# Commit 5435591

## Summary

Found one regression in the new split-tag buffering path for channel block replies.

## Findings

### Medium: text_end block replies can drop literal trailing final-tag prefixes

- Kind: data_loss
- File: `src/agents/pi-embedded-subscribe.ts`
- Line: 709
- Evidence: The new `stripBlockTags` buffers trailing known tag prefixes unless called with `{ final: true }` (`src/agents/pi-embedded-subscribe.ts:585`). Agent/SSE partial delivery passes that option on `text_end`, but `emitBlockChunk` still calls `stripBlockTags(text, state.blockState)` without final flushing (`src/agents/pi-embedded-subscribe.ts:709`). `text_end` then flushes block replies through `ctx.flushBlockReplyBuffer` (`src/agents/pi-embedded-subscribe.handlers.messages.ts:647`), and `message_end` skips the safety send once any text-end block reply was emitted (`src/agents/pi-embedded-subscribe.handlers.messages.ts:850`). Live reproduction with `onBlockReply`, `blockReplyBreak: "text_end"`, streamed delta `Answer ends with <fi`, then `text_end` and matching `message_end`, produced `[{"text":"Answer ends with",...}]` instead of preserving `<fi`.
- Impact: A legitimate assistant reply ending with a literal prefix such as `<`, `<fi`, `</fi`, `<think`, etc. can be truncated on channel block-streaming delivery. This is a regression from the parent behavior, where incomplete `<final>` tags were not stripped unless they completed the tag regex.
- Suggested fix: Make the text-end block-reply flush final-aware, for example by threading a `{ final: true }` option through `flushBlockReplyBuffer`/`emitBlockChunk` on `evtType === "text_end"` so pending literal fragments are emitted before `lastBlockReplyText` suppresses the message-end fallback. Add a block-reply regression alongside the new agent-event tests.
- Confidence: high
