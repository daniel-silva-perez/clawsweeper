## Summary

Keep this PR open. The patch changes the generic gateway-call fallback, but the linked WeCom media-send regression still travels through `message.action`, where current main injects an explicit 10000ms timeout before the changed fallback can apply.

## What This Changes

The PR branch modifies `src/gateway/call.ts` so gateway calls without a caller timeout resolve the shared 15000ms preauth handshake default instead of retaining the 10000ms fallback.

## Best Possible Solution

Align the gateway-executed `message.action` boundary so CLI plugin sends without a user-specified timeout do not inject an unrelated 10000ms budget, keep the behavior plugin-agnostic, simplify the generic fallback if retained, and add focused regression coverage for CLI gateway plugin sends with media-sized work.

## Reproduction Assessment

Yes. The linked `openclaw message send --channel wecom --media ...` report maps to `messageCommand` -> `runMessageAction` -> gateway plugin execution -> `callGatewayLeastPrivilege`, and current main visibly supplies `timeoutMs: 10_000` on that path; a fake gateway-plugin test can reproduce this without live WeCom credentials.

## Solution Assessment

No. The PR is a useful piece of generic cleanup, but it does not solve the reported path because that path passes an explicit timeout before the changed fallback runs; the safer fix is to align or centralize the message-action timeout boundary and cover it with tests.
