## Summary

Keep open for the automerge path. Current main still has the reported stale-label race, while this PR applies the narrow local-state fix, adds focused regression coverage, and has no blocking review or security findings.

## What This Changes

This PR moves Barnacle’s in-memory `labelSet` deletion outside the successful `removeLabel` call path and adds a regression test for a 404 label-removal race on GitHub App-authored PRs with stale `r: too-many-prs`.

## Best Possible Solution

Land this narrow PR through the enabled automerge/check gate so Barnacle treats a 404 label removal as an already-absent label and preserves the GitHub App active-PR-limit exemption without changing workflow permissions or Labeler policy.

## Reproduction Assessment

Yes. On current main, an App/Bot-authored PR with stale `r: too-many-prs` in the webhook payload reproduces the issue when `removeLabel` returns 404: the label remains in `labelSet` and the later active-limit rule can close the PR.

## Solution Assessment

Yes. Moving the local `labelSet.delete` after the 404-tolerant removal is the narrowest maintainable fix because current callers use `removeLabels` for active-limit cleanup and non-404 API errors still throw.
