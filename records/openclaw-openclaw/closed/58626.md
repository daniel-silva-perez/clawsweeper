## Summary

Keep open. Current main has partial cleanup for stale plugin config and bundled plugin load-path aliases, but it does not cover the reported upgrade case where `plugins.load.paths` points at a now-missing, versioned package directory from an older global install. That path still becomes a `plugins.load.paths` validation error, and stale plugin cleanup is intentionally blocked while discovery has errors.

## What This Changes

The issue asks OpenClaw to recover after upgrades leave versioned absolute bundled plugin paths in config, causing `openclaw gateway start` to fail until the paths are manually updated.

## Best Possible Solution

Add a targeted `doctor --fix` repair that recognizes missing old-version bundled plugin paths by package-layout plus plugin id, removes the redundant load-path entry when the current bundled plugin exists, and preserves strict errors for non-bundled or ambiguous paths.

## Reproduction Assessment

Yes. The source-level path is high-confidence: a missing configured `plugins.load.paths` entry emits `plugin path not found`, validation turns it into a `plugins.load.paths` issue, and the existing doctor repair only matches current-package aliases exactly.

## Solution Assessment

No, current main is not yet the best solution for this report. The narrow maintainable fix is to extend doctor bundled load-path repair, rather than adding startup-time config mutation or weakening all missing-path validation.
