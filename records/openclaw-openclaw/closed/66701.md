## Summary

Keep this open. Current main still preserves the reasoning/preamble text after stripping a trailing whitespace-delimited silent token, still treats glued suffixes like `here.NO_REPLY` as non-matches, and there are active open PRs with closing syntax proposing different fixes for the same leak class.

## What This Changes

The issue requests more tolerant silent-reply suppression when model output prepends reasoning or think text before `NO_REPLY`, including final-token and glued-token cases.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers choose and land one central silent-reply classifier used by normalization, final delivery, and any runner incomplete-turn checks, with regression coverage for bare `think`, `<think>...</think>`, whitespace-delimited trailing tokens, glued final-token examples, and the relevant streaming/block delivery behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the normalizer test for text ending in `NO_REPLY` expects a non-null preamble, and the token test keeps a glued suffix like `interject.NO_REPLY` unchanged; both feed the dispatcher path that queues channel delivery.

## Solution Assessment

Unclear. The current implementation is not the solution, and the open PRs intentionally choose different policies; the maintainable fix should centralize the classifier and avoid per-surface ad hoc checks while preserving any product-approved non-silent trailing-token cases.
