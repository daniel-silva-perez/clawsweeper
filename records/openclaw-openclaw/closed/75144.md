## Summary

Current main already fixes the systemd EADDRINUSE restart-loop problem in a broader and safer way than this PR. The main-branch fix handles the systemd-supervised healthy duplicate path that this PR's catch-only exit-code change misses, keeps unmanaged duplicate starts on the existing exit-0 path, adds targeted tests, and documents the behavior; the fix is currently unreleased on main.

## What This Changes

The PR changes gateway CLI lock-error handling so healthy `GatewayLockError` exits use code 78 and updates one coverage test expectation from `__exit__:0` to `__exit__:78`.

## Best Possible Solution

Close this PR as superseded by the current main implementation, keep the narrower systemd-aware behavior that preserves unmanaged duplicate-start semantics, and let the existing unreleased changelog entry carry the fix into the next release.

## Reproduction Assessment

Yes. The issue has a high-confidence unit-level reproduction: under a systemd supervisor, a duplicate gateway start that hits an already-running healthy gateway must resolve to exit 78; current main covers that path in `run.supervised-lock.test.ts`.

## Solution Assessment

No, this PR is not the best way to solve the issue as proposed because changing only the catch-path exit code misses the supervised healthy-gateway return path and broadens exit 78 to unmanaged duplicate starts. Current main is the safer solution because it targets systemd `Restart=always` behavior while preserving the old non-supervised success semantics.
