## Summary

Keep this PR open. Current main still does not implement the central WhatsApp inbound poll/reaction/event/RSVP support, and the open related reaction PR is only a narrower subset rather than a full supersession. The PR also has unresolved functional review concerns and mixes unrelated core/build changes, so the best maintainer path is review and narrowing rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer/contributor follow-up, but ask for a focused WhatsApp-owned patch: handle Baileys `messages.upsert`, `messages.update`, and `messages.reaction` through the inbound pipeline; preserve voter/reactor identity, original message IDs, selected poll options, reaction add/remove state, event time/location, and RSVP state; register and deregister named listeners; add targeted tests against the pinned Baileys contract; and split unrelated gateway, poll-parameter, and build-entry changes into separate PRs or drop them if main already covers them. Coordinate the reaction-only overlap with #46881 and the remaining reaction-querying work with #11460.
