## Summary

Keep open. Current main still routes `channels.telegram.*` config edits through a whole Telegram channel stop/start, and a fresh Telegram bot instance recreates the pending group-history map, so the channel-specific state-loss report remains valid even after the related restart-drain mitigation.

## What This Changes

Telegram config hot reload should avoid discarding Telegram channel in-memory context such as pending group history when `openclaw.json` changes.

## Best Possible Solution

Add a Telegram-owned state-retention path, or a generic channel reload seam if maintainers prefer, so safe Telegram config reloads either apply in place or carry pending group-history state across required stop/start cycles with account/token isolation and trimming to the new `historyLimit`. Keep #71465 as a mitigation rather than the canonical fix.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: change `channels.telegram.historyLimit`, observe it match the Telegram reload prefix, flow into `plan.restartChannels`, call whole-channel stop/start, and recreate `groupHistories` as an empty map in the new bot instance.

## Solution Assessment

Unclear. The current restart-drain behavior is a useful mitigation but not the complete fix; the narrow maintainable direction is Telegram-owned state preservation for pending group history unless maintainers choose to add a broader in-place channel reload API.
