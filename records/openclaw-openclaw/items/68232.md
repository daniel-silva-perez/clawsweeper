## Summary

Keep this issue open. Current main no longer treats every ordinary config path as a Telegram reload, and #71465 added active-work drain before restart-required reloads, but the reported Telegram-specific state-loss path still exists for `channels.telegram.*` changes: Telegram advertises a broad reload prefix, the gateway maps that to whole-channel stop/start, and Telegram group history is recreated as a fresh in-memory map on bot startup.

## Best Possible Solution

Keep this issue open. The best fix is a Telegram-owned or generic channel reload path that either applies safe Telegram config changes in place or preserves/rehydrates Telegram runtime state such as `groupHistories` across required stop/start cycles, with focused gateway reload and Telegram message-context regression tests. #71465 should stay treated as a related mitigation, not the canonical fix for this report.
