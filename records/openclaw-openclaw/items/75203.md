## Summary

Keep open: current main still has the reported source-level route. Synthetic isolated heartbeat sessions are persisted and listed like normal sessions, the Control UI picker adds them without a heartbeat/background filter, and WebChat sends user input to the currently selected session key.

## What This Changes

Ensure Control UI WebChat defaults user input to the primary main session and prevents synthetic `:heartbeat` isolated sessions from being selected as normal chat targets when the base session row was pruned.

## Best Possible Solution

Expose a marker-backed synthetic/background heartbeat-session contract to the Control UI, keep synthetic isolated heartbeat rows out of the normal chat picker or mark them read-only, and normalize the active WebChat target back to the resolved main session when the only visible sibling is the heartbeat row. Add focused regression coverage for session-list projection, picker option resolution, and `chat.send` target selection after the base main row is absent.

## Reproduction Assessment

Yes. The issue gives concrete config and current source inspection shows the necessary path: maintenance can prune the base row, isolated heartbeat saves a marked `:heartbeat` row, the session list and picker do not filter it, and WebChat sends to the selected key. I did not run a live gateway reproduction because this review is read-only.

## Solution Assessment

Yes for the requested direction, with one implementation constraint: the narrow maintainable fix should use `heartbeatIsolatedBaseSessionKey` or an explicit server-projected synthetic flag instead of relying only on a `:heartbeat` suffix. That preserves real custom sessions while making synthetic heartbeat rows safe in WebChat.
