## Summary

Keep open: current main still allows synthetic isolated-heartbeat sessions to appear as normal selectable Control UI chat sessions, and `chat.send` sends to whatever session key the UI has selected. The reported path remains plausible when maintenance prunes the base main row but a fresh `:heartbeat` row remains.

## What This Changes

The issue requests that Control UI/WebChat avoid routing user chat into synthetic `:heartbeat` sessions when the primary main session has been pruned or is absent from the visible session list.

## Best Possible Solution

Add a narrow session-list/UI contract for synthetic isolated-heartbeat rows: keep them available for audit where needed, but hide or mark them read-only/background in the Control UI chat picker and normalize default chat selection back to the resolved main session when the active selection is a synthetic heartbeat sibling. The main session should remain selectable even if its row was pruned, and user-originated `chat.send` should not target a synthetic heartbeat row by default.

## Reproduction Assessment

Yes. The issue gives concrete config and steps, and current code inspection shows the necessary path: maintenance can remove stale unpreserved rows, heartbeat writes a marked `:heartbeat` row, the session list does not filter it, and UI sends to the selected key. I did not run a live gateway reproduction because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the surface contract. The narrowest maintainable repair is likely to expose or use the existing `heartbeatIsolatedBaseSessionKey` marker so Control UI can hide/read-only those rows without relying on suffix matching, plus focused regression tests around session option resolution and `chat.send` target selection.
