## Summary

Current main still has the reported failure shape: isolated heartbeat rows can be listed as normal chat sessions, the picker has no heartbeat-background filter, and WebChat sends to the selected session key. Open #43984/#43737 overlap with the missing-main dropdown part, but they do not fully cover the synthetic heartbeat row/read-only routing problem described here.

## What This Changes

The issue asks for Control UI/WebChat to keep the primary main session available and prevent user-originated chat input from targeting synthetic isolated `:heartbeat` sessions after pruning.

## Best Possible Solution

Add a narrow session-list/UI contract for synthetic isolated heartbeat rows. Prefer exposing or using the existing `heartbeatIsolatedBaseSessionKey` marker so Control UI can hide or render those rows read-only/background, normalize default chat selection back to the resolved primary session, and keep the main session selectable even when aggressive maintenance has removed its stored row.

## Reproduction Assessment

Yes. The issue gives concrete config and current code inspection shows the path: pruning can remove the base session row, heartbeat writes a marked `:heartbeat` row, session listing/picker do not filter it, and `chat.send` uses the selected key.

## Solution Assessment

No patch is proposed here, so there is no implementation to bless. The narrowest maintainable fix is a shared session metadata contract plus focused Control UI selection/send normalization, coordinated with #43984 for the main-session visibility portion.
