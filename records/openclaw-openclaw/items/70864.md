## Summary

Keep open. Current main still lacks scoped mention-pattern policy, and the PR is a real implementation attempt, but the latest head still has a blocking Slack account-scope bug: the schema accepts account-level `mentionPatterns`, while the changed Slack runtime only resolves the top-level provider policy.

## What This Changes

The branch adds shared mention-pattern policy config and resolution, exposes it through channel runtime/SDK surfaces, wires allow/deny gating across channel handlers, and updates docs, generated config metadata, and tests.

## Best Possible Solution

Before merge, align the config contract with runtime behavior: pass each channel’s resolved account/provider `mentionPatterns` policy into `resolveMentionPatternsEnabled` where account-scoped config is exposed, starting with Slack, and add focused tests for named-account allow/deny behavior. If account-scoped policy is not intended for a channel, remove that schema/metadata exposure instead.

## Reproduction Assessment

Yes for the PR finding. A named Slack account with `channels.slack.accounts.<id>.mentionPatterns.mode: "deny"` and global pattern matching allowed would still evaluate regex mentions because the new Slack call reads only the top-level provider policy unless `providerPolicy` is passed.

## Solution Assessment

No. The general resolver approach is maintainable, but exposing account-scoped policy while omitting the merged account policy from Slack runtime resolution is not the narrowest correct implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor Slack account-scoped mention policy:** `extensions/slack/src/monitor/message-handler/prepare.ts:314-320`
  - body: `SlackAccountSchema` now accepts `channels.slack.accounts.<id>.mentionPatterns`, but this new resolver call only lets policy resolution see the top-level `channels.slack.mentionPatterns`. For named Slack accounts, account-scoped `mode`, `allowIn`, or `denyIn` settings are silently ignored, so configured regex mentions can be accepted or dropped against the wrong policy. Pass the merged account policy, for example from `account.config.mentionPatterns`, or stop exposing the account-scoped key.
