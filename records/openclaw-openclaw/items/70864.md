## Summary

Keep open. The feature is not on current main, and the PR is a real implementation attempt, but the latest diff still has merge-blocking contract issues: it exposes account-scoped mention-pattern policy for several channels whose runtime calls only read top-level provider policy, and it repurposes IRC's existing `mentionPatterns` array key as a policy object without aligning the public IRC types/local schema or preserving compatibility.

## What This Changes

The branch adds scoped mention-pattern policy config/resolution, exposes SDK/runtime helpers, gates configured regex mention matching across bundled channels, and updates docs, generated config metadata, and tests.

## Best Possible Solution

Keep the shared resolver approach, but align the config contract before merge: pass each channel's merged account policy into mention-policy resolution wherever account-scoped `mentionPatterns` is exposed, add focused account-scope tests for those channels, and avoid breaking IRC's existing `mentionPatterns: string[]` contract by using a separate policy key or a documented backward-compatible migration.

## Reproduction Assessment

Yes for the PR findings. A named Signal account with `channels.signal.accounts.work.mentionPatterns.mode: "deny"` and global/top-level policy allowing patterns will still evaluate regex mentions, and an existing IRC config using `channels.irc.mentionPatterns: ["openclaw"]` no longer matches the changed core schema shape.

## Solution Assessment

No. The central policy resolver is a maintainable direction, but the current patch is not the best mergeable solution until account-scoped schema exposure matches runtime behavior and the IRC key is handled without a public contract break.

Keep the old array key compatible or introduce the policy under a separate key and update all IRC types/schemas together.
