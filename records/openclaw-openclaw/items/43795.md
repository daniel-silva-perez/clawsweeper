## Summary

Keep open. Current main still has an unguarded OpenAI Responses/WebSocket assistant-content conversion path that can call array methods or iterate when provider output `message.content` is not an array; the prior linked fixes were closed unmerged, and newer replay cleanup only covers part of the malformed-history surface.

## What This Changes

The issue reports repeated `500 v.content is not iterable` failures during Feishu-routed embedded runs with a third-party OpenAI-compatible GLM provider returning malformed assistant content.

## Best Possible Solution

Add a focused defensive normalization boundary for malformed OpenAI Responses assistant `message.content` before array methods or iteration, and align replay/session sanitization so string, object, null, and invalid content shapes cannot re-enter provider payloads or retry loops. Preserve well-formed array behavior, add targeted regression tests for the converter and replay paths, and credit the prior #43836/#43864 analysis in the PR body.
