## Summary

Keep open. Current main still has provider-agnostic OpenAI Responses and replay paths that can iterate assistant `content` before proving it is array-shaped, and the two linked repair PRs for this report were closed unmerged.

## What This Changes

The issue reports a 500 retry loop when a third-party OpenAI-compatible GLM route returns malformed non-array assistant `content`.

## Best Possible Solution

Add a generic malformed assistant-content normalization boundary before OpenAI Responses output and replayed assistant turns are inspected. Preserve valid arrays, convert strings into text blocks where replayable, wrap only safe block-shaped objects, and drop or sentinel null/unrecognized values according to existing error-turn policy.

## Reproduction Assessment

Yes. A unit-level malformed Responses message with object-shaped `content`, or a replayed assistant turn with object/null `content`, reaches current-main code paths that call `.some(...)` or `for...of` without proving the value is an array.

## Solution Assessment

Yes. The narrowest maintainable path is provider-agnostic hardening at the Responses conversion and replay normalization boundaries, with regression tests for string, object, null, and undefined shapes rather than a GptGod-specific workaround.
