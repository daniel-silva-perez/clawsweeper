## Summary

Keep open: current main still has unsafe assistant-content iteration paths that can throw on malformed OpenAI-compatible provider payloads, and the two linked repair PRs were closed unmerged. Recent replay work covers legacy string content, but object/null assistant content can still pass through and later be iterated.

## What This Changes

The issue reports a Feishu/embedded agent regression where gptgod/glm-4-air repeatedly fails with `500 v.content is not iterable` after an error/tool-fetch path.

## Best Possible Solution

Add a generic malformed assistant-content normalization boundary before OpenAI Responses provider output and replayed assistant turns are inspected: preserve valid arrays, convert strings to text blocks where replayable, wrap safe single block-shaped objects, and drop or sentinel null/unrecognized content according to the existing error-turn policy. Cover the Responses converter and replay path with focused regression tests without adding provider-specific core logic.

## Reproduction Assessment

Yes. A malformed Responses message with object-shaped `content`, or a replayed assistant turn with object/null `content`, reaches current code paths that call `.some(...)` or `for...of` without an `Array.isArray` guard.

## Solution Assessment

Yes, the best fix is a small generic normalization/guard in the core agent/replay boundary. The closed PRs point at the right surface area, but a current-main replacement should be rebased around the newer replay policy and avoid provider-specific `gptgod` handling.
