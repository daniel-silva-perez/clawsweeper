## Repair Work Prompt

Fix openclaw/openclaw#43795. Third-party OpenAI-compatible providers such as the reported GptGod GLM route can return OpenAI Responses assistant output `message.content` as a string, single object, null, or another malformed shape instead of an array, causing repeated 500 failures like `v.content is not iterable`. Add a focused defensive normalization boundary in the agent response/replay path, starting with `buildAssistantMessageFromResponse` in `src/agents/openai-ws-message-conversion.ts`, so all `.some()` and `for...of` uses see an array. Preserve current behavior for well-formed arrays; handle strings as text blocks, valid single content-part objects as one-item arrays when safe, and null/invalid shapes without crashing. Review `sanitizeSessionMessagesImages` and `normalizeAssistantReplayContent` for persisted/replayed assistant object/null content and normalize or drop invalid content consistently. Do not change provider defaults, third-party provider IDs, auth routing, model selection, or broad gateway behavior unless a focused test proves that path is part of the failure. Use #43864 and #43836 as source context, but write the final patch against current main. Add focused regression tests for malformed string, object, null, and invalid assistant content in completed Responses conversion and replay sanitization. Add a single-line user-facing changelog fix entry if the repo's active changelog requires it, crediting the reporter/context contributors such as @1090963827 and @openperf, and do not credit bots.
