## Summary

Keep open: current main still has unsafe assistant-content iteration for malformed object/null content in OpenAI Responses conversion and replay, while the linked repair PRs were closed unmerged. Some string-content replay handling has landed, but it does not cover the non-iterable shapes that match the reported failure.

## What This Changes

The issue reports repeated `500 v.content is not iterable` failures from the embedded Feishu agent path when using the third-party `gptgod/glm-4-air` OpenAI-compatible provider after a web_fetch failure.

## Best Possible Solution

Add one generic normalization/guard boundary for malformed assistant content before provider responses or replayed assistant turns are inspected: preserve valid arrays, convert string content to a text block where replayable, wrap single block-shaped objects when safe, and drop or sentinel null/unrecognized content according to existing error-turn policy. Cover the OpenAI Responses converter and replay path with focused regression tests, without adding provider-specific `gptgod` logic to core.
