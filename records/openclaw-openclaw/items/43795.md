## Summary

Keep open. Current main still has reachable OpenAI Responses conversion and replay paths that assume assistant `content` is array-shaped, while the report and related closed PRs describe third-party OpenAI-compatible responses with string, object, or null content. The two repair PRs were closed unmerged, and a current search found no open paired PR for this issue.

## What This Changes

The report asks OpenClaw to stop repeated 500 `v.content is not iterable` failures when GptGod/GLM-4-Air returns malformed assistant content during Feishu-triggered agent runs.

## Best Possible Solution

Add a generic malformed assistant-content normalization boundary before OpenAI Responses output and replayed assistant turns are inspected. Preserve valid arrays, convert strings into text blocks where replayable, wrap only safe single block-shaped objects that match the existing content-block contract, and drop or sentinel null/unrecognized values according to current error-turn policy.

## Reproduction Assessment

Yes. A malformed Responses message with object-shaped `content`, or a replayed assistant turn with object/null `content`, reaches current-main code paths that call `.some(...)` or `for...of` without proving the value is an array.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-agnostic hardening at the Responses conversion and replay normalization boundaries, with focused regression tests for string, object, and null shapes rather than provider-specific GptGod handling.
