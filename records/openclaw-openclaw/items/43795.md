## Repair Work Prompt

Fix the malformed assistant-content regression behind `500 v.content is not iterable` for OpenAI-compatible Responses providers. Current main still lets `buildAssistantMessageFromResponse` call `.some()` and `for...of` on `item.content` before proving it is an array, lets `openresponses-http` iterate message content after only excluding strings, and only normalizes string assistant replay content. Add a generic normalization/guard boundary for assistant `message.content` so string, single object, null/undefined, and other non-array payloads cannot crash agent conversion or replay; preserve current behavior for well-formed arrays. Do not add provider-specific IDs, `gptgod` special cases, startup migrations, or broad core policy changes. Add focused regression tests for `buildAssistantMessageFromResponse`, replay/session sanitization, and `openresponses-http` if touched. Add a single-line changelog fix entry under the active version if required for this user-facing regression, crediting the reporter and prior analysis appropriately without bot credits.
