## Summary

Keep this issue open. Current main still exposes `enqueueSystemEvent` as an in-memory, session-scoped system-event queue with consecutive text duplicate suppression and a boolean return; it does not accept `idempotencyKey`, does not return an enqueue/dedupe result object, and is documented as ephemeral. Adjacent durable idempotency primitives exist, but they do not implement the requested public `enqueueSystemEvent` contract.

## Best Possible Solution

Keep this issue open for maintainer/API design. The best path is an additive core/plugin-SDK change that either adds scoped durable idempotency to `enqueueSystemEvent` itself or documents and exposes an equivalent durable resume primitive that preserves system-event queue behavior and heartbeat wake semantics. Coverage should include same key across different sessions, duplicate return shape, process-restart persistence, payload mismatch handling, and cleanup after event consumption.

## Risks / Open Questions

- This is a public plugin SDK/runtime surface, so the safest implementation would be additive and preserve existing callers that rely on the current boolean return unless a broader API migration is planned.
- Durable idempotency for system events needs a clear lifecycle: key scope, payload mismatch behavior, wake semantics, cleanup after consumption, and restart behavior should be tested explicitly.
