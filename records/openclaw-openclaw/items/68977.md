## Summary

Keep this issue open. Current main has adjacent idempotency primitives, but the public `enqueueSystemEvent` API still only supports an ephemeral in-memory queue with consecutive-text duplicate suppression, a boolean return, and no `idempotencyKey` or durable scoped dedupe contract. This remains a plausible generic plugin SDK reliability enhancement, not a ClawHub-only plugin request.

## Best Possible Solution

Keep the issue open for maintainer/API design. The likely best path is an additive plugin SDK/core change that either adds scoped durable idempotency to `enqueueSystemEvent` itself or explicitly redirects plugin resume flows to a documented durable alternative that also wakes the target session and preserves the needed system-event semantics. Coverage should include cross-session same-key behavior, duplicate return shape, process-restart persistence, and cleanup after consumption.
