## Summary

Keep this PR open. Current main still does not implement the PR’s intended whole-tool hard timeout for `memory_search`; the memory tool now lives under `extensions/memory-core`, where manager acquisition and search are awaited directly. The linked liveness bug #49524 remains a valid implementation target, but the PR needs to be ported/rebased onto the current plugin-owned files.

## Best Possible Solution

Keep the PR open as a relevant implementation candidate for #49524, but rework it against current main: add a bounded timeout around the current `extensions/memory-core` `memory_search` execution path covering manager acquisition plus search, add timeout-specific unavailable metadata in `extensions/memory-core/src/tools.shared.ts`, and add a fake-timer regression test in `extensions/memory-core/src/tools.test.ts`.

## Risks / Open Questions

- The PR no longer applies to the current owner files because `memory_search` has moved from `src/agents/tools` into `extensions/memory-core`.
- A fixed 30-second wall-clock timeout needs maintainer review against documented longer embedding timeouts and QMD behavior.
- A `Promise.race` timeout can let the user turn recover while underlying manager/search work continues; the port should consider cache and pending-manager side effects.
