## Summary

Keep this PR open for maintainer review. Current main still opens the QQBot gateway WebSocket directly without a proxy agent, and the PR is a narrow owner-local fix using an existing SDK helper with focused tests; I did not find a blocking correctness issue, but maintainers should confirm the ambient proxy default in light of the related Feishu proxy discussion.

## What This Changes

This PR replaces QQBot gateway WebSocket construction with a `createQQWSClient` helper that applies `resolveAmbientNodeProxyAgent` and adds proxy/no-proxy regression tests.

## Best Possible Solution

Land an owner-local QQBot WebSocket helper if maintainers agree QQBot gateway traffic should honor ambient HTTP(S) proxy env; otherwise adjust this PR to the chosen explicit or shared proxy policy before merge.

## Reproduction Assessment

Yes for the code-level behavior: current main creates the QQBot gateway `WebSocket` without an `agent`, while the proposed helper can be exercised with `HTTPS_PROXY` or `HTTP_PROXY` set and a mocked `ws` constructor. Full network reproduction requires a QQ/private-egress environment like the one described in the PR body.

## Solution Assessment

Yes for the narrow implementation shape: it stays inside the QQBot plugin, reuses the existing SDK helper, and mirrors current Feishu wiring without core changes. The only unresolved question is product policy around ambient proxy inheritance, not the mechanics of this patch.
