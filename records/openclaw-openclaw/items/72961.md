## Summary

Keep this PR open for maintainer review. Current main still creates the QQBot gateway WebSocket without an agent, while the PR adds a narrow plugin-local helper using the existing ambient proxy helper and tests the key proxy/no-proxy paths. I did not find a discrete blocking code-review bug, but the related Feishu proxy-default reports mean the remaining decision is proxy policy and normal CI/changelog review.

## What This Changes

This PR routes QQBot gateway WebSocket creation through a new helper that resolves an ambient HTTP(S) proxy agent and adds focused regression tests for proxy and no-proxy wiring.

## Best Possible Solution

Land the owner-local QQBot helper if maintainers accept ambient HTTP(S) proxy inheritance for QQBot gateway traffic; otherwise adjust this PR to the chosen cross-channel proxy policy before merge, ideally with a small docs/changelog note if user-visible proxy behavior changes.

## Reproduction Assessment

Yes for the code-level behavior: current main creates the QQBot gateway WebSocket without an agent, while the PR helper can be exercised with HTTPS_PROXY or HTTP_PROXY set and a mocked ws constructor. Full network reproduction still requires a QQ/private-egress environment like the one described in the PR body.

## Solution Assessment

Unclear at the product-policy level, but the mechanics are narrow and maintainable if ambient proxy inheritance is desired. If maintainers instead accept the Feishu opt-in direction from #65799/#65802, QQBot should use that explicit policy rather than copying the current Feishu default.
