## Summary

Current main still does not implement the requested isolation for pre-compaction memoryFlush. The flush is awaited in the auto-reply path, dispatched through runEmbeddedPiAgent without an explicit lane, and omitted lanes still resolve to CommandLane.Main. The related implementation PR #53154 was closed unmerged, so there is no shipped fix to close against.

## Best Possible Solution

Keep this issue open. The best fix is to route pre-compaction memoryFlush through an explicit background or dedicated compaction lane with its own bounded timeout/concurrency policy, and add regression coverage proving the captured runEmbeddedPiAgent call cannot fall back to CommandLane.Main. The design should also address same-session responsiveness, not only process-wide main-lane contention.

## Risks / Open Questions

- The default main-lane concurrency is now 4, which reduces process-wide starvation compared with a maxConcurrent=1 deployment, but it does not isolate memoryFlush from main and configs may still set maxConcurrent to 1.
- Moving only the global lane may still leave same-session inbound messages blocked by the session lane; maintainers need to decide whether memoryFlush can bypass, yield, or be preempted within per-session serialization.
