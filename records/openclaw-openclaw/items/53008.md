## Summary

Keep open. Current main still matches the reported compaction-starvation shape: the foreground reply flow awaits both preflight compaction and memoryFlush, those calls still omit a dedicated maintenance lane, omitted embedded lanes still resolve to `main`, and same-session serialization still precedes the global lane choice. Recent stuck-session recovery helps stale-lane cleanup, but active embedded/reply/lane work remains warning-only by default, so it does not solve this compaction-specific outage path.

## What This Changes

The issue reports that memory compaction and memoryFlush can run in-band on the main/same-session processing lane and make inbound Telegram messages wait behind a long compaction for 10 or more minutes.

## Best Possible Solution

Define an explicit compaction scheduling and recovery contract for both preflight compaction and memoryFlush: maintenance work must be bounded and unable to starve interactive inbound messages, lane choice must not silently fall back to `CommandLane.Main`, same-session consistency during transcript rotation must be documented in code/tests, and diagnostic recovery should distinguish safe compaction cancellation from ordinary long-running agent turns.

## Reproduction Assessment

Yes. The issue includes a concrete log timeline, and current source gives a high-confidence source-level reproduction path: trigger preflight compaction or memoryFlush during a foreground reply, observe that it awaits in the active reply operation, enters default `main`/session lanes, rejects steering while compacting, and leaves later same-session inbound messages queued.

## Solution Assessment

No single closed PR or current-main implementation is the best fix yet. The closed #53154-style lane move is too narrow because same-session serialization and cancellation/session-rotation semantics remain; the better fix needs maintainer-owned policy plus regression coverage for queued inbound messages during active compaction.
