## Summary

Keep open. Current main still runs preflight compaction and memoryFlush inside the foreground reply operation, omitted embedded lanes still resolve to `main`, and same-session serialization still happens before the global lane. Recent diagnostic recovery work helps stale lane cleanup, but it remains observe-only for active reply/embedded work by default and does not provide the dedicated compaction scheduling policy requested here.

## What This Changes

The issue reports that memory compaction and memoryFlush can occupy the main processing lane long enough to make inbound Telegram messages appear unresponsive for 10 or more minutes.

## Best Possible Solution

Define a compaction-specific scheduling and recovery policy for both preflight compaction and memoryFlush: memory maintenance should be bounded and unable to starve interactive inbound work, lane selection must not silently fall back to `CommandLane.Main`, same-session consistency rules must be explicit, and regression coverage should prove queued inbound messages drain safely while compaction is active.
