## Summary

Keep open. Current main still has the reported starvation path: preflight compaction and memoryFlush are awaited inside the reply flow, neither path selects a dedicated compaction/background lane, omitted lanes still resolve to `main`, and same-session compaction blocks steering/follow-up delivery until the active work finishes or is explicitly aborted.

## What This Changes

The issue asks for memory compaction and memoryFlush work to stop blocking inbound Telegram message processing for the full compaction timeout.

## Best Possible Solution

Define a compaction-specific scheduling policy for memoryFlush and preflight compaction: a bounded background or dedicated compaction path that cannot fall back to `CommandLane.Main`, has clear cancellation and recovery behavior, preserves transcript/session rotation consistency, and includes regression coverage for queued inbound messages during compaction.
