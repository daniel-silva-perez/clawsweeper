## Summary

Keep this open. Current main has some adjacent stuck-session recovery, but the specific memoryFlush compaction path still runs inline before the user turn continues, does not pass an explicit background/compaction lane, and omitted lanes still default to the shared main lane. The linked implementation candidate #53154 is closed unmerged, while #73581 is broader follow-up context rather than a shipped replacement for this specific compaction-lane bug.

## What This Changes

The issue asks for pre-compaction memoryFlush work to stop blocking inbound message processing, with shorter or configurable timeouts and recovery when compaction stalls.

## Best Possible Solution

Keep this issue open and choose a compaction-specific scheduling policy: route memoryFlush/pre-compaction work onto an explicit background or dedicated compaction lane with bounded timeout/concurrency, and decide how that work should yield, bypass, or be preempted within same-session serialization. Regression coverage should prove memoryFlush cannot fall back to CommandLane.Main and that inbound user messages do not starve while compaction is pending or stuck.
