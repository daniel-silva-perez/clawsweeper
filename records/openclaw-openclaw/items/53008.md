## Summary

Keep this open. Current main still runs pre-compaction memoryFlush as awaited foreground work in the reply path, the embedded memory run does not pass an explicit background/compaction lane, omitted lanes still resolve to `main`, and active stuck-session recovery remains warning-only by default. The prior narrow implementation candidate #53154 is closed unmerged, while #73581 is broader related follow-up rather than a shipped fix for this compaction-lane starvation case.

## What This Changes

The issue requests that memoryFlush/pre-compaction work stop blocking inbound message processing for minutes, with bounded timeout and recovery behavior.

## Best Possible Solution

Define and implement a compaction-specific scheduling policy: memoryFlush/pre-compaction work should use an explicit bounded background or dedicated compaction lane and have regression coverage proving it cannot fall back to `CommandLane.Main`, while same-session serialization, cancellation, and recovery semantics are handled deliberately.
