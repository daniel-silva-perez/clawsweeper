## Summary

Keep open. Current main still runs memoryFlush as awaited foreground work in the reply path, the embedded memory run does not request a background or compaction lane, omitted embedded lanes still default to `main`, and same-session work is serialized before the global lane. The newer default `steer` queue mode does not resolve compacting runs because compacting embedded runs refuse steering and fall back to queued followups. The closed #53154 PR was not merged, and #73581 is broader related follow-up rather than a shipped fix for this specific starvation path.

## What This Changes

The issue asks OpenClaw to prevent pre-compaction memoryFlush work from blocking inbound Telegram message processing for many minutes.

## Best Possible Solution

Define a compaction-specific scheduling policy: memoryFlush and preflight compaction should use an explicit bounded background or dedicated compaction path that cannot fall back to `CommandLane.Main` or block inbound same-session processing, with deliberate transcript consistency, cancellation, and recovery semantics plus regression coverage for the queue behavior.
