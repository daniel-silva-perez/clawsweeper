## Summary

Keep this PR open because current main still lacks WhatsApp inbound reaction forwarding, but the proposed patch is not mergeable as-is: it misreads the Baileys reaction actor contract and leaves a deterministic reaction-removal dedupe bug. The supplied discussion also shows repeated review attempts plus an overlapping broader open PR, so this needs maintainer selection/review rather than cleanup closure.

## What This Changes

The PR adds a WhatsApp `messages.reaction` listener that forwards non-empty reactions to `onMessage` with `reactionEmoji` and `reactionMessageId` metadata, and moves WhatsApp inbound activity recording after filtering.

## Best Possible Solution

Land one WhatsApp-owned inbound reaction implementation that follows Baileys' `{ key, reaction }` event contract, routes accepted reactions through the current access-control and claimable-dedupe patterns, handles removal and re-reaction behavior deliberately, cleans up listeners through the current lifecycle helper, and adds focused tests plus docs/changelog coverage for any agent-visible contract change.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to feed the proposed handler a Baileys-shaped group reaction where the actor is on `reaction.key` and no top-level `userJid` exists; the handler computes no reactor and skips the event. The removal/re-reaction bug is also deterministic because the falsey emoji branch runs before dedupe key clearing.

## Solution Assessment

No. Forwarding WhatsApp reactions is a valid direction, but this patch is not the best mergeable solution as-is; it should be rebased onto current main, read actor identity from the Baileys reaction event key, and use the current access-control and dedupe lifecycle deliberately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read the reactor from the reaction event key:** `extensions/whatsapp/src/inbound/monitor.ts:453-455`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`; the top-level `key` is the reacted-message key, while the reaction event key is on `reaction.key`. This handler reads a non-existent top-level `userJid`, so group reactions without that field are skipped and access-control attribution can be wrong.
  - confidence: 0.89
- **[P2] Clear reaction dedupe entries on removals:** `extensions/whatsapp/src/inbound/monitor.ts:442-443`
  - body: Reaction removals have falsey `reaction.text`, so this early `continue` runs before the dedupe key is computed or cleared. Re-applying the same emoji to the same message within the dedupe TTL can then be silently dropped.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds no supply-chain surface, but the new WhatsApp reaction input path has a concrete actor-identity/access-control concern.

Concerns:

- **[medium] Reaction actor identity can be misclassified:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: The handler derives the reactor from a top-level `userJid` that Baileys does not emit for `messages.reaction`; it should use the reaction event key. Misclassification affects whether a reaction is treated as self/outbound, allowed, blocked, or attributed to a group sender.
  - confidence: 0.86
