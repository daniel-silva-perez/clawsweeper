## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks WhatsApp inbound reaction forwarding, and the proposed handler has an access-control-sensitive reactor identity bug plus a removal/re-reaction dedupe edge case; maintainers also need to choose how this overlaps with the broader open WhatsApp event PR.

## What This Changes

The PR adds a WhatsApp `messages.reaction` listener that forwards non-empty reactions as inbound messages with reaction metadata, moves one inbound activity metric call, and extends the WhatsApp inbound message type.

## Best Possible Solution

Land one WhatsApp-owned inbound reaction implementation that follows the Baileys event contract, routes accepted reactions through the current inbound access-control and claimable-dedupe patterns, handles removal and re-reaction behavior deliberately, cleans up listeners through the current lifecycle helper, and adds focused tests plus docs/changelog coverage for any agent-visible contract change.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: feed the proposed handler a Baileys-shaped group reaction with reactor identity on `reaction.key` and no top-level `userJid`; it computes `reactorJid` as undefined and skips the event. The removal/re-reaction bug is also deterministic because the falsey emoji branch runs before the dedupe key is computed or cleared.

## Solution Assessment

No. The requested product direction is valid, but this patch is not the best mergeable solution as-is; it should read actor identity from the Baileys reaction event key, handle removal dedupe explicitly, rebase onto current mainâ€™s lifecycle/dedupe APIs, and be consolidated with or chosen over the broader WhatsApp event PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Read the reactor from `reaction.key`:** `extensions/whatsapp/src/inbound/monitor.ts:453-455`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`, where the top-level `key` is the reacted-message key and the reaction event key is on `reaction.key`; there is no top-level `userJid`. Because this handler reads `userJid`, group reactions without that non-existent field hit the `group && !reactorJid` guard and are never forwarded.
  - confidence: 0.89
- **[P2] Clear reaction dedupe entries on removals:** `extensions/whatsapp/src/inbound/monitor.ts:442-443`
  - body: Reaction removals have a falsey `reaction.text`, so this early `continue` runs before the dedupe key is computed or cleared. The previous reaction remains cached for the 20-minute window, and re-applying the same emoji to the same message can be silently dropped.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds no supply-chain surface, but the new reaction input path has a concrete actor-identity/access-control concern.

Concerns:

- **[medium] Reaction actor identity can be misclassified:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: The handler derives the reactor from a top-level `userJid` that Baileys does not emit for `messages.reaction`; it should use the reaction event key. Misclassification affects whether a reaction is treated as self/outbound, allowed, blocked, or attributed to a group sender.
  - confidence: 0.86
