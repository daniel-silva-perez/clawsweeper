## Summary

Keep this PR open. Current main still lacks WhatsApp inbound reaction forwarding, so the feature is not obsolete, but the proposed patch is not safe to merge as-is because it reads a non-Baileys actor field, mishandles reaction removals in the dedupe lifecycle, and is stale against the current WhatsApp inbound pipeline.

## What This Changes

The PR adds WhatsApp `messages.reaction` handling, exposes `reactionEmoji` and `reactionMessageId` on inbound WhatsApp messages, registers cleanup for the listener, and moves inbound activity recording later in the existing upsert path.

## Best Possible Solution

Land one WhatsApp-owned implementation on current main that follows Baileys’ `{ key, reaction }` event contract, derives the reactor from `reaction.key`, uses the current normalize/enrich/enqueue and claimable-dedupe lifecycle, deliberately handles removals and re-reactions, cleans up listeners through the current lifecycle helper, and adds focused inbound tests plus docs/changelog coverage for the agent-visible reaction contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to invoke the proposed handler with a Baileys-shaped group reaction where the actor exists on `reaction.key` and no top-level `userJid` is present; the handler treats the reaction as unattributable and skips it. The remove/reapply case is also deterministic because the falsey emoji branch runs before any dedupe invalidation.

## Solution Assessment

No. Forwarding WhatsApp reactions is valid work, but this PR is not the best merge path as-is because it relies on a non-contract actor field, leaves a dedupe lifecycle bug for removals, and needs rebasing against current main’s inbound access-control, listener, and claimable-dedupe structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read the reactor from reaction.key:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`, with the actor-sensitive message key on `reaction.key`; there is no typed top-level `userJid`. This handler reads `userJid`, so Baileys-shaped group reactions can be skipped or attributed through the wrong identity before access control.
  - confidence: 0.9
- **[P2] Clear reaction dedupe state on removals:** `extensions/whatsapp/src/inbound/monitor.ts:443`
  - body: Removal events have falsey `reaction.text`, so this early `continue` leaves prior reaction dedupe state intact. Removing and then re-applying the same emoji to the same message within the dedupe window can make the second reaction invisible to the agent.
  - confidence: 0.83
