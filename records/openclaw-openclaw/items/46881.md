## Summary

Keep this PR open for maintainer review. Current main still does not forward WhatsApp reaction events, but this branch is stale against the current WhatsApp inbound pipeline and has concrete Baileys actor-identity and reaction-removal dedupe defects; the overlapping open #48570 also needs maintainer scope selection before either implementation lands.

## What This Changes

The PR adds a WhatsApp `messages.reaction` listener, forwards non-empty reactions to `onMessage` with `reactionEmoji` and `reactionMessageId`, and moves WhatsApp inbound activity recording later in the filtering path.

## Best Possible Solution

Land one WhatsApp-owned reaction implementation on current main that follows Baileys' `{ key, reaction }` contract, reads actor identity from `reaction.key`, routes accepted events through the current access-control, debouncer, and claimable-dedupe lifecycle, handles removal/re-reaction behavior deliberately, cleans up listeners, and adds focused WhatsApp inbound tests plus docs/changelog coverage for the agent-visible contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass the proposed handler a Baileys-shaped group reaction where the top-level `key` is the reacted-message key and the actor is only present on `reaction.key`; the handler sees no `userJid` and skips it. The remove/reapply case is also deterministic because the falsey emoji branch runs before any dedupe eviction or prefix clearing.

## Solution Assessment

No. Forwarding WhatsApp reactions is valid product work, but this patch is not the narrowest maintainable solution as-is because it bypasses parts of the current inbound pipeline, uses an actor field outside the Baileys contract, and needs a rebase onto the current retry-safe dedupe and plugin SDK boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read the reactor from reaction.key:** `extensions/whatsapp/src/inbound/monitor.ts:453-455`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`, where the top-level `key` is the reacted-message key and the reaction event key is on `reaction.key`. This code reads a non-contract top-level `userJid`, so group reactions are skipped and DM self/peer reactions can be misattributed before access control.
  - confidence: 0.9
- **[P2] Handle removal before suppressing re-reactions:** `extensions/whatsapp/src/inbound/monitor.ts:442-443`
  - body: Reaction removals have falsey `reaction.text`, so this early `continue` leaves the previous reaction's dedupe entry intact. Removing and re-applying the same emoji to the same message within the dedupe TTL can make the second reaction invisible to the agent.
  - confidence: 0.83
