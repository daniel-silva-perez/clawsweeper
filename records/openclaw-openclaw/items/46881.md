## Summary

Keep this PR open. Current main still does not forward WhatsApp inbound reactions to the agent, so the intended feature is not obsolete; however, the branch is not merge-ready because it reads the Baileys reaction actor from a nonexistent top-level field, has a reaction-removal dedupe edge case, is stale against the current WhatsApp monitor/dedupe structure, and overlaps broader open WhatsApp reaction work.

## What This Changes

The PR adds WhatsApp `messages.reaction` handling that forwards non-empty reaction events as `WebInboundMessage` objects with `reactionEmoji` and `reactionMessageId` metadata, plus listener cleanup.

## Best Possible Solution

Land one WhatsApp-owned inbound reaction implementation that follows the Baileys event contract (`key` is the reacted message, `reaction.key` identifies the reaction event/actor), routes accepted reactions through the existing inbound access-control and dedupe patterns, handles removal/re-reaction behavior explicitly, cleans up listeners through the current lifecycle helper, and adds focused tests plus docs/changelog coverage if the agent-visible contract changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Read the reactor from `reaction.key`:** `extensions/whatsapp/src/inbound/monitor.ts:453-455`
  - body: Baileys v7 emits `messages.reaction` as `{ key, reaction }`, where `key` is the reacted-message key and the reaction event key is on `reaction.key`; there is no top-level `userJid`. Because this code reads `userJid` and never reads `reaction.key`, group reactions hit the `group && !reactorJid` guard and are never forwarded.
  - confidence: 0.89
- **[P2] Clear reaction dedupe entries on removals:** `extensions/whatsapp/src/inbound/monitor.ts:442-443`
  - body: Reaction removals have a falsey `reaction.text`, so this early `continue` runs before the dedupe key is computed. The previous reaction stays cached for the 20-minute window, and re-applying the same emoji to the same message is silently dropped.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff does not add supply-chain surfaces, but it introduces agent input from WhatsApp reactions with a concrete actor-identity/access-control bug.

Concerns:

- **[medium] Reaction actor identity can be misclassified:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: The handler derives the reactor from a top-level `userJid` that Baileys does not emit for `messages.reaction`; it should use the reaction event key. Misclassification here affects whether a reaction is treated as self/outbound, allowed, blocked, or attributed to a group sender.
  - confidence: 0.86
