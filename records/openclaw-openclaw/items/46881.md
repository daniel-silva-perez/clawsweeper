## Summary

Keep this PR open, but do not merge it as-is. Current main still has no WhatsApp inbound reaction forwarding, so the feature remains valid; however, the PR has actor-identity and removal/dedupe correctness defects, is stale against the current WhatsApp inbound pipeline, and overlaps the broader open WhatsApp media/event PR #48570.

## What This Changes

The PR adds WhatsApp `messages.reaction` handling, exposes `reactionEmoji` and `reactionMessageId` on inbound WhatsApp messages, and forwards accepted reaction events to the agent.

## Best Possible Solution

Land one WhatsApp-owned implementation on current main that follows Baileys’ `{ key, reaction }` contract, derives the reactor from `reaction.key`, routes accepted reactions through the current normalize/enrich/enqueue and claimable-dedupe lifecycle, deliberately handles removals and re-reactions, cleans up listeners, and adds focused WhatsApp inbound tests plus docs/changelog coverage for the agent-visible contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call the proposed handler with a Baileys-shaped group reaction where the reacted-message key is top-level and the reactor exists only on `reaction.key`; the handler sees no top-level `userJid` and skips it. The remove/reapply case is also deterministic because the falsey emoji branch runs before any dedupe clearing.

## Solution Assessment

No. Forwarding WhatsApp reactions is valid product work, but this PR is not the best merge path as-is because it reads a non-contract actor field, bypasses current-main inbound pipeline changes, and needs rebasing against the current claimable dedupe and plugin SDK boundaries.
