## Summary

Keep this issue open. Current main treats Pi transcripts as a tree/DAG and OpenClaw history readers follow the active branch, so a naive parentId linked-list walker is not the intended core reader model. However, I did not find the requested compaction-retry integrity guard: retry compaction entries are not superseded, deterministic, or reconciled, and tests still accept multiple retry-end events and preserve sibling branches.

## Best Possible Solution

Keep this open for maintainer product/API judgment. The best fix is a compaction-specific reconciliation contract, such as marking retry-superseded compaction entries or making retry compaction IDs deterministic, plus a regression test for duplicate same-parent compaction entries. If maintainers choose not to enforce that invariant, document clearly that downstream readers must use SessionManager/active-branch traversal and must not treat parentId as a globally linear list.
