## Summary

Current main has adjacent cleanup for terminal OpenClaw-owned ACP task sessions, but it still does not fix the direct `acpx codex sessions new` stale-record case reported here. The bundled plugin still consumes `acpx@0.6.1`, the OpenClaw wrapper delegates session lifecycle to `acpx/runtime`, and the pinned acpx direct CLI source still writes and displays stored `closed`/`pid` session metadata without reconciling `sessions show` against the actual process table.

## Best Possible Solution

Keep this open. The best path is to fix direct session lifecycle reconciliation in `openclaw/acpx`: make `sessions new/show/ensure/status` reconcile `closed`, `pid`, and last-exit fields against the actual queue owner/agent lifecycle, then release acpx, bump `extensions/acpx` to that version, and add focused regression coverage in this repo for the consumed behavior.
