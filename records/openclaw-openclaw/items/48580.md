## Summary

Keep open. Current main has adjacent OpenClaw-managed ACPX fixes for `sessions_spawn(runtime="acp")` and early `acpx exited ...` turn failures, but I did not find a source-backed fix for the specific direct `acpx codex sessions new` stale-session case where the stored record can remain `closed: false` with a dead `pid`. The issue comment’s session-state-drift framing still matches the remaining gap; related #50470 covers a narrower OpenClaw/Feishu turn path, not direct acpx CLI session metadata reconciliation.

## Best Possible Solution

Keep this open or move it to a canonical upstream `openclaw/acpx` bug only if maintainers prefer tracking dependency-owned behavior there. The concrete fix should make direct `acpx sessions new/show/ensure` reconcile session `closed`, `pid`, and last-exit fields against the actual agent lifecycle, then bump the bundled `acpx` pin and add targeted OpenClaw coverage proving the consumed behavior.
