## Summary

Current main still does not implement the issue's Phase 1 first-class session-chain contract. The code has partial direct-lineage support (`spawnedBy`, `parentSessionKey`, `spawnDepth`, `childSessions`), but no persisted `rootSessionKey`, no session-chain `traceId`, no gateway schema/list/event fields for those values, and no agent-facing `sessions_list` output for the full parent/root/trace/depth set. The related implementation PR #11043 was closed unmerged, and the prior Codex review comment on this issue already kept it open.

## Best Possible Solution

Keep this issue open until core implements the minimal backward-compatible Phase 1 chain metadata across `SessionEntry`, gateway protocol schemas/list rows/events, native and ACP spawn write paths, agent-facing `sessions_list` output, docs, and multi-hop regression tests. If maintainers reject first-class root/trace metadata as a core contract, close with that product-scope decision instead.

## Clownfish Work Prompt

Implement the minimal Phase 1 first-class session/task chain metadata for spawned sessions requested by this issue. Add optional `rootSessionKey?: string` and `traceId?: string` alongside the existing lineage fields, keep all fields backward-compatible, and do not change spawn permissions, nesting policy, delivery behavior, or add a new `sessions.chain` API/UI graph. Native `sessions_spawn` and ACP spawn should derive `parentSessionKey` from the requester, `rootSessionKey` from the parent root or parent key, a stable chain `traceId` from the parent or a new UUID, and `spawnDepth` from parent depth + 1. Expose `spawnedBy`, `parentSessionKey`, `rootSessionKey`, `traceId`, and `spawnDepth` through `SessionEntry`, gateway protocol schemas/list rows/events/snapshots, `sessions.patch` validation where appropriate, and agent-facing `sessions_list` output. Update docs for subagents/background tasks or protocol notes only where behavior/API changes are user-visible. Add focused multi-hop regression tests for native and ACP spawn/list/event/tool output. Do not revive unrelated changes from PR #11043, do not add broad observability/orchestration features, and do not move this into a ClawHub/plugin-only implementation.
