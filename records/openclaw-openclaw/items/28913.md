## Summary

Current main still lacks the requested source- or wrapper-aware autoCapture filter. The memory-lancedb config parser and manifest do not accept captureFilter.excludeWrapped, docs do not describe it, and the agent_end auto-capture path still stores user text solely through shouldCapture when broad trigger words match.

## Best Possible Solution

Keep this open and implement a backward-compatible memory-lancedb captureFilter.excludeWrapped option. The change should update the plugin-owned config parser, manifest schema/UI hints, docs, and agent_end auto-capture path, with focused tests proving EXTERNAL_UNTRUSTED_CONTENT-wrapped trigger text is skipped while normal direct preference capture still works. If marker detection needs core logic, add a small generic public SDK seam rather than deep-importing core security internals from the plugin.

## Risks / Open Questions

- Current autoCapture can still persist wrapped external text if it reaches a user message and matches broad trigger words such as always, never, or important without matching the existing prompt-injection regexes.
- A naive string check could drift from the randomized marker format or mishandle spoofed marker text; the fix should deliberately share or mirror the current marker semantics while respecting the extension/core boundary.
- Adding runtime filtering alone would leave config validation, manifest/UI metadata, docs, and tests out of sync.
- This narrow opt-in filter should not be treated as a full provenance or privilege-separation solution; #29442 and #48519 remain broader architecture discussions.
