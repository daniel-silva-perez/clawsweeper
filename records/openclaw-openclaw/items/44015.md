## Summary

Keep this PR open. Current main at 5828dcdb05aad587bff831467e7fa51f73312ff8 still does not implement a 512 KiB UTF-8 byte cap for `chat.send` messages, and the PR is a focused gateway hardening candidate rather than conservative cleanup. It should be revised before merge because the proposed schema character limit and post-NFC byte check do not exactly enforce the documented raw payload policy.

## Best Possible Solution

Keep this PR open as a useful gateway hardening candidate. Ask the contributor to define the intended raw-vs-normalized byte policy, enforce the 512 KiB cap in the actual `chat.send` server path, avoid or clearly document any schema character-count guard, update the protocol docs, and add focused ASCII boundary plus multi-byte/decomposed Unicode tests before running the relevant gateway/protocol changed checks.

## Risks / Open Questions

- Merging the current PR as-is could document a raw 512 KB request-body policy while enforcing byte length after NFC normalization, allowing some oversized raw decomposed Unicode inputs through the new check.
