## Summary

Keep open. Current main still lacks a 512 KiB `chat.send` message cap, but this PR should be revised before merge because the proposed byte check runs after NFC normalization and can accept oversized decomposed input. The schema change also needs clearer contract wording because its `maxLength` guard is character-based, not UTF-8 byte-based.

## What This Changes

The PR adds a 512 KiB UTF-8 `chat.send` message cap in the gateway sanitizer and TypeBox schema, adds sanitizer tests, and documents the gateway protocol limit.

## Best Possible Solution

Revise this PR so the 512 KiB policy is explicit and enforced in the actual `chat.send` path before expensive normalization when it is meant as a request-size guard. Keep the schema either clearly advisory/ASCII-aligned or replace it with a contract that cannot be mistaken for byte enforcement, then align docs and regression tests for ASCII, multi-byte, and decomposed-Unicode inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check raw message bytes before normalization:** `src/gateway/server-methods/chat.ts:238`
  - body: The new check measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `e\u0301` repeated enough times can be over 512 KiB on input, normalize below the cap, and still be accepted despite the documented request-body limit. Check the incoming string before normalization, or explicitly document and test normalized-message semantics.
  - confidence: 0.9
- **[P3] Clarify that the schema limit is character-based:** `src/gateway/protocol/schema/logs-chat.ts:35-36`
  - body: The added TypeBox `maxLength` is a JSON Schema string-length guard, not UTF-8 byte enforcement. Multi-byte messages can pass this layer well above 512 KiB and rely on the sanitizer, so the comment should say this is advisory/ASCII-aligned or the schema guard should be removed or changed.
  - confidence: 0.82
