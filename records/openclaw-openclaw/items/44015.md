## Summary

Keep open. Current main still lacks the requested `chat.send` message-size cap, and this PR is an active implementation candidate with concrete correctness/security review findings: the proposed byte check runs after NFC normalization and the schema guard is character-based. It needs revision and rebase rather than cleanup closure.

## What This Changes

The PR adds a 512 KiB UTF-8 byte limit for gateway `chat.send` messages, updates the TypeBox schema, adds sanitizer boundary tests, and documents the protocol limit.

## Best Possible Solution

Revise the PR so the 512 KiB policy is enforced against incoming UTF-8 message bytes before NFC normalization and other expensive processing. Keep the protocol schema clearly documented as advisory/character-based, or remove/rework that layer, then add regression coverage for ASCII, multi-byte, and decomposed-Unicode inputs and align the docs with the chosen semantics.

## Reproduction Assessment

Yes. Current main can be checked directly by reading the schema, sanitizer, and handler, and the PR bug reproduces with a focused Node byte check where decomposed Unicode is 600000 bytes before NFC and 400000 bytes after NFC.

## Solution Assessment

No. The requested direction is good, but this patch is not the best implementation because the actual size guard runs after normalization and the schema layer uses character length; raw-byte enforcement with clearer schema/docs is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check raw bytes before normalizing:** `src/gateway/server-methods/chat.ts:238`
  - body: The new limit measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it would be accepted despite the documented 512 KB request-body limit. Check the incoming string bytes before normalization, or explicitly change and test the policy as normalized-message size.
  - confidence: 0.9
- **[P3] Clarify the schema is not byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:35-36`
  - body: `maxLength` on the JSON Schema string is a character-count constraint, while the advertised limit is UTF-8 bytes. Multi-byte payloads can pass this schema well above 512 KiB and rely on the sanitizer, so the comment should state this is advisory/ASCII-aligned or the schema guard should be removed/reworked.
  - confidence: 0.82

## Security Review
