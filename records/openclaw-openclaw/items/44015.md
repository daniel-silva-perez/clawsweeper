## Summary

Keep this PR open. Current main still does not enforce the requested 512 KiB UTF-8 byte cap for `chat.send` messages, and the PR is a focused gateway hardening candidate with fixable review feedback rather than conservative cleanup.

## Best Possible Solution

Keep this PR open and ask for a small revision: define whether the 512 KiB policy applies to raw input bytes or normalized message bytes, enforce that policy in the actual `chat.send` path, avoid treating JSON Schema `maxLength` as byte enforcement, update the protocol docs, and add focused ASCII boundary plus multi-byte/decomposed Unicode tests before gateway/protocol validation.

## Risks / Open Questions

- The PR still needs revision before merge: the proposed schema `maxLength` is character-count based, while the stated policy is UTF-8 bytes, so it should be removed, narrowed, or clearly documented as advisory defense-in-depth.
- The proposed sanitizer measures bytes after NFC normalization, which may not match a documented raw request-body limit for decomposed Unicode inputs.
- Current main still admits `chat.send` message strings up to the broader gateway payload cap, so closing this PR now would discard a useful hardening candidate without a shipped replacement.
