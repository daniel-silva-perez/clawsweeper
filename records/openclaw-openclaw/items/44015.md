## Summary

Keep open. Current main still does not enforce the requested 512 KiB UTF-8 `chat.send` message cap, so the PR is useful, but the proposed patch is not merge-ready because it checks the byte limit after NFC normalization while documenting a request-body limit.

## What This Changes

The PR adds a 512 KiB `chat.send` message limit in gateway sanitization, adds a protocol schema `maxLength`, extends sanitizer tests, and documents the gateway protocol limit.

## Best Possible Solution

Revise this PR to define the byte policy explicitly, enforce the 512 KiB cap in the actual `chat.send` server path before expensive processing when the policy is raw payload size, make the schema guard clearly advisory or remove it, and keep protocol docs plus ASCII, multi-byte, and decomposed-Unicode regression tests aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure the cap before normalizing the message:** `src/gateway/server-methods/chat.ts:238`
  - body: The PR documents a 512 KB UTF-8 message body limit, but the new check measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload like `e\u0301` repeated enough times can be over 512 KiB on input and normalize below the cap, so the request is accepted despite exceeding the documented body limit. Check raw `message` bytes before normalization, or change the docs/tests to state a normalized-message limit.
  - confidence: 0.9
- **[P3] Clarify the schema guard is not byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:41`
  - body: The added schema `maxLength` is a character-count JSON Schema guard, while the server limit is bytes. Multi-byte messages can pass this validation at sizes well above 512 KiB and rely entirely on the sanitizer, so the schema comment should make that explicit or the schema guard should be removed/changed to avoid misleading future callers.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: No dependency, CI, or supply-chain changes were found, but the DoS hardening is incomplete because the new cap is checked after Unicode normalization.
