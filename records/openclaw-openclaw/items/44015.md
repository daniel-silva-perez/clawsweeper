## Summary

Keep open. Current main still has no 512 KiB command-level cap for gateway `chat.send`, so the PR addresses a real hardening gap, but the proposed patch should not merge as-is because it measures bytes after NFC normalization and can accept oversized raw decomposed Unicode payloads while documenting a request-body limit. Prior review discussion also flagged the schema `maxLength` character/byte mismatch.

## What This Changes

The PR adds a 512 KiB UTF-8 byte cap for gateway `chat.send` messages in the sanitizer, adds a TypeBox schema length constraint, adds sanitizer tests, and documents the protocol limit.

## Best Possible Solution

Revise the PR so the 512 KiB policy is explicit and enforced against incoming UTF-8 bytes before expensive normalization when used as a request-size guard. Keep the schema either clearly advisory/ASCII-aligned or replace it with a contract that cannot be mistaken for byte enforcement, then align docs and regression tests for ASCII, multi-byte, and decomposed-Unicode inputs.

## Reproduction Assessment

Yes. The PR diff plus a focused Node byte check reproduce the issue: `"e\u0301".repeat(200000)` is 600000 bytes before NFC and 400000 bytes after NFC, so a post-normalization cap would accept an oversized raw payload.

## Solution Assessment

No. The requested direction is good, but this patch is not the best implementation because it checks normalized bytes and adds a character-count schema guard; raw-byte enforcement plus clearer schema/docs is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check raw message bytes before normalization:** `src/gateway/server-methods/chat.ts:238`
  - body: The new check measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `e\u0301` repeated enough times can be over 512 KiB on input, normalize below the cap, and still be accepted despite the documented request-body limit. Check the incoming string before normalization, or explicitly document and test normalized-message semantics.
  - confidence: 0.9
- **[P3] Clarify that the schema limit is character-based:** `src/gateway/protocol/schema/logs-chat.ts:35-36`
  - body: The added TypeBox `maxLength` is a JSON Schema string-length guard, not UTF-8 byte enforcement. Multi-byte messages can pass this layer well above 512 KiB and rely on the sanitizer, so the comment should say this is advisory/ASCII-aligned or the schema guard should be removed or changed.
  - confidence: 0.82
