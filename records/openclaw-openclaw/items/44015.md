## Summary

Keep open. Current main still does not enforce a 512 KiB `chat.send` message cap, so the PR is not obsolete, but the proposed patch should not merge as-is because it measures the new byte limit after NFC normalization and presents the schema `maxLength` as a size guard even though actual byte enforcement remains in the sanitizer.

## What This Changes

The PR adds a 512 KiB UTF-8 byte cap for gateway `chat.send` messages, adds a protocol schema `maxLength`, extends sanitizer tests, and documents the gateway protocol limit.

## Best Possible Solution

Revise this PR to define the byte policy explicitly, enforce the 512 KiB cap in the actual `chat.send` path before expensive normalization when the policy is request-body size, make the schema guard clearly advisory or replace it with a contract that cannot be mistaken for byte enforcement, and align docs plus ASCII, multi-byte, and decomposed-Unicode regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure the cap before normalizing the message:** `src/gateway/server-methods/chat.ts:238`
  - body: The new check measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `e\u0301` repeated enough times can be over 512 KiB on input, normalize below the cap, and still be accepted despite the documented message-body limit. Check the raw message bytes first, or explicitly document and test a normalized-message limit.
  - confidence: 0.9
- **[P3] Clarify that the schema limit is character-based:** `src/gateway/protocol/schema/logs-chat.ts:35-36`
  - body: The added TypeBox `maxLength` is a JSON Schema string-length guard, not UTF-8 byte enforcement. Multi-byte messages can pass this layer well above 512 KiB and rely on the sanitizer, so the comment should say the schema guard is advisory/ASCII-aligned or the schema check should be removed or changed.
  - confidence: 0.82
