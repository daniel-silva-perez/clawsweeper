## Summary

Keep open. Current main still has no `chat.send` message byte cap, so the PR addresses a real gap, but the proposed patch has an unresolved correctness issue: it measures bytes after NFC normalization, allowing oversized decomposed Unicode input to shrink under the limit and be accepted. The schema `maxLength` addition is also only character-based, so it should be clarified or reworked before merge.

## What This Changes

The PR adds a documented 512 KiB `chat.send` message limit by changing the gateway sanitizer, TypeBox protocol schema, focused sanitizer tests, and gateway protocol docs.

## Best Possible Solution

Revise the PR so the 512 KiB policy is enforced against the incoming `chat.send` string bytes before NFC normalization and other processing. Keep the schema either clearly documented as a character-count advisory layer or remove/rework it, then add regression coverage for ASCII boundary, multi-byte characters, and decomposed Unicode that shrinks under NFC.

## Reproduction Assessment

Yes. Current main can be verified by reading the schema, sanitizer, and handler, and the PR bug reproduces with a focused byte check where decomposed Unicode is 600000 bytes before NFC but 400000 bytes after NFC.

## Solution Assessment

No. The requested direction is sound, but this patch is not the best implementation because the actual DoS guard runs after normalization and the schema layer is character-count based; raw-byte enforcement with clearer schema/docs is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Check raw bytes before normalizing:** `src/gateway/server-methods/chat.ts:238`
  - body: The new limit measures `Buffer.byteLength(normalized, "utf8")` after NFC normalization. A decomposed payload such as `"e\u0301".repeat(200000)` is about 600 KB on input but about 400 KB after NFC, so it would be accepted despite the documented 512 KB request-body limit. Check the incoming string bytes before normalization, or explicitly change and test the policy as normalized-message size.
  - confidence: 0.9
- **[P3] Clarify the schema is not byte enforcement:** `src/gateway/protocol/schema/logs-chat.ts:35-36`
  - body: `maxLength` on the JSON Schema string is a character-count constraint, while the advertised limit is UTF-8 bytes. Multi-byte payloads can pass this schema well above 512 KiB and rely on the sanitizer, so the comment should state this is advisory/ASCII-aligned or the schema guard should be removed/reworked.
  - confidence: 0.82
