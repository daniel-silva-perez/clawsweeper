## Summary

Keep open. Current main has important related fixes for duplicate tool-call IDs and WhatsApp group session isolation, but I did not find targeted proof that the specific WhatsApp group + Moonshot/Kimi `exec:26` duplicate path is fixed. The current Moonshot provider also intentionally opts out of the generic tool-call ID sanitizer, so the related generic mitigation is not enough by itself to close this report.

## Best Possible Solution

Add or run a targeted regression for a WhatsApp group session using the Moonshot provider with replay history containing duplicate raw `exec:26` tool-call IDs. If current Kimi K2.5 now emits only native `functions.<name>:<index>` IDs and replay passes, close this as implemented with the 2026.3.28 duplicate-ID mitigation plus the shipped Moonshot replay-policy evidence; otherwise add a Moonshot-safe duplicate-ID repair that preserves native Kimi IDs while rejecting or rewriting non-native repeated IDs.
