## Summary

Keep open. Current main has the pieces needed for a narrow fix, including a Kimi-aware strict sanitizer, but the bundled Moonshot plugin still opts out of tool-call ID sanitization, so duplicated raw IDs like `exec:26` can still bypass both replay-history and live-stream repair paths.

## What This Changes

The issue reports Moonshot/Kimi returning HTTP 400 for duplicated `exec:26` tool-call IDs in WhatsApp group chats, while the same setup works in direct messages and with other providers.

## Best Possible Solution

Add a focused Moonshot replay-policy regression that includes duplicate raw `exec:26` assistant/tool-result history and native `functions.<name>:<index>` IDs, then enable a Moonshot-safe sanitizer path so raw duplicates are rewritten while native Kimi IDs remain intact.

Yes for the source-level failure path: resolve the Moonshot OpenAI-compatible replay policy and pass duplicate raw `exec:26` history through replay sanitization; current main skips the ID sanitizer because the policy omits the sanitize flags. ## Solution Assessment

Yes, the narrowest maintainable direction is a Moonshot-safe replay sanitizer or policy activation now that strict sanitization preserves native Kimi IDs.
