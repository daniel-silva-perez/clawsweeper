## Summary

Keep open. Current main has related duplicate tool-call ID mitigations and WhatsApp group-session isolation, but the specific WhatsApp group + Moonshot/Kimi raw `exec:26` duplicate path is not proven fixed. Moonshot intentionally opts out of the generic replay ID sanitizer, so the generic OpenAI-compatible fix is not enough evidence to close this report.

## Best Possible Solution

Add a focused regression for a WhatsApp group session using the Moonshot provider with replay history containing duplicated raw `exec:26` assistant tool-call IDs. If current Kimi now emits only native `functions.<name>:<index>` IDs and the replay passes, close with that proof; otherwise add a Moonshot-safe repair that preserves valid native Kimi IDs while rewriting or dropping repeated non-native raw IDs before replay.

## Evidence

That means generic duplicate-ID rewriting does not directly cover raw repeated IDs such as `exec:26` on Moonshot.
