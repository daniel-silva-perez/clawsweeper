## Summary

Keep open. Current main has strong adjacent machinery for routed restart notices, one-shot restart continuations, durable session-delivery recovery, and Task Flow state, but it still does not implement the RFC’s first-class task checkpoint lifecycle with paused/resuming/resumed/failed resume states, plan cursor persistence, original/new run linkage, and automatic continuation for restart-required work.

## What This Changes

The issue requests a first-class checkpoint and auto-resume mechanism so in-progress tasks can continue across gateway restarts as a new follow-up run without another user prompt.

## Best Possible Solution

Keep this issue open as the design anchor for restart-owned task checkpoint semantics. The maintainable direction is to reuse restart sentinel/session-delivery infrastructure for wakeup, Task Flow for durable step progress where applicable, and the task registry for observable lifecycle, then add a deliberate checkpoint contract only where restart boundaries can be made idempotent and secret-safe.

## Reproduction Assessment

Not applicable as a feature RFC, but the current-main verification path is high-confidence: the relevant source and docs can be inspected statically, and they do not contain the requested checkpoint statuses or schema.

## Solution Assessment

No, current main is not the complete solution to the RFC. The one-shot continuation path is valuable partial progress, but the best fix needs maintainer design review before adding cross-cutting checkpoint state and resume policy.
