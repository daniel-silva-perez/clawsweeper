## Summary

Keep open. Current main has substantial adjacent restart recovery: routed restart sentinels, one-shot gateway.restart continuations, durable handoff through the session-delivery queue, main-session restart recovery from transcript locks, and Task Flow durable state. But the RFC asks for a first-class task checkpoint/resume model with statuses such as paused_for_restart/resuming/resumed/failed_resume, plan cursor persistence, and automatic continuation for restart-required work. Current task and flow status models do not include those states, and config/update restart sentinels currently carry notification metadata rather than a resumable task checkpoint. The related merged PR #63406 should be treated as useful partial progress, not as closing this broader RFC.

## Best Possible Solution

Keep this issue open as the canonical RFC for first-class task checkpoint and auto-resume semantics. The best path is to reuse the shipped restart sentinel/session-delivery queue for routed wakeup, Task Flow for durable step state where applicable, and the task registry for observable lifecycle, then add a deliberate checkpoint contract only for restart-owned task boundaries with explicit idempotency, failure states, and secret-safe persisted context.

## Risks / Open Questions

- Closing this now would conflate one-shot restart continuation with the broader checkpoint/resume lifecycle the RFC asks for.
- The remaining design overlaps existing restart sentinel, session recovery, task registry, and Task Flow primitives, so implementation should avoid adding a parallel orchestration system without maintainer product direction.
