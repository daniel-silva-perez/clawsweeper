## Summary

Keep open. Current main still does not implement the Prompt Inspector detector, and the open PR remains a security-sensitive feature proposal rather than cleanup-close material. The branch also needs maintainer security/product review and a rebase onto the current hook handler split before it can be evaluated for merge.

## What This Changes

The PR adds an opt-in Prompt Inspector HTTP client with PMTINSP environment controls, webhook pre-dispatch checks, async external-content scanning, and fire-and-forget scanning for web tool results.

## Best Possible Solution

Keep the security hardening discussion open. The best end state is either a reviewed core detector integration with explicit privacy/docs/config/tests and current hook-handler wiring, or a generic inspection seam that lets Prompt Inspector live as a plugin or ClawHub package without baking one vendor into core.

## Security Review

Status: needs_attention

Summary: The diff introduces a new outbound prompt-content/API-key path and needs maintainer security review before merge.

Concerns:

- **[medium] Plain HTTP detector endpoints can leak inspected content:** `src/security/pi-client.ts:132`
  - body: The client chooses the Node http transport for any non-HTTPS PMTINSP_BASE_URL and still sends sampled content plus X-App-Key. That is a concrete leakage risk for remote or misconfigured endpoints unless plaintext is limited to loopback or separately opted into.
  - confidence: 0.78
- **[medium] Third-party content inspection needs explicit policy review:** `src/security/pi-client.ts:128`
  - body: The feature posts webhook messages, external hook content, and web tool output to Prompt Inspector or a custom endpoint. Even with opt-in configuration, this changes the privacy boundary for sensitive user and tool content and should land only with explicit docs/config/redaction/retention review.
