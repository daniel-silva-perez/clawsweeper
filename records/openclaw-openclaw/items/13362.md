## Summary

Keep open. Current main still does not expose or propagate a `doctor --dry-run` mode, and the existing doctor flow can still prompt, persist config, install runtime dependencies, rewrite cron state, or repair services through paths that have no dry-run guard. The related global tool dry-run request and the open `--check-config` PR cover different scopes, so they do not supersede this issue.

## What This Changes

The issue requests a native `openclaw doctor --dry-run` mode that validates and previews doctor repair changes without writing to disk.

## Best Possible Solution

Add a focused native doctor dry-run contract: register and type the flag, thread it through the prompter/config/health contribution flows, collect and render the same repair plan without prompts or writes, return a defined non-zero result when issues are found, and document the command on https://docs.openclaw.ai/cli/doctor while keeping #41418 and #47302 separate.
