## Summary

Keep open. Current main still lacks a first-class `openclaw doctor --dry-run` contract: the CLI does not register the flag, doctor options cannot carry it, docs do not describe it, and doctor still has multiple mutation paths that would need explicit dry-run handling. The adjacent `--check-config` PR and global dry-run issue cover different scopes.

## What This Changes

Add a `--dry-run` flag to `openclaw doctor` so users can preview validation and repair recommendations without writing config or applying other doctor side effects.

## Best Possible Solution

Add a native doctor dry-run mode that is registered in the CLI, typed in `DoctorOptions`, threaded through repair mode, prompting, config flow, and health contributions, renders the same repair plan without prompts or mutations, returns a defined non-zero status when fixable issues are found, and updates the public doctor docs and changelog.

## Reproduction Assessment

Not applicable as a feature request, but the missing behavior is directly verifiable on current main: `openclaw doctor --dry-run` is not registered, cannot be represented by `DoctorOptions`, and is absent from the docs.

## Solution Assessment

Yes. A doctor-specific dry-run is the narrowest maintainable solution because it can cover doctor’s own config, cron, runtime-dependency, and service repair side effects without conflating this with the global agent dry-run or `--check-config` validation work.
