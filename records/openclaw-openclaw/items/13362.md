## Summary

Keep open. Current main still lacks a first-class doctor dry-run contract: the CLI does not register the flag, the doctor option type cannot carry it, docs do not describe it, and several doctor repair paths still mutate config, cron state, runtime dependency staging, or services without a dry-run guard. The related global dry-run issue and the open check-config PR cover different scopes, so they do not supersede this issue.

## What This Changes

Request to add `openclaw doctor --dry-run` so doctor validation and repair recommendations can be previewed without prompts or writes.

## Best Possible Solution

Add a native doctor dry-run mode that is registered in the CLI, typed in `DoctorOptions`, threaded through repair mode, prompting, config flow, and health contributions, renders the same repair plan without prompts or mutations, returns a defined non-zero status when fixable issues are found, and updates the public doctor docs and changelog.

## Reproduction Assessment

Not applicable as a feature request, but the missing behavior is directly verifiable on current main: `openclaw doctor --dry-run` is not registered, cannot be represented by `DoctorOptions`, and is absent from the doctor docs.

## Solution Assessment

Yes. A doctor-specific dry-run is the narrowest maintainable solution because doctor owns config repair, cron repair, runtime dependency staging, and service repair side effects that the global dry-run issue and `--check-config` PR do not cover.
