## Summary

Keep open. Current main still lacks a native `openclaw doctor --dry-run` contract: the flag is not registered, cannot be represented in `DoctorOptions`, is absent from the public doctor docs, and doctor still has multiple mutation paths that would need explicit dry-run handling. The related global dry-run issue and the open `--check-config` PR cover different layers and do not supersede this request.

## What This Changes

Requested `openclaw doctor --dry-run` mode to preview validation and repair changes without writing config or causing other doctor side effects.

## Best Possible Solution

Add a focused doctor-specific dry-run mode that is registered in the CLI, typed in doctor options, carried through repair mode and prompter behavior, and honored by config finalization, health contributions, bundled plugin dependency repair, cron repair, and gateway service helpers. It should render the same repair plan without prompts or mutations, return a defined non-zero result when fixable issues are found, and update the public doctor docs and changelog.

## Reproduction Assessment

Not applicable as a runtime bug reproduction, but the missing feature is directly verifiable on current main: `doctor --dry-run` is not registered, not typed in `DoctorOptions`, and not documented. The existing source also shows mutation paths that would run without any dry-run guard.

## Solution Assessment

Yes. A doctor-specific dry-run is the narrowest maintainable direction because doctor owns config repair, runtime dependency staging, cron repair, and gateway service repair semantics that neither #41418 nor #47302 covers.
