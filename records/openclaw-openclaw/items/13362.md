## Summary

Keep open. Current main still lacks a native `doctor --dry-run` contract: the CLI flag is not registered, the doctor option type cannot carry dry-run state, docs do not describe it, and existing doctor repair paths can still write config, stage plugin runtime dependencies, save cron state, or repair gateway services. The related global dry-run request and the open `--check-config` PR cover different scopes, so they do not supersede this issue.

## What This Changes

Request to add `openclaw doctor --dry-run` so doctor validation and repair recommendations can be previewed without prompts or writes.

## Best Possible Solution

Add a doctor-specific dry-run mode that is registered in the CLI, typed in `DoctorOptions`, threaded through repair mode, prompting, config flow, health contributions, and service/dependency/cron repair helpers, renders the same repair plan without prompts or mutations, returns a defined non-zero status when fixable issues are found, and updates the public doctor docs plus changelog.

## Reproduction Assessment

Not applicable as a feature request, but the missing behavior is directly verifiable on current main: `openclaw doctor --dry-run` is not registered, cannot be represented by `DoctorOptions`, and is absent from the doctor docs.

## Solution Assessment

Yes. A doctor-specific dry-run is the narrowest maintainable direction because doctor owns config repair, cron repair, runtime dependency staging, and gateway service side effects that neither the global dry-run issue nor the `--check-config` PR covers.
