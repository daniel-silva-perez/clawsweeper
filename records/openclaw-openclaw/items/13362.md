## Summary

Keep open. Current main still lacks a native `doctor --dry-run` contract: the doctor command does not register the flag, `DoctorOptions` cannot carry it, docs do not describe it, and doctor repair paths still include config writes plus dependency, cron, and gateway service side effects. The related global dry-run request and the open `--check-config` PR cover different scopes and do not supersede this work.

## What This Changes

Request to add `openclaw doctor --dry-run` so doctor validation and repair recommendations can be previewed without prompts, config writes, or other repair side effects.

## Best Possible Solution

Add a doctor-specific dry-run mode that is registered in the CLI, typed in `DoctorOptions`, threaded through repair mode, prompting, config flow, final config writing, health contributions, bundled plugin dependency repair, cron repair, and gateway service helpers. It should render the same repair plan without prompts or mutations, return a defined non-zero status when fixable issues are found, and update the public doctor docs and changelog.

## Reproduction Assessment

Not applicable as a feature request, but the missing behavior is directly verifiable on current main: `openclaw doctor --dry-run` is not registered, cannot be represented by `DoctorOptions`, and is absent from the doctor docs.

## Solution Assessment

Yes. A doctor-specific dry-run is the narrowest maintainable direction because doctor owns config repair, cron repair, runtime dependency staging, and gateway service side effects that neither the global dry-run issue nor the `--check-config` PR covers.
