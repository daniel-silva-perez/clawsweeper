## Summary

Keep this PR open. Current main still does not surface Telegram `Sticker.fileId` in the agent-visible inbound conversation-info prompt, so the PR is not obsolete; however, the proposed `sticker_file_id` key still needs maintainer review because Telegram sticker sending currently accepts `fileId` or `stickerId`, not `sticker_file_id`.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a narrow fix that surfaces a replayable Telegram sticker handle in per-turn conversation info while aligning the exposed key with the accepted send contract, either by exposing `fileId`/`stickerId` in the prompt context or by adding a documented `sticker_file_id` alias to the shared schema and Telegram runtime. Keep focused regression coverage for stable-system-metadata exclusion, direct webchat omission, and Telegram inclusion.

## Risks / Open Questions

- As written, the PR may not reliably enable sticker echo because a model may copy `sticker_file_id` into the send call while Telegram accepts `fileId` or `stickerId`.
- Exposing Telegram sticker file ids to the model is a data-scope expansion; it should stay in untrusted per-turn conversation info and out of stable system metadata.
