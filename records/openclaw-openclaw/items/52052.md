## Summary

Keep this PR open. Current main still captures Telegram sticker `fileId` on the inbound context, but it does not project a replayable sticker handle into `buildInboundUserContextPrefix()` conversation info. The unresolved review concern also still applies: the proposed `sticker_file_id` prompt key does not match the current Telegram send tool inputs, which accept `fileId` or `stickerId`.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a narrow fix that surfaces the replayable Telegram sticker handle in per-turn conversation info while aligning the exposed key with the send tool contract, either by using `fileId`/`stickerId` in the prompt context or by adding a documented `sticker_file_id` alias to both the shared schema and Telegram runtime. Keep regression coverage for system-metadata exclusion, webchat omission, and Telegram inclusion.
