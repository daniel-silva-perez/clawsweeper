## Summary

Keep this PR open. Current main still does not surface Telegram `Sticker.fileId` into the agent-visible inbound conversation-info prompt, so the PR is not obsolete. The remaining maintainer question is the contract shape: the PR exposes `sticker_file_id`, while the Telegram send path and shared message-tool schema currently accept `fileId` or `stickerId`, which could make sticker echo unreliable unless those names are aligned or an alias is added.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a narrow fix that surfaces a replayable Telegram sticker handle in per-turn conversation info while aligning the exposed key with the accepted send contract, either by using `fileId`/`stickerId` in the prompt context or by adding a documented `sticker_file_id` alias to the shared schema and Telegram runtime. Keep focused regression coverage for stable-system-metadata exclusion, direct webchat omission, and Telegram inclusion.

## Risks / Open Questions

- As written, a model may copy `sticker_file_id` into a Telegram send call, but current main accepts `fileId` or `stickerId`, so the intended sticker replay workflow may still fail unless the prompt key and send contract are aligned.
- Exposing Telegram sticker file ids to the model is a data-scope expansion; it should remain in untrusted per-turn conversation info and stay out of stable system metadata to avoid cache churn and misplaced trust.
