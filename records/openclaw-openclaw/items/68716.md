## Summary

Keep open. The linked Discord authorization bug is still present on current main, but this PR should not merge as-is because the proposed runtime-level fail-closed check also blocks trusted owner CLI guild-admin actions that do not carry a Discord requester sender id.

## What This Changes

The PR adds trusted-requester gating and Discord permission checks for guild-admin message actions, forwards requester sender identity into guild-admin runtime calls, and adds authorization regression coverage.

## Best Possible Solution

Keep the Discord guild-admin authorization fix, but model the trust boundary more narrowly: require a server-injected requester id for tool-driven Discord-origin actions, check that requester's Discord permissions before bot-credentialed guild mutations, and preserve trusted owner CLI/manual admin paths through an explicit owner/trusted-execution signal or equivalent scoped bypass. Add regression coverage for unauthorized Discord requester rejection and for owner CLI/direct runtime guild-admin actions continuing to work.

## Reproduction Assessment

Yes. The PR regression has a high-confidence source reproduction path: `openclaw message` calls `messageCommand`, which sets `senderIsOwner` but does not pass `requesterSenderId`; the PR then requires `senderUserId` in `handleDiscordGuildAction` before role/channel/event/emoji/sticker mutations.

## Solution Assessment

No. The requested direction is right, but this implementation puts the fail-closed check at a runtime layer that cannot distinguish untrusted Discord requesters from trusted local owner CLI calls; the safer fix is to carry trusted execution context or limit mandatory sender checks to tool/requester-driven paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Allow trusted CLI guild-admin actions without sender IDs:** `extensions/discord/src/actions/runtime.guild.ts:234`
  - body: `openclaw message` defaults CLI calls to `senderIsOwner` and does not pass `requesterSenderId`, so owner commands like `role-add`, `emoji-upload`, `sticker-upload`, `event-create`, and `channel-delete` reach this runtime without `senderUserId`. With the new unconditional `verifySenderGuildAdminPermission` call, those trusted CLI/admin flows now throw `Sender identity required...` before executing. Limit the mandatory sender check to tool/requester-driven contexts or pass an explicit trusted-owner execution signal through the runtime.
  - confidence: 0.91
