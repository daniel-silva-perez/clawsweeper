## Summary

Keep open. The PR addresses a real config UX gap, but the current patch is not correct against current main: the warning is attached to an obsolete direct config-write path, and the branch also includes an unrelated type-incompatible skills eligibility change.

## What This Changes

The PR adds a non-blocking `openclaw config set` warning for selected agent model config paths by checking string model refs against the model catalog, adds a path-detection test, and also changes skill eligibility to exclude parse-error entries.

## Best Possible Solution

Rework this as a narrow current-main config CLI change: detect model-ref assignments from `ConfigSetOperation` entries in `runConfigOperations` before config writes, warn non-blockingly for unknown catalog refs using existing model-ref/catalog helpers, cover single and batch assignments, and remove the unrelated skills eligibility edit from this PR.

## Reproduction Assessment

Yes for the PR defects: the static call path is clear from current main (`config set` -> `runConfigSet` -> `runConfigOperations` -> `replaceConfigFile`) and the PR diff hooks the warning into a stale direct-write path. I did not run the CLI because this was a read-only review.

## Solution Assessment

No. The user-facing warning direction is good, but the current implementation is not the best fix because it misses the current operation-based config-set flow and mixes in an unrelated skills-status change; the safer alternative is an operation-level warning with focused config CLI tests.
