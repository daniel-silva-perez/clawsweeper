Keep open. Current main still lacks the requested `config set` model-catalog warning, and this PR is not merge-ready because its warning hook is based on a stale inline config-write flow and the branch also includes an unrelated, untyped skills eligibility change.

The branch adds a non-blocking model-catalog warning for selected `openclaw config set` model paths, a small path-detection test, and an unrelated skills-status eligibility change for parse-error skills.

## Best Possible Solution

Keep the warning as a narrow config CLI change: inspect `ConfigSetOperation` entries in the current `runConfigSet`/`runConfigOperations` path before writing, cover both single and batch assignments, use existing model-ref/catalog helpers for matching, keep the warning non-blocking, and remove the unrelated skills-status edit from this branch.

## Solution Assessment

No. The user-facing warning direction is reasonable, but this implementation is not the best solution because it misses the current config operation flow and mixes in unrelated skills behavior; the safer alternative is an operation-level config CLI warning with focused tests.
