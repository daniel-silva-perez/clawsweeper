## Summary

Keep open. Current main has not implemented the requested `config set` model-catalog warning, and this PR still needs repair because it is dirty against main, hooks a stale config-write path, and includes an unrelated untyped skills eligibility change.

## What This Changes

The branch adds a non-blocking model-catalog warning for selected `openclaw config set` model paths, adds a small path-detection test, and changes skill status eligibility for parse-error entries.

## Best Possible Solution

Keep the warning-only UX, but implement it against the current operation-based config mutation pipeline so both single and batch `config set` assignments are covered, reuse the existing model-ref/profile/catalog lookup helpers, and leave skills parse-error eligibility to a separate typed change.

## Reproduction Assessment

Yes. Static inspection reproduces the PR defects: current main routes `config set` through `runConfigSet` and `runConfigOperations`, while the PR hooks the old inline writer, and `SkillEntry` has no `parseError` field.

## Solution Assessment

No. The requested warning is reasonable, but this patch is not the best implementation because it misses the current write path and mixes an unrelated skills-status change into a CLI config fix.
