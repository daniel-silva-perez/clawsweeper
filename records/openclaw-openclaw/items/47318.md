## Summary

Keep open. The PR targets a useful config warning, but it is not merge-ready on current main because the warning is wired into an obsolete inline config writer and the branch also includes an unrelated, untyped skills-status change.

## What This Changes

The branch adds non-blocking model-catalog warnings for selected `openclaw config set` model paths, adds a path-detection test, and changes skill status eligibility to exclude parse-error entries.

## Best Possible Solution

Keep the model warning feature, but implement it as an operation-level `config set` check that inspects `ConfigSetOperation` values before persistence, covers single and batch assignments, reuses existing model-ref/profile/catalog lookup helpers, and leaves skills parse-error eligibility to a separate typed change.

## Reproduction Assessment

Yes. Static inspection reproduces the PR defects: current main routes `config set` through `runConfigSet` and `runConfigOperations`, while the PR hooks a stale inline writer, and the `entry.parseError` read has no matching `SkillEntry` type member.

## Solution Assessment

No. A warning-only UX is a reasonable direction, but this patch is not the narrowest maintainable implementation because it misses the active config operation flow and mixes an unrelated skills-status behavior change into the config PR.
