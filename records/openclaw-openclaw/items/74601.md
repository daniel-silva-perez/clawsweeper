## Summary

Keep open. Current main still has the workspace-only Skill Workshop boundary described in the report, and the related skill-visibility/allowlist issues do not cover workshop-driven shared-skill modification. This should not be auto-closed or queued as a narrow repair because it expands write authority into shared skills and proposes cross-workspace LLM review, which needs maintainer product and security direction first.

## What This Changes

The issue requests opt-in Skill Workshop support for proposing shared-skill updates, detecting repeated patterns across workspaces, and promoting workspace skills into shared skill candidates.

## Best Possible Solution

Stage this inside `extensions/skill-workshop` with an explicit target model: separate shared/workspace proposal metadata and storage, validated roots, manual-approval-only shared applies, docs, and focused tests. Treat cross-workspace semantic promotion as a later design step after privacy and evidence rules are settled.
