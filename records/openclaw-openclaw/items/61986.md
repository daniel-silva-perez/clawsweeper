## Summary

Keep open. Current main still turns every queued Android `notifications.changed` event into an immediate heartbeat wake, and the focused tests assert that behavior. The issue is also cross-referenced by an open fix-intent PR, so it should stay open until that PR is reviewed/retargeted or an equivalent upstream fix lands.

## What This Changes

The issue requests Android companion notification events be buffered as heartbeat context until the next scheduled heartbeat instead of triggering one agent run per notification event.

## Best Possible Solution

Keep the issue open and fix the gateway notification path so `notifications.changed` can enqueue bounded, untrusted context without causing immediate model execution. If maintainers still want some proactive behavior, add an explicit debounce or next-heartbeat mode with tests proving noisy Android notifications do not produce one run per event.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send `handleNodeEvent` a valid `notifications.changed` payload and current tests expect `requestHeartbeatNow` to run; the wake layer then schedules a near-immediate wake rather than waiting for the configured 30-minute interval.

## Solution Assessment

No. Current main is not the best solution, and the linked PR’s stated `scopedHeartbeatWakeOptions` approach is not sufficient by itself because that helper only scopes the wake target and still uses `requestHeartbeatNow`; the safer fix needs a true passive or next-heartbeat batching path.
