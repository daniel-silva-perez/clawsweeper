## Summary

Keep open. Current main still wakes the heartbeat immediately for each queued Android `notifications.changed` event, and the gateway tests explicitly assert that behavior. Android-side filtering, duplicate suppression, and per-minute forwarding limits mitigate noise but do not batch notification context to the configured heartbeat interval. The provided timeline also shows an open unmerged PR with closing syntax, so this issue should stay open until that fix candidate is reviewed, landed, or rejected.

## Best Possible Solution

Review KChow-ctrl/openclaw#2 as the active fix candidate, but require source-backed proof that notification events are queued for the next scheduled heartbeat rather than merely short-coalesced with `requestHeartbeatNow`. The likely gateway fix belongs in `src/gateway/server-node-events.ts` with regression coverage in `src/gateway/server-node-events.test.ts` showing `notifications.changed` enqueues context without triggering an immediate run, plus any docs update if the user-visible notification behavior changes.
