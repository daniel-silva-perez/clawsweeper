## Summary

Keep this PR open. Current main still has the reported Zalo bug: `callZaloApi` parses `response.json()` before checking HTTP status, so non-JSON 4xx/5xx bodies can still surface as JSON parse failures instead of typed Zalo API errors. The PR is focused on the right surface, but should get maintainer review for regression tests and the remaining security/reliability concerns around reading and surfacing remote error bodies.

## Best Possible Solution

Keep the PR open and ask for a small hardening/test pass: add focused tests in `extensions/zalo/src/api.test.ts` for non-2xx non-JSON responses, non-2xx Zalo JSON errors, and HTTP 408 versus application polling timeout; then either cap/sanitize remote error body handling or intentionally document why the current bounded message behavior is acceptable. If those are addressed, this is a plausible owner-module fix for the Zalo plugin surface.

## Risks / Open Questions

- Landing the PR as-is may still read an unbounded HTTP error body with `response.text()` before truncating it, which can amplify memory/logging risk on malicious or broken upstream responses.
- The final patch appears to put remote response text into the thrown error message for non-JSON HTTP errors; maintainer review should decide whether that text must be sanitized, capped before buffering, or moved out of normal log/UI paths.
- The PR has no accompanying tests in the provided diff, so the HTTP non-JSON path, structured Zalo JSON error preservation, and HTTP 408 disambiguation could regress.
