## Summary

Keep open. Current main still matches the reported failure shape: `modelRun`/LLM-only runs remove all callable tools, inherited allowlist sources are still collected, and the empty allowlist guard still returns the pre-provider “No callable tools remain” error when `disableTools` is true. The closed related PRs were not merged, so there is no implemented fix on main.

## What This Changes

The issue requests that one-shot gateway `modelRun` and other explicitly LLM-only runs ignore inherited tool allowlists instead of failing before provider execution when tools are disabled.

## Best Possible Solution

Treat explicit no-tool runs as no-tool provider calls before the empty explicit allowlist guard fires, covering `modelRun`, `promptMode: "none"`, and plugin-owned `disableTools` paths. Add focused regressions for global `tools.profile`/`tools.alsoAllow` with both gateway modelRun and llm-task, while leaving strict allowlist errors in place for tool-enabled runs.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction: call `buildEmptyExplicitToolAllowlistError` with a non-empty source list, no callable tools, and `disableTools: true`; the existing unit test expects the same error that the reporter observed.

## Solution Assessment

Yes. The narrow maintainable fix is to make the empty explicit allowlist guard skip explicitly no-tool runs, or to clear inherited allowlists before that guard for those runs, rather than weakening allowlist enforcement globally.
