## Summary

Keep open. Current main still matches the reported pre-provider failure shape: LLM-only runs disable tools, but inherited explicit allowlist sources are still collected and the guard still errors when no callable tools remain. Related PRs #73031 and #74139 are open implementation candidates, so this should stay open until a focused fix lands.

## What This Changes

The issue asks for LLM-only `modelRun` and `llm-task` executions to ignore inherited tool allowlist entries so no-tool provider calls do not fail before reaching the provider.

## Best Possible Solution

Land one focused fix that treats intentional no-tool runs as no-tool provider calls even when inherited allowlist sources exist, while keeping strict empty-allowlist validation for normal tool-enabled runs. Regression coverage should include both gateway `modelRun` and `llm-task` with global `tools.profile` or `tools.alsoAllow` entries.

## Reproduction Assessment

Yes. The issue includes concrete CLI and `/tools/invoke` reproductions, and current-main source inspection confirms the same pre-provider condition: tools are empty, inherited allowlist sources remain, and the guard still errors.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: skip or neutralize the empty explicit-allowlist guard only for intentional LLM-only runs, rather than weakening allowlist validation for ordinary tool-enabled agent runs.
