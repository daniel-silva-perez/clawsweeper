## Summary

Keep open. Current main still has the reported failure shape: raw `model run` and `llm-task` runs intentionally disable tools, but inherited explicit allowlist sources are still collected and the guard still turns a zero-callable-tool set into the pre-provider error. Related open PRs #73031 and #74139 appear to cover overlapping parts of the fix, but neither is merged into current main.

## What This Changes

The issue requests that LLM-only `modelRun` and `llm-task` executions ignore inherited tool allowlists so non-empty global `tools.alsoAllow` settings do not abort no-tool model calls.

## Best Possible Solution

Land one focused fix that treats explicit no-tool runs as no-tool provider calls even when inherited allowlist sources exist, while keeping the empty explicit-allowlist error strict for normal tool-enabled runs. The regression coverage should include both gateway `modelRun` and `llm-task` with global `tools.profile` or `tools.alsoAllow` entries.

## Reproduction Assessment

Yes. The issue has concrete CLI and `/tools/invoke` reproductions, and current-main source inspection shows the same pre-provider path: tools are disabled, inherited allowlist sources remain, and the guard still errors on an empty callable set.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: skip or neutralize the empty explicit-allowlist guard only for intentional LLM-only runs, rather than weakening allowlist validation for ordinary tool-enabled agent runs.
