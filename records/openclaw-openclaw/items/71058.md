## Summary

Keep open. Current main still models Microsoft Teams as one top-level bot account, while the issue and follow-up discussion describe a real multi-agent identity gap that is not implemented or superseded by the related tracker work.

## What This Changes

Request to support multiple Azure/Bot Framework Microsoft Teams bot identities on one OpenClaw Gateway, with routing to distinct agents instead of one shared `channels.msteams` bot config.

## Best Possible Solution

Implement this inside the bundled Microsoft Teams plugin using the existing `channels.<channel>.accounts` plus `defaultAccount` pattern, preserve today’s top-level Teams config as the implicit default account, select and validate the account from Bot Framework identity/webhook context, carry `accountId` through routing/replies/pairing/status/setup, and update Teams plus multi-agent docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main inspection: Teams has one credential/config/runtime account, and the multi-agent docs omit Teams from the multi-account channel list.

## Solution Assessment

No for the exact proposed array plus `agentBinding` shape. The maintainable fix should reuse the standard account map and existing `bindings` routing pattern, with Teams-specific auth/account selection reviewed by maintainers.
