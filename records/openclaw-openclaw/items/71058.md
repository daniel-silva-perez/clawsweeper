## Summary

Keep open. Current main still implements Microsoft Teams as a single top-level bot account, and the issue discussion identifies a real multi-agent identity gap rather than an obsolete request. The closed related issue #72876 was explicitly closed as a duplicate of this issue, while #74163 is an open maintainer tracker, not an implementation that supersedes the requested behavior.

## What This Changes

Request support for multiple Microsoft Teams/Azure bot app identities on one OpenClaw Gateway, with per-agent routing while preserving the current single-object Teams configuration.

## Best Possible Solution

Implement this in the Microsoft Teams plugin using the existing channels.<channel>.accounts plus defaultAccount pattern: keep today’s top-level config as the implicit default, add named account config for additional Teams apps, select the account before routing from validated webhook identity, carry accountId through inbound routing and replies, and update setup/status/docs with plugin-level changes unless a generic SDK seam is proven necessary.

## Reproduction Assessment

Not applicable as an intermittent bug; this is a feature gap. The limitation is high-confidence from current-main source inspection: channels.msteams is strict top-level config, setup resolves only default, and runtime constructs one credential/JWT/webhook context.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The narrower maintainable direction is to reuse the existing accounts/defaultAccount and bindings contract already documented for multi-agent channel accounts.
