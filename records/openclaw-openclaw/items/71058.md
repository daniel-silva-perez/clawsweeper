## Summary

Keep open. Current main still models Microsoft Teams as one top-level bot credential/default account, and the related closed duplicate plus open Microsoft tracker do not implement the requested multi-bot identity support.

## What This Changes

Allow one OpenClaw Gateway to host multiple Microsoft Teams/Azure bot identities and route each bot account to a different agent.

## Best Possible Solution

Implement this in the Microsoft Teams plugin using the existing `channels.<channel>.accounts` plus `defaultAccount` pattern, keep today’s top-level config as the backward-compatible implicit default, carry `accountId` through inbound routing/outbound replies/status/setup/docs, and leave agent selection in the existing `bindings` system.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main source and docs: Teams exposes one configured/default account while other multi-account channels use `accounts` and `defaultAccount`.

## Solution Assessment

No for the exact proposed array plus `agentBinding` shape. The safer maintainable direction is the existing account-map/default-account contract plus a Teams-specific account-selection design reviewed for Bot Framework auth isolation.
