## Summary

Current main still does not support multiple Microsoft Teams bot identities/accounts on one gateway. The Teams channel remains modeled as a single top-level `channels.msteams` config with one credential set, setup forces the default account, runtime starts one SDK app/JWT validator from that one config, and inbound routing does not pass an `accountId`. Related #72876 was explicitly closed by a maintainer as a duplicate of this issue, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open and implement the feature in the bundled Microsoft Teams plugin: add a backward-compatible `channels.msteams.accounts` plus `defaultAccount` model, resolve credentials/policies per account, safely dispatch a shared Teams webhook to the validated bot account, pass `accountId` through inbound routing/session/reply paths, and update Teams plus multi-agent docs and focused tests.

## Risks / Open Questions

- A correct implementation is security-sensitive: the gateway must select the Teams account only after safe Bot Framework token validation, or validate against the correct audience without trusting an unauthenticated request body.
- Starting one webhook server per account could conflict on port/path; a shared endpoint needs a deliberate dispatch model keyed by validated bot identity or distinct configured paths.
- Backward compatibility must preserve existing top-level `channels.msteams` credentials and `MSTEAMS_*` environment variables as the implicit default account.
