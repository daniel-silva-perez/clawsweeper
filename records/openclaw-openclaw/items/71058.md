## Summary

Keep open. Current main still models Microsoft Teams as one top-level bot/app credential set and one default runtime account, while the issue discussion and related #72876 describe the same unsolved multi-agent identity gap. The open #74163 tracker is useful maintainer context but is not an implementation or superseding fix.

## What This Changes

Allow one OpenClaw gateway to configure and route multiple Microsoft Teams bot/app identities, with each bot account able to bind to a different agent.

## Best Possible Solution

Implement this inside the Microsoft Teams plugin using the existing `channels.<channel>.accounts` plus `defaultAccount` pattern, preserve today’s top-level Teams config as the implicit default account, select the account from verified Bot Framework identity/webhook context, and carry `accountId` through routing, replies, pairing, status, setup, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from static current-main inspection: Teams has a strict single-account config schema, default-only plugin adapter, default-only access path, and docs that omit Teams from the multi-account channel list.

## Solution Assessment

No for the exact proposed array plus `agentBinding` shape. The maintainable direction is to reuse the standard account map and existing `bindings` routing model, with maintainer-reviewed Teams-specific auth and account selection semantics.
