## Summary

Current main still does not support multiple Microsoft Teams bot identities on one gateway. The Teams config, setup, status/config adapter, runtime monitor, JWT validator, routing call, outbound send context, and docs are all still modeled around one top-level `channels.msteams` credential set and the default account. The discussion from dataoracle accurately identifies the gap: multi-agent bindings exist, but Teams does not expose a multi-account channel config surface that can bind different Teams bot apps to different agents.

## Best Possible Solution

Keep this issue open. Implement the feature in the bundled Microsoft Teams plugin using the existing multi-account channel contracts: add backward-compatible `channels.msteams.accounts` and `defaultAccount` support, resolve per-account credentials and policies, run one shared webhook that dispatches by validated bot identity, pass `accountId` into inbound routing/session/reply handling, scope outbound/conversation-store state by account, and update Teams plus multi-agent docs and tests.

## Risks / Open Questions

- A correct implementation needs a security-sensitive webhook design: account selection must happen only after validating the Bot Framework JWT audience/app identity so one bot cannot be routed through another bot's credentials or policy.
- Outbound sends and the Teams conversation store likely need account scoping; otherwise proactive replies could reuse a conversation reference created by a different Teams bot identity.
- The feature must preserve the current top-level `channels.msteams` shape and `MSTEAMS_*` environment variables as the default-account path.
