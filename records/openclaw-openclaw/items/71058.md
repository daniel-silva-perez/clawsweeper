## Summary

Keep open. Current main still exposes Microsoft Teams as one top-level bot account, while the issue discussion identifies a real multi-agent identity gap: Teams can route to agents, but cannot host multiple bot identities/accounts from one gateway. The related closed feature #72876 is duplicative, and open PR #74163 is a maintainer tracker rather than an implementation that supersedes this request.

## What This Changes

Request support for multiple Microsoft Teams/Azure bot configurations on one OpenClaw Gateway so each Teams bot identity can bind to a different agent.

## Best Possible Solution

Implement this in the bundled Microsoft Teams plugin using the existing account-scoped channel pattern: add channels.msteams.accounts plus defaultAccount, keep today’s top-level config as the implicit default account, select the correct account from validated Teams bot identity/webhook context, carry accountId through routing, replies, pairing, setup/status/docs, and add generic SDK seams only if the plugin cannot express the behavior through current contracts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main inspection: Teams exposes one credential/config/runtime account, and the multi-agent docs omit Teams from the multi-account channel list.

## Solution Assessment

No for the exact proposed array plus agentBinding shape; the maintainable solution should reuse the existing accounts/defaultAccount and bindings contract already used by other channels. Keeping this open for maintainer design review is the right path until that plugin-level implementation lands.
