## Summary

Keep open. Current main still implements Microsoft Teams as a single top-level bot account, and the discussion accurately identifies a real gap: multi-agent routing exists, but Teams cannot expose multiple bot identities from one gateway. The closed related feature #72876 appears duplicative, while open PR #74163 is a broad maintainer tracker rather than an implementation that supersedes this request.

## What This Changes

Add support for configuring multiple Microsoft Teams/Azure bot identities on one OpenClaw gateway and binding each bot identity to the appropriate agent.

## Best Possible Solution

Implement this in the bundled Microsoft Teams plugin using the existing multi-account channel pattern: add `channels.msteams.accounts` and `defaultAccount`, treat the current top-level config as the default account, dispatch a shared webhook to the correct account by validated bot identity, carry `accountId` through inbound routing, replies, pairing, policies, status, setup, docs, and tests, and add only generic core seams if the plugin cannot do this through current contracts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main inspection: Teams config/schema/runtime expose one top-level credential set and the multi-agent docs omit Teams from the multi-account list.

## Solution Assessment

No for the exact proposed array plus `agentBinding` shape; the maintainable path is to reuse the existing account map and binding contracts. Keeping this open is the right outcome until that plugin-level design and implementation land.

## Risks / Open Questions

- A correct implementation must select the Teams account only after validating the Bot Framework token against the intended bot app/audience; trusting only webhook path or activity fields could misroute messages across bot identities.
- The proposed array plus `agentBinding` shape differs from the repository's existing `channels.<channel>.accounts` and `bindings[].match.accountId` contracts, so the public config shape needs maintainer design confirmation.
- Backward compatibility for top-level `channels.msteams` credentials and `MSTEAMS_*` environment variables must be preserved as the implicit default account.
