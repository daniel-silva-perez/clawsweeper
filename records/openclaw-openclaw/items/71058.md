## Summary

Keep open. Current main still models Microsoft Teams as a single bot credential/runtime account, and the discussion describes a valid multi-agent identity gap that is not implemented or superseded by the closed duplicate or the open Microsoft tracker PR.

## What This Changes

Request support for multiple Microsoft Teams/Azure bot identities on one OpenClaw Gateway, with each bot/account routable to a specific agent.

## Best Possible Solution

Implement this inside the Microsoft Teams plugin using the existing channel account map pattern: keep today’s top-level config as the implicit default account, add accounts/defaultAccount support, select the correct account from verified Bot Framework identity and/or webhook context, carry accountId through inbound routing, outbound replies, status, setup, and docs, and keep agent routing in the existing bindings system.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main source and docs: Teams has one credential/config/runtime account, and the multi-agent docs omit Teams from the account-map channel list.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The maintainable path is to reuse channels.<channel>.accounts/defaultAccount and existing bindings, with a Teams-specific auth/account-selection design reviewed by maintainers.
