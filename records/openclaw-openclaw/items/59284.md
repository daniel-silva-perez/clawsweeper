## Summary

Keep open. Current main still matches the reported Slack ingress behavior: when `allowBots` resolves true, bot-authored room messages continue through normal routing, absent per-channel `users` allowlists are open by default, and the owner `allowFrom` match is only used for control-command gating rather than general message delivery. This is a security-sensitive Slack policy change, so it should stay open for maintainer/security review rather than be closed or auto-queued.

## What This Changes

Add a Slack `requireOwnerPresent` guard so `allowBots: true` in group channels cannot route bot-authored messages unless an authorized owner is present in the channel.

## Best Possible Solution

Keep the fix in the Slack plugin surface: add a documented, schema-backed owner-presence or authorized-human guard for bot-authored group-channel ingress, with cached membership probing and explicit fallback behavior approved by maintainers before changing defaults.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available with `prepareSlackMessage`: configure a Slack room with `allowBots: true`, no `users` allowlist, and a bot-authored message from a non-owner; the current path admits it after optional mention/channel policy gates.

## Solution Assessment

Unclear. The proposed `requireOwnerPresent` direction is plausible and correctly Slack-owned, but the safest default, API scopes, cache TTL, and behavior when membership lookup fails need maintainer/security review before implementation.
