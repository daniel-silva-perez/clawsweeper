## Summary

Keep open. Current main still lacks the requested Slack owner-presence guard: `allowBots=true` lets bot-authored Slack messages continue past the bot-message drop, room messages fall back to open sender authorization when no per-channel `users` allowlist is configured, and the owner allowlist is only wired into control-command gating rather than general message routing.

## Best Possible Solution

Implement this in the Slack plugin surface: add a typed and documented `requireOwnerPresent` option at the appropriate account/per-channel level, validate it in the Slack schema/config metadata, check channel membership with a short TTL cache before routing bot-authored or all guarded room messages, and add focused tests for owner present, owner absent, wildcard `allowFrom`, channel `users`, and control-command behavior. Mention gating can remain separate, but owner presence should become an explicit Slack authorization condition when enabled.

## Risks / Open Questions

- This is a security-relevant Slack bundled-plugin behavior, not a stale support request: closing it now would leave the requested owner-presence control untracked.
- A fix will need careful Slack API handling: owner IDs must be resolved from effective `allowFrom` without treating `*` as an owner, and membership checks should be cached/rate-limit aware.
