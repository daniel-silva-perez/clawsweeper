## Summary

Keep open. The PR targets a real reliability gap, but the proposed durable final-delivery state is cleared before the final payload is actually routed by the dispatcher, so a crash or send failure in that handoff window can still lose the completed response.

## What This Changes

The PR adds pending final-delivery state to main-session and subagent records, persists main-session final payloads, wires heartbeat/restart recovery for replay, updates subagent announce retry payload handling, adds a restart-recovery test, and changes heartbeat deferral behavior.

## Best Possible Solution

Revise the PR so pending final-delivery state is stored with enough routing context, replayed through the existing final dispatch path, and cleared only after a successful send or route result; add focused tests proving both first delivery and heartbeat replay preserve the marker when delivery fails.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: `runReplyAgent` and the heartbeat branch clear pending state before `dispatchReplyFromConfig` routes the returned final payload, so a failing `routeReply` or `dispatcher.sendFinalReply` after resolver return would drop the retry marker.

## Solution Assessment

No. The durable marker needs to survive until actual final dispatch success; clearing it inside the reply resolver is earlier than the existing delivery boundary and reopens the loss window the PR is meant to close.
