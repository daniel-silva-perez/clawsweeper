## Summary

Keep open. The PR targets a real durability gap and related prior work was closed unmerged, but the current patch still clears retry state before the actual final-delivery boundary, so it can lose the very marker it is trying to make durable if channel routing or sending fails.

## What This Changes

The PR adds pending-final-delivery persistence for main sessions and subagent records, restart/heartbeat replay handling, heartbeat deferral narrowing, and a main-session restart-recovery unit test.

## Best Possible Solution

Keep the durable pending marker through the real delivery boundary, clear it only after final dispatch or agent-command delivery succeeds, preserve retry metadata on route/send failure, add failure-path tests for first delivery and heartbeat replay, and add a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: the PR clears pending state in `runReplyAgent` and heartbeat replay before `dispatchReplyFromConfig` reaches `sendFinalPayload`, so a send failure after resolver return can lose the retry marker.

## Solution Assessment

No. The direction is good, but this patch clears durability state earlier than the existing delivery acknowledgment boundary; the safer implementation is to clear only after dispatch success and leave retry state intact on failure.
