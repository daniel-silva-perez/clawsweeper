## Summary

Keep open. The PR targets a real durable-delivery gap, but the current patch still clears the pending final-delivery marker before the final payload is actually routed or queued, so the same loss window can still occur after resolver return or heartbeat replay. It also lacks the required changelog entry for a user-visible reliability fix.

## What This Changes

The PR adds pending-final-delivery fields to session and subagent records, persists main-session final payloads, attempts heartbeat/restart recovery replay, narrows heartbeat deferral, and adds a main-session restart recovery unit test.

## Best Possible Solution

Keep the durable marker through the real delivery boundary: persist enough payload and route context before final dispatch, clear it only after `sendFinalPayload` or CLI delivery has actually succeeded, preserve retry metadata on route/send failure, and cover both first delivery and heartbeat replay failure paths with focused tests plus an Unreleased changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The static path is clear: the PR clears pending state in `runReplyAgent` and heartbeat replay before `dispatchReplyFromConfig` has routed or queued the returned final payload, so a later send failure loses the retry marker.

## Solution Assessment

Is this the best way to solve the issue? No. The patch moves durability state into useful places, but clearing it inside the resolver/replay path is earlier than the existing delivery boundary; the safer fix is to clear only after final dispatch success.
