## Summary

Keep this PR open. Current main still exposes and executes cron delivery as a single target, so the intended multi-channel cron fanout is not implemented on main. The PR remains relevant, but the submitted diff is still incomplete because it adds/resolves secondary targets without a complete dispatch, update-merge, trace, docs, CLI/UI, and test path.

## Best Possible Solution

Keep the PR open for a revised implementation against current main: add the `CronDeliveryTarget` / `delivery.additionalTargets` contract, normalize entries with the same helpers as the primary route, preserve and patch the field through cron updates, resolve and dispatch each secondary target independently after the primary delivery path, isolate optional-target failures, and add protocol, normalization, merge, dispatch, docs, CLI/UI, and trace coverage.

## Risks / Open Questions

- Merging the current PR shape would expose a public `delivery.additionalTargets` contract before the cron runner actually delivers to those targets.
- Without `mergeCronDelivery` support, cron updates could drop or ignore secondary targets once the field is accepted.
- Secondary delivery should be failure-isolated and auditable; one bad optional target should not fail an otherwise valid cron run.
- Multi-channel fanout sends the same output to more recipients, so validation, docs, and run traces need to make that privacy boundary visible.
