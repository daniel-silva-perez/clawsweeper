## Summary

Keep this PR open. Current main still has a single-target cron delivery contract and runtime path, so the requested multi-channel fanout is not implemented. The PR addresses a real gap, but the provided review context and current code show it still needs dispatch, patch-merge, UI/CLI/docs, tests, and failure-isolation work before maintainers can treat it as complete or obsolete.

## Best Possible Solution

Keep this PR open unless maintainers explicitly choose #44859 or another replacement as the canonical implementation. The complete path should add and document the additionalTargets contract across cron types, gateway schema, normalization, service patch merging, isolated-run dispatch, trace/status reporting, CLI/UI surfaces, and regression tests, with per-target failure isolation and a clear security review for recipient fanout.

## Risks / Open Questions

- Merging the current PR shape could expose a public delivery.additionalTargets contract before cron reliably dispatches to every configured recipient.
- Secondary target failures need isolation so one bad optional route does not fail or mask the primary cron delivery.
- Multi-channel fanout is privacy-sensitive because it sends the same generated output to more recipients; traces, docs, and UI/CLI affordances should make that recipient set visible.
- There is another open implementation candidate, #44859, so maintainers should choose a canonical PR before closing or landing duplicate work.
