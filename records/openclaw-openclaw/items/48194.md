## Summary

Keep this PR open. Current main still models cron delivery as a single target, and the PR remains a plausible implementation candidate but is not complete enough to close as implemented or obsolete.

## Best Possible Solution

Keep the PR open for a revised implementation against current main: add the cron `additionalTargets` contract, normalize and validate entries consistently, preserve and patch the field in job updates, resolve and dispatch secondary targets independently after primary delivery, isolate optional-target failures, record trace/status for each route, and add docs plus protocol/CLI/UI/runtime tests.

## Risks / Open Questions

- Merging the current PR shape could expose a public `delivery.additionalTargets` contract before cron reliably dispatches to every configured recipient.
- Secondary targets need failure isolation; one bad optional route should not fail or mask the primary cron delivery.
- Multi-channel fanout is privacy-sensitive because it sends the same generated output to more recipients, so traces, docs, and UI/CLI affordances should make the recipient set visible.
