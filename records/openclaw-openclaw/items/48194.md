## Summary

Keep this PR open. Current main still has a single-target cron delivery model, so the PR is not obsolete or implemented on main. The submitted PR is also not ready to close as already solved because the provided diff and review discussion show it only partially wires `additionalTargets`: secondary targets are resolved but not dispatched, update/merge preservation is missing, and there are no docs/tests for the new public cron contract.

## Best Possible Solution

Keep this PR open for a revised implementation against current main: add a public `CronDeliveryTarget`/`delivery.additionalTargets` contract, normalize entries with the same helpers as the primary delivery route, preserve the field through add/update/patch merging, resolve and dispatch each secondary target independently after the primary delivery path, avoid one optional target failing the whole run, and add focused protocol, normalization, merge, dispatch, docs, and CLI/UI coverage.

## Risks / Open Questions

- Closing this PR now would hide a still-missing cron fanout capability: current main neither accepts nor dispatches `delivery.additionalTargets`.
- The current PR head should not be merged as-is because resolving optional secondary targets without dispatch, merge preservation, docs, and tests can create silent no-op config or new run-failure paths.
- Fanout delivery has privacy and audit implications because one cron output is intentionally sent to multiple configured channels; the final implementation should make those routes explicit in config validation and run traces.
