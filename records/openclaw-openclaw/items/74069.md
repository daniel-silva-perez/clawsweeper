## Summary

Keep open. The report is a concrete npm-global downgrade failure, and current main still routes explicit npm downgrades through the same staged install plus exact-version verification path that can fail as `global install verify` while leaving the live install at the previous version. No relevant package-helper, verifier, or update-command changes landed after the report.

## What This Changes

The issue requests that `openclaw update --tag 2026.4.25` successfully downgrade an npm global install from 2026.4.26 instead of failing verification with the installed version still at 2026.4.26.

## Best Possible Solution

Fix the npm global downgrade path so an explicit older target installs into the staged prefix, verifies as that exact version, swaps into the live global root, and keeps post-update doctor/plugin/restart behavior intact. Add focused regression coverage for the 2026.4.26-to-2026.4.25 npm downgrade case and validate it against a real Linux npm global prefix before release.

## Reproduction Assessment

Yes for triage: the issue gives an exact command and output, and current main has a deterministic staged-verifier failure mode when the staged package version differs from the exact target. A live downgrade was not run in this read-only review.

## Solution Assessment

No, current main is not the best solution because it still lacks proof that the explicit older version reaches and survives the staged npm install/verify/swap flow. The narrow maintainable fix belongs in the package update helper and its regression tests, with installer review.
