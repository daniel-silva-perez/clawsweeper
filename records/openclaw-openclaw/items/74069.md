## Summary

Keep open. Current main has not changed the relevant npm global downgrade path since the prior ClawSweeper review: accepted downgrades still flow through staged npm install, exact-version verification, and swap-only-on-clean-verify behavior, and there is still no current-main proof that the 2026.4.26-to-2026.4.25 downgrade succeeds.

## What This Changes

The issue asks for `openclaw update --tag 2026.4.25` to successfully downgrade an npm global install from 2026.4.26 to 2026.4.25 instead of failing `global install verify` and remaining on 2026.4.26.

## Best Possible Solution

Fix the npm global downgrade path so an explicit older target installs into the staged prefix, verifies as that exact version, swaps into the live global root, and preserves post-update doctor, plugin sync, completion refresh, and restart behavior. Add a focused regression for the 2026.4.26-to-2026.4.25 npm downgrade plus Linux npm-global package proof before release.

## Reproduction Assessment

Yes for triage: the issue gives a concrete command and output, and current main contains the matching staged-verifier failure mode when the verified package version differs from the exact target. A live downgrade should still be reproduced in a disposable Linux npm global prefix before shipping a fix.

## Solution Assessment

No, current main is not the best solution because it lacks proof that an explicit older version reaches and survives the staged npm install, verification, and swap flow. The maintainable fix belongs in the package update helper and focused installer tests while preserving exact-version and inventory checks.
