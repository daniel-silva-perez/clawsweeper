## Summary

Current main still accepts the explicit npm downgrade and sends it through the staged global install plus exact-version verification path that can produce `global install verify` while leaving the live install unchanged; I did not find a merged fix or narrower canonical issue that supersedes this report.

## What This Changes

The issue asks for `openclaw update --tag 2026.4.25` to successfully downgrade an npm global install from 2026.4.26 instead of ending with `global install verify` and the old version still installed.

## Best Possible Solution

Fix the npm global downgrade path so an explicit older version installs into the staged prefix, verifies as that exact target, swaps into the live global root, and then preserves doctor, plugin sync, completion refresh, and restart behavior. Add a focused regression for the 2026.4.26-to-2026.4.25 case plus real Linux npm-global package proof.

## Reproduction Assessment

Yes for triage: the report gives an exact command and output, and current main has the matching staged verifier failure path. Full live reproduction still needs a Linux npm-global install because this review kept the checkout read-only.

## Solution Assessment

No, current main is not the best solution because it still lacks proof that the explicit older target reaches and survives the staged npm install, verify, and swap flow. The safer fix belongs in the package update helper with focused installer tests and maintainer supply-chain review.
