## Summary

Keep open. The report is a concrete npm global downgrade failure: current main recognizes exact-version downgrades and proceeds after confirmation, but the package update path still fails when the staged install does not contain the requested older version. The requested target exists on npm, and current tests cover downgrade confirmation plus verification failure, but not a successful recovery for this 2026.4.26-to-2026.4.25 downgrade path.

## Best Possible Solution

Keep this issue open and fix the npm global package downgrade path. The likely implementation is a narrow update-runner change plus regression tests that prove `openclaw update --tag 2026.4.25 --yes` from a 2026.4.26 npm install installs and verifies the exact older package, without weakening dist inventory verification or post-update safety checks. Because this touches package-manager execution and staging, maintainer review should choose the safest staging/argument strategy before automation writes the patch.
