## Summary

Keep open. Current main still routes the accepted npm downgrade through the staged global install and exact-version verification path that can fail as `global install verify` while preserving the old live install, and I did not find a merged fix or paired closing PR for this issue.

## What This Changes

The issue asks for `openclaw update --tag 2026.4.25` to successfully downgrade an npm global install from 2026.4.26 to 2026.4.25 instead of failing verification and staying on 2026.4.26.

## Best Possible Solution

Fix the npm global downgrade path so an explicit older target installs into the staged prefix, verifies as that exact version, swaps into the live global root, and preserves post-update doctor, plugin sync, and restart behavior. Add focused regression coverage for the 2026.4.26-to-2026.4.25 case plus real Linux npm-global proof before release.

## Reproduction Assessment

Yes for triage: the issue provides an exact command and output, and current main has the matching staged-verifier failure mode when the staged package version differs from the exact target. A live downgrade was not run because this review had to keep the checkout read-only.

## Solution Assessment

No, current main is not the best way to solve the issue because it still lacks proof that the explicit older version reaches and survives the staged npm install, verify, and swap flow. The fix belongs in the package update helper and focused installer tests, with maintainer installer review.
