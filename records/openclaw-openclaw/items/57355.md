## Summary

Keep open. Current main still lacks a native Plugin SDK `registerMcpServer` surface, and existing MCP support is limited to saved config and bundle-format MCP loading rather than native plugin lifecycle registration. The linked open implementation PR is the right place for maintainer review instead of stale cleanup.

## What This Changes

Request adding a native Plugin SDK `registerMcpServer` API so plugins can register managed stdio MCP servers that OpenClaw wires into embedded OpenAI/Pi-backed runs with lifecycle and settings management.

## Best Possible Solution

Keep the issue open and review the linked core implementation direction: a generic, backward-compatible Plugin SDK seam should register validated plugin-owned MCP servers, persist owner metadata, merge them deterministically into embedded runtime config below user/operator config, and include docs plus focused SDK/loader/runtime tests.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence current-main check is source/API inspection: `registerMcpServer` has no hits, `OpenClawPluginApi` does not expose it, and MCP loading currently comes from bundle/config paths.

## Solution Assessment

Unclear until the open implementation is reviewed. The requested core SDK seam appears to be the right layer for dynamic plugin-owned MCP lifecycle integration, but the exact process-execution and precedence model should be settled in PR #62160 or a narrower successor.
