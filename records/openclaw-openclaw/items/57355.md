## Summary

Keep open. Current main still lacks a native Plugin SDK `registerMcpServer` surface, and the only existing MCP wiring is through bundle metadata plus user/operator `mcp.servers` config. The related open PR #62160 is the active implementation candidate, so conservative cleanup should leave this issue open for maintainer review rather than close it as stale or implemented.

## What This Changes

The issue requests a native Plugin SDK `registerMcpServer` API so plugins can register managed stdio MCP servers and project dynamic MCP-backed capabilities into embedded OpenAI-backed runs.

## Best Possible Solution

Land a reviewed native SDK implementation, likely via #62160 or a replacement, that adds a typed `registerMcpServer` seam, captures plugin-owned MCP registrations, materializes them deterministically into embedded runtime config with user/operator precedence, documents the contract, and covers registry, workspace, status, and embedded MCP behavior in focused tests.

## Reproduction Assessment

Not applicable as a feature request. The verification path is source and docs inspection: current main lacks `api.registerMcpServer`, while MCP support is limited to bundle metadata and user/operator configuration paths.

## Solution Assessment

Yes in direction: a generic Plugin SDK registration seam is the maintainable place for dynamic plugin-owned MCP servers. The exact implementation should be settled through maintainer and security review of the open PR because it adds a managed stdio command-execution surface.
