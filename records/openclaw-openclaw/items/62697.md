## Summary

Keep open. Current main still drops inbound WhatsApp Channel/newsletter messages before they reach the message pipeline: non-group JIDs are resolved through phone/LID-only normalization, and the config surface has only DM and group policies. The related open PR is outbound message-tool support and explicitly leaves this inbound newsletter work separate.

## What This Changes

Add WhatsApp Channel/newsletter inbound handling with a separate access-control path so `@newsletter` messages are not treated as ordinary DMs.

## Best Possible Solution

Implement this inside the WhatsApp plugin: explicitly detect newsletter JIDs, preserve the newsletter JID as the conversation identity, add a conservative newsletter-specific policy/allowlist, and route allowed newsletter messages through the normal inbound processing without invoking DM pairing.

## Reproduction Assessment

Yes. A code-level reproduction is clear: a `messages.upsert` notify event with `key.remoteJid` ending in `@newsletter` is treated as non-group, cannot be converted by `resolveJidToE164`, and returns null before `onMessage`; tests were not run because this review is read-only.

## Solution Assessment

Yes, a separate plugin-owned newsletter access path is the maintainable direction. Reusing `dmPolicy` would keep conflating public broadcast channels with direct chats and would preserve the reported allowlist block.
