## Summary

Keep open. Current main still follows the reported hot path: embedded attempts build the full coding tool set, then mark `core-plugin-tools`, while plugin tool resolution still invokes each registered factory during resolution. The recent follow-up comment adds a second 2026.4.29 data point, and the related prep-regression issue tracks a different dominant bucket rather than superseding this plugin-tool cost.

## What This Changes

Investigate and reduce repeated `core-plugin-tools` prep latency on embedded runs, including possible plugin-tool opt-out, per-plugin timing, and context-safe factory reuse.

## Best Possible Solution

Keep this open and land a measured performance fix: add per-plugin factory timing around plugin tool resolution, expose a context-safe embedded agent/run policy to skip plugin tools when not needed, and only add memoization after separating reusable tool definitions from per-run wrappers or proving a complete cache key.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the field report: it includes repeated `[trace:embedded-run]` `core-plugin-tools` timings and a control where `disablePluginTools: true` drops the bucket to about 0 ms. Current main still follows the same factory-invocation path, though this review did not run a live timing replay.

## Solution Assessment

Is this the best way to solve the issue? No shipped/current-main solution exists yet. The reporter's memoization idea is plausible but incomplete by itself; the safer narrow fix starts with per-plugin instrumentation and an explicit plugin-tool skip path, then adds context-safe reuse only where the factory contract permits it.

## Repair Work Prompt

Fix the embedded-run `core-plugin-tools` prep latency tracked by https://github.com/openclaw/openclaw/issues/75290. Current main builds all OpenClaw coding tools in `runEmbeddedAttempt`, then marks `core-plugin-tools`; `resolvePluginTools` still calls every plugin `entry.factory(params.context)` per resolution. Implement a narrow, context-safe performance repair: add per-plugin timing/diagnostic marks around plugin tool factory resolution so slow factories are visible; thread an explicit plugin-tool-only skip policy from embedded run/agent config or runtime params into `createOpenClawCodingTools`/`createOpenClawTools` without disabling core tools; and avoid broad factory-result memoization unless the cache key covers all context-sensitive fields or the factory contract is split into reusable definition plus per-run wrapper. Keep core extension-agnostic, do not hard-code bundled plugin IDs, do not change dependency versions, and do not edit baseline/inventory files to silence checks. Add a user-facing changelog entry for the perf fix crediting the reporters with allowed handles, likely `Thanks @thanos-openclaw` and, if mentioning the second data point, `Thanks @sene1337`.
