## Summary

Keep open. Current main still pays plugin tool factory materialization during every tool-enabled embedded attempt: the `core-plugin-tools` mark sits after `createOpenClawCodingTools(...)`, and `resolvePluginTools(...)` invokes each registered plugin tool factory from the cached registry on every resolution. The report has concrete timing traces and no merged/current-main path yet provides per-agent plugin-tool opt-out, per-plugin timing, or safe request-time factory amortization.

## What This Changes

Investigate and reduce repeated `core-plugin-tools` latency during embedded runs, with possible factory memoization, plugin-tool opt-out configuration, and per-plugin timing diagnostics.

## Best Possible Solution

Add request-time plugin-tool profiling first, then reduce materialization cost with a generic, context-safe split between cached/static tool descriptors and per-run execution closures. If maintainers want an opt-out, expose it through the documented agent/tool policy surface with tests proving plugin tools are skipped before factory invocation.

## Reproduction Assessment

Yes. The issue includes repeated production trace lines, and current-main source inspection reproduces the same control flow: every tool-enabled embedded attempt reaches `resolvePluginTools`, which invokes registered factories before returning the tool list.

## Solution Assessment

No landed solution exists yet. The safest direction is not blanket memoization of returned tool objects, but a narrower design that times factories, skips provably unreachable optional/plugin tools before invoking factories, and caches only context-independent setup.
