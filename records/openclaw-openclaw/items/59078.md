## Summary

Keep open. Current main still cannot express or evaluate the reported “env vars OR skill-owned credential files” readiness path. Skill status and load-time filtering only treat `requires.env` as satisfied by process env, `skills.entries.<skill>.env`, or `skills.entries.<skill>.apiKey` tied to `primaryEnv`; docs describe the same contract, and the UI/CLI/onboarding still derive “needs setup” from that missing-env result. The reported ima-skill appears external, but the remaining blocker is a generic OpenClaw skill setup-status/API gap, not a closeable ClawHub-only cleanup.

## Best Possible Solution

Keep the issue open and design a generic skill setup-status extension point. A conservative path is either declarative alternative requirements, such as env var OR redacted file-presence checks, or a trusted plugin/skill setup-status hook that can report readiness without exposing secrets. Once that contract exists, ima-skill can publish corrected metadata through ClawHub without adding ima-specific behavior to core.
