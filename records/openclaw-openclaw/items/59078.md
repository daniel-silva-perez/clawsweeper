## Summary

Keep this issue open. Current main still derives skill eligibility from declared metadata requirements, with env requirements satisfied only by process env or OpenClaw skill config, and the Skills UI maps that ineligible state to Needs Setup. There is no current safe contract for skill-owned credential files or setup-status reporting, and a bundled Notion skill shows the same env-vs-file mismatch shape.

## What This Changes

The issue asks for skill setup/status detection to recognize credentials stored in skill-specific config files, or to let skills report setup status, instead of relying only on required env vars.

## Best Possible Solution

Add a safe declarative skill setup contract that can express credential alternatives such as env-or-file without executing untrusted skill code, then wire it through eligibility, skills.status, UI/CLI status rendering, docs, and regression tests. Bundled skills with file-backed instructions should be updated to use that contract once it exists.

## Reproduction Assessment

Yes. A skill that declares metadata.openclaw.requires.env while its credential exists only in a skill-owned config file remains ineligible on current main because the eligibility path never checks those files; the bundled Notion skill demonstrates the same shape.

## Solution Assessment

Unclear. The issue points at a real gap, but a broad self-report hook is probably not the safest first fix; a declarative credential-alternative requirement looks narrower and more maintainable, pending maintainer API review.
