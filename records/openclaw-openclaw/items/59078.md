## Summary

Keep this issue open. Current main still marks skill eligibility from declared metadata requirements, and env requirements are only satisfied by process env, `skills.entries.<skill>.env`, or `skills.entries.<skill>.apiKey`; the Skills UI then maps ineligible skills to Needs Setup. There is no current support for skill-specific credential files or a safe self-reported setup-status contract, and the bundled Notion skill shows the same env-vs-file mismatch pattern.

## What This Changes

The issue requests skill setup/status detection that recognizes credentials stored in skill-specific config files, or lets skills report their own setup status, instead of relying only on required env vars.

## Best Possible Solution

Add a safe declarative skill setup contract that can express credential alternatives such as env-or-file without executing untrusted skill code, then wire it through eligibility, `skills.status`, UI/CLI status rendering, docs, and regression tests. Bundled skills with file-backed instructions should be updated to use that contract once it exists.

## Reproduction Assessment

Yes. A skill declaring `metadata.openclaw.requires.env` while its credential is only present in a skill-owned config file will stay ineligible because current main never checks those files; the bundled Notion skill demonstrates the same shape.

## Solution Assessment

Unclear. The requested self-report option may be too broad if it requires running arbitrary skill code; a declarative metadata extension for credential alternatives looks safer, but that public API shape needs maintainer review.
