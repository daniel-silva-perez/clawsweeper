## Summary

Keep open. Current main has meaningful cron/session retention mitigation, including `cron.sessionRetention` and the cron run-session reaper, but the requested high-frequency cron guardrail is still missing from the creation/update surfaces. The same-author implementation PR #38731 remains open and uses closing syntax for this issue, so this issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and treat #38731 as the active implementation candidate. The best fix is a small shared high-frequency `every` schedule warning helper applied consistently to CLI add/edit, Gateway add/update, Control UI add/edit, and agent-tool-created cron jobs, while keeping short intervals valid and non-breaking. After that PR or an equivalent fix lands, close this as implemented with the landed SHA and release provenance.
