## Summary

Keep open. Current main has cron run-session retention/reaper mitigation and general session-store maintenance, but the requested creation-time high-frequency cron guardrail is still absent on the main creation/update surfaces. A same-author PR, #38731, explicitly links this issue for closure and remains open/unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and treat #38731 as the active implementation candidate. The best fix is a small shared high-frequency every-schedule warning helper applied consistently to CLI add/edit, Gateway add/update, Control UI add/edit, and agent-tool-created cron jobs, while keeping short intervals valid and non-breaking. Maintainers should also decide whether manual sessions cleanup should invoke the cron run-session reaper or remain documented as general session maintenance only.
