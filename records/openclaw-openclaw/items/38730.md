## Summary

Keep open. Current main has the cron run-session retention/reaper mitigation, but the requested creation-time high-frequency cron guardrail is still missing across the main creation paths. A same-author open PR, #38731, explicitly closes this issue and remains unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and use #38731 as the active implementation candidate. The best fix is a shared high-frequency every-schedule warning helper applied consistently to CLI, Gateway RPC logs/results, Control UI, and agent-tool-created cron jobs, plus a maintainer decision on whether manual sessions cleanup should invoke the cron run-session reaper or whether cron cleanup remains scheduler-owned and documented that way.
