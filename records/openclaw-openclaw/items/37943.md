## Summary

Keep this issue open. Current main still has the reported failure shape: the macOS node service is installed as `openclaw node run`, the invalid-config guard still exits for `node`, and the node runner still reads strict runtime config before connecting. Gateway startup has a separate last-known-good/best-effort recovery path, but that path is not shared with node startup.

## Best Possible Solution

Keep the issue open and decide the intended node startup contract. The likely fix is either to give `openclaw node run` a safe, explicit best-effort/last-known-good config path for node auth/TLS/plugin registry needs, or to keep node startup strict but emit node-specific diagnostics and repair guidance. Add regression coverage around invalid effective config and update the node docs once the policy is chosen.
