## Summary

Keep this issue open. Current main still has the reported failure shape: the macOS node service runs `openclaw node run`, the global invalid-config guard still does not allow `node`, and `runNodeHost()` still reads strict runtime config before connecting. Gateway startup has last-known-good/best-effort recovery, but that recovery path is still separate from node startup.

## Best Possible Solution

Keep this open and decide the intended node startup contract. The likely product choice is either to give `openclaw node run` a narrow, explicit best-effort/last-known-good config path for node auth/TLS/plugin registry needs, or to keep node startup strict but emit node-specific diagnostics and repair guidance. Add regression coverage for invalid effective config on `node run` and update `docs/cli/node.md` once the policy is chosen.
