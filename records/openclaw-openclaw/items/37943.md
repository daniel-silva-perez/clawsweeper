## Summary

Current main still matches the reported failure shape. The macOS node service launches `openclaw node run`; `node` is not in the invalid-config allowlist and does not bypass the config guard; and `runNodeHost()` still reads strict runtime config before creating the gateway client. Gateway startup has separate best-effort and last-known-good recovery logic that node startup does not share.

## Best Possible Solution

Keep this open for maintainer policy. Decide the intended `openclaw node run` startup contract, then either add a narrow node-specific best-effort or last-known-good config path for the TLS/auth/plugin-registry data it needs, or keep node startup strict but emit node-specific diagnostics and repair guidance. Add regression coverage for invalid effective config on node startup and update `docs/cli/node.md` if behavior changes.
