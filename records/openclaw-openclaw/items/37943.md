## Summary

Current main still matches the reported failure shape: the installed macOS node service runs `openclaw node run`, the invalid-config guard still exits for non-allowlisted commands and does not allow `node`, and `runNodeHost()` still reads strict runtime config before connecting. Gateway startup has separate best-effort and last-known-good recovery logic that node startup does not share.

## Best Possible Solution

Keep this open for maintainer policy. The likely fix is to decide the intended `openclaw node run` startup contract, then either add a narrow node-specific best-effort or last-known-good config path for the TLS/auth/plugin-registry data it needs, or keep node startup strict but emit node-specific diagnostics and repair guidance. Add regression coverage for invalid effective config on node startup and update `docs/cli/node.md` if behavior changes.
