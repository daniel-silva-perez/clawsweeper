## Summary

Current main still has the reported cron/main-session timeout gap. Main-session systemEvent jobs with wakeMode="now" enqueue the event, then await runHeartbeatOnce under the generic 10-minute non-agent timeout. systemEvent payloads still cannot set timeoutSeconds, and current tests continue to encode blocking heartbeat completion rather than fire-and-forget behavior. The provided related context also shows the later duplicate was closed into this issue, while the two targeted fix PRs were closed unmerged.

## Best Possible Solution

Keep this issue open and fix the main-session cron contract explicitly. The likely path is to change src/cron/service/timeout-policy.ts and/or src/cron/service/timer.ts so sessionTarget="main", payload.kind="systemEvent", wakeMode="now" either records successful enqueue-and-wake without waiting on the full heartbeat turn, or uses a documented/configurable timeout that is aligned with long-running main-session work. Add regression coverage beside the timeout-policy and main-job wake tests, then update the scheduled-tasks docs if any timeout remains user-visible.

## Risks / Open Questions

- Changing wakeMode="now" from blocking to fire-and-forget affects current run-history timing, one-shot deletion timing, and what cron can honestly report as success versus delivered/enqueued.
- A narrower 60-minute timeout would reduce false failures but would not fully satisfy the issue's stated fire-and-forget expectation unless docs and status semantics are updated accordingly.
