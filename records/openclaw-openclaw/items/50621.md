## Summary

Current main still matches the reported behavior: main-session systemEvent cron jobs use the generic non-agent cron timeout, wakeMode="now" still awaits runHeartbeatOnce, and systemEvent payloads still have no timeoutSeconds escape hatch. The duplicate report was already closed into this issue, while the two targeted fix PRs were closed unmerged, so this issue remains the canonical open tracker.

## Best Possible Solution

Keep this issue open and make the main-session cron contract explicit. The likely fix belongs in src/cron/service/timeout-policy.ts and/or src/cron/service/timer.ts: for sessionTarget="main", payload.kind="systemEvent", wakeMode="now", either record success after enqueue-and-wake without waiting for the full heartbeat turn, or expose/document a configurable timeout that applies to systemEvent jobs. Add regression coverage beside timeout-policy and main wake-now tests, and update https://docs.openclaw.ai/automation/cron-jobs if any timeout remains user-visible.

## Risks / Open Questions

- Changing wakeMode="now" to true fire-and-forget affects run-history timing, one-shot deletion timing, and what cron can honestly report as success versus only delivered/enqueued.
- Only increasing the timeout to 60 minutes would reduce false failures but would not fully satisfy the stated fire-and-forget expectation unless status semantics and docs are updated.
- Recurring main-session jobs have a busy-lane fallback now, but a heartbeat that starts and then runs longer than the generic systemEvent timeout still remains exposed to the reported failure mode.
