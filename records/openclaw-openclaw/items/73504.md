## Summary

Keep open. The PR still contains the useful missing wrapper forwarding for Discord message-action execution mode, but it is not merge-ready against current main because current main already narrowed Discord gateway routing to read/search and this branch's unconditional resolver would widen gateway routing for unrelated Discord actions.

## What This Changes

The PR adds Discord message-action execution-mode routing and forwards that resolver through the registered Discord channel wrapper so CLI message actions can execute via the gateway.

## Best Possible Solution

Update this PR so the only production behavior change is forwarding the existing read/search-only resolver through `extensions/discord/src/channel.ts`, then add focused coverage that the registered Discord plugin routes read/search to gateway while leaving other Discord actions local.

## Reproduction Assessment

Yes. The routing bug can be reproduced at source level: current main's registered Discord wrapper lacks `resolveExecutionMode`, and the shared runner defaults a missing resolver to local execution. The PR regression is also source-level reproducible because its added resolver is unconditional while current tests require non-read/search actions to stay local.

## Solution Assessment

No, not as-is. Forwarding the wrapper resolver is the right narrow fix, but replacing the adapter behavior with unconditional gateway execution is broader than the reported read/search hang and conflicts with current main's narrowed resolver contract.
