## Summary

Keep this open. The report is recent, concrete, and gateway-wide, and current main does not provide enough evidence that the WSL2 half-hour stall is fixed; there are plausible heartbeat, Telegram polling, and gateway responsiveness paths, but no single implemented change fully covers the reported Control UI plus Telegram stall pattern.

## What This Changes

The issue reports a WSL2 regression where the gateway stalls for several minutes at clock windows around :29/:59, affecting both Telegram and the Control UI.

## Best Possible Solution

Keep the issue as a focused WSL2 timing investigation, collect a diagnostics export and redacted config captured around a :29/:59 window, then fix the confirmed shared blocker in the owning path. If the root cause is heartbeat, land the disable semantics and add scheduling tests; if it is gateway event-loop blocking, add liveness coverage around the blocking surface rather than a Telegram-only workaround.
