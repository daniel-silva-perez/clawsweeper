## Summary

Keep open. Current main still has adjacent strict-agentic, liveness, abort, and wait-snapshot safeguards, but it does not define the requested public atomic run.begin/run.exec/run.commit-or-run.abort lifecycle contract with structured abort blockers; the existing maintainer discussion reached the same conclusion and current source remains aligned with that keep-open assessment.

## What This Changes

The issue requests a transaction-style run lifecycle that guarantees announced action intent is followed by execution evidence and a mandatory commit or abort state with a machine-readable blocker.

## Best Possible Solution

Define an additive protocol-level lifecycle contract that distinguishes intent, execution evidence, committed completion, and aborted/blocker completion, then wire it through AgentEvent types, Gateway schemas, embedded runtime emitters, agent.wait, SDK/client normalization, session state, docs, and regression tests without breaking existing lifecycle consumers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No concrete runtime interruption reproduction is provided, but current-main source and docs inspection gives high-confidence proof that the requested lifecycle contract is absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the public contract; the right direction appears to be an additive lifecycle protocol rather than more local heuristics, but the exact state model and rollout need human review.
