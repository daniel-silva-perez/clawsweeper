## Summary

Keep open. Current main has automatic per-channel restart paths for config hot reload, secrets reload, and health monitoring, but it still does not expose the requested manual channel reload CLI or Gateway RPC; the existing `channels.start` path is not a forced reload for an already-running account.

## What This Changes

The issue requests a targeted channel reload command or Gateway action that reconnects one named channel provider without restarting the whole Gateway or interrupting unrelated sessions.

## Best Possible Solution

Add an admin-scoped `channels.reload` Gateway method and an `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper that reuses the existing stop/start lifecycle, preserves unrelated channels and sessions, returns structured status, and updates protocol schemas, method scopes, CLI/Gateway docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main gap is high-confidence from inspection of the CLI registration, Gateway method list, admin scopes, protocol schema, docs, and existing start/stop behavior.

## Solution Assessment

Unclear pending maintainer API direction. A targeted reload surface is a maintainable direction, but the best implementation should align with existing admin scopes and the account-scoped lifecycle semantics being discussed in #43938.
