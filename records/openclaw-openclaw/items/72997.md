## Summary

Keep open. Current main has config-driven channel hot reload and health-monitor restart paths, but it still does not expose the requested manual `openclaw channels reload <provider>` command or a Gateway `channels.reload` RPC; `channels.start` is not equivalent because it returns when the target account is already running.

## What This Changes

Requests a targeted CLI or Gateway action to disconnect and reconnect one channel/provider without restarting the Gateway or interrupting unrelated channels and sessions.

## Best Possible Solution

Add an admin-scoped `channels.reload` Gateway method plus an `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper that reuses the existing stop/start lifecycle, preserves unrelated channels and sessions, returns structured status, and updates protocol schemas, method scopes, CLI/Gateway docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main gap is high-confidence from source and docs inspection of the CLI registration, Gateway method list, admin scopes, protocol schema, and existing start/stop behavior.

## Solution Assessment

Unclear pending maintainer API direction. A targeted reload is the right product direction for the reported use case, but the safest implementation should align with admin authorization and account-scoped lifecycle semantics before code is written.
