## Summary

Keep open. Current main has internal scoped restart paths for config hot reload, secrets reload, and channel health recovery, but it still does not expose the requested manual channel reload CLI command or Gateway RPC; the related open reload-planning items are adjacent rather than superseding.

## What This Changes

Request a targeted manual channel reload command or Gateway action that reconnects only the named channel provider without restarting the gateway or interrupting unrelated channels/sessions.

## Best Possible Solution

Add an additive admin-scoped `channels.reload` Gateway method and a matching `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper that reuses the existing stop/start lifecycle, preserves unrelated channels and active sessions, returns structured status, and updates protocol schemas, docs, changelog, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is high-confidence from source and docs inspection: no CLI command, Gateway method, protocol schema, validator, or admin scope exists for manual channel reload, and `channels.start` is idempotent rather than a forced reload.

## Solution Assessment

Unclear until a maintainer chooses the public API details. A targeted reload surface is the maintainable direction, but it should align with existing plural `channels.*` Gateway methods and account-scoped lifecycle semantics rather than adding an ad hoc singular action.
