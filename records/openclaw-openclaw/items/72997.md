## Summary

Keep open. Current main has automatic channel restart paths for config reload, secrets reload, and health monitoring, but it still lacks the requested manual `openclaw channels reload` command or `channels.reload` Gateway RPC; `channels.start` is not a forced reload for an already-running channel/account.

## What This Changes

The issue requests a targeted channel reload command or Gateway action that reconnects a single channel provider without restarting the gateway or disrupting other channels.

## Best Possible Solution

Add an admin-scoped `channels.reload` Gateway method and an `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper that reuses the existing stop/start lifecycle, preserves unrelated channels and sessions, returns structured status, and updates protocol schemas, method scopes, CLI/Gateway docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main gap is high-confidence from source inspection of the CLI registration, Gateway method list, method scopes, protocol schema, docs, and existing start/stop behavior.

## Solution Assessment

Unclear pending maintainer API direction. The requested targeted reload is a maintainable direction, but the best implementation should align with existing admin scopes and #43938's account-scoped lifecycle semantics before code is written.
