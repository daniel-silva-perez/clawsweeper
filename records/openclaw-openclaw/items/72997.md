## Summary

Keep open. Current main has internal per-channel restart paths for config hot reload, secrets reload, and health monitoring, but it still does not expose the requested manual channel reload CLI command or Gateway RPC method; the existing discussion and related items are adjacent rather than superseding.

## What This Changes

Request add a targeted `openclaw channel reload <provider>` or equivalent Gateway action that reconnects only one channel without restarting the gateway.

## Best Possible Solution

Add an additive admin-scoped `channels.reload` Gateway method plus a matching `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper that reuses the existing channel stop/start lifecycle, returns structured status, preserves unrelated channels and sessions, and updates protocol schemas, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is high-confidence from source and docs inspection: no CLI command, Gateway method, protocol schema, or validator exists for manual channel reload, and `channels.start` is not a forced reload.

## Solution Assessment

Unclear pending maintainer API direction. A targeted reload surface is the maintainable direction, but it should align with existing plural `channels.*` Gateway methods and account-scoped lifecycle semantics rather than adding an ad hoc singular action.
