## Summary

Keep open. Current main has automatic per-channel restart paths for config hot reload, secrets reload, and health monitoring, but it still does not expose the requested manual channel reload CLI command or Gateway RPC method; `channels.start` is not a forced reload for an already-running account.

## What This Changes

Request a targeted channel reload command or Gateway action that disconnects and reconnects only the named channel/provider without restarting the whole Gateway or disrupting unrelated sessions.

## Best Possible Solution

Add an admin-scoped `channels.reload` Gateway method plus a matching `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper that reuses the existing channel lifecycle, preserves unrelated channels and sessions, returns structured status, and updates protocol schemas, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main gap is high-confidence from inspection of the CLI registration, Gateway method list, admin scopes, protocol schema, docs, and lifecycle behavior.

## Solution Assessment

Unclear pending maintainer API direction. A targeted reload surface is maintainable, but the exact solution should align with existing admin scopes and account-scoped lifecycle semantics rather than adding an ad hoc `channel.reload` action.
