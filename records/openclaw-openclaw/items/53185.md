## Summary

Keep this issue open. Current main has external-content wrapping, special-token cleanup, and partial hidden HTML/invisible Unicode cleanup for web_fetch, but it still does not implement the requested content-sanitizer.py/script preprocessing pipeline, threat_level metadata, shared web_search sanitizer/classifier, or HIGH/CRITICAL block behavior.

## Best Possible Solution

Keep this issue open but narrow it. Do not integrate a private local Python path directly. Make a maintainer product/security decision on a trusted pre-model sanitizer contract for web_fetch and web_search: either a bundled sanitizer/classifier using the trusted tool-result middleware with documented threat_level and fail-closed behavior, or an explicit trusted plugin/script seam with audit logs, config/schema/docs, and tests. Keep broader provenance and execution-guardrail work connected to #54508.

## Risks / Open Questions

- Closing as implemented would conflate useful shipped web_fetch cleanup with the still-missing script preprocessing, threat_level metadata, shared web_search sanitizer, and HIGH/CRITICAL block semantics.
- A user-configured stdin/stdout script hook would rewrite model-visible tool output and execute local code, so it needs an explicit trust model, failure semantics, auditability, config/schema/docs, and regression tests before becoming a core surface.
- This should not be closed as clawhub today because the needed pre-model tool-result middleware is currently bundled/trusted-only, not a normal external plugin capability.
