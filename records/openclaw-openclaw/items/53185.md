## Summary

Keep open. Current main has meaningful external-content wrapping and partial `web_fetch` hidden-content cleanup, but it still lacks the requested sanitizer/script preprocessing contract, structured `threat_level` output, and HIGH/CRITICAL block behavior across `web_fetch` and `web_search`. The proposed local script hook is security-sensitive because it would execute configured code and rewrite untrusted tool output before model consumption, so this needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

Add a sanitizer or tool-preprocessing path for `web_fetch` and `web_search` that strips or classifies malicious content, emits threat metadata, and blocks high-risk results.

## Best Possible Solution

Narrow this into a trusted web result sanitizer/classifier contract. Avoid wiring a private local script path directly; decide between a bundled sanitizer or a documented trusted plugin/script seam with explicit threat metadata, fail-closed behavior, audit/logging semantics, config/schema coverage, docs, and regression tests for both `web_fetch` and managed `web_search`.
