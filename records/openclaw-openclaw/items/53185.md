## Summary

Keep open. Current main has meaningful external-content wrapping and partial web_fetch cleanup, but it still lacks the requested sanitizer/script preprocessing pipeline, structured threat_level metadata, shared web_search classifier, and HIGH/CRITICAL block behavior. The related #54508 is broader provenance/guardrail work, not a complete replacement for this narrower web tool sanitizer decision.

## What This Changes

The issue requests routing web_fetch and web_search results through a sanitizer or preprocessing layer that strips unsafe content, detects hidden/encoding attacks, emits threat_level metadata, and blocks HIGH/CRITICAL content.

## Best Possible Solution

Keep this issue open but narrow it around a trusted web result sanitizer/classifier contract. Do not wire a private local Python path directly; make a maintainer security/product decision between a bundled sanitizer for web_fetch/web_search or a documented trusted plugin/script seam with explicit threat metadata, fail-closed behavior, audit logs, config/schema coverage, docs, and regression tests.
