## Summary

Keep this issue open. Current main has useful external-content wrapping and partial `web_fetch` cleanup for hidden HTML/invisible Unicode, but it still does not implement the requested `content-sanitizer.py`/script preprocessing pipeline, `threat_level` metadata, shared `web_search` classifier, or HIGH/CRITICAL block behavior. The related #54508 is broader security/taint work, not a complete replacement for this narrower sanitizer/preprocessing request.

## Best Possible Solution

Keep this issue open but narrow it. Do not integrate a private local Python path directly. Make a maintainer product/security decision on a trusted pre-model sanitizer contract for `web_fetch` and `web_search`: either a bundled sanitizer/classifier with documented `threat_level` and fail-closed behavior, or an explicit trusted plugin/script seam with audit logs, config/schema/docs, and tests. Keep the broader provenance and execution-guardrail design connected to #54508.

## Risks / Open Questions

- Closing as implemented would conflate useful shipped `web_fetch` cleanup with the still-missing script preprocessing, `threat_level` metadata, shared `web_search` sanitizer/classifier, and HIGH/CRITICAL block semantics.
- Closing as `clawhub` would overstate current plugin capability: external plugins can register providers, but cannot currently register trusted pre-model tool-result middleware for arbitrary existing tool output.
- A user-configured stdin/stdout script hook would rewrite model-visible tool output and execute local code, so it needs explicit trust, failure, audit, config/schema, docs, and regression-test design.
