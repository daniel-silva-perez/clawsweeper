## Summary

Keep open: the underlying Discord multi-account routing problem is real on current main, but this PR is not mergeable as written. The branch is dirty against current main and the patch introduces blocking routing regressions around DMs, inactive accounts, plugin-boundary imports, current-main wildcard routes, and plain-name mention bypass semantics.

## What This Changes

This PR adds Discord cross-account route-owner selection, explicit mention bypass and bot-name mention plumbing, an application-id fallback for `botUserId` when `fetchUser("@me")` fails, and focused Discord monitor tests.

## Best Possible Solution

Refresh or replace the branch with a Discord-plugin-owned implementation on current main that ranks only active Discord accounts, suppresses non-owner guild/channel traffic by default, preserves explicit Discord metadata mention bypass, avoids owner blocking for DMs, supports current routing tiers including wildcard peer bindings, and keeps routing imports behind the plugin SDK/local Discord exports.

## Reproduction Assessment

Yes. The PR body gives a concrete two-account Discord reproduction, current main confirms the missing owner resolution and fail-closed identity lookup, and the PR-introduced regressions are directly traceable from the added owner-check and candidate-account code paths.

## Solution Assessment

No. The direction is useful, but this is not the best mergeable fix because it blocks valid DM cases, can pick inactive or stale accounts as owner, bypasses the extension boundary, misses current-main wildcard route precedence, and treats plain bot-name text as an explicit owner bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:442-452`
  - body: The owner guard runs for direct messages, but `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding, this bot can silently drop a DM even though the other bot will not receive that DM event.
  - confidence: 0.92
- **[P2] Restrict owner candidates to active Discord accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: `listDiscordCandidateAccountIds` includes account IDs that appear only in bindings, plus disabled or duplicate-token accounts. A stale binding can therefore win ownership and make the active bot suppress messages for an account that is not running.
  - confidence: 0.88
- **[P2] Import routing through the plugin SDK:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new Discord plugin production module imports config and routing internals through `../../../../src/**`. Bundled plugin code should use `openclaw/plugin-sdk/*` or local Discord exports, so this bypasses the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer matches in the priority table:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return `matchedBy: "binding.peer.wildcard"`, but the new priority table has no entry for it. After rebasing, wildcard owner routes will either fail the type contract or compare with an undefined priority.
  - confidence: 0.84
- **[P2] Require real Discord mentions for owner bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: `buildBotNameMentionRegexes` makes `@` optional and is merged into the owner-bypass mention set. That lets normal text containing the bot username bypass channel ownership even though the related routing contract wants Discord mention metadata, not plain names, to select an alternate bot.
  - confidence: 0.78
