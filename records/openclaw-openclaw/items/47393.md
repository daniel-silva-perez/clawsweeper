## Summary

Keep open. The underlying Discord multi-bot routing problem still exists on current main, but this PR branch is stale and has blocking correctness, extension-boundary, and data-scope issues that should be resolved before merge.

## What This Changes

The PR adds Discord route-owner resolution, explicit mention/name-based owner bypass, bot username preflight plumbing, an application-id fallback for bot user ID, and focused Discord monitor tests.

## Best Possible Solution

Refresh or replace the branch with a Discord-plugin-owned implementation on current main that ranks only active Discord accounts, suppresses non-owner guild/channel traffic by default, preserves explicit Discord metadata mention and reply-target precedence, excludes DMs from owner blocking, handles wildcard route precedence, keeps imports behind plugin SDK or local Discord exports, and coordinates with the existing open routing reports.

## Reproduction Assessment

Yes. The PR body gives a concrete two-account Discord guild-channel reproduction, and current main confirms the missing cross-account owner-resolution path plus fail-closed identity lookup. The PR-introduced regressions are also reproducible as static checks against the added owner guard and route-owner helper.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable fix because it can block valid DMs, choose inactive accounts as owners, bypass plugin boundaries, miss wildcard route precedence, and treat plain bot-name text as explicit owner bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:442-452`
  - body: The new owner check runs for direct messages, while `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this bot can silently drop a valid DM even though the other bot is not handling this DM event.
  - confidence: 0.92
- **[P2] Restrict owners to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: `resolveDiscordRouteOwner` considers account IDs that appear only in bindings, including disabled, removed, or duplicate-token accounts. A stale binding can win ownership and make active monitors suppress messages for an account that is not running.
  - confidence: 0.88
- **[P2] Import routing through the plugin SDK:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: The new Discord plugin production module imports config and routing internals through `../../../../src/**`. Bundled plugin code should use `openclaw/plugin-sdk/*` or local Discord exports, so this bypasses the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer priority:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return `matchedBy: "binding.peer.wildcard"`, but the new priority table has no entry for it. After rebasing, wildcard owner routes will fail the type contract or compare with an undefined priority.
  - confidence: 0.84
- **[P2] Require real Discord mentions for bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: `buildBotNameMentionRegexes` makes `@` optional and is merged into the owner-bypass mention set. That lets ordinary text containing the bot username bypass channel ownership even though the related routing policy requires Discord mention metadata, not plain names, to select an alternate bot.
  - confidence: 0.78
