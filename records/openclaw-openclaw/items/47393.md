## Summary

Keep this PR open. It addresses a real Discord multi-account routing gap, but the branch is not mergeable as-is against current main: the proposed owner guard can drop valid DMs, can choose inactive or stale accounts as owners, bypasses the plugin SDK boundary, is stale against wildcard route precedence, and treats plain bot-name text as an owner-bypass signal.

## What This Changes

The PR adds Discord route-owner selection, explicit mention bypass for non-owner bots, a `botUserName` preflight field, an application-id fallback for `botUserId`, and focused Discord monitor tests.

## Best Possible Solution

Refresh or replace the branch with a Discord-plugin-owned implementation on current main that ranks only active Discord accounts, suppresses non-owner guild/channel traffic by default, preserves explicit Discord metadata mention bypass, avoids owner blocking for DMs, handles wildcard route precedence, and keeps imports behind the plugin SDK or local Discord exports.

## Reproduction Assessment

Yes. The PR body gives a concrete two-account Discord reproduction, and current main confirms the missing owner-resolution path plus current fail-closed identity lookup. The blocking PR regressions are reproducible by reading the added guard and owner-resolution helper against the current routing and Discord account contracts.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable fix because it can block valid DMs, pick inactive accounts as owners, bypass extension boundaries, miss current wildcard route precedence, and treat plain bot-name text as explicit owner bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:442-452`
  - body: The new owner guard runs for direct messages, but `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding, this bot can silently drop a DM even though the other bot will not receive that DM event.
  - confidence: 0.92
- **[P2] Restrict owner candidates to active Discord accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: `resolveDiscordRouteOwner` considers account IDs that appear only in bindings, including disabled, removed, or duplicate-token accounts. A stale binding can therefore win ownership and make active monitors suppress messages for an account that is not running.
  - confidence: 0.88
- **[P2] Import routing through the plugin SDK:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new Discord plugin production module imports config and routing internals through `../../../../src/**`. Bundled plugin code should use `openclaw/plugin-sdk/*` or local Discord exports, so this bypasses the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer matches in the priority table:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return `matchedBy: "binding.peer.wildcard"`, but the new priority table has no entry for it. After rebasing, wildcard owner routes will either fail the type contract or compare with an undefined priority.
  - confidence: 0.84
- **[P2] Require real Discord mentions for owner bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: `buildBotNameMentionRegexes` makes `@` optional and is merged into the owner-bypass mention set. That lets ordinary text containing the bot username bypass channel ownership even though the linked routing policy requires Discord mention metadata, not plain names, to select an alternate bot.
  - confidence: 0.78
