## Summary

Keep open. Current main has a substantial SecretRef system, audit/configure/apply flows, and one-way scrubbing, but it still does not implement encrypted-at-rest `.env` values, `ENC:` transparent decryption, master-key/keyring setup, or key rotation; this remains security-sensitive product/API design rather than conservative cleanup.

## What This Changes

The issue requests built-in encrypted-at-rest storage for sensitive configuration values, including encrypted `.env` entries, transparent decryption, keyring-backed master key setup, and key rotation.

## Best Possible Solution

Design a first-party encrypted/keyring-backed SecretRef provider or local vault that fits the existing activation snapshot, audit/configure/apply workflow, onboarding, docs, fail-fast semantics, and secops review path, then add a deliberate migration and rotation story for plaintext `.env` and auth-profile values.

## Reproduction Assessment

Not applicable as a feature request rather than a runtime bug. Current-main source and docs provide a high-confidence negative check: the only supported secret sources are env/file/exec references, and the `.env` path scrubs plaintext instead of encrypting it.

## Solution Assessment

No current implementation or proposed patch is ready to approve. A quick transparent `ENC:` rewrite would be riskier than a reviewed key-management design that defines key storage, rotation, migration, runtime plaintext boundaries, and redaction behavior.
