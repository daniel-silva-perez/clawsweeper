## Summary

Keep open. Current main supports normal WhatsApp sender routing through `match.peer`, so the issue's `accountId` example is not the documented sender-binding shape, but WhatsApp still lacks the configured-binding provider and inbound configured-binding route integration used by several other channels. Because the report is about an agent privilege-routing boundary, this should not be closed as stale or implemented.

## What This Changes

The issue asks for WhatsApp `bindings[]` route rules to be honored or warned on instead of silently falling through to the default agent.

## Best Possible Solution

Define the intended WhatsApp binding contract, then either wire WhatsApp into the configured-binding provider and inbound route-resolution path with direct E.164 and group JID normalization, or add config validation/docs warnings that make unsupported or mis-scoped WhatsApp bindings impossible to mistake for an agent privilege boundary.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence reproduction for the missing WhatsApp configured-binding provider and missing inbound configured-binding route step, but the exact example uses `accountId` where current docs and router code require `match.peer` for sender routing.

## Solution Assessment

Unclear. Adding only the two provider hooks would address part of the configured-binding gap, but it would not fix the reporter's `accountId` sender-routing confusion; the safer maintainable solution likely combines explicit WhatsApp binding support with validation or docs around `accountId` versus `peer`.
