## Summary

Keep open. Current main does support normal WhatsApp route bindings through the generic router when the sender is expressed as `match.peer`, so the issue's `accountId` example appears to be the wrong shape for a per-sender route. However, WhatsApp still has no configured-binding provider and no explicit warning around that unsupported path, so this security-relevant routing-boundary report needs maintainer/security judgment rather than cleanup closure.

## What This Changes

The issue requests WhatsApp binding configuration to route inbound messages to the intended agent or warn when unsupported so messages do not unexpectedly fall through to the default agent.

## Best Possible Solution

Clarify and enforce the WhatsApp binding contract: sender-specific route bindings should use `match.peer`, `accountId` should remain the account/login selector, and unsupported WhatsApp configured bindings or sender-shaped `accountId` mistakes should produce validation or runtime warnings. If persistent ACP bindings are intended for WhatsApp, add a WhatsApp configured-binding adapter and inbound route integration with focused regression coverage.

## Reproduction Assessment

Unclear. Static source gives a high-confidence way to reproduce the missing WhatsApp configured-binding provider, but the reporter's exact route example uses `accountId` where current docs require `match.peer`, so a corrected config or live replay is needed to isolate the route fallback behavior.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address the configured ACP binding gap, but it would not make the issue's `accountId` example a per-sender route; the safer solution also needs validation or documentation around `accountId` versus `peer`.
