## Summary

Keep open. Current main still awaits `memory_search` manager setup and search execution without a memory-tool hard deadline, and open PR #49794 is still the active unmerged implementation candidate that links this issue as its close target.

## What This Changes

The issue asks for `memory_search` to fail open with a bounded timeout so memory manager, source, or provider stalls cannot wedge a live assistant turn.

## Best Possible Solution

Review and land PR #49794, or replace it with an equivalent memory-core fix that wraps the full `memory_search` tool boundary, including runtime load, manager context creation, QMD pending creation, builtin manager creation, and `manager.search`, in a hard deadline that always returns the existing unavailable payload. Add fake-timer regression coverage for never-settling manager-init and search promises before closing this issue.

## Reproduction Assessment

Yes for a code-level reproduction: stub `getMemorySearchManager` or `memory.manager.search` to return a never-settling promise and call `memory_search`; current main awaits that promise without a hard deadline. A live Telegram/iCloud reproduction was not run because this review was read-only.

## Solution Assessment

Yes, the requested hard fail-open boundary is the narrowest maintainable solution. Generic transcript/tool-result repair is not sufficient because current tool execution still awaits `tool.execute(...)`, and the memory tool ignores the abort signal while awaiting manager setup and search.
