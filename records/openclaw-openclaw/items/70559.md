## Summary

Keep this open. Current `main` still has the reported unsafe-reindex/sqlite-vec state mismatch: the unsafe full reindex drops `chunks_vec`, leaves vector availability cached as true, and then prepares vector DELETE statements before any later table recreation can occur.

## What This Changes

The issue requests a fix so `runUnsafeReindex` can rebuild an in-place memory index when sqlite-vec is enabled without crashing on a missing `chunks_vec` table.

## Best Possible Solution

Patch `memory-core` so unsafe full reindex keeps vector table existence and vector readiness state consistent before sync setup prepares vector DELETE statements, then add focused regression coverage for sqlite-vec-enabled unsafe reindex. Keep the Windows safe atomic-swap fallback work separate.

## Reproduction Assessment

Yes. The current source path is enough to reproduce with high confidence: force the unsafe reindex path after sqlite-vec has set `vector.available = true`, let `resetIndex()` drop `chunks_vec`, and the next sync setup prepares a DELETE against that missing table.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve and clear `chunks_vec` when sqlite-vec is available, or otherwise reset vector readiness so table recreation happens before vector DELETE statements are prepared; changing Windows rename behavior belongs to the separate safe-path issue.

In `extensions/memory-core/src/memory/manager-sync-ops.ts`, make `runUnsafeReindex()`/`resetIndex()` keep vector table existence and vector availability/readiness state consistent before `syncMemoryFiles()` or `syncSessionFiles()` prepare `DELETE FROM chunks_vec ...` statements.
