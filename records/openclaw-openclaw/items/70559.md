## Summary

Current main still has the unsafe-reindex/sqlite-vec state mismatch reported here. `ensureVectorReady()` can cache `vector.available = true`; the env-gated unsafe full reindex path calls `resetIndex()`, which drops `chunks_vec` and clears only `vector.dims`; then `syncMemoryFiles()` / `syncSessionFiles()` prepare `DELETE FROM chunks_vec ...` based only on availability before `indexFile()` can recreate the vector table. A direct in-memory `node:sqlite` probe confirms preparing that statement against a missing table throws `no such table: chunks_vec`. Related #71611 targets #64187's safe-path Windows rename/EBUSY issue and does not supersede this crash.

## Best Possible Solution

Keep this issue open and fix `memory-core` so unsafe full reindex keeps vector table state consistent: either preserve and clear `chunks_vec` when sqlite-vec is available, or reset/recreate vector state before any sync code prepares vector-table statements. Add a focused regression test with vector availability true that would fail on the missing-table prepare before the fix. Treat #64187/#71611 as the separate safe-path Windows EBUSY track.
