## Summary

Current main still has the reported unsafe-reindex/vector-table state mismatch: sqlite-vec can be marked available, `resetIndex()` drops `chunks_vec` without clearing availability, and the next sync prepares DELETE statements against the missing table. Related PR #71611 addresses the safe Windows EBUSY path, not this `runUnsafeReindex` crash.

## Best Possible Solution

Keep this issue open and fix `memory-core` so unsafe full reindex leaves vector table state consistent: either preserve and clear the existing `chunks_vec` table when sqlite-vec is available, or reset/recreate vector state before any sync code prepares statements. Add a focused regression test that exercises the unsafe reindex path with vector availability true and catches a missing `chunks_vec` prepare failure. Treat #64187/#71611 as the separate safe-path Windows EBUSY track.
