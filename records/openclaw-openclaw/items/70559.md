## Summary

Current main still has the unsafe-reindex/sqlite-vec state mismatch described in the report, so this should stay open as a narrow memory-core bug. The code still loads sqlite-vec and caches `vector.available = true`, the unsafe full reindex still drops `chunks_vec` without clearing availability, and both memory and session sync still prepare `DELETE FROM chunks_vec ...` before any later vector-table recreation can happen.

## What This Changes

The issue requests a fix for `OPENCLAW_TEST_FAST=1` plus `OPENCLAW_TEST_MEMORY_UNSAFE_REINDEX=1` full memory reindexing so sqlite-vec deployments do not crash after `resetIndex()` drops `chunks_vec`.

## Best Possible Solution

Patch `memory-core` so unsafe full reindex keeps vector table existence and vector availability state consistent before sync setup prepares vector DELETE statements, then add a focused regression test for the missing-table path. Keep the broader Windows atomic-swap fallback work separate under #64187/#71611.

## Reproduction Assessment

Yes. The source path is deterministic on current main, and an in-memory `node:sqlite` probe confirms that preparing the reported `DELETE FROM chunks_vec ...` statement without the table throws `no such table: chunks_vec`.

## Solution Assessment

Yes, the narrowest maintainable fix is to make `resetIndex()`/`runUnsafeReindex()` either preserve and clear `chunks_vec` when sqlite-vec is available or reset vector readiness/availability so later indexing recreates the table before vector writes. A broad Windows rename fallback would solve a different safe-path problem and should not be mixed into this fix.
