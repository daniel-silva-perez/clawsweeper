## Summary

Keep open. Current main still emits full inbound conversation and sender metadata for Telegram/direct-channel messages, and an open implementation PR already targets the same prompt-bloat problem but has not merged. The maintainer cleanup path is to review or land that compact-metadata PR rather than close this issue early.

## What This Changes

Request an optional minimal or compact inbound metadata mode for 1:1 direct chats so repeated Telegram DM messages do not prepend full conversation and sender JSON blocks on every turn.

## Best Possible Solution

Add a backwards-compatible prompt assembly option that compacts direct-chat inbound metadata while preserving full metadata for groups and rich message contexts, then close this issue through the merged implementation PR once tests and docs cover the chosen config contract.

## Reproduction Assessment

Yes. On current main, inspect or call buildInboundUserContextPrefix with ChatType "direct", OriginatingChannel or Provider "telegram", and sender metadata: the gate includes Conversation info and the sender branch appends the full Sender block.

## Solution Assessment

Unclear until the open PR is reviewed. A narrow optional compact-direct mode in prompt assembly looks like the right boundary, but maintainers should settle the config name and verify stripping/display behavior before merging.
