## Summary

Current main still appears to have the reported model-case gap in the local `infer model run` path. Provider IDs are normalized case-insensitively, but the model ID can retain the user-supplied casing through Anthropic dynamic resolution and into the provider call/error text, so this is actionable OpenClaw CLI/provider-resolution work rather than an implemented, stale, or duplicate item.

## Best Possible Solution

Keep this issue open. The best fix is to add provider/catalog-aware model-ref validation or canonicalization before local provider dispatch: either resolve a case-only Anthropic catalog/dynamic match to the canonical provider-owned model ID before calling the provider, or reject the user-supplied case mismatch upfront with `Unknown model: <input>` plus an optional suggestion. Add regression coverage for `anthropic/CLAUDE-OPUS-4-7` and `anthropic/Claude-Opus-4-7`, while preserving existing behavior for intentionally case-sensitive model IDs.
