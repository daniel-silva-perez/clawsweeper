## Summary

Keep open. Current main still preserves the user-supplied model casing on the `infer model run` local and gateway paths, and two open unmerged PRs already reference this issue with closing syntax, so the issue should stay open until maintainers review, land, or close those candidates.

## What This Changes

Reject or canonicalize case-mismatched `infer model run --model <provider/model>` model IDs before provider dispatch so users get a clear model-resolution result instead of a misleading no-output error.

## Best Possible Solution

Review and consolidate #73717 and #74945, then land a catalog-aware fix that canonicalizes unique known model matches before both local and gateway dispatch while preserving or clearly rejecting unmatched custom refs, with regression coverage in `src/cli/capability-cli.test.ts`.

## Reproduction Assessment

Yes. The issue provides a deterministic CLI matrix, and current source inspection confirms the normalization gap without a live provider: local dispatch forwards raw `params.model`, gateway parsing preserves the sliced model casing, and Anthropic runtime resolution can retain that mixed-case model identity.

## Solution Assessment

Unclear until a PR is selected. The narrower maintainable direction is unique catalog-match canonicalization; blanket lowercasing is riskier for custom provider/model IDs that intentionally use mixed case.
