## Summary

Keep open. Current main still has the routed `/status` context-window mismatch, so the PR is not obsolete, but the proposed resolver addition would expose only per-agent overrides and drop inherited `agents.defaults.contextTokens`; that fallback needs to be fixed before merge.

## What This Changes

The PR adds `contextTokens` to resolved agent config, makes `/status` prefer the target session entry's persisted context window, and adds regression tests for both paths.

## Best Possible Solution

Keep the PR focused, update the resolver line to `entry.contextTokens ?? agentDefaults?.contextTokens`, add an inherited-default regression test, and then rerun the author's targeted test, type, and lint commands before maintainer review.

## Reproduction Assessment

Yes. A focused unit path can set `params.contextTokens` to one value and `sessionStore[sessionKey].contextTokens` to another, then verify current main forwards the wrong value to `buildStatusReply`; the resolver finding reproduces with a config that only sets `agents.defaults.contextTokens`.

## Solution Assessment

No, not yet. The `/status` change is the narrow right direction, but the resolver addition should return the effective inherited context-token cap before the new field is safe for callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve inherited context-token caps:** `src/agents/agent-scope-config.ts:133`
  - body: The new resolved config field returns only `entry.contextTokens`. If an agent inherits `agents.defaults.contextTokens`, `resolveAgentConfig(cfg, id)?.contextTokens` becomes `undefined` even though other effective fields fold defaults in, so callers using this new field will drop the configured cap. Use `entry.contextTokens ?? agentDefaults?.contextTokens` and cover the inherited-default case.
  - confidence: 0.86
