## Summary

Keep this PR open. Current main still has the `/status` target-session context-window gap, and the PR moves in the right direction, but the branch is not merge-ready because the new resolved `contextTokens` field drops inherited `agents.defaults.contextTokens`; the user-facing fix also needs the required changelog entry.

## What This Changes

This PR adds `contextTokens` to resolved agent config, changes `/status` routing to pass the target session's persisted context window, and adds regression tests for both paths.

## Best Possible Solution

Keep the targeted `/status` routing fix, change the resolver addition to return the effective cap with `entry.contextTokens ?? agentDefaults?.contextTokens`, add inherited-default regression coverage, and add a single Unreleased/Fixes changelog line while leaving the broader model-limit capping work to #39859.

## Reproduction Assessment

Yes. The current-main status gap is reproducible with a `/status` command whose `sessionStore[sessionKey].contextTokens` differs from `params.contextTokens`; the review finding is reproducible with a config that sets only `agents.defaults.contextTokens` and then reads `resolveAgentConfig(cfg, id)?.contextTokens`.

## Solution Assessment

No, not yet. The `/status` change is the narrow maintainable direction, but the resolver field must preserve inherited defaults before callers can safely treat it as the effective context-window cap.
