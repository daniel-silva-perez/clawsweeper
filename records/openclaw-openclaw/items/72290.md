## Summary

Keep open. Current main still lacks the gateway/user-message protections this PR is trying to add, and the linked regression issue remains open, but the branch is currently dirty against main and the proposed inbound fallback does not cover the stated retry case where each attempt has a regenerated MessageSid.

## What This Changes

The PR adds a content-hash fallback to inbound dedupe, threads clientRunId into chat.send user transcript messages as an idempotencyKey, and reorders Control UI chat stream clearing with a regression test.

## Best Possible Solution

Rebase the gateway fix onto current main, preserve the current route-key helper contract, add a retry-safe dedupe path for missing or unstable message IDs without suppressing legitimate repeated messages, and cover it with focused inbound/chat.send tests. The Control UI ordering fix should either be split to #72004 or intentionally chosen as the canonical UI PR.

## Reproduction Assessment

Yes. The code-level reproduction is clear: current main returns no inbound dedupe key without MessageSid, and the PR diff still returns the primary key immediately whenever MessageSid exists, which misses the reported regenerated-id retry path.

## Solution Assessment

No, not as-is. The general direction is plausible, but the branch needs a current-main rebase, a dedupe design that handles regenerated IDs, and proof that transcript idempotency is enforced by an actual append/broadcast path.

If secondary user-message dedupe is required, wire it through the append/SSE path that actually checks for existing keys.
