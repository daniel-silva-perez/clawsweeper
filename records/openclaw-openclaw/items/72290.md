## Summary

Keep this PR open. Current main still lacks the intended inbound no-stable-ID fallback and user transcript idempotency, but the current branch is not merge-ready: the proposed fallback misses regenerated MessageSid/idempotencyKey retries, the transcript change does not invoke an actual transcript dedupe path, and the Control UI part overlaps the separate open #72004 UI fix.

## What This Changes

The PR adds a content-hash fallback to inbound dedupe, adds an idempotencyKey field to chat.send user transcript update payloads, and reorders Control UI stream clearing with a regression test.

## Best Possible Solution

Rebase onto current main, preserve the current route-key helper contract, implement a retry-safe inbound dedupe strategy for missing or unstable message IDs, wire user transcript idempotency through an append path that actually checks existing keys, and resolve the Control UI fix through either this PR or #72004, not both.

## Reproduction Assessment

Yes. A code-level reproduction is clear: two otherwise identical inbound contexts with different MessageSid/clientRunId values still take the PR's messageId fast path and produce different keys, while emitSessionTranscriptUpdate does not call transcriptHasIdempotencyKey.

## Solution Assessment

No, not as-is. The direction is plausible, but the current patch does not solve the regenerated-ID path described in #72176 and the transcript idempotency change is attached to a broadcast-only path rather than the duplicate-suppressing append path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Deduplicate regenerated message IDs:** `src/auto-reply/reply/inbound-dedupe.ts:75-77`
  - body: The fallback only runs when `messageId` is absent. The regression described here includes retries that generate a new idempotency key or MessageSid per attempt, and `chat.send` copies `clientRunId` into `ctx.MessageSid`, so those retries still return distinct primary keys and both run. Gate the fast path to truly stable provider IDs or add a retry-aware fallback before returning.
  - confidence: 0.86
- **[P2] Use a transcript append dedupe path:** `src/gateway/server-methods/chat.ts:767`
  - body: This only adds `idempotencyKey` to the transcript update broadcast payload. `emitSessionTranscriptUpdate` normalizes and notifies listeners, while `transcriptHasIdempotencyKey` is only checked by `appendAssistantTranscriptMessage`, so duplicate user transcript writes are not suppressed. Wire user-message persistence through a path that actually checks existing idempotency keys.
  - confidence: 0.78
