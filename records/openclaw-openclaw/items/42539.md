## Summary

Keep open. Current main still lacks a configurable Telegram TTS delivery mode: TTS attaches generated audio to the same reply payload, and Telegram still treats the visible text as a media/voice caption except for the existing caption-overflow or sendVoice-error fallback paths.

## What This Changes

Add a configurable Telegram TTS delivery mode that sends readable text as a normal message first and the generated voice note as a separate follow-up instead of bundling the text as a voice caption.

## Best Possible Solution

Keep this issue open and add a backwards-compatible delivery policy for TTS replies. In separate mode, Telegram should send the visible text as a normal message first and then send the voice note without a caption, with schema/types/docs and focused Telegram tests covering the bundled default and separate behavior.

## Reproduction Assessment

Yes. A read-only current-main trace is enough: messages.tts.delivery is absent from strict config, TTS returns one media-bearing ReplyPayload, and Telegram sends reply.text as the voice media caption by default.

## Solution Assessment

Unclear until maintainers choose the public config boundary. The requested text-first/voice-second behavior is maintainable, but the safest implementation path needs an API/product decision before an automated fix PR.
