## Summary

Keep this issue open. Current main still lacks the requested Telegram TTS delivery-mode option: the shared TTS config contract has no `delivery`/`separate` field, auto-TTS still returns one reply payload containing both visible text and generated audio, and Telegram still uses normal-length reply text as the voice/audio caption unless it exceeds Telegram’s caption limit. The existing overflow fallback and prior ClawSweeper comment are relevant context, but they do not implement a configurable default separate-text-plus-voice mode.

## Best Possible Solution

Keep this issue open. Implement a Telegram-owned delivery policy or shared `messages.tts.delivery` contract with `bundled` and `separate` modes, update strict config types/schema/generated docs, and add focused Telegram bot plus direct-send tests. In `separate` mode, Telegram should send visible text as a normal message first, then send the voice note without a caption.
