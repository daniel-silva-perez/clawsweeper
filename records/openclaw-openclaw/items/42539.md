## Summary

Keep this issue open. Current main still lacks a Telegram TTS delivery-mode option: shared TTS config has no `delivery`/`separate` field, auto-TTS still produces one reply payload containing both visible text and audio, and Telegram still uses normal-length reply text as the voice caption unless the caption overflow path is triggered.

## Best Possible Solution

Keep this issue open. Implement a Telegram-owned delivery policy or shared `messages.tts.delivery` contract with `bundled` and `separate` modes, update strict config types/schema/generated docs, and add focused Telegram bot/direct-send tests. In `separate` mode, Telegram should send visible text as a normal message first, then send the voice note without a caption.
