## Summary

Keep this issue open. Current main still has no shared or Telegram-specific TTS delivery mode for the requested default of sending readable text first and the voice note second. The implementation still produces one reply payload with text plus audio media, and Telegram still sends that text as a media caption unless the existing over-1024-caption fallback or sendVoice error fallback is triggered.

## Best Possible Solution

Keep this issue open. Add a Telegram-owned delivery policy or shared messages.tts.delivery contract with bundled and separate modes, update strict config types/schema/docs, and add focused Telegram tests. In separate mode, Telegram should send the visible text as a normal message first, then send the voice note without a caption.
