## Summary

Keep this issue open. Current main still lacks a shared or Telegram-specific TTS delivery mode for sending readable text first and a captionless voice note second; normal Telegram voice TTS still bundles the reply text as a media caption unless the existing long-caption or sendVoice-error fallback path is triggered.

## Best Possible Solution

Keep this issue open. Add a Telegram-owned delivery policy or shared messages.tts.delivery contract with bundled and separate modes, update strict config types/schema/docs, and add focused Telegram tests. In separate mode, Telegram should send the visible text as a normal message first, then send the voice note without a caption.
