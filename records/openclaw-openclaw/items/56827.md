## Summary

Keep open. Current main still has the two reported unresolved surfaces: `doctor` eagerly reads and parses the whole session store before state-integrity checks, and the inspected doctor/session cleanup paths do not detect or remove stale `sessions.json.*.tmp` atomic-write leftovers. The report is recent and concrete, so the stale label should not drive closure.

## What This Changes

The issue asks OpenClaw to make `openclaw doctor` complete on very large long-lived session stores and to handle stale `sessions.json.*.tmp` files that can accumulate beside `sessions.json`.

## Best Possible Solution

Harden the existing session diagnostics first: make `doctor` cheap and non-fatal on oversized stores, detect and safely clean stale same-directory session-store temp files, and include those artifacts in session cleanup/disk-budget maintenance. Treat a SQLite-backed session metadata store as a separate design follow-up rather than the first required repair.

Unclear. The report provides concrete store sizes and current source still follows the heavy full-store path, but this read-only review did not execute an end-to-end large-store fixture or reproduce the exact V8 stack.

## Solution Assessment

Yes for a narrow first fix, not for an immediate SQLite rewrite. The maintainable first step is targeted doctor/session-cleanup hardening around oversized stores and stale atomic-write temp files; a storage migration should be tracked separately if maintainers want that broader redesign.

## Repair Work Prompt

Fix the large session-store doctor failure and stale `sessions.json.*.tmp` accumulation reported in https://github.com/openclaw/openclaw/issues/56827. Implement a narrow first repair: make `openclaw doctor` tolerate oversized `sessions.json` without fatal eager parsing where possible; detect and safely report/remove stale same-directory `sessions.json.<uuid>.tmp` files left by atomic writes; include those temp artifacts in `openclaw sessions cleanup` and disk-budget maintenance. Use conservative guards so active writes are not deleted: only match the exact store basename pattern in the same sessions directory, require an age threshold, and keep dry-run/reporting behavior clear. Do not start a SQLite migration or broad session-storage redesign in this PR. Update focused tests for doctor state integrity, sessions cleanup, and disk budget, and update docs/changelog if user-visible output or cleanup behavior changes.
