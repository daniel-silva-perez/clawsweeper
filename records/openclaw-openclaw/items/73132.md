## Summary

Keep open. Current main still lets the macOS LaunchAgent stop path return after launchctl/bootout state changes without resolving the gateway port or verifying that the underlying listener process is gone. The loaded-service stop path also bypasses the unmanaged PID cleanup, so the report remains a valid, narrow macOS lifecycle bug distinct from the already-fixed Windows case.

## What This Changes

The issue asks for `openclaw gateway stop` on macOS to terminate lingering gateway processes and verify the gateway port is free before reporting success.

## Best Possible Solution

Extend the macOS managed stop path to resolve the service gateway port, wait for the listener to disappear after launchctl stop or bootout, terminate only verified lingering OpenClaw gateway processes with graceful-then-force cleanup, and return a failure if the port remains occupied.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps, and current source confirms a matching path where degraded bootout can return without gateway PID or port verification; live launchd validation should still be part of the eventual fix.

## Solution Assessment

No. Current main is not sufficient; the narrow maintainable fix is to add macOS post-stop listener verification and cleanup analogous to the Windows Scheduled Task stop behavior while preserving existing LaunchAgent disable/start recovery semantics.
