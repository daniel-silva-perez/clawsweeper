## Summary

Keep open. Current main still has the reported macOS managed-stop gap: `openclaw gateway stop` can finish the LaunchAgent stop/bootout path without verifying that the OpenClaw gateway listener exited or that the configured port is free.

## What This Changes

The issue reports that `openclaw gateway stop` on macOS can return success while the underlying gateway node process keeps the gateway port bound.

## Best Possible Solution

Add a macOS managed-stop postcondition: after `launchctl stop` or bootout, resolve the configured gateway port, wait for the verified OpenClaw listener to exit, escalate from graceful termination only for verified lingering gateway PIDs, and return a failure if the port remains occupied or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps, and current source confirms the loaded LaunchAgent stop path can return without listener or port verification; live macOS validation remains the final acceptance check.

## Solution Assessment

Yes. A narrow macOS lifecycle postcondition is the best fit; it should reuse existing verified gateway-process and port-inspection helpers rather than becoming a launchd redesign or an arbitrary `node` process killer.
