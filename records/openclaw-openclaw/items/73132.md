## Summary

Keep open. Current main still has a macOS managed-stop gap: the LaunchAgent path can report a degraded stop after launchctl/bootout state changes without verifying that the OpenClaw gateway listener process is gone or that the port is free. The report is recent, concrete, unpaired with an open PR, and analogous Windows cleanup code already exists as a model for a focused fix.

## What This Changes

Request that macOS `openclaw gateway stop` fully terminate any lingering OpenClaw gateway process and fail if the gateway port remains occupied.

## Best Possible Solution

Add a macOS managed-stop postcondition: after launchctl stop or bootout, resolve the configured gateway port, wait for the OpenClaw listener to exit, gracefully terminate verified lingering gateway PIDs, force them only after a short grace period, and return a clear non-zero failure if the port remains occupied or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps and current source confirms the matching path: a degraded LaunchAgent stop can return without PID or port verification. Final proof should include a macOS launchd run because this review only performed source and history inspection.

## Solution Assessment

Yes. Extending the macOS stop path to reuse verified gateway-process and port-release helpers is the narrow maintainable fix; it should not become a broad launchd redesign or arbitrary `node` process killer.
