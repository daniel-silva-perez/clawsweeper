## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the loaded LaunchAgent path can report a successful or degraded stop after launchctl state changes without verifying that the actual OpenClaw gateway listener exited or that the configured port is free.

## What This Changes

The issue requests that `openclaw gateway stop` on macOS terminate any verified lingering OpenClaw gateway process, verify port release, and fail clearly if the gateway is still running.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the configured gateway port, waits for the OpenClaw listener to exit after launchctl stop or bootout, terminates only verified lingering gateway PIDs with a short graceful window, and exits non-zero if the port remains occupied or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps and current source confirms the loaded LaunchAgent path can return without PID or port verification; final proof should include a macOS launchd smoke run.

## Solution Assessment

Yes, the requested direction is the right fix, with one constraint: use the existing verified gateway-process and port-inspection helpers rather than a broad `kill -9 $(pgrep node)` style cleanup.
