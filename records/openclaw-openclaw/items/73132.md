## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the loaded LaunchAgent stop path can return after launchctl stop/bootout state changes without verifying that the OpenClaw gateway listener process exited or that the configured port is free. The issue is recent, concrete, unprotected, and narrow enough for a focused repair.

## What This Changes

The issue requests that macOS `openclaw gateway stop` terminate or fail on lingering gateway listener processes instead of reporting a degraded stop while the port remains bound.

## Best Possible Solution

Add a macOS managed-stop postcondition: after launchctl stop or bootout, resolve the configured gateway port, wait for the OpenClaw listener to exit, terminate only verified lingering gateway PIDs with a short graceful window, and fail clearly if the port remains occupied or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source confirms the loaded LaunchAgent path can return without listener or port verification. Final proof should include a macOS launchd run because this review was read-only source inspection.

## Solution Assessment

Yes. The best fix is a narrow macOS lifecycle postcondition that reuses verified gateway-process and port-release helpers; it should not become a launchd redesign or an arbitrary `node` process killer.
