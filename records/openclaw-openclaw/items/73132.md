## Summary

Keep open. Current main hardens launchd state handling and restart paths, but macOS `gateway stop` still trusts launchctl/bootout state and does not verify or clear the underlying gateway listener before reporting success. The report is concrete, recent, and narrow enough for a focused fix PR.

## What This Changes

The issue reports that `openclaw gateway stop` on macOS can report a degraded LaunchAgent stop while leaving the underlying gateway process and port alive.

## Best Possible Solution

Extend the macOS managed stop path so it resolves the service's gateway port, waits for the listener to disappear after launchctl stop/bootout, terminates verified lingering gateway processes with a graceful-then-force path, and returns non-zero if the port remains occupied. Preserve the existing LaunchAgent disable/start/restart recovery semantics.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source confirms a matching failure path: degraded bootout can return without any gateway PID or port verification. A final fix should still be validated on macOS launchd because this worker cannot exercise LaunchAgent behavior live.

## Solution Assessment

No, current main is not the best solution yet. The suggested SIGTERM/SIGKILL plus port-free verification matches the existing Windows stop and restart cleanup direction, but it should be implemented through verified gateway-process helpers and fail closed when the remaining listener is not safe to kill.

## Repair Work Prompt

Fix the macOS `openclaw gateway stop` bug reported at https://github.com/openclaw/openclaw/issues/73132: a degraded LaunchAgent stop/bootout can leave the underlying `openclaw-gateway` process and listener alive while the CLI reports success. Keep the change scoped to gateway service lifecycle. Resolve the managed gateway port from LaunchAgent program args/env/config/profile as appropriate, and after `launchctl stop` or degraded `bootout`, verify the listener is gone before returning success. Terminate only verified OpenClaw gateway processes, using graceful termination first and a force path if needed; if the port remains busy or belongs to an unverified process, return a non-zero error with clear diagnostics. Preserve existing LaunchAgent disable semantics, start/restart recovery, and Windows/systemd behavior. It is fine to extract shared stop-cleanup helpers if that avoids duplicating the Windows Scheduled Task logic. Add focused tests for degraded bootout leaving a listener, successful cleanup releasing the port, and failure when the port remains occupied. If user-visible stop text changes, update the CLI gateway docs and add a changelog fix entry crediting @Adam-Researchh; otherwise keep docs/changelog untouched.
