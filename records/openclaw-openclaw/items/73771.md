## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the PR's central agent namespace isolation for the builtin memory index and QMD mcporter state, while the PR branch is an active stacked implementation with a blocking package-copy migration bug and additional performance regressions to resolve.

## What This Changes

The PR branch adds agent namespace scoping to memory schema/index read-write-search paths, memory_search scope/provenance handling, QMD mcporter daemon state, memory CLI provider registration, and related regression tests.

## Best Possible Solution

Rebase the stacked memory series onto current main, keep packages/memory-host-sdk as the schema source of truth, fix the package FTS rebuild check to use Set.size, preserve bounded/yielding large-index behavior, and then validate shared store.path isolation across schema migration, writes, deletes, keyword search, vector search, status, file state, and QMD mcporter daemon state.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main shared SQLite memory stores are unscoped by agent_id, and on the PR branch an existing package FTS table without agent_id will not rebuild because Set.length is undefined, causing later FTS writes with agent_id to fail.

## Solution Assessment

No, not as-is. The direction is the right owner-area fix for memory isolation, but the current branch should be rebased and corrected before merge, with the package migration bug and large-index regressions fixed first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use Set.size for the FTS rebuild check:** `packages/memory-host-sdk/src/host/memory-schema.ts:46`
  - body: `readColumns` returns a `Set`, so `existingFtsColumns.length` is always `undefined` and this condition never rebuilds old FTS tables that lack `agent_id`. Existing package-backed databases will keep the old virtual table, and later inserts that include `agent_id` will fail after upgrade.
  - confidence: 0.98
- **[P2] Keep fallback vector scoring bounded:** `extensions/memory-core/src/memory/manager-search.ts:254`
  - body: The fallback vector path now loads every matching chunk with `.all()`, scores all rows, sorts the full array, and then slices. Current main uses an iterator with a bounded top-K accumulator, so this regresses large indexes by materializing all chunks and embeddings in memory.
  - confidence: 0.9
- **[P2] Preserve yielding during cache seeding:** `extensions/memory-core/src/memory/manager-sync-ops.ts:319`
  - body: Changing cache seeding to `.all()` plus a fully synchronous loop removes the existing `setImmediate` yield during large rebuilds. Large embedding caches can now block the event loop long enough to delay gateway health probes and other async work.
  - confidence: 0.86
