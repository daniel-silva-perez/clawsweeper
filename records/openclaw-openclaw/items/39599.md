## Summary

Keep open. Current main still permits an announce cron delivery object for Feishu without `delivery.to` at the protocol, gateway validation, service validation, and CLI creation layers, and there are open PRs that explicitly target this issue, so the issue should wait for one canonical fix to merge rather than be closed early.

## What This Changes

The issue requests create-time validation so Feishu/Lark announce cron jobs without an explicit or deterministically resolved delivery target are rejected instead of failing at runtime.

## Best Possible Solution

Land one focused fix that validates Feishu/Lark announce cron delivery on create and update when neither explicit `delivery.to` nor a deterministic resolved target exists, with regression coverage at the gateway/service boundary and CLI/UI feedback kept as a user-experience layer.
