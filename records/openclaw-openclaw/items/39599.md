## Summary

Keep open. Current main still accepts a Feishu announce cron delivery without `delivery.to` at the protocol, gateway validation, service validation, and CLI creation layers. Two open PRs still explicitly target this issue, so they are implementation candidates rather than a reason to close it now.

## Best Possible Solution

Keep this open and land one canonical focused fix. Add create/update validation for Feishu/Lark announce cron delivery when neither explicit `delivery.to` nor deterministic resolved target exists, cover the cron service and gateway boundary, and optionally add CLI/UI fail-fast feedback without changing valid implicit-target flows for other channels.
