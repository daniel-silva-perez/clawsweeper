## Summary

Keep this issue open. Current main still permits a Feishu/Lark announce cron delivery object without `delivery.to` through the protocol, gateway, service, and CLI creation path, and the open PRs that reference this issue are implementation candidates rather than merged fixes.

## Best Possible Solution

Keep this issue open and land one focused canonical fix. Add create/update validation for Feishu/Lark announce cron delivery when neither explicit `delivery.to` nor a deterministic resolved target exists, with regression coverage at the cron service and gateway boundary and optional CLI/UI fail-fast feedback.
