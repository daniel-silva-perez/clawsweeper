## Summary

Keep this PR open. Current `main` still drops the reusable CLI `sessionId` on `auth-profile` and same-version `auth-epoch` mismatches, so the PR's requested behavior is not implemented on `main`; however the proposed auth reuse is too broad against current auth-epoch semantics and needs maintainer review before merge.

## What This Changes

The PR changes CLI session reuse so auth-profile/auth-epoch invalidations keep the existing CLI session id, prompt/MCP invalidations still force fresh sessions first, reset logging distinguishes reauth from reset, and helper tests cover the new behavior.

## Best Possible Solution

Implement the session-preservation behavior narrowly on current `main`: preserve native CLI resume for credential refresh or safe profile metadata rotation, keep prompt/MCP changes on fresh sessions, and keep identity/account/profile switches on fresh sessions unless a current-main contract proves they are the same principal. Cover token refresh, explicit profile switch, auth-epoch identity change, and MCP resume-hash cases in focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call `resolveCliSessionReuse` with a binding that has `sessionId` and matching prompt/MCP hashes but a changed `authProfileId` or same-version changed `authEpoch`; current `main` returns only `invalidatedReason`, and the runner therefore does not use native CLI resume.

## Solution Assessment

No. The PR fixes the context-loss mechanism, but after current `main` made auth epochs identity-oriented, resuming on every auth-profile/auth-epoch mismatch is not the narrowest maintainable solution; it should distinguish refresh-only changes from identity/profile switches.
