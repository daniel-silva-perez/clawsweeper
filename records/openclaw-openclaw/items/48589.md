## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the stable TUI affinity behavior, so the work is not obsolete, but the proposed patch has concrete routing bugs around current-main TUI client identity and explicit session selection.

## What This Changes

The PR adds stable TUI instance IDs, TUI affinity session keys, chat-origin affinity metadata, and orphan short-reply reattachment so a Gateway TUI main session can continue across restarts.

## Best Possible Solution

Rebase the branch onto current main, preserve current TUI client identity, synthesize affinity metadata for the real TUI client path, and restrict orphan short-reply recovery to default TUI main-session aliases with regression coverage for explicit session switches.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main's GatewayChatClient sends the TUI client id, while the PR's new chat-origin fallback checks gateway-client; separately, a missing explicit session key plus a short reply and a recent same-affinity awaiting session will enter the reattach block at line 508.

## Solution Assessment

No. The stable per-terminal main-session direction is reasonable, but the current patch is not the narrowest maintainable implementation because it applies reattachment to arbitrary missing keys and misses the current TUI client id used on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Match the current TUI client id for affinity origin:** `src/gateway/server-methods/chat.ts:1253-1254`
  - body: Current main registers the gateway TUI as `GATEWAY_CLIENT_NAMES.TUI`, but this fallback only builds `session:tui:<id>` when the client id is `GATEWAY_CLIENT_NAMES.GATEWAY_CLIENT`. After rebasing or merging onto current main, TUI sends will not persist the affinity origin metadata needed by the new reattach path, so the recovery mechanism will not work for the actual TUI client.
  - confidence: 0.86
- **[P1] Gate orphan reattach to default TUI main aliases:** `src/auto-reply/reply/session.ts:508-509`
  - body: This block runs for any missing session key, so a user who intentionally switches to a new explicit session and sends a short first message like `yes` can be silently moved back to a recent same-affinity TUI session awaiting confirmation. Limit the recovery to default or auto main-session aliases, and add a regression where `/session project-x` followed by `yes` stays in `agent:main:project-x`.
  - confidence: 0.92
