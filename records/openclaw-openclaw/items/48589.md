## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested stable TUI main-session continuity, and the proposed patch has blocking routing defects: it checks the wrong gateway client id for the actual TUI path and can reattach short replies from an explicit new session into an older TUI session.

## What This Changes

The PR adds stable TUI client instance IDs, TUI affinity session keys, gateway-origin synthesis, and orphan short-reply reattachment logic/tests to preserve gateway-mode TUI main-session continuity across reconnects.

## Best Possible Solution

Preserve the feature direction, but revise the PR before merge: key the origin synthesis to the actual TUI client identity, gate orphan short-reply reattachment to default/main TUI aliases only, add regressions for real TUI client id and explicit /session no-hijack behavior, then rebase and rerun the targeted TUI/gateway/session tests plus the changed gate.

## Reproduction Assessment

Yes for the PR defects. The static reproduction is direct: current TUI connects as openclaw-tui while the PR fallback checks gateway-client, and the orphan reattach branch only checks that the resolved session entry is missing, so an explicit missing session key plus a short yes/ok can be reassigned to another recent TUI session with the same OriginatingTo.

## Solution Assessment

No. The requested continuity feature is reasonable, but this implementation is not the safest maintainable fix until it uses the actual TUI client contract and narrows orphan reply recovery to default main-session aliases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Match the actual TUI client id before building affinity:** `src/gateway/server-methods/chat.ts:1250-1251`
  - body: `GatewayChatClient` registers the TUI as `GATEWAY_CLIENT_NAMES.TUI`, but this new fallback only runs for `GATEWAY_CLIENT_NAMES.GATEWAY_CLIENT`. Actual gateway-mode TUI sends therefore never receive the synthetic `session:tui:...` origin, so the new affinity and orphan-reattach path are bypassed; the added test uses the wrong client id and misses this.
  - confidence: 0.95
- **[P1] Gate orphan reattach to default main aliases:** `src/auto-reply/reply/session.ts:508-512`
  - body: This recovery runs for any missing session key, so an explicit switch such as `/session project-x` can have its first short `yes`/`ok` reply silently reassigned to an older awaiting TUI session with the same origin. That writes the reply to the wrong transcript and defeats the user's explicit session choice; limit this branch to default/main TUI aliases.
  - confidence: 0.93
