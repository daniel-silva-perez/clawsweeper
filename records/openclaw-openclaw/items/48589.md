## Summary

Keep this PR open. Current main still lacks stable TUI main-session affinity, and the branch is a useful implementation direction, but it has two correctness blockers that can route real TUI traffic incorrectly or override an explicit session switch.

## What This Changes

The branch adds persistent TUI instance IDs, TUI affinity session keys/origin metadata, and orphan short-reply reattachment tests for Gateway TUI main-session continuity.

## Best Possible Solution

Rebase the branch onto current main, keep the stable per-terminal TUI main-session direction, synthesize affinity metadata for the real current TUI client id, and restrict orphan short-reply recovery to default/main TUI aliases with explicit-session regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main sends TUI traffic with GATEWAY_CLIENT_NAMES.TUI while the PR fallback checks GATEWAY_CLIENT_NAMES.GATEWAY_CLIENT; separately, an explicit missing session key with a short first message can enter the PR's reattach block and be reassigned to a recent same-affinity session.

## Solution Assessment

No. The overall product direction is reasonable, but the proposed patch is not yet the best fix because it misses the current TUI client id and applies reattachment to arbitrary missing session keys instead of only default TUI main aliases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Match the current TUI client id for affinity origin:** `src/gateway/server-methods/chat.ts:1253-1254`
  - body: Current main registers the Gateway TUI as `GATEWAY_CLIENT_NAMES.TUI`, but this fallback only builds `session:tui:<id>` when the client id is `GATEWAY_CLIENT_NAMES.GATEWAY_CLIENT`. After rebasing onto current main, real TUI sends will not populate the affinity origin metadata needed by the new reattach path.
  - confidence: 0.86
- **[P1] Gate orphan reattach to default TUI main aliases:** `src/auto-reply/reply/session.ts:508-509`
  - body: This block runs for any missing session key, so a user who intentionally switches to a new explicit session and sends a short first message like `yes` can be silently moved back to a recent same-affinity TUI session awaiting confirmation. Limit this recovery to default/main TUI aliases and cover an explicit session switch regression.
  - confidence: 0.92
