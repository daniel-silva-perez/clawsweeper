## Summary

Keep this PR open. Current main does not implement the central TUI continuity behavior: the default TUI session still maps to the normal agent main key, the TUI gateway client still sends a fresh random instanceId per process, and there is no client-affinity module or orphan short-reply reattach path. The PR remains potentially useful, but it needs rebase/review against current client naming and the unresolved misrouting concern before it could land.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main, align the affinity detection with the current `openclaw-tui` client identity, keep the stable per-terminal instance ID scoped to the TUI, gate orphan short-reply reattachment to default TUI main-session aliases only, and retain focused tests for default affinity, wrong-client guards, and explicit-session non-reattachment.

## Risks / Open Questions

- The PR's orphan short-reply reattachment path needs a guard so it only applies to default/auto main-session aliases, not arbitrary explicit session keys.
- The PR patch appears to key one server-side affinity fallback on `GATEWAY_CLIENT_NAMES.GATEWAY_CLIENT`, while current main's TUI identifies as `GATEWAY_CLIENT_NAMES.TUI` with mode `ui`; that mismatch should be resolved during rebase.
- Security/supply-chain review found no workflow, dependency, lockfile, release, or install-script changes in the provided PR file list; the main security-relevant risk is session confidentiality/integrity if short replies are reattached to the wrong transcript.
