## Summary

Keep open. Current main has inbound debouncing, but the shared policy and common channel handlers still exclude media-bearing messages, while QQ Bot only merges already-queued group bursts and does not expose a configurable quiet window for image-plus-text turns.

## What This Changes

The issue requests configurable per-channel delayed aggregation so multi-part user inputs such as image followed by text are delivered to the agent as one conversation turn.

## Best Possible Solution

Add a media-aware inbound aggregation path using the existing `messages.inbound`/debouncer foundation or a clearly documented extension to it. The implementation should batch by channel, conversation, and sender; preserve text plus images/media/local paths; use the latest message for reply identity; keep command handling explicit; and include focused QQ Bot coverage plus one shared-channel contract or integration test.

## Reproduction Assessment

Yes. A static reproduction path is clear: send a media-bearing message followed by text on a shared-policy channel such as Discord, Slack, Signal, or iMessage, or a QQ Bot direct message; current code either bypasses debounce due to `hasMedia` or drains the QQ queue immediately instead of waiting for a quiet window.

## Solution Assessment

Unclear. The requested per-channel delay is directionally sound, but the maintainable path should probably extend the existing `messages.inbound` contract rather than add a QQ-only `messageAggregationDelayMs` unless maintainers want a channel-local config surface.
