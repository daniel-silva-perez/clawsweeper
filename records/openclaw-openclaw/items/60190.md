## Summary

Keep open. Current main has inbound debouncing, but it is explicitly text-only for the shared policy and common channel integrations, while QQ Bot only merges already queued group bursts instead of waiting a configurable quiet window for image+text multi-part turns. Narrow precedents exist, such as BlueBubbles split-send DM coalescing and Telegram forwarded-burst coalescing, but the requested general configurable multi-part media/text aggregation is not implemented.

## Best Possible Solution

Implement a media-aware inbound aggregation path on top of `messages.inbound` and `createInboundDebouncer`, likely as a shared SDK/channel helper plus opt-in per-channel defaults. It should batch by channel, conversation, and sender; preserve combined text plus images/media/local paths; use the latest message for reply threading; keep control-command bypass/flush rules explicit; and add focused tests/docs for QQ Bot plus at least one shared channel integration.

## Risks / Open Questions

- A broad default delay could add unwanted latency to command-heavy channels; this should stay opt-in or tightly scoped per channel/conversation.
- Existing channel-specific behavior, especially BlueBubbles split-send DM coalescing and Telegram forwarded-burst coalescing, should be preserved rather than replaced blindly.
