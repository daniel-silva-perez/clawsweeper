## Summary

Keep open. Current main still has the endpoint hardcoding this PR is trying to remove, so the work is not obsolete; however, the branch is stale against current xAI config ownership and still has blocking integration issues before it should merge.

## What This Changes

The PR adds configurable baseUrl handling, metadata/generated config updates, and tests for Gemini web_search, xAI Grok web_search, and xAI x_search.

## Best Possible Solution

Rebase or split the work so Gemini, Grok web_search, and x_search each use current plugin-owned config contracts, preserve legacy config fallbacks where existing installs already work, include baseUrl in cache keys, update public docs/generated metadata, and coordinate the Gemini slice with #57496.

## Reproduction Assessment

Yes. Code inspection on current main reproduces the missing behavior: Gemini, Grok web_search, and x_search still construct requests from fixed default endpoints, and the PR review findings are reproducible by comparing the diff to the current plugin-owned xSearch config path and legacy Grok scoped config path.

## Solution Assessment

No. The direction is maintainable, but this PR is not yet the best fix because it adds x_search.baseUrl to a legacy tools path, misses one legacy Grok fallback, and overlaps the separate Gemini provider-fallback implementation in #57496.

- **[P2] Move x_search baseUrl into plugin config:** `src/config/types.tools.ts:520-522`

  - confidence: 0.87

  - body: The new `x_search` resolver falls back from explicit x_search config to `plugins.entries.xai.config.webSearch.baseUrl`, but skips `tools.web.search.grok.baseUrl`. Existing installs that proxy Grok through the legacy scoped search config would still send `x_search` to the default xAI endpoint, recreating the inconsistency this PR is meant to fix.
  - confidence: 0.82
