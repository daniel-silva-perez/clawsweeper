## Summary

Keep open. Current main still hardcodes the Gemini, Grok web_search, and x_search endpoints, so the requested behavior is not implemented on main; however, the PR needs maintainer review because it adds new x_search config under the legacy core path and misses an existing Grok baseUrl fallback path.

## What This Changes

The PR adds configurable baseUrl handling, metadata, generated config schema updates, and tests for Gemini web_search, xAI Grok web_search, and xAI x_search.

## Best Possible Solution

Rebase or split the PR so Gemini, Grok web_search, and x_search each use the current plugin-owned config contracts, keep backward-compatible legacy fallbacks where existing configs already work, include baseUrl in cache keys, update the public docs, and coordinate the Gemini portion with #57496.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move x_search baseUrl into plugin config:** `src/config/types.tools.ts:520-522`
  - body: `x_search` is now configured under `plugins.entries.xai.config.xSearch.*`, while `tools.web.x_search.*` is legacy and doctor-migrated. Adding the new public override only to `tools.web.x_search.baseUrl` leaves the canonical plugin schema/UI unable to save `xSearch.baseUrl`, so users following current docs still cannot configure a per-tool x_search endpoint.
  - confidence: 0.87
- **[P2] Preserve the legacy Grok baseUrl fallback:** `extensions/xai/x-search.ts:57-68`
  - body: The new x_search resolver falls back from explicit x_search config to `plugins.entries.xai.config.webSearch.baseUrl`, but skips `tools.web.search.grok.baseUrl`. A config that makes Grok `web_search` use a proxy through the legacy scoped Grok setting would still send `x_search` to the default xAI endpoint, recreating the inconsistency this PR is meant to fix.
  - confidence: 0.82
