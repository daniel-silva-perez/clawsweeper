## Summary

Keep open. Current main still has several token-limit resolution paths that prefer configured or requested values before lower runtime/discovery/model caps, so the requested conservative minimum behavior is not implemented end to end.

## What This Changes

Use the minimum positive available context and output token limits from agent config, provider/model config, model discovery/runtime metadata, and defaults.

## Best Possible Solution

Add a shared token-limit candidate resolver that normalizes positive finite limits from agent defaults/per-agent config, provider/model config, discovery/runtime metadata, and fallback defaults, then applies the conservative minimum in the context-window guard, session context persistence, model merge/runtime normalization, and provider maxTokens request paths. Update docs and schema help to define the new contract.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure or request a token value above a lower runtime/discovery/model limit, and current main can still return the higher configured/requested value in contextTokens and maxTokens paths before the lower cap is considered.

## Solution Assessment

Yes. The requested minimum-of-available-limits behavior is the narrow maintainable direction, but it should be implemented through shared candidate collection and provider-qualified lookup handling rather than scattered Math.min calls.
