## Summary

Keep open. Current main still has several precedence paths that can return a configured or requested token value before lower runtime/discovery/model limits, so the requested conservative minimum behavior is not implemented end to end.

## What This Changes

Request to resolve `contextTokens` and `maxTokens` as the minimum positive finite value across agent config, provider model config, model discovery/runtime metadata, and defaults.

## Best Possible Solution

Introduce a shared token-limit resolver that normalizes positive finite candidates from agent defaults/per-agent config, provider/model config, runtime discovery/cache metadata, and fallback defaults, then chooses the conservative minimum while preserving provider-qualified lookup semantics and explicit long-context opt-ins. Apply the same resolver pattern to output `maxTokens` where runtime request params meet model/provider caps, and update tests plus docs to define the contract.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure a provider model context above discovered runtime metadata, or pass a per-agent `contextTokens` override through session persistence, and current code can return the higher override before lower model/discovery limits.

## Solution Assessment

Yes. The minimum-based direction is the narrow maintainable fix, but it should be implemented as normalized candidate collection rather than a bare `Math.min` over ambiguous model IDs.
