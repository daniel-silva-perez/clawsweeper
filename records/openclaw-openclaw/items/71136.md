## Summary

Keep open. Current main still does not implement conservative minimum token-limit resolution across the reported surfaces: contextTokens can still prefer an explicit override or configured provider value over a lower discovered/runtime value, and maxTokens handling remains inconsistent across providers.

## What This Changes

The issue requests resolving effective contextTokens and maxTokens as the minimum of agent config, provider model config, model discovery/runtime metadata, and the default limit.

## Best Possible Solution

Introduce a shared token-limit candidate resolver that normalizes positive finite limits from agent defaults/per-agent config, provider model config, runtime/discovery metadata, and fallback defaults, then use it consistently for context-window guards, session persistence, embedded-runner model metadata, model merge/runtime normalization, and provider maxTokens requests. Update tests and docs to define the conservative cap contract while preserving provider-qualified lookup safety.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure or request contextTokens or maxTokens above a lower provider/discovered/runtime model limit, and current main still has paths that return the higher configured/requested value before considering the lower cap.

## Solution Assessment

Yes. Minimum-of-available-limits is the narrow maintainable direction, but the safest implementation is a centralized resolver with provider-qualified lookup handling rather than scattered Math.min calls.
