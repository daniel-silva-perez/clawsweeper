## Summary

Keep this issue open. The relevant files are unchanged from the earlier ClawSweeper review, and current main still has strict-priority context-token resolution in auto-reply, session persistence, and Gateway defaults, plus maxTokens forwarding paths that can pass a configured request above the model/provider cap.

## What This Changes

Request to compute effective contextTokens and maxTokens as the minimum of positive agent config, provider model config, discovered model metadata, and defaults.

## Best Possible Solution

Add a shared positive-token-limit resolver for context windows and output-token limits, then route agent setup, auto-reply/session persistence, Gateway defaults/status rows, and provider request construction through it with focused regressions for high configured caps and lower model/discovery limits.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available: pass contextTokensOverride 500000 with a configured provider model contextWindow of 131072 and resolveContextTokensForModel returns the override before consulting model metadata; the auto-reply helper also returns agents.defaults.contextTokens first.

## Solution Assessment

Yes, the requested minimum-of-available-limits direction is the narrow maintainable fix. It should be implemented as a shared conservative resolver with explicit tests for provider-qualified discovery, Anthropic 1M behavior, and per-provider maxTokens request fields.
