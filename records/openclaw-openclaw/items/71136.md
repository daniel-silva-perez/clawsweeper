## Summary

Keep this issue open. Current main still resolves context limits by priority in the live agent, auto-reply, and Gateway session paths, and runtime maxTokens overrides can still bypass lower model/provider caps instead of using the minimum safe value.

## What This Changes

Request to compute effective contextTokens and maxTokens as the minimum of positive agent config, provider model config, discovered model metadata, and safe defaults.

## Best Possible Solution

Add a shared positive-token-limit resolver for context window and output-token limits, then route agent setup, auto-reply/session persistence, Gateway defaults/status rows, and provider request construction through it with focused regressions for high configured caps and lower model/discovery limits.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: pass contextTokensOverride 500000 with a configured provider model contextWindow of 131072 and resolveContextTokensForModel returns the override before consulting model metadata; the auto-reply helper also returns agents.defaults.contextTokens first.

## Solution Assessment

Yes, the requested minimum-of-available-limits direction is the right maintainable fix. The implementation should be shared and conservative, with explicit tests for provider-qualified discovery, Anthropic 1M handling, and per-provider maxTokens request fields.
