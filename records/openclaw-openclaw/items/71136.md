## Summary

Keep open. Current main has a partial embedded-runner context-budget cap, but the shared context-token resolver and session fallback path can still let an oversized configured override win, and runtime `maxTokens` params still flow into core transports without being capped by the known model output limit. The open #39857/#39859 pair is useful prior art for the contextTokens half, but it does not cover this issue's maxTokens scope.

## What This Changes

The issue requests resolving effective `contextTokens` and `maxTokens` as the minimum of agent config, provider/model config, discovered model metadata, and defaults so configured overrides cannot exceed known model limits.

## Best Possible Solution

Keep this issue open and finish the resolver contract in one coordinated change: make configured contextTokens act as ceilings when a model limit is known, preserve existing behavior when the model limit is unknown, and clamp runtime maxTokens params against known model output caps at a shared boundary before provider transports receive them. Reuse #39859 or a replacement for the contextTokens portion, then add focused transport/runtime regression coverage for maxTokens.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `resolveContextTokensForModel({ contextTokensOverride: 500000, provider, model })` currently returns the override before known model-window lookup, and `extraParams.maxTokens` can be traced directly into OpenAI/Anthropic request payloads without clamping.

## Solution Assessment

Yes, the requested minimum/ceiling direction is the right safety model when all values are known. The maintainable implementation should be centralized in the context/model/runtime boundary, not repeated separately in each caller, while preserving override behavior for unknown model limits.
