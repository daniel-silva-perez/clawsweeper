## Summary

Keep this issue open. Current main already includes Korean Control UI locale support, registration, docs, and tests, but the same issue also asks for default AI-agent language persistence and automatic Korean search localization; those platform behaviors are still not implemented, and the paired same-author PR #53346 is closed unmerged.

## Best Possible Solution

Keep this issue open but narrow the remaining maintainer scope. Preserve the shipped Korean Control UI locale, then design and test a default primary-language response policy in the agent prompt or conversation layer with override and literal-output preservation rules. Add Korean search localization only through provider-aware web-search defaults or config, not by assuming a universal `search_lang` field.

## Risks / Open Questions

- Closing now would drop the still-unimplemented agent-side language consistency and localized search portions of the request.
- A default primary-language policy needs explicit precedence for user overrides and should preserve code, logs, identifiers, paths, quotes, and raw tool output instead of translating everything blindly.
- Automatic Korean search localization must be provider-aware because `search_lang` is Brave-specific while other web-search providers expose different language/filter surfaces.
