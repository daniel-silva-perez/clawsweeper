## Summary

Keep this PR open. Current main partially supersedes the original gateway `agent` handler concern by evaluating freshness before choosing `resolvedSessionId`, but it does not implement the PR’s full intended change: node `voice.transcript` and `agent.request` events still forward a stored `sessionId` together with a canonical `sessionKey`, and downstream session resolution treats that explicit `sessionId` as authoritative. The PR is still a plausible implementation candidate for the remaining stale-session pinning path and should be rebased/reviewed rather than closed as implemented.

## Best Possible Solution

Rebase this PR on current main, preserve the remaining node-event fix so `voice.transcript` and `agent.request` dispatch by canonical `sessionKey` without forwarding stale stored `sessionId`, and reconcile the already-merged gateway `agent` freshness logic with the PR’s stricter handoff semantics. Add or keep focused regression tests for both node-event paths, then use the rebased PR as the implementation candidate for #54164 if maintainers agree with the behavior.
