## Summary

Keep this PR open because current main still does not implement the central behavior for #46341: recurring cron/every failures are not split into transient versus permanent errors before choosing the next run time. The PR is still a useful implementation candidate, but it should not merge as-is because the branch broadens the documented one-shot `cron.retry` contract and includes unrelated guardrail allowlists in scripts.

## What This Changes

The PR changes cron scheduling so transient recurring job failures retry on a short backoff path, adds cron regression/restart tests, and also carries unrelated script/test guardrail updates from CI drift cleanup.

## Best Possible Solution

Land a focused cron scheduler change that makes recurring `cron` and `every` jobs retry transient network, timeout, rate-limit, overload, and 5xx failures before the next natural schedule, while permanent errors keep the existing natural-schedule behavior. The final patch should preserve restart catch-up semantics, settle the config/docs contract with either a dedicated recurring retry policy or the fixed recurring backoff ladder, add long-interval recurring regression coverage, and drop unrelated script/CI guardrail changes.
