## Summary

Keep open. Current main still routes Feishu table-heavy reply, streaming fallback, and outbound card delivery through interactive-card sends without a 230099/11310 table-limit classifier or retry to the existing plain-text table-conversion path; open PR #50429 directly targets this issue and related PRs #73651/#42809 cover adjacent outbound and native table rendering work.

## What This Changes

The issue requests Feishu/Lark card delivery to handle markdown-table-heavy responses by falling back to plain text/raw rendering or safely converting/rendering tables when Feishu rejects cards for exceeding table limits.

## Best Possible Solution

A consolidated Feishu plugin fix should classify CardKit/table-limit errors such as 230099/11310, retry affected reply, streaming-start fallback, and outbound card sends through the existing plain-text/table-conversion path or another bounded safe rendering strategy, preserve threading and mentions, rethrow unrelated card failures, and add focused regression coverage for each delivery surface.

## Reproduction Assessment

Yes. The issue provides concrete Feishu 230099/11310 logs, and current-main source inspection shows table-heavy replies and outbound sends still enter card delivery without a table-limit catch; a live Feishu account would verify the end-to-end API rejection.

## Solution Assessment

Yes for the fallback direction: using the existing sendMessageFeishu path is the narrowest maintainable first fix because it already converts markdown tables for post-format messages. Native table or image rendering can be follow-up polish, but the core bug needs a table-limit-specific fallback and tests first.
