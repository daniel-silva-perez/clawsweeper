## Summary

Keep open. Current main still sends Feishu markdown/table content through interactive-card paths without a table-limit-specific classifier or retry to the plain-text/table-conversion path, and this issue is already paired with open implementation PRs that should be reviewed or consolidated before the issue is closed.

## What This Changes

The issue asks Feishu card delivery to fall back or otherwise avoid Feishu table-count failures when responses contain multiple markdown tables.

## Best Possible Solution

Land or rework a focused Feishu plugin fix that classifies CardKit table-limit errors such as 230099/11310, retries affected reply and outbound card sends through the existing plain-text/table-conversion path while preserving threading and mentions, rethrows unrelated card failures, and adds regression tests for reply, streaming-start fallback, and outbound delivery.
