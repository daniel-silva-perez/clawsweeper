## Summary

Keep open. Current main still has Feishu reply, streaming fallback, and outbound card paths that can send table-heavy markdown through interactive-card delivery without a 230099/11310 table-limit classifier or retry to the existing plain-text table-conversion path. The issue is also paired with open implementation PRs, so it should stay open until maintainers review and land or consolidate those fixes.

## What This Changes

The issue asks Feishu delivery to handle card table-count limit failures by falling back to plain text/table conversion or another representation instead of dropping the message.

## Best Possible Solution

Land or rework a focused Feishu plugin fix that classifies CardKit table-limit failures such as 230099/11310, retries affected reply and outbound card sends through the existing plain-text/table-conversion path while preserving threading and mentions, rethrows unrelated card failures, and adds regression tests for reply, streaming-start fallback, and outbound delivery.
