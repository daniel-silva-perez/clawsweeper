## Summary

Keep open. Current main still routes Feishu reply and outbound card text through direct interactive-card sends without a table-limit/plain-text fallback, while the plain text path remains the only path that applies Markdown table conversion. The issue is also paired with open implementation candidates, especially #50429, so it should not be closed before a linked or superseding fix lands.

## What This Changes

Feishu should recover when interactive card delivery hits the platform table-count limit by falling back to text or otherwise rendering tables safely instead of dropping the response.

## Best Possible Solution

Land or rework a Feishu plugin fix that specifically classifies CardKit table-limit failures such as 230099/11310, retries the affected reply and outbound card sends through the plain-text/table-conversion path, preserves routing/thread behavior, rethrows unrelated card failures, and adds focused regression tests. Coordinate with #42809 only for native table rendering so the fallback fix can remain narrow.
