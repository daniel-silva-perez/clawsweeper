## Summary

Keep open. Current main still routes Feishu table-heavy card replies, streaming-start fallback, and outbound card sends through interactive-card delivery without a table-limit classifier or retry to the existing plain-text table-conversion path. The issue is also paired with open implementation PRs, including a closing PR, so it should remain open until maintainers review and consolidate that work.

## What This Changes

The request is to prevent Feishu interactive-card delivery from dropping table-heavy markdown replies by falling back to plain text or rendering/converting tables before card creation.

## Best Possible Solution

A consolidated Feishu plugin fix should classify CardKit table-limit responses such as 230099/11310, retry affected reply, streaming-start fallback, and outbound card sends through the existing plain-text/table-conversion path or a safe table-rendering strategy, preserve threading and mentions, rethrow unrelated card failures, and add focused regression coverage for each delivery surface.
