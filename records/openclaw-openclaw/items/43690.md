## Summary

Keep open. Current main still lacks a Feishu CardKit/table-limit classifier or retry path for reply cards, streaming-card fallback, and outbound card sends, and this issue is paired with open implementation PRs that directly reference the same failure, including a closing PR. Maintainers should review and consolidate that existing work rather than closing this issue or spawning another duplicate repair branch.

## What This Changes

The issue requests that Feishu card replies with multiple markdown tables avoid the 230099/11310 table-count failure by falling back to plain text, converting tables before card creation, or rendering tables outside the limited card table path.

## Best Possible Solution

A consolidated Feishu plugin fix should classify CardKit/table-limit errors such as 230099/11310, retry affected reply-card, streaming-card fallback, and outbound card sends through the existing plain-text/table-conversion path or another bounded safe renderer, preserve threading and mentions, rethrow unrelated card failures, and add focused regression coverage for each delivery surface.

## Reproduction Assessment

Yes. The issue provides concrete Feishu 230099/11310 logs and steps, and current-main source inspection shows table-heavy auto/card replies and outbound sends still enter card delivery without a table-limit fallback; a live Feishu bot would verify the end-to-end API rejection.

## Solution Assessment

Yes for the fallback direction. The narrowest maintainable first fix is to reuse the existing sendMessageFeishu path because it already applies markdown table conversion; image or native-table rendering can remain follow-up polish if maintainers want richer presentation.
