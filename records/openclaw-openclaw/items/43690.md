## Summary

Keep open. Current main still lacks a Feishu table-limit classifier or retry path for card replies, streaming-card fallback, and outbound card sends; the issue is also paired with open implementation PRs, including a direct closing PR, so it should stay open until maintainers review and land or replace that work.

## What This Changes

The issue asks Feishu card delivery to fall back or render tables safely when table-heavy markdown hits the Feishu 230099/11310 table-count limit.

## Best Possible Solution

A consolidated Feishu plugin fix should classify CardKit/table-limit errors such as 230099/11310, retry affected reply, streaming-start fallback, and outbound card sends through the existing plain-text/table-conversion path or another bounded safe renderer, preserve threading and mentions, rethrow unrelated card failures, and add focused regression coverage for each delivery surface.

## Reproduction Assessment

Yes. The issue gives concrete Feishu 230099/11310 logs, and current-main source inspection shows table-heavy auto/card replies and outbound sends still enter card delivery without a table-limit fallback; a live Feishu bot would verify the end-to-end API rejection.

## Solution Assessment

Yes for the fallback direction. Using the existing sendMessageFeishu path is the narrowest maintainable first fix because it already applies markdown table conversion; image or native-table rendering can be follow-up polish if maintainers want richer presentation.
