## Summary

Keep this issue open. Current main still logs configured MCP startup/list failures at the bundle-MCP runtime boundary, drops the failed server from the live catalog, and passes only the reduced successful tool set into the embedded runner prompt. The requested `mcp.servers.<name>.required` / `onStartFailure` semantics and expected-vs-available tool diagnostics are not present in config, docs, runtime catalog types, materialization, or runner prompt assembly.

## Best Possible Solution

Keep this issue open and implement a generic bundled-MCP startup diagnostics path in core: preserve redacted failed-server records in the session MCP catalog, propagate diagnostics into embedded-runner prompt/channel delivery, add typed `mcp.servers.<name>.required` semantics that can block agent start, and add focused runtime/config/runner/docs coverage for stdio, SSE, and streamable-HTTP startup failures.

## Risks / Open Questions

- Startup diagnostics must redact command arguments, URLs, headers, environment values, and dependency error text before exposing anything to the model, transcript, logs, or channel delivery.
- `mcp.servers.<name>.required` needs a transport-agnostic contract for stdio, SSE, and streamable HTTP so optional failures can degrade while required failures block the run with a clear user-visible message.
- Expected-vs-available tool awareness may require a declared baseline or cached last-known catalog, because a failed or removed MCP package cannot always report the tools it would normally expose.
