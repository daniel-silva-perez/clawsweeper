## Summary

Keep this issue open. Current main still treats a configured MCP startup failure as a gateway warning plus a reduced tool catalog; it does not surface failed-server diagnostics to the agent or active channel, does not implement `mcp.servers.<name>.required`, and has no expected-vs-available MCP tool delta.

## Best Possible Solution

Keep this issue open and implement a generic bundled-MCP startup diagnostics path: preserve redacted failed-server records in the session MCP catalog, propagate diagnostics into embedded-runner prompt/channel delivery, add typed `mcp.servers.<name>.required` semantics that can block agent start, add expected-vs-available tool awareness where feasible, and cover stdio, SSE, and streamable-HTTP failures with focused runtime/config/runner/docs tests.

## Risks / Open Questions

- Startup diagnostics must redact command arguments, URLs, headers, environment values, and dependency error text before exposing anything to the model, transcript, logs, or channel delivery.
- `mcp.servers.<name>.required` needs a transport-agnostic contract for stdio, SSE, and streamable HTTP so optional failures can degrade while required failures block the run with a user-visible message.
- Expected-vs-available tool awareness may need a declared baseline or cached last-known catalog because a failed or removed package cannot always report its expected tools at failure time.
