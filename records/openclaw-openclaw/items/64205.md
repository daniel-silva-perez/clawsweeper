## Summary

Keep open. Current main still treats configured MCP startup failures as gateway-side warnings plus a reduced MCP tool catalog; the agent prompt/channel path receives only the surviving tool list, and the requested `required` or startup-failure notification contract is not present in config or docs.

## What This Changes

Configured MCP server startup failures should be surfaced to the agent/user, with optional required-server fail-fast semantics and awareness of missing MCP tools.

## Best Possible Solution

Add a generic, redacted MCP startup diagnostics path: preserve failed-server records in the session MCP runtime/catalog, pass safe diagnostics into embedded runner prompt or active channel delivery, define maintainer-approved `required` fail-fast semantics, and document/test stdio, SSE, and streamable HTTP behavior without Telegram-specific special casing.
