## Summary

Keep open. Current main still compares the tailnet IPv6 address by lowercased text form, so compressed and expanded spellings of the same Tailscale IPv6 address can miss; the PR's production direction matches the existing IP normalization contract but needs branch refresh, maintainer review, and fresh validation before merge.

## What This Changes

This PR normalizes stored tailnet IPv4/IPv6 addresses in `isLocalGatewayAddress` before comparison and adds gateway regression tests for equivalent IP forms.

## Best Possible Solution

Refresh the branch against current main, keep the normalized-both-sides comparison through the existing parser-backed IP normalizer, retain the focused regression tests, and land after gateway owner review with current validation.

## Reproduction Assessment

Yes. Mock `pickPrimaryTailnetIPv6()` to return `fd7a:115c:a1e0::1` and call current-main `isLocalGatewayAddress("fd7a:115c:a1e0:0:0:0:0:1")`; by inspection, the current IPv6 branch compares the lowercased raw strings and returns false for that equivalent address spelling.

## Solution Assessment

Yes. The production approach is the narrowest maintainable fix because it reuses the existing canonical IP parser on the stored tailnet values before equality comparison; the PR still needs refresh and validation, not a different design.
