## Summary

This PR should stay open. Current main still normalizes the candidate IP but compares tailnet IPv4/IPv6 values as raw lowercased strings, so equivalent expanded and compressed IPv6 tailnet forms can still miss; the PR applies the existing normalizer on both sides and its latest diff addresses the earlier test-mocking concern with a hoisted module mock.

## What This Changes

The PR changes `isLocalGatewayAddress` to normalize stored tailnet IPv4/IPv6 addresses before comparison and adds gateway tests for equivalent IPv6 spellings, IPv4-mapped IPv4, loopback, and empty inputs.

## Best Possible Solution

Keep the narrow normalization design if maintainers want this helper retained: compare both candidate and tailnet addresses through the shared IP normalizer, preserve regression coverage for equivalent textual IP forms, then merge only after a clean refresh and current gateway/changed-lane validation.

## Reproduction Assessment

Yes. A focused reproduction is to mock `pickPrimaryTailnetIPv6()` to return `fd7a:115c:a1e0::1` and call `isLocalGatewayAddress("fd7a:115c:a1e0:0:0:0:0:1")`; current main's raw string comparison returns false even though the addresses are equivalent.

## Solution Assessment

Yes. The PR's approach is the narrowest maintainable fix for the reported behavior because it reuses the existing `normalizeIp` contract on both operands rather than adding IPv6-specific comparison logic.
