## Summary

Keep open. Current main still resumes restart-aborted subagent sessions from persisted active run membership plus `abortedLastRun` with no age bound, the config surface has no `taskRecovery.maxAgeMinutes`, and tests/docs still preserve stale restart recovery. The discussion now includes a second concrete incident where an old recovered subagent caused real-world side effects, and a read-only GitHub API search found no open paired implementation PR.

## What This Changes

The issue requests an age-bound recovery policy so stale interrupted CLI/subagent work is marked lost instead of being automatically resumed after a gateway restart.

## Best Possible Solution

Add a bounded restart-aborted recovery policy in the core subagent/task recovery path: keep fresh interrupted work recoverable, mark stale interrupted run/task records terminal without synthetic redispatch, expose a documented config default, and add regression coverage for both stale-skip and fresh-resume behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction exists by persisting an old active subagent run with a child session marked `abortedLastRun: true`, restarting the registry/gateway, and observing orphan recovery schedule a synthetic resume; the current persistence test already asserts the stale three-hour recovery path.

## Solution Assessment

Yes. An age gate at the orphan-recovery decision point, paired with terminalizing the associated run/task and config/docs/tests, is the narrowest maintainable fix; filtering only task restore or only documentation would leave the redispatch path intact.

## Repair Work Prompt

Fix openclaw/openclaw#69778. Current main automatically resumes restart-aborted subagent sessions of any age when an active run has a child session with `abortedLastRun: true`; stale work can re-execute long after the user moved on. Add a bounded recovery policy, defaulting to a conservative max age such as 120 minutes unless an existing config pattern suggests a better name/default. Implement the check in the subagent orphan recovery decision path so stale aborted runs are not synthetically resumed, and make the associated run/task terminal as `lost` with a useful reason. Preserve fresh restart recovery within the configured window. Add `agents.defaults.taskRecovery.maxAgeMinutes` or the nearest maintainable config surface, update TS types, zod schema/help/generated config docs as required by the repo, and update public docs for tasks/subagents/config. Add regression tests for stale aborted run skipped/lost, fresh aborted run resumed, and task maintenance interaction. Keep #74864 separate: do not attempt a broad wedged-session health policy in this PR beyond avoiding stale resurrection. Add a user-facing changelog fix entry crediting @FelixRen-FY and @simonstephens if the repo’s active changelog requires one.
