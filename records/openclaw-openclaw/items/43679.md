## Summary

## What This Changes

The PR adds shared suspicious reply leakage detection, extends reply normalization with a `suspicious` skip reason, and adds tests for malformed orchestration text suppression plus false-positive cases.

## Best Possible Solution

Land a narrow, rebased shared guard in `normalizeReplyPayload` that uses the current `hasContent()`/`hasReplyPayloadContent` semantics, strips suspicious text while preserving non-text payload content, honors custom silent tokens, and keeps regression coverage for suppression, media/channel-data preservation, custom tokens, standalone route markers, pretty-printed tool JSON, and explanatory prose.

Yes. Source inspection gives a high-confidence path: final replies flow through `createReplyDispatcher` into `normalizeReplyPayload`, and current main has no suspicious route-marker/tool-JSON detector beyond silent-token, heartbeat, and generic sanitizer handling.

## Solution Assessment

No, not as currently written. The shared-normalization approach is the right direction, but the patch must be rebased onto the current content-helper normalizer before it can preserve current payload semantics and pass review.
