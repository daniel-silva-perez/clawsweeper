## Summary

Keep open. Current main still lacks the proposed suspicious-reply guard, but this PR is not merge-ready because its normalizer hunk is based on an older implementation shape with `hasMedia`/`hasChannelData` and Slack directive parsing, while current main uses `hasReplyPayloadContent`/`hasContent` and applies channel transforms after sanitization. Earlier review feedback on media preservation and false positives was iterated on; the remaining blocker is rebase/adaptation to current main.

## What This Changes

The PR adds a shared suspicious-reply detector, wires it into auto-reply normalization with a new `suspicious` skip reason, and adds normalizer/dispatcher coverage for malformed internal orchestration text.

## Best Possible Solution

Land a narrow, rebased shared guard in the current reply normalization or outbound-normalization path that preserves current content checks and transform semantics, strips suspicious text while retaining non-text payloads, and keeps focused coverage for text-only suppression, media/channelData preservation, custom silent tokens, explanatory examples, and dispatcher delivery.
