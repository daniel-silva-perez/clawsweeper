## Summary

Current main still does not implement the requested managed-skill symlink workflow. The loader now discovers symlinked skill directories, but then realpath containment rejects any candidate whose resolved target escapes the configured skill root, so a symlink under ~/.openclaw/skills pointing to an external skill folder remains skipped. The documented workaround is skills.load.extraDirs, and the related audit-boundary PRs were closed unmerged rather than implementing this behavior.

## Best Possible Solution

Keep this issue open for maintainer policy judgment. If OpenClaw should support symlinked managed skills that point outside ~/.openclaw/skills, add an explicit opt-in trust model and update the loader, security audit, docs, and regression tests together. If the current containment policy is intentional, document that root-escaping symlinks under managed skill roots are unsupported and direct users to skills.load.extraDirs or ClawHub installs.
