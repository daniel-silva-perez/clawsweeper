## Summary

Current main still does not implement the reported managed-skill symlink workflow. The loader now discovers symlinked skill directories, but candidates are realpath-contained to their configured root, so a symlink under `~/.openclaw/skills` that points to an external skill directory is still skipped. Docs point users to `skills.load.extraDirs` for external shared roots, and the related audit-boundary PRs were closed unmerged.

## Best Possible Solution

Keep this issue open for maintainer policy judgment. If OpenClaw should support symlinked managed skills that point outside `~/.openclaw/skills`, add an explicit opt-in trust model and update the loader, security audit, docs, and regression tests together. If the current containment policy is intentional, document that root-escaping symlinks under managed skill roots are unsupported and direct users to `skills.load.extraDirs` or ClawHub installs.
