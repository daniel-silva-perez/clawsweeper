## Summary

Current main only partially addresses the report: the skill scanner now notices symlinked directory entries, but it then resolves each candidate with realpath and rejects any skill directory or SKILL.md that resolves outside its configured skill root. A symlink from ~/.openclaw/skills/my-custom-skill to an external directory still matches the reported failure pattern. The documented workaround is to configure an external root with skills.load.extraDirs, but that does not implement the requested symlink-under-managed-skills behavior. The related closed PRs in the provided context focused on the security audit boundary and were not merged, so they do not supersede this loader request.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. If OpenClaw should support this workflow, add an explicit opt-in path for external skill roots or symlinked managed skills and update loader containment, security audit behavior, docs, and regression tests together. If the intended policy is to reject root-escaping symlinks, document that ~/.openclaw/skills symlinks to external directories are unsupported and point users to skills.load.extraDirs or ClawHub installs.
