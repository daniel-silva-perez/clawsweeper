## Summary

Keep open. Current main still builds the plugin metadata snapshot and manifest registry across the broader installed/bundled candidate set before the gateway startup plugin scope is applied, so the reported discovery-time manifest materialization cost remains a valid code-level performance bug.

## What This Changes

The issue asks OpenClaw to apply the plugin allowlist or equivalent startup plugin scope before reading and validating bundled plugin manifests during gateway boot.

## Best Possible Solution

Thread a conservative startup plugin scope into the cold metadata/discovery path so bundled directories outside the resolved startup set are skipped before package or plugin manifest reads, while keeping the existing manifest-first contracts for explicitly enabled channels, providers, slots, and config entries.

## Reproduction Assessment

Yes for the control-flow bug: current main shows gateway startup building the metadata snapshot and manifest registry before `onlyPluginIds` limits runtime loading. I did not run a live macOS heap profile to reproduce the exact RSS number.

## Solution Assessment

Yes, discovery-time scoping is the right direction, but the maintainable fix should derive a conservative plugin startup scope and pass it through discovery/manifest-registry APIs. A raw `tools.allow` filter would be the wrong contract because `tools.allow` controls tool access, not plugin inventory.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/70533 by reducing gateway boot plugin discovery/manifest materialization when the configured startup plugin set is small. Current main computes the startup plugin ids only after building a full metadata snapshot, and `loadOpenClawPlugins` applies `onlyPluginIds` after discovery and manifest registry loading. Implement a conservative cold-start plugin scope that is available before bundled discovery and installed-index manifest registry loading, then thread that scope through the relevant discovery/manifest-registry APIs so out-of-scope bundled plugin directories are skipped before package.json/openclaw manifest reads and zod validation.

Expected boundary: work primarily in `src/plugins/discovery.ts`, `src/plugins/manifest-registry.ts`, `src/plugins/manifest-registry-installed.ts`, `src/plugins/plugin-metadata-snapshot.ts`, `src/plugins/plugin-lookup-table.ts`, `src/plugins/loader.ts`, `src/gateway/server-startup-plugins.ts`, and `src/gateway/server-plugins.ts`, with focused tests. Preserve explicit channel enablement, provider activation, memory slot selection, plugin entries, deny rules, and existing external/custom path behavior. Do not repurpose `tools.allow`; plugin eligibility belongs to `plugins.allow` and derived startup scope. Do not add startup-time migrations or persistent manifest caches as a substitute for scoping.

Add regression coverage proving non-empty startup/plugin scopes avoid reading or validating unrelated bundled manifests while still pulling required explicitly enabled channels/providers/slots. Add a single changelog Fixes entry crediting `@jpippo364` if the behavior change lands.
