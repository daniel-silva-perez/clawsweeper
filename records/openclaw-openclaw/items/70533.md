## Summary

Current main still matches the reported shape: gateway startup derives a scoped `onlyPluginIds` set, but only after building a full plugin metadata snapshot and manifest registry. Bundled discovery still scans the stock plugin directory without a plugin-id scope, and manifest materialization still loads every candidate it receives unless a caller supplies `pluginIds`; the startup lookup path does not. Related #48644 is closed for startup/runtime import work and #74087 is an orthogonal compile-cache proposal, so neither supersedes this discovery-time manifest/materialization issue.

## Best Possible Solution

Keep this open and implement scoped cold discovery/manifest materialization for gateway startup: derive a conservative startup/plugin scope before bundled manifest registry creation, pass that scope into bundled discovery and installed-index manifest loading, skip out-of-scope bundled directories before package/manifest/schema reads where safe, and add regression tests proving skipped bundled manifests are not read while explicit channel/provider/slot activation semantics still work.
