## Summary

Keep open. Current main still has the reported stale-delivery path: interrupt mode aborts the active run, the embedded runner still passes `attempt.lastAssistant` into payload building, the payload builder can fall back to that assistant text when no new assistant text exists, and `runReplyAgent` still flushes and delivers payloads without an aborted-run gate. The provided discussion and open PR #50147 point at the same two affected surfaces, and that PR is still unmerged.

## What This Changes

The issue requests that interrupt-mode aborted runs produce no outgoing reply and avoid replaying the previous assistant response when a new inbound message interrupts an active run.

## Best Possible Solution

Land PR #50147 or an equivalent narrow patch with regression coverage for an interrupt-mode abort that produces no new assistant text: the aborted run should emit no payload, while normal non-aborted final-answer fallback remains intact.
