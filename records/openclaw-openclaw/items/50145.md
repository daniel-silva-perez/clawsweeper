## Summary

Current main still contains the reported interrupt-abort stale delivery path, and the issue is paired with open, unmerged PR #50147 using closing syntax. Repository policy says to keep this issue open until that PR or an equivalent fix lands.

## What This Changes

Fix interrupt queue mode so an aborted active run that produced no new assistant output does not deliver the previous assistant reply before the replacement run replies.

## Best Possible Solution

Keep this issue open and review or land PR #50147, or an equivalent patch, with regression coverage for an aborted interrupt-mode run that produces no new assistant text. The fix should guard the lastAssistant fallback only for aborted/no-output attempts and make `runReplyAgent` skip final payload delivery for aborted runs without regressing valid non-aborted fallback behavior.
