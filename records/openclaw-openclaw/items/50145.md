## Summary

Keep open. Current main still contains the reported stale-replay path, and open PR #50147 is the paired closing-syntax implementation candidate, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

The issue reports that interrupt queue mode can resend the previous assistant reply from session history after aborting an active run, before the replacement run delivers the correct reply.

## Best Possible Solution

Land PR #50147 or an equivalent narrow patch that prevents aborted interrupt runs with no new assistant output from constructing or delivering payloads from prior session history, with focused coverage around the payload-builder and reply-runner boundary.

## Reproduction Assessment

Yes. The issue gives concrete interrupt-mode steps, and current-main static inspection matches the path: interrupt aborts active work, an attempt can retain a prior `lastAssistant`, payload construction can fall back to it, and delivery has no aborted-run early exit.

## Solution Assessment

Yes, with verification. Guarding the aborted empty-attempt `lastAssistant` fallback and skipping aborted-run delivery is the narrow maintainable fix; regression tests should preserve legitimate non-aborted fallback behavior.
