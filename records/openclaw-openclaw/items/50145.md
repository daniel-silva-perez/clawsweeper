## Summary

Keep open. Current main still contains the reported stale replay path, and the linked closing-syntax PR #50147 remains the appropriate implementation candidate rather than a cleanup close.

## What This Changes

The issue reports that `messages.queue.mode: "interrupt"` can resend a previous assistant reply after an aborted active run before the newest turn replies.

## Best Possible Solution

Land PR #50147 or an equivalent narrow patch that prevents aborted interrupt runs with no new assistant output from constructing or delivering payloads from prior session history, with tests at the payload-builder and reply-runner boundary.

## Reproduction Assessment

Yes. The issue provides concrete interrupt-mode steps, and current-main static inspection confirms the abort path can retain an older `lastAssistant`, convert it into a payload, and deliver it without an aborted-run gate.

## Solution Assessment

Yes, with regression coverage. Guarding the aborted empty-attempt `lastAssistant` fallback and skipping aborted-run delivery is the narrow maintainable fix; the important safety check is preserving legitimate non-aborted fallback behavior.
