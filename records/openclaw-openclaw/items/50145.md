## Summary

Keep open. Current main still contains the reported stale replay path, and open PR #50147 uses closing syntax for this bug but is unmerged, so the issue should wait for that PR or an equivalent fix to land.

## What This Changes

The issue reports that interrupt queue mode can replay the previous assistant reply from an aborted run before delivering the correct follow-up response.

## Best Possible Solution

Land PR #50147 or an equivalent narrow patch that prevents aborted interrupt runs with no new assistant output from constructing or delivering payloads from prior session history, with focused regression coverage at the payload-builder and reply-runner boundary.

## Reproduction Assessment

Yes. The issue gives concrete interrupt-mode steps, and current-main static inspection matches the path: interrupt aborts active work, the attempt can retain a prior `lastAssistant`, payload construction can fall back to it, and delivery has no aborted-run early exit.

## Solution Assessment

Yes, with verification. Guarding the aborted empty-attempt `lastAssistant` fallback and skipping aborted-run delivery is the narrow maintainable fix; regression tests should preserve valid non-aborted fallback behavior.
