## Summary

Keep open. Current main still has the reported stale-delivery path: interrupt mode aborts the active run, the embedded runner still passes the prior `lastAssistant` into payload construction, the payload builder still falls back to that assistant text when streamed assistant text is empty, and the reply runner still has no aborted-run delivery gate. The open paired PR #50147 remains the right implementation path to review rather than closing this issue now.

## What This Changes

The issue requests that `messages.queue.mode: "interrupt"` silently suppress output from aborted active runs so a previous assistant reply is not replayed before the new turn's reply.

## Best Possible Solution

Review and land PR #50147 or an equivalent narrow patch that prevents aborted interrupt runs with no new assistant output from producing or delivering payloads, with focused regression coverage around the payload-builder and reply-runner boundary.
