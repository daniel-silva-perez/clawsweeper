## Summary

Keep open. Current main still has the reported stale replay path: interrupt mode aborts an active run, the embedded runner can still build payloads from the previous assistant message when the aborted attempt produced no assistant text, and reply delivery has no aborted-run gate. Open PR #50147 is still unmerged and uses closing syntax for this bug, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests that `messages.queue.mode: "interrupt"` abort an active run silently without replaying the previous assistant reply, then deliver only the new run's response.

## Best Possible Solution

Review and land PR #50147, or an equivalent narrow patch, so aborted interrupt runs with no new assistant output cannot construct or deliver payloads from prior session history; add focused regression coverage around the payload-builder and reply-runner boundary.

## Reproduction Assessment

Yes. The issue gives concrete interrupt-mode steps, and the current-main code path matches them statically: interrupt aborts active work, the aborted attempt can retain a prior `lastAssistant`, payload construction can fall back to it, and reply delivery lacks an aborted-run early exit.

## Solution Assessment

Yes, with verification. The proposed boundary of guarding `lastAssistant` for aborted empty attempts and skipping aborted-run delivery is the narrow maintainable fix, but it should include regression tests that preserve the legitimate non-aborted fallback cases.
