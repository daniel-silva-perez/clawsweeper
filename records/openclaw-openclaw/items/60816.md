## Summary

Keep this issue open. Current main has same-file edit serialization through the pinned Pi edit tool, which addresses part of the concurrency race, but it still does not provide the requested noop result semantics when a follower finds the file already in the target state.

## Best Possible Solution

Keep open and implement a real post-state classifier for edit results: one result for a request that writes, a distinct noop result when the requested final state is already satisfied after waiting/rechecking, and preserved errors for true mismatches. Add focused concurrent tests for canonical `edits[]`, already-up-to-date state, true failure, and non-identical concurrent edits; decide explicitly whether OpenClaw should also accept upstream Pi's flat `oldText`/`newText` compatibility shape.
