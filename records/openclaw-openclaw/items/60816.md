## Summary

Keep open: current main has same-file edit serialization, but it still does not expose the requested third result state. A concurrent follower can be reported as a normal success by OpenClaw recovery, while an already-up-to-date later call can still surface as a mismatch error instead of a noop.

## What This Changes

The issue requests edit-tool result semantics that distinguish mutation success, already-satisfied noop, and true failure for concurrent identical replacements.

## Best Possible Solution

Keep the issue open and add a focused post-state classifier around edit mismatches: normal success when this call actually wrote, distinct non-error noop when the requested final state is already satisfied, and unchanged error behavior for true mismatches or write failures. Cover canonical `edits[]`, already-up-to-date state, true failure, and non-identical edits; handle flat replacement input only if maintainers confirm that compatibility remains part of OpenClaw's contract.

## Reproduction Assessment

Yes. The source gives a high-confidence reproduction path: run two identical edit requests against the same file, where the queued follower sees `oldText` gone; current recovery can report that as success if its snapshot was stale, and a later already-up-to-date call is covered by an existing test that still expects an error.

## Solution Assessment

Yes for the requested semantics, but the safest implementation layer needs care. A narrow classifier for Pi mismatch errors in OpenClaw's wrapper, or the same change upstream in Pi, is better than broad serialization or suppressing all edit mismatches.
