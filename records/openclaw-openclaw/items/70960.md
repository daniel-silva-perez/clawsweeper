## Summary

Keep this open because current main still only partially addresses the bundled report: the gateway now exposes configured agent names and one Agents selector prefers them, but other Control UI display paths still prefer identity names, the topbar still does not show current token/cost/cache usage, and the QClaw Windows Electron upgrade/delete path is not present in this repository.

## What This Changes

The issue reports a QClaw Windows upgrade that removes the installed app, Control UI agent labels ignoring `agents.list[].name`, and a follow-up report that topbar token/cost/cache usage disappeared.

## Best Possible Solution

Split or route the bundled report into focused follow-ups: send the destructive QClaw Windows upgrade behavior to the QClaw installer owner with the supplied install log and registry details, then fix OpenClaw Control UI so every agent-label surface uses a single documented precedence for `agents.list[].name`, identity name, and id, and either restore the topbar current-session token/cost/cache display or deliberately replace it with an equivalent always-visible usage indicator backed by the existing session usage data.

## Reproduction Assessment

Yes for the in-repo UI parts by static verification: current main exposes `agent.name` from the gateway, but several Control UI render paths still prefer `identity.name`, and the topbar has no usage renderer. No high-confidence reproduction path exists in this repo for the QClaw Windows app deletion because that Electron installer surface is absent from the checkout.

## Solution Assessment

No, current main is not a complete solution: it only fixes the Agents selector path and provides usage information elsewhere, while the reported topbar usage and name-precedence consistency remain unresolved. The safer path is to split/reroute the external installer issue and make the Control UI behavior consistent in one owner-reviewed UI change.
