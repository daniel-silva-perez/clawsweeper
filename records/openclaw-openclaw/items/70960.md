## Summary

Keep open. Current main only partially addresses the report: configured agent names now reach the gateway and the Agents selector, but several Control UI paths still prefer identity names, the topbar still does not render token/cost/cache usage, and the QClaw Windows Electron installer failure is not present in this source tree.

## What This Changes

The issue requests fixes for QClaw disappearing after Windows upgrade installs, Control UI honoring `agents.list[].name`, and restoring top-right token usage display.

## Best Possible Solution

Split the report into focused ownership paths: route the QClaw Windows installer deletion to the QClaw installer owner with the supplied logs and registry detail, then decide and implement one consistent Control UI display-name precedence and separately decide whether current-session usage belongs in the topbar or should remain in existing usage/status surfaces.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: yes for the in-repo UI portions by source inspection, because configured names reach `agents.list` and the Agents selector while other UI paths still use identity-first precedence and the topbar lacks usage output. No high-confidence in-repo reproduction exists for the QClaw installer deletion because that installer is absent from this checkout.

## Solution Assessment

Is this the best way to solve the issue? No, a single automated OpenClaw core fix is not the best path because the report mixes an external installer failure with Control UI product behavior. The safer route is maintainer triage to split ownership and then land focused UI changes after precedence and placement decisions are explicit.
