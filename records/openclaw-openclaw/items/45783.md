## Summary

Keep this PR open. Current main at e476523082a6fff3a0c3048077d2f0e52413fbf2 still does not implement the proposed LLM-boundary privacy pipeline: there is no native privacy module, no privacy config/schema, no encrypted mapping store, and no outbound replacement plus inbound restoration wrapper around provider calls. The PR is also security-sensitive and still needs maintainer architecture/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer architecture and security review, or supersede it with a smaller maintainer-approved design. The best path is a first-class LLM-boundary privacy contract, whether in core or via a new explicit plugin API, covering config/schema/docs, outbound and inbound stream semantics, mapping persistence, fail-open versus fail-closed behavior, custom rule validation, cache traces, payload logging, BTW, TTS, and provider-specific message shapes.
