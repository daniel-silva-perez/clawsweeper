## Summary

Keep this PR open. Current main still skips Feishu `dmPolicy` persistence when a fresh config falls back to `pairing` and the user explicitly selects `pairing`; the contribution targets a real bug, but the patch is stale because the active DM-policy prompt code now lives in `src/flows/channel-setup.prompts.ts` while the PR patch modifies the old `src/commands/onboard-channels.ts` implementation path.

## Best Possible Solution

Keep this PR open for maintainer follow-up. Rebase or rewrite the fix against `src/flows/channel-setup.prompts.ts`, preferably as a focused Feishu missing-key write or a small `ChannelSetupDmPolicy` persisted-presence contract, and add a regression test for a fresh Feishu config where the user explicitly selects `pairing`. Avoid a global unconditional setter until non-idempotent channel setters are repaired or bypassed safely.
