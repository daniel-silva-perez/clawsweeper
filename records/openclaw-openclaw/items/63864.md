## Summary

Keep this PR open. Current main still does not implement configurable per-session lane concurrency, and the PR remains a relevant implementation candidate for the session-concurrency part of #53023. SweetSophia’s latest commit addressed the earlier hot-reload/test-reset feedback, but the current PR head still lacks the strict config schema/docs/test surface and includes unrelated gateway reload/auth/secrets and SDK/type-surface regressions that make it unsafe to close or land as-is.

## Best Possible Solution

Keep this PR open for contributor revision or maintainer reimplementation. The narrow path is to preserve existing gateway reload/secrets/auth, SDK export, and public config type surfaces; add a settled session-lane concurrency key to AgentDefaultsConfig, AgentDefaultsSchema, generated config metadata/help, docs, and tests; then wire startup and hot reload through shared queue state so existing session:* lanes are updated and drained. Leave #53023 open for the broader auto-yield and lane-wait-threshold work unless those are implemented separately.

## Risks / Open Questions

- The current PR head would expose a public config key without adding it to the strict zod schema, generated config metadata/help, docs, or focused validation tests.
- The PR removes or rewires gateway reload/auth/secrets handling outside the stated session-concurrency scope, including the managed config reloader export used by server startup.
- The PR removes public config type members that current runtime/schema/docs still use, such as startupContext, contextLimits, embeddedPi.executionContract, heartbeat.timeoutSeconds, and CliBackendConfig.jsonlDialect.
- The PR changes plugin SDK/runtime re-exports in a way that can break bundled plugin imports, especially WhatsApp runtime config helpers.
- Per-session parallelism can race transcript/session state, tool effects, and provider rate limits; it needs explicit tests and conservative docs/defaults before landing.
