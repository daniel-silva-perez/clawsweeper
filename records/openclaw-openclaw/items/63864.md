## Summary

Current main still does not implement configurable per-session lane concurrency. This PR remains a relevant implementation candidate for the session-concurrency slice of #53023, but it should not be closed or merged as-is because the public config key is incomplete and the PR diff includes unrelated, security-sensitive gateway reload/auth/secrets and SDK/type-surface regressions.

## Best Possible Solution

Keep this PR open for contributor revision or maintainer reimplementation. The narrow fix should preserve existing gateway reload/secrets/auth, SDK export, and config type surfaces; add a settled session-lane concurrency key to the config type, strict zod schema, generated config metadata/help/docs, and focused tests; then wire startup and hot reload through shared queue state so existing `session:*` lanes are updated and drained. Leave #53023 open for auto-yield and lane-wait-threshold work unless those are implemented separately.

## Risks / Open Questions

- Merging the current PR head would expose a config key that the strict schema does not accept, making the advertised usage fail validation.
- The PR mixes the intended queue change with unrelated gateway reload, secrets snapshot, shared-auth generation, SDK export, and config type deletions, creating runtime and public API regression risk.
- Per-session parallelism can race session history, tool state, and provider rate limits; the implementation needs explicit tests and conservative docs/defaults before it is safe to land.
