## Summary

Keep open. Current main still uses SQLite as a retrieval index over Markdown/session chunks, not as the requested durable typed memory source of truth with canonical records, revisions, entities, lifecycle statuses, confidence, or validity windows.

## What This Changes

The issue requests a v1 SQLite-backed durable memory schema with typed canonical records, revision history, lifecycle statuses, confidence, validity windows, normalized entities, and non-canonical embeddings.

## Best Possible Solution

Keep this open for maintainer-owned Memory MVP design. The durable path should be a versioned SQLite schema/migration plus exported record, revision, entity, status, confidence, provenance, and validity types, with write-path integration, tests for one canonical current record and non-destructive revisions, and docs explaining how Markdown memory, retrieval indexes, memory-wiki, LanceDB, and optional memory plugins interact.

## Reproduction Assessment

Not applicable as a runtime reproduction; this is a schema/product request. Source and docs inspection gives a high-confidence verification path: current main has no durable typed SQLite record schema and still documents Markdown files plus a SQLite chunk index as the default memory model.

## Solution Assessment

Unclear as an autonomous fix, because the requested direction is likely correct but needs maintainer product/API judgment before code lands. The maintainable solution is a coordinated Memory MVP schema and migration design, not a narrow opportunistic table addition.
