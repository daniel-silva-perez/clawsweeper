## Summary

Current main still implements the builtin SQLite memory database as a chunk/search index over Markdown memory files and indexed session material, not as the requested durable typed memory record schema with canonical rows, revisions, statuses, confidence, validity windows, and normalized entities. The memory-wiki and memory-lancedb plugins are useful adjacent memory layers, but the default memory-core storage contract and public search API do not satisfy this Memory MVP schema request.

## Best Possible Solution

Keep this issue open. The best path is a maintainer-owned design and implementation in memory-core plus the memory-host SDK: versioned SQLite migrations, exported contract/types, tests for canonical memory rows and memory_revisions, explicit type/status/confidence/timestamp/validity/entity fields, and docs explaining how Markdown memory, public artifacts, memory-wiki, and optional memory plugins interact with the new durable source-of-truth model. Coordinate this with #42647, #42648, and #42650 so provenance, write resolution, and operator review flows share one schema.
