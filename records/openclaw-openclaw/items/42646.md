## Summary

Keep open. Current main still implements the default memory SQLite database as a Markdown/session retrieval index, not the requested durable typed SQLite source of truth with canonical records, revisions, entities, lifecycle statuses, confidence, and validity windows. The related memory-wiki and memory-lancedb work is useful context, but it is plugin-layer/alternate-backend functionality and does not satisfy this Memory MVP schema anchor.

## Best Possible Solution

Keep this open for maintainer-owned Memory MVP design and implementation. The likely path is a versioned SQLite schema/migration in memory-host-sdk or memory-core with exported record, revision, entity, status, confidence, and validity types; tests for one canonical current record plus non-destructive revisions; write-path integration; and docs explaining how Markdown memory, public artifacts, memory-wiki, memory-lancedb, and optional memory plugins interact. Coordinate with #42647, #42648, and #42650 before implementation.
