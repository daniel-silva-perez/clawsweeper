## Summary

Keep open. Current main still models built-in memory as a Markdown-backed SQLite retrieval index, not the requested durable typed SQLite source of truth with canonical records, revisions, entities, statuses, confidence, and validity windows. Adjacent memory-wiki and memory-lancedb work is relevant context, but it does not satisfy the default memory-core schema request.

## Best Possible Solution

Keep this open for maintainer-owned Memory MVP design and implementation. The likely path is versioned SQLite migrations in memory-host-sdk/memory-core, exported record/revision/entity/status/confidence/validity types, tests for canonical current rows and non-destructive memory_revisions, write-path integration, and docs explaining how Markdown memory, public artifacts, memory-wiki, memory-lancedb, and optional memory plugins interact. Coordinate with #42647, #42648, and #42650 before implementation.
