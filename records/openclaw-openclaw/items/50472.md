## Summary

Keep open. Current main has substantially better generic crash diagnostics, stability bundles, Feishu WebSocket retry handling, and channel monitor exit logging, but I still cannot prove the reported Windows Gateway process termination is fixed. The issue has multiple confirmations, the original report asks for explicit process-exit/termination evidence, and current production code still does not intercept arbitrary `process.exit()` or capture external Windows process kills.

## Best Possible Solution

Keep this open and narrow it to Windows Gateway termination diagnostics. The strongest fix would add an explicit gateway process-exit/termination trail for native Windows runs: guarded `process.exit` logging, `process.on('exit')` metadata, last memory/uptime/state snapshot, Scheduled Task/startup-script stdout/stderr capture where applicable, and regression coverage proving Feishu channel monitor exits are logged/restarted without making the Gateway port disappear. A follow-up reproduction on v2026.4.27/main should include install mode, PID/service state, latest gateway logs, stability bundle, and Task Scheduler Last Run Result.
