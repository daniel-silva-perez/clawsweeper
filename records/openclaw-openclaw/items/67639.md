## Summary

Keep this PR open. Current main still preserves the status scan source config at the overview/result layer, but the memory status path and memory runtime bootstrap still receive only the resolved config. The PR's core fix remains relevant, though it needs a current-main rebase and the unrelated extension/type cleanup should be split or justified.

## Best Possible Solution

Keep this PR open and review a scoped rebase. The best implementation is to thread `overview.sourceConfig` as internal `activationSourceConfig` through `executeStatusScanFromOverview`, `resolveStatusMemoryStatusSnapshot`, `resolveSharedMemoryStatusSnapshot`, `status.scan.deps.runtime`, and `getActiveMemorySearchManager`/`ensureMemoryRuntime`, while preserving current targeted memory-plugin loading and passing only documented fields to memory plugin runtime hooks. Split unrelated extension/type cleanup, then run the focused status and memory-runtime regression tests plus the changed gate.
