## Summary

Keep this PR open. Current main still preserves the original status scan source config at the overview/result layer, but it does not thread that source config into the status memory probe or memory runtime bootstrap path. The PR’s intended fix is still relevant and not fully superseded by the closed related memory-search work.

## Best Possible Solution

Keep this PR open and review a scoped rebase. The best implementation is to thread `overview.sourceConfig` as an internal `activationSourceConfig` through `executeStatusScanFromOverview`, `resolveStatusMemoryStatusSnapshot`, `resolveSharedMemoryStatusSnapshot`, the status runtime dependency wrapper, and `getActiveMemorySearchManager`/`ensureMemoryRuntime`, while passing only `{ cfg, agentId, purpose }` to `MemoryPluginRuntime` and omitting optional `purpose` when absent. Split unrelated extension/type cleanup, then run the focused status and memory-runtime regression tests plus `pnpm check:changed`.
