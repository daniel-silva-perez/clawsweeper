## Summary

Keep this PR open. It targets a real ACP run-session cleanup gap that current main still does not solve immediately, but the proposed 24-hour fallback can close a legitimate no-timeout ACP run that has not emitted a terminal lifecycle event.

## What This Changes

The PR adds lifecycle-triggered cleanup for ACP `mode: "run"` runtimes, dispatch-failure cleanup using the initialized runtime handle, removal of `runTimeoutSeconds` from ACP runtime initialization options, and regression tests in `src/agents/acp-spawn.test.ts`.

## Best Possible Solution

Keep the prompt lifecycle cleanup and dispatch-failure cleanup direction, but revise the fallback so it only closes after confirmed terminal task state or after an explicit timeout-derived grace period; no-timeout ACP runs must remain valid. Leave the broader parent-reset and orphan-process recovery work to #68916.

## Reproduction Assessment

Yes. A focused code-level reproduction is `sessions_spawn({ runtime: "acp", mode: "run", runTimeoutSeconds: 0 })` or omitted timeout with a missed lifecycle terminal event; the PR head will close that active runtime after 24 hours despite the documented no-timeout path.

## Solution Assessment

No. Lifecycle `end`/`error` cleanup and dispatch-failure cleanup are the right narrow direction, but the fixed wall-clock fallback is not the safest policy because it bypasses the existing timeout contract.

- **[P2] Preserve no-timeout ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1030-1032`

Gate this fallback on terminal task state or derive it from an explicit timeout instead of using a fixed wall-clock close.
