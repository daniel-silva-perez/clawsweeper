## Summary

Keep this PR open. Current main still lacks equivalent plugin logging best-practice guidance, and the PR is a focused docs-only change with no observed security-sensitive surface. It should be revised before merge because the proposed optional-chaining wording conflicts with the current SDK contract: `api.logger` is required, and only `debug` is optional.

## Best Possible Solution

Keep the PR open for revision or maintainer edit. Add concise plugin logging guidance to `docs/plugins/building-plugins.md` or the SDK docs, explain why plugin registration should avoid `console.log` when CLI commands emit JSON on stdout, and align the example with the current contract: `api.logger.info(...)`, `api.logger.warn(...)`, `api.logger.error(...)`, and `api.logger.debug?.(...)`.
