## Summary

Keep this PR open. The requested plugin logging guidance is still missing on current main, but the proposed docs are not merge-ready because they describe required logger fields as optional and use an `openclaw config schema` example that is too specific for the current manifest-only schema path.

## What This Changes

The PR adds a 32-line logging best-practices section to `docs/plugins/building-plugins.md` advising plugin developers to use `api.logger` instead of `console.log`.

## Best Possible Solution

Revise the docs patch into concise plugin logging guidance in the plugin guide or SDK docs: recommend `api.logger` over stdout-writing `console.log`, show `api.logger.info/warn/error(...)` and `api.logger.debug?.(...)`, and phrase JSON-stdout corruption as a general CLI-output risk unless a current command-specific reproduction is verified.

## Reproduction Assessment

Not applicable as a runtime reproduction for this docs-only PR. The docs defect is high-confidence by comparing the PR diff with the current `OpenClawPluginApi` and `PluginLogger` types plus the current `openclaw config schema` implementation path.

## Solution Assessment

No, not as written. The direction is useful, but the best maintainable fix is a narrower docs update aligned with the current SDK contract and current manifest-based schema behavior.
