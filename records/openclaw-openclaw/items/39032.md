## Summary

Keep open. Current main still lets failed tool/toolResult message text become subagent completion fallback output: the extractor copies tool result text without checking isError/is_error or status markers, then stores that text as latestRawText and can forward it into requester-visible task_completion delivery. The direct fix PR #39367 was closed unmerged, and the adjacent merged hardening in #59649 did not touch this extractor path.

## Best Possible Solution

Keep this issue open and land a narrow extractor fix in src/agents/subagent-announce-output.ts: ignore failed tool/toolResult messages before they can populate latestRawText, reuse or mirror existing isToolResultError/status semantics, and add focused regression tests proving failed tool results are skipped while successful tool-only fallback still works.

Fix the subagent completion output leak tracked by https://github.com/openclaw/openclaw/issues/39032. Current main still lets failed tool/toolResult message text become latestRawText in src/agents/subagent-announce-output.ts and then requester-visible task_completion fallback output. Add focused regression tests in the existing subagent announce output/format test area proving: failed tool result alone returns undefined or falls back to a safe status, failed tool result after visible assistant text does not override the assistant text, camelCase/snake_case error markers are skipped, status error/timeout is skipped, and successful tool-only fallback still returns the tool output. Do not broaden this into provider-level reasoning filtering, message-tool policy changes, delivery route changes, or task/status sanitization refactors. Changelog: because this is user-facing leak hardening, add one single-line fix entry in the active changelog section if repo policy expects one, crediting @cryptopoly and @mvanhorn; avoid detailed exploit wording.

## Risks / Open Questions

- The fix must preserve the documented successful tool/toolResult fallback while dropping only failed tool results.
- The guard should cover marker shapes used in this codebase: top-level isError, top-level or block-level is_error, and details.status values such as error or timeout.
- If a subagent emits debugging as normal assistant text rather than failed tool output, that is a separate reasoning/final-answer separation problem and should not be hidden inside this narrow extractor fix.
