## Summary

Current main still preserves the reported failure path: an explicit `--profile` updates `OPENCLAW_PROFILE` but does not clear or re-derive a stale inherited `OPENCLAW_LAUNCHD_LABEL`, while the macOS launchd resolver still gives that label precedence over the profile-derived label. `gateway status` reaches the platform service resolver, so status can still inspect the wrong plist. PR #66000 is an open, unmerged implementation candidate linked to this issue, so the issue should remain open until that PR is reviewed and landed or rejected.

## Best Possible Solution

Keep this issue open and review PR #66000 as the active implementation candidate. The best fix is a narrow CLI/profile change that prevents a conflicting inherited `OPENCLAW_LAUNCHD_LABEL` from overriding an explicit `--profile`, while preserving any intentional same-profile/custom-label behavior maintainers want, with focused regression tests for conflicting, matching, custom, empty, and absent label cases.
