## Summary

Current main still appears to have the reported macOS profile/launchd precedence bug: the profile parser applies `OPENCLAW_PROFILE` but leaves an inherited `OPENCLAW_LAUNCHD_LABEL`, and the launchd resolver still prefers that label. Because #66000 is an open, unmerged PR with closing syntax for this issue, the issue should stay open until that PR is reviewed, landed, or closed.

## What This Changes

The issue asks for explicit `--profile` invocations to avoid a stale inherited `OPENCLAW_LAUNCHD_LABEL` so `gateway status` resolves the profile-specific launchd plist.

## Best Possible Solution

Land #66000 or an equivalent narrow fix that makes an explicit CLI `--profile` take precedence over an inherited gateway launchd label for status/service resolution, with focused unit coverage for conflicting, matching, absent, and custom label cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to apply `--profile` into an env that already has `OPENCLAW_LAUNCHD_LABEL`; current main leaves the label in place, and launchd plist resolution then selects the inherited label before reading `OPENCLAW_PROFILE`.

## Solution Assessment

Yes. Fixing the CLI profile env boundary is the narrowest maintainable solution; changing global launchd precedence would risk breaking intentional custom-label overrides that current tests explicitly cover.
