## Summary

Keep open. Current main still has the Telegram media/proxy fallback work noted in the prior review, but the exact reported `getaddrinfo ENOTFOUND api.telegram.org` media-download path is not covered by the Telegram media fallback predicate, so closing as implemented would overstate the source evidence.

## What This Changes

The issue asks OpenClaw to make Telegram inbound image/media downloads reliable when the gateway process hits DNS failures for `api.telegram.org`, or to document the required DNS/proxy configuration.

## Best Possible Solution

Keep the issue open until maintainers choose and implement one explicit policy: either treat Telegram media `ENOTFOUND`/`EAI_AGAIN` as pre-connect fallback candidates with regression coverage, or document that media downloads require working host DNS or a configured `channels.telegram.proxy`. Avoid hardcoding public DNS servers unless that operational tradeoff is explicitly accepted.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction path: a media fetch failure shaped as `TypeError: fetch failed` with a nested `ENOTFOUND` code will not satisfy `shouldRetryTelegramTransportFallback`, so `fetchRemoteMedia` will not advance to the fallback dispatcher attempts. I did not live-reproduce the reporter's VPN DNS setup.

## Solution Assessment

Unclear. The requested hardcoded-DNS behavior is not the best default, but a narrow reuse of the existing Telegram transport fallback for pre-connect DNS errors may be maintainable if maintainers accept the network-policy tradeoff; otherwise docs should state the host DNS/proxy requirement explicitly.
