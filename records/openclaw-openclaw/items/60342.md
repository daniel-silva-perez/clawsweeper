## Summary

Keep open. Current main has substantial related Telegram media/proxy/DNS work, including proxy-aware media fetches, RFC 2544 fake-IP allowance, IPv4-first defaults, and pinned-IP fallback attempts. But the specific reported failure is `getaddrinfo ENOTFOUND api.telegram.org`, and the media transport fallback predicate does not include `ENOTFOUND`/`EAI_AGAIN`, so current main does not prove the issue is fixed without a configured proxy or host DNS change.

## Best Possible Solution

Keep this issue open. The best fix is to make a deliberate Telegram media policy decision: either treat `ENOTFOUND`/`EAI_AGAIN` as pre-connect media fallback candidates and add a regression test for `MediaFetchError ... getaddrinfo ENOTFOUND api.telegram.org`, or document that Telegram media downloads require working host DNS or an explicit `channels.telegram.proxy` configuration. Avoid silently hardcoding public DNS servers unless maintainers explicitly accept that operational/security tradeoff.
