## Summary

Keep this issue open. Current main has substantial Telegram media/network hardening and docs, but the exact reported `MediaFetchError ... getaddrinfo ENOTFOUND api.telegram.org` path is still not covered well enough to close as implemented.

## Best Possible Solution

Keep this issue open. The best fix is a deliberate Telegram media networking decision: either make media fetch fallback handle pre-DNS failures such as `ENOTFOUND`/`EAI_AGAIN` with regression coverage for the reported `MediaFetchError`, including any needed change so pinned-IP fallback can work when local DNS resolution fails, or document that Telegram media downloads require working host DNS or an explicit `channels.telegram.proxy` configuration.

## Evidence

A prior Codex review on the issue also kept it open for this exact gap.

## Risks / Open Questions

- Closing as implemented would overstate the evidence: the exact `ENOTFOUND` media-download failure is still not covered by the inspected retry predicate.
- Adding `ENOTFOUND` to the predicate alone may not fully fix media downloads, because the guarded media fetch currently performs local hostname resolution before the pinned dispatcher override is useful.
- Hardcoding public DNS servers would be an operational and security policy change; maintainers should choose that explicitly if they want it.
