## Summary

Keep open. The PR addresses a real gateway approval edge case, and the provided bot/human comments agree with the intended behavior, but current main still fast-fails the shared pending-approval path before the two-phase accepted response when there are no approval clients, no turn-source route, and no successful delivery. The branch also appears stale against current main because that fast-fail logic now lives in the shared approval helper rather than the old in-file exec handler.

## What This Changes

The PR adds a two-phase guard to the exec approval no-route fast-fail path and adds regression coverage for two-phase pending behavior while preserving single-phase fast-fail behavior.

## Best Possible Solution

Carry the same behavior into the current shared pending-approval helper: two-phase approvals should return `accepted` and remain pending until resolution or timeout when no approval client is currently connected, while single-phase callers continue to fail fast. Add focused coverage around the shared helper or current server-method path, and coordinate with #56581 before landing overlapping work.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a two-phase `exec.approval.request` with no approval clients, no successful forwarder, and no turn-source route; current main's shared helper expires that request with `no-approval-route` before it can send the two-phase accepted response.

## Solution Assessment

Yes for the original code path, but not as-is for current main. The narrow maintainable update is to apply the `!twoPhase` distinction in `handlePendingApprovalRequest` and adjust the regression test to the current shared/server-method structure.
