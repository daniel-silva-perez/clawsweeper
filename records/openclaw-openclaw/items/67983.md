## Summary

Keep open. Current main still does not expose the offered tool schemas on the `llm_input` plugin hook: the public event type, embedded runner emit path, hook test, and docs all describe/pass only the existing prompt/history/image fields. This PR is the focused implementation candidate, but it is open, unmerged, dirty against its base, and still has reviewable implementation risks before it should land.

## Best Possible Solution

Keep the PR open as the active implementation candidate. Before landing, update it to current main and make the hook payload safe and faithful: expose additive optional `tools` and `clientTools` types, emit cloned or otherwise immutable descriptors after effective tool/client-tool resolution, normalize descriptions consistently with the adapter, add coverage for populated `tools` and `clientTools`, regenerate the plugin SDK API baseline, and rerun the plugin SDK/API and targeted hook tests.

## Risks / Open Questions

- If landed as the current PR diff shows, `llm_input` handlers would receive mutable `parameters` object references. `toToolDefinitions` and `toClientToolDefinitions` also retain those parameter objects for the live model/tool path, so a logging or redaction plugin could accidentally mutate the schema used for the turn.
- The current PR diff emits missing built-in/client tool descriptions as `undefined`, while the adapter normalizes missing descriptions to `""`; tracing or eval plugins that compare hook payloads to effective tool definitions could see false drift.
- The PR is dirty against its recorded base and should be rebased or reconciled with current main before relying on its verification results.
