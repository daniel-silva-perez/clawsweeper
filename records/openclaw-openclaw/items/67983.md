## Summary

Keep this PR open. Current main does not already expose `tools` or `clientTools` on `llm_input`, but the proposed patch has blocking correctness issues: it passes live parameter object references to observation hooks and does not mirror the description normalization used by the actual tool-definition path.

## What This Changes

The PR adds optional `tools` and `clientTools` descriptors to `PluginHookLlmInputEvent`, populates them from embedded-runner tool lists, updates the wired hook test fixture, and refreshes plugin SDK API baseline hashes.

## Best Possible Solution

Expose the tool list as an additive plugin hook contract using detached descriptors that faithfully match the model-facing normalized tool definitions, including `description ?? ""`, safe cloned parameter schemas, and focused coverage for both built-in tools and hosted client tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Clone tool parameters before emitting llm_input:** `src/agents/pi-embedded-runner/run/attempt.ts:2048-2053`
  - body: The hook payload reuses `tool.parameters` and `tool.function.parameters` from the live tool definitions. Since the model path carries those same objects into `customTools` and `runVoidHook` passes the event directly to handlers, an observation hook that redacts or mutates `event.tools[*].parameters` can alter the schema used for this turn. Build detached parameter descriptors before calling the hook.
  - confidence: 0.91
- **[P2] Mirror normalized tool descriptions in the hook payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2047-2052`
  - body: This maps descriptions directly from raw built-in and client tool objects, but the model-facing path normalizes missing descriptions to `""`. Plugins using `llm_input` for tracing or eval reproducibility will see `undefined` where the actual offered tool definition has an empty string, so the event does not faithfully describe the model request.
  - confidence: 0.87
