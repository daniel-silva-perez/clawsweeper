## Summary

Keep this issue open. Current main has authenticated Gateway WebSocket chat, browser device pairing/auth, WebChat/Control UI, and provider-backed Talk/WebRTC voice, but it still does not implement the requested native no-third-party browser-held E2EE chat transport with per-agent/per-device encrypted channels. The existing maintainer Codex review in the discussion also explicitly kept this open, and the relevant files have not changed since that review point.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. The likely path is an additive Gateway/browser chat protocol with encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, isolated routing, transcript/storage rules, relay/fallback behavior, Control UI/WebChat UX, docs, and tests. If maintainers want plugin-first delivery, define the missing core/plugin seams first so an external E2EE browser transport can integrate without bypassing Gateway pairing, auth scopes, routing, or transcript rules.
