## Summary

Keep this issue open. Current main still has authenticated Gateway WebSocket chat, browser pairing/auth, WebChat/Control UI, and provider-backed Talk/WebRTC voice, but it does not implement the requested native browser-held E2EE browser chat transport with per-agent/per-device encrypted channels and no third-party dependency. The existing maintainer Codex review in the issue discussion also explicitly kept this open, and the relevant files are unchanged since that review point.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. The likely path is an additive Gateway/browser chat protocol with encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, isolated routing, transcript/storage rules, relay/fallback behavior, Control UI/WebChat UX, docs, and tests. If maintainers want plugin-first delivery, first define the missing core/plugin seams so an external E2EE browser transport can integrate without bypassing Gateway pairing, auth scopes, routing, or transcript rules.
