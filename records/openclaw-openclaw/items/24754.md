## Summary

Keep this issue open. Current main already has browser Control UI/WebChat, device pairing, PWA/Web Push, configurable DM session isolation, and provider-backed Talk/WebRTC voice, but it still does not implement the requested browser-held E2EE browser chat transport with encrypted message envelopes, browser key negotiation, or per-device/per-agent encrypted browser channels. The prior maintainer/Codex comment reached the same conclusion, and current main at ef58307f still matches that assessment.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. The best path is an additive Gateway/browser chat design covering encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, isolated routing, transcript/storage semantics, relay/fallback behavior, Control UI/WebChat UX, docs, and contract tests. If maintainers want plugin-first delivery, define the missing Gateway/plugin seams first so an external E2EE browser transport can integrate without bypassing core auth, routing, and transcript policy.
