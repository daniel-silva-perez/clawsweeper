## Summary

Keep this issue open. Current main still provides authenticated Gateway WebSocket WebChat/Control UI chat, browser device identity for pairing/auth, and provider-backed realtime Talk/voice, but not the requested native no-third-party browser-held E2EE chat transport with per-agent/per-device encrypted channels. The prior maintainer Codex review kept it open, and no relevant browser chat/Gateway protocol files changed after that review point.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. The likely path is an additive Gateway/browser chat protocol with encrypted envelopes and browser-held key negotiation, per-device and per-agent key lifecycle, isolated session routing, transcript/storage rules, NAT/relay fallback behavior, Control UI/WebChat UX, docs, and tests. If maintainers prefer plugin-first delivery, define the missing core/plugin seams first so an external E2EE browser transport can integrate without bypassing Gateway pairing, auth scopes, routing, or transcript rules.
