## Summary

Keep this issue open. Current main still does not provide the requested native, no-third-party, browser-held E2EE chat transport. The shipped browser chat path is authenticated Gateway WebSocket RPC with plaintext `message` payloads, WebCrypto device keys are used for Control UI/device identity rather than chat envelope encryption, and WebRTC is scoped to provider-backed realtime Talk/voice. The existing plugin/channel surfaces are useful, but this request still needs core Gateway/browser protocol and security design before it can be treated as ClawHub-only.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. The likely path is an additive Gateway/browser chat protocol with encrypted envelopes and key negotiation, browser-held per-device/per-agent key lifecycle, isolated session routing, transcript/storage semantics, NAT/relay fallback behavior, Control UI/WebChat UX, docs, and regression tests. If maintainers want plugin-first delivery, first define the missing core/plugin seams so an external E2EE browser transport can integrate without bypassing Gateway pairing, auth scopes, session isolation, or transcript rules.
