## Summary

Keep open. Current main handles the immediate HEARTBEAT_OK heartbeat acknowledgement path, but the broader cron/system-event no-op sentinel suppression requested here is still missing, and open PR #73785 is explicitly paired with this issue via closing syntax.

## What This Changes

Request runtime-level silent/no-op delivery handling for cron, heartbeat, and system-event runs so acknowledgements such as HEARTBEAT_OK, NO_NEW_AUDIO, and SESSION_WATCHDOG_OK are not posted to user-visible channels.

## Best Possible Solution

Land or replace a narrow core fix that centralizes exact no-op sentinel detection at the heartbeat/system-event delivery boundary, preserves meaningful exec or cron summaries, and adds regression coverage for HEARTBEAT_OK, NO_REPLY, NO_NEW_AUDIO, and SESSION_WATCHDOG_OK. A structured noReply/suppressDelivery result can remain a follow-up if exact sentinel matching proves too limited.

## Reproduction Assessment

Yes. The issue discussion gives a concrete reproduction trace for the HEARTBEAT_OK delivery gap, and current-main code inspection shows the remaining requested sentinels are not recognized anywhere under src or docs.

## Solution Assessment

Yes for the immediate bug class. The narrowest maintainable fix is to suppress exact no-op sentinels at the runtime delivery boundary while keeping substantive summaries visible; broader structured no-reply plumbing is useful but not required before closing this report.
