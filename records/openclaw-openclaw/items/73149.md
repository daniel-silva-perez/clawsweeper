## Summary

Keep open. Current main has heartbeat-specific ack suppression and NO_REPLY handling, but it does not implement the broader job-defined no-op sentinel suppression requested here, and PR #73785 remains open and unmerged with explicit pairing to this issue. The discussion contains a concrete reproduction trace, and current code/docs still show remaining work outside the shipped HEARTBEAT_OK/NO_REPLY subset.

## What This Changes

Request runtime-level silent/no-op delivery handling so cron, heartbeat, and system-event jobs do not post sentinel outputs such as HEARTBEAT_OK, NO_NEW_AUDIO, or SESSION_WATCHDOG_OK to user channels.

## Best Possible Solution

Review and land, revise, or replace PR #73785 with a narrow delivery-boundary fix that centralizes exact no-op sentinel detection for heartbeat/system-event outputs while preserving meaningful exec and cron summaries. A structured noReply/suppressDelivery flag can remain a follow-up if sentinel matching is not enough for future jobs.

## Reproduction Assessment

Yes. The discussion gives a concrete HEARTBEAT_OK channel-delivery trace, and current-main code inspection shows no detector for the requested NO_NEW_AUDIO or SESSION_WATCHDOG_OK sentinel outputs.

## Solution Assessment
