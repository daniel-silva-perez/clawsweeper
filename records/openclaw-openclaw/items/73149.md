## Summary

Keep open. Current main now has heartbeat-specific HEARTBEAT_OK ack suppression and isolated cron NO_REPLY suppression, but it still does not implement the broader job-defined no-op sentinel or structured no-reply delivery contract requested here; the provided GitHub context also shows paired PR #73785 is open and unmerged.

## What This Changes

The issue requests runtime-level silent/no-op delivery handling for cron, heartbeat, and system-event jobs so sentinel-only successful no-op outputs are not posted to user-visible channels.

## Best Possible Solution

Land, revise, or replace the paired implementation so no-op recognition is centralized at the delivery boundary for exact sentinel-only outputs, while keeping substantive summaries and structured/media replies deliverable. A structured noReply or suppressDelivery job-result field can be a follow-up if exact sentinel matching is not enough for future job types.

## Reproduction Assessment

Yes. The issue discussion provides a concrete heartbeat channel-delivery trace, and current-main source inspection confirms that the requested custom no-op sentinels are not recognized by the shared delivery helpers.

## Solution Assessment

Yes. Keeping the issue open while PR #73785 or an equivalent narrow replacement is reviewed is the right workflow; closing before merge would drop the canonical tracker for an unshipped fix.
