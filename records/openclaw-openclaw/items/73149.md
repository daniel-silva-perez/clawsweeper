## Summary

Keep open. Current main partially handles heartbeat acknowledgments, but the broader cron/system-event no-op sentinel suppression requested here is still unresolved and is paired with open, unmerged PR #73785. The discussion includes a concrete reproduction trace, and the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Request a runtime-level silent/no-op delivery mode for cron, heartbeat, and system-event runs so sentinel replies such as HEARTBEAT_OK, NO_NEW_AUDIO, and SESSION_WATCHDOG_OK are not posted to user channels.

## Best Possible Solution

Land or replace a narrow core fix that centralizes no-op sentinel detection at the heartbeat/system-event delivery boundary, preserves meaningful exec summaries, and adds regression coverage. A first-class structured noReply flag can remain a follow-up if sentinel matching proves too limited.

## Reproduction Assessment

Yes. The discussion gives a concrete reproduction trace for the HEARTBEAT_OK delivery gap, and current main still lacks a detector for NO_NEW_AUDIO and SESSION_WATCHDOG_OK no-op outputs, making the remaining sentinel leak path reproducible by code inspection.

## Solution Assessment

Yes for the immediate regression. The narrowest maintainable fix is to suppress exact no-op sentinels at the heartbeat/system-event delivery boundary while keeping substantive exec or cron summaries deliverable; broader structured no-reply plumbing is useful but not required to close this bug.
