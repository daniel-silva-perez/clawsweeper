## Summary

Keep open. Current main has heartbeat-specific HEARTBEAT_OK ack suppression and generic NO_REPLY handling, but it still lacks the broader job-defined no-op sentinel/suppress-delivery contract requested here, and the paired PR #73785 is open and unmerged in the provided context.

## What This Changes

The issue requests first-class silent/no-op delivery handling for cron, heartbeat, and system-event runs so sentinel-only or ack-like outputs such as HEARTBEAT_OK, NO_NEW_AUDIO, and SESSION_WATCHDOG_OK do not post to user channels.

## Best Possible Solution

Review and land, revise, or replace PR #73785 with a narrow delivery-boundary fix that centralizes exact no-op sentinel handling for heartbeat/system-event outputs while preserving meaningful cron and exec summaries. A structured noReply or suppressDelivery result flag can remain a follow-up if exact sentinel matching is not enough for future jobs.

## Reproduction Assessment

Yes. The issue discussion gives a concrete heartbeat channel-delivery trace, and current-main inspection confirms custom no-op sentinels like NO_NEW_AUDIO and SESSION_WATCHDOG_OK are not recognized in shared delivery helpers.

## Solution Assessment

Yes for the immediate workflow. Keeping this issue open while the paired PR is reviewed is the narrowest maintainable path; closing before that PR or an equivalent fix merges would drop the canonical tracking issue.
