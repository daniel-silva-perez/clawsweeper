## Summary

Keep this PR open because current main still lacks the requested `sessions_send` plugin hook, but the proposed patch should not merge as-is. The branch is dirty against main, and the delegated path has a real trust-boundary flaw plus an ordering regression before hook dispatch.

## What This Changes

The PR adds a typed plugin `sessions_send` hook to the plugin hook registry and SDK exports, invokes it from `sessions_send` before default core delivery, updates the generated SDK API baseline hash, and adds sessions tool tests for declined and delegated hook paths.

## Best Possible Solution

Land a rebased implementation that adds the generic typed hook seam, invokes plugin-owned handling before core-only history reads, derives delegated announce state only from trusted hook output or actual waited run output, preserves current A2A safeguards, and updates the public SDK hook documentation and exports together.

## Reproduction Assessment

Yes. The PR diff shows `readSessionsSendHookTask` copying raw `task.runtime.roundOneReply` from tool arguments and the delegated branch passing it to `startA2AFlow`, while the current A2A flow treats that value as the primary reply for ping-pong and announce context.

## Solution Assessment

No. The hook direction is plausible, but this patch is not the best form until it is rebased and the hook path runs before core baseline-history reads without trusting caller-supplied announce metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Stop trusting caller-supplied announce metadata:** `src/agents/tools/sessions-send-tool.ts:445`
  - body: In delegated mode this passes `hookTask?.runtime?.roundOneReply`, parsed from raw tool args, into `startA2AFlow`. Since tool args can include extra fields, a caller can seed the A2A ping-pong/announce content without it coming from the delegated run or trusted plugin result; use hook output or the waited reply instead.
  - confidence: 0.86
- **[P2] Invoke the hook before reading core reply history:** `src/agents/tools/sessions-send-tool.ts:414`
  - body: For waited sends, the code still takes `baselineReply` before `runSessionsSend`; if `chat.history` fails, a plugin that would handle direct or delegated delivery never runs. Move the core baseline read into the fallback/core-delegated wait path so hook-owned delivery is not blocked by core history access.
  - confidence: 0.78
- **[P3] Export the hook task and cancel-target types:** `src/plugin-sdk/core.ts:102-106`
  - body: The new public event/result types refer to `PluginHookSessionsSendTask` and `PluginHookSessionsSendCancelTarget`, but `plugin-sdk/core` only re-exports the context/event/result types. Plugin authors cannot name the task or cancel-target contract from the same SDK surface without type gymnastics.
  - confidence: 0.74
