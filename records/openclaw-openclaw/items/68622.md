## Summary

Keep this PR open. Current main still does not implement a typed `sessions_send` plugin hook or equivalent delegation seam: `sessions_send` is handled directly by the core tool path, and the current plugin hook catalog has `before_tool_call`, `before_dispatch`, and `reply_dispatch` but no hook that can claim delegated `sessions_send` dispatch metadata. The PR is focused on the missing core plugin API requested by #68548, but it needs maintainer review/rebase and the existing review findings addressed before it is mergeable.

## Best Possible Solution

Keep the PR open as the current implementation candidate for the missing `sessions_send` plugin API seam. The best path is to rebase it onto current main, preserve newer core `sessions_send` behavior, move hook invocation ahead of baseline reads where appropriate, make delegated announce metadata trusted, export the full public hook subtype set, remove duplicated local typing, regenerate the plugin SDK API baseline, and rerun the targeted sessions test plus `pnpm plugin-sdk:api:gen` and `pnpm build`.

## Risks / Open Questions

- The PR patch, as reviewed, forwards caller-supplied `task.runtime.roundOneReply` into the delegated announce flow. Because tool args are untrusted, the announce seed should come from trusted hook output or the actual waited reply before merge.
- The PR invokes the hook after the waited-send baseline history read, so a `chat.history` failure can still abort before a plugin-owned direct or delegated handler gets a chance to handle the request.
- The PR is out of date against current main's newer `sessions_send` behavior, including dynamic nested lane resolution and ACP parent A2A skip handling, so it needs a careful rebase rather than cleanup closure.
- Review comments also note SDK polish issues: missing exported `PluginHookSessionsSendTask` / `PluginHookSessionsSendCancelTarget` subtypes and duplicated internal task typing.
