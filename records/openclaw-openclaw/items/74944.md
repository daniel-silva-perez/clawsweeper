## Summary

Keep open. Current main has partial update hardening, but the global update command environment still does not add the active Node directory or Homebrew Node paths needed for npm shims under launchd's minimal PATH, and no regression test covers the reported macOS Homebrew failure mode.

## What This Changes

The issue asks the macOS launchd/self-update path for Homebrew global npm installs to run npm with a PATH that can resolve Homebrew Node, avoiding npm exit 127 and retry loops.

## Best Possible Solution

Centralize update subprocess PATH construction so package-manager updates and any launchd update handoff include the active runtime directory, the owning package-manager prefix bin, Homebrew Node/bin locations, and safe system dirs without weakening service PATH hardening or trusting project-local bins; add regression coverage and a changelog entry for the macOS Homebrew update failure.

## Reproduction Assessment

Yes, for the target environment: a macOS Homebrew global npm install running `update.run` through launchd with PATH `/usr/bin:/bin:/usr/sbin:/sbin` should reproduce the npm shim failure. I did not execute launchd locally, so live confirmation remains part of fix validation.

## Solution Assessment

Unclear. The suggested shell `export PATH=...` is directionally correct, but the maintainable fix is a shared update environment helper used by both `update.run` global installs and any launchd handoff script, with tests for npm shims resolving Homebrew Node.
