## Summary

Keep open. Current main still has a plausible updater PATH gap for launchd/minimal environments: the global package update env copies the existing PATH and only adds portable Git, Windows package-install, and Corepack settings, while the separate helper that adds the active runtime and Homebrew dirs is not used by the update subprocess path. The report is recent, concrete, and narrow enough for a focused repair rather than cleanup closure.

## What This Changes

Fix macOS launchd-triggered self-updates so Homebrew npm shims can resolve Node under launchd's minimal PATH and avoid repeated npm exit 127 update failures.

## Best Possible Solution

Centralize update subprocess PATH construction so package-manager updates include the active runtime directory, the owning npm prefix bin, Homebrew Node/bin locations, and safe system dirs without adding project-local bins or weakening service PATH hardening; cover the macOS Homebrew launchd case and add a user-facing changelog fix.

Yes. A macOS Homebrew global npm install running the update path under launchd's default PATH `/usr/bin:/bin:/usr/sbin:/sbin` should reproduce the npm shim failing to resolve `node`; live launchd proof remains part of fix validation.

Unclear for the issue's suggested fix. Adding a hard-coded shell `export PATH` would address the immediate script, but the maintainable fix is a shared update-env helper used by `update.run`, CLI package updates, and any launchd handoff script that starts npm.

## Repair Work Prompt

Fix the macOS Homebrew launchd self-update failure reported at https://github.com/openclaw/openclaw/issues/74944. The observable bug is that update-triggered global npm installs can run under launchd's minimal PATH while invoking an absolute Homebrew npm shim whose shebang uses `env node`, causing `env: node: No such file or directory` and npm exit 127. Implement a shared update subprocess PATH/env helper, likely near `src/infra/update-global.ts` or `src/infra/path-env.ts`, and use it from `createGlobalInstallEnv` / `runGatewayUpdate` and the CLI package update path so package-manager subprocesses include the active runtime dir, the owning npm prefix bin, Homebrew Node/bin candidates such as `/opt/homebrew/bin`, `/opt/homebrew/opt/node/bin`, `/usr/local/bin`, `/usr/local/opt/node/bin`, and safe system dirs. Preserve service PATH hardening: do not put project-local `node_modules/.bin` ahead of trusted dirs, do not trust workspace-derived paths, and keep user/package-manager dirs ordered deliberately. Add focused tests for a darwin launchd minimal PATH plus Homebrew npm prefix where the install env must let npm resolve Node, and update CHANGELOG.md with a single-line user-facing fix entry crediting @jrsdhr. Do not change dependency versions or release metadata.
