## Summary

Keep open. Current main still leaves the global package-update subprocess environment without the active Node/Homebrew runtime path needed for npm shims under launchd's minimal PATH, so the reported Homebrew macOS failure is still actionable and narrow.

## What This Changes

Fix launchd-triggered self-update for macOS Homebrew global npm installs so npm can resolve `node` and the update does not retry into a gateway loop.

## Best Possible Solution

Centralize update subprocess PATH construction so gateway `update.run`, CLI package updates, and any launchd handoff use a shared env that includes the active runtime dir, safe system dirs, the owning npm prefix bin, and Homebrew Node/bin locations without trusting project-local bins or weakening service PATH hardening.

## Reproduction Assessment

Yes. A macOS Homebrew global npm install running `update.run` from launchd with PATH `/usr/bin:/bin:/usr/sbin:/sbin` should reproduce the npm shim `env node` failure; the report provides the exact log signature and runtime paths.

## Solution Assessment

No, the literal hard-coded shell `export PATH=...` is only a workaround. The maintainable solution is a shared update env helper with regression coverage for minimal launchd PATH and Homebrew npm shims.
