## Summary

Keep open. Current main still leaves the global package-update subprocess environment without an updater-specific PATH repair for Homebrew Node/npm shim resolution, while the existing restart-on-success guard only mitigates the crash-loop side effect. The report is concrete, narrow, and suitable for a focused repair PR.

## What This Changes

The issue asks OpenClaw to make macOS launchd-triggered self-updates resolve Homebrew Node when invoking the global npm updater from a minimal launchd PATH.

## Best Possible Solution

Introduce a shared update subprocess env builder that preserves service hardening while adding the active runtime directory, the owning package-manager prefix bin, Homebrew Node/bin locations, and trusted system dirs for package updates; wire it through CLI and gateway global-update paths, add regression coverage, and add a changelog fix crediting the reporter.

## Reproduction Assessment

Yes. A macOS Homebrew global npm install running the package-update path with launchd’s default PATH should reproduce the npm shim failure because `npm` can be invoked by absolute path while its shebang still needs `node` on PATH.

## Solution Assessment

No, the literal suggested shell export is not the most maintainable fix by itself. The better solution is a shared, tested update-env helper used by both CLI and gateway package-update paths without adding project-local bins or weakening service PATH rules.
