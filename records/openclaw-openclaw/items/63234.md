## Summary

Keep open. Current main still has the generic plugin-loader paths that validate `entry?.config` and fall back to `{}` before logging the exact missing-required-field error shape, and the current tests/docs only cover genuinely invalid plugin-local config degrading instead of valid required config being preserved through startup, active-registry reuse, and `tools.catalog`. The exact UnraidClaw plugin is not present in this repository, so the right fix is a synthetic generic regression rather than closing as implemented.

## What This Changes

Investigate and fix startup/post-ready plugin loading so a valid `plugins.entries.unraidclaw.config` is not repeatedly observed as missing and logged as invalid during gateway startup.

## Best Possible Solution

Land a narrow generic regression fix that preserves resolved `plugins.entries.<id>.config` for required native tool plugins across gateway startup, active registry reuse, `tools.catalog`, and post-ready plugin tool resolution, and fails if a valid config is later observed as `{}` or repeated invalid-config logs are emitted.
