## Summary

Current main has related generic fixes for plugin-local invalid config degradation and gateway-bound plugin registry reuse, but I do not see enough proof to close this exact report. The reported UnraidClaw plugin is not present in the source tree, the observed log prefix matches the plugin loader path, and current loader code can still emit `[plugins] <id> invalid config` whenever a repeated plugin load sees missing or invalid `entry.config`. This should stay open with a narrow synthetic-plugin regression/fix PR candidate.

## Best Possible Solution

Keep this open and add a narrow generic regression/fix: use a synthetic tool plugin with a schema requiring `serverUrl` and `apiKey`, configure it under `plugins.entries.<id>.config`, simulate gateway startup plus post-ready tool inventory/catalog paths, and ensure the configured plugin is not reloaded with an empty config or repeatedly logged as invalid. If current main already passes, land focused regression coverage and reference the existing generic fixes rather than adding Unraid-specific core behavior.
