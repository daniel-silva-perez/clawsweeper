## Summary

Keep open. Current main still has adjacent plugin-local invalid-config degradation and registry-reuse work, but it does not prove the reported valid-config startup spam is fixed: UnraidClaw is absent from this tree, the loader still emits the reported invalid-config path when a load sees missing config, and there is no synthetic required-config startup plus post-ready tool/catalog regression. A fork commit referenced from the issue timeline proposes log dedupe for this issue, but it is not on current main.

## What This Changes

The issue requests that a required-config tool plugin such as UnraidClaw stop repeatedly logging invalid config during gateway startup when its configured serverUrl and apiKey are valid and the plugin eventually registers successfully.

## Best Possible Solution

Keep this open and land a narrow generic regression/fix. Add a synthetic native tool plugin with required `serverUrl` and `apiKey`, configure it under `plugins.entries.<id>.config`, exercise gateway startup plus post-ready tool catalog/tool resolution paths, and assert the plugin is not reloaded with empty config and does not emit repeated invalid-config logs. The fork commit 4cbb5897 can be used as a reference for log dedupe, but the final fix should prove valid config propagation rather than only suppressing symptoms.
