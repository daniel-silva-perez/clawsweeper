## Summary

Keep open. Current main has adjacent fixes for plugin-local invalid-config degradation, gateway-bindable registry reuse, provider-discovery startup cost, and QMD startup laziness, but it still does not prove the reported valid-config startup spam is fixed. UnraidClaw is not present in this source tree, the plugin loader still validates `entry?.config` and emits the reported `[plugins] <id> invalid config` shape when a load sees missing config, and current tests cover adjacent registry reuse rather than a required-config tool plugin through startup plus post-ready tool/catalog paths.

## Best Possible Solution

Keep this open and land a narrow generic regression/fix. Add a synthetic native tool plugin whose config schema requires `serverUrl` and `apiKey`, configure it under `plugins.entries.<id>.config`, simulate Gateway startup plus the post-ready tool inventory/catalog path that can trigger plugin tool resolution or registry ensures, and assert the plugin is not reloaded with empty config and does not emit repeated invalid-config logs. If current main already passes, land the focused regression coverage and reference the existing generic startup/config fixes instead of adding behavior.
