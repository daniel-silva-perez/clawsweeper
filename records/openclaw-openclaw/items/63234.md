## Summary

Keep open. Current main still has generic plugin-loader paths that can emit the reported missing-required-field `invalid config` message when a required plugin config is observed as missing, while the current docs/tests mainly prove graceful handling of genuinely invalid plugin-local config rather than this issue’s valid-config-is-lost startup/catalog case.

## What This Changes

Investigate repeated UnraidClaw startup `invalid config` logs even though `plugins.entries.unraidclaw.config` contains the required values and the plugin later registers its tools successfully.

## Best Possible Solution

Land a narrow generic regression fix that preserves resolved `plugins.entries.<id>.config` for required native tool plugins across gateway startup, active registry reuse, `tools.catalog`, and post-ready plugin tool resolution, with coverage that fails if valid config is later observed as `{}` or if repeated invalid-config logs are emitted.
