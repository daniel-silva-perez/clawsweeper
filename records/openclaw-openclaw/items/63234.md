## Summary

Keep open. Current main still has the generic plugin-config validation paths that emit the reported missing-required-field error when a load sees an empty or missing plugin config, and the current invalid-plugin degradation work only covers genuinely invalid plugin-local config. There is still no UnraidClaw implementation or synthetic required-config startup/catalog regression proving a valid `plugins.entries.<id>.config` is preserved across the phases implicated by the report.

## What This Changes

The issue asks OpenClaw to stop repeatedly logging invalid-config errors during gateway startup when a plugin has valid required config under `plugins.entries.<id>.config` and later registers tools successfully.

## Best Possible Solution

Land a narrow generic regression fix that preserves resolved `plugins.entries.<id>.config` for required native tool plugins across gateway startup, active registry reuse, `tools.catalog`, and post-ready plugin tool resolution, with coverage that fails if valid config is later observed as `{}` or if repeated invalid-config logs are emitted.
