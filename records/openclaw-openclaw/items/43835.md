## Summary

Keep this PR open, but do not merge it as-is. Current main still leaves Docker/local sandbox workspace directories behind during prune, so the linked bug remains real; however, the PR adds recursive deletion with incomplete runtime-success, stored-scope, workspace-root, and active-reference guards, and the current backend-aware prune path has diverged from the PR's older base.

## What This Changes

The PR adds recursive cleanup of resolved sandbox workspace directories during sandbox-container prune, skips shared scope, and attempts to avoid deleting agent-scope workspaces that still have another registry entry.

## Best Possible Solution

Update or replace the branch with backend-aware sandbox-container workspace cleanup that deletes only a recorded, owned, inactive workspace after confirmed runtime removal, skips shared and still-referenced scopes, handles Docker/SSH/OpenShell semantics deliberately, and adds focused regression tests for scope-key, active-reference, and stale-root cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for workspace cleanup:** `src/agents/sandbox/prune.ts:66`
  - body: `entry.sessionKey` is already the normalized registry scope key. In agent scope, passing a stored key like `agent:ops` back through `resolveSandboxScopeKey()` treats it as malformed and can resolve to `agent:main`, deleting the wrong workspace while leaving the leaked one behind.
  - confidence: 0.91
- **[P1] Confirm runtime removal before deleting the workspace:** `src/agents/sandbox/prune.ts:83`
  - body: The cleanup runs after container removal is attempted, but Docker removal uses an allow-failure path and does not prove the runtime is gone. If the daemon is unavailable or removal fails, this can recursively delete a workspace still mounted by a live runtime.
  - confidence: 0.88
- **[P2] Guard cleanup against every remaining entry for the scope:** `src/agents/sandbox/prune.ts:70-75`
  - body: The remaining-entry check only covers agent scope and only re-reads the same registry. Session-scope duplicate entries and browser entries can still reference the same workspace, so pruning one stale entry can delete a workspace still in use.
  - confidence: 0.83
- **[P2] Persist the workspace root before pruning directories:** `src/agents/sandbox/prune.ts:82`
  - body: The PR derives the cleanup path from current `params.cfg.workspaceRoot`, but registry entries can outlive config or per-agent root changes. That can leave the original stale directory behind or delete a matching scope directory under the wrong current root.
  - confidence: 0.79
