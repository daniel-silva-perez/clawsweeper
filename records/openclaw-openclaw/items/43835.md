## Summary

Keep this PR open as a relevant implementation attempt for the still-open sandbox workspace leak, but the current patch is not merge-ready. Current main still prunes runtimes and registry entries without deleting local sandbox workspace directories, while this PR adds recursive deletion with unsafe scope-key, runtime-success, active-entry, and workspace-root semantics.

## What This Changes

The PR adds `fs.rm(..., { recursive: true, force: true })` cleanup of resolved sandbox workspace directories after sandbox-container prune, guarded away from shared scope and browser prune calls.

## Best Possible Solution

Keep this PR as useful source work, but require an update against current main before merge. The end state should be backend-aware sandbox-container workspace cleanup that deletes only a recorded, owned, inactive local workspace after runtime removal is confirmed safe, skips shared and still-referenced scopes, and includes focused regression tests for the unsafe cases already identified in review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for workspace cleanup:** `src/agents/sandbox/prune.ts:68`
  - body: `entry.sessionKey` is already the normalized registry scope key. In agent scope, passing a stored key like `agent:ops` back through `resolveSandboxScopeKey()` resolves it as `agent:main`, so prune can delete the wrong workspace while leaving the leaked one behind. Use the stored scope key directly, or store raw and normalized keys separately.
  - confidence: 0.91
- **[P1] Confirm runtime removal before deleting the workspace:** `src/agents/sandbox/prune.ts:83`
  - body: The patch deletes the workspace after `docker rm -f` is invoked with `allowFailure: true`. If Docker removal fails or the daemon is unavailable, the live container can still exist with this directory mounted while `fs.rm(..., { recursive: true })` removes its workspace. Gate cleanup on confirmed runtime removal.
  - confidence: 0.88
- **[P2] Guard cleanup against all remaining entries for the scope:** `src/agents/sandbox/prune.ts:70-72`
  - body: The remaining-entry check only runs for `agent` scope and only reads the sandbox-container registry. Session-scope duplicate entries and browser registry entries can still reference the same workspace, so pruning one stale entry can delete a workspace still used by another runtime. Check all entries for the same stored scope before removal.
  - confidence: 0.82
- **[P2] Persist the workspace root used by each registry entry:** `src/agents/sandbox/prune.ts:82`
  - body: The cleanup path is derived from the current `params.cfg.workspaceRoot`, but registry entries can outlive config or per-agent root changes. That means prune can leave the original stale directory behind or delete a matching scope directory under the wrong current root. Record the root/path with the entry or otherwise prove it still matches before deleting.
  - confidence: 0.79
