## Summary

Keep this PR open, but do not merge it as-is. The linked Docker sandbox workspace leak still exists on current main, and the branch usefully narrows cleanup to container pruning, but the proposed cleanup can target the wrong workspace in agent scope and can delete a workspace even when Docker container removal failed.

## What This Changes

The PR adds recursive sandbox workspace directory cleanup to the sandbox prune path, enabled for sandbox container pruning while skipping browser and shared-scope cleanup.

## Best Possible Solution

Land a current-main fix that treats workspace deletion as part of Docker sandbox runtime cleanup: delete only the intended non-shared sandbox workspace after the owning Docker container is confirmed removed, use the stored registry scope key, guard against remaining registry entries sharing that workspace, and cover session, agent, shared, browser, and failed-removal cases with tests.

## Reproduction Assessment

Yes. A source-level repro is enough: create an agent-scope registry entry whose stored `sessionKey` is `agent:ops`; the PR recomputes that as `agent:main` and targets the wrong directory, and a nonzero `docker rm -f` under `allowFailure: true` still reaches the new `fs.rm` call.

## Solution Assessment

No. The direction is valid, but the PR is not the narrowest safe fix on current main; cleanup should be tied to successful Docker runtime removal and stored scope/workspace identity instead of being bolted onto the shared prune helper with recomputed scope keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for workspace cleanup:** `src/agents/sandbox/prune.ts:68`
  - body: Registry entries already store the normalized scope key as `entry.sessionKey`. In agent scope, recomputing it turns a stored key like `agent:ops` into `agent:main`, so prune can delete the main workspace while leaving the stale agent workspace behind.
  - confidence: 0.93
- **[P1] Only delete workspaces after Docker removal succeeds:** `src/agents/sandbox/prune.ts:83`
  - body: `execDocker(..., { allowFailure: true })` can return a failed result without throwing, but the new cleanup still runs afterward. If Docker is unavailable or `rm -f` fails, this can remove a workspace while the container still exists or is still mounted.
  - confidence: 0.9
- **[P2] Check remaining entries for every cleaned scope:** `src/agents/sandbox/prune.ts:70`
  - body: The duplicate-workspace guard only runs for agent scope, but session scope can also have multiple entries with the same session key after a container-prefix/name change. Pruning the stale entry would delete the workspace used by the newer entry.
  - confidence: 0.78
