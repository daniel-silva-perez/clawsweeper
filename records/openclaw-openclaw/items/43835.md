## Summary

Keep this PR open. Current main still prunes sandbox runtimes and registry entries without deleting local sandbox workspace directories, so the linked disk-leak bug remains real. The PR is a relevant implementation candidate, but it is merge-dirty against current main and its destructive recursive cleanup still needs safer backend-aware semantics plus regression tests before it can merge.

## Best Possible Solution

Keep this PR open as a relevant implementation candidate for #43797, but require a current-main update before merge. The best fix is a backend-aware, container-prune-only cleanup that uses stored workspace identity and the original workspace root, skips shared or still-active same-scope cases, deletes only after runtime removal is known safe, and adds focused regression tests for session, agent, shared, browser, backend failure, duplicate-entry, and changed-workspaceRoot cases. If a safer patch lands elsewhere, close this PR as superseded by that merged fix.

## Risks / Open Questions

- Merging the PR as-is could recursively delete a workspace that is still used by another active sandbox, browser runtime, duplicate session entry, or agent-scoped runtime.
- The branch is dirty against current main, whose prune path is now backend-aware.
- The registry does not persist the original workspace root, so cleanup based on current config can miss stale directories or target the wrong path after config changes.
- No focused regression coverage currently proves safe workspace cleanup behavior for the risky prune cases.
