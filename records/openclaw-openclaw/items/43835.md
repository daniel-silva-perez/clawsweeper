## Summary

Keep open. Current main still has the sandbox workspace leak, but this PR branch is dirty against current main and the proposed recursive cleanup can delete the wrong or still-active workspace unless the scope key, workspace root, remaining-entry guard, and Docker-removal success cases are fixed first.

## What This Changes

The branch adds recursive non-shared sandbox workspace directory cleanup to the sandbox prune helper and enables it for sandbox container pruning.

## Best Possible Solution

Land a current-main sandbox fix that deletes only sandbox-owned non-shared Docker workspace directories after confirmed runtime removal, uses the stored registry scope/workspace identity, skips workspaces still referenced by any remaining runtime entry, and covers automatic prune plus recreate/config-refresh behavior with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an expired Docker sandbox registry entry is pruned, but current main only removes the container and registry entry; for the PR bugs, an agent-scope stored key like agent:ops or a failed docker rm under allowFailure demonstrates wrong-target or premature deletion.

## Solution Assessment

No. Cleaning stale sandbox workspaces is the right direction, but this patch is not the safest implementation because it recomputes stored identity, deletes after an allow-failure removal path, and only partially guards shared workspace use.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for workspace cleanup:** `src/agents/sandbox/prune.ts:68`
  - body: Registry entries already store the normalized scope key in `entry.sessionKey`. Recomputing it in agent scope treats a stored key like `agent:ops` as malformed and falls back to `agent:main`, so prune can delete the wrong workspace while leaving the stale one behind.
  - confidence: 0.93
- **[P1] Gate workspace deletion on successful Docker removal:** `src/agents/sandbox/prune.ts:83`
  - body: `execDocker(..., { allowFailure: true })` returns a nonzero result instead of throwing, but the new cleanup still runs afterward. If Docker is down or `rm -f` fails, this can remove the workspace while the container still exists or still has the mount.
  - confidence: 0.9
- **[P2] Check remaining entries before deleting any scoped workspace:** `src/agents/sandbox/prune.ts:70`
  - body: The remaining-entry guard only runs for `agent` scope, but `session` scope can also have multiple entries sharing a scope key after a container prefix/name change. Pruning the stale entry would delete the workspace used by the newer entry.
  - confidence: 0.78
- **[P2] Delete the workspace root recorded for the stale entry:** `src/agents/sandbox/prune.ts:82`
  - body: The cleanup uses the current `params.cfg.workspaceRoot` for every registry entry, but per-agent sandbox config can change the workspace root. A stale entry created under an older root can therefore leave the real directory behind and delete an unrelated directory under the current root.
  - confidence: 0.75
