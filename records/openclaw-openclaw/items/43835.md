## Summary

Keep this PR open. Current main still prunes sandbox runtimes and registry entries without removing the local sandbox workspace directories, so the linked disk-leak bug remains unresolved. The PR is relevant, but it is dirty against current main and its recursive workspace deletion still needs safer current-main semantics and regression tests before it can merge.

## Best Possible Solution

Keep this PR open as a relevant implementation candidate for #43797, but require a current-main update before merge. The best fix is a backend-aware, container-prune-only cleanup that uses stored workspace identity and the original workspace root, skips shared or still-active shared-scope cases, deletes only after runtime removal is known safe, and adds focused regression coverage for session, agent, shared, browser, backend, duplicate-entry, and failure cases. If a safer patch lands elsewhere, close this PR as superseded by that merged fix.

## Risks / Open Questions

- Recursive workspace deletion can remove a workspace still used by another active sandbox, browser runtime, duplicate session entry, or agent-scoped runtime if guards are incomplete.
- The PR is merge-dirty against current main, whose prune path is now backend-aware.
- The registry entry does not persist the original workspace root, so a cleanup based on current config can delete the wrong directory or leave the stale one behind after config changes.
- No current regression test covers sandbox prune workspace cleanup, including session, agent, shared, browser, backend failure, duplicate-entry, and changed workspaceRoot cases.
