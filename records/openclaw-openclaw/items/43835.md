## Summary

Keep this PR open for maintainer review. The linked disk-leak bug still appears valid on current main, but this patch is not merge-safe because it adds recursive workspace deletion without reliable scope/root ownership, full active-reference checks, or confirmed runtime removal.

## What This Changes

The PR adds recursive deletion of resolved sandbox workspace directories during sandbox-container pruning, skips shared scope, and adds an agent-scope remaining-entry check.

## Best Possible Solution

Update or replace the branch with backend-aware sandbox-container cleanup that deletes only an owned, recorded, inactive workspace after confirmed runtime removal, skips shared and still-referenced scopes across container and browser registries, preserves the workspace root used by the entry, and adds focused regression coverage for scope-key, active-reference, and removal-failure cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for cleanup:** `src/agents/sandbox/prune.ts:65`
  - body: `entry.sessionKey` is already the normalized registry scope key. In agent scope, passing a stored key such as `agent:ops` back through `resolveSandboxScopeKey()` can resolve to `agent:main`, deleting the wrong directory while leaving the leaked workspace behind.
  - confidence: 0.91
- **[P1] Delete only after confirmed runtime removal:** `src/agents/sandbox/prune.ts:80`
  - body: The workspace cleanup can run after a failed runtime removal because Docker removal uses `allowFailure` and swallows errors. That can delete a workspace still mounted by a live container; gate deletion on confirmed backend removal.
  - confidence: 0.88
- **[P2] Check every active reference before deleting:** `src/agents/sandbox/prune.ts:67-76`
  - body: The remaining-entry guard only checks agent entries in the main registry. Session-scope duplicate entries and browser registry entries can reference the same workspace, so pruning one stale entry can still delete state used by another runtime.
  - confidence: 0.84
- **[P2] Persist the workspace root before deleting:** `src/agents/sandbox/prune.ts:79`
  - body: The cleanup path is derived from the current `cfg.workspaceRoot`, but registry entries can outlive config or per-agent root changes. That can leave the original stale directory behind or remove a matching scope under the wrong root.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff introduces data-loss-sensitive recursive deletion of sandbox workspace state without sufficient ownership, runtime-success, and active-reference guards.

Concerns:

- **[medium] Recursive deletion can target live or wrong workspace:** `src/agents/sandbox/prune.ts:80`
  - body: `fs.rm(workspaceDir, { recursive: true, force: true })` is security-sensitive in sandbox cleanup because the path is recomputed from current config and deletion is not gated on confirmed runtime removal or every active reference to the stored scope.
  - confidence: 0.88
