## Summary

Keep this PR open, but do not merge it as-is. Current main still leaves Docker sandbox workspace directories behind during prune, so the requested cleanup is valid; the PR head has blocking data-loss risks around scope-key recomputation, deletion after unconfirmed Docker removal, incomplete active-reference checks, and stale pre-backend-manager prune code.

## What This Changes

The PR branch adds recursive deletion of non-shared sandbox workspace directories after expired sandbox containers are pruned.

## Best Possible Solution

Land a current-main fix that cleans up only Docker-owned non-shared sandbox workspaces after confirmed container removal, uses the stored registry scope identity, skips any workspace still referenced by remaining sandbox or browser registry entries, and covers prune behavior with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is clear: configure Docker sandbox pruning, let a registry entry expire, and current main removes the runtime/registry entry without deleting resolveSandboxWorkspaceDir for the stored scope key. The PR regressions are also reproducible from source by combining stored agent-scope keys, allowFailure Docker removal, or duplicate remaining entries.

## Solution Assessment

No. Cleaning stale Docker workspaces is the right direction, but this PR is not the best implementation because it recomputes stored scope identity, deletes after an allow-failure removal path, only partially checks active users of a workspace, and is not adapted to current main's backend-manager prune code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for cleanup:** `src/agents/sandbox/prune.ts:68`
  - body: Registry entries already store the normalized scope key in entry.sessionKey. Recomputing agent scope from a stored key like agent:ops can resolve as agent:main, deleting the wrong workspace while leaving the stale one behind.
  - confidence: 0.93
- **[P1] Delete workspaces only after confirmed runtime removal:** `src/agents/sandbox/prune.ts:83`
  - body: docker rm -f is invoked with allowFailure, so a nonzero Docker result does not stop the new fs.rm call. If Docker is down or removal fails, prune can delete a workspace while the container still exists or still has the mount.
  - confidence: 0.9
- **[P2] Check all remaining entries before deleting a scope:** `src/agents/sandbox/prune.ts:70-76`
  - body: The remaining-entry guard only applies to agent scope and only reads the same registry. Session-scope duplicate entries and browser entries can still reference the same workspace, so pruning one stale entry can remove a workspace still in use.
  - confidence: 0.84
- **[P2] Avoid assuming the current workspace root owns stale entries:** `src/agents/sandbox/prune.ts:82-83`
  - body: The cleanup builds the path from params.cfg.workspaceRoot for every stale entry. If the entry was created under a previous per-agent workspace root, prune can miss the real stale directory and delete an unrelated current-root directory with the same scope key.
  - confidence: 0.76
