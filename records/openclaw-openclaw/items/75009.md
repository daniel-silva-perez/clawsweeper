## Summary

Keep open. Current main does not implement the WhatsApp-specific DM visible-replies config, and this active contributor PR is the implementation path; the diff appears narrow and I did not find a blocking correctness or security finding during the read-only review.

## What This Changes

The PR adds an opt-in WhatsApp direct-chat `dmVisibleReplies` mode with per-account overrides, routes opted-in DMs through message-tool-only delivery with a same-chat final fallback, and updates prompt text, docs, config schema/UI hints, and targeted tests.

## Best Possible Solution

Land the feature after normal maintainer review and CI/Testbox validation if the branch remains consistent with the reviewed diff: keep `automatic` as the default, keep WhatsApp-specific policy in the WhatsApp plugin/account config, and retain the shared suppressed-final result only as an opt-in fallback hook.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a WhatsApp direct chat with `channels.whatsapp.dmVisibleReplies: "message_tool"`, process an inbound DM, and verify shared dispatch receives `sourceReplyDeliveryMode: "message_tool_only"` while same-chat `message(action=send)` suppresses fallback final text.

## Solution Assessment

Yes, with normal maintainer review still required. This is a narrow maintainable solution because it avoids changing global `messages.visibleReplies`, scopes the behavior to WhatsApp accounts, and reuses the existing source-reply delivery and same-target message-tool suppression seams.
