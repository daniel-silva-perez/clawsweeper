## Summary

Keep this PR open. Current main still has the reported audit behavior and the related duplicate PRs were closed rather than merged, but the current PR head is not safe to land as-is because its shared loopback-equivalence classifier can suppress critical gateway exposure findings for binds that may resolve or fall back to `0.0.0.0`.

## What This Changes

The PR adds a gateway bind classifier, uses it to tier several gateway security-audit findings, downgrades `gateway.trusted_proxies_missing` to `info` for loopback-equivalent binds, emits `warn` for non-loopback empty `trustedProxies`, and adds regression tests.

## Best Possible Solution

Revise the PR toward a conservative effective-bind audit model: downgrade only definitely local-only exposure, fail closed for ambiguous `auto` and fallback-prone `custom` binds, keep the trusted-proxy, auth, and Control UI origin findings coherent, and update the public audit-check docs if severity changes.

## Reproduction Assessment

Yes. Current-main behavior is reproducible by inspecting `collectGatewayConfigFindings` for a loopback Control UI config with empty `trustedProxies`; the PR regression is reproducible by evaluating `bind="auto"` or `bind="custom"` loopback cases where runtime bind resolution can fall back to `0.0.0.0`.

## Solution Assessment

No. The PR is close in direction, but the current classifier is not the safest fix because it infers loopback equivalence before proving the effective listener; a safer path is to fail closed or classify from an explicit/probed effective bind host.

## Review Findings

Full review comments:

- **[P2] Fail closed for auto bind classification:** `src/config/gateway-bind.ts:40`
  - body: `bind="auto"` is classified as loopback whenever the audit process is not in a container, but runtime can still resolve `auto` to `0.0.0.0` if the eventual gateway runs in a container or cannot bind `127.0.0.1`. Because the audit now uses this predicate to suppress `gateway.bind_no_auth` and `gateway.control_ui.allowed_origins_required`, exposed deployments can lose critical findings.
  - confidence: 0.9
- **[P2] Check custom bind reachability before suppressing exposure checks:** `src/config/gateway-bind.ts:32-38`
  - body: A custom loopback address is treated as loopback-equivalent based only on the configured string. `resolveGatewayBindHost` falls back to `0.0.0.0` when `canBindToHost(host)` fails, so this can suppress the non-loopback auth and allowed-origin criticals for a listener that is actually exposed.
  - confidence: 0.88
