## Summary

Keep open. Current main still matches all three reported lifecycle gaps: clean `gateway closed (1000 normal closure)` is not classified as recoverable for subagent wait supervision, `agent.wait` `pending` returns without scheduling another waiter, and stale active sweep can still terminally fail a run even when the backing child session entry is ordinary `running`. The adjacent closed issue #73064 and open PR #74131 are relevant context, but neither is a merged/current-main fix for this narrower combination.

## What This Changes

The issue requests source fixes so subagent runs remain supervised after clean gateway close, pending waits, or missing in-memory run context while the backing session is still running.

## Best Possible Solution

Implement a focused `src/agents` fix: classify clean gateway-close wait interruptions as recoverable for subagent supervision, rearm `waitForSubagentCompletion` after pending results, and re-wait or reattach when stale sweep finds a still-running backing session, while preserving terminal, missing-session, and stale failure semantics.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: a clean `gateway closed (1000 normal closure)` wait error is not recoverable, `agent.wait` pending returns without rearming, and stale sweep on a non-aborted `running` child session falls through to the lost-context terminal error.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `src/agents/run-wait.ts`, `src/agents/subagent-registry-run-manager.ts`, and `src/agents/subagent-registry.ts`, with regression tests for the three reported branches and no broader gateway or product-policy change.
