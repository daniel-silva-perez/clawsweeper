## Summary

Keep open. Current main still matches the three reported lifecycle gaps: clean gateway close code 1000 is not classified as recoverable for wait supervision, a pending `agent.wait` result returns without rearming supervision, and the stale-run sweeper can still fail a run as lost while the backing session entry remains `running`.

## What This Changes

The issue requests more resilient subagent lifecycle recovery after clean gateway closes, pending wait results, and missing in-memory run context when the backing session is still running.

## Best Possible Solution

Implement a focused `src/agents` fix that adds clean 1000 gateway-close wait interruption to recoverable supervision, re-arms `waitForSubagentCompletion` after pending waits, and reattaches or re-waits when stale sweep finds an ordinary still-running backing session, while preserving current terminal, missing-session, aborted, and stale failure semantics.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path by mocking `agent.wait` to throw `gateway closed (1000 normal closure)` or return `pending`, and by sweeping an unended run whose session store entry is `running` without `abortedLastRun`.

## Solution Assessment

Yes. The reporter’s source-level direction is the narrowest maintainable fix; broader gateway reconnect policy or cleanup rewrites are not needed for this issue.
