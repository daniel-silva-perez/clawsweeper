## Summary

Keep open. Current main still has the reported subagent lifecycle gaps: clean `gateway closed (1000...)` is not classified as recoverable, `agent.wait` `pending` returns without rearming supervision, and the stale-run sweeper can mark a run lost even when the backing child session is still `running`. Related #73064 and #74131 overlap, but neither is a merged complete fix for this narrower combination.

## What This Changes

The issue requests recoverable handling for clean gateway close, pending subagent waits, and still-running backing sessions so subagent runs are not falsely marked failed or lost.

## Best Possible Solution

Implement a focused `src/agents` fix: classify clean gateway-close wait interruptions as recoverable for subagent supervision, rearm `waitForSubagentCompletion` after `pending`, and have the stale active sweep re-wait or reattach when the child session store still says `running`, while preserving terminal, missing-session, stale, and restart-aborted semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by mocking `agent.wait` to return clean-close or `pending`, and by sweeping a stale active run whose child session store entry remains `status: "running"`; current main deterministically follows the reported failure branches.

## Solution Assessment

Yes. The suggested fix is the narrowest maintainable path because it extends the existing recoverable-wait and orphan-recovery seams instead of changing broader timeout, task, or gateway lifecycle semantics.
