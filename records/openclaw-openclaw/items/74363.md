## Summary

Keep open. Current main still matches the three reported lifecycle gaps: clean `gateway closed (1000 normal closure)` is formatted by the Gateway transport but not classified as recoverable for agent wait supervision, `agent.wait` `pending` returns without rearming `waitForSubagentCompletion`, and stale active sweep can still complete a run as `subagent run lost active execution context` when the backing child session is merely still `running`. The closed #73064 and open #74131 are useful adjacent context, but neither is a merged current-main fix for this exact issue.

## What This Changes

The issue requests hardening subagent lifecycle recovery so clean gateway close, pending waits, and missing in-memory run context with a still-running backing session do not falsely mark runs failed or lost.

## Best Possible Solution

Implement a focused `src/agents` fix: treat clean Gateway 1000 close as recoverable for subagent wait supervision, rearm `waitForSubagentCompletion` after pending results, and re-wait or reattach when stale sweep finds a still-running backing session, while preserving terminal, missing-session, stale, and drifted failure semantics.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path on current main: mock an `agent.wait` error with `gateway closed (1000 normal closure)`, mock an `agent.wait` pending response, and seed a stale active run whose backing session entry has `status: "running"` with no live run context.

## Solution Assessment

Yes. The narrowest maintainable fix is in the existing subagent wait/registry recovery seams, with regression tests for the three branches; a broad Gateway transport policy change or plugin/API change is not needed.
