## Summary

Keep open. Current main still has the application-level `tick` keepalive but no gateway WebSocket protocol ping in the connection handler, so this PR's intended change is not implemented on main. The patch is small and appears functionally sound, but it should be coordinated with open #74783, which proposes a broader single ping/pong owner for connection telemetry.

## What This Changes

The PR adds a 25 second per-connection `socket.ping()` interval in the gateway WebSocket handler and clears that interval during connection teardown.

## Best Possible Solution

Keep one gateway-owned WebSocket protocol ping path: either merge this focused keepalive PR, or land #74783 first and treat its maintenance-loop ping/pong implementation as the canonical owner while ensuring it covers the same idle-drop behavior.

## Reproduction Assessment

Yes. The linked report provides concrete long-tool-call WebChat steps and code 1006 logs, and current main source confirms there is no server-side WebSocket protocol ping; the only missing piece is live intermediary verification in this read-only review.

## Solution Assessment

Unclear. The per-connection timer is the narrowest fix for idle drops and appears correct, but if #74783 is landing, a single shared maintenance-loop ping owner with telemetry is the more maintainable end state.
