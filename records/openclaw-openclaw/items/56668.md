## Summary

Keep open. Current main still has only the application-level `tick` keepalive and does not implement a gateway WebSocket protocol ping, so this PR is not obsolete. The patch appears narrowly scoped and functionally sound, but it needs maintainer coordination with open #74783 so the gateway does not grow two independent ping owners.

## What This Changes

The PR adds a 25-second per-connection `socket.ping()` interval in the gateway WebSocket handler and clears that timer during connection teardown.

## Best Possible Solution

Keep exactly one gateway-owned WebSocket protocol ping path: either merge this focused keepalive PR, or land #74783 first and treat its maintenance-loop ping/pong implementation as canonical while verifying it covers the same idle-drop behavior.

## Reproduction Assessment

Yes. The linked report provides concrete WebChat long-tool-call steps and code 1006 logs, and current main source confirms there is no server-side WebSocket protocol ping; live intermediary behavior still needs deployment-level confirmation before merge.

## Solution Assessment

Unclear. The per-connection timer is the narrowest fix for idle drops and matches the `ws` contract, but if #74783 is landing, a single shared maintenance-loop ping owner with telemetry is the more maintainable end state.
