## Summary

Keep open. Current main has partial suppression for exec-like tool errors, but validation-only failures for mutating write/message-style tools are still converted into ordinary tool errors and can produce synthetic user-facing warning payloads. The related open PR #68070 is adjacent retryability work and does not change final chat-surface suppression.

## What This Changes

Request that tool-parameter validation failures be handled internally/logged and returned to the agent without sending raw or generic tool-failure warnings to user chat surfaces such as Feishu DMs.

## Best Possible Solution

Carry a typed validation-only error marker from the parameter validation wrapper through adapter normalization, tool-error summary storage, and final payload policy. Use that marker to suppress normal chat warning payloads for validation-only failures while still logging them and returning them to the model/runtime; keep visible warnings for real mutating side-effect failures and verbose diagnostics.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists in current main: construct buildEmbeddedRunPayloads with a mutating write/message lastToolError whose error is a required-parameter validation string, and the existing tests assert a visible warning payload. I did not run tests because this sweep is read-only.

## Solution Assessment

Yes. The best fix is typed validation-only classification through the existing tool error pipeline; it is narrower and safer than turning off mutating-tool warnings globally or relying on keyword matching alone.
