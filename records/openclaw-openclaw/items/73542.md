## Summary

Keep open. Current main still has the reported gap: `message_sending` is wired through the plugin hook runner, but no internal/user hook event is dispatched for that pre-send point. An open, unmerged PR explicitly references this issue as its fix, so the issue should stay open until that implementation is reviewed and merged or replaced.

## Best Possible Solution

Keep this issue open and review or amend PR #73570. The best fix is a tested internal pre-send hook bridge in the shared outbound delivery path and inbound/webchat before-deliver path, plus a documented event-key and mutation/cancel contract for user-defined hooks.

## Risks / Open Questions

- PR #73570 is the active implementation candidate, but maintainers still need to decide the public internal hook key contract: `message:sending`, plugin-style `message_sending`, or compatibility for both.
- Internal hook handlers currently return `void`, while plugin `message_sending` can rewrite or cancel delivery. The final fix should explicitly document whether user-defined internal hooks are observe-only or can mutate/cancel.
- Regression coverage should include normal send, rewritten content, cancellation, and the inbound/webchat before-deliver path so this does not remain fixed only on one outbound route.
