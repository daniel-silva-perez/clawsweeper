## Summary

Keep open. Current main still runs `message_sending` through the plugin hook runner only, while user-defined internal hooks are registered in a separate registry and no internal pre-send event is emitted. Open PR #73570 explicitly references this issue with closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

User requests that user-defined hooks from `hooks/` listening for `message_sending` fire before outbound delivery, matching the existing plugin hook behavior.

## Best Possible Solution

Review and land PR #73570 or a replacement that emits a documented internal pre-send hook before delivery across the relevant reply paths, with tests for registration keys, ordering, and the intended guardrail semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: user hooks can register arbitrary metadata event strings, but current main only calls plugin `message_sending` hooks and never emits an internal pre-send hook that such handlers can receive.

## Solution Assessment

Unclear. The issue points at the right missing bridge, but the best fix should also address the `message_sending` versus `message:sending` key mismatch, multi-path coverage, tests, and docs rather than adding a single fire-and-forget call in one path.
