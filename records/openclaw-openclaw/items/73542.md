## Summary

Keep open. Current main still lacks an internal/user-hook pre-send dispatch for `message_sending`, and open PR #73570 explicitly references this issue with closing syntax, so this should remain open until that implementation lands or is closed. The discussion about ordering and guardrail trust boundaries remains relevant because the current internal hook surface has no documented pre-send event.

## What This Changes

User-defined HOOK.md hooks registered for outbound pre-send behavior should fire before message delivery, matching the plugin `message_sending` hook surface.

## Best Possible Solution

Land a tested bridge through the active PR or a replacement that emits a documented internal pre-send event before channel delivery across the shared outbound and inbound reply paths, with a clear event-key contract and explicit observe-only versus mutate/cancel semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: HOOK.md event strings register directly, but current outbound and inbound reply code only run plugin `message_sending` hooks and never emit an internal `message:sending` or `message_sending` event.

## Solution Assessment

Unclear. A bare `triggerInternalHook(createInternalHookEvent("message", "sending", ...))` in one path is not enough unless it also resolves the event key mismatch, ordering, docs, tests, and non-shared reply paths.
