## Summary

Keep open. Current main still has a plugin-only pre-send `message_sending` path for the inspected outbound delivery surfaces, while user-defined HOOK.md handlers are registered in the separate internal hook registry. The open PR #73570 explicitly references this issue with closing syntax, so the issue should remain open until that PR lands or maintainers replace it.

## What This Changes

The issue reports that user-defined hooks registered for `message_sending` appear ready but do not fire on outbound delivery because the pre-send path only uses the plugin hook runner.

## Best Possible Solution

Add and document a canonical internal pre-send hook bridge for outbound delivery, with compatibility for existing HOOK.md event names or a clear migration path, preserving plugin modification/cancel ordering and adding focused coverage for shared outbound plus inbound reply dispatch paths.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: HOOK.md event strings are registered literally, internal dispatch only reaches `message` and `message:<action>` listeners, and the current pre-send delivery path only calls plugin `runMessageSending`.

## Solution Assessment

Unclear. Bridging the pre-send path is the right area, but the fix should also account for underscore-versus-colon event compatibility, documented semantics, and pre-delivery ordering before maintainers treat the issue as solved.
