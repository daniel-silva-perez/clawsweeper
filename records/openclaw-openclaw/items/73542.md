## Summary

Keep open. Current main still has the reported gap: `message_sending` is wired through the typed plugin hook runner, while standalone/internal hooks only have a post-send `message:sent` bridge and no documented or tested pre-send internal event. An open implementation PR, #73570, references this issue with GitHub closing syntax and is still unmerged, so the issue should stay open while maintainers review or reject that fix.

## Best Possible Solution

Keep this issue open and use #73570 as the implementation candidate. The best fix is to define the standalone/internal pre-send hook contract, bridge it before delivery in all relevant outbound paths, add regression tests for a HOOK.md/user-defined hook firing before delivery, and update `docs/automation/hooks.md` once the event name and mutation/cancel semantics are settled.

## Risks / Open Questions

- A notification-only internal bridge would not satisfy the full guardrail use case if maintainers expect standalone hooks to rewrite or cancel outbound delivery like plugin `message_sending` hooks can.
- Maintainers need to choose the public event key contract: existing internal hooks use colon keys such as `message:sent`, while the report uses plugin-style `message_sending`.
- A fix limited to `deliverOutboundPayloads` may still miss the inbound/webchat before-deliver path in `src/auto-reply/dispatch.ts`, which currently also uses only the plugin hook runner.
