## Summary

Keep open. Current main still runs `message_sending` only through the plugin hook runner in the relevant outbound paths, while the internal HOOK.md registry has no `message:sending` dispatch or documented pre-send event. Open PR #73570 explicitly references this issue with closing syntax, so the right cleanup is to keep the issue open until that PR lands, is replaced, or is closed.

## What This Changes

User asks OpenClaw to fire user-defined internal hooks for the `message_sending` pre-delivery lifecycle event, matching the existing plugin hook coverage.

## Best Possible Solution

Land #73570 or a replacement that introduces a documented internal pre-send event before channel delivery across the shared outbound and inbound reply paths, with tests for HOOK.md registration, event-key compatibility, ordering before delivery, and clear observe-only versus mutate/cancel semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main registers HOOK.md event strings but only emits plugin `message_sending` hooks before delivery, and a targeted search found no internal `createInternalHookEvent("message", "sending", ...)` bridge.

## Solution Assessment

Unclear. A single `triggerInternalHook` call in one path is not enough unless it also resolves event-key compatibility, documented contract, test coverage, ordering before delivery, and the non-shared reply path.
