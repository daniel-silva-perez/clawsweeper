## Summary

Keep open. Current main still dispatches pre-delivery `message_sending` only through the plugin hook runner, while user-defined HOOK.md hooks are registered in the internal hook registry and no current path emits an internal `message:sending` event. An open unmerged PR is already linked as the implementation candidate, so this issue should stay open until that PR is reviewed, amended, and merged or replaced.

## What This Changes

User-defined hook directories that declare a pre-delivery `message_sending` hook should run before outbound delivery, matching the plugin `message_sending` hook path.

## Best Possible Solution

Review and amend the linked PR or replace it with a narrow fix that bridges the internal pre-send hook at every shared pre-delivery path, documents the supported internal event key and observe/mutate semantics, and adds regression tests for normal, rewritten, and cancelled deliveries.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a HOOK.md event key is registered directly in `src/hooks/loader.ts`, but current main never dispatches an internal `message:sending` event while the plugin `message_sending` path does run before delivery.

## Solution Assessment

No for current main, and unclear for the linked PR as described. A notification-only bridge in one outbound path is not enough unless maintainers explicitly decide the event key, observe-only behavior, and coverage for the inbound before-deliver path.
