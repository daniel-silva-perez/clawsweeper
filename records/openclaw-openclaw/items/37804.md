## Summary

Keep open. Current main has useful partial support for config-injected provider models, including catalog visibility and some `models list` forward-compat handling, but it still does not implement the RFC's central invariant: one canonical resolved-model descriptor with identity, transport, capabilities, provenance, availability, and unsupported reason consumed consistently by status, list, runtime, pickers, thinking gates, and docs.

## Best Possible Solution

Keep this issue open until maintainers either land a shared resolved-model descriptor or intentionally split the RFC into narrower canonical follow-up issues. The implementation should unify registry rows, `models.providers` entries, provider dynamic/forward-compat hooks, runtime transport/API, capabilities, provenance, availability, and unsupported reasons, then migrate `models status`, `models list`, gateway/model-picker surfaces, runtime execution, auto-reply thinking gates, and docs to consume that single result.
