## Summary

Keep open. Current main has useful partial support for config-injected and forward-compatible models, but it still does not implement the RFC's central invariant: one canonical resolved-model descriptor with identity, transport, capabilities, provenance, and availability consumed consistently by status, list, runtime, thinking gates, picker/auto-reply consumers, and docs.

## Best Possible Solution

Keep this issue open until maintainers either land a shared resolved-model descriptor or explicitly split the RFC into narrower canonical follow-up issues. The implementation should unify registry entries, models.providers entries, provider dynamic/forward-compat hooks, transport/API, capabilities, provenance, availability, and unsupported reasons, then migrate models status, models list, runtime execution, picker/auto-reply consumers, thinking gates, and docs to consume that single result.
