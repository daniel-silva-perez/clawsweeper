## Summary

Keep this PR open. Current main still does not implement the requested proxy circuit breaker in the shared proxy fetch helper or current Discord plugin paths, and the PR is not merge-ready as-is because the proposed direct fallback has proxy-egress security implications and review comments identified real circuit-state bugs.

## Best Possible Solution

Keep the PR open as useful but not mergeable. The best path is a rebased current-main implementation that puts policy-aware proxy circuit breaking in the shared proxy helper and current Discord/Telegram-facing proxy paths, preserves managed-proxy fail-closed semantics, fixes the reviewed direct-error and half-open concurrency bugs, separates unrelated rate-limit matching changes, and adds focused regression tests for fallback, cooldown, single-probe behavior, and recovery.

## Risks / Open Questions

- The PR's direct fallback can bypass an operator-managed filtering proxy in deployments that require proxy egress for SSRF and audit controls.
- The proposed half-open logic can allow many concurrent re-probes at the cooldown boundary, which undercuts the CPU-exhaustion goal.
- The proposed gateway catch block can attribute direct-connection failures to the proxy circuit and extend proxy blackout incorrectly.
- The PR is stale against the current Discord plugin layout under `extensions/discord` and would need a rebase/adaptation.
- The diff mixes unrelated rate-limit regex changes into a network-proxy fix, which should be split or justified before review.
