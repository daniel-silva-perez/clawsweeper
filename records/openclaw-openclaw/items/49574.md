## Summary

Keep this PR open. Current main still does not implement the shared proxy-fetch circuit breaker this PR targets, but the proposed patch is not merge-ready as-is because it can conflict with the newer managed-proxy fail-closed contract and the review discussion identifies real circuit-state bugs.

## Best Possible Solution

Keep this PR open as useful but not mergeable in its current form. The best path is a rebased implementation that adds policy-aware proxy circuit breaking in the shared proxy helper and active Discord/Telegram-facing paths, preserves managed-proxy fail-closed semantics, fixes direct-error attribution and single half-open probe behavior, separates unrelated rate-limit matching changes, and adds focused regression tests for fallback policy, cooldown, single-probe behavior, and recovery.

## Risks / Open Questions

- Unconditional direct fallback can bypass an operator-managed filtering proxy in deployments relying on `proxy.enabled=true` for SSRF/audit controls.
- The proposed half-open state can still produce a burst of concurrent proxy probes and inflate backoff counters under traffic.
- The proposed gateway catch path can record direct-connection failures as proxy failures and delay proxy recovery.
- The branch needs a rebase/adaptation to the current `extensions/discord` layout and newer managed-proxy behavior.
- The diff mixes unrelated rate-limit regex changes into a network-proxy fix, which should be split or justified before review.
