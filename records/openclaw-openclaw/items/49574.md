## Summary

Keep this PR open. Current main still does not implement the requested shared proxy-fetch circuit breaker, and the proposed PR is not merge-ready because its direct fallback can conflict with operator-managed proxy fail-closed semantics and the review discussion identified real circuit-state bugs.

## Best Possible Solution

Keep this PR open as useful but not mergeable. The best path is a rebased current-main implementation that adds policy-aware proxy circuit breaking in the shared proxy helper and active Discord/Telegram-facing paths, preserves managed-proxy fail-closed semantics, fixes the direct-error attribution and single half-open probe behavior, separates unrelated rate-limit matching changes, and adds focused regression tests for fallback policy, cooldown, single-probe behavior, and recovery.

## Risks / Open Questions

- The PR's direct fallback can bypass an operator-managed filtering proxy in deployments using `proxy.enabled=true` for SSRF and audit controls.
- The proposed half-open state allows concurrent re-probes, which can recreate bursts of failed proxy connects and inflate backoff state.
- The proposed gateway catch logic can misattribute direct-connection failures to the proxy circuit and delay proxy recovery.
- The PR is stale against the current Discord plugin layout under `extensions/discord` and would need a rebase/adaptation.
- The diff mixes unrelated rate-limit regex changes into a network-proxy fix, which should be split or justified before review.
