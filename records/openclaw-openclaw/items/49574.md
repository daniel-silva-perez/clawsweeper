## Summary

Keep this PR open. Current main still does not implement the shared proxy-fetch circuit breaker the PR targets, and the prior review discussion identifies useful work plus real correctness issues. The branch also needs adaptation to the newer managed-proxy fail-closed contract before it can be considered mergeable.

## Best Possible Solution

Keep this PR open as useful but not mergeable as-is. The best path is a rebased, policy-aware proxy circuit breaker in the shared proxy helper and active Discord/Telegram-facing paths, with managed-proxy fail-closed semantics preserved, direct-error attribution fixed, only one half-open probe allowed per cooldown, unrelated rate-limit changes split out, and focused regression tests for fallback policy, cooldown, single-probe behavior, and recovery.

## Risks / Open Questions

- Security review: the proposed direct fallback after proxy connection errors is now security-sensitive because current main has an operator-managed proxy mode intended for egress control and SSRF/audit defense; bypassing it can violate that deployment contract.
- The proposed half-open logic can still create a burst of proxy probes and inflate backoff counters under concurrent traffic, preserving part of the CPU/churn risk the PR is trying to reduce.
- The proposed gateway catch path can misattribute direct-connection failures to the proxy and delay recovery.
- The PR diff targets older `src/discord/...` paths while current main has Discord under `extensions/discord/...`, so a rebase is required.
- The diff mixes unrelated rate-limit regex changes into a proxy/networking fix; those should be split or separately justified.
