## Summary

Keep this PR open. Current `main` still does not implement a shared proxy-fetch circuit breaker for unavailable proxies, and the PR discussion identifies real correctness and security issues that need a rebased, policy-aware revision before it can land.

## Best Possible Solution

Keep this PR open as useful but not mergeable as-is. The best path is a rebased implementation that adds proxy circuit-breaker behavior at the shared proxy helper/plugin-facing surface and active Discord/Telegram call sites, preserves managed-proxy fail-closed semantics, allows only one half-open probe per cooldown, fixes direct-error attribution, splits unrelated rate-limit matcher changes, and adds focused regression tests for fallback policy, cooldown, single-probe behavior, and recovery.

## Risks / Open Questions

- Security: direct fallback after proxy connection errors can bypass the operator-managed proxy contract used for egress control and SSRF/audit defense unless it is scoped to non-managed/debug proxy paths.
- Correctness: the PR discussion identifies direct-error attribution and half-open concurrency issues that could extend proxy blackout or preserve connection bursts under load.
- Integration: the PR diff targets older `src/discord/...` paths, while current main uses `extensions/discord/...`, so it needs a rebase across current plugin boundaries.
- Scope: unrelated rate-limit matching changes are bundled into a proxy circuit-breaker PR.
