## Summary

Keep open. Current main still has the reported Hyprland/Wayland screenshot gap, and the linked bug has concrete reproduction data. The PR is still the right narrow direction, but the latest head still has a blocking cleanup bug: if setup fails after creating the Hyprland virtual output, the output is not removed before falling back to CDP.

## What This Changes

The PR adds an opt-in `browser.hyprlandCapture` headed Hyprland/Wayland viewport screenshot fallback using `hyprctl` plus `grim`, updates browser config/schema/docs and a Plugin SDK subpath, and adds focused browser capture and route tests.

## Best Possible Solution

Revise this PR in place so every post-create setup failure removes the newly-created Hyprland output, add a unit assertion for create-output success followed by monitor or dispatch failure, and then land the opt-in browser-plugin fallback with targeted browser tests plus the normal changed gate for the linked bug.

## Reproduction Assessment

Yes. The linked bug gives a concrete Omarchy/Hyprland/Wayland headed Chromium screenshot timeout path, and the remaining PR defect is reproducible by the existing mocked setup-failure shape: create the output, fail `getActiveWorkspaceId` or dispatch, and observe no `hyprctl output remove` call.

## Solution Assessment

No, not as currently proposed. The feature direction is the narrow maintainable fix for #64317, but the branch should not merge until setup failure cleanup is made symmetrical with grim-failure and shutdown cleanup.
