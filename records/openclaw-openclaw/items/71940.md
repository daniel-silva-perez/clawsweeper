## Summary

Keep open. Current main still uses the CDP screenshot path and does not accept `browser.hyprlandCapture`, while the linked Hyprland screenshot bug remains open and concrete. The latest PR branch addresses several earlier review comments, but it still has a lifecycle bug that can orphan a created Hyprland virtual output when setup fails after `create-output`; the latest head also is not fully green.

## What This Changes

The PR adds an opt-in `browser.hyprlandCapture` headed viewport screenshot fallback using Hyprland `hyprctl` plus `grim`, wires config/schema/docs, adds a focused `plugin-sdk/resolve-system-bin` subpath, and adds browser capture and route tests.

## Best Possible Solution

Revise this PR in place so the browser-plugin fallback remains opt-in and limited to headed plain viewport screenshots, removes every created virtual output on setup, capture, and shutdown failure paths, keeps the SDK/config/docs/generated surfaces aligned, and then lands it to resolve #64317.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Remove created output when setup fails:** `extensions/browser/src/browser/hyprland-capture.ts:100-107`
  - body: After `createHeadlessOutput()` succeeds, `getActiveWorkspaceId()` or the `movetoworkspacesilent` dispatch can still throw. Because `_state` is assigned only after those steps and cleanup is not registered yet, the catch path clears memory but never removes the new `HEADLESS-*` output, so a failed setup can leak a virtual output and retries can create more. Wrap the post-create setup in a catch that removes `outputName` before rethrowing.
  - confidence: 0.92
