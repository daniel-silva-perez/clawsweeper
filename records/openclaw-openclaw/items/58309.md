## Summary

Keep open. Current main still has the targeted raw parse sites, so this is not implemented on main; the PR is useful, but it needs normal maintainer review because it mixes JSON error handling with security-adjacent identity key storage and has a low-severity permission default concern.

## What This Changes

The PR wraps selected JSON.parse calls in archive, Tailscale, and outbound delivery queue helpers with contextual errors, and adds a private atomic-write helper for device identity writes.

## Best Possible Solution

Adopt contextual parse errors for the affected helpers, align key-material writes with the existing private-by-default JSON/text file helper patterns, and add focused corrupt-JSON plus valid-JSON regression coverage before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: feed malformed JSON to the archive helper, outbound queue file loader, or Tailscale JSON parsing path on current main and the raw `JSON.parse` path is reached; I did not execute it because this review is read-only.

## Solution Assessment

Mostly yes, but not quite complete. The narrow parse wrappers address the reported UX, while the identity write helper should default to `0o600` or require a mode, tests should cover the new messages, and the remaining Tailscale funnel parse path should be considered for consistency.
