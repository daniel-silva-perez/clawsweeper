## Summary

Keep open: current main still lacks the WhatsApp config, account resolution, and Baileys send-option plumbing needed for outbound disappearing-message expiration. The related implementation PRs were closed unmerged, so the issue remains valid and narrow enough for a focused repair PR.

## What This Changes

Add WhatsApp channel-level and account-level disappearing-message expiration config and apply the effective value to outbound WhatsApp replies.

## Best Possible Solution

Implement the behavior inside the bundled WhatsApp plugin: add typed and schema-validated channel/account config, resolve a positive effective expiration with account override semantics, and merge it into Baileys MiscMessageGenerationOptions across text, media, quoted, voice-note companion text, poll, and tool send paths while leaving unset, null, or zero behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the proposed config is absent from strict schema/types, resolved account state cannot carry it, and current send paths never pass Baileys ephemeralExpiration despite the dependency exposing that option.

## Solution Assessment

Yes. The requested plugin-owned config and send-options plumbing is the narrowest maintainable solution; no new core channel contract is needed unless multiple channels later need a generic disappearing-message abstraction.

Add `channels.whatsapp.disappearingMessagesSeconds` and `channels.whatsapp.accounts.<accountId>.disappearingMessagesSeconds` to the typed config and strict zod schema, accepting an omitted or null value and treating zero/non-positive/non-finite values as no explicit OpenClaw override.
