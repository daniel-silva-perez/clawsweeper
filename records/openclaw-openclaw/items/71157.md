## Summary

Current main still lacks the WhatsApp config/schema field, account-level resolution, and Baileys send-option plumbing needed to set disappearing-message expiration on outbound replies. The two related implementation PRs were closed unmerged, so this remains a valid narrow WhatsApp plugin repair rather than a cleanup close.

## What This Changes

Allow WhatsApp channel/account configuration to set a default disappearing-message expiration for outbound OpenClaw WhatsApp replies.

## Best Possible Solution

Implement this inside the bundled WhatsApp plugin: add typed and schema-validated channel/account config, resolve a positive effective expiration with account override semantics, and merge it into Baileys MiscMessageGenerationOptions for text, media, quoted replies, voice-note companion text, polls, and tool send paths while preserving unset, null, and zero behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the proposed config key is absent from strict schema/types, resolved account state cannot carry it, and current WhatsApp send paths never pass Baileys ephemeralExpiration despite the pinned dependency exposing that option.

## Solution Assessment

Yes. The requested plugin-owned config and send-options plumbing is the narrowest maintainable solution; no new core channel abstraction is needed unless multiple channels later need a shared disappearing-message contract.

## Repair Work Prompt

Implement the WhatsApp disappearing-message expiration support requested by https://github.com/openclaw/openclaw/issues/71157. Add optional `channels.whatsapp.disappearingMessagesSeconds` and `channels.whatsapp.accounts.<accountId>.disappearingMessagesSeconds` config, with account-level override over channel-level default. Accept positive finite values, collapse unset/null/0/non-positive/non-finite to no explicit override, and keep existing behavior unchanged when not configured. Resolve the effective value in the bundled WhatsApp plugin account/send path, expose it through the local ActiveWebSendOptions path, and merge it into Baileys `MiscMessageGenerationOptions.ephemeralExpiration` for normal text, media/caption sends, quoted replies, voice-note companion text, polls, and tool/channel send paths. Treat #73080 and #73083 as useful source context only; do not depend on those branches being available. Keep the fix plugin-owned and do not add a generic core channel abstraction unless existing code already requires it. Update config schema/types, generated config docs/checks if metadata changes require it, WhatsApp docs, and CHANGELOG.md with appropriate contributor credit. Do not change unrelated WhatsApp auth, routing, or media behavior.
