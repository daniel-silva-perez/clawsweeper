## Summary

Keep this issue open. Current main still has no WhatsApp channel/account disappearing-message expiration config, no resolved account field to carry an effective value, and no Baileys `ephemeralExpiration` forwarding on the active outbound text, media, voice-note companion text, or poll paths; the related implementation PRs are closed and unmerged.

## What This Changes

The issue requests WhatsApp channel and per-account config that applies a default disappearing-message expiration to outbound OpenClaw replies.

## Best Possible Solution

Implement this inside the bundled WhatsApp plugin: add optional channel/account config and schema/docs metadata, resolve a positive effective value with account override semantics, and merge it into Baileys send options on every outbound transport path while preserving existing behavior when unset, zero, or null.

## Reproduction Assessment

Yes. A current-main source check reproduces the absence: the strict config schema lacks the proposed keys, resolved account state has no expiration field, and outbound send code never passes `ephemeralExpiration` to Baileys.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable solution because this is WhatsApp plugin-owned config and transport metadata, not a generic core channel contract; the closed PRs demonstrate a plausible shape but still need a merged, validated implementation.
