## Summary

Keep open. Current main still lacks a binding-level session target or alias for sharing a Discord guild/channel route with an agent main or DM session; the only route-binding session override is still `dmScope`, and that path is intentionally direct-message focused. Prior maintainer/ClawSweeper comments reached the same conclusion, and the remaining work needs product and privacy policy before implementation.

## What This Changes

The request asks for an optional route-binding session key, alias, or scope so a matched Discord guild/channel conversation can share continuity with the same agent's Telegram/DM session.

## Best Possible Solution

Add an explicit, opt-in, typed route-binding session target contract in config schema/types and route resolution, validate it to safe canonical agent-owned targets, preserve default group/channel isolation, and update generated schema/help/docs plus regression coverage for configured channel-to-main sharing and unchanged unconfigured channel behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Static inspection and existing tests show a Discord channel peer resolves to `agent:<agentId>:discord:channel:<id>`, while the strict binding session schema rejects any binding session field other than `dmScope`.

## Solution Assessment

Is this the best way to solve the issue? Unclear as proposed. A raw `sessionKey` on bindings may be too permissive; the safer maintainable direction is a typed, validated, explicit opt-in target or alias contract with channel isolation preserved by default.
