## Summary

Current main still lacks a binding-level session key/alias/scope target for non-direct channel peers. Binding session overrides are dmScope-only, Discord guild messages still resolve as channel peers, and channel peers still build isolated `agent:<agentId>:discord:channel:<channelId>` session keys.

## Best Possible Solution

Keep this issue open for maintainer product/security design. The best implementation would add a documented explicit opt-in binding session target such as `bindings[].session.target`, `sessionKey`, or `sessionScope`, preserve default group/channel isolation, validate canonical agent session keys or well-defined aliases, thread the target through route resolution and session-key construction, update config schema/help/docs, and add regression tests proving a Discord channel binding can intentionally share an agent main/alias session while unconfigured channels stay isolated.

## Risks / Open Questions

- A channel-to-main binding target would intentionally merge guild/channel conversation history into a shared agent session, so it needs explicit opt-in semantics and security docs to avoid accidental context leakage.
- Any implementation that adds a binding-level fixed session target must account for route-cache keys and last-route policy so different matched bindings cannot reuse stale session resolution.
- Existing Discord thread/session binding infrastructure can override target session keys for focused/thread-bound workflows, but that is not a documented general `bindings[]` session alias and should not be treated as satisfying this config request.
