## Summary

Keep open. Current main has the adjacent binding-level `session.dmScope` override from #71750, but it still only feeds DM scoping. There is no binding-level `sessionKey`, `sessionAlias`, or `sessionScope` target, and non-direct Discord guild/channel peers still build isolated `agent:<agentId>:discord:channel:<channelId>` session keys. Related #50875 remains open and is scoped to direct-message identity resolution, while #39049 is closed unmerged, so neither supersedes the requested channel-to-main session unification.

## Best Possible Solution

Keep this issue open until maintainers decide the product/security shape for non-direct route-binding session targets. A good implementation would add a documented, explicit opt-in binding-level target such as `bindings[].session.target`/`sessionKey`/`sessionScope`, preserve default channel isolation, validate canonical agent session keys or well-defined aliases, thread it through `resolveAgentRoute` and session-key construction, update config help/docs/generated schema, and add regression tests proving Discord channel bindings can intentionally share an agent main or alias session while unconfigured channels remain isolated.

## Risks / Open Questions

- Adding a channel-to-main binding target would intentionally merge guild/channel conversation history into a shared agent session, so it needs explicit opt-in semantics and clear docs/security framing.
- The current route cache key includes global `dmScope` but not a future binding-level fixed session target; any implementation must avoid stale route-cache collisions if session-target behavior varies by binding.
- Existing ACP/configured binding paths can override to stateful target session keys, but they are ACP/session-binding infrastructure and should not be conflated with a general route-binding session alias without maintainer design approval.
