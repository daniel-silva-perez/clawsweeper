## Summary

Current main still does not provide a binding-level `sessionKey`, `sessionAlias`, or non-DM session-scope override that would let a Discord guild/channel binding share the agent main session with Telegram DMs. The only binding session override is `dmScope`, and non-direct peers still construct isolated channel/group session keys, so this issue remains valid and needs maintainer design before implementation.

## Best Possible Solution

Keep this issue open for maintainer product/security design. The best path is an explicit opt-in binding session target, such as `bindings[].session.target`, `sessionKey`, or a well-defined alias/scope, with strict validation of canonical agent session keys or approved aliases; preserve default group/channel isolation; thread the target through route resolution and session-key construction; update generated config schema/help and public docs; and add regression tests proving a configured Discord channel can intentionally share an agent main/alias session while unconfigured channels remain isolated.

## Risks / Open Questions

- A binding-level target that intentionally merges guild/channel traffic into a shared agent session can leak channel context into DM history unless it is explicit, validated, documented, and opt-in.
- Any implementation needs route-cache and `lastRoutePolicy` handling so changing or matching a binding cannot reuse stale session resolution or misroute delivery.
- The public config/API shape needs maintainer judgment before an automated fix PR because `sessionKey`, `sessionAlias`, and `sessionScope` have different safety and migration implications.
