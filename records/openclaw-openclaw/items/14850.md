## Summary

Keep open. Current main still only supports `bindings[].session.dmScope` for direct-message session scoping, while Discord guild/channel peers remain isolated and there is no binding-level session target or alias for sharing a channel route with an agent main session.

## What This Changes

Request adding an opt-in binding-level session key, alias, or scope so a bound Discord guild channel can share continuity with the same agent's Telegram or direct-message session.

## Best Possible Solution

Add an explicit, opt-in, typed route-binding session target contract in config schema/types and the route resolver, validate it to safe canonical agent-owned targets, preserve default group/channel isolation, and update generated schema/help/docs plus regression coverage for configured channel-to-main sharing and unchanged unconfigured channel behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Static inspection and existing tests show a Discord channel peer resolves to `agent:<agentId>:discord:channel:<id>`, while the strict binding session schema rejects any binding session field other than `dmScope`.

## Solution Assessment

Is this the best way to solve the issue? Unclear as proposed. A raw `sessionKey` on bindings may be too permissive; the safer maintainable direction is a typed, validated, explicit opt-in target or alias contract with channel isolation preserved by default.
