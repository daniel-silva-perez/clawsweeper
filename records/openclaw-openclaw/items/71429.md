## Summary

Keep this open. Current main still has restart deferral and durable outbound/restart-continuation queues, but ordinary Telegram same-surface auto-replies still go through the in-process dispatcher and direct Telegram delivery path, so a Bot API/network failure can still leave the generated response visible internally without durable replay to Telegram.

## What This Changes

The issue requests durable replay or explicit visible failure for Telegram in-flight outbound replies when hot reload coincides with sendChatAction or Bot API network failures.

## Best Possible Solution

Add a focused Telegram regression for same-surface auto-replies during reload/network failure, then either journal normal Telegram final replies before Bot API delivery and replay them safely, or fail the turn visibly in Telegram and UI with retry guidance.

## Reproduction Assessment

Yes. A high-confidence unit/integration reproduction can simulate a Telegram inbound turn with Provider and Surface both set to telegram, force deliverReplies or sendChatAction-related delivery to fail during a reload window, and assert that no durable replay entry or guaranteed visible failure exists. A live Telegram run was not performed in this read-only review.

## Solution Assessment

No current-main behavior is not the best solution yet. The narrow maintainable fix is to reuse the existing durable outbound/session delivery primitives for normal Telegram same-surface final replies, or explicitly fail the turn with a retryable user-visible status if safe replay is not possible.
