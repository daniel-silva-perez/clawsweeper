## Summary

Keep open. Current main still routes ordinary Telegram-to-Telegram auto-replies through an in-process dispatcher and direct Telegram Bot API delivery, while the durable queues and restart-continuation hardening cover different outbound/restart paths. The maintainer discussion already calls out this exact gap, and the relevant scoped code has not gained durable replay or guaranteed visible failure handling for this path.

## What This Changes

The issue asks for Telegram in-flight outbound replies during hot reload and sendChatAction/network failure to be durably replayed or failed visibly instead of silently disappearing from the Telegram user surface.

## Best Possible Solution

Add a focused Telegram regression for same-surface auto-replies during reload/network failure, then reuse the existing durable outbound/session delivery primitives for final Telegram replies or fail the turn visibly in Telegram and UI with retry guidance.

## Reproduction Assessment

Yes. A high-confidence reproduction can be built by simulating a Telegram inbound turn where Provider and Surface are both telegram, forcing sendPayload/deliverReplies or sendChatAction-adjacent delivery to fail during a reload window, and asserting that current main has neither durable replay nor guaranteed user-visible failure.

## Solution Assessment

Yes, the issue's direction is the right maintainable shape if implemented through existing queue/recovery primitives or an explicit visible abort. An ad hoc Telegram-only journal would be a weaker direction unless the shared outbound queue cannot preserve Telegram preview/final semantics safely.
