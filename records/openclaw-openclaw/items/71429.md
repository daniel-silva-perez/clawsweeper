## Summary

Keep open. Current main still has reload deferral and durable outbound/restart-continuation queues, but the ordinary Telegram same-surface auto-reply path still sends through the in-process dispatcher and direct Telegram Bot API delivery without a write-ahead journal or guaranteed visible hard failure when delivery exhausts.

## What This Changes

Prevent Telegram long-poll gateway hot reload or sendChatAction/network delivery failures from silently dropping in-flight same-surface outbound replies.

## Best Possible Solution

Add a focused Telegram regression for same-surface auto-replies during reload/network failure, then either journal normal Telegram final replies before Bot API delivery and replay them safely, or fail the turn visibly in Telegram and UI with retry guidance.

## Reproduction Assessment

Yes at the focused unit/integration level: simulate a Telegram inbound turn with Provider and Surface both set to telegram, make final Bot API delivery fail during reload, and verify there is no durable replay entry or guaranteed visible failure. A live Telegram bot hot-reload run was not performed in this read-only review.

## Solution Assessment

No current-main behavior is not the best solution yet. The narrow maintainable fix is to reuse the existing durable outbound/session delivery primitives for normal Telegram same-surface final replies, or explicitly fail the turn with a retryable user-visible status if safe replay is not possible.
