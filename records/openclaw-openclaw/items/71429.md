## Summary

Keep open. Current main still has restart deferral and generic write-ahead outbound recovery, but ordinary Telegram-to-Telegram auto-replies remain on the in-process reply dispatcher and direct Telegram Bot API delivery path. The issue discussion already identifies this exact remaining gap, and the relevant files are unchanged since the prior keep-open review.

## What This Changes

Telegram same-surface auto-replies should not be silently lost when hot reload and sendChatAction or network delivery failures occur mid-turn.

## Best Possible Solution

Add a focused Telegram regression for same-surface auto-replies during reload/network failure, then either journal final Telegram replies through existing durable delivery primitives before Bot API delivery or fail the turn visibly in Telegram, UI, and agent state with retry guidance.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound turn where Provider and Surface are both telegram, force sendChatAction or direct deliverReplies failure during a reload window, and assert that current main has no durable replay for the final same-surface reply.

## Solution Assessment

Yes, the requested direction is the maintainable shape if it reuses the existing outbound/session delivery primitives or introduces an explicit visible abort. A Telegram-only ad hoc journal would be weaker unless the shared queue cannot preserve preview/final semantics safely.
