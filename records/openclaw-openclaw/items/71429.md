## Summary

Keep open. Current main still has restart deferral and durable queues for generic outbound/restart-continuation delivery, but the ordinary Telegram same-surface auto-reply path still sends final replies through the in-process dispatcher and direct Telegram Bot API delivery without a durable journal or visible hard failure on delivery exhaustion.

## What This Changes

The issue requests that in-flight Telegram replies during gateway hot reload either survive `sendChatAction`/network failures via durable resend or fail visibly instead of being silently dropped.

## Best Possible Solution

Keep the issue open and add a focused Telegram regression for same-surface auto-replies during reload/network failure, then journal normal Telegram final replies before Bot API delivery or make the failed turn visibly retryable in both Telegram and UI.

## Reproduction Assessment

Yes, there is a high-confidence focused reproduction path at the unit/integration level: simulate a Telegram inbound turn with `Provider` and `Surface` both set to `telegram`, make final Bot API delivery fail, and verify the reply is neither durably queued nor surfaced as a hard visible failure. A live Telegram bot hot-reload reproduction was not run in this read-only review.

## Solution Assessment

No current-main behavior is not the best solution yet. The narrow maintainable fix is to reuse the existing durable outbound/restart delivery primitives for normal Telegram same-surface final replies, or explicitly fail the turn with a retryable user-visible status if durable resend is not safe.

## Repair Work Prompt

Fix openclaw/openclaw#71429. Normal Telegram same-surface auto-replies currently flow through `extensions/telegram/src/bot-message-dispatch.ts` -> `deliverReplies` -> Telegram Bot API without the existing durable outbound queue, so a hot reload/network failure can leave the transcript/UI with a completed answer that never reaches Telegram. Add a focused regression that simulates a Telegram inbound turn with `Provider`/`Surface` set to `telegram`, an in-flight reload or dispatcher pending state, and a final delivery failure from `deliverReplies`/`sendMessage`. Implement the narrow durable path for ordinary Telegram final replies before Bot API delivery, preferably by reusing existing outbound delivery queue/recovery primitives without changing generic routeReply behavior. If durable resend is not safe for a specific payload class, fail the turn visibly and retryably instead of silently swallowing it. Preserve Telegram thread/quote/buttons/media/status-reaction hooks, avoid duplicate preview/final sends, and do not broaden this into generic session replay for all restart causes. Add a user-facing changelog fix entry if the repo policy requires one, credited to @sync-tachikoma.
