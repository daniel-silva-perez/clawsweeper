## Summary

Keep this open. Current main still has restart deferral and durable generic outbound recovery, but normal Telegram-to-Telegram auto-replies remain on the in-process dispatcher plus direct Telegram Bot API path, and maintainer discussion already identifies this exact gap as not fully closed.

## What This Changes

The issue requests durable replay or visible failure handling for Telegram replies that are in flight during gateway hot reload when Telegram sendChatAction or adjacent network delivery fails.

## Best Possible Solution

Add a focused Telegram regression for same-surface auto-replies during reload or Telegram network failure, then route final Telegram replies through an existing durable delivery primitive or fail the turn visibly in Telegram and UI with retry guidance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction can simulate a Telegram inbound turn with Provider and Surface both telegram, force deliverReplies or sendChatAction-adjacent Telegram API failure during a reload window, and assert that current main has neither durable replay nor guaranteed user-visible failure for the final reply.

## Solution Assessment

Yes, the issue's proposed direction is the maintainable shape if it reuses existing delivery queue or session-delivery primitives. A narrow Telegram-only journal would be less desirable unless the shared queue cannot safely preserve Telegram preview/final semantics.
