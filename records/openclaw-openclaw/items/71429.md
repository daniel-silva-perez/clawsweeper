## Summary

Keep open. Current main still has related reload deferral and durable queue primitives, but ordinary Telegram same-surface auto-replies still route through the in-process dispatcher to direct Telegram Bot API delivery without a durable write-ahead entry or a guaranteed visible hard failure when delivery/typing transport fails; that matches the maintainer comments already leaving this report open.

## What This Changes

The issue requests durable or visibly failing Telegram outbound reply delivery when sendChatAction/network failures occur during gateway hot reload, rather than silently losing in-flight Telegram messages.

## Best Possible Solution

Add a focused Telegram regression for same-surface auto-replies during reload/network failure, then either journal normal Telegram final replies before Bot API delivery and replay them safely, or fail the turn visibly in Telegram and the internal UI with retry guidance.

## Reproduction Assessment

Yes at the focused unit/integration level: simulate a Telegram inbound turn with Provider and Surface both set to telegram, make final Bot API delivery fail during reload, and verify the current path has no durable replay entry or guaranteed visible failure. A live Telegram hot-reload run was not performed in this read-only pass.

## Solution Assessment

No current-main behavior is not the best solution yet. The narrow maintainable direction is to reuse the existing durable outbound/session delivery primitives for normal Telegram same-surface final replies, or explicitly fail the turn with a retryable user-visible status if duplicate-safe replay is not possible.
