## Summary

Keep this PR open, but it is not merge-ready. Current main still uses a fixed two-hour stuck-lock cleanup window, so the PR is not obsolete, but the latest PR head clamps every finite timeout-derived cleanup threshold to the two-hour fallback; that can clear a valid long-running job lock before the configured cron timeout has expired.

## What This Changes

The PR imports cron timeout policy into job maintenance, replaces fixed stale running-marker cleanup with timeout-plus-buffer logic capped at a two-hour fallback, and adds or updates cron regression tests.

## Best Possible Solution

Request changes on this PR: keep timeout-derived cleanup for normal finite jobs, preserve the conservative no-timeout fallback, handle only overflow-sized timeout values specially, and add coverage for short custom timeouts, default systemEvent and agentTurn thresholds, no-timeout fallback, valid long finite timeouts, and overflow-sized timeoutSeconds.

## Reproduction Assessment

Yes. A source-level reproduction is an agentTurn cron job with timeoutSeconds set to three hours and runningAtMs just over two hours old; the PR clears that lock even though runtime would still allow the original execution to continue.

## Solution Assessment

No. The timeout-plus-buffer direction is narrow and maintainable, but clamping all finite values to two hours is too broad; the safer alternative is to cap, reject, or otherwise normalize only values beyond Node's timer ceiling while respecting valid configured long timeouts.
