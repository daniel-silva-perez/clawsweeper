## Summary

Keep open. Current main still uses a fixed two-hour stale-running marker threshold, so the intended per-job cleanup is not implemented, but this PR currently introduces a P2 correctness bug by capping every finite timeout at two hours even when runtime still honors longer valid timeouts.

## What This Changes

The PR changes cron maintenance to clear stale runningAtMs markers using resolveCronJobTimeoutMs plus a five-minute buffer, capped at a two-hour fallback, and adds or updates cron regression tests.

## Best Possible Solution

Request changes on this PR: keep timeout-derived stale-lock cleanup for normal finite jobs, preserve the conservative no-timeout fallback, handle only values beyond Node's timer ceiling specially, and add regression coverage for short custom timeouts, default thresholds, no-timeout fallback, valid long finite timeouts, and overflow-sized timeoutSeconds.

## Reproduction Assessment

Yes. A source-level reproduction is an agentTurn cron job with timeoutSeconds set to three hours and runningAtMs just over two hours old; this PR clears the lock even though runtime would still allow the original execution to continue.

## Solution Assessment

No. The timeout-plus-buffer direction is narrow and maintainable, but clamping all finite values to two hours is too broad; the safer alternative is to cap, reject, or otherwise normalize only values beyond Node's timer ceiling while respecting valid configured long timeouts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve locks until explicit long timeouts expire:** `src/cron/service/jobs.ts:393`
  - body: With an explicit timeoutSeconds above two hours but below Node's timer ceiling, runtime still allows the job to run for that configured duration. This clamp clears runningAtMs at the two-hour fallback, so a later maintenance tick can make the same job runnable while the original execution is still active. Handle overflow-sized values separately without shortening valid long timeouts.
  - confidence: 0.9
