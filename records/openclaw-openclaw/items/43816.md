## Summary

Keep open. Current main still uses a fixed two-hour maintenance threshold for clearing stale cron `runningAtMs` markers, so the intended per-job cleanup is not obsolete, but this PR head should not merge as-is because it caps every finite timeout-derived threshold at two hours and can make long configured jobs runnable before their configured runtime timeout expires.

## What This Changes

The branch replaces cron's fixed two-hour stuck-running lock cleanup with a timeout-derived per-job threshold, preserves a two-hour no-timeout fallback, and adds cron regression tests plus a comment update.

## Best Possible Solution

Request changes on this PR: keep the per-job stale-lock threshold for ordinary finite timeouts, preserve the conservative no-timeout fallback, handle overflow-sized timeout values with an explicit policy that does not shorten valid long timeouts, and add regression coverage for short custom timeouts, default systemEvent and agentTurn thresholds, no-timeout fallback, and long finite-timeout preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve locks until explicit long timeouts expire:** `src/cron/service/jobs.ts:393`
  - body: With an explicit `timeoutSeconds` above two hours, runtime still allows the job to run for that configured duration, but this clamp clears `runningAtMs` at the two-hour fallback. A later maintenance tick can then make the same job runnable while the original execution is still in progress. Handle overflow-sized values separately without shortening valid long timeouts.
