## Summary

Keep open. Current main still uses a fixed two-hour cron stale-running cleanup window, so the PR targets a real remaining reliability gap; however, the current PR head is not merge-ready because its two-hour ceiling can clear locks before explicit long-running job timeouts have elapsed.

## What This Changes

The PR changes cron maintenance to derive stale `runningAtMs` cleanup from `resolveCronJobTimeoutMs(job)` plus a buffer, keeps a fallback for no-timeout jobs, and adds focused cron regression coverage plus comment updates.

## Best Possible Solution

Request changes on this PR: keep the per-job stale-lock threshold for finite normal timeouts, preserve the conservative no-timeout fallback, handle overflow-sized timeout values with an explicit policy, and add regression coverage for short custom timeouts, default systemEvent and agentTurn thresholds, no-timeout fallback, and explicit long-timeout preservation.
