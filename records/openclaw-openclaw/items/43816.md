## Summary

Keep this PR open. Current main still uses a fixed two-hour cron stale-running cleanup threshold in timer maintenance, while execution timeouts are resolved per job. The PR addresses a real remaining reliability gap, but the current PR head should not merge as-is because its two-hour ceiling can clear locks before explicit long-running timeouts have elapsed.

## Best Possible Solution

Keep this PR open and request changes: rebase on current main; derive stale-running cleanup from `resolveCronJobTimeoutMs(job) + buffer` for finite normal timeouts; preserve a conservative fallback for `undefined` no-timeout jobs; handle overflow-sized timeout values with an explicit policy that does not shorten valid long configured timeouts; and add regression coverage for short custom timeouts, systemEvent defaults, no-timeout fallback, and explicit long-timeout preservation.

## Risks / Open Questions

- Merging the current PR head as-is could allow duplicate overlapping cron executions for explicit `timeoutSeconds` values above two hours, because maintenance can clear `runningAtMs` at the two-hour ceiling before the configured execution timeout has elapsed.
- Closing this PR would drop a focused implementation candidate for a still-present timer-maintenance reliability gap; the linked bug report #30096 is closed, but current main only partially addresses the broader stale-lock problem via startup/manual paths.
- The PR should be rebased and expanded with regression coverage for no-timeout jobs and explicit long-timeout preservation before merge.
