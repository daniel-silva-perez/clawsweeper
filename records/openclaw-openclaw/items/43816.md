## Summary

Keep open. Current main still uses a fixed two-hour stale-running cleanup window, so the PR targets a real remaining cron reliability gap, but the current PR head has a blocking correctness issue around explicit long-running job timeouts.

## What This Changes

The PR changes cron maintenance to derive stale `runningAtMs` cleanup from `resolveCronJobTimeoutMs(job)` plus a five-minute buffer, preserves a two-hour fallback for no-timeout jobs, and adds focused cron regression tests/comment updates.

## Best Possible Solution

Request changes on this PR: keep the per-job stale-lock threshold for ordinary finite timeouts, preserve the conservative no-timeout fallback, handle overflow-sized timeout values with a targeted policy, and add regression coverage for short custom timeouts, default systemEvent and agentTurn thresholds, no-timeout fallback, and explicit long-timeout preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve locks until explicit long timeouts expire:** `src/cron/service/jobs.ts:394`
  - body: With an explicit `timeoutSeconds` above two hours, runtime still allows the job to run for that configured duration, but this clamp clears `runningAtMs` at the two-hour fallback. A later maintenance tick can then make the same job runnable while the original execution is still in progress. Handle overflow-sized values separately without shortening valid long timeouts.
  - confidence: 0.9
