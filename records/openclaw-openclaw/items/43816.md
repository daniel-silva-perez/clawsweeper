## Summary

Keep open. Current main still uses a fixed two-hour stale-running threshold, so the PR addresses a real remaining cron recovery gap, but the latest PR head introduces a blocking correctness regression by capping valid finite timeouts above two hours to the two-hour fallback.

## What This Changes

The PR imports the cron timeout policy into job maintenance, replaces fixed two-hour stuck-lock cleanup with a per-job timeout-plus-buffer threshold capped at the two-hour fallback, and adds cron regression tests plus a comment update.

## Best Possible Solution

Request changes on this PR: keep the per-job stale-lock threshold for ordinary finite timeouts, preserve the conservative no-timeout fallback, handle overflow-sized timeout values separately, and add coverage for short custom timeouts, default systemEvent and agentTurn thresholds, no-timeout fallback, and valid long finite-timeout preservation.

Handle overflow-sized values separately without shortening valid long timeouts.
