## Summary

Keep open. The PR is still useful because current main retains a fixed two-hour stale-running-marker threshold, but the branch now introduces a correctness regression by capping every finite timeout-derived lock threshold at two hours, including valid explicit job timeouts longer than two hours.

## What This Changes

The branch imports cron timeout policy into job maintenance, replaces fixed stale running-marker cleanup with timeout-plus-buffer logic capped at a two-hour fallback, and adds/updates cron regression tests.

## Best Possible Solution

Request changes on this PR: keep timeout-derived cleanup for normal finite jobs, preserve the conservative no-timeout fallback, handle only overflow-sized timeout values specially, and add regression coverage for short custom timeouts, default systemEvent and agentTurn thresholds, no-timeout fallback, valid long finite timeouts, and overflow-sized timeoutSeconds.

## Reproduction Assessment

Yes. A source-level reproduction is an agentTurn cron job with a valid explicit timeout above two hours, such as three hours, and a runningAtMs lock just over two hours old; the PR clears that lock even though runtime would still allow the original execution to continue.

## Solution Assessment

No. The timeout-plus-buffer direction is narrow and maintainable, but clamping all finite values to two hours is too broad; the safer fix is to cap or reject only values beyond Node's timer ceiling while respecting valid configured long timeouts.
