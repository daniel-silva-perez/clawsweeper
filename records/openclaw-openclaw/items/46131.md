## Summary

Keep this PR open, but do not merge the current branch as-is. The model bug is still present on current main, while the PR branch is stale/mixed: the intended model fix needs to be ported to the current split inline-provider file, and the unrelated Telegram retry change weakens strict non-idempotent send retry semantics.

## What This Changes

The PR applies provider-level API defaulting to inline provider model construction, adds regression tests for that behavior, exports a config default helper, and also broadens Telegram send retry classification for rate-limit and grammY failed-after errors.

## Best Possible Solution

Land a model-only patch against current main that normalizes provider policy/default API before building inline provider models, preserves explicit model-level and provider-level API precedence, adds focused inline-provider regression coverage, and leaves Telegram retry behavior unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is an Anthropic inline provider with models and no explicit `api`: current `buildInlineProviderModels` leaves `api` undefined, and `resolveExplicitModelWithRegistry` then skips the inline match because `inlineMatch?.api` is false.

## Solution Assessment

No, not as this PR branch. The model-fix direction is the right narrow solution, but the implementation needs to target `src/agents/pi-embedded-runner/model.inline-provider.ts` on current main and drop the unrelated Telegram retry change.
