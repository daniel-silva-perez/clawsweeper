## Summary

Keep open. The model bug is still a valid current-main gap, but this PR should not merge as-is because the branch mixes the model fix with unrelated Telegram safe-send retry behavior that can replay non-idempotent sends.

## What This Changes

The branch applies provider-level API defaulting to inline provider models and adds regression tests, and it also changes Telegram rate-limit and grammY failed-after send retry handling.

## Best Possible Solution

Land a model-only replacement against current main: normalize provider policy before inline provider models are built, cover implicit Anthropic API defaulting in `model.inline-provider.test.ts`, and leave the current Telegram retry semantics unchanged.

## Reproduction Assessment

Yes. The source-level path is high confidence: a raw Anthropic inline provider with models and no `api` reaches `buildInlineProviderModels` with `model.api ?? entry?.api` undefined, then `resolveExplicitModelWithRegistry` skips it because `inlineMatch?.api` is false.

## Solution Assessment

No, not as this PR branch. The model-only direction is the right narrow fix, but current main should be patched in `model.inline-provider.ts` using the existing provider-policy normalization surface, without the unrelated Telegram retry changes.

## Review Findings
