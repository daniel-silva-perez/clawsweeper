## Summary

Keep open. The provider-level API defaulting bug still exists on current main, but this PR should not merge as-is because the branch also adds unrelated Telegram retry behavior that can replay non-idempotent sends.

## What This Changes

This PR applies provider-level API defaulting when building inline provider models, adds model regression tests, exports the default helper, and also broadens Telegram retry classification for rate-limit and grammY failed-after errors.

## Best Possible Solution

Land a model-only patch against current main that normalizes provider policy before inline provider models are built, adds focused inline-provider regression coverage, and leaves Telegram retry semantics unchanged.

## Reproduction Assessment

Yes. A current-main Anthropic inline provider with models and no explicit `api` reaches `buildInlineProviderModels` with `api` undefined, then `resolveExplicitModelWithRegistry` skips it because `inlineMatch?.api` is false.

## Solution Assessment

No, not as this PR branch. The model fix direction is narrow and valid, but the implementation needs to be ported to `model.inline-provider.ts` on current main and the unrelated Telegram retry change should be dropped.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep generic failed-after sends out of strict retries:** `src/telegram/network-errors.ts:216`
  - body: `isSafeToRetrySendError` is used for strict retries on non-idempotent Telegram sends, but this change treats generic grammY failed-after envelopes as safe even when the nested cause may be absent or a post-send transport error. That can replay `sendMessage` and duplicate visible messages; keep this predicate limited to pre-connect evidence and handle 429s separately.
