## Summary

Keep open. Current main contains a partial Windows fix that matches uizuKim's one-line diagnostic for compiled JavaScript plugin artifacts, but the source TypeScript plugin path named in this issue still intentionally routes through jiti and remains a plausible source-checkout/dev-runtime Windows latency lane.

## What This Changes

The issue reports pathological Windows latency when TypeScript source plugins load through the synchronous jiti-based plugin loader and asks for a faster TS-aware native import path for real production callers.

## Best Possible Solution

Keep the compiled-JS Windows fast path, but leave this issue open as the source TypeScript loader performance lane until maintainers choose and validate a Windows-safe strategy: either an async TS-aware native import path for source checkout callers or deterministic routing from source entries to built runtime artifacts where available.

## Reproduction Assessment

No high-confidence live reproduction from this review; I did not run a Windows source checkout. The static path is high-signal, though: current main still routes source `.ts` plugin entries through jiti, matching the reported bottleneck class.

## Solution Assessment

No. The current main fix is appropriate for compiled bundled JavaScript modules, but it is not the complete solution for the TypeScript source-loading problem described here; changing that path likely needs loader/caller design review and Windows profiling.
