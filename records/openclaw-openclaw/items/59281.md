## Summary

Keep open. Current main partially addresses the adjacent Windows bundled-plugin slowdown for built JavaScript artifacts, but the reported source TypeScript path remains routed through the synchronous jiti loader, and there is no current-main async native-import TS-aware path covering the real production call sites named in the issue.

## What This Changes

The issue asks OpenClaw to avoid pathological Windows latency when production plugin-loading paths load TypeScript source plugin entries through the synchronous jiti path.

## Best Possible Solution

Keep the issue open until maintainers decide and implement a supported Windows source-plugin loading strategy, such as a versioned async loader path or a documented built-runtime selection path that preserves plugin SDK aliasing and works across CLI, status/inspect, gateway bootstrap, HTTP tools, TTS, and media-understanding call sites with Windows profiling proof.

## Reproduction Assessment

Yes: the issue provides concrete Windows 11 source-checkout steps and timings across real production plugin callers. Current main still has a matching source path because TypeScript entries return `tryNative: false` and source TypeScript processes prefer `extensions/`.

## Solution Assessment

No: the current main-only native-require fast path is a useful partial mitigation for built JavaScript plugin artifacts, but it is not the requested async native-import TS-aware solution for source TypeScript entries. A broader loader design is needed before this can be closed as fixed.
