## Summary

Keep open. The underlying user problem is still a valid skills diagnostics gap, but this PR is incomplete: it only annotates skills that already survived loading and it does not render parse-error state consistently across the `skills info` and `skills check` commands.

## What This Changes

The PR adds a `parseError` field to skills/status entries, marks parse-error skills ineligible, and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Preserve read/parse diagnostics at the local skill discovery boundary, carry that diagnostic state through the skills status model, and render it consistently in `skills list`, `skills info`, and `skills check` text/JSON output with focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve diagnostics before dropping skills:** `src/agents/skills/workspace.ts:513-518`
  - body: The patch records `parseError` only while re-reading skills that were already returned by the loader. On current main, read/parse failures in `loadSingleSkillDirectory` return `null`, so malformed `SKILL.md` files never become status records that list, info, or check can render.
  - confidence: 0.88
- **[P2] Show parse errors in skill info:** `src/cli/skills-cli.format.ts:189-195`
  - body: `formatSkillInfo` still derives status only from eligible, disabled, and allowlist state. A parse-error skill would render as ready or needs setup, and the detailed info command would not show the actual error.
  - confidence: 0.9
- **[P2] Add a parse-error bucket to skills check:** `src/cli/skills-cli.format.ts:296-318`
  - body: `formatSkillsCheck` still groups every non-eligible, non-disabled, non-blocked skill under missing requirements. Parse-error skills need their own summary and JSON/text section so users do not see an empty or misleading missing-requirements entry.
  - confidence: 0.9
