## Summary

Keep open. Current main already shipped the tolerant frontmatter parser that fixes the original colon-in-description silent-drop case, but it does not implement this PR's explicit parse/read-error diagnostic surface; the submitted patch is also incomplete because it captures diagnostics after failed skills can already be dropped and only updates `skills list`, leaving `skills info` and `skills check` misleading.

## What This Changes

The PR adds `parseError` fields to skill entries and status records, marks parse-error skills ineligible, and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Keep the shipped tolerant parser. If maintainers still want explicit diagnostics for unrecoverable local skill read/parse failures, rework discovery to return diagnostic records before failed skills are filtered out, plumb them through the status contract, and render them consistently in `skills list`, `skills info`, and `skills check` text and JSON with focused regression coverage.

## Reproduction Assessment

Yes, for the PR defects. Code inspection gives a high-confidence path: a failed `loadSingleSkillDirectory` returns `null` before any `SkillEntry` exists, and even a synthetic parse-error status record would still be ignored or misbucketed by `skills info` and `skills check`.

## Solution Assessment

No. The submitted approach is too late in the loader pipeline and too narrow in the formatter layer; the safer solution is a discovery-level diagnostic contract plus complete list/info/check coverage while preserving the current safe loader behavior.
