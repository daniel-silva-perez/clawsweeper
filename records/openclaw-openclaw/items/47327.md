## Summary

Keep open. The PR targets a real skills diagnostics gap, but the latest diff is still incomplete: it records errors only after discovery has already dropped failed local skills, and it only updates `skills list` while `skills info` and `skills check` remain misleading for parse-error records.

## What This Changes

The PR adds `parseError` fields to skill entries/status records, marks parse-error skills ineligible, and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Preserve local skill read/parse diagnostics at the discovery boundary, carry that state through `SkillEntry` and `SkillStatusEntry`, and render it consistently in `skills list`, `skills info`, and `skills check` text/JSON output with focused regression tests.

## Reproduction Assessment

Yes, for the remaining defect. Code inspection gives a high-confidence reproduction path: make local skill discovery fail before `SkillEntry` creation, then verify no status record reaches the CLI; separately inject a parse-error status record and see `info`/`check` ignore or misbucket it.

## Solution Assessment

No. The PR is pointed in the right direction, but the current implementation is too late in the pipeline and too narrow in the formatter layer; the safer fix is discovery-level diagnostics plus complete CLI status/check/info coverage.
