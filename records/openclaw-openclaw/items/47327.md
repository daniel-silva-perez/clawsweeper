## Summary

Keep open. Current main still has no status model for skill parse diagnostics, and this PR is useful but incomplete: it only threads `parseError` into list output while leaving the loader-level drop path, `skills info`, and `skills check` misleading for parse-errored skills.

## What This Changes

The PR adds a `parseError` field to skills/status entries, marks parse-error skills ineligible, and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Preserve read/parse diagnostics at the local skill discovery boundary, carry that state through `SkillEntry` and `SkillStatusEntry`, and render it consistently in `skills list`, `skills info`, and `skills check` text/JSON output while keeping the current frontmatter parser's inline-colon recovery behavior.

## Reproduction Assessment

Partially. The original unquoted-colon example is already covered by current parser tests, but the remaining diagnostic gap is source-reproducible: loader read/parse failures return `null` before a skill status record exists, so the CLI cannot surface them.

## Solution Assessment

No. The PR is pointed in the right direction, but it records diagnostics too late and only renders one CLI surface; the safer fix is to preserve diagnostics during local loading and update all three documented inspection commands with focused tests.
