## Summary

Keep open. Current main fixed the common colon-in-description frontmatter case, but it still does not implement the PR's broader diagnostic goal, and the submitted patch is incomplete because dropped local skills can still disappear before any `parseError` field exists while `skills info` and `skills check` remain misleading.

## What This Changes

The branch adds a `parseError` field to skill entries and status records, marks parse-error skills ineligible, and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Keep the tolerant frontmatter parser. If maintainers still want explicit diagnostics for unrecoverable local skill read/parse failures, add a discovery-level diagnostic record before failed skills are filtered out, plumb it through the skill status contract, and render it consistently in `skills list`, `skills info`, and `skills check` text and JSON with focused regression tests.

## Reproduction Assessment

Yes, for the PR defects. Code inspection gives a high-confidence path: malformed or unreadable `SKILL.md` files can return `null` in local discovery before status records exist, and even a synthetic parse-error status would still be ignored or misbucketed by `skills info` and `skills check`.

## Solution Assessment

No. The submitted approach is too late in the loader pipeline and only updates the list formatter; a maintainable fix needs a discovery diagnostic contract plus complete list/info/check handling.
