## Summary

Keep this PR open. Current main still does not preserve SKILL.md parse/read diagnostics for skills status, and the PR remains relevant, but the branch is incomplete against the current loader/status/CLI shape: malformed local skills can still be dropped before status rendering, and the info/check formatters still do not consistently surface parse errors.

## What This Changes

The PR branch adds a parseError field to skill/status entries, marks parse-error skills ineligible, and shows parse errors in skills list text and JSON output.

## Best Possible Solution

Rebase the diagnostics change onto current main by preserving read/parse failures in the local skills loader, carrying a parse-error status through SkillEntry and SkillStatusEntry, rendering it consistently in list/info/check text and JSON, and adding focused regression tests while keeping the current tolerant frontmatter parser behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve diagnostics before the loader drops skills:** `src/agents/skills/workspace.ts:504-508`
  - body: On current main, malformed SKILL.md files are rejected inside the local loader before loadSkillEntries builds SkillEntry records. Capturing parseError only while re-reading already-loaded skills leaves parse-failed workspace skills absent from list/info/check, so the loader needs to return a diagnostic or placeholder record for failed SKILL.md files.
  - confidence: 0.88
- **[P2] Handle parse errors in skill info output:** `src/cli/skills-cli.format.ts:189-195`
  - body: The detailed info formatter still derives status only from eligible/disabled/allowlist state. A parse-error skill would render as ready or needs setup and the actual error text would not appear, which leaves one of the user-facing diagnostics paths misleading.
  - confidence: 0.9
- **[P2] Add a parse-error bucket to skills check:** `src/cli/skills-cli.format.ts:296-318`
  - body: skills check still groups every non-eligible, non-disabled, non-blocked skill under missing requirements. Parse-error skills should be counted and rendered separately in text and JSON; otherwise the check report can show an empty missing-requirements entry instead of the parse failure.
  - confidence: 0.9
