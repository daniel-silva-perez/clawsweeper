## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve legacy Telegram DM thread ids:** `src/config/sessions/delivery-info.ts:34-36`
  - body: `resolveSessionThreadIdForRouting()` drops every Telegram DM `:thread:` suffix unless the key also contains `:topic:`. Existing session-key consumers still accept `agent:main:telegram:dm:...:thread:<id>` as a real thread-bearing key, so run-context, ACP, and native subagent routing from those sessions would fall back to the DM root.
  - confidence: 0.86
- **[P2] Use the requester agent store for inferred delivery context:** `src/agents/command/run-context.ts:12-13`
  - body: The new fallback calls `extractDeliveryInfo(trimmedSessionKey)` without selecting the agent store for the supplied session key. For named-agent sessions resumed without `to`, this reads the default store, leaves `currentChannelId` empty, and same-turn threaded replies still route as if no channel context was restored.
  - confidence: 0.8
- **[P2] Normalize restored channel ids before reusing them:** `src/agents/command/run-context.ts:57-59`
  - body: This assigns stored `deliveryContext.to` directly to `currentChannelId`. Slack stores values such as `slack:C...` or `channel:C...`, while Slack auto-threading compares against the bare parsed channel id, so resumed Slack thread sessions can still miss auto-threading after this fallback runs.
  - confidence: 0.78
- **[P2] Keep failed Feishu explicit replies inside the topic:** `extensions/feishu/src/outbound.ts:52-56`
  - body: When an explicit `replyToId` is present, `resolveReplyContext()` forces `replyInThread: false` even if a `threadId` is also available. If Feishu later rejects that quoted target as withdrawn/missing, the send fallback posts at the chat root instead of failing closed in the topic, which can leak topic-scoped replies out of the thread.
  - confidence: 0.76
