## Summary

Keep open. The product fix direction is well aligned with the linked OpenAI Codex `xhigh` report, but this PR is not merge-ready because the new test overwrites OpenClaw's shared plugin-registry singleton with a partial object, which can break later files in the repo's non-isolated Vitest lanes.

## What This Changes

The PR changes provider thinking resolution to fall back to provider runtime hooks for startup-inactive providers, adds hook-alias matching and runtime-provider caching, and adds regression coverage for the OpenAI Codex `xhigh` path.

## Best Possible Solution

Keep the runtime fallback approach, but adjust the new regression test to use the real runtime helpers or preserve and restore the full `RegistryState` shape instead of replacing the global symbol with a partial object. After that, validate the provider-thinking path together with the gateway/session tests that consume it.

## Reproduction Assessment

Yes. The linked report has a concrete gateway/session path for `openai-codex/gpt-5.5` with `thinkingDefault: "xhigh"`, and current main shows why it can fail when the provider is not in the active startup registry; the PR-introduced test issue is reproducible in a non-isolated Vitest worker that imports runtime after this test replaces the global singleton.

## Solution Assessment

No, not as submitted. Loading provider-owned hooks on demand is the right product boundary, but the test fixture should not mutate the shared runtime singleton outside the runtime helper contract.
