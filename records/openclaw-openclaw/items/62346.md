## Summary

Keep open. The PR targets the right recovery symptom from the linked aborted-run report, but current `main` still has the old `incomplete-text -> prefill: true` helper behavior, the helper has no production caller in current `main`, and the proposed branch would delete valid assistant content for an `incomplete-text` assessment that can be triggered by any empty text block alongside meaningful output.

## What This Changes

This PR changes `sanitizeThinkingForRecovery` so `incomplete-text` drops the last assistant message and returns `prefill: false`, with the focused test updated to expect that behavior.

## Best Possible Solution

The recovery fix should operate on the actual current runtime path for aborted-run replay, avoid assistant prefill for truly incomplete assistant output, and preserve any meaningful assistant content in mixed-content turns; the broader state semantics should remain tracked with the linked recovery issue until the narrow behavior and production call path are confirmed.

## Reproduction Assessment

Yes for the PR regression: construct an assistant message with signed thinking, one meaningful text block, and one empty text block; current assessment returns `incomplete-text`, and the PR's new fall-through would remove the whole assistant message. For the broader live aborted-run inconsistency, the linked issue and PR comments provide controlled local observations, but I did not live-reproduce it in this read-only pass.

## Solution Assessment

No. Dropping all `incomplete-text` assistant turns is not the narrowest maintainable fix because the assessment currently includes mixed-content messages; a safer fix would distinguish reasoning-only or empty-visible-output turns from messages that still contain replayable assistant text, and wire that behavior into the production recovery path.
