## Summary

Keep this PR open. Current main has generic Feishu interactive reply rendering, so forwarded approval payloads can become native Feishu cards with quick-command buttons, but it does not implement the PR’s native Feishu exec-approval client: there is no `channels.feishu.execApprovals` config surface, no Feishu exec approval handler/runtime, no approver-DM/native target routing for Feishu, and no in-place pending-card update flow on resolution or expiry. The PR also still has security/authorization review work from its final unresolved review comment before it would be ready to land.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is to land a Feishu-owned native exec-approval implementation that adds the Feishu `execApprovals` config/docs, uses the shared native approval capability/runtime pattern, validates only callback-matchable `ou_` approver IDs, preserves existing `allowFrom` authorization behavior, and includes focused tests for card actions, routing, resolution updates, expiry, and multi-account handling. Current generic Feishu interactive cards can be reused, but they do not supersede the Handler-pattern work.
