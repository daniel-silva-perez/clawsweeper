## Summary

Keep open. Current main still lacks Feishu native exec approval cards, and this PR is the active implementation candidate for the related Feishu approval-card cluster, but the latest head still has a blocking approver-normalization bug and non-green checks.

## What This Changes

This PR adds Feishu exec approval interactive cards, a native approval handler/runtime adapter, card action resolution through the gateway, Feishu exec approval config, forwarding suppression hooks, SDK exports, outbound card sending support, and Feishu-focused tests.

## Best Possible Solution

Keep this implementation path, but require the approver parser to re-check that parsed direct IDs are valid Feishu `ou_` open_ids, add regression tests for prefixed invalid values, update the user-facing Feishu/exec-approval docs and changelog if the feature lands, and rerun the Feishu extension plus changed gates before merge.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with `approvers: ["user:foo"]`, the PR head records `foo` as an approver and enables the native client, but card callbacks later compare the clicker to `event.operator.open_id`, so the configured approver can never authorize a real click.

## Solution Assessment

Not yet. The Handler/native-runtime direction matches the existing shared approval architecture, but the current patch is not the best mergeable solution until the parsed-ID validation bug, tests, docs, and failing checks are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate parsed Feishu approver IDs:** `extensions/feishu/src/exec-approvals.ts:34`
  - body: `normalizeFeishuDirectApproverId` returns any value from `parseFeishuDirectConversationId` before checking that it is an `ou_` open_id. Since the parser accepts arbitrary `user:` suffixes, `user:foo` counts as an approver and enables the native client, but card callbacks authorize against `event.operator.open_id`, so no real click can match that value. Re-check `parsed.startsWith("ou_")` before returning it and add tests for prefixed invalid IDs.
