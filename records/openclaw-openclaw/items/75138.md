## Summary

Keep this PR open. Current main still omits `<stateDir>/gateway.env` from the global runtime dotenv chain, and the PR is a narrow, active fix for that gap; the remaining blocker is policy/docs polish for the new user-facing env-loading behavior.

## What This Changes

The PR adds `<stateDir>/gateway.env` to global runtime dotenv loading after `<stateDir>/.env` and adds a unit test that loads `GATEWAY_AUTH_TOKEN` while preserving `.env` precedence.

## Best Possible Solution

Land the focused loader/test fix after adding the required release note and aligning the environment documentation so users can discover that `$OPENCLAW_STATE_DIR/gateway.env` is now part of the trusted runtime dotenv chain.

## Reproduction Assessment

Yes. A source-level reproduction is clear: put `GATEWAY_AUTH_TOKEN` only in `$HOME/.openclaw/gateway.env`, leave it unset in the shell, and current main's loader will not read that file; the PR adds a focused unit test for the missing path.

## Solution Assessment

Functionally yes: inserting the state-dir `gateway.env` between state `.env` and the legacy XDG fallback is the narrowest maintainable code fix and preserves `.env` precedence. As submitted, it still needs the required changelog/docs alignment before merge.
