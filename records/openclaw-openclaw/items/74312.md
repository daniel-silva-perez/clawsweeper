## Summary

Keep open. Current main still has the reported auth-epoch sensitivity: Claude CLI token-shaped credentials and auth-profile token credentials include token material in the epoch hash, current tests still expect token changes to invalidate the epoch, and the parser can still fall back to token mode when a Claude keychain read lacks `refreshToken`. The linked closing PR #74493 is open and appears to be the right implementation path, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Prevent Claude CLI OAuth token rotations or transient access-token-only keychain reads from changing CLI auth epochs and resetting active sessions mid-conversation.

## Best Possible Solution

Land PR #74493 or an equivalent narrow fix that makes Claude CLI token-shaped epochs identity-only, removes token material from token auth-profile epoch hashes, bumps the auth epoch version to avoid migration resets, and keeps targeted tests proving token rotation stability plus real identity-change invalidation.

## Reproduction Assessment

Yes for the core reset trigger: current source and tests show that changing a token-shaped Claude credential changes `resolveCliAuthEpoch()`, and session reuse maps that mismatch to `reason=auth-epoch`. The exact macOS keychain race was not live-reproduced here, but the parser fallback and reporter logs make the reported path credible.

## Solution Assessment

Yes: the linked PR's identity-only hashing plus epoch-version bump is the narrowest maintainable solution for the reported symptom. Parser hardening could be a defense-in-depth follow-up, but it is not required to make transient token-only reads stop resetting sessions.
