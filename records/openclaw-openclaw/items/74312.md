## Summary

Keep open. Current main still hashes Claude token-shaped credentials and token auth-profile material into the CLI auth epoch, so token rotation or a transient token-shaped Claude credential can still invalidate reusable CLI sessions. The linked open PR #74493 is the right implementation path to review rather than closing this issue before it lands.

## What This Changes

Prevent Claude CLI OAuth token rotation or transient token-shaped keychain reads from changing the CLI auth epoch and resetting active sessions mid-conversation.

## Best Possible Solution

Review and land #74493 or an equivalent narrow change that makes Claude token-shaped auth epochs identity-based, removes raw token material from token auth-profile epoch hashing where appropriate, bumps the epoch version for migration, and adds regression tests for token rotation plus real identity changes.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: a Claude CLI credential parsed as `type: "token"` changes the epoch when `credential.token` changes, and same-version epoch mismatches reset reusable sessions with `reason=auth-epoch`.

## Solution Assessment

Yes. Identity-only hashing at the epoch layer is the narrowest maintainable fix for the observed token-rotation reset, while parser hardening can remain a defense-in-depth follow-up.
