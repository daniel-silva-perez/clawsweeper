## Summary

Keep this PR open. Current main still rolls stale daily/idle sessions into a new session and emits typed plugin lifecycle hooks, but it does not emit the internal `command:reset` event that the bundled `session-memory` hook consumes. The supplied PR context shows this is an active, focused fix with maintainer repair/validation comments and no obvious security or supply-chain red flags.

## Best Possible Solution

Keep this PR open and review or land it as the focused canonical fix for automatic daily/idle rollover memory persistence. The right path is to emit the existing internal `command:reset` event for non-command stale rollovers with `commandSource` values like `auto:daily` and `auto:idle`, preserve `previousSessionEntry` and workspace context for `session-memory`, keep daily/idle/fresh no-op coverage, and add a docs note if maintainers want the expanded command-event semantics documented.
