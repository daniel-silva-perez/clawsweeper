## Summary

Keep open. Current main has useful backend tool-event plumbing, including `tool-events` WebSocket delivery and `session.tool`, but it still does not provide the requested first-class Activity panel / observer mode. Control UI still projects tool events into chat state and renders them inside the single chat thread, `session.tool` is not consumed as an independent observer stream, and the TUI still gates tool-card rendering on verbose mode. The open draft PR #60814 remains an implementation candidate, not merged current-main behavior.

## Best Possible Solution

Keep this issue open as the broad observer-mode tracker. The best path is maintainer-scoped product work that builds on existing `tool-events` and `session.tool` plumbing, adds a dedicated Control UI Activity panel with independent state, scrolling, filtering, collapse/expand, and output truncation controls, then deliberately decides whether per-connection activity settings, TUI split-pane behavior, and external observer streams belong in the same milestone or narrower follow-ups. Treat #60814 as an open draft implementation candidate and #50762 as the narrower exec-log split, not as merged resolutions.
