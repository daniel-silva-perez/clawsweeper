## Summary

Keep open. Current main has useful observer plumbing, including `tool-events`, `session.tool`, and `command_output`, but it still does not provide the requested first-class Activity panel / observer mode. Control UI still turns live tool events into chat-thread tool cards, does not consume `session.tool` as an independent activity model, and the TUI still suppresses tool events when verbose is off. Open draft PR #60814 remains an implementation candidate, not merged current-main behavior.

## Best Possible Solution

Keep this issue open as the broad observer-mode tracker. The best path is maintainer-scoped product work that builds on existing `tool-events`, `session.tool`, `item`, `approval`, and `command_output` plumbing, adds a dedicated Control UI Activity panel with independent state, scrolling, filtering, collapse/expand, and output truncation controls, then deliberately decides whether per-connection activity settings, TUI split-pane behavior, and external observer streams belong in the same milestone or narrower follow-ups. Treat #60814 as an open draft implementation candidate and #50762 as the narrower exec-log split, not as merged resolutions.
