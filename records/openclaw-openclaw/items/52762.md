## Summary

Keep open. Current main and the latest release still have the eager-prune behavior and old bounded-map loop, so this PR is not obsolete; however, the PR has unresolved review concerns around saturated-cache performance and malformed numeric limit semantics on public plugin SDK helper surfaces, so it needs maintainer review rather than automatic cleanup.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete old map entries with a counted forward pass.

## Best Possible Solution

Land or replace this work with a focused shared-helper fix that removes per-message full scans in both normal and saturated cache states, preserves or explicitly validates invalid numeric limit handling, keeps Discord-owned behavior in the Discord plugin, and proves the helper plus Discord replay path with targeted tests and changed-gate validation.

## Risks / Open Questions

- The PR's lazy guard still calls the full TTL/size prune path whenever the dedupe cache is saturated, so the busiest Discord servers may retain the O(n) hot-path scan flagged in review.
- The `pruneMapToMaxSize()` rewrite can change malformed numeric limit behavior, including `NaN`, for webhook in-flight, rate-limit, and anomaly guard callers that depend on bounded in-memory tracking.
- Because `dedupe-runtime`, `persistent-dedupe`, and `collection-runtime` are plugin SDK surfaces used by bundled and third-party plugins, cache cleanup semantics need maintainer approval before merge.
