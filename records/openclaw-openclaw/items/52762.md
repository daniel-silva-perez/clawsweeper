## Summary

Keep this PR open. Current main still has `createDedupeCache().check()` pruning immediately after every newly recorded key, and `pruneMapToMaxSize()` still uses the old repeated `map.keys().next()` loop. The Discord inbound replay guard still uses a 5-minute, 5000-entry claimable dedupe cache backed by that shared helper. A later Discord listener/run-queue refactor helps the related Carbon listener timeout surface, but it does not implement this PR's lazy-prune/shared-cache change, and review comments flagged real correctness/performance concerns that need maintainer review before merge.

## Best Possible Solution

Keep this PR open for maintainer review, but do not land it as-is. The best path is a shared dedupe/map-size fix that avoids per-message full scans on saturated caches, preserves or explicitly validates invalid numeric limit behavior, adds focused regression tests for those semantics, and then verifies the Discord inbound replay path plus plugin-SDK callers.

## Risks / Open Questions

- The proposed `DedupeCache` change still runs the full TTL/size prune when the cache is over `maxSize`, so high-throughput Discord accounts may still pay O(n) work on new messages after the 5000-entry window saturates.
- The proposed `pruneMapToMaxSize()` rewrite changes invalid `NaN` max-size behavior from leaving the map untouched to deleting all entries, which can weaken shared webhook/dedupe guard tracking if malformed numeric config reaches the helper.
- Because these helpers are exported through plugin SDK subpaths, changes affect bundled and third-party plugins, not only Discord.
