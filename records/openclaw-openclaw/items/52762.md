## Summary

Keep open. Current main and the latest release still have the eager dedupe prune behavior this PR targets, but the PR should not merge as-is because it still full-prunes once the cache is saturated and changes shared map-pruning semantics in a way that can clear webhook guard state for invalid numeric limits.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only when the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete old map entries with a counted forward pass.

## Best Possible Solution

Land or replace this with a focused shared-helper fix that removes per-message full scans in both normal and saturated cache states, keeps memory bounded, preserves or explicitly validates invalid numeric limit handling, and proves the infra helper plus Discord and webhook guard callers with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. Discord uses a five-minute, 5000-entry replay guard; if the live unique-message set stays saturated, every new key still scans roughly 5000 entries, so the delay this PR targets can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit behavior in map pruning:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and `i++ >= deleteCount` never breaks, deleting every entry. Current behavior leaves `NaN` limits untouched, and webhook guards can pass `NaN` after numeric normalization, so this can make guard state forget active keys.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No CI, dependency, secret, or supply-chain change was found, but the shared map-pruning rewrite can weaken webhook guard state for invalid numeric limits.

Concerns:

- **[medium] Preserve webhook guard state for invalid limits:** `src/infra/map-size.ts:10`
  - body: The counted loop deletes every tracked entry when `maxSize` is `NaN`; in-repo webhook guards can normalize malformed `maxTrackedKeys` input to `NaN` before pruning, which can clear active in-flight or rate-limit tracking.
  - confidence: 0.84
