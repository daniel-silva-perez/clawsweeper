## Summary

Keep this PR open for maintainer review. Current main and the latest release still have the eager dedupe prune behavior the PR is trying to improve, but the proposed patch has unresolved correctness concerns: saturated caches still hit the full prune path on every new key, and the map-size rewrite changes invalid-limit behavior for public SDK guard callers.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete old entries with a counted forward pass.

## Best Possible Solution

Land or replace this with a focused shared-helper fix that removes per-message full scans in both normal and saturated cache states, preserves or explicitly validates invalid numeric limit handling, keeps public SDK helper semantics stable, and proves the helper plus Discord replay path with targeted tests and changed-gate validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: This still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's 5-minute, 5000-entry replay guard, a server above roughly 17 unique inbound messages per second keeps the cache saturated, so every new message still scans about 5000 entries and the delay this PR targets can continue.
  - confidence: 0.91
- **[P2] Preserve invalid-limit behavior in map pruning:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior is effectively a no-op for `NaN`; several SDK guard callers can pass `NaN` after `Math.max(..., Math.floor(...))`, so this can make in-flight or rate-limit guards forget active keys.
  - confidence: 0.88
