## Summary

Keep open. Current main and the latest release still have the eager dedupe prune path, so the PR is not obsolete, but the proposed patch should not merge as-is because it still does a full prune once the Discord replay cache is saturated and it changes shared map-pruning behavior for invalid numeric limits used by webhook guard helpers.

## What This Changes

The PR changes `createDedupeCache().check()` to call `prune(now)` only when the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete a computed number of oldest entries with a forward `for...of` pass.

## Best Possible Solution

Replace or revise the patch with a bounded pruning strategy that avoids per-message full TTL scans in both normal and saturated cache states, preserves current invalid numeric limit behavior or validates those inputs explicitly, and covers the shared helper plus Discord replay and webhook guard callers with focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by filling a 5000-entry dedupe cache with unique keys under the five-minute Discord TTL and observing that the PR still calls the full `prune(now)` path on each overflow; a small read-only node simulation also reproduces the `NaN` map-pruning semantic change.

## Solution Assessment

No. The direction is useful, but this patch is not the narrowest maintainable fix because it only delays the full scan until saturation and rewrites a public SDK helper in a way that changes guard-state semantics for invalid limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid full-cache prune once saturated:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a saturated unique-message stream grows to 5001 entries and scans the cache on every new key, so the event-loop delay this PR targets can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning semantics:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior is effectively a no-op for `NaN`; webhook guard callers can propagate `NaN` after numeric normalization, so this can make in-flight or rate-limit guards forget active keys.
  - confidence: 0.88
