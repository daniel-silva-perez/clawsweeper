## Summary

Keep open. Current main and the latest release still use the eager `DedupeCache` prune path this PR is trying to optimize, so the work is not obsolete; however, the proposed patch should not merge as-is because it still does a full prune for every new key once the cache is saturated and it changes `pruneMapToMaxSize(..., NaN)` from a no-op into clearing tracked guard state.

## What This Changes

The PR changes `createDedupeCache().check()` to call `prune()` only when the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete old entries with a counted forward pass.

## Best Possible Solution

Land or replace this with a focused shared-helper fix that removes per-message full scans in both normal and saturated cache states, keeps memory bounded, preserves or explicitly validates invalid numeric limit handling, and proves the infra helper plus Discord replay and webhook guard callers with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a saturated unique-message stream triggers that O(n) scan on every new key, so the event-loop delay this PR targets can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit behavior in map pruning:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior leaves the map untouched, and SDK webhook guards can pass `NaN` after numeric normalization, so this can make in-flight or rate-limit guards forget active keys.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No CI, dependency, secret, or supply-chain change was found, but the shared map-pruning rewrite can weaken SDK webhook guard state for invalid numeric limits.

Concerns:

- **[medium] Preserve webhook guard state for invalid limits:** `src/infra/map-size.ts:10`
  - body: The counted loop deletes every tracked entry when `maxSize` is `NaN`; in-repo webhook in-flight and rate-limit guards can normalize malformed `maxTrackedKeys` input to `NaN` before pruning, which can clear active request or rate-limit tracking.
  - confidence: 0.84
