## Summary

Keep open. Current main and the latest release still have the eager dedupe prune behavior, so this PR is not obsolete, but the proposed patch is not merge-ready because it leaves the saturated Discord hot path scanning the full cache and changes invalid numeric limit behavior in a shared SDK helper.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete older entries with a counted map-keys loop.

## Best Possible Solution

Replace or revise this with a focused shared-helper fix that removes per-message full scans even when the dedupe window is saturated, preserves or explicitly validates invalid numeric limit semantics, keeps plugin SDK helper contracts stable, and proves the infra, Discord replay, and webhook guard paths with targeted tests plus the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the dedupe window saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's 5-minute, 5000-entry replay guard, a server above roughly 17 unique inbound messages per second keeps the cache saturated, so every new message still scans about 5000 entries and the delay this PR targets can continue.
  - confidence: 0.91
- **[P2] Preserve invalid-limit behavior in map pruning:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the break condition never becomes true, so this loop deletes every entry. Current behavior is effectively a no-op for `NaN`; SDK guard callers can pass `NaN` after numeric normalization, causing in-flight or rate-limit guards to forget active keys.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No CI, dependency, secret, or supply-chain change was found, but the map-size helper rewrite can clear webhook guard state for malformed numeric limits.

Concerns:

- **[medium] Preserve webhook guard state on invalid limits:** `src/infra/map-size.ts:10`
  - body: The changed pruning loop deletes every tracked entry when `maxSize` is `NaN`. Because webhook guard helpers can propagate malformed numeric options into `pruneMapToMaxSize()`, this can clear in-flight or rate-limit state and weaken request protection.
  - confidence: 0.84
