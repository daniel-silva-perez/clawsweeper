## Summary

Keep open. Current main and the latest release still have the eager dedupe prune and old map-size loop, so this PR is not obsolete; the PR remains relevant but has unresolved review concerns around saturated-cache performance and invalid-limit behavior on shared SDK helpers.

## What This Changes

The PR modifies shared in-memory dedupe and bounded-map pruning helpers so dedupe misses prune only when the cache exceeds `maxSize`, and old map entries are deleted with a single forward pass.

## Best Possible Solution

Land or replace this PR with a focused shared-helper fix that removes per-message full scans in both normal and saturated cache states, preserves or explicitly validates invalid numeric limit handling, keeps Discord-owned behavior in the Discord plugin, and proves the helper plus Discord replay path with targeted tests and changed-gate validation.

## Risks / Open Questions

- The `pruneMapToMaxSize()` rewrite can change runtime `NaN` or malformed numeric limit behavior for webhook in-flight and rate-limit guards.
- Because these helpers are documented plugin SDK runtime surfaces used by bundled and third-party plugins, the cache-semantic tradeoffs need maintainer judgment before merge.
