## Summary

Keep this PR open, but do not merge it as-is. Current main still has the eager dedupe prune behavior the PR is trying to improve, but the proposed patch leaves the saturated-cache hot path with full scans and introduces an invalid-limit regression in a shared SDK collection helper used by webhook guards.

## What This Changes

The PR changes `createDedupeCache().check()` to call `prune()` only when `cache.size > maxSize` and rewrites `pruneMapToMaxSize()` to delete older entries with a counted map-key loop.

## Best Possible Solution

Revise or replace this with a focused shared-helper fix that removes per-message full scans in both normal and saturated cache states, preserves or explicitly validates invalid numeric limit behavior, and proves the infra helper, Discord replay path, and webhook guard callers with targeted tests before the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid saturated-cache full prune:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's 5-minute, 5000-entry replay guard, a live set above the cap keeps this condition true for each new unique message, so the hot path still scans the whole map and the reported delays can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit pruning semantics:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` is `NaN` and `i++ >= deleteCount` is never true, so this loop deletes every entry. Current behavior leaves `NaN` limits untouched; webhook guard options can normalize malformed input to `NaN` before calling this helper, which can clear active limiter state.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No CI, dependency, secret, or supply-chain change was found, but the shared map pruning rewrite can weaken webhook guard state for invalid numeric limits.

Concerns:

- **[medium] Preserve webhook guard state on invalid limits:** `src/infra/map-size.ts:10`
  - body: The changed counted loop deletes every tracked entry when `maxSize` is `NaN`. In-repo webhook guard helpers can pass `NaN` after numeric normalization, so this can clear in-flight or rate-limit tracking and weaken duplicate/request protection.
  - confidence: 0.84
