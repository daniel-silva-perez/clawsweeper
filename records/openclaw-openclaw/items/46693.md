## Summary

Keep this PR open. The OpenAI explicit cache-retention passthrough bug still appears valid on current main, but this branch needs repair because its provider-only resolver would lose current model/API eligibility checks and it still touches a stream-wrapper file that no longer exists on main.

## What This Changes

This PR extracts cache-retention resolution into a provider-neutral helper, switches extra-params to import it, re-exports the helper from the old Anthropic wrapper, and adds tests for explicit OpenAI/OpenAI-Codex cacheRetention passthrough.

## Best Possible Solution

Extend the existing src/agents/pi-embedded-runner/prompt-cache-retention.ts resolver to pass explicitly configured cacheRetention through for direct openai and openai-codex providers without adding an implicit default, while preserving current Anthropic, Bedrock, custom Anthropic, and Google eligibility rules and adding end-to-end regression coverage for stream options and OpenAI Responses payload mapping.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolveCacheRetention({ cacheRetention: "long" }, "openai", "openai-responses", "gpt-5") falls through to undefined on current main, and applyExtraParamsToAgent only forwards cacheRetention when that resolver returns a value.

## Solution Assessment

No. The requested behavior is correct, but this PR is not the best current-main solution because replacing the resolver with a provider-only helper loses eligibility data that current call sites already pass; the safer fix is to extend the existing resolver in place.
