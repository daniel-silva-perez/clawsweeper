## Summary

Keep open. The PR targets a real current-main OpenAI cache-retention plumbing gap, but the branch is stale against the current resolver shape and should not merge as-is because its provider-only helper would drop current model/API-aware eligibility rules for Google, custom Anthropic, and Bedrock models.

## What This Changes

This PR extracts cache-retention resolution into a provider-neutral helper, switches extra-params to import it, re-exports it from the Anthropic wrapper, and adds resolver tests for Anthropic, Bedrock, OpenAI, OpenAI-Codex, and unsupported providers.

## Best Possible Solution

Extend the existing `src/agents/pi-embedded-runner/prompt-cache-retention.ts` resolver to pass explicitly configured `cacheRetention` through for direct `openai` and `openai-codex` providers without adding an implicit default, while preserving the current Anthropic, Bedrock, custom Anthropic, and Google eligibility rules and adding end-to-end regression coverage for the stream options and OpenAI Responses payload mapping.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: on current main, `resolveCacheRetention({ cacheRetention: "long" }, "openai", "openai-responses", "gpt-5")` falls through to `undefined`, and `applyExtraParamsToAgent` only forwards cacheRetention when that resolver returns a value.

## Solution Assessment

No. The requested behavior is correct, but this PR is not the best current-main solution because replacing the resolver with a provider-only helper loses eligibility data that current call sites already pass; the safer fix is to extend the existing resolver in place.
