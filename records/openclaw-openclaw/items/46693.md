## Summary

Keep this PR open: the underlying OpenAI explicit cache-retention passthrough bug still appears valid on current main, but this branch is stale against the current resolver and its provider-only replacement would regress existing model/API eligibility gates.

## What This Changes

This PR extracts cache-retention resolution into a provider-neutral helper, updates extra-params to import it, re-exports the helper from the old Anthropic wrapper, and adds focused tests for OpenAI/OpenAI-Codex explicit cacheRetention passthrough.

## Best Possible Solution

Extend the existing `src/agents/pi-embedded-runner/prompt-cache-retention.ts` resolver to pass explicitly configured cacheRetention through for direct `openai` and `openai-codex` providers without adding an implicit default, while preserving Anthropic, Bedrock, custom Anthropic, and Google eligibility rules and adding end-to-end OpenAI Responses payload coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `resolveCacheRetention({ cacheRetention: "long" }, "openai", "openai-responses", "gpt-5")` falls through to undefined on current main, and `applyExtraParamsToAgent` only forwards cacheRetention when that resolver returns a value.

## Solution Assessment

No. The requested behavior is correct, but this PR is not the best current-main implementation because replacing the resolver with provider-only logic loses modelApi/modelId eligibility gates; extending the existing resolver in place is narrower and safer.
