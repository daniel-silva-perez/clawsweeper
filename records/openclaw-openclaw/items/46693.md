## Summary

Keep this PR open. Current main still drops explicit OpenAI/OpenAI-Codex cache retention before stream options, but the branch is stale against the current resolver shape and would regress model-aware cache eligibility if merged as-is.

## What This Changes

The PR extracts cache-retention resolution into a provider-neutral helper, re-exports it from the old Anthropic wrapper path, imports it from extra-params, and adds tests for explicit OpenAI/OpenAI-Codex `cacheRetention` passthrough.

## Best Possible Solution

Add explicit OpenAI/OpenAI-Codex passthrough to the current `src/agents/pi-embedded-runner/prompt-cache-retention.ts` resolver with no implicit defaults, preserve existing Anthropic, Bedrock, custom Anthropic, and Google rules, and add regression coverage for both config-to-stream-options and direct OpenAI Responses payload mapping.

## Reproduction Assessment

Yes. The current resolver path is enough to reproduce the issue: `resolveCacheRetention({ cacheRetention: "long" }, "openai", ...)` returns `undefined`, and `applyExtraParamsToAgent` only forwards cache retention when that resolver returns a value.

## Solution Assessment

No. The requested behavior is correct, but this branch is not the best current-main implementation because the provider-only helper loses model/API-aware eligibility; the safer fix is to extend the existing `prompt-cache-retention.ts` resolver in place.
