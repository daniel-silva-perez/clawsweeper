## Summary

Keep this PR open, but do not merge it as-is. Current main still drops explicit OpenAI/OpenAI-Codex cache retention before the OpenAI Responses transport can map it, while this branch replaces the current model/API-aware resolver with a provider-only helper that would regress Bedrock, custom Anthropic, and Google cache eligibility.

## What This Changes

The branch moves cache-retention resolution into a new provider-neutral helper, re-exports it from the Anthropic wrapper module, switches extra-params to the new helper, and adds resolver tests for Anthropic, Bedrock, OpenAI, OpenAI-Codex, and unsupported providers.

## Best Possible Solution

Extend the existing src/agents/pi-embedded-runner/prompt-cache-retention.ts resolver to allow explicitly configured cacheRetention for direct openai and openai-codex providers without adding an implicit default, while preserving the current Anthropic, Bedrock, custom Anthropic, and Google eligibility rules. Add regression coverage proving the value reaches stream options and the OpenAI Responses payload mapping.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: resolveCacheRetention({ cacheRetention: "long" }, "openai", ...) returns undefined on current main, and applyExtraParamsToAgent only forwards cacheRetention when that resolver returns a value.

## Solution Assessment

No. The requested behavior is correct, but this branch is not the best current-main solution because the new provider-only helper loses model/API-aware eligibility; the safer fix is to extend the existing prompt-cache-retention resolver in place.
