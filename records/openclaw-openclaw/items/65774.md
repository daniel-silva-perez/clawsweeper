## Summary

Current main has partial cron resilience improvements, but the requested safety behavior is still not implemented. Interrupted persisted runs are now marked failed instead of replayed, but other overdue cron slots are still selected by stored due time and replayed immediately, cron remove/delete does not cancel an active run, and gateway shutdown only stops scheduling/listeners rather than providing an emergency outbound-delivery stop.

## Best Possible Solution

Keep this open and fix it in cron/gateway core. Add explicit schedule-window-aware startup catch-up or durable deferred replay semantics, maintain a service-level registry of active cron AbortControllers, make cron remove/disable and gateway shutdown abort matching active executions, add an emergency outbound pause/stop surface for channel sends, and cover an America/Guatemala out-of-hours reconnect plus active delivery abort in regression tests. If #55160 becomes the canonical design, review and land or narrow it, but still include the active-run abort path before closing this issue.

## Risks / Open Questions

- Delayed cron work can still replay outside a user's intended contact window after restart or reconnect when nextRunAtMs is already past due.
- Deleting or disabling a cron job may not stop an already-started isolated agent run or outbound WhatsApp-style delivery.
- Gateway stop/shutdown may stop future scheduling while in-flight cron delivery continues until the process or underlying adapter exits.
- A fix needs careful run/session/task accounting so aborted, skipped, deferred, and delivered states are not misreported.
