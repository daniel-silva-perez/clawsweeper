## Summary

Keep open. Current main still starts the initial remote skill bin probe during node connect before the gateway sends `hello-ok`, while the Swift connect-response path still does not dispatch early event frames. The macOS app now advertises and handles `system.which`/`system.run`, so the remaining problem is the connect-time probe race and possibly same-host policy, not a missing command implementation.

## What This Changes

Avoid remote skill bin probe timeout spam for macOS app-backed nodes, especially same-host gateway/app nodes, while preserving legitimate remote Mac node probing.

## Best Possible Solution

Make the gateway/app node handshake robust so initial remote bin probes cannot be delivered before app-backed nodes are ready to receive `node.invoke.request`; handle any same-host loopback suppression as an explicit follow-up policy only if the race fix does not remove the observable timeout spam.

## Repair Work Prompt

Fix the remote skill bin probe race reported in https://github.com/openclaw/openclaw/issues/40527. Current main records node-role connections and starts `refreshRemoteNodeBins` before the gateway sends `hello-ok`, while Swift app-backed nodes do not dispatch early `node.invoke.request` events during `waitForConnectResponse`. Implement the smallest robust fix so initial remote bin probes cannot time out merely because they were emitted during the connect handshake. Prefer a generic readiness/defer solution over a macOS-only workaround; only add a same-host loopback skip if it is still needed after the race is fixed and keep it explicit. Do not weaken node command allowlists, exec approval checks, or remote skill bin cache invalidation. Add regression coverage for an invoke/probe arriving around node connect, plus coverage that macOS app nodes still advertise/handle `system.which` and remote skill probes still clear stale bins on real probe failure. Add a single-line changelog fix entry under the active version with `Thanks @Adam-Researchh and @tavaresgmg`.
