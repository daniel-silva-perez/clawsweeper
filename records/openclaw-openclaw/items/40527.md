## Summary

Keep open. Current main still has the timing shape reported in the follow-up: the gateway can start the automatic remote skill bin probe during node connect before `hello-ok`, while the Swift app node only starts normal event dispatch after the connect response. The older missing-handler part has improved because the macOS app now declares and handles `system.which`/`system.run`, but the connect-time race still remains.

## What This Changes

The issue asks OpenClaw to stop timing out automatic remote skill binary probes for macOS app-backed nodes, especially when the node is co-located with the gateway or still completing its connect handshake.

## Best Possible Solution

Move the initial node-side remote bin probe behind a real post-handshake readiness point, or buffer and dispatch pre-`hello-ok` node events in the Swift channel so app-backed nodes cannot drop `node.invoke.request`. Keep `system.which`/`system.run` declaration and approval semantics intact, and cover both co-located and real remote macOS app nodes with focused gateway and Swift tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current main can send the automatic `system.which` probe before `hello-ok`, while the Swift client is still waiting for the connect response and not dispatching events.

## Solution Assessment

Yes for a narrow handshake/readiness ordering fix; no for blanket `denyCommands`, a loopback-only skip, or a global remote-skills disable flag. The maintainable fix should make automatic probes wait until app-backed nodes can actually receive node events.
