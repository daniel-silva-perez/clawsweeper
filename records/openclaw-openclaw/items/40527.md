## Summary

Keep open. Current `main` still has the connect-ordering race reported in the follow-up: the gateway can fire the automatic remote skill bin probe during node connect before `hello-ok`, while the Swift app node starts normal event dispatch only after the connect response. The older missing-handler theory has improved because the macOS app now declares and handles `system.which`/`system.run`, so the remaining bug is the initial readiness/order path.

## What This Changes

The issue asks OpenClaw to stop timing out automatic remote skill binary probes for macOS app-backed nodes, especially when the node is co-located with the gateway or still completing its connect handshake.

## Best Possible Solution

Move the initial node-side remote bin probe behind a real post-handshake readiness point, or make the Swift channel safely buffer and dispatch pre-`hello-ok` node events. Keep `system.which`/`system.run` declaration, allowlist, device-token, and exec-approval semantics intact, with regression coverage for app-backed macOS nodes.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current `main` can send the automatic `system.which` probe before `hello-ok`, while the Swift client is still waiting for the connect response and not dispatching normal events.

## Solution Assessment

Yes for a narrow handshake/readiness ordering fix; no for blanket `denyCommands`, loopback-only skipping, or a global remote-skills disable flag. The maintainable fix should make automatic probes wait until the node can receive events, or make early event handling reliable.
