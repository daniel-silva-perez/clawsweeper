## Summary

Keep open. Current main still sends the automatic remote skill bin probe from the node-role connect path before the `hello-ok` response, while the Swift client starts normal event dispatch only after the connect response. The missing-handler part has improved, but the connect-time race described in the follow-up comment still matches the code.

## What This Changes

The issue asks OpenClaw to avoid remote skill binary probe timeouts for macOS app-backed nodes, especially when the macOS app node is co-located with the gateway.

## Best Possible Solution

Move the initial node-side remote bin probe behind a real post-handshake readiness point, or otherwise buffer/gate pre-hello node events so app-backed nodes cannot drop `node.invoke.request`. Preserve `system.which`/`system.run` support for remote macOS skills, and add focused TypeScript plus Swift coverage for the ordering regression.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a macOS app-backed node with a Darwin skill requiring a binary: current main sends the automatic `system.which` probe before `hello-ok`, while the Swift client is still waiting for the connect response and not dispatching events.

## Solution Assessment

No for the workaround-style solutions such as `denyCommands` or a blanket loopback skip. The narrow maintainable fix is a handshake/readiness ordering change that covers both co-located and real remote macOS app nodes.

## Repair Work Prompt

Fix the macOS app-backed node remote skill bin probe race from https://github.com/openclaw/openclaw/issues/40527. Current main registers a node and starts `refreshRemoteNodeBins` before sending `hello-ok`; Swift app clients start normal event dispatch only after the connect response, so pre-hello `node.invoke.request` frames can be dropped and the automatic `system.which` probe times out. Keep the fix narrow: defer or gate initial node post-connect side effects until after `hello-ok` is sent / the client can receive invoke events, or add an explicit readiness/buffering mechanism if that is cleaner. Preserve remote macOS skill eligibility and the existing `system.which`/`system.run` command support. Do not implement the reporter workaround as a broad `denyCommands`, do not add a global remote-skills disable flag, and do not remove macOS app command support. Add focused tests for the gateway ordering and Swift connect/listen behavior; update docs only if the protocol behavior changes. Add a single-line changelog fix entry if runtime behavior changes, crediting `@Adam-Researchh` and `@tavaresgmg` as appropriate under the active version rules.
