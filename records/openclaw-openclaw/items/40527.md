## Summary

Keep open. Current main still starts the initial remote skill binary probe during node-role connect before sending `hello-ok`, while the Swift connect path waits for the connect response without dispatching earlier event frames. The macOS app now advertises and handles `system.which`/`system.run`, so the linked missing-command-handler class is no longer the central blocker; the remaining issue is the connect-time probe race and possibly same-host policy.

## What This Changes

Avoid remote skill binary probe timeouts for macOS app-backed node connections, especially when the app node is co-located with the gateway, without blocking legitimate remote `system.run`/`system.which` nodes.

## Best Possible Solution

Make the gateway/app-node handshake robust so remote bin probes cannot be delivered before app-backed nodes are ready to receive `node.invoke.request`; preserve remote macOS skill support, and treat loopback/co-located suppression as a follow-up policy only if the handshake fix does not remove the observable timeout spam.
