## Summary

Keep open. Current main still starts the automatic remote skill bin probe from the node connect path before sending `hello-ok`, while the Swift app client only starts normal event dispatch after the connect response. The older missing-handler part has improved because the macOS node now advertises and handles `system.which`/`system.run`, but the connect-time race described in the follow-up comment still matches current source.

## What This Changes

The issue asks OpenClaw to avoid remote skill binary probe timeouts for macOS app-backed nodes, especially when the macOS app node is co-located with the gateway.

## Best Possible Solution

Move the initial node-side remote bin probe behind a real post-handshake readiness point, or buffer/gate pre-hello node events so app-backed nodes cannot drop `node.invoke.request`. Preserve existing `system.which`/`system.run` support and add focused TypeScript plus Swift coverage for the ordering regression.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a macOS app-backed node with a Darwin skill requiring a binary: current main can send the automatic `system.which` probe before `hello-ok`, while the Swift client is still waiting for the connect response and not dispatching events.

## Solution Assessment

Yes for a narrow handshake/readiness ordering fix; no for the workaround-style suggestions such as blanket `denyCommands`, a loopback-only skip, or a global remote-skills disable flag. The maintainable fix should cover both co-located and real remote macOS app nodes.

## Repair Work Prompt

Fix the macOS app-backed node remote skill bin probe race from https://github.com/openclaw/openclaw/issues/40527. Current main registers a node and starts `refreshRemoteNodeBins` before sending `hello-ok`; Swift app clients start normal event dispatch only after the connect response, so pre-hello `node.invoke.request` frames can be dropped and the automatic `system.which` probe times out. Keep the fix narrow: defer or gate initial node post-connect side effects until after `hello-ok` is sent and the client can receive invoke events, or add an explicit readiness/buffering mechanism if that is cleaner.
