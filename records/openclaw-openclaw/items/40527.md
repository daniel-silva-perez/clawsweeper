## Summary

Keep open. Current main has improved adjacent remote-skill behavior and the macOS app now declares and handles system.which/system.run, but the initial remote bin probe is still started during node connect before the gateway sends hello-ok, while the Swift connect-response wait path still ignores non-response event frames in that window.

## What This Changes

The issue requests a fix for remote skill binary probe timeouts for macOS app-backed nodes, especially when the macOS app and gateway run on the same host.

## Best Possible Solution

Fix the generic connect-time race so initial remote bin probes cannot be delivered before app-backed nodes are ready to handle node.invoke.request, then make any same-host loopback skip a separate explicit policy decision with regression coverage.

## Repair Work Prompt

Fix the remote macOS skill bin probe connect-time race tracked by openclaw/openclaw#40527. Current main calls refreshRemoteNodeBins from src/gateway/server/ws-connection/message-handler.ts while processing a role=node connect request before sendFrame sends hello-ok, and NodeRegistry.invoke can emit node.invoke.request immediately. The Swift GatewayChannel waitForConnectResponse path ignores event frames before the connect response, so macOS app-backed nodes can miss the initial system.which/system.run probe and time out. Implement a narrow generic fix: either move/defer the initial refreshRemoteNodeBins call until after hello-ok has been sent and the app listener can receive events, or buffer/dispatch early node.invoke.request frames in the Swift connect-response path. Preserve current remote skill eligibility semantics, system.which preference, 15s default timeout/log context, and node command allowlist behavior. Do not add a broad config flag, disable remote skills, or remove system.run/system.which from macOS nodes. Treat same-host/loopback probe skipping as optional maintainer policy unless it falls out as a small, well-tested generic improvement. Add regression coverage that proves a node connect with a darwin remote-skill bin does not deliver node.invoke.request before hello-ok, and/or that the Swift connection path handles early node.invoke.request frames. If adding a changelog entry, put it under the active Fixes section and credit Thanks @Adam-Researchh and @tavaresgmg.
