## Summary

Keep open. Current main still has the connect-order race described in the latest discussion: the gateway schedules the remote skill bin probe while handling the node-role connect, before the macOS app-backed node has finished the `hello-ok` handshake and started normal event dispatch. The missing-handler part has improved because the macOS node now declares and handles `system.run` and `system.which`, but that makes the remaining timing/handshake bug more plausible rather than resolved.

## What This Changes

The issue requests a fix for remote skill binary probing timing out for macOS app-backed node connections, especially when the app node is co-located with the gateway or probed too early during connect.

## Best Possible Solution

Make the gateway/app-node handshake robust so remote bin probes cannot be delivered before app-backed nodes can process `node.invoke.request`; preserve remote macOS skill eligibility and use loopback/co-located suppression only as a secondary policy if the handshake fix does not remove the timeout spam.
