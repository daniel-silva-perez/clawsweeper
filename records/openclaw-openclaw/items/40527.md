## Summary

Keep open. Current main still queues the remote skill bin probe during the node-role connect handler before sending `hello-ok`, while the Swift app-side listener starts only after connect completes. The older missing-handler part has improved because the macOS app now advertises and handles `system.which`/`system.run`, but the connect-time probe ordering described in the follow-up comment still matches the source.

## What This Changes

Avoid remote skill binary probe timeout warnings and startup/reconnect delay for macOS app-backed nodes, especially when the app node is co-located with the gateway.

## Best Possible Solution

Make node readiness explicit enough that remote bin probes cannot be delivered before app-backed nodes can dispatch `node.invoke.request`. Prefer deferring or gating the initial probe until after the connect response/listener path is ready, preserve remote macOS skill eligibility, and add focused TypeScript plus Swift regression coverage.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a macOS app-backed node with a Darwin skill requiring a binary: current source sends the automatic `system.which`/`system.run` probe before `hello-ok`, while the Swift client is still waiting for the connect response and not dispatching events.

## Solution Assessment

Unclear for the exact patch shape, but the best direction is a handshake/readiness fix rather than `denyCommands` or a blanket loopback skip. Co-located suppression may still be useful as policy, but it would not address the follow-up case where a real remote macOS app node supports manual `system.which` yet the automatic connect-time probe races.
