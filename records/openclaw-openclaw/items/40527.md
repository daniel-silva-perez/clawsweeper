## Summary

Current main still leaves the reported macOS app-backed remote skill bin probe race open. The original claim that the macOS app cannot implement `system.which`/`system.run` is outdated, but the gateway can still emit the automatic bin-probe `node.invoke.request` during the node connect path before the Swift client has entered its normal event listener, and there is still no loopback/co-located-node skip.

## Best Possible Solution

Keep this issue open. The best fix is to defer the initial `refreshRemoteNodeBins` call until after the node connect response is sent and the Swift listener can handle `node.invoke.request`, or teach the Swift connect-response wait path to buffer/dispatch early event frames. Maintainers should separately decide whether loopback/co-located macOS app nodes should skip remote bin probing when the gateway already has local shell access, then add regression coverage for connect-time probing and any co-located skip policy.
