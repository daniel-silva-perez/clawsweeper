## Summary

Keep open. Current main has partial npm failure diagnostics and a broad retry, but it still does not implement the requested structured npm error-code classification, retryable/non-retryable retry policy, actionable remediation map, Windows parity, or explicit Docker config-read failure handling.

## Best Possible Solution

Implement a scoped installer reliability change in the installer-owned surfaces: add tested npm error-code extraction/classification, retry only known transient npm/network failures with a bounded budget, print actionable remediation for terminal npm errors, make Docker gateway.controlUi.allowedOrigins config-get failures explicit without breaking first-run defaults, and mirror reasonable behavior in scripts/install.ps1 with focused Unix, Windows, and Docker setup coverage.

## Risks / Open Questions

- A broad retry after any npm failure can slow down terminal errors like EACCES, ENOSPC, E404, or EBADENGINE without giving users a clear next action.
- Changing Docker config-get handling needs to preserve quiet first-run defaults for an absent optional key while surfacing real CLI/container failures.
- Unix and Windows installers should stay aligned so retry behavior and remediation messages do not diverge by platform.
