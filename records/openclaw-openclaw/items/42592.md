## Summary

Keep open. Current main still has only partial installer diagnostics and broad npm retry behavior; it does not implement the requested structured npm error classification, retryable-only retry policy, code-specific remediation, Windows parity, or explicit Docker config-read failure handling. The existing issue discussion already identified these gaps, and a sweep through current main shows they remain.

## Best Possible Solution

Implement a scoped installer reliability change in the installer-owned surfaces: add tested npm error-code extraction/classification, retry only known transient npm/network failures with a bounded budget, print actionable remediation for terminal npm errors, make Docker gateway.controlUi.allowedOrigins config-get failures explicit without breaking first-run defaults, and mirror reasonable behavior in scripts/install.ps1 with focused Unix, Windows, and Docker setup coverage.

- Closing now would drop a concrete installer reliability request while terminal npm failures can still be retried broadly without actionable remediation.

- Unix and Windows installers should remain aligned so retry behavior and remediation messages do not diverge by platform.
