## Summary

Keep open for maintainer review. Current main still rejects root-level `paperclip` in `agent` params, and this PR is a small, behaviorally correct compatibility fix, but it overlaps the older open PR #69139 and the same-author broader issue #73653 about the generic adapter metadata contract.

## What This Changes

The PR adds an optional opaque `paperclip` field to `AgentParamsSchema` and adds validator tests for accepted Paperclip payloads plus missing-field and unknown-field rejections.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix for adapter-origin metadata in `agent` params, with focused schema and handler-level coverage, then consolidate the duplicate PRs and close the related Paperclip reports after the fix ships.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence reproduction path: send an `agent` request with required `message` and `idempotencyKey` plus a root-level `paperclip` field, and `validateAgentParams` rejects it through the closed schema before runtime handling.

## Solution Assessment

Unclear as the final product direction, but the patch is behaviorally correct for the reported Paperclip payload. The narrower `paperclip` shim is mergeable if maintainers want immediate compatibility; the safer long-term protocol shape may be a generic `adapterContext`/`adapterMeta` namespace from #73653.
