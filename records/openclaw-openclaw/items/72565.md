## Summary

Keep open for maintainer review. Current main still rejects the reported root-level `paperclip` field, so the PR addresses a real Gateway compatibility bug, but the patch is not merge-ready because it changes the Gateway protocol schema without updating generated Swift protocol models and it overlaps with other open Paperclip-metadata PRs plus an unresolved API-shape choice.

## What This Changes

The PR adds an optional opaque `paperclip` field to `AgentParamsSchema` and adds validator tests for accepted Paperclip payloads plus missing-field and unknown-field rejections.

## Best Possible Solution

Pick one additive Gateway `agent` metadata contract, update the schema, focused validator/handler tests, generated Swift protocol models, and any necessary protocol docs in the same branch, then close the duplicate Paperclip reports after the fix ships.

## Reproduction Assessment

Yes. A Gateway `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows current main into `validateAgentParams` and fails closed-schema validation before runtime dispatch.

## Solution Assessment

No as submitted. The schema change would unblock the reported payload, but a protocol-schema patch also needs generated client artifacts, and maintainers still need to choose between this Paperclip-specific field and a generic adapter metadata namespace.
