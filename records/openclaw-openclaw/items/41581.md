## Summary

Keep this issue open. Current main still forces Telegram DM partial answer previews through message transport, so the reported choppy `sendMessage`/`editMessageText` path remains the locked-in behavior rather than an already-shipped fix. The lower-level draft stream still supports DM-preferring `sendMessageDraft` when transport is left on `auto`, and the related PR attempts #62207 and #62210 were closed without merging.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin dispatcher: allow safe DM answer partial previews to use `auto`/`sendMessageDraft`, preserve final materialization and duplicate-final suppression, decide the intended transport for reasoning previews, update Telegram streaming docs, and add dispatcher regression tests for smooth DM partial preview transport without duplicate final messages.

## Risks / Open Questions

- A simple switch from forced `message` to `auto` could reintroduce duplicate-final or materialization timing problems; the fix needs dispatcher and lane-delivery regression coverage.
- Reasoning and answer preview lanes share the lane factory, so maintainers should explicitly decide whether reasoning previews should also use native draft transport in DMs.
