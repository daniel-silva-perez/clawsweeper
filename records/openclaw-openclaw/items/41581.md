## Summary

Keep open. Current main still forces Telegram DM partial answer previews through message-preview transport, while the lower draft stream still supports native `sendMessageDraft` when the dispatcher leaves DM previews on `auto`; the latest release has the same forced path and the focused fix PR is open but unmerged.

## What This Changes

Restore smooth Telegram DM partial streaming so normal direct-message answer previews can use draft-style `sendMessageDraft` behavior instead of coarse `sendMessage` plus `editMessageText` updates.

## Best Possible Solution

Restore the Telegram plugin so normal DM answer partial previews use the draft-preferring `auto` transport again, while preserving message transport where it is still needed for reasoning/tool-progress, native quote exclusions, stale preview cleanup, duplicate-final suppression, and fallback behavior. Update Telegram regression tests, streaming docs, and the changelog so the intended DM transport is explicit.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: with Telegram preview streaming enabled in a DM and no native quote or block-streaming exclusion, current main passes `previewTransport: "message"`; leaving it on `auto` would select native draft transport for DM threads.

## Solution Assessment

Yes, the narrow maintainable solution is in Telegram plugin transport selection and finalization logic, not a core streaming rewrite. #47631 is the active candidate, but it needs review/rebase and coverage before this issue can close.
