## Summary

Keep open. Current main still has the Slack routing split described by the linked report, so this PR is not obsolete; however, the PR as submitted needs cleanup and a rebase onto the current `prepare-routing.ts` helper before it is a safe merge candidate.

## What This Changes

The PR changes Slack routing and tests so top-level channel/group messages under `replyToMode="all"` use the message timestamp as the thread-scoped session key instead of the shared room session.

## Best Possible Solution

Land a rebased Slack-owned fix that makes inbound `SessionKey` and reply `MessageThreadId` agree for top-level room messages under `replyToMode="all"`, while keeping `off`/`first` continuity and current seeded-thread/runtime-binding behavior intact. If maintainers intentionally want shared room sessions for `all`, close this PR and the linked report with that product rationale.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `resolveSlackRoutingContext` for a Slack channel top-level message with `replyToMode="all"`, no `thread_ts`, and `isRoomish=true`; current main returns the base channel session while `resolveSlackThreadContext` sets `MessageThreadId` to the message timestamp.

## Solution Assessment

No, not as submitted. The intended routing change appears targeted to the reported split, but the PR branch leaves an unused `isRoomish` binding and targets the old `prepare.ts` shape; the safer current fix is to rebase the same behavior into `prepare-routing.ts` with the current seeded-room and runtime-binding cases preserved.
