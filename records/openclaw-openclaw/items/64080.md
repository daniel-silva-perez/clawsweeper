## Summary

Current main still does not implement this PR's intended Slack routing change. Ordinary top-level Slack channel messages under `replyToMode="all"` still resolve to the shared channel session, while reply delivery metadata can still carry the message timestamp as `MessageThreadId`. The paired bug report remains open and this PR is its implementation candidate, so closing it would be premature.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers accept the dual-session bug report, rebase the fix into `extensions/slack/src/monitor/message-handler/prepare-routing.ts`, preserve the newer seeded-thread and runtime-binding behavior, and update the thread-session tests that currently assert shared top-level channel sessions under `replyToMode="all"`. If maintainers intentionally want the current shared-session behavior, close both this PR and #64078 with that product rationale.
