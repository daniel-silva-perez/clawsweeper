## Summary

Keep open. The core LAN-to-loopback display fix is valid, but this PR is not safe to merge as submitted because it targets a stale status path on current main and mixes the URL fix with unrelated embedded-agent error-copy changes plus a sensitive Drop Box document-intake skill.

## What This Changes

The PR normalizes some Control UI display URL call sites from LAN bind to loopback, modifies embedded-agent transient and transport error text/payload tests, and adds a Drop Box document-intake skill with a PDF renderer.

## Best Possible Solution

Land a narrow current-main fix that centralizes display-only LAN-to-loopback normalization for browser Control UI URLs and applies it to dashboard, configure completion, status overview/status-all, daemon status, and setup finalization while preserving probe/runtime bind, custom bind, tailnet, basePath, and TLS semantics. Split the embedded-agent error-copy work and Drop Box skill into separate reviews.

## Reproduction Assessment

Yes. Code inspection on current main gives a high-confidence reproduction path: configure `gateway.bind: "lan"`, then any display caller that passes the raw bind to `resolveControlUiLinks` can render the LAN IP while dashboard already demonstrates the intended loopback-only display behavior.

## Solution Assessment

No. The LAN-to-loopback display direction is right, but this PR is not the best fix because it updates stale status code and bundles unrelated security/privacy-sensitive changes; the safer path is a narrow helper or display-mode resolver used by all current display call sites.
