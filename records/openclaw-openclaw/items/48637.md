## Summary

Keep open. The documentation gap from the linked Paperclip pairing report is real, but this PR still documents `openclaw devices approve --latest` as the recovery approval step even though current main and the latest release make that command preview-only; the persistent-key snippet also needs clearer Paperclip config-location context.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI docs and a Control UI cross-reference, including recovery commands and an `adapterConfig.devicePrivateKeyPem` note.

## Best Possible Solution

Update the PR to describe the current safe flow: trigger the pending Paperclip pairing request, inspect it with `openclaw devices list` or preview output, approve the exact `<requestId>` with any required remote auth flags, and add a Paperclip config-location sentence or link for `adapterConfig.devicePrivateKeyPem`.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The review has a high-confidence docs/code verification path: compare the PR text with current Devices CLI implementation and tests showing `--latest` exits before approval.

## Solution Assessment

No. A docs-only fix is a reasonable narrow path, but the current branch is not mergeable because it documents a preview command as an approval command and omits enough config-location context for first-time Paperclip users.
