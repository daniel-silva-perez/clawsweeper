## Summary

Keep open. Current main still lacks the Paperclip-specific pairing guidance, but this PR should not merge as-is because it documents `openclaw devices approve --latest` as an approval action even though current OpenClaw treats that form as preview-only.

## What This Changes

The PR adds Paperclip `openclaw_gateway` first-run pairing guidance to the Devices CLI docs and a cross-reference in the Control UI docs.

## Best Possible Solution

Keep the contribution open but update it to describe the current flow: trigger the pending pairing request, run `openclaw devices list` or `openclaw devices approve --latest` only to inspect it, then approve the exact `requestId` with any needed remote auth flags. Add a Paperclip config-location link or sentence for `adapterConfig.devicePrivateKeyPem`, or move the Paperclip-specific content to the external Paperclip/ClawHub docs if maintainers prefer that boundary.

## Reproduction Assessment

Not applicable as a runtime bug reproduction, but the review has a high-confidence verification path: current docs lack the Paperclip terms, and current CLI code/tests prove `--latest` is preview-only.

## Solution Assessment

No. The PR addresses a real documentation gap from #48566, but the current proposed command is stale and the persistent-key snippet lacks enough Paperclip config context for first-time users.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Document exact-request approval instead of --latest:** `docs/cli/devices.md:105-111`
  - body: Current `openclaw devices approve --latest` only previews the newest pending request and exits without calling `device.pair.approve`. This section tells Paperclip users to run that command as the recovery approval step, so following the docs will leave the device unapproved. Update the flow to list or preview the pending request, then approve the exact `<requestId>`; make the same correction in the Control UI cross-reference.
  - confidence: 0.97
- **[P3] Add Paperclip config location for the key snippet:** `docs/cli/devices.md:118-124`
  - body: The snippet shows `adapterConfig.devicePrivateKeyPem` but does not say which Paperclip config file or adapter section owns it. First-run users coming from this pairing failure will not know where to put the persistent key without a short location note or adapter-docs link.
  - confidence: 0.86
