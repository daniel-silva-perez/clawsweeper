## Summary

Keep open. Current main still does not implement this PR's central behavior: Gemini web_search only uses dedicated Gemini web-search config or GEMINI_API_KEY, still builds requests from the fixed Google Generative Language base URL, and exposes no webSearch.baseUrl config/docs surface. The PR remains a plausible Google-plugin fix, with credential/base URL precedence and SecretRef behavior needing maintainer review before landing.

## Best Possible Solution

Keep this PR open as the implementation candidate. The best path is to update the Google plugin-owned Gemini web_search provider so runtime execution and provider auto-detection both reuse models.providers.google.apiKey and models.providers.google.baseUrl only after the dedicated Gemini web_search config and documented env fallback, add webSearch.baseUrl to the manifest/config/docs, preserve raw SecretRef inputs until the shared secret resolver handles them, include baseUrl in the search cache key, and land targeted regression tests for explicit provider selection, auto-detection, missing-key behavior, SecretRef handling, and baseUrl precedence.

## Risks / Open Questions

- The PR changes credential resolution, so maintainer review should verify SecretRef handling and the intended precedence between dedicated web_search config, GEMINI_API_KEY, and models.providers.google.apiKey in both auto-detection and runtime execution.
- The PR adds custom base URL support; review should confirm the final endpoint remains guarded by withTrustedWebSearchEndpoint and that cache keys include the resolved base URL so proxy endpoints do not share stale results.
- Closing this PR now would discard a plausible implementation candidate for behavior that remains missing on current main and overlaps the open Gemini web_search baseUrl request #50213.
