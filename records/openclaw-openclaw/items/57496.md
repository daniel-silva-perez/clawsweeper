## Summary

Keep open. Current main still does not implement the PR's central behavior: Gemini web_search cannot reuse `models.providers.google.apiKey`, cannot reuse `models.providers.google.baseUrl`, and has no `webSearch.baseUrl` config surface. The PR remains a plausible implementation candidate, but the provided review thread shows credential precedence/SecretRef concerns that should be resolved before landing.

## Best Possible Solution

Keep this PR open as the implementation candidate for Gemini web_search provider-config reuse. The best path is to update the Google plugin so Gemini web_search resolves dedicated plugin config first, then `GEMINI_API_KEY`, then `models.providers.google.apiKey`, resolves base URL from dedicated `webSearch.baseUrl` then `models.providers.google.baseUrl`, preserves SecretRef and env precedence semantics in auto-detection/runtime secret resolution, and lands matching docs, metadata, and regression tests.

## Risks / Open Questions

- The PR changes credential resolution, and the provided review comments indicate the latest head still had a possible precedence mismatch where `models.providers.google.apiKey` could override `GEMINI_API_KEY` during runtime secret resolution.
- The PR adds support for a custom Gemini web_search base URL. That is reasonable for proxy use cases, but review should confirm the final implementation keeps the endpoint under the trusted web-tools guard and includes the resolved base URL in the cache key.
