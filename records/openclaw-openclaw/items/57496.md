## Summary

Keep open. Current main still lacks the requested Gemini web_search provider-config fallback, and the latest PR head is not merge-ready because its auto-detection credential lookup can reverse the documented `GEMINI_API_KEY` before provider-config precedence. The overlapping open #61972 covers baseUrl work but does not fully supersede this PR's API-key fallback goal.

## What This Changes

The PR makes Gemini web_search reuse `models.providers.google.apiKey` and `models.providers.google.baseUrl`, adds `webSearch.baseUrl` manifest/docs metadata, and adds Gemini web-search regression tests.

## Best Possible Solution

Land a rebased Google plugin change that supports Gemini web_search provider API-key and baseUrl fallback while keeping the same credential order in auto-detection, runtime secret resolution, and direct execution: dedicated web-search config first, then `GEMINI_API_KEY`, then `models.providers.google.*`. The baseUrl work should be reconciled with #61972 so one final implementation owns the manifest, docs, metadata, cache-key, and tests.

## Reproduction Assessment

Yes. A high-confidence static repro exists for the remaining PR defect: configure both `GEMINI_API_KEY` and `models.providers.google.apiKey` with no plugin-scoped key, then the PR head's configured credential lookup returns the provider key while the runtime resolver treats configured values as higher priority than env.

## Solution Assessment

No, not as currently patched. The requested behavior is the right product direction, but `getConfiguredCredentialValue` is the wrong place to expose a lower-priority provider fallback unless the runtime can preserve ordered credential sources through auto-detection and SecretRef resolution.
