## Summary

Keep open. Current main has generic empty-turn safety, but I did not find a targeted fix for a successful `anthropic-messages` tool_result follow-up that ends with `stop` and `content: []`; the current path still either skips the retry for custom Anthropic-compatible providers with nonzero usage or returns an error payload rather than preserving/synthesizing a successful user-visible reply.

## Best Possible Solution

Keep this issue open and add a targeted embedded-runner fix for `anthropic-messages` tool-result follow-ups: replay-safe successful tool calls that end in `stop` with empty assistant content should get one visible-answer continuation, while replay-unsafe side-effectful tool runs should not be blindly retried and should receive a sanitized user-visible fallback instead of a silent or raw tool-output reply. Add regression coverage for both read-like and mutating successful tool-result cases.

In current main, a custom provider using `modelApi: "anthropic-messages"` can complete a tool call successfully, then return a final assistant `stop` turn with `content: []`; the embedded runner skips the visible-answer retry unless the provider/model is already allowlisted or usage is zero, and then surfaces an error fallback.
