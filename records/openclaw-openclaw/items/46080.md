## Summary

Keep open. Current main has generic liveness guards for some empty assistant turns, but it still does not handle the reported custom `anthropic-messages` post-tool path where a successful tool result is followed by an empty nonzero-usage assistant `stop` turn.

## What This Changes

The issue asks OpenClaw to produce a visible final reply, retry safely, or synthesize a safe success acknowledgement when a custom `anthropic-messages` provider returns an empty final assistant turn after a successful `tool_result`.

## Best Possible Solution

Add a focused embedded-runner recovery for successful `anthropic-messages` post-tool empty final turns: replay-safe tool results should get one visible-answer continuation, while replay-unsafe successful tool executions should return a sanitized user-visible completion acknowledgement without retrying side effects or exposing raw tool output.

## Reproduction Assessment

Yes. A focused unit test can mock a successful tool result followed by a custom `anthropic-messages` assistant `stop` with `content: []` and nonzero usage; current main then bypasses the empty-response retry gate and reaches the incomplete-turn error path.

## Solution Assessment

No. Current main's generic empty-turn fallback is useful liveness protection, but it is not the narrowest correct solution for this issue because it treats a successful post-tool completion as an error instead of retrying safely or returning a safe success acknowledgement.

## Repair Work Prompt

Fix openclaw/openclaw#46080 in the embedded Pi runner. Current main serializes Anthropic `toolResult` messages correctly, but a successful tool result followed by a custom `anthropic-messages` assistant `stop` with `content: []` and nonzero usage still produces no successful terminal answer and falls into the incomplete-turn error path. Add a generic, provider-API-based recovery without hard-coding provider ids or custom model names: for successful post-tool empty final turns on `modelApi: "anthropic-messages"`, retry once with a visible-answer continuation only when the attempt is replay-safe; when the attempt had replay-unsafe successful side effects or mutating tools, return a sanitized success/completion acknowledgement instead of retrying side effects or exposing raw tool output. Keep existing zero-usage empty-stop, silent NO_REPLY, cron, and openai-completions behavior intact. Add focused tests in `src/agents/pi-embedded-runner/run.incomplete-turn.test.ts` for a read-like safe tool result and a mutating/replay-unsafe tool result. Update docs/changelog only if the implementation changes user-facing behavior enough to warrant it; do not change provider transport contracts or plugin APIs.
