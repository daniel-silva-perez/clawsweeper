## Summary

Keep open. Current main has generic empty-turn safety, but the reported custom `anthropic-messages` post-tool path is still not handled as a successful reply: nonzero-output empty `stop` turns are outside the retry gate, and successful tool results are not converted into a user-visible fallback.

## What This Changes

The issue requests that successful Anthropic-compatible `tool_result` runs preserve or synthesize a non-empty final assistant reply instead of ending with empty content and a no-reply outcome.

## Best Possible Solution

Add targeted embedded-runner recovery for successful `anthropic-messages` tool-result follow-ups: replay-safe empty final turns should receive one visible-answer continuation, while replay-unsafe successful tool executions should return a sanitized user-visible completion/fallback that acknowledges completion without exposing raw tool output or retrying side effects.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `runEmbeddedAttempt` to return a successful tool result followed by an `anthropic-messages` assistant `stop` with `content: []` and nonzero usage; current source shows that path produces no successful answer payload and falls through to an error fallback.

## Solution Assessment

No. The current generic empty-turn fallback is useful for liveness, but it is not the best solution for this issue because it treats a successful post-tool turn as an error and does not retry replay-safe custom Anthropic-compatible empty stops.
