## Summary

Keep open. Current main has generic liveness guards for empty assistant turns, but the specific custom `anthropic-messages` post-tool path still falls through to a user-visible incomplete-turn error instead of producing a useful final reply or a safe success acknowledgement from the successful tool result.

## What This Changes

The issue requests that custom `anthropic-messages` providers not lose the final assistant reply after a successful `tool_result` round trip.

## Best Possible Solution

Add focused embedded-runner recovery for successful custom `anthropic-messages` post-tool empty final turns: replay-safe successful tool results should get one visible-answer continuation, while replay-unsafe or mutating successful tool executions should return a sanitized completion acknowledgement without retrying side effects or dumping raw tool details.

## Reproduction Assessment

Yes. A high-confidence mocked reproduction can return a successful tool result followed by a custom `anthropic-messages` assistant `stop` with `content: []`, `payloadCount: 0`, and nonzero usage; current main skips the retry gate and reaches the incomplete-turn error path.

## Solution Assessment

No. The current implementation is useful generic liveness protection, but it is not the narrowest correct solution for this issue because it turns a successful post-tool completion into an error instead of safely recovering a visible success response.
