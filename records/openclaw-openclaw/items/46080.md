## Summary

Keep open. Current main has generic empty-turn safety, including an explicit error payload and side-effect-aware replay metadata, but it still does not implement a successful `anthropic-messages` post-tool answer path for the reported custom-provider case: nonzero-output empty `stop` turns are not retried for custom Anthropic-compatible providers, and successful tool results are not converted into a successful user-visible fallback.

## What This Changes

The issue requests that successful Anthropic-compatible `tool_result` runs preserve or synthesize a non-empty final user-visible assistant reply instead of ending with empty content and no useful response.

## Best Possible Solution

Add a targeted embedded-runner recovery for successful `anthropic-messages` tool-result follow-ups: replay-safe empty final turns should get one visible-answer continuation, while replay-unsafe successful tool executions should return a sanitized user-visible completion/fallback that acknowledges the completed action without exposing raw tool output or retrying side effects.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `runEmbeddedAttempt` to return a successful tool result followed by an `anthropic-messages` assistant `stop` with `content: []` and nonzero usage; current source shows that path produces no successful answer payload and falls back to an error.

## Solution Assessment

No. The current generic empty-turn fallback is useful but is not the best fix for this issue because it treats a successful post-tool turn as an error and does not retry replay-safe custom Anthropic-compatible empty stops.
