## Summary

Keep this open. Current main still restores last-known-good config with generic recovery logs/notices for the reported invalid config shapes, and the issue is paired with open PR #75081 using closing syntax, so it should be resolved by reviewing, landing, or superseding that PR rather than closing this issue early.

## What This Changes

The issue reports that manual edits to set `agents.defaults.execution` or a `{provider,name}` gateway password SecretRef are reverted to last-known-good config without field-specific diagnostics when `gateway.bind` is `lan`.

## Best Possible Solution

Review, land, or supersede PR #75081 with a diagnostics-only change that keeps last-known-good recovery and strict validation intact while surfacing concise rejected validation paths/messages in startup, reload, and main-agent recovery notices.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: add the unsupported `agents.defaults.execution` key or invalid `{provider,name}` password object, then current schema validation rejects it and the startup/reload recovery paths restore last-known-good with generic diagnostics.

## Solution Assessment

Yes. The narrow maintainable solution is to improve recovery diagnostics, not to relax LAN/direct execution policy or add aliases for invalid SecretRef/runtime shapes without explicit maintainer security review.
