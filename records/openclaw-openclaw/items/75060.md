## Summary

Current main still has the reported diagnostics gap: the edited config shapes are invalid under the strict current schemas, last-known-good recovery is expected, but the startup/reload recovery messages remain too generic to show the rejected validation paths. This should stay open because #75080 and #75081 both reference this issue with closing syntax and are active implementations of the narrow diagnostics-only fix.

## What This Changes

The issue asks OpenClaw to stop silently reverting unsupported direct-execution and malformed env-backed gateway-password config edits by either accepting them or reporting a clear validation failure.

## Best Possible Solution

Preserve the strict schema and last-known-good recovery model, but land one focused diagnostics change that includes concise rejected validation paths/messages in startup and reload recovery logs plus the main-agent recovery notice, with docs/tests updated as needed.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: add an unknown strict-schema key under `agents.defaults` or use a malformed `gateway.auth.password` SecretRef, then startup/reload recovery restores last-known-good and currently emits only generic recovery diagnostics.

## Solution Assessment

Yes, the diagnostics-only direction is the narrowest maintainable solution. Accepting `agents.defaults.execution` as an alias or relaxing SecretRef/LAN exposure behavior would be a separate product/security decision, not the best fix for the reported silent rollback.
