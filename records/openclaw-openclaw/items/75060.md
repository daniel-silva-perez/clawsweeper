## Summary

Current main still has a plausible rollback-with-too-generic-notice path. The reported config shapes are invalid under current strict schemas, and last-known-good recovery is intentional, but the recovery warning does not surface the rejected validation paths/messages that explain the restore.

## What This Changes

The issue asks for direct execution and env-backed gateway password config edits to be accepted or to fail loudly instead of being restored from last-known-good config when the gateway binds to LAN.

## Best Possible Solution

Keep the last-known-good recovery safety model, but make recovery diagnostics field-specific. The log and main-agent recovery notice should include concise validation issue paths/messages and point to the preserved `.clobbered.*` payload, without adding `agents.defaults.execution` as a supported alias or weakening SecretRef/LAN exposure rules.

## Reproduction Assessment

Yes. Static source inspection and existing recovery tests give a high-confidence reproduction path: an unknown strict-schema key under `agents.defaults` or a malformed `gateway.auth.password` SecretRef makes the config invalid, and startup/reload recovery restores last-known-good when available.

## Solution Assessment

Yes for a diagnostics-only repair. The narrow maintainable fix is to surface rejected validation details during recovery; accepting the reported unsupported config shapes is not the best default without explicit maintainer security/product review.
