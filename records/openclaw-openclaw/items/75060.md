## Summary

Keep open as a focused UX/diagnostic bug. Current main intentionally restores invalid direct edits from last-known-good config and does emit a generic warning, but the rejected field-level validation error is not surfaced in the recovery notice, which leaves the reporter’s central “silently reverted” experience plausible.

## What This Changes

The issue asks OpenClaw to either accept the reported `gateway.bind: "lan"` config combinations or fail with a clear user-facing error instead of restoring last-known-good config without showing the rejected fields.

## Best Possible Solution

Keep the current last-known-good recovery safety model, but make it field-specific enough that an operator immediately sees which rejected keys caused the restore. For this case, the recovery log/system event should include concise validation issue paths/messages and point to the preserved `.clobbered.*` payload, with targeted hints for the current SecretRef shape and the supported agent runtime/sandbox config surface.

## Reproduction Assessment

Yes, for the rollback class by static inspection: an unknown strict-schema key under `agents.defaults` or a malformed SecretRef object makes the startup snapshot invalid, and the gateway recovery path restores last-known-good when available. A full systemd restart was not executed in this read-only review.

## Solution Assessment

No for accepting the reported config as-is: `agents.defaults.execution` and `{provider,name}` SecretRefs are not current schema contracts. The narrower maintainable fix is to improve recovery diagnostics and tests while leaving any direct-LAN execution policy change to maintainer product/security review.
