## Summary

Keep open. Current main still has the Control UI draft as a Lit `@state()` property, and the textarea input path still writes every keystroke into that state while also requesting host updates through slash-menu handling. The related earlier PRs were closed unmerged or superseded, and open PR #62790 is the active implementation path rather than a reason to close before merge.

## What This Changes

The issue reports 500ms-1s lag on each Control UI/WebChat keystroke, suspected to come from draft typing/re-render behavior with reasoning or verbose chat content.

## Best Possible Solution

Resolve this through #62790 or an equivalent narrow UI fix that keeps normal typing out of full host re-renders, preserves explicit draft sync for abort/history/failure paths, and adds regression coverage proving draft edits do not rebuild the heavy chat thread.
