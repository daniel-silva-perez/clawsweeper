## Summary

Keep open. Current main still has the reported source-visible path: explicit short `[[tts:text]]` content is parsed and stripped, then the speech runtime returns before synthesis because the spoken text is under the generic 10-character guard. An open closing PR, #73911, already targets the narrow fix, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for `[[audio_as_voice]]` plus tagged `[[tts:text]]...[[/tts:text]]` output from the claude-cli/Telegram path to synthesize and deliver a voice note instead of being silently stripped and skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as opt-in to synthesis even below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding a regression case for `[[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A final payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current source path where hidden text is captured, visible text is stripped, the TTS runtime returns before synthesis because the spoken text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Is this the best way to solve the issue? Yes. The linked PR direction is the narrowest maintainable fix because it changes only the explicit tagged-speech short-text semantics and leaves provider configuration, directive parsing, Telegram delivery policy, and untagged short-text suppression intact.
