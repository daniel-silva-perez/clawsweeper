## Summary

Keep open. Current main still short-circuits explicit `[[tts:text]]hello[[/tts:text]]` before synthesis, and the open closing PR has not merged, so closing this issue now would be premature.

## What This Changes

The issue reports that Claude CLI-routed replies containing `[[audio_as_voice]]` plus short hidden `[[tts:text]]...[[/tts:text]]` content are parsed but never synthesized, causing Telegram voice delivery to be skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis below the generic short-text threshold, while preserving short-text suppression for untagged automatic TTS and adding regression coverage for `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Yes. A reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where hidden TTS text is captured and stripped, the runtime returns before synthesis because `hello` is under 10 characters, and Telegram then skips the empty voice-only payload.

## Solution Assessment

Yes. The narrow fix direction in #73911 is the best current path: change explicit tagged TTS semantics in the TTS runtime and add focused regression coverage, without changing provider configuration, Telegram delivery policy, or untagged auto-TTS noise suppression.
