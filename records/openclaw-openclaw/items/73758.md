## Summary

Keep open. Current main still strips hidden `[[tts:text]]` content and returns before synthesis when the explicit speech text is under 10 characters, and an open linked fix PR already targets the narrow behavior so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that short explicit TTS directives emitted through the `claude-cli` reply path are parsed but never synthesized, leaving Telegram with an empty `audioAsVoice` payload that is skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow patch that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis even below the generic short-text threshold, while preserving the short-text suppression for untagged automatic TTS and adding a regression case for the short hidden block.

## Reproduction Assessment

Yes. A final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where the hidden text is captured, visible text is stripped, the TTS runtime returns before synthesis because the speech text is under 10 characters, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes. The linked fix direction is the narrowest maintainable solution because it changes only explicit tagged TTS semantics in the shared runtime and avoids changing directive parsing, provider configuration, Telegram delivery policy, or untagged auto-TTS noise suppression.
