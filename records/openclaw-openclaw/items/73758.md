## Summary

Current main still matches the reported failure shape for short explicit tagged TTS text on a claude-cli-style final reply: `[[tts:text]]hello[[/tts:text]]` is parsed and stripped from visible text, then the TTS runtime returns before synthesis because the spoken text is under 10 characters. If `audioAsVoice` is still present, Telegram keeps the documented skip behavior for an empty voice-only payload.

## Best Possible Solution

Keep this open. Add a regression test for a final claude-cli-style payload such as `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`, then update the TTS fallback semantics so explicitly tagged hidden text is synthesized even when short, or so `audioAsVoice` is cleared and visible fallback text is preserved when synthesis is skipped. Telegram should not receive an empty voice-only payload.
