## Summary

Keep open. Current main still has a source-visible path where short explicit `[[tts:text]]` content is parsed and stripped, then returned before synthesis, and an open closing PR already targets the narrow fix so the issue should remain open until that lands or is replaced.

## What This Changes

The issue requests that explicitly tagged TTS output on the `claude-cli` final-reply path synthesize and deliver voice audio instead of being stripped and dropped before Telegram delivery.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as opt-in to synthesis even below the generic short-text threshold, while keeping untagged short auto-TTS suppression and adding a regression case for `[[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Yes. A final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where hidden text is captured, visible text is stripped, the TTS runtime returns before synthesis because the speech text is under 10 characters, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix because it changes the TTS runtime semantics for explicit tagged speech only, without changing provider configuration, directive parsing, Telegram delivery policy, or untagged auto-TTS noise suppression.
