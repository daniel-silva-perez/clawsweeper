## Summary

Keep this issue open. Current main still has the reported failure shape for a claude-cli-style final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`: the hidden TTS text is parsed and stripped from visible output, but the shared TTS hook returns before synthesis for sub-10-character text, leaving Telegram able to receive an empty `audioAsVoice` payload and skip it.

## Best Possible Solution

Keep this open and fix the shared final-reply TTS fallback semantics. Add a regression test for a final payload like `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`, then either allow explicitly tagged hidden TTS text to bypass the sub-10-character guard or clear `audioAsVoice` and preserve a visible fallback so Telegram never receives an empty voice-only reply.
