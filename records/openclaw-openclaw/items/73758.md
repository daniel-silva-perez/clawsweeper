## Summary

Keep open. Current main still strips explicit short hidden TTS text and returns before synthesis, and an open unmerged PR with closing syntax already targets the narrow fix, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `[[audio_as_voice]]` plus short `[[tts:text]]...[[/tts:text]]` output is parsed but not synthesized on the claude-cli/Telegram path, leaving no delivered voice message.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as opt-in speech text even below the generic short-text threshold, while preserving the short-text guard for untagged automatic TTS and adding regression coverage for short tagged hidden speech.

## Reproduction Assessment

Yes. A final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a current-main source path where hidden text is captured and stripped, the TTS runtime returns before synthesis because the speech text is under 10 characters, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes. The best current direction is the narrow approach in #73911: adjust the TTS runtime semantics for explicit tagged speech text and add focused coverage, without changing provider configuration, Telegram delivery policy, or untagged auto-TTS noise suppression.
