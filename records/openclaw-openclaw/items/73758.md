## Summary

Keep this issue open. Current main still has the reported failure shape for a claude-cli-style final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`: tagged TTS is recognized, but the hidden text is stripped from visible output and then skipped by the sub-10-character synthesis guard, leaving Telegram with an empty voice-only payload to skip.

## Best Possible Solution

Keep this open and fix the final-reply TTS fallback semantics. Add a regression test for a claude-cli-style final payload such as `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`, then either synthesize explicitly tagged short hidden text or clear `audioAsVoice` and preserve a visible fallback so Telegram never receives an empty voice-only payload.
