## Summary

Keep open. Current main still contains the reported short tagged TTS failure shape, and an open PR with closing syntax already targets the narrow fix, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for explicitly tagged TTS text such as `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` to synthesize and deliver voice output on the claude-cli final-reply path instead of being stripped and dropped.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis even below the generic short-text threshold, while preserving short-text suppression for untagged automatic TTS and adding regression coverage for `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Yes. A final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where the hidden text is captured and stripped, the TTS runtime returns before synthesis because it is under 10 characters, and Telegram then skips the empty voice-only payload.

## Solution Assessment

Yes, the narrow approach described by #73911 is the best current direction: change the TTS runtime semantics for explicitly tagged hidden speech text and add focused regression coverage, without changing provider configuration, Telegram delivery policy, or untagged auto-TTS noise suppression.
