## Summary

Keep open. Current main still has the direct `.bak` restore gap this PR targets, but the proposed scanner misses sensitive config fields and some actual `maskApiKey` outputs, so the patch is useful but not merge-ready as-is.

## What This Changes

The PR adds a redacted-secret backup pollution scanner and wires async/sync suspicious config recovery to refuse restoring polluted `.bak` files, with unit and integration tests.

## Best Possible Solution

Keep the `.bak` restore guard, but implement it through one shared redacted-placeholder detector aligned with current sensitive-path/schema behavior, then cover direct backup restore for gateway token, TLS key/passphrase, actual `maskApiKey` punctuation output, and the sync path.

## Reproduction Assessment

Yes. A source-level reproduction is clear: seed a recoverable `.bak` containing a redacted sensitive value and a live update-channel-only config; current main reaches `copyFile` without a pollution scan, while the PR still misses fields such as `models.providers.*.request.tls.passphrase`.

## Solution Assessment

No as-is. The guard belongs in this recovery path, but the proposed detector is too narrow; a safer fix should reuse or align with the current sensitive config path logic and actual redaction output shapes.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Detect redacted TLS key and passphrase fields:** `src/config/backup-pollution.ts:13-21`
  - body: `SECRET_FIELD_SUFFIXES` omits `key` and `passphrase`, but the config schema marks provider request TLS `key` and `passphrase` as sensitive. A polluted backup containing `request.tls.passphrase: "***"` or `request.tls.key: "***"` will still be treated as clean and copied over the live config.
  - confidence: 0.88
- **[P2] Accept the full maskApiKey character output:** `src/config/backup-pollution.ts:24`
  - body: `maskApiKey` preserves whatever first and last characters the original secret has, but this regex only allows a small punctuation set around the ellipsis. Masked placeholders containing characters such as `~` or `$` can be missed, allowing a polluted backup to restore.
  - confidence: 0.82
