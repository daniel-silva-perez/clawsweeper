## Summary

Keep open. Current main still implements SKILL.md/frontmatter-based skills, ClawHub scan visibility, and generic plugin hooks, but it does not implement the requested skill.yaml permission manifest, install-time consent review, skills audit/trust grading, signing, or attestation. The discussion also includes a recent maintainer keep-open signal and newer open RFCs, so this is active product/security design rather than conservative cleanup.

## What This Changes

Request to add a standardized skill.yaml manifest for skill permissions, with install-time review, audit/trust scoring, signing, attestation, and ClawHub integration.

## Best Possible Solution

A maintainer-designed, versioned skill capability manifest should land with parser/types, backward-compatible SKILL.md integration, direct install/update consent and audit display, ClawHub policy/display support, documented signing/attestation expectations, and a clear relationship to before_install and before_tool_call enforcement.

## Reproduction Assessment

Not applicable as a feature/security design request. The absence is high-confidence from static inspection of the current skills CLI, ClawHub installer, skill metadata types, hook docs, and public skills docs.

## Solution Assessment

Unclear; the external npm package and linked runtime-classification work are useful prior art, but directly adopting the full proposed manifest/signing/trust stack without maintainer security/API design would be premature. The safer path is an explicit RFC that phases visibility, consent, and enforcement.
