## Summary

Keep open. Current main still has SKILL.md/frontmatter-based skills, ClawHub scan visibility, and generic install/tool-call hooks, but it does not implement the requested skill.yaml permission manifest, install-time consent review, skills audit/trust grading, signing, or attestation; recent maintainer discussion and related RFC/security threads also keep this as active product/security design work.

## What This Changes

Request a standardized skill permission manifest format with declared network/filesystem/shell/credential permissions plus install-time review, audit/trust scoring, signing, and attestation support.

## Best Possible Solution

A maintainer-designed, versioned skills-first capability manifest should land with parser/types, backward-compatible SKILL.md integration, visible install/update/inspect or audit output, ClawHub policy/display support, documented signing and attestation expectations, and a clear relationship to before_install and before_tool_call enforcement.

## Reproduction Assessment

Not applicable as a feature/security design request. The absence is high-confidence from static inspection of the current skills CLI, ClawHub installer, skill metadata types, plugin hook contracts/docs, public skills docs, tests, and focused searches for manifest, audit, trust, signature, and attestation surfaces.

## Solution Assessment

Unclear whether the full proposed npm package model is the best direct implementation. A safer path is explicit maintainer RFC/design, likely starting with a small visible manifest vocabulary before deciding enforcement, trust grading, signing, and attestation policy.
