## Summary

Keep open. Current main still uses SKILL.md/frontmatter skills, ClawHub scan visibility, and generic install/tool-call hooks, but it does not implement the requested skill.yaml permission manifest, install-time consent review, skills audit/trust grading, signing, or attestation. Recent maintainer discussion also explicitly kept this open for product/security follow-up, and the related RFC/security threads remain open rather than merged or shipped.

## What This Changes

The issue requests a standardized skill.yaml permission manifest for skills, including declared filesystem/network/shell/credential needs, install-time review, audit/trust scoring, signing, attestation, and ClawHub integration.

## Best Possible Solution

A maintainer-designed, versioned skills-first capability manifest should land with parser/types, backward-compatible SKILL.md integration, direct install/update consent and audit display, ClawHub policy/display support, documented signing and attestation expectations, and a clear relationship to before_install and before_tool_call enforcement.

## Reproduction Assessment

Not applicable as a feature/security design request. The absence is high-confidence from static inspection of the current skills CLI, ClawHub installer, skill metadata types, plugin hook contracts/docs, public skills docs, tests, and focused searches for skill.yaml, audit, trust, signature, and attestation surfaces.

## Solution Assessment

Unclear whether the full proposed npm package model is the best direct implementation. The safer path is explicit maintainer RFC/design that can phase a small visible manifest vocabulary before deciding enforcement, trust grading, signing, and attestation policy.
