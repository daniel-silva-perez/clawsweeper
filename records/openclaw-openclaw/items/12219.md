## Summary

Keep open. Current main still has SKILL.md/frontmatter-based skills, ClawHub scan visibility, and generic plugin hooks, but it does not implement the requested skill.yaml permission manifest, install-time consent review, skills audit/trust grading, signing, or attestation. The issue also has a recent maintainer keep-open signal and newer related RFC/security threads, so this is active product/security design rather than conservative cleanup.

## What This Changes

Request a standard skill.yaml permission manifest for skills, with install-time permission review, skills audit/trust display, ClawHub badges, signing, and attestation.

## Best Possible Solution

A maintainer-designed, versioned skills-first capability manifest should land with parser/types, backward-compatible SKILL.md integration, direct install/update consent and audit display, ClawHub policy/display support, documented signing and attestation expectations, and a clear relationship to before_install and before_tool_call enforcement.

## Reproduction Assessment

Not applicable as a feature/security design request. The absence is high-confidence from static inspection of the current skills CLI, ClawHub installer, skill metadata types, hook docs, public skills docs, and focused searches for skill.yaml/audit/attestation surfaces.

## Solution Assessment

Unclear whether the full proposed npm package model is the best direct implementation. The safer path is explicit maintainer RFC/design that can phase a small visible manifest vocabulary before deciding enforcement, trust grading, signing, and attestation policy.
