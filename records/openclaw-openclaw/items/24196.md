## Summary

Keep open. Current main still exposes ordinary Discord sends as a single `mediaUrl`/`filename` upload path, and shared multi-media reply payloads are fanned out into separate Discord sends. Native command interaction replies are a partial exception, but they do not solve the message action or outbound reply delivery gap described here.

## Best Possible Solution

Implement native multi-file sending in the bundled Discord plugin: accept an ordered multi-attachment input shape while preserving existing single `media`/`path`/`filePath`/`filename` behavior, load and validate files through the existing media access helpers, batch eligible Discord uploads into one `files` array up to Discord limits, split overflow into additional messages, and update focused Discord tests plus docs.
