## Summary

Keep this PR open. Current main still has the ASCII-only Discord mention candidate and account-scoped directory cache, so the CJK bot-to-bot mention problem is not implemented on main. The PR is a plausible implementation candidate, but it still needs maintainer review and rework because the discussion identifies wrong-user mention risks from unscoped cross-account and nickname cache behavior. Related #43357 overlaps on Unicode parsing but is narrower and still open, so it does not make this PR shipped or safely obsolete.

## Best Possible Solution

Rebase and rework the PR against current main: land the Unicode-aware mention candidate with conservative boundaries and CJK regression tests, then handle bot-to-bot resolution with guild/account-scoped or unambiguous bot-identity cache entries, cache only after inbound policy gates pass, and coordinate the narrower Unicode parser work with #43357 instead of closing either candidate prematurely.

## Risks / Open Questions

- The proposed cross-account fallback can cause wrong-user pings when the same handle or nickname exists under different accounts or guilds unless the cache is scoped or ambiguity is rejected.
- The PR's widened mention boundary can corrupt non-mention text, links, or already-formatted Discord mentions unless boundaries are narrowed and covered by tests.
- The PR appears stale against current main's Discord reply-delivery path, which now delegates through `deliverOutboundPayloads` rather than the older chunk helper shown in the PR patch.
- The PR adds no dependency, workflow, lockfile, publish, or CI execution-surface changes, so the security concern is correctness/privacy from mention misrouting rather than supply-chain expansion.
