## Summary

Keep open. Current main still has the ASCII-only Discord mention candidate and account-scoped directory cache, so the reported CJK bot-to-bot mention behavior is not implemented. This PR is a useful implementation candidate, but it should not merge as-is because the proposed cache fallback and preflight seeding can resolve or learn handles outside the validated Discord context.

## What This Changes

The PR expands Discord outbound mention rewriting to Unicode/CJK handles, adds cross-account directory-cache lookup and preflight/startup cache seeding, and wires mention rewriting into additional Discord reply paths with focused tests.

## Best Possible Solution

Rebase and rework the PR against current main: keep the Unicode-aware parser with conservative boundaries and tests, then make bot-to-bot resolution guild/account-scoped or ambiguity-rejecting, write cache entries only after inbound policy gates pass, avoid globally caching guild nicknames, and coordinate the parser-only overlap with #43357.

## Reproduction Assessment

Yes. Seed the Discord directory cache with a CJK handle such as `架构师` and call `rewriteDiscordKnownMentions('@架构师', { accountId: 'default' })`; current main's ASCII-only parser will leave it plaintext, while the PR adds tests for that path.

## Solution Assessment

No. The Unicode parser direction is appropriate, but the current PR's cross-account and nickname cache behavior is too broad for a safe Discord mention resolver; a narrower scoped or unambiguous cache design is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject ambiguous cross-account handle matches:** `extensions/discord/src/directory-cache.ts:124-130`
  - body: The fallback now returns the first matching handle from any other account cache. In a process running bots for multiple guilds or accounts, duplicate display names can rewrite `@name` to the wrong Discord user ID, so the fallback should be guild-scoped, bot-identity-scoped, or reject ambiguous matches.
  - confidence: 0.94
- **[P1] Defer directory cache writes until after policy checks:** `extensions/discord/src/monitor/message-handler.preflight.ts:386-395`
  - body: Preflight writes author and mention handles before own-message, guild/channel allowlist, member allowlist, and bot-policy drops run. A blocked or ignored message can still poison the shared handle cache and affect later outbound mention rewriting in allowed contexts.
  - confidence: 0.91
- **[P1] Keep guild nicknames out of the account-global cache:** `extensions/discord/src/monitor/message-handler.preflight.ts:390-394`
  - body: `member.nick` is guild-scoped, but this writes it into the account-wide handle map consumed without guild context. Two guilds can reuse the same nickname for different users, causing later replies in another guild to ping the wrong user.
  - confidence: 0.92
- **[P2] Narrow mention boundaries to avoid URLs and formatted mentions:** `extensions/discord/src/mentions.ts:9`
  - body: The new `[^\p{L}\p{N}]` prefix treats `/` and `<` as valid separators, so cached handles can be rewritten inside URL paths such as `https://x.test/@alice` or inside already formatted Discord mentions. Preserve a tighter boundary or explicitly exclude those contexts.
  - confidence: 0.88
- **[P2] Cache display names during startup seeding:** `extensions/discord/src/monitor/provider.ts:1013-1018`
  - body: Startup seeds only `botUserName`, which is computed as `username ?? globalName`; when a bot has an ASCII username and a CJK display name, `@displayName` still cannot resolve until that bot is later seen in preflight traffic. Seed both typed handles if first-contact display-name mentions are part of the fix.
  - confidence: 0.82
