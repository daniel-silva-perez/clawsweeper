## Summary

Keep open. Current main still omits the `file` alias from the read/write/edit display resolver and read-start warning guard, while current execution validation has moved to canonical `path` only. An open unmerged PR with closing syntax is already tracking the implementation path, so this issue should stay open until maintainers settle the alias compatibility contract.

## What This Changes

Show verbose/block-streaming file-path details for read/write/edit tool summaries when args use the `file` parameter alias, and align the read-start diagnostic with that behavior.

## Best Possible Solution

Settle the path-parameter contract in one place: either land an updated display/diagnostic alias fix with regression coverage for `file` and `filePath`, or intentionally standardize on canonical `path` by aligning diagnostics, tests, and user-facing expectations.
