## Summary

Keep open. Current main still omits the `file` alias from the read/write/edit display path resolver and the read-start warning guard, while an open unmerged PR with closing syntax is already tracking the implementation path. Current runtime validation has also moved toward canonical `path` only, so maintainers should resolve the display/diagnostic compatibility contract rather than closing this as cleanup.

## What This Changes

Show read/write/edit verbose tool summaries and read-start diagnostics with file paths when tool arguments use the `file` alias instead of canonical `path`.

## Best Possible Solution

Keep this issue open until maintainers either review and land an updated #55121-style fix, or deliberately standardize on canonical `path` by updating diagnostics, tests, and user-facing expectations so aliases are rejected and displayed consistently.
