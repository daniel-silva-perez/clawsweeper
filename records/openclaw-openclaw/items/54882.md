## Summary

Keep open. Current main still does not implement the requested `file` alias in the tool-display path resolver or read-start warning guard, and the provided GitHub context shows open, unmerged implementation candidates with closing syntax for this issue.

## Best Possible Solution

Keep this issue open and review or close the linked implementation PRs on their merits. Maintainers should align `resolvePathArg()`, `handleToolExecutionStart()`, tests, and the shared display manifest around one explicit path-alias policy: either support `file`/`filePath`/`file_path` consistently in display and diagnostics, or document and enforce canonical `path` consistently so models no longer appear to execute one alias while display/diagnostics treat it as missing.
