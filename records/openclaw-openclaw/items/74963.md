## Summary

Keep open. Current main still derives the bundled runtime-deps stage key from a lexical `path.resolve(packageRoot)`, while this issue reports current-release packaged workers reaching different same-version runtime-deps roots and failing bundled channel imports. The provided GitHub context also shows open PR #75048 with explicit closing syntax for this issue, so this issue should stay open until that PR is reviewed and merged or replaced.

## What This Changes

The issue asks OpenClaw to make bundled plugin runtime-deps staging stable for multi-instance packaged installs so bundled channel dist files are found consistently after upgrading to 2026.4.27.

## Best Possible Solution

Review and land #75048, or an equivalent narrow replacement, so equivalent package roots canonicalize to one bundled runtime-deps stage key with focused regression coverage and without broad plugin SDK or channel-loader contract changes.

## Reproduction Assessment

Unclear for a live end-to-end reproduction: the issue has concrete 2026.4.27 Windows/PM2 logs and source tracing shows a plausible lexical-path hash split, but I did not execute that environment. The linked PR describes a narrower symlink/junction-style unit reproduction that should serve as regression proof.

## Solution Assessment

Yes, the linked PR direction appears to be the narrowest maintainable fix: canonicalize the package root before hashing the external runtime-deps stage key and add focused regression coverage. A docs-only workaround or broad channel-loader rewrite would not directly solve the shared staging-key divergence.
