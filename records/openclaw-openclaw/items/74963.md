## Summary

Keep open. Current main still derives packaged bundled runtime-deps stage keys from the lexical `path.resolve(packageRoot)`, while the loader and bundled-channel entry paths can reach the same physical install through different root forms; an open maintainer-labeled PR (#75048) already references this issue with closing syntax and should be reviewed before this issue is closed.

## What This Changes

Ensure multi-instance packaged Gateway workers resolve bundled plugin runtime deps to a stable usable per-version stage root so bundled channel dist imports do not fail with ENOENT.

## Best Possible Solution

Review and land #75048, or an equivalent narrow fix, so equivalent packaged OpenClaw roots canonicalize to one runtime-deps stage key and bundled channel/plugin imports use the staged dist files consistently; include focused regression coverage and a changelog entry if the fix ships.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully from this review alone: the report has concrete logs and the source shows a credible path-derived mismatch, but I did not run Windows/PM2 or the proposed symlink packageRoot regression test against current main.

## Solution Assessment

Is this the best way to solve the issue? There is no patch in this issue, but the linked PR's direction of canonicalizing `packageRoot` before hashing is the narrowest maintainable fix suggested by the source evidence; duplicate repair work should wait on that PR review.
