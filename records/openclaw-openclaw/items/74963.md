## Summary

Keep open. Current main still derives the packaged bundled runtime-deps stage key from the lexical `path.resolve(packageRoot)`, and the provided GitHub context shows an open PR, #75048, with closing syntax for this exact hash-divergence bug, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that OpenClaw 2026.4.27 multi-instance packaged installs can derive different `plugin-runtime-deps/openclaw-<version>-<hash>` directories and then fail to import bundled channel `dist/*.js` chunks.

## Best Possible Solution

Review and land #75048, or an equivalent narrow replacement, so packaged bundled runtime-deps hash the canonical filesystem path for the package root and include regression coverage for equivalent package-root views. After that lands, reassess whether disabled or unconfigured bundled channel entrypoints still create noisy status/health imports.

## Reproduction Assessment

Yes. A focused reproduction is to expose the same packaged OpenClaw root through real and symlink/junction-style lexical paths and compare the resolved external runtime-deps install roots; current main still has the lexical hash seam that can split those roots.

## Solution Assessment

Yes. Canonicalizing `packageRoot` before hashing the stage key appears to be the narrowest maintainable fix because it converges the runtime-deps directory without broadening hot bundled channel loading or changing plugin activation policy.
