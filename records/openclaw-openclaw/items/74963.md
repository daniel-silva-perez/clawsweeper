## Summary

Keep open. Current `main` still hashes the packaged bundled runtime-deps stage key from the lexical `path.resolve(packageRoot)`, while the issue discussion and linked #75048 show the failure comes from equivalent install roots reaching the staging seam under different path spellings. The open linked PR uses closing syntax for this issue and targets the exact hash divergence, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

This issue reports that OpenClaw 2026.4.27 multi-instance deployments can derive different `plugin-runtime-deps` stage hashes for the same install and then fail bundled channel imports with ENOENT on staged `dist/*.js` chunks.

## Best Possible Solution

Review and land #75048, or an equivalent narrow replacement, so the package root is canonicalized before deriving the runtime-deps stage hash and regression coverage proves equivalent package-root views converge to one install root. If disabled-channel discovery noise still appears afterward, track that as a narrower follow-up rather than closing this before the hash fix lands.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to expose the same packaged OpenClaw root through two equivalent filesystem paths, such as a real path and symlink or junction path, then compare the resolved external runtime-deps install roots; current `main` still supports divergent hashes through `path.resolve(packageRoot)`.

## Solution Assessment

Yes. Canonicalizing `packageRoot` at the hash seam is the narrowest maintainable fix because it converges all callers on one stage key without broad channel-loader or runtime mirror rewrites.
