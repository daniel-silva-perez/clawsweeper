## Summary

Keep open. Current main still derives the packaged bundled runtime-deps stage key from the lexical `path.resolve(packageRoot)`, and the linked implementation PR that targets this exact hash divergence remains open, so the issue should not be closed before that fix or an equivalent replacement lands.

## What This Changes

Ensure multi-instance packaged Gateway runs resolve one working bundled plugin-runtime-deps stage root per OpenClaw version/package root so bundled channel dist imports do not fail with ENOENT.

## Best Possible Solution

Review and land #75048, or an equivalent narrow replacement, so the stage-key hash uses the canonical filesystem path for the package root and includes focused regression coverage for equivalent package-root views.

## Reproduction Assessment

Yes for a focused reproduction path: expose the same packaged OpenClaw root through real and symlink/junction-style paths, then compare the resolved external runtime-deps install roots. A live Windows/PM2 reproduction was not run here, but current source still supports the hash-split mechanism described in the report.

## Solution Assessment

Yes, the linked PR direction appears to be the narrowest maintainable solution: canonicalize the package root at the hash seam rather than rewriting bundled channel loading or adding a docs-only workaround.
