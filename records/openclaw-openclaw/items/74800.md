## Summary

Keep open. Current main still has the undefined plugin-command result crash path, and an open implementation PR already explicitly targets this issue, so cleanup should wait for that PR to merge or be closed.

## What This Changes

The issue requests that Telegram/plugin command handling tolerate plugin command handlers that return undefined instead of crashing and surfacing the generic Telegram error.

## Best Possible Solution

Review and land #74989, or an equivalent narrow fix, that normalizes nullish/non-object plugin command returns at `executePluginCommand`, keeps `continueAgent` semantics intact, defensively protects Telegram/native consumers, and adds focused regression tests.

## Reproduction Assessment

Yes. A registered plugin command whose handler completes without returning a payload follows the current source path from `executePluginCommand()` to unguarded `result.continueAgent` or Telegram native `result.text` reads, which is enough to reproduce the TypeError without the reporter's local plugin.

## Solution Assessment

Yes. Normalizing at the shared plugin command executor boundary is the narrowest maintainable fix because both text command handling and Telegram native command handling consume that result; caller-side guards are useful defense-in-depth.
