## Summary

Keep open: current main still has the undefined plugin-command result crash path, and an open implementation PR already explicitly targets this issue, so cleanup should wait for that PR to merge or be closed.

## What This Changes

The issue requests defensive handling for Telegram/plugin command handlers that return undefined or no reply payload so they do not crash command dispatch with a generic Telegram error.

## Best Possible Solution

Review and land #74989, or an equivalent narrow fix, that normalizes nullish/non-object plugin command returns at `executePluginCommand`, keeps `continueAgent` semantics intact, defensively protects generic and Telegram native consumers, and adds focused regression tests for undefined command results.

## Reproduction Assessment

Yes. A registered plugin command whose handler completes without returning a payload follows the current source path from `executePluginCommand()` to unguarded `result.continueAgent` or Telegram native `result.text` reads, which is enough to reproduce the TypeError without the reporter's local plugin.

## Solution Assessment

Yes. Normalizing at the plugin command executor boundary is the narrowest maintainable fix because both text command handling and Telegram native command handling consume the same result; caller-side guards are appropriate defense-in-depth.
