## Summary

Keep this PR open because the reported gateway display-name bug still appears valid on current main, but the PR patch is stale against the current row projection path. Current main builds session rows through `buildGatewaySessionRow`, and session-change snapshots also load rows through that helper, so a guard inside only the old `listSessionsFromStore` fallback chain would miss part of the user-visible surface.

## What This Changes

The PR changes `src/gateway/session-utils.ts` so `listSessionsFromStore` treats configured per-agent and global main session keys as stable main sessions, preserving explicit `entry.displayName` and otherwise returning `Main Session` instead of transient origin labels.

## Best Possible Solution

Put the stabilization in `buildGatewaySessionRow` or a shared helper used by both `listSessionsFromStore` and `loadGatewaySessionRow`, preserve explicit `entry.displayName`, recognize `resolveAgentMainSessionKey` and `resolveMainSessionKey`, and add focused regression tests for default-agent main, configured main keys, global main, and stale heartbeat origin labels.

## Reproduction Assessment

Yes. A deterministic reproduction is to project a main session entry with no explicit label/displayName and `origin.label: "heartbeat"` through `buildGatewaySessionRow`; the resulting row can still carry `displayName: "heartbeat"`, and the UI prefers that row value before its `Main Session` fallback.

## Solution Assessment

No. The intended fix is correct, but the PR’s current shape patches an older list-only display-name path; on current main the narrow maintainable solution is a shared row-projection guard plus regression coverage for both list and event snapshot callers.
