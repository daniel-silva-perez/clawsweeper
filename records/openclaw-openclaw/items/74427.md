## Summary

Keep this PR open for normal maintainer review. Current `main` still hardcodes the OpenAI forward-compat default model base URLs, and the PR is a focused implementation rather than obsolete work; I found no discrete blocking correctness or supply-chain finding in the provided diff.

## What This Changes

The PR adds an OpenAI default base URL helper that reads `OPENAI_BASE_URL`, replaces hardcoded dynamic GPT fallback base URLs with that helper, and adds unit coverage for the helper behavior.

## Best Possible Solution

Land a focused OpenAI-provider fix that makes dynamic GPT defaults honor `OPENAI_BASE_URL` only as the no-config fallback, keeps explicit `models.providers.openai.baseUrl` precedence unchanged, and adds resolver-level coverage for that precedence before merge.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current `main`, resolve an OpenAI forward-compat model such as `openai/gpt-5.4` with only `OPENAI_BASE_URL` set and no `models.providers.openai.baseUrl`; the discovered model still carries the hardcoded `api.openai.com` base URL that the resolver uses as fallback.

## Solution Assessment

Yes, the proposed direction is a narrow owner-local fix: it changes OpenAI dynamic fallback metadata instead of core resolver precedence, preserving explicit config precedence while aligning with the OpenAI SDK’s env fallback. A safer merge version would add one resolver-level test in addition to the helper test. ([raw.githubusercontent.com](https://raw.githubusercontent.com/openai/openai-node/master/src/client.ts))
