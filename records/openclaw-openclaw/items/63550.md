## Summary

Keep open. Current main still implements memory-lancedb as one unpartitioned LanceDB table, and the tool, hook, duplicate-check, delete, and CLI paths do not carry a trusted user or agent partition into storage/search. The report is recent, concrete, and aligned with the multi-agent docs' promise of isolated agent data, but the exact partition source and migration behavior need maintainer/security product judgment before an automated repair lane should change it.

## What This Changes

The issue requests partition-aware memory-lancedb storage and search so multi-user deployments do not mix memory rows across users.

## Best Possible Solution

Add structural scoping inside the bundled memory-lancedb plugin, with any generic SDK/context seam kept additive. The end state should write a trusted partition key or use trusted per-partition storage, apply that scope consistently before recall results are ranked or used, cover store/recall/forget/list/query/stats/auto-capture/auto-recall/duplicate detection, and document migration plus default behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: configure memory-lancedb with one dbPath, write memories from two agent/user partitions, then recall from either partition; current db.search and db.store have no partition parameter or filter, so rows can mix. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear until maintainers choose the product boundary. Metadata filtering with a trusted partition key looks like the narrowest structural fix for the reported leakage, but dynamic dbPath variables alone would not cover existing rows, duplicate checks, forget/list/query behavior, or the CLI/global operator story.

## Risks / Open Questions

- Maintainers need to choose the trusted partition identity: agentId, channel sender/user id, an explicit gateway-level user partition, or a combination. A tool-argument or prompt-supplied partition would not provide structural isolation.
- Existing LanceDB rows need a migration/backfill/default-partition plan so old rows neither keep leaking across users nor silently disappear after upgrade.
- The memory-lancedb ltm CLI currently has no ambient user partition; list/search/query/stats will need an explicit scoped mode or a clearly documented operator/global mode.
