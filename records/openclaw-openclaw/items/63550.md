## Summary

Keep open. Current main still stores and searches memory-lancedb rows through one unpartitioned LanceDB table, while the issue and follow-up comment describe a concrete privacy/isolation problem for multi-user deployments. The existing plugin/tool context appears to have enough trusted agent/session context to start an owner-module fix, but the bundled plugin does not use it yet.

## Best Possible Solution

Keep this issue open and implement the fix in the bundled `memory-lancedb` plugin, with any needed generic SDK/context seam kept additive. A good fix would add a trusted partition key to the LanceDB schema, apply it to memory_store, memory_recall, memory_forget, auto-recall, auto-capture, duplicate checks, and CLI operations, filter before semantic result use, cover at least two agent/user partitions in tests, document migration/default behavior, and clarify whether the first supported partition source is agentId or a separate gateway-level user partition.

## Risks / Open Questions

- Partition semantics need an explicit maintainer choice: agentId, channel sender/user id, configured partition id, or a combination. Prompt/tool-argument supplied scope would not provide structural isolation.
- Existing LanceDB tables need a migration/backfill/default-partition plan so old rows do not keep leaking across users or disappear unexpectedly after upgrade.
- The `openclaw ltm` CLI path currently has no obvious ambient user partition; it will need explicit scoped options or an operator/global mode once memory rows are partitioned.
