## Summary

Keep open. Current main still does not implement multi-user or partition isolation for the bundled `memory-lancedb` plugin: rows have no trusted partition field, storage/search use one `memories` table, tool and hook paths do not scope operations by agent/user/session partition, and the public config/docs expose only a static `dbPath`.

## Best Possible Solution

Keep this issue open and implement the fix in `extensions/memory-lancedb` with any required SDK/context seam kept additive. A solid fix would add a trusted partition key or partition-aware DB resolution, apply it consistently to `memory_store`, `memory_recall`, duplicate checks, `memory_forget`, auto-recall, auto-capture, and CLI operations, cover at least two isolated partitions in tests, and document migration/default behavior.

## Risks / Open Questions

- Partition semantics need maintainer product choice: agent ID, sender/user ID, configured gateway partition, or a composite trusted scope. Prompt/tool-argument supplied scope would not provide structural isolation.
- Existing LanceDB tables need an upgrade/backfill/default-partition plan so old rows do not keep leaking across users or disappear unexpectedly.
- The `openclaw ltm` CLI has no obvious ambient requester partition today, so it likely needs explicit scoped options or an operator/global mode.
