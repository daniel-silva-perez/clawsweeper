## Summary

Keep open. Current main still treats `mcp.servers` as a gateway-wide registry and only applies per-agent tool policy after configured MCP servers have been merged and materialized, so the reported process-count and least-privilege problem remains valid.

## What This Changes

Request an optional per-agent scope on configured MCP servers so only matching agents start or connect to those MCP servers.

## Best Possible Solution

Add a documented, backward-compatible MCP server scope resolved against the active agent before stdio spawn, HTTP/SSE connection, tool cataloging, CLI backend injection, and cache fingerprinting. Omitted scope should preserve today’s all-agents behavior, with schema/help/docs and focused tests covering embedded Pi, CLI bundle MCP, config reload, and tool-policy interactions.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection: current main builds a full effective MCP server map and materializes it before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config contract. The narrow maintainable direction is pre-spawn filtering of the effective MCP server map; `tools.deny` or post-materialization filtering is not enough for the reported process and least-privilege boundary.
