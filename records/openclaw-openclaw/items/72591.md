## Summary

Keep open. Current main still treats `mcp.servers` as a global registry: configured servers are merged without agent identity, embedded Pi connects and catalogs every loaded server, CLI bundle overlays receive the full merged map, and per-agent tool policy is applied only after MCP tools have been materialized. The existing MCP tool filtering work is adjacent but does not solve the requested pre-spawn process and least-privilege boundary.

## What This Changes

The issue requests optional per-agent scoping for configured MCP servers so only matching agents start or connect to those servers.

## Best Possible Solution

Add a documented, backward-compatible MCP server scope that is resolved against the active agent before any stdio process is spawned or remote MCP connection is opened. Omitted scope should preserve current all-agents behavior, and the filtered effective server map should feed embedded Pi runtime creation, CLI bundle MCP generation, cache fingerprints, reload cleanup, schema/help, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Current main builds a full effective `mcpServers` map, connects/lists tools for every configured server, and only then applies per-agent tool policy.

## Solution Assessment

Unclear until maintainers approve the public config shape. The narrow maintainable implementation direction is pre-spawn filtering of the effective MCP server map; relying on `tools.deny` or filtering after tool materialization is not sufficient for the reported process and least-privilege problem.
