## Summary

Keep open. Current main still has a shared MCP server registry that is merged and materialized without an active agent scope, and per-agent tool policy runs after MCP server startup/cataloging, so the reported memory and least-privilege gap remains valid.

## What This Changes

Request adding an optional per-MCP-server agent scope so only matching agents spawn or connect that server while preserving the current all-agents default when omitted.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the normalized active agent before stdio spawn, HTTP/SSE connection, CLI backend MCP injection, cataloging, and fingerprint/cache decisions. The change should update config types, schema/help, docs, and focused tests for embedded Pi, CLI bundle MCP, config reload/cache behavior, and interaction with existing tool policy.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. A config with multiple agents and any `mcp.servers` entry feeds the same merged MCP server map into each normal MCP-enabled agent run before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config shape. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-materialization filtering is not sufficient for the reported memory and least-privilege boundary.
