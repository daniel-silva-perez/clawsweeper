## Summary

Keep open. Current main still builds each bundle/user MCP runtime from the full merged `mcp.servers` map and connects to every configured server before per-agent tool policy is applied, so the reported process/RAM problem remains. The related plugin-scoping reports are adjacent, but they do not cover MCP server process scoping, and the personal-data guardrail comment makes this worth explicit maintainer review.

## What This Changes

Request an optional per-server agent scope for `mcp.servers.*` so only matching agents spawn or connect to selected MCP servers while omitted scope preserves the current all-agent default.

## Best Possible Solution

Add an explicit, documented, backward-compatible MCP server scope that filters servers before any stdio process is spawned or HTTP/SSE connection is opened. Omitted scope should mean all agents, matching should reuse the existing glob-pattern helper, and the same filtered effective config should feed embedded Pi, CLI bundle MCP generation, cache fingerprinting, reload cleanup, and docs/tests.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. The runtime builds a full effective `mcpServers` map, iterates every entry during catalog creation, and applies per-agent tool policy only after MCP tools are materialized.

## Solution Assessment

Yes in direction, with one caveat: filtering before `resolveMcpTransport` and downstream CLI config generation is the narrowest maintainable way to solve both RAM/process use and least-privilege concerns. The exact public config syntax should be confirmed before implementation.
