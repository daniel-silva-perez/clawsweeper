## Summary

Current main still lacks per-agent MCP server scoping and still materializes the effective MCP server registry before per-agent tool policy is applied, so the reported process-count and least-privilege problem remains valid. This should stay open for maintainer product/security review because the requested field would become a public config contract and a pre-spawn/pre-connect security boundary.

## What This Changes

Request an optional per-MCP-server agent scope so configured MCP servers are spawned or connected only for matching agents while preserving the current all-agents default when no scope is set.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the active agent before any server is spawned, connected, injected into downstream CLI config, cataloged, or exposed to tool policy. Preserve omitted scope as all-agents behavior, update schema/help/docs, and add focused coverage for embedded Pi, CLI bundle MCP, config reload/fingerprints, and interaction with existing `tools.allow` / `tools.deny`.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Current main still builds a full MCP server map and materializes it before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config contract. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-materialization filtering is not enough for the reported process count or least-privilege boundary.
