## Summary

Keep this issue open. Current main carries Feishu accountId through monitor dispatch, route resolution, inbound context, and outbound reply selection, but the actual group session key still omits account/app scope. Feishu group peer ids are still chat/topic/sender scoped, and tests/docs continue to encode accountless group session keys, so the reported same-group multi-bot collision remains plausible when multiple Feishu accounts route to the same agent/session.

## Best Possible Solution

Keep the issue open. Implement account-aware Feishu group session isolation in the Feishu plugin or through an additive routing/session-key seam, likely appending a normalized accountId/appId for non-default Feishu group routes while preserving legacy default-account continuity. Add a regression test with two Feishu accounts in the same group routed to the same agent, and update Feishu/group routing docs to describe the compatibility behavior.

## Risks / Open Questions

- A global change to generic group session key shape could break existing transcript continuity for other group channels, so Feishu likely needs a compatibility path, explicit account-scoped mode, or migration-aware aliasing.
- Account-scoped route bindings can avoid some collisions when different Feishu accounts route to different agents, but they do not isolate multiple accounts in the same group when they resolve to the same agent/session.
- Feishu ACP/thread bindings and group history keys may need the same account/app scoping decision so reply routing and history do not drift from session storage.
