## Summary

Current main still does not implement the requested Feishu multi-bot group session isolation. Feishu inbound now carries the runtime account into routing and reply dispatch, but group session keys remain shaped only by agent, channel, peer kind, and group peer id. Two Feishu accounts in the same group routed to the same agent can still share `agent:<agent>:feishu:group:<chat_id>` transcript state.

## Best Possible Solution

Keep this issue open. Implement account-aware Feishu group session isolation, preferably as a Feishu-specific or generic opt-in account-scoped group mode using normalized `accountId` rather than raw app credentials. Preserve legacy session continuity where feasible, document the config/migration behavior, and add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct reply account identity.

## Risks / Open Questions

- A global change to all group/channel session key shapes could break existing transcript continuity across channels; the safer implementation likely needs Feishu-specific compatibility handling or an explicit account-scoped group-session mode.
- Account-scoped bindings can route different Feishu accounts to different agents, which avoids one class of persona mix-up, but they do not solve the reported same-agent/same-group transcript collision.
