## Summary

Current main still does not isolate Feishu group sessions by Feishu account/app. The Feishu handler passes accountId through routing and reply dispatch, but the group peer/session key is still derived from chat/group scope only, so multiple Feishu apps in the same group can collide when routed to the same agent.

## Best Possible Solution

Keep this issue open. Implement account-aware Feishu group session isolation in the Feishu plugin or an additive channel-routing seam, likely with a normalized accountId/appId suffix for non-default-account Feishu group routes while preserving legacy default-account continuity. Add a regression test with two Feishu accounts in the same group routed to the same agent, and document the compatibility behavior in the Feishu/group routing docs.

## Risks / Open Questions

- Changing the generic group session key globally would likely break existing transcript continuity for all group channels; a Feishu-specific compatibility path or explicit account-scoped group mode is safer.
- Account-scoped route bindings can send different Feishu accounts to different agents, but they do not solve same-agent, same-group transcript collisions.
- Existing cross-account broadcast dedup avoids duplicate broadcast dispatch, but it is separate from single-agent group session isolation.
