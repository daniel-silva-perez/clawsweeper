## Summary

Keep open. Current main still builds Feishu group session keys without an account/app component: Feishu passes accountId into routing and replies, but group peer IDs remain accountless and the single-agent dispatch uses route.sessionKey for transcript/session storage. Two Feishu apps in the same group routed to the same agent can still share one session bucket.

## Best Possible Solution

Keep this issue open. Implement account-aware Feishu group session isolation in the Feishu plugin, likely by adding a normalized accountId/appId-derived suffix for non-default account group routes while preserving legacy default-account session continuity. Add a regression test with two Feishu accounts in the same group routed to the same agent, asserting distinct session keys and correct outbound account identity, then document the compatibility behavior in the Feishu/channel-routing docs.

## Risks / Open Questions

- Changing the generic group session key globally would break existing transcript continuity; a Feishu-specific compatibility path or explicit account-scoped group mode is safer.
- Account-scoped bindings can route different Feishu accounts to different agents, but they do not solve the same-agent, same-group transcript collision described here.
- Feishu broadcast cross-account dedupe prevents duplicate broadcast dispatch, but it is separate from single-agent group session isolation.
