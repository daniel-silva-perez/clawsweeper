## Summary

Current main still carries accountId through Feishu routing and reply dispatch, but the group session key remains account/app agnostic. Multiple Feishu apps in the same group that resolve to the same agent can still converge on the same `agent:<agentId>:feishu:group:<chatId>` session bucket.

## Best Possible Solution

Keep this open and implement account-aware Feishu group session isolation in the Feishu plugin or through an additive routing/session-key option. A conservative fix would append a normalized Feishu accountId/appId for non-default multi-account group routes while preserving legacy default-account continuity, then add a regression test with two Feishu accounts in the same group routed to the same agent and update Feishu/channel-routing docs.

## Risks / Open Questions

- Changing the generic group session key shape would break transcript continuity for existing group channels; a Feishu-specific compatibility path or additive opt-in routing/session-key option looks safer.
- Account-scoped bindings can route different Feishu apps to different agents, but they do not solve the reported same-agent, same-group collision.
- Any migration should preserve existing default-account group sessions or provide doctor/fix guidance so operators do not unexpectedly lose context.
