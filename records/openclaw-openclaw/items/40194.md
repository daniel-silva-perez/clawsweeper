## Summary

Current main still does not isolate Feishu group sessions by Feishu account/app. The account id is carried through monitor dispatch, routing, and outbound reply selection, but the actual group session key remains based on agent + channel + group peer id, so two Feishu apps in the same group can still collide when routed to the same agent/session.

## Best Possible Solution

Keep this issue open. Implement account-aware Feishu group session isolation in the Feishu plugin or an additive routing/session-key seam, likely appending a normalized accountId/appId for non-default Feishu group routes while preserving legacy default-account continuity. Add a regression test with two Feishu accounts in the same group routed to the same agent, and document the compatibility behavior in the Feishu/group routing docs.

## Risks / Open Questions

- A global change to generic group session keys would break existing transcript continuity across group channels; Feishu likely needs a compatibility path, explicit account-scoped group mode, or migration-aware session aliasing.
- Account-scoped route bindings can send different Feishu accounts to different agents, which avoids some identity confusion, but they do not solve same-agent same-group transcript collisions.
- Existing cross-account broadcast dedupe prevents duplicate broadcast dispatch, but it is separate from single-agent group session isolation.
