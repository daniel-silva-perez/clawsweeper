## Summary

Keep this issue open. Current main still routes Feishu group messages through an account-aware route object, but the session key for non-direct peers remains `agent:<agentId>:feishu:group:<chatId>` and therefore can still collide for multiple Feishu apps in the same group when they resolve to the same agent.

## Best Possible Solution

Keep this issue open and implement account-aware Feishu group session isolation in the Feishu plugin or through an additive routing/session-key option. A conservative fix would include a normalized Feishu accountId/appId segment for non-default multi-account group routes while preserving legacy default-account continuity, add a regression test with two Feishu accounts in the same group routed to the same agent, and update the Feishu/channel-routing docs.

- Any fix needs a migration or continuity story for existing Feishu group transcripts, plus a regression test for two Feishu accounts in the same group routed to the same agent.
