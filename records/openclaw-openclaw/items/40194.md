## Summary

Keep open. Current main still passes Feishu accountId through routing and reply context, but the group session key itself remains account/app agnostic. Multiple Feishu apps in the same group that resolve to the same agent can still converge on the same `agent:<agentId>:feishu:group:<chatId>` session.

## Best Possible Solution

Implement account-aware Feishu group session isolation in the Feishu plugin or via an additive routing/session-key option. A conservative path would append a normalized accountId or appId for non-default Feishu group routes while preserving legacy default-account continuity, then add a regression test with two Feishu accounts in the same group routed to the same agent and update Feishu/group routing docs.

## Risks / Open Questions

- A blanket generic change to group session key shape could break transcript continuity for existing group channels, so the fix likely needs a Feishu-specific compatibility path or an additive opt-in session-key option.
- Account-scoped bindings can route different Feishu apps to different agents, but they do not solve the reported same-agent collision in one Feishu group.
