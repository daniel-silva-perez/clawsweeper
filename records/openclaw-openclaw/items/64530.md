## Summary

Keep this issue open. Current main still has live append-time guards, in-memory replay repair, restart resume, and orphan-user prompt merging, but it does not implement the requested durable reopen/startup repair seam that appends idempotent tail-only synthetic `toolResult` rows or an explicit assistant recovery marker for a persisted orphan-user tail.

## Best Possible Solution

Keep this issue open. The best fix is a narrow first-party reopen/startup transcript compensation seam near session load/open that inspects only the active tail, appends parentId-valid synthetic error `toolResult` rows for eligible unresolved tool calls, appends one explicit assistant recovery marker for a persisted orphan-user tail, never rewrites/deletes prior rows, is idempotent across repeated reopen, never re-executes tools, and adds focused tests for missing tail results, partial tail results, orphan-user tails, healthy no-op, and repeated reopen no-op.

## Risks / Open Questions

- Existing live and replay repair paths can mask the symptom during normal operation, so a durable reopen repair must be careful not to duplicate synthetic tool results on repeated reopen.
- The fix should append through `SessionManager.appendMessage(...)` or an equivalent parentId-aware path; raw JSONL appends risk severing the active transcript branch.
- Policy for errored or aborted assistant tool-call turns needs explicit treatment because current live and replay paths intentionally avoid synthesizing results for those turns.
