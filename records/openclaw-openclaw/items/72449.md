## Summary

Keep open. Current main still lacks the nullish guard this PR proposes for `parseAbsoluteTimeMs`, so the cron parser work is not obsolete. The PR should not be merged as-is because it also bundles an unrelated `openclaw update` early return that conflicts with current documented update semantics and touches a package-update execution path.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but split the work. The best path is to keep the cron parser guard, add parser tests, and add a targeted CLI regression for malformed `{ kind: "at" }` schedules. The update-command early return should be removed from this PR or handled separately with a design that preserves the documented package refresh, plugin sync, completion refresh, restart, and doctor semantics.
