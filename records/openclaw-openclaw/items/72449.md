## Summary

Keep open. The cron parser guard is useful, but the PR is not mergeable as-is because it also adds an unrelated same-version package-update early return that conflicts with current docs and tests and skips the established package refresh/post-update path.

## What This Changes

The PR widens `parseAbsoluteTimeMs` to accept nullish input with new parser tests, and adds an `updateCommand` early return when the installed package version already equals the resolved target version.

## Best Possible Solution

Split the branch before merge: keep the cron nullish parser guard, add parser and CLI malformed-`at` regression coverage, and remove the package-update early return. Any same-version update optimization should be handled separately against the documented refresh, plugin sync, completion, restart, and doctor contract.

## Reproduction Assessment

Yes. A malformed persisted cron job with `schedule: { kind: "at" }` and no `at` can flow through `printCronList` to `formatSchedule`, then into `parseAbsoluteTimeMs(undefined)` on current main. The PR-introduced update regression is also reproducible from the existing same-version package-update tests and docs contract.

## Solution Assessment

No. The cron parser guard is the right narrow direction, but this PR is not the best complete fix because it bundles an unrelated update-policy change that contradicts current behavior. The safer path is a cron-only patch plus targeted CLI regression coverage.
