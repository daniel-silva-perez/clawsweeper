## Summary

Keep open. Current main still has the cron nullish parser crash path, so the useful cron fix is not obsolete, but the branch should not merge as-is because it changes package-update policy in a way current docs and tests reject.

## What This Changes

The branch widens `parseAbsoluteTimeMs` to accept nullish input, adds parser unit tests, and adds a same-version package-update early return.

## Best Possible Solution

Split the branch before merge: keep the cron nullish parser guard, add an observable `cron list` malformed-`at` regression, and preserve the current same-version update refresh behavior. Any no-op package-update optimization should be designed separately against the package verification, plugin sync, completion, restart, and doctor contract.

## Reproduction Assessment

Yes. A malformed persisted cron job with `schedule: { kind: "at" }` can flow through `printCronList` to `formatSchedule`, then into `parseAbsoluteTimeMs(undefined)` on current main, which still calls `.trim()`.

## Solution Assessment

No. The cron guard is the right narrow direction, but this PR is not the best complete fix because it bundles an unrelated update-policy change; the safer path is a cron-only patch plus targeted CLI regression coverage.
