## Summary

Keep open. Current main has partial error sanitization and some WhatsApp error-payload suppression, but provider billing/rate-limit failures still become normal final reply text, WhatsApp lacks the requested suppress/custom/owner-only config surface, and Telegram's existing error policy is not wired as a shared provider-error final-reply policy.

## What This Changes

Request per-channel controls to suppress, customize, or owner-route provider/runtime error replies in WhatsApp and Telegram instead of sending raw billing, rate-limit, or system details to recipients.

## Best Possible Solution

Implement a shared outbound provider/runtime error-reply policy before channel delivery. Classify billing, rate-limit, auth, transient provider, and unclassified runtime failures; preserve diagnostics in logs/operator surfaces; and expose aligned WhatsApp and Telegram channel/account/group/topic/DM controls for normal reply, silent suppression, custom text, and owner-only routing, with docs and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the agent runner fail before reply with a billing or rate-limit error: current tests already assert the resulting final payload contains billing or rate-limit text, and the runner code emits that payload without a channel policy marker.

## Solution Assessment

Unclear for exact implementation. The requested direction is right, but the safest fix should be a shared reply-policy seam rather than independent WhatsApp and Telegram patches, and maintainers should decide defaults and owner-routing semantics before implementation.
