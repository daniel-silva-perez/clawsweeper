## Summary

Keep open. The PR addresses a real current workflow gap, but it is not mergeable as-is because it is stale against the current `exec-event` heartbeat contract and leaves generated/public config documentation inconsistent with the proposed default flip.

## What This Changes

The PR makes empty-output successful background exec completion notifications opt-out instead of opt-in, aligns the exec runtime default, updates tests and config help/types, and shortens the exec-exit wake coalescing delay.

## Best Possible Solution

If maintainers accept the default behavior change, rebase the PR onto current main, keep background exec completion wakes classified as `exec-event` with immediate coalescing, make the default consistently opt-out in both tool creation and runtime storage, regenerate schema/help artifacts, update public exec/background-process docs, and then adjust coding-agent guidance to rely on the parent-owned completion wake where appropriate.

## Reproduction Assessment

Yes. Current main can be checked statically and through the existing no-op notification test path: omitted `notifyOnExitEmptySuccess` flows through `=== true`, and the default no-op case expects no completion event.

## Solution Assessment

No, not as currently patched. The default-on direction may be the right maintainer-owned fix, but the branch should preserve the current heartbeat routing contract and update generated schema plus public docs before merge.
