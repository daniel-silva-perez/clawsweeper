## Summary

Keep open. Current main still skips empty-output successful background exec completion notifications by default, and the discussion shows a real workflow gap, but this branch is stale against the merged exec-event wake contract and leaves generated schema/public docs inconsistent.

## What This Changes

This PR makes empty-output successful background exec completion notifications opt-out, aligns runtime storage/tests/config help/types, and adds a 50 ms coalesce value to exec-exit heartbeat wake requests.

## Best Possible Solution

If maintainers accept the opt-out notification policy, rebase the change onto current main, keep completion wakes classified as `exec-event`, make the default consistently `!== false` in both tool creation and runtime storage, regenerate schema/help artifacts, update public exec/background-process docs, and then adjust bundled coding-agent guidance around parent-owned completion handling as a separate narrow docs follow-up if needed.

## Reproduction Assessment

Yes. A static current-main check gives a high-confidence reproduction path: omitted `notifyOnExitEmptySuccess` flows through `=== true`, empty successful completions return before event enqueue, and the existing no-op test matrix expects the default case to skip notifications. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as currently patched. The default-on direction may be the right maintainer-owned fix, but the branch should preserve the shipped `exec-event` heartbeat contract and update generated schema plus public docs before merge.
