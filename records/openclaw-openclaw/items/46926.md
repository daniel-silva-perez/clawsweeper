## Summary

Keep open. This PR addresses a real background exec completion gap and current main still defaults empty-success notifications off, but the branch is stale against the current exec-event wake contract and its docs/schema updates are incomplete, so it needs maintainer review rather than cleanup closure.

## What This Changes

The PR flips empty-success background exec completion notifications to default-on, aligns the runtime flag storage, updates tests and config help/types, and shortens the exit wake coalescing delay.

## Best Possible Solution

If maintainers accept the default behavior change, rebase the PR onto current main, keep the current `exec-event` wake reason with `coalesceMs: 0`, update generated schema/help plus public exec/background-process docs, and then make any coding-agent guidance reflect the shipped runtime behavior.

## Reproduction Assessment

Yes. Current main can be checked statically and through the existing no-op notification test path: omitted `notifyOnExitEmptySuccess` flows through `=== true`, and the current tests expect the empty successful background run to produce no completion event.

## Solution Assessment

No, not as currently patched. The default-on direction may be the right maintainer-owned fix, but the branch should preserve the current heartbeat routing contract and regenerate all config/doc surfaces before merge.
