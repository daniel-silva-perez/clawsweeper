## Summary

Keep this PR open. Current main still authenticates the Microsoft Teams webhook and then strict-parses JSON with `express.json`; it has no raw-body two-pass repair path and no HTTP 200 fallback for unrecoverable authenticated JSON parse failures. The PR remains a focused implementation candidate, with discussion already identifying and partly addressing repair-regex and test-mock concerns, but it needs maintainer/security review before landing.

## Best Possible Solution

Keep this PR open as the implementation candidate. Rebase it onto current main, review the Teams webhook security boundary, keep auth/JWT validation before raw body parsing, preserve the size limit, repair only truly bare invalid JSON escapes, acknowledge unrecoverable authenticated Bot Framework JSON with HTTP 200 only where intended, and add integrated Express middleware regression coverage before landing.

## Risks / Open Questions

- The PR changes parsing and acknowledgement behavior on an internet-facing Microsoft Teams webhook; maintainer review should explicitly preserve bearer precheck, Bot Framework JWT validation before body parsing, and the existing body-size limit.
- Returning HTTP 200 for unrecoverable JSON avoids Azure backoff but can also drop an authenticated activity silently; the landing version should make that fallback narrowly scoped and well logged.
- The proposed repair regression test mirrors the helper logic rather than exercising the actual Express middleware path end to end; integrated middleware coverage would reduce regression risk.
