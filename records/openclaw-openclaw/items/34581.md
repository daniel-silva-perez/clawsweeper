## Summary

Keep this PR open. The checked-out main at e583db63 still authenticates Microsoft Teams Bot Framework webhook requests and then strict-parses them with express.json, with parser error handling only for 413 payload-too-large responses. The PR's raw-body two-pass repair and authenticated HTTP 200 fallback are not implemented on main or in v2026.4.26, and the PR remains a focused implementation candidate that needs maintainer/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open as the implementation candidate. Rebase it onto current main, review the Teams webhook security boundary, keep auth/JWT validation before raw body parsing, preserve the size limit, repair only truly bare invalid JSON escapes, acknowledge unrecoverable authenticated Bot Framework JSON with HTTP 200 only where intended, and add integrated Express middleware regression coverage before landing.

## Risks / Open Questions

- The PR changes parsing and acknowledgement behavior on an internet-facing Microsoft Teams webhook. Maintainer review should explicitly preserve bearer precheck, Bot Framework JWT validation before body parsing, and MSTEAMS_WEBHOOK_MAX_BODY_BYTES enforcement.
- Returning HTTP 200 for unrecoverable authenticated malformed JSON intentionally acknowledges and drops that activity to avoid Azure Bot Service backoff; that product tradeoff should be covered by logs and integrated middleware tests.
- The added regression test mirrors the helper rather than exercising the actual Express middleware stack. Before landing, integrated coverage should verify repair success, unrecoverable authenticated acknowledgement, adapter non-invocation, middleware ordering, and payload-size handling.
