## Summary

Keep this PR open. Current main still strict-parses authenticated Microsoft Teams Bot Framework webhook bodies with `express.json()` and does not implement the PR’s raw-body parsing, JSON escape repair, or authenticated HTTP 200 acknowledgement for unrecoverable Bot Framework JSON. The PR remains a focused candidate, but it is dirty against main and needs a rebase/security review because the current branch parses raw bodies before the current main auth hardening.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is to rebase it onto current main, preserve auth-before-body-parse hardening and the existing body-size limit, implement repair only for truly bare invalid JSON escapes, return HTTP 200 only after authenticated unrecoverable Bot Framework JSON, and add integrated webhook regression coverage before landing.

## Risks / Open Questions

- Security-sensitive webhook behavior: the PR changes an internet-facing Microsoft Teams endpoint from strict JSON parsing to raw body parsing, repair, and HTTP 200 fallback; it must preserve current main’s bearer precheck, Bot Framework JWT validation before body parsing, and `MSTEAMS_WEBHOOK_MAX_BODY_BYTES`.
- Functional tradeoff: acknowledging unrecoverable JSON with HTTP 200 prevents Azure Bot Service backoff but intentionally drops that activity, so logs and integrated tests should make that behavior explicit.
- Coverage gap: the PR’s added test mirrors the helper directly; before landing, it should add integrated middleware/adapter coverage for repair success, unrecoverable authenticated acknowledgement, adapter non-invocation, middleware ordering, and payload-size handling.
