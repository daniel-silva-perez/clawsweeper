## Summary

Keep this PR open. Current main still uses strict `express.json()` on the authenticated Microsoft Teams webhook path and does not implement the PR's raw-body repair or authenticated HTTP 200 fallback for malformed Bot Framework JSON. The change is still relevant, but it needs a rebase and maintainer/security review before merge.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is a focused rebase of `extensions/msteams/src/monitor.ts` that keeps authentication before body parsing, keeps the body-size limit, repairs only truly bare invalid JSON escapes, returns HTTP 200 only for authenticated unrecoverable Bot Framework JSON to avoid Azure backoff, and adds integrated tests for repair success, unrecoverable acknowledgement, adapter invocation/non-invocation, middleware ordering, and payload limits.

## Risks / Open Questions

- Security-sensitive path: this changes an internet-facing Microsoft Teams webhook from strict JSON parsing to raw-body parsing plus repair and 200 fallback.
- A rebase must preserve the current bearer precheck, Bot Framework JWT validation before body parsing, and `MSTEAMS_WEBHOOK_MAX_BODY_BYTES`.
- The latest PR discussion still identifies a regex edge case; repair logic should be covered through integrated middleware tests, not only a copied helper in a standalone test.
