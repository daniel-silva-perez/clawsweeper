## Summary

Keep this PR open. Current main and the latest release still strict-parse authenticated Microsoft Teams Bot Framework webhook bodies with express.json and only special-case 413 parser errors, so the PR's raw-body two-pass repair and authenticated HTTP 200 fallback are not implemented. The PR discussion shows useful review iteration, but the change is security-sensitive enough to need maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open as the implementation candidate. Rebase it onto current main, review the webhook security boundary, keep auth-before-body-parse and size-limit behavior intact, repair only truly bare invalid JSON escapes, acknowledge unrecoverable authenticated Bot Framework JSON with HTTP 200 only where intended, and add integrated middleware regression tests before landing.

## Risks / Open Questions

- The PR changes parsing and acknowledgement behavior on an internet-facing Microsoft Teams webhook; maintainer review should preserve bearer precheck, Bot Framework JWT validation before body parsing, and the existing body-size limit.
- Returning HTTP 200 for unrecoverable authenticated malformed JSON intentionally acknowledges and drops that activity to avoid Azure Bot Service backoff; that product tradeoff should be covered by logs and integrated tests.
- The provided PR test coverage mirrors the repair helper; before landing, integrated Express middleware coverage should exercise repair success, unrecoverable authenticated acknowledgement, adapter non-invocation, middleware ordering, and payload-size handling.
