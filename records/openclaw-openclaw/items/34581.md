## Summary

Keep this PR open. Current main still authenticates Microsoft Teams webhook requests and then strict-parses the body with `express.json()`, with no raw-body repair path or 200 acknowledgement for authenticated but unrecoverable malformed JSON. The PR targets a plausible Teams delivery/backoff bug, but it needs maintainer rebase and security review because the same webhook path was later hardened to validate bearer/JWT auth before parsing request bodies.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is a focused rebase in `extensions/msteams/src/monitor.ts` that preserves the current bearer and Bot Framework JWT gates before body parsing, keeps the body-size limit, repairs only truly bare invalid JSON escapes, returns HTTP 200 only for authenticated unrecoverable Bot Framework JSON to avoid Azure backoff, and adds integrated middleware tests for repair success, unrecoverable acknowledgement, adapter invocation/no-invocation, ordering, and payload limits.

## Risks / Open Questions

- Security review: this PR changes an internet-facing Microsoft Teams webhook from strict JSON parsing to raw-body parsing plus repair and HTTP 200 fallback. A rebase must preserve current bearer/JWT validation before body parsing and keep `MSTEAMS_WEBHOOK_MAX_BODY_BYTES`.
- Returning HTTP 200 for unrecoverable malformed JSON may be correct for Azure backoff avoidance, but it should only happen after successful Bot Framework authentication and should not invoke the adapter on unparseable activity bodies.
- Regex-based repair has correctness risk around valid escaped backslashes mixed with invalid bare escapes; the PR discussion already found edge cases that need integrated tests against the current middleware chain.
