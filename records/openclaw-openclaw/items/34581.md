## Summary

Keep this PR open. Current main still uses strict `express.json()` for the Microsoft Teams webhook, so the reported malformed Bot Framework activity can still fail before the adapter sends the normal HTTP 200 acknowledgement; the PR remains a relevant implementation candidate and has already addressed the earlier regex/mock review feedback in its latest diff.

## What This Changes

The branch replaces MS Teams webhook JSON parsing with raw-body parsing plus a bare-invalid-escape repair pass, acknowledges unrecoverable authenticated JSON parse failures with HTTP 200, updates the lifecycle mock, and adds JSON-repair regression coverage.

## Best Possible Solution

Keep this PR open as the implementation candidate, then have the Teams maintainer review or rebase it on current main, preserve auth-before-body-parse and the existing size limit, add integrated middleware coverage for repaired and unrecoverable payloads, and land it if CI passes.

## Reproduction Assessment

Yes. A Bot Framework activity JSON string containing an invalid escape such as `Test\Project` fails `JSON.parse`; current main routes application/json through body-parser’s strict JSON parser before `adapter.process`, so the adapter’s normal 200 acknowledgement is not reached.

## Solution Assessment

Unclear whether this exact patch is ready to land. The owner-local raw-body repair approach is the right direction, but the production middleware should be covered directly and the HTTP 200 drop behavior should be accepted by the Teams/security maintainer before merge.
