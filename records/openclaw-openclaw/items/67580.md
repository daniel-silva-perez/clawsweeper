## Summary

Keep open. Current main still misses the reported inline `status: staged - Candidate:` contamination shape, rehydrates promotion snippets without checking whether the live range sits inside a managed dreaming fence, and uses total signal count rather than organic recall count for the documented `minRecallCount` gate. The open PR #68774 is a plausible targeted fix path, but this issue should stay open until that or an equivalent fix lands and maintainers decide whether broader distillation/threshold changes belong here or in #67363.

## What This Changes

Stop memory-core dreaming promotion from appending raw Candidate metadata and undistilled dreaming artifacts to `MEMORY.md`; promote only clean, durable memory snippets.

## Best Possible Solution

Review and land PR #68774 or an equivalent narrow memory-core guard for inline metadata and dreaming-fence rehydration, then keep #67363 or a follow-up RFC for the broader distillation and recall-threshold policy question so `MEMORY.md` stays curated without blocking the immediate leak fix.

## Reproduction Assessment

Yes for the core bug shape: applying the current detector logic to the reported inline `status: staged - Candidate:` sample leaves `hasNarrativeLead=false` and therefore does not mark the snippet contaminated. The full fleet bloat path is supported by production comments, but I did not rerun an end-to-end cron sweep in this read-only review.

## Solution Assessment

No, current main is not the best solution yet because it partially filters Candidate-led artifacts but misses inline metadata and lacks the rehydration fence guard. The narrow fix proposed by #68774 looks like the right immediate boundary, while LLM distillation and strict recall-only thresholds need separate maintainer direction.
