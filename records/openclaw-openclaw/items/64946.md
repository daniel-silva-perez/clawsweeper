## Summary

Keep open. Current main still preserves the documented wrapped plugin config contract, but the active config and loader validation paths can still validate an enabled plugin against `entry.config ?? {}` / `params.value ?? {}` and emit false-positive `invalid config` errors when a non-activating or metadata path has not materialized the wrapped config. The open related #70811 overlaps this loader boundary but is not merged and does not resolve this issue on main yet.

## What This Changes

The issue reports repeated false-positive plugin `invalid config` logs when a working plugin is configured under `plugins.entries.<plugin-id>.config` but another validation path behaves as if required plugin fields are missing.

## Best Possible Solution

Keep the issue open until plugin config validation consistently distinguishes real activation from metadata or snapshot paths, or carries the materialized `plugins.entries.<id>.config` value into every strict schema validation call. The fix should preserve strict validation for real activation and explicit config validation while adding regressions for strict-schema wrapped config and undefined-config non-activation paths.
