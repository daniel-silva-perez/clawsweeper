## Summary

Keep open. Current `main` still has the false-positive plugin config validation boundary: the documented contract validates plugin-owned fields under `plugins.entries.<id>.config`, but loader and config-validation paths can still validate a missing or non-materialized `entry?.config` as `{}` and emit required-property `invalid config` messages for otherwise valid plugins. The related #70811 remains an open, unmerged implementation candidate rather than proof that this issue is fixed.

## What This Changes

The issue requests eliminating false-positive plugin `invalid config` log spam when valid wrapped plugin config is validated by paths with mismatched or non-materialized config expectations.

## Best Possible Solution

Keep strict plugin config validation for real activation and explicit config checks, but make metadata/snapshot/preflight validation distinguish absent config from invalid config and add regressions for strict-schema plugins with wrapped config plus non-activating undefined-config paths.

## Reproduction Assessment

Yes. A strict plugin `configSchema` with required fields can reproduce the source-level failure when an enabled/discovered plugin reaches the loader or config validator without a materialized `entry.config`; current `main` validates that as `{}` and produces required-property `invalid config` output.

## Solution Assessment

No for current `main`. The narrow maintainable fix is context-aware validation, likely by extending or following #70811 so loader and config-validation paths defer only absent non-activation config while preserving fail-closed validation for real activation and explicit config.
