## Summary

Keep open. Current main still contains the validation boundary that can produce false-positive plugin `invalid config` noise: plugin-owned config is documented and schema-composed under `plugins.entries.<id>.config`, but loader paths still validate `entry?.config` after coercing missing config to `{}`, and the config validator does the same for enabled plugins. The related #70811 is open and overlaps the non-activating undefined-config part of the problem, but it is not merged and does not close this issue.

## What This Changes

The issue asks OpenClaw to stop repeatedly logging false plugin config validation errors when working plugins have valid config under `plugins.entries.<plugin-id>.config`.

## Best Possible Solution

Keep the issue open until plugin config validation consistently distinguishes real activation from metadata/snapshot paths, or carries materialized `plugins.entries.<id>.config` into every validation call. The fix should preserve strict validation for actual activation and explicit config validation, while adding regression tests for a strict-schema plugin with wrapped config and for non-activating undefined-config paths.
