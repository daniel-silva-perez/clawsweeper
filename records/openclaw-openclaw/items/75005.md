## Summary

Keep open. Current main still hard-requires provider API-key resolution before audio/video media provider callbacks, while this PR is an active, coherent fix for the linked no-auth local STT path and has addressed the earlier auth-precedence feedback; the remaining action is maintainer review and validation, not cleanup closure.

## What This Changes

The PR adds a `resolveSyntheticAuth` hook to media understanding providers, uses it as an audio/video execution fallback only after normal auth resolution misses provider API keys, and adds docs plus local no-auth regression tests.

## Best Possible Solution

Review and land the resolver-first, provider-owned synthetic auth hook for configured media execution, with its docs and regression coverage; keep broader auto-selection behavior or additional discovery semantics as an explicit follow-up if maintainers want that product behavior.

## Reproduction Assessment

Yes. Current main has a high-confidence code-path reproduction: a configured plugin-only audio provider with no auth reaches `resolveProviderExecutionAuth`, then `resolveApiKeyForProvider` and `requireApiKey`, before `transcribeAudio` can run, matching the linked #74644 report.

## Solution Assessment

Yes. The PR is the narrow maintainable solution for the reported execution path because it preserves normal auth precedence, keeps explicit profile failures hard-failing, and requires provider-owned opt-in instead of weakening core auth globally.
