## Summary

Keep open. Current main still sends audio/video media provider execution through model-provider auth resolution and `requireApiKey(...)` before calling the provider implementation, so the linked no-auth/local STT bug remains. This PR is the active unmerged implementation for #74644, and the latest diff appears reviewable because it keeps normal resolver precedence and only falls back to explicit provider-owned synthetic auth after a missing-provider-key failure.

## What This Changes

This PR adds a `MediaUnderstandingProvider.resolveSyntheticAuth` hook, threads the selected media provider into audio/video auth resolution, documents the local no-auth media provider pattern, and adds regression tests for synthetic auth, credential precedence, explicit profile failures, and fail-closed defaults.

## Best Possible Solution

Review and merge a resolver-first, explicit media synthetic-auth hook with the added docs and tests, keeping unmarked or remote providers fail-closed and preserving literal, env, stored-profile, and explicit-profile auth semantics.

## Reproduction Assessment

Yes. Current main can be reproduced with a configured plugin-only `mediaUnderstandingProviders` audio provider that has no model auth configured; `runCapability(...)` reaches `resolveProviderExecutionAuth(...)`, then `resolveApiKeyForProvider(...)`/`requireApiKey(...)`, and fails before `transcribeAudio(...)` runs.

## Solution Assessment

Yes. The current PR approach is the narrow maintainable fix for the reported path: it keeps the existing auth resolver first, treats explicit profile errors as hard failures, and requires provider-owned opt-in for no-auth media execution rather than weakening core auth globally.
