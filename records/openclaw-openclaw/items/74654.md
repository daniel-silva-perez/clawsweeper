## Summary

Keep this PR open, but do not merge it as-is. The service-layer validation is the right place to fix #74459, but the new enable-path guard validates the old stored cron expression even when the same update replaces that schedule with a valid non-cron schedule, which blocks a valid one-step repair path.

## What This Changes

The PR adds cron-expression pre-validation in the cron service update path, exports a schedule validation helper, adds service and gateway regression tests, and adds a changelog entry for #74459.

## Best Possible Solution

Keep the service-layer pre-mutation guard, but validate the candidate resulting schedule rather than the old persisted cron schedule for compound updates. A focused regression should cover a disabled job with a corrupted cron schedule being repaired by one `cron edit` update that replaces the schedule with `every` or `at` while enabling it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate the candidate schedule when enabling:** `src/cron/service/ops.ts:362-363`
  - body: This guard validates the old `job.schedule` whenever `patch.enabled === true`, so a patch that both enables a disabled job and replaces a corrupted cron schedule with a valid `every` or `at` schedule still throws on the stale cron expression. The CLI can send that compound patch with `cron edit <id> --every 10m --enable`, and the RPC schema accepts `enabled` plus `schedule` together. Validate the post-patch candidate on a clone, or only check the existing cron schedule when no replacement schedule is supplied.
  - confidence: 0.87
