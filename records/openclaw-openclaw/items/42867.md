## Summary

Keep open. Current main already handles the heartbeat-aware daily/idle freshness problem, and that part shipped in v2026.4.27, but normal sessions still do not expose or enforce a configurable max-turn TTL. The same author also has overlapping open TTL design in #45390, so the remaining work needs maintainer API and persistence direction rather than cleanup closure.

## What This Changes

The issue requests configurable session rotation after a maximum number of turns, plus heartbeat-aware staleness so maintenance traffic does not keep abandoned sessions fresh.

## Best Possible Solution

Define one core session TTL contract across this issue and #45390, then add schema-validated and documented max-turn rotation that counts real conversation turns, excludes maintenance/system-event traffic, persists or derives enough state to enforce the limit before session reuse, and preserves the shipped heartbeat-aware daily/idle freshness behavior.

## Reproduction Assessment

Yes. Static inspection of current main gives a high-confidence reproduction path: the proposed config keys are absent and strict-rejected, the freshness evaluator has no turn-count input, and session reuse/persistence have no counter to enforce max-turn TTL.

## Solution Assessment

Unclear. The current heartbeat-aware freshness implementation is the right direction for the second half, but the remaining max-turn behavior should be unified with #45390 before choosing the public config location, persistence semantics, and any memory-save hook behavior.
