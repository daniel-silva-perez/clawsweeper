## Summary

Keep open. Current main addresses the heartbeat-aware freshness problem, but it still has no public session reset config, schema support, or reuse-time freshness check for turn-count TTL. The related #45390 is broader, open, and same-author, so it is not a merged or maintainer-designated canonical replacement for this issue.

## What This Changes

The issue requests a configurable session reset policy that rotates sessions after a maximum number of conversation turns and prevents heartbeat maintenance traffic from keeping abandoned sessions fresh.

## Best Possible Solution

Define a core session TTL contract, then add a schema-validated and documented max-turn reset policy that counts real conversation turns, excludes heartbeat/cron/exec maintenance events, triggers rollover before session reuse, and preserves existing daily and idle reset behavior with focused regression coverage.
