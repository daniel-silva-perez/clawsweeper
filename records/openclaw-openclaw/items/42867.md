## Summary

Keep open. Current main has the heartbeat-aware daily/idle freshness behavior, but it still does not expose or enforce the requested turn-count TTL; the related same-author TTL issue is still open, so this should stay in maintainer/API review rather than being closed as implemented or superseded.

## What This Changes

The issue requests a configurable max-turn session reset plus heartbeat-aware staleness so maintenance traffic does not keep sessions fresh indefinitely.

## Best Possible Solution

Define one core session TTL contract, coordinated with #45390, then add schema-validated and documented max-turn rotation that counts real conversation turns, excludes maintenance/system-event traffic, persists enough state to enforce the limit, and checks it before session reuse while preserving the current heartbeat-aware daily/idle freshness behavior.

## Reproduction Assessment

Yes. Static inspection of current main is enough to reproduce the missing capability: the config type/schema do not accept a turn limit, the freshness evaluator has no turn input, and session reuse does not inspect a counter or transcript turn count.

## Solution Assessment

Unclear as proposed. The heartbeat-aware part is already solved, but the remaining max-turn behavior should be unified with the broader TTL contract in #45390 before choosing a public config location and persistence semantics.
