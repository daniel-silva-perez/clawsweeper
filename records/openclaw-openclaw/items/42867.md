## Summary

Keep open. Current main already has the heartbeat-aware daily/idle freshness behavior, but it still does not expose or enforce the requested turn-count TTL; the overlapping same-author TTL design in #45390 is still open, so this needs maintainer/API review rather than closure.

## What This Changes

The issue requests a configurable session reset policy that auto-rotates sessions after a maximum number of turns and prevents heartbeat or maintenance messages from keeping abandoned sessions fresh.

## Best Possible Solution

Define one core session TTL contract across this issue and #45390, then add schema-validated and documented max-turn rotation that counts real conversation turns, excludes maintenance/system-event traffic, persists enough state to enforce the limit, and checks it before session reuse while preserving the current heartbeat-aware daily/idle freshness behavior.

## Reproduction Assessment

Yes. Static inspection of current main is enough to reproduce the missing capability: the config/schema do not accept a turn limit, the freshness evaluator has no turn input, and session reuse does not inspect any counter.

## Solution Assessment

Unclear. The heartbeat-aware half is already solved; the remaining max-turn behavior should be unified with #45390 before choosing the public config location, persistence semantics, and whether rotation should include memory-save hooks.
