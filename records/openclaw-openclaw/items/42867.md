## Summary

Keep open. Current main and v2026.4.27 address the heartbeat-aware freshness half by separating `sessionStartedAt` and `lastInteractionAt` from bookkeeping `updatedAt`, but the requested turn-count TTL is still absent from the reset config type, strict schema, freshness policy, session store, docs, and reuse gate. Related #45390 is open and overlaps the remaining TTL contract, but it is not a merged or shipped canonical replacement.

## What This Changes

The issue requests a configurable maximum-turn session reset plus heartbeat-aware staleness so maintenance traffic cannot keep an overgrown session alive indefinitely.

## Best Possible Solution

Define one core session TTL contract, then add schema-validated and documented max-turn rotation that counts real conversation turns, excludes maintenance/system-event traffic, rolls before session reuse, and preserves the current heartbeat-aware daily/idle freshness behavior.
