## Summary

Keep open. Current main already addresses the heartbeat-aware daily/idle freshness half, but it still does not expose or enforce a turn-count TTL, and the overlapping same-author TTL design in #45390 is still open for maintainer API direction.

## What This Changes

The issue requests session auto-rotation after a configured maximum number of turns, plus heartbeat-aware staleness so maintenance traffic does not keep sessions alive indefinitely.

## Best Possible Solution

Define one core session TTL contract across this issue and #45390, then add schema-validated and documented max-turn rotation that counts real conversation turns, excludes maintenance/system-event traffic, persists or derives enough state to enforce the limit, and checks it before session reuse while preserving the shipped heartbeat-aware daily/idle freshness behavior.

## Reproduction Assessment

Yes. Static inspection of current main is enough: the proposed config keys are absent and strict-rejected, the freshness evaluator has no turn-count input, and session reuse/persistence have no counter to enforce a max-turn TTL.

## Solution Assessment

Unclear. The heartbeat-aware half is already solved; the remaining max-turn behavior should be unified with #45390 before choosing the public config location, persistence semantics, and any memory-save hook behavior.
