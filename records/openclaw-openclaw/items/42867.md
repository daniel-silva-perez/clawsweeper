## Summary

Keep open. Current main fixes the heartbeat-aware freshness half, but the core reset policy still has no schema/config field or freshness/reuse check for max-turn rollover; #45390 overlaps but is still open, same-author, and not a maintainer-designated canonical replacement.

## What This Changes

Request to add a configurable turn-count TTL that auto-rotates sessions after N conversation turns and keeps heartbeat/system maintenance traffic from masking stale sessions.

## Best Possible Solution

Define the core session TTL contract, then implement a schema-validated and documented max-turn reset policy that counts real conversation turns while excluding heartbeat/cron/exec maintenance events, triggers rollover before session reuse, and preserves existing daily/idle behavior with focused regression tests.
