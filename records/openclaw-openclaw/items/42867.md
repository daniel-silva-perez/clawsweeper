## Summary

Keep open. Current main has the heartbeat-aware freshness behavior, but the requested turn-count TTL is still missing from the public reset config, strict schema, freshness policy, persisted session entry, docs, and session reuse gate. Related #45390 overlaps the remaining TTL contract, but it is still open and same-author, so it should be handled by maintainer/API review rather than closing this issue as superseded.

## What This Changes

Request a configurable session reset limit after N conversation turns, plus heartbeat-aware staleness so maintenance turns do not keep abandoned sessions alive.

## Best Possible Solution

Define one core session TTL contract, likely coordinated with #45390, then add schema-validated and documented max-turn rotation that counts real conversation turns, excludes maintenance/system-event traffic, persists enough state to enforce the limit, and rolls before session reuse while preserving the current heartbeat-aware daily/idle freshness behavior.

## Reproduction Assessment

Yes. The missing turn-count TTL is reproducible by static inspection of current main: the config type/schema do not accept a turn limit, the freshness evaluator has no turn input, and session reuse does not inspect transcript or turn count.

## Solution Assessment

Unclear as proposed. The heartbeat-aware part is already solved, but the remaining max-turn solution should be unified with the broader TTL contract in #45390 before choosing a public config shape.
