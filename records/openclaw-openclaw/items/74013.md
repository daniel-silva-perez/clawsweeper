## Summary

Keep this PR open. Current main still blocks self-hosted Firecrawl endpoints, so the PR is not obsolete, but this exact diff is not safe to land as-is: it relaxes Firecrawl endpoint validation with a global allowSelfHosted resolver, sends the Firecrawl bearer token to the resolved endpoint, and overlaps the still-open canonical bug #63877 plus open implementation attempts #59666 and #63941.

## Best Possible Solution

Keep this PR open for maintainer review, but do not land it as-is. The best path is one canonical Firecrawl self-hosting fix for #63877 that preserves strict hosted defaults, scopes any explicit opt-in per operation if the flag remains, allows custom public hosts only over HTTPS, allows cleartext HTTP only for private or loopback targets after `assertHttpUrlTargetsPrivateNetwork`, routes private self-hosted endpoints through `withTrustedWebToolsEndpoint`, keeps the official hosted endpoint on the strict path, updates manifest/docs/tests, and supersedes #59666 and #63941 as appropriate.

## Risks / Open Questions

- The PR is security-sensitive because it broadens where Firecrawl requests, including the configured bearer token, may be sent.
- `allowSelfHosted` is resolved globally across `webSearch` and `webFetch`, so enabling it for one operation relaxes endpoint validation for the other.
- The PR permits arbitrary HTTP endpoints when `allowSelfHosted` is true, without private/loopback DNS/IP validation and without switching private endpoints to the trusted web-tools guard.
- There are multiple open overlapping implementation attempts for the same Firecrawl self-hosting bug, so maintainers should choose one canonical endpoint policy before landing any of them.
