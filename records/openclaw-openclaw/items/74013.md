## Summary

Keep open. Current main still rejects self-hosted Firecrawl endpoints, but this PR is not safe to merge as-is because the endpoint relaxation is broader than the reported local/private use case, the self-hosted opt-in leaks across search/fetch configuration, and private local endpoints would still go through the strict web-tools guard.

## What This Changes

The PR adds Firecrawl `allowSelfHosted` config fields and passes that flag into endpoint resolution so configured non-official Firecrawl base URLs can be accepted.

## Best Possible Solution

Land one canonical Firecrawl self-hosting change for #63877 that preserves strict defaults, allows custom public Firecrawl hosts only over HTTPS, allows cleartext HTTP only after private/loopback validation, routes private self-hosted endpoints through the trusted web-tools guard, and updates manifest, docs, and runtime tests consistently.

## Reproduction Assessment

Yes. Current-main source and tests reproduce the existing rejection path for HTTP and custom-host Firecrawl base URLs; the PR-specific bugs are source-reproducible from the diff by enabling `allowSelfHosted` on only one operation and by following an allowed local HTTP endpoint into the unchanged strict fetch guard.

## Solution Assessment

No. The PR addresses a real gap, but the proposed flag and endpoint policy are broader than the narrow maintainable fix; the safer path is the existing SearXNG-style split of HTTPS public hosts, validated private HTTP, and trusted guarded fetch for private endpoints.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Restrict cleartext self-hosted endpoints to private targets:** `extensions/firecrawl/src/firecrawl-client.ts:70-73`
  - body: When `allowSelfHosted` is true, this accepts any `http://` Firecrawl base URL while `postFirecrawlJson` sends an `Authorization` bearer header to that URL. HTTP should be limited to loopback/private targets with `assertHttpUrlTargetsPrivateNetwork`; public custom hosts should remain HTTPS-only.
  - confidence: 0.88
- **[P2] Scope allowSelfHosted to the selected Firecrawl config:** `extensions/firecrawl/src/config.ts:202-206`
  - body: `resolveFirecrawlAllowSelfHosted` ORs `webSearch` and `webFetch`, so opting one operation into self-hosted mode relaxes endpoint validation for the other operation too. Resolve the flag from the same config branch as the caller/base URL, or make it one explicit shared Firecrawl policy.
  - confidence: 0.86
- **[P2] Route private self-hosted calls through the trusted guard:** `extensions/firecrawl/src/firecrawl-client.ts:315-317`
  - body: The new resolver can return `http://127.0.0.1...` when opted in, but requests still use `withStrictWebToolsEndpoint`, whose default SSRF policy blocks localhost/private/internal targets. That means the advertised local self-hosted endpoint can pass the resolver test and still fail at runtime.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff broadens Firecrawl endpoint trust in a way that can leak credentials over public cleartext HTTP and can relax one operation from the other operation's config.

Concerns:

- **[high] Public HTTP endpoint can receive Firecrawl credentials:** `extensions/firecrawl/src/firecrawl-client.ts:70`
  - body: With `allowSelfHosted` true, the resolver accepts arbitrary `http://` hosts and the request path sends `Authorization: Bearer ...`; this should be constrained to validated private/loopback HTTP targets.
  - confidence: 0.88
- **[medium] Self-hosted opt-in leaks across search and fetch:** `extensions/firecrawl/src/config.ts:202`
  - body: The new resolver treats either operation's `allowSelfHosted` flag as global, so a fetch-only opt-in can relax validation for search and vice versa.
  - confidence: 0.86
