## Summary

Keep this PR open for maintainer review. Current main still rejects self-hosted Firecrawl endpoints, and this PR is a focused implementation candidate, but it is security-sensitive, overlaps other open Firecrawl self-hosting work, and has at least one review finding on the proposed allowSelfHosted scoping.

## Best Possible Solution

Keep this PR open for maintainer review, but do not land it as-is. The best path is a canonical Firecrawl self-hosting fix that keeps default cloud behavior strict, scopes any explicit self-hosted opt-in per operation, allows public custom hosts only over HTTPS, allows cleartext HTTP only for private or loopback targets, routes trusted private endpoints through the appropriate web-tools guard, preserves SecretRef tests, updates Firecrawl/web_fetch docs, and then resolves #63877 while superseding overlapping PRs #63941 and #59666 as appropriate.

## Risks / Open Questions

- The PR intentionally relaxes Firecrawl endpoint restrictions; maintainers should review the SSRF and credential-leakage implications before merge.
- With allowSelfHosted true, the proposed resolveEndpoint accepts arbitrary hosts and cleartext HTTP, so a public HTTP endpoint could receive the Firecrawl bearer token over plaintext if misconfigured.
- The proposed resolver ORs webSearch and webFetch allowSelfHosted flags, so enabling one surface widens the other surface unless scoped before merge.
- The proposed diff does not switch private self-hosted Firecrawl requests from withStrictWebToolsEndpoint to the trusted web-tools guard, so local/RFC1918 endpoints may still fail after endpoint validation passes.
- Docs still state Firecrawl baseUrl overrides are restricted to the official HTTPS host; landing any implementation needs docs alignment.
- The same user problem is already represented by open #63877 plus overlapping PRs #63941 and #59666, so maintainers should choose one canonical implementation rather than merging divergent variants.
