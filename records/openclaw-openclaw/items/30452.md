## Summary

Current main implements the compaction and memory-flush model override parts, but the requested flush-then-reset mode is still missing. The issue should remain open and be narrowed to a maintainer-approved `memoryFlush.resetAfterFlush` or equivalent reset policy.

## Best Possible Solution

Keep this open, but narrow the remaining scope to a maintainer-reviewed reset-after-flush design. The best path is to add an explicit config/schema/docs surface, reset only after a verified successful memory flush, skip redundant compaction when configured, reuse existing session reset or successor-transcript machinery, and cover queue/session metadata, transcript replay, active runs, subagents/ACP, and channel delivery behavior with focused tests.
