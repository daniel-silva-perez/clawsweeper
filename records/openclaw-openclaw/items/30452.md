## Summary

Current main implements the compaction and memory-flush model override portions, but it still does not support the requested flush-then-reset/reset-after-flush mode. The issue should stay open, narrowed to a maintainer-approved automatic reset policy after a verified successful memory flush.

## Best Possible Solution

Keep this open, but narrow it to a maintainer-reviewed reset-after-flush design. Add an explicit config/schema/docs surface only after deciding the lifecycle semantics; reset only after a verified successful memory flush, skip redundant compaction when configured, reuse the existing session reset/transcript replay machinery where appropriate, and cover session metadata, queued follow-ups, active runs, subagents/ACP, and channel delivery with focused tests.

## Risks / Open Questions

- The existing memory flush can trigger nested compaction/session rotation metadata updates, so reset-after-flush needs a clear ordering contract with compaction counts, successor transcripts, and queued follow-up session refresh.
- The model override portions are already implemented; keeping this issue broad could invite duplicate PRs unless maintainers narrow it to reset-after-flush semantics.
