## Summary

Keep open. Current main still honors short explicit isolated `agentTurn` timeouts after the embedded runner signals execution start, and the docs expose `lightContext`/`toolsAllow` without the reported cold-start timeout warning. An open fix PR, #74893, references this issue with closing syntax, so the right cleanup path is to review or land that PR rather than close the issue early.

## What This Changes

The issue asks for a guardrail or documentation update for isolated cron `agentTurn` jobs with short explicit timeouts that omit `lightContext` and a restricted `toolsAllow`.

## Best Possible Solution

Review and land a narrow fix that warns on create/update for risky isolated cron `agentTurn` jobs with short explicit timeouts and no lightweight context or tool allowlist, plus update the public cron docs to explain the cold-start budget tradeoff without changing explicit timeout semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an isolated `agentTurn` cron job with `timeoutSeconds: 60`, no `lightContext`, and no `toolsAllow`; current code starts the cron timeout before workspace/runtime bootstrap, though no live 60-second run was executed in this read-only review.

## Solution Assessment

Unclear until PR #74893 is reviewed, but the proposed direction is right: a targeted warning/docs guardrail is safer than silently inflating explicit user timeouts or moving broad scheduler semantics.
