## Summary

Keep open. Current main has addressed the older isolated-cron lane-wait/deadlock class, but this report is about a remaining cold-start budget gap: a short explicit `timeoutSeconds` is still honored, the cron timer starts when embedded execution is signaled, and that signal occurs before workspace/runtime bootstrap work that can consume the budget.

## What This Changes

The issue requests a warning, safer default, or prominent docs for isolated `agentTurn` cron jobs whose short explicit timeout can be spent on full context/tool bootstrap before useful work begins.

## Best Possible Solution

Add a narrow, tested guardrail for risky isolated/current/custom-session cron `agentTurn` create/update configurations, and document that short explicit timeouts should account for cold-start/bootstrap overhead by using `lightContext` and a restricted `toolsAllow` for simple jobs. Keep explicit timeout semantics stable rather than silently inflating user-provided values.

## Reproduction Assessment

Yes. A focused reproduction can create an isolated `agentTurn` job with a short explicit timeout, no `lightContext`, and no restricted `toolsAllow`, then simulate `onExecutionStarted` firing before bootstrap and verify the timeout fires before useful work; the current code path supports the reported live behavior even without running a full 60-second live job here.

## Solution Assessment

Yes, the requested warning/docs direction is the narrowest maintainable fix. Silently changing explicit timeout behavior would be more surprising; a guardrail plus prominent docs preserves user intent while making the failure mode discoverable.

## Repair Work Prompt

Fix the cold-start timeout guidance/guardrail for https://github.com/openclaw/openclaw/issues/74803. Observable problem: isolated/current/session cron `agentTurn` jobs with a short explicit `timeoutSeconds`, no `lightContext: true`, and no restricted `toolsAllow` can spend the whole budget on full bootstrap before useful work.
