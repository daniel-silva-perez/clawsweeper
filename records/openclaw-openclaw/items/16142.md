## Summary

Keep open. Current main still routes main-session cron reminder text through the heartbeat cron-event prompt and model-backed reply path; the cost concern is valid, but the fix needs maintainer product judgment because not every cron/system event can safely bypass model reasoning. Open PR #48883 is a related implementation candidate for constraining main-session cron events, but it is not merged and does not fully settle the requested deterministic low-cost delivery behavior.

## What This Changes

Request to reduce API usage by avoiding model-backed formatting of recurring main-session cron reminders and using deterministic formatting or delivery where it is safe.

## Best Possible Solution

Define a narrow low-cost delivery mode for simple main-session cron reminders, preferably through an explicit cron option or conservative simple-payload detector, while preserving the existing model-backed cron-event path for complex, internal, tool-using, or reasoning-dependent events. Fold in or reconcile the relevant parts of #48883, update docs/tool guidance, and add regression tests for both deterministic and model-backed paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: enqueue a tagged main-session cron system event, run heartbeat, and observe the cron-event prompt plus getReplyFromConfig call; current tests assert this model-backed behavior.

## Solution Assessment

Unclear. Deterministic formatting is directionally right for simple reminders, but the requested blanket replacement is too broad unless maintainers first define which cron events are safe to deliver without model interpretation and how that interacts with #48883's main-session tokenization approach.
