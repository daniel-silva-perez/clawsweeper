## Summary

Keep open. Current main still routes main-session cron system-event reminders through the heartbeat reply/model path: cron enqueues the payload as a `cron:<jobId>` system event, heartbeat builds a `cron-event` prompt asking the model to relay the reminder, and `getReplyFromConfig(...)` performs the model-backed reply turn. The prior maintainer-visible review also kept this open, and related PR #15403 is closed unmerged and concerns shell rc/env loading rather than the model-cost path.

## Best Possible Solution

Keep this open for maintainer product judgment, then implement a narrow deterministic delivery path only for clearly simple main-session reminder payloads, or add an explicit cron option that chooses deterministic delivery. Preserve the model-mediated path for cron payloads that need reasoning or tools, and cover routing, event draining, delivery visibility, and target=last behavior with focused tests.
