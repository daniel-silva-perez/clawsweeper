## Summary

Keep open for maintainer review. Current main still has the undefined-result crash path reported in the linked bug, and this PR is a plausible narrow fix with relevant CI green, but overlapping open fixes mean maintainers should choose the canonical branch and decide whether void command handlers become an explicit SDK contract.

## What This Changes

This PR normalizes nullish or non-object plugin command results in the shared executor and command dispatch paths, with regression coverage for shared, generic reply, and Telegram native command handling.

## Best Possible Solution

Land one canonical narrow fix for the linked regression that normalizes plugin command results at the shared executor boundary and keeps generic plus Telegram native dispatch resilient; align the SDK contract if void command handlers are intentionally supported.

## Reproduction Assessment

Yes. On current main, a plugin command handler that resolves `undefined` flows through `executePluginCommand` and reaches `handlePluginCommand`, which reads `result.continueAgent`; the linked release bug provides matching Telegram logs from v2026.4.27.

## Solution Assessment

Yes, mostly. Normalizing at `executePluginCommand` is the narrowest maintainable fix, and caller-side guards are reasonable defense for direct or mocked runtime paths; the remaining decision is whether to document/type void returns as supported API.
