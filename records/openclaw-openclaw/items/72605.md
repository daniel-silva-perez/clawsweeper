## Summary

Keep open. Current main has related WebChat duplicate-submit fixes, but it still does not preclaim `chat.send` idempotency before attachment/model preprocessing, so this PR is not obsolete and remains a plausible backend fix for the race described in #72536.

## What This Changes

The PR preclaims `chat:<idempotencyKey>` as `in_flight` before WebChat attachment/model preprocessing, teaches `agent.wait` to ignore stale same-run agent snapshots during that preclaim, and updates gateway tests plus the changelog.

## Best Possible Solution

Keep the PR open and have a gateway maintainer either rebase this narrow backend idempotency preclaim onto current main or land an equivalent source change that preserves the newer media offload path and locks the race with the focused gateway tests.

## Reproduction Assessment

Yes. The focused gateway test seam can hold the first attachment-backed `chat.send` at `loadGatewayModelCatalog` and send a duplicate with the same idempotency key; current main still proves the duplicate can become the `started` call before the first resumes.

## Solution Assessment

Yes, in direction. Claiming the gateway dedupe entry before the first async preprocessing boundary is the narrow backend fix for same-idempotency retries; the safer final form is the same idea rebased onto current main's attachment/offload code.
