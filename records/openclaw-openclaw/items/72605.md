## Summary

Keep this PR open. Current main still has the attachment/model preprocessing await window before the `chat.send` idempotency state is claimed, and the current focused test still expects the duplicate send to become the `started` request. The PR is directionally correct for the gateway idempotency contract, but it is stale over current main's newer chat attachment/media offload code and needs maintainer review or a careful rebase before merge.

## What This Changes

This PR preclaims `chat.send` idempotency entries as `in_flight` before attachment preprocessing, updates `agent.wait` stale-snapshot handling, adds gateway race tests, and adds a changelog entry.

## Best Possible Solution

Land this backend idempotency preclaim, or an equivalent patch, in the current gateway `chat.send` implementation while preserving the newer media offload path and keeping the focused gateway tests that prove duplicate same-id sends return `in_flight` before controller registration.

## Reproduction Assessment

Yes for the idempotency-order race this PR addresses: the focused gateway test can hold the first attachment-backed `chat.send` at `loadGatewayModelCatalog`, send a duplicate with the same idempotency key, and observe current main returning `started` for the duplicate. That seam does not independently prove every broader duplicate-response symptom from #72536, but it does reproduce the backend contract gap.

## Solution Assessment

Yes in direction. Claiming the gateway dedupe entry before the first async attachment/model preprocessing boundary is the narrowest maintainable backend fix; the exact branch needs rebase over current main before it can be considered merge-ready.
