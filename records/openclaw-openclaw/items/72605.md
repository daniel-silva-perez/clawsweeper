## Summary

Keep open. Current main still does not claim the `chat:<idempotencyKey>` dedupe entry before the attachment/model preprocessing await, and the existing gateway test still allows the duplicate request to become the `started` call. The PR is directionally correct but needs maintainer review and a careful rebase over newer gateway media offload changes.

## What This Changes

The PR preclaims WebChat `chat.send` idempotency entries as `in_flight` before attachment preprocessing, updates `agent.wait` stale-snapshot handling, and adds focused gateway tests plus a changelog entry.

## Best Possible Solution

Land a rebased equivalent of the early gateway dedupe preclaim and stale-snapshot guard, preserving the current media offload path and locking the behavior with the focused gateway tests.

## Reproduction Assessment

Yes. The gateway test seam can hold the first attachment-backed `chat.send` at `loadGatewayModelCatalog` and send a duplicate with the same idempotency key; current main shows the duplicate can return `started` before the first resumes.

## Solution Assessment

Yes, in direction. Claiming the chat dedupe entry before the first async attachment preprocessing boundary is the narrow backend fix for same-idempotency retries; the final form should be rebased onto current main's attachment/offload code.
