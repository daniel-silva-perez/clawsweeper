## Summary

Keep open. Current main still has the reported error-observability problem in pairing and config recovery, and the overlapping fixes are open or closed-unmerged rather than landed, so this is neither implemented nor safely superseded.

## What This Changes

The issue requests replacing or documenting empty catch blocks in pairing, infra, and config code so expected fallbacks are explicit and unexpected errors remain diagnosable.

## Best Possible Solution

Keep this as the canonical cleanup until maintainers land or replace the overlapping pairing/config PRs with a targeted audit that logs or rethrows unexpected I/O failures, documents intentional fallbacks, and adds focused regression coverage before considering a broader no-empty-catch guard.

## Reproduction Assessment

Yes. Source inspection on current main with `rg -n "catch\\s*\\{\\s*\\}" src/pairing src/infra src/config` still finds exact empty catches in pairing and config recovery, and nearby code shows those catches can hide stat/write/copy failures.

## Solution Assessment

No landed solution is present yet. The narrow maintainable fix is to classify each catch site by expected fallback versus unexpected failure, then log/rethrow or comment/test accordingly rather than applying a broad lint rule before the audit.
