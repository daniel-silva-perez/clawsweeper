## Summary

Keep open. This is a recent, concrete Linux heartbeat reliability report with a second user confirming similar behavior after the earlier #52270 re-arm fix. Current main has meaningful scheduler and wake-layer hardening, but I did not find source, changelog, test, or related-thread evidence that the reported long-gap/silent-stop behavior is definitively fixed or superseded.

## Best Possible Solution

Keep this issue open as the canonical Linux long-run heartbeat reliability report unless a narrower merged fix is identified. The next maintainer path should request a current-build diagnostics export, `openclaw system heartbeat last`, and a redacted gateway log slice from an affected user, then add instrumentation or regression coverage around last/next heartbeat state, wake pending/running flags, requests-in-flight retries, session-lane contention, event-loop saturation, and gateway restart/reload behavior.
