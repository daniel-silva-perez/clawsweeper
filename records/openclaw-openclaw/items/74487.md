## Summary

Keep open. The report is fresh, concrete, and filed against the latest release, and current main does not show a targeted fix for the Telegram plus Ollama/Kimi path. Source inspection shows `exec` is supposed to be part of the coding tool profile and that `/tools` and the embedded runner both derive their tool inventory from the same OpenClaw tool construction path, so a Telegram session advertising `exec` but returning `Tool exec not found` is a plausible real dispatch/registration mismatch rather than expected behavior.

## What This Changes

The issue requests that Telegram DM sessions using `ollama/kimi-k2.5:cloud` execute the advertised `exec` tool instead of returning `Tool exec not found`.

## Best Possible Solution

Add a focused regression for a Telegram-shaped embedded run using Ollama/Kimi tool calls and make the actual registered custom-tool allowlist match the `/tools` inventory for `exec`, preserving existing allow/deny policy and exact-name safety before adding any provider-name normalization.
