## Summary

Keep open. Current main has adjacent session-lock hardening, including a typed `SessionWriteLockTimeoutError` and failover suppression, but it does not definitely implement this PR’s intended change: the embedded attempt still calls `acquireSessionWriteLock` without passing `params.timeoutMs`, and the plugin SDK surface exports `acquireSessionWriteLock` without the matching lock-timeout error helper/type. The PR also needs rebasing against the newer error shape on main.

## Best Possible Solution

Keep this PR open for maintainer review or contributor update. The clean path is to rebase it onto current main, decide whether the embedded attempt should still pass `params.timeoutMs` as the lock wait or use a narrower/configurable wait, and, if a public programmatic signal is intended, expose the existing `SessionWriteLockTimeoutError`/code/helper through the supported plugin SDK surface instead of adding a second error shape. Add focused tests for the call-site timeout propagation and SDK export if those changes are accepted.
