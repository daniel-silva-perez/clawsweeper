## Summary

Current main still lacks the requested WhatsApp safeguard. Plain visible auto-replies are delivered through the inbound WhatsApp message's `reply`/`sendMedia` closures, which are bound to the originating chat, and the current config/schema/docs cover DM allowlists, last-route pinning, and explicit outbound sends but not a policy that blocks or forces explicit targeting for non-owner DM auto-replies.

## Best Possible Solution

Keep this open for maintainer product/security review. The likely fix belongs in the WhatsApp plugin's visible auto-reply path before `deliverWebReply` sends payloads, with a typed account/root config policy, docs, schema/help updates, and regression coverage for owner DMs, allowlisted non-owner DMs, and explicit `message` tool sends that should remain intentional.

## Risks / Open Questions

- The report describes a confidentiality leak class in WhatsApp DM sessions, so the fix should be treated as security-sensitive routing policy rather than a cosmetic config option.
- Blocking plain replies, auto-routing them to the owner, or changing defaults could alter existing WhatsApp automation behavior; maintainers need to choose the product semantics deliberately.
