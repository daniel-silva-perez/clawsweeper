## Summary

Keep this PR open because current main still has the reported FreeBSD gap, but the proposed fix should not merge as-is: it routes FreeBSD into a legacy write/copy path that pre-creates missing destination files and can regress all-or-nothing new-target behavior after a crash or kill.

## What This Changes

The PR adds FreeBSD to the platform checks that route `writeFileWithinRoot` and `copyFileWithinRoot` through the legacy filesystem fallback instead of the Python pinned-write helper.

## Best Possible Solution

Land a tested FreeBSD-compatible filesystem write/copy fix that avoids the Python-vs-Node device identity mismatch while preserving atomic new-target materialization, hardlink and symlink defenses, root-boundary checks, and post-write verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before writing the temp file. For a missing destination, that creates the final inode before rename, so a kill or crash can leave an empty target where the pinned helper would only materialize the path at `os.replace`. Please normalize pinned-helper identity or preserve atomic new-target behavior on FreeBSD.
  - confidence: 0.84
