## Summary

Keep this PR open for maintainer handling. Current main still routes FreeBSD write/copy operations through the Python pinned-write helper with strict device identity comparison, but the PR should not merge as-is because it extends legacy fallback semantics to FreeBSD and can regress interrupted new-target writes to a zero-byte final path.

## What This Changes

The PR adds `process.platform === "freebsd"` beside Windows in `writeFileWithinRoot` and `copyFileWithinRoot`, sending FreeBSD writes and copies through the existing legacy fallback instead of the Python pinned-write helper.

## Best Possible Solution

Land a tested FreeBSD-compatible filesystem write/copy fix that avoids the Python-vs-Node device identity mismatch while preserving pinned-helper atomic new-target materialization, hardlink rejection, symlink/root-boundary checks, and post-write verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve atomic new-target creation on FreeBSD:** `src/infra/fs-safe.ts:555`
  - body: Routing FreeBSD into `writeFileWithinRootLegacy`/`copyFileWithinRootLegacy` calls `openWritableFileWithinRoot` before the temp write. For a missing destination that creates the final inode before rename, so a crash can leave a zero-byte target where the pinned helper would not materialize the final path until replace. Please fix the pinned-helper identity flow or use a FreeBSD path that keeps new-target atomicity.
  - confidence: 0.84
