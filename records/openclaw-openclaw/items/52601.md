## Summary

Keep this PR open. The reported FreeBSD failure is plausible on current main, but the proposed fallback changes a hardened write path by extending legacy new-target creation semantics to FreeBSD, which needs maintainer judgment or an updated fix before merge.

## What This Changes

The PR adds FreeBSD to the legacy fallback guards in `writeFileWithinRoot` and `copyFileWithinRoot`.

## Best Possible Solution

Land a tested FreeBSD-compatible write/copy fix that avoids the Python-vs-Node device identity mismatch while preserving atomic new-target materialization, hardlink rejection, symlink and root-boundary checks, and post-write verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve atomic creation on FreeBSD writes:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write. For a missing destination, that creates the final inode before rename, so a crash can leave a zero-byte target where the pinned helper would not materialize the final path until replacement. Please normalize the pinned-helper identity flow or use a FreeBSD path that keeps new-target atomicity.
  - confidence: 0.84
