## Summary

Keep open. Current main still has the reported FreeBSD pinned-helper identity gap, but this PR should not merge as-is because it extends the legacy fallback's pre-created target window to FreeBSD writes and copies on a hardened filesystem path.

## What This Changes

The PR adds FreeBSD to the platform guards that route `writeFileWithinRoot` and `copyFileWithinRoot` through the existing legacy filesystem fallback helpers.

## Best Possible Solution

Land a tested FreeBSD-compatible write/copy fix that avoids the Python-vs-Node device identity mismatch while preserving the desired atomic new-target behavior, symlink and hardlink defenses, root-boundary checks, and post-write verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write. When the destination is missing, that creates the final inode before rename, so a crash can leave an empty target where the pinned helper would not materialize one. Please normalize the pinned-helper identity flow or use a FreeBSD path that preserves new-target atomicity.
  - confidence: 0.82
