## Summary

Keep this PR open, but do not merge it as-is. Current main still routes FreeBSD writes through the Python pinned-write helper, so the reported compatibility problem is plausible, but the PR extends the legacy fallback semantics to FreeBSD and creates a new-target interruption window that the current pinned path avoids.

## What This Changes

The PR adds FreeBSD to the platform guards that send `writeFileWithinRoot` and `copyFileWithinRoot` through the legacy filesystem fallback helpers instead of the Python pinned-write helper.

## Best Possible Solution

Land a tested FreeBSD-compatible filesystem write/copy fix that avoids the Python-vs-Node device identity mismatch while preserving atomic new-target materialization, hardlink and symlink defenses, root-boundary checks, and post-write verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:555`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write/copy. For a missing destination, that creates the final path before rename, so a crash can leave an empty target where the current pinned path only materializes the destination at `os.replace`; please normalize the pinned-helper identity flow or preserve new-target atomicity on FreeBSD.
