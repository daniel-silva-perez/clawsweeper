## Summary

Keep this PR open. Current main still routes FreeBSD write/copy operations through the pinned Python helper, so the reported FreeBSD/ZFS identity mismatch remains plausible and unresolved; however, the proposed fallback changes a hardened write path and needs maintainer judgment before merge.

## What This Changes

The PR changes `writeFileWithinRoot` and `copyFileWithinRoot` so FreeBSD uses the existing legacy filesystem fallback path instead of the Python pinned-write helper.

## Best Possible Solution

Land a tested FreeBSD-compatible filesystem write/copy fix that avoids the Python-vs-Node device identity mismatch while preserving the safety properties maintainers want for new destination creation, hardlink rejection, symlink boundaries, and post-write verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write; when the destination is missing, that creates the final inode before rename. If the process is killed in that window, FreeBSD can retain a zero-byte target where the pinned helper would not materialize the final path until replace. Please either normalize the pinned-helper identity flow or use a FreeBSD path that preserves new-target atomicity.
  - confidence: 0.82
