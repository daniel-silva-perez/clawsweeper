## Summary

Keep this issue open. Current main at 68561a8c still has activity-based idle timeout behavior and tool-call-only loop detection, but no shipped semantic repetition guard for streamed assistant text/thinking and no loop-aware quarantine of repetitive aborted assistant output. The related #39961 is still an open implementation candidate, not current main behavior.

## Best Possible Solution

Keep this issue open. Review or adapt #39961, or implement an equivalent core stream guard near the assistant stream subscription path: detect repeated text/thinking while the stream is active, abort with explicit loop metadata, prevent buffered repetitive content from being drained as a final reply, and quarantine loop-aborted assistant output from provider replay while preserving transcript visibility.

## Risks / Open Questions

- A streaming repetition breaker needs conservative false-positive handling for legitimate repeated logs, generated lists, tables, code blocks, and progress output.
- Loop-aware sanitization should preserve visible transcript/audit history while preventing loop-aborted assistant content from being replayed as normal provider context.
- The open #39961 may be a useful starting point, but it still needs maintainer review against current stream handling, block-buffer draining, and replay metadata behavior.
