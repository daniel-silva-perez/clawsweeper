## Summary

Keep this open. Current main only partially satisfies the request: batch embedding, exact hash reuse, safe-reindex cache seeding, entry-count status, and max-entry pruning exist, but fuzzy reuse, hit/miss/latency metrics, frequency-based warming, gradual model migration, and compressed vector storage remain unimplemented.

## What This Changes

Request to improve memory embedding cache with provider batching, fuzzy invalidation, cache warming, metrics, gradual model migration, and vector storage pruning or compression.

## Best Possible Solution

Keep the shipped batch and exact-cache behavior, then split the remaining work into staged memory-cache improvements: observable metrics first, opt-in fuzzy reuse with safeguards, a frequency-based warming policy, and backward-compatible schema changes for model migration and storage format updates.

## Reproduction Assessment

Not applicable as a single bug reproduction; this is a feature/performance request. The current gap is high-confidence from source and tests: exact hash lookup is the only reuse path, status lacks hit/miss/latency fields, and provider/model changes still trigger full reindex.

## Solution Assessment

No proposed patch is under review, and the current partial implementation is not the complete solution requested. The maintainable path is to split the broad request into smaller design-backed changes rather than landing one large cache rewrite.
