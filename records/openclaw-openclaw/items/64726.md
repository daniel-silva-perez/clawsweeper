## Summary

Keep this open. Current main partially satisfies the request with batch embedding, exact hash-based cache reuse, cache seeding during reindex, entry-count status, and max-entry pruning, but it still lacks fuzzy reuse for minor edits, queryable hit/miss/latency metrics, frequency-based cache warming, gradual model migration, and compressed vector storage.

## What This Changes

The issue requests a broader memory embedding cache optimization covering batch embedding, smarter invalidation, cache warming, metrics, model migration, and storage pruning/compression.

## Best Possible Solution

Keep the shipped batch and exact-cache behavior, then split the remaining work into staged memory-cache improvements: add observable cache metrics first, design opt-in fuzzy reuse with safeguards, define a frequency-based warming policy, and handle model migration and vector compression through backward-compatible schema changes.
