## Summary

Keep this open. Current main partially satisfies the request with batch embedding, exact hash-based cache reuse, cache seeding during safe reindex, entry-count status, and max-entry pruning, but it still lacks fuzzy reuse for minor edits, queryable hit/miss/latency metrics, frequency-based warming, gradual model migration, and compressed vector storage.

## What This Changes

The issue requests broader embedding-cache optimization: provider batch embedding, fuzzy invalidation, cache warming, cache metrics, gradual provider/model migration, vector compression, and pruning.

## Best Possible Solution

Keep the shipped batch and exact-cache behavior, then split the remaining work into staged memory-cache improvements: observable metrics first, opt-in fuzzy reuse with safeguards, a frequency-based warming policy, and backward-compatible schema changes for model migration and vector compression.
