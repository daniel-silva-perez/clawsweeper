## Summary

Keep this PR open. Current main still drops Codex project trust when ACPX creates an isolated CODEX_HOME, so the PR is aimed at a real user-visible prompt problem, but the patch is not ready because its line-oriented parser only handles one Codex TOML shape and the branch is dirty against current main.

## What This Changes

The PR copies Codex trusted project declarations and the ACPX workspace cwd into ACPX's isolated Codex config while adding auth-bridge tests that keep auth and notify hooks out.

## Best Possible Solution

Keep the fix owner-local in ACPX, rebase it onto current main, and implement TOML-aware trusted-project extraction that covers explicit project tables, inline entries under [projects], top-level inline projects maps, and single-quoted keys while continuing to exclude auth, notify hooks, and unrelated Codex config.

## Reproduction Assessment

Yes. Source inspection on current main shows the isolated ACPX Codex config is regenerated without reading CODEX_HOME/config.toml, so a trusted project declared in the user's Codex config is not available to the ACPX Codex session.

## Solution Assessment

No, not as written. The owner-local direction is the right narrow fix, but the ad hoc regex parser is incomplete for Codex's supported TOML project-trust forms; using a TOML-aware parser or equivalent complete handling is safer.
