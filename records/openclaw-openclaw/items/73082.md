## Summary

Keep open. Current main has host exec allowlists and the broader `autoAllowSkills` convenience path, but it does not implement the requested per-skill readonly scope or bounded readonly command allowlist; because this changes the exec approval trust model, it needs maintainer/security design rather than cleanup closure.

## What This Changes

Request to let readonly healthcheck/canary skills declare bounded read-only OpenClaw status commands in skill metadata so those commands can be auto-approved without per-command hash approvals.

## Best Possible Solution

Design an additive approval contract for skills or agent config that scopes auto-approval to explicit per-skill argv patterns, keeps outbound/mutating operations gated, updates the Gateway/node/macOS/UI/docs surfaces together, and includes regression tests for allowed readonly commands and denied side-effect commands.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing/types and exec approval schema/allowlist behavior; the reported real-world approval loop would need the reporter's exact workspace skill/session to replay end to end.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact `execScope: readonly` shape needs security/product review because the existing `autoAllowSkills` path is intentionally broad and already flagged as an implicit trust expansion.
