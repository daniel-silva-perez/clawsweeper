## Summary

Keep open. Current main still has host exec allowlists and the broader `autoAllowSkills` convenience path, but no per-skill readonly execution scope or bounded readonly command allowlist; the request changes the approval trust boundary and needs maintainer/security design rather than cleanup closure.

## What This Changes

Request to add `metadata.openclaw.execScope: readonly` and a per-skill `readonlyAllowlist` so healthcheck/canary skills can auto-approve bounded read-only host exec commands while keeping side-effect commands gated.

## Best Possible Solution

Design an additive approval contract that scopes auto-approval to explicit per-skill command patterns, treats workspace and third-party skills according to a clear trust model, keeps outbound/mutating/restart/filesystem-write operations fail-closed, and updates the protocol, runtime, UI/config docs, and tests together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The missing surface is high-confidence statically reproducible on current main by inspecting skill metadata parsing/types and exec approval schema/runtime behavior, while the exact canary incident would need the reporter's workspace session to replay end to end.

## Solution Assessment

Unclear. A per-skill readonly scope is a plausible direction, but the exact `execScope: readonly` metadata shape is not yet proven to be the safest contract; maintainers should compare it against an explicit agent/config allowlist or trusted-skill-only design before implementation.
