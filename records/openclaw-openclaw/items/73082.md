## Summary

Keep open. Current main still lacks a per-skill readonly exec scope or bounded readonly command allowlist, and the requested behavior would change host exec approval semantics enough that it needs maintainer/security design rather than conservative cleanup closure.

## What This Changes

Request a `metadata.openclaw.execScope: readonly` skill metadata contract with per-skill readonly command allowlists so canary/healthcheck skills can auto-approve declared read-only status commands while keeping side-effecting commands approval-gated.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit argv/pattern rules, keeps outbound delivery, filesystem writes, service restarts, cron mutations, and other side effects fail-closed, and updates protocol/config/UI/docs/tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing, exec approval schema, docs, and node-host allowlist behavior; replaying the exact WSL2 canary loop is not needed to confirm the requested API is absent.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact `execScope: readonly` contract should be settled by maintainers/security because current `autoAllowSkills` is intentionally broader and documented as implicit trust.
