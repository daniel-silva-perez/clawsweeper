## Summary

Keep open. Current main still lacks a per-skill readonly exec scope or per-skill readonly command allowlist; the only current auto-approval path is the broader agent-level `autoAllowSkills` trust for known skill CLIs, which the docs describe as an implicit convenience allowlist. The existing discussion already flagged this as maintainer/security design work, and the current fd0ca598 inspection still supports that conclusion.

## What This Changes

Request adds `metadata.openclaw.execScope: readonly` plus a bounded per-skill readonly command allowlist so healthcheck/canary skills can run approved read-only OpenClaw status commands without repeated exec prompts.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit argv/pattern rules, keeps outbound delivery, filesystem writes, service restarts, cron mutations, and other side effects fail-closed, and updates protocol/config/UI/docs/tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing, exec approval schema, docs, and node-host allowlist behavior; replaying the exact WSL2 canary loop is not needed to confirm the requested API is absent.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact `execScope` contract should be settled by maintainers/security because the current `autoAllowSkills` path is intentionally broader and documented as implicit trust rather than per-skill readonly approval.
