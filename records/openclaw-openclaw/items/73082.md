## Summary

Keep open. Current main still lacks the requested per-skill readonly exec scope and bounded readonly command allowlist, and the request changes host exec approval semantics enough that it needs maintainer/security design rather than cleanup closure.

## What This Changes

The issue requests a per-skill `readonly` exec approval scope and command allowlist for healthcheck/canary skills so known introspection commands can run without repeated approval prompts.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to explicit per-skill argv patterns and trusted skill identity, keeps outbound and mutating operations gated, updates Gateway/node/macOS/UI/docs surfaces together, and adds regression coverage for allowed readonly commands plus denied side-effect commands.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing/types and exec approval schema/allowlist behavior; replaying the exact WSL2 canary loop would require the reporter's workspace skill/session.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact `execScope: readonly` shape should be settled by maintainers/security because existing `autoAllowSkills` is intentionally broader and documented as implicit trust.
