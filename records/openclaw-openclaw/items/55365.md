## Summary

Keep this issue open. Current main still validates browser navigation URL shape, protocol, proxy mode, and SSRF policy, but it does not reject http(s) URLs with URL userinfo before forwarding the original URL into Chrome MCP, Playwright, or tab creation. The prior ClawSweeper comment remains accurate on current main, and related PR #51822 covers the same hardening pattern for cron webhooks rather than the browser navigation surface.

## Best Possible Solution

Add a narrow browser-navigation guard in extensions/browser/src/browser/navigation-guard.ts after URL parsing: for http: and https: URLs, reject when parsed.username or parsed.password is present, surface only a redacted URL, and point users to openclaw browser set credentials, cookies, or headers. Cover assertBrowserNavigationAllowed plus /navigate and /tabs/open behavior with regression tests, and preserve existing CDP endpoint userinfo support.

## Risks / Open Questions

- Credential-bearing page-navigation URLs can still reach Playwright, Chrome MCP, or CDP tab creation and may be echoed by downstream browser/runtime errors before OpenClaw can redact them.
- The fix needs to distinguish browser page navigation from remote CDP endpoint URLs, where userinfo support is intentional and already tested.
