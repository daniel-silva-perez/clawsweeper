## Summary

Keep this issue open. Current main still has the same browser-navigation gap described in the existing ClawSweeper comment: URL shape/protocol/proxy/SSRF checks run, but http(s) URL userinfo is not rejected before the original URL is handed to Chrome MCP, Playwright, or CDP tab creation. Related PR #51822 is useful prior art for cron webhooks, but it is a separate open PR and does not cover the browser tool surface.

## Best Possible Solution

Add a narrow browser-navigation guard after URL parsing in extensions/browser/src/browser/navigation-guard.ts: for http: and https: URLs, reject when parsed.username or parsed.password is present, use a redacted URL in the InvalidBrowserNavigationUrlError message, and direct users to `openclaw browser set credentials`, cookies, or headers. Add regression coverage for assertBrowserNavigationAllowed, /navigate, /tabs/open, and Playwright create/navigate paths, while preserving the existing CDP endpoint userinfo behavior and tests.

## Risks / Open Questions

- Credential-bearing page-navigation URLs can still reach downstream browser runtimes and may be echoed by Chromium, Playwright, Chrome MCP, CDP, or route-level errors before OpenClaw can redact them.
- A fix must distinguish page-navigation URLs from CDP endpoint URLs, where userinfo support is intentional and already covered by tests.
