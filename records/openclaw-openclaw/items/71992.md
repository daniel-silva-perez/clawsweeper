## Summary

Keep open. Current main still has the Control UI WebChat final/aborted ordering that can expose one assistant reply through both committed history and the live stream, and PR #72004 is already open as the direct closing implementation candidate.

## What This Changes

The issue asks to fix Control UI WebChat so assistant replies render once by clearing live stream state before committing final or aborted assistant messages to chat history.

## Best Possible Solution

Land PR #72004 or an equivalent narrow change that snapshots any fallback stream text, clears `chatStream`, `chatRunId`, and `chatStreamStartedAt`, then mutates `chatMessages` in both final and aborted paths with an order-sensitive regression test. Keep backend transcript idempotency and delivery-mirror cleanup separate from this client render race.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: process an active-run final or aborted chat event while `chatStream` is non-null, and current main can render both the committed assistant message and the still-live stream item.

## Solution Assessment

Yes for the focused client-side symptom. The requested clear-before-append ordering change is the narrowest maintainable fix; broad backend dedupe should not be folded into this issue except as separately tracked follow-up work.
