## Summary

Keep open. Current main still has the Control UI terminal-event ordering that can expose a committed assistant message and the still-live stream at the same time, and open PR #72004 is already the direct closing implementation candidate for the focused client-side race.

## What This Changes

The issue requests a Control UI WebChat fix so final and aborted assistant events clear live stream state before appending committed assistant messages, preventing duplicate visible replies.

## Best Possible Solution

Review and land PR #72004, or an equivalent narrow change, so final and aborted active-run events snapshot any fallback stream text, clear `chatStream`, `chatRunId`, and `chatStreamStartedAt`, then mutate `chatMessages`, with an order-sensitive regression test. Keep persisted transcript idempotency under the broader duplicate-transcript tracker.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: process an active-run `final` or `aborted` chat event while `chatStream` is non-null, and current main can present both the committed assistant message and the live stream item to the renderer.

## Solution Assessment

Yes for the focused client-side race. The clear-before-append ordering change is the narrowest maintainable fix for this UI symptom; broader backend transcript dedupe should remain separate under #69208 and related PR work.
