## Summary

Keep open. Current main still has the Control UI/WebChat terminal-event ordering that can render committed history and the live stream simultaneously, the latest stable release has the same ordering, and PR #72004 is already open as the direct closing implementation candidate.

## What This Changes

The issue requests that Control UI WebChat render each assistant reply once by clearing live chat stream/run state before committing final or aborted assistant messages.

## Best Possible Solution

Land PR #72004 or an equivalent narrow Control UI change that snapshots fallback stream text, clears chatStream/chatRunId/chatStreamStartedAt before mutating chatMessages in final and aborted paths, and adds an order-sensitive regression test. Keep backend transcript dedupe and delivery-mirror cleanup tracked separately under the broader duplicate-transcript work.

## Reproduction Assessment

Yes. The issue includes concrete browser steps across multiple reporters, and the source-level reproduction is high confidence: process an active-run final or aborted event while chatStream is non-null and current main can expose both the committed assistant message and the still-live stream item to renderChat.

## Solution Assessment

Yes for the focused client-side race. The clear-before-append ordering change is the narrowest maintainable fix for the visible duplicate render; broad backend dedupe is a separate solution path for the adjacent transcript reports.
