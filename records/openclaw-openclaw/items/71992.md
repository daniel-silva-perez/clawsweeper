## Summary

Keep open. Current main and the latest release still have the Control UI terminal-event ordering that can expose both the committed assistant message and the live stream item to rendering, and PR #72004 is already open with closing syntax as the direct implementation candidate.

## What This Changes

The issue reports duplicated Control UI WebChat assistant replies and asks for the final/aborted chat event handlers to clear live stream state before committing assistant messages.

## Best Possible Solution

Land PR #72004 or an equivalent narrow Control UI change that snapshots fallback stream text, clears chatStream/chatRunId/chatStreamStartedAt before mutating chatMessages in final and aborted paths, and adds an order-sensitive regression test. Keep backend transcript and delivery-mirror dedupe tracked separately under the broader duplicate-transcript work.

## Reproduction Assessment

Yes. The issue has concrete browser reports across versions/platforms, and the source-level path is clear: process an active-run final or aborted event while chatStream is populated and current main can render both committed history and the live stream item.

## Solution Assessment

Yes for the client-side duplicate-render race. The clear-before-append ordering change is the narrowest maintainable fix for this surface; broad backend dedupe would be a separate fix for adjacent transcript persistence reports.
