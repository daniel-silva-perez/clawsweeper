## Summary

Keep open. Current main still has the Control UI terminal-event ordering that can expose the same assistant reply through committed history and the live stream at the same time, and PR #72004 is already open as the direct closing implementation candidate.

## What This Changes

The issue requests fixing duplicate Control UI webchat assistant replies by clearing live stream state before committing terminal assistant messages, while keeping broader transcript-duplication work separate.

## Best Possible Solution

Land PR #72004 or an equivalent narrow change that snapshots fallback stream text, clears `chatStream`, `chatRunId`, and `chatStreamStartedAt`, then mutates `chatMessages` in both final and aborted paths, with an order-sensitive regression test. Keep persisted transcript idempotency under #69208.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: process an active-run final or aborted chat event while `chatStream` is non-null, and current main mutates `chatMessages` before clearing the stream that `buildChatItems` can still render.

## Solution Assessment

Yes for the focused client-side race. The requested ordering change is the narrowest maintainable fix for this UI symptom, while server-side duplicate transcript variants should remain separate under the umbrella issue.
