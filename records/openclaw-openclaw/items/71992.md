## Summary

Keep open. Current main still has the Control UI/WebChat terminal-event ordering that can expose the same assistant reply through committed history and the live stream at the same time, and PR #72004 is already open as the direct closing implementation candidate for https://github.com/openclaw/openclaw/issues/71992.

## What This Changes

Fix Control UI/WebChat so each assistant final or aborted reply renders once by clearing live stream state before committing the assistant message.

## Best Possible Solution

Land PR #72004 or an equivalent narrow change that snapshots any fallback stream text, clears `chatStream`, `chatRunId`, and `chatStreamStartedAt`, then mutates `chatMessages` in both final and aborted paths, with an order-sensitive regression test. Keep persisted transcript idempotency and delivery-mirror cleanup under the broader duplicate-transcript tracker.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: process an active-run final or aborted chat event while `chatStream` is non-null, and current main can render both the committed assistant message and the still-live stream item.

## Solution Assessment

Yes for the focused client-side race. The clear-before-append ordering change is the narrowest maintainable fix for this UI symptom; broad backend dedupe should not be folded into this issue except as separate #69208 follow-up work.
