## Summary

Keep open. Current main still mutates Control UI `chatMessages` before clearing live stream state on final and aborted chat events, so the reported duplicate-render window is still present; the linked PR #72004 is open and directly targets this issue with the requested narrow ordering fix.

## What This Changes

The issue requests fixing duplicate Control UI webchat assistant replies by clearing live stream state before committing terminal assistant messages in the final and aborted event paths.

## Best Possible Solution

Review and land PR #72004 or an equivalent narrow Control UI change that snapshots fallback stream text, clears `chatStream`, `chatRunId`, and `chatStreamStartedAt`, then mutates `chatMessages` in both final and aborted paths with an order-sensitive regression test.

## Reproduction Assessment

Yes. The source-level reproduction path is clear: process an active-run final or aborted chat event while `chatStream` is non-null, and current main mutates `chatMessages` before clearing the live stream that `buildChatItems` can still render.

## Solution Assessment

Yes for the focused client-side race. The requested ordering change is the narrowest maintainable fix for this visible UI symptom, while server-side transcript idempotency should remain separate under the broader duplicate-transcript tracker.
