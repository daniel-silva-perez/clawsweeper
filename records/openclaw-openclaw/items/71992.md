## Summary

Keep open. Current main still has the Control UI terminal-event ordering that can expose both the committed assistant message and the live stream item to the renderer, and open PR #72004 is already the direct implementation candidate with closing syntax. The comments also include cross-platform reproduction evidence, while the server-side transcript-duplication observations appear to belong with the broader #69208 umbrella rather than replacing this narrow client-side fix.

## What This Changes

The issue reports duplicate Control UI webchat assistant replies and asks for the final/aborted chat event handlers to clear live stream state before committing assistant messages.

## Best Possible Solution

Land PR #72004 or an equivalent narrow Control UI change that snapshots fallback stream text, clears chatStream/chatRunId/chatStreamStartedAt before mutating chatMessages in final and aborted paths, and adds an order-sensitive regression test. Keep backend transcript and delivery-mirror dedupe work separate under the existing duplicate-transcript umbrella.

## Reproduction Assessment

Yes. The issue has concrete browser reports across versions and platforms, and current source shows a reproducible state path: process an active-run final or aborted event while chatStream is populated, then the renderer can see both chatMessages and chatStream before the stream is cleared.

## Solution Assessment

Yes for the client-side duplicate-render race. The clear-before-append ordering change is the narrowest maintainable fix for this surface; broad backend transcript dedupe would address a different root cause family.
