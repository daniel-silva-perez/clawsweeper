## Summary

Keep this PR open. Current main still uses Feishu open_id-only bot mention matching, while this PR addresses the right reported problem but is incomplete and stale against the current Feishu monitor/message refactor.

## What This Changes

The PR threads a Feishu bot union_id through probe results, monitor identity/state, account handling, and message parsing so @mention detection can fall back from open_id to union_id in multi-bot groups.

## Best Possible Solution

Land a refreshed Feishu fix that uses a verified stable bot identity or documented cross-app mention-resolution strategy, then carries it through startup, recovery state, direct parsing, mention stripping, mention-forward extraction, debounce merging, and focused Feishu regression tests.

## Reproduction Assessment

Yes. The paired report gives concrete multi-bot Feishu group steps, and current main has an inspectable open_id-only requireMention path that will drop a synthetic event whose mention open_id does not match the bot's stored open_id.

## Solution Assessment

No. The union_id fallback direction is plausible, but this patch is not the best mergeable solution until the identity source is verified and the fallback is applied to all mention-derived paths, not only the final checkBotMentioned call.

## Review Findings

  - body: The new identity field is returned from fetchBotIdentityForMonitor, but the normal multi-account preflight still passes only botOpenId and botName into monitorSingleAccount. In that startup path the union-id map stays empty, so the fallback never runs.
  - confidence: 0.91

  - body: This reads bot?.union_id without a typed or tested response contract, while the paired report says the bot info API does not return union_id. If the field is absent, the new fallback is inert in the exact scenario being fixed.
  - confidence: 0.9
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After mentionedBot can become true via union_id, normalization still strips only a mention whose open_id equals botOpenId. In the reported mismatch case, @Bot /model can keep the mention placeholder and miss slash-command parsing.
  - confidence: 0.9
- **[P2] Pass fallback identity into debounce mention resolution:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives botUnionId, but the debounce path still resolves text and merged mentions with botOpenId-only parsing/filtering. Batched messages can still lose the bot mention and be dropped under requireMention.
  - confidence: 0.86
