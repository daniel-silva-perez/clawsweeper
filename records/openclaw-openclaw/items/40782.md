## Summary

Keep open. Current main still has the reported Feishu multi-bot mention gap, but this PR is not merge-ready: the branch is stale, relies on an unresolved bot identity source, and leaves related mention parsing paths incomplete while also regressing current @all handling.

## What This Changes

The PR threads a Feishu bot `union_id` through probe, startup, monitor state, and message parsing so bot mention detection can fall back when `open_id` does not match.

## Best Possible Solution

Refresh or replace the PR after verifying the Feishu/Lark stable bot identity contract. The final fix should carry the chosen identifier through probe results, monitor identity state, message parsing, mention normalization, mention-forward extraction, debounce merging, and sequential command parsing, with regression coverage for mismatched mention `open_id` plus matching stable identity.

  - body: This returns true whenever the raw content contains `@_all`, but current Feishu behavior documents and tests that broadcast-only `@all`/`@_all` are not bot mentions. With `requireMention` enabled, merging this would make every bot respond to a broadcast mention.

  - confidence: 0.96
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:790`
  - body: After this new call can set `mentionedBot` via `union_id`, content normalization still strips only a mention whose `open_id` equals `botOpenId`. In the multi-bot mismatch case, `@Bot /model` can keep the bot mention placeholder in the text, so slash-command detection no longer sees the leading `/`.
  - confidence: 0.9
- **[P2] Carry union_id through debounced merges:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This adds `botUnionId` to the normal message dispatch, but the debounce path still parses text and resolves merged mentions using only `botOpenId`. A burst where the bot mention is present only by matching `union_id` can still lose the mention signal and be dropped under `requireMention`.
  - confidence: 0.86
