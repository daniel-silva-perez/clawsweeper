## Summary

Keep open. Current main still has the reported Feishu multi-bot mention bug, but this PR is not safely mergeable as-is: the fallback identity source is unproven, adjacent mention paths remain open_id-only, and the branch is dirty against the current Feishu refactor.

## What This Changes

The PR threads a Feishu bot union_id from probe results through monitor state and account handling into message parsing so @mention detection can fall back from open_id to union_id.

## Best Possible Solution

Land a refreshed Feishu-owned fix that first verifies the stable bot identity or cross-app open_id mapping strategy, then applies it consistently across mention detection, bot-mention stripping, mention-forward targets, debounce preservation, monitor startup/recovery state, and focused regression tests.

## Reproduction Assessment

Yes. The paired report gives concrete multi-bot Feishu group steps, and current main has a synthetic reproduction path: send a group event whose mention.open_id differs from botOpenId under requireMention and mentionedBot remains false, causing the message to be dropped.

## Solution Assessment

No. The union_id fallback direction is plausible only after the identity source is proven, and this patch leaves several paths open_id-only; a safer fix should verify the Feishu API contract first and update all mention consumers together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

  - body: The patch reads bot?.union_id from the bot-info response, but the paired report says that endpoint does not return union_id and current main's probe contract exposes only botID/botName. If this field is absent, the fallback is inert for the reported multi-bot case.
  - confidence: 0.9
- **[P2] Propagate fallback identity through preflight:** `extensions/feishu/src/monitor.account.ts:510`
  - body: monitorSingleAccount can accept a prefetched botUnionId, but the normal multi-account startup preflight still passes only botOpenId and botName. That leaves the union-id state empty on the primary startup path, so the fallback will not run there.
  - confidence: 0.88
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:791`
  - body: After mentionedBot can become true through union_id, normalization still strips only the mention whose open_id equals botOpenId. In the mismatch case, @Bot /model can keep the bot placeholder and miss slash-command parsing.
  - confidence: 0.9
- **[P2] Carry fallback identity through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives botUnionId, but the debounce path still builds merged text and mention preservation with botOpenId-only parsing and filtering. Batched messages can still lose the bot mention and be dropped under requireMention.
  - confidence: 0.86
