## Summary

Keep open. Current main still does not implement stable Feishu multi-bot mention matching, the paired same-author bug report remains open, and the PR branch is a useful but stale and incomplete implementation candidate with blocking correctness issues.

## What This Changes

The PR threads a Feishu bot `union_id` from probe/startup state into message handling so mention detection can fall back from `open_id` in multi-bot groups.

## Best Possible Solution

Refresh the fix against current main after verifying the Feishu/Lark stable bot identity contract. The implementation should carry the chosen stable identifier through probe results, monitor identity state, `bot-content.ts`, `parseFeishuMessageEvent`, mention normalization, mention-forward extraction, debounce merge/filtering, and sequential-key command parsing, with focused tests for mismatched mention `open_id` plus matching stable identifier.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Add union_id to the bot info response type:** `extensions/feishu/src/probe.ts:137`
  - body: `bot` is typed as only `bot_name` and `open_id`, but this line reads `bot?.union_id`. In strict TypeScript this property access will fail, so the response type needs to include the field or the value needs to be read through a validated/narrowed shape.
  - confidence: 0.96
- **[P2] Strip union-id matched bot mentions:** `extensions/feishu/src/bot.ts:790-797`
  - body: After `mentionedBot` can become true via `union_id`, the content normalization still strips only a mention whose `open_id` equals `botOpenId`. In the multi-bot mismatch case, `@Bot /help` can keep the bot mention in the text, so slash-command detection no longer sees the leading `/`.
  - confidence: 0.9
- **[P2] Propagate union-id matching into debounce:** `extensions/feishu/src/monitor.account.ts:351-354`
  - body: The debounced path still parses text and filters merged mentions with only `botOpenId`. A burst where the bot mention is present only by matching `union_id` can still dispatch without a recognized bot mention and be dropped when `requireMention` is enabled.
  - confidence: 0.86
