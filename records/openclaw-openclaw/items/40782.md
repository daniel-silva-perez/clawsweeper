## Summary

This PR still addresses a real Feishu multi-bot mention bug that is not implemented on current main, but the proposed patch is incomplete and should not be merged as-is. The current Feishu mention gate is still based on exact `open_id` matching, and the PR leaves startup, typing, normalization, and debounced mention paths inconsistent for the same multi-bot scenario.

## What This Changes

The PR threads a Feishu bot `union_id` through probe, startup identity, monitor state, and message parsing so `@mention` detection can fall back from app-scoped `open_id` matching in multi-bot groups.

## Best Possible Solution

Keep the PR open as useful evidence for the canonical Feishu multi-bot bug, but refresh or replace it with a current-main fix after verifying the Feishu stable bot identity source. The final patch should carry the chosen identifier through probe/startup, monitor state, message parsing, bot mention stripping, mention-forward extraction, debounced merge handling, and sequential command parsing with focused regression tests.

  - confidence: 0.96
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After `mentionedBot` can be true via `union_id`, normalization still strips only a mention whose `open_id` equals `botOpenId`. In the reported mismatch case, `@Bot /model` can keep the mention placeholder and fail slash-command detection.
  - confidence: 0.9
- **[P2] Carry union_id through debounced merges:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives `botUnionId`, but the debounce path still parses and resolves merged mentions using open-id-only logic. A burst where the bot mention matches only by stable identity can still lose the mention signal and be dropped under `requireMention`.
  - confidence: 0.86
