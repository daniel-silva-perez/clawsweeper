## Summary

Keep open. Current main still has open_id-only Feishu mention gating, so the underlying multi-bot bug is not fixed on main, but this PR is not mergeable as-is because the fallback identity source is unverified and the new identity is not applied consistently across startup, normalization, and debounce paths.

## What This Changes

This PR threads a Feishu bot union_id from probe results through monitor state and account handling into message parsing so group @mention detection can fall back from open_id to union_id.

## Best Possible Solution

Land a refreshed Feishu fix that first verifies a stable bot identity or documented cross-app mention-resolution strategy, then applies that identity consistently to mention detection, bot-mention stripping, mention-forward extraction, debounce merging, startup/recovery state, and focused Feishu regression tests.

## Reproduction Assessment

Yes. The paired report provides concrete multi-bot Feishu group steps, and current main can be checked synthetically by sending a group event whose mention.open_id differs from botOpenId: mentionedBot remains false and requireMention drops the message.

## Solution Assessment

No. The union_id fallback direction is plausible, but this patch is not the best mergeable solution until the identity source is proven and every mention path uses the same fallback semantics.
