## Summary

Keep this PR open. Current main still lacks the proposed post-publish macOS asset verifier and root appcast CODEOWNERS protection, while the release docs and current macOS handoff explicitly require GitHub release assets plus an updated appcast for stable macOS readiness. The branch is relevant, but it has two concrete workflow reliability bugs that should be fixed before merge and still needs release-owner policy review.

## What This Changes

The PR adds a new GitHub Actions workflow that polls stable GitHub releases for three macOS assets and warns on missing appcast advancement, plus a CODEOWNERS rule for `/appcast.xml`.

## Best Possible Solution

Land a release-owner-approved guard, whether this branch or an equivalent, that verifies stable macOS release assets and appcast consistency without masking infrastructure failures, and keep it tied to the broader updater reliability tracker #48306.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fail real API errors instead of counting them as zero assets:** `.github/workflows/verify-release-assets.yml:53-54`
  - body: The `|| echo "0"` fallback makes every `gh api` failure look like a release with no assets. A 403, auth problem, rate limit, or GitHub outage would burn the full 45-minute loop and then report the wrong incident cause, so only genuinely retryable not-ready cases should continue polling while non-retryable API errors fail with their real message.
  - confidence: 0.91
- **[P2] Recheck assets before declaring the timeout:** `.github/workflows/verify-release-assets.yml:60-62`
  - body: The loop sleeps after the 45th poll and then fails without another API call. If the private publish uploads the assets during that final minute, the workflow still reports a missing-asset failure even though the assets arrived within the intended waiting window.
  - confidence: 0.88
