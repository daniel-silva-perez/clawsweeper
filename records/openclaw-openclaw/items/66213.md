## Summary

Keep this PR open. Current main still lacks the post-publish macOS asset verifier and the `/appcast.xml` CODEOWNERS rule, and the reported `v2026.4.12` zero-asset release gap is substantiated; however, the PR's polling loop still masks real GitHub API failures and can false-fail at the intended timeout, so it needs targeted fixes plus release-owner review before merge.

## What This Changes

This PR adds a GitHub Actions workflow that checks stable releases for at least three macOS assets and warns if `appcast.xml` is not updated, plus a CODEOWNERS rule for `/appcast.xml`.

## Best Possible Solution

Land a release-owner-approved guard, from this branch or an equivalent patch, that verifies stable macOS release assets and appcast consistency without masking infrastructure failures or reporting false missing-asset timeouts, while keeping the broader updater reliability work tied to #48306.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fail real API errors instead of counting them as zero assets:** `.github/workflows/verify-release-assets.yml:53-54`
  - body: The `|| echo "0"` fallback makes every `gh api` failure look like a release with no assets. A 403, rate limit, auth problem, or GitHub outage would burn the full polling window and report the wrong incident cause, so non-successful API calls should surface their real failure instead of being counted as zero assets.
  - confidence: 0.91
- **[P2] Recheck assets before declaring the timeout:** `.github/workflows/verify-release-assets.yml:60-62`
  - body: The loop sleeps after the 45th poll and then fails without another API call. If the private publish uploads the assets during that final minute, the workflow still reports missing assets even though they arrived within the intended waiting window.
  - confidence: 0.88
