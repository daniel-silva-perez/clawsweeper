## Summary

Keep this open. Current main still indexes memory through the fixed token/overlap chunker, has no heading-aware or section-aware chunking config, and persists only token/overlap metadata for reindex decisions; the broader entity-extraction and graph-memory pieces also remain separate, unshipped product scope.

## What This Changes

Request heading-aware or section-aware memory chunking to keep distinct Markdown sections from sharing embedding chunks, with stretch follow-ups for entity extraction, graph memory, reranking, and nightly consolidation.

## Best Possible Solution

Land a maintained heading-aware or section-aware strategy in the shared memory-host indexing path used by memory-core, expose and document the config, persist strategy-aware metadata so strategy changes trigger reindexing, and split entity extraction, graph memory, cross-encoder reranking, and nightly consolidation into separately scoped memory-wiki/QMD follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with default chunking, several short ## sections whose combined estimated size stays under the token-derived character budget remain in one chunk because current main flushes on size, not heading boundaries.

## Solution Assessment

Partly. Heading-aware chunking is a narrow maintainable fix for the blended-vector failure, but the broader entity/graph/reranking scope and the large multi-strategy PR need maintainer product and architecture review before choosing the final implementation shape.
