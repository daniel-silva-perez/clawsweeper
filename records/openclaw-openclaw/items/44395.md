## Summary

Keep this issue open. Current main still uses fixed token/character-window memory chunking, exposes only `tokens` and `overlap` in the memory-search config surface, and records only those fields for reindex decisions. The issue’s heading-aware chunking request and the related entity-extraction discussion remain unshipped; the provided GitHub context shows related PR #68019 is still open/unmerged, while earlier focused attempts were closed unmerged.

## Best Possible Solution

Keep this issue open. The maintainer path is to review #68019 or request a narrower PR that adds a documented heading/section-aware chunking strategy in the memory-host SDK path used by memory-core, exposes it through config types/schema/help/docs, persists the effective strategy/config for reindex invalidation, and adds tests proving adjacent `##`/`###` sections do not blend into one chunk. Track entity extraction, graph memory, reranking, and consolidation as separate follow-ups unless maintainers intentionally keep this issue as the umbrella.
