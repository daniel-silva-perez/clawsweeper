## Summary

Keep this open. Current main still uses fixed token/overlap memory chunking without a heading-aware strategy or strategy-aware reindex metadata, and the broader entity-extraction/graph-memory portion remains unshipped and needs separate scoping.

## What This Changes

Request heading-aware or section-aware chunking for Markdown memory files, with stretch goals for entity extraction, reranking, graph memory, and nightly consolidation.

## Best Possible Solution

Land a maintained heading-aware or section-aware strategy in the shared memory-host indexing path used by memory-core, expose and document the config, persist strategy-aware metadata so strategy changes trigger reindexing, and track entity extraction, graph memory, cross-encoder reranking, and nightly consolidation as separate memory-wiki/QMD/product follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with default chunking, several short ## sections whose combined estimated size stays under the token-derived character budget remain in one chunk because the current chunker flushes on size, not heading boundaries.

## Solution Assessment

Partly. Heading-aware chunking is a narrow maintainable fix for the reported blended-vector problem, but bundling it with multi-strategy chunking or graph-memory/entity extraction needs maintainer review and likely separate follow-up scope.
