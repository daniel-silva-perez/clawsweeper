## Summary

Keep this open. Current main still indexes memory Markdown through fixed token/overlap chunking with no heading-aware strategy or strategy-aware reindex metadata; #68019 is open and unmerged for the chunking portion, and the entity-extraction/graph-memory stretch remains unshipped and broader than this narrow fix.

## What This Changes

Request heading-aware or section-aware memory chunking for Markdown memory files, with a broader stretch goal around entity extraction, reranking, graph memory, and nightly consolidation.

## Best Possible Solution

Land a maintained heading-aware or section-aware strategy in the shared memory-host indexing path used by memory-core, expose and document the config, persist strategy-aware metadata so reindexing happens on strategy changes, and track entity extraction, graph memory, cross-encoder reranking, and nightly consolidation as separate memory-wiki/QMD/product follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with default chunking, several short `##` sections whose combined estimated size stays under the token-derived character budget remain in one chunk because the current chunker flushes on size, not heading boundaries.

## Solution Assessment

Partly. Heading-aware chunking is a narrow maintainable fix for the reported blended-vector problem; bundling it with six chunking strategies or graph-memory/entity extraction needs maintainer review and likely separate follow-up scope.
