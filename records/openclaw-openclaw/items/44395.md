## Summary

Keep this issue open. Current main still chunks memory with a fixed token/character window, exposes no heading/section-aware chunking strategy in the public config surface, and only tracks token/overlap values for reindex decisions. The related multi-strategy chunking PR is still open, so the requested heading-aware behavior and broader entity-extraction discussion remain unshipped.

## What This Changes

The issue requests heading-aware or section-aware markdown chunking for memory search, with broader follow-up ideas around entity extraction, graph memory, reranking, and nightly consolidation.

## Best Possible Solution

Keep the issue open until a maintained PR lands a documented heading/section-aware chunking strategy in the memory-host path used by memory-core, exposes the strategy through config types/schema/help/docs, persists effective strategy/config for reindex invalidation, and adds tests proving adjacent `##`/`###` sections do not blend into one chunk. Treat entity extraction, graph memory, cross-encoder reranking, and nightly consolidation as separate follow-ups unless maintainers intentionally keep this as an umbrella.
