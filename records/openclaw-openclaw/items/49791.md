## Summary

Keep open. Current main still implements the group-command authorization behavior described in the discussion: Feishu group slash/directive commands are authorized from per-group `allowFrom` or top-level `allowFrom`, and unauthorized text command-only/directive-only messages are silently ignored rather than producing an explicit group rejection. The reporter’s workaround confirms configuration can make `/think` work, but the remaining UX request and later group no-response report are not fully resolved by a merged PR.

## Best Possible Solution

Keep this issue open and implement a focused Feishu/group text-command UX fix: when an explicitly mentioned group text command or directive-only message such as `/think medium` is blocked by command authorization, return a visible, non-sensitive rejection that points to `allowFrom` or per-group sender authorization. Add regression coverage for both unauthorized and authorized `@Bot /think medium` group paths, and verify the later `allowall` report against the separate explicit-group/legacy-alias fixes before closing.
