## Summary

Keep open. Current main still has the reported SQL identifier interpolation in the active memory-host SDK package, so the PR addresses a real security-sensitive code path, but the branch is not merge-ready against current main because it edits a source bridge that is now package-owned and its table quoting contract can regress valid quoted/dotted SQLite identifiers.

## What This Changes

The PR adds SQL identifier quoting helpers to the memory schema helper and applies them to cache table, FTS table, and ALTER TABLE column/table interpolation in both the package file and the older source-side mirror file.

## Best Possible Solution

Land a revised security fix on current main that changes the package-owned memory schema implementation, leaves `src/memory-host-sdk/host/memory-schema.ts` as a thin bridge, defines the accepted SQLite identifier forms, quotes or validates identifiers consistently, and adds focused regression tests for malicious identifiers, schema-qualified names, and the chosen behavior for quoted dotted names.

## Reproduction Assessment

Yes. A high-confidence source-level and in-memory SQLite reproduction exists: pass a crafted table identifier such as `embedding_cache (x TEXT); DROP TABLE files; --` into the current helper's DDL path and SQLite executes the injected statement through `db.exec`.

## Solution Assessment

No, not as-is. The quoting direction is appropriate, but the current PR needs to be updated for the package/bridge split and should either preserve or explicitly reject quoted dotted identifiers with tests before merge.
