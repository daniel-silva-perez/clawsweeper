## Summary

Keep this PR open. Current main still lacks the PR's compatibility path for numeric Telegram targets accidentally stored in `delivery.channel`, but the proposed runtime inference turns that value into an explicit target and skips the existing implicit `allowFrom` protection, so it needs maintainer/security review before merge.

## What This Changes

The PR changes cron delivery resolution to infer the sole configured announce channel when `delivery.channel` is not a real channel, moves that raw value into `delivery.to`, adds resolver regressions for numeric targets, and includes unrelated macOS SwiftFormat-only hunks.

## Best Possible Solution

Keep the useful regression intent, but land a narrower cron-delivery fix that preserves the existing target authorization semantics. A safe end state would either repair legacy misplaced `delivery.channel` values through a doctor/migration path or mark runtime-inferred targets separately and apply the same `allowFrom` protections and ambiguity tests that implicit delivery already uses.

## Reproduction Assessment

Yes. The issue path is reproducible by resolving a cron delivery payload with one configured announce channel, `channel: "834732674"`, and no `to`; current main treats the numeric value as the channel, while the PR's new test demonstrates the intended recovery path.

## Solution Assessment

No, not as written. The PR is focused, but moving a raw non-channel value into `explicitTo` is broader than the current implicit fallback semantics; the safer alternative is to preserve or explicitly reapply `allowFrom` policy for inferred targets, or repair legacy config outside runtime delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve allowFrom for inferred targets:** `src/cron/isolated-agent/delivery-target.ts:104`
  - body: This assignment makes a raw `delivery.channel` value become `explicitTo`, which causes `resolveSessionDeliveryTarget` to return `mode: "explicit"`. `resolveDeliveryTarget` only applies configured/store `allowFrom` in implicit mode, so a recovered legacy value can be delivered to a recipient that implicit delivery would have rerouted or blocked. Track the inference separately and apply the same allowFrom policy, or keep this recovery out of runtime delivery.
  - confidence: 0.86
