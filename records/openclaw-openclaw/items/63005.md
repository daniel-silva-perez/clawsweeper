## Summary

Keep open. Current main still preserves arbitrary explicit inbound SessionKey values as raw normalized keys, while gateway/session-tool resolution canonicalizes bare non-agent keys to agent-scoped store keys and can prune the raw alias. The related #49202 fix/closure is narrower than this report because it covered Open WebUI UUID-style keys, not arbitrary structured external keys such as conversation:pair:... .

## Best Possible Solution

Keep this issue open and implement a generic session-key continuity fix. The likely maintainer path is to make explicit inbound SessionKey initialization and gateway resolveSessionStoreKey/resolveGatewaySessionStoreTarget share one store-key contract, or preserve durable raw aliases for external keys that future inbound traffic will continue to use. Add a regression test with a key such as conversation:pair:user_a::user_b:space:space_123, run a sessions.resolve or sessions_send/nested-agent path, then verify a later inbound message with the same raw key reuses the original sessionId and transcript.
