## Summary

Keep this open. Current main still has a concrete raw-versus-canonical session-key contract mismatch: explicit inbound SessionKey values can remain as raw external keys, while gateway/session-tool paths resolve the same bare key to an agent-prefixed canonical key and can migrate/prune the raw row, so the next inbound raw-key turn can miss the existing session entry.

## What This Changes

Preserve continuity for stable non-agent-prefixed external SessionKey values after sessions_send or nested agent activity so later inbound messages reuse the existing session and transcript.

## Best Possible Solution

Add a generic session-store key contract so inbound explicit SessionKey handling and gateway/session-tool resolution agree for external raw keys, or keep durable raw aliases when future inbound traffic is expected to continue using those keys. The fix should include a regression test with a conversation:pair-style key that survives sessions.resolve or sessions_send activity without rotating sessionId or transcript.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to seed or create a session under a raw key such as conversation:pair:user_a::user_b:space:space_123, run a sessions.resolve or sessions_send/nested-agent path that touches it, then send another inbound turn with the same raw SessionKey and inspect whether sessionId/sessionFile are reused; the current source shows the raw inbound lookup and canonical gateway lookup still diverge.

## Solution Assessment

Yes. The narrow maintainable solution is a generic raw-key continuity contract shared by inbound session initialization and gateway/session-tool resolution; the narrower closed #49202 UUID/WebUI case does not cover arbitrary structured external keys.
