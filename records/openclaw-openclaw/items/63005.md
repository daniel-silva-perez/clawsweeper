## Summary

Keep this issue open. Current main still has a mismatch between explicit inbound raw SessionKey handling and gateway/session-tool canonicalization for bare non-agent keys; the narrower closed #49202 UUID/WebUI case does not cover arbitrary structured external keys like `conversation:pair:...`.

## Best Possible Solution

Keep this issue open and implement a generic session-key continuity fix. The likely maintainer path is to make inbound explicit-key initialization and gateway `resolveSessionStoreKey` / `resolveGatewaySessionStoreTarget` share one store-key contract, or preserve durable raw aliases for external keys that future inbound traffic will continue to use. Add a regression test using a raw key such as `conversation:pair:user_a::user_b:space:space_123`, run a `sessions.resolve` or `sessions.send`/nested-agent path, then verify a later inbound message with the same raw key reuses the original `sessionId` and transcript.

## Risks / Open Questions

- External integrations that keep sending stable structured raw `SessionKey` values can still lose continuity if a gateway `sessions.resolve` or related session-tool path migrates the row to an agent-canonical key and removes the raw alias.
- A startup orphan-key migration exists, but it does not fix the live contract mismatch between future inbound explicit raw keys and gateway/session-tool canonicalization during an active gateway process.
