## Summary

Keep open. Current main and the latest release still have the Feishu receive-id fallback that this PR is trying to fix, so the PR is not obsolete; the proposed patch should not merge as-is because its nullable resolver return leaks into the send-target contract and downstream non-null send paths. Security pass found no workflow, dependency, lockfile, release-script, artifact-download, or secrets-handling changes.

## What This Changes

The PR makes Feishu receive-id type resolution nullable for invalid targets and adds guards in send-target resolution plus streaming reply startup before calling the Feishu/Lark API.

## Best Possible Solution

Land a Feishu-owned fix that rejects invalid send targets before API calls, preserves valid bare direct IDs, exposes a non-null FeishuIdType from the guarded send-target helper, handles every nullable resolver caller, and adds focused regression coverage for invalid targets plus the existing group/open_id/user_id routes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep receiveIdType non-null after target resolution:** `extensions/feishu/src/targets.ts:67`
  - body: Changing `resolveReceiveIdType` to return `null` widens `resolveFeishuSendTarget` because its `receiveIdType` field is still `ReturnType<typeof resolveReceiveIdType>`. Downstream send paths pass that value into `directParams`, whose `receiveIdType` is non-null, so the branch will fail typecheck until the guarded result is exposed as `FeishuIdType` or callers are narrowed.
  - confidence: 0.87
