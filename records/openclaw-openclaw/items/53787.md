## Summary

Keep open. The Feishu bug is still present on current main and in the latest release, while the current PR branch still has a blocking type-boundary defect before it should merge.

## What This Changes

The PR makes Feishu receive-id type resolution nullable for invalid targets and adds guards before send-target resolution and streaming-card startup call Feishu/Lark APIs.

## Best Possible Solution

Land a Feishu-owned fix that rejects truly invalid send targets before API calls, preserves valid bare direct IDs, exposes a non-null FeishuIdType from the guarded send-target helper, handles nullable resolver callers explicitly, and adds focused regression coverage for invalid targets plus existing group/open_id/user_id routes.

## Reproduction Assessment

Yes for the PR defect: the provided diff makes resolveReceiveIdType nullable while resolveFeishuSendTarget still exposes ReturnType<typeof resolveReceiveIdType>, and downstream direct send paths require a non-null ID type. The linked runtime bug has clear reporter steps in #53376, but this read-only review did not run live Feishu API validation.

## Solution Assessment

No. The PR is aimed at the right Feishu-owned boundary, but the current implementation should narrow the guarded helper return type instead of leaking null into send callers.
