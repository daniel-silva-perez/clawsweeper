## Summary

Keep this PR open. Current main still has the Feishu receive-id fallback that the PR is trying to fix, and the linked Feishu bug remains open, but the branch still needs a type-boundary fix before merge because the nullable resolver leaks into downstream direct-send contracts.

## What This Changes

The PR changes Feishu target ID resolution to return null for invalid IDs and adds guards in send-target resolution plus streaming reply startup before calling the Feishu/Lark message APIs.

## Best Possible Solution

Land a Feishu-owned fix that rejects truly invalid send targets before API calls, preserves valid bare direct IDs, exposes a non-null Feishu ID type from resolveFeishuSendTarget after the guard, updates the streaming caller consistently, and adds focused regression coverage for invalid targets plus the existing group/open_id/user_id routes.
