## Summary

Keep this PR open. Current main and the latest shipped release still have the Feishu receive-id classifier fallback that the PR is trying to harden, and the PR is paired with still-open #53376. The proposed patch is not ready to merge because widening `resolveReceiveIdType()` to include `null` also leaks a nullable `receiveIdType` through `resolveFeishuSendTarget()` while downstream send helpers require a non-null Feishu receive-id type.

## What This Changes

The PR changes Feishu receive-id type resolution to return null for invalid targets and adds guards before Feishu send-target resolution and streaming-card startup call the Feishu/Lark API.

## Best Possible Solution

Land a Feishu-owned fix that rejects invalid send targets before API calls, preserves valid bare direct IDs, exposes a non-null `FeishuIdType` from the guarded send-target helper, handles every nullable resolver caller, and adds focused regression coverage for invalid targets plus the existing group/open_id/user_id routes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep resolved receiveIdType non-null:** `extensions/feishu/src/targets.ts:67`
  - body: Changing `resolveReceiveIdType()` to return `null` widens `resolveFeishuSendTarget()` because `FeishuSendTarget.receiveIdType` is still `ReturnType<typeof resolveReceiveIdType>`. `sendMessageFeishu()` and `sendCardFeishu()` then pass that value into `directParams`, whose `receiveIdType` is non-null, so strict typechecking will fail until the guarded helper exposes a non-null Feishu ID type or callers narrow it.
  - confidence: 0.87
