## Summary

Keep open. Current main still has the Feishu receive-id fallback that the linked bug is about, and this PR is a plausible source-branch fix attempt, but the current patch is not merge-ready because its nullable resolver result leaks through the send-target return type into downstream send helpers that require a non-null receive-id type.

## What This Changes

The PR makes Feishu receive-id type resolution nullable for invalid targets and adds guards in send-target resolution plus streaming-card startup.

## Best Possible Solution

Land a Feishu-owned fix that rejects invalid send targets before API calls, preserves valid bare direct IDs, exposes a non-null `FeishuIdType` from the guarded send-target helper, handles every nullable resolver caller, and adds focused regression coverage for invalid targets plus existing group/open_id/user_id routes.

## Review Findings
