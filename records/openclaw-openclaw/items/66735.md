## Summary

Keep this PR open. Current main still uses a synchronous Linux/systemd restart path and lacks the marker-based SIGTERM restart semantics, while this PR remains the focused implementation candidate; it now needs maintainer rebase/review because GitHub reports the branch as dirty against current main.

## What This Changes

The PR adds Linux/systemd self-restart detection, a detached `systemd-run --user` restart handoff, systemd restart expectation markers, SIGTERM handling in the gateway run loop, regression tests, and a changelog entry.

## Best Possible Solution

Keep this PR as the canonical implementation candidate, rebase it onto current main, rerun the changed gateway/daemon checks, and land it after a Linux user-systemd smoke confirms self-restarts return `scheduled`, external restarts still return `completed`, and marker-triggered SIGTERM uses the bounded supervised restart path.

## Reproduction Assessment

Yes. The source-level reproduction path is to invoke a gateway restart from inside `openclaw-gateway.service` while the generated unit uses `KillMode=control-group`; current main still calls synchronous `systemctl --user restart`, and the PR discussion reports live disposable user-unit smokes for both external and in-unit restarts.

## Solution Assessment

Yes as a direction, pending rebase. The PR takes the narrowest maintainable approach I found: preserve synchronous external restarts, add a detached handoff only for current-process systemd self-restarts, and use a marker to distinguish expected restart SIGTERM from normal stop SIGTERM.
