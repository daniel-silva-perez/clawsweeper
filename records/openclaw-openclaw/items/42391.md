## Summary

Keep open. Current `main` still has typed plugin hook contracts for `before_tool_call`, `after_tool_call`, and `agent_end`, but none of those event payloads include the requested structured progress `metrics` object or fields such as `stepIndex`, `stepTotal`, `percent`, or `etaSec`. The previous ClawSweeper keep-open conclusion remains accurate at `891c7d9f1c1f5e0f52935fadfeacf695b0ef1eb8`; the only relevant hook-contract change since that review added per-hook `timeoutMs`, not progress metrics.

## Best Possible Solution

Keep this open and design an additive progress observation contract. The conservative path is to add a shared optional metrics type to relevant hook events, or introduce a dedicated progress hook, emit metrics only when OpenClaw has a defensible step model, document absent/unknown semantics in the public plugin docs, and add contract/runtime tests for monotonicity, successful completion, failure/cancellation, and compatibility with existing plugin integrations.

## Risks / Open Questions

- A numeric progress contract can mislead users unless unknown totals, dynamic plans, retries, subagents, failures, and cancellations have explicit semantics.
- This is a public plugin SDK/API change, so it needs shared types, docs, and focused runtime/contract tests for backwards compatibility and terminal success/failure behavior.
