## Summary

Keep open. Current main at e583db63c602d8d21b4ebdcd119061571a2e40a9 still has the relevant plugin hooks, but the typed hook contracts, public plugin API surface, runtime emitters, docs, and searched tests/code do not expose structured progress metrics such as stepIndex, stepTotal, percent, or etaSec for before_tool_call, after_tool_call, or agent_end. The related items cover adjacent hook/lifecycle observability, not the requested numeric progress contract.

## Best Possible Solution

Keep this open for additive plugin-facing progress observation work. The best path is to add an optional typed metrics contract to before_tool_call, after_tool_call, and agent_end, or introduce a dedicated progress hook, emitted only when OpenClaw has a defensible step model. The change should update hook types, runtime emitters across Pi and harness paths, public docs at https://docs.openclaw.ai/plugins/hooks, and focused tests for monotonicity, success reaching 100 only when appropriate, failure/cancellation behavior, and backward compatibility for existing plugins.

## Risks / Open Questions

- A numeric progress contract can mislead users unless OpenClaw defines unknown totals, dynamic plan changes, retries, subagents, failures, cancellations, and terminal success semantics before emitting percent-style values.
- Leaving this to plugin-side inference would keep Feishu and other live-card plugins deriving numeric progress inconsistently from textual phase transitions or tool events.
