## Summary

Keep this issue open. Current main still exposes typed plugin hooks for `before_tool_call`, `after_tool_call`, and `agent_end`, but their contracts, runtime emitters, tests, and public docs do not provide the requested structured progress `metrics` object (`stepIndex`, `stepTotal`, `percent`, `etaSec`). The issue is concrete, recent, and points to a real plugin API gap for live progress cards rather than a ClawHub-only plugin request.

## Best Possible Solution

Keep this open for an additive plugin API design/implementation. The best path is either an optional `metrics` object on `before_tool_call`, `after_tool_call`, and `agent_end`, or a dedicated progress hook, emitted only when OpenClaw has a defensible monotonic step model. Update `src/plugins/hook-types.ts`, runtime emitters across Pi and harness paths, tests for monotonic/success/failure behavior, and the public plugin hooks docs at https://docs.openclaw.ai/plugins/hooks.

## Risks / Open Questions

- Adding numeric progress without a defensible step model could mislead users, especially if failures or cancellations are shown as 100%.
- The contract should remain optional and backward-compatible for existing hook consumers.
- Pi, Codex/native harness, CLI, and embedded subscribe paths may need parity so progress cards do not behave differently by runtime.
