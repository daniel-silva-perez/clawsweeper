## Summary

Keep this PR open. Current main does not implement the proposed startup memory-source report: there is no Memoria/source-report module or prompt wiring, the embedded runner only passes existing workspace notes, and the command/system-prompt path still builds the system prompt without workspaceNotes. The existing startup-context feature loads recent daily memory contents for bare /new/reset, but it does not report local/cloud/file memory availability or counts.

## Best Possible Solution

Keep this PR open for revision rather than close it. The best path is to rebase onto current main, decide whether the report should use existing memory manager/status APIs or an explicit new Memoria integration, add bounded startup probes and accurate count/unknown rendering, cover both embedded and command/system-prompt flows with focused tests, and split the Feishu outbound logging into a separate PR if it is still wanted.

## Risks / Open Questions

- The PR mixes the memory-source report with unrelated Feishu outbound logging changes, so review and validation are harder than the title suggests.
- The proposed startup network probes need bounded timeouts and careful failure rendering; otherwise a configured slow cloud endpoint could delay prompt construction.
- Current main's memory surface is memory-core/QMD/Honcho-oriented and has no Memoria references, so maintainers should decide whether hard-coded Memoria env probes are still the right product direction before merging.
