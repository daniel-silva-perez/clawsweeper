## Summary

Current main still has the reported supervisor behavior: cancellation from manual replacement, no-output timeout, and overall timeout flows into `cancelAdapter`, which immediately calls `adapter.kill("SIGKILL")`. Adjacent adapter/process-tree safeguards exist, including SIGKILL wait fallbacks and process-tree graceful cleanup, but they do not implement the requested supervisor-level SIGTERM-first grace period for the direct supervised run. The related context also shows the previous broad PR #70094 was closed unmerged, duplicate reports #70004/#70026 were folded into this issue, and #71710 is a separate agent-side SIGTERM handling follow-up rather than a replacement for this supervisor-policy work.

## Best Possible Solution

Keep this issue open as the canonical supervisor cancellation-policy bug. Implement a narrow change in `src/process/supervisor/supervisor.ts`: on manual cancellation, scoped replacement, overall timeout, and no-output timeout, send SIGTERM first, start an unref'ed grace timer, and escalate to SIGKILL only if the run has not settled. Clear the escalation timer during settle/dispose, preserve existing child/PTY wait fallbacks and `killProcessTree` detached-safety behavior, and update supervisor/adapter tests to assert SIGTERM-first behavior plus hard-kill escalation. Track agent-side SIGTERM handling separately under #71710.

## Risks / Open Questions

- A focused fix must preserve the existing `detached:false` and process-group safety in the child adapter and `killProcessTree`; a naive group kill can terminate the gateway process group in service-managed or fallback-spawned cases.
- Windows signal semantics differ from POSIX, so the implementation should retain adapter-specific Windows fallbacks and add explicit tests for child and PTY behavior where feasible.
- SIGTERM responsiveness inside `openclaw agent` is separate from supervisor escalation; the supervisor should still guarantee a bounded SIGKILL fallback if the child ignores SIGTERM.
