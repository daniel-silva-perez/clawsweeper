## Summary

Keep open. Current main still lacks the requested external-chat workflow for `/agents` to list configured agents and `/agent <id>` to temporarily route one conversation to another configured agent; the existing chat `/agents` surface remains thread-bound subagent/session-binding focused, while configured-agent selection is available through static bindings and client-local surfaces like TUI.

## What This Changes

Request external-chat commands to list configured agents and temporarily switch the current channel conversation to another configured agent without restarting the Gateway.

## Best Possible Solution

Add an authorized Gateway/channel routing feature that lists configured agents in external chat, stores a temporary per-conversation configured-agent override before static binding fallback, supports reset or expiry, and preserves the selected agent’s workspace, auth, tool, and memory isolation. If maintainers want this plugin-led, first expose a documented route-override API for configured agent ids.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a missing feature. The gap is high-confidence from source/docs: external chat has `/agents` for thread-bound subagents, no `/agent <id>` command, and route resolution has no per-conversation configured-agent override.

## Solution Assessment

Unclear until maintainers decide the command namespace and security policy. A session-level route override is likely the narrowest maintainable direction; hot-editing bindings from chat would be riskier because it mutates global routing and can affect unrelated conversations.
