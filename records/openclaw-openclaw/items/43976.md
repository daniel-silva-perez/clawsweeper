## Summary

Keep open. Current main still lacks the requested external-chat workflow for listing configured agents and temporarily switching one conversation to another configured agent without restarting the Gateway; the existing `/agents` command is for subagent/thread-binding state, and configured-agent routing still resolves through static bindings or explicit CLI/UI/RPC surfaces. The prior ClawSweeper comment reached the same conclusion, and the current source/docs still support it.

## What This Changes

Request adding session-level chat commands to list configured agents and temporarily switch the current channel conversation to a different configured agent without restarting the Gateway.

## Best Possible Solution

Add an authorized Gateway/channel route-override feature that lists configured agents in external chat, stores a temporary per-conversation configured-agent override before static binding fallback, supports reset or expiry, and preserves the selected agent’s workspace, auth, tool, and memory isolation. Avoid hot-editing global bindings from chat unless maintainers explicitly choose that broader behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a missing feature, and the absence is high-confidence from the command registry, handler, route resolver, and docs inspection.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide namespace and security policy. A session-level route override is likely the narrow maintainable direction; mutating global bindings from chat would be riskier because it can affect unrelated conversations.
