## Summary

Current main still does not implement the requested Gateway/channel workflow for `/agents` plus `/agent <name>` to temporarily switch a single conversation to another configured agent without editing global bindings or restarting the Gateway. Existing functionality covers static multi-agent routing, TUI-local agent selection, Control UI agent listing, and thread/subagent focus bindings, but not a per-conversation configured-agent route override command.

## Best Possible Solution

Keep this open for maintainer-designed Gateway/channel routing work. A complete solution should add authorized `/agents` and `/agent <id|main>` handling in the Gateway command layer, list configured agents from `agents.list`, store a temporary per-conversation route override before normal binding fallback, create or select the target agent's main session as needed, and define reset, expiry, workspace/auth/memory isolation, and audit semantics. If maintainers prefer plugin-first behavior, core should expose a documented plugin API for per-conversation configured-agent route overrides without requiring global `bindings` edits.

## Risks / Open Questions

- There is overlapping terminology: TUI-local `/agent <id>`, Control UI `/agents`, Gateway thread-bound `/agents`, static `bindings`, and `/focus` session bindings all exist. A fix should define which surfaces share semantics and which remain client-local.
- Docs such as `docs/help/faq-models.md` mention using `/agent` to switch agents, but current source shows that behavior in the TUI path, not in Gateway channel commands. Maintainers may want to clarify docs while designing the channel workflow.
