## Summary

Keep open. Current main still lacks the requested external-chat workflow to list configured agents and temporarily switch one conversation to another configured agent; the existing chat `/agents` path is still subagent/thread-binding oriented, while configured-agent selection remains available through static routing bindings and client surfaces like TUI/WebChat.

## What This Changes

The issue requests `/agents` and `/agent <name>` chat commands that temporarily switch the current single-chat conversation to another configured agent without editing bindings or restarting the Gateway.

## Best Possible Solution

Add an authorized Gateway/channel routing feature that lists configured agents in external chat, stores a temporary per-conversation configured-agent override before static binding fallback, supports reset or expiry, and preserves the selected agent’s workspace, auth, tool, and memory isolation. If maintainers want this to be plugin-led, first expose a documented core route-override API for configured agent ids.
