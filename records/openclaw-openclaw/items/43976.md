## Summary

Keep open. Current main still lacks the requested external-channel workflow to list configured agents and temporarily switch one conversation to another configured agent without editing bindings or restarting the Gateway; the matching chat `/agents` surface is still for thread-bound subagents/session bindings, while configured-agent selection exists in static bindings and client surfaces such as TUI/WebChat.

## What This Changes

The issue requests `/agents` and `/agent <name>` chat commands that temporarily switch the current channel conversation to a configured agent without changing global bindings or restarting the Gateway.

## Best Possible Solution

Add an authorized Gateway/channel routing feature that lists configured agents in external chat, stores a temporary per-conversation configured-agent override before static binding fallback, provides reset or expiry behavior, and preserves the selected agent’s workspace, auth, tool, and memory boundaries. If maintainers want this to be plugin-led, first expose a documented core route-override API for configured agent ids.
