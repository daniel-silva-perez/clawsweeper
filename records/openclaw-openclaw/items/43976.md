## Summary

Keep open. Current main still does not provide the requested external-chat workflow to list configured agents and temporarily switch one conversation to another configured agent without restarting the Gateway; the existing `/agents` chat command is for thread-bound subagents/session bindings, while configured-agent switching exists only in client-local surfaces such as the TUI and static routing bindings.

## What This Changes

Request to add session-scoped chat commands `/agents` and `/agent <id>` so a single external conversation can temporarily route to a different configured agent without editing bindings or restarting the Gateway.

## Best Possible Solution

Add an authorized Gateway/channel route-override feature that lists configured agents in external chat, stores a temporary per-conversation configured-agent override ahead of static binding fallback, supports reset or expiry, and preserves the selected agent’s workspace, auth, tool, and memory isolation. If maintainers want this plugin-led, first expose a documented route-override API for configured agent ids.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a missing feature. The gap is high-confidence from source/docs: external chat has `/agents` for thread-bound subagents, no `/agent <id>` command, and route resolution has no per-conversation configured-agent override.

## Solution Assessment

Unclear until maintainers decide the command namespace and security policy. A session-level route override is likely the narrow maintainable direction; hot-editing bindings from chat would be riskier because it mutates global routing and can affect unrelated conversations.
