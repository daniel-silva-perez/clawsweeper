## Summary

Keep open. Current main still lacks the requested chat-level workflow to list configured agents and temporarily switch one channel conversation to another configured agent without editing bindings or restarting the Gateway. The existing `/agents` command is for thread/subagent binding status, `agents.list` is a Gateway/TUI RPC, and the TUI-only `/agent` switch does not affect inbound channel routing.

## What This Changes

The issue requests `/agents` and `/agent <name|main>` chat commands that temporarily switch the current channel conversation to another configured agent without changing global bindings or restarting the Gateway.

## Best Possible Solution

Add a maintainer-designed Gateway/channel routing feature that authorizes configured-agent selection, lists configured agents in chat, stores a temporary per-conversation route override ahead of static binding fallback, cleanly resets or expires that override, and preserves each selected agent's workspace, auth, and memory boundaries. If this should be plugin-first, expose a documented route-override API for configured agent ids first.
