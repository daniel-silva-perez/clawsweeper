## Summary

Keep open. Current main still does not provide the requested external-chat workflow to list configured agents and temporarily switch one conversation to another configured agent without restarting the Gateway; the existing `/agents` command is for subagent/thread-binding state, while configured-agent selection remains exposed through CLI, TUI/WebChat, SDK/RPC, and static routing bindings.

## What This Changes

The issue requests chat commands to list configured agents and temporarily switch a single channel conversation to another configured agent without restarting the Gateway.

## Best Possible Solution

Add an authorized Gateway/channel route-override feature that lists configured agents in external chat, stores a temporary per-conversation configured-agent override before static binding fallback, supports reset or expiry, and preserves the selected agent’s workspace, auth, tool, and memory isolation. If maintainers want this plugin-led, first expose a documented route-override API for configured agent ids.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a missing feature. The gap is high-confidence from source and docs: external chat has `/agents` for thread-bound subagents, no `/agent <id>` command, and route resolution has no per-conversation configured-agent selector.

## Solution Assessment

Unclear until maintainers decide command namespace and security policy. A session-level route override is likely the narrow maintainable direction; hot-editing global bindings from chat would be riskier because it can affect unrelated conversations.
