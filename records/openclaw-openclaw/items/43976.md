## Summary

Keep open. Current main still lacks the requested channel-chat workflow to list configured agents and temporarily switch the active channel conversation to another configured agent without editing bindings or restarting the Gateway. The existing `/agents` chat command is still for thread-bound subagent/session state, while configured-agent selection exists only in Gateway clients such as TUI/WebChat and does not alter external channel routing.

## What This Changes

The issue requests `/agents` and `/agent <name|main>` chat commands that temporarily switch the current channel conversation to another configured agent without changing global bindings or restarting the Gateway.

## Best Possible Solution

Add a Gateway/channel routing feature that authorizes configured-agent selection, lists configured agents in chat, stores a temporary per-conversation route override ahead of static binding fallback, resets or expires that override cleanly, and preserves each selected agent's workspace, auth, and memory boundaries. If this should be plugin-first, expose a documented route-override API for configured agent ids first.
