## Summary

Keep this issue open. Current main still serves the Control UI from the Gateway HTTP server on the Gateway port, with only an in-process SPA route fallback while the Gateway is alive; there is no crash-survivor static server, same-port takeover path, or gateway.serveStaticOnCrash-style config surface.

## What This Changes

Request an opt-in Gateway resilience feature that keeps a static WebUI or maintenance page reachable on the Gateway port after the Gateway process crashes.

## Best Possible Solution

Keep this open for maintainer product/design judgment. The best path is either to document supervised restart/rescue gateways as the supported resilience strategy, or design a core service-manager/Gateway fallback that can safely serve a minimal authenticated maintenance page on the configured Gateway port while the real Gateway is down, with typed config/schema/help, docs, and tests for port handoff, custom fallback content, restart recovery, auth/origin behavior, TLS, and information disclosure.
