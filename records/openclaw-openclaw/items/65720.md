## Summary

Keep this issue open. Current main still serves the Control UI from the Gateway process and port, and the only static fallback is the normal SPA route fallback while that process is alive. There is restart/supervisor recovery support, but no crash-time same-port static maintenance server or gateway.serveStaticOnCrash-style config surface.

## Best Possible Solution

Keep open for maintainer product/design judgment. The best path is either to document supervised restart as the supported resilience strategy, or design a core Gateway/service-manager fallback that can safely serve a minimal maintenance page on the configured Gateway port while the real Gateway is down, with typed config/schema/help, docs, and tests for port handoff, custom fallback content, restart recovery, auth/origin behavior, TLS, and information disclosure.
