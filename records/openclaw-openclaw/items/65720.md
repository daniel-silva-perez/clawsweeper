## Summary

Keep this issue open. Current main still serves the Control UI from the Gateway HTTP server on the same Gateway process/port, and the only static fallback present is the normal SPA index.html route fallback while that process is alive. There is restart/respawn/stability-bundle handling, but no same-port static maintenance server or gateway.serveStaticOnCrash-style config surface.

## Best Possible Solution

Keep this open for maintainer product/design judgment. The best path is to either explicitly document supervised restart as the supported resilience strategy, or design a core Gateway/service-manager fallback that can safely serve a minimal maintenance page on the configured Gateway port while the real Gateway is down, with typed config/schema/help, docs, and tests for port handoff, custom fallback content, restart recovery, auth/origin behavior, and information disclosure.
