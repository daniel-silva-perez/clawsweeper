## Summary

Keep open. Current main still forwards Firecrawl scrape targets to the Firecrawl API without local target validation, so the reported hardening is real, but this PR should not merge as-is because the latest diff imports `SsrFBlockedError` from a non-exported SDK subpath and would break Firecrawl client loading before the guard can run.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper and client tests for allowed and blocked target URLs, while the body also claims a cache-size cap that the final diff does not add.

## Best Possible Solution

Revise this as a narrow Firecrawl security hardening change: validate caller-supplied scrape targets inside `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF helpers from an exported plugin SDK subpath, keep focused regression coverage for blocked literal/internal targets and allowed public URLs, and align the PR body with the existing shared cache behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main forwards `params.url` as the Firecrawl scrape `body.url` without a local Firecrawl URL guard, and the PR defect is visible by comparing its `browser-security-runtime` import with current main's SDK exports.

## Solution Assessment

No as-is. The enforcement point and tests are the right shape, but the stale non-exported SDK import would break Firecrawl client loading; the safer fix is to rebase and import `SsrFBlockedError` from `openclaw/plugin-sdk/ssrf-runtime` or `openclaw/plugin-sdk/security-runtime`.
