## Summary

Keep open. Current main still forwards caller-supplied Firecrawl scrape URLs without local SSRF validation, so the security goal is valid, but this PR is not mergeable as-is because it imports `SsrFBlockedError` from an SDK subpath that is absent from current main.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper, calls it from `runFirecrawlScrape`, and adds Firecrawl client tests for allowed and blocked scrape URL cases while also describing cache-cap behavior.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF helpers from exported plugin SDK subpaths, keep focused regression tests for the helper and scrape caller surfaces, and align the PR body with the existing shared cache cap.
