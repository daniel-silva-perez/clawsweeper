## Summary

Keep open. Current main still forwards Firecrawl scrape targets without local SSRF validation, so the PR addresses a real security gap, but the branch should not merge as-is because the added Firecrawl client imports a non-exported SDK subpath and would fail before the guard can run.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper, calls it before forwarding scrape URLs to Firecrawl, and adds client tests for allowed and blocked URL inputs; the final diff does not add the cache-size cap described in the PR body.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in runFirecrawlScrape before cache lookup and API forwarding, import SSRF helpers and errors from exported plugin SDK subpaths, keep focused regression tests for the client and scrape caller surfaces, align the PR body with the actual shared cache behavior, and document or explicitly decide the Firecrawl hosted/self-hosted SSRF policy.

## Reproduction Assessment

Yes. Current main can be checked statically: runFirecrawlScrape forwards params.url as body.url to /v2/scrape with no SSRF guard; the PR defect is reproducible from package exports because openclaw/plugin-sdk/browser-security-runtime is not an exported SDK subpath.

## Solution Assessment

No as-is. The enforcement point and tests are directionally right, but the non-exported import breaks Firecrawl client loading; the safer fix is to import SsrFBlockedError from openclaw/plugin-sdk/ssrf-runtime or openclaw/plugin-sdk/security-runtime and then rerun the focused Firecrawl and SDK export checks.
