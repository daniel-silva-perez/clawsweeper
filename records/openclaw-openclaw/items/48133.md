## Summary

Keep open. Current main still forwards caller-supplied Firecrawl scrape URLs to the Firecrawl API without local target validation, so the reported SSRF gap is real. The PR is directionally useful and now includes focused tests, but it is not mergeable as-is because the latest diff imports `SsrFBlockedError` from a plugin SDK subpath that current main does not export.

## What This Changes

The PR adds `assertFirecrawlUrlAllowed()` and a new Firecrawl client test file to block non-HTTP(S), localhost/private, and metadata scrape targets before `runFirecrawlScrape` forwards URLs to Firecrawl.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF types from an exported plugin SDK subpath, decide and document the self-hosted Firecrawl policy, keep focused regression coverage for the helper and caller surfaces, and align the PR body with the existing shared cache behavior.
