## Summary

Keep open. The Firecrawl SSRF gap still exists on current main and the PR is directionally useful, but it is not merge-ready: the live PR is dirty/unmergeable with maintainer edits disabled, and the diff imports a plugin SDK subpath that is absent from current main.

## What This Changes

The PR adds Firecrawl scrape URL validation plus client tests for allowed and blocked URL inputs, while the body also claims a cache-size cap that the final diff does not implement.

## Best Possible Solution

Revise this as a narrow Firecrawl security hardening change: validate caller-supplied scrape targets inside runFirecrawlScrape before cache lookup and API forwarding, import SSRF helpers from an exported plugin SDK subpath, keep focused regression coverage for blocked literal/internal targets and allowed public URLs, and align the PR description with the existing shared cache behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: on current main, runFirecrawlScrape forwards params.url as body.url to Firecrawl with no local Firecrawl URL guard, and the PR defect is visible by comparing its browser-security-runtime import with current main's SDK exports.

## Solution Assessment

No as-is. The enforcement point and tests are the right shape, but the stale non-exported SDK import and dirty unmodifiable branch make this PR unsafe to merge without revision; the safer path is to rebase and use ssrf-runtime or security-runtime.
