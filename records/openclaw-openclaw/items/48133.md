## Summary

Keep open. The Firecrawl SSRF hardening is still relevant because current main forwards caller-supplied scrape URLs to Firecrawl without a local SSRF guard, but this PR should not merge as-is: the added client imports `SsrFBlockedError` from a non-exported plugin SDK subpath, so the Firecrawl client would fail to load before the protection runs.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper, calls it from `runFirecrawlScrape`, and adds tests for allowed and blocked scrape URL inputs while the body also describes cache-cap behavior.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF helpers and errors from exported plugin SDK subpaths, keep focused regression tests for the client and scrape caller surfaces, and align the PR body with the actual shared cache behavior and Firecrawl hosting policy.
