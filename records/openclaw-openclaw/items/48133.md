## Summary

Keep open. Current main still forwards Firecrawl scrape target URLs without local SSRF validation, so the security hardening remains useful, but this PR cannot merge as-is because it imports `SsrFBlockedError` from a non-exported plugin SDK subpath. The cache-size claim is not a remaining main-branch gap because the shared cache helper already evicts entries at 100.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper, calls it from `runFirecrawlScrape`, and adds Firecrawl client tests for allowed and blocked scrape URL cases while also describing cache-cap behavior.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF helpers from exported plugin SDK subpaths, keep focused regression tests for both the helper and scrape caller surfaces, and align the PR body with the existing shared cache cap.
