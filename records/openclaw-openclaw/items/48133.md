## Summary

Keep open. The security goal is valid because current main still forwards Firecrawl scrape URLs to Firecrawl without local SSRF validation, but this PR cannot land as-is: it imports `SsrFBlockedError` from a non-exported plugin SDK subpath, so the Firecrawl client would fail to load before the new guard can run. The cache-size part of the PR body is not a remaining main-branch gap because Firecrawl already writes through the shared capped cache helper.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper, calls it at the start of `runFirecrawlScrape`, and adds client tests for allowed and blocked URL inputs; the final diff does not implement the 500-entry cache cap described in the body.

## Best Possible Solution

Revise this as a narrow Firecrawl security hardening change: validate caller-supplied scrape targets inside `runFirecrawlScrape` before cache lookup and API forwarding, import both SSRF helpers from an exported SDK subpath such as `openclaw/plugin-sdk/ssrf-runtime`, keep focused regression tests for blocked literal/internal URLs and allowed public URLs, and align the PR description with the existing shared cache behavior.

## Reproduction Assessment

Yes. Current main can be checked statically: `runFirecrawlScrape` forwards `params.url` as `body.url` to Firecrawl without a local SSRF guard, and the PR defect is reproducible from package exports because `openclaw/plugin-sdk/browser-security-runtime` is absent.

## Solution Assessment

No as-is. The enforcement point and tests are directionally right, but the non-exported SDK import breaks module resolution; the safer fix is to import `SsrFBlockedError` from `openclaw/plugin-sdk/ssrf-runtime` or `openclaw/plugin-sdk/security-runtime` and keep the rest of the guard focused.
