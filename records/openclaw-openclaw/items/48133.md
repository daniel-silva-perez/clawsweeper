## Summary

Keep open. Current main still lacks a Firecrawl scrape target-URL SSRF guard, so the PR is not obsolete. The submitted branch is also not ready to close or land as-is because it imports a non-existent plugin SDK subpath, relies on a static hostname-only SSRF check that needs maintainer/security review for self-hosted Firecrawl, and the cache-size claim is already covered differently by the shared cache helper.

## What This Changes

This PR modifies the Firecrawl client and adds Firecrawl client tests to validate caller-supplied scrape URLs before forwarding them to Firecrawl.

## Best Possible Solution

Keep this PR open and revise it into a narrow Firecrawl security fix: validate caller-supplied scrape URLs in the shared `runFirecrawlScrape` path before cache lookup/API forwarding, use an exported plugin SDK SSRF helper, add Firecrawl SSRF regression tests for direct scrape/provider/facade coverage, correct the cache-cap claim, and get maintainer/security review on the static-vs-DNS resolution policy before landing.
