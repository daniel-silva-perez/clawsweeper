## Summary

Keep open. Current main still forwards Firecrawl scrape URLs to Firecrawl without a local SSRF guard, so the security hardening is still useful, but this PR is not mergeable as-is because its latest diff imports `SsrFBlockedError` from a non-exported SDK subpath that would break Firecrawl client module loading before the guard can run.

## What This Changes

The PR adds `assertFirecrawlUrlAllowed()` to the Firecrawl client, calls it before scrape requests, exports it for tests, and adds Firecrawl client tests for allowed and blocked target URLs while the body still describes a cache-size cap not present in the final diff.

## Best Possible Solution

Revise this as a narrow Firecrawl security hardening change: validate caller-supplied scrape targets inside `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF helpers from exported plugin SDK subpaths, keep focused client and caller-surface regression tests, preserve the shared cache behavior unless a separate configurable cache-size change is intended, and align the PR body with the final implementation.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main forwards `params.url` as the Firecrawl `/v2/scrape` request body without local SSRF validation, and the PR defect is visible by comparing its `browser-security-runtime` import with current main's package exports and `src/plugin-sdk` files.

## Solution Assessment

No as-is. The enforcement point and tests are the right shape, but the patch must import `SsrFBlockedError` from an exported subpath such as `openclaw/plugin-sdk/ssrf-runtime` or `openclaw/plugin-sdk/security-runtime` before it is safe to merge.
