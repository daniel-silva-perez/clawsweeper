## Summary

Keep open. The Firecrawl scrape-target SSRF gap still exists on current main, but this PR should not merge as-is because the head diff imports `SsrFBlockedError` from a non-exported plugin SDK subpath, which would break Firecrawl client module resolution before the guard can run. The cache-growth part is already covered by the shared cache helper on main, and the dirty, uneditable branch plus self-hosted Firecrawl policy interaction need maintainer/security handling.

## What This Changes

The PR adds Firecrawl scrape URL validation and tests that block non-HTTP(S), localhost, private, and metadata targets before `runFirecrawlScrape` forwards caller-supplied URLs to Firecrawl.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF helpers from exported SDK subpaths, keep focused regression tests for the helper and scrape caller surfaces, and align the PR body with the existing shared cache cap. Coordinate the DNS/private-network policy with the open self-hosted Firecrawl work instead of mixing it into an unmergeable branch.
