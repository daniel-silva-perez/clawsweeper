## Summary

Keep open. Current main still forwards caller-supplied Firecrawl scrape URLs to Firecrawl without target validation, so the core security concern remains valid. This PR should not be auto-closed or auto-landed because the current head is dirty against main, contains a non-exported plugin SDK import that would break Firecrawl client loading, and still needs maintainer/security judgment on the static hostname policy and cache-scope claim.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper and Firecrawl client tests to block non-HTTP(S), localhost/private, and metadata scrape targets before forwarding caller-supplied scrape URLs to Firecrawl, while the body also describes a cache-size cap.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF types from an exported plugin SDK subpath, decide and document the hosted versus self-hosted Firecrawl SSRF policy, keep regression coverage for the client and caller surfaces, and align the PR body with the actual shared cache behavior.
