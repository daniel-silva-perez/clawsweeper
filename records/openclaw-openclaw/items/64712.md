## Summary

Keep open. Current main still has the default-account status fallback, and this PR is not merge-ready because its guard does not distinguish Feishu's synthetic default account from a real configured account, its new regression test is missing required imports, and it can hide diagnostics for explicit configs whose account IDs are unresolved.

## What This Changes

The PR adds a `hasRealAccounts` guard around channel-summary construction and status-line emission, plus a regression test for channels with no configured accounts.

## Best Possible Solution

Add or use a typed way to tell authored or persisted account configuration apart from synthetic default fallbacks, then keep status diagnostics for explicit-but-unresolved configs and cover both the real Feishu helper path and an unresolved-config visibility case in tests.

## Reproduction Assessment

Yes. The source-level path is clear: current main can emit a fallback default status line, and Feishu's real account helper returns a synthetic `default` ID when no accounts exist, so the PR's empty-array fixture does not reproduce the linked Feishu behavior accurately.

## Solution Assessment

No. The PR's `accountIds.length > 0` heuristic is not the narrowest maintainable fix because `listAccountIds` may include synthetic defaults and some explicit configs resolve to no account IDs; the safer approach is to distinguish real authored/persisted accounts from fallback display IDs.

## Review Findings

- **[P1] Distinguish synthetic Feishu defaults from real accounts:** `src/infra/channel-summary.ts:194`
  - body: `hasRealAccounts` is true whenever `accountIds` is non-empty, but Feishu's shared `listAccountIds` helper returns `default` when there are no configured accounts. That means the linked Feishu phantom-default path still calls the status adapter and still pushes a status line; the regression test stubs `accountIds: []` and misses the real helper behavior.
  - confidence: 0.91

  - body: The new test calls `setActivePluginRegistry` and `createTestRegistry`, but this file does not import either name and the PR diff adds no import hunk. The test file will fail to typecheck/run before it can verify the regression.
  - confidence: 0.95
- **[P2] Keep unresolved configured channels visible:** `src/infra/channel-summary.ts:244`
  - body: Gating the status line on `hasRealAccounts` drops channels whose explicit config cannot resolve an account ID yet. For example, Nostr can have meaningful config with a SecretRef-backed `privateKey` while `listNostrAccountIds` returns `[]`, so this change would hide the status diagnostic instead of showing the user what needs attention.
  - confidence: 0.86
