## Summary

Keep open. Current main does not already implement the requested gateway MCP Apps WebSocket/resource surface, so the PR is not obsolete, but the branch is dirty against main and its MCP tool-call paths currently miss security-sensitive protections that main now requires before merge.

## What This Changes

The PR adds gateway-side MCP Apps support with new `mcp.*` WebSocket methods, TypeBox schemas/validators, MCP resource registry and HTTP loopback resource handlers, `_meta.ui` tool metadata, scope entries, docs, and tests.

## Best Possible Solution

Rebase the PR onto current main, keep the MCP Apps protocol additive, and integrate it through the existing MCP loopback security model: token-derived owner context, owner-only tool filtering, before-tool-call hooks/abort handling, and focused tests for both HTTP and the new WS `mcp.tools.call` path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve owner-only filtering in the MCP cache:** `src/gateway/mcp-http.runtime.ts:63-64`
  - body: Current main filters resolved loopback tools with `applyOwnerOnlyToolPolicy` before caching and building schemas. This branch stores `next.tools` directly, so non-owner MCP HTTP/WS callers can see and execute owner-only tools such as `cron` or tools marked `ownerOnly`.
  - confidence: 0.93
- **[P1] Run before-tool hooks before loopback calls:** `src/gateway/mcp-http.handlers.ts:126`
  - body: The current loopback `tools/call` path runs `runBeforeToolCallHook` and passes the abort signal before `tool.execute`. This branch's HTTP handler executes the tool directly, bypassing hook blocks/approval parameter rewriting and request cancellation behavior that main already tests.
  - confidence: 0.91
- **[P1] Apply hooks to the new WS call path:** `src/gateway/server-methods/mcp.ts:146`
  - body: The new `mcp.tools.call` handler also executes tools directly. Since the PR advertises reuse of the existing execution pipeline, WS callers need the same before-tool-call hook and abort handling as the MCP loopback path before this surface is safe to expose.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR adds new MCP tool execution surfaces and currently bypasses current-main owner-only and before-tool-call protections.

Concerns:

- **[high] Owner-only tools can leak to non-owner MCP callers:** `src/gateway/mcp-http.runtime.ts:63`
  - body: The PR head removes the current-main `applyOwnerOnlyToolPolicy` filtering from `McpLoopbackToolCache`, so non-owner callers can receive cached owner-only tools and invoke them through the HTTP or new WS MCP call paths.
  - confidence: 0.93
- **[high] Tool-call hooks are bypassed:** `src/gateway/mcp-http.handlers.ts:126`
  - body: Both the modified HTTP handler and the new WS handler call `tool.execute` directly. That skips the hook path used for approval/blocking, parameter rewriting, and abort propagation on current main.
  - confidence: 0.9
- **[medium] Owner context must stay token-derived after rebase:** `src/gateway/mcp-http.request.ts:139`
  - body: The PR head still contains the older single-token, header-derived owner context model in the loopback request path; current main derives owner/non-owner context from separate bearer tokens. A rebase must preserve the current token-derived model.
  - confidence: 0.82
