## Summary

Keep this PR open: current main does not yet implement the proposed MCP Apps gateway WebSocket/resource surface, but the PR head is not safe to merge because it drops current-main MCP loopback security behavior around owner-only tool filtering, before-tool-call hooks, abort propagation, and token-derived owner context.

## What This Changes

The PR adds gateway-side MCP Apps support with new `mcp.*` WebSocket methods, MCP resource registry and HTTP loopback resource handlers, `_meta.ui` tool metadata/types, scope entries, docs, and tests.

## Best Possible Solution

Rebase the PR onto current main, keep the MCP Apps protocol additive, and integrate it through the existing MCP loopback security model: token-derived owner context, owner-only tool filtering before cache/schema/resource sync, before-tool-call hooks with abort propagation, and focused regression tests for both HTTP loopback and the new WS `mcp.tools.call` path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve owner-only filtering in the MCP cache:** `src/gateway/mcp-http.runtime.ts:62-64`
  - body: Current main filters resolved loopback tools with `applyOwnerOnlyToolPolicy` before caching and building schemas. This branch stores `next.tools` directly, so non-owner MCP HTTP/WS callers can see and execute owner-only tools such as `cron` or tools marked `ownerOnly`.
  - confidence: 0.93
- **[P1] Run before-tool hooks before loopback calls:** `src/gateway/mcp-http.handlers.ts:126`
  - body: The current loopback `tools/call` path runs `runBeforeToolCallHook` and passes the abort signal before `tool.execute`. This branch's HTTP handler executes the tool directly, bypassing hook blocks, approval parameter rewriting, and request cancellation behavior that main already tests.
  - confidence: 0.91
- **[P1] Apply hooks to the new WS call path:** `src/gateway/server-methods/mcp.ts:146`
  - body: The new `mcp.tools.call` handler also executes tools directly. Since this exposes a new gateway tool-call surface, it needs the same before-tool-call hook and abort handling as the MCP loopback path before it is safe to expose.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds MCP tool execution/resource surfaces and currently bypasses current-main owner-only filtering, token-derived owner context, and before-tool-call protections.

Concerns:

- **[high] Owner-only tools can leak to non-owner MCP callers:** `src/gateway/mcp-http.runtime.ts:63`
  - body: PR head stores unfiltered `next.tools` in `McpLoopbackToolCache`; current main filters with `applyOwnerOnlyToolPolicy` before caching and schema creation, so this can expose owner-only tools to non-owner MCP callers.
  - confidence: 0.93
- **[high] Tool-call hooks are bypassed:** `src/gateway/mcp-http.handlers.ts:126`
  - body: Both the modified HTTP loopback handler and the new WS handler call `tool.execute` directly, skipping the hook path used for blocking, approval parameter rewriting, and abort propagation on current main.
  - confidence: 0.9
- **[medium] Owner context must stay token-derived:** `src/gateway/mcp-http.request.ts:139`
  - body: PR head still uses the older single-token plus header-derived sender-owner model, while current main derives owner/non-owner status from separate bearer tokens. A rebase must preserve the token-derived model.
  - confidence: 0.82
