## Summary

Keep open: current main still lacks the requested MCP Apps gateway protocol surface, but the PR head is stale against current main and would regress current MCP loopback security behavior around owner-only tools, before-tool-call hooks, and token-derived owner context.

## What This Changes

The PR branch adds gateway MCP Apps WebSocket methods, MCP resource listing/reading over WS and HTTP loopback, MCP UI metadata schemas, a resource registry, scopes, docs, and focused tests.

## Best Possible Solution

Rebase the PR onto current main and land the MCP Apps surface only after it preserves the existing MCP loopback security model: token-derived owner context, owner-only tool filtering before schema/cache exposure, before-tool-call hook and abort behavior on HTTP and WS tool calls, scoped resource access decisions, regenerated protocol artifacts, and tests that cover the new WS path plus the existing loopback regressions.

## Security Review

Status: needs_attention

Summary: The PR adds new MCP tool execution and resource-read surfaces while the current head bypasses current-main owner filtering, hook enforcement, and token-derived owner context.

Concerns:

- **[high] Owner-only tools can leak to non-owner MCP callers:** `src/gateway/mcp-http.runtime.ts:63`
  - body: The PR head caches and exposes `next.tools` directly instead of applying current-main `applyOwnerOnlyToolPolicy`, so non-owner callers could receive owner-only tool schemas and execute privileged tools through HTTP or the new WS MCP path.
  - confidence: 0.93
- **[high] Tool-call hooks are bypassed:** `src/gateway/mcp-http.handlers.ts:126`
  - body: Both the modified HTTP handler and the new WS handler call `tool.execute` directly. That skips the hook path used for approval/blocking, parameter rewriting, and abort propagation on current main.
  - confidence: 0.91
- **[medium] Owner context must remain token-derived:** `src/gateway/mcp-http.request.ts:53`
  - body: The PR head still uses the older single-token loopback request model and a caller-supplied owner header, while current main derives owner/non-owner context from separate bearer tokens. A rebase must preserve the current token-derived model.
  - confidence: 0.86
- **[medium] Global resource reads need scoping review:** `src/gateway/server-methods/mcp.ts:207`
  - body: The PR exposes all registered `ui://` resources through list/read methods while the registry supports file-backed resources and is documented as global. Maintainers should confirm this is acceptable or scope reads to the same session/account context used for tool resolution.
  - confidence: 0.78
