## Summary

Keep this PR open because current main does not implement the proposed MCP Apps gateway surface, but the patch is not merge-ready: the new WS tool-call path bypasses the current before-tool-call hook contract, and the proposed resource registry can serve process-global file-backed resources without session isolation. Earlier review discussion shows the advertised `mcp.tool.result` issue was deferred/removed and the initial registration-path concern was addressed, but the remaining security and scoping concerns need maintainer review before this protocol surface lands.

## What This Changes

The PR adds gateway-side MCP Apps support through new `mcp.*` WebSocket methods, HTTP loopback resource methods, tool/resource `_meta.ui` metadata, an in-process `ui://` resource registry, docs, and tests.

## Best Possible Solution

Rework the PR on current main so MCP Apps support preserves the existing tool-execution security contract, scopes resource listing/reads to the caller's resolved session/account context or removes file-backed/global resource serving, and documents the remaining host-side/client limitations clearly before any core protocol surface is advertised.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Run hooks before WS MCP tool calls:** `src/gateway/server-methods/mcp.ts:143`
  - body: `mcp.tools.call` executes the resolved tool directly with the raw arguments. Current MCP loopback and HTTP tool invocation both run `runBeforeToolCallHook` first, which is where blocking hooks and adjusted params are applied, so this new WS surface can bypass the existing tool-call guard path.
  - confidence: 0.91
- **[P1] Scope resource reads to the caller's session:** `src/gateway/server-methods/mcp.ts:240`
  - body: `mcp.resources.read` resolves any URI from the process-global registry after syncing the caller's cache, but the registry can also contain resources from other active sessions and supports file-backed content. A read-scoped caller can enumerate/read another session's registered `ui://` resource once it is in the shared registry.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces security-sensitive tool execution and resource-serving surfaces that bypass existing hooks and lack session isolation.

Concerns:

- **[high] WS tool calls bypass before-tool-call hooks:** `src/gateway/server-methods/mcp.ts:143`
  - body: The new `mcp.tools.call` handler calls `tool.execute` directly instead of running `runBeforeToolCallHook`, unlike the existing MCP loopback and `/tools/invoke` paths. That can bypass hook-based blocking, approval, and parameter-adjustment behavior.
  - confidence: 0.91
- **[high] Global MCP resource reads can cross session boundaries:** `src/gateway/server-methods/mcp.ts:240`
  - body: Resources are stored in a process-global registry and can be backed by files, while `mcp.resources.read` serves by URI without verifying the resource belongs to the requested session. This creates a plausible cross-session content exposure path.
  - confidence: 0.86
