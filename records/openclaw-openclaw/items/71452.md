## Summary

Keep open. Current main still has the observable mismatch: callers can pass limits for message reads, pins, and search, but the shared CLI formatter still truncates message-like tables to 25 rows and pagination metadata is only partially exposed by providers.

## What This Changes

Request support for caller-controlled limits and pagination for message list/read/list-pins surfaces instead of a hardcoded 25-row output cap.

## Best Possible Solution

Implement an additive message action contract that threads caller limits into text rendering, preserves the default 25-row display when no limit is supplied, and exposes provider pagination metadata consistently without rewriting unrelated provider behavior.

## Reproduction Assessment

Yes. A focused unit check can pass read, list-pins, or Discord-style search payloads with more than 25 message-like entries into formatMessageCliText and observe truncation from messages.slice(0, 25); tests were not run because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to thread existing caller limits and next-page metadata through the action/formatter boundary, with provider-specific support added only where the existing APIs already expose pagination.
