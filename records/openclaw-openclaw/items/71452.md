## Summary

Keep open. Current main still has the observable CLI mismatch: message rows are rendered through a shared hard cap of 25 even though the CLI accepts larger limits for read, pins, and search. Provider-side pagination is only partial and does not remove the shared formatter cap; hclsys's comment outlines a reasonable narrow implementation path.

## What This Changes

Request that message read/list-pins/search and related chat listing flows honor requested limits and expose pagination instead of a fixed 25-item display cap.

## Best Possible Solution

Implement a narrow message action contract fix: make the renderer limit configurable from the requested action limit, keep the default 25-row behavior when no limit is supplied, propagate list-pins limits where providers support them, expose next-page metadata consistently, and cover read/list-pins/search with regression tests.

## Reproduction Assessment

Yes. A focused unit reproduction can call formatMessageCliText with a read, list-pins, or search payload containing more than 25 messages and observe that renderMessageList truncates the table to 25 rows on current main.

## Solution Assessment

Yes, the proposed direction in the discussion is the right shape: thread the existing limit through the formatter and provider actions, preserve the default cap for backward compatibility, and add non-breaking pagination metadata rather than a broad core rewrite.
