## Summary

Keep open. Current main still has the reported message CLI contract bug: the shared formatter renders at most 25 message rows, the read, pins, and search text paths still route through it, and provider-side limit/pagination support remains incomplete rather than a shipped end-to-end solution.

## What This Changes

Request support for caller-controlled limits and pagination in message read/list-pins/search and related chat/message listing instead of truncating rendered output at a hardcoded 25 items.

## Best Possible Solution

Implement a narrow, backwards-compatible message action contract update: thread the requested display limit into the formatter, keep the existing default when no limit is supplied, propagate list-pins limits where providers support them or can safely post-filter, and expose next-page metadata consistently for providers that return cursors.

## Reproduction Assessment

Yes. A focused unit check can pass more than 25 message-like rows through formatMessageCliText for read, list-pins, or search and observe truncation through renderMessageList; I did not run tests because this review was required to stay read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to parameterize the shared renderer and action dispatch while adding provider cursor metadata additively; a broad provider rewrite is not needed to remove the observable CLI cap.
