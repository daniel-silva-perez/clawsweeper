## Summary

Keep open. Current main still hard-caps message CLI text rendering at 25 rows while limit-bearing `read`, `pins`, and `search` paths still use that renderer; provider-side pagination exists only in pieces and does not satisfy the shared CLI/action contract.

## What This Changes

The issue requests pagination and caller-controlled limits for message list/read/pins/search output instead of a hardcoded 25-row cap.

## Best Possible Solution

Implement a narrow, additive message action contract update: pass caller limits into the shared formatter, keep the default 25-row display when no limit is supplied, propagate limits through provider list-pins/read/search paths where supported, surface next-page metadata consistently, and document provider caps.

## Reproduction Assessment

Yes. A focused unit check can call `formatMessageCliText` with `read`, `list-pins`, or Discord-style `search` payloads containing more than 25 messages and observe truncation from `messages.slice(0, 25)`; I did not run tests because this review was required to stay read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to thread existing caller limits and additive pagination metadata through the formatter/action boundary, not to rewrite provider actions wholesale or close this as provider-only work.
