## Summary

Keep this issue open. Current main still has the shared text formatter hard-capping message rows at 25, and the affected read, list-pins, and search paths still route through that capped renderer. Provider-side pagination exists in places, but it does not solve the observable CLI/tool output contract requested here.

## Best Possible Solution

Keep the issue open and implement the narrow message action contract fix: thread the requested display limit into `formatMessageCliText` and `renderMessageList`, remove or configure the hardcoded 25-row cap, propagate limits and page tokens through read/list-pins/search where providers support them, expose next-page metadata consistently, and add regression tests for rendering more than 25 rows for read, list-pins, and search while preserving existing provider pagination behavior.
