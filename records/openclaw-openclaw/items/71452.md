## Summary

Keep open. Current main still has the reported message CLI contract bug: the shared renderer hard-caps text output at 25 rows while limit-capable actions continue to route through that renderer, and provider-side pagination/limit support remains partial rather than a complete fix.

## What This Changes

The issue requests pagination and caller-controlled limits for message/chat listing actions instead of a fixed 25-row message renderer cap.

## Best Possible Solution

Implement a narrow message action contract update: pass the requested display limit into the formatter, preserve the default 25-row display when no limit is supplied, propagate list-pins limits where provider APIs or local post-filtering support it, add non-breaking next-page metadata for providers with cursors, and document any provider maximums.

## Reproduction Assessment

Yes. A focused unit check can call `formatMessageCliText` for `read`, `list-pins`, or `search` results containing more than 25 items and observe truncation because `renderMessageList` maps `messages.slice(0, 25)`. I did not run tests because this review was required to be read-only.

## Solution Assessment

Yes. The narrow direction already discussed on the issue is the maintainable fix: thread the existing caller limit through the formatter and action dispatch, keep default behavior for omitted limits, and add cursor metadata without a broad core rewrite.

Fix https://github.com/openclaw/openclaw/issues/71452. Add non-breaking pagination metadata only where a provider already exposes a cursor/next link, and document provider maximums instead of inventing unsupported pagination. Add regression tests proving more than 25 rows render for read/list-pins/search when a higher limit is requested and that omitted limits still cap at 25. Update `docs/cli/message.md` if the user-visible contract changes. Add a single-line changelog fix entry if user-facing behavior changes, crediting the human reporter/contributor and not bots. Do not introduce new dependencies or broad plugin API rewrites.
