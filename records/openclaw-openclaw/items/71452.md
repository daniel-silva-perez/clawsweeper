## Summary

Keep open. Current main still has the reported message CLI contract bug: the shared formatter hard-caps text output at 25 rows while `read`, `list-pins`, and `search` continue to route through that formatter, and provider-side limit or pagination support remains partial.

## What This Changes

The issue requests pagination and caller-controlled limits for message/chat listing actions instead of a fixed 25-row message renderer cap.

## Best Possible Solution

Implement a narrow message action contract update: pass the requested display limit into the formatter, preserve the default 25-row display when no limit is supplied, propagate list-pins limits where provider APIs or local post-filtering support it, add non-breaking next-page metadata for providers with cursors, and document provider maximums.

## Reproduction Assessment

Yes. A focused unit check can call `formatMessageCliText` for `read`, `list-pins`, or `search` results containing more than 25 items and observe truncation because `renderMessageList` maps `messages.slice(0, 25)`. I did not run tests because this review was required to stay read-only.

## Solution Assessment

Yes. The issue discussion's proposed direction is the narrow maintainable fix: thread the existing caller limit through formatter/action dispatch, keep default behavior for omitted limits, and add cursor metadata without a broad core rewrite.
