## Summary

Keep open. Current main still has the observable mismatch: limit-bearing message read, pins, and search requests can flow through providers, but the shared CLI formatter still renders only the first 25 message-like rows and provider pagination metadata is not exposed consistently.

## What This Changes

The issue requests that message read/list-pins/search and chat listing paths honor caller limits and support pagination instead of truncating rendered message lists at 25 rows.

## Best Possible Solution

Implement an additive message action contract that threads caller limits into text rendering, preserves the default 25-row display when no limit is supplied, and exposes provider pagination metadata consistently where providers already return next-page cursors.

## Reproduction Assessment

Yes. A focused unit check can feed more than 25 read, list-pins, or Discord-style search messages into formatMessageCliText and observe truncation from messages.slice(0, 25); source inspection also shows the CLI limit options still reach capped rendering paths.

## Solution Assessment

Yes. The narrow maintainable solution is to pass the existing caller limit through the message action and formatter boundary, then add provider-specific page-token fields only where the existing provider APIs already expose pagination.
