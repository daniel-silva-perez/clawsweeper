## Summary

Keep open. Current main still has the reported CLI/message-action mismatch: the shared message formatter hard-caps text output at 25 rows while read, list-pins, and search still route through it and the CLI exposes larger limits. The issue discussion already contains a compatible implementation plan, and I did not find a paired open PR that closes this issue.

## What This Changes

Request to make message read/list-pins/search and related MS Teams listing paths honor requested limits and expose pagination instead of truncating rendered output at 25 rows.

## Best Possible Solution

Implement a narrow message action contract fix: make the renderer limit configurable from the requested action limit, keep the default 25-row behavior when no limit is supplied, propagate list-pins limits where providers support them, expose next-page metadata consistently, and cover read/list-pins/search with regression tests.

## Reproduction Assessment

Yes. A focused unit reproduction can call formatMessageCliText with read, list-pins, or search payloads containing more than 25 messages and observe that current main truncates the rendered table because renderMessageList uses messages.slice(0, 25).

## Solution Assessment

Yes. The discussion's proposed direction is the right shape: thread the existing limit through the CLI formatter and provider actions, preserve default behavior for callers that omit a limit, and add non-breaking pagination metadata instead of a broad core rewrite.
