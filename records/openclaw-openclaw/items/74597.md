## Summary

Keep open. The reporter narrowed the original incident to a manual source-level global install, but current main still lacks the requested guard: update/status classify a symlinked npm-global package root that resolves to a git checkout as a git install, and doctor only compares service entrypoint realpaths instead of flagging the mixed package/source layout.

## What This Changes

Request diagnostic and update/status/doctor hardening so an npm-global OpenClaw install cannot silently remain a symlink to a mutable source checkout and later break gateway/package layout expectations.

## Best Possible Solution

Add a shared install-layout diagnostic that records both the lexical package/global root and its realpath, detects package roots that unexpectedly resolve into a git/source checkout, and surfaces clear status, doctor, update pre/post verification, and repair guidance while preserving intentional dev-channel behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74597. Current main can classify a lexical npm-global OpenClaw package root that is a symlink to a git/source checkout as a git install, and doctor only compares entrypoint realpaths. Add a shared install-layout diagnostic that records the lexical package/global root and resolved realpath, detects when an npm/pnpm/bun package root unexpectedly resolves into a git/source checkout, and surfaces this in `openclaw status`, `openclaw update status --json`, `openclaw doctor`, and update pre/post verification with clear repair guidance. Do not flag normal package-manager symlinks such as pnpm `.pnpm/openclaw@.../node_modules/openclaw` unless the target is a git/source checkout. Also make skipped dirty git/source update results provide a reliable failing signal for JSON/caller paths if needed by the diagnostic contract. Add focused tests and a user-facing changelog fix entry credited with `Thanks @davelutztx`. Do not change unrelated package install, plugin sync, or service restart behavior.
