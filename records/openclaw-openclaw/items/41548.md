## Summary

Keep this open. Current main has Gateway session-event primitives and shared session routing, but the bundled TUI still does not subscribe to or handle live `session.message` / `sessions.changed` events, and its visible chat path remains scoped to the active session. The same-author related issue #40678 is still open, #51691 was closed back to this issue as the broader tracker, and open maintainer PR #73402 is only a narrow implementation candidate for active-session transcript updates, not the full TUI/Telegram/iMessage unified inbox request.

## Best Possible Solution

Keep this open as the canonical broader TUI/Gateway product tracker unless maintainers manually consolidate it with #40678. Review or replace #73402 for the narrow active-session `session.message` live-update gap, then design the remaining unified inbox/observer mode as an opt-in TUI feature using `sessions.subscribe` and/or `sessions.messages.subscribe`, with channel/session labels and privacy-aware routing boundaries.

## Risks / Open Questions

- Naive cross-surface mirroring can leak private DM, group, or channel context across local TUI, Telegram, and iMessage. Any implementation should be opt-in, source-labeled, and respect pairing, allowlists, route pinning, and DM isolation.
- The active implementation candidate #73402 has unresolved security-review discussion around transcript subscription authorization, event-rate/history reload DoS, terminal text sanitization, and subscription confusion; those need maintainer review before landing the narrow TUI fix.
- There is overlap with open #40678 and open PR #73402, so maintainers should consolidate the product scope rather than close this tracker automatically.
