## Summary

Keep open. Current main still has the stale assistant-media fallback path that can append a `gateway-injected` assistant transcript message without resolved media, and two open PRs already target this issue with closing syntax, so the issue should wait for one fix PR to land.

## What This Changes

The issue reports duplicate Control UI/WebChat assistant replies when a stale or non-playable assistant-media/TTS payload appends a second `:assistant-media` transcript entry containing the final answer text.

## Best Possible Solution

Land one narrow gateway fix that skips agent-run assistant-media transcript appends unless resolved media content exists, while preserving resolved auto-TTS audio supplements and non-agent command media fallback behavior.

## Reproduction Assessment

Yes. The issue gives a concrete stale-media/TTS sequence, and current source shows the static path: a media-bearing agent payload with no resolved media can still fall back to text and append a `:assistant-media` transcript message.

## Solution Assessment

Yes. The linked PR approach is the narrow maintainable fix: require actual persisted media content before writing the supplemental assistant-media transcript entry, with a regression test for stale local audio.
