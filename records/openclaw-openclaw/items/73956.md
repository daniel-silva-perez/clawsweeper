## Summary

Keep open. Current main still has a source-level path where an agent-run media-bearing reply with stale/non-playable media can fall back to visible assistant text and append a second `:assistant-media` transcript message, and two open PRs already target this issue with closing intent.

## What This Changes

Prevent Control UI/WebChat from storing duplicate gateway-injected assistant replies through the assistant-media path when TTS/media references are stale or non-playable.

## Best Possible Solution

Land one narrow gateway fix that appends the supplemental assistant-media transcript only when resolved non-text media content exists, while preserving the normal model transcript and the non-agent command/TTS media fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final payload with visible text plus a stale trusted local audio `mediaUrl`/`mediaUrls` and no `spokenText`; current main can fall back to that visible text and append an `:assistant-media` transcript message.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable solution: guard the agent-run supplemental append on resolved media content, without changing TTS configuration defaults, media serving, or command/non-agent media behavior.
