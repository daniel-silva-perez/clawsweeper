## Summary

Keep open. Current main still has the agent-run assistant-media append path that can fall back to visible text and append a `:assistant-media` transcript message when no playable media content is resolved, and two open PRs already target this issue with closing syntax.

## What This Changes

The issue reports duplicate Control UI/WebChat assistant replies after the assistant-media/TTS delivery path stores a second gateway-injected assistant message following the normal final answer.

## Best Possible Solution

Land one narrow gateway fix that appends the supplemental assistant-media transcript only when resolved non-text media content exists, while preserving the normal model transcript and the non-agent command/TTS fallback path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run payload with visible text plus a stale trusted local audio path: the media builder resolves no playable content, but the append path can still persist fallback text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix: guard the supplemental append on resolved media content, and keep command/non-agent media replies able to carry text when there is no model transcript.
