## Summary

Keep open. Current main still has the agent-run assistant-media append path that can persist a fallback text transcript with an `:assistant-media` idempotency key when a media-bearing payload does not resolve to playable media, and two open PRs already target this issue with closing syntax.

## What This Changes

The issue reports duplicate Control UI/WebChat assistant replies after the assistant-media/TTS delivery path appends a second gateway-injected transcript message that repeats the final answer text.

## Best Possible Solution

Land one narrow gateway fix that appends supplemental agent-run assistant-media transcript entries only after resolved non-text media content exists, while preserving normal model transcripts and non-agent command/TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final or block payload with visible text plus a stale or unembeddable media URL: the media-bearing guard passes, media resolution returns no media message, and the append path can persist fallback text with an `:assistant-media` key.

## Solution Assessment

Yes. The best fix direction is the one already proposed by the linked PRs: guard the supplemental append on resolved media content rather than changing TTS configuration, transcript storage, or Control UI rendering semantics.
