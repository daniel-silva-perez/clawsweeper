## Summary

Keep open. Current main still has the agent-run assistant-media append path that can persist a second `:assistant-media` assistant transcript from fallback text when media references do not resolve, and two open PRs already target this issue with closing syntax.

## What This Changes

The issue reports duplicate Control UI/WebChat assistant replies caused by the assistant-media transcript path appending a second gateway-injected text message after the normal assistant final answer.

## Best Possible Solution

Land one narrow gateway fix, choosing one canonical PR, that appends supplemental agent-run assistant-media transcript entries only when resolved non-text media content exists while preserving the normal model transcript and non-agent command/TTS fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final or block payload with visible text plus a stale or unembeddable media URL: the media-bearing guard passes, media resolution returns no media message, and the append path can persist fallback text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix because it guards the supplemental append on resolved media content instead of changing TTS configuration, transcript storage, or Control UI rendering semantics.
