## Summary

Keep open. Current main still has a plausible text-only fallback in the agent-run assistant-media transcript append path, and two open PRs explicitly target this issue with closing syntax, so the issue should remain open until one of those fixes is reviewed and merged.

## What This Changes

Control UI/WebChat should stop storing a second `gateway-injected` assistant message from the assistant-media/TTS supplement path when no playable media resolves.

## Best Possible Solution

Land one narrow gateway fix that appends an assistant-media transcript only after resolved non-text media content exists, with a regression for a stale trusted local audio path and a preserved non-agent command TTS fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final payload with visible text plus a stale trusted local audio `mediaUrl`; current main can still fall back to that visible text and append an `:assistant-media` transcript message.

## Solution Assessment

Yes. The narrowest maintainable fix is to guard the supplemental assistant-media append on resolved media content, while leaving the normal model transcript and non-agent command/TTS fallback behavior intact.
