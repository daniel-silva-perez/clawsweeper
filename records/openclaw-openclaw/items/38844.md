## Summary

Keep open. Current main has real mitigations for this report, especially dispatcher-path timeout wording, selector/batch action support, direct `inputRef`/`element` upload support, and browser skill guidance for stale refs. The remaining upload-with-trigger path is still stale-prone: `/hooks/file-chooser` arms the upload and then clicks the original `ref` without internal re-snapshot/re-resolution and without passing the route timeout or abort signal into the trigger click.

## Best Possible Solution

Keep this open and land a focused browser plugin fix: make upload-with-trigger robust as one request, pass timeout/abort context into trigger activation, re-snapshot or re-resolve stale trigger refs where current ref metadata supports it, preserve stale-ref/action errors, and add regression tests. Keep #51395 separate because it is about non-standard file-input fallbacks, not stale trigger activation.

## Clownfish Work Prompt

Fix the remaining browser file-chooser stale-trigger bug from this issue. In the browser plugin, make `/hooks/file-chooser` with `ref` behave as one robust upload-with-trigger request: preserve existing upload-root validation, arm the upload, activate the trigger with the same request timeout/AbortSignal context, and preserve specific stale-ref/action errors instead of mapping them to browser-control reachability failures. Where current ref metadata supports it, re-snapshot or re-resolve a stale trigger once before firing; do not add broad retry loops. Add focused tests that fail on current main: upload-with-ref should pass `timeoutMs` and request signal into trigger activation, stale trigger failures should remain stale/action-specific, and current happy paths for plain upload, `inputRef`, and `element` should remain green. Do not implement the separate ByteUploader/DataTransfer/clipboard fallback from #51395, do not weaken upload path validation, and do not move browser-specific behavior into core. Update docs/tool guidance only if the user-facing upload-with-ref behavior changes, and add a user-facing changelog entry with `Thanks @tigicion` if behavior changes are landed.
