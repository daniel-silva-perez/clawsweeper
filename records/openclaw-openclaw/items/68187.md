## Summary

Keep open. Current main and the latest stable tag still cache a per-session MCP client/transport and forward SSE-backed tool calls through that cached SDK client without a stale-session detector, per-server reconnect, catalog refresh, or retry. The related PR #71545 is open and unmerged, and its own body/diff scope is streamable-http only, so it does not yet resolve this SSE report.

## What This Changes

The issue requests automatic recovery when an SSE-backed MCP server restarts so OpenClaw invalidates stale client session state instead of failing subsequent calls with `Session not found`.

## Best Possible Solution

Add one bounded per-server recovery path in the session MCP runtime: when an SSE or streamable-http MCP call fails with a known lost-session error, dispose and recreate only that server's client/transport, refresh or validate catalog metadata as needed, retry the original tool call once, and leave ordinary server-down failures unchanged.

## Reproduction Assessment

Yes. The issue gives a concrete successful-call, downstream SSE restart, second-call path, and current main has an SSE probe harness plus a cached session runtime that can be extended into a deterministic regression test.

## Solution Assessment

Yes, a narrow per-server reconnect-and-retry in `src/agents/pi-bundle-mcp-runtime.ts` is the best maintainable direction. A full gateway restart workaround is too broad, and #71545 should be coordinated with but does not currently cover SSE.

Fix SSE-backed bundled MCP session recovery for openclaw/openclaw#68187. Current main caches session MCP clients in `src/agents/pi-bundle-mcp-runtime.ts`; after an SSE MCP server restart, `callTool` can keep using stale session state and fail with `Session not found` until the gateway restarts.
