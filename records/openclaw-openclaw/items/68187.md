## Summary

Keep open. Current main still reuses session-scoped bundled MCP clients and forwards SSE-backed tool calls through the cached SDK client without stale-session detection, reconnect, catalog refresh, or retry; the related streamable-http PR is open, unmerged, and explicitly does not cover SSE.

## What This Changes

Detect and recover stale SSE-backed MCP client sessions after a downstream MCP server restart instead of requiring an OpenClaw gateway restart.

## Best Possible Solution

Add one bounded stale-session recovery path in the session MCP runtime: when an SSE or HTTP MCP call fails with a known lost-session error, dispose and recreate only that server's client/transport, refresh or validate the catalog as needed, retry the original tool call once, and cover it with a restart-capable SSE regression test. Coordinate the helper shape with #71545 if that PR lands first.

## Reproduction Assessment

Yes. The issue gives a concrete restart-after-successful-call path, and current source confirms the same session runtime and client are reused across calls; a regression test can extend the existing SSE probe harness to restart or reset the downstream session before the second call.

## Solution Assessment

No current solution is in main. The narrow maintainable fix belongs in src/agents/pi-bundle-mcp-runtime.ts with focused tests; #71545 is a useful adjacent implementation candidate but is currently streamable-http-only and does not solve this SSE report.

Current main caches session MCP runtimes in src/agents/pi-bundle-mcp-runtime.ts and callTool forwards directly to the cached SDK client; after an SSE MCP server restart, stale POST/session errors such as "Session not found" should cause OpenClaw to recreate the affected MCP client/transport and retry the original tool call once.
