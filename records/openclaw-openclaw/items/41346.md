## Summary

Keep this issue open. Current main still lets cron.add callers omit enabled, normalizes and creates such jobs as enabled by default, exposes cron registration through agent/Gateway/plugin paths, and resolves isolated cron jobs without payload.model through the agent/default model selection. I found no current consent/diff flow, expensive inherited-model guard, doctor dry-run estimate for new jobs, or cron-specific usage/quota circuit breaker. The prior ClawSweeper keep-open review remains accurate: the relevant cron surfaces have no diff from that reviewed SHA to current main.

## Best Possible Solution

Implement an origin-aware cron safety contract in core. Persist origin/owner metadata, require explicit activation or a consent/diff flow for newly registered recurring external/plugin/agent-tool jobs, show cadence/resolved model/model source/estimated call volume in doctor or update dry-runs, warn or block high-cost inherited defaults unless the job explicitly sets payload.model, and add a configurable usage/quota circuit breaker that can pause non-critical recurring cron jobs. Cover Gateway cron.add, CLI cron add/edit, the agent cron tool, plugin hook getCron access, bundled managed reconciliation, docs, and migration/backcompat tests.

## Risks / Open Questions

- A blanket default-disabled change would likely break existing user-created reminders, one-shot jobs, and managed recurring jobs that rely on enabled-by-default behavior.
- The plugin hook cron contract currently requires enabled while Gateway and agent-tool paths allow or default it, so a fix needs a compatibility and migration story across public surfaces.
- A quota or usage circuit breaker needs provider-specific usage-window semantics and a clear definition of critical versus pausable cron jobs to avoid stopping expected automation incorrectly.
