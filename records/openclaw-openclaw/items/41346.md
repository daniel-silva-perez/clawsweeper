## Summary

Keep open. Current main still allows cron creation without an explicit disabled/consent state, normalizes omitted `enabled` to true, has bundled plugin code that creates and re-enables managed recurring cron jobs, and still resolves isolated cron jobs without `payload.model` through the agent/default model path. I found usage/cost recording, but not an origin-aware consent plan, expensive inherited-model guard, doctor/update dry-run estimate, or quota circuit breaker.

## What This Changes

The issue asks OpenClaw to stop externally registered recurring cron jobs from auto-enabling without confirmation, inheriting expensive default models silently, and running without usage/cost guardrails.

## Best Possible Solution

Design and implement an origin-aware cron safety contract in core: persist owner/origin metadata, require explicit activation or an auditable consent plan for newly registered external/plugin recurring jobs, surface cadence/resolved model/model source/estimated daily call volume in doctor/update dry-runs, warn or block expensive inherited defaults unless recurring jobs set `payload.model`, and add a configurable quota/usage circuit breaker that can pause non-critical recurring cron work.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call `cron.add` without `enabled` through the Gateway or agent tool and current main normalizes it to `enabled: true`; the bundled memory-core reconciliation path also creates an enabled recurring isolated job. I did not run a live token-burn scenario because this review is read-only, but the model-selection code and tests confirm the default-model inheritance path.

## Solution Assessment

Unclear as an implementation because the issue lists several safeguards with compatibility tradeoffs, but the direction is sound. The safest solution is not a blanket default flip; it should be origin-aware, migration-aware, and explicit about which plugin/managed jobs may auto-enable and which require user consent.
