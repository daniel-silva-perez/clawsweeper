## Summary

Keep this issue open. Current main still allows cron.add callers to omit enabled, normalizes missing enabled to true, documents the agent cron tool default as true, and keeps isolated cron model resolution falling through to agent/default model selection when payload.model is absent. I found no current cron consent/diff flow, expensive inherited-model guard, doctor dry-run estimate for new jobs, or cron-specific session-usage circuit breaker.

## Best Possible Solution

Implement an origin-aware cron safety contract in core: persist cron origin/owner metadata, require explicit activation or a consent/diff flow for newly registered recurring external/plugin/agent-tool jobs, show cadence/resolved model/model source/estimated call volume in doctor or update dry-runs, warn or block high-cost inherited defaults unless the job explicitly sets payload.model, and add a configurable usage/quota circuit breaker that can pause non-critical recurring cron jobs. Cover Gateway cron.add, the agent cron tool, plugin hook cron access, bundled managed reconciliation, CLI/docs, and migration/backcompat tests.

## Risks / Open Questions

- A blanket default-disabled change could break existing user-created reminders, one-shot jobs, and managed recurring jobs that rely on current enabled-by-default behavior.
- The plugin hook type currently requires enabled while Gateway and agent-tool paths allow/default it, so the fix should align public contracts and preserve compatibility.
- A quota circuit breaker will need provider-specific usage-window semantics and clear opt-out/critical-job handling to avoid pausing expected automation incorrectly.
