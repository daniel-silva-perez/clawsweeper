## Summary

Keep this issue open. Current main still defaults cron creation to enabled unless the caller explicitly disables the job, isolated agent cron runs without `payload.model` still resolve through the agent/default model path, and the current docs/source only provide reporting or legacy-repair surfaces rather than a consent diff, expensive inherited-model guard, doctor dry-run estimate, or cron usage circuit breaker. The provided related items cover different cron/UI/retry/cache topics and do not supersede this report.

## Best Possible Solution

Implement an origin-aware cron safety contract in core: persist owner/origin metadata, require explicit activation or an interactive/non-interactive consent plan for newly registered external/plugin/agent-tool recurring jobs, surface cadence/resolved model/model source/estimated daily call volume in doctor/update dry-runs, warn or block expensive inherited defaults unless a recurring job explicitly sets `payload.model`, and add a configurable quota/usage circuit breaker that can pause non-critical recurring cron jobs. Cover Gateway `cron.add`, CLI add/edit, the agent cron tool, plugin `getCron` access, doctor/store migration, docs, and backcompat tests.

## Risks / Open Questions

- A blanket default-disabled change could break existing reminders, one-shot jobs, and managed recurring jobs that rely on enabled-by-default behavior; the fix should be origin-aware and migration-aware.
- The public/plugin-adjacent cron surfaces are not perfectly aligned: Gateway and agent-tool paths allow omitted `enabled`, while plugin hook types require it, so a compatibility story is needed across API callers.
- A usage/quota circuit breaker needs provider-window semantics and a clear definition of critical versus pausable cron work to avoid stopping expected automation incorrectly.
