## Summary

Keep this issue open. Current main still exposes a public/plugin cron creation path where `enabled` is optional and defaults to true, memory-core still creates and re-enables a managed recurring cron job, isolated cron jobs without `payload.model` still resolve through the agent/default model path, and I did not find the requested consent/diff flow, inherited expensive-model guard, doctor dry-run estimate, or session-usage circuit breaker. Some adjacent improvements exist, including explicit cron model override validation/fallback handling, run telemetry/cost recording, recurring error backoff, failure alerts, and auth-profile cooldowns, but they do not close the central safety contract requested here.

## Best Possible Solution

Implement a generic cron safety contract in core: persist cron origin/owner metadata, require explicit activation or a consent/diff flow for newly registered recurring external/plugin jobs, show cadence/resolved model/model source/estimated call volume in doctor or update dry-runs, warn or block high-cost inherited defaults unless a job explicitly chooses a model, and add a configurable usage/quota circuit breaker that can pause non-critical recurring cron jobs. Cover the public cron API, plugin hook surface, bundled managed cron reconciliation, CLI/docs, and migration/backcompat behavior with focused tests.
