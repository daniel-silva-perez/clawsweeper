## Summary

Keep open. Current main still defaults cron jobs to enabled when callers omit `enabled`, isolated cron jobs without `payload.model` still resolve through agent/default model selection, and the plugin/Gateway cron surfaces do not yet provide origin-aware consent, expensive-model guards, doctor/update dry-run estimates, or a usage/quota circuit breaker.

## Best Possible Solution

Implement an origin-aware cron safety contract in core: persist owner/origin metadata, require explicit activation or an interactive/non-interactive consent plan for newly registered external/plugin/agent-tool recurring jobs, show cadence/resolved model/model source/estimated daily call volume in doctor/update dry-runs, warn or block expensive inherited defaults unless a recurring job explicitly sets `payload.model`, and add a configurable quota/usage circuit breaker that can pause non-critical recurring cron jobs. Cover Gateway `cron.add`, CLI add/edit, the agent cron tool, plugin `getCron`, doctor/store migration, docs, and backcompat tests.

## Risks / Open Questions

- A blanket default-disabled change could break existing reminders, one-shot jobs, and managed recurring jobs that intentionally rely on enabled-by-default behavior; the fix should be origin-aware and migration-aware.
- Gateway, CLI, agent-tool, and plugin cron surfaces are not aligned today: some allow omitted `enabled`, while the plugin hook create input requires `enabled`, so compatibility needs a deliberate API design.
- A quota/usage circuit breaker needs provider-window semantics and a clear definition of critical versus pausable cron work to avoid stopping expected automation incorrectly.
