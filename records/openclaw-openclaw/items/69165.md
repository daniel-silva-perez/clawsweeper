## Summary

Keep open. Current main has retry_after-aware retry for individual Telegram send helper calls and a generic write-ahead delivery recovery queue, but it still does not implement the requested per-chat outbound pacing queue, Telegram outbound queue config/depth/drop policy, or non-blocking queued submit semantics. Related #42246 remains complementary batching/digest work, not a duplicate of transport-layer throttling.

## Best Possible Solution

Keep this issue open for a Telegram plugin design pass. The best path is a plugin-owned outbound pacing layer keyed by account/chat/topic that reuses the existing retry_after-aware retry runner, adds documented Telegram queue config and depth/drop behavior, and explicitly decides how queued/non-blocking submission interacts with the shared outbound delivery contract, delivery results, hooks, cron status, and bestEffort handling. Keep #42246 as the separate batching/digest discussion.

## Risks / Open Questions

- Changing Telegram delivery to non-blocking queue submit could alter shared outbound delivery results, message_sent hooks, cron/task success semantics, and bestEffort error behavior unless the contract is designed deliberately.
- A per-chat queue must define lifecycle and durability carefully: in-memory-only pacing can lose queued cron notifications on restart, while reusing the durable recovery queue without pacing metadata would not satisfy rate limiting or depth/drop semantics.
