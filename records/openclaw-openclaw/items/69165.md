## Summary

Keep open. Current main has retry_after-aware retry for some Telegram send helpers, but it does not implement the requested per-chat outbound queue, configurable per-chat pacing, queue depth/drop policy, or non-blocking submit semantics. The related #42246 batching request is complementary, not a duplicate, because this issue is specifically about transport-layer throttling and 429 backoff for Telegram outbound sends.

## Best Possible Solution

Keep this issue open and treat it as a Telegram plugin design task: add a plugin-owned outbound pacing/queue layer that can serialize by chat/topic, reuse the existing retry_after-aware retry machinery, define how queued/non-blocking delivery reports message IDs and failures, expose documented config for rate/depth/drop policy, and keep #42246 as the separate batching/digest discussion.
