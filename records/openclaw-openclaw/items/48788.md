## Summary

Keep open. Current main still has private, narrow filename parsing in media fetch code plus a Feishu-local Latin-1 recovery helper; it does not yet provide the requested exported, shared, multi-encoding Content-Disposition filename utility or full channel migration. The issue discussion and open PR #71517 are useful active context, but that PR is still the implementation candidate rather than merged proof that this tracker can close.

## What This Changes

The issue requests a centralized reusable Content-Disposition filename decoder with RFC 5987 handling, mojibake recovery, explicit multi-charset support, and channel adapter reuse.

## Best Possible Solution

Keep the issue open until a shared filename decoder lands in the appropriate media/plugin SDK surface with focused tests for RFC 5987, Latin-1/Windows-1252 mojibake recovery, explicit CJK charset vectors, and migration of the relevant channel download paths. Review PR #71517 on its own merits first, then use this tracker for any remaining shared-utility scope.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Direct runtime reproduction is not applicable for this feature-tracking request, but source inspection gives high confidence that the requested central exported utility is absent on current main while only partial local helpers exist.

## Solution Assessment

Is this the best way to solve the issue? Yes, the maintainable direction is a shared media filename decoder exported through the existing plugin/runtime surface, with Feishu as an initial consumer and broader channel migration added incrementally. The current main implementation is not sufficient yet, and PR #71517 should be reviewed before spawning duplicate repair work.
