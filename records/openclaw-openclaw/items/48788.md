## Summary

Keep open. Current main has a Feishu-specific Latin-1 to UTF-8 repair and a private fetchRemoteMedia Content-Disposition parser, but it does not yet provide the requested centralized, exported multi-encoding filename decoder with explicit GBK, Shift-JIS, EUC-KR, or GB18030 handling across channel adapters. The provided timeline also shows open PR #71517, “media: add shared filename decoder,” as the current implementation candidate.

## Best Possible Solution

Keep this issue open and use #71517 as the implementation path if it is still viable. The best fix is an exported shared media filename decoder, likely under src/media with an SDK/runtime export if plugin adapters need it, with tests for RFC 5987, Latin-1 UTF-8 mojibake recovery, valid Latin-1 preservation, JSON metadata preservation, explicit GBK/Shift-JIS/EUC-KR/GB18030 decoding, and fallback behavior before migrating channel download paths.
