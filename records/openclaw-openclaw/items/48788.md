## Summary

Keep open. Current main still has only a private, narrower Content-Disposition parser in src/media/fetch.ts and a Feishu-local Latin-1-to-UTF-8 recovery helper; it does not yet provide the requested centralized, exported multi-encoding filename utility or migrate channel adapters to it. Open PR #71517 is the active implementation candidate and is explicitly scoped as partial/follow-up work for this issue.

## Best Possible Solution

Keep this issue open. Review and land PR #71517 if it remains viable, then use this issue to track the remaining central filename utility scope: exported shared decoder, RFC 5987 language-tag/charset handling, Latin-1/Windows-1252 mojibake recovery, explicit GB18030/GBK/Shift-JIS/EUC-KR coverage, tests, and migration of channel download paths or SDK/runtime exports where needed.
