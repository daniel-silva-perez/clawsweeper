## Summary

Keep open. Current main still has only a private fetch-local Content-Disposition parser plus a Feishu-local Latin-1-to-UTF-8 recovery helper; it does not yet provide the requested centralized, exported, multi-encoding filename utility or complete channel migration. Open PR #71517 is the active partial implementation candidate and explicitly says this issue should remain the follow-up tracker.

## Best Possible Solution

Keep this issue open. Review and land or close PR #71517 on its own merits, then use this issue to track the remaining shared filename utility scope: exported decoder contract, RFC 5987 charset/language handling, Latin-1/Windows-1252 mojibake recovery, explicit GBK/GB18030/Shift-JIS/EUC-KR coverage, tests, and migration of relevant channel download paths or SDK/runtime exports.
