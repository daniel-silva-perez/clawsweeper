## Summary

Keep open. Current main still exits from the `logs --follow` polling loop on non-fallback Gateway errors, and open PR #75059 is already linked with closing syntax as the implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

`openclaw logs --follow` should retry transient Gateway transport disconnects with warning/backoff instead of terminating the log tail session, while keeping non-recoverable errors fatal.

## Best Possible Solution

Add or land a focused `logs --follow` retry/backoff path around the CLI polling loop for transient Gateway transport failures, with tests for explicit URL and implicit loopback behavior while preserving the current local-file fallback boundary.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to make `callGatewayFromCli("logs.tail")` reject with a Gateway close/timeout error during `logs --follow --url ...`; current main catches that failure and exits with status 1, and the existing explicit-URL test asserts the same fatal behavior.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in `src/cli/logs-cli.ts` around the follow polling loop or a small shared one-shot retry helper, while keeping non-recoverable Gateway/auth/config errors fatal.
