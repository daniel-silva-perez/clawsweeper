## Summary

Keep open. The prior ClawSweeper comment remains accurate after rechecking current main: OpenClaw now has a narrow implicit-loopback local log-file fallback, but `openclaw logs --follow` still exits on non-fallback Gateway transport failures instead of reconnecting with backoff.

## What This Changes

Request auto-reconnect/backoff behavior for `openclaw logs --follow` after transient Gateway disconnects, with fatal handling preserved for non-recoverable auth or configuration errors.

## Best Possible Solution

Add bounded reconnect/backoff behavior inside the `logs --follow` polling loop for transient Gateway transport failures, while keeping terminal auth/configuration errors fatal and preserving the current local file fallback contract. Cover default loopback and explicit URL cases in focused CLI tests, then update docs and changelog for the user-visible behavior change.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: in current main, a non-fallback `fetchLogs` rejection during the follow loop reaches `process.exit(1)`, and the explicit-URL test already asserts that fatal path for a closed Gateway transport error.

## Solution Assessment

Yes. A scoped retry/backoff wrapper around the follow polling fetch is the narrowest maintainable fix; reusing or mirroring the Gateway client's transient/error classification is preferable to broadening the local file fallback.
