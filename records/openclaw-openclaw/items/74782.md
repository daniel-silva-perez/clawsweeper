## Summary

Keep open. Current main still exits from the `logs --follow` polling loop on non-fallback Gateway errors, the docs/tests only cover the implicit-loopback local-file fallback, and open PR #75059 is already linked as the implementation candidate for this issue.

## What This Changes

Request that `openclaw logs --follow` retry transient Gateway transport disconnects with warning/backoff instead of terminating the tail session.

## Best Possible Solution

Add or land a focused `logs --follow` retry/backoff path for transient Gateway transport failures in the CLI polling loop, with tests for explicit URL and implicit loopback behavior plus docs/changelog updates if the user-facing behavior changes.

## Reproduction Assessment

Yes. The current code-level reproduction is a `fetchLogs`/`callGatewayFromCli` transport rejection inside the follow loop; current tests already assert that an explicit Gateway URL close error exits with `process.exit(1)`.

## Solution Assessment

Yes in direction: bounded retry/backoff belongs in `src/cli/logs-cli.ts` around the follow polling loop, while keeping non-recoverable auth/config errors fatal and preserving the existing local fallback boundary.
