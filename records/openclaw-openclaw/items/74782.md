## Summary

Keep open. Current main still treats non-fallback `logs --follow` Gateway transport failures as fatal, and open PR #75059 is already the linked implementation candidate, so closing this issue before that PR or an equivalent fix lands would be premature.

## What This Changes

The issue requests automatic reconnect/backoff for `openclaw logs --follow` when transient Gateway transport failures occur.

## Best Possible Solution

Land or adapt a focused CLI change that retries transient `logs --follow` transport failures with bounded backoff and warning output, while preserving fatal handling for non-recoverable Gateway errors and the existing local fallback boundary.

## Reproduction Assessment

Yes. A closed or timed-out `callGatewayFromCli("logs.tail")` request on an explicit `--url` path reaches the current catch block and exits; the existing explicit-URL test already asserts that behavior.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in `src/cli/logs-cli.ts` around the follow polling loop, with tests that cover transient retry, fatal errors, and local fallback preservation.
