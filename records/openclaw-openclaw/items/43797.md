## Summary

Keep this open. Current main still prunes expired sandbox runtimes and registry entries without removing the non-shared sandbox workspace directory, and open PR #43835 still specifically targets this bug with closing syntax.

## What This Changes

The issue requests that sandbox prune cleanup remove OpenClaw-owned workspace directories under `~/.openclaw/sandboxes` along with expired Docker sandbox runtimes.

## Best Possible Solution

Add a narrow cleanup step that deletes only OpenClaw-owned, non-shared sandbox workspace directories after successful or attempted runtime removal, with focused regression coverage for prune and recreate paths; review and land PR #43835 or replace it with an equivalent maintained branch.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with pruning enabled and an expired regular sandbox registry entry, `maybePruneSandboxes` removes the runtime and registry entry while the `sandboxWorkspaceDir` created by `ensureSandboxWorkspaceLayout` is left untouched.

## Solution Assessment

No for current main; it does not solve the leak. The maintainable solution is a guarded workspace cleanup step in the sandbox runtime cleanup path, ideally through review and landing of PR #43835 or a narrow replacement.
