## Summary

Keep open. Current main still prunes expired sandbox runtimes and registry entries without deleting the corresponding non-shared Docker sandbox workspace directory, and open PR #43835 explicitly targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests cleanup of expired Docker sandbox workspace directories under the sandbox workspace root when pruning removes the corresponding runtime.

## Best Possible Solution

Review and land PR #43835 or a narrow replacement that removes safe non-shared Docker sandbox workspace directories only after confirmed runtime removal, preserves shared/rw workspaces, handles stale registry metadata conservatively, and adds focused prune/recreate regression coverage.

## Reproduction Assessment

Yes. A code-level reproduction path is clear: with sandbox pruning enabled and an expired regular Docker sandbox registry entry, maybePruneSandboxes removes the runtime and registry entry while the host workspace directory created by resolveSandboxContext remains untouched.

## Solution Assessment

No. Current main does not solve the reported leak; the maintainable fix should live in the sandbox runtime cleanup path and be guarded by registry identity, workspace ownership, and confirmed runtime removal.
