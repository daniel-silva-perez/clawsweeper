## Summary

Keep open. Current main still removes expired sandbox runtimes and registry entries without deleting the corresponding Docker sandbox workspace directory, and open PR #43835 explicitly targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests cleanup of expired Docker sandbox workspace directories under the sandbox workspace root when pruning removes the corresponding runtime.

## Best Possible Solution

Land PR #43835 or an equivalent sandbox-owned fix that removes safe non-shared Docker sandbox workspace directories after runtime removal, also covers manual recreate/config refresh if required, and adds focused regression coverage for prune and recreate behavior.

## Reproduction Assessment

Yes. A code-level reproduction path is clear: with sandbox pruning enabled and an expired regular Docker sandbox registry entry, maybePruneSandboxes removes the runtime and registry entry while the host workspace directory created by resolveSandboxContext remains untouched.

## Solution Assessment

Unclear for the open PR until reviewed. The direction of using sandbox scope/workspace helpers is right, but the complete fix should cover both automatic prune and the documented recreate/config-refresh workflow without deleting shared or rw workspaces.
