## Summary

Keep this issue open. Current main still removes sandbox runtimes and registry entries without deleting Docker sandbox workspace directories, and open PR #43835 remains an unmerged implementation candidate that references this issue with closing syntax.

## Best Possible Solution

Keep this issue open and review, land, or supersede #43835. The likely fix belongs in the sandbox owner surface: derive the non-shared workspace directory from `cfg.workspaceRoot` plus the registry `sessionKey`, remove it with safe recursive cleanup after runtime removal, skip shared scope and in-use workspaces, and add focused regression tests for prune plus manual recreate/removal behavior.

## Risks / Open Questions

- Current main can still leak non-shared Docker sandbox workspace directories after prune removes the runtime and registry entry.
- A fix needs to avoid deleting shared-scope workspaces and avoid removing agent-scope workspace state that may still be used by another active runtime/session.
- The manual `openclaw sandbox recreate` path appears to share the same workspace-retention behavior, so a complete fix should consider both automatic prune and explicit recreate/removal behavior.
