## Summary

Keep this open. Current main still removes expired sandbox runtimes and registry entries without deleting the corresponding non-shared sandbox workspace directory, and open PR #43835 still specifically targets this bug with closing syntax.

## What This Changes

The issue requests that sandbox pruning remove expired sandbox workspace directories as well as Docker containers.

## Best Possible Solution

Add a narrow cleanup path that deletes only OpenClaw-owned, non-shared sandbox workspace directories after runtime removal, preserves shared and rw workspaces, handles stale registry metadata conservatively, and adds focused prune/recreate regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with pruning enabled and an expired regular sandbox registry entry, maybePruneSandboxes removes the runtime and registry entry while the sandboxWorkspaceDir created by resolveSandboxContext is left untouched.

## Solution Assessment

No for current main; it does not solve the leak. The maintainable solution is a guarded workspace cleanup step in the sandbox runtime cleanup path, ideally by reviewing and landing PR #43835 or a narrow replacement.
