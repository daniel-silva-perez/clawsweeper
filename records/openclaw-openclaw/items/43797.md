## Summary

Keep this open. Current main still prunes expired sandbox runtimes and registry entries without deleting the corresponding non-shared sandbox workspace directory, and the linked implementation PR remains open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Sandbox pruning should remove expired non-shared sandbox workspace directories as well as the runtime containers and registry entries so disk usage and stale workspace state do not accumulate.

## Best Possible Solution

Review and land PR #43835 or a narrow replacement that deletes only owned non-shared sandbox workspace directories after runtime removal, preserves shared/rw workspaces, handles stale registry metadata conservatively, and adds focused prune/recreate regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with pruning enabled and an expired regular sandbox registry entry, maybePruneSandboxes removes the runtime and registry entry while the sandboxWorkspaceDir created by resolveSandboxContext remains untouched.

## Solution Assessment

No. Current main does not solve the leak; the maintainable fix should be in the sandbox runtime cleanup path with ownership and scope guards around filesystem deletion.
