## Summary

Keep this PR open, but do not merge it as-is. The direction addresses a real policy-parity bug, but the diff builds the full coding tool set for inline direct dispatch without carrying sandbox context, so sandboxed sessions can still diverge from the effective runtime policy.

## What This Changes

The PR changes inline `command-dispatch: tool` skill slash commands to use `createOpenClawCodingTools(...)` with runtime-policy session metadata, updates focused tests, and removes the old `openclaw-tools.runtime.ts` shim.

## Best Possible Solution

Inline tool dispatch should reuse the same effective tool construction as normal agent turns, including sandbox context or fail-closed behavior for sandboxed sessions, target-session metadata, owner/profile/group/provider/subagent filtering, and a preserved lazy runtime import boundary. Add a regression that proves sandbox policy removes an inline-dispatched tool such as `exec` or filesystem access.

## Reproduction Assessment

Yes. A code-level reproduction is clear: apply the patch, use a sandboxed session plus a `command-dispatch: tool` skill targeting a coding tool such as `exec`, and the new builder call lacks `options.sandbox` even though sandbox filtering is applied only from that option. I did not execute it because this review is read-only.

## Solution Assessment

No. `createOpenClawCodingTools(...)` is the right direction for shared profile/group/provider/subagent policy, but the patch is incomplete until it resolves and passes the sandbox context, or fails closed, and keeps the heavy tool builder behind the existing lazy branch.
