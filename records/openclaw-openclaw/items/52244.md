## Summary

Keep open. Current main still does not implement the PR's intended dynamic OpenRouter reasoning-default fix, and the proposed patch still has blocking correctness issues around reasoning visibility and fresh-cache ordering. Newer main has superseded the older bundled catalog-hook discovery objection, so this remains useful source work after revision rather than a cleanup close.

## What This Changes

This PR adds OpenRouter catalog augmentation and default-thinking fallback from cached model capabilities, exports an all-cached OpenRouter capability accessor through the provider-stream SDK surface, and reformats one daemon lifecycle test mock.

## Best Possible Solution

Revise or rebase the PR so dynamic OpenRouter capability metadata is hydrated before thinking and reasoning defaults are resolved, native OpenRouter reasoning remains visible without requiring an explicit /reasoning directive, and any new SDK/cache exposure is narrow, documented, and covered by fresh-cache plus cached dynamic-model tests.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction exists for a dynamic OpenRouter reasoning model absent from the static catalog: default resolution runs before prepareDynamicModel, and current tests confirm model reasoning defaults are skipped when thinking is active.

## Solution Assessment

No. The PR targets the right provider-owned area, but returning low thinking and reading a nonblocking all-cache accessor can still leave reasoning hidden or unavailable on first use; the safer fix aligns capability hydration with both defaulting paths and preserves reasoning visibility.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Preserve reasoning visibility when thinking is inferred:** `extensions/openrouter/index.ts:175-178`
  - body: Returning `"low"` here for reasoning-capable OpenRouter models makes current main treat thinking as active before reasoning defaults are resolved. `get-reply-directives.ts` then skips `resolveDefaultReasoningLevel()`, and the subscriber only extracts native thinking when `reasoningMode` is `on` or `stream`, so OpenRouter `reasoning_content` can still be dropped unless the user explicitly sets `/reasoning`.
  - confidence: 0.83
- **[P2] Await the OpenRouter catalog before using it:** `src/agents/pi-embedded-runner/openrouter-model-capabilities.ts:310-311`
  - body: `getAllCachedOpenRouterModels()` calls the nonblocking `ensureOpenRouterModelCache()` path and immediately returns `cache ?? new Map()`. On a fresh cache this starts the OpenRouter fetch but returns an empty map, while `resolveModelAsync()` prepares the dynamic model later than reply-default resolution, so first use can still miss the supplemental catalog row and keep defaults off.
  - confidence: 0.78
