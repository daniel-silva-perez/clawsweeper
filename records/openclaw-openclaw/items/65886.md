## Summary

Keep this PR open. Current main still does not implement configurable TUI status-line command rendering, and it is not cleanly closeable as ClawHub because the current plugin API exposes Control UI descriptors but not a TUI status/render insertion point. The author's linked petsonality workaround is useful evidence of real demand, but it is an external patch rather than a supported OpenClaw surface.

## Best Possible Solution

Keep this PR open for maintainer/product review. If accepted, land it as a core TUI/config feature with updated types, strict Zod schema, generated base schema/config docs, lifecycle and validation tests, and explicit trusted-local-command documentation. If maintainers prefer externalization, first add a public TUI status/render plugin seam, then move command rendering into a plugin.

## Risks / Open Questions

- Security: the PR starts a configured shell command automatically on TUI startup with `shell: true`, unlike the existing `!` local shell path that prompts once per session and warns about deleting files or leaking secrets.
- Reliability: the PR timeout and stop paths send `SIGTERM` only; a child process that ignores `SIGTERM` can continue running and leave the status runner stuck or a process alive.
- Completeness: the fetched diff updates runtime Zod validation and docs baseline hashes but does not include `src/config/schema.base.generated.ts`, focused tests, or user-facing trusted-local-command documentation.
