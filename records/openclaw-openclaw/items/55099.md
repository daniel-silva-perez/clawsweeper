## Summary

Keep open. Current main still has the reported Telegram transport behavior: a fetch-failed error graph with a qualifying timeout/socket code can promote the resolver to sticky IPv4-only fallback, and that fallback policy forces IPv4/ipv4first even when the account configured dnsResultOrder: "verbatim". I did not find a loopback IPv4 ECONNREFUSED suppression path or regression coverage for the reported /etc/hosts override case.

## Best Possible Solution

Keep this issue open and fix it in extensions/telegram/src/fetch.ts. Add a focused regression test for a Telegram fetch-failed aggregate containing loopback IPv4 ECONNREFUSED plus an IPv6 timeout or UND_ERR_SOCKET under dnsResultOrder: "verbatim", then narrow the fallback classifier/state transition so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when the configured or known-good path is IPv6/verbatim. Preserve existing sticky IPv4 and pinned-IP recovery for genuinely broken IPv6 hosts.
