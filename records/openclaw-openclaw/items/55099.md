## Summary

Keep open. Current main still lets Telegram transport fallback arm from a fetch-failed error graph that contains timeout/socket codes without checking whether the IPv4 failure was an intentional loopback refusal, and the armed fallback still forces IPv4-only transport instead of preserving the reported IPv6/verbatim path.

## What This Changes

The issue requests Telegram transport fallback to avoid latching onto dead IPv4 when a macOS `/etc/hosts` setup intentionally blocks IPv4 loopback while IPv6 is the working route.

## Best Possible Solution

Keep the Telegram-owned transport fix in the plugin: add loopback-aware fallback classification/state handling in `extensions/telegram/src/fetch.ts`, cover the reported aggregate error shape under `dnsResultOrder: "verbatim"`, and preserve the existing IPv4 and pinned-IP fallback path for real IPv6/TLS reachability failures.
