## Summary

Keep this issue open. Current main still has the reported Telegram transport behavior: fallback classification does not treat loopback IPv4 ECONNREFUSED as a blocker when the same fetch-failed error graph also contains a qualifying timeout/socket code, and the armed fallback still switches the resolver to sticky IPv4-only transport even when the account configured dnsResultOrder: "verbatim".

## Best Possible Solution

Keep this issue open and fix it in extensions/telegram/src/fetch.ts. Add a focused regression test for an aggregate Telegram fetch-failed error containing loopback IPv4 ECONNREFUSED plus an IPv6 timeout or UND_ERR_SOCKET under dnsResultOrder: "verbatim", then narrow the fallback classifier/state transition so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when the configured or known-good path is IPv6/verbatim. Preserve existing sticky IPv4 and pinned-IP recovery behavior for genuinely broken IPv6 hosts.
