## Summary

Keep open. Current main still accepts the reported Tailscale Serve plus explicit no-auth configuration, and the same-author security bug #50630 remains open. This PR is still a relevant fix candidate, but the runtime-only patch should address daemon install/doctor validation before merge.

## What This Changes

The PR adds a startup-time `resolveGatewayRuntimeConfig` guard and regression tests to reject Tailscale Serve when the resolved gateway auth mode is explicit `none`, while leaving Funnel password-only and Serve with token auth accepted.

## Best Possible Solution

Keep the focused runtime rejection, but add pre-start validation or repair in daemon install/doctor/service paths so the invalid config is blocked or normalized before a managed service is written. Preserve local loopback `auth.mode="none"`, Serve with real shared-secret auth or verified Tailscale identity where intended, and Funnel's password-only rule.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main accepts `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"` in runtime config resolution, and the auth path then accepts `mode="none"` callers without credentials.

## Solution Assessment

No, not yet. The runtime guard is the right core boundary, but the PR should also validate or normalize the same invalid configuration before daemon install/doctor/service repair writes a service that will fail at startup.
