## Summary

Keep this PR open. Current main still permits the reported `gateway.tailscale.mode="serve"` plus `gateway.auth.mode="none"` configuration, and the current tests explicitly preserve that success path. The PR remains a focused gateway-auth hardening candidate paired with the same-author open bug report, but it should be updated to cover the install/repair validation gap before landing.

## Best Possible Solution

Keep this PR open as the implementation candidate for the paired bug report. Rebase it onto current main, change the current Serve plus `auth.mode="none"` success test into a rejection test, add install/doctor/service-repair validation for the same configuration before a service is written, and preserve allowed-path coverage for local loopback `none`, Serve with token/password/trusted-proxy as intended, and Funnel's existing password-only rule.
