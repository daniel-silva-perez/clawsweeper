## Summary

Current main implements the missing manual-run handoff: main-session cron system events now treat the cron-active heartbeat skip as retryable, queue a targeted heartbeat after the cron active marker clears, and the heartbeat runner consumes the queued cron event as a cron-event prompt. The reported `deliveryStatus: "not-requested"` remains expected fallback-delivery status for main-session system events, not proof that the wake was skipped.

## What This Changes

Manual `cron run` for `sessionTarget="main"` and `payload.kind="systemEvent"` should wake the main session and execute the queued system event instead of finishing without a visible agent turn.

## Best Possible Solution

Keep the current-main active-marker fallback and cron-event prompt handoff, then ship it in the next release; if operators still confuse fallback delivery with heartbeat execution, add a distinct wake/execution diagnostic instead of changing the established fallback-delivery meaning.

## Reproduction Assessment

Yes. The issue provides concrete v2026.4.26 CLI/UI steps, and current-main regression coverage now models the relevant failure mode by making the synchronous heartbeat return `cron-in-progress` and verifying the cron service queues the targeted heartbeat instead of recording a skipped run.

## Solution Assessment

Yes. Treating the cron-active marker as a retryable busy condition and deferring the targeted heartbeat is the narrow fix because it preserves main-session systemEvent semantics, keeps fallback delivery status unchanged, and avoids a broader cron or heartbeat API redesign.
