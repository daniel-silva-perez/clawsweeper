## Summary

Keep open. Current main has the intended main-session cron handoff pieces, but the reporter-visible `deliveryStatus: "not-requested"` still appears by design and there is still no end-to-end proof that a manual Gateway or Control UI `cron.run` for `sessionTarget="main"` and `payload.kind="systemEvent"` produces a visible main-session turn rather than only queueing heartbeat bookkeeping.

## What This Changes

The issue reports that manually running a main-session cron `systemEvent` job on v2026.4.26 records `deliveryStatus: "not-requested"` and does not wake the main agent session.

## Best Possible Solution

Add a focused regression that drives manual `cron.run` through the Gateway path for a `sessionTarget="main"`, `payload.kind="systemEvent"`, `wakeMode="now"` job, proves the cron event is consumed by the intended main session, and proves a visible output path or explicit no-target execution result is recorded. If `deliveryStatus: "not-requested"` remains correct, add a distinct execution/wake status or clearer UI/docs wording so operators can distinguish successful main-session execution from missing fallback delivery.

## Reproduction Assessment

Unclear for current main. The issue provides concrete v2026.4.26 steps and current tests reproduce the `deliveryStatus: "not-requested"` diagnostic, but the read-only review did not produce a high-confidence current-main reproduction of the missing agent turn itself.

## Solution Assessment

Unclear. The existing implementation is directionally right because it enqueues the event and wakes heartbeat with `target: "last"`, but the best fix is to close the proof gap with an end-to-end regression and then clarify or extend run-history status if the diagnostic remains misleading.
