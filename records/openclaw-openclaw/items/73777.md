## Summary

Keep open. Current main has the intended cron-to-heartbeat plumbing, but it still records the reporter-visible `deliveryStatus: "not-requested"` for main-session system-event runs and the existing Gateway regression only locks that diagnostic, not a visible main-session turn. The report remains a plausible narrow regression around manual `cron.run`/Control UI execution proof rather than something safe to close.

## What This Changes

Manual `cron run` and Control UI Run for a `sessionTarget="main"`, `payload.kind="systemEvent"` cron job should wake the main session and create an agent turn instead of appearing to finish with `deliveryStatus: "not-requested"` and no work.

## Best Possible Solution

Add an end-to-end regression that drives manual `cron.run` through the Gateway for a `sessionTarget="main"`, `payload.kind="systemEvent"`, `wakeMode="now"` job and proves the queued event is consumed by the intended main-session heartbeat turn. If execution already succeeds, add a distinct wake/execution status or clearer run-history/UI/docs wording so operators can tell successful main-session execution apart from missing fallback delivery.

## Reproduction Assessment

Unclear. The issue has concrete v2026.4.26 steps and current tests reproduce the `deliveryStatus: "not-requested"` diagnostic, but this read-only review found no current-main end-to-end proof for the missing agent turn itself.

## Solution Assessment

Unclear. The current implementation is directionally narrow because it enqueues the event and queues or runs heartbeat with `target: "last"`, but the best repair is to close the Gateway-level proof gap first and then fix the minimal cron/heartbeat/status behavior that fails that regression.
