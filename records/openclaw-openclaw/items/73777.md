## Summary

Keep open. Current main still has the intended main-session cron handoff pieces, but it also still preserves the reporter-visible `deliveryStatus: "not-requested"` for this flow, and the existing Gateway coverage proves the diagnostic/run-history path rather than proving a visible main-session heartbeat turn after manual run.

## What This Changes

Manual `cron run` and the Control UI Run button should wake and execute a `sessionTarget="main"`, `payload.kind="systemEvent"` cron job instead of appearing to finish with `deliveryStatus: "not-requested"` and no agent work.

## Best Possible Solution

Add a Gateway-level regression for a manual main-session `systemEvent` cron run that proves the queued event is consumed by the intended heartbeat turn and produces visible work or a clear execution/wake result. If execution already works, separate fallback delivery status from wake/execution status in run history and UI/docs so operators can distinguish successful main-session execution from no requested fallback delivery.
