## Summary

Keep open. Current main still has the same relevant main-session cron/manual-run semantics as v2026.4.26: manual `cron.run` queues execution, main `systemEvent` jobs enqueue and wake heartbeat, but run history still records `deliveryStatus: "not-requested"` because that field tracks fallback delivery, not whether a main-session heartbeat turn actually ran. Existing tests cover the seams separately but do not prove the CLI/Control UI manual-run path creates a visible main-session agent turn, so this regression report is not closeable as implemented.

## Best Possible Solution

Keep this issue open. Add a focused regression test for manual `cron.run` through the Gateway/Control UI path using `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and `wakeMode: "now"`; assert the event reaches the intended session, heartbeat uses the cron-event prompt, the event is consumed, and a visible main-session agent turn/output occurs. If `deliveryStatus: "not-requested"` is the intended fallback-delivery status, add a separate wake/heartbeat execution status or clearer UI/docs so operators can distinguish a successful main-session wake from a lost cron event.
