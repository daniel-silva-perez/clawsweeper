## Summary

Keep open. Current main has code intended to enqueue main-session cron system events and wake the heartbeat, but the specific reported symptom is not closeable: manual `cron.run` history still records `deliveryStatus: "not-requested"` for main/systemEvent jobs, and the current tests do not prove the Gateway/Control UI manual-run path actually creates a visible agent turn after that run-history entry. The relevant main-session cron execution path in `v2026.4.26` also matches current main, so there is no post-release implemented-on-main fix to point to.

## Best Possible Solution

Keep the issue open and add a targeted regression test for manual `cron.run` through the Gateway/Control UI path using a main-session `systemEvent` job with default/`wakeMode: "now"`. The test should assert the event reaches the intended session, the heartbeat runner uses the cron-event prompt, the event is consumed, and the user-visible agent turn occurs. If the current run-history semantics are intended, also document or surface a separate wake/heartbeat execution status so `deliveryStatus: "not-requested"` is not mistaken for a lost cron event.

## Risks / Open Questions

- Existing coverage proves service-level wake behavior and Gateway run-history persistence, but not a full manual CLI/Control UI path that asserts a user-visible agent turn is created for `sessionTarget: "main"` plus `payload.kind: "systemEvent"`.
