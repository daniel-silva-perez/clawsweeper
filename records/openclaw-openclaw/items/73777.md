## Summary

Keep open. Current main and the v2026.4.26 source both enqueue main-session cron system events and request or run heartbeat, while `deliveryStatus: "not-requested"` is expected for main-session jobs without runner fallback delivery. The remaining report is the user-visible manual-run path: current Gateway/UI coverage still preserves the same `not-requested` run-history symptom and does not prove that a manual `cron.run` or Control UI Run creates and consumes a visible main-session agent turn.

## Best Possible Solution

Keep this issue open. Add a targeted regression test for manual `cron.run` through the Gateway/Control UI path with `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and both `wakeMode: "now"` and `wakeMode: "next-heartbeat"` if both are supported. The fix should either prove and surface the consumed heartbeat/agent turn, or add a separate wake/heartbeat execution status so `deliveryStatus: "not-requested"` is no longer treated as evidence that the cron event was lost.

## Risks / Open Questions

- `deliveryStatus` remains easy to misread because it describes cron runner fallback delivery, not whether a main-session heartbeat turn ran and consumed the system event.
- Current tests cover service-level wake calls and heartbeat cron-event prompting, but not a full manual CLI/Control UI flow that asserts the queued system event becomes a visible agent turn.
- Closing now could hide a real observability or manual-run timing bug for `wakeMode: "next-heartbeat"`, where run history can finish quickly while heartbeat execution is still queued separately.
