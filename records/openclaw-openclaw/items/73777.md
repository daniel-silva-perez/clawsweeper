## Summary

Keep open. Current main shows the main-session cron systemEvent path does enqueue and wake heartbeat, and `deliveryStatus: "not-requested"` is still expected for jobs without fallback delivery, but the remaining user-visible claim is that manual CLI/Control UI runs do not create an observable main-session turn. The existing coverage still proves the Gateway run-history/status seams and heartbeat cron-event prompt seams separately, not the full manual-run-to-visible-reply path.

## What This Changes

The issue reports that manually running a main-session cron job with a systemEvent payload on v2026.4.26 records `deliveryStatus: "not-requested"` and does not create an observable agent turn.

## Best Possible Solution

Add a focused regression that creates a main-session `systemEvent` cron job, manually runs it through the Gateway RPC path used by CLI and Control UI, and asserts the cron event reaches the intended session, uses the cron-event heartbeat prompt, is consumed, and produces a visible main-session reply. If fallback `deliveryStatus: "not-requested"` is intended, add a distinct wake/execution signal or UI/docs copy so operators can distinguish successful heartbeat execution from a lost cron event.

## Reproduction Assessment

Unclear. The report has concrete steps and current tests can reproduce the `deliveryStatus: "not-requested"` run-history value, but I did not find a high-confidence reproduction or regression test for the claimed missing visible agent turn on current main.

## Solution Assessment

Unclear. The current implementation path is plausible and should not blindly repurpose fallback delivery status, but the best maintainable fix is to add end-to-end regression proof first, then clarify or separate the operator-facing status if the manual run is already executing correctly.
