## Summary

Keep open. Current main still does not provide close-level evidence that manual `cron.run` or the Control UI Run button creates a visible main-session agent turn for `sessionTarget: "main"` plus `payload.kind: "systemEvent"`; run history is still expected to show `deliveryStatus: "not-requested"` for this path, so the reporter’s diagnostic remains ambiguous rather than disproven.

## Best Possible Solution

Keep the issue open. Add a targeted regression test for manual `cron.run` through the Gateway/Control UI path using a main-session `systemEvent` job, covering both default wake behavior and `wakeMode: "now"` if they are intended to differ. The fix should either make the manual path reliably create the visible heartbeat/agent turn or expose a separate wake/heartbeat execution status so `deliveryStatus: "not-requested"` is no longer interpreted as a lost cron event.

## Risks / Open Questions

- `deliveryStatus` is overloaded: it describes cron fallback delivery, not whether the main-session heartbeat turn actually executed.
- The current tests cover service-level wake calls and Gateway run-history persistence, but not a full CLI/Control UI manual-run flow that asserts the queued cron event is consumed and a user-visible agent turn occurs.
