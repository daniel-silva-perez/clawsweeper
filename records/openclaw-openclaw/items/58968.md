## Summary

Keep open. Current main still calls `listPagesViaPlaywright` for persistent remote-CDP tab listing without an internal timeout around the Playwright page enumeration loop, and the related fix PR was closed unmerged with the author saying they would resubmit if still relevant. Higher-level callers have some request timeouts, but that does not solve the stale Playwright connection remaining hung for future tab reads.

## What This Changes

The issue requests timeout protection and graceful handling for remote-CDP Playwright page enumeration when the CDP connection stalls.

## Best Possible Solution

Add a bounded, plugin-local timeout to the persistent Playwright page-listing path and make timeout cleanup recoverable, ideally evicting or disconnecting the stale cached Playwright connection so later browser requests can reconnect instead of repeatedly hitting the same stuck CDP pipe.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `BrowserContext.newCDPSession` or `page.title()` to never resolve and then call `listPagesViaPlaywright`; current main awaits those operations without an internal deadline.

## Solution Assessment

Yes. The narrowest maintainable fix is inside the browser plugin's persistent Playwright tab-listing primitive, with regression coverage; relying only on optional caller-level request timeouts leaves the cached Playwright/CDP session stuck.

## Repair Work Prompt

Fix openclaw/openclaw issue #58968. Current main still lets `extensions/browser/src/browser/pw-session.ts:listPagesViaPlaywright` await Playwright CDP page enumeration indefinitely when `newCDPSession`, `Target.getTargetInfo`, or `page.title()` stalls on a remote CDP profile. Implement a bounded timeout in the browser plugin, not in core. Prefer a small helper or `Promise.race` around the list-pages operation with a reasonable default around 15s, clear/unref the timer, preserve SSRF policy behavior, and evict or disconnect the cached Playwright connection on timeout so later requests can reconnect.
