## Summary

Current main still does not implement `chrome-extension://*` or any scheme-level wildcard support for `gateway.controlUi.allowedOrigins`. The origin checker only accepts the global `*` wildcard or exact normalized `URL.origin` values, the gateway WebSocket handshake still routes Control UI/WebChat browser origins through that checker, and the docs/tests still describe exact full origins plus global allow-all only. Related PR #39867 directly attempted this but is closed unmerged; open PR #38290 is intentionally narrower and does not add wildcard/prefix expansion.

## Best Possible Solution

Keep this issue open for explicit gateway product/security judgment. If accepted, implement a narrowly scoped `chrome-extension://*` matcher in `src/gateway/origin-check.ts`, preserving literal `Origin: null` rejection, existing HTTP(S) exact matches, host-header fallback, and loopback behavior. Add focused unit and WebSocket handshake tests for exact extension origins, `chrome-extension://*`, non-extension negatives, malformed origins, plus docs/config/security guidance updates. Coordinate with PR #38290 for exact non-standard origin normalization, but keep this as the wildcard tracking item unless a maintainer-approved follow-up replaces it.
