## Summary

Keep open: current main still has the Feishu token-request failure mode described here. The bundled SDK is now 1.62.0, but that version still catches token POST failures without rethrowing before destructuring `tenant_access_token`, and OpenClaw’s Feishu HTTP wrapper only adds timeout/User-Agent behavior rather than a token-error, proxy, or SDK workaround. Related #48949 and open PR #41943 cover a concrete proxy-triggered version of the same symptom, but they do not fully supersede this broader SDK-token error report.

## What This Changes

The issue asks OpenClaw to avoid the confusing Feishu `tenant_access_token` destructuring crash when the bundled Lark SDK token request fails.

## Best Possible Solution

Fix the Feishu plugin’s SDK HTTP/token boundary so token request failures surface the original cause and known proxy-triggered failures are handled deliberately. The fix should live in `extensions/feishu/src/client.ts` or a Feishu-owned helper, preserve timeout/User-Agent behavior, define NO_PROXY/proxy semantics for Feishu/Lark/custom domains, add focused regression tests, and update Feishu troubleshooting docs.

## Reproduction Assessment

Yes for the reported symptom: in the locked SDK 1.62.0 source, a rejected tenant token POST still flows through a catch that logs without rethrowing and then destructures undefined. The exact reporter-side Windows failure is not independently reproduced, but #48949 gives a concrete proxy configuration that can trigger the same symptom.

## Solution Assessment

No current solution is in place on main. Patching installed node_modules is not maintainable; the safer direction is an approved Feishu-owned adapter/SDK update or dependency patch that preserves existing timeout/User-Agent semantics and handles proxy policy explicitly.
