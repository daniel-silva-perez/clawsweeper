## Summary

Keep open. Current main has moved the Feishu plugin to @larksuiteoapi/node-sdk 1.62.0, but the upstream token manager still has the reported failure pattern: a rejected tenant-token POST is caught/logged without rethrowing, then `tenant_access_token` is destructured from the undefined result. OpenClaw's Feishu client wrapper still delegates that path to the SDK HTTP instance and only injects timeout/user-agent behavior. The older duplicate #44677 was already closed in favor of this issue, so this should remain the canonical tracker.

## Best Possible Solution

Keep this issue open and fix it in the Feishu-owned runtime surface, most likely `extensions/feishu/src/client.ts` or a narrow adapter around the Lark SDK token path. The fix should preserve the original HTTP/proxy/API error when tenant-token acquisition fails, avoid the undefined destructuring crash, and add a focused regression test that forces the tenant-token POST to reject. If upstream releases a real SDK fix, update the pinned dependency and keep a regression test proving the original failure no longer appears.
