## Summary

Keep open: current main still carries the reported Feishu tenant token failure mode. The bundled SDK is now 1.62.0, but that package still catches a failed tenant-access-token POST without rethrowing before destructuring the result; OpenClaw's Feishu client wrapper only injects timeout/User-Agent behavior and does not add a targeted guard.

## What This Changes

The issue asks OpenClaw to stop Feishu plugin startup/tool calls from failing with a confusing `tenant_access_token` destructuring error when the SDK token request fails.

## Best Possible Solution

Fix this in the Feishu plugin or through an approved upstream/dependency patch so tenant-token HTTP failures surface the original cause instead of the destructuring error, while preserving SDK response unwrapping, timeout injection, User-Agent behavior, and Feishu/Lark/custom-domain proxy semantics. Add a focused regression test for a rejected tenant token POST and document the user-facing troubleshooting path if proxy routing is part of the failure.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the SDK `httpInstance.post()` for `/tenant_access_token/internal` reject; the 1.62.0 SDK catch logs the error and returns `undefined`, then destructures `tenant_access_token` from it.

## Solution Assessment

No, current main is not the best solution because the SDK update to 1.62.0 does not remove the catch-and-destructure failure mode. The closed PR #44751 also was not the right fix because broad `data` unwrapping could strip valid Feishu API response bodies.
