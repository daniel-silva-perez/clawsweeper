## Summary

Keep open. Current main has an account-aware gateway health path, but the Feishu setup/status resolver still probes only top-level Feishu credentials, and open PR #74060 is an active, unmerged implementation candidate with closing syntax for this issue.

## What This Changes

The issue requests Feishu health/status detection to treat `channels.feishu.defaultAccount` plus `channels.feishu.accounts.*.appId/appSecret` as configured instead of requiring top-level credentials.

## Best Possible Solution

Review PR #74060 and either land it as the Feishu-owned fix or close it if maintainers decide the remaining setup/status probe gap is not part of the user-facing health surface. The desired end state is that Feishu status/probe paths consistently use the selected account resolver with regression coverage for multi-account credentials.

## Reproduction Assessment

Unclear for the exact live OFF table on current main because gateway health now resolves the selected account before probing. The narrower setup/status gap is reproducible by inspection: account-only credentials can set `configured` true while `resolveStatusLines` still inspects only top-level `channels.feishu` credentials before probing.

## Solution Assessment

Yes. Routing the Feishu setup/status probe through `resolveFeishuAccount` is the narrow maintainable fix, and the issue should stay tied to open PR #74060 rather than starting a duplicate repair branch.
