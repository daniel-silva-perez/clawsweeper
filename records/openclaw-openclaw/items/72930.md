## Summary

Keep open. The report is actionable and already paired with open PR #74060, which specifically targets the Feishu setup/status credential path; current main has account-aware Feishu resolution and gateway health probing, but the setup/status surface still probes from top-level credentials only, so this should wait for maintainer review of the linked PR or an equivalent fix.

## What This Changes

The issue requests that Feishu health/status treat `channels.feishu.defaultAccount` plus `channels.feishu.accounts.*.appId/appSecret` as configured instead of requiring top-level credentials.

## Best Possible Solution

Review and land PR #74060, or make an equivalent Feishu-plugin-owned change that resolves setup/status probes through the selected Feishu account and keeps regression coverage for account-scoped credentials.

## Reproduction Assessment

Unclear for the exact live `openclaw status --deep` table without Feishu credentials. The narrower setup/status defect has a clear source-level reproduction: account-only credentials can make Feishu configured while the status probe still inspects only top-level credentials.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix: reuse Feishu's existing account resolver in the Feishu setup/status surface and add focused regression coverage, without adding core-specific Feishu behavior.
