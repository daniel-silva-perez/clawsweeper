## Summary

Keep open. The report is actionable and still has an open linked implementation candidate, PR #74060, with closing syntax for this issue. Current main’s gateway health path is account-aware, but the Feishu setup/status status-lines path still probes only top-level credentials, so the maintainer path is to review the linked PR or decide that only the shipped gateway health behavior matters.

## What This Changes

The issue asks for Feishu health/status output to treat multi-account credentials under `channels.feishu.accounts.*.appId/appSecret` as configured and probeable.

## Best Possible Solution

Review PR #74060 and either land it as the Feishu-owned setup/status cleanup, or close it if maintainers decide the shipped gateway health path fully resolves the user-facing report. The desired end state is that every Feishu status/probe surface uses the same selected-account credential resolver and has account-scoped regression coverage.

## Reproduction Assessment

Unclear for the exact live health-table symptom without Feishu credentials. The narrower source-level gap is reproducible by using account-only Feishu credentials: the account resolver marks the account configured, while `resolveStatusLines` still probes only top-level credentials.

## Solution Assessment

Yes for the remaining setup/status gap: routing the probe through `resolveFeishuAccount` is the narrow Feishu-owned fix and avoids adding Feishu-specific logic to core. It is unclear whether that PR is still required for the exact gateway Health table because current main already resolves health accounts through the plugin adapter.
