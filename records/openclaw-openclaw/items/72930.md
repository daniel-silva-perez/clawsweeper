## Summary

Keep open. Current main is account-aware in the gateway health path, but this issue is paired with open PR #74060 using closing syntax, and the Feishu setup/status surface still verifies probes from top-level credentials instead of the selected account.

## What This Changes

The report asks Feishu health/status to treat multi-account credentials under `channels.feishu.accounts.<id>.appId/appSecret` as configured and probeable.

## Best Possible Solution

Land or close PR #74060 after maintainer review. The desired end state is for Feishu setup/status verification to resolve the selected Feishu account first and keep regression coverage for account-scoped credentials.

## Reproduction Assessment

Unclear for the exact live `openclaw status --deep` OFF output without real Feishu credentials. The remaining source-level mismatch is deterministic: account-scoped credentials make Feishu configured, while setup/status probing still reads top-level credentials before verifying.

## Solution Assessment

Yes. Routing Feishu setup/status probing through the existing resolved-account path is the narrow plugin-owned fix and avoids adding core special cases.
