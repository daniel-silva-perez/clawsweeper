## Summary

Keep open. The report is still paired with open PR #74060, which uses closing syntax and targets the same Feishu account-credential status/probe gap. Current main’s gateway health path is account-aware, but the Feishu setup/status resolver still probes only top-level `channels.feishu` credentials, so this should remain open until the linked PR is reviewed, landed, or explicitly closed.

## What This Changes

The issue requests that Feishu health/status treat `channels.feishu.defaultAccount` plus `channels.feishu.accounts.*.appId/appSecret` credentials as configured and probeable instead of requiring top-level credentials.

## Best Possible Solution

Land or close PR #74060 after maintainer review. The desired end state is that every Feishu status/probe surface resolves the selected Feishu account first and uses focused regression coverage for account-scoped credentials.

## Reproduction Assessment

Unclear for the exact live `openclaw status --deep` OFF output without real Feishu credentials. The narrower source-level gap is reproducible by inspection: account-scoped credentials can make Feishu configured, while `resolveStatusLines` still probes only top-level `channels.feishu` credentials.

## Solution Assessment

Yes. Routing Feishu setup/status probing through the existing selected-account resolver is the narrow plugin-owned fix, and the active linked PR is the right place to review that change.
