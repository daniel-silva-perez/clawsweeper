## Summary

Keep open. The report is still paired with open PR #74060, which uses closing syntax and targets the Feishu account-credential status/probe gap. Current main’s gateway health path is account-aware, but the Feishu setup/status resolver still probes only top-level credentials, so this should stay open until the linked PR is reviewed, landed, or explicitly closed by maintainers.

## What This Changes

The issue asks for Feishu health/status output to treat multi-account configs with credentials under `channels.feishu.accounts.*.appId/appSecret` as configured instead of showing Feishu as OFF/not configured.

## Best Possible Solution

Review PR #74060 and either land it as the Feishu-owned cleanup or close it if maintainers decide current gateway health already covers the user-facing symptom. The desired end state is that every Feishu status/probe surface uses the selected-account credential resolver with focused account-scoped regression coverage.

## Reproduction Assessment

Unclear for the exact live `openclaw status --deep` OFF output on current main because gateway health now resolves Feishu accounts before probing. The narrower source-level gap is reproducible by inspection: account-only credentials can make setup status configured while its probe still inspects only top-level `channels.feishu` credentials.

## Solution Assessment

Yes for the remaining setup/status gap: routing the probe through the existing Feishu account resolver is the narrow plugin-owned fix. The issue handling should stay tied to PR #74060 rather than starting a separate repair branch.
