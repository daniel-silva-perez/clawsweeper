## Summary

Current main still contains the ambiguous MCP transport selection behavior reported here, and open PR #72515 is already the paired implementation candidate with closing syntax, so this issue should remain open until that PR or a maintainer-chosen alternative lands.

## What This Changes

The issue requests a fix for MCP server configs that combine a spawned `command` with `transport: "streamable-http"` and `url`, where OpenClaw spawns the process but times out instead of connecting to the HTTP endpoint.

## Best Possible Solution

The desired end state is an explicit MCP transport contract: reject mixed stdio/HTTP server definitions with a clear diagnostic and regression tests, matching current docs, or deliberately design managed streamable-http child-process support with lifecycle coverage.

## Reproduction Assessment

Yes. Current main can be inspected to show a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves to stdio first; if the spawned process only serves HTTP, the existing 30-second MCP connection timeout is the expected failure path.

## Solution Assessment

Unclear until maintainers choose the intended boundary. PR #72515 is the narrowest docs-aligned repair if mixed configs should be invalid, while managed HTTP child-process support would be a larger feature and should not be folded into silent fallback behavior.
