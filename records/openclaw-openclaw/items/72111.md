## Summary

Keep open. Current main still selects the stdio transport before the streamable-http branch when a server config includes both `command` and `url`, so the reported timeout remains plausible, and open PR #72515 already carries closing syntax for this issue.

## What This Changes

The issue asks OpenClaw to handle an MCP `streamable-http` server config that includes both a spawned command and an HTTP URL without timing out after spawning the server process.

## Best Possible Solution

The desired end state is an explicit MCP transport contract: reject ambiguous mixed stdio/HTTP server definitions with a clear diagnostic and tests, or deliberately add managed streamable-http child-process support as a separate feature with lifecycle coverage.

## Reproduction Assessment

Yes. A config object with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` still resolves to stdio on current main; if the spawned process only serves HTTP and never speaks JSON-RPC over stdout, the runtime can hit the existing 30-second connection timeout.

## Solution Assessment

Unclear until maintainers choose the intended boundary. PR #72515 is the narrowest maintainable fix if OpenClaw should enforce the current docs and reject mixed configs; managed HTTP child-process support would be a larger feature.
