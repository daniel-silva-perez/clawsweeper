## Summary

Keep open. Current main still resolves a mixed MCP server config with `command`, non-empty `url`, and `transport: "streamable-http"` as stdio before the HTTP transport branch can run, which matches the reported 30-second timeout path. The linked #72515 PR remains the active closing implementation candidate, so this issue should stay open until that PR or a maintainer-selected alternative lands.

## What This Changes

The issue asks OpenClaw to handle a spawned MCP server configured for `streamable-http` without timing out after the child process starts and listens on its HTTP port.

## Best Possible Solution

Make MCP config semantics explicit: reject non-empty mixed `command`+`url` definitions with a clear diagnostic and regression tests, or deliberately add managed streamable-http child-process lifecycle support as a larger feature. Silent stdio fallback should not remain the user-visible behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a server config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio, then reaches the existing 30-second MCP connection timeout when the child process does not speak JSON-RPC over stdio.

## Solution Assessment

Unclear until maintainers choose the product boundary. If the docs remain authoritative, #72515 is the narrowest maintainable fix; if spawned streamable-http servers are intended to be supported, that should be a separate lifecycle design rather than preserving silent fallback.
