## Summary

Keep open. Current main still resolves a mixed MCP server config with both `command` and `url` as stdio before it ever reaches the `streamable-http` branch, so the reported timeout remains a plausible current-main bug; open PR #72515 is already the linked implementation candidate and should be reviewed instead of closing this issue early.

## What This Changes

The issue reports that an MCP server configured with `command`/`args`, `url`, and `transport: "streamable-http"` starts its child HTTP server but OpenClaw times out while connecting.

## Best Possible Solution

Align the implementation with the documented MCP transport contract: either land or revise #72515 to reject mixed `command` plus `url` configs with a clear diagnostic and focused resolver tests, or intentionally design managed HTTP child-process support as a separate documented feature.

## Reproduction Assessment

Yes. A config object with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` still resolves to stdio on current main; an HTTP-only child process would not complete JSON-RPC over stdout and can hit the existing 30-second connection timeout.

## Solution Assessment

Unclear until maintainers choose the intended boundary. If the docs are the source of truth, rejecting mixed `command` plus `url` configs is the narrowest maintainable fix; if OpenClaw should manage spawned HTTP MCP servers, that is a larger feature than the current linked PR.
