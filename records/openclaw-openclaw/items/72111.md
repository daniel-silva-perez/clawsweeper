## Summary

Keep open. Current main still resolves a mixed MCP server config with both `command` and `url` as stdio before it considers `transport: "streamable-http"`, so the reported timeout path remains plausible; an open linked fix PR, #72515, already targets this issue and should be reviewed before closing.

## What This Changes

The issue reports that MCP server entries combining `command`/`args` with `url` and `transport: "streamable-http"` spawn a process but time out instead of connecting to the HTTP endpoint or rejecting the ambiguous config.

## Best Possible Solution

The desired end state is an explicit MCP transport contract: either reject ambiguous mixed stdio/HTTP server definitions with a clear diagnostic, matching the current docs, or add a deliberate managed streamable-http child-process mode with readiness, connection, timeout, and cleanup semantics covered by tests.

## Reproduction Assessment

Yes. A config object with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` follows the stdio branch on current main; if the spawned process only serves HTTP and never speaks JSON-RPC over stdout, the runtime can hit the existing 30-second connection timeout.

## Solution Assessment

Unclear until maintainers decide the intended product boundary. Early rejection, as proposed in #72515, is the narrowest maintainable fix if managed HTTP child processes are unsupported; explicit child-process HTTP management would be a larger feature, not a cleanup patch.
