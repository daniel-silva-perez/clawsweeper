## Summary

Keep open: current main still silently resolves a mixed `command` + `url` MCP server as stdio before it can reach the streamable-http branch, which matches the reported timeout path. Open PR #72515 is the active linked implementation candidate with closing syntax, so this issue should remain open until that PR or a maintainer-selected alternative lands.

## What This Changes

The issue asks OpenClaw to handle an MCP `streamable-http` server configured with both a spawned command and HTTP URL without ending in the 30-second connection timeout.

## Best Possible Solution

The target state is explicit MCP config semantics: reject ambiguous mixed stdio/HTTP definitions with a clear diagnostic and regression tests, or deliberately add managed HTTP child-process support with lifecycle coverage. Silent stdio fallback should not remain the user-visible behavior.

## Reproduction Assessment

Yes. Source inspection on current main is enough: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio, and an HTTP-only child process then reaches the existing 30-second MCP connection timeout.

## Solution Assessment

Unclear until maintainers choose the boundary. If the docs are authoritative, PR #72515 is the narrowest maintainable fix; if managed spawned streamable-http servers are desired, that should be designed as a larger lifecycle feature instead of silent fallback.
