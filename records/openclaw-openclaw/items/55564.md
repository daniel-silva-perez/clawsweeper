## Summary

Keep this issue open. Current main still starts outbound delivery recovery on gateway startup and replays pending delivery-queue entries with retry/backoff gates only; there is still no stale-age, TTL, or restart-generation filter for old first-attempt entries. The related cleanup context points to this issue as the canonical tracker, and PR #54900 is an open but unmerged implementation candidate rather than a fix on main.

## What This Changes

The issue requests dropping, quarantining, or TTL-filtering queued outbound channel messages from before a gateway restart so stale replies are not batch-delivered later.

## Best Possible Solution

Keep this canonical tracker open until a core outbound recovery freshness policy lands. The best path is to review, rebase, and fix PR #54900 or replace it with a narrow PR that applies a stale-age or restart-generation guard in `src/infra/outbound`, preserves fresh crash recovery, covers reconnect/startup recovery tests, and updates config/docs/changelog if an operator knob is exposed.
