## Summary

Keep this issue open. Current main still stores and merges arbitrary model params, but the central OpenAI/OpenAI-compatible paths do not first-class forward `frequency_penalty`, `presence_penalty`, `repetition_penalty`, or config-level `top_p`/`top_k` into Responses, Chat Completions, or WebSocket payloads. Related PRs for this exact behavior were closed unmerged, and the broader open per-agent generation issues are adjacent rather than canonical replacements.

## Best Possible Solution

Keep this issue open as the canonical OpenAI/OpenAI-compatible sampling-penalty passthrough request. The best fix is a focused transport change that normalizes numeric snake_case and camelCase sampling params from the existing merged `params` sources, forwards only supported fields for compatible Responses, Chat Completions, Azure, and WebSocket payloads, preserves existing payload hooks and `extra_body` behavior, and documents the supported scope.

## Risks / Open Questions

- A blanket payload passthrough could break providers or models that reject unsupported sampling fields; the implementation should be scoped to compatible OpenAI/OpenAI-compatible API families or explicit provider capability.
- The `extra_body` workaround only covers `api: "openai-completions"` proxy payloads and can override generated keys, so first-class params should preserve existing `extra_body`, `chat_template_kwargs`, `onPayload`, and provider wrapper ordering.
- This overlaps with broader #47852/#71350 generation-parameter configuration discussions; a narrow fix should avoid inventing a competing public config shape beyond the existing flat `params` contract unless maintainers choose one.
