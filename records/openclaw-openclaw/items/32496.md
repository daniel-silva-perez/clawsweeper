## Summary

Keep this issue open. Current main still accepts and merges arbitrary model `params`, but the shared OpenAI/OpenAI-compatible transport path does not first-class forward `frequency_penalty` or `presence_penalty`; only native Ollama has provider-owned forwarding for related runtime options. The prior implementation PRs discussed in the thread are closed unmerged, and the newer related issues are broader rather than superseding this narrow request.

## Best Possible Solution

Keep this issue open as the canonical narrow request for OpenAI/OpenAI-compatible sampling-penalty passthrough. The best implementation is a focused transport change that normalizes finite numeric `frequency_penalty`/`frequencyPenalty` and `presence_penalty`/`presencePenalty` from the already-merged `params` sources, forwards them only on supported OpenAI-family payloads, adds targeted regression tests, and documents the supported scope without expanding into the broader per-agent generation-parameter programs in #47852 or #71350.

Expected boundary: add scoped support for finite numeric `frequency_penalty`/`frequencyPenalty` and `presence_penalty`/`presencePenalty` from the existing merged `params` sources. Treat `top_p`/`topP` as in-scope only where the same OpenAI-family contract supports it; leave `top_k` and `repetition_penalty` to provider-owned paths like Ollama unless dependency/provider contracts prove support.

Preserve existing `onPayload` chaining, `extra_body` precedence for OpenAI-compatible completions, transport selection, cache retention, OpenAI Responses payload policy, and non-native store stripping. Filter non-number, `NaN`, and infinite values.
