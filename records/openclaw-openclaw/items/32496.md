## Summary

Keep this issue open. Current main still accepts and merges arbitrary model params, but the central OpenAI/OpenAI-compatible transport paths do not first-class forward `frequency_penalty`, `presence_penalty`, `repetition_penalty`, or config-level `top_p`/`top_k` into Responses, Chat Completions, or WebSocket payloads. Native Ollama has a narrower provider-owned params path, and `extra_body` can work as an advanced OpenAI-compatible completions workaround, but neither fully implements the requested config-level sampling-penalty support.

## Best Possible Solution

Keep this issue open as the canonical OpenAI/OpenAI-compatible sampling-penalty passthrough request. The best fix is a focused transport change that normalizes numeric snake_case and camelCase sampling params from existing merged `params` sources, forwards only supported fields for compatible Responses, Chat Completions, Azure, and WebSocket payloads, preserves existing `onPayload`, provider wrappers, and `extra_body` behavior, and documents the supported scope.
