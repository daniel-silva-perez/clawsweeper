## Summary

Keep open. The PR addresses a real current-main Codex app-server state-isolation gap and is not obsolete, but it has a protected `maintainer` label and the new doctor warning misses one of the native Codex loader roots the PR itself documents, so it needs maintainer review and a small fix before merge.

## What This Changes

The PR adds per-agent `CODEX_HOME`/`HOME` wiring for local Codex stdio app-server launches, a Codex migration provider with doctor/docs updates for native Codex assets, and tests around Slack/Signal untrusted system-event handling.

## Best Possible Solution

Keep the per-agent stdio app-server isolation direction, but update the doctor scanner and tests to cover both native Codex loader roots (`$CODEX_HOME/skills` and `$HOME/.agents/skills`) before a maintainer lands the PR after targeted Codex, migration, doctor, and channel-trust validation.

## Reproduction Assessment

Yes for the review finding: configure a Codex-mode setup with only `$HOME/.agents/skills/<name>/SKILL.md` and no `CODEX_HOME` assets, then call the new doctor scanner; the PR code returns no warning because it scans only `CODEX_HOME` paths. For the original app-server state leak, source inspection on current main shows the stdio path has no managed `CODEX_HOME`/`HOME` injection, but I did not run a live app-server in this read-only pass.

## Solution Assessment

Not yet. The core per-agent `CODEX_HOME`/`HOME` injection is the right narrow fix for local stdio launches, but the doctor/migration support should consistently account for `$HOME/.agents/skills` before this is the best complete solution.
