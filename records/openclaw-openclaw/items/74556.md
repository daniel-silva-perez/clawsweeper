## Summary

Keep open. Current main still does not isolate local Codex stdio app-server state, while this PR adds the per-agent `CODEX_HOME`/`HOME` isolation and related migration, doctor, docs, channel-trust test, and auth-profile forwarding work. The protected `maintainer` label, XL scope, and auth/state boundary make this an explicit maintainer review item rather than cleanup-close material.

## What This Changes

The PR isolates local Codex stdio app-server state per agent, adds a Codex migration provider and doctor/docs coverage for native Codex assets, adds Slack/Signal untrusted system-event tests, and forwards OpenAI Codex auth profiles to configured Codex harness runs.

## Best Possible Solution

Land the PR after explicit maintainer review and green changed-surface validation, preserving the local stdio `CODEX_HOME`/`HOME` isolation, explicit migration/doctor transition path, harness auth-profile forwarding, and channel trust-boundary coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows local stdio app-server launches inherit the Gateway environment without managed `CODEX_HOME` or `HOME`, and the upstream Codex skill locations make that a credible state leak path; I did not launch a live app-server because this was a read-only review.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix: isolate only OpenClaw-owned local stdio app-server launches, leave WebSocket app-server ownership external, preserve explicit overrides, and keep native Codex plugins/hooks/config manual-review rather than auto-activating them.
