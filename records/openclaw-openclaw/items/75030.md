## Summary

Keep open. Current main still lacks the OpenAI Codex OAuth legacy-alias normalization and the config/store mode-mismatch diagnostic, while this PR directly addresses the linked auth-profile bug with a narrow parser and error-message change. I found no concrete correctness or security blocker in the diff; the remaining action is maintainer review and normal PR landing.

## What This Changes

The PR normalizes legacy OpenAI Codex OAuth fields during persisted auth-profile parsing, adds a secret-free auth-profile mode-mismatch hint to missing provider-auth errors, and adds regression tests for both failure modes.

## Best Possible Solution

Land this PR, or an equivalent narrow patch, after maintainer review and required CI, keeping the fix limited to persisted auth-profile compatibility plus metadata-only diagnostics and letting the linked bug close through the PR’s closing reference.

## Reproduction Assessment

Yes. On current main, an alias-only `openai-codex` OAuth profile is loaded without canonical fields and then rejected by PI credential conversion; a valid stored OAuth profile with `auth.profiles.<id>.mode = "api_key"` is filtered as a mode mismatch before the generic missing-auth error.

## Solution Assessment

Yes. Normalizing known OpenAI Codex OAuth aliases at persisted-profile parsing and adding a provider-auth-alias-aware, token-redacted mismatch hint is the narrowest maintainable solution I found for the two reported failure modes.
