## Summary

Keep this PR open. Current `main` still has the linked auth-profile compatibility gap, and the PR is a focused implementation that appears to address both observed failure modes; I found no blocking correctness issue in the latest diff, but CI was still pending at review time.

## What This Changes

The PR normalizes legacy OpenAI Codex OAuth alias fields during auth-profile parsing, adds a metadata-only mode-mismatch hint to missing-auth errors, and adds regression coverage for both paths.

## Best Possible Solution

Land this narrow auth compatibility PR after maintainer review and green required CI, then let the linked #47964 close through the PR’s closing reference while keeping the diagnostic limited to profile/provider/mode metadata.

## Reproduction Assessment

Yes. On current `main`, an alias-only `openai-codex` OAuth profile is not normalized into the canonical fields consumed by PI credential conversion, and a stored OAuth profile with `auth.profiles.<id>.mode = "api_key"` is filtered as a mode mismatch before the generic missing-key error.

## Solution Assessment

Yes. Normalizing the legacy aliases at persisted-profile parsing and appending an alias-aware, secret-free missing-auth hint is the narrowest maintainable fix I found; the latest PR head also addresses the earlier provider-auth-alias diagnostic gap.
