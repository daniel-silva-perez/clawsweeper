## Summary

Keep this issue open. Current main still does not expose a localhost/private-LAN opt-in for `web_fetch`; it only accepts and forwards `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`, which solves fake-IP proxy compatibility but not the reporter's local/private service use case. The discussion also has an open same-author implementation candidate, PR #39630, so the right maintainer path is to review, land, replace, or close that PR rather than close this issue early.

## Best Possible Solution

Keep this issue open while maintainers review, update, land, replace, or close PR #39630. The strongest core path is a default-off `tools.web.fetch.ssrfPolicy` surface with explicit private-network naming and/or narrow hostname controls, wired through the strict schema, exported types, help/docs, runtime policy forwarding, cache-key isolation, and focused SSRF regression tests.

## Risks / Open Questions

- A private-network opt-in broadens SSRF reach when enabled, so any implementation should stay default-off, use explicit dangerous naming or narrow hostname controls, isolate cache entries by policy, and cover direct host, DNS, redirect, and provider fallback paths.
- The existing RFC2544 opt-in can help fake-IP proxy environments but does not satisfy the localhost/private-LAN workflow described by the reporter and commenters.
