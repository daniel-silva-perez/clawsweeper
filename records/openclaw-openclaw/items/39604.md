## Summary

Keep this issue open. Current main still does not expose a private-network or localhost opt-in for `web_fetch`; it only exposes `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`, which covers fake-IP proxy compatibility, not the reporter's local/private service workflow. PR #39630 is an open same-author implementation candidate that should be reviewed, replaced, landed, or closed before this issue is closed.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review, update, land, replace, or close PR #39630. A strong core fix would expose a default-off `tools.web.fetch.ssrfPolicy` private-network option or narrower hostname controls through the strict schema, exported types, help/docs, runtime policy forwarding, provider fallback behavior, cache-key isolation, and SSRF regression tests.

## Risks / Open Questions

- A private-network opt-in broadens SSRF reach when enabled, so any implementation should remain default-off and include explicit naming, cache-key isolation, DNS/redirect coverage, and focused tests.
- The shipped RFC2544 option helps fake-IP proxy environments but does not satisfy the localhost/private-LAN workflow reported here and by commenters.
- Closing this issue before PR #39630 is resolved would hide an active implementation candidate and break the linked issue/PR cleanup policy.
