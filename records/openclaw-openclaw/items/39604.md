## Summary

Current main still does not expose a localhost/private-LAN opt-in for `web_fetch`. The low-level SSRF policy supports private-network allowances, but `tools.web.fetch` only accepts and forwards `ssrfPolicy.allowRfc2544BenchmarkRange`, which is limited to fake-IP proxy compatibility. The same-author PR #39630 remains the direct implementation candidate in the provided GitHub context, so this issue should stay open for maintainer review rather than be closed.

## Best Possible Solution

Keep this issue open while maintainers review, land, replace, or close PR #39630. The strongest core path is likely a default-off `tools.web.fetch.ssrfPolicy` surface using explicit dangerous private-network naming and/or narrow hostname controls, wired through the strict schema, exported types, generated help, runtime policy forwarding, cache isolation, docs, and focused SSRF regression tests.

## Risks / Open Questions

- A private-network opt-in expands SSRF reach when enabled, so the eventual design should stay default-off, use explicit dangerous naming or narrower host controls, isolate cache keys by policy, and cover direct host, DNS, redirect, and provider fallback paths.
- The existing RFC2544 support solves fake-IP proxy environments only; it does not address the reporter's localhost/private-LAN workflow.
