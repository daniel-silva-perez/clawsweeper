## Summary

Keep open. Current main still does not give `before_tool_call` hooks a trustworthy current browser page URL before browser `act`/`type` runs; the generic hook path forwards only existing params, while browser route URL resolution happens inside route execution and response assembly. The reporter's credential-vault workaround is useful and fail-secure, but it confirms the native hook-time contract gap remains.

## What This Changes

Expose the browser current tab URL, or an equivalent query seam, to `before_tool_call` hooks for browser `act`/`type` actions so credential plugins can enforce domain pins before text injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests proving blocked/private URLs are omitted and supported browser profiles behave consistently.

## Reproduction Assessment

Yes. The source path is enough to reproduce the contract gap: a browser `act` call reaches `runBeforeToolCallHook` with only the original params before the browser route code that can resolve a tab URL runs.

## Solution Assessment

No for the exact raw `_currentUrl` sketch; the maintainable solution should expose only a policy-safe current URL or supported query seam. The current workaround is fail-secure but duplicates state in each plugin and does not close the native contract gap.
