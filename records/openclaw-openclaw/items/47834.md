## Summary

Keep open. Current main still does not provide a current browser page URL to `before_tool_call` hooks before browser `act`/`type` executes, and the credential-vault workaround described by the reporter confirms a real plugin security contract gap rather than an obsolete request.

## What This Changes

Request that browser `before_tool_call` hook parameters for `act`/`type` include the current page URL, or provide an equivalent hook-safe way to query it before credential text injection.

## Best Possible Solution

Add an additive, documented browser/plugin contract that gives pre-action policies a policy-safe current tab URL for browser `act`/`type`, or an equivalent supported query seam, with tests covering managed and existing-session browser paths plus blocked/private URL omission.

## Reproduction Assessment

Yes. The source path is enough to reproduce the contract gap: a browser `act` call reaches `before_tool_call` with only original tool params, while active-tab URL resolution happens later inside browser execution and response construction.

## Solution Assessment

No for the issue's raw `_currentUrl` sketch as-is. A maintainable fix should expose a policy-safe current URL or explicit query seam rather than copying arbitrary tab URLs into every hook params object.
