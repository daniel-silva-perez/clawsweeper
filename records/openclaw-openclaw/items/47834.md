## Summary

Keep open. Current main still does not provide a current browser page URL to `before_tool_call` hooks before browser `act`/`type` executes; the hook receives the original params before the browser route can resolve and policy-check the active tab URL. The reporter's credential-vault workaround is fail-secure and useful, but it confirms the native hook-time contract gap remains.

## What This Changes

Expose the browser tab's current URL, or an equivalent supported query seam, to `before_tool_call` hooks for browser `act`/`type` actions so domain-pinned credential plugins can verify the target before text injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests proving blocked/private URLs are omitted and supported browser profiles behave consistently.

## Reproduction Assessment

Yes. The source path is enough to reproduce the contract gap: a browser `act` call reaches `before_tool_call` with only the original params, while the browser route code that can resolve the active tab URL runs after that decision point.

## Solution Assessment

No for the exact raw `_currentUrl` sketch. The maintainable solution should expose only a policy-safe current URL, or a supported browser query seam, rather than copying arbitrary tab URLs into every hook event.
