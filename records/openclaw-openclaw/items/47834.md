## Summary

Keep open. Current `main` still does not provide a trustworthy current browser page URL to `before_tool_call` hooks before browser `act`/`type` executes; the existing safe URL resolution is inside the browser route and response assembly, after the hook decision point. The reporter's credential-vault workaround is useful and fail-secure, but it confirms the native plugin hook contract gap remains.

## What This Changes

The issue requests adding the browser current page URL, or an equivalent supported query seam, to `before_tool_call` data for browser `act`/`type` actions.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests proving blocked/private URLs are omitted and supported browser profiles behave consistently.

## Reproduction Assessment

Yes. The source path is enough to reproduce the contract gap: a browser `act` call reaches `runBeforeToolCallHook` with only the original params before the browser route code that can resolve the active tab URL runs.

## Solution Assessment

No for the exact raw `_currentUrl` sketch; the maintainable solution should expose only a policy-safe current URL or supported query seam. The current workaround is fail-secure but duplicates state in each plugin and does not close the native contract gap.
