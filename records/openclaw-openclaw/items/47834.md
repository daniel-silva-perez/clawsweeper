## Summary

Keep open. Current main still exposes only the incoming browser tool parameters to `before_tool_call`; browser `act`/`type` calls are not enriched with a policy-checked current tab URL before plugins decide whether to rewrite, approve, or block. The credential-vault workaround is useful and fail-secure, but it is still an external in-memory cache workaround, and related PR #30323 is closed unmerged and targeted response URLs rather than hook-time policy input.

## What This Changes

Expose the browser tab’s current page URL to `before_tool_call` for browser `act`/`type` actions, or provide an equivalent hook-time query seam, so plugins can enforce domain-pinned credential injection.

## Best Possible Solution

Add a narrow, additive browser/plugin API contract that gives `before_tool_call` policy decisions a policy-checked current tab URL for browser `act`/`type` calls, or an equivalent supported query seam, with tests and docs covering managed browser, node proxy, and existing-session profiles.
