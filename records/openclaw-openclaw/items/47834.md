## Summary

Keep open. Current main still does not provide the current browser page URL to `before_tool_call` hooks before browser `act`/`type` execution, and the credential-vault workaround described in the discussion confirms a real plugin security contract gap rather than an obsolete request.

## What This Changes

The issue asks for browser `act`/`type` `before_tool_call` hooks to receive the current tab URL, or an equivalent way to query it, so credential plugins can enforce domain pins before injecting secrets.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests proving blocked/private URLs are omitted and managed plus existing-session profiles behave consistently.

## Reproduction Assessment

Yes. The source path is enough to reproduce the contract gap: a browser `act` call reaches `before_tool_call` with the original tool params before browser execution, while active-tab URL resolution happens later inside the browser route and response construction.

## Solution Assessment

No for the issue's raw `_currentUrl` sketch as-is. The maintainable solution should expose a policy-safe current URL through a typed hook context or explicit query seam rather than copying arbitrary tab URLs into every hook params object.
