## Summary

Keep open. Current main still does not provide a browser tab URL at the `before_tool_call` decision point for browser `act`/`type`; the generic hook runner only sees the original params before the browser tool executes, while the browser route can add a policy-checked `url` only to the post-execution response. The reporter's credential-vault workaround is useful and fail-secure, but it confirms the remaining hook-time API gap.

## What This Changes

The issue asks for browser `before_tool_call` events for `act`/`type` actions to include the current tab URL or provide an equivalent query path so plugins can enforce domain-pinned credential injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests proving blocked/private URLs are omitted and supported profiles behave consistently.
