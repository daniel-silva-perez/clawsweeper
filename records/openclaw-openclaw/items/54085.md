## Summary

Keep open. Current main has a narrower respawn policy than this PR proposes: it skips help/version and foreground `gateway` runs, but it still keeps the respawn path for `status`, `gateway status`, and other short-lived commands. The related lazy command-registration work does not supersede this PR because the respawn plan runs before the CLI command loader. Security review of the provided PR diff found no CI, dependency, lockfile, script, package publishing, secrets, or downloaded-artifact changes; the remaining concern is functional startup behavior.

## Best Possible Solution

Keep this PR open as a plausible implementation candidate. The best path is to rebase it onto the current `resolveCliArgvInvocation`/foreground-gateway respawn policy, preserve the existing gateway skip, decide the maintained command allowlist or safer equivalent, and add focused respawn-policy tests for root options, help/version, unknown commands, and sqlite-warning-safe command paths.
