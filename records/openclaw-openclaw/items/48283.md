## Summary

Keep open. The linked abort bug still appears valid on current main, but this PR branch is dirty and the proposed post-prompt cleanup is not a merge-ready shape because aborted assistant text can already pass through message_end delivery and persistence before the cleanup runs.

## What This Changes

The PR adds a post-abort helper in the embedded runner that removes trailing assistant messages with stopReason="aborted" from session state and SessionManager file entries.

## Best Possible Solution

Land a current-main fix that gates aborted assistant content before user-visible message_end delivery and transcript persistence, while sharing tail cleanup with the sessions_yield path and adding focused regression coverage for ordinary RunnerAbortError cleanup and channel/block-reply suppression.

## Reproduction Assessment

Yes. The linked issue gives a concrete Telegram rapid-message reproduction, and the source-level path is reproducible by injecting a stopReason="aborted" assistant message_end: current main can persist and emit it unless it is the special sessions_yield path.

## Solution Assessment

No. The PR direction is useful, but cleanup after activeSession.prompt rejects is too late for message_end delivery and the helper is stale against the current Agent API; the safer fix gates or strips aborted assistant output before subscription/session consumers see it.
