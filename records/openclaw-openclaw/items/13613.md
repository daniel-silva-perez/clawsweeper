## Summary

Keep open. Current main still treats cron as a Gateway-local scheduler with local job/state files, process-local serialization, local status and CLI surfaces, and no accepted distributed coordination, lease, leader-election, backend-health, or cron-leader contract. The broader #25883 horizontal-scaling proposal remains open, and #66712 explicitly scopes its dedup guard to process-local behavior, so neither supersedes this cron-specific HA request.

## What This Changes

Add distributed cron ownership or locking so multiple Gateway instances in HA deployments do not execute the same scheduled jobs concurrently.

## Best Possible Solution

Define a supported distributed cron ownership model for HA deployments, then implement it behind a versioned config or plugin/API boundary with explicit backend health, leader/lease visibility, CLI inspection, docs, and tests that simulate separate Gateway processes sharing scheduled jobs.
