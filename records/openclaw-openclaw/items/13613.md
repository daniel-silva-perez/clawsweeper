## Summary

Keep open. Current main still treats cron as a Gateway-local scheduler with local job/state storage, process-local serialization, local status/CLI surfaces, and no distributed lease, leader election, shared coordination backend, or HA ownership status. The broader horizontal-scaling proposal in #25883 is still open and does not supersede this cron-specific feature gap.

## What This Changes

The issue requests distributed coordination so scheduled cron jobs execute once across multiple high-availability Gateway instances.

## Best Possible Solution

Define a supported distributed cron ownership model for HA deployments, then implement it behind a versioned config or plugin/API boundary with health/status visibility, CLI inspection, docs, and tests that simulate separate Gateway processes sharing scheduled jobs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: two separate Gateway processes sharing the same due cron job can race because the only current lock is process-local and timer execution has no external lease before marking jobs running.

## Solution Assessment

Unclear. The issue’s Redis/etcd/DynamoDB approaches are plausible, but the narrowest maintainable solution needs maintainer decisions on backend support, failure behavior, security model, and its relationship to the broader #25883 scaling proposal.
