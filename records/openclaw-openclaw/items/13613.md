## Summary

Keep open. Current main still implements cron as Gateway-local scheduling with local job/state files, process-local locking, local status/CLI surfaces, and observer-only plugin hooks; it has not added distributed leases, leader election, shared coordination backends, HA ownership status, or a cron leader command. The broader horizontal-scaling proposal is still open and does not supersede this cron-specific feature gap.

## What This Changes

The issue requests distributed coordination for cron jobs so multiple OpenClaw Gateway instances do not execute the same scheduled work concurrently.

## Best Possible Solution

Define a supported distributed cron ownership model for HA deployments, then implement it behind a versioned config or plugin/API boundary with backend health, leader or lease visibility, CLI inspection, docs, and tests that simulate separate Gateway processes sharing scheduled jobs.

## Reproduction Assessment

Yes. The feature gap is reproducible by inspecting current main or by running two separate Gateway processes against the same cron state: current code has local state plus process-local locking and no distributed owner check before execution.

## Solution Assessment

Unclear. Redis, etcd, DynamoDB, or queue-backed approaches are plausible, but the best maintainable fix needs maintainer decisions on ownership semantics, dependency boundary, failure behavior, health/CLI surface, and how it relates to #25883.
