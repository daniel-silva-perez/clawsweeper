## Summary

Keep this issue open. Current main still has gateway-local cron scheduling with process-local serialization and no distributed cron ownership, lock backend, leader election, lease TTL config, coordination health, or `openclaw cron leader`-style status surface. The existing maintainer/Codex review correctly distinguishes related #69738 and #25883 from this narrower HA cron coordination request.

## Best Possible Solution

Keep this open for maintainer design. A complete path should define distributed cron ownership semantics, config/schema/docs, lease TTL and failure behavior, health and CLI visibility, tests that simulate multiple gateway processes or nodes, and the relationship to #25883. If core should not own Redis/etcd/DynamoDB backends, the best alternative is a documented, versioned lock-provider or scheduler-ownership API that plugins can implement.
