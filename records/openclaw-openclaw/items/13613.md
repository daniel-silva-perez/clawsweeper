## Summary

Keep open. Current main still treats cron as Gateway-local scheduling with local job/state files, process-local locking, local status/CLI surfaces, and no distributed lease, leader election, shared backend, or HA ownership status; the related horizontal-scaling proposal is still open rather than a merged replacement.

## What This Changes

Add distributed coordination for cron jobs so multi-Gateway HA deployments do not execute the same scheduled job concurrently.

## Best Possible Solution

Define a supported distributed cron ownership model for HA deployments, then implement it behind a versioned config or plugin/API boundary with backend health, leader or lease visibility, CLI inspection, docs, and tests that simulate separate Gateway processes sharing scheduled jobs.

## Reproduction Assessment

Yes. The feature gap is reproducible by inspecting current main or by running two Gateway processes against the same scheduled cron state: the code has only local state and process-local locking, and no distributed owner check exists before execution.

## Solution Assessment

Unclear. The requested Redis/etcd/DynamoDB approaches are plausible, but the best maintainable fix requires a maintainer decision on ownership semantics, backend/API boundary, failure behavior, health/CLI surface, and relationship to #25883 before implementation.
