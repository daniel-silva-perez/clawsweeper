## Summary

Keep this issue open. Current main still implements cron as a Gateway-local scheduler with local store serialization only; it has no distributed cron ownership, lease/lock backend, leader election, coordination config, coordination health, or `openclaw cron leader` status surface. Related #69738 addressed same-store duplicate protection in a local/same-host shape, and #25883 is a broader horizontal-scaling proposal with useful Redis/BullMQ context, but neither replaces this narrower HA cron coordination request.

## What This Changes

Request to add distributed coordination, locking, or leader election for cron jobs across multiple OpenClaw Gateway instances in HA deployments.

## Best Possible Solution

Keep this open for maintainer design. The best path is to define distributed cron ownership semantics, config/schema/docs, lease TTL and failure behavior, health and CLI visibility, tests that simulate multiple Gateway processes or nodes, and the relationship to #25883. If core should not own Redis/etcd/DynamoDB backends, add a documented, versioned lock-provider or scheduler-ownership API that plugins can implement.

## Risks / Open Questions

- Distributed cron coordination changes can create duplicate or missed deliveries if lease expiry, crash recovery, and idempotency semantics are not specified first.

- The current same-process duplicate prevention test can be mistaken for HA coverage even though it does not exercise separate Gateway processes or nodes.
