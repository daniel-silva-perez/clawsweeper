## Summary

Keep this PR open for maintainer follow-up. It targets a real Telegram health/status symptom, but current main has since moved the health path to runtime-backed account snapshots, and the PR's fallback would now conflate a successful Telegram probe with the bot runtime actually being active.

## What This Changes

The PR changes Telegram's channel summary to fill missing `tokenSource` from the resolved account and derive `running` from a successful probe, with regression tests for minimal health snapshots.

## Best Possible Solution

Keep `running` sourced from channel runtime lifecycle, keep `tokenSource` sourced from inspected account metadata, and fix any remaining Telegram health mismatch by refreshing or invalidating cached health after channel startup plus adding focused health snapshot coverage.

## Reproduction Assessment

Yes. The PR's old failure can be reproduced with a configured Telegram snapshot that lacks `running` and `tokenSource` but has a successful probe; current main should instead be checked through `getHealthSnapshot` with gateway runtime state and the cached-health path.

## Solution Assessment

No. The PR's approach is not the best current fix because a successful Telegram API probe proves the token works, not that the polling or webhook runtime is running; the safer fix belongs in the runtime-backed health/cache path.
