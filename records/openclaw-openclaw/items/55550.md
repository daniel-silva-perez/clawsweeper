## Summary

Keep this PR open for maintainer follow-up. It targets a real Telegram health/status mismatch, but the proposed fix would treat a successful Telegram credential probe as proof that the bot runtime is running; current main has moved the safer path toward runtime-backed health snapshots and the remaining mismatch is tracked in #59287.

## What This Changes

The PR changes Telegram status summaries to backfill missing `tokenSource` from the resolved account, infer `running` from successful probes when snapshots omit it, and add Telegram health fallback regression tests.

## Best Possible Solution

Keep `running` sourced from channel runtime lifecycle, keep `tokenSource` sourced from inspected account metadata, and fix any remaining Telegram health mismatch by refreshing or invalidating cached health after channel startup with focused health snapshot coverage.

## Reproduction Assessment

Yes. The old failure can be modeled with a configured Telegram snapshot that lacks `running` and `tokenSource` but has a successful probe; the current-main follow-up should reproduce through plain `openclaw health --json` plus the gateway cached-health path tracked in #59287.

## Solution Assessment

No. A successful Telegram API probe proves the token works, not that polling or webhook runtime is active; the safer fix belongs in the shared runtime-backed health/cache lifecycle path.
