## Summary

Keep this PR open for maintainer follow-up. It targets a real Telegram health/status symptom, but current main has since moved health snapshots toward runtime-backed account state, and this PR’s fallback would conflate a successful Telegram credential probe with the bot runtime actually running.

## What This Changes

The PR changes Telegram status summaries to fill missing `tokenSource` from the resolved account and derive `running` from probe success, with regression tests for minimal health snapshots.

## Best Possible Solution

Keep `running` sourced from channel runtime lifecycle, keep `tokenSource` sourced from inspected account metadata, and fix any remaining Telegram health mismatch by refreshing or invalidating cached health after channel startup with focused health snapshot coverage.

Yes. The PR’s old failure can be modeled with a configured Telegram snapshot that lacks `running` and `tokenSource` but has a successful probe; current-main follow-up should reproduce through `getHealthSnapshot` with gateway runtime state plus the cached-health path.

## Solution Assessment

No. A successful Telegram API probe proves the token works, not that polling or webhook runtime is active; the safer solution belongs in the runtime-backed health/cache lifecycle path.
