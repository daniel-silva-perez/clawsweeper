## Summary

Keep this issue open. Current main has adjacent pieces, including Web Push subscription/test plumbing and `chat.inject`, but it still lacks a first-class shell-facing `openclaw notify <message>` command, a typed admin-scoped Gateway notification RPC, and a Control UI event handler that renders visible in-app notifications for all connected UI sessions. The newer #20237 overlaps on the broader WebUI toast system, but it does not fully supersede the requested CLI/admin broadcast surface.

## What This Changes

The issue requests an `openclaw notify <message>` CLI command or equivalent that sends a visible notification bubble to all connected UI sessions through an admin-scoped broadcast without creating an agent turn.

## Best Possible Solution

Define an intentional core feature: a typed admin-scoped Gateway RPC and broadcast event for UI notifications, a CLI entry point such as `openclaw notify <message>`, and Control UI rendering for the event. It should document auth scope, rate limiting or abuse controls, delivery guarantees, and how this differs from `chat.inject`, `system event`, and Web Push, with #20237 used as the broader UI-design coordination point.
