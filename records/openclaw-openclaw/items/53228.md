## Summary

Keep open. Current main still lacks the requested shell-facing notification path, and the adjacent Web Push, chat injection, system-event, and broader WebUI toast work do not fully cover this CLI/admin broadcast surface. The remaining work crosses CLI, Gateway protocol/auth, broadcast scoping, and Control UI UX, so it needs maintainer product/API review before implementation.

## What This Changes

Add an `openclaw notify <message>` CLI command, or equivalent, that broadcasts a visible notification bubble to connected UI sessions through an admin-scoped Gateway path without running an agent turn.

## Best Possible Solution

Define an intentional core notification contract: a typed admin-scoped Gateway RPC and allowlisted broadcast event, a CLI entry point such as `openclaw notify <message>`, Control UI rendering for connected clients, and docs/tests covering authorization, delivery semantics, and its relationship to existing session, system-event, Web Push, and cross-session delivery paths.
