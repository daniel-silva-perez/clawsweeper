## Summary

Keep this issue open. Current main has an opt-in active-transcript byte trigger, but the default embedded gateway path still whole-file reads/parses the active session JSONL before opening it, so the large-transcript event-loop starvation described in the discussion remains plausible for default gateway/channel users.

## Best Possible Solution

Keep this issue open and implement bounded large-active-transcript recovery in the embedded gateway run path. Add a pre-open stat guard before repairSessionFileIfNeeded and SessionManager.open; for oversized active transcripts, either rotate/compact with yielding or worker isolation, or fail fast with an actionable recovery message while keeping gateway HTTP/RPC/channel handling responsive. Add regression coverage for an oversized JSONL and document the default recovery path plus the existing opt-in byte threshold.

## Evidence

- **Concrete reproduction in discussion:** A commenter reported an 89 MB / about 40,000-line main session JSONL causing 100% CPU and no further logs after the bootstrap session queue is acquired, freezing Telegram, WeChat, and Feishu until the large transcript is moved aside.
- **Embedded run still repairs then opens active transcript without a pre-open size guard:** The run path calls repairSessionFileIfNeeded, checks existence with stat, prewarms a small chunk, then calls SessionManager.open(params.sessionFile).
