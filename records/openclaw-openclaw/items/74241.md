## Summary

Current main has partial timezone support, but it still does not satisfy the reported multi-timezone drift: gateway turn timestamps, prompt time context, envelopes, and heartbeat active-hours resolution all depend on global `agents.defaults.userTimezone` or host-local fallback, while `agents.list[]` has no per-agent/profile timezone field. The reporter's follow-up confirms the production drift pattern, so this should remain open and is narrow enough for a focused fix PR.

## What This Changes

The issue requests user-local timezone context on agent turns, including per-agent/profile timezone configuration, so time-based reasoning and scheduled checks do not drift when the container timezone differs from the user's timezone.

## Best Possible Solution

Add a single effective-timezone resolver keyed by agent id/profile that falls back to `agents.defaults` and then host timezone, expose per-agent timezone/time-format/envelope overrides in the config contract, and route gateway turn timestamps, system prompt params, envelope formatting, startup/compaction time context, and heartbeat active-hours through that resolver without changing cron defaults unless the user explicitly configures a timezone.
