## Summary

Keep open. Current main still has only defaults-level timezone support, so the reported Docker host/user timezone split remains possible for per-agent profiles, gateway-injected turn timestamps, envelopes, startup/compaction time context, and heartbeat active-hours resolution.

## What This Changes

The issue asks OpenClaw to expose and inject the user's local timezone/time on every agent turn, including per-agent/profile configuration, so time-based reasoning and heartbeat scheduling do not follow the container clock.

## Best Possible Solution

Add one effective timezone/time-format resolver keyed by agent id/profile, fall back through agents.defaults and host timezone, and route gateway timestamp injection, prompt time context, envelope formatting, startup/compaction date selection, and heartbeat active-hours through that resolver while keeping cron semantics separate.

## Reproduction Assessment

Yes. A static reproduction is enough: define or imagine an agent in agents.list[] that needs Europe/London while the container/default is Europe/Berlin; current strict per-agent config has no timezone fields, and all inspected runtime paths resolve from agents.defaults or host fallback.

## Solution Assessment

Yes. Extending the existing timezone resolution seams is the narrow maintainable fix; it avoids adding a dynamic clock to the cached system prompt and keeps the current timestamp/envelope/heartbeat code paths as the integration points.
