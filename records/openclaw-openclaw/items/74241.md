## Summary

Keep open. Current main still has only defaults-level timezone handling: gateway timestamp injection, system-prompt params, envelopes, and heartbeat active-hours all resolve from `agents.defaults.userTimezone` or host-local fallback, while per-agent/profile timezone fields are absent from the public config type and strict schema. The reporter also confirmed the production drift pattern after the first review, so this is a current, concrete bug rather than stale cleanup.

## What This Changes

Requests per-turn user-local timezone context and per-agent/profile timezone configuration so agents do not reason from the container or gateway host timezone in multi-timezone deployments.

## Best Possible Solution

Add a single effective-timezone resolver keyed by agent id/profile, with fallback from per-agent settings to `agents.defaults` and then host timezone, and route gateway turn timestamps, system-prompt timezone params, envelope formatting where agent context is known, startup/compaction time context, and heartbeat active-hours through that resolver without changing cron semantics or default heartbeat cadence.
