## Summary

Keep open. Current main has useful partial timezone support, but it still resolves the relevant agent-visible time surfaces from global `agents.defaults.userTimezone` or host-local fallback and does not expose a per-agent/profile timezone contract. The reporter’s production follow-up matches the current code path, so this is a valid, narrow bug rather than stale cleanup.

## What This Changes

The issue requests user/profile timezone injection so agent turns, prompts, envelopes, and heartbeat scheduling use the user’s local timezone instead of the container or gateway host clock.

## Best Possible Solution

Add an additive effective-timezone resolver keyed by agent id/profile that falls back to `agents.defaults` and then host timezone, expose the needed per-agent config fields, and route gateway turn timestamps, prompt timezone context, envelope formatting, startup/compaction time context, and heartbeat active-hours through that resolver without changing cron `tz` semantics or defaults unexpectedly.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: run a gateway/container in one timezone with a user expecting another, leave only host/default timezone behavior available, then send `agent`/`chat.send` turns or heartbeat active-hours with `timezone: "user"`; current code resolves those paths from host/global defaults rather than a per-user/profile timezone.

## Solution Assessment

Yes, the requested direction is best solved as a narrow additive config/runtime resolver rather than injecting dynamic clocks into the cached system prompt or changing cron scheduling globally. The safer implementation is to extend existing timestamp and timezone seams, not create a parallel scheduling system.
