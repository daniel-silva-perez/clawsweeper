## Summary

Current main still uses the fixed 2-hour cron stuck-running marker cleanup. The PR's timeout-aware stale-marker recovery is not implemented or superseded by the adjacent restart/manual-run recovery paths.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best path is to port the timeout-aware stale-marker helper and regression test onto current main, then explicitly decide whether the threshold should use all effective cron timeouts or only jobs with explicit `timeoutSeconds`.

## Risks / Open Questions

- The functional review question from the PR discussion still matters: deriving stale cleanup from synthesized defaults would clear `systemEvent` jobs after about 11 minutes and default `agentTurn` jobs after about 61 minutes instead of 2 hours, so maintainers should decide whether the wider cleanup scope is intended or should apply only to explicit `timeoutSeconds` jobs.
- Earlier stale-marker cleanup can permit duplicate execution if a legitimately slow run remains active while its persisted marker is cleared, so the final implementation should keep focused regression coverage around scheduler concurrency.
