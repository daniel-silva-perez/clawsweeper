## Summary

Current main still uses a fixed 2-hour stale `runningAtMs` cleanup threshold. The PR’s timeout-aware stale-marker cleanup is not implemented or superseded by the adjacent manual-run, startup, or active-execution timeout paths, so this PR remains a plausible focused bug fix that needs maintainer review/rebase rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best path is to port the timeout-aware stale-marker helper and regression test onto current main, then explicitly settle the threshold policy for synthesized default timeouts versus explicit `timeoutSeconds` jobs.

## Risks / Open Questions

- Merging the PR as written may clear `runningAtMs` earlier for default-timeout `systemEvent` and `agentTurn` jobs, so maintainers should decide whether timeout-aware cleanup should apply to all effective cron timeouts or only explicit `timeoutSeconds` jobs.
- Earlier marker cleanup can permit duplicate execution if a legitimately slow job is still active while persisted state is cleared; the final implementation should keep regression coverage around scheduler concurrency and stale-marker recovery.
