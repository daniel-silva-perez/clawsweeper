## Summary

Keep open. Current main still uses a fixed 2-hour stale-running threshold in cron maintenance, while this PR addresses a real remaining gap by aligning stale marker cleanup with the existing cron timeout policy. I did not find a blocking correctness or security issue in the patch, but the branch is currently dirty against main and needs rebase plus fresh checks before merge.

## What This Changes

This PR changes cron maintenance to derive stale `runningAtMs` cleanup from `resolveCronJobTimeoutMs(job)` with a grace window and floor, logs the resolved stale threshold, and adds a regression test for short-timeout isolated `agentTurn` jobs.

## Best Possible Solution

Rebase the PR onto current main, keep the timeout-aware stale-marker threshold in `normalizeJobTickState`, and merge after the targeted cron regression test plus the changed gate are green.

## Reproduction Assessment

Yes. On current main, a cron `agentTurn` job with `timeoutSeconds: 900` and `runningAtMs` older than timeout plus grace but less than 2 hours remains blocked because maintenance still checks the fixed 2-hour `STUCK_RUN_MS`; the PR's added test models that path.

## Solution Assessment

Yes, with the rebase caveat. Reusing `resolveCronJobTimeoutMs` is the narrowest maintainable fix because the timer already uses that helper as the execution timeout contract, and the PR keeps a grace window plus minimum floor for conservative cleanup.
