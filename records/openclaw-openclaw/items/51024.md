## Summary

Keep open. Current main still uses the fixed 2-hour `STUCK_RUN_MS` cleanup for persisted `runningAtMs` markers, and the per-job timeout policy is still only applied to active execution paths. The PR remains a plausible focused cron recovery fix rather than an implemented, superseded, or out-of-scope cleanup item.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best implementation path is to port the timeout-aware stale-marker helper and short-timeout regression test onto current `main`, then explicitly settle the policy for explicit `timeoutSeconds`, synthesized default timeouts, and effectively unlimited jobs. Add or keep tests for short explicit timeouts, default-timeout jobs, `timeoutSeconds <= 0`, future `nextRunAtMs` preservation, and duplicate-execution avoidance.

## Risks / Open Questions

- The PR as written changes cleanup timing for all finite effective cron timeouts, not only jobs with explicit short `timeoutSeconds`; maintainers should decide whether synthesized defaults should clear after about 11 minutes for `systemEvent` and 61 minutes for default `agentTurn`, or whether only explicit timeouts should opt into the shorter stale threshold.
- Earlier marker cleanup can allow duplicate execution if a legitimate slow run is still active but its persisted `runningAtMs` is cleared; the final version should keep concurrency and stale-marker recovery coverage around that boundary.
- The branch may need a rebase onto current main, but the intended change is still narrow and not implemented on main.
- Security review found no supply-chain or secret-handling concern in the provided diff; the remaining review risk is functional scheduling policy, not infrastructure or dependency execution.
