## Summary

Current main still has the fixed 2-hour stale cron running-marker cleanup, so this PR’s timeout-aware recovery is not implemented. Existing restart/manual-run/active-timeout paths reduce adjacent failures, but they do not replace the per-job stale-marker threshold proposed here.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best path is to port the timeout-aware stale-marker helper and regression test onto current main, then decide whether default-timeout jobs should use their synthesized execution timeout window or whether earlier cleanup should apply only when `timeoutSeconds` is explicitly configured.

## Risks / Open Questions

- Functional review risk remains scheduler concurrency: applying synthesized default timeouts would clear `systemEvent` jobs after about 11 minutes and default `agentTurn` jobs after about 61 minutes, so maintainers should explicitly decide whether that earlier recovery is acceptable for slow-but-legitimate runs.
- No supply-chain regression was visible in the provided diff, but the runtime behavior change is shared cron infrastructure and should keep focused regression coverage when rebased.
