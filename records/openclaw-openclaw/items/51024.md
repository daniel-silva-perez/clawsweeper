## Summary

Keep open. Current main still clears persisted cron `runningAtMs` markers with a fixed two-hour `STUCK_RUN_MS`, while the timeout-aware policy is only used for active execution. This PR remains a plausible focused cron recovery fix rather than implemented, superseded, or out of scope. The security pass did not find supply-chain, secret, workflow, dependency, or permission changes in the provided diff; the remaining question is functional scheduling policy.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best implementation path is to port the timeout-aware stale-marker helper onto current main, explicitly decide whether cleanup should use explicit timeouts only or the full effective timeout policy, and keep focused regression coverage for short explicit timeouts, synthesized default timeouts, `timeoutSeconds <= 0`, future `nextRunAtMs` preservation, and duplicate-execution avoidance.

## Risks / Open Questions

- The PR changes cleanup timing for all finite effective cron timeouts, not only jobs with explicit short `timeoutSeconds`; maintainers should decide whether synthesized defaults should clear after about 11 minutes for `systemEvent` and 61 minutes for default `agentTurn` jobs.
- Earlier stale-marker cleanup can permit duplicate execution if an original run is still active or slow to unwind after its timeout window; the final version should keep concurrency and stale-marker recovery tests around that boundary.
- The branch may need a rebase onto current main, but the intended code change is still narrow and not implemented on main.
