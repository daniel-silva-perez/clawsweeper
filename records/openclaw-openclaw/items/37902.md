## Summary

Current main still does not enforce a configured subagent timeout floor or lock. The native `sessions_spawn` path normalizes the model/tool-supplied `runTimeoutSeconds` and the resolver returns that explicit value ahead of `agents.defaults.subagents.runTimeoutSeconds`; config, generated schema, docs, and tests all preserve fallback-only behavior. Related fix PRs are closed unmerged, so this issue remains actionable.

## Best Possible Solution

Keep this issue open until OpenClaw implements a maintainer-approved, backward-compatible timeout constraint for native subagent spawns. A focused fix should add an optional config key such as `agents.defaults.subagents.minRunTimeoutSeconds` or an explicit lock option, clamp model-supplied `sessions_spawn.runTimeoutSeconds` before run registration and gateway dispatch, preserve zero/unset behavior, update generated schema and docs, and add regression coverage for omitted timeouts plus explicit values above and below the configured constraint. Closed PRs #37973, #38028, and #38048 are useful prior art but do not close the issue on current main.

## Risks / Open Questions

- Changing `runTimeoutSeconds` itself from default to floor would be a behavior change for users who intentionally set shorter per-spawn timeouts; an opt-in `minRunTimeoutSeconds` or explicit lock key is safer.
- A fix changes the public config contract, so types, zod schema, generated schema, docs, and regression tests need to move together.
- No live multi-retry `sessions_spawn` reproduction was run in this read-only review, but source, schema, docs, release-tag inspection, and tests are sufficient to show the missing floor.
