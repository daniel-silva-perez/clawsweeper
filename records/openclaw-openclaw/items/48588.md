## Summary

Keep this PR open. Current main still has the Feishu HTTP shutdown behavior the PR targets: `server.close()` is fired without awaiting the close callback, map state is cleared immediately, and the public stop wrapper is still `void`. The linked bug #48183 is still open, older overlapping PR #48329 is closed unmerged, and the supplied PR discussion shows the author addressed several review comments, so this remains the active implementation candidate rather than something already implemented or superseded.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for #48183. The best path is to review the final diff, decide the public `stopFeishuMonitor()` contract, add targeted regression coverage for Feishu webhook shutdown/restart or HTTP close sequencing, and then either land this PR or close it only after a maintainer confirms the bug is not reproducible under supported OpenClaw/Node behavior.
