## Summary

Keep this issue open. Current main has adjacent safeguards, including owner-only gating for some core tools, per-sender group tool policy, cross-context message blocking, trusted sender metadata propagation, and plugin `before_tool_call` blocking/approval hooks. It still does not implement the requested per-action, per-sender authorization gate for `message.send`/`poll`, and the public pre-tool hook context still lacks trusted requester identity for a normal plugin to enforce this reliably.

## Best Possible Solution

Implement a core/API-backed authorization path instead of closing this as cleanup. The narrow fix should add a `tools.message` policy checked before `send` and `poll`, using trusted requester identity and returning a configured denial reason. The broader fix should expose a pre-tool policy surface that includes trusted requester identity, action classification, agent/session/channel/account context, and supports block or owner-confirmation decisions before tool params execute. Update config types, zod schema, docs, and targeted message-action tests.

## Risks / Open Questions

- Non-owner senders who can reach an agent with the `message` tool can still prompt it to call `message.send`; current main has no sender-based protected-action denial at the message action boundary.
- External plugins can block tool calls, but the public `before_tool_call` context lacks trusted requester identity, so a plugin cannot reliably implement the requested owner-only message-send policy without a core/API change.
- Current workarounds remain coarse for the reported use case: deny the whole `message` tool, restrict DM access, or block cross-context sends globally rather than validating only protected message actions by sender.
