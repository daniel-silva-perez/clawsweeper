## Summary

Keep this issue open. Current main has useful adjacent pieces, including pre-tool hooks, trusted tool policies, requester metadata flowing into the message action runner, cross-context blocking, and group-level per-sender tool allow/deny. It does not implement the requested per-action, per-sender authorization for the `message` tool, and the public hook/config surfaces do not currently expose enough sender context to implement the exact policy as an external plugin without a core/API change.

## Best Possible Solution

Keep the issue open and implement a core/API-backed authorization path: either a generic pre-tool policy surface that receives server-injected requester identity (`requesterSenderId`, `senderIsOwner`, channel/account/session) before model tool params execute, or a narrower `tools.message` policy checked in `runMessageAction` before protected actions such as `send`/`poll`. The fix should support per-action policies, deny reasons, and preferably approval/confirmation, with tests covering non-owner `message.send` blocked while lower-risk message actions remain available.
