## Summary

Keep this issue open. Current main has adjacent safety pieces, including trusted tool policies, owner-only tool execution, requester metadata propagated into message-action execution, cross-context send blocking, and group/channel per-sender tool policy. It still does not implement the requested per-action, per-sender authorization gate for `message.send`, and the public `before_tool_call` hook context still lacks trusted requester identity needed for an external plugin to implement this reliably.

## Best Possible Solution

Keep the issue open and implement a core/API-backed authorization path. The narrow fix would add a configurable `tools.message` policy checked before `send` and `poll`, using server-injected requester identity and returning a deny reason. The broader fix would expose a generic pre-tool policy surface that receives trusted requester identity, action classification, agent/session/channel/account context, and supports block or owner-confirmation decisions before tool params execute.

## Risks / Open Questions

- A non-owner who can trigger an agent and has access to the `message` tool can still ask the agent to perform `message.send`; current main has no sender-based protected-action denial for that path.
- Public plugin hooks can block or request approval, but their typed pre-tool context lacks trusted requester identity, so an external plugin cannot reliably implement the exact owner-only policy without a core/API change.
- Existing workarounds remain too coarse for the reported use case: denying the whole `message` tool removes read/react/low-risk message actions, and `allowFrom` blocks the non-owner DM interactions the reporter wants to preserve.
- No current test was found that asserts non-owner `message.send` is blocked while lower-risk message actions remain available.
