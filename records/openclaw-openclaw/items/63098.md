## Summary

Keep open. Current main and the latest shipped tag still preserve the reported Discord voice failure boundaries: initial `/vc join` waits only 15 seconds for `Ready` before destroying the connection, and the later `Disconnected` recovery path waits only 5 seconds for `Signalling` or `Connecting` before silently clearing the session and destroying it. Recent issue comments add independent Linux and macOS confirmations through 2026.4.26, so this remains an actionable Discord plugin reliability bug rather than conservative cleanup.

## What This Changes

The issue requests fixing a Discord voice regression where `/vc join` appears to connect but then aborts or leaves after the 2026.4.5 timeout changes, with suggested longer or configurable connection and recovery timeouts.

## Best Possible Solution

Keep the fix inside the Discord plugin: lengthen and/or make configurable the initial Ready wait and disconnected recovery grace, add a warning on the destroy path with guild/channel context, and add regression tests for both the initial join abort and post-join silent disconnect behavior. If new config fields are introduced, align types, schema, UI hints, docs, generated config metadata, and changelog.

## Reproduction Assessment

Yes. The issue has concrete `/vc join` and auto-join reproduction steps plus multiple independent confirmations, and current source provides a high-confidence simulated reproduction path by forcing `entersState(... Ready ...)` or the disconnected recovery waits to reject.

## Solution Assessment

Yes, a focused Discord plugin fix is the best path. Moving this into core would cross ownership boundaries; the maintainable repair is to adjust the plugin-owned voice manager behavior and its Discord voice config/docs/tests only.
