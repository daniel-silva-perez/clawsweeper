## Summary

Keep open. Current main still matches the reported Discord voice failure path: `/vc join` waits on a hard-coded 15s `Ready` transition and destroys the connection on timeout, while the post-join `Disconnected` recovery path still gives Signalling/Connecting only 5s before silently clearing the active session and destroying the connection. Recent comments confirm the same behavior through `2026.4.26`, so this is still an actionable Discord plugin bug rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to make Discord voice join/reconnect tolerate slower voice handshakes after the 2026.4.5 timeout split by increasing or configuring timeouts, adding retry/recovery logging, and keeping the bot connected.

## Best Possible Solution

Land a focused Discord plugin fix that lengthens and/or makes configurable the initial Ready wait and disconnected recovery grace, logs the destroy path with enough guild/channel context for operators, keeps the behavior inside the Discord plugin boundary, and adds regression coverage for both the initial join abort and post-join silent destroy paths.

Keep the implementation inside the Discord plugin unless a generic config/schema seam is strictly required.
