## Summary

Keep this PR open. Current main still resolves inbound BlueBubbles reply context only from webhook metadata or the process-local reply cache, with no opt-in HTTP API fallback/config/helper/test surface; the PR remains an active implementation of that missing behavior and includes security-relevant network-fetch handling that needs maintainer review rather than cleanup closure.

## What This Changes

The PR adds an opt-in BlueBubbles reply-context HTTP API fallback for cache misses, plus config/types/tests/changelog updates and related attachment failure logging and log-redaction changes.

## Best Possible Solution

Land one BlueBubbles-owned implementation after maintainer/security review that uses the typed `BlueBubblesClient`, keeps the fallback opt-in unless maintainers choose otherwise, preserves account-level inheritance, validates reply IDs before path construction, writes successful fetches back to the reply cache, dedupes concurrent misses, and keeps tests/docs/changelog aligned.
