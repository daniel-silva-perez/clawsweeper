## Summary

Keep open. Current main still lacks the BlueBubbles reply-context API fallback, and this PR remains a useful additive, default-off implementation; the earlier SSRF and account-inheritance review findings have been addressed in the PR head. The remaining cleanup is documentation for the new operator-facing config plus normal maintainer review/rebase because the PR is currently reported unmergeable.

## What This Changes

This PR adds an opt-in BlueBubbles reply-context API fallback with tests/config/types/changelog updates, routes the fetch through the typed BlueBubbles client, promotes attachment download failures to runtime errors, and extends BlueBubbles log redaction.

## Best Possible Solution

Land the BlueBubbles-owned, opt-in fallback after documenting the config knob and resolving mergeability, preserving the typed-client SSRF path, reply-id validation, in-flight dedupe, and cache write-back. Treat any default-on decision as a separate maintainer/product follow-up.

## Reproduction Assessment

Yes. A high-confidence reproduction path is an inbound BlueBubbles reply that has `replyToId` but lacks `replyToBody`/`replyToSender` after a process restart or reply-cache eviction; current main only checks the in-memory cache and then proceeds without fetching the original message.

## Solution Assessment

Yes, with one docs follow-up. Keeping the behavior extension-owned, default-off, best-effort, and routed through `BlueBubblesClient` is the narrow maintainable fix; the safer alternative to address before merge is documenting the new config rather than changing runtime shape again.
