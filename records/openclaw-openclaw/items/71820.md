## Summary

Keep open. Current main does not implement the opt-in BlueBubbles reply-context API fallback, and the competing #73241 is closed unmerged with parts folded into this PR, so this is not cleanup-closeable. The PR needs a focused fix before merge because the new `p:0/<guid>` normalization path caches under the stripped GUID while the caller continues looking up the raw prefixed id for cache reuse and short-id generation.

## What This Changes

The PR adds an opt-in BlueBubbles reply-context API fallback, config/schema/type surface, tests, a changelog entry, and attachment-download error visibility changes.

## Best Possible Solution

Keep the PR open, repair the reply-id normalization so the API fetch, cache lookup/write-back, and reply-tag short-id path use one canonical id, add the missing BlueBubbles docs for the default-off option, then rebase and send it back through targeted BlueBubbles validation and maintainer review.

## Reproduction Assessment

Yes. A code-level reproduction is clear: use a reply id like `p:0/msg-guid` on a cache miss; the helper fetches and caches `msg-guid`, but the caller later checks `getShortIdForUuid("p:0/msg-guid")` and future cache lookups with the prefixed id miss.

## Solution Assessment

No. The feature direction is reasonable and the SSRF route is the right client surface, but the implementation should thread the normalized reply id back to `processMessageAfterDedupe` or normalize before the first cache lookup so prefix-stripped ids amortize correctly.
