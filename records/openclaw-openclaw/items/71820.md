## Summary

Keep this PR open. Current main still does not implement the requested BlueBubbles reply-context API fallback: inbound reply handling only uses webhook metadata plus the process-local reply cache, and there is no shipped config flag, helper, tests, docs, or changelog entry for the proposed opt-in fallback. The PR is a live implementation candidate, with author follow-up addressing prior SSRF, inheritance, and dead-code review findings; the remaining path is maintainer review or choosing the overlapping #73241 as canonical.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land this BlueBubbles-owned opt-in fallback after scoped diff, CI, and security review, or explicitly choose #73241 as the canonical implementation and close the other PR as superseded. The implementation should stay in the BlueBubbles plugin, preserve channel/account config inheritance, use the existing SSRF-guarded BlueBubbles client policy, include cache-miss/failure/dedupe tests, and update docs/changelog if it lands.
