## Summary

Keep open. Current main still lacks the BlueBubbles reply-context API fallback, and the closed related fallback PR was folded into this PR rather than superseding it. The branch is useful but not merge-ready: it is dirty against current main, has a concrete prefixed-reply-id cache write-back bug, and still lacks user documentation for the new config knob.

## What This Changes

This PR adds a default-off BlueBubbles reply-context API fallback on inbound reply cache misses, with config/schema/types/tests/changelog updates plus attachment-download error visibility and log-redaction changes.

## Best Possible Solution

Rebase the branch onto current main, preserve the default-off typed-client fallback, normalize/cache prefixed `p:0/<guid>` reply ids so write-back and short-id lookup work, add BlueBubbles user docs for the new channel/account setting, then land only after the BlueBubbles targeted tests and changed gate pass.

## Reproduction Assessment

Yes. For the user-visible gap, reproduce on current main with an inbound BlueBubbles reply or tapback that has `replyToId` but no `replyToBody`/`replyToSender` after a cache miss; current main only checks RAM and never fetches the original message. For the PR bug, use `replyToId: "p:0/msg-1"`: the helper fetches/caches `msg-1`, while the monitor later looks up `p:0/msg-1`.

## Solution Assessment

No, not as currently patched. The default-off fallback through the typed client is the right direction, but the PR should first preserve cache/short-id behavior for prefixed reply ids and document the new operator-facing config; a safe repair is to return or share the canonical reply id with the monitor, or cache under both raw and normalized ids.
