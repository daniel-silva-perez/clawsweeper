## Summary

Keep this PR open. Current main still has cache-only BlueBubbles reply-context recovery: it normalizes webhook reply metadata and consults the process-local reply cache, but it does not expose `replyContextApiFallback`, add the proposed helper/tests, or fetch the quoted message from the BlueBubbles API after a cache miss. The PR remains a plausible implementation candidate, with author follow-up addressing the earlier SSRF, account inheritance, dead-code, and changelog concerns; maintainers should now choose whether this PR or overlapping #73241 is the canonical path.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land this BlueBubbles-owned opt-in fallback after CI and final diff-scope review, or explicitly pick #73241 as the canonical competing implementation. The accepted implementation should remain behind channel/account config, preserve account inheritance, route through the existing BlueBubbles SSRF policy/client path, and keep regression coverage for cache misses, failures, dedupe, and SSRF modes.
