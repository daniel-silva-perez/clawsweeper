## Summary

Keep open. Current main still only resolves BlueBubbles inbound reply context from the in-memory cache, while this PR adds a useful opt-in API fallback and has already incorporated the prior SSRF, account-inheritance, and dead-code review fixes; the remaining cleanup is documentation plus rebase/validation because GitHub currently reports the branch as dirty.

## What This Changes

The PR adds an opt-in BlueBubbles reply-context API fallback with config/types, a helper and tests, a changelog entry, and runtime-error logging for attachment download failures.

## Best Possible Solution

Land the BlueBubbles-owned fallback after adding user docs, rebasing cleanly, and rerunning the BlueBubbles changed gate, while keeping the behavior opt-in, best-effort, cache-populating, and routed through the typed client. Treat any default-on decision as a separate maintainer/product follow-up.

## Reproduction Assessment

Yes. A high-confidence reproduction path is an inbound BlueBubbles reply with replyToId but missing replyToBody/replyToSender after a gateway restart, multi-instance handoff, or reply-cache TTL eviction; current main checks only the in-memory cache and then continues without fetching the original message.

## Solution Assessment

Yes, with a docs/rebase follow-up. The PR keeps the fix extension-owned and default-off, uses the existing typed BlueBubbles client/SSRF policy, validates the reply id, dedupes concurrent fetches, and writes back to the reply cache; documenting the knob is the safer remaining repair than changing the runtime shape again.
