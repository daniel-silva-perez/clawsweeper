## Summary

Keep open. The direct TUI string-content gap is still present on current main, but this PR should not merge as-is because it applies tagged-reasoning parsing to generic string blocks that also render user and command transcript text, and its new extractor does not mirror the existing nested-tag depth contract.

## What This Changes

The PR adds a shared tagged-reasoning extractor, changes TUI string formatting to split inline `<think>...</think>` content from visible text, and adds formatter regression tests for assistant string payloads.

## Best Possible Solution

Revise this into a current-main, assistant-only TUI hardening patch: split assistant string content into reasoning and visible text, preserve literal tags in user and command transcript text, keep nested and malformed tag behavior aligned with the shared stripper, and cover formatter, live stream, and history reload paths.

## Reproduction Assessment

Yes. A focused unit path is enough: an assistant message with string content like `<think>hidden</think>Visible` reaches current TUI helpers with empty extracted thinking and unstripped visible string content.

## Solution Assessment

No, not as written. The direction is right, but the safer solution is a role-aware helper or assistant-only branch that reuses the existing reasoning-tag contract instead of changing the generic string block path for all transcript roles.
