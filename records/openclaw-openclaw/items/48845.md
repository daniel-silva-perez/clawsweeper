## Summary

Keep this PR open. Current main still lacks the missing-`chat_type` fallback and still uses the old UUID-based Feishu reaction synthetic event path, so the PR is not obsolete; however, the PR needs maintainer review or a refreshed patch because its deterministic reaction IDs, timeout increase, and topic metadata handling have code-backed functional risks.

## Best Possible Solution

Keep this PR open and ask for a refreshed Feishu patch: infer chat type from known Feishu `chat_id` shapes when `chat_type` is missing, keep reaction synthetic IDs event-unique and dedupe-safe, carry the original message ID separately for reply/thread anchoring, preserve topic metadata, and add regression tests for missing `chat_type`, p2p/private classification, created/deleted/repeated reactions, and topic-session routing.

## Risks / Open Questions

- Merging the PR as provided may still drop repeated same-user/same-emoji lifecycle events because the deterministic synthetic ID lacks an event-unique token such as `action_time`, `reaction_id`, or a UUID.
- The 10-second reaction verification timeout may interact badly with webhook callback timing and redelivery unless the handler/dedupe flow is adjusted deliberately.
- The chat-type fallback needs coverage for group, p2p/private, and `oc_dm` shapes so direct-message traffic is not misclassified as group traffic.
- The reaction reply-anchor change should preserve original `root_id`/`thread_id` or equivalent metadata so topic-session history is not forked.
