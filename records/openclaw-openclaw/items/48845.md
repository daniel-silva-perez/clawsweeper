## Summary

Keep this PR open. Current main still lacks the missing-`chat_type` fallback described by the PR, and Feishu reaction synthetic events still use the older UUID-stamped ID path. The branch is a plausible Feishu bug-fix candidate, but it is not obsolete or already implemented on main, and the current PR diff still needs maintainer review for dedupe, timeout, and topic-session behavior.

## Best Possible Solution

Keep this PR open and ask for a refreshed Feishu patch: infer chat type from known Feishu `chat_id` shapes when `chat_type` is missing, keep reaction synthetic IDs event-unique and dedupe-safe, carry the original message ID separately for reply/thread anchoring, preserve topic metadata, and add regression tests for missing `chat_type`, p2p/private classification, created/deleted/repeated reactions, and topic-session routing.

## Risks / Open Questions

- Merging the PR as provided may still drop repeated same-user/same-emoji lifecycle events because the deterministic synthetic ID lacks an event-unique token such as `action_time`, `reaction_id`, or a UUID.
- The 10-second reaction verification timeout may interact with webhook response timing because webhook mode waits for `eventDispatcher.invoke` before returning the 200 response.
- The PR should preserve original `root_id`/`thread_id` or equivalent metadata so topic-session routing and reply anchoring do not diverge.
- The missing-chat-type fallback needs coverage for group, p2p/private, and `oc_dm` shapes so direct-message traffic is not misclassified as group traffic.
