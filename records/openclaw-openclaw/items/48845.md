## Summary

Keep this PR open. Current main still lacks the PR’s missing-`chat_type` fallback and still uses the older UUID-based Feishu reaction synthetic event path, so the work is not obsolete or implemented on main. The PR should remain a maintainer review candidate, but it needs a refreshed patch or explicit maintainer acceptance for the dedupe, timeout, and topic-routing concerns already raised in review.

## Best Possible Solution

Keep this PR open and ask for a refreshed Feishu patch: infer chat type from known Feishu `chat_id` shapes when `chat_type` is missing, keep reaction synthetic IDs event-unique and dedupe-safe, carry the original message ID separately for reply/thread anchoring, preserve topic metadata, and add regression tests for missing `chat_type`, p2p/private classification, created/deleted/repeated reactions, and topic-session routing.

## Risks / Open Questions

- Merging the PR as provided may still drop repeated same-user/same-emoji lifecycle events because the deterministic synthetic ID shown in the PR lacks an event-unique token such as `action_time`, `reaction_id`, or a UUID.
- The 10-second reaction verification timeout may interact with Feishu webhook retry timing because webhook mode waits for `eventDispatcher.invoke` before returning the 200 response.
- The PR should preserve original `root_id`/`thread_id` or equivalent metadata so topic-session routing and reply anchoring do not diverge.
- The missing-chat-type fallback needs regression coverage for group, p2p/private, and `oc_dm` shapes so direct-message traffic is not misclassified as group traffic.
