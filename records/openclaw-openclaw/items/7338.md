## Summary

Keep this open. Current main still lacks opt-in, domain-scoped OpenClaw agent attestation for arbitrary outbound HTTP requests, and the existing provider-attribution work is deliberately limited to verified model-provider transports rather than general agent-origin proof.

## What This Changes

Request opt-in OpenClaw agent attestation headers or signed JWTs for outgoing HTTP requests, optionally scoped per skill and allowlisted domain so third-party services can distinguish agent traffic from direct human API calls.

## Best Possible Solution

Define the trust model and exact outbound-request scope first, then add an opt-in, domain-scoped config/API surface with privacy-preserving defaults, clear static-versus-cryptographic semantics, docs, and tests proving allowlisted injection plus non-leakage.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because implementation would assert OpenClaw agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how services obtain verification keys, how replay is prevented, and what issuer/audience semantics mean for OpenClaw agents.
  - confidence: 0.91
- **[medium] Prevent cross-domain identity leakage:**
  - body: Attestation must be opt-in and allowlisted so agent IDs, versions, skill context, or signed tokens are not attached to arbitrary external requests or redirects.
  - confidence: 0.9

## Risks / Open Questions

- The attestation trust model remains unresolved: signing authority, public-key distribution, replay windows, issuer/audience semantics, and whether OpenClaw should make a verifiable agent-origin claim need explicit security review.
- The outbound scope is broad across web_fetch, skills, plugin tools, browser automation, model-provider transports, provider fallbacks, and redirects; a narrow header patch could create inconsistent semantics or leak identity metadata.
- Stable agent identifiers, skill context, or signed tokens could enable cross-service tracking if attestation is not opt-in, domain-scoped, and carefully minimized.
