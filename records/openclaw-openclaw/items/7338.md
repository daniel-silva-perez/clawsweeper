## Summary

Keep this open. Current main still lacks a general, opt-in, domain-scoped outbound agent attestation surface, and the existing provider-attribution work is narrower vendor transport attribution rather than proof that arbitrary HTTP requests came from an OpenClaw agent. The issue discussion already treats this as unresolved product/security design work.

## What This Changes

The issue requests optional OpenClaw agent attestation headers or signed attestations for outbound external HTTP requests, scoped by user or skill configuration.

## Best Possible Solution

Define the identity and trust model first, then implement a privacy-preserving opt-in config/API surface with strict domain allowlisting, explicit static-versus-cryptographic semantics, public docs, and regression tests proving both injection and non-leakage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. We do have a high-confidence current-main verification path for the missing feature by inspecting web_fetch, config/schema, skill metadata, docs, and provider-attribution tests.

## Solution Assessment

Unclear until maintainers settle the design. Static headers alone would not be the best solution for agent-only services; a maintainable path needs a scoped trust model, explicit user consent, redirect/non-allowlist protections, and tests before implementation.
