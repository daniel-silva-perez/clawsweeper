## Summary

Current main still does not implement opt-in, domain-scoped OpenClaw agent attestation for arbitrary external HTTP requests. The web_fetch path sends ordinary fetch headers only, config/schema/skill metadata have no attestation surface, and existing provider attribution headers are scoped to model-provider transports rather than agent-origin requests to external services. Keep this open as the canonical product/security API design tracker, especially given the recent maintainer review already kept it open and noted #11137 as duplicate/superseded context.

## Best Possible Solution

Keep this open as the canonical product/security design tracker. The next step should be maintainer design review that defines the trust model and scope, then an opt-in domain-scoped config/API surface, static versus cryptographic semantics, consistent outbound-path wiring, privacy documentation, and tests for allowed-domain injection plus non-leakage to other destinations.

## Risks / Open Questions

- A static identifying header would be easy to spoof and could add cross-service tracking risk unless the privacy defaults, domain allowlist, and user-visible controls are designed first.
- A cryptographic attestation design needs explicit key ownership, audience binding, expiry, revocation, replay resistance, and verifier documentation before implementation.
- Outbound scope is broader than web_fetch: media/PDF helpers, provider-owned transports, plugin tools, MCP, and skill-run processes may use different HTTP paths, so partial wiring could give external services inconsistent or misleading signals.
- Reusing provider attribution machinery for arbitrary external requests could leak provider-specific hidden headers or conflate model-provider endpoint trust with user-approved agent attestation.
