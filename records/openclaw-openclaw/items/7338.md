## Summary

Keep this open. Current main still lacks opt-in, domain-scoped agent attestation for arbitrary outbound HTTP requests, and the existing attribution work is intentionally limited to verified model-provider transports. The issue also already has maintainer context keeping it open for product/security design rather than implementation cleanup.

## What This Changes

The issue requests opt-in OpenClaw agent attestation headers or signed tokens for outbound HTTP requests so third-party services can distinguish agent-origin traffic from direct human API calls.

## Best Possible Solution

Define the trust model and exact outbound-request scope first, then add an opt-in, domain-scoped config/API surface with privacy-preserving defaults, clear static-versus-cryptographic semantics, docs, and tests proving allowlisted injection plus non-leakage.

## Security Review

Status: needs_attention

Summary: This is security-sensitive because any implementation would assert agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how services obtain verification keys, how replay is prevented, and what issuer/audience semantics mean for OpenClaw agents.
  - confidence: 0.91
- **[medium] Prevent cross-domain identity leakage:**
  - body: Attestation must be opt-in and allowlisted so agent IDs, versions, skill context, or signed tokens are not attached to arbitrary external requests or redirects.
  - confidence: 0.9
