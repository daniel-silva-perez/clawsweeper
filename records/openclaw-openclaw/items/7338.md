## Summary

Current main still does not implement opt-in, domain-scoped OpenClaw agent attestation for external HTTP requests. The existing web_fetch path sends ordinary fetch headers only, the config/schema/docs/skill metadata have no attestation surface, and provider attribution headers are scoped to model-provider transports rather than arbitrary agent-origin HTTP requests. Prior context also keeps this as the canonical tracker after the related sender-identity request #11137 was closed as duplicate/superseded.

## Best Possible Solution

Keep this open as the canonical product/security API design tracker. A complete solution should decide the trust model first, then add an opt-in domain-scoped config/API surface, define static versus cryptographic semantics, wire the chosen outbound HTTP paths consistently, document privacy defaults, and add tests proving allowed-domain injection plus non-leakage to other destinations.

## Risks / Open Questions

- A naive default header implementation could leak stable agent, workspace, user, or skill identity to unrelated domains; opt-in behavior and domain allowlisting are central privacy requirements.
- Cryptographic attestation needs an explicit trust model: issuer identity, key management, public-key distribution, audience scoping, expiry, replay protection, and rotation.
- The intended scope is ambiguous across web_fetch, provider traffic, skill/plugin-owned fetches, browser automation, and arbitrary tool calls; each path may need different privacy and implementation handling.
