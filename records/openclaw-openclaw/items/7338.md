## Summary

Current main still does not implement opt-in, domain-scoped OpenClaw agent attestation for arbitrary external HTTP requests. The web_fetch path sends ordinary request headers only, config/schema/skill metadata have no attestation surface, and existing provider attribution is scoped to model-provider transports rather than external agent-origin requests. Keep this open as the canonical product/security design tracker.

## What This Changes

Request to add opt-in OpenClaw agent attestation headers, preferably cryptographic and domain-scoped, to outgoing external HTTP requests with config and skill metadata controls.

## Best Possible Solution

Keep this open as the canonical product/security design tracker. The next step should be maintainer design review that defines the trust model and outbound paths in scope, then an opt-in domain-scoped config/API surface, static versus cryptographic semantics, privacy documentation, and tests for allowlisted injection plus non-leakage.

## Risks / Open Questions

- Static identity headers would be spoofable and could create cross-service tracking unless opt-in controls, domain allowlists, and privacy defaults are designed first.
- Cryptographic attestation needs explicit key ownership, audience binding, expiry, replay resistance, revocation, verifier documentation, and rotation semantics.
- Outbound scope is broader than web_fetch; plugin tools, MCP, provider transports, media/PDF helpers, and skill-run processes may use different HTTP paths, so partial wiring could give external services inconsistent identity signals.
- Reusing provider attribution machinery for arbitrary external requests could conflate vendor-specific model-provider trust with user-approved external-service attestation.
