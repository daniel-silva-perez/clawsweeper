## Summary

Keep open. Current main still lacks a general, opt-in, domain-scoped OpenClaw agent attestation surface for arbitrary outbound HTTP requests; the existing attribution code is intentionally limited to selected model-provider transports, and the issue discussion already treats this as product/security design work rather than a completed implementation.

## What This Changes

The issue requests opt-in OpenClaw agent attestation headers or signed attestations for outbound HTTP requests so agent-only services can distinguish OpenClaw agent traffic from ordinary clients.

## Best Possible Solution

Define the identity/trust model and covered outbound paths first, then implement a privacy-preserving opt-in config/API surface with strict domain allowlisting, clear static-versus-cryptographic semantics, public docs, and regression tests proving both authorized injection and non-leakage.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because implementation would assert OpenClaw agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how services obtain verification keys, how replay is prevented, and what issuer/audience semantics mean for OpenClaw agents.
  - confidence: 0.92
- **[medium] Prevent cross-domain identity leakage:** `src/agents/tools/web-fetch.ts:425`
  - body: Attestation must be opt-in and allowlisted so agent IDs, versions, skill context, or signed tokens are not attached to arbitrary external requests, fallback providers, custom proxy routes, or redirects.
