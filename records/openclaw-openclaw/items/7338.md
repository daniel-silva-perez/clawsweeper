## Summary

Keep open. Current main still lacks a shared, opt-in, domain-scoped OpenClaw agent attestation contract for external HTTP requests. The adjacent provider-header work is model-provider scoped, and the linked before_llm_request PR is closed unmerged, so this remains a product/security API design item rather than something already implemented or superseded.

## Best Possible Solution

Keep this open as the canonical product/API design tracker. A complete solution should decide whether attestation belongs in core or behind a new plugin/API seam, define opt-in domain-scoped config, specify static versus cryptographic semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests for allowed-domain injection plus non-leakage to other destinations.

## Risks / Open Questions

- A naive default header implementation could leak stable agent or user identity across unrelated services; opt-in, domain allowlisting, and minimal claims are central privacy requirements.
- Cryptographic attestation needs explicit issuer, key management, audience, expiry, replay protection, verifier distribution, and rotation design before implementation.
- Adjacent model-provider attribution/header work can be mistaken for a fix, but it does not prove OpenClaw agent origin for general external HTTP requests.
