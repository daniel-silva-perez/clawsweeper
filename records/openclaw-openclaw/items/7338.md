## Summary

Keep open: current main still has no general, opt-in, domain-scoped outbound agent attestation surface. The nearby attribution code is provider-specific, and the issue discussion already treats this as unresolved product/security design work.

## What This Changes

Request to add user- or skill-configurable OpenClaw agent attestation headers or signed JWTs to outbound HTTP requests, scoped by domain with privacy-preserving defaults.

## Best Possible Solution

Define the identity model and exact outbound paths first, then implement a privacy-preserving opt-in config/API surface with strict domain allowlisting, explicit static-versus-cryptographic semantics, public docs, and regression tests for injection and non-leakage.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it would assert OpenClaw agent identity to third-party services and could expose stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how services obtain verification keys, how replay is prevented, and what issuer/audience semantics mean for OpenClaw agents.
  - confidence: 0.92
- **[medium] Prevent cross-domain identity leakage:** `src/agents/tools/web-fetch.ts:425`
  - body: Any attestation must be opt-in and allowlisted so agent IDs, versions, skill context, or signed tokens are not attached to arbitrary external requests, redirects, fallback providers, or custom proxy routes.
  - confidence: 0.91
