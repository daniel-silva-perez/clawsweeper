## Summary

Keep this open. Current main still lacks a general, opt-in, domain-scoped agent attestation mechanism for arbitrary external HTTP requests, and the existing OpenAI/OpenRouter attribution work is narrower provider-transport metadata rather than agent-origin proof. The prior maintainer/ClawSweeper comments also treat this as unresolved product and security design work.

## What This Changes

Request an opt-in OpenClaw agent attestation surface for outgoing HTTP requests, with possible static headers, signed JWT attestations, and domain or skill scoping.

## Best Possible Solution

Define the identity/trust model and exact outbound paths first, then implement a privacy-preserving opt-in config/API surface with strict domain allowlisting, static-versus-cryptographic semantics, public docs, and regression tests proving both attestation injection and non-leakage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence current-main verification path: inspect web_fetch, fetch config/schema, skill metadata, docs, and provider-attribution tests to confirm the requested surface is absent.

## Solution Assessment

Unclear until maintainers settle the design. Static headers alone are not the best solution for agent-only services; the maintainable path needs scoped consent, signing/replay semantics, redirect protections, and non-leakage tests.

## Security Review

Status: needs_attention

Summary: Security review needs attention because this feature would assert OpenClaw agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how third-party services obtain verification keys, how replay is prevented, and what issuer/audience claims mean for OpenClaw agents.
  - confidence: 0.93

  - body: Any attestation must be opt-in and allowlisted so agent ids, versions, skill context, or signed tokens are not attached to arbitrary external requests, redirects, custom proxies, or fallback providers.
