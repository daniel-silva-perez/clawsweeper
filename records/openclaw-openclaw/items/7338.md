## Summary

Keep open. Current main still lacks a general, opt-in, domain-scoped agent attestation mechanism for external HTTP requests, and the existing provider attribution code is deliberately scoped to known model-provider transports rather than proving agent origin to arbitrary third-party services.

## What This Changes

Request to add optional OpenClaw agent attestation headers or signed attestations to outgoing external HTTP requests, with user or skill-level opt-in and domain allowlisting.

## Best Possible Solution

Define the identity and trust model first, then implement a privacy-preserving opt-in config/API surface with strict domain allowlisting, signing/replay semantics, redirect protections, docs, and tests proving both allowlisted injection and non-leakage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence current-main verification path: inspect web_fetch headers, fetch config/schema, skill metadata, docs, and provider-attribution tests to confirm the requested surface is absent.

## Solution Assessment

Unclear until maintainers settle the design. Static headers alone are not the best solution for agent-only services; the maintainable path needs scoped consent, cryptographic semantics or explicit non-cryptographic limits, redirect protections, and non-leakage tests.
