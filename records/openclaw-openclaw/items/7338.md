## Summary

Keep open. Current main still lacks a general, opt-in, domain-scoped agent attestation mechanism for outbound external HTTP requests, and the existing provider attribution path is narrower model-provider transport attribution. The existing maintainer discussion also treats this as product/security design work, not a stale cleanup item.

## What This Changes

Request to add optional OpenClaw agent attestation headers or signed attestations to outgoing external HTTP requests, with user/skill configuration and domain allowlisting.

## Best Possible Solution

Define the identity and trust model first, then implement a privacy-preserving opt-in config/API surface with strict domain allowlisting, signing or explicitly non-cryptographic semantics, redirect protections, docs, and tests for allowlisted injection and non-leakage.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence current-main verification path: inspect web_fetch headers, runtime config/schema, skill metadata, docs, and provider-attribution tests/docs to confirm the requested attestation surface is absent.

## Solution Assessment

Unclear until maintainers settle the design. Static headers alone are not the best solution for agent-only services; the maintainable path needs scoped consent, cryptographic or explicitly limited semantics, redirect protections, and non-leakage tests.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it would assert OpenClaw agent identity to third-party services and could leak stable identifiers if scoped incorrectly.
