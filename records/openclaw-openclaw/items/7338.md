## Summary

Keep this open. Current main still does not provide opt-in, domain-scoped OpenClaw agent attestation for arbitrary outbound HTTP requests; the existing attribution code is limited to verified model-provider transports, and the discussion already treats this as product/security design work rather than a cleanup item.

## What This Changes

Request optional static or cryptographic OpenClaw agent attestation headers for external requests, with user/skill opt-in, domain allowlisting, privacy constraints, and verification semantics.

## Best Possible Solution

Define the trust model and exact outbound-request scope first, then add an opt-in, domain-scoped config/API surface with privacy-preserving defaults, clear static-versus-cryptographic semantics, docs, and tests proving allowlisted injection plus non-leakage.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because implementation would assert agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how services obtain verification keys, how replay is prevented, and what issuer/audience semantics mean for OpenClaw agents.
  - confidence: 0.91
- **[medium] Prevent cross-domain identity leakage:**
  - body: Attestation must be opt-in and allowlisted so agent IDs, versions, skill context, or signed tokens are not attached to arbitrary external requests or redirects.
  - confidence: 0.9
