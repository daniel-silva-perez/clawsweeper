## Summary

Current main still does not provide opt-in, domain-scoped agent attestation headers for external HTTP requests. The observable implementation remains ordinary web_fetch headers plus separate model-provider request/header plumbing; there is no shared attestation config, signed-token contract, skill metadata propagation, or non-leakage coverage. The prior maintainer/Codex comment correctly kept this open, and the related sender-identity request was closed as a duplicate of this issue rather than as a shipped fix.

## Best Possible Solution

Keep this open as the canonical product/security API design tracker. A complete solution should decide whether attestation belongs in core or behind a new plugin/API seam, define opt-in domain-scoped configuration, specify static versus cryptographic semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests for allowed-domain injection plus non-leakage to other destinations.

## Risks / Open Questions

- A naive default header implementation could leak stable agent, user, or workspace identity across unrelated services; opt-in behavior, domain allowlisting, and minimal claims are central privacy requirements.
- Cryptographic attestation requires explicit issuer, key management, verifier distribution, audience scoping, expiry, replay protection, and rotation design before implementation.
- Existing model-provider header and attribution work can be mistaken for a fix, but it covers provider traffic and does not attest OpenClaw-origin requests across external HTTP paths.
