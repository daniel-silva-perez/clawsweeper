## Summary

Keep open. Current main still matches the reported failure path: Memory Wiki bridge/status reads public artifacts through a plugin SDK import, while memory-core registers publicArtifacts through the host registry into module-local memory state. The prior Gateway-routing fix addresses the CLI/local process split but does not prove the managed plugin-runtime-deps mirror shares the same memory-state singleton.

## What This Changes

The issue reports that Memory Wiki bridge imports still return zero artifacts because memory-core and memory-wiki can resolve different memory-state module singletons across the installed runtime and the managed plugin-runtime-deps mirror.

## Best Possible Solution

Route Memory Wiki bridge artifact reads through a canonical host-provided memory capability seam, or make memory capability state process-global across mirrored dist chunks, then add packaged external-runtime-mirror coverage proving wiki.bridge.import sees artifacts registered by memory-core.

## Reproduction Assessment

Yes. The issue provides concrete packaged Gateway steps and instrumentation, and current main still has the same source shape: host registration writes canonical memory state while the bridge can read through a mirrored SDK/memory-state copy.

## Solution Assessment

Unclear which implementation is best, but leaving the bridge on a direct SDK singleton read is not the safest boundary. A canonical host-injected accessor or process-global memory state is narrower than generated mirror rewrites or broad externalization of all core modules.
