## Summary

Keep open. Current main still has Memory Wiki bridge/status artifact reads going through module-local memory capability state, and the managed runtime mirror boundary still makes the reported split-singleton failure plausible. The same author has draft PR #75064 open for this issue, so this should remain open while that paired implementation is reviewed and validated.

## What This Changes

The issue reports that Memory Wiki bridge/status return zero memory artifacts because memory-core registers public artifacts into one memory-state module instance while mirrored plugin-runtime-deps code reads a separate empty memory-state instance.

## Best Possible Solution

Land or replace the linked fix with a narrow canonical memory capability registry that is visible across duplicated runtime module instances, plus regression coverage for Memory Wiki bridge import and status under mirrored runtime loading.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and instrumentation, a second user confirms the same OpenClaw 2026.4.27 behavior, and current main still has the module-instance-sensitive bridge/status read path behind the reported runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. The maintainable direction is a narrow shared or host-owned memory capability seam with mirror regression tests, not patching generated plugin-runtime-deps files.
