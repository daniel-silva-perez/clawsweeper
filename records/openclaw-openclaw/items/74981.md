## Summary

Keep open. Current main still has Memory Wiki bridge and status artifact reads going through the plugin SDK memory-host-core singleton, while Memory Core registers publicArtifacts through the host registry into canonical memory state. The packaged runtime-deps mirror still remaps bundled plugin execution into a copied dist tree, so the reported split singleton remains a plausible and source-supported runtime-boundary bug rather than a duplicate of the earlier CLI-to-Gateway routing fix.

## What This Changes

Memory Wiki bridge import/status should see public artifacts registered by the active Memory Core plugin even when bundled plugins run from the managed plugin-runtime-deps mirror.

## Best Possible Solution

Route Memory Wiki bridge artifact reads through a canonical host-provided memory capability seam, or otherwise make the memory capability state shared across runtime-deps mirrored dist chunks, then add packaged external-runtime-mirror coverage proving wiki.bridge.import sees artifacts registered by memory-core. Keep the broader dependency-retention work in #74777 separate unless the same targeted change naturally covers both paths.

## Reproduction Assessment

Yes. The issue provides concrete packaged Gateway steps and instrumentation, and current main still has the bridge/status read path and memory-core registration path on separate module-instance-sensitive surfaces. A full live reproduction would require a packaged runtime-deps install, but the source path matches the reported failure.

## Solution Assessment

Unclear which implementation is best, but the current direct SDK singleton read from memory-wiki is not a robust boundary for mirrored packaged runtimes. A canonical host-injected accessor is narrower and safer than patching generated mirror files or broad externalization of all core modules.
