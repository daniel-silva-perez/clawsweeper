## Summary

Keep open. Current main still has a source-supported path where Memory Wiki bridge/status reads module-local memory capability state while packaged bundled plugin loading can execute from mirrored dist roots; #72615's CLI Gateway routing is present but does not remove the Gateway-side singleton split. A same-author draft PR #75064 is already open for this narrow fix, so the issue should remain open pending review and validation rather than be queued or closed.

## What This Changes

Make Memory Wiki bridge/status observe the public artifacts registered by memory-core when bundled plugins are loaded through managed plugin-runtime-deps mirrors, avoiding zero artifact imports from a split memory-state singleton.

## Best Possible Solution

Land a reviewed fix that makes memory capability state canonical across duplicated runtime module instances, with regression tests for duplicate memory-state imports plus Memory Wiki bridge import/status under mirrored runtime loading.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps on v2026.4.27 and a second same-version confirmation, and current main still has module-local memory state with bridge/status reads across the mirrored-runtime boundary.

## Solution Assessment

Unclear until #75064 is reviewed. The right direction is a narrow shared or host-owned memory capability registry; patching generated plugin-runtime-deps files would be brittle.
