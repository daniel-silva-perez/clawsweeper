## Summary

Keep open. Current main still keeps memory capability state in a module-local singleton and Memory Wiki bridge/status still read artifacts through that singleton, while bundled plugin runtime mirroring remains in the implicated path. The issue is also paired with open draft PR #75064 from the same reporter, so cleanup should wait for that implementation to be reviewed or replaced.

## What This Changes

Memory Wiki bridge import/status should see public artifacts registered by memory-core across packaged plugin-runtime-deps runtime mirrors instead of returning zero artifacts.

## Best Possible Solution

Review and either land or replace #75064 with a narrow shared or host-owned memory capability registry that all mirrored runtime module instances observe, while preserving the existing plugin SDK API and adding regression coverage for Memory Wiki bridge import/status across duplicate module instances.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and instrumentation, a second user confirms the same OpenClaw 2026.4.27 behavior, and current main still has the module-instance-sensitive bridge/status read path behind the runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. The maintainable direction is a canonical shared or host-owned memory capability registry with mirrored-runtime regression tests, not patching generated plugin-runtime-deps files.
