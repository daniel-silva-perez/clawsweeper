## Summary

Keep open. Current main still has Memory Wiki bridge/status artifact reads go through the plugin SDK import inside `memory-wiki`, while `memory-core` registers `publicArtifacts` through the host API into canonical memory state. The packaged runtime mirror still creates its own `openclaw/plugin-sdk` alias backed by the mirrored dist tree, so the reported split `memoryPluginState` path remains plausible and is not resolved by the earlier Gateway-routing fix.

## What This Changes

The issue reports that Memory Wiki bridge imports still return zero artifacts because `memory-core` and `memory-wiki` can resolve different `memory-state` module singletons across the installed runtime and the managed `plugin-runtime-deps` mirror.

## Best Possible Solution

Route Memory Wiki bridge artifact reads through a canonical host-provided memory capability seam, or make the memory capability state process-global across mirrored dist chunks, then add packaged external-runtime-mirror coverage that proves `wiki.bridge.import` sees artifacts registered by `memory-core`. Keep the broader runtime dependency retention work in #74777 separate unless the same patch touches both paths.

## Reproduction Assessment

Yes. The issue provides concrete packaged Gateway steps and instrumentation, and the current source path matches that diagnosis: host registration writes canonical memory state while the bridge can read from a mirrored SDK/memory-state copy.

## Solution Assessment

Unclear which implementation is best, but the current direct SDK singleton read from `memory-wiki` is not the safest fix boundary. A canonical host-injected accessor or process-global memory state is narrower than fragile generated mirror rewrites or broad externalization of all core modules.
