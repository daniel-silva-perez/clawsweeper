## Summary

Keep open. Current main still keeps memory capability state in a module-local object, and Memory Wiki bridge/status still read public artifacts through that module-local path while bundled runtime mirroring can load duplicated dist modules. The same author has open draft PR #75064 targeting this exact split-singleton bug, so this should stay open for PR review rather than be closed or queued as a parallel repair.

## What This Changes

Memory Wiki bridge should see memory-core publicArtifacts across plugin-runtime-deps runtime mirrors instead of returning zero artifacts from a split memory-state singleton.

## Best Possible Solution

Use a canonical shared or host-owned memory capability registry that all canonical dist and plugin-runtime-deps mirror module instances observe, keep the existing plugin SDK API source-compatible, and add regression tests for duplicate module instances plus Memory Wiki bridge/status artifact counts.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and instrumentation, a second user confirms the same release/install symptoms, and current main still has the module-instance-sensitive bridge/status read path behind the runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. The maintainable direction is a narrow shared or host-owned memory capability seam with mirrored-runtime regression coverage, not patching generated plugin-runtime-deps files.
