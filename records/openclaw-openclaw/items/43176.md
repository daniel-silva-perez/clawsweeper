## Summary

Keep open. Current main still does not implement this PR’s automatic 20s cap for broad `find $HOME` or `find /` exec scans, and the matching prompt/docs guidance is absent. The PR remains a plausible focused fix, but it touches a security-sensitive shell execution timeout path and the discussion identifies parser gaps that still need maintainer review before merge.

## Best Possible Solution

Keep this PR open and request changes before merge. The best path is a shared broad-`find` timeout classifier applied consistently before local exec, gateway approval follow-up execution, and node `system.run`; it should parse all `find` start paths, resolve relative roots against the command workdir, support `~`, `$HOME`, and `${HOME}`, preserve explicit timeout intent, honor only real traversal-limiting escape hatches, and include focused regression tests plus prompt/docs guidance.

## Risks / Open Questions

- Current main can still allow broad `$HOME` or `/` discovery scans without an explicit timeout to consume the configured default exec timeout.
- Merging this PR as-is could leave bypassable or inconsistent behavior for multi-root `find` commands and `${HOME}` forms in a shell execution path.
- The final implementation needs consistent application across local exec, gateway approval follow-up execution, and node `system.run`, otherwise one host path can still retain the old behavior.
