## Summary

Keep open. Current main still does not implement this PR's automatic 20s cap for broad `find $HOME` or `find /` exec scans, and the proposed prompt guidance is also absent. The PR remains a plausible focused fix, but it needs maintainer review/changes because existing review comments identify parser gaps in a shell execution timeout classifier.

## Best Possible Solution

Keep this PR open and request changes before merge. The best path is a shared broad-`find` timeout classifier applied consistently before local exec, gateway approval follow-up execution, and node `system.run`; it should parse every `find` start path, resolve relative roots against the command `workdir`, support `~`, `$HOME`, and `${HOME}`, preserve explicit timeout intent, honor only real traversal-limiting escape hatches, and add focused regression tests plus prompt/docs guidance.

## Risks / Open Questions

- Current main can still let broad `$HOME` or `/` discovery scans consume the configured default exec timeout when no explicit timeout is provided.
- Merging the PR as-is could leave inconsistent or bypassable timeout behavior in a security-sensitive shell execution path, especially for multi-root `find` commands and `${HOME}` forms.
- The final implementation should stay consistent across local exec, gateway approval follow-up execution, and node `system.run` execution.
