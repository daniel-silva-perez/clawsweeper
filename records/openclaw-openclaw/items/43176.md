## Summary

Keep open. Current main still lacks the PR’s automatic 20s cap for broad `find $HOME` or `find /` exec scans and lacks the matching prompt guidance. The PR is still a plausible focused fix, but it touches the shell execution timeout path and should get maintainer changes before merge.

## Best Possible Solution

Keep this PR open and request changes. The best implementation is a shared broad-`find` timeout classifier applied before local exec, gateway approval follow-up execution, and node `system.run`; it should parse all `find` start paths, resolve relative roots against the command workdir, support `~`, `$HOME`, and `${HOME}`, preserve explicit timeout intent, honor only real traversal-limiting escape hatches, and include focused regression tests plus prompt/docs guidance.

## Risks / Open Questions

- Current main can still allow broad `$HOME` or `/` discovery scans without an explicit timeout to consume the configured default exec timeout.
- Merging this PR as-is could leave inconsistent behavior because the proposed cap is not applied to every exec host path, especially approval follow-up and node `system.run`.
- The PR touches a security-sensitive shell execution path; parser gaps such as multiple find start paths, `${HOME}` roots, and escape-hatch scope should be resolved before merge.
