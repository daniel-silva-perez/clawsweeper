## Summary

Keep this PR open. Current main still does not implement Feishu missed-message recovery on startup or WebSocket reconnect, and this PR is a focused implementation attempt for the linked Feishu gap, but the patch is not mergeable as-is because it is not wired into the monitor lifecycle and conflicts with current Feishu SDK/client/event shapes.

## Best Possible Solution

Keep this PR open for rework or supersede it with a safer Feishu-owned implementation. The implementation should hook startup/reconnect inside `monitorSingleAccount`/`monitorWebSocket`, use `createFeishuClient` plus current `client.im.chat.list` and `client.im.message.list` contracts, reconstruct events from current message-list item fields, dispatch through the existing dedupe/processing-claim/sequential queue/shared-history path, respect structured Feishu mention and group policy rules, document required scopes, and add focused regression tests plus live API proof where feasible.

## Risks / Open Questions

- Merging the PR as-is would not implement the requested behavior because the new module is never called from the Feishu monitor lifecycle.
- The patch likely fails against current main because `ResolvedFeishuAccount` has no `client` property and `hasRecordedMessage` is not exported from `dedup.ts`.
- The proposed chat discovery and message-list calls do not match the current Lark client shape used elsewhere in the Feishu plugin.
- If wired without redesign, polling group histories every 30 seconds with `content.includes("@")` could replay unrelated messages and produce unintended or privacy-sensitive replies.
- Recovered messages need to reuse the live monitor's persistent dedupe, processing claims, shared `chatHistories`, and sequential dispatch path to avoid duplicate replies, context loss, and ordering regressions.
