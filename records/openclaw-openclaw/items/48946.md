## Summary

Keep this PR open. Current main still does not implement Feishu missed-message recovery on startup or reconnect, and this PR is a focused attempt at that behavior, but the patch is not mergeable as-is due to lifecycle wiring, type/API, routing, context, and privacy risks.

## Best Possible Solution

Keep the PR open for rework or supersede it with a safer Feishu-owned implementation. The recovery path should live in the existing monitor lifecycle for startup/reconnect, use `createFeishuClient` with `client.im.chat.list` and `client.im.message.list`, reuse persistent dedupe and processing claims, dispatch through the shared `chatHistories` and sequential queue, respect group policy and structured Feishu mention data, document required scopes/config, and add focused regression tests plus live Feishu API proof where feasible.

## Risks / Open Questions

- Merging the PR as-is could fail compile due to the missing `account.client` property and missing `hasRecordedMessage` export.
- The proposed recovery loop could create unwanted or privacy-sensitive replies by scanning group history every 30 seconds and treating any `@` character as a bot mention.
- The patch does not share the live monitor's `chatHistories` or existing sequential processing path, so recovered messages could lose conversation context and ordering.
- No tests or builds were run because this was a read-only review; the decision is based on source, docs, PR diff, and provided discussion.
