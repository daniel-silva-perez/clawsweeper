## Summary

Keep this PR open. Current main still lacks Feishu missed-message recovery on startup or reconnect, and this PR is a focused implementation attempt, but the patch is not mergeable as-is due to missing lifecycle wiring, current API/type mismatches, context/order bypasses, and runtime privacy risk.

## Best Possible Solution

Keep this PR open for rework or supersede it with a safer Feishu-owned implementation. The recovery path should integrate with `monitorSingleAccount` and reconnect lifecycle, use `createFeishuClient` with `client.im.chat.list` and `client.im.message.list`, reconstruct events from current message-list item fields, reuse persistent dedupe/processing claims plus shared `chatHistories` and sequential dispatch, respect structured Feishu mention data and group policy, document required scopes/config, and add focused regression tests plus live API proof where feasible.

## Risks / Open Questions

- Merging the PR as-is would not implement the requested behavior because the new module is not wired into the Feishu monitor lifecycle.
- The patch likely fails against current main due to the missing `account.client` property and non-exported `hasRecordedMessage` import.
- The proposed polling loop could create unwanted or privacy-sensitive replies by scanning group history every 30 seconds and treating any `@` character as a bot mention.
- Recovered messages would bypass the live monitor's shared `chatHistories`, debounce, and sequential processing path, risking context loss and ordering regressions.
- Security review found no workflow, dependency, lockfile, package metadata, install, build, or release-script changes, but the runtime group-history polling behavior needs privacy and scope review before merge.
