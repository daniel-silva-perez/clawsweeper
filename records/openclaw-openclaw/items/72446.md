## Summary

Keep this PR open. It targets a valid Microsoft Foundry provider bug and the picker-side guardrail is directionally right, but the patch still lets skipped Claude deployments be persisted into the provider model catalog, leaving another path to select the same unsupported OpenAI-routed model later.

## What This Changes

The PR adds a Microsoft Foundry onboarding guardrail that detects Claude model names, hides Anthropic deployments from the deployment picker, shows custom-provider guidance, errors when only unsupported deployments are present, and adds detector tests.

## Best Possible Solution

Finish the provider-local guardrail by deriving one supported-deployment set and using it consistently for selection, persisted catalog generation, and regression coverage; leave full Anthropic Foundry runtime/onboarding support to a separate provider/API implementation such as the broader Azure Claude work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a discovered Foundry resource with both GPT and Claude deployments: the patched picker hides Claude, but the auth flow still persists every raw deployment into the Microsoft Foundry catalog.

## Solution Assessment

No. Option B is a reasonable narrow fix, but this implementation is incomplete because it filters only the prompt surface and leaves the persisted model-switching surface able to expose the same unsupported Claude deployment.
