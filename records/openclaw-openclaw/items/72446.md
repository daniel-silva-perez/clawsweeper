## Summary

Keep this PR open. The linked bug remains valid on current main, and the PR is a useful provider-local guardrail, but the patch is incomplete because it filters the onboarding prompt without filtering the discovered deployment catalog that is persisted into the Microsoft Foundry provider models.

## What This Changes

The PR adds Microsoft Foundry onboarding logic to detect Claude model names, skip Anthropic deployments in the deployment picker with custom-provider guidance, error when only unsupported deployments exist, and test the detector.

## Best Possible Solution

Finish the provider-owned guardrail in this PR or a narrow replacement: use one supported-deployment set for selection, persisted catalog generation, and regression tests, while leaving full Anthropic Foundry runtime support for a separate provider/API implementation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mixed Foundry deployment list: the PR would hide the Claude entry in selectFoundryDeployment, but auth.ts would still persist that Claude deployment from resourceDeployments into the Microsoft Foundry model catalog, where it would later route through /openai/v1.

## Solution Assessment

No. Option B is a reasonable narrow guardrail, but this implementation is not yet the best form because it only filters the prompt surface and leaves another selectable Microsoft Foundry model path for the same unsupported Claude deployment.
