## Summary

Keep this issue open. Current main still does not capture or mirror the current login-shell PATH when installing the macOS Gateway LaunchAgent; it builds a curated minimal service PATH and only merges safe absolute segments from an existing service environment. The issue discussion already identified this distinction and the current code/docs still match that analysis.

## Best Possible Solution

Keep the issue open and retitle/rescope it around macOS LaunchAgent service PATH policy. The likely fix belongs in src/daemon/service-env.ts and src/commands/daemon-install-helpers.ts, with focused tests for macOS install PATH generation and reinstall behavior. Maintainers should decide whether to import a sanitized login-shell PATH at install time or keep a curated minimal PATH while adding missing common dirs and documenting the override path.

## Evidence

- **Gateway service env still uses minimal PATH generation:** buildServiceEnvironment resolves shared service fields and sets serviceEnv.PATH from buildMinimalServicePath for non-Windows platforms; it does not call getShellPathFromLoginShell or probe /bin/zsh -l for install-time PATH.
