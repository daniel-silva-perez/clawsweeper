## Summary

Keep open. Current main still treats the launchd PATH as a curated service environment rather than the current login-shell PATH, and the discussion correctly reframes this as a service PATH policy issue rather than a stale `getShellPathFromLoginShell()` cache bug.

## What This Changes

The issue requests that `openclaw gateway install` write a macOS LaunchAgent PATH matching the current login-shell PATH instead of an outdated or heuristic PATH.

## Best Possible Solution

Keep the issue open until maintainers choose the macOS service PATH contract. A maintainable fix should either document and improve the curated minimal PATH, including missing common user-bin paths and stale-entry pruning, or add a sanitized login-shell PATH capture path with explicit tests for LaunchAgent install, doctor repair, and service audit behavior.

## Reproduction Assessment

Yes for the code-level mismatch: current main deterministically builds the LaunchAgent PATH from `buildMinimalServicePath()` and existing service metadata, not from `/bin/zsh -l -c 'echo $PATH'`. A live macOS LaunchAgent reinstall was not run in this read-only review.

## Solution Assessment

Unclear. Mirroring the full login-shell PATH may not be the narrowest or safest fix because the current code and docs intentionally use a minimal durable service PATH; maintainers should choose between a sanitized login-shell import and a better curated minimal-path policy before implementation.
