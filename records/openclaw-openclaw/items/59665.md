## Summary

Keep open. Current main still builds a curated minimal Gateway service PATH for macOS instead of capturing the current login-shell PATH, and the install path does not call the login-shell PATH helper. Recent service PATH hardening and audit behavior also show this needs an explicit maintainer decision before broadening the LaunchAgent environment.

## What This Changes

The issue requests that macOS Gateway LaunchAgent install capture the current login-shell PATH at install time so user tool directories are not missing from the generated service environment.

## Best Possible Solution

Define the macOS service PATH policy explicitly: either keep the deterministic minimal PATH and extend the allowlisted home-relative tool directories with tests/docs, or add a sanitized login-shell PATH import that preserves the existing security filters and is covered by install-plan and launchd tests.

## Reproduction Assessment

Yes, by source-level reproduction. On current main, `buildGatewayInstallPlan()` reaches `buildMinimalServicePath()` for macOS and ignores the current login-shell PATH except for specific env-derived roots and preserved prior service PATH entries.

## Solution Assessment

Unclear. Mirroring the full login-shell PATH is not obviously the best fix because service PATH now has explicit security filtering; a narrower allowlist or sanitized opt-in import may be safer after maintainer review.
