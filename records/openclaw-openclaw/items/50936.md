## Summary

Keep this PR open. Current main still emits `location.update` from the iOS node, but the gateway drops unknown node events and the heartbeat runner has no location-specific event prompt or reason classification. The PR remains a focused implementation candidate, with one security/trust-boundary issue still worth maintainer review before merge.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a hardened version of the gateway `location.update` handler that validates finite coordinates, sanitizes node-provided fields, enqueues the location event as untrusted prompt data, wakes the scoped heartbeat, embeds the event deliberately in the heartbeat prompt, updates public node/protocol docs, and adds targeted tests. If passive location updates are not intended, the alternative is to remove or disable the existing iOS producer so main no longer emits an event the gateway drops.

## Risks / Open Questions

- Merging the PR as shown may promote node-provided location text into trusted `System:` prompt context unless the handler passes `trusted:false` or otherwise wraps the data as untrusted.
- The PR should add focused coverage for valid/invalid payloads, finite coordinate checks, source sanitization, dedupe/wake behavior, heartbeat prompt inclusion, and provider attribution.
- Location data is sensitive; maintainers should keep exact coordinates and accuracy out of info-level release logs.
