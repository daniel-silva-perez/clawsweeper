## Summary

Keep this PR open. Current main still has the iOS node emitting `location.update`, but the gateway does not handle that event, heartbeat has no `location-update` reason or location prompt path, and the docs still describe only `location.get`. The PR remains a focused implementation candidate, not obsolete cleanup. The prior bot review was useful: the PR discussion identified prompt delivery and location-PII logging risks, and later PR commits appear intended to address those areas, but they still need maintainer review before merge.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for the iOS-to-gateway gap. The best path is to land a sanitized, tested `location.update` handler that queues an untrusted system event, wakes the correct heartbeat session, embeds the location event deliberately in the heartbeat prompt, and documents the node event contract; alternatively, maintainers should remove or disable the iOS passive location producer if passive updates are not intended to be supported.
