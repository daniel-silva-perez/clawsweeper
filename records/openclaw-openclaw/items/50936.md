## Summary

Keep this PR open. Current main still has the iOS producer sending `location.update`, but the gateway does not handle that node event, heartbeat has no location-specific wake/prompt path, and the docs still only describe the `location.get` command. The PR remains a focused implementation candidate rather than obsolete cleanup. The security pass found that the PR addressed the prior info-level GPS logging and source-string injection comments in part, but maintainers should still review trust handling before merge because the proposed system event appears to use the default trusted path.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a hardened version of the gateway handler that validates finite coordinates, sanitizes node-provided fields, queues the event as untrusted prompt data, wakes the scoped heartbeat, embeds the location event deliberately in the heartbeat prompt, and updates docs/tests. If passive location updates are not intended, the alternative is to remove or disable the current iOS producer so main no longer emits an event the gateway drops.

## Risks / Open Questions

- The implementation should add targeted tests for valid and invalid payloads, source sanitization, dedupe/wake behavior, heartbeat prompt inclusion, and provider attribution.
- Location data is sensitive; keep exact coordinates and accuracy out of info-level release logs and preserve the debug-only logging direction from the PR discussion.
