## Summary

Keep open. Current main still lacks the ACP lifecycle behavior this PR targets, and the PR remains a useful implementation candidate, but the latest patch is not merge-ready because the spawn health probe only handles thrown status errors and still treats resolved unhealthy statuses as a successful spawn.

## What This Changes

This PR adds ACP spawn health probing, stale gateway-disconnect recovery for cancel/close/early-turn paths, and focused ACP manager plus `/acp` command regression tests.

## Best Possible Solution

Keep the PR open and revise the spawn health guard to apply the same unhealthy-status classification used by `AcpSessionManager` before binding or persisting the spawned session. Preserve the explicit gateway close-code allowlist, keep non-stale errors like `1008` visible, and add a `/acp spawn` regression where `getStatus` resolves with `dead` or `no-session` and triggers cleanup.

## Reproduction Assessment

Yes. The intended bug can be reproduced with the mocked ACP runtime paths described in the PR, and the remaining patch bug can be reproduced by making spawn-time `runtime.getStatus` resolve `{ summary: "status=dead", details: { status: "dead" } }` instead of throwing.

## Solution Assessment

No, not yet. The recovery direction is narrow and consistent with existing ACP manager patterns, but the current PR must also reject resolved unhealthy status values before it is the best maintainable fix.
