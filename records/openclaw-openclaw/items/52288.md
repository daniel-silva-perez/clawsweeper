## Summary

Current main still has the reported single-slot client tool call path. The embedded attempt records only one `clientToolCall`, `run.ts` wraps that one value into a one-element `pendingToolCalls` array, and `/v1/responses` emits only `pendingToolCalls[0]` for both non-streaming and streaming responses. The linked contributor PR #52287 is relevant but closed and unmerged in the provided GitHub context, so the issue still needs maintainer follow-up.

## Best Possible Solution

Keep this open and land an equivalent of the closed PR #52287: accumulate client tool calls as an ordered array in the embedded attempt result, preserve all calls in run metadata, emit every pending call in both non-stream and streamed OpenResponses responses, accept the corresponding tool-result round trip, and add regression tests that assert multiple pending client tool calls are returned in order.
