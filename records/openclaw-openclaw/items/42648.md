## Summary

Keep open. Current main still has memory search/indexing, Markdown flushes, short-term promotion, an optional LanceDB memory plugin, and a provenance-rich wiki companion, but it does not implement the requested resolver-backed durable write path with required source attachment, canonical records, create/merge/update/conflict outcomes, entity normalization, and revision history. The related schema, provenance, review/conflict, and operator-surface issues are still open.

## What This Changes

Request to build a conservative candidate-memory write pipeline that classifies memories, attaches sources, dedupes or merges near-duplicates, surfaces conflicts, and preserves revision history before durable storage.

## Best Possible Solution

Design the durable memory resolver as a first-class memory-core/host SDK layer coordinated with #42646, #42647, #42650, and #42651: typed candidates, required source references, canonical SQLite records, normalized entities, near-duplicate and contradiction lookup, explicit create/merge/update/conflict-for-review outcomes, and non-destructive revision history, while keeping memory-wiki as a companion knowledge layer.
