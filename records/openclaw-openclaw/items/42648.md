## Summary

Keep open. Current main still has file-backed memory search, append-style compaction flushes, thresholded short-term promotion, memory-wiki provenance surfaces, and a LanceDB plugin with duplicate checks, but it does not implement the requested resolver-backed durable write pipeline with required source attachment, canonical records, merge/update/conflict outcomes, or revision history. The smart-memory work discussed in #49108 is useful prior art for classifier/entity quality, but it does not replace the missing OpenClaw memory-core/product contract.

## What This Changes

The issue requests a conservative durable-memory write pipeline that classifies candidates, attaches sources, checks duplicates and conflicts, resolves create/merge/update/review outcomes, and preserves provenance plus revision history.

## Best Possible Solution

Keep this open and design the memory write path as a first-class durable resolver layer: typed candidates, required source refs, canonical SQLite records, entity normalization, duplicate and contradiction lookup, explicit create/merge/update/conflict-for-review outcomes, and non-destructive revision history coordinated with the related schema, provenance, and review-flow issues.
