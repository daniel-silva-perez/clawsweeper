## Summary

Keep open. Current main has memory indexing/search, Markdown flushes, short-term promotion, optional LanceDB-backed memory, and a provenance-rich wiki companion, but it still does not provide the requested conservative durable write resolver with required source attachment, canonical create/merge/update/conflict outcomes, entity normalization, and non-destructive revision history. The linked schema, provenance, review/conflict, and ingestion-surface follow-ups remain open, so this is valid broad memory architecture work rather than implemented cleanup.

## What This Changes

Request to build a Memory MVP candidate-write pipeline that classifies incoming memories, attaches provenance, dedupes and merges near matches, surfaces conflicts, preserves revisions, and blocks low-confidence or source-less writes.

## Best Possible Solution

Design and implement the durable memory resolver as a first-class memory host/plugin contract coordinated with #42646, #42647, #42650, and #42651: canonical typed records, required source references, normalized subjects/entities, near-duplicate and contradiction lookup, explicit create/merge/update/conflict-for-review outcomes, and non-destructive revision history while keeping Markdown, wiki, and indexing layers as supporting surfaces.
