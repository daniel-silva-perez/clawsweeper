## Summary

Keep open. Current main has adjacent memory search, Markdown flush, short-term promotion, optional LanceDB storage, and a provenance-rich Memory Wiki companion, but it still lacks the requested conservative durable write resolver with required source attachment, canonical create/merge/update/conflict outcomes, entity normalization, and non-destructive revision history. The latest implementation-experience comment adds useful design input and supports keeping this as real memory architecture work.

## What This Changes

Build a conservative durable memory write pipeline that classifies candidate memories, attaches sources, deduplicates or merges near matches, and routes conflicts for review before durable writes.

## Best Possible Solution

Design and implement the durable memory resolver as a first-class memory host/plugin contract coordinated with #42646, #42647, #42650, and #42651: canonical typed records, required source references, normalized subjects/entities, near-duplicate and contradiction lookup, explicit create/merge/update/conflict-for-review outcomes, and non-destructive revision history while keeping Markdown, wiki, LanceDB, and indexing layers as supporting surfaces.

## Reproduction Assessment

Not applicable as a feature architecture request. The high-confidence verification path is source/docs inspection: current main exposes search, append-only flush, promotion, and optional LanceDB storage, but no canonical resolver contract or durable write schema.

## Solution Assessment

Unclear for a concrete patch because no implementation is under review. The requested direction is valid, but the maintainable solution should be a coordinated canonical memory schema/resolver design rather than a one-off change to Markdown flush or the optional LanceDB plugin.
