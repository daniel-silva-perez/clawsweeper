## Summary

Keep open. Current main has adjacent memory search, short-term promotion, Memory Wiki, and optional LanceDB pieces, but the default durable write path still lacks the requested conservative resolver with canonical create/merge/update/conflict outcomes, mandatory source attachment, entity normalization, and non-destructive revision history. The latest discussion adds useful production experience that reinforces the need for a deliberate memory-write design rather than cleanup closure.

## What This Changes

Build a conservative memory ingestion write pipeline that classifies candidates, attaches provenance, deduplicates and merges near-duplicates, and routes conflicts to review before durable writes.

## Best Possible Solution

Design and implement the memory write resolver as a coordinated durable memory contract: typed canonical records, required source references, normalized subjects/entities, near-duplicate and contradiction lookup, explicit create/merge/update/conflict-for-review outcomes, and append-only revision/provenance history, with Markdown, Memory Wiki, LanceDB, and indexing layers serving that contract rather than bypassing it.

## Reproduction Assessment

Not applicable as a feature architecture request. The high-confidence verification path is source and docs inspection: current main exposes recall, append-style promotion, wiki claim metadata, and optional LanceDB duplicate checks, but no canonical durable write resolver or storage model.

## Solution Assessment

Unclear for a concrete patch because no implementation is under review. The requested direction is valid, but the best fix needs maintainer design across schema, plugin/API boundaries, provenance, and review flows rather than a one-off change to promotion, Markdown flush, or LanceDB storage.
