## Summary

Keep this issue open. Current main still lacks a `silent` field on the cron failure-alert contract and still does not propagate any silent delivery option from cron failure alerts into the shared outbound/Telegram path. The existing issue discussion is accurate: Telegram/message delivery can already send silently, but cron failure alerts cannot request it yet.

## Best Possible Solution

Implement a narrow additive cron contract change: add `silent?: boolean` to per-job `CronFailureAlert` and global `cron.failureAlert`, update TypeBox/Zod/generated config docs and Control UI state/forms, propagate the resolved boolean through `resolveFailureAlert`, `emitFailureAlert`, `sendCronFailureAlert`, `sendGatewayCronFailureAlert`, `sendCronAnnouncePayloadStrict`, and `deliverOutboundPayloads`, and add focused tests proving Telegram cron failure alerts set `disable_notification` when `silent: true` while preserving default behavior when omitted.

## Risks / Open Questions

- The implementation should preserve the current audible default when `silent` is omitted or false.
- Maintainers should decide whether `silent` is valid only for announce/channel failure alerts or whether webhook-mode payloads should reject/ignore it consistently.
- A Telegram-only change would not solve the issue because cron config, gateway validation, UI state, and runtime delivery currently omit the field before Telegram delivery is reached.
