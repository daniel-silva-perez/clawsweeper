## Summary

Keep open. Current main still lacks a `silent` field in the cron failure-alert contract, gateway validation, config schema, runtime failure-alert plumbing, and Control UI form state, while outbound delivery and Telegram already support silent sends.

## What This Changes

Add a boolean `silent` option to cron job `failureAlert` so Telegram failure alerts can be delivered without sound or push notification while keeping audible delivery as the default.

## Best Possible Solution

Add a backward-compatible cron failure-alert `silent?: boolean` contract that resolves from cron alert config, passes through only the announce/channel delivery path into `deliverOutboundPayloads`, updates schema/docs/UI/tests, and leaves omitted or false values audible.

## Reproduction Assessment

Yes. A `cron.add` or `cron.update` payload with `failureAlert.silent` can be checked against current `CronFailureAlertSchema`, which omits the field and rejects additional properties; source inspection also shows no runtime or UI path for preserving it.

## Solution Assessment

Yes. The requested additive field is the narrowest maintainable path because the downstream outbound and Telegram layers already support silent delivery; the fix should avoid Telegram-specific branching in cron core and keep the default audible.
