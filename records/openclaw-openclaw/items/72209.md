## Summary

Keep this PR open. Current main still does not contain the passive recovery checkpoint surface, and the proposed patch is not merge-ready because the persisted Recovery Bundle is not wired into the real prompt creation path, with additional review findings around free-text redaction and synchronous checkpoint I/O.

## What This Changes

The PR adds agent session-recovery prompt helpers, local Task Ledger/Recovery Bundle persistence under the OpenClaw state directory, and checkpoint writes from `update_plan`, `sessions_yield`, and `sessions_spawn` with focused tests.

## Best Possible Solution

Keep the PR open for author/maintainer iteration: load the right Recovery Bundle when a session is recreated, pass it through the actual embedded/CLI prompt builders, keep recovered context passive and non-authorizing, make checkpoint persistence non-blocking or queued, tighten free-text redaction, and then validate the update_plan/yield/spawn and prompt-composition paths together.

## Reproduction Assessment

Yes for the review findings: static source inspection is enough. The PR writes checkpoint files from tools, but its changed files do not add any runtime caller that loads a bundle and passes `sessionRecovery` into live prompt construction; the redaction and synchronous I/O issues are directly visible in `session-recovery-state.ts`.

## Solution Assessment

No. The direction is plausible, but the current patch is not the narrowest complete solution because it records recovery data without delivering it to automatically recreated sessions and does blocking best-effort writes on tool execution paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire recovered bundles into real prompt creation:** `src/agents/system-prompt.ts:502`
  - body: This adds `sessionRecovery` only to the leaf prompt builder, but the actual embedded/CLI prompt callers are not changed and no runtime path loads a saved Recovery Bundle for a recreated session. As written, the checkpoints are persisted but never appear in live prompts, so the advertised recovery context is unavailable.
  - confidence: 0.93
- **[P2] Avoid redacting normal recovery prose:** `src/agents/session-recovery-state.ts:115`
  - body: The bare `token|secret|authorization|password` pattern treats ordinary summaries like `authorization: waiting for review` as secret material and stores them as redacted text. That silently damages the recovered context this feature is meant to preserve; keep the high-confidence secret patterns or require stronger value shapes.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: Every enabled `update_plan`, `sessions_yield`, and `sessions_spawn` checkpoint calls synchronous filesystem writes before returning the tool result. Since these checkpoints are best-effort, slow local or network-backed state directories can block unrelated agent progress; queue the write or use async persistence.
  - confidence: 0.88
