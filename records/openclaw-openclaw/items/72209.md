## Summary

Keep this PR open. Current main still does not implement the passive session recovery checkpoint layer proposed here: there is no session recovery prompt section, no Recovery Bundle / Task Ledger state surface, and no checkpoint side effects in update_plan, sessions_yield, or sessions_spawn. The existing code only covers active restart/orphan recovery for interrupted sessions, which is adjacent but not equivalent. The related #72436 was closed unmerged and does not supersede this PR.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase against current main, address the redaction and synchronous-I/O concerns, preserve the passive/informational-only contract, prove approvals/auth/permissions are never inherited, add focused tests for update_plan, sessions_yield, sessions_spawn, prompt composition, and persisted state, then decide whether this new recovery-state surface needs user-facing docs.

## Risks / Open Questions

- Greptile's P2 review comments remain relevant for maintainer review: broad redaction patterns can corrupt legitimate recovery summaries, and synchronous checkpoint writes can block the event loop on plan/yield/spawn paths.
- The PR adds persistent local recovery context and prompt text. Even if intended as informational only, prompt-injection handling, secret redaction, local file permissions, and explicit non-inheritance of approvals/auth/permissions need careful review before merge.
- The design needs reconciliation with existing active restart recovery, subagent orphan recovery, transcript compaction, reset/archive behavior, and session-store maintenance so the new passive bundle does not conflict with shipped recovery paths.
