## Summary

Keep this PR open. Current main still lacks the passive recovery prompt discipline, Task Ledger / Recovery Bundle files, and checkpoint writes proposed here, but the PR should not merge as-is because the new recovery-state path still has concrete review issues around free-text redaction and synchronous checkpoint I/O.

## What This Changes

This PR adds passive session recovery prompt guidance, local session recovery ledger/bundle persistence, and checkpoint writes from update_plan, sessions_yield, and sessions_spawn under src/agents.

## Best Possible Solution

Keep the passive recovery direction only if it remains strictly informational, stores bounded local state with secure permissions, does not inherit approvals or authority, uses non-blocking best-effort writes, and has focused tests around prompt composition, checkpoint persistence, spawn/yield/plan behavior, stale bundles, and existing restart/orphan recovery interactions.

## Reproduction Assessment

Not applicable as a feature PR rather than a reported runtime bug. The high-confidence check is code inspection: current main lacks the passive recovery symbols and tool side effects, while the PR diff adds them.

## Solution Assessment

No, not as-is. The proposed direction is plausible, but the implementation should first tighten redaction for free-text summaries and move best-effort checkpoint persistence off the hot tool execution path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Tighten recovery redaction to avoid losing context:** `src/agents/session-recovery-state.ts:115`
  - body: Recovery summaries are free text, but this pattern redacts bare labels like `authorization: waiting for review` or `token: refresh task` even when no secret is present. That silently degrades the context the ledger is meant to preserve; please restrict this to higher-confidence credential shapes or explicit key/value secret fields.
  - confidence: 0.86
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:200-201`
  - body: `appendTaskLedgerEvent` performs synchronous mkdir/append work, and the checkpoint path then reaches synchronous bundle writes as well. Since update_plan, sessions_yield, and sessions_spawn call this best-effort side channel during normal tool execution, slow state storage can block the Node event loop; queue/offload the writes or make this path async.
  - confidence: 0.83
