## Summary

Keep this open. Current main still only satisfies the sandbox portion through the existing agent-scoped sandbox config, while `routing.rules`, a single-config `instances` model, and core `gateway.nodes.denyPaths` remain absent from the strict config schema and from the gateway node policy chokepoint. The remaining work is broad and security-sensitive, so this should stay with maintainers for product/security design rather than conservative cleanup closure.

## What This Changes

The issue requests config schema and gateway enforcement support for a global sandbox mode, tag-based provider routing rules, multi-instance declarations, and gateway-wide node path denial rules.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox` path, then split the remaining work into maintainer-designed tracks: provider-routing rule semantics and tag provenance, multi-instance declaration versus existing profile isolation, and a gateway-wide path-deny contract with a clearly documented enforcement boundary. The file-transfer plugin policy can inform the path-deny design but does not replace the requested core policy.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence check is source and docs inspection: current main has agent-scoped sandboxing, while the requested `routing.rules`, single-config `instances`, and core `gateway.nodes.denyPaths` surfaces are absent.

## Solution Assessment

No, not as one bundled schema change. The safer solution is to separate the already-supported sandbox concern from the routing, multi-instance, and path-deny designs before any implementation PR.
