## Summary

Keep this issue open. Current main only satisfies the sandbox part through the existing agent-scoped sandbox configuration; the requested `routing.rules`, single-config `instances`, and core `gateway.nodes.denyPaths` contract are still absent from the strict config schema and from the gateway node policy chokepoint. The remaining work is broad and security-sensitive, so it needs maintainer design rather than conservative cleanup closure.

## What This Changes

Request adding config schema surfaces for global sandbox mode, tag-based provider routing rules, multi-instance declarations, and gateway-wide node path denial.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox` path for sandboxing, then split the remaining request into maintainer-designed tracks: tag-based provider routing, multi-instance declaration/validation versus existing profile isolation, and a gateway-wide path-deny contract with a clearly documented enforcement boundary. The file-transfer plugin’s `denyPaths` can inform the path-policy slice but should not be treated as the core implementation.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection: current main has agent-scoped sandboxing, while `routing.rules`, `instances`, and core `gateway.nodes.denyPaths` remain absent.

## Solution Assessment

No, not as one bundled schema addition. The safer maintainable path is to separate the already-supported sandbox concern from the routing, multi-instance, and path-deny designs before implementation.

- `instances.*` overlaps the current profile/config/state isolation model and needs product direction before one config file becomes a multi-gateway authority.
