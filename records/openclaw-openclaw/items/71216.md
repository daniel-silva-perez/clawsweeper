## Summary

Keep this issue open. Current main only satisfies the sandbox part through the existing agent-scoped sandbox configuration; `routing.rules`, single-config `instances`, and a core `gateway.nodes.denyPaths` policy are still absent from the strict config schema and from the gateway node command-policy chokepoint. The remaining request is broad and security-sensitive, so it needs maintainer design rather than conservative cleanup closure.

## What This Changes

The issue requests config schema and gateway enforcement support for global sandbox mode, tag-based provider routing, multi-instance declarations, and `gateway.nodes.denyPaths`.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox` path, then split the remaining work into maintainer-designed tracks: tag provenance plus provider-route resolution, multi-instance declaration/validation versus existing profile isolation, and a gateway-wide path-deny contract with an explicit enforcement boundary. The existing file-transfer plugin denyPaths policy can inform the file-transfer slice but does not replace the requested core policy.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence check is source and docs inspection: current main has agent-scoped sandboxing, while the requested `routing.rules`, `instances`, and core `gateway.nodes.denyPaths` surfaces are absent.

## Solution Assessment

No, not as one bundled schema change. The safer path is to separate the already-supported sandbox concern from the remaining routing, multi-instance, and path-deny designs before implementation.
