## Summary

Keep this open. Current main covers the sandbox portion through existing agent-scoped sandbox config, but the requested `routing.rules`, single-config `instances`, and core `gateway.nodes.denyPaths` schema/runtime contract are still absent and security-sensitive enough to require maintainer design before implementation.

## What This Changes

The issue requests config schema and gateway enforcement support for global sandbox mode, tag-based provider routing, multi-instance declarations, and gateway-wide node path denial.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox` and per-agent sandbox model, then split the rest into maintainer-designed tracks: trusted provider-routing metadata, multi-instance declaration versus the documented separate-profile isolation model, and a gateway-wide path-deny contract with explicit coverage and non-coverage.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection: current main has agent-scoped sandboxing, while `routing.rules`, `instances`, and core `gateway.nodes.denyPaths` are not present in the strict schema or gateway node policy.

## Solution Assessment

No, not as one bundled schema addition. The safer maintainable solution is to separate the already-supported sandbox concern from routing, instance validation, and path denial so each gets a clear enforcement boundary and targeted tests.
