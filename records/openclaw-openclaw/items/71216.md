## Summary

Keep this issue open. Current main still lacks the requested first-class config and runtime surfaces, and the request combines config schema, routing/product design, multi-instance management, and security-boundary enforcement that needs maintainer direction rather than conservative cleanup closure.

## What This Changes

Request first-class config support for a root sandbox mode, tag-based provider routing rules, multi-instance gateway declarations, and gateway node path deny policies.

## Best Possible Solution

Keep the issue open and split it deliberately: decide whether a root sandbox key should alias or replace agent-scoped sandbox config, design tag-aware model/provider routing with explicit metadata provenance, decide whether multi-instance declarations belong in config or profile/service management, and implement deny-path policy only with schema, docs, doctor/audit coverage, runtime enforcement, and security tests.

## Reproduction Assessment

Not applicable as a runtime bug report. The config gap has a high-confidence verification path by inspecting the strict config schema/types plus sandbox, routing, node-policy, and multi-gateway docs on current main.

## Solution Assessment

Unclear that the bundled issue shape is the best single fix. The maintainable path is to split the four asks and require runtime enforcement for any deny-path security boundary rather than accepting a schema-only implementation.
