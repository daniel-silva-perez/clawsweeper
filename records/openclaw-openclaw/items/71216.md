## Summary

Keep open. Current main only satisfies the sandbox portion through the existing agent-scoped sandbox config; the requested tag-based provider routing, single-config gateway instances, and gateway-wide node path denial remain absent from the strict schema/runtime contract and are security-sensitive enough to need maintainer design before implementation.

## What This Changes

Request to add config/runtime support for top-level sandbox policy, tag-based provider routing rules, single-config gateway instances, and `gateway.nodes.denyPaths`.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox` and per-agent sandbox model, then split the remaining request into maintainer-designed tracks: provider-routing metadata with trusted tag provenance, multi-instance declaration versus the documented separate-profile isolation model, and a gateway-wide path-deny contract with explicit enforcement boundaries and tests.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection: current main has agent-scoped sandbox config, while `routing.rules`, top-level `instances`, and `gateway.nodes.denyPaths` are absent from the strict schema, exported types, docs, and node command policy.

## Solution Assessment

No, not as one bundled schema addition. The safer maintainable path is to separate the already-supported sandbox concern from routing, instance validation, and path denial so each can get a clear runtime enforcement boundary and targeted tests.

## Security Review

Status: needs_attention

Summary: The issue requests security-sensitive policy controls whose enforcement boundaries are not yet defined in current main.

Concerns:

- **[medium] Define the global path-deny enforcement boundary:** `src/gateway/node-command-policy.ts:200`
  - body: A `gateway.nodes.denyPaths` field could overpromise protection unless maintainers decide whether it covers node-host file transfer, `system.run`, local filesystem tools, sandbox bridges, elevated exec, and plugin-owned tools. Current gateway node policy only filters command ids.
  - confidence: 0.86
- **[medium] Define trusted tag provenance for provider routing:**
  - body: Sensitivity-based provider routing can leak data to the wrong model provider if tags are optional, user-forgeable, dropped during delegation/subagent calls, or ambiguous when multiple rules match.
  - confidence: 0.8
