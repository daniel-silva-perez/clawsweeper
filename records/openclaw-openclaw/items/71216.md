## Summary

Keep this open. Current main has adjacent mechanisms for agent-scoped sandboxing, binding-based agent routing, profile-based multi-gateway isolation, and node command allow/deny lists, but it does not implement the requested `routing.rules`, `instances`, or `gateway.nodes.denyPaths` config contracts, and the requested top-level `sandbox` shape is only partially covered by `agents.defaults.sandbox` / `agents.list[].sandbox`. Because these are security- and product-contract changes, this needs maintainer design before an automated fix PR.

## What This Changes

The issue requests config schema and gateway enforcement for global sandbox policy, tag-based provider routing, declarative multi-instance validation, and gateway node deny paths.

## Best Possible Solution

Keep the issue open for maintainer design. The likely end state is a split design: preserve existing `agents.defaults.sandbox` and `bindings` semantics where they already solve use cases, then add any approved new schema/runtime contracts with matching enforcement, doctor validation, docs, and focused tests rather than schema-only placeholders.

## Risks / Open Questions

- The request bundles four separate public config contracts; landing them together could create unclear precedence and migration behavior.
- `gateway.nodes.denyPaths` would need canonical path handling across gateway host tools, node-host `system.run`, sandbox filesystem bridges, plugin tools, symlinks, and platform path formats.
- `routing.rules` depends on a stable tag propagation contract for skill/agent calls; the current routing model is channel-to-agent, not tag-to-provider.
- `instances.*` overlaps the documented profile-based multi-gateway isolation model and may require a product decision on whether one config should validate several Gateway processes.
