## Summary

Keep open. The report is concrete and still paired with open draft PR #62441, while the maintainer discussion says the proposed `senderName` to OpenAI API `name` root cause has not been verified. Current main still preserves raw Telegram display names in channel metadata and runner context, but the inspected OpenAI serializer path does not show user sender names being emitted as API message `name` fields.

## What This Changes

The issue asks OpenClaw to prevent Telegram forum-topic sender display names from causing GitHub Copilot/OpenAI-compatible 400 request-body failures and provider-wide cooldowns.

## Best Possible Solution

Keep this issue open until a redacted failing outbound request body identifies the rejected field. Then review PR #62441 or a replacement fix that sanitizes only the actual provider-bound field and adds a regression test for Telegram display names plus provider cooldown behavior.

## Reproduction Assessment

Unclear. The report gives concrete reproduction steps, but the current review did not have a live Telegram forum-topic plus GitHub Copilot/OpenAI payload capture; the missing high-confidence reproduction artifact is the redacted outbound request body that returns 400.

## Solution Assessment

Unclear. Sanitizing sender names could be right if the payload proves they reach an OpenAI-compatible `name` field, but the safer path is to first identify the exact rejected field and fix that narrow serialization boundary.
