## Summary

Keep open. Current main still has the mixed-image gaps described in the issue: HTTP(S) prompt image URLs are explicitly excluded from native image detection, and the fallback `image` tool remains available to vision-capable runs with only descriptive guidance. An open linked PR, #63710, uses closing syntax for this issue, so the right cleanup action is to review, land, or revise that PR rather than close this issue ahead of merge.

## What This Changes

The issue asks OpenClaw to make Web UI mixed image input deterministic for vision-capable models so uploaded images, prompt-referenced images, and fallback tool behavior do not cause ignored images or hallucinated image-tool calls.

## Best Possible Solution

Keep the issue open until the linked implementation path is resolved. The maintainable end state is a single prompt-image assembly path for vision-capable models, with regression coverage for uploaded inline plus HTTP(S) refs and a conscious maintainer decision on whether the fallback `image` tool should be hidden or otherwise guarded when native images are already present.

## Reproduction Assessment

Yes for the deterministic core of the bug: current helper behavior can be reproduced by inspecting or testing `detectImageReferences()` with a prompt that contains a local image path and an HTTP(S) image URL; current tests already assert that only the local path is detected. The exact hallucinated tool call still depends on the selected model, but the runtime condition that leaves the URL as plain text remains present.

## Solution Assessment

No, closing this issue now is not the best way to solve it. The best next step is maintainer review of #63710, with scope adjusted if needed to cover the original weak-tool-calling image-tool misuse as well as HTTP(S) mixed-source loading.
