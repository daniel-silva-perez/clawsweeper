## Summary

Current main still does not expose the requested host-owned Memory Plugin API for file scan/watch/change events. OpenClaw has memory slots, memory runtime registration, public artifacts, event-journal helpers, and reusable file/chunk helpers, but the actual startup scan, chokidar watcher, hash-based incremental sync, and stale cleanup remain private to memory-core. The bundled memory-lancedb plugin still registers tools and lifecycle hooks, not a file-indexing runtime that receives memory file events.

## Best Possible Solution

Keep this issue open and design an additive memory file-indexing capability for the active memory plugin: a host-owned scanner/watcher service that can deliver full-scan batches and add/change/delete events built on existing list/chunk helpers, with contract tests, docs, and a memory-core dogfood path before third-party memory plugins rely on it.

## Risks / Open Questions

- A future API needs explicit semantics for full scans, add/change/delete events, unlinkDir behavior, batching/backpressure, extraPaths changes, active memory-slot gating, error handling, and stale cleanup.
- The existing helper exports reduce duplicated code, but treating them as the requested pipeline would still leave external memory plugins responsible for implementing their own watcher and sync state.
