## Summary

Current main still does not implement the requested host-owned Memory Plugin API for file scan/watch/change events. OpenClaw now exposes useful memory file and chunking helpers, but the startup scan, chokidar watcher, incremental hash sync, and stale cleanup remain owned by memory-core, while memory-lancedb remains tool/lifecycle-hook based rather than file-indexing capable.

## Best Possible Solution

Keep this issue open and design an additive memory file-indexing capability for the active memory plugin: a host-owned scanner/watcher service that can deliver full-scan batches and add/change/delete events built on the existing list/chunk helpers, with contract tests, docs, and a memory-core dogfood path before third-party plugins rely on it.

## Risks / Open Questions

- An eventual API needs explicit lifecycle semantics for full scans, incremental add/change/delete events, unlinkDir, batching/backpressure, extraPaths changes, active memory slot gating, and stale cleanup without freezing memory-core internals wholesale.
- The existing helper exports may reduce duplicated code, but treating them as the requested pipeline would still leave external memory plugins responsible for reimplementing watching and sync state.
