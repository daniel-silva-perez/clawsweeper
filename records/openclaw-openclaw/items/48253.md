## Summary

Keep open. Current main has memory-plugin runtime and helper exports for memory files/chunking, but it does not implement the requested host-owned scan/watch/change-event API for third-party memory plugins. The built-in memory-core plugin still owns the chokidar watcher, hash-based sync, indexing, and stale cleanup path internally, while the bundled LanceDB memory plugin remains hook/tool based and does not receive file-indexing events or extraPaths automatically. The related #32966 is closed/stale and broader, so it is not a canonical open tracker for this narrower SDK gap.

## Best Possible Solution

Keep the issue open and design an additive MemoryPluginCapability/MemoryPluginRuntime file-indexing API: a host-owned scanner/watcher service that can deliver full-scan batches and add/change/delete events built on the existing list/chunk helpers, gated by the active memory slot. Dogfood it through memory-core, document it in the plugin SDK docs, and add contract tests plus a migration path for memory-lancedb-style plugins.

## Risks / Open Questions

- Closing this as ClawHub would be premature because the core gap is the missing SDK/runtime seam for host-owned file scan/watch events, not merely a request to bundle a third-party memory plugin.
- The eventual API should avoid freezing memory-core internals wholesale; it likely needs an additive, narrow contract with explicit lifecycle, batching, delete semantics, extraPaths handling, and contract tests.
