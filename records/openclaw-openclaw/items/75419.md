## Summary

Close as implemented on current main. The reported TUI input failure against v2026.4.29 is covered by current raw-stdin progress handling: main suppresses readline-backed CLI progress while the TUI owns input, has regression coverage for that behavior, and the TUI input listeners preserve normal character input.

## What This Changes

Fix OpenClaw v2026.4.29 on macOS so `openclaw tui` accepts typed prompt input again.

## Best Possible Solution

Keep the current raw-stdin progress suppression and ship it in the next release; if the problem persists after a build containing the main fix, a new report should include the terminal app, `TERM_PROGRAM`, whether `openclaw chat` differs from Gateway-backed `openclaw tui`, and any visible status text.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for maintainer cleanup: the report gives the direct path `openclaw tui`, and current main has a focused regression test for the likely raw-stdin spinner/input ownership failure mode rather than a live macOS TUI recording.

## Solution Assessment

Is this the best way to solve the issue? Yes. Suppressing readline-backed progress when `process.stdin.isRaw` is the narrowest maintainable fix because it preserves normal CLI spinners outside the TUI while preventing progress UI from interfering with terminal input ownership.
