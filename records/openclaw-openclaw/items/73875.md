## Summary

Current main still matches the reported failure mode. OpenClaw documents Claude as an ACPX target, `sessions_spawn(runtime:"acp")` now forwards model/thinking/timeout into ACP runtime options, the control plane maps those generic options to `model`, `thinking`, `approval_policy`, and `timeout`, and the ACPX wrapper still forwards non-Codex config keys unchanged. The pinned ACPX runtime does not advertise strict config keys, while the pinned Claude adapter rejects unknown config IDs and exposes `mode`, `model`, and optional `effort`, so a Claude-managed ACP session can still fail before output when unsupported generic keys are applied.

## Best Possible Solution

Keep this issue open and fix the ACP runtime-control path generically. The likely fix is to derive supported config option IDs from the active ACP session or ACPX status when available, map OpenClaw `thinking` to adapter-specific keys such as Claude `effort` when advertised, keep run timeout as an OpenClaw/ACPX execution timeout unless the adapter explicitly advertises a timeout config option, skip unsupported generic keys without failing the turn, and add regression coverage with a Claude-like non-Codex adapter that advertises only `mode`, `model`, and `effort`.
