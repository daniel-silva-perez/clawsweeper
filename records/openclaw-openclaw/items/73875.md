## Summary

Keep open. Current main still has the reported ACP runtime-control behavior, and an open PR already uses closing syntax for this issue, so the issue should remain open until that implementation is reviewed and landed or replaced.

## What This Changes

The issue asks OpenClaw-managed ACP sessions to avoid sending generic unsupported runtime config options to Claude ACP and instead apply only advertised or adapter-supported options.

## Best Possible Solution

The ACP runtime-control layer should derive supported config IDs from backend/session capabilities, skip or gracefully degrade unsupported generic keys for strict adapters, keep timeout as an OpenClaw execution timeout unless explicitly advertised by the adapter, and add regression coverage for no-list and strict-key backends.

## Reproduction Assessment

Yes. The issue gives a concrete live reproduction path, and current main source shows the static failure precondition: `sessions_spawn` can store generic options, those options become ACP config IDs, and the manager forwards every pair when no strict key list is present.

## Solution Assessment

Unclear until the linked PR is reviewed. The direction of skipping unsupported config options is right, but the maintainable fix should align the capability contract and include regression tests before the issue is considered solved.
