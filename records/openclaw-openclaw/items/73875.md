## Summary

Keep open. Current main still documents Claude ACP as a supported ACPX harness and still sends generated runtime config options when the backend exposes `session/set_config_option` without advertising `configOptionKeys`. An open PR, #74048, already uses closing syntax for this issue, so the issue should remain open until that implementation is reviewed and either landed or replaced.

## What This Changes

OpenClaw should stop sending unsupported generic ACP runtime config options such as `timeout`, `thinking`, and `approval_policy` to strict Claude ACP adapters unless those options are advertised or mapped safely.

## Best Possible Solution

Land or replace #74048 with a tested ACP runtime-control fix that derives supported config IDs from backend capabilities, avoids unsupported generic keys for strict adapters, keeps timeout as an OpenClaw execution timeout unless explicitly advertised, and deliberately handles Claude-style `effort` mapping for thinking controls.

## Reproduction Assessment

Yes. The issue gives a concrete live path, and current main source shows the static failure precondition: `sessions_spawn` can persist generic runtime options, those options become ACP config IDs, and the manager forwards every pair when `configOptionKeys` is absent.

## Solution Assessment

Unclear. Skipping unsupported config options is the right direction, but the linked PR should align the runtime capability contract and add regression coverage before this is considered the best maintainable fix.
