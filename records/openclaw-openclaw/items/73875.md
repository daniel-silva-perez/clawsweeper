## Summary

Keep open. Current main still exposes Claude as a supported ACPX target and still forwards generated runtime config options when `session/set_config_option` exists but no strict `configOptionKeys` list is advertised. The open PR #74048 uses closing syntax for this issue, so the issue should remain open until that implementation is reviewed, landed, or replaced.

## What This Changes

OpenClaw should stop sending unsupported generic ACP runtime config options such as `timeout`, `thinking`, and `approval_policy` to strict Claude ACP adapters unless those options are advertised or mapped safely.

## Best Possible Solution

Land or replace #74048 with a tested ACP runtime-control fix that derives supported config IDs from backend capabilities, avoids unsupported generic keys for strict adapters, keeps timeout as an OpenClaw execution timeout unless explicitly advertised, and handles Claude-style thinking/effort mapping deliberately.

## Reproduction Assessment

Yes. The issue gives a concrete live path, and current main shows the static failure precondition: ACP spawns can persist generic runtime options, those options become ACP config IDs, and the manager forwards every pair when `configOptionKeys` is absent.

## Solution Assessment

Unclear. Skipping unsupported config options is the right direction, but the linked PR needs maintainer review for the `configOptionKeys` contract and regression coverage before this is the best maintainable fix.
