## Summary

Keep open. Current main still builds generic ACP config-option pairs and forwards them when the backend exposes set_config_option but no strict configOptionKeys list; the linked closing PR #74048 is still open and unmerged, so this issue should wait for that PR or a successor to land.

## What This Changes

The issue asks OpenClaw-managed Claude ACP sessions to apply only adapter-supported ACP config options instead of sending generic runtime keys such as thinking, approval_policy, and timeout.

## Best Possible Solution

Land a tested ACP runtime-control fix that filters or skips config options based on backend-advertised support, maps generic thinking to adapter-specific keys such as Claude effort only when supported, keeps run timeout as an OpenClaw/ACPX execution timeout unless the adapter advertises timeout, and preserves Codex ACP startup/config behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by running an ACP manager path with persisted model/thinking/permissionProfile/timeoutSeconds and a non-Codex backend that has set_config_option but no configOptionKeys; current tests already assert those generic keys are sent.

## Solution Assessment

Yes for the direction, but not yet for current main. Capability-based filtering plus adapter-specific mapping is the narrow maintainable fix; the linked PR is a plausible candidate but needs tests and review before this issue can close.
