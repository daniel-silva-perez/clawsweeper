## Summary

Keep open. Current main still does not expose `X-Actual-Model` or any equivalent actual runtime provider/model response metadata on the Gateway OpenAI-compatible HTTP endpoints. The handlers still preserve the request-side `model` value in `/v1/chat/completions` and `/v1/responses`, while streaming headers are flushed before fallback-selected runtime model metadata could be sent as a normal response header. The prior maintainer/Codex review remains accurate, and related #51441 and #54985 cover broader session observability and outbound request headers rather than this response-header contract.

## What This Changes

Request to add an `X-Actual-Model` HTTP response header that exposes the actual runtime model used for Gateway OpenAI-compatible responses, especially when fallbacks or model routing change the backend model.

## Best Possible Solution

Keep this open for maintainer API-contract work: define actual runtime provider/model metadata for `/v1/chat/completions` and `/v1/responses`, preserve the existing OpenAI-compatible request-side `model` field, decide non-streaming header/body plus streaming SSE/trailer behavior, then thread selected provider/model from `onModelSelected` or `agentMeta` into the handlers with focused tests and docs.

## Risks / Open Questions

- A simple late response header is not sufficient for streaming because the current SSE path flushes headers before the agent run can report a fallback-selected model.
- Adding actual runtime metadata must preserve the existing OpenAI-compatible `model` response semantics, which currently represent the request-side agent target rather than the backend provider/model.
- This is a public Gateway HTTP contract decision spanning non-streaming JSON, SSE events, and possibly trailers, so maintainers should choose the shape before an automated implementation PR.
