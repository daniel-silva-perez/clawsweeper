## Summary

Keep this issue open. Current main still does not expose `X-Actual-Model` or an equivalent actual runtime provider/model field on the Gateway OpenAI-compatible HTTP responses. The Chat Completions and Responses handlers preserve the request-side compatibility `model` value in response bodies/SSE chunks, while the actual selected model callback remains available only on internal reply/channel paths. The recent prior Codex review comment remains accurate against current `main` at `ab5c8025c9d0a31873d311ae3e73f3664b0a1185`.

## Best Possible Solution

Keep this open. Maintainers should define a Gateway HTTP contract for actual runtime provider/model metadata, thread selected provider/model from the fallback execution path into `src/gateway/openai-http.ts` and `src/gateway/openresponses-http.ts`, preserve the existing compatibility `model` response semantics, explicitly define non-streaming and SSE behavior, document it on the public Gateway HTTP API docs, and add focused tests for both compatibility endpoints.
