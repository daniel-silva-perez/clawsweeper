## Summary

Keep open. Current main still lacks the requested actual-runtime-model transport surface on the Gateway OpenAI-compatible HTTP endpoints; the code echoes request-side model values, the actual selected-model callback remains below the HTTP layer, and streaming responses flush headers before fallback-selected model data can be represented as a normal response header.

## What This Changes

Request to add an `X-Actual-Model` or equivalent Gateway HTTP response metadata surface so clients can see which backend provider/model actually processed each request, including fallback cases.

## Best Possible Solution

Define an additive Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire existing selected provider/model data into that contract with focused tests and public docs.
