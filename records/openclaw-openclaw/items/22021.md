## Summary

Keep open. Current main still lacks the requested actual-runtime-model transport metadata on the OpenAI-compatible Gateway endpoints: there is no `X-Actual-Model` or equivalent response surface, the HTTP responses use the request-side agent target model, and the actual selected-model callback remains below the HTTP layer.

## What This Changes

The issue requests a Gateway HTTP response metadata surface, specifically an `X-Actual-Model` header, so clients can see which model actually processed each request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming responses, then wire selected provider/model from `onModelSelected` or equivalent run metadata with focused endpoint tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by inspecting current main: exact searches find no actual-model response metadata, and the Gateway HTTP handlers emit the request-side `model` while selected-model data remains below the HTTP layer.

## Solution Assessment

Unclear. The requested header is a useful non-streaming shape, but it is not sufficient for streaming fallback cases because SSE headers are flushed before late model selection; the maintainable solution needs a documented streaming metadata mechanism as well.
