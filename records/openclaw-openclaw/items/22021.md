## Summary

Keep open. Current main still lacks a Gateway actual-runtime-model response surface on the OpenAI-compatible HTTP endpoints, and the remaining problem is a public API contract decision because ordinary headers cannot describe a fallback selected after streaming headers flush.

## What This Changes

The issue requests Gateway HTTP response metadata, proposed as `X-Actual-Model`, to expose the provider/model that actually handled each OpenAI-compatible request.

## Best Possible Solution

Define and document an additive Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire authoritative provider/model data from the execution path into that contract with focused tests.
