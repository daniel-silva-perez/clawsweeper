## Summary

Current main still does not implement `X-Actual-Model` or an equivalent actual runtime provider/model field on the Gateway OpenAI-compatible HTTP responses. The Chat Completions and OpenResponses handlers preserve the request-side compatibility `model` value, while actual model selection is only exposed through internal reply/channel callback paths. The related open items cover broader observability or outbound request headers but do not supersede this transport-response request.

## Best Possible Solution

Keep this issue open. Maintainers should define a Gateway HTTP contract for actual runtime provider/model metadata on `/v1/chat/completions` and `/v1/responses`, thread selected provider/model from fallback execution into the compatibility handlers, preserve the current request-side `model` response semantics, explicitly define non-streaming versus SSE behavior, document it on the public Gateway HTTP API page, and add focused tests for both compatibility endpoints.

- The naming and scope overlap with #51441, #7006, and #54985, so the HTTP response contract should be aligned with broader resolved-model observability work.
