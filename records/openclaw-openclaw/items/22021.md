## Summary

Keep open. Current main still lacks the requested actual-runtime-model response surface on the Gateway OpenAI-compatible HTTP endpoints, and the existing code continues to echo the request-side agent target model while internal selected-model plumbing is not passed into these handlers. The prior maintainer/Codex comment remains consistent with current source, and related #51441 and #54985 do not supersede this transport-layer response contract.

## What This Changes

Add an HTTP response header such as `X-Actual-Model` to expose the actual provider/model that processed each Gateway OpenAI-compatible request.

## Best Possible Solution

Define a Gateway HTTP contract for actual runtime provider/model metadata across `/v1/chat/completions` and `/v1/responses`: preserve the OpenAI-compatible request `model` as the agent target, expose selected provider/model for non-streaming responses, specify SSE/trailer behavior for streaming, then wire the selected metadata from `onModelSelected` or `agentMeta` with focused tests and docs.
