## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the Gateway OpenAI-compatible HTTP endpoints, and the prior maintainer review in the discussion correctly identifies the remaining problem as an API-contract decision rather than cleanup-close material.

## What This Changes

Request to add an `X-Actual-Model` or equivalent response metadata surface exposing the actual backend model used by OpenAI-compatible Gateway responses.

## Best Possible Solution

Define and document an additive Gateway actual-model metadata contract across `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming behavior, then wire existing selected provider/model data from the model-selection path into that contract with focused tests.
