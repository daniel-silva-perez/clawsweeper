## Summary

Keep open. Current main still lacks the requested actual-runtime-model HTTP response metadata on the Gateway OpenAI-compatible endpoints, and the discussion already identifies the remaining work as an API-contract decision rather than cleanup-close material.

## What This Changes

Request adding `X-Actual-Model` or equivalent transport metadata to Gateway OpenAI-compatible HTTP responses so clients can see the provider/model that actually handled each turn.

## Best Possible Solution

Define and document an additive Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming behavior, then wire authoritative selected provider/model data from the execution path into that contract with focused tests.
