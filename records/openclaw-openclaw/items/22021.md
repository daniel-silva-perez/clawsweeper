## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the OpenAI-compatible Gateway HTTP endpoints: the handlers serialize the request-side agent target model, docs only describe request overrides, and the existing actual selected-model callback remains below the HTTP layer. The remaining blocker is API-contract design, especially for streaming responses after SSE headers are flushed.

## What This Changes

Add a Gateway HTTP response metadata surface, proposed as `X-Actual-Model`, that exposes the actual runtime provider/model used for each OpenAI-compatible response.

## Best Possible Solution

Define and document an additive Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming behavior, then wire authoritative selected provider/model data from the execution path into that contract with focused tests.
