## Summary

Keep open. Current main still lacks the requested actual-runtime-model HTTP response surface for the Gateway OpenAI-compatible endpoints: the handlers return the request-side agent target model, while the actual selected model callback remains below the HTTP layer. The existing maintainer review and the open related PR are useful context, but neither establishes an implemented response header or streaming metadata contract.

## What This Changes

The issue requests an HTTP response metadata surface, proposed as `X-Actual-Model`, that exposes the actual provider/model that processed each Gateway OpenAI-compatible request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming responses, then wire the selected provider/model from execution into that contract with focused endpoint tests.

## Reproduction Assessment

Yes for the missing surface: inspect or exercise the Gateway OpenAI-compatible endpoints with a runtime-selected or fallback model that differs from the request-side agent target. Current source shows the responses use the request `model` value and no `X-Actual-Model` header is implemented.

## Solution Assessment

Unclear that the proposed header alone is the best solution. A header may work for non-streaming responses, but streaming needs a documented SSE/body/trailer-style metadata path because the current code flushes headers before model fallback resolution can complete.
