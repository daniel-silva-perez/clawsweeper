## Summary

Keep open. Current main still lacks the requested actual-runtime-model transport surface on the OpenAI-compatible Gateway endpoints: no `X-Actual-Model` or equivalent response metadata is present, response payloads use the request-side `model`, and the actual selected-model callback remains below the HTTP layer. The related open PR #54985 is adjacent request-context work, but it does not implement this response metadata contract.

## What This Changes

The issue requests an `X-Actual-Model` response header so clients can see which backend model actually processed each OpenAI-compatible Gateway request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming responses, then wire the selected provider/model from `onModelSelected` or equivalent run metadata with focused endpoint tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: inspect the current Gateway HTTP handlers and observe that no actual-model response header or SSE/body metadata is emitted while response resources use the request-side `model` value.

## Solution Assessment

Unclear. A plain response header may work for non-streaming responses, but it is not sufficient for streaming fallback cases because headers are flushed before late model selection; the maintainable solution needs an explicit streaming metadata contract.
