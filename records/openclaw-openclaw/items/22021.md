## Summary

Keep this issue open. Current main still does not implement `X-Actual-Model` or an equivalent actual runtime provider/model response contract on the Gateway OpenAI-compatible HTTP endpoints. `/v1/chat/completions` and `/v1/responses` preserve the request-side `model` value in response payloads, while streaming headers are flushed before runtime fallback/model selection can be reflected as a normal response header. The prior maintainer comment remains accurate, and related open items cover broader session observability or outbound request headers rather than superseding this transport-response request.

## Best Possible Solution

Keep this open. Maintainers should define the Gateway HTTP contract for actual runtime provider/model metadata on `/v1/chat/completions` and `/v1/responses`, preserve the existing OpenAI-compatible request-side `model` response semantics, decide the non-streaming header/body and SSE event/trailer behavior, then thread actual selected provider/model from `agentMeta` or a new selected-model callback into the compatibility handlers with focused tests and docs.
