## Summary

Keep open. Current main still lacks the requested actual-runtime-model HTTP response metadata: the OpenAI-compatible Gateway handlers echo the request-side agent target model, while the internal selected-model callback remains below the HTTP layer. The related open PR #54985 is unmerged and appears to cover request context/tracing headers rather than this exact response contract.

## What This Changes

Expose the model that actually processed each Gateway OpenAI-compatible HTTP request, originally proposed as an `X-Actual-Model` response header.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for both `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire selected provider/model data from `onModelSelected` or equivalent run metadata with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: inspect the current Gateway HTTP handlers and docs and observe that no actual-model response header or streaming metadata is emitted, while response bodies/chunks use the request-side `model` value.

## Solution Assessment

Unclear. The requested header is plausible for non-streaming responses, but it is not sufficient by itself for streaming fallback because headers are flushed before late model selection; the maintainable fix needs an explicit streaming metadata contract.
