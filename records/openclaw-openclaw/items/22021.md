## Summary

Keep this issue open. Current main still does not expose `X-Actual-Model` or an equivalent actual runtime provider/model field on Gateway OpenAI-compatible HTTP responses. The Chat Completions and Responses handlers return the request-side compatibility model (`openclaw`, `openclaw/<agent>`, or the requested model), and the internal selected-model callback is still not wired through the HTTP ingress path. The prior April 26 Codex review remains accurate against current main.

## Best Possible Solution

Keep this open. Define a Gateway HTTP contract for actual runtime provider/model metadata, thread selected provider/model from the fallback execution path into `src/gateway/openai-http.ts` and `src/gateway/openresponses-http.ts`, preserve the existing compatibility `model` field semantics, choose explicit non-streaming and SSE behavior, document the contract on the Gateway HTTP API docs, and add focused tests for both compatibility endpoints.

## Risks / Open Questions

- Streaming semantics are the main unresolved design point: current SSE paths flush headers before runtime model selection/fallback is observable, so satisfying the exact header request for streamed responses likely needs a deliberate contract such as a metadata event, trailer strategy, or delayed-start behavior.
- The OpenAI-compatible `model` response field currently has documented agent-target semantics. A fix should keep actual backend provider/model metadata separate to avoid breaking compatibility clients.
- Adjacent work such as #51441 and #7006 may overlap in metadata shape, so the Gateway HTTP contract should avoid inventing a one-off model identity format that conflicts with broader resolved-model observability.
