## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the Gateway OpenAI-compatible HTTP endpoints: the handlers continue to serialize the request-side agent target model, the selected-model callback remains below the HTTP layer, and streaming responses flush ordinary HTTP headers before fallback-selected model data is available. The prior maintainer review in the discussion also identifies this as a valid gap that needs API-contract judgment rather than cleanup closure.

## What This Changes

Request to add an HTTP response metadata surface, specifically `X-Actual-Model`, that exposes the actual backend model used for OpenAI-compatible Gateway responses.

## Best Possible Solution

Define and document an additive Gateway actual-model metadata contract across `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming behavior, then wire existing selected provider/model data from the model-selection path into that contract with focused tests.
