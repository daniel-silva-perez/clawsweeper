## Summary

Keep open. Current main still lacks the requested actual-runtime-model HTTP response metadata: the OpenAI-compatible Gateway handlers use the request-side agent target model in responses, while the actual selected-model callback remains below the HTTP layer. The remaining blocker is API-contract design, especially streaming behavior after headers have already flushed.

## What This Changes

The issue requests adding an `X-Actual-Model` response header to Gateway OpenAI-compatible HTTP responses so clients can see the model that actually processed a request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for both `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire selected provider/model data from `onModelSelected` or equivalent run metadata with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: inspect the current Gateway HTTP handlers and observe that no actual-model response header or SSE/body metadata is emitted, while response bodies and chunks use the request-side `model` value.

## Solution Assessment

Unclear. A header is plausible for non-streaming responses, but by itself it is not enough for streaming fallback cases because headers are flushed before late model selection; the maintainable fix needs an explicit streaming metadata contract.
