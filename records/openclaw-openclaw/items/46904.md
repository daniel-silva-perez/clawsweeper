## Summary

Keep this PR open. Current main still rejects `message` tool `action: "send"` calls when non-empty poll-looking params are present, so the proposed bug fix is still relevant; however the PR is stale/unmergeable against current main and overlaps with #46455, so the next step is maintainer review, rebase, and product choice rather than closure or a separate automated repair branch.

## What This Changes

The PR changes the message action runner to strip detected poll-creation fields from `action: "send"` params, adds a poll-param stripping helper, and updates focused tests to expect send actions with stray poll params to proceed.

## Best Possible Solution

Land one current-main implementation that strips stray poll-creation params only for `action: "send"`, preserves `action: "poll"` validation and plugin poll dispatch, handles the current shared/channel-specific detection rules including snake_case aliases, and resolves the overlap with #46455 in a single reviewed branch.

## Reproduction Assessment

Yes. Current main has a direct source guard and focused send-validation tests that reproduce the reported rejection for structured, string-encoded, snake_case, and negative-duration poll params on `action: "send"`; no live LLM run is required for a high-confidence code-path reproduction.

## Solution Assessment

Unclear. The PR is a narrow plausible fix for the poll-param slice of #41199, but it needs rebase against current poll-param refactors and maintainer choice versus #46455 before it can be called the best maintainable solution.
