## Summary

Keep this PR open. Current main still rejects `message` tool sends when real poll-creation params are present, and the shipped #52150 work only covers zero/default poll params. This PR remains a focused candidate for the unresolved message-tool part of #41199, but it needs maintainer review of the silent-stripping behavior and a current-main rebase.

## What This Changes

The PR changes the message action runner to strip poll-creation parameters from `action: "send"` calls, adds a poll-param stripping helper, and updates tests to expect sends with stray poll params to proceed.

## Best Possible Solution

Land a current-main implementation that strips stray poll-creation params only for `action: "send"`, preserves all `action: "poll"` validation and plugin poll dispatch behavior, and adds regression coverage for camelCase, snake_case, mixed, boolean/string, non-zero numeric, and channel-specific poll-field noise.
