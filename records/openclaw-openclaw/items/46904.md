## Summary

Keep this PR open. Current main and v2026.4.27 still reject `message` tool `action: "send"` calls when non-empty poll creation params are present, and the proposed behavior has not landed; the branch remains useful but needs maintainer review, a current-main rebase, and coordination with overlapping #46455.

## What This Changes

The PR strips poll-creation fields from `message` tool `action: "send"` params instead of throwing, adds a `stripPollCreationParams` helper, and updates focused tests to expect the send path to proceed.

## Best Possible Solution

Land a current-main implementation that strips stray poll-creation params only for `action: "send"`, preserves all `action: "poll"` validation and plugin poll dispatch behavior, remains symmetric with current poll-param detection including snake_case and channel-specific `poll*` keys, and coordinates the overlap with #46455.

## Reproduction Assessment

Yes. Current main has both a direct source guard and focused send-validation tests that reproduce the reported rejection for structured, string-encoded, snake_case, and negative-duration poll params on `action: "send"`; no live LLM run was needed for this read-only review.

## Solution Assessment

Unclear. The PR is a narrow plausible fix for the poll-param slice of #41199, but maintainers still need to choose between strip-all behavior here and the narrower overlapping policy in #46455, then rebase against current poll-param detection.
