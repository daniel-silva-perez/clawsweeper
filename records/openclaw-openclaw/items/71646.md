## Summary

Keep open. Current main and the latest tagged release still have the reported MCP channel bridge pending-map retention pattern, and the same-author closing PR #71648 is still open and unmerged, so this issue should wait for maintainer review or an equivalent landed fix.

## What This Changes

The issue asks `OpenClawChannelBridge` to bound or expire `pendingClaudePermissions` and `pendingApprovals`, clear them on close, and avoid unbounded growth in long-running `openclaw mcp serve` sessions.

## Best Possible Solution

Land #71648 or an equivalent maintainer-approved MCP bridge patch that clears both pending maps on close, prevents post-close writes, expires stale Claude permission and approval entries, preserves the MCP protocol surface, and resolves the cap question with either an included bound or a clearly tracked follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: unique Claude permission requests without matching replies grow `pendingClaudePermissions`, approval requested events remain in `pendingApprovals` unless a matching resolved event arrives, and `close()` does not clear either map.

## Solution Assessment

Yes for keeping this issue open. Because #71648 is an open PR with closing syntax for this issue, the narrow maintainable path is to review, land, rebase, or replace that implementation rather than closing the issue before a fix reaches main.
