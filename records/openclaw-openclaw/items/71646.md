## Summary

Keep open. Current main still has the reported MCP bridge retention pattern: both pending maps are unbounded, `close()` clears only waiters, Claude permission entries are removed only on a matching channel reply, and approval entries are removed only on a matching resolved event. The linked closing PR #71648 remains open, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests bounding and cleaning up `OpenClawChannelBridge` pending Claude permission and approval maps so long-running `openclaw mcp serve` processes do not retain stale entries indefinitely.

## Best Possible Solution

Land #71648 or an equivalent maintainer-approved MCP bridge patch that clears both pending maps on close, prevents post-close writes, expires stale Claude permission and approval entries, preserves the MCP protocol surface, and adds focused regression tests. If burst caps are outside that patch, track them explicitly as a follow-up before considering this fully resolved.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: repeated unique Claude permission requests without matching `yes/no <id>` replies grow `pendingClaudePermissions`, and approval requested events remain in `pendingApprovals` unless a matching resolved event arrives; `close()` does not clear either map.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable fix belongs in `src/mcp/channel-bridge.ts` with focused MCP tests, and the already-linked closing PR #71648 is the right implementation path to review or replace rather than queueing a second repair job.
