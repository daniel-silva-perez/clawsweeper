## Summary

Keep open. Current main still has the reported MCP channel-bridge leak pattern: both pending maps are unbounded instance Maps, are not cleared by `close()`, and only delete through narrow success paths; the paired closing PR #71648 is still the appropriate implementation path while it remains open.

## What This Changes

Request cleanup and bounding for `OpenClawChannelBridge` `pendingClaudePermissions` and `pendingApprovals` so long-running `openclaw mcp serve` processes do not retain stale pending entries indefinitely.

## Best Possible Solution

Review and land #71648 or an equivalent maintainer-approved MCP bridge patch that clears both pending maps on close, prevents post-close writes, expires stale Claude permission and approval entries, preserves the MCP notification/tool protocol, and adds targeted regression tests. If maintainers want burst hardening, add a cap in the same patch or track it as a narrow follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: repeated unique `handleClaudePermissionRequest()` calls without matching `yes/no <id>` replies grow `pendingClaudePermissions`, and `trackApproval()` entries remain until matching resolved events even though `expiresAtMs` is stored; `close()` does not clear either map.

## Solution Assessment

Yes for keeping this open. The narrow maintainable fix belongs in `src/mcp/channel-bridge.ts` with focused MCP tests; the existing open closing PR appears aligned with that direction, while cap policy can be decided by maintainers.
