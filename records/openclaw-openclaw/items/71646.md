## Summary

Keep open. Current `main` still retains `pendingClaudePermissions` and `pendingApprovals` without TTL, close-clear, or cap behavior, and the same-author closing PR #71648 is still open rather than merged.

## What This Changes

The issue requests bounding and cleanup for MCP channel bridge pending Claude permission and approval maps so long-running `openclaw mcp serve` processes do not retain stale entries indefinitely.

## Best Possible Solution

Land #71648 or an equivalent maintainer-approved MCP bridge patch that clears both pending maps on close, prevents post-close writes, expires stale Claude permission and approval entries, preserves the MCP protocol surface, and includes focused regression coverage; if burst caps are intentionally deferred, keep that as an explicit follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current `main`: repeated unique Claude permission requests without matching replies grow `pendingClaudePermissions`, and approval request events remain in `pendingApprovals` unless a matching resolved event arrives; `close()` does not clear either map.

## Solution Assessment

Yes for keeping this open. Because #71648 is an open closing PR for the same report, the narrow maintainable path is to review, land, or replace that PR rather than closing early or queuing duplicate automation.
