## Summary

Keep open. Current main and the latest tagged release still have the reported MCP channel bridge pending-map retention pattern, and #71648 is an open, unmerged closing PR for the same report.

## What This Changes

The issue requests TTL expiry, close-time cleanup, and a cap for `OpenClawChannelBridge` pending Claude permission and approval maps to prevent long-running `openclaw mcp serve` memory growth.

## Best Possible Solution

Land #71648 or an equivalent maintainer-approved MCP bridge patch that clears both pending maps on close, prevents post-close writes, expires stale Claude permission and approval entries, preserves the MCP protocol surface, and resolves the cap question either in the same patch or a clearly tracked follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: unique Claude permission requests without matching replies grow `pendingClaudePermissions`, approval requested events remain in `pendingApprovals` unless a matching resolved event arrives, and `close()` does not clear either map.

## Solution Assessment

Yes for keeping this issue open. Because #71648 is an open PR with closing syntax for this report, the narrow maintainable path is to review, land, rebase, or replace that implementation rather than closing the issue before a fix reaches main.
