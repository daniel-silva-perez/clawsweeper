## Summary

Keep this open. Current main still gates prompt text on `fullAccessAvailable`, but the main auto-reply and command-system-prompt paths still derive that value from elevated config and allowlist state when no explicit runtime signal is supplied, and the focused resolver test still asserts the optimistic fallback.

## What This Changes

Request a real provider/runtime signal for embedded-agent `/elevated full` availability, fail ambiguous availability closed, and add regression coverage across providers and prompt construction.

## Best Possible Solution

Add a generic runtime capability source for auto-approved full access before prompt construction, thread it through both auto-reply and command prompt paths, make ambiguous resolver input unavailable with a clear blocked reason, and update the existing resolver/system-prompt tests plus provider/runtime coverage so docs and parity claims match behavior.
