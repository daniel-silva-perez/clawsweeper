## Summary

Keep this open. Current main still derives `fullAccessAvailable` from elevated config and allowlist state when no explicit runtime/provider signal is supplied, so the prompt gate can still advertise `/elevated full` from an optimistic fallback rather than a proven capability.

## What This Changes

Request that elevated full-access prompt availability be driven by real provider/runtime capability signals, fail closed when ambiguous, and gain regression coverage across blocked runtime states.

## Best Possible Solution

Add a generic runtime capability source before prompt construction, thread it through the auto-reply and command prompt paths, make ambiguous availability unavailable with an explicit blocked reason, and update resolver, prompt, and provider/runtime coverage so elevated guidance matches actual execution capability.
