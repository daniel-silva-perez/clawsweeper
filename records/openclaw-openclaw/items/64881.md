## Summary

Keep open. Current main still has the reported source-level gap: ambiguous enabled-and-allowed elevated config resolves as full-access available, both prompt construction paths pass only config/allowlist state into that resolver, and the focused resolver test still asserts the optimistic fallback.

## What This Changes

The issue requests that elevated full-access prompt guidance be driven by an explicit runtime/provider capability signal, fail closed when ambiguous, and gain regression coverage.

## Best Possible Solution

Add a generic runtime capability source before prompt construction, thread it through both auto-reply and command-system-prompt paths, make ambiguous resolver input unavailable with an explicit blocked reason, and cover the resolver, system prompt, and provider/runtime matrix in focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call `resolveEmbeddedFullAccessState` with `{ enabled: true, allowed: true, defaultLevel: "full" }` and no `fullAccessAvailable`; current main returns available, and the existing test asserts that behavior.

## Solution Assessment

No, current main is not the best solution because it treats config permission as runtime capability. The maintainable fix should introduce or reuse a generic provider/runtime capability signal before changing the fallback semantics broadly.
