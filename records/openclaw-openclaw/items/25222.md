## Summary

Keep this issue open. Current main still serializes same-session contention by queueing, steering, dropping heartbeats, or waiting for the shutdown guard; it does not implement the requested immediate channel-visible busy/status reply with cancel/new-session guidance for ordinary messages sent while a session run is active. The latest provided maintainer/Codex discussion also explicitly kept this open, and the relevant source files have not changed since that review SHA.

## Best Possible Solution

Keep this issue open and implement a generic, configurable active-session busy-reply path around `resolveActiveRunQueueAction`, `resolvePreparedReplyQueueState`, and `runReplyAgent`. The path should return channel-safe status text/action guidance for ordinary same-session contention, optionally expose reason codes, and add focused queue-policy, runner, and dispatch/bridge regression coverage.
