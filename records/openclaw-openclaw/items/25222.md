## Summary

Keep this issue open. Current main still has queueing, `/stop`, `/new`, typing, and status-reaction-adjacent behavior, but ordinary same-session messages arriving during an active run still enqueue/serialize or wait rather than returning the requested immediate channel-visible busy/status reply with cancel/new-session guidance. The prior Codex review comment in the issue reached the same conclusion, and I did not find a newer main-branch implementation that changes that behavior.

## Best Possible Solution

Keep this issue open. The best implementation path is a generic active-session queue/runner change around `resolveActiveRunQueueAction`, `resolvePreparedReplyQueueState`, and `runReplyAgent` that can optionally return a configurable, channel-safe busy/status reply for ordinary messages arriving during an active run, including `/stop` and `/new` guidance where appropriate. Add focused runner tests plus at least one bridge-level regression test for Telegram same-topic serialization.
