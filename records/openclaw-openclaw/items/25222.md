## Summary

Keep this issue open. Current main still treats same-session active-run contention as queue/follow-up behavior, not as an immediate channel-visible busy/status reply with cancel or new-session guidance. The existing busy/status code is internal account/UI state or a narrow shutdown race message, and the reporter's Telegram-side ticket is useful platform context but does not implement the requested OpenClaw bridge behavior.

## Best Possible Solution

Keep this open and implement a generic active-session busy-reply path around queue policy, runReplyAgent, and dispatch. It should return configurable channel-safe status text when a same-session message is queued during an active run, include safe /stop and /new guidance where appropriate, optionally expose stable busy reason codes, and add focused tests for queue policy, runner return payloads, dispatch, and at least one channel path.
