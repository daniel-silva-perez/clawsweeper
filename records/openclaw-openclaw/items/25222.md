## Summary

Keep this issue open. Current main still serializes or queues same-session messages during an active run, and it has adjacent `/stop`, `/new`, internal busy state, and queue-mode support, but it does not send the requested immediate channel-visible busy/waiting reply with cancel/new-session guidance for ordinary follow-up messages. The reporter’s Telegram processing-indicator ticket and the prior Codex review comment remain useful context, but they do not change the remaining OpenClaw core UX gap.

## Best Possible Solution

Keep this issue open. The best fix belongs in the generic active-session queue/runner path around `resolveActiveRunQueueAction`, `resolvePreparedReplyQueueState`, and `runReplyAgent`: add a configurable channel-safe busy/status reply for ordinary messages that arrive during an active run, include optional `/stop` and `/new` guidance, preserve current queue modes, and add focused runner plus channel-level regression tests.
