## Summary

Keep open. Current main has meaningful partial fixes for session-corpus noise and cursoring, but the central request is still incomplete: daily/session ingestion still uses fixed source scores, session dedupe is still tied to timestamp or line position, and the scan budget still lacks a proven multi-file rotation guard.

## What This Changes

The issue requests memory-core dreaming to pre-filter and dedupe noisy session corpus entries, assign content- and recency-aware scores instead of flat source scores, and rotate scan budget so session corpus entries continue advancing as the corpus grows.

## Best Possible Solution

Keep the existing heartbeat/cron/sentinel filtering, then finish the memory-core owner-local path with deterministic content-level dedupe, a bounded source/content/recency scoring helper before `recordShortTermRecalls`, and regression coverage that proves many session files make progress under tight sweep budgets.

## Reproduction Assessment

Yes for the remaining source-level defects: current main directly shows fixed ingestion scores and timestamp-or-line-scoped session dedupe, and test search did not show multi-file rotation coverage. The exact production April 9 stall was not live-reproduced in this read-only pass.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if kept bounded inside memory-core ingestion/session export and tests. It should avoid unrelated memory search, Dream Diary output, or core plugin API changes.
