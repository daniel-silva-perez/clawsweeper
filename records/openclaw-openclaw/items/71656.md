## Summary

Keep open. Current main has partial fixes for session-corpus noise and checkpointing, but the main request is still not fully implemented: daily and session ingestion still use fixed source scores, repeated session content can still be distinct when it appears at different transcript positions, and the scan-budget coverage does not prove fairness across many large corpus files.

## What This Changes

Request to improve memory-core dreaming by filtering or deduplicating noisy session corpus entries, scoring ingested content by quality and recency, and rotating scan budget so session corpus entries do not stall behind other sources.

## Best Possible Solution

Keep the existing shipped heartbeat/cron/sentinel sanitization, then finish the memory-core ingestion layer with deterministic content-level dedupe, source/content/recency-aware scores before `recordShortTermRecalls`, and regression coverage for large multi-file session corpus rotation under tight budgets.

## Reproduction Assessment

Yes for the source-level defects: current main directly shows fixed daily/session scores and position-scoped session dedupe, and tests cover only a single large transcript drain. A live production corpus reproduction was not run in this read-only review, so the exact reported April 9 stall remains source-inferred rather than live-confirmed.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable path is to keep the existing filtering and add bounded ingestion scoring, content-level dedupe, and multi-file scan fairness tests inside `memory-core` rather than changing Dream Diary artifact behavior or unrelated memory search paths.
