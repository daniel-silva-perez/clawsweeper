## Summary

Keep this PR open for maintainer/author follow-up. The linked Windows CLI hang is a real, still-open problem, but this patch fixes it by forcing every completed launcher path through `process.exit(...)`, which introduces a concrete stdout/stderr truncation risk and bypasses the existing graceful-exit contract.

## What This Changes

The PR adds a final `process.exit(process.exitCode ?? 0)` to `openclaw.mjs` after the launcher help fast paths or built entry import path completes.

## Best Possible Solution

Fix the Windows lingering-process symptom at the leaking handle, launcher cleanup, or command-completion boundary while preserving `process.exitCode` and allowing stdout/stderr to drain; add proof for Windows `version`/`status`, piped or redirected output, and foreground commands that should intentionally stay alive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drain output before forcing launcher exit:** `openclaw.mjs:317`
  - body: Calling `process.exit(...)` at the launcher boundary applies to every completed CLI command. Node documents that this can terminate while stdout/stderr writes are still pending, so help, JSON, or other piped output can be truncated; use a narrower termination path or explicitly drain the streams before a hard exit.
  - confidence: 0.86
