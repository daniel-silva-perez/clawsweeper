## Summary

Current main does not implement the PR’s intended Windows process-exit change, but this active contributor PR should stay open because the patch adds a hard launcher-wide `process.exit()` that can truncate pending stdout/stderr writes under Node’s documented process I/O behavior. The next step is author or maintainer revision with Windows smoke proof, not closure or an automated replacement.

## What This Changes

The PR adds a final `process.exit(process.exitCode ?? 0)` to `openclaw.mjs` after launcher help handling or built entry import completes.

## Best Possible Solution

Preserve the Windows termination fix goal while avoiding a blanket hard exit: identify the handle keeping finite commands alive, clean it up at the owner boundary or add a narrow drained-exit helper, and prove `version`/`status`, piped or redirected output, and intentionally long-running foreground commands still behave correctly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drain output before forcing launcher exit:** `openclaw.mjs:317`
  - body: Calling `process.exit(...)` at the launcher boundary applies to every completed CLI command. Node documents that this can terminate while stdout/stderr writes are still pending, so help, JSON, or other piped output can be truncated; use a narrower termination path or explicitly drain the streams before a hard exit.
  - confidence: 0.86
