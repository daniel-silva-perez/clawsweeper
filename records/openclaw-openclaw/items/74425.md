## Summary

Keep this PR open, but do not merge it as-is. The patch addresses a real Windows CLI lifecycle report, yet the proposed global launcher `process.exit()` can cut off pending stdout/stderr writes after otherwise successful commands, which conflicts with Node's documented process-exit contract. citeturn1view1

## What This Changes

The PR adds a final `process.exit(process.exitCode ?? 0)` to `openclaw.mjs` after the launcher help fast paths or built entry import path completes.

## Best Possible Solution

Fix the Windows lingering-process symptom at the leaking handle, command cleanup, or a narrowly scoped launcher termination boundary while preserving `process.exitCode`, allowing stdout/stderr to drain, and proving that foreground commands still stay alive until stopped.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drain output before forcing launcher exit:** `openclaw.mjs:317`
  - body: This new hard exit runs after every completed launcher path. Node documents that `process.exit()` can terminate before pending stdout/stderr I/O drains, so help, JSON, or piped output can be truncated; drain streams before a hard exit or fix the leaking handle at a narrower boundary. citeturn1view1.
  - confidence: 0.86
