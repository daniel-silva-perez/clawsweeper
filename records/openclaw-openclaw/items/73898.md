## Summary

Keep open. Current main still lacks a default-off Control UI setting, persisted playback preference, attachment autoplay handler, or blocked-playback prompt for assistant TTS audio; the existing TTS backend already produces media, but the UI path remains manual or incomplete for rich audio blocks.

## What This Changes

Add an explicit Control app / Control UI option to auto-play newly generated assistant TTS audio after user opt-in, with graceful loading and blocked-autoplay handling.

## Best Possible Solution

Implement a browser-local, persisted, default-off assistant audio auto-play preference in the Control UI, wire it into chat rendering, auto-play only newly arrived assistant audio after the opt-in gesture, show a compact enable/playback-blocked prompt when play() rejects, and make rich audio blocks render through the same visible audio attachment path without changing backend TTS or channel delivery semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an assistant message with audio media reaches the Control UI renderer, but the renderer only emits a manual audio control and settings persistence has no autoplay preference; the reporter's direct-MP3 check further isolates the problem away from audio generation.

## Solution Assessment

Yes. A default-off browser-local Control UI setting with play() rejection handling is the narrow maintainable solution; backend TTS generation, provider selection, channel delivery, and realtime Talk should stay unchanged.
