## Summary

Current main still lacks the requested opt-in Control UI autoplay behavior. Backend TTS can generate assistant audio and the Gateway can persist webchat audio blocks, but the Control UI has no persisted playback preference, no autoplay/playback-consent path for assistant audio attachments, and its current audio attachment renderer is a manual `<audio controls preload="metadata">` element. The reporter's follow-up about missing visible playback also remains plausible because Gateway `type: "audio"` content blocks are produced, while the UI normalizer/render path only handles attachment-style audio and transcript `MediaPath` entries.

## Best Possible Solution

Keep this issue open and implement a focused Control UI enhancement: add a browser-local, persisted, default-off assistant audio autoplay setting; expose it using existing chat/settings controls; normalize/render Gateway `type: "audio"` content blocks as visible audio attachments; autoplay only newly arrived assistant audio after the opt-in gesture; show a compact prompt when `play()` is blocked; improve the audio loading state; and add targeted UI/storage tests plus a docs/changelog note.
