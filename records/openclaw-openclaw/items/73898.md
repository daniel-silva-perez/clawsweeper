## Summary

Keep open: current main still generates assistant TTS media, but the Control UI only renders assistant audio as a manual attachment and has no persisted playback opt-in or blocked-autoplay prompt. The related assistant-media duplicate-message report is distinct and does not cover this requested auto-play behavior.

## What This Changes

Add an explicit default-off Control app / Control UI option to automatically play newly generated assistant TTS audio attachments after user opt-in.

## Best Possible Solution

Implement a focused Control UI enhancement: persist a browser-local, default-off assistant audio auto-play preference, expose it with existing settings patterns, pass it into chat rendering, auto-play only newly arrived assistant audio attachments after opt-in, show a compact blocked-playback prompt when `play()` rejects, and leave backend TTS generation, channel delivery, and realtime Talk behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: auto-TTS adds `mediaUrl` to assistant replies, the Control UI normalizes that into an audio attachment, and the renderer outputs a manual `<audio controls preload="metadata">` element with no persisted auto-play setting or playback-consent path.

## Solution Assessment

Yes. The requested browser-local opt-in is the narrowest maintainable solution; changing backend TTS or channel semantics would not address browser autoplay policy and would risk surprising sound on other surfaces.
