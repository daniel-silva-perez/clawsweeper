## Summary

Current main still lacks the requested opt-in Control UI behavior. Auto-TTS already produces assistant audio media, but the browser Control UI renders it as a normal manual audio attachment, persisted UI settings have no playback preference, and there is no attachment autoplay/playback-consent path outside the separate realtime Talk implementation.

## Best Possible Solution

Keep this issue open and implement a focused Control UI enhancement: add a browser-local, persisted, default-off assistant audio auto-play setting; expose it with the existing settings UI patterns; pass it into chat rendering; auto-play only newly arrived assistant audio attachments after the opt-in gesture; show a compact blocked-playback prompt if `play()` rejects; and improve the audio attachment loading state without changing backend TTS or channel semantics.
