## Summary

Keep this issue open. Current main still documents the 1Password exec-provider example without disabling the 1Password CLI cache, and the exec resolver still passes only configured args/env without an automatic `OP_CACHE=false` override or a new `globalArgs`/`prependArgs` config surface. The manual workaround is expressible today, but the docs/defaults hazard remains actionable and is not superseded by broader external-secret batching work in #72278.

## Best Possible Solution

Keep the issue open. Update the public secrets docs to show the 1Password exec provider with `--cache=false` before `read` and/or `env: { OP_CACHE: "false" }` for LaunchAgent/headless setups, then consider a targeted doctor warning or explicit 1Password helper. Track the broader batching/model work separately in #72278.
