## Summary

Keep this issue open. Current main still exposes the same 1Password exec-provider hazard: the official 1Password example uses `op read` without `--cache=false` or `OP_CACHE=false`, and the exec resolver only passes the configured command args/env without any 1Password-specific cache default or helper. The existing generic `args` and `env` fields can express the workaround manually, but the docs/defaults problem remains actionable and is not fully superseded by the broader external-secret batching issue #72278.

## Best Possible Solution

Keep the issue open. Update the public secrets docs to show `--cache=false` before `read` and/or `env: { OP_CACHE: "false" }` for 1Password CLI exec providers in LaunchAgent/headless setups, then consider a narrow doctor warning or explicit 1Password helper. Coordinate separately with #72278 for the broader external-secret batching model.
