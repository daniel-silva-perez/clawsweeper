## Summary

Keep open. Current main still has the DeepSeek V4 disabled-thinking gap when `extra_body` overwrites the final OpenAI-completions payload, and this PR is a narrow functional fix with no blocking correctness or security concern. The only actionable issue I found is the missing required changelog entry for a user-facing fix.

## What This Changes

The PR deletes `reasoning_content` from OpenAI-completions messages after `extra_body` makes the final payload `thinking.type: "disabled"`, and adds a DeepSeek V4 regression test for that path.

## Best Possible Solution

Land the final-payload cleanup with regression coverage and a changelog entry, while keeping the behavior narrow enough that enabled-thinking replay and unrelated provider-specific reasoning formats are preserved.

## Reproduction Assessment

Yes. The code path is reproducible through the existing payload mutation harness: DeepSeek V4 can add or preserve `reasoning_content`, then `extra_body` can overwrite `thinking` to disabled on the final payload. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, with one mechanical release-note gap. Cleaning up after `extra_body` matches the final request state and the documented DeepSeek disabled-thinking contract; if maintainers reject the generic scope, the safer alternative is to guard the same cleanup to DeepSeek V4-compatible payloads.
