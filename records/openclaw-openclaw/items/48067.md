## Summary

Keep open. Current main has adjacent improvements for existing session rows, but the specific defaults fallback described here is still not resolved: TUI asks for sessions with an agent id, then applies `result.defaults` when the active session has no persisted row, while `listSessionsFromStore` still computes those defaults from global `agents.defaults.model` only. The closed related PR #48110 directly proposed passing `agentId` into `getSessionDefaults`, but that change is not present on main.

## Best Possible Solution

Keep this issue open and implement the narrow TUI/session-list fallback fix: make session-list defaults agent-aware when `opts.agentId` is supplied, or synthesize the active current session row with `buildGatewaySessionRow` for brand-new sessions. Add a regression test with `agents.defaults.model.primary = A`, `agents.list[].model.primary = B`, no persisted row for `agent:<id>:<session>`, and assert the TUI/session-list fallback reports B.

## Risks / Open Questions

- Existing session rows now resolve model identity with `sessionAgentId`, so the remaining fix should target only the fallback/defaults path and avoid regressing the selected-override behavior covered by the current session row test.
