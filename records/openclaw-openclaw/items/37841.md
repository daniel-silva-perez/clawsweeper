## Summary

Current main still has the reported media-only same-target suppression path. `buildReplyPayloads` and `resolveFollowupDeliveryPayloads` suppress all final/follow-up payloads based on same-target message-tool metadata without requiring that the message tool actually sent text, while the tool handler can commit target/media metadata without text. The linked fix attempts for this behavior were closed unmerged, and no open PR currently references this issue.

## Best Possible Solution

Patch `buildReplyPayloads` and `resolveFollowupDeliveryPayloads` so same-origin target suppression drops final/follow-up session text only when the message tool actually sent text to that same target. Keep media URL dedupe for already-sent media. Add focused regression tests for same-target WhatsApp media-only preservation, same-target text-send duplicate suppression, and the follow-up delivery path.

There is no gate requiring `messagingToolSentTexts.length > 0` before suppressing the final session text.
