## Summary

Keep open for maintainer review. Current main still has the reported fallback-load gap, while the PR’s short-circuit matches the documented global plugin-disable contract and I found no blocking functional or security finding in the proposed small diff.

## What This Changes

The PR short-circuits plugin capability provider resolution when `plugins.enabled: false` and updates the capability-provider fallback tests to assert disabled configs do not build bundled compat config or call the runtime loader.

## Best Possible Solution

Land a narrow capability-runtime fix that treats `plugins.enabled: false` as a hard stop before bundled capability compat or runtime snapshot loading, with regression coverage for both list and targeted provider fallback paths.

## Reproduction Assessment

Yes. A high-confidence reproduction is the current-main unit path where `resolvePluginCapabilityProviders` receives `cfg.plugins.enabled: false`, the compat helper rewrites the config to enabled, and a mocked fallback registry can return a bundled provider.

## Solution Assessment

Yes. The proposed short-circuit is the narrowest maintainable fix because it enforces the documented master-disable contract before discovery/load work, instead of trying to thread a disabled activation source through a compat helper that intentionally re-enables bundled ids.
