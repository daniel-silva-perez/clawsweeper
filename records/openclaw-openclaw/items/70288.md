## Summary

Keep open. Current main still returns `queuedFinal: false` when a plugin-bound `inbound_claim` handler returns handled, while Telegram still uses `!queuedFinal` plus no recorded delivery to send the silent-reply fallback. The linked issue and PR describe a real current-main behavior, and the latest release tag still contains the old branch. The PR is narrowly scoped and has no supply-chain-sensitive diff surface, but it needs rebase/update because current main has drifted and now includes an additional handled-with-reply expectation that still asserts the old result.

## Best Possible Solution

Keep this PR open as the implementation candidate for #70286. Rebase it onto current main, change the plugin-bound handled branch to return `queuedFinal: true`, update all handled plugin-bound expectations including the newer returned-reply case, and run targeted dispatch plus Telegram fallback tests before merge.
