Keep open. Current main does not implement the PR's intended change, and the latest PR patch still only flips the direct `dispatchReplyFromConfig` result/tests. The linked `openclaw-codex-app-server` reproduction is useful and matches the production Telegram path, but that path still goes through `dispatchInboundMessage` finalization, which can turn an async plugin claim back into `queuedFinal: false` when no final reply was enqueued.

Update this PR so a plugin-bound handled claim survives the production wrapper path without pretending a final reply was delivered. Preserve `queuedFinal: false` for missing plugin, no handler, declined, error, and cancelled-final cases; add integration coverage through `dispatchInboundMessage` or the provider dispatcher plus Telegram fallback coverage for the async no-reply claim; then merge it as the implementation for #70286.

## Clownfish Work Prompt

Implement a narrow fix for #70286, using this PR as context. Current main returns `queuedFinal: false` for plugin-bound `inbound_claim` handled results, and this PR flips only the direct `dispatchReplyFromConfig` return/tests. That is not enough because `dispatchInboundMessage` finalizes with `queuedFinal: result.queuedFinal && counts.final > 0`, so an async plugin claim with no immediate final reply can still reach Telegram as false and trigger the silent-reply fallback.

Expected fix boundary: update the core dispatch result/finalization path so a plugin-bound handled async claim suppresses channel fallback through `dispatchReplyWithBufferedBlockDispatcher` without counting a nonexistent final delivery. Preserve false results for missing plugin, no handler, declined, error, and cancelled final delivery. Prefer a typed internal result reason/metadata or another small explicit signal over broad channel-specific special cases. Do not add plugin ids, service-specific behavior, dependencies, CI changes, or public plugin SDK changes unless a typed additive seam is truly necessary.

Tests: keep/update the direct plugin-bound handled tests, add a `dispatchInboundMessage` or provider-dispatcher regression test for handled-with-zero-final-count, and add Telegram coverage showing the silent-reply fallback is not sent for a handled async no-reply plugin claim while the real no-visible-response fallback still fires. Changelog expectation: this is user-facing Telegram/plugin fallback behavior, so add a single-line active-version Fixes entry crediting the reporter if repo release policy requires it.
