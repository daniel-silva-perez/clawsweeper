## Summary

Keep this PR open. The reported bug is real on current main, but the PR’s current two-file patch likely does not fully fix the Telegram-observed path because `dispatchInboundMessage` normalizes `queuedFinal` back to `false` whenever the final count is zero. An async plugin claim with `{ handled: true }` normally has no in-cycle final delivery, so Telegram can still see `queuedFinal: false` and fire the silent-reply fallback unless the wrapper/finalizer path is updated or explicitly covered.

## Best Possible Solution

Keep this PR open as the implementation lane for #70286, but update it before merge so the plugin-bound handled signal survives the production wrapper path that Telegram actually calls. The fix should preserve false results for missing/no-handler/declined/error paths, avoid pretending a final was delivered when the plugin will respond asynchronously, and add an integration-level test through `dispatchInboundMessage` or `dispatchReplyWithBufferedBlockDispatcher` plus Telegram fallback coverage for the async no-reply claim case.
