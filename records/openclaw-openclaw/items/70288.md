## Summary

Keep this PR open. The reported Telegram/plugin-bound async fallback bug is real, and the PR changes the right direct branch, but the current two-file patch still does not preserve that signal through the production dispatcher wrapper that Telegram observes.

## What This Changes

The PR changes the plugin-bound handled branch in `dispatchReplyFromConfig` to return `queuedFinal: true` and updates three direct dispatch tests to expect that result.

## Best Possible Solution

Preserve an explicit plugin-claimed/response-committed signal through `dispatchInboundMessage` and the buffered dispatcher path without treating missing, no-handler, declined, or error outcomes as final responses. Add an integration-level wrapper test and Telegram fallback regression coverage for a handled async claim with no immediate reply.

## Reproduction Assessment

Yes. Current main plus the PR diff provide a high-confidence code reproduction: a handled plugin claim can return true from direct dispatch, but the wrapper returns false when `counts.final` remains zero, and Telegram’s fallback condition then fires when no delivery occurred.

## Solution Assessment

No. Returning `queuedFinal: true` in `dispatchReplyFromConfig` is necessary but not sufficient because the production wrapper still conflates committed async response with an enqueued final reply count; the fix needs to carry that distinction through the wrapper.

## Review Findings

Please carry the plugin-claimed response signal through the wrapper and cover that path.
