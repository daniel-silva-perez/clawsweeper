## Summary

Keep open. Current main still has the reported Volcengine/BytePlus Kimi catalog problem after the catalog moved from the old shared TypeScript helper into plugin manifests: the affected rows still cap `maxTokens` at `4096` and omit `reasoning`, which the manifest loader turns into `false`. Together still advertises Kimi K2.5 with `reasoning: true` and `maxTokens: 32768`; official BytePlus ModelArk docs also make the current 4096 cap look like the default response length rather than the configurable maximum for Kimi K2. citeturn1view0

## What This Changes

The issue requests correcting Volcengine/Doubao and BytePlus Kimi model metadata so reasoning capability and output-token limits match the provider’s real capabilities instead of the current 4096-token, non-reasoning catalog entries.

## Best Possible Solution

Keep this issue open and fix it in the provider-owned manifests with vendor-backed values for Volcengine and BytePlus, adding focused tests that assert the affected Kimi rows propagate through the provider builders and augmented catalog. Update provider docs only if corrected output-token or reasoning metadata becomes user-visible in model/status surfaces.
