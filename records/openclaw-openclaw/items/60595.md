## Summary

Close this PR as implemented on current main. The session-store hot read-modify-write paths now opt out of the defensive clone with `clone: false`, while ordinary reads keep the safer clone boundary; later main also removed the `structuredClone` clone path in favor of serialized JSON cloning.

## What This Changes

The PR returns the parsed session store directly for `loadSessionStore(..., { skipCache: true })` and adds a regression test that mutations of that fresh store do not affect later reads.

## Best Possible Solution

Keep the current main design: explicit `clone: false` for trusted mutating hot paths, defensive cloning for ordinary loads, and serialized JSON cloning instead of `structuredClone` for the remaining cache boundary.

## Reproduction Assessment

Yes. The original overhead is reproducible by source inspection of the old branch shape, and current main verifies the fix by routing the session-store update paths through `{ skipCache: true, clone: false }`; no runtime benchmark was run because this was a read-only review.

## Solution Assessment

Yes. The current implementation is the narrower maintainable solution because it avoids the redundant clone where the caller mutates and writes the fresh store, without making every `skipCache` read return a mutable direct object.
