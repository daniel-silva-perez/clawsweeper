## Summary

Keep this PR open. Current main has a `clone: false` escape hatch and uses it in the locked session-store update paths, but `loadSessionStore(..., { skipCache: true })` by itself still returns a `structuredClone(store)`. That means main does not definitely implement the PR's stated behavior, even though it partially addresses the hottest update path.

## Best Possible Solution

Review or adapt the PR against the current `LoadSessionStoreOptions` contract. Maintainers should decide whether `skipCache` should globally imply returning the fresh parsed object, or whether no-clone reads should remain explicit via `clone: false`; if accepted, keep focused coverage for the cache-aliasing invariant.
