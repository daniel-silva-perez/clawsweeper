## Summary

Keep open. Current main still has only a global stuck-session warning threshold, while cron job definitions, protocol validation, CLI/tool surfaces, and diagnostic session state have no per-job warning threshold, so the reporter’s slow-successful-cron false-positive path remains valid.

## What This Changes

The issue asks for cron jobs to declare an optional per-job stuck-session warning threshold so legitimately slow jobs do not emit false-positive `stuck session` warnings at the global 120s default.

## Best Possible Solution

Add an additive cron job warning-threshold field, validate it in cron normalization/protocol/tool/CLI paths with a bounded millisecond range, capture the effective value onto diagnostic session state when a cron run starts, and fall back to the current global/default threshold when absent. Keep stuck-session abort/recovery behavior separate from this configuration fix.

## Reproduction Assessment

Yes. A focused code reproduction is to start the diagnostic heartbeat with default config, put a cron session in `processing`, and let age exceed the 120s default; current main emits `session.stuck` because it compares every session against one global threshold. The reporter’s production logs and successful run durations provide matching live evidence.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it preserves the aggressive default for interactive/fast sessions while letting known slow cron jobs opt into a higher SLA without downstream log filtering. Raising the global threshold or relying on `timeoutSeconds` would either mask unrelated stuck sessions or change abort behavior rather than warning noise.

## Repair Work Prompt

Implement per-job configurable stuck-session warning thresholds for cron jobs. Current main only uses global `diagnostics.stuckSessionWarnMs`, which causes false-positive `stuck session` warnings for successful long-running cron jobs. Add an additive per-job field with millisecond units and bounded validation, preferably named consistently with the existing global warning knob unless maintainers direct otherwise. Wire it through cron normalization, persisted job types, Gateway protocol add/update/job schemas, the cron agent tool schema/flat recovery, CLI add/edit surfaces, and docs. Capture the effective threshold at cron run start on diagnostic session state so the heartbeat compares each processing session against its per-run threshold when present, otherwise falling back to `diagnostics.stuckSessionWarnMs` and then the 120000 ms default. Do not conflate this with stuck-session abort/recovery in #71127, and do not change `timeoutSeconds` semantics. Add/update focused tests for default behavior, per-job override behavior, range validation, protocol rejection/acceptance, CLI/tool passthrough, and documentation/changelog.
