## Summary

Keep open. Current main still applies one global stuck-session warning threshold to every processing session, while cron job definitions, gateway validation, agent-tool schema, CLI/docs, and diagnostic session state have no per-job warning-threshold field. The reporter's slow-but-successful cron false-positive path remains valid and is not solved by the existing cron `timeoutSeconds` abort behavior.

## What This Changes

The issue requests a per-cron-job configurable stuck-session warning threshold so legitimately slow cron jobs do not trigger the global 120s stuck-session log.

## Best Possible Solution

Add an additive bounded per-job warning-threshold field for cron jobs, validate it across persisted jobs, protocol, tool, and CLI paths, capture the effective threshold onto diagnostic session state when a cron run starts, and fall back to the existing global/default threshold when absent. Keep this separate from cron `timeoutSeconds` execution aborts and from the broader stuck-session recovery work tracked elsewhere.

## Reproduction Assessment

Yes. A focused code reproduction is to start the diagnostic heartbeat with default config, move a cron session into `processing`, and let its age exceed the 120s default; current main compares that session against the one global threshold and emits `session.stuck` even if the cron job later finishes successfully.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable direction because it preserves the aggressive global default for interactive sessions while letting known slow cron jobs declare their expected SLA. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated stuck sessions or change abort behavior.
