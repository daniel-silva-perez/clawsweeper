## Summary

Keep open. Current main still applies one global diagnostics stuck-session warning threshold to every processing session, and the cron job schema/tool/CLI/docs surfaces do not provide a per-job threshold, so the reported cron false-positive control remains unimplemented.

## What This Changes

Request an optional per-cron-job stuck-session warning threshold so legitimately slow cron jobs can avoid false-positive 120s stuck-session warnings without weakening interactive-session detection.

## Best Possible Solution

Add an additive, bounded cron job stuck-warning threshold, persist/validate it through cron normalization, protocol, tool, CLI, and docs surfaces, capture the effective threshold into diagnostic session state when a cron run starts, and have the heartbeat fall back to the current global/default threshold when absent.

## Reproduction Assessment

Yes. A focused reproduction is to start the diagnostic heartbeat with default config, mark a cron-backed diagnostic session as `processing`, advance past 120s, and observe that current main compares it to the single global threshold; the reporter's successful 130-210s cron runs match that code path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable direction because it keeps the aggressive default for interactive sessions while allowing known slow cron jobs to declare their expected duration; raising the global threshold or reusing `timeoutSeconds` would conflate unrelated warning and abort semantics.
