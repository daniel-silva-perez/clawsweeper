## Summary

Keep open. Current main still uses one global `diagnostics.stuckSessionWarnMs` threshold for every processing session, while cron job schemas, tool/CLI surfaces, session diagnostic state, and docs have no per-job stuck-warning threshold. The adjacent stuck-session recovery/watchdog issues do not supersede this narrower cron false-positive control.

## What This Changes

Request to let cron job definitions configure an optional per-job stuck-session warning threshold so legitimately slow cron jobs do not trigger the default 120s stuck-session warnings.

## Best Possible Solution

Add an additive cron job stuck-warning threshold with bounded millisecond validation, persist it through cron normalization/protocol/tool/CLI surfaces, capture the effective value into diagnostic session state when a cron run starts, and have the heartbeat fall back to the current global/default threshold when absent. Update status/doctor/docs/tests so operators can see the active threshold without changing existing jobs.

## Reproduction Assessment

Yes. A focused reproduction is to start the diagnostic heartbeat with default config, mark a cron-backed session as `processing`, and advance beyond 120s; current main compares that session to the single global threshold and emits `session.stuck`. The reporter's successful 130-210s cron runs match that code path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it keeps the aggressive default for interactive sessions while allowing known slow cron jobs to declare their expected duration. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated stuck sessions or change abort semantics.
