## Summary

Keep open. Current main has a configurable global stuck-session warning threshold, but cron jobs still cannot declare a per-job threshold and the diagnostic heartbeat applies one resolved value to every processing session, so the reported false-positive path remains valid.

## What This Changes

Add an optional per-cron-job stuck-session warning threshold so legitimately slow scheduled jobs can avoid default 120s stuck-session warnings without raising the threshold for all sessions.

## Best Possible Solution

Introduce an additive cron job threshold field, validate it in the cron store/protocol/tool/CLI surfaces, capture the effective value on the diagnostic session when a cron run starts, and fall back to the existing global/default threshold when the field is absent.

## Reproduction Assessment

Yes. A code-level reproduction is clear: start the diagnostic heartbeat with the default config, mark a cron run session as `processing`, and let it exceed 120 seconds; the current heartbeat will emit `session.stuck` because it only compares against the global threshold. The reporter also provided production run durations and successful `status=ok` outcomes for slow cron jobs.

## Solution Assessment

Yes. A per-job threshold is the narrowest maintainable fix because it preserves the aggressive default for interactive sessions and fast jobs, while allowing known slow cron jobs to opt into a higher SLA without downstream log filtering.
