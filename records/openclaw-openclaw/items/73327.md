## Summary

Keep open. Current main still applies a single global stuck-session warning threshold to every processing session, and cron job definitions, protocol schemas, tool/CLI surfaces, and diagnostic session state still have no per-job warning threshold. The previous ClawSweeper review reached the same result; current main has added adjacent stuck-session recovery behavior, but not the requested cron-specific false-positive control.

## What This Changes

Allow cron job definitions to set an optional per-job stuck-session warning threshold so legitimate slow cron runs do not trigger the default 120s warning.

## Best Possible Solution

Add an additive cron job warning-threshold field with bounded millisecond validation, capture the effective value into diagnostic session state when a cron run starts, and have the heartbeat fall back to the current global/default threshold when absent. Keep stuck-session abort/recovery work, such as #71127, separate from this warning-threshold configuration.

## Reproduction Assessment

Yes. A focused code reproduction is to start the diagnostic heartbeat with default config, mark a cron session as `processing`, and advance past 120s; current main emits `session.stuck` because every session is compared to one global threshold. The reporter’s production logs and successful 130-210s cron runs match that path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it preserves the aggressive global default for interactive sessions while allowing known slow cron jobs to declare their expected duration. Raising the global threshold or relying on `timeoutSeconds` would either mask unrelated stuck sessions or change abort semantics.

Add an additive millisecond field for cron jobs, preferably named consistently with the existing diagnostics knob (`stuckSessionWarnMs`), validate it in the 1000..3600000 range, and preserve current behavior when the field is absent. Add focused tests for default behavior, custom 60s/300s thresholds, invalid range rejection, schema/tool/CLI acceptance, and current fallback behavior.
