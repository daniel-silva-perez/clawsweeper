## Summary

Keep open. Current main still applies one global stuck-session warning threshold to all processing sessions, while cron job types, protocol schemas, CLI/UI surfaces, and docs have no per-job warning threshold. The adjacent stuck-session recovery/watchdog issues are related but do not supersede this narrower cron false-positive control.

## What This Changes

The issue requests an optional per-cron-job stuck-session warning threshold so legitimate long-running cron jobs do not trigger the global 120-second stuck-session warning.

## Best Possible Solution

Add a bounded additive cron job stuck-warning threshold, preserve the existing global/default behavior when absent, capture the effective threshold into diagnostic session state at cron run start, and surface it through cron schema/tool/CLI/UI/status/docs without changing cron execution timeout or stuck-session recovery semantics.

## Reproduction Assessment

Yes. A focused reproduction is to start the diagnostic heartbeat with default config, mark a cron-backed session as `processing`, and advance beyond 120 seconds; current main compares that session to the single global `diagnostics.stuckSessionWarnMs` threshold. The reporter's successful 130-210 second cron runs match that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it keeps the aggressive default for interactive sessions while allowing known slow cron jobs to declare their expected warning window. Raising the global threshold or reusing `timeoutSeconds` would either mask unrelated sessions or change abort semantics.
