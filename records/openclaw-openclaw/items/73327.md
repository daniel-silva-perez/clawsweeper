## Summary

Keep open. Current main still has only a global stuck-session warning threshold and applies it uniformly to all processing sessions; cron jobs can set execution `timeoutSeconds`, but cannot declare a per-job stuck-warning threshold, so the reported false positives for legitimate slow cron runs remain actionable.

## What This Changes

The issue requests an additive per-cron-job stuck-session threshold so slow scheduled jobs can raise the warning window without changing the global default for other sessions.

## Best Possible Solution

Add a backward-compatible cron job threshold field, validate and persist it through cron normalization, protocol, tool, CLI and UI surfaces, capture the effective value on the diagnostic session when a cron run starts, and have the heartbeat use that per-session value with the current global/default threshold as fallback.

## Reproduction Assessment

Yes. A code-level reproduction is clear: create or simulate a cron-backed processing session that runs past the default 120 seconds while still healthy; current main compares it only against the one global `diagnostics.stuckSessionWarnMs` value and will emit `session.stuck`. The reporter also provided production durations and successful `status=ok` run outcomes for the slow jobs.

## Solution Assessment

Yes. A per-job threshold is the narrowest maintainable solution because it preserves the aggressive default for interactive sessions and fast jobs while letting known slow cron jobs opt into a higher SLA without downstream log filtering.
