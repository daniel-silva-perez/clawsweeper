## Summary

Keep this PR open. Current main covers the exec PATH fallback via the newer `host=auto` routing and covers the cron deadlock shape via `cron-nested`, but the browser diagnostics work is still not implemented in the current `extensions/browser` plugin layout. The PR also has unresolved security/responsiveness review concerns that should be addressed during a split or rebase.

## Best Possible Solution

Keep this PR open, but require a split or rebase. Drop the exec and cron changes already covered on current main, then port the still-missing browser layered status diagnostics and bounded HTML upstream-error summarization into `extensions/browser`. Resolve the two existing review comments, keep config-derived diagnostics redacted and timeout-bounded, and verify with focused browser client/server/CLI tests plus a security review for reflected error text and config exposure.

## Risks / Open Questions

- Merging the PR as-is would target stale root browser/CLI paths instead of the current `extensions/browser` plugin layout.
- The browser HTML error summarization in the PR touches untrusted upstream response text; the existing review comment about fallback length and tag stripping should be resolved before porting.
- The PR's browser status path adds a `config.get` call; a rebased implementation should keep it redacted and timeout-bounded so status does not hang or expose secrets.
- The exec and cron changes touch command execution and scheduling surfaces that current main now handles differently, so carrying those older hunks forward could create semantic drift.
