## Summary

Keep open. Current main still computes and uses the effective context-token cap internally, but the plugin hook contract and the `llm_output` / `model_call_*` emitters do not expose that resolved cap to hook consumers. The related #64260 is adjacent naming/context-window work, not a shipped replacement for this narrower effective-limit request.

## What This Changes

The issue requests exposing the resolved effective context-token limit from agent `contextTokens` configuration in LLM hook event context or payloads.

## Best Possible Solution

Add a backward-compatible hook contract field for the resolved effective context-token limit, preferably with source/reference metadata, and thread it through the embedded, model-call, and CLI/Codex hook paths with docs and focused tests.

## Reproduction Assessment

Yes. The source-level path is high-confidence: current tests prove the 20k effective cap from a 200k model window, while the hook types and emitters build payloads without exposing that value.

## Solution Assessment

Yes. The narrow maintainable fix is an additive plugin hook contract update fed from existing `ctxInfo.tokens`; plugin-side workarounds cannot recover this internal runtime value reliably.
