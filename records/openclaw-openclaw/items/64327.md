## Summary

Keep open. Current main still resolves the effective context-token cap internally, but the plugin hook contract and the embedded, CLI, Codex, and model-call hook emitters do not expose that resolved cap to hook consumers. The related #64260 is adjacent naming work for native context-window and max-token metadata, not a replacement for this narrower effective-limit request.

## What This Changes

The issue requests adding the agent's resolved effective context-token limit to LLM hook event context so plugins can calculate utilization against the runtime cap rather than only the model's native window.

## Best Possible Solution

Add a backwards-compatible plugin hook contract field for the resolved effective context-token budget, preferably with clear source/native-vs-effective naming, then populate it from the already resolved runtime budget in the hook emitters that can honestly provide it and document the new field.
