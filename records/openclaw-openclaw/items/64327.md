## Summary

Current main still does not expose the resolved effective context-token limit to plugin hook consumers. The runtime resolves and uses the effective budget internally, and final run metadata records it, but `llm_output`, `model_call_started`, and `model_call_ended` hook event/context types and emitters still only expose provider/model/run/session/usage metadata. The related #64260 remains open and adjacent, not a shipped replacement for this narrower effective-limit request.

## Best Possible Solution

Keep this issue open. The best fix is an additive plugin hook contract update that threads the resolved effective context-token limit from `resolveContextWindowInfo()` / `ctxInfo.tokens` into `llm_output` and sanitized `model_call_started` / `model_call_ended` payloads or context, then updates embedded, CLI, and Codex hook emitters, hook docs, and focused hook tests. Coordinate field names with #64260 so native context-window metadata and effective runtime caps are clearly distinct.
