## Summary

Keep open. Current main still resolves and uses the effective context-token cap internally, but the plugin hook contract and the embedded/model-call hook emitters do not expose that resolved cap to hook consumers. The related #64260 remains open and adjacent, not a replacement for this narrower effective-limit request.

## What This Changes

The issue requests adding the agent’s resolved effective context-token limit to LLM hook event context so plugins can calculate utilization against the runtime cap rather than only the model’s native window.

## Best Possible Solution

Add a backwards-compatible hook contract field for the resolved effective context-token limit, ideally with clear native-vs-effective naming and source metadata, then thread it through the embedded and sanitized model-call hook emitters with docs and focused tests.

Add a backwards-compatible field to the plugin hook contract that exposes the resolved effective context-token limit used by the runtime, and include enough naming/source clarity that plugins can distinguish it from a model's native context window. Thread the value from resolveContextWindowInfo()/ctxInfo.tokens or EmbeddedRunAttemptParams.contextTokenBudget into embedded llm_output and sanitized model_call_started/model_call_ended hook payloads or context. Keep the surface generic and do not add provider-specific logic. Update the CLI/Codex-style hook paths only when they can honestly provide the same field; otherwise preserve omission rather than inventing values. Update docs/plugins/hooks.md and focused hook tests, including a case where agents.defaults.contextTokens caps a larger model window.
