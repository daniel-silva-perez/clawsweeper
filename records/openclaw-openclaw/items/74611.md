## Summary

Close as implemented on current main. The reported Telegram placeholder leak from 2026.4.26 is now handled by a shared user-facing sanitizer used by final replies, partial/block replies, and tool-result delivery, and current main also has a lifecycle terminal backstop for the reported stall-like no-completion shape. The broader concern about arbitrary text between tool calls remains covered by open #25592, but the literal `[tool calls omitted]` Telegram symptom is fixed on main.

## What This Changes

The issue requests that Telegram stop showing the literal `[tool calls omitted]` placeholder after tool-call turns and provide a better outcome when the agent stalls.

## Best Possible Solution

Keep the internal replay placeholder for provider transcript validity, strip it at every user-facing delivery boundary, and leave the broader policy question about non-final tool-call text leakage to #25592.

## Reproduction Assessment

Partial. There is a high-confidence deterministic reproduction for the visible placeholder leak through `sanitizeUserFacingText` tests and delivery-path inspection, but the exact provider-specific Telegram stall was not live-reproduced.

## Solution Assessment

Yes. Stripping only standalone placeholder lines in the shared sanitizer is the narrowest maintainable fix because it protects all channels without changing provider replay repair, while the lifecycle backstop addresses missing terminal events without altering tool execution semantics.
