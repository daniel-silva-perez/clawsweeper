## Summary

Keep this PR open. Current main still has the old WhatsApp reconnect retry window and does not surface WhatsApp reply delivery failures as agent-visible system events. The PR is stale against the current plugin layout and needs review fixes, but its core reliability fix remains relevant.

## Best Possible Solution

Keep this PR open as a plausible WhatsApp reliability fix. Rebase it onto extensions/whatsapp, widen the current send retry helper/defaults to cover the reconnect window, wire dispatcher delivery errors through ReplyDispatcherOptions.onError and openclaw/plugin-sdk/system-event-runtime, scope events to the intended final delivery failures, handle event trust/sanitization, and add regression coverage for the 5-attempt/1000ms retry window plus the agent-visible failure signal.

## Risks / Open Questions

- The PR likely needs a non-trivial rebase because it modifies removed src/web/auto-reply paths instead of the current extensions/whatsapp plugin files.
- The Greptile review concern remains valid for any adapted implementation: delivery-failure system events should be explicitly scoped to the intended final reply failures, not accidental tool/block dispatcher errors.
- System events default to trusted unless trusted: false is passed, so an adapted event that includes raw WhatsApp error text or phone/JID targets should sanitize content or mark it untrusted.
- A wider retry window improves the reconnect gap but is not a durable outbound queue; reconnects longer than the retry window can still fail.
