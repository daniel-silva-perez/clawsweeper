## Summary

Keep this PR open. Current main still has the old WhatsApp retry window and does not surface WhatsApp final delivery failures to the agent. The PR needs rework for the current `extensions/whatsapp` plugin layout and the prior review concern about scoping events to final replies is still relevant, but the core reliability fix remains useful.

## Best Possible Solution

Keep this PR open as a plausible WhatsApp reliability fix, but rebase and adapt it to `extensions/whatsapp`. The best implementation should widen the current WhatsApp retry helper/defaults to 5 attempts with 1000ms linear backoff, wire `ReplyDispatcherOptions.onError` in the WhatsApp dispatch path, emit agent-visible system events only for intended final delivery failures, handle trust/sanitization for error and recipient text, and add regression coverage for both the retry window and failure signal.

## Risks / Open Questions

- The PR is stale against the current plugin layout: it edits old `src/web/auto-reply` paths, while current main owns this behavior under `extensions/whatsapp/src/auto-reply`.
- The Greptile review concern remains valid for an adapted implementation: delivery-failure system events should be scoped to intended final reply failures, not accidental tool/block delivery errors.
- System events default to trusted on main, so an adapted event that includes raw transport errors, phone numbers, or JIDs should sanitize the text and consider `trusted: false` where appropriate.
- A wider retry window mitigates common reconnect gaps but is not a durable outbound queue; reconnects beyond the retry window can still fail.
