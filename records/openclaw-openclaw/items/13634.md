## Summary

Keep open. Current main already covers the workspace-file half through shared Project Context/bootstrap injection, but the Slack account/global `systemPrompt` surface is still missing from the typed config, strict schema, runtime prompt resolution, and docs.

## What This Changes

The issue requests Slack parity with Telegram by adding a global Slack `systemPrompt` and automatic workspace-file context injection so operators do not have to repeat the same prompt in every Slack channel entry.

## Best Possible Solution

Keep workspace-file loading on the shared Project Context path, and add a documented Slack account/global prompt contract that validates in strict config, inherits through top-level-to-account resolution, stacks predictably with per-channel overrides, and reaches Slack channels, private channels, DMs, MPIM/group DMs, and native slash commands.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by configuring `channels.slack.systemPrompt` against current main: the strict Slack schema has no such account/global key, and the runtime prompt path only reads per-channel `systemPrompt`.

## Solution Assessment

Yes. The best fix is a narrow Slack config/schema/runtime/docs/test change; adding a separate Slack-specific workspace bootstrap injector would duplicate the generic Project Context mechanism that already covers workspace files.
