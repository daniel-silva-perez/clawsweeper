## Summary

Keep this issue open. Current main still forces Telegram DM draft lanes through `previewTransport: "message"`, while the lower draft stream still defaults `auto` to native `sendMessageDraft` for DM threads; the latest release tag has the same selector, and PR #47631 is an open implementation candidate rather than a merged fix.

## What This Changes

Telegram DM reply streaming should preserve the draft stream's auto/draft behavior instead of forcing DM answer previews to message transport, optionally adding a user-facing transport preference.

## Best Possible Solution

Keep this issue open and resolve it through PR #47631 or an equivalent Telegram plugin patch: let the DM answer lane pass `auto` so `createTelegramDraftStream` can choose native draft transport, keep or explicitly decide reasoning-lane transport, preserve materialization and duplicate-final guards, and update dispatcher tests plus Telegram docs/config only if a transport preference is added.
