## Summary

Keep this issue open. Current main still matches the reported regression: Telegram DM streaming lanes are created with `previewTransport: "message"`, which bypasses the lower draft stream's `auto` behavior that would prefer `sendMessageDraft` for DM threads. The issue comments correctly identify the transport-precedence problem. PR #47631 is an open implementation candidate, but it is not merged and current main is not fixed.

## Best Possible Solution

Keep the issue open and resolve it in the Telegram plugin dispatcher. The likely path is to land PR #47631 or an equivalent patch: let the answer lane pass `auto` so `createTelegramDraftStream` can choose native draft transport for DM threads, keep or explicitly decide the reasoning-lane transport, preserve duplicate-final/materialization guards, update the tests that currently assert forced `message`, and document any Telegram DM transport preference if a config knob is added.

## Risks / Open Questions

- The current forced message transport was added to avoid duplicate-final/materialization problems, so a fix should preserve duplicate-final safeguards while restoring DM answer draft transport.
- Answer and reasoning previews share the lane factory today; the fix should explicitly test the intended answer-lane `auto` behavior and reasoning-lane behavior so they do not regress each other.
