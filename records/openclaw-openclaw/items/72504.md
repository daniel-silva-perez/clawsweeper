## Summary

Keep this open. Current `main` still has the reported Feishu prompt-loss path, and open PR #72661 explicitly closes this issue, so the conservative outcome is to leave the issue open until that PR, or the smaller open alternative #72532, is reviewed and either merged or rejected.

## What This Changes

The issue reports that Feishu inbound mention normalization strips the receiving bot's own mention from multi-bot group prompts, causing agents to decide they were not addressed and return `NO_REPLY`.

## Best Possible Solution

Review and land one focused Feishu fix that preserves #35994 leading-address slash-command behavior while keeping inline or repeated self-mentions visible to the agent prompt, with regression coverage in `extensions/feishu/src/bot.stripBotMention.test.ts`; close this issue only after that fix merges.

## Reproduction Assessment

Yes. A source-level reproduction is to parse a Feishu group text event with `mentions[]` for Bot A and Bot B while `botOpenId` is Bot A; current `normalizeMentions()` removes Bot A's placeholder before `buildFeishuAgentBody()` builds the model-facing prompt.

## Solution Assessment

No for the issue's broad suggested removal of all `botStripId` behavior, because current code and changelog history show the strip also protects #35994 slash-command routing. The narrower open PR #72661 direction, strip only a leading addressing prefix and preserve inline self-mentions, appears to be the best maintainable fix path.
