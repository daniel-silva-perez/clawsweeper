## Summary

Keep this open. Current main still strips the receiving Feishu bot's own mention from normalized content, and the issue is already paired with open fix PRs, including #72661 with closing syntax and a narrower test-backed approach.

## What This Changes

The issue asks Feishu mention normalization to preserve the receiving bot's own `<at>` mention in multi-bot group messages so each bot can tell it was explicitly addressed instead of returning `NO_REPLY`.

## Best Possible Solution

Land or rework the #72661-style fix: strip only a leading addressing self-mention needed for command UX, preserve inline self `<at>` tags for model routing, and add regression tests for multi-bot group messages before closing this issue.

## Reproduction Assessment

Yes. A unit-level reproduction can be built through `parseFeishuMessageEvent()` with group text like `Hey @_bot_a @_bot_b please coordinate`, mentions for both bots, and `botOpenId` set to Bot A; current main will remove Bot A's own mention from `ctx.content`.

## Solution Assessment

Yes for the #72661 direction, not for removing all self-stripping outright. The narrow leading-prefix approach fits the existing #35994 slash-command tests while preserving the semantic inline self-mentions that this report needs.
