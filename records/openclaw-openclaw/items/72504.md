## Summary

Keep this open. Current main still removes every Feishu mention whose open_id matches the receiving bot before building the agent-facing prompt, and the latest release contains the same behavior. The issue also has open implementation candidates, including #72661 with closing syntax, so it should stay open until maintainers review, land, or reject that fix path.

## What This Changes

Request Feishu inbound mention normalization to preserve the receiving bot's own `<at>` mention in multi-bot group messages so each addressed bot can see it was mentioned.

## Best Possible Solution

Land a Feishu-owned fix that preserves self-mention context for non-leading or semantic bot mentions while keeping mention-prefixed slash commands normalized for command routing, with regression tests for both multi-bot groups and the existing slash-command case. Then close or supersede the extra open PR so there is one canonical implementation path.

## Reproduction Assessment

Yes. A unit-level Feishu event with text like `Hey @_bot_a @_bot_b please coordinate`, both mentions in `event.message.mentions`, and `botOpenId` set to Bot A will flow through `parseFeishuMessageEvent` and remove Bot A's own tag from `ctx.content` on current main.

## Solution Assessment

No, removing `botStripId` entirely is probably not the best narrow fix because the existing code and tests preserve #35994 slash-command behavior by stripping a leading bot address. The safer candidate is a narrower strip that only removes the leading addressing prefix and leaves later self-mentions visible to the agent.
