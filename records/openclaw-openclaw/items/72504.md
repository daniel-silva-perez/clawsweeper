## Summary

Keep open. Current main still passes the receiving bot id into Feishu mention normalization and strips every matching self-mention from the prompt body, while the linked open PR #72661 uses closing syntax and proposes a narrower fix for the same regression. Because that PR is still open, the issue should stay open until a fix merges.

## What This Changes

Request to preserve enough Feishu self-mention context so each bot addressed in a multi-bot group mention can respond instead of returning NO_REPLY.

## Best Possible Solution

Review and land a focused Feishu fix such as PR #72661 if it passes maintainer review and CI: strip only a leading addressing self-mention for command UX, preserve inline self-mentions as `<at>` tags for model routing, and keep targeted regression tests for multi-bot group messages.

## Reproduction Assessment

Yes for the code-level regression: construct a group `parseFeishuMessageEvent()` input with text like `Hey @_bot_a @_bot_b please coordinate` and mentions where `@_bot_a` has the current bot open id. Current main sets `mentionedBot=true` from metadata but strips the current bot's `<at>` tag from `ctx.content`; live confirmation of the final NO_REPLY outcome was not run in this read-only pass.

## Solution Assessment

Yes, the best fix direction appears to be the narrower leading-prefix strip proposed by PR #72661 rather than removing all stripping, because current comments and tests tie leading self-mention stripping to the #35994 slash-command UX. The issue itself should remain open until that or an equivalent fix merges.
