## Summary

Current main still has the reported Feishu behavior: the parser passes the receiving bot open id into mention normalization, and normalization removes every matching self-mention key before the model-facing prompt is built. Because open PR #72661 explicitly closes this issue, the conservative maintainer cleanup outcome is to keep this issue open until that PR is reviewed and merged or rejected.

## What This Changes

The issue reports that Feishu mention normalization removes the receiving bot's own `<at>` tag everywhere in the prompt body, causing multi-bot group mentions to route to `NO_REPLY`.

## Best Possible Solution

Land or reject the focused PR that strips only a leading Feishu bot addressing prefix while preserving inline self-mentions as `<at>` tags, then close this issue after the regression is covered in `bot.stripBotMention.test.ts`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: parse a Feishu group text event whose `mentions[]` contains the receiving bot and another bot, and `normalizeMentions()` will remove every occurrence of the receiving bot's mention key from `ctx.content` before `buildFeishuAgentBody()` creates the model-facing prompt.

## Solution Assessment

Yes, the linked leading-prefix direction appears narrower than removing self-mention stripping entirely because it preserves the documented #35994 slash-command behavior while fixing the inline multi-bot context loss. The issue itself should remain open until that implementation is merged or a maintainer chooses an alternative.
