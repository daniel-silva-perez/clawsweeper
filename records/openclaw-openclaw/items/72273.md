## Summary

Keep open. Current main still hardcodes the Discord startup READY timeout to 15s and runtime READY timeout to 30s, while the strict Discord config schema, generated metadata hints, and user docs expose adjacent timeout knobs but no READY timeout override. The report's exact log string is still produced by the current lifecycle path, so the request remains actionable in the Discord plugin surface.

## What This Changes

The issue requests configurable Discord gateway startup and runtime READY timeouts because the fixed 15s startup budget can be too short for large multi-account Discord deployments.

## Best Possible Solution

Add Discord plugin-owned startup and runtime READY timeout settings that inherit from `channels.discord` into `channels.discord.accounts.<id>`, optionally read env fallbacks only when config is unset, pass the resolved values into the lifecycle wait/watch paths, and document the distinction from EventQueue and gateway metadata timeouts.

## Reproduction Assessment

Yes. The deterministic reproduction path is the existing lifecycle fake-timer behavior: keep `gateway.isConnected` false past 15s or emit `Gateway websocket opened` without READY past 30s, and current main follows the reported timeout/error path.

## Solution Assessment

Yes, with a boundary adjustment. The maintainable fix is to keep this in the Discord plugin/config surface and update schema/types/docs/tests; adding a core `gateway.discord.*` setting would cross the owner boundary without evidence of a cross-plugin need.
