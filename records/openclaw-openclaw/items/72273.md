## Summary

Keep open. Current main still has fixed Discord gateway READY budgets of 15s for startup and 30s for runtime reconnects, the config schema/docs only expose the adjacent gateway metadata timeout, and the issue describes a plausible multi-account restart loop that matches current lifecycle behavior.

## What This Changes

The issue requests configurable Discord gateway startup and runtime READY timeouts instead of hardcoded 15s and 30s values.

## Best Possible Solution

Add Discord plugin-owned account-inheritable READY timeout settings under `channels.discord` for startup and runtime waits, with env fallbacks only when config is unset, schema/types/config metadata/docs coverage, and lifecycle tests proving custom values are honored. Keep this out of core `gateway.*` config unless multiple plugins need the same contract.

## Reproduction Assessment

Yes. Current fake-timer lifecycle tests can reproduce both mechanics: keep `gateway.isConnected` false past 15s for startup restart, or emit `Gateway websocket opened` without READY for 30s for runtime force-stop.

## Solution Assessment

Yes, with a scope correction. The maintainable fix is a Discord plugin/config change following existing account inheritance and timeout metadata patterns; editing bundled output or adding a core `gateway.discord.*` setting would be the wrong boundary.
