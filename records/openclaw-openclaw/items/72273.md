## Summary

Keep open. Current main and the latest release still hard-code the Discord startup/runtime READY waits at 15s/30s, while the Discord plugin already staggers multi-account startup and exposes only the adjacent gateway metadata timeout. The reported restart loop matches the current lifecycle code and fake-timer coverage, so this is a valid narrow Discord plugin repair rather than cleanup-close material.

## What This Changes

Make the Discord gateway startup and runtime READY timeouts configurable because the fixed 15s/30s budgets can be too short for staggered multi-account startup.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<accountId>`, with env fallbacks only when config is unset, schema/type/config metadata/docs coverage, and lifecycle tests proving custom startup and runtime values are honored. Keep this out of core `gateway.*` config unless a generic multi-plugin contract is needed.

## Reproduction Assessment

Yes. The current fake-timer tests already reproduce the startup READY timeout restart and runtime READY force-stop mechanics, and the multi-account channel test confirms staggered startup before provider launch.

## Solution Assessment

Yes, with a boundary correction. The narrow maintainable fix is a Discord plugin/config change following existing account inheritance and timeout metadata patterns; patching bundled output or adding a core `gateway.discord.*` setting would put owner-specific behavior in the wrong layer.

## Repair Work Prompt

Fix the Discord READY timeout configurability gap reported in https://github.com/openclaw/openclaw/issues/72273. Current behavior hard-codes startup READY wait to 15s and runtime READY wait to 30s in `extensions/discord/src/monitor/provider.lifecycle.ts`; multi-account startup in `extensions/discord/src/channel.ts` staggers later accounts, so the fixed startup wait can restart the gateway before later accounts finish. Add plugin-owned, account-inheritable config fields under `channels.discord` and `channels.discord.accounts.<accountId>` for startup and runtime READY waits, with env fallbacks only when config is unset. Follow the existing `gatewayInfoTimeoutMs` schema/types/config-ui/docs pattern, but do not add core `gateway.discord.*` config and do not edit bundled dist output. Wire the resolved values into `runDiscordGatewayLifecycle` and tests. Update docs and changelog if the behavior is user-facing; credit the issue reporter as `Thanks @sergionsantos` if adding a changelog entry. Related context: PR #59019 changed Discord gateway READY/reconnect supervision, so preserve Carbon-owned reconnect behavior and only make the OpenClaw supervision budgets configurable.
