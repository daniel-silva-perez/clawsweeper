## Summary

Keep open. Current main and the latest release still hardcode the Discord startup and runtime READY waits, while only the adjacent gateway metadata timeout is configurable; the report describes a plausible multi-account restart loop that matches the existing lifecycle timeout paths.

## What This Changes

The issue requests configurable Discord gateway startup and runtime READY timeouts because the fixed 15s/30s waits can restart multi-account Discord setups during staggered startup or reconnects.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` for startup and runtime waits, with env fallbacks only when config is unset, schema/types/config metadata/docs coverage, and lifecycle tests proving custom values are honored.

## Reproduction Assessment

Yes. Existing fake-timer lifecycle tests can reproduce the mechanics by keeping `gateway.isConnected` false past 15s for startup restart, and by emitting `Gateway websocket opened` without READY for 30s for the runtime force-stop path.

## Solution Assessment

Yes, with a scope correction. The narrow maintainable fix belongs in the Discord plugin/config surface and should thread resolved timeout values into `runDiscordGatewayLifecycle`; adding a core `gateway.discord.*` setting or editing bundled output would be the wrong boundary.
