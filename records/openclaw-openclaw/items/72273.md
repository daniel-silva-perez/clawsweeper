## Summary

Current main still hardcodes the Discord startup READY timeout to 15s and runtime READY timeout to 30s in the Discord plugin lifecycle. Multi-account startup is intentionally staggered by 10s per account, so the reporter's 9-account case can exceed the startup READY budget before later accounts even begin connecting. The existing Discord config schema/docs expose related EventQueue, inbound worker, and gateway metadata timeout knobs, but not these READY timeouts, and no exact open PR appears to implement the requested override.

## Best Possible Solution

Implement plugin-owned Discord READY timeout configurability on current main. Prefer top-level and per-account `channels.discord` settings with optional env fallback only when config is unset, pass the resolved values into the lifecycle wait/watch paths, update schema/types/UI hints/generated config metadata/docs, and cover both default and custom timeout behavior in focused Discord tests.
