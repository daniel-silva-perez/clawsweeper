## Summary

Keep open. Current main still has no upfront duration expectation in the classic `openclaw onboard` flow, while the flow can continue through channels, workspace/session setup, web search, skills, plugins, hooks, and daemon work. The request is narrow, current, unprotected by maintainer labels, and suitable for a focused UX/docs repair rather than closure.

## What This Changes

Request to show an upfront ETA or long-running setup warning when `openclaw onboard` starts.

## Best Possible Solution

Add a concise early note to the classic CLI wizard that sets expectations before the main prompts begin, explains that provider auth, channel setup, daemon install, skills, and optional dependencies can take a while, and mirrors the user-facing expectation in the onboarding docs.

## Repair Work Prompt

Fix the onboarding ETA expectation gap for this issue. Add conservative upfront copy to the classic `openclaw onboard` CLI wizard before the main prompt flow begins, likely near `runSetupWizard` after the intro and before/around risk acknowledgement. Do not hard-code a guaranteed 40-minute ETA unless there is measured data; instead set expectations that setup can take a while depending on provider auth, channel setup, daemon install, skills, optional dependencies, and network/platform conditions. Keep the copy concise and beginner-friendly. Update focused tests so the note is covered without making the flow brittle. Mirror the expectation in the public onboarding docs (`docs/start/wizard.md` and/or `docs/cli/onboard.md`, plus `docs/reference/wizard.md` only if the step reference changes). Add a user-facing changelog entry if runtime copy changes, crediting `Thanks @sanjarcode`. Do not change provider/channel setup behavior, install behavior, plugin APIs, or unrelated onboarding flow semantics.
