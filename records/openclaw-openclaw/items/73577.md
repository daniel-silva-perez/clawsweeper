## Summary

Keep open. Current main still restores invalid config from last-known-good without surfacing the original validation path/message in the successful recovery log or main-agent recovery notice; the related provider-schema PR is adjacent but does not resolve the operator-visible diagnostic gap.

## What This Changes

The issue asks OpenClaw to show the specific config validation path/schema reason when an invalid provider edit is restored from last-known-good instead of only logging a generic recovery message.

## Best Possible Solution

Carry the original invalid `snapshot.issues` through successful last-known-good recovery and surface a concise path/message in both reload and startup logs plus the main-agent recovery notice. Add regression tests for recovered invalid snapshots, and leave the credential-only provider schema question to #39766 or a separate schema-focused change.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the existing reload/startup tests create invalid snapshots and prove successful recovery only emits generic recovery output; the same path would handle the reported provider validation failure.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve the current last-known-good recovery behavior while passing the original validation issue summary into the recovery log/notice; changing provider schema alone would not solve the operator diagnostics bug.
