## Summary

Keep open. Current main still restores last-known-good config after invalid reload/startup snapshots without surfacing the original rejected path/message in the successful recovery log or prompt-facing recovery notice; the related provider-schema PR is adjacent but does not solve the diagnostic gap.

## What This Changes

The issue requests operator-visible validation diagnostics when a direct provider config edit is rejected and last-known-good config recovery restores the active file.

## Best Possible Solution

Preserve the original invalid snapshot issues through successful last-known-good recovery and emit a concise sanitized path/message summary in both reload and startup recovery logs plus the main-agent recovery notice, with regression coverage for recovered invalid snapshots.

## Reproduction Assessment

Yes. The current source-level path is high confidence: the reload and startup tests construct invalid snapshots with issue details, recover successfully, and assert only generic recovery output; no live gateway run was needed for this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to thread existing `snapshot.issues` into the successful recovery reporting paths; changing provider schema alone would only address one adjacent rejection case and would not solve opaque recovery diagnostics.
