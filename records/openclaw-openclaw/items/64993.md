## Summary

Keep open. Current main mitigates the exact owner-flag spoof by deriving `senderIsOwner` from distinct owner/non-owner bearer tokens, but the broader token-bound scope requested here is still missing because session key, account id, and message channel continue to come from mutable loopback MCP request headers; #67376 is the open same-author implementation path for that remaining work.

## What This Changes

Bind loopback MCP request scope to bearer-token identity instead of trusting mutable request headers for session, account, channel, and owner status.

## Best Possible Solution

Land a reviewed loopback MCP design where the gateway registers authoritative per-backend scope server-side, resolves all loopback scope from token identity, stops sending or accepting bundled scope headers, invalidates scoped tool-cache entries on token revocation, and keeps regression coverage for owner-header spoofing plus cross-session/account/channel isolation.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence for the remaining bug: send a valid non-owner loopback bearer with altered `x-session-key`, `x-openclaw-account-id`, or `x-openclaw-message-channel` headers and current main passes those header values into scoped tool resolution; the exact owner-flag spoof no longer reproduces because `senderIsOwner` comes from the matched bearer token.

## Solution Assessment

Yes, the requested token-bound server-side scope is the narrow maintainable direction. Current main is only a partial mitigation, and #67376 appears to target the right boundary but still needs maintainer/security review before this issue can close.
