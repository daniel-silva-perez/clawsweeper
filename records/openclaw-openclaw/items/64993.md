## Summary

Keep open. Current main mitigates the exact owner-flag header spoof by deriving `senderIsOwner` from separate owner/non-owner bearer tokens, but session key, account id, and message channel still come from mutable loopback MCP request headers, and the paired open implementation PR still needs maintainer/security review.

## What This Changes

The issue requests binding loopback MCP request scope to bearer-token identity instead of accepting session, account, channel, and owner scope from mutable request headers.

## Best Possible Solution

Land a reviewed loopback MCP design where the gateway records authoritative per-backend scope server-side, resolves all loopback scope from token identity, stops sending or accepting bundled backend scope headers, invalidates scoped tool-cache entries on token revocation, and keeps regression coverage for owner spoofing plus cross-session/account/channel isolation.
