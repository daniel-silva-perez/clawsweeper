## Summary

Keep open. Current main only mitigates the exact owner-flag spoof by deriving `senderIsOwner` from owner/non-owner bearer tokens; the broader token-bound scope requested here is still missing because session key, account id, and message channel continue to come from mutable loopback MCP request headers, and a same-author implementation PR is still open for maintainer review.

## What This Changes

The issue reports a high-severity loopback MCP authorization bug where a backend with the bearer token can spoof mutable HTTP scope headers, and requests binding the full loopback scope to token identity.

## Best Possible Solution

Land a reviewed loopback MCP design where the gateway records authoritative per-backend scope server-side, resolves loopback scope from token identity, stops sending or accepting scope headers for bundled backend MCP, invalidates scoped tool-cache entries on token revocation, and keeps regression coverage for owner-header spoofing plus cross-session/account/channel isolation. If maintainers intentionally accept the remaining header-derived non-owner scope, that decision should be explicit and #67376 should be closed or narrowed accordingly.
