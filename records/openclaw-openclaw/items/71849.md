## Summary

Keep open. Current main still does not implement the requested Voice Call realtime fast-context path: remembered-context turns still go through the full realtime consult tool, non-final transcript chunks are ignored before call-state processing, and the strict config/manifest/docs expose no `realtime.fastContext` surface. The maintainer discussion also explicitly says this remains a real gap rather than a duplicate of the merged realtime voice work.

## What This Changes

The issue requests an opt-in Voice Call realtime fast-context retrieval path for live phone calls before falling back to full `openclaw_agent_consult`.

## Best Possible Solution

Implement a default-off, plugin-owned `realtime.fastContext` mode in Voice Call that accumulates and normalizes partial user transcript chunks, detects context-seeking turns, performs hard-deadline memory/session search through the public memory-host seam, injects snippets or an explicit miss into the realtime session, resets for follow-up questions, and preserves full `openclaw_agent_consult` as fallback.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no fast-context config/docs/manifest surface, registers only the full consult handler for remembered-context turns, and discards non-final transcript chunks before call-state processing.

## Solution Assessment

Yes. The narrow maintainable solution is an opt-in Voice Call plugin path using existing memory-host search surfaces; moving this into core or replacing the full consult tool would be broader than the issue requires.
