## Summary

Keep open. Current main still lacks the requested Voice Call realtime fast-context path: remembered-context turns go through the full realtime consult tool, partial transcript chunks are discarded before call-state processing, and the strict config/manifest/docs expose no `realtime.fastContext` surface. The issue discussion also includes maintainer confirmation that this is a real current-main gap, not a duplicate of the existing realtime consult work.

## What This Changes

The issue requests an opt-in, low-latency Voice Call realtime context lookup for memory/session questions before falling back to full `openclaw_agent_consult`.

## Best Possible Solution

Implement a default-off, plugin-owned `realtime.fastContext` feature in Voice Call that accumulates and normalizes partial user transcript chunks, detects context-seeking turns, performs a bounded memory/session search through public plugin SDK memory-host seams, injects snippets or an explicit miss into the realtime session, resets for follow-up questions, and keeps full `openclaw_agent_consult` available as fallback.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: on current main, Voice Call realtime exposes only the full consult handler for remembered-context questions when `realtime.toolPolicy` is enabled, while non-final transcript chunks are ignored before call-state processing.

## Solution Assessment

Yes. The narrowest maintainable direction is an opt-in Voice Call plugin path using existing memory-host search surfaces; moving this into core or replacing the full consult tool would be broader than the issue requires.
