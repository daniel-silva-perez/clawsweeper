## Summary

Keep this issue open. The related pre-commit OOM report appears handled in current main, but this issue is specifically about `openclaw update --channel dev` reporting failure after preflight lint failures. Current main still has an update-runner status aggregation path that can treat earlier failed preflight candidates as blocking even after a later candidate passes, which matches the reported log shape.

## Best Possible Solution

Keep the issue open and fix the dev update preflight aggregation: failed lint/build/deps steps for candidates that were intentionally skipped should be recorded as diagnostic candidate failures, not blocking final failures once a later `selectedSha` passes. Add a regression test where the first candidate’s `preflight lint` exits nonzero or SIGKILL-like and the second candidate passes, and assert the update result is `ok` with the selected fallback SHA. Separately consider whether dev preflight lint should use a lower-memory profile or a clearer OOM error.
