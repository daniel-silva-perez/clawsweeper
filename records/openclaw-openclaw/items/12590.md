## Summary

Keep open. Current main and the latest stable tag still contain the reported counter synchronization path, and open PR #51421 already targets this bug with closing syntax, so this issue should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue reports that configured memoryFlush can run only on alternating eligible auto-compaction cycles because the runner records the post-increment compaction count as the flush marker.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction, updates the synchronized-counter test expectation, and adds regression coverage for consecutive eligible auto-compaction cycles.

## Reproduction Assessment

Yes. The static current-main path is high-confidence: after a flush-triggered compaction, the runner can persist the post-increment count, and the next eligible cycle sees equal counters and skips the flush.

## Solution Assessment

Yes. The narrow maintainable fix is to stop recording the post-increment nextCount as the flush marker, with regression coverage and a coordination check against #47143.
