## Summary

Keep open. Current main still persists the post-increment compaction count as the memory-flush marker while the flush gate skips equal counters, and the issue is already paired with open PR #51421 using closing syntax, so it should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue reports that `memoryFlush` fires on alternating eligible auto-compaction cycles instead of before every compaction cycle because `memoryFlushCompactionCount` is synchronized to the incremented `compactionCount`.

## Best Possible Solution

Keep this issue open until #51421 or an equivalent focused patch lands, preserving the pre-increment flush marker after flush-triggered compaction, updating the synchronized-counter test expectation, and adding coverage for consecutive eligible auto-compaction cycles.

## Reproduction Assessment

Yes. The static current-main path is high-confidence: after a flush-triggered compaction, the runner persists the post-increment count, and the next eligible cycle sees equal counters and skips the flush.

## Solution Assessment

Yes. The narrow maintainable solution is to stop recording the post-increment `nextCount` as the flush marker, with regression coverage for consecutive eligible compaction cycles and a coordination check against #47143.
