## Summary

Keep open. Current main still matches the reported counter synchronization path, and open PR #51421 already targets the focused fix with closing syntax, so this issue should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue reports that `memoryFlush` fires only on alternating eligible auto-compaction cycles because `memoryFlushCompactionCount` is persisted to the same post-increment value as `compactionCount`.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction, updates the synchronized-counter expectation, adds consecutive eligible auto-compaction coverage, and keeps the documented once-per-cycle behavior.
