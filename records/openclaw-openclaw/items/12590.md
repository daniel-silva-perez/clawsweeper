## Summary

Keep open. Current main still persists the post-increment compaction count as the memory-flush marker while the gate skips equal counters, and open PR #51421 already targets this exact bug with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

`memoryFlush` should run before every eligible auto-compaction cycle instead of being skipped after flush-triggered compaction synchronizes `memoryFlushCompactionCount` with `compactionCount`.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction, updates the synchronized-counter test expectation, adds consecutive eligible auto-compaction coverage, and preserves the documented once-per-cycle behavior.

## Reproduction Assessment

Yes. The static path on current main is high-confidence: after a flush-triggered compaction, the runner persists the post-increment count, and the next eligible cycle sees equal counters and skips the flush.

## Solution Assessment

Yes. A narrow runner/test fix that stops recording the post-increment count as the flush marker is the maintainable path, with a cross-check against #47143 before merge.
