## Summary

Keep this issue open. Current main still persists the post-increment compaction count as `memoryFlushCompactionCount`, while the flush gate skips when that marker equals `compactionCount`; the test suite still asserts that synchronized state. The provided GitHub context and `git ls-remote` also show an open closing PR, so the issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review PR #51421, or land an equivalent focused patch, so a flush-triggered compaction preserves the pre-increment flush marker instead of writing the post-increment `compactionCount` into `memoryFlushCompactionCount`. The fix should update the synchronized-counter expectation and add regression coverage for consecutive eligible auto-compaction cycles.
