## Summary

Keep open. Current main still has the WebChat history/reset races this PR targets, and the latest PR head is a draft with a concrete cleanup gap: synchronous WebChat resets clear visible messages but leave tool-stream artifacts until a later successful history reload or gateway event.

## What This Changes

The branch changes Control UI WebChat and Gateway chat-history/session projection code, related tests, and QA scenario timeouts to stabilize history refreshes, reset handling, streaming/tool rendering, and terminal error display.

## Best Possible Solution

Keep this PR as the implementation candidate, add the same resetToolStream cleanup to clearChatHistory before the synchronous reset path reloads history, cover reset-after-tool-output behavior when history is delayed or fails, then rerun the focused WebChat/Gateway checks on the exact head.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a WebChat /new or /reset after tool/stream output while chat.history or sessions.changed delivery is delayed or fails: clearChatHistory clears chatMessages, but buildChatItems still renders chatStreamSegments and chatToolMessages.

## Solution Assessment

No, not as-is. The PR is scoped to the right WebChat/Gateway surfaces, but the synchronous reset path should reuse the existing resetToolStream cleanup already used by PR-head sessions.changed reset handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear tool stream state during synchronous resets:** `ui/src/ui/app-chat.ts:649-655`
  - body: The new bare `/new` and `/reset` path clears visible chat state after `sessions.reset`, but leaves `chatStreamSegments`, `chatToolMessages`, `toolStreamById`, `toolStreamOrder`, and the sync timer untouched until history succeeds or a later gateway event arrives. Those artifacts render independently of `chatMessages`, so old tool/stream cards can remain visible after a reset when history or event delivery is delayed or fails. Call `resetToolStream` in this path before reloading history.
