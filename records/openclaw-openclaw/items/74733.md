## Summary

Keep open. Current main still has the WebChat history and stream-ordering problems this PR targets, but the latest PR head still introduces two blocking correctness issues around stream-prefix trimming and deferred final-message dedupe.

## What This Changes

The PR changes Gateway chat-history/session metadata and Control UI chat reconciliation/rendering to stabilize WebChat history reloads, stream/tool ordering, render keys, deferred finals, and visible error messages.

## Best Possible Solution

Revise the PR so live stream text has one clear representation contract across controller state, final fallback, and render-time chat item building, and preserve distinct deferred finals using stable run/message metadata or by avoiding text-only dedupe for that queue; then land after targeted WebChat and Gateway validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep stream-prefix trimming at one boundary:** `ui/src/ui/controllers/chat.ts:867`
  - body: `state.chatStream` is stored as an already-trimmed suffix, but `streamedAssistantText` and `buildChatItems` trim `chatStream` against the same `chatStreamSegments` again. If the post-tool suffix legitimately starts with the committed prefix text, the live bubble and fallback final text drop that leading text. Keep the stream cumulative until render/final time, or make downstream consumers treat it as suffix-only.
  - confidence: 0.9
- **[P2] Preserve distinct deferred final messages:** `ui/src/ui/controllers/chat.ts:312`
  - body: Deferred finals are queued through `appendUniqueDisplayMessages`, which ignores run/message ids and dedupes by role plus rendered text/content. Two unrelated background runs that both finish with the same visible text collapse before flush, so one visible result is lost. Preserve run/message identity when available, or avoid text-only dedupe for this queue.
  - confidence: 0.86
