## Summary

Keep open. Current main still has the WebChat history/reset races this PR is trying to fix, but the current PR head is not merge-ready because the new synchronous bare `/new` and `/reset` path still leaves live tool-stream artifacts uncleared until a later history reload or gateway event catches up.

## What This Changes

The PR changes Control UI WebChat and Gateway chat-history/session projection code, related tests, and QA scenario timeouts to stabilize history refreshes, reset handling, streaming/tool rendering, and terminal error display.

## Best Possible Solution

Keep this PR as the implementation candidate, add the same `resetToolStream` cleanup to `clearChatHistory` before the synchronous reset path reloads history, cover reset-after-tool-output behavior when history is delayed or fails, then rerun the focused WebChat/Gateway tests plus exact-head CI.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a WebChat reset after tool/stream output while `chat.history` or `sessions.changed` delivery is delayed or fails: `clearChatHistory` clears `chatMessages`, but `buildChatItems` still renders `chatStreamSegments` and `chatToolMessages`.

## Solution Assessment

No, not as-is. The PR is scoped to the right WebChat/Gateway surfaces, but the synchronous `/new` and `/reset` path should reuse the existing `resetToolStream` cleanup already used by the PR's `sessions.changed` reset handling.

## Review Findings

Overall correctness: patch is incorrect
