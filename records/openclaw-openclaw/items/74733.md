## Summary

Keep open. The PR targets real current-main WebChat ordering and reconciliation problems, but the latest head still leaves synchronous reset handling dependent on a successful history reload to clear tool-stream artifacts, so stale stream/tool cards can remain visible after bare `/new` or `/reset`.

## What This Changes

The branch changes Control UI WebChat and Gateway history projection to stabilize transcript timestamps and render keys, reconcile stale history loads, defer unrelated finals during active streams, reset active chat views after session resets, trim tool-boundary stream prefixes, and render assistant stream failures as visible sanitized errors.

## Best Possible Solution

Keep the PR as the implementation candidate, add the existing tool-stream reset cleanup to the synchronous reset path before reloading history, cover a reset-after-tool-output case where history is delayed or fails, then rerun the focused WebChat/Gateway tests and wait for exact-head CI to pass.

## Reproduction Assessment

Yes. Current main has source-level reproduction paths for stale `chat.history` replacing newer local state and unrelated finals appending during an active stream; the PR finding can be reproduced by resetting after a tool/streamed response while `chat.history` is delayed or fails.

## Solution Assessment

No, not as-is. The broad approach is scoped to the right Control UI/Gateway surfaces, but the synchronous reset path should use the same `resetToolStream` cleanup already used by the PR's `sessions.changed` reset path before history reload.
