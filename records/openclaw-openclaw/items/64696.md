## Summary

Keep open. Current main still lacks the requested memory-wiki bridge and lint durability repairs: link extraction still scans raw Markdown, broken-link lint still compares raw targets without page-relative resolution, stale-pages native links still use vault-root paths from inside `reports/`, bridge render fingerprints lack a renderer-contract token, raw reply tags remain inside fenced bridge content, and cached source-page skips still trust file existence. The earlier upstreaming PR #65197 was closed unmerged, and open PR #73722 is scoped to partial lint normalization rather than the bridge render/cache fixes.

## What This Changes

The issue requests maintained memory-wiki fixes for page-relative report links, reply-tag handling, bridge renderer invalidation, and malformed cached source-page repair.

## Best Possible Solution

Land an extension-owned memory-wiki repair with regression coverage that handles Markdown-aware or otherwise safe link extraction, page-relative native Markdown link resolution, report-local native links, bridge reply directive immunity, renderer-contract invalidation in bridge fingerprints, and source-page frontmatter validation before cached import skips.

## Reproduction Assessment

Yes. The source path gives a high-confidence reproduction without mutating the checkout: a native link emitted from `reports/stale-pages.md` is extracted as a raw target, bridge fenced content containing `[[reply_to_current]]` is still scanned, and lint compares those raw targets against only extensionless page paths and basenames.

## Solution Assessment

Yes, the memory-wiki plugin is the right fix boundary. The safer implementation should prefer a Markdown-aware extractor or equivalent fenced-content immunity, while coordinating the narrower lint resolver overlap with #73722.
