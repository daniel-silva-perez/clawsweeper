## Summary

Keep open. Current main still loads and reports only documented skill roots, while `openclaw skills check`, status reporting, and the existing security audit have no orphan `SKILL.md` discovery path for arbitrary home/workspace-adjacent folders. The related open PR #63312 addresses symlinked skills inside recognized `.agents/skills` roots, not orphan skill directories outside all load roots.

## What This Changes

Request to warn or diagnose when skill-like directories containing `SKILL.md` exist outside OpenClaw's documented skill roots and therefore are not loaded.

## Best Possible Solution

Add a bounded diagnostic that compares candidate `SKILL.md` directories against the documented load roots and reports folders that will not be loaded, preferably through `openclaw skills check`, doctor, or security audit after maintainers decide whether scanning is opt-in, configured, or limited to safe candidate roots. Update CLI/docs/tests so the canonical-root behavior and any orphan scan are explicit.

## Reproduction Assessment

Yes. A valid `$HOME/my-new-skill/SKILL.md` outside the documented roots will not be returned by `loadWorkspaceSkillEntries`, and `openclaw skills check` only renders that discovered entry set.

## Solution Assessment

Unclear. The requested broad home-directory or startup scan would catch the example, but a bounded explicit diagnostic is safer unless maintainers approve implicit filesystem scanning.
