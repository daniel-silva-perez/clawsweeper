## Summary

Keep open. Current main still loads and reports only bounded, documented skill roots; `openclaw skills check`, gateway skill status, and the existing security audit do not discover arbitrary valid `SKILL.md` directories outside those roots. The remaining decision is where to put a bounded diagnostic and what filesystem scan scope is acceptable.

## What This Changes

The issue asks OpenClaw to warn or diagnose valid skill directories created outside canonical skill roots and to steer scaffolding toward `<workspace>/skills/<name>`.

## Best Possible Solution

Add a bounded diagnostic that compares candidate `SKILL.md` directories against the documented load roots and reports folders that will not be loaded. The likely maintainable destination is `openclaw skills check`, doctor, or the security audit after maintainers decide whether scanning is opt-in, configured, or limited to safe candidate roots; docs and focused tests should make the canonical-root behavior explicit.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a valid `$HOME/my-new-skill/SKILL.md` outside the documented roots is not reachable from `loadWorkspaceSkillEntries`, so `skills list`, `skills check`, and gateway skill status cannot report it.

## Solution Assessment

Unclear. The requested broad home-directory or startup scan would catch the example, but a bounded explicit diagnostic is safer unless maintainers approve implicit filesystem scanning.
