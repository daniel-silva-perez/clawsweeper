## Summary

Keep open. Current main still lacks orphan-skill detection: the loader and docs define bounded skill roots, `openclaw skills check` reports only discovered skills from those roots, and the existing security audit checks symlink escapes inside workspace skills rather than stray skill-like folders elsewhere. The remaining design needs maintainer judgment because a default home-directory scan has privacy and performance tradeoffs.

## What This Changes

The issue requests warnings or diagnostics for valid skill directories created outside canonical OpenClaw skill load roots so users do not leave orphaned skills that OpenClaw never loads.

## Best Possible Solution

Add a bounded diagnostic that compares candidate `SKILL.md` directories against the documented load roots and reports folders that will not be loaded, preferably in `openclaw skills check`, doctor, or security audit after maintainers choose the scan scope. Update CLI/docs/tests to make the canonical-root behavior and any opt-in orphan scan explicit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a valid `$HOME/my-new-skill/SKILL.md` outside the documented roots is never included by `loadWorkspaceSkillEntries`, and `openclaw skills check` only reports that discovered set.

## Solution Assessment

Unclear. The requested broad home/startup scan would solve the example but may not be the narrowest maintainable default; an explicit, capped diagnostic is safer unless maintainers approve implicit filesystem scanning.
