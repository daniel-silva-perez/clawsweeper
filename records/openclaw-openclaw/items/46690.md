## Summary

Keep open. Current main still only loads and reports skills from documented, bounded roots, and the issue's orphan-skill diagnostic is not implemented in `skills list`, `skills check`, doctor/status, startup, or the workspace skill audit.

## What This Changes

The issue requests guardrails and diagnostics for valid `SKILL.md` folders created outside OpenClaw's documented skill roots so users can find non-loaded orphan skill directories.

## Best Possible Solution

Add a bounded diagnostic that compares candidate `SKILL.md` folders against the documented load roots and reports folders that will not be loaded. The most maintainable home is likely `openclaw skills check`, doctor, or the existing security audit, with docs and focused tests clarifying canonical-root behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a valid `$HOME/my-new-skill/SKILL.md` outside all documented/configured roots is unreachable from `loadWorkspaceSkillEntries`, so `skills list`, `skills check`, and current status/audit paths cannot report it.

## Solution Assessment

Unclear. The requested broad home-directory or startup scan would catch the example, but a bounded explicit diagnostic is safer unless maintainers approve implicit filesystem scanning.
