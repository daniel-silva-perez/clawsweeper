## Summary

Keep open. Current main partially hardens skill path handling, but it does not implement the reported orphan-skill guard: arbitrary skill-like directories under home are not detected by `openclaw skills check`, `skills.status`, `openclaw status all`, doctor workspace status, or the bundled skill-creator initializer. The related ClawSweeper matches are title-overlap false positives about other guard fixes, not a canonical duplicate for orphan skills.

## Best Possible Solution

Keep the issue open and implement a focused core diagnostic: add a reusable orphan-skill scanner that compares shallow candidate `SKILL.md` directories against workspace, project/personal agent, managed, bundled, plugin, and `skills.load.extraDirs` roots; surface it in `openclaw skills check` or doctor/security audit rather than every startup path; and update `skills/skill-creator/scripts/init_skill.py` plus docs so new skills default to or clearly warn toward `<workspace>/skills/<name>`.

## Risks / Open Questions

- A naive startup or status scan over all of `~/` could be slow, noisy, and privacy-sensitive; any implementation should be bounded and probably limited to explicit diagnostics such as `openclaw skills check` or `openclaw doctor/security audit`.
- `skills.load.extraDirs` makes some outside-workspace skill directories intentional, so orphan detection must compare against all configured and documented roots before warning.
