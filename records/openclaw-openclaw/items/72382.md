## Summary

Keep open. Current main still matches the reported installer failure mode: the Unix npm installer can warn that the user's original PATH lacks the npm global bin directory, skip verification by default, and still print the installed-successfully banner; the non-root smoke then repairs PATH after the installer exits, so it does not prove the reporter's next `openclaw` command works.

## What This Changes

The issue reports that the curl bash installer on Ubuntu 26.04 prints success after npm/PATH warnings, but `openclaw onboard` and `which openclaw` fail immediately afterward.

## Best Possible Solution

Make the Unix installer's normal success path line up with what the user can run next: either ensure the CLI is discoverable from the user's next shell without manual PATH repair, or make the missing-original-PATH case an action-required/non-success outcome with focused installer tests and matching troubleshooting docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run the npm-method installer with the npm global bin directory absent from the parent shell PATH and `OPENCLAW_VERIFY_INSTALL` unset; current main can warn about that PATH gap and still continue to a success banner.

## Solution Assessment

Yes. A narrow installer fix is the best direction; docs-only guidance or optional verification would not address the misleading default success path for new installs.
