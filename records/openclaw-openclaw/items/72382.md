## Summary

Keep open. Current main still matches the reported installer failure mode: the documented Unix curl installer can warn that the user's original PATH lacks the npm global bin directory, skip verification by default, and still print the installed-successfully banner; the non-root smoke then repairs PATH after the installer exits, so it does not prove the reporter's next `openclaw` command works.

## What This Changes

The issue asks the Unix curl installer to leave OpenClaw installed and immediately runnable after a reported success on Ubuntu 26.04.

## Best Possible Solution

Make the Unix installer's normal success path match what the user can run next: either ensure the CLI is discoverable from the user's next shell without manual PATH repair, or make the missing-original-PATH case an action-required/non-success outcome with focused installer tests and matching troubleshooting docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run the npm-method installer with the npm global bin directory absent from the parent shell PATH and `OPENCLAW_VERIFY_INSTALL` unset; current main can warn about that PATH gap and still continue to a success banner.

## Solution Assessment

Yes. A narrow installer fix in `scripts/install.sh` with targeted installer and smoke coverage is the best direction; docs-only guidance or optional verification would not address the misleading default success path for new installs.

## Repair Work Prompt

Fix the Unix npm installer behavior reported in https://github.com/openclaw/openclaw/issues/72382. The observable bug is that `curl -fsSL https://openclaw.ai/install.sh | bash` can warn that the user's original PATH lacks the npm global bin directory, skip verification by default, print `OpenClaw installed successfully`, and then the user's next `openclaw onboard` command is `command not found`. Keep the fix scoped to the installer success/PATH contract: either make the installed CLI discoverable for the user's next shell without manual PATH repair, or make this missing-original-PATH case an action-required/non-success result rather than a success banner. Add focused coverage around `VERIFY_INSTALL` default behavior, `warn_shell_path_missing_dir`, and the non-root smoke path so the smoke no longer masks the immediate shell gap by adding PATH only after installer exit. Check whether the `openclaw@latest` to `openclaw@next` fallback in the report is a separate npm/dist-tag issue before claiming it is fixed. Do not broaden this into Windows/MSYS/fish support or the separate stdin-truncation/hang report unless the same minimal patch genuinely covers it. Update troubleshooting docs only if behavior or required user action changes. Changelog is expected if the installer behavior changes user-facing output or success/failure semantics.
