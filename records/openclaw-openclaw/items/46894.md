## Summary

Keep open. Current main still routes Control UI `/model status` and `/model list` through the non-empty `/model <name>` path, which calls `sessions.patch`; open PR #47069 uses closing syntax for the same regression, with #46895 also targeting it, so the issue should remain open until maintainers land, replace, or close that fix path.

## What This Changes

The issue requests that Control UI `/model status` and `/model list` behave as informational model commands instead of treating `status` or `list` as model ids passed to `sessions.patch`.

## Best Possible Solution

Keep the issue open and land or replace a narrow Control UI fix that routes normalized informational arguments such as `status`, `list`, and any maintainer-approved alias through the model-info path, with focused executor tests proving those inputs do not call `sessions.patch` while real `/model <provider/model>` changes still work.

## Reproduction Assessment

Yes. The source-level reproduction is deterministic on current main: `/model status` parses as command `model` with args `status`, then `executeModel` calls `sessions.patch` for that non-empty arg, matching the reporter's gateway error logs.

## Solution Assessment

Yes. The best fix is localized to `ui/src/ui/chat/slash-command-executor.ts` plus its colocated tests; the shared directive path already handles `status` and `list`, so this does not need a new core protocol or plugin API.
