## Summary

Keep open. Current main implements the WebChat/Control UI upload transport for non-video files and can stage PDF/TXT-like files into the media path, but the issue also asks for Word document text extraction. Current main still only defaults input-file extraction to text/json/pdf, the bundled document-extract plugin advertises only PDF extraction, and the media-understanding file-block path treats Office/vendor MIME types as binary and skips them before extraction. Related #57612 is the clearest DOCX extraction candidate, but it remains open and does not yet close the WebChat media-understanding path or legacy `.doc` policy.

## Best Possible Solution

Keep this open and narrow the remaining work to document extraction policy: review or adapt #57612 for native DOCX extraction, wire DOCX through the WebChat/media-understanding file-block path if maintainers want automatic extraction from uploads, explicitly reject or defer legacy `.doc` unless a safe extractor is approved, and align tests/docs across Control UI uploads, Gateway staging, input-file extraction, media-understanding, and the document-extract plugin contract.

## Risks / Open Questions

- Closing now would overstate Word support: WebChat can upload `.doc`/`.docx`, but current main does not extract Word text into the prompt by default.
- Legacy `.doc` support is security-sensitive because OLE/Word binary files can be misclassified as text; related open work is currently about safely skipping or guarding that path, not extracting it.
- DOCX support has an open implementation candidate in #57612, but maintainers still need to decide how it should connect to WebChat media-understanding and whether legacy `.doc` remains unsupported.
