## Summary

Keep this PR open. It targets a real linked onboarding bug, but the current patch does not preserve the new boolean `cask` metadata through skill frontmatter parsing, so the CodexBar installer still reaches the command builder without `spec.cask`; it also needs release-note and public hook-type follow-through before merge.

## What This Changes

The PR adds brew-cask metadata support to skill installer types/commands/status, marks the bundled model-usage CodexBar installer as a cask, and switches the bundled summarize installer from a Homebrew tap to the `@steipete/summarize` node package.

## Best Possible Solution

Land a revised version that treats brew casks as first-class parsed skill install metadata from SKILL.md through command generation, status display, install-scan/hook contracts, focused tests, and changelog, while keeping the summarize installer on a validated current package source.

## Reproduction Assessment

Yes. The linked bug is reproducible from current-main source: the bundled CodexBar skill is a brew formula install and `buildInstallCommand()` emits `brew install <formula>`; the PR’s added `"cask": true` is still dropped by `parseInstallSpec()` before command generation.

## Solution Assessment

No. The direction is maintainable, but the patch is incomplete because the new field is not parsed end to end and the release/public-contract follow-through is missing; the safer fix is to normalize `cask?: boolean` at the frontmatter boundary and cover it with focused tests.
