## Summary

Keep open. Current main still has the unguarded iMessage RPC stdin write path the report describes, and the adjacent Codex app-server stdio client already has the missing stdin error/write-callback hardening, so this is a valid narrow reliability bug rather than stale cleanup.

## What This Changes

The issue requests hardening `IMessageRpcClient.request()` so broken child stdin writes reject the awaiting request instead of surfacing as uncaught exceptions that can crash the gateway.

## Best Possible Solution

Patch the iMessage plugin-owned RPC client to treat child stdin write failures as transport shutdown: reject and clean up the affected pending request on write-callback errors, route stdin `error` through `failAll()`, and add focused unit coverage for broken-pipe/write-after-close behavior. Broader one-shot stdin write sites can be handled separately if they produce user-visible failures.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: current `IMessageRpcClient.request()` writes to a child stdin stream with no callback and the client has no stdin error listener, so a broken pipe can escape the request promise. A live `imsg` run was not needed to confirm the missing handling point.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `extensions/imessage/src/client.ts`, mirroring the existing Codex app-server stdio handling without moving iMessage-specific recovery policy into core.
