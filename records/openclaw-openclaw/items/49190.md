## Summary

Keep open. Current main still has cron run statuses of `ok`, `error`, and `skipped` only; the gateway protocol and Control UI also lack a `warning` state. The reported unresolved announce target path still returns `status: "error"` with `errorKind: "delivery-target"` unless best-effort delivery is explicitly enabled. Related #64413 is adjacent delivery-reason work, not a clear replacement for this issue's warning-state/UI request.

## Best Possible Solution

Implement this in the cron owner surface: add either a first-class `warning` run status or an explicit execution/delivery split across `src/cron/types.ts`, cron run persistence, gateway protocol schemas/validators, run-log filtering, and Control UI rendering. Classify successful execution plus failed delivery-target/direct delivery as warning, keep execution failures as error, do not increment execution error backoff for warning, and coordinate delivery reason codes with #64413.

## Risks / Open Questions

- Adding `warning` is a public protocol and UI change; schema validators, run-log filtering, persisted job state, and clients need additive compatibility handling.
- Implementation must preserve existing delivery diagnostic fields and best-effort behavior while ensuring delivery-only warnings do not increment `consecutiveErrors` or trigger execution failure alerts.
- #64413 may change delivery reason codes; coordinating the status split with that work would avoid two incompatible delivery semantics models.
