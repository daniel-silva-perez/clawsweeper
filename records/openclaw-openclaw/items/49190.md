## Summary

Keep open. Current main still has a three-state cron run model (`ok`, `error`, `skipped`) and no first-class `warning` status in the core type, gateway protocol, or Control UI. Delivery diagnostics are persisted separately, and best-effort delivery can keep some delivery failures from failing the job, but the reported non-best-effort unresolved announce target path still returns `status: "error"` with `errorKind: "delivery-target"`. The broader open #64413 is related, but it does not clearly supersede this issue's specific warning-state/UI request.

## Best Possible Solution

Implement this in the cron owner surface: extend the cron run status or add an execution/delivery split in `src/cron/types.ts`, gateway protocol schemas, run log filtering, and Control UI rendering; classify execution success plus delivery failure as warning, avoid incrementing `consecutiveErrors` for that state, and coordinate the delivery reason codes with the broader #64413 work.
