## Summary

Keep open. Current main still models cron run status as `ok | error | skipped`, the gateway protocol and Control UI do not accept or render `warning`, and the unresolved announce target path still returns `status: "error"` with `errorKind: "delivery-target"` unless best-effort delivery is enabled. Related #64413 is adjacent delivery-reason work, not a replacement for the warning-state/UI request.

## Best Possible Solution

Keep this issue open and implement the status split in the cron-owned surface: add either a first-class `warning` run status or explicit execution/delivery outcome fields across `src/cron/types.ts`, isolated delivery dispatch, `applyJobResult`, run-log persistence/filtering, gateway protocol schemas, and Control UI/i18n rendering. Successful execution plus failed delivery-target/direct delivery should surface as warning, execution failures should remain error, and delivery-only warnings should not increment execution-error backoff. Coordinate delivery reason codes with #64413.

## Risks / Open Questions

- Adding `warning` is a public cron protocol and Control UI change; schema validators, run-log filters, persisted state, generated clients, and UI labels need additive compatibility handling.
- The implementation must preserve existing `lastDeliveryStatus`, `lastDeliveryError`, and best-effort delivery behavior while ensuring delivery-only warnings do not increment `consecutiveErrors` or trigger execution failure alerts.
- Related #64413 is open for normalized delivery failure reasons; coordinating the status split with that issue would avoid two incompatible delivery semantics models.
