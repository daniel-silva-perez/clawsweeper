## Summary

Keep open. Current main still has no cron `warning` run status, and the unresolved announce-target and direct-delivery failure paths can still persist the whole run as `error` even when execution produced a summary; related #64413 is useful adjacent delivery-reason work but does not replace the requested status/UI split.

## What This Changes

Request to distinguish successful cron execution from failed announce delivery so Control UI and persisted run status can show warning instead of error.

## Best Possible Solution

Add an additive cron-owned warning outcome, or an equivalent execution/delivery split that serializes as warning for successful execution plus failed post-execution delivery. Keep delivery diagnostics (`lastDelivered`, `lastDeliveryStatus`, `lastDeliveryError`) visible, avoid incrementing execution-error backoff for warning runs, update gateway schemas and Control UI rendering/filters, and coordinate normalized delivery reason codes with #64413 without waiting on that broader issue.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: an isolated `agentTurn` cron job with requested announce delivery and no resolvable delivery target reaches `failDeliveryTarget`, which current main records as `status: "error"` even though the execution summary can exist.

## Solution Assessment

No. The current `ok | error | skipped` model plus separate delivery metadata is not the best solution because the primary persisted/UI status still conflates execution failure with notification failure; a narrow additive warning status or explicit execution/delivery status split is the maintainable fix.
