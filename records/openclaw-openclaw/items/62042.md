## Summary

Keep open. Current main still sends Slack user targets through `conversations.open` before any `chat.postMessage`, and the tests/docs codify that behavior. That is the same failure mode reported here: a Slack app can DM a user ID directly, but OpenClaw's proactive/direct delivery path can fail earlier while opening the DM conversation.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the bundled Slack plugin, probably around `extensions/slack/src/send.ts`: distinguish text/blocks user-target sends from file uploads, or add a narrow fallback when `conversations.open` fails with a scope error to post text directly to the user ID. Add regression tests for `user:<id>` text delivery with `conversations.open` failing, and keep media upload tests proving D-channel resolution still happens for files.
