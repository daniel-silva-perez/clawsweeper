## Summary

Keep open. Current main still routes Slack user-target sends through `conversations.open` before posting, which matches the reported missing-scope failure mode instead of the reported working `chat.postMessage(channel=<USER_ID>)` path.

## What This Changes

The issue requests that proactive Slack DM delivery through cron or `message.send` work for a Slack user ID when the same bot can send a direct `chat.postMessage` to that user ID.

## Best Possible Solution

Keep the upload/file path resolving a concrete Slack DM channel ID, but let text-only and block-only user-target sends post directly to the Slack user ID, with tests proving `conversations.open` is not required for the reported proactive DM case.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `client.conversations.open` to throw `missing_scope`, call `sendMessageSlack("user:U...", "text", ...)`, and observe current main failing before `chat.postMessage` is attempted.

## Solution Assessment

No. The current implementation uses one DM-channel resolution path for all Slack sends; the maintainable fix is narrower routing based on payload type so file uploads still get a D-channel while normal proactive DMs use the working user-ID post path.
