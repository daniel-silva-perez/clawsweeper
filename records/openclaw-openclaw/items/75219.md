## Summary

Keep open. Current main still has the provider transient retry gap, and this active external PR appears to be the right implementation vehicle for #60422 rather than a cleanup-close candidate. The latest head addresses the earlier AbortError timeout concern, so the remaining path is normal maintainer review and validation.

## What This Changes

The branch adds an opt-in same-key transient retry path to provider API-key execution, enables it for media audio/video and Google embeddings, and adds focused retry/rotation regression tests.

## Best Possible Solution

Land a narrow opt-in same-key transient retry path for idempotent provider operations, preserving 429 alternate-key rotation and fail-fast behavior for permanent/auth/caller-aborted errors, after maintainer review and validation pass.

## Reproduction Assessment

Yes. Against current main, call executeWithApiKeyRotation with one API key and an execute function that throws an HTTP 500-style Error or AbortError; the wrapper calls execute once and rethrows because the default predicate is rate-limit-only.

## Solution Assessment

Yes. Extending the existing API-key rotation wrapper with an explicit opt-in transient retry option is the narrow maintainable direction; the latest PR head also addresses the prior generic AbortError timeout concern with code and regression coverage.
