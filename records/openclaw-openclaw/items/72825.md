## Summary

Keep open. Current main still has the reported Copilot capability gap, and the PR is directionally useful, but the patch has a concrete correctness issue: the new session/context fallback uses the Anthropic 1,048,576-token constant for Copilot variants whose reported and tabled cap is 1,000,000 tokens.

## What This Changes

The PR adds GitHub Copilot known-capability overrides for Claude Opus 4.6/4.7 1m variants, enables xhigh for the 4.7 1m internal variant, adjusts context/session fallback behavior, and adds targeted tests.

## Best Possible Solution

Land one provider-owned Copilot capability fix that keeps runtime model metadata, status/session context, output caps, and thinking levels consistent with the exact Copilot /models limits; adjust this PR to use the 1,000,000-token Copilot cap in the context fallback or derive it from the known-capability table, then leave dynamic catalog discovery to a separate maintainer-approved follow-up.

## Reproduction Assessment

Yes. On current main, selecting github-copilot/claude-opus-4.7-1m-internal still reaches the synthetic Copilot model path with 128k/8k defaults and no xhigh entry; the PR diff reproduces the remaining review finding by returning 1,048,576 for a Copilot cap documented in the same PR as 1,000,000.

## Solution Assessment

No, not as currently patched. The narrow provider-owned override is a maintainable fix boundary, but the session/context path should use Copilot's exact cap instead of Anthropic's binary 1M constant before the PR is merge-ready.
