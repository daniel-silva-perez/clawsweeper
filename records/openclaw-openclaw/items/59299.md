## Summary

Keep open. The report is recent, has a concrete Windows AC-to-battery trigger, and current main still installs the native Windows Gateway service as a Scheduled Task without setting battery-safe Task Scheduler options. Microsoft Task Scheduler defaults can stop a task when the machine switches to battery and can prevent starts on battery, so the source still has a plausible unhandled crash/termination path despite the existing Windows restart hardening. citeturn1view0turn1view1

## What This Changes

The issue asks OpenClaw to keep the native Windows Gateway daemon running when a laptop switches from AC power to battery.

## Best Possible Solution

Add a native Windows Scheduled Task install/update repair that explicitly allows Gateway tasks to start and continue on battery power, with a doctor/status hint for existing tasks and targeted daemon tests proving the battery settings are applied without changing macOS, Linux, or Startup-folder fallback behavior.

## Reproduction Assessment

Yes. ## Solution Assessment

Yes, a focused Windows daemon fix is the best direction. The narrowest maintainable solution is to set `DisallowStartIfOnBatteries=false` and `StopIfGoingOnBatteries=false` for OpenClaw's Scheduled Task while preserving other task settings, rather than changing core Gateway shutdown or restart logic.

## Repair Work Prompt

Fix openclaw/openclaw#59299. Native Windows Gateway installs use Task Scheduler, and the Gateway can be stopped when a laptop switches from AC to battery because the task install path does not explicitly allow battery operation. In `src/daemon/schtasks.ts`, update the Windows Scheduled Task creation/update flow so the OpenClaw Gateway task has `DisallowStartIfOnBatteries=false` and `StopIfGoingOnBatteries=false` (PowerShell exposes this as `-AllowStartIfOnBatteries` and `-DontStopIfGoingOnBatteries`). Preserve existing task settings when updating/reinstalling; do not clobber unrelated Task Scheduler options. Keep macOS, Linux, and Startup-folder fallback behavior unchanged. Add focused unit coverage for task creation/update applying battery-safe settings and an actionable failure path if the settings update cannot be applied. Add a single-line changelog fix under the active version with `Thanks @zutony`. Do not touch generated baselines or unrelated daemon restart behavior. Validate with targeted daemon tests and formatting.
