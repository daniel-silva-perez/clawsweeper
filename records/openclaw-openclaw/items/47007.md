## Summary

Keep open. The intended Feishu subagent live streaming is not present on current main, but this PR is not mergeable as-is: the diff review confirms duplicate delivered-run streaming, Feishu thread target loss, a block-streaming regression, unthrottled card replacements, and a plugin-boundary type import.

## What This Changes

The PR adds a channel-level outbound dispatcher hook, wires Feishu outbound agent/subagent callbacks into Feishu streaming cards, adds Feishu block-streaming config, and changes streaming-card replacement/status rendering.

## Best Possible Solution

Revise the feature into a narrow, SDK-safe subagent streaming path that only attaches where no inbound dispatcher already exists, preserves replyToId/threadId/rootId routing end to end, coordinates with final delivery to avoid duplicate replies, keeps Feishu block streaming semantics backwards-compatible, coalesces replacement-card updates, and adds focused Feishu plus agent-command tests and plugin-contract docs.

## Reproduction Assessment

Yes. The blockers are reproducible by code inspection against the PR diff and current main: the PR creates a bridge for opts.deliver while current main still final-delivers outbound payloads, and the Feishu bridge reduces threadId to a boolean before the streaming session chooses its target.

## Solution Assessment

No. The proposed implementation is not the best solution as-is because it broadens a public channel contract and reuses Feishu reply streaming without preserving delivery and thread semantics; a subagent-only bridge integrated with the existing delivery route is safer.
