## Summary

Keep this PR open for maintainer review: current main does not implement the proposed Feishu native subagent streaming bridge, but the PR as proposed is not mergeable because it targets a now-refactored command path and introduces concrete Feishu delivery, routing, and plugin-boundary regressions.

## What This Changes

The PR adds a ChannelPlugin outbound dispatcher hook, wires Feishu agent-command output into Feishu streaming-card reply handling, and changes Feishu streaming-card UI/status plus block-streaming config behavior.

## Best Possible Solution

Revise this into a narrow, SDK-safe outbound streaming path for native subagent runs only: preserve replyToId/root/thread routing identifiers, coordinate with normal final delivery, keep block replies opt-in, coalesce card replacement updates, expose any new contract through the plugin SDK, and add focused Feishu plus agent-command regression coverage.

## Reproduction Assessment

Yes. The blocking regressions are reproducible by code inspection against the PR diff and current main: delivered runs would still use normal final delivery, Feishu thread-only routing loses the actual thread id, and unset blockStreaming changes existing block-chunk behavior.

## Solution Assessment

No. The proposed approach is not the best fix as-is because it broadens a plugin-facing channel seam and reuses Feishu dispatcher streaming before preserving routing and final-delivery semantics; a subagent-only bridge in the refactored agent-command path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit outbound streaming to subagent runs:** `src/commands/agent.ts:738`
  - body: The bridge is created for delivered main-agent runs as well as subagent runs. Those main runs still go through normal final outbound delivery, so Feishu can stream a card and then send the final reply again for the same turn.
  - confidence: 0.9
- **[P1] Preserve Feishu thread targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher reduces threadId to a boolean and drops the actual topic/root id. In thread-only contexts without replyToId, the streaming session has no concrete thread target and can fall back to a top-level card.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:313-314`
  - body: account.config?.blockStreaming !== false enables block delivery when the setting is unset. Current Feishu semantics drop internal block chunks unless they can be safely consumed by the streaming-card path, so default accounts can receive fragmented internal chunks.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: replaceContent bypasses the existing streaming-card update throttle while the new status/model/reasoning paths can call it repeatedly, including per-character model-name animation. This can burst Feishu CardKit updates and trigger rate or sequence failures.
  - confidence: 0.82
- **[P2] Use a public SDK type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports ModelSelectedContext through ../../../src/auto-reply/types.js, crossing the bundled plugin boundary. This type needs an SDK/local facade instead of a core src import.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI execution surface, but it introduces a concrete Feishu routing/privacy risk by losing thread targets for streamed output.

Concerns:

- **[medium] Preserve Feishu thread targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher converts threadId to Boolean(params.threadId) and does not pass the actual Feishu topic/root id to the streaming session. In topic or thread-only contexts, subagent output can be posted outside the intended thread and exposed to the broader chat.
