## Summary

Keep open. Current main still does not implement native Feishu subagent character streaming, but the proposed branch is not ready to merge because it adds a plugin-facing seam with concrete correctness and routing regressions already visible from the PR diff and current delivery contracts.

## What This Changes

The PR adds a `ChannelPlugin.createOutboundDispatcher` hook, wires Feishu outbound/subagent agent runs into reply-dispatcher streaming callbacks, adds Feishu block-streaming config, and changes streaming-card update behavior for richer live status rendering.

## Best Possible Solution

Revise the design into a narrow, SDK-safe outbound streaming path for native subagent runs only: preserve channel reply/thread identifiers, coordinate or suppress duplicate final delivery, keep Feishu block chunks backward-compatible, coalesce high-frequency card updates, and add focused Feishu plus agent-command regression coverage with public SDK/docs updates if the hook remains plugin-facing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit outbound streaming to subagent runs:** `src/commands/agent.ts:738-771`
  - body: The bridge is created for delivered main-agent runs as well as subagent runs. Main delivered runs still go through normal final outbound delivery, so Feishu can emit a streamed card and then send the final payload again for the same turn.
  - confidence: 0.9
- **[P1] Preserve the Feishu thread target:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher reduces `threadId` to a boolean and drops the actual thread/root id. In thread-only contexts without `replyToId`, the streaming session has no concrete topic target and can post subagent output as a top-level group message.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:313-314`
  - body: `account.config?.blockStreaming !== false` enables block chunks when the config is unset. Current Feishu behavior suppresses internal block payloads unless card streaming can consume them, so default accounts can receive fragmented internal block replies.
  - confidence: 0.86
- **[P2] Throttle replaceContent updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent()` bypasses the existing card update throttle while the new model/tool/reasoning paths can call it repeatedly, including per-character animation. That can burst Feishu CardKit updates and trigger rate-limit or sequencing failures.
  - confidence: 0.82
- **[P2] Use a public SDK type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: The Feishu plugin imports `ModelSelectedContext` via a relative path into core `src/**`. Extension production code should depend on exported plugin SDK/runtime facades, not private core source paths.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no third-party dependency execution, but it introduces a concrete Feishu routing/privacy risk by dropping the actual thread target for streamed output.

Concerns:

- **[medium] Preserve Feishu thread targets:** `extensions/feishu/src/channel.ts:496`
  - body: The proposed outbound dispatcher converts `threadId` to a boolean and does not pass the actual Feishu thread/root id to the streaming session.
