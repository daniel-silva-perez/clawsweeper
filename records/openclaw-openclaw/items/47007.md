## Summary

Keep this PR open for normal maintainer review rather than cleanup-closing it. Current main does not implement the proposed `createOutboundDispatcher` contract, and the PR diff still has blocking delivery/routing regressions already surfaced in review discussion, including duplicate Feishu delivery for main-agent runs and a Feishu thread-target privacy risk.

## What This Changes

The PR adds a `ChannelPlugin.createOutboundDispatcher` hook, wires Feishu outbound agent and subagent callbacks into Feishu streaming cards, adds Feishu `blockStreaming` config handling, and changes streaming-card replacement/status rendering.

## Best Possible Solution

Revise this into a narrow, SDK-safe native subagent streaming path that only attaches to subagent lanes without an inbound dispatcher, preserves Feishu `replyToId`/`threadId`/`rootId` routing, coordinates with final delivery to avoid duplicate messages, keeps block replies opt-in, coalesces replacement-card updates, and adds focused Feishu plus agent-command tests and contract docs for any new plugin API.

## Reproduction Assessment

Yes. The main blockers are reproducible by code inspection against the PR diff and current main: the PR bridge is created for delivered main-agent turns while current main still final-delivers payloads, and the Feishu dispatcher reduces `threadId` to a boolean before the streaming session chooses its message target.

## Solution Assessment

No. The proposed approach is not the best fix as-is because it broadens a public channel contract and reuses Feishu reply streaming without preserving delivery/thread semantics; a subagent-only bridge integrated with the current delivery path is the safer direction.

Overall confidence: 0.9

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: The new guard also creates the streaming bridge for ordinary `opts.deliver === true` main-agent turns. Those turns still go through final outbound delivery, so Feishu can show a streamed card and then send the final reply again for the same run.
  - confidence: 0.9
- **[P1] Preserve Feishu thread routing targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher converts `threadId` into `replyInThread: Boolean(params.threadId)` and drops the actual topic/root id. In thread-only contexts without `replyToId`, the streaming card has no concrete thread target and can be posted outside the originating topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-433`
  - body: `account.config?.blockStreaming !== false` enables block delivery when the setting is unset, reversing the existing Feishu behavior that suppresses internal block chunks unless the streaming-card path can safely consume them.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent` bypasses the existing card update throttle while the new status/model-name paths can call it repeatedly, including per-character animation. This can burst Feishu CardKit updates and trigger rate or sequence failures.
  - confidence: 0.82
- **[P2] Use an SDK type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports `ModelSelectedContext` through a relative `../../../src/**` path, crossing the bundled plugin boundary. The new plugin-facing contract should expose this through the SDK or a local runtime facade instead.
  - confidence: 0.78
