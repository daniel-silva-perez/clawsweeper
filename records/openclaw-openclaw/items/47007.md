## Summary

Keep this PR open because current main still lacks the native Feishu outbound/subagent streaming seam and the docs still describe native subagents as announce/final-delivery based. The proposed patch is not mergeable as-is: the reviewed diff has concrete duplicate-delivery, thread-routing, block-streaming default, and Feishu CardKit update-rate regressions.

## What This Changes

The PR adds a channel outbound dispatcher hook, wires Feishu delivered/subagent agent runs into reply-dispatcher-backed streaming cards, adjusts Feishu streaming-card replacement behavior and config, and threads streaming callbacks through the agent command path.

## Best Possible Solution

Revise the PR into a narrow, SDK-safe outbound/subagent streaming design: scope live streaming to native subagent runs that lack an inbound dispatcher, preserve Feishu reply/thread transport identifiers, avoid duplicate final delivery, coalesce or throttle card replacement updates, and add focused Feishu plus agent-command regression coverage and public SDK/docs updates if the hook remains plugin-facing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit outbound streaming to subagent runs:** `src/commands/agent.ts:738-771`
  - body: The bridge is created for `opts.deliver === true`, but delivered main-agent runs still finish through `deliverAgentCommandResult`. On Feishu that can emit a streamed card and then send the normal final payload for the same turn, so the bridge should be scoped to subagent runs or final delivery must be coordinated explicitly.
  - confidence: 0.9
- **[P1] Pass the Feishu thread id through:** `extensions/feishu/src/channel.ts:496`
  - body: The dispatcher converts `threadId` to `replyInThread: Boolean(...)` and drops the actual identifier. In thread-only contexts without `replyToId`, `FeishuStreamingSession.start` has no root/thread target and falls back to top-level message creation, so streamed subagent output can leave the originating topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:313-314`
  - body: `account.config?.blockStreaming !== false` enables block chunks whenever the config is unset. That changes the previous fallback from dropping block chunks when card streaming is unavailable to sending fragmented messages for default Feishu accounts.
  - confidence: 0.86
- **[P2] Throttle replaceContent updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: The new `replaceContent` path bypasses the 100 ms throttle used by `update()`, while the model-name animation calls it every 30 ms per character. That can burst Feishu CardKit updates and trigger sequence or rate-limit failures.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff does not add third-party code or dependency execution, but it changes Feishu message routing in a way that can expose thread-bound output outside the intended topic.

Concerns:

- **[medium] Preserve Feishu thread targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher drops the actual `threadId` and only passes a boolean thread flag, so a thread-only subagent stream can be posted as a top-level group message instead of the intended Feishu topic. That is a concrete routing/privacy regression for group chats.
  - confidence: 0.83
