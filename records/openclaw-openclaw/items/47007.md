## Summary

Keep this PR open for maintainer and author follow-up, but do not merge the current diff. The intended Feishu subagent streaming behavior is not implemented on current main, and the proposed patch still has concrete routing, duplicate-delivery, block-streaming, throttling, and plugin-boundary defects.

## What This Changes

The PR adds a plugin-facing outbound dispatcher hook, wires Feishu agent-command runs into the Feishu reply dispatcher for streaming cards, adds Feishu block-streaming config, and changes streaming-card replacement behavior for live status/reasoning/tool UI.

## Best Possible Solution

Revise this into a narrow SDK-safe outbound streaming path for native subagent runs only: preserve replyToId/thread/root routing identifiers, coordinate with normal final delivery, keep block replies explicitly opt-in, coalesce replacement-card updates, expose any new types through the plugin SDK, and add focused Feishu plus agent-command regression coverage.

## Reproduction Assessment

Yes. The blocking regressions are reproducible by code inspection: main delivered runs enter both the proposed bridge and normal deliverOutboundPayloads, thread-only Feishu routing loses the actual thread id, and unset blockStreaming flips from opt-in to enabled in the PR diff.

## Solution Assessment

No. The current proposed design is not the best fix because it broadens a plugin-facing seam and Feishu dispatcher reuse before preserving routing and delivery semantics; a subagent-only bridge with explicit route/final-delivery coordination is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit outbound streaming to subagent runs:** `src/commands/agent.ts:738`
  - body: The bridge is created for opts.deliver === true as well as subagent runs. Delivered main-agent runs still perform the normal final outbound send, so Feishu can stream one card and then send the final reply again for the same turn.
  - confidence: 0.9
- **[P1] Preserve Feishu thread targets:** `extensions/feishu/src/channel.ts:496`
  - body: Boolean(params.threadId) keeps only the fact that a thread exists and drops the actual topic/root id. In thread-only contexts with no replyToId, the streaming session has no concrete thread target and falls back to a top-level card.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:313-314`
  - body: account.config?.blockStreaming !== false enables block delivery when the setting is unset. Current channel semantics require explicit blockStreaming: true for non-Telegram block replies, so default Feishu accounts can receive fragmented internal block chunks.
  - confidence: 0.86
- **[P2] Throttle replaceContent updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: replaceContent bypasses the existing update throttle while the new model/tool/reasoning paths can call it repeatedly, including per-character model-name animation. This can burst Feishu CardKit updates and cause rate-limit or sequence errors.
  - confidence: 0.82
- **[P2] Use a public SDK type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports ModelSelectedContext through ../../../src/auto-reply/types.js. Bundled plugins follow third-party plugin boundaries, so this type should be exposed through an SDK/local facade instead of a core src import.
  - confidence: 0.78
