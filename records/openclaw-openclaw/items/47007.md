## Summary

Keep this PR open, but it is not merge-ready: current main does not implement the proposed native Feishu subagent character streaming bridge, and the PR diff still has concrete delivery, Feishu thread-routing/privacy, block-streaming, CardKit throttling, and plugin-boundary blockers.

## What This Changes

The PR adds a `ChannelPlugin.createOutboundDispatcher` hook, wires Feishu outbound agent/subagent output into Feishu streaming cards, adds Feishu `blockStreaming` config handling, and changes Feishu streaming-card replacement/status behavior.

## Best Possible Solution

Revise the design into a narrow, SDK-safe native subagent outbound streaming path that only runs for subagent lanes lacking an inbound dispatcher, preserves Feishu `replyToId`/`threadId`/`rootId` routing, coordinates with final delivery to avoid duplicates, keeps block replies opt-in, coalesces replacement-card updates, and documents/tests any new plugin contract.

## Reproduction Assessment

Yes. The main blockers are reproducible by code inspection of the PR diff against current main: delivered runs satisfy the new bridge guard while final delivery still runs, and Feishu `threadId` is reduced to a boolean before streaming starts.

## Solution Assessment

No. The proposed approach is not the best fix as-is because it broadens a public channel contract and reuses Feishu reply streaming without preserving delivery and thread-routing semantics; a subagent-only bridge in the current refactored command/delivery path is safer.

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: This guard also creates the streaming bridge for ordinary delivered main-agent turns. Those turns still go through the normal final delivery path, so Feishu can show a streamed card and then receive the final reply again for the same run.
  - confidence: 0.9
- **[P1] Preserve Feishu thread routing targets:** `extensions/feishu/src/channel.ts:496`
  - body: The dispatcher converts `threadId` into a boolean and drops the actual topic/root id. In thread-only contexts without `replyToId`, the streaming card has no concrete thread target and can be posted at the top level instead of the originating topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-433`
  - body: `account.config?.blockStreaming !== false` enables block delivery when the setting is unset, reversing the existing Feishu behavior that suppresses internal block chunks unless the streaming-card path can safely consume them.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent` bypasses the existing card update throttle while the new model/status/reasoning paths can call it repeatedly, including per-character model-name animation. This can burst Feishu CardKit updates and trigger rate or sequence failures.
  - confidence: 0.82
- **[P2] Use an SDK type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports `ModelSelectedContext` through a relative `../../../src/**` path, crossing the bundled plugin boundary. This type needs to come from an SDK or local facade before the plugin-facing contract expands.
  - confidence: 0.78
