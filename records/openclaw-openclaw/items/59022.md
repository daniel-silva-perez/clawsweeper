## Summary

Keep open. Current main has the Gateway A2UI route and scoped canvas auth, so the direct unauthenticated curl returning 401 is expected; the reported macOS node failure still maps to current source because `MacNodeRuntime` resolves A2UI through `GatewayConnection.shared` and never refreshes the node-scoped canvas capability that the macOS node session already holds.

## What This Changes

The issue requests that Gateway-hosted A2UI be reachable for macOS node `canvas.a2ui.*` commands instead of failing with `A2UI_HOST_UNAVAILABLE`.

## Best Possible Solution

Fix the macOS node A2UI path so `MacNodeRuntime` resolves A2UI from the active node `GatewayNodeSession` scoped canvas URL, refreshes that capability and retries when readiness fails, and preserves the existing Gateway behavior where direct unauthenticated `/__openclaw__/a2ui/` requests return 401.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: invoke `canvas.a2ui.push` on a macOS node after the shared operator canvas URL is missing or stale, and current `MacNodeRuntime.ensureA2UIHost()` will navigate once and return `A2UI_HOST_UNAVAILABLE` instead of refreshing the node session capability.

## Solution Assessment

Yes. The narrowest maintainable fix is in the macOS node runtime/coordinator boundary, reusing the existing `GatewayNodeSession.currentCanvasHostUrl()` and `refreshNodeCanvasCapability()` contract rather than broadening Gateway canvas auth or changing A2UI routing.
