## Summary

Keep open. Current main and the latest shipped tag still leave the bundled OpenRouter catalog entry as bare `auto`, while the pinned pi-ai catalog uses `openrouter/auto` and pi-coding-agent merges custom models by raw provider/id equality, so the reported canonical-id mismatch has not been fixed. The linked fix PR #62745 was closed unmerged, and current tests do not add the regression coverage that would prevent this from coming back.

## What This Changes

The issue asks OpenClaw to stop producing duplicate or unsafe OpenRouter auto model choices by making the bundled OpenRouter plugin use the canonical `openrouter/auto` model id instead of bare `auto`.

## Best Possible Solution

Update the bundled OpenRouter provider catalog so the auto row uses `openrouter/auto`, then add focused regression coverage proving the bundled auto entry does not create a second raw `auto` choice beside the pi-ai built-in. Leave broader provider-id normalization, OpenRouter vendor/model ids like `moonshotai/kimi-k2.6`, and media-understanding defaults alone unless a separate reproducer shows they are part of the same bug.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main still emits a bundled OpenRouter model row with id `auto`, while the pinned pi-ai catalog emits `openrouter/auto` and the pinned pi-coding-agent merge uses raw provider/id equality. A live picker run was not performed in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the bundled OpenRouter auto catalog id canonical at the source and add a regression test; changing broad core/model normalization is not necessary for this report and would carry more compatibility risk.
