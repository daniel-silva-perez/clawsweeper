## Summary

Keep open. Current main still registers the bundled OpenRouter auto catalog entry as bare `auto`, while the picker/catalog merge paths preserve raw model ids strongly enough for `auto` and `openrouter/auto` to remain distinct. The prior fix PR was closed unmerged, and the current tree lacks its regression test, so the issue remains a valid narrow bug rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to stop exposing a duplicate OpenRouter auto model by making the bundled OpenRouter catalog use canonical `openrouter/auto` instead of `auto`.

## Best Possible Solution

Update the bundled OpenRouter provider catalog so the auto entry uses `openrouter/auto`, then add focused regression coverage proving the bundled auto entry is canonical and does not create a second picker/catalog choice alongside the pi-ai built-in. Leave broader provider-id normalization and unrelated media-understanding defaults alone unless a separate reproducer shows they are part of the same failure.
