## Summary

Keep open. Current main still writes the Windows Startup-folder fallback as a `.cmd` launcher containing `start "" /min cmd.exe /d /c`, has no `wscript`/`.vbs` sidecar generation or cleanup, and the startup-fallback test still asserts the minimized-cmd launcher shape.

## What This Changes

Suppress the visible Windows login-time console flash for the Startup-folder gateway fallback by launching through a hidden mechanism while preserving the existing `OpenClaw Gateway.cmd` entry and cleanup behavior.

## Best Possible Solution

Implement the fix in the core Windows daemon Startup-folder fallback: preserve the existing `.cmd` entry name for service detection, delegate to a truly hidden launcher or equivalent no-window mechanism, remove any sidecar during uninstall, and update the focused fallback tests plus changelog.

## Reproduction Assessment

Yes. A source-level reproduction is clear: force the `schtasks` fallback path and inspect the generated Startup-folder script; current tests already do this and assert the minimized-cmd command that the report identifies as flashing.

## Solution Assessment

Yes, the requested boundary is the right one: the bug is in the core Windows daemon Startup-folder fallback, not in plugin code. The exact hidden-launcher mechanism should be implemented with careful quoting and sidecar cleanup, with tests enforcing that the login item no longer uses `start /min cmd.exe`.
