## Summary

Keep open. The prior ClawSweeper comment is still accurate on current main: the Windows Startup-folder fallback still generates an `OpenClaw Gateway.cmd` file that launches `gateway.cmd` through `start "" /min cmd.exe /d /c`, and there is no `.vbs`/`wscript` sidecar generation or cleanup. The related Microsoft tracker #74163 lists this as remaining Windows platform work, but it is not an implementation PR or superseding fix.

## What This Changes

The issue requests changing the Windows Startup-folder fallback so login startup launches the gateway without a visible `cmd.exe` flash, while preserving the existing `.cmd` entry path for service detection and adding sidecar cleanup if needed.

## Best Possible Solution

Implement the fix in the core Windows daemon Startup-folder fallback: keep the existing `OpenClaw Gateway.cmd` entry name for service detection, delegate login startup to a no-window launcher or equivalent hidden mechanism with robust quoting, remove any sidecar on uninstall, and update the focused fallback tests plus user-facing changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: force the `schtasks` fallback path and inspect the generated Startup-folder script; the current test already does this and asserts the minimized-cmd command that the report identifies as flashing.

## Solution Assessment

Yes, the requested boundary is the right one because the bug is in the core Windows daemon Startup-folder fallback. The exact `wscript` implementation should be reviewed for quoting and deprecation fallback, but replacing the login-item `start /min cmd.exe` handoff is the narrow maintainable fix.
