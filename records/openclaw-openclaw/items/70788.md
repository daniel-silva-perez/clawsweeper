## Summary

Keep this issue open. Current main still writes the Windows Startup-folder login item as a `.cmd` file that launches `gateway.cmd` through `start "" /min cmd.exe /d /c`, and there is no sidecar `.vbs`/`wscript` hidden launcher or cleanup path. The existing startup-fallback test still asserts the minimized-cmd launcher shape, so the reported login-time window flash remains plausible and actionable.

## Best Possible Solution

Keep this issue open and fix the core Windows daemon Startup-folder fallback: preserve the existing `OpenClaw Gateway.cmd` path/name for detection, launch the gateway through a truly hidden sidecar or equivalent no-window mechanism, remove any sidecar on uninstall, update startup-fallback tests, and add a user-facing changelog fix entry.

## Clownfish Work Prompt

Fix the Windows daemon Startup-folder fallback for this issue. Current main generates `%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\OpenClaw Gateway.cmd` with `start "" /min cmd.exe /d /c <gateway.cmd>`, which can flash a console window at login. In `src/daemon/schtasks.ts`, preserve the existing `.cmd` startup entry path/name for service detection, but make it delegate to a truly hidden launcher or equivalent no-window mechanism such as a sidecar `OpenClaw Gateway.vbs` using `wscript.exe //nologo` with window style 0. Write the sidecar during fallback install and remove it during uninstall. Preserve robust cmd/VBS quoting, line-break validation, and existing immediate launch/restart behavior. Update `src/daemon/schtasks.startup-fallback.test.ts` to assert the sidecar is written, the `.cmd` delegates to it, uninstall cleans it up, and existing fallback detection/restart behavior remains intact. Add a single-line user-facing changelog fix entry with proper attribution. Do not change unrelated Windows restart, update, exec-tool, or provider behavior unless a narrow helper reuse is required.
