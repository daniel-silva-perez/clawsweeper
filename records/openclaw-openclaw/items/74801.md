## Summary

Keep open. Current main has bounded file-log rotation, but the Linux systemd/journald storage concern is still not explicitly handled in code or docs: service output can still reach stdout/stderr, generated systemd units do not set stream or journal policy, and the Linux-facing docs show journalctl usage without retention guidance.

## What This Changes

The issue requests that OpenClaw address journald storage pressure for Linux systemd user services instead of only rotating the Gateway file log.

## Best Possible Solution

Keep this open until maintainers choose an explicit Linux service logging policy that covers both bounded OpenClaw file logs and bounded service-manager output, likely starting with journald retention docs or doctor/setup guidance before changing generated unit streams or vacuuming behavior by default.

## Reproduction Assessment

No. Static checks show a credible remaining path through systemd stdout/stderr into journald, but high-confidence reproduction would require running the current Gateway as a Linux systemd user service, generating logs, and comparing journalctl disk usage with the rolling file log.

## Solution Assessment

Unclear. The report lists several possible fixes, and the narrowest maintainable path appears to be docs or diagnostic guidance first; changing generated service streams or vacuuming journald needs explicit maintainer policy.
