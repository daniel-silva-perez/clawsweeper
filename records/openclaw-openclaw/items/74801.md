## Summary

Keep open. Current main no longer matches the exact cron/script details in the report because file logs now use an in-process rotating logger, but the systemd journald storage-pressure concern remains plausible and unaddressed: Linux services still emit console output to stdout/stderr, the generated systemd unit does not redirect or cap that stream, and docs/hints point operators to `journalctl` without retention guidance.

## What This Changes

Limit or document journald growth for Linux systemd Gateway services because OpenClaw file-log rotation does not cap journal storage.

## Best Possible Solution

Keep the issue open and choose an explicit Linux service logging policy: either document supported journald retention settings for systemd installs, add a safe setup/doctor recommendation, or change the generated service output behavior while preserving the existing rotating file logs and operator access to recent diagnostics.
