## Summary

Keep open. Current main no longer matches the exact old cron/script rotation details because file logs now use in-process rolling logs, but the Linux systemd journald storage-pressure concern remains unaddressed: service output still reaches stdout/stderr, the generated systemd unit has no journal retention or stream redirection policy, and docs/hints point users to journalctl without cap guidance.

## What This Changes

The issue requests that Linux systemd installs stop accumulating unbounded journald logs by adding journald cleanup, documenting journald retention settings, or changing gateway/service logging behavior.

## Best Possible Solution

Keep the issue open until a maintainer chooses an explicit Linux service logging policy: user-facing journald retention docs, a doctor/setup recommendation, generated systemd stream handling, or another bounded approach that preserves file logs and recent service diagnostics.

## Reproduction Assessment

No. Static source checks show systemd output can still reach journald and no retention guidance exists, but a high-confidence repro would run the current gateway as a Linux systemd user service, generate logs, and compare `journalctl --disk-usage` with the rolling file log behavior.

## Solution Assessment

Unclear. The report lists several possible fixes, but the safest path needs maintainer policy judgment; a docs/doctor recommendation is likely lower risk than silently vacuuming journald or changing service streams by default.
