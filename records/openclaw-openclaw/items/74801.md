## Summary

Keep open. Current main has moved past the exact cron/script details because file logs now use the in-process rotating logger, but the reported Linux systemd journald storage-pressure gap remains: service output still reaches stdout/stderr, the generated unit does not set output or journal retention policy, and docs point users to journalctl without retention guidance.

## What This Changes

The issue asks OpenClaw to address systemd journald growth for gateway logs instead of only relying on the file-log rotation path.

## Best Possible Solution

Define a Linux service logging policy that preserves OpenClaw's existing rotating file logs while giving systemd users a clear, supported way to bound journald growth, likely through docs and doctor/setup guidance before considering automatic journal vacuuming or unit output changes.

## Reproduction Assessment

No, not as a full live reproduction from this checkout: the reported 152 MB journal growth requires a Linux systemd user-service host. The source-level path is clear enough to keep open because stdout/stderr still feed the service log surface and no current unit or docs retention policy was found.

## Solution Assessment

Unclear. The issue's automatic journald vacuum option is not clearly the best fix without maintainer policy review; a narrower supported retention recommendation or doctor/setup warning may be safer than mutating host-level journal state.
