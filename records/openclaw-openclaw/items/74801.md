## Summary

Keep open. Current main has replaced the old file-log cron/truncate premise with in-process rolling file logs, but the reported Linux systemd journald storage-pressure concern is still not addressed in source, service generation, or docs.

## What This Changes

The issue asks OpenClaw to stop systemd installs from accumulating unbounded journald logs by aligning log rotation, journald retention guidance, or service/file-log behavior.

## Best Possible Solution

Choose an explicit Linux service logging policy and encode it in the narrowest maintainable surface: likely user-facing journald retention guidance or doctor/setup hints first, with service stream changes only if maintainers want OpenClaw to own that behavior by default.

## Reproduction Assessment

No. Static checks show current systemd installs can still emit to stdout/stderr and docs still point operators to journalctl, but a high-confidence reproduction would run the current gateway as a Linux systemd user service, generate logs, and compare journal disk usage with the rolling file log behavior.

## Solution Assessment

Unclear. The report lists multiple possible fixes; docs or doctor guidance appears lower risk than silently vacuuming journald or changing service streams, but maintainers should choose the intended Linux service policy first.
