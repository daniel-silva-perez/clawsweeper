## Summary

Keep open. Current main has moved past the exact old cron/script rotation details because file logs now rotate in-process, but the Linux systemd journald storage-pressure concern remains unaddressed: service output still reaches stdout/stderr, generated systemd units have no journal retention or stream redirection policy, and Linux docs point users to journalctl without retention guidance.

## What This Changes

The issue asks OpenClaw to address Linux systemd journald growth because current log rotation watches file logs while service output is stored in journald.

## Best Possible Solution

Keep this open until maintainers choose an explicit Linux service logging policy that covers both bounded file logs and bounded service output, likely starting with user-facing journald retention docs or doctor/setup guidance before changing generated service streams by default.

## Reproduction Assessment

No. Static checks show current Linux service output can still reach journald and no journald retention guidance exists, but high-confidence reproduction would require running the current gateway under a Linux systemd user unit, generating logs, and comparing journalctl disk usage with the rolling file log.

## Solution Assessment

Unclear. The report lists several possible fixes, but the narrowest maintainable path likely starts with docs or doctor guidance rather than silently vacuuming journald or changing service stream behavior by default.
