## Summary

This issue should stay open. Current main still runs load-time session-store maintenance synchronously for oversized stores, and `chat.history` is still gated during Gateway startup until sidecars are ready. Later code added high-water batching and honored configured maintenance settings, but it has not made the default startup path deferred, chunked, or budgeted.

## Best Possible Solution

Keep this issue open and fix the default startup path: make oversized session-store maintenance deferred, chunked with event-loop yields, or explicitly moved to a bounded background cleanup after the Gateway is responsive; preserve an operator/debug opt-out such as `session.maintenance.mode: "warn"`; and add regression coverage that `chat.history`/Gateway readiness is not held behind pruning or capping a large `sessions.json`.

Work in the session-store owner path first. Make `loadSessionStore` stop doing expensive prune/cap work synchronously on the caller path, or bound it with chunked/event-loop-yielding/background maintenance after the Gateway is responsive.
