## Summary

Current main still matches the reported failure mode. Session maintenance defaults to enforce, `loadSessionStore` still performs oversized-store pruning and capping inline after synchronous `sessions.json` read/parse, and `capEntryCount` still sorts the removable key set synchronously. Later work improved config handling, startup gating, and model-catalog avoidance for `chat.history`, but it has not deferred, chunked, or budgeted load-time session-store cleanup.

## Best Possible Solution

Keep this issue open and make default session-store startup maintenance startup-safe: defer it until the gateway is responsive, chunk it with event-loop yields, or otherwise bound it with a startup budget while preserving `session.maintenance.mode: "warn"` as an operator/debug opt-out and preserving OOM protection for oversized stores. Add regression coverage showing a large `sessions.json` does not hold gateway readiness or `chat.history` behind synchronous pruning/capping.

## Clownfish Work Prompt

Fix the startup regression reported in openclaw/openclaw#70050. Current main defaults session maintenance to enforce and `loadSessionStore` prunes/caps oversized stores inline, which can block Gateway startup and keep `chat.history` unavailable. Implement a startup-safe fix in the session maintenance path: keep the OOM/size protection from #69404, but avoid synchronous unbounded prune/cap work on the critical load path by deferring cleanup until after Gateway responsiveness, chunking with event-loop yields, or adding a bounded startup budget. Preserve the explicit operator opt-out `session.maintenance.mode: "warn"`. Do not broadly refactor gateway startup, plugin loading, or session transcript formats. Add focused regression coverage showing large session stores do not force synchronous default cleanup during Gateway readiness/history startup, while explicit/manual cleanup and configured maintenance behavior still work. Update docs if the final default behavior or operator guidance changes; add a changelog entry because this is user-visible startup behavior.
