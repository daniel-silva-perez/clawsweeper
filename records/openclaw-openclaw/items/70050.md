## Summary

Keep open: current main has mitigations around batching and session-store caching, but the root startup path still performs session-store read, parse, migration, normalization, stale pruning, and cap checks synchronously before returning. The linked triage comment is directionally supported by current source and tests, and the open WebUI timeout PR only masks an infinite loading state rather than fixing the gateway event-loop/startup stall.

## What This Changes

The issue reports that after upgrading, gateway startup and `chat.history` responsiveness can be blocked for about 20 minutes by load-time session maintenance, and asks for a way to defer, disable, or sequence that maintenance safely.

## Best Possible Solution

Keep the issue open and land a root fix that keeps gateway startup and `chat.history` responsive while preserving bounded session-store cleanup: move expensive load-time maintenance out of the hot synchronous read path, run it with an explicit budget/background pass or save/cleanup trigger, and retain warn/enforce semantics plus tests for oversized stores.

## Reproduction Assessment

Partially. Current source and tests give a high-confidence reproduction of the synchronous load-time maintenance path with an oversized `sessions.json`, but the exact 20-minute Windows plus Enter-to-unblock symptom still needs reporter-scale data or a targeted startup harness.

## Solution Assessment

No. The current implementation is not the best final solution because it keeps potentially expensive cleanup in `loadSessionStore`; the safer fix is bounded or deferred maintenance that preserves the #69404 OOM protection without blocking startup reads.

## Repair Work Prompt

Fix the root startup responsiveness issue for https://github.com/openclaw/openclaw/issues/70050. Current main still performs stale pruning and cap maintenance synchronously inside `loadSessionStore`, and `chat.history` reaches that path via `loadSessionEntry`. Preserve the #69404 goal of bounding oversized session stores, but move expensive load-time cleanup out of the hot synchronous read path or make it bounded/deferred so gateway startup and `chat.history` can respond quickly. Keep `session.maintenance.mode` semantics, `warn` behavior, configured `maxEntries`, `pruneAfter`, disk-budget behavior, and existing transcript/archive safety. Do not solve this only by adding a WebUI timeout; #44897 is a symptom-level UI PR. Add or update focused tests that prove oversized stores do not force immediate cap/prune work on a normal read/startup path, and that explicit cleanup or write-path maintenance still enforces limits. Update docs/changelog only if behavior or user-facing config changes.
