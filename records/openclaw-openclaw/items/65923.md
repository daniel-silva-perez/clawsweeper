## Summary

Current main still matches the report: omitted `session.reset` resolves to an implicit daily reset at 4 AM, normal sessions evaluate that policy and can roll over silently, and tests lock that default. Related #57369 and open PR #64028 cover an explicit manual-only/off mode, while #61238 covers the broader silent-reset UX, but none of those currently supersede this narrower unset-default contract question.

## What This Changes

The issue requests changing or clearly surfacing the behavior when `session.reset` is omitted so sessions do not silently reset daily at 4 AM without operator opt-in or prominent disclosure.

## Best Possible Solution

Keep this issue open for maintainer product/compatibility judgment. The best path is to decide the omitted `session.reset` contract explicitly: either make unset mean no automatic reset and update types, Zod schema, generated config schema/help, runtime policy, tests, docs, and operator surfaces, or intentionally keep daily-at-4-AM as the default and make that implicit behavior prominent in config help, FAQ/setup/status output, and startup/operator warnings. Coordinate with #57369/#64028 for the explicit manual-only mode and with #61238 for the broader silent-reset UX, but do not treat them as merged fixes for this narrower default-semantics report.
