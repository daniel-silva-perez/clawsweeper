## Summary

Current main still matches the report: omitted session reset config resolves to an implicit daily reset at 4 AM, runtime applies that policy to non-provider-owned sessions, and tests assert the default. The issue should stay open because the remaining choice is product/compatibility policy: change unset semantics to no automatic reset, or keep the daily default and make it much more visible across operator surfaces. The broader #61238 discussion overlaps but does not supersede this narrower unset-default report.

## Best Possible Solution

Keep this open as a focused product/UX bug. Maintainers should decide the contract for omitted `session.reset`: either add a supported manual-only/no-reset mode and update types, schema, resolver, runtime, tests, docs, and config/status surfaces, or intentionally keep the daily default and make the implicit policy prominent in config help, FAQ, setup/status output, and startup/operator warnings. Coordinate with #61238 without treating it as a replacement tracker.
