## Summary

Current main still matches the reported UX bug: omitting `session.reset` resolves to an implicit `daily` policy at 4am, the auto-reply session path still applies that policy to roll stale sessions, and both unit/integration tests intentionally lock the default. The narrower unset-default report should remain open because the broader daily-reset discussion overlaps but does not replace a maintainer decision on whether unset should mean disabled/no automatic reset or whether the implicit daily default should become much more visible.

## Best Possible Solution

Keep this open as a focused product/UX bug. Maintainers should decide the contract for omitted `session.reset`: preferably add a supported manual-only/no-reset mode and update schema/types/runtime/tests/docs/config UI, or deliberately keep the daily default and make it visible in config help, FAQ, setup/status output, and startup/operator warnings. Coordinate with #61238 for broader warning/recovery work, but do not treat that broader issue as closing this narrower unset-default report.
