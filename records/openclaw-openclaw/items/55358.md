## Summary

Keep this issue open. Current main still has no source-backed fix for the reported Slack DM 160-character inbound truncation: the Slack inbound path accepts and forwards `message.text`, but the internal message event type and content resolver still do not model or parse top-level Slack `blocks`/rich-text content as an agent-body fallback. The existing 160-character slice is only a system-event/log preview; `BodyForAgent` and `RawBody` receive the unsliced `rawBody`, so it does not explain or fix a case where Slack-provided `text` is already a preview.

## Best Possible Solution

Keep this issue open and fix it in the Slack plugin inbound content path. Capture or construct a representative >160-character Slack DM payload, extend the Slack event type/parser to preserve top-level block or rich-text content, use that as a fallback when `message.text` is missing or appears to be only a preview, and add a regression test asserting `BodyForAgent` and `RawBody` include the full tail while the 160-character system-event preview remains compact.
