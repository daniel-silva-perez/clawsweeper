## Summary

Current main still uses `zh-CN` and `zh-TW` as the Control UI runtime locale IDs, lazy import keys, browser-detection outputs, saved-settings validation values, generator metadata, workflow matrix entries, docs, and tests. The related implementation attempts are closed and unmerged, so the requested script-based `zh-Hans`/`zh-Hant` rename remains actionable.

## Best Possible Solution

Keep this issue open and implement a focused Control UI i18n contract change: make `zh-Hans` and `zh-Hant` the canonical runtime locale IDs for Simplified and Traditional Chinese, preserve the existing translated content, add legacy alias/migration support for saved global and scoped browser settings, update the generator metadata, workflow matrix, docs, selector/test expectations, and run the generated i18n checks. If maintainers intentionally want the region-coded IDs to remain stable, record that product/API decision on the issue instead.
