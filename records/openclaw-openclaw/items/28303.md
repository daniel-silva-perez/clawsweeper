## Summary

Current main still uses region-based `zh-CN` and `zh-TW` throughout the Control UI i18n runtime contract, generated locale pipeline, persisted settings validation, selector values, docs, workflow, and tests. The related implementation/docs/glossary PRs are closed and unmerged, so the script-based `zh-Hans`/`zh-Hant` rename plus compatibility migration remains actionable.

## Best Possible Solution

Keep this issue open and implement a focused Control UI i18n contract change: use `zh-Hans`/`zh-Hant` as runtime locale IDs, preserve the existing translated content, add legacy alias/migration support for saved global and scoped browser settings, update the generator metadata/workflow/docs/tests, and document the compatibility behavior. If maintainers decide the old region-coded IDs are intentionally stable, record that product/API decision on the issue instead of closing silently.
