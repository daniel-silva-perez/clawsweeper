## Summary

Keep this PR open. Current main still does not accept `moonshotai` or `moonshot-ai` as aliases for the direct `moonshot` provider: the shared normalizer returns those strings unchanged, model refs are split at the first slash, and the bundled Moonshot provider/catalog/docs still register the direct provider as `moonshot`. The linked #73876 user problem remains valid. The PR diff is narrow and consistent with the requested fix, and the security pass found no workflow, dependency, lockfile, script, secret-handling, or supply-chain surface in the changed files.

## Best Possible Solution

Keep this PR open and review/land it, or apply an equivalent narrow fix: add `moonshotai` and `moonshot-ai` to the shared provider-id normalization table so they resolve to `moonshot`, keep the canonical plugin id/auth profiles/docs as `moonshot`, and include focused regression coverage for direct `moonshotai/kimi-k2.6` model refs plus unchanged OpenRouter nested refs.
