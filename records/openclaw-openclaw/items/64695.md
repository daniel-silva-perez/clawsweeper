## Summary

Keep this issue open. Current main still routes both task registry and Task Flow registry persistence through `node:sqlite` with no runtime availability probe or fallback store; the scheduled sweep no longer leaks an unhandled rejection, but the reported Homebrew runtime would still fail to restore or persist task state.

## What This Changes

The issue asks OpenClaw to avoid gateway crashes and task persistence failures when the task registry reaches `node:sqlite` on Node builds that do not provide that built-in module.

## Best Possible Solution

Add a maintainer-approved runtime-safe storage selection layer for both registries: probe `node:sqlite` before default store use, keep SQLite where available, provide a durable and hardened fallback or supported dependency where unavailable, and document the resulting storage behavior.

## Risks / Open Questions

- The storage fallback decision is not purely mechanical: replacing `node:sqlite` with a native dependency changes install and packaging behavior, while JSON fallback needs explicit durability, locking, validation, and symlink/path-safety semantics.
- A fix must cover both task registry and Task Flow registry; addressing only one storage path would leave a similar startup or persistence failure in the other.
- The scheduled sweep now swallows failed sweep promises, so a storage fix should also preserve visible diagnostics when persistence is disabled or degraded.
