## Summary

Keep this issue open. Current main still uses `node:sqlite` as the default durable store for both task records and Task Flow state, with no availability probe, alternate dependency, or fallback store; the scheduled sweep now consumes promise rejections, but the underlying restore/upsert path still fails on runtimes where `node:sqlite` is missing.

## What This Changes

The issue requests a runtime-safe fix for Gateway task registry persistence when Homebrew Node.js lacks the `node:sqlite` built-in module.

## Best Possible Solution

Add a maintainer-approved runtime-safe storage selection layer for both task registries: probe `node:sqlite` before default use, keep SQLite where available, provide a durable and hardened fallback or supported dependency where unavailable, and update docs/tests for restore, upsert/update, stale-task maintenance, and Task Flow persistence.

## Reproduction Assessment

Yes. On a Node runtime where `require('node:sqlite')` throws, current main still routes task restore and maintenance-triggered task updates through `requireNodeSqlite()`; the exact Homebrew runtime was not available in this runner, but the source path matches the reported stack.

## Solution Assessment

Unclear. The issue's suggested `better-sqlite3` or JSON fallback may solve the runtime crash, but the best fix needs maintainer selection of the storage backend and durability/security semantics before implementation.
