## Summary

Keep this issue open. Current main still models and validates the relevant compaction token settings as absolute numbers only, and the runtime resolver ignores non-number values instead of parsing percentage strings against the model context window.

## What This Changes

Issue requests support for percentage string values such as `"10%"` in compaction token settings so configuration scales with each model context window.

## Best Possible Solution

Add a small typed compaction token-budget value contract that preserves existing numeric behavior while accepting documented percentage strings for the requested compaction settings, resolves them against the active context window at runtime, and updates schema generation, docs, changelog, and focused tests.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence static verification path: current schema/type/runtime code accepts numbers only and has no parser for percentage strings.

## Solution Assessment

Yes, percentage strings are a maintainable way to solve the portability problem as long as absolute numeric values remain backward-compatible. Auto-scaling existing numbers would be riskier because it would silently change established configs.
