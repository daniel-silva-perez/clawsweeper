## Summary

Keep open. Current main still has the split described in the report for the central lifecycle events: managed/internal hooks receive message:received and message:sent, but before_dispatch and agent_end remain plugin-hook-only. The requested bridge is plausible but should not be auto-closed or auto-implemented without maintainer API/privacy judgment because it would expand raw conversation lifecycle payloads into the internal HOOK.md surface and the proposed before:dispatch naming does not fit the current internal event taxonomy.

## What This Changes

Managed hooks installed through the internal HOOK.md system should be able to observe lifecycle events currently exposed only through plugin hooks, especially before_dispatch and agent_end.

## Best Possible Solution

Define an explicit supported internal-hook contract for the missing lifecycle observations, document the event names and payloads, and add focused tests at the dispatch and agent-runner call sites. If maintainers want these to remain plugin-only, the docs should say that before_dispatch and agent_end require a code plugin rather than a managed HOOK.md script.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a managed hook can register agent:end or before:dispatch through HOOK.md metadata, but current main has no triggerInternalHook call for before_dispatch or agent_end on the relevant paths, so those handlers cannot fire from normal dispatch/agent completion.

## Solution Assessment

Unclear. Adding triggerInternalHook beside each plugin hook is the obvious mechanical fix, but the best maintainable solution should first settle event naming, observation-only semantics, and raw conversation access boundaries rather than copying the plugin hook catalog wholesale.
