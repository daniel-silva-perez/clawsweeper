## Summary

Keep open. Current main partially addresses the report by bridging `message_received` and `message_sent` into internal `message:*` hooks, but the central requested managed-hook lifecycle coverage is still missing for `before_dispatch` and `agent_end`. Those remain plugin-only surfaces on current main, and the issue is concrete, recent, and not superseded by the related closed author-withdrawn report.

## Best Possible Solution

Keep this as the focused tracker for extending the managed/internal hook surface. A good fix would add explicit, documented internal events for the remaining lifecycle points, with tests in the dispatch and agent-end paths, and define stable names/semantics instead of blindly mirroring plugin hook names.

## Risks / Open Questions

- A direct bridge is not purely mechanical: plugin `before_dispatch` can handle a message and return a decision, while internal hooks currently return `void`, so maintainers need to define whether managed hooks are observation-only or get a new decision contract.
- The proposed `before:dispatch` name does not fit the current `InternalHookEventType` union, which only allows `command`, `session`, `agent`, `gateway`, and `message`; naming should be designed before implementation.
