## Summary

Keep this PR open. Current main still does not dedupe idless inbound retries, but the branch is dirty against the current inbound route-key implementation and needs a focused rebase/adaptation before it is mergeable.

## What This Changes

The PR adds a text/timestamp content fingerprint fallback for inbound dedupe when `MessageSid` is missing, preserves provider-id dedupe, and adds retry/non-collapse tests.

## Best Possible Solution

Rebase the fallback onto current main’s inbound dedupe design: keep `channelRouteDedupeKey()` and `JSON.stringify([sessionScope, routeKey, messageKey])`, use the raw provider message id when present, use `content:<sha256-128>` only for idless text/timestamp retries, and add focused coverage plus a user-facing changelog entry.

Yes. Current main has a high-confidence source-level reproduction: call inbound dedupe with provider/peer data but no `MessageSid`; `buildInboundDedupeKey()` returns `null`, `claimInboundDedupe()` returns `invalid`, and dispatch does not skip the retry.

## Solution Assessment

No, not in its current branch form. The fallback direction is narrow and maintainable, but the submitted patch must be rebased onto the current route-key JSON shape before it is the best mergeable solution.
