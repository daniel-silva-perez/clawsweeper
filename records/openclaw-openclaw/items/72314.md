## Summary

Keep open. Current main still has the idless inbound retry gap, and this PR is a relevant focused fix candidate, but the branch is dirty against current main after the route-key refactor and needs rebase/adaptation before it can merge safely.

## What This Changes

The PR adds a SHA-256 content/timestamp fallback inbound dedupe key and tests for idless retries while preserving provider message-id dedupe.

## Best Possible Solution

Rebase the PR onto current main and apply the fallback inside the current route-key tuple: keep `channelRouteDedupeKey(...)`, preserve raw provider message IDs as the final key element when present, use a namespaced `content:<hash>` only for idless text retries, and add current-main tests for idless retry dedupe plus timestamp separation.

## Reproduction Assessment

Yes. A current-main inbound context with provider, peer, body, and timestamp but no `MessageSid` produces no dedupe key, and `claimInboundDedupe()` returns `invalid`, which dispatch does not treat as a duplicate.

## Solution Assessment

Not yet. The content/timestamp fallback is a narrow maintainable direction, but the current PR branch should not merge until it is rebased onto the current `channelRouteDedupeKey`/JSON key shape instead of carrying forward the old pipe-delimited return.
