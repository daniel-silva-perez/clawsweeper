## Summary

Keep this PR open. Current main still requires a provider message id for inbound dedupe, so idless inbound retries are not deduped before dispatch. The PR remains a focused implementation candidate, but it is now dirty against main after the route-key refactor and needs rebase/review rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review. Best path: rebase it onto current main, adapt the content/timestamp fallback to the current `JSON.stringify([sessionScope, routeKey, messageKey])` key shape, preserve raw provider-id key compatibility, clarify empty-body/timestamp assumptions, and rerun the targeted inbound-dedupe test plus the changed gate before merge.
