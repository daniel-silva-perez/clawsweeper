## Summary

Keep open. Current main still requires explicit delivery before a gateway agent reply can leave the session, while Telegram group and topic session keys remain active. The same reporter has an open implementation candidate in a fork, so the right path is maintainer review, landing, or porting that focused fix instead of closing this issue.

## What This Changes

Default Telegram group-bound gateway agent sessions to external delivery when `deliver` is omitted, while preserving `deliver: false`, internal-channel suppression, and no auto-delivery for unrelated session shapes.

## Best Possible Solution

Add or land a small gateway delivery-flag helper plus focused tests that auto-enable delivery only for `agent:*:telegram:group:*` and `agent:*:telegram:group:*:topic:*` when `deliver` is omitted, while keeping `deliver:false` authoritative and suppressing external sends for the internal message channel.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send a gateway `agent` request with a Telegram group or topic session key and no `deliver`; current main computes `wantsDelivery` and final `deliver` as false, so transcript state can update without external Telegram delivery.

## Solution Assessment

Yes. The requested gateway-local helper is the narrowest maintainable fix because the decision is made before dispatching the agent run, and tests can cover Telegram group/topic keys without changing Telegram DM, webchat, main, or other channel semantics.
