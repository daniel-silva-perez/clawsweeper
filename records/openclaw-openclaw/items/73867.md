## Summary

Keep open. Current main does not make `openclaw tasks maintenance --apply` prune the session registry. The task maintenance command is scoped to task registry plus Task Flow maintenance, while session-store cleanup and cron-run session reaping live on separate paths and do not implement the requested tasks-maintenance sweep with a durable jobs-state active guard.

## Best Possible Solution

Keep this issue open and implement a focused `tasks maintenance` session-registry sweep, likely by sharing session cleanup primitives rather than duplicating store mutation code. The implementation should define the exact session-key scope, derive the protected active cron run set from durable `jobs-state.json`/`CronJobState.runningAtMs`, preview/apply counts in the existing JSON output, preserve currently running rows, and add targeted tests plus docs for the new `tasks maintenance` behavior.
