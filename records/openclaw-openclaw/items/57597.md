## Summary

Keep this PR open. Current main and the latest release still persist ACP spawn labels through the requester store, while the PR head routes the write through the spawned ACP session key and the earlier broken-test assertion has been corrected.

## What This Changes

The PR changes ACP spawn label persistence to resolve the target session store from the spawned ACP session key and adds a regression test for requester-store versus target-store mismatch.

## Best Possible Solution

Land a rebased version of this PR after targeted ACP command regression proof, preserving the current boundary: label persistence should resolve the spawned target session store, same-store in-memory updates should still work, and spawn routing/binding/runtime policy should remain unchanged.

## Reproduction Assessment

Yes. The reported flow is reproducible by tracing `/acp spawn codex --bind here --label <name>` from a requester session whose store differs from the spawned `agent:codex:acp:*` store; the PR body also reports a local Telegram-bound reproduction.

## Solution Assessment

Yes. Resolving the target store from the spawned ACP session key is the narrow maintainable fix because ACP metadata already uses the same resolver and session stores are agent-specific.
