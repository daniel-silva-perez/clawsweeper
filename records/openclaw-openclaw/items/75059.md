## Summary

Keep open. Current main still treats explicit `--url` Gateway transport failures as fatal in the logs follow loop, and this PR is a live implementation candidate for the linked reconnect behavior. The patch is scoped to the CLI/docs/tests surface and has no blocking correctness or security issue from this read-only review, though there is one small writer-handling cleanup worth addressing.

## What This Changes

The PR adds bounded exponential-backoff retry handling for `openclaw logs --follow` transport disconnects, focused tests for fallback and explicit URL behavior, and a CLI docs note.

## Best Possible Solution

Land a narrow logs CLI retry implementation that preserves implicit-loopback file fallback, retries explicit transient transport failures with bounded backoff, keeps auth/pairing/configuration failures fatal, and documents the behavior after the minor writer-return handling is cleaned up.

## Reproduction Assessment

Yes. The linked issue gives a concrete runtime path for `openclaw logs --follow` plus a Gateway restart, and current main has a source/test-level reproduction for the explicit URL case because the existing test still asserts `process.exit(1)` on a Gateway close error.

## Solution Assessment

Yes, with a small cleanup. Localizing retry behavior in the `logs --follow` loop is the narrowest maintainable fix compared with changing the shared Gateway client, and it preserves the existing local-file fallback; the retry warning should follow the surrounding `errorLine` return-value pattern.
