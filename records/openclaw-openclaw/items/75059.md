## Summary

Keep open for normal PR review because current main still exits on explicit Gateway URL follow-loop disconnects, and the PR is useful but has one blocking retry-classification bug for plain post-handshake close errors.

## What This Changes

The PR adds bounded exponential-backoff retry handling to `openclaw logs --follow`, focused CLI tests for retry and fatal cases, and a docs note for reconnect behavior.

## Best Possible Solution

Keep the localized follow-loop retry design, but parse close codes from plain `gateway closed (...)` errors before retrying so 1008 and 4xxx auth/rate-limit closures remain fatal, with a regression test for the plain-error path.

## Reproduction Assessment

Yes. The linked issue gives a runtime path with `openclaw logs --follow` plus a Gateway restart, and current main source/test coverage shows explicit `--url` close errors still exit. The review finding has a source-level reproduction by passing a plain `Error("gateway closed (4001): unauthorized")` through `--follow` and observing the new retry branch treat it as transient.

## Solution Assessment

No, not as currently written. The narrow CLI retry approach is the right shape, but the transient classifier needs to handle plain close strings consistently with typed `GatewayTransportError` codes before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not retry plain 4xxx close errors:** `src/cli/logs-cli.ts:164-168`
  - body: GatewayClient pending request failures can still reach this helper as plain `Error` strings like `gateway closed (4001): ...` when `client.request` rejects on close. This branch treats every plain close string as transient, so `--follow` retries auth or rate-limit 4xxx closes despite the PR's fatal-error contract. Parse the close code before returning true and add a plain-error 4xxx regression test.
  - confidence: 0.89
