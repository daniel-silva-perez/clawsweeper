## Summary

Keep open. Current main supports `/acp spawn --bind here`, but the programmatic `sessions_spawn` ACP path still has no current-bind selector and routes `thread: true` through channel default placement; for Discord that default is child placement, which creates or binds a separate thread instead of attaching to the current one.

## What This Changes

Add a programmatic `sessions_spawn` option equivalent to `/acp spawn --bind here` so an agent can bind an ACP session to the current Discord thread or conversation.

## Best Possible Solution

Add a first-class ACP current-conversation binding path to `sessions_spawn`, reusing the `/acp spawn --bind here` policy and placement checks while preserving existing child-thread behavior for `thread: true`. Cover Discord existing-thread binding with focused tests and update the tool/docs surface.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: call `sessions_spawn` with `runtime: "acp"`, `mode: "session"`, and `thread: true` from a Discord thread context; the current code has no current-bind parameter and will select Discord child placement rather than current placement.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if implemented as an additive option. The safer version is to add an explicit current-bind field rather than redefining `thread: true`, because existing Discord top-level child-thread spawns depend on that behavior.
