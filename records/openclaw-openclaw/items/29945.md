## Summary

Keep this issue open. Current main still hardcodes branded pairing replies for unpaired DM pairing requests, has no `unpairedResponse` config contract, and the linked maintainer PR #39065 remains open and unmerged as the active implementation path.

## Best Possible Solution

Keep this issue open until #39065 or an equivalent maintainer-approved change lands on main. The fix should add a documented `unpairedResponse` policy for DM pairing responses, wire `silent`, `code-only`, and `branded` behavior through shared pairing helpers and affected channel/plugin call sites, update config schema/metadata/docs, cover all modes with regression tests, and audit adjacent pairing logs/error responses for raw sender identifiers.
