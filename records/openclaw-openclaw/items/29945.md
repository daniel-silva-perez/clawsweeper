## Summary

Current main still implements branded, always-sent DM pairing replies for unpaired senders, has no `unpairedResponse` config contract, and the linked closing PR #39065 remains open and unmerged. The issue should stay open until that PR or an equivalent maintainer-approved security hardening change lands.

## Best Possible Solution

Keep this open and review/land or deliberately supersede #39065. The accepted fix should add a documented `unpairedResponse` policy, wire `silent`, `code-only`, and `branded` through shared pairing helpers and channel/plugin call sites, update config schema/metadata/docs, cover the modes with regression tests, decide the default explicitly, and audit adjacent pairing logs/error paths for raw sender identifiers.
