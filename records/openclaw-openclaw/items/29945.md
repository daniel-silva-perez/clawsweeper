## Summary

Current main still has the reported behavior: DM pairing replies are hardcoded as branded `OpenClaw: access not configured.` messages, newly-created pairing challenges still send a reply, and there is no `unpairedResponse` config surface. The linked implementation PR #39065 remains open and unmerged with closing syntax for this issue, so this issue should stay open for maintainer review rather than be closed by cleanup.

## Best Possible Solution

Keep this issue open and review, land, or deliberately supersede #39065. The accepted fix should add a documented `unpairedResponse` policy, wire `silent`, `code-only`, and `branded` through shared pairing helpers and channel/plugin call sites, update config schema/metadata/docs, add regression tests for the modes, decide the default explicitly, and audit adjacent pairing logs/error paths for raw sender identifiers.
