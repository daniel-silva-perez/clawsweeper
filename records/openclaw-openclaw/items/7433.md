## Summary

Keep this issue open. Current main still lacks the requested Baileys `getMessage` retry callback, socket-level `cachedGroupMetadata`, proactive group metadata/participant event handling, and explicit `lid-mapping.update` persistence. Open PR #73732 is relevant to the group-inbound subset, but it is still open and does not cover the broader retry/cache/LID reliability request.

## Best Possible Solution

Keep this issue open. The best path is to review, land, or supersede #73732 for the group inbound reconnect subset, then implement the broader WhatsApp plugin reliability hooks in a focused follow-up: bounded account-scoped message retry storage wired through Baileys `getMessage`, a shared group metadata cache exposed through `cachedGroupMetadata`, proactive cache updates from group/participant events, and an explicit decision on whether `lid-mapping.update` persistence is needed beyond the current LID lookup paths. Add focused WhatsApp plugin regression tests before closing.
