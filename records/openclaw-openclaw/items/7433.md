## Repair Work Prompt

Fix the remaining WhatsApp/Baileys group reliability gaps tracked by this issue. Treat #73732 and #66920 as partial context only: current main has reconnect-time group metadata fallback, but still lacks Baileys socket-level `getMessage`, `cachedGroupMetadata`, proactive group metadata event handling, and focused tests. Keep the change inside the WhatsApp plugin boundary. Add a bounded, account-scoped, in-memory recent message store for `getMessage`; do not persist raw message content to disk unless maintainers explicitly approve it. Verify Baileys v7 `lid-mapping.update` behavior against source/types and prefer `sock.signalRepository.lidMapping` over a parallel stale map. Update focused tests for socket options and inbound event wiring, and add a user-facing changelog fix entry crediting @octopuslabs-fl and relevant related reporters if runtime behavior changes. Do not change unrelated WhatsApp provider defaults, core channel APIs, or broad reconnect watchdog policy.
