## Summary

Keep this issue open. Current main still lacks the requested Baileys `getMessage` retry callback, socket-level `cachedGroupMetadata`, proactive group metadata/participant event maintenance, and explicit `lid-mapping.update` persistence. Open PR #73732 is relevant implementation work for the group inbound subset, but it is still open and does not complete the broader retry/cache/LID request.

## Best Possible Solution

Keep this issue open. The best path is to review or supersede #73732 for the group inbound subset, then finish the broader WhatsApp plugin reliability work: add a bounded account-scoped message retry store wired through Baileys `getMessage`, expose a shared group metadata cache through `cachedGroupMetadata`, seed and update it from connect hydration plus group/participant events, and decide whether current `signalRepository.lidMapping` plus auth-dir reverse files are sufficient or whether explicit `lid-mapping.update` persistence is needed. Add focused WhatsApp plugin regression tests before closing.
