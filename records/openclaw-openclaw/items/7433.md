## Risks / Open Questions

- A `getMessage` implementation retains recent message content; it should be bounded, account-scoped, and in-memory by default unless maintainers explicitly approve disk persistence.
- `lid-mapping.update` should be implemented only after checking the Baileys v7 source/types because Baileys already persists several LID mappings through `sock.signalRepository.lidMapping`; a parallel stale map would be a regression.
- The current regression tests exercise reconnect fallback, not high-volume retry/decryption behavior, so the fix needs focused socket-option and event-handler tests rather than relying on the existing group reconnect case.
