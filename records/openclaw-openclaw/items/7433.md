## Repair Work Prompt

Implement the remaining WhatsApp/Baileys group reliability hooks for the current issue. Keep the change scoped to the WhatsApp plugin. Add an account-scoped bounded recent-message store for Baileys `getMessage` retry support, preferably in-memory with TTL/count bounds and no disk persistence of message content unless explicitly approved. Wire socket-level `cachedGroupMetadata` from the shared group metadata cache so Baileys can use it during group sends/processing. Add proactive cache updates for `groups.upsert`, `groups.update`, and `group-participants.update`, refreshing or invalidating the existing cache without unbounded participant retention. Handle `lid-mapping.update` through the existing Baileys `sock.signalRepository.lidMapping`/auth-store path after verifying the installed Baileys v7 type shape; do not create a separate long-lived mapping store if Baileys already persists it. Preserve existing reconnect fallback behavior from #73732 and avoid core changes or dependency bumps. Add focused tests for the new socket options, message retry lookup bounds, group update listeners, participant update refresh/invalidation, and LID mapping event handling. Add a single-line user-facing changelog fix with `Thanks @octopuslabs-fl` if behavior changes are landed.
