## Summary

Keep this issue open. Current main includes the narrower PR #73732 reconnect/group-metadata fallback, but the broader Baileys reliability request is still not implemented: the WhatsApp socket is not wired with `getMessage` or socket-level `cachedGroupMetadata`, and the plugin still lacks proactive group metadata/participant event handlers and `lid-mapping.update` handling.

## What This Changes

Request improved bundled WhatsApp/Baileys group reliability by adding retry `getMessage`, socket-level group metadata caching, proactive group cache updates, and LID mapping persistence/update handling.

## Best Possible Solution

Keep this open and handle the remaining work in a focused WhatsApp plugin PR: add account-scoped bounded retry message storage wired into Baileys `getMessage`, pass a socket-level `cachedGroupMetadata` callback, update the shared group metadata cache from group/participant events, and decide/implement `lid-mapping.update` handling through the existing Baileys signal/auth-store path. Treat #73732 as partial context, not as the closing fix.
