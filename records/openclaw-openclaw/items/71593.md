## Summary

Keep open. Current main still has a closed SecretRef source/provider model limited to env, file, and exec, while the active implementation candidate #72548 is open and unmerged; the engram authToken comment also confirms the schema coverage gap is user-visible.

## What This Changes

Request plugin-defined SecretRef providers and broader SecretRef object support across core config schema fields so users do not need core PRs for each secret backend or sensitive path.

## Best Possible Solution

Land a versioned, documented, security-reviewed plugin SecretRef provider contract or an equivalent maintainer-approved seam, then align core types, zod schemas, Gateway wire validation, resolver dispatch, plugin manifest/SDK/runtime registration, docs, and the remaining SecretInput field coverage such as skills env values and plugin config sensitive paths.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence verifiable by static inspection of current main: the type, zod schema, Gateway schema, docs, and resolver dispatch only support env, file, and exec.

## Solution Assessment

Unclear until maintainer review of the API shape. A plugin-defined SecretRef provider seam matches the plugin-first architecture, but the active PR must be reviewed for security, versioning, and whether it covers enough schema surfaces or needs a follow-up.
