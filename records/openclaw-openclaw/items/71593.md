## Summary

Keep open. Current main still has a closed SecretRef provider/source model limited to env/file/exec, current docs and the supported credential surface only cover targeted SecretInput paths, and open PR #72548 is already a closing implementation candidate that should be reviewed before this issue is closed.

## What This Changes

Request to let plugins define SecretRef provider sources that can apply across core config schemas instead of requiring core schema edits for each secret-bearing field.

## Best Possible Solution

Review and either land #72548 or ask for a narrower maintainer-approved replacement that adds a versioned, documented plugin SecretRef provider contract, then track any remaining coverage gaps for skills env values, generic plugin config sensitive paths, and channel credential schemas as follow-up work.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence verifiable by static inspection: current type, zod, Gateway protocol, docs, and resolver dispatch only support env, file, and exec, while the engram comment provides a concrete plugin config case.

## Solution Assessment

Unclear until maintainer review of #72548. A plugin-defined SecretRef provider seam fits the plugin-first direction, but the proposed implementation needs API/security/versioning review and may need follow-up work for the broader config-field coverage.
