## Summary

Keep open. Current main still has a closed SecretRef source/provider model limited to `env`, `file`, and `exec`, and the linked implementation candidate #72548 is still open and unmerged in the provided GitHub context. The engram authToken comment is useful confirmation that the remaining gap is user-visible, not only an abstract schema concern.

## What This Changes

The issue requests a plugin-based SecretRef provider mechanism that can cover core config schemas without repeated core schema edits.

## Best Possible Solution

Review #72548 as the active implementation candidate, then either land an equivalent versioned plugin SecretRef provider contract or explicitly document `exec` as the supported universal extension point. If a new contract lands, align core types, schemas, Gateway wire validation, resolver dispatch, plugin manifest/SDK/runtime registration, docs, tests, and the remaining SecretInput surfaces such as `skills.entries.*.env`.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is still high-confidence verifiable by static inspection of current main: the type, zod schema, Gateway schema, resolver dispatch, and docs only accept `env`, `file`, and `exec`.

## Solution Assessment

Unclear until maintainer review of the API shape. A plugin-defined SecretRef provider seam matches the repo's capability/plugin pattern, but the safer outcome may be either landing #72548 plus field-coverage follow-up or documenting `exec` as the supported external-provider boundary.
