## Summary

Keep open. Current main still exposes a closed SecretRef provider model limited to env, file, and exec, while plugin config support only marks specific manifest-declared secret input paths. The open closing implementation PR #72548 is the right item to review before this issue can be closed.

## What This Changes

Request a plugin-based SecretRef provider API so plugins can define new secret-reference sources and reduce repeated core schema changes for secret-capable config fields.

## Best Possible Solution

Land or replace PR #72548 with a versioned, documented plugin SecretRef provider contract that preserves strict built-in env/file/exec behavior, honors plugin activation and origin boundaries, and separately tracks any remaining SecretRef field-coverage gaps for skills env values, plugin config, and channel credentials.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main rejects non-env/file/exec SecretRef sources in config, SDK, gateway wire schemas, and resolver dispatch.

## Solution Assessment

Unclear until maintainer review of PR #72548. A plugin-defined provider seam matches the plugin-first architecture, but the open implementation explicitly defers broader schema field coverage, so maintainers need to decide whether that split is acceptable.
