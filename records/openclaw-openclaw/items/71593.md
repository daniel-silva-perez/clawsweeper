## Summary

Keep open. Current main still has a closed SecretRef source/provider model limited to env, file, and exec; plugin config support only marks declared SecretInput paths, and the linked open implementation PR remains the right review path before closing this issue.

## What This Changes

The issue requests a plugin-defined SecretRef provider contract so users can add secret backends and cover sensitive config fields without repeated core schema changes.

## Best Possible Solution

Add a versioned, documented, activation-aware plugin SecretRef provider contract that works for bundled, installed, and workspace plugins without changing env/file/exec behavior, then explicitly track any remaining SecretRef field-coverage gaps for skills env values, generic plugin config, and channel credential schemas.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main accepts and resolves only env/file/exec SecretRef sources, and the engram comment supplies a concrete plugin-config field affected by the missing broader coverage.

## Solution Assessment

Unclear until maintainer review of PR #72548. A plugin-defined SecretRef provider seam matches the plugin-first architecture, but the current implementation candidate needs API/security review and should honor plugin activation and third-party plugin origins before it can close this issue.
