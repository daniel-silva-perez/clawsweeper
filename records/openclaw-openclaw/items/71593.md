## Summary

Keep open. Current main still has a closed SecretRef source/provider contract limited to env, file, and exec, and the active closing PR #72548 is open and unmerged; the engram authToken comment adds a concrete user-visible plugin config coverage gap.

## What This Changes

The issue requests a plugin-defined SecretRef provider mechanism so users can use SecretRef objects across core config schemas without core changes for every provider or sensitive field.

## Best Possible Solution

Land a versioned, documented, security-reviewed plugin SecretRef provider contract or equivalent maintainer-approved seam, then align core types, zod schemas, Gateway wire validation, resolver dispatch, plugin manifest/SDK/runtime registration, docs, and the remaining SecretInput field coverage such as skills env values and plugin config sensitive paths.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence verifiable by static inspection of current main: source types, zod validation, Gateway schemas, docs, and resolver dispatch still only support env, file, and exec.

## Solution Assessment

Unclear until maintainer review of the API shape. A plugin-defined SecretRef provider seam fits OpenClaw's plugin-first direction, but #72548 is still open and explicitly defers broader field coverage, so it may need either a narrower merge plus follow-up or additional changes before this issue is complete.
