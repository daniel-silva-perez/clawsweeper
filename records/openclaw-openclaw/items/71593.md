## Summary

Keep open. Current main still has a closed SecretRef provider/source model limited to env, file, and exec; the manifest SecretInput path support helps plugin-owned config fields but does not let plugins register new SecretRef provider sources, and #72548 is still the active unmerged implementation candidate linked to this issue.

## What This Changes

The issue requests a plugin-based SecretRef provider mechanism so users can add secret providers and cover sensitive config schema fields without repeated core schema edits.

## Best Possible Solution

Review and land #72548 or a narrower maintainer-approved replacement that adds a versioned, documented plugin SecretRef provider contract across config types, Zod, Gateway protocol, SDK docs, and resolver dispatch, then follow through on the remaining SecretInput coverage for the requested config surfaces.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence verifiable by static inspection of current main: the type, Zod schema, Gateway schema, docs, and resolver dispatch only support env, file, and exec, while the engram comment supplies a concrete plugin config case.

## Solution Assessment

Unclear until maintainer review of the active API proposal. A plugin-defined SecretRef provider seam fits OpenClaw's plugin-first direction in VISION.md, but #72548 needs security/versioning review and may not cover the full schema-field scope requested here.
