## Summary

Keep open. Current `main` still logs pre-handshake WebSocket closes at WARN except for the SwiftPM helper, so the PR targets a real gateway log-noise path, but the proposed branch is not clearly merge-ready because its suppression depends on an `Origin` header that current `GatewayClient` construction does not set.

## What This Changes

The PR adds an internal-probe close classifier in `src/gateway/server/ws-connection.ts` so loopback, code-1000 pre-handshake closes with the expected gateway origin log at debug instead of warn.

## Best Possible Solution

Keep the PR open, but require a small revision before merge: either make probe clients send an explicit, test-covered internal marker such as a known Origin/header, or classify current-main probe closes using data the server actually receives without broadening the downgrade to external/proxied clients.

## Reproduction Assessment

Yes. A focused reproduction path is a gateway probe using current `GatewayClient` that closes after a successful probe; source inspection shows `client.stop()` runs after settlement while the close handler still warns when `client` is not registered.

## Solution Assessment

No. The direction is narrow and appropriate, but the current implementation keys on `requestOrigin`, while current `GatewayClient` plus `ws` does not send an Origin unless configured; a safer fix should align the probe signal and the server-side classifier and add a regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Classify probes using a header they actually send:** `src/gateway/server/ws-connection.ts:244`
  - body: The new helper requires `origin === http://${effectiveProbeHost}:${port}`, but current `GatewayClient` builds `new WebSocket(url, wsOptions)` without `origin` or `headers`, and `ws` only sends an Origin when that option is set. That means the gateway's own `probeGateway()` closes still arrive with no Origin on current main and continue through the WARN path. Please either set an explicit probe Origin/header on the client side or match on a server-side signal these probes already provide.
