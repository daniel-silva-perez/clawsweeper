## Summary

Keep open. Current main has meaningful provider-transport, polling, timeout, and diagnostic hardening, but the reported four-agent Telegram workload is not proven fixed and current code still preserves the main coupling points called out in the report: Telegram runner concurrency and the Gateway main lane both use `agents.defaults.maxConcurrent`, and model fallback still runs candidates sequentially without an explicit per-candidate timeout budget.

## What This Changes

The issue reports simultaneous LLM API timeouts across four Telegram-bound agents and requests concurrency, fallback-timeout, service-env, DNS/network, and watchdog/lock configurability fixes.

## Best Possible Solution

Keep this as an active reliability investigation until maintainers can capture current-main telemetry for the reported workload, then land the smallest measured fix in the layer that actually stalls: fallback budgeting, provider transport, Gateway lane scheduling, Telegram polling, service env, or session locking.

Unclear. The report gives concrete reproduction steps and timestamped curl/log evidence, but this review did not have the live Telegram bots, provider accounts, and concurrent browser/tool workload needed to reproduce the failure on current main.

## Solution Assessment

Unclear because there is no patch for this issue. The best path is instrumentation first; a broad speculative patch across concurrency limits, DNS, Undici, launchd, and watchdog settings would be harder to validate than a measured fix at the stalled layer.
