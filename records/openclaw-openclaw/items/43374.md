## Summary

Keep this issue open. Current main has partial fallback and transport hardening, but the central multi-agent Telegram reliability report is still not proven fixed: Telegram polling concurrency remains tied to the agent concurrency cap, inbound runs still enter a process-wide main lane, the LaunchAgent service environment does not default UV_THREADPOOL_SIZE, and generic model-provider request transport still does not expose Telegram-style DNS result-order controls.

## Best Possible Solution

Keep this as an active reliability investigation. Reproduce or instrument a four-agent Telegram workload on current main, measuring Telegram update sink pressure, main-lane queue wait, session-lock acquire/hold timing, LLM connect/header/body/idle timing, provider fallback attempts, abort reasons, event-loop lag, and libuv threadpool pressure. Then fix the measured layer: decouple Telegram update processing from agent-run concurrency if polling is starved, add a generic configurable LLM network policy if provider transport is implicated, tune service/threadpool defaults only with evidence, and add focused regression coverage for the all-agents-timeout pattern.
