## Summary

Keep open. Current main has meaningful hardening for provider transport, fallback classification, Telegram polling watchdogs, and process-wide queueing docs, but it still does not prove the reported four-agent Telegram failure is fixed. The same global agent limit still feeds Telegram runner sink concurrency and the main inbound lane, service defaults do not set `UV_THREADPOOL_SIZE`, and the exact simultaneous timeout bottleneck remains unmeasured on current main.

## What This Changes

The issue requests a fix for simultaneous LLM provider timeouts during four concurrent Telegram-bound agent sessions, with suspected bottlenecks in fallback timeout budgeting, Telegram polling concurrency, provider transport/DNS policy, service threadpool defaults, and session-lock/watchdog tuning.

## Best Possible Solution

Keep this as an active reliability investigation. Reproduce or instrument a four-agent Telegram workload on current main, measuring Telegram update sink pressure, main-lane queue wait, session-lock acquire/hold timing, provider connect/header/body/idle timing, fallback attempts, abort reasons, event-loop lag, and libuv threadpool pressure, then make one focused fix in the measured layer with targeted regression coverage.
