## Summary

Keep open. Current main contains meaningful fallback, provider-transport, Telegram polling watchdog, and queueing improvements, but it still does not prove the reported four-agent Telegram failure is fixed: Telegram runner concurrency remains coupled to the global agent cap, inbound runs still pass through the process-wide main lane, service defaults do not set `UV_THREADPOOL_SIZE`, and the report spans several subsystems that need current-main instrumentation before selecting a narrow fix.

## What This Changes

The issue reports simultaneous LLM request timeouts across multiple Telegram-driven agents despite provider APIs responding directly, and proposes timeout, transport, threadpool, Telegram concurrency, and lock/watchdog changes.

## Best Possible Solution

Keep this as an active reliability investigation. Reproduce or instrument a four-agent Telegram workload on current main, measuring Telegram update sink pressure, main-lane queue wait, session-lock acquire/hold timing, provider connect/header/body/idle timing, fallback attempts, abort reasons, event-loop lag, and libuv threadpool pressure, then make one focused fix in the measured layer with regression coverage.
