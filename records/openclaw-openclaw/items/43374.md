## Summary

Keep open. Current main has meaningful hardening for provider HTTP transport, timeout-classified model fallback, Telegram polling liveness, and command lanes, but the reported four Telegram-bot/four-agent simultaneous LLM timeout pattern is not proven fixed. Several reported pressure points still exist or remain unmeasured: Telegram runner sink concurrency is still tied to agent maxConcurrent, the Gateway main lane uses the same agent concurrency limit, service environments do not set a libuv threadpool default, and there is no current-main workload test or instrumentation that isolates the bottleneck.

## What This Changes

The issue requests a reliability fix for simultaneous LLM provider timeouts when multiple Telegram-bound agents run concurrently, including fallback timeout behavior, service environment threadpool sizing, Telegram polling concurrency, and provider transport DNS/Undici behavior.

## Best Possible Solution

Keep this as an active reliability investigation until maintainers can run or instrument a current-main four Telegram-bot/four-agent workload, then land a focused fix in the measured layer with targeted coverage for that layer and without broad speculative tuning.
