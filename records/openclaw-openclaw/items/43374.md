## Summary

Keep open. Current main still preserves the central coupling points from the report, and no relevant code changed since the prior ClawSweeper review that kept this active: Telegram long-poll runner concurrency and Gateway main-lane concurrency still derive from the same agent limit, while fallback candidates still run sequentially without an explicit per-candidate fallback budget. Main does have useful transport, timeout, polling watchdog, and diagnostics hardening, but that is not enough to prove the reported four-agent Telegram timeout storm is fixed.

## What This Changes

The issue reports simultaneous LLM API timeouts across four Telegram-backed agents while direct provider API calls remain reachable, and requests concurrency, fallback-timeout, DNS/transport, service-env, and lock/watchdog improvements.

## Best Possible Solution

Keep this as an active reliability investigation until maintainers capture current-main telemetry for the reported workload, then land the smallest measured fix in the layer that actually stalls: fallback budgeting, provider transport, Gateway lane scheduling, Telegram polling, service environment, or session locking.

## Reproduction Assessment

Unclear. The report has concrete steps and timestamped curl/log evidence, but this review did not have the live Telegram bots, provider accounts, and concurrent browser/tool workload needed to reproduce the failure on current main.

## Solution Assessment

Unclear because there is no patch for this issue. The best path is instrumentation first; a broad speculative change across concurrency limits, DNS, Undici, launchd, and watchdog settings would be less maintainable than a measured fix at the stalled layer.
