## Summary

Keep open. Current main includes meaningful fallback and transport hardening, but it still does not prove or fully address the reported multi-agent Telegram failure mode: Telegram polling concurrency remains tied to the agent concurrency cap, inbound agent runs still enter a process-wide main lane, LaunchAgent service env does not default `UV_THREADPOOL_SIZE`, and model-provider transport lacks the Telegram-specific DNS result-order surface.

## What This Changes

The issue reports simultaneous LLM API timeouts across multiple Telegram-backed agents and providers, with suspected shared fallback timeout, Telegram concurrency coupling, transport/DNS behavior, and service threadpool defaults.

## Best Possible Solution

Keep this as an active reliability investigation. Reproduce or instrument a four-agent Telegram workload on current main, measuring Telegram update sink pressure, main-lane queue wait, session-lock acquire/hold timing, provider connect/header/body/idle timing, fallback attempts, abort reasons, event-loop lag, and libuv threadpool pressure. Then fix the measured layer with focused regression coverage rather than applying all hypothesized knobs at once.
