## Summary

Keep this open. Current main and the latest release still use the narrow memory-wiki lint target check, while the open linked PR #73722 is the active implementation candidate and explicitly closes this issue.

## What This Changes

The issue requests broader memory-wiki lint normalization so title, slug, fragment-stripped, basename/path, and imported source-path Obsidian link variants resolve without false `broken-wikilink` warnings.

## Best Possible Solution

Land PR #73722, or an equivalent memory-wiki-owned fix, that expands lint target lookup keys, strips fragments and queries consistently, preserves imported provenance, and adds focused regression coverage for the reported link forms.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a page titled `Alpha Database` linked as `[[Alpha Database#Evidence]]` leaves the fragment in `linkTargets`, while current lint only accepts extensionless paths and basenames. I did not run a vault command because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable solution is to fix the memory-wiki lint resolver and its colocated tests; rewriting imported source evidence would damage provenance and is not the right fix.
