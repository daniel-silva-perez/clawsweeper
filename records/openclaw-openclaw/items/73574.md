## Summary

Keep this open. Current main still has the narrow memory-wiki lint target resolver, and the provided GitHub context shows open PR #73722 is the active closing implementation candidate for the same reported false positives.

## What This Changes

The issue asks for `openclaw wiki lint` to normalize title, slug, fragment, basename/path, and imported source-path variants before reporting Obsidian wiki links as broken.

## Best Possible Solution

Land PR #73722, or an equivalent memory-wiki-owned fix, that expands lint lookup keys, strips fragments and queries consistently, preserves imported provenance, and adds focused regression coverage for the reported link forms.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a page titled `Alpha Database` linked as `[[Alpha Database#Evidence]]` leaves the fragment in `linkTargets`, while current lint accepts only extensionless paths and basenames. I did not run a vault command because this review is read-only.

## Solution Assessment

Yes. The best fix is a narrow change in the memory-wiki lint resolver with colocated regression tests; rewriting imported source evidence would damage provenance and is not the right solution.
