## Summary

Keep this open. Current main and the latest release tag still use the narrow memory-wiki broken-link resolver, while open PR #73722 is already the focused implementation candidate that explicitly closes the issue.

## What This Changes

Wiki lint should resolve existing memory-wiki pages when Obsidian links use title, slug, fragment/query, or imported source-path variants.

## Best Possible Solution

Land PR #73722 or an equivalent memory-wiki-owned change that expands lint target lookup keys, strips fragment/query suffixes consistently, preserves imported provenance, and adds focused regression coverage for the reported link forms.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a page titled `Alpha Database` linked as `[[Alpha Database#Evidence]]` will be linted as broken because Obsidian extraction preserves the fragment while current valid targets only include the extensionless relative path and basename. I did not run a vault command because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable solution is to fix the memory-wiki lint resolver and colocated tests; rewriting imported evidence would damage provenance, and no core/plugin API change is needed.
