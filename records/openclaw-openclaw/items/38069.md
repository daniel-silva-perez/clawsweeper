## Summary

Keep this issue open. Current main still has one-shot transient retry and a generic recurring error-backoff floor, but it does not implement configurable delayed re-execution for failed recurring cron jobs before the next natural daily/weekly schedule. The related implementation PR #69252 is closed unmerged.

## Best Possible Solution

Keep this issue open and implement an additive recurring cron retry policy: per-job retry config in cron types and gateway schemas, CLI add/edit flags, durable retry-attempt state distinct from natural schedule state, transient-error classification for recurring executions, scheduling at `endedAt + configured delay` before the next natural slot while attempts remain, and docs/tests that distinguish one-shot retry, recurring backoff floor, and recurring execution retry.

## Clownfish Work Prompt

Implement openclaw/openclaw#38069 as a focused, additive cron reliability PR. Current main has one-shot transient retry plus a generic recurring error-backoff floor; it does not re-run failed recurring cron jobs before the next natural daily/weekly schedule. Add configurable delayed retry for recurring cron execution failures without changing existing one-shot behavior or broad scheduler ownership.

Expected boundary:
- Add a backwards-compatible per-job retry policy to cron domain types/store and gateway add/update schemas, likely shaped like `{ maxAttempts?: number; backoffMs?: number[]; retryOn?: CronRetryOn[] }` or an equivalent repo-consistent name. Do not require migration for existing jobs.
- Keep global `cron.retry` one-shot semantics clear; avoid silently changing its documented contract unless maintainers intentionally choose a default for recurring retry.
- In `applyJobResult`, when an enabled recurring job fails with a transient error and retry attempts remain, schedule `nextRunAtMs` at `endedAt + configuredDelay` even when that is before the natural next cron/every slot. After attempts are exhausted or the error is non-transient, resume the natural schedule plus the existing storm-prevention backoff behavior.
- Track retry-attempt state durably and distinctly from natural schedule state so restarts do not lose pending retry intent. Do not overload unrelated delivery/failure-alert state.
- Add CLI support for `openclaw cron add|edit` retry controls such as `--retry-max`, `--retry-delay`, `--retry-on`, and `--no-retry`, using existing duration parsing patterns. Keep flags additive and validate invalid combinations.
- Update user docs in `docs/automation/cron-jobs.md` and `docs/cli/cron.md`; clarify the difference between one-shot retry, recurring execution retry, and the existing recurring error-backoff floor.
- Add regression tests for a daily/weekly recurring job that fails transiently and retries before the next natural slot, retry budget exhaustion, non-transient failure, restart/recompute behavior if state is added, gateway schema validation, and CLI parsing.
- Do not touch unrelated provider retry, transport retry, release metadata, generated baselines, or plugin surfaces unless required by the typed contract.

Related refs: openclaw/openclaw#38069, #69252, #24355, #11641.
