## Summary

Keep this issue open. Current main still only has global one-shot transient retry plus recurring error backoff as a storm guard; it does not add per-job retry policy, CLI add/edit retry flags, Gateway job retry schema, or durable retry-attempt state for delayed recurring re-execution before the next natural daily/weekly slot.

## Best Possible Solution

Implement a core scheduler feature for additive per-job recurring execution retry: cron job retry config in types and Gateway schemas, CLI add/edit flags, durable retry attempt state, transient error classification, and scheduling at endedAt plus configured delay while attempts remain and before the next natural slot. Preserve existing one-shot retry behavior and recurring backoff storm guards, and document the distinction clearly.

## Repair Work Prompt

Implement recurring cron execution retry with configurable delay after transient failures. The central bug is that recurring daily/weekly cron jobs still wait for the later of the natural next schedule and the error backoff floor after a transient failure; they need optional retry re-execution before the next natural slot while retry attempts remain.

Scope:
- Add an additive job-level retry policy to cron job types and Gateway schemas, compatible with existing jobs and clients. Suggested shape may follow the issue: maxAttempts, delay/backoff list, retryOn, and disable/clear support.
- Add durable retry attempt state so retries survive Gateway restart and do not duplicate already-consumed attempts.
- Update scheduling in src/cron/service/timer.ts and any maintenance/recompute path in src/cron/service/jobs.ts so recurring jobs schedule endedAt + configured delay for transient errors while attempts remain, then fall back to the natural schedule after exhaustion. Preserve existing one-shot retry behavior and existing recurring backoff storm guard semantics where applicable.
- Add CLI add/edit flags for retry configuration and clearing/disabling retry, using existing cron CLI parsing style.
- Update Gateway protocol validation and tests for the new fields. Keep protocol changes additive.
- Update docs/help for docs/automation/cron-jobs.md, docs/cli/cron.md, and src/config/schema.help.ts as needed. Add a changelog entry only if repo policy says this user-facing feature needs one, and include a real contributor attribution if added.
- Do not change unrelated provider retry behavior, connection-level retry, release metadata, dependency pins, or bundled plugin boundaries.

Related refs: this issue, #69252, #11616, #24355, and #31104. Treat #69252 as prior art only; it is closed unmerged, so verify the final design against current main rather than copying blindly.
