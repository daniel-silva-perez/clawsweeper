## Summary

Keep this issue open. Current main still has one-shot transient retry through global `cron.retry` and recurring error backoff as a storm guard, but recurring daily/weekly jobs still schedule the later of the natural next run and the backoff floor. There is no per-job retry policy, CLI add/edit retry flags, Gateway schema field, or durable retry-attempt state for delayed recurring re-execution. Related PR #69252 directly targeted this gap but is closed unmerged.

## Best Possible Solution

Implement an additive recurring cron execution retry policy: job-level retry config in cron types and Gateway schemas, CLI add/edit flags, durable retry attempt state, transient-error classification, scheduling at `endedAt + configured delay` before the next natural slot while attempts remain, and docs/tests that preserve existing one-shot retry plus recurring backoff safeguards.
