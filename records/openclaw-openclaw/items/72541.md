## Summary

Keep open. Current main still matches the reported false-negative: a zero-payload terminal turn after `sessions_spawn` is classified through the generic incomplete-turn path, while the subscriber records only tool name/meta and does not carry the accepted spawn result into the classifier.

## What This Changes

The issue requests that the gateway treat a successful terminal `sessions_spawn` as completing a zero-payload pure-relay turn instead of surfacing the incomplete-turn warning.

## Best Possible Solution

Carry a minimal structured success marker for `sessions_spawn` through the current attempt, then let the incomplete-turn classifier treat `status: "accepted"` plus a non-empty `childSessionKey` as terminal completion while preserving warnings for failed spawns, malformed results, and empty turns with no successful terminal action.

## Reproduction Assessment

Yes. A focused unit test can build an attempt with `payloadCount: 0`, no assistant text, `toolMetas: [{ toolName: "sessions_spawn" }]`, replay metadata from `buildAttemptReplayMetadata`, and a non-error stop; current main still returns the side-effect incomplete-turn warning.

## Solution Assessment

Yes. The narrowest maintainable fix is to propagate the actual successful spawn result into the runner completion classifier, not to disable empty-turn detection globally or require agent-specific acknowledgement text.
