## Summary

Keep open. Current main still classifies a zero-payload terminal assistant turn as incomplete even when the turn successfully executed `sessions_spawn`; the accepted spawn result exists, but the incomplete-turn classifier only sees side-effect metadata, not a successful terminal handoff signal.

## What This Changes

Request to treat pure-relay agent turns that successfully call `sessions_spawn` and then stop without text as complete rather than surfacing the gateway incomplete-turn warning.

## Best Possible Solution

Add structured current-attempt completion metadata for successful `sessions_spawn` results and let the incomplete-turn classifier return no warning only when the same attempt has `status: "accepted"` plus a non-empty `childSessionKey`; keep the existing warning for failed spawns, no-tool empty turns, and replay-unsafe side effects without confirmed terminal completion.

## Reproduction Assessment

Yes. A focused helper test can construct a zero-payload stop attempt with `toolMetas: [{ toolName: "sessions_spawn" }]` and replay metadata from the current helper; current `resolveIncompleteTurnPayloadText` will return the side-effect incomplete-turn warning instead of `null`.

## Solution Assessment

Yes, with a narrower implementation than a global config flag. The maintainable fix is to preserve successful spawn result metadata through the current attempt and exempt only accepted terminal spawns, not to disable empty-turn detection or add agent-specific acknowledgement text.
