## Summary

Keep open for maintainer handling. The Feishu filename-threading goal is still useful because current main does not carry explicit filename values through the send path, but this PR is not mergeable as-is because it introduces Feishu media regressions and mixes in unrelated security-sensitive CI, CODEOWNERS, docs, skills, scripts, and lockfile changes.

## What This Changes

The PR threads optional filename fields through outbound message contracts into Feishu media upload while also changing Feishu media upload behavior, Feishu sendPayload handling, CI workflows/actions, CODEOWNERS, local docs/scripts, skills, OpenAI WebSocket headers, and a uv lockfile.

## Best Possible Solution

Land a narrow Feishu/outbound patch that carries an explicit filename through the message tool, ReplyPayload, normalized payload, delivery overrides, ChannelOutboundContext, and Feishu sendMedia call while preserving loadWebMedia validation, remote URL download behavior, native-card payload handling, and upload filename sanitization.

## Reproduction Assessment

Yes. The blocking regressions are reproducible by static control-flow inspection of the PR diff: remote non-image URLs can reach fs.createReadStream as paths, local absolute non-image files bypass mediaLocalRoots, and CODEOWNERS is deleted.

## Solution Assessment

No. A focused filename-plumbing change is the maintainable solution; this branch changes unrelated CI/security/docs/skills surfaces and weakens existing Feishu media safety checks.
