## Summary

Keep open. Current main has safe Feishu upload-layer filename handling, but it still does not thread caller-provided filenames through the normal `send` payload path into Feishu media sends, and this PR branch is not safe to merge because it introduces functional regressions plus unrelated security-sensitive repository changes.

## What This Changes

The PR threads optional filename values through outbound and Feishu media send paths, but also adds a Feishu local upload fast path, removes CODEOWNERS, changes CI/cache/action refs, adds local docs/scripts/skills, modifies OpenAI WebSocket headers, and adds a uv lockfile.

## Best Possible Solution

Land a narrow Feishu/outbound patch that carries an explicit `filename` from message-tool send params and reply payloads into Feishu `sendMediaFeishu`, while preserving `loadWebMedia`, `mediaLocalRoots`, upload filename sanitization, CODEOWNERS, CI workflows, local docs, skills, scripts, and lockfiles unchanged.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path for the PR regressions: send a remote non-image Feishu media URL or a disallowed absolute local non-image path through the PR branch; both paths bypass the current guarded `loadWebMedia` behavior.

## Solution Assessment

No. The filename plumbing direction is useful, but this branch is not the best solution because it couples that change to unrelated repository/security changes and weakens existing Feishu media safety checks.
