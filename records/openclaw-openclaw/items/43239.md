## Summary

Current main still does not implement the PR's central filename-preservation path for normal `message` tool `send` delivery to Feishu. The PR addresses a real repository gap, but it remains too broad and security-sensitive to close as implemented or merge as-is: the provided diff includes unrelated CI/CODEOWNERS/local-script/skill changes and Feishu media regressions already called out in review.

## Best Possible Solution

Keep this PR open only as an implementation candidate and request a narrow revision or replacement. The maintainer path should thread `filename` through `MessageSendParams`, `executeSendAction`, reply/outbound payload normalization, `ChannelOutboundContext`, Feishu `sendPayload`/`sendMedia`, and `sendMediaFeishu`; preserve `loadWebMedia`/`mediaLocalRoots` and `sanitizeFileNameForUpload`; add focused regression tests for explicit filename forwarding, basename fallback, remote non-image URLs, local-root rejection, and upload sanitization; and remove all unrelated workflow, CODEOWNERS, local-script, docs, skills, lockfile, and OpenAI WebSocket changes.

## Risks / Open Questions

- Closing as implemented would drop a real remaining gap: normal `send` delivery still cannot carry the message-tool `filename` through core outbound contracts into Feishu media upload.
- Merging the PR as provided could regress Feishu remote document/file sends by treating URL strings as filesystem paths in the upload layer.
- Merging the PR as provided could bypass `mediaLocalRoots` validation for non-image local Feishu uploads, creating a local-file exposure risk.
- Merging the PR as provided could weaken Feishu multipart filename hardening for control characters, quotes, and backslashes.
