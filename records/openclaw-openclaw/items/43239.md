## Summary

Keep open. The Feishu filename plumbing is useful and current main still does not carry an explicit message-tool filename through the normal outbound send contracts, but this PR is not mergeable as-is because it mixes the Feishu fix with broad CI/CODEOWNERS/local docs/skills changes and introduces concrete Feishu media and repository-protection regressions.

## What This Changes

The PR threads optional filename values through outbound message contracts into Feishu media upload, adds Feishu sendPayload/local-file handling, and also changes CI workflows/actions, CODEOWNERS, local docs/scripts, skills, OpenAI WebSocket headers, and a uv lockfile.

## Best Possible Solution

Land a narrow Feishu/outbound change that threads an explicit filename through the message-tool, ReplyPayload, normalized outbound payload, ChannelOutboundContext, and Feishu sendMedia call while preserving current loadWebMedia local-root validation, remote URL download behavior, native card sendPayload behavior, and upload filename sanitization. Drop unrelated CI, CODEOWNERS, local docs/scripts, skills, WebSocket header, and uv lockfile changes.

## Reproduction Assessment

Yes. The blocking regressions are reproducible by static control-flow inspection: remote non-image mediaUrl values reach uploadFileFeishu as raw URL strings, local non-image paths bypass mediaLocalRoots, and .github/CODEOWNERS is deleted in the PR diff.

## Solution Assessment

No. The requested filename preservation should be solved as a focused outbound/Feishu contract change; this branch is too broad and removes existing safety controls unrelated to the Feishu filename bug.
