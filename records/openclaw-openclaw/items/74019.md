## Summary

Keep open. Current main still follows the reported failing path: `llm-task` starts an embedded run with tools intentionally disabled, the runner creates an empty callable tool set, then the empty explicit-allowlist guard still evaluates inherited config allowlists and can produce the reported error. An open PR already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Allow Lobster workflows to invoke `llm-task` when inherited `tools.alsoAllow` or agent tool allowlists include `lobster` and `llm-task`, without the no-tools embedded run failing the empty allowlist guard.

## Best Possible Solution

Review and land #74139 or an equivalent focused fix that skips config-derived empty allowlist failures for intentional no-tool embedded runs while preserving runtime `toolsAllow` conflicts; then close this issue as implemented.

## Reproduction Assessment

Yes. The issue provides a concrete Lobster plus `/tools/invoke` reproduction, and current main statically confirms the same path: `llm-task` disables tools, effective tools are empty, inherited allowlist sources are still collected, and the guard blocks prompt submission.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. The safest implementation should distinguish inherited config allowlists from explicit runtime `toolsAllow` so no-tool JSON/model runs can proceed without weakening caller-requested tool constraints.
