## Summary

Keep open. Current main still has the reported disabled-tools/explicit-allowlist path, and the issue is already paired with open closing PR #74139, so cleanup should wait for a reviewed fix to land rather than closing early.

## What This Changes

The issue requests that `llm-task` JSON-only embedded model runs work from Lobster workflows when inherited `tools.alsoAllow` entries are present, without treating intentionally disabled tools as an allowlist failure.

## Best Possible Solution

Review and update #74139 or land an equivalent narrow fix that ignores config-derived allowlist sources for intentional no-tool model runs, preserves runtime `toolsAllow` fail-closed behavior, adds focused regression coverage for `llm-task` and model-run paths, then close this issue after the fix is merged.

## Reproduction Assessment

Yes. The issue provides concrete config and `/tools/invoke` steps, and the current source trace on `98b96182f8e18a96a20eda6de51a16d1ae0b09ad` deterministically reaches the same empty-tools guard with inherited allowlist sources.

## Solution Assessment

No for current main, and unclear for the open PR as written. The safer solution is not to skip all disabled-tool guard checks, but to ignore inherited config allowlists for no-tool runs while keeping explicit runtime tool requests fail-closed.
