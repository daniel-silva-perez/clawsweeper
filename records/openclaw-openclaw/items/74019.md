## Summary

Keep open. Current main still matches the reported failure path: llm-task starts a JSON-only embedded run with disableTools=true while forwarding the caller config, and the embedded runner still feeds config-derived allowlists from tools.alsoAllow/agents.main.tools.alsoAllow into the empty-tool precheck. That can still produce the reported “No callable tools remain” error before prompt submission.

## Best Possible Solution

Land a narrow embedded-runner fix: for disableTools JSON-only embedded runs such as llm-task, ignore ordinary config-derived allowlist sources when no tools are meant to be callable, while preserving fail-closed behavior for explicit runtime toolsAllow and real tool-enabled runs. Add focused regression coverage for the guard/attempt path plus llm-task, and include a changelog fix entry crediting @technicallysecure.
