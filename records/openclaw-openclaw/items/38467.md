## Summary

Keep open. Current main still has generic directory/name-resolution plumbing, but the bundled WhatsApp plugin only exposes config-backed allowlist peers and does not subscribe to or persist Baileys contact-list events, so saved phone contacts cannot be resolved by name.

## What This Changes

The issue requests that the WhatsApp directory ingest Baileys contact-list events, persist saved phone contacts locally, expose them through `openclaw directory peers`, and allow outbound messaging by contact name.

## Best Possible Solution

Implement the feature in the bundled WhatsApp plugin using the existing directory and outbound resolver seams: opt-in, account-scoped local contact storage fed by Baileys contact events or a verified equivalent sync path, directory entries with stable id/name/handle/raw metadata, unique-match outbound resolution, and preserved allowFrom/outbound authorization checks. Update WhatsApp and directory docs after the behavior is real.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, `openclaw directory peers list --channel whatsapp --query Emily` can only search entries derived from `allowFrom`, so a saved phone contact named Emily cannot appear unless the user manually put that number into config.

## Solution Assessment

Unclear as an automated change, but the requested direction is the right product boundary: the fix belongs in the WhatsApp plugin, not core. The safest implementation is narrower than the old message-store PR and should add only opt-in contact persistence plus directory/name-resolution integration first.
