## Summary

Keep this issue open. Current main still has generic directory/name-resolution plumbing, but the bundled WhatsApp plugin only lists config-backed `allowFrom` peers, does not persist Baileys contact-list events, and does not expose saved WhatsApp contact names/phone numbers for outbound name resolution. Related PRs found in the discussion/search do not supersede it: #33924 was closed unmerged and covered outbound allowlist policy, while #14834 was a closed unmerged message-store/contact-resolution attempt.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin, not core: subscribe to Baileys contact-list events, persist account-scoped contacts locally behind an explicit privacy/config choice, expose them through the WhatsApp directory as `ChannelDirectoryEntry` rows with stable id/name/handle/raw metadata, and wire outbound name resolution through unique directory matches while preserving `allowFrom`/future outbound policy checks. Add focused tests for ingestion, query filtering, ambiguity, account isolation, persistence/logout cleanup, and privacy behavior, plus docs for the opt-in and `openclaw directory peers list --channel whatsapp`.

## Risks / Open Questions

- Persisting the full WhatsApp phone contact list is privacy-sensitive and needs an explicit opt-in/default decision, account scoping, local storage permissions, deletion/logout behavior, and docs.
- Name-based outbound resolution must fail safely on ambiguous/stale contacts and must preserve WhatsApp outbound authorization checks rather than turning contact discovery into an allowlist bypass.
- A prior closed unmerged implementation attempt (#14834) mixed contact resolution with a broader message store and extra tool actions; a narrower implementation may be easier to review.
