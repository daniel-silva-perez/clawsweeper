## Summary

Keep open. Current main still routes each Telegram inbound update to one agent for the receiving account, has no Telegram cross-account forwarding or broadcast-account config surface, and the existing docs/tests describe or assert singular routing rather than multi-bot group fan-out.

## What This Changes

Request cross-account Telegram group routing so a message received by one bot account can be forwarded or broadcast to other configured bot accounts/agents while preserving separate bot identities.

## Best Possible Solution

Design an additive Telegram cross-account or broadcast routing model with loop prevention, per-account identity preservation, explicit allowlists, bot-authored message handling, docs, and focused Telegram multi-account group coverage; if maintainers decide not to support it, document the supported alternative clearly.

## Risks / Open Questions

- A naive fan-out model could create duplicate replies or reply loops if bot-authored messages, self-echoes, and mention handling are not designed explicitly.
- Account identity and authorization semantics need a clear allowlist model so one Telegram bot cannot accidentally cause another account/agent to reply in the wrong identity.
- Telegram bot-to-bot observability depends on BotFather Bot-to-Bot Communication Mode plus privacy/admin setup, so product docs and live QA need to separate platform prerequisites from OpenClaw routing behavior.
