## Summary

Keep open. Current main still routes each Telegram inbound message to one route for the receiving account, deliberately drops non-default Telegram group fallback without an explicit binding, and has no `forwardToAccounts`, `broadcastToAccounts`, or Telegram account fan-out config surface. The prior ClawSweeper comments are still consistent with the current checkout, so the stale label is not a closure signal here.

## What This Changes

Request an opt-in Telegram multi-account routing or broadcast feature so group messages received by one bot account can be forwarded to other bot accounts while preserving separate bot identities.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or broadcast routing model with per-account allowlists, identity preservation, loop prevention, bot-authored message handling, docs, and focused multi-account group coverage; if maintainers do not want to support the pattern, document the supported alternative clearly.

## Risks / Open Questions

- A naive Telegram fan-out model could duplicate replies or create loops unless bot-authored messages, self-echoes, mention handling, and BotFather bot-to-bot mode are designed explicitly.
- Cross-account routing needs an explicit authorization model so one Telegram account cannot implicitly authorize another account to process or answer a group message.
- Maintainers still need to choose between a Telegram-specific forwarding setting, a generalized broadcast-groups expansion beyond WhatsApp, or documenting this multi-bot group pattern as unsupported.
