## Summary

Keep open. The issue body and prior ClawSweeper comments describe a concrete Telegram multi-bot group gap, and current main still routes each Telegram inbound update to one route for the receiving account; non-default group fallback is still dropped without an explicit binding, and there is no Telegram cross-account forwarding, broadcast, or bot-message routing config surface.

## What This Changes

Request an opt-in Telegram multi-account routing feature so group messages received under one bot account can be forwarded or broadcast to other configured bot accounts and agents while preserving bot identity.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or broadcast routing model with per-account identity preservation, explicit destination allowlists, loop prevention, bot-authored message handling, docs, and focused multi-account group coverage; if maintainers decide not to support this pattern, document the supported alternative clearly.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is the existing named-account Telegram group case: a non-default account receives a group message without an explicit binding and buildTelegramMessageContext returns null, which the test at extensions/telegram/src/bot-message-context.named-account-dm.test-support.ts:123 asserts.

## Solution Assessment

Unclear. The requested forward/broadcast direction is plausible, but the best maintainable solution requires maintainer product and security choice between Telegram-specific account forwarding, expanding broadcast groups beyond WhatsApp, or explicitly documenting the pattern as unsupported.

## Security Review

Status: needs_attention

Summary: No patch was reviewed, but the requested feature is security-sensitive because it would let one Telegram account cause other accounts or agents to process group traffic.

Concerns:

- **[medium] Define cross-account authorization before fan-out:**
  - body: Forwarding inbound messages from one Telegram account to another could bypass the destination account's intended group, sender, and tool policy unless the feature requires explicit destination allowlists and preserves account identity boundaries.
  - confidence: 0.87
- **[medium] Prevent bot-to-bot reply loops:**
  - body: Bot-authored group messages need self-echo handling, mention or activation rules, and loop prevention before cross-account routing is safe to enable.
  - confidence: 0.84
