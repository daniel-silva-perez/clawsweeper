## Summary

Keep open. Current main still selects a single Telegram route for the receiving account and drops named-account group fallback; there is no Telegram config, route resolver, or docs surface for forwarding or broadcasting inbound group messages across bot accounts.

## What This Changes

Requested an opt-in Telegram multi-account routing feature so messages observed by one bot account in a group can be delivered to other configured bot accounts or agents while preserving bot identity.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or broadcast routing model with loop prevention, per-account identity preservation, explicit allowlists, bot-authored message handling, docs, and focused multi-account group coverage; if maintainers decide not to support this pattern, document the supported alternative clearly.

## Risks / Open Questions

- A naive fan-out model could create duplicate replies or reply loops unless bot-authored messages, self-echoes, mention handling, and Telegram BotFather bot-to-bot mode are designed explicitly.
- Account identity and authorization semantics need an explicit allowlist model so one bot account cannot implicitly grant another account permission to process or answer a group message.
- Maintainers need to choose whether this belongs in Telegram-specific account forwarding, a generalized broadcast-groups expansion beyond WhatsApp, or documentation of an unsupported pattern.
