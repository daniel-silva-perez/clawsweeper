## Summary

Keep open. Current main still routes Telegram inbound messages to a single route for the receiving account, intentionally drops named-account group fallback without an explicit binding, and has no `forwardToAccounts`, `broadcastToAccounts`, or equivalent Telegram fan-out config. The stale label does not make this closeable because the requested behavior remains a real product gap.

## What This Changes

The issue requests opt-in Telegram multi-account routing so a group message received through one bot account can be forwarded or broadcast to other configured bot accounts/agents while preserving separate bot identities.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or broadcast routing model with per-account allowlists, identity preservation, bot-authored message handling, loop prevention, docs, and focused multi-account group coverage; if maintainers do not want to support this pattern, document the supported alternative clearly.

## Security Review

Status: needs_attention

Summary: No patch was reviewed, but the requested routing feature is security-sensitive because it would let one Telegram account cause other accounts or agents to process group traffic.

Concerns:

- **[medium] Define cross-account authorization before fan-out:**
  - body: Forwarding inbound messages from one Telegram account to another could bypass the destination account's intended group and sender policy unless the feature requires explicit destination allowlists and preserves account identity boundaries.
  - confidence: 0.86
- **[medium] Prevent bot-to-bot reply loops:**
  - body: Bot-authored group messages need loop prevention, self-echo handling, and mention/activation rules before a cross-account routing feature is safe to enable.
  - confidence: 0.82
