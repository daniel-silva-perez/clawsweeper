## Summary

Keep open. Current main still has the Telegram multi-account gap described in the report: inbound Telegram messages resolve to one route for the receiving account, named-account group fallback is intentionally dropped without an explicit binding, and there is no `forwardToAccounts`, `broadcastToAccounts`, or equivalent Telegram fan-out configuration. The stale label and prior ClawSweeper comments do not change that; the previous keep-open rationale remains consistent with the current checkout.

## What This Changes

Add opt-in Telegram multi-account routing or broadcast behavior so a group message received by one bot account can be routed to other configured bot accounts or agents without losing account identity.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or broadcast routing model with per-account allowlists, identity preservation, loop prevention, bot-authored message handling, docs, and focused multi-account group coverage; if maintainers do not want to support this pattern, document the supported alternative clearly.

## Risks / Open Questions

- A naive fan-out model could duplicate replies or create loops unless bot-authored messages, self-echoes, mention handling, and Telegram bot-to-bot behavior are designed explicitly.
- Cross-account routing needs an explicit authorization model so one Telegram account cannot implicitly authorize another account or agent to process and answer a group message.
- Maintainers still need to choose between Telegram-specific forwarding, a generalized broadcast-groups expansion beyond WhatsApp, or documenting this multi-bot group pattern as unsupported.
