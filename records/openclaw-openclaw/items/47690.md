## Summary

Keep this issue open. Current main supports Telegram multi-account bindings, but routing still selects a single agent for each inbound update and there is no shipped `forwardToAccounts`/`broadcastToAccounts` or Telegram group broadcast path that satisfies the requested cross-account bot-to-bot fan-out. The provided related PR #63380 is relevant prior work for account-level agent routing, but it is still open and does not prove this feature is shipped on main.

## Best Possible Solution

Keep the issue open and design an additive routing feature, likely by extending the generic binding/broadcast model to support Telegram peer/account fan-out after normal Telegram access checks. The implementation should define whether the target is accounts, agents, or bindings; preserve outbound replies on the target bot token; include loop prevention; and add Telegram tests for two bot accounts in one group plus docs for BotFather bot-to-bot requirements.

## Risks / Open Questions

- Telegram bot-to-bot group delivery depends on Telegram/BotFather behavior and should be live-verified before implementing the routing design.
- Cross-account forwarding can create duplicate replies or loops unless self-authored messages, mention gating, and account identity are handled deliberately.
- Any fan-out must preserve per-account allowlists, agent isolation, and reply-token identity so one bot account cannot leak or impersonate another agent's context.
