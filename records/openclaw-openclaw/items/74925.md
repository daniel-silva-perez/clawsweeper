## Summary

Keep this PR open. Current main still lacks outbound WhatsApp PN-to-LID resolution, and the PR is a focused implementation for the linked ghost-chat delivery bug with no blocking code-review finding from this read-only pass; it needs maintainer review and targeted validation before merge.

## What This Changes

The PR adds a WhatsApp outbound JID resolver that reads Baileys forward LID mappings from the account authDir, threads authDir into createWebSendApi, and tests mapped, unmapped, and no-authDir text, poll, and composing sends.

## Best Possible Solution

Land a focused WhatsApp plugin fix that resolves plain phone-number recipients against the active account's Baileys LID forward mapping at the final active send boundary, preserves unmapped PN fallback, and leaves broader relink/reverse-mapping repair as separate follow-up work.

## Reproduction Assessment

Yes. Current main can be source-reproduced by creating an active send API and sending to a phone number with a matching lid-mapping-{digits}.json available in the account authDir: the current code still sends {digits}@s.whatsapp.net because it never receives authDir.

## Solution Assessment

Yes. Passing authDir into createWebSendApi and resolving only final outbound text/media, poll, and composing targets is a narrow maintainable fix that matches the existing inbound authDir mapping pattern without adding a new core API surface.
