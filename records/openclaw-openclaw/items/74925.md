## Summary

Keep open for maintainer review and possible landing. Current main still lacks outbound phone-number to LID resolution for WhatsApp proactive sends, while this PR implements a focused authDir-backed resolver at the active send boundary, adds targeted coverage, and shows no blocking correctness or security finding from the reviewed diff.

## What This Changes

The PR adds an authDir-aware WhatsApp outbound JID resolver using Baileys LID forward mapping files, threads authDir into createWebSendApi, and adds unit/integration coverage for mapped, unmapped, and no-authDir sends.

## Best Possible Solution

Land a focused WhatsApp plugin fix that resolves plain phone-number recipients against the active account's Baileys LID forward mapping at the final active send boundary, keeps unmapped numbers on the existing PN fallback, and tracks any re-link reverse-mapping problem separately.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: with a forward lid-mapping-{phoneDigits}.json present, createWebSendApi still sends +15555550000 through toWhatsappJid as 15555550000@s.whatsapp.net; the linked issue provides concrete WhatsApp steps and a compiled-bundle workaround.

## Solution Assessment

Yes. Resolving at createWebSendApi with the listener authDir is the narrowest maintainable fix for the central delivery bug because text/media, polls, and composing presence all flow through that active send API, while the PR preserves existing behavior when authDir or a mapping is absent.
