## Summary

Keep open. Current main still has restart/drain mitigations and SQLite-backed task records, but I did not find the durable inbound/SQLite message store wiring requested here. Slack ingress still dispatches directly from the Bolt message handler into the reply/agent path, and the command queue/restart recovery visible on main is in-memory or session-recovery based rather than a persistent pre-dispatch inbound queue. The related #49431 remains the right implementation candidate from the supplied context, but it is open and unmerged, so it is not a close reason.

## Best Possible Solution

Keep this issue open and use it to review, land, or supersede #49431. The best fix is a durable pending-inbound store at the channel/gateway ingress boundary before active Slack dispatch, plus startup replay/re-dispatch, shutdown/restart handling, and targeted regression coverage for an accepted Slack turn interrupted before final reply delivery.
