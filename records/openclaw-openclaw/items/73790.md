## Summary

Keep open. Current main still writes fallback-used provider/model values into durable session runtime fields, so the PR targets a real behavior gap, but the latest patch still misclassifies explicit live-session switches that are consumed inside the fallback runner on both direct-agent and cron paths.

## What This Changes

The PR adds fallback-vs-primary guards around direct agent and isolated cron session runtime model persistence, plus regression tests for fallback no-persist and live-switch behavior.

## Best Possible Solution

Carry an explicit selection-source outcome from the fallback runner, or persist the live-switch target before retry, so primary success, automatic fallback, and explicit live-switch targets cannot be inferred from final provider/model comparisons. Apply the same contract to direct-agent and isolated cron paths, then cover both outer-throw and internal-redirect cases.

## Reproduction Assessment

Yes. For the reported bug, configure a primary plus fallback and have `runWithModelFallback` return a different provider/model; current main writes that value through `setSessionRuntimeModel`. For the PR regression, use the existing later-candidate `LiveSessionModelSwitchError` path in `src/agents/model-fallback.test.ts` and assert direct/cron runtime persistence semantics.

## Solution Assessment

No. The PR is directionally relevant, but the current approach infers fallback-ness after the fallback loop from provider/model comparisons; an explicit outcome/source signal is the narrower maintainable solution because live switches can be handled inside the fallback runner.
