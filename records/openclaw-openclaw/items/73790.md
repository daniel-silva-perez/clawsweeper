## Summary

Keep open: the underlying sticky fallback problem is real on current main, but this PR still has two correctness gaps around live model switches that are consumed inside the fallback runner. The diff should not merge until automatic fallback and explicit user switch outcomes are carried through without relying only on final provider/model comparison.

## What This Changes

The PR adds fallback-vs-primary guards around direct agent and isolated cron session runtime model persistence, plus regression tests for direct agent, cron, and session-store fallback no-persist behavior.

## Best Possible Solution

Carry an explicit selection-source or outcome through the fallback runner and cron executor, or persist live-switch targets before retry, so automatic fallback is transient while explicit user-selected models remain durable. Add regression coverage for both direct agent and cron internal redirect paths using the existing `runWithModelFallback` live-switch redirect case.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: make a primary attempt throw `LiveSessionModelSwitchError` to a later fallback candidate, let `runWithModelFallback` consume it as shown in `model-fallback.test.ts`, then observe the PR classify the returned target as fallback and skip the runtime write.

## Solution Assessment

No. The direction is relevant, but the implementation is not the safest fix because it infers source after the fact from provider/model equality; a durable source flag from the fallback runner or explicit live-switch persistence would avoid conflating fallback with user intent.
