## Summary

This PR addresses a real fallback persistence problem, but it should stay open because the current patch still conflates automatic fallback with live user-initiated model switches when the fallback runner handles the redirect internally. The provided discussion already caught one live-switch edge case and the follow-up commit improves the direct outer-retry path, but the source still has the same issue for fallback-internal redirects, including the cron path.

## What This Changes

The PR adds fallback-vs-primary guards around direct agent and isolated cron session runtime model persistence, extends `updateSessionStoreAfterAgentRun` with an `isFromFallback` flag, and adds regression tests for fallback no-persist and live-switch behavior.

## Best Possible Solution

Keep the fix focused on an explicit selection-source contract: automatic fallback must not become the durable next-turn selection, live user switches must persist even when handled inside `runWithModelFallback`, and direct-agent plus cron paths should consume the same reliable fallback outcome metadata.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a primary plus a later fallback candidate, trigger `LiveSessionModelSwitchError` from the primary to that later candidate, and observe `runWithModelFallback` consume the redirect before the PR's outer live-switch tracking sees it.

## Solution Assessment

No. The direction is right, but the patch is not the best safe fix because it infers fallback-ness after the run; a safer repair should carry explicit fallback/live-switch outcome metadata from the fallback runner or otherwise persist the live-switch target before retry.
