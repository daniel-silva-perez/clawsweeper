## Summary

Keep open: current main still has the runtime-model write path this PR targets, but the proposed guard is not merge-ready because it misses live-switch redirects handled inside `runWithModelFallback`; that can suppress persistence of an explicit user model switch.

## What This Changes

The PR adds fallback-vs-primary guards around direct agent and isolated cron session runtime model persistence, plus regression tests for fallback no-persist and live-switch behavior.

## Best Possible Solution

Keep the fix focused on the selection-source contract: automatic fallback must not become the durable next-turn user selection, explicit live switches must persist even when the fallback runner handles the redirect internally, and runtime/status fields should continue to reflect the intended product semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a primary plus fallback, have `runWithModelFallback` return a different provider/model, and current main writes that provider/model through `setSessionRuntimeModel`; for the PR regression, use the existing later-candidate `LiveSessionModelSwitchError` path in `model-fallback.test.ts`.

## Solution Assessment

No. The PR is directionally relevant, but the current implementation is not the best safe fix because it infers fallback-ness after the fact; a safer solution should carry an explicit outcome from the fallback runner or persist/mark live-switch targets before retry so automatic fallback and user intent cannot be conflated.
