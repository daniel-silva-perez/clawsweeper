## Summary

Keep this issue open. Current main still has Gmail/gog setup, watcher startup, and model-OAuth doctor coverage, but it does not provide the deterministic revoked-token recovery path, Gmail/Drive validation signal, or auth-state lifecycle UX requested here; the related open issue #43557 covers an overlapping cron/keyring symptom rather than superseding this broader operability report.

## What This Changes

The issue requests a durable Google/Gmail/Drive OAuth recovery baseline for production-like OpenClaw instances, including documented headless recovery, clearer CLI auth-state UX, loop prevention, and immediate Gmail/Drive validation after reauth.

## Best Possible Solution

Keep the issue open and add an OpenClaw-owned Gmail/gog operability path: a documented revoked-token/headless recovery runbook, auth-error diagnostics in the Gmail watcher/run path, and an optional validation command or checklist that proves Gmail read, Gmail send, and Drive access after reauth without changing core credential storage prematurely.

## Reproduction Assessment

No high-confidence live reproduction was performed for the exact invalid_grant state. Static reproduction is clear for the missing OpenClaw behavior: current docs, skill text, watcher code, and tests do not expose the requested recovery runbook or validation closure signal.

## Solution Assessment

Unclear for the full request because the right long-term boundary between OpenClaw and upstream gog OAuth needs maintainer judgment. The safest first step is a narrow docs plus diagnostic/validation improvement, reusing existing Gmail hook surfaces and avoiding token-store changes until upstream behavior is confirmed.
