## Summary

Keep open. Current main has a bundled `gog` Google Workspace skill plus OpenClaw Gmail Pub/Sub setup and Gateway watcher support, but it still assumes `gog` OAuth is already healthy. I did not find the deterministic revoked-token recovery path, stale/manual auth-state UX, or Gmail read/send plus Drive validation that this issue asks for.

## Best Possible Solution

Keep the issue open as an operability bug for the existing Gmail/gog surface. The best maintainer path is to add an OpenClaw-owned recovery runbook and/or CLI/doctor diagnostic that detects `gog` invalid_grant or stale auth state, guides one headless-safe reauth path, and optionally validates Gmail read, Gmail send, and Drive before declaring recovery. If the actual auth repair must live in `gog`, OpenClaw docs and watcher/setup errors should still point to the exact upstream recovery command and failure mode.
