## Summary

Keep open. Current main still has the requested exposure-matrix gap, and this PR remains the narrow canonical path, but the PR head has a concrete false-positive bug because it treats legacy `dm.policy="open"` as effective even when modern `dmPolicy` is explicitly safer.

## What This Changes

The PR changes the security audit exposure-matrix collector so open DM policies are included alongside open group policies in the existing elevated/runtime tool exposure findings.

## Best Possible Solution

Update the DM-policy collector to mirror effective policy resolution, then add focused tests proving modern `dmPolicy` wins over legacy `dm.policy` while legacy-only and account-level open DM policies still trigger the existing exposure checks.

## Reproduction Assessment

Yes. The patch-level reproduction is a config section with `dmPolicy: "allowlist"` and stale `dm: { policy: "open" }`; the PR head's line 351 still reports the legacy path even though the schema resolves the modern key first.

## Solution Assessment

No. Extending the exposure matrix to DMs is the right direction, but the current implementation should first honor modern-over-legacy policy precedence and add current-branch regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor dmPolicy before legacy dm.policy:** `src/security/audit-extra.sync.ts:351-352`
  - body: The new legacy scan reports `dm.policy="open"` even when the same config section explicitly sets a safer `dmPolicy`. Discord and Slack resolve these as `dmPolicy ?? dm.policy`, so a migrated config with `dmPolicy: "allowlist"` plus stale `dm.policy: "open"` would get a false CRITICAL exposure finding. Gate the legacy branch on the modern key being absent, including the account-level branch.
