## Summary

Current main does not fully implement this issue. It now distinguishes `modelOverrideSource` and clears auto fallback overrides on `/new`, `/reset`, and `sessions.reset`, but explicit `/model` selections are still intentionally preserved as user/legacy overrides. That leaves the reported core case, a user-set session model override continuing to beat changed config defaults across restarts and resets, still actionable.

## Best Possible Solution

Keep this issue open for a focused product/code decision. The clean implementation path is to either make `/new`, `/reset`, and `sessions.reset` clear user `providerOverride`/`modelOverride` as the issue requests, or keep preservation but add first-class user-facing visibility and clearing: `/status` should label config default versus session override, and chat/docs should expose an obvious `/model default` or `/model clear` path. Any code change should update the current preservation tests and docs under `docs/concepts/models.md`, `docs/concepts/session.md`, and `docs/tools/slash-commands.md`.
