## Summary

Keep open. Current main still makes the `/cron` History button an in-page state update only: it selects/loads the job runs while the Run history card remains lower in the same page, with no scroll, focus, drawer, modal, or other visible navigation feedback. The related context is adjacent Control UI/history work, not a canonical duplicate or merged fix for this specific UX problem.

## What This Changes

The issue requests clearer feedback when clicking the `/cron` page History button, such as scrolling/focusing the run-history section or opening a dedicated history panel.

## Best Possible Solution

Keep the issue open and add a narrow Control UI cron affordance: give the Run history section a stable target, then after a History click loads/selects a job, scroll and focus that section or otherwise provide equivalent visible navigation feedback. Add focused view coverage so the behavior is not only an `onLoadRuns` callback assertion.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: render the cron view with a job, click History, and observe that the handler only calls `onLoadRuns`; the updated Run history card remains lower on the same page.

## Solution Assessment

Yes for a repair lane: a stable scroll/focus target after the existing run-load path is the narrowest maintainable solution. A dedicated modal or drawer could also satisfy the report, but it is a larger design change than the issue requires.
