## Summary

Keep open. Current main still makes the cron History action an in-page run-history reload with no scroll/focus/modal feedback, and the existing tests only assert that the click loads/selects history rather than making the lower panel visible.

## What This Changes

Make the Control UI /cron History action visibly take the user to run history, such as by scrolling/focusing the run-history section or opening a dedicated history panel.

## Best Possible Solution

Keep the existing cron backend and run-history panel, but make the explicit History button move attention to that panel after loading a job's runs, preferably by scrolling and focusing a stable Run history section target with focused view coverage. A modal or drawer should be reserved for a broader Control UI redesign.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction path: render the /cron page with enough jobs or viewport height for the Run history card to sit below the fold, click a job's History button, and current main only reloads/selects history without moving the viewport or focus.

## Solution Assessment

Yes. The reporter's scroll/focus option is the narrowest maintainable fix because the cron API and run-history rendering already exist; adding a modal/drawer would solve the confusion but expands the UI surface unnecessarily.

Current main renders the Run history card below the jobs list and the History button only calls `props.onLoadRuns(job.id)`, so on shorter viewports the click can look like a no-op.
