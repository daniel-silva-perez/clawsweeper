## Summary

Keep this issue open because current main still renders the Usage tab through both the shared content header and the Usage view's local header. The related open PR #67290 targets the same regression, but it is not merged, so the reported behavior is still present on main.

## What This Changes

The issue requests fixing the Control UI v2 Usage tab so its header/title area is rendered only once.

## Best Possible Solution

Land #67290 or an equivalent narrow Control UI patch that removes exactly one Usage header source, keeps the shared/local header choice consistent with the surrounding tab architecture, and adds a focused render regression check proving the Usage tab exposes only one title/subtitle region.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: set the Control UI tab to usage and the app shell emits content-header while renderUsageTab calls renderUsage, which emits usage-page-header inside the tab body.

## Solution Assessment

Yes. The narrow maintainable fix is to remove one of the two header sources rather than redesign the Usage page; if the local subtitle key is removed, the English source and generated locale outputs should be kept in sync.
