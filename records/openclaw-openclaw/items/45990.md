## Summary

Keep open. Current main still emits raw `llm_input` before the embedded prompt submission and raw `llm_output` once after the attempt with aggregate text/usage, so the PR’s intended per-round raw hook behavior is not implemented on main. The proposed branch is useful source work for #64268, but it introduces blocking regressions around failed prompt observability and usage normalization, so it needs maintainer API-contract review or author revision before merge.

## What This Changes

The PR moves `llm_input` and `llm_output` hook emission from embedded attempt start/end into assistant `message_start`/`message_end` handling, adds subscription hook context fields and per-round tests, and updates hook comments.

## Best Possible Solution

Define a stable per-call raw observability contract alongside #64268, then revise this branch so input hooks still cover failed provider submissions, per-call output usage is normalized, aggregate/final-run semantics are preserved or separately labeled, and plugin docs/changelog clearly explain the migration.

## Reproduction Assessment

Yes. The blocking findings reproduce by inspection and focused unit tests: make `activeSession.prompt(...)` fail before any assistant `message_start`, and feed assistant usage shaped like `{ totalTokens: 42 }` through the proposed per-round output path.

## Solution Assessment

No. The PR moves raw input emission too late for failed submissions and casts output usage more narrowly than the existing normalization contract; a safer solution keeps input at the prompt/provider-call boundary and normalizes or explicitly versions per-call output events.

## Review Findings

- **[P2] Emit input hooks before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:72`
  - body: Moving `llm_input` to `message_start` means provider failures before the first assistant stream event never emit an input hook. Current main emits before prompt submission, so tracing plugins can capture auth, rate-limit, network, or provider-rejection failures; this PR would lose that data for the failing rounds users most need to diagnose.
  - confidence: 0.92
- **[P2] Normalize assistant usage before emitting output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: The new output helper forwards `message.usage` through a narrow cast instead of the existing `normalizeUsage` path. Provider aliases such as `totalTokens`, `prompt_tokens`, cache-token fields, and timing-shaped usage can therefore drop `usage.total` or token breakdowns for per-round `llm_output` consumers.
  - confidence: 0.88
