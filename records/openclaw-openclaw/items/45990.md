## Summary

Keep open. Current main still emits raw `llm_input` once before prompt execution and raw `llm_output` once after attempt completion, so the PR is not obsolete; however, the branch has merge-blocking regressions around pre-stream provider failures and usage normalization, and the public plugin hook contract should be settled with the related per-call observability issue before merge.

## What This Changes

The PR moves raw `llm_input` and `llm_output` hook emission from the embedded attempt boundary into assistant message stream start/end handling, extends subscription parameters/context, updates hook comments, and adds per-round hook coverage.

## Best Possible Solution

Define a stable raw per-call observability contract alongside the related open issue, then revise the branch so input hooks still cover failed provider submissions, output usage is normalized, per-call identity is explicit, aggregate/final-run semantics are preserved or separately labeled, and docs/changelog explain the plugin-visible migration.

## Review Findings

- **[P2] Emit input hooks before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:72`
  - body: Moving `llm_input` to `message_start` means provider failures before the first assistant stream event no longer emit an input hook. The current attempt-level hook fires before prompt submission, so tracing plugins can still capture auth, rate-limit, network, or provider-rejection failures; this PR would lose that data for the failing rounds users most need to diagnose.
  - confidence: 0.92
- **[P2] Normalize assistant usage before emitting output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: The new output helper forwards `message.usage` through a narrow cast instead of the existing `normalizeUsage` path. Provider aliases such as `totalTokens`, `prompt_tokens`, cache-token fields, and timing-shaped usage can therefore drop `usage.total` or token breakdowns for per-round `llm_output` consumers.
  - confidence: 0.88
