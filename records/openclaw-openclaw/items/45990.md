## Summary

Keep this PR open. Current main still emits raw `llm_input` once before embedded prompt execution and `llm_output` once after the attempt, so the requested per-round raw hook behavior is not implemented; however, the proposed patch has blocking correctness issues around losing input hooks for pre-stream provider failures and bypassing existing usage normalization.

## What This Changes

The PR moves `llm_input`/`llm_output` emission from embedded attempt start/end into assistant message stream start/end handling, extends subscription hook context, updates hook comments, and adds per-round hook test coverage.

## Best Possible Solution

Define a stable raw per-call observability contract alongside #64268, then revise the branch so input hooks still cover failed provider submissions, output usage is normalized, per-call identity or round index is explicit, aggregate/final-run semantics are preserved or separately labeled, and plugin docs/changelog explain the migration.

## Reproduction Assessment

Yes. The two blocking findings are reproducible by inspection and with focused tests: make `activeSession.prompt(...)` fail before any assistant `message_start`, and emit assistant usage shaped like `{ totalTokens: 42 }` through the proposed per-round output path.

## Solution Assessment

No. Moving raw input emission to `message_start` is too late for failed provider submissions, and casting `message.usage` is narrower than the existing normalization contract; a safer solution keeps input at the prompt/provider-call boundary and normalizes per-call output usage before dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Emit input hooks before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:72`
  - body: Moving `llm_input` to `message_start` means provider failures before the first assistant stream event no longer emit an input hook. The current attempt-level hook fires before prompt submission, so tracing plugins can still capture auth, rate-limit, network, or provider-rejection failures; this PR would lose that data for the failing rounds users most need to diagnose.
  - confidence: 0.92
- **[P2] Normalize assistant usage before emitting output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: The new output helper forwards `message.usage` through a narrow cast instead of the existing `normalizeUsage` path. Provider aliases such as `totalTokens`, `prompt_tokens`, cache-token fields, and timing-shaped usage can therefore drop `usage.total` or token breakdowns for per-round `llm_output` consumers.
  - confidence: 0.88
