## Summary

Keep this PR open for maintainer review rather than cleanup-closing it. Current `main` still emits `llm_input`/`llm_output` at the attempt boundary and does not implement the PR's per-round raw conversation hook behavior, but the branch changes a public plugin contract and has concrete blocking review findings around failed-prompt observability and usage normalization.

## What This Changes

The PR moves embedded-runner `llm_input` and `llm_output` hook emission from attempt-level start/end into subscription `message_start`/`message_end`, threads provider/model/context data into the subscription layer, and updates hook tests and comments for per-round semantics.

## Best Possible Solution

Define a stable per-LLM-call observability contract alongside #64268, then revise this branch so input hooks still cover failed provider submissions, per-call output usage is normalized, aggregate/final-run semantics are preserved or explicitly renamed, and plugin docs/changelog explain the migration.

## Reproduction Assessment

Yes. The blocking issues reproduce by inspection and focused tests: make `activeSession.prompt(...)` fail before any assistant `message_start`, and feed assistant usage shaped like `{ totalTokens: 42 }` through the proposed per-round output path.

## Solution Assessment

No. The direction is useful, but moving raw input emission to `message_start` is later than the provider-call boundary and the output helper bypasses the normalized usage contract; a safer fix keeps failed-submission observability and labels or versions per-call versus aggregate output semantics.
