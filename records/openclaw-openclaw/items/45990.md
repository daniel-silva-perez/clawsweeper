## Summary

Keep this PR open. Current main still does not implement raw per-round `llm_input`/`llm_output`, and the PR is a plausible implementation candidate, but the proposed patch has blocking contract/regression issues that need maintainer/API follow-up before merge.

## What This Changes

The branch moves raw `llm_input`/`llm_output` emission from `runEmbeddedAttempt` into embedded session `message_start`/`message_end` handling and extends subscription params, tests, and hook comments for per-round events.

## Best Possible Solution

Define the public hook contract alongside #64268, then update the implementation to emit per-call raw observability with a stable `roundIndex` or `callId`, normalized per-call usage, preserved or separately labeled final aggregate semantics, and regression tests for multi-round success plus failures before streaming starts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Emit input hooks before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:72`
  - body: Triggering `llm_input` only from `message_start` means provider failures before the first assistant stream event no longer produce an input hook. The previous attempt-level emission ran before `activeSession.prompt(...)`, so tracing plugins could still capture auth, rate-limit, network, or provider-rejection failures.
  - confidence: 0.92
- **[P2] Normalize per-round usage before emitting output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: The new output helper forwards `message.usage` through a narrow cast instead of `normalizeUsage`. Existing usage normalization accepts provider aliases such as `totalTokens`, `prompt_tokens`, and cache-token fields, so per-round hook consumers can lose `usage.total` or token breakdowns when the assistant message is not already in normalized shape.
  - confidence: 0.88

## Risks / Open Questions

- The proposed `llm_output` payload changes observable plugin semantics from final aggregate output/usage to per-round output/usage without a compatibility signal or migration path.
- Moving `llm_input` emission to stream `message_start` misses provider failures that happen before the first stream event.
- The public API shape is still unsettled between reusing raw hooks per call, adding a new `llm_call` hook, or preserving aggregate hooks plus per-call events.
- A revised patch must keep raw conversation data behind the existing `allowConversationAccess` policy and make an explicit decision about raw-model/probe runs.
