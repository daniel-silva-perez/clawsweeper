## Summary

Keep open. The branch has a useful narrow intent, but current main has not already landed it, the linked bloat report now points at a different explicit-config/plugin-discovery path, and the proposed patch introduces blocking correctness and security concerns around the omitted-config fallback.

## What This Changes

The PR changes `resolveModelsConfigInput(config?)` so an omitted config plans `models.json` from the loaded runtime config while keeping the runtime source snapshot for secret markers, and adds a regression test for the omitted-config path.

## Best Possible Solution

Keep the PR open only for maintainer/author follow-up: either revise it into a tested fallback-path fix that plans from a true lean source config while preserving all SecretRef markers across provider `request.*`, or close it manually in favor of a separate provider-discovery fix for the empty-provider bloat described in #65020.

## Reproduction Assessment

Yes for the review findings. Static inspection reproduces the test failure path, and the code trace from `getRuntimeConfig()` through the secrets runtime shows how the proposed fallback can plan from a resolved runtime snapshot containing plaintext request secrets.

## Solution Assessment

No. This is not the best fix as proposed because it targets an omitted-config branch while the linked startup bloat path uses explicit config, and the fallback change weakens SecretRef persistence unless the full provider request surface is source-managed first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep omitted config from writing resolved request secrets:** `src/agents/models-config.ts:121`
  - body: When this branch uses `config: loaded`, `loaded` can be the active resolved runtime snapshot from `getRuntimeConfig()`, not a lean source config. Provider `request.*` SecretRefs are resolved to plaintext there, while the models config marker pass only restores `apiKey` and top-level `headers`, so an omitted `ensureOpenClawModelsJson()` call can serialize request tokens, headers, or TLS material into `models.json`. Use a true source config for planning or restore markers for the full request surface before changing this path.
  - confidence: 0.88
- **[P1] Fix the added regression test harness call:** `src/agents/models-config.runtime-source-snapshot.test.ts:376-389`
  - body: The new test calls `withTempHome` without importing it in this file, then calls `expectGeneratedProviderApiKey("openai", "OPENAI_API_KEY")` even though the helper requires `(agentDir, providerId, expected)`. As written, the added test will fail at runtime or during test typechecking instead of validating the fallback behavior.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff can persist resolved provider request secrets into `models.json` when the omitted-config branch runs with an active runtime snapshot.

Concerns:

- **[high] Plaintext provider request secrets can be written:** `src/agents/models-config.ts:121`
  - body: Switching the planned config to `loaded` can feed the resolved runtime snapshot into `models.json` generation. Current source-managed enforcement only restores provider `apiKey` and top-level `headers`, so nested `request.auth`, `request.headers`, `request.tls`, and proxy TLS secrets can remain plaintext in the serialized provider config.
  - confidence: 0.86
