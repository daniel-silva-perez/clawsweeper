## Summary

Keep this PR open, but do not merge it as-is. The linked bug is real on current main, yet the proposed patch still misses the cold-session `/tools` reproduction, leaks the MCP materialization lease, and mixes in unrelated plugin-install changes on a sensitive dependency path.

## What This Changes

The PR adds an `mcp` effective-tool source, appends materialized session MCP tools to `/tools`, and also changes plugin package install peer-dependency handling and its test expectation.

## Best Possible Solution

Land a focused `/tools` MCP inventory fix that uses the same naming, policy, and lifecycle semantics as the run path, covers both cold and already-warm sessions, releases any materialized lease after extracting display metadata, and removes the installer dependency changes from this PR.

## Reproduction Assessment

Yes. Configure `mcp.servers`, start a new session, and run `/tools verbose` before an agent turn: current main renders only `resolveEffectiveToolInventory()`, while the actual run path later materializes MCP tools separately.

## Solution Assessment

No. The proposed direction is close, but this patch only lists MCP tools after a runtime has already been cached, does not dispose the materialized MCP lease, and includes unrelated installer changes that should be split or removed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Create MCP inventory for cold sessions:** `src/auto-reply/reply/commands-info.ts:170-171`
  - body: This guard means `/tools` only appends MCP tools after a previous agent turn has already created and bound an MCP runtime. The reported reproduction runs `/tools verbose` in a new session before using MCP, so configured `mcp.servers` tools are still omitted.
  - confidence: 0.9

  - body: `materializeBundleMcpToolsForRun()` acquires a runtime lease and releases it only via the returned `dispose()`. This command keeps just display metadata, so it should dispose after building the inventory or repeated `/tools` calls can keep active leases and delay idle cleanup.
  - confidence: 0.84
- **[P2] Keep peer stripping reachable:** `src/infra/install-package-dir.ts:65-72`
  - body: The new `peerDependencies.openclaw` removal sits after early returns that require workspace devDependencies to be rewritten. Packages with only an `openclaw` peer, or no workspace devDependency entries, still keep that peer and do not get the stated bloat fix.
  - confidence: 0.82

  - body: The test now expects `--omit=peer`, but the production command still omits only dev dependencies and the nearby comment says preserving other peers is intentional. As written, the test fails or pressures production toward dropping legitimate runtime peers.
  - confidence: 0.93
