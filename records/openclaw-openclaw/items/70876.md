## Summary

Keep this PR open. Current main still omits configured MCP server tools from `/tools`, but the PR is not merge-ready because it only lists MCP tools after a session runtime has already been cached and it mixes the inventory fix with unrelated plugin install dependency-resolution changes.

## What This Changes

The PR adds an MCP tool source/group to the effective tool inventory, augments `/tools` to append materialized session MCP tools, and also changes plugin package install peer-dependency handling and tests.

## Best Possible Solution

Implement a focused `/tools` MCP inventory fix that shares the run path's name collision handling, `bundle-mcp` ownership, final policy filtering, and compact/verbose rendering, with regression tests for first-run and already-warm sessions. Keep plugin package install dependency-resolution changes out of this PR and handle them separately if still needed.

## Reproduction Assessment

Yes. Current main can be checked by configuring `mcp.servers` and running `/tools verbose` before an agent turn: the command handler only renders `resolveEffectiveToolInventory()`, while the run path separately materializes MCP tools. The PR still misses that first-run reproduction because it only proceeds when `resolveSessionId(params.sessionKey)` already has a cached runtime mapping.

## Solution Assessment

No. The proposed direction is close, but this implementation is not the narrowest maintainable fix because it depends on an already-existing runtime, does not release the materialized MCP lease after building read-only inventory, and includes unrelated installer changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Create MCP inventory before a runtime is cached:** `src/auto-reply/reply/commands-info.ts:170-171`
  - body: This guard means `/tools` only appends MCP tools after `getOrCreateSessionMcpRuntime()` has already run for the session. In the linked reproduction, `/tools verbose` is run in a new session before asking the agent to use MCP, so `resolveSessionId()` is empty and the command still omits configured `mcp.servers` tools.
  - confidence: 0.87
- **[P2] Release the materialized MCP lease:** `src/auto-reply/reply/commands-info.ts:182-185`
  - body: `materializeBundleMcpToolsForRun()` acquires a runtime lease and releases it only through the returned `dispose()`. This command keeps only metadata and never disposes `mcpTools`, so repeated `/tools` calls can leave active leases behind and block idle runtime cleanup.
  - confidence: 0.78
- **[P2] Keep the installer test aligned with the command:** `src/infra/install-package-dir.test.ts:323`
  - body: The test now expects `--omit=peer`, but the implementation still runs npm without that flag and explicitly comments that omitting all peers would break legitimate runtime peer imports. As written, the updated test fails or pressures production toward the unsafe behavior the implementation avoids.
  - confidence: 0.92
