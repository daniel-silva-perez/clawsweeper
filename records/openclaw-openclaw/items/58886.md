## Summary

Keep this PR open. Current main still has a mid-turn transient-error path that formats rate-limit and overload messages without the active provider, but the submitted hunk is stale against the current shared formatter and should be adapted before merge.

## What This Changes

The PR branch changes `src/auto-reply/reply/agent-runner-execution.ts` to derive a display label from `fallbackProvider` and prepend it to overload and rate-limit reply payloads.

## Best Possible Solution

Rebase or replace the PR with a provider-aware formatter path that accepts the active provider/model, preserves specific upstream rate-limit details, falls back to the existing generic text when provider context is absent, and uses stable display labels for common provider IDs.

## Reproduction Assessment

Yes. The current-main test fixture constructs a Telegram run with `provider: "anthropic"` and a mid-turn `429 Too Many Requests` error, while the asserted output remains the generic `API rate limit reached` text.

## Solution Assessment

No, not as-is. The requested product behavior is reasonable, but the implementation should be moved into or alongside `formatRateLimitOrOverloadedErrorCopy` on current main so it does not discard specific rate-limit messages or produce rough provider branding.
