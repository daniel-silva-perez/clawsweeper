## Summary

Current main still implements `/v1/models` as an OpenAI-compatible agent-target list, while provider/model discovery exists only through the Gateway `models.list` RPC and Control UI path. The requested HTTP discovery surface remains valid but needs maintainer API/product judgment because changing `/v1/models` would conflict with the documented agent-first contract.

## Best Possible Solution

Keep this open for maintainer API design. The likely path is an authenticated, read-scoped, additive HTTP discovery endpoint or explicit extension fields backed by the existing `loadGatewayModelCatalog()` / `models.list` behavior, while preserving the documented agent-first `/v1/models` contract unless maintainers intentionally choose a compatibility break. The design should define whether returned models are configured, discovered, allowed, or live-probed reachable.

## Risks / Open Questions

- Replacing `/v1/models` outright would break the documented agent-target contract used by OpenAI-compatible clients that select `openclaw/default` or `openclaw/<agentId>`.
- The issue asks for models that are actually reachable; current catalog and allowlist code can represent configured or discovered models, but live reachability would need explicit probe, cache, timeout, and error semantics.
- A new HTTP discovery surface should preserve the existing operator read-scope/auth boundary and avoid leaking provider configuration details beyond intended operator clients.
