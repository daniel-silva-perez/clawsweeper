## Summary

Keep open. Current main still exposes terminal-looking session `status` values as the row/tool contract without an additive resumability or status-scope signal, and the agent-facing docs/system prompt still do not clearly tell orchestrators that existing visible sessions can usually be resumed with `sessions_send`.

## What This Changes

The issue requests clearer session status semantics or additive metadata so orchestrator agents do not mistake last-turn states such as `failed`, `timeout`, and `done` for permanently dead sessions.

## Best Possible Solution

Add a backward-compatible session row/tool contract that preserves existing `status` values while making resumability and status scope explicit, then update the tool descriptions, system prompt, and public session-tool docs so orchestrators prefer `sessions_send` for visible existing sessions unless the session is inaccessible or intentionally removed.
