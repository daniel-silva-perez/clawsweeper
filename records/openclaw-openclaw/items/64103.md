## Summary

Keep open. Current main and the latest release still expose terminal-looking session status values through the persisted session, gateway row, and agent-facing `sessions_list` contracts without an explicit resumability or status-scope signal, and the docs/system prompt still do not tell orchestrators how to interpret those values.

## What This Changes

The issue requests clearer session status semantics or additive metadata so orchestrator agents do not mistake last-turn states such as `failed`, `timeout`, and `done` for permanently dead sessions.

## Best Possible Solution

Add a backward-compatible session row/tool contract that preserves existing `status` values while making resumability and status scope explicit, then update the tool descriptions, system prompt, and public session-tool docs so orchestrators prefer `sessions_send` for visible existing sessions unless the session is inaccessible, deleted, or policy-denied.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to list or construct a subagent row with `status: "failed"`, `"timeout"`, or `"done"`; current main returns that bare status through `sessions_list` with no resumability/scope metadata, matching the agent-facing ambiguity in the report.

## Solution Assessment

Yes. The narrow maintainable solution is a non-breaking additive contract plus prompt/docs guidance; renaming the existing `status` values would be cleaner conceptually but is riskier for current clients.
