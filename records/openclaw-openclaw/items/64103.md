## Summary

Keep open. Current main still exposes `running | done | failed | killed | timeout` as the session row/status contract without an additive resumability or status-scope field, and the agent-facing docs/tool descriptions still do not clearly tell orchestrators that terminal-looking session statuses can be followed up with `sessions_send`.

## What This Changes

The issue asks OpenClaw to make session status semantics clear enough that orchestrator agents resume existing sessions instead of spawning duplicates after `failed`, `timeout`, or `done` statuses.

## Best Possible Solution

A backward-compatible session API should make each listed row self-describing: preserve the existing `status` values, add an explicit resumability/status-scope signal, and update agent tool descriptions plus https://docs.openclaw.ai/concepts/session-tool so orchestrators prefer `sessions_send` for visible existing sessions unless the session is gone or inaccessible.

## Repair Work Prompt

Fix openclaw/openclaw#64103 with a narrow backward-compatible API/docs/test PR. Current `sessions_list` and gateway session rows expose `status: running|done|failed|killed|timeout`, which orchestrator agents can misread as session lifecycle/deadness and then spawn duplicate sessions. Do not rename or remove existing status strings, do not add startup/load-time migrations, and do not broaden into compaction recovery or lifecycle cleanup. Add an explicit additive signal to the agent-facing session row contract, preferably `resumable: boolean`, with semantics documented as true for visible existing sessions that can be targeted by `sessions_send` and false/omitted only when the row cannot be sent to under existing access/policy rules. If a separate status-scope/name field is needed, keep it additive and generic; do not break clients that branch on the old `status` values. Update the gateway row type/building path and `sessions_list` row type/output, plus focused tests that prove rows with `done`, `failed`, `timeout`, and `killed` expose the resumability/scope signal and still preserve the old status value. Update `src/agents/tool-description-presets.ts` and `docs/concepts/session-tool.md` so agents are told that `status` is not by itself a reason to spawn a replacement and that they should use `sessions_send` for a visible existing session before `sessions_spawn`. Consider `docs/concepts/system-prompt.md` only if the runtime prompt text is generated from or documented there. Add a changelog fix entry if repo policy treats this as user-facing, crediting `Thanks @lykeion-dev`. Validate with targeted session/gateway tests and formatting; run the changed gate in Testbox before handoff if available.
