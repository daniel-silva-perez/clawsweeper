## Summary

Keep open. Current main still has the agent-facing browser tool gap this PR targets, while the PR diff is a narrow implementation that wires existing managed download endpoints into the tool surface; I found no discrete blocking correctness finding in the patch, but it needs maintainer review and CI/security validation before merge.

## What This Changes

The PR adds `download` and `waitfordownload` actions to the browser tool schema/routing, exports existing download helpers, adjusts download request/proxy timeouts, updates browser docs/changelog text, and adds browser-tool/client-helper regression tests.

## Best Possible Solution

Land this PR, or a narrow replacement, after confirming the new agent actions route only through the existing managed download API, preserve download-path restrictions, keep existing-session profiles unsupported, and document the agent action surface alongside the CLI/control API.

## Reproduction Assessment

Yes. On current main, the agent tool schema lacks `download`/`waitfordownload` and the switch has no matching cases, so an agent call for those actions cannot route even though the lower-level `/download` and `/wait/download` endpoints exist.

## Solution Assessment

Yes. The PR is the narrow maintainable path: expose the already-supported managed download endpoints at the agent tool boundary, reuse the existing server-side path/profile constraints, and add host/node routing plus timeout regression coverage.
