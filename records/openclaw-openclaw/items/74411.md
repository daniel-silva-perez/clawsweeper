## Summary

Keep open. Current main still lacks the agent-facing browser download actions, while the lower-level managed browser download endpoints and path guards already exist; this PR is a focused implementation of that missing surface. I did not find a concrete correctness blocker in the diff, but it should receive maintainer approval and normal CI because it exposes an existing managed download/file-write capability to agents.

## What This Changes

The PR adds `download` and `waitfordownload` to the bundled browser agent tool schema and routing, exports existing download helpers, adjusts host/node download timeouts, documents the action surface, adds tests, and updates the changelog.

## Best Possible Solution

Land this PR, or a narrow equivalent, after maintainer approval and CI confirm the new agent actions only route through the existing managed download API, preserve guarded download-path handling, keep existing-session profiles unsupported, and document the action surface beside the browser control API.

## Reproduction Assessment

Yes. Current main reproduces the gap by inspection: the browser tool schema and routing omit `download` and `waitfordownload`, while the managed `/download` and `/wait/download` endpoints already exist.

## Solution Assessment

Yes. The proposed approach is the narrow maintainable fix because it wires existing browser download helpers into the agent tool contract and aligns transport timeouts without changing the lower-level download implementation or path guards.
