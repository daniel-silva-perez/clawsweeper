## Summary

Keep open. Current main has useful session-list, labeling, navigation, deletion, and backend creation primitives, but the visible WebChat/Control UI New session button still routes through reset semantics instead of creating and switching to a preserved dashboard session. The open related PR #73725 is the right implementation path to review rather than closing this issue now.

## What This Changes

Request WebChat/Control UI multi-session management so users can list, switch, preserve, name, and delete conversations instead of losing the old conversation when starting a new chat.

## Best Possible Solution

Land #73725 or an equivalent narrow change that makes the Control UI/WebChat plus button call sessions.create, switches Chat to the returned session key, refreshes session options, preserves the old session in sessions.list, and leaves /new or sessions.reset as explicit reset behavior with focused UI/controller tests and docs updates.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: clicking the New session affordance calls the same onNewSession callback that sends /new, and that reset path rewrites the active store key instead of creating a new switchable session.

## Solution Assessment

Yes for the direction, not yet for current main. Reusing the existing sessions.create backend contract and switchChatSession UI path is the narrowest maintainable fix; adding another metadata store or changing reset semantics would be broader than needed.
