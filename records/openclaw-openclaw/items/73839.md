## Summary

Keep this PR open: the bug is valid and the patch is in the right area, but the latest head still misses provider-backed safeguard mode because it passes only the raw `compaction.mode` into the Pi guard. The earlier CLI excess-property issue from Greptile appears addressed in the latest diff, so the remaining blocker is the resolved safeguard predicate plus the required changelog entry.

## What This Changes

The PR updates the Pi auto-compaction guard to accept compaction mode, disables Pi auto-compaction for explicit safeguard mode, threads that mode through embedded and CLI compaction paths, reapplies the guard after embedded resource reload, and adds focused `pi-settings` tests.

## Best Possible Solution

Land a narrow fix that reuses or exposes one provider-aware “OpenClaw owns compaction” resolver for both explicit safeguard mode and provider-forced safeguard mode, then use that resolved value in embedded and CLI guard calls while preserving context-engine `ownsCompaction` behavior. Add focused coverage for explicit safeguard, provider-backed safeguard with raw default mode, default mode, and post-reload guard reapplication, plus a one-line changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists with `agents.defaults.compaction.provider` set while raw `compaction.mode` is absent or `"default"`: current extension resolution enables safeguard, but the PR guard would see only the raw mode and leave Pi auto-compaction enabled.

## Solution Assessment

No. The PR is close, but the best fix should share the provider-aware safeguard predicate instead of passing only `agents.defaults.compaction.mode`; otherwise one supported safeguard path still hits the same Pi/OpenClaw compaction conflict.
