## Summary

Keep open. The user-facing symlink warning problem is still not implemented on current main, but this PR should not merge as-is because it broadens skills containment to bundled targets without preserving bundled source semantics, which can bypass `skills.allowBundled`.

## What This Changes

The PR updates the skills workspace loader to accept symlinked skill paths whose realpaths resolve under configured skill roots and adds regression coverage for workspace and managed symlink cases.

## Best Possible Solution

Implement a single source-preserving symlink policy for configured skill roots: allow cross-root links among explicitly trusted non-bundled roots, keep external escapes rejected, and either exclude bundled targets from non-bundled aliases or relabel them so bundled allowlist checks still apply. Add coverage for cross-root success, external rejection, bundled allowlist enforcement, grouped skills, and docs for the final contract.

## Reproduction Assessment

Yes. Use a bundled skill excluded by `skills.allowBundled`, then create a workspace or managed symlink to that bundled skill; this PR would accept the realpath while the local loader keeps the entry source non-bundled.

## Solution Assessment

No. Accepting configured-root realpaths may be a good direction for non-bundled roots, but including `bundledSkillsDir` without preserving bundled source classification is not the narrowest maintainable fix; excluding bundled roots or relabeling bundled targets is safer.
