## Summary

Keep open. Current main still only scans JavaScript/TypeScript-family extensions, so the PR addresses a real remaining scanner gap, but the proposed patch is not merge-ready because extensionless shell entrypoints and variable-based Python command execution can still bypass the new rules. Related broader scanner hardening in #63826 is still open, not a merged superseding fix.

## What This Changes

The PR extends the skill scanner and tests to scan `.py`, `.sh`, and `.bash` files, add Python and shell detection rules, and add line-rule extension include/exclude guards.

## Best Possible Solution

Revise this PR, or replace it with a focused reviewed scanner-hardening patch, so Python and shell coverage includes extension-based files plus extensionless shebang or declared entrypoints, and Python command-exec matching catches non-literal `os.*` payloads without reintroducing noisy subprocess false positives.

## Reproduction Assessment

Yes. The current-main gap is reproducible by inspecting `isScannable()` and the extension set, and the PR-specific bypasses are reproducible from the diff: extensionless shell files still fail the extension gate, and `os.system(cmd)` does not match the quoted-literal Python regex.

## Solution Assessment

No. The direction is right for the reported blind spot, but this patch is not the best complete fix until it handles extensionless shell entrypoints and non-literal Python command payloads, with regression tests for both bypasses and false-positive boundaries.
