## Summary

Keep open. Current main still only scans JS/TS-family files, so the reported Python and shell blind spot remains real; however, this PR is not ready to merge because the patch leaves extensionless shell entrypoints outside the scanner and misses non-literal Python command payloads. Related #63826 overlaps scanner hardening, but it does not supersede this narrower Python/shell coverage.

## What This Changes

The PR extends the built-in skill scanner and tests to scan `.py`, `.sh`, and `.bash` files with new Python and shell detection rules plus line-rule extension guards.

## Best Possible Solution

Revise this PR, or replace it with a focused reviewed scanner-hardening patch, so Python and shell coverage includes extension-based files plus extensionless shebang or declared entrypoints, and Python command-exec matching catches non-literal `os.*` payloads without reintroducing noisy subprocess false positives.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main excludes `.py`, `.sh`, and `.bash`, and the PR head still skips extensionless files via `path.extname()` while `os.system(cmd)` does not match the quoted-literal Python regex.

## Solution Assessment

No. The direction is right, but this patch is not the best complete fix until it handles extensionless shell entrypoints and non-literal Python command payloads with regression tests for both bypasses and false-positive boundaries.
