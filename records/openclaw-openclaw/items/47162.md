## Summary

Keep open. Current main still has the reported JS/TS-only scanner gap, and this PR is a plausible core security hardening change, but the current branch still leaves concrete Python and shell bypasses in the new coverage. The second commit addressed earlier review feedback around duplicate Python eval findings, subprocess false positives, network-method coverage, and raw.githubusercontent.com noise; the remaining action is revision or maintainer security review, not closure.

## What This Changes

The PR extends the skill scanner and its tests to scan `.py`, `.sh`, and `.bash` files, add Python and shell detection rules, and add line-rule include/exclude extension guards.

## Best Possible Solution

Revise this PR, or replace it with an equivalent focused scanner-hardening patch, so Python and shell coverage includes extension-based files plus extensionless shebang or declared entrypoints, Python command-exec matching catches non-literal `os.*` payloads without reintroducing noisy subprocess false positives, and regression tests cover both bypasses and false-positive boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scan extensionless shell entrypoints:** `src/security/skill-scanner.ts:49-50`
  - body: Adding `.sh` and `.bash` leaves the scanner extension-gated through `path.extname()`, including for forced entry files. Common shebang entrypoints such as `bin/install` or `postinstall` still skip the new shell rules during install and audit scans, so this does not cover a common shell-script shape.
  - confidence: 0.93
- **[P2] Match non-literal Python command payloads:** `src/security/skill-scanner.ts:186`
  - body: The new `os.system`/`os.popen`/`os.exec*` arm only matches quoted first arguments. A script can assign the command to a variable and call `os.system(cmd)`, which is the same dangerous behavior but avoids the new critical finding.
  - confidence: 0.88
