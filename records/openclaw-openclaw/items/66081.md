## Summary

Keep this PR open. Current main still finalizes `codex-cli` runs from stdout/JSONL only, and the bundled backend does not request `--output-last-message`. The PR remains a plausible implementation candidate, but the proposed diff has unresolved command-ordering, parser-scoping, temp-cleanup, and verbose-log issues in a command execution path.

## Best Possible Solution

Keep this PR open and request a scoped revision, or have a maintainer apply the same behavior directly on main: add a Codex-only `--output-last-message` artifact before positional prompt args, prefer a non-empty artifact for final reply text while preserving stdout-derived session/usage/raw text, scope Codex JSONL content fallback to `codex-cli` or an explicit dialect, mask the artifact path in verbose logs, create and clean the artifact inside the existing cleanup block, and keep focused tests for artifact precedence, empty-artifact fallback, and Codex JSONL content parsing.

## Risks / Open Questions

- Merging the PR as provided could fold unrelated `message`, `content`, `result`, or `response` fields from future JSONL backends into final reply text unless the fallback is scoped to `codex-cli` or a named dialect.
- Appending `--output-last-message` after the positional prompt can be fragile for `input: "arg"` backends; the flag should be inserted before prompt placement or through a backend-owned base-arg seam.
- The new temp artifact should be created inside the existing guarded cleanup flow so failures do not bypass cleanup for system-prompt files, image payloads, or Claude skills plugin resources.
- Verbose CLI argv logging should mask or abbreviate the artifact path for consistency with other injected file/path arguments.
