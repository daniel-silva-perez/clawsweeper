## Summary

Keep open. Current `main` has improved trusted span tracking for child lifecycle spans, but the root `openclaw.run` path can still start without an OpenTelemetry parent context when a non-HTTP run has only OpenClaw diagnostic trace context, so the exported trace can still diverge from `evt.trace.traceId`.

## What This Changes

The issue asks for the bundled `diagnostics-otel` plugin to create spans under the trusted diagnostic event trace so run, model-call, and tool spans share `evt.trace.traceId` and Cloud Logging trace links resolve.

## Best Possible Solution

Update `diagnostics-otel` so trusted lifecycle spans fall back to a parent context derived from the OpenClaw diagnostic trace when no active exported parent span exists, while keeping active exported-span parenting for child topology and rejecting untrusted event trace injection. Add focused tests for non-HTTP run roots, child spans, and untrusted events.

## Reproduction Assessment

Yes. A source-level reproduction is clear: emit a trusted non-HTTP `run.started` event whose `trace.parentSpanId` is not an active exported span, then observe that current `recordRunStarted` calls `tracer.startSpan` with `parentContext: undefined` instead of a context in `evt.trace.traceId`.

## Solution Assessment

Yes, a narrow plugin fix is the best path. The async event bus and diagnostic trace schema already carry the needed trace context; the missing behavior is the trusted OpenTelemetry parent-context selection in `extensions/diagnostics-otel/src/service.ts`.
