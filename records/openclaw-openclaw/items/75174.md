## Summary

Keep open. Current main still has a source-level path where trusted diagnostics-otel spans can be started without an event-derived parent context, so non-HTTP or async-drained runs can still produce OTEL trace IDs that do not match the diagnostic `evt.trace.traceId` used by log records. The reporter’s linked SnappStats workaround is consistent with the code path: it reconstructs context around listener invocation, while the bundled plugin should do that correlation itself.

## What This Changes

Fix the bundled diagnostics-otel plugin so exported spans correlate with the diagnostic event trace ID instead of becoming orphan root traces for non-HTTP or async diagnostic events.

## Best Possible Solution

Update diagnostics-otel to reconstruct a safe trusted parent context from validated diagnostic trace data when no tracked active span exists, while preserving the untrusted-event guard and avoiding invalid or self-parented contexts. Add regression tests that prove run/model/tool spans share the event trace ID across async diagnostic drains, then document or changelog the user-facing observability fix.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level: emit trusted run/model/tool diagnostic events with `traces: true` and no active OTEL request context, then inspect `tracer.startSpan` parent contexts; current main leaves root and completed-only lifecycle paths undefined instead of deriving a context from `evt.trace`.

## Solution Assessment

Unclear as written. The reporter’s direction is right, but the fix should be a careful trusted-context fallback for span creation rather than a blanket one-line pass-through that could self-parent spans or accept untrusted plugin-supplied trace IDs.
