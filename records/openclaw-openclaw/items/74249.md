## Summary

Keep this issue open. Current main still lacks a `channelTimeoutSeconds` config surface and the shared agent-run timeout resolver remains global plus explicit request overrides; recent timeout work helps adjacent Telegram/Codex failure modes but does not implement channel-aware agent-run timeout policy.

## What This Changes

The issue requests per-channel agent-run timeout overrides for Telegram, Discord, WebChat, and cron while preserving existing global timeout behavior.

## Best Possible Solution

Add a backward-compatible channel-aware agent-run timeout contract only after confirming the public config shape and defaults: preserve current global `agents.defaults.timeoutSeconds` fallback, keep cron job `payload.timeoutSeconds` as the explicit top priority, and apply normalized channel context in Gateway/WebChat/channel request paths.

## Reproduction Assessment

Yes for the config gap: current main has no accepted `channelTimeoutSeconds` schema/type/docs field and no resolver call that can vary by channel. The provider soft-throttle hang itself is not live-reproduced here, so that part remains production-report evidence rather than local proof.

## Solution Assessment

Unclear. A channel-aware resolver is the right maintainable direction, but the proposed default values and per-agent field shape are public config decisions; the safer implementation would add optional overrides first and preserve existing behavior unless configured, unless maintainers explicitly approve new channel defaults.
