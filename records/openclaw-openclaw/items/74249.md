## Summary

Keep this issue open. Current main still has only a global agent-run timeout plus explicit request/cron overrides; there is no channel-aware timeout config, schema/type surface, docs entry, or runtime resolution for Telegram/Discord/WebChat whole-run timeouts. Recent Telegram transport and Codex app-server idle-timeout fixes are useful adjacent mitigations, but they do not implement the requested per-channel agent-run policy.

## What This Changes

Requests backward-compatible per-channel agent-run timeout overrides in global and per-agent agent config, with cron payload timeouts remaining highest priority and config/schema/docs/runtime support added.

## Best Possible Solution

Add a shared channel-aware agent-run timeout resolver and public config surface, preserving existing global `agents.defaults.timeoutSeconds` behavior when no override is configured, keeping cron `payload.timeoutSeconds` first, and applying the normalized source/reply channel in Gateway/WebChat/channel request paths with schema, generated metadata, docs, and focused regression tests.

## Reproduction Assessment

Yes for the config/runtime gap: current main can be checked by searching for `channelTimeoutSeconds`, inspecting the strict agent schemas, and following the Gateway/WebChat/Cron timeout calls. The live OpenAI OAuth soft-throttle scenario remains environment-dependent and was not replayed here.

## Solution Assessment

Unclear until maintainers approve the public API shape and defaults. The likely maintainable direction is a single shared resolver plus schema/docs/tests, rather than more channel-specific transport timeout tweaks.
