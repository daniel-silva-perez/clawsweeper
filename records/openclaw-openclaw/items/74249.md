## Summary

Keep this issue open. Current main still has only a global agent-run timeout plus explicit request/cron overrides; there is no `channelTimeoutSeconds` config surface, and the Gateway/WebChat/cron timeout call sites do not pass a normalized channel into timeout resolution. The related Telegram stall and active-memory reports are useful context for why fast-fail channel policy matters, but they do not implement or supersede this configuration feature.

## What This Changes

The issue requests backward-compatible per-channel agent-run timeout overrides for interactive channels and cron, with schema validation, runtime resolution, defaults, and agent config documentation.

## Best Possible Solution

Add a small, documented, backward-compatible channel-aware agent-run timeout resolver in core config/runtime: preserve existing `agents.defaults.timeoutSeconds` behavior when no channel override is configured, keep explicit request and cron `payload.timeoutSeconds` overrides highest priority, add schema/types/generated docs/help for the approved map shape, and apply it from Gateway/WebChat/channel request paths using normalized source or reply channel.

## Reproduction Assessment

Yes for the repository-side gap: current main can be inspected and has no `channelTimeoutSeconds` schema/type/docs key, while all relevant runtime paths still call the global timeout resolver. The external soft-throttle hang scenario is plausible from the report but was not live-reproduced here.

## Solution Assessment

Unclear pending maintainer API approval. The requested channel-aware timeout direction is likely the narrow maintainable fix, but the public key shape, default values, and channel naming should be confirmed before an automated implementation PR.
