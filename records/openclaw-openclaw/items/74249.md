## Summary

Keep this issue open. Current main still has only global agent-run timeout resolution plus explicit request or cron overrides; there is no channel-aware timeout config, schema/type surface, docs entry, or runtime selection for Telegram, Discord, WebChat, or cron defaults.

## What This Changes

Request per-channel agent timeout overrides in global and per-agent config, with cron job-specific timeouts remaining the highest-priority override.

## Best Possible Solution

Add a shared channel-aware agent-run timeout resolver and public config surface, preserving existing global `agents.defaults.timeoutSeconds` behavior when no override is set, keeping cron `payload.timeoutSeconds` first, and covering Gateway/WebChat/channel request paths with schema, generated metadata, docs, and focused regression tests.

## Reproduction Assessment

Yes for the config/runtime gap: current main can be verified by searching for `channelTimeoutSeconds`, inspecting the strict schemas, and following the Gateway/WebChat/Cron timeout calls. The provider soft-throttle scenario remains environment-dependent.

## Solution Assessment

Unclear until maintainers approve the public API shape and default values. The maintainable direction appears to be one shared resolver used by all run entry points, rather than more channel-specific transport timeout tweaks.
