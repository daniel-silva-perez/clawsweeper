## Summary

Keep open. Current main still registers `openclaw ltm search --limit <n>` and passes `Number.parseInt(opts.limit, 10)` directly into `db.search`; `MemoryDB.search` then forwards that value to LanceDB `.limit(limit)`. That means the radix cleanup is already present, but the finite/positive guard and warning proposed by this PR are still not implemented on main. The provided PR diff is narrow and does not introduce workflow, dependency, package, lockfile, install, release, or secret-surface changes.

## Best Possible Solution

Keep this PR open and update it against current main: preserve the existing `Number.parseInt(opts.limit, 10)`, add a finite positive limit guard before `db.search`, warn on invalid input, and add a focused memory-lancedb CLI unit test for invalid, zero, negative, and valid limits. No core/plugin API change appears necessary; a small changelog entry is worth maintainer judgment because this changes user-visible CLI behavior.
