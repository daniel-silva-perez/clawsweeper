## Summary

Keep this PR open. Current main still parses `openclaw ltm search --limit` with `Number.parseInt(opts.limit, 10)` and passes the result directly into `db.search`; `MemoryDB.search` then forwards that value into LanceDB `.limit(limit)`. The PR remains relevant and narrow, with no security-sensitive file surface in the provided diff, but it should add focused regression coverage before merge.

## What This Changes

The PR changes the memory-lancedb `ltm search --limit` CLI path to parse base-10 input, fall back to 5 for invalid or non-positive values, warn when user input is ignored, and pass the sanitized limit to search.

## Best Possible Solution

Keep this PR open and finish it against current main: validate `opts.limit` for `openclaw ltm search` before calling `db.search`, fall back to 5 for invalid, zero, or negative input, warn when user input is ignored, and add focused memory-lancedb CLI tests for invalid and valid limits. No core/plugin API change is needed.
