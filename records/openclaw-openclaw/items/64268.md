## Summary

Current main still does not implement per-LLM-call output/usage events for agentic loops. The embedded runner continues to accumulate assistant-message usage and emits a single `llm_output` after the attempt, while the existing per-call `model_call_*` hooks are sanitized telemetry without raw output or token usage. The related PR #45990 remains an open implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open. The best path is to review or replace #45990 with an additive, documented per-call observability contract: either a new `llm_call` hook or clearly flagged per-call plus aggregate `llm_output` events, including per-call usage, provider/model identifiers, run/session/agent context, and focused multi-round agentic-loop tests.

## Risks / Open Questions

- Changing `llm_output` from one aggregate event per attempt to per-call events without a compatibility marker could break existing plugins that assume aggregate semantics.
- The existing `model_call_started` and `model_call_ended` hooks address sanitized per-call telemetry and latency, but they intentionally do not expose raw output or token usage, so they do not satisfy this request.
- The open PR #45990 should be reviewed for API compatibility, provider/model threading, aggregate preservation, docs, and focused multi-round embedded-runner tests before this issue can close.
