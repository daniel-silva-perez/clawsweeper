## Summary

Keep open. Current main still matches the reported Twitch lifecycle failure: the bundled Twitch gateway task resolves after monitor setup, and the gateway supervisor treats that clean resolution as a channel exit that triggers auto-restart. The paired open Twitch report #60041 tracks overlapping symptoms from the same author, so this should stay open and be fixed by one narrow lifecycle PR rather than closed during cleanup.

## What This Changes

The issue reports that the bundled Twitch channel connects to IRC but immediately enters the gateway auto-restart loop because the provider startup task returns after setup instead of staying alive.

## Best Possible Solution

Keep the gateway restart contract intact and fix the bundled Twitch plugin locally: start the monitor, keep the account task pending until the account aborts, call the returned stop handle exactly once on shutdown, and add a focused Twitch lifecycle regression test alongside a changelog fix entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: Twitch `startAccount` resolves as soon as `monitorTwitchProvider` returns `{ stop }`, and existing gateway tests show that a resolved channel task is marked stopped and auto-restarted.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-local use of the existing passive monitor lifecycle helper or equivalent pending-until-abort wrapper, plus a Twitch regression test; changing the generic gateway supervisor would broaden the contract unnecessarily.
