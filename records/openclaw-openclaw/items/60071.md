## Summary

Keep open. Current `main` still has the Twitch lifecycle pattern described by the reporter: Twitch `startAccount()` awaits a monitor that returns a `{ stop }` handle immediately, and the generic gateway supervisor treats a settled `startAccount()` task as an account exit and schedules auto-restart.

## What This Changes

The issue requests fixing the Twitch plugin so a successfully started IRC connection stays running instead of entering the gateway auto-restart loop immediately after startup.

## Best Possible Solution

Make the Twitch gateway adapter keep `startAccount` pending until `ctx.abortSignal` fires, using the existing passive lifecycle helper for `{ stop }` monitors, and add a Twitch lifecycle regression test matching the fixed Nextcloud Talk pattern.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `twitchPlugin.gateway.startAccount` with a mocked `monitorTwitchProvider` returning `{ stop }`; it will settle before abort, and the current gateway supervisor path will treat that settlement as an exit and schedule auto-restart.

## Solution Assessment

Yes. The narrowest maintainable fix is Twitch-local lifecycle wrapping plus a regression test; the generic gateway behavior is already the contract relied on by other fixed passive/webhook channel plugins.

## Repair Work Prompt

Fix the Twitch plugin restart-loop lifecycle reported in https://github.com/openclaw/openclaw/issues/60071. Current `extensions/twitch/src/plugin.ts` awaits `monitorTwitchProvider(...)`, and `extensions/twitch/src/monitor.ts` returns `{ stop }` immediately, so the gateway supervisor treats the Twitch account as exited and schedules auto-restart. Keep the fix Twitch-local: wrap the returned monitor with the existing passive lifecycle helper (`runStoppablePassiveMonitor` or `runPassiveAccountLifecycle`) so `startAccount` stays pending until `ctx.abortSignal` fires, and call the monitor stop hook exactly once on shutdown. Preserve existing `stopAccount` cleanup through `removeClientManager`; do not change generic gateway restart semantics unless a targeted test proves it is necessary. Add a colocated Twitch lifecycle regression test using `startAccountAndTrackLifecycle` / `expectStopPendingUntilAbort`, modeled on `extensions/nextcloud-talk/src/channel.lifecycle.test.ts`, and keep existing Twitch config/status behavior unchanged. Consider whether `TwitchClientManager.getClient()` should await or otherwise surface `client.connect()` failures, but do not broaden the PR beyond lifecycle/error propagation needed for this bug. Add a user-facing changelog fix entry credited with `Thanks @nhaener`.
