## Summary

Keep open. Current main has Discord thread parent detection, thread starter context, optional parent transcript inheritance, and a manual `list-pins` action, but I found no implementation that automatically fetches parent channel pins or injects them into normal Discord thread message context.

## What This Changes

Request automatic inheritance of parent Discord channel pinned messages into Discord thread session context or metadata, optionally controlled by configuration.

## Best Possible Solution

Add Discord-plugin support that resolves a thread's parent channel, fetches that parent channel's pins with the existing REST surface, and exposes the pin text as bounded untrusted supplemental context or explicit metadata for thread messages. Avoid putting user-authored pin content into `GroupSystemPrompt`; document the config/default and add regression tests around parent-thread context, visibility filtering, and the existing manual `list-pins` action.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: in current main, normal Discord thread message processing computes thread parent/session data but never fetches or carries parent pins, and no `inheritPins`-style config exists.

## Solution Assessment

Unclear for the exact requested shape. The user problem is real, but injecting pins as system context is likely not the safest default because Discord pins are user-authored; an opt-in untrusted context or metadata implementation would be the narrower maintainable path.
