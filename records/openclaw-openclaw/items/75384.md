## Summary

Keep open. The report and PR target a real-looking empty-prompt path in the memory flush flow, but the proposed fix should not merge as-is because it makes the full internal memory-flush prompt the visible/persisted user turn, regressing the existing runtime-context transcript boundary.

## What This Changes

The PR removes the empty `transcriptPrompt` override from pre-compaction memory flush runs, updates two memory-flush test expectations, and adds a changelog entry for #75305.

## Best Possible Solution

Preserve the runtime/transcript separation for memory flushes while making the provider-facing turn non-empty, then cover both behaviors with regression tests: Anthropic-compatible runs should not receive an empty text block, and chat/session history should not expose the full memory-flush prompt as a normal user message.

## Reproduction Assessment

Yes. Static tracing is high-confidence: current main passes `transcriptPrompt: ""`, the splitter converts that into a runtime-only empty prompt, and the runner submits that empty prompt; the PR changes the same path so the full flush prompt becomes the submitted user prompt.

## Solution Assessment

No. The proposed deletion is the narrowest way to avoid an empty prompt, but it solves that by undoing the transcript-hygiene boundary; a safer fix should use a non-empty provider-compatible marker or runner-level runtime-only handling while keeping the internal flush prompt out of visible transcript text.
