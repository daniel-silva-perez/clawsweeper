## Summary

Keep this PR open for maintainer review. Current main still uses the 10s Gateway RPC wrapper fallback, while the PR is a narrow resolver/test update to make that fallback 30s without changing explicit per-call timeouts; I did not find a blocking correctness or security issue in the proposed patch.

## What This Changes

The PR raises the Gateway RPC wrapper default timeout from 10s to 30s in `src/gateway/call.ts` and updates `src/gateway/call.test.ts` coverage for event-loop readiness and default wrapper timeout behavior.

## Best Possible Solution

Land a reviewed version of the narrow resolver change with focused Gateway call tests and the normal Gateway changed-gate proof; an additional assertion for a configured/env handshake timeout above 30s would strengthen coverage but is not required to make the approach sound.

## Reproduction Assessment

Yes. The deterministic path is the silent `callGateway({ method: "health" })` wrapper-timer case in `src/gateway/call.test.ts`: current main falls through the resolver to a 10,000ms wrapper timeout, while the PR expects the default wrapper timeout at 30,000ms.

## Solution Assessment

Yes. The proposed fix changes the resolver that owns the wrapper timeout and leaves explicit per-call request timeout forwarding untouched, which is the narrowest maintainable place for this behavior.
