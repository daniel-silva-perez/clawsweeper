## Summary

Keep open. Current main does not implement the PR's scoped working-memory lane: there is no `workingMemoryPath`/`.openclaw/working-memory` surface in source or docs, cron and heartbeat only pass lightweight bootstrap context, `/context` still reports a single injected workspace file list, and the tool inventory has no scoped-working-memory filter for `memory_search`, `memory_get`, or `lcm_*`. The PR is also not merge-obsolete because the existing review discussion includes an unresolved correctness concern about bootstrap stats being confused with scoped memory files.

## Best Possible Solution

Keep the PR open for maintainer review if scoped working-memory lanes are desired. The next useful path is to rebase on current main, resolve the bootstrap-stat review comment by keeping scoped files out of bootstrap matching, add/update the cron CLI, gateway protocol, heartbeat/cron plumbing, `/context` memory taxonomy, docs, and regression tests together, then run the appropriate changed gate. If maintainers decide this should be plugin/ClawHub-only, close it by explicit product decision rather than as already implemented.
