## Summary

Keep open. Current main does not implement scoped working-memory lanes, and the PR remains useful, but the patch has a concrete correctness issue: it mixes scoped working-memory files into the bootstrap stats path, which can corrupt startup-memory/context reporting because the existing stats helper still performs basename fallback matching.

## What This Changes

The PR adds an explicit scoped working-memory file option for isolated cron/heartbeat runs, separates startup/searchable/recall/working memory in context reporting, filters memory/recall tools for scoped runs, and adds related docs and regression tests.

## Best Possible Solution

Keep the PR open, but require a small revision before merge: pass only bootstrap context files into bootstrap stats and system-prompt report bootstrap inputs, keep scoped working-memory files solely in the dedicated working-memory lane, and add a basename-collision regression test.

## Reproduction Assessment

Yes. A focused unit test can create a bootstrap file named `notes.md` and inject only `.openclaw/working-memory/cron/notes.md`; because `buildBootstrapInjectionStats` falls back by basename, the bootstrap file is incorrectly reported as injected from scoped content when the PR passes the mixed file list.

## Solution Assessment

No. The product direction is reasonable, but the current implementation is not the narrowest maintainable fix until bootstrap stats/reporting consume bootstrap-only files and scoped memory remains separate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

  - body: The PR appends scoped working memory to `contextFiles` and then passes that mixed list into `buildBootstrapInjectionStats`.
