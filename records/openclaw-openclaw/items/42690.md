## Summary

Current main still does not implement the issue’s requested status-card split between the selected/session-default model and the routed active model. The shared renderer computes both selected and active model refs and can use transcript fallback for active model lookup, but it only renders the selected `🧠 Model:` line plus `↪️ Fallback:` when fallback state is active. Non-fallback runtime routing still has no `↪️ Active:` line, and the `session_status` tool can collapse no-override runtime identity into the primary model label instead of keeping the configured default visible separately. The adjacent #55425 fix shipped persisted-runtime fallback behavior, but that is narrower than this issue.

## Best Possible Solution

Keep this issue open. Implement the split in the shared status rendering path: keep `🧠 Model:` tied to the configured or explicit session selection, add `↪️ Active: provider/model` when the resolved active runtime model differs and fallback state is not active, preserve existing `↪️ Fallback:` behavior for failover, and add focused coverage in `src/auto-reply/status.test.ts` plus `src/agents/openclaw-tools.session-status.test.ts`. If live session state lacks active identity, use the existing bounded transcript usage/model fallback without mutating the session default.
