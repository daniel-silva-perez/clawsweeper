## Summary

Keep open. Current main has solid abort primitives (`chat.abort`, `sessions.abort`, `sessions.steer`, and queue `interrupt`), plus some partial edit/delete event awareness in Slack, Signal, and the QA channel, but it still does not implement the requested lifecycle path: correlate a recalled/edited/deleted inbound platform message to the active run and cancel or rerun it. Related #52993 is useful Feishu recall/delete API prior art, but it is an open PR for Feishu message actions and does not solve broad cancel-on-recall behavior for channel runs.

## Best Possible Solution

Keep this open as a core/channel lifecycle feature. Add an additive typed plugin SDK contract for inbound message edit/delete/recall events, persist enough platform message-id to session/run correlation to identify the active run, expose a safe channel runtime abort/steer helper or route through `sessions.abort`/`chat.abort`, then implement and test bundled channels such as Telegram, Discord, Feishu, Slack, Signal, WhatsApp, and QA-channel where their platforms expose reliable events. Treat edit as cancel-and-rerun first unless maintainers explicitly choose live transcript mutation.

## Risks / Open Questions

- A future implementation must strictly correlate platform account, conversation, sender, original inbound message id, session key, and active run ownership so an unrelated delete/edit event cannot abort the wrong run.
- Edit support needs a product decision on transcript semantics; cancel-and-rerun is safer than mutating prompt history after the model has already consumed it.
