## Summary

Keep open. Current main has solid abort primitives and queue interruption, plus partial platform edit/delete awareness in Slack, Signal, Telegram update selection, and QA-channel test transport, but it still does not implement the requested lifecycle path: correlate a recalled/edited/deleted inbound platform message to the active OpenClaw run and cancel or rerun it. Related #52993 is useful Feishu prior art, but it is an open, Feishu-specific message recall/delete tool PR and does not supersede this broader cross-channel cancellation feature.

## Best Possible Solution

Keep this open as a core/channel lifecycle feature. Add an additive typed plugin SDK contract for inbound message edit/delete/recall events, persist enough platform-message-id to session/run correlation to identify the active run, expose a safe channel runtime abort/steer helper or route through sessions.abort/chat.abort, then implement and test bundled channels where their platforms expose reliable events. Treat edit as cancel-and-rerun first unless maintainers explicitly choose live transcript mutation.

## Risks / Open Questions

- A future implementation must strictly correlate platform account, conversation/thread, sender, original inbound message id, session key, and active run ownership so an unrelated edit/delete event cannot abort the wrong run.
- Edit support needs a product decision on transcript semantics; cancel-and-rerun is safer than mutating prompt history after a model has already consumed the original message.
- Platform support and event payload quality vary by channel, so the core seam should be additive and allow channel plugins to opt in only when they can prove reliable correlation.
