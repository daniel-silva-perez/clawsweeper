## Summary

Keep open. Current main has usable run abort primitives, normal queue interruption, and some channel-level edit/delete awareness, but it still lacks the requested cross-channel lifecycle path that correlates a recalled/edited platform message to the active OpenClaw run and cancels or reruns it. The related Feishu PR is useful prior art for Feishu message recall/delete tooling, but it is open/unmerged and does not cover this broader channel lifecycle feature.

## Best Possible Solution

Keep this open as a core/channel lifecycle feature. Add an additive typed plugin SDK contract for inbound message edit/delete/recall events, persist enough platform-message-id to session/run correlation to identify the active run, expose a safe channel runtime abort/steer helper or route through sessions.abort/chat.abort, then implement and test bundled channels where reliable platform events exist. Treat edit as cancel-and-rerun first unless maintainers explicitly choose live transcript mutation.

## Risks / Open Questions

- A future implementation needs strict correlation across platform account, conversation/thread, sender, original platform message id, session key, and active run ownership so an unrelated edit/delete event cannot abort the wrong run.
- Edit support needs a product decision on transcript semantics; cancel-and-rerun is safer than mutating prompt history after the model has consumed the original message.
- Platform event support varies by channel, so the core seam should be additive and opt-in per channel/plugin.
