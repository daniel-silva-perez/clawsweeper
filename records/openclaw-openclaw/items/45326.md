## Summary

Keep open. Current `main` still clears the TUI editor before any active-run/busy check and still routes normal text into `sendMessage` without rejecting `state.activeChatRunId`, so the swallowed-input path remains present. The merged adjacent TUI reconciliation work in #59800 did not add the active-generation submit guard, and the direct prior fix PR #45586 was closed unmerged.

## What This Changes

The issue requests that the TUI stop swallowing, buffering, or replaying text submitted during active model generation by interrupting, rejecting, or preserving that input instead of sending stale text on the next turn.

## Best Possible Solution

Add a TUI-local guard for normal chat submissions before `editor.setText("")`, preserve the typed draft when blocked, surface a short busy/abort hint, and keep slash commands and bang-lines usable during active runs. The fix should stay at the TUI submit boundary and avoid changing gateway `chat.send` concurrency or queue semantics unless maintainers make a separate product decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text while `state.activeChatRunId` is non-null: the current submit handler clears the editor first and then calls `sendMessage`, which does not reject active-run state.

## Solution Assessment

Yes. The narrowest maintainable repair is at the TUI submit boundary before the editor is cleared; a guard only inside `sendMessage` would still lose the draft, and a gateway-wide concurrency ban would be broader than the reported TUI UX bug.
