## Summary

Keep open. Current main still lacks the TUI active-run submit guard: normal submit clears the editor before any busy check, production wiring passes no active-run predicate, and the normal send path still submits chat without rejecting an existing active run. The closed unmerged PR #45586 remains useful prior art, while adjacent merged TUI busy-state work did not cover this submit boundary.

## What This Changes

The issue requests that TUI text submitted during active model generation be interrupted intentionally or rejected while preserving the draft, instead of being swallowed and later replayed as a stale prompt.

## Best Possible Solution

Add a TUI-local guard before clearing the editor: block normal chat submissions while `state.activeChatRunId` is non-null, preserve the typed draft, and show a short busy or Esc-to-abort hint. Keep slash commands and bang-lines available during active runs, and leave gateway chat concurrency or queue semantics to a separate product decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call the current submit handler with normal text while `state.activeChatRunId` is non-null: the handler clears the editor and routes to `sendMessage`, and the send path has no active-run rejection. The reporter’s manual flow maps directly onto that path.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before `editor.setText("")`; guarding only inside `sendMessage` would still lose the draft, while a gateway-wide queue or concurrency change would be broader than the reported TUI behavior bug.
