## Summary

Keep open. Current main and the latest release still clear the TUI editor before any busy-state check and route normal input to `sendMessage` without rejecting an active run, so the reported swallowed-input path remains valid. The closed prior PR is useful implementation evidence but was not merged.

## What This Changes

The issue requests that TUI input entered during active model generation not be silently swallowed, queued, or replayed as a later prompt.

## Best Possible Solution

Implement a TUI-local normal-message submit guard before `editor.setText("")`, preserve the typed draft when blocked, show a short status hint pointing to Esc or `/abort`, and add focused regression tests for active-run chat blocking plus allowed command/bang-line paths. Keep gateway `chat.send` concurrency semantics unchanged unless maintainers make a separate product decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text while `state.activeChatRunId` is non-null: the current submit handler clears the editor first and then calls `sendMessage`, which does not reject the active run state.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before the editor is cleared; a guard only inside `sendMessage` would still lose the draft, and a gateway-wide concurrency ban would be broader than the reported TUI UX bug.
