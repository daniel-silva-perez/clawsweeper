## Summary

Keep open. Current main still lacks the active-run submit guard: normal TUI submit clears the editor before any busy check, production wiring passes no active-run predicate, and the send path still submits normal chat while an existing run may be active. The closed unmerged PR #45586 is useful prior art, while merged adjacent PR #59800 did not touch this submit boundary.

## What This Changes

The issue requests that TUI text submitted during an active model run be rejected, preserved, or converted into an explicit interrupt instead of being swallowed and later replayed.

## Best Possible Solution

Add a TUI-local guard before clearing the editor: block normal chat submissions while `state.activeChatRunId` is non-null, preserve the typed draft, and show a short busy or Esc-to-abort hint. Keep slash commands and bang-lines available during active runs, and avoid changing gateway chat concurrency semantics unless maintainers make a separate product decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call the current submit handler with normal text while `state.activeChatRunId` is non-null: the handler clears the editor and routes to `sendMessage`, and the send path has no active-run rejection. The reporter’s manual flow maps directly onto that path.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before `editor.setText("")`; guarding only inside `sendMessage` would still lose the draft, while a gateway-wide queue/concurrency change would be broader than the reported TUI behavior bug.
