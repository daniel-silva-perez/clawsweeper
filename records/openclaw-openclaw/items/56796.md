## Summary

Keep this PR open, but it is not ready to merge. Current main already fixes the timeout lock-contention path with an isolated `gateway-fallback-*` embedded session, while this branch still uses broad message matching and would fail closed for every `gateway closed` error, including recoverable pre-request closes; GitHub also reports the branch as merge-conflicted against current main.

## What This Changes

The PR branch makes `openclaw agent` throw instead of using embedded fallback for gateway timeout, gateway request-timeout, and gateway-closed error messages, with focused tests for those fail-closed cases.

## Best Possible Solution

Rebase the work onto current main and implement a structured, phase-aware policy for ambiguous agent gateway failures: preserve fresh-session timeout fallback, preserve embedded fallback for pre-request recoverable closes, and fail closed or isolate fallback only when the gateway-side agent request may actually still be active.

## Reproduction Assessment

Yes. A focused unit path can mock `callGateway` to reject with a timeout `GatewayTransportError`, `gateway request timeout for agent`, a pre-request closed transport error, and a pending-request `gateway closed` error, then assert whether `agentCommand` is called with a fresh fallback session or not called.

## Solution Assessment

No. The branch’s message-substring classifier is not the best current solution because main now has structured transport errors and a documented fresh-session timeout fallback; the safer direction is explicit phase/kind classification.

## Review Findings

- **[P2] Distinguish pre-request gateway closes:** `src/commands/agent-via-gateway.ts:191`

Please classify the request phase instead of matching all close messages.
