## Summary

Current main still has the Discord `thread-create` partial-success gap. The Discord plugin creates the thread, then awaits a separate initial-message post for non-forum threads; if that second REST call fails, the created thread id is not returned and the message tool falls through to the generic tool-error result, leaving agents likely to retry a non-idempotent create.

## Best Possible Solution

Keep this issue open and fix it in the Discord plugin `thread-create` path. Preserve and return the created thread id when the non-forum follow-up message post fails, using a structured partial/error result that lets agents continue in the known thread instead of retrying creation. Add regression coverage where the first REST post succeeds and the second rejects, including the runtime `threadCreate` result shape if that contract changes.
