## Summary

Current main still has the reported Discord thread-create partial-success gap. The Discord message action forwards `thread-create` content into `createThreadDiscord`, which creates the thread and then separately awaits the initial message post; if that second REST call rejects, the success return with the created thread id is bypassed, so agents can retry a non-idempotent create and produce duplicate ghost threads.

## Best Possible Solution

Keep this issue open and fix it in the Discord plugin thread-create path. The conservative implementation is to preserve and return the created thread id when the non-forum follow-up message post fails, using a structured partial/error payload that lets agents continue in the known thread instead of retrying creation. Add a regression test where the first REST post succeeds and the second rejects, covering both `createThreadDiscord` and the runtime `threadCreate` result shape if that contract changes.
