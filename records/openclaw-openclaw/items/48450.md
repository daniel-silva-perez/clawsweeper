## Summary

Current main still has the reported Discord thread-create partial-success gap. The message tool forwards `thread-create` content into the Discord plugin, `createThreadDiscord` creates the thread, then separately posts the initial message and awaits it without preserving a partial success result. If that second post fails, the caller gets an error instead of the created thread id, matching the issue discussion’s side-effect/return-state divergence.

## Best Possible Solution

Keep this issue open and fix it in the Discord plugin thread-create path. The conservative implementation is to preserve the created thread id if the follow-up initial message post fails, returning a structured partial/error payload that lets agents continue in the known thread instead of retrying thread creation; add a regression test where the first REST post succeeds and the second rejects.
