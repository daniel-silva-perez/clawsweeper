## Summary

Keep this PR open for maintainer handling, but do not merge it as-is. Current main still lacks the intended in-process gateway-tool dispatch, but the PR's early local return is not WebSocket-equivalent and would change gateway authorization, token, timeout, and JSON-frame semantics on a security-sensitive control-plane path.

## What This Changes

The PR adds a gateway-local dispatch registry, registers `dispatchGatewayMethod` during gateway startup, and makes `callGatewayTool` prefer that in-process path for embedded non-`expectFinal` calls while adding focused tests.

## Best Possible Solution

Rebase and rework this as an explicit in-process gateway request adapter that preserves the existing WebSocket contract: resolved credential intent, method-specific or explicit scopes, bounded timeout behavior, deliberate final-response handling, and JSON-frame-equivalent params normalization.

## Reproduction Assessment

Yes. A focused check is to register a local dispatcher in the gateway tool test harness, then call `callGatewayTool` with `gatewayToken`, `timeoutMs`, explicit scopes, or params containing optional `undefined` fields; the PR path returns locally before the current WebSocket contract is applied.

## Solution Assessment

No. The local registry is a reasonable direction, but this implementation returns too early and should instead route through a WebSocket-equivalent in-process adapter before it is safe to merge.

## Review Findings

Full review comments:

- **[P1] Preserve caller authorization before local dispatch:** `src/agents/tools/gateway.ts:152-158`
  - body: The local branch returns before `callGatewayTool` resolves method-specific or explicit scopes, then `dispatchGatewayMethod` uses an ambient runtime client or a synthetic client. That changes authorization behavior for calls such as admin-only cron methods, approval methods, and dynamic explicit-scope callers, so local dispatch needs to carry the same effective client/scopes as the WebSocket path.
  - confidence: 0.9
- **[P1] Honor gatewayToken and timeoutMs locally:** `src/agents/tools/gateway.ts:152-158`
  - body: The guard skips local dispatch for `gatewayUrl` and `expectFinal`, but calls with `gatewayToken` or `timeoutMs` still return through the local path. That silently ignores explicit auth intent and removes the bounded request timeout that the WebSocket transport applies.
  - confidence: 0.88
- **[P2] Normalize params before in-process dispatch:** `src/agents/tools/gateway.ts:154-155`
  - body: The local path forwards the live `params` object directly, while the WebSocket path serializes the request frame with `JSON.stringify`. Optional properties set to `undefined` can therefore reach gateway validators locally even though they would be omitted over WebSocket.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds an in-process gateway control-plane path without preserving the existing auth, scope, token, timeout, and transport-normalization contract.

Concerns:

- **[high] Local dispatch bypasses gateway authorization intent:** `src/agents/tools/gateway.ts:152`
  - body: The new early return can execute a gateway method before resolving explicit tokens, least-privilege scopes, or explicit scope overrides, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:152`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
