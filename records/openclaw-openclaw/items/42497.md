## Summary

Keep this PR open, but it is not mergeable as-is. Current main still lacks an in-process agent-tool dispatch fix, and the linked self-contention bug is real, but this patch returns before the gateway tool contract preserves authorization, explicit credentials, request timeouts, and WebSocket-equivalent params semantics.

## What This Changes

The PR adds a gateway local-dispatch registry, registers `dispatchGatewayMethod` during gateway startup, and makes `callGatewayTool` prefer in-process dispatch for embedded non-`expectFinal` calls with focused tests.

## Best Possible Solution

Rebase and rework this into an explicit in-process gateway request adapter that preserves the existing WebSocket contract: resolved credential intent, method-specific or explicit scopes, bounded timeout behavior, deliberate final-response handling, and JSON-frame-equivalent params normalization.

## Reproduction Assessment

Yes. A focused test can register a local dispatcher in the gateway tool harness, then call `callGatewayTool` with `gatewayToken`, `timeoutMs`, explicit scopes, or params containing optional `undefined` fields and verify the PR path returns locally before the current WebSocket contract is applied.

## Solution Assessment

No. The registry direction is plausible, but this early-return implementation is not the safest fix because it bypasses existing auth, timeout, and serialization behavior instead of adapting the in-process path to match it.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Pass effective scopes into local dispatch:** `src/agents/tools/gateway.ts:152-158`
  - body: This local branch runs before `resolveLeastPrivilegeOperatorScopesForMethod` and before `extra.scopes` are applied. In fallback mode `dispatchGatewayMethod` uses a synthetic write-scoped client, while an ambient scope may differ from the caller's explicit intent, so admin methods like `cron.add`/`cron.run` or dynamic scope overrides can authorize differently than the WebSocket path.
  - confidence: 0.9
- **[P1] Honor explicit auth and timeout locally:** `src/agents/tools/gateway.ts:152-158`
  - body: The guard only excludes explicit `gatewayUrl` and `expectFinal`; calls with `gatewayToken` or `timeoutMs` still take the in-process path. That silently ignores explicit credential intent and removes the bounded request timeout that existing `node.invoke` callers rely on.
  - confidence: 0.88
- **[P2] Normalize params before local dispatch:** `src/agents/tools/gateway.ts:154-155`
  - body: The local path forwards the live `params` object directly, while the WebSocket path serializes the request frame. Optional properties set to `undefined` can therefore reach gateway validators locally even though they would be omitted over WebSocket.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces an in-process gateway control-plane path without preserving the existing authorization, credential, and timeout contract.

Concerns:

- **[high] Local dispatch bypasses caller authorization intent:** `src/agents/tools/gateway.ts:152`
  - body: The early local return can execute a gateway method before resolving explicit tokens, least-privilege scopes, or explicit scope overrides, so embedded calls may run under ambient or synthetic credentials rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Local dispatch skips bounded request timeout:** `src/agents/tools/gateway.ts:152`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
