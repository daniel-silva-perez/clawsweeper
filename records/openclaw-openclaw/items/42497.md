## Summary

Keep open. The PR targets a real gateway self-contention problem, but the proposed local-dispatch path is not equivalent to the current gateway tool auth, scope, token, timeout, and request-normalization contract.

## What This Changes

The PR adds a local gateway dispatch registry, registers in-process dispatch during gateway startup, and makes `callGatewayTool` prefer that dispatcher for embedded calls except explicit `gatewayUrl` and `expectFinal` cases, with tests.

## Best Possible Solution

Keep the in-process dispatch direction, but make it an explicit gateway request adapter with WebSocket-equivalent behavior: resolved token intent, least-privilege or explicit scopes, default-deny unknown methods, timeout handling, `expectFinal` support or deliberate fallback, and JSON-frame-equivalent params normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve scopes before local dispatch:** `src/agents/tools/gateway.ts:151-158`
  - body: The new local branch returns before `callGatewayTool` resolves least-privilege scopes or honors explicit `extra.scopes`. `dispatchGatewayMethod` then uses the ambient request client or a synthetic write-scoped client, so embedded admin cron calls can fail and scoped callers can run under a different authorization contract than the existing WebSocket path.
  - confidence: 0.9
- **[P1] Honor token and timeout options on local calls:** `src/agents/tools/gateway.ts:151-158`
  - body: The guard only skips local dispatch for explicit URLs and `expectFinal`, so calls with `timeoutMs` or `gatewayToken` still bypass `resolveGatewayOptions` and the gateway client. That silently ignores explicit auth and timeout behavior used by existing `node.invoke` and approval paths, and can leave a stuck in-process handler blocking the active turn.
  - confidence: 0.88
- **[P2] Normalize params like the WebSocket path:** `src/agents/tools/gateway.ts:155-157`
  - body: The local path casts and forwards the live `params` object directly, while the WebSocket path validates a request frame and sends it through `JSON.stringify`. Optional properties set to `undefined` can therefore reach gateway validators locally even though the same call over WebSocket would omit them.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds a security-sensitive in-process gateway dispatch path without preserving the existing auth, scope, token, and timeout contract.

Concerns:

- **[high] Local dispatch bypasses auth and scope resolution:** `src/agents/tools/gateway.ts:151`
  - body: `callGatewayTool` can return from local dispatch before resolving explicit gateway tokens, least-privilege method scopes, or explicit scope overrides, so embedded calls may execute under ambient or synthetic gateway identity rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Timeout bypass can hang privileged operations:** `src/agents/tools/gateway.ts:151`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the bounded timeout behavior used by the WebSocket request path.
  - confidence: 0.84
