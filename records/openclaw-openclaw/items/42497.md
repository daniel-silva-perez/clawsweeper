## Summary

Keep this PR open. The reported gateway WebSocket self-contention bug is still plausible on current main because embedded gateway tools still route through the WebSocket-backed `callGateway` path, but this patch is not ready to merge: its local-dispatch branch changes authorization, timeout, token, and request-shape semantics compared with the current gateway client contract.

## What This Changes

The PR adds a local gateway dispatch registry, registers `dispatchGatewayMethod` during gateway startup, and makes `callGatewayTool` use that in-process dispatcher for embedded calls except explicit gateway URL and `expectFinal` calls.

## Best Possible Solution

Keep the in-process dispatch direction, but rework it around a request contract that is equivalent to `callGatewayTool`/`callGateway`: resolved token intent, explicit and least-privilege scopes, timeout handling, default-deny unknown methods, `expectFinal` support or explicit WebSocket fallback, and WebSocket-equivalent JSON request normalization. Add regression coverage for embedded cron read/admin methods, node invocation, explicit URL/token/timeout/scope overrides, unknown methods, `expectFinal`, and optional `undefined` params.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve scopes before taking the local dispatch path:** `src/agents/tools/gateway.ts:151-158`
  - body: The local branch returns before the current helper resolves least-privilege scopes or honors explicit scope overrides. In fallback gateway contexts, `dispatchGatewayMethod` uses a synthetic write-scoped client, so admin cron calls like `cron.add`/`cron.run` can fail with missing admin scope; in scoped contexts it can also inherit a different ambient scope than the helper requested.
  - confidence: 0.9
- **[P1] Keep timeout and token semantics on local calls:** `src/agents/tools/gateway.ts:151-158`
  - body: The guard skips local dispatch only for explicit `gatewayUrl` and `expectFinal`, so calls with `timeoutMs` or `gatewayToken` still bypass `resolveGatewayOptions` and the gateway client. Existing `node.invoke` and approval paths rely on timeouts, and token overrides should not be silently ignored inside the gateway process.
  - confidence: 0.88
- **[P2] Normalize params before in-process dispatch:** `src/agents/tools/gateway.ts:152-155`
  - body: The local path forwards the live `params` object directly to `dispatchGatewayMethod`, while the WebSocket path validates a request frame and sends it through `JSON.stringify`. That changes behavior for `undefined` and other non-JSON values, so embedded calls can reach gateway validators in a shape the WebSocket path would not send.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive in-process gateway dispatch path that can bypass the current auth, token, scope, and timeout contract for embedded calls.

Concerns:

- **[high] Local dispatch bypasses auth and scope resolution:** `src/agents/tools/gateway.ts:151`
  - body: `callGatewayTool` can return from local dispatch before resolving `gatewayToken`, least-privilege scopes, or explicit scope intent, so embedded calls may execute under ambient or synthetic gateway identity rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Timeout bypass can hang privileged gateway operations:** `src/agents/tools/gateway.ts:151`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can keep an active agent turn blocked instead of failing under the timeout behavior used by the WebSocket request path.
  - confidence: 0.84
