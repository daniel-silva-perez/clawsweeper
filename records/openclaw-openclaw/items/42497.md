## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the intended embedded in-process gateway-tool dispatch, so the PR is not obsolete; however, the proposed local path is not equivalent to the current WebSocket-backed helper and would regress auth, scope, timeout, and serialization behavior on security-sensitive gateway calls.

## What This Changes

The PR adds a `src/gateway/local-dispatch.ts` registry, registers `dispatchGatewayMethod` from gateway startup, and makes `callGatewayTool` use that in-process dispatcher for embedded calls unless an explicit `gatewayUrl` or `expectFinal` call requires the WebSocket path.

## Best Possible Solution

Keep the useful design direction but rework the implementation around a local dispatcher contract that accepts the same effective request shape as `callGatewayTool` or `callGateway`: resolved URL/token intent, timeout, `expectFinal`, explicit scopes, least-privilege default scopes, default-deny unknown methods, and WebSocket-equivalent JSON normalization. The regression suite should cover embedded cron status/list/add/update/remove/run/runs plus gateway token, timeout, explicit-scope, unknown-method, `expectFinal`, and optional/undefined parameter cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve scopes before taking the local dispatch path:** `src/agents/tools/gateway.ts:156-162`
  - body: The new local branch returns before the current helper resolves least-privilege scopes or honors `extra.scopes`. On current main, fallback `dispatchGatewayMethod` uses a synthetic `operator.write` client when there is no request-scope client, so the admin cron calls this PR is meant to fix (`cron.add`, `cron.update`, `cron.remove`, `cron.run`) can fail with `missing scope: operator.admin`; in scoped contexts it can also inherit a broader caller scope than the existing helper would request.
  - confidence: 0.9
- **[P1] Keep timeout and token semantics on local calls:** `src/agents/tools/gateway.ts:153-162`
  - body: This path only falls back to WebSocket for explicit `gatewayUrl` and `expectFinal`, so calls that pass `timeoutMs` or `gatewayToken` still take local dispatch and never run through `resolveGatewayOptions`. Existing callers such as cron and node invocation use `timeoutMs` to bound gateway work, and `gatewayToken` overrides should not be silently ignored inside the gateway process.
  - confidence: 0.88
- **[P2] Normalize params like the WebSocket request path:** `src/agents/tools/gateway.ts:157-160`
  - body: The local branch forwards the live `params` object directly, while the current WebSocket path validates a request frame and sends it through `JSON.stringify`. That means object properties with `undefined` and other non-JSON values can reach gateway validators in-process even though the same call over WebSocket would have omitted or serialized them differently.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive gateway dispatch path that bypasses current token and scope resolution for some embedded calls.

Concerns:

- **[high] Local dispatch bypasses auth and scope resolution:** `src/agents/tools/gateway.ts:156`
  - body: `callGatewayTool` can return from local dispatch before resolving `gatewayToken`, least-privilege scopes, or explicit `extra.scopes`, so embedded calls may execute under ambient/synthetic gateway identity instead of the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Timeout bypass can hang privileged gateway operations:** `src/agents/tools/gateway.ts:156`
  - body: Because `timeoutMs` is ignored on the local path, a stuck in-process handler can keep an active agent turn blocked indefinitely instead of failing under the same transport timeout used by the WebSocket path.
  - confidence: 0.84
