## Summary

Keep open. The PR addresses a real gateway WebSocket self-contention bug, and the earlier discussion usefully narrowed two initial problems, but the current branch still changes gateway authorization and transport semantics in ways that are not safe to merge.

## What This Changes

The branch adds a local gateway dispatch registry, registers it during gateway startup, makes agent gateway-tool calls prefer in-process dispatch except for explicit gateway URLs and `expectFinal`, and adds focused tests for the new path.

## Best Possible Solution

Keep the in-process-dispatch direction, but implement it as an explicit gateway request adapter that preserves WebSocket-equivalent behavior: resolved token intent, method-specific scopes or explicit scope overrides, default-deny unknown methods, timeout handling, deliberate `expectFinal` support or fallback, and JSON-frame-equivalent params normalization.

## Reproduction Assessment

Yes. A focused reproduction is to register a local dispatcher in the gateway tool test harness and call `callGatewayTool` with `gatewayToken`, `timeoutMs`, custom scopes, or params containing optional `undefined` fields; the PR path returns before the current WebSocket contract is applied.

## Solution Assessment

No. The proposed registry is a useful starting point, but returning before the existing gateway request contract is preserved is not the narrowest maintainable fix for a security-sensitive control-plane path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve gateway scopes before local dispatch:** `src/agents/tools/gateway.ts:154-158`
  - body: The local branch returns before `callGatewayTool` computes explicit or least-privilege scopes. `dispatchGatewayMethod` then runs under the ambient request client or a synthetic write-scoped client, so embedded cron/admin calls, dynamic scope overrides, and default-deny behavior no longer match the WebSocket path.
  - confidence: 0.9
- **[P1] Honor token and timeout options locally:** `src/agents/tools/gateway.ts:154-158`
  - body: The guard skips local dispatch for explicit URLs and `expectFinal`, but calls with `gatewayToken` or `timeoutMs` still bypass `resolveGatewayOptions` and the WebSocket request timer. That silently ignores explicit auth intent and can leave embedded tool operations unbounded.
  - confidence: 0.88
- **[P2] Normalize params before dispatching locally:** `src/agents/tools/gateway.ts:155-157`
  - body: The local path forwards the live `params` object directly, while the WebSocket path sends a JSON frame. Optional properties set to `undefined` can therefore reach validators locally even though they would be omitted over WebSocket.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff adds an in-process gateway dispatch path without preserving the existing auth, scope, token, and timeout contract.

Concerns:

- **[high] Local dispatch bypasses auth and scope resolution:** `src/agents/tools/gateway.ts:154`
  - body: `callGatewayTool` can return from local dispatch before resolving explicit gateway tokens, least-privilege method scopes, or explicit scope overrides, so embedded calls may execute under ambient or synthetic gateway identity rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Timeout bypass can hang privileged operations:** `src/agents/tools/gateway.ts:154`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the bounded timeout behavior used by the WebSocket request path.
  - confidence: 0.84
