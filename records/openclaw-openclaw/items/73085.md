## Summary

Keep open: current main still has the Slack DM main-route contamination path, and this PR is the active implementation candidate. The patch addresses the global `session.dmScope` case and CI is green, but it should be adjusted before merge because the new guard does not use the effective route policy and can miss route-binding `dmScope` overrides.

## What This Changes

The PR gates Slack DM main-session last-route updates in `prepareSlackMessage` and `dispatchPreparedSlackMessage` on `cfg.session.dmScope` being `main`, and adds focused dmScope regression tests for both seams.

## Best Possible Solution

Merge a narrow Slack fix that bases main-route updates on the effective resolved route policy, preserving default `main` and owner-pin behavior while skipping main-route writes for any non-main DM session, including binding-specific overrides. Add regression coverage for both global and binding-level dmScope cases in prepare and dispatch.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence seam reproduction: configure a Slack DM with non-main effective DM scope, then the current main prepare/dispatch paths still target `route.mainSessionKey` for last-route updates; the PR tests cover the global config version of that scenario.

## Solution Assessment

Not quite. The PR direction is narrow and appropriate, but the implementation should use `route.lastRoutePolicy` or the existing inbound last-route policy helper rather than only `cfg.session.dmScope`, so route-binding overrides cannot keep contaminating `agent:main:main`.
