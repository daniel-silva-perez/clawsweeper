## Summary

Keep open. The PR targets a real Slack DM route-contamination bug on current main, and its prepare-side policy change is in the right direction, but the latest dispatch change still writes non-main threaded DMs to the base peer session instead of the prepared turn session. The focused CI checks are green, but the new dispatch tests do not cover the threaded DM case that exposes the remaining bug.

## What This Changes

The PR changes Slack DM last-route persistence in `prepareSlackMessage` and `dispatchPreparedSlackMessage` to avoid main-session writes for non-main `dmScope` routes, and adds focused dmScope regression tests.

## Best Possible Solution

Revise this PR so dispatch mirrors the prepared turn session for non-main DM last-route updates, preserves main-DM owner pinning only for `lastRoutePolicy: "main"`, adds a threaded non-main DM regression, and includes the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence seam reproduction is a prepared Slack DM with non-main `lastRoutePolicy: "session"`, `route.sessionKey` set to the base peer key, and `ctxPayload.SessionKey` set to the same key with a `:thread:<threadTs>` suffix; the PR dispatch code writes `updateLastRoute.sessionKey` to the base key.

## Solution Assessment

No. The prepare-side policy is the right shape, but dispatch should target `prepared.ctxPayload.SessionKey ?? route.sessionKey` for non-main routes and only attach ctx when updating that prepared turn session.
