## Summary

Keep open: the latest PR revision addresses the earlier global-config-only `dmScope` problem by using the resolved route policy and adding binding-override coverage, but the dispatch path still targets the base route session for non-main threaded Slack DMs instead of the prepared turn session. That can refresh the peer base session with a thread route, so the branch needs one more narrow fix before merge.

## What This Changes

The branch changes Slack DM last-route persistence in `prepareSlackMessage` and `dispatchPreparedSlackMessage` to avoid main-session route writes for non-main `dmScope` routes, and adds focused dmScope regression tests.

## Best Possible Solution

Keep the route-policy based direction, but make dispatch choose the prepared turn session for non-main DM routes, not the base route session. Add a regression where a non-main Slack DM has a thread-suffixed `ctxPayload.SessionKey` and verify both prepare and dispatch update that session while preserving main-session owner-pin behavior only for `lastRoutePolicy: "main"`.

## Reproduction Assessment

Yes. A high-confidence seam reproduction is a Slack direct message with non-main effective `dmScope` plus threaded DM routing (`replyToMode="all"` or an incoming DM thread), which yields a prepared `SessionKey` with `:thread:` while the PR dispatch code still targets `route.sessionKey`.

## Solution Assessment

Not quite. The PR’s policy-based prepare change is the right shape, but dispatch should mirror the prepared turn/session key for `lastRoutePolicy: "session"`; using `route.sessionKey` is narrower than main contamination but still writes the wrong session for threaded DMs.
