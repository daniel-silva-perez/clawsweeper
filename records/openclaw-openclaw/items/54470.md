## Summary

Current main still leaves the reported native Windows Gmail setup failure actionable. The Gmail setup command remains documented and routed to `runGmailSetup`, dependency checks can recognize PATHEXT shims, but the Gmail helper execution paths still invoke bare `gcloud`, `gog`, and `tailscale`. The shared Windows command shim only special-cases npm/npx/corepack/pnpm/yarn and does not resolve arbitrary PATH/PATHEXT `.cmd` tools such as `gcloud.cmd`, while some Gmail serve paths bypass that wrapper and call `spawn("gog", ...)` directly. No canonical related item in the provided context is a merged fix or superseding open implementation.

## Best Possible Solution

Keep this issue open. Add a Windows-aware executable resolution path for Gmail helper tools before spawning `gcloud`, `gog`, and `tailscale`, ideally reusing the existing executable resolver or extending the shared exec shim narrowly without broad `shell: true`. Cover setup, run, and gateway watcher flows with win32 regression tests for PATH/PATHEXT `.cmd` shims.

## Risks / Open Questions

- Native Windows was not live-tested from this Linux checkout, but the reporter supplied concrete PowerShell evidence and current main still shows the same command-resolution gap.
- A fix should preserve the existing no-shell argv hardening; using broad `shell: true` would introduce unnecessary command-injection risk.
- The direct `spawn("gog", ...)` serve paths need coverage in addition to `runCommandWithTimeout`, otherwise setup/run/watcher behavior can diverge.
