## Summary

Current main still leaves this Windows Gmail setup crash actionable. The documented command remains supported, the CLI still routes it into `runGmailSetup`, and the Gmail helper paths still spawn bare `gcloud`, `gog`, and `tailscale` rather than resolving Windows PATH/PATHEXT shims such as `gcloud.cmd`. The existing Windows command shim is deliberately limited to package-manager commands, so it does not cover this reported helper-tool path.

## What This Changes

The issue asks for native Windows `openclaw webhooks gmail setup --account ...` to locate installed Gmail helper tools and complete setup instead of exiting with `spawn gcloud ENOENT`.

## Best Possible Solution

Add a Windows-aware helper-tool resolution path for Gmail commands that resolves PATH/PATHEXT shims to explicit invocations before spawning, keeps argv-based execution without broad shell mode, and covers setup, run, and gateway watcher behavior with win32 regression tests.
