## Summary

Keep open. This is a protected maintainer-labeled PR, and the latest head still has a security-sensitive false-positive in the denied-destination validation semantics: a plain HTTP 403 is accepted as proof that the proxy blocked the target even though the fetch result cannot distinguish proxy-generated denial from an origin response.

## What This Changes

The PR adds `openclaw proxy validate`, a proxy validation engine with CLI/runtime tests, CLI and Network Proxy docs, and an Unreleased changelog entry.

## Best Possible Solution

Keep the command in the proxy-owned CLI/security area, but make denied checks pass only on transport failure or an unambiguous proxy-generated deny signal. If HTTP status denial remains supported, the result type needs enough source information and tests proving ordinary origin 403 responses still fail validation.

## Reproduction Assessment

Yes. The injected `fetchCheck` seam can return `{ ok: false, status: 403 }` for a denied URL, and `runProxyValidation` reports success even though the same result shape can represent an origin 403 after the proxy allowed the connection.

## Solution Assessment

No. The command and docs direction are appropriate, but accepting a bare 403 is not the safest maintainable solution for a security preflight because it cannot prove the proxy blocked the destination.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require a verifiable proxy-deny signal:** `src/infra/net/proxy/proxy-validation.ts:248-254`
  - body: `runDeniedCheck` treats any HTTP 403 as a successful denied check, but the fetch result only carries `ok` and `status`. If a denied target is reachable and returns 403 itself, validation passes even though the proxy allowed the request.
  - confidence: 0.86
