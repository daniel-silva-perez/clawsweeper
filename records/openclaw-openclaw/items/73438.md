## Summary

Keep open. This draft PR is maintainer-authored and carries the protected `maintainer` label, and the latest head still has a blocking security-sensitive correctness issue in denied-destination validation: a bare HTTP 403 is treated as proof of proxy denial even though the validator cannot distinguish a proxy response from a reachable origin response.

## What This Changes

This draft PR adds `openclaw proxy validate`, a proxy validation engine, CLI/runtime tests, Network Proxy and CLI proxy documentation updates, and an Unreleased changelog entry.

## Best Possible Solution

Keep the validation command in the proxy owner area, but make denied checks pass only on a transport failure or an unambiguous proxy-deny signal. If HTTP status-based denial is required, the accepted statuses should be explicit, documented with the false-positive tradeoff, and tested so ordinary origin responses still fail.

## Reproduction Assessment

Yes. The proposed test seam reproduces it by returning `{ ok: false, status: 403 }` for a denied URL and observing `result.ok === true`; because the validator receives only `ok` and `status`, that same shape can represent an origin 403 after the proxy allowed the request.

## Solution Assessment

No. The command boundary and docs direction are appropriate, but accepting a bare 403 is not the safest maintainable solution for a security preflight; the validator should require a transport failure, a proxy-specific denial marker, or an explicit audited policy for status-based denial.
