## Summary

Keep open. Current main does not have the requested proxy validation command, but this PR should not automerge because the new denied-destination check can report success from a bare HTTP 403 without proving the proxy blocked the request. The PR is also maintainer-authored and labeled `maintainer`, so cleanup closure is not appropriate.

## What This Changes

The PR adds `openclaw proxy validate`, a proxy-validation runtime, CLI/runtime tests, proxy documentation updates, and an Unreleased changelog entry for operator-managed outbound proxy preflight checks.

## Best Possible Solution

Keep the CLI/security direction, but revise denied checks so success requires a transport failure or a verifiable proxy-generated denial signal. If HTTP denial remains supported, the result contract should include enough source metadata and tests should prove ordinary origin 403 responses fail validation; docs should describe the same fail-closed behavior.

## Reproduction Assessment

Yes. The injected `fetchCheck` seam can return `{ ok: false, status: 403 }` for a denied URL, and `runProxyValidation` reports overall success even though that result shape can also represent a reachable origin returning 403.

## Solution Assessment

No. Adding the validation command is the right product direction, but accepting a bare HTTP 403 as proof of proxy denial is not the narrowest safe solution for a security preflight; the validator should fail closed unless it can verify that the proxy blocked the destination.
