## Summary

Keep open. Current main does not already implement `openclaw proxy validate`, and this is a draft MEMBER PR with the protected `maintainer` label, so it needs explicit maintainer handling. The proposed patch is useful but not merge-ready because the validation engine can pass when proxy routing is disabled, and the security docs contradict the new 403-as-deny behavior.

## What This Changes

This draft PR adds an `openclaw proxy validate` subcommand, a proxy validation engine, CLI/runtime tests, Network Proxy docs, and a changelog entry.

## Best Possible Solution

Keep the draft open and require maintainer/security review. Before merge, make the validation result match the actual `proxy.enabled` runtime contract and align docs, tests, and CLI output around the chosen denied-response semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fail disabled proxy configs before probing:** `src/infra/net/proxy/proxy-validation.ts:257-258`
  - body: The command is meant to validate the effective proxy configuration, but `runProxyValidation` only returns early for URL errors or a missing URL. With `OPENCLAW_PROXY_URL` set and `proxy.enabled` absent/false, or with a disabled config that still contains `proxyUrl`, this can return `ok: true` even though current `startProxy` skips routing unless `proxy.enabled === true`. That makes the preflight pass for a setup OpenClaw will not actually use.
  - confidence: 0.87
- **[P3] Align denied-response docs with validation behavior:** `docs/security/network-proxy.md:128`
  - body: The security guide says any completed HTTP response for a denied destination is reachable and should fail, but the implementation and tests accept HTTP 403 as a successful explicit proxy deny. Operators will get the opposite expectation from the docs and the command output, so the docs or the validation semantics need to be updated together.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive outbound proxy validation and needs semantics cleanup before merge.

Concerns:

- **[medium] Disabled proxy configs can validate as passing:** `src/infra/net/proxy/proxy-validation.ts:257`
  - body: The validator can report success when only a proxy URL is present but `proxy.enabled` is false or absent, while the runtime proxy lifecycle will not route traffic in that state. That weakens the operator-facing preflight guarantee.
  - confidence: 0.86
- **[low] Denied-response guidance contradicts 403 handling:** `docs/security/network-proxy.md:128`
  - body: The Network Proxy docs tell operators denied checks fail on any completed HTTP response, but the implementation treats HTTP 403 as a successful proxy deny. Security-sensitive operator guidance should match the command behavior.
  - confidence: 0.9
