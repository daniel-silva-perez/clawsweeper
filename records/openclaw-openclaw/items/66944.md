## Summary

Keep open. Current main still lacks a plugin-contributed native Control UI page/tab system: navigation and page rendering are static, bootstrap config does not carry plugin page/import-map metadata, manifests do not declare `ui.pages`, and the existing descriptor API is narrower than this request. Related Control UI contribution-slot work is active, but it is not a shipped implementation of plugin-owned native pages.

## What This Changes

The issue requests a plugin UI extension system where plugins can declare Control UI pages that load as native tabs inside the main Lit-based shell.

## Best Possible Solution

Keep this as active design work for a versioned Control UI plugin contribution/page API. Maintainers should decide whether the first supported shape is host-owned panels/slots or full Custom Element pages, then document manifest or SDK declarations, asset serving, navigation grouping, auth and gateway access rules, unload/disable semantics, security boundaries, and contract/UI coverage before external plugin authors depend on it.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would place plugin-owned browser behavior inside the authenticated operator Control UI.

Concerns:

- **[high] Define the Control UI plugin code trust boundary:**
  - body: A native plugin page/tab system could share DOM, theme, gateway client state, and operator auth context with third-party plugin JavaScript. The design needs explicit CSP, allowed asset origins, dependency/runtime sharing, operator scopes, data access, disable/unload behavior, and trusted-vs-untrusted plugin rules before implementation.
  - confidence: 0.92
