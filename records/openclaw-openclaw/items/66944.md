## Summary

Keep open. Current main still does not provide a plugin-contributed native Control UI page/tab system: navigation is static, bootstrap config carries no plugin page or import-map metadata, manifests do not declare `ui.pages`, and the existing Control UI descriptor seam is a narrower metadata surface. The Openclaw-Quest discussion is concrete evidence that the current standalone HTTP-route path is not enough, but the remaining work needs product/API and security design before implementation.

## What This Changes

Allow plugins to declare native Control UI pages or tabs, loaded at runtime inside the main Control UI shell from plugin manifest or SDK metadata.

## Best Possible Solution

Keep this as active design work for a versioned Control UI plugin-page or contribution API. Maintainers should decide the supported shape first, then document manifest or SDK declarations, asset serving, navigation grouping, auth/gateway access, security boundaries, disable/unload semantics, and add contract plus UI coverage before external plugin authors depend on it.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would place plugin-owned browser behavior inside the authenticated operator Control UI.

Concerns:

- **[high] Define the Control UI plugin code trust boundary:**
  - body: A native plugin page/tab system could share DOM, theme, gateway client state, and operator auth context with third-party plugin JavaScript. The design needs explicit CSP, allowed asset origins, dependency/runtime sharing, operator scopes, data access, disable/unload behavior, and trusted-vs-untrusted plugin rules before implementation.
