## Summary

Keep this open. Current main has an adjacent `registerControlUiDescriptor` host-hook surface, but it is limited to session/tool/run/settings descriptors and is not a native Control UI page/tab contribution system. The requested manifest page declarations, plugin page/import-map bootstrap metadata, dynamic UI routing, and browser-side Custom Element loader are still missing; the thread also has live Openclaw-Quest context and unresolved API/security questions.

## What This Changes

Allow plugins to declare and serve native Control UI pages or tabs that render inside the main UI shell with shared theme and gateway context.

## Best Possible Solution

Keep this as the canonical design track for a versioned Control UI plugin-page API. A good end state would define manifest or SDK page declarations, safe asset serving, bootstrap metadata, navigation grouping, a browser Custom Element loader, gateway/client access rules, docs, and contract/UI tests, while documenting the security model before broad plugin authors depend on it.
