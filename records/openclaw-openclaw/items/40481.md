## Summary

Keep this PR open: current main still shows generic exec-approval fallback guidance, so the UX change is not implemented there, but the PR head should not merge as-is because it hard-codes `allow-always` into fallback copy even when the request-specific allowed-decision set excludes it.

## What This Changes

The PR changes exec-approval fallback chat notifications and tests to render ready-to-copy `/approve <id> <decision>` lines, with markdown wrapping for capable channels and plain text for non-markdown channels.

## Best Possible Solution

Rebase and update the fallback text generation to build visible command lines from the same request-specific allowed-decision descriptors used by shared approval actions, while preserving markdown-aware wrapping, the `ask=always` unavailable explanation, and focused coverage for default, non-markdown, and `ask=always` cases.

## Reproduction Assessment

Yes. Static inspection is enough to reproduce the defect: an `ask=always` forwarded request must omit `allow-always`, while the PR diff adds that command unconditionally and the resolver rejects it.

## Solution Assessment

No. Pre-filled commands are the right UX direction, but hard-coding every decision is not the narrow maintainable fix; the command list should be derived from `resolveExecApprovalRequestAllowedDecisions` or `buildExecApprovalActionDescriptors`.
