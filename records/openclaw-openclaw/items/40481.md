## Summary

Keep open: current main still has the generic fallback guidance, so the intended UX is not implemented, but this PR introduces a blocking correctness regression by rendering `allow-always` even when current approval policy filtering makes that decision unavailable.

## What This Changes

The PR changes exec-approval fallback notifications and tests to render ready-to-copy `/approve <id> <decision>` lines, with inline-code wrapping for markdown-capable channels and plain text for non-markdown channels.

## Best Possible Solution

Rebase the branch onto current main and generate fallback command lines from the same allowed-decision descriptors used for approval actions, wrapping only for markdown-capable channels while preserving the `ask=always` unavailable explanation and focused coverage for default, non-markdown, and `ask=always` fallback text.

## Reproduction Assessment

Yes. Current-main source and tests provide a high-confidence static reproduction path: an `ask=always` forwarded request must omit `allow-always`, while the PR diff adds that command unconditionally and the resolver rejects it.

## Solution Assessment

No. Pre-filled commands are the right UX direction, but hard-coding every decision is not the narrow maintainable fix; the branch should derive the visible commands from the request-specific allowed decision set.
