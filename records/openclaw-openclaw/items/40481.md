## Summary

Keep open: the requested UX is still useful, but the PR needs rework because it renders an unavailable `allow-always` command for `ask=always` approval requests that current main intentionally filters and rejects.

## What This Changes

The PR changes exec-approval fallback notifications and tests to render ready-to-copy `/approve <id> <decision>` lines, with inline-code wrapping for markdown-capable channels and plain text for non-markdown channels.

## Best Possible Solution

Rebase and update the fallback text generation to derive visible command lines from the same allowed-decision descriptors used for approval actions, preserving markdown-aware wrapping, the `ask=always` unavailable explanation, and focused tests for default, non-markdown, and `ask=always` cases.

## Reproduction Assessment

Yes. Static inspection of current main is enough: an `ask=always` forwarded request must omit `allow-always`, while the PR diff adds that command unconditionally and the resolver rejects it.

## Solution Assessment

No. Pre-filled commands are the right UX direction, but hard-coding all decisions is not the narrow maintainable fix; the command list should be generated from the request-specific allowed decisions.
