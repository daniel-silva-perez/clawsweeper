## Summary

Current main still has the generic exec-approval forwarded fallback text, so this PR is not obsolete. The existing ClawSweeper comment remains accurate on the newer main SHA: the useful direction is still to render concrete `/approve <actual-id> <decision>` commands, but the submitted patch needs rework because it hard-codes `allow-always` instead of preserving the existing allowed-decision filtering.

## Best Possible Solution

Keep this PR open and rework it against current main. The best fix is to update `src/infra/exec-approval-forwarder.ts` so the generic forwarded fallback renders concrete `/approve <actual-id> <decision>` lines from the existing allowed-decision path, applies markdown only for markdown-capable channels, handles caller-supplied ids safely, and adds focused coverage in `src/infra/exec-approval-forwarder.test.ts`.

## Risks / Open Questions

- As submitted, the patch advertises `allow-always` unconditionally, which conflicts with the current `ask=always` policy path that allows only `allow-once` and `deny`.
- Approval ids can be caller-supplied non-empty strings, so inline markdown wrapping should be robust against formatting edge cases instead of assuming UUID-only ids.
- The PR should add focused tests for markdown-capable fallback text, non-markdown fallback text, and `ask=always` filtering before merge.
