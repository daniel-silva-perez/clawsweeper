## Summary

Keep open: current main still renders the generic forwarded fallback hint, so the requested UX is not implemented there. The PR is useful but not mergeable as-is because it can advertise an `allow-always` command for requests whose effective policy filters that decision out, and the resolver rejects that decision.

## What This Changes

The PR changes forwarded exec-approval fallback notifications and tests to show ready-to-copy `/approve <id> <decision>` command lines, using inline code only for markdown-capable channels.

## Best Possible Solution

Generate forwarded fallback command lines from the same allowed-decision descriptors used by approval actions, apply markdown wrapping only for markdown-capable channels, and keep coverage for markdown, plain-text, and `ask=always` fallback text.

## Reproduction Assessment

Yes. The current-main text path and tests provide a high-confidence check: `ask=always` omits `allow-always`, while the PR diff adds an unconditional `allow-always` fallback command.

## Solution Assessment

No. The requested UX is the right direction, but the branch should generate commands from the allowed decision set instead of hard-coding all three decisions.
