## Summary

Keep this PR open. The UX request is still valid because current main still shows a placeholder `/approve <id> ...` fallback, but the proposed patch hard-codes all three decisions and would advertise `allow-always` even when effective `ask` is `always`, where current main filters and rejects that decision.

## What This Changes

The branch changes forwarded exec-approval fallback notifications and tests to render ready-to-copy `/approve <id> <decision>` command lines, using inline code for markdown-capable channels and plain text for non-markdown channels.

## Best Possible Solution

Generate fallback command lines from the same allowed-decision descriptors used for approval actions, wrap them only for markdown-capable channels, preserve the existing `ask=always` explanation, and add focused coverage for default, non-markdown, and `ask=always` fallback text.

## Reproduction Assessment

Yes. Current-main tests and source provide a high-confidence static reproduction path: an `ask=always` forwarded request must omit `allow-always`, while the PR diff adds that command unconditionally and the resolver rejects it.

## Solution Assessment

No. Pre-filled commands are the right UX direction, but hard-coding every decision is not the narrow maintainable fix; the branch should derive commands from the allowed decision set already modeled in the approval reply helpers.
