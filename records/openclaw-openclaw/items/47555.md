## Summary

Keep open. Current main still allows an explicit root `--profile` to coexist with an inherited `OPENCLAW_LAUNCHD_LABEL`; the macOS launchd resolver prefers that stale label, and `gateway restart` builds its service target from the stale label. The related fix PR #47594 is closed and unmerged, while #65643 remains open for the same stale-label root cause on `gateway status`.

## What This Changes

The issue requests that an explicit global `--profile` selection take precedence over inherited agent-session launchd context when running `gateway restart`.

## Best Possible Solution

Explicit root `--profile` selection should make the profile-derived service identity win for launchd lifecycle, status, plist resolution, and detached restart handoff paths, while custom launchd labels remain honored in contexts that did not pass a root profile.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, `resolveLaunchAgentLabel` returns the stale label and `restartLaunchAgent` targets it. I did not run a live macOS launchctl restart in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow profile-environment/service-identity repair: when a root profile is explicitly accepted, clear or re-derive stale launchd label state before lifecycle/status code resolves service identity. PR #47594 is closed and unmerged and does not cover the remaining launchd-label precedence path.
