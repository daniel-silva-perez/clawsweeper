## Summary

Keep open. Current main still leaves the stale launchd-label failure path in place: global profile setup updates `OPENCLAW_PROFILE`, but launchd resolution still prefers an inherited `OPENCLAW_LAUNCHD_LABEL`, and `gateway restart` builds its target from that resolved label. PR #66000 is now an open implementation candidate for the same root cause and refs this issue, so this should stay open until that PR, or an equivalent patch, lands or is rejected.

## What This Changes

The issue asks for an explicit global `--profile` selection to take precedence over inherited agent-session launchd service identity when running `gateway restart`.

## Best Possible Solution

Review and land PR #66000, or an equivalent narrow patch, so explicit profile setup clears stale OpenClaw-managed launchd labels before lifecycle/status code resolves service identity while preserving intentional custom labels; then close this issue and #65643 as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, `resolveLaunchAgentLabel` returns the inherited label and `restartLaunchAgent` targets it. This review did not restart a real macOS LaunchAgent.

## Solution Assessment

Yes, the best direction is a narrow profile-environment/service-identity fix rather than changing generic lifecycle routing. PR #66000 appears to target that boundary, but it should be reviewed for restart coverage and custom-label semantics before this issue closes.
