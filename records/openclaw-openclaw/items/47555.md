## Summary

Keep open. Current main still lets an explicit root `--profile` coexist with an inherited `OPENCLAW_LAUNCHD_LABEL`, and the macOS launchd lifecycle resolver still uses the inherited label first. That leaves the reported wrong-target restart path valid; the related fix PR is closed and unmerged, and the status variant remains open in #65643.

## What This Changes

The issue reports that `openclaw --profile <name> gateway restart` from inside an agent session can ignore the requested profile and restart the inherited session gateway instead.

## Best Possible Solution

Explicit root `--profile` selection should make profile-derived service identity win for lifecycle and status commands, including plist resolution and detached restart handoff, while preserving custom launchd label overrides for non-profile managed contexts.

## Repair Work Prompt

Fix the stale launchd-label precedence bug for explicit root profiles. When a user runs `openclaw --profile <name> gateway restart` or `gateway status` from a process that inherited `OPENCLAW_LAUNCHD_LABEL`, the explicit root profile must target the profile-derived service identity (`ai.openclaw.<name>`) instead of the inherited session LaunchAgent. Address this issue and #65643 together. Keep the boundary narrow: prefer sanitizing or rederiving profile-owned service identity at the root profile env handoff and/or the lifecycle/status env handoff; preserve intentional custom `OPENCLAW_LAUNCHD_LABEL` behavior for managed service contexts that did not pass an explicit root `--profile`. Cover restart target resolution, plist/status resolution, and detached launchd restart handoff behavior.
