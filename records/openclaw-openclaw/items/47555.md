## Summary

Keep open. Current main still has a deterministic path where an inherited `OPENCLAW_LAUNCHD_LABEL` wins over an explicit `--profile`, so a gateway lifecycle command run from a managed gateway/agent environment can still target the current LaunchAgent instead of the requested profile. The related PR #47594 was closed unmerged in the provided context, and the newer #65643 tracks the same stale-label root cause for status only, not the destructive restart case here.

## Best Possible Solution

Keep this issue open. The best fix is to make explicit root `--profile`/`--dev` take precedence over inherited service labels for gateway lifecycle/status commands, likely by clearing or recomputing `OPENCLAW_LAUNCHD_LABEL` when `applyCliProfileEnv` handles an explicit profile, then adding focused regression tests for cross-profile `gateway restart` and `gateway status` from an environment that already contains the current profile's launchd label.
