## Summary

Keep open. Current main does not implement persisted exec approval decision audit logging, and this PR is an active implementation attempt, but it is not safe to merge as-is because the branch introduces concrete access-control, secret-redaction, event-loop availability, and protocol-contract issues. The related open #63557 is useful adjacent context for gateway tool-call auditing, but it does not supersede this approval-decision audit query.

## What This Changes

The PR adds an append-only exec approval decision JSONL audit log, a new `exec.approval.auditLog` gateway RPC with filters, protocol params/exports, method registration, and custom persisted-argument redaction.

## Best Possible Solution

Keep the feature direction, but revise it so historical approval audit reads use the same approval/admin access boundary as live approval operations, persisted command/argv fields reuse the existing tool redaction coverage, audit reads are async and bounded or streaming, and the gateway protocol docs plus generated client artifacts are updated together.

## Reproduction Assessment

Yes. A focused branch test can show `authorizeOperatorScopesForMethod("exec.approval.auditLog", ["operator.read"])` allows the new RPC, and unit tests against the redactor can show `--token secret`, lower-case `token=secret`, and bearer-style argv entries remain unmasked.

## Solution Assessment

No. Persisting approval decisions is a reasonable additive gateway feature, but this implementation is not the narrowest maintainable version until scope gating, redaction reuse, nonblocking reads, and protocol artifact sync are fixed.

Full review comments:

- **[P1] Gate audit-log reads with approval scope:** `src/gateway/method-scopes.ts:110`
  - body: This adds `exec.approval.auditLog` under `operator.read`, while current approval methods and approval broadcasts require `operator.approvals`. The returned history includes command text, args, agent/session IDs, and resolver identity, so read-only operator tokens would gain access to approval data they cannot see live.
  - confidence: 0.91
- **[P1] Reuse tool redaction for persisted argv:** `src/infra/exec-approval-audit.ts:35-40`
  - body: The array redactor only handles uppercase env-var style `KEY=value` entries. Common argv forms such as `--token secret`, `--password=secret`, lower-case `token=secret`, and bearer-header arguments can still be written to `approvals.jsonl`; reuse the existing tool redaction path or equivalent coverage before persisting argv.
  - confidence: 0.88
- **[P2] Avoid blocking the gateway on audit reads:** `src/infra/exec-approval-audit.ts:85`
  - body: `readFileSync` loads and parses the whole append-only audit log before applying `limit`. A long-lived gateway can accumulate a large file, and one `exec.approval.auditLog` request would block the Node event loop; use an async bounded or streaming read path instead.
  - confidence: 0.83
- **[P2] Register audit params in protocol artifacts:** `src/gateway/protocol/schema/exec-approvals.ts:153-161`
  - body: Adding `ExecApprovalAuditLogParamsSchema` here is not enough for the public gateway protocol. The new params/result surface is missing from `PROTOCOL_SCHEMA_DEFINITIONS`, generated schema types/Swift models, and protocol docs, so generated clients will not learn about the new RPC contract.
  - confidence: 0.79
