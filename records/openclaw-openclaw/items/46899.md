## Summary

Current main still leaves the reported cron delivery mismatch class unresolved. Explicit string-shaped Telegram delivery is supported, but object-shaped persisted `delivery.channel` values such as `{ id: "telegram" }` are still treated as non-strings: direct delivery resolution falls back to `last`, and store-load normalization can delete the channel before execution. The related #46907 fix attempt is closed unmerged, and I did not find equivalent landed code or regression coverage on current main.

## Best Possible Solution

Keep this issue open and land a narrow compatibility fix in the cron delivery owner path: accept object-shaped channel values such as `{ id: "telegram" }` in both `resolveCronDeliveryPlan()` and persisted-job normalization, canonicalize them back to the string channel id, and add focused regression tests for resolver behavior plus store-load normalization.
