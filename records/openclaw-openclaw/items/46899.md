## Summary

Current main does not yet implement the diagnosis from the closed, unmerged #46907. A persisted object-shaped `delivery.channel` is still treated as non-string, stripped or normalized to undefined, and `resolveCronDeliveryPlan()` then falls back to `last`, which is the same class of channel misrouting reported here.

## Best Possible Solution

Keep this issue open and land a narrow cron delivery compatibility fix: accept object-shaped channel values such as `{ id: "telegram" }` at the persisted-job normalization and delivery-plan boundaries, preserve canonical string output, and add regression tests for both direct `resolveCronDeliveryPlan()` and store-load normalization so the route cannot fall back to `last` for this shape.

## Evidence

- **Resolver still falls back to last when channel is not a string:** `resolveCronDeliveryPlan()` reads `delivery.channel` through `normalizeChannel()`. That helper only accepts values handled by `normalizeOptionalLowercaseString()`, so an object such as `{ id: "telegram" }` produces `undefined`; announce delivery then uses `deliveryChannel ?? "last"`.
