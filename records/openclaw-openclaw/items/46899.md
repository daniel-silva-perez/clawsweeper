## Summary

Current main still treats an object-shaped persisted `delivery.channel` as non-string, so it can be stripped during cron store normalization or resolve to `undefined` and fall back to `last`. That leaves the same Telegram-to-Feishu misrouting class reported here unresolved; the related #46907 fix attempt was closed unmerged and no equivalent landed change or regression test is present.

## Best Possible Solution

Keep this issue open and land a narrow cron delivery compatibility fix: accept object-shaped channel values such as `{ id: "telegram" }` at both persisted-job normalization and delivery-plan resolution, preserve canonical string output, and add regression tests for direct resolver behavior plus store-load normalization.
