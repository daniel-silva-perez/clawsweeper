## Summary

Keep open. Current main still routes exit-0 `codex-cli` JSONL output through normal parsing, carries no semantic sandbox-denied/failure flag, and records returned CLI output as a successful completed run. The focused fix candidate #39460 is closed unmerged, and the related #66081 artifact-finalization work explicitly does not change failure classification.

## What This Changes

Make the `codex-cli` runner treat exit-0 Codex JSONL sandbox-denied or failed tool/action runs as failures instead of successful completed runs.

## Best Possible Solution

Land a focused parser/runner failure-classification fix: inspect structured Codex JSONL records for sandbox-denied or failed tool/action/command/file-change/status/error signals after an exit-0 process, fail before `buildCliRunResult`, avoid matching ordinary assistant message text, and cover both parser and runner behavior with regression tests.

## Reproduction Assessment

Yes. The report provides a concrete `codex exec --json --sandbox read-only` reproduction, and current source proves that any exit-0 JSONL stream carrying denial/failure records bypasses OpenClaw failure handling and is recorded as success. I did not run the live mutation repro because this review was read-only.

## Solution Assessment

Yes. A focused #39460-style structured JSONL detection path is the narrowest maintainable fix; #66081's final-message artifact work is complementary but does not solve semantic failure classification.
