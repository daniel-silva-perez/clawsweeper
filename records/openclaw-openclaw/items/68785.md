## Summary

Keep open. The requested command-lane circuit breaker is not implemented on current main, but this PR is dirty against main and carries stale timeout work that has since landed under a different API; it should not merge as-is.

## What This Changes

The PR adds command-lane circuit-breaker options and `CommandLaneCircuitBreakerError`, enables default main-lane saturation rejection, and adds command-queue tests while also carrying an older `maxExecutionMs` timeout commit.

## Best Possible Solution

Reimplement the circuit breaker as a small current-main patch: extend `CommandQueueEnqueueOptions`, add the rejection error and depth/wait checks without reintroducing `maxExecutionMs`, remove or implement priority ordering, and cover default main-lane behavior with focused command-queue tests and a changelog entry if the user-visible behavior is accepted.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a feature/stability PR. The saturation scenario is reproducible in unit tests by blocking a lane, filling queued work to the configured threshold, and asserting that the next enqueue rejects immediately.

## Solution Assessment

No, this PR is not currently the best way to solve it. The maintainable path is to rebase the circuit-breaker portion onto current main's `taskTimeoutMs` command-queue API and avoid adding undocumented or unimplemented queue priority behavior.
