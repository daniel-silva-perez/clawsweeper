## Summary

Keep this issue open. Current main still does not implement Discord trusted webhook ingress: Discord preflight can read a webhook id, but bot-authored webhook messages are still admitted only through broad `allowBots` modes or PluralKit handling, and the Discord config types/schema expose no `trustedWebhooks` or `mapToUser` field. The generic HTTP `/hooks/agent` automation endpoint is useful but does not satisfy the requested same-channel Discord webhook workflow.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it in `extensions/discord` with typed and zod-validated guild `trustedWebhooks` config, an allowlist check before the default bot-message drop, optional `mapToUser` validation against configured guild/channel users, docs/config metadata updates, and focused preflight tests for allowed, denied, mention-only, mapped-user, and bound-thread echo cases.

## Risks / Open Questions

- `mapToUser` changes effective sender identity, so an implementation needs explicit validation against configured guild/channel users and clear audit/log behavior.
- Trusted webhook ingress must preserve existing own-message and bound-thread webhook echo suppression to avoid reply loops.
- `allowBots="mentions"` is a broader workaround than the requested allowlist because any bot that mentions OpenClaw can pass that gate.
