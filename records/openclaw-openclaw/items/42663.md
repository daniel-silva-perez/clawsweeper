## Summary

Keep this issue open. Current main still does not implement Discord trusted webhook ingress: webhook IDs are resolved, but bot-authored webhook messages are still gated only by broad `allowBots` / `allowBots="mentions"`, there is no guild-level `trustedWebhooks` config, and sender identity cannot map a trusted webhook to a configured user. The prior ClawSweeper keep-open comment remains accurate against current main.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it in `extensions/discord` with typed and zod-validated guild `trustedWebhooks` config, a trusted-webhook allowlist check before the default bot-message drop, `mapToUser` authorization against configured guild/channel users, preserved echo guards, docs/config metadata, and focused preflight tests for allowed, denied, mention-only, mapped-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` would alter effective sender identity, so any implementation needs explicit authorization against configured guild/channel users and clear audit behavior.
- Trusted webhook ingress must preserve existing own-message and bound-thread webhook echo suppression to avoid response loops.
- The implemented HTTP hook feature is related automation, but it does not satisfy the reporter's same-channel Discord webhook workflow.
