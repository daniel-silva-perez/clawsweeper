## Summary

Keep this issue open. Current main still does not implement Discord trusted webhook ingress: Discord preflight reads a webhook ID, but bot-authored webhook messages are still controlled only by broad `allowBots` modes plus PluralKit handling, and the Discord config types/schema do not expose `trustedWebhooks` or `mapToUser`. The related HTTP hooks surface is useful automation, but it runs external HTTP-triggered agent turns and does not satisfy the reporter's same-channel Discord webhook workflow.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it in `extensions/discord` with typed and zod-validated guild `trustedWebhooks` config, an allowlist check before the default bot-message drop, optional `mapToUser` validation against configured guild/channel users, preserved echo guards, docs/config metadata updates, and focused preflight tests for allowed, denied, mention-only, mapped-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` would alter effective sender identity, so any implementation needs explicit validation against configured guild/channel users and clear audit behavior.
- Trusted webhook ingress must preserve existing own-message and bound-thread webhook echo suppression to avoid reply loops.
- `allowBots="mentions"` is a broader workaround than the requested allowlist because any bot that mentions OpenClaw can pass that gate.
