## Summary

Keep this issue open. Current main still does not implement Discord trusted webhook ingress: webhook IDs are read, but bot-authored webhook messages are still gated by broad `allowBots` / `allowBots="mentions"`, guild config has no `trustedWebhooks` field, strict schema validation would reject the proposed key, and sender identity has no `mapToUser` path. The related HTTP hooks feature is useful automation, but it does not satisfy the reporter's same-channel Discord webhook workflow.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it in `extensions/discord` with typed and zod-validated guild `trustedWebhooks` config, an allowlist check before the default bot-message drop, `mapToUser` validation against configured guild/channel users, preserved echo guards, docs/config metadata updates, and focused preflight tests for allowed, denied, mention-only, mapped-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` would change effective sender identity, so any implementation needs explicit authorization against configured guild/channel users and clear audit behavior.
- Trusted webhook ingress must preserve own-message and bound-thread webhook echo suppression to avoid response loops.
- Using `allowBots="mentions"` as a workaround is broader than the requested allowlist because any bot that mentions the OpenClaw bot can pass that gate.
