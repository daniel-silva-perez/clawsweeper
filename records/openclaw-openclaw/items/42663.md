## Summary

Keep open. Current main still lacks Discord trusted webhook ingress: the preflight path can read a webhook ID, but bot-authored webhook messages are only admitted through broad `allowBots` modes or PluralKit, and the Discord config types/schema/docs expose no `trustedWebhooks` or `mapToUser` field. The documented HTTP hook surface is related but does not satisfy the requested same-channel Discord webhook workflow.

## What This Changes

Add a Discord guild-level `trustedWebhooks` allowlist so selected Discord webhooks can trigger agent responses, optionally mapped to a configured user identity.

## Best Possible Solution

If maintainers accept the product and trust model, implement this inside the Discord plugin with typed and zod-validated guild `trustedWebhooks` config, an allowlist check before the default bot-message drop, optional mapped-user validation against configured users, docs/config metadata updates, and focused preflight tests for allowed, denied, mention-only, mapped-user, and bound-thread echo cases.

## Reproduction Assessment

Yes. A guild message with `author.bot=true`, a Discord `webhookId`, default `allowBots` behavior, and no PluralKit identity follows the current preflight path to the bot-message drop; the proposed config key would also be rejected by the strict Discord guild schema.

## Solution Assessment

Unclear. A guild-level webhook allowlist is a narrow direction, but `mapToUser` crosses an identity/security boundary, so maintainer and Discord owner review should settle the exact validation and audit semantics before implementation.
