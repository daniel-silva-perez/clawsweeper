## Summary

Keep open. Current main still lacks the future-`lastRunMs` clock-skew guard, while this PR is a narrow, relevant implementation with focused unit coverage; the remaining work is maintainer review, rebase or cherry-pick, changelog handling, and validation.

## What This Changes

This PR changes heartbeat task due calculation so future `lastRunMs` values are treated as due and adds `isTaskDue` unit coverage for never-run, elapsed, not-elapsed, and clock-skew cases.

## Best Possible Solution

Heartbeat task due checks should treat future last-run timestamps as due, preserve normal elapsed and not-elapsed behavior, keep focused `isTaskDue` regression coverage, and include any required release-note entry before merge.

## Reproduction Assessment

Yes. The pure helper path is enough: call `isTaskDue(now + 5 * 60 * 1000, "30m", now)` on current main and the source logic returns false because the elapsed delta is negative.

## Solution Assessment

Yes. Updating `isTaskDue` is the narrowest maintainable fix because both due-task filtering and post-run timestamp updates already go through that helper; the safer merge path is to rebase or cherry-pick this minimal change and validate it.
