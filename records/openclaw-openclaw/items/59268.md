## Summary

Current main still defaults macOS WebChat to the gateway-wide main session and derives the visible session label from shared session metadata, so the reported cross-Mac shared history and swapped-looking label remain plausible. This needs maintainer direction before changing the macOS default session model.

## What This Changes

Request to make the macOS app chat picker default to a device-unique node session and keep labels/history isolated when multiple Macs are paired to one gateway.

## Best Possible Solution

Define the intended macOS WebChat identity first, then implement it consistently: either keep the gateway main session shared and prevent sender labels from overwriting the shared session label, or make the macOS app create/select a stable per-device session key based on the paired node or `InstanceIdentity.instanceId` with a local display name.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: two macOS apps on the same gateway resolve the same preferred session key, and the displayed label can come from shared session origin metadata updated by the latest sender. I did not run a live two-Mac smoke in this read-only review.

## Solution Assessment

Unclear. The issue's device-unique handshake direction is plausible, but current main already has a unique `instanceId`; the missing piece appears to be WebChat session-key selection and label metadata rather than only replacing the static `clientId`.
