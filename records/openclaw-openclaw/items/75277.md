## Summary

Keep open. Current main still makes generated dreaming reports eligible for ordinary QMD recall, while docs and tests only establish special handling for promotion and artifact listing; the remaining question is the intended product contract for recall, not whether the reported shape exists.

## What This Changes

The issue asks whether generated dreaming report files under `memory/dreaming/**` should participate in ordinary QMD-backed memory recall or be excluded, tagged, ranked lower, or separately filterable.

## Best Possible Solution

Define a clear QMD recall contract for generated dreaming artifacts. Normal recall should either exclude `memory/dreaming/**` by default or surface those hits as an explicit generated/dream-report source with ranking and filter controls, while keeping a review/debug path and updating QMD memory docs and focused tests to match.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to enable QMD defaults and place a matching markdown file under `memory/dreaming/deep/`; the default collection pattern indexes it and the QMD search assembly path has no dream-report filter or generated-artifact source classification.

## Solution Assessment

Unclear. Excluding or demoting generated reports is likely the right retrieval-hygiene direction, but the issue intentionally asks for the intended model, so maintainers should choose exclude-vs-tag-vs-rank semantics before implementation.
