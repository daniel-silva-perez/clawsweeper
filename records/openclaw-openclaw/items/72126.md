## Summary

Keep this PR open: current main still passes the raw OpenAI realtime API-key fallback through to the browser-session and backend bridge paths, and the PR is a narrow owner-plugin fix for the reported keychain-ref failure. I did not find a blocking correctness or supply-chain issue in the diff, but it should still get normal maintainer review, rebase/mergeability handling, and preferably targeted test coverage before landing.

## What This Changes

The PR modifies the OpenAI realtime voice provider to resolve exact `keychain:<service>:<account>` API-key refs via `/usr/bin/security`, cache resolved values in-process, and use the resolved key for browser-session and bridge creation while keeping configuration detection as an input-presence check.

## Best Possible Solution

Land a small OpenAI plugin-owned fix that resolves the keychain ref before any realtime OpenAI auth header is constructed, with focused tests for both `createBrowserSession` and `createBridge`; keep any broader SecretRef/keychain-provider design separate from this bug fix.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to set `OPENAI_API_KEY=keychain:openclaw:OPENAI_API_KEY` and exercise `talk.realtime.session` or backend realtime bridge creation; current main reads that env value literally, and the linked report includes the resulting OpenAI 401 plus a successful local `security find-generic-password` check.

## Solution Assessment

Yes, with a test gap. The PR keeps the fix inside the OpenAI plugin, applies it at the two API-key consumption points, uses a fixed binary path with argv rather than a shell string, and avoids the earlier concern of resolving keychain refs during `isConfigured`.
