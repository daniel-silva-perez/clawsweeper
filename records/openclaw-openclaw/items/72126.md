## Summary

Keep open. The underlying OpenAI Realtime SecretRef bug is still present on current main, and this PR is a useful narrow fix attempt, but it needs a bounded Keychain lookup, a changelog entry, and rebase/conflict follow-through before merge.

## What This Changes

The PR adds provider-local `keychain:<service>:<account>` resolution and in-process caching for OpenAI Realtime API-key lookup, then routes browser session creation and server-side bridge creation through that resolver.

## Best Possible Solution

Land a provider-local OpenAI Realtime API-key resolver that handles `keychain:<service>:<account>` values from provider config and `OPENAI_API_KEY`, caches successful resolutions, bounds the synchronous Keychain call, preserves current main's realtime transport/header behavior, covers the env/config ref paths with focused tests, and records the user-facing fix in the Unreleased changelog.

## Reproduction Assessment

Yes. On current main, set `OPENAI_API_KEY=keychain:openclaw:OPENAI_API_KEY` and invoke the OpenAI Realtime browser-session path; the code passes the literal env value into the bearer header, matching the 401 captured in #72120.

## Solution Assessment

No, not as-is. The provider-local resolver/cache is the right narrow direction, but the synchronous Keychain subprocess needs a timeout and regression tests, and the branch must be updated against current main before it is the maintainable fix.
