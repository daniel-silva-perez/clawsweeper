## Summary

Keep open. The underlying OpenAI Realtime Keychain-ref bug is still present on current main and this PR targets the right provider-local path, but the proposed Keychain command is unbounded in the gateway request path and the PR also needs the required user-facing changelog entry before merge.

## What This Changes

The PR adds a Keychain SecretRef resolver/cache in `extensions/openai/realtime-voice-provider.ts` and routes OpenAI Realtime browser-session and bridge API-key lookup through it.

## Best Possible Solution

Land a provider-local OpenAI Realtime API-key resolver that resolves `keychain:<service>:<account>` for both browser sessions and backend bridges, bounds the synchronous Keychain lookup, covers env/config ref behavior with focused tests, preserves current main's realtime transport behavior, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. On current main, setting `OPENAI_API_KEY` to `keychain:openclaw:OPENAI_API_KEY` and invoking the OpenAI realtime browser-session or bridge path reaches the literal value in the auth path; #72120 includes the resulting OpenAI 401 for the browser-session path.

## Solution Assessment

No, not as-is. The provider-local resolver/cache is the right narrow direction, but the synchronous Keychain lookup needs a fail-closed timeout and the PR needs rebase, tests, and changelog follow-through before it is the maintainable fix.
