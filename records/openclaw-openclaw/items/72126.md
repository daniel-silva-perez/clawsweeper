## Summary

Current main does not implement this PR's intended fix. The OpenAI realtime voice provider still reads `config.apiKey || process.env.OPENAI_API_KEY` directly for both browser client-secret sessions and backend bridge creation, so an env value like `keychain:openclaw:OPENAI_API_KEY` would still be used as the literal bearer token. The PR is a focused fix for the linked #72120 repro and should stay open for review/iteration rather than be closed as obsolete.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate for #72120. The best path is a narrow OpenAI realtime fix that resolves `keychain:<service>:<account>` before both `createBrowserSession` and `createBridge` use the API key, with targeted tests proving the unresolved marker is not sent in Authorization headers. If maintainers want broader SecretRef coverage for realtime provider config, that should be a follow-up or a small generic seam rather than closing this focused PR.
