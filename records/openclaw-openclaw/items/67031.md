## Summary

Keep open. Current main still only partially addresses the report: `agents.defaults.imageMaxDimensionPx` and OpenAI-compatible chat-completion image budgets are configurable, but the requested shared `media.image` byte/input-pixel/dimension contract is absent and several sanitizer, media, gateway, sandbox, and ACP limits remain hardcoded.

## What This Changes

The issue requests configurable image size limits, especially max bytes, max input pixels, max dimensions, and per-request totals, threaded consistently across OpenClaw media sanitization layers.

## Best Possible Solution

Define a backwards-compatible, documented `media.image` contract with global defaults and scoped overrides, preserve existing aliases such as `agents.defaults.imageMaxDimensionPx`, and thread resolved limits through tool-result sanitization, image-ops input guarding, generic media caps, sandbox/media-store staging, gateway attachment parsing, ACP attachment reads, and OpenAI-compatible HTTP paths with generated schema/help/docs updated together.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path by source and test inspection on current main: `media.image.*` is absent from the schema, oversized image tests assert fixed rejection before resize, and gateway attachment tests assert rejection above `MAX_IMAGE_BYTES` before offload.

## Solution Assessment

Unclear. A shared configurable image-limit contract is directionally appropriate, but the exact schema, defaults, hard ceilings, and precedence with existing endpoint-specific settings need maintainer design before implementation.
