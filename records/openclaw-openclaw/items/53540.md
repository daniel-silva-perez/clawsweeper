## Summary

Keep open. Current main has timeout plumbing that can mitigate configured setups, but it does not prove a complete fix for the default OpenRouter to Anthropic large tool-parameter path: OpenRouter still uses the OpenAI-compatible transport, native Anthropic fine-grained tool streaming is not applied to that route, and there is no regression covering a silent OpenAI-compatible stream while large tool-call arguments are generated.

## What This Changes

The issue reports embedded-runner `Network connection lost` failures when large LLM-generated tool-call parameters take longer than the underlying idle/request timeout, and asks OpenClaw to succeed, retry with appropriate backoff, or surface a clearer error.

## Best Possible Solution

Keep this as the canonical tracker for OpenRouter/Anthropic large tool-parameter generation failures. The durable fix should add a synthetic OpenAI-compatible streaming regression that withholds tool-call argument chunks, then implement the chosen provider/runtime policy: supported provider-side tool-argument streaming, a safe timeout policy for silent tool-argument generation, or a tool-loop strategy that avoids requiring one huge `Write` payload.
