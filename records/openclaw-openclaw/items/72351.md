## Summary

Keep open. The PR addresses a real current-main mismatch: the TUI can pass `0` through as a `chat.history` limit while the gateway protocol requires the limit to be at least `1`. The branch fixes that at the TUI boundary with focused regression coverage, and I found no blocking functional or security issue in the diff.

## What This Changes

The branch adds a zero-history guard in TUI session history loading and a regression test so `--history-limit 0` clears the local log/session marker without calling backend history loading.

## Best Possible Solution

Merge this narrow TUI-side fix after normal maintainer and CI validation; keep the gateway `chat.history` contract as a positive page-size limit and treat CLI `--history-limit 0` as a local no-history mode before making a backend request.

## Reproduction Assessment

Yes. Current main parses the CLI value as a number, forwards `historyLimit: 0` through `createSessionActions.loadHistory`, and the gateway `chat.history` schema rejects `limit` values below `1`. A focused unit seam can reproduce this without a live gateway by asserting the backend receives `limit: 0` on current main.

## Solution Assessment

Yes. The PR solves the mismatch at the TUI boundary, where `0` means no history for the user-facing option, instead of loosening the gateway protocol to accept a non-positive history page size.
