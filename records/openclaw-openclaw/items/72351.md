## Summary

Keep this PR open. Current main still preserves an explicit `--history-limit 0`, passes it into the TUI history loader, forwards it as `limit: 0` to `chat.history`, and the Gateway protocol still rejects zero. The PR is narrow, relevant, and not superseded by the related group-chat history-limit item.

## Best Possible Solution

Keep this PR open and review/rebase/land the narrow TUI-side guard with its regression test, or recreate the same behavior on main. The Gateway protocol should continue rejecting `chat.history.limit: 0`; the TUI should treat explicit `--history-limit 0` as local no-history mode and avoid sending `chat.history` at all.
