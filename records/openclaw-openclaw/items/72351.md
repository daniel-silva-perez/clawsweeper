## Summary

Keep this PR open. The runtime change addresses a real current-main bug: the TUI currently forwards `--history-limit 0` to `chat.history`, whose protocol schema rejects zero, and the proposed guard is narrow and covered by a regression test. The remaining issue is a small release/docs gap for the new public zero-history behavior.

## What This Changes

The PR adds a TUI `historyLimit === 0` branch that clears local history state, skips the gateway history call, refreshes session info, renders the session marker, and adds regression coverage.

## Best Possible Solution

Merge the narrow TUI-side guard after documenting that zero disables initial history loading and adding the required changelog entry; the gateway RPC should keep its positive-limit contract.

## Reproduction Assessment

Yes. Current main parses `--history-limit 0`, forwards it as `limit: 0`, and the `chat.history` schema rejects limits below 1; the PR test captures the expected no-gateway-call path.

## Solution Assessment

Yes for the runtime fix. Handling zero before the TUI calls `chat.history` is narrower than relaxing the gateway protocol for all clients, but the public docs and changelog should be updated before merge.
