## Summary

Keep this PR open. Current main still has the CSS-only code-block copy label behavior this PR is trying to fix, so the PR is not obsolete or implemented. The patch needs current-main refinement because main now has separate fenced and indented code-block renderers, while the PR diff only updates the fenced renderer and raises a low-risk but real sanitizer-scope question by globally allowlisting `hidden`.

## Best Possible Solution

Keep the PR open and ask for a current-main update that applies the hidden fallback to both fenced and indented code-block copy buttons, toggles `HTMLElement.hidden` together with the copied CSS class in `renderChat`, and adds regression coverage for the default label visibility. Prefer setting `hidden` only on generated `.code-block-copy__done` elements rather than globally allowlisting `hidden`, unless maintainers explicitly accept that sanitizer behavior.

## Risks / Open Questions

- A direct rebase of the current PR could fix fenced code blocks but leave indented code blocks with the same `CopyCopied!` fallback behavior.
- Globally allowlisting `hidden` in DOMPurify is low XSS risk on current main because raw HTML is escaped, but it broadens sanitizer behavior more than necessary for a generated copy-button span.
- The click handler must keep the CSS class and `HTMLElement.hidden` states symmetric so the copied label does not get stuck after the timeout.
