## Summary

Keep this PR open. Current main still renders code-block copy buttons with both `Copy` and `Copied!` spans and relies on CSS/class toggling to hide the inactive label, so the PR’s intended non-CSS fallback is not implemented. The PR remains a plausible focused UI bug fix, but it needs rebase/refinement for current main’s separate fenced and indented code-block renderers plus a targeted decision on the `hidden` sanitizer allowlist concern raised in review.

## Best Possible Solution

Keep the PR open and ask for a current-main rebase or replacement patch that applies the hidden fallback to both fenced and indented code-block copy buttons, toggles `HTMLElement.hidden` alongside the copied state in `renderChat`, and adds regression coverage for the default hidden/visible label state. Prefer avoiding a global DOMPurify `hidden` allowlist unless maintainers explicitly accept that sanitizer behavior.

## Risks / Open Questions

- The PR patch shown in the provided context predates current main’s split fenced and indented code-block renderers; if applied without rebasing, it could leave the indented-code path with the same CSS-only label behavior.
- Adding `hidden` to the global DOMPurify allowlist is low XSS risk, especially with current raw-HTML escaping, but it still broadens sanitized output. A targeted post-sanitization or generated-element-only hidden assignment would be tighter.
