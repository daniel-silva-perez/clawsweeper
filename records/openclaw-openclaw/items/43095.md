## Summary

Current main still has the reported ambiguity: after `sessions_send` successfully starts the target agent run, a later `agent.wait` timeout is returned as plain `status: "timeout"` without delivery/accepted or reply-wait metadata. The supplied related context shows #53527 was closed while pointing back to this issue, so this remains the canonical open contract issue rather than a duplicate to close.

## Best Possible Solution

Keep this issue open. Implement a backward-compatible `sessions_send` result-contract clarification: after the target `agent` RPC accepts a run, a later wait timeout should include explicit accepted/delivery and reply-wait metadata, while preserving pre-accept failures as errors. Add focused regression coverage for accepted run plus `agent.wait` timeout, and update the public session-tool docs to distinguish delivery failure, accepted delivery, reply wait timeout, and success.

## Risks / Open Questions

- Changing the existing top-level `status: "timeout"` contract could break callers; an additive field such as explicit delivery/accepted state plus reply-wait timeout metadata is safer.
- A fix should only mark post-acceptance wait timeouts as delivered or accepted; failures before the `agent` RPC accepts a run should remain send/start errors.
