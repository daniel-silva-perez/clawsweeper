## Summary

Keep open. Current main still has the reported ambiguity: after the target `agent` RPC has accepted the inter-session run, `sessions_send` can return a bare `status: "timeout"` when only the synchronous reply wait expires, and the overlapping fix PRs are still open rather than merged.

## What This Changes

The issue requests clearer `sessions_send` result semantics so accepted/delivered target runs whose reply wait times out are distinguishable from delivery failures.

## Best Possible Solution

Keep this issue open until one focused contract change lands that reports target acceptance/delivery separately from reply-wait completion, adds regression coverage for accepted-then-timeout behavior, and updates the relevant agent-to-agent documentation if the public result shape changes.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock the target `agent` call as accepted and `agent.wait` as `status: "timeout"`; current `sessions_send` then returns a bare timeout even though the target run was already accepted.

## Solution Assessment

No for current main. The narrow maintainable solution is an additive accepted/pending or delivery-state response for the accepted-then-wait-timeout path, rather than treating it like a delivery failure.
