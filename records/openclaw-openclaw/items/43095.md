## Summary

Keep open. Current main still has the reported ambiguity: `sessions_send` can successfully start the target agent run, then return a bare `status: "timeout"` when the reply wait expires, without accepted/delivery metadata. Two open PRs already target this area, so this should be resolved by maintainer review and consolidation rather than cleanup closure.

## What This Changes

The issue asks for `sessions_send` to distinguish target delivery or acceptance from reply-wait timeout instead of returning the same timeout shape for both cases.

## Best Possible Solution

Land or revise a focused fix that preserves existing consumers while adding explicit accepted/delivery or reply-wait metadata for the accepted-then-timeout path, with regression coverage and docs; coordinate the outcome with #58133 and #68196 while keeping #52497 scoped to model-error context on timeout.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock `agent` as accepted and `agent.wait` as `status: "timeout"`; current `sessions_send` then returns a bare timeout even though the target run was already accepted.

## Solution Assessment

No for current main. The narrow maintainable solution is an additive response contract that separates target acceptance or delivery from reply wait completion, plus focused tests around the accepted-then-timeout path.
