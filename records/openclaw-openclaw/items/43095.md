## Summary

Current main still has the reported contract ambiguity: `sessions_send` first gets an accepted target run, but if the later `agent.wait` deadline expires it returns a plain `status: "timeout"` without delivery/accepted or reply-wait metadata. The related #53527 was closed back to this canonical issue, while #52497 is a narrower open follow-up for model-error context rather than a replacement tracker.

## What This Changes

Distinguish `sessions_send` delivery/start failure from an accepted target-session delivery whose reply wait later times out.

## Best Possible Solution

Add a backward-compatible `sessions_send` result contract that makes the accepted delivery phase explicit when reply waiting times out, document the distinction between delivery/start failures and reply wait timeouts, and cover the accepted-then-timeout path with focused regression tests.
