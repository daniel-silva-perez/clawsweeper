## Summary

Keep open. Current main still has the reported ambiguity: after the target agent run is accepted, `sessions_send` can return a bare `status: "timeout"` when the reply wait expires, and overlapping open PRs are still under review rather than merged.

## What This Changes

The issue requests that `sessions_send` distinguish target acceptance or delivery from reply-wait timeout instead of using the same timeout shape for both cases.

## Best Possible Solution

Land or revise a focused, backward-compatible fix that adds explicit accepted/delivery or reply-wait metadata for the accepted-then-timeout path, with regression coverage and docs for delayed/no-inline-reply behavior; keep the model-error context request tracked separately unless maintainers choose one broader timeout taxonomy update.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock `agent` as accepted and `agent.wait` as `status: "timeout"`; current `sessions_send` then returns a bare timeout even though the target run was already accepted.

## Solution Assessment

No for current main. The narrow maintainable solution is an additive response contract that separates target acceptance/delivery from reply wait completion, plus focused tests around the accepted-then-timeout path.
