## Summary

Keep open. Current main still has the reported ambiguity: `sessions_send` can successfully start the target agent run, then return a bare `status: "timeout"` when the reply wait expires, without accepted or delivery metadata. Open PRs already target this contract area, so cleanup closure would be premature.

## What This Changes

The issue requests that `sessions_send` distinguish target delivery or acceptance from reply-wait timeout instead of using the same timeout shape for both cases.

## Best Possible Solution

Land or revise a focused fix that preserves existing consumers while adding explicit accepted/delivery or reply-wait metadata for the accepted-then-timeout path, with regression coverage and docs; keep #52497 scoped to model-error context on timeout.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to mock `agent` as accepted and `agent.wait` as `status: "timeout"`; current `sessions_send` then returns a bare timeout even though the target run was already accepted.

## Solution Assessment

No for current main. The narrow maintainable solution is an additive response contract that separates target acceptance or delivery from reply wait completion, plus focused tests around the accepted-then-timeout path.
