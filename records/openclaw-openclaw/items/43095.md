## Summary

Keep open. Current main and the latest release still return a bare `status: "timeout"` from `sessions_send` after the target `agent` RPC has already accepted the run, so the reported delivery-vs-reply-wait ambiguity remains. The closed #53527 is a duplicate back to this canonical tracker, while #52497 is a narrower follow-up about timeout error context.

## What This Changes

The issue asks for `sessions_send` to distinguish target-run delivery or acceptance from an inline reply wait timeout.

## Best Possible Solution

Add an additive `sessions_send` result contract that makes accepted target delivery explicit when reply waiting times out, document the distinction between start/delivery failure and reply-wait timeout, and cover the accepted-then-timeout path with focused regression tests.

## Repair Work Prompt

Fix the `sessions_send` accepted-then-reply-timeout ambiguity tracked by https://github.com/openclaw/openclaw/issues/43095. Current behavior: `sessions_send` calls the gateway `agent` method, receives/uses an accepted target run, then if `agent.wait` times out it returns only `{ runId, status: "timeout", sessionKey }`, which looks like delivery failure even though the target session may receive/process the message later. Implement a backward-compatible additive contract: preserve existing `status: "timeout"` unless a stronger compatible local pattern already exists, but include explicit accepted/delivery or reply-wait metadata that distinguishes accepted target delivery from start/delivery failure. Thread only metadata the gateway can know reliably; do not invent durable queued/running state unless it is already available. Update `docs/concepts/session-tool.md` to describe delivery/start failure vs reply wait timeout. Add focused regression coverage for accepted target run plus `agent.wait` timeout in `sessions_send`; update lower-level `agent.wait`/run-wait tests only if new metadata is threaded there. Add a user-facing changelog entry under the active version with attribution, e.g. Thanks @jesseliu1, @marcus-daemon, and @jp9573. Related refs: #43095 canonical, #53527 duplicate closed to this issue, #52497 narrower model-error-context follow-up, #20939 fallback/rate-limit context. Do not change unrelated model fallback/rate-limit policy from #20939 in this PR.
