## Summary

Current main still has the reported ambiguity: after `sessions_send` gets an accepted target run, a later `agent.wait` deadline is returned as plain `status: "timeout"` without explicit accepted/delivery or reply-wait metadata. The related #53527 was closed back to this issue, while #52497 is a narrower open follow-up about surfacing model-error reasons, so this remains the canonical contract issue.

## What This Changes

The issue asks for `sessions_send` to distinguish target-session delivery or acceptance from a later reply-wait timeout instead of making that timeout look like a possible delivery failure.

## Best Possible Solution

Add a backward-compatible `sessions_send` result-contract clarification: once the target `agent` RPC has accepted a run, a later wait timeout should carry explicit accepted/delivery and reply-wait metadata, with focused regression coverage and session-tool docs that separate delivery failure, accepted delivery, reply wait timeout, and success.
