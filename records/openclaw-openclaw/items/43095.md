## Summary

Keep open. Current `main` still returns a bare `status: "timeout"` from `sessions_send` after the target `agent` RPC has accepted the run, so the reported delivery-vs-reply-wait ambiguity remains. The closed #53527 is a duplicate back to this canonical tracker, #52497 is a narrower timeout-error-context follow-up, and open PRs #68196/#58133 are related implementation candidates that need maintainer review rather than another automated repair lane.

## What This Changes

The issue requests that `sessions_send` distinguish target delivery/start acceptance from reply-wait timeout instead of returning the same timeout shape for both.

## Best Possible Solution

Land or revise a focused fix that preserves the existing `status: "timeout"` compatibility while adding explicit accepted/delivery or reply-wait metadata for the accepted-then-timeout path, with docs and regression tests; coordinate that outcome with #68196/#58133 and keep #52497 scoped to timeout error context.
