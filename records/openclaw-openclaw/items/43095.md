## Summary

Current main still has the reported contract ambiguity. `sessions_send` starts the target run successfully, but if the subsequent `agent.wait` times out it returns plain `status: "timeout"` without accepted/delivery or reply-wait metadata, so callers still cannot distinguish delivery failure from post-delivery wait timeout. Related #53527 was closed while pointing back to this issue, so this remains the canonical open item.

## Best Possible Solution

Keep this issue open. Implement a backward-compatible `sessions_send` result-contract clarification: after the target `agent` RPC accepts a run, a later wait timeout should include explicit accepted/delivery and reply-wait timeout metadata while preserving pre-accept failures as errors. Add focused regression coverage for accepted run plus `agent.wait` timeout, and update the public session-tool docs to distinguish delivery failure, accepted delivery, reply wait timeout, and success.

## Risks / Open Questions

- Changing the existing top-level `status: "timeout"` value could break callers; an additive field such as accepted/delivery state plus reply-wait status is safer.
- A fix must only mark post-acceptance wait timeouts as accepted or delivered; failures before the `agent` RPC accepts the run should remain send/start errors.
- `agent.wait` currently may not know queued versus running state when it times out, so any state metadata should be best-effort unless the gateway adds a reliable in-flight snapshot source.
