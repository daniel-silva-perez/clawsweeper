## Summary

Keep open. The PR is aimed at the right ACPX boundary and preserves the current bundled 0.12.0 path, but the new guard misses explicit old npm ranges such as `@0.11`, so it does not fully prevent the opaque stale-adapter startup failure it is meant to catch.

## What This Changes

The PR adds a Codex ACP `gpt-5.5` compatibility guard for explicitly pinned pre-0.12.0 adapter commands, plus focused ACPX runtime tests, docs, and a changelog entry.

## Best Possible Solution

Keep the wrapper-boundary guard, but make it evaluate explicit npm version/range specs conservatively so any command that can resolve below `@zed-industries/codex-acp@0.12.0` fails before `session/new`, while unpinned commands, local wrappers, and `0.12.0+` specs continue to pass.

## Reproduction Assessment

Yes for the review finding: an explicit command like `npx @zed-industries/codex-acp@0.11` has no patch segment, so the added `parseSemver` returns `undefined` and `supportsCodexAcpGpt55` treats it as supported. That path can still forward `openai-codex/gpt-5.5` to the old adapter instead of failing early.

## Solution Assessment

No as written. The boundary and error shape are maintainable, but the version check should parse npm semver ranges or conservatively reject explicit old ranges rather than assuming every unparsable explicit spec supports `gpt-5.5`.
