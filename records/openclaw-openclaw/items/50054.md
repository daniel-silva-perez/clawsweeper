## Summary

Keep this PR open. Current main still does not implement ACP distributed session locking: there is no ACP-specific lock manager or `OPENCLAW_ACP_SESSION_LOCK_*` config, automatic ACP dispatch and `/acp steer` still call `acpManager.runTurn()` directly, and the current ACP manager only serializes turns inside one process. The PR also still needs maintainer review because the provided review comments flag unresolved behavior around dropped queued turns, Redis renewal failure, and missing steer coverage, even though the author did address the earlier per-command timeout and generic `REDIS_URL` fallback feedback.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to revise the design so distributed ACP locking is applied at the ACP control-plane/turn boundary shared by all production `runTurn` entrypoints, preserves existing queued-turn semantics, handles lost Redis renewals without silently losing exclusivity, covers `/acp steer`, cleans up failed Redis setup sockets, and ships with focused unit/integration tests plus env/docs updates.

## Risks / Open Questions

- The PR is a plausible feature/fix candidate, but current main has not already shipped it, so closing would discard still-relevant work.
- The proposed lock placement can skip queued same-session turns in the default local-lock path instead of preserving the existing in-process queue behavior.
- Best-effort Redis renewals can let a long turn outlive its distributed lock and allow another worker to start the same session concurrently.
- The PR does not cover `/acp steer`, which is another production ACP turn entrypoint.
- The Redis implementation is security-sensitive runtime networking code; even without dependency or CI changes, it needs careful review for timeout cleanup, secret handling, and fail-closed behavior.
