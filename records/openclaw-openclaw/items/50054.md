## Summary

Keep this PR open. Current main still does not implement distributed ACP session locking, Redis fail-closed locking, the `OPENCLAW_ACP_SESSION_LOCK_*` env surface, or `acp_execution_locked`. The PR remains a plausible implementation candidate, but the current diff is not safe to close or merge as-is because it changes default queued-turn behavior, only wraps normal ACP reply dispatch, leaves `/acp steer` uncovered, and adds a custom Redis TCP/TLS lock path that needs maintainer review.

## Best Possible Solution

Keep the PR open for focused rework against current main. The best path is to place the distributed lock at a shared ACP turn boundary that covers normal bound-message dispatch and `/acp steer`, preserve current queued-turn semantics or deliberately document and test a new policy, fail closed only when `OPENCLAW_ACP_SESSION_LOCK_REDIS_URL` is explicitly configured, handle Redis command timeouts/socket cleanup/owner-checked renew and release, define what happens when renewal fails during a long turn, and update `.env.example`, public ACP docs, and targeted tests.

## Risks / Open Questions

- Closing this PR would drop still-unmerged distributed ACP locking work; current main only has in-process ACP turn serialization plus unrelated file-based transcript write locks.
- Merging the PR as-is could regress current queued-turn semantics because the default local lock path skips a contended second turn instead of letting `AcpSessionManager` queue and run it.
- The proposed lock is wired only into normal ACP reply dispatch; `/acp steer` remains a direct `runTurn` entrypoint.
- The proposed renewal heartbeat logs renewal failures and lets long turns continue, so a Redis lock can expire during a long turn and allow cross-worker overlap.
- The custom Redis protocol/client code is a new network-sensitive path and should receive explicit maintainer reliability/security review before merge.
