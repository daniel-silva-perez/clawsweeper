## Summary

Keep this PR open. The author addressed earlier review feedback around Redis command timeouts and removing the generic REDIS_URL fallback, but current main still does not implement distributed ACP session locking, Redis fail-closed behavior, the OPENCLAW_ACP_SESSION_LOCK_* env surface, or acp_execution_locked. The PR remains a plausible implementation candidate, but it still needs maintainer rework because the proposed lock only wraps normal ACP reply dispatch, leaves /acp steer outside the boundary, changes current same-session queued-turn behavior, and adds a custom Redis TCP/TLS path that deserves explicit reliability and security review.

## Best Possible Solution

Keep this PR open for focused rework against current main. The best implementation should put distributed locking at a shared ACP turn boundary that covers normal bound-message dispatch and /acp steer, preserve queued same-session semantics or explicitly document and test a changed policy, fail closed only when OPENCLAW_ACP_SESSION_LOCK_REDIS_URL is configured, define abort behavior when renewal fails during a long turn, and update .env.example, public ACP docs, and targeted tests before merge.

## Risks / Open Questions

- Closing this PR would drop still-unmerged distributed ACP locking work; current main only provides in-process ACP turn serialization, not cross-worker Redis locking.
- Merging the current PR as-is could regress default same-session behavior by skipping contended turns instead of preserving the existing queued execution semantics.
- The proposed lock is not applied to /acp steer, so a multi-worker deployment could still overlap steer and prompt turns for the same ACP session.
- The proposed renewal heartbeat is best-effort; if renewal fails during a long turn, the Redis lock can expire while the original turn continues.
- The custom Redis protocol/client path handles credentials and network I/O directly, so socket cleanup, command timeouts, TLS behavior, and owner-checked release/renew need focused maintainer review.
