## Summary

Keep this PR open. Current main does not already implement the proposed distributed ACP session locking: ACP reply dispatch and `/acp steer` still call `AcpSessionManager.runTurn()` directly, and there is no `session-lock-manager` module or ACP Redis lock env documentation on main. The PR is still a plausible implementation candidate, but the open review feedback is substantive: it can drop queued same-session ACP turns in the default local mode, does not cover `/acp steer`, lets long turns continue after Redis lock renewal failure, and still appears to leak a Redis socket if AUTH/SELECT fails. Security pass found no workflow, dependency, lockfile, or package-publishing changes, but the new raw Redis TCP/TLS path needs more hardening before merge.

## Best Possible Solution

Keep the PR open, but revise the design before merge. The best path is to put a distributed ACP session lease at the shared ACP turn boundary so both prompt dispatch and `/acp steer` are covered, preserve same-session queueing instead of dropping contended turns, define fail-closed behavior for renewal loss during long turns, close Redis sockets on all initialization failures, and add regression tests plus env/docs updates for the final contract.

## Risks / Open Questions

- The PR's default local lock path can turn normal same-session overlap into `acp_execution_locked` skips, dropping a user turn that current main queues and executes.
- Distributed locking is only wired into `tryDispatchAcpReply`; `/acp steer` remains able to overlap with normal ACP prompts in multi-worker Redis deployments.
- Redis renewal failures are log-only in the PR, so a long turn can outlive its lease and allow another worker to acquire the same session.
- The raw Redis TCP/TLS client still needs cleanup hardening for AUTH/SELECT failures, and the per-command timeout is not covered by an executable network-level test.
