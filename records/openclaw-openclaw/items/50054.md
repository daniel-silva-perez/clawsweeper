## Summary

Keep this PR open. Current main still does not implement distributed ACP session locking or the ACP-specific Redis lock env surface, while the PR remains a plausible but not merge-ready implementation candidate. The current ACP manager only serializes turns inside one process, and the PR discussion/review context still flags real behavior and hardening gaps around queued same-session turns, `/acp steer`, renewal loss, and Redis socket cleanup.

## Best Possible Solution

Keep the PR open and revise the design before merge. The best path is to place a distributed ACP session lease at a shared ACP turn boundary so prompt dispatch and `/acp steer` use the same contract, preserve same-session queueing in the default local mode, define fail-closed behavior when renewal is lost during a long turn, close Redis sockets on all initialization failures, and update tests plus env/docs for the final contract.

## Risks / Open Questions

- Default local-lock behavior in the PR can drop a same-session ACP turn that current main queues and executes.
- A dispatch-only lock does not cover `/acp steer`, leaving another production ACP turn path outside the proposed distributed lease.
- Redis renewal failures are described as best-effort/log-only, which can let a long turn outlive its lease and overlap with another worker after TTL expiry.
- The PR introduces a custom Redis TCP/TLS command path; review feedback still calls out socket cleanup hardening for AUTH/SELECT failure cases.
