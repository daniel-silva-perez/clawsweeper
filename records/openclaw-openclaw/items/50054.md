## Summary

Keep this PR open. Current main still does not implement distributed ACP session locking, Redis fail-closed locking, the `acp_execution_locked` skip reason, or the proposed env/docs surface. The PR is a plausible implementation candidate, but the review discussion and diff still need maintainer follow-up around queued-turn semantics, renewal failure behavior, and `/acp steer` coverage before merge.

## Best Possible Solution

Keep the PR open for a focused current-main rework. The best path is to put the distributed lock at a shared ACP turn boundary that covers both normal bound-message dispatch and `/acp steer`, preserve current queued-turn behavior or document and test an intentional new policy, fail closed only when `OPENCLAW_ACP_SESSION_LOCK_REDIS_URL` is explicitly configured, handle command timeouts/socket cleanup/owner-checked renew and release, and add targeted tests plus `.env.example` and public ACP docs updates. The author already addressed earlier Redis command timeout and generic `REDIS_URL` fallback comments in the PR branch, which is useful context for the rework.

## Risks / Open Questions

- Closing this PR would drop still-unmerged distributed ACP locking work; current main only has in-process ACP turn serialization.
- Merging the PR as-is could regress current queued-turn semantics because the proposed default local lock path skips a contended second turn instead of letting `AcpSessionManager` queue and run it.
- The proposed renewal heartbeat logs failures and lets long turns continue, so a Redis lock can expire during a long turn and allow another worker to overlap the same ACP session.
- The proposed wiring only covers normal ACP reply dispatch; `/acp steer` remains a direct `runTurn` entrypoint in current main.
- The custom Redis protocol/client code is a new network-sensitive path and should get explicit maintainer reliability/security review even though the diff does not add dependencies or CI/package execution changes.
