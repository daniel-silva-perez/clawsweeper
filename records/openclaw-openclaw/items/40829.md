## Summary

Keep this PR open. The PR discussion correctly identifies a useful media fix, but current main still converts under-cap non-alpha PNGs to JPEG and still only renames JPEG outputs for HEIC sources. The PR cannot land as-is because it targets the old `src/web/*` media paths; it needs a rebase onto the current shared media loader plus the validation concern from review should be handled before merge.

## Best Possible Solution

Rebase the useful parts of this PR onto `src/media/web-media.ts` and current media tests. The best implementation is to validate/decode PNG bytes first, preserve already-under-cap verified PNGs without JPEG re-encoding, normalize filenames whenever the actual output is JPEG, and add regression coverage for under-cap PNG preservation plus PNG-to-JPEG filename normalization.

## Risks / Open Questions

- The PR is stale and targets old `src/web/*` files, so it cannot be merged directly into current main.
- A direct rebase of the early return could accept malformed or mislabeled `.png` inputs without decoding them first; the fix should preserve under-cap PNGs only after validation.
- The filename normalization change is in a shared plugin SDK media helper, so regression coverage should include the shared loader and a plugin-facing path such as WhatsApp.
