## Summary

Keep this PR open. Current main still re-encodes under-cap non-alpha PNGs through the JPEG optimizer, still only normalizes JPEG output filenames for HEIC sources, and this PR now targets deleted `src/web/*` paths. The earlier ClawSweeper review remains accurate after rechecking current main at `16906780fd2c481308ec4b80f96eb3b93d0add46`.

## Best Possible Solution

Keep this PR open and ask for a rebase onto `src/media/web-media.ts` plus current media tests. The best fix is to preserve already-under-cap, validated PNG images without JPEG re-encoding, normalize filenames for every actual JPEG output, and add regression coverage for under-cap PNG preservation plus PNG-to-JPEG filename normalization.

## Risks / Open Questions

- The PR is stale against current main because it targets deleted `src/web/*` media files.
- The proposed early return needs PNG payload validation before returning original bytes; extension-derived `image/png` can otherwise bypass the current decode/re-encode failure path for malformed inputs.
- Because the shared loader is exposed through the plugin SDK and used by WhatsApp, the rebased fix should keep coverage at the shared loader boundary and at least one plugin-facing path.
