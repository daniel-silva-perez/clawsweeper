## Summary

Keep this PR open. Current main still does not implement the PR’s intended behavior: under-cap non-alpha PNGs are still routed through JPEG optimization, and JPEG output filenames are still normalized only for HEIC sources. The PR is also stale against the relocated shared media loader and should address the existing PNG payload-validation concern before merge.

## Best Possible Solution

Rebase the useful behavior onto `src/media/web-media.ts` and current media tests. Preserve verified under-cap PNGs only after decoding or otherwise validating PNG bytes, normalize filenames whenever the actual optimized output is JPEG, and add regression coverage for both the shared loader and the WhatsApp/plugin-facing media path.

## Risks / Open Questions

- A direct rebase of the PR’s early return could preserve malformed or mislabeled `.png` bytes without decoding or validating them first.
- The patch is stale against current main because it targets relocated `src/web/media*` files instead of the active shared `src/media/web-media.ts` loader.
- Filename normalization affects the shared plugin SDK helper, so regression coverage should include the shared loader and a plugin-facing path such as WhatsApp.
