## Summary

Keep open. Current main still hardcodes MiniMax usage polling to the CN host, so this PR is not obsolete or implemented. The branch addresses a valid bug, but its config-priority lookup can choose a usage host that does not match the credential source, so it needs maintainer revision alongside #63073 and #65054 rather than cleanup closure.

## What This Changes

The PR adds a `baseUrl` argument to MiniMax usage fetching, derives the coding-plan usage host from configured MiniMax provider config with CN fallback, and passes that config from MiniMax provider registration.

## Best Possible Solution

Implement MiniMax-owned usage host resolution that follows the same auth/config source that produced the usage token, preserves an explicit CN/global fallback policy, updates the MiniMax usage docs, and adds focused URL-routing tests for global, CN, missing/invalid baseUrl, and both-provider-config cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the credential source when choosing the usage host:** `extensions/minimax/provider-registration.ts:246-250`
  - body: When both `minimax` and `minimax-portal` configs exist and no portal OAuth token is available, `resolveUsageAuth` can resolve the API key from `minimax` first, but this new lookup prefers the `minimax-portal` `baseUrl`. That can send a CN/global bearer token to the other regional usage host and reproduce 403s or wrong usage data. Choose the usage host from the same source that produced `ctx.token`.
  - confidence: 0.9
