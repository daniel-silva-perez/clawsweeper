## Summary

Keep open. Current main still hardcodes MiniMax usage polling to the CN host and does not pass configured endpoint data into `fetchMinimaxUsage`, so this PR is not obsolete or implemented. The PR addresses a real bug, but the current diff needs maintainer revision because the proposed host selection can diverge from the credential source and lacks URL-routing regression coverage.

## What This Changes

This PR adds a `baseUrl` argument to MiniMax usage fetching, derives the coding-plan usage host from the configured MiniMax provider base URL with CN fallback, and passes that config from MiniMax provider registration.

## Best Possible Solution

Implement source-aligned MiniMax usage host resolution: derive the usage host from the same provider config or auth source that supplied the token, preserve documented CN/global fallback behavior, and add focused request-URL tests for global, CN, missing/invalid baseUrl, and both-provider-config cases while reconciling #63073 and #65054.

## Risks / Open Questions

- The PR’s config lookup prefers `minimax-portal` baseUrl while API-key fallback resolves credentials from `minimax` before `minimax-portal`, so both-provider configurations can send a token to the wrong regional MiniMax host.
- The PR changes the outbound host for a bearer-token request; without credential-source alignment this is a security-adjacent regression risk even though no supply-chain files are touched.
- URL routing is not covered by current tests or the PR diff shown in context, so global, CN, invalid/missing baseUrl, and both-provider-config behavior could regress.
- Open PR #63073 proposes a different global-host fix for overlapping MiniMax usage behavior, so maintainers should reconcile default CN/global behavior before landing either change.
