## Summary

Keep open. Current main still hardcodes the MiniMax CN usage endpoint, so the reported bug remains valid, but this PR should not merge as-is because the new host selection can mismatch the credential source when `minimax` and `minimax-portal` configs point at different regions.

## What This Changes

The PR adds a `baseUrl` argument to MiniMax usage fetching, derives the usage polling host from MiniMax provider config with a CN fallback, and passes provider config from MiniMax registration into the usage fetch hook.

## Best Possible Solution

Implement source-aligned MiniMax usage host resolution: carry or derive the usage host from the same credential source that supplied the token, preserve CN/global fallback behavior intentionally, add focused URL-selection tests for global, CN, missing/invalid `baseUrl`, and dual-provider-config cases, and reconcile #63073 with #65054.

## Reproduction Assessment

Yes. The current-main bug is provable from source because MiniMax usage polling ignores configured global `baseUrl` and always calls the CN host; the PR-specific regression is provable by configuring both `minimax` and `minimax-portal` with different regions and no portal OAuth token.

## Solution Assessment

No. The configured-host direction is better than the current hardcoded endpoint, but this implementation is not the best fix because it selects host priority independently from credential priority; a safer fix should bind token source and usage host together and add regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Select the usage host from the credential source:** `extensions/minimax/provider-registration.ts:247-250`
  - body: The PR picks `minimax-portal` `baseUrl` before `minimax`, but usage auth falls back to API keys from `minimax` before `minimax-portal` when portal OAuth is absent. With both provider configs present in different regions, a `minimax` key can be sent to the other region's usage host, preserving the 403 or wrong-quota failure this patch is trying to fix.
  - confidence: 0.91
