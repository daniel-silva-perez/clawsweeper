## Summary

Current main still lacks configurable `fallbackChannel`/`fallbackChannels` support for hook mapping delivery. Hook mappings, schema validation, mapping transforms, dispatch payloads, cron delivery, docs, and the final outbound send path still model one delivery channel/target. The existing `fallbackChannel` identifiers in outbound code are pre-delivery selection helpers, not post-failure hook delivery failover. The prior maintainer/Codex review comment also kept this open, and current main does not add the requested feature.

## Best Possible Solution

Keep this open for a typed hook delivery fallback feature. Add `fallbackChannel` or ordered `fallbackChannels` to hook mapping config and schema, carry it through mapping resolution/transforms and `HookAgentDispatchPayload`, persist it in `CronDelivery`, and make outbound delivery try alternates only after the primary delivery fails. Document the difference from `channel: "last"`, `bestEffort`, and `failureDestination`, and add focused tests for success, failover, and all-targets-failed behavior.
