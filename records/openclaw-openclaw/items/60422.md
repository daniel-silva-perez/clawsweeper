## Summary

Keep this issue open. Current main still routes media transcription through `executeWithApiKeyRotation` without a transient-error retry policy: the default retry decision is only `isApiKeyRateLimitError`, the classifier only matches rate-limit/quota patterns, and single-key setups still break after the first failed attempt. The related media pipeline reports are useful context, but they do not supersede the concrete retry/backoff gap for 5xx, timeout, or network failures.

## Best Possible Solution

Implement a conservative transient retry path for provider API-key execution, likely as an explicit option or helper used by media understanding and embedding call sites. Cover single-key behavior with tests for HTTP 500/502/503/504, AbortError/timeout, and network errors, preserve existing API-key rotation on 429, and keep retry budgets/backoff small and opt-in where idempotency is clear.
