## Summary

Keep this open. Current main still implements API-key rotation as a rate-limit-only alternate-key retry, and the media audio/video plus Google embedding call sites do not add a transient 5xx, timeout, or network retry policy. The related media issues fixed logging and provider availability symptoms, but they do not supersede this concrete single-key retry/backoff gap.

## What This Changes

Retry provider API-key execution on transient 5xx, timeout, and network errors, including single-key setups, instead of only rotating keys on 429/rate-limit errors.

## Best Possible Solution

Add a conservative transient retry path for idempotent provider API-key execution, preserving existing 429 alternate-key rotation while adding same-key retry/backoff for 5xx, AbortError/timeout, and network errors where media transcription/video and embedding calls are safe to retry. Cover single-key and multi-key behavior with focused tests and keep docs aligned if the public API-key rotation contract changes.

## Reproduction Assessment

Yes. A source-level reproduction is straightforward: call `executeWithApiKeyRotation` with one key and an `execute` function that throws `Error("Audio transcription failed (HTTP 500)")` or an AbortError; current code makes one call and rethrows because the default retry predicate is rate-limit-only.

## Solution Assessment

Yes, but the narrowest maintainable solution is an explicit transient retry policy or option for idempotent provider operations, not an unscoped global retry of every API-key wrapped call. Existing `retryAsync`/provider error helpers can likely be reused, with tests proving retry budgets and non-retryable 4xx behavior.
