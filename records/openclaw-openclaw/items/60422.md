## Summary

Keep this issue open. Current main still retries API-key execution only by rotating to another key on rate-limit-looking errors; it does not provide same-key retry/backoff for 5xx, timeout, or network failures.

## Best Possible Solution

Keep this issue open. Implement a conservative transient retry policy either in `executeWithApiKeyRotation` or as an explicit opt-in helper used by media-understanding and embedding call sites. Cover single-key and multi-key behavior for HTTP 500/502/503/504, `AbortError`/timeout, and network errors, while preserving existing 429 API-key rotation semantics.

## Evidence

A single configured key therefore gets no retry attempt for transient server or network failures.

## Risks / Open Questions

- A fix should use a small, explicit retry budget with backoff/jitter and avoid retrying non-idempotent multi-step flows unexpectedly.
- Provider SDKs may already perform some retries, so OpenClaw should avoid duplicating long internal retry behavior while still covering the single-key helper path requested here.
