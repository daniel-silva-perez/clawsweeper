## Summary

Current main does not implement usage-aware load balancing for multi-account auth profiles. Auth selection still resolves explicit/configured/stored profile order, prefers OAuth over token/API-key, sorts by lastUsed, and moves cooldowned/disabled profiles later. Anthropic usage-window data exists and is exposed through provider-usage/status surfaces, but it is not used as a request-time routing score or proactive circuit breaker. Related issues #41135, #24107, and #45771 remain open and overlap this design rather than superseding it.

## Best Possible Solution

Keep this issue open and consolidate it with #41135, #24107, and #45771 into an opt-in generic auth-profile routing contract. The design should consume provider-owned usage snapshots, preserve today’s session-sticky/default behavior, support proactive thresholds and fallback for profiles without usage data, expose routing decisions/status, and update config schema, docs, and tests before enabling Anthropic OAuth usage-aware balancing.
