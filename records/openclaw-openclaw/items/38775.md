## Summary

Current main still does not implement usage-aware auth-profile routing. Anthropic/provider usage snapshots exist for status and usage reporting, but Gateway request selection remains explicit-order, credential-type/lastUsed, and cooldown based, with no `auth.balancing` config, proactive utilization circuit breaker, polling/cache/shadow accounting, or async per-profile routing API.

## Best Possible Solution

Keep this issue open for maintainer product/API design. A solid implementation should add first-class usage-aware auth-profile routing across Gateway request paths: provider-owned Anthropic usage parsing mapped to per-profile headroom, cached or polled utilization state, proactive 5h/7d circuit thresholds, shadow accounting for concurrent starts, fallback to current reactive cooldown rotation for profiles without usage data, `auth.balancing` schema/help/docs, and regression tests for utilization-based profile choice plus cooldown interaction. If maintainers prefer plugin ownership, first add a documented async pre-request profile-selection API with usage-cache integration.
