## Summary

Current main still does not implement usage-aware load balancing for multi-account auth profiles. Anthropic and other provider usage snapshots are available for reporting/status surfaces, but request-time Gateway auth selection remains explicit order or profile type plus lastUsed, with reactive cooldown/failover after failures. Related routing/rate-limit items remain open and do not supersede this issue yet.

## Best Possible Solution

Keep this open and consolidate design with #41135, #24107, and #45771. The best path is an opt-in generic auth-profile routing contract that can consume provider-owned usage snapshots, preserve existing session stickiness/default behavior, support proactive thresholds and fallback for profiles without usage data, expose route decisions/status, and update config schema, docs, and tests before enabling Anthropic OAuth usage-aware balancing.

## Risks / Open Questions

- A narrow Anthropic-only implementation would likely violate the core/plugin boundary; the routing contract should stay generic and let provider plugins contribute usage snapshots or scoring inputs.
- Polling or querying usage during request selection could add latency, quota pressure, and concurrency races unless the design includes caching, freshness bounds, and in-flight/shadow accounting.
- The request overlaps open broader work in #41135, #24107, and #45771, so config shape and routing semantics need maintainer alignment before implementation.
