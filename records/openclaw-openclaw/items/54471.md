## Summary

Keep open. Current main still drops ACP runtime status events before the parent relay, and this PR only adds a handler for an agent-event stream that current ACP code does not emit. The branch also does not count relayed system text as progress, so it can still produce false no-output stall notices after visible clarification/progress messages.

## What This Changes

The PR adds a `system_event` branch to `startAcpSpawnParentStreamRelay()` so non-empty text can be emitted to the parent session with a fallback ACP context key.

## Best Possible Solution

Update the branch in place so the actual ACP runtime status/progress source is translated into parent-session system events, treat relayed text as progress for stall/resume state, preserve existing assistant/commentary/lifecycle behavior, and add focused parent-relay coverage plus a user-facing changelog fix if merged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an ACP `runTurn` event with `type: "status"` during a `streamTo: "parent"` spawn: `agent-command.ts` drops it before `emitAgentEvent`, and the parent relay only sees assistant/lifecycle streams on current main.

## Solution Assessment

No. The PR is pointed at the right file but not the actual source contract; the maintainable fix should wire ACP runtime `status` events into the relay and update progress accounting rather than only listening for a currently unproduced `system_event` stream.
