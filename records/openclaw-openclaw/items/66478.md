## Summary

Keep open. Current main still trims Feishu reaction input and passes it unchanged into both the add and remove/list reaction API paths, so Unicode emoji such as thumbs-up would still reach Feishu as raw emoji. This PR remains a focused implementation candidate for the linked Feishu regression, but it should be revised before merge to preserve the Feishu lazy runtime boundary and validate the mapping table against the supported Feishu/Lark emoji type vocabulary.

## Best Possible Solution

Keep this PR open for maintainer review, or land an equivalent Feishu-owned fix. The best implementation is a lightweight Feishu emoji normalizer that can be imported without loading the runtime client, applied before both addReactionFeishu and listReactionsFeishu in the react action, with tests for Unicode thumbs-up, Feishu type passthrough, mixed-case input, whitespace trimming, unknown input behavior, and remove/list filtering.

## Risks / Open Questions

- Merging the PR as-is could weaken the Feishu staged/lazy runtime dependency model because channel.ts would import reactions.ts at top level, and reactions.ts imports the runtime client path that loads @larksuiteoapi/node-sdk.
- The proposed Unicode mapping table should be verified against the currently supported Feishu/Lark emoji_type vocabulary before landing, otherwise OpenClaw could normalize to unsupported values or miss common supported values.
