## Summary

Keep this PR open. It targets a real Feishu reaction bug and the helper-level normalization direction is right, but the current branch still maps several Unicode inputs to undocumented reaction type strings and still pulls the runtime-heavy reaction module into the channel entry path.

## What This Changes

The PR adds Feishu reaction emoji normalization in `extensions/feishu/src/reactions.ts` and `extensions/feishu/src/channel.ts`, adds focused reaction tests, and records the Feishu reaction fix in the changelog.

## Best Possible Solution

Revise the PR so Feishu reaction normalization uses the official exact emoji_type vocabulary, preserves documented mixed-case values, and keeps normalization either inside the lazy reaction helpers or in a small runtime-free helper shared by `channel.ts` and `reactions.ts`. Then land the PR as the canonical fix for #66406 and the closed duplicate #69979.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: `message(action="react", emoji="👍")` reaches `reaction_type.emoji_type` unchanged, while Feishu documents that the reactions API expects a valid `emoji_type` string and returns 231001 for invalid types.

## Solution Assessment

No, not as currently patched. Normalizing inside the Feishu plugin is the right maintainable boundary, but this branch should use documented Feishu values and avoid statically importing `reactions.ts` into the channel entry path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Use the documented Feishu emoji_type values:** `extensions/feishu/src/reactions.ts:163-190`
  - body: The normalizer maps several Unicode inputs to values Feishu does not document, such as `THUMBSDOWN`, `FIRE`, `CHECK`, and `CROSS`, while the current Feishu reference lists `ThumbsDown`, `Fire`, `CheckMark`, and `CrossMark`. It also uppercases valid mixed-case user inputs like `Fire` to `FIRE`, so those reactions can still fail with 231001. Please build the map and pass-through set from the exact documented vocabulary, or limit conversion to values verified by the API.
  - confidence: 0.94
- **[P2] Keep reaction normalization behind the lazy runtime boundary:** `extensions/feishu/src/channel.ts:63`
  - body: Importing `normalizeFeishuEmoji` from `./reactions.js` makes `channel.ts` statically load `reactions.ts`, which imports `client.ts`; that imports the Lark SDK and mutates its HTTP interceptor at module evaluation. This defeats the existing `channel.runtime.ts` lazy boundary for reaction API work. Since the PR also normalizes inside `addReactionFeishu` and `listReactionsFeishu`, remove the channel-level import or move only the pure normalizer into a runtime-free helper.
  - confidence: 0.91
