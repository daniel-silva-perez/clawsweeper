## Summary

Keep this PR open. Current main still forwards Feishu reaction emoji input unchanged into add and remove/list reaction API calls, and the linked Feishu reaction bug remains unresolved on main. This PR is still a plausible implementation candidate, but it needs maintainer review for the Feishu emoji_type vocabulary and the Feishu lazy runtime boundary before merge.

## Best Possible Solution

Keep this PR open for maintainer review and either land this repaired contributor branch or an equivalent Feishu-owned fix. The best implementation is a lightweight Feishu emoji normalizer that can be used without eagerly importing the Feishu runtime client, applied before addReactionFeishu and listReactionsFeishu, with tests for Unicode thumbs-up/heart/fire, Feishu type passthrough including mixed-case official values, trimming, unknown input behavior, add, and remove/list filtering.

## Risks / Open Questions

- Merging the PR as-is may weaken the Feishu lazy runtime boundary because channel.ts would statically import reactions.ts, which currently imports client.ts and the Lark SDK at module load.
- The PR's Unicode mapping should be checked against Feishu's current emoji_type vocabulary before landing; the official docs include mixed-case values for several reactions, while the PR maps some common Unicode inputs to uppercase names that may not match the current Feishu table.
