## Summary

Current main still does not support a job-level model or provider override for cron jobs that use sessionTarget="main" with payload.kind="systemEvent". The schema and TypeScript cron payload contract keep systemEvent payloads text-only, normalization prunes model/provider-like fields from system events, main-session cron execution only enqueues text and wakes heartbeat, and the CLI/UI/docs limit cron model selection to isolated agent-turn jobs. The related implementation PR #42708 was closed unmerged, so it is useful context but not a shipped fix.

## Best Possible Solution

Keep this issue open. The best implementation is an additive cron contract for per-job model/provider selection on main-session systemEvent jobs, with allowed-model validation and transient runtime plumbing through schema/types, normalization, main-session cron execution, heartbeat/system-event dispatch, CLI, Control UI, docs, and regression tests. PR #42708 is useful prior design context, but it was closed unmerged and is not shipped.

## Risks / Open Questions

- Closing this now would hide a still-current feature gap for users who want cheaper recurring main-session cron runs without editing internal session metadata.
- A future implementation needs the override to be job-scoped and transient so a cron run does not silently mutate the shared interactive main-session model selection.
- The change crosses protocol schema/types, normalization, cron runtime, heartbeat/system-event plumbing, CLI, Control UI, docs, and tests; a partial fix could make different surfaces disagree.
