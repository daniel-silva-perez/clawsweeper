## Summary

Keep this issue open. Current main still does not support a job-level model/provider override for `sessionTarget: "main"` cron jobs with `payload.kind: "systemEvent"`. The current contract accepts model selection only on `agentTurn` payloads, strips model-like fields from `systemEvent` payloads, and the main-session cron execution path has no model/provider handoff into the system-event queue or heartbeat wake. The related implementation PR #42708 is closed unmerged, so it is prior design context rather than a shipped fix.

## Best Possible Solution

Keep this issue open. The best path is an additive cron contract for per-job model/provider selection on `main + systemEvent` jobs, validated against the existing allowed-model behavior, then passed transiently through the main-session cron heartbeat/run path without mutating unrelated interactive session state. Use #42708 as prior implementation/design context, but do not treat it as shipped.

## Risks / Open Questions

- Closing now would hide a still-current feature gap for cheaper recurring main-session cron runs without editing internal session metadata.
- A future fix should avoid persisting a cron job's transient model/provider choice into shared main-session state where it could affect later interactive turns.
- The implementation is cross-surface: protocol schema, runtime types, normalization, main-session heartbeat/system-event plumbing, CLI, UI, docs, and regression tests need to move together.
