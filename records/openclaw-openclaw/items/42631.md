## Summary

Keep open. Current main still does not support a job-level model/provider override for cron jobs using `sessionTarget="main"` with `payload.kind="systemEvent"`; the schema, types, normalizer, runtime wake path, CLI, UI, and docs still keep cron model selection on the isolated `agentTurn` path. The related implementation PR #42708 is useful design context but was closed unmerged and had review feedback showing the override must be transient and allowlist-validated.

## What This Changes

Request to support job-level model/provider selection for main-session system-event cron jobs so recurring jobs can use a cheaper model without manual session-store edits.

## Best Possible Solution

Add an additive, job-scoped cron model contract for main-session system events, most likely `payload.model` using the existing provider/model ref format, validate it with the same allowed-model logic as isolated cron, and thread it transiently through cron execution into heartbeat processing for both `wakeMode="now"` and queued wake paths without mutating the shared session entry. Update schema/types, normalization and patch merge behavior, Gateway cron wiring, CLI, Control UI, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes. A deterministic source-level repro exists: submit or normalize a main-session systemEvent cron payload with `payload.model` and current main either rejects it at the protocol schema or prunes it back to text-only before runtime, and the main cron execution path has no model override plumbing.

## Solution Assessment

Unclear for a completed fix because there is no merged patch yet. The requested direction is sound if implemented as a transient, allowlist-validated per-job override; the closed PR #42708 is not the best final solution because review comments identified persistence, race, schema, and allowlist issues.

## Repair Work Prompt

Implement job-level model selection for cron jobs using `sessionTarget: "main"` and `payload.kind: "systemEvent"`. Prefer the existing cron `payload.model` provider/model ref pattern unless maintainers explicitly choose another public shape. The override must be transient for that cron-triggered heartbeat only: do not write `providerOverride`/`modelOverride` into the shared main-session store, and do not encode model data through normalized `contextKey`. Validate the requested model with the same allowlist/error behavior used for isolated cron `payload.model`. Preserve existing isolated cron semantics. Cover `wakeMode="now"`, `wakeMode="next-heartbeat"`, and busy fallback paths. Update protocol schema/types, normalization and patch merge behavior, cron service deps/state/timer, Gateway cron wiring, heartbeat wake/runner plumbing as needed, CLI add/edit, Control UI form/display, docs, changelog, and focused regression tests.
