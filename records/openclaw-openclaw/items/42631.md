## Summary

Current main still lacks job-level model/provider selection for cron jobs that use `sessionTarget: "main"` with `payload.kind: "systemEvent"`. The current protocol, runtime types, normalizer, main-session execution path, CLI/UI, and docs still scope cron model selection to `agentTurn`/isolated jobs, while the linked implementation PR #42708 was closed unmerged.

## Best Possible Solution

Keep this issue open. The best path is an additive cron contract for per-job model/provider selection on `main + systemEvent` jobs, validated with the existing allowed-model behavior, then passed transiently into the cron-triggered main-session heartbeat/run without mutating unrelated interactive session state. Use #42708 as prior design context, not as a landed fix.

## Risks / Open Questions

- Closing now would hide a still-current feature gap for cheaper recurring main-session cron runs without editing internal session metadata.
- A future implementation should avoid persisting a cron job's transient model choice into shared main-session state where it could leak into later interactive work.
- The fix is cross-surface: protocol schema, runtime types, normalization, main-session heartbeat/run plumbing, CLI, UI, docs, and regression tests need to move together.
