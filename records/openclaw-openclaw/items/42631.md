## Summary

Keep this issue open. Current main still does not support a job-level model/provider override for cron jobs using `sessionTarget: "main"` with `payload.kind: "systemEvent"`; model selection remains scoped to isolated `agentTurn` jobs, and the related implementation PR #42708 was closed unmerged.

## Best Possible Solution

Keep this issue open. The best path is an additive cron contract for per-job model/provider selection on `main + systemEvent` jobs, validated against the existing allowed-model behavior, then passed transiently into the cron-triggered main-session heartbeat/run without mutating unrelated interactive session state. Use #42708 as prior design context, not as a landed fix.

## Risks / Open Questions

- Closing now would hide a still-current feature gap for cheaper recurring main-session cron runs without editing internal session metadata.
- A future implementation should avoid persisting a cron job's transient model/provider choice into shared main-session state where it could affect later interactive work.
- The fix is cross-surface: protocol schema, runtime types, normalization, main-session heartbeat/run plumbing, CLI, UI, docs, and regression tests need to move together.
