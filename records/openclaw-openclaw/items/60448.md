## Summary

Keep this issue open. Current main still renders sender metadata into the model-facing prompt path: `chat.send` copies Gateway/TUI client identity into sender fields, Telegram copies real sender name/id/username into the same context fields, and `buildInboundUserContextPrefix` serializes those fields as `Sender (untrusted metadata)` JSON before `get-reply-run` prepends it to the prompt body. The existing stripping code is display/history hygiene, not a fix for AI-visible prompt context.

## Best Possible Solution

Keep this issue open and land a scoped prompt-context fix: preserve trusted internal routing/requester identity, but stop exposing internal UI client identity and sensitive channel sender identifiers in the model-facing user prompt unless a channel explicitly opts into a privacy-minimized label. #47216 is a useful TUI/WebChat implementation candidate, but the broader Telegram/privacy behavior should also get regression coverage.
