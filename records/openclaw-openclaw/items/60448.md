## Summary

Keep open. Current main still builds a model-facing `Sender (untrusted metadata)` block from sender fields and prepends it to the prompt body; the TUI/chat.send and Telegram paths still populate those fields with transport/client identity or real Telegram identity. The related open PR #47216 is useful for the direct webchat/TUI half, but it does not resolve the broader Telegram/privacy behavior described here.

## What This Changes

The issue requests stripping internal sender/routing metadata from the AI-facing prompt for TUI and Telegram messages while keeping routing identity internal to OpenClaw.

## Best Possible Solution

Land a scoped prompt-context change that separates internal routing/requester identity from model-visible context, suppresses transport-only client labels for direct webchat/TUI, and privacy-minimizes Telegram sender identifiers unless a channel explicitly opts into a safe prompt label. Regression coverage should include both direct webchat/TUI and Telegram direct messages.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: send through TUI or Telegram, observe sender fields populated in the channel/gateway context, then follow `buildInboundUserContextPrefix` into `get-reply-run` where that metadata is prepended to the prompt.

## Solution Assessment

No, current main is not the best solution because it still exposes sender metadata in the prompt. #47216 is a reasonable partial TUI/webchat fix, but the maintainable end state should address the shared prompt-context boundary and Telegram privacy path too.
