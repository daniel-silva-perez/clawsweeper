## Summary

Keep this PR open for maintainer follow-up. Current main still lacks heartbeat-specific fallback guidance, so the documentation gap is real, but the submitted patch is stale against the current `ParamField` docs layout and its strongest cross-provider warning is inaccurate because current runtime can still try the configured primary after the explicit heartbeat model fails. The broader structured fallback feature remains separately tracked by open issue #23718.

## Best Possible Solution

Request a narrow revision rather than closing: add an accurate fallback note inside the current `heartbeat.model` `ParamField`, align it with `docs/concepts/model-failover.md`, state that no override and same-provider overrides follow the applicable fallback chain, state that different-provider overrides skip unrelated configured fallbacks unless the override is already in the configured fallback chain, and mention that the configured primary can still be tried when no explicit fallback override is supplied. Keep #23718 separate as the broader feature request for structured heartbeat fallback configuration.
