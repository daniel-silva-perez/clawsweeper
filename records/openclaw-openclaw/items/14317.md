## Summary

Keep open. Current main still does not implement Slack `groupPolicy: "private"`: the shared config contract rejects it, Slack runtime routing has no private-channel branch, docs/metadata omit it, and the security audit has no classification for the proposed trust boundary.

## What This Changes

Request to add a Slack-specific `groupPolicy: "private"` mode that allows invited private channels, blocks public channels unless allowlisted, and updates the security audit accordingly.

## Best Possible Solution

After maintainer/security approval, implement a Slack-scoped private policy end to end: config types/schema/metadata, normal and slash-command authorization that carries Slack privacy metadata, public-channel allowlist fallback, exposure-audit classification, docs with Slack Connect caveats, and focused regression tests while preserving fail-closed behavior when Slack config is missing.

## Reproduction Assessment

Yes for the missing feature at source/config level. Current main rejects `groupPolicy: "private"` in the shared schema and has no private-channel branch in normal or slash-command Slack routing, so no live Slack runtime reproduction is required to verify the gap.

## Solution Assessment

Unclear until maintainers approve the trust model. If approved, the narrow maintainable solution is a Slack-specific `private` policy with explicit audit/docs coverage, not a generic enum addition that relies on existing policy fallthrough.
