## Summary

Keep open. Current main still lacks Slack `groupPolicy: "private"` end to end: the shared config contract rejects it, Slack docs and generated metadata omit it, and normal plus slash-command routing only evaluate `open`, `allowlist`, and `disabled`. The request is concrete, but it changes Slack authorization and audit semantics, so it should get maintainer/security approval before implementation.

## What This Changes

Add a Slack `groupPolicy: "private"` mode that allows invited private Slack channels while requiring explicit allowlist entries for public channels.

## Best Possible Solution

After approval, implement this as a Slack-scoped policy end to end: config types/schema/metadata, normal and slash route authorization that carries Slack privacy metadata, public-channel allowlist fallback, exposure-audit classification, docs with Slack Connect caveats, and focused regression tests while preserving current fail-closed behavior when Slack config is missing.

## Reproduction Assessment

Not applicable as a runtime bug, but the missing feature is high-confidence. Current main rejects `groupPolicy: "private"` at the shared type/schema layer and has no private-channel branch in Slack normal-message or slash-command routing.

## Solution Assessment

Unclear until maintainers approve the trust model. If approved, the narrow maintainable solution is a Slack-specific private policy with explicit audit/docs coverage, not a generic enum addition that falls through existing shared helpers.
