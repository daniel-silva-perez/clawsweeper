## Summary

Keep open. Current main still does not implement Slack `groupPolicy: "private"`: the shared config type and schema reject it, Slack docs and generated metadata omit it, and both normal message routing and slash-command routing still use the existing open/disabled/allowlist gate without Slack privacy metadata. The request is concrete, but it changes Slack authorization and exposure-audit semantics, so it needs maintainer/security approval before an automated repair PR changes the trust boundary.

## What This Changes

Request to add a Slack `groupPolicy: "private"` mode that allows private channels, ignores public channels unless allowlisted, and updates security-audit handling.

## Best Possible Solution

After maintainer/security approval, implement this as a Slack-scoped policy end to end: config types/schema/metadata, normal and slash route authorization that carries Slack privacy metadata, public-channel allowlist fallback, exposure-audit classification, Slack docs with Slack Connect caveats, and focused regression tests while preserving current fail-closed behavior when Slack config is missing.

## Reproduction Assessment

Not applicable as a runtime bug, but the missing feature is high-confidence: current main rejects `groupPolicy: "private"` at the schema/type layer and has no private-channel branch in the Slack routing paths.

## Solution Assessment

Unclear until maintainers approve the trust model. If approved, the narrow maintainable solution is a Slack-specific private policy with explicit audit/docs coverage, not a generic enum addition that falls through existing open-policy helpers.
