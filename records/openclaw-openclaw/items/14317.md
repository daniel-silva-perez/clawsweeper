## Summary

Keep open. Current main still lacks the requested Slack `groupPolicy: "private"` behavior end to end: the shared config contract rejects it, Slack docs/metadata omit it, normal and slash-command routing only implement the existing three policies, and the security audit only classifies `open`. The feature is concrete, but it changes Slack authorization and exposure-audit semantics, so it needs maintainer/security product approval before implementation.

## What This Changes

The issue requests a Slack-only `groupPolicy: "private"` mode that allows invited private Slack channels by default while requiring explicit allowlist entries for public channels.

## Best Possible Solution

After maintainer/security approval, implement a Slack-scoped private policy end to end: config types/schema/metadata, normal and slash-command authorization that carries Slack privacy metadata, public-channel allowlist fallback, exposure-audit classification, docs with Slack Connect caveats, and focused regression tests while preserving the current fail-closed default when Slack config is missing.

## Reproduction Assessment

Yes for the missing feature at source level. Current main rejects `groupPolicy: "private"` in the shared schema and has no private-channel branch in normal or slash-command Slack routing; no runtime bug reproduction is needed for this feature request.

## Solution Assessment

Unclear until maintainers approve the trust model. If approved, the narrow maintainable solution is a Slack-specific private policy with explicit audit/docs coverage, not a generic `GroupPolicy` enum addition that relies on existing fallthrough behavior.
