## Summary

Keep this issue open. Current main still does not implement Slack `groupPolicy: "private"`: the shared config type/schema and generated metadata only expose `open`, `disabled`, and `allowlist`; Slack message and slash routing still evaluate only those generic policies without using Slack `is_private`; docs and tests still describe the three-value policy model. The prior steipete Codex review comment from 2026-04-26 remains accurate against current main, with one update: the report's default-open motivation is now obsolete because current Slack config/runtime docs default/fallback to `allowlist`, but the requested private-channel policy is still missing.

## Best Possible Solution

Keep this issue open and implement a Slack-scoped `groupPolicy: "private"` option in the bundled Slack plugin. The fix should accept the value in Slack config types/schema/generated metadata, carry or fetch Slack `is_private` channel metadata for message and slash routing, allow private channels where the bot is present, preserve explicit public-channel allowlist and explicit deny behavior, update security audit/fix guidance so `private` is not treated like `open`, and document/test Slack Connect caveats. Do not change the Slack default from `allowlist` as part of the narrow fix unless a maintainer explicitly asks for that product decision.
