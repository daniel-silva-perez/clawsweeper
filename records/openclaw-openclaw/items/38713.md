## Summary

Keep this PR open. Current main still resolves `openclaw skills info <name>` with exact-only `name`/`skillKey` matching, so the intended fix is not implemented on main. The PR is focused and relevant, but its current head still has the unresolved case-insensitive ambiguity noted in review, so the best path is a small follow-up before landing rather than closure.

## Best Possible Solution

Keep this PR open. Update `resolveSkillByName` so exact matches still win, both case-insensitive and separator-normalized fallbacks require exactly one match, and the not-found sanitization remains. Keep or extend the focused regression tests in `src/cli/skills-cli.test.ts`, then run the targeted skills CLI test and the normal changed gate before landing. Track broader agent skill activation separately if it remains after the CLI lookup mismatch is fixed.
