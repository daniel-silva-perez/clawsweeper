## Summary

Keep open. Current main still lacks a session-owned runtime envelope in the Gateway session RPC contract, and the same reporter already has open PR #74799 as the active implementation candidate. The remaining work is authorization-boundary design and maintainer review, not cleanup closure or a duplicate automated repair job.

## What This Changes

Request adding an optional per-session runtime envelope to `sessions.create` and `sessions.patch`, with persisted tool/path/network policy enforcement for all tool calls in that session.

## Best Possible Solution

Keep this issue open while maintainers review #74799 or decide on a narrower protocol design. The durable end state should be an additive, documented Gateway session field with typed schemas, persisted session state, deterministic pre-execution enforcement that cannot be bypassed by later parameter rewrites, focused tests for create/patch/tool-call rejection, and a separate explicit plan for network or external callback enforcement if those cannot be implemented reliably in the same layer.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: adding `envelope` to a `sessions.create` or `sessions.patch` payload is rejected today because the TypeBox/AJV schemas omit the field and disallow additional properties. I did not run a live Gateway call because this review had to remain read-only.

## Solution Assessment

Unclear. A session-owned envelope is a plausible maintainable boundary for the reported gap, but the best implementation depends on maintainer review of the authorization layer and whether network/callback support belongs in the same change; the active same-author PR is the right candidate to evaluate.
