## Summary

Keep open. Current main still has no session-owned runtime envelope in the gateway session RPC contract: `sessions.create` and `sessions.patch` schemas omit the field and reject additional properties, while existing tool allow/deny and `before_tool_call` hooks are adjacent policy mechanisms rather than a declarative per-session gateway envelope. The linked same-author PR #74799 is the right active implementation candidate to review rather than closing this issue or queueing a duplicate repair job.

## What This Changes

Add an optional per-session runtime envelope to `sessions.create` and `sessions.patch`, persist it with the session, and enforce declared tool/path/network policy for every tool call in that session, with an optional external authorization callback follow-up.

## Best Possible Solution

Keep this issue open while maintainers review #74799 or replace it with an accepted narrower design. The durable end state should be an additive protocol field with typed schema/docs, persisted `SessionEntry` state, pre-execution enforcement that survives policy/hook/approval parameter rewrites, focused tests for create/patch and tool-call blocking, and a separate explicit decision for network enforcement and external callbacks if those cannot be enforced reliably in the same layer.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: add an `envelope` property to a `sessions.create` or `sessions.patch` RPC payload and the current TypeBox/AJV schema rejects it because the field is absent and additional properties are disabled. I did not run a live gateway test because this review was required to remain read-only.

## Solution Assessment

Unclear. A session-owned envelope is a plausible narrow product boundary for the reported gap, but the exact enforcement layer and the network/callback scope need maintainer review; #74799 is the active candidate rather than something to supersede with a new automated fix.
