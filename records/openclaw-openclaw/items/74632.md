## Summary

Keep open. Current main and the latest release still define `sessions.create` and `sessions.patch` without an `envelope`, store no session-owned envelope on `SessionEntry`, and rely on existing config/plugin hook policy surfaces instead. The open paired PR #74799 is the right implementation candidate to review rather than closing this issue early.

## What This Changes

Request additive Gateway session-envelope support on `sessions.create` and `sessions.patch`, persisted on the session entry and enforced for tool, path, network, and optional external-authorization policy during runs.

## Best Possible Solution

Add an explicit, additive session-envelope contract in the Gateway protocol and session store, then enforce the stored policy in the narrowest reliable layer before every tool side effect. Tool and path checks can compose with the existing before-tool-call policy path; network restrictions should only be exposed when backed by an enforceable sandbox/proxy boundary and documented protocol semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send `sessions.create` or `sessions.patch` with an `envelope` property against current main; both validators use fixed TypeBox objects with `additionalProperties: false`, and no storage field exists to preserve the value.

## Solution Assessment

Unclear. The core `sessions.create`/`sessions.patch` envelope direction is a plausible maintainable solution, but the network and external-callback parts need explicit security design; review the narrower paired PR #74799 rather than closing this issue now.
