## Summary

Keep open. Current main still lacks a first-class session-owned runtime envelope in the Gateway protocol and session store, and a same-author open PR (#74799) is already paired to this request, so the right cleanup action is maintainer review of that PR rather than closing this issue.

## What This Changes

The issue requests an additive Gateway protocol field for per-session runtime/tool/path/network policy on `sessions.create` and `sessions.patch`, with enforcement on subsequent tool calls.

## Best Possible Solution

Keep this issue open while reviewing PR #74799. A maintainable fix should add an additive typed Gateway protocol field, persist validated envelope state on the session record, enforce it in the tool-call authorization path with tests for create/patch and rewritten params, and defer network constraints until they can be backed by a real sandbox or proxy boundary.

## Reproduction Assessment

Yes. A static current-main check is enough: `sessions.create` and `sessions.patch` both use closed TypeBox schemas with no `envelope` field, and the handler/store paths have no envelope persistence or enforcement.

## Solution Assessment

Unclear for the paired PR until maintainer review completes. The issue direction is plausible for a core Gateway/session contract, but the safest design is a narrow additive envelope for enforceable tool/path/command constraints plus a separate security design for network policy.
