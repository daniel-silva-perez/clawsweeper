## Summary

Keep open. Current main still lacks the proposed smartExtraction path, so this PR is not obsolete, but the patch is not merge-ready because the new LLM memory path is not wired through the manifest, bypasses existing capture safety gates, treats stored memories as trusted prompt text, and hides extractor failures.

## What This Changes

This PR adds opt-in smartExtraction config, a new memory-lancedb smart-extractor module with parser tests, and an auto-capture branch that calls a chat model to decompose conversations into atomic memories.

## Best Possible Solution

Keep this PR open for maintainer-directed rework in the memory-lancedb plugin: align the manifest, UI hints, docs, and changelog with the new config; run smart extraction only after the existing capture safety gate; treat stored memories as untrusted prompt data; make extractor failures observable or fall back safely; and add targeted config, parser, hook, and safety tests.

## Reproduction Assessment

Yes. The manifest/config failure is reproducible by enabling smartExtraction in memory-lancedb config: the manifest schema has additionalProperties:false and lacks that property, and the loader validates the manifest schema before runtime parsing. The safety bypass is also visible by inspection: the PR calls extractMemories with raw texts before shouldCapture is applied.

## Solution Assessment

No. The single-shot extraction design may be a viable opt-in feature, but this implementation is not the best merge path until it preserves the current safety/trust contracts and wires the public config surface through manifest validation and docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Sync smartExtraction with the manifest schema:** `extensions/memory-lancedb/config.ts:139`
  - body: The runtime parser now accepts `smartExtraction`, but the plugin manifest still has `additionalProperties: false` without that property. Since the loader validates plugin config against the manifest before runtime config parsing, users who enable this option will get an invalid-config load failure before the feature can run.
  - confidence: 0.94
- **[P1] Gate smart extraction through shouldCapture:** `extensions/memory-lancedb/index.ts:633-637`
  - body: The smart branch sends every collected user text to `extractMemories` before applying `shouldCapture`, while the existing filter is only used in the fallback branch below. That bypasses `captureMaxChars` and the prompt-injection/content filters that currently prevent unsafe or oversized messages from being captured.
  - confidence: 0.96
- **[P1] Escape existing memories before prompting the extractor:** `extensions/memory-lancedb/smart-extractor.ts:109-110`
  - body: `existingMemories` are persisted user-controlled strings, but the extractor prompt inserts them verbatim as bullet text. A stored memory containing instruction-like content can steer the extraction or dedup result, unlike the recall path that escapes memory text and marks it as untrusted.
  - confidence: 0.9
- **[P2] Make extraction failures observable:** `extensions/memory-lancedb/smart-extractor.ts:202-203`
  - body: If the chat completion fails because of auth, model, base URL, or provider outage, this catch returns an empty result and the caller treats it as a normal no-op. That silently disables smart auto-capture and gives operators no warning or fallback path.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a new LLM chat-completion path for user conversation text and stored memories, and the current patch bypasses established capture and prompt-trust safeguards.

Concerns:

- **[high] Capture safety bypass before outbound LLM call:** `extensions/memory-lancedb/index.ts:633`
  - body: The smart extraction branch calls the chat extractor with raw collected user text before applying shouldCapture or captureMaxChars, so content the current plugin rejects can be sent to the extraction provider and possibly stored.
  - confidence: 0.96

  - body: Existing memories are user-controlled persisted data but are interpolated into the extractor prompt without escaping or an untrusted-data boundary, creating a prompt-injection route specific to smart extraction.
