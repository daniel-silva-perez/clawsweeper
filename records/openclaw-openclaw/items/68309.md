## Summary

Keep this issue open. Current main still routes any truthy `choice.delta.content` straight into the OpenAI-completions text append path, so an array-shaped Mistral thinking/text delta can still be coerced into `[object Object]`; the exact fix is represented by open PR #68418, which uses closing syntax and has not merged.

## What This Changes

The issue reports that `mistral/mistral-small-latest` with thinking enabled renders repeated `[object Object]` text before the normal assistant output.

## Best Possible Solution

Land #68418 or an equivalent parser fix that treats array-shaped OpenAI-completions `delta.content` as ordered text/thinking blocks, preserves the existing string fast path, and adds regression coverage for Mistral typed-block streams before closing this issue.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an OpenAI-completions stream chunk whose `choice.delta.content` is an array of `{type:"thinking"}` and `{type:"text"}` blocks; current main passes that non-string value into the text append path, which can produce `[object Object]` output.

## Solution Assessment

Yes, the linked fix direction appears to be the narrow maintainable solution: fix the shared OpenAI-completions stream parser generically instead of hiding thinking in the UI or adding a Mistral-only workaround. The open PR still needs normal maintainer review/landing before this issue should close.
