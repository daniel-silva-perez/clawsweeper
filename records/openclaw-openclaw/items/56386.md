## Summary

Keep open. The PR targets a real remaining gap in current main, but the patch should not merge as-is because the new awaited ACP child cancellation can hang stop/reset when an ACP backend does not resolve cancellation.

## What This Changes

The PR makes subagent stop propagation async, awaits stop/reset callers, cancels ACP-backed child sessions, persists child aborted state, and adds fast-abort regression coverage.

## Best Possible Solution

Update the PR to keep the ACP child-stop cascade, but wrap each child `cancelSession` in the same bounded cleanup pattern used by gateway ACP cleanup, then rebase onto current main and refresh the regression tests for active ACP child cancellation and hung-cancel fallback.

## Reproduction Assessment

Yes for the blocking review finding: the code path is directly traceable from the PR's new awaited child cancel to `AcpSessionManager.cancelSession`, which awaits backend cancellation without its own timeout. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No, not as proposed. Propagating cancellation to active ACP-backed subagents is the right direction, but the maintainable fix needs bounded cancellation and a rebase against current main before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Bound ACP child cancellation during stop cascade:** `src/auto-reply/reply/abort.ts:204-209`
  - body: The new child-stop loop awaits `acpManager.cancelSession(...)` for each active child. `AcpSessionManager.cancelSession` waits on the backend `runtime.cancel` promise without its own timeout, and session reset now awaits this cascade before returning, so one stuck ACP backend can hang stop/reset. Wrap this child cancel in the same bounded cleanup pattern used for gateway ACP cleanup and continue/log on timeout.
  - confidence: 0.88
