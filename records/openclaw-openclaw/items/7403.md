## Summary

Keep open. Current main still lacks an enforceable per-session Private/Secret/Incognito mode: the Gateway protocol, session config, persisted session entry, Control UI session row model, and transcript write paths have no no-persist/private field, while the issue discussion asks for a program-level no-read/no-write/no-history guarantee. The related #12242 was closed as a duplicate of this tracker, and #52478 was closed unmerged after only proposing partial validation scaffolding.

## What This Changes

Request a first-class Private Mode for demos and shared/public use that suppresses workspace and memory context, prevents durable session history and workspace writes, and exposes UI/channel controls for that mode.

## Best Possible Solution

Keep this as the canonical Private/Secret/Incognito mode tracker and design an additive private/ephemeral session contract across Gateway protocol, SessionEntry/store, transcript/history APIs, bootstrap and memory, tool/media persistence, channel triggers, and Control UI, with regression tests proving private sessions neither read private workspace/memory context nor write durable history.

## Reproduction Assessment

Not applicable as a feature/privacy request rather than a runtime bug. The source inspection gives a high-confidence verification path: protocol schemas, session config/types, session creation/transcript helpers, bootstrap context loading, and Control UI session models all lack the requested private/no-persist contract.

## Solution Assessment

No proposed patch is currently the best complete solution. The requested direction is valid, but a safe fix needs a design-approved cross-surface privacy contract rather than a UI-only toggle or config-only validation phase.
