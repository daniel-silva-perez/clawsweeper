## Summary

Keep open. Current main still lacks an enforceable per-session Private/Secret/Incognito mode: the protocol, session config, persisted session entry, transcript write paths, bootstrap/memory context, and Control UI session model do not carry a private/no-persist contract, while the discussion asks for a program-level no-read/no-write/no-history guarantee.

## What This Changes

Add a Private Mode for demos, content creation, and shared/private sessions that suppresses workspace context, avoids durable session/history/memory writes, uses temporary scratch state, and exposes an active-state toggle/indicator in Mission Control.

## Best Possible Solution

Keep this as the canonical Private/Secret/Incognito mode tracker and design an additive private/ephemeral session contract across Gateway protocol, SessionEntry/store, transcript/history APIs, bootstrap and memory, tool/media persistence, channel triggers, and Control UI, with regression tests proving private sessions neither read private workspace/memory context nor write durable history.

## Reproduction Assessment

Not applicable as a feature/privacy request rather than a runtime bug. The absence check is high-confidence: protocol schemas, session config/types, session creation/transcript helpers, bootstrap context loading, and Control UI session models all lack the requested private/no-persist contract.

## Solution Assessment

No proposed patch is currently the best complete solution. The requested direction is valid, but a safe fix needs a maintainer-approved cross-surface privacy contract rather than a UI-only toggle or the closed config-validation-only scaffolding.
