## Summary

Keep open. Current main still lacks the requested enforceable Private/Secret/Incognito session contract: there is no private/no-persist field in the Gateway session protocol, session config, persisted session entry, transcript creation path, memory/bootstrap flow, or Control UI session model. The discussion also broadens the request into a program-level privacy guarantee, so this needs maintainer design review rather than cleanup closure.

## What This Changes

Add a Mission Control Private Mode that suppresses workspace context, prevents durable session and memory persistence, and visibly marks private sessions in the UI.

## Best Possible Solution

Keep this as the canonical Private/Secret/Incognito mode tracker and design an additive private/ephemeral session contract across Gateway protocol, SessionEntry/store, transcript/history APIs, bootstrap and memory, tool/media persistence, channel triggers, and Control UI, with regression tests proving private sessions neither read private workspace/memory context nor write durable history.

## Reproduction Assessment

Not applicable as a feature/privacy request rather than a runtime bug. The absence check is high-confidence: current schemas, session store types, transcript creation tests, bootstrap loading, memory flush code, docs, and UI session rows all lack the requested private/no-persist contract.

## Solution Assessment

No current patch is the best complete solution. The requested direction is valid, but the maintainable fix needs a maintainer-approved cross-surface privacy contract rather than a UI-only toggle or the closed config-validation-only scaffolding.

## Security Review

Status: needs_attention

Summary: This privacy-sensitive feature needs maintainer threat-model review before OpenClaw advertises a no-read/no-write/no-history guarantee.

Concerns:

- **[medium] Define an enforceable private session boundary:** `src/gateway/protocol/schema/sessions.ts:85`
  - body: The session protocol and persisted session entry do not carry a private/ephemeral bit, so downstream transcript, memory, media, and tool paths have nothing to enforce. A private-mode implementation needs explicit security semantics before exposing the guarantee.
  - confidence: 0.9
- **[medium] Block durable transcript writes before UI claims:** `src/gateway/server-methods/sessions.ts:944`
  - body: sessions.create currently creates a JSONL transcript and persists its path. Any Private Mode UI or channel trigger must block this and related append paths, or sensitive conversation data can still be written to disk.
  - confidence: 0.88
- **[medium] Suppress workspace and memory reads through runtime enforcement:** `src/agents/bootstrap-files.ts:241`
  - body: Existing contextInjection controls can skip bootstrap context globally, but they are not a per-session privacy boundary and do not enforce memory/tool/media read and write limits. Private mode needs runtime propagation to those surfaces.
  - confidence: 0.84
