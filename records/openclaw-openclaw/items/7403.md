## Summary

Keep open. Current main still lacks the enforceable per-session Private/Secret/Incognito contract requested here: there is no private/no-persist bit in the Gateway session protocol, session config, persisted session entry, transcript creation path, or Control UI session row model, while existing context-injection controls only mitigate bootstrap prompt injection globally and do not guarantee no reads, writes, or durable history.

## What This Changes

Request a Mission Control Private Mode toggle that suppresses workspace/persona/memory context and runs a temporary no-read/no-write/no-history session for demos, content creation, and public use.

## Best Possible Solution

Keep this as the canonical Private/Secret/Incognito mode tracker and design an additive private/ephemeral session contract across Gateway protocol, SessionEntry/store, transcript/history APIs, bootstrap and memory, tool/media persistence, channel triggers, and Control UI, with regression tests proving private sessions neither read private workspace/memory context nor write durable history.

## Reproduction Assessment

Not applicable as a feature/privacy request rather than a runtime bug. The absence check is high-confidence: protocol schemas, session config/types, session creation/transcript helpers, bootstrap context loading, and Control UI session models all lack the requested private/no-persist contract.

## Solution Assessment

No current patch is the best complete solution. The requested direction is valid, but the maintainable fix needs a maintainer-approved cross-surface privacy contract rather than a UI-only toggle or the closed config-validation-only scaffolding.

## Security Review

Status: needs_attention

Summary: This privacy-sensitive feature needs maintainer threat-model review before OpenClaw advertises a no-read/no-write/no-history guarantee.

Concerns:

- **[medium] Define an enforceable private session boundary:** `src/gateway/protocol/schema/sessions.ts:85`
  - body: The session protocol and persisted session entry do not carry a private/ephemeral bit, while downstream transcript, memory, media, and tool paths have nothing to enforce. A private-mode implementation needs explicit security semantics before exposing the guarantee.
  - confidence: 0.9
- **[medium] Block durable transcript writes before shipping UI claims:** `src/gateway/server-methods/sessions.ts:944`
  - body: sessions.create currently ensures a JSONL transcript file and persists its path. Any Private Mode UI or channel trigger must block this and related append paths, or sensitive conversation data can still be written to disk.
  - confidence: 0.88
- **[medium] Suppress workspace and memory reads through runtime enforcement:** `src/agents/bootstrap-files.ts:229`
  - body: The existing contextInjection control can skip bootstrap context globally, but it is not a per-session privacy boundary and does not enforce memory/tool/media read and write limits. Private mode needs runtime propagation to those surfaces.
  - confidence: 0.84
