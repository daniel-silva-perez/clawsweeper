## Summary

Keep open. Current main still has the risky display path described here: chat `/status` builds context usage from `entry.totalTokens` without checking `totalTokensFresh`, and when transcript fallback is enabled it promotes a larger transcript-derived prompt total over the stored value. Current compaction/session accounting can also preserve an older total as stale when no provider usage or `tokensAfter` is available, so an already-compacted session can still have a stale larger total available for status rendering.

## Best Possible Solution

Keep this issue open and fix the status rendering contract: chat `/status` should treat `totalTokensFresh: false` as unknown unless a trustworthy post-compaction source is available, prefer valid post-compaction `tokensAfter`/checkpoint data over stale pre-compaction totals, and avoid using a larger whole-transcript estimate as current context after compaction. Add a focused regression test that creates a compacted session with stale pre-compaction `totalTokens` plus a larger transcript estimate and verifies `/status` does not render values like `151k/128k` as current usage.
