## Summary

Keep open. Current main has useful partial mitigations, but the requested public session write-lock timeout/stale configuration is still absent, and the embedded attempt still holds the lock through prompt execution until end-of-attempt cleanup. Recent discussion also includes a fresh v2026.4.26 operator reproduction and an overlapping open PR for only part of the timeout/error-handling surface, so this is not cleanup-closeable.

## What This Changes

The issue requests configurable session write-lock timeout/stale settings, a narrower lock lifecycle or single-writer queue, and lock-timeout handling that does not surface as model failover.

## Best Possible Solution

Add a supported session write-lock configuration contract with schema, exported types, generated help/metadata, and docs; wire it consistently into embedded attempts, transcript rewrite/compaction, session-store mutation, and startup/doctor lock paths where applicable; preserve the typed lock-timeout failover suppression; and split any single-writer or prompt-lifecycle lock redesign into a clearly scoped follow-up if maintainers want to ship the operator-unblocking config first.
