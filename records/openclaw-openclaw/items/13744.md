## Summary

Keep open. Current main has partial mitigation for session-lock contention and failover masking, but it still lacks a supported timeout/stale configuration surface and the embedded attempt still holds the session write lock through prompt execution. The latest discussion also includes a fresh v2026.4.26 operator report of the same 10s timeout, and open PR #69159 overlaps only part of the remaining work.

## What This Changes

Request configurable session write-lock timeout/stale settings, a narrower embedded session-lock lifecycle, and lock-timeout handling that does not surface as a model/provider failure.

## Best Possible Solution

Add a supported session write-lock configuration contract with schema, exported types, generated help/metadata, and public docs; wire it into embedded attempts and transcript/session-store mutation paths; retain the typed lock-timeout failover suppression; and make an explicit maintainer call on whether lock-scope narrowing is safe now or should be staged as a deeper single-writer refactor.

## Reproduction Assessment

Yes. The issue discussion provides a fresh v2026.4.26 operator reproduction, and current-main source inspection shows the central embedded path still uses the 10s acquire default and keeps the lock through `activeSession.prompt(...)`; this read-only review did not run a live gateway load test.

## Solution Assessment

Unclear. The requested direction is valid and a config-only first stage looks maintainable, but the exact public config/env contract and the depth of lock lifecycle changes need maintainer design before an automated replacement fix is appropriate.
