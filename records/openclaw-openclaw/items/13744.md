## Summary

Keep open. Current main has partial mitigations for lock contention and failover masking, but it still lacks a public timeout/stale configuration surface and the embedded attempt still holds the session write lock through prompt execution and cleanup; the latest discussion also includes a fresh v2026.4.26 operator report of the same 10s timeout.

## What This Changes

Request to make session write-lock timeout and stale-lock settings configurable, narrow the embedded session lock lifetime, and surface lock contention separately from model/provider failover.

## Best Possible Solution

Add a supported session write-lock configuration contract with schema, exported types, generated help/metadata, and public docs; wire it consistently into embedded attempts and session-store/transcript mutation paths; keep the typed lock-timeout failover suppression; and stage any deeper single-writer or lock-lifecycle redesign as an explicitly scoped follow-up.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The discussion includes a fresh v2026.4.26 operator reproduction, and current-main source inspection confirms the missing config surface plus the embedded prompt still running under the same session lock; I did not run live tests because this review is read-only.

## Solution Assessment

Is this the best way to solve the issue? Unclear for implementation details. The requested direction is valid, but the maintainable fix needs a maintainer decision on the public config/env contract and whether lock-scope narrowing should be incremental or a larger single-writer refactor.
