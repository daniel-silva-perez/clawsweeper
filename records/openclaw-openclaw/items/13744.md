## Summary

Keep open. Current main has useful partial mitigations for session-lock cleanup, failover classification, and delayed lock acquisition, but it still lacks a public session write-lock timeout/stale configuration surface and the embedded runner still holds the session lock through prompt execution and end-of-attempt cleanup. The recent v2026.4.26 operator report plus overlapping open PR #69159 make this a maintainer staging item rather than conservative cleanup-close material.

## What This Changes

The issue requests configurable session write-lock timeout/stale settings, narrower lock scope, and clearer non-model handling for session lock timeout failures.

## Best Possible Solution

Add a supported session write-lock configuration contract, wire it through embedded attempts plus session-store and transcript mutation paths, preserve the typed lock-timeout failover suppression, and split any deeper single-writer or prompt-lifecycle lock redesign into an explicit follow-up if maintainers want to ship the operator-unblocking config first.
