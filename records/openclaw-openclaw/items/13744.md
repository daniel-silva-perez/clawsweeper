## Summary

Keep open. Current main still only partially addresses the report: session write-lock timeout errors are typed and excluded from model failover, and acquisition was moved after some cold setup, but there is still no supported timeout/stale configuration surface and the embedded attempt still holds the transcript lock through prompt execution and cleanup.

## What This Changes

The issue requests configurable session write-lock timeout/stale settings, a narrower embedded-run lock scope, and lock-contention handling that does not surface as model/provider failure.

## Best Possible Solution

Add a supported session write-lock configuration contract with schema, exported types, config help/metadata, and public docs; wire it into embedded attempts and session-store/transcript mutation paths; keep the typed lock-timeout failover suppression; and make an explicit maintainer decision on whether deeper lock-scope narrowing lands now or as a staged single-writer refactor.

## Reproduction Assessment

Yes. The discussion provides a fresh v2026.4.26 operator reproduction, and current-main source inspection shows the embedded path still uses the 10s acquire default while holding the lock through prompt execution; this read-only review did not run a live gateway load test.

## Solution Assessment

Unclear. The requested direction is valid and a config-first stage looks useful, but the exact public config/env contract and safe lock-lifecycle boundary need maintainer design before an automated replacement fix is appropriate.
