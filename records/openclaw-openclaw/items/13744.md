## Summary

Keep open. Current main has partial mitigation for lock-timeout failover masking, but it still lacks a public timeout/stale configuration surface and the embedded attempt still keeps the session write lock across prompt execution; the discussion also includes a fresh v2026.4.26 operator report of the same 10s timeout.

## What This Changes

The issue requests configurable session write-lock timeout/stale settings, a narrower session-lock lifetime, and clearer handling of lock timeout errors.

## Best Possible Solution

Add a supported session write-lock configuration contract with schema, exported types, generated help/metadata, and public docs; wire it into embedded attempts and session/transcript mutation paths; keep the typed lock-timeout failover suppression; and stage any deeper single-writer or lock-lifecycle redesign as an explicit follow-up if it cannot be made safely in the same change.

## Reproduction Assessment

Yes. The issue discussion provides a fresh v2026.4.26 operator reproduction, and current-main source inspection shows the embedded lock still defaults to 10s acquisition and stays held through `activeSession.prompt(...)`; I did not run live tests because this review is read-only.

## Solution Assessment

Unclear for implementation details. The requested direction is valid, with config likely the safest first stage, but the exact public config/env contract and the depth of lock-scope narrowing need maintainer design before an automated fix PR is appropriate.
