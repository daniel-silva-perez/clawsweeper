## Summary

Keep open. Current main has partial mitigations for session-lock contention and failover masking, but it still lacks a public session write-lock timeout/stale configuration surface and the embedded runner still holds the session lock through the model prompt lifecycle; the latest discussion also reports the same 10s timeout on v2026.4.26.

## What This Changes

The issue requests configurable session write-lock timeout/stale settings, narrower lock lifetime around transcript mutation, and lock-timeout handling that does not surface as model/provider failover.

## Best Possible Solution

Add a supported session write-lock configuration contract, wire it consistently into embedded attempts and session-store/transcript mutation paths, keep the typed lock-timeout failover suppression, and split the deeper lock-scope or single-writer redesign into an explicit staged follow-up if needed.
