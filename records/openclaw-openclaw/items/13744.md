## Summary

Keep open. Current main has partial mitigations for session-lock contention and failover masking, but it still lacks a public timeout/stale config or env surface and the embedded attempt still holds the session write lock through prompt execution and final cleanup. The latest issue discussion also contains a fresh v2026.4.26 operator reproduction, so this is still live work rather than cleanup-closeable.

## What This Changes

The issue requests configurable session write-lock timeout/stale settings via config or env, narrower lock scope or single-writer serialization, and lock-timeout handling that does not surface as model failover.

## Best Possible Solution

Add a supported session write-lock configuration contract with schema, exported types, generated help/metadata, and docs; wire it consistently into embedded attempts and other transcript/session-store mutation paths; keep the typed lock-timeout failover suppression; and split any deeper single-writer or prompt-lifecycle refactor into a clearly scoped follow-up if needed.
