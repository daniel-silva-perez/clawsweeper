## Summary

Keep open. Current main still has the core requested gaps: session write-lock `timeoutMs`/`staleMs` are parameter-only fallbacks rather than public config/env settings, and the embedded attempt still holds the session lock through prompt/model execution and cleanup. The error-classification part is partially fixed with `SessionWriteLockTimeoutError`, so lock timeouts are no longer treated as model-timeout failover.

## Best Possible Solution

Keep this issue open until OpenClaw adds a supported session write-lock timeout/stale configuration surface, wires it into embedded run and transcript mutation paths, and either narrows the lifecycle lock to actual transcript critical sections or explicitly documents why the broader lock is intentional. Preserve the current `SessionWriteLockTimeoutError` failover suppression and add focused concurrency/config regression coverage.
