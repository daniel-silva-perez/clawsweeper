## Summary

Keep open. Current main still has only partial mitigations: session write-lock timeout errors are now typed and suppressed from model failover, and lock acquisition moved after some cold startup work, but there is still no supported timeout/stale config surface and the embedded run still holds the session lock through prompt execution and cleanup.

## What This Changes

Request to make session write-lock timeout/stale settings configurable, narrow the lock lifecycle, and surface lock contention separately from model/provider failure.

## Best Possible Solution

Add a supported session write-lock configuration contract with schema, exported types, config help/metadata, and public docs; wire it into embedded attempts and transcript/session-store mutation paths; preserve the typed lock-timeout failover suppression; and make an explicit design call on whether deeper lock-scope narrowing should land now or as a staged single-writer refactor.

## Reproduction Assessment

Yes. The discussion provides a fresh v2026.4.26 operator reproduction, and current-main source inspection shows the embedded path still uses the 10s acquire default while holding the lock through `activeSession.prompt(...)`; this read-only review did not run a live gateway load test.

## Solution Assessment

Unclear. The requested direction is valid and a config-only first stage looks useful, but the exact public config/env contract and the safe lock-lifecycle boundary require maintainer design before an automated replacement fix is appropriate.
