## Summary

Keep open. Current main has partial mitigations for lock cleanup/scope and failover classification, but the requested public timeout/stale configuration is still absent and the embedded runner can still hold the session lock through the prompt lifecycle; the fresh v2026.4.26 operator report and overlapping open PR make this a maintainer staging item, not cleanup-close material.

## What This Changes

The issue requests configurable session write-lock timeout/stale settings, narrower lock holding around transcript mutations, and lock-timeout handling that does not surface as model/provider failure.

## Best Possible Solution

Add a supported session write-lock configuration contract, wire it into embedded attempts and session-store/transcript mutation paths, preserve typed lock-timeout failover suppression, and split any deeper single-writer or lock-lifecycle redesign into an explicit follow-up if maintainers want to stage the operator-unblocking config first.
