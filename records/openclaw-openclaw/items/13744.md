## Summary

Keep open. Current main has partially improved this area by adding a typed session write-lock timeout error and moving lock acquisition later in the embedded runner, but it still does not expose `timeoutMs`/`staleMs` through public config or env, and the embedded run still holds the lock while `AgentSession.prompt()` runs the model/tool loop before cleanup releases the lock.

## Best Possible Solution

Keep this issue open until OpenClaw adds a supported session write-lock timeout/stale configuration surface, threads it through embedded run, compaction, transcript rewrite/truncation, and session-store mutation paths as appropriate, and either narrows the lifecycle lock to real transcript critical sections or introduces a clearly documented per-session single-writer queue. Preserve the current `SessionWriteLockTimeoutError` failover suppression and add focused concurrency/config regression coverage.
