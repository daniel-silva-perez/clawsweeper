## Summary

Keep open. Current main still lacks this focused Windows no-Docker/WSL hardening fallback, so the PR is not obsolete, but the patch should not merge as-is because its verification step can pass without proving the ACL hardening the guide asks users to apply.

## What This Changes

This PR adds a native Windows host-hardening fallback docs page, registers it in the docs navigation, and cross-links it from the Windows and sandboxing docs.

## Best Possible Solution

Keep a Windows host-hardening fallback only if maintainers want to document this weaker non-container path, then rebase it onto the current Windows docs, preserve WSL2/Docker as the primary recommendation, and change verification to test an explicitly denied sibling path or reader-supplied blocked path.

## Reproduction Assessment

Yes for the docs defect. Inspect the PR head: step 4 denies sibling directories, but the forbidden-path test at line 158 checks `$env:USERPROFILE`, so it can succeed without validating the sibling-deny ACLs the guide introduced.

## Solution Assessment

No, not as-is. The docs direction is narrow and plausible, but the verification snippet needs to target the actual denied path and the branch needs to be reconciled with current Windows documentation before this is the best merge path.

## Review Findings

Overall correctness: patch is incorrect
