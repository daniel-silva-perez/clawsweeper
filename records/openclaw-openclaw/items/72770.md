## Summary

Keep open. The PR is a plausible gateway feature but is not merge-ready: it exposes token-minting RPCs while the returned token is not wired into the gateway auth path, the JSONL persistence path is only initialized in tests, and the automatic-pruning claim remains unsupported by the diff.

## What This Changes

This PR adds gateway protocol schemas, WRITE-scoped gateway methods, handler registration, and an in-memory/JSONL-backed store plus tests for time-limited session share tokens.

## Best Possible Solution

Land this as a complete gateway feature: token creation/revocation, startup persistence and compaction, connect-auth resolution to read-only session-bound scopes, explicit authorization rules for share-token issuance, protocol docs, and integration tests proving restart, expiry, revocation, and token-scoped read access.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the PR returns a token from sessions.createSharedToken, but neither current main nor the PR diff wires resolveSessionShareToken into the websocket connect/auth path, and the persistence initializer is only exercised by tests.

## Solution Assessment

No. The current PR is not the best way to solve the issue because it publishes public gateway methods before the token can authenticate and before production persistence is initialized; the safer path is to merge the store, auth middleware, startup hook, docs, and integration tests together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Wire shared tokens into auth before exposing them:** `src/gateway/server-methods/session-share-tokens.ts:24-29`
  - body: `sessions.createSharedToken` returns a bearer token, but the diff never teaches the websocket connect/auth path to resolve that token into read-only, session-bound scopes. Callers can mint a token that cannot actually share a session, so either wire the auth middleware in this PR or keep the method unregistered until that follow-up exists.
  - confidence: 0.9
- **[P2] Initialize token persistence during gateway startup:** `src/gateway/server-methods.ts:99`
  - body: Registering the new handlers is not enough to enable the advertised JSONL persistence: `persistencePath` is set only by `initSessionShareTokens`, and the PR only calls that initializer from tests. In a real gateway process token creation will stay memory-only and restart loading will never happen unless startup initializes the store.
  - confidence: 0.88
- **[P2] Compact expired tokens automatically:** `src/infra/session-share-tokens.ts:132`
  - body: Every token creation appends a JSONL record, while expired entries are compacted only on explicit revoke or a manual `pruneExpiredTokens()` call. Since the PR does not schedule pruning or compact during create/resolve/startup, the file can retain every token ever minted despite the PR description promising automatic pruning.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces bearer session-share tokens and public mint/revoke RPCs, so the auth and session-authorization boundary needs attention before merge.

Concerns:

- **[medium] Token issuance is not bound to an authorized session:** `src/gateway/server-methods/session-share-tokens.ts:24`
  - body: `sessions.createSharedToken` accepts a caller-supplied `sessionKey` and mints a bearer token without proving the caller may delegate read access to that session. Before the auth middleware follow-up lands, the PR should define and enforce the session access rule at issuance time.
  - confidence: 0.78
- **[medium] Returned tokens are not enforced by auth middleware:** `src/gateway/server-methods/session-share-tokens.ts:24`
  - body: The patch adds token creation and storage but does not integrate token resolution with the gateway connect path, leaving the new bearer token semantics unvalidated and incomplete at the security boundary.
  - confidence: 0.86
