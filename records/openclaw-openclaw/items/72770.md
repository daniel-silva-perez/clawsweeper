## Summary

Keep open. The PR is a partial gateway security feature and should not merge until the bearer-token auth path, startup persistence, compaction, and session authorization rules are completed together.

## What This Changes

This PR adds gateway protocol schemas, WRITE-scoped RPC registration, token handlers, and an in-memory/JSONL-backed store plus tests for time-limited session share tokens.

## Best Possible Solution

Land session sharing as a complete gateway feature: mint/revoke APIs, startup persistence initialization, automatic expiry compaction, connect-auth resolution into read-only session-bound scopes, explicit session-authorization checks for issuance, protocol/docs updates, and integration tests covering restart, expiry, revocation, and token-scoped read access.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the PR returns a bearer token from sessions.createSharedToken, but the connect/auth path inspected on current main and in the PR diff never resolves that token, and the persistence initializer is not wired into gateway startup.

## Solution Assessment

No. The current PR is not the best way to solve the issue because it exposes public gateway methods before the tokens can authenticate or persist as advertised; the safer path is to merge the auth, startup, compaction, authorization, docs, and tests together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Wire shared tokens into auth before exposing them:** `src/gateway/server-methods/session-share-tokens.ts:27-37`
  - body: sessions.createSharedToken returns a bearer token, but the connect/auth path still only handles the configured shared secret, bootstrap tokens, and device tokens. Callers can mint a token that cannot actually authenticate or share a session, so wire this into WebSocket auth or keep the RPC unregistered until the feature is complete.
  - confidence: 0.91
- **[P2] Initialize token persistence during gateway startup:** `src/gateway/server-methods.ts:102`
  - body: Registering the handlers does not call initSessionShareTokens, so persistencePath stays unset in a real gateway process and the advertised JSONL restart loading never happens. Add a startup initializer before exposing the methods.
  - confidence: 0.88
- **[P2] Compact expired tokens automatically:** `src/infra/session-share-tokens.ts:138-139`
  - body: Token creation appends every entry to the JSONL file, while expired entries are rewritten only on explicit revoke or a manual pruneExpiredTokens call. Without a scheduler or create/resolve-time compaction trigger, the file can retain every token ever minted despite the PR claiming automatic pruning.
  - confidence: 0.84
- **[P2] Require session authorization before minting tokens:** `src/gateway/server-methods/session-share-tokens.ts:27-30`
  - body: The create handler accepts an arbitrary sessionKey from any WRITE-scope caller and stores a bearer token for it. Once token auth is wired in, that can delegate read access to sessions the caller may not be allowed to share, so validate session ownership or visibility before minting.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR introduces bearer session-share tokens and public mint/revoke RPCs before the authorization boundary is fully defined and enforced.

Concerns:

- **[medium] Token issuance is not bound to session authorization:** `src/gateway/server-methods/session-share-tokens.ts:27`
  - body: sessions.createSharedToken accepts a caller-supplied sessionKey and mints a bearer token without proving the caller may delegate read access to that session. The PR should define and enforce that rule before this token can become an auth credential.
  - confidence: 0.8
- **[medium] Returned tokens are not enforced by auth middleware:** `src/gateway/server-methods/session-share-tokens.ts:27`
  - body: The patch creates and persists bearer tokens, but it does not integrate token resolution with the WebSocket connect path. That leaves the new security credential unvalidated and incomplete at the gateway auth boundary.
  - confidence: 0.86
