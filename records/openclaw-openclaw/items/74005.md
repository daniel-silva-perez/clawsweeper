## Summary

Keep open. Current main still has a plausible HTTPS_PROXY-only failure mode for model-provider OpenAI-family requests: the OpenAI transports now pass a guarded fetch into the OpenAI SDK, but that guarded fetch uses strict SSRF mode without enabling trusted env-proxy mode for ordinary inherited HTTP_PROXY/HTTPS_PROXY settings. The reporter's proxy-log evidence and the existing strict-mode test line up with current source rather than showing an obsolete report.

## What This Changes

The issue asks OpenClaw to make OpenAI-family model-provider requests to api.openai.com work in HTTPS_PROXY-only gateway environments, or to surface/document the proxy routing gap clearly.

## Best Possible Solution

The best end state is a maintainer-approved generic provider-transport policy: either apply the existing guarded auto-upgrade pattern used by media-understanding to model-provider HTTP requests when HTTP_PROXY/HTTPS_PROXY is present and NO_PROXY does not match, or explicitly document and doctor the requirement to set `models.providers.*.request.proxy.mode: "env-proxy"` or managed `proxy.enabled=true` for proxy-only sandboxes. The fix should not be OpenAI-id-specific.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction path exists: configure a model-provider request without `request.proxy` or managed `proxy.enabled`, set only HTTP_PROXY/HTTPS_PROXY, and current `buildGuardedModelFetch` reaches `fetchWithSsrFGuard` in strict mode, where tests assert env proxy is ignored by default.

## Solution Assessment

Unclear for implementation, because there is no proposed patch and the narrowest maintainable fix needs maintainer security judgment. A generic provider transport change using the existing NO_PROXY/HTTP_PROXY-gated trusted-env-proxy pattern is safer than adding an OpenAI-specific bypass.
