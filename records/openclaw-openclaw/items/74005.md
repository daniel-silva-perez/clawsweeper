## Summary

Keep open. Current main still supports the reported failure class: native OpenAI completions calls are routed through OpenClaw's guarded provider fetch, and that guard does not honor inherited `HTTPS_PROXY`/`HTTP_PROXY` in strict mode unless explicit provider proxy config or managed OpenClaw proxy mode is active. The issue is recent, detailed, and security-sensitive enough that the remaining contract should be decided by maintainers rather than swept closed.

## What This Changes

The issue asks OpenClaw to make `openai-completions` requests to `api.openai.com` work in HTTPS_PROXY-only environments or emit a clear diagnostic when the OpenAI-specific path bypasses proxy routing.

## Best Possible Solution

Preserve the strict SSRF default, then make the proxy contract explicit for model-provider traffic. A maintainer should either add a narrowly trusted provider-transport path for public native model endpoints in proxy-only deployments, or improve docs/doctor/runtime errors so users are directed to `models.providers.*.request.proxy.mode: "env-proxy"` or managed `proxy.enabled` without implying plain inherited `HTTPS_PROXY` is enough.

## Reproduction Assessment

Yes. The report gives concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current source/tests independently show why provider requests without explicit `request.proxy` or managed proxy mode stay on strict guarded fetch instead of inherited env-proxy routing.

## Solution Assessment

Unclear. The requested automatic inherited-env behavior would address the reported deployment, but the current strict-mode test documents a deliberate SSRF/DNS-pinning invariant; the safer solution needs maintainer security/product judgment first.
