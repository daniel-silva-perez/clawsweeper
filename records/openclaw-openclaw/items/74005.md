## Summary

Keep this open. Current main routes OpenAI Responses and Chat Completions through OpenClaw's guarded provider fetch, but that guard still ignores plain inherited HTTP_PROXY/HTTPS_PROXY in strict mode unless the explicit provider request proxy mode or the managed OpenClaw proxy lifecycle is active. That matches the reporter's HTTPS_PROXY-only failure mode rather than proving it fixed. The linked #56841 and #42311 context is useful, but neither is a canonical merged fix for this specific env-only OpenAI provider path.

## Best Possible Solution

Keep this issue open for maintainer follow-up. The best fix is a generic proxy semantics decision in the provider transport/fetch guard layer: either make trusted public model-provider requests honor inherited env proxy settings without weakening SSRF protections, or improve doctor/docs/errors so HTTPS_PROXY-only deployments are directed to request.proxy.mode: "env-proxy" or proxy.enabled=true. Do not add an OpenAI-id-specific bypass unless the generic contract requires a provider-owned exception.
