## Summary

Keep open. Current main still matches the reported failure class: native OpenAI provider turns use OpenClaw's guarded provider fetch, and that guard intentionally does not honor plain inherited HTTPS_PROXY/HTTP_PROXY in strict mode unless an explicit provider request proxy or managed OpenClaw proxy mode is configured.

## What This Changes

The issue reports that OpenAI provider traffic to api.openai.com fails in HTTPS_PROXY-only sandboxes while other upstreams work from the same gateway process.

## Best Possible Solution

Preserve the SSRF guard's secure default while making the proxy contract explicit. Maintainers should either add a generic provider-transport path that safely honors env proxies for trusted public model endpoints, or improve doctor/errors/docs so HTTPS_PROXY-only deployments are directed to models.providers.*.request.proxy.mode: "env-proxy" or managed proxy.enabled without implying OpenAI-specific behavior.

## Reproduction Assessment

Yes. The issue gives a concrete HTTPS_PROXY-only sandbox reproduction, and current source/tests independently show why a provider request without request.proxy or managed proxy mode will remain in strict guarded fetch rather than using the env proxy.

## Solution Assessment

Unclear. Automatically honoring inherited env proxy settings might solve this environment, but the existing strict-mode test shows that behavior is intentionally constrained for DNS-pinning/SSRF reasons; the safer path needs maintainer security/product judgment before implementation.
