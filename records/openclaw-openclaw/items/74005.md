## Summary

Keep open. Current main still matches the reported class: OpenAI Chat Completions goes through OpenClaw's guarded provider fetch, and the strict guard does not honor inherited HTTPS_PROXY/HTTP_PROXY unless the provider request explicitly opts into env-proxy mode or managed OpenClaw proxy mode is active.

## What This Changes

The issue reports OpenAI completions calls to api.openai.com failing in an HTTPS_PROXY-only sandbox while other provider and channel traffic from the same gateway process reaches the proxy.

## Best Possible Solution

Preserve the secure strict-mode default while making the proxy contract explicit. Maintainers should choose either a safe generic provider-transport path for trusted public model endpoints to honor env proxies, or clearer doctor/error/docs guidance that HTTPS_PROXY-only deployments must use models.providers.*.request.proxy.mode: "env-proxy" or managed proxy.enabled.

## Reproduction Assessment

Yes. The issue gives a concrete HTTPS_PROXY-only sandbox reproduction, and current source plus SSRF tests independently show why a provider request without explicit request.proxy or managed proxy mode remains in strict guarded fetch instead of using the env proxy.

## Solution Assessment

Unclear. Automatically honoring inherited HTTPS_PROXY would address this environment, but the existing strict-mode invariant is tied to SSRF/DNS-pinning behavior; the safer solution needs maintainer security and product judgment before implementation.
