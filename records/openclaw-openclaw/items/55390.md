## Summary

Keep open. Current main supports manual config schema export and root `$schema` validation, but it does not implement the PR’s local `openclaw_schema.json` maintenance or `update.maintainConfigJsonSchema` opt-out. The PR remains useful as the implementation candidate for the paired feature issue, but the proposed startup path still needs maintainer review because it can rewrite a JSON5 config during read-style CLI runs just to add `$schema`.

## What This Changes

The PR adds local `openclaw_schema.json` maintenance beside `openclaw.json`, defaults missing `$schema` values to that file, adds `update.maintainConfigJsonSchema`, and updates config CLI docs/tests/schema metadata.

## Best Possible Solution

Keep the feature direction, but make startup schema maintenance non-destructive: write the sidecar schema best-effort, add `$schema` with a source-preserving root-file update or only during explicit config writes, keep the opt-out documented, and add regression coverage for invalid configs, included configs, JSON5 comments, help/version fast paths, and JSON output modes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve JSON5 source when adding the startup schema ref:** `src/config/local-json-schema.ts:98`
  - body: The startup maintenance path calls `writeConfigFile` only to add `$schema` when it is missing. That writer serializes the parsed config with `JSON.stringify`, so a normal read-style CLI command against a JSON5 config can drop comments and formatting while adding one field. Use a source-preserving insertion path or avoid rewriting the config during startup maintenance.
