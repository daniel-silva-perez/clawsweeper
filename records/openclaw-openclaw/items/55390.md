## Summary

Keep open. Current main still only exposes `openclaw config schema` on demand and does not maintain a sibling schema artifact, so the PR is not obsolete. The proposed branch has a blocking config-persistence regression around `$include`-owned configs that needs maintainer/author follow-up before merge.

## What This Changes

The PR adds best-effort local `openclaw_schema.json` generation, `$schema` backfilling on startup/config writes, an `update.maintainConfigJsonSchema` opt-out, docs, and targeted tests.

## Best Possible Solution

Rebase the feature onto current main and make schema backfill use an authored-root-aware config mutation path, or skip root-file mutation when the authored root uses `$include`; keep schema-file generation best-effort, preserve custom `$schema` values, retain the camelCase opt-out, and add include-owned regression coverage.

## Reproduction Assessment

Yes. The source-level path is clear: use an `openclaw.json` with a root `$include` and no authored root `$schema`, then trigger the PR's startup schema maintenance; the branch writes a resolved snapshot back through `writeConfigFile` instead of preserving the authored root structure.

## Solution Assessment

No. The product direction is reasonable, but this implementation is not the safest merge shape until `$schema` backfill respects authored includes or avoids mutating include-owned roots.
