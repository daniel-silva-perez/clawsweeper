## Summary

Keep this PR open. Current main still has a plausible ByteString crash path for pasted non-Latin1 API keys in header-bound flows, so the PR is not obsolete. However, the proposed fix is too broad because it changes generic SecretInput normalization and can mutate valid Unicode credentials outside HTTP-header usage.

## Best Possible Solution

Keep this PR open and request a narrower patch. Preserve trim-only semantics for `normalizeSecretInputString`, `hasConfiguredSecretInput`, and non-header credentials. Apply `normalizeSecretInput` or an explicitly header-value-safe helper only where values are assembled into HTTP headers, starting with Comfy Cloud `X-API-Key` and shared provider request header/auth assembly. Add regression coverage for pasted U+2502 in a header-bound path and guard coverage proving Unicode Matrix/gateway password values and configured-presence checks remain unchanged.

## Risks / Open Questions

- Merging this PR as written would globally strip code points above `0xff` from generic SecretInput strings and can change `hasConfiguredSecretInput` presence checks for all-non-Latin1 values.
- Non-header credentials can be valid Unicode values; Matrix plaintext passwords are a concrete current-main path where broad stripping would mutate authentication material before login.
- Leaving the PR unresolved preserves the real header-bound crash class for Comfy Cloud `X-API-Key` and other shared provider request header/auth paths that currently pass values through to `new Headers`.
- No test suite was run because this was a read-only cleanup review; source inspection and read-only Node header-construction probes were used instead.
