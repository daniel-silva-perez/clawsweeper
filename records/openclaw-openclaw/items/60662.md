## Summary

Keep open. Current main has a real macOS screen.snapshot node primitive, but the agent-facing screenshot workflow and failure-response policy still do not clearly prevent the reported behavior of handing the failed operation back to the user.

## What This Changes

The issue asks OpenClaw to handle action failures without telling the user to perform the same requested operation themselves, using screenshot capture failure as the concrete example.

## Best Possible Solution

Add a first-class agent-facing screen_snapshot/media path, with explicit permission or unusable-capture failure handling, then cover the response behavior so OpenClaw explains its limitation and offers OpenClaw-owned next steps instead of delegating the same failed action back to the user.

## Reproduction Assessment

No, not with high confidence from this review. The report gives a plausible macOS/TCC scenario, and source inspection confirms remaining gaps, but there is no transcript/log evidence and no live denied-screen-recording run was performed.

## Solution Assessment

No. Current main’s generic invoke plus screen_record support is not the best solution for this issue because it leaves screenshot capture without dedicated media handling and leaves the failed-action response policy model-dependent.
