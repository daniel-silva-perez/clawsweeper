## Summary

Keep open. Current main and the latest release still recognize only camelCase tool-call block types in the session repair path, while the issue's linked implementation PR remains open and unmerged, so the reported Anthropic snake_case corruption path is still actionable.

## What This Changes

Recognize snake_case tool call block types in session transcript repair so malformed Anthropic `tool_use` blocks missing `input` are sanitized before replay.

## Best Possible Solution

Land a focused fix that treats `tool_use`, `tool_call`, and `function_call` as tool-call blocks consistently across session repair, tool-call extraction, and embedded replay normalization, with regression tests for malformed snake_case blocks missing `input` or `arguments`. After merge, close this issue with the fixing SHA and release provenance.

## Reproduction Assessment

Yes. A code-level reproduction is to pass an assistant message containing `{ type: "tool_use", id: "call_1", name: "read" }` without `input` or `arguments` through `sanitizeToolCallInputs`; current main's predicate does not recognize it, so the malformed block survives.

## Solution Assessment

Yes, recognizing the snake_case variants in the shared tool-call recognition path is the narrowest maintainable fix. The safer version is to centralize the type set or update all duplicate recognizers together so replay repair and tool-call extraction agree.
