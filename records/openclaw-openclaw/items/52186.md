## Summary

Keep open. Current main has the diagnostic/observability work from #57953 and #57954, but the remaining provider-stickiness behavior called out in the maintainer comment is still not implemented for normal configured TTS or `/tts audio`: a configured ElevenLabs primary can still fall through to another configured provider such as OpenAI after a provider failure.

## Best Possible Solution

Keep this issue open for a focused TTS fallback policy fix. The likely best path is to decide whether an explicitly configured provider or `/tts provider <id>` should fail closed by default, or add a documented config/prefs switch to disable fallback for pinned providers, then update `/tts status`, docs, and regression tests covering ElevenLabs primary with OpenAI configured as a fallback.
