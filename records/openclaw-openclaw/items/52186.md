## Summary

Keep open. Current main has the diagnostic/fallback observability work from #57953 and #57954, but the remaining provider-stickiness behavior is still not implemented for normal configured TTS or `/tts audio`: a configured ElevenLabs primary can still fall through to another configured speech provider such as OpenAI after a provider failure.

## Best Possible Solution

Keep this issue open for a focused TTS fallback policy decision. The likely implementation path is to decide whether an explicitly configured provider or `/tts provider <id>` should fail closed by default, or add a documented config/preference switch to disable fallback for pinned providers; then thread that through automatic TTS and `/tts audio`, update `/tts status` and docs, and add regression tests covering ElevenLabs primary with OpenAI configured as a fallback.
