## Summary

Keep this issue open. Current main still rejects `timeoutSeconds: 0` in the Control UI and still omits an explicit zero from the submitted cron payload, while gateway/runtime cron handling already treats zero as valid no-timeout input. The same-author implementation candidate PR #41275 remains open and uses closing syntax, so this issue should stay open until that PR or an equivalent complete fix lands.

## Best Possible Solution

Keep this issue open and review, update, or replace PR #41275. A complete fix should allow zero in `validateCronForm`, preserve explicit `timeoutSeconds: 0` in `buildCronPayload`, update Control UI regression tests so zero is accepted while negative values are rejected, and align the English timeout help/error copy with the gateway/runtime no-timeout contract. No core protocol change appears necessary.
