## Summary

Keep open. Current main still rejects `timeoutSeconds: 0` in the Control UI and drops explicit zero from the submitted cron payload, while the gateway/runtime cron contract accepts zero as no-timeout. Same-author PR #41275 remains an open implementation candidate with closing syntax, so this issue should not be closed before that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review or replace #41275. A complete fix should allow zero in `validateCronForm`, preserve explicit `timeoutSeconds: 0` in `buildCronPayload`, update Control UI regression tests, and align timeout help/error copy with the existing gateway/runtime no-timeout contract; no core protocol change appears necessary.
