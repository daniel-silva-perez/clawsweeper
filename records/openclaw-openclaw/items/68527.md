## Summary

Keep open. Current main has meaningful partial mitigations for the linked bundled-MCP process leak, including runtime reuse, idle cleanup, and stdio process-tree teardown, but the broader request remains unresolved: there is still no per-MCP backoff/circuit breaker across new runtimes, `openclaw mcp set` still does not preflight reachability, status does not expose MCP child process/RSS accounting, and the generated systemd unit lacks the requested memory/task/OOM guardrails.

## What This Changes

Add MCP launch backoff, per-server circuit breaking, MCP config preflight, status resource accounting, and default Linux service guardrails so misconfigured MCP servers cannot exhaust VM resources.

## Best Possible Solution

Implement a split hardening plan: add per-server MCP failure state keyed by config fingerprint so repeated fail-fast launches cool down and clear on config change, add an explicit MCP definition preflight path for saved configs, and separately decide the default Linux service cgroup/swap/watchdog policy with docs.

## Reproduction Assessment

Yes, for the missing guardrail class: configure a fail-fast stdio MCP server under `mcp.servers` and trigger new bundled-MCP session runtimes; current source shows startup is attempted during catalog build without persisted per-server cooldown or circuit state. The exact 312-process VM incident was not live-replayed.

## Solution Assessment

No. Current main is a useful partial fix for orphaned and duplicate MCP process accumulation, but the best solution still needs bounded failed-start behavior plus a separate maintainer decision on default systemd and operator guardrails.
