## Summary

Keep open. Current main has shipped partial MCP lifecycle cleanup that reduces the duplicate-process leak from the companion report, but the broader requested safeguards are still missing: persistent per-server startup backoff/circuit state, live MCP config preflight, MCP child/RSS status accounting, and default systemd memory/task ceilings.

## What This Changes

The issue requests MCP startup backoff, per-server circuit breaking, MCP config preflight validation, process/resource accounting, and deployment guardrails to prevent misconfigured MCP servers from exhausting VM resources.

## Best Possible Solution

Define a generic MCP launch-health policy keyed by server and config fingerprint, with bounded startup retry/cooldown semantics, an operator-visible unhealthy state, an explicit MCP preflight/check path, resource visibility for MCP children, and a deliberate systemd/doctor/docs rollout for memory and task guardrails.

## Reproduction Assessment

Yes for the missing safeguards: a misconfigured stdio MCP server can be modeled by a command that exits during catalog materialization, and current main shows immediate start/list failure handling without persistent backoff or circuit state across future runtime creation. The full VM-level resource exhaustion incident was not live-reproduced in this read-only review.

## Solution Assessment

No, current main is only a partial mitigation. The cleanup shipped in v2026.4.27 is useful, but the narrow maintainable solution still needs explicit MCP launch policy, preflight/status surfaces, and maintainer-chosen Linux service guardrails rather than relying only on cleanup after failed starts.
