## Summary

Keep open. Current main does not already implement the requested caller-context injection, and the PR is useful but not mergeable as written because it drops existing non-string headers and is stale against main's disabled-tools bundle MCP isolation on a security-sensitive caller/session forwarding path.

## What This Changes

The PR adds an optional `mcp.servers.<name>.injectCallerContext` config flag, bundle MCP caller-header injection and placeholder env handling, generated config schema/baseline updates, changelog entries, and focused unit coverage for Claude, Codex, and Gemini bundle MCP paths.

## Best Possible Solution

Rebase the branch onto current main, keep the per-server owner-managed opt-in shape, preserve the disabled-tools bundle MCP gate for both env creation and bundle MCP preparation, merge caller placeholders into a copy of the original header object without dropping non-string values, then add focused tests for numeric/boolean header preservation and disabled-tools env behavior before maintainer review of the caller-context trust boundary.

## Reproduction Assessment

Yes. The feature gap is visible on current main because there is no `injectCallerContext` config key or injection helper, and the PR bugs can be reproduced by using an opted-in remote server with a numeric or boolean header and by comparing PR head against current main's disabled-tools bundle MCP test and gate.

## Solution Assessment

No, not as written. The per-server opt-in is the right narrow direction, but the implementation must preserve existing header values and rebase around `bundleMcpEnabled` before it is a maintainable and safe solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve existing non-string MCP headers:** `src/agents/cli-runner/bundle-mcp-caller-context.ts:53-61`
  - body: `McpServerConfig.headers` and the zod schema allow `string | number | boolean`, but this helper normalizes existing headers through `normalizeStringRecord` and then assigns that string-only map back to `server.headers`. In Claude config-file mode, an opted-in server with an existing header such as `{ "X-Tenant": 42 }` loses that header when caller context is enabled, despite the non-destructive merge contract. Build the lowercased key set from all original header names and merge caller placeholders into a copy of the original header object.
  - confidence: 0.89
- **[P2] Preserve the disabled-tools bundle MCP gate:** `src/agents/cli-runner/prepare.ts:181-182`
  - body: Current main gates bundle MCP with `params.disableTools !== true`, but this branch's new placeholder-env path keys off `backendResolved.bundleMcp`, and the PR head still passes that value to bundle MCP preparation. If the rebase keeps this shape, a run with tools disabled can still prepare bundle MCP and expose caller/session env for opted-in MCP injection. Rebase this block around `bundleMcpEnabled` for both env creation and `prepareCliBundleMcpConfig`.
  - confidence: 0.78
