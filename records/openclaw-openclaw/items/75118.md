## Summary

The PR should stay open: current main still lacks the requested feature and the linked feature issue remains open, but the branch is not merge-ready because the added caller-context helper drops existing non-string header values even though the MCP config contract allows them. The PR is also currently dirty against main, so it needs author/maintainer follow-up rather than conservative cleanup.

## What This Changes

The PR adds an opt-in `mcp.servers.<name>.injectCallerContext` config/schema field, injects OpenClaw caller headers into trusted bundled CLI remote MCP server configs, updates placeholder env handling, and adds changelog, config-baseline, and unit-test coverage.

## Best Possible Solution

Keep the per-server, owner-managed opt-in shape and the plugin self-opt-in block, but fix the header merge to preserve every existing header entry before maintainers review and land the rebased PR. If maintainers reject the caller/session forwarding trust model, keep the linked feature issue open for a revised design.

## Reproduction Assessment

Yes. The feature gap is reproducible by adding a bundled CLI remote MCP server with `injectCallerContext: true` and observing that current main has no config key or merge pass; the PR bug is reproducible by adding an opted-in server with a numeric or boolean header and seeing the helper replace the original header map with string-only entries.

## Solution Assessment

No, not as written. The per-server owner opt-in is the narrow maintainable direction, but the implementation should preserve the full existing header map and only add missing caller headers before it can satisfy the stated non-destructive contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve non-string headers when injecting caller context:** `src/agents/cli-runner/bundle-mcp-caller-context.ts:53-54`
  - body: `McpServerConfig.headers` and the zod schema allow `string | number | boolean` values, but this code normalizes the existing map through `normalizeStringRecord` and then replaces `server.headers` with that filtered map. In Claude config-file mode, an opted-in server with an existing non-string header such as `{ "X-Tenant": 42 }` loses that header whenever caller context is enabled, contradicting the PR's "existing headers always win" contract. Build the lowercased key set from all existing header names and merge caller headers into a copy of the original header object instead.
  - confidence: 0.88
