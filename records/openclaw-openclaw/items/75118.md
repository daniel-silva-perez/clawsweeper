## Summary

Keep open. Current main does not implement the requested MCP caller-context option, and this is an active implementation PR paired with open feature issue #75116. The final PR head appears to address the earlier security concern by requiring owner-managed config or OpenClaw runtime-supplied config before caller headers are forwarded, so I found no blocking review finding in the current diff.

## What This Changes

This PR adds `mcp.servers.<name>.injectCallerContext` for bundled CLI MCP remote servers, injects OpenClaw caller headers into trusted generated MCP configs, updates schema/baseline/changelog, and adds bundle-MCP tests.

## Best Possible Solution

Review and land this PR if maintainers accept the owner-managed trust model for forwarding caller/session context; otherwise keep the same narrow per-server API shape and tighten the trust gate or docs before merge.

## Reproduction Assessment

Yes. The feature path is reproducible by adding a bundled CLI remote MCP server with `injectCallerContext: true` and inspecting the generated Claude/Codex/Gemini MCP config; current main lacks both the config key and the merge pass, while the PR adds unit coverage for the expected generated config.

## Solution Assessment

Yes, with normal maintainer judgment on the trust boundary. The final diff is a narrow per-server opt-in, strips the private flag from emitted MCP config, preserves existing headers case-insensitively, skips stdio, and no longer lets plugin bundle MCP definitions self-grant caller-header forwarding.
