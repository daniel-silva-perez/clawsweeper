## Summary

Keep open. Current main does not implement the requested `mcp.servers.<name>.injectCallerContext` contract, and this PR is the active implementation paired with the open feature issue. The patch needs maintainer/security review before merge because the implementation honors the flag after bundle plugin MCP defaults and owner config have been merged, so an enabled plugin bundle can opt into forwarding caller identity/session headers without an explicit owner-managed `mcp.servers` override.

## What This Changes

The PR adds an opt-in bundled CLI MCP `injectCallerContext` server option, injects OpenClaw caller headers into remote URL MCP servers, populates `OPENCLAW_MCP_*` env placeholders when needed, and updates config schema, generated docs baseline, changelog, and bundle-MCP tests.

## Best Possible Solution

Keep this PR open and either tighten the implementation so caller-context forwarding is honored only from an explicit owner-managed config/trust decision, or document and test plugin-bundle opt-in as an intentional trusted-plugin capability before merging.

## Reproduction Assessment

Yes. The feature can be reproduced by adding a remote bundled MCP server with `injectCallerContext: true` and inspecting the generated Claude/Codex/Gemini MCP config; the security concern is statically reproducible because plugin bundle MCP defaults and owner config are merged into one map before the new helper runs.

## Solution Assessment

Unclear. Per-server opt-in and non-overwriting header merge are the right narrow shape, but applying the opt-in to plugin-provided bundle MCP layers may not be the safest boundary for forwarding caller/session context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Require owner opt-in before forwarding caller headers:** `src/agents/cli-runner/bundle-mcp.ts:189`
  - body: Because caller-context injection runs after plugin bundle MCP defaults and owner config are merged, an enabled bundle can set `injectCallerContext: true` and receive `x-session-key` plus caller IDs even when the user's `mcp.servers` never opted in. Please restrict this to an explicit owner-managed trust decision, or add a dedicated trusted-plugin permission/docs path for this data flow.
  - confidence: 0.82
