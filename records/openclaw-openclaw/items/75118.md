## Summary

Keep open. Current main still lacks the requested opt-in caller-context injection, so the PR is not obsolete, but the branch has blocking issues around non-destructive header merging and a stale conflict with current main's disabled-tools bundle MCP gate.

## What This Changes

The PR adds a per-server `injectCallerContext` MCP config flag, caller-header injection and placeholder env handling for bundled CLI remote MCP configs, generated config schema and changelog updates, and unit coverage across Claude, Codex, and Gemini bundle MCP paths.

## Best Possible Solution

Rebase the branch onto current main, keep the per-server owner-managed opt-in and plugin self-opt-in block, preserve the disabled-tools bundle MCP gate, fix header merging to copy the original header map while adding only missing caller placeholders, and add focused coverage for numeric/boolean header preservation plus disabled-tools env behavior before maintainer review of the trust boundary.

## Reproduction Assessment

Yes. The feature gap is reproducible on current main by configuring a remote bundle MCP server with the proposed flag and observing that no such config key or injection pass exists; the PR bugs are reproducible with an opted-in server containing a numeric or boolean header, and by checking the PR head against current main's `disableTools` gating in `prepare.ts`.

## Solution Assessment

No, not as written. The per-server owner opt-in is the narrow maintainable direction, but the implementation must preserve all existing header values and be rebased to use current main's `bundleMcpEnabled` gate before it is a safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve existing non-string MCP headers:** `src/agents/cli-runner/bundle-mcp-caller-context.ts:53-61`
  - body: `McpServerConfig.headers` and the zod schema allow `string | number | boolean`, but this helper builds `existing` with `normalizeStringRecord` and then replaces `server.headers` with that string-only map. In Claude config-file mode, an opted-in server with an existing header such as `{ "X-Tenant": 42 }` loses that header when caller context is enabled, contradicting the non-destructive header contract. Build the lowercased key set from all existing header names and merge the caller placeholders into a copy of the original header object.
  - confidence: 0.88
- **[P2] Preserve the disabled-tools bundle MCP gate:** `src/agents/cli-runner/prepare.ts:180-182`
  - body: Current main added `bundleMcpEnabled = backendResolved.bundleMcp && params.disableTools !== true`, but this branch's new placeholder-env path keys off `backendResolved.bundleMcp`, and the PR head still passes that value to bundle MCP preparation. If the rebase keeps this shape, a run with tools disabled can still prepare bundle MCP and expose caller/session env for an opted-in server. Rebase this block around `bundleMcpEnabled` before merge.
  - confidence: 0.76
