## Summary

Keep open. Current main does not implement caller-context injection for bundled remote MCP servers, and this PR is a plausible focused implementation for the paired open feature request, but the branch is not merge-ready because it introduces a test type error and still needs maintainer judgment for forwarding caller/session context across the MCP boundary.

## What This Changes

The PR adds an opt-in `mcp.servers.<name>.injectCallerContext` config property, caller-header merge logic for bundled remote MCP servers, placeholder env handling, schema/changelog/baseline updates, and bundle-MCP tests for Claude, Codex, and Gemini paths.

## Best Possible Solution

Keep the narrow per-server, owner-managed opt-in shape if maintainers accept the trust boundary, fix the test type error, rebase the PR onto current main, and validate the bundle-MCP/config-doc lanes before merge.

## Reproduction Assessment

Yes. The feature path is reproducible by configuring a bundled remote MCP server with `url` and `injectCallerContext: true` and inspecting the generated CLI MCP config; current main lacks the config key, injection pass, and non-loopback placeholder env path. The PR defect is also high-confidence from static TypeScript inspection of the changed test type and its new `.headers` assertions.

## Solution Assessment

Yes in direction, but not yet merge-ready. A per-server owner opt-in with plugin-supplied flags stripped is the narrowest maintainable API shape I found; the safer alternative is to keep this trust gate and fix the test type/rebase issues before final maintainer review.
