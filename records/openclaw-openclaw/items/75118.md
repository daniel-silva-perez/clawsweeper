## Summary

Keep open. Current main still lacks the requested bundle MCP caller-context config and merge path, but the PR head is not ready to merge because it drops existing non-string headers during injection and is currently dirty against main.

## What This Changes

The branch adds an optional `mcp.servers.<name>.injectCallerContext` config/schema field, injects caller headers for trusted bundled CLI remote MCP server configs, updates placeholder env handling, generated config baseline, changelog, and unit tests.

## Best Possible Solution

Keep the per-server, owner-managed opt-in shape and the plugin self-opt-in block, but change the injection helper to copy the original header object, build the case-insensitive existing-name set from all original header keys, and only add missing caller placeholders before maintainers review the rebased PR.

## Reproduction Assessment

Yes. The feature gap is reproducible by adding a bundled CLI remote MCP server with `injectCallerContext: true` and observing that current main has no config key or merge pass; the PR bug is reproducible by adding an opted-in server with a numeric or boolean header and seeing the helper replace the original header map with string-only entries.

## Solution Assessment

No, not as written. The per-server owner opt-in is the narrow maintainable direction, but the implementation must preserve all existing header values before it satisfies the non-destructive contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve non-string headers when injecting caller context:** `src/agents/cli-runner/bundle-mcp-caller-context.ts:53-54`
  - body: `McpServerConfig.headers` allows `string | number | boolean` values, but this helper filters existing headers through `normalizeStringRecord` and then writes that filtered map back to `server.headers`. In Claude config-file mode, an opted-in server with an existing header like `{ "X-Tenant": 42 }` loses that header whenever caller context is enabled, contradicting the PR's existing-headers-win contract. Build the lowercased key set from all original header names and merge caller headers into a copy of the original header object instead.
  - confidence: 0.88
