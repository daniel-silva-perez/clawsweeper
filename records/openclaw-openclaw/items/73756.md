## Summary

Keep this issue open. Current main still has the reported Telegram fail-open shape: `dmPolicy: "open"` is an unconditional DM accept path, account-level `allowFrom: ["*"]` can replace a restrictive top-level allowlist after account config merging, and existing audit/docs coverage warns about public DMs without fixing the runtime conflict.

## Best Possible Solution

Keep this open for Telegram-owned access-control work. Define fail-closed effective-policy semantics when a restrictive top-level allowlist conflicts with account-level public DM settings, add regression coverage for `mergeTelegramAccountConfig`, `enforceTelegramDmAccess`, `TelegramConfigSchema`, and the security/status surfaces, then document and migrate the intentional-public-bot path explicitly.

## Risks / Open Questions

- Changing `dmPolicy="open"` semantics could break operators who intentionally configured public Telegram DMs; maintainers should choose an explicit migration or high-friction confirmation path.
- Existing audit coverage already warns about generic open DM access, so the remaining work should focus on the conflict/effective-policy case without duplicating generic warnings.
- The related auto-heal PR touches nearby account allowlist behavior; any fix should coordinate with that work to avoid widening sibling-account access.
