## Summary

Keep this issue open. Current main still treats Telegram account-level `dmPolicy: "open"` as an unconditional DM accept path, and account-level `allowFrom: ["*"]` overrides the visible top-level allowlist after account config merging. The issue is recent, concrete, and not fully superseded by the related audit/auto-heal items.

## Best Possible Solution

Keep the issue open and handle it in the Telegram-owned access-control surface. The best fix is to define fail-closed effective-policy semantics for top-level restrictive allowlists versus account-level public overrides, add regression tests around `mergeTelegramAccountConfig`, `enforceTelegramDmAccess`, and `TelegramConfigSchema`, and update security audit/status/docs to show when a named account is effectively public.

## Risks / Open Questions

- Changing `dmPolicy="open"` semantics may break operators who intentionally configured public Telegram DMs; this needs an explicit migration or high-friction confirmation path.
- There is partial security-audit coverage already, so the remaining fix should target the conflict/effective-policy case without duplicating generic open-DM warnings.
- The related auto-heal PR touches nearby account allowlist behavior; maintainers should coordinate any runtime/schema change with that PR to avoid widening sibling account access.
