## Summary

Keep this issue open. Current main still has the core Telegram account-level public-DM behavior: an effective `dmPolicy: "open"` returns before allowlist matching, and account config can override a restrictive top-level `allowFrom` with `allowFrom: ["*"]`. Existing schema and audit checks are partial mitigations, not a fail-closed runtime or effective-policy fix.

## Best Possible Solution

Keep this issue open and fix it in the Telegram-owned access-control surface. Define fail-closed effective-policy semantics for restrictive channel allowlists versus account-level public overrides, add regression tests for `mergeTelegramAccountConfig`, `TelegramConfigSchema`, and `enforceTelegramDmAccess`, and update security audit/status/docs so named accounts clearly show when they are effectively public.

## Risks / Open Questions

- Changing `dmPolicy: "open"` semantics could break operators who intentionally configured public Telegram bots, so a fix likely needs migration, explicit confirmation, or clear effective-policy UI/audit output.
- The exact redacted shape with top-level `dmPolicy: "open"` and restrictive top-level `allowFrom` is now rejected by schema, but the account-level override variant remains; the fix should target the remaining cross-layer conflict without duplicating existing generic open-DM warnings.
- The related auto-heal work in #38683 touches the same account allowlist area and should be coordinated with any runtime/schema change.
