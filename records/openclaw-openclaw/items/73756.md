## Summary

Keep this issue open. Current main still has the reported Telegram fail-open shape: `dmPolicy: "open"` is an unconditional runtime accept path, account-level `allowFrom` can replace a restrictive inherited top-level allowlist, and existing audit/docs coverage warns about public DMs without fixing the runtime conflict.

## Best Possible Solution

Keep this open for Telegram-owned access-control work. Define fail-closed effective-policy semantics when a restrictive top-level allowlist conflicts with account-level public DM settings, preserve an explicit intentional-public-bot path, add regression coverage for `mergeTelegramAccountConfig`, `enforceTelegramDmAccess`, event authorization, `TelegramConfigSchema`, and security/status surfaces, then update docs and coordinate with related PR #38683.
