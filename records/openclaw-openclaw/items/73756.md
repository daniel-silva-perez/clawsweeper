## Summary

Keep this issue open. Current main still has the reported Telegram access-control conflict: `dmPolicy: "open"` accepts DMs before allowlist matching, account-level config can override a restrictive top-level `allowFrom`, validation permits an effective account wildcard for open policy, and audit/docs warn about public DMs without making the runtime fail closed.

## Best Possible Solution

Keep this open for Telegram-owned access-control work. Define fail-closed effective-policy semantics for restrictive top-level allowlists versus account-level public DM settings, add regression coverage around `mergeTelegramAccountConfig`, `enforceTelegramDmAccess`, schema validation, audit/status surfaces, and document or migrate the intentional-public-bot path explicitly.
