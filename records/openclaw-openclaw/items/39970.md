## Summary

Keep this PR open. Current main has partial shared support for `audioAsVoice` propagation and iMessage text chunking, but it still does not implement the PR's iMessage-owned `audioAsVoice` payload path. The current iMessage plugin exposes only `sendText`/`sendMedia`, the attached-results SDK wrapper does not synthesize `sendPayload`, and the iMessage runtime/sender still has no `audioAsVoice` option, so voice-intent audio can still fall into the generic attachment placeholder path.

## Best Possible Solution

Keep this PR open and ask for a narrowed rebase against `extensions/imessage`. The best fix is an iMessage-owned `sendPayload` path, or a generic SDK helper that attached-results channels can safely opt into, that threads `audioAsVoice` through `channel.ts` -> `channel.runtime.ts` -> `send.ts`, suppresses the generic `<media:audio>` placeholder for voice-intent audio, sends visible captions separately when needed, preserves `channels.imessage.textChunkLimit`, and adds focused iMessage regression tests plus a live macOS Messages smoke when feasible.
