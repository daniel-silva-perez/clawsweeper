## Summary

Keep this PR open. Current main has partial shared support for audioAsVoice and iMessage text chunking, but it still does not implement the PR's iMessage-owned audioAsVoice payload path. The iMessage plugin exposes only attached sendText/sendMedia, the SDK wrapper does not synthesize sendPayload, and the runtime/sender boundary still lacks audioAsVoice handling, so empty voice-intent audio can still become the generic <media:audio> placeholder.

## Best Possible Solution

Keep this PR open and ask for a narrowed rebase or rewrite against extensions/imessage. The best fix is an iMessage-owned sendPayload path, or an SDK attached-results helper that safely creates one, that forwards audioAsVoice through channel.ts -> channel.runtime.ts -> send.ts, suppresses the generic <media:audio> placeholder for voice-intent audio, sends visible caption text separately when needed, preserves configured text chunking, and adds focused iMessage regression tests.
