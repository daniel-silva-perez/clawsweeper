## Summary

Keep open. Current main still lets a one-line local `MEDIA:` directive absorb trailing serialized JSON as part of the media source, and that polluted string can flow into outbound delivery where the local media loader attempts to read it as a file path.

## What This Changes

Harden `MEDIA:` directive parsing so local file paths do not include trailing serialized tool-result JSON before outbound media delivery.

## Best Possible Solution

Add shared parser-level boundary handling for local `MEDIA:` sources so a valid media/document extension followed by serialized JSON is either trimmed to the valid source or rejected before delivery. Cover it with focused parser tests, and avoid per-channel workarounds or throw-based parsing that would make malformed directives fatal.

## Reproduction Assessment

Yes. The source trace on current main is high-confidence: the regex captures the full line, the validator accepts any slash-prefixed candidate, and outbound delivery later reads that candidate as a local file path. I did not run a live channel test because this review was read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared `splitMediaFromOutput` parsing/validation path, with regression coverage; channel-level guards can remain defense-in-depth but should not be the primary fix.
