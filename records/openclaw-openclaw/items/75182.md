## Summary

Keep open. Current main still has the shared parser path that can accept a slash-prefixed `MEDIA:` candidate with embedded serialized JSON, and an open closing PR already targets this exact bug, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue reports that shared `MEDIA:` directive parsing can absorb trailing serialized JSON into a local file path and asks for parser-level sanitization or rejection before outbound delivery.

## Best Possible Solution

Review and land the focused shared-parser fix in #75202, or replace it with an equivalent parser-level boundary check plus regression coverage. Avoid a channel-specific workaround unless it is added only as defense-in-depth after the shared parser stops emitting polluted paths.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: the regex captures the whole directive line, cleanup does not trim embedded JSON after a file extension, local path validation accepts slash-prefixed candidates, and outbound delivery later reads that value as a local path.

## Solution Assessment

Yes. A shared `splitMediaFromOutput` cleanup/validation fix with parser regression tests is the narrowest maintainable solution; channel-level guards alone would leave other outbound media paths exposed to the same parsed value.
