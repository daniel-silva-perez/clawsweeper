## Summary

Current main has the fixed/renewable inbound debounce that was tracked by related #55835, but it does not implement the adaptive/incremental timing requested here. The shared config and schema only expose `messages.inbound.debounceMs` plus `byChannel`, and the debouncer reschedules with a resolved fixed window rather than increasing the wait after each same-key message or applying an adaptive cap.

## Best Possible Solution

Keep this issue open for maintainer product/design judgment. A concrete implementation would likely extend the shared `messages.inbound` config and schema with an additive adaptive policy, update the generic inbound debouncer to track same-key burst count and cap the delay, then cover it with focused debouncer tests and docs for the fixed versus adaptive behavior.
