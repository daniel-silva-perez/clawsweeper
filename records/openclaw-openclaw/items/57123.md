## Summary

Keep this issue open. Current main still implements fixed/renewable inbound debounce, including per-channel fixed windows and a per-item override hook, but it does not implement the requested adaptive policy with an initial wait, per-message increment, and maximum cap. The related closed debounce work covers batching/concatenation, not this separate latency strategy.

## Best Possible Solution

Keep this issue open for maintainer product/design follow-up. A concrete implementation would add a documented adaptive inbound debounce policy under `messages.inbound`, extend config types/schema/generated metadata, track same-key burst count in the shared debouncer or a clear channel wrapper seam, cap the computed delay, and add tests/docs that distinguish fixed renewable debounce from adaptive incremental timing.
