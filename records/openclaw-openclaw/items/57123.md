## Summary

Keep this issue open. Current main still implements a fixed, renewable inbound debounce window, not the requested adaptive/incremental policy with an initial delay, per-message increment, and max cap. Related #55835 covered renewable debounce with concatenation, but the reporter explicitly distinguished this issue as complementary adaptive timing.

## Best Possible Solution

Keep this issue open for maintainer product/design follow-up. A likely implementation would add a documented adaptive inbound debounce policy to `messages.inbound`, extend the schema/types/generated config metadata, track same-key burst count in the generic debouncer, cap the computed delay, and add focused tests plus docs distinguishing fixed and adaptive debounce behavior.
