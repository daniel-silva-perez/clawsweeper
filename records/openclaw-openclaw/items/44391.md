## Summary

Current main still has the followup delivery bug this PR describes: followup payloads run text/media messaging-tool dedupe before checking whether the messaging tool sent to the originating target. The PR is still useful, but it should not land as-is because it is stale against the current helper split and bundles unrelated model retry, delivery serialization, and Mattermost dependency changes.

## Best Possible Solution

Keep this PR open, but ask for a narrowed version that ports the same-target dedupe guard into `resolveFollowupDeliveryPayloads`, preserves legacy dedupe when target metadata is unavailable, and adds followup text/media cross-target regression tests. Split model retry, delivery serialization, and Mattermost dependency work into separate PRs with their own review and validation.
