## Summary

Keep this PR open. Current main still uses the Telegram native slash envelope session key when persisting inbound session metadata, even though the same context carries a separate CommandTargetSessionKey for the routed/bound target session. The PR remains a focused implementation candidate for the bug described in #57509 rather than work already implemented on main.

## Best Possible Solution

Keep the PR open and have a maintainer review, land, or replace it with an equivalent narrow fix that persists Telegram native command metadata to ctxPayload.CommandTargetSessionKey?.trim() || ctxPayload.SessionKey || route.sessionKey, with the existing session-meta regression tests updated to assert the target-session key for private, helper-topic, configured ACP binding, and bound-topic flows.
