## Summary

Keep open. Current main still omits both session elapsed/start timing from the shared `/status` and `session_status` card and per-response duration from the visible usage footer; the related #68226 and #49762 items cover narrower pieces but do not fully resolve this broader report.

## What This Changes

Restore elapsed or response-time timing information to the `session_status`/`/status` display and related response footer after it disappeared for the reporter in 2026.3.28.

## Best Possible Solution

Keep this as the broad timing-observability tracker until the narrower pieces are resolved. Implement session start or elapsed duration in the shared status renderer using `sessionStartedAt` with lifecycle fallback, then coordinate the response-duration footer with #49762 or a replacement that preserves existing `/usage` modes and formatting.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the current status/session-status render path never reads session start data, and the response usage formatter has no duration input even though diagnostics record `durationMs`.

## Solution Assessment

Unclear for the whole report. The narrow maintainable direction is to put session lifetime in the shared status renderer and handle response duration in the usage footer, but the open partial PR and the newer #68226 tracker need maintainer coordination before this broad issue should close.
