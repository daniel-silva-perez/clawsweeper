## Summary

Keep open. Current main still lacks the direct single-job cron read path, and open PR #75117 is already linked as the active implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Add a direct cron job read path by id through `openclaw cron get <id>` and Gateway `cron.get`, returning the stored job definition or a clean not-found error.

## Best Possible Solution

Keep this issue as the tracking item and resolve it by reviewing and landing PR #75117 or an equivalent narrow additive implementation that adds `cron.get` to the Gateway registry, read scopes, protocol schema/types/validators, server handler, CLI, agent tool, docs, changelog, and focused tests without changing scheduling, execution, delivery, or existing list/show behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: `cron show` still pages through `cron.list`, and the Gateway registry, scopes, protocol validators, server handlers, docs, and agent tool expose no `cron.get` or `cron get` surface.

## Solution Assessment

Yes. The requested direct read path is the narrow maintainable direction because cron jobs are already id-addressable for mutation and `CronService.getJob(id)` exists internally; the active PR should be reviewed instead of creating duplicate repair work.
