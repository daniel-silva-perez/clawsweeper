## Summary

Keep open. Current main still lacks the requested direct single-job cron read path: the CLI single-job display pages through `cron.list`, the gateway registry/scopes/protocol/server handler expose list/status/add/update/remove/run/runs but not `cron.get`, and the agent cron tool has no `get` action. The related implementation PR #67677 is closed unmerged, so it is useful prior art rather than shipped behavior.

## What This Changes

The issue requests an additive single-job cron inspection path, including `openclaw cron get <id>` and gateway RPC `cron.get`, so callers can fetch one stored cron job definition without listing and filtering the full job set.

## Best Possible Solution

Implement the additive core read API: add `cron.get` to the gateway method registry, read scopes, protocol schema/types/validators, and server handler, backed by `CronService.getJob(id)` with a clean not-found error. Then expose `openclaw cron get <id>` or make `cron show` use the direct RPC, add an agent cron `get` action if agent workflows remain in scope, and update focused tests/docs/changelog without changing scheduling, execution, delivery, or existing list behavior.
