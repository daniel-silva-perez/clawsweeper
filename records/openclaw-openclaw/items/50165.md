## Summary

Keep this issue open. Current main handles the narrower nested-subagent/descendant announce race, but the issue discussion also reports registered or shell-backed background exec/process work outliving the subagent run. Current source and docs still make the agent run terminal event authoritative for subagent task completion, while background exec completion is surfaced through a separate process notification path rather than blocking subagent completion.

## Best Possible Solution

Keep this open for maintainer product/architecture judgment. The best path is to define the contract explicitly: either subagent completion remains agent-run authoritative and `/tasks`/`session_status` must surface active child-session process work clearly, or subagent finalization must defer on owned background exec/process sessions through a generic process-liveness seam with timeout/cancel semantics and regression tests.

## Risks / Open Questions

- Changing subagent completion to wait for all child-session background processes could hang or substantially delay completion announcements unless there is explicit timeout, cancel, and ownership policy.
- Leaving the current contract as-is means users can still see a subagent task as terminal while a registered background exec session from that child session is still running, unless status surfaces make that distinction clear.
- The issue has behavioral reports but no focused lifecycle log or deterministic repro harness, so the next step should include a small regression scenario before changing the contract.
