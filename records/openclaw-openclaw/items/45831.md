## Summary

Keep this PR open. Current main still has only partial attachment-cache hardening: local attachment reads use a custom readLocalBuffer helper instead of the shared readLocalFileSafely path, and temp cache writes still omit explicit 0o600 permissions. The PR remains a plausible core security-hardening candidate, but it needs to be ported onto the current attachment-cache shape and keep the SafeOpenError("too-large") to MediaUnderstandingSkipError("maxBytes") mapping from the review thread.

## Best Possible Solution

Port the PR onto current main by replacing or refactoring readLocalBuffer to use the shared readLocalFileSafely/openLocalFileSafely identity checks while preserving local-root validation and MediaUnderstandingSkipError mappings, add { mode: 0o600 } to the temp write in getPath, add targeted tests for oversized SafeOpenError mapping, identity/path-mismatch behavior, and temp-file mode, then run the focused media-understanding attachment tests plus the normal changed gate before merge.
