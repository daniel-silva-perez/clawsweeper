## Summary

Keep this PR open. Current main still has only partial attachment-cache hardening: local attachment reads go through a custom `readLocalBuffer()` helper instead of the shared `readLocalFileSafely()` path, and temp attachment cache writes still omit an explicit `0o600` mode. The PR remains a plausible core security-hardening candidate, but it needs to be ported to the current attachment-cache implementation and keep the review-thread fix for `SafeOpenError("too-large")` mapping.

## Best Possible Solution

Port the PR onto current main by refactoring `readLocalBuffer()` or its call site to use `readLocalFileSafely()` while preserving local-root checks and media-understanding skip errors, add `{ mode: 0o600 }` to the temp write in `getPath()`, and add targeted tests for shared safe-read identity behavior, oversized-file error mapping, and temp-file mode. Then run the focused media-understanding attachment tests and the changed gate before merge.
