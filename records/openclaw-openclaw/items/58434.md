## Summary

Keep open. Current main does not implement the per-request deny override, while the proposed patch appears coherent and the earlier docs issue was fixed; there is no strong close reason. It needs normal maintainer review, rebase if necessary, and validation rather than ClawSweeper closure.

## What This Changes

The PR adds an optional `tool_deny` array to `/v1/responses`, documents it, and threads it through gateway and embedded agent execution as an extra run-local deny policy.

## Best Possible Solution

Adopt a rebased, tested per-request deny override that is documented as an OpenClaw-specific subtractive policy on `/v1/responses`, while leaving broader session and binding policy design to the related open issues.

## Reproduction Assessment

Yes. On current main, a request body containing `tool_deny` should fail schema validation because the strict request schema lacks that key, and the HTTP handler has no path to forward such an override into agent execution.

## Solution Assessment

Yes. The proposed direction is narrow: parse an optional array, keep it run-local, and apply a deny-only step through the existing group-expanding tool-policy pipeline before stricter sandbox/subagent filters remain in force.
