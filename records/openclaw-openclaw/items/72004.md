## Summary

Keep this PR open. Current main still has the Control UI/WebChat terminal-event ordering that can render both the committed assistant message and the live stream item, and the PR is a focused, low-risk implementation candidate for the linked client-side duplicate-render bug.

## What This Changes

This PR snapshots streamed fallback text, clears Control UI chat stream/run state before appending final or aborted assistant messages, and adds a regression test plus changelog entry.

## Best Possible Solution

Review and land this PR, or an equivalent narrow patch, in the Control UI chat controller. The end state should preserve foreign-run terminal behavior, snapshot streamed fallback text, clear active stream/run state before committing terminal assistant messages, and keep order-sensitive regression coverage plus user-facing changelog credit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start with an active run and non-null `chatStream`, dispatch a `final` or `aborted` payload, and observe current main writes `chatMessages` while `chatStream` is still non-null; the render path then has both a committed message and a live stream item available.

## Solution Assessment

Yes. The PR’s local reorder is the narrowest maintainable fix for this client-render race because it changes only the intermediate mutation order while preserving the final state and streamed fallback semantics; broader transcript or delivery dedupe work would not address this specific Lit batching window.
