## Summary

Keep open. Current main does not contain this PR's reusable bare PowerShell cmdlet allowlist fallback, but the PR is not merge-ready because it can make a bare cmdlet allowlist-satisfied while the Windows node-host execution path still drops the PowerShell host and tries to spawn the cmdlet token directly.

## What This Changes

The PR adds a Windows-only PowerShell cmdlet fallback for exec allowlist matching and allow-always persistence, extracts PowerShell path resolution into infra, and adds cmdlet-specific tests.

## Best Possible Solution

Keep the PR open for maintainer direction: either close it as superseded by exact-command durable approvals for the linked report, or revise it so cmdlet-backed allowlist matches preserve the PowerShell host through execution and are covered by allowlist evaluation, allow-always persistence, and Windows node-host execution tests.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: after the PR, `Get-ChildItem -LiteralPath ...` can become allowlist-satisfied via a synthetic PowerShell match, then `resolveSystemRunExecArgv` rewrites the run to `segments[0].argv`, which is the bare cmdlet token rather than a PowerShell executable invocation.

## Solution Assessment

No. The PR changes allowlist evaluation and persistence but not the execution plan, so the proposed fix is incomplete; the safer alternative is to carry cmdlet-match metadata into execution or otherwise keep the PowerShell wrapper for these matches.
