## Summary

Keep this PR open. Current main still lacks the proposed Swift ignore generalization, large-file guard, exec-surface guard, and additional-boundary wiring; the branch is still an active implementation candidate, but the exec guard has a blocking false-negative in its import matcher before it should become required CI.

## What This Changes

The PR generalizes Swift kit entries in `.gitignore` and adds additional-boundary checks for tracked large files and unjustified `child_process` exec or `shell: true` usage, with package scripts and script-level tests.

## Best Possible Solution

Keep the contribution open, rebase it on current main, refresh the allowlists against the current exec surface, replace the import regex with parser-backed or otherwise statement-safe detection, make `shell: true` detection comment/string-safe, and then run the two targeted script tests plus the changed/additional-boundary lane.

## Reproduction Assessment

Yes. The current-main gap is reproducible by inspecting `.gitignore`, `package.json`, and `scripts/run-additional-boundary-checks.mjs`; the blocking PR bug is reproducible with a semicolonless unrelated import followed by `import { exec } from "node:child_process"`, which the proposed regex can swallow as one match.

## Solution Assessment

No. The hygiene direction is reasonable, but the current exec guard is not the best mergeable solution until import detection is semicolon-independent and `shell: true` scanning ignores comments and strings; a parser-backed check or a small TypeScript-aware scanner would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Detect semicolonless child_process imports:** `scripts/check-no-unjustified-exec.mjs:88`
  - body: The import matcher can start at a previous semicolonless import and consume through the risky `node:child_process` import. In that case `classifyImportSpecifier` inspects only the first brace block, so `import { foo } from "x"\nimport { exec } from "node:child_process"` yields no violation and defeats the required guard.
  - confidence: 0.94
- **[P3] Ignore comments and strings before counting shell: true:** `scripts/check-no-unjustified-exec.mjs:168`
  - body: `findShellTrueViolations` scans raw file text, so a comment or string containing `shell: true` in a non-allowlisted source file would fail CI even though no shell option is used. Parse or strip comments and strings before counting matches.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR adds a security-oriented exec-surface CI guard, but the current import regex can miss new `exec` or `execSync` imports in semicolonless files.

Concerns:

- **[medium] Exec import guard can be bypassed:** `scripts/check-no-unjustified-exec.mjs:88`
  - body: Because `CHILD_PROCESS_IMPORT_STMT_RE` uses `[^;]*?` as the statement boundary, a semicolonless import before a `node:child_process` import can be consumed as one match and the risky binding can be missed. That undermines the guard's security value for new shell execution surfaces.
  - confidence: 0.94
