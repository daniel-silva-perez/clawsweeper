## Summary

Keep this PR open. Current main still has the ambiguous outbound delivery contract, and this branch is the active implementation path, but the proposed gateway cancellation response still needs a contract fix before merge.

## What This Changes

The PR changes outbound delivery to return structured outcome metadata, adds partial-send `DeliveryError` metadata, updates result-consuming callers, and adds delivery/gateway/cron tests for hook cancellation and partial delivery behavior.

## Best Possible Solution

Keep the PR open, fix the gateway cancelled-send response so callers and CLI/JSON formatting receive an explicit cancellation status, confirm the deferred partial-replay scope with maintainers, then land this core contract before the dependent delivery-status PRs consume it.

## Reproduction Assessment

Yes. A focused source-level reproduction is clear: configure a `message_sending` hook to return `cancel: true`; current main reaches the cancellation `continue`, returns an empty result array, and gateway send treats that as `No delivery result`.

## Solution Assessment

No, not merge-ready as-is. The structured delivery outcome is the right direction, but the gateway success branch must preserve a typed cancellation contract instead of returning an `ok` payload that existing callers still interpret as a message-id send.

- **[P2] Preserve a typed cancelled-send response:** `src/gateway/server-methods/send.ts:532-546`
