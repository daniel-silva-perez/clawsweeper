## Summary

Keep this issue open. Current main has adjacent safety controls, including skill allowlists, name/profile-based tool policy, owner-only tools, gateway HTTP/MCP tool filtering, exec/plugin approvals, and before_tool_call governance hooks. It still does not implement the requested capability manifest, persisted grant/revoke state, default-deny sensitive capability taxonomy, pre-model filtering by granted capability, central execution-boundary capability authorization, or durable high-risk capability audit records. The linked @vainplex/openclaw-governance plugin is useful hook-based prior art, but the discussion explicitly treats it as complementary rather than a replacement for a core/plugin-API grant layer.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. The best path is a staged implementation: manifest/skill metadata for required and optional sensitive capabilities, persisted grants and revocations, CLI/UI/doctor diagnostics, capability filtering before tool schemas reach the model, the same capability decision enforced at every execution boundary, and durable audit records for high-risk decisions. Treat @vainplex/openclaw-governance as useful dynamic before_tool_call prior art that can layer on top of static grants, not as the canonical grant system.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policies, owner-only gates, HTTP/MCP filtering, approvals, and hook governance with the requested static capability/grant/audit system.
- A future implementation needs one central authorization decision reused by model-exposed tools, direct HTTP tools, MCP loopback tools, native harness paths, and plugin-owned command paths so pre-exposure filtering cannot be bypassed.
- A default-deny rollout needs staged schema, diagnostics, migration, and operator consent UX so existing trusted skills/plugins are not silently broken.
