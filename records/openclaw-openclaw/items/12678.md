## Summary

Keep this issue open. Current main has adjacent controls for tool allow/deny policy, owner-only tools, gateway HTTP tool deny defaults, operator scopes, and before_tool_call governance, but it still lacks the requested capability manifest, explicit grant/revoke state, default-deny sensitive capability taxonomy, central execution-boundary authorization, and durable high-risk capability audit model for skills/plugins/tools.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. A good staged path is: add manifest/skill metadata for required and optional sensitive capabilities; add persisted grants/revocations with CLI/UI/doctor diagnostics; filter ungranted high-risk tools before model exposure; enforce the same decision at every execution boundary; and write durable audit records for high-risk capability decisions. Treat @vainplex/openclaw-governance as useful dynamic before_tool_call prior art that can layer on top of static grants, not as the canonical grant mechanism.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policies, owner-only gates, sandboxing, HTTP deny defaults, operator scopes, and before_tool_call governance with the requested capability manifest/grant/audit system.
- A future implementation needs one central authorization decision reused by model-exposed tools, direct HTTP tools, MCP tools, and native harness paths so pre-exposure filtering cannot be bypassed.
- A default-deny rollout needs migration, diagnostics, and operator consent UX so existing trusted skills/plugins are not silently broken.
