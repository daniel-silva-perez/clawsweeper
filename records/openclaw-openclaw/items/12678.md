## Summary

Keep this issue open. Current main has meaningful adjacent safeguards, including tool allow/deny policy, per-agent/group/provider filtering, sandbox/exec approval controls, owner-only tool guards, and plugin before_tool_call hooks. It still does not implement the requested manifest-declared sensitive capability taxonomy, persisted grants/revocations, default-deny high-risk authorization before tool exposure, cross-surface execution-boundary checks, or durable audit records for capability decisions. The linked @vainplex/openclaw-governance plugin is useful prior art for call-time governance, but the discussion correctly distinguishes it from the requested core/API permission layer.

## Best Possible Solution

Keep this open as a core/plugin-API hardening item. The best path is staged: add manifest and skill metadata for required/optional sensitive capabilities, add persisted grant/revoke state and diagnostics, filter ungranted high-risk tools before model exposure, enforce the same checks at execution boundaries, and record durable audit decisions. Treat @vainplex/openclaw-governance as valuable dynamic before_tool_call prior art that can layer on top of static grants, not as the canonical grant mechanism.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policy and hook-time governance with the requested manifest/grant/enforcement/audit permission model.
- A future implementation needs explicit security design across model-facing tools, direct HTTP/RPC/MCP/non-LLM invocation paths, and durable audit storage so filtering cannot be bypassed by alternate surfaces.
- Grant storage location, migration behavior, and default-deny rollout need maintainer review to avoid breaking existing trusted plugin and skill deployments.
