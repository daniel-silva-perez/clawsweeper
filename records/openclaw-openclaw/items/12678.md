## Summary

Keep this issue open. Current main has useful adjacent controls, including skill allowlists, name-based tool allow/deny policies, owner-only tools, gateway HTTP tool deny defaults, operator scopes, exec/plugin approvals, and before_tool_call hooks. It still does not implement the requested static capability manifest plus explicit grant/revoke state, default-deny sensitive capability taxonomy, pre-model tool filtering by granted capability, central execution-boundary authorization, or durable high-risk capability audit model. The recent discussion also explicitly distinguishes @vainplex/openclaw-governance as useful hook-based prior art rather than a replacement for a core/plugin-API grant system.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. A good staged path is to add manifest/skill metadata for required and optional sensitive capabilities, add persisted grants and revocations with CLI/UI/doctor diagnostics, filter ungranted high-risk tools before model exposure, enforce the same decision at each execution boundary, and write durable audit records for high-risk capability decisions. Treat @vainplex/openclaw-governance as useful dynamic before_tool_call prior art that can layer on top of static grants, not as the canonical grant mechanism.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policies, owner-only gates, HTTP deny defaults, operator scopes, approvals, and hook governance with the requested static capability/grant/audit system.
- A future implementation needs one central authorization decision reused by model-exposed tools, direct HTTP tools, MCP tools, native harness paths, and plugin-owned commands so pre-exposure filtering cannot be bypassed.
- A default-deny rollout will need migration, diagnostics, and operator consent UX so existing trusted skills/plugins are not silently broken.
