## Summary

Keep this issue open. Current main has adjacent safety controls, including skill allowlists, name/profile-based tool policy, owner-only tools, Gateway HTTP default denies, exec approvals, and before_tool_call hooks, but it still lacks the requested capability manifest, operator grant/revoke state, default-deny sensitive capability taxonomy, pre-model capability filtering, central execution-boundary authorization, and durable high-risk capability audit records.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. A maintainable path is staged: add manifest/skill metadata for required and optional sensitive capabilities, persist grants and revocations, surface CLI/UI/doctor diagnostics, filter ungranted high-risk tools before model exposure, enforce the same decision at every execution boundary, and write durable audit records for high-risk decisions. Treat @vainplex/openclaw-governance as useful dynamic before_tool_call prior art that can layer on top of static grants, not as the canonical grant system.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policies, owner-only gates, HTTP default denies, exec approvals, and hook governance with the requested static capability/grant/audit system.
- A future implementation needs one authorization decision reused by model-exposed tools, direct HTTP tools, MCP/loopback tools, native harness paths, and plugin-owned command paths so pre-exposure filtering cannot be bypassed.
- A default-deny rollout needs staged schema, diagnostics, migration, and operator consent UX so existing trusted skills/plugins are not silently broken.
