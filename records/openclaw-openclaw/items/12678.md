## Summary

Current main has adjacent controls, including name/profile-based tool policies, before_tool_call hooks, exec approvals, plugin install scanning, and a default HTTP /tools/invoke deny list. It still does not implement the requested capability-manifest plus explicit grant/revoke, pre-model default-deny filtering, execution-boundary authorization across tool surfaces, or durable high-risk capability audit records for skills/tools. The @vainplex/openclaw-governance plugin is useful prior art for dynamic call-time governance, but the remaining core/plugin-API capability grant layer is still missing.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. A reviewable path is schema/parser/diagnostics first for required and optional sensitive capabilities, then persisted grants/revocations, CLI/UI/doctor visibility, pre-model filtering after existing tool policy resolution, execution-boundary checks for HTTP/MCP/non-prompt paths, and durable audit records. Coordinate with #12219 for skill manifest shape and #63881 for MCP deny-list parity while treating @vainplex/openclaw-governance as complementary runtime policy.
