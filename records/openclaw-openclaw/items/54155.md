## Summary

Keep open. Current main has useful partial mitigations for the session-store clone path, seqByRun cleanup, stale run-context sweeping, bounded preview reads, diagnostics, and session maintenance, but the broader multi-day Gateway RSS/session-accumulation problem is not proven fixed. A maintainer comment explicitly keeps this open for soak/profiling proof, and current code still has auth-profile structuredClone paths plus full-transcript materialization in several Gateway history/usage paths.

## What This Changes

Requests that long-running Gateway memory usage plateau by reclaiming or bounding accumulated session, cron, and subagent state after pruning and compaction.

## Best Possible Solution

Keep this as the broad Gateway memory tracker until maintainers either split the remaining work into narrower issues or finish the auth-profile clone and hot transcript-read follow-ups, then close only after targeted regression tests plus a representative multi-day Gateway soak/profile run demonstrate stable RSS.
