## Summary

Keep open. The PR fixes the deterministic pre-aborted-signal path, but the cleanup backstop does not fully close the remaining prompt-start race because Pi `AgentSession.prompt()` performs async preflight before creating the internal `Agent` run that `agent.abort()` can cancel.

## What This Changes

This PR adds an embedded-runner pre-prompt abort guard, finally-block Pi Agent cleanup, and a focused abort-before-prompt regression test with test-harness support methods.

## Best Possible Solution

Keep the narrow fix direction, but make the prompt-start boundary atomic with respect to `runAbortController.signal`: recheck after async prep and immediately before each prompt start, or wrap prompt creation in a lazy abortable helper that cannot evaluate `activeSession.prompt()` after the run is aborted. Cleanup should cancel and settle the session-level prompt path when it may have escaped, with a regression test for the abort-after-guard/mid-preflight window.

## Reproduction Assessment

Yes. The linked issue provides a deterministic pre-aborted `AbortSignal` test shape, and current main source confirms the direct `abortable(activeSession.prompt(...))` evaluation order; the remaining PR finding can be reproduced by aborting after the new guard but before `AgentSession.prompt()` creates `Agent.activeRun`.

## Solution Assessment

No. The pre-prompt guard is the right minimal direction for the pre-aborted case, but the added cleanup is not sufficient as a mid-prompt backstop because `agent.abort()` is a no-op until the Pi `Agent` has an active run.
