## Summary

Keep open. The linked bug is valid and current main still has the prompt promise created before `abortable()` can reject; this PR is useful but still leaves the abort-after-guard, before-`Agent.prompt()` preflight window uncancelled.

## What This Changes

The PR modifies the embedded Pi attempt runner to guard prompt submission when the run is already aborted, abort and clear the Pi Agent during abort/cleanup, and adds abort-before-prompt regression coverage with test harness support.

## Best Possible Solution

Make the actual Pi prompt-start boundary abort-aware: check the run abort signal immediately before Pi calls `Agent.prompt()` and fail without starting `activeRun` when cancellation has arrived, then keep focused coverage for already-aborted, abort-before-activeRun, and abort-after-activeRun cases.

## Reproduction Assessment

Yes. #74859 gives a deterministic pre-aborted-signal reproduction, and the remaining PR gap can be reproduced by delaying `AgentSession.prompt()` preflight and aborting after this PR's guard but before `Agent.prompt()` creates `activeRun`.

## Solution Assessment

No. The current PR is a useful partial fix for the already-aborted path, but the guard plus `agent.abort()` backstop is not sufficient because `agent.abort()` is a no-op before Pi creates `activeRun`.
