## Summary

Keep this PR open: it targets a real current-main bug in `/hooks/agent`, but the branch is not merge-ready because it conflicts with current hook trust sanitization and the current cron announce message-tool policy, and it still needs docs/changelog cleanup plus maintainer confirmation of the hook account-selection trust model.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization/error handling, adjusts shared hook message-tool policy, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased gateway fix that preserves and trims `accountId` through `HookAgentPayload`, replay scope, and `CronJob.delivery`, keeps sanitized hook metadata in prompt context, preserves current announce message-tool behavior, documents the hook `accountId` authority model, and credits the contributor in the changelog.

## Reproduction Assessment

Yes. Current-main code inspection gives a high-confidence reproduction path: send `/hooks/agent` with `accountId`, and the normalizer cannot return that field while dispatch cannot place it into `job.delivery`.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this PR is not the best merge-ready form because it regresses hook-name sanitization, rewrites cron message-tool policy that current main has since changed, and omits required docs/changelog follow-through.
