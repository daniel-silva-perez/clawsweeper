## Summary

Keep open. The linked bug is valid on current main, but this PR should not merge as-is because the branch is stale against current cron message-tool policy and still has hook trust/documentation issues that need maintainer review.

## What This Changes

The PR adds accountId parsing and forwarding for /hooks/agent, routes hook dispatch through cron normalization, adjusts shared hook message-tool policy, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased gateway-owned fix that trims and forwards accountId, includes it in hook replay scope, preserves sanitized hook metadata in cron prompt context, keeps current main's announce message-tool behavior, documents the request field and trust model, and credits the contributor in the changelog.

## Reproduction Assessment

Yes. Current-main code inspection gives a high-confidence reproduction path: send /hooks/agent with accountId and the normalizer cannot return that field, while dispatch cannot place it into job.delivery.

## Solution Assessment

No. Forwarding accountId is the right direction, but this PR is not the best merge-ready form because it regresses current hook-name sanitization in prompt metadata and rewrites cron message-tool policy that current main has since changed.
