## Summary

Keep this PR open: it targets a real remaining `/hooks/agent` multi-account delivery bug, and the linked #43866 issue should stay paired with this implementation until a fix lands. The branch is not merge-ready because it needs a rebase onto the active hook request handler plus maintainer review for hook account-selection authority, sanitized hook metadata, and current cron announce message-tool policy.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization, adjusts isolated cron message-tool policy, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased, narrow gateway fix that trims and preserves `accountId` through `HookAgentPayload`, direct and mapped hook replay scopes, and `CronJob.delivery`; keeps sanitized hook names in prompt-visible metadata; preserves current announce message-tool behavior unless a separate maintainer decision changes it; and updates public webhook docs plus changelog credit.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: send `/hooks/agent` with `accountId`, and the normalizer plus dispatch builder cannot carry that field into `job.delivery`.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this PR is not the best merge-ready form because it targets stale handler code, can reintroduce raw hook names into prompt context, and changes current announce message-tool policy beyond the accountId fix.
