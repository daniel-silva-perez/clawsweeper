## Summary

Keep this PR open: it targets a real current-main bug, but the branch is not merge-ready. Current main still drops `accountId` from `/hooks/agent`, while this PR is stale against the active hook handler and introduces prompt-metadata and cron message-tool policy regressions that need maintainer review before merge.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization, changes isolated cron message-tool policy, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow gateway fix that trims and preserves `accountId` through `HookAgentPayload`, direct hook replay scope, mapped hook scope if supported, and `CronJob.delivery`; keeps sanitized hook names in prompt-visible metadata; preserves current announce message-tool behavior unless handled in a separate maintainer-approved change; and updates public webhook docs plus changelog credit.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: send `/hooks/agent` with `accountId`, and current main cannot carry that field through `normalizeAgentPayload()` or the hand-built hook `delivery` object into cron delivery.

## Solution Assessment

No. Forwarding `accountId` is the right fix direction, but this PR is not the best current implementation because it is stale against the active handler, regresses sanitized hook job names, and changes announce message-tool policy beyond the bug boundary.
