## Summary

Keep open. Current main still drops `accountId` before `/hooks/agent` reaches cron delivery, so the linked bug is real, but this PR is not ready to merge because the patch no longer applies cleanly to current main and still has a prompt-safety regression plus docs/changelog follow-up.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization with replay/error-handling changes, adjusts cron message-tool policy, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a rebased gateway-owned fix that preserves sanitized hook metadata, trims and forwards `accountId`, includes it in hook replay scope, propagates it into `CronDelivery.accountId`, documents the request field and trust model, and aligns with current main's cron message-tool policy rather than reviving stale interfaces.

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow changes, but it weakens prompt-safety around hook names and adds hook-driven account selection that needs documented policy.

Concerns:

- **[high] Raw hook name reaches prompt context:** `src/gateway/server/hooks.ts:60`
  - body: The patch passes caller-controlled `value.name` into `CronJob.name`; external hook prompts include that job name in a `Task:` line before the untrusted-content wrapper, bypassing current main's hook-name sanitization.
