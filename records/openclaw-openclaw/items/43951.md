## Summary

Keep open: current main still lacks `accountId` on the direct hook payload, replay scope, and hand-built delivery path, so the reported bug remains real. The PR is useful, but it is not ready to merge because it drops current hook-name sanitization into prompt context, is stale against the current cron message-tool policy, and still needs docs/changelog/security-policy cleanup.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization, adjusts cron message-tool policy, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a rebased gateway-owned fix that trims and forwards `accountId`, includes it in hook replay scope, preserves sanitized hook metadata in cron prompt context, keeps current main's message-tool policy semantics, documents the new request field and trust model, and closes #43866 after merge.
