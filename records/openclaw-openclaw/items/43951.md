## Summary

Keep this PR open: it targets a real `/hooks/agent` multi-account delivery bug that still exists on current `main`, but the branch is stale against the active hook request handler and introduces prompt-metadata/account-selection concerns that need maintainer review before merge.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization, changes isolated cron message-tool policy, updates hook replay/error handling and tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased, narrow gateway fix that preserves a trimmed `accountId` from direct `/hooks/agent` payloads through the active request handler, replay scope, dispatch payload, and `CronJob.delivery`; keeps sanitized hook names in prompt-visible job metadata; preserves current announce message-tool policy unless intentionally changed in a separate review; and updates public webhook docs plus changelog attribution.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: send `/hooks/agent` with `accountId` on current `main`; `normalizeAgentPayload()` drops the field and `dispatchAgentHook()` builds `delivery` without it, so `resolveDeliveryTarget()` cannot use the requested account.

## Solution Assessment

No. Forwarding `accountId` is the right fix direction, but this PR is not currently the best implementation because it updates an older handler location, regresses sanitized job names, changes announce message-tool policy beyond the accountId bug boundary, and omits docs/changelog policy follow-through.
