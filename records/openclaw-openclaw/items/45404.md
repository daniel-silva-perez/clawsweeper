## Summary

Keep this issue open. Current main still canonicalizes the managed browser profile to `openclaw` and preserves `clawd` only as a config compatibility alias, but there is no browser-owned doctor/repair/docs path that detects and safely archives unconfigured legacy `browser/clawd/user-data` residue.

## What This Changes

The issue requests a low-priority doctor or repair cleanup flow for stale `~/.openclaw/browser/clawd/user-data` profile data left behind by rename-era installs.

## Best Possible Solution

Implement a browser-owned doctor/repair check that scans for stale unconfigured `browser/clawd/user-data` residue when the canonical managed profile is `openclaw`, reports a clear warning, and offers an explicit safe archive/trash move. Keep the `clawd` config-driver alias for compatibility and document the canonical managed profile path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: place an unconfigured legacy `CONFIG_DIR/browser/clawd/user-data` directory beside the canonical `openclaw` profile, then current doctor/browser doctor/delete-profile paths will not detect or archive it because they only inspect configured profiles and selected runtime state.

## Solution Assessment

Yes. The best fix is the issue's narrow doctor/repair-style warning plus explicit archival flow; automatic deletion or a core startup migration would create unnecessary data-loss and ownership risk.
