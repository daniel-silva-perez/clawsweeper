## Summary

Keep this open. Current main has canonicalized the managed browser profile to `openclaw` and preserves `clawd` only as a config compatibility alias, but there is still no browser-owned doctor/repair/docs path that detects and safely archives unconfigured legacy `browser/clawd/user-data` residue. The issue discussion’s source pass remains consistent with current code.

## What This Changes

Request to detect legacy managed browser profile residue under `browser/clawd/user-data` after the rename, warn before cleanup, offer safe archival, and document the canonical `openclaw` managed profile path.

## Best Possible Solution

Implement a browser-owned doctor/repair check that scans for stale unconfigured `browser/clawd/user-data` residue when the canonical managed profile is `openclaw`, reports a clear warning, and offers an explicit safe archive/trash move. Keep it out of startup/load-time core migrations, document the canonical managed profile path, and preserve the `clawd` config-driver alias for compatibility.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main can launch/use `openclaw` profiles, but doctor and browser cleanup paths never scan an unconfigured `browser/clawd/user-data` directory, and `deleteProfile` rejects unknown profiles before cleanup.

## Solution Assessment

Yes. The narrow maintainable solution is a browser plugin doctor/repair path with explicit archival and docs; automatic deletion or a core startup migration would carry unnecessary data-loss and ownership risk.
