## Summary

Keep this issue open. Current main still canonicalizes managed browser state to the `openclaw` profile but does not detect or safely archive leftover on-disk `browser/clawd/user-data` residue; existing migration coverage is limited to config keys and Chrome MCP readiness.

## What This Changes

The issue asks for a doctor/repair/docs path that detects and safely archives legacy managed Brave profile data left under `browser/clawd/user-data` after the OpenClaw rename.

## Best Possible Solution

Implement a browser-owned doctor/repair check that scans for stale unconfigured `CONFIG_DIR/browser/clawd/user-data` only when the active managed profile is `openclaw`, reports a clear warning, and offers explicit safe archival or Trash movement. Keep the legacy `clawd` driver alias for config compatibility and document the canonical managed profile path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an unconfigured `CONFIG_DIR/browser/clawd/user-data` beside the canonical `openclaw` profile, then current browser doctor and delete-profile paths will not detect or archive it because they only inspect configured profiles and selected runtime state.

## Solution Assessment

Yes. A doctor/repair-style warning plus explicit archival flow is the narrowest maintainable solution; startup migration or silent deletion would create unnecessary data-loss and ownership risk.
