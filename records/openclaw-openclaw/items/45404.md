## Summary

Keep this issue open. Current main still canonicalizes the managed browser profile to `openclaw`, but the browser doctor/reset/delete paths do not detect or safely archive an unconfigured legacy `browser/clawd/user-data` directory left behind by rename-era installs.

## What This Changes

Request a doctor or repair flow that detects legacy `clawd` managed browser profile residue after the rename and safely archives it when the canonical `openclaw` profile is active.

## Best Possible Solution

Implement a browser-owned doctor/repair check that scans for stale unconfigured `CONFIG_DIR/browser/clawd/user-data` only when the canonical managed profile is `openclaw`, reports a clear warning, and offers explicit safe archival or Trash movement. Preserve the legacy `clawd` driver alias for config compatibility and document the canonical managed profile path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: place an unconfigured `CONFIG_DIR/browser/clawd/user-data` beside the canonical `openclaw` profile, then current browser doctor and profile cleanup paths will not report or archive it because they only inspect configured/default profiles and selected runtime state.

## Solution Assessment

Yes. A doctor/repair warning plus explicit archival flow is the narrowest maintainable solution; startup migration or silent deletion would add unnecessary data-loss risk for local browser state.
