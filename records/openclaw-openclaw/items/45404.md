## Summary

Current main still uses `openclaw` as the canonical managed browser profile and still treats `clawd` only as a legacy driver/config alias, but there is no doctor, repair, migration, or docs path that detects and safely archives an unconfigured `~/.openclaw/browser/clawd/user-data` residue directory. The GMTekAI source-pass comment and the prior ClawSweeper comment remain consistent with current main at `1b25dcf57a9fca5d00cf275046e20a2e3ea3730c`, so this is a valid low-priority cleanup gap rather than a close candidate.

## Best Possible Solution

Keep this issue open and implement a browser-owned doctor/repair check: scan for `~/.openclaw/browser/clawd/user-data` when the active managed profile is `openclaw`, report it as legacy residue, and offer an explicit interactive archive/trash move rather than deletion. Update browser/doctor docs to document the canonical managed profile path and the safe cleanup behavior.
