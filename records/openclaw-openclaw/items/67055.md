## Summary

Keep open. Current main still has the Google Chat off-mode bug at the implicit inbound thread source and typing placeholder, so the PR is addressing a real problem. The branch is not correct as-is because it also guards final delivery on account config, which drops explicit reply targets that the shared reply-threading contract intentionally preserves when `replyToMode` is `off`; it also overlaps the broader open Google Chat threading work in #74235.

## What This Changes

The PR adds Google Chat `replyToMode: "off"` guards around inbound thread propagation, typing placeholder sends, final text/media sends, and adds config-resolution tests for channel-level and per-account off mode.

## Best Possible Solution

Revise or fold the branch into the current Google Chat threading work: suppress only implicit inbound `ReplyToId`/`ReplyToIdFull` and typing-placeholder thread propagation when the resolved Google Chat account has `replyToMode: "off"`, keep final delivery honoring `payload.replyToId`, and add a focused monitor regression for off-mode DM replies plus explicit reply preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve explicit reply targets in off mode:** `extensions/googlechat/src/monitor.ts:437`
  - body: `payload.replyToId` has already passed through the shared reply-threading filter, which preserves explicit `[[reply_to:*]]` and `[[reply_to_current]]` directives even when `replyToMode` is `off`. Guarding final sends on `account.config.replyToMode` drops those explicit targets, and the same pattern appears in the media path, so intentional Google Chat thread replies are forced flat. Filter the implicit inbound and typing thread sources instead and let final delivery use `payload.replyToId`.
  - confidence: 0.88
