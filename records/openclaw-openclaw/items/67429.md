## Summary

Keep open. Current main still cannot clear cron payload model, fallbacks, or thinking overrides with null, so the PR is not obsolete; however, the current patch has a blocking create-path regression because it preserves null in the shared normalizer used by both cron.update and cron.add.

## What This Changes

The PR adds null-as-clear handling for cron.update payload model, fallbacks, and thinking overrides across cron types, normalization, service merge logic, gateway schema, and regression tests.

## Best Possible Solution

Land a revised version that defines null-clear as a cron.update patch-only contract, deletes the three stored payload overrides when patch values are null, preserves cron.add create-time optional-null tolerance, and covers both the normalize-to-apply path and create compatibility in regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current cron.update normalizes the patch before validation, current normalization drops these nulls, and current merge ignores null for the requested fields; the PR’s remaining regression is also reproducible from the shared normalizeCronJobCreate/normalizeCronJobPatch path.

## Solution Assessment

No. The requested direction is the narrow maintainable fix, but the current implementation is not the best way because it changes shared create normalization; a safer fix should gate nullable clears to patch normalization only.
