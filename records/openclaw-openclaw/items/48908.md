## Summary

Keep this PR open. Current main still does not retry Telegram HTTP 421 / Misdirected Request responses in either the shared channel retry regex or the strict Telegram send path, while the PR is a focused implementation candidate for the closed linked bug #48892.

## Best Possible Solution

Keep this PR open for maintainer review and either land it or an equivalent fix. The implementation should add 421 / Misdirected Request handling to the default channel retry path and, more importantly, teach extensions/telegram/src/network-errors.ts that Telegram 421 responses are safe for strict non-idempotent send retries, with regression tests for both paths and a retry-doc update if the behavior ships.
