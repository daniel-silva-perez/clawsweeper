## Summary

Keep open. Current main still implements per-run fallback/cooldown probing and sticky auto-fallback session overrides, but it does not implement the requested background health-probe recovery loop, recovered-primary state for future work, configurable recovery interval, or coordinated recovery notification behavior.

## What This Changes

The issue requests health-probe-based automatic recovery from a fallback model back to the primary model for new sessions, subagents, and cron work while keeping existing sessions pinned to their fallback model.

## Best Possible Solution

Design a core fallback-recovery state machine that reuses existing auth/model probe contracts, records primary availability for future work, keeps active sessions pinned to their chosen fallback, exposes documented configuration, and coordinates user-visible recovery notifications with the companion notification issue.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request involving provider outage timing. Static inspection gives high confidence that current main lacks the requested background recovery loop and future-work routing state.

## Solution Assessment

Unclear until maintainer design is done. The health-probe direction is plausible, but the maintainable solution should add an explicit recovery state/routing contract rather than clearing sticky session overrides or blindly switching sessions back.
