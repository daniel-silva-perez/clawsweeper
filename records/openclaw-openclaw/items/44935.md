## Summary

Keep open. Current main still has per-run fallback, cooldown probing, sticky auto-fallback session overrides, and fallback lifecycle/status notices, but it does not implement the requested background recovery loop, durable recovered-primary state for future work, configurable recovery interval, or recovery notification behavior tied to the companion notification request.

## What This Changes

The issue requests health-probe-based automatic recovery from fallback back to the primary model for new work while keeping existing sessions pinned to the fallback.

## Best Possible Solution

Design a core fallback-recovery state machine that reuses existing model/auth probe contracts, records primary availability for future work, routes only new sessions/subagents/cron runs back to the recovered primary, keeps active sessions pinned to their chosen fallback, documents config defaults, and coordinates recovery notifications with #44941.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request involving provider outage timing. Static inspection gives high confidence that current main lacks the requested background recovery loop and future-work routing state.

## Solution Assessment

Unclear until maintainer design is done. The health-probe direction is plausible, but the maintainable fix should add an explicit recovery state/routing contract rather than clearing sticky session overrides or blindly switching active sessions back.
