## Summary

Keep open: current main has sticky auto fallback state and inline cooldown probes, but it still lacks the requested configurable background recovery loop that health-probes the primary, marks it healthy for future work, and coordinates recovery notification behavior.

## What This Changes

The issue requests health-probe-based automatic recovery from fallback models to the primary model for new sessions, subagents, and scheduled work while keeping existing sessions sticky.

## Best Possible Solution

Design a core fallback-recovery state machine that reuses existing model/auth probe contracts, leaves active sessions pinned to their selected fallback, exposes a documented configuration surface, routes only new sessions/subagents/cron work back to the primary after a successful probe, and coordinates user-visible recovery messaging with #44941.
