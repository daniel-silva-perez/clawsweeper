## Summary

Keep open. Current main has per-run fallback/cooldown probing and sticky auto fallback overrides, but it still lacks the requested background health-probe recovery loop, persisted recovered-primary state for future work, configurable recovery interval, and coordinated recovery notifications with #44941.

## What This Changes

The issue requests health-probe-based automatic recovery from fallback to the primary model for future sessions, subagents, and cron jobs while keeping existing sessions pinned to their fallback model.

## Best Possible Solution

Design a core fallback-recovery state machine that reuses existing model/auth probe contracts, records recovered-primary availability for future work, keeps active sessions pinned to their chosen fallback, exposes documented configuration, and coordinates user-visible recovery notifications with #44941.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request involving provider outage timing. Static inspection gives high confidence that current main lacks the requested background recovery loop and future-work routing state.

## Solution Assessment

Unclear for implementation until maintainer design is done. The requested health-probe direction is plausible, but the safest solution needs an explicit state machine and routing contract rather than clearing sticky session overrides blindly.
