## Summary

Keep open: current main has sticky auto fallback state and inline cooldown probes, but it still does not implement the requested configurable background recovery loop that health-probes the primary, records recovered availability for future work, and coordinates recovery messaging.

## What This Changes

The issue requests health-probe-based recovery from fallback to the primary model for new sessions, subagents, and scheduled work while keeping existing sessions pinned to their active fallback model.

## Best Possible Solution

Design a core fallback-recovery state machine that reuses existing model/auth probe contracts, leaves active sessions pinned to their selected fallback, exposes documented configuration, routes only new sessions/subagents/cron work back to the primary after a successful probe, and coordinates user-visible recovery messaging with #44941.
