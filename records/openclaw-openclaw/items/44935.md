## Summary

Keep open. Current main has per-run cooldown probes and sticky auto fallback overrides, but it still does not implement the requested background recovery loop, configurable probe interval, persisted recovered-primary signal, or routing change for new sessions, subagents, and cron runs.

## What This Changes

The issue requests health-probe-based auto-recovery from fallback models back to a recovered primary for new work while keeping existing sessions pinned to their fallback model.

## Best Possible Solution

Design a core fallback-recovery state machine that reuses the existing model/auth probe contracts, records recovered-primary availability for future work, keeps active sessions pinned to their selected fallback, exposes documented configuration, and coordinates user-visible recovery notifications with #44941.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run because this is primarily a feature request involving provider outage timing. Static inspection gives high confidence that the requested recovery mechanism is missing from current main.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current path. A narrow automated patch would be risky because the requested behavior spans background scheduling, persisted recovery state, routing semantics, config, and notification policy.
