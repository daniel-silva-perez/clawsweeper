## Summary

Keep open. Current main still lacks a typed per-agent delivery default, and message-tool target inference still depends on explicit arguments or already-captured trusted run/session routing context rather than an agent-level fallback.

## What This Changes

Add an optional per-agent default delivery target/channel so programmatic, webchat-routed, cron, and cross-session agent runs can send messages without reconstructing long channel IDs.

## Best Possible Solution

Define a typed, documented agents.list[].delivery contract using DeliveryContext-style fields such as channel, to, accountId, and threadId; validate it in config/schema/help, carry it through resolved agent config, and apply it only as the lowest-precedence trusted fallback after explicit message args and per-run/session deliveryContext.

## Risks / Open Questions

- A broad default target needs explicit precedence rules so it cannot silently route messages to the wrong channel, account, DM/group, or thread.
- A channel/target-only shape may be too narrow for existing DeliveryContext semantics because accountId and threadId matter for several channel surfaces.
- Cron defaults in #9155 overlap conceptually but would not satisfy fresh programmatic or cross-session agent runs unless a separate agent-level contract is designed.
