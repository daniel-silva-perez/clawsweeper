## Summary

Keep this issue open. Current main still does not expose a typed per-agent delivery default, and the message tool target fallback still depends on explicit arguments or trusted run/session routing context rather than an agent-level default target/channel. Recent deliveryContext work preserves or seeds known routes, but it does not solve fresh programmatic or cross-session runs that have no delivery target to seed.

## Best Possible Solution

Keep this issue open for a core product/design fix. Add a typed, documented per-agent delivery default, preferably using DeliveryContext-style fields such as channel, to, accountId, and threadId under agents.list[].delivery rather than only defaultTarget/defaultChannel strings. Validate it in the config schema, carry it through resolveAgentConfig, and apply it as the lowest-precedence trusted fallback after explicit message args and per-run/session deliveryContext. Add focused tests for message tool normalization, sessions_send, webchat-routed sessions, cron-created runs, and normal channel-triggered sessions, with clear precedence rules to avoid accidental cross-channel delivery.

## Risks / Open Questions

- A broad default target without clear precedence could silently route messages to the wrong channel, account, DM/group, or thread in multi-channel deployments.
- The nearby session deliveryContext and cron delivery-default work can look similar, but they only help when a route is already captured or explicitly supplied; conflating them would leave the reported fresh-programmatic-run case unsolved.
