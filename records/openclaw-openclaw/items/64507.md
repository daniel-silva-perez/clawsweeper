## Summary

Keep open. Current main still lacks a typed per-agent delivery default, and targetless message-tool sends still depend on explicit arguments or already-captured trusted route context rather than an agent-level fallback. The two existing ClawSweeper comments remain accurate against `68912111cfeafa72b7c62eb6f2ebb82fa5e10a3b`.

## What This Changes

The issue requests a per-agent default delivery target and channel so programmatic, webchat-routed, or cross-session agent runs can send messages without repeating long target IDs.

## Best Possible Solution

Define a typed, documented `agents.list[].delivery` contract using DeliveryContext-style fields such as `channel`, `to`, `accountId`, and `threadId`; validate it in config/schema/help/docs, carry it through resolved agent config, and apply it only as the lowest-precedence trusted fallback after explicit message args and per-run/session delivery context.
