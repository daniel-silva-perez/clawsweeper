## Summary

Keep open. Current main still does not provide a typed per-agent delivery default, and targetless message sends still depend on explicit tool arguments or already-captured trusted route context rather than an agent-level fallback.

## What This Changes

The issue requests an agent-level default delivery target/channel so cross-session, webchat-routed, and scheduled agent runs can send messages without reconstructing long channel target IDs.

## Best Possible Solution

Define a typed, documented agents.list[].delivery contract using DeliveryContext-style fields such as channel, to, accountId, and threadId; validate it in config/schema/help/docs, carry it through resolved agent config, and apply it only as the lowest-precedence trusted fallback after explicit message arguments and per-run/session delivery context.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: agents.list[].delivery is not in the strict agent schema, and a targetless message send still throws unless toolContext.currentChannelId is populated from existing route context.

## Solution Assessment

No. The issue's minimal fallback sketch is not the best final shape because it omits accountId/threadId and precedence rules; a DeliveryContext-style contract with explicit trust boundaries is safer and more maintainable.
