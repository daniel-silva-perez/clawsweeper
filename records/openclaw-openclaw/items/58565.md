## Summary

Keep open. Current main still does not implement the requested dead-key policy: ambiguous 401/403 payloads such as `invalid_api_key` stay in the `auth` lane, and `auth` is persisted as a cooldown rather than a disabled credential state. The related merged #60404 work shortened `auth_permanent` backoff, but it did not make ordinary dead-key 401s operator-disabled or propagate provider `Retry-After` into auth-profile cooldown timing.

## What This Changes

The issue asks auth-profile failover to distinguish dead-key 401 authentication failures from transient 429/529 responses, disabling invalid credentials while using transient cooldowns for rate limits and overloads.

## Best Possible Solution

Define a conservative auth-health policy that promotes only high-confidence dead credentials out of transient cooldown, preserves retryable handling for ambiguous provider incidents, and aligns classifier coverage, auth-profile persistence, doctor/status guidance, model fallback, and docs.
