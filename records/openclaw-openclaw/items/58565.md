## Summary

Keep open. Current main has partial auth-profile failover differentiation, but the central dead-key behavior remains unresolved: generic HTTP 401/authentication failures still classify as `auth` and are persisted as a cooldown, while only high-confidence `auth_permanent` strings use the disabled lane. The related open reports cover neighboring failover issues, not this exact 401 dead-key policy.

## What This Changes

The issue requests error-specific auth-profile failover so dead credentials are skipped differently from transient 429/529 throttling/overload and billing failures.

## Best Possible Solution

Keep this open and design a conservative failover policy that promotes high-confidence dead credentials, such as provider `authentication_error`/invalid or expired API-key 401s, out of the transient cooldown lane while preserving short rotation/backoff behavior for 429/529 and the existing billing lane. The implementation should align classifier tests, auth-profile state tests, doctor/status messaging, and docs before changing scheduler-facing behavior.
