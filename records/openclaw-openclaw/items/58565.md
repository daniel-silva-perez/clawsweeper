## Summary

Keep open. Current main has typed auth-profile failure reasons and an `auth_permanent` disabled lane, but generic HTTP 401 / `invalid_api_key` still classifies as `auth`, and `auth` still goes through the regular cooldown path rather than an operator-cleared disabled state. 429 and 529 are also still handled through fixed cooldown/profile-rotation controls rather than a Retry-After-derived 429 cooldown and distinct short 529 policy. Related reports cover adjacent failover/backoff issues but do not supersede this exact request.

## Best Possible Solution

Keep #58565 open as the focused tracker for error-specific auth-profile failover. The best fix is to make provider/auth error classification and `markAuthProfileFailure` carry enough structured data to distinguish dead-key 401s from ambiguous auth, use Retry-After when setting 429 profile cooldowns, keep 529 on a short overloaded path with prompt/assistant coverage, and update docs/config help for the resulting operator-visible states. PR #49800 may be relevant for the overloaded slice but does not replace this issue.
