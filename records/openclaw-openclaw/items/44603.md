## Summary

Current `main` does not implement the requested global PTY default. The exec config schema and exported config/default types still omit `pty`, the runtime still derives PTY mode only from the per-call parameter, and the related implementation PR was closed unmerged for PR-slot management rather than because the work was obsolete.

## What This Changes

The issue requests a global `tools.exec.pty` configuration option so exec commands can default to PTY mode and stream long-running command output without relying on each tool call to pass `pty: true`.

## Best Possible Solution

Add a typed, documented optional `tools.exec.pty` default that preserves explicit per-call override semantics, keeps sandbox behavior unchanged, and covers both direct local execution and the gateway approval execution path with focused config and exec tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current `main`: `tools.exec` is a strict config object without `pty`, and exec runtime code uses only `params.pty`; no live command run was needed for this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing exec-default plumbing rather than changing PTY implementation; the prior closed PR is a useful starting point but needs the gateway approval path, tests, docs, generated config checks, and changelog coverage.
