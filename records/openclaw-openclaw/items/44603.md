## Summary

Current main still only supports `pty` as a per-call exec argument. The requested `tools.exec.pty` global/default config is not present in the config schema, generated schema, config type, exec defaults type, default propagation path, or docs. The related implementation PR is closed and unmerged in the provided context, so this issue still tracks real remaining work.

## Best Possible Solution

Keep this issue open. The best fix is a focused exec-config change: add optional `pty` to `tools.exec` and per-agent exec defaults if that scope is intended, propagate it through `ExecToolConfig`, `ExecToolDefaults`, `resolveExecConfig`, and `createLazyExecTool`, compute PTY mode as per-call override first then configured default, keep sandbox behavior explicit, and cover it with config-schema/default-propagation/runtime tests plus docs updates.

## Risks / Open Questions

- A global PTY default changes command I/O behavior across many exec calls, so an implementation should preserve explicit per-call override precedence and the current sandbox-forces-pipes behavior unless maintainers intentionally change that contract.
- Because config schemas are strict and generated metadata backs docs/UI, adding this requires updating source schema/types, generated schema/help/labels as applicable, docs, and regression tests rather than only changing `bash-tools.exec.ts`.
