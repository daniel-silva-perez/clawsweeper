## Summary

Keep open. Current main still matches the reported routing bug: main-session cron system events force `target: "last"`, the gateway adapter shallow-merges that over heartbeat defaults without clearing inherited destination fields, and outbound heartbeat routing treats inherited `heartbeat.to` as explicit before falling back to the session route. An open implementation PR, #73968, has a closing reference to this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Ensure `sessionTarget: "main"` cron `systemEvent` heartbeat wakes bound to a Telegram forum topic deliver replies back to the bound topic session instead of inheriting a global DM `heartbeat.to`.

## Best Possible Solution

Keep this issue open and land #73968 after it covers the queued wake path, or land an equivalent narrow fix that clears inherited explicit destination fields for cron-forced `target: "last"` main/systemEvent wakes while preserving truly explicit heartbeat destinations.

## Reproduction Assessment

Yes. The issue body provides concrete config, cron payload, session key, and observed logs, and the current source statically reproduces the merge and delivery-resolution path without requiring a live Telegram run.

## Solution Assessment

Yes for the requested direction: the maintainable fix is to clear or distinguish inherited heartbeat destination defaults at the cron-forced `target: "last"` boundary. The linked PR should ensure the same behavior for queued wakes, not only the direct gateway wrapper.
