## Summary

Keep open. Current main still matches the reported routing bug: cron-forced `target: "last"` is merged over resolved heartbeat defaults without clearing inherited explicit destinations, and outbound heartbeat routing treats that inherited `to` as explicit before session-bound `last` routing. There is also an open implementation PR (#73968) with a closing reference to this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Ensure `sessionTarget: "main"` cron `systemEvent` heartbeat wakes bound to a Telegram forum topic reply to the bound session/topic route instead of inheriting a global `heartbeat.to` DM default.

## Best Possible Solution

Keep this issue open and land #73968 or an equivalent narrow fix that clears inherited explicit destination fields for cron-forced `target: "last"` main/systemEvent wakes, with regression coverage proving a Telegram forum-topic session stays on the bound group/topic route when global heartbeat defaults include a DM `to`.

## Reproduction Assessment

Yes. The issue body provides a concrete config, cron payload, session key, and observed route leak, and the current source statically reproduces the merge and delivery-resolution path without needing a live Telegram run.

## Solution Assessment

Yes. The narrow maintainable solution is to fix cron/systemEvent wake inheritance or explicitly distinguish inherited defaults from same-call explicit destinations; changing general heartbeat `target: "last"` semantics globally would be broader and riskier.
