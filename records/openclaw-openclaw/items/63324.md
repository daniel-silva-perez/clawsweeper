## Summary

Keep open. The behavior remains unfixed on current main, and this PR is useful but stale/incomplete: it covers the async write stat catch and an old sync helper, while current sync reads route through `src/pairing/allow-from-store-file.ts`, where non-`ENOENT` `statSync` and `readFileSync` errors still return empty allowlists. #51556 is related sync-only work and #63423 is the broader cleanup tracker, so this PR should be repaired rather than closed or superseded.

## What This Changes

This PR changes pairing allowlist stat handling so async write and sync read helpers rethrow non-`ENOENT` filesystem errors instead of treating them as missing allowlists.

## Best Possible Solution

Port the useful source work to current main: keep the async write non-`ENOENT` rethrow in `writeAllowFromState`, apply sync non-`ENOENT` rethrows inside `readAllowFromFileSyncWithExists` for both `statSync` and `readFileSync`, add focused regression tests, and leave the broad empty-catch cleanup to #63423.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by mocking `fs.promises.stat` to throw `EACCES` after an allowlist write, and mocking `fs.statSync` or `fs.readFileSync` to throw `EACCES` in the sync reader; current main swallows those errors in the inspected code paths.

## Solution Assessment

No as-is. The proposed behavior is the right direction, but the sync hunk targets the old `pairing-store.ts` implementation instead of the current shared `allow-from-store-file.ts` helper, and regression tests are still needed.
