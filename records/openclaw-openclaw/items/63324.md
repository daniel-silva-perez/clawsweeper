## Summary

Keep open. Current main still contains the pairing allowlist error-swallowing behavior, and this PR remains relevant but stale: it targets the old helper layout while the sync reader now lives in the shared allow-from store file. Related open items do not fully supersede it because #51556 covers the sync reader only and #63423 is a broader cleanup umbrella.

## What This Changes

The PR changes pairing allowlist stat handling so async write and sync read helpers rethrow non-ENOENT stat errors instead of treating them as missing allowlists.

## Best Possible Solution

Update the PR for current main by preserving the async write non-ENOENT rethrow in src/pairing/pairing-store.ts, applying sync non-ENOENT rethrows in src/pairing/allow-from-store-file.ts for both statSync and readFileSync, and adding focused regression coverage for those failure modes. Keep #63423 as the broader cleanup tracker rather than using it to replace this pairing-specific fix.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: mock fs.promises.stat to throw EACCES after an allowFrom write, and mock fs.statSync or fs.readFileSync to throw EACCES in the sync reader; current main will swallow those errors instead of surfacing them.

## Solution Assessment

No as-is. The proposed behavior is the right direction, but the PR needs to be rebased and moved onto the current shared helper split before it is the narrowest maintainable fix.
