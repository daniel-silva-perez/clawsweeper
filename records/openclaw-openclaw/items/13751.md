## Summary

Current main still has the reported least-privilege problem: Feishu sender-name resolution is enabled by default and still reaches the Feishu contact user API, while live peer listing still enumerates contact users. The issue should stay open because it is valid, security-sensitive, and overlaps assigned maintainer follow-up plus open maintainer-labeled PR #73399.

## What This Changes

The issue requests making Feishu sender-name resolution and live directory behavior avoid broad Feishu contact-directory read access by default.

## Best Possible Solution

Make ordinary inbound Feishu messages contact-free by default: use verified event or chat-member metadata when available, fall back quietly to a stable sender id, and put contact-backed profile lookup plus live user-directory listing behind explicit operator opt-in with docs and regression tests.

## Reproduction Assessment

Yes. A focused source check reproduces the default path: with `resolveSenderNames` unset, `handleFeishuMessage` calls `resolveFeishuSenderName`, which calls `client.contact.user.get`; config, docs, and tests confirm the default remains true.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Feishu plugin boundary rather than core. The implementation should coordinate with PR #73399 and change the default sender-label and live-directory behavior without weakening existing explicit opt-in contact lookup support.
