## Summary

Keep open: current main still has the reported least-privilege problem because Feishu sender-name resolution remains default-on and still calls the Feishu contact user API, while live peer listing still enumerates contact users. The issue is valid, but it overlaps assigned maintainer follow-up and open maintainer-labeled PR #73399, so this should be routed for maintainer coordination rather than closed or queued as a competing automated repair.

## What This Changes

Request to make the Feishu plugin avoid contact-directory API access by default for sender-name resolution and live directory listing.

## Best Possible Solution

Make ordinary inbound Feishu messages contact-free by default inside the Feishu plugin: use verified event or chat-member metadata when available, quietly fall back to a stable sender id, and put contact-backed profile lookup plus live user-directory listing behind explicit operator opt-in with docs and regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with `channels.feishu.resolveSenderNames` unset on a configured Feishu account, `handleFeishuMessage` calls `resolveFeishuSenderName`, which calls `client.contact.user.get`; the live directory peer path similarly calls `client.contact.user.list`.

## Solution Assessment

Yes for the requested direction, but not as a close decision. The maintainable fix should coordinate with PR #73399 and change default sender-label and live-directory behavior without removing explicit opt-in contact lookup support.
