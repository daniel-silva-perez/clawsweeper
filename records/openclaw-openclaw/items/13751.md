## Summary

Current main still has the reported least-privilege problem: default Feishu sender-name resolution calls the contact user API, live peer listing still enumerates contact users, and the latest release tag contains the same behavior. This should stay open because the issue is valid and overlaps assigned maintainer follow-up plus open maintainer-labeled PR #73399.

## What This Changes

The issue asks for basic Feishu inbound sender handling to stop depending on broad contact-directory read permissions, with contact-backed directory lookup moved behind explicit opt-in or fallback behavior.

## Best Possible Solution

Make ordinary inbound Feishu messages contact-free by default: use verified event or chat-member metadata when available, fall back quietly to a stable sender id, and put contact-backed profile lookup plus live user-directory listing behind explicit operator opt-in with docs and regression tests.
