## Summary

Current main still has the least-privilege problem: ordinary Feishu sender-name resolution is enabled by default and still calls the Feishu contact user API, while live peer listing still enumerates contact users. A newer change suppresses one stale `contact:contact.base:readonly` permission notice path, but it does not make the default inbound sender-label path contact-free, so the issue should stay open for the assigned maintainer and overlapping maintainer-labeled PR #73399.

## What This Changes

The issue requests that the Feishu plugin stop depending on broad Feishu contact-directory access for basic sender-name resolution and avoid live contact-directory listing by default.

## Best Possible Solution

Make ordinary inbound Feishu messages contact-free by default: use verified event or chat-member metadata when available, fall back quietly to a stable sender id, and put contact-backed profile lookup plus live user-directory listing behind explicit operator opt-in with docs and regression tests.

## Reproduction Assessment

Yes. A focused source check reproduces the default path: with `resolveSenderNames` unset, `handleFeishuMessage` calls `resolveFeishuSenderName`, which calls `client.contact.user.get`; config, docs, and tests confirm the default remains true.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Feishu plugin boundary rather than core. It should coordinate with PR #73399 and change the default sender-label and live-directory behavior without removing explicit opt-in contact lookup support.
