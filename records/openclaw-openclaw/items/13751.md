## Summary

Current main at f2405c830bd925bd24f91b6bac5d417c6d36e5ba still keeps Feishu sender-name resolution contact-backed and default-on. The inbound path calls `resolveFeishuSenderName` unless `resolveSenderNames` is disabled, the resolver still calls `client.contact.user.get`, and live peer listing still calls `client.contact.user.list`; open PR #73399 overlaps display-name behavior but is still open and preserves contact fallback rather than making basic inbound operation contact-free by default.

## Best Possible Solution

Keep this issue open and resolve it with a focused Feishu plugin change: make basic inbound bot operation contact-free by default by using webhook sender fields when available and otherwise falling back to open_id/user_id without calling `contact.user.get`; gate contact-backed profile lookup and live contact-directory peer listing behind explicit opt-in config; coordinate with open PR #73399 so DM display-name fallback, permission-scope reduction, docs, config metadata, changelog, and focused regression tests land coherently.
