## Summary

Current main and the latest release still have the reported least-privilege problem: default Feishu sender-name resolution calls the contact user API, and live peer listing still enumerates contact users. The issue should stay open because the report is valid, security-sensitive, and already overlaps assigned maintainer follow-up plus open maintainer-labeled PR #73399.

## What This Changes

The issue requests making Feishu sender-name resolution and directory listing work without requiring broad contact-directory read permission by default.

## Best Possible Solution

Make ordinary inbound Feishu messages contact-free by default: use verified event or chat-member metadata when available, fall back quietly to a stable sender id, and put contact-backed profile lookup plus live user-directory listing behind explicit operator opt-in with docs and regression tests.

## Reproduction Assessment

Yes. A focused source check reproduces the default path: with resolveSenderNames unset, handleFeishuMessage calls resolveFeishuSenderName, which calls client.contact.user.get; config/docs/tests confirm that default remains true.

## Solution Assessment

Yes, the right fix belongs in the Feishu plugin boundary rather than core: change the default sender-label and live-directory behavior, document the opt-in contact lookup, and coordinate with the overlapping maintainer PR before implementation.
