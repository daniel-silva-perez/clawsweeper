## Summary

Current main still has the reported least-privilege problem: Feishu sender-name resolution is enabled by default and calls the contact user API, while live peer listing still enumerates contact users. The issue should stay open because it is assigned for maintainer follow-up and overlaps open maintainer-labeled PR #73399 rather than being safe for a competing automated repair.

## What This Changes

Request that the Feishu plugin stop depending on organization-wide contact-directory permission for basic sender-name resolution, and gate or avoid live contact-directory listing by default.

## Best Possible Solution

Make ordinary inbound Feishu messages contact-free by default inside the Feishu plugin: use verified event or chat-member metadata when available, quietly fall back to a stable sender id, and put contact-backed profile lookup plus live user-directory listing behind explicit operator opt-in with docs and regression tests.

## Reproduction Assessment

Yes. A focused source check reproduces the default path: with `resolveSenderNames` unset, `handleFeishuMessage` calls `resolveFeishuSenderName`, which calls `client.contact.user.get`; config, docs, and tests confirm the default remains true.

## Solution Assessment

Yes for the requested plugin-boundary direction, but not as a pre-merge close. The maintainable fix should coordinate with PR #73399 and change the default sender-label and live-directory behavior without removing explicit opt-in contact lookup support.
