## Summary

Keep this issue open. Current main still has no source, test, changelog, or live Windows/QQBot proof that the reported Chinese transcript JSONL corruption is fixed. The normal QQBot and transcript paths are UTF-8-oriented, but the remaining persistence path still delegates message appends to the pinned Pi SessionManager and there is no narrow regression covering QQBot Chinese text through JSONL storage and memory/search indexing.

## Best Possible Solution

Keep this open and add a focused regression/hardening path: feed a QQBot Chinese message fixture through gateway decode, event dispatch, inbound content/body assembly, reply runner transcript prompt persistence, SessionManager append, and memory/search indexing; assert the JSONL bytes decode as valid UTF-8 and preserve the original Chinese text. If corruption appears before persistence, fix the QQBot gateway/content pipeline; if it appears at persistence, wrap SessionManager transcript appends or update the pinned dependency so writes pass explicit UTF-8 and keep the parentId chain intact.
