## Summary

Keep open. Current main still has the same missing-runtime-dependency shape reported for WhatsApp QR rendering: the QR helper dynamically imports `qrcode`, but packaged runtime-deps mirroring does not include `qrcode`, and the existing drift guard incorrectly exempts it as type-only.

## What This Changes

The issue requests fixing the WhatsApp setup QR rendering failure in packaged 2026.4.29 installs where `qrcode` is missing from the plugin runtime dependency root.

## Best Possible Solution

Mirror `qrcode` into the packaged root runtime-deps set, remove the stale `qrcode` drift-guard exception, and add focused package/runtime-deps coverage so WhatsApp QR rendering can resolve the shared SDK QR helper from staged packaged installs.

Yes. The high-confidence static reproduction path is a packaged WhatsApp QR render where `extensions/whatsapp` calls the SDK QR helper, `src/media/qr-runtime.ts` dynamically imports `qrcode`, and the packaged runtime-deps root lacks `qrcode` because it is not mirrored.

## Solution Assessment

Yes. The narrow maintainable fix is package/runtime-deps metadata plus guard coverage; moving QR rendering into WhatsApp or adding startup migrations would miss the shared SDK helper boundary and create more duplication.
