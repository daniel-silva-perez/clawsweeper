## Summary

Keep this issue open. Current main still injects `window.openclawSendUserAction`, but that helper only posts to native iOS/Android bridge objects, the browser WebSocket still only handles live reload, and there is no authenticated browser action POST route or WebSocket action protocol for Gateway-dispatched A2UI user actions.

## Best Possible Solution

Implement the browser fallback in core canvas/gateway code: define a typed A2UI user-action payload, route it through an authenticated `/__openclaw__/canvas/action` endpoint or explicit `/__openclaw__/ws` message, dispatch `openclaw:a2ui-action-status` back to the page, update the A2UI host/helper to use it consistently, and add gateway/canvas tests for successful browser sends plus auth and validation failures. Coordinate with #66983 if the broader web canvas-node design becomes the chosen canonical path.

## Risks / Open Questions

- Browser-originated A2UI actions would let same-origin canvas JavaScript request agent work through the Gateway, so a fix needs explicit auth/capability checks, payload validation, rate/error handling, and status semantics.
- There is product overlap with #66983; maintainers should decide whether to implement a narrow action bridge first or fold it into first-class web canvas-node support.
