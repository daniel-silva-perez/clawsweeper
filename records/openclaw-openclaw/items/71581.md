## Summary

Keep open. Current main still routes forced same-agent heartbeat wakes and explicit cron session keys through the literal canonical session key without applying the `session.dmScope="main"` single-owner rule, and open PR #72648 references this issue with closing syntax, so the issue should remain open until that PR is reviewed, replaced, or merged.

## What This Changes

Request that exec-event and cron-event heartbeat wakes for single-owner `session.dmScope="main"` agents resolve to the agent main session instead of a per-channel direct-message session.

## Best Possible Solution

Land or replace #72648 with a narrow core fix that applies the existing single-owner non-wildcard `dmScope="main"` rule to internal same-agent exec and cron wake routing, while preserving multi-owner, wildcard, non-main dmScope, subagent, and cross-agent behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `session.dmScope="main"` with a single-owner allowlist, enqueue an exec or cron system event under `agent:main:telegram:default:direct:<userId>`, and request a heartbeat with that forced key; current main resolves and peeks the per-channel queue instead of `agent:main:main`.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing dmScope single-owner precondition for internal event routing and ensure enqueue, preflight, and drain semantics agree; documenting the split would preserve the context-fragmentation bug.
