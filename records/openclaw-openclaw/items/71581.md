## Summary

Keep open. Current main still preserves same-agent forced heartbeat and cron session keys literally, while dmScope=main single-owner handling is only wired into inbound channel routing; the issue is also paired with open, unmerged PR #72648 using closing syntax.

## What This Changes

Honor session.dmScope=main for internal exec-event and cron-event heartbeat wakes so same-agent direct-message system events are processed in the canonical main session.

## Best Possible Solution

Review, land, or replace #72648 with a narrow core change that normalizes both system-event enqueue keys and forced heartbeat wake targets for same-agent direct-message sessions when dmScope=main and the existing single-owner allowlist rule is satisfied, while preserving explicit group/channel/custom cron routing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure dmScope=main with a single-owner Telegram allowlist, enqueue an exec or cron system event under agent:main:telegram:default:direct:<userId>, and request an exec-event or cron wake with that forced key; current main resolves and peeks the per-channel queue instead of agent:main:main.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it normalizes enqueue and wake routing together. A heartbeat-only rewrite would be incomplete because system events are keyed before preflight peeks and drains them.
