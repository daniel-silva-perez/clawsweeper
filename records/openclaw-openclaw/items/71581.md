## Summary

Keep open. Current main still routes forced same-agent heartbeat wakes and cron explicit session keys through the literal per-channel session key, while the `session.dmScope="main"` single-owner helper is only used by inbound channel routing. The issue is also paired with open PR #72648 using closing syntax, so it should remain open until that PR is reviewed, replaced, or merged.

## What This Changes

Request that exec-event and cron-event heartbeat wakes honor `session.dmScope="main"` so same-agent direct-message system events process in `agent:<id>:main` instead of orphan per-channel DM sessions.

## Best Possible Solution

Land or replace #72648 with a narrow core fix that routes same-agent direct-message exec and cron system-event wakes to `resolveAgentMainSessionKey` only when `session.dmScope="main"` and the existing single-owner non-wildcard allowlist precondition is satisfied, with regression coverage for exec, cron, and unaffected non-main cases.

## Reproduction Assessment

Yes. The source-level reproduction is to configure `session.dmScope="main"` with a single-owner allowlist, enqueue an exec or cron system event under `agent:main:telegram:default:direct:<userId>`, and request a heartbeat with that forced key; current main resolves and peeks the per-channel queue instead of `agent:main:main`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if it normalizes both event enqueue/drain semantics and the heartbeat wake target. A wake-only rewrite would be incomplete because preflight and consumption already read the resolved session queue.
