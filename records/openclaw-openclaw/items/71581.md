## Summary

Keep open. Current main still preserves forced same-agent per-channel session keys for exec-event and cron-event heartbeat wakes, while the `session.dmScope="main"` single-owner helper is only used by inbound channel routing; the issue is also paired with open PR #72648 using closing syntax, so it should not be closed before that implementation is reviewed, replaced, or merged.

## What This Changes

The issue requests that internal exec-event and cron-event heartbeat wakes honor `session.dmScope="main"` so same-agent direct-message session keys resolve to the agent main session.

## Best Possible Solution

Review, rebase, land, or replace #72648 with a narrow core fix that normalizes same-agent direct-message system-event routing to `resolveAgentMainSessionKey` when `session.dmScope="main"` and the same single-owner non-wildcard allowlist precondition applies. The fix should preserve multi-owner, wildcard allowlist, subagent, non-DM explicit session, and isolated heartbeat behavior.

## Reproduction Assessment

Yes. The source-level reproduction is to configure `session.dmScope="main"` with a single-owner allowlist, enqueue an exec or cron system event under a same-agent per-account direct-message key, and run heartbeat with that forced key; current main resolves and consumes the per-channel key instead of `agent:<id>:main`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, provided it normalizes the event queue and wake target together. Rewriting only `runSessionKey` would not be sufficient because preflight and consumption already use the resolved session queue.
