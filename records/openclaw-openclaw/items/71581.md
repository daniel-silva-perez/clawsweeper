## Summary

Keep open. Current main still preserves same-agent forced heartbeat and cron session keys literally, while `session.dmScope=main` single-owner handling is wired into inbound channel routing only; the reporter’s production follow-up provides concrete orphan-session evidence, and open PR #72648 is already the linked implementation candidate.

## What This Changes

Honor `session.dmScope=main` for exec-event and cron-event heartbeat wakes so single-owner DM events are processed in the agent main session instead of per-channel orphan sessions.

## Best Possible Solution

Review, land, or replace #72648 with a narrow core change that normalizes same-agent direct-message system-event enqueue keys and forced heartbeat wake targets to the agent main session only when `session.dmScope=main` and the existing single-owner allowlist rule is satisfied, while preserving explicit group/channel/custom cron routing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `dmScope=main` with a single-owner Telegram allowlist, enqueue an exec or cron system event under `agent:main:telegram:default:direct:<userId>`, and request an `exec-event` or `cron-event` wake with that forced key; current main resolves and peeks the per-channel queue instead of `agent:main:main`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if enqueue and wake routing are normalized together. A heartbeat-only rewrite would be incomplete because system events are keyed before preflight peeks and drains them.
