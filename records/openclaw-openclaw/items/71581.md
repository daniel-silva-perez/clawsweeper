## Summary

Keep open. Current `main` still preserves same-agent forced heartbeat and cron session keys literally, while the `session.dmScope=main` single-owner helper is only used by inbound channel routing; the reporter's production follow-up matches that split. The issue is also paired with open, unmerged PR #72648 using closing syntax, so it should remain open until that PR is reviewed, repaired, replaced, closed, or merged. https://github.com/openclaw/openclaw/issues/71581

## What This Changes

The issue requests that exec-event and cron-event heartbeat wakes honor `session.dmScope=main` by routing same-agent direct-message system events into the agent main session when the existing single-owner allowlist rule applies.

## Best Possible Solution

Land or replace #72648 with a narrow core fix that normalizes both system-event queue keys and forced heartbeat wake targets for same-agent direct-message exec and cron events only when `session.dmScope` is `main` and the existing single-owner non-wildcard allowlist rule is satisfied, while preserving explicit group/channel cron routing.

## Reproduction Assessment

Yes. A source-level reproduction is to configure `session.dmScope=main` with a single-owner Telegram allowlist, enqueue an exec or cron system event under `agent:main:telegram:default:direct:<userId>`, and request an `exec-event` or `cron-event` wake with that forced key; current `main` resolves and peeks that per-channel queue instead of `agent:main:main`.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if enqueue, preflight peek, wake target, and drain semantics are normalized together; changing only `resolveHeartbeatSession` would leave queued events under the original key.
