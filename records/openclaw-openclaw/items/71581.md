## Summary

Keep open. Current main still preserves same-agent forced heartbeat and cron session keys literally, while the dmScope=main single-owner helper is only used by inbound channel routing; the reporter's follow-up production data is consistent with that split. The issue is also paired with open, unmerged PR #72648 using closing syntax, so it should remain open until that PR is reviewed, replaced, closed, or merged. https://github.com/openclaw/openclaw/issues/71581

## What This Changes

Request that exec-event and cron-event heartbeat wakes route same-agent direct-message system events into the agent main session when session.dmScope is main and the single-owner allowlist precondition is satisfied.

## Best Possible Solution

Land or replace #72648 with a narrow core fix that normalizes both system-event queue keys and forced heartbeat wake targets for same-agent direct-message exec and cron events only when session.dmScope is main and the existing single-owner non-wildcard allowlist rule is satisfied, while preserving explicit group/channel cron routing.

## Reproduction Assessment

Yes. A source-level reproduction is to configure session.dmScope=main with a single-owner Telegram allowlist, enqueue an exec or cron system event under agent:main:telegram:default:direct:<userId>, and request an exec-event or cron wake with that forced key; current main resolves, peeks, and consumes the per-channel queue instead of agent:main:main.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it normalizes enqueue/drain semantics and the wake target together. Rewriting only resolveHeartbeatSession would be incomplete because the system-event queue is keyed before preflight peeks and consumes it.
