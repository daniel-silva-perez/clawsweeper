## Summary

Current main still has the reported `ensureTabAvailable()` race: the local-managed browser path discards the `openTab()` result, immediately relists tabs, and filters candidates to entries with `wsUrl`. The linked fix PR #63355 was closed unmerged at the author's request because the branch was not isolated, and I did not find a replacement PR that landed this fix.

## Best Possible Solution

Keep this issue open and land a clean, isolated browser plugin fix. The fix should capture the cold-start `openTab()` result, refresh/poll briefly when local-managed wsUrl filtering removes otherwise visible tabs, tolerate transient `listTabs()` failures during that recovery window, and add targeted tests that avoid real one-second waits. The separate AbortSignal follow-up in the issue body should stay out of this narrow fix unless a maintainer opens a separate item.

## Repair Work Prompt

Fix the browser plugin race described by this issue. Current `extensions/browser/src/browser/server-context.selection.ts` still discards the cold-start `openTab("about:blank")` result, immediately relists tabs, and filters local-managed candidates to tabs with `wsUrl`, which can throw `BrowserTabNotFoundError` while CDP `/json/list` is still lagging. Implement a clean current-main fix in the browser plugin only: capture the `openTab()` return value, keep a fresh tab snapshot before selection, add a bounded wsUrl-discovery recovery path for `supportsPerTabWs` when tabs exist but all lack `wsUrl`, tolerate transient `listTabs()` failures during that recovery, and use the opened tab directly when CDP list remains empty after a successful open. Be careful not to return stale closed tabs after polling; tests should cover latest-snapshot behavior, transient polling failures, no-wsUrl fallback, and direct openTab fallback without adding real one-second waits. Use #63355 as context but do not copy it blindly because current browser tab alias/target-swap code has changed. Do not broaden this PR into the separate AbortSignal follow-up mentioned in the issue body. Changelog likely not required unless maintainers want a user-facing browser fix entry.
