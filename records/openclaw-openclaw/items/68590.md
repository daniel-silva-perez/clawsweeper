## Summary

Keep this PR open, but do not merge it as-is. The index rewrite addresses a real gap on current main, but the proposed delegation makes the duplicate-document/null-byte repair path potentially idempotent when it needs to force a database-level collection reset, and the generated YAML is not safely quoted.

## What This Changes

The PR adds a hand-written QMD `index.yml` rewrite in `ensureCollections()`, changes managed collection repair to call `ensureCollections()` instead of directly removing and re-adding each managed collection, and updates qmd-manager tests to expect no remove calls during repair.

## Best Possible Solution

Revise the PR so bootstrap and repair share a safe `index.yml` writer, while the repair path still forces a managed collection reset or uses an explicit force mode. Add focused tests for generated YAML and for duplicate-document repair when QMD still lists matching collections.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to mock `qmd collection list --json` returning matching `memory-root-main` and `memory-dir-main`, then trigger the duplicate-document update failure; the PR would skip remove/add and retry against unchanged state.

## Solution Assessment

No. Rewriting `index.yml` is a reasonable direction, but replacing the forced repair loop with the normal idempotent ensure path is not the safest fix; factor the writer separately and preserve the reset semantics.
