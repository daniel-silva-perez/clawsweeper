## Summary

Keep open. Current main has broad `apiRoot` plumbing for Telegram, but the reported polling migration gap still appears valid: persisted update offsets are not scoped to the Bot API endpoint, and the polling startup path still treats Telegram 400 `Logged out` from `deleteWebhook` as a fatal setup error rather than a narrow local-server migration state. The merged `apiRoot` and local-media PRs are useful adjacent context, but they do not cover this inbound polling/offset case.

## What This Changes

The issue requests that Telegram local Bot API `apiRoot` migrations dispatch inbound updates correctly by handling endpoint-specific polling offsets and non-fatal `deleteWebhook` `Logged out` responses.

## Best Possible Solution

Fix this inside the Telegram plugin by storing a normalized endpoint identity with polling offset state, treating missing or mismatched endpoint identity as stale during startup, preserving existing bot-token mismatch safety, and narrowly allowing the local Bot API migration `Logged out` cleanup response to continue or retry without hiding real auth failures.
