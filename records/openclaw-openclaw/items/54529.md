## Summary

Keep this issue open. Current main has broad Telegram apiRoot passthrough, but the reported local Bot API migration gaps remain: persisted polling offsets are not scoped to the Bot API endpoint, and deleteWebhook setup still treats a non-network 400 Logged out response as fatal instead of a narrow migration/cooldown condition.

## Best Possible Solution

Keep this issue open and fix it in the bundled Telegram plugin: persist a normalized apiRoot or endpoint fingerprint with update-offset state, ignore or reset stored offsets when the endpoint changes, handle Telegram 400 Logged out narrowly during local Bot API migration/cooldown, and add focused tests plus Telegram docs for local Bot API migration and manual offset recovery.

## Risks / Open Questions

- Closing now would leave a concrete migration failure where valid local Bot API updates may be ignored because the stored official-API watermark is higher.
- The deleteWebhook fix needs to be narrow so unrelated Telegram 400 authentication or configuration errors still fail clearly.
- The offset fix needs to preserve replay protection for normal restarts against the same endpoint while resetting or ignoring offsets only when endpoint identity changes.
