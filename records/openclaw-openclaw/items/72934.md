## Summary

Keep open. The report is recent, concrete, and current main still has a source-backed snapshot timeout gap: the browser tool accepts a top-level `timeoutMs`, but the snapshot action does not pass it into the `/snapshot` client/proxy path, while the client still applies a fixed 20s timeout.

## What This Changes

Requests fixing a browser automation regression where `action=snapshot` times out after successful `open` or `navigate` calls on macOS OpenClaw 2026.4.25.

## Best Possible Solution

Keep the issue open and fix the browser plugin snapshot path so `timeoutMs` and cancellation are handled consistently across host, gateway `browser.request`, and node-proxy execution, then add focused regression tests and a targeted live/browser smoke if available.
