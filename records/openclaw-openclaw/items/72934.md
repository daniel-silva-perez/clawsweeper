## Best Possible Solution

Keep the issue open and repair the browser plugin snapshot path so an explicit or configured/default snapshot timeout is normalized once and carried consistently from the agent tool through the host client, node proxy, `/snapshot` route, and Playwright/CDP/Chrome-MCP snapshot execution, with focused regression coverage.

## Repair Work Prompt

Fix the browser snapshot timeout regression described in https://github.com/openclaw/openclaw/issues/72934. Current main exposes top-level `timeoutMs`, and gateway/node proxy code can honor supplied timeouts, but `executeSnapshotAction`, `browserSnapshot`, and the `/snapshot` route do not carry a snapshot timeout or abort signal through consistently. Implement a narrow browser plugin repair: normalize explicit `timeoutMs` for snapshot actions, use the existing configured/default browser action timeout when no explicit timeout is supplied where appropriate, pass the effective timeout through the direct host client and node proxy request, and carry it into Playwright/CDP/Chrome-MCP snapshot execution including AI, role, ARIA, labels, and existing-session paths as far as supported. Add focused regression tests for host snapshot timeout propagation, node proxy snapshot timeout propagation, and route-level helper timeout/signal forwarding. Do not rewrite gateway routing, browser profile migration, node pairing, or unrelated navigate/load-state behavior; related #47203 may share timeout plumbing but should not broaden this PR unless the same helper change naturally covers it. Add a user-facing `CHANGELOG.md` fix entry with `Thanks @wkomers-hash`.
