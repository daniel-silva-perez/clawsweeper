## Summary

Keep open. The report is recent, concrete, and current main still has a source-backed snapshot timeout gap: the browser tool exposes `timeoutMs` and gateway/node proxy code can honor it when supplied, but the snapshot action and host snapshot client do not pass it through to `/snapshot`.

## What This Changes

The issue reports that browser `open` or navigation succeeds, but `snapshot` consistently times out on macOS OpenClaw 2026.4.25 instead of returning page content or the accessibility tree.

## Best Possible Solution

Keep this open and repair the browser plugin snapshot path so caller `timeoutMs` and cancellation are carried consistently from the agent tool through host client, gateway `browser.request`, node proxy, and `/snapshot` execution, with focused regression coverage.

## Reproduction Assessment

Yes for triage: the issue provides a concrete open/navigate/snapshot sequence, affected version, platform, and repeated cross-URL failure. Current main source inspection independently shows a matching timeout-handling defect, although live macOS reproduction was not run.

## Solution Assessment

Yes, the narrow maintainable solution is to fix snapshot timeout/cancellation plumbing in the browser plugin. A core gateway rewrite or browser profile/default change would be broader than the evidence supports.
