## Summary

Keep open. Current main still has a source-backed browser snapshot timeout plumbing gap: the tool schema exposes a top-level timeout, and the gateway/node proxy can honor one when supplied, but the snapshot action, host snapshot client, and snapshot route do not consistently carry that timeout into `/snapshot` or the underlying Chrome MCP/Playwright snapshot calls.

## What This Changes

The issue reports that browser snapshot times out after successful open/navigate operations on OpenClaw 2026.4.25 for macOS loopback gateway browser automation.

## Best Possible Solution

Repair the browser plugin snapshot path so a caller-provided or default snapshot timeout is normalized once and propagated from the agent tool through the host client, gateway/node proxy, /snapshot route, Chrome MCP snapshot calls, and Playwright/CDP snapshot helpers, with focused regression tests for host and node-proxy paths.

## Reproduction Assessment

Yes for triage: the issue provides a precise open-then-snapshot sequence, version, platform, and repeated cross-URL failure. Current-main source inspection independently confirms that snapshot timeoutMs cannot affect the /snapshot route or inner snapshot calls.

## Solution Assessment

Yes: the best fix is a narrow timeout-propagation change inside the browser plugin snapshot stack. A core gateway rewrite or broad browser profile migration would be larger than the evidence supports.
