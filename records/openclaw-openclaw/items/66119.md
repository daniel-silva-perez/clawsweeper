## Summary

Keep this issue open. Current main still auto-starts the bundled Signal plugin's `signal-cli` daemon without a non-mutating account-file preflight for `registered:false`, without automatic account-state backup, and without recovery/guard docs for the destructive state described in the report. The existing ClawSweeper comment is still consistent with current main.

## Best Possible Solution

Keep the issue open and investigate in the bundled Signal plugin. The best path is a Signal-owned, non-mutating startup preflight before daemon auto-start where feasible, fail-closed warning behavior when a local account file is detectably `registered:false`, focused mocked regression coverage around startup, and user-facing backup/recovery docs. If the destructive transition is ultimately upstream signal-cli behavior, OpenClaw should still avoid auto-starting the daemon when it can safely detect an invalid local account state.
