## Summary

Keep open. Current main still has the strict minute-field equality check, and this PR targets a real cron stagger bug, but the proposed helper does not fully match Croner-compatible minute-field parsing and should be revised before merge.

## What This Changes

The PR adds a minute-field helper and regression assertions so top-of-hour cron stagger detection recognizes step, list, range, range-step, and wildcard minute fields as firing at minute zero.

## Best Possible Solution

Revise and land the cron stagger fix so the helper follows Croner-compatible minute-field semantics for 5-field and 6-field expressions, including recursive comma-list entries, while avoiding unsupported numeric-prefix stepping; then use the merged fix to resolve #51587.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before applying range or step parsing to each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step coverage:** `src/cron/stagger.ts:23`
  - body: OpenClaw constructs Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless sloppy range parsing is enabled. This helper and its new test should not lock in `0/15` as supported OpenClaw cron syntax; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
