## Summary

Keep open. Current main still has the literal minute-field check that the PR is trying to fix, and the PR is useful source work, but it is not merge-ready because the new list handling still misses valid composite Croner minute fields such as `0-5,30` and `*/15,45`. The diff is limited to cron runtime helper/tests and does not introduce a security or supply-chain sensitive change.

## What This Changes

This PR adds a minute-field helper and tests so top-of-hour cron stagger detection treats step, list, range, range-step, and wildcard minute fields as firing at minute zero.

## Best Possible Solution

Land this PR after tightening the helper to follow Croner-compatible minute-field semantics for both 5-field and 6-field expressions, including composite comma-list entries, then close #51587 once service, normalization, and doctor migration regression coverage proves the persisted default-stagger paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step coverage:** `src/cron/stagger.ts:23`
  - body: OpenClaw uses Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless sloppy range parsing is enabled. This helper and its new test should not lock in `0/15` as supported OpenClaw cron syntax; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
