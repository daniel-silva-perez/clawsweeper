## Summary

Keep this PR open. The linked cron default-stagger bug is still present on current main and the PR is a useful narrow fix, but the proposed helper does not fully match Croner’s minute-field semantics and should not merge as-is.

## What This Changes

The PR adds a minute-field zero detector and tests so top-of-hour cron stagger detection recognizes step, list, range, range-step, and wildcard minute fields for 5-field and 6-field cron expressions.

## Best Possible Solution

Tighten the helper to follow Croner-compatible minute-field semantics for both 5-field and 6-field expressions, including composite comma-list entries, then land the fix with persisted/default-stagger regression coverage and close linked #51587 after merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step coverage:** `src/cron/stagger.ts:23`
  - body: OpenClaw uses Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless `sloppyRanges` is enabled. This helper and its new test should not lock in `0/15` as supported OpenClaw cron syntax; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
