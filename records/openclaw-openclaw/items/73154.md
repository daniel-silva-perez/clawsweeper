## Summary

Keep this PR open. The underlying queue-loss problem is still valid and current main does not implement refresh/reconnect localStorage queue persistence, but the proposed patch is not merge-ready because it conflicts with the current helper-owned session-switch path, replays restored attachment messages without recoverable payload bytes, and introduces a browser-side retention tradeoff for queued chat text.

## What This Changes

The PR adds localStorage-backed WebChat chatQueue persistence, restores queued items during lifecycle/session changes, and drains restored queue entries after gateway hello.

## Best Possible Solution

Rebase onto current main and implement queue durability through the central chat queue/session-switch helpers, with explicit retention behavior and focused coverage for refresh, reconnect, session switch, TTL/quota handling, and attachment-bearing queued messages. Keep conversation-history, draft restoration, and server-side ACK/queue semantics scoped to their own linked follow-ups unless this PR intentionally expands.

## Reproduction Assessment

Yes for the review findings: source inspection shows queued attachments are metadata-only while chat.send requires a resolvable dataUrl, so a refresh-restored attachment queue would replay without files. GitHub's public PR API also provides a high-confidence branch-state check by reporting the PR dirty against its base.

## Solution Assessment

No as written. localStorage can be a narrow text-queue persistence layer, but the implementation needs to integrate with current main's shared session-switch helper and either reject or recover attachment-bearing persisted entries instead of replaying incomplete messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve attachment payloads before replaying queued messages:** `ui/src/ui/chat-queue-persistence.ts:22-23`
  - body: Queued attachments are stored as metadata while their dataUrl payloads live only in the in-memory attachment payload store. After a browser refresh, restored items from localStorage still have an attachments array but no payload, and buildApiAttachments filters them out, so the queued message is sent without its files. Either skip or reject attachment-bearing persisted entries with visible feedback, or persist a recoverable payload path.
  - confidence: 0.89
- **[P2] Restore persisted queues through the shared session switch helper:** `ui/src/ui/app-render.ts:1575`
  - body: This restore is added to the old inline app-render session-switch block, but current main routes session changes through switchChatSession and stores per-session queues in app-render.helpers.ts. After rebasing, restoring persisted queues only at this caller would miss other session-switch paths and conflict with the helper-owned queue lifecycle.
  - confidence: 0.78

## Security Review

Status: needs_attention

Concerns:
