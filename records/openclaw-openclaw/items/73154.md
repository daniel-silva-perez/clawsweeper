## Summary

Keep open. The PR targets a real WebChat queue-loss path and the author addressed the earlier Greptile compile/drain/quota comments, but the branch is currently dirty against main and still has blocking correctness issues around attachment replay and current session-switch integration.

## What This Changes

The PR adds localStorage-backed WebChat chatQueue persistence, restores queued items during lifecycle/session changes, and drains restored queue entries after gateway hello.

## Best Possible Solution

Rebase onto current main and implement queue durability through the central chat queue/session-switch helpers, with coverage for refresh, reconnect, session switch, TTL/quota behavior, and attachment-bearing queued messages; keep broader conversation-history and draft restoration scoped to the linked issues unless this PR expands intentionally.

## Reproduction Assessment

Yes for the review findings: source inspection shows queued attachments are metadata-only while chat.send requires a resolvable dataUrl, so a refresh-restored attachment queue would replay without files. GitHub's PR API also provides a high-confidence mergeability reproduction by reporting the branch as dirty.

## Solution Assessment

No as written. localStorage can be a narrow text-queue persistence layer, but the current patch needs to align with the main-branch session-switch helper and attachment payload model; server-side ACK/queueing remains the more complete answer for the broader reconnect-loss issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve attachments before replaying queued messages:** `ui/src/ui/chat-queue-persistence.ts:22-23`
  - body: Queued attachments are stored as metadata while their dataUrl payloads live only in the in-memory attachment payload store. After a browser refresh, restored items from localStorage still have an attachments array but no payload, and buildApiAttachments filters them out, so the queued message is sent without its files. Either skip/reject attachment-bearing persisted entries with visible feedback or persist a recoverable payload path.
  - confidence: 0.89
- **[P2] Move restore into the shared session switch path:** `ui/src/ui/app-render.ts:1575`
  - body: This restore is added to the old inline app-render session-switch block, but current main routes session changes through switchChatSession and already keeps per-session queues in app-render.helpers.ts. After rebasing, restoring persisted queues only at this caller would miss the other session-switch paths and conflict with the helper-owned queue lifecycle. Integrate the persistence restore/save behavior in the shared helper instead.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch introduces localStorage retention of raw queued chat text and file metadata, which is an intentional feature tradeoff but needs explicit privacy review.

Concerns:

- **[low] Queued chat text is retained in localStorage:** `ui/src/ui/chat-queue-persistence.ts:23`
  - body: persistChatQueue writes queued user message text under a predictable per-session localStorage key for up to 24 hours. That expands browser-side retention of potentially sensitive prompts and file names; before merge, maintainers should confirm this retention model or add opt-in/clearable behavior.
