## Summary

Keep this PR open. Current main has partial overlap through ClawDock and the first-class `openclaw backup create/verify` CLI, but it does not implement this PR’s Mac-hosted Docker keep-awake helper or low-disruption backup/restore workflow. The PR also includes security-sensitive host restore scripts, so the right path is maintainer review rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open and review it against the current `openclaw backup` CLI, generic migration docs, and ClawDock helper surface. The best product path is likely to reuse the first-class backup archive/verification machinery where possible, decide whether keep-awake belongs in ClawDock or Docker docs, and require focused tests or manual proof for macOS portability, checksum validation, restore safety, and stop-before-restore behavior before merging.

## Risks / Open Questions

- The PR touches host-operator scripts that archive and restore OpenClaw state, auth profiles, channel/provider credentials, repo `.env`, and Docker setup files; that needs explicit security and recovery-path review before merge.
- The proposed restore path can overwrite live config/workspace data and runs shell tools such as `tar`, `rsync`, and `docker compose`; maintainers should verify archive integrity checks, traversal safety, stop-before-restore behavior, macOS portability, and secret-handling docs.
- Current main’s `openclaw backup create/verify` overlaps with archive creation but does not satisfy the PR’s keep-awake helper or Docker-specific restore workflow, so closing as implemented would lose remaining functionality.
