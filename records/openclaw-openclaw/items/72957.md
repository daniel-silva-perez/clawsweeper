## Summary

Keep open. Current main still has the threshold-only Control UI chat auto-follow behavior this PR targets: `scheduleChatScroll` and `handleChatScroll` use the 450px near-bottom state, the chat view wires only `@scroll`, and there is no manual-scroll lock, wheel-intent handler, tight reacquire gate, or regression coverage for the proposed interrupt behavior. The PR is still open and unmerged, and it directly addresses the same user problem described in #37500 and #60837.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Review the proposed Control UI scroll state-machine change, request any needed fixes, then land it or apply equivalent manual-scroll release locking, wheel-intent cancellation, return-to-bottom reacquire behavior, and focused `ui/src/ui/app-scroll.test.ts` coverage on main.

That preserves the broad threshold behavior the PR is trying to harden.
