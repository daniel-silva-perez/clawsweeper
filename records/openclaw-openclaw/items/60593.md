## Summary

Keep this issue open. Current main has improved some fallback behavior and tolerant tool-call partial parsing, but it still does not fully implement the reported Anthropic streaming JSON parse recovery: the reported SyntaxError text is not classified as a failover reason, unhandled assistant error formatting can still return the raw message, and the related PR only targets a narrower Discord sanitization path while remaining open.

## Best Possible Solution

Keep this issue open and fix it in the agent/provider error boundary: add a precise streaming JSON parse SyntaxError detector, classify it as a retryable/failover-worthy provider stream failure, sanitize the user-facing message across channels including Telegram, and add regression tests for `formatAssistantErrorText`, `sanitizeUserFacingText`, and the embedded fallback decision path. Review #59118 as useful prior art, but do not treat that open Discord-only PR as resolving this broader issue.

## Evidence

- **Issue scope:** The report is not only malformed tool-call partial JSON; it specifically says Anthropic streaming SyntaxErrors like `Expected ',' or '}' after property value in JSON at position NNN` arrive with `failoverReason=null`, fail to recover, and are exposed through Telegram. The only related PR in the supplied timeline, #59118, is open and scoped to Discord last-mile sanitization, not this issue's Telegram/failover behavior.
- **Anthropic stream still depends on SDK event parsing before OpenClaw sees deltas:** Current Anthropic transport iterates `client.messages.stream(...)`; if that upstream async iterator throws while parsing SSE JSON, OpenClaw only catches the thrown error and stores `error.message` on the assistant output. The tolerant `parseStreamingJson` path applies to `input_json_delta` after a valid event has already been yielded, so it does not prove the reported SDK/SSE parse failure is fixed.
