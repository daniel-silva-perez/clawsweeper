## Summary

Keep this PR open for maintainer handling or a narrow replacement: current main and v2026.4.27 already contain the early search/filter/sort/limit-before-row-build behavior, but they do not contain the child-session reverse index. The remaining index idea is still relevant, but the proposed implementation is stale against current child-link semantics and would need to preserve the newer stale/moved subagent filters before it is mergeable.

## What This Changes

The PR changes `src/gateway/session-utils.ts` to prefilter/search/sort/limit session entries before row enrichment and to add a per-request child-session reverse index for child lookups.

## Best Possible Solution

Keep the shipped prefilter/search/limit behavior, then land a small current-main patch that indexes child relationships while reusing the existing latest-run, lifecycle, and store-only child-link predicates. Leave broader direct lookup, summary indexing, caching, and pagination work to the canonical session-list performance issues.

## Reproduction Assessment

Yes. The supplied stage repro gives high-confidence performance evidence for the activeMinutes+limit hot path, and current tests provide high-confidence semantic checks for stale and moved child-session relationships that any indexed implementation must preserve.

## Solution Assessment

No, not as-is. The requested direction is good, but the PR’s child-index implementation needs to be adapted to current main’s child-link filtering instead of indexing raw `spawnedBy` and `parentSessionKey` links unconditionally.
