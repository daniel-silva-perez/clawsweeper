## Summary

Keep open. Current main still matches the reported custom-session cron transcript gap: the cron runner can target a persistent `session:<id>` and direct delivery can send to the channel, but the direct delivery call does not pass outbound mirror metadata, while transcript persistence is gated on that mirror object. The issue is also already paired with open PR #74913 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that custom `session:<id>` cron jobs deliver visible responses to the destination channel but do not persist those responses in that session transcript, so later channel turns cannot remember the cron message.

## Best Possible Solution

The durable end state is a focused cron fix that mirrors successful direct delivery text into the resolved destination session transcript through the existing outbound mirror contract, with idempotency and regression coverage for `session:<id>` jobs.

## Reproduction Assessment

Yes. The reporter's steps map directly to current main: a custom-session cron job reaches direct delivery, but that delivery call omits `mirror`, and transcript append only runs when `params.mirror` is present.

## Solution Assessment

Yes. Reusing the existing outbound mirror contract is the narrow maintainable solution; the existing linked PR should be reviewed and landed or replaced with an equivalent focused patch.
