## Summary

Keep open. Current main still matches the reported cron custom-session persistence gap: session-bound cron runs can deliver to the requested channel, but direct delivery does not pass the transcript `mirror` object that outbound delivery requires before appending the assistant response to the session transcript. The linked fork PR is an implementation candidate and should be reviewed instead of closing this issue early.

## What This Changes

The issue reports that cron jobs targeting `session:<id>` deliver visible channel output but do not persist the assistant response into that destination session transcript.

## Best Possible Solution

Land or revise a focused cron delivery fix that mirrors successful custom/current session cron sends into the targeted persistent session transcript, preserves isolated-cron duplicate-delivery protections, and clarifies the `session:<id>` session-key contract in docs or CLI messaging.
