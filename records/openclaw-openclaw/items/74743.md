## Summary

Keep open. Current main still matches the reported transcript gap: custom-session cron jobs resolve to a persistent destination session and can deliver directly to the channel, but the direct delivery call does not pass outbound mirror metadata, while transcript persistence is gated on that mirror object. An open PR already targets this exact bug with closing syntax, so the issue should remain open until that fix, or an equivalent one, lands.

## What This Changes

Custom-session cron jobs should persist the directly delivered assistant response in the destination session transcript so follow-up turns can see it.

## Best Possible Solution

Review and land PR #74913, or an equivalent focused fix, so successful direct cron deliveries pass outbound mirror metadata for the resolved destination session with an idempotency key and regression coverage for `session:<id>` jobs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: run a custom-session announced cron job through direct delivery, then observe the delivery call lacks `mirror` while outbound transcript append requires `params.mirror`. The reporter’s live steps are plausible and match this code path.

## Solution Assessment

Yes. Keeping this open while reviewing the existing focused PR is the narrow maintainable path; direct cron delivery should reuse the outbound mirror contract rather than adding a separate transcript persistence path.
