## Summary

Keep open. Current main still has the reported custom-session cron persistence gap: custom-session cron runs use the requested persistent session key, but the direct channel delivery path does not pass the outbound mirror parameter that writes delivered assistant text into the session transcript. A linked fork PR is already open for this exact bug, so maintainers should review or revise that PR rather than close this issue or queue duplicate automation.

## What This Changes

Cron jobs targeting `--session session:<id>` should persist the delivered assistant response in the destination persistent session transcript so later turns in that session can see what the agent just sent.

## Best Possible Solution

Land a focused cron delivery fix that mirrors successful custom/current-session cron sends into the targeted persistent session transcript via the outbound mirror contract, preserves isolated-cron duplicate-delivery and awareness protections, adds regression coverage around session-bound direct delivery, and clarifies the `session:<id>` key contract if user-facing docs imply stored `sessionId`.

## Reproduction Assessment

Yes. The high-confidence code-level reproduction is a session-bound agent-turn cron job with announce/direct delivery: current main reaches `deliverViaDirect`, sends through `deliverOutboundPayloads` without `mirror`, and the only fallback awareness path is intentionally isolated-only.

## Solution Assessment

Yes for the direction, but the exact PR needs maintainer review. The narrow maintainable fix is to use the existing outbound mirror contract for successful non-empty session-bound cron delivery rather than adding a separate transcript writer or broad core migration.
