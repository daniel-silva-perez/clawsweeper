## Summary

Keep open. Current main still matches the reported custom-session cron transcript gap: the job can run against the persistent destination session and deliver to the channel, but the direct cron delivery call does not pass mirror metadata, while outbound transcript persistence is gated on that mirror object. PR #74913 and the linked fork PR already target this exact bug, so the issue should remain open until one fix lands.

## What This Changes

Custom-session cron jobs should persist the directly delivered assistant response in the destination session transcript so follow-up turns can see it.

## Best Possible Solution

Review and land PR #74913, or an equivalent focused fix, so successful direct cron deliveries mirror their visible assistant text into the resolved destination session transcript with an idempotency key and regression coverage for `session:<id>` jobs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: create a custom-session announced cron job and force direct delivery, then observe that `dispatchCronDelivery` calls `deliverOutboundPayloads` without `mirror` while transcript append requires `params.mirror`. The reporter's live steps match that code path.

## Solution Assessment

Yes. Keeping this issue open while reviewing the existing focused PR is the narrow maintainable path; the fix should reuse the outbound mirror contract rather than adding a separate cron-specific transcript writer.
