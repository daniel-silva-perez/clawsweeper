## Summary

Keep open. Current main still has the reported custom-session cron persistence gap: successful direct channel delivery does not pass a transcript mirror, and the only awareness fallback is scoped to isolated cron jobs. A linked fork PR is already open for the same bug, so the issue should remain open until that PR lands or maintainers choose a different fix.

## What This Changes

The issue asks for cron jobs targeting `--session session:<id>` to persist delivered assistant responses into the destination session transcript so the channel agent can remember them.

## Best Possible Solution

Land a focused cron delivery fix that mirrors successful custom/current session cron sends into the targeted persistent session transcript, preserves isolated-cron duplicate-delivery protections, and clarifies or bridges the `session:<id>` key versus stored `sessionId` contract.
