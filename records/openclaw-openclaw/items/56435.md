## Summary

Keep open. Current main still does not implement Feishu `update` card actions, and the draft PR remains useful but incomplete: broadcast replies miss the new card-update context, long final replies bypass existing Feishu chunk handling, and the Signal SDK entrypoint change conflicts with the current curated SDK contract.

## What This Changes

The PR adds a Feishu interactive-card `update` action flow with pending update registration, in-place processing/final card patches, duplicate final-text suppression, Feishu tests/docs, and Signal runtime API/plugin SDK entrypoint adjustments.

## Best Possible Solution

Rebase the branch onto current main, keep the implementation inside the Feishu plugin, carry card-update context through every reply-dispatch path including broadcast groups, apply the existing Feishu size/chunk policy or a safe fallback before final card patches, and align the Signal runtime/build fix with the current SDK contract instead of reviving `signal-core`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: in a Feishu group configured with `broadcastAgents`, click an `update` card action and the active broadcast dispatcher lacks `cardUpdate`, so the final answer follows normal reply delivery instead of patching the original card. A long final text similarly reaches one card patch without the existing chunking path.

## Solution Assessment

No. The implementation direction is reasonable and the earlier processing-card bug is addressed, but the current patch is incomplete for broadcast groups and long replies, and the SDK entrypoint edit is stale against current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass card-update context through broadcast dispatch:** `extensions/feishu/src/bot.ts:1141`
  - body: The new `cardUpdate` wiring only reaches the single-agent dispatcher. In broadcast-enabled Feishu groups, the active-agent dispatcher is still created without that context, so final text will be sent through the normal reply path instead of patching the original card and duplicate follow-up replies can reappear.
  - confidence: 0.93
- **[P2] Guard final card patches with Feishu size handling:** `extensions/feishu/src/reply-dispatcher.ts:404-410`
  - body: Card-update finals embed the entire model text in one `updateCardFeishu` payload, bypassing the existing `textChunkLimit`/`resolveTextChunksWithFallback` path used for normal Feishu replies. Long final outputs can make the patch fail and leave the interactive card stuck in the processing state.
  - confidence: 0.86
- **[P2] Drop the stale signal-core SDK entrypoint:** `scripts/lib/plugin-sdk-entrypoints.json:188`
  - body: Current main explicitly denies `signal-core` as a public plugin SDK subpath and no longer has `src/plugin-sdk/signal-core.ts`. Adding it back to `plugin-sdk-entrypoints.json` reintroduces a retired surface and will fail or undo the current SDK export policy after rebase.
  - confidence: 0.84
