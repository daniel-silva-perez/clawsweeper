## Summary

Keep open. Current main still contains the reported false-positive path: the doctor Gateway auth health check detects a configured SecretRef, evaluates auth through a resolver that does not resolve that configured SecretRef, and emits the unavailable warning even when the SecretRef could resolve through the normal SecretRef-aware helper. The latest release tag still contains the same branch, and the related PRs are not merged fixes in the target repository.

## What This Changes

Doctor should stop warning that `gateway.auth.token` is unavailable when the configured SecretRef resolves successfully and the gateway can authenticate with it.

## Best Possible Solution

Use the existing SecretRef-aware gateway token resolver inside `runGatewayAuthHealth` before emitting the unavailable warning, keep the warning only for genuinely unresolved SecretRefs, and add a focused regression test for a resolvable env or file SecretRef.

## Reproduction Assessment

Yes. The high-confidence reproduction path is local token mode with `gateway.auth.token` configured as a resolvable SecretRef, then running doctor; current main statically reaches the warning branch because the health check uses `resolveGatewayAuth`, which strips configured SecretRefs.

## Solution Assessment

No for the closed one-line `process.env` approach, because that still would not resolve arbitrary configured SecretRef providers such as file refs. The maintainable fix is to call the existing `resolveGatewayAuthToken`/service helper before deciding the SecretRef is unavailable.
