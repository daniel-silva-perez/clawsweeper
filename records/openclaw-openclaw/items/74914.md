## Summary

Keep open. The requested native Google Vertex config support is valid and current main still rejects `api: "google-vertex"`, but the proposed patch is incomplete: it accepts the new provider/model API value in schema without preserving provider-level `api: "google-vertex"` through runtime inline model materialization, so common provider-level configs can still miss the native Vertex stream path.

## What This Changes

The PR adds `google-vertex` to the model API config enum/generated schema and adds tests for native Google Vertex config validation and Google provider normalization.

## Best Possible Solution

Keep the PR open and extend it so `google-vertex` is treated consistently across the config schema, inline provider API inheritance, and Google provider runtime selection, with a focused test proving a provider-level native Vertex config materializes models with `api: "google-vertex"`.

## Reproduction Assessment

Yes. A source-backed reproduction path is available by tracing a provider-level `models.providers.google-vertex.api` value through `buildInlineProviderModels`: after the PR makes the schema accept it, `normalizeResolvedTransportApi` still drops it unless every model repeats the API value.

## Solution Assessment

No. Adding the enum/schema entry is necessary, but the narrow maintainable fix also needs the runtime allowlist and coverage for provider-level inheritance so accepted configs keep the native Vertex transport.
