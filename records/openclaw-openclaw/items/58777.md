## Summary

Keep open. Current main persists requested and fallback runtime models into session/session-status paths, but the detached task record and lifecycle event contracts still do not expose requested-vs-effective model/provider/thinking fields, and fallback attempts still reuse the precomputed thinking level rather than deriving it for each fallback candidate.

## What This Changes

The issue requests additive sub-agent telemetry and reconciliation so callers can compare requested model pins with the effective provider, model, and thinking level after fallback or routing changes.

## Best Possible Solution

Add a generic core runtime-result metadata path that carries requested and effective provider/model/thinking through fallback, session persistence, subagent task records, and lifecycle broadcasts, with candidate-level thinking compatibility re-clamp during fallback and regression tests around subagent fallback mismatches.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to run a subagent with an explicit model and a configured fallback that succeeds after the primary fails, then inspect the lifecycle event and detached task record: the current contracts have no effective provider/model/thinking fields, while session-store persistence can show only the final runtime model.

## Solution Assessment

Yes. The issue’s P1 direction is the narrow maintainable fix if implemented as optional core metadata plus per-fallback thinking re-clamp; the task-router/plugin rewrite vector should remain a separate plugin-owned follow-up unless a missing core API is identified.
