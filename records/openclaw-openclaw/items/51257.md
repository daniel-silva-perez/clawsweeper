## Summary

Keep open. Current main still drops `label` and `displayName` before `/status` and the TUI status summary, so the PR is not obsolete; however, the branch should not merge as-is because it misses the current CLI report-data path and introduces unresolved selector and terminal-rendering issues.

## What This Changes

The PR adds optional `label` and `displayName` fields to the status payload/types and renders recent sessions with a `displayName ?? label ?? key` fallback in CLI and TUI status output.

## Best Possible Solution

Land a corrected implementation that carries `label` and `displayName` through the status payload, updates the active CLI report-data/session-row path and TUI formatter, displays a friendly sanitized name without losing a resolvable key or label, and adds focused tests for labeled sessions plus control-character/ANSI inputs.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available on current main: create or inspect a `SessionEntry` with `label`/`displayName`, then follow `getStatusSummary` into `buildStatusSessionsRows` and `formatStatusSummary`; those paths still emit only `key`.

## Solution Assessment

No. The PR is pointed at the right product problem, but the best fix should update the current shared CLI renderer, preserve a resolvable selector when friendly text differs, and sanitize friendly names before terminal output.

## Review Findings

Full review comments:

- **[P2] Preserve a resolvable session selector:** `src/commands/status.command.ts:607`
  - body: Using `displayName ?? label ?? key` in the Key column can show a value that users cannot pass back to session commands, because gateway resolution accepts `key`, `sessionId`, or `label`, not `displayName`.   - confidence: 0.91

  - body: `label` and `displayName` can come from `sessions.patch` or channel/thread metadata and may contain newlines, control characters, or ANSI escapes. Rendering them directly lets a crafted name inject extra status rows or terminal control sequences in the CLI/TUI output.
  - confidence: 0.93
- **[P2] Move the CLI change into the active row builder:** `src/commands/status.command.ts:607`
  - body: Current main renders `openclaw status` sessions through `buildStatusCommandReportData` and `buildStatusSessionsRows`, which still use `sess.key`. Updating the older row-construction site in `status.command.ts` will not update the active CLI table after rebase.
  - confidence: 0.86
