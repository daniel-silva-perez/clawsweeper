## Summary

Keep open. Current main still does not carry session `label` or `displayName` through the `/status` payload or render them in the active CLI/TUI status paths, so this PR is not obsolete. The provided diff remains a plausible implementation candidate for #51184, but it needs rework against the current CLI report path and should address the existing resolver and terminal-sanitization review concerns.

## Best Possible Solution

Keep this PR open for maintainer review or rework: add `label` and `displayName` to the status payload/types, populate them from `SessionEntry`, update the current CLI report-data/session-row path plus the TUI gateway-status formatter, preserve a resolvable selector when friendly display text differs, sanitize all terminal-rendered friendly names, and add targeted regression tests for friendly labels and unsafe control characters.

## Risks / Open Questions

- The proposed `displayName ?? label ?? key` fallback can make the visible value unresolvable when `displayName` differs from the stored `label`, because gateway resolution does not accept displayName.
- Friendly session names need terminal sanitization before printing; labels and thread/group display names can contain newlines, control characters, or ANSI escapes.
- The PR currently misses the active CLI status table builder on main, so merging it as-is would not update `openclaw status` output.
- No focused regression coverage currently proves friendly names or unsafe friendly-name text in the status summary, CLI sessions table, or TUI gateway-status formatter.
