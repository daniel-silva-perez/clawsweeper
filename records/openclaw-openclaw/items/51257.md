## Summary

Keep open. Current main still does not surface session label/displayName in the `/status` payload, CLI status table, or TUI status summary, so this PR is not obsolete. It remains a plausible implementation candidate for the still-open #51184, but the patch is stale against the current CLI render path and the resolver/sanitization concerns from review are still valid.

## Best Possible Solution

Rework this PR or a replacement implementation against current main: add `label` and `displayName` to the status payload/types, copy them from `SessionEntry` in `status.summary.ts`, render a sanitized friendly session label in the current CLI and TUI status paths, keep a resolvable selector visible when displayName differs from label, and add targeted regression tests for CLI/TUI output including unsafe control characters.

## Risks / Open Questions

- The PR's `displayName ?? label ?? key` fallback can hide the resolvable `label` when `displayName` is present, because gateway resolution does not accept displayName.
- The proposed CLI/TUI output would print label/displayName values without terminal sanitization, allowing newlines, control characters, or ANSI escape sequences from session metadata to affect terminal rendering.
- The patch is stale against current main because CLI status sessions now render through `status.command-report-data.ts` and `status.command-sections.ts`, not the old direct row construction in `status.command.ts`.
- No focused regression coverage is present for friendly status names or unsafe label/displayName text in the current status summary, CLI table, or TUI formatter.
