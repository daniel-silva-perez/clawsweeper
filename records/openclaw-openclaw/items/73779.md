## Summary

Keep this PR open with changes requested. Current main still reloads `sessions.list` immediately on `sessions.changed`, so the PR is not obsolete, but the submitted debounce patch leaves the existing synchronous unit test stale and does not clear the new pending timer during UI teardown.

## What This Changes

The PR adds Control UI `sessionsRefreshTimer` state and debounces the full `loadSessions` reconciliation for `sessions.changed` events while keeping the incremental session update immediate.

## Best Possible Solution

Request changes on this PR: keep the immediate incremental session-row update, debounce and coalesce only the full `sessions.list` reconciliation, clear any pending timer during Control UI disconnect cleanup, add fake-timer coverage for delayed and coalesced reloads, initialize the test fixture state, and add the required user-facing changelog entry if maintainers classify this as a visible Control UI performance fix.

## Reproduction Assessment

Yes. Current main can be verified by reading `handleGatewayEventUnsafe`: a `sessions.changed` event calls `loadSessions` immediately, and the existing unit test encodes that synchronous behavior; the PR regression is also reproducible by that test unless timers are advanced.

## Solution Assessment

No, not as submitted. The debounce direction is a narrow maintainable fix for the reported load pattern, but the implementation needs timer lifecycle cleanup, updated tests, and changelog handling before it is the best solution.
