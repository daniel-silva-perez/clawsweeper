## Summary

Keep open. Current main still preserves `emoji` in the gateway identity contract and `agent.identity.get` response, but the Control UI assistant identity state, bootstrap normalization, and chat rendering path only carry name/avatar metadata, so the assistant chat footer still cannot render the configured identity emoji. The linked #45898 attempt is closed unmerged and targeted the selector dropdown rather than the reported chat transcript header/footer path.

## What This Changes

The issue reports that the v2 Control UI chat no longer renders the configured agent identity emoji inline with the agent name.

## Best Possible Solution

Carry assistant identity emoji through the existing bootstrap/RPC normalization and Control UI app state, then render it inline with the assistant name for persisted and streaming assistant chat groups. Preserve backend identity resolution, local avatar override behavior, and the existing Sessions-table emoji behavior, with focused UI/gateway contract tests.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: mock `agent.identity.get` or bootstrap config with an `emoji`, then render assistant chat groups; current normalization and render props discard the value before `.chat-sender-name` is produced.

## Solution Assessment

Yes. A focused Control UI propagation/rendering fix is the narrowest maintainable solution because the gateway already resolves and exposes emoji; the closed selector-dropdown approach would not fix the reported transcript path.
