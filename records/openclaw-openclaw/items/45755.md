## Summary

Keep open. Current main still preserves `emoji` in the gateway identity contract and RPC response, but the Control UI assistant identity state and chat transcript rendering only carry name/avatar metadata, so the assistant message footer cannot render the configured identity emoji. The related selector-scoped PR #45898 is closed unmerged and would not have fixed the chat transcript path described in the discussion.

## What This Changes

The issue asks to restore the configured agent identity emoji in the Control UI chat message identity label after the v2 UI regression.

## Best Possible Solution

Carry assistant identity emoji through the existing Control UI bootstrap/RPC normalization and app state, then render it inline with the assistant name for persisted and streaming assistant chat groups. Keep the existing backend contract, avatar override behavior, and Sessions-table emoji behavior intact, and cover the chat transcript path with focused UI tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure or mock `agent.identity.get` with an `emoji`, then render assistant chat groups; current UI normalization and render props discard the value before `.chat-sender-name` is rendered.

## Solution Assessment

Yes. A focused Control UI state/rendering fix is the narrowest maintainable path because the gateway already resolves and returns the emoji; changing only selector/session-list surfaces would not address the reported chat transcript regression.
