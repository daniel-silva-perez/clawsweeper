## Summary

Keep this issue open. Current main carries agent emoji through the gateway identity contract, and some non-chat UI surfaces use identity emoji, but the v2 chat message header/footer path still renders only the assistant name. The closed linked PR #45898 was not merged and, from the supplied context, targeted the selector dropdown rather than the reported chat bubble/header regression.

## Best Possible Solution

Keep the issue open and fix it in the Control UI chat path: preserve `emoji` in the browser assistant identity model, plumb it through `ChatProps` into `renderMessageGroup` and streaming footer rendering, and add a focused UI regression test that an agent identity emoji appears next to the assistant name in chat while existing avatar behavior remains intact.
