## Summary

Keep this PR open for maintainer handling, but do not merge it as submitted. Current main still sends pairing codes only for newly created shared DM pairing requests, while this one-line guard change conflicts with the existing shared and channel contracts, fires creation-oriented callbacks for repeat pending requests, and broadens repeated pairing-code disclosure without a defined cross-channel reminder policy.

## What This Changes

The PR changes `src/pairing/pairing-challenge.ts` so an existing pending pairing request with a nonempty code sends another pairing reply instead of returning early when `created` is false.

## Best Possible Solution

If repeat pairing reminders are desired, add an explicit reminder path that preserves `created` and `onCreated` as creation-only signals, sends existing codes through named reminder behavior with cooldown or suppression, and updates shared helper tests, affected channel tests, docs, and changelog together.

## Reproduction Assessment

Yes. Mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`, or send two unauthorized Telegram/Slack/Signal DMs where the second upsert returns `created: false`; current main sends once, while this PR falls through and sends again.

## Solution Assessment

No. The one-line guard change is not the best way to solve the UX issue because it conflates reminder delivery with request creation; the safer alternative is an explicit reminder contract with preserved creation-only callbacks and cooldown policy.
