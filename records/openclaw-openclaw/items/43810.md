## Summary

Keep open. The UX goal is plausible, but the submitted one-line guard change conflicts with the current shared pairing contract: repeat pending requests return a code with `created: false`, current main intentionally returns before replying, existing tests assert that behavior, and channel callbacks treat `onCreated` as a first-request side effect. Matrix already implements the safer pattern separately with reminder cooldown semantics, so the cross-channel policy needs maintainer judgment before merge.

## What This Changes

The PR changes the shared DM pairing challenge guard so existing pending requests with a code send another pairing reply instead of returning early.

## Best Possible Solution

If repeat reminders are desired, add an explicit reminder contract that preserves `created` and `onCreated` for first-time creation only, reuses only valid pending codes, applies a documented cooldown or suppression policy comparable to Matrix, and updates shared helper tests, affected channel tests, docs, and changelog together.

## Reproduction Assessment

Yes. Mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }`; current main returns `{ created: false }` without sending a reply, while this PR's `!code` guard would fall through because the code is truthy.

## Solution Assessment

No. The submitted patch is not the best way to solve the UX request because it conflates reminder delivery with request creation; a named reminder path with cooldown and creation-only callback semantics is safer.
