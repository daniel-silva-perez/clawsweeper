## Summary

Keep this PR open as a real but unresolved behavior proposal. Current main, docs, and tests intentionally send DM pairing replies only for newly created shared-helper requests, while the patch changes that contract without preserving creation-only side effects, updating repeat-request tests, or defining a cross-channel resend/cooldown policy.

## What This Changes

The PR changes `issuePairingChallenge` so repeat pending unapproved senders with an existing pairing code fall through and receive another pairing reply.

## Best Possible Solution

If maintainers want repeat pairing replies, implement it as an explicit resend/reminder path: reuse the existing code only when valid, preserve `created` and `onCreated` for true first creation or add a separate resend signal, add a cooldown/rate-limit policy, and update shared tests, channel tests, and pairing docs to match.

## Review Findings

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve creation-only side effects:** `src/pairing/pairing-challenge.ts:31`
  - body: With the new guard, `{ code, created: false }` falls through to `params.onCreated` and the final `{ created: true, code }` return. Channel callers use `onCreated` for first-request logs and side effects, so repeat `/start` messages become false creation events unless the resend path keeps those semantics separate.
  - confidence: 0.92

  - body: The shared pairing test and multiple channel tests currently assert that an existing pending request does not send a second pairing reply. This line flips that contract without updating tests or docs, so the submitted patch is inconsistent with the current expected behavior.
  - confidence: 0.9
