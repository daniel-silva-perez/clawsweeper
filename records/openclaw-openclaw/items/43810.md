## Summary

Keep this PR open but do not merge it as-is. Current main and the latest shipped release still intentionally avoid resending DM pairing codes for an existing pending request, while the PR's one-line guard change would also fire creation-only callbacks and leave existing shared/channel tests and security docs inconsistent.

## What This Changes

The PR changes `issuePairingChallenge` so a pending sender with a stored pairing code receives the pairing reply again by returning early only when no code exists.

## Best Possible Solution

If maintainers want this UX, implement repeat pairing replies as an explicit resend path: resend only when a valid existing code is present, keep `onCreated` and the returned creation state tied to true first creation or add a separate resend signal, and update the shared tests, channel tests, pairing docs, and security docs with the chosen cooldown policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve creation-only side effects:** `src/pairing/pairing-challenge.ts:31`
  - body: With `created === false` and a stored code, the new guard falls through to `onCreated` and the final `{ created: true, code }` return. Channel callers use `onCreated` for first-time pairing-request logs and possible one-time side effects, so repeat messages become false creation events; keep those semantics behind `created` while adding any resend path separately.
  - confidence: 0.92
- **[P2] Update the repeat-request contract tests:** `src/pairing/pairing-challenge.ts:31`
  - body: The shared pairing test and channel tests already assert that an existing pending request does not send a second pairing reply. This one-line change flips that contract without updating the tests, so the targeted pairing/channel suites will fail or remain inconsistent with the intended behavior.
  - confidence: 0.9
