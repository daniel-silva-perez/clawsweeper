## Summary

Keep this PR open. Current main has not implemented repeat pairing-code replies, but the proposed one-line branch change is incomplete because it breaks the existing shared/channel contract and changes creation-only side effects.

## What This Changes

This PR changes shared DM pairing challenge issuance so an existing pending sender with a stored code falls through to send the pairing reply again instead of returning early.

## Best Possible Solution

If maintainers want this UX, make repeat pairing replies an explicit path: resend only when a valid existing code is present, preserve `onCreated` for first creation only, return the true creation state or add a separate resend signal, and update the shared tests, channel tests, and pairing docs with any cooldown policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve the creation-only side effects:** `src/pairing/pairing-challenge.ts:31`
  - body: With `created === false` and a stored code, the new guard falls through to `onCreated` and the final `{ created: true, code }` return. Slack, Telegram, Signal, and BlueBubbles use this callback for first-time pairing-request logs, so repeat `/start` messages become false creation events; keep those side effects and the return state behind `created` while adding any resend path.
  - confidence: 0.91
- **[P2] Update the repeat-request contract tests:** `src/pairing/pairing-challenge.ts:31`
  - body: The shared test already stubs `{ code: "ABCD", created: false }` and expects no reply; Telegram, Slack, Signal, and BlueBubbles tests likewise assert one send for repeat pending requests. This PR changes that contract without updating tests, so current targeted suites will fail.
  - confidence: 0.89
