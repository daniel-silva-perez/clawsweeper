## Summary

Keep this PR open. Current main still sends a DM pairing reply only when a pairing request is newly created, so the requested repeat `/start` code reply is not already implemented. The submitted one-line change is still incomplete because it would change `onCreated` semantics, break current shared/channel tests, and alter an access-control reply policy without docs or throttling.

## Best Possible Solution

Keep this PR open for a complete implementation if maintainers want this UX change: resend the existing pending code when `upsertPairingRequest` returns a non-empty code, preserve `onCreated` so it only fires for `created === true` or add a separately named reply/reminder callback, add a cooldown or explicit repeat-reply policy if desired, then update shared pairing tests, affected channel tests, and the public pairing/security docs.

## Risks / Open Questions

- Merging the current one-line patch would make `onCreated` fire for repeat pending requests, producing duplicate pairing-request logs and possibly repeated future one-time side effects in channel code.
- Always resending a still-valid pairing code to unapproved senders without a cooldown or explicit policy could increase DM spam and code exposure during the one-hour validity window.
- Current shared tests, channel tests, and public docs all encode no-resend behavior, so the PR would leave implementation, tests, and user-facing contract inconsistent.
