## Summary

Keep this issue open. Current main still has the assistant-anywhere session-history guard and the fallback retry marker is still hard-coded with no config or hook surface; the only related implementation PR, #69109, was closed unmerged and did not ship.

## What This Changes

The issue requests that attempt-execution treat session history as answered only when an assistant message appears after the latest user turn, and add a supported override or hook for the fallback retry continuation prompt.

## Best Possible Solution

Land a focused fix that changes the session scan to track assistant-after-latest-user, adds the missing regression coverage, and exposes a small documented operator override for the retry marker while keeping the current default behavior unchanged.

## Reproduction Assessment

Yes. A JSONL transcript shaped user -> assistant -> fresh user will currently return true because the scan exits on the prior assistant record before considering the later user turn; the existing tests do not cover that shape.

## Solution Assessment

Yes for the guard: the narrow maintainable fix is to keep the existing JSONL parser but track/reset state by user and assistant roles. For the prompt hook, the safest first solution is a small config-backed override rather than a broader plugin/event hook unless maintainers want a larger extension API.
