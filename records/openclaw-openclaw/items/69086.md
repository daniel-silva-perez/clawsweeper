## Summary

Keep this issue open. Current main still treats any prior assistant record in the session JSONL as sufficient history for fallback retry handling, and the retry prompt text is still fixed in code with no supported config, plugin, or callback override; the only focused implementation PR, #69109, is closed and unmerged.

## What This Changes

Fix fallback retry handling so session history only counts assistant messages after the latest user turn, and add a supported operator override for the retry continuation prompt.

## Best Possible Solution

Land a focused core fix that keeps the default retry behavior compatible, changes the history guard to assistant-after-latest-user semantics, adds the missing JSONL regression tests, and documents a small config-backed retry prompt override for operators.

## Reproduction Assessment

Yes. A JSONL transcript with user -> assistant -> fresh user currently returns true because the scan exits at the earlier assistant record; the missing regression fixture described in the issue can reproduce the wrong result with `sessionFileHasContent()`.

## Solution Assessment

Yes. The maintainable fix is to keep the existing bounded JSONL parser but track role state across the scanned records, and to add a narrow config override rather than a larger plugin or event hook unless maintainers want a broader extension API.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/69086 in one focused PR. Change the attempt-execution session-history guard so it answers whether an assistant message exists after the latest user message, not whether any assistant record exists anywhere in the JSONL transcript. Keep the existing bounded JSONL parsing, symlink/file guards, and error swallowing behavior. Add regression tests for user -> assistant -> fresh user returning false and user -> assistant -> fresh user -> assistant returning true. Then expose a small supported operator override for the fallback retry continuation prompt, preferably under `agents.defaults` with the current retry marker as the unchanged default; plumb it through `runAgentAttempt()`/`resolveFallbackRetryPrompt()` without changing fallback selection, provider routing, model cooldown logic, CLI resume behavior, or the default prompt when unset.
