## Summary

Keep open. The report is fresh, concrete, and still appears valid on current main: heartbeat prompts are explicitly hidden from Control UI/WebChat history, but the inspected Telegram/channel delivery path still treats non-final delivered text as visible without filtering the heartbeat transcript marker. There is also an open PR, #75167, with closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Telegram DM delivery never expose the internal heartbeat poll prompt text and that OK-only heartbeat acknowledgments remain suppressed by default.

## Best Possible Solution

Review and land PR #75167, or an equivalent narrow fix, so the heartbeat transcript marker and configured heartbeat prompt are treated as internal control text before any external channel delivery while preserving alert delivery and optional `showOk` behavior.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.27 Telegram DM configuration and current main has a focused unit-test seam via `runHeartbeatOnce`/Telegram outbound stubs to assert the marker is never sent.

## Solution Assessment

Yes for keeping this issue open: the linked PR is the correct implementation candidate and uses closing syntax, so the issue should close only after a fix merges. For the implementation itself, the safest direction is a shared heartbeat/channel visibility guard with Telegram regression coverage rather than relying only on UI history filtering.
