## Summary

Keep open. The report describes a real remaining external-channel leak class: current heartbeat delivery can forward `includeReasoning` payloads to channel delivery even when the final heartbeat result is `HEARTBEAT_OK`, and that path does not filter the internal heartbeat transcript prompt before Telegram/other channel sends. The only unresolved detail is the report's "no explicit target" claim, because current target resolution defaults to `none` unless another wake/commitment/config path resolves a deliverable Telegram target.

## What This Changes

Prevent the internal heartbeat poll prompt and OK-only heartbeat acknowledgments from appearing as visible Telegram DM messages, including heartbeat runs with reasoning enabled.

## Best Possible Solution

Add a generic heartbeat outbound sanitization step before any channel delivery so heartbeat reasoning and final payloads cannot carry `HEARTBEAT_TRANSCRIPT_PROMPT`, the configured heartbeat prompt, task heartbeat prompts, or OK-only acknowledgments to Telegram or other plugins. Keep the filtering in shared heartbeat delivery code rather than Telegram-specific code, and add a focused regression that mocks a Telegram heartbeat with `includeReasoning: true`.

## Reproduction Assessment

Yes for the likely leak class: a targeted `runHeartbeatOnce` unit test can mock `getReplyFromConfig` to return `[{ text: "Reasoning:\n[OpenClaw heartbeat poll]" }, { text: "HEARTBEAT_OK" }]` with `includeReasoning: true` and a Telegram target, then assert no internal prompt text is sent. The exact no-target part remains unclear because current code defaults unresolved heartbeat targets to `none`.

## Solution Assessment

Unclear for the exact production route until the target mismatch is confirmed, but the narrow maintainable fix is clear: sanitize heartbeat outbound payloads in the shared heartbeat runner before `deliverOutboundPayloads`, not in the Telegram formatter.

Fix the heartbeat prompt leakage reported in https://github.com/openclaw/openclaw/issues/75134. Scope: shared heartbeat delivery, not a Telegram-only formatter hack. Ensure `runHeartbeatOnce` never passes internal heartbeat trigger text (`HEARTBEAT_TRANSCRIPT_PROMPT`, the configured/default heartbeat prompt, and task heartbeat prompt forms recognized by `isHeartbeatUserMessage`) to `deliverOutboundPayloads`, including `includeReasoning: true` reasoning payloads and final payloads. Preserve useful `Reasoning:` delivery when it does not contain internal heartbeat prompt text, and preserve existing `HEARTBEAT_OK` suppression semantics. Add or update focused tests in the heartbeat-runner suite for a Telegram target with `includeReasoning: true`, a mocked reply array like `Reasoning:\n[OpenClaw heartbeat poll]` plus `HEARTBEAT_OK`, and assert no outbound send contains the internal prompt. Also add a control case where normal reasoning plus a real alert still delivers. Do not change channel plugin boundaries or make Telegram-specific core assumptions.
