## Summary

Keep this PR open. Current main still compares browser-origin allowlists against `URL.origin`, which is `"null"` for `chrome-extension://`, `app://`, and similar opaque-origin schemes, so exact extension origins cannot match. The PR remains a focused fix for a live gateway allowlist bug; scheme wildcard support is separate in #39739.

## Best Possible Solution

Keep this PR open and land it after rebase, or land an equivalent narrow gateway patch. The fix should symmetrically normalize incoming `Origin` headers and configured `gateway.controlUi.allowedOrigins`, using `URL.origin` for standard origins and a `protocol//host` fallback only when `URL.origin` is `"null"`. Keep focused unit coverage plus a WebSocket handshake regression for accepted and rejected extension origins, and leave wildcard pattern support to #39739.

## Risks / Open Questions

- Origin allowlisting is security-sensitive; the final patch should preserve literal `Origin: null` rejection, exact matching semantics, global `"*"` behavior, host-header fallback behavior, and token/device auth behavior.
- The PR branch is older than current main and should be rebased with tests aligned to the current table-driven `src/gateway/origin-check.test.ts` style before landing.
- Scheme-level wildcard support such as `chrome-extension://*` remains a separate product/security decision tracked by #39739 and should not be bundled into this narrow fix.
- The PR's allowlist normalization drops unparsable configured entries instead of retaining raw lowercased strings; that appears safe because malformed incoming origins already fail parsing, but it should stay covered by regression tests or reviewer acknowledgement.
