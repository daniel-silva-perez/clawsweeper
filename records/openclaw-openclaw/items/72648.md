## Summary

Keep open. This PR targets a real gap from #71581, but the latest diff is still not safe to merge because it regresses exec completion trust, heartbeat classification, delivery-context preservation, and canonical session ownership checks on the same core routing path it is trying to harden.

## What This Changes

The PR adds a shared event-routing-key normalizer and rewrites exec notify-on-exit plus heartbeat session resolution so single-owner DM system events can route toward an agent main session.

## Best Possible Solution

Repair this branch narrowly: add dmScope=main single-owner routing after the existing canonicalization and ownership checks, and preserve the current exec completion trust, notify delivery context, structured summary, background-only guard, and exec-event heartbeat classification.

## Reproduction Assessment

Yes. The regressions are reproducible from the latest PR diff against current main and existing tests: the diff removes trusted:false, uses exec-exit/coalesMs, reads a non-existent deliveryContext field, drops the background guard, and bypasses canonical heartbeat session checks.

## Solution Assessment

No. The PR rewrites too much of the session and exec-notification contract; the maintainable path is a small dmScope-aware routing seam that leaves the existing safety and delivery behavior intact.

  - body: The notify-on-exit enqueue omits `trusted: false`. `enqueueSystemEvent` defaults missing trust to true, so command output can be injected as trusted System context instead of the current untrusted exec-completion path.

  - body: The wake now uses `reason: "exec-exit"`, and the immediate wake option is misspelled as `coalesMs`. The heartbeat classifier only recognizes `exec-event`, so these completions miss the exec-specific prompt/provider path and no longer request the zero-delay wake.
  - confidence: 0.94
- **[P1] Read the stored notify delivery context:** `src/agents/bash-tools.exec-runtime.ts:360-361`
  - body: `ProcessSession` stores the origin route in `notifyDeliveryContext`, but this reads `deliveryContext` through a loose cast. Delayed background exec completions will lose their pinned channel/topic and fall back to mutable session routing.
  - confidence: 0.93
- **[P1] Honor dmScope during routing:** `src/agents/bash-tools.exec-runtime.ts:380-382`
  - body: The normalizer is passed `cfg.session?.scope` instead of `cfg.session?.dmScope`. That mixes storage scope with DM routing scope, so configured `dmScope` values can be ignored or misinterpreted on the isolation path this PR is meant to fix.
  - confidence: 0.91
- **[P1] Preserve canonical heartbeat session checks:** `src/infra/heartbeat-runner.ts:319-327`
  - body: The new resolver returns raw forced/configured keys and falls back to `mainEntry`, skipping alias canonicalization and agent-id ownership checks. A mixed-case alias or another agent's key can select or drain the wrong heartbeat session.
  - confidence: 0.94
- **[P2] Keep notify-on-exit background-only:** `src/agents/bash-tools.exec-runtime.ts:325-326`
  - body: The current guard only sends notify-on-exit events for `session.backgrounded`; this replacement guard drops that condition. Foreground execs with notify-on-exit enabled can now enqueue system events and wake heartbeats.
  - confidence: 0.89
- **[P2] Keep structured exec completion summaries:** `src/agents/bash-tools.exec-runtime.ts:364-366`
  - body: The new completion text drops the exec id and output tail. Existing behavior and tests rely on those fields so the heartbeat has the command result context it is supposed to process.
  - confidence: 0.9
