## Summary

Keep this PR open, but it is not safe to merge in its current form. Current main still does not implement the linked dmScope-aware exec/cron heartbeat routing, and this patch drops several existing exec-notification and heartbeat-session contracts while trying to add that behavior.

## What This Changes

The PR adds a shared event-routing key normalizer and rewrites exec system-event notification plus heartbeat session resolution to route background execution and system events toward agent sessions.

## Best Possible Solution

Preserve the current exec notification and heartbeat session-resolution contracts, then add a narrow dmScope-aware routing helper that reads session.dmScope and the same single-owner allowlist inputs used by inbound DM routing. Cover exec-event and cron-event wakes with regression tests for dmScope=main, non-main dmScope isolation, subagent guards, configured heartbeat sessions, and delivery-context preservation.

## Reproduction Assessment

Yes. A high-confidence path exists from static diff plus existing tests: notify-on-exit wake tests and configured/forced heartbeat-session tests exercise the contracts this PR removes; a focused #71581 regression test should add dmScope=main single-owner exec/cron routing.

## Solution Assessment

No. The patch is broader than needed and rewrites stable exec/heartbeat behavior; the safer solution is a small normalization seam after existing canonicalization and before event drain/wake, without changing ProcessSession typing, output formatting, or subagent/agent ownership guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wake the heartbeat after queuing exec notifications:** `src/agents/bash-tools.exec-runtime.ts:358-362`
  - body: `maybeNotifyOnExit` now queues the completion event but never calls `requestHeartbeatNow`, so a background command can finish and leave the exec-event pending until an unrelated heartbeat. Current behavior wakes the scoped session immediately after enqueueing.
  - confidence: 0.94
- **[P2] Keep the notify-on-exit opt-in guard:** `src/agents/bash-tools.exec-runtime.ts:315`
  - body: The new function no longer checks `backgrounded` or `notifyOnExit` before queuing. Since `maybeNotifyOnExit` is called on normal process exits, foreground or non-opt-in execs with a session key would start creating system events.
  - confidence: 0.9
- **[P2] Use the real DM routing inputs:** `src/agents/bash-tools.exec-runtime.ts:349-352`
  - body: The normalizer is fed `session.scope` and `agents.defaults.allowFrom`, but the linked behavior is controlled by `session.dmScope` plus channel allowlists. A single-owner Telegram config like #71581 will not be normalized correctly, and isolated dmScope configs can be collapsed when `scope` is unset.
  - confidence: 0.91
- **[P1] Preserve canonical forced-session resolution:** `src/infra/heartbeat-runner.ts:267-275`
  - body: The heartbeat resolver now returns the raw forced/configured key and falls back to `mainEntry`, dropping `toAgentStoreSessionKey`, alias canonicalization, and agent ownership checks. Configured sessions such as `alerts` or mixed-case agent keys will not read the intended queue, and unrelated agent keys can be accepted.
  - confidence: 0.93
- **[P2] Read exit details from ProcessSession:** `src/agents/bash-tools.exec-runtime.ts:323-336`
  - body: The rewrite reads `exitCode` and `signal` from `session.process`, but `ProcessSession` stores `exitCode`, `exitSignal`, `tail`, and `aggregated` directly. Failed commands can be reported with exit code 0, and the useful output tail is dropped from the notification.
  - confidence: 0.88
