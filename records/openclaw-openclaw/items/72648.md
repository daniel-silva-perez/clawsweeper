## Summary

Keep open. The linked problem in #71581 is valid, but this PR branch introduces blocking regressions in exec completion notifications and heartbeat session resolution, and it weakens session-isolation checks on a security-sensitive routing path.

## What This Changes

The PR adds a shared event-routing key normalizer and rewrites exec system-event notification plus heartbeat session selection to route background execution events toward agent sessions based on session/config state.

## Best Possible Solution

Preserve the existing exec notification and heartbeat session-resolution contracts, then add a narrow dmScope-aware normalization seam after current canonicalization and agent ownership checks. The fix should read session.dmScope and the relevant channel/agent allowlist inputs, keep ProcessSession formatting and delivery context intact, and add regression coverage for #71581 exec-event and cron-event routing.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from the PR diff plus current tests: notify-on-exit wake tests and forced/configured heartbeat-session tests exercise the contracts the patch removes, and a focused #71581 test can cover dmScope=main single-owner rerouting. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No. This is not the best way to solve the linked issue because it rewrites stable exec and heartbeat routing behavior instead of adding a narrow dmScope-aware step on top of the existing canonicalization and ownership checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wake after queuing notify-on-exit events:** `src/agents/bash-tools.exec-runtime.ts:358-362`
  - body: maybeNotifyOnExit now stops after enqueueSystemEvent and never calls requestHeartbeatNow, so a background command can finish and leave its completion event pending until an unrelated heartbeat. Current main and docs expect an immediate exec-event wake after enqueueing.
  - confidence: 0.94
- **[P2] Restore the notify-on-exit opt-in guard:** `src/agents/bash-tools.exec-runtime.ts:317-318`
  - body: The rewritten maybeNotifyOnExit no longer checks backgrounded or notifyOnExit before queuing. Since this helper is called when a process exits, foreground or non-opt-in execs with a session key can start creating system events.
  - confidence: 0.9
- **[P2] Read exit details from ProcessSession:** `src/agents/bash-tools.exec-runtime.ts:326-338`
  - body: ProcessSession stores exitCode, exitSignal, tail, and aggregated directly, but the patch reads session.process and drops the output tail. Failed commands can be reported as completed with exit code 0, and the useful completion output is lost.
  - confidence: 0.88
- **[P1] Preserve canonical heartbeat session resolution:** `src/infra/heartbeat-runner.ts:266-274`
  - body: resolveHeartbeatSession now returns the raw forced or configured key and falls back to mainEntry, dropping alias canonicalization, agent ownership checks, and configured-session validation. Existing forced/configured session and subagent-guard cases can route or drain the wrong session.
  - confidence: 0.93
- **[P2] Use dmScope for routing decisions:** `src/agents/bash-tools.exec-runtime.ts:350-352`
  - body: The normalizer is passed session.scope, but the linked behavior is controlled by session.dmScope. A single-owner config with dmScope=main is not being checked through the intended field, and non-main dmScope settings can be collapsed incorrectly.
  - confidence: 0.91
- **[P2] Do not default normalized events to the main agent:** `src/agents/bash-tools.exec-runtime.ts:443-445`
  - body: emitExecSystemEvent adds agentId but callers are not updated, so opts.agentId is usually missing and resolveAgentMainSessionKey runs with an empty id. Under a single-owner allowlist this can rewrite another agent's event to the default main agent session.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff changes session-isolation routing and currently bypasses existing forced-session validation paths.

Concerns:

- **[medium] Forced heartbeat keys bypass ownership checks:** `src/infra/heartbeat-runner.ts:271`
  - body: resolveHeartbeatSession accepts the raw candidate key and uses store[candidateKey] or mainEntry after dropping current canonicalization and agent-id verification. Session-key-bearing wake APIs are part of the runtime surface, so this weakens isolation between agent sessions.
  - confidence: 0.86
- **[medium] Wrong DM scope source can collapse isolated DMs:** `src/agents/bash-tools.exec-runtime.ts:352`
  - body: The new normalizer reads session.scope instead of session.dmScope. When dmScope is explicitly non-main but the store scope is unset, a single-owner allowlist can still collapse events toward the main session.
  - confidence: 0.82
