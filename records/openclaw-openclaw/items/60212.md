## Summary

Keep open. Current main still lacks the requested empty-output cron repair, and the PR is useful source work, but it is not merge-ready because the CLI-backed repair path can lose the original session context and the changelog entry misses required attribution.

## What This Changes

The PR adds a cron empty-output repair pass, threads `disableTools` through embedded and CLI runs, disables bundled MCP for CLI repair mode, converts unrecovered empty delivery to an explicit error, and adds regression tests plus a changelog entry.

## Best Possible Solution

Implement the empty-output repair in the current cron executor architecture, keep the repair retry in the same CLI/live-session context while disabling tools and bundled MCP, fail explicitly when the retry still has no deliverable text, and keep the focused cron/CLI regression tests plus a credited changelog entry.

## Reproduction Assessment

Yes. The original bug has a clear mocked reproduction path: a cron isolated run with no deliverable payload currently reaches delivery with no synthesized text, and the PR’s own test plan models repair success and repair failure.

## Solution Assessment

No. The direction is right, but the proposed CLI repair changes the system prompt context for the retry; the safer fix is to disable tools/MCP without changing the reusable CLI session fingerprint or losing the original cron task context.

## Review Findings

- **[P1] Keep the repair pass in the existing CLI session:** `src/cron/isolated-agent/run.ts:532-542`
  - body: The repair retry adds a new `extraSystemPrompt` when `disableTools` is true.
