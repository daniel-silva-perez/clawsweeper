## Summary

Keep this issue open. Current main sends a Telegram typing cue before dispatching the agent reply, but only after `buildTelegramMessageContext` completes. That context-building path still resolves the inbound body and can run voice/audio preflight transcription before the typing cue is fired, so the requested receipt-time, pre-STT feedback is not implemented. The related #39075 PR is an open Telegram pipeline optimization candidate, but the provided PR context does not show a receipt-time `sendChatAction` fix.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the Telegram plugin’s inbound voice/audio path: send a best-effort `sendChatAction("typing")` after cheap eligibility/thread resolution but before media download and definitely before `transcribeFirstAudio`, with tests proving ordering and blocked-sender behavior. Add a generic channel/plugin seam only if the same pre-STT receipt cue is needed by multiple channel plugins.
