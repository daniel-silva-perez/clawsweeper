## Summary

Keep this issue open. Current main still waits for Telegram media resolution and then builds the inbound message context before sending the early typing cue; that context path can await audio preflight transcription, so the requested receipt-time, pre-STT voice feedback is not implemented. The duplicate #58889 was already closed in favor of this canonical tracker, and #39075 remains an open broad optimization PR rather than a shipped fix.

## What This Changes

The issue requests that Telegram voice messages trigger a typing indicator as soon as the message is received, before audio download or STT transcription delays the agent loop.

## Best Possible Solution

Implement a Telegram-plugin-only receipt-time typing cue for authorized inbound voice/audio messages after cheap access/thread resolution but before media download and transcribeFirstAudio, using the existing sendChatAction backoff/thread helpers and preserving the current model-loop typing behavior as a later refresh path.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: send an authorized Telegram DM voice note with audio preflight enabled and observe that current main reaches resolveMedia and transcribeFirstAudio before the existing sendTyping call in createTelegramMessageProcessor.

## Solution Assessment

Yes, with a narrower boundary than the issue's unconditional wording: the maintainable fix is to emit a best-effort Telegram typing action only after cheap eligibility checks, not from core startup or before authorization. A generic core seam should wait until multiple channel plugins need the same pre-STT contract.
