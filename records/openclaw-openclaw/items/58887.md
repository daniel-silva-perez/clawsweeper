## Summary

Keep this open. Current main still sends Telegram typing only after the message context is built, and that context build awaits voice-note media handling plus `transcribeFirstAudio`, so the reported pre-STT feedback gap remains. The related Telegram performance PR is open and adjacent, but it does not supersede this specific typing-order request.

## What This Changes

Request to show a Telegram typing indicator as soon as an authorized voice message is received, before audio download/transcription finishes.

## Best Possible Solution

Add a small authorized early-typing path for Telegram voice/audio messages before media download and STT, with focused tests proving the cue fires before `resolveMedia`/`transcribeFirstAudio` for allowed messages and does not fire for blocked messages.

## Reproduction Assessment

Yes. The static call path is enough to reproduce the ordering: `resolveMedia` and `transcribeFirstAudio` complete before `context.sendTyping()` can run, and a focused unit test can assert the same with mocks.

## Solution Assessment

Yes, a narrow Telegram handler fix is the best first step. It should reuse the existing `sendChatActionHandler`/threading helpers and preserve current access gates rather than moving broad policy into core.
