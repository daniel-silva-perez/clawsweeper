## Summary

Keep this open. Current main still uses path-backed filesystem session persistence: `session.store` resolves to `sessions.json`, transcripts resolve to per-session `.jsonl` files, and the plugin/runtime helpers expose those same file-store helpers rather than a replaceable Redis/MongoDB/PostgreSQL-style backend contract. The existing ClawSweeper comment correctly kept this open, and current main at 17811480 still has the same core gap. Related #57497 is narrower PostgreSQL-specific work and does not supersede this broader unified storage request.

## Best Possible Solution

Keep this open for a core session persistence design. The likely path is a versioned storage contract covering both `sessions.json` metadata and JSONL transcript/history semantics, filesystem as the default backend, config/schema/docs plus migration/doctor support, one proven multi-instance-safe backend, and then a narrow plugin-facing backend registration API for additional stores. Related #57497 can stay linked as a narrower PostgreSQL-specific request but should not replace this broader unified storage issue.

## Risks / Open Questions

- Closing as implemented would be premature because both metadata and transcripts still depend on local filesystem semantics.
- Closing as ClawHub would be premature because current plugin APIs can annotate/project session state and use path-backed helpers, but cannot replace the core session metadata and transcript persistence backend.
- A correct implementation needs maintainer design for distributed concurrency/locking, transcript append and replay semantics, metadata consistency, TTL/pruning, migration/doctor behavior, config/schema/docs, and secret handling for external backend credentials.
