## Summary

Keep open. Current main still persists session metadata and transcripts through local filesystem paths, and the public config/SDK surfaces expose path-backed helpers rather than a replaceable Redis/MongoDB/PostgreSQL-style storage backend. The existing ClawSweeper comment correctly kept this open; related #57497 was narrower PostgreSQL-only work and does not supersede this broader unified storage request.

## What This Changes

Request a pluggable external session storage abstraction so multiple OpenClaw Gateway instances can share session metadata and transcripts without sticky routing.

## Best Possible Solution

Keep this as a core session persistence design item: define a versioned storage contract covering sessions.json-equivalent metadata and JSONL transcript semantics, keep filesystem as the default backend, add config/schema/docs/doctor and migration behavior, prove one distributed-safe backend, then expose a narrow plugin-facing backend registration API for additional stores.
