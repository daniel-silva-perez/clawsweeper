## Summary

Keep this issue open. Current main still persists sessions through local filesystem metadata (`sessions.json`) plus per-session JSONL transcript files, and the config/plugin surfaces remain path-backed rather than a replaceable Redis/MongoDB/PostgreSQL-style storage backend. The existing ClawSweeper comment correctly kept this open, and the newer main SHA still has the same core gap.

## Best Possible Solution

Keep this open for a core session persistence design. The likely path is a versioned storage contract covering both `sessions.json` metadata and JSONL transcript/history semantics, filesystem as the default backend, config/schema/docs plus migration/doctor support, one proven multi-instance-safe backend, and then a narrow plugin-facing backend registration API for additional stores. Related #57497 can remain linked as a narrower PostgreSQL-specific request but does not supersede this broader unified storage issue.

## Risks / Open Questions

- A plugin-only or ClawHub close would be premature because current plugin APIs can annotate or inspect session state but cannot replace core metadata and transcript persistence.
- A correct implementation needs maintainer design for distributed concurrency and locking, transcript append/replay semantics, metadata consistency, TTL/pruning, migration/doctor behavior, config/schema/docs, and secret handling for backend credentials.
