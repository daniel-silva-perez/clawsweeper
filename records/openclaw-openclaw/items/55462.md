## Summary

Keep open. Current main still sends generic Telegram inline-button callbacks through the agent path with the callback query id as the message id, while the originating Telegram message id is only available to plugin interactive handlers; the linked fix PR remains open and unmerged, so this issue should not be closed yet.

## What This Changes

Expose the originating Telegram callback_query message_id to agents so inline-button callbacks can call editMessage on the source message.

## Best Possible Solution

Keep this issue open until PR #55548 or an equivalent maintainer-reviewed fix lands. The fix should expose callbackMessage.message_id and chat_id to the generic agent-visible callback metadata or prompt context, add a focused regression test for inline-button callbacks driving editMessage, and preserve callback query ids where they are needed for callback answering or dedupe.

## Reproduction Assessment

Yes. A source-level reproduction is enough: a Telegram callback_query with message.message_id reaches the fallback path, but current main passes callback.id as messageIdOverride, and MessageSid then becomes that override.

## Solution Assessment

Yes, keeping this open is the best current handling. The requested behavior is narrow and valid, but the existing open closing PR should be reviewed, rebased, or replaced rather than closing this issue before a fix lands.
