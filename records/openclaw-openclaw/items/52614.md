## Summary

Keep open. Current main still lacks historical embedded subscribe replay suppression, but this PR is not merge-ready against current main because the assistant eligibility helper needs to preserve the existing delivery-mirror/gateway-injected transcript-only filter after rebase.

## What This Changes

The PR snapshots preexisting embedded session messages, suppresses initial replay/preexisting assistant lifecycle handling in embedded subscribe, and adds a regression test for attaching to a session with historical assistant output.

## Best Possible Solution

Land or replace a rebased narrow embedded subscribe fix that snapshots preexisting session messages, suppresses only historical replay/preexisting assistant outward delivery, preserves fresh assistant replies and current transcript-only filters, and keeps focused regression coverage for the attach-to-existing-session case.

## Reproduction Assessment

Yes. The proposed synchronous replay harness gives a high-confidence reproduction path: replay `agent_start`, historical assistant `message_start`/`message_end`, and `agent_end` during `session.subscribe`, then verify current main would treat the historical assistant final like a fresh outward event because no replay/preexisting guard exists.

## Solution Assessment

No, not as-is against current main. The implementation direction is narrow and maintainable, but the PR needs a rebase that folds the current transcript-only assistant predicate into the new eligibility helper before merge.
