## Summary

Keep open. Current main has partial WebChat audio support: the picker advertises `audio/*`, `chat.send` accepts non-image attachments, Gateway can build assistant audio content blocks, and normalized audio attachments render with HTML audio controls. The remaining user-visible gaps are still concrete: `audio/webm` uploads are rejected by filename, and Gateway-produced `{type:"audio", source:{...}}` TTS content blocks are not normalized/rendered as playable audio attachments in the UI.

## Best Possible Solution

Keep this issue open and make one focused WebChat audio follow-up: accept `audio/webm` uploads without accepting video WebM, normalize/render Gateway-produced assistant audio content blocks as playable audio attachments, preserve existing `attachment.kind === "audio"` playback and non-image `chat.send` routing, and add targeted UI/media/Gateway regression coverage.
