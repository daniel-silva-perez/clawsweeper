## Summary

Close as implemented. Current main accepts audio files in the Control UI/WebChat picker, sends non-image attachments with their MIME type, classifies audio attachments, and renders received audio media with an inline HTML audio player; the fix is included in a shipped release.

## What This Changes

Add audio MIME support to WebChat uploads and render received audio attachments with inline playback controls.

## Best Possible Solution

Keep the shipped WebChat implementation: audio upload uses the existing non-video attachment path, received audio uses the shared attachment renderer with browser-native controls, and remaining TTS telemetry/status work should stay with the separate #57297 follow-up.

## Reproduction Assessment

Yes. The focused reproduction path is to inspect or exercise the Control UI file picker with an audio file and render a MEDIA audio attachment; current main’s source and focused UI tests show those checks now pass instead of reproducing the reported behavior.

## Solution Assessment

Yes. The current solution is appropriately narrow: it broadens the picker to `audio/*`, preserves MIME metadata through `chat.send`, and adds inline playback in the existing attachment renderer without adding a new core channel or provider contract.
