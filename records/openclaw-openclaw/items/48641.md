## Summary

Keep this issue open. Current main still treats Discord account-level DM allowlists as taking precedence over the top-level allowlist rather than unioning them, and the related implementation PR was closed unmerged; because this is access-control behavior, the remaining decision should be made explicitly by maintainers.

## What This Changes

The issue reports Discord inbound DMs being dropped in a multi-account `dmPolicy: "allowlist"` setup even though outbound DMs and guild channel messages work.

## Best Possible Solution

Define the intended Discord DM allowlist semantics for multi-account setups, then update the Discord owner module and preflight/runtime tests to match that contract. If top-level `allowFrom` is meant to be shared, implement a documented union or inheritance rule, add regression coverage for default and named accounts, and update the Discord docs plus changelog.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: current code shows that a sender present only in top-level `channels.discord.allowFrom` is excluded when the selected account has its own `allowFrom`, and DM preflight then denies the message. The reporter's exact four-account live path still needs account/token confirmation.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the security semantics; the direct union proposed by #49365 may solve the reported drop but could widen access. The safer path is to first decide and document whether top-level `allowFrom` is shared, inherited only when unset, or overridden per account.
