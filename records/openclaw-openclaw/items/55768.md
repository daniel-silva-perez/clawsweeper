## Summary

Keep open. Current main still routes anonymous OpenAI-compatible chat-completion requests through the persisted agent-session path, while the gateway docs still lack clear external-monitor guidance telling uptime checks to use the existing lightweight health/readiness probes instead. The linked docs PR that directly addressed the gap was closed unmerged, and the broader passthrough-mode issue does not supersede this narrower monitoring documentation cleanup.

## What This Changes

The issue asks OpenClaw to prevent uptime-monitor session bloat by documenting and optionally warning/rate-limiting callers so health checks use `/health` or `/healthz` instead of `POST /v1/chat/completions`.

## Best Possible Solution

Add public gateway monitoring guidance that keeps the existing lightweight probes as the recommended contract: use `/health` or `/healthz` for liveness, use `/readyz` when readiness matters, and avoid `/v1/chat/completions` for uptime checks because it is a full agent endpoint that creates a persisted session without a stable session key or `user`. Cross-link that guidance from the OpenAI HTTP docs and leave passthrough mode, warnings, and rate limits to separately scoped product/runtime work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: `POST /v1/chat/completions` without `x-openclaw-session-key` or `user` resolves to an `openai:<uuid>` agent session key and calls `agentCommandFromIngress`, while `/healthz` returns a shallow live-probe response without loading the full request path.

## Solution Assessment

Yes for the narrow repair lane. A docs-first fix is the smallest maintainable step because the lightweight endpoints already exist and are tested; changing session persistence, adding warning rate counters, or rate-limiting anonymous session creation would require separate runtime/product review.
