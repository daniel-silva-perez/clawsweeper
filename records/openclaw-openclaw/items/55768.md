## Summary

Keep open. Current main has lightweight probe endpoints and some deployment templates use `/health` or `/healthz`, but the reported OpenAI-compatible path still creates a fresh `openai` session key for anonymous `/v1/chat/completions` requests and runs the normal agent command path. The docs still describe that behavior as “stateless per request” without warning uptime monitors away from the endpoint, and there is no gateway warning or rate limit for high-rate anonymous session creation. Related PR #55785 attempted docs but was closed unmerged; open #62436 tracks broader stateless passthrough, not the narrower operational health-check guidance.

## Best Possible Solution

Keep this open and implement the narrow operational fix: document `/health`/`/healthz` as the health-check endpoints in `docs/gateway/health.md` and `docs/gateway/openai-http-api.md`, explicitly warn that `/v1/chat/completions` creates agent sessions and may make LLM calls, update deployment examples where needed, and consider a low-noise Gateway warning for high-rate anonymous OpenAI-compatible session creation. Track any true no-persistence relay mode in #62436 rather than folding it into this health-check documentation task.
