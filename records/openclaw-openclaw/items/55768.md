## Summary

Keep open. Current main still has the reported operational footgun: anonymous OpenAI-compatible chat requests create fresh persisted agent sessions and run the normal agent ingress path, while the docs only partially show lightweight probe endpoints and still do not clearly warn uptime monitors away from `/v1/chat/completions`. The related docs PR was closed unmerged, and the broader passthrough-mode issue does not supersede this narrower monitoring guidance cleanup.

## What This Changes

The issue requests health-check documentation and deployment guidance to direct uptime monitors to `/health` or probe endpoints, plus optional gateway diagnostics/rate limiting for repeated session-creating chat probes.

## Best Possible Solution

Keep the existing lightweight probe endpoints and add clear public monitoring guidance in the gateway health/OpenAI HTTP docs: `/health` or `/healthz` for liveness, `/readyz` where readiness matters, and `/v1/chat/completions` labeled as a full agent endpoint that creates persisted sessions when callers omit a stable session key or `user`. Runtime detection or rate limiting should be handled separately as generic gateway diagnostics work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: send `POST /v1/chat/completions` without `x-openclaw-session-key` and without an OpenAI `user`; current code generates an `openai:<uuid>` agent session key and enters `agentCommandFromIngress`, while `/health` and `/healthz` return a shallow probe response.

## Solution Assessment

Yes for the immediate docs-first repair, because the problematic endpoint behavior is currently documented without the health-monitor warning and the safe probe endpoints already exist. Runtime warnings, metrics, passthrough mode, or session rate limits are not the narrowest fix for this issue and should be separately scoped.
