## Summary

Keep open. Current main still sends anonymous OpenAI-compatible chat-completion requests through the normal persisted agent-session path, while the public gateway docs do not clearly warn uptime monitors to use the lightweight health/readiness probes instead. The prior docs PR that explicitly addressed this was closed unmerged, and the open passthrough-mode issue is broader than this monitoring guidance cleanup.

## What This Changes

The issue requests monitoring docs/deployment guidance and optional gateway safeguards so uptime monitors use `/health`, `/healthz`, or `/readyz` instead of `POST /v1/chat/completions`.

## Best Possible Solution

Keep the existing probe endpoints and add clear public monitoring guidance in the gateway health/OpenAI HTTP docs: use `/health` or `/healthz` for liveness, `/readyz` where readiness matters, and do not use `/v1/chat/completions` for uptime checks because it is a full agent endpoint that creates a persisted session when callers omit a stable session key or `user`.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: send `POST /v1/chat/completions` without `x-openclaw-session-key` and without an OpenAI `user`; current code generates an `openai:<uuid>` agent session key and enters `agentCommandFromIngress`, while `/healthz` returns a shallow live probe response.

## Solution Assessment

Yes for a docs-first repair. The narrowest maintainable fix for this issue is to document and cross-link the existing health endpoints and warn against monitoring the chat-completion endpoint; passthrough mode, runtime warnings, and rate limits are separate product/runtime decisions.
