## Summary

Keep open. Current main already has lightweight unauthenticated HTTP probes and some deployment docs use them, but the OpenAI-compatible chat endpoint still generates a fresh `openai:<uuid>` agent session and runs the normal agent ingress path when callers omit `user` and `x-openclaw-session-key`. The specific monitoring guidance and warning against using `/v1/chat/completions` as an uptime probe is still missing; the related docs PR was closed unmerged, and the broader passthrough-mode issue does not supersede this narrower health-check cleanup.

## What This Changes

Request deployment guidance and safeguards so uptime monitors use the lightweight `/health` probe instead of `/v1/chat/completions`, avoiding per-ping agent sessions and LLM calls.

## Best Possible Solution

Keep the existing lightweight probe endpoints, add clear public monitoring guidance under the gateway health/OpenAI HTTP docs, and explicitly label `/v1/chat/completions` as a full agent endpoint that creates persisted sessions. If runtime detection is desired, implement it separately as a generic gateway diagnostics feature rather than a health-docs-only repair.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: omit both `user` and `x-openclaw-session-key` on `POST /v1/chat/completions`, and current code generates a new `openai:<uuid>` session key and sends it through `agentCommandFromIngress`; `/health` is the contrasting shallow path.

## Solution Assessment

Unclear as a full runtime solution. Documentation is the narrowest immediate fix for misconfigured monitors, but warning/metrics/rate limiting should be a separately scoped maintainer decision because it changes gateway operational behavior.
