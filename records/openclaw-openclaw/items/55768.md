## Summary

Keep open. Current main still sends anonymous OpenAI-compatible chat-completion requests through the persisted agent-session path, while public gateway docs still do not give external uptime-monitor guidance to use `/health`, `/healthz`, or `/readyz` instead of `/v1/chat/completions`. The closed docs PR that directly targeted this issue was not merged, and the broader passthrough-mode feature request does not supersede this narrow monitoring documentation gap.

## What This Changes

The issue requests monitoring guidance and guardrails so uptime checks use lightweight gateway health endpoints instead of creating persisted `/v1/chat/completions` agent sessions.

## Best Possible Solution

Add public gateway monitoring guidance that names the lightweight contract: use `GET /health` or `/healthz` for liveness, `/readyz` when readiness matters, and avoid `/v1/chat/completions` for uptime checks because it is a full agent endpoint that can create persisted sessions and call an LLM. Cross-link that guidance from the OpenAI HTTP API docs and keep passthrough mode, warning counters, and rate limits as separately scoped follow-ups.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: a `POST /v1/chat/completions` request without `x-openclaw-session-key` or `user` resolves to an `openai:<uuid>` session key and enters the agent command path, while `/healthz` returns a shallow liveness response.

## Solution Assessment

Yes for the narrow repair lane. A docs-first fix is the smallest maintainable step because the lightweight probe endpoints already exist and are tested; changing session persistence, warning heuristics, or rate limits should be separate runtime/product work.
