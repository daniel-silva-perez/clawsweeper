## Summary

Keep this issue open. Current main still has targeted cleanup guards and some narrow lifecycle helpers, but it does not implement the requested standard async resource lifecycle protocol across retry/tool, cleanup, and plugin/channel paths. One motivating Feishu cleanup path is still unresolved, and the related retry/tool PR remains open rather than merged.

## Best Possible Solution

Keep this issue open as the broad lifecycle-design tracker. Maintainers should review or close PR #48512 for the retry/tool-call cluster, fix the Feishu HTTP cleanup path directly, and then decide whether to add a small shared lifecycle contract in core or the plugin SDK. A good design should define stop-accepting-new-work, drain or cancellation behavior, timeout and forced-dispose semantics, tests for retry and server-close races, and clear owner boundaries before migrating call sites.

## Risks / Open Questions

- Closing now would drop the broad lifecycle-design tracker while the Feishu cleanup race remains visible on current main and PR #48512 is still an open, narrower retry/tool candidate.
- A broad ResourceManager could over-centralize plugin-owned behavior; the next design should preserve owner boundaries and expose only generic lifecycle seams third-party plugins can use.
- The current helper landscape is partial: keepHttpServerTaskAlive can likely help HTTP channel cleanup, but retry/tool cleanup needs a separate drain or cancellation contract.
