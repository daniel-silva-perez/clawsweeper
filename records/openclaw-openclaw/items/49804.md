## Summary

Keep this issue open. Current main still has targeted cleanup guards, but not the requested standard ResourceManager/shutdown protocol across retry, cleanup, and plugin/channel lifecycle paths. The related retry/tool PR remains open, and the Feishu cleanup path still closes HTTP servers while immediately deleting state.

## Best Possible Solution

Keep this issue open as the broad lifecycle-design tracker. Maintainers should review or close PR #48512 separately for the retry/tool-call cluster, resolve the Feishu HTTP cleanup path directly, then decide whether a small shared core or plugin-SDK lifecycle contract is warranted. A good design should define stop-accepting-new-work, drain or cancellation behavior, timeout/forced-dispose semantics, and owner boundaries before migrating call sites.

## Risks / Open Questions

- Closing now would hide that a motivating cleanup path, Feishu HTTP server shutdown, still has immediate state deletion around asynchronous `server.close()`.
- Treating PR #48512 as resolving this issue would conflate a narrow retry/tool-call guard with the broader lifecycle protocol requested here; it is still open in the provided context.
- A future fix could overreach if implemented as a large cross-cutting abstraction. The safer path is a small typed shutdown/drain contract adopted only where ownership boundaries are clear.
