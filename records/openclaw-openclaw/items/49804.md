## Summary

Keep open. Current main has several targeted cleanup primitives, but it does not implement the requested standard ResourceManager/shutdown protocol across retry and cleanup paths. The retry/tool path still uses a narrow wait-for-idle flush guard, Feishu monitor cleanup still closes HTTP servers and immediately deletes/clears state, and the provided discussion/related context shows the author and commenters already narrowed some related bugs into separate targeted fixes rather than resolving the broader lifecycle design.

## Best Possible Solution

Keep this issue open as the broad lifecycle-design tracker. Maintainers should review or close PR #48512 on its own merits for the retry/tool-call cluster, resolve the Feishu cleanup path separately, then decide whether a small shared core/SDK lifecycle contract is warranted. A good implementation would define stop-accepting-new-work, drain or cancellation behavior, timeout/forced-dispose semantics, and owner boundaries before migrating call sites.

## Risks / Open Questions

- Closing now would hide that at least one motivating cleanup path is still unresolved and that current fixes remain local, best-effort guards rather than a shared lifecycle protocol.
- A future fix should avoid a large cross-cutting abstraction without clear owner boundaries; the safer path is a small, typed shutdown/drain contract with targeted adoption in the retry/tool and plugin/channel cleanup paths.
