## Summary

Keep open. Current main still does not add the requested startup dangerous-config warnings, and this PR is useful source work, but the branch is now dirty against the refactored dangerous-flag helper and still has a concrete security-warning gap for trusted-proxy allowUsers semantics.

## What This Changes

The PR branch adds startup dangerous-config warnings for explicit `agents.defaults.sandbox.mode="off"`, `gateway.auth.mode="none"`, and an explicit empty `gateway.auth.trustedProxy.allowUsers` array.

## Best Possible Solution

Rebase the branch onto current main, add the explicit sandbox-off startup warning in the current core dangerous-flag helper, cover it with focused tests, and either drop the extra auth warnings or make them match runtime and audit semantics, especially `allowUsers ?? []`.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a config with `gateway.auth.mode="trusted-proxy"`, a valid `trustedProxy.userHeader`, and omitted `trustedProxy.allowUsers`: runtime auth treats that as an empty allowlist, while the PR collector skips the warning because `Array.isArray(undefined)` is false.

## Solution Assessment

No. The explicit sandbox-off equality check is the right narrow direction, but the PR is not the best current-main fix until it is rebased into the refactored helper and the trusted-proxy warning is normalized consistently with runtime and audit behavior.
