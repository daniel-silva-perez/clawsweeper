## Summary

Keep open. The sandbox-off warning is still useful, but this PR is not merge-ready against current main: the dangerous-config helper has moved to a core/helper split, and the PR’s added trusted-proxy startup warning misses the omitted-allowUsers case that runtime, configure, docs, and audit all treat as allow-all.

## What This Changes

This PR adds dangerous startup warnings for explicit `agents.defaults.sandbox.mode="off"`, `gateway.auth.mode="none"`, and `gateway.auth.trustedProxy.allowUsers=[]` in `src/security/dangerous-config-flags.ts`.

## Best Possible Solution

Rebase onto current main, add the explicit sandbox-off startup warning in the current core dangerous-flag helper, add focused tests proving absent sandbox config stays quiet and explicit `mode="off"` warns, then either remove the auth-warning expansion or make trusted-proxy allowlist detection match runtime/audit semantics with `allowUsers ?? []`.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="trusted-proxy"`, a configured `trustedProxy.userHeader`, and omitted `trustedProxy.allowUsers` reproduces the PR bug by static inspection: runtime allows all users, but the PR condition does not warn because `Array.isArray(undefined)` is false.

## Solution Assessment

No. The explicit sandbox-off equality check is the right narrow direction, but the current PR is not the best solution until it is rebased and the trusted-proxy warning is normalized or split out of scope.
