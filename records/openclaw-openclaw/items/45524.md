## Summary

Keep open. Current main does not already implement the requested startup warning, and the PR is not safe to merge as submitted because the added trusted-proxy warning misses the omitted-allowUsers path that runtime treats as allow-all.

## What This Changes

The PR modifies `src/security/dangerous-config-flags.ts` to add startup dangerous-config warnings for explicit sandbox-off mode, disabled gateway auth, and an empty trusted-proxy user allowlist.

## Best Possible Solution

Rebase onto current main, add the explicit sandbox-off warning in the current core dangerous-flag helper with tests proving absent sandbox config stays quiet and explicit `mode="off"` warns, then either remove the auth-warning expansion or make it match runtime/audit semantics for omitted and empty trusted-proxy allowlists.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="trusted-proxy"`, a configured `trustedProxy.userHeader`, and omitted `trustedProxy.allowUsers` reproduces the PR bug because runtime treats the omitted list as allow-all while the PR warning only detects an explicit empty array.

## Solution Assessment

No. The explicit sandbox-off equality check is the right narrow direction, but the current PR is not the best solution until it is rebased and the trusted-proxy warning is normalized or split out of scope.
