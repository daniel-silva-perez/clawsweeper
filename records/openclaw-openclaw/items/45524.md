## Summary

Keep open. The requested startup warning is still not present on current main, and this PR cannot merge as-is because it is dirty against the current helper split and its added trusted-proxy warning misses the omitted `allowUsers` case that runtime treats as allow-all. The original sandbox absent-config false positive was fixed in the PR head.

## What This Changes

The PR adds dangerous-config startup warnings for explicit `agents.defaults.sandbox.mode="off"`, `gateway.auth.mode="none"`, and trusted-proxy `allowUsers=[]` in `src/security/dangerous-config-flags.ts`.

## Best Possible Solution

Rebase onto current main, implement the explicit sandbox-off startup flag in the current core dangerous-flag helper with tests proving absent sandbox config stays quiet and explicit `mode="off"` warns, then either remove the auth-warning expansion or make it match runtime/audit semantics with `allowUsers ?? []` coverage.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="trusted-proxy"`, a valid `trustedProxy.userHeader`, and omitted `trustedProxy.allowUsers` reproduces the PR bug because runtime treats the omitted list as allow-all while the PR warning only detects an explicit empty array.

## Solution Assessment

No. The explicit sandbox-off equality check is the right narrow direction, but the current PR is not the best solution until it is rebased and the trusted-proxy warning is normalized or split out of scope.
