## Summary

Keep open. Current main still lacks the requested startup dangerous-config warning, so the PR is not obsolete, but the branch is currently dirty against main and the added trusted-proxy warning misses the omitted-allowUsers case that runtime treats as allow-all.

## What This Changes

The PR adds startup dangerous-config warnings for explicit `agents.defaults.sandbox.mode="off"`, `gateway.auth.mode="none"`, and explicit empty `gateway.auth.trustedProxy.allowUsers` in `src/security/dangerous-config-flags.ts`.

## Best Possible Solution

Rebase onto current main, put the explicit sandbox-off warning in the current core dangerous-flag helper, add focused tests around absent vs explicit sandbox mode, and either drop the extra auth warnings or normalize them to the same runtime/audit contracts.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="trusted-proxy"`, a valid `trustedProxy.userHeader`, and omitted `trustedProxy.allowUsers` exercises the PR bug: runtime authorizes any proxy-authenticated user while the PR's new warning does not fire.

## Solution Assessment

No. The explicit sandbox-off equality check is the right narrow direction, but the PR is not the best current-main fix until it is rebased and the trusted-proxy warning is made consistent with `allowUsers ?? []` semantics or removed from scope.
