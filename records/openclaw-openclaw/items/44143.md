## Risks / Open Questions

- Closing this PR as implemented would be incorrect because current main still lacks a per-destination outbound delivery serializer.
- Merging the current diff as-is may still fail FIFO-by-call-initiation because the proposed lock is acquired after enqueueDelivery, so queue I/O completion order can invert same-target sends.
- The proposed key uses raw accountId/to and omits normalized recipient/thread identity, which can either bypass serialization for equivalent destinations or serialize distinct threaded conversations unnecessarily.
- The proposed queue can introduce head-of-line blocking if a channel send hangs; current Discord webhook sends use fetch without an obvious timeout in the inspected path.
- The PR contains unrelated Mattermost package and lockfile dependency churn, which is a supply-chain/package-resolution change outside the stated outbound serializer fix.
