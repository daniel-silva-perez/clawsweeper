## Risks / Open Questions

- Closing this PR as implemented would be incorrect because current main and v2026.4.26 still lack a destination-level outbound delivery serializer.
- Merging the current diff as-is may still fail FIFO-by-call-initiation because the proposed lock is acquired after enqueueDelivery, so queue I/O completion order can invert same-target sends.
- The proposed key uses raw accountId/to and omits normalized recipient/thread identity, so equivalent destinations can bypass serialization and distinct threaded conversations can be unnecessarily serialized together.
- The serializer design can introduce head-of-line blocking if a channel send hangs; the inspected Discord webhook path uses fetch without an obvious timeout in that call.
- The PR contains unrelated Mattermost package and lockfile dependency churn, which is a supply-chain/package-resolution change outside the stated outbound serializer fix.
