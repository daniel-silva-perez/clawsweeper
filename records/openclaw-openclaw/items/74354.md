## Summary

Keep open. Current main still lacks the Control UI composer safeguards and regression coverage for numeric-only chat input, and open PR #74369 explicitly targets this issue with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The requested fix is to preserve pure numeric Control UI webchat messages exactly as typed instead of allowing browser or OS input behavior to rewrite them with separators.

## Best Possible Solution

Land PR #74369, or an equivalent narrow Control UI change, so the chat composer discourages browser/OS numeric rewriting and tests exact numeric payload preservation without adding server-side deformatting that could corrupt intentional hyphenated input.

## Reproduction Assessment

Unclear. The report has concrete steps and a screenshot, and static inspection shows the current path would send a rewritten textarea value unchanged, but this pass did not live-reproduce the Windows/browser/IME-specific rewrite.

## Solution Assessment

Yes, the linked direction is the best narrow fix: add plain-text composer attributes and UI/send-boundary tests, rather than changing Gateway sanitization to guess when hyphens are unwanted.
