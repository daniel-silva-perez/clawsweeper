## What This Changes

The PR adds a Matrix guarded-fetch helper, transport hook, targeted tests, and changelog entry to rewrite narrowly matched `/keys/upload` one-time-key collision 400 responses into a synthetic 200 with empty `one_time_key_counts`.

## Best Possible Solution

Review and land the narrow Matrix-owned transport mitigation if maintainers agree with the temporary synthetic response semantics, while leaving matrix-rust-sdk issue #6520 as the durable upstream OTK ID-tracking fix.

## Reproduction Assessment

Yes for the failure mechanics: current main passes the 400 through unchanged, matrix-js-sdk does not retry non-429 4xx responses, and the upstream issue gives concrete steps and the exact collision body. I did not independently reproduce against a live homeserver in this read-only review.

## Solution Assessment

Yes, with maintainer sign-off: the proposed mitigation is scoped to the Matrix SDK fetchFn path and excludes non-collision 400s and other endpoints. The safer long-term solution remains an upstream matrix-rust-sdk fix for OTK ID generation/tracking.
