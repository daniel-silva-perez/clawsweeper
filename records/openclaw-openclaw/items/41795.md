## Summary

Keep open. Current main supports configured Telegram proxies and standard HTTP(S) env proxy routing, but it still does not implement the PR's Telegram-specific `OPENCLAW_TELEGRAM_PROXY` override or resolver, and the submitted diff targets the old `src/telegram` layout rather than the current `extensions/telegram` plugin.

## Best Possible Solution

Refresh the PR narrowly against `extensions/telegram`: add a Telegram-owned resolver with precedence `channels.telegram[.accounts.<id>].proxy` > `OPENCLAW_TELEGRAM_PROXY` > existing standard HTTP(S) env-proxy behavior, thread it through send, monitor/webhook, probe, audit, and chat lookup paths, and add focused tests for precedence, blanks, malformed values, NO_PROXY behavior, and redaction. If maintainers decide the existing configured proxy plus standard env proxy support is sufficient product surface, close after that explicit decision.
