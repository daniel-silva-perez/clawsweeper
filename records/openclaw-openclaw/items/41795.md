## Summary

Keep this PR open. Current main supports configured Telegram proxies and standard HTTP(S) env proxy routing, but it still does not implement the PR's Telegram-specific `OPENCLAW_TELEGRAM_PROXY` override. The submitted diff is also stale against the current bundled plugin layout and needs a security-focused refresh before it could be considered mergeable.

## Best Possible Solution

Refresh the PR narrowly against `extensions/telegram`: add a Telegram-owned resolver with precedence `channels.telegram[.accounts.<id>].proxy` > `OPENCLAW_TELEGRAM_PROXY` > existing standard HTTP(S) env-proxy behavior, thread it through send, monitor/webhook/polling, probe, audit, and chat lookup paths, and add focused tests for precedence, blank values, malformed values, `NO_PROXY` interaction, and credential-redacted logging. If maintainers decide configured proxy plus standard env proxy support is enough product surface, close after that explicit product decision.
