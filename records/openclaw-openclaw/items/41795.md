## Summary

Keep this PR open. Current main has configured Telegram proxy support and standard HTTP(S) env-proxy routing for Telegram transports, but it still does not implement the PR's Telegram-specific `OPENCLAW_TELEGRAM_PROXY` override, and the submitted diff targets the old `src/telegram` layout rather than the current `extensions/telegram` plugin.

## Best Possible Solution

Keep this PR open for a narrow refresh against `extensions/telegram`. If maintainers want the override, implement a Telegram-owned resolver with precedence `channels.telegram[.accounts.<id>].proxy` > `OPENCLAW_TELEGRAM_PROXY` > existing standard HTTP(S) env-proxy behavior, then thread it through send, monitor/webhook, probe, audit, and chat lookup paths with focused tests for precedence, blanks, malformed values, NO_PROXY behavior, and redaction. If configured/global proxy support is considered sufficient, close only after that product decision is explicit.
