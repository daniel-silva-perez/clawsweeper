## Summary

Keep this issue open. Current main still has the reported degraded Discord DM path: inbound message preflight only marks a DM when fetched channel info says `ChannelType.DM`; channel-info failures return `null`; route construction then falls through to a `channel` peer/session; and session-key normalization only repairs contexts that were already classified as direct.

## Best Possible Solution

Keep this issue open and fix the Discord-owned inbound preflight classification path. The likely implementation is a small helper that derives Discord chat kind from `channelInfo` first, then from trusted payload/channel/sender metadata when `guild_id` is absent and channel lookup fails, while preserving group-DM handling. Add a regression test where `fetchChannel` returns `null` or throws for an absent-guild one-to-one DM and the resolved route/session remains `agent:<agent>:discord:direct:<userId>`.
