## Summary

Keep this issue open. Current main has adjacent Discord DM continuity work, including direct-user binding lookup and explicit session-key normalization for contexts already marked direct, but the specific degraded channel-info path in this report still appears present: if Discord channel lookup returns null during inbound preflight, the message is not inferred as a DM from the absent guild id and can still fall into channel routing.

## Best Possible Solution

Keep the issue open and fix the Discord inbound preflight/classification path. The likely product direction is a small Discord-owned helper that derives chat kind from channelInfo first, then from trusted payload signals such as absent guild_id and sender/direct metadata, with a regression test where fetchChannel returns null or throws and a one-to-one DM still resolves to ChatType direct and agent:<agent>:discord:direct:<userId>. Existing direct-user binding and session-key normalization can stay as supporting safeguards, but they do not replace the missing degraded-lookup classification.
