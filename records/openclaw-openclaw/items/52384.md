## Summary

Keep open. Current main has adjacent protections for model failover, provider/auth-profile cooldowns, and restart/orphan recovery, but it still does not implement a persisted session-level circuit breaker that suppresses or backs off automatic resume-message injection after consecutive model errors. The related open PR #42933 remains an implementation candidate, not proof this specific auto-resume loop is fixed on main.

## Best Possible Solution

Keep this issue open and route it through the agent/session recovery path. The best fix is a generic persisted session-level model-error breaker around automatic resume attempts: count consecutive model errors for main-session and subagent recovery, suppress or exponentially back off further synthetic resume injection after the threshold, reset on the first successful model response or explicit reset, and add focused tests for restart/orphan recovery plus model-fallback exhaustion. Coordinate with #42933/#45389 so the breaker state is shared rather than ad hoc.

## Evidence

- **Issue discussion defines a distinct auto-resume loop:** The supplied GitHub discussion describes a 2026-03-22 Gemini 429 cascade where automatic `Continue where you left off` prompts were injected repeatedly, appeared as user messages, and kept consuming tokens. That is narrower than generic model fallback: it asks to cap/suppress the resume injection itself and reset the breaker after a successful model response.
- **Subagent orphan recovery still injects synthetic resume messages:** `resumeOrphanedSession` builds a `Please continue where you left off` prompt and sends it through `callGateway({ method: "agent" })` with a fresh idempotency key.
