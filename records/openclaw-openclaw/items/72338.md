## Summary

Keep open. Current main contains several partial mitigations related to this report, including session-store rotation removal, interactive channel-login blocking, liveness diagnostics, stuck-session recovery, plugin manifest-load caching, and a gateway CPU scenario test harness. But the remaining Gateway high-CPU/status-timeout problem is still actionable: outbound reply planning still calls a synchronous pending-spawn predicate, that predicate still counts through the subagent registry, and production reads still synchronously load `subagents/runs.json`. The status-path hardening also falls back to another bounded Gateway `status` RPC, so it does not solve an event-loop saturation case where the Gateway cannot service local RPCs. The issue has recent multi-platform reproductions and should stay open as the canonical broad Gateway stall item rather than being closed against narrower partial fixes like #73532 or #73581.

## Best Possible Solution

Keep this issue open as the broad canonical Gateway stall/high-CPU item. The best next fix is to remove synchronous disk reads from the outbound pending-spawn path by using an in-memory or cached subagent-run snapshot with explicit invalidation or bounded refresh, then add a regression/performance test proving empty `subagents/runs.json` is not reread per outbound plan. In parallel, keep #73532 and #73581 as narrower follow-ups only if maintainers want separate plugin-loader and stuck-session workstreams, and verify the Gateway RPC/status path under event-loop saturation or self-loopback timeout conditions.
