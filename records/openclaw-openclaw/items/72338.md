## Summary

Keep open. Current main contains several relevant unreleased mitigations for plugin runtime-deps staging, subagent registry reads, Gateway client readiness, handshake budgets, and diagnostics, but the broad Gateway CPU/event-loop starvation report is not proven fixed. The remaining source/docs contract still treats liveness saturation as diagnostic-only and active stuck embedded/reply/lane work as warning-only by default, so the reported “alive but unresponsive” failure class still needs maintainer retest and follow-through before closure.

## What This Changes

The issue asks OpenClaw to prevent or recover from Gateway high-CPU/event-loop starvation that stalls Telegram/WebChat replies and causes local Gateway status or channel probes to time out.

## Best Possible Solution

Keep this issue as the canonical broad Gateway CPU/liveness tracker until affected packaged installs are retested on current main or the next release. Close only after the Gateway idles normally, loopback probes stay responsive, a Telegram/WebChat turn completes without pre-model CPU starvation, and any remaining active stuck-session recovery work is either fixed or explicitly tracked by narrower canonical issues with release provenance.
