## Summary

Keep open. Current main has useful partial mitigations and diagnostics, but the concrete hot path called out in the issue discussion is still present: Telegram delivery can invoke outbound reply planning with a session key, that planner still calls the pending-spawn predicate, and the registered predicate still reaches a production disk read of `subagents/runs.json`. The issue is recent, corroborated by multiple reporters across Linux/macOS/Windows and Node 22/24/25, and related reports are being funneled into this item rather than superseding it.

## Best Possible Solution

Keep this issue open as the canonical Gateway CPU/event-loop starvation report. The best fix is to remove synchronous persisted subagent-registry reads from normal outbound reply planning, use an in-memory or bounded cached snapshot with explicit invalidation for cross-process visibility, and add a regression/performance test proving `subagents/runs.json` is not reread per outbound plan. Keep the narrower plugin-loader and stuck-lane work linked, but do not close this issue until the main stall path is fixed and shipped.
