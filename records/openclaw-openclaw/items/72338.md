Keep open. Current main contains several meaningful mitigations for the reported Gateway CPU/event-loop starvation cluster, but the thread is recent, multi-host, and still waiting on maintainer-requested main/dev retesting; one successful 2026.4.27 report is not enough to prove the broader idle CPU, loopback probe, and Telegram/WebChat reply stall class is fixed or shipped.

## What This Changes

The issue asks OpenClaw to fix a Gateway high-CPU/event-loop stall where Telegram replies stop progressing and local status or channel probes time out, including safer handling for interactive channel login commands and better diagnostics for stuck runtimes.

## Best Possible Solution

Keep this as the broad Gateway CPU/event-loop starvation tracker until maintainers can tie the resolution to a specific shipped release or explicitly split the remaining work into canonical narrower issues. The desired end state is a packaged Gateway that idles normally, answers loopback status/probe calls reliably, and completes Telegram/WebChat turns without runtime mirror, registry, or stuck-session hot paths starving the event loop.
