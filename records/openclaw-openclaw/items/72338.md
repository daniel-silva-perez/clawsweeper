## Summary

Keep open. Current main includes partial mitigations related to this report, including a manifestless-plugin startup fix, session-store rotation cleanup, agent-exec blocking for interactive channel login, and liveness diagnostics. But the remaining Gateway stall/high-CPU report is still actionable: the outbound pending-spawn path still synchronously reads `subagents/runs.json` through the subagent registry snapshot during normal runtime, and `channels status` still depends on a live Gateway WebSocket RPC that will time out if the event loop is saturated. The newer comments also add concrete reproductions on Node 22, Linux arm64 and x86_64, and identify `runs.json` polling plus CLOSE-WAIT growth, so this is neither stale nor safely superseded by the closed related reports.

## Best Possible Solution

Keep this issue open as the current canonical Gateway/runtime performance bug. The best fix is to remove synchronous disk reads from the outbound pending-spawn hot path by using an in-memory or cached subagent-run snapshot with explicit invalidation or bounded refresh, while preserving cross-process visibility where it is actually needed. Add a regression/performance test around `resolvePendingSpawnedChildren` or outbound payload planning that proves an empty `subagents/runs.json` is not reread per outbound plan. Separately investigate the CLOSE-WAIT/self-loopback report and consider an in-process Gateway dispatch path or bounded internal RPC pool with guaranteed client close on timeout.
