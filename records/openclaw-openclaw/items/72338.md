## Summary

Keep open. Current main contains several meaningful mitigations for the reported Gateway CPU/event-loop starvation cluster, but the issue is recent, multi-host, and still waiting on maintainer-requested main/dev retesting after the latest changes. The remaining risk spans plugin runtime mirroring, subagent registry hot paths, client/probe readiness, and stuck-session recovery, so the broad report should stay open until affected packaged installs prove idle CPU, loopback probes, and Telegram/WebChat turns are stable.

## What This Changes

The issue requests fixing a Gateway high-CPU/event-loop starvation state that stalls Telegram replies and times out local status/deep probes, with suspected triggers in interactive channel login, plugin runtime mirroring, subagent registry polling, and session recovery.

## Best Possible Solution

Keep this issue as the broad Gateway CPU/event-loop starvation tracker until maintainers get affected-host confirmation on current main/dev or the next packaged release. The end state should tie a shipped release or specific fix set to stable idle CPU, reliable `openclaw gateway probe` and `openclaw channels status --deep`, and at least one successful Telegram/WebChat turn on previously affected installs.
