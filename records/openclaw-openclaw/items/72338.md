## Summary

Keep open. Current main still contains the reported performance-sensitive path: Telegram/core outbound reply planning can synchronously query pending spawned children for a live session, and that query still reads `subagents/runs.json` from disk in production. The issue is also recent, has multiple corroborating reproductions across Linux/macOS/Windows and Node 22/24/25, and related reports have been closed into this issue rather than superseding it.

## Best Possible Solution

Keep this issue open as the canonical Gateway CPU/event-loop starvation report. The best fix is to remove synchronous persisted subagent-registry reads from normal outbound reply planning, using an in-memory or bounded cached snapshot with explicit invalidation where cross-process visibility is required, and add a regression/performance test proving `subagents/runs.json` is not reread per outbound plan. Separately track or link narrower follow-ups for the plugin-loader hot loop and CLOSE-WAIT/internal RPC cleanup without closing this issue until the main stall is fixed and shipped.
