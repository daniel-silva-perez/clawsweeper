## Summary

Keep open. Current main has useful partial mitigations for plugin manifest/runtime-deps caching, stuck-session recovery, Gateway handshake budgets, and event-loop diagnostics, but it does not prove the reported Gateway CPU/event-loop starvation is fixed. The current code still runs bundled runtime mirror prep during plugin runtime staging, and the Telegram outbound/subagent pending-run path still reaches the production disk-backed `subagents/runs.json` snapshot path. Related #73532 and #73581 remain open rather than superseding this broader report.

## What This Changes

The issue reports Gateway CPU/event-loop starvation in local loopback mode that stalls Telegram replies and causes status probes to time out.

## Best Possible Solution

Keep this issue open as the canonical Gateway CPU/event-loop starvation report. The best next step is a focused performance fix for the plugin runtime mirror hot path tracked by #73532, followed by or paired with a narrower fix for the outbound/subagent `runs.json` read path if profiling still shows it on message delivery. Close this only after current main and a shipped release prove the main stall/status-timeout symptom is fixed.
