## Summary

Keep open. The report is recent, multi-host, and still broader than the partial mitigations now on current main. Main has unreleased improvements for `subagents/runs.json` read caching, client/event-loop readiness, plugin manifest/runtime-deps caching, and stuck-session recovery, but the central Gateway CPU/event-loop starvation symptom is not proven fixed or shipped, and related narrower trackers such as #73532 and #73581 remain open.

## What This Changes

The issue reports Gateway CPU/event-loop starvation in local loopback mode that stalls Telegram replies and causes `openclaw channels status --deep` or other WebSocket probes to time out.

## Best Possible Solution

Keep this issue open as the canonical Gateway CPU/event-loop starvation tracker until the narrow related fixes are reviewed, affected packaged installs are retested, and a shipped release can be tied to a specific fix commit. The desired end state is a Gateway that idles normally, accepts loopback status probes reliably, and processes Telegram/WebChat replies without plugin-runtime or subagent-registry hot loops starving the event loop.
