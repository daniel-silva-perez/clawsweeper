## Summary

Keep open. Current main still lacks the requested opt-in Responses flag and still builds `/v1/responses` output from assistant messages plus caller-provided client-tool calls only; built-in tool events exist but are not forwarded into JSON or SSE output. The issue is paired with open closing PR #75075, while the parallel #75107 is closed unmerged, so the right next step is maintainer review of the open PR rather than cleanup closure.

## What This Changes

Add an opt-in `gateway.http.endpoints.responses.exposeBuiltInToolCalls` flag so `/v1/responses` JSON and SSE output can include built-in agent tool calls as `function_call` items.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with the behavior default-off and audit-only. The change should cover config types/schema/generated docs, JSON and SSE output ordering, client-tool dedupe, focused gateway tests, public docs, changelog, and a security review of argument exposure; tool results, compact summaries, and built-in delegate mode should remain separate follow-ups.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: run `/v1/responses` with an agent turn that invokes built-in tools, and the response output will contain only assistant text unless caller-provided `pendingToolCalls` are present, while built-in `stream: "tool"` events are emitted separately.

## Solution Assessment

Unclear until the paired PR is reviewed. A default-off `function_call` audit-item flag is a plausible narrow solution, but maintainers should verify security/redaction behavior, output ordering, SSE indices, and client-tool dedupe before merge.
