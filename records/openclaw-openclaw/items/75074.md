## Summary

Keep open. Current main still lacks the requested opt-in flag and still builds `/v1/responses` output from assistant messages plus caller-provided client-tool calls only; built-in tool events exist on the agent event stream but are not forwarded into JSON or SSE output. The issue is paired with open closing PR #75075, while parallel PR #75107 is closed unmerged, so the right next step is maintainer review of the open PR rather than cleanup closure or a duplicate repair lane.

## What This Changes

Add an opt-in Gateway HTTP Responses config flag that surfaces built-in agent tool invocations as Responses `function_call` output items in JSON and SSE.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with the new behavior default-off and audit-only, covering config types/schema/generated baselines, JSON and SSE output, client-tool dedupe, focused gateway tests, docs, and changelog. Tool results, compact trace summaries, and built-in delegate mode should remain separate follow-up design work.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: run `/v1/responses` with an agent turn that invokes built-in tools, and current main will return only the assistant message unless the stop reason contains caller-provided `pendingToolCalls`; built-in `stream: "tool"` events are emitted but not folded into the response output.

## Solution Assessment

Unclear until the paired PR is reviewed. The opt-in, default-off `function_call` audit-item approach is a plausible narrow solution aligned with the Responses surface, but maintainers should verify security/redaction behavior, output ordering, and client-tool dedupe before merge.
