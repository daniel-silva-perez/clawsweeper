## Summary

Keep open. Current main still lacks the requested opt-in Responses flag and still only returns assistant output plus caller-provided client-tool `function_call` items, while built-in tool start events are emitted separately and ignored by the `/v1/responses` JSON/SSE output path. The issue is also paired with open closing PR #75075, so the right cleanup outcome is maintainer review of that PR rather than closing the issue before merge.

## What This Changes

The issue requests an opt-in `/v1/responses` configuration flag that surfaces built-in agent tool invocations as Responses-compatible `function_call` output items in both JSON and SSE responses.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with the behavior default-off and audit-only. The finished change should cover config types/schema/generated docs, JSON and SSE output ordering, client-tool dedupe, focused gateway tests, public docs, changelog, and a security review of argument exposure; tool results, compact digests, and delegate mode should stay separate follow-ups.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: run `/v1/responses` with an agent turn that invokes built-in tools, and the response output will contain only assistant text unless caller-provided `pendingToolCalls` are present, while built-in `stream: "tool"` events are emitted separately.

## Solution Assessment

Unclear until the paired PR is reviewed. A default-off `function_call` audit-item flag is a plausible narrow solution, but maintainers should verify redaction, output ordering, SSE indices, and client-tool dedupe before merge.
