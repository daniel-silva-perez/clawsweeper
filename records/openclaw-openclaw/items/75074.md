## Summary

Keep open. Current main still omits built-in tool calls from `/v1/responses` output while emitting those tool starts on the agent event stream, and the issue is paired with open closing PR #75075 rather than already implemented on main.

## What This Changes

Request to add an opt-in `gateway.http.endpoints.responses.exposeBuiltInToolCalls` flag that surfaces built-in agent tool starts as `function_call` output items in `/v1/responses` JSON and SSE responses.

## Best Possible Solution

Review and land PR #75075, or replace it with an equivalent narrow implementation that updates the gateway config types/schema/generated config docs, JSON and SSE Responses output, focused gateway tests, public docs, and changelog while keeping the behavior opt-in and preserving the existing client-tool delegate path.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: run `/v1/responses` with an agent turn that invokes built-in tools, and current main builds output from assistant text plus caller-provided `pendingToolCalls` only while built-in `stream: "tool"` start events remain outside JSON and SSE output.

## Solution Assessment

Unclear until the paired PR is reviewed. The opt-in `function_call` audit-item design is a plausible narrow fix aligned with the Responses surface, but maintainers should verify redaction, output ordering, client-tool dedupe, and the choice to expose only `phase: "start"`.
