## Summary

Keep open. Current main still returns only assistant messages plus caller-provided client-tool `function_call` items from `/v1/responses`, while built-in tool start events exist on the agent event stream but are not folded into JSON or SSE output. The issue is also paired with open closing PR #75075, with parallel PR #75107 closed unmerged, so the next step is normal maintainer review of the PR rather than closure or a separate repair job.

## What This Changes

The issue requests an opt-in `gateway.http.endpoints.responses.exposeBuiltInToolCalls` setting that would expose built-in agent tool invocations as Responses-compatible `function_call` output items in JSON and SSE responses.

## Best Possible Solution

Review and land PR #75075, or replace it with an equivalent narrow implementation that updates the gateway config types/schema/generated docs, JSON and SSE Responses output, focused gateway tests, public docs, and changelog while keeping the new behavior opt-in and preserving the existing client-tool delegate path.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: run `/v1/responses` with an agent turn that invokes built-in tools, and current main builds the response output from assistant text and `pendingToolCalls` only while built-in `stream: "tool"` events remain outside the Responses output.

## Solution Assessment

Unclear until the paired PR is reviewed. The default-off `function_call` audit item approach is a plausible narrow fix aligned with the Responses surface, but maintainers should verify security/redaction behavior, output ordering, and client-tool dedupe semantics before merge.
