## Summary

Keep open for maintainer review. The PR contains useful terminal-truth and stream-capture work, and it already incorporated review fixes for EEXIST-only run-id retries and route-mirror failures, but it is not merge-ready against the current ACPX boundary: current main delegates turns to the published `acpx` runtime, while the PR patches the older wrapper-owned subprocess loop and introduces a concrete raw-route-argument credential persistence risk plus an abort-listener cleanup bug.

## What This Changes

The PR adds ACPX wrapper-owned per-run `route.json`, `terminal.json`, `stdout.log`, and `stderr.log` artifacts, strict terminal-state finalization, collision-safe run IDs, abort-before-spawn handling, and focused tests.

## Best Possible Solution

Implement durable ACPX terminal truth and post-mortem stdout/stderr capture at the current runtime ownership boundary, preferably in the published `acpx` runtime when reusable, then consume it from the bundled plugin with redacted artifact metadata, documented retention/location behavior, and tests for terminal states, setup-failure cleanup, route-mirror failure, stderr capture, and MCP env redaction.

## Reproduction Assessment

Not applicable for the feature request as a whole. Yes for the blocking security finding: on the PR branch, configure ACPX `mcpServers.*.env`, run an ACPX turn, and inspect `route.json` to see `route_args` persist the `--agent` payload that can encode those env values.

## Solution Assessment

No. The current PR is not the best way to solve this as-is because it patches the older wrapper-owned subprocess path and persists unsanitized route metadata; the safer approach is a current-runtime artifact contract with sensitive route data omitted or redacted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact route args before persisting artifacts:** `extensions/acpx/src/runtime.ts:579`
  - body: `args` can include `--agent <payload>` when MCP servers are configured, and that payload encodes `mcpServers` entries including `env` values. Persisting it verbatim in `route.json` copies credentials into durable artifacts, so store a sanitized route view or omit sensitive flags.
  - confidence: 0.92
- **[P2] Clean up abort listeners when artifact setup fails:** `extensions/acpx/src/runtime.ts:572`
  - body: The abort listener is registered before `createRunArtifacts`, but listener removal only happens in the later `finally`. If artifact setup throws, for example on an unwritable state directory, the signal keeps a stale listener that can call `cancel` later and accumulate across retries.
  - confidence: 0.86
- **[P3] Avoid cancelling twice after pre-spawn aborts:** `extensions/acpx/src/runtime.ts:607`
  - body: If the signal fires after the listener is attached but before the post-artifact abort check, `onAbort` already called `cancelOnAbort`; this branch calls it again. Drop the second call or gate it so abort-before-spawn does not send duplicate cancel requests.
  - confidence: 0.73

## Security Review

Status: needs_attention

Summary: The PR introduces a credential-exposure risk by durably persisting raw ACPX route arguments.

Concerns:

- **[high] Secret-bearing route args are persisted:** `extensions/acpx/src/runtime.ts:579`
  - body: `routeArgs: args` is written into durable `route.json`; in MCP mode those args can include an `--agent` payload built from configured MCP servers, including `env` values. This can copy API tokens or other secrets into per-run artifacts.
  - confidence: 0.92
