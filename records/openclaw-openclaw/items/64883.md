## Summary

Keep open. Current main has adjacent one-action planning guards, but the reported shell no-op path is still not resolved: the retry helper only bypasses non-plan tool suppression for read-like tools, treats `exec`/`bash` as side-effectful by name, and still rejects terminal non-`stop` assistant reasons before prose classification. The current regression tests cover safe `read`, `update_plan`, side-effectful, and unclassified cases, but not a dummy shell command such as `sleep 0.1`.

## What This Changes

The issue requests that strict-agentic planning-only detection treat dummy or no-op tool calls as non-progress and add regression coverage for a shell `sleep` call followed by planning prose.

## Best Possible Solution

Add a conservative structural progress classifier in the embedded runner planning-only path. It should keep real shell, messaging, session, cron, and file mutations replay-unsafe, but allow known no-op/no-output shell calls such as `sleep 0.1` to be classified as non-progress for strict-agentic retry and blocked-exit behavior, with regression tests proving the exact reported path.

## Repair Work Prompt

Fix the strict-agentic planning-only dummy shell-tool gap tracked by this issue. A model that performs a single no-op shell call such as `exec`/`bash` command `sleep 0.1` and then returns plan-only prose should still trigger the planning-only retry path and, after the strict-agentic retry cap, the blocked exit. Keep the boundary conservative: do not mark arbitrary shell commands replay-safe, do not weaken messaging/session/cron/file side-effect guards, and fail closed when command args or tool results are unavailable. Thread command/result metadata into the planning/replay classifier only as needed to recognize a small set of known no-op/no-output shell commands. Add focused regression coverage for the exact dummy shell case, plus negative coverage showing real shell mutations still suppress retry.
