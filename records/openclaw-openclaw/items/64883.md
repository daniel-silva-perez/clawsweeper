## Summary

Keep open. Current main still treats a lone `exec`/`bash` call as non-retry-safe, potentially side-effectful activity by name, and the planning-only retry path has no command/result-aware way to distinguish a no-op shell command such as `sleep 0.1` from real shell progress. The adjacent single-action tests cover safe read-style tools, `update_plan`, side-effectful tools, and unclassified tools, but not the dummy shell evasion described here.

## What This Changes

The issue asks to strengthen the embedded Pi strict-agentic planning-only detector so a dummy or no-op tool call followed by planning prose still triggers retry and eventual blocked behavior.

## Best Possible Solution

Add a conservative structural progress classifier for the strict-agentic planning-only path. It should keep real shell, messaging, cron, write, and session actions replay-unsafe by default, but allow known no-op/no-output shell calls to be treated as non-progress and covered by retry-then-block tests.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction can construct an attempt with one `exec`/`bash` tool meta, planning-only assistant prose, and strict-agentic GPT-5 context; current source returns no retry because shell is treated as non-plan activity with potential side effects before prose classification.

## Solution Assessment

No, a plain tool-name whitelist is not the best complete fix for this issue. The safer approach is command/result-aware classification that can catch `sleep 0.1` while preserving fail-closed behavior for shell commands that may have changed user state.
