## Summary

Keep open. Current main still suppresses the planning-only retry path for a lone `exec`/`bash` call by treating shell tools as side-effectful by name, and the retry helper still rejects terminal non-`stop` assistant reasons before prose classification. The specific dummy shell/no-op case described here is not covered by current regression tests.

## What This Changes

The issue asks OpenClaw to treat dummy or no-op tool calls as non-progress so strict-agentic planning-only retries still fire, with regression coverage for a lone `sleep` call followed by plan-only prose.

## Best Possible Solution

Add a conservative structural progress classifier in the embedded runner planning-only path. It should preserve the current side-effect guard for real shell and messaging actions, but thread enough command/result metadata to classify known no-op/no-output shell calls such as `sleep 0.1` as non-progress and exercise the strict-agentic retry-then-blocked behavior in tests.

## Repair Work Prompt

Fix the strict-agentic planning-only dummy tool-call evasion described in openclaw/openclaw#64883. Current main treats any lone `exec`/`bash` as side-effectful progress and returns before prose classification; a no-op/no-output shell call such as `sleep 0.1` followed by plan-only prose should instead trigger the planning-only retry path and, after the strict-agentic retry limit, the blocked exit. Keep the fix narrowly scoped to the embedded runner/tool metadata surface. Do not make arbitrary shell commands replay-safe and do not weaken side-effect protection for real mutations, messaging sends, cron adds, spawned sessions, or tool errors. Prefer threading structured command/result metadata from tool completion into the planning-only/replay classifier over broad name whitelisting. Add focused regression coverage for a lone dummy shell call plus plan-only prose, and preserve the existing `update_plan`, safe read/search, side-effectful tool, unclassified tool, and completion-language tests. Add a user-facing changelog fix entry only if the final behavior change qualifies under OpenClaw policy, crediting @100yenadmin and not bot accounts.
