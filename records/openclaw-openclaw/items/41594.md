## Summary

Keep open. Current main still does not implement per-model or alias-scoped context profiles for workspace bootstrap injection, tool prompt visibility, or skills prompt visibility. The existing controls are useful but partial: bootstrap injection is global, model alias entries only carry alias/params/streaming, `compat.supportsTools` is capability metadata, and tool policy does not provide the requested workspace-file or skills profile.

## Best Possible Solution

Keep this issue open. The best path is a backward-compatible selected-model or alias-scoped context profile under the model configuration surface, with config types/schema/docs/tests and runtime wiring through model selection, bootstrap-file filtering, skills prompt assembly, effective tool/prompt visibility, `/context` reporting, and model/status UI surfaces. Keep `compat.supportsTools` as capability fallback metadata rather than treating it as the whole user-controlled prompt policy.

## Risks / Open Questions

- Implementation needs an explicit precedence model among a future context profile, `compat.supportsTools`, `tools.byProvider`, runtime `toolsAllow`, `disableTools`, and `localModelLean`.
- The related open model-key proposal #59168 may help alias/profile design, but it does not supersede this issue because it does not implement workspace-file, tool-prompt, or skills-prompt policy.
- Bootstrap hooks can mutate context files, but they are not a typed user-facing per-model config surface and do not cover schema validation, status reporting, or skills/tool prompt suppression.
