## Summary

Keep this issue open. Current main still lacks an alias- or selected-model-scoped context profile for workspace bootstrap files, skills prompt visibility, and tool prompt/schema exposure; existing controls are global, per-agent, provider-capability, or runtime-policy knobs rather than the requested per-model profile surface.

## Best Possible Solution

Keep this issue open and implement a backward-compatible selected-model or alias-scoped contextProfile on the model-entry/config surface, or fold it into the canonical model-profile design if #59168 lands first. The implementation should thread the resolved selected model entry into bootstrap file filtering, skills prompt assembly, tool/client-tool schema exposure, and system prompt rendering, then update zod/generated schemas, docs, Control UI, and tests for defaults, validation, /model switching, workspace allowlists, skills suppression, tool suppression, and precedence with existing policy controls.

## Risks / Open Questions

- Closing this now would conflate existing provider capability and runtime tool-policy controls with the requested alias-scoped context profile surface.
- A real implementation needs careful precedence with compat.supportsTools, disableTools, toolsAllow, localModelLean, per-agent skill allowlists, bootstrap context modes, and /model session overrides so it does not accidentally widen or over-restrict tool and skill visibility.
- The schema, generated config docs, Control UI, and focused tests would need to change together because agents.defaults.models is currently strict.
