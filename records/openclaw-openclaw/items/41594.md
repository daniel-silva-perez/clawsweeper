## Summary

Keep open. Current main still does not implement per-model or alias-scoped context profiles for workspace file injection, skills prompt visibility, or user-controlled tool prompt policy. Existing controls are partial: `compat.supportsTools` can suppress tools for a model definition, global `contextInjection` can change bootstrap timing, per-agent skill allowlists can filter skills, and internal bootstrap hooks can mutate files, but none provides the requested opt-in `contextProfile` surface on model aliases.

## Best Possible Solution

Keep this open and implement a backward-compatible selected-model or alias-scoped context profile in the model configuration surface. Wire it through model selection into bootstrap file filtering, skills prompt assembly, effective tool visibility, `/context` reporting, status/model UI surfaces, config types/schema/generated docs, and focused tests. Treat `compat.supportsTools` as capability fallback metadata rather than the whole user-controlled prompt policy.

## Risks / Open Questions

- A future implementation needs a clear precedence model among `contextProfile`, `compat.supportsTools`, global and provider tool policy, `toolsAllow`, `disableTools`, and `agents.defaults.experimental.localModelLean`.
- Internal `agent:bootstrap` hooks can mutate bootstrap files, but their context is agent/session scoped rather than model/alias scoped and they do not cover schema validation, UI/status reporting, skills prompt suppression, or tool policy.
