## Summary

Keep this issue open. Current main still has only global, per-agent, provider-capability, and runtime tool-policy controls; it does not implement an alias- or selected-model-scoped `contextProfile` for workspace file injection, skills prompt visibility, or tool exposure. The prior discussion correctly identifies a real multi-model workflow gap, and the related #59168 remains a broader model-key proposal rather than an implementation of this request.

## Best Possible Solution

Keep this open. Implement a backward-compatible selected-model or alias-scoped `contextProfile` on the model-entry/config surface, or fold it into the canonical model-profile design if #59168 lands first. Wire the resolved selected model entry into bootstrap file filtering, skills prompt assembly, and effective tool/client-tool exposure; update zod/generated schemas, docs, Control UI, and focused tests for defaults, strict validation, `/model` switching, workspace allowlists, skills suppression, tool suppression, and precedence with existing capability and policy controls.
