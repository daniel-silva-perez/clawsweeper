## Summary

Keep this PR open. Current main still keeps Slack thread participation in a process-local dedupe cache with no disk hydrate/write path, while non-mention Slack thread continuation still depends on that cache. The overlapping persistence PRs #58731 and #70626 remain open/unmerged and do not provide a shipped replacement for this Slack behavior.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to preserve the Slack restart-survival behavior, then either migrate Slack thread participation to the shared async persistence primitive or explicitly approve a hardened Slack-local store. The final branch should trim unrelated churn and retain focused tests for restart survival, clear semantics, TTL expiry, corrupt/oversized files, bounds, and security-sensitive state-file handling.
