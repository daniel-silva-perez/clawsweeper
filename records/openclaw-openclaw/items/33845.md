## Summary

Keep open. Current main still uses an in-process Slack thread participation dedupe cache, so non-mention Slack thread continuation can still be lost across gateway restarts. The related persistence work in #58731 and #70626 is open/unmerged in the provided context and does not provide a shipped replacement for this PR’s intended behavior.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to preserve the Slack restart-survival behavior, then either migrate Slack thread participation to the existing async persistence primitive or wait for/coordinate with the persistent keyed-store work if that becomes the approved SDK contract. The final branch should trim unrelated churn and retain focused tests for restart survival, clear semantics, TTL expiry, corrupt/oversized files, bounds, and state-file hardening.
