## Summary

Keep this PR open. The bug is real and current main still keeps Slack thread participation only in memory, but the latest PR head still has concrete local file-handling hardening gaps in its custom JSON persistence path; the merged plugin state store also makes the final storage direction a maintainer decision rather than a cleanup close.

## What This Changes

This PR persists Slack thread participation to a JSON state file, adds restart/TTL/clear/capacity/security tests, and restricts Slack API exports to production cache helpers.

## Best Possible Solution

Land one focused Slack restart-survival implementation after maintainer storage-direction signoff. The final patch should preserve the in-memory hot path semantics, account/channel/thread keying, 24h TTL, and 5k cap; then either use api.runtime.state.openKeyedStore with an approved async/fire-and-forget pattern or harden the local JSON path by rejecting non-regular files and cleaning failed temp writes, with restart/TTL/clear/corrupt/oversized/capacity/failure coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject non-regular state files before reading:** `extensions/slack/src/sent-thread-cache.ts:103-110`
  - body: `hydrateFromDisk` calls `statSync` and then `readFileSync` on the configured persist path without checking `lstat`/`isFile`. A symlink, FIFO, or device node under the state directory can bypass the regular-file assumption or block the first Slack event while hydration runs. Reject non-regular paths before the size check/read, or move this persistence onto the shared keyed store.
  - confidence: 0.84
- **[P3] Remove temp files when rename fails:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: `persistToDisk` writes a UUID temp JSON file and then renames it, but if `renameSync` throws the scheduled persist path swallows the error and leaves the temp file behind. Repeated locked-path or permission failures can accumulate stale `*.tmp` files containing Slack thread identifiers; remove the temp path on failure before rethrowing.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds custom local filesystem persistence with no dependency or CI supply-chain change, but two local file-handling hardening issues remain.

Concerns:

- **[medium] Hydration follows non-regular state files:** `extensions/slack/src/sent-thread-cache.ts:103`
  - body: The PR uses `statSync` and `readFileSync` on the final persisted path without rejecting symlinks or special files, which can turn a tampered local state path into a local DoS or unintended read before Slack message handling continues.
  - confidence: 0.84
- **[low] Failed atomic rename leaves state temp files:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: If the final rename fails, the temp JSON file is not removed and the scheduled write swallows the error, allowing repeated failures to leave Slack thread identifiers in leftover temp files.
  - confidence: 0.82
