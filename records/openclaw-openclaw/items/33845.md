## Summary

Keep open. Current main still keeps Slack thread participation in process memory only, so the restart-survival bug is real, but the latest patch is not ready to merge because it adds a custom filesystem persistence path with a remaining non-regular-file hydration risk and two concrete cleanup/test defects.

## What This Changes

The branch adds JSON-backed disk persistence, lazy hydration, clear/restart coverage, and an API barrel comment for the Slack thread participation cache.

## Best Possible Solution

Land one owner-approved Slack restart-survival implementation that persists account/channel/thread participation for 24 hours with bounded growth, clear semantics, corrupt-state handling, and restart tests; now that the shared plugin state store exists, prefer using that store or explicitly documenting why Slack needs a hardened custom JSON path.

## Reproduction Assessment

Yes. On current main, recordSlackThreadParticipation writes only to a process-global dedupe cache, and prepare.ts uses that cache for bot_thread_participant implicit mentions; a gateway restart clears the process map and makes previously participated non-mention Slack threads stop matching.

## Solution Assessment

No. The PR targets the right user-visible bug, but the current patch is not the narrowest maintainable merge shape while openKeyedStore exists and while the custom filesystem path still has non-regular-file, temp-cleanup, and isolated-test defects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reject non-regular state files before hydration:** `extensions/slack/src/sent-thread-cache.ts:103-110`
  - body: `hydrateFromDisk` calls `statSync` and then `readFileSync` on the configured state path without rejecting symlinks, FIFOs, devices, or other non-regular files. A tampered local state path can block the first Slack event or read unintended local content before the cache falls back, so use `lstat`/`isFile` or move this persistence onto the keyed state store.
  - confidence: 0.87
- **[P3] Remove temp files when rename fails:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: `persistToDisk` writes a UUID-named temp JSON file and then renames it, but a thrown `renameSync` leaves the temp file behind while the scheduled persist path swallows the error. Wrap the rename so failed writes remove `tmpPath` and do not accumulate Slack thread identifiers in leftover temp files.
  - confidence: 0.82
- **[P3] Initialize tempDir before the directory-mode test:** `extensions/slack/src/sent-thread-cache.test.ts:311-312`
  - body: This test builds `path.join(tempDir, ...)` without calling `setup()`. Running it by itself can leave `tempDir` undefined and fail before the directory-mode assertion, so create a temp dir in the test or call the shared setup helper first.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff adds local filesystem persistence for Slack thread state and still has concrete hardening gaps around non-regular state files and failed temp-file cleanup.

Concerns:

- **[medium] Hydration follows non-regular state files:** `extensions/slack/src/sent-thread-cache.ts:103`
  - body: The persisted state path is read with `statSync` and `readFileSync` without rejecting symlinks, FIFOs, or device files, which can turn local state-path tampering into a gateway event-loop block or unintended local file read before Slack message handling continues.
  - confidence: 0.87
- **[low] Failed rename leaves state temp files:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: If the final rename fails, the temp JSON file is not removed and the scheduled write swallows the error, allowing repeated failures to leave Slack thread identifiers in leftover temp files.
  - confidence: 0.82
