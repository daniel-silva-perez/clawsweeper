## Summary

Keep this PR open. Current main still keeps Slack thread participation in memory only, so the restart-survival bug remains real, but the PR now needs maintainer direction because the merged plugin state store gives a newer shared persistence path and the branch still has filesystem hardening cleanup to resolve before merge.

## What This Changes

The PR adds a Slack-local JSON persistence layer and persistence tests for `sent-thread-cache`, adds a Slack API export comment, and still carries unrelated formatter/import churn in several non-Slack-cache files.

## Best Possible Solution

Preserve the Slack restart-survival behavior, but implement it through the approved plugin state store or an explicitly approved hardened Slack-local store. The final patch should be focused on Slack thread participation, keep the in-memory hot path semantics clear, cover restart, TTL, clear, corrupt/oversized state, bounded-entry, and storage-failure cases, and consolidate with the overlapping Slack persistence PR before landing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Reject non-regular persist files before reading:** `extensions/slack/src/sent-thread-cache.ts:103-110`
  - body: `hydrateFromDisk` follows the final path with `statSync` and then `readFileSync` without checking `lstat`/`isFile`. A symlink, FIFO, or device node at the state path can bypass the intended file-size guard or block the first Slack event while the gateway reads it; reject non-regular files or use the shared state-store primitive before reading.
  - confidence: 0.78
- **[P3] Clean up temp files when rename fails:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: `persistToDisk` writes a UUID temp file and then renames it into place, but if `renameSync` throws the scheduled persist path swallows the error and leaves the temp file behind. Repeated locked-path or permission failures can accumulate stale `*.tmp` files containing thread participation data in the state directory; wrap the rename so the temp path is removed on failure.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces local filesystem persistence for Slack thread state; no dependency or CI supply-chain regression was found, but the custom store still has concrete file-handling concerns.

Concerns:

- **[medium] Hydration follows non-regular state files:** `extensions/slack/src/sent-thread-cache.ts:103`
  - body: The PR uses `statSync`/`readFileSync` on the final persist path without rejecting symlinks or special files, which can turn a tampered state path into a local DoS or unintended read before Slack message handling continues.
  - confidence: 0.78

  - body: If the final rename fails, the temp JSON file is not removed and the caller swallows the write error, allowing repeated failures to leave Slack thread identifiers in leftover temp files.
  - confidence: 0.82
