## Summary

Keep this PR open. Current main still has the reported Slack restart-survival bug, but the current branch is not ready to merge because the custom JSON persistence path still has concrete hardening gaps and current main now provides a SQLite-backed plugin state store that maintainers should consider for the final storage direction.

## What This Changes

The PR adds JSON-backed lazy persistence, clear/restart coverage, test-only helpers, and an API-barrel comment for the Slack thread participation cache.

## Best Possible Solution

Land one owner-approved Slack restart-survival implementation that keeps the in-memory hot path semantics, persists account/channel/thread participation for the existing 24-hour policy, preferably uses api.runtime.state.openKeyedStore where feasible, and keeps restart, TTL, clear, corrupt-state, cap, and split-module behavior covered by focused Slack tests.

## Reproduction Assessment

Yes. Record participation in a Slack thread, restart the gateway so the process-global cache is empty, then send a non-mention reply in that thread; current main has no durable state for prepare.ts to classify it as bot_thread_participant.

## Solution Assessment

No. The PR targets the right user-visible bug, but the current patch is not the best merge shape while openKeyedStore exists and while the custom filesystem path still has non-regular-file, failed-rename cleanup, and isolated-test defects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject non-regular state files before hydration:** `extensions/slack/src/sent-thread-cache.ts:103-110`
  - body: `hydrateFromDisk` calls `statSync` and then `readFileSync` on the configured state path without rejecting symlinks, FIFOs, devices, or other non-regular files. A tampered local state path can block the first Slack event or read unintended local content before the cache falls back, so validate the opened file with `lstat`/`fstat` or move this persistence onto the keyed state store.
  - confidence: 0.87
- **[P3] Clean up temp files when rename fails:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: `persistToDisk` writes a UUID-named temp JSON file and then renames it, but a thrown `renameSync` leaves the temp file behind while the scheduled persist path swallows the error. Wrap the rename so failed writes remove `tmpPath` and do not accumulate Slack thread identifiers in leftover temp files.
  - confidence: 0.82
- **[P3] Initialize tempDir before the directory-mode test:** `extensions/slack/src/sent-thread-cache.test.ts:311-312`
  - body: This test builds `path.join(tempDir, ...)` without calling `setup()`. Running it by itself can leave `tempDir` undefined and fail before the directory-mode assertion, so create a temp dir in the test or call the shared setup helper first.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff adds local filesystem persistence for Slack thread state and still has concrete hardening gaps around non-regular state files and failed temp-file cleanup.

Concerns:

- **[medium] Hydration follows non-regular state files:** `extensions/slack/src/sent-thread-cache.ts:103`
  - body: The persisted state path is read with `statSync` and `readFileSync` without rejecting symlinks, FIFOs, or device files, which can turn local state-path tampering into a gateway event-loop block or unintended local file read before Slack message handling continues.
  - confidence: 0.87

  - body: If the final rename fails, the temp JSON file is not removed and the scheduled write swallows the error, allowing repeated failures to leave Slack thread identifiers in leftover temp files.
  - confidence: 0.82
