## Summary

Keep this PR open. The reported Slack restart-survival bug is still real on current main, but this branch is not ready to merge because the new JSON persistence still has concrete filesystem hardening and test-isolation issues, and it now overlaps with the merged plugin state-store foundation plus the open broader Slack persistence PR.

## What This Changes

This PR adds JSON-backed disk persistence, lazy hydration, clear/restart tests, and a Slack API-barrel comment for the Slack thread participation cache.

## Best Possible Solution

Land one owner-approved Slack restart-survival implementation, preferably using the merged plugin state store unless maintainers explicitly choose a hardened JSON path, while preserving account/channel/thread keying, 24h TTL, bounded growth, clear semantics, restart/corrupt-state coverage, and the current Slack mention-gating contract.

## Reproduction Assessment

Yes. Current main records participation only in a process-global dedupe cache, and prepare.ts uses that cache to allow non-mention replies in participated Slack threads, so a gateway restart drops the state needed for that behavior.

## Solution Assessment

No. The PR solves the right user problem in principle, but the current custom JSON persistence path is not the safest merge shape now that the plugin state store exists and the branch still has concrete filesystem and test issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reject non-regular persist files before hydration:** `extensions/slack/src/sent-thread-cache.ts:103-110`
  - body: `hydrateFromDisk` calls `statSync` and then `readFileSync` on the configured state path without rejecting symlinks, FIFOs, devices, or other non-regular files. A tampered local state path can block the first Slack event or read unintended local content before the cache falls back, so reject non-regular files before the size check/read or move this onto the keyed state store.
  - confidence: 0.87
- **[P3] Remove temp files when rename fails:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: `persistToDisk` writes a UUID-named temp JSON file and then renames it, but a thrown `renameSync` leaves the temp file behind while the scheduled persist path swallows the error. Wrap the rename so failed writes remove `tmpPath` and do not accumulate Slack thread identifiers in leftover temp files.
  - confidence: 0.82
- **[P3] Initialize tempDir before the directory-mode test:** `extensions/slack/src/sent-thread-cache.test.ts:311-312`
  - body: This test builds `path.join(tempDir, ...)` without calling `setup()` in the test. Running it by itself can leave `tempDir` undefined and fail before the directory-mode assertion, so create a temp dir in the test or call the shared setup helper first.
  - confidence: 0.79

## Security Review

  - body: The persisted state path is read with `statSync` and `readFileSync` without rejecting symlinks, FIFOs, or device files, which can turn local state-path tampering into a gateway event-loop block or unintended local file read before Slack message handling continues.
