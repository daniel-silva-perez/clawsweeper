## Summary

Keep this open. Current main still keeps Slack thread participation in process memory only, so the restart-survival bug is real, but the latest PR head still has concrete local file-handling issues in its custom JSON persistence. The newly merged plugin state store also gives maintainers a clearer storage direction to choose before landing this work.

## What This Changes

This PR persists Slack thread participation to a JSON state file, adds restart/TTL/clear/capacity/security tests, and keeps test-only cache helpers out of the Slack API barrel.

## Best Possible Solution

Land one focused Slack restart-survival implementation after maintainer storage-direction signoff. The final design should preserve the sync in-memory hot path for mention gating, account/channel/thread keying, 24h TTL, and bounded growth, while either using the approved plugin state store with a safe async/fire-and-forget pattern or fully hardening the JSON path with non-regular-file rejection, failed-temp cleanup, and restart/TTL/clear/corrupt/oversized/capacity/failure coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject non-regular state files before reading:** `extensions/slack/src/sent-thread-cache.ts:103-110`
  - body: `hydrateFromDisk` uses `fs.statSync` and then `fs.readFileSync` on the persisted path. Because `statSync` follows symlinks and does not reject FIFOs/devices, a tampered state path can block the first Slack event or read unintended file content before gating continues. Check `lstatSync`/`isFile` before the size check/read, or move this persistence onto the approved keyed store.
  - confidence: 0.86
- **[P3] Clean up temp files when rename fails:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: `persistToDisk` writes `${filePath}.${randomUUID()}.tmp` and then calls `fs.renameSync`, but if the rename throws, the scheduled persist path swallows the error and leaves a temp JSON file containing Slack thread identifiers. Wrap the rename in cleanup so the temp path is removed before rethrowing.
  - confidence: 0.82
- **[P3] Initialize tempDir inside the directory-mode test:** `extensions/slack/src/sent-thread-cache.test.ts:311-312`
  - body: This test builds `path.join(tempDir, ...)` without calling `setup()` in the test itself. It relies on an earlier test having assigned `tempDir`, so filtered or isolated runs of this case can fail before exercising the directory-mode assertion. Create a temp dir in this test or call the shared setup helper first.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds custom local filesystem persistence with no dependency or CI supply-chain change, but local file-handling hardening issues remain.

Concerns:

- **[medium] Hydration follows non-regular state files:** `extensions/slack/src/sent-thread-cache.ts:103`
  - body: The persisted state path is read with `statSync` and `readFileSync` without rejecting symlinks, FIFOs, or device files, which can turn local state-path tampering into a gateway event-loop block or unintended file read before Slack message handling continues.
  - confidence: 0.86
- **[low] Failed atomic rename leaves state temp files:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: If the final rename fails, the temp JSON file is not removed and the scheduled write swallows the error, allowing repeated failures to leave Slack thread identifiers in leftover temp files.
  - confidence: 0.82
