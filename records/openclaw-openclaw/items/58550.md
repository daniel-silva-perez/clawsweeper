## Summary

Keep open: current main still uses the JSON session store, so the PR is not obsolete, but the proposed SQLite backend is not mergeable as-is because the diff can hide or lose session updates when SQLite writes or JSON-to-SQLite migration fail.

## What This Changes

This PR adds an opt-in SQLite-backed session store, `session.storeType` config/schema entries, migration and store-info CLI commands, and SQLite store tests while keeping existing JSONL transcripts as cold storage.

## Best Possible Solution

Keep the PR open only if maintainers want a SQLite session-store direction, but require a single authoritative backend model, explicit migration failure handling, and focused regression tests for SQLite write failure, corrupt JSON migration, retry behavior, and legacy session-store migrations before merge.

## Reproduction Assessment

Yes. The reviewed defects are reproducible from the code path: enable SQLite, trigger a SQLite save failure after a populated DB, and the fallback writes JSON while subsequent reads still prefer SQLite; for migration, a corrupt or unreadable `sessions.json` is caught and reported as `0` migrated rows.

## Solution Assessment

No, not as proposed. A SQLite backend may be a reasonable long-term solution for the scale problem, but this implementation should not fall back per-operation to a second store or treat failed migrations as empty successful migrations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve a single source of truth on SQLite write failure:** `src/config/sessions/store-facade.ts:136-138`
  - body: In sqlite mode this catch writes the updated store to `sessions.json` after a SQLite write error, but later reads still select SQLite whenever it is available. A transient `SQLITE_BUSY` or similar write failure can therefore make the just-saved session change disappear on the next load because the stale SQLite DB remains the source of truth. Surface the write error or switch the effective backend persistently before saving.
  - confidence: 0.93
- **[P1] Propagate migration failures instead of returning no data:** `src/config/sessions/store-sqlite.ts:560-562`
  - body: `migrateJsonToSqlite` catches JSON parse and SQLite write failures and returns `0`, which callers treat like a normal empty migration. If an existing `sessions.json` is corrupt or a SQLite insert fails, the command/facade can proceed as though there were no sessions and leave users looking at an empty SQLite store. Only truly missing or empty valid input should return `0`; failed migrations need to fail visibly or fall back to JSON.
  - confidence: 0.91
- **[P2] Honor failed migrations before loading SQLite:** `src/config/sessions/store-facade.ts:111`
  - body: The return value from `ensureSqliteMigrated` is ignored, so even when that helper reports `false` the facade still loads SQLite instead of using the JSON store or failing. That can expose an empty or stale SQLite DB after a failed first migration attempt. Branch on the failure before reading SQLite, and only memoize a migration attempt after it succeeds.
  - confidence: 0.86
