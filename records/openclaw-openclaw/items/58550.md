## Summary

Keep open. Current main still has no SQLite session-store backend, but this PR is not merge-ready because the latest diff still has data-integrity bugs in SQLite write fallback and migration failure handling, plus required user-facing release/docs follow-through is missing.

## What This Changes

The PR adds opt-in `session.storeType: "sqlite"` support with a SQLite session metadata store, JSON-to-SQLite migration helpers, `openclaw sessions migrate/store-info` commands, config schema/help updates, SQLite store tests, and a repository PR description file.

## Best Possible Solution

Keep the PR open only as source work until the storage direction is confirmed. A mergeable solution should preserve one authoritative store in SQLite mode, fail or deliberately switch source-of-truth on SQLite errors, normalize legacy JSON rows before import, add focused regression tests for those cases, and update docs plus the active changelog.

## Reproduction Assessment

Yes for the blocking PR defects: configure sqlite mode and force `saveSessionStoreSqlite` or JSON-to-SQLite migration to throw; the diff shows writes can fall back to JSON while later reads prefer SQLite, and migration failures can be treated as zero-row success. I did not run tests because this sweep is read-only.

## Solution Assessment

No. The high-level performance goal is valid, but this implementation is not the best fix until the SQLite backend has strict source-of-truth semantics, migration failures surface correctly, legacy rows are normalized, and the branch is rebased onto current main's session-store split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid fallback JSON writes after SQLite save failures:** `src/config/sessions/store-facade.ts:136-138`
  - body: In sqlite mode, this catch writes the updated store to `sessions.json` when SQLite throws, but future reads still prefer SQLite as soon as it is readable again. A transient `SQLITE_BUSY` or filesystem error can therefore acknowledge a save that disappears on the next load because the SQLite DB remains stale. Propagate the error or make the fallback source-of-truth switch explicit and durable.
  - confidence: 0.92
- **[P1] Surface SQLite migration failures:** `src/config/sessions/store-sqlite.ts:560-562`
  - body: `migrateJsonToSqlite` rolls back and then converts every migration failure into `0`, which callers treat like a normal empty store. On a failed first migration, the facade can proceed to read SQLite and show an empty or stale hot store instead of falling back to JSON or failing the CLI migration. Let the error propagate or return an explicit failure result.
  - confidence: 0.9
- **[P2] Apply session migrations before importing rows:** `src/config/sessions/store-sqlite.ts:534-546`
  - body: The existing JSON loader runs `applySessionStoreMigrations` and normalizes entries before use, but this migration inserts raw parsed JSON directly. Older stores that still have fields like `provider`/`lastProvider` will be imported with null indexed `channel`/`last_channel` columns, so routing/list metadata can be wrong until each session is rewritten. Reuse the current loader migration/normalization path before binding rows.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/cli/program/register.status-health-sessions.ts:226`
  - body: This user-facing feature adds a config key and new `openclaw sessions` commands, but the diff does not add an entry to `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` changes to be recorded under the active version with contributor credit.
  - confidence: 0.95
