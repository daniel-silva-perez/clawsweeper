## Summary

Keep this PR open. Current main does not implement the PR’s intended SQLite-backed session store or an equivalent replacement: session metadata is still stored, discovered, documented, loaded, and saved as `sessions.json`. The PR remains potentially useful unmerged core work, but it needs maintainer review because the provided PR discussion still records unresolved data-consistency issues around SQLite write fallback and migration failure handling.

## Best Possible Solution

Keep this PR open as unmerged core session-store work. The best path is to either continue this PR or a successor PR with a generic session-store abstraction, SQLite implementation, explicit migration/error semantics, docs/config updates, and regression tests for transient SQLite failures, legacy `sessions.json` migrations, and large-store `sessions.list` behavior. If maintainers choose a different storage direction, this PR should be superseded by that canonical issue/PR rather than closed as implemented today.

- The PR is large and touches core session persistence; it should not be closed as obsolete, but it also should not merge without targeted data-integrity tests and a fresh maintainer review.
