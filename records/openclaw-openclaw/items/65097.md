## Summary

Keep open. Current main still has the reported Dream Diary language gap: the narrative system/user prompts are English-only, the runner and phase call sites only carry timezone/model, the resolved memory-core Dreaming config has no language or locale field, the strict plugin manifest would reject those keys, and the public docs expose no language setting.

## What This Changes

Add a language or locale configuration option for memory-core Dreaming narrative generation so Dream Diary entries can be generated in a user-selected language.

## Best Possible Solution

Implement a schema-backed explicit Dream Diary language hint in memory-core, resolve it through the memory-host Dreaming config path, pass it through light/REM narrative generation and detached runs, add a concise output-language directive to the narrative subagent and request-scoped fallback paths, and update focused tests/docs/changelog. Keep `auto` inference and custom prompt templates as separate follow-up scope.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. The current gap is high-confidence from source inspection because the schema rejects language/locale keys and the narrative generation path never reads or carries an output-language setting.

## Solution Assessment

Yes, with scope adjustment. The narrow maintainable solution is an explicit memory-core `dreaming.language` or `dreaming.locale` config contract; auto-detection and full custom templates are broader product choices and should not be mixed into the first fix.
