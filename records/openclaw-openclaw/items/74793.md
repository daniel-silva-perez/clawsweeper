## Summary

## What This Changes

The PR adds optional `agentId` plumbing to `PluginCommandContext`, forwards it through plugin command execution, and adds tests for present and omitted values.

## Best Possible Solution

If maintainers accept the API shape, land the additive command-context field with the included tests, update any generated public SDK or docs artifacts required by release checks, and explicitly include or exclude nonstandard command call sites such as diagnostics.

Yes. Current main can be inspected directly: `HandleCommandsParams.agentId` exists, but `handlePluginCommand`, `executePluginCommand`, and `PluginCommandContext` do not carry it into plugin handlers; the PR adds focused tests for the present and omitted cases.

## Solution Assessment

Yes, with maintainer API review. An optional `ctx.agentId` field is the narrowest maintainable way to unblock agent-aware plugin commands, but the public field name and exact propagation boundary should be accepted before merge.
