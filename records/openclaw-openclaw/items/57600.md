## Summary

Keep this PR open for maintainer follow-up. Current main still lacks the proposed read-tool guard for PDF/Office binary documents, the linked issue and broader same-author DOCX extraction PR are still open, and the draft diff has a concrete path-resolution bug that should be fixed before it could land.

## What This Changes

The PR adds MIME-based binary document fallback handling to the read wrapper for PDF and Office formats, wires an extra buffer-read hook through host and sandbox read paths, and adds a DOCX fallback regression test.

## Best Possible Solution

Resolve the product choice between a lightweight fallback for all binary documents and the broader native DOCX extraction path. If this fallback remains desired, wire MIME detection through the same resolved path or buffer used by the read operation, preserve workspace and sandbox guards, and cover host plus sandbox reads including path aliases.

## Reproduction Assessment

Yes. The high-confidence reproduction path is to read a .docx/.pdf-style non-image file through the current read tool: upstream read converts non-image buffers to UTF-8 text, and current OpenClaw only post-processes images, so PDF/Office files can still surface raw binary-looking text.

## Solution Assessment

No. The PR targets the right narrow behavior, but the added buffer hook re-resolves the original path differently from the underlying read tool; the safer solution should share the resolved read path or file buffer with the base read operation, then apply the fallback after the existing guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reuse the resolved read path for MIME sniffing:** `src/agents/pi-tools.ts:394-398`
  - body: The new fallback sniffs by re-reading the original `toolPath`, but the underlying read tool first normalizes inputs such as `@foo`, `~/foo`, and macOS filename variants via `resolveReadPath`. Those valid reads can succeed in the base tool while this hook reads a different/nonexistent path, catches the error, and returns the raw binary text the PR is meant to suppress. Pass the resolved path/buffer from the read operation or mirror the same resolver before sniffing.
