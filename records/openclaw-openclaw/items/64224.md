## Summary

Keep open. Current main still treats billing failures as auth-profile-wide disabled state, while model-scoped cooldown bypass only applies to rate-limit cooldowns. The LiteLLM/proxy setup in the report remains supported by docs, and the related billing-summary work only improved the user-facing error message; it did not change cooldown scope or add a zero/disable opt-out.

## Best Possible Solution

Keep this issue open. The best fix is a deliberate proxy-aware or model-scoped billing cooldown policy in the auth-profile usage/fallback path for OpenAI-compatible aggregators such as LiteLLM, or a documented supported opt-out for operators who manage failover in the proxy. Add regression coverage with one LiteLLM auth profile serving two sibling models where a billing failure on one model does not block the other, while preserving profile-wide behavior for true provider-key billing exhaustion.

## Risks / Open Questions

- A model-scoped billing policy needs to distinguish true provider-key balance exhaustion from a proxy/upstream-model billing failure, or OpenClaw could retry an actually exhausted account too aggressively.
- A supported `billingBackoffHours: 0` or disable flag needs explicit operator-managed failover semantics and retry-loop guardrails.
