## Summary

Current main still treats billing failures as auth-profile-wide disabled state, not model-scoped state. The reported LiteLLM/proxy scenario remains actionable because one billing-classified upstream model failure can still make the shared `litellm:default` profile unavailable for sibling models, and there is still no supported `0` or `disable` opt-out for billing backoff. The existing ClawSweeper comment is still accurate against the newer main SHA, and the related #66315 work only improved user-facing billing guidance for fallback-summary errors; it did not change cooldown scope.

## Best Possible Solution

Keep this issue open. The best fix is a deliberate proxy-aware or model-scoped billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or a documented supported opt-out for operators who manage failover in their proxy. The likely code path is auth-profile usage state, `isProfileInCooldown` / fallback selection, and config schema/help/docs if a public opt-out is added. Add regression coverage with one LiteLLM auth profile serving two sibling models where a billing failure on one model does not block the other, while preserving profile-wide behavior for true provider-key billing exhaustion.
