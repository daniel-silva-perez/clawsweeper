## Summary

Keep open. Current main still does not implement the proposed `tools.exec.denyPatterns` config surface, and the PR touches exec command-admission behavior that overlaps the open security-labelled denylist design in #71097, so it needs maintainer security/product review rather than conservative cleanup closure.

## What This Changes

This PR adds global and per-agent `tools.exec.denyPatterns` regex config, threads those patterns into exec tool defaults and chat-bash exec calls, blocks matching commands before execution, updates generated config schema artifacts, and adds deny-pattern tests.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers want the feature, rebase it onto current main and decide whether `tools.exec.denyPatterns` is explicitly a soft routing guardrail for steering coding CLIs to `sessions_spawn` or part of the broader denylist security mode tracked in #71097. A landable version should align with that decision, use shell-aware matching or very explicit documentation, preserve additive global/per-agent semantics, cover all exec call sites, update current generated schema/docs/changelog surfaces, and add focused tests for merge behavior, invalid patterns, wrappers/chains, and approval interactions.
