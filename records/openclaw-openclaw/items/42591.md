## Summary

Keep open. Current main still ships the requested installer responsibilities through one monolithic `scripts/install.sh`, and the tests, docs, CI scope rules, and Docker smoke workflow still treat that file and the public installer URL as the stable surface. The report is valid, but this is a broad public installer refactor that needs maintainer agreement on module boundaries and compatibility before an implementation PR is safe.

## What This Changes

The issue asks to split the monolithic macOS/Linux `scripts/install.sh` into a small entrypoint plus sourced modules for UI, downloading, platform detection, npm/git installation, and build-tool setup.

## Best Possible Solution

Keep `scripts/install.sh` and `https://openclaw.ai/install.sh` stable as the public entrypoint, then extract narrowly scoped sourced helpers behind that wrapper with matching installer docs, unit coverage, Docker smoke coverage, and changed-lane/release-validation updates.

- Splitting a curl-piped installer can break remote installs if helper files are sourced from paths that are not available in the streamed script.

- The desired module boundary is a maintainer/product decision because the public `scripts/install.sh` wrapper must stay compatible with existing one-liners.
