## Summary

Keep this PR open. Current main still builds PDF extraction fallback contexts without `systemPrompt`, while the pinned Codex Responses provider maps `context.systemPrompt` to the required `instructions` request field. The latest PR head is narrowly scoped to the PDF tool and its tests, covers both fallback branches, and I did not find a concrete correctness or security blocker in the diff.

## What This Changes

This PR adds a Codex-only `systemPrompt` to PDF extraction fallback contexts, passes the resolved model into both fallback context builders, and adds regression tests for Codex and non-Codex fallback behavior.

## Best Possible Solution

Land this focused patch, or an equivalent maintainer patch, so Codex PDF extraction fallback requests include a minimal `systemPrompt` only when `model.api === "openai-codex-responses"`, while preserving non-Codex fallback behavior and the existing PDF extraction flow.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure a non-native PDF fallback model resolved as `openai-codex-responses`; current main sends a user-only context, and the pinned provider only populates the Codex `instructions` field from `context.systemPrompt`.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I found: it gates on the resolved model API, threads that model into both existing fallback branches, and adds regression coverage without changing provider defaults, extraction policy, dependencies, or non-Codex behavior.
