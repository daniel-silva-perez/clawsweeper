## Summary

Keep open. Current main still uses the existing `tar.c` backup archive path and does not implement archive-time ENOENT/ENOTDIR skipping, while this PR is tied to open bug #67417 and overlaps with open draft PR #72251. The PR is useful, but it needs maintainer review or revision around node-tar Pack error semantics, manifest/verify consistency, and test fidelity before merge or supersession.

## Best Possible Solution

Keep this PR open for maintainer review or revision, or replace it with a superseding fix that closes #67417. The best implementation should add archive-time ENOENT/ENOTDIR resilience in `src/infra/backup-create.ts`, keep manifest/result/verify semantics consistent for omitted payloads, add a focused regression test for a session transcript disappearing during packing, and coordinate explicitly with #72251's volatile-file filtering work.

## Risks / Open Questions

- The proposed `tar.Pack` error listener may not be enough to continue archiving: tar 7.5.13 emits errors from queued stat/open/read work without marking the job skipped, so swallowing ENOENT needs proof that packing completes rather than hanging or leaving a corrupt temp archive.
- The branch records archive-time skipped paths after writing `manifest.json`; current verify semantics require manifest-declared payloads to exist, so omitted top-level payloads can create inconsistent backup artifacts.
- The change overlaps with #72251's volatile-file filter/retry work; merging either PR without coordination could duplicate or conflict in the backup archive path.
- Security review found no unexplained workflow, dependency, lockfile, package script, secret, permission, or downloaded-artifact changes. The remaining risk is local backup correctness and consistency.
