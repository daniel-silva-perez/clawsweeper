## Summary

Keep this PR open. Current main still uses the existing `tar.c` backup path and has no archive-time ENOENT/ENOTDIR skip handling, so the linked backup race is not implemented on main. The PR is a focused implementation candidate for #67417, but the submitted diff still needs revision and proof around manifest consistency, verification behavior, tar stream semantics, and the backup test mock.

## Best Possible Solution

Keep this PR open and request revision, or land an equivalent superseding fix. The maintainer path should handle archive-time ENOENT/ENOTDIR for volatile session files in `src/infra/backup-create.ts`, keep manifest/result/verify semantics consistent for omitted payloads, add a regression test that simulates a session transcript disappearing during packing, and coordinate explicitly with #67417 and #72251 before closing either implementation candidate.

## Risks / Open Questions

- Merging the PR unchanged could create archives whose embedded manifest still declares payloads omitted during packing, making `backup verify` fail or misrepresenting the artifact.
- The proposed mock, as described in review context, cannot faithfully exercise the swallowed file-level ENOENT success path because it still emits a stream error.
- Replacing `tar.c` with direct `tar.Pack` stream wiring needs explicit proof for gzip, portable mode, path remapping, close/error ordering, temp archive cleanup, and atomic publish behavior.
- Related #72251 is still open and broader; it is not a shipped implementation of this PR's intended change on main.
