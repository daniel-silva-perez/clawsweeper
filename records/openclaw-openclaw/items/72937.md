## Summary

Keep this PR open for maintainer review rather than close it as cleanup. The linked bug is plausible and the manager-level registration change is small, but current main already has generic capability fallback for cold provider lookup, another open PR (#73769) overlaps with a CLI-specific bootstrap and tests, and this PR currently carries a stale memory-host SDK shim rewrite that should not merge as-is.

## What This Changes

This PR registers built-in memory embedding providers before `MemoryIndexManager` creates an embedding provider and expands the source memory-host SDK embeddings shim to expose provider registration.

## Best Possible Solution

Adopt one canonical cold-registry fix for local memory embeddings, paired with a targeted regression test for `openclaw memory status --deep` or manager initialization with `provider: "local"`; keep `src/memory-host-sdk/engine-embeddings.ts` as the thin package bridge.

## Reproduction Assessment

Unclear against current main. The linked issue gives a concrete installed-package path (`openclaw memory status --deep` with `memorySearch.provider = "local"`), but current main also has manifest/capability fallback, so a live or targeted test is needed to prove the error still reproduces there.

## Solution Assessment

Unclear as submitted. The manager-level registration is a plausible narrow fix, but the stale memory-host SDK shim rewrite should be removed, and maintainers should decide whether manager bootstrap, CLI bootstrap, or the existing capability fallback is the intended contract.
