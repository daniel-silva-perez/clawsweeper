## Summary

Keep this PR open. Current main supports Microsoft Teams passwordless auth through explicit federated certificate and managed-identity configuration, but it still does not implement the PR's distinct `defaultAzureCredential` public auth type, DAC credential-chain behavior, or local `az login` development flow. The enterprise deployment comment and review discussion identify real product/security questions, so this is not conservative cleanup material.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto the current `@microsoft/teams.apps` Teams plugin runtime, decide whether OpenClaw wants a separate `defaultAzureCredential` mode alongside explicit federated auth, model any separate identity client ID instead of overloading the bot app ID, constrain and document DAC fallback behavior including local `az login`, update schema/types/generated config metadata/docs/tests, and carry forward the setup-reset fix if maintainers accept the feature.

## Risks / Open Questions

- The PR touches authentication, token acquisition, and dependency/lockfile surfaces; it needs maintainer security review before merge.
- Identity semantics remain unresolved: DAC identity pinning or fallback could fail or authenticate as an unintended principal unless modeled and documented explicitly.
- Current main has moved to the `@microsoft/teams.apps` runtime, while the PR was written against older token-provider paths, so the implementation likely needs a careful rebase and adaptation.
