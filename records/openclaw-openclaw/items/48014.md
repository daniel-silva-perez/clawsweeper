## Summary

Keep this PR open. Current main has Microsoft Teams passwordless support through `authType: "federated"` with certificate or managed identity, but it still does not implement this PR's explicit `defaultAzureCredential` auth type, DAC credential-chain behavior, or local `az login` flow. The setup reset fix described in the PR also still looks potentially relevant, and the PR needs a rebase/security review against the current Teams SDK architecture rather than cleanup closure.

## Best Possible Solution

Keep the PR open as a possible implementation candidate. The best path is to rebase it onto the current `@microsoft/teams.apps` implementation, decide explicitly whether OpenClaw wants a DAC mode in addition to `federated` managed identity, address the separate identity-client-id concern, document the local `az login` and Azure deployment behavior if supported, and carry forward the setup reset fix if it still reproduces.

## Risks / Open Questions

- The PR targets the older `@microsoft/agents-hosting` / `MsalTokenProvider` architecture, while current main uses `@microsoft/teams.apps` `App` plus `createMSTeamsTokenProvider`; a direct merge would need a substantial rebase.
- Credential selection is security-sensitive: the PR discussion flags that tying DAC managed/workload identity client IDs to the bot app ID can fail or authenticate the wrong principal when the bot app registration and managed identity differ.
- The setup-wizard reset fix may still be valid, but it needs to be adapted to the current `secret`/`federated` config names and current setup-core flow.
