## Summary

Keep this PR open. Current main has Microsoft Teams passwordless auth through explicit federated certificate and managed-identity modes, but it still does not implement the PR’s public `defaultAzureCredential` auth type, DAC credential-chain behavior, or local Azure CLI-style development flow. The enterprise deployment comment and the related #40855/#40884/#47934 history are substantive, so this remains a real implementation candidate rather than cleanup closure.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto the current `@microsoft/teams.apps` runtime, decide whether OpenClaw wants a separate `defaultAzureCredential` mode alongside existing federated managed identity, model any separate identity client ID explicitly, constrain and document DAC fallback behavior, and carry forward the setup-reset fix with current field names if maintainers accept the feature.

## Risks / Open Questions

- The PR touches authentication, token acquisition, and lockfile/dependency surfaces, so it needs maintainer security review before merge.
- The identity-selection concern remains material: bot app registrations and user-assigned managed identities can be distinct, so pinning DAC client IDs to the bot app ID needs an explicit product decision.
- DAC fallback is useful for local development, but production fallback to Azure CLI/env/other principals could authenticate as an unintended identity unless constrained or clearly documented.
- Because current main now uses `@microsoft/teams.apps`, the implementation likely needs a careful rebase and adaptation before it can be evaluated as merge-ready.
