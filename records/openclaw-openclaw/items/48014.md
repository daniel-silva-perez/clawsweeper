## Summary

Keep this PR open. The discussion and related items are substantive: it builds on #40855, #40884, and #47934, and the enterprise-deployment comment confirms this auth pattern is useful. Current main has passwordless Microsoft Teams auth through federated certificate and managed-identity modes, but it still lacks this PR's explicit `defaultAzureCredential` public config mode, DAC credential-chain behavior, and local Azure CLI-style development flow.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto the current `@microsoft/teams.apps` Teams runtime, decide whether OpenClaw wants a separate `defaultAzureCredential` mode alongside existing `federated` managed identity, model identity client-id selection explicitly, document supported Azure/local credential behavior, and carry forward the setup-reset fix with current field names if maintainers accept the feature.

## Risks / Open Questions

- The PR touches authentication, token acquisition, and lockfile/dependency surfaces, so it needs maintainer security review before merge even though the dependency is consistent with the stated feature.
- The unresolved identity-selection concern remains material: bot app registrations and user-assigned managed identities can be distinct, so pinning DAC client IDs to the bot app ID needs an explicit product decision.
- DAC fallback can be useful for local development, but in production webhook auth it may silently authenticate as an unintended principal unless OpenClaw deliberately constrains or documents the chain.
- The PR appears based on the older `@microsoft/agents-hosting` architecture, while current main uses `@microsoft/teams.apps`; it likely needs a rebase rather than cleanup closure.
