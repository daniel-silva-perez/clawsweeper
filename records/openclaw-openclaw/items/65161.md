## Summary

Keep open. Current main has partial heartbeat improvements, including lightweight bootstrap filtering and the narrower exec-event prompt-body fixes, but the central operator-visible contract is still unresolved: `last-heartbeat` is still an in-process singleton, `system heartbeat last` and status still read that surface, exec-event wakes can still populate it, and the docs still describe it as persisted while no durable heartbeat receipt writer exists.

## What This Changes

The issue requests reliable isolated heartbeat cadence evidence, separation of exec-event wakeups from canonical heartbeat status, effective lightweight context behavior, and clarification or implementation of `memory/heartbeat-state.json` state writing.

## Best Possible Solution

Define a canonical heartbeat receipt contract for actual scheduled/manual heartbeat runs, persist it in a runtime-owned location, make `system heartbeat last` and status read or clearly label that source, keep exec-event wakeups separate from cadence proof, and update public docs to distinguish runtime receipts from agent-managed `memory/heartbeat-state.json` bookkeeping.
