## Summary

Keep open. Current main still has the core visibility gap: the heartbeat CLI/status path reads an in-process singleton, docs describe that RPC as persisted, and exec-event wakes can still populate the same last-heartbeat surface even though narrower exec-event prompt-body bugs have been fixed.

## What This Changes

The issue requests reliable isolated heartbeat cadence evidence, separation of exec-event wakeups from canonical heartbeat status, effective lightweight context behavior, and clarification or implementation of `memory/heartbeat-state.json` state writing.

## Best Possible Solution

Define a canonical runtime-owned heartbeat receipt contract for actual scheduled/manual heartbeat runs, persist it durably, make `system heartbeat last` and status read or clearly label that source, keep exec-event wakeups separate from cadence proof, and update public docs to distinguish runtime receipts from agent-managed `memory/heartbeat-state.json` bookkeeping.

## Reproduction Assessment

Yes for the remaining source-level visibility problem: the CLI/RPC path can be traced to `last-heartbeat` -> `getLastHeartbeatEvent()` -> process-global `state.lastHeartbeat`, and exec-event ok paths can write that same surface. The cadence-stall and token-cost symptoms have corroborating live comments but were not fully live-reproduced in this read-only review.

## Solution Assessment

No, the current implementation is not the best complete solution because it still lacks a durable scheduled/manual heartbeat receipt source. The narrower exec-event payload fixes should stay, but the remaining fix should be a maintainer-designed receipt/status contract rather than more ad hoc filtering in the CLI.
