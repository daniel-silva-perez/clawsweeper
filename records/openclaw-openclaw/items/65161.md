## Summary

Keep open. Current main contains several partial heartbeat fixes, including lightweight bootstrap filtering, scheduler/busy-lane work, and the closed exec-event prompt-body fixes from #66382 and #68386, but the central visibility problem remains: `system heartbeat last` still returns a process-global in-memory event, exec-event runs can still be emitted through that same heartbeat event surface, and there is no durable scheduled/manual heartbeat receipt contract for operators to use as cadence proof.

## What This Changes

The issue asks OpenClaw to make isolated heartbeat behavior trustworthy by preserving cadence evidence, separating exec-event wakes from canonical heartbeat receipts, making lightweight context actually cheap, and clarifying or implementing heartbeat-state persistence.

## Best Possible Solution

Narrow this issue around the durable heartbeat visibility contract: define and implement runtime-owned receipts for actual scheduled/manual heartbeat runs, make `system heartbeat last` and status surfaces read or clearly label that canonical receipt source, ensure exec-event wakes cannot masquerade as cadence evidence, and clarify in public docs whether `memory/heartbeat-state.json` is agent-managed template state or a runtime-owned file.
