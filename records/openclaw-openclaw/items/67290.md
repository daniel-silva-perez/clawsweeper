## Summary

Keep open. Current main still renders both the shared app Usage header and the local Usage page header, so the underlying bug remains live, but this PR is not ready to merge because its locale cleanup removes the English key while leaving several shipped locale and translation-memory outputs on current main with the removed key.

## What This Changes

The PR removes the local Usage page title/subtitle block, deletes `usage.page.subtitle` from English plus a subset of generated locale and translation-memory files, and adds small Usage CSS helper rules.

## Best Possible Solution

Land a focused Usage-header fix that removes the desktop duplicate, keeps the Filters card context, updates English as the source of truth, regenerates every shipped locale plus `.i18n` metadata, and documents or tests the intended mobile header behavior.

## Reproduction Assessment

Yes. Current main can be reproduced statically: the app shell renders the shared Usage title/subtitle and `renderUsage` also renders a local Usage title/subtitle block; related issue #45709 supplies screenshot evidence for the same path.

## Solution Assessment

No, not as currently authored. Removing the local Usage header is the narrow right direction, but the locale deletion must go through the normal i18n sync flow and the mobile header behavior needs an explicit maintainer call before merge.
