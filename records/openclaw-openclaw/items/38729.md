## Summary

Keep this PR open, but it is not merge-ready against current main. The proposed callback targets a deprecated legacy hook, while Anthropic now advertises thinking support through a provider profile that is returned before that hook is consulted.

## What This Changes

The PR adds an Anthropic `supportsXHighThinking` callback for Claude Opus 4.6 model IDs in `extensions/anthropic/register.runtime.ts`.

## Best Possible Solution

Keep Claude thinking capabilities centralized in `resolveClaudeThinkingProfile`. If Opus 4.6 should expose max effort, update that shared profile plus the Anthropic/provider thinking tests and thinking docs; otherwise ask for the PR to be closed or revised with the explicit policy that Opus 4.6 remains adaptive-only in the advertised profile.

## Reproduction Assessment

Yes. A static reproduction path is enough here: current main returns the Anthropic provider profile before checking `supportsXHighThinking`, and the existing thinking test locks that precedence.

## Solution Assessment

No. Adding a legacy `supportsXHighThinking` callback is not the best fix because Anthropic already uses `resolveThinkingProfile`; the maintainable path is to change the shared Claude profile if the capability policy is approved.
