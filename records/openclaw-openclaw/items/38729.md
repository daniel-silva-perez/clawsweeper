## Summary

Keep this PR open, but it is not merge-ready because the added legacy hook is bypassed by the current Anthropic provider profile path. Current main already resolves Anthropic thinking levels through `resolveThinkingProfile`, and that non-empty profile returns before `supportsXHighThinking` is consulted.

## What This Changes

The PR adds an Anthropic `supportsXHighThinking` callback for Claude Opus 4.6 model-id prefixes in `extensions/anthropic/register.runtime.ts`.

## Best Possible Solution

Keep Claude thinking capabilities centralized in `resolveClaudeThinkingProfile`. If maintainers want Opus 4.6 max effort exposed to users, update that shared profile plus Anthropic/provider thinking tests and thinking docs; otherwise ask for the PR to be closed or revised with the explicit policy that Opus 4.6 remains adaptive-only in the advertised profile.

Yes. A static reproduction path is high-confidence: current main returns the Anthropic provider profile before checking `resolveProviderXHighThinking`, and the existing Anthropic test locks Opus 4.6 as adaptive-only without `xhigh` or `max`.

## Solution Assessment

No. Adding a legacy `supportsXHighThinking` callback is not the best way to solve this because Anthropic already uses `resolveThinkingProfile`; the maintainable path is to change `resolveClaudeThinkingProfile` if the capability policy is approved.
