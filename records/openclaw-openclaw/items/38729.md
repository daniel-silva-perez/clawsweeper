## Summary

Keep this PR open, but it is not merge-ready: current main resolves Anthropic thinking through a non-empty provider profile before the legacy xhigh hook is consulted, so the proposed hook would not expose `xhigh`/`max` for Claude Opus 4.6 in the current user-facing path.

## What This Changes

The PR adds an Anthropic `supportsXHighThinking` callback for Claude Opus 4.6 model-id prefixes in `extensions/anthropic/register.runtime.ts`.

## Best Possible Solution

Keep Claude thinking support defined through `resolveClaudeThinkingProfile`. If Opus 4.6 max effort should be user-facing, update that shared profile and the matching Anthropic/provider thinking tests and docs; otherwise close or revise this PR with the explicit policy that Opus 4.6 remains adaptive-only in the advertised profile.

## Reproduction Assessment

Yes. A static reproduction path is high-confidence: current main returns the Anthropic provider profile before checking `resolveProviderXHighThinking`, and the Anthropic regression test locks Opus 4.6 as adaptive-only without `xhigh` or `max`.

## Solution Assessment

No. Adding a legacy `supportsXHighThinking` callback is not the best way to solve the issue because Anthropic already uses `resolveThinkingProfile`; the maintainable path is to change `resolveClaudeThinkingProfile` if the capability policy is approved.
