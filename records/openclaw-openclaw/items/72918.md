## Summary

Keep open. Current main has useful display-layer mitigations for heartbeat prompts and HEARTBEAT_OK rows, but the broader report is still active because internal heartbeat/exec turns are not source-tagged as internal system input, and overlapping open PRs are still trying to fix the remaining transcript/session contamination paths.

## What This Changes

The issue requests that Control UI/WebChat consume async completion and heartbeat-only events internally instead of showing internal prompts or HEARTBEAT_OK in the visible chat transcript.

## Best Possible Solution

Keep the issue open until maintainers choose and land a coordinated fix that tags internal heartbeat/cron/exec turns at origin, keeps chat.history/SSE/Control UI suppression based on that provenance, and suppresses no-op exec heartbeat delivery without blocking meaningful user-facing async completion summaries.

## Reproduction Assessment

Unclear: there is no minimal live repro, but the issue comments provide transcript-store evidence and current source/tests cover the heartbeat prompt plus HEARTBEAT_OK history path. A maintainer can validate the fix with targeted heartbeat/chat-history tests and a Control UI run that triggers a metadata-only async exec completion.

## Solution Assessment

No: the current display filtering is a mitigation, not the best complete fix for the issue as reported. The better direction is source-level internal provenance plus no-op heartbeat delivery suppression, with #68262, #66749, and #73785 reviewed as candidate pieces.
