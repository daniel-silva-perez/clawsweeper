## Summary

Keep open. Current main still has only bundled, plugin, managed, and workspace hook sources, all behind the existing user-configured loading and enable/disable model; there is no admin-owned policy tier, policy-first ordering, policy inventory, or disable immunity. This is a valid core/security design request rather than something current plugins can enforce outside the runtime.

## What This Changes

Request an admin-managed, machine-wide hook discovery tier that loads before user-controlled hooks and cannot be disabled by user config or `openclaw hooks disable`.

## Best Possible Solution

Design and implement a core policy-hook tier with hardened machine-wide discovery, policy-first execution ordering, immunity from user hook toggles, policy-aware CLI inventory, public docs, and regression coverage for both POSIX and Windows trust-boundary behavior.

## Reproduction Assessment

Not applicable as a feature request, but the gap is high-confidence: inspecting current main and v2026.4.27 shows only the four existing hook tiers plus user-configurable load/disable gates.

## Solution Assessment

Unclear pending maintainer security design. A core policy tier appears to be the right layer if OpenClaw accepts this capability, because current plugin or hook-pack surfaces cannot enforce OS-owned discovery roots, trusted ordering ahead of user hooks, or immunity from user config.
