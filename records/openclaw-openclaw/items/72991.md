## Summary

Keep open. Current main still has only user/plugin/workspace-managed hook sources, user-controlled internal hook gating, registration-order dispatch, and no policy-only CLI inventory; the requested machine-wide, admin-owned policy tier remains unimplemented. Related #38670 is open and agent-scoped, so it does not supersede this security/product design request.

## What This Changes

Add an admin-managed machine-wide hook discovery tier that loads first, cannot be disabled by user config or `openclaw hooks disable`, and is visible as policy-owned inventory.

## Best Possible Solution

Design a core policy-hook tier first: hardened OS-specific discovery, deterministic policy-before-user ordering, disable immunity, policy-aware CLI/status/docs, clear deny versus fail-open semantics, and regression tests. Existing plugin and hook-pack surfaces should not be treated as sufficient unless a new core trust-boundary API makes administrator enforcement possible.

## Reproduction Assessment

Not applicable as a feature request. The gap is high-confidence from current-main source and docs: only bundled/plugin/managed/workspace sources are modeled, and user configuration still gates internal hook discovery and dispatch.

## Solution Assessment

Unclear pending maintainer security design. If OpenClaw accepts this capability, a core runtime tier appears to be the narrow maintainable layer because existing plugin and hook-pack mechanisms cannot enforce OS-owned roots, trusted ordering, or immunity from user disablement.
