## Summary

Keep open. Current main still lacks the requested machine-wide, admin-owned hook policy tier: hook source types, discovery, dispatch, CLI inventory, and docs remain limited to user/plugin/workspace-managed hooks and user-controlled config. This is core security/product design work, not ClawHub work, because a plugin cannot enforce OS-owned policy roots, policy-first ordering, or immunity from user disablement.

## What This Changes

The issue requests an admin-managed machine-wide hook discovery tier that always loads before user-controlled hooks and cannot be disabled by user config or `openclaw hooks disable`.

## Best Possible Solution

Design a core policy-hook tier with hardened POSIX and Windows discovery, deterministic policy-first ordering, clear deny versus fail-open semantics, policy-aware CLI inventory, docs, and regression tests before implementation.

## Reproduction Assessment

Not applicable as a feature request. The gap is high-confidence from current-main source and docs: only the existing hook sources are modeled, and user configuration still controls whether internal hooks load.

## Solution Assessment

Unclear pending maintainer security design. If OpenClaw accepts the capability, a core runtime tier appears to be the maintainable layer because plugin and hook-pack surfaces cannot enforce machine-owned roots, trusted ordering, or user-disable immunity.
