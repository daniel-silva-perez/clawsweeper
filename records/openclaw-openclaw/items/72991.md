## Summary

Keep open. Current main still does not provide the requested admin-owned machine-wide hook policy tier, and the remaining work is a core security/product design problem rather than a plugin-only addition because it needs OS-owned discovery roots, policy-first ordering, disable immunity, and auditable CLI inventory.

## What This Changes

Add an admin-managed, machine-wide hook discovery tier that always loads before user hooks, cannot be disabled by user config or `openclaw hooks disable`, and exposes policy-specific inventory.

## Best Possible Solution

Design a core policy-hook tier with hardened POSIX and Windows discovery, deterministic policy-first dispatch, disable immunity, fail-open error handling, policy-aware CLI output, docs, and focused regression tests before accepting implementation.

## Reproduction Assessment

Not applicable as a feature request. The gap is high-confidence from current-main source and docs because the existing hook model, discovery, CLI, and docs only cover bundled, plugin, managed, and workspace hooks gated by user-controlled config.

## Solution Assessment

Unclear pending maintainer security design. If OpenClaw accepts this enterprise governance capability, core hook loading and dispatch appear to be the maintainable layer because current plugin and hook-pack surfaces cannot enforce machine-owned roots, trusted ordering, or user-disable immunity.
