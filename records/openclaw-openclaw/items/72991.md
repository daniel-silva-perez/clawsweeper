## Summary

Keep open. Current main still lacks the requested machine-wide policy hook tier: the hook model exposes only bundled, plugin, managed, and workspace sources, discovery only scans user/workspace/plugin/bundled roots, user config can still disable internal hook loading, and the hooks CLI has no policy inventory or non-removable policy semantics. This remains a valid core/security design request rather than ClawHub/plugin work because the runtime would need to enforce OS-owned roots, trusted ordering, and disable immunity.

## What This Changes

The issue requests an admin-managed machine-wide hook discovery tier whose hooks run before user-controlled hooks, are loaded from hardened OS policy roots, and cannot be disabled by user config or `openclaw hooks disable`.

## Best Possible Solution

Design a core policy-hook tier with hardened POSIX and Windows discovery, deterministic policy-first ordering, explicit block/fail-open semantics, policy-aware CLI inventory, docs, and regression tests for trust-boundary behavior before implementing code.

## Reproduction Assessment

Not applicable as a feature request. The gap is high-confidence from current-main source and docs: only the four existing hook sources are represented, and user config still controls internal hook loading.

## Solution Assessment

Unclear pending maintainer security design. If OpenClaw accepts this capability, a core runtime policy tier appears to be the right layer because current plugin and hook-pack surfaces cannot enforce OS-owned discovery roots, trusted ordering before user hooks, or immunity from user config.
