## Summary

Keep open. Current main still has only bundled, plugin, managed, and workspace internal hook sources, all governed by the existing user-configured enable/disable model; there is no admin-owned policy tier, policy-first execution lane, policy-only inventory, or disable immunity. The request is a concrete core/security design gap rather than ClawHub/plugin work, because current extension surfaces cannot enforce OS-owned discovery roots or bypass user-controlled config.

## What This Changes

Request an admin-managed, machine-wide policy hook discovery tier with hardened roots, first-run ordering, CLI inventory, and immunity to user hook disable/config controls.

## Best Possible Solution

Design a core policy-hook tier with hardened machine-wide discovery, policy-first registration/execution semantics, immunity from user hook toggles, policy-aware CLI inventory, public docs, and regression coverage for POSIX and Windows policy-root behavior.

## Reproduction Assessment

Not applicable as a feature request, but the gap is high-confidence: current main source and docs show only the existing four hook sources and user-configured loading/disable gates.

## Solution Assessment

Unclear pending maintainer security design. If OpenClaw accepts the capability, a core runtime policy tier is the maintainable layer because plugins and hook packs cannot enforce OS-owned roots, trusted ordering ahead of user hooks, or immunity from user config.
