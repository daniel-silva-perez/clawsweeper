## Summary

Keep this PR open. Current main still rejects Telegram `dmPolicy="open"` unless the effective `allowFrom` includes `"*"`; the implemented recovery path is explicit `openclaw doctor --fix`, not startup/pre-validation auto-heal. The PR’s diff is small and does not touch dependencies, workflows, or package execution paths, but it would auto-insert a wildcard into Telegram access control, so it needs maintainer security/product judgment against current main and the later open-DM policy work.

## Best Possible Solution

Keep this PR open for maintainer security/product judgment. If startup recovery is still wanted, rework it on current main around #74112 semantics: mutate only the intended Telegram top-level or account surface, never widen sibling inherited allowlists, avoid module-level warning state, cover missing/empty/non-wildcard/inherited account cases, and update docs/changelog. If maintainers prefer the current explicit contract, close this PR intentionally and keep `openclaw doctor --fix` plus clear validation guidance as the supported repair path.

## Risks / Open Questions

- Auto-inserting `"*"` at startup can change Telegram DM access from restrictive-looking to public, which is security-sensitive for bots connected to tool-capable agents.
- The PR branch predates current main’s merged #74112 open-DM allowlist changes, so any implementation needs a rebase and review against the current effective-policy semantics.
- The PR diff still uses a module-level warning singleton, which can make warning behavior process-global and order-dependent if tests or validation flows start asserting it.
- Current main already has an explicit `openclaw doctor --fix` repair path, so replacing cold-start failure with implicit startup mutation is a maintainer product/security decision rather than a narrow bug fix.
