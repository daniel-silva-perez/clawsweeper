## Summary

Keep this PR open. Current main still rejects Telegram `dmPolicy="open"` without an effective `allowFrom` wildcard during normal config validation/startup. Main has a doctor-side repair for this shape, but that does not implement the PR's requested pre-validation/startup auto-heal. The PR also changes a security-sensitive Telegram access-control contract and now intersects with open security issue #73756.

## Best Possible Solution

Keep this PR open for maintainer security/product judgment. If maintainers want startup recovery, implement a narrow Telegram pre-validation normalization that only mutates the intended top-level channel or account, never widens sibling inherited allowlists, emits a testable warning without module-order coupling, and adds tests for missing, empty, non-wildcard, inherited, and sibling-account cases plus docs/changelog. If maintainers prefer the current explicit contract, close this PR intentionally after documenting that `openclaw doctor --fix` is the supported repair path and reconciling the related #73756 security posture.
