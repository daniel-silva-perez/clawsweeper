## Summary

Current main still does not make `backup create --no-include-workspace` archive-real for default workspaces nested under `~/.openclaw`. The flag removes workspace candidates from the backup plan, but the state directory remains a broad included source and the tar writer only filters installed plugin `node_modules`, not workspace paths under state. Docs and CLI help still describe the flag as excluding workspace files, so the reported UX/behavior gap remains.

## Best Possible Solution

Keep this issue open. The best fix is to make `--no-include-workspace` carry canonical nested workspace exclusions from backup planning into archive creation, then add a real archive regression test proving a file under `~/.openclaw/workspace` is absent when workspace inclusion is disabled. If maintainers choose the broader #40786 exclude-pattern path, that implementation should explicitly cover the default nested-workspace semantics before this issue closes.

## Risks / Open Questions

- Users may still archive large or sensitive workspace contents despite passing `--no-include-workspace` when those workspaces live under the default state directory.
- A fix needs canonical path handling for default, profile, custom, and symlinked workspaces so it excludes workspace payloads without dropping required state such as config, credentials, sessions, managed plugins, or auth profiles.
- Leaving implementation unchanged would require an explicit docs/help contract change, because current docs and help text still imply archive-level workspace exclusion.
