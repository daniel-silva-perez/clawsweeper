## Summary

Close as implemented on main. Current main preserves main-session cron systemEvent text through enqueue, cron-context tagging, and heartbeat prompt construction, and the same implementation is present in the shipped v2026.4.27 tag.

## What This Changes

The issue reports that cron jobs targeting the main session with a systemEvent payload record successful runs but lose payload.text before the agent prompt/context is built.

## Best Possible Solution

Keep the shipped transient-runtime-context implementation: cron systemEvent text should stay out of visible chat transcripts while being embedded in the cron-event prompt body for the agent. Leave ghost-run diagnostics and cross-route delivery leaks to their separate open reports.

## Reproduction Assessment

Yes. The issue has concrete reproduction steps, and the current-main source/test path maps directly to that flow: executeMainSessionCronJob enqueues payload text and runHeartbeatOnce builds a cron-event prompt body containing that text.

## Solution Assessment

Yes. The current approach is the narrowest maintainable fix because it preserves the 4.25 design goal of avoiding visible synthetic chat messages while ensuring the model receives the cron payload as transient runtime context.
