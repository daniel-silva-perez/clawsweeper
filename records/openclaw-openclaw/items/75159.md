## Summary

Keep open. The linked Feishu bug is valid and the extension-local direction is reasonable, but this PR currently adds an unconditional message-tool instruction that conflicts with supported automatic group delivery; it also lacks the required changelog entry for a user-facing fix.

## What This Changes

The PR adds Feishu topic-scope prompt helpers and tests, then appends a thread-reply message-tool hint to `GroupSystemPrompt` for `group_topic` and `group_topic_sender` sessions.

## Best Possible Solution

Keep the fix Feishu-owned, but make the topic guidance compatible with delivery mode. A corrected patch should either emit the override only for message-tool-only visible replies, phrase it conditionally for cases where the model chooses the message tool, or use the Feishu/plugin threading boundary so same-conversation topic sends inherit the right thread target without conflicting with automatic delivery.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current main prompts message-tool-only Feishu topic group turns toward `action=send`, while Feishu only sets native thread delivery for `action=thread-reply`; the PR regression is reproduced by configuring automatic group visible replies and observing the contradictory prompt instructions.

## Solution Assessment

No, not as written. The extension-local approach is the right ownership boundary, but the proposed prompt hint must respect source reply delivery mode or be conditional so automatic-delivery sessions are not told to call the message tool.
