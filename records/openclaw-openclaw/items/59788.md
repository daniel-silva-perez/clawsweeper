## Summary

Keep open: current main now has outbound transcript mirroring, but it still does not solve the reported non-default routed-agent case. The CLI path passes the default agent into outbound sends, and both CLI and Gateway send paths mirror using that effective agent instead of resolving the agent from the channel/account/peer routing rule first.

## What This Changes

Request that outbound `openclaw message send` and similar channel sends create or resume the routed agent session and persist the first outbound turn before an inbound reply arrives.

## Best Possible Solution

Add a shared outbound route-agent resolver that, for sends without an explicit agent or session key, resolves the agent from the same channel/account/peer binding semantics used by inbound routing, then mirrors the outbound turn into that routed session. Preserve existing explicit overrides and keep the channel/plugin APIs unchanged unless a genuinely reusable SDK seam is needed.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure a non-default agent bound to a WhatsApp account, call `message send` with that account, and the current code path selects the default agent before creating the outbound mirror route, so the transcript is written under the wrong agent.

## Solution Assessment

No. Current main is only a partial solution because it persists outbound sends, but not into the routed non-default agent session requested here; the narrow maintainable fix is to resolve the outbound agent from existing route bindings before building the mirror session.

## Repair Work Prompt

Fix outbound-first routed sessions for `openclaw message send` and Gateway `send`. Observable bug: when a channel/account is bound to a non-default agent, an outbound send with that explicit account is mirrored under the default agent instead of the routed agent, so the first outbound turn is missing from the bound agent's transcript when the peer replies. Implement a narrow shared helper or local logic that resolves the effective outbound agent from existing routing bindings using channel/account/target/peer when no explicit agent or session key is supplied. Preserve precedence for explicit `agentId`/future `--agent`, provided `sessionKey`, and existing channel target normalization. Reuse existing inbound routing/session-key contracts where possible; do not add owner-specific WhatsApp logic to core and do not change plugin SDK APIs unless strictly necessary. Add regression coverage for a non-default account-bound WhatsApp or generic direct-message channel send, proving the mirror/session key uses the bound agent and respects `session.dmScope`, and add Gateway send coverage if that path is changed. Add a changelog fix entry with reporter credit `Thanks @fridary`.
