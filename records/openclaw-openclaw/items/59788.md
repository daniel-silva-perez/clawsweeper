## Summary

Keep open: current main has generic outbound session mirroring, but this issue’s non-default, account-routed WhatsApp case is still not implemented because `openclaw message send` selects the default CLI agent before outbound routing. https://github.com/openclaw/openclaw/issues/59788

## What This Changes

Add support for outbound WhatsApp/account sends to create or resume the routed agent session and persist the first outbound turn before the peer replies.

## Best Possible Solution

Add a generic outbound agent resolver before session-route derivation: when no explicit agent/session is supplied, resolve the selected channel/account through route bindings with inbound-compatible account-match semantics, then mirror the delivered opener into that agent’s derived DM session. Preserve default-agent fallback when no binding matches and avoid a WhatsApp-only special case.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure a non-default agent with a WhatsApp account binding, run `openclaw message send --channel whatsapp --account <bound-account>`, and the current CLI still passes the default agent into outbound routing.

## Solution Assessment

No. Current main solves generic outbound transcript mirroring, but not the requested routed-account agent selection; a small shared resolver plus focused tests is the narrowest maintainable fix.

Fix https://github.com/openclaw/openclaw/issues/59788. Observable bug: `openclaw message send --channel whatsapp --account <bound-account> --target <peer> --message <text>` currently mirrors the outbound opener using the default CLI agent, so account-scoped bindings to a non-default agent do not create/populate that agent's session transcript. Implement a generic outbound agent-resolution path, not a WhatsApp-only special case: when no explicit agent/session is supplied, resolve the selected channel/account through route bindings with the same account-match semantics as inbound routing; preserve current default-agent behavior when no binding matches. Add an explicit `--agent <id>` only if needed for override/ambiguity, but keep automatic unambiguous account-bound routing working. Do not change WhatsApp transport/auth behavior or the transcript message format. Add focused tests for a WhatsApp account bound to a non-default agent, dmScope direct-message session key derivation, explicit `--account`, default fallback, and existing explicit session/agent precedence. Update docs/changelog only if user-facing CLI behavior or flags change, with proper reporter credit.
