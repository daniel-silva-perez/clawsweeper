## Summary

Keep open. Current main still treats `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` as a literal Discord target rather than resolving it from the environment. The cron status persistence path only records `delivered` when the isolated runner returns `delivered: true`; requested delivery with no delivered flag still becomes `lastDeliveryStatus: "unknown"`. Existing docs/tests cover explicit `channel:<id>`/`user:<id>` targets and delivery status persistence, but not env-var delivery targets.

## Best Possible Solution

Implement explicit, documented cron delivery target environment or SecretRef resolution before outbound target normalization, then add focused tests proving `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` resolves to the configured channel, Discord delivery reports `delivered: true`, and cron state persists `ok` plus `lastDeliveryStatus: "delivered"`. If maintainers prefer existing SecretRef syntax instead of `env:`, update validation/docs and error messages so this configuration fails clearly rather than becoming a literal Discord target.
