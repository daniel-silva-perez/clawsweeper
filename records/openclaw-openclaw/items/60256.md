## Summary

Keep open. Current main still forwards `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` as a literal Discord target instead of resolving it, so Discord delivery can fail after the agent succeeds and cron then persists an error/unknown delivery state. The existing docs describe explicit Discord target prefixes and the general SecretRef/env-template mechanisms, but not `env:` cron delivery targets.

## What This Changes

The issue asks cron Discord announce delivery to resolve environment-variable delivery targets so successful sends persist as `ok` and `delivered` instead of `error` and `unknown`.

## Best Possible Solution

Add an explicit cron delivery target resolution contract before outbound normalization, preferably reusing the existing SecretRef/env-template machinery and documenting the supported syntax. If `env:` is not accepted as a public cron API, reject it during job create/edit with an actionable error instead of letting it become a literal Discord target.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: create an isolated cron announce job with `channel: "discord"` and `to: "env:DISCORD_BACKUPS_CHANNEL"`; current main forwards that literal target into Discord delivery and leaves `delivered` unset on failure. I did not run a live Discord cron job because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the supported syntax. The narrowest maintainable fix is centralized cron delivery target resolution or validation before channel-specific outbound normalization, not a Discord-only special case.
