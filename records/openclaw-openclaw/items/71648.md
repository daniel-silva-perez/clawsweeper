## Summary

Keep this PR open. Current main still has the MCP channel bridge pending-map leak pattern the PR targets, and the linked same-author issue remains open, so this should stay available for maintainer review rather than being closed before merge.

## What This Changes

The PR wraps MCP bridge pending Claude permission and approval entries with timestamps, adds a lazy TTL sweeper plus close-time clearing, guards post-close pending-map writes, and adds focused regression tests with one fixture update.

## Best Possible Solution

Land or rebase a focused MCP bridge memory-bound fix that keeps current main's gateway startup/readiness behavior, evicts stale pending permission and approval entries by stable timestamps, clears both maps on close, and leaves any strict FIFO/cap policy as a separate follow-up if maintainers want stronger burst bounds.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: add a Claude permission request without a matching yes/no reply, or observe an approval requested event without the matching resolved event; the corresponding map entry has no TTL or close-clear cleanup path.

## Solution Assessment

Yes for the scoped slow-leak bug. A lazy self-terminating TTL sweeper plus close cleanup is the narrowest maintainable fix for missed replies and missed resolved frames, while a separate cap/FIFO follow-up is a reasonable safer boundary for burst-policy decisions.
