## Summary

Keep open. Current main has useful adjacent mitigations for reminder promises, strict-agentic plan-only turns, and terminal empty/reasoning-only runs, but it still does not implement the requested general reply-pipeline guard for unbacked follow-up promises such as promising to come back after a memory lookup. The reporter's 2026.4.23 healthy datapoint is useful, but it was explicitly non-exhaustive and the source still leaves this broader class uncovered.

## What This Changes

The issue requests a runtime guard that prevents final user-facing replies from promising a later follow-up unless the same turn actually starts a follow-up mechanism or rewrites/annotates the promise.

## Best Possible Solution

Add a small runtime consistency guard near the existing reminder guard: detect final user-visible follow-up/future-work commitments, check same-turn evidence for real backing actions such as cron creation, sessions_spawn/background task creation, queued follow-up, or committed message-tool delivery, and retry/rewrite/annotate only when unsupported. The implementation should keep the existing reminder guard behavior intact while adding targeted tests for the memory_search example and backed follow-up cases.

## Reproduction Assessment

Yes, at a focused unit level: simulate an embedded run with one `memory_search` tool meta and final text like 'I need to read the files and then I’ll come back with one short message.' Current main does not match the reminder guard and the planning-only retry path can skip because `memory_search` is not in the single-action safe-tool list; no live channel reproduction was run in this read-only review.

## Solution Assessment

No. The current partial mitigations are useful for related plan-only/reminder failures, but they are not the narrowest complete solution for this issue because they do not validate generic follow-up promises against same-turn backing actions.
