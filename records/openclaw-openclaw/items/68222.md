## Summary

Keep open. Current main still exposes the Kimi Code provider, still rewrites Kimi tagged tool-call text into executable tool calls, and still treats any executed `sessions_yield` as a clean paused/end-turn exit. I did not find source, tests, release, or discussion evidence that current main prevents or recovers from unwanted Kimi-generated `sessions_yield` calls; the open related Kimi loop report is adjacent but does not clearly cover this specific premature-yield/manual-resume symptom.

## What This Changes

The issue requests that Kimi Code long-running conversations continue autonomously instead of stopping on model-emitted `sessions_yield` calls that require manual input to resume.

## Best Possible Solution

Add a focused guard or recovery path around `sessions_yield` execution for cases where there is no pending spawned-child completion or expected follow-up event, and cover the Kimi tagged-tool stream path with a regression test that proves premature yield does not leave the parent run waiting for manual input.

## Reproduction Assessment

Unclear. The issue gives plausible manual steps and a screenshot for Kimi Code on version 2026.4.15, and current-main code inspection confirms the interruption mechanics, but there is no captured transcript or live reproduction proving the first unwanted yield state.

## Solution Assessment

Unclear. Blocking all Kimi `sessions_yield` calls would be too broad because the tool is a documented subagent orchestration mechanism; the safer direction is a narrow runtime guard/recovery keyed to missing pending child/follow-up state plus Kimi-specific regression coverage.
