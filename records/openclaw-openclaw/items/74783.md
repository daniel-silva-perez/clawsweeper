## Summary

Keep this PR open. It is a useful implementation candidate for the open connection-telemetry request and current main does not already expose per-client `health.connection`, but the patch should not merge as-is because the proposed stale check never expires clients that have not produced their first pong.

## What This Changes

The PR adds an optional `HealthSummary.connection` block, per-WebSocket ping/pong RTT tracking in gateway maintenance and connection handling, docs for `health.connection`, and focused gateway/CLI health tests.

## Best Possible Solution

Land the additive `health.connection` overlay after tightening stale semantics so an authenticated socket with missed protocol pongs eventually reports `connected: false`, while keeping connection data out of the shared cached `HealthSummary` and documenting the exact first-pong and stale-threshold behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Expire clients that never answer protocol pings:** `src/gateway/server/connection-health.ts:53`
  - body: `lastHeartbeatAt` stays `null` until the first accepted pong, and this branch treats `null` as fresh forever. A client or network path that stops answering protocol pings before the first pong will keep returning `connected: true` even after many missed 5s pings, so the new stale threshold does not actually protect the initial no-pong state.
  - confidence: 0.88
