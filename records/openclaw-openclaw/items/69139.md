## Summary

Keep this PR open. Current main still keeps Gateway `AgentParamsSchema` closed with no root-level `paperclip` field, and the `agent` handler validates params before dispatch, so the Paperclip adapter payload described in the PR is still rejected. The PR diff is small and not supply-chain sensitive, but maintainers should decide whether the compatibility field should stay Paperclip-specific or become a generic adapter metadata seam.

## Best Possible Solution

Keep this PR open for maintainer review. The remaining work is to land an additive compatibility fix for adapter metadata on `agent` params, preferably as a generic extension-safe metadata field unless maintainers explicitly approve the Paperclip-specific shim, with focused protocol/server tests and any needed user-facing changelog entry.
