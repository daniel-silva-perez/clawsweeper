## Summary

Keep open. Current main still excludes Responses-family transports from the malformed tool-call argument repair gate, and this report is asking for maintainer/provider-contract guidance before a PR rather than reporting an observed Codex failure. The HTTP Responses/Codex path appears technically close to the existing repair wrapper because it emits `toolcall_delta`, but the native WebSocket Responses path is different enough that a blind gate expansion would be premature.

## What This Changes

The issue asks whether malformed tool-call argument repair should be enabled for `openai-codex-responses` and possibly other Responses-family transports.

## Best Possible Solution

Decide the Responses-family policy explicitly. If maintainers confirm these transports can emit recoverable malformed argument deltas, add the exact supported model APIs to the gate with focused tests for OpenAI, Codex, and Azure Responses, and separately cover or document the WebSocket final-response path.

## Reproduction Assessment

Unclear. We can statically reproduce the current exclusion from the gate and tests, but the issue explicitly says no malformed Codex arguments have been observed, so there is not yet a high-confidence runtime reproduction of user-facing failure.

## Solution Assessment

Unclear. A narrow gate expansion may be the right implementation for HTTP/SSE Responses streams because they emit `toolcall_delta`, but the WebSocket path does not use the same internal event stream for tool arguments, so the best fix needs maintainer/provider-contract review first.
