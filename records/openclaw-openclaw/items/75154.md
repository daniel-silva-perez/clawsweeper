## Summary

Keep open. Current main still excludes Responses-family transports from the malformed tool-call argument repair gate; the static behavior is real, but the issue asks for maintainer/provider-contract guidance before broadening because no malformed Codex trace is reported and the HTTP and WebSocket Responses paths differ.

## What This Changes

The issue asks whether malformed tool-call argument repair should be enabled for `openai-codex-responses` and possibly other Responses-family transports.

## Best Possible Solution

Decide the Responses-family policy explicitly. If maintainers confirm Codex/Responses can emit recoverable malformed argument deltas, add only the supported transport IDs to the gate with focused tests for `openai-responses`, `openai-codex-responses`, and `azure-openai-responses`, and either cover or document the separate WebSocket final-response behavior.

## Reproduction Assessment

Unclear. We can reproduce the current exclusion statically from the gate and tests, but the report explicitly says no malformed Codex arguments have been observed, so there is no high-confidence runtime reproduction of a user-facing failure yet.

## Solution Assessment

Unclear. A narrow gate expansion may be best for HTTP/SSE Responses because that path emits `toolcall_delta`, but the WebSocket path is different, so the maintainable fix needs provider-contract review before implementation.
