## Summary

Keep open. Current main still excludes `openai-codex-responses` and other Responses-family transports from the malformed tool-call argument repair gate, and an open linked PR is already the concrete implementation candidate for this issue.

## What This Changes

The issue asks whether malformed tool-call argument repair should be broadened from Kimi/Anthropic and `openai-completions` to `openai-codex-responses` and possibly other Responses transports.

## Best Possible Solution

Make the Responses-family policy explicit in the linked fix path: enable the repair only for transport IDs whose HTTP/SSE stream emits compatible `toolcall_delta` events, keep or document any WebSocket-only exclusions, and add focused positive and negative gate tests for the chosen matrix.

## Reproduction Assessment

Unclear. We can reproduce the current exclusion statically from the gate and tests, but the issue says no malformed Codex arguments have been observed, so there is not yet a high-confidence runtime reproduction of a user-facing failure.

## Solution Assessment

Unclear. A narrow transport-specific expansion may be the best fix for Codex/Azure Responses if maintainers accept the provider contract evidence, but the WebSocket path means the exact policy should be settled in review rather than closed by cleanup.
