## Summary

Keep this issue open. Current main still hard-codes isolated cron run summaries to 2000 UTF-16 code units before they become the `summary` stored in cron run-log entries, while `cron.runLog` only configures file pruning by bytes and retained lines. The requested configurable or larger per-entry summary limit remains unimplemented.

## Best Possible Solution

Add a cron-owned `cron.runLog.maxSummaryChars` config field with a conservative default, validation, help/generated schema, and docs. Plumb the resolved value through isolated cron summary selection and delivery/subagent resummary paths so long agent outputs can be retained up to the configured cap while `maxBytes` and `keepLines` still bound file growth. Add focused tests for default behavior, configured >2000-character summaries, invalid config handling, and UTF-16-safe truncation.
