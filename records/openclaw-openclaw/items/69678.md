## Summary

Current main still hard-codes isolated cron run summaries to 2000 UTF-16 code units before they are emitted to cron run history. The `cron.runLog` config/schema/docs still only expose pruning controls, so the requested configurable or larger per-entry summary cap remains valid and narrow enough for a focused fix PR.

## What This Changes

The issue requests a configurable or larger cron run-log summary character limit so long agent outputs remain readable and auditable in `openclaw cron runs`.

## Best Possible Solution

Keep this issue open and implement a cron-owned `cron.runLog.maxSummaryChars` setting with a conservative default such as 20000, schema/help/label/docs coverage, and plumbing through isolated cron summary selection and subagent follow-up paths while preserving `maxBytes` and `keepLines` pruning.
