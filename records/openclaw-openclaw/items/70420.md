## Summary

Keep open. Current main still separates Discord/channel runtime health from persisted session-store listing, and the docs still lack the explicit negative contract that `openclaw sessions`, Gateway `sessions.list`, and agent `sessions_list` are not channel/provider liveness checks. A narrow docs guidance PR is appropriate.

## Best Possible Solution

Land a narrow guidance fix: document that `openclaw sessions`, Gateway `sessions.list`, and agent `sessions_list` enumerate persisted conversation/session-store entries, not provider socket or channel liveness. Point heartbeat/connectivity checks to `openclaw channels status --probe`, `openclaw status --deep`, `openclaw health --verbose`, and readiness/channel-status surfaces. Do not create synthetic session rows unless maintainers intentionally choose that cross-channel behavior.
