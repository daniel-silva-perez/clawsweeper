## Summary

Keep open. Current main still separates channel/provider runtime health from persisted session rows: Discord reconnect updates channel status, while `sessions.list`/`sessions_list` enumerates the session store. The docs point users toward live channel probes, but they still do not explicitly say that session listing is not a provider socket-liveness signal, so the reported false-alarm path remains actionable.

## What This Changes

The issue asks OpenClaw to align Discord reconnect/session guidance so session-based checks do not treat a healthy reconnected Discord provider as disconnected.

## Best Possible Solution

Land a narrow docs clarification that `openclaw sessions`, Gateway `sessions.list`, and the agent `sessions_list` tool list persisted conversation/session-store entries, not provider socket or channel liveness. Keep live connectivity guidance pointed at `openclaw channels status --probe`, `openclaw status --deep`, `openclaw health --verbose`, and readiness/channel-status surfaces; avoid synthetic session rows unless maintainers intentionally change the cross-channel contract.

## Reproduction Assessment

Yes. The issue supplies runtime logs, and current-main source gives a high-confidence reproduction path: after a quiet Discord reconnect, runtime status can be connected while `sessions.list` still returns only persisted store rows that already exist.

## Solution Assessment

Yes. A docs-first fix is the narrowest maintainable solution because current code intentionally separates runtime liveness from session persistence; materializing heartbeat sessions would be a broader cross-channel behavior change.
