## Summary

Current main still matches the reported shape: the channel health monitor restarts the Discord account runtime, Discord reports runtime status back through channel status patches, and `sessions.list`/`sessions_list` is still backed by persisted session-store rows rather than channel/provider liveness. The existing docs say sessions are stored conversations and `channels status --probe` is the live channel path, but they do not explicitly warn heartbeat/session checks away from using `sessions_list` as a Discord connectivity signal.

## Best Possible Solution

Keep this open and land a narrow docs/guidance fix: explicitly document that `sessions_list`/`openclaw sessions` list persisted conversation sessions, not provider socket liveness, and point heartbeat/connectivity checks at `openclaw channels status --probe`, `openclaw status --deep`, or Gateway readiness/channel status surfaces. Avoid changing session-store materialization unless a maintainer chooses that product behavior for all relevant channels.
