## Summary

Keep open. Current main still makes Discord reconnect health and persisted session listing separate signals, and the user-facing docs still do not explicitly warn that `sessions_list`, `sessions.list`, or `openclaw sessions` are not provider socket-liveness checks.

## What This Changes

The issue requests that Discord reconnect/session health guidance be aligned so session listing is not mistaken for provider socket liveness after a health-monitor restart.

## Best Possible Solution

Land a narrow docs clarification that `openclaw sessions`, Gateway `sessions.list`, and the agent `sessions_list` tool enumerate persisted conversation/session-store rows, not provider socket or channel liveness. Keep live connectivity guidance pointed at `openclaw channels status --probe`, `openclaw status --deep`, `openclaw health --verbose`, and readiness/channel-status surfaces; do not synthesize session rows unless maintainers intentionally change the cross-channel contract.

## Reproduction Assessment

Yes. The issue provides reconnect logs and current-main source gives a high-confidence reproduction path: after a quiet Discord reconnect, channel runtime status can be connected while `sessions.list` still returns only existing persisted session-store rows.

## Solution Assessment

Yes. A docs-first fix is the narrowest maintainable solution because current code intentionally separates runtime liveness from session persistence; creating heartbeat/session rows would be a broader cross-channel behavior change.
