## Summary

Keep open. Current main still runs Discord voice turns through the voice-channel-derived route with text delivery disabled, while the strict Discord voice config/schema/docs do not expose a bridge-to-text option; the maintainer discussion also explicitly kept this as dedicated feature work rather than cleanup.

## What This Changes

Request a Discord voice mode where spoken transcripts are posted into a bound text channel, processed in that text-channel agent session, and the same assistant reply is both posted to text and played through TTS in voice.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed config/schema/docs, explicit bridge target selection, speaker-attributed transcript posting, routing through the selected text-channel session, normal Discord text delivery, and TTS fan-out of the same final reply; keep realtime audio provider support as a separate follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice joins resolve a session from the voice channel id, voice turns call agentCommandFromIngress() with deliver:false, and strict config/schema/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin, but the precise config and permission model need maintainer design before an automated implementation is the safest path.
