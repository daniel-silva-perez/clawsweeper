## Summary

Keep open. Current main still does not implement the requested Discord voice-as-I/O bridge: voice joins resolve the session from the voice channel, voice turns call the agent with outbound text delivery disabled, and the Discord voice config/schema/docs expose no bridge-to-text mode. The issue discussion also has a maintainer pass explicitly leaving this as dedicated Discord voice feature work rather than cleanup already covered by routing fixes.

## What This Changes

Request to add a Discord voice mode where STT transcripts enter a bound text-channel agent session, text replies are posted normally, and the same replies are spoken back into the voice channel.

## Best Possible Solution

Build a dedicated bundled Discord plugin feature that adds typed bridge configuration, validates a text-channel target, posts speaker-attributed transcripts into that text session, lets the normal Discord text delivery path publish replies, and fans the final reply out to voice TTS without losing the existing isolated voice mode.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice joins bind the route to the voice channel id, voice segments invoke `agentCommandFromIngress()` with `deliver: false`, and the strict config schema has no bridge fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin and matches the observed user problem, but the exact config names, defaults, target selection, transcript visibility, and permission inheritance need maintainer design before a narrow implementation is clearly the best fix.
