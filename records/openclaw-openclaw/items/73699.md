## Summary

Keep open. Current main still does not implement the requested Discord voice-as-I/O bridge: voice auto-join and manual joins route voice turns through the voice channel session, voice segments call the agent with delivery disabled, and the Discord voice config/schema/docs do not expose bridge-to-text fields. The issue discussion also includes a maintainer pass explicitly leaving this open as dedicated feature work.

## What This Changes

Request a Discord voice mode where STT transcripts are posted into a selected text-channel session and the resulting text-channel reply is both delivered to Discord text and spoken back over voice.

## Best Possible Solution

Build this as a dedicated bundled Discord plugin feature: add typed config/schema/docs for bridge mode and target selection, route STT into the selected text-channel session, post speaker-attributed transcripts, deliver the assistant reply through normal Discord text delivery, and enqueue that same final reply for voice playback.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: Discord voice joins resolve a route from the voice channel id, voice turns call `agentCommandFromIngress()` with `deliver: false`, and the config schema rejects the proposed bridge fields.

## Solution Assessment

Unclear. The requested bridge direction belongs in the Discord plugin, but the exact config and permission semantics need maintainer design; a narrow `sessionChannelId` field alone is not the best complete solution for this issue.
