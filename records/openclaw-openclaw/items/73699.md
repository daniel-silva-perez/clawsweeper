## Summary

Keep open. Current main still does not implement the requested Discord voice-as-I/O bridge: voice joins bind the voice channel id as the session channel, voice ingress still calls the agent with `deliver: false`, and the typed/config/docs surface has no `bridgeToText`, bridge mode, transcript prefix, or bridge text-channel target. The related #53562/#53741 work overlaps but is narrower than this issue’s full transcript-posting plus text-and-TTS reply fan-out behavior.

## Best Possible Solution

Keep this open unless maintainers explicitly choose a canonical tracker. The likely implementation belongs in the bundled Discord plugin voice path: add typed/schema/docs config for bridge mode and target text channel, resolve voice input against the chosen text-channel route/session, post speaker-attributed transcripts, deliver replies through normal Discord text delivery, and enqueue the same final reply for voice TTS playback. Coordinate with #53562/#53741 if that narrower `sessionChannelId` work becomes the base.
