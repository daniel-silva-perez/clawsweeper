## Summary

Keep open. Current main still lacks the requested Discord voice-as-I/O bridge: voice joins resolve the agent route from the voice channel, segment processing calls `agentCommandFromIngress` with `deliver: false`, and the typed config/schema/docs do not expose bridge fields. The maintainer discussion explicitly says to leave this open for a dedicated feature change; related #53562/#53741 cover only narrower `sessionChannelId` routing work and are still open.

## Best Possible Solution

Keep this open for a dedicated bundled Discord plugin feature. The best implementation path is to add typed/schema/UI/docs config for bridge mode and target text channel selection, route voice STT into the selected text-channel session, post speaker-attributed transcripts, deliver the assistant reply through normal Discord text delivery, and enqueue that same final reply for voice TTS playback. Coordinate with #53562/#53741 if the narrower `sessionChannelId` work becomes the base, but do not close this broader tracker until the full voice-as-I/O behavior is implemented or superseded by a maintainer-chosen canonical issue.
