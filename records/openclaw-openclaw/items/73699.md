## Summary

Keep open. Current main still routes Discord voice turns through the voice-channel session, suppresses normal text delivery, and exposes no bridge-to-text config/schema/docs; the maintainer discussion also explicitly left this as dedicated Discord voice feature work rather than cleanup to close.

## What This Changes

The issue requests a Discord voice-as-I/O mode that posts voice transcripts into a bound text channel, runs the existing text-channel agent session, posts the reply to text, and also speaks it in voice via TTS.

## Best Possible Solution

Implement this as a bundled Discord plugin feature with typed bridge-mode config/schema/docs, explicit target-channel resolution and permission checks, speaker-attributed transcript posting, text-channel session routing, normal Discord text delivery, and TTS playback of the same final reply. Keep realtime audio provider work separate from the bridge unless a later design intentionally combines them.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice join resolves a route from the voice channel id, voice segment processing calls the agent with `deliver: false`, and the strict config schema has no bridge fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin, but the proposed config/defaults are not yet proven as the narrowest maintainable design; a simple `sessionChannelId` field alone would be an incomplete solution for the full voice-as-I/O behavior.
