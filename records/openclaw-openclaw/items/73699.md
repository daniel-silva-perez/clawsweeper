## Summary

Keep open. Current main still routes Discord voice through the joined voice-channel session, suppresses Discord text delivery for voice turns, and exposes no bridge-to-text config/schema/docs fields; the maintainer discussion also explicitly left this as dedicated feature work rather than cleanup that can close.

## What This Changes

Request to let Discord voice channels act as microphone-and-speaker I/O for a bound text-channel agent session instead of an isolated voice-channel session.

## Best Possible Solution

Implement this as a bundled Discord plugin feature with typed config/schema/docs, explicit bridge target selection, speaker-attributed transcript posting, routing through the selected text-channel session, normal Discord text delivery, and TTS fan-out of the same final reply; keep realtime audio provider support as a separate follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice join resolves the route from the voice channel id, voice turns call agentCommandFromIngress() with deliver:false, and strict config/schema/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin, but the exact config shape, fallback behavior, permission model, transcript visibility, and reply fan-out semantics need maintainer design before a narrow automated fix is safe.
