## Summary

Keep open. Current main still does not implement the requested Discord voice-as-I/O bridge: voice joins resolve against the joined voice channel, voice agent turns still suppress outbound Discord text delivery, and the typed/schema/docs surfaces still lack bridge mode, transcript prefix, or bridge target configuration. The related session-channel proposal remains narrower than this issue, and a maintainer comment explicitly left this feature open for dedicated design work.

## Best Possible Solution

Keep this open for a dedicated Discord plugin feature. The likely path is to add typed/schema/docs config for bridge mode and target text channel, resolve voice input against the selected text-channel route/session, post speaker-attributed transcripts, deliver replies through normal Discord text delivery, and enqueue the same final reply for voice TTS playback. Coordinate with #53562/#53741 if that narrower sessionChannelId work is revived or landed first.
