## Summary

Keep open. Current main still does not implement the requested Discord voice-as-I/O bridge, and the maintainer discussion explicitly says this needs a dedicated feature change. Voice auto-join still joins only a voice channel, routing still uses the voice channel id as the session peer, voice agent turns still call `agentCommandFromIngress` with `deliver: false`, and the typed config/schema/docs do not expose bridge-to-text fields. Related #53562/#53741 are narrower `sessionChannelId` routing work and remain open, so they are possible implementation inputs rather than a reason to close this broader tracker.

## Best Possible Solution

Keep this open for a dedicated bundled Discord plugin feature. The likely implementation should add typed config/schema/docs for bridge mode and text-channel target selection, route voice STT into the selected text-channel session, post speaker-attributed transcripts, deliver the assistant reply through normal Discord text delivery, and enqueue the same final reply for voice TTS playback. Coordinate with #53562/#53741 if that narrower `sessionChannelId` work becomes the base, but do not close this broader tracker until the full voice-as-I/O behavior is implemented or a maintainer chooses a canonical replacement.
