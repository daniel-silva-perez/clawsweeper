## Summary

Keep open. Current main still does not implement the requested Discord voice-as-I/O bridge to a bound text-channel session, and the maintainer discussion explicitly says this should remain open for a dedicated feature change.

## Best Possible Solution

Keep this open unless maintainers choose a canonical successor. The likely fix belongs in the bundled Discord plugin voice path: add typed/schema/UI/docs config for bridge mode, target text channel, and transcript formatting; route voice STT into the selected text-channel session; post speaker-attributed transcripts; deliver the assistant reply through normal Discord text delivery; and enqueue the same final reply for TTS playback in the voice channel. Coordinate with #53562/#53741 if that narrower sessionChannelId work becomes the base.
