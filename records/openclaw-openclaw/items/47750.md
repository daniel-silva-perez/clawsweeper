## Summary

Keep this issue open. Current main supports interactive OpenAI Codex browser OAuth, manual redirect paste during that prompt flow, and a device-code pairing method, but it still does not provide the requested non-interactive callback URL or environment-driven Codex OAuth setup path.

## What This Changes

Request a non-interactive OpenAI Codex OAuth setup path for model provider auth, such as accepting a callback URL from an environment variable.

## Best Possible Solution

Add a documented, secure Codex OAuth automation path that can complete a PKCE callback without an interactive prompt, preferably through a generic provider-auth input such as a callback URL/code option or env-var reader with explicit redaction and validation, then cover it in OpenAI provider tests, CLI auth tests, and the OpenAI/automation docs.

## Reproduction Assessment

Yes. A source-level reproduction is enough: `models auth login` is TTY-gated and the OpenAI Codex provider has only interactive OAuth/device-code methods, so current main cannot consume a callback URL from env non-interactively.

## Solution Assessment

Unclear. The requested env-var idea solves the user workflow, but the best implementation needs maintainer/security design around PKCE state, one-time code handling, redaction, and whether the surface should be provider-specific or a generic provider-auth callback input.

## Security Review

Status: needs_attention

Summary: The issue requests a new OAuth callback ingestion path, so any implementation must treat callback URLs and authorization codes as sensitive credential material.
