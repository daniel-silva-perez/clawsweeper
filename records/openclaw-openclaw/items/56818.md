## Summary

Keep open. Current main still appears to propagate the temporary top-level Feishu card callback ID into the synthetic message reply target when both top-level and context message IDs are present, so streaming card startup can call Feishu `message.reply` with `card-action-c-*`. Existing coverage checks context-only and flat callback IDs, but not the conflicting callback-versus-original-message shape described in the report.

## What This Changes

The issue asks Feishu card-action callbacks to avoid using temporary callback `open_message_id` values as streaming reply targets and instead use the original message ID or send a new message.

## Best Possible Solution

Fix this inside the Feishu plugin boundary by separating callback event IDs from reply target message IDs: prefer a real `context.open_message_id` when present, ignore temporary `card-action-c-*` IDs for reply targeting, and suppress reply targeting when no real message anchor exists so replies use `receive_id` create mode. Add a regression test for the conflicting top-level/context ID payload and keep the synthetic `card-action-${token}` message ID for dedupe/session identity.

## Reproduction Assessment

Yes. A unit/lifecycle reproduction can feed `card.action.trigger` a payload with top-level `open_message_id: "card-action-c-xxx"` and `context.open_message_id: "om_real"`; current main statically resolves the top-level value into `replyToMessageId`, which is the failing path described by the reporter.

## Solution Assessment

Yes, keeping this open and fixing the Feishu parser/reply-target normalization is the narrow maintainable path. A broader core migration or disabling Feishu streaming would be less precise and would risk unrelated channel behavior.
