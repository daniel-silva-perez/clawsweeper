## Summary

Keep open. Current main still lets a temporary top-level Feishu card callback `open_message_id` win over the real `context.open_message_id`, so the downstream streaming card path can still call Feishu `message.reply` with `card-action-c-*`. The related #71673/#38609 work covers synthetic-only and context-only cases, but not the conflicting top-level-versus-context payload described here.

## What This Changes

The issue asks Feishu card-action callbacks to avoid using temporary callback `open_message_id` values as streaming reply targets and instead use the original message ID or send a new message.

## Best Possible Solution

Fix this inside the Feishu plugin boundary by separating callback event IDs from reply target message IDs: prefer a real `context.open_message_id` when present, ignore temporary `card-action-c-*` IDs for reply targeting, and suppress reply targeting when no real message anchor exists so replies use receive-id create mode. Add regression coverage for the conflicting payload shape while preserving synthetic `card-action-${token}` IDs for dedupe/session identity.

## Reproduction Assessment

Yes. A unit/lifecycle reproduction can invoke `card.action.trigger` with top-level `open_message_id: "card-action-c-xxx"` and `context.open_message_id: "om_real"`; current main copies the top-level value into the reply-target path.

## Solution Assessment

Yes. Keeping this open for a narrow Feishu parser/reply-target normalization fix is the best path; disabling streaming or moving Feishu-specific policy into core would be broader and less maintainable.

## Repair Work Prompt

Fix Feishu card-action callbacks that carry a temporary top-level `open_message_id` such as `card-action-c-xxx` while the original message ID is in `context.open_message_id`. Keep the change inside the Feishu plugin. Preserve the synthetic `message_id: card-action-${token}` used for dedupe/session identity, but do not use temporary callback IDs as `reply_target_message_id` or `replyToMessageId`. Prefer a real `context.open_message_id` when present; when only a temporary/non-message callback ID exists, suppress reply targeting so Feishu sends via receive-id create mode. Preserve flat SDK-style `open_message_id` behavior when it is a real message ID. Add regression coverage for top-level `open_message_id: "card-action-c-xxx"` plus `context.open_message_id: "om_real"`, expecting `replyToMessageId: "om_real"`, and for temporary-only callback IDs expecting no reply target.
