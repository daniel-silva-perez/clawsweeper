## Summary

Keep this PR open. Current main still does not hydrate attachment payloads for plugin-handled `send` actions, and the PR head remains a plausible implementation candidate because it adds a guarded `preloadSendMedia`/`hydrateSendMedia` path rather than unconditional media preloading. I did not find an implemented-on-main fix or a merged superseding PR.

## Best Possible Solution

Keep this PR open for maintainer review or contributor refinement. The best path is a guarded public plugin action contract for send-media preloading, wired from `runMessageAction` into `hydrateAttachmentParamsForAction`, with tests proving both default URL/path forwarding and opt-in hydrated `buffer`/`contentType`/`filename`. Coordinate with #41779, #42674, #53641, and #57335, but do not close this PR as implemented or duplicate while current main lacks the plugin send media hydration path.

- This is a public plugin channel contract change; if accepted, it should include SDK/docs coverage for the new opt-in semantics and API/contract checks as appropriate.
