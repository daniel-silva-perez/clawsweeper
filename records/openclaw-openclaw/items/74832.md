## Summary

Keep open. Current main and the latest release contain the intended Telegram config-only accessor behavior with regression coverage, but provided GitHub context shows open PR #74833 by the same author still references this issue with closing syntax, so conservative cleanup should leave the issue open until that PR is reviewed, landed, or explicitly superseded.

## What This Changes

The issue requests that Telegram status and metadata accessors avoid strict-resolving file-backed `botToken` SecretRefs while preserving strict runtime token resolution.

## Best Possible Solution

Resolve the paired PR path first: review and land #74833 if it remains the desired implementation, or close it as superseded by the shipped v2026.4.29 behavior, then close this issue as implemented while keeping runtime Telegram token resolution strict.

## Reproduction Assessment

Yes. The issue gives a concrete Telegram config shape, and current `extensions/telegram/src/shared.test.ts` covers the file-backed `botToken` SecretRef while reading `allowFrom` and `defaultTo` through the config adapter.

## Solution Assessment

Yes. A config-only `resolveAccessorAccount` is the narrow maintainable fix because read-only status metadata no longer dereferences `botToken`, while `resolveTelegramAccount` and `resolveTelegramToken` keep runtime token use strict.
