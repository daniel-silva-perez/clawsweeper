## Summary

Keep open. The Docker proxy documentation idea is useful and is not already present on current main, but the PR still documents `NO_PROXY` CIDR ranges that OpenClaw and Undici do not match, and it adds stale `extra_hosts` guidance even though current main already has an active host-gateway mapping.

## What This Changes

The PR adds commented `HTTP_PROXY`/`HTTPS_PROXY`/`NO_PROXY` examples to `docker-compose.yml` and a Docker install docs section for local outbound proxy configuration.

## Best Possible Solution

Keep the docs-only direction, but update the examples to document only `NO_PROXY` host/domain entries that OpenClaw and Undici actually honor, remove stale duplicate `extra_hosts` guidance from the bundled Compose file, and link or phrase the section so users can distinguish ambient env proxy use from the managed `proxy.enabled` network-proxy feature.

Yes. The review issue is reproducible by comparing the PR's `NO_PROXY` sample against `src/infra/net/proxy-env.ts` and Undici v8.1.0 `EnvHttpProxyAgent`; CIDR entries are parsed as literal host strings, not ranges.

## Solution Assessment

No. The PR is a narrow and useful docs change, but the current sample is not the best solution until it stops promising CIDR bypass behavior that the runtime dependency does not implement.
