## Summary

Current main still omits `standardFontDataUrl` in the active bundled PDF extraction fallback renderer, the render path remains reachable, and tests do not assert PDF.js document options. The issue should stay open because open PR #70936 has explicit closing linkage and is the active narrow implementation candidate.

## Best Possible Solution

Keep this issue open while PR #70936 is reviewed. The best fix is a narrow `document-extract` plugin change that resolves the installed `pdfjs-dist/standard_fonts` directory from the package root, passes a slash-terminated filesystem path as `standardFontDataUrl` to `getDocument`, widens the local PDF.js type shim if needed, and adds regression coverage in `extensions/document-extract/document-extractor.test.ts`. Close this issue only after that PR or an equivalent fix merges.
