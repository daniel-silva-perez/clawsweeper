## Summary

Keep open. Current main still omits `standardFontDataUrl` in the bundled `document-extract` PDF.js call, the fallback page-render path remains active, the latest release has the same omission, and open PR #70936 explicitly tracks this issue with closing syntax while #74355 is another active implementation candidate.

## What This Changes

Configure the PDF extraction fallback renderer to pass PDF.js a standard-font data path in Node so fallback rendering avoids `standardFontDataUrl` warnings and handles standard-font PDFs correctly.

## Best Possible Solution

Land one narrow plugin-owned fix that resolves the installed `pdfjs-dist/standard_fonts/` directory as a slash-terminated filesystem path, passes it through the local `PdfJsModule.getDocument` type and active call site, and adds focused regression coverage that would fail without the option.

## Reproduction Assessment

Yes. The issue body and linked PR context provide a concrete fallback-render reproduction with standard-font PDFs, and current main statically preserves the missing option and reachable render path that make the warning plausible.

## Solution Assessment

Yes, the requested direction is the right boundary: fix `extensions/document-extract` rather than core. The safest next step is maintainer review/consolidation of #70936 and #74355, then landing one narrow plugin-owned implementation with real regression coverage.
