## Summary

Current main still omits `standardFontDataUrl` in the active `document-extract` PDF fallback renderer, the image-render fallback path remains reachable, and open PR #70936 is the explicit unmerged implementation candidate for this issue. The issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Configure the PDF extraction fallback renderer to pass PDF.js a slash-terminated standard font asset path so Node fallback rendering no longer emits `standardFontDataUrl` warnings.

## Best Possible Solution

Keep this issue open while PR #70936 is reviewed. The best fix is the narrow plugin-owned change already proposed there: resolve the installed `pdfjs-dist` package root, pass a slash-terminated filesystem `standardFontDataUrl` to `getDocument`, update the local PDF.js type surface if needed, and keep focused regression coverage in `extensions/document-extract/document-extractor.test.ts`. Close this issue only after that PR or an equivalent fix merges.
