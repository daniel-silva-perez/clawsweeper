## Summary

Keep this PR open. Current main still lets IRC inherit the global block-streaming default when `channels.irc.blockStreaming` is unset, while the streaming docs say non-Telegram channels should only block-stream after an explicit channel opt-in; the PR is a narrow IRC-owned fix with regression coverage and no definite defect found in the diff.

## What This Changes

The PR changes IRC inbound reply options so unset or false IRC `blockStreaming` disables block streaming, explicit true opts back in, and adds policy tests for all three config states.

## Best Possible Solution

Merge or port the IRC-owned default so final IRC replies are preserved unless IRC block streaming is explicitly opted in, keep regression coverage for unset, false, and true account config values, then validate the IRC test surface and changed gate before landing.

## Reproduction Assessment

Yes. On current main, set `agents.defaults.blockStreamingDefault` to `on` with `channels.irc.blockStreaming` unset: IRC passes `disableBlockStreaming: undefined`, and the shared resolver enables block streaming from the global default.

## Solution Assessment

Yes. The PR keeps the fix at the IRC ownership boundary, preserves explicit per-account opt-in with `blockStreaming: true`, and avoids changing the shared block-streaming resolver for other channels.
