## Summary

Keep open. Current main still lets IRC inherit the global block-streaming default when the IRC account setting is unset, and the latest release contains the same behavior; the PR is a narrow IRC-owned fix that matches the docs and needs maintainer review plus validation, not cleanup closure.

## What This Changes

The PR changes IRC inbound reply options so unset or false IRC `blockStreaming` disables block streaming while explicit true opts in, and adds policy tests for those three config states.

## Best Possible Solution

Land or port the IRC-owned default so final IRC replies are preserved unless IRC block streaming is explicitly opted in, keep regression coverage for unset, false, and true account config values, then validate the IRC test surface and changed gate before merging.

## Reproduction Assessment

Yes. With `agents.defaults.blockStreamingDefault` set to `on` and IRC `blockStreaming` unset, current main passes `disableBlockStreaming: undefined`, and the shared resolver enables block streaming from the global default.

## Solution Assessment

Yes. The PR keeps the behavioral change inside the IRC plugin boundary, preserves explicit per-account opt-in with `blockStreaming: true`, and avoids changing the shared resolver or other channels.
