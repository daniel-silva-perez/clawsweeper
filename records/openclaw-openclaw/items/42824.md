## Summary

Keep open. Current main and the latest release still let IRC inherit the global block-streaming default when IRC account config leaves `blockStreaming` unset, while this PR is a focused IRC-owned fix with matching regression coverage and no blocking review finding.

## What This Changes

The PR changes IRC inbound reply options so unset or false IRC `blockStreaming` disables block streaming, explicit true opts back in, and adds policy tests for all three config states.

## Best Possible Solution

Land or port the IRC-owned default so final IRC replies are preserved unless IRC block streaming is explicitly opted in, keep regression coverage for unset, false, and true account config values, then validate the IRC test surface and changed gate before landing.

## Reproduction Assessment

Yes. On current main, set `agents.defaults.blockStreamingDefault` to `on` with `channels.irc.blockStreaming` unset: IRC passes `disableBlockStreaming: undefined`, and the shared resolver enables block streaming from the global default.

## Solution Assessment

Yes. The PR keeps the fix at the IRC ownership boundary, preserves explicit per-account opt-in with `blockStreaming: true`, and avoids changing the shared resolver for other channels; a direct port is the safer path if the branch no longer rebases cleanly.
