## Summary

Keep open for maintainer review. Current main still lacks the reconnect-grace behavior: running disconnected channels are marked unhealthy immediately after startup connect grace, and neither the health monitor nor readiness checker projects typed `lastDisconnect.at` into shared health policy. The PR is a scoped implementation candidate based on #45712; I found no discrete blocking correctness or security finding in the diff, but the reconnect-grace default/readiness semantics and changed-gate proof still need maintainer signoff.

## What This Changes

This PR adds shared reconnect-grace handling based on typed `lastDisconnect.at`, wires it through the gateway health monitor and readiness checker, adds focused regression tests, and records a changelog fix crediting #45712.

## Best Possible Solution

Land this or an equivalent narrow shared gateway health-policy change after maintainer review confirms the default grace window and readiness behavior, then require targeted policy/monitor/readiness tests plus the changed gate before merge.

## Reproduction Assessment

Yes. A unit-level reproduction is clear on current main: evaluate or monitor a managed account with `running: true`, `connected: false`, `lastStartAt` older than startup grace, and recent typed `lastDisconnect.at`; current main returns/restarts as `disconnected`, while the PR expects `reconnect-grace`.

## Solution Assessment

Mostly yes. The PR uses the right shared boundary by adding one policy helper and wiring both monitor and readiness through typed snapshot projection; the remaining uncertainty is product-level default duration/readiness semantics, not a clearly better code shape.
