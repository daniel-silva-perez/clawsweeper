## Summary

Keep open. Current main has fixed adjacent Telegram media-loss paths, but the reported local/generated Markdown-image path is still not implemented: ordinary reply construction and Telegram delivery still require mediaUrls to already exist, while the shared Markdown-image parser only promotes validated remote HTTPS images and tests still assert local Markdown-image targets remain text.

## What This Changes

Request to make ordinary Telegram assistant replies deliver image attachments when a text reply includes a local/generated Markdown image instead of delivering only the text.

## Best Possible Solution

Add a narrow reply-pipeline implementation that promotes only safe local/generated Markdown-image destinations into media payloads for Telegram, routes them through the existing reply media normalizer/access boundary, wires the extraction option through ordinary Telegram reply paths, and covers parser, final-payload, block-stream, and Telegram delivery regressions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call the current parser/buildReplyPayloads path with extractMarkdownImages enabled and a local Markdown-image target; existing tests assert mediaUrl/mediaUrls stay undefined, which prevents Telegram media delivery before the send layer is reached.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix is not to broadly lift all local Markdown-image paths in parse.ts; maintainers should first define the safe generated/local media boundary and then reuse the existing reply media normalization path.
