## Summary

Keep this open. Current main has the Telegram outbound hooks needed for markdown-image parsing and can deliver media once a media URL exists, but the shared markdown-image parser still only lifts remote HTTP(S) image destinations and the current tests explicitly reject local markdown image targets, which is central to the reported ordinary-assistant image loss.

## What This Changes

Fix Telegram ordinary assistant replies so text-plus-image replies, including local/generated markdown image references, preserve image attachments through reply parsing, block/final delivery, and Telegram delivery.

## Best Possible Solution

Implement a channel-scoped, agent-root-gated fix that promotes safe local/generated markdown image destinations to media payloads for Telegram, while adding regression coverage for ordinary assistant final text plus media after block streaming/lane splitting. Keep the separate read-tool media bridge tracked by #48979 unless the same validated media-access helper can cover both paths cleanly.

## Reproduction Assessment

Yes. The parser-level reproduction is high-confidence: with extractMarkdownImages enabled, current tests still assert that local markdown image targets produce no mediaUrls; a Telegram harness regression should then verify that a normal assistant final payload with text plus a generated image sends media once.

## Solution Assessment

Unclear until maintainers approve the local-path trust boundary. The safest direction is not to lift all local markdown image paths blindly, but to reuse existing agent-scoped media access/local-root checks and prove Telegram text-plus-media delivery does not double-send or drop attachments.
