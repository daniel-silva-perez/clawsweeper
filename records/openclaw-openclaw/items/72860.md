## Summary

Keep open. Current main has fixed adjacent Telegram media-loss paths, but the remaining reported local/generated Markdown-image path is still not implemented: Telegram opts into Markdown-image extraction, while the shared parser only promotes validated remote HTTP(S) Markdown images and current tests explicitly keep local Markdown image targets as text without media URLs.

## What This Changes

The issue asks OpenClaw to preserve image attachments in ordinary Telegram assistant replies, including local/generated Markdown image content, across reply parsing, block-stream final suppression, and Telegram delivery.

## Best Possible Solution

Add a narrow reply-pipeline implementation that promotes only safe local/generated Markdown image destinations into media payloads for Telegram, runs them through the existing reply media normalizer/access boundary, and preserves the current block-stream and lane-delivery safeguards with focused regression tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with extractMarkdownImages enabled, current parser tests still assert local Markdown image targets remain text with mediaUrls undefined, matching the reported text-only outcome before Telegram media delivery is reached.

## Solution Assessment

Unclear. The high-level direction is right, but the best fix should not broadly lift every local Markdown path in parse.ts; it needs maintainer agreement on the safe local/generated path boundary and should reuse reply media normalization.
