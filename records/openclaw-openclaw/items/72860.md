## Summary

Keep open. Current main has fixed adjacent Telegram media-loss paths, but the central local/generated Markdown-image path is still not implemented: Telegram opts into Markdown-image extraction, while the shared parser only promotes validated remote HTTP(S) Markdown images and tests still assert local Markdown image targets remain plain text without media URLs.

## What This Changes

Request to make Telegram normal assistant replies deliver image attachments when a text reply includes a local/generated Markdown image, instead of delivering only the text.

## Best Possible Solution

Add a narrow reply-pipeline change that promotes only safe local/generated Markdown image destinations into media payloads for Telegram, then runs them through the existing reply media normalizer/access boundary and keeps the current block-stream and lane-delivery safeguards covered by focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with extractMarkdownImages enabled, current parser and final-payload tests still assert local Markdown image targets remain text with mediaUrl/mediaUrls undefined before Telegram media delivery can occur.

## Solution Assessment

Unclear. The direction is right, but the best fix should not broadly lift every local Markdown path in the parser; it needs maintainer agreement on the safe generated/local path boundary and should reuse existing reply media normalization.
