## Summary

Keep open. Current main covers several adjacent Telegram media delivery failure modes, but it still does not implement the central local/generated Markdown-image promotion described here: Telegram opts into Markdown image extraction, while the shared parser only promotes remote HTTP(S) Markdown image destinations and tests still pin local Markdown image targets as plain text with no media URLs.

## What This Changes

The issue requests that ordinary Telegram assistant replies preserve image attachments, especially local/generated Markdown image content, across parsing, block-stream final suppression, and Telegram lane delivery.

## Best Possible Solution

Add a narrow reply-pipeline fix that promotes only safe local/generated Markdown image destinations into media payloads for Telegram, routes those media sources through the existing agent-scoped media normalizer, preserves the current block-stream and lane-delivery safeguards, and adds regression coverage for local Markdown images plus final text/media delivery.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with extractMarkdownImages enabled, current parser tests still assert local Markdown image targets remain text with mediaUrls undefined, matching the reported text-only outcome before Telegram media delivery is reached.

## Solution Assessment

Unclear. The suggested direction is correct at a high level, but the maintainable fix should not broadly lift every local Markdown path in parse.ts; it should reuse the reply media access boundary and prove Telegram does not double-send or drop media after streaming.
