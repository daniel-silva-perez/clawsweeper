## Summary

Keep open. Current main has several adjacent Telegram media-delivery protections, but it still does not implement the central local/generated markdown-image promotion described here: markdown image extraction is Telegram-enabled, yet the shared parser only lifts HTTPS markdown image destinations and current tests explicitly assert that local markdown image targets remain plain text with no mediaUrls.

## What This Changes

Telegram ordinary assistant replies with text plus a local or generated markdown image should preserve and deliver the image media instead of delivering only the text.

## Best Possible Solution

Add a narrow, channel/session-scoped implementation that promotes safe local/generated markdown image destinations into media payloads for Telegram normal replies, then runs them through the existing agent-scoped media normalizer and adds regression coverage for text plus media after block streaming and Telegram lane delivery.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with extractMarkdownImages enabled, current parser tests still assert local markdown image targets produce no mediaUrls, matching the reported text-only Telegram outcome before media delivery is reached.

## Solution Assessment

Unclear. The suggested direction is correct at a high level, but the maintainable fix should not blindly lift every local markdown path in parse.ts; it should reuse the existing reply media access boundary and prove Telegram text-plus-media delivery does not double-send or drop attachments.
