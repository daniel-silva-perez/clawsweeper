## Summary

Keep open. Current main has session-pruning docs, cache-TTL pruning knobs, and a Control UI context-pressure notice, but it still lacks the central requested pruning preview/history, pruning lifecycle metadata/events, user-facing pruning notifications, and keyword/tag retention strategy.

## What This Changes

Request to improve context pruning/trimming visibility and control with previews/logs, UI usage indicators, pruning notifications, and configurable retention behavior.

## Best Possible Solution

Keep this as a design umbrella unless maintainers split it: add safe pruning metadata to the context-pruning path, expose a pruning lifecycle event/hook, surface concise Control UI/chat notices, document existing and new controls, and leave overflow-message wording to #9409.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The source check gives high-confidence behavioral evidence that pruning can run silently and currently returns only transformed messages, while UI/docs expose context pressure rather than pruning history or notifications.

## Solution Assessment

Unclear until maintainers approve the product boundary. The narrowest maintainable direction is metadata/events first, with UI/docs layered on top and no raw pruned-output disclosure by default.
