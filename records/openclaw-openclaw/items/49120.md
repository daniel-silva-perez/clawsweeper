## Summary

Keep this PR open. Current main still accepts private/internal cron webhook URLs during validation and persistence because `normalizeHttpWebhookUrl` only trims, parses, and checks for http(s). Runtime dispatch remains protected by `fetchWithSsrFGuard`, but that is later than this PR's stated pre-persistence defense-in-depth goal.

## Best Possible Solution

Review and land this focused security hardening, or reimplement the same small change on main: call `isBlockedHostnameOrIp(parsed.hostname)` inside `src/cron/webhook-url.ts`, add the dedicated `src/cron/webhook-url.test.ts` coverage from this PR, and keep the existing runtime `fetchWithSsrFGuard` protection as the DNS-aware dispatch-time guard.
