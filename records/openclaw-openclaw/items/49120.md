## Summary

Keep this PR open. Current main still accepts private/internal cron webhook URLs at create/patch time and relies on the later runtime fetch guard to block them, so the PR's pre-persistence SSRF hardening remains unimplemented and useful.

## Best Possible Solution

Review and land this focused PR, or reimplement the same change on main: call `isBlockedHostnameOrIp(parsed.hostname)` from `normalizeHttpWebhookUrl`, keep `fetchWithSsrFGuard` as the DNS-aware runtime guard, and update/create tests so private, loopback, link-local, metadata, `.local`/`.internal`, `0.0.0.0`, and IPv4-mapped IPv6 webhook targets are rejected before cron jobs are stored.

## Risks / Open Questions

- Current main still allows private/internal cron webhook URLs to be persisted, although runtime dispatch is guarded by `fetchWithSsrFGuard`.
- Landing this PR likely requires updating gateway cron test expectations that currently assert private webhook URLs are accepted and blocked only during runtime fetch.
- Maintainers may want an explicit product/security decision on whether private-network cron webhooks should ever be allowed by configuration; the PR chooses stricter default rejection.
