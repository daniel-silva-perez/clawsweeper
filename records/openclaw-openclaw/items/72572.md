## Summary

Keep open. Current `main` still rejects a symlinked `~/.openclaw` below the resolved home root, and this issue is already paired with two open implementation PRs, so it should remain open until a reviewed fix lands.

## What This Changes

The issue asks for exec approvals and `openclaw exec-policy` writes to accept a trusted first-level `~/.openclaw` symlink while continuing to reject unsafe deeper symlink traversal.

## Best Possible Solution

Land a reviewed fix that permits exactly the trusted first-level `.openclaw` symlink shape under the user home on supported platforms, validates the symlink target safely, keeps the final approvals file and deeper path components protected, and adds focused regression coverage plus docs if the supported shape changes.

## Reproduction Assessment

Yes. The source path is high-confidence: a symlinked `.openclaw` below the resolved home root is encountered by `assertNoSymlinkPathComponents()` and rejected before `saveExecApprovals()` can write the approvals file.

## Solution Assessment

Unclear between the two open PRs, but the requested direction is sound if implemented narrowly. The best fix should be the minimal safe relaxation at the home boundary, with owner/mode validation and continued rejection of deeper symlinks.
