## Summary

Keep open. Current `main` still rejects the reported symlinked `~/.openclaw` shape, the focused store test still expects that rejection below a symlinked trusted home, and two open PRs already reference this issue with closing syntax, so the issue should wait for review and merge rather than be closed or sent to another repair lane.

## What This Changes

The issue requests that `openclaw exec-policy preset yolo` accept a trusted first-level symlinked `~/.openclaw` directory under the normal home directory while preserving deeper exec-approvals symlink hardening.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, user-owned, non-group/other-writable first-level `.openclaw` symlink and continues rejecting chained or deeper symlinks plus symlinked approval-file destinations, with focused store tests and docs if the supported layout changes.

## Reproduction Assessment

Yes. The high-confidence path is to set normal home to a directory containing `.openclaw` as a symlink and call `openclaw exec-policy preset yolo`; current source reaches `saveExecApprovals()`, then `ensureDir()`, which rejects the symlinked `.openclaw` component.

## Solution Assessment

Yes in direction, but not yet implemented on `main`. The narrow, owner/mode-checked first-level `.openclaw` exception is the maintainable fix shape; broader macOS/state-dir changes should be included only if review shows they are necessary and equivalently hardened.
