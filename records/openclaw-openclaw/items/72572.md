## Summary

Keep open. Current main still rejects the reported symlinked `~/.openclaw` layout, and the issue is already paired with open closing PRs #72650 and #72713, so it should remain open until one reviewed implementation lands.

## What This Changes

The issue requests that `openclaw exec-policy preset yolo` support a trusted first-level `~/.openclaw` symlink while still rejecting unsafe deeper symlinks in the exec approvals path.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, user-owned, non-group/other-writable first-level `.openclaw` symlink and continues rejecting chained or deeper symlinks plus symlinked `exec-approvals.json` destinations, with focused store tests and docs if the supported layout changes.

## Reproduction Assessment

Yes. A source-level reproduction is clear: put `.openclaw` as a symlink below the trusted home root and run `openclaw exec-policy preset yolo`; the command calls `saveExecApprovals()`, which reaches `ensureDir()` and rejects the symlinked `.openclaw` component before writing.

## Solution Assessment

Yes in direction, but not yet on main. The maintainable fix is a narrow, owner/mode-checked exception for the first-level `.openclaw` directory while preserving strict rejection for deeper symlink traversal and symlinked approval-file destinations.
