## Summary

Keep open. Current main still rejects the reported symlink shape, and two open PRs already reference this as closing work, so the issue should remain open until a reviewed fix lands or maintainers choose a different security boundary.

## What This Changes

The issue asks OpenClaw to allow a trusted first-level `~/.openclaw` symlink for local exec approvals while continuing to reject unsafe deeper symlink traversal.

## Best Possible Solution

Land one reviewed, narrow fix that permits only a trusted user-owned first-level `~/.openclaw` symlink, resolves it safely, keeps `exec-approvals.json` destination protection, and preserves rejection of deeper or chained symlinks across the relevant Node and macOS approval paths.

## Reproduction Assessment

Yes. By source inspection, a symlinked `.openclaw` below the trusted home root is the first segment walked by `assertNoSymlinkPathComponents()`, and the current test suite explicitly expects that shape to throw.

## Solution Assessment

Yes, the issue’s proposed direction is the right maintainable shape if paired with ownership, mode-bit, and deeper-symlink checks. The immediate next action should be review and consolidation of #72650 or #72713, not closing this before merge.
