## Summary

Keep open. The branch is useful but not merge-ready: current main does not already have this Bedrock config surface, and the PR introduces a manifest/type mismatch plus security-sensitive credential fields without sensitive hints; the docs also advertise auth modes before the current runtime/dependency pin is proven to consume them.

## What This Changes

The PR adds Amazon Bedrock plugin manifest configSchema/uiHints for Cline-style auth options, a BedrockAuthConfig normalization helper with tests, setup-api type re-exports, Bedrock docs/changelog entries, and a private plugin version bump.

## Best Possible Solution

Land a coordinated Bedrock config/auth contract that matches the shipped pi-ai resolver: manifest schema, uiHints, helper types, docs, secret handling, and tests should contain the same fields, and user-facing auth-mode docs should land with or after the dependency/runtime support that actually consumes them.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: normalizeBedrockAuthConfig emits awsBedrockCustomSelected, while the PR manifest keeps additionalProperties:false and omits that property, and current plugin config validation uses AJV without removing additional properties.

## Solution Assessment

No, not yet. The PR is the right owner boundary for Bedrock plugin config metadata, but the current patch needs schema/type alignment, sensitive secret hints, and a shipped resolver/dependency contract before it is the narrow maintainable solution.
