## Summary

Keep open. Current main already has a partial Telegram transport fallback: direct requests can promote from normal DNS to sticky IPv4 and then to a hard-coded pinned Telegram API IP, and proxy support is already documented. The remaining request is still meaningful because there is no `fallbackIps`/`disableFallback` config surface or DoH discovery, and timeout-triggered paths such as `sendChatAction` are not covered by the same forced fallback behavior.

## What This Changes

Request that Telegram Bot API transport automatically or manually fall back to known-good `api.telegram.org` IPs in restricted networks while preserving TLS/SNI and proxy fallback behavior.

## Best Possible Solution

Keep this behavior in the Telegram plugin and define a safe public policy for fallback IPs: document proxy precedence, preserve TLS/SNI and existing SSRF boundaries, decide whether fallback IPs are configurable or auto-discovered, and add focused tests for timeout-triggered fallback paths before changing runtime behavior.

## Reproduction Assessment

Unclear. The report includes concrete logs and environment, and the existing unit tests reproduce low-level `ETIMEDOUT`/`EHOSTUNREACH` fallback behavior, but there is not a high-confidence live reproduction for the exact China-mainland timeout and `sendChatAction` restart loop path.

## Solution Assessment

Unclear. The current hard-coded pinned-IP fallback is a useful partial solution, but the requested configurable/discoverable fallback list and timeout-path behavior need maintainer policy decisions before this can be called the best fix.
