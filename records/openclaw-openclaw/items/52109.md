## Summary

Keep this PR open. Current main still persists the computed next run directly for successful non-cron completions, so the overrun `every` schedule refire bug remains and the PR is not obsolete. The branch is not merge-ready as-is because its unresolved cron fallback conflicts with the newer current-main behavior that clears unresolved cron next-runs instead of synthesizing a 2s retry.

## What This Changes

This PR changes cron `applyJobResult` so successful `every` jobs get a 2s refire floor when the computed next run is not after `endedAt`, and adds regression coverage for overrun and exact-boundary every schedules.

## Best Possible Solution

Keep the bug fix in the cron scheduler, but rebase it onto current main and preserve the existing unresolved-next-run clearing contract. The maintainable end state is a small completion-path helper that floors successful `every` jobs only when a valid computed next run is at or before `endedAt`, leaves malformed every schedules unscheduled, keeps cron completion/error behavior routed through the current lower-bound resolver, and adds regression tests for overrun, exact-boundary, malformed `everyMs`, preserved manual schedules, and unresolved cron next-runs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an `every` job whose execution reaches or exceeds its interval: `computeJobNextRunAtMs` can return `endedAt`, and current main's successful non-cron branch persists that value directly; the PR body also includes live Docker reproduction details for the same symptom.

## Solution Assessment

No, not as currently proposed. The direction is right and narrow, but the branch needs to be adjusted to current main so it does not reintroduce the cron unresolved-next-run refire behavior that #66083/#66113 already fixed.
