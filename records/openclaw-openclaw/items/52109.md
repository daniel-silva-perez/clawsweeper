## Summary

Keep this PR open. Current main still does not implement the PR’s intended persisted refire-gap behavior for successful `every` schedules: `applyJobResult` lower-bounds only cron-expression schedules, while non-cron recurring jobs store `naturalNext` directly. The supplied PR context shows a focused scheduler/test change with prior review narrowing the fix, so the right outcome is maintainer review/rebase rather than cleanup closure.

## Best Possible Solution

Keep this PR open. The best path is to rebase or replace it with a narrow cron scheduler fix that conditionally guards `every` schedules when the computed next run would refire at or before the just-finished run, preserves valid future ticks and forced-run anchors according to the chosen product semantics, keeps malformed/unresolved schedules unscheduled, and adds focused regression tests for overrun, exact-boundary, malformed `everyMs`, and `preserveSchedule` cases.
