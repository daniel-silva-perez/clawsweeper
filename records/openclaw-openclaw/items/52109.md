## Summary

Keep open. Current main still does not implement this PR's intended `every`-schedule refire-gap behavior in `applyJobResult`: the completion path lower-bounds only cron-expression schedules, while non-cron recurring jobs keep `naturalNext` directly. The PR is still a focused scheduler bug-fix candidate, but it needs maintainer review/rebase for exact-boundary cadence, preserved-anchor behavior, and current unresolved-next-run handling before landing.

## Best Possible Solution

Keep this PR open. The best path is to rebase it onto current main and land or replace it with a narrow scheduler fix that clamps `every` schedules only when the computed next run is at or before the just-finished run, preserves valid future ticks and forced-run anchors, keeps malformed/unresolved schedules unscheduled according to current main, and adds focused regression coverage for overrun, exact-boundary, malformed `everyMs`, and `preserveSchedule` cases.
