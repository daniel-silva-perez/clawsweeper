## Summary

Keep open. Current main still intentionally persists generated gateway install tokens with token auth, while this PR changes that path to persist a generated token under no-auth mode. That is a gateway auth product/security policy change and the current diff would make the saved generated token ineffective rather than solving the local-friction problem safely.

## What This Changes

The PR changes gateway install-token persistence to save auto-generated tokens with `gateway.auth.mode: "none"` when no mode exists, and adds a unit test expecting that behavior.

## Best Possible Solution

Keep token auth as the generated-token default. If maintainers want a no-auth local mode, add a narrower policy that only applies after proving loopback-only exposure, refuses or warns for daemon/non-loopback/private-ingress paths, and updates gateway docs plus regression coverage across bind and Tailscale modes.

## Reproduction Assessment

Yes for the proposed behavior: the existing unit seam can call `resolveGatewayInstallToken()` with empty `gateway.auth`, `autoGenerateWhenMissing: true`, and `persistGeneratedToken: true` to inspect the persisted mode. I did not run tests because this review is read-only, but current source and tests show the invariant clearly.

## Solution Assessment

No. Persisting `mode: "none"` in the generated-token path is not the narrowest safe solution because this branch is reached when install policy has decided a token is needed; a safer design would keep token mode unless a separate exposure-aware policy explicitly selects no-auth.

## Review Findings

Full review comments:

- **[P1] Preserve auth when persisting generated tokens:** `src/commands/gateway-install-token.ts:62`
  - body: This branch still saves a generated token, but changing the fallback mode to `"none"` makes that token unused by later gateway auth. The install-token path is reached when the policy decided a token is needed, so this can leave users with a config that looks token-protected while the runtime follows the no-auth path; keep the fallback as `"token"` unless a separate exposure-aware policy has explicitly opted into no-auth.
