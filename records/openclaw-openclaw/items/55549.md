## Summary

Keep this PR open, but do not merge it as-is. The current bug is still present on main, and the PR fixes the exec-ID race for the original path, but the proposed handler loses the approval kind after queue pruning and can send plugin approvals to the exec resolver.

## What This Changes

The PR passes a render-time approval ID from Control UI approval buttons into the shared approval decision handler and updates the AppViewState handler type.

## Best Possible Solution

Keep the render-time capture approach, but carry the approval kind or concrete resolver method together with the ID so both exec and plugin approvals resolve through the correct backend method after the queue entry is pruned. Add focused UI regression coverage for clicked buttons after queue mutation before landing.

## Review Findings

Overall correctness: patch is incorrect

  - body: When the queue entry has already been pruned, `active` is undefined and this falls back to `exec.approval.resolve` even if the rendered modal was for a plugin approval. The gateway uses separate exec and plugin approval managers, so plugin IDs only resolve through `plugin.approval.resolve`; this leaves plugin approval clicks failing in the same race window. Capture/pass the kind or method alongside the rendered ID.
