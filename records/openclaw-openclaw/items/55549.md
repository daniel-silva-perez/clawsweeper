## Summary

Keep this PR open. Current main still has the reported Control UI stale-render approval race, and the PR’s render-time ID capture is the right direction, but the proposed handler re-derives approval kind from the mutable queue and can route stale plugin approvals to the exec resolver.

## What This Changes

The PR updates Control UI approval buttons and the shared view-state handler signature so approval clicks pass a render-time approval ID into `handleExecApprovalDecision`.

## Best Possible Solution

Keep the render-time capture approach, but capture stable routing metadata too, such as approval kind or the exact resolver method, and add focused stale-render button coverage for both exec and plugin approvals.

## Reproduction Assessment

Yes. The issue can be reproduced at the UI unit level by rendering an approval prompt, pruning `execApprovalQueue`, then invoking a rendered decision button; the same path exposes the PR’s remaining plugin-resolver bug.

## Solution Assessment

No. Passing the ID is the narrow right fix for exec approvals, but the handler must also preserve kind or resolver metadata so plugin approvals do not fall back to `exec.approval.resolve` after queue cleanup.
