## Summary

Keep open: current main still prints only the bare not-found error, and this PR is a focused implementation candidate. I did not find a blocking correctness or security issue, but maintainers should decide whether the suggestion inventory should cover the same plugin/channel schema paths already exposed by config schema metadata.

## What This Changes

The PR adds a config-path suggestion helper with unit tests and prints `Did you mean` hints from `openclaw config get` and `openclaw config unset` when a path is not found.

## Best Possible Solution

Land a config-path suggestion feature after deciding whether candidates should come from the shared schema/UI-hint inventory, or explicitly scope the feature and tests to core static config paths.

## Reproduction Assessment

Not applicable as a standalone bug report. The PR behavior has a clear before/after check: current main emits only `Config path not found`, while the PR would add hints for close static paths such as `gateway.prot`.

## Solution Assessment

Unclear. The get/unset integration point is narrow and maintainable, but using only `FIELD_HELP` is not the complete current config-path inventory because plugin and channel metadata are merged elsewhere.
