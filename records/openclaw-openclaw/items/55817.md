## Summary

Keep this PR open. Current main still persists a new sessionFile during rollover without creating the corresponding transcript header immediately; header creation only happens later on transcript append paths. The PR is aimed at a real remaining consistency gap, but its final patch still needs the non-clobber/atomic create review concern addressed before it should land.

## Best Possible Solution

Keep the PR open and revise it rather than close it. The best path is to create the rollover transcript header before publishing the new sessionFile in sessions.json, use an atomic exclusive create so existing transcript content is never overwritten, rethrow non-ENOENT filesystem failures, and keep the daily/stale rollover plus explicit reset regression tests, ideally with one non-clobber test for an already-created transcript.

## Risks / Open Questions

- The PR's final patch, as described in the review context, still appears to use stat followed by writeFile for header creation. That can truncate a transcript created by a concurrent worker unless changed to exclusive creation with flag: "wx" and EEXIST-as-success handling.
- No supply-chain or CI security regression was visible from the PR file list: it only touches session runtime code and tests, with no workflow, dependency, lockfile, install, release, or secret-handling changes.
