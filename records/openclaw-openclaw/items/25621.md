## Summary

Keep open. Current main still throws for OpenBSD and every non-darwin/linux/win32 platform in `resolveGatewayService()`, and both `openclaw status` and `openclaw gateway status` still resolve the service before their degraded status paths can render. The related PR #25690 is closed and unmerged, so it remains useful prior art rather than a canonical fix.

## Best Possible Solution

Land a narrow gateway service/status fix: either add a conservative unsupported-platform `GatewayService` adapter whose read-only methods return safe unknown/not-installed state while mutating methods reject clearly, or guard status-only call sites before resolving managed service adapters. Cover `openclaw status` and `openclaw gateway status` regression behavior on an unsupported `process.platform`, keep lifecycle commands failing explicitly, and update docs/changelog if user-visible status wording changes.

## Clownfish Work Prompt

Fix the unsupported-platform Gateway service status failure tracked by openclaw/openclaw#25621. Current `resolveGatewayService()` throws on platforms outside darwin/linux/win32, which prevents read-only `openclaw status` and `openclaw gateway status` diagnostics from rendering on OpenBSD and similar platforms.
