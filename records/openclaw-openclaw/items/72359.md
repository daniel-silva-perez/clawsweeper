## Summary

Keep open. Current main still implements Active Memory as a foreground `before_prompt_build` pass that runs `runEmbeddedPiAgent`; there is no `mode` config, manifest field, docs entry, or test path for an opt-in single-shot recall. The request is narrower than the broader Active Memory latency tracker and looks technically feasible inside the active-memory plugin using existing memory-search and simple-completion runtime surfaces.

## What This Changes

The issue requests an opt-in Active Memory single-shot mode that avoids the embedded agent loop and performs one memory search plus one LLM completion for lower-latency preflight summaries.

## Best Possible Solution

Add an opt-in `config.mode` with the existing embedded-agent behavior as the default, and implement a `single-shot` branch inside the Active Memory plugin that performs one memory-manager search, formats bounded hits into a deterministic prompt, calls the configured recall model once, parses `NONE` or a bounded summary, and updates manifest/docs/tests without changing the current multi-turn mode.

## Reproduction Assessment

Yes. The issue gives concrete config/log context via its predecessor, and current-main source statically reproduces the architecture: the prompt hook awaits `maybeResolveActiveRecall`, which calls `runEmbeddedPiAgent` with memory tools and has no single-shot branch.

## Solution Assessment

Yes for this narrow request. An additive opt-in mode in `extensions/active-memory` using existing `memory-host-search` and simple-completion helpers is the smallest maintainable path; it should not replace the broader open latency tracker or change default embedded-agent behavior.
