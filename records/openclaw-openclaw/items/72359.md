## Summary

Keep open. Current main still lacks the requested opt-in Active Memory single-shot path: config has no `mode` value, runtime normalization ignores such a setting, eligible prompt-build hooks still await recall, and recall still goes through the embedded Pi agent with memory tools. The existing hard-timeout, partial-summary, cache, and circuit-breaker work mitigates latency, but it does not implement the one-search plus one-completion architecture requested here.

## What This Changes

Add an opt-in Active Memory single-shot mode that performs one memory search and one direct LLM completion without running the embedded agent loop.

## Best Possible Solution

Implement a documented opt-in `plugins.entries.active-memory.config.mode` with the current embedded-agent behavior as the default and a `single-shot` branch for low-latency preflight recall. The new branch should use plugin-safe memory search and simple-completion APIs, return `NONE` or a bounded summary, and leave recall-heavy/contextual multi-turn behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure Active Memory for an eligible session and inspect the current prompt-build path, which always awaits `maybeResolveActiveRecall` and then `runEmbeddedPiAgent`; `mode: "single-shot"` is neither accepted by the manifest nor read by runtime config.

## Solution Assessment

Likely yes. For the narrow `preference-only` and preflight use case, an opt-in single-search plus direct-completion mode is the narrowest maintainable solution; replacing the existing multi-turn recall path globally would be riskier.
