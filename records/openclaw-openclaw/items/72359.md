## Summary

Keep open. Current main still has only the bounded blocking Active Memory sub-agent path: the config surface has no `mode` or `single-shot` option, eligible prompt-build hooks await recall, and recall still invokes `runEmbeddedPiAgent` with memory tools. The hard-timeout, partial-summary, cache, and circuit-breaker mitigations help related timeout reports, but they do not implement the requested one-search, one-completion low-latency architecture.

## What This Changes

Add an opt-in Active Memory single-shot mode that performs one memory search plus one direct LLM completion without running the embedded agent loop.

## Best Possible Solution

Implement a documented opt-in `plugins.entries.active-memory.config.mode` with the current embedded-agent behavior as the default and a `single-shot` branch for low-latency preflight use. The single-shot branch should use plugin-safe memory search and simple-completion helpers, return `NONE` or a bounded summary, and leave recall-heavy/contextual multi-turn behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure Active Memory for an eligible session and inspect the current prompt-build path, which always awaits `maybeResolveActiveRecall` and then `runEmbeddedPiAgent`; adding `mode: "single-shot"` is not accepted by the schema or read by runtime config.

## Solution Assessment

Likely yes. For the narrow `preference-only`/preflight use case, an opt-in single-search plus direct-completion mode is the maintainable path; replacing the existing multi-turn recall path globally would not be the safest fix.
