## Summary

Keep open. Current main has hard-deadline and circuit-breaker work for the existing Active Memory path, but it still does not implement the requested opt-in single-shot architecture: config, manifest schema, docs, tests, and runtime all remain centered on the blocking embedded sub-agent loop.

## What This Changes

The issue requests an opt-in Active Memory `single-shot` mode that runs one memory search and one simple LLM completion instead of invoking the embedded agent loop.

## Best Possible Solution

Add a default-preserving Active Memory config mode, keep the current embedded-agent path as the default, and implement a plugin-owned `single-shot` branch that uses the existing memory search manager for one bounded search and the simple-completion runtime for one bounded model call, with manifest, docs, changelog, and regression tests.

## Reproduction Assessment

Yes for the implementation gap: on current main, a static check shows the requested config key is absent from types and manifest schema, and eligible turns still call `runEmbeddedPiAgent`. The exact latency numbers from the report are empirical, but reproducing the missing single-shot capability does not require live provider access.

## Solution Assessment

Yes in direction: an opt-in default-preserving mode is the narrowest maintainable way to address the latency architecture request without changing recall-heavy or contextual behavior. The implementation should reuse existing plugin SDK memory-search and simple-completion surfaces rather than adding owner-specific core behavior.
