## Summary

Keep this PR open: it targets a real CLI error-message bug, but the proposed missing-provider check is narrower than the media-understanding resolver and can misreport configured media/CLI attempts as missing configuration.

## What This Changes

The PR adds CLI fallback errors and regression tests so `infer image describe` and `infer audio transcribe` distinguish missing media provider configuration from configured-provider empty output.

## Best Possible Solution

Revise the PR so the CLI missing-configuration decision uses the same resolved-entry or attempt-state semantics as media understanding: emit the new missing-provider message only when no configured or auto-resolved route exists, and preserve the existing empty-output errors after any configured provider, model entry, or CLI route was attempted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve empty-output errors after configured media attempts:** `src/cli/capability-cli.ts:493-502`
  - body: The new fallback check only looks for registry providers with generic auth/config. A user can configure `tools.media.audio.models` or `tools.media.image.models` with a CLI entry, have that entry run and return empty output, and then this helper still returns false, so the CLI reports “No ... provider configured” instead of preserving the intended empty-result error. Please key this branch off the resolver/attempt state or include configured media entries before throwing.
  - confidence: 0.87
