## Summary

Keep open. The PR targets a real CLI error-message bug, but the new missing-provider check is narrower than the media-understanding resolver and can mislabel configured media/CLI attempts that return empty output as “no provider configured.”

## What This Changes

The PR adds CLI fallback logic and tests so `infer image describe` and `infer audio transcribe` emit missing-provider configuration errors separately from configured-provider empty-output errors.

## Best Possible Solution

Keep the PR open and align the CLI fallback decision with the media-understanding resolver or returned decision metadata: emit the missing-configuration message only when no entries were resolved or attempted, and preserve the empty-output error whenever a configured provider, configured CLI entry, shared media model, `agents.defaults.imageModel`, or auto-detected local CLI was actually tried.

## Reproduction Assessment

Yes. The linked issue gives a deterministic current-main no-provider reproduction, and the PR-specific regression can be reproduced by configuring `tools.media.audio.models` or `tools.media.image.models` with a CLI entry that returns empty output while no generic provider auth/config exists.

## Solution Assessment

No. The requested UX direction is correct, but the proposed helper is not the best implementation because it duplicates only part of media-understanding resolution instead of using the same resolved-entry or attempt-state signal as the runner.
