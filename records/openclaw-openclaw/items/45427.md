## Summary

Keep this issue open. Current main has sender display fields in channel context, but the trusted inbound metadata envelope intentionally excludes human names and renders sender names only in user-role untrusted metadata, so the requested trusted sender display-name field remains unimplemented.

## Best Possible Solution

Keep the issue open and implement a generic trusted sender display-name contract in the inbound metadata path: define the field and trust semantics in the core MsgContext/prompt envelope or channel plugin contract, populate it from verified channel directory/profile data at ingestion, keep untrusted conversational labels separate, and add regression tests proving agents see the trusted field without relying on user-role context.

## Risks / Open Questions

- Closing this now would leave the reported identity-misrouting gap unresolved because agents still do not receive a trusted sender display-name field in the authoritative inbound metadata block.
- A correct fix needs explicit trust semantics: which channel/plugin sources may populate a trusted display name, how stale directory entries are handled, and how the dynamic field interacts with prompt-cache stability.
