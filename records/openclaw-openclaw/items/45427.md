## Summary

Keep open. Current main carries `SenderName` through internal context, hooks, and Discord ingestion, but the agent-facing trusted inbound metadata deliberately excludes human names and renders sender name/id in user-role untrusted metadata instead. The issue is recent, has a concrete production failure mode in the discussion, and the remaining work crosses the prompt/security trust boundary rather than being already implemented.

## What This Changes

The issue requests a trusted platform-resolved sender display name in inbound metadata so agents do not infer names from untrusted conversational context.

## Best Possible Solution

Keep the issue open for a maintainer-owned identity contract: expose a current-turn platform-resolved display label without conversational drift, keep `sender_id` as the authoritative routing/security key, document whether the label is human-mutable, and add prompt/Discord tests that prevent agents from using stale conversational names for recipient selection.

## Reproduction Assessment

Yes. A source-level reproduction is sufficient: pass a context with `SenderName` into `buildInboundMetaSystemPrompt` and `buildInboundUserContextPrefix`; current tests and source show it is absent from trusted metadata and present only in untrusted user-context blocks.

## Solution Assessment

Unclear. The requested trusted `sender_display_name` field addresses the reported failure mode, but a direct trusted-name promotion conflicts with current prompt-security comments; the safer path is a typed identity contract that distinguishes authoritative ids from display labels.
