## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main intentionally distinguishes ordinary authorized channel senders from internal gateway clients with scope metadata, and this branch would reject normal external `/phone arm` and `/phone disarm` callers by treating missing gateway scopes as missing admin authority.

## What This Changes

The PR removes the WebChat-only condition from the phone-control arm/disarm admin guards and updates tests to expect Telegram callers without `operator.admin` to be rejected.

## Best Possible Solution

Keep the current internal gateway-scope guard unless maintainers intentionally change phone-control into an owner-only external-channel command. If that stronger policy is desired, add an explicit owner/channel authorization seam and cover normal channel senders, WebChat/internal gateway callers, and non-admin scoped gateway callers in tests.

## Reproduction Assessment

Yes. A unit-level reproduction is clear: invoke the phone-control command handler on current main with `channel: "telegram"` and no `gatewayClientScopes`, which is allowed for arm/disarm today; invoke it with `gatewayClientScopes: ["operator.write"]`, which is rejected.

## Solution Assessment

No. The submitted patch is not the best way to solve the reported security concern because `operator.admin` gateway scopes are internal control-plane metadata, while ordinary external channel commands normally lack that field. A safer solution would use explicit owner authorization if maintainers want to narrow external access.
