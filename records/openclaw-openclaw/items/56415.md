## Summary

Keep this PR open for maintainer judgment, but do not merge it as-is. The original security concern for scoped gateway callers is already handled on current main, while this PR’s unconditional scope check would block normal authorized external-channel `/phone` usage because external channel commands do not carry internal gateway scopes.

## What This Changes

The PR changes the phone-control plugin so `/phone arm` and `/phone disarm` require `operator.admin` for every channel, and updates tests for Telegram callers without that scope.

## Best Possible Solution

Keep current main’s internal gateway-scope guard unless maintainers intentionally change `/phone` into an owner-only external-channel command; if they do, implement that with channel owner identity or command-owner policy rather than treating absent internal gateway scopes as failed admin scope.

## Reproduction Assessment

Yes. The current unit-level path is clear: a `telegram` command context without `gatewayClientScopes` reaches the arm/disarm path on main, while scoped `operator.write` gateway callers are blocked; the PR changes the former expected behavior to rejection.

## Solution Assessment

No. The submitted patch is not the best fix because it conflates missing internal gateway scopes with unauthorized external channel senders; a safer design would preserve sender allowlist/owner semantics and only require `operator.admin` when an internal scoped gateway client is present, or add an explicit owner-only policy for `/phone`.
