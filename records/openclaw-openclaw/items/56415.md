## Summary

Keep open for maintainer review. The PR targets an auth-sensitive phone-control surface, but current main now deliberately separates internal gateway-scoped callers from ordinary authorized channel command senders; the proposed unconditional check would reintroduce the same external-channel regression that was restored after the earlier admin-scope hardening attempt.

## What This Changes

The PR makes `/phone arm` and `/phone disarm` reject every caller without `operator.admin`, and updates phone-control tests so Telegram callers without that scope are blocked.

## Best Possible Solution

Keep the current internal gateway-scope guard unless maintainers intentionally change phone-control into an owner-only external-channel command. If that stronger policy is desired, add a clear owner/channel authorization seam and tests for normal channel senders, WebChat/internal gateway callers, and non-admin scoped gateway callers.

## Reproduction Assessment

Yes. The current unit path is high-confidence: a Telegram command context without `gatewayClientScopes` reaches arm/disarm on main, while a Telegram context with only `operator.write` is rejected; the PR changes the first case to rejection.

## Solution Assessment

No. The submitted patch is not the best solution because it treats absent internal gateway scopes as failed admin scope for ordinary external command senders; a safer fix would model external owner/admin authorization directly if maintainers want that policy.
