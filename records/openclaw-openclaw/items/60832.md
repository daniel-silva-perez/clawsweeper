## Summary

Keep this issue open. Current main still presents the Control UI composer plus action as “New session” while routing it through the `/new` reset path for the active chat, and the Gateway already has separate create-vs-reset session concepts that the UI does not expose clearly.

## What This Changes

The issue requests renaming or otherwise distinguishing the Control UI composer plus action so users know it resets current context instead of creating a separate session entry.

## Best Possible Solution

Update the Control UI so the existing composer plus action accurately reads as a reset-context action, with matching tooltip, aria label, confirmation text, and tests. If maintainers want a real new conversation container from the same area, add that as a separate `sessions.create`-backed action rather than leaving the reset action labeled as session creation.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main renders the composer plus as `New session`, sends `/new` with reset confirmation, and never calls `sessions.create` from that composer path.

## Solution Assessment

Unclear for the broader product direction. The narrowest maintainable fix is the requested reset-context wording update, while adding a second true new-session-entry control would be a larger UX decision.
