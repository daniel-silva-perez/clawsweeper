## Summary

Keep open. Current main still hardcodes short-term promotion writes to `MEMORY.md`, exposes no `promotionTarget` or equivalent memory-core config/CLI/runtime option, and docs still describe deep promotion as writing only to `MEMORY.md`; adjacent memory-growth and promotion-filter PRs do not implement this target-file feature.

## What This Changes

Add a backward-compatible memory-core configuration option so short-term memory promotions can write to a configurable, searchable target file instead of always appending to `MEMORY.md`.

## Best Possible Solution

Implement the target selection in the bundled memory-core plugin, defaulting to the current `MEMORY.md` behavior, with a safe workspace-relative Markdown path resolver, deterministic date-token expansion, manual CLI and managed dreaming plumbing, automatic indexing/search coverage for the resolved target, focused tests, public docs, and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: any successful `applyShortTermPromotions()` call on current main writes to `path.join(workspaceDir, "MEMORY.md")`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable solution is a memory-core-owned config setting that preserves `MEMORY.md` as the default while routing both manual and scheduled promotion through the same safe target resolver and indexing contract. Asking users to maintain `memorySearch.extraPaths` manually would not satisfy the reported behavior, especially for writes and root-level target files.
