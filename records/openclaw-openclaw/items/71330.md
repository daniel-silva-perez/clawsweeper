## Summary

Keep open. Current main at f2405c830bd925bd24f91b6bac5d417c6d36e5ba still hardcodes short-term memory promotions to MEMORY.md. There is no promotionTarget-style config in the memory-core manifest or resolved dreaming config, the manual CLI has no target option, managed dreaming reports and writes MEMORY.md, and docs/tests still assert that destination. The related memory/wiki reports are broader lifecycle or vault-structure ideas and do not supersede this specific target-file feature.

## Best Possible Solution

Implement a backward-compatible memory-core promotion target setting defaulting to MEMORY.md. Resolve a safe workspace-relative Markdown path with documented date tokens, use it from manual memory promote --apply and managed deep dreaming, create missing parent directories/files with the existing long-term-memory header behavior, and ensure the resolved target is indexed/searchable by builtin and QMD memory search. Update manifest schema, config resolution, focused promotion/CLI/dreaming/indexing tests, public docs, and changelog.

## Clownfish Work Prompt

Implement configurable memory promotion target support for the bundled memory-core plugin, resolving this issue without changing unrelated memory architecture. Add a backward-compatible public setting under `plugins.entries.memory-core.config`, preferably `promotionTarget`, defaulting to `MEMORY.md`. Resolve it as a safe workspace-relative Markdown path; reject or ignore unsafe absolute paths, `..` escapes, non-Markdown targets, and symlink/path traversal hazards with focused tests. Support documented date tokens such as `YYYY`, `MM`, and `DD`, using the same timezone basis as dreaming/manual promotion. Thread the resolved target through `ApplyShortTermPromotionsOptions`, manual `openclaw memory promote --apply`, and managed deep dreaming so both write/report the actual target. Create missing parent directories and missing target files with the existing long-term-memory header behavior. Ensure promoted content remains searchable in both builtin and QMD memory search when the target is outside the default `MEMORY.md` or `memory/**/*.md` collections, avoiding duplicate indexing when the target is already covered. Update memory-core manifest schema, config resolution tests, promotion/CLI/dreaming/indexing tests, docs at `docs/cli/memory.md` and `docs/reference/memory-config.md`, and add a user-facing changelog entry with proper attribution. Do not move this into core beyond generic memory-host seams needed for indexing, and do not alter unrelated memory-wiki or adaptive-memory behavior.
