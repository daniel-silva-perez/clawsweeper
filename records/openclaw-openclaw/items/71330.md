## Summary

Keep open. Current main still hardcodes short-term promotion writes to `MEMORY.md`, the manual and managed promotion call paths expose no target override, and the memory-core plugin schema has no `promotionTarget`-style option. The adjacent open PR #74088 addresses size budgeting for `MEMORY.md`, but it does not implement configurable promotion destinations.

## What This Changes

Add a backward-compatible memory-core configuration option so short-term promotions can be written to a configurable target file instead of always appending to `MEMORY.md`.

## Best Possible Solution

Implement this in bundled memory-core as a backward-compatible promotion target resolver shared by manual `memory promote --apply` and managed dreaming. The end state should default to `MEMORY.md`, accept only constrained workspace-relative Markdown targets with deterministic date-token expansion, create missing target files with the expected header behavior, and ensure the resolved target is indexed by memory search.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `applyShortTermPromotions()` always sets `memoryPath` to `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. The requested memory-core-owned config path is the narrowest maintainable direction because the write target lives in the promotion writer; `memorySearch.extraPaths` alone would not change where promotions are written. The open size-budget PR is useful adjacent work, but it does not solve the target-file requirement.
