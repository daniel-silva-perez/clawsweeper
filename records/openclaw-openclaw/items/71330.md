## Summary

Keep open. Current main still writes short-term promotions to `MEMORY.md` unconditionally, the manual and managed promotion paths have no target override, and the memory-core schema/docs do not expose a `promotionTarget`-style setting. The adjacent open PR #74088 addresses size budgeting for `MEMORY.md`, but it explicitly does not make the promotion destination configurable.

## What This Changes

Add a memory-core configuration option that lets promoted short-term memory write to a configured Markdown target instead of always using `MEMORY.md`, while keeping defaults backward compatible and search indexing aligned.

## Best Possible Solution

Implement a memory-core-owned promotion target resolver shared by manual `memory promote --apply` and managed dreaming. The end state should default to `MEMORY.md`, validate workspace-relative Markdown targets, support deterministic date-token expansion if accepted, create missing files with the existing header behavior, and automatically include the resolved target in memory search indexing without requiring duplicate user config.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `applyShortTermPromotions()` always resolves `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review was required to stay read-only.

## Solution Assessment

Yes. A memory-core-owned config path is the narrowest maintainable direction because the writer owns the destination; memory search `extraPaths` can help indexing but cannot change where promotions are written. The adjacent size-budget PR is useful but does not address the target-file requirement.

## Repair Work Prompt

Implement configurable short-term promotion target files for the bundled memory-core plugin. The observable issue is that manual `openclaw memory promote --apply` and managed dreaming always append promoted short-term memory to `MEMORY.md`, which can bloat the bootstrap file and mix generated metadata with curated memory. Add a config option under `plugins.entries.memory-core.config.dreaming` or the nearest memory-core-owned promotion config shape, defaulting to `MEMORY.md`. Use one shared resolver for manual promotion and managed dreaming. Validate targets conservatively: Markdown only, workspace-relative unless an existing approved memory-search path pattern already allows the destination, no absolute paths, no traversal outside the workspace, no symlink escapes, deterministic date-token expansion only for an explicitly supported small token set such as `YYYY`, `MM`, and `DD`. Create missing files with the existing `# Long-Term Memory` style header or a clearly documented equivalent. Ensure the resolved target is searchable by default, including root-level targets such as `MEMORY_PROMOTIONS.md`, without forcing users to duplicate memory-search config. Update CLI help, plugin config schema/UI hints if needed, docs, focused tests, and CHANGELOG.md with credit to `@oc-mostlycopy`. Do not move owner-specific behavior into core, do not implement broad per-agent wiki vaults from #63829, and do not replace this feature with a size cap only; #74088 is adjacent and may require rebase/conflict coordination.
