## Summary

Keep open. Current main still writes short-term promotions to `MEMORY.md` unconditionally, the manual and managed promotion call paths expose no target override, and the memory-core config schema/docs have no `promotionTarget`-style option. The adjacent open PR #74088 addresses `MEMORY.md` size budgeting for a related failure mode, but it does not make the promotion destination configurable.

## What This Changes

Request a backward-compatible memory-core configuration option for choosing the short-term promotion target file instead of always appending promoted memories to `MEMORY.md`.

## Best Possible Solution

Implement this in the bundled memory-core plugin as a shared promotion target resolver used by manual `memory promote --apply` and managed dreaming. The end state should default to `MEMORY.md`, validate constrained Markdown targets, support deterministic date-token expansion if accepted, create missing files with the existing header behavior, and ensure the resolved target is included in memory search without requiring users to duplicate config manually.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `applyShortTermPromotions()` always resolves `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review was required to stay read-only.

## Solution Assessment

Yes. A memory-core-owned config path is the narrowest maintainable direction because the write target lives in the promotion writer; `memorySearch.extraPaths` can help indexing but cannot change where promotions are written. PR #74088 is useful adjacent size-budget work, but it does not solve the target-file requirement.
