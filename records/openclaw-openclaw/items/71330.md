## Summary

Keep open. Current main still hardcodes short-term promotion writes to `MEMORY.md`, has no `promotionTarget`/equivalent config in the memory-core manifest schema, and the manual CLI, managed dreaming sweep, tests, and docs all still describe/apply promotion to `MEMORY.md`. The related issues in the provided context cover broader adaptive memory/wiki structure but do not implement this specific promotion-target configuration.

## Best Possible Solution

Implement this in the bundled memory-core plugin as a backward-compatible config option defaulting to `MEMORY.md`, validate and resolve safe workspace-relative target paths with date-token support, thread the resolved target through manual `memory promote --apply` and managed deep dreaming, create parent directories/new markdown files as needed, and include the resolved target in memory indexing/search/watch paths. Update the memory-core manifest schema, focused tests, and the public memory/dreaming docs.

## Risks / Open Questions

- A configurable promotion target must be constrained to safe workspace-relative markdown paths, with absolute paths, `..` traversal, and symlink escape behavior considered before writing files.
- Searchability is part of the requested behavior: builtin indexing, watchers, and QMD default collections need to include the resolved promotion target without requiring users to duplicate it in `memorySearch.extraPaths`.
- Date-token templating should use deterministic token semantics and the same timezone source as promotion headers/dreaming to avoid surprising monthly file rollovers.
