## Summary

Keep open. Current main still writes memory-core short-term promotions to hardcoded `MEMORY.md`, exposes no `promotionTarget` or equivalent CLI/config option, and documents manual and managed deep promotion as `MEMORY.md`-only. The related open memory-core PRs cover contamination, size budgeting, and recall gates, but they do not implement the requested target-file feature.

## What This Changes

Add a memory-core configuration option for where short-term promotions are written, defaulting to `MEMORY.md` while allowing safe alternate Markdown targets that remain searchable.

## Best Possible Solution

Implement this in bundled memory-core as a backward-compatible target resolver shared by manual `memory promote --apply` and managed dreaming. The end state should default to `MEMORY.md`, allow constrained workspace-relative Markdown targets with deterministic date-token expansion, create missing target files with the expected header/frontmatter behavior, and ensure the resolved target is indexed by memory search.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `applyShortTermPromotions()` always sets `memoryPath` to `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. The requested memory-core-owned config path is the narrowest maintainable direction because the write target lives in the promotion writer, while memory-search `extraPaths` alone would not change where promotions are written.
