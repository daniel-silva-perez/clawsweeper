## Summary

Keep open. Current main still routes short-term promotion writes to `MEMORY.md` with no promotion-target option, the manual CLI and managed dreaming paths do not pass a target override, and the strict memory-core manifest/docs expose no equivalent config. The related open memory-core PRs address contamination, size budgeting, and recall gates, but not this target-file feature.

## What This Changes

Requests a memory-core config option to route short-term promotion writes away from hardcoded `MEMORY.md`, while keeping the default backward-compatible and preserving search coverage for the chosen target file.

## Best Possible Solution

Implement this inside bundled memory-core as a backward-compatible target resolver shared by manual `memory promote` and managed dreaming. The end state should default to `MEMORY.md`, allow safe workspace-relative Markdown targets with deterministic date-token expansion, create missing target files with the expected header/frontmatter, and ensure the resolved target is indexed by memory search.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `applyShortTermPromotions()` always sets `memoryPath` to `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without any target override. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. The requested memory-core-owned config path is the narrowest maintainable direction because the write target lives in memory-core promotion, while extra memory-search paths alone would not change where promotions are written.
