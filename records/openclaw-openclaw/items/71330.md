## Summary

Keep open. Current main still hardcodes short-term promotion writes to `MEMORY.md`, has no `promotionTarget` or equivalent memory-core config/CLI/runtime option, and the current docs still describe deep promotion as writing only to `MEMORY.md`; adjacent open memory-growth and promotion-filter PRs do not implement this target-file feature.

## What This Changes

Add a backward-compatible memory-core option so short-term memory promotions can write to a configurable, searchable target file instead of always appending to `MEMORY.md`.

## Best Possible Solution

Implement the target selection inside the bundled memory-core plugin, defaulting to current `MEMORY.md` behavior, with a safe workspace-relative Markdown path resolver, deterministic date-token expansion, shared manual/scheduled promotion plumbing, automatic indexing coverage for the resolved target, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: any successful `applyShortTermPromotions()` call on current main writes to `path.join(workspaceDir, "MEMORY.md")`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. A memory-core-owned config setting is the narrow maintainable solution because it preserves the current default while routing both manual and scheduled promotion through one validated target resolver and indexing contract. Requiring users to maintain `memorySearch.extraPaths` manually would not solve the write target and would miss root-level target files by default.
