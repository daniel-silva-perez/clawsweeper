## Summary

Keep this PR open. Current main still does not implement the PR's two intended changes: restart-health probes still use only process environment gateway credentials, and rate-limit failover still rotates one same-provider auth profile by default before model fallback. The PR is also not obsolete as a security/supply-chain cleanup item because its diff is limited to targeted TS source/tests, but the restart auth change is credential-sensitive and needs maintainer review on precedence and unresolved SecretRef warnings.

## Best Possible Solution

Keep this PR open for maintainer follow-up. If the restart-health behavior is desired, rebase the restart probe change onto current `restart-health.ts` and wire it to the existing config-aware probe-auth helper with explicit warning handling. If the rate-limit default should change, update the current `failover-policy.ts`, docs, and tests to skip same-provider rotation when a fallback model is configured; otherwise maintainers should close that portion as a rejected policy change while preserving the restart-health work separately if useful.

## Risks / Open Questions

- The PR head targets an older embedded runner shape (`src/agents/pi-embedded-runner/run.ts` helper exports) while current main has split failover policy and assistant failover modules, so it likely needs a rebase rather than direct merge.
- Restart-health probe auth is credential-sensitive: maintainers should decide config-vs-env precedence, local-vs-remote credential fallback, and how to surface unresolved SecretRef warnings before accepting that part.
- The rate-limit behavior is now documented and tested as product policy: one same-provider rotation by default with `auth.cooldowns.rateLimitedProfileRotations: 0` for immediate fallback. Changing the default should be an explicit maintainer policy decision, not treated as already implemented.
