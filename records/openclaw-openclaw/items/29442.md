## Summary

Keep open. Current main has meaningful prompt-injection hardening for cron and external content, but it still uses marker wrapping, warnings, special-token sanitization, tool policy, and docs-guided reader-agent mitigation. It does not implement the requested default dual-LLM privilege separation, read-only worker handoff, strict JSON sanitizer middleware, or manager continuation boundary for cron jobs.

## Best Possible Solution

Keep this open for maintainer security/product design. The best path is to decide whether OpenClaw should add a first-class read-only/tool-disabled reader-worker flow for cron and other external-content ingestion, or a narrower structured sanitizer/taint hook that can replace or block wrapped content before any privileged manager turn continues. The design should account for ClawMoat, #28913, closed unmerged scanner/sanitizer proposals, and tests proving untrusted external content cannot silently reach privileged cron execution or memory capture as ordinary prompt text.
