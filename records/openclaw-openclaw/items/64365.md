## Summary

This PR should stay open. Current main still lacks outbound Slack `replyBroadcast` support, Slack's SDK supports the underlying `reply_broadcast` parameter for threaded `chat.postMessage` calls, and the branch remains a focused implementation candidate rather than something already implemented or clearly superseded by #63904. The remaining work is merge-readiness: rebase to the current message-tool discovery seam and resolve the docs and media/upload semantics before merge.

## What This Changes

The PR branch adds Slack `replyBroadcast` plumbing across docs, action runtime, action adapter, message dispatch, tool schema, send logic, and tests so explicit threaded sends can request Slack `reply_broadcast`.

## Best Possible Solution

Keep this PR open as the narrow outbound implementation candidate. The desired end state is Slack-owned support for an explicit `replyBroadcast` message-tool field that is discoverable through the current plugin schema contribution API, forwarded through Slack dispatch/action/send layers, emitted as `reply_broadcast: true` only for valid threaded `chat.postMessage` sends, documented with current message-tool params, and covered by focused Slack schema/dispatch/send tests while #63904 continues tracking broader mirroring policy.

- The PR branch appears based on an older Slack message-tool schema shape; it likely needs a rebase and migration to the current `message-tool-api.ts` schema contribution surface before it can be reviewed accurately.
