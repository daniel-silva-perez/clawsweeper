## Summary

Keep this PR open. Current main still lacks outbound Slack `replyBroadcast` / Slack API `reply_broadcast` support, while the PR remains a plausible focused implementation candidate for a real Slack API behavior. The related #63904 is broader and still open; current main only ships inbound `thread_broadcast` handling, not outbound broadcast sends.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best path is to rebase it onto current main, wire `replyBroadcast` through Slack-owned message tool discovery/schema, message-action dispatch, action runtime, the `actions.ts` adapter, and `send.ts`; emit Slack `reply_broadcast: true` only for explicit threaded `chat.postMessage` sends; either support or reject upload/media broadcast semantics explicitly; update Slack docs to the current message tool contract; and add targeted tests. Keep broader automatic inbound/outbound mirroring policy tracked in #63904.

## Risks / Open Questions

- Merging as-is may still document or expose `replyBroadcast` for upload/media paths where the PR context shows the media send path may not honor it, creating a silent no-op.
- The PR docs example in the provided review context appears to need alignment with the current message tool contract: `action`, `message`, and `threadId`/`replyTo` rather than unsupported `content`/`threadTs` fields for the generic message tool path.
- `reply_broadcast` intentionally expands visibility from a thread into the parent channel, so the final implementation should keep the flag explicit, thread-gated, and covered by focused Slack send/schema/dispatch tests.
