## Summary

Keep this issue open. Current main still has the WhatsApp inbound auto-reply path where final embedded-runner tool-error warning payloads can be delivered through `deliverWebReply`, and WhatsApp dispatch still does not pass `suppressToolErrorWarnings` or an equivalent channel-scoped suppression policy.

## Best Possible Solution

Implement a channel-safe tool-error policy for WhatsApp and similar external-recipient channels. The likely narrow fix is to pass `suppressToolErrorWarnings` or a channel-scoped equivalent into WhatsApp inbound auto-reply runs, then add regression coverage proving read/edit/write/message failure warnings are not delivered to WhatsApp while normal replies and intentional tool media still deliver.
