## Summary

Keep this issue open. Current main still has the WhatsApp inbound auto-reply path where embedded-runner tool-error warning payloads can become final reply payloads and pass through `deliverWebReply`; the WhatsApp dispatch path still does not set `suppressToolErrorWarnings` or an equivalent external-channel suppression policy.

## Best Possible Solution

Keep this issue open. Implement a channel-safe tool-error policy for WhatsApp and similar external-recipient channels, likely by passing `suppressToolErrorWarnings: true` or a channel-scoped equivalent into WhatsApp inbound auto-reply runs, then add regression coverage proving read/edit/write/message failure warnings are not delivered while normal replies and intentional tool media still work.
