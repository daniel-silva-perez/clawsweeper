## Summary

Keep this issue open. Current main already has the runtime path needed for oMLX through the OpenAI-compatible embedding adapter, and the docs mention generic OpenAI-compatible embedding endpoints, but the oMLX-specific discoverability gap described in the report remains because current docs/source do not mention `oMLX` or `omlx`.

## What This Changes

The issue requests documentation, and optionally native support, for using oMLX as an OpenAI-compatible local embedding provider for memorySearch.

## Best Possible Solution

Add a focused docs-only oMLX recipe to the memory provider documentation that explains using the existing OpenAI-compatible path, warns not to configure oMLX as Ollama, and tells users to reindex after changing embedding models or vector space.
