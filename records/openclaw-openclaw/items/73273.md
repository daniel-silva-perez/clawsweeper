## Summary

Keep this open. Current main has related handshake-timeout and plugin-registry startup mitigations, but the report describes a more specific failure: the CLI receives `connect.challenge` and then appears to loop reading the 92KB OpenClaw package manifest before signing the nonce. I did not find source, tests, changelog, or release evidence proving that exact pre-auth package.json read loop is fixed.

## What This Changes

The issue reports that after a 4.24 to 4.25 upgrade, normal CLI commands such as `openclaw cron list` time out before sending the signed `connect.challenge` nonce, with strace pointing at repeated reads of `node_modules/openclaw/package.json`.

## Best Possible Solution

Keep the issue open and route it to a maintainer to collect the strace or a current-main repro, then make the CLI pre-auth path deterministic: signing and sending `connect` should not depend on plugin registry materialization or repeated package manifest resolution. A good fix would include a focused minimal-gateway regression that proves a CLI RPC emits `connect` under the handshake budget before plugin/package metadata work can block it.

## Reproduction Assessment

Unclear. The issue gives actionable v4.24/v4.25 steps and strong strace observations, but current-main reproduction depends on the reporter's poisoned local install/state and the raw traces are not attached here.

## Solution Assessment

No proposed patch is under review. The current timeout/config mitigations are useful but not sufficient proof of the best fix, because they do not directly eliminate or diagnose the reported pre-auth package.json walk before nonce signing.
