## Summary

Keep this issue open. Current main has related mitigations for cold CLI handshakes and plugin-registry startup, but I still do not see source, tests, docs, changelog, or release provenance proving the reported post-challenge package.json read loop is fixed before nonce signing.

## What This Changes

The issue reports that after upgrading from 4.24 to 4.25, CLI RPC commands receive `connect.challenge` but then hang while repeatedly reading the OpenClaw package manifest instead of signing and sending the nonce.

## Best Possible Solution

Keep the issue open and have a maintainer collect the strace or a current-main repro, then isolate the CLI pre-auth path so receiving `connect.challenge`, signing the device payload, and sending `connect` cannot depend on plugin-registry materialization or repeated package/version resolution. A focused regression should prove a CLI RPC emits `connect` within the handshake budget even when plugin/package metadata is slow or stale.

## Reproduction Assessment

Unclear. The report provides strong v4.24/v4.25 reproduction steps and strace observations, but the raw traces and local poisoned state are not available in the repository, and I did not run a live reproduction against the reporter's environment.

## Solution Assessment

No proposed patch is under review. The current timeout/config and registry-repair paths are useful mitigations, but they are not the best complete solution unless paired with proof that the pre-auth signing path no longer blocks on package or plugin metadata.
