## Summary

Keep this open. Current main has related timeout, event-loop readiness, and plugin-registry startup mitigations, but I did not find source, tests, docs, changelog, or release evidence proving the specific post-challenge package-manifest read loop is fixed before nonce signing.

## What This Changes

The issue reports CLI commands hanging after receiving `connect.challenge`, repeatedly reading the OpenClaw package manifest, and timing out before signing the device nonce after a 4.24 to 4.25 gateway upgrade.

## Best Possible Solution

Keep the issue open and make the CLI pre-auth path deterministic: after `connect.challenge`, signing and sending `connect` should not depend on plugin registry materialization or repeated package/version metadata resolution. A focused regression should prove a minimal gateway receives `connect` within the handshake budget even when package metadata or plugin registry reads are slow or stale.
