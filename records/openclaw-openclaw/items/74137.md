## Summary

Keep this issue open. Current main has several blank-input defenses, but they are either ingress-specific or after-the-fact repair. I could not prove that current main prevents the reported Telegram/WeChat-style blank user turn before provider submission, and the embedded runner still has a runtime-only path that can submit an empty visible prompt when there is existing session context.

## Best Possible Solution

Keep this open and fix the pre-provider path: add a last-chance guard so user-triggered channel runs cannot persist or submit a blank visible user prompt, while preserving explicit internal/runtime-only events that are intentionally not user-authored. Cover the shared embedded runner path plus Telegram/channel ingress tests, and keep session-file repair as a safety net rather than the primary defense.
