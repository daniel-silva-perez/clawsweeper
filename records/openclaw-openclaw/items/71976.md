## Summary

Keep open. Current main still has the reported memory-core behavior: Light dreaming refreshes daily/session signals before sorting by recency ahead of real recall count, deep promotion still gates on a combined signal count that includes daily signals, and snippet relocation still uses the narrow span. The provided GitHub context also shows open PR #74051 with closing syntax for this issue, so the issue should wait for that PR or a maintainer-approved replacement.

## What This Changes

The issue requests a memory-core fix so real recall signals drive Light dreaming selection and promotion eligibility instead of daily-refreshed entries, with a wider rehydration search span.

## Best Possible Solution

Land #74051 or a focused replacement that makes real recall signals control default Light selection and recall-gate eligibility, keeps daily/session ingestion useful for scoring without satisfying recall gates by itself, widens robust live-note rehydration, and updates the memory-core tests/docs that define the signal contract.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the reported behavior: daily/session ingestion can refresh `lastRecalledAt`, Light sorts by that timestamp first, promotion gates on combined signal count, and relocation still uses the narrow span.

## Solution Assessment

No for a sort-only fix; the reporter's follow-up correctly shows the bug spans Light ordering, recall-gate semantics, score calibration, and rehydration. The best maintainable fix should handle those surfaces together while preserving grounded-evidence promotion behavior.
