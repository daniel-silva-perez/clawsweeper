## Summary

Keep open. Current main still has the reported Memory Dreaming behavior, and open PR #74051 is linked with closing syntax for the same bug, so the issue should remain open until that PR or a focused replacement lands.

## What This Changes

The issue reports that Memory Dreaming Light sorting and promotion/rehydration logic let daily/session signals crowd out genuine recall entries, causing valid short-term recall candidates not to appear or promote.

## Best Possible Solution

Review and land #74051, or a focused replacement, so real recall signals drive Light ordering and recall-gate eligibility, daily/session ingestion can still inform scoring without satisfying recall gates by itself, rehydration handles shifted snippets, and memory-core tests/docs define the corrected contract.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the mechanism: daily/session ingestion refreshes lastRecalledAt, Light sorts by that timestamp before recallCount, and promotion ranking/apply filters use combined signalCount for minRecallCount.

## Solution Assessment

No for the original sort-only fix. The maintainable fix should separate recall-gate semantics from daily/session scoring, preserve intentional grounded evidence, widen rehydration, and cover the behavior in tests/docs; the linked PR appears aimed at that broader boundary.
