## Summary

Keep open: current main and the latest release still contain the reported Light recency-first selection, combined signal-count recall gate, and narrow rehydration span. An open, unmerged PR is already linked with closing syntax, so closing this issue now would be premature.

## What This Changes

The issue requests memory-core changes so real recall signals, not daily/session ingestion recency or combined daily counts, control Light dreaming visibility and short-term promotion eligibility.

## Best Possible Solution

Memory-core should distinguish real recall signals from daily/session reinforcement when enforcing recall gates, keep non-recall signals useful for scoring and phase reinforcement, order Light candidates so recalled entries remain visible within the default limit, widen robust live-note rehydration, and update the tests/docs that define this signal contract.

## Reproduction Assessment

Yes. The current source path is enough to reproduce the reported mechanism: daily/session ingestion can refresh `lastRecalledAt`, Light sorts by that timestamp first, promotion gates on combined signal count, and tests still expect daily-only signals to clear default gates.

## Solution Assessment

No, this should not be closed yet and a sort-only fix would be incomplete. The maintainable solution needs to cover Light ordering, recall-gate semantics, scoring expectations, live-note rehydration, and the existing open implementation PR or a focused replacement.
