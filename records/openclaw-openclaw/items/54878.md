## Summary

Keep this issue open. Current main at 17811480dae20f65b526454d75e971ae8424ec10 still checks cross-agent access before spawned-session lineage, so a parent-owned ACP child like agent:codex:acp:<uuid> is denied under default tree visibility and still requires the child harness id in tools.agentToAgent.allow under visibility=all. The related fix PRs in the provided context were closed unmerged, so there is no shipped or canonical resolved item to close against.

## Best Possible Solution

Keep this issue open and implement a narrow lineage-aware session visibility fix in src/plugin-sdk/session-visibility.ts. For parent-owned spawned targets, allow proven spawned-tree lineage before generic cross-agent rejection, including ACP child keys with a different agent id. Add guard-level and sessions_history regression coverage for agent:main:main reading agent:codex:acp:<id> or agent:opencode:acp:<id>, and audit session_status for its pre-guard cross-agent gate.

## Risks / Open Questions

- Closing now would leave ACP parent workflows unable to reliably retrieve their own child history unless operators broaden tools.agentToAgent.allow to include harness ids such as codex or opencode.
- The fix is security-sensitive: it should allow only proven spawned parent-child lineage before generic cross-agent denial, without broadening unrelated cross-agent session access.
- session_status has a separate cross-agent access check that may remain inconsistent if only sessions_history is patched.
