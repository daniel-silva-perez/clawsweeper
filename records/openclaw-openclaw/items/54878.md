## Summary

Keep this issue open. Current main still rejects a parent-owned ACP child session with a different `agent:` id before the spawned-session lineage check can allow it, and the `visibility=all` repro still requires the ACP harness id to match `tools.agentToAgent.allow`. The linked fix PRs were closed unmerged, so there is no canonical resolved item to close against.

## Best Possible Solution

Keep this issue open and implement a lineage-aware session visibility fix in `src/plugin-sdk/session-visibility.ts`: for parent-owned spawned targets, allow proven spawned-tree lineage before generic cross-agent rejection, including ACP child keys with a different agent id. Add guard-level and `sessions_history` regression coverage for a requester such as `agent:main:main` reading `agent:codex:acp:<id>` or `agent:opencode:acp:<id>`, and audit `session_status` for its extra cross-agent gate.

## Risks / Open Questions

- Closing now would leave ACP parent workflows unable to reliably retrieve their own child history unless operators broaden `tools.agentToAgent.allow` to include harness ids such as `codex` or `opencode`.
- The fix needs to stay narrow: allow proven spawned parent/child lineage before generic cross-agent rejection without broadening unrelated cross-agent session access.
- `session_status` has an additional pre-guard cross-agent gate that should be audited so status and history follow the same lineage rule.
