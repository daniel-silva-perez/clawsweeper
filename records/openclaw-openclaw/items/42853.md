## Summary

Keep this PR open. Current main still does not implement the proposed gateway heap-limit config, daemon service NODE_OPTIONS heap injection, node-service threading, or setup wizard prompt. The PR still addresses a real unresolved OOM path also tracked by open issue #52092, but it needs maintainer review and refresh against current main and the current NODE_OPTIONS security policy before it could be merged.

## Best Possible Solution

Keep this PR open for maintainer review or an equivalent replacement implementation. The best path is to add a documented gateway.nodeMaxOldSpaceMb config contract, compute a conservative default, emit a deterministic heap-only startup setting for both gateway and node services without importing ambient NODE_OPTIONS, detect both max-old-space-size spellings, update the current setup wizard path, and add focused tests for gateway service env, node service env, config schema/help/docs, wizard blank-to-auto behavior, hostile NODE_OPTIONS, and the related cron/OOM scenario.

## Risks / Open Questions

- The PR is stale against current main because it edits src/wizard/onboarding.gateway-config.ts, while current setup code lives in src/wizard/setup.gateway-config.ts.
- The PR's NODE_OPTIONS merge strategy can preserve arbitrary startup flags such as --require, conflicting with current service/env hardening that strips NODE_OPTIONS from untrusted sources.
- The heap-flag detector in the PR discussion still needs to handle Node's accepted underscore spelling, --max_old_space_size, before appending a second heap flag.
- The proposed default of 85% of total system RAM is a product/runtime-policy choice that may be too aggressive for containers, shared hosts, or memory-constrained machines.
