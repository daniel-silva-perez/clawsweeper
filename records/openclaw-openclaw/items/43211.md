## Summary

Keep this PR open. Current main still performs exact normalized provider/model allowlist checks and does not implement hyphen-stripped provider variants, but the PR should not be closed as obsolete or merged as-is because unconditional hyphen stripping can broaden authorization to a distinct custom provider ID. The prior review comments are material: they identify the same authorization risk, while the Greptile comment also noted a secondary allowedKeys/allowedCatalog divergence in the PR implementation.

## Best Possible Solution

Keep this PR open for maintainer or contributor rework. The best path is to add a failing reproduction for the specific runtime path that loses provider hyphens, then either fix that resolver to preserve the configured provider ID or add an explicit provider/plugin alias mechanism for the equivalence. Regression coverage should prove declared aliases work while distinct providers such as my-provider and myprovider remain separately allowlisted unless explicitly linked.

## Risks / Open Questions

- Merging the PR as written could allow a distinct provider ID, for example accepting myprovider/some-model when only my-provider/some-model was configured.
- The PR body does not include a concrete reproduction for the runtime path that strips provider hyphens, so the right fix may be preserving the provider ID in that resolver rather than expanding the allowlist.
- The Greptile review noted that the PR's allowedKeys expansion does not also create matching synthetic allowedCatalog entries for stripped variants, leaving catalog and authorization views divergent.
