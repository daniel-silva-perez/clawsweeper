## Summary

Current main still does not implement this PR's intended Slack reaction idempotency. The shared Slack action helpers still propagate `already_reacted` from reaction adds and `no_reaction` from removals, while only the status-reaction adapter has local catches. Overlapping open PRs #58242 and #71843 cover `already_reacted` in narrower ways, but they do not fully supersede this PR's removal/no-reaction race handling.

## Best Possible Solution

Keep this PR open for maintainer review or contributor rebase. The best fix is a small Slack-owned change on current main that treats `already_reacted` as success for reaction adds, treats `no_reaction` as success for explicit removal and remove-own race cleanup, preserves the current `cfg`/token/client/write-client contracts, and adds focused helper or action-runtime regression tests. Coordinate with #58242 and #71843 so only one current implementation lands.
