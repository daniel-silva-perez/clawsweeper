Keep open. Current main still maps inbound WhatsApp GIF-picker video messages to plain `<media:video>` and does not surface `videoMessage.gifPlayback` or `contextInfo.externalAdReply` metadata. The closed companion PR #49103 demonstrates a plausible narrow implementation path, but it was not merged and current main has not adopted the behavior.

Keep this issue open and implement the feature in the WhatsApp plugin: distinguish `videoMessage.gifPlayback` as GIF media, surface `externalAdReply.title/sourceUrl/body` with explicit WhatsApp media provenance, carry provider-supplied values through the existing untrusted structured-context sanitization path, add focused inbound tests, and update the WhatsApp docs and changelog.
