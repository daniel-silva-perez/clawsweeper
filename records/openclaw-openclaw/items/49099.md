## Summary

Keep open. Current main still drops the requested inbound WhatsApp GIF/ad metadata: video messages return the fixed `<media:video>` placeholder, `gifPlayback` and `contextInfo.externalAdReply` are not read, and the closed companion PR did not merge. This is a narrow WhatsApp plugin repair candidate, with the important caveat that provider-supplied metadata should flow through untrusted structured context rather than inline prompt text.

## What This Changes

Extract inbound WhatsApp GIF picker metadata such as `gifPlayback`, `externalAdReply.title`, `externalAdReply.body`, and `externalAdReply.sourceUrl` so agents can understand GIF/media context without video processing.

## Best Possible Solution

Implement the feature in the WhatsApp plugin: classify `videoMessage.gifPlayback` media as GIF, extract bounded `externalAdReply` title/body/source URL for relevant inbound media, and forward that data through the existing untrusted structured context channel with focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. A direct source-level reproduction is to pass a Baileys `proto.IMessage` with `videoMessage.gifPlayback: true` and `contextInfo.externalAdReply` into `extractMediaPlaceholder`; current main still returns `<media:video>` and the monitor has no media metadata path.

## Solution Assessment

No, the inline attribute form suggested in the issue is not the safest final shape. The narrow maintainable fix is still WhatsApp-owned, but provider-fetched metadata should be represented as explicit untrusted structured context while the visible placeholder can distinguish GIF from video.
