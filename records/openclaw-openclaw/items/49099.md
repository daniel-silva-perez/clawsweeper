## Summary

Keep open. Current main still drops the requested WhatsApp GIF/ad metadata: the inbound extractor returns a fixed `<media:video>` placeholder for all video messages, the monitor only forwards contact structured context, and the companion implementation PR #49103 was closed without merging.

## What This Changes

The issue requests extracting WhatsApp GIF picker metadata from inbound media so GIFs and provider title/source/description can reach the agent without video processing.

## Best Possible Solution

Implement a focused WhatsApp plugin fix: classify `videoMessage.gifPlayback` as GIF, extract bounded `contextInfo.externalAdReply` title/body/source URL for relevant inbound media, and pass that metadata through `untrustedStructuredContext` with docs, changelog credit, and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a Baileys-shaped `proto.IMessage` with `videoMessage.gifPlayback: true` and `contextInfo.externalAdReply` into `extractMediaPlaceholder`; current main returns `<media:video>` and has no monitor path for the metadata.

## Solution Assessment

The issue's inline attribute sketch is not the safest final form. The narrow maintainable solution is still WhatsApp-owned, but external provider metadata should be forwarded as untrusted structured context while the visible placeholder distinguishes GIF from video.
