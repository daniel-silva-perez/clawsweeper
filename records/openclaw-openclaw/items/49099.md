## Summary

Keep open. Current main still normalizes inbound WhatsApp GIF-picker messages as plain `<media:video>` and does not surface `videoMessage.gifPlayback` or `contextInfo.externalAdReply` metadata; the companion implementation PR #49103 was closed unmerged by the contributor and has not been adopted.

## What This Changes

Expose WhatsApp GIF-picker media as GIF placeholders and surface inbound media `externalAdReply` title/source/body metadata so agents can understand GIF context without video processing.

## Best Possible Solution

Implement this in the WhatsApp plugin: distinguish `videoMessage.gifPlayback` from regular video, extract `externalAdReply.title/sourceUrl/body` from inbound video/image media, carry those values through explicit WhatsApp media provenance and the existing untrusted structured-context sanitization path, then add focused tests, WhatsApp docs, and a changelog entry crediting the reporter.
