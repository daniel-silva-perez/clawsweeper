Keep open. Current main still drops WhatsApp GIF-picker and `externalAdReply` metadata, and the companion implementation PR was closed unmerged, so the reported user problem remains actionable in the WhatsApp plugin.

## What This Changes

The issue asks the WhatsApp plugin to distinguish inbound GIF-picker videos from normal videos and expose `externalAdReply` title, source URL, and body metadata for inbound media.

## Best Possible Solution

WhatsApp inbound should emit a distinct GIF media placeholder and carry `externalAdReply` title/body/source URL as explicit untrusted media metadata, documented and tested, without moving WhatsApp-specific behavior into core.

## Repair Work Prompt

Implement the remaining WhatsApp inbound GIF/media metadata work for this issue. Current `main` still returns plain `<media:video>` for `videoMessage.gifPlayback` and drops `contextInfo.externalAdReply` metadata.

Scope:
- Work in the WhatsApp plugin, not core, unless a generic test fixture needs to assert existing prompt formatting.
- Distinguish inbound `videoMessage.gifPlayback === true` from normal video, producing a GIF media placeholder where the body fallback is needed.
- Extract `externalAdReply.title`, `externalAdReply.sourceUrl`, and `externalAdReply.body` from inbound video/image media.
- Carry these values as explicit untrusted WhatsApp media metadata through `untrustedStructuredContext` or an equivalently safe existing structured-context path. Do not place raw third-party preview text inline as normal message body text or XML-like attributes.

- Treat #49103 as useful prior art, but do not copy its inline placeholder-attribute design without the untrusted metadata boundary. #39037 is adjacent CTWA metadata work; avoid broadening this PR into CTWA/session attribution unless a tiny shared helper is clearly needed.
