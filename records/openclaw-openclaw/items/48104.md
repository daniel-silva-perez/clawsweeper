## Summary

Keep open. Current main has adjacent execution-bias, strict-agentic, exec-approval event, diagnostics, and trajectory-export work, but it still does not implement the central request: a first-class distinction between technical SSH/tool failure, authorization or approval denial, and model-level soft refusal for explicitly authorized read-only operational diagnostics. The issue discussion, especially Ryce's authorization/config-state framing and the reporter's follow-up, remains actionable.

## Best Possible Solution

Keep this issue open for a core agent-runtime observability and policy design pass. The best path is to add first-class run metadata/events that classify tool execution failure, authorization/approval denial, and model/tool-routing soft refusal; refine prompt/runtime guidance for explicitly authorized read-only infrastructure diagnostics; and add regression coverage with a simulated provider response that refuses SSH/log inspection instead of calling the available exec path.

## Risks / Open Questions

- Closing now would hide a real remaining product gap: users still cannot reliably tell model soft refusal apart from permission/approval denial or actual SSH/tool failure.
- A future operator/trusted-admin mode could broaden execution authority, so the design needs explicit security review around exec approvals, sandboxing, and channel/operator authorization.
- Diagnostics and trajectory exports may help support investigate incidents, but they are not a user-facing live failure taxonomy and diagnostics intentionally omit payloads that may be needed to classify model soft refusals.
