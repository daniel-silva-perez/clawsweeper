## Summary

Keep open. Current main has adjacent machinery for exec approvals, elevated host execution, operator scopes, trajectory export, and a general execution-biased prompt, but it does not implement the issue’s requested distinction between technical tool failure, permission denial, and model-level soft refusal for authorized operational diagnostics. The issue comments are useful: Ryce frames this as authorization/config state drift, and the reporter clarifies that visibility between SSH/tool failure, missing authorization, and model veto is the central ask.

## Best Possible Solution

Keep this open for a core design/implementation pass. The likely path is to add first-class observability that classifies runs as tool execution failure, authorization/approval denial, or model/tool-routing refusal; refine system prompt/policy for explicitly authorized read-only operational diagnostics; and add regression coverage with a simulated provider response that tries to stay in cautionary guidance instead of calling the available exec/tool path.
