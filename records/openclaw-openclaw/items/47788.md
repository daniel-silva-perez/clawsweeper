## Summary

Keep this PR open, but it needs changes before merge. Current main still contains the Discord setup allowlist cast that the PR targets, but the proposed replacement asserts the wrong concrete shape for the actual Discord resolver output, so it does not really tighten the setup boundary.

## What This Changes

The PR adds a local Discord guild/channel allowlist entry type and replaces the Discord setup apply path’s `resolved as never` cast with a cast to that explicit entry array type.

## Best Possible Solution

Type the Discord setup allowlist boundary around the real resolver output, then map `guildId`/`channelId` results into the config writer’s `guildKey`/`channelKey` shape before applying the allowlist. Add a focused Discord setup test that exercises a resolved guild/channel entry so this mismatch cannot regress.

## Reproduction Assessment

Yes for the review finding: inspect the setup surface and resolver tests, which show group allowlist resolution returns `guildId`/`channelId`, then inspect the apply path and setter, which read `guildKey`/`channelKey`. A focused unit test that applies a resolved Discord guild/channel setup entry would reproduce the current mismatch without live Discord.

## Solution Assessment

No, not as-is. A named setter-entry cast is only maintainable if the resolved values actually have that shape; the safer fix is to map or type the resolver result explicitly before calling `setDiscordGuildChannelAllowlist`.
