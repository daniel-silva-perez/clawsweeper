## Summary

Keep this PR open. The reported Discord bug is still present on current main, because `thread-reply` only reads `media` while the sibling `send` action accepts `media`, `path`, and `filePath`; however, the PR is stale/incomplete as-is because it targets a removed shared test file and does not propagate `mediaReadFile` through the Discord thread-reply path.

## What This Changes

The PR changes Discord `thread-reply` attachment parsing to fall back from `media` to `path` to `filePath`, passes `mediaLocalRoots` into the runtime call, and adds alias precedence tests under the old shared action test path.

## Best Possible Solution

Land a current-main Discord extension fix that normalizes `thread-reply` media aliases with the same `media > path > filePath` precedence as `send`, propagates both `mediaLocalRoots` and `mediaReadFile` through the Discord action adapter/parser/runtime options, and moves coverage into the active Discord extension test suite.

## Reproduction Assessment

Yes. The linked bug gives concrete steps, and current main has a high-confidence static reproduction: `thread-reply` reads only `media`, while `send` reads `media`, `path`, and `filePath`.

## Solution Assessment

No, not as currently proposed. The alias fallback is the right narrow direction, but the PR needs a current-main repair because it omits `mediaReadFile` propagation and places tests in a removed shared test file.
