## Summary

Keep this PR open. Current main still only reads `media` for Discord `thread-reply` while the sibling `send` path accepts `media`, `path`, and `filePath`, so the linked #43570 bug remains valid; this PR is a useful start but needs a current-main repair before merge because it misses full media read callback propagation and puts coverage in a removed shared test file.

## What This Changes

The PR changes Discord `thread-reply` attachment parsing to accept `media`, `path`, then `filePath`, forwards `mediaLocalRoots`, and adds alias precedence/passthrough tests under `src/channels/plugins/actions/actions.test.ts`.

## Best Possible Solution

Land a current-main Discord extension fix that normalizes `thread-reply` media aliases with `media > path > filePath` precedence, threads the existing media access/read callbacks from the channel adapter through the guild-admin parser into the messaging runtime, and covers the active extension action, adapter, and runtime paths.

## Reproduction Assessment

Yes. Static reproduction on current main is high-confidence: call Discord `message(action="thread-reply")` with `path` or `filePath`; the active parser reads only `media`, so the attachment field is dropped before the runtime sends the message.

## Solution Assessment

No. The alias fallback is the right narrow direction, but this PR is not the best merge as proposed because it omits `mediaReadFile` propagation and places tests in a file that no longer exists on current main.
