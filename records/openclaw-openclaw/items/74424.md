## Summary

Keep open. Current main still has the reported in-flight coalescing gap: the gateway tool only writes the restart continuation from an emit hook, while the already-emitted restart path returns a successful coalesced response before that hook can run. The linked open PR #73812 is adjacent, but its public diff does not currently change the `hasUnconsumedRestartSignal()` coalescing path.

## What This Changes

The issue requests that `gateway` restart calls with `continuationMessage` either durably queue and deliver the continuation when coalesced with an in-flight restart or explicitly report that the continuation was not queued.

## Best Possible Solution

Keep restart coalescing, but make the accepted continuation contract truthful: either atomically merge the explicit continuation into the already-persisted restart handoff exactly once, or return a clear non-queued/error result so the agent can recover without assuming the follow-up was scheduled.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by taking the `gateway` restart action with `continuationMessage` while `hasUnconsumedRestartSignal()` is true: the scheduler returns a successful coalesced result before the tool's sentinel-writing hook can run, and startup only resumes continuations that reached the sentinel.

## Solution Assessment

Yes for the requested direction: preserving coalescing while guaranteeing or explicitly denying continuation queuing is the narrowest maintainable fix. The linked #73812 diff is useful adjacent work, but it does not currently appear to solve this exact in-flight early-return path.
