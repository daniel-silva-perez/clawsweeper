## Summary

Keep open. Current main still has the reported coalesced-restart continuation gap: the gateway tool only persists the continuation from restart emit hooks, while the already-emitted/in-flight coalescing branch returns success before those hooks are registered or run. The linked open PR should be reviewed as the likely fix path, but the issue is not implemented on main yet.

## What This Changes

The issue asks OpenClaw to preserve or truthfully report gateway restart `continuationMessage` delivery when a restart request coalesces with an already in-flight restart.

## Best Possible Solution

Review and land PR #73812 if it actually preserves the accepted continuation through coalesced restart paths; otherwise make a narrow fix in the gateway restart tool/scheduler so coalesced success either durably queues the continuation exactly once or returns an explicit non-queued/error result.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available: emit or schedule a restart until `hasUnconsumedRestartSignal()` is true, then call the gateway restart tool with `continuationMessage`; current main returns a coalesced success before the sentinel write hook can run.

## Solution Assessment

Yes, the issue’s requested contract is the narrow maintainable direction. A broader checkpoint/auto-resume system is not required for this specific accepted-parameter loss; the fix should stay in the restart coalescing and sentinel handoff path.
