## Summary

Keep open. Current main still has a reachable path where the gateway restart tool accepts a continuation, returns a coalesced success, and never runs the hook that writes the restart sentinel. The related discussion is useful: #74443 proposed the exact sentinel-merge fix but was closed unmerged, while #73812 is adjacent restart-continuation work and should be checked before creating duplicate work.

## What This Changes

The issue reports that a gateway `restart` tool call with `continuationMessage` can return coalesced success while dropping the post-restart continuation when it overlaps an in-flight restart.

## Best Possible Solution

Patch current main so a continuation-bearing restart request that coalesces with an emitted or pending restart either durably preserves the continuation before returning success, or explicitly reports that no continuation was queued. Keep the fix narrow to the gateway restart tool/sentinel scheduler path with regression coverage for the in-flight coalesced branch.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to emit a gateway restart so `hasUnconsumedRestartSignal()` is true, then call the gateway restart tool with `continuationMessage`; current main returns `coalesced: true` before running the hook that writes the continuation sentinel.

## Solution Assessment

Yes, the narrowest maintainable fix is to preserve or explicitly reject the continuation at the coalesced restart boundary. This should not be broadened into the larger checkpoint/auto-resume RFC or inbound message replay work.
