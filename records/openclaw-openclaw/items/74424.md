## Summary

Keep open: current main still has the exact early-return path that can accept a restart continuation and return `coalesced: true` without running the sentinel-writing hook. The discussion usefully points to open PR #73812 as the intended fix path, but that work is not merged on main, so this issue should remain open until that PR or an equivalent narrow fix lands.

## What This Changes

The issue reports that a `gateway` tool restart with `continuationMessage` can coalesce with an in-flight gateway restart and return success while dropping the promised post-restart continuation.

## Best Possible Solution

Keep this issue open until a merged fix either durably persists or merges the continuation when `scheduleGatewaySigusr1Restart` coalesces after a restart signal is already in flight, or explicitly reports that no continuation was queued. The fix should include a regression test for the already-in-flight branch, not only the already-scheduled/preparing coalescing paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to have an unconsumed SIGUSR1 restart signal, then call the gateway restart tool with `continuationMessage`; current `scheduleGatewaySigusr1Restart()` returns from the `hasUnconsumedRestartSignal()` branch before the gateway tool's `beforeEmit` sentinel write can run.

## Solution Assessment

No, current main is not the best solution because it can still report a successful coalesced restart while dropping the continuation. A maintainable fix should make continuation queuing explicit and covered by tests, either through PR #73812 if it covers this exact branch or through a narrower follow-up.
