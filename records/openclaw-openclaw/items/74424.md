## Summary

Current `main` still has the reported restart-continuation gap: the gateway tool only persists the continuation from an emit hook, while the in-flight coalesced path returns before any hook is registered or run. The issue should stay open until the linked restart-continuation PR is merged or adjusted to cover this exact coalesced path.

## What This Changes

The issue requests that `gateway` restart calls with `continuationMessage` either durably queue the continuation across coalesced restarts or explicitly report that the continuation was not queued.

## Best Possible Solution

Keep the issue open and land a narrow restart-continuation fix that covers both in-flight and already-scheduled coalescing, with regression tests proving the accepted `continuationMessage` is persisted or the tool response reports that it was not queued.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can schedule or emit a restart, then make a second `gateway` restart call with `continuationMessage` in the coalescing window and assert the continuation sentinel is not written on current `main`.

## Solution Assessment

Unclear whether the currently open PR is the best final patch until it is checked against this exact in-flight coalescing case. The narrow maintainable solution is to persist or merge continuation intent at the scheduler/tool boundary, or return an explicit non-queued result instead of normal success.
