## Summary

Keep this open. Current main has useful advisory/proactive memory surfaces, including `memory_search`, memory-lancedb auto-recall, and the Active Memory plugin, but it still does not provide the requested opt-in runtime policy that forces recall before selected model responses and can block the turn when recall cannot run. The linked implementation attempt, #43825, is closed and unmerged.

## Best Possible Solution

Keep this open for maintainer design. The best path is an opt-in runtime policy or plugin/context-engine API that can run recall before selected model responses, inject recalled rule context, and optionally fail closed with a controlled error when recall cannot run. It should explicitly define scope for embedded runs, CLI/harness execution, subagents/workflows, and existing Active Memory or memory-lancedb behavior rather than adding an unconditional global recall step.

## Risks / Open Questions

- A blanket enforced recall step could add latency, embedding/model cost, and hidden personalization to automation paths unless it is opt-in and scoped.
- Changing existing prompt hooks from best-effort mutation to hard gates could break plugin expectations; a new explicit policy/result shape may be safer.
- Active Memory and memory-lancedb already cover some proactive recall use cases, so the remaining design should avoid duplicating those plugins while covering enforcement and non-chat surfaces deliberately.
