## Summary

Keep open. Current main does not contain the new `agent.capabilities` RPC, so the PR is not obsolete, but the current PR head has a concrete correctness regression: listed agents that omit their own `skills` array will report no skills instead of inheriting `agents.defaults.skills`, which conflicts with the existing agent skill filter contract and tests. The Greptile import/export comments appear addressed, and the security pass found no supply-chain concern.

## What This Changes

The PR adds an `agent.capabilities` Gateway RPC with protocol schemas, READ-scope and method registration, generated Swift models, and a handler that aggregates model, tools, MCP servers, skills, channels, and session count for an agent.

## Best Possible Solution

Keep the additive RPC direction, but make the response use the existing effective skill-filter semantics or the raw agent-entry override check, then add focused handler/protocol coverage for default skill inheritance, explicit per-agent replacement, tool counts, READ scope, and generated client surfaces.

## Reproduction Assessment

Yes. A config with `agents.defaults.skills: ["github"]` and `agents.list: [{ id: "writer" }]` should report `github` for `writer`, but the PR's static code path returns `[]` because the resolved agent config owns `skills` as `undefined`.

## Solution Assessment

No. The Gateway surface is a reasonable narrow solution for discoverability, but this implementation is not the best fix until skills are resolved through the existing effective filter contract rather than by testing a resolved object for raw-property ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve inherited default skills:** `src/gateway/server-methods/agent-capabilities.ts:90-92`
  - body: `resolveAgentConfig()` always returns an object with a `skills` property, set to `undefined` when the raw agent did not configure one. Because this branch tests `Object.hasOwn(agentEntry, "skills")`, any listed agent without its own skills array will report `[]` instead of inheriting `agents.defaults.skills`, diverging from the existing runtime skill filter behavior. Use the raw agent entry or `resolveAgentSkillsFilter`/`resolveEffectiveAgentSkillFilter` before exposing this as `agent.capabilities`.
  - confidence: 0.95
