## Summary

Keep open. Current main does not already provide `agent.capabilities`, and the PR has a blocking correctness regression: listed agents that inherit `agents.defaults.skills` would be reported with no skills because the new handler tests `Object.hasOwn` on a resolved config object that always owns a `skills` property, even when it is `undefined`.

## What This Changes

The PR adds an `agent.capabilities` Gateway RPC, protocol schemas, READ-scope/method registration, generated Swift models, and a handler that aggregates model, tools, MCP server names, skills, channel bindings, and session count for an agent.

## Best Possible Solution

Keep the additive RPC direction, but make the response reflect effective agent capabilities by reusing the existing skill-filter semantics or raw agent-entry override check, then add targeted handler/protocol tests for default skills inheritance, explicit per-agent replacement, tool counts, READ scope, and generated client surfaces.

## Reproduction Assessment

Yes. A config with `agents.defaults.skills: ["github"]` and `agents.list: [{ id: "writer" }]` should report `github` for `writer`, but the PR's static code path returns `[]` because `resolveAgentConfig` owns `skills` as `undefined`.

## Solution Assessment

No. The proposed RPC is a reasonable Gateway surface, but this implementation is not the best fix until skills resolution uses the existing effective skill filter contract instead of testing a resolved object for raw-property ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve default skills for configured agents:** `src/gateway/server-methods/agent-capabilities.ts:90-92`
  - body: `resolveAgentConfig()` always returns an object with a `skills` property, set to `undefined` when the raw agent did not configure one. Because this branch tests `Object.hasOwn(agentEntry, "skills")`, any listed agent without its own `skills` array will report `[]` instead of inheriting `agents.defaults.skills`, which diverges from the existing runtime skill filter behavior. Use the raw agent entry or the existing effective skill-filter helper before exposing this as `agent.capabilities`.
  - confidence: 0.95
