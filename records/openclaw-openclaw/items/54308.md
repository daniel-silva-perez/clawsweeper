## Summary

Keep open. Current main still lacks the PR's intended Discord idle-drain/managed-identity teardown work, but the proposed patch is not merge-ready because it can start Discord without a resolved bot identity and is stale against the current provider cleanup/run-queue architecture.

## What This Changes

This PR adds Discord managed bot identity tracking, threads an idle-drain teardown path through inbound handling, adds SenderManagedAccountId context, updates Discord queue/provider tests, and includes small Telegram message schema and plugin type changes.

## Best Possible Solution

Rebase and narrow the change to a Discord-owned teardown fix that preserves fetchDiscordBotIdentity fail-closed startup behavior, adds a minimal idle-drain primitive to the current createChannelRunQueue/provider.cleanup path, and splits Telegram or plugin contract edits into separate PRs if they are still needed.

## Reproduction Assessment

Yes. The blocking PR defect can be reproduced with the current provider test shape by making client.fetchUser('@me') reject: current main rejects before handler creation, while the PR's catch path can continue with botUserId undefined.

## Solution Assessment

No. The idle-drain direction is plausible, but this patch is not the best current solution because it weakens fatal bot-identity startup semantics, is stale against the current queue/cleanup shape, and bundles unrelated surfaces.
