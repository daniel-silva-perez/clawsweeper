## Summary

Keep open. Current main still lacks the PR's intended Discord managed-identity idle-drain teardown, but the proposed branch is not safe to merge as-is because it changes Discord bot identity lookup from fail-closed to best-effort while Discord self-message filtering still depends on a resolved botUserId.

## What This Changes

The PR adds Discord managed bot identity tracking and wait-for-idle teardown plumbing, adds SenderManagedAccountId context, extends KeyedAsyncQueue, updates Discord tests, and includes small Telegram/plugin type-surface changes.

## Best Possible Solution

Rebase and narrow the work to a Discord-owned teardown fix that preserves fetchDiscordBotIdentity fail-closed behavior, adds a small wait-for-idle lifecycle primitive to the current createChannelRunQueue path, and splits unrelated Telegram or plugin contract edits into separate PRs if they are still needed.

## Reproduction Assessment

Yes. The high-confidence reproduction for the blocking PR defect is the existing provider test shape where client.fetchUser('@me') rejects: current main rejects before handler creation, while the PR branch catches that error and can continue with botUserId undefined.

## Solution Assessment

No. The idle-drain direction is plausible, but this PR is not the best current fix because it regresses bot identity startup semantics and targets stale queue files; the safer path is a rebased, Discord-only patch on the current shared channel queue seam.
