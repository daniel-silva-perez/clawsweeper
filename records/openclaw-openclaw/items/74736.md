## Summary

Keep open. Current main still collapses bundled channel entry load failures into a flat warning, while this active contributor PR adds a narrow structured lock-timeout signal with focused tests and green PR check runs. The patch looks mergeable from a correctness standpoint, with one low-severity parser edge and a small maintainer judgment point about whether this is still the best timeout surface after recent bundled runtime-deps changes.

## What This Changes

The PR adds lock-timeout detection/parsing to `src/channels/plugins/bundled.ts` so bundled channel entry load failures can emit structured warning metadata, and adds tests for lock-timeout versus ordinary import-error warnings.

## Best Possible Solution

Land or adjust a narrow observability fix that preserves the full runtime-deps timeout diagnostic in the warning message and emits stable structured metadata on the actual bundled timeout path, while leaving broader runtime-deps recovery and gateway deadlock work under the related operator report #73874.

## Reproduction Assessment

Partly. The PR has a high-confidence unit reproduction for the catch branch by mocking `loadChannelPluginModule` to throw the lock-timeout message, but I did not establish a live current-main gateway reproduction for that exact entry-load path because recent code passes `installRuntimeDeps: false` through lazy bundled channel loads.

## Solution Assessment

Mostly yes, with a small correction. The additive metadata fits the existing logger contract and preserves non-timeout fallback behavior, but the parser should use the stable ` (waited=` marker and maintainers should confirm the annotated catch is the right runtime-deps timeout boundary after the recent staging changes.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86

Full review comments:

- **[P3] Parse the lock path using the waited marker:** `src/channels/plugins/bundled.ts:298-299`
  - body: `indexOf(" (")` stops at the first space-parenthesis sequence after the prefix, so a valid install path such as `/tmp/OpenClaw (prod)/...` would emit a truncated `lockDir` while still classifying the timeout. Since this PR adds `lockDir` for structured logging, parse up to the stable ` (waited=` marker or avoid returning a misleading partial path.
  - confidence: 0.84
