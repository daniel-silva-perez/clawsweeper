## Summary

Keep open. Current main has ACP conversation/thread binding and completion announce plumbing, but it does not implement the requested `sessions_spawn({ runtime: "acp", delivery: ... })` path for directing a spawned ACP run to an arbitrary channel conversation or topic. The related reports in the provided context cover ACP hangs/history/heartbeat delivery and do not supersede this narrower API gap.

## Best Possible Solution

Keep open. The best path is a generic, policy-checked ACP `sessions_spawn` delivery design that reuses `DeliveryContext`, `SessionBindingAdapter`/conversation binding, and task-completion delivery semantics. It should define whether live streaming, final announce, or both are supported; validate channel/account/thread permissions; and include tests for schema exposure, ACP run-mode behavior, plugin channel binding, and forbidden cross-channel targets.

## Risks / Open Questions

- Adding arbitrary per-call channel delivery to `sessions_spawn` would broaden where an agent can post, so the implementation needs explicit policy/authorization checks rather than schema plumbing alone.
- There is overlapping existing ACP support through `/acp spawn --bind here`, `--thread`, persistent `bindings[]`, and `subagent_delivery_target` hooks; a fix should reuse those routing and binding contracts instead of creating a parallel channel path.
