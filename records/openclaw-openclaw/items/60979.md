## Summary

Keep open. Current main still lacks the requested per-call ACP delivery target: `sessions_spawn` only exposes the existing ACP-only `resumeSessionId` and `streamTo`, rejects direct channel-target aliases, and run-mode ACP dispatch reports through parent/task completion rather than a caller-specified channel conversation. Related open PRs address parent or thread delivery pieces, but none specifically closes this API gap.

## What This Changes

Request adds a `sessions_spawn({ runtime: "acp", delivery: ... })` option so ACP output can stream or announce directly to a specified Zulip/Discord-style channel conversation or topic.

## Best Possible Solution

Define a generic, policy-checked ACP `sessions_spawn` delivery contract that reuses `DeliveryContext`, conversation binding/session binding adapters, and task-completion delivery instead of adding a parallel channel path. The finished change should update the tool schema, runtime validation, permission checks, tests, and public docs for channel/account/thread behavior.

## Reproduction Assessment

Yes, as a static API-gap reproduction. On current main, inspecting the tool schema and execution path shows no `delivery` field, and passing direct channel target fields would hit the explicit unsupported-parameter guard.

## Solution Assessment

No, not as schema plumbing alone. The requested direction is plausible, but the maintainable solution needs a generic delivery design with authorization, binding resolution, and clear streaming versus completion semantics.
