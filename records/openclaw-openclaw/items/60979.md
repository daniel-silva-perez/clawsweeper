## Summary

Keep open. Current main has ACP conversation/thread binding, `streamTo: "parent"`, and task-completion announce routing, but it still does not implement the requested `sessions_spawn({ runtime: "acp", delivery: ... })` path for sending a spawned ACP run to an arbitrary channel conversation/topic such as the referenced Zulip plugin binding. The existing ClawSweeper comment remains accurate on current main; the related reports cover ACP hangs/history/heartbeat delivery and do not supersede this narrower API gap.

## Best Possible Solution

Keep open. The best path is a generic, policy-checked ACP `sessions_spawn` delivery design that reuses existing delivery context and session binding infrastructure, exposes a typed schema/docs contract, validates channel/account/thread permissions, and adds tests for ACP run-mode delivery, bound plugin channels, schema exposure, and forbidden cross-channel targets. The referenced external Zulip plugin is useful evidence that plugin-side binding can exist; the missing piece is the core per-call delivery API.

## Risks / Open Questions

- Adding arbitrary per-call channel delivery would broaden where an agent can post, so the implementation needs explicit authorization and channel/account/thread policy checks, not schema plumbing alone.
- The design should choose and document whether the new delivery path supports live streaming, final announce, or both, and should reuse existing `DeliveryContext`, session binding, and task-completion routing instead of creating a parallel channel path.
