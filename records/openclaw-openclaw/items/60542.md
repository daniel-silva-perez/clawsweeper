## Summary

Keep open. Current main has adjacent fixes for lifecycle persistence and restart-interrupted running sessions, but I did not find a current-main repair for the specific stale terminal main-session row being older than its transcript; the session init path still reuses fresh-looking rows without comparing transcript freshness, and the closed related PR that targeted this case was not merged.

## What This Changes

The issue reports that a persisted main-session registry row can become terminal but stale relative to its transcript, causing new input from multiple surfaces to stop until that stale row is removed.

## Best Possible Solution

Add a focused recovery path in session initialization, or a small helper it calls, that detects main-session terminal rows whose transcript/lifecycle tail is newer than the registry snapshot and safely rolls the session forward to an input-accepting state. Keep it narrowly scoped to the stale terminal-row shape and preserve existing normal terminal sessions, explicit resets, and running-session restart recovery.

## Reproduction Assessment

Yes. A focused regression can create an `agent:main:main` store row with terminal status and an older updatedAt/endedAt, pair it with a newer transcript containing a later lifecycle/timeout tail, then call `initSessionState` and assert current main reuses/preserves the stale row instead of rotating or reconciling it; I did not run it because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable solution is the targeted main-session stale-row reconciliation described by the closed PR, updated against current main with regression coverage; a broad startup migration or generic session-store sweep would be riskier than fixing the owner path that reuses the row.
