## Summary

Keep open. Current main has lifecycle persistence and locked session-store updates, but I did not find the recovery guard this report asks for: a terminal persisted main-session row is still judged mainly by reset freshness and can be reused without checking whether its transcript advanced later. The related #60642 PR proposed the exact stale terminal-row detection and regression test, but it was closed unmerged.

## What This Changes

Ensure a persisted main-session registry row cannot become stale relative to its transcript and block new input across chat surfaces after restart or recovery.

## Best Possible Solution

Add a narrow, well-tested recovery path in session initialization for the canonical main/direct session: when a persisted terminal row is older than its resolved transcript, rotate to a fresh session instead of reusing the wedged row, while leaving normal subagent/child-session terminal history semantics intact.
