## Summary

Keep this issue open. Current main has partial runtime handling for OpenAI-compatible reasoning deltas, reasoning-only retry recovery, and vLLM/Nemotron visible-content enforcement when thinking is off, but it still lacks the requested benchmark/reporting schema, separate reasoning-aware versus visible-content scores, extraction-source counters, and first-class model/profile policy knobs. The issue discussion also confirms this broader work should remain separate from #74017.

## What This Changes

The issue requests opt-in reasoning-field ingestion, visible final-answer enforcement, extraction-source reporting, model/profile policy fields, and separate `oc llm` scoring/reporting modes for native reasoning model outputs.

## Best Possible Solution

Keep this open for maintainer design. The best end state is a reporting-first measurement layer that records content/reasoning presence, extraction source, compatibility counters, and separate visible-content versus reasoning-aware scores, followed by an explicit runtime/provider-profile contract for opt-in reasoning-field use and visible-content policy.

## Reproduction Assessment

Not applicable as a direct runtime bug; this is a feature/reporting request. The implementation gap is high-confidence from current-main source and docs searches, while reproducing the exact 197/210 versus 151/210 benchmark split requires the reporter's redacted fixtures or an equivalent checked-in dataset.

## Solution Assessment

Unclear until maintainer design review. The proposed reporting-only first step looks like the safest direction, while blindly reading `message.reasoning` in normal runtime paths would not be the best fix because of hidden-reasoning privacy and compatibility concerns.
