## Summary

Keep this issue open. Current main has partial runtime handling for OpenAI-compatible reasoning fields, a reasoning-only visible-answer retry path, and vLLM/Nemotron request shaping, but it does not implement the requested benchmark/report schema, extraction-source counters, separate reasoning-aware versus visible-content scores, or first-class profile policy knobs; the reporter’s follow-up also confirms this is broader product/runtime/reporting work rather than the already closed scorer bug.

## What This Changes

The issue requests reasoning-aware output handling and reporting for native reasoning models, including opt-in extraction from `message.reasoning`, visible-content compatibility scoring, and explicit benchmark/profile metadata.

## Best Possible Solution

Keep this open for maintainer design. A safe first slice would add reporting-only instrumentation for content/reasoning presence, extraction source, and separate visible-content versus reasoning-aware scoring using redacted fixtures, then follow with an explicit provider/profile contract for opt-in reasoning-field use and visible-content enforcement.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/reporting request. We do have high-confidence current-main verification for the missing surfaces via focused source/docs searches and inspection, while the exact benchmark numbers require the reporter’s local artifacts.

## Solution Assessment

Yes, keeping this open for design review is the best current handling. The existing runtime safeguards are useful but do not solve the requested reporting and policy contract, and expanding raw reasoning handling needs privacy review before implementation.
