## Summary

Keep open. Current main has partial runtime/provider handling for reasoning fields, reasoning-only retry/fallback, and vLLM/Nemotron visible-content request shaping, but it still lacks the requested benchmark/reporting schema, extraction-source counters, separate reasoning-aware versus visible-content scores, and first-class profile policy knobs. The discussion also confirms this broader work should remain separate from the closed scorer bug in #74017.

## What This Changes

Request OpenClaw and `oc llm` support for opt-in reasoning-field extraction, visible final-answer compatibility scoring, extraction-source reporting, and model/profile policy for native reasoning models.

## Best Possible Solution

Keep this open for maintainer design. The best path is an opt-in reporting-first slice that records content/reasoning presence and extraction source without raw reasoning by default, produces visible-content versus reasoning-aware versus strict-content score breakdowns, and only then extends provider/profile contracts once privacy and routing semantics are settled.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a feature/reporting request. There is a high-confidence current-main verification path for the missing surfaces through exact search and code/docs inspection, but the exact benchmark percentages require the reporter's local artifacts.

## Solution Assessment

Yes, keeping this open for design is the best current handling. A narrow automated fix would be unsafe because the remaining work crosses report schema, runtime/provider profile policy, routing behavior, and reasoning-data privacy.
