## Summary

Keep this open. Current main has partial runtime handling for OpenAI-compatible reasoning deltas, reasoning-only retry/fallback classification, and vLLM/Nemotron visible-content request shaping, but it still does not implement the requested benchmark/reporting schema, extraction-source counters, separate reasoning-aware versus visible-content scores, or first-class profile policy knobs. The issue discussion also clarifies that this broader product/runtime/reporting work should stay separate from the closed scorer bug in #74017.

## What This Changes

The issue requests reasoning-aware OpenClaw runtime and `oc llm` benchmark/reporting support, including opt-in extraction from `message.reasoning`, visible-content enforcement, extraction-source reporting, and separate reasoning-aware versus visible-content compatibility scores.

## Best Possible Solution

Keep the issue open for maintainer design review. The best path is a reporting-first slice that records content/reasoning presence, extraction source, compatibility counters, and separate visible-content versus reasoning-aware scores with redacted fixtures, followed by an explicit provider/profile contract for opt-in reasoning-field use and visible-content policy.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a feature/reporting request. We do have a high-confidence current-main verification path for the missing surfaces through focused search and code/docs inspection, but the reporter's exact benchmark numbers depend on local artifacts not present in the repository.

## Solution Assessment

Yes, keeping this open for design is the best current handling. Current main's partial runtime safeguards are useful but are not the narrowest complete solution because the remaining work requires explicit reporting semantics and privacy policy before runtime/profile behavior is expanded.
