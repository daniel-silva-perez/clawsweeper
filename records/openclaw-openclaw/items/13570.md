## Summary

Keep open. Current main still lacks the requested `forcePruneRatio` config, schema, docs, and runtime setting, and the context-pruning hook still returns during a fresh cache-TTL window before usage is estimated. The earlier implementation PR #13572 is closed unmerged, while open PR #43238 is adjacent rolling-mode work rather than this thresholded TTL bypass.

## What This Changes

Add an optional `agents.defaults.contextPruning.forcePruneRatio` setting that bypasses cache-TTL pruning when estimated context usage reaches a configured ratio, with schema and test coverage.

## Best Possible Solution

Implement the feature end to end as an additive, unset-by-default cache-TTL bypass: accept `forcePruneRatio` in config types/Zod/generated schema, carry it through effective settings, estimate finite usage before the fresh-TTL early return, bypass TTL only at or above the threshold, document it, and add focused runtime/schema regressions.

## Reproduction Assessment

Yes. Configure cache-TTL pruning with a recent `lastCacheTouchAt` and an over-threshold context; current `extension.ts` returns before calling `pruneContextMessages`, and existing coverage asserts a fresh second cache-TTL pass returns `undefined`.

## Solution Assessment

Yes. The requested additive threshold is a narrower maintainable solution than always-running pruning: it preserves cache-TTL behavior until high usage, while #43238’s rolling mode remains adjacent but not equivalent.
