## Summary

Keep open. Current main still lacks the requested `forcePruneRatio` config/schema/docs/runtime surface, and the cache-TTL hook still exits during a fresh TTL window before context usage is estimated. The earlier implementation PR #13572 is closed unmerged, while #43238 is adjacent rolling-mode work and does not provide a thresholded TTL bypass.

## What This Changes

The issue requests an optional `agents.defaults.contextPruning.forcePruneRatio` setting that bypasses cache TTL when estimated context usage reaches a configured threshold.

## Best Possible Solution

Implement the feature end to end as an additive, unset-by-default cache-TTL bypass: accept `forcePruneRatio` in config types, Zod, generated schema, effective settings, and docs; estimate finite context usage before the fresh-TTL early return; bypass TTL only when usage is at or above the configured threshold; preserve existing TTL behavior otherwise; and add focused schema/runtime regressions.

## Reproduction Assessment

Yes. A code-level reproduction is to configure cache-TTL pruning with `lastCacheTouchAt` inside the TTL window and an over-threshold context; current `extension.ts` returns before `pruneContextMessages`, and the existing TTL reset test asserts the second pass returns `undefined`.

## Solution Assessment

Yes. The requested additive threshold is the narrowest maintainable fix because it preserves cache-TTL behavior until high usage, unlike always-running pruning or the adjacent rolling-mode proposal in #43238.
