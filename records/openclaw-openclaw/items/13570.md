## Summary

Keep open. Current main still has TTL-gated context pruning without the requested high-usage bypass setting, and the related implementation PR is closed unmerged while the open rolling-mode PR does not implement this thresholded config surface.

## What This Changes

Add an optional `contextPruning.forcePruneRatio` setting so cache-TTL pruning can bypass the TTL when estimated context usage reaches a configured threshold.

## Best Possible Solution

Implement the feature end to end as an additive, unset-by-default cache-TTL bypass: accept and document `forcePruneRatio`, carry it through effective settings and runtime setup, estimate finite usage before the fresh-TTL early return, bypass TTL only at or above the configured threshold, and preserve existing TTL behavior otherwise.

## Reproduction Assessment

Yes. A focused code-level reproduction is to configure cache-TTL pruning with `lastCacheTouchAt` inside the TTL window and an over-threshold context; current `extension.ts` returns before `pruneContextMessages`, and the existing TTL reset test asserts the second pass returns `undefined`.

## Solution Assessment

Yes. The requested additive ratio is the narrowest maintainable solution for this issue because it preserves cache-TTL behavior unless context usage is high; the adjacent rolling mode changes trigger timing more broadly and does not provide the requested config knob.
