## Summary

Keep open. Current main still has TTL-gated context pruning without the requested high-usage bypass setting: the config/schema/docs surface has no `forcePruneRatio`, and the context hook returns inside a fresh TTL window before usage is estimated. The closed reference PR #13572 was not merged, and the adjacent rolling-mode PR #43238 does not provide the requested thresholded cache-TTL bypass.

## What This Changes

Add an optional `agents.defaults.contextPruning.forcePruneRatio` setting so cache-TTL context pruning can bypass TTL when estimated context usage reaches the configured threshold.

## Best Possible Solution

Implement the feature end to end as an additive, unset-by-default cache-TTL bypass: validate and document `forcePruneRatio`, carry it through effective settings and runtime setup, estimate finite context usage before the fresh-TTL early return, bypass TTL only when usage meets the configured threshold, and preserve existing TTL behavior otherwise.

## Reproduction Assessment

Yes. Configure cache-TTL pruning with `lastCacheTouchAt` inside the TTL window and an over-threshold context; current `extension.ts` returns `undefined` before `pruneContextMessages`, and the existing TTL reset test already asserts the second pass is skipped inside the fresh TTL window.

## Solution Assessment

Yes. The requested additive ratio is the narrowest maintainable solution for this issue because it preserves cache-TTL behavior unless context usage is high; the rolling-mode direction is broader and does not provide the requested thresholded config surface.
