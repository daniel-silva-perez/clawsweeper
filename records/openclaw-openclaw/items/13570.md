## Summary

Keep open. Current main still has only TTL-gated cache pruning: there is no `contextPruning.forcePruneRatio` config/schema/runtime/docs surface, and the context hook returns during a fresh TTL window before it calls the pruner that estimates context usage. The paired implementation PR is closed unmerged, so this remains a narrow valid enhancement rather than a cleanup close.

## What This Changes

The issue requests an optional `contextPruning.forcePruneRatio` setting so high estimated context usage can bypass cache TTL and prune immediately.

## Best Possible Solution

Add the feature in the context-pruning owner path as an additive config knob: accept and document `agents.defaults.contextPruning.forcePruneRatio`, carry it through effective settings/runtime setup, estimate usage before the fresh-TTL early return, bypass TTL only when known usage is at or above the threshold, preserve existing TTL behavior otherwise, and cover schema plus runtime regressions.
