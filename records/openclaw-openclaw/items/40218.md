## Summary

Keep open. Current main still lacks the PR’s optional stable cron job id and duplicate-id validation behavior, and the supplied PR context shows a recent, scoped implementation candidate rather than obsolete work.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate. The best path is to land an additive cron contract change that accepts optional stable ids through the service, gateway schema, CLI, cron tool flat recovery, docs, and generated clients; preserves UUID fallback for omitted or invalid ids; and rejects duplicates case-insensitively inside the locked add path after focused compatibility review.
