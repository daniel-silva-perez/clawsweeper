## Summary

Keep open. Current main still generates random cron job IDs and rejects an `id` field in the gateway `cron.add` schema, so the PR is not obsolete. The latest PR diff fixes several earlier review points, but it still makes the generated Swift `CronAddParams` initializer source-breaking by adding an optional wire field as a required initializer argument.

## What This Changes

This PR adds optional stable cron job IDs across cron creation, duplicate-id validation, gateway/CLI/agent-tool inputs, docs, tests, and generated Swift gateway models.

## Best Possible Solution

Keep the additive stable cron-id contract, but update the Swift protocol generator so optional initializer parameters default to nil, regenerate both Swift model copies, and then land the feature with the existing cron service/gateway/CLI/tool coverage intact.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Existing Swift source that calls `CronAddParams(name:agentid:...)` without an `id:` argument would fail to compile against the PR diff because `id: String?` is a new required initializer parameter.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed. The TypeScript service and protocol direction is narrow, but the generated Swift API must preserve additive source compatibility by defaulting optional initializer parameters to `nil`.
