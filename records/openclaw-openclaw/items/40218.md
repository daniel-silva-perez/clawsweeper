## Summary

Keep open: current main still lacks stable cron job IDs, and the latest PR head remains source-breaking for Swift clients because the optional `id` field is added to `CronAddParams`' public initializer without a default. The latest diff appears to address the earlier gateway, CLI, tool, id-shape, and duplicate-id concerns, so this needs a small compatibility repair rather than closure.

## What This Changes

This PR adds optional stable cron job IDs across cron creation, duplicate-id validation, gateway/CLI/agent-tool inputs, docs, tests, and generated Swift gateway models.

## Best Possible Solution

Keep the additive stable cron-id contract, but make the generated Swift API source-compatible by defaulting optional initializer parameters to `nil` and regenerating both Swift model copies before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Existing Swift source that calls `CronAddParams(name:agentid:...)` without an `id:` argument would fail after the PR diff because the generated initializer adds `id: String?` as a required parameter.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently proposed. The TypeScript service/protocol direction is narrow, but the Swift generated API should preserve additive compatibility by giving optional initializer parameters default `nil` values.
