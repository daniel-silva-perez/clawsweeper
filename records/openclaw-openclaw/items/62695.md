## Summary

Keep open. Current main has runtime `toolsAllow` support for embedded/cron runs and static subagent tool policies, but `sessions_spawn` still has no per-run allowlist parameter in its schema, no `SpawnSubagentParams` field, and no plumbing from the spawn call into the child `agent` run. The discussion correctly identifies this as separate from the merged cron/lightContext plumbing work.

## Best Possible Solution

Keep this open and implement it as a native `sessions_spawn` API addition: add an optional per-run `toolsAllow` field to the tool schema and `SpawnSubagentParams`, normalize/validate it, pass it through the child `agent` call to the existing embedded runner `toolsAllow` consumer, and add seam tests proving the allowlist narrows spawned subagent tools while remaining constrained by existing static policies.

## Risks / Open Questions

- Implementing this needs clear intersection semantics with existing profile, provider, agent, sandbox, and `tools.subagents.tools` policies so a per-run allowlist cannot add back tools removed by static policy.
- Tests should cover both schema/forwarding and fail-closed behavior when the runtime allowlist resolves to no callable tools.
