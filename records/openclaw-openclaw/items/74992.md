## Summary

Keep open for maintainer review. Current main still has the resolver gap for configured/custom model metadata without complete numeric cost fields, the pinned Pi dependency expects `Model.cost` to exist, and the PR addresses that boundary narrowly with no blocking review finding found in the diff.

## What This Changes

The branch adds resolver-level zero-cost normalization for resolved models, adds a regression test for configured provider models without cost metadata, and switches the skip-Pi-discovery empty registry path to `ModelRegistry.inMemory`.

## Best Possible Solution

Land the narrow resolved-model normalization with the regression test after maintainer validation, keeping the compatibility fix at the resolver boundary rather than changing the public config schema, adding startup migrations, or weakening the pinned Pi model contract.

## Reproduction Assessment

Yes. A configured provider model with `api` but no `cost` follows the inline-match path on current main and is returned from `normalizeResolvedModel` without a complete cost object; the added unit test is a direct reproduction of that path.

## Solution Assessment

Yes. Normalizing cost after provider/runtime/canonical model normalization is the narrowest maintainable fix because it covers inline, dynamic, and configured override sources without changing provider ownership or dependency contracts.
