## Summary

Keep open. Current main has skill gating by OS, binary, environment, config, bundled allowlists, per-agent skill filters, and plugin activation, but it does not parse or evaluate the requested `fallback_for_*` / `requires_*` tool or toolset fields. This is a core skill-loader/spec gap rather than a ClawHub-only skill request, because plugin-published skills still flow through the same loader and there is no current SKILL.md gate based on the effective runtime tool set.

## Best Possible Solution

Keep this issue open. The best path is a focused core design/implementation: extend the OpenClaw skill metadata type and parser, feed effective tool/toolset availability into skill snapshot construction, add eligibility tests for `requires_*` and `fallback_for_*`, update the public Skills docs, and explicitly document snapshot refresh semantics. If maintainers prefer one umbrella, link it under #52861 as a focused subtask rather than closing it as solved.

## Risks / Open Questions

- A correct fix needs a clear definition of `toolset` across core tools, provider-native tools, optional plugin tools, and per-agent tool policy.
- Because OpenClaw snapshots eligible skills per session, maintainers should define whether tool availability changes refresh skills mid-session or only on new sessions.
- Related #52861 tracks broader prompt overhead work, but this issue is a narrower SKILL.md spec/API request and should not be closed unless maintainers intentionally fold it into that canonical tracker.
