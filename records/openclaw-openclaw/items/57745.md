## Summary

Keep open. Current main supports existing skill gates by OS, binaries, environment, config, bundled allowlists, per-agent skill filters, plugin activation, and remote binary eligibility, but it still does not parse or evaluate the requested `fallback_for_*` / `requires_*` tool or toolset fields. This remains a core skill-loader/spec gap because plugin-published skills flow through the same loader, and effective runtime tool availability is not passed into skill snapshot construction.

## Best Possible Solution

Keep this issue open and implement it as a focused core skill-spec change: extend the skill metadata types and frontmatter parser, pass effective tool/toolset availability into skill snapshot construction, add eligibility tests for `requires_*` and `fallback_for_*`, update the Skills docs, and document snapshot refresh semantics. If maintainers prefer one umbrella, link it under #52861 as a focused subtask rather than closing it as solved.

## Risks / Open Questions

- A correct implementation needs a clear definition of `toolset` across core tools, plugin tools, provider-native tools, sandbox/tool policy, owner-only tools, and per-agent restrictions.
- Skill eligibility is snapshotted per session today, so maintainers need to define whether tool availability changes refresh skills mid-session or only on new sessions.
- #52861 may become the canonical umbrella if maintainers explicitly fold this work into it, but it is not currently enough to close this focused spec request.
