## Summary

Keep open. Current main still documents and schemas `approvals.exec.mode` as `session | targets | both`, but the runtime forwarder still trusts the raw configured mode; an invalid string reaches exact `session`/`targets`/`both` branches and produces no approval forwarding targets. The linked #59187 discussion is directly relevant, but that PR is closed unmerged and its described fallback is not present on current main.

## What This Changes

The issue asks OpenClaw to reject or safely handle invalid `approvals.exec.mode` values and make approval forwarding reliable for main, channel, and subagent approval flows.

## Best Possible Solution

Add a defensive runtime mode normalizer in the approval forwarder that accepts only `session`, `targets`, and `both`, falls back to `session` for invalid values, logs a warning, and covers it with a focused regression test. Keep the existing schema validation; if subagent routing still fails afterward, track that as a separate, reproducible channel/subagent bug.

## Reproduction Assessment

Yes for the invalid-mode portion: source inspection gives a high-confidence reproduction path by passing a malformed `approvals.exec.mode` into the forwarder config and observing that neither forwarding branch runs. The subagent-specific portion is unclear because the issue does not include concrete logs or a narrowed failing route.

## Solution Assessment

No, current main is not the best solution because it relies on schema validation but leaves the runtime forwarding consumer fail-open-to-empty for malformed values. The narrow best fix is the #59187-style consumption-point fallback plus a regression test, without changing approval grant semantics.
