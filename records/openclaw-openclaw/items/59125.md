## Summary

Keep open. Current main still lets an invalid `approvals.exec.mode` reach the approval forwarder, where it matches none of the valid routing branches and can produce zero forwarding targets; the linked fix PR was closed unmerged, so this is not implemented or shipped.

## What This Changes

Validate `approvals.exec.mode` so invalid values cannot silently break approval forwarding across main, channel, and subagent-routed workflows.

## Best Possible Solution

Add a consumption-point guard in the approval forwarder that accepts only `session`, `targets`, and `both`, warns on any other value, and falls back to the documented default `session`; add focused tests for invalid strings without changing broader subagent routing policy.

## Reproduction Assessment

Yes for the invalid-mode part: set `approvals.exec.mode` to an invalid string in a forwarder test and current main will skip both routing branches. The intermittent subagent claim is unclear without the logs referenced in the issue.

## Solution Assessment

Yes for the narrow fix: runtime validation in `normalizeMode` is the maintainable fix because schema validation already exists but config value mode and manual edits can bypass it. A broader subagent repair would need separate reproduction evidence.
