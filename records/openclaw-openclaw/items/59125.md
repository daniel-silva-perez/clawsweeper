## Summary

Keep this issue open. Current main has schema/config-write validation for `approvals.exec.mode`, but it does not contain the linked PR’s consumption-point fallback: the approval forwarder still returns the configured mode unchecked and only routes on exact `session`, `targets`, or `both`, so the specific invalid-mode forwarding failure is not fully covered in the forwarder or regression tests. The broader same-chat/channel routing path has useful current coverage, but the invalid-mode bug remains a concrete open fix candidate.

## Best Possible Solution

Keep this issue open and either revive the substance of #59187 or implement the same fix directly: make `normalizeMode` defensively accept only `session`, `targets`, and `both`, log/fallback to `session` for invalid runtime values, and add a regression test in `src/infra/exec-approval-forwarder.test.ts`. If maintainers consider schema/write validation sufficient, add an explicit test proving invalid `approvals.exec.mode` is rejected through the supported config paths before closing.
