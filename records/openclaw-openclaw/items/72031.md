## Summary

Keep open. Current main still unconditionally requires a non-empty API key in the media-understanding image and audio/video auth paths, while Bedrock is documented and tested elsewhere as an `aws-sdk` auth provider that can run without a static key. The linked fix PR #72092 is open, unmerged, and explicitly closes this issue, so the issue should remain open until that PR lands or is superseded.

## What This Changes

The issue requests that Bedrock-backed image and related media-understanding paths accept `auth.mode === "aws-sdk"` without a static API key, matching the chat/simple-completion path.

## Best Possible Solution

Review, land, or supersede #72092. The end state should let the media-understanding image, MiniMax fallback, and audio/video provider auth paths pass an empty API key only when resolved auth mode is `aws-sdk`, avoid storing empty runtime keys, preserve existing errors for other auth modes, and add regression coverage for Bedrock aws-sdk media calls.

## Reproduction Assessment

Yes. The issue body gives concrete steps, and current source shows the failure occurs before any live Bedrock call: an `aws-sdk` auth result with no `apiKey` reaches `requireApiKey`, which throws immediately.

## Solution Assessment

Yes. The narrow maintainable fix is to mirror the simple-completion missing-key allowance in the media-understanding auth resolvers, limited to `aws-sdk`, rather than weakening `requireApiKey` globally.
