## Summary

Keep open. Current main still routes Bedrock `aws-sdk` media auth through unconditional API-key requirements, and the linked closing PR #72092 is still open rather than merged.

## What This Changes

The issue requests Bedrock media-understanding image and audio/video auth paths to accept `aws-sdk` auth without a static API key, matching chat/completion behavior.

## Best Possible Solution

Land or supersede #72092 with the narrow media-understanding fix: allow empty API-key material only for `aws-sdk` auth in the image and audio/video media auth resolvers, avoid storing empty runtime keys, preserve missing-key failures for all other modes, and keep regression coverage for the Bedrock image path plus provider execution auth.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a Bedrock image model with `auth.mode === "aws-sdk"` and no `apiKey`, then call the image tool; current main reaches `requireApiKey` in media auth resolution and throws before pi-ai can use the AWS SDK credential chain.

## Solution Assessment

Yes. Mirroring the chat path's missing-key allowance in the media-specific auth resolvers is the narrow maintainable fix; weakening `requireApiKey` globally would broaden behavior beyond the reported Bedrock `aws-sdk` case.
