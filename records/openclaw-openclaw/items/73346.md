## Summary

Keep this draft PR open. Current main still sends the default `anthropic-beta` header on API-key `anthropic-messages` transport requests without checking provider or custom `baseUrl`, and the repo already supports/documented custom Anthropic-compatible providers. The PR is a plausible focused bug fix with a local regression test, but it should receive normal maintainer review rather than cleanup closure; the proposed endpoint guard also needs tightening before merge.

## Best Possible Solution

Keep the PR open for normal review. The best path is to land a focused transport fix that gates default Anthropic beta headers on an exact direct-Anthropic endpoint classification, adds regression coverage for direct Anthropic, custom provider/baseUrl, and embedded-host/schemeless edge cases, and documents or tests the intended OAuth behavior.

## Risks / Open Questions

- The proposed helper in the PR patch has a hostname-classification edge case for malformed or schemeless base URLs because it falls back to substring matching; it should reuse `resolveProviderEndpoint` or exact hostname parsing before merge.
- The PR patch shown gates the API-key header path only; maintainers should explicitly decide whether OAuth-token/custom-baseUrl requests should retain or suppress the OAuth/default beta bundle.
- No local tests were run because this was a required read-only cleanup review, but source inspection is enough to decide that current main has not already implemented the PR's intended behavior.
