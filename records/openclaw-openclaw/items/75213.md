## Summary

Keep this PR open. Current main still has the reported reliability gap: shared Codex app-server notifications and server requests can update the active run's silence timer before they are scoped to that run's thread and turn, while the PR applies a narrow current-turn gate with matching regression coverage.

## What This Changes

The PR scopes Codex app-server terminal-idle activity tracking to the active thread/turn and adds regression tests for unrelated shared-client notification and server-request traffic.

## Best Possible Solution

Land the narrow scoped-activity fix after maintainer review and terminal CI, keeping the silence timeout tied to the active Codex thread/turn without changing the shared client contract.

## Reproduction Assessment

Yes. On current main, start a Codex app-server turn with the terminal-idle watch armed, then deliver a shared-client notification or server request for another thread/turn just before the timeout; the current handler updates `turnCompletionLastActivityAt` before rejecting the unrelated traffic.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I found: it preserves shared-client fan-out and existing request handlers, but only resets the terminal-idle watch after the traffic is confirmed to belong to the active turn.
