## Summary

Keep open. Current main still resets the Codex app-server terminal-idle activity timer before scoping shared-client notifications and server requests to the active thread/turn, while this unmerged PR is a narrow implementation with regression coverage and no blocking correctness finding found in review.

## What This Changes

The PR changes Codex app-server run-attempt activity tracking so notifications and server requests reset the terminal idle timer only after matching the active thread/turn, and adds regression tests for unrelated shared-client traffic.

## Best Possible Solution

Land the narrow scoped-activity fix after maintainer review, keeping the shared app-server client contract intact while tying the terminal-idle deadline only to traffic for the active Codex thread/turn.

Yes. On current main, start a Codex app-server turn with the terminal-idle watch armed, then deliver a shared-client notification or server request for another thread/turn just before the timeout; current code updates `turnCompletionLastActivityAt` before rejecting that unrelated traffic.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I found because it preserves the shared client fan-out model and scopes only the terminal-idle activity reset to current-turn traffic, with regression tests for both notification and request paths.
