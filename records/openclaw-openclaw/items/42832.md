## Summary

Keep this PR open. Current main still has the core Nostr DM delivery problems the PR targets: the Nostr gateway start task resolves immediately, which the gateway treats as a channel exit/restart condition, and the DM subscription still passes an array-wrapped filter while the pinned nostr-tools 2.23.3 API documents a single `Filter` parameter. The PR is stale against current main and its `sendMedia` fallback would now conflict with the channel's no-media contract, so it should be refreshed rather than closed as obsolete.

## Best Possible Solution

Refresh this PR or land an equivalent Nostr-owned patch on current main: keep `startNostrGatewayAccount()` pending until abort using `runPassiveAccountLifecycle` or equivalent idempotent cleanup, pass the single `Filter` object required by pinned `nostr-tools` 2.23.3 for the DM subscription path, and add focused lifecycle plus subscription-shape regression tests. Leave media unsupported unless a separate public-URL-only media design updates the Nostr channel metadata, docs, and tests. Keep #53858 as the canonical user-facing restart-loop report until the corrected fix lands.

## Risks / Open Questions

- Merging this PR as-is would not apply cleanly to the current lifecycle ownership because the startup implementation now lives in `extensions/nostr/src/gateway.ts`.
- The proposed `sendMedia` fallback would expose arbitrary `mediaUrl` strings in encrypted DM text despite Nostr being declared and documented as media-unsupported; local/private paths could be leaked or delivered as unusable links.
- The subscription-shape test in the PR is mock-based, so a refreshed fix should include a focused contract/regression test that asserts the argument shape against the pinned API expectations where feasible.
