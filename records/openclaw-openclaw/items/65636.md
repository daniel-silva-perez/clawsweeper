## Summary

Keep open. Current main still lacks the requested `cron add --id` path, and open PR #66023 already uses closing syntax for this issue, so the issue should remain open until that implementation is reviewed and merged or replaced.

## What This Changes

Add an optional `openclaw cron add --id <id>` surface so users can create cron jobs with stable human-readable IDs instead of generated UUIDs.

## Best Possible Solution

Review and land a narrow implementation that carries the id through the CLI, protocol schema, normalization, and service create path with a conservative slug format and duplicate-id rejection, or close the linked PR and replace it with an equivalent focused fix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `cron add --id daily-brief` is not registered by the command, direct `cron.add` rejects unknown `id`, and the service currently generates `crypto.randomUUID()` on every create.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if validation is conservative and collisions are rejected in the cron service. The current linked PR appears to target that boundary, so the best next step is PR review rather than closing this issue early.
