## Summary

Keep this open. Current main still treats any truthy OpenAI-compatible `choice.delta.content` as visible text before checking reasoning fields, while Mistral native reasoning can return structured `thinking` and `text` content chunks, so the reported `[object Object]` corruption path remains valid and narrowly fixable.

## What This Changes

The issue requests that Mistral thinking responses stop leaking `[object Object]` into assistant replies, saved session transcripts, and memory extraction.

## Best Possible Solution

Normalize structured Mistral content chunks at the OpenAI-compatible completions stream boundary: send `thinking` chunks to hidden thinking blocks, append only `text` chunks to visible assistant text, and keep corrupted text out of persisted session and memory-ingested content.

## Reproduction Assessment

Yes. The issue has concrete versioned steps, and current-main static inspection gives a high-confidence mocked reproduction path by feeding `processOpenAICompletionsStream` a Mistral-style `delta.content` array containing `thinking` and `text` objects.

## Solution Assessment

Yes. The narrowest maintainable fix is in the stream parser/provider-normalization boundary; filtering memory ingestion or sanitizing `[object Object]` after persistence would leave live replies and saved transcripts corrupted.
