## Summary

Keep open. Current main still matches the reported Slack directory gap: the public directory CLI list commands call only the config-backed directory methods, Slack's token-backed live readers are registered separately, and Slack still inherits the default null self resolver. The related #28103 work fixed Slack user-token sourcing, but it did not bridge the CLI to those live directory readers or add Slack current-account identity resolution.

## What This Changes

The issue asks for Slack `openclaw directory self`, `peers list`, and `groups list` to return token-backed Slack workspace identity, peers, and groups instead of `null` or empty results when Slack auth is valid.

## Best Possible Solution

Add a generic live-aware directory CLI path that prefers live list methods when present while preserving config-backed fallback, then add Slack-owned current-account identity support using the existing directory entry contract.

## Repair Work Prompt

Fix the Slack directory bug reported in https://github.com/openclaw/openclaw/issues/50776. Current main routes `openclaw directory peers list` and `openclaw directory groups list` through config-backed `plugin.directory.listPeers/listGroups`, while Slack exposes token-backed live readers as `listPeersLive/listGroupsLive`; `openclaw directory self` calls `plugin.directory.self`, but Slack inherits the default null resolver. Implement a generic live-aware directory CLI list path that prefers live list methods when present and preserves config-backed fallback. Add Slack-owned current-account identity support using the resolved Slack user token before bot-token fallback and return only the existing `ChannelDirectoryEntry` shape. Keep Slack-specific behavior in `extensions/slack`; do not add bundled Slack ids or raw token/profile leakage to core. Add focused regression tests for CLI live preference/fallback, Slack live directory behavior, and Slack self. Add a single-line active changelog fix with `Thanks @pjaillon`.
