## Summary

Keep open. Current `main` still matches the reported Slack directory failure path: the public directory CLI uses only config-backed peer/group methods, Slack exposes token-backed live directory readers separately, and Slack `self` still falls through to the generic null resolver. The issue is concrete, not maintainer-authored, has no protected label, and is narrow enough for a focused repair PR.

## What This Changes

The issue asks Slack directory CLI commands to return live workspace self, peers, and groups when Slack is connected with valid Slack bot/app/user tokens instead of returning null or empty arrays.

## Best Possible Solution

Add a generic CLI path that can prefer live directory methods when available while preserving config-backed fallback, then add Slack-owned `self` support and regressions for token-backed peers, groups, and current-account identity.

## Repair Work Prompt

Fix the Slack directory CLI bug reported in this issue. Current `openclaw directory peers/groups list --channel slack` only calls config-backed `listPeers`/`listGroups`, even though Slack exposes token-backed `listPeersLive`/`listGroupsLive`; `openclaw directory self --channel slack` also returns null because Slack does not provide `directory.self`. Implement the missing runtime path generically in `src/cli/directory-cli.ts` so live directory methods are used when available while config-backed fallback remains intact for channels that only support config directories. Add Slack-owned `self` support using an appropriate Slack auth/profile source and degrade cleanly when no read token is available. Keep the fix narrow: do not change unrelated Slack token-selection policy, writes, onboarding, provider behavior, or plugin SDK contracts unless an additive seam is strictly necessary. Add focused regression tests for CLI live preference/config fallback and Slack peers/groups/self behavior. Update the changelog under the active release fixes with `Thanks @pjaillon` if user-facing behavior changes.
