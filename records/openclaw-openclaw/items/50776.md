## Summary

Keep open. Current main still matches the reported Slack directory gap: the public directory CLI reads the config-backed peer/group methods, Slack exposes token-backed live readers separately, and Slack still inherits the default null self resolver. Related PR #28103 fixed adjacent Slack user-token sourcing, but it did not wire these CLI commands to live Slack directory reads or add Slack current-account identity.

## What This Changes

The report asks for Slack directory CLI commands to return workspace self, peers, and groups when valid Slack tokens and Slack API auth are available instead of returning empty/null results.

## Best Possible Solution

Make `openclaw directory peers/groups list` prefer live directory methods when a channel adapter provides them while preserving config-backed fallback, and add Slack-owned token-backed self identity support that returns the existing `ChannelDirectoryEntry` shape with focused regressions for peers, groups, and self.

## Repair Work Prompt

Fix the Slack directory CLI gap reported in https://github.com/openclaw/openclaw/issues/50776. Current `openclaw directory peers/groups list --channel slack` uses config-backed directory methods only, even though Slack exposes token-backed `listPeersLive` and `listGroupsLive`; `openclaw directory self --channel slack` returns null because Slack does not override the default self resolver. Implement a narrow fix that makes the generic directory CLI prefer live list methods when a channel adapter provides them while preserving config-backed fallback, and add Slack-owned self identity resolution using the effective Slack read token. Keep the result within the existing `ChannelDirectoryEntry` shape and do not expose raw tokens or extra Slack profile fields. Do not change Slack token write policy, broad channel config semantics, or unrelated channel behavior. Add focused tests for live preference, config fallback, Slack live peer/group use, and Slack self. Include a user-facing changelog fix entry crediting @pjaillon; mention related PR #28103 only as adjacent context, not as the fix.
