## Summary

Current main still has the reported Telegram gap: the Telegram dispatch path does not pass a resolved humanDelay into the shared block dispatcher, while the shared dispatcher only delays later block replies when that option is supplied. The issue should remain open because PR #69022 is still open, unmerged, and explicitly linked as the implementation candidate.

## What This Changes

The issue asks for Telegram streamed block replies to respect `agents.defaults.humanDelay` and per-agent `humanDelay` by wiring `resolveHumanDelayConfig(cfg, route.agentId)` into the Telegram block reply dispatcher.

## Best Possible Solution

Land the existing Telegram-scoped fix path: expose `resolveHumanDelayConfig` through the Telegram runtime barrel, pass it into `dispatchReplyWithBufferedBlockDispatcher`, keep the focused regression test proving the forwarded config, and then let this issue close after the PR merges.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure Telegram block streaming with `humanDelay`, observe that current main never passes `dispatcherOptions.humanDelay`, and the shared dispatcher only sleeps between later block replies when that option is present.

## Solution Assessment

Yes. The narrow PR #69022 approach matches the existing Slack/Discord/Signal pattern and fixes the Telegram plugin boundary without changing the shared dispatcher contract.
