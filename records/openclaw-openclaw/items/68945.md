## Summary

This issue should stay open. Current main still omits Telegram `humanDelay` wiring, the core dispatcher only applies pacing when that option is provided, and open PR #69022 is the active implementation candidate for the same bug.

## What This Changes

The issue requests that Telegram block-streamed replies honor `agents.defaults.humanDelay` and per-agent overrides by passing the resolved delay config into the Telegram block reply dispatcher.

## Best Possible Solution

Keep the fix in the Telegram plugin boundary: expose or import `resolveHumanDelayConfig` through the existing Telegram runtime seam, pass the resolved config into `dispatchReplyWithBufferedBlockDispatcher`, and keep focused regression coverage proving Telegram forwards the configured delay before closing this issue.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with Telegram block streaming, current main never passes `dispatcherOptions.humanDelay`, while the shared dispatcher only sleeps between later block replies when that option is supplied.

## Solution Assessment

Yes. The narrow Telegram-only wiring used by PR #69022 matches the existing Slack/Discord/Signal pattern and avoids changing the shared dispatcher contract; the branch needs maintainer review or refresh because it is still open and dirty.
