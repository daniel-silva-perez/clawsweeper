## Summary

Keep this open. Current main still has size-proportional hot session-store updates: active-session writes reload and parse the whole store, mutate one row, stringify the whole store, and write the whole file while large `skillsSnapshot` and `systemPromptReport` payloads remain inline. The added maintenance controls help stale, over-entry, or explicitly over-disk-budget stores, but they do not solve recent below-cap stores made large by duplicated snapshots.

## What This Changes

The issue requests reducing session-store update latency so large per-agent `sessions.json` files with repeated inline skill snapshots do not consume the 60-second agent run budget and surface as `No reply from agent`.

## Best Possible Solution

Adopt a backward-compatible session metadata design where updating one active session does not scale with all retained sessions. The end state should preserve reads of existing inline rows while writing repeated snapshot-like payloads out of line or moving session metadata to a sharded/database-backed store, with a regression for a large recent store below the entry cap.

## Reproduction Assessment

Yes. The issue provides concrete production file sizes and gateway logs, and current main confirms the full-file parse/stringify/write path; a focused repro can synthesize many recent entries with repeated large `skillsSnapshot` payloads and time a single `persistSessionEntry` or `updateSessionStore` call.

## Solution Assessment

Unclear for the implementation choice. Content-addressed `skillsSnapshot` storage is a plausible narrow first step, but sharding or a database-backed store may be the more maintainable fix if maintainers want to solve this together with #55334 and #69451.
