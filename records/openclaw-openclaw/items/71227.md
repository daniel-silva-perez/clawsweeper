## Summary

Keep this open. Current main still performs size-proportional hot session-store updates: updates re-read and parse the full `sessions.json`, mutate one entry, stringify the whole store, and atomically rewrite it, while large `skillsSnapshot` and `systemPromptReport` payloads remain inline. The newer maintenance controls can prune or cap entries, but they do not remove the per-write O(file size) path or deduplicate repeated snapshots, and related #55334 and #69451 are still open rather than superseding this latency-specific report.

## What This Changes

The issue requests reducing large per-agent `sessions.json` parse/write latency so agent runs do not exhaust the 60-second wait budget, with proposed fixes including snapshot deduplication, TTL/LRU cleanup, sharding or SQLite, or async writes.

## Best Possible Solution

Keep the issue open and choose a backward-compatible session metadata storage design that removes large inline payloads from hot writes and prevents updating one active session from scaling with total retained store size. A maintainable end state should read existing inline entries, preserve current on-disk compatibility, and add regression coverage with repeated large snapshots plus enough retained sessions to prove one update no longer rewrites multi-MB metadata.

## Reproduction Assessment

Yes. The issue provides concrete production sizes and gateway logs, and current main source confirms the same full-file parse/stringify/write path; a focused regression can synthesize repeated `skillsSnapshot` entries and time one session update.

## Solution Assessment

Unclear. The requested content-addressed `skillsSnapshot` storage is a plausible first step, but the durable fix may be per-session sharding or SQLite-backed metadata; the current maintenance controls alone are not the narrowest complete solution for the latency failure.
