## Summary

Keep this open. Current `main` still has the size-proportional session metadata update path reported here: active writes re-read and parse `sessions.json`, mutate one row, then stringify and atomically rewrite the whole store, while `skillsSnapshot` and `systemPromptReport` remain inline on each `SessionEntry`. The existing cache, pruning, capping, and disk-budget maintenance are useful mitigations, but they do not remove the monolithic parse/write cost for hot updates or deduplicate repeated skill snapshots. Related #55334 and #69451 overlap, but they remain open and do not supersede this latency/run-budget-focused report.

## What This Changes

The issue requests fixing large `sessions.json` updates so repeated inline `skillsSnapshot` data and whole-store parse/write latency no longer cause agent waits to time out with no visible reply.

## Best Possible Solution

Keep the issue open and fix the core session metadata storage path with a backward-compatible design. A maintainable first step would deduplicate or content-address repeated `skillsSnapshot` payloads while continuing to read existing inline entries, then add regression coverage showing that updating one active session no longer scales with a large retained `sessions.json`; the broader durable direction is sharded or database-backed metadata plus non-critical-path writes where feasible.

## Reproduction Assessment

Yes. The issue gives a production reproduction with multi-megabyte per-agent stores and current `main` source confirms the size-proportional update path; a focused regression can synthesize many entries with repeated `skillsSnapshot` data and time a single session update.

## Solution Assessment

Unclear for a single implementation choice. Content-addressing `skillsSnapshot` is a plausible narrow first fix, but the best durable solution may be per-session metadata sharding or SQLite-backed storage, so maintainers should choose the storage contract before an automated repair branch rewrites it.
