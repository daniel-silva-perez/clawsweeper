## Summary

Keep this open. Current main still has a size-proportional hot session-store update path: writes re-read the full `sessions.json`, parse it, mutate one entry, stringify the full store, and atomically write the full file while `skillsSnapshot` and `systemPromptReport` remain inline. The existing maintenance controls help stale, over-entry, or explicitly over-disk-budget stores, but they do not address recent below-cap stores made large by duplicated snapshots, which is the latency class reported here.

## What This Changes

The issue asks OpenClaw to stop monolithic `sessions.json` parse/write latency from consuming the agent run budget when per-agent stores grow due to inline session metadata and repeated `skillsSnapshot` payloads.

## Best Possible Solution

Adopt a backward-compatible session metadata design where updating one active session does not scale with all retained sessions. The end state should read existing inline rows, then either store repeated snapshot-like payloads out of line or move session metadata to a sharded/database-backed store, with regression coverage for a large recent store below the entry cap.

## Reproduction Assessment

Yes. The issue provides production file sizes, affected agents, and gateway log timing, and current main confirms the full-file parse/stringify/write path. A focused repro can synthesize many recent entries with repeated large `skillsSnapshot` payloads and time one `persistSessionEntry` or `updateSessionStore` call.

## Solution Assessment

Unclear. Content-addressed `skillsSnapshot` storage is a plausible narrow first step, but maintainers should choose whether to solve this together with #55334 and #69451 through sharding, database-backed storage, async persistence, or snapshot dedupe.
