## Summary

Keep this open. Current main still has the size-proportional session metadata update path: hot writes re-read, parse, mutate, stringify, and atomically rewrite the whole `sessions.json`, while large `skillsSnapshot` and `systemPromptReport` fields remain inline. Related #55334 and #69451 overlap, but both are still open and neither supersedes this latency/run-budget-specific report.

## What This Changes

The issue asks OpenClaw to prevent large per-agent `sessions.json` files from causing run-budget timeouts by reducing whole-file parse/write cost and repeated inline session snapshot data.

## Best Possible Solution

Keep the issue open and choose a backward-compatible session metadata storage design. A maintainable first step would deduplicate repeated `skillsSnapshot` payloads while still reading existing inline entries; the durable direction may be per-session metadata sharding or a database-backed store plus regression coverage showing that updating one active session no longer scales with the retained store size.

## Reproduction Assessment

Yes. The issue provides concrete production stores and logs, and current main source confirms the size-proportional read/parse/write path; a focused regression can synthesize many retained entries with repeated snapshots and time a single session update.

## Solution Assessment

Unclear for one implementation choice. Content-addressed `skillsSnapshot` storage is a plausible narrow first fix, but the best long-term fix may be sharding or SQLite-backed metadata, so maintainer design input should come before a large persistence rewrite.
