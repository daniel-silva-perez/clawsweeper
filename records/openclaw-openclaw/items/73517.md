## Summary

Keep open. The report is actionable and current `main` still has the repeated maintenance lookup shape, but the issue is already paired with open PR #74406 using closing syntax, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Prevent gateway task-registry maintenance from sustaining high CPU on stale `runs.sqlite` data by avoiding repeated backing-session and chat-type lookups during one maintenance pass.

## Best Possible Solution

Land #74406 or an equivalent narrow task-maintenance change that creates a per-pass backing-session lookup context, caches session-store reads and duplicate CLI chat-type derivations, and preserves the fresh recheck around recovery hooks and task changes.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run maintenance, and count repeated `loadSessionStore`/chat-type derivation calls. The exact production database is unavailable, so full live CPU reproduction is less certain.

## Solution Assessment

Yes, the linked per-maintenance-pass cache is the narrowest maintainable direction because it addresses the reported hot path without changing task schema, retention policy, or startup-time migration behavior. Automatic quarantine of task registries would be broader follow-up scope.
