## Summary

Keep open. The report is current and actionable: current main still performs per-task backing-session checks that can reload/cloned session stores and repeat CLI chat-type derivation during maintenance, and open PR #74406 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Prevent Gateway task registry maintenance from repeatedly reloading/cloning session stores and deriving CLI chat types while reconciling stale records from `runs.sqlite`.

## Best Possible Solution

Land #74406 or an equivalent narrow task-maintenance change that creates a per-pass backing-session lookup context, caches session-store reads and duplicate CLI chat-type derivations, and preserves the fresh recheck around recovery hooks and task changes.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run maintenance, and count repeated `loadSessionStore` or chat-type derivation calls. Full production CPU reproduction is less certain without the reporter's private database.

## Solution Assessment

Yes, the linked per-maintenance-pass cache is the narrowest maintainable direction because it addresses the reported hot path without changing the task schema, retention policy, or startup migration behavior. Automatic quarantine of task registries would be broader follow-up scope.
