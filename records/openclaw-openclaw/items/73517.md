## Summary

Keep open. Current main still has the repeated per-task backing-session lookup shape described by the CPU profile, and the issue is already paired with open PR #74406 using closing syntax, so the right cleanup path is to review or land that PR rather than close this issue early.

## What This Changes

The issue asks OpenClaw to prevent gateway task registry maintenance from sustaining high CPU when stale historical task records cause repeated session-store loads and CLI chat-type derivation.

## Best Possible Solution

Land #74406 or an equivalent narrow task-maintenance change that caches backing-session lookups and duplicate CLI chat-type derivation within one maintenance pass while preserving the fresh recheck around recovery hooks and task changes.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run maintenance, and count repeated loadSessionStore/chat-type derivation calls. The exact production CPU saturation requires the reporter's private historical task database or a sanitized equivalent.

## Solution Assessment

Yes, the linked per-maintenance-pass cache direction is the narrowest maintainable solution because it addresses the reported repeated synchronous work without changing task schema, retention policy, startup migrations, or user config.
