## Summary

Keep open. Current main still has the repeated task-maintenance lookup shape described in the CPU profile, and open PR #74406 is already linked as the intended fix, so this issue should remain open until that PR or an equivalent change lands.

## What This Changes

The issue asks OpenClaw to prevent gateway task registry maintenance from becoming CPU-bound when stale task records repeatedly trigger session-store loads and CLI session chat-type derivation.

## Best Possible Solution

Review and land #74406, or an equivalent narrow task-maintenance change, so each maintenance pass reuses backing-session store reads and duplicate CLI chat-type derivations while preserving the fresh recheck around recovery hooks and changed tasks. Broader malformed-database quarantine belongs in the separate task-registry corruption track such as #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share the same session store or channel session key, run maintenance, and count repeated `loadSessionStore` and `deriveSessionChatType` calls. The full production CPU profile still depends on the reporter's original private task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass cache direction is the narrowest maintainable fix for the reported hot path because it avoids task schema, retention, startup repair, and automatic database quarantine changes. The safer path is normal review of #74406 rather than starting a second repair branch.
