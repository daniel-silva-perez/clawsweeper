## Summary

Keep open. Current main still has the repeated task-maintenance lookup shape from the reported CPU profile, and the linked closing PR is still the active implementation candidate rather than a merged fix.

## What This Changes

The issue asks OpenClaw to prevent task registry maintenance from sustaining high CPU on stale or large `runs.sqlite` state by avoiding repeated session-store loads, duplicate chat-type derivation, and/or unsafe handling of pathological task registries.

## Best Possible Solution

Review and land PR #74406 or an equivalent narrow task-maintenance change that reuses backing-session store reads and duplicate CLI chat-type derivations within one maintenance pass while preserving the fresh recheck around recovery hooks and changed tasks. Broader malformed-database quarantine should remain in the separate task-registry corruption track such as #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and `deriveSessionChatType` calls. The exact production CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass cache direction is the narrowest maintainable fix for the reported hot path because it avoids task schema, retention, startup repair, and automatic database quarantine changes. Current main's batch yielding may improve responsiveness, but it does not eliminate the repeated lookup/clone cost.
