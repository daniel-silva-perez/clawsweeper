## Summary

Keep open. Current `main` still has the repeated task-maintenance lookup shape described in the CPU profile, and open PR #74406 already targets this issue with a narrow implementation, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that gateway task registry maintenance can burn CPU on stale `runs.sqlite` data by repeatedly loading/cloning session state and resolving CLI session metadata during stale-task checks.

## Best Possible Solution

Land #74406 or an equivalent narrow task-maintenance change that reuses backing-session store reads and duplicate CLI chat-type derivations within a maintenance pass while preserving the fresh recheck around recovery hooks and changed tasks. Broader malformed-database quarantine should stay in the separate task-registry corruption track such as #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share the same session store or channel session key, run maintenance, and count repeated `loadSessionStore` and `deriveSessionChatType` calls. The full production CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the per-maintenance-pass cache direction in #74406 is the narrowest maintainable fix for the reported hot path because it avoids task schema, retention, startup repair, and automatic database quarantine changes. The safer path is normal review of that PR rather than starting a second repair branch.
