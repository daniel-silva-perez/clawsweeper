## Summary

Keep this issue open. Current main still exposes only current-segment `chat.history`: the protocol has no logical/current scope parameter, the handler reads the active `sessionId` transcript only, and rollover code archives the previous transcript instead of linking it into a logical history response.

## Best Possible Solution

Implement this as an additive Gateway protocol change, likely `chat.history({ sessionKey, scope: "current" | "logical" })`, where `current` preserves todayâ€™s behavior and `logical` returns ordered segment metadata plus bounded, normalized messages across reset archives for the same logical session key. Update WebChat/Control UI consumers and add focused tests for manual reset, idle reset, daily reset, missing/pruned archives, and remote Gateway behavior; coordinate naming and remaining archive-access work with #45003 rather than closing this issue now.

- Logical history must preserve existing `chat.history` redaction, display-normalization, byte-budget, and operator-scope contracts across multiple transcript segments.
- Reset archives can be pruned by `session.maintenance.resetArchiveRetention`, so any implementation needs explicit behavior when older segments are no longer available.
