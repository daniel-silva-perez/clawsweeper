## Summary

Keep open. Current main still uses a single `launchctl bootstrap` path, so the PR is not obsolete, but the proposed legacy `launchctl load -w` fallback is not mergeable as-is because it can report success outside the intended per-user LaunchAgent domain and without proving registration/startup.

## What This Changes

This PR modifies launchd LaunchAgent bootstrap to retry after explicit already-loaded errors, fall back to `launchctl load -w` for GUI-domain failures, and adds unit tests for those paths.

## Best Possible Solution

Keep the duplicate-registration retry direction only if it is tied to a verified duplicate-load signal, and gate any fallback on wrong-user/root rejection, legacy load output parsing, post-load registration/runtime verification, explicit session semantics, and live macOS validation.

## Reproduction Assessment

Yes for source-level behavior: current main has a single bootstrap attempt, and the PR tests simulate duplicate-registration plus missing-GUI-domain failures. The blocking fallback risks are reproducible by inspection against the launchctl legacy contract and need live macOS confirmation before shipping.

## Solution Assessment

No. The retry portion is plausible, but the unconditional legacy load fallback is not the narrowest maintainable fix without root guards, load-output validation, registration checks, and a documented headless session model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate legacy load output before returning success:** `src/daemon/launchd.ts:230-231`
  - body: The fallback returns as soon as `launchctl load -w` exits 0, but the legacy load/unload contract returns zero except for improper usage and can report real plist or registration failures in output. Parse stdout/stderr or verify the service is registered before reporting install success.
  - confidence: 0.9
- **[P1] Reject root legacy load fallback:** `src/daemon/launchd.ts:229-231`
  - body: This fallback also runs after sudo-triggered `gui/0` bootstrap failures. Legacy launchctl targets the system domain when run as root, so this can report success while leaving the intended per-user LaunchAgent uninstalled.
  - confidence: 0.86
- **[P2] Prove the fallback starts a headless-capable agent:** `src/daemon/launchd.ts:229-231`
  - body: The PR uses `launchctl load -w` without `-S Background`, and the generated plist has no Background session policy. In SSH/headless contexts the load path can accept the plist while the agent still waits for a GUI Aqua session, contradicting the claimed headless install success.
  - confidence: 0.8
