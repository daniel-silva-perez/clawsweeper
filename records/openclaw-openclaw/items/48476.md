## Summary

Keep open. Current main does not implement the requested retry/fallback behavior, but the PR should not merge as-is because the legacy `launchctl load -w` fallback can report success in the wrong or unverified launchd domain.

## What This Changes

This PR adds retry-after-bootout handling for already-bootstrapped `launchctl bootstrap` failures, a legacy `launchctl load -w` fallback for GUI-domain bootstrap failures, and launchd install tests for those paths.

## Best Possible Solution

Keep the duplicate-registration retry direction, but only ship a fallback after it rejects sudo/root wrong-domain success, parses or verifies legacy load results, defines the intended headless session type, and confirms the service is registered/running in the intended domain before reporting success.

## Reproduction Assessment

Yes for source-level behavior: current main has a single bootstrap attempt, and the PR tests model duplicate-registration plus missing-GUI-domain failures. The real `launchctl load` exit/output, root-domain, and session-type semantics still need live macOS validation.

## Solution Assessment

No, not as currently proposed. The retry piece is narrow and plausible, but the unconditional legacy `load -w` success path is not the safest maintainable fix without output validation, root/sudo guarding, explicit session semantics, and post-load verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Validate load output before returning success:** `src/daemon/launchd.ts:231-232`
  - body: Legacy `launchctl load` can report load or registration failures in stdout/stderr while still exiting 0. Returning here on `load.code === 0` can make `openclaw gateway install` report success even when the agent was not registered.
  - confidence: 0.9
- **[P1] Reject root legacy load fallback:** `src/daemon/launchd.ts:230`
  - body: When the GUI bootstrap failure comes from running under sudo, this fallback invokes legacy `launchctl load -w` as root. That can target the root/system context instead of the intended per-user LaunchAgent domain, turning the existing wrong-user guidance into a misleading success path.
  - confidence: 0.86
- **[P2] Verify a headless-capable session before success:** `src/daemon/launchd.ts:230-232`
  - body: The fallback uses `launchctl load -w` without `-S Background`, while the generated plist has no Background session policy. In SSH/headless contexts this can accept the plist but leave the agent waiting for GUI login, contradicting the claimed headless success path.
  - confidence: 0.8
