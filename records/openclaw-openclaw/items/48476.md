## Summary

Keep this PR open. Current main still performs a single `launchctl bootstrap` for LaunchAgent activation and throws on unsupported GUI domains, with no duplicate-registration retry and no `launchctl load -w` fallback. The PR is relevant to #8619, but the fallback changes macOS launchd session/domain behavior and needs maintainer review before merge.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it onto current main, preserve or split out the duplicate-registration retry if maintainers agree it is safe, and only keep the headless/sudo `launchctl load -w` fallback if it verifies the actual loaded domain/session and parses `load` diagnostics before reporting success. If headless LaunchAgent fallback is out of scope, narrow this PR to reinstall recovery and handle headless/sudo as documented guidance or a separate LaunchDaemon design discussion.

## Risks / Open Questions

- The `launchctl load -w` fallback changes per-user LaunchAgent session/domain behavior; accepting exit code 0 without parsing diagnostics or verifying registration can report a successful install when the service is not actually loaded for the intended user.
- Sudo/root invocation is especially sensitive: current code derives `gui/<uid>` from `process.getuid()` and paths from `HOME`, while legacy launchctl subcommands can target the system/root context, so the fallback may hide the current wrong-user guidance.
- The duplicate-registration retry is plausibly useful, but the headless/sudo fallback may need to be split into a separate LaunchDaemon or verified-background-session design instead of merged as a small launchd helper change.
- This PR overlaps adjacent launchd bootstrap/restart resilience work such as #43443, so maintainers should decide whether to merge, narrow, or consolidate the retry behavior.
