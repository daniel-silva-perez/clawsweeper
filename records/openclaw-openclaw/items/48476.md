## Summary

Keep this PR open. Current main still lacks the intended launchd install resilience, but the PR is not merge-ready because the legacy `launchctl load -w` fallback can report success without proving the intended user LaunchAgent was registered or started.

## What This Changes

This PR changes launchd install activation to retry `bootstrap` after explicit already-loaded errors, fall back to `launchctl load -w` for unsupported GUI domains or missing-domain text, and adds launchd install tests for those paths.

## Best Possible Solution

Keep the duplicate-registration retry as a narrow launchd install repair, but only ship a headless fallback if it parses `launchctl load` diagnostics, avoids sudo/root wrong-domain success, and verifies the loaded service in the intended domain/session. If that cannot be made reliable, leave headless macOS Gateway installs as documented GUI-session or LaunchDaemon work.

## Reproduction Assessment

Yes for the source-level behavior: current main has a single bootstrap attempt, and the PR tests can model duplicate-registration and missing-GUI-domain failures. Live macOS validation is still required for the real `launchctl load` domain/session semantics.

## Solution Assessment

No, not as currently proposed. The retry direction is narrow, but the unconditional legacy `load -w` fallback is not the safest fix until it verifies load output, target domain, and startup state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Parse load output before returning success:** `src/daemon/launchd.ts:241-242`
  - body: Legacy `launchctl load` can print real plist or registration failures while still returning exit code 0. Returning immediately on `load.code === 0` can make `openclaw gateway install` report success even when the LaunchAgent was not registered.
  - confidence: 0.9
- **[P1] Guard the root legacy load fallback:** `src/daemon/launchd.ts:240`
  - body: When the GUI bootstrap failure comes from running under sudo, this fallback runs legacy `launchctl load -w` as root. The legacy subcommand targets the system domain in that context, so the install can succeed for the wrong domain/user instead of preserving the current wrong-user guidance.
  - confidence: 0.86
- **[P2] Verify a headless-capable session before success:** `src/daemon/launchd.ts:240-242`
  - body: `launchctl load` defaults to the Aqua session, and the generated Gateway plist does not set `LimitLoadToSessionType`. In SSH/headless installs this can accept the plist but leave the agent waiting for a GUI login, which contradicts the fallback's claimed headless success path.
  - confidence: 0.8

## Security Review

Status: needs_attention
