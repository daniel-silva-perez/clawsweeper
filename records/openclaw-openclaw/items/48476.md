## Summary

Keep this PR open. Current main still does not implement the PR's intended launchd install resilience: `bootstrapLaunchAgentOrThrow()` performs a single `launchctl bootstrap` against `gui/<uid>` and immediately throws GUI-session guidance for unsupported GUI domains, with no duplicate-registration retry and no `launchctl load -w` fallback. The PR is small and relevant, but the proposed legacy `launchctl load -w` fallback changes macOS launchd domain/session behavior and still needs maintainer judgment.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it onto current main, preserve or split out the duplicate-registration retry, and only keep the headless/sudo `launchctl load -w` fallback if it verifies the actual loaded domain/session and parses load diagnostics before reporting success. If maintainers decide headless LaunchAgent fallback is out of scope, narrow this PR to reinstall recovery and leave headless/sudo behavior as documented guidance or a separate LaunchDaemon design discussion.

## Risks / Open Questions

- The PR's `launchctl load -w` fallback may report success in SSH/headless/sudo contexts while registering outside the intended `gui/<uid>` user LaunchAgent domain.
- Legacy `launchctl load` can return zero despite actionable diagnostics, so accepting exit code 0 alone risks false successful installs.
- The low-risk duplicate-registration retry and the broader headless/sudo fallback may need to be split so reinstall recovery can ship without expanding the supported macOS headless boundary.
