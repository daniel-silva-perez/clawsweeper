## Summary

Keep open. Current main still has only a single `launchctl bootstrap gui/<uid>` attempt for install activation, so the retry work is not implemented on main, but this PR should not merge as-is because the new legacy `launchctl load -w` fallback can report success without proving registration, can target the wrong root/system domain under sudo, and does not verify a headless-capable launchd session.

## What This Changes

This PR changes macOS launchd activation to retry explicit already-loaded bootstrap failures, fall back to `launchctl load -w` for unsupported GUI-domain errors, and adds launchd install tests for those paths.

## Best Possible Solution

Keep the duplicate-registration retry as a narrow launchd install repair if it can be proven against current macOS output, but only ship a headless fallback if it avoids sudo/root wrong-domain success, parses legacy `load` diagnostics, selects or declares the intended session type, and verifies the service is registered/running in the intended domain before reporting success. Otherwise, keep headless macOS Gateway installs on the documented logged-in GUI session or custom LaunchDaemon path.

## Reproduction Assessment

Yes for source-level behavior: current main has a single bootstrap attempt, and the PR tests model duplicate-registration and missing-GUI-domain failures. Live macOS validation is still needed for the real `launchctl load` exit/output, root-domain, and session-type semantics.

## Solution Assessment

No, not as currently proposed. The retry direction is narrow and useful, but the unconditional legacy `load -w` success path is not the safest solution without output parsing, root/sudo guarding, and post-load verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Parse load output before returning success:** `src/daemon/launchd.ts:230-231`
  - body: Legacy `launchctl load` can report plist or registration failures in stdout/stderr while still returning exit code 0. Returning on `load.code === 0` can make `openclaw gateway install` report success even when the LaunchAgent was not registered.
  - confidence: 0.9
- **[P1] Guard the root legacy load fallback:** `src/daemon/launchd.ts:229`
  - body: When the GUI bootstrap failure comes from running under sudo, this fallback runs legacy `launchctl load -w` as root. Legacy launchctl subcommands target the system domain in that context, so the install can succeed for the wrong domain/user instead of preserving the wrong-user guidance.
  - confidence: 0.86
- **[P2] Verify a headless-capable session before success:** `src/daemon/launchd.ts:229-231`
  - body: `launchctl load` defaults to the Aqua session, and the generated Gateway plist does not set `LimitLoadToSessionType`. In SSH/headless installs this can accept the plist but leave the agent waiting for GUI login, contradicting the fallback's claimed headless success path.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The PR changes macOS service registration in a daemon path and can turn sudo/root invocation into a misleading success path outside the intended per-user LaunchAgent domain.
