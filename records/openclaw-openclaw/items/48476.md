## Summary

Keep open. Current main still has the single `launchctl bootstrap gui/<uid>` install path and the PR addresses a real macOS LaunchAgent install surface, but the proposed legacy `launchctl load -w` fallback is not merge-ready because it can report success without validating legacy load diagnostics, can run in the wrong root/system scope under sudo, and does not prove the service is usable in a headless session.

## What This Changes

This PR changes macOS LaunchAgent activation to retry explicit already-loaded bootstrap failures, fall back to legacy `launchctl load -w` for GUI-domain failures, and adds launchd install tests for those paths.

## Best Possible Solution

Keep the duplicate-registration retry as a narrow launchd install repair if it is backed by current macOS output, but only ship a headless fallback after it rejects sudo/root wrong-domain success, handles legacy load diagnostics, chooses or documents the intended session type, and verifies the service is registered/running in the intended domain before reporting success.

## Reproduction Assessment

Yes for source-level behavior: current main has a single bootstrap attempt and the PR tests model duplicate-registration plus missing-GUI-domain failures. Live macOS validation is still needed for the real `launchctl load` exit/output, root-domain, and session-type semantics.

## Solution Assessment

No, not as currently proposed. The retry direction is narrow and useful, but the unconditional legacy `load -w` success path is not the safest maintainable fix without output parsing, root/sudo guarding, session semantics, and post-load verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Validate load output before returning success:** `src/daemon/launchd.ts:231-232`
  - body: Legacy `launchctl load` can print load or registration failures while still returning exit code 0. Returning here on `load.code === 0` without inspecting stdout/stderr or verifying the service can make `openclaw gateway install` report success when the agent was not registered.
  - confidence: 0.9
- **[P1] Reject root legacy load fallback:** `src/daemon/launchd.ts:230`
  - body: When the GUI bootstrap failure comes from running under sudo, this fallback invokes legacy `launchctl load -w` as root. Legacy launchctl targets the system domain in that context, so this can succeed for the wrong domain instead of preserving the wrong-user guidance for the intended per-user LaunchAgent.
  - confidence: 0.86
- **[P2] Verify a headless-capable session before success:** `src/daemon/launchd.ts:230-232`
  - body: `launchctl load` defaults to the Aqua session, and the generated Gateway plist does not set `LimitLoadToSessionType`. In SSH/headless installs this can accept the plist but leave the agent waiting for GUI login, contradicting the fallback's claimed headless success path.
  - confidence: 0.8

## Risks / Open Questions

- The fallback accepts `launchctl load -w` exit code 0 without parsing legacy diagnostics or verifying registration, so install can report success when the agent was not loaded.
- When invoked through sudo/root, the fallback can run legacy launchctl in root/system scope instead of preserving the current wrong-user guidance for a per-user LaunchAgent.
- The headless fallback does not select or verify a Background session and the generated plist has no Background session policy, so it may accept the plist while leaving the gateway unavailable until GUI login.
- No live macOS proof is provided for the proposed legacy load behavior across desktop, SSH/headless, and sudo/root contexts.
