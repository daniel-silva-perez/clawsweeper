## Summary

Keep open. Current main still conflates fast log growth with a reset, while this PR is a focused fix with matching regression tests and successful CI; the remaining step is maintainer review and merge/changelog judgment.

## What This Changes

The PR removes `reset = true` from the fast-growth branch of `readLogSlice` and adds `log-tail` unit coverage for fast growth versus real file shrink/rotation.

## Best Possible Solution

Land the semantic split so `reset` is reserved for cursor-past-EOF/file shrink cases and `truncated` covers byte-budget skips during fast growth, with the new regression tests kept in place.

## Reproduction Assessment

Yes. The focused path is to call `readConfiguredLogTail`, append more than `maxBytes` while keeping the prior cursor valid, then call it again; current main sets both `truncated` and `reset`, which the CLI renders as both truncation and rotation notices.

## Solution Assessment

Yes. The PR uses the narrowest maintainable fix: it changes only the incorrect fast-growth flag assignment and adds tests for both affected branches without changing schema or downstream strings.
