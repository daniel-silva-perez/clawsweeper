## Summary

Keep this PR open. Current main still conflates fast log growth with true file shrink/rotation: `readLogSlice` sets `reset` in the `size - start > maxBytes` branch, and the CLI renders any `reset` as `Log cursor reset (file rotated).` The PR is a narrow, relevant fix with focused tests, and the provided PR diff shows no workflow, dependency, package, secret, permission, network, artifact-download, or code-execution surface change.

## What This Changes

The PR removes `reset = true` from the fast-growth branch of `readLogSlice` and adds log-tail tests for fast growth versus real shrink/rotation behavior.

## Best Possible Solution

Review and land this PR if CI stays green. The best fix is the PR's narrow semantic split: keep `reset` true only when the previous cursor is past EOF, keep `truncated` true for byte-budget skips, and keep regression tests for both fast growth and real shrink/rotation.
