## Summary

Keep open. Current main still sets `reset` during fast log growth and the CLI still renders `reset` as the file-rotation notice, so the PR addresses a real remaining bug. The patch is narrow, adds focused regression coverage for fast growth versus file shrink, and I found no discrete correctness or security blocker in the diff.

## What This Changes

The PR removes `reset = true` from the fast-growth branch of `readLogSlice` and adds two `log-tail` unit tests for fast growth versus actual file shrink/rotation.

## Best Possible Solution

Land the narrow semantic split: keep `reset` reserved for a cursor past EOF/file shrink, use `truncated` for byte-budget skips during fast growth, and keep regression tests for both cases.
