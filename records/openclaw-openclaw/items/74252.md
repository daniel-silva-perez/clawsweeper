## Summary

Current main still has the reported behavior: `readLogSlice` sets `reset = true` when a valid cursor falls more than `maxBytes` behind a growing log file, and the CLI renders any `payload.reset` as `Log cursor reset (file rotated).` The PR is a narrow, relevant fix with focused regression tests, is not implemented on main, and is not superseded by another open item found in exact-title/search checks. Security review found only `src/logging/log-tail.ts` and `src/logging/log-tail.test.ts` changed, with no workflow, dependency, package, secret, permission, network, artifact-download, or code-execution surface change.

## Best Possible Solution

Keep this PR open for normal maintainer review and landing if CI stays green. The best fix is the PR's narrow semantic split: leave `reset` true only when the previous cursor is past EOF, keep `truncated` true for byte-budget skips, and keep the focused regression tests for fast growth and real shrink/rotation. Because this changes a user-visible CLI notice, maintainers should decide whether a one-line changelog Fixes entry is warranted before merge.
