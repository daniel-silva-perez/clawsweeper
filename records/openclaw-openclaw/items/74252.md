## Summary

Keep open. Current main still reports `reset` during fast-growth truncation, and the CLI still maps any `reset` payload to the user-visible file-rotation notice. The PR is narrow and appears aligned with the observed bug, but it is not implemented on main and should go through normal maintainer review and CI rather than being closed.

## What This Changes

The PR removes the fast-growth `reset = true` assignment in `readLogSlice` and adds unit coverage for fast-growth truncation versus real file shrink/rotation.

## Best Possible Solution

Land the narrow semantic split after review and validation: `reset` should mean the previous cursor is past EOF, while `truncated` should report byte-budget skips, with regression coverage for both paths.
