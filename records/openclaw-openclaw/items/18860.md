## Summary

Keep this PR open. Current main still does not implement `after_tools_resolved` or an equivalent plugin-facing API for discovering the final per-attempt tool list and schemas, and the latest provided maintainer comment explicitly asks to keep the PR open for follow-up. The PR remains a plausible implementation path, but it still needs maintainer security/product review around exposing full tool metadata and passing schema objects by reference.

## Best Possible Solution

Keep this PR open as the active implementation/design path. A maintainer-quality resolution is to review and either land an additive resolved-tools hook with SDK baseline, docs, and focused tests, or explicitly supersede it with a pull-based `getResolvedTools` API that safely exposes final per-attempt schemas.

## Risks / Open Questions

- Security review pass: the PR touches plugin API, hook execution, agent attempt tool resolution, docs, tests, and generated SDK baseline files; it does not add workflows, dependencies, lockfile changes, package scripts, or external code execution paths in the provided diff.
- The proposed hook would expose full post-policy tool names and provider-facing parameter schemas to loaded plugins, including tools that are never called. Maintainers should confirm this fits the trusted-plugin boundary.
- The PR body and tests say `parameters` are passed by reference. If the hook runs before provider/session use of those schema objects, accidental plugin mutation could become a behavioral or integrity risk unless the implementation clones, freezes, or clearly constrains timing.
