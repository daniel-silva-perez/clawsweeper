## Summary

Keep this issue open. Current main has partial mitigations, but it still does not implement the requested per-job cron elevated flag or heartbeat provenance/explicit elevated marker. Cron public types, gateway schemas, agent tool docs, CLI/UI/docs surfaces, and heartbeat runtime still rely on provider-class allowlists such as `cron-event` and `heartbeat` rather than per-job/per-file trust metadata.

## Best Possible Solution

Keep this open and implement an additive trust model: add a cron job elevated/trust field to `CronJob`, gateway schemas/validators, persistence normalization, cron tool, CLI/UI surfaces, docs, and tests; restrict setting or changing it to owner/elevated-approved senders; and add heartbeat provenance or an explicit heartbeat elevated marker so `HEARTBEAT.md`-driven runs do not inherit broad internal-event elevated access by wildcard alone.
