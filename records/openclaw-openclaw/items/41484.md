## Summary

Keep this issue open. Current main still has provider/sender elevated allowlists and some cron/heartbeat hardening, but it does not implement a persisted per-job cron elevated flag, an approval rule for changing that flag, or heartbeat provenance/explicit elevated metadata.

## Best Possible Solution

Keep this open for maintainer/secops design. The best path is an additive trust model: persist cron job elevated/trust metadata, gate setting or changing it on an approved owner/elevated sender rather than the internal `cron-event` wildcard, thread it through cron normalization/storage/protocol/CLI/UI/agent tool/docs/tests, and add heartbeat provenance or an explicit heartbeat elevated marker so HEARTBEAT.md-driven runs do not rely on broad internal event allowlists.
