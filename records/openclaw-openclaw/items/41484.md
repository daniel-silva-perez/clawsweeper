## Summary

Keep this issue open. Current main still has sender/provider elevated allowlists and some defensive cron/heartbeat mitigations, but it does not implement the requested persisted per-job elevated flag, approval rule for setting that flag, or heartbeat provenance/explicit elevated marker.

## Best Possible Solution

Keep this open for maintainer/secops design. The best path is an additive trust model: add persisted cron job elevated/trust metadata, gate setting or changing it on an approved owner/elevated sender rather than the internal `cron-event` wildcard, thread it through cron normalization/storage/protocol/CLI/UI/agent tool/docs/tests, and add heartbeat provenance or an explicit heartbeat elevated marker so HEARTBEAT.md-driven runs do not rely on broad internal event allowlists.
