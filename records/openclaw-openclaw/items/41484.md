## Summary

Keep this issue open. Current main has useful partial mitigations, especially owner-only cron tool exposure and isolated cron sessions that do not copy stored elevatedLevel, but it still does not implement the requested per-job/per-heartbeat elevated trust boundary. Cron job schemas, protocol validators, CLI/tool surfaces, and docs have no elevated flag or approval provenance, while elevated availability is still resolved from broad provider allowlists such as heartbeat or cron-event.

## Best Possible Solution

Keep the issue open and implement an additive trust model: add a cron job elevated/trust field to CronJob, gateway schemas, store normalization, CLI/tool/UI surfaces, docs, and tests; restrict setting or changing it to owner/elevated-approved senders; and add heartbeat provenance or an explicit heartbeat elevated marker so HEARTBEAT.md-driven runs do not inherit broad internal-event elevated access by wildcard alone.
