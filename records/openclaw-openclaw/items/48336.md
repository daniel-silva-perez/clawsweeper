## Summary

Keep this PR open. Current main still does not ship a general per-IP Gateway HTTP request limiter: it only has failed-auth throttling and a separate control-plane write throttle. The PR remains a plausible implementation candidate, but it is not safe to close or merge as-is because its submitted HTTP integration keys by the raw socket address before trusted-proxy client-IP resolution, which undermines the reverse-proxy use case it is meant to protect.

## Best Possible Solution

Keep the PR open for maintainer follow-up or explicit consolidation with #44884. The best implementation path is to add an operator-facing `gateway.requestRateLimit` config/schema/docs surface if maintainers want this configurable, instantiate and dispose the limiter in Gateway startup, enforce it after `resolveRequestClientIp(req, trustedProxies, allowRealIpFallback)`, and add regression tests for direct clients, trusted reverse proxies, loopback proxy sources, forwarded headers, and 429 Retry-After behavior.

## Risks / Open Questions

- Merging the PR as submitted would leave common same-host reverse-proxy deployments unprotected because loopback proxy traffic is exempted before the real client IP is resolved.
- For non-loopback reverse proxies, the submitted implementation can make unrelated clients share one limiter bucket and cause cross-user 429s.
- Closing this PR as implemented would be inaccurate while current main lacks the module, config surface, docs, startup wiring, and HTTP enforcement for a general per-IP request limiter.
- The overlapping public-network hardening PR #44884 and linked security issue #20373 are still open in the provided GitHub context, so they do not prove this work has shipped or been superseded by a landed implementation.
