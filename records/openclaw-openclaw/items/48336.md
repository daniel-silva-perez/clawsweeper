## Summary

Keep this PR open. Current main still does not implement the proposed general per-IP Gateway HTTP request limiter, and the PR remains a plausible implementation candidate for the open Gateway hardening work. It is not merge-ready as submitted because the limiter keys on the raw socket peer before trusted-proxy client-IP resolution, and it lacks the operator-facing config/schema/docs/startup lifecycle needed for normal Gateway runs.

## Best Possible Solution

Keep this PR open for maintainer follow-up or explicit consolidation with #44884/#20373. The best implementation path is to add a `gateway.requestRateLimit` config/schema/help/docs surface, create and dispose the limiter during Gateway startup/shutdown, enforce it after `resolveRequestClientIp(req, trustedProxies, allowRealIpFallback)` in the HTTP pipeline, and add regression coverage for direct clients, trusted reverse proxies, same-host loopback proxies, forwarded headers, 429 `Retry-After`, and bounded limiter pruning/memory behavior.

## Risks / Open Questions

- Merging the PR as submitted could leave same-host reverse-proxy deployments effectively unprotected because loopback proxy traffic is exempted before the real client IP is resolved.
- For non-loopback reverse proxies, the submitted integration can collapse many unrelated users into one limiter bucket and cause cross-user 429s.
- Without config/schema/docs/startup creation and disposal wiring, the new limiter module would not be enabled in normal Gateway runs even though the code compiles as an optional parameter.
- No tests were executed because this was a read-only cleanup review; the decision is based on source/docs inspection and the provided PR diff/comments.
