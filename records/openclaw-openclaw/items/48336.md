## Summary

Keep this PR open. Current main still does not implement a general per-IP Gateway HTTP request limiter; it only has failed-auth throttling, a narrow control-plane write limiter, and a preauth WebSocket connection budget. The PR is still a plausible implementation candidate for #20373, but the submitted integration has a real security/design issue already called out in review: it keys on the raw socket peer before trusted-proxy client IP resolution, so reverse-proxy deployments can bypass or misbucket the limiter.

## Best Possible Solution

Keep this PR open for maintainer follow-up or explicit consolidation with #44884/#20373. The best implementation path is an operator-facing gateway.requestRateLimit config/schema/docs surface, startup creation/disposal of the limiter, enforcement after resolveRequestClientIp(req, trustedProxies, allowRealIpFallback), and regression coverage for direct clients, trusted reverse proxies, loopback proxy sources, forwarded headers, 429 Retry-After behavior, and bounded limiter memory/pruning behavior.

## Risks / Open Questions

- Merging the PR as submitted could leave same-host reverse-proxy deployments effectively unprotected because the limiter keys by the proxy socket address before trusted-proxy client-IP resolution and loopback is exempt by default.
- For non-loopback reverse proxies, the submitted integration can make unrelated clients share one limiter bucket and produce cross-user 429s.
- The PR currently adds an optional limiter parameter, but the provided diff does not add operator-facing config/schema/docs or gateway startup instantiation/disposal, so normal gateway runs would not be protected without additional wiring.
