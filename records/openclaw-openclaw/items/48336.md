## Summary

Keep this PR open. Current main still does not implement a general per-IP Gateway HTTP request limiter; it only has failed-auth throttling, a narrow control-plane write limiter, and a preauth WebSocket connection budget. The PR is a plausible implementation candidate for #20373, but it is not merge-ready as submitted because the rate-limit check keys on the raw socket peer before trusted-proxy client-IP resolution and the PR does not add the operator-facing config/startup/docs surface needed for normal Gateway runs.

## Best Possible Solution

Keep this PR open for maintainer follow-up or explicit consolidation with #44884/#20373. The best implementation path is to add an operator-facing gateway.requestRateLimit config/schema/docs surface, create and dispose the limiter during gateway startup/shutdown, enforce it after resolveRequestClientIp(req, trustedProxies, allowRealIpFallback), and add regression coverage for direct clients, trusted reverse proxies, loopback proxy sources, forwarded headers, 429 Retry-After behavior, and bounded limiter memory/pruning behavior.

## Risks / Open Questions

- Merging the PR as submitted could leave same-host reverse-proxy deployments effectively unprotected because the limiter uses the proxy socket address before trusted-proxy client IP resolution and loopback is exempt by default.
- For non-loopback reverse proxies, the submitted integration can place unrelated clients into one shared limiter bucket and cause cross-user 429s.
- Even with the new limiter module, the PR currently lacks gateway.requestRateLimit schema/docs/startup creation and disposal wiring, so normal Gateway runs would not enable the limiter without additional work.
