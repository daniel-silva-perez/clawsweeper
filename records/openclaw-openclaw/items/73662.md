## Summary

Keep open. The PR targets a real status regression from #73525, but the current diff still only enriches rows when a live `channels.status` payload is available; the default `openclaw status` path keeps that live channel status disabled, so the reported plain command can still return the read-only header-only Channels table.

## What This Changes

The PR adds a helper to synthesize missing Channels table rows from the live gateway `channels.status` payload, wires it into status overview, adds focused tests, and updates the changelog.

## Best Possible Solution

Revise the PR so default text status gets populated channel rows while preserving the fast-status contract, then add a regression test through `scanStatus({ json: false })` for the #73525 scenario. The maintainable fix should either use a bounded live gateway enrichment path for the default command or a read-only/setup-discovery source that can surface configured channels without reintroducing the slow checks.

## Reproduction Assessment

Yes. #73525 gives concrete default-command steps, and source inspection shows the default `scanStatus({ json: false })` path still disables both live channel status and setup runtime fallback unless `--all` or `--deep` is set.

## Solution Assessment

No. The helper is a reasonable additive row source for direct overview, `--all`, or `--deep` callers, but as wired it does not cover the plain `openclaw status` path the PR body and changelog claim to fix.
