## Summary

Keep this PR open: current main still lacks the gateway-snapshot row synthesis, but the proposed wiring still misses the reported plain `openclaw status` path because that caller disables the live channel payload the new helper needs.

## What This Changes

The PR adds `mergeChannelsTableWithGatewayStatus`, wires it into `collectStatusScanOverview`, adds focused tests, and records the CLI status fix in the changelog.

## Best Possible Solution

Revise the PR so plain `openclaw status` gets populated channel rows while preserving the fast-status contract, then cover it with a `scanStatus({ json: false })` regression test. The maintainable fix is either a read-only/setup-discovery row source for configured channels or an explicit bounded gateway enrichment path for the default command.

## Reproduction Assessment

Yes. #73525 gives concrete default-command steps, and the PR head source shows `scanStatus({ json: false })` still disables the live payload required by the new merge helper.

## Solution Assessment

No. The helper is a reasonable augmentation for `--all`, `--deep`, or direct overview callers, but as wired it does not fix the plain `openclaw status` path that the PR body and changelog claim to address.
