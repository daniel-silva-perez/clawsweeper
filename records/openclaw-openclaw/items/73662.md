## Summary

Keep open. The PR is a relevant implementation candidate for the channel-summary regression, but the current diff still misses the reported plain `openclaw status` path and would misclassify gateway snapshots that omit the optional `connected` field.

## What This Changes

The PR adds a helper that appends missing Channels rows from the live `channels.status` gateway snapshot, wires it into `collectStatusScanOverview`, adds focused tests, and updates the changelog.

## Best Possible Solution

Revise the status path so the default text command gets a lightweight, populated Channels row source for configured/running accounts without reintroducing heavyweight probes, and make synthesized row health follow the existing snapshot contract by treating only `connected === false` as disconnected.

## Reproduction Assessment

Yes. #73525 gives concrete default-command steps, and current main has a static/test-backed path showing plain `scanStatus({ json: false })` disables the live gateway payload that this PR depends on.

## Solution Assessment

No. Merging gateway rows is a reasonable seam for reachable-gateway cases, but this wiring does not cover the reported default command and uses stricter connection semantics than the current channel snapshot contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Wire the default status path to the synthesized rows:** `src/commands/status.scan-overview.ts:262-264`
  - body: This merge only runs when `channelsStatus` is present, but plain `openclaw status` still passes `includeLiveChannelStatus: false` from `scanStatus`, so the reported default command never requests the gateway snapshot and can continue rendering the empty read-only table. Add coverage through `scanStatus({ json: false })` or route default status to the intended lightweight row source.
  - confidence: 0.94
- **[P2] Do not require optional connected snapshots to be true:** `src/commands/status-all/channels.ts:236`
  - body: `connected` is optional in `ChannelAccountSnapshot`, and existing status/health code only treats `connected === false` as disconnected. Requiring every account to report `connected === true` makes running/configured webhook-style snapshots that omit the field synthesize a WARN/disconnected row instead of matching current channel status semantics.
  - confidence: 0.88
