## Summary

Keep this PR open for revision. The proposed gateway-snapshot merge is a plausible seam, but the latest PR head still does not reach the reported plain `openclaw status` path, and it treats an omitted optional `connected` field as a disconnect even though current channel status and health code treat only `connected === false` as disconnected.

## What This Changes

The PR adds `mergeChannelsTableWithGatewayStatus`, wires it into status overview channel-row construction, adds merge-focused tests, and updates the changelog for the Channels table regression.

## Best Possible Solution

Revise the branch so default text status can populate configured/running channel rows without loading heavyweight plugin runtimes, add a command-level `scanStatus({ json: false })` regression test for #73525, and align synthesized health with the existing optional `connected` contract.

## Reproduction Assessment

Yes. #73525 gives concrete command steps, and current main has both source and test evidence showing the default text status path does not request the live `channels.status` payload that the PR merge requires.

## Solution Assessment

No. The helper is a reasonable fallback for reachable-gateway overview paths, but this wiring misses the reported default command and uses stricter connection semantics than the existing channel snapshot and health contracts.
