## Summary

Keep open. The PR is still a useful targeted fix for the Channels table regression, but the current wiring does not cover the plain `openclaw status` path reported in #73525, and the synthesized row logic misclassifies snapshots where `connected` is absent rather than false.

## What This Changes

The PR adds a helper that synthesizes missing status Channels rows from the gateway `channels.status` snapshot, wires it into `collectStatusScanOverview`, adds focused tests, and updates the changelog.

## Best Possible Solution

Revise the PR so the default text status command gets populated channel rows for configured/running accounts while preserving the fast-status contract, and align synthesized row health with the existing channel snapshot semantics where `connected: false` is a failure but an absent `connected` field is unknown.

## Reproduction Assessment

Yes. #73525 gives concrete default-command steps, current main has a test that keeps default status off `channels.status`, and the PR head still gates the merge on the live payload that default status does not request.

## Solution Assessment

No. The gateway-snapshot merge is a reasonable seam for reachable-gateway cases, but as wired it misses the reported default command path and uses stricter connection semantics than the existing channel status/health contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Fix default status to invoke the row source:** `src/commands/status.scan-overview.ts:262-264`
  - body: This merge only runs when `channelsStatus` is non-null, but plain `openclaw status` still calls the overview with live channel status and setup fallback disabled from `scanStatus`. That is the path reported in #73525, so the PR can still render the header-only table from read-only discovery and never reach the synthesized gateway rows. Please wire the default command to the chosen row source, or add a `scanStatus({ json: false })` regression test that fails today.
  - confidence: 0.94
- **[P2] Treat absent connection state as unknown:** `src/commands/status-all/channels.ts:236`
  - body: `ChannelAccountSnapshot.connected` is optional, and existing status/health code treats only `connected === false` as disconnected. Requiring every account to have `connected === true` means a running configured channel that omits the field, such as webhook-style snapshots, will be synthesized as `WARN` with `disconnected` even though the gateway did not report a disconnect.
  - confidence: 0.88
