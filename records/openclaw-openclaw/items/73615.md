## Summary

Current main still appears to match the reported Telegram regression. Telegram preflight transcription frames the transcript for the agent and marks the audio attachment as already transcribed, but the only `sendTranscriptEcho` call is in the shared media-understanding path that runs only when new audio outputs are produced. The preflight-marked path therefore skips duplicate STT and also skips echo delivery.

## Best Possible Solution

Keep this issue open. The likely fix is to carry the raw Telegram preflight transcript into the shared media-understanding/echo path, or add a plugin-safe echo helper used after Telegram builds the finalized inbound context, so `echoTranscript` fires once before agent processing while preserving the current `MediaTranscribedIndexes` no-double-STT behavior. Add a Telegram DM voice-note regression test with `tools.media.audio.echoTranscript=true` that proves framed agent context and outbound echo delivery both happen.
