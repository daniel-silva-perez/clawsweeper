## Summary

Keep this PR open. Current main still builds the embedded-runner tool inventory before applying `toolsAllow`, so a restrictive config allowlist can remove core tools before the runtime allowlist post-filter sees them. The PR addresses a real, still-open bug, but it changes a command/data-access policy boundary and the branch is stale against current main, so it needs maintainer review and a rebase rather than cleanup closure.

## What This Changes

The PR threads embedded-runner `toolsAllow` into tool construction as `forceAllowTools`, merges requested known core tools into restrictive allow policies without overriding deny rules, and adds regression tests for re-exposure and deny precedence.

## Best Possible Solution

Rebase or rework the fix on current main so embedded runs can request known core tools through the tool-policy pipeline before the final `toolsAllow` post-filter, while preserving deny, sandbox, group, subagent, owner-only, provider, and runtime capability gates. Keep the regression coverage for restrictive allowlist re-exposure and explicit-deny precedence, then review and land it together with the linked bug.

## Reproduction Assessment

Yes. The source path on current main is enough to reproduce: configure `tools.profile: "full"` and `tools.allow: ["group:openclaw"]`, run an embedded attempt with `toolsAllow: ["exec"]`, and `exec` is removed by the policy pipeline before the post-filter at `attempt.ts:786`.

## Solution Assessment

Unclear until maintainer policy review. The PR's implementation is narrow and includes the right guardrails in the diff, but it is security-sensitive and stale against current main, so the best mergeable solution is a rebased version with the same deny-preserving semantics.
