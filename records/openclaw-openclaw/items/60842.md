## Summary

Keep this PR open. Current main still does not implement the PR's central behavior: `runEmbeddedAttempt` builds the tool inventory with `createOpenClawCodingTools()` before applying the runtime `toolsAllow` post-filter, and `createOpenClawCodingTools()` has no `forceAllowTools` or equivalent pre-policy re-exposure path. Under a restrictive config allowlist such as `tools.allow: ["group:openclaw"]`, `exec`/`read`/`write` can still be filtered out before the later `toolsAllow` filter runs. The linked same-author bug report remains open, and related PR #66710 was complementary forwarding work, not the config-allowlist fix proposed here.

## Best Possible Solution

Keep this PR open as a focused implementation candidate, or recreate its fix on main: forward runtime `toolsAllow` into tool creation as a narrowly scoped force-allow list, merge only known core tool ids into restrictive config/profile/provider allowlists before the policy pipeline, preserve deny-wins semantics and later sandbox/subagent/provider filtering, and add regression coverage for `group:openclaw` plus explicit-deny precedence.
