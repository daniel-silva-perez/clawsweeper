## Summary

Keep open. Current main still installs Playwright Chromium under `/home/node/.cache/ms-playwright`, the Docker/browser docs still point users at that hidden cache path, and repository search found no Dockerfile-created `chrome` or `chromium` command on PATH. The issue is also paired with open implementation PR #42771, so it should remain open until maintainers review, refresh, land, replace, or close that PR.

## What This Changes

The issue requests Docker browser-enabled images install Playwright Chromium in a stable non-hidden path and expose a stable `chrome` or `chromium` command instead of requiring versioned cache paths.

## Best Possible Solution

Implement the Docker packaging contract directly in the Dockerfile with aligned docs and regression coverage: browser assets should live outside `/home/node/.cache`, `chrome` or `chromium` should resolve through PATH in browser-enabled images, and existing Docker persistence guidance should point at the new stable path.

## Reproduction Assessment

Yes. Static current-main reproduction is high confidence: the Dockerfile still installs to `/home/node/.cache/ms-playwright`, docs still direct users there, and repository search found no Dockerfile-created Chrome/Chromium PATH symlink.

## Solution Assessment

Yes. The narrow maintainable solution is a Dockerfile, Docker/browser docs, and focused Dockerfile contract test update; the linked PR matches that boundary, but maintainers need to review or refresh it before this issue can close.
