## Summary

Keep open. Current main still starts the default-enabled model-pricing refresh during initial Gateway runtime setup before the post-attach sidecar/channel readiness boundary, so the reported boot-path delay is not fixed by the shipped opt-out alone.

## What This Changes

The issue reports macOS Gateway boot still waiting about 75 seconds on OpenRouter/LiteLLM pricing fetch timeouts after the earlier manifest-reuse fix, and asks that pricing failures not block channel/sidecar readiness.

## Best Possible Solution

Move the default pricing refresh fully off the Gateway boot readiness path so channels and sidecars can become ready before any OpenRouter/LiteLLM fetch waits or times out, while keeping `models.pricing.enabled: false` as an operator mitigation and leaving broader runtime degradation to the canonical tracker.

## Reproduction Assessment

Yes. Force OpenRouter/LiteLLM pricing fetches to hang or time out under default config, restart the Gateway, and watch whether sidecar/channel readiness waits; current source still starts pricing before the readiness boundary and retains the 60s timeout path.

## Solution Assessment

Yes, the narrow maintainable direction is to defer pricing refresh until after sidecars/channels are ready. The shipped config opt-out helps affected users but is not the best default-path fix because pricing enrichment is optional and should not compete with startup readiness.
