## Summary

Keep open. Current main still starts the default-enabled model-pricing refresh from initial Gateway runtime services before the post-attach sidecar/channel readiness path, so the reported boot critical-path problem is not implemented away. The existing opt-out helps affected users, but #73486 remains the open, unmerged narrow fix candidate for moving pricing refresh after readiness while #73323 tracks broader runtime degradation.

## What This Changes

The issue reports that macOS Gateway startup in OpenClaw 2026.4.26 still waits roughly 75 seconds on OpenRouter/LiteLLM model-pricing fetch timeouts before channel/sidecar readiness, despite the earlier manifest-reuse pricing fix.

## Best Possible Solution

Move model-pricing refresh out of initial Gateway runtime setup so optional OpenRouter/LiteLLM catalog work cannot compete with sidecar/channel readiness, keep the existing `models.pricing.enabled: false` escape hatch, and verify with a forced-hanging pricing fetch on macOS and Windows. Leave #73323 open for any broader long-running Gateway degradation after the startup-order fix lands.

## Reproduction Assessment

Yes. The issue provides concrete restart steps, logs, timing, and same-host standalone fetch comparisons; current-main source inspection also preserves the suspect ordering. Live OS reproduction was not run during this read-only review.

## Solution Assessment

No. Current main is not the best way to solve the issue because optional pricing refresh still starts before sidecars are ready by default; the narrower maintainable direction is the after-ready scheduling approach proposed by #73486, with regression coverage around startup ordering.
