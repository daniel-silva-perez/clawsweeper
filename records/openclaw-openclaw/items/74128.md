## Summary

Keep open. Current main and the latest release still start the default-enabled model-pricing refresh from early Gateway runtime setup, before the post-attach sidecar/channel readiness boundary, so the reported startup-readiness problem is not resolved by the shipped opt-out alone.

## What This Changes

The issue reports that macOS Gateway startup still waits about 75 seconds on OpenRouter/LiteLLM pricing fetch timeouts after the earlier manifest-reuse pricing fix.

## Best Possible Solution

Land or adapt #73486 so the default pricing refresh starts only after the post-attach sidecars-ready path and scheduled-service activation, while keeping `models.pricing.enabled: false` as an operator mitigation and leaving broader runtime/network degradation to #73323.

## Reproduction Assessment

Yes. A high-confidence path is to make the OpenRouter/LiteLLM pricing fetches hang under default config, restart the Gateway, and verify whether channel/sidecar readiness waits; current source still starts pricing before the readiness boundary and retains 60s fetch timeouts.

## Solution Assessment

Yes, for this issue's narrow scope the best fix is to move optional pricing refresh fully behind Gateway/channel readiness. The existing opt-out helps affected users but is not the best default behavior because remote pricing enrichment should not compete with startup readiness.
