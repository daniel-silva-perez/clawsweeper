## Summary

Keep open. Current main still starts the default-enabled model-pricing refresh from initial Gateway runtime services, before post-attach sidecars/channels reach the `gateway ready` path; the open, unmerged #73486 remains the narrow implementation candidate for moving that optional refresh after readiness, while #73323 tracks broader runtime degradation.

## What This Changes

The issue reports that Gateway startup still waits on 60s OpenRouter/LiteLLM pricing fetch timeouts before channel/sidecar readiness on macOS even after the earlier manifest-reuse fix.

## Best Possible Solution

Land or adapt #73486 so optional model-pricing refresh starts only after post-attach sidecars/channels have reached the ready path, then verify on macOS and Windows with OpenRouter/LiteLLM fetches forced to hang; keep #73323 open for any remaining long-running Gateway network/timer degradation.

## Reproduction Assessment

Yes for triage: the report gives deterministic restart steps, timing logs, and direct network counterchecks, and current-main code still has the same early pricing-start ordering. I did not run a live macOS LaunchAgent restart in this read-only review.

## Solution Assessment

Yes, the best narrow fix is to defer default-enabled pricing refresh until after sidecar/channel readiness while preserving the existing opt-out and pricing feature. The existing `models.pricing.enabled: false` workaround helps restricted networks but does not address the default startup-order bug.
