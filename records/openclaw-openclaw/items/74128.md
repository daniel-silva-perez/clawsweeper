## Summary

Keep open. Current main still starts the optional model-pricing refresh from early Gateway runtime setup before the sidecars/channels-ready boundary, so the reported startup-readiness problem is not resolved on main; the active narrow implementation candidate remains PR #73486.

## What This Changes

Issue #74128 reports that Gateway startup can still wait roughly 75 seconds on OpenRouter/LiteLLM pricing fetch timeouts after the earlier manifest-reuse pricing fix.

## Best Possible Solution

Land or adapt PR #73486 so the default pricing refresh starts only after post-attach sidecars/channels have reached the ready path, keep `models.pricing.enabled: false` as a mitigation, and leave #73323 to track the broader runtime/network degradation separately.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to make the OpenRouter/LiteLLM pricing fetches hang under default config, restart the Gateway, and verify whether channel/sidecar readiness waits; current source and tests still show pricing starts before the ready boundary with 60s fetch timeouts.

## Solution Assessment

Yes. For this issue's narrow scope, deferring optional pricing refresh until after sidecars/channels are ready is the narrowest maintainable fix; relying only on the documented opt-out leaves the default startup path vulnerable to remote catalog stalls.
