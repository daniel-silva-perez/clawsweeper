## Summary

Keep open. Current main still intentionally ignores explicit HTTP(S) prompt URLs for native image injection, so the bug this PR targets is not implemented; however, the PR patch introduces blocking network-fetch and SSRF-policy issues and is paired with the same-author open bug report #62514, so it should be reworked rather than cleaned up.

## What This Changes

The PR adds HTTP(S) URL detection and preloading to `detectImageReferences()` / `detectAndLoadPromptImages()` so mixed inline-image plus remote-image prompts can pass both images to native vision requests.

## Best Possible Solution

Keep this PR and #62514 open, then rework the native prompt image path to recognize clearly intended remote image refs, avoid fetching arbitrary links, thread the configured media SSRF policy, and cover mixed inline plus remote image prompts, no-fetch non-image links, punctuation/dedupe edge cases, and SSRF-policy propagation.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: the detector returns only one ref for a local image plus an HTTP(S) image URL, and the embedded runner only passes loaded `imageResult.images` to native prompt submission.

## Solution Assessment

No. The PR changes the right surface, but the current patch is not the best fix because it fetches every HTTP(S) URL and omits the existing SSRF-policy contract; a safer fix should narrow detection and thread policy through the preload path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid fetching every HTTP URL as a prompt image:** `src/agents/pi-embedded-runner/run/images.ts:40`
  - body: `HTTP_URL_REGEX_SOURCE` matches any HTTP(S) URL, and the new preload path fetches each detected ref before discarding non-images. A normal prompt that mentions a docs page or tracking link would now trigger an outbound gateway request, adding latency and privacy exposure outside the stated image-URL scope.
  - confidence: 0.95
- **[P1] Forward the configured SSRF policy for remote images:** `src/agents/pi-embedded-runner/run/images.ts:458-461`
  - body: The `remote-url` branch calls `loadWebMedia` with only `maxBytes`, unlike the existing image tool path that passes `remoteMediaSsrfPolicy`. This means configured `tools.web.fetch.ssrfPolicy` allowlists or private-network controls are not honored for native remote image preloads.
  - confidence: 0.94
- **[P2] Do not mark skipped bracket URLs as seen:** `src/agents/pi-embedded-runner/run/images.ts:337-345`
  - body: When a `[media attached: path | url]` entry has a local path, this branch adds the URL dedupe key even though it does not emit a `remote-url` ref. If the same URL is repeated explicitly elsewhere in the prompt, the later scan suppresses it and the remote image is never loaded.
  - confidence: 0.86
- **[P2] Use URL-specific dedupe keys for remote refs:** `src/agents/pi-embedded-runner/run/images.ts:277-281`
  - body: `addRemoteUrlRef` reuses `normalizeRefForDedupe`, which lowercases keys on Windows for filesystem paths. HTTP URL paths can be case-sensitive, so Windows gateways can collapse two distinct remote image URLs into one native image input.
  - confidence: 0.84
- **[P2] Trim closing parentheses from detected image URLs:** `src/agents/pi-embedded-runner/run/images.ts:94-106`
  - body: The URL regex allows `)` and the punctuation trimmer does not remove it, so Markdown or parenthesized links such as `(https://example.com/image.png)` are fetched as `image.png)`. That causes avoidable native image-load failures for common prompt formatting.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR expands outbound network fetching from prompts and bypasses the existing configurable media SSRF policy on the new native remote-image path.

Concerns:

- **[medium] Prompt text can trigger unintended outbound fetches:** `src/agents/pi-embedded-runner/run/images.ts:40`
  - body: Because the new regex treats every HTTP(S) URL as an image candidate, ordinary links in vision-model prompts cause gateway network requests even when they are not image URLs.
  - confidence: 0.95

  - body: The new `remote-url` loader calls `loadWebMedia` without passing the configured `tools.web.fetch.ssrfPolicy`, so native image preloads do not honor the same policy used by the existing image tool path.
