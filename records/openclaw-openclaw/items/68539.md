## Summary

Keep open. Current main has inbound audio transcription, but `openai-codex` is not registered as an audio transcription provider: the bundled OpenAI plugin exposes audio only on `openai`, the media runner requires a `transcribeAudio` implementation for audio providers, and the docs list OpenAI Codex OAuth for image understanding but not audio/STT.

## What This Changes

The issue requests voice/audio transcription through the `openai-codex` ChatGPT/Codex-authenticated path without an OpenAI API key, API billing, or coupling to Codex text-message limits.

## Best Possible Solution

Keep this as provider-owned OpenAI plugin work: either add a documented `openai-codex` audio transcription implementation with live proof of the Codex-authenticated endpoint and limit/billing behavior, or explicitly document that Codex OAuth STT is unsupported and direct users to the existing API/CLI STT providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to select `openai-codex` as the active or configured audio provider: current main will not choose it for audio because it lacks `transcribeAudio`, and an explicit audio entry would hit the runner's provider-not-available path.

## Solution Assessment

Unclear. The narrowest maintainable implementation would live in `extensions/openai`, but the exact requested no-cost, separate-limit behavior needs upstream/API confirmation before an automated code change is safe.
