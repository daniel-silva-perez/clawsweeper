## Summary

Keep open: current main does not provide the requested Codex-authenticated batch audio transcription path. The OpenAI plugin has a Codex OAuth media-understanding provider, but it is image-only today, and the shared audio runner only selects providers with a `transcribeAudio` hook and then resolves API-key-style execution credentials before posting audio.

## What This Changes

Request inbound audio and voice-note transcription for `openai-codex` through ChatGPT/Codex OAuth, without `OPENAI_API_KEY`, OpenAI API billing, or coupling to normal Codex text-message limits.

## Best Possible Solution

If maintainers confirm a supported Codex/ChatGPT transcription transport, implement it inside the OpenAI plugin as provider-owned media-understanding behavior: add an `openai-codex` audio capability and `transcribeAudio` implementation using the existing Codex OAuth auth-profile resolver, avoid falling back to `OPENAI_API_KEY` for that route, cover Telegram/shared media runner behavior with tests, and document the supported quota/auth semantics. If no supported upstream route exists, document the current alternatives: OpenAI API STT, other STT providers, or local CLI transcription.

## Reproduction Assessment

Yes. A source-level reproduction is to use `openai-codex` as the active model with no `OPENAI_API_KEY` and no local/audio fallback; the runner skips `openai-codex` for audio because it lacks `transcribeAudio`, so voice input does not reach a Codex OAuth transcription path.

## Solution Assessment

Not applicable yet because there is no proposed patch. The narrow maintainable fix would be provider-owned in the OpenAI plugin, but only after confirming a supported upstream Codex transcription endpoint and quota contract.
