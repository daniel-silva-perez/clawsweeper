## Summary

Keep open. Current main has strong adjacent rate-limit handling, model-scoped auth-profile cooldown metadata, cooldown probing, and ordered fallback, but it still does not implement the requested dynamic rate-limit-driven model routing policy: no model strategy config, no per-model health store, no round-robin/health-aware selector, and no automatic downgrade plus upgrade-back behavior. The aashari fork linked in the discussion remains useful prior art rather than shipped upstream behavior.

## Best Possible Solution

Keep this open as a core model-routing design issue. A good upstream implementation would document and implement policies such as failover versus health-aware/round-robin/downgrade routing, add provider/model/profile health state with success/failure recovery semantics, expose observable downgrade and upgrade-back events, update config schema and docs, and add regression tests for rate-limit scenarios. Treat the aashari fork linked in the discussion as prior art, not shipped OpenClaw behavior.
