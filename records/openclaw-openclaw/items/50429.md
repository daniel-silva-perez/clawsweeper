## Summary

Keep this PR open. Current main still sends non-streaming Feishu reply cards directly and has no plain-text fallback for table-limit card failures, so the PR is not obsolete. It is also not cleanup-closeable because it is linked to open bug #43690 and contains relevant work, but the proposed catch-all fallback still needs tightening around error classification, chunk duplication, and block payload handling.

## Best Possible Solution

Keep this PR open and request a focused revision: share or align with the table-limit predicate from #70651, fall back only for Feishu table-limit failures, limit fallback to final user-visible replies when no prior card chunk was delivered or resume from the failed chunk, preserve block payload suppression, and add regression tests for table-limit fallback, non-table-limit rethrow, later-chunk failure, and block payload handling.

## Risks / Open Questions

- The PR fallback catches every structured-card failure rather than only Feishu table-limit errors such as code 230099 / ErrCode 11310, so it can hide authentication, permission, rate-limit, or malformed-card failures.
- If a later chunk fails after earlier card chunks were already delivered, retrying the entire reply as plain text can duplicate content and first-chunk mentions.
- If a block payload reaches the fallback after streaming startup fails, the PR can expose preview or reasoning-like content as a normal Feishu message before the final answer.
- The added tests in the provided PR context cover first-card failure and success only; they do not cover non-table-limit rethrow, later-chunk failure, or block payload suppression.
