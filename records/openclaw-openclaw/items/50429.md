## Summary

Keep open. The Feishu table-limit bug is real and current main still has no reply-dispatcher text fallback, but this PR is not ready to merge as-is: the latest diff falls back for every card error, retries the entire reply after any chunk failure, and can send block/preview payloads as ordinary Feishu text. The related outbound classifier/fallback work in #73651 is still open and should be coordinated rather than closing or replacing this PR prematurely.

## What This Changes

The PR wraps non-streaming Feishu reply card delivery in a catch block that logs card-send failures, retries through the existing plain-text chunk sender, and adds two reply-dispatcher tests for card failure fallback and successful card delivery.

## Best Possible Solution

Revise the reply-dispatcher fallback to degrade only known Feishu table-limit errors, preserve block/preview suppression, and avoid duplicating already delivered card chunks by falling back only before the first card chunk is sent or by resuming from the failed chunk. Share or align the table-limit classifier with #73651 if that lands first, and add regression tests for table-limit fallback, non-table rethrow, block payload handling, and multi-chunk partial delivery.

## Review Findings

Full review comments:

- **[P2] Avoid retrying after delivered card chunks:** `extensions/feishu/src/reply-dispatcher.ts:419-448`
  - body: If sendChunkedTextReply() fails on chunk 2 or later, earlier card chunks have already been sent. The catch then retries the original full text as plain text, duplicating the beginning of the answer and first-chunk mentions. Track whether any card chunk was delivered and only fall back before the first send, or resume from the failed chunk.
  - confidence: 0.88
- **[P2] Limit fallback to final reply payloads:** `extensions/feishu/src/reply-dispatcher.ts:441-464`
  - body: When streaming startup has failed, info.kind === "block" can still reach this catch path. Sending those failed card chunks through sendMessageFeishu can expose block/preview content as normal chat text before the final answer, so the fallback should stay limited to final user-visible replies or explicitly drop block payloads here.
  - confidence: 0.84
- **[P2] Classify table-limit errors before fallback:** `extensions/feishu/src/reply-dispatcher.ts:441-448`
  - body: The catch handles every sendStructuredCardFeishu failure. That hides auth, permission, malformed-card, and rate-limit problems by retrying them as plain text; only the known 230099 / 11310 table-limit rejection should degrade to text, and unrelated failures should still propagate.
  - confidence: 0.8
