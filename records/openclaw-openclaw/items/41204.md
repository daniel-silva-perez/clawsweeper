## Summary

Keep open. The PR is still an open, unmerged hardening proposal and current main does not contain the session-store queue cap it proposes, so it is not obsolete; however, the branch is not safe to merge because its subagent sweep and installer-validation changes regress current behavior, matching the prior review discussion.

## What This Changes

The branch adds a per-store session lock queue cap, subagent orphan cleanup during sweeping, skills installer value validation, and three related Vitest files.

## Best Possible Solution

Do not merge the current branch. Preserve main's subagent cleanup and installer validation behavior, and if maintainers still want the resource-exhaustion hardening, land a narrow session-store queue/backpressure patch with deterministic coverage and an explicit queue-limit policy.

## Reproduction Assessment

Yes. The PR diff itself reproduces the main regressions: non-due subagent records can fall through into cleanup, delete failures lose retry state, shared installer validation blocks valid specs, and the queue-limit test serializes 1100 delayed lock tasks.

## Solution Assessment

No. The current PR mixes three security-sensitive surfaces and regresses two areas that main already handles more carefully; a narrow queue-cap patch, if desired, is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Restore the archive guard before sweep cleanup:** `src/agents/subagent-registry.ts:784-791`
  - body: After the orphan branch, the loop reaches standard archive cleanup for entries without archiveAtMs or with a future archiveAtMs. That can delete active or recently ended subagent records and call sessions.delete before archival is due.
  - confidence: 0.93
- **[P2] Keep archive records until session deletion succeeds:** `src/agents/subagent-registry.ts:728-745`
  - body: The due-archive branch deletes the run record and attachments before sessions.delete, then ignores delete failures. If transcript deletion fails, the registry loses the state needed to retry cleanup.
  - confidence: 0.9
- **[P2] Use installer-specific validators for package specs:** `src/agents/skills-install.ts:197-217`
  - body: The shared validator rejects valid node, Go, and uv installer syntax that current metadata accepts, including uv extras/version pins and npm or Go version forms with characters such as ~, +, and =. Legitimate skills can fail installation at runtime.
  - confidence: 0.9
- **[P2] Bound the queue-limit stress test:** `src/config/sessions/store.lock-queue.test.ts:18-37`
  - body: This test queues 1100 lock tasks that each sleep for 100 ms behind the serial lock, so Promise.allSettled can take roughly 100 seconds. That is likely to time out or make the targeted suite impractically slow.
  - confidence: 0.86
