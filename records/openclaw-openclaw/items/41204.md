## Summary

Keep this PR open as a signal for the remaining session-store queue/backpressure hardening idea, but do not merge the current branch. Current main already has stronger skills-installer validation and subagent cleanup behavior, while it still lacks the proposed lock-queue cap.

## What This Changes

The PR adds a session-store lock queue cap, subagent orphan cleanup in the sweeper, skills-install package/formula validators, and three new test files.

## Best Possible Solution

Retain the current main behavior for skills installation and subagent cleanup, and either ask for a narrow rebase/replacement that only implements a justified session-store queue/backpressure policy with focused tests or record a maintainer decision that no cap is wanted.

Full review comments:

- **[P1] Restore the archive guard before cleanup:** `src/agents/subagent-registry.ts:784-785`
  - body: After the new orphan branch, the loop falls into the standard archive cleanup block for any entry that is not orphaned, including active runs and runs whose `archiveAtMs` is unset or still in the future. That can remove live/recent subagent records and invoke session deletion on the next sweep instead of waiting for archive time.
  - confidence: 0.93
- **[P2] Keep archive entries until session deletion succeeds:** `src/agents/subagent-registry.ts:730-745`
  - body: The new due-archive branch deletes the run record and attachments before calling `sessions.delete`, then ignores delete failures. If the gateway delete fails, the registry no longer has a record to retry cleanup, leaving child session artifacts behind despite delete-mode cleanup.
  - confidence: 0.89
- **[P2] Use per-installer validators for package specs:** `src/agents/skills-install.ts:197-217`
  - body: The shared `validatePackageName` whitelist is applied to node, Go, and uv install specs, but it rejects valid syntax that current metadata accepts, such as uv extras/version pins and npm or Go version forms with `~`, `+`, or `=`. Those skills would now fail installation with a validation error.
  - confidence: 0.9
- **[P2] Avoid serial sleeps in the queue-limit test:** `src/config/sessions/store.lock-queue.test.ts:18-37`
  - body: The new stress test accepts roughly a thousand lock tasks, each sleeping for 100 ms under the serialized lock, before `Promise.allSettled` can finish. That makes the test take around 100 seconds and likely time out in Vitest; stub the lock/work duration or expose the cap for a small deterministic test.
  - confidence: 0.86
