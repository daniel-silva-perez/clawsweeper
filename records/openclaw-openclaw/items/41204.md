## Summary

Keep open. Current main already has stronger skills-install validation and subagent sweep/archive behavior than parts of this PR, but the session-store lock queue cap is still not implemented; the PR branch is also not safe to merge because its subagent cleanup and shared installer validation introduce concrete regressions.

## What This Changes

The PR adds a session-store lock queue limit, orphaned subagent cleanup during sweep, skills-install package/formula validators, and three new test files.

## Best Possible Solution

Keep the current main behavior for skills installation and subagent cleanup. If maintainers still want session-store queue hardening, land it as a narrow backpressure change with explicit cap semantics and deterministic lock-queue tests, without replacing current per-installer validators or archive retry behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restore the archive guard before cleanup:** `src/agents/subagent-registry.ts:784-785`
  - body: After the new orphan branch, the loop can fall through into archive cleanup for entries that are not due for archive, including active or recently ended runs. That can remove live subagent records and invoke session deletion before the configured archive time.
  - confidence: 0.93
- **[P2] Keep archive entries until session deletion succeeds:** `src/agents/subagent-registry.ts:730-745`
  - body: The due-archive branch deletes the run record and attachments before calling sessions.delete, then ignores delete failures. If deletion fails, the registry loses the state needed to retry transcript cleanup.
  - confidence: 0.89
- **[P2] Use per-installer validators for package specs:** `src/agents/skills-install.ts:197-217`
  - body: The shared package-name validator is applied to node, Go, and uv specs, but it rejects valid syntax that current metadata accepts, including uv extras/version pins and npm or Go version forms with characters such as ~, +, or =.
  - confidence: 0.9
- **[P2] Avoid serial sleeps in the queue-limit test:** `src/config/sessions/store.lock-queue.test.ts:18-37`
  - body: The stress test accepts roughly a thousand serialized lock tasks that each sleep for 100 ms before Promise.allSettled can finish, making the test take about 100 seconds and likely time out in Vitest.
  - confidence: 0.86
