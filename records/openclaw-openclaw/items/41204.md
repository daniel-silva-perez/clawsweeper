## Summary

Keep open, but do not merge this PR as-is. Current main already has stronger implementations for the skills-install validation and subagent stale/orphan recovery parts, while the session-store lock queue cap is still not present; the PR branch also introduces clear regressions in subagent sweeping and installer compatibility.

## What This Changes

The PR adds a session-store lock queue cap, subagent orphan cleanup in `sweepSubagentRuns`, skill installer package/formula validation, and three new test files.

## Best Possible Solution

Keep the current main implementations for skills installer validation and subagent orphan recovery, and handle the remaining queue-cap concern as a narrow session-store change with deterministic tests around `withSessionStoreLock` and lock-runtime reset helpers.

## Reproduction Assessment

Yes for the PR regressions: diff inspection shows non-archivable subagent runs fall through into cleanup, and the shared installer regex rejects valid specs that current main tests accept. The C1 resource-exhaustion concern has a clear code-level path because the lock queue is unbounded, but I did not run a live stress test in this read-only review.

## Solution Assessment

No. The PR is broader than the remaining problem, duplicates or regresses areas current main already handles better, and should be replaced or heavily revised rather than merged.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Restore the archive guard before sweep cleanup:** `src/agents/subagent-registry.ts:784-785`
  - body: After the new orphan branch, any run that is not due for archive and is not recognized as an orphan falls through into the standard cleanup block. That means active runs, session-mode runs, or runs with a future `archiveAtMs` can be deleted on the next sweep; keep the old `!archiveAtMs || archiveAtMs > now` guard before cleanup.
  - confidence: 0.94
- **[P2] Use installer-specific safe specs:** `src/agents/skills-install.ts:128`
  - body: The shared package regex rejects valid installer metadata that the repo already accepts, such as uv extras/version pins and Go or npm version syntax with characters outside this whitelist. This would make legitimate skill installers fail at runtime; use the existing per-kind validation contracts and pass the trimmed value to command builders.
  - confidence: 0.88
