## Summary

Keep open. Current main already has safer subagent cleanup and installer validation than this branch, but it does not implement the session-store lock queue cap; the PR is still not safe to merge because its subagent sweep and installer validator changes introduce concrete regressions.

## What This Changes

The PR adds a session-store lock queue cap, an orphaned subagent sweep path, skills-install package/formula validation, and three related Vitest files.

## Best Possible Solution

Do not merge the current branch. Keep main's subagent cleanup and installer validation, and if maintainers still want the resource-exhaustion hardening, land a narrow session-store queue/backpressure patch with bounded deterministic coverage and no changes to subagent cleanup or installer syntax.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore the archive guard before sweep cleanup:** `src/agents/subagent-registry.ts:784-791`
  - body: After the orphan branch, the loop reaches standard archive cleanup for entries without archiveAtMs or with a future archiveAtMs. That deletes active or recently ended subagent records and calls sessions.delete before they are due for archival.
  - confidence: 0.93
- **[P2] Keep archive records until session deletion succeeds:** `src/agents/subagent-registry.ts:728-745`
  - body: The due-archive branch deletes the run record and attachments before sessions.delete, then swallows delete failures. If transcript deletion fails, the registry loses the state needed to retry cleanup.
  - confidence: 0.9
- **[P2] Use installer-specific validators for package specs:** `src/agents/skills-install.ts:197-217`
  - body: The shared validator rejects valid node, Go, and uv installer syntax that current metadata accepts, including uv extras/version pins and npm or Go version forms with characters such as ~, +, and =. Legitimate skills can fail installation at runtime.
  - confidence: 0.9
- **[P2] Make the queue-limit stress test bounded:** `src/config/sessions/store.lock-queue.test.ts:18-37`
  - body: This test queues 1100 lock tasks that each sleep for 100 ms behind the serial lock, so Promise.allSettled can take roughly 100 seconds. That is likely to time out or make the targeted suite impractically slow.
  - confidence: 0.86
