## Summary

Keep this PR open, but do not merge the current branch. Current main already has stronger skills installer validation and subagent orphan/archive behavior than the PR, and the PR diff has security-sensitive functional regressions, but current main still does not implement the session-store lock queue cap that is one of the PR's stated fixes.

## Best Possible Solution

Keep this PR open only as a signal for the remaining queue-cap idea, but ask for a narrow rebase rather than merging this branch. The maintainable path is to drop the superseded skills-installer and subagent changes, add a justified session-store queue limit in `src/config/sessions/store.ts`, and cover it with production-path tests. If maintainers decide a queue cap is not wanted, close after opening or linking a narrower tracking issue or documenting that decision.
