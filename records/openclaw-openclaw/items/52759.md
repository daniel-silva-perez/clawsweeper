## Summary

Keep open. Current main still leaves tool-loop detection disabled by default and still has no read-aware out-of-range offset recovery, so this PR is not obsolete; however the branch is not merge-ready because it mixes the loop-default decision with read-offset recovery, carries local generated artifacts, and misses current config/docs surfaces after main moved on.

## What This Changes

The PR flips tool-loop detection on by default, updates related tests/docs/schema metadata, and catches read-tool out-of-range offset errors to return a diagnostic result instead of throwing.

## Best Possible Solution

Land two focused, rebased changes if maintainers want both behaviors: one canonical loop-detection default change that preserves run-id scoping and updates tests, exported type comments, schema help, generated checksums, and current docs; and one read-offset recovery fix inside adaptive paging with coverage for explicit out-of-range offsets and shrinking-file partial output.

## Reproduction Assessment

Yes. The loop-default behavior is reproducible by recording repeated tool calls and calling detectToolCallLoop without config; the read-offset path is reproducible by invoking tools.read with an offset beyond EOF. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The direction may be valid, but this branch is not the best merge shape because it mixes unrelated fixes, commits local generated baselines, misses current config/type/docs surfaces, and handles EOF recovery at a layer that can discard partial output.
