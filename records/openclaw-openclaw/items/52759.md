## Summary

Keep open. Current main still has loop detection disabled by default, so the main behavior is not implemented, but this PR is not merge-ready: it mixes the default flip with read-offset handling, is stale against current run-scoped loop history and current docs/generated-baseline structure, and has concrete correctness issues that should block merge.

## What This Changes

The PR enables tool-loop detection by default, updates a default-behavior unit test plus docs/schema metadata, commits generated config baseline artifacts, and wraps read-tool offset-beyond-EOF errors into a diagnostic result.

## Best Possible Solution

Use two focused, rebased changes if maintainers want this direction: one canonical loop-detection default change that preserves run-id scoping and synchronizes the exported type comments, schema help, current docs, and tracked generated SHA baselines; and one separate read-offset recovery change inside adaptive paging if a read-specific diagnostic is still needed. Drop the full generated JSON/JSONL baselines from the branch.

## Reproduction Assessment

Yes. The loop-default behavior is reproducible by recording repeated tool calls and calling detectToolCallLoop without config; current main’s test still asserts the old disabled default. The read-offset path is reproducible by invoking tools.read with an offset beyond EOF, and the partial-output loss follows from the PR catch wrapping the whole adaptive pager.

## Solution Assessment

No. The safety goal may be valid, but this PR is not the best merge shape because it combines unrelated runtime changes, updates stale or local-only generated docs artifacts, misses current config/type surfaces, and handles read EOF recovery outside the pager where partial output can be lost.
