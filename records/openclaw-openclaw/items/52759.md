## Summary

Keep open. Current main still leaves `tools.loopDetection` disabled by default, so the central loop-default change is not obsolete, but this PR is not merge-ready because it mixes that decision with a read-offset change, commits local-only generated baselines, misses current config/doc/type surfaces, and needs reconciliation with overlapping loop-detection PRs.

## What This Changes

The PR enables tool-loop detection by default, updates related tests/docs/config metadata and generated baseline files, and adds read-tool out-of-range offset handling that returns a diagnostic result instead of throwing.

## Best Possible Solution

Use one focused, rebased loop-detection default change if maintainers want default-on behavior: preserve run-id scoping, sync exported type comments, schema help, generated SHA baselines, current docs, and tests. Treat read-specific clamping or richer EOF diagnostics as a separate follow-up only if the existing adapter-level diagnostic result is not sufficient.

## Reproduction Assessment

Yes. The loop-default behavior is reproducible by recording repeated tool calls and calling `detectToolCallLoop` without config; the read-offset path is reproducible by invoking `tools.read` with an offset beyond EOF, though I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The safety direction may be valid, but this PR is not the best merge shape because it combines unrelated behavior changes, commits ignored generated artifacts, misses current config/doc/type surfaces, and handles read EOF recovery outside the adaptive pager where partial output can be lost.
