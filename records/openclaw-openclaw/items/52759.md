## Summary

Keep open. Current main still ships loop detection off by default and documents/tests that opt-in behavior, while the read-tool wrapper does not have the PR's read-specific out-of-range offset handling. The PR is not merge-ready as-is because it mixes a product/safety default change with a read-tool behavior change, overlaps several open loop-detection PRs, and needs rebasing against current run-scoped loop history and generated-doc layout.

## What This Changes

The PR flips `tools.loopDetection.enabled` to true by default, updates related loop-detection tests/docs/schema metadata, regenerates config baselines, and adds a read-tool diagnostic result for offset-beyond-EOF errors.

## Best Possible Solution

Rebase and likely split this work: handle loop detection defaulting in one canonical PR that preserves current run-id scoping and updates runtime tests, config schema/help, generated checksums, and docs together; handle read offset recovery separately inside the adaptive read path so it preserves partial output and has focused regression coverage.
