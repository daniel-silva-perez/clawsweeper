## Summary

Keep open. Current main still has loop detection disabled by default and does not provide read-aware out-of-range offset recovery, but this branch is not merge-ready: it mixes two fixes, commits ignored generated baselines, misses current config docs/type comments, and handles offset recovery outside adaptive paging. The overlapping open loop-default PRs make this a maintainer reconciliation item rather than a cleanup close.

## What This Changes

The PR enables tool-loop detection by default, updates related tests/config/docs, commits generated config baselines, and wraps read-tool out-of-range offset failures with a diagnostic result.

## Best Possible Solution

Land two focused, rebased changes if maintainers want both behaviors: one canonical loop-detection default change that preserves run-id scoping and updates tests, schema help, exported type comments, generated checksums, and current docs; and one read-offset recovery change inside the adaptive pager with regression coverage for explicit out-of-range offsets and shrinking-file partial output.

## Reproduction Assessment

Yes. The loop-default behavior is reproducible with repeated detectToolCallLoop calls without config, and current tests encode the disabled default; the read-offset path is reproducible by calling tools.read with an offset beyond EOF. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction may be valid, but this is not the best merge shape because it mixes unrelated fixes, updates stale/generated-local docs artifacts, misses current config/type surfaces, and handles EOF recovery at a layer that can discard partial output.
