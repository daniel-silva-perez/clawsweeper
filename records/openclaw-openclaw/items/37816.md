## Summary

Keep this issue open. Current main still does not implement the requested Telegram mode for durable, separate, Web-UI-style activity messages. Telegram currently supports preview-message edits, tool-progress lines inside that preview, and a Telegram-only reasoning stream, but the public config/schema is still limited to `off | partial | block | progress`, `progress` maps to `partial`, and current dispatch suppresses standalone default progress messages when preview streaming is off or already owned by the Telegram draft preview path.

## Best Possible Solution

Keep this open as bundled Telegram plugin/product work. The best path is an explicit opt-in `channels.telegram` mode or flag for durable activity messages, with defaults unchanged, NO_REPLY/internal-run suppression, sensitive-output safeguards, rate limiting/coalescing, docs/schema/UI metadata, and focused Telegram tests. Reuse existing replyOptions callbacks where sufficient; add a narrow generic channel/plugin activity-event seam only if current callbacks cannot represent the required Web-UI-style events.
