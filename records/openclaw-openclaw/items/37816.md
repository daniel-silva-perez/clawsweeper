## Summary

Keep open. Current main still has Telegram preview/activity-adjacent behavior, but not the requested opt-in Web UI parity mode where each agent activity step is sent as a durable separate Telegram message. The public config/schema only allow preview modes `off | partial | block | progress`; Telegram folds tool/progress events into an edited `Working…` preview when preview streaming is active, and docs explicitly describe `off` as final-only with generic progress chatter suppressed.

## Best Possible Solution

Keep this open as bundled Telegram plugin/product work. The best path is an explicit opt-in `channels.telegram` mode or flag for durable activity messages, with schema/docs/tests, default behavior unchanged, NO_REPLY/internal-run suppression, rate-limit controls, and sensitive-output safeguards. Reuse current replyOptions callbacks where sufficient, and add a narrow generic channel/plugin activity event seam only if current callbacks cannot represent the needed Web UI-style events.

## Risks / Open Questions

- Telegram rate limits and chat noise are real constraints, especially in groups and tool-heavy turns.
- The existing replyOptions callbacks cover many activity types, but full Web UI parity may need a narrow additive channel/plugin activity event contract if memory or internal activity sources are not represented.
