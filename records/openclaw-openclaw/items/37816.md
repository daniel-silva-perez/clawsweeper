## Summary

Keep this issue open. Current main has Telegram activity-adjacent behavior, including edited preview streaming, short tool/progress lines inside the edited preview, and a Telegram reasoning preview lane, but it still does not provide an opt-in mode that mirrors Web UI activity as durable separate Telegram messages. The config/schema only allow off/partial/block/progress preview modes; final-only mode explicitly suppresses generic progress chatter; and the dispatcher folds tool, plan, command, approval, and patch summaries into one edited “Working…” preview rather than sending persistent per-activity messages.

## Best Possible Solution

Keep this open as bundled Telegram plugin/product work. The best path is an explicit opt-in `channels.telegram` mode or flag for durable activity messages, with schema/docs/tests, default behavior unchanged, NO_REPLY/internal-run suppression, rate-limit controls, and sensitive-output safeguards. Reuse existing replyOptions callbacks where they are sufficient, and add a narrow generic channel/plugin activity event seam only if current callbacks cannot represent the needed Web UI-style events.

## Risks / Open Questions

- A durable per-activity Telegram mode can expose reasoning, command output, tool results, memory paths, or other sensitive data; it needs explicit opt-in and redaction/filtering decisions.
- Telegram rate limits and chat noise are real product constraints, especially in groups or tool-heavy turns.
- The existing callback surface covers many activity types, but full Web UI parity may require a narrow additive channel/plugin event contract for any missing reasoning, memory, or internal activity sources.
- Related open issue #66459 is about Telegram delivery reliability for thinking/text turns and should not be confused with this feature request, but it affects the same surface.
