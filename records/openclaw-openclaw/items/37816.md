## Summary

Current main still does not implement the requested Telegram mode for durable, separate, Web-UI-style activity messages. Telegram exposes preview streaming and tool-progress edits, plus reasoning/verbose surfaces, but the public config/schema remains `off | partial | block | progress`; `progress` maps to `partial`, and current Telegram dispatch suppresses standalone default progress messages when preview streaming is off.

## Best Possible Solution

Keep this open as bundled Telegram plugin/product work. The best path is an explicit opt-in `channels.telegram` mode or flag for durable activity messages, with defaults unchanged, NO_REPLY/internal-run suppression, sensitive-output safeguards, rate limiting/coalescing, docs/schema/UI metadata, and focused Telegram tests. Reuse existing replyOptions callbacks where sufficient; add a narrow generic channel/plugin activity-event seam only if current callbacks cannot represent the required Web UI-style events.
