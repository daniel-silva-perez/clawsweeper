## What This Changes

The issue reports that `openclaw tui -h` hangs at 100% CPU because help banner generation synchronously loads the heavy config graph through `readCliBannerTaglineMode()`.

## Best Possible Solution

Review #62485 or replace it with a maintainer-approved fix that keeps help rendering off the heavy config import path while preserving `cli.banner.taglineMode` semantics and the `readCliBannerTaglineMode(env)` contract.

## Reproduction Assessment

Yes. The reported command path is concrete, and current main still shows `openclaw tui -h` reaching a synchronous `createConfigIO().loadConfig()` call during Commander help banner generation; I did not run the potentially hanging command in this read-only sweep.

## Solution Assessment

Unclear for the existing PR until maintainer review, because the dynamic runtime-boundary direction is narrow and plausible but cache/config-env behavior still needs verification. The best fix belongs in the CLI banner config helper rather than the TUI runtime itself.
