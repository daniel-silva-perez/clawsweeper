## Summary

Keep this PR open. Current main has not implemented the requested human-readable cron timestamp fields, but the proposed helper can throw from best-effort logging/event paths for finite epoch values outside JavaScript Date range, so the patch should be corrected before merge.

## What This Changes

The PR adds ISO-8601 companion fields for cron scheduler epoch-ms log/event timestamps plus unit coverage for the new formatter helper.

## Best Possible Solution

Keep the additive raw-plus-human timestamp direction, but make the formatter non-throwing for invalid Date ranges and add focused coverage for an out-of-range finite value before merge.

## Reproduction Assessment

Yes. The focused reproduction is `new Date(9e15).toISOString()`, which throws `RangeError`, combined with `parseAbsoluteTimeMs` accepting large positive finite digit strings that can become cron schedule state.

## Solution Assessment

No. The proposed implementation is close, but the best fix is to validate the constructed Date's time value or catch `RangeError` and return `undefined` so logging and event emission remain best-effort.
