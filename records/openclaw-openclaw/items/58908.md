## Summary

Keep this PR open. Current main does not implement the intended human-readable cron timestamp fields: cron logs still emit only numeric `nextAt` / `nextRunAtMs`, and `CronEvent` has no optional `nextRunAt` string. The PR is a plausible implementation candidate for the still-open linked feature request, but it needs review fixes before landing.

## Best Possible Solution

Keep the PR open as the active implementation candidate for the linked cron readability request. The best path is to update the helper so invalid or out-of-range dates return `undefined`, settle the human-readable field naming, keep the raw epoch fields unchanged, and add focused tests before maintainer review/merge.

## Risks / Open Questions

- The PR’s proposed `formatEpochMs` guards `undefined`, `NaN`, and `Infinity`, but a finite out-of-range epoch value can still make `new Date(ms).toISOString()` throw `RangeError` from a logging/event path.
- The naming split between `nextAtHuman` and `nextRunAt` should be resolved or intentionally accepted before this becomes a structured-log/event contract.
