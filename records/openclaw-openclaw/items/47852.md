## Summary

Keep open. Current main has a partial per-agent parameter surface via flat `agents.list[].params`, and that covers `temperature`/`maxTokens`, but it does not accept the requested `generation` block or generically forward `topP`/`top_p`, `topK`/`top_k`, presence penalties, or frequency penalties across the agent stream path. Related #32496 is the closest open sampling-parameter passthrough item, while #71350 and #59168 are broader adjacent design discussions, not exact replacements.

## Best Possible Solution

Keep this open until maintainers choose the public config contract. The narrow path is to extend the existing `agents.list[].params` contract, or add a documented `agents.list[].generation` alias if that shape is preferred, then normalize camelCase and snake_case sampling keys and forward only supported fields per provider/transport. Coordinate the sampling-field forwarding with open #32496, and keep #71350/#59168 linked as broader design context rather than treating them as replacements.

## Risks / Open Questions

- Closing this now would lose the remaining requested fields: top-p/top-k and penalties are still not provider-generically available from per-agent config.
- Any implementation should preserve the existing `agents.list[].params` merge precedence and avoid breaking users who already use flat params for `temperature`, `maxTokens`, cache retention, and transport options.
- Provider support varies; blindly forwarding unsupported sampling fields may cause upstream API errors, so the public contract should define normalization and provider-specific filtering or warnings.
