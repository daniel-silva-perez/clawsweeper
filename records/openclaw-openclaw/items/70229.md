## Summary

Keep open. Current main still hard-codes Docker `--security-opt no-new-privileges` and has no `sandbox.docker.noNewPrivileges` config surface, so this PR remains a relevant implementation candidate for the linked sandbox failure. It needs a refresh against current main's dangerous-config collector and maintainer/security review because the opt-out intentionally weakens sandbox hardening.

## What This Changes

The PR adds a default-on `sandbox.docker.noNewPrivileges` config field, conditionally omits Docker `--security-opt no-new-privileges`, documents the compatibility escape hatch, updates config schema/labels, and adds resolver, create-args, and dangerous-config tests.

## Best Possible Solution

Land a refreshed default-on Docker sandbox compatibility knob that only omits `--security-opt no-new-privileges` when explicitly configured false, documents it as a break-glass host-compatibility setting, and reports the false value through the current core dangerous-config diagnostics.

## Reproduction Assessment

Yes. The linked bug provides a focused Docker command that fails only with `--security-opt no-new-privileges`, and current main unconditionally supplies that flag when creating sandbox containers.

## Solution Assessment

Yes in direction, but not as the current branch. A default-on config knob with dangerous-config reporting is the narrowest maintainable fix for incompatible hosts, but the PR must be rebased onto current main's collector and reviewed for the sandbox security tradeoff.
