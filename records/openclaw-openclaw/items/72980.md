## Summary

Keep this PR open. Current main still lacks the requested env marker, but the proposed patch introduces a security-sensitive runtime contract and has a concrete node-host scope mismatch that should be resolved before merge; the broader open #40423 also needs maintainer product-shape consideration.

## What This Changes

This PR injects `defaults.sessionKey` as `OPENCLAW_SESSION_KEY` into exec environments and adds a new session-key exec test file covering gateway injection, absence, and caller override behavior.

## Best Possible Solution

Define the runtime marker contract once, then either land this narrow exec-only path with an explicit non-node guard, docs/security notes, and gateway plus sandbox tests, or fold the work into #40423 if maintainers want one shared env resolver across supported child-process surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard session-key injection away from node hosts:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: The PR documents `host=node` as out of scope, but the new block runs before the node branch. Current node execution sends the remote env from `params.requestedEnv` only while approval/allowlist analysis reads `params.request.env`, so a node command can be analyzed with `OPENCLAW_SESSION_KEY` present and then run remotely without that variable. Gate the injection to gateway/sandbox paths or move it after the node return.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency or CI supply-chain change, but it deliberately exposes a session identifier to every affected exec child process and needs maintainer approval of that security contract.

Concerns:

- **[medium] Session key is exposed through subprocess env:** `src/agents/bash-tools.exec.ts:1605`
  - body: The new env variable makes the agent session key available to spawned commands. That is useful for signed-session integrations, but env values are commonly echoed, logged, traced, inherited by child processes, or captured in crash diagnostics, so this should be documented and explicitly accepted before merge.
  - confidence: 0.88
