## Summary

Keep open. Current main still only supports `streamTo="parent"`, so the requested ACP thread streaming is not implemented. The PR is useful source work, but the proposed target resolver has a concrete Slack/Mattermost thread-routing regression already called out in review comments, and it needs to be rebased onto the newer conversation-resolution code before it can be considered merge-ready.

## What This Changes

The PR adds `streamTo: "thread"` for ACP `sessions_spawn`, relays ACP progress to a bound thread via gateway `send`, updates ACP spawn/tool tests and docs, and includes follow-up test fixes for channel model overrides and browser trash path expectations.

## Best Possible Solution

Rebase the feature onto current main, implement `streamTo="thread"` through the existing conversation-resolution and delivery-target helpers, preserve `streamTo="parent"` behavior, and add a regression covering Slack/Mattermost current-placement bindings where the binding lacks `parentConversationId` but the requester origin has the parent channel.

## Reproduction Assessment

Yes. The blocking PR bug is reproducible by static path inspection: a Slack/Mattermost current-placement ACP thread bind can record only the thread timestamp as `conversationId`, then the PR resolver falls through to `channel:<thread_ts>` without `threadId`.

## Solution Assessment

No, not as proposed. The feature direction is reasonable, but the resolver should fall back to the requester parent channel or current-main conversation-resolution helpers before sending direct progress messages to Slack/Mattermost threads.
