## Summary

Keep open. Current main still uses UTF-16 string length for Markdown table column padding, so the CJK Telegram table problem remains; the linked #55512/#55596 path covers only that table half and is still open. The paragraph-break request is separate and not implemented by the Telegram formatter today, so this should not be closed as implemented or as a pure duplicate.

## What This Changes

The issue requests better Telegram formatting for agent messages by preserving paragraph breaks and fixing CJK-aware Markdown table alignment.

## Best Possible Solution

The end state should be a display-width-aware shared table renderer, or an equivalent landed #55596-style fix, plus a deliberate Telegram paragraph policy that preserves code, lists, and intentional soft wraps with focused formatter and delivery tests.

## Reproduction Assessment

Yes. The CJK table bug has a high-confidence static reproduction because code-mode table padding uses `cell.text.length`; the paragraph behavior is also reproducible at formatter level because single Markdown softbreaks remain single `\n`, while only true paragraph closes produce blank-line separation.

## Solution Assessment

No for the suggested blanket Gateway post-processing. The table half should be fixed in the shared Markdown table renderer, while the paragraph half needs a narrower Telegram formatting decision or prompt policy rather than globally converting every single newline.
