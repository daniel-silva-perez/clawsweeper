## Summary

Keep this issue open. Current main still uses character count, not display width, when rendering markdown tables as code blocks, so the CJK table-alignment half remains real and is already tracked by the open #55512 plus draft PR #55596. The paragraph-break request is not covered by that canonical table item, and a blanket single-newline postprocessor would be risky for lists, code blocks, and intentional line breaks.

## What This Changes

The issue asks Telegram delivery to preserve readable paragraph spacing in agent replies and align markdown tables containing CJK characters.

## Best Possible Solution

Keep this open until the maintainer path is split or clarified: land a width-aware markdown table renderer through the canonical #55512/#55596 path, and handle Telegram paragraph readability with a channel-local, markdown-aware prompt/formatting approach that preserves code, lists, and intentional line breaks.

## Reproduction Assessment

Partially yes. The CJK table issue is reproducible from current source because code-mode table padding uses cell.text.length, but the paragraph wall-of-text symptom depends on model output and Telegram rendering and was not live-verified in this read-only review.

## Solution Assessment

Unclear. A width-aware table renderer is the right direction for CJK alignment, but blanket newline postprocessing is not the best paragraph fix; Telegram-specific formatting guidance or a conservative markdown-aware formatter would be safer.
