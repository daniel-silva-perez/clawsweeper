## Summary

Current main does not fully implement the requested Telegram formatting fixes. The CJK table portion is already covered by open issue #55512 and draft PR #55596, but the paragraph-break complaint is a separate live behavior: markdown soft/hard breaks are still rendered as single newlines, and Telegram delivery sends the rendered HTML without a post-processing step that promotes single newlines to blank-line paragraph breaks.

## Best Possible Solution

Keep this open. Treat the CJK table portion as tracked by #55512/#55596, and either split or implement the paragraph-break behavior separately in the Telegram formatter/delivery path with regression tests. A complete fix should cover both `extensions/telegram/src/format.ts` delivery output and `src/markdown/ir.ts` table code rendering, without changing global markdown semantics accidentally.

## Risks / Open Questions

- Promoting every single newline to a blank-line paragraph could change intentional line breaks in lists, code-like output, compact status messages, and streaming previews; this likely needs a narrowly scoped Telegram formatter policy plus tests.
- The CJK table fix should use display-width-aware measurement in the shared markdown table renderer or a channel-specific rendering path; relying on a transitive `string-width` dependency would need normal dependency review if made direct.
