## Summary

Keep open. Current main still has a credible Google native transport path where a configured Google model id shaped as `google/<model>` can survive resolution and then be sent as `models/google/<model>`, matching the reported 404 class and the later confirming config comment. The work should be consolidated with the overlapping open prefix-normalization PRs instead of closed as implemented.

## What This Changes

The issue reports that Google Gemini models that work in a main session fail with a 404 when used through sub-agent spawn or session reset.

## Best Possible Solution

Keep this open and land one narrow Google/provider-model normalization fix that strips only redundant active-provider prefixes before Google model matching and native Google URL construction, preserves valid nested ids for wrapper providers such as OpenRouter, and adds regression coverage for configured Google model entries plus sub-agent spawn/reset.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to configure `models.providers.google.models[].id` as `google/gemini-3-flash-preview` or `google/gemini-3.1-flash-lite-preview`, resolve it through the Google provider, and observe that current URL construction can build `models/google/<model>` for the Google Generative AI API.

## Solution Assessment

No. Current main is not the best solution because it still preserves the redundant same-provider prefix into the native Google transport path; the safer fix is scoped prefix stripping coordinated with #41398 and #41517.
