## Summary

Keep open. Current main still has a credible path where a configured Google model id like `google/gemini-3.1-flash-lite-preview` is matched as an inline Google model and then preserved into the native Google Generative AI request path, while overlapping prefix fixes remain open rather than shipped.

## What This Changes

The issue reports that Google Gemini models can work in the main session but fail with a 404 during sub-agent spawn or session reset when a provider-prefixed Google model id is sent to the Google API.

## Best Possible Solution

Land one focused Google model-resolution fix that strips only redundant active-provider prefixes before native Google model matching and URL construction, preserves valid provider-native nested ids for wrapper providers, and adds regression coverage for configured Google model entries plus sub-agent spawn/reset inheritance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure the Google provider with `api: google-generative-ai` and an inline model id `google/gemini-3.1-flash-lite-preview`, resolve a bare inherited Google model through the inline path, and observe that current URL construction can turn the preserved id into `models/google/gemini-...`.

## Solution Assessment

No. Current main has partial stored-override protection, but it does not fully solve the configured inline Google model path; the best next step is to consolidate the overlapping prefix-normalization PRs into a narrow same-provider-prefix fix.
