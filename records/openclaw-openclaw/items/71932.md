## Summary

Keep open. The latest discussion includes a concrete Google provider config with provider-prefixed model IDs, and current main still has a credible path that preserves `google/<model>` into the native Google request URL as `models/google/<model>`. The explicit sub-agent model override split helps one path, but it does not cover inline configured Google model entries; overlapping open PRs #41398 and #41517 should be consolidated instead of closing this issue or queueing another duplicate fix.

## What This Changes

The issue reports Google Generative AI 404s when sub-agents or session reset use configured Gemini model IDs such as `google/gemini-3.1-flash-lite-preview`.

## Best Possible Solution

Land one consolidated Google-owned normalization fix that removes redundant active-provider prefixes before Google native transport URL construction and model matching, while preserving legitimate nested model IDs for wrapper providers. Add regression coverage for `sessions_spawn`/reset and inline `models.providers.google.models[].id` values written as `google/<model>`.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to configure a Google inline model with id `google/gemini-3-flash-preview`, resolve it through the Google provider, and observe that the native Google transport would build a `models/google/gemini-3-flash-preview` URL; a live debug capture would still be useful to confirm the reporter's exact sub-agent/reset flow.

## Solution Assessment

No, current main is not a complete solution for the reported class. The narrow maintainable solution is a scoped Google same-provider prefix normalization change, coordinated with the two already-open related PRs, rather than closing this as implemented.
