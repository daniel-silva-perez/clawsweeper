## Summary

Keep open. Current main still has a credible native Google Generative AI path where an inline configured Google model id shaped as `google/<model>` can be preserved into request URL construction as `models/google/<model>`, matching the reported 404 class and the later confirming config comment.

## What This Changes

The issue requests fixing Google Gemini sub-agent and session-reset 404s when provider-prefixed model ids are used with the Google Generative AI API.

## Best Possible Solution

Land one narrow Google-owned normalization fix that strips only redundant active-provider prefixes before Google model matching and native Google transport URL construction, while preserving valid cross-provider nested ids such as OpenRouter model refs.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to configure `models.providers.google.models[].id` as `google/gemini-3-flash-preview` or `google/gemini-3.1-flash-lite-preview`, resolve it through the Google provider, and observe that current URL construction would build `models/google/<model>`.

## Solution Assessment

No. Current main is not the best solution because it still preserves the redundant same-provider prefix into the Google native transport; the safer fix is scoped same-provider prefix stripping coordinated with the open related PRs.
