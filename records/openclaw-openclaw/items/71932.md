## Summary

Keep this issue open. Current main still has a credible native Google path where a provider-prefixed configured model id can be preserved as `google/<model>` and then sent to the Google Generative AI API as `models/google/<model>`, while the overlapping prefix-normalization PRs remain open rather than shipped.

## What This Changes

Reports that Google Gemini model refs such as `google/gemini-3.1-flash-lite-preview` work in the main session but fail with 404 during sub-agent spawn or session reset.

## Best Possible Solution

Land one focused Google/model-resolution fix that strips only redundant active-provider prefixes before native Google model matching and URL construction, preserves provider-native nested ids, and adds regression coverage for configured Google models plus sub-agent spawn/reset inheritance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a Google provider model entry as `google/gemini-3-flash-preview` or `google/gemini-3.1-flash-lite-preview` with `api: google-generative-ai`, resolve the inherited/bare Google model, and observe the native URL builder can construct `models/google/<model>`.

## Solution Assessment

No. Current main is not the best solution because it can preserve the redundant same-provider prefix into the native Google transport path; the safer fix is scoped prefix stripping coordinated with the existing open normalization PRs.
