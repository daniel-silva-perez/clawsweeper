## Summary

Keep open. Current main still silently filters same-agent group/channel sessions from `sessions_list` under the default `tree` visibility; the documented `agent` visibility is an operator workaround, not a fix for the setup-diagnostic mismatch described in the issue discussion.

## What This Changes

The issue requests that `sessions_list` surface active same-agent Telegram/Discord group or channel sessions, or at least provide a filter that can reveal them.

## Best Possible Solution

Define the intended session-tool UX for setup diagnostics, then add focused coverage. A maintainable fix would either safely expose same-agent group/channel sessions for `sessions_list` under an explicit mode or filter, or keep `tree` as the default while returning clear hidden-by-visibility metadata and pointing operators to `tools.sessions.visibility="agent"`.

## Reproduction Assessment

Yes, at source level. With default `tree` visibility, a requester session and a separate same-agent Telegram group or Discord channel session will pass through gateway enumeration but be rejected by `createSessionVisibilityGuard` unless the target is the requester or a spawned child; no test command was run because this review was read-only.

## Solution Assessment

Unclear. The current opt-in `tools.sessions.visibility="agent"` workaround is not the best diagnostic experience, but automatically widening the default may not be safe; the narrower solution is a maintainer-approved visibility/diagnostic contract plus regression coverage.
