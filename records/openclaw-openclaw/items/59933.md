## Summary

Keep open. Current main still has the reported mismatch under the default session-tool visibility contract: stored same-agent group/channel sessions can exist, but `sessions_list` silently filters sessions outside the current session tree unless `tools.sessions.visibility` is widened to `agent` or `all`. The Discord follow-up is useful narrowing because it shows persistence and CLI enumeration work while the tool layer still hides the other same-agent channel session.

## Best Possible Solution

Keep the issue open. Maintainers should decide whether `sessions_list` should include same-agent group/channel sessions by default, whether Telegram/Discord setup diagnostics should temporarily widen to `tools.sessions.visibility="agent"`, or whether the tool result should explicitly report that same-agent sessions were hidden by `tree` visibility. Add a focused regression test with a requester DM key and an existing same-agent Telegram/Discord group key, especially for `sessions_list({ kinds: ["group"] })`.

## Risks / Open Questions

- Changing the default from tree to same-agent-wide could expose unrelated same-agent users or channels, so the product/security tradeoff needs maintainer judgment.
- The current `agent` visibility mode is a workaround for operators who know to configure it, but it does not make initial Telegram/Discord group setup diagnostics self-explanatory by default.
