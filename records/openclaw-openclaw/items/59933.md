## Summary

Keep open. Current main still matches the reported same-agent group/channel visibility mismatch: `sessions_list` can receive stored sessions from the gateway, but the tool silently drops sessions outside the requester session tree before `kinds` filtering, while the documented default remains `tools.sessions.visibility="tree"`. The related Discord comment is useful corroboration because it narrows the problem to the tool layer rather than persistence or CLI enumeration.

## Best Possible Solution

Keep this issue open. Maintainers should decide whether `sessions_list` should include same-agent group/channel sessions by default, expose a clear hidden-session/visibility notice, or provide a setup-diagnostic path that temporarily uses `tools.sessions.visibility="agent"`. Add a targeted regression test for a requester DM/current session plus an existing same-agent Telegram or Discord group/channel session, especially with `kinds: ["group"]`.

## Evidence

- **Issue discussion narrows the symptom:** The issue reports same-agent Telegram group transcript files with valid messages/responses that are absent from `sessions_list`; the follow-up Discord comment reports the same family where the transcript and store entry exist and `openclaw sessions --agent main --json` sees the session, but tool-facing `sessions_list` hides the other same-agent channel session.
- **Tool filters by visibility before kind filters:** `sessions_list` calls gateway `sessions.list`, then creates a `createSessionVisibilityGuard` and skips entries whose keys are not allowed. The later `kinds` filter only runs after this skip, so `kinds: ["group"]` cannot surface a same-agent group/channel session hidden by default tree visibility.

## Risks / Open Questions

- Widening the default from `tree` to `agent` could expose unrelated same-agent users or channels, so the final behavior needs explicit product/security judgment.
- There is no focused regression test proving the desired diagnostic behavior for `sessions_list({ kinds: ["group"] })` with a requester DM key and a stored same-agent group/channel key.
