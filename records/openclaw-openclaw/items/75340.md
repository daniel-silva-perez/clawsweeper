## Summary

Keep open. Current main still routes plain `openclaw status --usage` through the fast status scan without the setup/runtime fallback, while the bundled channel read-only loaders intentionally skip runtime-deps registration on discovery paths; that matches the reported missing-module warnings and empty Channels table for packaged installs.

## What This Changes

The issue requests that packaged CLI status register or otherwise use managed bundled runtime dependencies before loading bundled channel status/setup modules so fast status can show configured channels without false missing-module warnings.

## Best Possible Solution

Keep plain status fast and read-only, but make its channel table use lightweight setup/manifest metadata for configured bundled channels or prepare the active packaged runtime-deps root before any runtime-dependent bundled channel import. Avoid turning plain status into a live probe or broad plugin preload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is visible: run packaged `openclaw status --usage` with configured bundled channels whose dependencies exist only under the managed runtime-deps stage; current main keeps the fast path off setup fallback and skips runtime-deps registration for bundled discovery loads.

## Solution Assessment

No current-main solution is sufficient yet. The narrowest maintainable fix is status/read-only-channel specific: show configured bundled channel rows without importing full channel runtime, or register the packaged runtime-deps root before the existing bundled import path that can require those packages.
