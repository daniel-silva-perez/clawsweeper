## Summary

Keep open. Current main still has the reported fast-status/runtime-deps gap: `openclaw status --usage` does not change the status scan mode, so normal text status keeps setup/runtime fallback disabled while bundled channel loaders deliberately skip managed runtime-deps registration on discovery paths. The reporter's workaround and the earlier ClawSweeper comment line up with the current source and packaged-install docs.

## What This Changes

The issue requests that packaged `openclaw status --usage` register managed bundled runtime dependencies before loading bundled channel status/setup modules so configured channels appear without manual `NODE_PATH` or `--deep`.

## Best Possible Solution

Keep plain status fast and read-only, but make configured bundled channel rows work in packaged installs by using lightweight manifest/setup metadata where possible or by narrowly registering the active managed runtime-deps root before the existing bundled channel setup/status import path. Avoid making normal status a live probe or broad plugin preload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is visible: run packaged `openclaw status --usage` with configured bundled channels whose dependencies exist only under the managed runtime-deps root; current main keeps setup/runtime fallback disabled and the bundled discovery loads skip runtime-deps registration.

## Solution Assessment

Yes, with a narrow boundary. The right fix is status/read-only-channel specific: either build configured bundled channel rows from lightweight metadata or register the packaged runtime-deps root before the existing status/setup import path, without changing fast status into a live channel check.
