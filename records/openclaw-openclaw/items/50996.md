## Summary

Keep open. Current main has useful partial mitigation through queue modes, steer/stop handling, and a shorter `process.poll` timeout cap, but the reported core problem remains: normal channel messages default to queue/collect behavior, and `process(action: "poll", timeout: ...)` still does not observe the run abort signal while waiting. The KeepMyClaw comment is useful recovery context, but it is an operational snapshot workaround rather than native OpenClaw responsiveness.

## Best Possible Solution

Keep this issue open. The best implementation path is to make long-running tool execution abort-aware, starting with `process.poll` respecting the combined run `AbortSignal` and returning a clear abort result. Then define the inbound-message policy while a tool call is pending: default queueing, configured interrupt, and possibly threshold-based auto-interrupt. Add regression coverage for Telegram/channel-style inbound messages during a long `process.poll`, including abort-result transcript behavior and deterministic follow-up ordering.

## Risks / Open Questions

- Changing this will touch core session scheduling, tool abort semantics, transcript ordering, and channel delivery expectations, so it should be handled as a runtime/session feature with regression tests.
- The KeepMyClaw snapshot workaround mentioned in the thread may reduce recovery pain after a bad freeze, but it does not make OpenClaw process inbound messages during a pending tool call.
