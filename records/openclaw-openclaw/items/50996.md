## Summary

Keep open. Current main has partial mitigation through configurable message queue modes, including `messages.queue.mode: "interrupt"`, but the reported core problem is not fully implemented: normal channel behavior still queues busy-session messages by default, and a pending `process(action: "poll", timeout: ...)` tool call does not observe the run abort signal while sleeping. The existing comment’s KeepMyClaw link is useful as an operational snapshot workaround, but it does not make OpenClaw respond to inbound messages during a long tool call.

## Best Possible Solution

Keep this issue open. The best path is to make long-running tool execution abort-aware, starting with `process.poll` respecting the combined run `AbortSignal`, then define a session policy for inbound messages while a tool call is pending: default queueing, configured interrupt, or a threshold-based auto-interrupt. Add regression coverage around Telegram/channel inbound messages arriving during a long `process.poll`, plus transcript ordering and abort-result behavior.

## Risks / Open Questions

- Implementing this touches core run scheduling, tool abort semantics, transcript ordering, and channel delivery behavior, so it should be handled as a runtime/session feature rather than a plugin-only cleanup.
- The existing `interrupt` queue mode is close enough to be easy to over-credit, but it does not make `process.poll` itself abortable and is not the default Telegram/channel behavior described in the report.
