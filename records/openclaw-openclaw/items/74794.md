## Summary

Keep open. Current main still treats a non-empty `delivery.channel` as a resolved announce target, so the PR addresses a real cron delivery inference bug; the diff is focused and I did not find a blocking correctness issue in review. It still needs maintainer review plus the targeted cron-tool test and changed gate before merge.

## What This Changes

The PR changes cron-tool add delivery inference to infer announce `delivery.to` when a caller provides `delivery.channel` without `delivery.to`, preserves explicit `delivery.to`, and adds three regression tests.

## Best Possible Solution

Land the focused cron-tool inference fix after maintainer review and validation, keeping the existing boundaries that explicit `delivery.to` wins and `webhook`/`none` delivery modes do not infer chat announce targets.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is a cron `add` for an isolated `agentTurn` from `agent:main:slack:channel:general` with `delivery: { mode: "announce", channel: "slack" }`; current main leaves `to` unset because the channel-only delivery trips the existing `hasTarget` gate.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I found: it changes the inference gate to explicit `to`, preserves explicit targets, and adds regression coverage without changing scheduler, resolver, dependency, or public schema surfaces.
