## Summary

Current main still only partially satisfies the request. Gateway `sessions.create`, `sessions.patch`, labels, and the separate Sessions page management flows exist, and WebChat can switch among existing sessions, but the visible WebChat New Session action still sends `/new`, which current code and tests define as resetting the current transcript. Open maintainer-labeled PR #73725 remains the active candidate for the narrow WebChat create/switch slice; agent-controlled create/relabel and broader tab/sidebar/rename/delete session management still need maintainer scoping.

## Best Possible Solution

Keep this issue open as the canonical WebChat multi-session tracker. Review, adjust if needed, and land or close PR #73725 for the narrow New Chat -> `sessions.create` create/switch flow. After that, explicitly split or scope the remaining agent-controlled create/relabel API plus broader rename/delete/sidebar/tab session-management work.
