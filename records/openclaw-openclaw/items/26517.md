## Summary

Keep this issue open. Current main has backend session labels, `sessions.create`, `sessions.patch`, and a WebChat selector for existing sessions, but the visible WebChat New Session button still sends `/new`, which current tests define as resetting the same session. Open PR #73725 remains the active implementation candidate for the narrow WebChat create/switch slice; broader agent-controlled create/relabel and session management scope is still unresolved.

## Best Possible Solution

Keep this issue open as the canonical tracker. Review, fix if needed, and land or close PR #73725 for the narrow WebChat New Session -> `sessions.create` create/switch flow. After that, maintainers should split or explicitly scope the remaining agent-controlled create/relabel API and broader rename/delete/sidebar session-management work.
