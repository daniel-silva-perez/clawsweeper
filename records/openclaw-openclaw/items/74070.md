## Summary

Current main still matches the reported ACP background-task behavior: ordinary successful non-blocked output is accumulated as `progressSummary`, capped at 240 characters, but `resolveBackgroundTaskTerminalResult` returns `{}` unless it recognizes a blocked/permission case, so `terminalSummary` is written as `null` and terminal delivery can fall back to the bare `Background task done` message. Keep this issue open; the fix should be narrow and preserve the existing anti-leak guard for raw progress chatter.

## Best Possible Solution

Keep this open and land a focused ACP background-task fix: derive a sanitized terminal summary for useful successful ACP output, preserve blocked outcome detection, avoid leaking internal progress-only text, and raise the ACP progress accumulation cap to a more useful bounded length with regression coverage.

## Clownfish Work Prompt

Fix the ACP background-task completion summary bug reported here. In current `main`, successful parent-owned ACP runs accumulate non-thought `text_delta` output into `taskProgressSummary`, but `resolveBackgroundTaskTerminalResult` returns `{}` for ordinary non-blocked output and `markBackgroundTaskTerminal` stores `terminalSummary: null`, so users see only the bare `Background task done` message. Update `src/acp/control-plane/manager.core.ts` so useful successful ACP output can become a sanitized `terminalSummary`, while preserving existing blocked/permission detection and the current guard against leaking generic internal progress chatter. Strip/suppress exact or trailing silent announce tokens such as `ANNOUNCE_SKIP`/`NO_REPLY` where appropriate, and do not dump raw tool/progress noise into terminal channel messages. Raise `ACP_BACKGROUND_TASK_PROGRESS_MAX_LENGTH` from 240 to a more useful bounded value around 1200 unless code review finds a better local constant. Add focused tests in `src/acp/control-plane/manager.test.ts` and/or `src/tasks/task-registry.test.ts` for: ordinary successful ACP output sets a useful terminal summary; blocked permission output still maps to `terminalOutcome: "blocked"`; the existing no-leak progress-detail behavior remains protected; long progress accumulation no longer clips at 240. Add a user-facing changelog fix entry under the active version with `Thanks @dcapclaw`. Do not change broad task delivery semantics, A2A skipping, plugin APIs, or unrelated task registry behavior.
