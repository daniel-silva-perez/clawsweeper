## Summary

Keep open. Current `origin/main` still has the reported ACP background-task behavior, and PR #74103 is still open and explicitly targets this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for successful ACP background-task completions to carry a useful `terminalSummary` derived from progress output and for the progress summary cap to increase from 240 characters to about 1200.

## Best Possible Solution

Land PR #74103 or an equivalent narrow ACP manager patch that derives a sanitized successful terminal summary, preserves blocked-outcome handling, raises the bounded progress cap, and adds regression coverage for both useful completion summaries and internal-detail suppression.

## Reproduction Assessment

Yes. The static current-main path is high confidence: non-thought ACP output builds `progressSummary`, successful non-blocked text returns `{}`, the task records `terminalSummary: null`, and terminal delivery formats only `terminalSummary`.

## Solution Assessment

Unclear for the currently linked PR as-is. The fix belongs in the ACP background-task terminal-result boundary, but the safest maintainable solution should include tests and sanitization constraints so routine or internal progress text is not exposed in channel notifications.
