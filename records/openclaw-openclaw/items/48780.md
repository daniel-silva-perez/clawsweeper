## Summary

Keep open. Current main still lacks a `</arg_value>>` argument sanitizer or provider-specific repair for valid JSON tool arguments containing that suffix, and the active `exec` and `read` paths still forward the received command/path strings unchanged. An open implementation PR already targets this issue with closing syntax, so closing or queueing a separate repair would be premature.

## What This Changes

The issue reports Windows `exec` and `read` tool arguments being corrupted with a trailing `</arg_value>>` artifact when using the Qwen/DashScope route.

## Best Possible Solution

Keep this issue open until PR #48792 or a replacement lands a narrow current-main fix at the active tool-argument or provider repair boundary, with regression coverage for the reported suffix and negative coverage for legitimate XML-like payload text.

## Reproduction Assessment

Yes for the local boundary behavior: if `exec` receives `command: "echo test</arg_value>>"` or `read` receives a suffixed `path`, current main forwards that string unchanged. The live Windows/DashScope emission itself was not reproduced in this read-only pass.

## Solution Assessment

No current-main solution exists yet. The maintainable fix should be narrow to the malformed tool-argument boundary or affected provider stream path; broad record-wide stripping is riskier unless tests prove it preserves valid content.
