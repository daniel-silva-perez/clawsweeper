## Summary

Keep open. Current main still leaves the Feishu SDK HTTP token path on the Lark SDK default axios instance with only timeout/User-Agent wrapping, so the reported proxy-triggered tenant_access_token failure is still plausible and not covered by tests or docs.

## What This Changes

The issue asks OpenClaw to fix Feishu message delivery failures where the Lark SDK token request inherits HTTP proxy environment variables and surfaces a confusing tenant_access_token destructuring error.

## Best Possible Solution

Add a Feishu-owned HTTP proxy policy at the Lark SDK httpInstance seam that preserves the existing timeout and User-Agent behavior, explicitly defines direct versus ambient proxy behavior for Feishu/Lark/custom domains, honors NO_PROXY where selected, and documents the supported configuration/troubleshooting path.

## Reproduction Assessment

Yes. The issue gives concrete env/setup steps, and current source plus the published SDK show the same failure path: an env-proxied SDK token POST can fail, the SDK catch returns undefined, and tenant_access_token is then destructured from undefined.

## Solution Assessment

No for the issue’s proposed blanket SDK patch or unconditional proxy:false default. The safer solution is a Feishu plugin policy that distinguishes NO_PROXY/direct/ambient or managed proxy modes, with regression coverage, because proxy routing can be required for some deployments and security postures.
