## Summary

Keep open. Current main still lacks this PR’s exact direct raw final-output delivery, but the patch should not merge as-is because it can corrupt streamed whitespace, duplicate the existing ACP task terminal delivery, and bypass the documented parent-completion handoff boundary.

## What This Changes

This PR adds routeReply-based final and error delivery from the ACP parent stream relay for non-threaded parent sessions, plus tests for direct-chat success, threaded suppression, lookup failure, and error delivery.

## Best Possible Solution

Use one delivery owner for ACP completion. Prefer extending the task-completion or parent handoff path so the final visible child result is delivered with sanitization, thread policy, idempotency, and duplicate suppression; if maintainers explicitly choose direct relay delivery, rework it to preserve streamed whitespace, suppress the generic terminal notification for the same run, and cover success, error, truncation, threaded, and duplicate paths.

## Reproduction Assessment

Yes for the review defects. A non-threaded direct relay receiving assistant deltas like "hello", " ", "world" would lose the standalone space before the new direct buffer, and the existing task-registry quiet-relay test shows the same run can still send the generic terminal notification while the PR adds routeReply.

## Solution Assessment

No. The PR is a plausible implementation candidate, but the direct routeReply side path is not the narrowest maintainable fix unless maintainers explicitly choose to bypass the current task-completion handoff; the safer direction is to carry final child output through the existing completion model or add duplicate suppression and policy checks to a direct relay.
