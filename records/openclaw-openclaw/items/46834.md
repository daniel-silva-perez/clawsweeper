## Summary

Keep open. Current main still does not implement the PR's specific non-threaded direct-chat delivery of the child ACP final visible output, and the proposed branch needs maintainer rework because it can corrupt whitespace and collide with the newer task-terminal delivery path.

## What This Changes

The PR adds best-effort direct-channel final/error delivery from the ACP parent stream relay for non-threaded parent sessions, plus tests for direct-chat success, threaded suppression, lookup failure, and error delivery.

## Best Possible Solution

Choose one current-main owner for ACP completion delivery. Prefer carrying the final visible child result through the existing task-completion/parent handoff with sanitization and duplicate suppression; if maintainers explicitly want a direct relay, rework it to preserve whitespace, avoid duplicate terminal notifications, respect threaded contexts and silent/sanitization policy, and cover truncation/error behavior in regression tests.

Full review comments:

  - body: Current main already sends an ACP task terminal notification to the requesterOrigin when the same run ends.   - confidence: 0.78

  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a new direct outbound delivery path for raw ACP child output that needs maintainer review against the existing completion-handoff and sanitization model.

Concerns:

- **[medium] Direct ACP output bypasses parent completion handoff:** `src/agents/acp-spawn-parent-stream.ts:329`
  - body: Parent-owned ACP sessions are documented to report through task completion so the parent can rewrite the child result for user-facing delivery. The new routeReply call sends accumulated child output directly to the originating channel, which may bypass that policy boundary even though routeReply still normalizes payload text.
