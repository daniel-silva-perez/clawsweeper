## Summary

Keep open. Current main still does not implement this PR's direct raw final-output relay for explicit non-threaded `streamTo="parent"` chats, but the proposed patch should not merge as-is because it can corrupt streamed whitespace, duplicate existing ACP task terminal delivery, and bypass the documented parent-completion handoff boundary.

## What This Changes

This PR adds routeReply-based final and error delivery from the ACP parent stream relay for non-threaded parent sessions, plus tests for direct-chat success, threaded suppression, lookup failure, and error delivery.

## Best Possible Solution

Use one delivery owner for ACP completion. Prefer extending the task-completion or parent handoff path so the final visible child result is delivered with sanitization, thread policy, idempotency, and duplicate suppression; if maintainers explicitly choose direct relay delivery, rework it to preserve streamed whitespace and suppress duplicate terminal notifications.

## Reproduction Assessment

Yes for the review defects. A non-threaded direct relay receiving assistant deltas like `hello`, ` `, `world` would lose the standalone space before the new direct buffer, and the current quiet-relay task test shows the same run can still send the generic terminal notification while the PR adds `routeReply`.

## Solution Assessment

No. The PR is a plausible implementation candidate, but the direct `routeReply` side path is not the narrowest maintainable fix unless maintainers explicitly choose to bypass the current task-completion handoff.
