## Summary

Keep open. Current main does not already implement this PR's intended direct final-output relay for non-threaded `streamTo: "parent"` ACP chats, but the proposed branch introduces correctness and security-boundary issues that should block merge until maintainers decide whether raw child output should bypass the existing parent-owned task completion path.

## What This Changes

The PR adds best-effort final/error `routeReply` delivery from the ACP parent stream relay for non-threaded parent delivery contexts, plus tests for direct chat delivery, threaded suppression, lookup failure, and error delivery.

## Best Possible Solution

Keep a single ACP completion owner. If maintainers want final child text delivered for non-threaded direct chats, implement it through the task-completion/announce owner or explicitly revise that owner with sanitization, idempotency, whitespace-preserving aggregation, truncation coverage, and suppression of duplicate terminal notifications.

## Reproduction Assessment

Yes. On the PR branch, a relay receiving assistant chunks like `hello`, ` `, `world` drops the standalone whitespace before direct delivery; a non-threaded ACP run with a pending task terminal notification can also send both the new direct reply and the existing terminal task message.

## Solution Assessment

No. The side-path `routeReply` call is not the narrowest maintainable fix against current main's parent-owned ACP completion model; extending or deliberately replacing the task-completion path is safer.

Overall correctness: patch is incorrect

  - body: The new direct-reply buffer is appended only after the `!delta.trim()` early return, so standalone spaces or newlines from streaming are dropped. A split like `"hello"`, `" "`, `"world"` can become `"helloworld"` in the user-visible final reply.
  - confidence: 0.86

  - body: On lifecycle end this sends raw child output directly, but the ACP task remains pending for the existing terminal delivery path. Current main will still send the task completion notification to the same requester origin, producing two completion messages for one child run.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff adds a new direct outbound ACP child-output path that needs maintainer review against the parent-owned completion, sanitization, and idempotency model.

Concerns:

- **[medium] Direct ACP output bypasses completion handoff:** `src/agents/acp-spawn-parent-stream.ts:296`
  - body: Parent-owned ACP sessions are documented to report through task completion so the parent can rewrite child results for user-facing delivery. This new `routeReply` call sends accumulated child output directly to the originating channel, bypassing that handoff and its duplicate-suppression/sanitization policy.
