## Summary

Keep open. Current `main` and latest `v2026.4.27` still emit `multiple_gateways` solely from the number of reachable probe targets, while the SSH tunnel path keeps a configured remote target when the URLs differ, so the reported false-positive path remains source-backed.

## What This Changes

The issue requests that `openclaw gateway probe` distinguish multiple transports to one logical gateway from multiple distinct reachable gateways.

## Best Possible Solution

Fix the gateway-status warning path by grouping reachable targets with a conservative non-null gateway self identity, suppressing or downgrading `multiple_gateways` only when all reachable targets identify as the same gateway, preserving the warning for distinct or unknown identities, and updating focused tests plus CLI docs/changelog.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass two reachable `GatewayStatusProbedTarget` entries with the same non-null `self` identity to `buildGatewayStatusWarnings`; current code still emits `multiple_gateways` because it only checks `reachable.length > 1`.

## Solution Assessment

Yes. Identity-aware warning classification is the narrowest maintainable fix; skipping the configured remote target when a tunnel exists would remove a useful direct-path check and only mask this transport pairing.
