## Summary

Keep open. Current main still builds the warning from the number of reachable probe targets, so an SSH tunnel and a configured remote URL that both reach the same gateway can still produce the reported false positive. The report is recent, concrete, and narrow enough for a focused gateway-status fix.

## What This Changes

The requested change is for `openclaw gateway probe` to distinguish one logical gateway reached through two transports from multiple distinct reachable gateways.

## Best Possible Solution

Keep this open and fix the gateway-status warning path. Classify reachable targets by a conservative non-null gateway self identity, suppress or downgrade `multiple_gateways` when all reachable targets share that identity, preserve the warning for distinct or unknown/mixed identities, and add focused tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: pass two reachable `GatewayStatusProbedTarget` entries with the same non-null `self` to `buildGatewayStatusWarnings`; current main still emits `multiple_gateways` from `reachable.length > 1`. The live repro is the reporter's SSH tunnel plus direct `gateway.remote.url` setup.

## Solution Assessment

Yes. Identity-aware warning classification is the narrowest maintainable fix; skipping the configured remote target when a tunnel is active would remove a useful direct-path check and only masks this specific transport pairing.

## Repair Work Prompt

Fix the `openclaw gateway probe` false-positive `multiple_gateways` warning reported in https://github.com/openclaw/openclaw/issues/69135. Current main can probe an SSH tunnel target and a configured `gateway.remote.url` that both reach the same logical gateway; `buildGatewayStatusWarnings` then warns solely because two targets are reachable. Implement a conservative gateway identity classifier using the existing `GatewayStatusProbedTarget.self` data from `pickGatewaySelfPresence`. Suppress or downgrade the `multiple_gateways` warning when all reachable targets have the same strong non-null identity, and keep the warning for distinct identities, missing identity, or mixed/partial identity cases. Prefer the owner path under `src/commands/gateway-status/`; do not remove configured remote probing just because an SSH tunnel is active, and do not fold in the separate daemon/status severity issue from #46930. Add focused tests in `src/commands/gateway-status/output.test.ts` for same-identity SSH tunnel plus configured remote URL and for genuinely distinct or unknown identities. Update user-facing docs if the warning semantics or copy changes, and add a single-line changelog fix crediting @sk8man121. Validate with targeted gateway-status tests and the changed gate in Testbox before handoff.
