## Summary

Keep open. Current main still does not implement the named `readonly` tool profile requested by this PR, and the PR is not obsolete. The proposed change remains useful, but it needs to be refreshed against the current tool-profile contracts and generated schema surfaces before it could land safely.

## Best Possible Solution

Keep this PR open and ask the author or a maintainer to refresh it against current main. The best path is to add `readonly` consistently across `src/agents/tool-catalog.ts`, `src/config/zod-schema.agent-runtime.ts`, `src/config/types.tools.ts`, generated config schema artifacts, Gateway protocol/server catalog types, UI profile option fallbacks if needed, docs, and focused tests that prove both allowed non-mutating tools and denied mutating tools. The docs should describe it as a non-mutating/data-readable observer profile rather than a confidentiality boundary.

## Risks / Open Questions

- Merging the PR as described without updating generated config schema artifacts would leave schema-driven clients and validation/help surfaces out of sync with runtime parsing.
- The PR must update the shared exported config `ToolProfileId` contract, not only the local agent catalog type, or TypeScript config consumers will still reject `tools.profile: "readonly"`.
- Security review: the changed files are limited to tool catalog/config/protocol/docs/tests, with no workflow, dependency, lockfile, install, or release-script changes in the provided PR file list. The main security/product risk is wording and expectation management: `readonly` is non-mutating, but it still grants data access to files, memory, session history/status, web search/fetch, image understanding, and agent listing.
