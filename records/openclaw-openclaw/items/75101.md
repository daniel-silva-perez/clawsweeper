## Summary

Keep open. The proposed security control addresses a valid problem, but the patch is not merge-ready because the new deny gate is wired only into the node `system.run` handler while the ordinary gateway exec path remains separate and does not receive or enforce `denyPathPatterns`.

## What This Changes

The PR adds a `tools.exec.denyPathPatterns` config field, schema/docs/changelog entries, a new deny-path matcher helper, node-host enforcement before allowlist/approval checks, and focused unit tests.

## Best Possible Solution

Land a version where `denyPathPatterns` is a shared host-exec policy input evaluated before allowlist, approval, safeBins, and elevated-full execution for both gateway and node hosts, with config threading, gateway/node integration tests, and docs that match the accepted v1 scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `tools.exec.denyPathPatterns` and use normal `host=gateway` exec: current gateway execution does not receive the new field and never calls the new matcher, so the proposed deny gate would not run there.

## Solution Assessment

No. The narrow helper is reasonable, but the enforcement point is not the best solution because it protects only node `system.run`; the safer design is a shared pre-execution deny phase used by every host-exec path before any bypass or approval decision.
