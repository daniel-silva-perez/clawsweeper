## Summary

Keep open. Current main still lacks this exec path denylist, and the PR is a useful implementation attempt, but the current head still leaves concrete bypasses for shell-wrapper commands and bare relative secret filenames, so it should not merge as-is.

## What This Changes

The PR adds opt-in `tools.exec.denyPathPatterns` config/schema/docs/changelog support plus deny-path matching wired into gateway and node exec policy paths.

## Best Possible Solution

Land a maintainer/security-reviewed update that evaluates parsed argv segments and inner shell payloads before every host execution path, treats cwd-resolvable sensitive filenames as deny candidates, removes stale duplicate helper code, and backs the behavior with gateway, sandbox/elevated, node-host, schema, and docs tests.

## Reproduction Assessment

Yes. A high-confidence static repro exists against the PR head: configure `tools.exec.denyPathPatterns: ["~/.openclaw/secrets/**"]` and run gateway or elevated-full `bash -c "cat ~/.openclaw/secrets/foo.env"`; the new gate receives raw wrapper text at `src/agents/bash-tools.exec.ts:1622` and will not match the tilde path pattern. A second repro is `cat .env` with `**/.env`, which is skipped by `isPathLikeArg` before cwd resolution.

## Solution Assessment

No. The proposed boundary and config direction are plausible, but the current implementation is not the best fix until it reuses the existing shell-wrapper parsing seams and broadens relative-path candidate detection to match the documented behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Parse shell wrappers before deny matching:** `src/agents/bash-tools.exec.ts:1622-1623`
  - body: The pre-dispatch gate passes raw `params.command` as `shellPayload`, so `bash -c "cat ~/.openclaw/secrets/foo.env"` is tokenized as one quoted inner command with a leading `cat ` instead of a standalone path. With the documented `~/.openclaw/secrets/**` pattern, gateway/sandbox/elevated-full execution can still read the denied file.
  - confidence: 0.88
- **[P2] Treat bare relative secret names as path candidates:** `src/infra/exec-deny-path.ts:86-95`
  - body: `isPathLikeArg` rejects bare filenames such as `.env` and `credentials.json` because they lack a slash or tilde. The docs recommend `**/.env` and promise cwd-resolved relative matching, but `cat .env` is skipped before resolution and bypasses the configured denylist.
  - confidence: 0.86
