## Summary

Keep this issue open. Current main still lacks the requested configurable subagent notify-policy surface: subagent task registration does not pass a policy, the task registry still falls back to `done_only` for deliverable records, and the config schema/types/docs expose no `silent` or `inherit_from_parent` default. Current main has a narrower mitigation that suppresses most automatic terminal delivery for subagent task records, but that does not implement the requested user-configurable policy.

## What This Changes

Add a configurable default notification policy for subagent tasks spawned from cron jobs, including `silent`, `done_only`, and `inherit_from_parent` behavior.

## Best Possible Solution

Add a backwards-compatible `agents.defaults.subagents.notifyPolicy` setting, preserve current behavior when unset, resolve cron-run inheritance to `silent`, and keep ordinary sessions on the existing default unless a real parent task policy is available through a clean generic seam. Update schema/types/generated config metadata, public docs, and focused tests together.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a cron task is explicitly stored as `silent`, while a subagent spawned inside that cron session creates a pending session task with no `notifyPolicy`, causing the registry fallback to store `done_only`; the schema and docs confirm there is no config override.

## Solution Assessment

Yes, the issue’s narrow config-backed subagent default is a reasonable fix if scoped conservatively. Current main’s hardcoded subagent auto-delivery suppression reduces the notification symptom, but it is not the requested configurable policy and does not settle the inheritance contract.
