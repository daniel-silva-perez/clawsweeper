## Summary

Keep this issue open. Current main still has the reported mismatch: cron task records are explicitly silent, while default subagent task records omit a notification policy and resolve to `done_only`; the broader #64081 overlaps but does not fully settle the requested cron parent-inheritance behavior.

## What This Changes

The issue requests a configurable default notification policy for subagents so cron jobs can suppress or inherit subagent completion notifications.

## Best Possible Solution

Add a backwards-compatible notification policy resolver that preserves current `done_only` behavior when unset, supports explicit `silent` and `done_only`, and handles cron parent inheritance deliberately; alternatively fold the cron-specific behavior into the broader task-default design in #64081 with schema, docs, and tests aligned.

## Reproduction Assessment

Yes. A static source reproduction is high-confidence: a cron job creates a silent cron task, then `sessions_spawn` creates a deliverable subagent task without `notifyPolicy`, which the task registry resolves to `done_only`.

## Solution Assessment

Unclear. A subagent-level setting would solve the reported cron case, but #64081 may be a cleaner long-term API if core wants one configurable task notification policy surface across runtimes.
