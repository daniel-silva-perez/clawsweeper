## Summary

Keep this PR open, but do not merge it as-is. The patch is a useful subset for chat-template literal leaks, and the stale changelog-drop review appears resolved in the latest provided diff, but it closes the broader #69943 report while leaving raw `<tool_call>` XML, `NO_REPLY`, and role-marker artifacts on the session-memory persistence path.

## What This Changes

The PR exports the existing LLM special-token literal sanitizer, applies it when session-memory reads string and text-block transcript content, adds transcript regression tests, and adds a changelog entry.

## Best Possible Solution

Land a session-memory-specific sanitizer that treats persisted conversation content as untrusted before reuse: remove or elide chat-template tokens, raw tool-call XML/function-call scaffolding, `NO_REPLY`-only turns, and orphaned role-marker lines, with tests at the transcript read seam and the handler write seam while preserving clean prose.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction: a synthetic session JSONL assistant text containing raw `<tool_call>` XML or `NO_REPLY` but none of the listed special-token literals still flows through the current extraction path into the memory markdown; #69943 provides real deployment evidence for the same path.

## Solution Assessment

No. Reusing the existing literal sanitizer is a good component, but it is not the best complete fix for the issue this PR closes because it leaves non-token model artifacts that the reporter explicitly showed can continue the poisoning loop.
