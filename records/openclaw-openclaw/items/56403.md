## Summary

Keep open. The reported gap is still present on current main, but this PR is not merge-ready: the branch is dirty against current main, leaves contradictory docs, and adds a critical generic-repeat block that can stop progress-making repeated calls without no-progress evidence.

## What This Changes

The PR enables `tools.loopDetection.enabled` by default, updates related tests/help/types/docs, adds exec outcome normalization for volatile fields, and escalates consecutive generic repeats to critical blocking.

## Best Possible Solution

Land a narrow, rebased fix that enables loop detection by default, preserves explicit opt-out, keeps current main's run-scoped history and stable exec outcome hashing, updates every public/default metadata surface consistently, and only hard-blocks generic repeats when there is no-progress evidence.

## Reproduction Assessment

Yes. Current main has a high-confidence source/test reproduction: repeated same tool calls with omitted config still resolve to `enabled: false`, and the existing unit test asserts that no loop is detected by default.

## Solution Assessment

No as-is. Enabling the guard by default is the right direction, but this implementation should be rebased and narrowed so generic critical blocking is gated by no-progress evidence and the docs/config surfaces agree.
