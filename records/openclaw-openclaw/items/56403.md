## Summary

Keep this PR open. Current main still ships loop detection disabled by default, so the central fix is not obsolete, but the PR head is not ready to merge because it hard-blocks generic repeats without no-progress evidence and leaves the public loop-detection docs internally contradictory. The related overlapping PR #52759 is also open, so maintainers should choose one canonical rebased fix rather than merge stale overlapping branches.

## What This Changes

This PR enables `tools.loopDetection.enabled` by default, updates tests/config help/types/docs metadata, normalizes some exec outcomes for loop hashing, and escalates consecutive `generic_repeat` detections to critical blocking.

## Best Possible Solution

Land one narrow, rebased canonical fix that enables loop detection by default, preserves explicit opt-out, keeps current main's stable exec outcome hashing and run-scoped history, syncs every user-facing config/docs surface, and gates critical blocking on no-progress evidence rather than repeated arguments alone.

## Reproduction Assessment

Yes. Current main has a source and unit-test reproduction: repeated identical calls with omitted config still resolve through `enabled: false`, and the existing default test asserts no loop is detected.

## Solution Assessment

No as-is. Enabling the guard by default is the right direction, but this PR should be narrowed so generic repeats do not hard-block while results are changing and the documentation is fully synchronized.
