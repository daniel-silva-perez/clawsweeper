## Summary

Keep this PR open, but do not merge it as-is. Current main still leaves tool-loop detection disabled by default, so the central request remains relevant; the PR also adds progress-blind critical blocking for generic repeats and leaves the loop-detection docs internally contradictory.

## What This Changes

This PR enables `tools.loopDetection.enabled` by default, updates related tests/help/types/docs metadata, normalizes some exec outcomes for loop hashing, and escalates consecutive generic repeats to critical blocking.

## Best Possible Solution

Land one narrow, rebased fix that enables loop detection by default, preserves explicit opt-out and current main's run-scoped/stable-outcome behavior, updates every public config/doc surface consistently, and only hard-blocks generic repeats when there is no-progress evidence.

## Reproduction Assessment

Yes. Current main has a source and unit-test reproduction: repeated identical calls with omitted config still resolve through `enabled: false`, and the existing test asserts no loop is detected by default.

## Solution Assessment

No as-is. Enabling the guard by default is the right direction, but this PR should be rebased and narrowed so generic critical blocking is progress-aware and the documentation is fully synchronized.
