## Summary

Keep open. Current main still ships tool-loop detection disabled by default, and the overlapping related PRs are still open or closed unmerged, so the central behavior is not implemented. The branch is useful but should not merge as-is because it can hard-block progressing repeated calls and leaves the loop-detection docs internally inconsistent.

## What This Changes

The PR flips `tools.loopDetection.enabled` to true by default, updates config help/types/docs/tests, adds exec outcome hashing for volatile fields, and changes `generic_repeat` to critical-block consecutive identical tool calls.

## Best Possible Solution

Rebase or replace this with a narrow patch that flips the runtime default to true, preserves explicit opt-out, keeps current main's stable exec hashing and run-scoped history, aligns every public/default metadata surface, and only hard-blocks generic repeats when there is no-progress evidence.

## Reproduction Assessment

Yes. Current main reproduces the reported gap with a focused source/test check: repeated tool calls without an explicit config still use `enabled: false`, and the existing test asserts that default-disabled behavior.

## Solution Assessment

No as-is. Enabling the guard by default is the right direction, but the PR should avoid critical blocking on args-only progress-blind streaks and should update the full docs page, not only one field bullet.
