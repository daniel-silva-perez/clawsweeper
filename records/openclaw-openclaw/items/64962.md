## Summary

Keep this issue open. Current main still has the reported timeout-compaction non-reduction path: after a high-context timeout, the runner attempts compaction, retries only when `compacted` is true, and logs the reported fall-through when compaction returns `compacted: false`; the downstream result is still a normal timeout error payload rather than an escalation ladder.

## What This Changes

The issue requests timeout-triggered compaction recovery to escalate when compaction runs but does not reduce context, instead of falling through to a user-visible context/timeout failure.

## Best Possible Solution

Add an explicit timeout-compaction recovery ladder in `src/agents/pi-embedded-runner/run.ts`: after a timeout recovery compaction returns `compacted: false` or throws, try any approved fallback compaction model path, then reuse the existing compact-then-truncate/tool-result truncation recovery where applicable, and finally return a clear blocked-session or fresh-session recovery outcome. Update the timeout-triggered compaction tests so they assert escalation instead of the current timeout fall-through.
