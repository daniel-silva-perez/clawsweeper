## Summary

Keep this issue open. Current main still treats a timeout-triggered compaction that returns `compacted: false` as a fall-through path: it logs the reported non-reduction warning and current regression tests still assert a terminal timeout error payload rather than a timeout-specific recovery ladder into compaction fallback, compact-then-truncate, or fresh-session recovery.

## What This Changes

The issue requests escalation after timeout-triggered compaction fails to reduce context, including fallback compaction models, aggressive truncation, and fresh-session recovery instead of surfacing a timeout/context overflow failure.

## Best Possible Solution

Define one maintainer-approved timeout-compaction recovery ladder in the embedded runner: treat `compacted: false` and compaction exceptions as recoverable signals, reuse the approved compaction fallback model design, apply existing tool-result truncation when it can actually reduce the prompt, and only then surface a clear blocked or fresh-session recovery outcome with focused regression tests.
