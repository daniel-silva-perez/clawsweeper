## Summary

Keep open: current main still has the reported timeout-compaction non-reduction path. The runner attempts timeout recovery compaction on high prompt usage, retries only when `compacted` is true, and otherwise falls through to the normal timeout error payload; the focused tests still encode that behavior.

## What This Changes

The issue asks for timeout-triggered compaction to escalate when compaction completes but does not reduce context enough.

## Best Possible Solution

Add an explicit timeout-compaction recovery ladder in `src/agents/pi-embedded-runner/run.ts`: after timeout recovery compaction returns `compacted: false` or throws, try an approved compaction fallback path if available, then reuse existing compact-then-truncate/tool-result truncation recovery where it can help, and only then return a clear blocked or fresh-session recovery outcome. Update the timeout-triggered compaction tests so the non-reducing path asserts escalation instead of the current timeout fall-through.

## Reproduction Assessment

Yes. The focused test `falls through to normal handling when timeout compaction fails` reproduces the core path by mocking a high-context timeout and a `compacted: false` compaction result, and current source shows the same fall-through branch.

## Solution Assessment

No for current main: the existing behavior is not the best solution because it treats a non-reducing compaction as terminal timeout handling. The safest repair direction is a small explicit recovery ladder, but session rotation and compaction model fallback policy need maintainer design alignment with #64960.
