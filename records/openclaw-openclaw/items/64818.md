## Summary

Keep open. Current main still has the reported failure shape: the generated A2UI hash is tracked while ignored, the build preflight can rewrite it, and the dev update fallback loop reuses the same preflight worktree without cleaning or resetting before trying the next checkout.

## What This Changes

The issue asks OpenClaw to prevent `openclaw update` preflight walkback from being blocked by a dirty tracked `src/canvas-host/a2ui/.bundle.hash` generated during a failed candidate build.

## Best Possible Solution

Harden the git-install updater so each disposable preflight candidate starts from a clean worktree before checkout, then add a regression test that simulates a failed candidate leaving a tracked generated file dirty before the next fallback. The A2UI hash tracking/reproducibility cleanup can be a companion change, but the updater should be resilient to this whole class of generated-file dirtiness.

## Reproduction Assessment

Yes. The code path is high-confidence: current main tracks an ignored generated hash, the preflight build can rewrite it, and the fallback loop then attempts plain detached checkouts in the same worktree without cleanup.

## Solution Assessment

Yes, with a narrower emphasis on updater robustness. Re-untracking the hash file would help this specific artifact, but cleaning or resetting the disposable preflight worktree before each candidate is the more maintainable fix for similar future generated-file conflicts.

Fix the git-install `openclaw update` preflight walkback bug from https://github.com/openclaw/openclaw/issues/64818. Current main can dirty tracked generated files such as `src/canvas-host/a2ui/.bundle.hash` during a failed preflight build, then the next fallback candidate uses plain `git checkout --detach` in the same disposable worktree and can fail with local-change conflicts. Scope the fix to `src/infra/update-runner.ts` and tests: ensure each fallback candidate starts from a clean disposable preflight worktree before checkout, using an approach appropriate for a temporary worktree. Do not rely only on untracking the A2UI hash, do not mutate A2UI generated artifacts unless a maintainer explicitly chooses that companion cleanup, and preserve existing Windows cleanup behavior.
