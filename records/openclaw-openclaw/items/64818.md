## Summary

Keep open. Current main still matches the reported failure shape: the generated A2UI hash is tracked while ignored, the build path can rewrite it during preflight, and the dev update preflight loop still reuses one disposable worktree across fallback candidates without resetting or cleaning before the next checkout. The minimal fix PR was closed unmerged, and the related A2UI reproducibility issue does not replace the updater checkout-conflict fix.

## What This Changes

The issue requests that `openclaw update` preflight walkback stop failing when a candidate build dirties the tracked generated A2UI bundle hash before fallback checkout.

## Best Possible Solution

Harden the git-install updater preflight loop so each fallback candidate starts from a clean disposable worktree before checkout, and cover it with a regression test that simulates a failed candidate leaving a tracked generated file dirty. The A2UI hash tracking/reproducibility cleanup can be handled as a companion or follow-up, but the updater should be robust against this class of generated-file dirtiness.
