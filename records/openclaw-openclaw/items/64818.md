## Summary

Keep open. Current main still has the reported failure shape: the A2UI hash is both tracked and ignored, the build path can rewrite it, and the dev update preflight loop still retries candidate checkouts in the same disposable worktree without resetting or cleaning after failed deps/build/lint steps.

## What This Changes

The issue requests that `openclaw update` stop failing its dev preflight walkback when generated A2UI hash changes leave the disposable worktree dirty before fallback checkouts.

## Best Possible Solution

Make the dev update preflight loop robust against candidate-local dirtiness by cleaning or resetting the disposable worktree before each fallback checkout, with a regression test that simulates a failed candidate leaving a tracked generated file modified. The A2UI hash tracking/reproducibility cleanup can remain a companion or follow-up, but the updater should tolerate this class of generated-file dirtiness.

## Repair Work Prompt

Fix the dev git update preflight walkback bug reported in this issue. Current main reuses one disposable preflight worktree while iterating candidate SHAs; if deps/build/lint for one candidate dirties a tracked generated file such as `src/canvas-host/a2ui/.bundle.hash`, the next `git checkout --detach <sha>` can fail and the updater returns `preflight-no-good-commit` even though an older good commit may exist.

Expected fix boundary: harden `src/infra/update-runner.ts` so each fallback candidate checkout starts from a clean disposable preflight worktree state. Prefer a targeted reset/clean before retrying the next candidate, or an equivalently safe worktree strategy. Do not make the primary fix depend on changing checked-in A2UI artifacts. Avoid mutating generated A2UI bundle/hash files unless deliberately choosing and explaining a companion cleanup.

Add a regression test in `src/infra/update-runner.test.ts` that simulates at least two preflight candidates: the first candidate checks out and then fails after deps/build/lint while leaving a tracked generated file dirty; the second candidate should still be checked out and selected after the cleanup/reset behavior. Assert the relevant git cleanup command ordering and that the result is not `preflight-no-good-commit`.

Related refs: this issue, #64822 closed unmerged minimal hash untracking PR, #67547 adjacent A2UI reproducibility issue, #58544 symptom-level preflight-no-good-commit report.

Likely files: `src/infra/update-runner.ts`, `src/infra/update-runner.test.ts`, and `CHANGELOG.md` if the fix is user-facing. If adding a changelog entry, put it under the active version `### Fixes` section as a single line and credit `Thanks @protomota`.

Validation: run `pnpm test src/infra/update-runner.test.ts`. Before handoff, run `pnpm check:changed` in Testbox if code/test/runtime files changed. Do not run raw `vitest`, do not edit baseline/ignore files, and do not use `git reset --hard` on the user's checkout outside the disposable preflight worktree code path.
