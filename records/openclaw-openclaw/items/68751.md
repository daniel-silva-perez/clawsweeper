## Summary

Keep open. Current main still has the reported session-memory behavior: the bundled hook fires on both `/new` and `/reset`, extracts prior user/assistant transcript text, and writes it under `## Conversation Summary` without untrusted-data framing or neutral summarization. Current startup-context wrapping helps one first-turn path, but the persisted memory artifact remains transcript-shaped and is still indexed/read by memory search/get surfaces. Related sanitizer PRs such as #69961 and #73768 target malformed token poisoning for #69943; they do not fully address raw prior-user command replay or `/reset` should-forget semantics.

## Best Possible Solution

Keep this issue open and fix the bundled `session-memory` hook directly. The conservative path is to stop persisting prior transcript material on `/reset`, and for `/new` either wrap saved conversation content in explicit untrusted historical markers before it reaches `memory/*.md` or replace raw turns with a neutral generated summary. Add regression coverage for the saved file shape, `/reset` behavior, and coordination with the malformed-token sanitizer work around #69943/#69961/#73768.
