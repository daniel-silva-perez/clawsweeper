## Summary

Keep open. Current main still persists raw prior-session turns on both `/new` and `/reset` without untrusted framing or neutral summary conversion, and the existing startup-context wrapper only covers one injection path while memory files remain searchable/readable later.

## What This Changes

The issue asks the bundled `session-memory` hook to stop saving prior-session transcript-shaped turns in a way that can be replayed as actionable current input after `/reset`.

## Best Possible Solution

Fix the bundled `session-memory` boundary: make `/reset` stop carrying prior transcript material forward, and ensure any `/new` continuity is stored or returned as clearly untrusted historical reference or a neutral summary before it reaches memory files, search results, or `memory_get`. Preserve the newer startup-context untrusted wrapper and coordinate with the malformed-token sanitizer work rather than treating token stripping alone as complete.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: create a session JSONL with user/assistant messages, trigger the hook with `action: "reset"`, and current tests show the raw `user:` and `assistant:` lines are written into `memory/YYYY-MM-DD-*.md`. Live model re-execution is model-dependent, but the unsafe persistence and recall surface are directly reproducible from current main.

## Solution Assessment

Yes, the requested direction is the right fix boundary, but it should combine reset semantics with persisted-content framing. Token-only or metadata-only sanitizers are useful subcase fixes, but they do not neutralize ordinary prior user commands saved as historical `user:` turns.
