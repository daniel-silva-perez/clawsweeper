## Summary

Keep open. Current main still writes prior-session user/assistant turns into session-memory Markdown on both `/new` and `/reset` without a persisted untrusted boundary or neutral-summary conversion, and the startup-context wrapper only mitigates one injection path while `memory_search`/`memory_get` can still reach the durable files. The active related PRs target narrower sanitizer subcases and do not yet resolve this broader reset/framing policy.

## What This Changes

The issue asks for bundled `session-memory` to stop replaying prior-session transcript-shaped turns as current instructions by wrapping or summarizing stored content and likely changing `/reset` persistence semantics.

## Best Possible Solution

Fix the bundled session-memory trust boundary end to end: `/reset` should break continuity by default, while any `/new` continuity should be stored and later returned only as clearly untrusted historical reference or as a neutral summary. The fix should cover durable Markdown files, startup prelude loading, and memory tool retrieval behavior, and it should coordinate with the malformed-token and metadata sanitizer PRs rather than treating those subcases as complete.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: create a JSONL transcript with ordinary user/assistant messages, trigger the session-memory hook with `action: "reset"`, and current code/tests show raw transcript-shaped lines written into `memory/YYYY-MM-DD-*.md`. Live autonomous re-execution is model- and tool-environment-dependent, but the unsafe persistence and recall surface are directly reproducible from current main.

## Solution Assessment

Yes, the requested direction is the right fix boundary, but the best solution is broader than token stripping. A narrow durable-content trust boundary, plus reset-specific semantics, is safer than only sanitizing malformed chat-template tokens or channel metadata.
