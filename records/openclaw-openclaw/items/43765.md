## Summary

Keep open. The PR head still has blocking static defects, current main only partially supersedes the branch, and an overlapping Feishu health-signal PR remains open, so this is not conservative cleanup material and should not be closed as implemented.

## What This Changes

The PR branch changes heartbeat system-event consumption, Feishu channel health snapshots and status sinks, exec/process terminal status handling, ACPX/Codex ACP launch behavior, ACP spawn guidance, first-visible reply diagnostics/config/docs, lockfile entries, and related tests.

## Best Possible Solution

Replace or split the branch into current-main-sized changes: reconcile Feishu health reporting with #39489, keep heartbeat event consumption on the current prefix-consumption model, decide exec non-zero exit semantics with compatibility-focused tests, review ACPX launch behavior against the current 0.12.0/0.6.1 baseline, and land first-visible diagnostics separately if still desired.

## Reproduction Assessment

Yes for the PR defects. Read-only raw-source inspection of the PR head reproduces the duplicate top-level declarations, missing imported module, and drained heartbeat events followed by an alerts-disabled return path.

## Solution Assessment

No. The current PR is not the best way to solve the runtime recovery issues because it combines multiple subsystems, stale dependency changes, overlapping Feishu work, docs, diagnostics, and blocking static defects in one XL branch; smaller rebased PRs are safer.
