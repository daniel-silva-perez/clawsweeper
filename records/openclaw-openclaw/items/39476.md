## Summary

Current main still permits the reported reverse sessions_send path: A2A prompts expose the requester session key, peer A2A still runs reply/announce turns as normal agent runs, and there is no per-run tool deny policy to suppress sessions_send during those nested turns.

## Best Possible Solution

Keep this issue open and fix the core A2A sessions_send lifecycle. Add a framework-level delivery-intent or per-run tool policy that prevents target/reply/announce A2A turns from calling sessions_send back to the requester, narrow requester session-key prompt exposure where possible, preserve the parent-owned ACP child skip, and add regression coverage for normal peer A2A duplicate visible delivery. Treat #44309 as a related dispatch-only semantics discussion unless it lands the same hard guard.
