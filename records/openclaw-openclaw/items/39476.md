## Summary

Keep this issue open. Current main still exposes the requester session key to A2A target/reply/announce prompts, still runs the normal reply-back ping-pong plus announce path for peer `sessions_send`, and still lacks a per-run hard guard that removes or denies `sessions_send` during nested A2A turns. The existing skip is limited to requester-owned background ACP children, and #44309 is a broader dispatch-only feature request rather than a shipped fix for this reverse-call duplicate path.

## Best Possible Solution

Keep this issue open and fix the core A2A `sessions_send` path. Add a framework-level guard that prevents A2A target/reply/announce turns from calling `sessions_send` back to the requester, remove or narrow requester-session prompt exposure where feasible, preserve the existing parent-owned ACP-child skip, and add regression coverage for normal peer A2A duplicate visible delivery. Treat #44309 as a separate dispatch-only semantics discussion, not the replacement for this bug fix.
