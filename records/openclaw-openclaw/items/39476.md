## Summary

Current main still permits the reported reverse `sessions_send` path: the requester session key is still rendered into A2A prompts, target/reply/announce turns still run as normal agent turns with normal tool policy, and existing tests/docs describe ping-pong plus announce as active behavior. Keep this open for a core A2A guard rather than closing as implemented or superseded by the broader dispatch-only request.

## Best Possible Solution

Keep this issue open and fix the core A2A `sessions_send` lifecycle. Add a framework-level per-run guard or delivery-intent policy that prevents target/reply/announce A2A turns from calling `sessions_send` back to the requester, remove or narrow requester session-key prompt exposure where feasible, preserve the existing requester-owned ACP child skip, and add regression coverage for normal peer A2A duplicate visible delivery. Treat #44309 as a separate dispatch-only semantics discussion unless it lands the same hard guard.
