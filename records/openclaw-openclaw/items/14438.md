## Summary

Current main still lacks the requested plugin developer hot-reload or cache-clear workflow. Docs explicitly say native plugin runtime code requires a Gateway restart, `openclaw plugins` has no reload or clear-cache command, and the Gateway reload planner still treats `plugins.*` changes as restart work. Internal loader/Jiti cache helpers exist, but they are not exposed as a supported CLI/RPC workflow with live dispose and re-registration semantics. The related #74122 is adjacent watcher debounce/scoped reload work, not a replacement for this issue.

## Best Possible Solution

Keep this open as core plugin loader and Gateway lifecycle developer-UX work. The best path is maintainer-approved design for `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, backed by a Gateway method that invalidates loader/Jiti/module state, safely disposes plugin-owned runtime registrations/resources, re-registers tools/hooks/providers/services, and documents Docker/cache-location behavior. If native plugin runtime hot reload remains intentionally unsupported, the CLI and plugin docs should document the supported restart/cache workflow clearly.
