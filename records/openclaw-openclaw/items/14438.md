## Summary

Keep open. Current main still documents native plugin runtime changes as restart-only, the `openclaw plugins` CLI has no reload or clear-cache command, and the Gateway reload path is config-reload/restart machinery rather than supported Jiti/module invalidation plus live plugin re-registration.

## Best Possible Solution

Keep this issue open as core plugin loader and Gateway lifecycle developer-UX work. The best path is a maintainer-approved design for `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, backed by a Gateway RPC that invalidates loader/Jiti/module state, safely disposes plugin-owned registrations/resources, re-registers tools/hooks/providers/services, and documents Docker behavior. If native plugin runtime hot reload is intentionally unsupported, the plugin and CLI docs should instead document the supported restart/cache workflow clearly.
