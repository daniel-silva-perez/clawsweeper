## Summary

Keep open. Current main still lacks a supported in-process plugin reload or plugin cache-clear workflow, and the latest maintainer discussion explicitly leaves this for plugin-loader and Gateway lifecycle design rather than cleanup closure.

## What This Changes

The issue requests a faster plugin development workflow that invalidates Jiti/plugin loader caches and reloads TypeScript plugin runtime changes without restarting the container or Gateway process.

## Best Possible Solution

Define a supported plugin development reload path in the plugin loader and Gateway lifecycle model: invalidate Jiti/module/runtime caches, safely replace the active registry, run plugin lifecycle cleanup for stale registrations, expose an additive Gateway/CLI UX such as openclaw plugins reload or clear-cache, and document the security and container behavior. If live runtime reload remains unsupported, keep restart-only behavior documented with accurate cache guidance.
