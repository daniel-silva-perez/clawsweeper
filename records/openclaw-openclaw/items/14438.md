## Summary

This should stay open: current main still documents native plugin runtime changes as restart-only, the plugin CLI has no reload or cache-clear command, and Gateway config reload planning still maps plugin config changes to a Gateway restart. The latest maintainer comment also explicitly keeps the item open for plugin-loader and Gateway lifecycle design, so conservative cleanup should not close it or queue a narrow automated fix.

## What This Changes

The issue requests a supported plugin development reload or cache-clear workflow so TypeScript plugin edits do not require manually clearing Jiti cache files and restarting the OpenClaw process or container.

## Best Possible Solution

Design a supported plugin development reload flow backed by Gateway/API lifecycle semantics that can invalidate loader/Jiti/module state, safely replace plugin registrations, and expose either `openclaw plugins reload <id>` or `openclaw plugins clear-cache`; if live runtime reload is intentionally out of scope, document the restart-only workflow with precise container/cache guidance.
