## Summary

Keep this issue open. Current main still documents native plugin runtime changes as restart-only, exposes no supported `openclaw plugins reload` or cache-clear workflow, and the loader/Gateway pieces are internal cache/state reset and config-reload machinery rather than live plugin module/Jiti invalidation plus safe re-registration.

## Best Possible Solution

Keep this issue open as core plugin loader and Gateway lifecycle developer-UX work. The best path is a maintainer-approved design for `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, backed by a Gateway RPC that invalidates loader/Jiti/module state, safely disposes plugin-owned registrations/resources, re-registers tools/hooks/providers/services, and documents Docker behavior. If maintainers decide native plugin runtime hot reload is intentionally unsupported, the public plugin and CLI docs should instead document the supported restart/cache workflow clearly.

## Risks / Open Questions

- A cache-clear-only command could give users false confidence because current code has OpenClaw registry caches, Jiti loader/module caches, and live Gateway registrations to reconcile.
- Live re-execution of third-party plugin code is lifecycle- and security-sensitive: a fix needs disposal/re-registration semantics for tools, hooks, providers, services, commands, memory/runtime state, and channel/plugin-owned resources.
- Docker/container behavior needs explicit design because the docs already distinguish restarting the actual `openclaw gateway run` child from wrapper process restarts.
