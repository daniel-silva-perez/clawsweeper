## Summary

Keep open. Current main still lacks a supported plugin hot-reload or cache-clear workflow, the public docs explicitly describe native plugin runtime changes as restart-only, the plugin CLI has no reload/cache-clear command, plugin config changes still plan a Gateway restart, and a maintainer comment already keeps this as plugin-loader/Gateway lifecycle design work.

## What This Changes

The issue requests a faster plugin development loop by invalidating plugin/Jiti caches and reloading plugin tools without restarting the OpenClaw process or container.

## Best Possible Solution

Define a supported plugin development reload lifecycle in the loader and Gateway: invalidate Jiti/module/runtime caches, safely replace active plugin registrations, dispose stale resources, expose an additive CLI/RPC UX such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`, and update the plugin docs. If live runtime reload remains unsupported, keep restart-only behavior documented with accurate container/cache guidance.

## Reproduction Assessment

Not applicable as a normal bug reproduction, but the feature gap is high-confidence. Current docs, CLI registration, reload planner rules, and config reload tests all show restart-only behavior for native plugin runtime/config changes.

## Solution Assessment

Unclear until maintainers choose the lifecycle design. A path-specific cache delete command alone would be too narrow; the maintainable solution needs loader cache invalidation plus safe active registry/lifecycle replacement and Gateway/CLI UX.
