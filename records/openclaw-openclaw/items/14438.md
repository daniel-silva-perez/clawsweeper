## Summary

Keep open. Current main still documents native plugin runtime changes as restart-only, has no plugin reload or cache-clear CLI path, and continues to classify plugin config changes as Gateway restart work; the recent maintainer discussion also explicitly keeps this as plugin-loader and Gateway lifecycle design work.

## What This Changes

The issue requests a supported plugin development reload workflow, such as a plugin reload command, watcher, or cache-clear command, to invalidate Jiti/module state and apply TypeScript plugin edits without restarting the container.

## Best Possible Solution

Define a supported plugin development reload path in the plugin loader and Gateway lifecycle model: invalidate Jiti/module/runtime caches, safely replace active plugin registrations, dispose stale lifecycle resources, expose an additive CLI/RPC UX such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`, and document the container/cache/security behavior. If live runtime reload remains unsupported, keep restart-only behavior documented with accurate cache guidance.
