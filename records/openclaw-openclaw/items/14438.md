## Summary

Current main still does not implement the requested plugin developer hot-reload/cache-clear workflow. Native plugin runtime changes remain documented as restart-only, `openclaw plugins` has no reload or clear-cache command, and the Gateway reload planner treats `plugins.*` changes as restart work rather than supported Jiti/module invalidation plus live plugin re-registration. The open cross-reference #74122 is adjacent watcher/reload ergonomics work, not a replacement for this request.

## Best Possible Solution

Keep this open as core plugin loader and Gateway lifecycle developer-UX work. The best path is a maintainer-approved design for `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, backed by a Gateway method that invalidates loader/Jiti/module state, safely disposes plugin-owned registrations/resources, re-registers tools/hooks/providers/services, and documents Docker/cache-location behavior. If native plugin runtime hot reload remains intentionally unsupported, the CLI and plugin docs should document the supported restart/cache workflow clearly.
