## Summary

Current main still lacks the requested plugin hot-reload or cache-clear workflow. The public docs explicitly say native plugin runtime changes require a Gateway restart, `openclaw plugins` exposes no reload or clear-cache subcommand, and Gateway config reload planning still treats `plugins.*` changes as restart-required. The related #74122 is adjacent watcher debounce/scoped reload work, not a replacement for Jiti/module cache invalidation and live plugin re-registration.

## What This Changes

The issue requests a supported plugin development workflow for reloading TypeScript plugins or clearing Jiti/native loader cache without restarting the OpenClaw process/container.

## Best Possible Solution

Keep this open for maintainer-approved core plugin loader and Gateway lifecycle design. The best path is a supported `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache` workflow backed by a Gateway method that invalidates loader/Jiti/module state, safely disposes plugin-owned runtime registrations/resources, re-registers tools/hooks/providers/services, and documents Docker/cache-location behavior. If native runtime hot reload remains intentionally unsupported, the CLI and plugin docs should document the supported restart/cache workflow clearly.
