## Summary

Keep open. Current main still documents native plugin runtime changes as restart-only, the plugin CLI has no reload or cache-clear command, plugin config changes still route to Gateway restart behavior, and the latest maintainer discussion explicitly keeps this as plugin-loader and Gateway lifecycle design work.

## What This Changes

Add a supported plugin development hot-reload or cache-invalidation workflow so TypeScript plugin changes can take effect without restarting the OpenClaw process or container.

## Best Possible Solution

Define a supported plugin development reload lifecycle in the loader and Gateway: invalidate Jiti/module/runtime caches, safely replace active plugin registrations, dispose stale resources, expose an additive CLI/RPC UX such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`, and document container/cache/security behavior. If live runtime reload remains unsupported, keep restart-only behavior documented with accurate cache guidance.

## Reproduction Assessment

Not applicable as a normal bug reproduction, but the feature gap is high-confidence: current docs, CLI registration, reload planner, and tests all show restart-only behavior for native plugin runtime/config changes.

## Solution Assessment

Unclear until maintainers choose the lifecycle design. An ad hoc /tmp/jiti delete command alone would be too narrow; the maintainable solution needs loader cache invalidation plus safe active registry/lifecycle replacement and Gateway/CLI UX.
