## Summary

Keep open. Current main still parses large iOS share/node `agent.request` image attachments through `server-node-events.ts`, but it only forwards `message`, `images`, and `imageOrder` to `agentCommandFromIngress`; it does not route `parsed.offloadedRefs` into transcript `MediaPath`/`MediaPaths`. The `chat.send` path has the media persistence implementation and regression coverage, but that does not cover this node-event ingress path.

## Best Possible Solution

Keep this issue open and implement the same transcript media persistence contract for `node.event` / `agent.request` that `chat.send` now has: carry ordered offloaded refs through the node ingress path, persist or rewrite the user transcript entry with `MediaPath`/`MediaPaths` and matching media types, and add a focused `server-node-events` regression test for a large offloaded iOS/node image.

## Risks / Open Questions

- The current turn may still work for some vision paths because offloaded image markers can be hydrated from prompt text, but history/UI/replay consumers that depend on transcript `MediaPath`/`MediaPaths` still lose the structured media metadata on the node-event path.
- A fix should avoid raw JSONL appends; the scoped Gateway guidance says Pi session transcript messages should go through `SessionManager.appendMessage(...)` or a wrapper that preserves the parent chain.
