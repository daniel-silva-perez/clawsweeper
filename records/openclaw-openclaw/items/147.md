## Summary

Keep open. Current main has generic node, Voice Wake, VoiceClaw realtime, Talk/TTS, and OpenAI-compatible Gateway surfaces that make an external Raspberry Pi voice bridge feasible, and the discussion already includes a working external reference. It still does not ship or document a first-class Brabble/Raspberry Pi distributed voice-wake node, and the issue discussion includes maintainer-facing product/security questions that conservative cleanup should not preempt.

## What This Changes

Request a first-class Brabble/Raspberry Pi distributed voice-wake node that connects to the Gateway, forwards wake-word audio or text to an agent, and returns responses via TTS.

## Best Possible Solution

Maintainers should choose the official boundary first: document a minimal external voice-node contract and promote or link the Raspberry Pi companion through docs/ClawHub, or define a small generic node/plugin API only if current Gateway surfaces are insufficient. Brabble-specific core code should wait for that product and security decision.

## Reproduction Assessment

Not applicable as a feature/product request rather than a bug. The current-main verification path is high confidence: search/read the voice, node, Gateway, plugin, and docs surfaces to confirm Brabble/Pi distributed voice wake is still not shipped or documented as a first-class feature.

## Solution Assessment

No, conservative closure is not the best way to solve this yet. The narrowest maintainable path is to decide and document the supported external-node contract and trust boundary first; native or plugin work should follow only if that review finds a missing OpenClaw API.
