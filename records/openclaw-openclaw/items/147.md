## Summary

Keep open. Current main has generic Gateway, node, Voice Wake, Talk/TTS, plugin, and OpenAI-compatible HTTP primitives that make an external Raspberry Pi voice bridge feasible, and the discussion includes a working external reference, but OpenClaw still does not ship or document a first-class Brabble/Raspberry Pi distributed voice-wake node. The remaining decision is product and security boundary work, not conservative cleanup.

## What This Changes

The issue requests a Brabble/Raspberry Pi-style distributed voice wake node that connects to the OpenClaw Gateway, sends wake-triggered speech to an agent, and returns spoken responses.

## Best Possible Solution

Maintainers should choose the official boundary first: document a minimal external voice-node contract and promote or link the Raspberry Pi companion through docs/ClawHub, or define a small generic node/plugin API only if current Gateway surfaces are insufficient. Any official path should include clear security guidance for operator credentials and always-on microphone use.

## Reproduction Assessment

Not applicable as a feature/product request rather than a bug. The verification path is high-confidence: current-main source and docs show generic voice/node/Gateway primitives but no first-class Brabble or Raspberry Pi distributed voice-wake node.

## Solution Assessment

No, implementing native Brabble support or closing as external-only is not clearly the best resolution yet. The narrower maintainable path is a maintainer decision on the supported external-node contract and trust boundary before code, docs, or plugin packaging work starts.
