## Summary

Keep open. Current main has generic Gateway/node voice primitives, Voice Wake routing, VoiceClaw realtime support, and an OpenAI-compatible HTTP path that make an external Raspberry Pi voice bridge feasible, and the discussion includes a working external reference. It still does not ship or document a first-class Brabble/Raspberry Pi distributed voice-wake node, so the remaining work is a product and security boundary decision rather than conservative cleanup.

## What This Changes

Request to support a Brabble/Raspberry Pi distributed voice-wake node that connects to the Gateway, sends wake-word audio or text into an agent session, and returns responses through TTS.

## Best Possible Solution

Maintainers should choose the official boundary first: document a minimal external voice-node contract and promote or link the Raspberry Pi companion through docs/ClawHub, or define a small generic node/plugin API only if current Gateway surfaces are insufficient. Brabble-specific core code should wait for that product and security decision.

## Reproduction Assessment

Not applicable as a feature/product request rather than a bug. The current-main verification path is high confidence: search/read the voice, node, Gateway, plugin, and ClawHub docs plus the Gateway voice transcript implementation to confirm the requested official Brabble/Pi node is still absent.

## Solution Assessment

No, a conservative close is not the best way to solve this yet. The narrowest maintainable path is first to decide and document the supported external-node contract and trust boundary; native or plugin work should follow only if that review finds a missing OpenClaw API.
