## Summary

Keep open. Current main still rejects mixed-case valid `--resources` values because `parse_resources()` validates stripped tokens against lowercase `ALLOWED_RESOURCES` before normalization, and the related fix PRs are closed and unmerged.

## What This Changes

The issue requests case-insensitive parsing for `skills/skill-creator/scripts/init_skill.py --resources` while preserving validation, order-preserving deduplication, and regression coverage.

## Best Possible Solution

Normalize each non-empty resource token by stripping and lowercasing before validation and first-seen deduplication, return canonical lowercase resource names, and add focused `parse_resources()` regression tests for mixed-case valid input, case-insensitive duplicates, empty-token handling, and invalid-resource rejection.

Yes. Importing `parse_resources()` with bytecode disabled reproduces current-main behavior: lowercase resource values succeed, while `Scripts,References,ASSETS` exits with unknown-resource errors.

## Solution Assessment

Yes. A narrow parser normalization plus focused Python unittest coverage is the smallest maintainable fix; no core architecture, plugin API, or product decision is needed.
