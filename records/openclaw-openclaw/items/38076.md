## Summary

Keep open. Current main still validates stripped `--resources` tokens with their original casing, so valid mixed-case values fail; the latest release tag has the same parser, and the related implementation PRs in the supplied context are closed and unmerged.

## What This Changes

Request `skills/skill-creator/scripts/init_skill.py` to parse `--resources` values case-insensitively while preserving validation, order-preserving deduplication, and adding regression tests.

## Best Possible Solution

Normalize each non-empty resource token by stripping and lowercasing before validation and first-seen deduplication, return canonical lowercase resource names, add focused `parse_resources()` tests for mixed-case valid input, case-insensitive duplicates, empty-token handling, and invalid-resource rejection, and add a changelog fix entry if maintainers treat bundled skill tooling fixes as user-facing.

## Reproduction Assessment

Yes. Importing `parse_resources()` with `PYTHONDONTWRITEBYTECODE=1 python3 -B` reproduces current-main behavior: lowercase values succeed, while mixed-case valid values and case-only duplicates exit with unknown-resource errors.

## Solution Assessment

Yes. A narrow parser normalization plus focused Python unittest coverage is the smallest maintainable fix; no core architecture, plugin API, or product decision is needed.
