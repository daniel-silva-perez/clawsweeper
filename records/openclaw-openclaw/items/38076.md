## Summary

Keep open. Current main still rejects mixed-case valid `--resources` values because `parse_resources()` validates stripped tokens against lowercase allowed names before any normalization, and the related fix PRs are closed and unmerged.

## What This Changes

The request is to make `skills/skill-creator/scripts/init_skill.py --resources` accept valid resource names case-insensitively while preserving validation and order-preserving deduplication.

## Best Possible Solution

Normalize each non-empty resource token by stripping and lowercasing before validation and first-seen deduplication, return canonical lowercase resource names, and add focused parser regression tests for mixed-case valid input, case-insensitive duplicates, empty-token handling, and invalid-resource rejection.

## Reproduction Assessment

Yes. Importing `parse_resources()` with bytecode disabled reproduces the current-main behavior: lowercase values succeed, while `Scripts,References,ASSETS` exits with unknown-resource errors.

## Solution Assessment

Yes. A narrow parser normalization plus focused Python unittest coverage is the smallest maintainable fix; no core architecture, plugin API, or product decision is needed.
