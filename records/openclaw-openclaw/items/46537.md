## Summary

Keep this PR open: the underlying ACP identity bug still exists on current main, but the proposed state flip is unsafe as-is because it lets ensure-derived IDs become authoritative over already-resolved status/event IDs.

## What This Changes

The PR marks ACP ensure-derived identities as resolved when stable ACPX or agent session IDs exist, adds focused unit coverage for ensure identity construction, and updates an ACP spawn fixture expectation.

## Best Possible Solution

Keep the intended resolved-on-ensure behavior, but make merge authority source-aware: ensure results with stable IDs may resolve new or still-unresolved identities, while already-resolved status/event IDs should not be replaced by later ensure-derived stale IDs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with this PR, a resolved ensure identity is accepted by `mergeSessionIdentity()`, persisted by `ensureRuntimeHandle()`, and can remain unrepaired when the `failOnStatusError: false` status refresh path returns early.

## Solution Assessment

No. The current PR fixes the misleading pending state, but it conflates identity resolvedness with identity authority; the safer fix is an explicit precedence rule for ensure versus status/event identities.
