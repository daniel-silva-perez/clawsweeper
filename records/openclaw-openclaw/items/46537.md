## Summary

This PR targets a real ACP identity bug that is still present on current main, but the proposed fix is too broad: making every ensure-derived identity with stable IDs `resolved` lets later ensure metadata overwrite already-resolved status/event IDs. The PR needs a narrower source-precedence fix before it can merge.

## What This Changes

The PR marks ensure-derived ACP identities as resolved when `ensureSession()` returns backend or agent session IDs, adds focused `createIdentityFromEnsure()` tests, and updates an ACP spawn fixture expectation.

## Best Possible Solution

Keep the ACP identity fix, but add source-aware precedence: allow ensure results with stable IDs to resolve new or still-unresolved identities, while preserving already-resolved status/event identifiers from later stale ensure results. Add focused coverage for stable ensure, record-only ensure, stale ensure versus fresh status/event, status-refresh failure, and startup skip behavior.

## Reproduction Assessment

Yes. The regression path is reproducible by source inspection: a resolved ensure identity is accepted by `mergeSessionIdentity()`, `ensureRuntimeHandle()` can persist it, and the `failOnStatusError: false` status-refresh path can return before status repairs stale IDs. Existing manager tests already model stale ensure IDs versus fresh status IDs.

## Solution Assessment

No. Marking all ensure handles with stable IDs as resolved fixes the reported pending state but also changes merge authority; the safer solution is an explicit source/precedence rule around ensure-derived identities.
