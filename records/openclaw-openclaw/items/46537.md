## Summary

This PR should stay open because current main still stores ensure-derived ACP identities as pending, but the proposed one-line state flip is not safe as-is. It makes ensure results authoritative in the existing merge path, so stale IDs from a later ensure can replace fresher resolved status/event IDs when status refresh is skipped or fails.

## What This Changes

The PR marks `createIdentityFromEnsure()` identities as resolved when stable ACPX or agent session IDs exist, adds focused unit coverage, and updates one ACP spawn fixture expectation.

## Best Possible Solution

Keep the intended ACP identity cleanup, but make merge authority source-aware: ensure results with stable IDs may resolve new or still-unresolved identities, while already-resolved status/event IDs should not be replaced by later ensure-derived stale IDs. Cover stable ensure, record-only ensure, stale ensure versus fresh status/event, status-refresh failure, and startup skip behavior.

## Reproduction Assessment

Yes. The risky path is reproducible by source inspection: a resolved ensure identity is accepted by `mergeSessionIdentity()`, persisted by `ensureRuntimeHandle()`, and can remain unrepaired when the `failOnStatusError: false` status refresh path returns early.

## Solution Assessment

No. Resolving all ensure handles with stable IDs fixes the reported pending state, but it conflates identity resolvedness with authority; the safer solution is an explicit precedence rule for ensure versus status/event identities.
