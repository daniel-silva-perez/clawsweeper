## Summary

Keep open: current main still has the reported pending ensure-identity behavior, but this PR should not merge as written because it makes ensure-derived IDs authoritative over already resolved status/event IDs.

## What This Changes

The PR changes ACP ensure-derived identity creation to mark handles with stable backend or agent session IDs as resolved, adds focused session-identity tests, and updates a direct ACP spawn fixture expectation.

## Best Possible Solution

Use a source-aware ACP identity merge that lets stable ensure results resolve new or still-unresolved identities while preserving status/event authority over already resolved identifiers; cover stable ensure, record-only ensure, stale ensure versus fresh status/event, startup skip behavior, and status-refresh failure behavior.

## Reproduction Assessment

Yes. The review finding is reproducible from the current code path: ensureRuntimeHandle() merges ensure identity into existing metadata, mergeSessionIdentity() accepts resolved incoming values, and the existing tests already model stale ensure IDs followed by fresher status IDs.

## Solution Assessment

No. The PR fixes the reported state flag too broadly; the safer solution is to resolve ensure identities only where they cannot demote source authority or to adjust merge precedence so ensure-sourced resolved values cannot replace already resolved status/event IDs.
