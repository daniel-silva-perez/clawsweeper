## Summary

Keep open. The PR addresses a real shared media regression, but the proposed fallback is too broad to merge: returning the raw source from `optimizeImageToJpeg()` can bypass the decoded-pixel guard and can make non-JPEG bytes get labeled as JPEG. The open consolidation PR #74243 tracks the safer policy boundary.

## What This Changes

The PR changes `optimizeImageToJpeg()` to return the original source buffer with `resizeSide: 0` and `quality: 0` when every JPEG optimization attempt fails instead of throwing.

## Best Possible Solution

Implement the fallback at the shared loader layer where the original content type and byte cap are still available, only for valid under-cap images after distinguishing optimizer backend failures from HEIC conversion, pixel/dimension guard, unknown-dimension, and over-cap failures. Add focused regression coverage for optimizer backend failure, MIME preservation, and byte-small pixel-oversized images; #74243 appears to be the closest current candidate for that end state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep pixel-limit failures fail-closed:** `src/media/web-media.ts:657-662`
  - body: `resizeToJpeg()` enforces the decoded image pixel limit before backend resize work. This fallback catches every resize failure and returns the original bytes, so a byte-small but pixel-oversized image that current main rejects can pass through as long as it is under `maxBytes`. Limit fallback to optimizer/backend failures after preserving the dimension guard.
  - confidence: 0.9
- **[P2] Preserve the original MIME type on fallback:** `src/media/web-media.ts:657-662`
  - body: For non-HEIC sources, `source` is the original buffer. `optimizeImageWithFallback()` wraps every `optimizeImageToJpeg()` return as `format: "jpeg"`, so falling back with PNG/WebP bytes makes `optimizeAndClampImage()` emit `image/jpeg` for non-JPEG data. Move the raw fallback to a layer that can keep the original content type or only fallback for actual JPEG input.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, or secret-handling changes are present, but the runtime fallback can bypass an existing image resource guard.
