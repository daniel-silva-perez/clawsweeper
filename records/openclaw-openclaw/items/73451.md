## Summary

Close as superseded. The useful fallback idea is covered by the open consolidated media fallback PR #74243, while this narrower patch should not merge because it returns original bytes from the JPEG optimizer boundary, which would bypass current image input guards and can mislabel non-JPEG originals as `image/jpeg`.

## What This Changes

The PR changes `optimizeImageToJpeg` to return the original source buffer with `resizeSide` and `quality` set to `0` when no JPEG optimization attempt succeeds.

## Best Possible Solution

Use #74243 or an equivalent consolidated fix: perform under-cap original fallback at the `loadWebMedia`/`optimizeAndClampImage` boundary, preserve the original content type and filename, rethrow HEIC and image input guard errors, keep optimizer diagnostics, and cover the behavior with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: make `resizeToJpeg` throw, or use the existing oversized-pixel guard path, and the proposed fallback would return the source buffer after all attempts instead of preserving the guard failure; I did not run tests because this review is read-only.

## Solution Assessment

No. Returning original bytes from `optimizeImageToJpeg` is not the safest boundary because it lacks MIME and guard classification context; the safer approach is the caller-level fallback used by #74243.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve image input guard failures:** `src/media/web-media.ts:657-662`
  - body: `resizeToJpeg` is also where current main rejects oversized pixel-count images before decode. Returning `source` after every resize attempt fails will swallow those guard errors, and `optimizeAndClampImage` only checks byte length afterward, so a byte-small image with huge declared dimensions can pass through unoptimized. The fallback needs to rethrow pixel-limit/unknown-dimension guard failures or happen at a caller boundary that can classify them.
  - confidence: 0.88
- **[P2] Keep original MIME type on fallback:** `src/media/web-media.ts:657-662`
  - body: This fallback returns the original buffer from `optimizeImageToJpeg`, but `optimizeImageWithFallback` wraps every result from that helper as `format: "jpeg"`. If the original is PNG or WebP and the backend fails, the caller will send those bytes with `contentType: "image/jpeg"`, which can break downstream media parsing. Preserve the original content type when falling back.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete media input-guard bypass by converting all JPEG resize failures, including pixel-limit guard failures, into original-buffer fallback.

Concerns:

- **[medium] Image pixel guard can be bypassed:** `src/media/web-media.ts:657`
  - body: Current main rejects oversized declared image dimensions inside `resizeToJpeg`; this patch catches those failures along with ordinary optimizer backend failures and returns the original buffer. For byte-small images with huge pixel dimensions, the later cap check only sees bytes and would allow the guarded image to proceed.
  - confidence: 0.88
