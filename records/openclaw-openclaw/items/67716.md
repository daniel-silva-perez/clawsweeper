## Summary

Keep open. Current main and the latest release still have the host-side bootstrap scheduling gap: bootstrap maintenance runs foreground-only, while deferred compaction debt execution is only enabled in the background worker that is scheduled for turn maintenance. The hot-cache threshold behavior still looks partly plugin-owned, but the OpenClaw host path remains actionable and is not superseded by the broader open architecture issue.

## What This Changes

The issue reports that context-engine bootstrap/reconcile and hot-cache maintenance can leave deferred compaction debt pending without a valid debt-consuming follow-up.

## Best Possible Solution

Keep the issue open and make a narrow host-level change so bootstrap/reconcile maintenance for background-capable context engines can schedule or request a debt-consuming follow-up with allowDeferredCompactionExecution enabled. If hot-cache policy intentionally supersedes debt, expose that through an additive context-engine contract or plugin-side telemetry rather than embedding lossless-claw policy in core.

## Reproduction Assessment

Partly yes: the host scheduling gap is reproducible from current source with a fake context engine by calling bootstrapHarnessContextEngine and observing foreground maintenance without allowDeferredCompactionExecution or a queued background follow-up. The exact hot-cache sticky-debt case still lacks a minimized synthetic repro, but the issue and follow-up comments provide concrete live traces.

## Solution Assessment

Yes for the host half: the narrow maintainable fix is to add a generic bootstrap/reconcile debt follow-up or additive engine request contract, covered by focused context-engine maintenance tests. The hot-cache half should remain plugin-owned unless core needs a generic API to distinguish debt-needed from policy-superseded states.

## Repair Work Prompt

Fix the host-side bootstrap/reconcile deferred compaction debt gap tracked by https://github.com/openclaw/openclaw/issues/67716. Current main runs bootstrap maintenance in foreground and only queues the background debt-consuming worker for reason "turn", so a context engine with turnMaintenanceMode: "background" can record deferred compaction debt during bootstrap/reconcile without receiving a follow-up run where runtimeContext.allowDeferredCompactionExecution is true. Implement a generic OpenClaw host fix only: do not hard-code lossless-claw ids, thresholds, hot-cache reason strings, or plugin internals. Prefer a narrow scheduling/contract change in src/agents/pi-embedded-runner/context-engine-maintenance.ts and/or src/agents/harness/context-engine-lifecycle.ts so bootstrap/reconcile-created debt can be consumed by an allowed background maintenance pass, while preserving the existing turn-maintenance coalescing, session-lane yielding, and synchronous overflow/manual compaction behavior. Add focused fake-engine regression coverage proving bootstrap maintenance for a background-capable engine schedules or runs a debt-consuming follow-up with allowDeferredCompactionExecution true, and keep existing turn deferral tests green. Update docs only if a public context-engine contract changes. No changelog entry is required unless the PR changes user-facing behavior wording or public plugin API docs.
