## Summary

Keep open. Current main still has the native macOS WebChat transcript-rendering gap: Talk Mode submits the user turn through Gateway `chat.send`, and the Gateway/browser Control UI path can broadcast and consume `session.message`, but the SwiftUI chat transport/view model still neither subscribes to nor maps those transcript-update events.

## What This Changes

Talk Mode voice transcripts should render as user messages in the WebChat / Control UI thread when spoken input is submitted.

## Best Possible Solution

Add native macOS SwiftUI WebChat handling for external session transcript updates: subscribe to the active session's transcript events, map `session.message` into the shared chat transport/view model, and refresh or merge the active thread so Talk Mode user turns appear like typed input while preserving browser Control UI behavior.

## Reproduction Assessment

Yes. The source-level reproduction path is clear: open native macOS WebChat on the main session, submit a Talk Mode transcript, and the Gateway can emit `session.message` while the native transport neither subscribes to nor maps that event, so the user turn is not added by the native view model.

## Solution Assessment

Yes. The narrow maintainable fix is native event/subscription handling plus regression coverage, not a new opt-in `talk.echoTranscript` setting; the only implementation detail to handle carefully is showing the spoken transcript rather than the full Talk model prompt if needed.
