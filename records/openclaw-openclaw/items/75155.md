## Summary

Keep open. Current main still has a real macOS/native WebChat gap: Talk Mode submits through `chat.send` outside the chat view model's typed-input path, gateway/browser code can broadcast and react to `session.message`, but the native SwiftUI WebChat transport neither subscribes to session transcript updates nor maps `session.message` into the chat view model.

## What This Changes

User voice transcripts from macOS Talk Mode should appear as user messages in the WebChat/Control UI thread alongside assistant replies.

## Best Possible Solution

Add native macOS SwiftUI WebChat handling for external transcript updates: subscribe to the active session's `session.message` stream, map the event into the shared chat transport/view model, and refresh or merge history for the active session when Talk Mode submits a user turn. Preserve browser Control UI behavior and make the visible card show the spoken transcript rather than Talk's model-facing wrapper.

## Reproduction Assessment

Yes. A source-level reproduction is clear: open native macOS WebChat on the main session, submit a Talk Mode transcript, and the Talk path uses `GatewayConnection.chatSend` while native WebChat does not subscribe to or handle the gateway `session.message` update that would reveal the user turn. I did not run a live macOS app reproduction in this read-only review.

## Solution Assessment

Yes, the requested behavior is the right fix target, but the narrow maintainable implementation is native event/subscription handling plus transcript display sanitation rather than a new opt-in `talk.echoTranscript` setting. The browser Control UI path already has the corresponding reload behavior and should remain unchanged except for regression coverage if shared contracts move.
