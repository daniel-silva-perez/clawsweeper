## Summary

Keep open. Current main still has the reported native macOS gap: Talk Mode sends through `chat.send` outside the chat view model's optimistic typed-input path, while the native SwiftUI WebChat transport does not subscribe to or map `session.message` transcript broadcasts. Browser Control UI has a reload path for those events, so the remaining bug is the macOS/native WebChat event handling path.

## What This Changes

The issue requests that macOS Talk Mode speech transcripts render as user messages in the WebChat/Control UI thread alongside assistant replies.

## Best Possible Solution

Add native macOS SwiftUI WebChat handling for external session transcript updates: subscribe to the relevant gateway session-message stream, map `session.message` into the shared chat transport/view model, and refresh or merge the active thread when Talk Mode submits a user turn. Preserve the browser Control UI behavior, and ensure the visible user text is the spoken transcript rather than the full Talk model prompt wrapper.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: submit a macOS Talk Mode turn while native WebChat is open, and Talk Mode uses `GatewayConnection.chatSend` while native WebChat neither subscribes to nor maps the resulting `session.message` transcript update. I did not run a live macOS session in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is native WebChat session-message subscription/mapping plus regression coverage. A new opt-in `talk.echoTranscript` setting is not the best first fix because typed and spoken user turns should share the visible chat history contract by default.
