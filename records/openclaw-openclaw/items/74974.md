## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` params, so the PR addresses a real Paperclip interoperability bug, but the proposed vendor-named public protocol field conflicts with the repo's generic core-boundary guidance and overlaps with open #69139; this needs maintainer/API-owner judgment before merge.

## What This Changes

The PR adds an optional opaque `paperclip` field to Gateway `AgentParamsSchema`, mirrors it into generated Swift `AgentParams` models, and adds validator tests for Paperclip metadata while preserving rejection of unrelated unknown fields.

## Best Possible Solution

Land one additive, documented, generic opaque Gateway agent metadata field such as `adapterMeta` or `pluginParams`, regenerate affected client models, keep strict rejection for unrelated root fields, add Paperclip regression coverage, and then close the related Paperclip reports after the accepted fix merges.

## Reproduction Assessment

Yes. A Gateway `agent` request with `message`, `idempotencyKey`, and a root `paperclip` field has a high-confidence rejection path on current main because the schema is closed, AJV preserves unknown fields, and dispatch validates before runtime handling; #74635 provides the matching live error string.

## Solution Assessment

No. The PR unblocks the immediate payload shape, but a vendor-named root field is not the narrowest maintainable core protocol shape under the repo's generic-seam guidance unless a maintainer explicitly approves that exception.
