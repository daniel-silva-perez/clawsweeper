## Summary

Keep this PR open. The Paperclip regression is real on current main, but the proposed fix adds a Paperclip-named root field to the public Gateway protocol; that needs maintainer/API-owner approval or a generic opaque metadata field before merge.

## What This Changes

The PR adds optional opaque `paperclip` metadata to `AgentParamsSchema`, regenerates Swift Gateway models, and adds validator coverage for Paperclip payloads.

## Best Possible Solution

Land one additive, documented, generic opaque Gateway agent metadata field such as `adapterMeta` or `pluginParams`, regenerate affected client models, keep strict rejection for unrelated root fields, add Paperclip regression coverage, and close the related Paperclip reports after that fix merges.

## Reproduction Assessment

Yes. A Gateway `agent` request with `message`, `idempotencyKey`, and root `paperclip` has a high-confidence rejection path on current main because the schema is closed, AJV preserves unknown fields, and dispatch validates before runtime handling; #74635 provides a live Paperclip setup and error string.

## Solution Assessment

No. The PR unblocks the immediate payload shape, but a vendor-named root field is not the narrowest maintainable core protocol shape under the repo’s generic-seam guidance unless a maintainer explicitly approves the exception.
