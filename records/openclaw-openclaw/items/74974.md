## Summary

Keep this PR open. Current main still rejects a root-level `paperclip` field on Gateway `agent` params, so the reported Paperclip failure is valid; however, the proposed fix adds a vendor-named field to the public Gateway protocol and needs maintainer/API-owner judgment or a generic metadata namespace before merge.

## What This Changes

The PR adds an optional opaque root-level `paperclip` field to Gateway `AgentParamsSchema`, mirrors it into generated Swift `AgentParams` models, and adds validator coverage for accepting Paperclip metadata while rejecting unrelated unknown fields.

## Best Possible Solution

Land one additive, documented, generic opaque Gateway agent metadata field such as `adapterMeta` or `pluginParams`, regenerate affected client models, keep strict rejection for unrelated root fields, add Paperclip regression coverage, and close #74635 plus related duplicates only after the accepted fix merges.

## Reproduction Assessment

Yes. A Gateway `agent` request containing `message`, `idempotencyKey`, and a root `paperclip` property will fail validation on current main because the schema is closed and the handler validates before dispatch; #74635 provides the live Paperclip error and setup path.

## Solution Assessment

No. The PR unblocks the immediate Paperclip payload, but adding a vendor-named root field is not the narrowest maintainable core protocol contract unless a maintainer explicitly approves that exception; a namespaced opaque metadata field is the safer direction.
