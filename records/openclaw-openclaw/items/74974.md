## Summary

Keep this PR open. The underlying Paperclip Gateway bug is still real on current main, but this branch makes a Paperclip-named root field part of the public Gateway protocol; that needs maintainer/API-owner judgment and consolidation with the overlapping open Paperclip PR rather than automatic cleanup closure.

## What This Changes

The PR adds an optional opaque root-level `paperclip` field to Gateway `AgentParamsSchema`, mirrors it into generated Swift Gateway models, and adds validator tests for Paperclip metadata while preserving rejection of unrelated unknown root fields.

## Best Possible Solution

Land one additive, documented, generic opaque Gateway agent metadata field such as `adapterMeta` or `pluginParams`, regenerate affected client models, keep strict rejection for unrelated root fields, add Paperclip regression coverage, and close the related Paperclip reports after that fix merges.

## Reproduction Assessment

Yes. A Gateway `agent` request with `message`, `idempotencyKey`, and root `paperclip` has a high-confidence rejection path on current main because the schema is closed, AJV preserves unknown fields, and dispatch validates before runtime handling; #74635 provides the matching live error string.

## Solution Assessment

No. The PR unblocks the immediate payload shape, but a vendor-named root field is not the narrowest maintainable core protocol shape under the repo’s generic-seam guidance unless a maintainer explicitly approves that exception.
