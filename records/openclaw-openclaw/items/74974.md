## Summary

Keep this PR open. The Paperclip failure is a valid Gateway protocol bug on current main, but the proposed vendor-named root `paperclip` field needs maintainer/API-owner judgment because it makes one external adapter name part of the public `agent` params contract and overlaps the still-open #69139 work.

## What This Changes

The PR adds an optional opaque root-level `paperclip` field to `AgentParamsSchema`, mirrors it into generated Swift Gateway models, and adds validator tests for Paperclip metadata while preserving rejection of unrelated unknown root fields.

## Best Possible Solution

Land one additive, documented, generic opaque Gateway agent metadata field such as `adapterMeta` or `pluginParams`, regenerate affected client models, keep strict rejection for unrelated root fields, add Paperclip regression coverage against the accepted contract, and close the Paperclip reports after the fix merges.

## Reproduction Assessment

Yes. A current-main Gateway `agent` request containing `message`, `idempotencyKey`, and a root `paperclip` property has a high-confidence rejection path because the schema is closed, AJV preserves unknown fields, and the handler validates before dispatch; #74635 provides the matching live error string.

## Solution Assessment

No. The PR unblocks the immediate Paperclip payload, but a vendor-named root field is not the narrowest maintainable core protocol shape under the repo’s generic-seam and protocol-boundary guidance unless a maintainer explicitly approves that exception.
