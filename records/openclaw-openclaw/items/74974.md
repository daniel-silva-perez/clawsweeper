## Summary

Keep open. Current main still has the Paperclip interoperability failure, but this PR makes a vendor-named field part of the public Gateway agent protocol where repo guidance favors generic core seams; the overlap with #69139 also means maintainers should choose one canonical protocol shape before merging.

## What This Changes

The PR adds an optional opaque `paperclip` field to Gateway `AgentParamsSchema`, mirrors it into generated Swift `AgentParams` models, and adds validator coverage for accepting Paperclip metadata while rejecting unrelated unknown fields.

## Best Possible Solution

Land one additive, documented, generic opaque adapter metadata contract for Gateway `agent` params, regenerate affected client models, preserve strict rejection for unrelated root fields, keep Paperclip regression coverage, and then consolidate the related Paperclip reports after the canonical fix merges.

## Reproduction Assessment

Yes. A Gateway `agent` request with `message`, `idempotencyKey`, and a root `paperclip` field has a high-confidence rejection path on current main because the schema is closed, AJV preserves unknown fields, and dispatch validates before runtime handling.

## Solution Assessment

No. The PR likely unblocks the immediate payload, but a vendor-named root field is not the narrowest maintainable core protocol shape unless a maintainer explicitly approves that exception; a generic opaque adapter metadata field is safer.
