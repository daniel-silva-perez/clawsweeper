## Summary

Keep this PR open for maintainer/API-owner review. Current main still rejects the reported Paperclip `agent` payload, so the bug is real, but the proposed public protocol shape bakes a vendor-named root field into Gateway `AgentParams` and overlaps the still-open #69139 fix path.

## What This Changes

The PR adds an optional opaque `paperclip` field to Gateway `AgentParamsSchema`, regenerates Swift Gateway models, and adds validator coverage for Paperclip metadata while preserving rejection of unrelated unknown fields.

## Best Possible Solution

Land one additive Gateway `agent` metadata contract that is extension-safe: preferably a generic opaque namespace such as `adapterMeta` or `pluginParams`, with docs/tests/generated clients updated, and either coordinate Paperclip to send that field or explicitly approve a temporary Paperclip compatibility shim.

## Reproduction Assessment

Yes. A Gateway `agent` request containing `message`, `idempotencyKey`, and a root `paperclip` property follows a high-confidence rejection path on current main: closed schema, AJV with `removeAdditional: false`, and pre-dispatch validation; #74635 supplies the matching live error.

## Solution Assessment

No. The PR solves the immediate payload rejection, but a vendor-named root field is not the narrowest maintainable core protocol shape unless a maintainer explicitly approves that exception; a generic opaque adapter metadata namespace is safer.
