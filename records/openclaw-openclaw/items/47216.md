## Summary

Keep this PR open. Current main still emits the sender metadata block for direct webchat/TUI turns because `buildInboundUserContextPrefix` suppresses direct-webchat conversation identifiers but pushes `Sender (untrusted metadata)` whenever sender fields are present; the PR’s narrow guard matches the observed gateway/TUI path and I found no blocking review findings.

## What This Changes

The PR adds a `buildInboundUserContextPrefix` guard to suppress sender metadata for direct webchat contexts and adds a regression test for TUI/gateway-client sender identities.

## Best Possible Solution

Land this focused prompt-context fix, keeping sender metadata for external direct and group/channel chats while omitting transport-only sender blocks from direct webchat/TUI turns, with the new regression test and the changed gate run before merge.
