## Summary

Keep this PR open. Current main still emits the direct-webchat/TUI sender block the PR targets, and the proposed prompt-boundary guard is narrow and consistent with the existing direct-webchat conversation-info suppression; the remaining issue is that the source branch needs to be ported to current main before it can land.

## What This Changes

The PR suppresses `Sender (untrusted metadata)` for direct webchat inbound prompt context and adds a regression test for TUI/gateway transport sender identities.

## Best Possible Solution

Port the same direct-webchat sender guard onto the current `buildInboundUserContextPrefix` implementation, add the colocated regression test, and keep gateway sender construction unchanged so internal identity remains available for authorization, routing, and attribution.

## Reproduction Assessment

Yes. On current main, a direct `TemplateContext` whose channel resolves to `webchat` and includes `SenderId`/`SenderName` reproduces by source trace: conversation info is suppressed, but the sender block still renders.

## Solution Assessment

Yes, conceptually. Filtering at the prompt helper is the narrowest maintainable fix because it removes transport-only direct-webchat sender metadata from model context while preserving sender metadata for group chats and external-channel direct chats; the branch just needs a current-main port.
