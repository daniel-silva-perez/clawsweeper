## Summary

Keep open. Current main has bundle-backed LSP startup and a narrow Pi tool surface, but it does not collect LSP diagnostics, track server health/performance, or expose the requested in-editor diagnostics dashboard. The paired same-author LSP setup issue remains open, so this should be scoped with that related work rather than closed independently.

## What This Changes

The issue requests real-time LSP diagnostics in the editor, including server status, diagnostic streams, performance metrics, searchable logs, troubleshooting suggestions, history, reports, and CI/pre-commit integration.

## Best Possible Solution

Keep this open with #65251 and first define a generic LSP diagnostics contract: capture server lifecycle, stderr/errors, request latency, and publishDiagnostics events in an owner runtime or plugin service, then expose that data through a documented plugin/gateway API before choosing Control UI, plugin-served UI, or external editor integration.
