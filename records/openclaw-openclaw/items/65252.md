## Summary

Keep open. Current main has bundle-backed stdio LSP startup plus generic Gateway diagnostics/log viewing, but it still does not collect or expose LSP publishDiagnostics events, server health, request latency, searchable LSP logs, or an editor diagnostics dashboard. The same-author LSP setup proposal #65251 is still open in the provided GitHub context, so this should be scoped with that work rather than closed independently.

## What This Changes

Request to add real-time in-editor LSP diagnostics with server status, diagnostic feed, performance metrics, logs, troubleshooting suggestions, trend analysis, export/share, and notifications.

## Best Possible Solution

Define a generic LSP diagnostics contract first: capture server lifecycle, stderr/errors, request timing, and publishDiagnostics events, expose that data through a documented plugin/gateway API, then decide whether the first UI belongs in Control UI, a plugin-served view, or an external editor integration.

## Reproduction Assessment

Not applicable; this is a feature proposal, not a bug report. The current-main verification path is source/docs/UI inspection showing that LSP diagnostics are not captured or surfaced today.

## Solution Assessment

No, a monolithic dashboard is not the best first fix. The maintainable path is a narrow diagnostics/event contract with tests before adding UI, smart repair, history, or CI integrations.
