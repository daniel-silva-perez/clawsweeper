## Summary

This PR should stay open because the reported Telegram ACP-bound DM-topic bug is still source-provable on current main, but the branch changes retired `src/telegram` paths instead of the active bundled Telegram plugin under `extensions/telegram/src`. The guard idea is the right shape, but the patch as written would not affect the runtime code that currently appends the DM-topic suffix after bound-route resolution.

## What This Changes

The PR adds a Telegram route-level flag in `src/telegram` to skip DM-topic thread suffixing for already-bound ACP routes and adds inbound-message and native-command regression coverage in the same retired tree.

## Best Possible Solution

The active Telegram plugin should surface whether route resolution produced an authoritative bound session key and honor that signal at every DM-topic suffixing site, with regression coverage proving bound DM topics keep the exact ACP key while unbound DM topics still get isolated thread sessions.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: runtime conversation binding can replace `route.sessionKey` with an ACP target key, and the active inbound/native-command DM-topic paths then append `:thread:<chatId>:<topicId>` after route resolution.

## Solution Assessment

No, not as written. The route-level guard is a narrow maintainable fix, but it must be ported to `extensions/telegram/src` and consumed by the active inbound, native-command, and session-state paths instead of the retired `src/telegram` tree.
