## Summary

Current main still has the reported identity-keyed web-provider snapshot cache behavior. The issue is recent, has no protected labels or maintainer author association in the provided GitHub context, and the related #73728/#73729 reports are adjacent performance problems rather than a canonical replacement for this one.

## Best Possible Solution

Keep this issue open and fix the shared web-provider cache in `src/plugins/web-provider-runtime-shared.ts`: either canonicalize config identity before lookup or replace the outer WeakMap with a bounded content-keyed cache that includes the existing env/workspace/plugin-scope/origin inputs plus a stable config/runtime-config key. Add regression coverage for web-search and web-fetch calls with fresh but equivalent config objects so the second call reuses the snapshot within TTL.
