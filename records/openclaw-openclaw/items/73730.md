## Summary

Current main still has the reported identity-keyed web-provider snapshot cache behavior. The issue is recent, unprotected, not maintainer-authored, and the related #73728/#73729 items are adjacent cache/performance reports rather than a canonical replacement. The discussion also confirms the diagnosis and the remaining design choice, so this should stay open for a real fix rather than be swept closed.

## Best Possible Solution

Keep this issue open and fix the shared cache in `src/plugins/web-provider-runtime-shared.ts`: either canonicalize equivalent resolved configs before lookup or replace the outer identity WeakMap with a bounded content-keyed cache whose key includes provider-affecting config content plus the existing workspace/env/origin/scope inputs. Add regression tests for both web-search and web-fetch where two fresh but equivalent config objects reuse the snapshot within TTL, and tests for genuinely different plugin/web config content not colliding.

## Risks / Open Questions

- A content-keyed cache rewrite must include the config fields that affect provider availability, especially plugin allowlists, enabled entries, plugin web-search/web-fetch config, workspace, origin, onlyPluginIds, env-derived cache key, bundled allowlist compatibility, and TTL.
- Existing tests currently assert cache retention after in-place config mutation; maintainers should decide whether that behavior remains intended or should change as part of a safer config-fingerprint invalidation model.
