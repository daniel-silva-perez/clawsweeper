## Summary

Keep this open. Current main still has the reported identity-keyed web-provider snapshot cache, and the related #73728/#73729 items are adjacent performance reports rather than a canonical replacement. The hclsys follow-up also confirms the diagnosis and the key design constraint: a fix cannot key only on the existing string cache key because config content is not encoded there.

## Best Possible Solution

Keep this issue open and choose a maintainer-approved cache shape. The cleaner fix is to make the shared web-provider snapshot cache content-keyed with a bounded TTL entry whose key includes the existing workspace/env/scope/origin inputs plus the relevant plugin and web provider config fingerprint; alternatively, intern immutable resolved configs before the existing WeakMap. Add focused regression tests for both web-search and web-fetch where two fresh but equivalent config objects reuse the snapshot, plus a negative test where different plugin/web config content does not collide.

## Risks / Open Questions

- A naive fix that drops the outer WeakMap and keys only on the current cacheKey could collide across different plugin allowlists, disabled plugin entries, or web provider config values.
- Prematurely closing would leave at least the web-fetch provider path, and any web-search path without a stable runtime snapshot match, paying repeated plugin-load cost for fresh equivalent configs.
