## Summary

Current main still has the reported identity-keyed web-provider snapshot cache behavior. The stable string cache key and TTL are only consulted inside a WeakMap bucket keyed by the exact config object, so fresh but equivalent config objects still miss. The web-search and web-fetch runtime paths still feed caller/runtime config into that shared resolver, and the existing memoization tests cover same-reference reuse rather than fresh equivalent config objects. Related #73728 and #73729 are adjacent performance reports, not a canonical replacement for this issue.

## Best Possible Solution

Keep this issue open and fix the shared web-provider snapshot cache in src/plugins/web-provider-runtime-shared.ts. Either canonicalize equivalent runtime config identity before lookup or replace the outer WeakMap with a bounded content-keyed cache that includes the existing workspace/env/plugin-scope/origin/bundled-compat inputs plus a stable config/runtime-config fingerprint. Add regression coverage for both web-search and web-fetch where the second call uses a fresh but equivalent config object and reuses the cached snapshot within TTL.

## Risks / Open Questions

- A fix must preserve invalidation for workspaceDir, env/cache-control values, plugin scope, origin, bundled allowlist compatibility, and TTL; a content key that omits those inputs could make provider availability stale.
- Replacing WeakMap identity caching with a plain content-keyed Map should be bounded or aggressively expired so long-lived gateways do not retain unbounded config-derived entries.
