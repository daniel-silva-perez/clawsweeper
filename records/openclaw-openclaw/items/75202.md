## Summary

Keep this PR open. Current main still has the linked MEDIA parser bug, but the proposed regex is too broad and can truncate already-valid media paths or HTTPS URLs that contain a comma after an extension.

## What This Changes

The PR changes `src/media/parse.ts` so `cleanCandidate` truncates a MEDIA candidate at a file extension before JSON-like boundary characters and adds one regression case in `src/media/parse.test.ts`.

## Best Possible Solution

Keep the fix in the shared MEDIA parser, but narrow the suffix stripping to clear serialized tool-result JSON after a media source and add regression coverage for both the polluted #75182 shape and valid paths/URLs containing comma or bracket characters after an extension.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: `MEDIA_TOKEN_RE` captures through the end of the line, `cleanCandidate` does not remove embedded JSON after `.png`, and slash-prefixed local candidates are accepted as media.

## Solution Assessment

No. The proposed direction is in the right file and fixes the reported example, but applying the extension-boundary truncation to every candidate is broader than necessary and can alter valid attachment sources.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Narrow the JSON suffix trim to preserve valid media paths:** `src/media/parse.ts:39`
  - body: The new cleanup runs for every MEDIA candidate, including already-valid local paths and HTTPS URLs. Because comma is in the boundary class, a valid source like `MEDIA:/tmp/report.pdf,final` or `MEDIA:https://example.com/img/photo.png,large` is truncated before validation, so OpenClaw would send or fetch a different attachment. Please limit this trim to an actual serialized JSON suffix shape, or add coverage that preserves valid comma-bearing sources.
  - confidence: 0.88
