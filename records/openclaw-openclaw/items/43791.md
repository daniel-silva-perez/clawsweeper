## Summary

Keep this PR open. Current main still does not preserve full embedded tool-result `ReplyPayload` values in the final embedded Pi run result, so the work is not obsolete, but the current PR head has a real timeout regression and a changelog policy miss before it can merge.

## What This Changes

The PR buffers embedded Pi runner tool-result `ReplyPayload` values, prepends buffered payloads to final non-streaming run payloads, clears them on compaction retry, updates subscribe types/tests, and adds a changelog entry.

## Best Possible Solution

Rebase the useful full-payload preservation work onto current main, buffer complete `ReplyPayload` objects only for final consumers that did not receive live tool results, preserve the existing timeout/error synthesis when no assistant reply was produced, keep compaction-retry buffer clearing, and add the required changelog attribution.

## Reproduction Assessment

Yes. The original gap is reproducible with a mocked embedded attempt that emits a tool-result payload plus assistant text, and source inspection confirms current main only preserves pending tool media. The PR regression is reproducible with a mocked `timedOut: true` attempt containing a buffered media payload and no assistant output.

## Solution Assessment

No. The implementation direction is maintainable, but the current branch makes buffered tool callbacks count as final payloads before timeout handling. The safer fix is to preserve full buffered payloads after keeping timeout/error detection based on assistant or intentional final reply production.
