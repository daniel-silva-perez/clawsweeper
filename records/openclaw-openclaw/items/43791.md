## Summary

Keep this PR open. Current main still lacks full embedded tool-result `ReplyPayload` preservation, so the PR addresses a real gap, but the current branch is not merge-ready because buffered tool-result payloads can suppress the explicit timeout error path and the changelog entry lacks the required attribution.

## What This Changes

The PR buffers embedded Pi runner tool-result `ReplyPayload` values in attempt results, replays them into final run payloads for non-streaming callers, resets the buffer on compaction retry, updates subscribe types/tests, and adds a changelog entry.

## Best Possible Solution

Rebase the useful payload-preservation work onto current main, preserve full `ReplyPayload` metadata only for final consumers that did not already receive live tool results, make timeout synthesis depend on assistant/final reply production rather than buffered tool callbacks, keep compaction-retry buffer reset coverage, and credit the changelog entry per repo policy.

## Reproduction Assessment

Yes. The PR body gives a focused mocked embedded-run path for the original preservation bug, and the timeout regression can be reproduced by mocking a timed-out attempt with non-empty `toolResultPayloads`, no assistant payloads, and no live `onToolResult` callback.

## Solution Assessment

No. The direction is maintainable, but the current implementation should not let buffered tool callbacks satisfy the timeout guard; the safer fix is to assemble/replay buffered payloads without changing the no-assistant timeout contract.
