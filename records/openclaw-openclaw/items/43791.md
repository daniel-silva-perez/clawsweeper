## Summary

Keep open. Current main still only carries pending tool media into embedded run results, not full embedded tool-result `ReplyPayload` values, so the PR is not obsolete; however the branch should not merge as-is because buffered tool callbacks can now suppress the explicit timeout error path, and the changelog entry misses required attribution.

## What This Changes

The PR buffers embedded Pi runner tool-result `ReplyPayload` values in attempt results, prepends them to final non-streaming run payloads, clears the buffer on compaction retry, updates subscribe types/tests, and adds a changelog entry.

## Best Possible Solution

Keep the useful full-payload preservation direction, but rebase it onto current main and make timeout/error synthesis depend on assistant or intentional final reply production rather than buffered tool callbacks. Preserve full `ReplyPayload` metadata only for final consumers that did not receive live tool results, keep the compaction-retry buffer reset, and add the required changelog credit.

## Reproduction Assessment

Yes. The original gap is visible by comparing current main's attempt contract and final assembly, which only carry pending media, with the PR's mocked embedded-run tests for returned `toolResultPayloads`. The timeout regression can be reproduced with a mocked timed-out attempt that has buffered tool payloads, no assistant payloads, and no live `onToolResult` callback.

## Solution Assessment

No. The implementation direction is maintainable, but this branch folds buffered tool callbacks into `payloads` before timeout handling; the safer fix is to replay buffered payloads after preserving the existing no-assistant timeout contract.
