## Summary

Keep this PR open. Current main still does not implement the PR's intended full embedded tool-result `ReplyPayload` preservation; it only carries a narrow orphaned tool-media fallback into final payload assembly.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is to rebase onto current main, preserve full `ReplyPayload`s only for callers that are not already streaming `onToolResult`, reconcile that with the existing orphaned tool-media fallback, preserve explicit timeout failures when no assistant reply exists, clear buffers on compaction retry, and add focused tests plus the required changelog attribution.

## Risks / Open Questions

- The PR changes the embedded final-result contract; maintainers still need to decide which `ReplyPayload` metadata is safe and expected to replay for non-streaming consumers.
- Timeout semantics still need explicit review: buffered tool-result payloads must not hide a timeout when no assistant reply was produced.
- The implementation needs to stay aligned with current main's orphaned tool-media fallback so media is not duplicated, reordered, or stripped.
- The PR branch predates current main's split embedded-runner test layout and reset paths, so it needs a rebase and targeted regression proof before merge.
