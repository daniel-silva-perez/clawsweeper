## Summary

Keep open. Current main still does not preserve full embedded tool-result `ReplyPayload` values in final non-streaming embedded Pi run results, so the branch addresses a real remaining gap. The branch is not ready to merge because it can let buffered tool-result payloads suppress the existing timeout error path, and its changelog entry is missing the required contributor attribution.

## What This Changes

The PR buffers `ReplyPayload` values emitted through embedded Pi runner `onToolResult`, prepends buffered payloads to non-streaming final run payloads, resets the buffer on compaction retry, extends related attempt/subscribe types, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the preservation work, but update it so complete buffered `ReplyPayload` values are available to non-streaming final consumers without letting ordinary buffered callbacks mask timeout/error handling. The final branch should preserve the no-duplicate streamed path, clear abandoned retry buffers on compaction retry, rebase onto current main, and add the required changelog attribution.

## Reproduction Assessment

Yes. The original gap is reproducible with a mocked embedded attempt that emits a tool-result payload plus assistant text and then inspecting `runEmbeddedPiAgent(...).payloads`; current main has no `toolResultPayloads` contract. The PR regression is reproducible by a mocked timed-out attempt with buffered tool-result payloads and no assistant payload, which the branch currently treats as a non-error final payload.

## Solution Assessment

No. The implementation direction is maintainable, but assembling buffered tool-result payloads before the timeout guard is not the safest final contract. A safer fix should decide timeout/error outcomes from assistant or intentional final deliverables first, then add buffered tool-result payloads only where they do not hide run failure state.
