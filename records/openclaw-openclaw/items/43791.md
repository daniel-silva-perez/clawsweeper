## Summary

Keep this PR open. Current main still lacks a full `ReplyPayload` preservation path for embedded tool-result callbacks, so the branch is not obsolete, but the current PR is not merge-ready because it makes buffered tool-result callbacks count as final payloads before timeout handling and still needs the required changelog attribution.

## What This Changes

The PR buffers embedded Pi runner tool-result `ReplyPayload` values, prepends buffered payloads to final non-streaming run payloads, clears them on compaction retry, updates subscribe types/tests, and adds a changelog entry.

## Best Possible Solution

Keep the useful preservation work in this PR, but make the final assembly preserve complete buffered `ReplyPayload` values only for non-streaming final consumers after the timeout/error decision still reflects whether an assistant or intentional final deliverable was produced; keep compaction retry buffer clearing and add the required changelog credit.

## Reproduction Assessment

Yes. The original gap is reproducible with a mocked embedded attempt that emits a tool-result reply payload plus assistant text and then inspecting `runEmbeddedPiAgent(...).payloads`; current main only assembles assistant payloads plus pending tool media. The PR regression is reproducible with a mocked timed-out attempt that has buffered tool-result payloads and no assistant output.

## Solution Assessment

No. The implementation direction is maintainable, but this branch folds buffered payloads into `payloads` before timeout handling and adds an uncredited changelog bullet. The safer fix is to preserve buffered payloads without letting ordinary tool callbacks suppress timeout/error synthesis.
