## Summary

Keep this PR open. Current main still does not preserve full embedded tool-result `ReplyPayload` objects in final run payloads; it only merges a narrower orphaned tool-media fallback. The PR is still relevant, but it should not merge as-is because it broadens timeout handling so a timed-out run with only buffered tool output can look like a partial success, and it also needs rebase/current-surface review plus changelog attribution cleanup.

## What This Changes

The PR buffers embedded runner tool-result `ReplyPayload`s, prepends them to final run payloads when no live tool-result callback is present, clears that buffer on compaction retry, and adds regression tests plus a changelog entry.

## Best Possible Solution

Rebase the useful payload-preservation work onto current main, preserve full `ReplyPayload` metadata only for non-live final-payload consumers, keep the existing orphaned-media fallback from duplicating or reordering media, base timeout errors on whether an assistant/final reply was produced rather than any buffered tool callback, and add targeted tests for metadata, singular media, compaction retry reset, streamed suppression, and timed-out tool-only runs.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve timeout errors when only tool payloads are buffered:** `src/agents/pi-embedded-runner/run.ts:1519-1520`
  - body: The branch combines buffered tool-result payloads with assistant payloads before the timeout guard. If the model times out after emitting a tool result but before producing an assistant reply, `payloads.length` is non-zero and the run returns a partial-looking tool payload instead of the explicit timeout error. Keep the timeout decision based on assistant/final reply content, or settle that behavior explicitly with a regression test.
