## Summary

Keep open. Current main still accepts hook-supplied session keys after prefix checks without rejecting internal `subagent:`, `acp:`, or `cron:` namespaces, while this PR addresses that boundary and incorporates the earlier mapping fail-fast and whitespace-bypass review feedback. I found no blocking correctness or supply-chain issue in the proposed diff, but it is security-sensitive and still needs maintainer review plus CI/Testbox validation before merge.

## What This Changes

The PR adds a gateway hook session-key guard for reserved internal namespaces across request, default, static mapping, and templated mapping paths, with new helper and regression tests.

## Best Possible Solution

Land a maintainer-reviewed version of the reserved-namespace guard that keeps detection centralized at the hook session-key boundary, fails fast for non-templated agent mapping config, preserves wake mapping compatibility, and verifies the gateway hook server path in CI/Testbox.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: enable request hook session keys with `allowedSessionKeyPrefixes` including `agent:` and send `agent:main:subagent:worker` to `/hooks/agent`; the current resolver accepts that key after prefix validation.

## Solution Assessment

Yes. The PR is the narrow maintainable fix because it normalizes the hook-supplied key, unwraps routed agent prefixes, and applies one reserved-prefix check at the config-load and dispatch boundaries that can supply hook session keys.
