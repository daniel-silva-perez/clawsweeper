## Summary

Keep open. Current main still accepts hook-provided session keys after only request-policy and prefix checks, while this PR adds the missing reserved-namespace guard across request, default config, static mapping, and runtime mapping paths. The branch is useful and security-sensitive, so the next step is maintainer review and validation rather than cleanup closure or an automated replacement.

## What This Changes

The PR adds gateway hook validation that rejects external hook session keys targeting `subagent:`, `acp:`, or `cron:` internal namespaces, including default and mapping configuration paths, with gateway regression tests.

## Best Possible Solution

Land a maintainer-reviewed version of this reserved-namespace guard that keeps detection centralized at the hook session-key boundary, fails fast for non-templated agent mapping config, and preserves the existing runtime policy for templated mapping and transform-provided keys.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists on current main by enabling hook request session keys with `allowedSessionKeyPrefixes` including `agent:` and sending `agent:main:subagent:worker` to `/hooks/agent`; the current resolver accepts the value before dispatch.

## Solution Assessment

Yes. The PR is the narrow maintainable direction: it checks the normalized key after recursively unwrapping routed `agent:` prefixes and applies that shared check in the config-load and dispatch paths that can supply hook session keys.
