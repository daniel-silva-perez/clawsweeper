## Summary

Keep open for maintainer review. Current main still accepts externally supplied hook session keys after request and prefix policy without any reserved internal namespace check, while this PR addresses the valid gateway boundary by adding reserved namespace validation at config load and dispatch; source review found no blocking introduced bug, but the change is security-sensitive and needs maintainer review plus CI/Testbox validation rather than cleanup closure.

## What This Changes

This PR adds a gateway hook session-key guard that rejects reserved `subagent:`, `acp:`, and `cron:` namespaces from direct hook request keys, default hook session keys, static or templated mapping keys, and transform-provided agent session keys, with unit and server regressions.

## Best Possible Solution

Land a maintainer-reviewed version of the reserved-namespace guard that keeps detection centralized at the hook session-key boundary, preserves wake mapping compatibility, and validates request, default, static mapping, templated mapping, and transform-provided agent session keys on the final merge ref.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: enable request hook session keys with `allowedSessionKeyPrefixes` including `agent:` and send `agent:main:subagent:worker` to `/hooks/agent`; the current resolver accepts it after request and prefix policy checks.

## Solution Assessment

Yes. The proposed approach is the narrow maintainable fix because it normalizes hook-supplied keys, unwraps routed agent prefixes before reserved-prefix checks, and applies the guard at config-load and dispatch boundaries that can supply hook session keys.
