## Summary

Keep open. Current main still accepts hook-provided session keys after request/prefix checks without rejecting internal `subagent:`, `acp:`, or `cron:` namespaces, while this PR adds a narrow guard across request, default, static mapping, templated mapping, and transform-provided paths. I found no discrete blocking correctness or supply-chain finding in the diff; the remaining action is maintainer security review plus CI/Testbox validation.

## What This Changes

The PR adds a gateway hook session-key helper, reserved namespace rejection in hook config/session resolution, static mapping validation, and unit/server regression tests.

## Best Possible Solution

Land a maintainer-reviewed version of the reserved-namespace guard that keeps detection centralized at the hook session-key boundary, fails fast for non-templated agent mapping config, preserves wake mapping compatibility, and verifies the gateway hook server path on the final merge ref.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: enable request hook session keys with `allowedSessionKeyPrefixes` including `agent:` and send `agent:main:subagent:worker` to `/hooks/agent`; the current resolver accepts that key after request/prefix policy checks.

## Solution Assessment

Yes. The proposed approach is the narrow maintainable fix because it normalizes hook-supplied keys, unwraps routed agent prefixes before reserved-prefix checks, and applies the guard at the config-load and dispatch boundaries that can supply hook session keys.
