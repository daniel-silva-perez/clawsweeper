## Summary

Current `main` still does not implement this PR's central feature. The CLI banner tagline contract remains limited to `random`, `default`, and `off`; there is no accepted `taglineScriptFile` config key; docs and generated schema/help metadata still document only the static modes; and the startup/banner path has no script-loading or script-execution boundary. The related earlier PR #41147 was closed unmerged, so this PR is not obsolete; it remains the active implementation candidate and still needs maintainer product/security review because the proposed diff executes user-configured JavaScript during CLI startup/help rendering.

## Best Possible Solution

Keep this PR open for maintainer review. If arbitrary JS taglines are acceptable, rebase onto the current startup/help presentation surfaces, execute scripts only when banner/help rendering genuinely needs them, use a defensible execution boundary or document a narrower safety model, update strict/generated schema/help/public docs consistently, and add targeted tests for TTY, non-TTY, JSON, version, hidden-banner, root help, and subcommand help behavior. If maintainers do not want code execution at startup, close with that rationale and consider a safer static custom-tagline config instead.

## Risks / Open Questions

- The proposed feature intentionally executes arbitrary local JavaScript from OpenClaw config during CLI startup/help rendering, so maintainers should explicitly accept or reject that product/security model before merge.
- A same-event-loop timeout does not reliably stop synchronous blocking work or slow top-level module evaluation in the configured script; a worker/child-process boundary or a safer static custom-tagline model may be needed.
- Suppressed-output paths matter: JSON, version, hidden-banner, non-TTY, root help, and subcommand help flows should not execute user scripts unexpectedly or allow script stdout/stderr to pollute machine-readable output.
