## Summary

Keep open. Current main at 6d323ee73640afdf02ed9fa3aded13da23359b98 still does not implement this PR's central feature: CLI banner tagline mode is limited to random/default/off, there is no taglineScriptFile config field, and no script-loading helper. The earlier related PR #41147 was closed unmerged, so this PR remains the live candidate, but it needs maintainer product/security review because it would execute user-configured JavaScript during CLI startup.

## Best Possible Solution

Keep this PR open for maintainer review. Decide first whether arbitrary JS in CLI banner config is an acceptable product/security tradeoff. If accepted, rebase onto the current applyCliExecutionStartupPresentation/startup-policy/help surfaces, execute scripts only when banner or help rendering genuinely needs them, use an abortable execution boundary or clearly document a narrower safety model, update schema/help/docs consistently, and add targeted tests for TTY, non-TTY, JSON, version, hidden-banner, root help, and subcommand help behavior. If maintainers do not want code execution at startup, close with that rationale and consider a safer static custom-tagline config instead.

## Risks / Open Questions

- The proposed feature intentionally executes arbitrary local JavaScript with the CLI user's privileges, so a configured or compromised file could read OpenClaw state/credentials, spawn processes, write files, or make network calls.
- A Promise.race-style timeout in the same event loop would not reliably stop synchronous blocking work or slow top-level module evaluation; a worker/child-process boundary or a narrower non-code customization model may be needed.
- Suppressed-output paths matter: JSON, version, hidden-banner, non-TTY, root help, and subcommand help flows should not execute tagline scripts unexpectedly or allow script stdout/stderr to pollute machine-readable output.
- The PR targets banner/startup plumbing that has since moved through applyCliExecutionStartupPresentation and startup policy helpers, so a landable version needs a careful rebase and focused tests across help/banner routes.
