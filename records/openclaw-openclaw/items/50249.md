## Summary

Keep open. Current main still does not implement this PR's central feature: CLI banner taglines are limited to `random`, `default`, and `off`; there is no `script` mode, no `taglineScriptFile` config field, and no script-loading helper. The PR remains a live implementation candidate, but it needs maintainer product/security review because it executes user-configured JavaScript during CLI startup.

## Best Possible Solution

Keep this PR open for maintainer review. The next step is to decide whether arbitrary JS in CLI banner config is an acceptable product/security tradeoff. If accepted, rebase onto the current `applyCliExecutionStartupPresentation` and help/startup surfaces, execute scripts only when banner/help rendering genuinely needs them, use a real abortable execution boundary or document a narrower safety model, update schema/help/docs consistently, and add focused tests for TTY, non-TTY, JSON, version, hidden-banner, root help, and subcommand help behavior. If maintainers do not want code execution at startup, close with that rationale and consider a safer static custom-tagline config instead.

## Risks / Open Questions

- The proposed feature intentionally executes arbitrary local JavaScript with the CLI user's privileges during startup, so a configured or compromised file could read OpenClaw credentials/state, spawn processes, write files, or make network calls.
- A same-event-loop `Promise.race` timeout would not reliably stop synchronous blocking work or slow module evaluation; a worker/child-process boundary or a narrower non-code customization model may be needed.
- Suppressed-output paths matter: JSON, version, hidden-banner, non-TTY, root help, and subcommand help flows should not execute tagline scripts unexpectedly or allow script stdout/stderr to pollute machine-readable output.
- The PR targets older banner/startup plumbing, while current main now uses startup policy helpers; rebasing without focused tests could regress help/banner behavior.
