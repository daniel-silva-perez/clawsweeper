## Summary

Keep open. Current main still does not implement this PR's central feature: CLI banner taglines are limited to `random`, `default`, and `off`; there is no `script` mode, no `taglineScriptFile` config field, and no script-loading helper. The PR remains a real implementation candidate, but it is security- and reliability-sensitive because it executes arbitrary local JavaScript during CLI startup, so it needs maintainer product/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The next step is to decide whether arbitrary JS in CLI banner config is an acceptable product/security tradeoff. If accepted, rebase onto current `applyCliExecutionStartupPresentation` and root-help startup paths, execute scripts only when banner/help rendering genuinely needs them, use a real abortable execution boundary or document a narrower safety model, update schema/help/docs consistently, and add focused tests for TTY, non-TTY, JSON, version, hidden-banner, root help, and subcommand help behavior. If maintainers do not want code execution at startup, close with that rationale and consider a safer static custom-tagline config instead.

## Risks / Open Questions

- The proposed feature intentionally executes arbitrary local JavaScript with the CLI user's privileges during startup, so a malicious or compromised configured file could read secrets, write state, run subprocesses, or make network calls.
- Timeouts implemented inside the same event loop would not reliably stop synchronous blocking work or slow module evaluation; an abortable boundary such as a worker or child process, or a narrower non-code customization model, may be needed.
- Suppressed-output paths matter: JSON, version, hidden-banner, non-TTY, root help, and subcommand help flows should not unexpectedly execute user scripts or let script stdout/stderr pollute machine-readable output.
- The PR appears to target older banner/startup plumbing; current main has newer startup policy helpers, so the branch needs a careful rebase and focused tests before merge consideration.
