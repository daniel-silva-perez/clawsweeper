## Summary

Keep this PR open. Current main still does not implement Xiaomi video media understanding: Xiaomi registers speech only, its manifest lacks a media-understanding contract, and the generic fallback only covers image-capable config providers. The contribution is still relevant, but it needs a rework against the current manifest-first provider shape, SDK exports, provider HTTP policy helpers, and test expectations.

## Best Possible Solution

Keep this PR open and ask for a rework on current main. The best patch should add a Xiaomi media-understanding provider under extensions/xiaomi, declare contracts.mediaUnderstandingProviders and mediaUnderstandingProviderMetadata with image/video defaults to mimo-v2-omni, use the shared OpenAI-compatible video helpers plus openclaw/plugin-sdk/provider-http request-policy helpers, register through the current single-provider plugin entrypoint, and add focused Xiaomi tests plus the media docs update.

## Risks / Open Questions

- Merging the PR as-is would not compile cleanly against current main because provider HTTP helpers are no longer exported from openclaw/plugin-sdk/media-understanding.
- The PR sends user video bytes and a Xiaomi bearer token through a hand-built HTTP path without current resolveProviderHttpRequestConfig policy handling for dispatcher/private-network/proxy behavior.
- Registering Xiaomi as image/video capable without manifest defaultModels risks reusing the default text-only xiaomi/mimo-v2-flash model instead of mimo-v2-omni for media understanding.
- The PR lacks focused Xiaomi media-understanding tests for manifest metadata, registration, default model selection, request shape, and transport-policy forwarding.
