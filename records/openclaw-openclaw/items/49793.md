## Summary

Keep this PR open. Current main still does not implement Xiaomi video media understanding: the Xiaomi plugin only registers speech, its manifest does not declare a media-understanding provider, the generic config-provider fallback only covers image-capable models, and the video runner still requires a real `describeVideo` implementation. The PR remains relevant, but it is stale against the current plugin entrypoint, manifest metadata contract, SDK helper layout, and provider HTTP policy path.

## Best Possible Solution

Keep the PR open and ask for a rebase/rework. The best path is a current Xiaomi media-understanding provider in the Xiaomi bundled plugin that declares `contracts.mediaUnderstandingProviders` plus `mediaUnderstandingProviderMetadata`, registers through the modern plugin entrypoint, defaults image and video media understanding to `mimo-v2-omni`, uses the shared OpenAI-compatible video helpers and `openclaw/plugin-sdk/provider-http` request policy helpers, and adds focused tests for Xiaomi image/video registration and default routing.

## Risks / Open Questions

- The submitted patch should not be merged as-is without a rebase because the Xiaomi plugin now uses `defineSingleProviderPluginEntry`, and media-understanding contract metadata is manifest-driven for discovery/defaults.
- Security review: the PR sends base64 video bytes and the Xiaomi API key through a new remote HTTP path. The reworked provider should use the current `provider-http` policy helpers so base URL overrides, proxy/private-network behavior, dispatcher policy, headers, and timeouts stay aligned with core transport policy.
- The current PR has no focused Xiaomi media-understanding tests; a rework should cover default image routing to `mimo-v2-omni`, video routing to `mimo-v2-omni`, request shape, and provider registration metadata.
