## Summary

Keep this PR open. Current main already covers Xiaomi image understanding through `mimo-v2-omni` in the model catalog plus the generic image-capable config-provider fallback, but it still does not provide Xiaomi video understanding. The PR is stale against the current plugin entrypoint, manifest metadata contract, and provider HTTP helper split, but the remaining video feature is not implemented on main.

## Best Possible Solution

Keep the PR open and ask for a current rebase/rework. The best path is a Xiaomi media-understanding provider in the bundled Xiaomi plugin, declared through `contracts.mediaUnderstandingProviders` and `mediaUnderstandingProviderMetadata`, defaulting image and video to `mimo-v2-omni`, using the shared OpenAI-compatible video helpers plus `provider-http` request policy helpers, and adding focused tests for Xiaomi registration, routing, default model selection, and HTTP request behavior.

## Risks / Open Questions

- Security review: the PR adds a new remote HTTP path that sends base64 video bytes and a Xiaomi API key. A current rework should use `openclaw/plugin-sdk/provider-http` request policy helpers so proxy, private-network, dispatcher, timeout, and header behavior stay aligned with other providers.
- The PR's default model/base URL behavior needs rework: Xiaomi onboarding defaults to `xiaomi/mimo-v2-flash`, while video should default to `mimo-v2-omni`; the provider should not accidentally reuse a text-only active model for video.
- The patch is stale against `defineSingleProviderPluginEntry`, manifest-driven media-understanding metadata, and the current SDK helper layout.
- The PR has no focused Xiaomi media-understanding tests for manifest metadata, registration, image/video model defaults, video request shape, or transport policy.
