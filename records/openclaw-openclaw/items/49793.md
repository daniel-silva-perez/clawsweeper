## Summary

Keep open. Current main has Xiaomi MiMo V2 Omni as an image-capable model and can cover image understanding through the generic config-provider fallback, but it still does not implement the PR's central video-understanding feature for Xiaomi. The contribution is relevant, especially given the #49214 follow-up and the author's Xiaomi API platform context, but it needs a current rework against the manifest-first plugin contract, current entrypoint shape, provider HTTP policy helpers, and focused tests.

## Best Possible Solution

Keep this PR open and ask for a rework on current main. The best implementation is to add a Xiaomi media-understanding provider under `extensions/xiaomi`, declare it through `contracts.mediaUnderstandingProviders` and `mediaUnderstandingProviderMetadata`, default Xiaomi image and video media understanding to `mimo-v2-omni`, use the shared OpenAI-compatible video helpers plus `openclaw/plugin-sdk/provider-http` policy helpers, and add focused Xiaomi registration/default-model/request tests. The existing #49214 Xiaomi model-catalog work should be treated as useful groundwork, not as a replacement for the missing video provider.

## Risks / Open Questions

- Security review: the PR adds a new remote HTTP path that would send base64 video bytes and a Xiaomi API key. A current patch should use `openclaw/plugin-sdk/provider-http` request-policy helpers so proxy, timeout, TLS/private-network, dispatcher, and attribution behavior match other provider media paths.
- The PR currently has no focused Xiaomi media-understanding tests for manifest metadata, registration, default image/video model selection, request shape, or transport-policy forwarding.
- Current main's default Xiaomi model is `mimo-v2-flash`, which is text-only; a rework needs explicit media-understanding defaults to `mimo-v2-omni` rather than reusing the active text model for image/video.
