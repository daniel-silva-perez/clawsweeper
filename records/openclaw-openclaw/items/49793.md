## Summary

Keep this PR open. Current main now covers the Xiaomi image side through the MiMo V2 Omni catalog plus generic image-capable provider fallback, matching the related image-only cleanup, but it still does not implement Xiaomi video media understanding. The PR remains relevant, and the prior discussion correctly points at a rework rather than closure: the patch is stale against the current Xiaomi plugin entrypoint, manifest metadata contract, SDK helper layout, and provider HTTP policy path.

## Best Possible Solution

Keep the PR open and ask for a current rebase/rework. The best path is a Xiaomi media-understanding provider in the bundled Xiaomi plugin, declared through `contracts.mediaUnderstandingProviders` and `mediaUnderstandingProviderMetadata`, defaulting image and video to `mimo-v2-omni`, using the shared OpenAI-compatible video helpers plus `provider-http` request policy helpers, and adding focused tests for Xiaomi image/video registration, routing, and HTTP request behavior.

## Risks / Open Questions

- Security review: the PR adds a new remote HTTP path that sends base64 video bytes and a Xiaomi API key. A rework should use `openclaw/plugin-sdk/provider-http` policy helpers so proxy, private-network, dispatcher, timeout, and header behavior stays aligned with other providers.
- The submitted patch is stale against current main: the Xiaomi plugin entrypoint moved to `defineSingleProviderPluginEntry`, and media-understanding discovery/defaults are manifest-driven through `contracts.mediaUnderstandingProviders` plus `mediaUnderstandingProviderMetadata`.
- The PR has no focused Xiaomi media-understanding tests for registration metadata, default image/video model routing to `mimo-v2-omni`, request shape, or transport policy. Existing review comments also identified base URL/model selection and `reasoning_content` fallback concerns that should be handled in the rework.
