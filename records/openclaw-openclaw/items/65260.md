## Summary

Keep open. Current main has transport, cache, priority, config-preservation, and diagnostic instrumentation for OpenAI/Codex paths, but no source, changelog, test, or related-report evidence proves that the reported steady 12-13s `openai-codex` OAuth server-side latency is fixed. A newer related latency report also points to the same provider family, so this needs maintainer live timing rather than cleanup closure.

## What This Changes

Investigate whether the `openai-codex` OAuth Responses path has a configuration, transport, cache, or provider bottleneck causing consistent 12-13s server-side latency independent of cache hits, model choice, prompt size, and session history.

## Best Possible Solution

Keep the issue open and investigate it with the related OpenAI Codex latency reports. Reproduce on current main or the latest release with supported Codex OAuth models, compare `transport: "auto"`, `"sse"`, and `"websocket"`, compare `/fast` or `serviceTier: "priority"`, and capture model-call time-to-first-byte, total duration, transport, cache counters, request payload size, response bytes, prep-stage timings, and bounded request-id hashes.
