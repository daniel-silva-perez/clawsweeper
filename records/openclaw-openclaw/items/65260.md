## Summary

Keep open. The report is recent, detailed, and still describes a specific `openai-codex` OAuth latency symptom that current `main` does not clearly fix or supersede. Current code has useful transport, cache, priority, and diagnostic controls, but those are investigation tools rather than proof that the steady 12-13s server-side duration is resolved.

## What This Changes

Investigate whether the `openai-codex` OAuth Responses path has a configuration or implementation bottleneck causing consistent 12-13s server-side latency independent of cache hits, model choice, prompt size, and session history.

## Best Possible Solution

Keep the issue open for provider investigation. Reproduce on current main or the latest release with supported Codex OAuth models, compare `transport: "auto"`, `"sse"`, and `"websocket"`, compare `/fast` or `serviceTier: "priority"`, and capture model-call time-to-first-byte, total duration, transport, cache counters, and bounded request-id hashes.
