## Summary

Keep this PR open for maintainer review. Current main still has no repotype config or equivalent repo-structure gate, and the author has already addressed several earlier glob issues, but the proposed config is not merge-ready because it would fail against the current tree and still needs an explicit tooling/supply-chain decision before becoming an OpenClaw gate.

## What This Changes

The PR adds a root `repotype.yaml` and `schemas/skill.frontmatter.schema.json` to validate repository structure, including skill frontmatter, package files, plugin manifests, docs, scripts, and other repo paths in permissive unmatched-file mode.

## Best Possible Solution

Keep the PR open and require a refresh before merge: make the proposed rules pass the current tree, add or deliberately exempt missing skill frontmatter, cover plugin-packaged skills if the rule claims all skills, and decide whether repotype should be a pinned dependency, package script, and CI lane.

## Reproduction Assessment

Yes. A focused static reproduction is enough: the PR schema requires `name` and `description` for `skills/*/SKILL.md`, while current main has `skills/canvas/SKILL.md` with no frontmatter, so validation would fail once this config is used.

## Solution Assessment

No. The direction may be useful, but the current patch is not the best mergeable solution because it introduces a gate that does not pass OpenClaw's existing skill tree and leaves dependency/CI adoption as an unresolved maintainer decision.
