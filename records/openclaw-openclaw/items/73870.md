## Summary

Keep open. Current main still has a global `subagent` lane plus a per-session active-child reject guard, but it does not implement the requested per-parent queued concurrency throttle; related open issues #8061 and #10467 are adjacent but track per-target-agent and named-lane controls rather than this exact parent-keyed queueing behavior.

## What This Changes

The issue requests a configurable per-parent subagent spawn concurrency cap, with excess spawns queued instead of letting one parent saturate the shared subagent lane or hit a reject-only child cap.

## Best Possible Solution

Add a documented per-requester spawn concurrency throttle distinct from the global `subagent` lane and the reject-only `maxChildrenPerAgent` cap. Queued native and ACP child runs should remain visible in task/subagent state, start when that parent has capacity, preserve unrelated-parent fairness, and define restart/cancel behavior explicitly.

## Reproduction Assessment

Yes for the missing product behavior: current source and tests show `maxConcurrent` is global, not parent-keyed, and `maxChildrenPerAgent` rejects instead of queueing. The reported large-fleet Slack timeout symptom was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but the default, fairness rules, task-state representation, ACP/native parity, and restart/cancel semantics need maintainer design before this is a safe narrow implementation.
