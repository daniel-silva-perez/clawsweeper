## Summary

Keep open. Current main still has a single global subagent execution lane and a per-session active-child guard that rejects excess spawns; it does not implement the requested queued per-parent concurrency throttle. Adjacent open issues #8061 and #10467 cover per-target and named-lane concurrency, but not this parent-keyed queueing behavior.

## What This Changes

Add a configurable per-parent concurrency cap for subagent spawns so multi-wave fanout queues past the parent limit instead of saturating gateway/WebSocket work.

## Best Possible Solution

Design a parent/requester-keyed spawn throttle that is separate from the global subagent lane and the reject-only active-child cap. Native and ACP child runs should queue visibly, preserve unrelated-parent fairness, keep sessions_spawn non-blocking, and define cancellation, stop, restart, and task-registry behavior explicitly.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show a global subagent lane plus reject-only per-session child cap, with no queued per-parent throttle. The production Slack timeout symptom itself was not reproduced live in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but the safest implementation needs maintainer design around defaults, fairness, queue observability, and lifecycle semantics before it becomes a narrow repair.
