## Summary

Keep open. Current main still has a global `subagent` execution lane plus a per-session active-child reject guard, but it does not implement the requested queued per-parent spawn throttle; adjacent open issues #8061 and #10467 cover per-target and named-lane concurrency, not this exact parent-keyed queueing behavior.

## What This Changes

The issue requests a configurable per-parent concurrency cap for subagent spawns, with excess children queued instead of overloading the gateway or being rejected.

## Best Possible Solution

Add a documented parent/requester-keyed spawn throttle that is distinct from the global `subagent` lane and the reject-only child-count cap. Queued native and ACP child runs should be visible in task/subagent state, start later when that parent has capacity, preserve unrelated-parent fairness, and define restart, stop, and cancellation behavior explicitly.

## Reproduction Assessment

Yes for the missing product behavior: source and tests on current main show `maxConcurrent` is global, not parent-keyed, and `maxChildrenPerAgent` rejects instead of queueing. The reported large-fleet Slack timeout symptom was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle appears aligned with the report, but the exact defaults, fairness rules, task-state representation, and restart/cancel semantics need maintainer design before this is a safe narrow repair.
