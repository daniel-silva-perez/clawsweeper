## Summary

Keep open. Current main still has a global subagent command-lane cap plus a per-session active-child reject guard, but it does not implement the requested per-parent queued spawn throttle with a lower default such as 2-3.

## What This Changes

Request a configurable per-parent concurrency limit for subagent spawns so excess child work from one requester queues instead of saturating the gateway event loop.

## Best Possible Solution

Add an explicit per-requester/session spawn throttle that queues excess native and ACP subagent starts behind each parent while preserving the global subagent lane as the process-wide safety valve. The implementation should document the new config, prove unrelated parents can still make progress, and keep maxChildrenPerAgent as a separate hard cap if it remains useful.

## Reproduction Assessment

Yes for the missing behavior: current source and tests show maxConcurrent is global and not a per-parent gate, and maxChildrenPerAgent rejects rather than queues. No live Slack CPU reproduction was run under the read-only review constraint.

## Solution Assessment

Unclear until maintainers choose the config contract and default, but the current global lane plus reject guard is not the best solution for this report. A dedicated per-parent queued throttle is the maintainable direction if the product wants this behavior.
