## Summary

Keep open. Current main has a global subagent command lane and a per-session active-child reject guard, but it does not implement the requested per-parent queued spawn throttle with a default around 2-3; excess children are either admitted to the global lane or rejected at `maxChildrenPerAgent`.

## What This Changes

Request a configurable per-parent concurrency cap for subagent spawns so excess child runs queue instead of one parent saturating gateway/WebSocket work.

## Best Possible Solution

Add a documented per-requester spawn concurrency throttle distinct from the existing global lane and reject-only `maxChildrenPerAgent` cap. Excess native and ACP child runs should be represented in task/subagent state, start later when that parent has capacity, and continue to let unrelated parents use the global `subagent` lane fairly.

## Reproduction Assessment

Yes for the missing product behavior: current source and `src/agents/subagent-spawn.depth-limits.test.ts` show `maxConcurrent` is not a per-parent gate and `maxChildrenPerAgent` rejects instead of queueing. No live Slack fleet reproduction was available in this read-only pass.

## Solution Assessment

No. The current global lane plus active-child reject guard reduces total fanout pressure, but it is not the narrow queued per-parent throttle requested here and can still either over-admit one parent up to the global lane or reject work instead of deferring it.
