## Summary

Keep open. Current main still has a single global `subagent` execution lane plus a reject-only per-session active-child guard; it does not implement the requested queued per-parent spawn throttle. Related open items #8061 and #10467 cover per-target and named-lane concurrency, but neither subsumes parent-keyed queueing for burst fanout.

## What This Changes

The issue requests a configurable per-parent concurrency cap for subagent spawns, with excess child spawns queued to reduce gateway event-loop and WebSocket contention during large fanout.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that is distinct from the global subagent lane and the active-child reject cap. Queued native and ACP child runs should be visible in task/subagent state, start when that parent has capacity, preserve fairness for unrelated parents, and define stop, restart, cleanup, and completion behavior.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane concurrency plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but scheduler defaults, fairness, task-state representation, and lifecycle semantics are broad enough to need maintainer design before a narrow automated repair is safe.
