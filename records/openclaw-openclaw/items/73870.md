## Summary

Keep open. Current main has a global `subagent` command-lane cap and a per-session `maxChildrenPerAgent` reject guard, and it also queues some announce delivery while the requester session is active. It does not implement the requested per-parent queued spawn throttle with a default around 2-3; a current test explicitly asserts that `agents.defaults.subagents.maxConcurrent` is not used as a per-parent spawn gate.

## Best Possible Solution

Keep this issue open and design a real per-parent spawn throttle instead of closing it as covered by the existing global lane. The likely implementation belongs around `src/agents/subagent-spawn.ts` and the subagent registry/task tracking: add a configurable per-parent queued limit, probably separate from the existing reject-only `maxChildrenPerAgent`, apply it consistently to native and ACP subagent spawns, preserve the non-blocking `sessions_spawn` contract, and add docs/schema/tests that prove excess children from one requester queue while unrelated parents can still use the global lane fairly.

## Risks / Open Questions

- The existing global `subagent` lane cap may reduce total pressure, but it does not provide per-parent fairness or the requested 2-3 child limit for a single orchestrator.
- The active-requester announce queue may already cover part of the completion-delivery contention path; a fix should confirm whether the failing hot spot is child start, completion announce, or both.
- A per-parent queue needs to compose carefully with the existing session lane and global subagent lane to avoid deadlocks, starvation, or breaking `sessions_spawn`'s non-blocking accepted result contract.
- No live 30-agent Slack reproduction was run because this review was constrained to read-only inspection; the keep-open decision is based on current source, docs, tests, and related cleanup evidence.
