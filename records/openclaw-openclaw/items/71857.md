## Summary

Keep open. Current main still rejects owner-prefixed ClawHub skill slugs, and the PR’s parser/test approach is focused, but the submitted diff still puts its user-facing changelog entry under an already released section instead of Unreleased, so it should not merge as-is.

## What This Changes

The PR adds a ClawHub skill slug parser that accepts requested `owner/slug` inputs, canonicalizes them to the bare slug for install/update registry and storage paths, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Move the changelog entry to the current Unreleased/Fixes section after rebasing, then land the narrow parser and regression tests if maintainers want OpenClaw to tolerate ClawHub URL-style `owner/slug` inputs while continuing to fetch, install, and track by canonical bare slug.

## Reproduction Assessment

Yes. On current main, `openclaw skills install spclaudehome/skill-vetter` deterministically reaches `validateRequestedSlug`, which calls `normalizeTrackedSlug` and throws on `/` before any network call.

## Solution Assessment

No, not as submitted: the code/test approach is the narrow maintainable fix, but the changelog placement would falsely claim the fix shipped in an already released version. The safer path is to move that entry to Unreleased and then review/land the parser change.
