## Summary

## Repair Work Prompt

Create a focused replacement for the stale Telegram forum command-menu PR on current main. Implement under `extensions/telegram` only; do not edit old `src/telegram` paths and do not move behavior into core.

Observable bug: Telegram forum/group slash command menus can show commands as unavailable because OpenClaw registers native commands only in the default Bot API scope. Current main also clears only default-scope commands when native commands are disabled, and the plugin-command `Command not found.` fallback sends without topic thread params.

Expected fix boundary:
- In `extensions/telegram/src/bot-native-command-menu.ts`, register the same capped command list for both default scope and `{ scope: { type: "all_group_chats" } }`.
- Clear/delete both default and `all_group_chats` scopes, including the explicit native-disabled cleanup path in `extensions/telegram/src/bot-native-commands.ts`.
- Preserve existing process-local command hash cache, bot identity/account isolation, delete-before-set ordering, and `BOT_COMMANDS_TOO_MUCH` retry behavior. Make the cache scope-aware if needed so default and group scopes do not suppress each other.
- In the plugin-command mismatch path, send `Command not found.` with topic params from `resolveTelegramThreadSpec` / `buildTelegramThreadParams`, preserving DM topics and forum topics while omitting General topic id 1.
- Preserve current command authorization behavior; menu registration must not allow unauthorized command execution.
- Add focused regression tests for dual registration, dual cleanup, cache/retry behavior across scopes, plugin not-found thread params, and General-topic id 1 omission.

Related refs: this PR, #74032, #6833, #19213. Changelog: add a user-facing Telegram fix entry only if this becomes a product-facing PR; credit the reporter/contributor, not bots.

Validation to run: targeted Telegram tests and formatter first, then `pnpm check:changed` through the normal maintainer/Testbox path if broad lanes are selected.
