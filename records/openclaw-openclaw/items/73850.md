## Summary

Keep this open. Current main still only has provider-side interruption handling; the Google Meet Chrome command-pair and paired-node paths continue forwarding shared input without a local human barge-in side channel or loopback suppression, and #73834 remains the linked open draft implementation candidate for this behavior.

## What This Changes

The issue requests Google Meet realtime voice support that lets human speech interrupt assistant playback, avoids treating assistant loopback as user speech, and preserves existing Chrome command-pair and external bridge behavior.

## Best Possible Solution

Keep this tracked until #73834 or a reshaped PR lands an additive realtime voice interruption hook plus Google Meet-owned optional side-channel human speech detection, loopback suppression during assistant playback, local output-buffer clearing, config/docs/tests, and representative command-pair validation.

## Reproduction Assessment

No high-confidence live reproduction was run. The source-level check is still strong: current main forwards shared command-pair input while assistant output may be active and lacks the requested side-channel barge-in path.

## Solution Assessment

Unclear until the draft PR is reviewed. The proposed additive provider hook plus Google Meet-owned optional side-channel command fits the extension boundary, but thresholds, trust model, live behavior, and security implications need maintainer review.
