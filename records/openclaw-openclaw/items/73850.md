## Summary

Keep this issue open. Current main still lacks transport-owned human barge-in support for Google Meet Chrome command-pair realtime audio, and the linked draft PR remains open, draft, unmerged, and dirty, so this is still active maintainer review work rather than conservative cleanup.

## What This Changes

Request Google Meet realtime voice behavior that lets human speech interrupt assistant playback quickly, avoids treating shared BlackHole loopback as user speech, and preserves existing Chrome command-pair and external bridge behavior.

## Best Possible Solution

Review and either land or reshape the linked draft PR so the final design adds a backwards-compatible realtime interruption hook plus Google Meet-owned optional side-channel barge-in detection, loopback suppression during playback, local output-buffer clearing, config/docs/tests, and live or strongly representative command-pair validation.

## Reproduction Assessment

Unclear: the issue describes a plausible Chrome command-pair plus shared BlackHole reproduction path and current source matches that path, but this review did not run a live Google Meet/BlackHole session. Source inspection is enough to keep the issue open, not enough to claim live reproduction proof.

## Solution Assessment

Unclear pending maintainer architecture review. The proposed additive SDK hook plus optional Google Meet side-channel command is a plausible narrow direction, but the API name, local command trust model, capture thresholds, and external-bridge boundaries should be reviewed before implementation lands.
