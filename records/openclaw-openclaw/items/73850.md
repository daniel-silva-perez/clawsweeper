## Summary

Keep this issue open. Current main still lacks a Google Meet-owned local barge-in path: the Chrome command-pair and chrome-node paths continue forwarding shared input into the realtime provider, while the only barge-in handling found is OpenAI provider-internal and not exposed as a transport hook. The linked draft PR remains the active implementation candidate rather than a reason to close this issue.

## What This Changes

The issue requests Google Meet realtime voice support for human barge-in during assistant playback, including optional side-channel human speech detection, shared-loopback suppression, local output-buffer clearing, and an additive realtime voice interruption hook.

## Best Possible Solution

Keep this tracked until #73834 or a reshaped PR lands an additive realtime voice interruption hook plus Google Meet-owned optional side-channel human speech detection, loopback suppression during assistant playback, local output-buffer clearing, config/docs/tests, and representative command-pair validation.

## Reproduction Assessment

No high-confidence live reproduction was run. The source-level path is clear: with Chrome command-pair audio using a shared BlackHole device, current main forwards shared input to the provider while assistant output may be active and has no local side-channel barge-in path.

## Solution Assessment

Unclear until the draft PR is reviewed. The proposed additive provider hook plus Google Meet-owned optional side-channel command fits the plugin boundary, but thresholds, live behavior, and local-command trust/security details need maintainer review.
