## Summary

Keep this issue open. Current main still lacks the requested Google Meet human barge-in behavior: the Chrome command-pair and paired-node paths forward shared input while playback may be active, the realtime voice bridge has no transport-facing interruption hook, and only the OpenAI provider has an internal speech-start barge-in handler. The linked draft PR #73834 remains the open implementation candidate rather than a reason to close this issue.

## What This Changes

Request Google Meet realtime voice support for human-first barge-in during assistant playback, including optional side-channel human speech detection, loopback suppression, provider interruption, and local output-buffer clearing.

## Best Possible Solution

Keep this issue tracked until #73834 or a reshaped PR lands an additive realtime voice interruption hook plus Google Meet-owned optional side-channel human speech detection, loopback suppression during assistant playback, local output-buffer clearing, config/docs/tests, and representative command-pair validation.

## Reproduction Assessment

No; I do not have a high-confidence live reproduction because no Google Meet plus shared BlackHole session was run. The static source path is strong evidence for the missing behavior: current main forwards shared input during playback and lacks the requested side-channel barge-in path.

## Solution Assessment

Unclear until the draft PR is reviewed. The proposed additive provider hook plus optional Google Meet side-channel command fits the plugin boundary, but thresholds, command trust, live behavior, and provider semantics need maintainer and security review.
