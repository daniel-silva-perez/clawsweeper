## Summary

Keep open. Current main has useful mitigations for this failure mode, including global main/subagent lane caps, per-session subagent child limits, heartbeat busy-session skips, opt-in subagent/nested lane deferral, and liveness diagnostics. It still does not fully close the reported multi-agent heartbeat fanout problem because subagent/nested lane deferral is opt-in and I did not find a heartbeat-origin cap or serialization path at subagent spawn time.

## What This Changes

The issue requests Gateway and agent-scheduling changes so simultaneous multi-agent heartbeat runs with subagent fanout do not saturate the Gateway or delay channel, probe, and heartbeat queues.

## Best Possible Solution

Add a heartbeat-origin resource policy in core scheduling: before starting heartbeat turns, account for shared subagent and nested lane backlog by default or through a clearly documented safe mode; when heartbeat turns spawn subagents, cap or serialize that fanout so channel responsiveness and probe latency remain bounded. Back it with a focused regression test that simulates several heartbeat-enabled agents spawning background subagents and verifies deferred heartbeats or bounded queue growth.
