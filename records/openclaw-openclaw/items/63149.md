## Summary

Keep this issue open. Current main has useful generic mitigations, including configurable main/subagent lanes, per-session child caps, heartbeat busy-session skips, and liveness diagnostics, but I did not find a shipped heartbeat-origin admission policy that accounts for shared subagent backlog before starting heartbeat turns. The reported multi-agent heartbeat/subagent fanout path remains plausible and still needs a focused fix or reproduction-backed regression test.

## Best Possible Solution

Keep this issue open as a core Gateway and agent-scheduling performance bug. The best maintainer path is to add or validate heartbeat-origin resource policy: check shared subagent-lane active/queued depth before starting heartbeat turns, cap or serialize subagent spawns initiated by heartbeat turns, expose lane/backlog/drain diagnostics that identify heartbeat-origin work, and add a focused stress/regression test for multiple heartbeat-enabled agents spawning background subagents while channel responsiveness stays bounded.
