## Summary

Keep this issue open. Current main has generic mitigations for lane concurrency, busy-session heartbeat skips, per-session child caps, and liveness diagnostics, but I still do not see a heartbeat-origin admission policy that accounts for shared subagent-lane backlog before starting heartbeat turns. The reported multi-agent heartbeat fanout under load remains plausible and is not clearly implemented, duplicated, or stale.

## Best Possible Solution

Keep this issue open as a core Gateway and agent-scheduling performance bug. The best path is to add or validate heartbeat-origin resource policy: check shared subagent-lane active/queued depth before starting heartbeat turns, cap or serialize subagent spawns initiated by heartbeat turns, expose diagnostics that identify heartbeat-origin queued/running work, and add a focused stress/regression test for multiple heartbeat-enabled agents spawning background subagents while channel responsiveness stays bounded.
