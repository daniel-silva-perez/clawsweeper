## Summary

Keep this PR open. Current main still has no Nacos-backed config source, so the PR is not obsolete, but the branch is a broad core gateway/config change with blocking runtime semantics and a remote-config trust boundary that needs maintainer review before merge.

## What This Changes

This PR adds an opt-in Nacos-backed gateway config source with file/Nacos adapters, env-based source resolution, long-poll reload subscription, config IO/CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic config-source seam where file remains the default and Nacos is an explicit, validated adapter used consistently by startup metadata reads, runtime reads, writes, reload notifications, docs, and security guardrails. If this should live outside core, define the missing pre-start config-source plugin API first and move the Nacos adapter to plugin space.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route startup metadata through the active source:** `src/gateway/server-startup-config.ts:205-206`
  - body: `loadGatewayStartupConfigSnapshot()` sets the Nacos source, but immediately calls `readConfigFileSnapshotWithPluginMetadata()`, whose top-level helper still creates file-backed ConfigIO. A Nacos-only gateway can therefore boot from local/default config while the reloader subscribes to Nacos, so startup reads need to use the resolved source.
  - confidence: 0.91
- **[P2] Notify subscribe reloaders after Nacos writes:** `src/config/io.ts:2501`
  - body: The Nacos branch updates runtime snapshot/refresher state and returns, but subscribe-mode reloaders skip `subscribeToWrites`. Successful `config.apply`/`config.patch` calls can update `config.get` without applying hot-reloadable services until an external Nacos edit or restart.
  - confidence: 0.87
- **[P3] Describe standalone CLI Nacos writes accurately:** `docs/gateway/configuration.md:744`
  - body: The docs say CLI `config set` updates the in-memory snapshot in Nacos mode, but standalone CLI now initializes the Nacos source and `writeConfigFile()` throws when there is no runtime snapshot or refresh handler. Users get an error instead of a process-local update.
  - confidence: 0.84
