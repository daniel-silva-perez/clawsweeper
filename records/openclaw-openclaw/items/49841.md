## Summary

Keep this PR open. Current main still has no Nacos config-source surface, and the proposed branch is not safe to merge yet because startup config reads can still bypass the active source and Nacos in-process writes do not notify the gateway reload path. The remote-config trust boundary also needs explicit maintainer security/product review before this belongs in core.

## What This Changes

The PR adds an opt-in Nacos-backed gateway config source with env-based source resolution, file/Nacos adapters, long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic source seam with file as the default and Nacos as an explicit, validated adapter that routes every startup/read/write/reload path through the active source, preserves source/runtime snapshots, emits correct reload notifications, documents persistence limits and trust requirements, and has real Nacos validation. If this belongs outside core, define the needed pre-start config-source plugin API first and move the Nacos adapter to plugin space.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read startup config through the active source:** `src/gateway/server-startup-config.ts:202-206`
  - body: `loadGatewayStartupConfigSnapshot()` resolves the Nacos source, but the next read still goes through `readConfigFileSnapshotWithPluginMetadata()`, which delegates directly to `createConfigIO()` and reads the local file. In Nacos mode the server can start from local/default config while the CLI preflight and reloader are pointed at Nacos, so the gateway runs a different config than the configured remote source.
  - confidence: 0.91
- **[P2] Notify reloaders after Nacos writes:** `src/config/io.ts:2481-2501`
  - body: The Nacos branch updates the runtime snapshot and returns without emitting a config write notification, and subscribe mode never registers `subscribeToWrites`. Successful `config.set`/`config.patch` calls can update `config.get` while hot-reloadable gateway services keep running the old config until a remote Nacos edit or restart.
  - confidence: 0.87
