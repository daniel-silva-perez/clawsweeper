## Summary

Keep open. Current main still has only file-backed config loading/reload behavior, and the PR head is not ready to merge because the Nacos source is not used for the gateway startup snapshot, Nacos-mode in-process writes do not drive the reload planner, and the new remote config trust boundary needs maintainer security review.

## What This Changes

The PR branch adds an opt-in Nacos-backed gateway config source with file/Nacos adapters, env-based source resolution, Nacos long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic source seam where file remains the default and Nacos is an explicit, validated adapter used consistently by startup, plugin metadata reads, preflight, reads, writes, and reloads. If Nacos should live outside core, define the missing pre-start config-source plugin/API seam first and move the adapter to plugin space.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Read startup snapshots through the active source:** `src/gateway/server-startup-config.ts:205-206`
  - body: `loadGatewayStartupConfigSnapshot()` initializes the Nacos source, but then calls `readConfigFileSnapshotWithPluginMetadata()`, which still delegates straight to `createConfigIO()` in this branch. A Nacos-only gateway can therefore start from local/default config while the later reloader is pointed at Nacos. Make the plugin-metadata snapshot path source-aware before enabling this mode.
  - confidence: 0.91
- **[P2] Notify the reloader for Nacos writes:** `src/config/io.ts:2481-2501`
  - body: The Nacos branch of `writeConfigFile()` updates the runtime snapshot and returns without emitting a runtime write notification. Since subscribe-mode reloaders do not register the file write path, successful `config.set`/`config.patch` calls can update `config.get` while hooks, cron, channel state, or other hot-reloadable services keep running the old config until an external Nacos edit or restart.
  - confidence: 0.88
- **[P3] Document standalone CLI writes accurately:** `docs/gateway/configuration.md:744`
  - body: The docs say CLI `config set` updates the in-memory snapshot in Nacos mode, but the proposed `writeConfigFile()` throws in a standalone process with no runtime snapshot or refresh handler. Update the docs and CLI wording, or route CLI writes through a running gateway session.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces opt-in remote HTTP loading of gateway configuration, so the new trust boundary needs maintainer security review before merge.

Concerns:

- **[medium] Review remote config auth and transport guardrails:** `src/config/sources/nacos-client.ts:52`
  - body: `createNacosConfigClient()` fetches gateway config and posts listener requests to an operator-provided Nacos URL, while the docs show plain HTTP examples and the client has no Nacos auth parameters, TLS requirement, or documented ACL expectations.
