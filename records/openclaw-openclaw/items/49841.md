## Summary

Keep this PR open. Current main does not already include Nacos config-source support, and the PR is still an open, dirty, failing-check core gateway/config change with blocking correctness issues in startup source routing and Nacos write/reload semantics plus an unsettled remote-config security model.

## What This Changes

The PR adds an optional Nacos-backed gateway config source with env-based source resolution, file/Nacos adapters, long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a consistent generic config-source seam where file remains default and Nacos is an explicit adapter used by startup metadata reads, runtime reads, writes, reload notifications, docs, and security guardrails. If this belongs outside core, define the missing pre-start config-source plugin API first and move the Nacos adapter to plugin space.

## Reproduction Assessment

Yes. Diff inspection gives high-confidence focused reproductions: enable OPENCLAW_CONFIG_SOURCE=nacos and startup still reaches the file-backed metadata helper, and drive config.apply/config.patch in Nacos mode to see runtime snapshot change without a subscribe-mode write notification to the reloader. A live Nacos smoke would still be needed before shipping.

## Solution Assessment

No. The proposed direction may be product-worthy, but this implementation is not the best mergeable form because it applies the source seam inconsistently and introduces remote-config security semantics before the startup, write/reload, docs, and guardrail behavior is settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the active source for startup snapshot reads:** `src/gateway/server-startup-config.ts:205-206`
  - body: `loadGatewayStartupConfigSnapshot()` resolves the Nacos source, but then reads `readConfigFileSnapshotWithPluginMetadata()`, which still delegates to file-backed `createConfigIO()`. A Nacos-only gateway can therefore boot from local/default config while the reloader subscribes to Nacos, so startup metadata reads need to go through the active source.
  - confidence: 0.92
- **[P2] Notify reloaders after Nacos config writes:** `src/config/io.ts:2466-2501`
  - body: In Nacos mode this branch returns after updating the runtime snapshot/refresh path, but subscribe-mode reloaders do not register the existing write-notification path. Successful `config.apply` or `config.patch` calls can update `config.get` while hot-reloadable services remain on the old config until an external Nacos edit or restart.
  - confidence: 0.88
- **[P3] Fix the Nacos CLI write documentation:** `docs/gateway/configuration.md:744`
  - body: The docs say standalone CLI `config set` updates the in-memory snapshot in Nacos mode, but the PR initializes the Nacos source in standalone CLI processes and `writeConfigFile()` throws when there is no running runtime snapshot or refresh handler. Users get an error instead of a process-local update.
  - confidence: 0.84
- **[P3] Credit the changelog entry:** `CHANGELOG.md:18`
  - body: The added changelog bullet is user-facing but lacks the required `Thanks @...` attribution used by the active changelog section. Add the contributor credit before this can pass the repo’s changelog policy review.
  - confidence: 0.9
