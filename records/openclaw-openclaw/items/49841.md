## Summary

Keep open. Current main still has no Nacos config source, and this open PR remains a broad gateway/config feature with a dirty branch, a blocking write-to-reloader correctness issue, and unresolved remote-config security guardrails before it can merge.

## What This Changes

The PR adds an optional Nacos-backed gateway config source with env-based source resolution, file/Nacos adapters, Nacos HTTP long-polling, subscribe-mode reload wiring, CLI/doctor integration, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic source seam where file remains default and Nacos is an explicit, validated adapter that preserves current read/write/reload semantics, emits correct runtime reload notifications, documents persistence limits accurately, includes changelog credit, and has a reviewed security model. If this belongs outside core, define the needed pre-start config-source plugin API first and move the Nacos adapter to plugin space.

## Reproduction Assessment

Yes. Diff inspection gives a focused reproduction: enable Nacos source, call a gateway config write that changes a hot-reloadable setting, and observe that the Nacos write branch updates runtime state without the subscribe-mode reloader receiving an in-process write event. A live Nacos smoke is still needed before shipping.

## Solution Assessment

No. The proposed direction may be useful, but this branch is not the best mergeable form yet because it applies the config-source seam inconsistently across write/reload behavior and introduces remote-config security semantics before guardrails are settled.
