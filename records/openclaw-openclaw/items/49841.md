## Summary

Keep this PR open for maintainer review. Current main still has only file-backed config loading, while this branch adds a broad Nacos remote-config path and is currently dirty against main; the latest patch also still has a blocking Nacos in-process write/reload gap that should be fixed before merge.

## What This Changes

The PR adds an optional Nacos-backed gateway config source with env-based source resolution, file/Nacos adapters, long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic source seam with file as the default and Nacos as an explicit, validated adapter that preserves current config semantics, emits correct runtime reload notifications, handles Nacos auth/error/retry behavior, documents CLI/write persistence limits, and has real Nacos validation. If this belongs outside core, define the needed pre-start config-source plugin API first and move the Nacos adapter to a plugin path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Notify the reloader for Nacos in-process writes:** `src/config/io.ts:2466-2481`
  - body: In Nacos mode this branch returns from `writeConfigFile()` after updating the runtime snapshot/secret refresh path, but there is no file watcher and subscribe mode does not register the existing `subscribeToWrites` path. As a result, successful `config.set`/`config.patch` calls can update `config.get` while leaving hot-reloadable gateway services such as hooks, cron, or channel state unchanged until a remote Nacos edit or restart occurs. Emit a write notification for the virtual Nacos snapshot and let the reloader consume it, or explicitly route these writes through the same reload planner.
  - confidence: 0.87
