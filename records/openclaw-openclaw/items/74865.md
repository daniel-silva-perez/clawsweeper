## Summary

Keep open. Current main still has a narrow, source-backed regression candidate: the native subagent spawn path resolves the active context engine before ensuring the built-in legacy engine is registered, while the adjacent subagent cleanup path already performs that initialization first.

## What This Changes

Restore `/subagents spawn` so default legacy context-engine registration happens before subagent spawn preparation resolves an engine.

## Best Possible Solution

Wire built-in context-engine initialization into the spawn-side context-engine resolution boundary and add a regression test that isolates or clears the registry before a default legacy `spawnSubagentDirect` or `/subagents spawn` path succeeds.

## Reproduction Assessment

Yes, at source level: a default `/subagents spawn` reaches `spawnSubagentDirect`, then `prepareContextEngineSubagentSpawn`, then `resolveContextEngine` without first registering `legacy`. The registry implementation has an explicit default-engine `not registered` throw path, and current spawn tests mock the resolver instead of exercising that cold registry case.

## Solution Assessment

Yes. The narrowest maintainable fix is to initialize built-in context engines before the spawn path resolves the active engine, mirroring `src/agents/subagent-registry.ts`; changing the global registry resolver to auto-initialize would broaden side effects beyond this regression.
