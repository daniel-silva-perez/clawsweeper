## Summary

Keep open. Current main still lacks an original-aware UI filter for stale redaction sentinels, while gateway config writes still reject `__OPENCLAW_REDACTED__` when the original config has no matching value. The focused fix in #60971 was closed unmerged, so the reported dashboard save failure remains actionable.

## What This Changes

The issue asks the Control UI to avoid submitting `__OPENCLAW_REDACTED__` as literal config data when saving unrelated settings, especially for stale `gateway.remote.token` placeholders.

## Best Possible Solution

Add a UI-side, original-aware sanitization step before form-mode `config.set` and `config.apply`: drop redaction sentinels for sensitive paths absent from the original raw/source config, preserve placeholders that correspond to real existing secrets so the gateway can restore them, and keep the backend rejection of unrestorable sentinels intact.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to seed `configForm.gateway.remote.token` with `__OPENCLAW_REDACTED__` while the original config lacks that path, then save in form mode: current UI serialization includes the sentinel and the gateway restore path rejects it as unrestorable.

## Solution Assessment

Yes. The narrowest maintainable fix is the UI-side original-aware filter described by #60971, because globally relaxing `restoreRedactedValues` would risk accepting reserved sentinel strings as real config data.
