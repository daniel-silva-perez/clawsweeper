## Summary

Keep open. Current main and the latest release still preserve the route-relative empty-basePath Control UI logo fallback, the sidebar uses that helper directly, and an open unmerged PR already claims the closing implementation for this regression.

## What This Changes

The issue reports a 2026.3.24 regression where the Control UI sidebar logo and browser favicon render blank even though the favicon asset is present.

## Best Possible Solution

Keep this issue open until PR #56357 or an equivalent current-main port lands. The fix should make empty-basePath logo/favicon URLs absolute, preserve mounted `gateway.controlUi.basePath` behavior, handle static SVG CSP safely, and update focused regression tests.

## Reproduction Assessment

Yes. Current source gives a high-confidence reproduction path: default empty `basePath` makes the sidebar request route-relative `favicon.svg`, and the Gateway 404s route-local static asset misses; the existing unit test also asserts that old behavior.

## Solution Assessment

Yes, the linked fix direction is the narrow maintainable path: normalize the empty-basePath asset URL to `/favicon.svg`, keep base-mounted URLs under the mount, and scope static-asset CSP only as needed. The open PR still needs review/update against current main before this issue should close.
