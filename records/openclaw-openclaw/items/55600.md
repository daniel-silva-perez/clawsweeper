## Summary

Keep open. Current main still returns a route-relative `favicon.svg` for the empty Control UI base path, the sidebar header still uses that helper for the brand logo, and the latest shipped release has the same behavior. An open unmerged PR already uses closing syntax for this regression, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that after updating to OpenClaw 2026.3.24, the default Control UI header logo and browser favicon render as missing even though `/favicon.svg` is accessible.

## Best Possible Solution

Land PR #56357 or an equivalent current-main fix that makes empty-basePath logo URLs root-absolute, preserves configured `gateway.controlUi.basePath` mounting, and updates the focused UI/Gateway regression tests before closing this issue.

## Reproduction Assessment

Yes. Current source gives a high-confidence source-level reproduction: the default empty base path produces `favicon.svg`, the sidebar renders that value, and route-local static asset misses such as `/webchat/favicon.svg` return 404.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize the empty-basePath asset URL to `/favicon.svg` while keeping non-empty base paths mounted under their configured prefix and covering both cases with focused tests.
