## Summary

Keep open. Current main still has no trusted-proxy default/fallback identity: trusted-proxy auth reads the configured identity header and fails with `trusted_proxy_user_missing` when it is absent. The related discussion distinguishes this Cloudflare/WebSocket missing-header case from the closed local-direct password fallback work, and the requested behavior changes a security-sensitive auth boundary that needs maintainer approval.

## What This Changes

Add an explicit trusted-proxy fallback/default user option for trusted proxy requests where the configured identity header is absent, especially Cloudflare Zero Trust WebSocket upgrade paths.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, add an explicit trusted-proxy fallback identity option across exported config types, zod/generated schemas, docs, security audit guidance, and auth/control-ui tests, with precise rules for trusted proxy source checks, required headers, loopback policy, browser origin policy, and whether `allowUsers` must include the fallback identity. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers on WebSocket upgrades.

## Risks / Open Questions

- A fallback identity would change the trusted-proxy trust boundary: any request from a configured trusted proxy that lacks an identity header could be admitted as the fallback user unless semantics are very tight.
- The reporter's Cloudflare Tunnel setup uses a loopback proxy source; current main also requires explicit `gateway.auth.trustedProxy.allowLoopback = true`, so a default-user fix would need to define how it interacts with loopback proxy policy.
- This read-only review did not run a live Cloudflare Tunnel reproduction, but the missing OpenClaw config/auth/docs/test support is clear from current main.
