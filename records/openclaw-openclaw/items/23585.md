## Summary

Keep open. Current main still fails closed when the trusted-proxy identity header is absent, the config/schema contract has no `defaultUser` or `fallbackUser` field, and the existing docs/tests describe a required identity-header trust boundary. The related discussion also distinguishes this Cloudflare WebSocket missing-header case from the closed local-direct password fallback work, so this remains a maintainer/security decision rather than cleanup closure.

## What This Changes

The issue requests a trusted-proxy `defaultUser` or `fallbackUser` option so requests from a configured trusted proxy can authenticate when the configured identity header is absent.

## Best Possible Solution

Keep the issue open until maintainers decide the trusted-proxy contract. If accepted, add an explicit fallback identity option across exported config types, zod and generated schemas, docs, security audit guidance, and gateway/control-ui auth tests with precise fail-closed rules. If rejected, document Cloudflare Zero Trust IP-whitelist WebSocket paths without forwarded identity as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers.
