## Summary

Keep open. Current main still has no trusted-proxy fallback/default user: missing or blank identity headers return `trusted_proxy_user_missing`, the config/schema/docs expose only the existing trusted-proxy fields, and tests assert the fail-closed behavior. Related discussion also separates this from the closed local-direct fallback work, so this needs maintainer/security judgment rather than cleanup closure.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, implement an explicit trusted-proxy fallback identity option across `GatewayTrustedProxyConfig`, zod and generated schemas, config help/docs, security audit guidance, and auth tests. Define that it only applies after trusted proxy source, loopback policy, required-header, and browser-origin checks pass, and define `allowUsers` semantics for the fallback identity. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably injects identity headers on WebSocket upgrades.
