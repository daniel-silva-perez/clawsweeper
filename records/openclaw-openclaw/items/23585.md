## Summary

Keep open. Current main still fails closed when the trusted-proxy identity header is absent, and the config/types/schema/docs/tests do not expose a default or fallback identity option. The discussion distinguishes this Cloudflare/WebSocket missing-header case from the closed local-direct password fallback and Control UI pairing work, so this remains a security-sensitive product decision rather than conservative cleanup.

## What This Changes

Request adding an explicit default or fallback trusted-proxy user identity when a trusted proxy request lacks the configured identity header, especially for Cloudflare Zero Trust WebSocket upgrades.

## Best Possible Solution

Keep the issue open until maintainers decide the trusted-proxy contract. If accepted, add an explicit fallback identity option across exported config types, zod and generated schemas, docs, security audit guidance, and gateway/control-ui auth tests with precise fail-closed rules. If rejected, document Cloudflare Zero Trust IP-whitelist WebSocket paths without forwarded identity as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers.
