## Summary

Keep open. Current main still rejects trusted-proxy requests when the configured identity header is missing, and the config contract does not expose a default or fallback user. The issue discussion also separates this Cloudflare WebSocket missing-header case from the closed Control UI pairing and local-direct password fallback work, so this remains an auth-boundary product/security decision rather than cleanup closure.

## What This Changes

The issue requests a trusted-proxy default or fallback user option so requests from configured trusted proxies can authenticate when the configured identity header is absent.

## Best Possible Solution

Keep the issue open until maintainers decide the trusted-proxy contract. If accepted, add an explicit fallback identity option across exported config types, zod and generated schemas, docs, security audit guidance, and gateway/control-ui auth tests with precise fail-closed rules. If rejected, document Cloudflare Zero Trust IP-whitelist WebSocket paths without forwarded identity as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers.
