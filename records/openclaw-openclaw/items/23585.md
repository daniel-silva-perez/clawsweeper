## Summary

Keep open. Current main still fails closed when the trusted-proxy identity header is absent, the config/schema contract has no default or fallback identity field, and existing discussion distinguishes this Cloudflare/WebSocket missing-header case from the closed local-direct password fallback work. The requested behavior changes an authentication trust boundary, so it needs maintainer/security judgment rather than cleanup closure.

## What This Changes

The issue requests a trusted-proxy auth option such as defaultUser or fallbackUser so requests from configured trusted proxies can authenticate when the configured identity header is absent.

## Best Possible Solution

Keep this issue open until maintainers decide the trusted-proxy contract. If accepted, add an explicit fallback identity option across exported config types, zod and generated schemas, docs, security audit guidance, and gateway/control-ui auth tests with precise fail-closed rules. If rejected, document Cloudflare Zero Trust IP-whitelist WebSocket paths without forwarded identity as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers.
