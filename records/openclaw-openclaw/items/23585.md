## Summary

Keep open. Current main still fails closed when the trusted-proxy identity header is absent, the config/schema contract has no default or fallback identity field, and the existing discussion separates this Cloudflare/WebSocket missing-header case from the closed local-direct password fallback work. The requested change would alter an authentication trust boundary, so it needs maintainer/security judgment rather than conservative cleanup closure.

## What This Changes

Request to add a trusted-proxy default or fallback user used when a request comes from a trusted proxy but the configured identity header is absent, especially on Cloudflare Tunnel WebSocket upgrades.

## Best Possible Solution

Keep the issue open until maintainers decide the trusted-proxy contract. If accepted, add an explicit fallback identity option across exported config types, zod and generated schemas, docs, security audit guidance, and gateway/control-ui auth tests with precise fail-closed rules. If rejected, document Cloudflare Zero Trust IP-whitelist WebSocket paths without forwarded identity as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers.
