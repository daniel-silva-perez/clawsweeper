## Summary

Keep open. Current main still has no trusted-proxy fallback/default identity: trusted-proxy auth reads only the configured identity header and returns `trusted_proxy_user_missing` when it is absent, while config types, strict schemas, generated schema, docs, audit guidance, and tests all continue to model an identity header as required. The issue discussion also separates this Cloudflare/WebSocket missing-header case from the already-closed local password fallback work, so this remains a maintainer/security product decision rather than cleanup closure.

## What This Changes

Add an explicit trusted-proxy fallback/default user option for trusted proxy requests where the configured identity header is absent, especially Cloudflare Zero Trust WebSocket upgrade paths.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, add an explicit trusted-proxy fallback identity option across exported config types, zod/generated schemas, docs, security audit guidance, and auth/control-ui tests, with precise semantics for trusted proxy source checks, required headers, loopback policy, browser origin policy, and whether `allowUsers` must include the fallback identity. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers on WebSocket upgrades.
