## Summary

Keep open. Current main still rejects trusted-proxy requests when the configured identity header is absent, the config/type/schema/docs surface has no default or fallback user option, and the related discussion distinguishes this Cloudflare/WebSocket missing-header case from the closed local-direct password fallback work. Because the requested fallback would authorize no-identity requests at a gateway auth boundary, it needs maintainer/security judgment rather than cleanup closure or an automated fix lane.

## What This Changes

Add a trusted-proxy `defaultUser` or `fallbackUser` option so requests from configured trusted proxies can authenticate when the configured identity header is absent.

## Best Possible Solution

Keep this open for maintainer/security review. If accepted, add an explicit fallback identity option across exported config types, zod/generated schemas, docs, security audit guidance, and auth/control-ui tests, with precise constraints for trusted proxy source checks, loopback opt-in, required headers, browser origin policy, and whether `allowUsers` must include the fallback identity; if rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point operators to an auth/proxy path that forwards identity headers reliably.

## Reproduction Assessment

Yes for the OpenClaw behavior: a trusted-proxy auth request from a trusted source with no configured user header reproduces `trusted_proxy_user_missing`, and the current unit test asserts that rejection. The Cloudflare-specific WebSocket header loss was not live-reproduced here, but the repository-side missing fallback path is clear.

## Solution Assessment

Unclear. A `defaultUser` or `fallbackUser` could address the reporter's deployment, but it is not automatically the best fix because it would authorize requests without per-request identity; the maintainable path needs explicit security semantics or a documented unsupported configuration.
