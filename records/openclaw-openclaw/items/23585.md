## Summary

Keep open. Current main still has no trusted-proxy default/fallback identity path: the auth helper reads the configured identity header and returns `trusted_proxy_user_missing` when it is absent, while config types, strict zod validation, generated schema, tests, and docs still model `userHeader` as required. The linked discussion also separates this Cloudflare/WebSocket missing-header case from the closed local-direct fallback and Control UI pairing issues, so there is no implemented or canonical duplicate close path.

## What This Changes

The issue requests a trusted-proxy `defaultUser`/`fallbackUser` option so Cloudflare/WebSocket requests from a trusted proxy can authenticate when the configured identity header is missing.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, add an explicit trusted-proxy fallback identity option across exported config types, zod/generated schemas, docs, security audit guidance, and auth/control-ui tests. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers on WebSocket upgrades.
