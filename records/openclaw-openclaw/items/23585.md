## Summary

Keep open. Current main still fails closed when a trusted-proxy request lacks the configured identity header, and the public config/schema/docs expose no `defaultUser`, `fallbackUser`, or equivalent. The remaining request is a security-sensitive auth-boundary product decision, not conservative cleanup closure.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, implement an explicit trusted-proxy fallback identity option across the exported config type, zod schema, generated schema, docs, config help, security audit guidance, and auth tests. Semantics should specify that trusted proxy source checks, required headers, loopback policy, and browser-origin policy must pass before fallback applies, and whether `allowUsers` must include the fallback identity. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers on WebSocket upgrades.
