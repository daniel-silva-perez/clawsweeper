## Summary

Keep open. Current main still rejects trusted-proxy requests when the configured identity header is absent, the public config contract has no fallback/default identity field, and the issue discussion separates this Cloudflare/WebSocket missing-header case from the already-closed local-direct password fallback work. The requested behavior changes an authentication trust boundary, so it needs maintainer/security judgment rather than conservative cleanup closure.

## What This Changes

The issue requests a trusted-proxy `defaultUser` or `fallbackUser` option so requests from configured trusted proxies can authenticate when the configured identity header is missing.

## Best Possible Solution

Keep this open until maintainers decide the trusted-proxy contract. If accepted, add an explicit opt-in fallback identity field across exported config types, zod and generated schemas, docs, security audit guidance, and gateway/control-ui auth tests with fail-closed rules; if rejected, document Cloudflare Zero Trust IP-whitelist WebSocket paths without identity headers as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers.

## Reproduction Assessment

Yes for the OpenClaw behavior: a trusted-proxy request from a trusted source without the configured `userHeader` reaches `authorizeTrustedProxy` and returns `trusted_proxy_user_missing`, which is also asserted in `src/gateway/auth.test.ts`. The external Cloudflare header-drop portion is plausible from the report but not independently live-reproduced here.

## Solution Assessment

Unclear. A fallback/default user is a narrow functional fix for the reporter’s setup, but it is not automatically the best solution because it changes trusted-proxy auth from per-request identity evidence to proxy-source-only trust; the safer path is explicit maintainer/security approval with a tightly documented opt-in contract or a documented unsupported configuration.
