## Summary

Keep open. Current main still fails closed when a trusted-proxy request lacks the configured identity header: there is no `defaultUser`/`fallbackUser` config field, the zod and generated schemas remain strict, docs still require an identity header, and tests still assert `trusted_proxy_user_missing`. The discussion also explicitly separates this Cloudflare/WebSocket missing-header case from related trusted-proxy fixes, so this needs maintainer/security product judgment rather than cleanup closure.

## What This Changes

Add a trusted-proxy fallback/default user option so trusted proxy requests without the configured identity header can authenticate as a configured user.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, implement an explicit trusted-proxy fallback identity option across the exported config type, zod schema, generated schema/help, docs, security audit guidance, and auth tests. Semantics should specify that trusted proxy source checks, required headers, loopback policy, and browser-origin policy must pass before fallback applies, and whether `allowUsers` must include the fallback identity. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers on WebSocket upgrades.

## Risks / Open Questions

- Adding a fallback/default trusted-proxy identity changes the gateway authentication boundary by allowing headerless trusted-proxy requests to collapse to one configured user.
- The reported Cloudflare WebSocket behavior appears to be an upstream/proxy limitation; OpenClaw fallback support could mask proxy misconfiguration unless the semantics and audit guidance are explicit.
- Maintainers need to decide whether fallback should require `allowUsers`, required headers, browser-origin checks, loopback policy, and/or stronger proxy assertions before accepting headerless identity.
