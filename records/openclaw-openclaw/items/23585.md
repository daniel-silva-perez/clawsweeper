## Summary

Keep open. Current main still does not support a trusted-proxy default/fallback user when the configured identity header is absent. The auth path still returns trusted_proxy_user_missing, the config/schema/docs expose only userHeader/requiredHeaders/allowUsers/allowLoopback, tests assert the missing-header rejection, and maintainer discussion explicitly separates this from the already-related local-direct fallback work.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, implement an explicit trusted-proxy fallback/default identity option across GatewayTrustedProxyConfig, zod and generated schemas, config help/docs, security audit guidance, and auth tests. Define that it only applies after trusted proxy source, allowLoopback, required-header, and browser-origin checks pass; define allowUsers semantics for the fallback identity. If rejected on security grounds, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password/token auth, Tailscale Serve, or a proxy path that reliably injects identity headers on WebSocket upgrades.
