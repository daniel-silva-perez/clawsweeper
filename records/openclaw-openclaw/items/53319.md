## Summary

Keep this issue open. Current main has newer ACPX/Claude adapter work, timeouts, background tasks, and live ACP bind coverage, but I did not find code, tests, docs, history, or live-smoke evidence proving that two rapid Claude ACP `sessions_spawn` runs both launch their underlying process and emit output.

## Best Possible Solution

Keep this open until a maintainer runs a focused live smoke on current main: spawn two `sessions_spawn({ runtime: "acp", agentId: "claude", mode: "run" })` children in rapid succession and verify both create/own a Claude ACP process and emit assistant output. If it still reproduces, fix the ACPX/OpenClaw launch path with a backend-level launch guard, startup health check, or retry for silent launch hangs, then add regression coverage at the closest injectable launch-failure seam.
