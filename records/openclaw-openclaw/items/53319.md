## Summary

Keep this issue open. Current main has meaningful ACPX/Claude improvements, including bundled acpx 0.6.1, Claude adapter packaging, turn timeout handling, result-before-idle behavior, background task tracking, and live ACP bind coverage. I did not find source, tests, docs, release notes, or live-smoke evidence proving the specific reported case: two rapid Claude ACP `sessions_spawn` children both launch their underlying Claude process and emit output.

## Best Possible Solution

Keep the issue open until a maintainer runs a focused live smoke on current main: spawn two `sessions_spawn({ runtime: "acp", agentId: "claude", mode: "run" })` children in rapid succession, then verify both create/own a Claude ACP process and emit assistant output. If it still reproduces, fix the ACPX/OpenClaw launch path with a backend-level launch guard, startup health check, or retry for silent launch hangs, and add regression coverage at the closest injectable launch-failure seam.
