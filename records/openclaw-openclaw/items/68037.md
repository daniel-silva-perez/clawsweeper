## Summary

Keep open. Current main still maps HTTP MCP tool-result content blocks down to only `type` and `text`, so image/audio/resource payload fields are still dropped on successful `tools/call` responses. The linked #68046 remains open and unmerged with closing syntax for this issue, so maintainer review of that PR is the right path rather than closing early.

## What This Changes

Preserve non-text MCP content block fields from HTTP `tools/call` responses instead of normalizing every block down to `{type, text}`.

## Best Possible Solution

Review and revise #68046 as the focused implementation candidate: preserve valid MCP text/image/audio/resource/resource_link blocks, fall back to text for strings or malformed entries, add regression coverage for the HTTP `tools/call` path, and let the issue close after the fix lands.
