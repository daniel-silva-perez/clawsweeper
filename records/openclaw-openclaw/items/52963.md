## Summary

Keep this issue open. Current main still matches the prior ClawSweeper assessment: the bundled Slack plugin is bot-event and bot-token centered for Socket Mode monitor DMs, while `userToken` remains a read/preferred-action credential plus limited write fallback rather than an inbound user-event identity.

## What This Changes

Support Slack Socket Mode `message.im` events subscribed on behalf of a configured user token and route eligible DM replies through that user token.

## Best Possible Solution

Implement the feature in the bundled Slack plugin: add documented `user_events` manifest guidance for `message.im`, resolve the configured user token identity at monitor startup, distinguish bot-level and user-level DM events, select the xoxp token only for eligible opt-in replies, prevent self-loops, and cover manifest output, event routing, token selection, docs, and reply delivery with regression tests.
