## Summary

Keep this issue open. Current main still treats Slack Socket Mode as a bot-token monitor: setup/docs only advertise bot-event `message.im`, monitor startup resolves bot/app credentials and bot identity, and reply dispatch still sends monitor replies with `ctx.botToken` rather than a configured xoxp user token.

## What This Changes

The issue requests Slack user-level `message.im` event support so DMs to a configured Slack user account can be routed through OpenClaw and answered with the configured xoxp user token.

## Best Possible Solution

Implement this in the bundled Slack plugin after maintainer approval: document `user_events` setup for `message.im`, resolve the configured xoxp user identity during monitor startup, distinguish bot-level and user-level DM events, select the user token only for explicit opt-in replies, prevent self-loops, and cover manifest/docs, routing, token selection, and live Slack behavior with targeted regression tests.
