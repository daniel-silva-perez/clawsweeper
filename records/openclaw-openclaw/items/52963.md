## Summary

Keep this open. Current main still documents and implements Slack Socket Mode monitoring around bot-event subscriptions, bot/app credentials, and `ctx.botToken` reply delivery; `userToken` remains a read/preferred-action credential plus limited write fallback, not an inbound user-event identity for DM routing.

## What This Changes

The issue requests Slack Socket Mode support for `message.im` events subscribed on behalf of a configured Slack user, with replies routed through the xoxp user token so DMs to that regular user are handled by OpenClaw.

## Best Possible Solution

Implement the feature in the bundled Slack plugin after maintainer product/security direction: add documented `user_events` manifest guidance for `message.im`, resolve the configured user token identity at monitor startup, distinguish bot-level and user-level DM events, select the xoxp token only for eligible opt-in replies, prevent self-loops, and cover manifest output, event routing, token selection, docs, and delivery behavior with regression tests plus live Slack validation.
