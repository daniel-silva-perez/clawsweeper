## Summary

Keep this issue open. Current main still supports Slack user tokens only as supplemental action/read credentials and limited fallback writes, while the Socket Mode monitor, manifest, docs, inbound identity, and reply dispatch remain bot-token centered.

## What This Changes

Support Slack Socket Mode DM events subscribed on behalf of a configured Slack user and route replies through the configured user token when explicitly enabled.

## Best Possible Solution

Implement this in the bundled Slack plugin after maintainer approval: document and generate the `user_events` setup, resolve the configured user identity at monitor startup, distinguish bot-level from user-level DM events, select the user token only for explicit opt-in replies, prevent self-loops, and add targeted unit plus live Slack validation.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: current main can be inspected with `userToken` and `userTokenReadOnly: false` in mind and still shows no generated `user_events`, no monitor user identity, and bot-token reply dispatch. A live Slack reproduction remains useful for payload details, but it is not needed to see the missing implementation path.

## Solution Assessment

No automated fix path is best yet. The maintainable solution needs a Slack-plugin change, but xoxp user impersonation and inbound identity routing should be approved and live-validated before a repair PR is queued.
