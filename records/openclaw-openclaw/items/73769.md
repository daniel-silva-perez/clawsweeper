## Summary

Keep open. The PR is aimed at a real memory CLI/provider initialization boundary, but the proposed guard can skip the built-in registration path in the cold-registry shape that current main explicitly supports through capability fallback, and the overlapping open #72937 means maintainers should choose the canonical bootstrap layer before this lands.

## What This Changes

The PR re-exports memory embedding provider registry helpers through the memory CLI host runtime, calls a new provider-registration guard before memory manager creation, and adds CLI tests for cold and already-populated provider lists.

## Best Possible Solution

Adopt one cold-registry strategy for memory embeddings: either rely on manifest/capability fallback and keep CLI coverage focused on that behavior, or add a bootstrap that delegates idempotency to the registered-only built-in helper and includes a regression for the capability-fallback non-empty list case.

## Reproduction Assessment

Yes. At source/unit level, make `listMemoryEmbeddingProviders()` return a capability-only adapter while the registered-provider set is empty; this PR returns before `registerBuiltInMemoryEmbeddingProviders`, which is the failure path the current tests miss.

## Solution Assessment

No as submitted. The CLI bootstrap is narrow, but the early return should use a registered-only check or be removed so `registerBuiltInMemoryEmbeddingProviders` owns idempotency; maintainers should also reconcile this with #72937's manager-level approach.
