## Summary

Keep this PR open for maintainer review. Current main still lacks the PR’s explicit memory CLI bootstrap and CLI-specific regression coverage, but the proposed guard likely does not seed built-ins under current capability-aware provider-list semantics, so the branch needs revision or an architecture decision before merge.

## What This Changes

This PR re-exports embedding-provider registry helpers through the memory CLI host runtime, calls a provider-registration guard before memory manager creation, and adds CLI tests for cold and already-populated provider lists.

## Best Possible Solution

Adopt one canonical cold-registry strategy for local memory embeddings: either rely on manifest/capability fallback and add targeted CLI regression coverage for `memory status/index/search`, or add an explicit CLI or manager bootstrap that checks registered adapters only and lets the existing built-in registration helper handle idempotency.

## Reproduction Assessment

Yes, for the review finding at source/unit level: with a cold registered-provider set but manifest-capability providers available, `listMemoryEmbeddingProviders()` can be non-empty and the PR’s guard returns before registration. A focused regression should mock that production shape rather than only the empty-list case.

## Solution Assessment

No as submitted. The CLI bootstrap idea is narrow and follows an adjacent pattern, but the guard should use a registered-only check or delegate idempotency to `registerBuiltInMemoryEmbeddingProviders`; otherwise the PR may not implement its stated registration behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Check the registered provider list before skipping bootstrap:** `extensions/memory-core/src/cli.runtime.ts:107-109`
  - body: `listMemoryEmbeddingProviders()` is capability-aware: it merges manifest-resolved providers into the result. On a cold registry with declared providers available, this branch can return before `registerBuiltInMemoryEmbeddingProviders(...)`, leaving the built-in `local` adapter unregistered in the path the PR is trying to seed. Use a registered-only check or let the built-in registration helper perform the idempotent filtering.
  - confidence: 0.87
