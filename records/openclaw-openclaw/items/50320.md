## Summary

Keep this PR open. Current main has already solved the export-template regression and appears to have replaced the old google-vertex sentinel approach, but it still does not implement the full intended PR: current-run HEIC/HEIF chat attachments can still flow as `image/heic`/`image/heif`, and `openclaw config set gateway.nodes.denyCommands ...` still has no write-time warning. The PR also has unresolved review-identified risks in the auth and denyCommands changes.

## Best Possible Solution

Keep this PR open as a candidate, but ask for an update before merge: drop or rework the already-solved export/auth pieces against current main, implement HEIC/HEIF normalization in `parseMessageWithAttachments` before `parsed.images` reaches active prompts with post-conversion size checks, and add denyCommands config-set warnings based on a real full node command catalog rather than the effective allowlist.

## Risks / Open Questions

- Security review: the PR touches provider auth resolution and node command policy UX, both security-sensitive surfaces. The provided changed-file list does not include workflows, dependencies, lockfiles, install scripts, release metadata, package publishing, or generated/vendor blobs, so the main concern is behavioral rather than supply-chain.
- The google-vertex ADC change in the PR returns oauth without an apiKey, but current runtime paths still require an apiKey unless mode is aws-sdk. That could turn an auth-status fix into a runtime failure.
- The denyCommands warning approach shown in the PR diff uses the effective allowlist as a command catalog, which can mislabel real dangerous command IDs such as camera.snap or screen.record as unknown.
- The HEIC conversion proposed in the PR needs post-conversion size revalidation before returning prompt images, because converted JPEG payloads can exceed the original attachment budget.
