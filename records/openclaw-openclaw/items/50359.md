## Summary

Current `main` at `189c91eae6e1f4f280179961bca2c4153cf42e86` still does not implement this PR’s intended fallback from embedded `read` image results to original tool start args. The extractor only sees the result, current tests still expect pathless image content to produce no media path, and the paired #41744 issue remains open.

## Best Possible Solution

Keep this PR open for maintainer review. The best fix is a narrow equivalent of the PR on current `main`: either pass stored `tool_execution_start` args into trusted read-result media extraction, or make `createOpenClawReadTool` attach structured `details.media.mediaUrl` or `details.path` before extraction. Keep regression coverage for `path`, `file_path`, `filePath`, and final outbound media delivery, then resolve #41744 after the fix lands or the duplicate PRs are consolidated.
