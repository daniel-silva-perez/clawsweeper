## Summary

Keep open. Current main still lacks the audio-level private-network opt-in and local OpenAI-compatible audio auth behavior this PR targets, while the patch appears functionally sound aside from a low-priority test-helper fidelity issue.

## What This Changes

The PR adds audio-scoped `tools.media.audio.request.allowPrivateNetwork` support, threads audio request overrides into media provider execution, updates generated config/docs/changelog surfaces, and adjusts OpenAI-compatible audio auth for local endpoints.

## Best Possible Solution

Land this as the narrow audio-media fix after aligning the test helper and completing maintainer review of the SSRF/auth scope: keep private-network access explicit and audio-only, preserve image/video rejection, and avoid broadening provider credential behavior beyond OpenAI-compatible local audio.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the schema rejects the requested audio request key, runtime sanitization drops audio-level `allowPrivateNetwork`, and OpenAI-compatible audio always builds a provider Authorization header.

## Solution Assessment

Yes, with a small test-fidelity correction. The PR extends the existing model-provider request contract into audio config instead of bypassing the guarded HTTP path, and it scopes synthetic local auth to OpenAI-compatible audio endpoints.
