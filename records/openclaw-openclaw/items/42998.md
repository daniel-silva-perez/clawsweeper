## Summary

Keep this PR open. Current main fixes the duplicate rerun for empty cron output by no longer treating empty text as an interim acknowledgement, but it still does not implement the PR's broader intended behavior: recovering arbitrary successful raw-stdout toolResult text as the final isolated cron output when assistant-visible text is empty. The submitted patch is directionally related but needs a rewrite because it uses prompt keyword detection and transcript promotion that is not safely tied to the current attempt.

## Best Possible Solution

Keep this PR open, but require a rebase/rewrite on current main. The best fix is structural current-attempt toolResult recovery in the embedded runner or cron outcome flow: recover only verified successful current-run toolResult text, use it when assistant output is empty, suppress interim retry only when recovered output or an exact silent token proves completion, and add focused tests for raw stdout success, acknowledgement-without-tool-result retry, stale/older or missing timestamps, errored tool results, malformed transcript entries, and exact NO_REPLY.

## Risks / Open Questions

- The submitted transcript scan could promote stale, unrelated, missing-timestamp, or cross-run tool output unless recovery is tied to verified current-attempt state.
- The PR test uses a fixed /tmp transcript path and writes its timestamp before runStartedAt is captured, making it timing-sensitive and cleanup-dependent.
