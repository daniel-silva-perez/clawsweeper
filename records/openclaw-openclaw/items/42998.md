## Summary

Keep this PR open. Current main appears to have removed the narrow empty-output-as-interim retry path, but it still does not implement the PR’s intended completion behavior: using verified successful current-run toolResult text as the isolated cron output when assistant-visible text and reply payloads are empty. The submitted patch remains relevant, but should be rewritten around current-attempt runner state rather than keyword detection plus transcript-file scanning.

## Best Possible Solution

Keep this PR open for a rebase/rewrite. The best fix is a structural current-attempt recovery path: have the embedded runner expose verified latest successful current-run toolResult text, use it only when assistant output and reply payloads are empty, suppress interim retry only when recovered output or exact silent-token completion proves terminal success, and add focused tests for raw stdout success, acknowledgement-without-tool-result retry, stale/older or malformed transcript data, errored tool results, missing timestamps, and exact `NO_REPLY`.

## Risks / Open Questions

- Closing this PR now would lose the remaining behavior request: successful raw stdout toolResult text can still be absent from the final isolated cron output when no assistant-visible reply or payload is produced.
- The submitted implementation should not merge as-is because keyword-based gating can suppress the only interim retry even when no successful tool result was recovered.
- The transcript-file scan is data-integrity-sensitive unless tied to verified current-attempt runner state; stale, unrelated, missing-timestamp, or malformed transcript entries should not become cron output.
- Security review: the PR does not add dependency, workflow, lockfile, package-publish, or lifecycle-hook changes, but it does add runtime filesystem transcript reads on the cron path, so the safer design is to pass verified current-run tool result state from the embedded runner.
