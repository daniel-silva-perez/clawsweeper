## Summary

Keep this PR open. Current main has addressed the narrow duplicate-rerun trigger by not treating empty cron output as an interim acknowledgement, but it still does not implement the PR's broader intended behavior: recovering successful raw-stdout toolResult text as the final isolated cron output when assistant-visible text and reply payloads are empty. The submitted patch is still relevant but needs a safer rewrite.

## Best Possible Solution

Keep this PR open, but require a rebase/rewrite on current main. The best fix is structural current-attempt toolResult recovery in the embedded runner or cron outcome flow: recover only verified successful current-run toolResult text, use it when assistant output is empty, suppress interim retry only when recovered output or an exact silent token proves completion, and add focused tests for raw stdout success, acknowledgement-without-tool-result retry, stale/older or missing timestamps, errored tool results, malformed transcript entries, and exact NO_REPLY.

## Risks / Open Questions

- Current main can still finish an isolated cron run without surfacing arbitrary successful raw stdout when the embedded result has no reply payloads and no final assistant-visible text.
- The submitted patch's message keyword gate can suppress the only interim-ack retry for a raw-stdout prompt even when no successful tool result was captured.
- The submitted transcript scan is a data-integrity/security concern unless tied to verified current-attempt state; otherwise stale, unrelated, missing-timestamp, or cross-run tool output could be promoted as the cron result.
- The submitted test uses a fixed /tmp transcript path and writes its timestamp before runStartedAt is captured, making it timing-sensitive and shared-state prone.
