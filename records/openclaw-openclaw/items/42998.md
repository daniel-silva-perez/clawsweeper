## Summary

Keep this PR open. Current main has partially removed the narrow empty-output retry trigger, but it does not implement the PR's intended raw-stdout completion behavior: recovering a successful current-run toolResult as the isolated cron output when assistant-visible text and reply payloads are empty. The submitted patch remains relevant, but should be rewritten around verified current-attempt runner state rather than message keyword detection and transcript-file scanning.

## Best Possible Solution

Keep this PR open for a rebase/rewrite. The best fix is a structural current-attempt toolResult recovery path in the embedded runner or cron outcome flow: recover only verified successful current-run toolResult text, use it only when assistant output and reply payloads are empty, suppress interim retry only when recovered output or exact silent-token completion proves terminal success, and add focused tests for raw stdout success, acknowledgement-without-tool-result retry, stale/older or missing timestamps, errored tool results, malformed transcript entries, and exact `NO_REPLY`.

## Risks / Open Questions

- Closing this PR now would lose the remaining behavior request: raw stdout from a successful current-run toolResult can still be absent from the final isolated cron output when no assistant payload is produced.
- The submitted implementation should not be merged as-is because message-keyword gating can suppress the only interim retry even when no successful tool result was recovered.
- The submitted transcript-file scan is a data-integrity risk unless it is tied to verified current-attempt runner state; stale, unrelated, missing-timestamp, or malformed transcript entries should not be promoted as cron output.
- The submitted test uses a fixed `/tmp/transcript.jsonl` path and writes before the run captures `runStartedAt`, making it timing-sensitive and shared-state prone.
