## Summary

Keep open. Current main still lacks the direct inbound Feishu interactive-card fetch/raw-content path, and this PR should not merge as-is because the raw-card query shape is likely wrong and the RawBody change regresses normal message compatibility.

## What This Changes

The PR adds a shared Feishu interactive-card parser, fetches richer inbound interactive-message content, propagates raw card JSON through Feishu context fields, and adds focused Feishu tests.

## Best Possible Solution

Land a narrow Feishu plugin repair that preserves current-main quoted-card extraction, fetches inbound interactive-card payloads with `card_msg_content_type=raw_card_content`, keeps RawBody/CommandBody normalized for ordinary messages, exposes raw card JSON only for interactive-card context, and covers direct inbound cards plus existing quoted/thread card tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock an inbound Feishu `interactive` event whose event content is only `[Interactive Card]`; current main lacks the fetch/rawContent path, and this PR's mock would still pass even if the real fetch parameter does not request raw card content.

## Solution Assessment

No. The PR targets the right Feishu-owned boundary, but the raw-card fetch parameter and RawBody compatibility change need correction before this is the best maintainable fix.

Keep RawBody on normalized content except for an explicitly scoped interactive-card raw field.
  - confidence: 0.88
