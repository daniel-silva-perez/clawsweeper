## Summary

Keep open. Current main still does not solve the broader inbound Feishu interactive-card path: the shipped parser improvements cover fetched/quoted card content, while direct inbound interactive events still lack a dedicated fetch/raw-card context path. The PR is pointed at the right Feishu plugin surface, but it should not merge as-is because the follow-up fetch uses the wrong raw-card query shape and the RawBody change regresses normal non-interactive message compatibility.

## What This Changes

The PR adds a shared Feishu interactive-card parser, fetches richer inbound interactive-message content, propagates raw card JSON through Feishu context fields, and adds focused Feishu tests.

## Best Possible Solution

Land a narrow Feishu plugin repair that preserves current-main quoted-card extraction, fetches inbound interactive-card payloads with Feishu's raw-card message query shape, keeps RawBody/CommandBody normalized for ordinary messages, exposes raw card JSON only for interactive-card context, and covers direct inbound cards plus existing quoted/thread card tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock an inbound Feishu interactive event whose event content is only the placeholder; current main lacks an inbound interactive fetch/rawContent path, while this PR's current mock would still pass even if the real fetch parameter does not request raw card content.

## Solution Assessment

No. The PR targets the right Feishu-owned boundary, but the fetch parameter and RawBody compatibility change need correction; the safer solution is a smaller rebased patch that preserves existing body semantics and current-main parser improvements.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Use Feishu's raw-card message parameter:** `extensions/feishu/src/send.ts:274`
  - body: The new fetch sends `raw_card_content=true`, but the SDK type surface for `im.v1.message.get` only exposes `user_id_type`, and the related raw-card path uses `card_msg_content_type=raw_card_content`. With this parameter, the follow-up fetch can keep returning placeholder content, so inbound cards remain unreadable while the unit mock still passes. Send the raw-card query key Feishu expects and assert it in the mock.
  - confidence: 0.9
- **[P2] Preserve RawBody for normal messages:** `extensions/feishu/src/bot.ts:942`
  - body: `parseFeishuMessageEvent` now stores `event.message.content` in `ctx.rawContent` for every message type, and this line makes `RawBody` prefer that raw envelope. Regular text messages would expose values like `{"text":"hello"}` instead of the legacy clean body, breaking hooks/templates that use RawBody as the CommandBody alias. Keep RawBody on normalized content except for an explicitly scoped interactive-card raw field.
  - confidence: 0.88
