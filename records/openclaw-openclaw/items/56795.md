## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the broader inbound Feishu interactive-card parsing path, and the related Feishu issue remains open, but the PR has blocking functional problems in the raw-card fetch parameter and RawBody compatibility.

## What This Changes

The PR adds shared Feishu interactive-card parsing, fetches richer inbound interactive-message content, propagates raw card JSON through Feishu context fields, and adds focused Feishu tests.

## Best Possible Solution

Land a narrow Feishu plugin fix that keeps current-main quoted-card improvements, fetches inbound card payloads with the documented Feishu message.get card format parameter, preserves RawBody semantics for non-interactive messages, and adds targeted tests for placeholder inbound cards, raw card propagation, normal text RawBody compatibility, and representative card schemas.

## Reproduction Assessment

Yes. The main product gap is reproducible from source by sending or mocking an inbound Feishu interactive event whose event content is only the placeholder; current main has no interactive branch in parseMessageContent and no rawContent propagation, while the PR's mock test would still pass despite using an undocumented fetch parameter.

## Solution Assessment

No. The PR is pointed at the right Feishu-owned surface, but the fetch parameter should follow Feishu's documented card_msg_content_type contract and RawBody should stay normalized except for an explicitly scoped interactive-card raw field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use Feishu's documented raw-card query parameter:** `extensions/feishu/src/send.ts:274`
  - body: The new fetch sends `raw_card_content=true`, but Feishu documents `card_msg_content_type` for controlling card return format, and the SDK types only expose `user_id_type`. With this unknown parameter, the API can keep returning the placeholder payload, so inbound cards that need the follow-up fetch remain unreadable even though the unit mock passes. Use the documented card content parameter and cover it in the mock expectation.
  - confidence: 0.9
- **[P2] Preserve RawBody for non-interactive messages:** `extensions/feishu/src/bot.ts:942`
  - body: `parseFeishuMessageEvent` now stores `event.message.content` in `ctx.rawContent` for every message type, so this fallback changes normal text `RawBody` from the normalized message body to the Feishu JSON envelope. Existing hooks/templates that treat RawBody as the body alias will start seeing values like `{"text":"hello"}`. Keep RawBody on `ctx.content` unless the message is an interactive card with fetched raw card JSON, or expose the card JSON under a separate field.
  - confidence: 0.88
