## Summary

Keep open. Current main still rejects Telegram account-level `agentId` because the strict account schema and exported account type do not include that field, while docs route Telegram account-to-agent mapping through top-level `bindings[]`; #63380 is open and unmerged as the paired implementation candidate.

## What This Changes

The issue reports that upgrading from 2026.4.5 to 2026.4.8 made Telegram multi-account entries with account-level `agentId` fail strict config validation.

## Best Possible Solution

Review #63380 as the active implementation candidate. The durable outcome should either restore backward-compatible `channels.telegram.accounts.<id>.agentId` with defined routing semantics and regression coverage, or explicitly keep `bindings[]` as the only supported account-routing path and improve upgrade docs or doctor output so users can find the migration.
