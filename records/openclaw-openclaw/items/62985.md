## Summary

Keep open. Current main still rejects the reported Telegram account-level `agentId` shape, and #63380 is open, unmerged, and explicitly paired as the implementation candidate, so this issue should stay open until that PR is reviewed, landed, or closed.

## What This Changes

Telegram multi-account configs that used `channels.telegram.accounts.<id>.agentId` before the 2026.4.8 upgrade should continue to validate and route, or receive a clear migration path.

## Best Possible Solution

Review #63380 as the active implementation candidate. The durable end state should either restore backward-compatible account-level `agentId` with routing semantics and regression coverage, or make `bindings[]` the explicit migration with doctor and upgrade guidance that prevents this crash-on-restart path.

## Reproduction Assessment

Yes. The issue provides a concrete multi-account config, and current source confirms `channels.telegram.accounts.<id>` is parsed by a strict schema that omits `agentId`, which is enough to reproduce the validation failure at config load.

## Solution Assessment

Unclear until #63380 is reviewed. Restoring compatibility is a plausible narrow fix, but the best solution must include routing semantics or a migration path rather than only allowing the field in the schema.
