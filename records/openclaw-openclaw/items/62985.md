## Summary

Keep open. Current main still rejects Telegram account-level `agentId` in strict account config, while the supported documented routing path is top-level `bindings[]`; #63380 remains open and unmerged as the paired implementation candidate. The discussion also includes a verified workaround using `bindings[]`, but that does not resolve the reported upgrade regression in the repository.

## What This Changes

Restore or intentionally migrate compatibility for Telegram multi-account configs that put `agentId` under `channels.telegram.accounts.<id>` after upgrading from 2026.4.5 to 2026.4.8.

## Best Possible Solution

Review #63380 as the active implementation candidate. The durable outcome should either accept and correctly route or migrate legacy `channels.telegram.accounts.<id>.agentId` with regression coverage for the reported multi-account config, or explicitly keep `bindings[]` as the only supported path and improve upgrade docs or doctor output so users can find the migration.
