## Summary

Keep this issue open. Current main still has the reported Telegram inline-button contract mismatch: the agent prompt and Telegram docs advertise legacy/top-level `buttons` with `callback_data`, while the shared message tool schema exposes `presentation.blocks[{type:"buttons"}]` and the Telegram runtime renders buttons from normalized presentation/interactive payloads, not raw assistant text like `[[ ... ]]`.

## Best Possible Solution

Treat this as a contract-alignment bug. Update the agent prompt and Telegram docs to tell models to call `message(action="send")` with `presentation.blocks[{type:"buttons", buttons:[{label,value,style?}]}]`, keep Telegram's existing presentation renderer, and add regression coverage for the prompt/schema plus Telegram final delivery behavior. Only add legacy raw `[[...callback_data...]]` or top-level `buttons` compatibility if maintainers explicitly want it and it is tightly validated.
