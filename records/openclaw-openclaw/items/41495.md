## Summary

Keep this issue open. Current main still has the reported contract mismatch: the agent system prompt and prompt test advertise legacy Telegram `buttons=[[{text,callback_data,style?}]]`, while the shared message tool schema and Telegram runtime use structured `presentation.blocks[].buttons` / `interactive` payloads. Raw assistant text containing JSON-like button rows is still treated as text; I did not find a current-main parser or shipped fix that would convert that text into Telegram inline buttons.

## Best Possible Solution

Keep this open as a contract-alignment bug. Update `src/agents/system-prompt.ts` and `src/agents/system-prompt.test.ts` to instruct models to call `message(action="send")` with `presentation.blocks[{type:"buttons", buttons:[{label,value,style?}]}]`, align the Telegram docs with the public message-presentation contract, and add a Telegram regression showing presentation button blocks render as inline keyboards. Only add legacy raw-text or top-level `buttons` compatibility if maintainers intentionally want that narrower compatibility path and validate it explicitly.

## Risks / Open Questions

- Leaving this open behavior unchanged keeps Gemini and other less tool-strict providers prone to rendering JSON-like button rows as user-visible text instead of Telegram inline keyboards.
- A raw-text regex parser would need careful validation; parsing arbitrary assistant text into callback actions could misinterpret normal text or create surprising interactive controls.
