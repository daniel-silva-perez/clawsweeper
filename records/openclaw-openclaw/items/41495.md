## Summary

Keep this issue open. Current `main` still contains the Telegram inline-button contract mismatch described in the report and in the prior ClawSweeper comment: the system prompt and part of the Telegram docs advertise legacy/top-level `buttons=[[{text,callback_data,...}]]`, while the shared `message` tool exposes `presentation.blocks` and Telegram renders inline buttons from normalized presentation/interactive payloads, not raw assistant text.

## Best Possible Solution

Treat this as a contract-alignment bug. Update the agent prompt and Telegram docs to tell models to call `message(action="send")` with `presentation.blocks[{type:"buttons", buttons:[{label,value,style?}]}]`, keep Telegram's existing presentation renderer, and add regression coverage for the prompt/schema plus Telegram final delivery behavior. Only add legacy raw `[[...callback_data...]]` or top-level `buttons` compatibility if maintainers explicitly want it and it is narrowly validated.
