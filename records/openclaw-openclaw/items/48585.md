Keep this PR open. Current main still lacks `privacy.redactPII`, and the LLM-facing inbound context still serializes raw sender identity, chat id, conversation labels, reply/forward labels, and history senders. The PR remains the focused implementation candidate, but it needs rebase/review follow-up for the privacy and test gaps already raised in review.

## Best Possible Solution

Keep this PR open as the implementation candidate. Rebase it onto current main, preserve the prompt-cache separation already on main, add the `privacy.redactPII` config/docs/generated-schema surface if maintainers accept it, address the pseudonymization strength and forwarded-name test feedback, then run focused inbound metadata validation plus the changed gate.

## Risks / Open Questions

- The PR is privacy-sensitive: unsalted SHA-256 pseudonyms for phone-number-like identifiers are reversible by lookup/brute force, so maintainers should decide whether to use a keyed HMAC, document the weaker guarantee, or constrain the feature wording.
- The submitted test coverage has a known gap for capitalized `ForwardedFrom` leakage, per the existing review comment.
- The branch likely needs a rebase because current main has since moved stable trusted metadata away from volatile chat ids for prompt-cache stability, while the PR was authored against an older inbound metadata shape.
- No tests were run in this read-only review; focused inbound metadata tests and the normal changed gate are still needed after updates.
