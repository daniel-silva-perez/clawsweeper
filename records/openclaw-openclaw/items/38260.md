## Summary

Keep this issue open. Current main still accepts WebChat image attachments, can offload images as inbound media refs, then rehydrates those refs through `loadWebMedia` with image optimization enabled. On Linux/Node this still reaches the media-understanding core `sharp` artifact in-process, and I did not find code, docs, changelog, release, or related-item evidence that the reported `SIGILL` in `libvips-cpp` has been fixed or made obsolete. The prior ClawSweeper comment also kept it open, and the related closed title-search matches cover different crash paths such as Skia/PDF, config crash loops, or Docker CPU regressions.

## Best Possible Solution

Keep this issue open. A maintainer should reproduce on current main or the latest release in a Linux npm-global Node setup, collect a core/backtrace around sharp/libvips, determine whether bundled libvips, system libvips selection, CPU dispatch, or OpenClaw’s optimization path is responsible, then add a safe fallback or process-isolation strategy plus a WebChat image-flow regression smoke.
