## Summary

Keep open. Current main still permits the reported split: status can show persisted WhatsApp direct-session rows from the session store, while agent session tools apply the default `tools.sessions.visibility=tree` policy and can hide or reject same-agent direct sessions that were not spawned by the requester.

## What This Changes

The issue reports that WhatsApp direct sessions visible in `openclaw status --json` and usable for cron delivery are not consistently discoverable or targetable by agent-side session tools.

## Best Possible Solution

Keep this open until maintainers choose the intended policy. If on-demand automation should target existing same-agent WhatsApp DMs, add a generic deliverable-session discovery or target-resolution path that remains visibility-aware, respects agent-to-agent policy and sandbox clamps, and documents the required configuration instead of special-casing WhatsApp.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to persist a same-agent WhatsApp direct session under `session.dmScope=per-channel-peer`, keep `tools.sessions.visibility` unset or `tree`, then compare `openclaw status --json` with `sessions_list` or `sessions_send` from a different same-agent requester session that did not spawn the WhatsApp session.

## Solution Assessment

Unclear as a product decision. Always exposing WhatsApp DMs would bypass an intentional session-tool privacy boundary; the safer direction is a generic visibility-aware deliverable-session mechanism or clear docs telling operators to use `tools.sessions.visibility=agent` when they want same-agent DM targeting.
