## Summary

Keep this open. Current main still allows the reported split: status can show persisted WhatsApp direct session rows from the session store, while agent-side session tools still apply the default tree visibility policy and can hide same-agent WhatsApp DM sessions that were not spawned by the requester.

## What This Changes

The issue asks for WhatsApp direct sessions that appear in status and work for scheduled delivery to be consistently discoverable and targetable through agent-side session tools.

## Best Possible Solution

Keep the issue open until maintainers choose the intended policy. If on-demand automation should target existing same-agent WhatsApp DMs by default, implement a generic deliverable-session discovery or target-resolution path that respects tools.sessions.visibility, agent-to-agent policy, and sandbox clamps, with WhatsApp direct coverage and clear docs for the required visibility setting.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to create a persisted same-agent WhatsApp direct session under session.dmScope=per-channel-peer, leave tools.sessions.visibility at its default tree value, then compare status output with sessions_list or sessions_send from a different same-agent requester session that did not spawn the WhatsApp session.

## Solution Assessment

Unclear as a product decision. Making WhatsApp DMs always visible would bypass an intentional session-tool privacy boundary, so the better direction is either a generic visibility-aware deliverable-session mechanism or documentation/config guidance that tells operators to use agent visibility when they want same-agent DM targeting.
