## Summary

Keep open. Current main does not provide high-confidence evidence that WhatsApp direct sessions are consistently discoverable from agent-side session tools: both status/session listing paths read persisted session rows, but the agent `sessions_list` tool applies the default `tools.sessions.visibility="tree"` guard, which hides same-agent non-current direct sessions such as `agent:main:whatsapp:direct:<phone>` unless configuration is widened or the caller is already in that session.

## Best Possible Solution

Keep the issue open and reproduce on current main with a linked WhatsApp account, `session.dmScope=per-channel-peer`, a successful cron delivery, and `sessions_list`/`sessions_send` from the same automation context. If maintainers want on-demand automation to target existing WhatsApp DMs, implement a generic, policy-aware session discovery or target-resolution path for deliverable same-agent channel sessions, with WhatsApp direct coverage and docs clarifying the required `tools.sessions.visibility` setting. Avoid a core special case for WhatsApp ids.

## Risks / Open Questions

- A fix that broadens default session discovery could leak session metadata across users or agents; it should preserve the documented `tools.sessions.visibility` and sandbox clamp semantics unless maintainers intentionally change that policy.
- The remaining reproduction should distinguish between a visibility-policy mismatch and a missing session-store write after cron/WhatsApp delivery; both can produce the same user-visible 'no WhatsApp sessions' result.
