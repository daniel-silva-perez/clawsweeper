## Summary

Keep this issue open. Current main still couples the macOS app Launch at login toggle to live launchd state for `ai.openclaw.mac`; there is no separate persisted user intent, stale/unloaded/repair-needed state, or Sparkle/post-update reconciliation path. The related #40213 fix is present, but it only removed `KeepAlive` from the app LaunchAgent and does not address update-time persistence or UI drift.

## Best Possible Solution

Keep this issue open and fix the macOS Launch at login contract in the app: persist the user's intended launch-at-login preference separately from live launchd status, distinguish disabled from stale/unloaded/repair-needed LaunchAgent states, reconcile or re-bootstrap `ai.openclaw.mac` on app startup and after Sparkle updates, and add focused tests for stale plist, unloaded job, and UI reconciliation behavior.
