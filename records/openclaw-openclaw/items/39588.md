## Summary

Keep this issue open. Current main still drives the macOS app Launch at login setting from the live `ai.openclaw.mac` launchd job, without a separate persisted user intent or Sparkle/post-update repair path. The related #40213 change removed `KeepAlive` from the app LaunchAgent, but it does not address update-time persistence or UI drift.

## Best Possible Solution

Keep this issue open and fix the macOS app Launch at login contract: persist the user's intended preference separately from live launchd state, model disabled versus stale/unloaded/repair-needed LaunchAgent states, reconcile or re-bootstrap `ai.openclaw.mac` on app startup and after Sparkle updates, and add focused tests for stale plist, unloaded job, UI reconciliation, and post-update repair behavior.
