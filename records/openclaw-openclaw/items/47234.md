## Summary

Keep open. Current main still does not implement the requested group/channel-to-main session scope, and this PR remains a real implementation candidate, but the proposed patch has blocking routing gaps and mixes the separate IRC proactive-send change into the same branch.

## What This Changes

This PR adds a route-binding `groupScope` option for matched group/channel peers to use an agent main session and adds an IRC active-client registry so proactive sends can reuse the monitor connection.

## Best Possible Solution

Refresh this as a first-class route-binding session-scope change on current main, preferably after splitting or landing the IRC proactive-send fix separately. The accepted implementation should choose the canonical config shape, update TypeScript/Zod/generated schema/help/docs, carry the scope through inbound routing, outbound session derivation, Telegram topic overrides, and focused tests, and document the privacy consequence of merging a selected group/channel into main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: When a binding sets `groupScope: "main"`, inbound routing can collapse group/channel traffic onto the main session, but outbound helpers still derive group targets from only global `dmScope`. Sends that omit an explicit session key, such as gateway or tool-driven sends, can fork the same conversation back into a per-channel session instead of the configured main session.
  - confidence: 0.89
- **[P2] Preserve groupScope in Telegram topic overrides:** `src/routing/resolve-route.ts:673`
  - body: Telegram forum topics with `topicAgentId` recompute `route.sessionKey` using `buildAgentSessionKey()` and only `dmScope`. A topic that matches `groupScope: "main"` will be overwritten back to a per-topic group session, so that supported channel path will not honor the new option.
  - confidence: 0.84
- **[P3] Fall back when the active IRC send fails:** `extensions/irc/src/send.ts:72-74`
  - body: When an active monitor client is found, the new IRC path calls `active.sendPrivmsg()` and returns. If the monitor connection drops after `isReady()` but before the write, proactive sends fail instead of trying the transient fallback preserved for missing active clients.
  - confidence: 0.72
