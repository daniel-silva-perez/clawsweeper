## Summary

Keep this PR open. Current main still allows Discord startup to continue with an undefined bot identity when `fetchUser("@me")` rejects, and that undefined identity still flows into the message listener paths that enforce self-message filtering and mention gating. The submitted patch is a useful direction, but it is stale against the current plugin layout and incomplete because it only covers a thrown lookup, not a successful response without a usable id.

## Best Possible Solution

Keep this PR open as a useful but stale implementation candidate. The best maintainer path is to port the fix to `extensions/discord/src/monitor/provider.startup.ts`, make `fetchDiscordBotIdentity` abort startup when lookup rejects or yields no valid bot id, add focused regression tests for both paths, and address the related mention-gate defense-in-depth tracked in #42219 before closing or superseding this PR.
