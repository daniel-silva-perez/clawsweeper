## Summary

Keep this PR open. Current main still allows cron run session keys to flow into heartbeat wake/session resolution, so the underlying bug is real, but the current patch still has a blocking routing regression because cron-originated wakes lose agent scope and can become unscoped multi-agent heartbeat fan-out.

## What This Changes

The PR changes heartbeat wake option construction and heartbeat session resolution so cron session keys are not used as heartbeat targets, with regression tests for scoped wake options and forced cron-session fallback.

## Best Possible Solution

Revise the PR so cron run session keys are never used as heartbeat run sessions while preserving the originating agent target, likely by converting a cron session key into an `agentId`-scoped wake or that agent's main heartbeat session. Add non-default and multi-agent regression coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve agent targeting when dropping cron session keys:** `src/routing/session-key.ts:46`
  - body: Returning `wakeOptions` unchanged for cron keys removes the only targeting data current wake callers provide. `requestHeartbeatNow` then enqueues an unscoped wake, and `startHeartbeatRunner` runs every enabled agent instead of the originating cron job's agent main session, so a background exec in `agent:ops:cron:...` can trigger unrelated heartbeats.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain change, but it introduces a concrete cross-agent routing risk by turning cron-originated heartbeat wakes into unscoped wakes.

Concerns:

- **[medium] Cron wake can lose agent isolation:** `src/routing/session-key.ts:46`
  - body: When the cron session key is dropped without an `agentId` replacement, the wake becomes unscoped and may run unrelated agents' heartbeats, which can send messages or run prompts outside the originating cron job's agent boundary.
