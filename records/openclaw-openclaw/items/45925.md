## Summary

Keep this PR open, but do not merge it as-is. The PR addresses a real heartbeat/cron routing bug and the latest branch has already fixed the earlier raw string-match review feedback by using `isCronSessionKey()`, but its helper change drops all targeting for cron-originated exec wakes; current wake dispatch then treats the wake as unscoped and can run unrelated agents.

## What This Changes

The PR changes heartbeat wake option construction and forced heartbeat session resolution so cron session keys are not used as heartbeat targets, with tests for the session-key helper and `runHeartbeatOnce` fallback behavior.

## Best Possible Solution

Keep cron run sessions out of heartbeat execution while preserving the originating agent boundary, likely by converting cron session keys into an `agentId`-scoped wake or that agent's main heartbeat session, with non-default and multi-agent regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a cron session key passed through the PR's `scopedHeartbeatWakeOptions` returns unscoped options, and the existing wake runner handles unscoped wakes by iterating every enabled heartbeat agent.

## Solution Assessment

No. Refusing to run heartbeat inside an isolated cron session is the right direction, but dropping the only target data is not the narrowest maintainable fix; the patch should preserve the originating agent target.
