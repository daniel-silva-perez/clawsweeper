## Summary

Keep this PR open. Current main still has the cron-session heartbeat targeting bug the PR is trying to fix, and the latest PR revision resolved the earlier raw-string cron check concern, but the patch still drops all targeting for cron-originated wakes and can fan out heartbeats to unrelated agents.

## What This Changes

The PR branch changes heartbeat wake option construction and forced heartbeat session resolution so cron session keys are not used as heartbeat targets, with regression tests for the session-key helper and runHeartbeatOnce fallback behavior.

## Best Possible Solution

Revise the PR so cron run session keys are never used as heartbeat run sessions while still preserving the originating agent target, likely by converting cron session keys into an `agentId`-scoped wake or that agent's main heartbeat session. Add non-default and multi-agent regression coverage before merge.

## Reproduction Assessment

Yes. The source path is high-confidence: exec completion passes only `sessionKey` through `scopedHeartbeatWakeOptions`, the PR returns no target fields for cron keys, and the heartbeat runner fans out unscoped wakes across enabled agents.

## Solution Assessment

No. Dropping cron session keys is the right direction, but the patch should preserve the originating agent boundary instead of turning cron-originated wakes into unscoped global wakes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve agent targeting when dropping cron session keys:** `src/routing/session-key.ts:46`
  - body: Returning `wakeOptions` unchanged for cron keys removes the only targeting data current wake callers provide. `requestHeartbeatNow` then enqueues an unscoped wake, and `startHeartbeatRunner` runs every enabled agent instead of the originating cron job's agent main session, so a background exec in `agent:ops:cron:...` can trigger unrelated heartbeats.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain change, but it introduces a concrete cross-agent routing risk by turning cron-originated heartbeat wakes into unscoped wakes.

Concerns:

- **[medium] Cron wake can lose agent isolation:** `src/routing/session-key.ts:46`
  - body: When the cron session key is dropped without an `agentId` replacement, the wake becomes unscoped and may run unrelated agents' heartbeats, which can send messages or run prompts outside the originating cron job's agent boundary.
