## Summary

Keep this PR open. Current main still routes cron session keys into heartbeat wake targeting, so the reported bug is real, but this patch is not merge-ready because the helper change drops the only target data from cron-originated exec completions and can turn a single-agent wake into an unscoped wake across all heartbeat-enabled agents.

## What This Changes

The PR changes heartbeat wake option construction and forced heartbeat session resolution so cron session keys are not used as heartbeat targets, with focused regression tests for the helper and runHeartbeatOnce fallback behavior.

## Best Possible Solution

Keep cron run sessions out of heartbeat execution while preserving the originating agent boundary, likely by converting cron session keys into an agentId-scoped wake or that agent's main heartbeat session, with regression coverage for non-default and multi-agent cron exec wakes.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a cron session key passed through the PR's scopedHeartbeatWakeOptions returns no sessionKey or agentId, and current wake dispatch runs every enabled heartbeat agent when no target is present.

## Solution Assessment

No. Refusing to run heartbeat inside an isolated cron session is the right direction, but dropping the only target data is not the narrowest maintainable fix; the patch should preserve the originating agent target.

## Review Findings
