## Summary

Keep this PR open. Current main still has the cron-session heartbeat targeting bug, but the PR branch needs revision because dropping a cron session key without preserving the originating agent turns a scoped exec wake into an unscoped heartbeat wake.

## What This Changes

The PR changes heartbeat wake/session resolution to avoid targeting cron run session keys and adds regression coverage for scoped wake options and forced cron session fallback.

## Best Possible Solution

Revise the PR so cron run session keys are never used as heartbeat run sessions, while converting the cron key into an explicit originating agent target, typically the same agent's main heartbeat session or an `agentId`-scoped wake. Add non-default and multi-agent regression coverage before landing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve agent targeting when dropping cron sessions:** `src/routing/session-key.ts:46-47`
  - body: Returning `wakeOptions` unchanged for cron keys removes the only target data exec-completion callers provide. `requestHeartbeatNow` then enqueues an unscoped wake, and `startHeartbeatRunner` runs every enabled agent rather than the originating cron job's agent main session, so a background exec in `agent:ops:cron:...` can trigger unrelated/default heartbeats.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain change, but it introduces a concrete cross-agent routing risk by dropping agent scope from cron-originated heartbeat wakes.

Concerns:

- **[medium] Cron wake can lose agent isolation:** `src/routing/session-key.ts:46`
  - body: When a cron session key is dropped without an `agentId` replacement, the wake becomes unscoped and may run unrelated agents' heartbeats, which can send messages or run prompts outside the originating cron job's agent boundary.
