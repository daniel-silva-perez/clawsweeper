## Summary

Keep open. Current main still routes webhook agent runs through the isolated cron runner with `forceNew: true`, so stable hook session keys cannot reuse a fresh existing session; the PR is a narrow implementation candidate paired with open bug #70894 and should be reviewed or landed rather than cleaned up.

## What This Changes

The PR changes isolated cron session resolution so webhook-sourced isolated agent runs with hook provenance do not force a new session, and adds a focused regression test for that behavior.

## Best Possible Solution

Land this PR or a narrow equivalent on main: derive `forceNew` from true scheduled isolated-cron semantics, allow hook/webhook-sourced isolated agent turns with stable hook provenance to reuse fresh stored sessions, and keep regression coverage for both webhook reuse and ordinary isolated-cron freshness.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is direct `/hooks/agent` dispatch with the same permitted `sessionKey`: the handler creates an isolated cron job, current main passes `forceNew: true`, and the resolver therefore creates a new session id instead of reusing the fresh stored one.

## Solution Assessment

Yes. The proposed fix is narrow and uses hook provenance that `prepareCronRunContext` already computes, while preserving `forceNew: true` for non-hook isolated cron runs; this is preferable to a new config flag or resolver-level special case.
