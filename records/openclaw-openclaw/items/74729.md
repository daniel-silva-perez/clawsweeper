## Summary

Keep open. Current main still advertises Feishu `read`, but the handler only accepts `messageId` and calls the single-message lookup path; the requested chat-history pagination path is still missing.

## What This Changes

The issue requests paginated Feishu chat-history reads through the message read action using chatId/target, limit, before/after, and page-token style parameters.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path while preserving the existing `messageId` lookup path, using the Feishu message-list endpoint for chat containers and documenting the supported parameters.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: call Feishu `read` with a chat target and no `messageId`, and the handler throws before any list call; call it with `messageId`, and it uses the single-message `get` path.

## Solution Assessment

Yes, a narrow Feishu plugin change is the best direction. Core already passes generic fetch parameters, so the maintainable fix belongs in `extensions/feishu` with tests and docs rather than a shared-core migration.
