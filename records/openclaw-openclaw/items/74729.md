## Summary

Keep open. Current main still advertises Feishu `read`, but the Feishu handler only accepts `messageId` and calls the single-message lookup path; chatId-based paginated history reads are still missing. The broader pagination issue #71452 is related, but it does not supersede this Feishu-specific API gap.

## What This Changes

Feishu `message read` should support chatId-based paginated history retrieval with limit, page token, and time filters instead of only single-message lookup by messageId.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path using `client.im.message.list` with `container_id_type: "chat"`, preserving the existing `messageId` lookup behavior. Cover chatId/pageToken/time-filter reads in Feishu tests, update the Feishu action alias/docs surface, add a user-facing changelog entry, and leave the broader cross-channel 25-item rendering/pagination cleanup to #71452.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action=read` with only `chatId`, `limit`, `before`, `after`, or `pageToken` cannot reach a list path and instead fails before dispatching any Feishu history request.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the best path because the shared runner already passes action parameters through to channel plugins; core should only need a small target-alias or normalization test if `chatId` must satisfy the generic `read` target check.
