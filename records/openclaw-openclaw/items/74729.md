## Summary

Keep open. Current `main` still exposes Feishu `read`, but the implementation only supports single-message lookup by `messageId`; the requested paginated chat-history path is still missing and the broader open pagination issue does not supersede this Feishu-specific API gap.

## What This Changes

Feishu message `read` should support paginated chat-history listing with chatId, limit, page_token, and before/after filters instead of only single-message lookup by messageId.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path while preserving the existing `messageId` lookup path, update Feishu action aliases/docs/tests for chatId-based reads, and leave broader cross-channel pagination cleanup to #71452.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: a Feishu `read` with only chatId/limit/pageToken cannot reach a list path, while a read with `messageId` calls only the single-message `get` helper.

## Solution Assessment

Yes. A narrow Feishu plugin change is the maintainable fix because the shared runner already passes action params through to channel plugins; core should only need a small alias/normalization test if chatId must satisfy target validation.
