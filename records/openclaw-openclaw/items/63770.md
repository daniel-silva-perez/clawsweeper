## Summary

Current main still has the at-only `deleteAfterRun` guard in the cron scheduler, so recurring `every` and `cron` jobs with `deleteAfterRun: true` continue into next-run scheduling instead of removal. The issue should stay open because PR #64375 is open, unmerged, and explicitly targets this bug with closing syntax and focused regression tests.

## Best Possible Solution

Keep this issue open and review PR #64375 as the canonical implementation candidate. If maintainers agree that explicit `deleteAfterRun: true` should self-delete `every` and `cron` jobs after a successful run, land the narrow guard removal with the regression tests and a user-facing changelog entry. If maintainers reject that semantic change, close or revise the PR and record the decision here.
