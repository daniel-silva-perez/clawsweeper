## Summary

Keep this issue open. Current main still lacks the requested model-facing exec `description`/`intent` parameter, and the related #61562 was closed as a duplicate of this older tracker rather than as a fix.

## What This Changes

Request an optional exec tool description or intent field that models can populate and hooks, approval prompts, and audit logs can consume as informational metadata.

## Best Possible Solution

Keep the tracker open and design an additive optional `description` or `intent` field as untrusted display/audit metadata, then thread it through the exec schema, runtime params, hook-visible params, approval request protocol, approval UI, docs, and focused tests while preserving command-based safety enforcement.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of current main: the exec schema/runtime, hook types, approval protocol, approval UI, and public docs all lack the requested field.

## Solution Assessment

Unclear until maintainer/security review sets trust semantics. The narrow additive field is plausible, but the maintainable solution must explicitly document that intent text is informational and cannot replace command-based approval policy.
