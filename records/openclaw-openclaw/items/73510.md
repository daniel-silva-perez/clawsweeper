## Summary

Keep open. Current main has added conservative stale-lane recovery and several adjacent mitigations, but active embedded runs, active reply work, and active lane tasks still remain warning-only by default, so the central permanent stuck-session outage is not fully resolved.

## What This Changes

The issue requests automatic stuck gateway-session recovery plus startup compaction, trajectory-size, memory-limit, and Feishu ping mitigations for a v2026.4.26 production outage.

## Best Possible Solution

Keep this as the production outage umbrella until maintainers decide and land a diagnostics-owned, configurable, race-safe active-run abort/reset policy; then split or close the remaining startup compaction, Feishu, trajectory, and service guardrail pieces against narrower trackers such as #64585 and #60858.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests reproduce that active embedded/reply/lane work remains warning-only past the stuck threshold. The full Feishu end-to-end outage has credible report logs, but no live reproduction was attempted in this read-only review.

## Solution Assessment

No, current main is not yet the complete fix because it only releases stale lanes with no active work. The maintainable direction is a narrower diagnostics-owned abort threshold and recovery path, with separate follow-ups for startup compaction and channel/service guardrails.
