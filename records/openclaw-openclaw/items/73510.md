## Summary

Keep open. Current main has partial watchdog recovery and several mitigations, but active run abort/reset and startup compaction policy remain unresolved; the open diagnostics abort PR and related trackers make this a maintainer split-or-land decision, not conservative cleanup.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus startup compaction, trajectory bounds, memory-limit guidance, Feishu probe isolation, and service PATH repair after a Feishu outage on v2026.4.26.

## Best Possible Solution

Keep this as the outage umbrella until maintainers either land a configurable, race-safe active stuck-session abort/reset policy or choose a different recovery model, then split remaining startup compaction, service memory tuning, Feishu probe, and PATH cleanup into focused follow-ups with clear owners.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show active embedded runs, active reply operations, and active lane tasks remain warning-only after the stuck-session threshold. The complete Feishu outage chain is log-backed but was not live-reproduced here.

## Solution Assessment

No, current main is not the complete solution because it only releases stale lanes when no active work is present. The narrow maintainable direction is a diagnostics-owned configurable abort/reset path for confirmed stuck active runs, with the other requests split after the central policy is settled.
