## Summary

Keep open. Current main has partial mitigations, including conservative stale-lane recovery, trajectory caps, Feishu probe timeout handling, and docs for memory controls, but it still does not satisfy the central request: active stuck embedded/reply/lane work remains warning-only by default, and the open diagnostics-abort PR has not merged.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus startup compaction, trajectory limits, memory guardrails, and Feishu heartbeat hardening after a Feishu WebSocket outage on v2026.4.26.

## Best Possible Solution

Keep this as the production-outage umbrella until maintainers review and either land or replace #73243 for a configurable, race-safe stuck-session abort path. Then split any remaining startup compaction, transcript-growth, Feishu liveness, and Linux service memory guardrail work into narrower follow-ups tied to their owner modules.

## Reproduction Assessment

Yes. The issue includes concrete production logs, and the current source-level reproduction path is a session left in `processing` past `diagnostics.stuckSessionWarnMs` with active embedded/reply/lane work; main will warn and intentionally skip active recovery in that path.

## Solution Assessment

No. Current main’s stale-lane release is a useful mitigation, but it is not the complete fix for the reported active stuck-session outage; the safer direction is a reviewed configurable abort/cleanup contract like #73243 plus separate follow-ups for the non-diagnostics subproblems.
