## Summary

Keep open. Current main has partial mitigations, including conservative stale-lane release, trajectory caps, and Feishu probe timeout handling, but it still intentionally leaves active embedded runs, active reply work, and active lane tasks observe-only. The central production outage request for configurable active stuck-session abort/reset is not fully implemented, and the remaining work is already tied to open related trackers and PR #73243.

## What This Changes

The issue reports a Feishu/Linux production outage where stuck processing sessions, large transcripts and trajectory files, Feishu probe timeouts, service PATH gaps, and missing memory/process limits prevent gateway self-recovery.

## Best Possible Solution

Keep this as the production outage umbrella until diagnostics can safely recover genuinely stuck active runs, then split or close the remaining startup compaction, transcript growth, Feishu liveness, and Linux service guardrail items against narrower owners with focused validation.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show active embedded/reply/lane work past the stuck threshold remains warning-only. The full Feishu outage chain has credible production logs but was not live end-to-end reproduced in this read-only review.

## Solution Assessment

No. Current main is not the complete solution because it only releases stale lanes with no active work; the safer path is a configurable diagnostics-owned active abort/reset policy, reviewed alongside PR #73243, with separate focused follow-ups for startup compaction and service/runtime guardrails.
