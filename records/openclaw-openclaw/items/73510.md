## Summary

Keep open. Current main has added partial stuck-session recovery for stale lanes, but active embedded runs, active reply work, and active lane tasks are still intentionally observe-only, so the central production outage report is not fully implemented. Several adjacent sub-requests now have mitigations, but startup compaction and active-run abort/reset policy remain unresolved and are already tied to open related trackers/PRs.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus startup compaction, trajectory size limits, gateway memory guidance, Feishu probe isolation, and Linux service PATH repair after a Feishu WebSocket outage on v2026.4.26.

## Best Possible Solution

Keep this as the production outage umbrella until diagnostics can safely recover genuinely stuck active runs, then close or split the remaining startup compaction, transcript-growth, Feishu liveness, and Linux service guardrail items into narrower trackers with clear owners and validation paths.

## Reproduction Assessment

Yes for the central code-path gap: current tests and source inspection reproduce that active embedded/reply/lane work past the stuck threshold remains warning-only. The full Feishu outage chain has credible production logs but no live end-to-end reproduction in this review.

## Solution Assessment

No, current main is not yet the best complete solution because it only releases stale lanes with no active work. A narrower maintainable fix would add diagnostics-owned, configurable, race-safe abort/reset behavior for confirmed stuck active runs and leave compaction/Feishu service concerns to focused follow-ups.
