## Summary

Keep this issue open. Current main still only warns/emits diagnostics for stuck processing sessions; it does not abort the active run, clear the lane, reset the session, or restart the gateway. Main has adjacent mitigations for restart recovery, trajectory caps, Feishu probe timeout handling, Linux PATH repair/audit, and memory diagnostics, but those do not resolve the central auto-recovery outage reported here. Related PR #73243 remains the narrower implementation candidate and is not merged in the provided context.

## Best Possible Solution

Keep this issue open as the production outage umbrella. The best maintainer path is to review, land, or replace PR #73243 for diagnostics-owned stuck-session abort and queue cleanup, then split remaining startup compaction, transcript growth, Feishu liveness, and Linux service memory guardrail work into narrower follow-ups if they are not fixed by that change.
