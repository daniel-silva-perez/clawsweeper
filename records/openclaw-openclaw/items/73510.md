## Summary

Keep open. Current main has partial stuck-session mitigation, trajectory size caps, Feishu probe timeout/caching, and Linux memory/PATH diagnostics, but the central production outage remains unresolved because active embedded runs, active reply work, and active lane tasks are still warning-only by default. Startup compaction for existing large transcripts is also not implemented, and the remaining active-run abort/reset policy is already tied to open related work that needs maintainer judgment.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus startup compaction, trajectory size limits, gateway memory guardrails, Feishu ping isolation, and service PATH repair after a v2026.4.26 Feishu outage.

## Best Possible Solution

Keep this as the production outage umbrella until maintainers decide the active-run abort/reset policy, then land a focused diagnostics-owned recovery fix and split any remaining startup compaction, transcript growth, Feishu probe, and Linux service guardrail work into narrower owner-specific follow-ups.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show that active embedded runs, active reply work, and active lane tasks past the stuck threshold remain warning-only by default. The full Feishu outage chain has credible logs in the report but was not live-reproduced in this review.

## Solution Assessment

No, current main is not the complete solution because it only recovers stale lanes with no active work and leaves active stuck runs observe-only. The safer path is a configurable, race-safe abort/reset policy for confirmed stuck active runs, with startup compaction and Feishu/service mitigations handled as focused follow-ups.
