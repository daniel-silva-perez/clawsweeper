## Summary

Keep open. Current main has partial stuck-session recovery for stale lanes, plus trajectory and Feishu probe mitigations, but active embedded runs, active reply work, and active lane tasks still remain warning-only by default, and there is no configurable abort threshold on main.

## What This Changes

The issue reports a Feishu gateway outage where stuck processing sessions are detected but not recovered, with related startup compaction, trajectory growth, memory-limit, heartbeat, and PATH guardrail requests.

## Best Possible Solution

Keep this as the production outage umbrella until maintainers either land an active stuck-session abort/reset policy or deliberately choose a different recovery model, then split any remaining startup compaction, service memory tuning, Feishu probe, and PATH cleanup into narrower trackers with clear owners.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show active embedded runs, reply work, and active lane tasks remain warning-only after the stuck-session threshold. The complete Feishu outage sequence is log-backed but was not live-reproduced in this read-only review.

## Solution Assessment

No, current main is not yet the complete solution because it only releases stale lanes with no active work. The narrower maintainable direction is a diagnostics-owned, configurable, race-safe abort/reset path for confirmed stuck active runs, with the other sub-requests handled as focused follow-ups.
