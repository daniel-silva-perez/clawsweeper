## Summary

Keep open. Current main still lacks TUI subscription and routing for gateway transcript events, and open implementation PRs are explicitly paired with this bug, so closing before one lands would be premature.

## What This Changes

The issue asks for `openclaw tui --session ... --deliver` to live-tail active session messages, including inbound Telegram messages and agent replies, without relaunching or manual refresh.

## Best Possible Solution

Land a focused TUI fix that subscribes to `sessions.messages.subscribe` for the active session, unsubscribes on session switch or shutdown, handles `session.message` for the active session, and adds regression coverage for external Telegram-style transcript updates, active-run deferral, session filtering, and duplicate prevention.

## Reproduction Assessment

Yes. The issue gives a concrete command and current source inspection provides a high-confidence code-path reproduction: gateway transcript events exist, but the TUI neither subscribes to nor handles `session.message`, so external session transcript updates cannot live-tail through the TUI.

## Solution Assessment

Yes for keeping this open. The best fix path is the transcript-subscription approach already represented by #73402, rather than broadcasting external channel ingress through generic `chat` events as proposed by #45417.
