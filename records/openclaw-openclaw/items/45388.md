## Summary

Keep open. Current main has the Gateway transcript subscription/event contract and a partial non-local chat-final history reload path, but the bundled TUI still does not subscribe to or route `session.message`, so the live-tail bug remains and is already paired with open implementation work.

## What This Changes

The issue requests that `openclaw tui --session ... --deliver` live-tail the active Telegram-backed session instead of requiring manual refresh or relaunch.

## Best Possible Solution

Land a focused TUI fix, likely via #73402 or an equivalent replacement, that subscribes to the active session transcript stream, unsubscribes on session switch/shutdown, handles `session.message` only for the active session, and includes duplicate-prevention and history-overlap regression coverage.

## Reproduction Assessment

Yes. The issue gives a concrete command and current source provides a high-confidence code-path reproduction: Gateway transcript events exist, but the TUI neither subscribes to nor routes them.

## Solution Assessment

Yes for the implementation direction. The narrow maintainable fix is to use the existing `sessions.messages.subscribe` / `session.message` transcript contract rather than widening run-scoped `chat` event fanout for external channel traffic.
