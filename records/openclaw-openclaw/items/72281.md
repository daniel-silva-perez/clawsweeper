## Summary

Keep this PR open. Current main still has the unguarded PowerShell `Write-Host` wrapper while `Main` is still captured into `$installSucceeded`, so the PR is not implemented on main and remains a narrow candidate for the linked Windows installer CI failure. The diff has no concrete blocking correctness or security finding from this review, but it should be validated on the affected PowerShell/CI lane before merge.

## What This Changes

The PR changes `scripts/install.ps1` so the custom PowerShell `Write-Host` wrapper explicitly discards the inner `Microsoft.PowerShell.Utility\Write-Host` command result with `[void](...)`.

## Best Possible Solution

Review and land the one-line installer wrapper guard if the focused install-ps1 test, the `core-support-boundary` shard, and an affected Windows/PowerShell validation lane pass on the rebased PR. If validation still shows return-value pollution, keep the fix inside `scripts/install.ps1` and broaden only the installer host-output handling that is proven to leak.

## Reproduction Assessment

Yes. The linked report names the failing CI shard, and current main contains a focused PowerShell fixture that fails if `Main` returns an array or anything other than `$true`; this review did not run the full test because the task is read-only.

## Solution Assessment

Yes for the reported wrapper leak: making the custom `Write-Host` wrapper side-effect-only is the narrowest maintainable change and does not alter installer control flow. If Windows validation shows the direct npm stdout/stderr host writes are the real source, the safer follow-up is to guard those installer output sites as well rather than changing core completion behavior.
