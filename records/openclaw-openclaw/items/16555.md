## Summary

Keep open. Current main still lacks an outbound delivery-queue TTL/expiry policy: startup recovery and reconnect drains replay otherwise-eligible pending files using retry/backoff/max-retry state, not queued age, and strict messages config has no `messages.delivery` surface. The production Telegram reproduction and maintainer follow-up comments remain applicable; the related SQLite/outbox PRs are closed and unmerged rather than canonical fixes.

## What This Changes

Add configurable TTL/expiry for delivery queue messages so stale pending outbound entries are skipped, deleted, or failed instead of being replayed after gateway restart or reconnect.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: add optional strict `messages.delivery` config, expire stale pending entries before replay in both startup recovery and reconnect drains, preserve current behavior when unset, log expiry counts, update generated config docs/schema/help/labels as needed, and cover storage, policy, recovery, reconnect, and config acceptance in focused tests.

## Reproduction Assessment

Yes. Seed a delivery-queue JSON entry with an `enqueuedAt` timestamp far in the past and retry/backoff state that is otherwise eligible, then invoke startup recovery or a matching reconnect drain; current main will consider it for delivery because no age check exists.

## Solution Assessment

Yes. A direct TTL/expiry policy on the current file-backed queue is the narrowest maintainable fix for the reported stale replay behavior; the broader SQLite/outbox rewrites are useful prior art but are closed and unmerged.
