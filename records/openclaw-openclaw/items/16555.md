## Summary

Keep open. Current main still replays pending outbound delivery queue entries by retry/backoff/max-retry policy only, and the strict messages config has no delivery TTL or expiry action. The production reports and the prior maintainer review still match current code; related SQLite/outbox attempts are closed and unmerged rather than canonical fixes.

## What This Changes

Add configurable TTL/expiry for outbound delivery queue messages so stale pending entries are not replayed indefinitely after gateway restart or reconnect.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: add optional strict messages.delivery config, expire stale pending entries before replay in both startup recovery and reconnect drains, preserve default no-TTL compatibility, log expiry counts, and cover recovery, reconnect, storage, policy, and config acceptance with focused tests.
