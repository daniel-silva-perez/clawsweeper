## Summary

Keep open. Current main still lacks configurable TTL/expiry for outbound delivery queue entries: startup recovery and reconnect drains replay pending files using retry/backoff/max-retry policy only, and the strict `messages` config has no delivery TTL surface. The production repro in the discussion still matches current behavior, while the related SQLite/outbox attempts are closed and unmerged rather than canonical fixes.

## What This Changes

Request to add configurable TTL/expiry for file-backed outbound delivery queue entries so stale messages are not replayed after restarts or reconnects.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: add optional strict `messages.delivery` config, expire stale pending entries before replay in both startup recovery and reconnect drains, preserve default no-TTL compatibility, log expiry counts, and cover recovery, reconnect, storage, policy, and config acceptance with focused tests.

## Repair Work Prompt

Implement optional TTL/expiry for the existing file-backed outbound delivery queue requested by this issue. Keep the fix narrow: do not replace the queue with SQLite/outbox, do not change channel adapter contracts, and preserve the default no-TTL behavior unless maintainers choose otherwise. Add a strict `messages.delivery` config surface, likely `maxAgeMs` plus a closed expiry action, and apply the policy before replay in both `recoverPendingDeliveries` and `drainPendingDeliveries` after rereading the claimed entry. Expired entries should not be delivered; prefer a debuggable failed/quarantine path unless maintainer guidance selects a different action. Log useful expiry counts without noisy per-entry spam. Update types, Zod schema, config docs/generated schema as needed, and add focused tests for startup recovery, reconnect drain, storage/policy behavior, and config acceptance/rejection. Add a user-facing changelog entry because this adds operator config; credit `@rohan-ixlayer` for the report and `@wildhuman` for the production reproduction. Related refs: #16555, #32063, #29147, #29953, #22376.
