## Summary

Keep open. Current main still has no delivery-queue TTL/expiry policy: startup recovery and reconnect drains replay eligible pending entries by retry/backoff state, not queued age, and the strict messages config has no delivery expiry surface. The production Telegram reproduction and prior maintainer comments remain applicable; the related SQLite/outbox attempts are closed and unmerged rather than canonical fixes.

## What This Changes

Add configurable expiry for file-backed outbound delivery-queue messages so stale pending entries are skipped, deleted, or moved aside instead of flooding channels on gateway restart.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: accept optional strict messages delivery config, expire stale entries before replay in both startup recovery and reconnect drains, preserve current behavior when unset, log expiry counts, and cover storage, policy, recovery, reconnect, config, and docs/schema generation.

## Reproduction Assessment

Yes. Seed a delivery-queue JSON entry with an enqueuedAt timestamp far in the past and retryCount/backoff state that is otherwise eligible, then invoke startup recovery or a matching reconnect drain; current main will consider it for delivery because no age check exists.

## Solution Assessment

Yes. A direct file-backed queue expiry fix is the narrowest maintainable solution; the broader SQLite/outbox rewrites in the related context are useful prior art but are closed and unmerged.

## Repair Work Prompt

Implement the delivery-queue TTL/expiry behavior requested by this issue for the existing file-backed outbound queue. Keep the fix narrow: do not replace the queue with SQLite/outbox infrastructure and do not change default replay behavior unless an operator configures TTL. Add optional strict config under the messages delivery surface for max age and expiry action, choose a closed action vocabulary aligned with existing failed-entry handling, and document generated config if the public schema changes. Apply the age policy before replay in both recoverPendingDeliveries and drainPendingDeliveries, using each entry's enqueuedAt. Expired entries should not be delivered; support a safe failed/quarantine-style action and clear logs/counts. Preserve existing retry/backoff/permanent-error behavior for non-expired entries. Add focused tests for startup recovery, reconnect drain, storage/policy helpers, and strict config acceptance/rejection. Do not alter unrelated channel adapters, provider behavior, or broad message-lifecycle architecture. Related prior art: closed/unmerged #29147/#28941/#32063; use them only as context.
