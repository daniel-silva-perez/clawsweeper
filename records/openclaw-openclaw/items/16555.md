## Summary

Keep open. Current main still replays file-backed outbound delivery-queue entries based on retry count, backoff, and delivery outcome, not queued age, and there is still no `messages.delivery` TTL/expiry config surface. The issue discussion includes a concrete production reproduction on 2026.3.24 plus a workaround that avoids one Telegram restart-notification trigger but does not fix stale pending-entry replay. Related SQLite/outbox PRs that proposed TTL semantics are closed unmerged, so this remains actionable in core.

## Best Possible Solution

Keep this issue open and implement a narrow core delivery-queue expiry policy: optional `messages.delivery.maxAgeMs` plus a small closed `expireAction`, applied after claiming and re-reading entries in both startup recovery and reconnect drains. Preserve default no-TTL behavior, add clear logging, move expired entries to `failed/` by default when configured, and document the config surface with targeted regression tests.

## Clownfish Work Prompt

Implement optional TTL/expiry for the existing file-backed outbound delivery queue described in this issue. Keep the fix narrow: do not replace the queue with SQLite/outbox, do not change channel adapter contracts, and preserve default backward compatibility with no age check unless `messages.delivery.maxAgeMs` is configured.

Expected behavior:
- Add a typed/configured `messages.delivery` block, likely `maxAgeMs?: number` and a closed `expireAction` such as `move-to-failed | skip | delete` (choose names consistent with repo style; default to a debuggable failed/expired action when maxAgeMs is set).
- In `recoverPendingDeliveries`, after claiming and re-reading each current entry, compare `Date.now() - enqueuedAt` to the configured max age before max-retry/backoff delivery. Expired entries must not be delivered.
- In `drainPendingDeliveries`, apply the same expiry policy after re-reading the claimed current entry so reconnect drains cannot replay stale matched entries.
- Preserve existing max-retry, permanent-error, active-claim, delivered-marker, and backoff behavior for non-expired entries.
- Add clear log messages and tests for default no-TTL compatibility, expired startup recovery, expired reconnect drain, retry/backoff interaction, and the chosen terminal action.
- Update config help/labels/docs and add a user-facing changelog entry with reporter/context attribution. Do not add `Thanks @codex`, `Thanks @openclaw`, or `Thanks @steipete`.

## Risks / Open Questions

- A TTL that defaults on could drop legitimate delayed deliveries for users relying on current at-least-once behavior; preserve backward compatibility unless maintainers choose otherwise.
- Implementing expiry only in startup recovery would leave reconnect drains able to replay stale matched entries, so both paths need the same policy.
- Deleting expired files silently would remove debugging evidence; a failed/expired terminal action is safer unless explicitly configured otherwise.
