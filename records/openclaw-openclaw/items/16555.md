## Summary

Keep open. Current main still lacks configurable TTL/expiry for the file-backed outbound delivery queue: startup recovery and reconnect drains replay pending entries using retry/backoff/max-retry policy only, and the strict `messages` config has no `delivery.maxAgeMs` or `expireAction` surface. The production reproduction and maintainer follow-up in the discussion remain applicable; the related SQLite/outbox alternatives are closed and unmerged rather than canonical fixes.

## What This Changes

Issue requests configurable TTL/expiry for outbound delivery queue messages so stale persisted messages are not replayed after gateway restarts.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: add optional strict `messages.delivery` config, expire stale pending entries before replay in both startup recovery and reconnect drains, keep default no-TTL behavior, log expiry counts, and cover recovery, reconnect, storage, policy, and config acceptance with focused tests.
