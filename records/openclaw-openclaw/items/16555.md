## Summary

Keep open. Current main still has a file-backed outbound delivery queue with no configurable TTL/expiry policy for stale queued messages. Startup recovery and reconnect drains replay eligible entries based on max retries, backoff, and permanent-error handling, not queued age. The linked SQLite/outbox PRs that proposed TTL/expiry are closed unmerged, and the latest maintainer review comment already reached the same conclusion against recent main.

## Best Possible Solution

Keep this issue open and implement a bounded delivery-queue expiry policy in core. The likely focused fix is optional `messages.delivery` config/types/schema/help/docs, age checks in both `recoverPendingDeliveries` and `drainPendingDeliveries` after re-reading the claimed entry and before replay, a conservative terminal action such as moving expired entries to `failed/` with clear logging, and regression tests covering default backward compatibility, stale startup entries, reconnect drains, retry/backoff interaction, and delivered-marker cleanup.
