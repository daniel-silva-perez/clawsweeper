## Summary

Keep open. Current main still lacks an outbound delivery-queue TTL: startup recovery and reconnect drains replay pending file-backed entries based on retry/backoff/max-retry state, not queued age, and strict messages config has no delivery expiry surface. The production reproduction and recent maintainer follow-up remain applicable, while the SQLite/outbox PRs that claimed this work are closed and unmerged.

## What This Changes

The issue requests configurable TTL/expiry for file-backed outbound delivery-queue entries so stale queued messages are not replayed on gateway restart or reconnect drain.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: accept optional strict messages delivery config, expire stale entries before replay in both startup recovery and reconnect drains, preserve current no-TTL behavior when unset, log expiry counts, and cover recovery, reconnect, storage, policy, and config acceptance with focused tests.

## Reproduction Assessment

Yes. Seed an old delivery-queue JSON entry with an enqueuedAt timestamp far in the past, then invoke startup recovery or a matching reconnect drain; current main will still consider it for replay because only retry/backoff/max-retry gates are applied.

## Solution Assessment

Yes, keeping this open and fixing the file-backed queue directly is the narrowest maintainable path. A SQLite/outbox rewrite is not required to solve this issue and the related rewrite PRs are closed and unmerged.
