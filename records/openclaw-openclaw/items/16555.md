## Summary

Keep open. Current main still replays outbound delivery-queue entries by retry/backoff/permanent-error policy only; it has no queued-age TTL, no `messages.delivery.maxAgeMs`, and no expiry action. The production repro and the latest maintainer review in the issue remain applicable, while the related SQLite/outbox TTL PRs are closed and unmerged rather than canonical fixes.

## Best Possible Solution

Implement a narrow core outbound delivery-queue expiry policy: optional `messages.delivery.maxAgeMs` plus a closed `expireAction`, applied after claim/re-read in both `recoverPendingDeliveries` and `drainPendingDeliveries`. Preserve default no-TTL behavior, log expired counts, prefer a failed/quarantine path when configured, and update config types/schema/help/docs plus focused recovery, reconnect-drain, storage/policy, and config tests.

## Repair Work Prompt

Preserve the default current behavior when `maxAgeMs` is unset. ## Risks / Open Questions

- Expiry changes core outbound delivery semantics: defaults must preserve current no-TTL behavior so existing pending queues are not silently dropped on upgrade.
- The policy must run after claim/re-read in both startup recovery and reconnect drains; adding it only to startup recovery would leave reconnect message dumps possible.
- `skip` can leave old files indefinitely, while `delete` loses debugging evidence; tests should pin each action and logging behavior.
- Do not revive the broad closed SQLite/outbox PRs as part of this narrow fix; that would mix a focused TTL request with a larger architecture change.
