## Summary

Keep open. Current main still has the file-backed outbound delivery queue without any queued-age TTL or expiry action in startup recovery, reconnect drain, storage cleanup, or strict messages config. The issue also has a production reproduction and maintainer follow-up context, while the related SQLite/outbox attempts in the provided timeline are closed and unmerged rather than canonical fixes.

## What This Changes

Add configurable TTL and expiry handling for persisted outbound delivery queue messages so stale entries are not replayed after gateway restart.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: accept optional strict messages delivery config, evaluate stale entries before replay in both startup recovery and reconnect drains, preserve current retry/backoff behavior for non-expired entries, keep unset config backward-compatible, log expiry counts, and cover the policy with focused recovery, storage, reconnect, and config tests.

## Repair Work Prompt

Implement the TTL/expiry request for the current file-backed outbound delivery queue. Add optional strict messages delivery config, for example `messages.delivery.maxAgeMs` plus a closed `expireAction` enum that follows repo naming conventions. Preserve the current default behavior when the config is unset: no age check and no silent deletion. Apply expiry before replay in both `recoverPendingDeliveries` and `drainPendingDeliveries`, after re-reading/claiming the entry so the queue file remains the source of truth. For expired entries, prefer a debuggable failed/quarantine path unless the user explicitly configures delete/skip; log concise expiry counts or per-entry summaries without leaking message bodies. Keep existing retry, backoff, permanent-error, active-claim, skipQueue, and delivered-marker semantics for non-expired entries. Add focused tests for startup recovery, reconnect drain, storage/failed movement, policy helper behavior, and strict config acceptance/rejection. Update generated config docs/schema if needed and add a user-facing changelog entry crediting the reporter/repro context, e.g. Thanks @rohan-ixlayer and @wildhuman. Do not replace the file queue with SQLite/outbox, add dependencies, change channel adapters, or alter default behavior for users without the new config.
