## Summary

Keep open. Current main still lacks a configurable outbound delivery-queue TTL: startup recovery and reconnect drains replay pending JSON entries using retry/backoff/max-retry policy only, while the strict messages config has no delivery expiry surface. The production reports still match current behavior, and the related SQLite/outbox attempts are closed and unmerged rather than canonical fixes.

## What This Changes

The issue requests optional TTL/expiry handling for outbound delivery queue entries so stale persisted messages are not replayed after gateway restart.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: add optional strict messages.delivery config, expire stale pending entries before replay in both startup recovery and reconnect drains, preserve default no-TTL compatibility, log expiry counts, and cover recovery, reconnect, storage, policy, and config acceptance with focused tests.

Add a strict `messages.delivery` config surface with a finite nonnegative `maxAgeMs` and a small closed expiry action vocabulary agreed from the existing issue context; prefer a safe failed/quarantine style over silent deletion unless the local patterns clearly point elsewhere.
