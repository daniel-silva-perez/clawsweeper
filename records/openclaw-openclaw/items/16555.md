## Summary

Keep open. Current main still lacks a delivery-queue TTL/expiry path: startup recovery and reconnect drains replay pending file-backed entries based on max-retry and backoff state, not queued age, and strict messages config has no delivery expiry surface. The production reproduction remains applicable, and the SQLite/outbox rewrite attempts in the related context are closed and unmerged rather than canonical fixes.

## What This Changes

Request adds configurable TTL/expiry handling for file-backed outbound delivery queue entries so stale messages are skipped, deleted, or moved to failed during recovery.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: accept optional strict messages delivery config, expire stale entries before replay in both startup recovery and reconnect drains, preserve current no-TTL behavior when unset, log expiry counts, and cover recovery, reconnect, storage, policy, config, and generated docs/schema surfaces.

## Reproduction Assessment

Yes. Seed a delivery-queue JSON entry with an enqueuedAt timestamp far in the past, then invoke startup recovery or a matching reconnect drain; current main still considers it for replay when retry/backoff gates allow it.

## Solution Assessment

Yes. A direct file-backed queue expiry fix is the narrowest maintainable path; the broader SQLite/outbox rewrites that attempted similar behavior are closed and unmerged, so they should be treated as prior art rather than the active solution.
