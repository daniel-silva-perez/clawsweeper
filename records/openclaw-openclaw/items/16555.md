## Summary

Keep open. Current main still lacks configurable TTL/expiry for the outbound delivery queue: both startup recovery and reconnect draining replay pending queue entries using retry/backoff policy only, while the strict messages config has no `messages.delivery.maxAgeMs` or `expireAction` surface. The production repro and prior maintainer review remain applicable, and the related SQLite/outbox PRs in the discussion are closed and unmerged rather than canonical fixes.

## What This Changes

The issue requests configurable TTL/expiry handling for persisted outbound delivery queue messages so stale entries can be skipped, deleted, or moved aside instead of replayed after gateway restart or reconnect.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: add a strict, documented optional `messages.delivery` config, expire old entries before replay in both startup recovery and reconnect drains, keep the default no-TTL behavior, log expiry counts, and cover the policy with focused recovery, reconnect, storage, and config tests.
