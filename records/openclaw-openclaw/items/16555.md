## Summary

Keep open. Current main still has file-backed outbound delivery recovery and reconnect drains, but they only enforce retry count/backoff/permanent-error handling; they do not expire by queued age and there is still no `messages.delivery` TTL/expiry config surface. The March production repro and latest maintainer comment remain applicable, and the related SQLite/outbox TTL PRs are closed and unmerged rather than canonical fixes.

## Best Possible Solution

Implement a narrow core delivery-queue expiry policy: optional `messages.delivery.maxAgeMs` plus a closed `expireAction`, applied after claim/re-read in both `recoverPendingDeliveries` and `drainPendingDeliveries`. Preserve default no-TTL behavior, log expired counts, prefer moving expired entries to `failed/` when configured, and update config types/schema/help/docs plus focused tests.

## Clownfish Work Prompt

Implement the delivery queue TTL/expiry feature for this issue as a narrow core change. Add an optional `messages.delivery` config block with `maxAgeMs?: number` and a small closed `expireAction` set; preserve default no-TTL behavior unless the operator configures `maxAgeMs`. Apply the expiry policy after claiming and re-reading queue entries in both startup `recoverPendingDeliveries` and reconnect/plugin `drainPendingDeliveries`, so stale entries are not replayed through either path. Prefer a debuggable failure action such as moving expired entries under `delivery-queue/failed/` when configured; do not revive the broader SQLite/outbox rewrites from the closed PRs. Update config types, strict zod schema, labels/help/docs/generated config surfaces as required, and add focused tests for startup recovery, reconnect drain, default backwards compatibility, and configured expiration. ## Risks / Open Questions

- A default TTL could discard legitimate delayed delivery after a long outage, so a narrow fix should preserve current behavior unless `messages.delivery.maxAgeMs` is explicitly configured.
- Applying expiry only to startup recovery would leave reconnect drains able to replay old entries; both recovery paths need the same policy.
- Moving or deleting expired entries must not disturb the existing two-phase `.delivered` cleanup that prevents replay after successful delivery.
