## Summary

Keep open. Current main still lacks outbound delivery-queue TTL/expiry: startup recovery and reconnect draining replay pending file-backed entries based on retry/backoff/max-retry state, not queued age, and strict messages config has no delivery expiry surface. The production reproduction and recent maintainer follow-up remain applicable, while the related SQLite/outbox PRs in the provided context are closed and unmerged.

## What This Changes

The issue requests an optional TTL/expiry policy for outbound delivery queue entries so stale persisted messages are not replayed after gateway restart or reconnect.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: accept optional strict messages delivery config, expire stale entries before replay in both startup recovery and reconnect drains, preserve current no-TTL behavior when unset, log expiry counts, and cover recovery, reconnect, storage, policy, config, and docs/schema surfaces with focused tests.

## Reproduction Assessment

Yes. Seed a delivery-queue JSON entry with an enqueuedAt timestamp far in the past, then invoke startup recovery or a matching reconnect drain; current main will still consider it for replay because only retry/backoff/max-retry gates are applied.

## Solution Assessment

Yes. Keeping this open for a direct file-backed queue expiry fix is the narrowest maintainable path; a SQLite/outbox rewrite is not required to solve the observable stale-replay bug and the related rewrite PRs are closed and unmerged.
