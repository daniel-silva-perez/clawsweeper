## Summary

Keep open. Current main and the latest release still lack the requested outbound delivery-queue TTL: recovery and reconnect drains use retry/backoff/max-retry rules, not queued age, and the strict messages config has no delivery expiry surface. The production reproduction and recent maintainer follow-up remain applicable, while the related SQLite/outbox PRs are closed and unmerged.

## What This Changes

Add configurable TTL/expiry handling for file-backed outbound delivery queue entries so stale pending messages are not replayed on gateway restart or reconnect drain.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: accept optional strict messages delivery config, expire stale entries before replay in both startup recovery and reconnect drains, preserve current no-TTL behavior when unset, log expiry counts, and cover recovery, reconnect, storage, policy, and config acceptance with focused tests.

## Repair Work Prompt

Implement a narrow fix for stale outbound delivery-queue replay in openclaw/openclaw. The observable bug is that file-backed entries under the outbound delivery queue can be replayed hours or days later on gateway startup or reconnect drain because recovery only checks retry/backoff/max-retry state and never compares Date.now() against entry.enqueuedAt.

Expected boundary:
- Work in the existing file-backed outbound delivery queue; do not migrate to SQLite/outbox, do not require channel adapter changes, and do not change unrelated session-delivery queues.
- Add optional messages delivery config in the strict config/type/schema surface, preserving backward compatibility when unset. Use a closed, documented expiry action set; the issue proposes skip/delete/move-to-failed, with move-to-failed as the conservative debugging-friendly action.
- Apply the same expiry policy before replay in recoverPendingDeliveries and drainPendingDeliveries after re-reading the current queue entry.
- Keep non-expired retry/backoff/max-retry behavior unchanged.
- Ensure storage helpers can support the chosen action safely, including failed/ placement and suffix/logging if needed.
- Log an aggregate or clear per-entry summary when entries expire, without logging message bodies or secrets.
- Update generated config docs/schema only if the config generator requires it, and add a user-facing changelog entry with the original reporter/context attribution required by repo policy.
