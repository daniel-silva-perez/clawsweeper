## Summary

Keep this PR open. Current main still has single-target cron delivery, and the PR is useful source work, but it is not mergeable yet because the latest branch still leaves webhook fan-out semantics and session-bound `channel: "last"` routing unresolved.

## What This Changes

The PR adds optional cron `delivery.additionalTargets` fan-out across cron types, gateway schema, normalization, runtime delivery dispatch, cron edit CLI flags/tests, and lazy channel-runtime access in `commands-session.ts`.

## Best Possible Solution

Land one canonical, documented cron fan-out implementation that keeps schema, normalization, delivery planning, runtime dispatch, trace/status reporting, CLI/UI surfaces, docs, changelog, and regression tests aligned, with explicit behavior for announce, webhook, no-delivery, primary-success gating, `channel: "last"`, and per-target failure isolation.

## Reproduction Assessment

Yes for the review findings. A static reproduction is enough: webhook delivery accepts `additionalTargets` but never reaches a direct-send `sendOccurred` path, and `channel: "last"` additional targets are resolved with the cron run session key rather than the job delivery session context.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best mergeable solution until it either supports or rejects webhook fan-out and resolves `last` targets from the correct job session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/cron/isolated-agent/run.ts:772`
  - body: `additionalTargets` is added to the shared delivery schema, so webhook-mode cron jobs can accept and persist extra targets. The runtime fan-out only runs when `deliveryResult.sendOccurred === true`, but webhook delivery is posted later by the gateway notification path rather than direct cron delivery, so webhook jobs silently skip all additional targets. Reject the field for unsupported modes or trigger fan-out from the webhook completion path too.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets can use `channel: "last"`, but this helper passes the cron run's `agentSessionKey` into `resolveDeliveryTarget`. Primary delivery resolves against the job's configured session context, so session-bound jobs can route fan-out to the wrong channel/thread or fail closed. Pass the job delivery session key into this helper and cover it with a regression test.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces routing-sensitive outbound fan-out, and one code path can resolve `channel: "last"` additional targets against the wrong session.

Concerns:

- **[medium] Fan-out can resolve the wrong last recipient:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: `deliverToAdditionalTargets` passes the cron run session key to `resolveDeliveryTarget` instead of the job delivery session context. For session-bound cron jobs, `channel: "last"` can route cron output to an unintended channel or thread.
