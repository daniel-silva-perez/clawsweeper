## Summary

Keep this PR open. Current main still has single-target cron delivery, so the feature is not implemented, but the branch should not merge as-is because webhook/no-delivery modes can accept additional targets without any fan-out path and session-derived `last` targets are resolved against the wrong cron session context.

## What This Changes

The branch adds cron `delivery.additionalTargets` types, gateway schema support, normalization, CLI edit flags, isolated-run fan-out dispatch and tests, plus lazy channel-runtime lookups in `commands-session.ts`.

## Best Possible Solution

Land one canonical cron fan-out implementation that keeps schema, normalization, delivery planning, runtime dispatch, trace/status reporting, CLI/UI/docs/changelog, and tests aligned, with explicit behavior for announce, webhook, no-delivery, primary-success gating, `channel: "last"`, and per-target failure isolation.

## Reproduction Assessment

Yes for the review findings. Static inspection is sufficient: the PR accepts `additionalTargets` in shared delivery schema while fan-out only runs after a direct primary send, and it resolves additional `last` targets with `params.agentSessionKey` instead of the job delivery session key.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best mergeable solution until it either rejects or implements non-announce fan-out, resolves additional targets from the same session context as primary delivery, and is reconciled with current main's delivery-plan structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/cron/isolated-agent/run.ts:763-789`
  - body: `additionalTargets` is added to shared delivery properties, so webhook and no-delivery jobs can accept and persist extra targets. The runtime fan-out only runs when `deliveryResult.sendOccurred === true`, which webhook delivery never sets because webhook completion is posted by the gateway notification path. Restrict the field to supported modes or trigger fan-out from the webhook completion path too.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:914-920`
  - body: Additional targets can use `channel: "last"`, but this helper resolves them with the cron run's `agentSessionKey`. Primary delivery resolves against the job's configured session key, so session-bound jobs can route fan-out to the wrong conversation or fail unexpectedly. Pass the job delivery session key into this helper and cover it with a regression test.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces routing-sensitive outbound fan-out and one path can resolve `channel: "last"` additional targets against the wrong session.

Concerns:

- **[medium] Fan-out may resolve the wrong last recipient:** `src/cron/isolated-agent/delivery-dispatch.ts:914`
  - body: `deliverToAdditionalTargets` resolves additional targets using the cron run session key. For session-bound jobs using `channel: "last"`, that can send cron output to an unintended channel/thread instead of the job's preserved delivery context.
