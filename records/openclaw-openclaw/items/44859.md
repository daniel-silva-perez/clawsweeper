## Summary

Keep this PR open, but it is not merge-ready. Current main still has single-target cron delivery, so the feature is not obsolete; however the branch accepts `additionalTargets` for delivery modes that never run the fan-out path and still resolves `channel: "last"` additional targets from the cron run session rather than the job delivery session.

## What This Changes

This PR adds cron `delivery.additionalTargets` types, gateway schema support, normalization, CLI edit flags, isolated-run fan-out dispatch and tests, plus lazy channel-runtime lookups in `commands-session.ts`.

## Best Possible Solution

Land one canonical, documented cron fan-out implementation that keeps schema, normalization, delivery planning, runtime dispatch, trace/status reporting, CLI/UI surfaces, docs, changelog, and regression tests aligned, with explicit behavior for announce, webhook, no-delivery, primary-success gating, `channel: "last"`, and per-target failure isolation.

## Reproduction Assessment

Yes for the review findings. Static inspection is sufficient: webhook and none deliveries validate `additionalTargets` but cannot satisfy the direct-send fan-out gate, and `channel: "last"` additional targets are resolved with `params.agentSessionKey` instead of the primary job delivery session key.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the narrowest maintainable solution until it rejects or implements non-announce fan-out, passes the correct session context into additional-target resolution, and rebases onto current main's cron delivery-plan structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject or implement non-announce fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: Adding `additionalTargets` to the shared delivery properties makes webhook and none deliveries accept and persist extra targets, but the runtime fan-out only runs after a direct primary chat send. Webhook delivery is posted later by the gateway notification path, so configured extra targets are silently skipped. Restrict the field to supported modes or trigger fan-out from the webhook completion path too.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets can use `channel: "last"`, but this call resolves them with the cron run's `agentSessionKey`. Primary delivery resolves against the job's configured session key, so session-bound jobs can route fan-out to the wrong conversation or fail unexpectedly. Pass the job delivery session key into this helper and cover it with a regression test.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces routing-sensitive outbound fan-out and one path can resolve `channel: "last"` additional targets against the wrong session.

Concerns:

- **[medium] Fan-out can resolve the wrong last recipient:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: `deliverToAdditionalTargets` uses the cron run session key when resolving additional targets. For session-bound jobs using `channel: "last"`, that can route cron output to an unintended channel/thread or fail closed instead of using the job's preserved delivery context.
