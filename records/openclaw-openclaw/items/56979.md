## Summary

Keep this issue open. Current main has auth cooldown and cooldown-probe throttling, but repeated non-probe `model-fallback/decision` warnings are still emitted directly, so the reported log-spam problem is not clearly fixed.

## What This Changes

Reduce repeated model fallback decision logging when an auth provider token expires and fallback/cooldown handling runs repeatedly.

## Best Possible Solution

Add bounded duplicate coalescing for repeated model fallback decision warnings, keyed by the stable decision/requested/candidate/reason/next/error fingerprint, emitting the first event and periodic summary/count updates while preserving structured fallback-step data for diagnostics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a cooldowned auth profile with model fallbacks and call `runWithModelFallback` repeatedly; current main reaches `logModelFallbackDecision` on each run and has no duplicate suppression for normal `model-fallback` warnings.

## Solution Assessment

No, current main is not yet the best solution because it handles auth cooldown state but still logs each fallback decision. The narrowest maintainable fix is log coalescing in the model-fallback observability path, not a change to provider fallback policy.
