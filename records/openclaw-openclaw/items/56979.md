## Summary

Keep open. Current main has auth cooldown and cooldown-probe throttling, but the reported `model-fallback/decision` log spam is still not coalesced: the decision observer emits a warning for every fallback decision, and existing tests cover retry/probe throttling rather than repeated warning suppression.

## What This Changes

The issue asks to reduce repeated model-fallback/auth-cooldown warning logs when an auth provider token expires and model fallback runs repeatedly.

## Best Possible Solution

Add bounded, test-covered warning coalescing around fallback decision logging, keyed by stable decision/provider/model/reason/error fingerprint and scoped to preserve first-occurrence diagnostics plus periodic or summarized repeats without changing fallback selection or auth cooldown behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to run `runWithModelFallback` repeatedly while the selected provider’s auth profiles are in cooldown and fallbacks are configured; current main will call `logModelFallbackDecision` for each decision and that function always warns.

## Solution Assessment

Yes, a narrow logging coalescer is the best fix direction. Changing auth cooldown or fallback routing would be broader and riskier, because current cooldown/probe behavior is already documented and tested separately from warning spam.
