## Summary

Keep open. Current main still lacks the proposed devcontainer, benchmark workflow, and release SBOM workflow, so the PR is not obsolete; however, the latest patch still has concrete workflow and supply-chain defects that should block merge until maintainer revision.

## What This Changes

The PR adds a Dev Container configuration plus GitHub Actions workflows for hyperfine-based performance benchmarking and release-time SBOM generation/attachment.

## Best Possible Solution

Land only a revised, focused implementation that reuses the repository setup action and startup benchmark scripts, generates SBOMs from the shipped tarball or production dependency set through one permissions-minimal upload path, and splits the devcontainer from release/benchmark workflow policy if needed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Verify hyperfine before installing it with sudo:** `.github/workflows/benchmark.yml:31-32`
  - body: The workflow downloads a `.deb` from GitHub releases and immediately runs `sudo dpkg -i` on it. Add a pinned checksum/signature check or install hyperfine from a trusted package source so a compromised release asset cannot execute arbitrary code in CI.
  - confidence: 0.89
- **[P2] Generate the SBOM from the release artifact:** `.github/workflows/sbom.yml:24-28`
  - body: This job installs the full workspace and scans `path: .`, so the SBOM attached to a release can include dev/test tooling and omit the exact npm tarball boundary consumers receive. Scan the prepared release package or a production dependency set instead.
  - confidence: 0.86
- **[P2] Use one SBOM release upload path:** `.github/workflows/sbom.yml:26-30`
  - body: `anchore/sbom-action` uploads artifacts and release assets by default on release events, but the job also uploads the same file as an artifact and attaches it with `softprops/action-gh-release`. Disable the Anchore upload modes or remove the duplicate steps and grant the permissions required by the chosen path.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds CI/release code-execution and release-asset paths with concrete supply-chain and permissions concerns.

Concerns:

- **[medium] Unverified privileged package install:** `.github/workflows/benchmark.yml:31`
  - body: The benchmark job downloads a Hyperfine `.deb` and installs it with sudo without verifying a checksum or signature, so a compromised release asset or delivery path could execute code in CI.
  - confidence: 0.9
- **[medium] SBOM target does not match shipped artifact:** `.github/workflows/sbom.yml:24`
  - body: The release job scans the full checkout after a full dependency install, which can publish an SBOM for development inputs rather than the actual npm release tarball or production dependency set.
  - confidence: 0.86
- **[low] Duplicate release upload behavior:** `.github/workflows/sbom.yml:26`
  - body: Anchore documents artifact and release-asset uploads as default behavior and requires `actions: read` plus `contents: write` for release uploads; this workflow also attaches the same SBOM with a second action, so the permissions and upload ownership need to be made explicit. ([github.com](https://github.com/anchore/sbom-action))
  - confidence: 0.78
