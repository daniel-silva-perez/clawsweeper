## Summary

Keep open. Current main and the latest tagged release still start the Android node as a targetSdk 36 `dataSync` foreground service and call `ServiceCompat.startForeground` without handling `ForegroundServiceStartNotAllowedException` or Android 15+ foreground-service timeout callbacks, so the reported crash path is still live.

## What This Changes

Fix the Android companion app crash when `NodeForegroundService` starts a `dataSync` foreground service after the Android foreground-service quota is exhausted.

## Best Possible Solution

Update the Android foreground-service lifecycle so quota exhaustion cannot crash the app: handle `startForeground` denial, implement the Android 15+ timeout callback, stop or demote the service cleanly, avoid restart loops, and document or test any remaining background-connection limitation.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: target SDK 36, `dataSync` foreground-service type, exhausted dataSync quota, and an unguarded `startForeground` call during `NodeForegroundService.onCreate`; final proof should use an Android 15/16 device or emulator with foreground-service time limits forced low.

Unclear on the exact final policy choice, but the suggested fix area is correct. The narrow maintainable fix belongs in the Android service lifecycle and tests, not in gateway connectivity code.
