## Summary

Keep open. Current main still starts the Android node as a targetSdk 36 `dataSync` foreground service and calls `ServiceCompat.startForeground` without handling dataSync quota exhaustion or implementing the Android 15+ timeout callback, so the reported crash path is still live.

## What This Changes

The issue reports Android app crashes when `NodeForegroundService` calls `startForeground` after Android reports the `dataSync` foreground-service time limit is exhausted.

## Best Possible Solution

Fix the Android app’s foreground-service lifecycle so the gateway connection does not crash when `dataSync` quota is exhausted: handle `startForeground` failures, implement Android 15+ timeout demotion/stop behavior, avoid restart loops, and update docs/tests for whatever background-connection behavior remains supported.

## Repair Work Prompt

Fix the Android crash reported in openclaw/openclaw#64903: `NodeForegroundService` can crash on Android 15/16 targetSdk 36 when `ServiceCompat.startForeground` is called for `dataSync` after the foreground-service quota is exhausted (`ForegroundServiceStartNotAllowedException: Time limit already exhausted for foreground service type dataSync`). Keep the change scoped to the Android app unless a docs/changelog update is required. Do not change core gateway protocol, plugin code, package IDs, release versions, or broaden permissions unless Android platform docs require it. Read the Android foreground-service timeout docs first. Implement graceful handling around `startForeground`/`startForegroundService` failures, add Android 15+ `Service.onTimeout(int, int)` handling for dataSync quota timeout, prevent restart/crash loops, and prefer stopping/demoting/surfacing reconnect status over repeatedly starting a quota-exhausted dataSync foreground service. Preserve or deliberately update Talk Mode microphone escalation behavior and its docs/tests. Add focused tests around service type selection and the new failure/timeout handling; update `docs/platforms/android.md` if background connection behavior changes. Add a changelog entry with appropriate reporter/contributor credit if the fix is user-facing.
