## Summary

Keep this issue open. Current main supports `delivery.accountId` as an optional cron delivery field and has adjacent validation for ambiguous channels, but it does not implement `openclaw cron lint`, startup lint warnings/fail-closed behavior, or add/edit validation requiring `delivery.accountId` for announce jobs on multi-account channel configs. The linked #69055 is a companion logging/forensics request, not a duplicate or replacement.

## Best Possible Solution

Keep this issue open and implement a cron-owned validation surface: add a reusable lint helper that detects announce-mode jobs missing `delivery.accountId` when the resolved channel has multiple configured accounts; wire `openclaw cron lint --warn-only`, gateway startup warnings, and cron add/edit mutation validation; document the migration path; and add focused unit/integration tests. Preserve explicit `accountId: "default"` for jobs intentionally using the default bot.
