## Summary

Keep this issue open. Current main still has no `openclaw cron lint` command, no startup lint/fail-closed path, and no create/update validation requiring `delivery.accountId` for announce-mode jobs on multi-account channel configs. The existing code only treats `accountId` as an optional delivery field and validates adjacent channel ambiguity. The cross-linked #69055 is a companion logging/forensics request, and open PR #69562 covers invalid explicit account IDs rather than the missing-account lint rule requested here.

## Best Possible Solution

Keep this issue open and implement a cron-owned validation/lint surface: add a reusable helper that detects announce-mode cron jobs missing `delivery.accountId` when the resolved delivery channel has multiple configured accounts; wire `openclaw cron lint --warn-only`, gateway startup warnings with a staged fail-closed path, and cron add/edit validation; document the migration path in the public cron docs; and add focused tests for single-account no-op, multi-account findings, explicit `accountId: "default"`, CLI exit codes, gateway startup behavior, and mutation rejection. Review #69562 as adjacent invalid-account work, but do not treat it as closing this missing-account lint request unless it grows to cover the full rule.
