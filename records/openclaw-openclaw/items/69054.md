## Summary

Keep this issue open. Current main still has optional `delivery.accountId` plumbing and adjacent ambiguous-channel validation, but it does not add `openclaw cron lint`, does not require an explicit account for announce jobs on multi-account channels, and does not warn before cron scheduler startup. The linked #69055 is complementary forensic logging work, not a replacement for this config-time validation.

## What This Changes

Requests a cron lint command plus gateway startup and cron add/edit validation requiring explicit `delivery.accountId` for announce-mode cron jobs targeting multi-account channel configurations.

## Best Possible Solution

Implement a cron-owned validation surface: reusable lint findings for announce jobs missing `delivery.accountId` only when the resolved channel has multiple configured accounts, a read-only `openclaw cron lint` command with the requested exit codes and `--warn-only`, startup warnings before scheduling, add/update rejection for new invalid jobs, docs for migration, and focused tests. Keep explicit `accountId: "default"` valid for intentional default-bot delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to call `cron.add` or `cron.update` with `delivery.mode: "announce"`, a configured multi-account channel, and no `delivery.accountId`; current validation reaches `assertValidCronAnnounceDelivery`, checks only channel ambiguity, and then accepts the job.

## Solution Assessment

Yes. The requested shared lint/validation helper is the narrow maintainable fix if it is implemented through cron and channel plugin contracts rather than provider-specific branches. The safer rollout is warning/lint/add-edit validation first, with fail-closed startup deferred to explicit maintainer approval after migration.
