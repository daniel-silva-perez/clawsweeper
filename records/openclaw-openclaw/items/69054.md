## Summary

Keep this issue open. Current main still has the optional `delivery.accountId` plumbing and adjacent ambiguous-channel validation, but it does not add `openclaw cron lint`, does not require `delivery.accountId` for announce jobs on multi-account channels, and does not warn or fail at cron scheduler startup for existing jobs. The linked #69055 is a companion logging/forensics request, not a duplicate of this config-time validation request.

## What This Changes

Requests a cron lint command plus gateway startup and cron add/edit validation requiring explicit `delivery.accountId` for announce-mode cron jobs targeting multi-account channel configurations.

## Best Possible Solution

Implement this as a cron-owned validation surface: a reusable lint helper that reports announce jobs missing `delivery.accountId` only when the resolved channel has multiple configured accounts, a read-only CLI command with the requested exit codes and `--warn-only`, startup warnings before scheduling, add/update rejection for new invalid jobs, migration docs, and focused tests. Keep explicit `accountId: "default"` valid for jobs intentionally using the default bot.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to invoke `cron.add` or `cron.update` with `delivery.mode: "announce"`, `delivery.channel` set to a configured channel that has multiple accounts, and no `delivery.accountId`; current source shows validation reaches `assertValidCronAnnounceDelivery`, which checks only channel ambiguity and then accepts the job.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if implemented in shared cron validation rather than per-channel special cases. The safer rollout is lint and warning-only startup first, with fail-closed startup behavior deferred until the documented deprecation window is explicitly approved.
