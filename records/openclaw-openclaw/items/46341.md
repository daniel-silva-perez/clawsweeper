## Summary

Keep open. Current main still has the reported recurring cron/every asymmetry: one-shot jobs classify transient errors and retry on short backoff, while recurring error handling chooses the later of the natural schedule and the backoff floor without transient classification. The issue is also paired with open, unmerged PR #46423 using closing syntax, so it should remain open until that PR or a narrower replacement lands.

## What This Changes

The issue requests that recurring cron/every jobs classify transient execution errors and retry with backoff instead of waiting for the next natural schedule slot.

## Best Possible Solution

Land a focused scheduler change, likely by reviewing/refining #46423 or replacing it with a narrower PR, that classifies recurring cron/every execution errors, schedules retryable transient failures on the short backoff path, preserves natural-schedule behavior for permanent errors, and adds regression coverage plus docs/type alignment.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: run a daily cron or every job whose result.error contains a transient token such as network or timeout; applyJobResult increments consecutiveErrors but then chooses the natural next run over endedAt + 30s for recurring jobs.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, but the issue should not be closed before the linked implementation PR merges. The implementation should explicitly document the recurring retry policy rather than accidentally broadening the one-shot config contract.
