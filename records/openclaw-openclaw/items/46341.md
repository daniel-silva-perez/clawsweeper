## Summary

Keep open. Current `main` still has the reported asymmetry: one-shot `at` jobs classify transient cron errors and retry on a short backoff path, while recurring `cron`/`every` errors still apply backoff only as a lower bound against the natural next schedule. The issue is also paired with open PR #46423 using closing syntax, so the right path is to review, refresh, merge, or close that PR rather than close this issue early.

## Best Possible Solution

Keep this issue open and use PR #46423 as the implementation candidate. The focused fix should live in `src/cron/service/timer.ts`: classify recurring `cron` and `every` errors with the existing transient-error helper, schedule transient failures on the short backoff path, preserve natural-schedule behavior for permanent errors, and add regression coverage for long-period recurring jobs plus restart/backoff interactions. Update public config/docs wording if recurring jobs intentionally adopt `cron.retry` settings.
