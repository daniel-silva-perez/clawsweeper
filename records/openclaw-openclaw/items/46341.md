## Summary

Keep this issue open. Current main still treats recurring cron errors as a generic backoff floor: it computes the natural next schedule and then picks the later of the natural run and backoff, so a daily or hourly recurring job can still wait for the next scheduled run after a transient error. The same-author PR #46423 remains open and explicitly proposes the missing transient-specific retry behavior, so this issue should stay open until that PR is reviewed/merged or an equivalent fix lands.

## Best Possible Solution

Keep the issue open and resolve it by reviewing/landing PR #46423 or applying an equivalent fix in `src/cron/service/timer.ts` that checks transient errors for recurring cron/every jobs, schedules the retry at the configured backoff instead of the natural schedule, and adds regression coverage for both transient and permanent recurring failures.
