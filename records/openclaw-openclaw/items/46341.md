## Summary

Keep open. Current main still applies recurring cron error backoff as a lower bound against the natural next schedule without transient-error classification, so hourly/daily recurring jobs can still wait for their natural slot after a retryable network/rate-limit/server error. The issue is also paired with open PR #46423, which is configured to close this issue on merge, so the right path is to review or refresh that PR rather than close this issue early.

## Best Possible Solution

Keep this issue open and use #46423 as the implementation candidate. The focused fix should live in `src/cron/service/timer.ts`: classify recurring `cron` and `every` errors for built-in transient classes, schedule transient failures on the short backoff path, preserve natural-schedule behavior for permanent errors, and add regression tests for long-period `cron` and `every` jobs plus restart/backoff interactions. Update docs/config wording if recurring jobs intentionally adopt `cron.retry` settings.
