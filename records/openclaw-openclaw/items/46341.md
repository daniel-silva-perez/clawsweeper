## Summary

Keep open. Current main still treats recurring cron/every failures as ordinary recurring backoff, using the later of the natural next schedule and the backoff floor without transient-error classification; the issue is also paired with open, unmerged PR #46423, which uses closing syntax for this issue, so it should stay open until that PR or a narrower replacement lands.

## What This Changes

Recurring cron/every jobs should classify transient errors and retry on the short backoff path instead of waiting for the next natural schedule slot.

## Best Possible Solution

Land a focused scheduler fix, likely by reviewing/refining #46423 or replacing it with a narrower PR, that classifies recurring cron/every errors, schedules retryable transient failures on the short backoff path, preserves natural-schedule behavior for permanent errors, and adds regression coverage for long-period cron/every jobs plus docs alignment.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: run a daily cron or every job whose result.error contains a transient token such as network or timeout; applyJobResult increments consecutiveErrors but then picks the natural next run over endedAt + 30s for recurring jobs.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, but the final PR should settle whether recurring jobs reuse cron.retry.maxAttempts/backoffMs/retryOn or keep a separate recurring policy and then update docs/types accordingly.
