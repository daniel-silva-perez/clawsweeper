## Summary

Keep open. Current main at eecd758e399277f3b6eeea2b20786257a4df4827 still has the reported scheduler asymmetry: one-shot jobs classify transient errors and retry on short backoff, while recurring cron/every error handling only applies a backoff lower bound and still selects the later natural schedule. The latest user comment reinforces this as a real operational bug, and open PR #46423 already targets the issue with closing syntax, so the issue should remain open until that PR or a focused replacement lands.

## What This Changes

Recurring cron/every jobs should retry transient execution errors with backoff instead of waiting until the next natural schedule.

## Best Possible Solution

Land a focused scheduler change, likely by reviewing/refining PR #46423 or replacing it with a narrower PR, that classifies recurring cron/every execution errors, schedules retryable transient failures on the short backoff path, preserves natural-schedule behavior for permanent errors, and updates regression tests plus docs/type comments to match the final contract.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: a daily cron or every job that returns a transient error such as network timeout enters the recurring error branch, computes endedAt + 30s backoff, and then selects the later natural schedule instead of retrying soon.

## Solution Assessment

Yes. The narrow maintainable direction is to apply transient classification in the recurring error path while keeping permanent-error natural scheduling; the issue should not close before the linked implementation PR merges or a focused replacement lands.
