## Summary

Current main still returns several hook gateway 4xx responses without a hook warning/journal line, including invalid JSON, direct wake/agent validation rejects, and mapped-hook validation rejects. The direct prior fix PR #69267 is closed unmerged, so this remains actionable and is a good narrow fix-PR candidate if the implementation sanitizes logged metadata.

## Best Possible Solution

Keep this issue open and implement a small centralized hook rejection logging helper in `src/gateway/server/hooks-request-handler.ts`. Route every non-2xx hook response through it, emit one sanitized warn line with a stable internal reason code and safe request metadata, preserve response compatibility unless deliberately improving ambiguous errors, and add focused tests for the direct and mapped 4xx branches.

## Clownfish Work Prompt

Fix the hook gateway diagnosability gap reported in openclaw/openclaw#69192. Implement a small centralized rejection logging helper in `src/gateway/server/hooks-request-handler.ts` and route every hook response with a non-2xx status through it, including query-token, non-POST, unauthorized/throttled, empty path/not found, invalid JSON/body timeout/body too large, direct wake/agent validation, direct policy/session rejects, mapped-hook validation, mapped channel/agent/session rejects, and mapping exceptions where appropriate. The warning must be sanitized: do not log hook tokens, raw bodies, authorization/cookie headers, channel targets, secret values, or full user payloads. Prefer stable reason codes such as `invalid-json`, `hook-mapping-requires-message`, `agent-not-allowed`, `session-key-disallowed`, plus safe metadata like status, normalized path shape, content type, body type, and byte count. Preserve existing HTTP response bodies/statuses unless tests intentionally cover a clearer distinct mapping-message error. Add focused tests in `src/gateway/server.hooks.test.ts` and, if stable reason codes require mapping changes, `src/gateway/hooks-mapping.test.ts`. Treat PR #69267 as prior art only; do not depend on it being merged. Avoid conflicts with open PR #50527 and do not add raw metadata logging. Add a single-line changelog fix entry if the repo’s active changelog section expects one for user-facing gateway fixes, crediting the reporter and relevant prior context, and keep wording to plugin/docs conventions.
