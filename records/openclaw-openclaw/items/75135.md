## Summary

Keep open. Current main does not already implement stack-bearing gateway logs for these attachment failure paths, and the related #63432 discussion shows a still-useful diagnostic gap. The branch is narrow and generally aligned with the gateway boundary, with one non-blocking review finding about misleading log wording in the chat path.

## What This Changes

The PR adds error-level gateway logging of attachment parse failures in `chat.send` and `agent`, using `err.stack` for `Error` instances while preserving existing error-code classification and mostly preserving response text.

## Best Possible Solution

Keep the PR open for normal maintainer review. The desired end state is a small diagnostics patch that logs a redacted, stack-bearing error at the gateway boundary, keeps client error classification stable, names parse versus staging failures accurately, and leaves the underlying #63432 root-cause fix to a follow-up with the captured stack.

## Reproduction Assessment

Yes for the diagnostic gap: current main has directly inspectable `chat.send` and `agent` catch paths that respond with `String(err)` and do not log a stack. The exact iPad/Tailscale RangeError from #63432 still needs live reproduction to identify the root cause, but that is outside this PR’s stated scope.

## Solution Assessment

Yes, logging at the gateway handler boundary is a narrow maintainable way to make these attachment failures diagnosable without changing protocol classification. The chat log label should be broadened because that catch also wraps staging, and a seam test for a thrown `RangeError` would make the fix more durable.
