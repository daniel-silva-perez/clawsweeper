## Summary

Keep open. The PR addresses a real diagnostics gap on current main, but the proposed patch introduces a likely targeted-test failure because the existing direct `chat.send` test context does not provide the new `logGateway.error` method used on an already-tested failure path.

## What This Changes

The PR adds stack-bearing gateway error logs and Error-instance response formatting to the `agent` and `chat.send` attachment parsing catch blocks.

## Best Possible Solution

Land a narrow diagnostics fix after the test harness and coverage are updated: log the full server-side stack for attachment parsing failures, preserve the existing response classification and client-facing error text, and leave the related RangeError issue open for the actual root-cause repair.

## Reproduction Assessment

Yes. The existing `chat.directive-tags.test.ts` non-image staging failure test runs through the modified `chat.send` catch with a partial `logGateway` mock, so the PR's new `context.logGateway.error(...)` call should throw before the expected response.

## Solution Assessment

No, not as submitted. Logging the stack at this boundary is a maintainable direction, but the mergeable solution should update the direct handler test context and preferably assert the new stack-bearing log while preserving response compatibility.
