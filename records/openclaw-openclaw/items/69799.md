## Summary

Keep this open. Current main has partial mitigations for macOS TCC exposure, including app-owned automation surfaces, a user-space app installer path, stable Homebrew Node resolution, and optional service wrappers, but the supported macOS Gateway install path can still run through an external/shared Node binary when installed globally or without a wrapper, so the requested default OpenClaw-owned TCC boundary is not fully implemented.

## What This Changes

Request that macOS OpenClaw ship or install a dedicated Node/runtime identity so Full Disk Access and Accessibility grants are not applied to a shared node binary.

## Best Possible Solution

Keep this as core macOS security and packaging work. The desired end state is a stable OpenClaw-owned identity for TCC-sensitive Gateway and node-host operations, with installer, LaunchAgent repair/update, doctor, onboarding, and docs aligned with the related FDA and Accessibility reports.

## Reproduction Assessment

Yes. The source-level reproduction is `openclaw gateway install --runtime node` without `--wrapper`: current code resolves the active or system Node path and writes it into service ProgramArguments, so the macOS TCC identity can still be the selected Node executable rather than an OpenClaw-owned binary.

## Solution Assessment

No. The current mitigations are useful but are not the narrowest complete fix because they leave the npm/global and unwrapped service path using a shared Node identity; the safer direction is a supported default OpenClaw-owned runtime or app-owned TCC execution boundary.
