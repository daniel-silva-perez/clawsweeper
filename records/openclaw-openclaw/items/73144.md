## Summary

Keep this issue open. Current main has adjacent improvements for DeepSeek onboarding, non-destructive config patching, default-model admission, and unmanaged gateway restart fallback, but the broader model-switch UX is not fully resolved because explicit allowlists still intentionally constrain visible/configured models and current-session model selection remains an explicit per-session override policy.

## What This Changes

The issue requests a smoother provider/model switching flow from `bailian/qwen3.6-plus` to `deepseek/deepseek-v4-pro`, covering config patching, dual model config files, allowlist mismatches, gateway restart behavior, and active-session model refresh.

## Best Possible Solution

Split this into maintainer-approved UX changes: keep the bundled DeepSeek onboarding path, add high-signal doctor/model-status warnings for provider or default-model entries hidden by allowlists or stale per-agent model config, and design a first-class `openclaw models add/set` plus explicit current-session reset/switch flow instead of silently mutating active sessions.

## Reproduction Assessment

Partially. The report gives exact model refs, product version, OS, and symptoms, and current source/tests confirm the remaining allowlist and explicit session-selection semantics; the exact patch payload and config files needed to replay all five steps are missing.

## Solution Assessment

Unclear. Automatic allowlist expansion and automatic active-session remapping may violate intentional user scoping, while warnings plus a first-class model setup/reset command look like the narrower maintainable direction.
