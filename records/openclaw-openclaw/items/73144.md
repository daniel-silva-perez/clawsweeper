## Summary

Keep open. Current main has adjacent improvements for direct Gateway config patching, DeepSeek onboarding, and unmanaged Gateway restarts, but the reported model-switch setup UX is not fully implemented: the agent-facing Gateway tool still blocks `models.providers` mutations, configured model views still intentionally prefer `agents.defaults.models` when present, and active sessions only switch models through explicit session patch/reset paths rather than automatically inheriting a changed default after reload.

## What This Changes

The issue requests a lower-friction model-switching flow from `bailian/qwen3.6-plus` to `deepseek/deepseek-v4-pro`, covering provider config mutation, global-vs-agent model config visibility, allowlist diagnostics, Gateway restart behavior, and active-session model updates.

## Best Possible Solution

Split the issue into maintainable policy decisions: preserve the agent-facing config security boundary, keep DeepSeek onboarding as the recommended provider setup path, add high-signal `models status` or `doctor` warnings when defaults/providers are hidden by `agents.defaults.models`, and consider a first-class `openclaw models add/set` flow instead of silently mutating allowlists or existing session overrides.

## Reproduction Assessment

Partially. The report includes concrete symptoms and environment details, and current source/tests verify the remaining allowlist and agent-facing config-patch behaviors; reproducing all five symptoms exactly still needs the reporter’s patch payload and final config files.

## Solution Assessment

Unclear. The requested automatic allowlist expansion and active-session re-evaluation are not obviously the safest fixes because they can override intentional scoping; warning/diagnostic improvements plus first-class setup commands look safer for maintainers to evaluate.
