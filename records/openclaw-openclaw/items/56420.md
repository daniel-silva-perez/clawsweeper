## Summary

Keep open. Current main still has the request-id-only Claude permission reply path, so this PR is not obsolete; the branch targets the right MCP bridge surface but needs maintainer review because it changes the Claude permission request contract and still has a concrete blank-session-key bug.

## What This Changes

The PR adds a required `session_key` to Claude permission request notifications, stores it with pending Claude permissions, only accepts `yes/no` replies from the matching session, and adds regression coverage for cross-session rejection.

## Best Possible Solution

Land a session-bound permission reply fix in the MCP channel bridge after tightening the `session_key` boundary, documenting or otherwise validating the Claude notification contract, and coordinating with the pending-map lifecycle work so both security and memory behavior compose cleanly.

## Reproduction Assessment

Yes. A unit-level reproduction is clear from current main: seed `pendingClaudePermissions` with `abcde`, send a `session.message` from any other non-empty `sessionKey` containing `yes abcde`, and the current handler will emit the permission notification because it checks only the request id.

## Solution Assessment

Unclear. Binding replies to the originating session is the right product boundary, but this exact patch should normalize/reject blank session keys and needs maintainer confirmation that making `permission_request.session_key` required is compatible with the active Claude client path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Normalize permission session keys at the schema boundary:** `src/mcp/channel-shared.ts:122`
  - body: This adds `session_key` as a plain `z.string()`, so `""` or whitespace-only values pass validation and are stored on the pending permission. Replies read `payload.sessionKey` through `toText`, which trims/empties blanks, so those pending permissions can never match and remain until process exit. Use a trimmed, non-empty schema or normalize before storing so only resolvable keys enter the map.
