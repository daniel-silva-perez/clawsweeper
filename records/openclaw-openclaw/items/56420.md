## Summary

Keep open. Current main still accepts Claude permission replies by request id alone, so this PR is not obsolete, but the branch should not merge as-is because the new required session_key contract is not fully tightened or propagated to all current permission_request callers.

## What This Changes

The PR adds session_key to Claude permission request notifications, stores it with pending Claude permissions, only accepts yes/no replies from the matching session, and adds regression coverage for cross-session rejection.

## Best Possible Solution

Land session-bound permission replies with a trimmed, non-empty session_key contract, update the repo e2e client and user-facing MCP docs for that contract, and compose cleanly with the pending-map lifecycle work so unresolved requests do not accumulate.

## Reproduction Assessment

Yes. Against current main, a unit-level reproduction is to create a pending permission for abcde, then send a session.message from a different non-empty sessionKey containing yes abcde; the handler emits the permission notification because it checks only the request id.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right boundary, but the patch should reject or normalize blank session keys and update every permission_request sender before the required schema field is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:122`
  - body: By making session_key required in the notification schema, this patch requires every permission_request sender to include it. The current Docker MCP channel e2e client still sends notifications/claude/channel/permission_request without session_key, so that flow will fail once validation uses this schema; update that caller or keep the schema backward-compatible while deriving the session another way.
  - confidence: 0.84
- **[P2] Reject blank permission session keys:** `src/mcp/channel-shared.ts:122`
  - body: This accepts session_key as a plain z.string() and the server stores params.session_key raw, while replies read payload.sessionKey through toText, which trims and converts blanks to undefined. Empty or whitespace-only request keys can enter pendingClaudePermissions and never match a reply, leaving the permission unresolved; use a trimmed non-empty schema or normalize before storage.
  - confidence: 0.88
