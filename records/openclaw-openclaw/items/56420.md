## Summary

Keep open. Current main still accepts Claude permission replies by short request id alone, so the PR addresses a real security-sensitive bug, but the submitted patch is not merge-ready because it makes a required session_key contract without updating all senders and still accepts blank or unnormalized session keys on the approval path.

## What This Changes

The PR adds session_key to Claude permission-request notifications, stores it with pending Claude permissions, checks yes/no replies against the originating session, and adds cross-session regression coverage.

## Best Possible Solution

Land session-bound Claude permission replies with a trimmed, non-empty session key contract, update the e2e sender and user-facing MCP docs if the field remains required, and coordinate the map value shape with the pending-map TTL/close-clear work in #71648.

## Reproduction Assessment

Yes. Against current main, create a pending Claude permission for abcde and send a session.message from a different non-empty sessionKey containing yes abcde; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right boundary, but the patch should reject or normalize blank session keys and update every permission_request sender or derive the session server-side before the required schema field is safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:122`
  - body: By making session_key required in the notification schema, this patch requires every permission_request sender to include it. The current Docker MCP channel e2e client still sends notifications/claude/channel/permission_request without session_key, so that flow will fail once validation uses this schema; update that caller and docs, or keep the schema backward-compatible while deriving the session another way.
  - confidence: 0.86
- **[P2] Reject blank permission session keys:** `src/mcp/channel-shared.ts:122`
  - body: This accepts session_key as a plain z.string() and stores params.session_key raw, while replies read payload.sessionKey through toText, which trims and converts blanks to undefined. Empty or whitespace-only request keys can enter pendingClaudePermissions and never match a reply, leaving the permission unresolved; use a trimmed non-empty schema and normalized storage.
  - confidence: 0.88
