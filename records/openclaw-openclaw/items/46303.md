## Summary

Keep this PR open. Current main still does not implement the PR's central behavior: SIGUSR1 restart drains active command/task/runtime-deps work, but it does not flush closure-local inbound debounce buffers or wait for the global followup queues before teardown. The PR is still an implementation candidate, though the current branch needs maintainer review for lifecycle ordering, SDK compatibility, and security-sensitive global-state design.

## Best Possible Solution

Keep this PR open as an implementation candidate, but request a narrower maintainer-reviewed revision. The best path is a lifecycle-safe drain surface for active inbound debouncers, backward-compatible SDK additions, restart ordering that flushes debounce work and waits for active turns plus followup queues including dropped-summary state, explicit handling for channel worker queues such as Discord, and focused tests for restart ordering, teardown races, SDK compatibility, and the security boundary.

## Risks / Open Questions

- Closing this PR would discard a plausible fix path for a real current-main gap: buffered inbound debounce messages and followup queues are still not explicitly drained during SIGUSR1 restart.
- The PR branch is broad across core, gateway, tests, UI-adjacent files, and several bundled channel plugins, so it needs maintainer review before merge rather than automatic cleanup closure.
- The proposed Symbol.for-backed process-global debouncer registry is security-sensitive because same-process plugin code and stale lifecycle state can retain or mutate restart-time handles unless the API is narrowed and teardown is robust.
- The PR changes plugin-facing debounce contracts, including handle shape and flushKey semantics, which can break third-party plugins unless kept backward-compatible or versioned.
