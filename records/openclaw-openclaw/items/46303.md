## Summary

Keep open. Current main still lacks a restart-time flush for inbound debounce buffers and followup queues, so the PR is not obsolete, but the latest PR head is not merge-ready because it introduces restart-drain correctness, public SDK compatibility, and availability-sensitive security issues.

## What This Changes

The PR adds a process-global inbound debouncer registry and flush path, a followup-queue drain waiter, SIGUSR1 restart integration, and bundled channel teardown/test updates to reduce message loss during gateway reloads.

## Best Possible Solution

Revise this branch around a backward-compatible restart-drain design: pause or gate external ingress at restart start while allowing trusted internally flushed work, mirror the followup drain predicate including dropped summaries, keep live debouncers discoverable until their transports are fully closed, and preserve public SDK return contracts with additive APIs for new drain metadata.

## Reproduction Assessment

Yes. The original user problem is reproducible by inspection on current main: debounced messages live in closure-local timers, while SIGUSR1 restart only drains active command work before server close. The PR-specific bugs are also high-confidence static reproductions from the latest head and current queue/command semantics.

## Solution Assessment

No. The PR addresses a real gap, but this is not yet the best way to solve it because it delays the external enqueue gate, undercounts followup summary state, can unregister live idle debouncers before ingress closes, and changes a public SDK return type. A safer fix needs an explicit internal-drain bypass or transport pause plus compatible SDK additions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count summary-only followup work:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps a queue alive while `queue.items.length > 0 || queue.droppedCount > 0`, but the new waiter only counts queued items plus the active drain sentinel. A summarize-overflow queue with only `droppedCount` pending can report drained and let restart close before the summary followup is sent.
  - confidence: 0.91
- **[P2] Keep live debouncers registered until ingress is closed:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: This flush path auto-unregisters debouncers idle for five minutes before `server.close()` runs, while channel ingress can still deliver events. A live but quiet channel can then buffer a late message on a handle removed from the global registry, so no later global sweep can flush it before teardown.
  - confidence: 0.86
- **[P2] Gate external command enqueues at restart start:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The branch waits through debounce flush, active-task drain, and followup drain before setting `gatewayDraining`. During that async window external chat traffic can start new command work, and `waitForActiveTasks()` only waits for tasks active at its snapshot, so restarts can be prolonged and late work can still be lost.
  - confidence: 0.82
- **[P2] Preserve the public flushKey return contract:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is re-exported through public plugin SDK subpaths, but this changes `flushKey` from `Promise<void>` to `Promise<boolean>`. Existing plugins or helpers typed to the old contract can fail to type-check; keep `flushKey` compatible and expose delivery/count state through additive APIs.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces an availability-sensitive restart window where external inbound traffic can enqueue new LLM/tool work before the gateway is marked draining.

Concerns:

- **[medium] External work accepted during restart drain:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: `markGatewayDraining()` runs only after debounce flushing, active-task drain, and followup drain. Remote chat traffic can create additional command work during that window, undermining the restart drain and creating cost/DoS exposure; pause ingress early or add an internal-only drain bypass before accepting this design.
  - confidence: 0.82
