## Summary

Keep open. Current main still lacks a restart-time inbound-debounce flush and followup-queue drain, but this PR is not ready to merge because the latest branch still has blocking restart-drain, followup accounting, SDK compatibility, and security/availability concerns, and the head is dirty with failing CI checks.

## What This Changes

The PR adds a global inbound debouncer registry and flush path, a followup-queue drain waiter, SIGUSR1 restart-loop integration, and bundled channel teardown/test updates to reduce lost messages during gateway reloads.

## Best Possible Solution

Land a backward-compatible restart drain that stops external ingress at restart start or uses an explicit trusted internal enqueue bypass, mirrors the existing followup drain predicate including dropped summaries, keeps debouncer handles tracked until transport shutdown, and exposes any new delivery counts through additive SDK APIs while preserving existing signatures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count dropped summaries while waiting for followup drain:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps running while `queue.items.length > 0 || queue.droppedCount > 0`, but this waiter only counts queued items plus the active-drain sentinel. A summary-only queue can report `drained: true` and let restart close before the summary followup is sent.
  - confidence: 0.91
- **[P2] Keep idle live debouncers registered until ingress closes:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: This pre-close flush can unregister any debouncer idle for five minutes even though the server may still accept channel ingress. A live but quiet channel can then receive a late message and buffer it on a handle no longer visible to the global restart flush.
  - confidence: 0.86
- **[P2] Gate external enqueues before restart drain:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The branch waits through debounce flush, active-task drain, and followup drain before setting `gatewayDraining`. During that window inbound users can still enqueue command work that the active-task snapshot may not wait for, prolonging restarts and reintroducing restart-time loss or cost spikes.
  - confidence: 0.82
- **[P2] Preserve flushKey's SDK return contract:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is re-exported through public plugin SDK subpaths, and this changes `flushKey` from the current `Promise<void>` shape to `Promise<boolean>`. Keep `flushKey` compatible and expose delivery counts through an additive method.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces an availability and cost risk by leaving external inbound command enqueues open during the restart drain window.

Concerns:

- **[medium] External work accepted during restart drain:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: Moving `markGatewayDraining()` after debounce, active-task, and followup drain steps lets remote chat traffic start additional LLM/tool work while a restart is draining. Fix by pausing ingress early or allowing only trusted internally flushed work during the drain window.
  - confidence: 0.82
