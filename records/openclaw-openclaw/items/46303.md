## Summary

Keep this PR open. Current main still lacks the PR's central SIGUSR1 restart behavior: inbound debounce buffers remain closure-local and undiscoverable, the restart path does not flush them, and there is no all-followup-queue drain waiter before teardown. The PR is still a live implementation candidate, but it needs maintainer review because the proposed fix changes plugin-facing debounce contracts and introduces process-global lifecycle/security concerns.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a maintainer-reviewed revision that adds a lifecycle-safe drain surface for active inbound debouncers, preserves plugin SDK compatibility, orders restart draining so debounce flushes and busy-session followups are both observed before teardown, covers channel worker queues such as Discord, and includes focused tests for SIGUSR1 ordering, followup dropped-summary state, teardown races, and SDK compatibility.

## Risks / Open Questions

- The PR's Symbol.for-backed process-global debouncer registry is security- and lifecycle-sensitive: stale same-process handles can retain closures, keep timers alive, or enqueue work after teardown unless ownership and disposal are robust.
- The PR changes plugin-facing debounce handle shape and flushKey semantics; because createInboundDebouncer is exported through the plugin SDK, third-party plugin compatibility needs explicit preservation or versioning.
- Several bundled channel lifecycles have shutdown race concerns in the review comments: unregistering before ingress is fully stopped can make late buffers undiscoverable during restart.
- The PR is broad and marked dirty-candidate, with unrelated formatting/test churn mixed into a restart/lifecycle fix; maintainers should require a narrower, green revision before merge.
