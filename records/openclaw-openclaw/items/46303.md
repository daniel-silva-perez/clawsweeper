## Summary

Keep open. Current main still does not implement the restart-time debounce/followup drain behavior, and the PR branch still has blocking lifecycle, SDK compatibility, and security-sensitive command-admission issues that need a revised maintainer-reviewed branch rather than cleanup closure.

## What This Changes

The PR adds a process-global inbound debouncer registry and flush path, a followup-queue drain waiter, SIGUSR1 restart integration, and bundled channel teardown/test updates to reduce message loss during gateway reloads.

## Best Possible Solution

Revise the branch to preserve the public debouncer contract or add an additive method, keep live debouncers discoverable until ingress is definitely closed, gate or pause external work at restart start while allowing trusted internal restart-flush work, mirror the followup drain predicate including dropped summaries, and cover the sequence with focused run-loop, queue, debounce, channel teardown, and SDK compatibility tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count dropped summaries while waiting for followup drain:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps draining while `queue.items.length > 0 || queue.droppedCount > 0`, but this waiter only counts queued items plus an active drain sentinel. A queue that only has summarized dropped messages can return `drained: true` and let restart close before the summary followup is sent.
  - confidence: 0.91
- **[P2] Keep idle live debouncers registered until ingress closes:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: This pre-close flush can unregister any debouncer idle for five minutes even though the server is still accepting channel ingress. A live but quiet channel can then receive a late message and buffer it on a handle that is no longer in the global registry, so no later global flush can rescue it before shutdown.
  - confidence: 0.86
- **[P2] Gate external enqueues before restart draining:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The branch waits through debounce flush, active-task drain, and followup drain before setting `gatewayDraining`. During that window inbound channel users can still enqueue new command work that the active-task snapshot may not wait for, prolonging restarts and reintroducing restart-time loss or cost spikes.
  - confidence: 0.82
- **[P2] Preserve flushKey return type for SDK compatibility:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is re-exported through public plugin SDK subpaths, and this change makes `flushKey` return `Promise<boolean>` instead of the current `Promise<void>` shape. Third-party plugins or shared test helpers typed against the old contract can fail to type-check on upgrade; keep `flushKey` compatible and expose delivery counts through an additive method.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete restart-window availability and cost-control concern by accepting external command work until late in the drain sequence.

Concerns:

- **[medium] External work accepted during restart drain:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: Moving `markGatewayDraining()` after the debounce, active-task, and followup drain steps means remote chat traffic can start additional LLM/tool work while the operator expects a restart drain. Fix by pausing ingress early or allowing only trusted internally flushed work during the drain window.
