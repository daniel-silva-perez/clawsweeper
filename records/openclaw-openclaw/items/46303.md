## Summary

Keep this PR open. Current main still lacks the SIGUSR1 debounce/followup drain behavior, but the latest branch has blocking restart-lifecycle issues: it can report followup queues drained while dropped-summary work remains, it can unregister quiet live debouncers before ingress is closed, and it delays the command-queue draining gate until after several awaited restart steps.

## What This Changes

The PR adds a process-global inbound debouncer registry, a followup-queue drain waiter, SIGUSR1 restart integration, and bundled channel teardown/test updates to reduce message loss during gateway reloads.

## Best Possible Solution

Revise this into a narrower maintainer-reviewed restart lifecycle fix: preserve the public debouncer contract or version any SDK expansion, keep live debouncers discoverable until ingress is fully closed, stop accepting external command work at restart start while still allowing trusted restart-flush work, wait for active turns before followup queues, count dropped-summary state as pending, and cover the behavior with focused run-loop, queue, debounce, channel teardown, and SDK compatibility tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count dropped summaries while waiting for followup drain:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps draining while `queue.items.length > 0 || queue.droppedCount > 0`, but this new waiter only counts queued items plus an active drain sentinel. A queue that only has summarized dropped messages can return `drained: true` and let restart close before the summary followup is sent.
  - confidence: 0.91
- **[P2] Keep idle live debouncers registered until ingress closes:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: This pre-close flush can unregister any debouncer that has been quiet for five minutes even though the server is still accepting channel ingress. A live but idle channel can then receive a late message and buffer it on a handle that is no longer in the global registry, so no later global flush can rescue it before shutdown.
  - confidence: 0.86
- **[P2] Gate external enqueues before restart draining:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The branch waits through debounce flush, active-task drain, and followup drain before setting `gatewayDraining`. During that window inbound channel users can still enqueue new command work that the active-task snapshot may not wait for, prolonging restarts and reintroducing restart-time loss or cost spikes. Pause ingress early or add an internal-only bypass for trusted flushed work instead.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces a restart window where external channel traffic can enqueue new command work before the gateway is marked draining.

Concerns:

- **[medium] External work accepted during restart drain:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: Moving `markGatewayDraining()` after the debounce, active-task, and followup drain steps means remote chat traffic can start additional LLM/tool work while the operator expects a restart drain. That is a concrete availability and cost-control concern, and it should be fixed by pausing ingress early or allowing only trusted internally flushed work during the drain window.
  - confidence: 0.82
