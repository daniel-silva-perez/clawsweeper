## Summary

Keep open. Current main still treats top-level `openclaw.json` `env.vars` and env sugar as string-only environment injection, while SecretRef resolution is implemented for other credential surfaces. The related ACPX MCP env SecretRef work and closed AWS provider PR are useful context, but neither implements this requested global `env.vars` SecretRef surface.

## What This Changes

Requests support for SecretRef objects in the top-level `openclaw.json` `env.vars` map so runtime environment variables can be sourced from external secret providers such as 1Password.

## Best Possible Solution

Implement this as a core config/secrets feature: widen the approved env surface to `SecretInput`, resolve refs through the existing SecretRef runtime pipeline before env application, keep resolved plaintext runtime-only unless explicitly designed otherwise, preserve blocked-env checks, and update schema, target registry data, generated docs, public docs, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a structured `{source, provider, id}` SecretRef under `env.vars`: current schema/type validation accepts only strings, and the runtime collector skips non-string values if raw JSON reaches that path.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable solution should extend the generic SecretInput resolver and registry instead of adding a 1Password-specific or exec-only path.
