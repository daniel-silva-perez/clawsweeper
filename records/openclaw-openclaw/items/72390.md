## Summary

Keep open. Current main still treats top-level `openclaw.json` `env.vars` and env sugar as plain string-only environment injection, while SecretRef resolution is wired for credential/plugin surfaces such as ACPX MCP server env values. The requested exec-backed SecretRef workflow for global env vars is valid but remains unimplemented and should not be closed as a duplicate of the narrower MCP work or the closed AWS/provider PR.

## What This Changes

The issue requests SecretRef support for top-level `openclaw.json` `env.vars` so dynamic secret providers can populate runtime environment variables.

## Best Possible Solution

Implement this as a core config/secrets feature: widen `env.vars` to `SecretInput`, collect and resolve those refs through the existing runtime SecretRef pipeline before env application, keep resolved plaintext out of persisted config/service snapshots, update the generated schema/docs, and add focused tests for env/file/exec refs plus unresolved-ref and guardrail behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to put a structured SecretRef object under `env.vars`: current schema/type definitions reject it as non-string, and the runtime env collector skips non-string `env.vars` values even if they are present in a raw JSON object.

## Solution Assessment

Unclear until maintainer design review, but the requested direction is plausible. The narrow maintainable solution is to extend the existing SecretInput resolver and env guardrails rather than adding a separate ad hoc 1Password or exec-only path.

## Risks / Open Questions

- This changes a global environment injection path, so resolution order must avoid persisting resolved plaintext into config snapshots or durable service metadata unless explicitly intended.
- Existing guardrails need to survive the change: no overriding already-set env vars, dangerous env var filtering, portable key normalization, and skipping unresolved `${VAR}` placeholders.
- Top-level env sugar currently accepts only strings, so maintainers should decide whether SecretInput support belongs only under `env.vars` or also in the sugar form.
