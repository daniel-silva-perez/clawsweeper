## Summary

Keep open. Current main still treats top-level `openclaw.json` `env.vars` as string-only env injection and does not include that surface in the SecretRef runtime or target registry. The related ACPX MCP env SecretRef work is useful precedent, but it covers plugin-scoped MCP server env, not global config env vars.

## What This Changes

Support structured SecretRef values in the top-level `openclaw.json` `env.vars` block so external secret managers can provide runtime environment variables.

## Best Possible Solution

Implement this as a core config/secrets feature: widen the approved `env.vars` surface to `SecretInput`, resolve refs through the existing SecretRef runtime before env application, keep resolved plaintext runtime-only unless explicitly designed otherwise, preserve blocked-env checks, and update schema, registry/docs, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a structured `{source, provider, id}` SecretRef under `env.vars`: current schema accepts only strings, and if raw JSON bypasses schema, `applyConfigEnvVars` skips the object value.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable solution should extend the generic SecretInput resolver and env materialization boundary instead of adding a 1Password- or exec-specific path.
