## Summary

Keep open. Current main still treats top-level `openclaw.json` `env.vars` and env sugar as string-only environment injection, while SecretRef resolution remains registered for other credential surfaces. The related ACPX MCP env SecretRef support and AWS provider PR are useful context, but they do not implement the requested global `env.vars` SecretRef surface.

## What This Changes

The issue requests SecretRef support in top-level `openclaw.json` `env.vars` so dynamic providers such as exec-backed 1Password can populate runtime environment variables.

## Best Possible Solution

Implement this as a core config/secrets feature: widen the approved env surface to `SecretInput`, resolve refs through the existing SecretRef runtime pipeline before env application, preserve dangerous-env guardrails, keep resolved plaintext runtime-only, and update schema, target registry data, generated docs, public docs, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a structured `{source, provider, id}` SecretRef under `env.vars`: current schema/type validation rejects it as non-string, and the runtime collector skips non-string values if raw JSON reaches that path.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable solution should extend the generic SecretInput resolver and registry rather than adding a 1Password-specific or exec-only path.
