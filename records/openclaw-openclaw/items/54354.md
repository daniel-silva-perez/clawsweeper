## Summary

Keep open. Current main has related `openclaw.install.minHostVersion` and ClawHub compatibility gates, but it does not implement the PR's `engines.openclaw` npm version-selection behavior. The branch is a plausible feature, but it changes the public plugin compatibility contract and npm resolution semantics and has blocking correctness and supply-chain findings before it can merge.

## What This Changes

The PR adds `engines.openclaw`-aware npm version resolution for plugin install/update, new semver and npm registry helpers, an `--ignore-engine` flag, load-time compatibility warnings, and related tests.

## Best Possible Solution

Decide the public plugin compatibility contract first. If `engines.openclaw` is accepted, rework the branch so npm tags, exact versions, prereleases, invalid ranges, non-release host versions, and external registry metadata all preserve user intent and fail safely with focused install/update/load tests; otherwise keep the shipped `openclaw.install.minHostVersion` plus ClawHub compatibility model as the supported path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Honor requested npm selectors during engine resolution:** `src/plugins/install.ts:743-747`
  - body: For specs like `pkg@beta`, the PR ignores the requested dist-tag and scans all published versions with `allowPrerelease` derived from `selectorIsPrerelease`, which is false for tags. That can rewrite an explicit beta-channel request to an unrelated exact stable version, changing existing npm/tag semantics.
  - confidence: 0.9
- **[P1] Block incompatible exact-version installs:** `src/plugins/install.ts:743`
  - body: Exact-version specs skip the new engine resolver entirely, so `pkg@1.2.3` can still install even when that version declares an incompatible `engines.openclaw` range. The new `--ignore-engine` flag implies incompatible exact pins should be blocked unless explicitly bypassed.
  - confidence: 0.86
- **[P1] Reject malformed engine ranges:** `src/infra/semver-range.ts:237-238`
  - body: When a non-empty branch produces no comparator tokens, `allComparators.every(...)` returns true. Unsupported ranges such as `>=2026.3` or `foo` therefore match every host version and bypass the install/update compatibility gate.
  - confidence: 0.92
- **[P2] Validate registry engine values before resolving:** `src/infra/npm-registry-versions.ts:82-87`
  - body: The registry `engines` object is cast to `Record<string, string>` without checking `engines.openclaw`. A non-string value can reach `satisfiesRange`, which calls `trim()` and throws during install/update instead of handling malformed external metadata defensively.
  - confidence: 0.88
- **[P2] Preserve prerelease lower bounds:** `src/infra/semver-range.ts:161-175`
  - body: Caret and tilde ranges replace the lower-bound prerelease with `null`, so `^2026.3.14-beta.1` becomes `>=2026.3.14` instead of including the beta lower bound. That diverges from npm semver behavior and can select or reject the wrong versions for prerelease engine constraints.
  - confidence: 0.82
- **[P3] Suppress warnings for unparsable host versions:** `src/plugins/loader.ts:989-995`
  - body: The loader only checks that `VERSION` is truthy before evaluating ranges. Development or local builds with non-semver versions will fail every constrained range and warn for all constrained plugins even when compatibility is unknown.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff introduces npm registry metadata reads and client-side version selection with concrete fail-open and selector-mismatch risks.

Concerns:

- **[medium] Explicit npm dist-tags can be overridden:** `src/plugins/install.ts:743`
  - body: The resolver treats tag selectors as broad package scans and can rewrite a user-requested tag such as `@beta` to a different exact version, changing the requested release channel during plugin installation or update.
  - confidence: 0.9
- **[medium] Exact incompatible versions bypass the engine gate:** `src/plugins/install.ts:743`
  - body: Exact-version specs skip engine resolution, so a package version with incompatible `engines.openclaw` metadata can still install without using the new bypass flag.
  - confidence: 0.86
- **[medium] Malformed compatibility ranges fail open:** `src/infra/semver-range.ts:237`
  - body: Unsupported `engines.openclaw` ranges can evaluate as compatible because an empty comparator list passes `every()`, bypassing the new install/update gate and hiding load warnings.
  - confidence: 0.92
- **[low] External registry metadata is not parsed defensively:** `src/infra/npm-registry-versions.ts:82`
  - body: The npm registry `engines.openclaw` value is trusted as a string, so malformed metadata can throw during install/update instead of producing a controlled compatibility result.
  - confidence: 0.88
