## Summary

Keep this PR open. Current main has overlapping plugin host compatibility checks through `openclaw.install.minHostVersion` and ClawHub `pluginApi` / `minGatewayVersion`, but it still does not implement the PR's intended npm install/update behavior of selecting the newest compatible package version from per-version `engines.openclaw` metadata. The PR also has unresolved review and supply-chain-sensitive concerns around npm tag semantics, malformed ranges, and untrusted registry metadata.

## Best Possible Solution

Keep this PR open for maintainer review. The best direction is to either rework it around the existing `openclaw.install.minHostVersion` / ClawHub compatibility model, or explicitly decide that npm `engines.openclaw` is a supported public plugin metadata surface. Any rework should preserve npm dist-tag and prerelease semantics, fail malformed compatibility metadata safely, guard all registry-derived values, and add focused install/update tests for stable, beta, exact-version, malformed-range, and non-string metadata cases.
