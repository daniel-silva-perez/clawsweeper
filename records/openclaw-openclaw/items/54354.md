## Summary

Keep this PR open for maintainer review. Current main has overlapping compatibility gates through `openclaw.install.minHostVersion` and ClawHub `pluginApi` / `minGatewayVersion`, but it still does not implement the PR's intended npm install/update behavior of choosing the newest compatible package version from per-version `engines.openclaw` metadata. The branch also needs product/API and supply-chain review before merge because it adds a new public metadata contract and client-side npm registry version selection.

## What This Changes

The PR adds custom semver and npm registry helpers for `engines.openclaw`, wires plugin install/update to resolve a compatible npm package version with a new `--ignore-engine` escape hatch, and adds load-time compatibility diagnostics plus tests.

## Best Possible Solution

Keep this PR open for maintainer review. The best direction is to either rework it around the existing `openclaw.install.minHostVersion` / ClawHub compatibility model, or explicitly bless `engines.openclaw` as a supported public plugin metadata surface. Any rework should preserve npm dist-tag and prerelease semantics, guard all registry-derived values, fail malformed compatibility metadata safely, prefer the existing semver dependency or prove equivalence, and add focused install/update tests for stable, beta, exact-version, malformed-range, and non-string metadata cases.
