## Summary

Keep open. Current main still lacks the requested WhatsApp configured ACP binding path, but this PR should not merge as-is because the proposed ACP error cleanup and media loading changes introduce concrete availability and file-read security regressions.

## What This Changes

The PR adds WhatsApp configured ACP session routing, WhatsApp media/history threading into ACP prompts, ACP prompt hook dispatch, delivery visibility tweaks, and schema/test updates for WhatsApp ACP bindings.

## Best Possible Solution

Rework the feature on top of current main by adding a WhatsApp channel binding provider and configured-route resolution in the WhatsApp inbound path, preserving history media metadata where needed, and reusing the existing dispatch-acp-attachments/MediaAttachmentCache hardening instead of duplicating file-read policy in ACP dispatch.

## Reproduction Assessment

Yes for the blocking patch issues: bind two conversations to one ACP session and force an ACP_TURN_FAILED path to observe session-wide unbinding, and place a symlinked directory under an allowed media root to show raw-prefix checks can read outside the root. Current main source provides the service semantics needed for both focused tests.

## Solution Assessment

No, not as proposed. The maintainable fix is narrower: add WhatsApp's missing ACP binding adapter/routing seam, keep current canonical media path enforcement, and only clean up the current binding or a confirmed stale canonical session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Restrict ACP cleanup to the current binding:** `src/auto-reply/reply/dispatch-acp.ts:596-605`
  - body: The PR's error path unbinds by targetSessionKey for ACP_SESSION_INIT_FAILED/ACP_TURN_FAILED. That input removes every conversation bound to the same ACP session, so one failing WhatsApp turn can disconnect unrelated conversations that share the session; resolve and unbind the current binding, and keep session-wide cleanup only for confirmed stale sessions.
  - confidence: 0.88
- **[P1] Canonicalize attachment paths before reading:** `src/auto-reply/reply/dispatch-acp.ts:112-145`
  - body: The new attachment loader allowlists the raw path and rejects only a final-component symlink before fs.readFile. A path under an allowed root can still traverse a symlinked directory to a file outside the root, so this can exfiltrate local files to ACP; reuse MediaAttachmentCache or realpath-check the candidate against realpath roots before opening.
  - confidence: 0.9
- **[P2] Avoid exposing host media paths in prompts:** `src/auto-reply/reply/inbound-meta.ts:247-254`
  - body: Replacing history media placeholders with the literal local mediaPath sends host filesystem details into the ACP prompt and plugin hooks. The agent only needs an opaque attachment reference or separate attachment payload, so avoid embedding full local paths in untrusted prompt context.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security-sensitive regressions in ACP session cleanup and local media file handling.

Concerns:

- **[high] Session-wide ACP unbind can be remotely triggered:** `src/auto-reply/reply/dispatch-acp.ts:596`
  - body: Unbinding by targetSessionKey on generic ACP turn/init failures can remove every conversation bound to that session; a crafted or unlucky failing inbound turn can cause cross-conversation availability loss.
  - confidence: 0.86
- **[high] Raw attachment path checks permit symlink-directory escape:** `src/auto-reply/reply/dispatch-acp.ts:112`
  - body: The PR's direct lstat/readFile path checks do not canonicalize intermediate symlink components, allowing an allowed-root prefix to resolve to an outside file that is then base64-attached to ACP.
  - confidence: 0.9
- **[medium] Prompts expose local media file paths:** `src/auto-reply/reply/inbound-meta.ts:247`
  - body: Embedding mediaPath values in ACP prompt history leaks host filesystem paths to the harness and prompt hooks, which is unnecessary once media is carried as an attachment.
  - confidence: 0.76
