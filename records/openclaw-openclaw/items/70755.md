## Summary

Keep this issue open. Current main still authenticates the BlueBubbles webhook by comparing caller-supplied `guid`/`password` material to `target.account.config.password`, the docs still publish `?password=<password>` setup guidance, and paired PR #70757 is open and unmerged with closing syntax for this issue. Because the remaining work is security-sensitive and already has a live implementation candidate, this should not be auto-closed or queued as a separate repair job.

## What This Changes

The issue asks OpenClaw to stop reusing the BlueBubbles server/API password as the public inbound webhook secret and add a distinct webhook credential surface.

## Best Possible Solution

Land PR #70757 or an equivalent maintainer-approved security fix that adds a dedicated BlueBubbles `webhookSecret`, authenticates inbound webhooks only against that credential, keeps the API password reserved for BlueBubbles server access, updates setup/docs/SecretRef metadata, and ships regression coverage for top-level and per-account credential resolution.

## Reproduction Assessment

Yes. Configure BlueBubbles with `channels.bluebubbles.password`, send a POST to `/bluebubbles-webhook?password=<password>` with a caller-controlled `handle.address`, and current main authenticates against the server password before the allowlist and command-gating flow consumes that sender identity.

## Solution Assessment

Yes. A distinct inbound `webhookSecret` is the narrow maintainable fix; header-only transport hardening would reduce URL exposure but would not separate the reusable BlueBubbles API credential from the public webhook credential.
