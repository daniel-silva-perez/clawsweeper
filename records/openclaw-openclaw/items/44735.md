## Summary

Keep open. Current main still auto-enables the bundled Feishu channel when another enabled Feishu/Lark owner claims the same channel without `preferOver` metadata, while this PR's submitted guard has a real allowlist correctness bug and now targets the pre-split auto-enable file layout.

## What This Changes

The PR adds an auto-enable guard and regression test so configured Feishu does not enable the bundled `feishu` channel when `plugins.entries.openclaw-lark.enabled=true` already claims the Feishu channel.

## Best Possible Solution

Keep the contribution alive but adapt it to the current split implementation. The final fix should live in `src/config/plugin-auto-enable.shared.ts` or a small helper it owns, and should suppress bundled Feishu only when another same-channel owner is effective, or made effective by the same repair pass, under the normal plugin activation rules.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: configured `channels.feishu` plus manifest claims from `feishu` and `openclaw-lark` without `preferOver` still produces bundled `feishu` as the auto-enable candidate on current main, and there is no later enabled-alternative guard before registration.

## Solution Assessment

No, not as submitted. The intended direction is good, but the PR's guard must be ported to the current split files and must use effective activation semantics, including `plugins.enabled`, allow, deny, and explicit disable, instead of treating an entry flag alone as active.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require allowlist before treating alternatives as active:** `src/config/plugin-auto-enable.ts:485`
  - body: This guard treats `plugins.entries[record.id].enabled === true` as an active alternative, but on this branch a restrictive `plugins.allow` can still prevent a non-bundled owner from running. With Feishu configured and `openclaw-lark` enabled but not allowed, this `continue` can skip the bundled fallback and leave no effective channel owner. Check the same activation/allowlist rules before suppressing the bundled candidate.
  - confidence: 0.86
