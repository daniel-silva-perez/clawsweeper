## Summary

Keep open. Current main still does not implement this PR's intended behavior: Control UI markdown `http://`/`https://` images are flattened to alt text, while `data:image/...` images render. The PR remains a focused implementation candidate for open #40945, but it needs maintainer product/security direction because a renderer-only change conflicts with the current served CSP/docs and would enable arbitrary browser remote-image fetches from chat output.

## Best Possible Solution

Keep this PR open as an implementation candidate for #40945. A complete maintainer-approved solution should align the markdown renderer, served CSP, docs, and tests around an explicit remote-media policy, likely HTTPS-only at minimum and preferably an allowlist, opt-in setting, managed-media proxy, `referrerpolicy="no-referrer"`, and the existing `markdown-inline-image` styling. If maintainers choose to preserve the current no-remote-fetch policy, record that decision on #40945 or #64607 before closing implementation attempts.

## Risks / Open Questions

- Merging the PR as-is would allow arbitrary remote image URLs in chat/model output to trigger browser fetches, exposing viewer IP address, user agent, and read timing to third-party hosts.
- The PR allows `http://` image URLs as well as `https://`, which creates mixed-content and downgrade concerns for HTTPS Control UI deployments.
- A renderer-only change conflicts with the current Control UI CSP and docs, so served deployments may still block the images unless CSP, tests, and documentation change together.
- The PR likely needs a rebase/design update to preserve the current `markdown-inline-image` styling expectations and align with the current markdown-it renderer shape.
