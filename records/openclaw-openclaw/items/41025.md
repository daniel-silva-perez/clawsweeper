## Summary

Keep this PR open. Current main still blocks remote http/https markdown images in the Control UI, the linked bug #40945 remains open in the provided GitHub context, and the PR is still a plausible implementation candidate. It needs maintainer product/security review because allowing arbitrary remote image fetches changes the browser privacy posture and conflicts with current CSP/docs/tests unless those are intentionally updated too.

## Best Possible Solution

Keep this PR open as a focused implementation candidate for #40945. Maintainers should make an explicit remote-media policy decision, then either update the renderer plus CSP/docs/tests with mitigations such as HTTPS-only, allowlisting or proxying, lazy loading, and referrerpolicy="no-referrer", or document that remote image fetches remain intentionally blocked and close implementation attempts in favor of #40945 or the broader media discussion in #64607.

## Risks / Open Questions

- Merging the renderer-only patch would not fully solve browser rendering because the current served CSP still blocks remote image loads unless img-src is deliberately relaxed.
- If CSP is relaxed, arbitrary chat/model content could trigger third-party image requests from every viewer's browser, exposing IP address, user agent, and read timing to the image host.
- The PR allows http:// as well as https://, which adds mixed-content and downgrade concerns for HTTPS Control UI deployments.
- Current docs and tests explicitly document and enforce a no-remote-fetch posture, so the final change needs aligned docs, CSP tests, renderer tests, and product/security sign-off.
- Security review found no workflow, dependency, lockfile, package-publishing, or supply-chain changes in the provided diff; the meaningful risk is the runtime browser network/privacy behavior.
