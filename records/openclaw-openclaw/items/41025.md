## Summary

Keep open. Current main still does not implement the PR’s intended behavior: Control UI markdown renders `data:image/...` images, but `http://` and `https://` markdown image URLs are flattened to alt text. The PR is still a focused implementation candidate for open #40945, but it needs maintainer product/security direction because the current CSP, docs, tests, and changelog describe an intentional no-remote-image-fetch policy.

## Best Possible Solution

Keep this PR open as an implementation candidate for #40945. A complete maintainer-approved solution should align the markdown renderer, served CSP, docs, and tests around an explicit remote-media policy, likely HTTPS-only at minimum and preferably an allowlist, opt-in setting, managed-media proxy, `referrerpolicy="no-referrer"`, and the existing `markdown-inline-image` styling. If maintainers choose to preserve the current no-remote-fetch policy, record that decision on #40945 or the broader #64607 before closing implementation attempts.

## Risks / Open Questions

- Merging the PR as-is would allow arbitrary `http://`/`https://` image URLs in chat/model output to trigger browser fetches, exposing viewer IP address, user agent, and read timing to third-party hosts.
- The PR allows `http://` as well as `https://`, which creates mixed-content and downgrade concerns for HTTPS Control UI deployments.
- A renderer-only change conflicts with the current served CSP, docs, changeloged policy, and adjacent exported-HTML hardening, so the complete product decision needs more than the two-file patch.
