## Summary

Keep open. Current main still has the reported numeric ceiling and the central config validation mapper still only enriches allowed-value cases, while open PR #73220 explicitly targets this issue with closing syntax and should be reviewed or closed before the issue is resolved.

## What This Changes

Request that schema-validated numeric ceiling rejections explicitly state the configured maximum allowed value in operator-facing config errors.

## Best Possible Solution

Keep this issue open until PR #73220 lands or an equivalent central `mapZodIssueToConfigIssue()` fix adds generic `too_big`/`too_small` bound hints with regression coverage for both maximum and minimum constraints.

## Reproduction Assessment

Yes. The source-level reproduction is setting `session.agentToAgent.maxPingPongTurns` to `10`: the schema still caps it at `5`, and the mapper currently has no numeric-bound handling beyond returning the native Zod message.

## Solution Assessment

Yes. A generic central mapper enhancement with focused validation tests is the narrowest maintainable solution; per-field custom schema messages would not cover all schema-validated ceilings and floors.
