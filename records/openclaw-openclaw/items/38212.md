## Summary

Keep open. Current main still does not show a source/test proof that raw `codex-cli/gpt-5.4` reaches the session-memory embedded helper as an `openai-codex` model-resolution provider; current code instead points toward a newer product split where legacy CLI refs are migrated to canonical provider refs plus `agentRuntime`.

## Best Possible Solution

Keep this issue open until maintainers choose one explicit product path and prove it. Either normalize legacy `codex-cli/*` helper/embedded runs to the intended `openai-codex/*` Codex OAuth route before model resolution, with focused regression coverage for the session-memory slug generator, or enforce and document that raw legacy `codex-cli/*` primary refs are unsupported for PI helper hooks and must be migrated to canonical provider refs plus runtime configuration, with coverage proving loaded config cannot reach the helper as raw `codex-cli/gpt-5.4`.
