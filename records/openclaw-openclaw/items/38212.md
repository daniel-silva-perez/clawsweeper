## Summary

Current main still does not prove the reported raw `codex-cli/gpt-5.4` helper path is fixed. The session-memory slug helper forwards the configured provider/model unchanged, the embedded runner still resolves that exact provider/model before auth, and `codex-cli` remains a CLI backend/auth alias rather than an `openai-codex` model-resolution provider alias.

## Best Possible Solution

Keep this issue open until maintainers choose and prove one product path. Either normalize legacy `codex-cli/*` helper/embedded runs to the intended `openai-codex/*` Codex OAuth route before model resolution and add a focused session-memory slug regression, or explicitly enforce/document that raw legacy `codex-cli/*` primary refs are unsupported for PI helper hooks and must be migrated to canonical model refs plus runtime configuration, with a regression proving loaded config cannot reach the helper as raw `codex-cli/gpt-5.4`.
