## Summary

Keep this PR open. Current main still ignores `gateway.remote.transport` when selecting the Gateway URL, so the intended SSH-loopback mapping is not implemented; the PR is a focused candidate for the related open SSH transport bug, but it touches a client-side plaintext WebSocket security guardrail and has a small URL-preservation issue to resolve before maintainer landing.

## What This Changes

The PR adds SSH-transport URL mapping in `src/gateway/connection-details.ts` so configured remote WebSocket URLs become local loopback tunnel endpoints, with a new `buildGatewayConnectionDetails` regression test in `src/gateway/call.test.ts`.

## Best Possible Solution

Land a narrow SSH-transport fix in the gateway connection-detail seam that selects the effective local tunnel endpoint, preserves URL path/search/hash semantics, keeps direct-transport `ws://` rejection intact, and coordinates with #26250. Add regression coverage for SSH mapping, bare URLs, direct remote rejection, and override precedence.

## Reproduction Assessment

Yes. A focused current-main check is enough: configure `gateway.mode=remote`, `gateway.remote.transport="ssh"`, and a non-loopback plaintext `gateway.remote.url`; current main selects that raw URL in `buildGatewayConnectionDetailsWithResolvers()` and rejects it with `isSecureWebSocketUrl()` before any loopback tunnel endpoint is used.

## Solution Assessment

Unclear as-is. The PR targets the right narrow surface, but the URL reconstruction should preserve bare URL shape and maintainers should verify the SSH tunnel contract because the changed code maps to loopback without itself establishing the tunnel.
