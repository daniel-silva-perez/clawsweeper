## Summary

Keep this PR open. Current main still does not implement the PR's intended SSH-transport URL mapping: `buildGatewayConnectionDetailsWithResolvers()` reads `gateway.remote.url` directly, ignores `gateway.remote.transport`, and runs the plaintext WebSocket security check against that selected URL. The PR remains a focused implementation candidate for the related open SSH transport bug #26250, but it changes a security guardrail and should get normal maintainer/security review rather than cleanup closure.

## Best Possible Solution

Keep the PR open for normal maintainer review. The likely path is a narrowly tested fix coordinated with #26250 that makes SSH transport validate and connect to the effective loopback tunnel endpoint, preserves the existing direct-transport security rejection, and reconciles the implementation with the documented `gateway.remote.url`/`gateway.remote.sshTarget` contract.

## Risks / Open Questions

- The PR changes a security-sensitive plaintext WebSocket guardrail. Maintainers should verify SSH mapping applies only to the intended SSH tunnel contract and that direct `ws://` remote hosts still fail closed.
- The PR's mapping derives a local loopback URL from the configured remote URL without proving a tunnel has been established in the call/bootstrap path; review should decide whether tunnel lifecycle belongs in this layer or whether config should continue to store the local tunnel URL directly.
- The bot review's trailing-slash finding is real for bare URLs because `URL.pathname` is `/`; add or adjust coverage if this PR proceeds.
