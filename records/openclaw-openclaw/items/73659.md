## Summary

Keep this PR open. Current main still does not implement the PR's core SSH-transport URL mapping, and the patch is a focused candidate for the related SSH transport bug, but it introduces a fail-open malformed-URL fallback in the WebSocket security path that should be fixed before merge.

## What This Changes

The PR rewrites SSH-transport gateway remote URLs to a 127.0.0.1 tunnel endpoint in `buildGatewayConnectionDetails` and adds a gateway call test for that mapping.

## Best Possible Solution

Keep the PR open for maintainer review and require a narrow revision that preserves fail-closed URL validation, keeps direct-transport non-loopback `ws://` rejection intact, and adds regression coverage for malformed, bare-host, and explicit-path SSH remote URL cases before coordinating with #26250.

## Reproduction Assessment

Yes. A high-confidence focused reproduction for the PR-introduced bug is `gateway.mode=remote`, `gateway.remote.transport=ssh`, no URL override, and a malformed nonempty `gateway.remote.url`; the patch catches the parse failure and returns `ws://127.0.0.1:18789` instead of letting current validation reject it.

## Solution Assessment

No as written. Mapping an SSH transport to an effective loopback endpoint may be the right core direction if maintainers want to support that config shape, but the current implementation should not silently replace invalid remote URLs with a trusted loopback target.
