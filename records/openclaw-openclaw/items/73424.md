## Summary

Keep open. Current main still routes explicit image-tool and shared channel media through `loadWebMedia` before the VLM/provider call, still optimizes every non-GIF image by default, and still collapses total optimizer backend failure into the exact `Failed to optimize image` error. The issue is also paired with open PR #74243 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to stop valid JPEG/PNG media from failing in shared image preprocessing with the opaque `Failed to optimize image` error before provider/VLM execution.

## Best Possible Solution

Keep this issue open and review #74243 as the canonical user-facing resilience fix, or land an equivalent change in `src/media/web-media.ts` that falls back to the original valid under-cap image at the shared loader layer, preserves MIME/content type, fails closed for HEIC/pixel/over-cap guard failures, and logs or preserves the underlying optimizer cause. Treat #74216-style runtime-dependency staging as complementary packaging work, not a replacement for the shared loader behavior.
