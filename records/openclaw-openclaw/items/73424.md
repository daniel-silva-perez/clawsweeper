## Summary

Keep open. Current main improves the missing-runtime-dependency path by loading `media-understanding-core` through staged public-surface runtime deps, but the shared media loader still has no original-image fallback when optimization throws, and open PR #74243 explicitly closes this issue with that policy.

## What This Changes

The issue asks OpenClaw to stop failing JPEG image analysis before the VLM call when shared image preprocessing reports `Failed to optimize image`.

## Best Possible Solution

Review and either land #74243 or an equivalent `src/media/web-media.ts` change that falls back only for valid under-cap originals, preserves the original MIME/content type, keeps HEIC conversion, decoded-pixel, over-cap, and unsafe-path failures closed, and preserves actionable optimizer diagnostics. Keep #74392 as the separate packaged scan/doctor repair follow-up.

## Reproduction Assessment

Yes. The issue and comments provide concrete LINE, WhatsApp, Telegram, and local `loadWebMedia()` reproductions, and current source confirms the image tool still calls `loadWebMedia()` before the provider call.

## Solution Assessment

Yes, the paired PR direction appears to be the best available fix boundary: fallback at the shared media-loader layer can preserve MIME and safety guards, while returning original bytes directly from `optimizeImageToJpeg()` would be broader and risks mislabeled output.
