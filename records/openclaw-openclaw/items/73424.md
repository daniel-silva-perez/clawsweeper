## Summary

Keep open. Current main fixes part of the missing-runtime-dependency path, but the shared media loader still has no guard-aware fallback for valid under-cap images when optimization throws, and this issue is explicitly paired with open closing PR #74243.

## What This Changes

The issue reports that JPEG image analysis fails before the VLM call because shared media preprocessing throws `Failed to optimize image`.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` change that falls back only for valid under-cap original images, preserves original MIME/content type, and keeps HEIC conversion, decoded-pixel limits, unsafe-path, unknown-dimension, and over-cap failures closed.

## Reproduction Assessment

Yes. The issue and comments provide concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, macOS, and Linux; current source confirms optimizer failures still abort shared media loading before the provider call.

## Solution Assessment

Yes, the guard-aware loader-level fallback proposed by #74243 is the best narrow fix. Returning original bytes directly from `optimizeImageToJpeg()` would risk MIME mismatch and weakening guard boundaries.
