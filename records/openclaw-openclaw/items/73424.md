## Summary

Keep open. Current main appears to address the missing `sharp` runtime-dependency path, but the shared media loader still has no original-image fallback when optimization throws, and open PR #74243 explicitly closes this issue with the conservative fallback policy.

## What This Changes

The issue asks the image/VLM media preprocessing pipeline to stop failing JPEG/PNG analysis with `Failed to optimize image` before the provider call.

## Best Possible Solution

Keep the runtime-dependency staging fix, then review and either land #74243 or an equivalent `src/media/web-media.ts` change that falls back only for valid under-cap originals, preserves MIME/content type, fails closed for HEIC conversion, pixel-limit, over-cap, and unsafe-path guard failures, and keeps actionable optimizer diagnostics.

## Reproduction Assessment

Yes. The issue and comments provide concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, and local `loadWebMedia()` checks, and current source confirms the image tool still reaches the provider only after shared media loading succeeds.

## Solution Assessment

Unclear. Current main’s runtime-dependency staging is the right fix for the missing-`sharp` root cause, but the user-facing resilience policy is still represented by open PR #74243 and should be reviewed before closing this issue.
