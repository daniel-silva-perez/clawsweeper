## Summary

Keep open. Current main still routes image-tool inputs and shared channel image media through the optimizing `loadWebMedia` path, and the optimizer can still collapse total backend failure into the exact `Failed to optimize image` error. PR #74243 is an open closing implementation for this issue, so the issue should remain open until that PR merges or maintainers explicitly choose a different fix.

## What This Changes

The issue requests fixing the image/media preprocessing pipeline so valid in-limit JPEG images do not fail with `Failed to optimize image` before the configured vision model is called.

## Best Possible Solution

Review and land #74243 or an equivalent `src/media/web-media.ts` fix that falls back only for valid under-cap originals, preserves MIME/content type, keeps HEIC/pixel/over-cap guard failures closed, and preserves the underlying optimizer cause. Treat packaged `sharp` runtime-deps staging as complementary rather than a substitute for shared loader resilience.
