## Summary

Keep open. Current main partially addresses the packaged missing-`sharp` path and improves optimizer diagnostics, but it still has no loader-level fallback that returns valid under-cap original image bytes when optimization throws. The issue is also explicitly paired with open closing PR #74243, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports JPEG/PNG image analysis failing before the VLM call with `Failed to optimize image` in the shared media preprocessing path.

## Best Possible Solution

Review and either land #74243 or an equivalent `src/media/web-media.ts` change that falls back only for valid under-cap original images, preserves MIME/content type, keeps HEIC conversion, pixel-limit, unsafe-path, and over-cap failures closed, and retains actionable optimizer diagnostics. Keep the packaged runtime-dependency scan/doctor follow-up in #74392 separate unless maintainers decide it must land first.

## Reproduction Assessment

Yes. The issue and comments provide concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, macOS, Linux, and direct `loadWebMedia()` checks; current source confirms an optimizer throw can still abort shared media loading before the provider call.

## Solution Assessment

Unclear for current main. The runtime-dependency mitigation is useful for missing `sharp`, but the best narrow fix for the remaining user-visible failure is the guard-aware loader-level fallback in #74243 rather than a broad fallback inside `optimizeImageToJpeg()`.
