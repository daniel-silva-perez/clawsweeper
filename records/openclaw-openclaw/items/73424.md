## Summary

Keep open. Current main appears to mitigate the missing `sharp` runtime-dependency path for fresh packaged installs, but the shared media loader still has no original-image fallback when optimization throws, and open PR #74243 explicitly closes this issue with the more conservative fallback policy.

## What This Changes

The issue requests fixing the image preprocessing path so valid JPEG/PNG images from channels such as LINE reach the configured vision model instead of failing with `Failed to optimize image` before the provider call.

## Best Possible Solution

Review and either land #74243 or an equivalent `src/media/web-media.ts` change that falls back only for valid under-cap originals, preserves MIME/content type, fails closed for HEIC conversion, pixel-limit, over-cap, and unsafe-path guard failures, and keeps actionable optimizer diagnostics.

## Reproduction Assessment

Yes. The issue and comments provide concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, and local `loadWebMedia()` checks, and current source confirms the image tool still reaches the provider only after shared media loading succeeds.

## Solution Assessment

Unclear. Current main’s runtime-dependency staging is the right direction for the missing-`sharp` root cause, but the user-facing resilience policy is still represented by open PR #74243 and should be reviewed before closing this issue.
