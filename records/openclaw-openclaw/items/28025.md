## Summary

Keep open. Current main still does not implement a Slack `channels.slack.thread.replyOnToolUse` option or an equivalent tool-use-gated reply target. Slack thread routing is still selected from `replyToMode`, existing incoming thread state, and streaming eligibility before tool callbacks observe tool activity.

## Best Possible Solution

Keep this issue open. If maintainers accept the behavior, implement it in the Slack plugin as a default-false typed config option, carry a reliable per-turn tool-use signal into reply target planning before the first Slack transport action, preserve explicit thread replies and existing `replyToMode`/`thread.inheritParent` semantics, update Slack docs/config reference, and add Slack tests covering DMs, channels, existing threads, streaming/native streaming, progress callbacks, and `replyToMode` interactions.
