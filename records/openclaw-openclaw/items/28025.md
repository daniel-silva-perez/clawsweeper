## Summary

Keep open. Current main still does not implement `channels.slack.thread.replyOnToolUse` or an equivalent tool-use-gated Slack reply target. Slack reply routing is still controlled by `replyToMode`, incoming thread state, and reply plans created before tool callbacks run.

## Best Possible Solution

Keep this open for maintainer product review. If accepted, implement it in the Slack plugin as a default-false typed config option, carry a reliable per-turn tool-use signal into Slack reply target planning before the first Slack transport action, preserve existing explicit thread replies and `replyToMode`/`replyToModeByChatType`/`thread.inheritParent` semantics, update Slack docs and generated config metadata, and add focused Slack tests for DMs, channels, existing threads, native streaming, draft preview streaming, progress callbacks, and reply-mode interactions.
