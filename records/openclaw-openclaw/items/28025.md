## Summary

Keep open. Current main still does not implement `channels.slack.thread.replyOnToolUse` or an equivalent tool-use-gated Slack reply target. Slack reply/thread routing is still planned from `replyToMode` and incoming thread state before tool callbacks observe tool activity, so the requested top-level conversational replies plus threaded tool-use replies behavior remains missing.

## Best Possible Solution

Keep this open for maintainer product review. If accepted, implement it in the Slack plugin as a default-false typed config option, carry a reliable per-turn tool-use signal into reply target planning before the first Slack transport action, preserve existing explicit thread replies and `replyToMode`/`thread.inheritParent` semantics, update Slack docs/config metadata, and add focused Slack tests for DMs, channels, existing threads, streaming/native streaming, progress callbacks, and reply-mode interactions.
