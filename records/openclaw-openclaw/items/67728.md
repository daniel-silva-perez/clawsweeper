## Summary

Keep open. Current main has useful remote-CDP hardening, and v2026.4.26 even documents a limited fix, but the remaining post-open transport failure is still not implemented: `withCdpSocket` retries only handshake/open failures and current tests assert that a mid-command socket close uses one callback, one connection, and throws `CDP socket closed`. The related implementation PR #67748 was closed unmerged, so there is still actionable browser-plugin work here.

## What This Changes

Requests remote CDP WebSocket auto-reconnect/backoff, or an equivalent proxy/configurable retry path, so browser actions recover from transient CDP drops in WSL2 and other remote Chrome setups.

## Best Possible Solution

Keep the shipped handshake and safe-read recovery, then add bounded recovery for recoverable post-open transport closes in the browser plugin. The end state should evict stale cached CDP/Playwright connections for later actions, retry only operations classified as safe/idempotent, leave ambiguous mutating commands single-shot, and cover close-after-open, mid-command close, next-action recovery, retry exhaustion, and unsafe non-replay with targeted tests.
