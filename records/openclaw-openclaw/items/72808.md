## Summary

Keep open. Current main still has a hardcoded finite Slack Socket Mode reconnect cap, the provider still throws when that cap is reached, and the issue discussion includes current-main reproduction plus live logs. There is also an open unmerged PR with closing syntax for this issue, so the issue should remain open until that PR lands or maintainers choose the upstream Slack SDK update path.

## What This Changes

The issue reports Slack becoming silently non-responsive after working for several days, while the OpenClaw web interface remains usable.

## Best Possible Solution

Land a Slack-owned transport fix that removes or configures the finite Socket Mode reconnect cap while preserving fail-fast behavior for non-recoverable auth errors, then add regression coverage for repeated disconnect/start-failure behavior and validate the upstream Slack SDK reconnect fix once the dependency is updated.

## Reproduction Assessment

Yes. The discussion provides current-main reproduction and live logs, and the current source has a deterministic path where repeated Socket Mode disconnects or start failures eventually hit the hardcoded `maxAttempts: 12` guard.

## Solution Assessment

Unclear. Removing the cap is the narrow fix for permanent reconnect exhaustion, but the best final solution should also account for the upstream Slack SDK pong stall and transient reconnect-window delivery failures described in later comments.
