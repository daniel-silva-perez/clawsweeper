## Summary

Keep open. The report remains valid on current main: Slack Socket Mode still has a hardcoded finite reconnect cap, current config/docs expose ping timeout tuning but not reconnect attempts, and the discussion includes current-main reproduction plus live logs. There is also an open unmerged PR with closing syntax for this issue, so the issue should stay open until that PR or the upstream Slack SDK path lands.

## What This Changes

Slack Socket Mode can stop responding after long-running reconnect or pong-timeout instability, leaving web chat usable while Slack DMs and mentions receive no replies.

## Best Possible Solution

Land a Slack-owned transport fix that removes or configures the finite Socket Mode reconnect cap while preserving non-recoverable auth fail-fast behavior, add regression coverage for repeated disconnect/start-failure and status behavior, and update the Slack SDK when the upstream pong/reconnect fix is available and validated.

## Reproduction Assessment

Yes. The discussion gives a current-main reproduction path and live logs, and the code has a deterministic unit-level path: repeated Socket Mode disconnect or start-failure events eventually hit the hardcoded `maxAttempts: 12` guard.

## Solution Assessment

Unclear whether the open one-line uncap is the complete fix. It is a narrow fix for permanent reconnect exhaustion, but the best final solution should also account for the upstream Slack SDK pong stall and transient reconnect-window delivery failures described in later comments.
