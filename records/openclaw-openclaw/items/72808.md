## Summary

Keep open. Current main still hardcodes a finite Slack Socket Mode reconnect cap and throws when that cap is reached; the discussion includes current-main reproduction/logs plus an open unmerged PR with closing syntax, so the reported Slack outage is not implemented away on main.

## What This Changes

The issue asks OpenClaw to recover from or clearly surface Slack Socket Mode connection loss instead of silently dropping Slack messages while web chat continues to work.

## Best Possible Solution

Review #73162 as the current implementation path, land or adjust it with Slack-owned regression coverage for repeated disconnect/start-failure behavior and status reporting, then validate the upstream Slack SDK reconnect fix when it reaches the dependency version used by OpenClaw.

## Reproduction Assessment

Yes. The discussion provides current-main reproduction and live logs, and the current code has a deterministic path where repeated Socket Mode disconnect or start-failure events hit the hardcoded 12-attempt guard.

## Solution Assessment

Unclear whether uncapping alone is the complete fix. It is the narrowest fix for permanent reconnect exhaustion, but the full solution should also account for the upstream pong-stall behavior and transient reconnect-window delivery failures described in later comments.
