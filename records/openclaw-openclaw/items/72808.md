## Summary

Keep open. Current main and the latest release still retain the hardcoded finite Slack Socket Mode reconnect cap, and the issue already has a linked open fix PR plus maintainer discussion about validating the upstream Slack SDK pong-stall path before closing.

## What This Changes

Slack Socket Mode can silently stop replying after long-lived connection failures while web chat remains usable and Slack still shows the bot online.

## Best Possible Solution

Keep this issue open until #73162 or a replacement lands with Slack-owned coverage for repeated disconnect/start-failure behavior, then validate a Slack SDK update that includes the upstream pong-stall fix. If transient delivery loss during reconnect windows remains afterward, split that into a narrower follow-up.

## Reproduction Assessment

Yes. The current source has a deterministic path where repeated Socket Mode disconnect or start-failure events increment `reconnectAttempts` until the hardcoded `maxAttempts: 12` guard throws, and the discussion includes current-main reproduction plus live logs for the same failure family.

## Solution Assessment

Unclear whether the open uncap alone is the complete solution. It is the narrowest fix for permanent reconnect exhaustion, but the full resolution should also account for the upstream Slack SDK pong stall and reconnect-window delivery failures described in later comments.
