## Summary

Keep open. Current main still has a hardcoded finite Slack Socket Mode reconnect cap and no Slack reconnect-attempt config override, while the discussion includes current-main reproduction/logs plus an open implementation PR with closing syntax. This should be handled by reviewing that PR and the upstream Slack SDK fix path, not by closing this issue now.

## What This Changes

Slack Socket Mode can stop replying after long-running reconnect or pong failures, leaving web chat usable while Slack DMs and mentions receive no response.

## Best Possible Solution

Review #73162 as the current implementation path, land or adjust it with Slack-owned coverage for repeated disconnect/start-failure behavior and status reporting, then validate an upstream Slack SDK update when the pong-stall fix is available. If transient reconnect-window delivery loss remains after that, track it as a narrower follow-up.

## Reproduction Assessment

Yes. The discussion provides current-main reproduction and live logs, and the source has a deterministic path where repeated Socket Mode disconnect or start-failure events increment `reconnectAttempts` until the hardcoded `maxAttempts: 12` guard throws.

## Solution Assessment

Unclear whether the open uncap alone is the complete fix. It is the narrowest fix for permanent reconnect exhaustion, but the complete solution should also account for the upstream Slack SDK pong stall and transient reconnect-window delivery failures described in later comments.
