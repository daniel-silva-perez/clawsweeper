## Summary

Keep open. Current main still has the hardcoded finite Slack Socket Mode reconnect cap, the Slack config/docs do not expose a reconnect-attempt override, and the discussion includes current-main reproduction plus an open closing PR, so this issue should remain the tracker until that PR or a replacement lands and the upstream Slack SDK path is validated.

## What This Changes

The issue asks OpenClaw to stop losing Slack connectivity silently after long-running Socket Mode heartbeat or reconnect failures while other surfaces remain usable.

## Best Possible Solution

Keep this issue open until #73162 or a replacement lands with Slack-owned coverage for repeated disconnect/start-failure behavior and status reporting, then validate a Slack SDK update that includes the upstream pong-stall fix. If transient delivery loss during reconnect windows still remains afterward, track that as a narrower follow-up.

## Reproduction Assessment

Yes. The source has a deterministic path where repeated Socket Mode disconnect or start-failure events increment `reconnectAttempts` until the hardcoded `maxAttempts: 12` guard throws, and the issue discussion adds current-main/live logs for that failure family.

## Solution Assessment

Unclear whether the open uncap alone is the complete fix. It is the narrowest fix for permanent reconnect exhaustion, but the complete solution should also account for the upstream Slack SDK pong stall and transient reconnect-window delivery failures described in later comments.
