## Summary

Keep open. Current main still has the hardcoded finite Slack Socket Mode reconnect cap and no Slack reconnect-attempt config override, while the discussion includes current-main reproduction evidence and open PR #73162 with closing syntax for this issue.

## What This Changes

The report asks OpenClaw to stop silently losing Slack connectivity after a long-running Socket Mode connection becomes unresponsive while web chat remains healthy.

## Best Possible Solution

Keep this issue open until #73162 or a replacement lands with Slack-owned coverage for repeated disconnect/start-failure recovery and status reporting, then validate the Slack SDK update path for the pong-stall behavior. If reconnect-window message loss remains after that, track it as a narrower follow-up.

## Reproduction Assessment

Yes. Repeated Socket Mode disconnect or start-failure events deterministically increment `reconnectAttempts` until the current hardcoded `maxAttempts: 12` guard throws, and the issue discussion adds current-main/live logs for that failure family.

## Solution Assessment

Unclear. Removing or configuring the finite reconnect cap is the narrowest fix for the permanent dead-forever state, but the complete solution should also account for the upstream Slack SDK pong stall and transient reconnect-window delivery failures described in later comments.
