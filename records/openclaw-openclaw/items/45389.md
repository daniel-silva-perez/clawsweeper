## Summary

Keep open. Current main still has adjacent timeout mitigations, but not the requested durable session-level circuit breaker that counts consecutive LLM timeouts across heartbeat/message/cron runs, marks or pauses the session as degraded, and emits an operator notification. Related PR #42933 remains an open implementation candidate, not proof that this is implemented.

## What This Changes

Requests a configurable per-session LLM timeout circuit breaker that stops automatic retries, marks the session degraded, and emits a system event or webhook after N consecutive timeouts.

## Best Possible Solution

Keep this issue open and consolidate it with the active session-breaker work. The desired end state is a documented per-agent/default timeout breaker threshold with persisted per-session timeout state across heartbeat, cron, and interactive runs; success/operator reset clears the state; tripping the breaker pauses or marks the session degraded; and a sanitized system event or webhook tells operators what happened.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. The missing capability is high-confidence from source inspection: current strict config schemas, session state, heartbeat/cron paths, and timeout tests lack the requested durable breaker surface.

## Solution Assessment

Unclear for the exact proposed shape. A persisted configurable session-level breaker is the right direction, but maintainers should decide the pause/reset/alert semantics and likely review or rework #42933 rather than treating existing timeout compaction/failover as sufficient.
