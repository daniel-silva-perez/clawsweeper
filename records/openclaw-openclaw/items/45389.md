## Summary

Keep open. Current main has adjacent mitigations for high-context timeout recovery, cron backoff/failure alerts, tool-loop blocking, and Active Memory recall timeouts, but it still does not implement the requested durable session-level LLM timeout circuit breaker with default/per-agent configuration, persisted timeout streak state, degraded or paused session handling, and operator notification. PR #42933 remains open and unmerged, so the remaining action is maintainer consolidation rather than cleanup closure.

## What This Changes

The issue requests a configurable session-level circuit breaker that stops automatic LLM retries after consecutive timeouts, marks the session degraded, and emits an operator notification.

## Best Possible Solution

Keep this feature request open and consolidate it with the active session-breaker work. The desired end state is a documented core agent/session breaker with default and per-agent configuration, persisted per-session timeout/error streak state across heartbeat, cron, and interactive runs, success/manual reset clearing the streak, a clear degraded or paused state, and sanitized system event or webhook notification for operators.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. The high-confidence check is source inspection: current main lacks the requested config fields, durable session state, and heartbeat/cron/session enforcement paths, while existing timeout tests cover only in-run compaction fallthrough.

## Solution Assessment

Unclear for the exact proposed config shape. A persisted configurable session-level breaker is the right direction, but maintainers should decide pause/reset/alert semantics and review or rework PR #42933 rather than treating current timeout compaction, cron cleanup, or Active Memory recall guards as sufficient.
