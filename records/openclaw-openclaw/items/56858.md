## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested opt-in media caching behavior, so the work is not obsolete; however, the PR branch is stale against the current `pi-hooks` context-pruning path and the patch still has concrete correctness issues in the disk-cache and marker handling paths.

## What This Changes

The PR adds an optional `cacheMedia` context-pruning setting, a disk media cache module and tests, cached-media markers for pruned tool-result media, schema/baseline updates, and image-marker rehydration in the embedded runner.

## Best Possible Solution

Port the implementation onto the current `pi-hooks` context-pruning files, keep the setting default-off, reuse or align with existing media-store access and local-root semantics, make cache writes atomic with bounded cleanup, and add focused tests for multi-image, soft-trim plus hard-clear, MIME normalization, and marker rehydration before merge.

## Reproduction Assessment

Yes. Current-main inspection reproduces the missing feature: `cacheMedia` is absent from settings/schema/docs, pruning still strips image blocks to text, and the image parser has no cached-media marker path; PR-specific failures can be reproduced from the diff with stale-path, concurrent-write, and MIME-normalization unit cases.

## Solution Assessment

No. The product direction may be reasonable, but this patch is not the best mergeable solution because it targets a stale path and reimplements cache persistence without atomic write, cleanup, and current local-media policy follow-through.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Port the cache hook to the active pruning path:** `src/agents/pi-extensions/context-pruning/extension.ts:1`
  - body: Current main loads context pruning from `src/agents/pi-hooks/context-pruning/extension.ts`, but this patch changes the old `src/agents/pi-extensions/context-pruning/*` files. After a rebase/merge onto current main, the runtime hook would still call the unchanged `pi-hooks` implementation, so `cacheMedia` would be accepted by config but never cache pruned media.
  - confidence: 0.93
- **[P2] Write cache files atomically:** `src/media/media-cache.ts:39-47`
  - body: `cacheMediaToDisk()` checks for the destination with `access()` and then writes it in place. A concurrent caller can observe the path after the first writer created it but before the write is complete, return that path, and let the image loader read a partial file. Write to a unique temp file and `rename()` into place, or create with exclusive open and handle `EEXIST`.
  - confidence: 0.88
- **[P2] Normalize MIME before computing cached media kind:** `src/media/media-cache.ts:67-68`
  - body: `mediaCacheKind()` passes the raw MIME string to `mediaKindFromMime()`, which expects normalized lower-case MIME values. Values like `IMAGE/PNG` or `image/png; charset=binary` become `document`, and the image rehydration parser ignores cached markers unless `kind=image`, so cached images can be silently left as text.
  - confidence: 0.9
- **[P2] Preserve only cached marker tokens during hard clear:** `src/agents/pi-extensions/context-pruning/pruner.ts:16-30`
  - body: `extractCachedMediaMarkers()` keeps the entire text block when it contains a cached-media marker. If a prior soft-trim block contains head/tail tool output plus a cached marker, a later hard-clear keeps that surrounding text too, defeating the clear and leaving large old tool-result text in context.
  - confidence: 0.78
- **[P3] Gate cached-media prompt guidance on the feature:** `src/agents/system-prompt.ts:456-457`
  - body: The new system-prompt section is added for every agent even though `cacheMedia` defaults to false. That adds prompt tokens and tells agents to look for markers that cannot exist for the default configuration; pass the effective setting into prompt construction or inject this guidance only when the feature is enabled.
  - confidence: 0.77

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, or package-publishing changes were found, but the diff adds persistent local media writes and model-visible cached paths that need security/resource-bound review before merge.

Concerns:

- **[low] Unbounded persistent media cache:** `src/media/media-cache.ts:27`
  - body: The new cache writer stores unique pruned media under the OpenClaw config media directory, but the diff does not implement TTL cleanup, size limits, or eviction despite the PR body describing TTL-based cleanup. Long media-heavy sessions could grow local disk usage without bound when `cacheMedia` is enabled.
  - confidence: 0.84
- **[low] Non-atomic cache writes:** `src/media/media-cache.ts:39`
  - body: The `access()` then `writeFile()` sequence can expose an incomplete destination to a racing caller that returns the path as soon as the file exists. This is primarily a data-integrity issue, but for a security-sensitive local media cache it should use atomic temp-file replacement.
  - confidence: 0.86
