## Summary

Keep open. Current main still maps the no-provider/no-attempt media-understanding path to generic file-path no-output errors, and three open PRs already reference this issue with closing syntax, so the issue should remain open until one canonical fix lands.

## What This Changes

The issue requests clearer missing-provider configuration errors for `infer audio transcribe` and `infer image describe` instead of generic no-output messages that name the input path.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors for providers that actually ran, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and `infer image describe-many` to emit actionable configuration errors with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current main shows the deterministic path: no resolved provider returns a skipped/no-output decision, the runtime converts that to `text: undefined`, and the CLI throws the generic file-path error.

## Solution Assessment

Is this the best way to solve the issue? Yes for the requested behavior: the narrow maintainable fix is to distinguish missing-provider/no-attempt from provider-ran-but-empty-output at the media runtime or CLI boundary. Which open PR should land is a maintainer review choice.
