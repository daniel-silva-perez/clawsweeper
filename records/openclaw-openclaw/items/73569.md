## Summary

Keep open. Current main still maps the no-provider/no-attempt media-understanding path to the generic no-description/no-transcript file-path errors, and three open unmerged PRs already reference this issue with closing syntax, so the issue should stay open until one canonical fix lands.

## What This Changes

The issue requests clearer `infer audio transcribe` and `infer image describe` errors when no relevant provider is configured, instead of generic no-transcript/no-description path errors.

## Best Possible Solution

Land one canonical fix that preserves the current empty-result errors for providers that actually ran, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and `infer image describe-many` to emit actionable configuration errors with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current main shows the deterministic path: no resolved provider returns a skipped/no-output decision, the runtime converts that to `text: undefined`, and the CLI throws the generic file-path error.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable solution is to distinguish missing-provider/no-attempts from provider-ran-but-empty-output at the media runtime or CLI boundary, then keep the existing generic errors only for true empty provider responses.
