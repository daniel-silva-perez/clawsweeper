## Summary

Keep open. Current main still conflates a skipped no-provider media-understanding run with a provider returning empty output, and three open PRs already target the bug with closing syntax, so the maintainer action is to choose and land a canonical fix rather than close or queue another repair.

## What This Changes

The issue requests clearer missing-provider configuration errors for `infer audio transcribe` and `infer image describe` when no relevant media provider is configured, while preserving the existing empty-output errors after a provider actually runs.

## Best Possible Solution

Land one focused fix that preserves provider-called-but-empty transcript/description errors, carries or detects the no-provider/no-attempts state at the media runtime or infer CLI boundary, and adds regression coverage for image describe, image describe-many, and audio transcribe.

## Reproduction Assessment

Yes. The reporter provided deterministic CLI steps, and current-main source gives a high-confidence static reproduction path: no resolved media entries become a skipped empty result, which the CLI converts into the generic path-blaming error.

## Solution Assessment

Yes, the narrow maintainable solution is to separate missing provider/model configuration from provider-empty-output at the media runtime or CLI boundary. Closing this issue or starting another automated repair is not the best path while the linked fix PRs remain open.
