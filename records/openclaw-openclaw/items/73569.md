## Summary

Keep open. Current main still turns the no-provider media-understanding path into the same generic no-description/no-transcript file-path error, and three open PRs already reference this issue with closing syntax, so this should stay open until one canonical fix merges.

## What This Changes

The issue requests clearer missing-provider configuration errors for `infer audio transcribe` and `infer image describe`, while preserving the existing empty-result errors when a configured provider actually returns no text.

## Best Possible Solution

Land one canonical fix that carries missing-provider/no-attempt state far enough for the infer CLI to emit actionable audio and image-understanding configuration errors, with focused tests for audio transcribe, image describe, and image describe-many.

## Reproduction Assessment

Yes. Current main can be reproduced from source with high confidence: no resolved media provider becomes a skipped decision with no outputs, `runMediaUnderstandingFile` returns `text: undefined`, and the CLI converts that into the generic file-path error.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: distinguish missing-provider/no-attempts from provider-ran-but-empty-output at the media runtime or CLI boundary, then keep the existing empty-result errors for real provider responses.
