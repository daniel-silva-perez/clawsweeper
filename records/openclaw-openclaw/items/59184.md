## Summary

Keep open. Current main does not implement the PR's central chief task ledger, inbound receipt continuity, Paperclip-backed tracked work, or chief continuation runner, and the draft patch still has blocking correctness and security concerns that need maintainer review before it can merge.

## What This Changes

The draft PR adds broad chief/task orchestration, inbound receipt ledgers, Paperclip issue sync, Telegram task-continuity delivery, quality-guard/status-tag handling, runtime hardening, Windows-oriented scripts, package scripts, and many tests.

## Best Possible Solution

Split this into narrow owner-reviewed PRs: keep generic ledgers and continuation runners in core behind private or versioned runtime seams, keep Telegram-owned behavior in the Telegram plugin, avoid exposing Paperclip credential helpers through broad SDK barrels, package only supported scripts, and add docs/API-baseline coverage for any public contract that remains.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep Paperclip helpers out of the public SDK:** `src/plugin-sdk/channel-runtime.ts:27`
  - body: The new `export * from "../infra/paperclip-issues.js"` publishes helpers that read the local Paperclip API key and send it as a Bearer token to a caller-supplied `apiUrl`. Because `channel-runtime` is a plugin-facing compatibility shim, this turns an internal credentialed client into a public plugin API and should stay behind a narrow core-owned runtime seam instead.
  - confidence: 0.84
- **[P2] Package the added receipt reconciliation script:** `package.json:1115`
  - body: `package.json` adds `reconcile:chief-receipt-state`, but the `files` whitelist only adds the archive and task-authority scripts. Published installs will not contain `scripts/reconcile-chief-receipt-state.ts`, so the new package script will fail outside the repo checkout.
  - confidence: 0.9
- **[P2] Load the requested config through supported APIs:** `scripts/archive-chief-task-ledger.ts:46-47`
  - body: The added scripts pass `configPath` into `loadConfig`, but the current exported `loadConfig()` takes no arguments. This should fail type checking and, at runtime, would ignore the requested `--config` path; set `OPENCLAW_CONFIG_PATH` before loading or use the config IO API that accepts an explicit path.
  - confidence: 0.88
