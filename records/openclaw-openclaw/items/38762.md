## Summary

Keep open. The misleading browser-service reachability wrapping appears improved for service-side 400s, but current main still does not support the reported mixed `act` parameter shape: when `request` is an object, top-level `ref` is ignored, the body is forwarded without a merge, and `/act` validation still fails because the body lacks `ref` or `selector`. The related compatibility PR #38790 was closed unmerged and no open PR currently tracks the fix.

## Best Possible Solution

Land a narrow browser-tool compatibility normalizer: when `action: "act"` receives an object `request`, clone it and fill only missing `request.ref`, `request.targetId`, and finite numeric `request.timeoutMs` from top-level params; preserve nested request precedence, keep flattened legacy behavior unchanged, and add focused tests plus a user-facing changelog entry.

Fix the browser `act` mixed-parameter compatibility bug reported in openclaw/openclaw#38762. Current behavior: `browser action="act" ref="e146" request={"kind":"click"}` ignores the top-level `ref` because `readActRequestParam` returns object `params.request` unchanged, so `/act` receives `{ kind: "click" }` and rejects it. Do not change the browser control service contract, gateway/browser runtime behavior, auth, networking, or route validation. Add focused tests in `extensions/browser/src/browser-tool.test.ts` for: top-level `ref` filling a nested click request, nested `request.ref` taking precedence over top-level `ref`, top-level `targetId` filling only when missing, finite top-level `timeoutMs` filling only when missing, and non-finite timeout values not being copied. Add a single-line user-facing changelog fix entry crediting the reporter and the prior PR contributor as appropriate, following repository changelog attribution rules.
