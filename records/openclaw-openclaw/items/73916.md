## Summary

Keep open. The uploaded diagnostics make the 2026.4.26 old-state Gateway hang credible, but current main only proves adjacent observability and startup hardening; it does not prove the old-state post-bind HTTP/WS starvation is fixed, especially because the sanitized logs show both a no-plugin sidecar stall and later Bonjour churn.

## What This Changes

The issue reports that on Intel macOS the Gateway can bind `127.0.0.1:18789` but never answer HTTP or WebSocket requests when started with existing `~/.openclaw` state.

## Best Possible Solution

Keep the issue open for diagnostics-driven triage: retest the preserved old state on current main or the latest release, capture startup trace/diagnostics around `sidecars.total`, and then patch the smallest owner path so a stuck sidecar, channel, or discovery advertiser cannot leave the Gateway bound but unresponsive without clear diagnostics.

## Reproduction Assessment

Unclear: the diagnostics give a strong 2026.4.26 reproduction shape with old local state, but there is no public fixture or live Intel macOS state in the checkout. A high-confidence reproduction needs the saved broken state or a minimized config/state trigger from it.

## Solution Assessment

No: closing as implemented or queuing a blind repair is not the best path yet. The safer solution is to identify whether the stall is in channel startup, internal hooks, Bonjour/mDNS, runtime-deps staging, or another sidecar phase, then add a focused regression test for that trigger.
