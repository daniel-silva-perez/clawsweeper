Keep open. Current main contains an adjacent fix for the credential-free `tools.web.fetch` pre-bind regression tracked in #74896, but I did not find high-confidence proof that it fixes this report's Intel macOS old-state path where the listener exists yet HTTP/WebSocket diagnostics still time out.

## What This Changes

The issue reports that an Intel macOS Gateway using an existing `~/.openclaw` state directory binds `127.0.0.1:18789` but then stops answering HTTP/WebSocket requests, while a clean state directory works.

## Best Possible Solution

Keep this open until a maintainer retests the preserved old state against current main or extracts a sanitized minimal state/config trigger. If the current web-fetch fix resolves it, close with that provenance; otherwise add a focused regression test around the specific post-bind service, state loader, or provider-discovery path that blocks the event loop.
