## Summary

Keep open. Current main has the Codex ACP timeout guard, but the auth half remains unresolved: the acpx Codex wrapper still forces an isolated CODEX_HOME, creates only generated config there, and regression coverage still asserts source Codex auth is not copied. The open #73943 PR references this issue with closing syntax, so this should wait for that PR or a successor to merge.

## What This Changes

The issue asks OpenClaw-managed Codex ACP to stop failing by either safely reusing or documenting Codex auth for the isolated CODEX_HOME and by avoiding unsupported timeout config handoff to Codex ACP.

## Best Possible Solution

Keep the issue open until #73943 or a successor lands a narrow Codex ACP auth path: bridge only the required Codex auth.json or provide a documented isolated-home login flow, preserve generated config isolation, keep the current Codex timeout guard, and update regression tests plus the public ACP docs.

## Reproduction Assessment

Yes. The issue has concrete reproduction steps and an independent macOS confirmation, and current main source/tests still reproduce the relevant precondition: an isolated Codex home is forced while source auth.json is intentionally absent from generated ACP auth paths.

## Solution Assessment

Unclear as a merged solution because the remaining fix is not on main. The best direction is a narrow, security-reviewed #73943-style auth bridge or documented isolated-home login flow, not closing this issue while the linked closing PR remains open.
