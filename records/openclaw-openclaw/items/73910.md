## Summary

Keep open. Current main has the Codex ACP timeout guard, but the isolated CODEX_HOME auth half remains: the ACPX Codex wrapper still forces a generated codex-home, writes only config.toml there, and current tests still assert source Codex auth is not bridged. The linked open #73943 PR is the right place to review or replace the auth fix before this issue closes.

## What This Changes

OpenClaw-managed Codex ACP should safely reuse or document Codex auth for its isolated CODEX_HOME and avoid sending unsupported timeout config to Codex ACP.

## Best Possible Solution

Keep the timeout guard already on main, then land or revise #73943 as a narrow Codex ACP auth fix: bridge only the required Codex auth.json or document a supported isolated-home login flow, preserve generated config isolation, add regression coverage, and update the public ACP docs.

## Reproduction Assessment

Yes. The source-level reproduction is clear: set a source CODEX_HOME with auth.json, call prepareAcpxCodexAuthConfig, and inspect the generated codex-home/wrapper; current tests still expect no auth bridge, while runtime tests cover the fixed timeout path.

## Solution Assessment

Unclear until #73943 or a successor receives maintainer security review. The narrow auth.json bridge or a documented isolated-home login flow is the maintainable direction, but it must keep config hooks and broader Codex state out of the generated home.
