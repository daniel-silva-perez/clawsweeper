## Summary

Keep open. Current main still matches the report: completion announce dispatch preserves the direct-send failure only in phase data, returns the successful queued or steered fallback as the top-level delivery result, and the caller/lifecycle paths do not surface that recovered direct-primary error.

## What This Changes

Report a warning or diagnostic when a subagent completion announce direct-send fails but queue fallback later recovers delivery.

## Best Possible Solution

Add a narrow recovered-delivery diagnostic that warns once when completion dispatch has a failed direct-primary phase with an error and a later queue-fallback phase delivers, while preserving successful delivery semantics and existing retry behavior.

## Reproduction Assessment

Yes. The unit-level path is clear: run completion dispatch with direct returning delivered=false/path=direct/error and queue returning queued or steered, then observe the top-level delivered=true result and the caller's top-level-only log guard.

## Solution Assessment

Yes. The narrowest maintainable fix is to inspect dispatch phases after delivery or add a small dispatch logger/callback seam; changing fallback delivery semantics would be the wrong fix because the announce did recover.

## Repair Work Prompt

Fix openclaw/openclaw#62968. Current completion-mode subagent announce dispatch can return delivered=true/path queued or steered after a direct-primary phase failed with an error, and the caller/lifecycle do not surface that recovered direct-send error. Add a narrow warning/diagnostic for this recovered path without marking the announce failed and without duplicating transient retry logs from runAnnounceDeliveryWithRetry. Prefer implementing near runSubagentAnnounceFlow after delivery is returned, or via a small dispatch callback/logger seam if that fits existing patterns. Add focused Vitest coverage for delivered=true with path queued or steered plus phases containing failed direct-primary and delivered queue-fallback. Do not change queue fallback semantics, delivery status, public plugin APIs, or unrelated announce routing behavior. Changelog is not needed unless the maintainer wants user-facing diagnostics called out.
