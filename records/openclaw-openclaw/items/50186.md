## Summary

Keep open. Current main already uses `KillMode=control-group` for newly rendered Linux systemd gateway units, and the latest release tag contains that partial default. The main remaining request is still not implemented: systemd runtime/status and doctor audit paths do not collect or surface `KillMode`, `TasksCurrent`, `MemoryCurrent`, or a hygiene warning for degraded cgroups. The issue has concrete production evidence, so the stale label is not a good close reason.

## Best Possible Solution

Keep this issue open and implement the remaining observability as a focused daemon/systemd PR: extend the Linux systemd runtime/audit path to read `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`; show those values in gateway/status surfaces; add a conservative doctor warning and remediation hints for `KillMode=process` plus suspicious task or memory counts. Leave the existing `KillMode=control-group` default unchanged unless maintainers explicitly choose a `mixed`/migration policy.
