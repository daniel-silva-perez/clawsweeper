## Summary

Keep open. Current main has already addressed the generated Linux systemd unit default by emitting `KillMode=control-group`, including in `v2026.4.27`, but the main diagnostics request remains unimplemented: doctor/status still do not collect, model, print, or warn on `KillMode`, `TasksCurrent`, or `MemoryCurrent`.

## What This Changes

The issue requests Linux systemd gateway hygiene diagnostics in `openclaw doctor` and `openclaw gateway status --deep`, plus consideration of safer generated gateway unit `KillMode` defaults.

## Best Possible Solution

Add a read-only Linux systemd hygiene collector shared by gateway status and doctor/service audit: request and parse `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`; include the values in deep status/JSON; warn on suspicious `KillMode=process` plus high task or memory counts; and print concrete `systemctl --user show`, `systemd-cgls --user-unit`, restart, and explicit repair guidance without silently migrating existing units.

## Reproduction Assessment

Yes for the missing diagnostic behavior: mocked `systemctl show` output with `KillMode=process`, high `TasksCurrent`, and high `MemoryCurrent` would be ignored today because the runtime query/parser and status/audit types do not request or model those fields. The original production cgroup growth was not live-reproduced in this read-only review.

## Solution Assessment

Yes for the remaining work: a narrow read-only doctor/status observability PR is the best next step because the generated-unit default is already `KillMode=control-group`. Any `mixed` versus `control-group` policy change should be handled as a separate maintainer decision.
