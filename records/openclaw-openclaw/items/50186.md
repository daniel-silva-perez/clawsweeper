## Summary

Keep open. Current main already uses a safer generated Linux systemd unit default, but the main requested diagnostics remain missing: systemd runtime/status and doctor service-audit paths still do not collect, model, print, or warn on `KillMode`, `TasksCurrent`, or `MemoryCurrent`.

## What This Changes

Request to add Linux systemd gateway hygiene visibility to `openclaw doctor` and `openclaw gateway status --deep`, plus consider safer generated unit `KillMode` behavior.

## Best Possible Solution

Add a read-only Linux systemd hygiene collector shared by gateway status and doctor/service audit: request and parse `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`; include them in deep status/JSON; warn when `KillMode=process` combines with suspicious task or memory counts; and print concrete `systemctl --user show`, `systemd-cgls --user-unit`, restart, and explicit repair guidance without silently migrating existing units.

## Reproduction Assessment

Yes for the missing diagnostic behavior: mocked `systemctl show` output containing `KillMode=process`, high `TasksCurrent`, and high `MemoryCurrent` would be ignored today because the runtime query/parser and status/audit types do not request or model those fields. The original production cgroup growth was not live-reproduced in this read-only review.

## Solution Assessment

Yes for the remaining work: a narrow read-only doctor/status observability PR is the most maintainable next step because the generated-unit default is already `KillMode=control-group`. Any `mixed` versus `control-group` migration policy should remain a separate maintainer decision.
