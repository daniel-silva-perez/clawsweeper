## Summary

Keep open. Current main and v2026.4.27 already render newly generated Linux systemd gateway units with `KillMode=control-group`, but the requested doctor/status visibility is still missing: the runtime reader, service audit, and deep status paths do not collect or warn on `KillMode`, `TasksCurrent`, or `MemoryCurrent`. The remaining work is a bounded read-only diagnostics improvement, so no close reason applies.

## What This Changes

Request Linux systemd gateway hygiene diagnostics in `openclaw doctor` / `openclaw gateway status --deep`, plus consideration of safer generated-unit `KillMode` defaults.

## Best Possible Solution

Add a read-only Linux systemd hygiene collector shared by gateway status and doctor/service audit: report `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`; warn on suspicious `KillMode=process` plus high task or memory counts; include concrete `systemctl --user show`, `systemd-cgls --user-unit`, and restart commands; keep existing-unit migration as a separate explicit repair decision.

## Reproduction Assessment

Yes for the code path: feed mocked `systemctl show` output with `KillMode=process`, high `TasksCurrent`, and high `MemoryCurrent` through the systemd runtime/audit/status paths, and current main has no field model or warning path to surface it.

## Solution Assessment

Is this the best way to solve the issue? Yes for the remaining work: a read-only doctor/status warning is the narrowest maintainable next step, while silent migration of existing units should stay separate and maintainer-reviewed.

## Repair Work Prompt

Implement the remaining diagnostics for https://github.com/openclaw/openclaw/issues/50186. Current main already renders new Linux systemd gateway units with `KillMode=control-group`; do not change that default in this PR, and do not silently migrate existing user units. Add a read-only Linux systemd hygiene collector used by `openclaw gateway status --deep` and `openclaw doctor`/service audit. It should collect or parse `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` from `systemctl --user show openclaw-gateway*.service`, expose the values in status JSON/human output where appropriate, and warn when `KillMode=process` combines with suspicious task or memory counts. Warning text should include concrete next-step commands such as `systemctl --user show openclaw-gateway.service -p KillMode -p TasksCurrent -p MemoryCurrent -p MainPID`, `systemd-cgls --user-unit openclaw-gateway.service`, and `systemctl --user restart openclaw-gateway.service`. Keep thresholds conservative and centralized. Add regression tests with mocked systemctl output; update docs for gateway status/doctor if human output or JSON shape changes; add a user-facing changelog entry crediting the reporter/commenters, not bots. Do not touch unrelated stale-port cleanup, Slack, JSON file locking, or open PR #47776 behavior.
