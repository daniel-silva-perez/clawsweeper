## Summary

Keep open. Current main and the latest release already cover the safer generated systemd unit default with `KillMode=control-group`, but the central remaining request is still missing: doctor/status do not collect or surface `KillMode`, `TasksCurrent`, `MemoryCurrent`, or a conservative cgroup hygiene warning.

## What This Changes

Request Linux/systemd gateway hygiene visibility in `openclaw doctor` and `openclaw status --deep`, plus consideration of safer generated unit KillMode defaults.

## Best Possible Solution

Keep this issue open and implement the remaining observability as a focused Linux/systemd doctor/status PR: read `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`; carry those fields through `GatewayServiceRuntime`; display them in deep gateway/status output; and emit a conservative doctor warning with concrete `systemctl --user show`, `systemd-cgls`, and restart/remediation hints when `KillMode=process` combines with suspicious task or memory counts. Leave the existing `KillMode=control-group` default unchanged unless maintainers separately choose a `mixed` or migration policy.
