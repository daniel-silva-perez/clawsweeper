## Summary

Keep open. Current main already uses a safer generated Linux systemd unit default, but the main requested observability is still missing: systemd runtime/status and doctor/service-audit paths do not collect, model, print, or warn on `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so the reported cgroup-contamination failure mode would still be hard to see from OpenClaw diagnostics.

## What This Changes

Add Linux systemd gateway hygiene visibility to doctor/status and consider safer defaults or explicit repair for gateway systemd `KillMode`.

## Best Possible Solution

Add a read-only Linux systemd hygiene collector shared by gateway status and doctor/service audit: request and parse `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID`; print those values in deep status/JSON; warn when `KillMode=process` combines with suspicious task or memory counts; include concrete `systemctl --user show`, `systemd-cgls --user-unit`, restart, and explicit repair guidance without silently migrating existing units.

## Reproduction Assessment

Yes for the code path: mocked `systemctl show` output containing `KillMode=process`, high `TasksCurrent`, and high `MemoryCurrent` would be ignored today because the runtime query and parser do not request or model those fields. The production cgroup-growth scenario was not live-reproduced in this read-only review, but current source provides a high-confidence focused reproduction for the missing diagnostic behavior.

## Solution Assessment

Yes for the remaining work: read-only doctor/status observability is the narrowest maintainable next step because the generated-unit default is already `KillMode=control-group`. Silent migration or another default-policy change should be handled separately after maintainer review of the `mixed` versus `control-group` tradeoff.
