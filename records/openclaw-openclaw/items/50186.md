## Summary

Keep open. Current main already uses a safer generated systemd unit default, but the requested doctor/status hygiene visibility is still missing: the Linux systemd runtime path does not collect KillMode, TasksCurrent, or MemoryCurrent, and the service audit/status surfaces only expose existing service/runtime/config checks.

## What This Changes

Request to add Linux systemd gateway hygiene diagnostics to doctor/status and evaluate safer KillMode defaults for generated or repaired gateway units.

## Best Possible Solution

Add a read-only Linux systemd hygiene collector used by gateway status and doctor/service audit: report KillMode, TasksCurrent, MemoryCurrent, and MainPID; warn on suspicious KillMode=process plus high task or memory counts; include concrete inspection and restart commands; leave existing-unit migration explicit and prompted.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the code path: mock or provide systemctl show output with KillMode=process, high TasksCurrent, and high MemoryCurrent, and current main has no field collection or warning path to surface it.

## Solution Assessment

Is this the best way to solve the issue? Yes for the remaining work: a read-only doctor/status warning is the narrowest maintainable next step, while any existing-unit KillMode migration should stay explicit and owner-reviewed.

## Repair Work Prompt

Implement the remaining Linux systemd gateway hygiene visibility for this issue. Add a read-only collector for the gateway systemd unit that can surface KillMode, TasksCurrent, MemoryCurrent, and MainPID in `openclaw gateway status --deep` and `openclaw doctor`/service audit. Use the existing systemd user-scope helpers, profile/OPENCLAW_SYSTEMD_UNIT handling, and resilient parsing patterns; do not shell out through ad hoc strings. Warn conservatively when the combination is suspicious, especially KillMode=process with unusually high TasksCurrent or MemoryCurrent while the gateway probe/runtime otherwise looks healthy. Include concrete next steps such as `systemctl --user show openclaw-gateway.service -p KillMode -p TasksCurrent -p MemoryCurrent -p MainPID`, `systemd-cgls --user-unit openclaw-gateway.service`, and `openclaw gateway restart`. Keep the current generated default `KillMode=control-group`; do not silently migrate existing units or switch to `KillMode=mixed` unless maintainers explicitly choose that policy. Update user-facing docs for doctor/gateway status and add a changelog entry with Thanks @openclawq and @Hollychou924. Add focused unit tests for parsing, status JSON/human output, and doctor audit warnings.
