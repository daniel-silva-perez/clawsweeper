## Summary

Keep open: current main and the latest release still keep exponent notation out of `parseConfigValue`'s numeric branch, while the PR is focused and I found no definite correctness or security defect in the final diff. The remaining decision is whether maintainers want unquoted exponent-looking values coerced to numbers on these command paths.

## What This Changes

The PR extends `parseConfigValue` numeric detection to accept exponent notation and adds colocated Vitest coverage for existing value types plus scientific notation.

## Best Possible Solution

Land or adapt the focused parser change with the added unit coverage if maintainers agree that unquoted exponent notation should be treated as numeric input on `/config`, `/mcp`, `/debug`, and `openclaw mcp set`; otherwise close the PR after documenting that exponent-looking input is intentionally preserved as a string on those paths.
