## Summary

Keep this draft PR open. Current main and v2026.4.29 still do not implement the PR's lower Active Memory defaults or QMD startup jitter, and the branch introduces a blocking hook-timeout regression for valid custom `timeoutMs` values above the new default.

## What This Changes

The PR lowers Active Memory default timeout/setup-grace values, changes the registered `before_prompt_build` hook timeout, adds randomized QMD startup staggering, updates related tests, and revises Active Memory docs toward lower recall budgets.

## Best Possible Solution

Revise the branch so the hook envelope is derived from the resolved per-agent timeout or documented maximum, then make a maintainer decision on the lower default/setup-grace policy and whether QMD startup should use jitter, a concurrency limit, lazy startup, or a config knob before landing against #72015.

## Reproduction Assessment

Yes for the main PR regression: configure Active Memory with `timeoutMs` above 3000ms, then the proposed registration timeout at `DEFAULT_TIMEOUT_MS + setupGraceTimeoutMs + 1000` can expire before the plugin's own recall watchdog. The current hook runner proves this because registration timeouts are enforced with `Promise.race`.

## Solution Assessment

No, not as proposed. Lowering the default recall budget may be a reasonable direction, but the hook timeout must not be fixed to the default when current config allows values up to 120000ms; QMD startup also needs a deliberate concurrency/jitter policy rather than a small random delay wrapped in broad `Promise.all`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Size the hook timeout from the configured recall budget:** `extensions/active-memory/index.ts:2434`
  - body: This registers every Active Memory `before_prompt_build` hook with a fixed 4s timeout, while `normalizePluginConfig` still accepts larger per-agent `config.timeoutMs` values and the docs recommend 5000ms+ for full mode. The hook runner enforces this registration timeout before the plugin's own watchdog can fire, so valid custom recall configs will silently drop memory context after 4s. Use the resolved `config.timeoutMs` or the 120000ms cap when computing the hook envelope, and add a test with a custom timeout above the default.
  - confidence: 0.92
