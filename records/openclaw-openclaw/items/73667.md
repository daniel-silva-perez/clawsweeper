## Summary

Keep this PR open. Current main has related Active Memory timeout work for #72606, but it does not implement this PR's central #72015 changes: the default recall timeout is still 15s, setup grace is still 30s, the hook budget remains 150s, docs still recommend 15s for recent/full modes, and QMD startup still has no jitter path. The PR is also draft, paired with open issue #72015 via closing syntax, and has an unresolved functional review concern.

## Best Possible Solution

Keep this draft PR open as an implementation candidate for #72015. The best path is to revise the hook envelope so it respects the resolved per-agent timeout or documented cap, make a deliberate maintainer decision on the default timeout and setup-grace policy, add deterministic tests for custom timeoutMs values above the default, and add targeted QMD startup jitter/concurrency coverage before landing or replacing it with a narrower maintainer-owned fix.

## Risks / Open Questions

- Merging the PR as-is can silently skip Active Memory recall for users with config.timeoutMs above 3000ms because the proposed hook timeout is fixed at 4000ms while the current contract permits up to 120000ms.
- The QMD startup change should be reconciled with current eager/deferred startup semantics and QMD manager boot behavior; the PR's Promise.all plus small random delay is not enough evidence of bounded startup concurrency.
