## Summary

Keep this PR open. Current main has related Active Memory timeout work for #72606, but it does not implement this PR's central changes for #72015: the default recall timeout is still 15s, setup grace is still 30s, docs still show 15s budgets for recent/full modes, and QMD startup has no jitter implementation. The PR is also draft and has an unresolved functional review concern around the proposed hook timeout envelope.

## Best Possible Solution

Keep this PR open for maintainer review as an implementation candidate for #72015. The best path is to revise the patch so the hook envelope respects the resolved per-agent timeout or documented cap, deliberately decide the setup-grace/default timeout policy, preserve current QMD startup deferral semantics while adding bounded jitter or concurrency control, and add targeted Active Memory plus gateway startup tests before landing.

## Risks / Open Questions

- The proposed hook timeout in the PR context uses DEFAULT_TIMEOUT_MS + setupGraceTimeoutMs + 1_000, which would be 4_000ms with the PR defaults and can undercut valid per-agent timeoutMs values above 3_000ms, including the PR's own docs guidance for full mode.
- The QMD startup change should be reconciled with current main's existing eager/deferred agent startup policy so it does not accidentally replace maintainer-owned deferral behavior.
- The jitter behavior needs deterministic test coverage around startup ordering/timing or concurrency before landing.
