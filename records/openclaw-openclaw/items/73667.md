## Summary

Keep this draft PR open. Current main has related Active Memory timeout hardening, but it does not implement the PR's central changes: the default recall timeout remains 15s, setup grace remains 30s, the hook envelope remains 150s by default, Active Memory docs still recommend 15s for recent/full/default examples, and QMD startup still initializes eager agents sequentially with no jitter. The PR is also paired with open issue #72015 by the same author and still has an unresolved functional review concern about truncating valid custom timeoutMs values.

## Best Possible Solution

Keep this draft PR open as an implementation candidate for #72015. The best path is to revise the hook envelope so it respects the resolved per-agent timeout or documented cap, make a deliberate maintainer decision on the default timeout and setup-grace policy, and add deterministic tests for high custom timeoutMs values plus QMD startup jitter/concurrency behavior before landing or replacing it with a narrower maintainer-owned fix.

## Risks / Open Questions

- Merging the PR as-is can silently skip Active Memory recall for users with config.timeoutMs above 3000ms because the proposed hook timeout is fixed at 4000ms while current main permits up to 120000ms.
- The QMD startup change uses random jitter plus Promise.all but does not add an explicit concurrency limit, so it should be reconciled with current eager/deferred startup semantics before landing.
