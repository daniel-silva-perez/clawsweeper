## Summary

Current main still does not implement an operator approval gate for agent-facing `gateway config.patch` outside the narrow allowlist. The model-facing gateway tool continues to reject non-allowlisted config paths before forwarding the RPC, while the approval subsystem is still scoped to exec and plugin approvals, not config mutation approvals.

## Best Possible Solution

Keep this open for maintainer/security design. The best path is a first-class config mutation approval policy: preserve the strict default, classify paths as always allowed, promptable, or never promptable, bind approval to the exact base hash and patch/diff, show an operator-readable diff, optionally support a reviewed config allowlist, and update Telegram/Discord/native approval delivery, gateway protocol schemas, docs, and regression tests.

## Risks / Open Questions

- Broadening `config.patch` with a generic approval could let a model alter auth, plugin loading, MCP servers, hooks, sandbox settings, tools, browser paths, or command execution policy after one poorly scoped approval.
- Exec approvals are documented as reducing accidental execution risk, not as a per-user auth boundary; reusing that UX for config writes needs config-specific approver, expiry, replay, and exact-diff semantics.
- The design should coordinate with open #39528, which asks for recent consent before currently allowed model-initiated gateway writes/restarts/updates, so OpenClaw does not grow two inconsistent control-plane approval models.
