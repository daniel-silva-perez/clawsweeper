## Summary

Keep open. Current main still enforces a narrow fail-closed guard for agent-facing `gateway config.patch` and rejects non-allowlisted paths before forwarding the RPC, while the approval system only models exec and plugin approvals. The request is valid, but it changes a GHSA-related model-to-operator trust boundary and needs maintainer/security design rather than conservative cleanup closure.

## What This Changes

Request to add an optional operator approval flow for agent-facing `gateway config.patch` when a patch touches paths outside the current allowlist, with support for an operator-configured expanded allowlist.

## Best Possible Solution

Add a first-class config mutation approval policy: keep strict rejection as the default, define explicit path classes, present an operator-readable diff, bind approval to the exact base hash and patch, support a reviewed expanded allowlist, and cover Telegram/Discord/native delivery plus gateway-tool/protocol tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call the agent-facing `gateway` tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current code computes changed paths and throws before forwarding the RPC, and tests encode those blocked cases.

## Solution Assessment

No, not as a direct reuse of exec approvals. The maintainable solution is a config-specific approval surface because config writes need path classification plus exact diff/base-hash semantics that exec/plugin approvals do not currently model.
