## Summary

Keep open. Current main still intentionally rejects agent-driven `config.patch` mutations outside a narrow allowlist, and there is no config-specific approval surface; the request is valid as a product/security design item, but it should not be auto-closed or queued as an autonomous repair.

## What This Changes

The issue requests an optional operator approval gate and configurable expanded allowlist for `gateway config.patch` changes outside the current strict allowlist.

## Best Possible Solution

Add a first-class config mutation approval policy: keep strict rejection as the default, classify paths as always allowed, promptable, or never promptable, show an operator-readable diff, bind approval to the exact base hash and patch, support an explicit reviewed allowlist, and cover Telegram/Discord/native approval delivery plus gateway-tool and protocol tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call the agent-facing `gateway` tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current code throws before forwarding the RPC, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approvals for every non-allowlisted config patch is not the safest maintainable solution; this needs config-specific approval semantics with path classification, exact diff/base-hash binding, and explicit never-promptable paths.
