## Summary

Keep open. Current main still enforces a narrow fail-closed allowlist for agent-facing gateway config.patch/config.apply and has no first-class config approval kind, approval RPC, or configurable configPatch allowlist for promptable paths; the remaining request is a security-sensitive product/design change, not stale cleanup.

## What This Changes

Request an optional operator approval gate when agent-facing gateway config.patch targets paths outside the existing allowlist, plus an optional expanded allowlist for trusted paths.

## Best Possible Solution

Add a first-class config mutation approval policy: keep the strict default, classify paths as always allowed, promptable, or never promptable, show an operator-readable diff, bind approval to the exact base hash and patch payload, support a reviewed config allowlist for trusted paths, and cover Telegram/Discord/native delivery plus gateway-tool/protocol tests.

## Reproduction Assessment

Yes. A static reproduction is to invoke the gateway tool with config.patch changing a protected path such as tools.exec.ask or plugins.allow; current tests assert the tool throws and never forwards config.patch to the gateway RPC.

## Solution Assessment

Unclear. The requested direction is useful, but reusing exec approval as-is is not the best complete fix; a config-specific approval contract with path policy and diff/base-hash binding is the safer maintainable solution.
