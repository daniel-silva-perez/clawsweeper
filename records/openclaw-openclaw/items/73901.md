## Summary

Current main still intentionally fails closed for agent-driven protected config mutations, and the requested approval gate would broaden a GHSA-related model-to-operator trust boundary. This should stay open for maintainer/security design rather than be closed as implemented or queued as an autonomous repair.

## What This Changes

The issue requests an optional operator approval gate for agent-facing `gateway config.patch` writes outside the current config mutation allowlist.

## Best Possible Solution

Add a first-class config mutation approval policy: keep strict rejection as the default, classify paths as always allowed, promptable, or never promptable, display an operator-readable diff, bind approval to the exact base hash and patch, add an explicit reviewed allowlist for operators, and cover Telegram/Discord/native approval delivery plus gateway-tool and protocol tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call the agent-facing `gateway` tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current code computes changed paths and throws before forwarding the RPC, and tests encode those blocked cases.

## Solution Assessment

No, not as a direct reuse of exec approvals for every non-allowlisted patch. The maintainable solution is config-specific approval semantics with path classification, exact diff/base-hash binding, and explicit never-promptable paths.
