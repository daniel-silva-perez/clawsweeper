## Summary

Keep this PR open. Current main still writes timestamped `.clobbered.*` snapshots in both config persistence paths without any per-path cap, and existing hash-based suspicious-read dedupe only suppresses identical payloads. The PR remains a plausible defensive fix, though the count-then-write concurrency review should be resolved before landing.

## Best Possible Solution

Keep this PR open. The best path is to finish the defensive cap in both `src/config/io.ts` and `src/config/io.observe-recovery.ts`, address the concurrent count/write race, keep audit logging independent of snapshot persistence, and land focused regression coverage for async and sync paths, max file count, and one warning per path.

## Risks / Open Questions

- The PR's async cap counts existing files and writes the new snapshot in separate operations, so concurrent suspicious reads for the same path can overshoot 32 unless the implementation serializes per path or uses an atomic slot/allocation approach.
- Because the cap intentionally skips later forensic snapshots, tests should also prove the audit record remains complete and warning output is bounded per path for both async and sync recovery paths.
