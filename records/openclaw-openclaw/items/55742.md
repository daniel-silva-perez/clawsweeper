## Summary

Keep open. Current main still does not implement execute-time plugin tool context injection: `agentId` and session fields are resolved into the plugin tool factory context, but `resolvePluginTools()` returns the plugin tool objects without wrapping `execute`, and the Pi tool adapter still calls tools with only `toolCallId`, params, `AbortSignal`, and `onUpdate`. The PR addresses a real plugin API gap, but it needs a refreshed review/rebase rather than cleanup closure.

## Best Possible Solution

Review and land a refreshed, narrowly scoped version of this PR: add a public typed execute-context contract in the plugin tool SDK surface, wrap plugin-owned tools in `resolvePluginTools()` so `execute(callId, params, signal, onUpdate, context)` receives a frozen identity/session snapshot, preserve tool metadata and existing runtime args, add focused tests for fifth-argument injection and no config/secret leakage, and drop or separately rebase the unrelated compaction-safeguard changes.

## Risks / Open Questions

- Security review: the intended change would expose trusted agent/session identity to every plugin tool execute method, so the implementation should keep the context identity-only, frozen/read-only, and exclude config, filesystem paths, credentials, and other secret-bearing fields.
- Functional review: the wrapper must preserve own enumerable tool fields, plugin metadata, `AbortSignal`, and `onUpdate`; the PR discussion shows these were real failure modes in earlier commits.
- Maintenance risk: the PR contains an unrelated compaction auth hunk against an old path, while current main already has a newer `pi-hooks` implementation that preserves request headers.
