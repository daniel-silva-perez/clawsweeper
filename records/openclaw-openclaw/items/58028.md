## Summary

Keep this open. Current main does not implement a top-level `/resume` command or the requested previous-session picker/restore flow. The only documented resume command is `/codex resume`, which controls the Codex harness, and the existing gateway restore API is limited to compaction checkpoints, not arbitrary reset/archived sessions. The directly related PR #36481 was closed unmerged, and related restore work remains an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep the issue open and implement or review a focused first-party session resume/restore flow in the core chat command surface, likely spanning `src/auto-reply` command registration/handling plus gateway session-store/archive support. The shipped design should list recent reset/archived sessions, support a safe selection by id or `last`, and update docs/tests for the user-facing `/resume` behavior.

## Risks / Open Questions

- The implementation should distinguish top-level OpenClaw session resume from `/codex resume` and compaction checkpoint restore so users do not get a misleading partial feature.
- Restoring archived JSONL transcripts should include authorization, session ownership, active-run interruption, and transcript integrity handling; related PR #51889 may be useful implementation context if maintainers want this feature.
