## Summary

Keep open. Current main still lacks a proactive fallback-chain test command: the CLI and `/models` slash surfaces do not register a `test-fallback` action, `models status --probe` remains an auth-profile probe, and the runtime fallback resolver still begins with the effective primary before configured fallbacks. The direct implementation PR #43342 is closed unmerged, while #63145 and #17876 are broader model-health or migration workflow requests rather than exact substitutes.

## What This Changes

Add a `/models test-fallback` command, with an optional all-chain mode, to verify configured model fallback candidates without waiting for a real primary-model failure.

## Best Possible Solution

Ship a shared fallback-chain probe that reuses OpenClaw's real model/auth resolution, intentionally begins at configured fallback candidates, reports the responding provider/model plus per-candidate failures, supports first-fallback and all-fallback modes, and documents how it differs from auth-profile probes and broader health checks.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is static inspection of current main's CLI registration, slash parser/handler, docs, tests, and fallback runtime, which shows the requested command is absent.

## Solution Assessment

Yes. A first-class fallback-chain probe is the narrow maintainable direction; folding it silently into the existing auth-profile `models status --probe` path would blur auth reachability with configured fallback-chain execution unless maintainers intentionally combine it with #63145.
