## Summary

Keep open. Current main still does not provide a proactive fallback-chain test command: the slash `/models` path only routes provider/list/add flows, `openclaw models status --probe` remains an auth-profile probe, and the runtime fallback loop starts from the requested primary before configured fallbacks. The direct implementation PR is closed and unmerged, while the broader per-model health-check issue does not replace this narrower request.

## What This Changes

Request to add a `/models test-fallback` command, with an optional all-fallbacks mode, so users can verify configured model fallback chains without waiting for a real provider failure.

## Best Possible Solution

Ship a first-class fallback-chain probe that reuses OpenClaw's real model/auth fallback resolution, intentionally begins at configured fallback candidates rather than the primary, reports the responding provider/model plus per-candidate failures, supports first-fallback and all-fallback modes, and documents how it differs from auth-profile probes and broader per-model health checks.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is static inspection of the current CLI/slash command registration, docs, and fallback runtime, which shows the requested command is absent.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution if implemented as a shared fallback-chain probe rather than overloading auth-profile probes. A broader all-model health checker should remain tracked separately by #63145 unless maintainers decide to merge the surfaces intentionally.
