## Summary

Keep open. Current main still lacks a proactive fallback-chain test command: the CLI has status/list/set/fallback-management/auth-probe surfaces, the slash `/models` parser only dispatches provider/list/add flows, and `models status --probe` remains an auth-profile probe rather than a primary-skipping fallback walk. The direct implementation PR #43342 is closed unmerged, and #63145/#17876 are broader follow-up ideas rather than exact substitutes.

## What This Changes

Request to add a `/models test-fallback` command, with optional all-chain mode, to proactively verify configured model fallback candidates without waiting for a real primary-model failure.

## Best Possible Solution

Ship a shared fallback-chain probe that reuses OpenClaw's real model/auth resolution, intentionally begins at configured fallback candidates, reports the responding provider/model and per-candidate failures, supports first-fallback and all-fallback modes, and documents how it differs from auth-profile probes and broader health checks.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is static inspection of current main's CLI registration, slash parser/handler, docs, and fallback runtime, which shows the requested command is absent.

## Solution Assessment

Yes. A first-class fallback-chain probe is the narrow maintainable direction; hiding this under the existing auth-profile `models status --probe` path would blur two different checks unless maintainers explicitly decide to combine it with the broader #63145 health-check design.
