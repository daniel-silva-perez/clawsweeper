## Summary

Keep this issue open. Current main still has model fallback runtime and live `models status --probe` auth checks, but it does not ship `/models test-fallback` or `openclaw models test-fallback`, and the existing probe path does not intentionally skip the primary or walk configured fallback candidates. The directly related PR #43342 is closed and unmerged, so it is useful implementation context rather than a close reason.

## Best Possible Solution

Keep this open until maintainers add or explicitly decline a first-class fallback-chain probe. The best implementation is a `/models test-fallback` slash command plus an `openclaw models test-fallback` CLI equivalent that reuses the real fallback/auth resolution path with an intentional primary skip, supports first-fallback and all-fallback modes, respects active agent/session auth context, avoids session/config mutation, adds focused slash/CLI/runtime tests, and documents how it differs from `models status --probe`.

## Clownfish Work Prompt

Implement a first-class fallback-chain probe for this issue. Add `/models test-fallback` and an `openclaw models test-fallback` CLI equivalent that verifies configured text-model fallbacks without waiting for a real provider failure. Default mode should intentionally skip the primary and test only the first configured fallback; support an all-chain mode (`--all` for CLI and `all` or equivalent for slash) that tests each configured fallback in order. Reuse the real model/auth resolution path as much as practical, but do not mutate OpenClaw config, auth profiles, auth-state cooldowns, session model overrides, or transcripts beyond unavoidable temporary probe artifacts. Pin each probe to the fallback being tested so later fallbacks cannot mask an earlier failure. Respect active agent/session context, including agent-level fallback overrides and auth-profile/session auth context where available. Keep `models status --probe` semantics unchanged and document the difference: status probe checks provider/profile reachability; test-fallback validates the configured fallback chain. Related refs: this issue, closed unmerged #43342, related fallback work #6381, #4217, and #3253. Likely files include `src/auto-reply/reply/commands-models.ts`, `src/cli/models-cli.ts`, a new or existing `src/commands/models/*test-fallback*` helper, `src/agents/model-fallback.ts` only if a narrow generic seam is needed, tests under `src/auto-reply/reply/commands-models.test.ts`, `src/cli/models-cli.test.ts`, and focused fallback probe tests. Update `docs/cli/models.md`, `docs/tools/slash-commands.md`, possibly `docs/concepts/model-failover.md`, and add a user-facing changelog entry with `Thanks @jonwiley` plus any other directly credited discussion contributor. Do not broaden model fallback policy, do not change normal run fallback behavior, and do not add provider-specific logic to core.
