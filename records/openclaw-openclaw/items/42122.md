## Summary

Keep this PR open. Current main still does not implement the smoke-only Control UI build skip: the Dockerfile runs `pnpm ui:build` unconditionally, and the install-smoke workflow passes only `OPENCLAW_EXTENSIONS=matrix` to the root Dockerfile smoke image. The PR remains a plausible focused CI performance change, but it needs a rebase against the current split fast/full install-smoke workflow and careful scoping so placeholder Control UI assets cannot leak into package/update smoke paths.

## Best Possible Solution

Keep this PR open as a focused performance candidate. The best path is to rebase it onto current main, keep `OPENCLAW_SKIP_UI_BUILD` re-declared inside the Docker build stage, pass the skip arg only to the intended fast Docker smoke image build, and preserve real Control UI asset coverage for package/update and release-adjacent lanes.

## Risks / Open Questions

- The PR is stale against current main: the Dockerfile now uses Node 24 base images, runs `pnpm qa:lab:build` after `pnpm ui:build`, and install-smoke now has split fast/full paths with one matrix-extension image tagged as both root and extension smoke images.
- If `OPENCLAW_SKIP_UI_BUILD` is applied to the full install-smoke image, package/update smoke paths may pack placeholder Control UI assets because they copy `/app/dist` from `openclaw-dockerfile-smoke:local` and run `npm pack --ignore-scripts`.
- Security-sensitive review: the submitted diff changes Dockerfile build behavior and workflow build args only, with no new GitHub Actions permissions, third-party action refs, dependency sources, lockfile changes, lifecycle hooks, or secrets handling. The remaining risk is functional coverage: a public Docker build arg can intentionally produce an image with placeholder Control UI assets.
