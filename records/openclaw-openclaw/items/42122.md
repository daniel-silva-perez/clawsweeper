## Summary

Keep this PR open. Current main still does not implement the smoke-only Control UI build skip: the Dockerfile runs the full UI build unconditionally, and install-smoke does not pass an OPENCLAW_SKIP_UI_BUILD build arg. The PR remains a plausible focused performance change, but it needs a rebase and careful scoping because current install-smoke can reuse the Docker image dist for package/update smoke paths.

## Best Possible Solution

Keep the PR open as a focused performance candidate. The best path is to rebase it onto current main, keep OPENCLAW_SKIP_UI_BUILD re-declared inside the Docker build stage, pass the skip arg only to the intended smoke Docker image build, and validate that package/update lanes still exercise real Control UI assets where required.

## Risks / Open Questions

- Security/supply-chain review: the submitted diff does not add new Actions permissions, third-party action refs, dependency sources, lockfile changes, lifecycle hooks, or secrets handling. The main risk is functional coverage: a public Docker build arg can intentionally produce an image with placeholder Control UI assets, so it must remain limited to smoke-only image builds and not leak into release, package acceptance, or package/update smoke artifacts.
- The PR is stale against current Docker/install-smoke layout: current main uses Node 24 base images, tags one matrix-extension smoke image as both root and extension smoke images, and also runs pnpm qa:lab:build after pnpm ui:build.
- The current full install-smoke path can reuse /app/dist from openclaw-dockerfile-smoke:local for package/update smokes. If the skip is applied there, it may reduce coverage of packaged Control UI assets unless maintainers explicitly scope the skip to the intended fast Docker smoke path or add validation that real assets are still tested elsewhere.
