## Summary

Keep this PR open. Current main has not implemented the requested smoke-only UI-build skip: the Dockerfile still runs `pnpm ui:build` unconditionally, and the install-smoke workflow does not pass an `OPENCLAW_SKIP_UI_BUILD` build arg. The PR is now outdated against the current Docker/install-smoke layout, but it remains a plausible implementation candidate after rebase and maintainer review.

## Best Possible Solution

Keep the PR open as a focused performance candidate. The best path is to rebase it onto current main, keep the Docker ARG re-declared inside the `build` stage, pass the skip arg only to the intended smoke image builds, and validate the current install-smoke workflow so package/update lanes do not accidentally consume placeholder Control UI assets where real assets are required.

## Risks / Open Questions

- The submitted diff is against an older install-smoke/Dockerfile shape. Current main has Node 24 base images, a single matrix-extension smoke image build tagged both ways, and an additional `pnpm qa:lab:build` after `pnpm ui:build`, so the PR needs a real rebase rather than being closed as obsolete.
- Security review: the submitted diff does not add new GitHub Actions, permissions, secrets, dependency sources, lockfile changes, or remote installer execution. The relevant risk is functional/supply-chain coverage: a public Docker build arg can intentionally produce an image with a placeholder Control UI, so it must remain smoke-only and must not leak into release or package-acceptance image builds.
- Current full install-smoke reuses the root Docker image's `dist/` for package/update smoke paths. If the skip is applied to that full path, it may reduce coverage of packaged Control UI assets unless maintainers confirm those lanes only need CLI/runtime dist or scope the skip to the fast Docker smoke path.
