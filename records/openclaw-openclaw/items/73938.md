## Summary

Keep this PR open. Current main still resolves and passes `remote.batch.timeoutMinutes` into provider batch execution, but the shared memory-host HTTP helpers and OpenAI/Gemini/Voyage remote batch upload/create/status/error-read/download paths do not accept or forward a remaining timeout budget. The PR is a live replacement for the stale #49981 idea credited to @yintamaa, and it needs maintainer follow-up on the unresolved timeout/security review before landing.

## Best Possible Solution

Keep this PR open, address the timeout resolver/retry-loop review so expired budgets fail closed instead of disabling request timeouts, then land this PR or an equivalent narrow replacement that threads one remaining-time budget through memory-host SDK HTTP helpers and OpenAI/Gemini/Voyage batch upload, create, status/error-read, retry delay, and download phases. Preserve the #49981 credit, keep the fix scoped to remote provider-batch HTTP budget propagation, and validate with focused memory-host/provider-batch tests plus the changed gate.
