## Summary

Keep open. Current main still does not implement the monitor-client reuse proposed here: IRC proactive sends only reuse an explicitly supplied client, otherwise they create a transient connection and quit after PRIVMSG. The related #47234 stack is still open and unmerged, so it is not a shipped replacement. The PR diff is narrow and has no obvious supply-chain drift, but the implementation still needs maintainer review around process boundaries, JOIN readiness, and regression coverage.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused IRC delivery fix that reuses a verified joined monitor or gateway-owned connection when available, preserves the no-monitor transient fallback deliberately, handles CLI/process-boundary behavior explicitly, and adds targeted regression tests for monitor reuse, stop cleanup, fallback behavior, and startup JOIN timing. Coordinate with #47234 so the small IRC delivery fix can land separately from the broader groupScope routing proposal if maintainers want the bug fix first.
