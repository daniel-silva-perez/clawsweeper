## Summary

Keep this PR open. Current main still lacks the persistent IRC monitor-client reuse this PR proposes: proactive sends without an explicit ready client still create a transient IRC client, send PRIVMSG, and quit. The related same-author #47234 stack is open and includes broader routing work, so it is not a shipped supersession. The PR discussion also matters: the Greptile identity-cleanup concern was addressed in the PR head, but remaining process-boundary and JOIN-readiness concerns still need maintainer review.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused IRC transport fix that reuses a verified joined monitor or gateway-owned connection when available, preserves a clear no-monitor fallback, handles process boundaries intentionally, and adds targeted tests for reuse, fallback, identity-safe cleanup, and JOIN readiness. Coordinate with #47234 so the proactive-send fix can land separately from the groupScope routing proposal if maintainers want the smaller bug fix first.

## Risks / Open Questions

- A process-local client registry would not fix proactive IRC sends launched from a separate CLI process unless those sends are routed through a gateway-owned or otherwise shared monitor execution path.
- Publishing a monitor client as active immediately after connect can still race IRC channel membership because current readiness resolves after JOIN commands are sent, not after JOIN confirmation.
- Current main lacks targeted regression tests for monitor-client reuse, transient fallback, identity-safe cleanup, and JOIN/readiness timing.
- #47234 includes this transport fix plus broader groupScope routing behavior, so maintainers should avoid merging or closing this focused PR as superseded until the transport fix is separated, landed, or explicitly rejected.
