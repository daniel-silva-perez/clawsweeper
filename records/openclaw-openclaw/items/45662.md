Keep open. Current main still does not implement the monitor-client reuse this PR proposes, and the same-author related PR #47234 is open and unmerged rather than a shipped replacement. The PR is focused and has no obvious supply-chain drift, but it still needs maintainer judgment on process-boundary behavior, JOIN readiness, and targeted regression coverage.

## Best Possible Solution

Coordinate with #47234 so the small IRC delivery fix can land separately from the broader groupScope routing proposal if maintainers want the bug fix first.

## Risks / Open Questions

- Closing this PR now would discard a focused fix candidate while current main still has the transient IRC channel-send behavior the PR targets.
- A process-local active-client registry may not solve standalone CLI or other separate-process proactive sends unless outbound IRC sends are routed through the gateway or another monitor-owned process.
- Publishing a monitor client immediately after connect may still race channel membership because current readiness resolves after issuing JOIN commands, not after confirmed membership.
- The current PR needs targeted tests for monitor-client reuse, no-monitor transient fallback, identity-safe deregistration, and JOIN-ready publication before it is safe to land.
