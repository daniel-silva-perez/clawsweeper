## Summary

Keep open. Current main has manual ClawHub-backed skill/plugin update flows and an opt-in OpenClaw package auto-updater, but it does not implement the requested background auto-update policy for installed ClawHub skills and plugins. This is also security-sensitive enough that it should remain a maintainer/product design item rather than be swept into ClawHub-only cleanup.

## Best Possible Solution

Keep this issue open for maintainer product/security design. The most conservative path is to extend the existing manual `openclaw skills update --all` and `openclaw plugins update --all` primitives into an explicitly disabled-by-default gateway scheduler or documented update service, with trust/pinning, audit, rollback, and failure semantics before any silent background apply.

## Risks / Open Questions

- Auto-updating skills/plugins can silently pull new third-party code or instructions into user workspaces; the threat model already treats update poisoning as high residual risk.
- Current `openclaw update` can run plugin sync after a core update, which is useful partial coverage, but it does not address plugin/skill releases that happen between OpenClaw package releases.
- A good implementation likely needs explicit opt-in config, pin/trust semantics, rollback/audit logging, failure backoff, and clear separation between workspace skills and shared/global plugin installs.
