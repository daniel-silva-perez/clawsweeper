## Summary

Keep open. Current main has meaningful partial fixes for Telegram dispatcher failures, context-overflow fallback replies, exact NO_REPLY suppression, and managed gateway restart cleanup, but it still does not fully cover the reported cron no-op failure mode or make gateway PID/port mismatch self-heal from status/diagnostics without operator restart semantics.

## What This Changes

The issue asks OpenClaw to avoid silent Telegram non-replies on context/tool-loop failures, treat cron no-op outcomes as successful without a final NO_REPLY model call, and stabilize or self-heal gateway PID/port ownership mismatches.

## Best Possible Solution

Keep this open or split it into narrower follow-ups: verify the Telegram/context-overflow fallback path for the reported assistantError/terminated case, define a cron no-op terminal-result contract for explicit business outcomes like {send:false}, and decide whether gateway status/doctor/restart should actively reconcile non-owner listeners while the service runtime is still running.
