## Summary

Keep open. Current main has partial safeguards for Telegram dispatch failures, context-overflow fallback text, exact `NO_REPLY` suppression, and gateway restart cleanup, but it still does not fully cover the reported cron no-op timeout mode or make gateway PID/port mismatch self-heal from status/diagnostics. The issue is recent, has an independent corroborating Telegram/context-overflow comment, and spans multiple live reliability surfaces.

## What This Changes

The issue requests reliability fixes for Telegram no-reply after context overflow, cron no-op runs failing while emitting `NO_REPLY`, and gateway PID/port ownership mismatch.

## Best Possible Solution

Keep the partial safeguards already on main, then split or track the remaining work by subsystem: verify the Telegram assistantError/terminated fallback, define a cron terminal no-op contract for explicit no-send business outcomes, and decide whether gateway status/doctor/restart should actively reconcile non-owner listeners.
