## Summary

Keep open. The feature remains valid and current main still lacks a Brave Goggles parameter, but this PR now targets the stale core web-search implementation instead of the active Brave provider-owned schema/runtime, so it should not merge as-is.

The PR branch adds a Brave-only optional `goggles` web_search parameter, forwards it to Brave web and llm-context requests, separates cached results by Goggles value, and updates tests, docs, and changelog.

## Best Possible Solution

Land a rebased provider-owned port in `extensions/brave` that exposes a per-query `goggles` option, forwards it to both Brave endpoints, preserves exact Goggles values in cache separation, and updates the current provider docs, overview docs, focused tests, and credited changelog.

Yes. The current-main check is a static trace: `createWebSearchTool` delegates to `resolveWebSearchDefinition`, the runtime calls `provider.createTool`, and the active Brave provider has no `goggles` schema, forwarding, or cache-key handling.

## Solution Assessment

No. The requested capability is a good fit, but the PR’s current approach is no longer the narrow maintainable solution because it changes stale core files instead of the active Brave plugin-owned contract and runtime.
