## Summary

Keep open. The feature direction is valid, but current `main` moved `web_search` onto provider-owned contracts, while the PR still edits the old core web-search implementation; as written, the rebased patch would not expose or forward Brave Goggles through the active Brave provider.

## What This Changes

The PR adds a Brave-only `goggles` parameter to the old core `web_search` schema/runtime, forwards it to Brave web and llm-context requests, updates caching/tests, docs, and changelog.

## Best Possible Solution

Land a rebased port that exposes a Brave-only per-query `goggles` option from the active Brave provider, forwards it to both Brave web and llm-context endpoints, separates cached results by Goggles value without lowercasing it, and updates the public provider docs, overview docs, tests, and credited changelog.

## Reproduction Assessment

Yes. The current-main path is reproducible by static trace: `createWebSearchTool` delegates to `resolveWebSearchDefinition`, which calls `provider.createTool`, and the active Brave provider schema/runtime currently has no `goggles` field or forwarding.

## Solution Assessment

No. The requested capability is useful, but this PR is no longer the best implementation path because it changes stale core files instead of the active `extensions/brave` provider-owned contract and runtime.
