## Summary

Keep open. Current main still has memory search, promotion/dreaming previews, a destructive LanceDB forget tool, and an adjacent memory-wiki knowledge layer, but it does not provide first-class operator review/list, edit, non-destructive forget/tombstone/supersede, or conflict-resolution flows for canonical durable memory records. The KinthAI production comment supports the requested direction, and the related schema, write-pipeline, and operator-surface issues remain open.

## What This Changes

Add first-class operator flows to review/list, edit, non-destructively forget or supersede, and resolve conflicts for durable memory records.

## Best Possible Solution

Sequence this with #42646, #42648, and #42651: establish a canonical durable memory record/status/revision model, route writes through provenance-aware dedupe and conflict handling, then expose operator review/list, edit, tombstone/forget, supersede, and conflict-resolution flows through a small CLI/RPC or UI surface. Keep memory-wiki as a companion provenance/dashboard layer unless maintainers explicitly promote it into the canonical active-memory record system.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: source and docs inspection show the requested operator flows are absent, and the existing LanceDB forget path is explicitly hard-delete behavior.

## Solution Assessment

Unclear as a final implementation until the schema and write-pipeline issues are settled. The requested direction is maintainable only if it lands on a canonical durable memory model with revisions/statuses, not as an isolated wrapper around LanceDB deletion or memory-wiki metadata edits.
