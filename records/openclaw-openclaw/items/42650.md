## Summary

Keep this issue open. Current main still has memory search, short-term promotion/dreaming review output, LanceDB recall/store/forget tools, Active Memory recall, and a companion memory-wiki layer, but it does not provide the requested first-class operator review/list, edit, non-destructive forget/tombstone or supersede, and conflict-resolution flows for canonical durable memory records. The KinthAI production-context comment supports the requested direction rather than making the OpenClaw work obsolete.

## Best Possible Solution

Keep this issue open and implement it in the memory owner modules alongside #42646, #42648, and #42651. The likely path is a canonical durable memory record/status/revision model, provenance-aware write pipeline, and operator-facing CLI/RPC or UI flows for review/list, edit, tombstone/forget, supersede, and conflict resolution, with focused docs and tests. Memory-wiki concepts can inform dashboards and provenance, but should not be treated as a substitute unless maintainers explicitly make it the canonical memory-record surface.
