## Summary

Keep open. Current main has Telegram transport code and tests that should honor HTTP_PROXY/HTTPS_PROXY for Telegram API fetches, and the v2026.4.26 tag already contains that same env-proxy path. That makes this unsafe to close as implemented: the reported failure may be a packaged/runtime environment issue, service env inheritance issue, or the remaining ALL_PROXY/Telegram-specific env gap. Related PR #41795 is open and proposes a broader Telegram proxy resolver, including OPENCLAW_TELEGRAM_PROXY.

## Best Possible Solution

Keep this issue open. A maintainer should reproduce the packaged v2026.4.26 gateway with uppercase HTTP_PROXY inherited by the actual gateway process, then either document the required service env setup or refresh a narrow Telegram proxy PR. If support for ALL_PROXY or OPENCLAW_TELEGRAM_PROXY is desired, it belongs in the Telegram plugin transport layer with focused coverage for polling, send, probe, audit, and docs; #41795 is the closest existing implementation candidate but needs review against the current plugin layout and security expectations.
