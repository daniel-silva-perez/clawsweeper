## Summary

Keep this issue open. Current main still has no configurable image-tool local media root allowlist for external attachment directories: the image tool passes only default/workspace roots into local media loading, `tools.media.image` is a strict schema without `mediaLocalRoots`, and existing tests still preserve rejection of non-workspace host image paths. The OpenClaw-managed inbound-media case is now handled, and related PR #39034 remains useful context, but it is open/maintainer-labeled and not a merged fix for the requested config surface.

## What This Changes

The issue requests a configurable image-tool local media root allowlist so external attachment directories such as iMessage attachments can be read without copying files into the workspace.

## Best Possible Solution

Keep this open for maintainer/security design. The likely fix is an explicit opt-in image/media local-root allowlist, probably under the existing `tools.media.image` config surface or a narrowly documented channel-derived bridge, with strict absolute path validation, clear global/per-agent precedence, documented `tools.fs.workspaceOnly` behavior, tests for rejection and allowed external roots, and preservation of the current OpenClaw-managed inbound-media exemption.

## Risks / Open Questions

- A configurable external local-root allowlist changes the host file-read boundary for the image tool, so it needs strict absolute-path validation, narrow defaults, and clear rejection of filesystem-root or broad wildcard entries.

- Related PR #39034 overlaps the iMessage attachment-root direction and is maintainer-labeled, so a replacement or repair PR should not be queued without maintainer design direction.
