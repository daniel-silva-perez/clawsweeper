## Summary

Keep open. Current main still does not expose a public `tools.media.image.mediaLocalRoots`-style allowlist for the explicit image tool: the tool resolves only workspace/default local roots, the schema/types have no config field, and existing coverage still preserves rejection of outside local paths. The related open maintainer-labeled #39034 overlaps the iMessage attachment-root case, so this needs maintainer/security contract review rather than cleanup closure.

## What This Changes

The issue requests a configurable local media root allowlist for the image tool so iMessage attachment paths can be read without copying them into the OpenClaw workspace.

## Best Possible Solution

Define and document a narrow opt-in local media root contract for image/media tools, or a channel-owned attachment-root propagation contract if maintainers prefer that boundary. It should validate explicit non-filesystem-root directories, preserve managed inbound-media behavior and existing rejection coverage, integrate through the shared resolver/`loadWebMedia`, and update schema/types/docs/tests together.

## Reproduction Assessment

Yes. A focused reproduction path exists: call the explicit image tool with an absolute local image path outside the workspace/default roots and observe the current rejection, which is covered by `src/agents/tools/image-tool.test.ts:1336`. Managed inbound media has separate acceptance coverage, so external iMessage attachment directories remain the relevant failing path.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but the exact best fix needs maintainer/security judgment because it expands local read access and overlaps #39034's narrower channel attachment-root approach.
