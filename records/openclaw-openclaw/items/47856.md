## Summary

Keep open. Current main still lacks a public configurable local-media-root allowlist for the explicit image tool; the managed inbound-media exception helps OpenClaw-owned media, but it does not cover external iMessage attachment directories. Because the requested allowlist expands model-facing local filesystem reads and overlaps open maintainer-labeled PR #39034, this should stay with maintainer review.

## What This Changes

Request adding a `tools.image.mediaLocalRoots` or similar config field to extend the image tool's allowed local directories so iMessage attachment paths can be read without copying them into the workspace.

## Best Possible Solution

Define and document a narrow opt-in local media root allowlist, or a channel-owned attachment-root propagation contract if maintainers prefer that boundary, with explicit non-filesystem-root validation, clear `workspaceOnly` behavior, shared resolver integration, and regression tests for both configured acceptance and existing rejection cases.

## Reproduction Assessment

Yes. Current main has a focused reproduction path: call the explicit image tool with an absolute local image path outside the workspace/default media roots and observe the existing rejection covered by `src/agents/tools/image-tool.test.ts:1336`. Managed inbound media now has separate acceptance coverage, so external iMessage attachment directories remain the relevant failing path.

## Solution Assessment

Unclear. A configurable allowlist is plausible, but the exact best contract needs maintainer/security judgment because it expands local read access and overlaps #39034's narrower channel attachment-root approach.
