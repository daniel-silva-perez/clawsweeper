## Summary

Keep open. Current main has adjacent fixes for iMessage inbound attachment roots and managed inbound media, including the media/inbound case discussed in the comments and the related closed #36507 thread, but it still does not provide a configurable image-tool local-root allowlist. The explicit image tool still resolves local reads to default media roots plus the workspace, or only the workspace under workspaceOnly, and tests assert that arbitrary host paths remain rejected even when workspaceOnly is false.

## Best Possible Solution

Keep this open. Implement a constrained, documented config surface for explicit media tool local reads, likely under tools.media.image or a shared media tool policy, that safely extends image-tool local roots while preserving workspaceOnly semantics. Update zod schema, generated config metadata/docs, and tests for iMessage Messages attachment paths, managed media/inbound, and rejected unsafe roots.

## Risks / Open Questions

- Adding configurable local roots is security-sensitive host file access. It needs absolute-path validation, filesystem-root rejection, canonical/symlink checks, and clear interaction with tools.fs.workspaceOnly.
- The iMessage-specific attachmentRoots path should remain channel-owned; a core image-tool root setting should not bypass channel inbound-root validation for untrusted or mixed-trust deployments.
