## Summary

Keep this PR open. Current main still has the reported Google Chat threading gap, so the work is useful, but the submitted patch is not merge-ready because it reintroduces threading after off-mode filtering and changes shared implicit reply semantics in a way that can affect Telegram topics.

## What This Changes

The PR propagates Google Chat thread resource names through inbound context, auto-reply delivery, routed outbound delivery, and message-tool sends, adds a thread-resource validator, and expands core plus Google Chat regression tests.

## Best Possible Solution

Land a revised PR that keeps Google Chat resource-name normalization inside the Google Chat plugin boundary, preserves currentMessageId as the generic implicit reply target for other channels, and respects replyToMode across natural delivery, routed delivery, and message-tool sends with Google Chat off-mode plus Telegram topic regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep implicit replies on message ids:** `src/auto-reply/reply/agent-runner-utils.ts:169`
  - body: `resolveAutoThreadingTargets` now treats any `MessageThreadId` as the implicit `replyToId` for shared reply plumbing. Telegram stores forum/topic ids in `MessageThreadId`, while its send path parses `replyToId` as a message id and `threadId` as the topic id, so topic replies can target the topic id as a message reply instead of preserving message/thread separately.
  - confidence: 0.84
- **[P2] Respect off mode before inbound thread fallback:** `extensions/googlechat/src/monitor-reply-delivery.ts:23`
  - body: `resolveOutboundThread` falls back to `inboundThreadId` whenever the filtered payload has no thread-shaped `replyToId`. When `replyToMode: "off"` stripped the implicit target, this restores threading anyway, so off-mode Google Chat replies still land inside the inbound thread.
  - confidence: 0.88
- **[P2] Do not restore routed threads after off-mode filtering:** `extensions/googlechat/src/channel.adapters.ts:145`
  - body: `resolveReplyTransport` promotes a thread-shaped `threadId` to `replyToId` without considering the resolved account reply mode. Routed follow-ups that have already had `replyToId` suppressed for off mode can become threaded again.
  - confidence: 0.8
- **[P2] Respect off mode in message-tool defaults:** `extensions/googlechat/src/actions.ts:136`
  - body: The action handler defaults omitted `threadId`/`replyTo` to `toolContext.currentThreadTs`, but it never checks Google Chat `replyToMode`. A same-thread message-tool send with no explicit target will thread in off-mode accounts, bypassing the generic message-tool reply-mode guard.
  - confidence: 0.78
