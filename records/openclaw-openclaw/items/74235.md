## Summary

Keep this PR open. Current main still has the Google Chat thread-preservation gap the branch targets, but the proposed fallback paths reintroduce inbound thread targets after shared `replyToMode: "off"` filtering, so the patch needs focused revisions before merge.

## What This Changes

The branch carries Google Chat inbound thread resource names through reply delivery and message-tool paths, normalizes outbound thread targets, validates Chat thread resource names, and adds core plus Google Chat regression tests.

## Best Possible Solution

Revise this PR so Google Chat preserves valid inbound thread resources for normal threaded replies while still honoring `replyToMode: "off"` for implicit fallbacks in direct delivery, routed/follow-up delivery, and message-tool defaults. Add focused off-mode regression coverage for those paths, then land this branch or a maintainer-approved variant and reconcile #39554, #69422, and #67055.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor off-mode before falling back to inbound threads:** `extensions/googlechat/src/monitor-reply-delivery.ts:23`
  - body: `replyToMode: "off"` strips `payload.replyToId` in shared reply filtering, but this fallback turns the missing value back into `params.inboundThreadId`. That makes normal Google Chat text/media replies land in-thread even when the account config asks for unthreaded replies; only use the inbound thread when threading is allowed or when replacing an explicit message-resource reply target.
  - confidence: 0.88
- **[P2] Preserve off-mode in routed Google Chat replies:** `extensions/googlechat/src/channel.adapters.ts:143-144`
  - body: `resolveReplyTransport` promotes any valid Chat `threadId` to `replyToId` without checking the account reply mode. Routed and follow-up replies can pass the session `MessageThreadId` separately from the filtered payload, so a reply whose `replyToId` was removed for off-mode will still be sent threaded.
  - confidence: 0.84
- **[P2] Gate message-tool thread defaults on reply mode:** `extensions/googlechat/src/actions.ts:131-136`
  - body: The message action now defaults omitted `threadId`/`replyTo` to `toolContext.currentThreadTs` even when the current account resolved `replyToMode: "off"`. That makes `message` tool sends from a Google Chat turn thread implicitly despite the same off-mode setting that suppresses natural replies; follow the existing Slack-style gating before using the current thread as an implicit default.
  - confidence: 0.78
