## Summary

Keep this PR open, but do not merge it as written. The PR addresses a real current-main Google Chat threading gap and the latest head appears to resolve the earlier off-mode fallback concerns, but the current diff introduces a media-delivery regression where valid explicit Google Chat thread targets are dropped for outbound media sends.

## What This Changes

The branch preserves Google Chat thread resource names through webhook context, reply delivery, routed outbound sends, and message-tool sends, adds thread-resource validation, and expands core plus Google Chat regression tests.

## Best Possible Solution

Revise this PR so Google Chat resource-name normalization stays inside the plugin boundary while valid explicit thread resources are preserved consistently for text and media paths. After that, land with the existing regression coverage plus a media-thread test, then reconcile #39554, #69422, and #67055 against the shipped behavior.

## Reproduction Assessment

Yes. The underlying bug is source-reproducible from current main's missing MessageThreadId and raw replyToId delivery, and the PR finding is reproducible by calling googlechatOutboundAdapter.attachedResults.sendMedia with threadId set to a valid spaces/.../threads/... resource and no replyToId.

## Solution Assessment

No, not as currently patched. The overall direction is maintainable, but the media path should preserve explicit valid threadId-only delivery the same way the text path does, or otherwise distinguish explicit thread targets from suppressed implicit reply-mode fallbacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve explicit media thread targets:** `extensions/googlechat/src/channel.adapters.ts:362`
  - body: `sendMedia` now calls `resolveGoogleChatOutboundThread` without `allowThreadIdOnly`, unlike `sendText`. A generic outbound media delivery with `threadId: "spaces/.../threads/..."` and no `replyToId` will upload successfully but send with `thread: undefined`, so the media lands top-level instead of in the requested Google Chat thread. Preserve valid explicit thread resources here and add a regression test for media sends.
