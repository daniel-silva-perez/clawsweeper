## Summary

Keep open. The PR targets a real Slack stale-thread-session bug that still exists on current main, but the proposed freshness gate can diverge from the session reset decision because it passes only `updatedAt` into `evaluateSessionFreshness`; current session reuse also considers `sessionStartedAt` and `lastInteractionAt`.

## What This Changes

The PR widens Slack thread-history loading for stale reset-policy sessions by passing config into thread-context resolution, re-exporting existing session reset helpers, and adding targeted Slack regression tests.

## Best Possible Solution

Land a narrow Slack-owned fix that preserves the fresh-session history skip but evaluates stale sessions with the same session entry lifecycle data used by `initSessionState`; track the active-session missing-assistant-turn report separately if it reproduces.

## Reproduction Assessment

Yes. For the stale-session bug, current main can be reproduced with a Slack thread session-store entry that has a prior timestamp and a daily/idle reset boundary, causing `resolveSlackThreadContextData` to skip `conversations.replies`; the proposed tests cover an updatedAt-stale path but not the lifecycle timestamp divergence.

## Solution Assessment

No, not yet. The direction is the narrow maintainable fix, but the implementation should read the full session entry or share the same freshness helper inputs as session initialization instead of checking freshness from `updatedAt` alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use lifecycle timestamps for the freshness check:** `extensions/slack/src/monitor/message-handler/prepare-thread-context.ts:170-181`
  - body: `evaluateSessionFreshness` is not an `updatedAt`-only contract: daily resets use `sessionStartedAt`, idle resets use `lastInteractionAt`, and `initSessionState` passes those lifecycle values when deciding whether to reset the transcript. This new Slack gate passes only `updatedAt`, so a thread session touched by bookkeeping after the reset boundary can still skip Slack history while the real session initialization resets to an empty transcript. Please load the session entry/lifecycle timestamps or share the same freshness inputs before skipping thread history.
  - confidence: 0.84
