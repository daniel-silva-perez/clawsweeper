## Summary

Keep open. Current main still has the Slack thread-history skip that this PR targets, and the PR is a focused active implementation path, but its freshness check does not yet fully match the session lifecycle contract used by core session reuse/reset decisions.

## What This Changes

The PR re-exports existing session reset helpers to the Slack monitor, passes Slack config into thread-context resolution, and adds tests while widening Slack thread-history loading for stale reset-policy sessions.

## Best Possible Solution

Land a narrow Slack-owned fix that preserves the fresh-session history skip but evaluates staleness with the same session entry lifecycle data as `initSessionState`; track the within-minutes assistant-turn omission separately in prompt assembly if it reproduces.

## Reproduction Assessment

Yes for the stale-session bug: current main can be reproduced by a Slack thread session store entry with a prior timestamp and a daily/idle reset boundary, where `resolveSlackThreadContextData` skips `conversations.replies`. The PR tests cover that updatedAt-stale path, but not the lifecycle timestamp divergence.

## Solution Assessment

No, not yet. The direction is the narrowest maintainable fix, but the implementation should read the full session entry or share the same freshness helper inputs used by session initialization instead of passing only `updatedAt`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use lifecycle timestamps for the freshness check:** `extensions/slack/src/monitor/message-handler/prepare-thread-context.ts:170-181`
  - body: `evaluateSessionFreshness` is not an updatedAt-only contract: core session initialization passes `sessionStartedAt` for daily resets and `lastInteractionAt` for idle resets, and existing tests assert those fields can make a session stale even when `updatedAt` is recent. This new gate only reads `updatedAt`, so a thread session touched after the reset boundary can still skip history while `initSessionState` resets to an empty transcript. Please load the session entry/lifecycle timestamps or share the same freshness helper before deciding to skip Slack thread history.
  - confidence: 0.84
