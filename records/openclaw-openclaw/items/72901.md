## Summary

Keep open. The PR addresses a valid Slack stale-session bug and the latest force-push fixes the earlier updatedAt-only problem for stored lifecycle fields, but it still does not match the lifecycle fallback used by current session initialization for legacy entries with only a JSONL session header. That leaves a concrete stale-session path where the agent resets to an empty transcript while Slack thread history is still skipped.

## What This Changes

The PR re-exports existing session reset/store helpers to Slack, loads the Slack thread session entry before resolving thread context, passes Slack config into that resolver, and adds regression tests for stale daily/idle thread sessions.

## Best Possible Solution

Land a narrow Slack-owned fix after the thread-history gate reuses or exposes the same lifecycle timestamp resolution as `initSessionState`, including the JSONL header fallback, with one regression test for a legacy `sessionFile` entry. Track the within-minutes assistant-turn omission from the PR discussion separately in prompt assembly if it reproduces.

## Reproduction Assessment

Yes. The original stale-session bug is reproducible by a Slack thread session with a stored timestamp after a daily or idle reset boundary; the remaining PR gap is reproducible with a legacy entry whose `updatedAt` is recent, `sessionStartedAt` is missing, and the session file header predates the reset boundary.

## Solution Assessment

No, not yet. The PR direction is the narrowest maintainable fix, but the implementation should use the lifecycle resolver contract already used by session initialization instead of passing only fields directly stored on the entry.
