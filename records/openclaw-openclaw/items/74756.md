## Summary

Keep this PR open. The force-push fixed the earlier `initializedPid` compile blocker, and the requested observability gap is real on current main, but the patch still does not solve real ACPX-backed PID propagation because it reads fields that the pinned ACPX runtime does not expose as typed handle/status data.

## What This Changes

The PR adds optional `pid` fields to ACP runtime/session metadata, propagates them through the ACPX wrapper, ACP control-plane metadata, and `sessions_spawn` accepted results, and adds PID-present/PID-absent unit coverage.

## Best Possible Solution

Define a documented additive ACP process-metadata contract in ACPX first, expose the adapter and/or child-agent PID through typed runtime handle/status data with staleness semantics, then propagate that contract through OpenClaw spawn results, session status, persisted metadata, docs, and integration-style coverage against the real ACPX runtime boundary.

## Reproduction Assessment

Yes. Current main reproduces the original issue by source inspection: `sessions_spawn(runtime: "acp")` accepted results and ACP session/runtime metadata have no PID field, and the troubleshooting docs still rely on manual `ps` correlation.

## Solution Assessment

No. The direction is useful, but this PR is not the best fix because it reads PID from top-level or `details.pid` fields that `acpx@0.6.1` does not provide; the safer path is to add a typed upstream ACPX process metadata surface before OpenClaw persists and returns it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Use a real ACPX PID source:** `extensions/acpx/src/runtime.ts:517`
  - body: The wrapper now forwards `pid` only when the delegate returns a top-level or `details.pid` value, but the pinned `acpx@0.6.1` runtime does not expose either field: `ensureSession` builds a handle without `record.pid`, and `getStatus` only puts `pid=...` in the summary string. Real ACPX-backed spawns will still omit `pid`; add/support a typed ACPX PID field before propagating it here.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The patch does not add dependencies or CI execution paths, but it intentionally exposes host OS process IDs through agent-visible and persisted ACP metadata.

Concerns:

- **[medium] Confirm host PID exposure policy:** `src/agents/acp-spawn.ts:1465`
  - body: Returning and persisting host PIDs lets agents correlate OpenClaw sessions with local OS processes and may later combine with process-list or kill controls. That may be acceptable for ACP debugging, but it should land with a scoped contract for which PID is exposed, who can see it, and how stale or reused identifiers are handled.
  - confidence: 0.74
