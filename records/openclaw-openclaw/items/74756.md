## Summary

Keep open. The underlying bug is valid on current main, but this PR is not merge-ready because it propagates PID fields from locations the pinned ACPX runtime does not expose, so real ACPX-backed `sessions_spawn` calls would still omit the PID despite the new mocked coverage.

## What This Changes

The PR adds optional `pid` fields to ACP runtime/session metadata and `sessions_spawn` accepted results, wraps ACPX delegate results to copy PID values, and adds PID-present/PID-absent unit tests.

## Best Possible Solution

Define an additive typed ACPX process-metadata surface first, make clear whether it reports the adapter PID, child agent PID, or both, then propagate that through OpenClaw spawn results, session status, persisted metadata, docs, and integration-style coverage with unavailable/stale semantics.

## Reproduction Assessment

Yes. Current main reproduces the original issue by source inspection: `sessions_spawn(runtime: "acp")` and ACP session/runtime metadata have no PID field, and ACP troubleshooting still requires manual `ps` correlation.

## Solution Assessment

No. The direction is useful, but this PR reads PID from fields that `acpx@0.6.1` does not provide; the narrower maintainable path is to add or consume a typed ACPX PID source before persisting and returning it from OpenClaw.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Use an actual ACPX PID source:** `extensions/acpx/src/runtime.ts:517`
  - body: The wrapper now forwards `pid` only when the delegate returns a top-level or `details.pid` value, but the pinned `acpx@0.6.1` runtime does not expose either field: `ensureSession` builds a handle without `pid`, and `getStatus` only includes `pid=...` in the human summary while omitting it from `details`. Real ACPX-backed spawns will still return no PID, so add or consume a typed ACPX PID field before propagating it here.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The patch has no new dependencies or CI execution paths, but it intentionally exposes and persists host process identifiers without a scoped policy.

Concerns:

- **[medium] Confirm host PID exposure policy:** `src/agents/acp-spawn.ts:1465`
  - body: Returning and persisting host PIDs lets agent-visible tools correlate OpenClaw sessions with local OS processes and may later combine with process-list or kill controls. That may be acceptable for ACP debugging, but it should land with a scoped contract for which PID is exposed, who can see it, and how stale or reused identifiers are handled.
  - confidence: 0.74
