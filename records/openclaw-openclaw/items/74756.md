## Summary

Keep this PR open. The linked bug is valid on current main, but this patch is not merge-ready because it introduces a TypeScript scope error, reads PID data from ACPX surfaces that do not actually expose it, and needs an explicit host-process metadata policy before agent-visible results and persisted session metadata include OS PIDs.

## What This Changes

This PR adds optional `pid` fields to ACP runtime/session metadata, propagates them into ACP `sessions_spawn` accepted results, and adds ACPX/runtime unit tests around PID propagation.

## Best Possible Solution

Define an additive ACP process-metadata contract, have ACPX expose real adapter and/or child-agent process identifiers through that contract, then propagate the data through `sessions_spawn`, ACP manager status, the user-facing `session_status` surface, docs, and tests that exercise the real ACPX runtime behavior.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the missing surface: ACP `sessions_spawn` accepted results and persisted ACP metadata have no PID field, and the pinned ACPX public handle/status types do not expose one.

## Solution Assessment

No. This patch is not the best solution yet because it has a compile blocker and depends on mocked PID fields that ACPX does not return; the safer path is to add a real, documented process-metadata contract and source it from ACPX lifecycle state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Hoist initializedPid before returning it:** `src/agents/acp-spawn.ts:1464`
  - body: `initializedPid` is declared with `const` inside the initialization `try` block, but both accepted-result returns are after that block. TypeScript will fail with `Cannot find name 'initializedPid'` until the variable is declared in the outer scope and assigned inside the `try`.
  - confidence: 0.98
- **[P2] Read PID from the actual acpx runtime surface:** `extensions/acpx/src/runtime.ts:517`
  - body: The pinned `acpx@0.6.1` public `ensureSession` handle and `getStatus` result do not expose a typed `pid`, so wrapping those delegate results only passes mocked tests and leaves real `sessions_spawn` calls without the requested PID.
  - confidence: 0.91
