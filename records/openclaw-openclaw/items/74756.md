## Summary

Keep open. The linked bug is still real on current main, but this PR is not merge-ready because the proposed PID propagation depends on delegate fields that the pinned ACPX runtime does not expose, and the host-PID visibility policy needs maintainer approval before it is persisted and returned to agents.

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata, extracts PID values from ACPX delegate handles/status, persists them in ACP session metadata, returns them from `sessions_spawn`, and adds unit coverage for PID-present and PID-absent cases.

## Best Possible Solution

Add a typed ACPX process-metadata contract first, then propagate that additive contract through the OpenClaw plugin SDK runtime types, ACP control-plane metadata, `sessions_spawn` results, `session_status`, docs, and integration-style tests with explicit semantics for adapter PID versus child-agent PID and stale identifiers.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the missing surface: `sessions_spawn(runtime: "acp")`, ACP runtime handle/status types, and persisted ACP metadata have no PID, while the ACP troubleshooting docs still require manual `ps` correlation.

## Solution Assessment

No. The direction is useful, but this PR is not the best fix because it reads PID from top-level or `details.pid` fields that `acpx@0.6.1` does not emit; the safer path is to expose typed process metadata from ACPX first and then persist/return it with documented visibility rules.

## Review Findings

Overall correctness: patch is incorrect
