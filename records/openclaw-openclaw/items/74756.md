## Summary

Keep this PR open. It targets a valid linked ACP observability gap, but the proposed patch is not mergeable as-is: the new `initializedPid` variable is scoped inside the initialization `try` block and used after that block, and the ACPX wrapper reads `pid` from surfaces that the pinned `acpx@0.6.1` runtime does not actually expose. The host-PID exposure also needs maintainer confirmation of the ACP process-metadata security contract before merge.

## What This Changes

This PR adds optional `pid` fields to ACP runtime/session metadata and attempts to propagate ACPX delegate PIDs into ACP `sessions_spawn` results, session status, and related unit tests.

## Best Possible Solution

Define an additive ACP process-metadata contract that states which PID or process identifiers are exposed, update ACPX or its wrapper to source real adapter/child process metadata, then propagate that contract through ACP session metadata, `sessions_spawn`, `session_status`, docs, and tests that exercise the actual ACPX runtime behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Hoist initializedPid before returning it:** `src/agents/acp-spawn.ts:1464`
  - body: `initializedPid` is declared with `const` inside the initialization `try` block, but both accepted-result returns are after that block. TypeScript will fail with `Cannot find name 'initializedPid'` until the variable is declared in the outer scope and assigned inside the `try`.
  - confidence: 0.98
- **[P2] Read PID from the actual acpx runtime surface:** `extensions/acpx/src/runtime.ts:517`
  - body: The pinned `acpx@0.6.1` public `ensureSession` handle and `getStatus` result do not expose a typed `pid`, so wrapping those delegate results only passes mocked tests and leaves real `sessions_spawn` calls without the requested PID.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff intentionally exposes host OS process IDs to agent-visible tool results and persisted ACP session metadata, which needs an explicit maintainer-approved ACP process-metadata policy.

Concerns:

- **[medium] Confirm PID exposure policy:** `src/agents/acp-spawn.ts:1464`
  - body: Returning and persisting host PIDs gives agents process-correlation metadata that can be combined with other tools to inspect or signal host processes. That may be acceptable for ACP debugging, but the PR should land with a scoped contract for which PID is exposed, who can see it, and how stale process identifiers are handled.
