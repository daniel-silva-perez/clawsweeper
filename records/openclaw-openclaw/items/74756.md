## Summary

Keep this PR open. The underlying bug is valid on current main, but the proposed patch is not merge-ready because it propagates `pid` only from delegate fields that the pinned ACPX runtime does not currently expose, so real `sessions_spawn(runtime: "acp")` calls would still omit the PID.

## What This Changes

The PR adds optional `pid` fields to ACP runtime/session metadata, forwards them through the ACPX wrapper, ACP session manager, and `sessions_spawn` results, and adds PID-present/PID-absent unit tests.

## Best Possible Solution

Define an additive ACPX runtime process-metadata contract first, expose the correct live adapter and/or child-agent PID through typed handle/status data with staleness semantics, then propagate it through OpenClaw spawn results, session status, persisted metadata, docs, and integration coverage against the real ACPX boundary.

## Reproduction Assessment

Yes. Current main reproduces the original issue by source inspection: `sessions_spawn(runtime: "acp")` accepted results and ACP session/runtime metadata have no PID field, and ACP troubleshooting still relies on manual `ps` correlation.

## Solution Assessment

No. The direction is useful, but this PR is not the best fix because it reads `pid` from top-level or `details.pid` fields that `acpx@0.6.1` does not provide; the safer path is to add or consume a typed upstream ACPX process metadata surface first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Use a real ACPX PID source:** `extensions/acpx/src/runtime.ts:517`
  - body: The wrapper now forwards `pid` only when the delegate returns a top-level or `details.pid` value, but the pinned `acpx@0.6.1` runtime does not expose either field: `ensureSession` builds a handle without `pid`, and `getStatus` only includes PID in the human summary string. Real ACPX-backed spawns will still omit `pid`, so add/support a typed ACPX PID field before propagating it here.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The patch does not add dependencies or CI execution paths, but it intentionally exposes and persists host OS PIDs without a documented visibility or staleness policy.

Concerns:

- **[medium] Confirm host PID exposure policy:** `src/agents/acp-spawn.ts:1465`
  - body: Returning and persisting host PIDs lets agent-visible tool results correlate OpenClaw sessions with local OS processes. That may be acceptable for ACP debugging, but it should land with a scoped contract for which PID is exposed, who can read it, and how stale or reused identifiers are treated.
  - confidence: 0.74
