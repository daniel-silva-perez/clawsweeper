## Summary

Keep this PR open, but do not merge it as-is. It targets a real ACP observability gap that current main still has, but the patch is incorrect because the PID variable is scoped inside the initialization block while returned later, and the pinned ACPX runtime does not publicly return the PID values this wrapper tries to propagate.

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata, attempts to propagate ACPX delegate PIDs into accepted `sessions_spawn` results, and adds PID-present/PID-absent unit coverage.

## Best Possible Solution

Define a documented additive ACP process-metadata contract, expose real adapter and child-agent identifiers from ACPX or a supported wrapper surface, then propagate those fields through spawn results, status/session metadata, docs, and tests that exercise the actual ACPX runtime behavior.

## Reproduction Assessment

Yes. On current main, a `sessions_spawn({ runtime: "acp" })` accepted result cannot include PID because the spawn result, runtime handle/status, and persisted ACP session metadata types have no PID field; the docs still describe manual `ps` correlation for stalled ACP processes.

## Solution Assessment

No. The general direction is useful, but this patch is not the best fix because it does not compile as written, it does not use a real ACPX PID source, and it skips the policy contract for exposing host process identifiers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Hoist initializedPid before returning it:** `src/agents/acp-spawn.ts:1466`
  - body: `initializedPid` is declared inside the initialization `try` block, but both accepted-result returns are after that block. TypeScript cannot resolve this name at the return sites, so the branch will fail to compile until the variable is declared in the outer scope and assigned during initialization.
  - confidence: 0.98
- **[P2] Use a real ACPX PID source:** `extensions/acpx/src/runtime.ts:517`
  - body: The pinned `acpx@0.6.1` public handle/status contract does not return a top-level or `details.pid` value, so these tests pass only by mocking delegate output real ACPX-backed spawns will not produce. Expose PID through ACPX or another supported runtime surface before propagating it.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or CI execution paths, but it intentionally exposes host OS process IDs through agent-visible and persisted ACP metadata.

Concerns:

- **[medium] Confirm host PID exposure policy:** `src/agents/acp-spawn.ts:1466`
  - body: Returning and persisting host PIDs lets agents correlate OpenClaw sessions with local OS processes and may later combine with process-list or kill controls. That may be acceptable for ACP debugging, but it should land with a scoped contract for which PID is exposed, who can see it, and how stale or reused identifiers are handled.
  - confidence: 0.74
