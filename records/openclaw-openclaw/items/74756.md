## Summary

Keep open. The PR addresses a valid ACP observability gap, but it is not merge-ready because the ACPX wrapper reads PID only from fields that the pinned acpx@0.6.1 runtime does not expose, so real ACPX-backed sessions_spawn calls can still omit pid. Host PID exposure and persistence also need maintainer policy before this becomes a stable agent-visible contract.

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata, wraps the ACPX delegate to extract PID from handle/status objects, propagates PID into ACP session status and sessions_spawn accepted results, and adds PID-present/PID-absent unit coverage.

## Best Possible Solution

Add or consume a typed ACPX process metadata contract first, including adapter-vs-child-agent meaning and staleness semantics, then propagate that contract through OpenClaw runtime handle/status data, persisted ACP metadata, sessions_spawn, session_status/docs, and integration-style coverage at the real ACPX boundary.

## Reproduction Assessment

Yes. Current main reproduces the original issue by source inspection: sessions_spawn ACP accepted results and ACP runtime/session metadata have no PID field, and docs still instruct manual ps correlation. The PR defect is also reproducible by comparing the patched field reads with the pinned acpx@0.6.1 public runtime contract.

## Solution Assessment

No. The intended direction is useful, but this implementation is not the best fix because it propagates a field the real dependency does not provide; the safer path is to expose typed ACPX PID metadata before OpenClaw persists and returns it.

## Review Findings
