## Summary

This PR should stay open and should not merge in its current form. Current main still lacks the PID surface requested by #74684, but the branch has a TypeScript scoping blocker, relies on mocked PID fields that acpx@0.6.1 does not expose in its typed runtime handle/status responses, and needs an explicit maintainer policy for exposing host process identifiers.

## What This Changes

The branch adds optional `pid` fields to ACP runtime/session metadata, propagates those fields through the ACP control-plane manager and `sessions_spawn` results, wraps ACPX ensure/status responses to extract PID values, and adds ACPX/acp-spawn unit coverage.

## Best Possible Solution

Define an additive ACP process-metadata contract that specifies which process identifiers are exposed, how unavailable and stale identifiers are represented, and which callers can see or act on them; then update ACPX or its wrapper to source real adapter/child metadata and propagate it through `sessions_spawn`, ACP manager status, user-facing status surfaces, docs, and integration tests.

## Security Review

Status: needs_attention

Summary: The diff intentionally exposes host OS process IDs through agent-visible ACP results and persisted session metadata, which needs an explicit maintainer-approved policy.
