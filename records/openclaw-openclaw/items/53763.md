## Summary

Keep open. Current main has a bundled browser plugin with managed profiles, headless launch mode, remote CDP, existing-session attach, and optional Docker browser installation, but the normal managed launch path still depends on an explicit/default/system Chromium-family executable and fails when none exists. The remaining request is a first-class bundled or managed browser runtime that works without a host-installed browser, and that needs maintainer product/security/packaging review before an implementation PR would be safe.

## What This Changes

The issue requests a first-class bundled headless Chromium-style browser runtime so agents can access JS-rendered and login-required pages without depending on host Chrome, third-party extraction APIs, or a Chrome relay.

## Best Possible Solution

Define a maintainer-approved browser-runtime packaging path, then wire executable discovery, doctor/fix repair, Docker/package docs, and update policy around it so the managed openclaw browser profile can start in headless/server/container deployments without requiring a preinstalled host browser.
