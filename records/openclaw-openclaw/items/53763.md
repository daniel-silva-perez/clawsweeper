## Summary

Keep this issue open. Current main has a first-class bundled browser plugin, headless local launch support, remote CDP/Browserless paths, Chrome MCP existing-session attach, and optional Docker browser installation, but it still does not bundle a Chromium runtime for normal installs or make local managed browser launch independent of a separately installed/discoverable browser.

## Best Possible Solution

Keep this issue open and narrow it to the remaining product/packaging work: either ship a pinned Chromium runtime by default, publish an official browser-enabled Docker image, or add installer/doctor repair that installs and persists Playwright Chromium and wires that path into executable discovery. Reuse the existing bundled browser plugin, headless mode, extraArgs, remote CDP, node-host proxy, and optional Docker build arg.

## Risks / Open Questions

- Bundling Chromium by default would materially increase package and image size, so maintainers need a deliberate packaging decision rather than an incidental browser-plugin change.
- A bundled browser runtime needs an explicit Chromium security update cadence and clear sandbox/no-sandbox handling for root and container deployments.
- OpenClaw already has several browser paths: managed local CDP, remote CDP, node-host proxy, Chrome MCP existing-session, Browserless/Browserbase-style CDP, and optional Docker install. A fix should reuse the existing browser plugin rather than add a parallel browser stack.
