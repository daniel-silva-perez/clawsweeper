## Summary

Keep open. Current main has a bundled browser plugin with headless local launch, remote CDP/Browserless support, Chrome MCP existing-session attach, and optional Docker browser installation, but it still does not satisfy the core request: normal installs do not bundle or resolve a Chromium runtime independent of a host-installed browser.

## What This Changes

The issue requests a built-in headless Chromium/browser runtime so agents can access JS-rendered or login-required pages without relying on user Chrome, third-party APIs, or browser relay workarounds.

## Best Possible Solution

Narrow the issue to the remaining product/packaging decision: either ship a pinned Chromium runtime by default, publish an official browser-enabled image, or add installer/doctor repair that installs and persists Playwright Chromium and wires that path into existing executable discovery.

## Risks / Open Questions

- Bundling Chromium by default would materially increase package or image size and needs an explicit distribution/update policy.
- A bundled browser runtime needs clear security handling for Chromium sandboxing, root/container no-sandbox behavior, and timely browser security updates.
- OpenClaw already has managed local CDP, remote CDP, node-host proxy, Browserless/Browserbase-style CDP, Chrome MCP existing-session attach, and optional Docker browser install paths; a fix should extend the existing browser plugin rather than create a parallel stack.
