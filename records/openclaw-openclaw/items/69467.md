## Summary

Keep open. Current main still has the unsafe combination described: `sessionTarget: "current"` is persisted as `session:<live session key>`, one-shot `at` jobs default `deleteAfterRun` to true, non-isolated session targets run against the requested persistent session key, and direct-delivery cleanup still calls `sessions.delete` on that same `agentSessionKey` without checking that it is cron-owned. I found no open PR or merged fix specifically covering this live-DM deletion/rebind case.

## Best Possible Solution

Keep this open and land a focused cron fix that makes delete-after-run session cleanup target only cron-owned disposable run sessions, not live or persistent `session:*` conversation keys. The smallest safe direction is to guard `cleanupDirectCronSessionIfNeeded` by `job.sessionTarget === "isolated"` and/or an explicit cron-owned session-key predicate, add a regression for a non-default Signal-style direct DM key created through `sessionTarget: "current"`, and verify that isolated deleteAfterRun direct deliveries from #67807 still clean up exactly once.

## Clownfish Work Prompt

Fix the cron deleteAfterRun live-session deletion bug described in openclaw/openclaw#69467. Current main resolves `sessionTarget: "current"` to `session:<live session key>`, passes explicit `session:*` targets into `runCronIsolatedAgentTurn`, and `dispatchCronDelivery` deletes `params.agentSessionKey` whenever `job.deleteAfterRun` is true. The fix should preserve #67807 cleanup for truly isolated cron-owned sessions, but must not call `sessions.delete` for live/persistent `session:*` bindings such as `agent:<agentId>:signal:default:direct:<recipient>`.

Expected boundary: `deleteAfterRun` may delete the cron job after success and may clean up cron-owned isolated run sessions, but it must not delete or rebind a user conversation session created by `current` or explicit `session:<id>`. Prefer a small helper/predicate if needed so the cleanup decision is explicit and testable. Do not change provider/channel routing policy, cron schedule semantics, or broad retention/reaper behavior.

Add focused regression coverage for a non-default agent live DM/session-bound job with `deleteAfterRun: true`, direct announce delivery, and `delivery.mode: "none"`/silent-message path if feasible. Assert `sessions.delete` is not called for the live key, and keep existing assertions that isolated deleteAfterRun direct deliveries clean up exactly once. Update docs only if the user-facing semantics of `deleteAfterRun` vs persistent session targets need clarification. Changelog: add a single user-facing fix entry with reporter credit if this changes runtime behavior.
