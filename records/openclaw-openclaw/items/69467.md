## Summary

Keep open. Current `main` still has the unsafe combination described: `sessionTarget: "current"` is normalized to a persistent `session:<live key>`, one-shot `at` jobs default to `deleteAfterRun: true`, explicit session-bound cron runs keep the requested persistent session key, and direct-delivery cleanup still deletes `params.agentSessionKey` without proving the key is cron-owned or isolated.

## Best Possible Solution

Land a focused cron fix that limits delete-after-run session deletion to cron-owned disposable run sessions. Preserve explicit `session:*` execution semantics, but make direct-delivery cleanup skip live/current/custom persistent session keys unless the run can prove the key is isolated or cron-owned. Add regression coverage for a non-default Signal-style direct DM key created through `sessionTarget: "current"` or explicit `session:<live-key>`, and keep the existing #67807 isolated cleanup cases green.
