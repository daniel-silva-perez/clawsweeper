## Summary

Keep open: current main still has the reported downgrade path, and the PR's core runtime direction is right, but the branch should not merge as-is because it leaves behind an exported `maxSecurity()` helper that encodes the rejected escalation behavior and it lacks the required user-facing changelog entry.

## What This Changes

The PR changes exec policy resolution to ignore model-supplied `security` arguments in favor of the configured operator policy, preserves explicitly granted elevated-full mode, and adds regression coverage plus a new policy helper test.

## Best Possible Solution

Merge a narrow fix that makes configured exec security authoritative, preserves the explicit elevated-full operator override, removes the unused unsafe helper and its tests, adds focused regression coverage for full/allowlist/deny cases, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main maps configured `full` plus model `security:"allowlist"` through `minSecurity` to `allowlist`, and the gateway allowlist path then emits `exec denied: allowlist miss` when no allowlist match exists.

## Solution Assessment

No, not as currently packaged. Ignoring model-supplied `security` is the safest runtime fix, but the branch should remove the exported `maxSecurity()` helper and add the missing release-note coverage before merge.
