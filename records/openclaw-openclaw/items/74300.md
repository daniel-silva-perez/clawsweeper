## Summary

Keep this open. Current main has adjacent future-config and state repair pieces, but it still lacks the requested migration-specific source/target version preflight, and the machine migration docs still allow installing a different target version before copying state.

## What This Changes

Request a migration preflight and documentation change so machine-to-machine state transfers detect or require OpenClaw source/target version alignment.

## Best Possible Solution

Add a narrow read-only migration-version preflight in doctor and make the machine migration guide require same-version alignment before copying state. Reuse existing version parsing and trusted metadata such as config `meta.lastTouchedVersion`, backup manifest `runtimeVersion`, or a documented source-version stamp, without adding automatic schema rewrites in the first fix.

## Reproduction Assessment

Yes. The repo-level gap is reproducible by inspecting current main: the migration guide lacks a source/target version alignment step, and the doctor command/type surface has no migration-version preflight option or field.

## Solution Assessment

No, current behavior is not the best solution because post-copy doctor repair and action-scoped future-config guards do not protect the cross-environment cutover boundary. The safer solution is an explicit preflight warning/check plus docs, not startup-time migrations.
