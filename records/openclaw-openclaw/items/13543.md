## Summary

Keep open. Current main still has session/agent-level sandbox placement, sandbox-only tool allow/deny filtering, browser target selection, and elevated exec escape behavior, but no schema-backed per-tool execution placement policy that can force web_fetch/browser into a sandbox while keeping other tools host-routed in the same session.

## What This Changes

Add per-tool sandbox execution placement so untrusted tools such as web_fetch and browser can be sandbox-routed while trusted tools such as exec, message, cron, and gateway remain host-routed in the same session.

## Best Possible Solution

Add a documented core tool-placement contract with schema, typed config, runtime resolution, and tests proving selected tools can be sandbox-routed while other trusted tools remain host-routed in the same session. Keep it separate from tool allow/deny policy and from exec elevation.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature/security architecture request. The absence is reproducible by inspecting the current schema, sandbox runtime-status resolver, tool-policy code, and tool construction pipeline, all of which lack a tool-keyed placement decision.

## Solution Assessment

No, the current behavior is not the best way to solve the request. Per-agent sandboxing, subagent delegation, browser target selection, and elevated exec are useful adjacent controls, but they do not provide enforceable per-tool placement for web_fetch/browser in an otherwise host-routed session.
