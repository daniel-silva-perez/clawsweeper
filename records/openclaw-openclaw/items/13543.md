## Summary

Keep open. Current main still implements sandbox placement at the session/agent level and sandbox tool policy as allow/deny filtering; it does not provide the requested schema-backed per-tool execution placement that can sandbox web_fetch/browser while leaving exec/message/cron/gateway host-routed in the same session. The related merged Docker work is useful but explicitly partial/unblocking, and the existing discussion already frames this as security-sensitive maintainer design work rather than implemented behavior.

## What This Changes

The issue requests configurable per-tool sandbox execution placement so selected untrusted tools can always run in a sandbox while trusted tools remain on the host in the same session.

## Best Possible Solution

Design and document a core tool-placement contract that is schema-backed, typed, and evaluated where tool identity is available, with tests proving web_fetch/browser can be sandbox-routed while trusted tools remain host-routed and existing sandbox filtering/elevated exec behavior remains distinct.

## Reproduction Assessment

Not applicable as a runtime bug report. The missing feature is reproducible by inspection: current schema, typed config, runtime status, tool construction, docs, and tests lack a tool-keyed placement decision.

## Solution Assessment

No; current main is not the best solution for the requested security posture. Session-level sandboxing, sandbox tool allow/deny, browser target controls, and elevated exec are adjacent controls, but they do not enforce selective per-tool placement.
