## Summary

Keep open. Current main still resolves sandbox execution at the session/agent level and treats `tools.sandbox.tools` as tool availability filtering, not as a schema-backed placement policy that can force `web_fetch` and `browser` into a sandbox while leaving trusted tools host-routed in the same session. The related Docker deployment work is useful but explicitly partial/unblocking, and the existing discussion already frames this as security-sensitive maintainer design work.

## What This Changes

The issue requests configurable per-tool sandbox execution placement so selected untrusted tools can always run in a sandbox while trusted tools remain on the host in the same session.

## Best Possible Solution

Design and document a core tool-placement contract that is schema-backed, typed, and evaluated where tool identity is available, with tests proving `web_fetch` and `browser` can be sandbox-routed while trusted tools remain host-routed and existing sandbox filtering/elevated exec behavior remains distinct.

## Reproduction Assessment

Not applicable as a runtime bug report. The missing feature is reproducible by inspection: current schema, typed config, runtime status, tool construction, docs, and tests lack a tool-keyed placement decision.

## Solution Assessment

No. Current main's session-level sandboxing, sandbox allow/deny policy, browser target controls, and elevated exec are adjacent controls, but they do not enforce selective per-tool placement.
