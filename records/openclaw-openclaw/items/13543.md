## Summary

Keep open. Current main still provides session/agent-level sandbox placement, sandbox tool allow/deny filtering, browser target controls, and elevated exec, but not the requested schema-backed per-tool execution placement policy. The maintainer discussion also already treats the related Docker deployment work as partial/unblocking rather than a fix for this request.

## What This Changes

Add per-tool sandbox execution placement so selected untrusted tools such as web_fetch and browser can always run in a sandbox while trusted tools remain host-routed in the same session.

## Best Possible Solution

Design and document a core tool-placement contract that is schema-backed, typed, version-conscious, and evaluated at runtime with tool identity available. Add tests proving selected tools can be sandbox-routed while other trusted tools remain host-routed in the same session, with placement kept separate from sandbox tool filtering and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature/security architecture request. The absence is reproducible by inspecting the current schema, typed config, sandbox runtime resolver, tool-policy code, docs, and tool construction tests, all of which lack a tool-keyed placement decision.

## Solution Assessment

No; current main is not the best solution for the requested security posture. Per-agent sandboxing, sandbox tool filtering, browser target controls, and elevated exec are adjacent controls, but none provides enforceable per-tool placement for web_fetch/browser in an otherwise host-routed session.
