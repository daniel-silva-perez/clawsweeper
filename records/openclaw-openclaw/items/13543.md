## Summary

Keep open. Current main still has session/agent-level sandbox placement and sandbox tool allow/deny filtering, but no schema-backed per-tool execution placement policy that can force web_fetch/browser into a sandbox while leaving exec/message/cron/gateway host-routed in the same session. The existing maintainer discussion and merged Docker sandbox PR also treat this as still-open or only partially unblocked work.

## What This Changes

Request a per-tool sandbox placement mode that can isolate selected tools such as web_fetch and browser while keeping trusted tools such as exec, message, cron, and gateway on the host in the same session.

## Best Possible Solution

Design and document a core tool-placement contract that is schema-backed, typed, and evaluated where tool identity is available. The end state should let selected tools run in a sandbox runtime while trusted tools remain host-routed in the same session, with tests proving placement is separate from tool filtering, browser target controls, and elevated exec.

## Reproduction Assessment

Not applicable as a runtime bug report. The missing feature is reproducible by inspecting current schema, typed config, sandbox runtime status, sandbox context resolution, tool construction, docs, and targeted searches, all of which lack a tool-keyed placement decision.

## Solution Assessment

No; current main is not the best solution for the requested security posture. Existing per-session sandboxing, sandbox tool allow/deny filtering, browser controls, and elevated exec are adjacent controls but do not enforce per-tool placement.
