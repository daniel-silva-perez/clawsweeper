## Summary

Keep open. Current main still has session/agent-level sandbox placement plus sandbox-only tool allow/deny filtering, browser target controls, and elevated exec escape behavior, but it does not implement the requested schema-backed per-tool execution placement policy. The issue discussion also treats this as still-open security architecture work rather than something fixed by the Docker sandbox deployment work.

## What This Changes

Request a core configuration and runtime contract that can force selected tools such as `web_fetch` and `browser` into a sandbox while keeping trusted tools such as `exec`, `message`, `cron`, and `gateway` host-routed in the same session.

## Best Possible Solution

Add a documented, schema-backed core tool placement contract that can sandbox selected tools such as `web_fetch` and `browser` while leaving selected trusted tools host-routed in the same session, with regression tests proving placement remains separate from tool filtering, browser target selection, and elevated exec behavior.
