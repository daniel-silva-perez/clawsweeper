## Summary

Keep open. Current main still has session/agent-level sandbox placement plus sandbox tool allow/deny filtering, but it does not implement the requested schema-backed per-tool execution placement that can sandbox web_fetch/browser while keeping exec/message/cron/gateway host-routed in the same session. The existing discussion also treats the related Docker sandbox work as partial/unblocking rather than a fix, so this needs maintainer security/design follow-up.

## What This Changes

Add per-tool sandbox execution placement configuration so selected tools can always run in a sandbox while other tools remain host-routed in the same session.

## Best Possible Solution

Design and document a core tool-placement contract that is schema-backed, typed, and evaluated where tool identity is available, with tests proving web_fetch/browser can be sandbox-routed while trusted tools remain host-routed and existing sandbox filtering/elevated exec semantics remain separate.

## Reproduction Assessment

Not applicable as a runtime bug report. The missing feature is reproducible by inspection: current schema, typed config, sandbox runtime status, tool construction, docs, and tests all lack a tool-keyed placement decision.

## Solution Assessment

No; current main is not the best solution for the requested security posture. Session-level sandboxing, sandbox tool allow/deny, browser target controls, and elevated exec are adjacent controls but do not enforce selective per-tool placement.
