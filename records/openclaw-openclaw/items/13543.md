## Summary

Keep this issue open. Current main still provides session/agent-level sandbox selection plus sandbox-only tool availability filtering, not a per-tool execution placement policy that can force `web_fetch` or `browser` into a sandbox while leaving `exec`, `message`, `cron`, and `gateway` on host in the same run. Related Docker sandbox work such as #29974 improved sandbox availability for Docker deployments, but it only partially unblocked this request and did not implement selective tool-level routing.

## Best Possible Solution

Keep this open as a core sandbox/tool-routing enhancement. The best path is a documented, schema-backed per-tool execution placement policy that can route selected tools such as `web_fetch` and `browser` through a sandbox backend while selected host tools such as `exec`, `message`, `cron`, and `gateway` remain on host in the same session, with regression tests that distinguish placement from `tools.sandbox.tools` filtering and `tools.elevated` exec escape behavior.
