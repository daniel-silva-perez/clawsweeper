## Summary

Keep this issue open. Current main still has adjacent generic agent streams, live tool/session broadcasts, session lifecycle status, task records, and an update_plan tool, but it does not expose the requested durable session-level process/step event contract with named process.* and step.* events, stable replayable step identity, and related tool/message/artifact ids.

## Best Possible Solution

Keep this issue open and implement an additive Gateway protocol surface for durable session process events, likely a typed process or session.process event family plus a per-session/per-run replayable process ledger. It should derive from existing lifecycle, item, plan, tool, message, and artifact signals where possible, provide stable step ids and sequence/timestamp/type fields, document the contract, and add tests for live streaming plus reconnect/reload recovery.

## Risks / Open Questions

- This is a public Gateway protocol addition, so the implementation should be additive, typed, documented, and covered across schema, server broadcasts, and clients.
- Existing agent item/plan/tool/session broadcasts are close enough to be reusable, but a fix must avoid treating live-only UI events as durable replay state.
- Reconnect/reload durability is the core missing behavior; a future implementation should define the persistence and sequence source instead of relying only on in-memory broadcasts.
