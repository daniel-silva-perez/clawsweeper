## Summary

Keep this PR open. Current main still lacks a `memory-lancedb` memory capability/runtime registration, so the doctor/status false-negative remains, but the branch needs maintainer review and rebase against the current provider-backed embedding configuration path before merge.

## What This Changes

The PR adds a `memory-lancedb` memory runtime via `registerMemoryCapability`, exposes search/status/probe/read hooks, and adds regression coverage for doctor/status runtime discovery.

## Best Possible Solution

Land a rebased plugin-owned runtime adapter in `extensions/memory-lancedb` that registers `registerMemoryCapability({ runtime })`, reports status through the current `MemorySearchManager` contract, preserves provider-backed embedding and storage option behavior, and keeps focused regression coverage proving doctor/status can discover the active LanceDB memory plugin.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction path: configure `plugins.slots.memory` to `memory-lancedb`, then doctor/status resolve through `resolveActiveMemoryBackendConfig`/`getMemorySearchManager`, while the plugin never registers a runtime, so those paths report no active runtime or return null.

## Solution Assessment

Partly. A plugin-owned `registerMemoryCapability({ runtime })` adapter is the right boundary, but the current PR needs rebase work so the probe/status manager honors current provider-backed embeddings without requiring `embedding.apiKey`.
