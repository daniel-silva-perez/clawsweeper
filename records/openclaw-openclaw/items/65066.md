## Summary

Keep open. Current main still lacks a `memory-lancedb` memory runtime registration, so the PR targets a real remaining doctor/status false-negative, but the branch is stale against current `memory-lancedb` behavior and introduces a runtime probe that would still fail supported provider-auth configurations without a plugin-local API key. I did not find unexplained security-sensitive changes: the diff is limited to the plugin implementation and colocated tests, with no workflow, dependency, lockfile, install, or release-script changes.

## What This Changes

The PR adds a `MemoryPluginRuntime` adapter, `MemoryDB.get()` helper, and regression coverage in `extensions/memory-lancedb` so doctor/status can discover the LanceDB memory backend through `registerMemoryCapability({ runtime })`.

## Best Possible Solution

Rebase the plugin-owned runtime adapter onto current main, keep the `registerMemoryCapability({ runtime })` approach, and make the adapter follow current `memory-lancedb` config semantics, including provider-backed embeddings without a plugin-local API key. Regression coverage should prove doctor/status sees the active memory plugin for both explicit API-key and provider-auth configurations.

## Review Findings

Overall correctness: patch is incorrect
