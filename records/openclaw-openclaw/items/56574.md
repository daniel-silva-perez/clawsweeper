## Summary

Keep open. Current main still lacks the registry-level tool dedupe and pluginConfig threading that this PR targets, so the plugin fixes remain useful, but the PR branch should not merge as-is because its live-test env changes can leak profile-sourced credentials into normal tests and can stage stale live config paths.

## What This Changes

This PR changes plugin registry/loading/tool resolution to dedupe repeated per-plugin tool registrations, pass per-plugin config into plugin tool factories, relax gateway-bindable registry reuse for default requests, and update live-test HOME/profile staging docs and tests.

## Best Possible Solution

Rebase the plugin fixes onto current main, keep exact per-plugin registration dedupe and pluginConfig threading in the plugin registry/tool contract with focused regression tests, preserve the already-present gateway-bindable compatibility behavior, and fix or split the live-test HOME/profile staging changes before merge.

## Reproduction Assessment

Yes. The current-main gaps are reproducible by source inspection: repeated registerTool calls still append multiple registry.tools entries, and tool factories are called without pluginConfig; the PR bugs are reproducible from the PR head test-env flow around profile loading and live env snapshot timing.

## Solution Assessment

No as currently proposed. Registry-level dedupe and tool-context threading are the right maintainable boundary, but the branch also changes live-test env staging in a way that broadens credential exposure and can stage the wrong config; those changes should be fixed or split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict profile env loading to live runs:** `test/test-env.ts:338`
  - body: installTestEnv calls loadProfileEnv(realHome) before checking whether this is a live run, so ordinary isolated tests can import missing variables from the developer's ~/.profile, including provider keys or config paths. Gate profile loading behind live/real-home mode or an explicit option.
  - confidence: 0.92
- **[P2] Snapshot env after loading the profile:** `test/test-env.ts:336-338`
  - body: liveEnvSnapshot is captured before loadProfileEnv(realHome), but stageLiveTestState later uses that snapshot to resolve OPENCLAW_CONFIG_PATH and OPENCLAW_STATE_DIR. If those are only exported from ~/.profile, live runs stage the default state instead of the operator-selected config/auth state.
  - confidence: 0.9
