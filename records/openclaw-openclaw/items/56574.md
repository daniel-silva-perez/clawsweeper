## Summary

Keep open. Current main still lacks the core plugin fixes this PR targets, so the work is not obsolete, but the PR branch should not merge as-is because its test-env changes load profile variables in all test runs and snapshot live config env before profile loading.

## What This Changes

This PR changes plugin registry/loading/tool resolution to dedupe repeated per-plugin tool registrations, pass per-plugin config into plugin tool factories, relax gateway-bindable registry reuse for default requests, and update live-test HOME/profile staging docs and tests.

## Best Possible Solution

Rebase the plugin fixes onto current main, keep the dedupe and pluginConfig threading in the plugin registry/tool contract with focused regression tests, preserve current cache-key compatibility dimensions, and either fix or split the live-test HOME/profile staging changes before merge.

## Reproduction Assessment

Yes. The remaining current-main gaps are reproducible by source inspection: repeated api.registerTool calls append multiple registry.tools entries, and tool factories are called without pluginConfig; the PR bugs are likewise reproducible from the head source around installTestEnv profile loading and liveEnvSnapshot staging.

## Solution Assessment

No as currently proposed. Registry-level dedupe and tool-context threading are the right boundary, but the PR also changes test credential/config staging in a way that makes default tests non-deterministic and can stage the wrong live config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict profile env loading to live runs:** `test/test-env.ts:338`
  - body: `installTestEnv` now calls `loadProfileEnv(realHome)` for every test run before checking `live`. That imports variables missing from `process.env` out of the developer's `~/.profile`, including provider keys or `OPENCLAW_CONFIG_PATH`, so normal isolated tests can become machine-dependent and touch real credentials/config. Gate profile loading behind live/real-home mode or an explicit option.
  - confidence: 0.92
- **[P2] Snapshot env after loading the profile:** `test/test-env.ts:336-338`
  - body: `liveEnvSnapshot` is captured before `loadProfileEnv(realHome)`, but `stageLiveTestState` later uses that snapshot to resolve `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. If those are only exported from `~/.profile`, live runs stage the default `~/.openclaw` instead of the operator-selected config/auth state. Capture the snapshot after profile loading or pass post-load env values to staging.
  - confidence: 0.9
