## Summary

Keep open. Current main still defaults the Feishu startup bot-identity probe to 30 seconds and awaits that probe before starting the account transport; the related #74187 fix is open and unmerged in the provided GitHub context.

## What This Changes

The issue reports that Windows Gateway startup waits on a 30-second Feishu API timeout before continuing, even though the system network and later Feishu WebSocket behavior work normally.

## Best Possible Solution

Feishu startup should use a short, startup-safe bot-identity and WebSocket token wait, start the transport promptly when identity lookup is slow, recover identity in the background, and cover that behavior with focused Feishu regression tests.
