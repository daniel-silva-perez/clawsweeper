## Summary

This should stay open. Current main still has a Feishu startup preflight that can wait up to 30s before starting account monitoring, and the test suite explicitly covers the “probe timed out; continuing startup” behavior. The provided related items are Windows gateway/WebSocket handshake reports, but this issue is about the bundled Feishu plugin’s startup API probe and has a confirming user comment, so it is not a strong duplicate/implemented close.

## Best Possible Solution

Keep this open and fix it in the bundled Feishu plugin startup path. The best direction is to decouple long Feishu bot-identity/API probes from gateway startup so the WebSocket account monitor can start promptly, while keeping group mention gating fail-closed until bot identity recovery succeeds and preserving sequential/rate-limit protections for background recovery.

## Clownfish Work Prompt

Fix the bundled Feishu plugin startup delay reported in openclaw/openclaw issue 63257: on Windows, Feishu startup API/auth probing can time out after 30s and delay `openclaw gateway start`, even though the WebSocket later works. Keep the fix scoped to `extensions/feishu` unless a clearly generic plugin seam is unavoidable. Expected behavior: the Feishu WebSocket/account monitor should start promptly when the initial bot identity probe is slow or times out; bot identity recovery may continue in the background; `requireMention` group handling must stay fail-closed until bot identity is known; multi-account startup must not burst Feishu API probes. Do not weaken Feishu auth, webhook verification, allowlists, or core gateway startup policy. Add focused tests that simulate a hanging/timeout startup probe and assert account monitoring starts without waiting for the full 30s budget, plus tests preserving sequential/background recovery behavior. Update Feishu docs only if you add or expose a supported timeout/config knob; otherwise no changelog is needed unless maintainers want a user-facing fix entry.
