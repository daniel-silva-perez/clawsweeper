## Summary

Keep open. Current main still has the reported Feishu startup shape: a 30-second startup bot-identity timeout is passed into the Feishu probe and awaited before account transport startup, while the focused PR #74187 remains open rather than merged.

## What This Changes

The issue reports that Windows gateway startup waits on a 30-second Feishu API timeout before continuing, even though system networking is normal and Feishu works after startup.

## Best Possible Solution

Land or revise #74187 so Feishu startup uses bounded startup-safe bot-identity and WebSocket token waits, starts account transport promptly when identity lookup is slow, recovers identity in the background, and adds focused regression coverage for the Windows startup timeout path.
