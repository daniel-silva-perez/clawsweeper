## Summary

Keep open. Current main still lets the Feishu startup bot-identity probe use a 30-second default and awaits that probe before account transport starts, while the focused PR that would bound the startup path is open and unmerged.

## What This Changes

The issue requests that Windows Gateway startup avoid a 30-second Feishu API timeout and connect to Feishu promptly.

## Best Possible Solution

Land or revise #74187 so Feishu startup uses startup-safe bot-identity and WebSocket token waits, starts account transport promptly when identity lookup is slow, recovers identity in the background, and covers the behavior with focused Feishu regression tests.
