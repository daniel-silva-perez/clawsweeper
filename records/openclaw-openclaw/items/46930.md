## Summary

Keep open. Current main still routes the best-effort multi-gateway advisory through the CLI error channel, while the gateway docs describe the condition as advisory output and explicitly allow intentional isolated multi-gateway setups. I did not find an exact open implementation PR or merged fix for this severity downgrade.

## What This Changes

The issue asks for the best-effort multi-gateway detection advisory to be emitted as warning/info-style output instead of error-level output.

## Best Possible Solution

Downgrade only the non-empty `extraServices` advisory branch in `printDaemonStatus` to warning/info-style human output, preserving error output for real service, port, RPC, probe, and config failures. Add a focused unit test that locks the advisory onto the non-error channel and include a user-facing changelog fix crediting the reporter.

## Reproduction Assessment

Yes. A deterministic source-level reproduction is to call `printDaemonStatus` with `json: false` and a non-empty `extraServices` array; current main routes that advisory through `defaultRuntime.error(...)`.

## Solution Assessment

Yes. The narrowest maintainable solution is to change only this advisory output path, likely using `defaultRuntime.log` with `warnText`, without changing gateway detection, JSON status shape, doctor remediation, or unrelated diagnostics.
