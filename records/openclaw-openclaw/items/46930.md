## Summary

Keep this issue open. Current main still sends the `gateway status --deep` extra-gateway advisory through the error/stderr path, while the docs and the separate gateway probe path describe intentional multi-gateway setups as supported-with-isolation and warning-level diagnostics.

## Best Possible Solution

Update `src/cli/daemon-cli/status.print.ts` so the extra gateway detection block and recommendation use warning or informational output, likely `defaultRuntime.log(warnText(...))` or a dedicated warning helper, while preserving `defaultRuntime.error` for real failures such as non-listening ports. Add a focused `status.print.test.ts` case asserting the advisory is not written through `runtime.error`.

## Risks / Open Questions

- Changing the output channel can affect scripts that currently scrape stderr from `openclaw gateway status --deep`; the fix should keep true failure cases on `runtime.error` and only downgrade the advisory block.
