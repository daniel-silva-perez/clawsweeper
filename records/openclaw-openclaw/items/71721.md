## Summary

Keep open. Current main partially handles the reported ciao/Bonjour `uv_interface_addresses` crash, but the gateway LAN and tailnet discovery helpers still call the throwing network-interface reader directly, and an open closing PR already targets the remaining issue.

## What This Changes

The issue asks OpenClaw to keep the Gateway running in Nemoclaw/OpenShell sandbox environments where `os.networkInterfaces()` throws a `uv_interface_addresses` SystemError.

## Best Possible Solution

Review PR #71950 or land an equivalent narrow fix that uses the existing safe network-interface helper for gateway/tailnet discovery paths that should degrade gracefully, with tests changed from thrown failures to `undefined` or empty-address fallbacks.

## Reproduction Assessment

Yes. Current tests provide a high-confidence unit reproduction by mocking `os.networkInterfaces()` to throw `uv_interface_addresses failed`; `pickPrimaryLanIPv4()` and `listTailnetAddresses()` currently propagate that failure.

## Solution Assessment

Yes, the paired PR direction appears to be the narrowest maintainable fix: keep the raw reader available for callers that want propagation, but switch startup-critical gateway/tailnet discovery to the safe fallback helper.
