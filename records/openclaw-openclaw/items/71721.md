## Summary

Keep open. Current main has Bonjour-specific handling for the reported ciao `uv_interface_addresses` stack, but gateway LAN and tailnet helpers still propagate throwing `os.networkInterfaces()` calls, and PR #71950 remains the open closing-syntax implementation candidate for the remaining gateway-critical paths.

## What This Changes

The issue reports that sandboxed Nemoclaw/OpenShell gateway startup can crash when Node's network-interface enumeration throws `uv_interface_addresses returned Unknown system error 1`, disconnecting the TUI.

## Best Possible Solution

Keep the shipped Bonjour hardening, then review and land PR #71950 or an equivalent narrow fix so startup-critical, best-effort LAN and tailnet discovery paths use `safeNetworkInterfaces()` while preserving the raw reader for callers that intentionally need error propagation.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists by mocking `os.networkInterfaces()` to throw and calling `pickPrimaryLanIPv4()` or `listTailnetAddresses()`; current tests still assert those helpers throw on that path.

## Solution Assessment

Yes. The best direction is the narrow linked PR approach plus the existing Bonjour-specific mitigation: harden best-effort gateway/tailnet discovery callers without changing the global `readNetworkInterfaces()` contract for all consumers.
