## Summary

Keep open. Current main still has best-effort gateway LAN and tailnet helper paths that propagate `os.networkInterfaces()` failures, and the current tests still lock in that throwing behavior. The linked open PR #71950 is the active implementation candidate with closing syntax, so this issue should remain open until that PR, or an equivalent fix, lands.

## What This Changes

The issue asks for gateway startup and discovery to tolerate sandbox `os.networkInterfaces()` / `uv_interface_addresses` failures instead of crashing and disconnecting the TUI.

## Best Possible Solution

Land PR #71950 or an equivalent narrow fix so startup-critical gateway LAN and tailnet discovery paths use the existing safe network-interface helper and gracefully degrade to missing-address fallbacks, while preserving `readNetworkInterfaces()` for callers that intentionally need propagation.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists by mocking `os.networkInterfaces()` to throw and then calling `pickPrimaryLanIPv4()` or `listTailnetAddresses()`; current main tests still assert those helpers throw.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse `safeNetworkInterfaces()` at the best-effort gateway and tailnet call sites instead of changing the raw reader contract globally.
