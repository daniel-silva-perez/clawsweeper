## Summary

Keep open. Current main still propagates throwing `os.networkInterfaces()` failures through gateway LAN and tailnet helpers, and an open PR with closing syntax is already the active implementation candidate rather than a reason to close this issue early.

## What This Changes

The bug report asks OpenClaw gateway startup and TUI connectivity to tolerate sandbox `uv_interface_addresses` failures from `os.networkInterfaces()` instead of crashing.

## Best Possible Solution

Review and land PR #71950, or an equivalent narrow fix, so startup-critical gateway LAN and tailnet discovery paths use `safeNetworkInterfaces()` and gracefully fall back to missing addresses while preserving the raw reader for callers that intentionally want error propagation.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists by mocking `os.networkInterfaces()` to throw and then calling `pickPrimaryLanIPv4()` or `listTailnetAddresses()`; current tests still assert those helpers throw.

## Solution Assessment

Yes. The best direction is the narrow PR #71950 approach: switch only best-effort gateway/tailnet discovery callers to the existing safe helper rather than changing the global `readNetworkInterfaces()` contract.
