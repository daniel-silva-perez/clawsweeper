## Summary

Keep this PR open. Current main still does not implement the healthcheck-specific rule proposed here: the bundled healthcheck skill has no instruction telling alerts, reports, logs, or memory summaries to prefer the user's local timezone, while the generic cron prompt still exposes both local time and a UTC suffix that the model can copy. The PR remains a narrow, low-risk skill-instruction change, with the main maintainership question being wording rather than whether the underlying gap still exists.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase the one-line healthcheck rule onto current main, refine it so user-local time is primary while UTC remains allowed as secondary context when useful, and add narrow prompt/composition coverage if maintainers want stronger proof before merging.
