## Summary

Current main still has the reported behavior. The shared exec approval follow-up fallback prepends the exact internal resume-failure banner to direct external delivery after session resume fails, including for successful `Exec finished (... code 0)` results. The report also has a Telegram confirmation, so this is a shared follow-up bug rather than a Discord-only channel issue.

## Best Possible Solution

Keep this issue open. Fix `src/agents/bash-tools.exec-approval-followup.ts` so a session-resume failure that falls back to successful direct delivery sends only the sanitized command result, while logging resume diagnostics internally. Update `src/agents/bash-tools.exec-approval-followup.test.ts` so successful `code 0` fallback delivery expects `all good` without the internal prefix. Maintainers can review #68129 as a candidate, but a small targeted patch would also solve the confirmed bug; later #72143 looks duplicate or adjacent rather than canonical unless maintainers choose to consolidate there.
