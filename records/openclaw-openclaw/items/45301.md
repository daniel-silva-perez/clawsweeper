## Summary

Keep this issue open. Current main still has the reported failure class: Feishu wiki `space_id` fields are schema-declared as strings, but runtime params are cast and forwarded directly to the Lark API, while the GLM/Z.AI path uses the shared OpenAI-compatible tool-call parser that can already have rounded bare 19-digit JSON numbers before the plugin sees them.

## What This Changes

The issue asks OpenClaw to stop Feishu/Lark knowledge-base tool calls from altering user-provided `space_id` values when listing, creating, or moving wiki nodes.

## Best Possible Solution

Add a focused Feishu wiki hardening path that accepts quoted string IDs, rejects numeric `space_id` and `target_space_id` inputs with a clear precision-loss error before any Lark API call, strengthens schema/skill wording so models pass IDs as quoted strings, and covers the regression with targeted Feishu tool tests. A later shared parser extraction can be considered separately if maintainers want cross-provider protection.

Current `extensions/feishu/src/wiki-schema.ts` declares wiki space IDs as strings, but `extensions/feishu/src/wiki.ts` casts runtime params and forwards `p.space_id` / `p.target_space_id` directly to Lark; OpenAI-compatible/Z.AI tool-call parsing can round bare 19-digit JSON numbers before the plugin sees them. Add a plugin-local guard for all Feishu wiki actions that use `space_id` or `target_space_id`: accept non-empty strings; reject numeric values with a clear error explaining that Feishu/Lark space IDs must be passed as quoted strings because numeric JSON may already have lost precision; do not silently stringify unsafe numbers and send a possibly wrong ID.
