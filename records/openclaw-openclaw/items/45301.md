## Summary

Keep this issue open. Current main still has the reported failure class: Feishu wiki IDs are schema-declared as strings, but the plugin forwards runtime values without rejecting numeric IDs after tool-call JSON parsing has already rounded unsafe integer literals.

## What This Changes

Fix Feishu/Lark knowledge-base tool handling so user-provided `space_id` and related wiki space IDs remain exact instead of being rounded or altered.

## Best Possible Solution

Feishu wiki tools should treat wiki space identifiers as opaque strings end to end, reject numeric `space_id` and `target_space_id` values with a clear precision-loss message before any Lark API call, strengthen schema/skill wording so models quote IDs, and cover the behavior with focused Feishu tool tests.
