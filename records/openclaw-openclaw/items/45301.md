## Summary

Keep open. Current main still has the Feishu/Lark wiki ID precision-loss path: the schema declares `space_id` values as strings, but the runtime casts tool params and forwards `space_id` / `target_space_id` to the Lark client without rejecting number-typed values that may already have been rounded. The issue discussion’s unsafe-integer explanation matches a local parse probe, and the latest release tag still shows the same affected wiki runtime/schema/skill behavior.

## What This Changes

The issue reports that Feishu/Lark wiki knowledge-base IDs supplied as `space_id` are rounded or altered before Feishu document or wiki-node operations reach Lark.

## Best Possible Solution

Feishu wiki tools should treat wiki space identifiers as opaque strings end to end: preserve quoted string values, reject number-typed `space_id` and `target_space_id` with a clear precision-loss message, strengthen schema and skill guidance so models quote long IDs, and add focused Feishu plugin regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `JSON.parse` rounds the reported bare numeric IDs, and current main forwards those rounded values from Feishu wiki tool params into Lark request fields without a type guard.

## Solution Assessment

Yes. The narrowest maintainable fix is Feishu-owned validation plus schema/skill wording; a global big-integer parser would have broader blast radius and still cannot recover digits after a provider or client has already rounded a bare number.
