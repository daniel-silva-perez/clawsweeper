## Summary

Keep this issue open. Current main still has the reported Feishu/Lark wiki ID failure class: wiki space IDs are schema-declared as strings, but parsed runtime parameters are cast and forwarded to the Lark SDK without rejecting numeric, already-rounded unsafe integer values.

## What This Changes

The issue asks OpenClaw to stop Feishu/Lark knowledge-base `space_id` values from being changed when calling wiki document or node tools.

## Best Possible Solution

Feishu wiki tools should treat wiki space identifiers as opaque strings end to end: accept quoted string IDs unchanged, reject numeric `space_id` and `target_space_id` values with a clear precision-loss message before any Lark API call, strengthen schema/skill wording so models quote IDs, and cover the behavior with focused Feishu tool tests.
