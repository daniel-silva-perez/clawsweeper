## Summary

Keep open. Current main still has the reported Feishu/Lark wiki ID precision-loss failure path: wiki IDs are schema-declared as strings, but runtime execution accepts already-parsed numeric values and forwards them to the Lark SDK without rejecting unsafe numbers or warning the model strongly enough to keep long IDs quoted.

## What This Changes

The issue requests lossless handling of long Feishu/Lark knowledge-base `space_id` values when calling Feishu wiki/document tools.

## Best Possible Solution

Feishu wiki tools should treat wiki space identifiers as opaque strings end to end: preserve quoted string IDs unchanged, reject number-typed `space_id` and `target_space_id` before any Lark API call with a clear precision-loss message, and strengthen schema/skill wording so models quote long IDs.
