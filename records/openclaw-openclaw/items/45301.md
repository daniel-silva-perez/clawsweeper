## Summary

Keep open. Current main still has the reported Feishu/Lark wiki ID precision-loss path: wiki IDs are schema-declared as strings, but the bundled Feishu wiki tool casts raw runtime params and forwards `space_id` / `target_space_id` without rejecting already-rounded number values. The reported suffix changes match JavaScript unsafe-integer parsing, so this is a narrow bundled-plugin regression rather than a closeable stale report.

## What This Changes

The issue reports that Feishu/Lark knowledge-base `space_id` values are modified when reused in wiki document/node tools, causing Lark to reject the request as an unknown space.

## Best Possible Solution

Feishu wiki tools should treat wiki space identifiers as opaque strings end to end: preserve quoted string IDs, reject number-typed `space_id` and `target_space_id` before any Lark API call with a clear precision-loss message, strengthen schema/skill wording so models quote long IDs, and add focused regression coverage in the Feishu plugin.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: JavaScript parsing of the reported bare numeric IDs rounds them, and current main forwards those rounded values from Feishu wiki tool params into Lark request fields without a type/unsafe-number guard.

## Solution Assessment

Yes. The narrowest maintainable fix is Feishu-owned validation plus schema/skill guidance; a global big-integer JSON parser would have broader blast radius and still cannot recover digits once a provider/client has already rounded the value.
