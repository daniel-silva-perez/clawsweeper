## Summary

Keep this issue open. Current main still has the reported Feishu/Lark wiki ID precision-loss path: long numeric-looking wiki IDs are schema-declared as strings, but the Feishu wiki tool casts raw runtime params and forwards `space_id` / `target_space_id` without rejecting already-rounded number values.

## What This Changes

The issue reports that Feishu/Lark knowledge-base IDs passed to wiki document/node tools are altered before the Lark API call.

## Best Possible Solution

Feishu wiki tools should treat wiki space identifiers as opaque strings end to end: keep quoted string IDs unchanged, reject number-typed `space_id` and `target_space_id` before any Lark API call with a clear precision-loss message, strengthen schema/skill wording so models quote long IDs, and add focused regression coverage in the Feishu plugin.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: the local `JSON.parse` probe reproduces the reported rounding, and current Feishu wiki execution forwards the resulting value without a string/unsafe-number guard.

## Solution Assessment

Yes, the best repair is a narrow Feishu-owned validation and prompt/schema hint update. A global big-integer JSON parser would have much wider blast radius and still cannot recover IDs once provider/client parsing has already rounded them.
