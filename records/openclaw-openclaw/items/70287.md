## Summary

Keep open. Current main still routes `msteams:search` through Graph `$search`, so the PR addresses a real live bug, but the patch still pushes sender filtering into Graph with `$filter`; Microsoft’s current chat-message docs only allow date filters and channel-message docs do not support `$filter`, so `from` searches can still fail on the same app-only list path. The changelog entry also needs the required contributor attribution before landing. ([github.com](https://github.com/openclaw/openclaw/pull/70287))

## What This Changes

The PR updates the MS Teams message-search helper, regression tests, and changelog to replace Graph `$search` with bounded recent-message listing, pagination, HTML-stripped local matching, and removal of the `$search` consistency header.

## Best Possible Solution

Land a corrected version that uses only app-only-supported chat/channel message listing plus pagination, applies both text matching and sender matching locally over the bounded recent window, keeps the regression tests focused on the URL contract and local filters, and credits the reporter/contributor in the changelog.

## Reproduction Assessment

Yes. The current-main reproduction is high-confidence at the URL-construction seam: the helper and tests still build `$search`, and the linked bug report supplies an app-only Graph 400 for that request. The remaining PR defect is also reproducible by passing `from`, which still appends an unsupported sender `$filter` to the list endpoint.

## Solution Assessment

No, not yet. Removing `$search` and paging recent messages is the right direction, but pushing sender filtering through Graph is not the safest maintainable fix for app-only compatibility; sender filtering should happen locally after listing.
