## Summary

Keep open. Current main still uses the runtime workspace resolver as the interactive `agents add` prompt default, and that resolver still nests non-default agents under `agents.defaults.workspace`; the docs still describe peer-level `workspace-<agentId>` paths, and active implementation PRs already target this bug.

## What This Changes

The issue requests changing the interactive `openclaw agents add <id>` workspace prompt default from nested `~/.openclaw/workspace/<id>` to the documented peer-level `~/.openclaw/workspace-<id>` layout.

## Best Possible Solution

Keep this issue open and review the existing implementation PRs, preferably landing one narrow fix that changes the interactive `agents add` default for brand-new agents to a peer-level `workspace-<id>` path while preserving explicit workspaces and existing runtime resolver compatibility. Add focused regression coverage for the wizard prefill/resolver edge cases and a user-facing changelog entry when the fix merges.

## Reproduction Assessment

Yes. The source path is high-confidence: setup persists `agents.defaults.workspace`, the wizard prompts with `resolveAgentWorkspaceDir(cfg, agentId)`, and that resolver joins the fallback workspace with the non-default id, producing `~/.openclaw/workspace/testbug` for the reported setup.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: change the wizard default for brand-new agents, not the whole runtime resolver. The exact PR to land is still a maintainer choice because several open PRs implement similar fixes with different compatibility scopes.
