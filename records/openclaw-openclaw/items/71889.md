## Summary

Keep open. Current main and the latest release still default the interactive `agents add` workspace prompt through the resolver path that nests non-default agents under `agents.defaults.workspace`, while the docs still describe peer-level `workspace-<agentId>` workspaces. Three open PRs already target this issue, so the right cleanup is maintainer review and selection of one fix, not closing the issue early.

## What This Changes

The issue reports that the interactive `agents add <id>` wizard pre-fills new agent workspaces under the main workspace instead of the documented peer-level `workspace-<id>` path.

## Best Possible Solution

Land one narrow fix from the existing PR set that changes the interactive default for newly added agents to a documented peer-level workspace path, preserves explicit and legacy workspace paths, adds focused resolver/wizard regression coverage, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The source path is deterministic: setup persists `agents.defaults.workspace`, the add wizard uses `resolveAgentWorkspaceDir` as the prompt default, and the resolver joins that fallback with the non-default agent id.

## Solution Assessment

Yes, the requested direction is the right end state. The safest implementation should change the new-agent wizard prefill while avoiding an unreviewed runtime migration of existing agents that already have nested or explicit workspace paths.
