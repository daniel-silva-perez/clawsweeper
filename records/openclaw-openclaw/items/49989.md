## Summary

Keep this issue open. Current main has some newer Synology Chat inbound-reply routing that may avoid the exact webhook reply path described, but the requested client-side fix is still absent: `sendMessage` still omits `user_ids` for a channel-prefixed target such as `synology-chat:4`, and the related fix PR was closed unmerged.

## Best Possible Solution

Keep this issue open and land a narrow Synology Chat fix: normalize or strictly parse channel-prefixed numeric IDs at the `sendMessage`/`buildWebhookBody` boundary, add a regression test proving `synology-chat:4` sets `user_ids: [4]`, and preserve the current inbound `chatUserId` reply-delivery path.

## Evidence

- **Client still parses prefixed IDs as NaN:** `buildWebhookBody` calls `parseNumericUserId(userId)`, and `parseNumericUserId` still runs `Number.parseInt(userId, 10)` directly for strings. That does not strip `synology-chat:` before parsing, so `synology-chat:4` still resolves to `undefined` and does not set `payload.user_ids`.
  - file: [extensions/synology-chat/src/client.ts:272](https://github.com/openclaw/openclaw/blob/14f140d6f0d31fed802869622c38c7e8f8e19587/extensions/synology-chat/src/client.ts#L272)

  - sha: [14f140d6f0d3](https://github.com/openclaw/openclaw/commit/14f140d6f0d31fed802869622c38c7e8f8e19587)
- **Outbound adapter passes `to` through unchanged:** The Synology Chat `sendText` adapter passes `to` directly into `sendMessage`. Its separate `messaging.normalizeTarget` helper strips the prefix, but the outbound boundary itself does not normalize before calling the client.
  - file: [extensions/synology-chat/src/channel.ts:338](https://github.com/openclaw/openclaw/blob/14f140d6f0d31fed802869622c38c7e8f8e19587/extensions/synology-chat/src/channel.ts#L338)

## Risks / Open Questions

- The current inbound webhook route may already avoid the exact lost-reply symptom in common cases, so a fix should be scoped to the client/outbound target parsing boundary rather than undoing the newer `chatUserId` reply-delivery flow.
- Docs currently show `openclaw message send --channel synology-chat --target synology-chat:123456`; a regression test should cover that prefixed target shape so the public behavior stays aligned.
