## Summary

Keep open. Current main only confirms the Control UI toolbar New Session path; typed WebChat `/new` and `/reset`, shared channel/native reset command handling, and gateway `agent` reset handling still reset immediately, and the shipped changelog explicitly describes that partial scope.

## What This Changes

The issue requests a confirmation flow for destructive `/new` and `/reset` session wipes, ideally configurable and showing what would be lost.

## Best Possible Solution

Define one shared destructive-session-reset confirmation contract for bare `/new` and `/reset`, with safe default behavior, clear timeout semantics, and an explicit opt-out config only if maintainers want it; route WebChat typed commands, channel/native command handling, and gateway agent reset paths through that contract while preserving reset-tail and `/reset soft` semantics.

## Reproduction Assessment

Yes. Current source and regression tests give a high-confidence static reproduction: typed WebChat `/new` or `/reset` bypasses `confirmReset`, and shared reset-command tests assert immediate bare reset acknowledgements.

## Solution Assessment

Unclear. The current toolbar-only confirmation is not enough for this issue, but a broader fix should be chosen by maintainers because it changes default destructive command behavior across multiple public surfaces.
