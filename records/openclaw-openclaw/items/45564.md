## Summary

Keep open. Current main only confirms the WebChat New Session button path; typed `/new` and `/reset` still bypass that UI guard, the gateway reset handler still acknowledges bare reset commands immediately, docs still describe them as direct session reset commands, and the session config surface has no `confirmWipe`/confirm-steps style option. The related WebChat-only issue #45800 is closed and PR #45863 is closed unmerged, so neither is a canonical open implementation tracker for the broader command-confirmation request.

## Best Possible Solution

Keep this issue open for maintainer product review. If accepted, implement one central confirmation flow for destructive bare `/new` and `/reset` commands, probably in the gateway reset-command path with UI/channel helpers only where needed. Preserve `/reset soft`, ACP-bound reset behavior, and reset-with-tail semantics deliberately. Add tests for first command creating pending confirmation, second command within the timeout executing, timeout/cancel behavior, typed WebChat commands, and at least one channel/native command path. Update slash-command and config docs if a `session.confirmWipe` style setting is added.
