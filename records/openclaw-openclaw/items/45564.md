## Summary

Keep open. Current main only confirms the WebChat New Session button path; typed WebChat `/new` and `/reset`, plus the gateway reset-command handler used by chat/channel surfaces, still execute or acknowledge bare resets without a pending confirmation flow or config knob.

## What This Changes

The issue requests a configurable confirmation step for destructive `/new` and `/reset` session wipes, with discussion converging on a short double-send confirmation and optional loss summary.

## Best Possible Solution

Define one central destructive-reset confirmation contract for bare `/new` and `/reset`, then layer WebChat and native/channel affordances on top of it. Document any config switch and explicitly preserve or exclude reset tails, `/reset soft`, and ACP-bound reset behavior.

## Reproduction Assessment

Yes. The current code and tests provide a high-confidence static reproduction: typed WebChat `/new` or `/reset` calls `handleSendChat(host)` without `confirmReset`, and the gateway tests assert immediate bare reset acknowledgements.

## Solution Assessment

Unclear until maintainers pick the UX contract. A single short double-send confirmation in the central reset-command path looks more maintainable than adding more WebChat-only prompts, but the requested multi-step/config shape needs product judgment.
