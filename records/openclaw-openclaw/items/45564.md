## Summary

Keep open. Current main only confirms the Control UI New Session button path; typed WebChat `/new` and `/reset`, central gateway reset command handling, and agent RPC reset handling still allow immediate destructive resets or immediate reset acknowledgement without the requested confirmation contract.

## What This Changes

The issue requests a confirmation step before `/new` and `/reset` wipe session state, with discussion converging on a short double-send confirmation and optional opt-out configuration.

## Best Possible Solution

Define one central destructive-reset confirmation contract for bare `/new` and `/reset`, likely a short double-send window with a safe default and an explicit opt-out config, then layer Control UI and channel/native affordances on that same contract while preserving documented reset tails and `/reset soft`.

## Reproduction Assessment

Yes. The current code and tests give a high-confidence static reproduction: typed WebChat `/new` or `/reset` bypasses `confirmReset`, and gateway reset-command tests assert immediate bare reset acknowledgement.

## Solution Assessment

Unclear until maintainers choose the UX and compatibility contract. A central double-send confirmation is likely more maintainable than more WebChat-only prompts, but the exact config shape and surface coverage need product review first.
