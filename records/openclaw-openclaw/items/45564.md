## Summary

Keep open. Current main only confirms the Control UI New Session toolbar path; typed WebChat `/new` and `/reset`, the shared reset command handler, and gateway agent reset handling still perform immediate reset behavior. The shipped v2026.4.27 changelog explicitly describes that partial fix as leaving typed commands immediate, so this still needs maintainer UX and compatibility review rather than cleanup closure.

## What This Changes

Add a default-safe, ideally configurable, confirmation step before destructive `/new` and `/reset` session wipes.

## Best Possible Solution

Define one shared destructive-session-reset confirmation contract for bare `/new` and `/reset`, with a safe default, clear timeout semantics, and an explicit opt-out config if desired; route WebChat, channel/native command handling, and gateway reset paths through that contract while preserving `/reset soft` and reset-tail behavior.

## Reproduction Assessment

Yes. The current source and tests give a high-confidence static reproduction: typed WebChat `/new` or `/reset` bypasses `confirmReset`, and the shared reset-command tests assert immediate bare reset acknowledgements.

## Solution Assessment

Unclear. The current toolbar-only confirmation is not enough for the request, but the best broader fix should be chosen by maintainers because it changes default behavior across shared command and gateway surfaces.
