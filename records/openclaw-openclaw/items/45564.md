## Summary

Keep open. Current main only confirms the Control UI button-triggered `/new` path; typed `/new` and `/reset` in WebChat still dispatch without confirmation, the mobile shared chat view model resets immediately, and the backend reset command handler acknowledges/resets directly with no confirmation state. The issue discussion also narrowed the request to a one-step repeat-confirm pattern with useful loss context, which is not implemented on main.

## Best Possible Solution

Keep this issue open and implement a shared confirmation gate for destructive session resets. The likely direction is a default-on one-step repeat-confirm flow for `/new` and `/reset`, applied in the backend command path so text/native channels are covered, with client UI/mobile affordances layered on top and an optional config escape hatch if maintainers want one.
