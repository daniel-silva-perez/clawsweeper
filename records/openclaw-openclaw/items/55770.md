## Summary

Keep this open. Current main still has a hard-coded 120-character exec command display limit and no `verboseLimit` session contract, while this PR is not merge-ready because it leaves stale TUI state paths, misses the web UI refresh after setting the new limit, and needs to preserve current `full` verbose command support when rebased.

## What This Changes

This PR adds a per-session `verboseLimit` setting and `/verbose limit <n>` handling across directive parsing, gateway session patching, TUI, web UI, and exec tool display plumbing.

## Best Possible Solution

Land the feature only after rebasing and making `verboseLimit` a first-class optional session field with consistent set, clear, reload, and render semantics across gateway, web UI, TUI, and verbose tool-event paths, while preserving existing `full` verbose behavior and adding targeted tests plus docs.

## Reproduction Assessment

Yes. The PR bugs have a high-confidence static reproduction path: set a limit, switch or reload to a session without `verboseLimit`, or clear it via `sessions.patch` with `null`, and the TUI paths retain the old limit; in web UI, `/verbose limit <n>` does not refresh the active session snapshot after patching.

## Solution Assessment

No, not yet. The design direction is reasonable, but the current patch is not the narrowest maintainable implementation until it clears optional session state consistently, refreshes or locally mirrors the web session row, preserves `full`, and validates the new contract with focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Refresh the web session after setting the limit:** `ui/src/ui/chat/slash-command-executor.ts:270`
  - body: `executeVerbose` patches `verboseLimit` but returns only content, while `dispatchSlashCommand` reloads session state only for `action: "refresh"`. The active session keeps the previous `verboseLimit`, so web tool cards continue truncating at the old length until a manual refresh or navigation.
  - confidence: 0.91
- **[P2] Clear the TUI limit when the session omits it:** `src/tui/tui-session-actions.ts:180-183`
  - body: This branch only calls `chatLog.setVerboseLimit` when `entry.verboseLimit` is defined. After a clear patch or a refresh for a session with no override, both `state.sessionInfo.verboseLimit` and `chatLog` keep the previous limit, so future exec cards render with stale truncation.
  - confidence: 0.92
- **[P2] Reset the TUI limit when loading another session:** `src/tui/tui-session-actions.ts:310-311`
  - body: Using `record.verboseLimit ?? state.sessionInfo.verboseLimit` carries the prior session's limit into a newly loaded session that has no override. That breaks the per-session contract and makes unset sessions render exec commands with another session's limit.
  - confidence: 0.93
- **[P2] Keep `full` in verbose command choices:** `src/auto-reply/commands-registry.shared.ts:654`
  - body: Current main exposes and tests `/verbose full`, but this change replaces the verbose choices with `on`, `off`, and `limit`. Native command menus would lose the documented `full` option even though the parser still accepts it.
  - confidence: 0.86
