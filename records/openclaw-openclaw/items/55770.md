## Summary

Keep this PR open. Current main still hard-codes exec command display truncation at 120 characters and has no `verboseLimit` session, gateway protocol, agent display, TUI, web UI, or docs plumbing. The PR remains a plausible implementation candidate, but the provided review discussion identifies stale-state and refresh issues that should be resolved before merge.

## Best Possible Solution

Keep this PR open as the focused implementation candidate. The best path is to rebase onto current main, fix the TUI stale-state and web refresh issues already called out in review, add a bounded `verboseLimit` session/protocol contract, thread it through exec tool display across agent, TUI, web UI, and messaging paths, add narrow parser/session/display tests, and document `/verbose limit <n>` if maintainers accept the feature.

## Risks / Open Questions

- The PR as posted appears to leak stale `verboseLimit` across TUI session switch and session-entry clear paths based on existing review comments.
- The web slash-command path in the PR context may not refresh the active session snapshot after setting a limit, so users may not see the new limit until refresh/navigation.
- The proposed integer is minimum-bounded but not maximum-bounded in the compact diff, so maintainers should consider a reasonable cap to avoid noisy or expensive verbose tool cards.
- No current-main tests cover parser handling, `sessions.patch` persistence/clear behavior, display propagation, or UI/TUI refresh semantics for this feature.
