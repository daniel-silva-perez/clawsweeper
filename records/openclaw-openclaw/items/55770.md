## Summary

Keep this PR open. Current main still hard-codes exec command display truncation at 120 characters and has no `verboseLimit` session/protocol/display plumbing, so the PR is not obsolete. It remains a plausible implementation candidate, but the existing review discussion identifies stale TUI state and web refresh issues that should be fixed before merge.

## Best Possible Solution

Keep this PR open as the focused implementation candidate. The best path is to rebase onto current main, fix the TUI stale-state and web refresh issues already called out in review, add a bounded `verboseLimit` session/protocol contract, thread it through exec tool display across agent, TUI, web UI, and messaging paths, add narrow parser/session/display tests, and document `/verbose limit <n>` if maintainers accept the feature.

## Risks / Open Questions

- The PR as posted can leak a stale `verboseLimit` across TUI session switch and session-entry clear paths, based on existing review comments.
- The web slash-command path in the PR context may not refresh the active session snapshot after setting a limit, so users may not see the new limit until refresh/navigation.
- The proposed integer is minimum-bounded but not maximum-bounded, so maintainers should choose a reasonable cap before exposing it across verbose tool-card surfaces.
- The PR body says no new tests were added, but this feature crosses parser, session persistence, gateway protocol, agent display, TUI, web UI, and docs surfaces.
