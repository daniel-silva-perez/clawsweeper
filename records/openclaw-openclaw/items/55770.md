## Summary

Keep this PR open. Current main still has the hard-coded 120-character exec command display truncation and no `verboseLimit` session/protocol/UI plumbing, so the PR’s central change is not implemented on main. The PR is a plausible focused implementation candidate, but the existing review discussion identifies unresolved stale-state and refresh issues that should be fixed before merge.

## Best Possible Solution

Keep this PR open as the focused implementation candidate. The best path is to rebase it onto current main, fix the TUI stale-state and web refresh issues already called out in review, add a bounded `verboseLimit` protocol/session contract, thread it through exec tool display across agent, TUI, web UI, and messaging paths, add narrow parser/session/display tests, and document `/verbose limit <n>` if maintainers accept the feature.

## Risks / Open Questions

- The PR as posted appears to leak stale `verboseLimit` across TUI session switch/clear paths based on existing review comments.
- The web slash-command path in the PR may not refresh the active session snapshot after setting a limit, so users may not see the new limit until refresh/navigation.
- The proposed integer field is minimum-bounded but not maximum-bounded in the PR context; very large values could make verbose tool cards noisy or expensive to render.
- No targeted tests are present on current main for parser handling, `sessions.patch` persistence/clear behavior, display propagation, or UI/TUI refresh semantics.
