## Summary

Keep open. Current main still returns retryable `UNAVAILABLE` immediately for startup-gated `chat.history` and `models.list`, so this PR is not obsolete; the proposed server-side barrier addresses a remaining cached-client race that the shipped TUI retry work does not cover.

## What This Changes

This PR adds a Gateway startup barrier that waits for post-attach sidecars before dispatching startup-gated `chat.history` and `models.list` requests, with a timeout fallback and focused gateway tests.

## Best Possible Solution

Keep a gateway-owned server-side wait or equivalent queue for startup-gated methods if maintainers want cached or older clients to recover without relying on client retries. The finished change should preserve the retryable timeout fallback and keep coverage for dispatch-after-sidecars, timeout fallback, non-gated bypass, and legacy no-barrier contexts.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction path: call `handleGatewayRequest` for `models.list` or `chat.history` while `context.unavailableGatewayMethods` contains the method, and it immediately returns retryable `UNAVAILABLE` without dispatching.

## Solution Assessment

Yes, with maintainer validation. A Gateway-owned barrier is the narrowest maintainable fix for cached clients that cannot retry, while retaining the existing retryable timeout fallback for sidecar failures or legacy contexts.
