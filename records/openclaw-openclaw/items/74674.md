## Summary

Keep open. Current main still joins every matching assistant text block before the embedded Slack/Telegram-style delivery path, while transcript-oriented callers still have explicit joined-text behavior, so this remains a real delivery selector policy decision rather than cleanup that can be closed.

## What This Changes

Request a delivery-layer policy that selects one user-visible assistant text item instead of posting concatenated multiple matching text content items.

## Best Possible Solution

Add an explicit delivery-boundary selector that leaves shared transcript extraction as concatenation by default. Cover same-phase multi-text assistant messages in shared extraction, embedded helper behavior, and `handleMessageEnd`, while preserving TUI, gateway preview/session dump, and chat-history joined transcript behavior unless maintainers deliberately change those surfaces.

## Reproduction Assessment

Yes. A synthetic assistant message with two matching `text` blocks is enough to trace `extractAssistantVisibleText` into `handleMessageEnd`; current source shows the resulting newline-joined string reaches the delivery path without a selector.

## Solution Assessment

Unclear until maintainers choose the selector rule. The narrowest maintainable direction appears to be a delivery-only selector, while changing the shared extractor default would risk transcript, TUI, gateway, and history behavior.
