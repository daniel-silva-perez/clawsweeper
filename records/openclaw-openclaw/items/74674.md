## Summary

Keep open. Current main still routes delivery-visible assistant text through extractors that concatenate all matching text blocks, and I found no delivery-boundary pick-one policy or regression coverage for the reported multi-text case. The split from #69737 is also valid because that related issue tracks raw error delivery, not duplicate visible text selection.

## What This Changes

The issue requests a delivery-boundary policy so assistant messages with multiple visible text content items do not get concatenated into one user-facing Slack or Telegram post.

## Best Possible Solution

Add an explicit delivery-scoped selector, or a shared extractor option whose default remains concat, so Slack/Telegram-style delivery posts one chosen visible assistant text block while TUI rendering, gateway previews, and chat-history transcript views keep their current concatenation unless deliberately changed. Cover the selected behavior with focused extractor and handleMessageEnd regression tests.
