## Summary

Keep open. Current main still concatenates all matching assistant text blocks before delivery, and I found no delivery-scoped selector or regression coverage that would make this obsolete; the split from #69737 remains valid because that related issue is still open and scoped to raw error fallback behavior.

## What This Changes

The issue requests a delivery-layer policy so user-facing channels select one visible assistant text block instead of concatenating multiple text content items.

## Best Possible Solution

Add an explicit delivery-scoped selector, or a shared extractor option whose default remains concatenation, so user-facing channel delivery posts one chosen visible assistant text block while TUI rendering, gateway previews, and chat-history transcript views keep full-text behavior unless deliberately changed. Cover it with focused extractor and `handleMessageEnd` regression tests.
