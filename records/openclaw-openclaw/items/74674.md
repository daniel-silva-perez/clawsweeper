## Summary

Keep open. Current main still concatenates all matching assistant text content items before the channel delivery path, and the existing shared/transcript-oriented surfaces still depend on joined text behavior, so this remains a real delivery-policy issue rather than cleanup that can close.

## What This Changes

The issue requests a delivery-scoped policy for assistant messages with multiple visible text blocks so user-facing channels send one selected reply instead of a concatenated near-duplicate response.

## Best Possible Solution

Add an explicit delivery-boundary selector that leaves shared transcript extraction as concatenation by default, then cover a multi-text assistant message through the shared extractor, embedded visible-text helper, and `handleMessageEnd` delivery behavior. If maintainers want the selector reusable, expose it as an opt-in extractor option rather than changing the shared default.

## Reproduction Assessment

Yes. A synthetic assistant message with two matching visible `text` content blocks currently flows through `extractAssistantVisibleText` into `handleMessageEnd` as a newline-joined string, which is enough to reproduce the reported concatenation path without private Slack data.

## Solution Assessment

Unclear until maintainers choose the selector policy. A delivery-only selector is the narrowest maintainable fix; a shared extractor option is reasonable only if more than delivery needs the same pick-one behavior.
