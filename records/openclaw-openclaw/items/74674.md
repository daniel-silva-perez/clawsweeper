## Summary

Keep open. Current main still joins all matching assistant text blocks before the embedded delivery path, while existing transcript-oriented surfaces still rely on joined text behavior, so this remains a real delivery-policy decision rather than cleanup that can close.

## What This Changes

Request a delivery-layer policy for assistant messages with multiple visible text content items so user-facing channel delivery selects one deterministic reply instead of posting concatenated variants.

## Best Possible Solution

Add an explicit delivery-boundary selector that preserves shared transcript extraction as concatenation by default. Cover same-phase multi-text assistant messages in the shared extractor, embedded helper, and `handleMessageEnd` delivery behavior; expose a reusable opt-in extractor option only if another surface needs the same selector.

## Reproduction Assessment

Yes. A synthetic assistant message with two matching visible `text` content blocks is enough to trace the current newline-joined value into `handleMessageEnd`; private Slack data is not required to reproduce the source-level path.

## Solution Assessment

Unclear until maintainers choose the selector rule. The narrowest maintainable direction appears to be delivery-only selection, while changing the shared extractor default would risk transcript, TUI, gateway, and history behavior.
