## Summary

Current main still does not implement automatic discovery of agent configurations from external workspaces. Agent definitions remain explicit via `agents.list`, the schema rejects an `agents.autoDiscover` object, workspace enumeration is limited to configured/default agent workspaces, and the plugin SDK does not expose an agent-definition contribution API. #27706 and the current `agents.list` hot-reload path are useful related zero-restart context, but they do not supersede the external workspace discovery request.

## Best Possible Solution

Keep this open for maintainer design. The best path is to define either a first-class `agents.autoDiscover` config contract or a documented plugin-backed agent-definition contribution API that runs before normal agent enumeration. It should include trusted roots, accepted filenames, schema validation, deterministic scan order, explicit precedence versus `agents.list`, reload/watch behavior, docs, and regression tests. Treat #27706 and the current `agents.list` hot-reload work as related zero-restart context, not a substitute for external workspace discovery.

## Risks / Open Questions

- Scanning external workspace roots can become a config-injection path unless roots, filenames, schema validation, and trust boundaries are explicit.
- Glob-based discovery needs deterministic ordering and clear precedence against `agents.list` to avoid nondeterministic routing.
- A script or plugin can approximate this by rewriting `agents.list`, but that is an out-of-band workaround and does not provide a supported discovery contract.
