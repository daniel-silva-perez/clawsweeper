## Summary

Keep open. The PR targets real gateway crash reports and is not fully obsolete: current main has related uncaught-exception suppression for selected pre-connect network errors, but it does not add the PR's raw upgrade-socket error listener or the same TLS/socket message coverage. The proposed patch still needs revision because the uncaughtException path suppresses transient-looking nested errors before preserving fatal-error precedence.

## What This Changes

The PR adds strict transient network/TLS classification to the global uncaughtException handlers in `src/index.ts` and `src/cli/run-main.ts`, attaches a raw upgrade-socket error listener in `src/gateway/server-http.ts`, and adds unit coverage for the new classifier.

## Best Possible Solution

Preserve the useful crash hardening, but route it through a narrow uncaught-exception policy that keeps fatal/config-style errors fatal, targets concrete TLS/undici/socket signatures or the raw socket source, and adds regression coverage for both CLI entrypoints plus the upgrade-socket path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve fatal precedence before suppressing uncaught errors:** `src/cli/run-main.ts:164-169`
  - body: This handler now returns for `isStrictTransientNetworkError(error)` before any fatal/config-style classification. The strict classifier walks nested causes, so a fatal uncaught wrapper such as `ERR_WORKER_UNCAUGHT_EXCEPTION` with a transient network cause can be logged as non-fatal and leave the CLI/gateway running after a fatal exception. Please keep fatal checks ahead of transient suppression or make the strict check ignore fatal wrappers.
  - confidence: 0.86
