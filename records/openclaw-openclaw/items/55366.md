## Summary

Keep open. Current main still resets the active session after context/compaction overflow and returns a retry message, but it does not inject the task goal, last user instruction, or progress state into the new session. Existing compaction memory flush, post-compaction context, and startup-context mechanisms help normal compaction or manual reset paths, but they do not satisfy this hard-reset recovery request. The related open #63216 is useful context for repeated reset loops, but it does not fully supersede this state-loss issue.

## Best Possible Solution

Keep this issue open and implement a core hard-overflow recovery path. Before rotating the session, persist a bounded recovery note from the current command, last user instruction, active run metadata, and any available progress summary, then inject it into the next fresh-session prompt with an explicit note that context overflow caused the reset. Add focused tests around the context-overflow terminal path and coordinate with #63216 so retry-loop fixes and recovery UX do not diverge.
