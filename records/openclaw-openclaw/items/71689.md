## Summary

Keep open. Current main still routes task-registry startup restore through the SQLite store and only logs restore exceptions; it does not classify malformed SQLite errors, preserve/quarantine the database and WAL sidecars, recover or rebuild safely, or expose a task-registry restore-failed audit signal. The related #73517/#74406 work is adjacent maintenance hot-loop work, not a malformed `runs.sqlite` restore fix.

## What This Changes

Handle malformed `tasks/runs.sqlite` during task-registry restore by detecting corruption, preserving or recovering the registry, rebuilding when needed, and surfacing an operator/audit signal.

## Best Possible Solution

Implement task-registry-owned malformed SQLite handling in the store/restore/audit path: detect corrupt or not-a-database failures during open, schema setup, statement preparation, or load; close any handle; preserve the database plus WAL sidecars; attempt conservative recovery only when safe; otherwise reopen a clean empty registry after preservation; and surface the condition through task audit/operator diagnostics.

## Reproduction Assessment

Yes. The issue gives a concrete corrupted `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path with `ERR_SQLITE_ERROR` and `PRAGMA integrity_check` failures, and current source shows that path reaches SQLite load and then only logs the exception. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable direction is to fix corruption handling in the task registry SQLite store plus restore/audit surfaces; a generic gateway startup catch-all or the #74406 maintenance-cache path would not solve malformed-image restore failure.
