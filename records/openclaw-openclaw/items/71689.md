## Summary

Keep open. Current main still routes task-registry startup restore through the SQLite store and only logs restore exceptions; it does not detect malformed database errors, quarantine the bad database and WAL sidecars, rebuild/recover the registry, or expose a task-registry restore-failed audit signal. The related hot-loop report is adjacent maintenance work, not a replacement for malformed SQLite restore handling.

## What This Changes

The issue asks OpenClaw to recover or quarantine a corrupted task registry SQLite database so gateway startup can restore durable tasks cleanly.

## Best Possible Solution

Add task-registry-owned SQLite corruption handling in the store/restore/audit path: classify malformed/not-a-database SQLite failures during open, schema setup, statement creation, and row load; close any partially opened handle; preserve runs.sqlite plus WAL sidecars in a timestamped local quarantine; recover when safe or reopen a clean empty registry; and surface a task-registry restore-failed audit/operator diagnostic.

## Reproduction Assessment

Yes. The issue provides a concrete corrupted $OPENCLAW_STATE_DIR/tasks/runs.sqlite startup path with ERR_SQLITE_ERROR and PRAGMA integrity_check failures, and current source still loads that file through the SQLite restore path with only a catch-and-warn. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in src/tasks task-registry store/restore/audit code; the adjacent maintenance hot-loop work should not be treated as the full solution because it does not handle startup restore on a malformed SQLite image.
