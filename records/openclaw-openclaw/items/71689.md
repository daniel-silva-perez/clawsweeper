## Summary

Keep open. Current main still treats malformed task-registry SQLite restore as a warning-only failure: the default store opens and reads `tasks/runs.sqlite` directly, and no corruption-specific snapshot, recovery/rebuild, or operator-facing audit path exists. The related title-overlap items cover different registry/provider/task symptoms and do not supersede this malformed SQLite recovery request.

## What This Changes

The issue requests task-registry recovery when gateway startup encounters a malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` database.

## Best Possible Solution

Implement task-registry-owned SQLite corruption handling: detect malformed/corrupt/not-a-database errors during open/schema/load, close any handle, preserve the database and WAL sidecars, attempt a conservative recovery/rebuild when available, fall back to an explicit empty registry only after preserving evidence, and surface the condition through audit/operator diagnostics with regression tests.

## Reproduction Assessment

Yes. The issue gives a concrete reproduction path with a malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite`, and current source shows the default gateway restore path will hit the SQLite load and catch/log the failure without recovery.

## Solution Assessment

No. The current warning-only restore path is not the best solution; the narrow maintainable fix belongs in the task-registry SQLite store and restore/audit surfaces, not in a generic startup catch-all or unrelated core migration.

## Repair Work Prompt

Fix malformed SQLite recovery for the task registry reported in https://github.com/openclaw/openclaw/issues/71689. Current main only logs `Failed to restore task registry` when `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` is malformed. Keep the fix owned by the task registry: likely `src/tasks/task-registry.store.sqlite.ts`, `src/tasks/task-registry.ts`, `src/tasks/task-registry.audit.ts`, and colocated tests. Detect corruption/malformed/not-a-database SQLite errors during open, PRAGMA/schema setup, and load; close any cached handle; snapshot `runs.sqlite` plus `runs.sqlite-wal` and `runs.sqlite-shm` before any destructive action; attempt a conservative recovery/rebuild when feasible; otherwise replace with a fresh empty registry only after preserving the bad files and emitting an explicit operator/audit diagnostic. Do not add a broad startup migration, do not touch unrelated Task Flow behavior except where shared helpers are truly needed, and do not require an external `sqlite3` binary unless it is optional with a safe fallback. Add regression tests that create or simulate malformed SQLite restore failures and prove the bad DB is preserved and startup can continue with an operator-visible finding. Update docs/changelog if the user-facing recovery behavior changes.

## Risks / Open Questions

- A recovery implementation could silently lose durable task state unless it snapshots `runs.sqlite` plus `-wal` and `-shm` before replacement or rebuild.
- Malformed SQLite can fail during open, PRAGMA/schema migration, SELECT, or later writes, so a fix needs corruption-specific classification across those phases rather than only wrapping one read.
- Depending on the external `sqlite3 .recover` binary may not be portable for packaged installs unless it is optional with a safe documented fallback.
