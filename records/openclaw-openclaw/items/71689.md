## Summary

Keep open. Current main still restores the task registry by loading the SQLite store and only logs restore exceptions; it does not classify malformed SQLite errors, preserve the bad database and WAL sidecars, recover or rebuild safely, or expose a restore-failed audit signal. The related #73517/#74406 work targets maintenance hot-loop cost on stale records, not startup restore failure on a malformed `runs.sqlite`.

## What This Changes

The issue reports that gateway startup cannot restore the durable task registry when `tasks/runs.sqlite` is a malformed SQLite database and asks for corruption-aware recovery or rebuild handling.

## Best Possible Solution

Add task-registry-owned SQLite corruption handling in the store/restore/audit path: detect corrupt or not-a-database SQLite failures during open, schema setup, statement creation, and row load; close any partially opened handle; preserve the database plus WAL sidecars under a timestamped quarantine/snapshot; recover when safe or reopen a clean empty registry; and surface a task-registry restore-failed audit/operator diagnostic.

## Reproduction Assessment

Yes. The issue gives a concrete corrupted `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path with `ERR_SQLITE_ERROR` and `PRAGMA integrity_check` failures, and current source still routes startup restore through the SQLite load path with only a catch-and-warn. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. The best fix is a narrow task-registry store/restore/audit change; the adjacent #73517/#74406 maintenance-cache work reduces stale-record CPU cost but does not address malformed SQLite restore failure.
