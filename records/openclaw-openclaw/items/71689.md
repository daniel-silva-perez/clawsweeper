## Summary

Keep open. Current main still has a warning-only task-registry restore path for SQLite failures, with no corruption classification, sidecar preservation, rebuild/recovery path, or task-registry restore-failed audit signal. The adjacent stale-registry maintenance report and open PR are useful context but do not cover malformed SQLite restore.

## What This Changes

The issue requests durable task-registry recovery when `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` is malformed, including detection, preservation, recovery or rebuild, and an explicit operator signal.

## Best Possible Solution

Implement task-registry-owned SQLite corruption handling in the store and restore/audit surfaces: detect malformed/corrupt/not-a-database failures during open, schema setup, or load; close any handle; preserve the database and WAL sidecars; attempt conservative recovery when available; fall back to a clean rebuilt registry only after preservation; and expose the condition through operator diagnostics and task audit.

## Reproduction Assessment

Yes. The issue provides a concrete malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup reproduction with `ERR_SQLITE_ERROR` and `PRAGMA integrity_check` failures, and current source shows that restore will enter the SQLite load path and only catch/log the exception. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. The requested direction is the right owner-boundary: fix corruption handling in the task registry SQLite store plus restore/audit surfaces. A generic gateway startup catch-all or the #74406 maintenance-cache path would not solve malformed-image restore failure.
