## Summary

Keep open. Current main still has a warning-only task-registry restore path for SQLite failures, and the SQLite store has no corruption-specific quarantine, recovery, rebuild, or audit/operator signal. The prior ClawSweeper comment remains aligned with the code after re-checking current main.

## What This Changes

The issue requests automated recovery for a malformed durable task registry database at `tasks/runs.sqlite`.

## Best Possible Solution

Implement task-registry-owned SQLite corruption handling: detect malformed/corrupt/not-a-database failures during open/schema/load, close any handle, preserve the database and WAL sidecars, attempt a conservative recovery or rebuild when available, fall back to an explicit empty registry only after preserving evidence, and surface the condition through task audit/operator diagnostics with regression tests.

## Repair Work Prompt

Fix malformed SQLite recovery for the durable task registry reported at https://github.com/openclaw/openclaw/issues/71689. Current main restores through `src/tasks/task-registry.ts` -> `src/tasks/task-registry.store.ts` -> `src/tasks/task-registry.store.sqlite.ts`, and malformed `tasks/runs.sqlite` currently only logs `Failed to restore task registry`. Add task-registry-owned handling for SQLite corruption/not-a-database/malformed-image failures during open, schema setup, and snapshot SELECT. Surface an operator/audit diagnostic, likely by extending task-registry audit codes similarly to the task-flow `restore_failed` pattern. Add regression tests that simulate malformed DB/open/select failure and verify preservation, recovery/fallback behavior, logging/audit, and normal restore behavior. Do not add a generic startup migration, do not touch unrelated task-flow behavior except for shared helper reuse if it is clearly justified, do not download or execute new third-party recovery tools, and update docs/changelog only if the final behavior is user-visible.
