## Summary

Keep open. Current main still only logs task-registry restore failures and does not classify malformed SQLite errors, preserve/quarantine the database and WAL sidecars, rebuild/recover the registry, or expose a task-registry restore-failed operator/audit signal; the linked #73517/#74406 work is adjacent maintenance CPU work, not a malformed `runs.sqlite` restore fix.

## What This Changes

The issue requests durable task-registry recovery when `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` is malformed, including detection, preservation, recovery or rebuild, and an explicit operator signal.

## Best Possible Solution

Implement task-registry-owned malformed SQLite handling in the store/restore/audit path: detect corrupt/not-a-database failures during open, schema setup, statement preparation, or load; close any handle; quarantine the database plus WAL sidecars; attempt conservative recovery only when safe; otherwise reopen a clean empty registry after preservation; and surface the condition through task audit/operator diagnostics.

## Reproduction Assessment

Yes. The issue gives a concrete corrupted `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path with `ERR_SQLITE_ERROR` and `PRAGMA integrity_check` failures, and current source shows that path reaches SQLite load and then only logs the exception. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. The requested owner-boundary is the narrow maintainable direction: fix corruption handling in the task registry SQLite store plus restore/audit surfaces. A generic gateway startup catch-all or the #74406 maintenance-cache path would not solve malformed-image restore failure.
