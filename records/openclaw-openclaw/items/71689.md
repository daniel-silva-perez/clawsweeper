## Summary

Keep open. Current main still restores the task registry by loading the SQLite-backed snapshot and only logs `Failed to restore task registry` on any restore error; the SQLite store has no corruption-specific quarantine, recovery, rebuild, or operator/audit signal. The existing ClawSweeper comment remains aligned with current main.

## What This Changes

The issue asks for task registry restore to recover gracefully when `tasks/runs.sqlite` is a malformed SQLite database image.

## Best Possible Solution

Implement task-registry-owned SQLite corruption handling: detect malformed/corrupt/not-a-database failures during open, schema setup, and load; close any handle; preserve the database plus WAL sidecars; attempt conservative recovery or rebuild when available; fall back to an explicit empty registry only after preserving evidence; and surface the condition through task audit/operator diagnostics with regression tests.

## Reproduction Assessment

Yes. The issue gives a concrete malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path, and current source shows restore will hit the SQLite open/load path and then only catch/log the failure. I did not create a corrupt database during this read-only review, but the source path is high-confidence.

## Solution Assessment

No. The current warning-only restore path is not the best solution; the narrow maintainable fix belongs in the task-registry SQLite store plus restore/audit surfaces, not in a generic startup catch-all or unrelated migration.
