## Summary

Keep open. Current main still has a warning-only task-registry restore path for SQLite failures, with no corruption classification, snapshot/quarantine/rebuild path, or task-registry restore-failed audit signal; the related maintenance hot-loop issue and PR are adjacent but do not supersede this corruption-recovery bug.

## What This Changes

The issue reports that restoring the durable task registry fails when `tasks/runs.sqlite` is malformed and asks for corruption-aware recovery or quarantine handling.

## Best Possible Solution

Implement task-registry-owned SQLite corruption handling: detect malformed/corrupt/not-a-database failures during open, schema setup, or load; close any handle; preserve the database and WAL sidecars; attempt conservative recovery when available; fall back to an explicit rebuilt empty registry only after preserving evidence; and surface the condition through task audit/operator diagnostics with regression tests.

## Reproduction Assessment

Yes. The issue gives a concrete malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` reproduction, and current source shows gateway restore will enter the SQLite open/load path and then only catch/log the failure. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes, the requested direction is the right owner-boundary: fix corruption handling in the task registry SQLite store plus restore/audit surfaces. A generic gateway startup catch-all or the #74406 maintenance-cache path would not solve the malformed-image restore failure.
