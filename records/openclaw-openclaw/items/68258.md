## Summary

Keep open. Current main still lacks a top-level auth/auth.profiles reload rule, so auth.profiles.* remains an unmatched config path that requests a gateway restart; the issue is also paired with open PR #68257, which is the right maintainer review path before closing.

## What This Changes

The report asks the gateway reload planner to treat OAuth auth profile refreshes under auth.profiles.* as dynamic/no-op changes while preserving restart behavior for gateway.auth.*.

## Best Possible Solution

Keep the issue open until PR #68257 or an equivalent patch lands, with an explicit dynamic-read/no-op classification for auth.profiles.* and regression tests proving auth profile refreshes do not restart the gateway while gateway.auth.token and gateway.auth.mode still do.

## Reproduction Assessment

Yes. A high-confidence code reproduction is to pass auth.profiles.openai-codex:user@example.com into buildGatewayReloadPlan on current main; there is no matching auth rule, so the unmatched-path branch marks restartGateway and records the path as a restart reason.

## Solution Assessment

Yes, if scoped to the reported auth.profiles.* path. The paired PR direction is the narrow maintainable fix, but maintainers should confirm whether a blanket auth prefix no-op is acceptable or whether auth.profiles/auth.order should be classified more specifically.
