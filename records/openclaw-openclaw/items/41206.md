## Summary

Keep this PR open, but do not merge it as-is. Current main still uses the single `jobs.json` cron store with a `jobs-state.json` sidecar, so the requested per-job directory store is not already implemented; however, the PR only adds helper functions/tests, does not wire the directory store into Gateway/CLI persistence, and has concrete filename, deletion, state-splitting, and path-safety defects already surfaced in review discussion.

## What This Changes

This PR adds optional directory-based cron job store helpers and tests for storing one cron job per JSON file under `cron/jobs/` while keeping `jobs.json` available.

## Best Possible Solution

Rework the feature as a first-class cron storage backend on current main: keep `jobs.json` compatibility, define activation and precedence, key files by stable collision-safe job IDs, preserve `jobs-state.json` or an equivalent runtime sidecar, wire Gateway/CLI/doctor/docs behavior, and cover migration, overlap, rename, delete, malformed filenames, and sidecar semantics with focused tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: a job with id `brew-1` and name `Daily Brew` is saved as `daily-brew.json`, while removal compares `jobIdFromFilename("daily-brew.json")` to `brew-1` and leaves the file behind; current main also has no call path that would load `cron/jobs/`.

## Solution Assessment

No. The directory-store direction may be useful, but this patch is not the best implementation because it adds unreachable helpers, keys files by mutable display names, serializes runtime state into job config files, and omits the current service/config/docs contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` is exported, but the PR never switches Gateway, service add/update/remove, CLI, config, or doctor paths to use it. Creating `cron/jobs/` would not affect runtime behavior, so the advertised opt-in store is unreachable.
  - confidence: 0.97
- **[P1] Use immutable IDs for job filenames:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives filenames from `job.name`, but cron job names are mutable and nonunique while service-created IDs are stable UUIDs. Duplicate names can overwrite each other, and renames can leave multiple files for the same job.
  - confidence: 0.96
- **[P1] Delete files by persisted job IDs:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested ID with the filename-derived ID, but files written by this patch are name-derived. A normal job whose stable ID differs from its display name will not be removed.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, `slugifyCronJobName` returns the raw input. Values containing path separators can then be joined into the output path, so the fallback must be safe and the resolved path must stay under `jobs/`.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main strips mutable runtime fields from `jobs.json` and stores them in `jobs-state.json`, but this writer serializes the entire `CronJob` into each per-job file. If wired in, runtime execution state becomes user-editable config again.
  - confidence: 0.85
