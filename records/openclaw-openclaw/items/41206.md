## Summary

Keep open. This is a real cron directory-store feature tied to the existing cron GitOps requests, but current main does not implement it and this PR is not merge-ready: the diff has blocking filename-keying, delete, service-wiring, and path-safety issues. The existing review discussion already called out the same data-integrity concerns, which still apply to the fetched PR diff.

## What This Changes

The PR adds `src/cron/store.ts` helpers plus `src/cron/store.dir.test.ts` coverage for an optional `cron/jobs/` directory where each JSON file represents one cron job while preserving `jobs.json` support.

## Best Possible Solution

Rework the feature on current main as an additive cron directory-store design: key files by immutable ids, keep `jobs.json` compatibility, preserve `jobs-state.json` runtime state semantics, wire Gateway/service load and add/edit/remove paths, add config/docs/doctor support, and cover migration, path confinement, duplicate ids, rename/delete, permissions, and state-sidecar behavior in focused cron tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds the filename from `job.name ?? job.id`. Current cron jobs have UUID ids and mutable names, so two jobs with the same name can overwrite each other and renaming a job writes a second file with the same persisted id that reloads as a duplicate.
  - confidence: 0.96
- **[P1] Delete files by persisted job ids:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with `jobIdFromFilename(entry.name)`, but files created by this patch are named from the display name. A normal UUID-backed job named `Daily Brew` saves as `daily-brew.json`, so removal by id leaves the file behind.
  - confidence: 0.97
- **[P2] Wire directory store into service persistence:** `src/cron/store.ts:187-236`
  - body: The new directory helpers are not selected by the service load or persist paths. Current startup, add, edit, and remove flows still use `loadCronStore(state.deps.storePath)` and `saveCronStore(...)`, so creating `cron/jobs/` would not make the Gateway use this store.
  - confidence: 0.88
- **[P2] Keep raw fallback strings out of paths:** `src/cron/store.ts:151`
  - body: When slugification removes every character, `slugifyCronJobName` returns the raw input and `saveCronJobFile` joins it into a filesystem path. A punctuation-only name containing path separators can escape `cron/jobs/`; the fallback should be a safe id-derived filename or be rejected.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds local file creation from cron job names and has a concrete path-safety issue in the slug fallback.

Concerns:

- **[medium] Raw slug fallback can escape jobs directory:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` falls back to the raw job name/id when the sanitized slug is empty, and `saveCronJobFile` joins that value into the output path. Names containing only separators such as `../` can write outside the intended `cron/jobs/` directory.
  - confidence: 0.82
