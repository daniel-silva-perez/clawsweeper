## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks directory-backed cron job storage, so the proposed feature is still relevant; however the PR branch is stale/unmergeable against the current cron store and the diff has blocking filename-keying bugs that can orphan, overwrite, or duplicate jobs.

## What This Changes

The PR adds `cron/jobs/` directory-store helpers and tests for loading, saving, removing, and migrating one JSON file per cron job while leaving the existing `jobs.json` store path in place.

## Best Possible Solution

Rework this into an additive directory-store implementation on current main: keep existing `cron.store`/`jobs.json` compatibility, key persisted job files by immutable ids or otherwise match by persisted ids, wire directory loading and add/edit/delete into the cron service and Gateway startup paths, preserve the state sidecar model, and update schema/help, doctor, docs, and focused cron store/Gateway tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Key job files by immutable ids:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds the path from `job.name`, but cron job names are mutable and not guaranteed unique while current jobs get UUID ids. Two jobs with the same name can overwrite each other, and renaming a job writes a new file while the old file remains, producing duplicate entries for the same id on reload.
  - confidence: 0.96
- **[P1] Delete files by the persisted job id:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the id inferred from the filename, but files saved by this patch are named from the job name. For normal UUID-backed jobs where the name slug differs from the id, removal returns `false` and leaves the job file behind to be loaded again.
  - confidence: 0.97
- **[P2] Wire the directory store into cron service operations:** `src/cron/store.ts:187-236`
  - body: The patch adds `loadCronStoreDir`, `saveCronJobFile`, and `removeCronJobFile`, but current cron service load/persist paths still call the aggregate `loadCronStore` and `saveCronStore` functions. As written, Gateway startup and cron add/edit/remove will not actually use the new directory store mode.
  - confidence: 0.88
- **[P2] Keep raw fallback strings out of paths:** `src/cron/store.ts:151`
  - body: When a name contains no alphanumeric characters, `slugifyCronJobName` returns the raw input. A name like `../` can therefore be joined into a path outside `cron/jobs/`; the fallback should be a safe generated filename or sanitized id, not raw user text.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds local file creation from cron job names and has a concrete path-safety issue in the slug fallback.

Concerns:

- **[medium] Raw slug fallback can escape jobs directory:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` falls back to the raw job name/id when the sanitized slug is empty. Because `saveCronJobFile` joins that value into a filesystem path, punctuation-only names containing path separators can write outside `cron/jobs/` instead of staying confined to the job directory.
  - confidence: 0.82
