## Summary

Keep this PR open, but it is not merge-ready. Current main still only documents and wires the cron store through jobs.json plus jobs-state.json, so the requested per-job directory store remains unimplemented; however, this branch currently adds helper-only storage code with blocking correctness issues, a filename path traversal risk, and drift from the split runtime-state contract now on main.

## What This Changes

The PR adds exported cron directory-store helpers and tests for storing one cron job per JSON file under cron/jobs/, including disabled filename prefixes, filename-derived IDs, and a migration helper from jobs.json.

## Best Possible Solution

Rework the feature on current main as a real cron storage backend: keep jobs.json backward compatibility, make directory activation and precedence explicit, use stable sanitized id-keyed filenames or another collision-safe scheme, preserve the jobs-state.json runtime sidecar contract, wire load/add/update/remove/migration/status paths, and add docs plus doctor coverage for overlap and migration behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wire the directory store into cron service persistence:** `src/cron/store.ts:196`
  - body: The new directory-store helpers are exported but never used by the live cron service. ensureLoaded/add/update/remove still load and save the configured jobs.json store, so creating cron/jobs/ would not change gateway behavior.
  - confidence: 0.97
- **[P1] Use immutable IDs for per-job filenames:** `src/cron/store.ts:246-247`
  - body: saveCronJobFile derives the filename from job.name, but names are mutable and nonunique while cron ids are stable. Duplicate names can overwrite each other, and renaming a job writes a second file for the same id.
  - confidence: 0.96
- **[P1] Delete files by the persisted job ID:** `src/cron/store.ts:271-272`
  - body: removeCronJobFile compares the requested id with the filename-derived id, but files written by this patch are name-derived. A normal job whose id differs from its display name will not be deleted.
  - confidence: 0.96
- **[P2] Keep filename fallbacks inside the jobs directory:** `src/cron/store.ts:151`
  - body: slugifyCronJobName returns raw input when no alphanumeric slug remains. A crafted name containing relative path segments can reach path.join for the output file, so the fallback must be guaranteed safe and the resolved path should be checked under jobs/.
  - confidence: 0.9
- **[P2] Preserve the runtime-state sidecar contract:** `src/cron/store.ts:248`
  - body: Current main strips runtime state from jobs.json and stores mutable fields in jobs-state.json, but this patch writes the whole CronJob into each per-job file. If wired in, runtime fields would become user-editable config again.
  - confidence: 0.85
