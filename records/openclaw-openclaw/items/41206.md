## Summary

Keep open. Current main still lacks a directory-backed cron job store, and the open #20950 tracker covers the same remaining user need, so this PR is not obsolete; however, the branch is currently unmerged/unmergeable and the patch should not merge as-is because it adds unused store helpers plus data-integrity and filename-safety regressions.

## What This Changes

This PR adds optional cron directory-store helpers and tests for loading, saving, removing, and migrating one cron job per JSON file under `cron/jobs/` while keeping `jobs.json` available.

## Best Possible Solution

Rework the feature as a first-class cron storage backend on current main: keep `jobs.json` compatibility, define explicit activation and precedence, key per-job files by stable collision-safe IDs, keep runtime state in the `jobs-state.json` sidecar or an equivalent non-config state store, wire Gateway/CLI/doctor/docs paths, and cover migration, overlap, rename, delete, malformed filename, and sidecar behavior with focused tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: a job with id `brew-1` and name `Daily Brew` would be saved as `daily-brew.json`, while removal by `brew-1` compares against the filename-derived `daily-brew` and leaves the file behind.

## Solution Assessment

No. The requested direction is valid, but this PR is not the best implementation because it adds unused helpers, keys files by mutable display names, serializes runtime state into config files, and does not define the live storage contract on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` is exported here, but the cron service, Gateway startup, config schema, CLI, and doctor paths still use the existing single-file `loadCronStore`/`saveCronStore` flow. Creating `cron/jobs/` would not change runtime behavior, so the advertised opt-in store is unreachable.
  - confidence: 0.97
- **[P1] Use immutable IDs for job filenames:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives the file path from `job.name`, but cron job names are mutable and nonunique while `createJob` assigns stable UUID IDs. Duplicate names can overwrite each other, and renaming a job can leave two files for the same persisted job ID.
  - confidence: 0.96
- **[P1] Delete files by persisted job IDs:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested ID with the filename-derived ID, but files written by this patch are name-derived. A normal job whose stable ID differs from its display name will not be removed.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, this helper returns the raw input. Values containing path separators can then reach `path.join(jobsDir, ...)`, so the fallback must be safe and the resolved file path must remain under `jobs/`.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main strips mutable runtime fields from `jobs.json` and stores them in `jobs-state.json`, but this writer serializes the whole `CronJob` into each per-job file. If wired in, runtime execution state becomes user-editable config again.
  - confidence: 0.85
