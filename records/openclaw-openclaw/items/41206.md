## Summary

Keep this PR open. Current main still uses a single `jobs.json` cron store, and this PR remains a plausible implementation candidate for the open per-job cron storage request. It should not be closed as implemented or obsolete, but it should not merge as-is because the posted review comments identify real filename-keying/data-integrity bugs.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but require a revision that uses a stable id-based filename contract or otherwise loads/removes by persisted job id, adds tests where job id and name differ plus rename/remove/collision cases, and updates cron docs/config/doctor behavior before maintainer merge review.
