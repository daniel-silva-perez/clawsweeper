## Summary

Keep open. Current main still enforces the CLI no-output watchdog on stdout/stderr silence, including the claude-cli live-session path, with no source evidence of a Monitor or pending-tool expected-idle exemption; the related cap-raising work may reduce the symptom but does not clearly solve the Monitor-specific failure mode.

## What This Changes

The issue requests that the CLI no-output watchdog stop killing claude-cli sessions that are silent because they are waiting on a long-running Monitor task.

## Best Possible Solution

Keep this open until the CLI runner distinguishes expected idle from a stalled backend during long-running Monitor/tool waits, or maintainers explicitly choose the timeout-cap/config-override path as the intended product behavior. The end state should include focused regression coverage for silent claude-cli waits that are still making progress through a pending tool/background job.

## Reproduction Assessment

Yes. The issue provides a concrete Monitor wait and watchdog log sequence, and current-main code/tests show that a silent CLI process still follows the no-output-timeout failure path.

## Solution Assessment

Unclear. Raising watchdog caps in #40982/#41022 would reduce false kills for 10-15 minute jobs, but the narrower semantic fix is likely a pending-tool expected-idle signal or heartbeat so truly stalled CLI processes can still be terminated.
