## Summary

Keep open. Current main still applies the CLI no-output watchdog to silent claude-cli live turns and generic supervised CLI runs, with no Monitor or pending-tool expected-idle signal in the timeout calculation or timer reset path; the related cap-raising work in #40982/#41022 may mitigate some long waits but does not clearly solve the Monitor-specific failure mode.

## What This Changes

The issue asks for the CLI no-output watchdog to stop killing healthy sessions while the agent is expectedly idle waiting on a long-running Monitor task.

## Best Possible Solution

Keep this open until the CLI runner can distinguish expected idle during long-running Monitor/tool waits from a genuinely stalled backend, or maintainers explicitly choose cap/config tuning as the product contract. The finished path should include focused regression coverage for silent claude-cli waits that are still making progress through a pending tool or background job.

## Reproduction Assessment

Yes. The issue provides a concrete Monitor wait and watchdog log sequence, and current-main source plus existing tests show silent CLI processes still follow the no-output-timeout failure path without an expected-idle exemption.

## Solution Assessment

Unclear. A pending-tool expected-idle signal or bounded heartbeat is the narrow semantic fix, while #40982/#41022-style cap raising is only a mitigation unless maintainers decide longer silence is the intended contract.
