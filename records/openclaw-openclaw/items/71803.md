## Summary

Keep this issue open. Current main still has a CLI no-output watchdog that is armed for claude-cli runs, resets only on subprocess stdout/stderr, and treats 180s of silence as a timeout; I found no Monitor-aware expected-idle exemption or heartbeat path in the current watchdog, live-session, docs, or tests. Related #40982/#41022 cover the broader cap-raising problem, but they do not yet prove the Monitor-specific idle state is handled.

## Best Possible Solution

Keep this issue open and fix the Monitor-specific expected-idle behavior in the CLI backend path. The best direction is either to suspend/touch the claude-cli no-output watchdog while a real pending Monitor task is active, or to add a narrowly scoped heartbeat/liveness signal that is not user-visible. Coordinate with #40982/#41022 so the broader cap-raising work does not close this Monitor-specific bug without regression coverage.
