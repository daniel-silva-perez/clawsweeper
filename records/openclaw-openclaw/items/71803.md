## Summary

Keep open. Current main and the latest tagged release still apply a CLI no-output watchdog to silent CLI turns without any Monitor-aware expected-idle path, and the related open cap-raising PR is only a mitigation because it lengthens the timer rather than recognizing a healthy pending Monitor wait.

## What This Changes

The issue requests making the CLI no-output watchdog tolerate deliberate idle periods while a long-running Monitor task is pending.

## Best Possible Solution

Add an explicit proof-of-life or expected-idle contract for pending Monitor/MCP waits in the CLI runner so healthy long-running background work does not kill the supervising CLI process, while preserving no-output and overall timeouts for genuinely stalled backends.

## Reproduction Assessment

Yes. The issue gives a concrete Monitor wait plus matching watchdog logs, and current main source shows silent CLI processes still hit no-output timeout paths with no Monitor-aware exemption.

## Solution Assessment

Unclear. Exempting verified pending Monitor work is the right product direction, but maintainers should choose whether that state belongs in the Claude stream-json parser, the MCP/tool wrapper, or a bounded heartbeat before implementation.
