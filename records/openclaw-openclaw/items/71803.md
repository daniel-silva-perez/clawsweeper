## Summary

Keep open. Current main still has a numeric CLI no-output watchdog for claude-cli, still uses the 180s resumed cap, and still refreshes liveness from subprocess stdout/stderr rather than from a Monitor-aware expected-idle state. The related cap-raising issue #40982 and PR #41022 are still open and would not by themselves prove that a pending Monitor/OpenClaw MCP long-running tool is exempted safely.

## Best Possible Solution

Keep this issue open and implement a narrow Monitor-aware expected-idle path for claude-cli. The fix should suspend or refresh the no-output watchdog only while a real pending long-running Monitor/OpenClaw MCP tool is active, preserve the watchdog for genuinely silent stalled subprocesses, and add regression coverage for both claude-stdio live sessions and the non-live JSONL/supervisor path if applicable. Coordinate with #40982/#41022 so a broad cap change does not close the Monitor-specific case without coverage.
