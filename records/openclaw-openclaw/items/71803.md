## Summary

Keep open. Current main still treats stdout/stderr silence as a CLI no-output timeout for both the Claude live-session path and the generic supervised CLI path, with no code evidence that a pending Monitor/tool wait is classified as expected idle.

## What This Changes

The issue requests that CLI watchdog behavior stop terminating Claude CLI sessions that are intentionally silent while waiting for a long-running Monitor task.

## Best Possible Solution

Add a CLI-runner expected-idle or heartbeat contract for pending tool/background-job waits, with focused regression coverage proving that a silent pending Monitor wait survives while a genuinely stalled CLI backend is still terminated. If maintainers intentionally choose timeout tuning instead, document that as the product behavior and make the cap configurable enough for long Monitor workflows.

## Reproduction Assessment

Yes. The issue includes a concrete Monitor wait command and matching watchdog logs, and current-main source shows the same output-driven no-output timeout path is still active.

## Solution Assessment

Unclear. Raising caps in #40982/#41022 is a useful mitigation, but the narrower maintainable fix is likely a pending-tool expected-idle signal or heartbeat so the watchdog can distinguish supported waiting from a stalled CLI process.
