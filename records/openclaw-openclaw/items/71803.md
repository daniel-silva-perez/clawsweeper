## Summary

Keep this open: current `main` still enforces a hard no-output watchdog on CLI runs, including `claude-cli` live sessions, and I found no Monitor-aware expected-idle exemption or heartbeat path. The related #40982/#41022 work is useful context, but it is still open and only raises generic watchdog caps rather than proving this Monitor-specific failure mode is fixed.

## What This Changes

Prevent the CLI no-output watchdog from killing correctly idle `claude-cli` sessions while a long-running Monitor/tool task is pending.

## Best Possible Solution

Add an explicit expected-idle contract for CLI-backed runs: pause or heartbeat the no-output watchdog while a provider-visible long-running tool/Monitor operation is pending, or expose a documented per-backend/session watchdog override if the provider stream cannot identify that state. Cover both the process-supervisor and Claude live-session paths with focused regression tests.

## Reproduction Assessment

Yes for the watchdog failure mode: a CLI run that emits no stdout/stderr longer than the resolved `noOutputTimeoutMs` deterministically reaches `no-output-timeout`, and the issue log matches the current runner’s timeout text. The exact Monitor-tool state still needs provider-stream verification before choosing the final implementation shape.

## Solution Assessment

Unclear. Raising the generic cap, as in #41022, is a partial mitigation; the best fix should either recognize pending Monitor/tool work as expected idle or provide a narrowly documented override when that state is not observable.
