## Summary

Keep open. Current main still computes and enforces a CLI no-output watchdog for both normal CLI runs and Claude live sessions, with no source evidence of a Monitor or pending-tool expected-idle exemption; the related open timeout-cap work may mitigate the symptom but does not clearly resolve the Monitor-specific behavior.

## What This Changes

Request that the CLI no-output watchdog avoid terminating Claude CLI sessions that are intentionally idle while a long-running Monitor/background task is pending.

## Best Possible Solution

Keep this open until the CLI runner distinguishes expected idle from a stalled backend for long-running tool waits, or maintainers explicitly decide that a documented backend/session watchdog override plus the cap increase is the intended product behavior. The end state should include focused regression coverage around no-output watchdog behavior during a pending long-running tool/background wait.

## Reproduction Assessment

Yes. The issue provides concrete logs showing a 180000 ms no-output timeout during a Monitor wait, and current main's source and tests show that a silent CLI process still reaches the no-output timeout path.

## Solution Assessment

Unclear. Raising the timeout cap would reduce failures for 10-15 minute jobs, but the narrower maintainable fix is likely a Monitor/pending-tool expected-idle signal or heartbeat so genuinely stalled CLI processes can still be killed.
