## Summary

Keep open. Current main still lacks stable behavioral capability descriptors for tools across the core tool type, plugin tool metadata, before_tool_call/trusted-policy payloads, tools.catalog/tools.effective protocol schemas, and effective inventory projection. The related #48503 remains open complementary design context, not a merged implementation or superseding tracker.

## What This Changes

Add optional behavioral capability descriptors to tool definitions so policy plugins can reason about tool behavior without brittle tool-name matching.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, let built-in and plugin tools declare it optionally, project it through tools.catalog and tools.effective, and surface it to trusted policies plus before_tool_call payloads with API baseline checks, focused contract tests, and public docs.
