## Summary

Keep open. Current main still lacks stable behavioral capability descriptors on the core tool type, plugin tool metadata registration, before_tool_call/trusted-policy payloads, Gateway tools.catalog/tools.effective schemas, and effective inventory projection. The related #48503 proposal is complementary and open, not a merged superseding implementation.

## What This Changes

The issue requests optional, unknown-by-default behavioral capability descriptors on tool definitions so plugins and policies can reason about tool behavior without brittle tool-name matching.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, allow built-in and plugin tools to declare it optionally, project it through tools.catalog and tools.effective, surface it to trusted policies plus before_tool_call payloads, and update API baselines, contract tests, docs, and changelog together.
