## Summary

Keep open. Current main still does not expose stable behavioral capability descriptors for tools: the core tool type, plugin tool metadata registration, before_tool_call event/dispatch, tools.catalog/tools.effective protocol schemas, and effective inventory projection all stop at names, params, source, risk, tags, and display metadata. The related #48503 hook-enrichment proposal remains complementary open design context, not a merged implementation or superseding tracker.

## What This Changes

The issue requests optional structured capability descriptors on built-in and plugin tool definitions so policy hooks and audit surfaces can reason about tool behavior without brittle tool-name matching.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, let built-in and plugin tools declare it optionally, project it through tools.catalog and tools.effective, and surface it to trusted policies plus before_tool_call payloads with focused contract tests and public docs.
