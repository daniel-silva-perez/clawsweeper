## Summary

Current main does not implement the requested optional per-tool capability descriptors. OpenClaw has tool names, descriptions, parameters, display summaries, plugin ownership metadata, and plugin tool catalog metadata, but no typed `readsExternalData`/`sendsExternally`/`writesPersistentState`-style descriptor on tool definitions, no built-in tool self-declarations, and no capability metadata in `before_tool_call`. The related #48503 is still open and complementary rather than a superseding implementation.

## Best Possible Solution

Keep this issue open and handle it as an additive core/plugin SDK design task: define a typed optional tool capability descriptor, let built-in tools and plugin-registered tools declare it, preserve it through tool wrapping/schema normalization, expose it in effective tool inventory or CLI/docs surfaces, and coordinate with #48503 so `before_tool_call` policy plugins can consume the metadata with conservative unknown defaults.

## Risks / Open Questions

- Capability declarations should be treated as advisory for third-party plugins and unknown when absent; policy code must not silently equate missing descriptors with safe behavior.
- Overloading current free-form `tags` or `risk` metadata would likely recreate brittle string matching and would not cover built-in tools, so the cleaner path is an explicit typed contract.
