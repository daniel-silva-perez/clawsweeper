## Summary

Keep open. Current main still does not provide stable behavioral capability descriptors for tools: the plugin tool type, tool metadata registration, before_tool_call event, trusted policy dispatch, tools.catalog/tools.effective schemas, and effective inventory projection all stop at names, params, source/provenance, risk, tags, and display metadata. The related #48503 proposal remains a complementary open design item rather than a merged superseding implementation.

## What This Changes

The issue requests optional structured behavioral capability descriptors on tool definitions so policy plugins can reason about tool effects without hardcoded tool-name matching.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, let built-in and plugin tools declare it optionally, project it through tools.catalog and tools.effective, and surface it to trusted policies plus before_tool_call hook payloads with docs, API baseline updates, and focused contract tests.
