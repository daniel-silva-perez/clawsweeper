## Summary

Keep this open. Current main still has adjacent tool display metadata and working before_tool_call enforcement, but it does not define stable behavioral capability descriptors, expose them through tool catalog/effective inventory, or pass them to trusted policies or plugin hook events. The related #48503 proposal is complementary and still open rather than a merged superseding fix.

## What This Changes

Add optional behavioral capability descriptors to tool definitions so core and plugin tools can declare actions such as external reads, persistent writes, outbound sends, privileged execution, and autonomous scheduling for policy, onboarding, docs, and audit use.

## Best Possible Solution

Design an additive, versioned tool capability descriptor contract owned by core/plugin SDK, let built-in and plugin tools optionally declare it, project it through tools.catalog and tools.effective, and pass it to trusted tool policies plus before_tool_call hook payloads with docs, API baseline updates, and focused contract tests.
