## Summary

Keep this issue open. Current main has adjacent plugin tool metadata (`displayName`, `description`, `risk`, `tags`) and functioning `before_tool_call` policy hooks, but it does not define stable behavioral capability descriptors on tool definitions, project them through `tools.catalog` or `tools.effective`, or pass them to trusted policies and plugin hook payloads. Related #48503 is complementary and remains open, not a merged superseding fix.

## What This Changes

The issue requests optional behavioral capability descriptors on core and plugin tool definitions so policy hooks, catalogs, docs, and audit logging can reason about stable tool capabilities instead of tool-name lists.

## Best Possible Solution

Keep this open for maintainer design. The best path is an additive, versioned tool capability descriptor contract that core and plugin tools can declare, projected through tools.catalog/tools.effective and passed to trusted tool policies plus before_tool_call hooks. Coordinate the hook payload and provenance parts with #48503, then add focused SDK/protocol/docs/tests and audit the built-in tools before exposing the descriptors as policy signals.
