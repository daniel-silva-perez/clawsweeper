## Summary

Keep open. Current main still lacks stable behavioral capability descriptors for tools: the hook event, trusted policy dispatch, plugin tool metadata, gateway catalog/effective schemas, and effective inventory projection all stop at name, params, source/provenance, risk, tags, and display metadata. The related #48503 proposal is complementary and remains open, so there is no merged superseding implementation.

## What This Changes

Add optional behavioral capability metadata to built-in and plugin tool definitions so policies, onboarding, docs, and audit logs can reason about tool effects without brittle tool-name matching.

## Best Possible Solution

Design an additive, versioned core/plugin SDK contract for behavioral tool capability descriptors, let built-in and plugin tools declare it optionally, project it through tools.catalog and tools.effective, and surface it to trusted policies plus before_tool_call hook payloads with docs, API baseline updates, and focused contract tests.
