## Summary

Keep open. Current main still lacks the requested stable tool behavioral capability descriptor contract: core tool types, plugin tool metadata registration, before_tool_call/trusted-policy payloads, tools.catalog/tools.effective schemas, and effective inventory projection all stop at names, params, provenance/source, display metadata, risk, and tags. Related #48503 remains a complementary open design item, not a merged superseding implementation.

## What This Changes

The issue requests optional behavioral capability descriptors on tool definitions so plugins and policy hooks can reason about stable tool capabilities instead of brittle tool-name lists.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, allow built-in and plugin tools to declare it optionally, project it through tools.catalog and tools.effective, and surface it to trusted policies plus before_tool_call payloads with API baselines, docs, and focused contract tests updated together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is static inspection of the current tool type, plugin metadata registration, hook payload, dispatch code, gateway schemas, docs, and related open design context.

## Solution Assessment

Unclear pending maintainer design. Capability descriptors are a plausible direction, but the exact taxonomy and relationship to #48503 should be settled before implementation because this becomes public SDK, Gateway protocol, and security-policy input.
