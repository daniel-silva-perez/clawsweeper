## Summary

Keep open. Current main still lacks the requested stable behavioral capability descriptor contract for tools; the central SDK, hook, registry, catalog, and effective-inventory surfaces still expose names, params, display metadata, risk, tags, source, and provenance-like session context, but not self-declared tool behavior flags. The existing ClawSweeper comments match the current code, and related #48503 remains an open complementary design item rather than a merged implementation.

## What This Changes

Add optional capability descriptor metadata to tool definitions so policy plugins can reason about tool behavior instead of matching tool names.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, let built-in and plugin tools declare it optionally, project it through tools.catalog and tools.effective, and surface it to trusted policies plus before_tool_call hook payloads with API baselines, docs, and focused contract tests updated together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is static inspection of the current tool type, plugin metadata registration, hook payload, dispatch code, gateway schemas, docs, and the related open design context.

## Solution Assessment

Unclear pending maintainer design. Capability descriptors are a plausible direction, but the exact taxonomy and relationship to #48503 should be settled before implementation because this becomes public SDK, Gateway protocol, and security-policy input.
