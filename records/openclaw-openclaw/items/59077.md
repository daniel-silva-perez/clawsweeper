## Summary

Keep open. Current main and the latest release still ship the reported shared status reaction defaults, including `stallSoft: "🥱"` and `stallHard: "😨"`; Discord and Slack inherit those defaults, Telegram resolves from the same defaults and still falls back through the old stall variants, and existing tests preserve the current behavior. The report is narrow, reproducible from source/tests, and has supporting user discussion rather than being stale cleanup.

## What This Changes

Request clearer default lifecycle status reaction emoji, especially replacing the default soft/hard stall indicators with more readable slow/stuck status markers while preserving config overrides.

## Best Possible Solution

Make a small shared channel UX change in `src/channels/status-reactions.ts`, update Telegram-supported fallback variants, and adjust focused Discord/Slack/Telegram coverage plus docs/help/changelog so the shipped defaults and documented override path stay aligned.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to instantiate the status reaction controller with no `messages.statusReactions.emojis` override and advance the stall timers; the source and existing tests show it emits `DEFAULT_EMOJIS.stallSoft` and `DEFAULT_EMOJIS.stallHard`, which are still `🥱` and `😨`.

## Solution Assessment

Yes, changing the shared defaults is the narrowest maintainable fix because config overrides only help power users after they notice the confusing behavior. The safer implementation should keep the existing override API and timing behavior, while adding Telegram-specific supported fallbacks for any shared defaults Telegram cannot use directly.
