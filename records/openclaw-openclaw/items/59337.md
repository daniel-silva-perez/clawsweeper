## Summary

Keep open. Current main intentionally removed the legacy node shell wrapper and still lacks an equivalent direct CLI command for batch remote shell execution; the supported path is the session-scoped exec tool with host=node, so the user’s requested CLI ergonomics remain unresolved and need maintainer product/security judgment rather than cleanup closure.

## What This Changes

Requested restoration of direct CLI-based remote shell command execution on paired OpenClaw nodes after the legacy nodes run command was removed.

## Best Possible Solution

Maintainers should decide whether OpenClaw wants a first-class non-session CLI for node shell execution; if accepted, it should be a narrow CLI wrapper around the existing exec host=node approval pipeline, not a raw `nodes invoke system.run` path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug repro, but the current behavior is source-backed: `nodes run` is intentionally unregistered and `nodes invoke --command system.run` is blocked by code and tests; no live test was run because this review was read-only.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Restoring the old `nodes.run` shape may not be best because current node shell execution is approval-bound through exec host=node; the safer direction is a maintainer-approved CLI facade over that same pipeline if direct batch execution is desired.
