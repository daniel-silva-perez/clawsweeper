## Summary

Keep open. Current main serializes Slack sends per account/token/recipient/thread, but it still lacks the requested configurable minimum-delay throttle needed to force distinct Slack timestamps; the related broad outbound rate-limiting PR is closed unmerged.

## What This Changes

Add a configurable Slack-specific outbound send throttle so rapid sequential Slack messages get distinct timestamps and preserve display order.

## Best Possible Solution

Add a focused Slack account-level throttle setting with strict schema/types/UI hints/docs and apply one shared delay helper consistently across normal sends, DMs, thread replies, block/presentation sends, chunked text, media-caption follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call sendMessageSlack twice rapidly, or send a chunked Slack reply with a fake Slack client, and observe that current main serializes calls without any timer-enforced minimum delay.

## Solution Assessment

Yes. A Slack-owned configurable delay in the Slack plugin send paths is the narrow maintainable fix; the closed broad multi-channel throttle PR was more invasive than this Slack timestamp-ordering issue requires.
