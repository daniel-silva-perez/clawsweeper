## Summary

Keep open. Current main has per-target Slack send serialization, but it does not implement the requested configurable minimum-delay throttle for Slack timestamp ordering, and the related broad outbound rate-limiting PR is closed unmerged.

## What This Changes

Request a Slack-specific configurable outbound send throttle so rapid sequential Slack messages get distinct timestamps and render in order.

## Best Possible Solution

Add a focused Slack account-level throttle setting with types, strict schema validation, config UI hints, docs, and tests, then apply one shared delay helper consistently to Slack normal sends, DMs, thread replies, block/presentation sends, text chunks, media-caption follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call sendMessageSlack with two rapid sends or a long chunked message using a fake Slack client and observe that current main serializes calls without any timer-enforced minimum delay; a live Slack display-order repro would need a workspace but is not required to see the missing throttle.

## Solution Assessment

Yes. A Slack-owned configurable delay in the Slack plugin send paths is the narrow maintainable fix; the closed broad PR that tried a shared multi-channel throttle is more invasive than this Slack timestamp-ordering problem requires.
