## Summary

Keep open. Current main serializes Slack sends per target, but it still lacks the requested configurable minimum-delay throttle and leaves multipart/slash follow-up delivery loops without timestamp spacing. The broad related throttle PR was closed unmerged, so there is still actionable Slack-owned work here.

## What This Changes

Request to add a Slack-specific configurable outbound send throttle so rapid agent messages receive distinct Slack timestamps and display in order.

## Best Possible Solution

Implement the throttle inside the Slack plugin/config surface, not as a broad core multi-channel throttle: add a typed account-level setting with schema/UI/docs coverage, apply it consistently to Slack post surfaces including chunks and slash response_url follow-ups, and keep existing per-target serialization semantics intact.

## Repair Work Prompt

Implement the Slack outbound message throttle requested by this issue. Current main only serializes Slack sends per account/token/recipient/thread; it does not enforce a minimum inter-message delay. Add a Slack-owned, typed, configurable account-level setting such as `sendThrottleMs` with a conservative default chosen by maintainer policy, strict schema validation, config UI hints, docs, and changelog entry. Apply the effective delay consistently to all Slack outbound post surfaces: normal `sendMessageSlack` channel/DM/thread sends, block sends, text chunks, media-caption follow-up chunks after uploads, streaming fallback delivery through `deliverReplies`, and slash command `response_url` follow-ups from `deliverSlackSlashReplies`. Keep the implementation inside the Slack plugin/config surface unless a small generic helper is clearly reusable; do not revive the broad multi-channel throttle from PR #26732 and do not add dependency or lockfile changes. Add focused fake-timer tests for serialization plus spacing, configured zero/override behavior, multi-account isolation, chunk/media follow-ups, and slash response follow-ups. Update public Slack docs and add a single-line changelog entry with reporter credit.
