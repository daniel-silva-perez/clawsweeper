## Summary

Current main still lacks a configurable Slack outbound minimum-delay throttle. Slack sends are serialized by target/thread, but there is no last-send timestamp spacing in the send path, slash follow-up path, Slack config types/schema, docs, or tests; the related PR #26732 was closed unmerged.

## Best Possible Solution

Keep this issue open and implement a focused Slack outbound minimum-delay feature in the bundled Slack plugin. The likely shape is a typed, validated, documented per-account/top-level Slack setting with a default around 1000ms, applied to normal channel sends, DMs, thread replies, chunked text/media follow-ups, and any in-scope slash/respond follow-ups. PR #26732 is useful prior art but was closed unmerged and should not be treated as the shipped solution.

## Clownfish Work Prompt

Implement a focused Slack outbound minimum-delay throttle for timestamp ordering.

Observable problem: Slack can display multiple outbound messages sent within the same second out of order. Current main serializes Slack sends but does not enforce a configurable delay between posts.

Expected fix boundary:
- Keep the change Slack-specific in the bundled Slack plugin unless an existing generic seam already fits cleanly.
- Add a typed, validated Slack config setting for outbound inter-message delay, available at top-level `channels.slack` and per-account. Prefer a clear name such as `outbound.interMessageMs` if it fits existing config style; document defaults and how to disable/tune it.
- Default should preserve the issue intent, around 1000ms for Slack timestamp ordering, unless maintainers have an established config-default convention that requires opt-in.
- Apply the delay to normal channel sends, DMs, thread replies, chunked text posts, and media follow-up text posts. Include slash `response_url` follow-ups only if they can use the same Slack account/target scoping without awkward core changes; otherwise leave a narrow TODO or separate follow-up note.
- Preserve existing serialization by account/token/recipient/thread and avoid unbounded maps/timers.
- Do not change Telegram, Discord, or generic channel behavior in this PR.
- Add or update public docs and generated config metadata/schema as required.
- Add a changelog entry under the active version with `Thanks @doomclaw`; do not credit Codex/OpenClaw/steipete.

Useful prior context: PR #26732 attempted a broad Telegram/Discord/Slack rate limiter but was closed unmerged; do not resurrect its broad scope. Avoid the race/unbounded-map concerns called out in that PR discussion.
