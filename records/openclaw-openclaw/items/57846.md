## Summary

Keep open. Current main still treats Mattermost mention gating as a single account/group/channel decision: it routes Mattermost thread replies with `threadRootId`, but the mention gate does not use that thread context to allow unmentioned follow-ups after bot participation. The config schema, types, docs, and tests still expose `requireMention`/`replyToMode` only, with no Mattermost thread mention policy equivalent.

## Best Possible Solution

Implement this in the bundled Mattermost plugin. Add an account/group-level thread mention policy, likely aligned with Slack’s `thread.requireExplicitMention` semantics or the issue’s proposed `requireMentionInThread`; wire it into `evaluateMattermostMentionGate` using `threadRootId` plus bot-participation or active-session evidence; keep top-level channel `requireMention` unchanged; update the strict config schema/types/docs and add monitor-gating tests for `oncall` channel posts requiring mentions while bot-participated thread follow-ups do not.

## Risks / Open Questions

- A naive implementation that treats every Mattermost CRT reply as an implicit mention could make the bot respond in unrelated followed threads. The bypass should require bot participation, active session ownership, or similarly strong evidence that the thread is already directed at the bot.
- The config naming/default should be aligned with existing Slack thread mention behavior where possible, while preserving Mattermost’s current top-level `requireMention` semantics.
