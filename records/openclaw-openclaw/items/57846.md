## Summary

Keep open. Current main still applies one Mattermost mention-gating decision per channel/group: it detects Mattermost thread roots for routing, but the mention gate does not use the thread root to relax `requireMention`. The config schema and docs expose `chatmode`, `replyToMode`, and per-channel/group `requireMention`, but no `requireMentionInThread` or equivalent Mattermost thread mention policy. Related Slack/Feishu work is channel-specific and does not implement the Mattermost request.

## Best Possible Solution

Implement this in the Mattermost plugin, not core: add an account/group-level thread mention policy field, wire it into `evaluateMattermostMentionGate` using `threadRootId` plus bot-participation or active-session evidence, preserve top-level `requireMention`, and add config schema, docs, and monitor-gating regression tests for channel mention required plus thread follow-up allowed.

## Risks / Open Questions

- A naive default that bypasses mentions for every Mattermost CRT reply could make the bot respond in unrelated followed threads; the implementation should prove bot participation or active session ownership before bypassing top-level mention gating.
- Config naming/defaults should be aligned with existing channel behavior where possible so Mattermost does not diverge unnecessarily from Slack's thread mention controls or Feishu's active-topic approach.
