## Summary

Keep this PR open: current main still has the misleading empty-result errors, and the latest PR head is a plausible narrow fix, but it still needs normal maintainer review because its tests only accept either old or new wording and the PR's aggregate `check` status is not green.

## What This Changes

The PR adds a `hasConfiguredCapabilityProvider` helper and uses it in the image describe and audio transcribe empty-result paths, while loosening two CLI tests to accept either the old path-based message or the new missing-provider message.

## Best Possible Solution

Land a focused CLI fix that keeps the provider-configuration check at the empty-result boundary, preserves the old empty-provider-response wording when a provider is configured, adds branch-specific tests for configured and no-provider cases, and passes the focused CLI test plus the changed gate.
