## Summary

Keep this PR open. The Feishu/Lark webhook setup bug is still present on current `main`, and this PR contains useful source work for #58905, but the branch is currently dirty against `main` and the patch should be ported with one additional auth-boundary guard for malformed encrypted challenge bodies.

## What This Changes

The PR adds Feishu webhook authorization helpers that accept `url_verification` challenges when the normal signature is valid or the configured verification token matches the plaintext or encrypted challenge payload, plus e2e coverage for unsigned plaintext and encrypted challenges.

## Best Possible Solution

Port the token-verified `url_verification` exception onto the current Feishu webhook transport while preserving current pre-auth body limits, required webhook secrets, fail-closed non-challenge signature checks, and security regression coverage for wrong-token and malformed encrypted payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard SDK challenge parsing for malformed encrypted bodies:** `extensions/feishu/src/monitor.transport.ts:301-303`
  - body: This calls `Lark.generateChallenge()` before any signature or token authorization. In `@larksuiteoapi/node-sdk@1.62.0`, `generateChallenge()` decrypts and `JSON.parse`s `encrypt` payloads without catching failures, so an unsigned body such as `{ "encrypt": "bad" }` throws into the outer handler and returns 500 instead of failing closed with 401. Parse/decrypt the challenge through a caught helper before calling the SDK, or catch SDK challenge parsing failures and reject them.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR does not add dependencies, workflows, secrets access, or execution paths, but its auth-relaxation path should fail closed for malformed unsigned encrypted challenge bodies before merge.

Concerns:

- **[low] Malformed encrypted challenges can reach a 500 path:** `extensions/feishu/src/monitor.transport.ts:301`
  - body: Because the patch calls `Lark.generateChallenge()` before authorizing the request, an unauthenticated malformed `encrypt` payload can trigger SDK decrypt/parse errors and return `500 Internal Server Error` instead of the intended `401 Invalid signature` boundary.
  - confidence: 0.86
