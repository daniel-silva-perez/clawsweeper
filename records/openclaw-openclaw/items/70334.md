## Summary

Keep this open. Current main has a useful Codex app-server timeout that should reduce one likely way the reported `openai-codex/gpt-5.4` retry can stay active forever, but I did not find issue-specific source-and-test proof that the overflow-compaction success path clears the active session and lets follow-up channel messages run after the retried prompt hangs or errors.

## What This Changes

The issue reports that after overflow auto-compaction succeeds and the prompt is retried, the session remains in `processing` with `queueDepth=0` and blocks later channel messages until restart.

## Best Possible Solution

Add a focused regression around overflow-compaction success followed by a retried prompt that succeeds, errors, and never produces a terminal event; the fixed behavior should clear the active embedded run, leave diagnostics idle or bounded-error, drain the lane, and allow a follow-up channel message without requiring gateway restart.

## Reproduction Assessment

Yes for an automated regression path: the issue supplies a concrete log sequence and current code still has the same overflow-compaction success retry branch. A live long-session reproduction was not run in this read-only review, but the repo can mock the overflow prompt error, compaction success, and silent retried Codex attempt.

## Solution Assessment

Unclear. The current Codex app-server timeout is a good partial mitigation for the reporter's model, but closing would need a regression proving the overflow-compaction retry path itself releases the session lane and handles follow-up messages.
