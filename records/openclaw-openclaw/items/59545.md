## Summary

Keep open. Current main still contains the macOS General settings path described in the report: the local General view constructs the Tailscale settings section, that section hydrates SwiftUI state from config, and its change handlers route into a save path that rewrites gateway config and stamps metadata. I found smoke coverage for the view, but no regression that proves hydration is read-only.

## What This Changes

The issue asks for the macOS app General settings page to load existing config without rewriting `~/.openclaw/openclaw.json` unless the user explicitly changes a setting.

## Best Possible Solution

Keep the behavior owned in the macOS app: make Tailscale/General settings distinguish initial hydration from user edits, avoid calling `ConfigStore.save` on view entry, and add focused regression coverage that proves loading existing local config leaves `gateway`, `meta`, and auth fields unchanged until a user action occurs.

## Reproduction Assessment

Yes. The report provides a concrete mtime/diff-based macOS UI reproduction path, and current-main source inspection still shows the same General-tab hydration and save-on-change route that can produce the write.

## Solution Assessment

Yes. The narrow maintainable fix is to gate or dirty-track the Tailscale settings save handlers during initial config hydration; changing core config stamping, gateway defaults, or auth policy would be broader than the reported UI side effect.
