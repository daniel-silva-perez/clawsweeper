## Summary

Keep open. Current main still has a source-backed path where opening General constructs the Tailscale settings section, hydrates SwiftUI state from config, and uses asynchronous onChange handlers that can call the same persistence path intended for user edits. The save path rewrites the full config and stamps meta fields, and the current macOS tests only smoke-render this section rather than asserting that hydration performs no save.

## What This Changes

Opening macOS Settings > General should load the existing config without rewriting `~/.openclaw/openclaw.json` unless the user changes a setting.

## Best Possible Solution

The macOS Tailscale settings section should separate initial hydration from user-driven edits: loading config into state must not call `ConfigStore.save`, while explicit picker/toggle/password actions should keep the existing Tailscale config update behavior. Add a focused macOS test that injects ConfigStore overrides and fails if the initial General/Tailscale load attempts any save.

## Reproduction Assessment

Yes. The issue gives concrete mtime/diff steps, and current source still exposes the same General > Tailscale hydrate-then-save path; the only missing piece in this review is a live macOS app run.

## Solution Assessment

Yes. The narrow maintainable fix is to suppress persistence during initial `TailscaleIntegrationSection` hydration, likely with an `isHydrating`/loaded gate before scheduling save tasks, rather than weakening generic config save stamping or audit behavior.
