## Summary

Keep open. Current main still does not use `plugins.entries.openai.config.stt` for browser Talk realtime sessions, and the reported error matches the current resolver path when only that batch-STT-style config is present. The remaining decision is product/API scope: improve the Talk realtime config contract and error guidance, or intentionally support local Whisper through a realtime voice/provider plugin rather than an implicit OpenAI STT alias.

## What This Changes

The issue requests that Talk mode initialize the `openai` voice path from a local Whisper-compatible STT endpoint configured under `plugins.entries.openai.config.stt` instead of reporting the realtime provider as unconfigured.

## Best Possible Solution

Define the Talk realtime contract explicitly: validate or warn that `plugins.entries.openai.config.stt` and `tools.media.audio` do not configure browser Talk, improve the `talk.realtime.session` error to point to `talk.providers.<provider>` or `plugins.entries.voice-call.config.realtime.providers.<provider>`, and support local Whisper Talk only through a dedicated realtime voice/provider plugin if maintainers want that workflow.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: with only `plugins.entries.openai.config.stt` set, the Talk realtime builder ignores that block, and selecting `openai` reaches the realtime resolver with no usable provider config, producing the reported `Realtime voice provider "openai" is not configured` path.

## Solution Assessment

No. Treating an OpenAI-compatible batch transcription endpoint as a browser Talk realtime provider via `plugins.entries.openai.config.stt` is not the narrowest maintainable fix; the safer path is explicit config guidance plus a dedicated realtime provider/plugin if local Whisper Talk support is desired.
