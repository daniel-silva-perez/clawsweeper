## Summary

Keep open. Current main still does not route `plugins.entries.openai.config.stt` into Control UI Talk realtime provider resolution, and the reported unconfigured OpenAI realtime provider error still matches the current resolver path. The remaining decision is product/API scope: make this unsupported configuration fail clearly, or support local Whisper through a dedicated realtime voice provider/plugin path.

## What This Changes

The issue asks Control UI Talk to use an OpenAI-compatible local Whisper STT configuration from `plugins.entries.openai.config.stt` instead of failing provider resolution as unconfigured.

## Best Possible Solution

Define the browser Talk realtime contract explicitly. At minimum, surface a clear diagnostic that `plugins.entries.openai.config.stt` and `tools.media.audio` do not configure Control UI Talk realtime; if local Whisper Talk is desired, add or document a dedicated realtime voice provider plugin path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure only `plugins.entries.openai.config.stt`, request `talk.realtime.session`, and current main builds no matching realtime provider config before reaching the unconfigured OpenAI realtime provider path.

## Solution Assessment

No. Treating an OpenAI-compatible batch transcription endpoint as an implicit realtime Talk provider is not clearly the narrowest maintainable fix; explicit diagnostics plus a real realtime voice provider/plugin path is safer if maintainers want local Whisper Talk support.
