## Summary

Keep open. Current main still does not route `plugins.entries.openai.config.stt` into browser Talk realtime provider resolution, and the reported `Realtime voice provider "openai" is not configured` error remains the expected resolver outcome for that configuration. This is a real product/API boundary gap, not a stale or implemented report.

## What This Changes

The issue requests that Control UI Talk use an OpenAI-compatible local Whisper STT configuration from `plugins.entries.openai.config.stt` instead of failing realtime provider resolution as unconfigured.

## Best Possible Solution

Define the browser Talk realtime contract explicitly. At minimum, surface a clear diagnostic that `plugins.entries.openai.config.stt` and `tools.media.audio` do not configure Control UI Talk realtime; if local Whisper Talk is desired, add or document a dedicated realtime voice provider plugin path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure only `plugins.entries.openai.config.stt`, request `talk.realtime.session`, and current main builds no matching realtime provider config before reaching the unconfigured OpenAI realtime provider path.

## Solution Assessment

No. Treating an OpenAI-compatible batch transcription endpoint as an implicit realtime Talk provider would conflate batch STT with duplex realtime voice; clearer diagnostics plus a real realtime voice provider/plugin path is the safer maintainable direction.
