## Summary

Keep open. Current main still does not route `plugins.entries.openai.config.stt` into browser Talk realtime sessions, and the reported error matches the current provider resolver when only that batch-STT-style config is present. The remaining decision is product/API scope: clarify the Talk realtime contract and diagnostics, or add/support a local Whisper realtime voice provider path.

## What This Changes

The issue requests that Control UI Talk mode honor a local OpenAI-compatible Whisper STT configuration under `plugins.entries.openai.config.stt` instead of reporting the `openai` realtime voice provider as unconfigured.

## Best Possible Solution

Define the browser Talk realtime contract explicitly. At minimum, warn or error clearly that `plugins.entries.openai.config.stt` and `tools.media.audio` do not configure Control UI Talk realtime; if local Whisper Talk is desired, add or document a dedicated realtime voice provider plugin path rather than treating batch OpenAI-compatible transcription as an implicit realtime OpenAI alias.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure only `plugins.entries.openai.config.stt`, request `talk.realtime.session` with provider `openai`, and current main builds no matching realtime provider config before reaching the `Realtime voice provider "openai" is not configured` resolver path.

## Solution Assessment

No. Routing an OpenAI-compatible batch transcription endpoint directly into browser realtime Talk is not clearly the narrowest maintainable fix; the safer solution is explicit config diagnostics plus a real realtime voice provider/plugin if maintainers want local Whisper Talk support.
