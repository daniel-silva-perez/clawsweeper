## Summary

Keep this issue open. Current main does not wire `plugins.entries.openai.config.stt` into browser Talk realtime sessions: `talk.realtime.session` resolves registered realtime voice provider config from `talk.providers.*` and the Voice Call realtime config, while OpenAI realtime voice is considered configured only when its realtime provider config or `OPENAI_API_KEY` supplies an API key. The report is recent, has concrete logs/config, and points to a real docs/config/product gap around local Whisper batch STT versus Talk realtime voice rather than something already fixed or clearly closeable.

## Best Possible Solution

Clarify and/or fix the Talk realtime configuration path: at minimum, document and validate that `plugins.entries.openai.config.stt` and `tools.media.audio` batch STT do not configure browser Talk realtime, and point users to `talk.providers.<provider>` or `plugins.entries.voice-call.config.realtime.providers.<provider>`. If local Whisper Talk is desired, build or bless a local/openai-compatible realtime voice plugin using `registerRealtimeVoiceProvider`, possibly combining local Whisper STT with a configured TTS/model loop.

## Risks / Open Questions

- The issue says "start talk" but the log is from `talk.realtime.session`, so maintainers should confirm whether the reporter used browser realtime Talk rather than macOS/iOS/Android on-device Talk before changing behavior.
- A local Whisper `/v1/audio/transcriptions` server is batch STT, not a duplex realtime voice provider. Implementing this directly in core would need a product decision because the existing architecture points provider-specific local backends toward plugins.
