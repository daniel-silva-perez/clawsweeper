## Summary

Keep open. Current main still ignores `plugins.entries.openai.config.stt` for Control UI Talk realtime sessions, and the reported `Realtime voice provider "openai" is not configured` error matches the current resolver path when only that batch-STT-style config is present. The remaining decision is a product/API boundary: improve diagnostics for unsupported STT config, or explicitly support local Whisper through a realtime voice provider plugin path.

## What This Changes

The issue asks Control UI Talk to use an OpenAI-compatible local Whisper STT configuration from `plugins.entries.openai.config.stt` instead of failing provider resolution as unconfigured.

## Best Possible Solution

Define the browser Talk realtime contract explicitly. At minimum, warn or error clearly that `plugins.entries.openai.config.stt` and `tools.media.audio` do not configure Control UI Talk realtime; if local Whisper Talk is desired, add or document a dedicated realtime voice provider plugin path rather than treating batch OpenAI-compatible transcription as an implicit realtime OpenAI alias.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure only `plugins.entries.openai.config.stt`, request `talk.realtime.session`, and current main builds no matching realtime provider config before reaching the unconfigured OpenAI realtime provider path.

## Solution Assessment

No. Routing an OpenAI-compatible batch transcription endpoint directly into browser realtime Talk is not clearly the narrowest maintainable fix; explicit diagnostics plus a real realtime voice provider/plugin is safer if maintainers want local Whisper Talk support.
