## Summary

Keep open. Current main still does not consume the reported `plugins.entries.openai.config.stt` path for browser Talk realtime sessions, and the reported `Realtime voice provider "openai" is not configured` error still matches the resolver path. This is not implemented, not stale, and not clearly safe to close as ClawHub because the user-facing contract/error around Talk realtime still needs maintainer resolution.

## What This Changes

The issue requests that Talk mode use a local OpenAI-compatible Whisper STT endpoint configured under `plugins.entries.openai.config.stt`.

## Best Possible Solution

Clarify and enforce the browser Talk realtime contract: make the runtime error point users to `talk.providers.<provider>` or `plugins.entries.voice-call.config.realtime.providers.<provider>`, document that batch `tools.media.audio` and `plugins.entries.openai.config.stt` do not configure browser Talk, and only add local Whisper Talk support through an explicit realtime voice provider or bridge contract.

## Reproduction Assessment

Yes. A source-level reproduction path is high confidence: configuring only `plugins.entries.openai.config.stt` leaves `buildTalkRealtimeConfig` without an OpenAI realtime provider config, so `talk.realtime.session` reaches the resolver and can throw the reported not-configured error without requiring a live Whisper server.

## Solution Assessment

Unclear. Directly mapping a REST batch Whisper transcription endpoint into duplex browser Talk would not be the narrowest maintainable fix unless maintainers intentionally design that bridge; the safer path is contract/error/docs cleanup plus a dedicated realtime provider if support is desired.
