## Summary

Keep this issue open. Current main still does not wire `plugins.entries.openai.config.stt` into browser Talk realtime sessions, and the reported `Realtime voice provider "openai" is not configured` error matches the current resolver path. The remaining work is actionable: clarify/validate the Talk realtime config contract and decide whether local Whisper Talk support should be a dedicated realtime voice plugin/provider rather than the OpenAI batch STT config.

## Best Possible Solution

Keep this open for maintainer follow-up. The best path is to clarify and enforce the browser Talk realtime config contract: document and/or validate that `plugins.entries.openai.config.stt` and batch `tools.media.audio` STT do not configure browser Talk, improve the `talk.realtime.session` error to point users at `talk.providers.<provider>` or `plugins.entries.voice-call.config.realtime.providers.<provider>`, and, if local Whisper Talk is desired, implement it through a dedicated realtime voice provider/plugin or explicit bridge contract.
