## Summary

Keep open. Current main still keeps RTL RLI/PDI isolation private to the TUI formatter, while the shared assistant-visible/user-facing sanitizers, gateway live chat projection, plain-text sanitizer, and outbound delivery path still forward RTL text without a shared bidi-isolation stage. The prior TUI fix and Telegram-preview report are useful context but do not supersede this outbound boundary bug.

## What This Changes

The issue requests moving the existing RTL bidi-isolation behavior out of the TUI-only formatter and applying it to user-visible outbound replies for non-TUI channel and gateway surfaces.

## Best Possible Solution

Add a shared RTL isolation helper under `src/shared/text`, reuse it from the TUI formatter, and apply it on the canonical user-visible outbound path before channel chunking/sending; include gateway live projection only if that surface is intentionally user-visible under the same contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass Hebrew or Arabic text through the current shared sanitizers or outbound delivery path and it reaches `sendFormattedText`/`sendTextChunks` without the TUI RLI/PDI wrapping; live client rendering would be useful confirmation but is not the blocker.

## Solution Assessment

Yes. Extracting the existing TUI helper into a shared text module and applying it at delivery/projection boundaries is the narrowest maintainable fix, as long as tests lock no-double-wrap behavior and avoid applying bidi controls to internal prompt or transcript data.
