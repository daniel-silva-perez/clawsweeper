## Summary

Keep open. The PR targets a real sandbox configuration problem, but current main has not implemented the intended admin-configured Docker env bypass, and the submitted branch still introduces merge-blocking regressions around skill eligibility and existing sanitizer options.

## What This Changes

The PR adds a `forceAllowKeys` env-sanitizer option for Docker sandbox env values, exports a sandbox env validation helper, keeps `/root` in the host-path denylist, and changes skill filtering so explicitly named skills bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow sandbox env fix: allow only resolved operator-configured `sandbox.docker.env` keys to bypass the credential-name blocklist, preserve existing sanitizer options and value validation, keep `shouldIncludeSkill` eligibility as a prerequisite before agent allowlists unless a separate policy change is approved, and add focused sanitizer, Docker create-args, and skill-gating regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve skill eligibility when filtering skills:** `src/agents/skills/workspace.ts:79`
  - body: The explicit-name shortcut runs before `shouldIncludeSkill`, which is the path that enforces `skills.entries.<key>.enabled === false`, `skills.allowBundled`, and runtime `requires` gates. A disabled or runtime-ineligible skill that remains in `agents.*.skills` would be loaded again; keep eligibility as the prerequisite and apply the agent allowlist afterward.
  - confidence: 0.91
- **[P2] Preserve existing env sanitizer options:** `src/agents/sandbox/docker.ts:370-372`
  - body: This call replaces the options argument with `{ forceAllowKeys }`, so current `envSanitizationOptions` such as `strictMode` or custom blocked/allowed patterns are not passed through. Merge the existing options and add `forceAllowKeys` so callers can still enforce stricter filtering.
  - confidence: 0.86
