## Summary

Keep this PR open for contributor/maintainer revision. Current main does not implement the requested admin-configured sandbox env bypass, and the PR still has blocking correctness and security issues around skill eligibility and preserving existing env sanitizer policy; the earlier `/root` denylist regression noted in review was corrected in the latest PR head.

## What This Changes

The PR adds a `forceAllowKeys` option to sandbox env sanitization, uses it for configured Docker env keys, adds env validation scaffolding, keeps `/root` in the sandbox host-path denylist, and lets explicitly listed agent skills bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow sandbox env fix: allow only resolved operator-configured `agents.*.sandbox.docker.env` keys to bypass the credential-name blocklist, preserve all existing sanitizer options and value validation, keep skill eligibility gates unchanged unless a separate reviewed policy change approves them, and add focused tests for forced env keys plus strict/custom sanitizer options.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: with this PR head, a disabled or runtime-ineligible skill named in an agent allowlist passes the `explicitNames` branch, and Docker sandbox creation no longer forwards current main's `params.envSanitizationOptions`.

## Solution Assessment

No. The env bypass direction may be useful, but the submitted patch is not the narrowest maintainable solution because it drops existing sanitizer policy and mixes in an unrelated skill eligibility bypass that weakens current gating semantics.
