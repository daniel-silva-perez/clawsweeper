## Summary

Keep open. Current main still lacks the PR's central configured Docker sandbox env bypass: `sandbox.docker.env` is resolved as explicit config, but `buildSandboxCreateArgs` still sends those keys through the default credential blocklist. The PR remains useful, but it needs a focused revision because the current diff also changes skill eligibility policy and adds an unused env-validation export in the sandbox security module.

## Best Possible Solution

Keep this PR open for a focused revision. Implement a configured-key escape hatch in `sanitizeEnvVars`, apply it only when building Docker sandbox create args from resolved `sandbox.docker.env`, preserve value validation and host/process-env blocking, add sanitizer plus Docker create-args regression tests, and remove the unrelated skill eligibility bypass unless maintainers explicitly approve that policy change separately.

## Risks / Open Questions

- Merging the PR as-is can re-enable disabled, disallowed bundled, or runtime-ineligible skills whenever they appear in an agent skill filter.
- The configured-env bypass is security-sensitive and should stay scoped to resolved admin Docker sandbox config, not inherited host `process.env` or SSH/backend spawn env.
- The PR currently has no sanitizer or Docker create-args regression tests for the intended pass-through behavior and the still-blocked inherited/unconfigured credential behavior.
- The PR adds `validateEnvVars` to `validate-sandbox-security.ts` without a current call site in the live diff, which creates extra sandbox security surface without proving the central behavior.
