## Summary

Keep this PR open for maintainer/contributor revision. Current main still blocks credential-shaped keys in configured Docker sandbox env, so the requested behavior is not implemented, and the latest PR head still introduces blocking regressions in skill eligibility and Docker env sanitizer option handling.

## What This Changes

The PR adds `forceAllowKeys` to sandbox env sanitization and Docker env handling, exports env validation scaffolding, keeps `/root` in the sandbox host-path denylist on the latest head, and lets explicitly listed agent skills bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow sandbox env fix: allow only resolved operator-configured `agents.*.sandbox.docker.env` keys to bypass the credential-name blocklist, preserve existing sanitizer options and value validation, leave skill eligibility gates unchanged unless a separate reviewed policy change approves them, and add focused tests for forced env keys plus strict/custom sanitizer options.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: with this PR head, a disabled or runtime-ineligible skill named in an agent allowlist passes the `explicitNames` branch, and Docker sandbox creation no longer forwards current main's `params.envSanitizationOptions`.

## Solution Assessment

No. The env bypass direction may be useful, but the submitted patch is not the narrowest maintainable solution because it drops existing sanitizer policy and mixes in an unrelated skill eligibility bypass that weakens current gating semantics.
