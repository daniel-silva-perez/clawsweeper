## Summary

Keep open. Current main still lacks the PR's central configured Docker sandbox env bypass, so the contribution is still relevant, but the live PR diff is not safe to merge as-is because it lets explicit agent skill filters bypass disabled, bundled-allowlist, and runtime eligibility gates.

## What This Changes

The PR adds a `forceAllowKeys` sanitizer option, applies it to Docker sandbox configured env values, adds an exported sandbox env validation helper, reorders `/root` in the blocked host-path list, and lets explicitly listed skills bypass eligibility filtering.

## Best Possible Solution

Keep the PR open for a focused revision: add the configured-key escape hatch to `sanitizeEnvVars`, apply it only when building Docker create args from resolved `sandbox.docker.env`, preserve value validation and host/process-env blocking, remove the unrelated skill eligibility bypass and unused validation helper unless separately approved, and add sanitizer plus Docker create-args regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve skill eligibility when filtering skills:** `src/agents/skills/workspace.ts:80-82`
  - body: The explicit-name short-circuit makes `agents.*.skills` bypass `shouldIncludeSkill`. That helper enforces `skills.entries.<key>.enabled === false`, `skills.allowBundled`, and runtime `requires` gates, so a disabled or ineligible skill that remains in an agent allowlist would be loaded again. Keep the allowlist as a final filter after eligibility, or split this policy change into a separate reviewed design.
  - confidence: 0.91
