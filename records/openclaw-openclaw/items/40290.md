## Summary

Keep open. Current main still strips credential-shaped keys from explicitly configured Docker sandbox env, so the PR’s central bug fix is not implemented on main. The PR should be revised rather than closed because its current diff also mixes in unrelated skill eligibility bypass behavior that can override existing disabled/eligibility gates.

## Best Possible Solution

Keep this PR open for a focused revision. Implement a configured-key escape hatch in sanitizeEnvVars, apply it only when building Docker sandbox create args from resolved sandbox.docker.env, preserve value validation and host/process-env blocking, add sanitizer plus Docker create-args regression tests, and remove the unrelated skill eligibility bypass unless maintainers separately approve that policy change.

## Risks / Open Questions

- Merging the PR as-is can re-enable skills that config, bundled allowlist policy, or runtime eligibility would otherwise suppress.
- The configured-env escape hatch is security-sensitive and should be scoped only to resolved admin Docker sandbox env, not host process.env or arbitrary per-call env.
- The env sanitizer change needs focused tests for both sides: explicitly configured sandbox.docker.env credential keys pass, while unconfigured or host/process env credentials remain blocked and null-byte values remain blocked.
