## Summary

Keep open. The Docker env passthrough problem is still real on current main, but the branch is not mergeable as-is because it bypasses skill eligibility gates and drops existing Docker env sanitizer options.

## What This Changes

The PR adds a `forceAllowKeys` env-sanitizer option for sandbox Docker `env`, adds an exported sandbox env validation helper, reorders `/root` in the host-path denylist, and changes skill filtering so explicit skill names bypass eligibility checks.

## Best Possible Solution

Revise this PR into a narrower sandbox env fix: keep the configured-key escape hatch for resolved `sandbox.docker.env`, preserve existing sanitizer options and value validation, keep skill eligibility as a prerequisite before agent allowlists, and add focused sanitizer/Docker create-args plus skill-gate regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve skill eligibility when filtering skills:** `src/agents/skills/workspace.ts:80-82`
  - body: The explicit-name short-circuit makes `agents.*.skills` bypass `shouldIncludeSkill`. That helper enforces `skills.entries.<key>.enabled === false`, `skills.allowBundled`, and runtime `requires` gates, so a disabled or ineligible skill that remains in an agent allowlist would be loaded again. Keep the allowlist as a final filter after eligibility, or split this policy change into a separate reviewed design.
  - confidence: 0.91
- **[P2] Preserve existing env sanitizer options:** `src/agents/sandbox/docker.ts:370-372`
  - body: The new Docker create-args call passes only `{ forceAllowKeys }` to `sanitizeEnvVars`, so existing `envSanitizationOptions` such as `strictMode` or custom blocked patterns are ignored. Merge the existing options and then add the configured-key set so callers can still enforce stricter sandbox env filtering.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no workflow, dependency, lockfile, or release-script changes, but it does affect sandbox secret exposure and introduces a skill-gating bypass.

Concerns:

- **[medium] Skill allowlists bypass disabled and runtime gates:** `src/agents/skills/workspace.ts:80`
  - body: Explicit skill names are treated as sufficient to include a skill before `shouldIncludeSkill` runs, so config-disabled, bundled-denied, or runtime-ineligible skills can become available again through an agent allowlist.
  - confidence: 0.91

  - body: The Docker env sanitizer call no longer passes through existing custom options, which could weaken stricter caller-supplied filtering if this exported helper is used with custom blocked patterns or strict mode.
  - confidence: 0.86
