## Summary

Keep open. Current main still blocks credential-shaped `sandbox.docker.env` keys, so the PR addresses a real missing behavior, but the PR head introduces merge-blocking regressions by bypassing skill eligibility gates and dropping existing sandbox env sanitizer options.

## What This Changes

The PR adds a `forceAllowKeys` env sanitizer option for Docker sandbox config env vars, adds sandbox env validation scaffolding, restores `/root` in the host-path denylist after review, and lets explicitly listed agent skills bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow sandbox env change: allow only resolved operator-configured `agents.*.sandbox.docker.env` keys to bypass the credential-name blocklist, merge that forced-key set with existing sanitizer options, keep skill eligibility and disabled-skill gates intact, and add focused tests for forced env keys plus strict/custom sanitizer options.

## Reproduction Assessment

Yes. Static inspection provides a high-confidence reproduction path: on the PR head, an explicit skill name short-circuits `shouldIncludeSkill`, and the Docker env sanitizer call no longer forwards the current-main `params.envSanitizationOptions`.

## Solution Assessment

No. The env bypass direction is plausible, but this PR is not the best solution because it mixes an unrelated skill policy bypass into a sandbox env fix and weakens caller-supplied sanitizer policy.
