## Summary

Keep this PR open for maintainer review. The sandbox env bypass idea is plausible, and earlier review concerns about removing `/root` and the strict-mode gap have been addressed on the latest PR head, but the branch still introduces a skill eligibility bypass and drops current-main sanitizer options in a security-sensitive path.

## What This Changes

The PR adds a `forceAllowKeys` option to sandbox env sanitization for configured Docker env keys, adds unused env validation scaffolding, restores `/root` in the sandbox host-path denylist after review, and changes skill filtering so explicitly listed skills bypass eligibility checks.

## Best Possible Solution

Revise this into a narrow sandbox env change: allow only resolved operator-configured `agents.*.sandbox.docker.env` keys to bypass the credential-name blocklist, merge the forced-key set with existing sanitizer options, keep skill eligibility and disabled-skill gates intact, and add focused tests for the env bypass and preserved sanitizer policy.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: on the PR head, an explicit skill name short-circuits `shouldIncludeSkill`, and the Docker env sanitizer call no longer forwards current-main `params.envSanitizationOptions`.

## Solution Assessment

No. The env bypass direction may be useful, but this PR is not the best fix because it mixes an unrelated skill policy bypass into a sandbox env change and weakens caller-supplied sanitizer policy.
