## Summary

Keep this PR open. Current main still does not pass video width/height to Telegram `sendVideo` on either outbound path, so the intended bug fix is not implemented. The PR is also not ready to close as obsolete or merge as-is because its patch crosses the bundled-plugin boundary, uses bare `ffprobe` execution instead of the trusted resolver path, changes unrelated shared video size defaults, and lacks regression coverage for both Telegram video send paths.

## Best Possible Solution

Keep this PR open and request a focused revision: implement video dimension probing through `openclaw/plugin-sdk/media-runtime` by using `runFfprobe` safely or adding an exported helper there, pass width/height only for regular Telegram `sendVideo` in `extensions/telegram/src/send.ts` and `extensions/telegram/src/bot/delivery.replies.ts`, leave GIF animations and video notes untouched, add targeted regression tests for both send paths, and remove the unrelated shared `MAX_VIDEO_BYTES` change.
