## Summary

Current main still does not implement this PR's intended Telegram behavior: both regular outbound sendVideo paths build params without width/height. The PR remains useful directionally, but it should stay open for revision rather than be closed or merged as-is because the submitted diff crosses the bundled-plugin boundary, adds bare ffprobe execution, includes an unrelated global media limit change, and lacks regression coverage for width/height forwarding.

## Best Possible Solution

Keep this PR open and request a focused revision: add or expose a video-dimension helper through openclaw/plugin-sdk/media-runtime using the trusted ffprobe resolver path, pass width and height only for regular Telegram sendVideo calls in extensions/telegram/src/send.ts and extensions/telegram/src/bot/delivery.replies.ts, keep GIF animations and video notes excluded, add regression tests for both outbound paths, and remove or split the unrelated MAX_VIDEO_BYTES change.
