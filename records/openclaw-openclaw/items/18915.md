## Summary

Keep this PR open. Current main still does not pass video width/height to Telegram `sendVideo` in either outbound path, and the requested behavior is supported by Telegram. The submitted patch is still not ready to close or merge as-is because it crosses the bundled-plugin boundary, uses bare `ffprobe` execution instead of the trusted binary resolver path, changes an unrelated shared video-size constant, and lacks regression coverage for the two Telegram send paths.

## Best Possible Solution

Keep this PR open and request a focused revision: use the existing `openclaw/plugin-sdk/media-runtime` seam, either by calling `runFfprobe` safely or by adding an exported video-dimension helper there; pass width/height only for regular Telegram `sendVideo` calls in `extensions/telegram/src/send.ts` and `extensions/telegram/src/bot/delivery.replies.ts`; keep GIF animations and video notes excluded; add regression tests for both outbound paths; and drop the unrelated shared `MAX_VIDEO_BYTES` change.
