## Summary

Keep open. Current main still rejects `sessions_send` calls that provide both `sessionKey` and `label`, so this PR’s intended fix is not implemented or obsolete. The PR is narrow and security-sensitive only in the sense that it relaxes tool-call validation; downstream session visibility and agent-to-agent guards remain in the current send path, but the PR should gain a focused regression test before landing.

## Best Possible Solution

Keep this PR open as a valid implementation candidate. The best maintainer path is to select one canonical `sessions_send` fix, require or add a regression test proving `sessionKey` wins when both selectors are provided, merge that implementation, then close the duplicate PRs and the `sessions_send` portion of #41199 as implemented.

## Risks / Open Questions

- The PR currently lacks a focused regression test for the mixed `sessionKey` plus `label` input path; a test should prove `sessionKey` is authoritative and `sessions.resolve` is skipped when it is present.
- The behavior change is intentionally more permissive: if a caller sends mismatched `sessionKey` and `label`, the proposed behavior would route by `sessionKey`. That precedence should be explicitly tested and documented in the tool behavior.
- Security review found no supply-chain or secrets-handling changes in the PR diff, but because this changes a command-tool execution path, maintainers should verify the existing visibility and agent-to-agent guards remain downstream of target resolution before merge.
- There are multiple open implementation candidates for the same `sessions_send` bug, including #39551, #56203, and #59324; maintainers should choose one canonical patch and close the rest after a fix lands.
