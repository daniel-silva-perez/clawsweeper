## Summary

Keep open. Current main has agent runtime harness selection, per-agent config/state, sandbox backends, and Gateway/node/operator protocol roles, but it still runs agent turns from the Gateway process and lacks a control-plane-to-per-agent-runtime protocol or per-agent lifecycle commands. The discussion’s Phase 0 slice is valid but still architectural and security-sensitive, so it needs maintainer design review rather than cleanup closure.

## What This Changes

The issue proposes splitting `openclaw gateway` into a lightweight control plane and separately managed per-agent compute runtimes, starting with per-agent start/stop/restart and later distributed/container orchestration.

## Best Possible Solution

Keep this as an architecture tracker and split it into additive milestones: preserve embedded mode, define the Gateway-to-runtime contract for registration, health, capabilities, session ownership, turn delivery, cancellation, and reconnect behavior, then implement local per-agent process lifecycle before adding container/Kubernetes backends and per-agent secret/resource isolation.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. The current-main capability gap is high-confidence because docs, protocol roles, CLI registration, and Gateway dispatch still show embedded/harness-selected execution without separately managed per-agent runtimes.

## Solution Assessment

Unclear as a direct implementation plan. The direction is plausible, but the best maintainable solution is a phased maintainer-approved protocol and lifecycle design rather than an immediate broad patch.
