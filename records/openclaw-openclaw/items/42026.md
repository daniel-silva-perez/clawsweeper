## Summary

Keep this open. Current main has adjacent infrastructure such as multi-agent routing, agent runtime/harness selection, runtime-plan policy bundles, CLI backends, node hosts, sandbox runtimes, and Gateway hot reload. It still does not implement the RFC’s requested split between a lightweight Gateway control plane and separately registered per-agent compute runtimes, nor the Phase 0 lifecycle commands TinyTb called out for starting, stopping, or restarting a single agent without restarting the Gateway. Related #12429 is narrower Kubernetes node-pairing work and does not supersede this broader agent compute runtime architecture request.

## Best Possible Solution

Keep this open for maintainer architecture review. The best implementation path is an additive design that preserves today’s embedded Gateway mode, defines a stable Gateway-to-agent-runtime protocol and role/capability handshake, adds local per-agent process start/stop/restart plus health reporting, then layers container/Kubernetes backends and stronger per-agent secret/resource isolation while deliberately integrating the existing harness selection, runtime-plan policy, node transport, sandboxing, hot reload, and session routing pieces.

## Risks / Open Questions

- The RFC is broad and should likely be split into maintainer-owned milestones: runtime protocol, local per-agent process lifecycle, runtime registration/health, per-agent secret/resource isolation, then orchestration backends.
- Existing surfaces named runtime, harness, node, sandbox, and ACP are useful building blocks but could be confused with the requested distributed agent compute runtime unless future design draws explicit boundaries.
- Related #12429 remains relevant for Kubernetes worker pairing, but it is narrower infrastructure work and should not be treated as the canonical tracker for this control-plane/runtime split.
