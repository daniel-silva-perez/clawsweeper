## Summary

Keep open. Current main still has embedded agent runtime/harness selection, multi-agent routing inside one Gateway process, node/operator Gateway protocol roles, and adjacent diagnostics/reload hardening, but it does not implement a Gateway control plane that registers and routes turns to separately managed per-agent compute runtimes. The discussion’s narrower Phase 0 request for per-agent start/stop/restart without restarting the Gateway is also still absent from the documented and registered agent command surfaces.

## What This Changes

This issue requests splitting `openclaw gateway` into a lightweight control plane plus separately managed per-agent runtime processes or containers for isolation, scaling, lifecycle control, and future distributed deployment modes.

## Best Possible Solution

Keep this as an architecture tracker and sequence it additively: preserve embedded mode, define a stable Gateway-to-runtime protocol with registration, health, capabilities, session ownership, delivery, and failure semantics, add local per-agent process lifecycle first, then layer container/Kubernetes backends and per-agent secret/resource isolation.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The current-main capability check is high confidence because the docs, protocol roles, method inventory, and CLI registrations still describe embedded/local execution rather than separately managed per-agent runtimes.

## Solution Assessment

Unclear as a direct implementation path; the requested direction is plausible, but it is not yet a narrow best fix. The maintainable path is an explicit architecture decision and phased protocol/lifecycle design before implementation is promoted to focused PRs.
