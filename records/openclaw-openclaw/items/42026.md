## Summary

Keep open. Current main still has adjacent embedded runtime selection, multi-agent routing, node-host control-plane plumbing, and Gateway diagnostics work, but it does not implement a Gateway control plane that registers and routes turns to separately managed per-agent compute runtimes. The discussion’s narrower Phase 0 ask, per-agent start/stop/restart without restarting the Gateway, is also still missing from the documented and registered agent command surface.

## What This Changes

The issue requests splitting the monolithic Gateway into a lightweight control plane plus separately managed per-agent runtimes for model loops, tools, MCP servers, memory, workspace state, and isolated compute.

## Best Possible Solution

Keep this as an architecture tracker and sequence it additively: preserve embedded mode, define a stable Gateway-to-runtime protocol with registration, health, capability, session, and delivery semantics, add local per-agent process lifecycle first, then layer container/Kubernetes backends and per-agent secret/resource isolation.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The current-main capability check is high confidence because docs, protocol roles, config help, and the registered agent CLI still describe embedded/local execution rather than separately managed per-agent runtimes.

## Solution Assessment

Unclear as a direct implementation path; the requested direction is plausible, but not yet a narrow best fix. The maintainable path is an explicit architecture decision and phased protocol/lifecycle design before any autonomous implementation PR.
