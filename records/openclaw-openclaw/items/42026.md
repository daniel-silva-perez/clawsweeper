## Summary

Keep this open. Current main still describes and implements OpenClaw as a single Gateway process with embedded/CLI/ACP agent-runtime selection, multi-agent routing, and node capability hosts, but not the RFC's Gateway control-plane plus separately registered per-agent compute runtimes. TinyTb's narrower Phase 0 request for per-agent start/stop/restart without restarting the Gateway also remains missing.

## Best Possible Solution

Keep this open for maintainer architecture review. The best path is an additive design that preserves embedded Gateway mode, defines a stable Gateway-to-agent-runtime protocol and runtime role/capability/health handshake, adds local per-agent process start/stop/restart, then layers container/Kubernetes backends plus per-agent secret and resource isolation while deliberately reusing existing runtime-plan, ACP, node transport, sandbox, config reload, and session routing pieces.

## Risks / Open Questions

- The RFC is broad and should likely be split into milestones, but closing it now would lose the unimplemented runtime protocol, per-agent lifecycle, health, and isolation work.
- Existing node transport, ACP runtime descriptors, runtime-plan, sandboxing, hot reload, and session routing are useful building blocks, but none is a drop-in substitute for the requested per-agent compute runtime boundary.
