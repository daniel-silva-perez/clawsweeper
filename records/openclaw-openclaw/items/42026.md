## Summary

Keep open. Current main has adjacent agent runtime harness selection, per-agent config/state, sandbox policy, and Gateway operator/node protocol roles, but it still dispatches agent turns through the Gateway-side embedded path and does not provide a Gateway-to-per-agent-runtime protocol or per-agent start/stop/restart lifecycle. The discussion also narrowed a useful Phase 0 slice, but that remains a broad runtime/protocol/security design item rather than a cleanup closure.

## What This Changes

RFC proposes splitting the Gateway into a lightweight control plane and separately managed per-agent runtimes for local-split and distributed deployment modes while preserving embedded mode.

## Best Possible Solution

Keep this as an architecture tracker and split it into additive, reviewable milestones: preserve embedded mode; define the Gateway-to-runtime contract for registration, health, capabilities, session ownership, turn delivery, cancellation, reconnect, and event delivery; implement local per-agent process lifecycle first; then add container/Kubernetes backends and per-agent secret/resource isolation after the threat model is explicit.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. The current-main capability gap is high-confidence because docs, CLI registration, protocol roles, and Gateway dispatch code all still show Gateway-side embedded execution without separately managed per-agent runtimes.

## Solution Assessment

Unclear as a direct implementation plan. The direction is plausible, but the best maintainable path is a phased maintainer-approved protocol and lifecycle design rather than a broad autonomous patch.
