## Summary

Keep open. Current main has adjacent runtime-selection, harness, sub-agent, node, and multi-agent routing machinery, but it still does not implement the requested control-plane plus separately registered per-agent compute runtime split. The Phase 0 lifecycle request from the discussion is also still missing: the CLI exposes agent config/routing commands, not per-agent runtime start/stop/restart without a gateway restart.

## What This Changes

The issue proposes splitting the monolithic gateway into a lightweight control plane and independently managed per-agent compute runtimes, with embedded mode retained for compatibility.

## Best Possible Solution

Keep this as an architecture tracker and sequence it additively: preserve embedded mode, define a stable gateway-to-runtime protocol with registration, health, capability, session, and delivery semantics, add local per-agent process lifecycle first, then layer container/Kubernetes backends and per-agent secret/resource isolation.
