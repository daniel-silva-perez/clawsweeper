## Summary

Keep open. Current main has adjacent agent runtime harness selection, per-agent state/config, sandbox policy, and Gateway node/operator protocol roles, but it still runs agent turns through the Gateway-side embedded dispatch path and does not expose a control-plane-to-per-agent-runtime protocol or per-agent start/stop/restart lifecycle. The issue discussion also narrowed a valuable Phase 0 slice, but that remains a cross-cutting runtime/protocol/security design item rather than cleanup closure.

## What This Changes

The issue requests splitting the Gateway into a lightweight control plane and separately managed per-agent runtimes with lifecycle, protocol, scaling, and isolation boundaries while preserving embedded mode.

## Best Possible Solution

Keep this as an architecture tracker and split it into additive, reviewable milestones: preserve embedded mode; define a Gateway-to-runtime contract for registration, health, capabilities, session ownership, turn delivery, cancellation, reconnect, and event delivery; implement local per-agent process lifecycle first; then add container/Kubernetes backends and per-agent secret/resource isolation after the threat model is explicit.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. The current-main capability gap is high-confidence because docs, CLI registration, protocol roles, and Gateway dispatch code all still show embedded Gateway-side execution without separately managed per-agent runtimes.

## Solution Assessment

Unclear as a direct implementation plan. The direction is plausible, but the best maintainable solution is a phased maintainer-approved protocol and lifecycle design rather than a broad autonomous patch.
