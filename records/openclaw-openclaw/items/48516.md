## Summary

Keep open. Current main has useful partial mitigation: Android notification forwarding is now disabled by default and has allowlist/blocklist controls. But the requested native-provider exclusion is not implemented: if forwarding is enabled in the default blocklist mode with no configured packages, all packages except OpenClaw are eligible, and the gateway still turns every valid `notifications.changed` payload into an untrusted system event plus heartbeat wake. The issue comments' dual-source routing and tuple-lock concerns remain relevant because there is still no native-channel package deny default, server-side package filter, or notification-origin outbound route lock.

## Best Possible Solution

Keep this issue open and implement a layered fix: Android notification forwarding should default to a narrow allowlist or ship native-channel deny defaults for WhatsApp, Telegram, Discord, Signal, and similar packages; the gateway should have a server-side `notifications.changed` package deny/allow policy before enqueueing and waking heartbeat; and message sends from node/notification-originated system events should be denied or route-locked unless a trusted originating channel/peer/session tuple exists.

## Risks / Open Questions

- Closing this now would treat off-by-default forwarding and manual package controls as equivalent to native-provider suppression, but users who enable broad notification forwarding can still forward WhatsApp notifications into a node session.
- Gateway-side handling still lacks a defense-in-depth package deny filter, so a future Android client or custom node can send the same problematic `notifications.changed` payload.
- The current outbound guard is not a session/channel/peer tuple lock for notification-originated system events; wrong-channel sends remain a plausible failure mode if the agent can call `message` with an explicit channel target.
