## Summary

Keep open. Current main still only partially mitigates the report: Android notification forwarding is disabled by default and configurable, but enabled blocklist mode still forwards all non-OpenClaw packages unless the operator manually blocks them. The gateway still accepts every valid `notifications.changed` payload as an untrusted system event and heartbeat wake, and the outbound guard still depends on having an originating channel/target context rather than enforcing a notification-origin channel/peer/session tuple lock. The Ryce and Kelisi808 comments about dual-source routing and tuple-lock guardrails remain relevant.

## Best Possible Solution

Keep this issue open and implement a layered fix: Android should default to a narrow allowlist or built-in deny defaults for native OpenClaw channel packages such as WhatsApp, Telegram, Discord, and Signal; the gateway should reject or ignore denied notification packages before enqueueing and waking heartbeat; and notification-originated runs should be denied or route-locked for outbound message sends unless a trusted originating channel/peer/session tuple exists. Add Android policy tests, gateway `notifications.changed` filter tests, and outbound route-lock coverage.

- A paired Android client or custom node can still send `notifications.changed` for native channel packages because the gateway has no server-side package deny/allow policy.
- The current outbound cross-context guard does not bind notification-origin system events to a trusted channel/peer/session tuple when no originating channel target exists.
