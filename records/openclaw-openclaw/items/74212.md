## Summary

Keep open. Current main and the latest tagged release still mask the OpenAI Codex device-pairing code in remote mode, while an open linked PR with closing syntax already targets this exact bug; this issue should remain open until that PR is reviewed, landed, or replaced.

## What This Changes

Fix OpenAI Codex device-code auth so `models auth login --provider openai-codex` surfaces the pairing code in SSH/headless terminals without writing it to persistent logs.

## Best Possible Solution

Land the narrow OpenAI plugin fix that prints `Code: <userCode>` in the remote prompter note, keeps persistent runtime logs code-free, and updates the existing provider unit test so remote mode requires terminal visibility and log hygiene.

## Reproduction Assessment

Yes. The source/test path is high-confidence: run the OpenAI Codex device-code auth method with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in the SSH-visible prompt.

## Solution Assessment

Yes. The narrowest maintainable solution is provider-local in `extensions/openai/openai-codex-provider.ts`, because core already supplies remote detection and the device-code helper already supplies `userCode`; #74258 matches that boundary better than the closed alternative that logged the code.
