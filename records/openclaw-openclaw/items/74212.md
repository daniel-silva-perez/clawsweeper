## Summary

Keep open. Current main and the latest tagged release still mask the OpenAI Codex device-pairing code in remote mode, and an open PR with closing syntax already targets this exact bug, so the issue should remain open until that PR is reviewed, landed, or replaced.

## What This Changes

The issue asks OpenClaw to surface the OpenAI Codex device-pairing code in the SSH-visible terminal flow for `models auth login --provider openai-codex` instead of rendering the placeholder text.

## Best Possible Solution

Review and land #74258, or an equivalent provider-local fix, so the remote device-code flow prints `Code: <userCode>` on the user-facing prompter note while keeping the persistent `runtime.log` output limited to the verification URL and locking both behaviors with OpenAI provider unit tests.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the existing provider unit-test setup with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in that SSH-visible note.

## Solution Assessment

Yes. The best fix is narrow and provider-local in `extensions/openai/openai-codex-provider.ts`, because the device-code helper already supplies `userCode` and core already supplies remote detection; #74258 matches that boundary while preserving log hygiene.
