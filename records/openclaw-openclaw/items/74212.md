## Summary

Keep this issue open. Current main still masks the available OpenAI Codex device-pairing code in remote mode, and open PR #74258 already uses closing syntax for the same bug, so the issue should remain open until the chosen fix lands.

## What This Changes

The issue reports that `models auth login --provider openai-codex` masks the OpenAI Codex device-pairing code in SSH/remote sessions, making the device-code auth flow unrecoverable from the terminal.

## Best Possible Solution

Review and land #74258 or an equivalent narrow OpenAI plugin fix that shows `Code: <userCode>` in the prompter note for remote/SSH sessions while keeping persistent runtime logs code-free, then update the regression test to require both behaviors.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: SSH env makes `isRemote` true, the provider receives a real `userCode`, and current source/test logic replaces it with the placeholder in the terminal-facing note.

## Solution Assessment

Yes. The best fix is the narrow provider-owned approach already described in #74258: surface the short-lived code on the user-facing TTY note and keep `runtime.log` limited to the verification URL.
