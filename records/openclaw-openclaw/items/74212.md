## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote mode, v2026.4.29 has the same behavior, and open PR #74258 already uses closing syntax for this exact bug.

## What This Changes

The issue requests that `openclaw models auth login --provider openai-codex` surface the OpenAI Codex device-pairing user code in SSH or remote sessions without writing it to persistent logs.

## Best Possible Solution

Review and land or replace the narrow OpenAI plugin fix from PR #74258: print `Code: <userCode>` through the prompter note in remote mode, keep `runtime.log` URL-only, and update provider tests to enforce both terminal visibility and log hygiene.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run the device-code auth method with `ctx.isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in the SSH-visible prompt.

## Solution Assessment

Yes. A provider-local change in `extensions/openai/openai-codex-provider.ts` is the narrowest maintainable fix because the helper already supplies `userCode`, and the CLI already passes remote/SSH state into provider auth.
