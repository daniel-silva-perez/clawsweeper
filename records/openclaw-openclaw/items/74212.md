## Summary

Keep this issue open. Current main still masks the OpenAI Codex device-pairing code in remote/SSH mode, the latest tagged release still has the same behavior, and an open PR with closing syntax is already tracking the narrower fix path.

## What This Changes

Request to make `openclaw models auth login --provider openai-codex` surface the OpenAI Codex device-pairing code in SSH/remote sessions instead of displaying a placeholder.

## Best Possible Solution

Land a narrow OpenAI plugin fix that prints `userCode` in the remote-mode prompter note, leaves persistent runtime logs code-free, and updates the provider unit tests to cover both invariants.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The current provider code path and existing remote-mode unit test show that an `isRemote: true` device-code flow receives `CODE-12345` but renders the placeholder instead of the code.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow plugin-local direction described by the open fix PR is the maintainable path: change only the prompter note output and preserve the runtime-log boundary.
