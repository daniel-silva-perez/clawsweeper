## Summary

Keep open. Current main and the latest tagged release still mask the OpenAI Codex device-pairing code in remote mode, and the issue is already paired with open PR #74258 using closing syntax, so this should stay open until that PR is reviewed, landed, or replaced.

## What This Changes

Surface the OpenAI Codex device-pairing user code in `models auth login --provider openai-codex` remote or SSH terminal output without writing it to persistent runtime logs.

## Best Possible Solution

Review and land #74258, or a narrow equivalent OpenAI plugin fix, so the remote prompter note prints `Code: <userCode>` while the persistent runtime log continues to show only the verification URL and tests lock in both behaviors.

## Reproduction Assessment

Yes. The source and unit-test path is high-confidence: invoke the OpenAI Codex device-code auth method with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and does not expose the code in the SSH-visible prompt.

## Solution Assessment

Yes. A provider-local change is the narrowest maintainable fix because core already passes remote detection and the device-code helper already supplies `userCode`; #74258 matches that boundary better than the closed alternative that also wrote the code to `runtime.log`.
