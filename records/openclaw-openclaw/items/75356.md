## Summary

Keep open. The linked bug is still real on current main, and this PR is the right general implementation candidate, but the patch is incomplete because it retargets `chat.send` to the base session while leaving the Control UI's active session state on the heartbeat row; base-session chat events will then be ignored by the existing event filter.

## What This Changes

This PR projects isolated-heartbeat base-session metadata through Gateway session rows, hides marker-backed heartbeat rows from the Control UI session picker, retargets stale heartbeat `chat.send` requests to the base session in UI and Gateway code, and adds focused UI regression tests.

## Best Possible Solution

Land a corrected version that keeps the marker-backed heartbeat contract, hides isolated heartbeat rows from normal chat selection, and normalizes the Control UI's active chat target consistently for send, stream/final event handling, abort, history reload, and session overrides. Add a changelog Fixes entry and focused tests that prove a stale heartbeat selection both sends to and receives completion from the base session.

## Reproduction Assessment

Yes. Seed `sessionsResult` with only `agent:main:main:heartbeat` carrying `heartbeatIsolatedBaseSessionKey: "agent:main:main"`, send a message, then deliver a `chat` event for `agent:main:main`; code inspection shows the PR would send to the base session while `handleChatEvent` still filters on the stale heartbeat `state.sessionKey`.

## Solution Assessment

No. The marker-backed routing direction is right, but the current implementation only rewrites the outbound `chat.send` key and does not keep the rest of the UI chat state on the same resolved session; the safer fix is to resolve the active session once and use that same key for send, event acceptance, abort, history, and session controls.
