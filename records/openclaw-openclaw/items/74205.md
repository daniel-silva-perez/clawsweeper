## Summary

Keep open. Current main improves the gateway-status probe classification for post-connect read timeouts, but the core reported failure remains unproven: the bundled Kimi Coding provider still exposes `kimi-code` and legacy `k2p5`, docs steer K2.6 users to `moonshot/kimi-k2.6`, and the embedded runner still classifies the reported terminal payload as an incomplete format result. I found no current-main source, test, or related PR proving an exact `kimi/k2p6` agent turn now succeeds.

## Best Possible Solution

Keep this issue open. A maintainer should validate the exact `kimi/k2p6` embedded-runner path with a live Kimi credential or faithful fixture, then either add explicit supported-model/alias handling with Kimi provider tests or reject unsupported `kimi/k2p6` with a clearer diagnostic and docs guidance toward `kimi/kimi-code` or `moonshot/kimi-k2.6`. Treat the gateway-status degraded-probe improvement as a separate partial fix until it ships.
