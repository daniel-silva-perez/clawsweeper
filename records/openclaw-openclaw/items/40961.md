## Summary

Keep this PR open. Current main still has the `sessions_send` A2A wait capped at 60 seconds, so the core problem is not fully implemented on main, but the branch is stale against the current 120-second subagent announce default and removes the cap without adding an explicit bounded timeout policy.

## What This Changes

The PR raises the subagent announce default/test expectation from 60 seconds to 180 seconds and removes the hard 60-second cap from the `sessions_send` A2A wait phase.

## Best Possible Solution

Rework the A2A timeout handling on current main: keep or intentionally revise the 120-second subagent announce default, add an explicit bounded/configurable policy for `sessions_send.timeoutSeconds`, remove inappropriate 60-second caps consistently across the A2A wait and nested announce/reply steps, and cover waits above 60 seconds plus timeout background-announce behavior with focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: This line changes the wait to use `params.announceTimeoutMs` directly, but `sessions_send.timeoutSeconds` is only lower-bounded and execution accepts any finite value. A caller can therefore request a very long wait and keep the A2A flow/RPC pending far beyond the intended budget; add an explicit maximum or shared clamp before honoring values above 60 seconds.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR does not touch supply-chain or secret-handling surfaces, but it removes a runtime wait cap without replacing it with a bounded timeout policy.

Concerns:

- **[medium] Unbounded A2A wait duration:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: After the cap removal, `sessions_send.timeoutSeconds` can drive `agent.wait` for very long durations because the schema only requires a nonnegative number and Gateway timers clamp only to the platform-safe maximum. This is a resource-use/availability risk for model-initiated tool calls and should be bounded deliberately.
  - confidence: 0.82
