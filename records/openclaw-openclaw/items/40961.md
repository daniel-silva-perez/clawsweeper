## Summary

Keep open, but do not merge as-is. The PR is aimed at a real A2A timeout problem, yet current main has already moved the subagent announce default to 120s while the proposed cap removal would let `sessions_send.timeoutSeconds` drive an unbounded gateway wait; related remaining behavior is better handled as a current-main timeout-policy rework.

## What This Changes

The PR raises the subagent announce default/test expectation to 180 seconds and removes the 60-second cap from the initial `sessions_send` A2A wait phase.

## Best Possible Solution

Define one bounded A2A timeout policy on current main, enforce it through a shared helper or schema maximum, apply it consistently to the initial wait and nested reply/announce waits, and add focused coverage for waits above 60 seconds plus timeout-triggered background announce behavior.

## Reproduction Assessment

Yes for the PR-introduced defect: by inspection, a large `sessions_send.timeoutSeconds` value is schema-valid, becomes `announceTimeoutMs`, and this PR would pass it directly into the initial `agent.wait`. I did not run tests because the review was explicitly read-only.

## Solution Assessment

No. The patch is directionally related to #40863, but current main already uses a 120s subagent announce default, and removing the first wait cap without a replacement bound leaves inconsistent and potentially excessive A2A waits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: This change lets `sessions_send.timeoutSeconds` drive the initial `agent.wait` duration without an upper bound. The schema accepts any nonnegative finite number and `waitForAgentRun` forwards the duration to the gateway wait, so large model-supplied values can hold the A2A flow for an excessive time. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
