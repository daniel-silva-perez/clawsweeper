## Summary

Keep open, but do not merge as-is. The PR still targets a real A2A timeout problem, but current main has moved the adjacent subagent announce default to 120 seconds, still caps the first A2A wait at 60 seconds, and this patch would remove that cap without adding a replacement bound or aligning the nested A2A waits.

## What This Changes

The PR raises the subagent announce default/test expectation to 180 seconds and changes the first `sessions_send` A2A wait to use the configured announce timeout instead of a hard 60-second cap.

## Best Possible Solution

Define one bounded current-main A2A timeout policy, enforce it through a shared helper or schema maximum, apply it consistently to the initial wait plus nested reply/announce waits, and add focused regression tests for timeout values above 60 seconds and timeout-triggered background announce behavior.

## Reproduction Assessment

Yes. The PR-introduced problem is reproducible by inspection: a large `sessions_send.timeoutSeconds` value is schema-valid, becomes `announceTimeoutMs`, and the PR diff would pass that duration directly into the initial `agent.wait`.

## Solution Assessment

No. Removing the 60-second cap is directionally related to the false-timeout report, but this patch does not add a replacement bound and leaves the rest of the A2A wait flow on different timeout semantics.
