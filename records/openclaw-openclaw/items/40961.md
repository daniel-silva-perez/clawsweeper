## Summary

Keep open, but do not merge as-is. Current main has not implemented the PR's cap-removal behavior, and the branch is stale against the current announce-timeout implementation while removing the only A2A wait cap without a replacement bound.

## What This Changes

The PR raises the subagent announce default and matching test expectation to 180 seconds, and changes the first `sessions_send` A2A wait to use the configured announce timeout instead of a 60-second cap.

## Best Possible Solution

Define one current-main A2A timeout policy: choose the intended default and maximum, clamp `sessions_send.timeoutSeconds` through a shared helper or schema rule, align the initial wait, nested reply/announce waits, and timeout-triggered background announce behavior, then cover those cases with focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: The changed line lets `sessions_send.timeoutSeconds` drive the initial `agent.wait` with no upper bound. The schema only requires a nonnegative number and the wait helper forwards that duration, so large tool-supplied values can hold the A2A flow for a very long time. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
  - confidence: 0.9
