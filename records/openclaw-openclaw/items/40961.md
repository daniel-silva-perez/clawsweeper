## Summary

Keep open. Current main still does not implement the PR's intended A2A cap removal, and the proposed branch is not safe to merge as-is because it makes the initial A2A wait unbounded while related waits and timeout behavior remain inconsistent.

## What This Changes

The PR raises the subagent announce default/test expectation to 180 seconds and changes the first `sessions_send` A2A wait to use the configured announce timeout instead of a 60-second cap.

## Best Possible Solution

Define a single bounded current-main A2A timeout policy, enforce it through a shared helper or schema maximum, make the initial wait, nested reply/announce waits, and timeout-triggered background announce behavior consistent, and then update focused regression tests around those paths.

## Reproduction Assessment

Yes. The PR-introduced problem is reproducible by inspection: a large `sessions_send.timeoutSeconds` value is schema-valid, converted to milliseconds, and the PR diff passes that duration directly into the initial `agent.wait`.

## Solution Assessment

No. Removing the 60-second cap is related to the false-timeout report, but the maintainable fix needs an explicit upper bound and consistency across all A2A wait phases rather than making only the first wait unbounded.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
