## Summary

Keep open. Current main has not implemented the PR's intended cap removal, and the proposed branch is not safe to merge as-is because it removes the first A2A wait cap without adding a replacement maximum while other A2A waits remain capped.

## What This Changes

The PR raises the subagent announce default/test expectation to 180 seconds and changes the first `sessions_send` A2A wait to use the configured announce timeout instead of a 60-second cap.

## Best Possible Solution

Define one current-main A2A timeout policy with an explicit maximum, enforce it through a shared helper or schema rule, align the first wait, nested reply/announce waits, and timeout-triggered background announce behavior, then update focused regression tests around those cases.

## Reproduction Assessment

Yes. The issue introduced by the PR is reproducible by inspection: a large `sessions_send.timeoutSeconds` value is schema-valid, is converted to milliseconds, and the PR diff passes that value directly into the initial `agent.wait`.

## Solution Assessment

No. Removing the 60s cap is directionally related to the false-timeout report, but the best fix needs a bounded policy and consistency across all A2A waits rather than making only the first wait unbounded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: This line lets `sessions_send.timeoutSeconds` drive the initial `agent.wait` without an upper bound.
