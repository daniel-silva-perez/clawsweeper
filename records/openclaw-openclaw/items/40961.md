## Summary

Keep this PR open but do not merge it as-is. Current main partially addresses the original false-timeout complaint by shipping a 120s default announce timeout, but the PR is stale against that implementation and removes the A2A wait cap without replacing it with a bounded timeout policy.

## What This Changes

The PR raises the subagent announce default/test expectation to 180 seconds and changes the initial `sessions_send` A2A wait to use the configured announce timeout instead of the existing 60-second cap.

## Best Possible Solution

Define one bounded A2A timeout policy on current main, enforce it through a shared helper or schema maximum, apply it consistently to the initial wait plus nested reply/announce waits, and add focused regression coverage for values above 60 seconds and timeout-triggered background announce behavior.

## Reproduction Assessment

Yes for the PR-introduced issue: by inspection, a large `sessions_send.timeoutSeconds` value is schema-valid, becomes `announceTimeoutMs`, and this PR would pass it directly into the initial `agent.wait`. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No. Removing the 60-second cap is directionally related to the false-timeout report, but this patch does not add a replacement bound and leaves nested A2A waits on different timeout semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: This change lets `sessions_send.timeoutSeconds` drive the initial `agent.wait` duration without an upper bound. The schema only requires a nonnegative number and the wait helper forwards the duration, so large tool-supplied values can hold the A2A flow for a very long time. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
