## Summary

Keep this PR open. Current main has partially addressed the surrounding subagent announce timeout area, but the PR’s central sessions_send A2A cap-removal behavior is still not implemented, and the old patch is stale against the current 120s default and needs a bounded timeout policy before it would be merge-ready.

## Best Possible Solution

Keep this PR open for maintainer review, but rework it against current main: preserve or explicitly change the shipped 120s subagent announce default, add a bounded/configurable timeout policy for sessions_send A2A, remove inappropriate 60s caps consistently, and add focused regression coverage for waits above 60s plus the timeoutSeconds: 0 A2A path.

## Risks / Open Questions

- Rebasing the old diff as-is would remove a defensive 60s upper bound while sessions_send.timeoutSeconds has no maximum, creating a resource/DoS hardening concern for accidental or malicious long waits.
- Changing the current shipped 120s subagent announce default to 180s is a product/runtime behavior decision, not an automatic stale-PR cleanup decision.
- The current sessions_send path also has a 30s fire-and-forget announce budget and adjacent nested-step 60s cap, so a narrow one-line cap removal may leave inconsistent A2A timeout behavior.
