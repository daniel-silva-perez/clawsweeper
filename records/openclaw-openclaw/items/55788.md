## Summary

Keep this PR open. Current main still lacks the requested OpenAI Codex audio media-understanding support, but the PR is not ready to merge because it needs refresh against the current metadata-driven media defaults and must preserve explicit audio endpoint overrides.

## What This Changes

The PR adds `openai-codex` audio capability, a transcription handler wrapper, a default audio model entry, and a test assertion for the OpenAI media-understanding provider.

## Best Possible Solution

Refresh the PR against current main so `openai-codex` audio is represented consistently in the runtime provider, plugin manifest metadata, default model metadata, auto-selection priority, tests, and docs, while only avoiding ChatGPT backend URLs for transcription rather than discarding all explicit endpoint overrides.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve explicit codex audio endpoints:** `extensions/openai/media-understanding-provider.ts:29`
  - body: `transcribeOpenAiCodexAudio` overwrites `baseUrl` with `undefined` for every request. The runner already passes entry/config/provider base URLs into `transcribeAudio`, and the shared OpenAI-compatible helper honors them; dropping the value forces audio to `api.openai.com` even when an operator configured an OpenAI-compatible relay or proxy. Strip only the ChatGPT backend URL case.
  - confidence: 0.9
- **[P2] Wire codex into audio metadata and auto selection:** `extensions/openai/media-understanding-provider.ts:36`
  - body: Adding `"audio"` to the runtime provider is not enough on current main. Default model and auto-provider selection now come from provider `defaultModels`/`autoPriority` and manifest metadata, where `openai-codex` is still image-only, so codex-authenticated users without explicit `tools.media.audio.models` still will not get the intended automatic transcription path.
  - confidence: 0.84
