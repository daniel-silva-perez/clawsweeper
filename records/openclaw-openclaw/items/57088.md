## Summary

Keep this PR open. Current main still has the unbounded cron run-log read problem, and this PR is aimed at the right surface, but the current head is dirty against main and still has blocking correctness issues in the proposed read-side prune implementation.

## What This Changes

The PR adds read-side cron run-log pruning before pagination, threads runtime prune limits through the gateway request context, updates related tests, and adjusts one pi embedded runner test harness mock.

## Best Possible Solution

Rebase or repair the PR on current main, then make cron run-log reads bounded even when read-side pruning fails. The prune/read helper should avoid fixed per-line byte assumptions, keep runtime-configured limits, stay serialized with appends, and add regression coverage for oversized prune failures and long JSONL entries.

## Reproduction Assessment

Yes. Current main can be reproduced by creating an oversized cron run JSONL file and calling `readCronRunLogEntriesPage` or `cron.runs`; the source shows the unbounded `fs.readFile` path. The PR-specific failures are reproducible with an oversized file plus prune failure, and with recent entries longer than the fixed 2KB estimate.

## Solution Assessment

No. The PR direction is good, but the current implementation is not the best fix because failed read-side pruning still falls through to an unbounded full-file read, and the streaming prune uses a fixed line-size estimate that can drop recent history.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound reads when prune fails:** `src/cron/run-log.ts:423-427`
  - body: `serializedPruneIfNeeded(...).catch(() => undefined)` makes the defensive prune best-effort, but if it fails on an already-oversized log, the next line still reads the unpruned file with `fs.readFile`. That leaves the OOM path open under the disk-pressure or permission scenario this PR is meant to fix; fall back to a bounded tail read or bounded error result when prune cannot shrink the file.
  - confidence: 0.89
- **[P2] Remove the fixed per-line tail window:** `src/cron/run-log.ts:166`
  - body: The streaming prune only looks back `keepLines * 2048` bytes, so recent JSONL entries larger than 2KB can make the helper see fewer than `keepLines` complete records and rewrite away valid history. Expand until enough complete lines are found, or use a bounded algorithm that does not depend on a fixed per-line size.
  - confidence: 0.86
