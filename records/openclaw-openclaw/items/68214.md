## Summary

Keep open. Current main appears to have fixed the WhatsApp allowlist privacy leak path by making pairing replies happen only for the shared `dmPolicy="pairing"` decision and silently blocking non-allowlisted `allowlist` DMs, with tests for that behavior. But the actual requested operator surface is still missing: there is no documented `message:unauthorized` / `access:denied` internal hook, no plugin hook for access denial that can suppress default outbound behavior, and no per-channel `suppressPairingReply` config flag. The only `suppressPairingReply` in current source is an internal historical-message grace check, not the requested config knob.

## Best Possible Solution

Keep the issue open and narrow it to the still-missing access-denial extension point. The best product direction is a generic, documented channel access-denial hook or plugin SDK seam that reports channel/account/sender/deny reason/default action and can suppress or replace default outbound behavior. After that design lands, maintainers can decide whether a separate `suppressPairingReply` config flag is still needed, since current main already makes pairing replies opt-in to `dmPolicy: "pairing"` for the WhatsApp path.
