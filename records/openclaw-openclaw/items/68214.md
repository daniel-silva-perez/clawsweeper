## Summary

Keep open. Current main appears to have fixed the reported WhatsApp allowlist pairing-reply leak, but the requested unauthorized-inbound hook and suppress-default policy surface still do not exist; related #42438 and PR #39065 cover adjacent response posture, not this access-denial hook seam.

## What This Changes

Request a generic unauthorized-inbound hook that can suppress default replies plus a per-channel `suppressPairingReply` flag for unauthorized direct messages.

## Best Possible Solution

Design a generic, privacy-reviewed channel access-denial extension point that reports minimal channel/account/deny-reason/default-action metadata and can suppress or replace the default outbound behavior. Keep pairing replies opt-in to pairing-mode decisions, and coordinate any simple config knob with the broader silent/notify response work.

## Reproduction Assessment

Yes for the remaining feature gap: current docs, hook types, and config schema directly show no unauthorized/access-denied hook or suppress-default config. No live reproduction was performed for the original 2026.4.12 WhatsApp incident, but current source and tests show the allowlist pairing leak itself is no longer present.

## Solution Assessment

No, current main is not a complete solution to this issue. A WhatsApp-only `suppressPairingReply` flag would be narrower than the operator-policy problem; the more maintainable path is a generic access-denial hook/SDK seam with privacy-minimized payloads and explicit suppression semantics.
