## Summary

Keep open. Current main fixes the reported WhatsApp allowlist pairing-reply leak, but the requested access-denial hook/suppression surface is still absent; related #42438 and open PR #39065 cover adjacent silent/unpaired response configuration, not the requested unauthorized-inbound hook seam.

## What This Changes

The issue requests an unauthorized-inbound hook that reports deny context and can suppress default outbound handling, plus a per-channel suppressPairingReply option for unauthorized DMs.

## Best Possible Solution

Keep the issue focused on a generic channel access-denial extension point: a documented internal or plugin SDK hook that reports channel/account/sender/deny reason/default action with a privacy-reviewed payload and can suppress or replace the default outbound behavior. After that seam is designed, maintainers can decide whether #42438 or PR #39065 should cover the remaining config-only response posture.

## Reproduction Assessment

Partially. Current main gives a high-confidence source/test path for the former WhatsApp allowlist leak being fixed, and the missing hook/config surface is reproducible by direct docs/types/source inspection; the original 2026.4.12 live WhatsApp incident was not rerun in this read-only review.

## Solution Assessment

No, current main is not a complete solution to this issue. The narrow maintainable direction is a generic access-denial hook/SDK seam with privacy controls, while a WhatsApp-only `suppressPairingReply` flag would leave the broader operator policy problem unresolved.
