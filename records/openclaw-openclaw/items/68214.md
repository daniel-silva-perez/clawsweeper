## Summary

Keep open. Current main appears to have fixed the WhatsApp allowlist privacy leak path by making pairing replies happen only for the shared `dmPolicy="pairing"` decision, with tests for silent allowlist blocking. The requested operator surface is still missing: no `message:unauthorized` or `access:denied` internal hook, no plugin access-denial hook that can suppress or replace default outbound behavior, and no `suppressPairingReply` config flag.

## Best Possible Solution

Keep the issue open and narrow it to a generic, documented channel access-denial extension point. The best path is a core/Plugin SDK seam that fires before any default pairing/deny outbound, reports channel, account, sender identifier or hash, deny reason, and default action, and can suppress or replace that default action. Once that lands, maintainers can decide whether a separate `suppressPairingReply` config flag is still useful or whether pairing-only default behavior is enough.
