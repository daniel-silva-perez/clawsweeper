## Summary

Keep open. The user-visible gap still exists on current main, but this PR is not safe to merge as-is because its fallback submits a new gateway agent turn and delivers that turn's generated payload rather than a guaranteed raw recovered notice. The current direct outbound delivery seam is a better replacement path.

## What This Changes

The PR adds `notifyAnnounceGiveUp` in `src/agents/subagent-registry.ts` and calls it from resumed and live announce retry-exhaustion paths to send recovered result text through gateway `agent` delivery.

## Best Possible Solution

Implement the give-up notice in the current lifecycle and announce-delivery layers using direct outbound `sendMessage` delivery when a deliverable requester origin exists. Cover resumed and live retry exhaustion, missing origin, killed-run suppression, per-run idempotency, and failed last-resort delivery logging.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: exhaust announce retries so `resolveDeferredCleanupDecision` returns `give-up` or resume a run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT`; current main logs give-up and completes cleanup without a final requester send.

## Solution Assessment

No. The requested fallback notification is the right direction, but this PR's gateway-agent approach is not the narrowest maintainable fix because it creates another model/session turn; direct outbound delivery through the existing announce-delivery seam is safer.
