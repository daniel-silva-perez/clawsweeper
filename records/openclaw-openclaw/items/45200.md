## Summary

Keep open. Current main still has a user-visible gap at subagent announce retry exhaustion, but this PR should not merge as-is because its fallback sends the recovered notice through a new gateway agent/model turn instead of a direct outbound delivery path, and the branch predates the current lifecycle/announce-delivery split.

## What This Changes

The PR adds a best-effort `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` and invokes it from announce retry-exhaustion paths to send recovered subagent output through gateway `agent` delivery.

## Best Possible Solution

Implement the final give-up notice in the current lifecycle and announce-delivery layers using direct outbound `sendMessage` delivery when a normalized requester origin has a deliverable channel and target. The end state should cover live and resumed retry exhaustion, killed-run suppression, missing origin, per-run idempotency, failed last-resort logging, and no additional requester model turn.

## Reproduction Assessment

Yes. Exhaust announce retries until `resolveDeferredCleanupDecision` returns `give-up`, or resume a run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT`; current main logs give-up and completes cleanup without a final requester send.

## Solution Assessment

No. The fallback notification is the right direction, but the PR's gateway-agent approach is not the narrowest maintainable fix because it turns the recovered notice into another model/session run instead of a direct outbound channel send.
