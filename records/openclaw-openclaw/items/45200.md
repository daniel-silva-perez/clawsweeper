## Summary

Keep open. The reported silent-loss path is still a valid current-main concern, but this PR should not merge as-is because its last-resort notice goes through the gateway `agent` path, which starts another requester agent/model run instead of directly sending the recovered notice to the user's channel.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` and invokes it from live and resumed announce retry-exhaustion paths to send recovered subagent output through gateway `agent` delivery.

## Best Possible Solution

Implement the give-up notice in the current lifecycle and announce-delivery layers using direct outbound `sendMessage` delivery when the normalized requester origin has a deliverable channel and target. The final behavior should avoid an extra requester model turn, use per-run idempotency, skip intentionally killed runs, preserve cleanup semantics, log failed last-resort sends, and cover both live and resumed retry exhaustion.

## Reproduction Assessment

Yes. Force `runSubagentAnnounceFlow` to fail until `resolveDeferredCleanupDecision` returns `give-up`, or resume a run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT`; current main then logs give-up and completes cleanup without a final requester send.

## Solution Assessment

No. The fallback-notification direction is right, but this PR's gateway `agent` approach is not the narrowest maintainable fix because it turns the recovered notice into another requester agent/model run instead of directly sending the recovered text through the outbound delivery seam.
