## Summary

Keep open. The underlying bug still appears valid on current main, but this PR should not merge as-is because its fallback notice goes through the gateway `agent` model/session path rather than direct outbound delivery, and the affected code has since moved into the split subagent lifecycle and announce-delivery modules.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` and invokes it from announce retry-exhaustion paths to send recovered subagent result text through gateway `agent` delivery.

## Best Possible Solution

Implement the final give-up notice in the current lifecycle and announce-delivery layers using direct outbound `sendMessage` delivery when a deliverable requester origin exists. It should cover live and resumed retry exhaustion, missing origin, killed-run suppression, per-run idempotency, failed last-resort delivery logging, and avoid creating a new requester model turn.

## Reproduction Assessment

Yes. Exhaust announce retries until `resolveDeferredCleanupDecision` returns `give-up`, or resume a run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT`; current main logs give-up and completes cleanup without a final requester send.

## Solution Assessment

No. The requested fallback notification is the right direction, but this PR's gateway-agent approach is not the narrowest maintainable fix because it turns the notice into another model/session run; direct outbound delivery through the existing announce-delivery seam is safer.
