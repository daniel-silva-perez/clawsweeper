## Summary

Keep open. Current main still has no final requester notification when subagent announce retry exhaustion reaches give-up, but this PR should not merge as-is because its fallback calls the gateway `agent` method and starts another requester agent turn instead of directly sending the recovered notice.

## What This Changes

The PR adds `notifyAnnounceGiveUp` in `src/agents/subagent-registry.ts` and invokes it from retry-exhaustion paths to send a best-effort requester notification with recovered frozen result text.

## Best Possible Solution

Implement the give-up notice in the current lifecycle and announce-delivery layers with direct outbound `sendMessage` delivery when a deliverable requester origin exists. Cover resumed and live retry exhaustion, missing origin, killed-run suppression, idempotency, and failed last-resort delivery logging.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence path: when `announceRetryCount` reaches `MAX_ANNOUNCE_RETRY_COUNT` or the live cleanup decision returns `give-up`, current main records failed delivery and logs give-up without a final send; the PR finding reproduces by tracing its `method: "agent"` call through gateway agent dispatch and delivery.

## Solution Assessment

No. The requested user-visible fallback is the right product direction, but this PR’s gateway-agent approach is not the narrowest maintainable fix because it creates another model/session turn; the safer fix is a direct outbound send in the existing announce-delivery seam.
