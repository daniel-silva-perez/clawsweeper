## Summary

Keep this PR open. Current main still has the reported exec allowlist behavior, while the PR remains a useful active implementation candidate but needs maintainer/security review before merge because the new unsupported-syntax denial still appears to be bypassed by durable exact-command approvals.

## What This Changes

The PR splices POSIX backslash-newline continuations before exec allowlist analysis, adds an `unsupported-shell-syntax` denial reason through gateway and node-host paths, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land an adapted version that normalizes POSIX backslash-newline before allowlist analysis with quote/heredoc/comment-aware parsing, reserves `allowlist-miss` for successful analysis with a non-allowlisted target, hardens or explicitly documents durable-approval behavior for unparseable commands, and updates tests plus the attributed changelog entry together.

## Reproduction Assessment

Yes. The current-main reproduction is high-confidence from source and tests: a command containing backslash-newline reaches `hasShellLineContinuation()` and returns `analysisOk=false`, and the existing test asserts that failure path.

## Solution Assessment

No, not as currently proposed. The parser direction and distinct denial signal are the right narrow fix, but the PR should move unsupported-syntax denial ahead of durable approval or otherwise make the durable override an explicit maintainer-approved policy with tests.
