## Summary

Keep this PR open. Current main still has the reported exec allowlist behavior, and this PR is the active implementation candidate, but the head diff still leaves unparseable shell syntax bypassable by durable approval on both gateway and node-host paths and its changelog entry is missing required contributor credit.

## What This Changes

The PR normalizes POSIX backslash-newline continuations before exec allowlist analysis, adds an `unsupported-shell-syntax` denial signal across gateway/node-host/user messaging, updates regression tests, and adds a changelog entry.

## Best Possible Solution

Land an adapted version that splices POSIX backslash-newline continuations with quote/heredoc/comment-aware semantics, reserves `allowlist-miss` for successfully analyzed non-allowlisted commands, hard-denies or explicitly maintainer-documents unparseable syntax before durable approval reuse, and ships matching gateway, node-host, user-message, test, and changelog updates together.

## Reproduction Assessment

Yes. Current main has a high-confidence source/test reproduction: a command containing backslash-newline reaches `hasShellLineContinuation()` and returns `analysisOk=false`, and the existing test asserts that failure path. The PR blocker is also source-reproducible by setting `analysisOk=false` with durable approval satisfied in the gateway/node policy paths.

## Solution Assessment

No, not as currently proposed. The parser and distinct denial direction are the right narrow fix, but unsupported shell syntax needs to be denied before durable trust is considered, or the durable override needs explicit maintainer-approved policy and tests.
