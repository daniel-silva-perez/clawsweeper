## Summary

Current main still lacks a default BOOT.md startup delivery route. The boot hook runs BOOT.md, but it invokes the agent with `deliver:false` and no channel, target, `to`, or run context, while the gateway startup hook context only carries config, deps, and workspace. Keep this open for a maintainer decision on the startup delivery contract.

## What This Changes

BOOT.md startup runs should be able to send user-visible startup messages, or the docs/config should clearly require an explicit delivery destination.

## Best Possible Solution

Define one explicit BOOT.md startup contract: either provide a configured/default delivery channel and target through the boot run and message-tool context, or update the template/docs/validation to require explicit channel plus target for BOOT.md sends. Preserve NO_REPLY as final assistant reply suppression and add a regression test for the chosen gateway-startup behavior.
