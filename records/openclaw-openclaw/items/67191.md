## Summary

Current main still does not provide the default BOOT.md startup delivery route reported here. The boot-md hook runs BOOT.md, but runBootOnce still invokes the agent with deliver:false and no channel, target, to, replyTo, or runContext. Current code now prompts the agent to use channel + target for message-tool sends, so a hard-coded destination may work, but the documented template and reported Telegram startup-status workflow remain unresolved.

## Best Possible Solution

Keep this issue open. Maintainers should define the BOOT.md delivery contract: either pass a well-defined startup delivery route into the boot run and message-tool context, or update the docs and validation to require explicit channel and target for BOOT.md sends. Preserve NO_REPLY as final assistant reply suppression and add a regression test for the chosen gateway-startup BOOT.md message-tool behavior.

## Risks / Open Questions

- A BOOT.md file that hard-codes a valid channel and target may be able to send today, so the remaining product decision is whether BOOT.md should provide a default startup delivery route or explicitly document and validate hard-coded destinations.
- Changing only deliver:false to deliver:true could create unsolicited startup deliveries or duplicate final replies unless the startup delivery contract, target resolution, and NO_REPLY suppression behavior are defined together.
