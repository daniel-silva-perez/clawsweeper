## Summary

Current main still does not implement a default user-visible BOOT.md startup delivery route. The boot-md hook runs BOOT.md per agent, but it invokes the agent with `deliver: false` and does not pass a channel, target, reply target, `to`, or run context that the message tool can use to infer the requester chat. A BOOT.md that hard-codes a valid channel and target may work, but the reported Telegram startup-status workflow and the documented template remain unresolved.

## Best Possible Solution

Keep this issue open. Maintainers should define the BOOT.md delivery contract: either pass a well-defined startup delivery route into the boot run and message-tool context, or update docs and validation to require explicit channel and target in BOOT.md. Add a regression test for the chosen gateway-startup BOOT.md message-tool behavior while preserving `NO_REPLY` as final assistant reply suppression.

## Risks / Open Questions

- A BOOT.md file with a hard-coded valid channel and target may be able to send today, so the remaining problem is the default startup delivery contract rather than total message-tool inability.
- Changing only `deliver: false` to `deliver: true` could create unsolicited startup replies or duplicate final replies unless maintainers define target resolution and `NO_REPLY` suppression together.
