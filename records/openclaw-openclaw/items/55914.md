## Summary

Keep this issue open. Current main still supports QR/setup-code pairing and the bundled `/pair` setup-code flow, but it does not implement the requested `openclaw invite` command, short `CLAW-xxxx` invite codes, cloud invite register/resolve endpoints, `gateway.cloud` config, or mobile invite-code resolver. The related Cloud Relay request #55915 is still open, and the comments add demand signal around TestFlight/code-based onboarding rather than evidence that the feature shipped.

## Best Possible Solution

Keep this open for maintainer product/security review. The strongest path is to design `openclaw invite` together with #55915, reusing the current bootstrap-token and `openclaw devices approve` flow where possible, or have a maintainer explicitly decide that documented `openclaw qr --setup-code-only`, `/pair`, and direct/Tailscale pairing are the supported non-QR alternatives.

## Risks / Open Questions

- Cloud-backed short invite codes need explicit product and security design for expiry, collisions, rate limiting, backend trust, privacy, and how bootstrap tokens are exposed or avoided.
- The current setup-code flow helps copy/paste pairing, but it still does not provide the short phone-dictated cloud discovery UX requested here.
- The TestFlight/code-access comments suggest user confusion around invite-code onboarding that may need maintainer support or product clarification independent of the core CLI implementation.
