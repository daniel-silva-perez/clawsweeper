## Summary

Keep open. Current main has QR/setup-code and `/pair` setup-code paths for mobile pairing, but it still does not implement the requested `openclaw invite` command, human-dictated `CLAW-xxxx` codes, cloud invite register/resolve endpoints, `gateway.cloud` config, or a mobile invite resolver. The linked cloud relay feature in #55915 remains open, and the comments add useful demand signal around TestFlight/code-based onboarding rather than evidence that this is already shipped.

## Best Possible Solution

Keep this open for maintainer product/security review. The strongest path is to design `openclaw invite` together with #55915, reusing the current device bootstrap and `openclaw devices approve` flow where possible, or have a maintainer explicitly decide that the documented `openclaw qr --setup-code-only` and `/pair` setup-code paths are the supported non-QR alternative and close the feature request on product-scope grounds.

## Risks / Open Questions

- Cloud-backed short invite codes need explicit product and security design for expiry, collision handling, rate limiting, backend trust, privacy, and how bootstrap tokens are exposed or avoided.
- The existing setup-code flow solves copy/paste pairing but not the short phone-dictated cloud discovery UX requested here, so closing as implemented would hide a real remaining capability gap.
- The TestFlight/comment thread suggests some user confusion around invite-code access that may need maintainer support or product clarification independent of the core CLI implementation.
