## Summary

Keep open. Current main solves the Telegram album side of the request, but LINE still downloads and dispatches each media webhook event as its own agent turn, with no LINE media batching config or docs surface.

## What This Changes

Request optional media batching so rapid LINE images and Telegram album photos can be forwarded to the agent as one turn instead of one turn per image.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: add an optional media batching delay, buffer downloadable LINE media by account plus conversation plus sender, flush on timer and before same-key text or non-media events, aggregate media into one LineInboundContext, and leave existing Telegram album buffering plus shared text-only debounce semantics unchanged.

## Repair Work Prompt

Fix the remaining LINE side of this issue. Implement optional media batching inside the LINE plugin boundary, not in generic core gateway code. Add an account-level/top-level LINE config option such as `channels.line.mediaBatchDelayMs` / `channels.line.accounts.<id>.mediaBatchDelayMs` with a safe default of disabled unless maintainers prefer a small default. Buffer downloadable LINE media events by account id plus conversation id plus sender id. If another media event with the same key arrives before the timer, aggregate it. If same-key text arrives before the timer, build one `LineInboundContext` from the text event plus the pending media so the agent gets the user's follow-up text and all media in one turn. If the timer expires with only media, emit one media-only context using a stable representative event. Preserve current access checks, group mention behavior, replay/dedupe semantics, reply-token handling, media size limits, and Telegram album buffering. Do not introduce bundled channel ids into core or change the shared text-only `messages.inbound` semantics silently. Update LINE config schema/types, docs, focused tests, and changelog with reporter credit if the change is user-facing.
