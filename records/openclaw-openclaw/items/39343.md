## Summary

Keep open. Current main still only solves the Telegram album side: Telegram buffers `media_group_id` updates and processes one aggregated turn, but LINE still downloads at most the current event's media into a fresh `allMedia` array and immediately calls `processMessage`. LINE has no `mediaBatchDelayMs` or equivalent config/docs surface, and shared inbound debouncing remains text-only, so the central LINE image-burst problem is still valid.

## What This Changes

Request an optional media batching buffer for rapid image/media updates, using Telegram `media_group_id` where available and sender/chat/time proximity for LINE.

## Best Possible Solution

Keep this open and implement the missing LINE plugin behavior in the LINE plugin boundary: add an optional media batching delay, buffer downloadable LINE media by account + conversation + sender, flush on timer and before same-key text/non-media events, aggregate all media into one `LineInboundContext`, preserve access control/mention/replay/reply-token behavior, document the option, and leave existing Telegram `media_group_id` buffering unchanged.
