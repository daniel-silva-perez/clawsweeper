## Summary

Keep open. Current main already buffers Telegram albums by `media_group_id`, but the LINE half of the request remains unresolved: LINE image/media webhook events are still downloaded, contextualized, and dispatched one event at a time with no same-sender media buffer or LINE batching config/docs surface.

## What This Changes

Request optional media batching so rapid image/media messages from Telegram albums or LINE same-sender bursts are delivered to the agent as one turn instead of separate image turns.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: add an optional LINE media batching delay, buffer downloadable LINE media by account plus conversation plus sender, flush on timer and before same-key text or non-media events, aggregate the media into one `LineInboundContext`, and leave existing Telegram album buffering plus shared text debounce semantics unchanged.

## Reproduction Assessment

Yes. A high-confidence reproduction path is visible from source: send two LINE image webhook message events from the same sender/chat and `handleLineWebhookEvents` will call `handleMessageEvent` twice, each with a fresh single-event `allMedia` array and immediate `processMessage`.

## Solution Assessment

No for a generic gateway-layer fix; the safer maintainable path is plugin-owned LINE batching using existing context/media seams. Telegram already solves the album case, so the narrow remaining fix should not move channel-specific heuristics into core unless a concrete SDK gap appears.
