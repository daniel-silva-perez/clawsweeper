## Summary

Keep open. Current main already handles Telegram media groups, but the LINE part of the request is still unresolved: LINE image/media webhook events are downloaded and dispatched one event at a time, with no same-sender batching buffer or LINE config/docs surface.

## What This Changes

The issue requests optional media batching for rapid multi-image/media messages, using Telegram `media_group_id` where available and a LINE sender/chat/time-proximity heuristic otherwise.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: add an optional media batching delay, use LINE `imageSet.id` for simultaneous image sets when present, fall back to account plus conversation plus sender plus short time proximity for other downloadable media, flush on timer and before same-key text/non-media events, and aggregate through the existing `allMedia` context fields without changing Telegram album handling or shared text debounce semantics.

## Reproduction Assessment

Yes. Send two LINE image webhook message events from the same source; current `handleLineWebhookEvents` calls `handleMessageEvent` twice, and each call creates a fresh single-message `allMedia` list before immediate `processMessage`.

## Solution Assessment

No for the proposed generic gateway-layer placement. The maintainable fix is a LINE-plugin-scoped buffer using existing media/context seams and LINE SDK `imageSet` metadata, while leaving the already-implemented Telegram media-group path unchanged.
