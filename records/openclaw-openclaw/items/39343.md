## Summary

Keep open. Current main implements Telegram album buffering, but the LINE half of the request remains unresolved: LINE media webhook events are still downloaded and dispatched one event at a time, with no LINE media batching config or docs surface.

## What This Changes

The issue requests optional batching for rapid image/media messages so Telegram albums or LINE same-sender media bursts reach the agent as one turn instead of separate image turns.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: add an optional LINE/account media batching delay, use `imageSet.id` for simultaneous LINE image sets when present, fall back to account plus conversation plus sender plus time proximity for other downloadable media, flush on timer and before same-key text/non-media events, and aggregate through the existing `allMedia` context fields without moving channel-specific heuristics into core.

## Reproduction Assessment

Yes. Send two LINE image webhook message events from the same sender/chat and current `handleLineWebhookEvents` will call `handleMessageEvent` twice, each with a fresh single-event `allMedia` array and immediate `processMessage`.

## Solution Assessment

No for the proposed generic gateway-layer placement. The narrow maintainable fix is LINE-plugin scoped batching using existing context/media seams and the LINE SDK `imageSet` metadata where available, while leaving the already-implemented Telegram album buffering unchanged.
