## Summary

Keep open. Current main still does not implement authority-link materialization, and this PR remains a potentially useful but security-sensitive sandbox filesystem change. The current PR head narrows the first version with an allowlist and nested-symlink skipping, but it still needs maintainer/security review for resolved-target root policy, alias and hardlink handling, reused-sandbox refresh semantics, Windows test guards, and docs before it is safe to merge.

## What This Changes

This PR adds sandbox workspace hydration for allowlisted top-level authority symlinks such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, plus unit tests for materialization, allowlisting, and nested symlink skipping.

## Best Possible Solution

Define a documented authority-link model for sandbox seed workspaces, then implement it through the same boundary primitives used for bootstrap reads: explicit supported names, resolved-target validation against the intended root policy, symlink and hardlink rejection throughout copied trees, deliberate refresh behavior for reused sandboxes, Windows-safe tests, and user-facing docs for the supported authority names.

## Risks / Open Questions

- The authority-link trust model is unresolved: the PR needs a maintainer decision on whether allowlisted authority symlinks may intentionally resolve outside the agent workspace, and which roots are trusted if they can.
- The PR head still appears to copy resolved allowlisted targets without shared root/alias validation or hardlink rejection, so an approved name such as `README.md` or a file under `docs` could expose unintended host content in `workspaceAccess: "none"` or `"ro"`.
- The copy-if-missing behavior leaves reused sandbox workspaces vulnerable to stale or self-modified authority docs unless refresh semantics are deliberately defined.
- The new symlink tests in the PR are not guarded like the current Windows-sensitive sandbox symlink coverage.
