## Summary

Keep this PR open: current main still lacks the requested authority-link hydration, but the proposed branch changes the sandbox filesystem boundary and has unresolved P1 security issues around copying resolved symlink targets and hardlinked files into non-rw sandboxes.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level symlinked authority files/directories such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, plus unit tests for materialization, allowlisting, nested symlink skipping, and non-symlink exclusion.

## Best Possible Solution

Define a documented authority-link model before merging: supported names, resolved-target policy, symlink and hardlink rejection throughout copied trees, deliberate refresh semantics for reused sandboxes, Windows-safe tests, and docs that explain exactly which authority files can be hydrated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority links before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted workspace name and then copies that target without proving it is inside the seed workspace or another explicitly trusted root. A workspace can point `README.md` or `docs` at credentials or a sibling checkout and materialize that content into a non-rw sandbox.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. That bypasses the hardlink alias rejection used by `openBoundaryFile`, so same-filesystem host content can be smuggled through an allowed authority tree into the sandbox.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: The copy helper returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later source changes or sandbox-side edits can leave future sessions reading stale authority content instead of current workspace docs.
  - confidence: 0.86
- **[P2] Guard symlink tests on Windows:** `src/agents/sandbox.workspace.test.ts:25-34`
  - body: The new tests create file and directory symlinks unconditionally. Existing sandbox alias tests are skipped on `win32` because Windows runners may lack symlink privileges, so this file can fail before reaching the intended assertions.
  - confidence: 0.88
