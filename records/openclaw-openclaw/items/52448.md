## Summary

Keep open. Current main still only seeds boundary-checked bootstrap files, and this PR changes the sandbox filesystem trust boundary with unresolved P1 issues around trusted-root validation and hardlink aliases. The author did address earlier feedback by narrowing the broad symlink walk to an allowlist and skipping nested symlinks, but the remaining boundary concerns need maintainer/secops review before merge.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level authority symlinks such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, plus unit tests for materialization, allowlisting, nested symlink skipping, and non-symlink exclusion.

## Best Possible Solution

Define a documented authority-link model before merging: explicit supported names, resolved-target validation against the intended root policy, symlink and hardlink rejection throughout copied trees, deliberate refresh behavior for reused sandboxes, Windows-safe tests, and docs that explain exactly which authority files can be hydrated.

## Reproduction Assessment

Yes. The review findings can be reproduced from the PR head by pointing an allowlisted `README.md` or `docs` symlink at an outside path, or by placing a same-filesystem hardlink inside an allowed authority tree; the PR copy path does not use the current `openBoundaryFile` root and hardlink checks.

## Solution Assessment

No. Copying the resolved target into the sandbox without root validation or hardlink rejection is not the safest maintainable solution for this sandbox boundary; a narrower approved model should preserve the current seed contract unless maintainers explicitly expand it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority links before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted name and then copies that resolved path without checking it is inside the seed workspace or another explicitly trusted root. A workspace can point `README.md` or `docs` at a credential directory or sibling checkout and have that content materialized into a `workspaceAccess: "none"`/`"ro"` sandbox, bypassing the current seed contract.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. Current bootstrap seeding rejects hardlink aliases through `openBoundaryFile`, but this path would copy a same-filesystem hardlink inside an allowed authority tree into the sandbox as readable content.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: `copyAuthorityTreeIfMissing` returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later changes to `README.md` or `docs` symlink targets, or edits left in the copied sandbox files, will not be refreshed and later sessions can read stale authority content.
  - confidence: 0.86
- **[P2] Guard symlink tests on Windows:** `src/agents/sandbox.workspace.test.ts:25-34`
  - body: The new tests create file and directory symlinks unconditionally. Existing sandbox symlink/hardlink tests are skipped on `win32` because Windows runners may lack symlink privileges, so this added file can fail the Windows lane before reaching the sandbox assertions.
  - confidence: 0.88
