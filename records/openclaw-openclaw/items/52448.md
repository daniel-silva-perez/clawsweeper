## Summary

Keep this PR open, but it should not merge in its current form. The user problem is real and current main still only seeds bootstrap files, but the patch changes a sandbox filesystem boundary and leaves unresolved host-path exposure and hardlink alias risks.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level symlinked authority files/directories such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, plus unit tests for materialization, allowlisting, nested symlink skipping, and non-symlink exclusion.

## Best Possible Solution

Define a documented authority-link model before merging: explicit supported names, resolved-target validation against the intended trusted roots, symlink and hardlink rejection throughout copied trees, deliberate refresh behavior for reused sandboxes, Windows-safe tests, and matching workspace/sandbox docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority links before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted workspace name and then copies that target without proving it is inside the seed workspace or another trusted root. A workspace can point `README.md` or `docs` at credentials or a sibling checkout and materialize that content into a `workspaceAccess: "none"`/`"ro"` sandbox.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `copyAuthorityTreeIfMissing` skips symlinks but copies hardlinked files as ordinary files. That bypasses the hardlink alias rejection used by `openBoundaryFile`, so an allowed authority tree can smuggle same-filesystem host content into the sandbox.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: The copy helper returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later source changes or sandbox-side edits can leave future sessions reading stale authority content instead of the current workspace docs.
  - confidence: 0.86
- **[P2] Guard symlink tests on Windows:** `src/agents/sandbox.workspace.test.ts:27-38`
  - body: The added tests create file and directory symlinks unconditionally. Existing sandbox alias tests use `it.runIf(process.platform !== "win32")` because Windows runners may not have symlink privileges, so this file can fail before reaching the assertions.
  - confidence: 0.88
