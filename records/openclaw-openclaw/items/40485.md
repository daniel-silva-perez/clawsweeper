## Summary

Keep this PR open as useful source work, but it should not merge as-is. Current main still routes Telegram native-command delivery through the active plugin under `extensions/telegram/src`, while this PR changes the removed legacy `src/telegram/*` path, so the intended direct-chat `/new` and `/reset` suppression is not applied to shipped code.

## What This Changes

The PR changes legacy Telegram native-command delivery so direct-chat `/new` and `/reset` replies use `replyToMode: "off"`, and adds a legacy `/new` regression test.

## Best Possible Solution

Port the narrow behavior into the active Telegram plugin so direct-chat native core `/new` and `/reset` suppress implicit reply threading, while group/forum topics, plugin commands, explicit reply targets, menus, and other native commands keep current semantics.

## Reproduction Assessment

Yes. A code-level reproduction exists on current main: configure Telegram `replyToMode` to a threaded mode and invoke direct-chat native `/new` or `/reset`; the active handler injects the command `message_id`, and delivery only suppresses it when `replyToMode` is `off`.

## Solution Assessment

No. The idea is narrow and appropriate, but this PR is not the best implementation because it changes removed legacy files instead of the active `extensions/telegram` plugin path.
