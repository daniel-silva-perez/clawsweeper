## Summary

Close as implemented on main. The deleted-message path this PR was trying to avoid is already fixed by the shipped main implementation: bare `/new` and `/reset` reset turns deny stale implicit current-message threading, Telegram native-command delivery falls back to the visible command message when no explicit reply target exists, and the linked bug was closed with that release evidence. This PR also targets the removed legacy `src/telegram` path, so it should not be merged as-is.

## What This Changes

This PR changes legacy Telegram native-command delivery so direct-chat native `/new` and `/reset` force `replyToMode: "off"`, and adds a legacy `/new` regression test.

## Best Possible Solution

Close this PR and keep the shipped main implementation. If deleted-message replies are observed again on a release after `v2026.4.22`, the follow-up should be a new report with a current Telegram repro and should start from the active `extensions/telegram` plugin path plus the auto-reply reply-threading guard.

## Reproduction Assessment

Yes. The original bug has a high-confidence code-level path: pre-fix bare reset turns could inherit an implicit current-message reply target, while current main denies that inheritance and anchors native Telegram command replies to the visible command `message_id`.

## Solution Assessment

No for this PR. The shipped main implementation is the better solution because it fixes the stale target source and preserves normal reply threading semantics, while this PR patches removed legacy files and would not affect the active Telegram plugin.
