## Summary

Keep open. Current main still lacks the requested Discord stuck-session circuit breaker, and the PR targets a real auto-reply queue failure mode, but the patch is not merge-safe as written because the new timer lifecycle can reuse stale deadlines and the recovery drain can restart before the aborted run has actually released the session key.

## What This Changes

The PR adds a Discord-only stuck-session circuit breaker for queued follow-ups, wires active-run state hooks through `runPreparedReply` and `runReplyAgent`, exports the queue helper, and adds focused Vitest coverage.

## Best Possible Solution

Keep the feature scoped to Discord follow-up queues, but make the breaker session-aware and drain-aware: replace or clear stale per-queue timers when queued work drains or a new active run begins, and wait for abort settlement or use the existing abort-and-drain/force-clear pattern before starting recovery follow-ups.

## Reproduction Assessment

Yes for the blocking review findings. The static reproduction path is clear from the PR diff and current main: a normal drain leaves the new breaker timer in the global map, and the breaker calls `scheduleDrain` immediately after an abort helper that only signals cancellation while the reply-run registry can still reject same-session work.

## Solution Assessment

No, not as written. The requested circuit breaker is a reasonable narrow fix, but the implementation should tie timer lifetime to queue/session lifetime and reuse the established abort-and-drain semantics before attempting recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Refresh stale breaker timers for new queued work:** `src/auto-reply/reply/queue/circuit-breaker.ts:72-73`
  - body: A breaker timer can remain in `DISCORD_STUCK_SESSION_BREAKER_TIMERS` after a queue drains normally, because the drain cleanup deletes the queue/callback but never clears this new map. A later Discord session on the same `queueKey` will return here and inherit the old deadline/callback, so it can fire too early or miss a fresh five-minute protection window. Clear the breaker when the queue drains or replace the timer when new queued work arrives.
  - confidence: 0.86
- **[P2] Wait for abort to settle before draining:** `src/auto-reply/reply/queue/circuit-breaker.ts:112`
  - body: The breaker starts the recovery drain as soon as `abortActiveRun` returns true, but the current abort helper only signals cancellation and the reply-run registry keeps the session key active until the operation clears. For a stuck run that does not exit promptly, the recovery follow-up can hit `ReplyRunAlreadyActiveError` and keep the queue blocked. Wait for the aborted run to drain, or use the existing force-clear recovery path, before scheduling recovery work.
  - confidence: 0.9
