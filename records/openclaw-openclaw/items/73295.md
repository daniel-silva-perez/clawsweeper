## Summary

Keep this PR open. Current main still lacks the proposed Discord follow-up queue circuit breaker: there is no circuit-breaker queue module/export, the enqueue-followup path only restarts draining when the active run is already inactive, and runPreparedReply does not pass the abort/session/streaming hooks that this PR adds. Main does have a related Discord inbound worker timeout with docs and tests, but that is a separate worker-level timeout/fallback, not this PR's non-streaming active-run breaker with recovery notice ordering. The PR remains the open unmerged successor to closed PRs #73233 and #73239, and review comments still flag correctness issues that need maintainer follow-up.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to fix or adjudicate the stale-timer replacement/clear behavior, narrow isRunActive before passing it, decide whether queued recovery must refresh session IDs after abort, and confirm whether the threshold belongs in Discord channel policy/config. Then validate with focused queue tests plus Discord-safe operational proof, or replace this PR with a narrower main-branch implementation that covers the same stuck follow-up queue failure mode.
