## Summary

Keep open. Current main still lacks the PR’s specific Discord stuck-session circuit breaker: there is no queue/circuit-breaker module or export, the enqueue-followup path only drains if the active run is already inactive, and runReplyAgent does not receive streaming/active-session/abort hooks. Main does have a Discord inbound worker timeout safety valve, but that is a configurable 30-minute queued-handler timeout, not the proposed 5-minute non-streaming active-run abort with a recovery notice. The related earlier PRs are closed unmerged, so this PR remains the active implementation candidate.

## Best Possible Solution

Keep this PR open for normal maintainer review. The best path is to review the focused circuit-breaker implementation against current main, decide whether the 5-minute threshold and recovery notice are appropriate or should be configurable channel policy, and require focused tests plus any Discord-safe operational proof maintainers want before landing or replacing it with a narrower main-branch implementation.
