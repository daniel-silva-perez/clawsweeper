## Summary

Keep open. The current `sessions.list` path still maps every stored row that survives broad key filters into a Gateway session row, and `buildGatewaySessionRow` still emits `updatedAt: null` plus an omitted `sessionId` when the store entry lacks those fields. The related fix PR #57419 was closed unmerged, so it is useful context but not shipped behavior.

## Best Possible Solution

Implement a server-side guard in the session listing path, likely in `listSessionsFromStore` or the combined-store normalization boundary, that excludes store placeholder/non-session rows before building Gateway session rows. Add a regression test with an `agent:<id>:sessions` row lacking `sessionId` and `updatedAt` plus at least one real session for the same agent. The prior PR #57419 is relevant implementation context, but the fix should be reviewed against current `main` semantics before landing.

## Risks / Open Questions

- Control UI and clients may still need defensive filtering for rows without `sessionId` or valid `updatedAt`.
- A narrow `:sessions` suffix filter alone may miss other malformed non-session rows; the safer fix should define the server-side validity rule for listable session rows.
