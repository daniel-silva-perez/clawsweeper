## Summary

Keep open. Current main still allows a malformed store placeholder such as `agent:sean:sessions` to flow through `sessions.list`: the Gateway handler passes the combined store directly to `listSessionsFromStore`, that list path only filters by key scope/search/activity/limit, and `buildGatewaySessionRow` still serializes rows with `updatedAt: null` and an omitted `sessionId` when the retained store entry lacks those fields. The related implementation PR #57419 is closed and unmerged in the provided GitHub context, and no later commits on the inspected session-listing files changed this behavior after the prior Codex review commit.

## Best Possible Solution

Implement the fix server-side in `listSessionsFromStore` or the combined-store normalization boundary so non-session placeholder rows are excluded before `buildGatewaySessionRow`. Add a regression test with an `agent:<id>:sessions` placeholder lacking `sessionId`/`updatedAt` alongside a real session for that agent. The closed PR #57419 is useful context, but the current fix should be reviewed against main and avoid relying only on a suffix check.

## Risks / Open Questions

- A narrow `:sessions` suffix-only filter may miss other malformed non-session rows; the fix should define a server-side validity rule for listable session rows, likely requiring a non-empty `sessionId` and valid activity timestamp.
- Client-side filtering would hide the Control UI symptom but leave the `sessions.list` API contract leaking malformed rows to other clients.
