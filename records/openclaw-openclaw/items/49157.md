## Summary

Keep open. Current main still has the reported live-PID stale-lock behavior: session lock stale detection defaults to 30 minutes, a lock owned by an alive PID is not stale until its age exceeds that window, and effectively no-timeout embedded runs can still derive a 2,147,000,000 ms maxHoldMs. Current cleanup/finally paths narrow some leak cases, but they do not implement heartbeat leases, a shorter stale window, or bounded recovery for a process-alive abandoned lock context.

## Best Possible Solution

Keep this issue open as a focused session-lock hardening bug. The best next step is to cap embedded session-lock maxHoldMs to a short bounded window, add regression coverage for a process-alive abandoned lock context, and then decide whether heartbeat/lease semantics or explicit nonfatal unhandled-rejection lock cleanup is needed. Coordinate with #13744 for configurability and lock-scope work, but do not close this as a duplicate unless a canonical issue explicitly covers leaked live-PID session locks.
