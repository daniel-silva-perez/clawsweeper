## Summary

Keep open. Current main still permits process-alive session lock stalls: live-PID lock files are only stale by age, the default stale window remains 30 minutes, and embedded runs derive session-lock maxHoldMs from long/no-timeout run budgets instead of a short bounded lease. Recent orphan-lock, PID-starttime, exit-cleanup, and watchdog work is useful partial hardening, but it does not implement heartbeat/lease semantics or a bounded recovery path for an abandoned live-PID lock context.

## Best Possible Solution

Keep this issue open as the focused session-lock recovery bug. The best next implementation is to add a short bounded recovery path for abandoned process-alive session locks, likely by capping embedded-run maxHoldMs independently from agent run timeout and adding regression coverage for a process that remains alive while its lock-holding context stops making progress. Then decide whether heartbeat/lease mtime updates or nonfatal unhandled-rejection lock cleanup are needed for longer operations. Coordinate with #13744 for configurability and lock-scope changes, but do not close this until the live-PID leak case has direct source and test coverage.
