## Summary

Keep open. Current main still treats a live-PID session lock as valid until the 30-minute stale age or the in-process watchdog budget expires, and embedded runs still derive that watchdog budget from long run plus compaction timeouts. Recent cleanup and lock-scope work reduce some stalls, but they do not directly cover an abandoned lock context inside a still-running Gateway process.

## What This Changes

The issue requests bounded recovery for session write locks that can remain blocked when a live Gateway process leaves a `.jsonl.lock` file behind after an abandoned lock-holding context.

## Best Possible Solution

Keep this issue open as the focused live-PID session-lock recovery bug. The maintainable end state is direct recovery coverage for a process that remains alive while its lock-holding context stops making progress, then a bounded lease/watchdog or narrower critical-section implementation that cannot unlock genuinely active transcript writes; #13744 can carry broader configurability and lock-scope follow-up.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a `.jsonl.lock` payload with the current/Gateway PID and a `createdAt` younger than 30 minutes while no matching in-process `HELD_LOCKS` entry exists; current `inspectLockPayload` and acquire contention logic treat it as non-stale and wait until the caller timeout.

## Solution Assessment

No. Current main includes useful partial mitigations, but it is not the best complete solution because the remaining recovery behavior still depends on a 30-minute age check or a potentially huge watchdog budget rather than a bounded live-PID lease/progress signal.
