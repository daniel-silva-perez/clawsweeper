## Summary

Keep open. Current main still leaves the reported live-PID session-lock stall path unresolved: session locks use a 30-minute stale default, live PID locks are only stale by age/recycled/dead PID state, and embedded runs still derive maxHoldMs from long run timeouts rather than a short bounded lease. Recent cleanup, watchdog, PID-starttime, and orphan-lock hardening are useful partial fixes, but they do not yet cover the abandoned live-PID lock context with direct source/test coverage.

## Best Possible Solution

Keep this issue open as the focused session-lock recovery bug. The best next implementation is to add a short bounded recovery path for abandoned process-alive session locks, likely by capping embedded-run maxHoldMs independently from agent runtime timeout and adding regression coverage for a process that remains alive while its lock-holding context stops making progress. Then decide whether heartbeat/lease mtime updates or nonfatal unhandled-rejection lock cleanup are needed for longer operations. Coordinate with #13744 for configurability and lock-scope changes, but do not close this until the live-PID leak case has direct source and test coverage.
