## Summary

Keep open. Current main does not implement the requested `ACP_BACKEND_UNAVAILABLE` classification for plain `acpx exited with code 1` turn failures, but this PR should not merge as-is because #61495 moved the process-exit loop into the published `acpx` package and the PR branch now changes stale in-tree runtime code while broadening classification beyond genuinely silent exits.

## What This Changes

The PR changes ACPX/ACP error classification so selected non-zero `acpx` process exits surface as `ACP_BACKEND_UNAVAILABLE`, with runtime, manager, fixture, and error-text test updates.

## Best Possible Solution

Move the exit classification to the current ACPX boundary: only genuinely silent prompt/backend exits should become `ACP_BACKEND_UNAVAILABLE`, while stderr-backed failures, permission-denied exits, control commands, session initialization, and recovery paths keep their existing semantics. OpenClaw should then consume the published or explicitly pinned ACPX change with focused wrapper, manager, and error-text coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review findings: the PR diff deterministically remaps prompt/control exits and early session-init errors, and current-main tests still expect plain `acpx exited with code 1` after a done event to surface as `ACP_TURN_FAILED`. I did not execute tests because this pass was read-only.

## Solution Assessment

Is this the best way to solve the issue? No. The PR targets stale in-tree ACPX runtime code and broadens classification beyond truly silent exits; the safer solution is package-level ACPX classification plus a narrow OpenClaw consumption/update layer.
