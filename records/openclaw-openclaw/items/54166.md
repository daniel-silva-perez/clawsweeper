## Summary

Keep this issue open. On current main, `llm-task` still derives provider and model independently, only strips a duplicate same-provider prefix, and dispatches that unresolved pair to the embedded runner. The reporter's distinction from the haiku-only alias issue remains correct: this is still a broader canonical model-ref resolution gap for bundled `llm-task` overrides.

## Best Possible Solution

Keep this issue open. Add an additive plugin-facing model-ref resolution helper, or equivalent runtime seam, then route `llm-task` `model` and `defaultModel` through the same canonical resolver used by normal agent and subagent model selection. Use the resolved `provider/model` for thinking policy, `allowedModels`, and `runEmbeddedPiAgent`, with focused regression tests for `gemini-flash`, explicit `google/gemini-3-flash-preview` under a different default provider, an Anthropic alias, and canonical allowlist validation.

## Risks / Open Questions

- Closing now would hide an active bundled-plugin routing bug for cross-provider `llm-task` model overrides and bare aliases.
- A fix must resolve aliases and explicit provider/model refs before thinking policy lookup, `allowedModels` validation, and embedded dispatch, otherwise allowlists can reject valid canonical targets or validate the wrong key.
- Because `llm-task` is a bundled plugin, the fix should add or reuse a narrow plugin runtime/SDK seam rather than importing core `src/agents` internals from extension production code.
