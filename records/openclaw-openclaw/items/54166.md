## Summary

Keep this issue open. Current main still routes `llm-task` model overrides through a separately selected provider plus raw model string, so a bare alias such as `gemini-flash` or a cross-provider ref such as `google/gemini-3-flash-preview` can still be dispatched under the default/selected provider instead of being resolved as a canonical model ref first. The reporter's distinction from the related haiku-only alias issue remains correct.

## Best Possible Solution

Keep this issue open. Add a plugin-facing model-ref resolution helper, or otherwise route `llm-task` `model` and `defaultModel` through the same canonical alias/model-ref resolver used by normal agent and subagent model selection before thinking policy checks, `allowedModels` validation, and `runEmbeddedPiAgent`. Add focused regression tests for `gemini-flash`, explicit `google/gemini-3-flash-preview` with an Anthropic/OpenAI default provider, an Anthropic alias, and allowlist validation against the resolved canonical key.

## Risks / Open Questions

- Closing now would hide a still-actionable bundled plugin routing bug for cross-provider `llm-task` model overrides and bare aliases.
- A correct fix must resolve aliases and provider/model refs before `allowedModels` validation so policy checks apply to canonical `provider/model` keys.
- Because `llm-task` is a bundled plugin, the implementation should use or add a narrow plugin runtime/SDK model-ref resolver seam rather than importing core internals from extension code.
