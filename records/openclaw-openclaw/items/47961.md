## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested Feishu delayed acknowledgment behavior, while the PR branch is stale against the current turn-kernel dispatch path and has two concrete P2 correctness/test defects already visible from the diff and current dispatcher code.

## What This Changes

The PR adds an 8-second delayed Feishu p2p/private-message acknowledgment timer around Feishu bot dispatch and adds bot tests for slow, fast, and group-chat delayed-ack behavior.

## Best Possible Solution

Land a Feishu-local revision that integrates with the current turn-kernel runDispatch path, sends the acknowledgment only for p2p/private chats after a real no-visible-output interval, suppresses it as soon as any user-visible Feishu reply starts or is delivered, preserves normal final reply delivery, and covers slow DM, fast DM, group chat, and early-visible-reply cancellation deterministically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1835-1847`
  - body: The timer is armed before `withReplyDispatcher` and only canceled in the outer `finally`, but Feishu can start streaming/card/plain-text/media output before the dispatcher settles. A long run that already produced visible output can still send the extra acknowledgment at 8 seconds, violating the PR's no-visible-reply condition and creating duplicate/confusing messages.
  - confidence: 0.9
- **[P2] Use a fast dispatcher in the fast-path test:** `extensions/feishu/src/bot.test.ts:536-537`
  - body: This case advances fake timers by only 7,999 ms and then awaits `pending`, while the shared dispatch mock resolves after 9,000 ms. Under fake timers the promise cannot settle there, so the test can hang instead of proving the delayed ack was canceled before the threshold.
  - confidence: 0.88
