## Summary

Keep open and do not merge as-is. Current main still lacks the Feishu delayed-ack behavior, so this PR is not obsolete, but the proposed patch has two P2 correctness issues: the acknowledgment can be sent after visible output has already started, and the fast-path fake-timer test can hang.

## What This Changes

The PR adds an 8-second delayed acknowledgment timer for slow Feishu p2p/private direct-message dispatches and adds Feishu bot tests for slow, fast, and group-chat behavior.

## Best Possible Solution

Land a Feishu-local revision that sends the delayed acknowledgment only for p2p/private chats after a real interval with no visible output, cancels or suppresses it as soon as any visible Feishu reply starts or is delivered, preserves the final reply flow, and covers slow DM, fast DM, group chat, and early-visible-reply cancellation deterministically.

## Reproduction Assessment

Yes. The review defects reproduce by inspection with a long Feishu DM run that starts visible output before 8 seconds but settles after 8 seconds, and by the fake-timer fast-path test awaiting a 9,000 ms mock after only 7,999 ms.

## Solution Assessment

No. The PR is appropriately Feishu-local, but as written it is not the best fix because the timer is tied to dispatcher settlement rather than visible-output start, and one new negative test does not settle deterministically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1833-1848`
  - body: The timer is only canceled after `withReplyDispatcher` settles. Feishu visible replies can start in `onReplyStart`/`deliver` before settlement, so a long run that already produced a streaming/card/text/media reply can still send the extra acknowledgment after 8s, contrary to the no-user-visible-reply condition.
  - confidence: 0.9
- **[P2] Use a fast dispatcher in the fast-path test:** `extensions/feishu/src/bot.test.ts:537`
  - body: This case advances fake timers by only 7,999 ms and then awaits `pending`, while the shared dispatch mock resolves after 9,000 ms. Under fake timers that promise cannot settle there, so the test can hang instead of proving cancellation before the threshold.
  - confidence: 0.88
