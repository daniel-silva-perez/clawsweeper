## Summary

Keep this PR open, but it needs changes before merge. Current main still lacks the requested Feishu delayed-ack behavior, and the PR is Feishu-local with no supply-chain-sensitive file changes, but the proposed timer can still send an acknowledgment after visible output has already begun and one added fake-timer test can hang instead of proving the fast path.

## What This Changes

The PR adds an 8-second delayed Feishu p2p/private-message acknowledgment timer in the Feishu bot dispatch path and bot tests for slow, fast, and group-chat behavior.

## Best Possible Solution

Land a Feishu-local revision that sends the delayed acknowledgment only for p2p/private chats after a real no-visible-output interval, cancels or suppresses it as soon as any user-visible Feishu reply starts or is delivered, preserves the normal final reply flow, and covers slow DM, fast DM, group chat, and early-visible-reply cancellation deterministically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1833-1848`
  - body: The timer is only canceled after `withReplyDispatcher` settles, but the Feishu reply dispatcher can start streaming/card/plain-text/media output before settlement. A long run that already produced visible output can still send the delayed acknowledgment after 8s, which violates the PR's no-user-visible-reply condition and creates duplicate/confusing messages.
  - confidence: 0.9
- **[P2] Use a fast dispatcher in the fast-path test:** `extensions/feishu/src/bot.test.ts:537`
  - body: The fast direct-message case advances fake timers by only 7,999 ms and then awaits `pending`, but the shared dispatch mock resolves after 9,000 ms. Under fake timers, that promise cannot settle there, so the test can hang instead of validating that the ack was canceled before the threshold.
  - confidence: 0.88
