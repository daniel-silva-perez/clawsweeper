## Summary

Keep open. Current main still lacks the Feishu delayed acknowledgment behavior, and the PR is a plausible Feishu-local feature, but the proposed patch still sends the ack after visible output has already started and includes a fake-timer test that can hang.

## What This Changes

The PR adds an 8-second delayed acknowledgment for slow Feishu p2p/private-message dispatches and bot tests for slow, fast, and group-chat paths.

## Best Possible Solution

Land a Feishu-local revision that sends the delayed acknowledgment only for p2p/private chats after a real no-visible-output interval, cancels or suppresses it as soon as any user-visible Feishu reply starts or is delivered, preserves the normal final reply flow, and covers slow DM, fast DM, group chat, and early-visible-reply cancellation deterministically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1833-1848`
  - body: The timer is only canceled after `withReplyDispatcher` settles, but Feishu can start streaming/card/plain-text/media output before settlement. A long run that already produced visible output can still send the delayed acknowledgment after 8 seconds, which violates the PR body's no-user-visible-reply condition and creates duplicate/confusing messages.
  - confidence: 0.9
- **[P2] Use a fast dispatcher in the fast-path test:** `extensions/feishu/src/bot.test.ts:537-538`
  - body: The fast direct-message case advances fake timers by only 7,999 ms and then awaits `pending`, but the shared dispatch mock resolves after 9,000 ms. Under fake timers, that promise cannot settle there, so the test can hang instead of validating that the ack was canceled before the threshold.
  - confidence: 0.88
