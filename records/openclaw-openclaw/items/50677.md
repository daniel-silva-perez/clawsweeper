## Summary

Keep this issue open. Current main still drops prompt-facing skills by count and by character budget without returning omitted skill names, the truncation notice is still injected into the model prompt rather than surfaced to the operator, and the config/schema surface still has no priority, pinned, or required-skill contract. The related omitted-name PR and broader priority issue are both still open, so the requested visibility and prioritization behavior has not landed.

## What This Changes

The issue requests warnings, documentation, and priority or pinning controls so user-created skills are not silently omitted when the skills prompt budget is exceeded.

## Best Possible Solution

Keep this open until OpenClaw ships an omitted-skill discovery or operator-visible warning path, documents the skills prompt budget and truncation outcome on the public skills docs, and explicitly decides whether priority, pinning, or context-required skills belong in the skills config contract or remain tracked by narrower follow-ups.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to build a workspace skills prompt with many eligible skills and a low `skills.limits.maxSkillsPromptChars` or `maxSkillsInPrompt`; existing tests already assert count and compact truncation, including an `included 0 of 3` case.

## Solution Assessment

No. The current implementation is not the best resolution because it preserves only a count-only model-facing note; #42637 looks like a plausible narrow omitted-name improvement, while priority/pinning and user-visible warning semantics need maintainer API/product review.
