## Summary

Keep open. Current main partially addresses the older invalid cron shape by requiring main-session jobs to use `systemEvent`, but it does not resolve the reported `ANNOUNCE_SKIP` confusion: exact assistant `ANNOUNCE_SKIP` replies are still globally hidden from chat/history, and cron docs/source do not warn or validate main-session system-event text that asks the agent to produce that token.

## What This Changes

The issue requests clearer cron documentation plus warning or validation when a main-session cron payload contains `ANNOUNCE_SKIP`, because that exact assistant reply is treated as an internal silent control token.

## Best Possible Solution

Keep the global exact-control-token suppression for subagents and chat projection, but add cron-local ergonomics: document that main-session system events are prompts for the next heartbeat, warn or fail closed when main-session cron text instructs an exact `ANNOUNCE_SKIP` reply, and add a focused regression test proving users get a diagnostic instead of a silent-looking success.

## Reproduction Assessment

Yes. A source-backed reproduction is clear: create a main-session cron job with `payload.kind="systemEvent"` whose text asks the agent to reply exactly `ANNOUNCE_SKIP`; cron enqueues that text, and gateway chat projection/history suppress exact `ANNOUNCE_SKIP` assistant replies.

## Solution Assessment

Yes, the best fix is narrow cron-specific docs plus warning/validation and tests. Changing the global `ANNOUNCE_SKIP` suppressor would risk breaking intentional subagent announce-skip behavior already documented elsewhere.
