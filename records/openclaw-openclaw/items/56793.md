## Summary

Keep open. Current main still supports separate managed OpenClaw profiles and existing-session Chrome MCP attach profiles, but I found no CLI command, server route, or service path that imports or copies source browser data into a managed profile. Because the requested behavior would intentionally transfer sensitive signed-in browser state, this needs maintainer product/security review rather than stale cleanup.

## What This Changes

Add an explicit one-shot browser profile import flow that copies an existing user browser profile into a managed OpenClaw browser profile, leaves the source untouched, and documents it as non-automatic.

## Best Possible Solution

If maintainers accept the product direction, implement this inside the bundled browser plugin as an explicit opt-in CLI and server route that stops or resets only the destination managed profile, resolves and validates the source profile, copies with a documented exclusion list for lock/runtime artifacts, redecorates/cleans the destination, updates persistent-mutation guards, and adds user-facing docs and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. For this feature request, current-main verification is a source/docs check: the available CLI, route, service, and tests show attach/reset/create/delete behavior but no import path.

## Solution Assessment

Unclear. The issue's explicit one-shot import direction is safer than automatic startup copying, but the best implementation needs maintainer product and security decisions around what state may be copied and how consent is presented.
