## Summary

Keep open. Current main still separates managed OpenClaw browser profiles from existing-session Chrome MCP attach profiles; I found no CLI command, server route, or profile-service path that imports or clones user browser state into a managed profile, and the requested state transfer is product/security-sensitive.

## What This Changes

Add an explicit one-shot CLI/API import flow that copies a user browser profile into a managed OpenClaw browser profile while leaving the source untouched.

## Best Possible Solution

If maintainers accept the product direction, implement this inside the bundled browser plugin as an explicit opt-in import path that stops or resets only the destination managed profile, resolves and validates the source profile, copies with a documented exclusion list for transient and high-risk artifacts, redecorates/cleans the destination, protects the route as a persistent browser mutation, and updates CLI/API docs plus focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction. For this feature request, the high-confidence check is source/docs inspection: current main documents and implements attach, reset, create, and delete profile behavior, but no import or clone path.

## Solution Assessment

Unclear. The requested one-shot explicit import is safer than automatic startup copying, but the best implementation depends on maintainer product and security decisions about consent, copied state, and exclusions.
