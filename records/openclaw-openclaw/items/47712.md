## Summary

Keep open. Current main still lacks the requested shared `messages.tts.filterEmoji` config and does not consistently filter emoji before TTS synthesis; only provider/channel-local sanitizers exist, so the issue remains valid and narrow.

## What This Changes

The issue requests native TTS support for a `messages.tts.filterEmoji` option so emoji are skipped before speech generation.

## Best Possible Solution

Add a public `messages.tts.filterEmoji?: boolean` config field with a documented default, route all shared TTS text preparation through one helper before provider synthesis, and keep visible chat reply text unchanged while only sanitizing spoken text.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: `messages.tts.filterEmoji` is outside the strict config schema, and the shared synthesis paths do not call any config-controlled emoji filter before `textToSpeech`.

## Solution Assessment

Yes. The narrow maintainable fix is a shared config/runtime/docs/test change; provider-local sanitizers alone are not sufficient because they leave auto-TTS, the agent tool, Gateway conversion, and third-party speech providers inconsistent.
