## Summary

Keep open. Current main still does not expose or honor `messages.tts.filterEmoji`: the strict TTS config schema, public type surface, generated help/docs, auto-TTS path, gateway conversion path, and agent `tts` tool all lack a config-controlled emoji filtering step. There is partial provider-local emoji stripping in Discord voice replies and the Local CLI speech provider, but that does not satisfy the requested native TTS config behavior.

## What This Changes

Add native `messages.tts.filterEmoji` support so synthesized TTS text can omit emoji while preserving visible chat text.

## Best Possible Solution

Implement a shared, config-controlled TTS text preparation step that accepts `messages.tts.filterEmoji?: boolean`, documents the default, applies only to text sent to speech providers and spoken-text/tool metadata, and leaves visible chat payload text unchanged.
