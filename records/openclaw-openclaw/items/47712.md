## Summary

Keep open. Current main still does not accept or honor `messages.tts.filterEmoji`; the shared TTS config contract, generated schema, docs, automatic TTS path, agent tool, and Gateway `tts.convert` path all lack config-controlled emoji filtering.

## What This Changes

Request to add `messages.tts.filterEmoji` support so TTS filters emoji before speech synthesis, defaulting to enabled.

## Best Possible Solution

Add a public `messages.tts.filterEmoji?: boolean` config field with a documented default, route all shared TTS text preparation through one helper, and keep visible chat reply text unchanged while only sanitizing text sent to speech providers.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: adding `messages.tts.filterEmoji` is outside the strict config schema, and the current synthesis paths do not call any shared emoji filter before `textToSpeech`.

## Solution Assessment

Yes. The narrow maintainable fix is a shared config/runtime/docs/test change; provider-local sanitizers alone are not the best solution because they leave other TTS surfaces inconsistent.
