## Summary

Keep open. Current main still does not define or honor `messages.tts.filterEmoji`; the strict config schema, public TTS type, generated schema, docs, automatic reply TTS path, agent `tts` tool, and Gateway `tts.convert` path all lack config-controlled emoji filtering. Local CLI and Discord voice have provider/path-local emoji stripping, but that does not satisfy the requested shared TTS behavior.

## What This Changes

Request support for `messages.tts.filterEmoji` so text-to-speech can skip emoji, preferably enabled by default before synthesis.

## Best Possible Solution

Add `messages.tts.filterEmoji?: boolean` to the public config contract with a documented default, implement a shared TTS text-preparation helper that strips markdown and optionally emoji, and route auto-TTS, the agent `tts` tool, and Gateway `tts.convert` through it while preserving the visible chat reply text and provider selection behavior.
