## Summary

Keep open. Current main still allows the reported cron isolated-agent shape: incomplete embedded-runner terminal output is not classified before the model fallback call accepts a result, direct cron delivery can count a sent fatal payload as delivered, and cron state persists execution status and delivery status independently.

## What This Changes

The issue requests clearer cron isolated-agent handling when provider fallback yields incomplete-turn text, including retry or alert behavior and non-contradictory delivery state for error runs.

## Best Possible Solution

Fix the cron isolated-agent path so embedded incomplete-turn terminal results are classified before fallback accepts them, then ensure failure text is handled as failure notification or failed delivery metadata rather than normal cron output. If a public state field changes, align docs, schema/types, and monitoring expectations.

## Repair Work Prompt

Fix openclaw/openclaw#72985. Current main calls runWithModelFallback from src/cron/isolated-agent/run-executor.ts without classifying EmbeddedPiRunResult, so an embedded-runner incomplete-turn error payload like "Agent couldn't generate a response" can be accepted as the successful fallback candidate, delivered through direct cron announce, and persisted as lastRunStatus=error with lastDelivered=true. Wire classifyEmbeddedPiRunResultForModelFallback, or a cron-specific equivalent, into the cron isolated-agent model fallback call so incomplete terminal results trigger the fallback chain or final failure instead of normal announce delivery. Add a backstop so this specific error fallback text is not counted as normal cron output delivery; prefer the narrowest behavior change and do not change legitimate successful message-tool delivery semantics. Add regression tests for primary failure plus fallback incomplete result, final incomplete result not being delivered as normal output, and cron state avoiding the reported contradiction or exposing a separate error-delivery field if that product shape is chosen. Update docs/schema only if a public cron state field changes. Do not edit generated baselines, unrelated provider fallback behavior, or closed unrelated delivery semantics.
