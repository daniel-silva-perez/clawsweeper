## Summary

Keep open. Current main still leaves the cron isolated-agent path without the embedded-runner result classifier, so incomplete-turn error payloads can be accepted as fallback results, and cron state persistence still records delivery state independently from execution status.

## What This Changes

The issue requests fixing isolated cron fallback/error delivery so incomplete-turn failure text is not treated as normal cron output and `status: "error"` rows do not report ordinary delivery success.

## Best Possible Solution

Implement a cron-owned fix that reuses the embedded-runner fallback classifier before cron accepts fallback attempts, then keep normal delivery state reserved for usable assistant output. If maintainers want to record that an error notification was sent, add a separate schema/docs field instead of overloading `lastDelivered`.

## Reproduction Assessment

Yes. A focused source-level reproduction can mock an isolated cron embedded result containing the incomplete-turn error payload, let direct announce delivery succeed, and observe that current code can return/persist `status: "error"` with `delivered: true`.

## Solution Assessment

Yes. Keeping this open and landing a narrow cron/runtime fix is the right path; a persistence-only regex would hide one symptom but would not make the fallback chain reject incomplete embedded-runner results.

## Repair Work Prompt

Fix the isolated cron fallback/delivery bug from https://github.com/openclaw/openclaw/issues/72985. Current main calls `runWithModelFallback` from `src/cron/isolated-agent/run-executor.ts` without `classifyResult`, even though `classifyEmbeddedPiRunResultForModelFallback` already recognizes incomplete-turn error payloads. Wire that classifier into the cron embedded-runner fallback path so incomplete/empty/reasoning-only terminal results advance the configured fallback chain or fail with a classified error when no fallback remains. Also adjust cron delivery/state semantics so fatal/incomplete error payloads are not counted as normal delivered cron output; if an error notification needs tracking, keep it separate from `lastDelivered`. Add focused tests for the incomplete-turn warning, fallback advancement/no-fallback failure, and persistence of error outcomes without `lastDelivered: true`. Update docs only if schema/CLI output semantics change, and add a user-facing changelog fix entry crediting `Thanks @Allenbluff`. Do not change provider defaults, add a new fallback policy, or touch unrelated channel/provider behavior.
