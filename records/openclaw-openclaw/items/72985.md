## Summary

Keep open. Current main still leaves cron isolated-agent fallback accepting incomplete embedded-runner results when no classifier hook is supplied, and a successful channel send of the resulting error payload can still flow through to `status: "error"` plus `delivered: true` state.

## What This Changes

The issue asks cron isolated-agent fallback handling to retry or alert on incomplete-turn fallback failures and to stop recording failed cron runs as normal delivered output.

## Best Possible Solution

Cron should reuse the existing embedded-runner result classifier before accepting model-fallback results, then reserve normal delivered state for usable assistant output. If maintainers want to track that an error notification reached the user, add a separate explicit error-delivery field and document it instead of overloading `lastDelivered`.

## Reproduction Assessment

Yes. A deterministic source-level reproduction can mock the isolated embedded run to return the incomplete-turn warning as an `isError` payload, let announce delivery succeed, and observe the current path returning/persisting `status: "error"` with `delivered: true`.

## Solution Assessment

Yes. Keeping this open and fixing the cron fallback/delivery semantics is the narrow maintainable path; an ad hoc persistence regex alone would leave fallback acceptance and monitoring semantics split across layers.
