## Summary

Keep open. Current main still accepts cron isolated-agent incomplete-turn warning payloads as successful model-fallback results, can deliver the fatal text through cron announce/direct delivery, and persists execution status independently from delivery status, so the reported `error` plus `delivered=true` state remains possible.

## What This Changes

Request to fix isolated cron runs so provider-fallback incomplete-turn warnings are retried or alerted instead of silently delivered, and error runs are not persisted as delivered normal output.

## Best Possible Solution

Implement a cron-owned fix that reuses the embedded-runner result classifier before fallback accepts a result and reserves normal delivery state for usable assistant output. If maintainers want to record that an error notification was sent, add a separate schema/docs field rather than overloading `lastDelivered`.

## Reproduction Assessment

Yes. The supplied production trace matches a source-level reproduction: mock cron isolated execution to return the incomplete-turn error payload, let delivery succeed, and current main can return and persist `status: "error"` with `delivered: true`.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrow maintainable fix is to wire the existing classifier into the cron fallback path and adjust delivery/state semantics, rather than only adding an ad hoc regex at persistence time.

## Repair Work Prompt

Fix this issue: cron isolated-agent runs can accept the embedded-runner incomplete-turn warning as a successful model-fallback result, deliver that fatal text to the announce target, and persist `lastRunStatus: "error"` together with `lastDelivered: true`. Wire `classifyEmbeddedPiRunResultForModelFallback` or an equivalent cron-specific classifier into `createCronPromptExecutor`'s `runWithModelFallback` call so incomplete embedded results continue through fallback or fail cleanly. Then ensure fatal/incomplete error payloads are not counted as normal cron output delivery: either keep `lastDelivered` false for terminal error runs or add a separate error-notification delivery field with docs/schema updates. Add focused regression coverage for the fallback classifier wiring and for error-plus-delivered persistence. Preserve existing side-effect guards so runs that committed message/tool output are not retried incorrectly. Do not change provider defaults, channel plugin behavior, or unrelated cron scheduling policy. Changelog only if the persisted public state contract or user-facing cron behavior changes.
