## Summary

Keep open. Current main still appears to allow the reported cron isolated-agent path: incomplete embedded-runner terminal output is not classified before model fallback returns, direct cron delivery can send that fatal text, and cron state records execution status and delivery status independently.

## What This Changes

The issue requests clearer cron isolated-agent handling when provider fallback yields incomplete-turn text, including retry or alert behavior and non-contradictory delivery state for error runs.

## Best Possible Solution

Fix this in the cron isolated-agent path by applying the embedded-runner incomplete-result classifier, or an equivalent cron-specific classifier, before model fallback accepts a result. Error fallback text should not be treated as normal cron output delivery; state should either keep normal delivery false for these runs or expose a separate error-notification delivery signal with tests and docs/schema updates if public state changes.

## Repair Work Prompt

Fix openclaw/openclaw#72985. Current main calls runWithModelFallback from src/cron/isolated-agent/run-executor.ts without classifying EmbeddedPiRunResult, so an embedded-runner incomplete-turn error payload like "Agent couldn't generate a response" can be accepted as the successful fallback candidate, delivered through direct cron announce, and persisted as lastRunStatus=error with lastDelivered=true. Wire classifyEmbeddedPiRunResultForModelFallback, or a cron-specific equivalent, into the cron isolated-agent model fallback call so incomplete terminal results trigger the fallback chain or final failure instead of normal announce delivery. Add a backstop so this specific error fallback text is not counted as normal cron output delivery; prefer the narrowest behavior change and do not change legitimate successful message-tool delivery semantics. Add regression tests for primary failure plus fallback incomplete result, final incomplete result not being delivered as normal output, and cron state avoiding the reported contradiction or exposing a separate error-delivery field if that product shape is chosen. Update docs/schema only if a public cron state field changes. Do not edit generated baselines or unrelated provider fallback behavior.
