## Summary

Keep open. Current main still has the reported cron isolated-agent gap: cron calls the model fallback runner without the existing embedded-runner result classifier, fatal embedded-runner error payloads remain deliverable, and the cron service persists delivery state independently from run error state, so an error run can still end with delivered=true.

## What This Changes

Cron isolated-agent fallback failures can deliver the embedded-runner incomplete-turn warning as normal cron output and persist an error run with delivered=true.

## Best Possible Solution

Implement a cron-owned fix that reuses the existing embedded-runner fallback classifier before cron accepts a fallback attempt, and keep normal delivery state reserved for usable assistant output. If maintainers want to record that an error notification was sent, use a separate error-notification state or existing failure alert path instead of overloading lastDelivered.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock cron isolated execution so runWithModelFallback returns an EmbeddedPiRunResult with the incomplete-turn isError payload, allow direct delivery to succeed, and observe current code returning/persisting status error with delivered true.

## Solution Assessment

Yes. Keeping this open and fixing it in the cron isolated-agent path is the narrow maintainable direction; a persistence-only regex would hide the symptom after delivery instead of preventing fallback from accepting an incomplete terminal result.

## Repair Work Prompt

Fix openclaw/openclaw#72985. Current main lets cron isolated-agent fallback accept an embedded-runner incomplete-turn error payload as a successful fallback because createCronPromptExecutor calls runWithModelFallback without classifyEmbeddedPiRunResultForModelFallback; direct delivery can then send that isError payload and CronService persists lastRunStatus=error plus lastDelivered=true. Implement the fix in the cron-owned path: wire the existing embedded-runner result classifier into cron fallback before accepting a fallback attempt, and ensure fatal/error cron payloads do not count as normal assistant-output delivery. Do not add provider-specific defaults, broaden the core fallback contract, or solve this only with an ad hoc persistence regex. Add focused tests for runWithModelFallback classifier forwarding in cron, incomplete-turn fallback rejection, and persistence of error runs as not delivered or separately error-delivered.
