## Summary

Keep open. Current main still has the reported source-level gap: isolated cron calls the model fallback runner without the embedded-runner result classifier, and the delivery/persistence path can still turn a fatal error payload into `status: "error"` plus `delivered: true` when the error text is successfully sent to the announce target.

## What This Changes

The issue asks cron isolated-agent runs to avoid silently delivering incomplete-turn fallback warnings as normal output and to stop recording failed runs as successfully delivered user content.

## Best Possible Solution

Cron should classify incomplete embedded-runner outputs before accepting a fallback attempt, and failed cron output should be persisted separately from successful user-content delivery. If all fallback candidates fail, the run should remain an error and use failure notification semantics without marking the fallback warning as delivered content.

## Reproduction Assessment

Yes. A deterministic test can mock an isolated embedded run returning only the incomplete-turn warning as an error payload, let announce delivery succeed, and observe the current path returning an error run whose delivered flag is persisted as delivered.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing embedded-runner fallback classifier in the cron fallback call and adjust delivery persistence semantics; a regex-only `lastDelivered` patch would leave fallback acceptance and monitoring behavior split across layers.
