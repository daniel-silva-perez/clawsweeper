## Summary

Keep open. Current main still has the reported source-level gap: isolated cron calls the model fallback runner without the embedded-runner result classifier, and cron persistence still records delivery success independently from a terminal error status.

## What This Changes

The issue reports that isolated cron fallback can deliver the embedded-runner incomplete-turn warning as chat output and persist `lastRunStatus: "error"` together with `lastDelivered: true`.

## Best Possible Solution

Cron should reuse the embedded-runner fallback classifier before accepting a fallback attempt, then persist delivery state so useful assistant output is distinct from an error notification or fallback warning text.

## Reproduction Assessment

Yes. A focused unit reproduction can mock an isolated embedded run to return only the incomplete-turn warning as an `isError` payload, let announce delivery succeed, and observe the current path returning an error run with delivered metadata.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing result classifier into isolated cron fallback and adjust delivery persistence semantics; a regex-only `lastDelivered` patch would leave fallback acceptance and monitoring behavior split across layers.

## Repair Work Prompt

Fix the isolated cron fallback/delivery bug reported in https://github.com/openclaw/openclaw/issues/72985. The observable bug is that a cron isolated-agent fallback can accept an embedded-runner incomplete-turn warning as the fallback result, deliver that warning as chat output, and persist `lastRunStatus: "error"` with `lastDelivered: true`. Keep the fix narrow: reuse `classifyEmbeddedPiRunResultForModelFallback` in the isolated cron `runWithModelFallback` call, and adjust cron run/job-state delivery semantics so error/fallback-warning delivery is not counted as useful assistant output. Prefer an explicit error-notification state or `lastDelivered: false` for fatal error payloads over a broad regex-only persistence patch. Add focused regression tests for the fallback classifier being invoked by isolated cron and for job state persistence when status is error and delivery transport succeeds. Do not refactor unrelated cron scheduling, provider selection, timeout, or channel routing behavior. Add a changelog entry under the active `### Fixes` section if the change is user-facing, with appropriate non-bot reporter credit.
