## Summary

Keep this PR open. Current main still has the shared pnpm store cache mount on the `pnpm install` layer only; the `runtime-assets` stage still runs `pnpm prune --prod` without that cache mount, and the repository still uses the hoisted pnpm linker premise described in the PR. The PR is a focused Docker build optimization, but it needs rebasing onto the newer multi-command `runtime-assets` block before maintainer validation.

## Best Possible Solution

Keep this PR open. The best path is to rebase or update the patch so the existing `runtime-assets` RUN block starts with the same `openclaw-pnpm-store` cache mount used by `pnpm install`, optionally add a regression assertion that the prune RUN has the mount, then validate with the Docker build lane or an equivalent `docker build --progress=plain .` check.
