## Summary

Keep open. Current main still has targeted runtime parameter checks for read/write/edit and handler-level checks for tools such as exec, but it does not provide the requested generic configurable pre-dispatch required-parameter validation or automatic retry/prompt policy across tool schemas. The related #56562 is closed, but it does not leave a canonical open implementation path or prove this behavior shipped.

## What This Changes

The issue requests configurable automatic validation of required tool parameters before handlers run, with optional retry/auto-prompt behavior for missing parameters.

## Best Possible Solution

Add a core dispatch-level validation design that can reject missing required fields from normalized tool schemas before invoking handlers, while preserving existing richer per-tool validators and treating retry/auto-prompt behavior as a separate opt-in policy with maintainer-reviewed loop semantics.

## Reproduction Assessment

Yes. A focused source check reproduces the remaining gap: exec missing command is rejected inside the handler, read/write/edit use targeted wrappers, and there is no generic schema-derived validation/config path before tool execution.

## Solution Assessment

Unclear. Pre-dispatch validation is a plausible direction, but the requested config plus automatic retry prompting is broader than the existing targeted validators; the safer path is a maintainer-designed generic validator first and a separate retry policy decision.
