## Summary

Keep open. Current main has some targeted runtime parameter checks for read/write/edit and an in-handler exec command check, but it does not implement the requested generic, configurable pre-dispatch validation or auto-retry/auto-prompt behavior across tool schemas.

## What This Changes

The issue asks for configurable pre-call validation of required tool parameters, plus optional retry prompting, so malformed tool calls can be rejected before invoking tool handlers.

## Best Possible Solution

Add a core tool-dispatch validation design that can reject missing required fields from normalized tool schemas before invoking handlers, while preserving the existing richer validators for file tools and treating retry/auto-prompt behavior as a separately reviewed opt-in policy.

## Reproduction Assessment

Yes. A focused source check shows missing exec `command` is rejected inside `createExecTool.execute`, while read/write/edit use hard-coded validators; the requested generic configurable pre-dispatch path is absent.

## Solution Assessment

Unclear. The direction is plausible, but the requested config plus automatic retry prompting is broader than the current targeted validators and should be designed with maintainer input before implementation.
