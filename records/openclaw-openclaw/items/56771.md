## Summary

Keep open. Current main has useful targeted runtime checks for some file tools and handler-level checks for exec, but it still lacks the requested generic, configurable pre-dispatch validation layer and automatic missing-parameter retry/prompt policy across tool schemas.

## What This Changes

The issue requests configurable pre-call validation for missing required tool parameters, with optional automatic prompting or retry before invoking tool handlers.

## Best Possible Solution

Add a core dispatch-level validator that can reject missing required fields from normalized tool schemas before invoking handlers, while preserving richer per-tool validators. Treat automatic retry or prompting as a separate opt-in policy after maintainer review of loop behavior and plugin/client-tool contracts.

## Reproduction Assessment

Yes. A focused source check reproduces the remaining gap: `exec` missing `command` is rejected inside `createExecTool.execute`, read/write/edit use targeted wrappers, and no config or dispatch layer validates all tool schemas before handler invocation.

## Solution Assessment

Unclear. Schema-derived pre-dispatch validation is a plausible narrow core seam, but the requested automatic prompting/retry behavior is broader than validation and should be designed separately.
