## Summary

Keep this PR open. Current main still has the reported behavior: pinned write helper failures are normalized to the generic `path is not a regular file under root`, and the helper still emits raw stderr or a generic helper-code message without the PR's `python3 helper failed` context or install hints. The PR is focused and its second commit addresses the Greptile platform-guard review comment, but it still needs maintainer review plus regression coverage before landing.

## Best Possible Solution

Keep this PR open as a focused bug-fix candidate. The best path is to update it with regression tests for pinned write error preservation, verify or fix the actual missing-`python3` spawn-error path, then review and land if the error-surfacing behavior is acceptable for plugin/archive install failures.

## Risks / Open Questions

- Security review: the PR diff is limited to local infra error construction in `src/infra/fs-safe.ts` and `src/infra/fs-pinned-write-helper.ts`; it does not touch CI, dependencies, lockfiles, install scripts, publishing metadata, permissions, or downloaded code execution paths.
- The PR should add regression tests before landing, especially for preserving the underlying pinned-write error through `normalizePinnedWriteError`.
- The PR body claims Linux-without-python3 behavior, but the current helper can also fail via a child-process spawn `error` event when `python3` is absent; the proposed patch should be checked against that path instead of only non-zero helper exits.
- Surfacing raw helper stderr can include local filesystem paths, which is useful for diagnosis here but should remain a deliberate CLI/user-facing error choice.
