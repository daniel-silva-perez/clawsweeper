## Summary

Keep open. Current main does not make this PR obsolete: `sessions_spawn` still lacks the proposed `taskFlow` surface and the documented CLI only lists flow list/show/cancel, not retry. The PR branch is useful and the author addressed earlier review feedback around ACP registration and lane wiring, but the final patch still has blocking retry, cancellation, and UI staleness bugs that should be fixed before merge.

## What This Changes

The PR adds managed child TaskFlow creation from `sessions_spawn`, child-completion synchronization, progress/guidance delivery, retry operations, CLI/gateway/runtime/UI controls, and related docs/tests.

## Best Possible Solution

Revise the branch into a mergeable TaskFlow v1 control-surface slice: preserve the useful `sessions_spawn(taskFlow=...)` and retry concepts, but reopen terminal flows before child linking, give cancellation precedence over child completion, require precise child wait identifiers, derive UX guidance from the same retry eligibility logic, and make Control UI flow loads session-token safe before broad validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reopen terminal flows before retry child spawns:** `src/tasks/task-executor.ts:569`
  - body: `resolveRetryableManagedChildTaskFlow` allows retries from `failed` and `lost`, but this branch passes `parentFlowId` into the child spawn while the flow is still terminal. Task creation rejects terminal parent flows, so subagent retries fail before relaunch and ACP retries can start detached from the flow. Move the flow back to a nonterminal waiting/retrying state before spawning/linking the child, or link only after a successful reopen.
  - confidence: 0.91
- **[P1] Give cancellation precedence over completion sync:** `src/tasks/task-registry.ts:1121`
  - body: `updateTask` calls `syncManagedFlowCompletionFromTask` before `syncManagedFlowCancellationFromTask`, and the completion sync does not skip flows with `cancelRequestedAt`. If the user cancels a managed flow and the child later exits, the flow can be finalized as succeeded/failed/lost first, after which the cancellation sync returns because the flow is terminal. Check cancellation first or make completion sync bail out when cancellation was requested.
  - confidence: 0.89
- **[P2] Require wait identifiers when matching child completions:** `src/tasks/task-registry.ts:815-825`
  - body: `matchesManagedChildWaitState` only rejects mismatches when both the wait state and task provide an identifier. If `waitJson` names a specific `runId` or `childSessionKey` and another linked terminal task lacks that field, it still matches and can close the flow with the wrong outcome. When the wait state has an identifier, require the task to have the same identifier.
  - confidence: 0.86
- **[P2] Base retry guidance on actual retry eligibility:** `src/tasks/task-flow-guidance.ts:41-47`
  - body: `getTaskFlowGuidance` reports `retry_available` for every failed or lost managed flow, but the retry resolver can reject flows with `launch.retryable: false`, missing launch metadata, or an active child task. That makes CLI/UI text say retry is available while the retry section says it is not. Derive guidance from the same eligibility check or suppress retry guidance when retry is blocked.
  - confidence: 0.9
- **[P2] Discard stale flow loads after session switches:** `ui/src/ui/controllers/task-flows.ts:23-33`
  - body: `loadLatestTaskFlow` returns early while a load is in flight and then writes the result using mutable state after the await. If the user switches sessions during that request, the old response can populate the new chat with the previous session's flow card until another refresh happens. Capture the requested session key or request token and ignore stale responses, while allowing a new-session load to proceed.
  - confidence: 0.87
