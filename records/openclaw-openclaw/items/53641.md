## Summary

Keep open. Current main still has the core false-success gap for ordinary Discord `message.send` with direct `buffer` attachment fields: the public send schema accepts `buffer`/filename/MIME inputs, but attachment hydration still excludes the `send` action, the shared send runner only turns media/path-style strings into `mediaUrl`, and the Discord bridge/runtime only uploads when `mediaUrl` is present. The path/filePath multipart route appears implemented and tested, so the remaining work should stay narrowed to buffer-backed send delivery or explicit rejection while the overlapping open PRs are reviewed.

## What This Changes

The issue asks for Discord `message.send` to deliver local image or buffer attachments as real Discord uploads, or fail instead of returning a successful message id while dropping the file.

## Best Possible Solution

Review or consolidate #57335 and #63578 into one narrow fix that aligns ordinary Discord `message.send` with its advertised schema by hydrating and delivering `buffer`/`filename`/`contentType` through the existing attachment and multipart upload path, or by rejecting unsupported attachment fields clearly with regression coverage.
