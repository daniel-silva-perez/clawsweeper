## Summary

Keep open. Current main still has the direct `buffer` false-success gap for ordinary Discord `message.send`: the public send schema accepts buffer attachment fields, but the shared send hydration gate excludes `send`, the runner/Discord bridge only forward media path or URL fields, and the Discord runtime only enters multipart upload when `mediaUrl` is present.

## What This Changes

Discord `message.send` should upload local or buffer-backed attachments, or fail loudly instead of returning success with a message id while omitting the attachment.

## Best Possible Solution

Make ordinary `message.send` either hydrate and deliver `buffer`/`filename`/`contentType` through the shared attachment path and Discord multipart upload with regression coverage, or reject those fields explicitly for `send`. The open candidate PRs should be reviewed or consolidated before closing this issue.
