## Summary

Keep open. Current main still advertises buffer-backed attachments on the shared message send schema, but ordinary send actions are not hydrated through the attachment pipeline, and the Discord send bridge/runtime only upload when a path-like mediaUrl/path/filePath reaches sendMessageDiscord. Path/filePath Discord uploads are implemented and tested; the remaining false-success risk is the direct buffer/contentType/mimeType message.send path already called out in the prior Codex review and open candidate PRs #57335 and #63578.

## Best Possible Solution

Keep this issue open and narrow the remaining work to direct buffer attachments on ordinary Discord message.send. The best fix is to make buffer, filename, and contentType/mimeType flow through shared send hydration and the Discord multipart upload path with regression coverage, or explicitly reject those fields for ordinary send if direct buffers are not supported. Review #57335 and #63578 as candidate fixes, but close only after a fix is merged and verified on main.
