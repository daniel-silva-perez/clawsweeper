## Summary

Keep open. Current main still has the direct buffer-backed Discord `message.send` false-success gap: the public send schema accepts `buffer`/filename/content-type fields, but ordinary `send` is excluded from attachment hydration and the shared runner/Discord bridge only forward path or URL media references into the upload path. The path/filePath media upload route itself is present and has multipart serialization coverage, so the remaining bug is direct buffer delivery or an explicit fail-fast rejection.

## What This Changes

The issue reports that Discord `message.send` accepts attachment inputs and returns a successful message id, but Discord receives a message with no uploaded attachment.

## Best Possible Solution

Align ordinary Discord `message.send` with its advertised schema by hydrating and delivering `buffer`/`filename`/`contentType` through the existing attachment pipeline and Discord multipart upload path with regression coverage, or reject those fields clearly for `send` so callers cannot receive a false success.
