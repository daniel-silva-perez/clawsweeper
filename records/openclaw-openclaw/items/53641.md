## Summary

Keep open. Current main still has the direct-buffer false-success gap: `message.send` accepts `buffer`/filename/MIME fields, but ordinary `send` is excluded from attachment hydration, the shared send runner and Discord bridge only pass path/URL media, and the Discord upload branch is gated on `mediaUrl`. The path/filePath multipart path appears implemented and tested, so the remaining issue is direct buffer delivery or explicit rejection while the overlapping open PRs are reviewed.

## What This Changes

The issue reports Discord `message.send` returning success and a message id while dropping provided attachment payloads instead of uploading visible Discord attachments.

## Best Possible Solution

Review or consolidate #57335 and #63578 into one narrow fix that aligns ordinary Discord `message.send` with its advertised schema by hydrating and delivering `buffer`/`filename`/`contentType` through the existing attachment and multipart upload path, or by rejecting those fields clearly for `send` with regression coverage.
