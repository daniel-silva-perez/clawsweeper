## Summary

Keep open. Current main still accepts `buffer` attachment fields in the ordinary `send` schema, but `send` is excluded from attachment hydration and the Discord bridge never forwards buffer or MIME fields into the multipart upload path. The path/filePath media route appears implemented and tested, so the remaining bug is narrowed to direct buffer-backed delivery or explicit rejection; open PRs #57335 and #63578 are unmerged implementation candidates.

## What This Changes

Discord `message.send` should not return success while dropping requested attachments, especially when callers provide buffer, filename, or MIME fields.

## Best Possible Solution

Align the advertised `send` contract with runtime behavior by either hydrating and delivering `buffer`/`filename`/`contentType` through the existing attachment pipeline and Discord multipart upload path with focused regression coverage, or rejecting unsupported attachment fields with a clear error.
