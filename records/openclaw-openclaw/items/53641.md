## Summary

Keep open. Current main still has the direct buffer-backed Discord `message.send` false-success gap: the shared send schema accepts `buffer` attachment fields, but the send hydration gate excludes `send`, the shared runner and Discord bridge only promote path/URL media references, and the Discord upload runtime only enters multipart upload when `mediaUrl` is present. The path/filePath multipart route appears implemented and tested, so the remaining bug should stay narrowed to direct buffer delivery or fail-fast rejection; open PRs #57335 and #63578 target this area but are not merged.

## What This Changes

The issue requests fixing Discord `message.send` so attachment inputs, especially local or buffer file payloads, are actually uploaded instead of returning success with an empty Discord attachments array.

## Best Possible Solution

Make ordinary Discord `message.send` either hydrate and deliver `buffer`/`filename`/`contentType` through the shared attachment pipeline and Discord multipart upload path with regression coverage, or explicitly reject those fields for `send` so callers cannot get a false success.
