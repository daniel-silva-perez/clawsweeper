## Summary

Keep open. Current main still does not route direct buffer-backed ordinary Discord sends into the upload pipeline: the public send schema accepts `buffer`, but attachment hydration excludes `send`, and the shared/Discord send paths only forward media/path/filePath style references. The path/filePath multipart upload route appears implemented and tested, so the remaining bug is direct buffer delivery or explicit fail-fast rejection.

## What This Changes

Discord `message.send` should deliver attachment inputs such as `buffer`, `filePath`, or `path`, or reject unsupported attachment fields instead of returning success without an uploaded Discord attachment.

## Best Possible Solution

Align ordinary Discord `message.send` with its advertised schema by either hydrating `buffer`/`filename`/`contentType` into the existing attachment upload pipeline with regression coverage, or rejecting those fields clearly when used with `send`.
