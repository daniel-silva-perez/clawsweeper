## Summary

Keep open. Current main still lacks the requested built-in `/hooks/message` ingress, and the companion issue was already closed as superseded by this PR, but the PR is not merge-ready because the diff introduces a backwards-incompatible routing regression for existing mapped hooks on the `message` subpath and drops structured `sessions.create` error status in the new dispatch path.

## What This Changes

The PR adds a built-in `POST /hooks/message` webhook route, message/event dispatch plumbing, hook replay/idempotency behavior, `/hooks/agent` boundary changes, cron delivery-contract wiring, tests, and docs.

## Best Possible Solution

Land the `/hooks/message` feature only after preserving existing mapped-hook behavior for the `message` subpath and carrying structured gateway method errors through the new HTTP dispatcher, with docs and tests covering both compatibility cases.

## Reproduction Assessment

Yes. A config with `hooks.mappings[].match.path: "message"` currently reaches `applyHookMappings` on main, while the PR intercepts `/hooks/message` before mappings and validates it against the new strict message contract.

## Solution Assessment

No. The proposed feature boundary is useful, but the implementation should either let explicit mappings keep precedence or provide a deliberate compatibility path, and it should map `sessions.create` `INVALID_REQUEST`/`UNAVAILABLE` errors the same way the new `chat.send` path does.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve mapped hooks on the message path:** `src/gateway/server-http.ts:674`
  - body: Current main lets arbitrary hook subpaths, including `message`, fall through to `hooks.mappings`. This new branch runs before mappings, so an existing `hooks.mappings[].match.path: "message"` integration will be intercepted by the built-in contract and can start returning validation errors instead of running its configured mapping. Preserve explicit mappings or add a compatibility path before reserving this subpath.
  - confidence: 0.94
- **[P2] Propagate session creation status codes:** `src/gateway/server/hooks.ts:388`
  - body: `sessions.create` can return structured gateway errors such as `INVALID_REQUEST` or `UNAVAILABLE`, but this branch wraps them in a plain `Error`. The outer HTTP layer then treats recoverable or caller-fixable session creation failures as a generic 500; map the structured error code to `statusCode` here as the `chat.send` failure path does below.
  - confidence: 0.86
