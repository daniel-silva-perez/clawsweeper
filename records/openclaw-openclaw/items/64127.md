## Summary

Keep open. Current main still lacks the quota-exhaustion circuit breaker, so the PR is not obsolete; however, the current branch has blocking correctness issues that prevent the new quota lane from reliably taking effect.

## What This Changes

The PR adds a `quota_exhausted` failover/auth-profile reason, routes it through billing-style disabled backoff, maps it to HTTP 429 status, emits an `agent:provider_tripped` internal hook event, and updates related failover tests.

## Best Possible Solution

Land a corrected narrow circuit breaker that classifies periodic quota exhaustion even when it arrives as an HTTP 429 payload, persists and scores `quota_exhausted` consistently across auth-profile state, skips quota-disabled providers for the intended disabled window with a deliberate probe policy, and locks the hook event with type-guard and regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add quota_exhausted to unavailable-reason scoring:** `src/agents/auth-profiles/usage.ts:565`
  - body: This branch writes `quota_exhausted` into `disabledReason`, but `resolveProfilesUnavailableReason` only scores values present in `FAILURE_REASON_PRIORITY`. Because the new reason is missing there, a provider disabled for quota exhaustion can later infer as `unknown`, taking the transient cooldown path instead of the intended persistent-disabled path.
  - confidence: 0.94
- **[P1] Preserve quota_exhausted for HTTP 429 payloads:** `src/agents/pi-embedded-helpers/errors.ts:812`
  - body: The new message classifier can return `quota_exhausted`, but `classifyFailoverSignal` applies the HTTP status classifier first and status 429 still unconditionally returns `rate_limit`. Real quota payloads that include a 429 status will therefore stay in the transient lane.
  - confidence: 0.92
- **[P1] Avoid 30-second probes for quota-disabled single providers:** `src/agents/model-fallback.ts:699`
  - body: Adding `quota_exhausted` to the billing branch reuses `shouldProbeSingleProviderBilling`, which allows attempts every 30 seconds when no fallback exists. That reintroduces repeated quota retries throughout a multi-hour disabled window, which is the loop this PR is intended to stop.
  - confidence: 0.86
