## Summary

Keep this PR open, but do not merge it as-is. Current main still has the broad Control UI `connect-src 'self' ws: wss:` policy, so the hardening request is real, but the latest PR diff still allows blanket `wss:` connections and conflicts with documented/configured remote gateway URL behavior unless a narrower allowlist or proxy design is chosen.

## What This Changes

The PR changes the Control UI CSP builder to remove the blanket `ws:` scheme, keep `wss:`, add an `extraConnectSrc` option, and add tests for that connect-src policy.

## Best Possible Solution

Define a Control UI CSP model that blocks arbitrary WebSocket exfiltration by default while preserving supported remote gateway workflows through explicit configured origins or a same-origin gateway/proxy path, then update the CSP builder, serving call sites, docs, and regression tests together.

## Reproduction Assessment

Yes. The source check on current main reproduces the reported policy because `buildControlUiCspHeader()` emits `connect-src 'self' ws: wss:`, and the PR diff can be reviewed directly to see that `wss:` remains allowed.

## Solution Assessment

No. The current PR shape is not the best fix because it keeps blanket `wss:` and only removes `ws:`, while a maintainable solution needs either explicit allowed gateway origins or a documented same-origin path for remote gateway access.
