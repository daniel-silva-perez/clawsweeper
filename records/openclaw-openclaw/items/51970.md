## Summary

Keep this PR open because current main still uses the suffix-revealing `maskApiKey` behavior, so the contribution is addressing a real remaining credential-display problem. The patch is not ready to merge as written: its `> 4` branch reduces long-key suffix disclosure but increases exposure for 5- and 6-character secrets, and the test suite needs a small cleanup around the misplaced 5-character case.

## What This Changes

The PR changes `maskApiKey` to reveal only the first character for keys of length 4 or less and the first four characters for longer keys, then updates unit tests to assert suffix nondisclosure.

## Best Possible Solution

Revise the shared `maskApiKey` policy so auth displays never expose suffixes and never reveal a larger fraction of short secrets than current main, then cover empty, <=4, 5-6, medium, long, whitespace-trimmed, and suffix-nondisclosure cases in the colocated unit test plus the affected auth overview/status tests.

## Reproduction Assessment

Yes. The current-main helper can be inspected or exercised with examples like `maskApiKey("1234567890abcdefghijklmnop")`, which returns `12345678...ijklmnop`; the PR regression can be reasoned directly from the changed `> 4` branch, where `maskApiKey("abcde")` would return `abcd...`.

## Solution Assessment

No. The PR is directionally useful for long keys, but the catch-all first-four policy is not the best fix because it increases short-secret exposure; a safer version should keep or reduce short-key disclosure while still removing suffix disclosure.
