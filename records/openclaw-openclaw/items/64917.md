## Summary

Keep open. Current main still starts the A2A ping-pong/announce flow after the initial `deliver:false` target run, uses `webchat` as the internal message channel, and persists that request channel into fresh target session delivery metadata; existing tests also assert `delivery.mode: announce` rather than strict internal behavior.

## What This Changes

The issue asks for `sessions_send` from a visible webchat-backed session to an internal session to remain internal, without exposing the target reply or rebinding the target session to webchat delivery context.

## Best Possible Solution

Keep this open and define the `sessions_send` contract for visible webchat callers. The maintainable fix likely belongs in the session-send/A2A boundary so internal targets do not inherit `webchat` delivery metadata and raw target output is not surfaced to the visible requester unless an explicit announce/delivery mode is requested.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists in current main: `sessions_send` uses `deliver:false` but still returns announce metadata, starts A2A follow-up agent calls, and passes the internal `webchat` channel through the gateway path that persists session delivery fields.

## Solution Assessment

No, the current behavior is not the best solution for the reported strict-internal use case. A docs-only response would preserve the leak hazard; a safer path is to add or enforce an internal-only mode for webchat-originated internal sends while preserving explicit announce behavior where it is intended.
