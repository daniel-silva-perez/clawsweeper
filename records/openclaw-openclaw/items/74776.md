## Summary

Keep open: current main still has the lazy, in-process built-in local embedding path described by the report. The post-release memory startup changes are QMD/primary-model warmup related and do not prewarm or isolate the local node-llama embedding model/context from the gateway event loop.

## What This Changes

The issue asks for memory-core local embedding initialization after restart to stop blocking the first message and channel/WebSocket/HTTP timers, likely through warmup, off-main-thread initialization, and non-gated channel startup.

## Best Possible Solution

Add an owner-approved local embedding runtime path where node-llama model/context initialization and first query embedding cannot starve gateway timers, with observable warmup/status behavior and docs for local cold starts while channels remain responsive during background indexing or warmup.

## Reproduction Assessment

Yes. The report gives a concrete first-message-after-restart path, and current source confirms that an indexed non-empty memory_search still awaits lazy provider initialization before embedding the query; only the exact stall duration needs live Windows/node-llama confirmation.

## Solution Assessment

No. Current main is not the best solution because it still relies on lazy in-process local provider/context initialization; prewarming alone would reduce first-message latency but would not fully protect gateway timers without off-event-loop isolation or equivalent backpressure.
