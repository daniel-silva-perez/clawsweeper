## Summary

Keep open: current main still has the lazy built-in local embedding initialization path described by the report, and I did not find a shipped or main-only mitigation that prewarms it or moves the local model/context load off the gateway event loop. The related local-embedding title matches cover provider lookup and input-limit fixes, not this event-loop blocking behavior.

## What This Changes

The issue requests that memory-core avoid blocking the Node.js event loop when the local embedding provider initializes on the first post-restart memory search, and that startup indexing not delay channel readiness.

## Best Possible Solution

Keep this open and fix the built-in local embedding path so model/context initialization and first query embedding cannot starve gateway WebSocket/HTTP timers. The clean end state is probably a worker-thread or subprocess-backed local provider plus an optional, post-channel-start warmup/status path with docs for local embedding cold starts.
