## Summary

Keep this issue open. Current main has fixed the most dangerous silent-fallback path for explicit agent/user model selections and now rejects remote custom providers without an apiKey, but the Cloudflare AI Gateway multi-provider credential problem remains: credentials are still provider-scoped, Cloudflare onboarding wires only the bundled provider/profile, and the Cloudflare docs do not explain same-gateway custom provider key reuse.

## Best Possible Solution

Keep this open for a generic provider-auth policy: document the current per-provider apiKey or SecretRef requirement for Cloudflare AI Gateway custom providers, add an explicit shared credential mechanism if maintainers want same-gateway reuse, and keep regression coverage so a second remote Cloudflare-backed provider without apiKey fails loudly while intentional fallback chains still work.

## Risks / Open Questions

- Naively sharing credentials by matching baseUrl could leak one provider's credential to an unrelated custom provider; any fix should use explicit SecretRef reuse or a named/scoped gateway credential.
- Changing all 401/403 behavior globally could break intentional configured fallback chains, auth-profile rotation, and cooldown recovery; strict failure should remain tied to explicit selections or a clear operator policy.
