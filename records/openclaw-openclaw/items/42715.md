## Summary

Keep open. Current main has addressed the dangerous silent fallback part for explicit agent model selections and remote custom providers without credentials, but the remaining Cloudflare AI Gateway multi-provider credential-sharing and documentation request is still not implemented.

## What This Changes

The issue requests clearer handling for Cloudflare AI Gateway setups where additional providers share a gateway but lack their own apiKey, including loud auth failures, possible shared credentials, and onboarding documentation.

## Best Possible Solution

Keep the strict missing-auth behavior, document that additional Cloudflare AI Gateway custom providers need their own models.providers.<id>.apiKey or SecretRef plus any gateway auth header, and add a shared credential mechanism only if maintainers choose an explicit, non-leaky config design.

## Reproduction Assessment

Yes for the remaining config behavior: current main can be checked by configuring a remote custom provider without apiKey and observing resolveApiKeyForProvider throw No API key found, while no sibling provider credential is consulted. The original silent default-model fallback no longer reproduces for explicit agent primaries because current agent fallback resolution supplies an empty fallback override.

## Solution Assessment

No, current main is not yet the complete solution for this issue. The loud-failure fix is the right direction, but the remaining docs/shared-credential behavior needs an explicit design rather than implicit same-baseUrl inheritance.
