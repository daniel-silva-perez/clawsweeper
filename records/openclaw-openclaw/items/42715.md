## Summary

Keep this issue open. Current main partially fixes the dangerous silent fallback path for explicit per-agent model selections and remote custom providers without an apiKey now fail with a missing-key error, but the remaining Cloudflare AI Gateway multi-provider credential problem is still present: credentials are provider-scoped, onboarding only wires the bundled cloudflare-ai-gateway provider/profile, and docs do not cover same-gateway custom provider key reuse.

## Best Possible Solution

Keep this open for a generic provider-auth policy: document the current per-provider apiKey or SecretRef requirement for Cloudflare AI Gateway custom providers, add an explicit shared credential mechanism if maintainers want same-gateway reuse, and add regression coverage for a second remote Cloudflare-backed provider without apiKey so strict per-agent selections fail loudly while intentional fallback chains still work.

## Risks / Open Questions

- Implicitly sharing credentials by matching baseUrl could leak one provider's credential to an unrelated custom provider; a fix should use explicit SecretRef reuse or a named/scoped gateway credential.
- Changing all 401/403 behavior globally could break intentional configured fallback chains, auth-profile rotation, and provider cooldown handling; strict failure should stay tied to explicit user/agent selections unless maintainers choose otherwise.
