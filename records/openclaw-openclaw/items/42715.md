## Summary

Keep this issue open. Current main appears to have fixed the most dangerous silent-fallback path for explicit agent/user model selections and now rejects remote custom providers without an apiKey, but the remaining Cloudflare AI Gateway multi-provider credential problem is still present: credentials are provider-scoped, Cloudflare onboarding wires only the bundled cloudflare-ai-gateway provider/profile, and the docs do not cover same-gateway custom provider key reuse.

## Best Possible Solution

Keep this open for a generic provider-auth policy: document the current per-provider apiKey or SecretRef requirement for Cloudflare AI Gateway custom providers, add an explicit shared credential mechanism if maintainers want same-gateway reuse, and keep regression coverage for a second remote Cloudflare-backed provider without apiKey so strict per-agent selections fail loudly while intentional fallback chains still work.

## Risks / Open Questions

- Naively sharing credentials by matching baseUrl could leak one provider's credential to another custom provider; any fix should use explicit SecretRef reuse or a named/scoped gateway credential.
- Changing all 401/403 behavior globally could break intentional configured fallback chains, auth-profile rotation, and cooldown recovery; strict failure should stay tied to explicit selections or a clear operator policy.
