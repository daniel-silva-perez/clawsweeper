## Summary

Keep open. Current main still persists `/exec` host/security/ask/node values into session state and then resolves stored session values before agent/global exec defaults; Discord inbound turns flow through that same session-key auto-reply path. The behavior is also documented and tested as sticky per-session state, so changing it needs maintainer review because it affects exec security semantics.

## What This Changes

The issue asks for Discord session exec overrides to stop persistently outranking later global `tools.exec` security and allowlist changes.

## Best Possible Solution

Define the intended precedence for admin exec policy changes, then update the generic auto-reply/session exec resolver, persistence behavior, tests, and docs so stricter global policy changes take effect for existing Discord sessions without a Discord-only workaround.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: persist a session with `execSecurity: "full"`, then resolve a later turn without an inline directive after global config changes; current main reads the stored session value before agent/global defaults.

## Solution Assessment

Unclear. The report identifies a real behavior, but the safest fix needs maintainer choice between ephemeral `/exec`, stricter-global precedence, config-versioned cleanup, or an operator reset path because current docs/tests treat `/exec` as session-scoped state.
