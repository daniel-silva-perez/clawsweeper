## Summary

Keep open. Current main still persists `/exec` options into durable session entries and resolves those stored values before configured exec defaults, including the global `tools.exec` policy the report expects to take effect after a config change. The behavior is also documented and tested as sticky session state, so this is a valid security-sensitive policy issue rather than a cleanup close.

## What This Changes

The issue requests that Discord sessions stop indefinitely honoring persisted per-session `/exec` overrides after global `tools.exec` configuration changes.

## Best Possible Solution

Define the intended admin-policy precedence for exec settings, then update the generic session exec persistence/resolution path, reset or patch behavior if needed, docs, and focused regression tests so tightened global exec policy takes effect for existing Discord sessions without a Discord-only workaround.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by resolving a persisted session entry with `execSecurity: "full"` after global config changes to a stricter value; current main reads the stored session value before configured defaults. I did not run a live Discord gateway reproduction because this review was required to stay read-only.

## Solution Assessment

Unclear. The report identifies a real policy conflict, but the best fix depends on maintainer semantics for durable session overrides versus operator-enforced global exec policy; a narrow code change should wait for that direction.
