## Summary

Keep this issue open. Current main still only partially addresses the reported stale-task recovery class: cron task cancellation remains unsupported, offline `openclaw tasks maintenance --apply` intentionally does not treat its own empty cron runtime as authoritative, and restart/reload drain still waits indefinitely unless `gateway.reload.deferralTimeoutMs` is configured. The latest issue discussion also reports a v2026.4.26 restart/update failure mode, so this is not obsolete.

## Best Possible Solution

Keep this issue open and scope the next maintainer fix to the remaining operator recovery paths: add a cron/task-flow stale-running force-cancel or recovery path, give `tasks maintenance --apply` an explicit Gateway-authoritative or force mode for stale cron rows, and make restart/update preflight stale tasks, exclude proven-stale records from drain counts, or enforce a safe bounded timeout by default. Treat #60299 as a partial fix, not the canonical closure for this broader issue.
