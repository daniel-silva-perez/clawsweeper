## Summary

Keep this issue open. Current main has partially addressed the stale-task class by adding Gateway-authoritative cron reconciliation and a bounded default restart deferral, but ordinary cron task cancellation still falls through the unsupported runtime path and offline `tasks maintenance --apply` intentionally remains non-authoritative for cron liveness.

## What This Changes

The issue requests operator recovery for zombie `running` tasks through `tasks cancel`, task maintenance reconciliation/pruning, TaskFlow cancellation, and restart/update handling.

## Best Possible Solution

Keep the issue open and define a safe operator recovery path: Gateway-authoritative maintenance should reconcile or cancel proven-stale cron tasks, the CLI should expose an explicit force/Gateway-backed mode for offline recovery, TaskFlow cancel should not deadlock on unrecoverable children, and restart/update paths should continue to avoid unbounded waits on stale active-work accounting.

## Reproduction Assessment

Yes. A source-backed reproduction exists: a stale cron task with `cronRuntimeAuthoritative: false` remains `running` under maintenance, and a cron task with a child session still reaches the unsupported cancel branch in `cancelTaskById`.

## Solution Assessment

No. Current main contains good partial fixes for Gateway-owned cron reconciliation and restart deferral, but it is not yet the complete fix because cron cancellation and a deliberate offline force/recovery path are still missing.
