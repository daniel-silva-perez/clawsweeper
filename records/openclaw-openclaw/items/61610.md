## Summary

Keep this issue open. Current main has partial fixes for stale task maintenance, especially the narrower #60299-style cron/CLI reconciliation cases, but the central operator recovery problem is not fully resolved: cron task cancellation still falls through to the unsupported-runtime path, offline `tasks maintenance --apply` is explicitly not authoritative for cron active-job liveness, and restart/reload deferral still waits indefinitely by default unless an operator configures a positive timeout. The April 28 comment also reports a remaining restart/update failure mode on v2026.4.26, so this is not obsolete.

## Best Possible Solution

Keep this issue open and narrow the next maintainer fix to the remaining operator recovery paths: add an explicit cron/task-flow stale-running recovery or force-cancel path, make `tasks maintenance --apply` provide an actionable Gateway-authoritative/force mode for stale cron rows, and make restart/update either preflight task audit/maintenance, exclude proven stale records from drain counts, or enforce a safe bounded timeout by default. Treat the #60299 work as a partial fix, not the canonical closure for this broader report.

## Risks / Open Questions

- Closing now would lose the remaining cron-specific cancellation gap and the operator-facing restart/update wedge described in the newest comment.

- Related #48488 remains open for the broader lane timeout class; this issue still adds a distinct task-ledger and restart recovery surface.
