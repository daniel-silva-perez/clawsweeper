## Summary

Keep this open. Current main has partial safeguards for accidental empty completions and timeout-without-payload cases, but it does not implement the requested always-visible fallback or `dispatch.requireReply` behavior, and the exact `queuedFinal=false, replies=0` outcome is still a documented result for intentional silent replies and silent-allowed group/channel turns.

## What This Changes

The issue asks OpenClaw to send a user-visible fallback when an agent run completes or times out without producing replyable content.

## Best Possible Solution

Define an explicit require-reply policy, likely as an opt-in dispatch or surface setting, that overrides silent delivery only where maintainers want that behavior. It should preserve intentional `NO_REPLY` for group/channel/internal workflows by default, keep the existing timeout and empty-direct safeguards, and avoid duplicate replies when messaging tools or block streaming already delivered content.
