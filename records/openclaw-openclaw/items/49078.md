## Summary

Keep this issue open. Current main still has the cron delivery model described in the prior Codex review: a single `delivery` object with `none`, `announce`, or `webhook`, plus automatic channel fallback and limited main-session awareness after successful isolated delivery. It does not implement deterministic cron `output.targets` for file writes, multiple output targets, or explicit main/named-session result injection independent of prompt compliance.

## Best Possible Solution

Keep the issue open and design an additive core cron output contract. The best path is either a versioned extension of `delivery` or a new `output.targets` field that covers validated file append/write targets, channel/webhook delivery, explicit main or named-session injection, silent/discard mode, per-target status in run logs, mixed success/failure behavior, CLI and Control UI editing, docs, and focused protocol/runtime tests. If maintainers prefer plugin ownership later, first add a core plugin API that can receive per-job output target config and deterministically route final cron responses.

## Risks / Open Questions

- A future implementation needs strict workspace/path confinement, append/overwrite semantics, and redaction/logging rules before adding file output targets.
- The existing `delivery` status/run-log semantics already cover part of the problem; adding a parallel `output` path could create inconsistent success/failure reporting unless target fan-out status is modeled explicitly.
- Session injection can affect prompt context and privacy boundaries, so main/named-session targets should be additive, auditable, and clearly scoped to the owning agent/session.
