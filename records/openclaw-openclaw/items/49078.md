## Summary

Keep this issue open. Current main still exposes cron output as a single `delivery` object with `none`, `announce`, or `webhook`; it does not have deterministic `output.targets` for file writes, multi-target fan-out, or explicit main/named-session result injection independent of prompt compliance.

## Best Possible Solution

Keep the issue open and design an additive core cron output contract. The best path is either a versioned extension of `delivery` or a new `output.targets` field covering validated file append/write targets, channel/webhook delivery, explicit main or named-session injection, silent/discard mode, per-target status in run logs, mixed success/failure behavior, CLI and Control UI editing, docs, and focused protocol/runtime tests. If maintainers prefer plugin ownership later, first add a core plugin API that can receive per-job output target config and deterministically route final cron responses.

## Risks / Open Questions

- Adding file output targets needs strict workspace/path confinement, append/overwrite semantics, and redaction/logging rules.
- Adding a parallel `output` path beside existing `delivery` could create inconsistent run success/failure reporting unless per-target status and mixed-success behavior are modeled explicitly.
- Session injection affects prompt context and privacy boundaries, so main/named-session targets should be explicit, auditable, and scoped to the owning agent/session.
