## Summary

Keep this issue open. Current main only supports the existing single-runner delivery model (`announce`, `webhook`, or `none`) with a single chat/webhook target and delivery diagnostics. It does not implement the requested deterministic cron `output.targets` contract for file writes, multiple targets, or explicit main/named-session injection independent of prompt compliance. This also does not fit a conservative `clawhub` close: VISION prefers plugins for optional capabilities, but the requested behavior is a cron schema/protocol/runtime delivery contract, and current plugin hooks expose cron change summaries/status rather than a per-job output-target API that can guarantee final response routing.

## Best Possible Solution

Keep the issue open and design an additive cron output contract in the core cron/Gateway protocol: either add an `output` field or extend `delivery` into a versioned multi-target result-routing model. The implementation should cover schema/types, Gateway validation, runtime fan-out after final response capture, per-target status/run-log metadata, CLI and Control UI creation/editing, docs, and tests for file append/write, channel delivery, session/main injection, silent/discard mode, and mixed success/failure handling. If maintainers prefer plugin ownership later, first add a core plugin API that can safely receive per-job output target config and route final cron responses deterministically.

## Risks / Open Questions

- The current announce/webhook/none delivery behavior already covers part of the problem, so a fix should extend it carefully instead of creating a parallel delivery path with inconsistent status/error semantics.
- Adding file output targets needs explicit workspace/path confinement, append/overwrite rules, per-target failure status, and clear behavior when one target succeeds and another fails.
