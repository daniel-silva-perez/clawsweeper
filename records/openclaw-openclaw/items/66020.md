## Summary

Keep open. Current main still rejects existing local `<qqmedia>` paths outside the OpenClaw media roots instead of staging or mirroring them into QQ Bot media storage. The security objection in the discussion is valid, so this needs a constrained QQ Bot-owned staging design rather than a broad copy-any-path change.

## Best Possible Solution

Keep this open and implement a QQ Bot-owned constrained staging path: pass or reuse agent-scoped `mediaAccess` / `mediaLocalRoots` for `<qqmedia>` delivery, realpath-check candidates against workspace/media allowlists, copy only allowed files into managed OpenClaw/QQ Bot outbound media storage, and preserve existing size, type, symlink, and storage-boundary checks. Update QQ Bot tests and docs/skill guidance for allowed workspace files and rejected outside-root files.

## Risks / Open Questions

- A naive mirror-any-existing-local-path fix could let a model or remote user send arbitrary host files through QQ Bot.
- Fixing only generic reply payload media staging will not cover `<qqmedia>` tags because QQ Bot parses and sends those tags before the plain reply/media-normalizer path.
- Current tests assert outside-root rejection, so a fix needs deliberate allowlist/staging coverage for workspace files, outside-root files, symlink escapes, and managed media roots.
