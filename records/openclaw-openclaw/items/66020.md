## Summary

Keep open. Current main still rejects existing local `<qqmedia>` files outside OpenClaw/QQ Bot media roots, and the discussion correctly flags that a blanket auto-mirror could disclose arbitrary host files.

## What This Changes

Request that QQ Bot outbound `<qqmedia>` automatically stage existing local files from outside `~/.openclaw/media/qqbot` into managed QQ Bot media storage before sending.

## Best Possible Solution

Keep this open for a QQ Bot-owned secure staging design: accept only files proven to be inside agent-scoped workspace/media allowlists, realpath-check them, enforce size/type limits, copy them into managed QQ Bot outbound media storage, keep rejecting arbitrary outside-root or symlink-escape paths, and update QQ Bot tests plus `qqbot-media` guidance.

## Risks / Open Questions

- The fix needs explicit agent-scoped allowlists, realpath and symlink-escape checks, size/type limits, and cleanup semantics before automation should implement it.
- Current skill guidance still works around the bug by telling agents to pre-copy files, so the user-facing workflow remains brittle.
