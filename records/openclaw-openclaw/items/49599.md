## Summary

Keep this issue open. Current main does not implement custom HTTP headers for CLI Gateway WebSocket connections: the CLI options expose URL/token/password/timeout only, the runtime only forwards URL/token/password into callGateway, CallGateway/GatewayClient have no header field, and the ws ClientOptions are constructed without headers. The related implementation PR #49646 was closed unmerged, so it is useful context but not a canonical shipped fix.

## Best Possible Solution

Implement the feature in core CLI/Gateway code: add a repeatable `--header` option plus `OPENCLAW_GATEWAY_HEADERS` and `gateway.remote.headers` config support with explicit precedence, thread a sanitized `Record<string,string>` through `GatewayRpcOpts` -> `CallGatewayBaseOptions` -> `GatewayClientOptions`, set `ClientOptions.headers` before `new WebSocket`, update the strict config schema/types/help/docs, and add focused tests for parsing, precedence, WebSocket upgrade delivery, and secret redaction.

## Risks / Open Questions

- This touches the Gateway control-plane authentication path; header values may be secrets and should be parsed, stored, redacted, and excluded from logs with the same care as tokens/passwords.
- A complete fix should cover all CLI Gateway call surfaces consistently, not only `openclaw gateway call`, because many user-facing CLI commands use `callGatewayFromCli`.
