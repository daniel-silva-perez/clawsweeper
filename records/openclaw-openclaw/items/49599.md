## Summary

Current main still does not implement custom HTTP headers for CLI Gateway WebSocket connections. The request is concrete and belongs in OpenClaw core because the missing path crosses the shared Gateway CLI option layer, Gateway call/client options, WebSocket `ws` client construction, and the strict `gateway.remote` config contract. The related implementation PR #49646 was closed unmerged, so there is no shipped replacement to close against.

## Best Possible Solution

Keep this issue open. Implement the feature in the core Gateway CLI/client path: add a repeatable `--header` option plus `OPENCLAW_GATEWAY_HEADERS` and `gateway.remote.headers`, define explicit CLI > env > config precedence, validate and parse header names/values, treat values as sensitive, thread a `Record<string,string>` through `GatewayRpcOpts`/`CallGatewayBaseOptions`/`GatewayClientOptions`, merge it into `ws` `ClientOptions.headers`, update Gateway remote docs/config help, and add focused tests for parsing, precedence, WebSocket upgrade delivery, and redaction.
