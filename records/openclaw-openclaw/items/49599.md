## Summary

Keep this issue open. Current main still has no custom HTTP header path for CLI Gateway WebSocket connections: the CLI option types and command builders omit a header option, callGateway only forwards URL/auth/TLS/client metadata, GatewayClient constructs ws ClientOptions without headers, and the strict gateway.remote config schema has no headers field. The related implementation PR #49646 was closed unmerged, so it is useful context but not shipped behavior.

## Best Possible Solution

Implement this in the core Gateway CLI/client path: add a repeatable --header option plus OPENCLAW_GATEWAY_HEADERS and gateway.remote.headers with explicit CLI > env > config precedence, parse and validate header names/values, mark config values sensitive, thread a Record<string,string> through GatewayRpcOpts -> CallGatewayBaseOptions -> GatewayClientOptions, merge into ws ClientOptions.headers before new WebSocket, update gateway remote docs/config help, and add focused tests for parsing, precedence, upgrade delivery, and secret redaction.
