## Summary

Keep open. Current main still lacks the stricter pre-write path guidance and exact audit remediation command, but this PR is not merge-ready because the proposed schema walker rejects valid nested config paths behind current SecretRef combinators and plugin `$ref` schemas; it also needs rebase onto current main's config runner and test layout.

## What This Changes

The PR adds schema-backed path validation to `openclaw config set`, workspaceOnly suggestions/remediation text, and related CLI, config, security, test-environment, and secret-registry coverage.

## Best Possible Solution

Keep the useful direction, rebase it onto current main, and make path validation follow the same JSON Schema semantics as runtime validation: recursive combinators, internal `$ref`/`$defs` resolution, arrays, object maps, and plugin/channel schemas before rejecting a path.

## Reproduction Assessment

Yes. Static source proof is enough: the PR rejects `models.providers.openai.apiKey.provider` through an `anyOf` branch containing nested `oneOf`, and it also rejects `$ref`-backed plugin secret paths such as `plugins.entries.feishu.config.appSecret.provider` before AJV-backed validation could accept a completed SecretRef object.

## Solution Assessment

No. The requested direction is narrow and useful, but this implementation duplicates only part of JSON Schema traversal; recursive combinator and `$ref` handling should be fixed before merge, ideally through a shared schema-path helper rather than another partial walker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Traverse nested schema combinators:** `src/config/validate-config-path.ts:40-53`
  - body: `expandSchemaVariants` only includes direct `anyOf`/`oneOf`/`allOf` members. Current generated config schema represents SecretRef fields as `anyOf` with a nested `oneOf` of object shapes, so valid updates like `models.providers.openai.apiKey.provider` can be rejected as an unknown config path before normal config validation accepts them. Recursively flatten combinator wrappers before rejecting the path.
  - confidence: 0.88
- **[P2] Resolve `$ref` schemas before rejecting paths:** `src/config/validate-config-path.ts:99-103`
  - body: When a known property schema is only a `$ref`, `schemaAllowsPath` recurses into that wrapper and then treats it as non-traversable. Runtime plugin schemas such as Feishu use `$defs`/`$ref` for SecretRef inputs, so nested paths under those fields are rejected even though AJV validation resolves the reference. Resolve local refs, or delegate to a schema traversal that does, before returning `Unknown config path`.
  - confidence: 0.82
