## Summary

Keep this PR open, but not merge-ready. Current main still lacks the requested up-front config-path guidance and exact security-audit command, while the PR’s schema walker would reject valid nested SecretRef config paths because it only expands combinator wrappers one level.

## What This Changes

The PR adds pre-write `openclaw config set` path validation, suggests canonical `tools.fs.workspaceOnly` paths, updates the open-group audit remediation command, and adds related CLI, config, security-audit, and test-environment coverage.

## Best Possible Solution

Keep the PR open and revise the validator to use a schema traversal that fully expands nested combinators before declaring a path unknown, then keep the audit-command wording and path suggestions once the traversal matches the generated runtime schema.

## Reproduction Assessment

Yes. Against the PR head, a config containing an existing SecretRef object under a field such as `models.providers.<id>.apiKey` can be updated via a nested path today, but the proposed one-level combinator expansion rejects that path as unknown before the final config remains schema-valid.

## Solution Assessment

No. The PR’s direction is the right narrow fix for the user-facing bug, but the schema walker is not yet the best implementation because it duplicates partial schema traversal and misses nested combinator wrappers used by current generated schemas.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Traverse nested schema combinators:** `src/config/validate-config-path.ts:40-53`
  - body: `expandSchemaVariants` only includes direct `anyOf`/`oneOf`/`allOf` members. Current config schema represents SecretInput fields as `anyOf` with a nested `oneOf` of SecretRef object shapes, so a valid update to an existing nested field like `models.providers.openai.apiKey.provider` is rejected as an unknown config path before the normal config validation can accept it. Recursively flatten combinator wrappers, or reuse the existing schema lookup traversal, before rejecting the path.
  - confidence: 0.87
