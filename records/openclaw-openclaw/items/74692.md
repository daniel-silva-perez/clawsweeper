## Summary

Keep open. Current main still has the same packaging gap: builtin memory imports sqlite-vec through the memory host SDK, but sqlite-vec is neither declared by the memory-core plugin package nor included in the root runtime-dependency mirror list that feeds packaged plugin-runtime-deps snapshots.

## What This Changes

The issue requests that OpenClaw packaged installs stage sqlite-vec correctly for builtin memory/vector search after upgrading to OpenClaw 2026.4.27.

## Best Possible Solution

Update the packaged bundled-runtime-deps metadata so memory-core’s staged runtime contains sqlite-vec and its platform package, then add a regression test and a user-facing changelog fix credit.

## Repair Work Prompt

Fix the packaged builtin-memory sqlite-vec runtime dependency gap reported in https://github.com/openclaw/openclaw/issues/74692. In packaged installs, `memory-core` loads sqlite-vec through the memory host SDK, but sqlite-vec is not currently staged into `~/.openclaw/plugin-runtime-deps/.../node_modules`. Keep the fix within packaging/runtime-deps metadata and tests: either declare `sqlite-vec` as a memory-core runtime dependency or add it to the root mirrored runtime dependency path after verifying which plan correctly stages both `sqlite-vec` and its platform optional package. Do not edit node_modules, do not change the sqlite-vec version/platform pins without maintainer approval, and keep core/plugin boundaries generic. Add regression coverage in `src/plugins/bundled-runtime-deps.test.ts` proving the default memory slot/package-level runtime-deps plan includes sqlite-vec for packaged installs. Add a `CHANGELOG.md` fix entry in the active version with `Thanks @mozi1924`.
