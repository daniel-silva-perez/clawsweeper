## Summary

Keep open. Current main still implements the reported Discord behavior: a non-empty guild `channels` map makes an unlisted channel resolve to a denied channel config, and preflight then drops the message even when `groupPolicy` is `open`. The related reports are adjacent broader access-policy or dynamic matching requests, not a merged canonical replacement for this narrower semantics change.

## What This Changes

The issue requests Discord guild `channels` entries to behave as additive per-channel overrides over guild defaults instead of making all unlisted channels denied.

## Best Possible Solution

Keep this as bundled Discord plugin work: make unlisted channels inherit guild defaults only for open guild routing, preserve deny-by-default behavior for `groupPolicy: "allowlist"`, keep explicit `enabled: false` channel disables authoritative, and update source, docs, and focused regression tests across the shared Discord channel-access path.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Discord guild message in an unlisted channel with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map; current code resolves that channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear until maintainer compatibility review. The additive override model is a plausible narrow fix for open guild routing, but it should be constrained so existing allowlist and explicit-deny semantics do not loosen unexpectedly.
