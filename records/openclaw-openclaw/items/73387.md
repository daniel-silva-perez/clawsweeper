## Summary

Keep open. Current main still implements the implicit Discord guild channel allowlist that the report describes: a non-empty guild `channels` map resolves an unlisted channel to `allowed: false`, and preflight drops that message before guild-level mention/default handling. The related reports are adjacent broader access-policy or pattern-matching requests, not a merged canonical replacement for this narrower behavior change.

## What This Changes

The issue requests Discord guild `channels` entries to act as additive per-channel overrides so unlisted channels inherit guild-level defaults instead of being denied when guild routing is open.

## Best Possible Solution

Keep this as bundled Discord plugin work: make unlisted channels inherit guild defaults for the intended open-guild override use case, preserve explicit channel disables and allowlist-mode denial semantics, and update Discord docs plus focused regression tests for listed overrides, unlisted inheritance, and explicit denial.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear until maintainer compatibility review. The additive override model is a plausible narrow fix for open guild routing, but it should be guarded so existing allowlist and explicit-deny semantics do not loosen unexpectedly.
