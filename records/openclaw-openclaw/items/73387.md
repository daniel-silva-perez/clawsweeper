## Summary

Keep this issue open. Current main still treats an unmatched Discord guild channel entry as `{ allowed: false }`, and the active message preflight path then drops that guild message even under `channels.discord.groupPolicy: "open"`. The related open items are broader or adjacent, not a clean canonical replacement for this narrower additive-override semantic change.

## Best Possible Solution

Keep open for bundled Discord plugin work. Update Discord channel-policy resolution so `guilds.*.channels` entries act as per-channel overrides when `channels.discord.groupPolicy` is `"open"`, with unmatched channels inheriting guild-level defaults such as `requireMention`, sender restrictions, tools, skills, and prompts. Preserve current allowlist semantics for `groupPolicy: "allowlist"`, keep explicit disabled/deny entries blocking, document precedence, and add regression tests for listed overrides, unlisted inherited channels, and explicit channel denial.

## Risks / Open Questions

- Changing this can widen Discord guild access for existing configs that relied on `guilds.*.channels` as an implicit allowlist, so maintainers need an explicit compatibility/migration decision.
- The implementation needs to distinguish an unmatched channel that should inherit guild defaults from an explicit deny such as `enabled: false`, and preserve current deny-by-default behavior for `groupPolicy: "allowlist"`.
