## Summary

Keep open. Current main still implements the Discord guild `channels` block as an implicit channel allowlist: unmatched configured guild channels resolve to denied and the message preflight gate drops them, while docs and tests still describe and protect that behavior. The related reports are broader or adjacent, so none is a canonical replacement for this narrower config-semantics change.

## What This Changes

The issue requests Discord guild `channels` entries to behave as additive per-channel overrides under guild defaults instead of causing all unlisted channels to be denied.

## Best Possible Solution

Update the bundled Discord plugin policy so unlisted guild channels inherit guild-level defaults only when the effective Discord `groupPolicy` is `open`, while preserving channel allowlist semantics for `groupPolicy: "allowlist"`, honoring explicit channel disables, and documenting/tests-covering precedence for guild defaults, channel overrides, wildcard entries, parent-thread fallback, sender restrictions, skills, tools, and prompts.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure Discord with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that does not include the incoming channel; current resolver returns `{ allowed: false }` and preflight drops the message.

## Solution Assessment

Unclear until a maintainer makes the compatibility call. The requested additive override model is a plausible narrow fix for `groupPolicy: "open"`, but it changes access-control semantics for existing Discord configs, so the safest solution needs explicit migration and release-note handling.
