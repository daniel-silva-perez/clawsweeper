## Summary

Keep open. Current main still implements the implicit Discord guild channel allowlist: a non-empty `guilds.*.channels` map yields a fallback denied channel config for unlisted channels, and preflight drops that message even when the broader guild policy is `open`. The related reports are adjacent broader access-policy requests, not a merged or canonical replacement for this narrower behavior change.

## What This Changes

The issue requests Discord guild `channels` entries to act as additive per-channel overrides so unlisted channels inherit guild-level defaults under open guild routing.

## Best Possible Solution

Keep this as bundled Discord plugin work: make unlisted channels inherit guild defaults only when Discord guild routing is `groupPolicy: "open"`, preserve deny-by-default behavior for `groupPolicy: "allowlist"`, keep explicit channel disables authoritative, and update docs plus focused regression tests for listed overrides, unlisted inheritance, and explicit denial.

## Reproduction Assessment

Yes. A high-confidence reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current source resolves that channel to `{ allowed: false }` and preflight drops it before mention handling.

## Solution Assessment

Unclear until maintainer compatibility review. The requested additive override model is a plausible narrow fix for open guild routing, but it should be guarded so existing allowlist and explicit-deny semantics do not loosen unexpectedly.
