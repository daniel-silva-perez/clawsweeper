## Summary

Keep open. Current main still implements and documents the Discord guild channel `channels` map as an implicit allowlist, so the reported open-guild override use case remains blocked; the related items are broader or unmerged and do not supersede this narrower behavior change.

## What This Changes

The issue asks for Discord guild `channels` entries to act as additive per-channel overrides so unlisted channels inherit guild defaults instead of being denied.

## Best Possible Solution

Keep this as bundled Discord plugin work: make unlisted channels inherit guild defaults for `groupPolicy: "open"` additive override use cases, preserve explicit `enabled: false` channel disables and `groupPolicy: "allowlist"` denial semantics, and update Discord docs plus regression tests across message preflight and adjacent auth surfaces.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the channel to `{ allowed: false }` and preflight drops the message.

## Solution Assessment

Unclear until maintainer compatibility review. The additive override model is a plausible narrow fix for open guild routing, but the safest implementation should preserve explicit-deny and allowlist-mode behavior instead of globally removing the fallback deny.
