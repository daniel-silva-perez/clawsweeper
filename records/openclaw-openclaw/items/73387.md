## Summary

Keep open. Current main still treats a non-empty Discord guild `channels` map as an implicit channel allowlist: unmatched guild channels resolve to a denied config and preflight drops the message before mention handling. The related items cover broader Discord access-list redesign or dynamic channel matching, but neither is a merged canonical replacement for this narrower compatibility-sensitive behavior change.

## What This Changes

The issue requests Discord guild `channels` entries to act as additive per-channel overrides over guild defaults instead of denying unlisted channels.

## Best Possible Solution

Keep this as bundled Discord plugin work: make unlisted channels inherit guild defaults only for intentionally open guild routing, preserve deny-by-default behavior under `groupPolicy: "allowlist"`, keep explicit channel disables authoritative, and update docs plus regression tests for listed overrides, unlisted inheritance, and explicit denial.

## Reproduction Assessment

Yes. A high-confidence reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current source resolves that channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear as written. The additive override model is a good fit for `groupPolicy: "open"`, but applying it universally would weaken existing allowlist semantics; the safer solution is policy-aware inheritance with explicit deny/disable preserved.
