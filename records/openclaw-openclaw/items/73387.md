## Summary

Keep this issue open. Current main still implements the documented implicit allowlist behavior for Discord guild `channels`: when a guild has a non-empty `channels` map and the incoming channel does not match, the resolver returns `{ allowed: false }`, and the message preflight path drops that guild message even if the broader Discord `groupPolicy` is `open`. The related issues in the provided context are broader or adjacent, not a direct canonical replacement for this narrower semantic change.

## Best Possible Solution

Keep open for bundled Discord plugin work. Update Discord channel-policy resolution so `guilds.*.channels` entries act as per-channel overrides when `channels.discord.groupPolicy` is `"open"`, with unlisted channels inheriting guild-level defaults such as `requireMention`, sender restrictions, tools, skills, and prompts. Preserve current allowlist semantics for `groupPolicy: "allowlist"`, keep explicit disabled channel entries blocking, document precedence, and add regression tests for listed overrides, unlisted inherited channels, and explicit channel denial.

## Risks / Open Questions

- Changing this behavior can widen Discord guild access for existing configs that rely on `guilds.*.channels` as an implicit allowlist, so maintainers need an explicit compatibility and migration decision.
- The implementation must distinguish unmatched channels that should inherit guild defaults from explicit channel denial such as `enabled: false`, and it should preserve deny-by-default behavior for `groupPolicy: "allowlist"`.
