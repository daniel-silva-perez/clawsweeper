## Summary

Keep this issue open. Current main still treats a non-empty Discord `guilds.*.channels` map as a denial source for unlisted channels in the active preflight path, so the requested additive-override behavior is not implemented. The related open items are overlapping but broader or adjacent, not a clear superseding tracker for this narrower semantic change.

## Best Possible Solution

Keep open for bundled Discord plugin work. Update the Discord preflight/channel-policy semantics so `guilds.*.channels` entries are per-channel overrides when `groupPolicy: "open"`, unmatched channels inherit guild-level `requireMention`, tools, sender policy, and other defaults, and explicit deny/disabled entries still block. Preserve current allowlist behavior for `groupPolicy: "allowlist"`, document the precedence, and add regression tests for listed override channels, unlisted inherited channels, and explicit channel denial.

## Risks / Open Questions

- Changing this behavior can widen Discord guild access for existing configs that relied on `guilds.*.channels` as an implicit allowlist, so the implementation needs an explicit migration/compatibility decision and tests for `groupPolicy: "allowlist"` versus `"open"`.
- The fix should preserve a clear way to deny specific channels, such as `enabled: false`, while allowing unlisted channels to inherit guild-level defaults only in the intended policy mode.
