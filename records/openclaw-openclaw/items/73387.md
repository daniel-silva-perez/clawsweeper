## Summary

Keep open. Current main still implements the reported Discord behavior: once a guild has a non-empty `channels` map, an unlisted channel resolves to a fallback denied config and guild preflight/native/component/reaction paths treat that as a hard block, even when the broader guild policy is `open`. The related reports cover broader or adjacent access-policy work rather than a merged replacement for this narrower semantics change.

## What This Changes

Request Discord guild `channels` entries to behave as additive per-channel overrides so unlisted channels inherit guild defaults instead of becoming implicitly denied.

## Best Possible Solution

Keep this as bundled Discord plugin work: make unlisted channels inherit guild-level defaults when the guild is open, preserve allowlist-mode and explicit channel-disable behavior, and update Discord docs plus focused tests for listed overrides, unlisted inheritance, explicit denial, native commands, and reaction/component access where the shared resolver is used.

## Reproduction Assessment

Yes. Configure Discord with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the omitted channel to `{ allowed: false }` and preflight drops it before mention handling.

## Solution Assessment

Unclear until maintainer compatibility review. The additive override model is a plausible narrow fix for open guild routing, but the implementation should explicitly preserve current restrictive semantics for allowlist mode and explicit channel disables.
