## Summary

Keep open. Current main still does not implement the requested Qwen cache marker path, but this active PR should not merge as-is because it extends unconditional provider-side cache markers to a new OpenRouter model family without honoring the documented cacheRetention opt-out.

## What This Changes

The PR adds a closed-source Qwen model matcher and expands the OpenRouter system-cache wrapper so selected Qwen model IDs receive Anthropic-style cache_control markers, with unit coverage for included and excluded IDs.

## Best Possible Solution

Make Qwen cache marker injection retention-aware, verify it live against at least one matched Qwen commercial model with cache write/read usage, and update prompt-caching docs plus OpenRouter cache-TTL metadata if Qwen is meant to be a first-class cache-eligible OpenRouter family.

## Reproduction Assessment

Yes. Current main behavior is reproducible by the wrapper tests and code inspection: non-Anthropic OpenRouter models are left unchanged. The PR regression is reproducible by configuring cacheRetention: "none" for a matched Qwen model because the proposed wrapper path has no retention gate.

## Solution Assessment

No as proposed. The matcher is narrow, but the maintainable fix should either rebase on the cacheRetention path from #42961 or add an equivalent Qwen-aware opt-out before enabling cache_control markers.

## Review Findings

## Security Review

Status: needs_attention

Summary: The diff extends provider-side prompt caching to Qwen without honoring an explicit cacheRetention opt-out, which is a concrete data-retention/privacy concern.

Concerns:
