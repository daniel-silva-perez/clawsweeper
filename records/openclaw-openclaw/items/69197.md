## Summary

Keep this PR open. Current main and the latest release still lack an active-poll suppression guard, and the latest PR revision usefully replaced the earlier boolean with a refcount, but the patch still has a concrete abort race that can drop a configured completion notification when a poll is cancelled after the exec exits but before the poll consumes the exit.

## What This Changes

The PR adds a `ProcessSession.pollActiveCount` refcount, uses it to suppress `notifyOnExit` while `process poll` is active, marks exits consumed by poll/kill as notified, and adds focused poll ghost regression tests.

## Best Possible Solution

Amend the PR so active-poll suppression only replaces the notification when a poll actually consumes the exit, or hands the completion back to the normal notification path when the poll aborts before consumption. Keep the refcount design, add an abort-race regression test, run the targeted agent tests plus the changed gate, and include a user-facing changelog credit if this ships as the fix.

- **[P2] Preserve notifications when an active poll aborts:** `src/agents/bash-tools.exec-runtime.ts:277`
  - body: The new guard suppresses `maybeNotifyOnExit()` whenever `pollActiveCount > 0`, but `process poll` only sets `exitNotified` after the wait loop reaches the exited branch. If the poll's `AbortSignal` fires after the process exits but before the 250 ms sleep wakes, `sleepPollInterval()` rejects, `finally` only decrements the count, and no poll result or `notifyOnExit` event is delivered. Add a handoff/retry path or consume the exit before rethrowing.
