## Summary

Current main and the latest release still lack the PR's active-poll suppression for background exec completion notifications. The PR remains a plausible focused implementation candidate, but it should stay open for maintainer review because the current boolean `pollActive` design has an overlapping-poll race called out in review and the added tests do not fully prove the actual system-event/heartbeat suppression path.

## Best Possible Solution

Keep this PR open as the active implementation candidate. The best path is to amend or explicitly justify the active-poll design, preferably with an in-flight poll counter/token unless maintainers decide and test that same-session process polls are serialized. The final patch should preserve normal `notifyOnExit` behavior for unpolled background exits, suppress the active-poll ghost, keep timeout/later-exit behavior intact, and add direct regression coverage for overlapping polls or a proven serialization invariant.

## Risks / Open Questions

- A single `pollActive` boolean can be cleared by the first finishing poll while another long poll is still active, reopening the same notification race.
- The new PR tests mostly assert session flags; the final fix should include a regression that observes the actual `enqueueSystemEvent`/heartbeat suppression path or a realistic exec/process integration harness.
- Exec completion notifications are an observability boundary. A bad fix could either keep leaking ghost `Exec completed` messages or suppress legitimate background completion/failure events.
- The linked broader async-routing reports such as #25592 and #52305 are larger than this active-poll race, so merging this PR should not be treated as a complete fix for every related channel-leak or completion-routing issue.
