## Summary

Current main and the latest release still lack the PR's intended active-poll suppression: `maybeNotifyOnExit()` only checks `backgrounded`, `notifyOnExit`, and `exitNotified`, while `process` poll waits on a 250ms timer without marking the notification consumed before the exec completion promise can fire. The PR is a plausible focused fix for the ghost exec-notification behavior linked to #25592/#66135, but it should stay open for maintainer review because the proposed boolean `pollActive` approach has an overlapping-poll race called out in review and the current tests show process polls are not structurally serialized.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate. The best path is to amend or confirm the active-poll design, preferably using an in-flight poll counter/token rather than a boolean unless maintainers explicitly document and enforce single active poll per session. The final PR should keep normal notifyOnExit behavior for unpolled background exits, suppress the active-poll ghost, cover poll timeout/later-exit behavior, and include regression tests for overlapping polls or a tested serialization invariant.

## Risks / Open Questions

- The PR changes exec lifecycle notification behavior, so a bad fix could either keep leaking ghost `Exec completed` messages or suppress legitimate background completion events.
- The proposed boolean `pollActive` state is racy for overlapping long polls; a counter/token would keep suppression active until the last poll finishes.
- The linked broader issues include async completion routing problems beyond this runtime race, so merging this PR should not be treated as a complete fix for every related notification-routing complaint.
