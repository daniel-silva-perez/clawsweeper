## Summary

Keep open. Current main still resolves pending system-event entries from the base heartbeat session before isolation, creates the `:heartbeat` run session without copying or aliasing those entries, and then active-session inspection reads only the isolated key. The issue remains a real contract gap, although the exact transfer scope needs maintainer judgment because current tests intentionally keep base-session untrusted events from affecting isolated-run ownership flags.

## What This Changes

The issue requests carrying pending system-event entries from a base heartbeat session into the isolated `:heartbeat` run session, or otherwise making pre-isolation events visible to isolated-session consumers.

## Best Possible Solution

Add a narrow, documented heartbeat-boundary transfer or alias for the relevant pre-isolation system-event entries so isolated-session consumers can observe them exactly once, while preserving base-queue consumption order, duplicate/drain semantics, and the intentional stale routing/trust protections.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that consumers inspecting the isolated `:heartbeat` key still see an empty queue on current main; the existing ghost-reminder tests also show the trust-flag edge currently stays false for base-only untrusted events.

## Solution Assessment

Yes in direction, but the implementation scope needs maintainer confirmation. A scoped transfer-or-alias at isolated heartbeat creation is narrower than rewriting all event consumption, but it must explicitly decide how wake-triggered and untrusted base events should behave.
