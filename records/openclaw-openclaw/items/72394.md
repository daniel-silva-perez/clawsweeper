## Summary

Keep open. Current main still creates the isolated heartbeat session without transferring pending base-session system-event entries into that active session; adjacent tests document the current asymmetry rather than proving it fixed. The related #73297 routing work is adjacent, but it addresses stale delivery routing and does not close this active-session event-queue contract gap.

## What This Changes

The issue asks OpenClaw to carry pending base-session system-event entries into or through an isolated heartbeat session so active-session event consumers do not see an empty queue.

## Best Possible Solution

The desired end state is a stable active-session system-event contract for isolated heartbeats: consumers of `peekSystemEventEntries(runSessionKey)` should see the relevant pre-isolation entries exactly once without restoring stale base-session delivery routing or replaying consumed events.

## Reproduction Assessment

Yes. A focused reproduction is to enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and observe that the model context uses the `:heartbeat` session while active-session event inspection reads only that isolated key.

## Solution Assessment

Yes, a narrow transfer-or-alias seam around isolated heartbeat session creation looks like the best fix, provided it preserves the current delivery-target guard and exactly-once consumption. A broad rewrite of system-event consumption would have a larger blast radius than this issue needs.
