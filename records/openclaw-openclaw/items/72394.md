## Summary

Keep open. Current main still peeks pending system events on the base heartbeat session before isolation, creates the isolated `:heartbeat` run session without copying or aliasing the event queue, and then active-session consumers inspect only the isolated key. The issue remains a real contract gap, with the main unresolved point being the trust/routing semantics of any transfer.

## What This Changes

The issue requests transferring or otherwise exposing pending system event entries queued on the base heartbeat session to consumers running under an isolated `:heartbeat` heartbeat session.

## Best Possible Solution

Add a narrow, documented heartbeat-boundary transfer or alias for the relevant pre-isolation system-event entries so isolated-session consumers can observe them exactly once, while explicitly preserving base-queue consumption order, duplicate handling, delivery-context routing, and the intended trust-boundary behavior.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the isolated `:heartbeat` key remains empty on current main; I did not run it because this review is read-only.

## Solution Assessment

Unclear. A scoped transfer-or-alias at isolated heartbeat creation looks narrower than rewriting all event consumers, but maintainers should first decide how wake-triggered and untrusted base events are supposed to affect isolated-session inspection.
