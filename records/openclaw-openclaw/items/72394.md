## Summary

Keep open. Current main still reads pending system-event entries from the base heartbeat session before isolation, creates the isolated `:heartbeat` session without copying or aliasing the queue, and then reads active-session events from the isolated key. The linked Telegram routing report was closed as a distinct not-reproducible issue, and the related open routing PRs do not make this contract implemented on main.

## What This Changes

The issue requests that pending system-event entries queued on the base heartbeat session remain visible across the isolated `:heartbeat` model-run boundary.

## Best Possible Solution

Add a narrow heartbeat-boundary transfer or alias for the relevant pre-isolation system-event entries so active isolated-session consumers can observe them exactly once, while preserving base-queue consumption, duplicate/drain behavior, and the stale delivery-routing guard.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and verify that the model context uses the isolated `:heartbeat` key while active-session event inspection currently sees no copied base entries.

## Solution Assessment

Yes. A scoped transfer-or-alias seam at isolated heartbeat creation is the narrowest maintainable fix; a larger rewrite of all event consumption would touch more routing and delivery behavior than this contract-hardening issue needs.

## Repair Work Prompt

Fix the isolated heartbeat system-event boundary described in https://github.com/openclaw/openclaw/issues/72394. Current main peeks pending system-event entries from the base session before `heartbeat.isolatedSession` switches the model run to `<base>:heartbeat`; active-session consumers then call `peekSystemEventEntries(runSessionKey)` and see an empty isolated queue. Add a narrow transfer or alias seam so the relevant pre-isolation entries are visible through the active isolated run key exactly once, while preserving base-session consumption semantics and existing stale delivery-routing guards. Do not broaden delivery routing, do not reintroduce base-session `turnSource` routing for isolated wake runs, and do not change unrelated heartbeat/session behavior. Coordinate with PR #72814 if it has landed or conflicts, because it also changes heartbeat inspection-session routing. Add focused regression coverage in `src/infra/heartbeat-runner.isolated-key-stability.test.ts` or `src/infra/heartbeat-runner.ghost-reminder.test.ts`, plus `src/infra/system-events.test.ts` if a queue helper is added. Changelog is probably not needed unless the fix changes user-visible heartbeat behavior.
