## Summary

Keep open. Current main still peeks pending system events from the base heartbeat session before isolation, creates or reuses the isolated `:heartbeat` run session without copying or aliasing those events, and later inspects pending events by the isolated run key. The gap is real, but the transfer scope needs maintainer judgment because existing tests intentionally keep some base-session hook and untrusted events from changing isolated-run classification or ownership flags.

## What This Changes

Transfer or alias pending system event entries queued on the base heartbeat session so consumers inspecting the isolated `:heartbeat` run session can see the relevant events.

## Best Possible Solution

Add a narrow, documented heartbeat-boundary transfer or alias for the relevant pre-isolation system-event entries so isolated-session consumers can observe them exactly once, while preserving base queue ordering, drain semantics, delivery routing, and the existing trust protections for untrusted base events.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that an isolated-key consumer still sees no copied entry on current main; the existing ghost-reminder tests also show the sensitive trust edge that must be preserved or intentionally changed.

## Solution Assessment

Unclear until maintainers choose the trust and routing scope. A scoped transfer-or-alias at isolated heartbeat session creation is likely narrower than rewriting every consumer, but it must explicitly decide how wake-triggered, cron, exec, delivery-context, and untrusted events cross the boundary.

## Risks / Open Questions

- A naive copy of all base-session events into the isolated queue could flip `ForceSenderIsOwnerFalse` for base-only untrusted events, contradicting current tests.
- A transfer implemented at the wrong time could duplicate or lose events because preflight consumes only the inspected prefix and later queued entries must remain intact.
- Delivery-context comments in the runner warn that reusing base-session event context for isolated runs can pin stale channel/thread routing, so the event transfer must define which metadata moves.
