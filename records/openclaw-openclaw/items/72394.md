## Summary

Keep open. Current main still peeks pending system-event entries from the base heartbeat session before isolation, creates the `:heartbeat` run session without copying or aliasing those entries, and then reads active-session pending entries only from the isolated key. The related Telegram routing report was closed as a distinct not-reproducible issue, and the active routing PRs are unmerged and do not make this contract implemented on main.

## What This Changes

The issue requests that pending system-event entries queued on the base heartbeat session remain visible when `heartbeat.isolatedSession` switches the model run to the isolated `:heartbeat` session.

## Best Possible Solution

Add a narrow heartbeat-boundary transfer or alias for relevant pre-isolation system-event entries so active isolated-session consumers can observe them exactly once, while preserving base-queue consumption, duplicate/drain behavior, and stale delivery-routing safeguards.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and verify that the model context uses the isolated `:heartbeat` key while active-session event inspection currently sees no copied base entries.

## Solution Assessment

Yes. The narrowest maintainable solution is a scoped transfer-or-alias seam at isolated heartbeat creation; rewriting all event consumption would touch more routing and delivery behavior than this contract-hardening issue needs.

## Repair Work Prompt

Fix the isolated heartbeat system-event boundary described in https://github.com/openclaw/openclaw/issues/72394. Current main peeks pending system-event entries from the base session before `heartbeat.isolatedSession` switches the model run to `<base>:heartbeat`; active-session consumers then call `peekSystemEventEntries(runSessionKey)` and see an empty isolated queue. Add a narrow transfer or alias seam so the relevant pre-isolation entries are visible through the active isolated run key exactly once, while preserving base-session consumption semantics, duplicate/drain behavior, and existing stale delivery-routing guards. Do not broaden delivery routing, do not reintroduce base-session `turnSource` routing for isolated wake runs, and do not change unrelated heartbeat/session behavior. Coordinate with PR #72814 if it has landed or conflicts, because it also changes heartbeat inspection-session routing. Add focused regression coverage in `src/infra/heartbeat-runner.isolated-key-stability.test.ts` or `src/infra/heartbeat-runner.ghost-reminder.test.ts`, plus `src/infra/system-events.test.ts` if a queue helper is added. Changelog is probably not needed unless the fix changes user-visible heartbeat behavior.
