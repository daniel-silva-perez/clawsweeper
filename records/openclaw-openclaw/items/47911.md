## Summary

Keep open. Current main has several observability building blocks, including status JSON, per-agent session summaries, model auth status, channel probes, logs, and service status, but it still does not provide the requested single `openclaw fleet status` surface or an equivalent unified JSON payload for fleet-wide auth, channel, session, service, stuck-session, and recent-error state.

## What This Changes

Request a new `openclaw fleet status` command with JSON output that combines multi-agent session/service state, provider auth health, channel health, stuck-session detection, and recent errors.

## Best Possible Solution

Define one supported read-only fleet observability schema that reuses the existing status, sessions, models/auth, channels, logs, and service collectors with explicit redaction guarantees. The final CLI surface could be `openclaw fleet status --json` or an expanded `openclaw status --all --json`, but it should avoid parallel duplicated logic.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The missing behavior is verifiable with high confidence by inspecting the current CLI catalog, docs, status schema, status JSON payload, and models/channels/logs surfaces, none of which provide the requested unified fleet command.

## Solution Assessment

Unclear until maintainers choose the product/API surface. The requested direction is coherent, but the best implementation needs a design pass around command naming, JSON schema stability, sensitive-output redaction, and reuse of existing collectors.
