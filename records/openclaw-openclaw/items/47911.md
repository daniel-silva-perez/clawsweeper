## Summary

Keep open. Current main still has separate status, health, sessions, models, channels, and logs surfaces, but no `openclaw fleet status` command or equivalent single JSON payload that combines multi-agent state, provider/auth health, channel probes, service/log state, stuck-session detection, and recent error aggregation. The related observability PR and signal proposal are closed and do not supersede the remaining request.

## What This Changes

Add a unified `openclaw fleet status` command and JSON output that aggregates multi-agent health, provider/auth state, channel health, stuck sessions, service state, and recent errors.

## Best Possible Solution

Define one supported read-only observability schema that composes the existing status, health, sessions, models/auth, channels, service, and logs inputs with explicit redaction guarantees. The maintainer-approved surface could be a new `openclaw fleet status --json` or an expanded `openclaw status --all --json`, but it should avoid duplicating logic across parallel commands.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The missing behavior is verifiable from current main by checking the CLI catalog/docs and the status JSON types, which show no fleet command or equivalent unified payload.

## Solution Assessment

Unclear until maintainers choose the product/API surface. The requested direction is coherent, but the best implementation likely needs a design pass around command naming, schema stability, and redaction before code is written.
