## Summary

Keep this PR open. Current main still hard-times out after `agent.wait` returns `timeout`, the proposed helper is not present on main, and the PR is paired with the still-open bug it says it fixes; this needs normal maintainer review rather than cleanup-close.

## What This Changes

The PR adds post-timeout compensation for `sessions_send` by re-reading the target session’s latest assistant reply after an `agent.wait` timeout, returning `ok` for a new late reply or `accepted` when the run was accepted but no new reply is visible yet, with focused regression tests.

## Best Possible Solution

Review this PR as the implementation candidate for the open `sessions_send` timeout bug, but make the timeout states explicit: a successful post-timeout reply should return `ok`, an accepted still-running run should return `accepted` without being confused with delivery failure, and the accepted-no-reply path should preserve the background announce/follow-up behavior expected for cross-session sends.

## Reproduction Assessment

Yes. On current main, a focused mock path where `agent` succeeds and `agent.wait` returns `timeout` reproduces the behavior: `sessions_send` returns a hard `timeout` immediately without a follow-up history read.

## Solution Assessment

Unclear as written. The compensation direction is narrow and matches the reported misclassification, but the accepted-no-reply branch should be reviewed for whether it must start the existing A2A announce flow with the run id before this is the best complete fix.
