## Summary

Keep open. Current main still has the durable restore failure mode this PR targets, but the PR branch remains incorrect because its rollback only restores `subagentRuns` and the retry latch while leaving `resumedRuns` markers from a failed restore attempt behind. A later retry can therefore skip restored run resume/cleanup work after the map rollback, matching the existing review finding and making this not ready to close or merge as-is.

## What This Changes

The PR snapshots `subagentRuns` before subagent registry restore, restores that map and resets `restoreAttempted` on restore exceptions, adds a restore-retry regression test, and hardens adjacent registry persistence mocks and cleanup.

## Best Possible Solution

Update the restore rollback to restore all in-memory restore/resume bookkeeping consistently, including `resumedRuns` for ids touched by a failed attempt, and add a regression case where restore fails after at least one resumed marker is recorded so the next init retries the run normally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Roll back resumed-run markers on restore failure:** `src/agents/subagent-registry.ts:815-819`
  - body: The catch block restores `subagentRuns` and resets `restoreAttempted`, but the try body can already call `resumeSubagentRun()`, which records ids in `resumedRuns`. If a later restore step throws, the next `initSubagentRegistry()` retry will restore the map but skip those ids through `resumedRuns.has(runId)`, so cleanup or wait recovery is not re-run. Snapshot or clear the touched `resumedRuns` entries as part of rollback and cover a failure after one resume attempt.
