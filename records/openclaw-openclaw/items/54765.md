## Summary

Keep open. Current main still has the durable restore failure path, and this PR targets the right latch/map poisoning problem, but the proposed rollback leaves `resumedRuns` mutated after a failed restore attempt, so a later init retry can still skip resumed runs and leave cleanup/wait recovery stuck.

## What This Changes

The PR changes subagent registry restore to snapshot and roll back `subagentRuns`, reset the restore latch on failure, adds a focused restore-retry test, and hardens shared registry test mocks and temp cleanup.

## Best Possible Solution

Land a narrow fix that treats restore as an atomic in-memory transaction: on failure, restore the run map, restore or clear the resumed-run set for ids touched by the failed attempt, reset the latch, and cover the partial-restore-then-retry path with a focused regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Roll back resumed run ids on restore failure:** `src/agents/subagent-registry.ts:815-820`
  - body: The new catch restores `subagentRuns` and the retry latch, but restore can already have called `resumeSubagentRun()` for earlier entries. Those paths add run ids to `resumedRuns`, and the next `initSubagentRegistry()` retry immediately returns for those ids, so their wait/cleanup resume never runs. Snapshot and restore or clear the affected `resumedRuns` entries alongside `subagentRuns`.
