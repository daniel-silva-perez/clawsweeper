## Summary

Keep open. Current main still has the one-shot restore latch with ignored restore failures, so the PR is addressing a real remaining recovery gap, but the proposed rollback is incomplete because it restores `subagentRuns` and the latch without undoing `resumedRuns` side effects from the same restore attempt.

## What This Changes

The PR snapshots the in-memory subagent registry before restore, rolls back `subagentRuns` and resets `restoreAttempted` on restore exceptions, and adds focused restore-retry coverage plus test mocks/cleanup hardening.

## Best Possible Solution

Revise the rollback so a failed restore attempt restores all in-memory restore/resume bookkeeping consistently, especially `resumedRuns` for ids touched during the failed pass, then add a regression case where failure happens after at least one run has been marked resumed and the next init retries that run normally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Roll back resumed run markers on restore failure:** `src/agents/subagent-registry.ts:815-819`
  - body: The catch block restores `subagentRuns` and resets `restoreAttempted`, but the try body also calls `resumeSubagentRun()` for restored entries. That function records ids in `resumedRuns`; if a later restore step throws, the next `initSubagentRegistry()` retry restores the map but skips those ids via `resumedRuns.has(runId)`, so cleanup/wait resume is never re-run. Please snapshot or clear the restored run ids in `resumedRuns` as part of rollback and cover a failure after one resume attempt.
  - confidence: 0.86
