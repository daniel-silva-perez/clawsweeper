## Summary

Keep open. The PR targets a real durable-restore bug on current main, but the proposed rollback is incomplete: restored runs can remain marked in `resumedRuns` after a failed restore pass, so the next init retry can skip the resume/cleanup path. The existing review thread already flags the same P2 blocker, and no current-main change supersedes the PR.

## What This Changes

The PR snapshots `subagentRuns` before durable restore, restores that map and resets `restoreAttempted` on exceptions, adds restore-retry coverage, and hardens related registry test mocks and cleanup.

## Best Possible Solution

Revise the rollback to restore all in-memory restore/resume bookkeeping consistently, especially `resumedRuns` for ids touched during the failed pass, and add a regression where failure occurs after at least one run is marked resumed and the next init retries that run normally.

## Reproduction Assessment

Yes. A focused unit test can mock two restored runs, let the first resume path add its id to `resumedRuns`, then force a later restore/resume step to throw; the next `initSubagentRegistry()` would retry disk restore but hit the `resumedRuns.has(runId)` guard. I did not run tests because this was a read-only review.

## Solution Assessment

No. Snapshotting only `subagentRuns` and resetting the latch fixes the map/latch part but not the resume bookkeeping; snapshotting/restoring `resumedRuns` or clearing only the touched restored ids is the narrower maintainable fix.

## Review Findings

- **[P2] Clear resumed-run markers on restore rollback:** `src/agents/subagent-registry.ts:815-819`
  - body: The catch block restores `subagentRuns` and resets `restoreAttempted`, but the try body can already have added restored ids to `resumedRuns`. If a later restore/resume step throws, the next init retry restores the map but `resumeSubagentRun()` returns at the `resumedRuns.has(runId)` guard, so wait/cleanup/orphan recovery is not re-registered for those runs.
