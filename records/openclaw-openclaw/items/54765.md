## Summary

Keep open. Current main still lacks the durable restore retry fix, and the PR branch has a concrete correctness issue: rollback restores the run map and latch but leaves resume bookkeeping dirty, so a retry after a partial restore can skip recovery for already marked run ids.

## What This Changes

The PR snapshots subagent runs before durable restore, restores that map and resets the restore latch on exceptions, adds focused restore-retry coverage, and hardens related registry test mocks and cleanup.

## Best Possible Solution

Revise the rollback so a failed restore attempt restores all in-memory restore/resume bookkeeping consistently, especially `resumedRuns` for ids touched during the failed pass, then add a regression where failure occurs after at least one run has been marked resumed and the next init retries that run normally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Roll back resumed-run markers on restore failure:** `src/agents/subagent-registry.ts:815-819`
  - body: The catch block restores `subagentRuns` and resets `restoreAttempted`, but the try body can already have called `resumeSubagentRun()`, which records ids in `resumedRuns`. If a later restore step throws, the next `initSubagentRegistry()` retry restores the map but skips those ids via `resumedRuns.has(runId)`, so cleanup or wait recovery is not re-run. Snapshot or clear the restored run ids in `resumedRuns` as part of rollback and cover a failure after one resume attempt.
  - confidence: 0.86
