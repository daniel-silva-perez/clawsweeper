## Summary

Keep open for maintainer review. Current main still has no stale-activity watchdog or clear-all path, so the PR is not obsolete, but the proposed patch should not merge as-is because the fixed 3-minute eviction can hide legitimate quiet long-running work and one new test asserts behavior that contradicts the eviction threshold. The author’s link to #60947 is useful context: this is a downstream macOS display-state symptom of missing terminal run events, distinct from the broader provider degraded-mode work.

## What This Changes

The PR adds a macOS WorkActivityStore watchdog that evicts stale job/tool display activity after three minutes, refreshes activity timestamps on streaming/tool events, exposes a clearAllActivities helper, and expands WorkActivityStore tests.

## Best Possible Solution

Add a macOS display-state recovery path that clears orphaned activities when the gateway/run lifecycle is known lost or explicitly stale, while preserving visible activity for legitimate quiet long-running jobs/tools. Keep the role-recompute fix, make the stale-state tests age activity deterministically, and avoid changing background work semantics.

## Reproduction Assessment

Yes. Current-main stale display can be reproduced at the WorkActivityStore level by sending a started job without any terminal state; the PR regression can be reproduced by aging an active non-streaming job or long tool past 180 seconds and observing the watchdog report idle while work may still be running.

## Solution Assessment

No. A watchdog is a reasonable UI recovery mechanism, but evicting active work on a fixed three-minute display timestamp is not the narrowest maintainable fix; the safer direction is lifecycle-aware cleanup or an explicit stale/unknown display state plus deterministic tests for aged activities.

## Review Findings

- **[P2] Avoid evicting quiet active work:** `apps/macos/Sources/OpenClaw/WorkActivityStore.swift:337-342`
  - body: The watchdog removes any job/tool once lastUpdate is older than 180 seconds, but lastUpdate is only refreshed on streaming job events and tool starts. A healthy non-streaming provider call or long shell tool with no progress events can therefore flip the menu bar to idle while work is still running, which regresses the active-work display contract.
  - confidence: 0.89
- **[P2] Age the activity before expecting eviction:** `apps/macos/Tests/OpenClawIPCTests/WorkActivityStoreTests.swift:174-175`
  - body: This test starts a fresh job and immediately calls evictStaleActivities(), but the implementation only evicts entries older than jobStaleThreshold. As written the job should remain working, so the new test will fail or pressure the implementation toward evicting fresh activity.
  - confidence: 0.95
