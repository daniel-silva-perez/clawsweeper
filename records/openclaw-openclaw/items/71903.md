## Risks / Open Questions

- The PR currently skips the existing same-PID/no-starttime reclamation test instead of replacing it with direct regression coverage for a live self-PID legacy lock not being reclaimed and separate stale/dead/recycled cleanup coverage.
- The proposed valid-starttime branch may reclaim a same-PID lock when `getProcessStartTime(process.pid)` cannot be read, which is not clearly aligned with the PR body’s conservative no-verifiable-starttime rule.
- Changing legacy self-PID lock reclamation can trade data-corruption prevention against stale-lock hangs on platforms or containers where starttime cannot be verified, so maintainers should review the behavior boundary before landing.
