## Summary

Keep open. Current main still has the same-PID/no-starttime lock reclamation behavior this PR targets, and the linked session-history/message-loss reports make the bug plausible and important. The latest PR diff is still not merge-ready because it can remove a live same-PID lock when the current process starttime cannot be verified.

## What This Changes

The PR changes session write-lock self-orphan classification to use lock payload starttime and PID liveness, and skips the existing same-PID/no-starttime reclamation test.

## Best Possible Solution

Refine this branch so a same-PID lock is kept whenever the PID is alive and current starttime cannot be verified, while still reclaiming dead-PID and verified recycled-PID locks. Replace the skipped test with passing regression coverage for live same-PID/no-starttime and stored-starttime/current-starttime-null cases.

## Reproduction Assessment

Yes. Current main has a focused unit-test path that writes the current PID without starttime and expects reclamation, matching the behavior the PR changes; the remaining PR bug can be reproduced by forcing getProcessStartTime(process.pid) to return null for a same-PID lock with stored starttime.

## Solution Assessment

No. The PR direction is narrow and relevant, but this implementation is not the best fix until the stored-starttime branch falls back to PID liveness when current starttime is unavailable and the skipped test is replaced with affirmative coverage.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Keep live locks when starttime cannot be verified:** `src/agents/session-write-lock.ts:452`
