## Summary

Keep open. Current main has Telegram inline-button infrastructure and native exec-approval callbacks, but the DM pairing approval path remains text/CLI-only and has no pairing-specific approve/deny callback handling.

## What This Changes

The issue requests Telegram pairing notifications with inline Approve/Deny buttons that trigger the existing pairing approve/deny flow.

## Best Possible Solution

Implement a secure owner/operator pairing approval path rather than adding buttons to the requester-facing challenge. The maintainable shape is likely a shared channel-pairing approve/reject seam plus Telegram-owned inline buttons delivered only to an authorized operator surface, with tests for first-owner bootstrap, account scoping, callback authorization, and stale/denied requests.

## Reproduction Assessment

Yes. Code and tests on current main reproduce the missing behavior: Telegram DM pairing issues a text-only challenge containing the CLI approve command, and the callback handler has no pairing approve/deny branch.

## Solution Assessment

No, not as a direct inline-button patch on the current pairing reply. A safe solution must first define the authorized operator notification surface and deny semantics, then route callbacks through the same authorization and state transitions as CLI pairing approval.
