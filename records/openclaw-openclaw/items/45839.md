## Summary

Current main has generic Telegram inline-button support and Telegram exec-approval buttons, but the specific pairing approve/deny workflow is not implemented. DM pairing still produces text with an `openclaw pairing approve ...` command, `openclaw pairing` has no deny/reject command, and the Telegram callback handler has direct paths for plugin interactions, exec approvals, model/menu callbacks, and synthetic agent text, but no native pairing approve/deny resolver. The related closed generic-button and exec-approval issues do not supersede this narrower pairing workflow request.

## Best Possible Solution

Keep this issue open. The best implementation is a generic channel-pairing action path, backed by core APIs to approve and reject pending DM pairing requests, then a Telegram inline-button renderer/handler that authorizes the operator, resolves `approve` or `deny`, notifies the requester when appropriate, and edits the Telegram message to remove or mark the buttons. Reuse the existing Telegram plugin interactive/callback infrastructure where possible, and add focused tests for Telegram and at least one non-Telegram pairing owner path so the shared pairing contract stays channel-neutral.
