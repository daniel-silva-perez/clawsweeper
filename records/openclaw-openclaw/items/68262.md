## Summary

Keep this PR open. Current main still does not implement the source-tagged `internal_system` provenance contract for hiding internal heartbeat/cron/exec-event user turns, and the PR is still pointed at the right leak class; however, the branch needs a trust-boundary fix before merge because provenance-based hiding must only rely on trusted internal origins.

## What This Changes

The branch tags heartbeat/cron/exec-event turns with `InputProvenance.kind = "internal_system"`, hides matching user messages in gateway and Control UI history, adds focused regressions, and includes some unrelated formatting/import churn.

## Best Possible Solution

Land the source-tagged design after tightening the trust boundary: set `internal_system` only at trusted internal heartbeat/cron/exec origins, reject or ignore untrusted `internal_system` provenance at gateway ingress, keep the gateway projection as the shared suppression point with Control UI defense-in-depth, and preserve legacy text/envelope cleanup for old transcripts.

## Reproduction Assessment

Yes. The original leak has concrete Control UI/webchat reports in the PR discussion and #72918, and current-main source inspection shows the source-tagged provenance path is absent; the branch-level trust bug is reproducible by persisting a user turn with `provenance.kind === "internal_system"` through an ingress path that accepts request `inputProvenance`.

## Solution Assessment

No, not as-is. Source-tagging is the maintainable direction, but the current patch should first make `internal_system` a trusted-origin-only marker before gateway/UI history filters treat it as authority to hide user-role messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.8

Full review comments:

- **[P2] Gate hidden provenance on trusted ingress:** `src/gateway/chat-display-projection.ts:396`
  - body: The new history filter drops any user message with `provenance.kind === "internal_system"`, but `agent.run` currently normalizes request `inputProvenance` without the admin-scope gate that `chat.send` uses for system provenance. A gateway caller can persist an arbitrary user turn with that kind and this branch will hide it from chat history/session-history, so restrict `internal_system` to trusted internal origins or reject it at untrusted ingress before filtering on it.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces provenance-based history suppression, but the hiding marker is not yet proven to be trusted-only across gateway ingress paths.

Concerns:

- **[medium] Untrusted provenance can hide chat entries:** `src/gateway/chat-display-projection.ts:396`
  - body: Because `agent.run` accepts request `inputProvenance` and the PR hides messages solely on `provenance.kind === "internal_system"`, a caller with gateway access could make a visible user turn disappear from chat/session history. The patch should gate or strip that kind unless it was set by trusted internal code.
