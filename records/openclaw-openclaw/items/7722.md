## Summary

Keep open. Current main still only has the narrower `tools.fs.workspaceOnly` boolean guard; it does not accept `tools.fileAccess`, `mode`, `allowedPaths`, or `denyPaths`, and `exec` is not covered by a filesystem allow/deny policy. The related open PR #60981 is useful context but is not merged and explicitly leaves arbitrary shell execution out of scope.

## Best Possible Solution

Keep this issue open until OpenClaw implements a versioned filesystem access policy surface, likely under `tools.fs` or a successor config contract, with deny-wins `allowedPaths`/`denyPaths` enforced across read/write/edit/apply_patch/local media and a clearly designed exec strategy. Review #60981 as a related partial implementation candidate, but do not treat it as completion before merge or before the exec gap is explicitly resolved or split into a tracked follow-up.
