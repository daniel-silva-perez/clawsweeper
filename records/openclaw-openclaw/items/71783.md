## Summary

Keep open. Current main still leaves the reported metadata-free, different-name QMD path/pattern conflict unresolved: verified same-path/pattern conflicts can rebind, but name-only `collection list` output plus an add conflict still falls through to a skipped add while later searches use the configured collection name. Open PR #72297 is the active implementation candidate for the remaining repair-hint behavior, so this should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to reconcile or clearly surface QMD collection conflicts when an existing collection for the same path and pattern has a different name than the configured managed collection.

## Best Possible Solution

Keep the issue open until PR #72297 or an equivalent memory-core change lands: parse the stale conflict name from QMD stderr, show a targeted manual repair hint when list metadata is unverifiable, keep automatic rebind limited to verified path/pattern matches, and retain regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The issue includes concrete QMD stderr for the `life` versus `life-main` conflict, and current tests already model the metadata-free list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

No for current main; yes for the linked implementation direction. The narrow maintainable solution is metadata-gated automatic rebind plus an explicit manual repair hint for stderr-only conflicts, rather than blindly removing a collection name parsed from stderr.
