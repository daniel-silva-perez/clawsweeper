## Summary

Keep open. Current main still only auto-rebinds QMD collection conflicts when it can verify path and pattern metadata from `collection list`; the reported stale-name case remains unresolved when QMD only exposes the conflict through add stderr, and PR #72297 is still the focused implementation candidate rather than a landed fix.

## What This Changes

The issue reports that QMD can already have a collection registered for the same path and glob under a different name, causing OpenClaw boot registration for the configured `life` collection to fail and later searches to fail with `Collection not found: life`.

## Best Possible Solution

Land PR #72297 or an equivalent memory-core change that preserves automatic removal/rebind only for verified path/pattern matches, while parsing or clearly surfacing the stale conflict name from QMD stderr as a manual repair hint when list metadata is unavailable.

## Reproduction Assessment

Yes. The issue includes concrete QMD stderr for the `life` versus `life-main` conflict, and current unit coverage models the metadata-free `collection list` plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

No for current main. The narrow maintainable solution is metadata-gated automatic rebind plus explicit manual repair guidance for stderr-only conflicts, rather than blindly deleting a name parsed from QMD stderr.
