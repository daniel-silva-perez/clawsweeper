## Summary

Keep open. Current main still leaves the metadata-free different-name QMD conflict unresolved: verified path/pattern matches can be rebound, but name-only conflicts still fall through to the skipped-add warning while searches continue to target the configured managed collection name. The existing discussion and open PR #72297 are aligned with that remaining gap.

## What This Changes

The issue reports that boot-time QMD collection registration fails when the configured collection name differs from an existing QMD collection at the same path/pattern, causing searches for the configured name to fail.

## Best Possible Solution

Implement the narrow memory-core behavior where verified path/pattern conflicts can be safely rebound, while metadata-free different-name conflicts produce a clear manual repair hint naming the conflicting QMD collection instead of silently leaving later searches to fail.

## Reproduction Assessment

Yes. The issue provides concrete QMD stderr, and current regression coverage models the key path with name-only collection metadata plus an add conflict that still results in `qmd collection add skipped for workspace-main`.

## Solution Assessment

Yes for the #72297-style direction, not for current main. The safest solution is to keep destructive repair metadata-gated and add a targeted manual repair hint when QMD only exposes a stale conflicting name.
