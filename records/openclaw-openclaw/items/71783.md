## Summary

Keep open. Current main still leaves the reported QMD stale-name conflict unresolved when `collection list` cannot verify path/pattern metadata, and open PR #72297 remains the active implementation candidate rather than a landed fix.

## What This Changes

The issue requests that QMD startup collection registration reconcile or clearly surface an existing same path/pattern collection registered under a different name so searches for the configured collection name do not fail.

## Best Possible Solution

Keep the issue open until PR #72297 or an equivalent memory-core change lands: preserve automatic removal/rebind only for verified path/pattern matches, parse or surface the stale conflict name from QMD stderr as a manual repair hint when metadata is missing, and keep regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The issue includes concrete QMD stderr for the `life` versus `life-main` conflict, and current tests model the metadata-free list plus add-conflict path that still produces a skipped-add warning.

## Solution Assessment

No for current main. The narrow maintainable fix is the linked implementation direction: metadata-gated automatic rebind plus an explicit manual repair hint, rather than blindly deleting a name parsed from stderr.
