## Summary

Keep this draft PR open. Current main still lacks the requested working-context persistence and user-visible integration, and the proposed storage primitive has a blocking temp-file collision plus unresolved schema/privacy review before it should merge.

## What This Changes

The PR adds a new `src/agents/session-working-context.ts` utility and colocated Vitest coverage for resolving, normalizing, reading, writing, and clearing a sibling JSON working-context capsule beside a session transcript.

## Best Possible Solution

Settle the working-context schema/privacy contract, fix the atomic writer collision, then either land this primitive as the storage contract or fold it into the first integration that updates the capsule from runtime/tool paths and exposes it through status and compaction/new-session flows.

## Reproduction Assessment

Yes. The current-main gap is reproducible by repository search and inspection of session metadata, status output, successor transcript rotation, and subagent workspace inheritance; the PR-introduced writer race is statically reproducible from the shared temp-path expression in the diff.

## Solution Assessment

Unclear as proposed. A small storage primitive is a plausible first step, but the current implementation is not merge-ready until each write gets a unique temp path and maintainers decide the durable local-state schema/privacy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use a unique temp path for each context write:** `src/agents/session-working-context.ts:176`
  - body: `writeWorkingContext` derives the temp filename from only `process.pid` and `Date.now()`. Two writes to the same session in the same process and millisecond can share one temp file, so one rename/unlink can race the other and either reject persistence or write the wrong final capsule; use `randomUUID()` or an existing atomic helper per call.
  - confidence: 0.87
