## Summary

Keep this issue open. Current main still starts outbound delivery recovery on non-minimal Gateway startup, loads every pending outbound queue entry, and treats first-replay crash entries as immediately eligible without any stale-age cutoff or replay-count cap for old retryCount=0 messages. Recent recovery hardening reduces duplicate sends and budget deferrals, but it does not address the reported stale successful replay behavior.

## Best Possible Solution

Implement a conservative outbound recovery policy in src/infra/outbound: expire or move stale pending deliveries to failed/ before replay, preserve recent crash recovery, and add focused tests for old retryCount=0 entries without lastAttemptAt, maxRecoveryMs interaction, startup recovery, and reconnect drains. If operator control is wanted, add documented config/schema for max pending age and possibly max replay count.
