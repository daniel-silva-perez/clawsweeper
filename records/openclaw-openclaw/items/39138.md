## Summary

Keep this issue open. Current main still replays every pending outbound delivery queue entry on non-minimal Gateway startup, and old retryCount=0 entries without lastAttemptAt are still treated as immediately eligible with no age cutoff or replay-count cap. The prior ClawSweeper keep-open comment remains accurate after re-checking current main at 955b4df0939f90b4b6de5e7e6f3149dd125e1e50.

## Best Possible Solution

Implement a conservative outbound recovery policy in src/infra/outbound: expire or move stale pending deliveries to failed/ before replay, keep recent crash recovery behavior, and add focused tests for old retryCount=0 entries without lastAttemptAt, maxRecoveryMs interaction, startup recovery, and reconnect drains. If operator control is desired, add documented config/schema for max pending age and possibly max replay count.
