## Summary

Keep open. The PR targets a real voice-call startup race, but the proposed sequencing leaves an accepted Twilio start frame in the pre-start pending state while awaiting STT readiness, so the existing 5s pending guard can close valid streams before the 10s OpenAI STT connect timeout completes.

## What This Changes

The PR changes Twilio voice-call media-stream startup to await realtime STT connection before firing `onConnect`, closes the media stream on STT connect failure, adds media-stream regression tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep the STT-readiness sequencing, but transition a validated and accepted Twilio `start` frame out of pending/pre-start state before waiting for provider readiness. Preserve the intended 1011 close plus `onDisconnect` behavior for actual STT connect failures, and add focused coverage for the slow-connect path.

## Reproduction Assessment

Yes. The original bug has detailed live reproduction in #75197, and the PR regression can be reproduced with a focused media-stream test that sends a valid `start`, delays `connect()` past `preStartTimeoutMs`, and observes an unintended 1008 `Start timeout` close.

## Solution Assessment

No. The direction is maintainable, but the implementation should clear or cancel the accepted-start pending timer before awaiting `sttSession.connect()` so the anti-idle guard only protects sockets that have not sent a valid start.
