## Summary

Keep this PR open: current main still uses Telegram's ordinary per-chat/per-topic sequential key, so the underlying busy-topic queueing problem is not implemented on main. The PR is a useful implementation candidate, but it is not merge-ready because the new wrapper diverges from core queue semantics for default configs, misses embedded-only busy runs, and ignores the bot instance config that webhook and polling callers pass in.

## What This Changes

The PR wraps Telegram sequential-key selection with a busy-session control lane for immediate queue modes, exports a reply-run active lookup through the plugin SDK, and adds Telegram/reply-run tests for the new routing behavior.

## Best Possible Solution

Land an equivalent Telegram-owned busy-session bypass after aligning it with core resolveQueueSettings semantics, using the bot instance config, checking embedded-run busy state through the same core runtime path, and rebasing onto the current plugin SDK/config-runtime surface with focused Telegram and reply-run tests.

## Reproduction Assessment

Yes. A focused test can create a busy Telegram session with no explicit messages.queue.mode and show core resolves steer while the PR wrapper falls back to collect; another can seed an embedded active run without a reply operation and show the PR's reply-run-only check misses it.

## Solution Assessment

No. The direction is narrow and maintainable, but the current patch should not merge until it mirrors the core queue resolver/defaults, honors opts.config, and uses a busy-state seam that covers embedded-only runs.

Full review comments:

- **[P2] Match the core queue default:** `extensions/telegram/src/sequential-key.runtime.ts:95`
  - body: Core resolveQueueSettings defaults an unset queue mode to `steer`, but this wrapper falls back to `collect`. For the default Telegram config, a busy session will stay on the ordinary topic lane even though core would treat the turn as immediate steering, so the main user path remains blocked.
  - confidence: 0.92
- **[P2] Treat embedded-only runs as busy:** `extensions/telegram/src/sequential-key.runtime.ts:173`
  - body: Core busy-state resolution covers embedded runs that may not have a reply-run registry entry, but this gate only calls `isReplyRunActiveForSessionKey`. In that embedded-only state, immediate Telegram turns still fall back to the topic lane and cannot reach core interrupt/steer handling promptly.
  - confidence: 0.88
- **[P2] Use the bot instance config for lane policy:** `extensions/telegram/src/bot-core.ts:551`
  - body: `createTelegramBotCore` can be constructed with `opts.config`, and webhook/polling callers pass that instance config through. The new wrapper reloads the process/global config instead, so a bot created with an override can choose the wrong queue mode or command policy when deciding whether to bypass the topic lane.
  - confidence: 0.86
