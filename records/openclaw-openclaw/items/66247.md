## Summary

Keep this PR open. Current main still uses Telegram's plain per-chat/per-topic sequential key, so ordinary busy-topic follow-ups cannot bypass the adapter lane for immediate queue modes before core queue policy sees them. The PR remains a relevant unmerged implementation candidate, but the supplied review thread still has embedded-run and instance-config concerns that should be resolved before merge.

## Best Possible Solution

Keep the PR open and finish the Telegram-specific busy-session bypass or land an equivalent current-main implementation. The best path is to preserve existing control lanes, match core queue-mode precedence and aliases, cover text and media-only turns, include embedded-run busy-state handling, honor per-instance config, and keep the SDK seam narrow and documented if it remains necessary.
