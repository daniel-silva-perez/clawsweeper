## Summary

Keep this PR open. Current main still wires Telegram through the plain per-chat/per-topic sequential key, so ordinary follow-up messages in a busy topic cannot bypass the adapter lane for immediate queue modes before core queue policy sees them. The PR remains an unmerged, relevant implementation candidate, but it still needs maintainer review for the outstanding embedded-run and instance-config concerns before landing.

## Best Possible Solution

Keep the PR open. The best path is to review and finish the Telegram-specific busy-session bypass on top of the current PR shape: preserve existing control lanes, match core queue-mode precedence and aliases, cover text and media-only turns, and resolve the embedded-run plus instance-config review comments with tests before merge.
