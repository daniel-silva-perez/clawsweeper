## Summary

Current main does not implement this opt-out, and the PR is a plausible fix for a real user-visible false-positive class, so it should stay open. It is not merge-ready because the new config key is read through the typed config path without updating `MessagesConfig`, and the public generated config metadata/docs surface would drift; maintainers also need to choose between this config knob and overlapping reminder-note approaches.

## What This Changes

This PR gates the unscheduled-reminder note behind optional `messages.unscheduledReminderNote`, adds the Zod field and label, and adds reminder-guard unit tests.

## Best Possible Solution

Settle the reminder-guard UX first, then land one complete implementation: either a fully documented and generated `messages.unscheduledReminderNote` config with runtime wiring, public types, schema/help metadata, docs/changelog, and integration coverage, or the competing strip/system-annotation approach that prevents the internal note from appearing in delivered user-visible text.
