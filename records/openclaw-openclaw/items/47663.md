## Summary

Keep open. Current main still has the reported user-visible reminder-note behavior with no config opt-out, and the PR is a plausible small fix, but the proposed public config key is incomplete: it is read from typed config without adding the `MessagesConfig` field, and the generated/help metadata contract is not updated. Maintainers also need to choose whether this config opt-out or the related stripping approach in #39330 is the intended UX.

## What This Changes

This PR adds a `messages.unscheduledReminderNote` boolean gate around the unscheduled-reminder note append path, adds the key to the Zod session schema and field labels, and adds reminder-guard unit tests.

## Best Possible Solution

Settle the reminder-guard UX first, then land one complete implementation: either a fully typed, documented, generated, and tested `messages.unscheduledReminderNote` config option, or the competing delivery-strip/system-annotation path that prevents this internal note from appearing as agent-authored user text.

## Reproduction Assessment

Yes. Current main has a focused source and test path: a reply payload like `I'll remember that` or `I'll remind you tomorrow morning` with `successfulCronAdds: 0` and no same-session active cron reaches `appendUnscheduledReminderNote`, and the existing tests assert the appended note behavior.

## Solution Assessment

Unclear. A default-on config opt-out can address the reported pain, but this branch has not completed the public config contract and maintainers should choose it deliberately over #39330's narrower delivery-stripping approach.
