## Summary

Keep open. Current main still runs legacy iMessage by spawning the configured `imsg rpc` child process from the Gateway, and the user-facing docs still describe Full Disk Access as required for the process context running OpenClaw/`imsg`. Related PR #72898 is relevant because it improves permission-denied handling, but it does not provide the stable macOS TCC process identity requested here; BlueBubbles remains a documented alternative channel, not an implementation of this legacy-path request.

## What This Changes

The issue asks for legacy `imsg` access to Messages `chat.db` to run under a stable `imsg` or OpenClaw-owned process identity so Homebrew Node upgrades do not invalidate the parent Node Full Disk Access grant.

## Best Possible Solution

Keep this for maintainer product/security handling. The durable end state should either run legacy `imsg` database access behind a stable signed OpenClaw- or `imsg`-owned process identity, or explicitly retire the legacy path in favor of BlueBubbles with clear migration, diagnostics, and permission guidance.
