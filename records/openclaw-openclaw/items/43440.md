## Summary

Keep open. Current main still launches legacy iMessage through gateway/plugin-owned `imsg rpc` child processes, and the docs still require Full Disk Access for the process context running OpenClaw/`imsg`. Related PR #72898 improves permission-denied handling but does not provide the requested stable macOS TCC process identity.

## What This Changes

The issue requests legacy `imsg` database access to run under a stable `imsg` or OpenClaw process identity so Homebrew Node upgrades do not invalidate the parent Node process Full Disk Access grant.

## Best Possible Solution

Keep this for maintainer product/security handling. The durable end state should either run legacy `imsg` database access behind a stable signed OpenClaw- or imsg-owned process identity, or explicitly retire the legacy path in favor of BlueBubbles with clear migration, diagnostics, and permission guidance.
