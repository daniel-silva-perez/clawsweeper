## Summary

Keep this issue open. Current main still implements legacy iMessage as gateway/plugin-owned `imsg rpc` child processes, and the public docs still describe the integration as stdio RPC with no daemon plus Full Disk Access for the OpenClaw/`imsg` process context. The requested stable macOS permission identity is not implemented on main.

## Best Possible Solution

Keep this open for bundled iMessage plugin product/security handling. If legacy `imsg` remains supported, move Messages database access behind a stable macOS process identity such as a signed OpenClaw-owned helper, imsg-owned daemon, launchd/XPC bridge, or another explicit helper boundary; otherwise document and execute a retirement path toward BlueBubbles. The work likely belongs in `extensions/imessage/src/client.ts`, probe/send/monitor paths, setup copy, public iMessage docs, and macOS TCC reproduction coverage.
