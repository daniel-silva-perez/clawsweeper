## Summary

Keep open. Current main still runs the legacy iMessage path by spawning the configured `imsg rpc` child from the Node/Gateway process, and the public docs still require Full Disk Access for the same process context that runs OpenClaw/`imsg`. The related diagnostics PR and LaunchAgent issue are useful context, but neither provides the requested Node-upgrade-stable TCC identity.

## What This Changes

Request that legacy `imsg` read `~/Library/Messages/chat.db` under its own stable process identity so an `imsg` Full Disk Access grant remains sufficient after Homebrew Node upgrades.

## Best Possible Solution

The durable end state should either move legacy `imsg` database access behind a stable signed OpenClaw- or `imsg`-owned process identity with clear macOS TCC guidance, or explicitly deprecate/retire the legacy `imsg` path in favor of BlueBubbles with migration docs and actionable permission-denied diagnostics.

## Reproduction Assessment

No. Static inspection confirms current main lacks the requested stable process boundary, but a high-confidence live reproduction requires macOS TCC state, a real `~/Library/Messages/chat.db`, an `imsg` FDA grant, and a Homebrew Node resolved-path upgrade.

## Solution Assessment

Unclear. The requested direction is valid, but the narrowest maintainable fix depends on maintainer decisions about macOS signing, helper or XPC architecture, least-privilege Full Disk Access, and whether legacy `imsg` should continue beside the recommended BlueBubbles path.
