## Summary

Keep open. Current main still uses the legacy iMessage path by spawning the configured `imsg rpc` child process from the Gateway/Node runtime, and the docs still say the process context running OpenClaw/`imsg` needs Full Disk Access. Related PR #72898 is useful for permission-denied handling, but it does not provide the stable macOS TCC process identity requested here.

## What This Changes

The issue requests that legacy iMessage `imsg` access `chat.db` from its own stable process context so the parent Node/Gateway process does not need Full Disk Access after Homebrew Node upgrades.

## Best Possible Solution

The durable end state should either move legacy `imsg` database access behind a stable signed OpenClaw- or `imsg`-owned process identity with clear TCC guidance, or deliberately retire the legacy path in favor of BlueBubbles with migration, diagnostics, and permission documentation.
