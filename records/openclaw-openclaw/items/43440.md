## Summary

Keep open. Current main still runs the legacy iMessage path by spawning the configured `imsg rpc` process from the Gateway/Node runtime, and the docs still require Full Disk Access for the process context running OpenClaw/`imsg`. Related PR #72898 is useful permission-denied handling, but it does not provide the stable macOS TCC process identity requested here.

## What This Changes

The issue requests that legacy iMessage `imsg` read `chat.db` under a stable `imsg`- or OpenClaw-owned process identity so Homebrew Node upgrades do not invalidate Full Disk Access for the parent Gateway process.

## Best Possible Solution

The durable end state should either move legacy `imsg` database access behind a stable signed OpenClaw- or `imsg`-owned process identity with clear TCC guidance, or deliberately retire the legacy path in favor of BlueBubbles with migration, diagnostics, and permission documentation.

## Reproduction Assessment

No. A high-confidence live reproduction requires macOS TCC state, Full Disk Access grants, and a Homebrew Node binary path change; this read-only Linux checkout can only verify that current main still lacks the requested process boundary.

## Solution Assessment

Unclear. A stable signed helper or OpenClaw-owned runtime could solve the request, but maintainers should first choose the security/product direction across macOS TCC, signing, helper architecture, and legacy iMessage support.
