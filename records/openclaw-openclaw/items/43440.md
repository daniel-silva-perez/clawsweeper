## Summary

Keep this open. The checked-out main tree still runs legacy iMessage through gateway/plugin-owned `imsg rpc` child processes, and the public docs still describe that model as no-daemon stdio RPC with Full Disk Access required for the OpenClaw/`imsg` process context. The open related PR #72898 improves permission-denied error handling, but it does not provide the requested stable macOS TCC process identity.

## What This Changes

The issue requests that legacy iMessage database access happen under a stable `imsg` process identity so the parent Node process does not need Full Disk Access after Node binary path changes.

## Best Possible Solution

Keep this open for maintainer product/security handling. The durable end state should either move legacy `imsg` database access behind a stable OpenClaw- or imsg-owned macOS process identity, or explicitly retire the legacy path in favor of BlueBubbles with clear migration and permission guidance.
