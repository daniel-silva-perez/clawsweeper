## Summary

Keep open. Current main still runs legacy iMessage through OpenClaw-spawned `imsg rpc` child processes, and the public docs still require Full Disk Access for the OpenClaw/`imsg` process context. Related PR #72898 improves permission-denied handling, but it does not provide the stable macOS TCC process identity requested here.

## What This Changes

The issue requests that legacy `imsg` database access run under a stable `imsg` or OpenClaw-owned process identity so Homebrew Node upgrades do not invalidate the parent Node process Full Disk Access grant.

## Best Possible Solution

Keep this for maintainer product/security handling. The durable end state should either run legacy `imsg` database access behind a stable signed OpenClaw- or imsg-owned process identity, or explicitly retire the legacy path in favor of BlueBubbles with clear migration, diagnostics, and permission guidance.
