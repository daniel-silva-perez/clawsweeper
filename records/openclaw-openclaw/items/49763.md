## Summary

Keep this issue open. Current main still lets Ctrl+O replace an active busy TUI status with non-busy tool-toggle text, which tears down the Loader spinner; the related queued-message paths have some safeguards, but the Ctrl+O reproduction from the issue remains actionable and untested.

## Best Possible Solution

Keep the issue open and fix the TUI status model in src/tui/tui.ts so transient tool-output toggle notices do not replace an active busy state. A conservative fix is to preserve the current Loader whenever activeChatRunId or pendingOptimisticUserMessage is present, while still toggling chatLog.setToolsExpanded; add a focused regression test for Ctrl+O during an active run and keep the existing queued-message/BTW safeguards intact.
