## Summary

Keep this issue open. Current main still has a TUI status-state path that can stop the spinner during an active run: Ctrl+O toggles tool output by setting a non-busy activity status, and the status renderer tears down the Loader for non-busy states. The queued-message path has had related TUI fixes, but the Ctrl+O reproduction described in the issue remains enough to keep this actionable.

## Best Possible Solution

Keep the issue open and fix it in `src/tui/tui.ts` by decoupling transient tool-output toggle notices from the run activity state. A conservative fix would preserve the current busy status and Loader whenever `activeChatRunId` or `pendingOptimisticUserMessage` is present, while still toggling `chatLog.setToolsExpanded`; add a focused TUI test for Ctrl+O during an active run and, if feasible, a queued-send regression test that verifies the activity indicator remains busy.
