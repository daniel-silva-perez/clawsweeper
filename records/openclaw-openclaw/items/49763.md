## Summary

Keep this issue open. Current main still has a source-level path where toggling tool output during an active TUI run can replace the busy Loader with static non-busy status text: Ctrl+O sets `tools expanded`/`tools collapsed`, those are not busy statuses, and the renderer stops the Loader for non-busy statuses. The queued-message path has some active-run safeguards, but the Ctrl+O reproduction from the report remains actionable and is not covered by a focused regression test.

## Best Possible Solution

Keep the issue open and fix the TUI status model in `src/tui/tui.ts` so transient tool-output toggle notices do not replace an active busy state. A conservative path is to preserve the current busy Loader whenever `activeChatRunId` or `pendingOptimisticUserMessage` is present, while still toggling `chatLog.setToolsExpanded`; add a focused test for Ctrl+O during an active run and keep the existing queued-message safeguards intact.
