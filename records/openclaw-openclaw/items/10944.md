## Summary

Keep open. Current main still has no supported Telegram parse-mode config at the top level or per account, while outbound Telegram sends, media captions, draft previews, and lower-level delivery helpers still force HTML parse mode with plain-text fallback only after formatting failures. The closed #10758 custom-emoji/raw-HTML request is narrower and not a canonical replacement for this broader config request.

## Best Possible Solution

Implement this in the bundled Telegram plugin/config surface: add a typed channels.telegram.parseMode setting, consider a per-account override if maintainers want that granularity, and wire it consistently through sends, media captions, edits, draft previews, and delivery helpers. Update strict schema, exported Telegram config types, generated config metadata/help, public Telegram docs, and tests for current HTML behavior plus plain/no-parse behavior; treat MarkdownV2/raw HTML/custom emoji support as explicit maintainer/security choices or narrower follow-up work.
