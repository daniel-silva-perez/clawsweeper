## Summary

Keep open. Current main still has no supported Telegram parse-mode config at either the channel or account level, and the send, caption, edit, draft preview, and lower-level delivery paths still force Telegram HTML formatting with plain-text fallback. The linked #10758 custom-emoji/raw-HTML request and open #16121 custom-emoji request are related but narrower, not canonical replacements for this broader config surface.

## What This Changes

Request adding optional top-level and per-account Telegram parse-mode configuration so users can choose plain/no-parse, Markdown, or HTML behavior instead of relying on hardcoded Telegram formatting.

## Best Possible Solution

Add a typed Telegram plugin config surface for parse-mode behavior, preserve current HTML rendering as the compatibility default unless maintainers intentionally change it, support a safe plain/no-parse path, and wire the setting consistently through sends, captions, edits, draft previews, and delivery helpers with schema/types/metadata/docs/tests updated together.

## Reproduction Assessment

Yes. The missing feature is reproducible by source inspection or by validating a config containing channels.telegram.parseMode against the strict Telegram schema; the relevant schema/types omit the key and current outbound paths still set parse_mode: "HTML".

## Solution Assessment

Unclear until maintainers choose the exact supported values and default. The narrow maintainable direction is a bundled Telegram plugin config option with safe escaping semantics, while raw HTML, MarkdownV2, and custom emoji support should be explicit design choices or narrower follow-ups.
