## Summary

Keep open. Current main still has the Telegram outbound remote-media proxy gap this PR targets: direct Telegram sends and reply delivery still call the generic media loader without the Telegram transport, while inbound Telegram media already uses the proxy-aware transport. The PR remains useful source work, but it is stale against current transport and media-loader APIs and needs a real port before merge.

## What This Changes

This PR threads Telegram proxy transport options through outbound media prefetch for direct sends and bot replies, extends shared media load options, adds regression tests, and includes a small WhatsApp test adjustment.

## Best Possible Solution

Port the central fix onto current main: thread the current TelegramTransport from bot creation and direct-send account resolution into outbound send/reply media prefetch, teach the outbound media loader to carry sourceFetch, dispatcherAttempts, shouldRetryTelegramTransportFallback, and trusted explicit-proxy DNS semantics, and cover both direct sendMessageTelegram mediaUrl and delivered reply media paths with focused regression tests.
