## Summary

Keep open. The underlying Telegram outbound media proxy bug is still present on current main, but this PR branch is stale against the current Telegram transport and media-loader APIs, so it needs a current-main port or replacement before merge.

## What This Changes

This PR threads Telegram transport options into outbound remote media loading for direct sends and bot reply delivery, extends shared media load option plumbing, touches the old WhatsApp media helper, and adds regression tests.

## Best Possible Solution

Port the fix onto current main by carrying TelegramTransport.sourceFetch, dispatcherAttempts, trusted explicit-proxy DNS state, and shouldRetryTelegramTransportFallback through direct sends, normal reply delivery, and native-command reply delivery into the shared web-media/fetchRemoteMedia path, with focused regression tests for each outbound route.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure Telegram in a proxy-required environment and send or reply with a remote media URL; source inspection shows current main still reaches loadWebMedia without Telegram transport options on both direct-send and reply-delivery paths.

## Solution Assessment

No. The requested direction is correct, but this branch is not the best mergeable solution because it targets older pinned-dispatcher names and the old WhatsApp media implementation instead of current main's dispatcherAttempts and src/media/web-media.ts contracts.
