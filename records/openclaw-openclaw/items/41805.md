## Summary

Keep this issue open. Current main has made meaningful progress: Discord component clicks are routed back into the agent flow, sender ID/username are present on the inbound context, callbackData can be used as the fallback body, and plugin interactive handlers get structured messageId/kind/values fields. The specific requested normal agent inbound metadata is still not implemented: the fallback agent turn does not receive a structured interaction object with the source Discord message id, component id/custom id, button label, and clicker identity together.

## Best Possible Solution

Add a Discord component interaction metadata payload to normal agent fallback turns, likely in a structured context block or explicit finalized context fields. It should include interaction id, source message id, component id/custom id, kind, button/select label, callbackData/value, selected values when present, and sender id/username, while keeping the existing plugin handler context and BodyForAgent compatibility.
