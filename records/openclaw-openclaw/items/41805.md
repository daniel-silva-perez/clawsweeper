## Summary

Keep this issue open. Current main still routes Discord component clicks back to the normal agent flow as plain fallback text or callbackData, while plugin interactive handlers receive a structured subset. The specific requested normal agent inbound metadata bundle is still missing: source message id, component/custom id, button/select label, callbackData/value, selected values, interaction id, and clicker identity are not exposed together to the ordinary agent turn. The earlier Codex review comment remains accurate; since that review, the touched Discord files changed only in docs unrelated to interaction metadata.

## Best Possible Solution

Add a Discord component interaction metadata payload to normal agent fallback turns, likely through UntrustedStructuredContext or explicit finalized context fields. It should include interaction id, source message id, component id/custom id, kind, button/select label, callbackData/value, selected values when present, and sender id/username, while keeping existing plugin handler context and BodyForAgent compatibility. #40289 remains related for surrounding card text/context, but it does not supersede this metadata request.
