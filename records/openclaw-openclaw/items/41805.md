## Summary

Keep this issue open. Current main still routes Discord component clicks into the normal agent fallback as plain event text or callbackData, while structured interaction data is only exposed to plugin interactive handlers. The requested ordinary agent context metadata bundle remains missing, even though the source Discord interaction wrapper already preserves the raw custom id, source message, selected values, and clicker identity needed to build it.

## What This Changes

The issue requests structured Discord button/select interaction metadata in normal agent inbound context instead of only a plain callback body.

## Best Possible Solution

Expose a typed untrusted Discord component interaction block on normal fallback agent turns, reusing the existing structured context renderer and keeping the change inside the Discord plugin unless maintainers want a shared channel-interaction contract.

## Reproduction Assessment

Yes. The existing Discord monitor test harness can register a component entry, run the button or select control, and inspect lastDispatchCtx; current main shows only plain BodyForAgent plus sender/routing fields, without structured component metadata.

## Solution Assessment

Yes. Reusing UntrustedStructuredContext from the Discord fallback path is the narrowest maintainable solution because core already renders structured untrusted JSON and the Discord wrapper already exposes the needed raw interaction fields.
