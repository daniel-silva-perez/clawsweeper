## Summary

Keep open: current main still suppresses Discord ack/status reactions for guild/channel source turns that resolve to `message_tool_only`, and the issue's `ackReactionScope: "all"` configuration matches an existing regression-style test that expects no reaction call. The channel-resolution failure is useful live context, but current fallback returns the configured guild entries; the remaining question is the product contract between ack/status reactions and the v2026.4.27 privacy change.

## What This Changes

The issue requests restoring automatic Discord ackReaction in guild channels configured with `ackReactionScope: "all"` after v2026.4.27, especially when channel resolution falls back to configured entries.

## Best Possible Solution

Define the Discord/group-channel contract explicitly. If configured ack reactions are allowed even in tool-only rooms, narrow the Discord ack/status gate and update the matching tests; if not, document that `messages.groupChat.visibleReplies: "automatic"` is the supported visible-reaction workaround and add clearer logging when ack reactions are suppressed by source reply mode.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists in `extensions/discord/src/monitor/message-handler.process.test.ts`: a guild/channel route with `ackReactionScope: "all"` and `ackReaction: "👀"` currently expects no Discord reaction call because source delivery is `message_tool_only`.

## Solution Assessment

Unclear. Letting `ackReactionScope: "all"` bypass the tool-only suppression would solve this report, but it may conflict with the shipped #73046 privacy behavior that deliberately suppresses automatic status-reaction output in always-on rooms.
