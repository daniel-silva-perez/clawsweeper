## Summary

Keep open: current main still suppresses Discord automatic ack/status reactions for guild/channel source turns when the source reply mode is `message_tool_only`. The channel-resolution fallback log is relevant operational context, but source inspection shows configured guild entries are retained after that failure; the direct suppression point is the visible-reply privacy gate introduced around #73046.

## What This Changes

The issue reports that Discord automatic ack reactions stopped firing in guild channels on v2026.4.27 when `ackReaction` and `ackReactionScope: "all"` are configured.

## Best Possible Solution

Define the Discord/group contract explicitly. If configured ack reactions are allowed as processing feedback even in tool-only rooms, narrow the Discord ack/status gate and update tests/docs; if they are considered visible output, document the suppression clearly and add an operator-visible log or troubleshooting note pointing to `messages.groupChat.visibleReplies: "automatic"`.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: the current guild-route test configures `ackReactionScope: "all"` and `ackReaction: "👀"` and expects no Discord reaction call because source delivery is `message_tool_only`.

## Solution Assessment

Unclear. Letting `ackReactionScope: "all"` bypass tool-only suppression would solve the reported symptom, but it may conflict with the shipped #73046 privacy behavior that deliberately suppresses automatic status-reaction output in group/channel rooms.
