## Summary

Keep open: current main still suppresses Discord ack/status reactions for guild/channel inbound turns when source delivery resolves to tool-only mode, and the reported `ackReactionScope: "all"` configuration matches existing test coverage that expects no Discord reaction call. The reported channel-resolution/TLS log is useful context, but source inspection points to the later visible-output gate as the current-main skip path rather than the #73133 inbound reaction-notification fetch path.

## What This Changes

The issue asks OpenClaw to restore automatic Discord `ackReaction` behavior in guild channels on v2026.4.27 for `ackReactionScope: "all"`, including cases where Discord channel resolution logs TLS/fetch failures.

## Best Possible Solution

Define the Discord/group-room feedback contract explicitly. If ack/status reactions are allowed in tool-only rooms, narrow the Discord gate and update the matching tests/docs; if suppression is intentional, keep the gate but add clearer Discord docs/logging that point operators to `messages.groupChat.visibleReplies: "automatic"` when they want legacy visible feedback.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists in `extensions/discord/src/monitor/message-handler.process.test.ts`: a guild route with `ackReactionScope: "all"` and `ackReaction: "👀"` currently expects no Discord reaction calls because source delivery is `message_tool_only`.

## Solution Assessment

Unclear. Treating the channel-resolution failure as the sole fix target is not the best-supported path from current source; the maintainable fix depends on a product/safety decision about whether ack/status reactions should be exempt from the #73046 visible-output suppression.
