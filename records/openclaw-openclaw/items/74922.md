## Summary

Keep open: current main still suppresses Discord ack/status reactions for guild/channel source turns that resolve to `message_tool_only`, including the reported `ackReactionScope: "all"` shape. The reported channel-resolution failure is relevant live context, but source inspection shows that fallback returns configured guild entries; the remaining decision is whether configured ack/status reactions should bypass the v2026.4.27 private visible-output default.

## What This Changes

The issue asks OpenClaw to restore Discord automatic `ackReaction` behavior in guild channels after v2026.4.27, especially when `ackReactionScope: "all"` is configured and channel resolution falls back to configured guild entries.

## Best Possible Solution

Define the Discord/group-channel contract explicitly. If configured ack reactions are allowed in tool-only rooms, narrow the Discord ack/status reaction gate and update the matching tests; if they are not allowed, document the interaction between `ackReaction`, `ackReactionScope`, and `messages.groupChat.visibleReplies`, and add clearer logging when the reaction is suppressed by source reply mode.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists on current main: a guild/channel route with `ackReactionScope: "all"` and `ackReaction: "👀"` currently expects no Discord reaction call because source delivery is `message_tool_only`.

## Solution Assessment

Unclear. Letting `ackReactionScope: "all"` bypass the tool-only suppression would address the report, but it may conflict with the shipped v2026.4.27 privacy behavior that intentionally suppresses automatic status-reaction output in always-on rooms.
