## Summary

Keep open: current main still suppresses Discord ack/status reactions for guild/channel source turns when the reply mode is `message_tool_only`, and the issue's `ackReactionScope: "all"` shape matches an existing test that expects no reaction call. The reported channel-resolution failure is relevant context, but the current fallback keeps configured guild entries, while the linked #73133 path is for inbound reaction notifications rather than automatic ack reactions.

## What This Changes

The issue reports that Discord automatic ack reactions no longer fire in guild channels on v2026.4.27 when channel resolution fails despite `ackReaction: "👀"` and `ackReactionScope: "all"`.

## Best Possible Solution

Define the Discord/group contract explicitly. If configured ack reactions are allowed to be visible even in tool-only rooms, change the Discord ack/status gate narrowly and update the matching tests/docs; if not, document that `messages.groupChat.visibleReplies: "automatic"` is the supported visible-reaction workaround and add clearer logging when ack reactions are suppressed by source reply mode.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists in `extensions/discord/src/monitor/message-handler.process.test.ts`: a guild/channel route with `ackReactionScope: "all"` and `ackReaction: "👀"` currently expects no Discord reaction call because source delivery is `message_tool_only`.

## Solution Assessment

Unclear. Letting `ackReactionScope: "all"` bypass the tool-only suppression would solve this report, but it may conflict with the shipped #73046 privacy behavior that deliberately suppresses automatic status-reaction output in always-on rooms.
