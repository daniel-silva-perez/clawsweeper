## Summary

Keep open. Current main still classifies ended subagent orchestrators with pending descendants as active and uses that same active-run count for the reported spawn quota rejection; the later stale-unended orphan fix addresses a different liveness case.

## What This Changes

The issue asks for ended subagent runs with pending descendants to stop appearing active, stop consuming spawn quota, and be archived or pruned after a configurable duration.

## Best Possible Solution

Define a split subagent liveness policy: ended orchestrators with pending descendants should remain cleanup-aware for announce and descendant settlement, but should not be counted as active user work or consume requester spawn quota if maintainers accept that product behavior. Update list/status, registry count helpers, native and ACP quota checks, tests, and subagent docs together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an ended parent run with a pending child; current tests construct that shape and assert it remains active and quota-counted, matching the reporter's failure path. I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers confirm the policy. The narrow maintainable fix is not only hiding rows from `/subagents list`; it should split user-active/quota counting from pending cleanup and announce gating so nested completions still settle correctly.
