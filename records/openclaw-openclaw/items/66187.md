## Summary

Keep this issue open. Current main still treats an ended subagent run as active when it has pending descendants, and the per-session spawn quota uses that same active-run count, so the reported quota-blocking behavior is still present. The related stale-unended orphan work addressed a different liveness case and does not supersede this report.

## Best Possible Solution

Keep the issue open and resolve it with an explicit subagent liveness policy split: ended orchestrators with pending descendants can remain cleanup-aware for announce/descendant settlement, but should not appear as active user work or consume the requester session's spawn quota if maintainers agree with the reporter's expected behavior. Update `subagent-list`, `subagent-registry-queries`, native/ACP spawn quota behavior as needed, then replace the current opposite-behavior tests with regression coverage for the chosen policy and update the subagents docs.
