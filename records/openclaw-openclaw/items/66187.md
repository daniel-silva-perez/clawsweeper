## Summary

Keep this issue open. Current main still classifies an `endedAt`-set subagent run as active when it has pending descendants, and the same active-run count is used by the per-session spawn quota. The related closed item addressed stale unended orphans, but its own scope notes treat this ended-parent descendant case as separate.

## Best Possible Solution

Keep the issue open and resolve it with an explicit subagent liveness policy change: separate display/quota liveness from descendant cleanup gating, so ended orchestrators with pending descendants can remain cleanup-aware without consuming the requester’s spawn quota or appearing as active work if maintainers agree with the reporter’s expected behavior.

## Risks / Open Questions

- The issue uses `maxConcurrent` wording, but current main’s shown error comes from the per-session `maxChildrenPerAgent` gate via `countActiveRunsForSession`; the user-visible quota block is still present.
- Changing this behavior would affect existing announce/cleanup gating tests that intentionally keep ended orchestrators active while descendant cleanup is pending, so the fix needs a maintainer decision on the intended liveness model.
