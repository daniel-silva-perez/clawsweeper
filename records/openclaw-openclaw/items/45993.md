## Summary

Keep open. Current main still provides provider-based imports from other agent systems plus backup create/verify, but it does not provide same-product cross-platform `migrate export`, archive `migrate import`, or backup restore behavior; the open fork PR that explicitly proposes this surface should be reviewed, replaced, landed, or closed before this tracker is resolved.

## What This Changes

Add same-product cross-platform OpenClaw migration export/import commands with selective archive contents, path adaptation, environment-variable handling, memory merge/replace options, and dependency checks.

## Best Possible Solution

Keep this issue as the canonical tracker for same-product OpenClaw migration. Maintainers should decide whether the public API is `migrate export/import`, `backup restore`, or a coordinated pair, then review or replace the open fork PR with a focused implementation that includes archive validation, secret handling, path remapping, docs, and tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The feature-gap check is high-confidence: current CLI registration, implementation, docs, release state, and repository search show no same-product `migrate export`, archive `migrate import`, or `backup restore` surface on main.

## Solution Assessment

Unclear for the exact proposed `migrate export/import` command shape. The user problem is valid, but the narrowest maintainable solution should be reviewed against existing backup/restore semantics, provider migration semantics, and archive-import security constraints.
