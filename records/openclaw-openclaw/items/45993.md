## Summary

Keep open. Current main still provides provider-based imports from other agent systems and backup create/verify, but it does not implement same-product cross-platform archive export/import or backup restore. The issue is also paired with an open implementation PR in a fork, so it should remain open until that PR is reviewed, replaced, landed, or closed by maintainers.

## What This Changes

The issue requests `openclaw migrate export` and `openclaw migrate import` commands for cross-platform OpenClaw workspace/state migration with path adaptation, selective export, env-var detection, memory merge handling, and dependency checks.

## Best Possible Solution

Keep this issue as the canonical tracker for same-product OpenClaw migration. Maintainers should review the open fork PR against the existing provider-based migration and backup architecture, then land or replace a narrowed implementation with archive validation, secret handling, path remapping, docs, and tests; if that design is rejected, record the preferred backup/restore direction here.

## Reproduction Assessment

Not applicable as a bug reproduction. The high-confidence verification path is static: current CLI registration, command implementation, docs, tests, and repository search show no same-product `migrate export` or archive `migrate import` surface on main.

## Solution Assessment

Unclear for the exact proposed command shape. The user problem is valid, but the best maintainable fix should be reviewed against backup/restore semantics and security constraints rather than accepted solely as `migrate export/import`.
