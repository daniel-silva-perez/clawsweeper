## Summary

Keep open. Current main still provides provider-based imports from other agent systems plus backup create/verify, but it does not provide same-product cross-platform archive export/import or backup restore; an open fork PR explicitly proposes this command surface and should be reviewed, replaced, landed, or closed before this tracker is resolved.

## What This Changes

Request for `openclaw migrate export` and `openclaw migrate import` commands to package selected OpenClaw local state, adapt paths across platforms, detect missing environment variables, and merge or replace memory/config on import.

## Best Possible Solution

Keep this issue as the canonical tracker for same-product OpenClaw migration. Maintainers should review the open fork PR against the existing migration-provider and backup architecture, then land or replace a focused implementation with archive validation, secret handling, path remapping, docs, and tests; if `migrate export/import` is not the preferred API, record the backup/restore direction here.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The feature-gap check is high-confidence: current CLI registration, implementation, docs, and repository search show no same-product `migrate export`, archive `migrate import`, or `backup restore` surface on main.

## Solution Assessment

Unclear for the exact proposed `migrate export/import` command shape. The user problem is valid, but the narrowest maintainable solution should be reviewed against backup/restore semantics, secret handling, and archive-import security constraints.
