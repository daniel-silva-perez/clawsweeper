## Summary

Keep open. Current main still does not expose typed ACP process or PID metadata through the ACP spawn result, persisted session metadata, ACP runtime handle/status types, or the agent-facing session_status surface; the issue is also paired with open unmerged PR #74756, so it should remain open until that PR or an equivalent replacement lands.

## What This Changes

Expose OS process metadata for ACP sessions created by sessions_spawn so operators can correlate running child sessions with their adapter or child-agent processes.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that represents adapter and child-agent process identifiers when available, persists that metadata with ACP session state, exposes it through sessions_spawn and ACP status surfaces, and keeps cancellation/control semantics separate from manual PID killing.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path for this API-shape bug: the result types, accepted return object, persisted metadata, runtime status contract, and status tool output all omit typed ACP process metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but the stable schema and access-control/security model should be settled before treating the open PR as the final implementation.
