## Summary

Current main does not expose the requested process-correlation surface. The ACP spawn path returns logical run/session identifiers, the stored ACP metadata tracks backend/session identity but not process handles, and the `session_status` path renders stored session/status text rather than live adapter or child-agent PIDs. This is related to the open cleanup/orphan-process reports, but it is a narrower observability/control gap that remains actionable.

## What This Changes

The issue asks OpenClaw to expose process metadata or a first-class control path for ACP sessions spawned through `sessions_spawn`.

## Best Possible Solution

Define an additive ACP runtime process-metadata contract, have ACPX populate adapter and child-agent process identifiers when available, persist enough metadata for correlation, and expose it through `sessions_spawn`, `session_status`, and task/status surfaces with an explicit unavailable state for backends that cannot provide it.
