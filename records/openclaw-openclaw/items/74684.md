## Summary

Keep open. Current main still lacks typed ACP process/PID metadata in the spawn result, persisted ACP session metadata, and `session_status`; the issue is also already paired with open unmerged PR #74756, so it should remain open until that PR or an equivalent replacement lands.

## What This Changes

Request to expose process metadata, especially OS PID correlation, for local ACP child sessions created through `sessions_spawn(runtime="acp")` and related status/control surfaces.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that represents available/unavailable adapter and harness process identifiers, persists the metadata, exposes it through `sessions_spawn` and ACP status surfaces, and documents cancellation separately from raw PID exposure.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path: the ACP spawn result types, accepted response construction, tool return path, persisted metadata, and status tool all omit typed PID/process metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but maintainers should decide the stable schema and security model before treating the paired PR as the final solution.
