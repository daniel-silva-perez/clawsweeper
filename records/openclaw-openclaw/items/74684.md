## Summary

Keep open. Current main still does not expose ACP child process PID metadata through `sessions_spawn`, persisted ACP session metadata, or `session_status`, and this issue is already paired with open unmerged PR #74756 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Request to expose process metadata, especially OS PID correlation, for local ACP child sessions created through `sessions_spawn(runtime="acp")` and related status/control surfaces.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that represents available and unavailable adapter/harness PIDs, persists that metadata, and exposes it through spawn/status surfaces without over-broad termination authority.

## Reproduction Assessment

Yes. The missing field is reproducible by static inspection of current main: the ACP spawn result type, accepted response construction, tool return path, and persisted metadata all omit PID/process metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but maintainers should decide whether the best schema is one `pid`, separate adapter and child-agent PIDs, or status-only metadata before landing the paired PR.
