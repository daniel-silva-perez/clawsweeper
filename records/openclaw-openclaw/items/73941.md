## Summary

Keep open. Current main has ordinary path normalization and workspace boundary checks, but no configurable middleware that removes model-inserted spaces between Chinese/Han and ASCII characters in file paths. The related CJK items in the provided context cover encoding/mojibake, and the only nearby implementation found is a QQBot media-path encoding repair, not this cross-tool path correction feature.

## Best Possible Solution

Keep this open for maintainer product/security review. If accepted, the best implementation is likely an opt-in, existence-aware correction in OpenClaw-owned path-parameter wrappers, with raw exec command strings handled separately through prompt guidance or explicit tooling rather than heuristic command rewriting.

## Risks / Open Questions

- Naively removing spaces can rewrite legitimate filenames, especially when both spaced and unspaced Windows or network-share paths exist.
- Applying correction inside raw shell command strings is security-sensitive because it can alter quoted arguments, command text, and approval matching.
- A cross-tool feature needs product decisions around opt-in/default behavior, existence-based fallback, auditability, and Windows/CJK regression coverage.
