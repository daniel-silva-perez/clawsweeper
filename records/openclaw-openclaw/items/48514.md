## Summary

Keep this issue open. Current main has useful size-bounding mitigations for WebChat history, but the reported product gap remains: the Control UI still uses the WebSocket `chat.history` latest-window path, there is no WebSocket cursor/hasMore contract, the handler still materializes the transcript snapshot before trimming, and the rendered chat list is capped with Lit `repeat()` rather than paginated or virtualized. The related pagination and virtualization PRs are closed unmerged, so they are not shipped fixes.

## Best Possible Solution

Keep the issue open and implement a current-main-native WebChat history window: either extend the WebSocket `chat.history` protocol with cursor/hasMore metadata or route the Control UI chat history through the existing paginated `/sessions/:key/history` endpoint, then add a load-earlier or infinite-scroll UI with session-switch and streaming safety. Pair that with list virtualization or chunked markdown/rendering coverage for realistic long transcripts.
