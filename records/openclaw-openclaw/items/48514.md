## Summary

Keep this issue open. Current main has useful WebChat history bounding and sanitization, plus a separate paginated REST/SSE session-history endpoint, but the affected Control UI path still uses the WebSocket `chat.history` latest-window request with no cursor/hasMore contract, still materializes the transcript before trimming, and still renders a capped Lit `repeat()` list rather than paginated or virtualized history. The related pagination and virtualization PRs are closed unmerged, so they are not shipped fixes.

## Best Possible Solution

Keep the issue open and implement a current-main-native WebChat history window: either add cursor/hasMore metadata to the WebSocket chat.history contract or route Control UI history through the existing paginated /sessions/:key/history endpoint, then add load-earlier or infinite-scroll UI behavior with session-switch and streaming safety. Pair that with list virtualization or chunked markdown/rendering coverage and tests using realistic long transcripts with large tool results.
