## Summary

Keep open. Current main has useful size-bounding mitigations, but it still does not implement the issue's requested WebChat pagination or virtualization path: the WebSocket `chat.history` contract has no cursor/before parameter, the handler still materializes the transcript and returns a latest-window response, and the Control UI still requests/renders the latest 200 messages with Lit `repeat()`. The related pagination PRs in the provided context (#49169, #49359, #53812) were closed unmerged, so they are not a shipped fix for this issue.

## Best Possible Solution

Keep this issue open and implement a current-main-native WebChat history window: either extend the WebSocket `chat.history` protocol with cursor/hasMore metadata or route Control UI history through the existing paginated `/sessions/:key/history` endpoint, then add load-earlier or infinite-scroll UI plus regression coverage for large transcripts. Pairing that with list virtualization or chunked markdown rendering would address the remaining DOM/main-thread risk.

## Risks / Open Questions

- Existing size caps, markdown caching, and final-event refresh reductions may reduce the severity, so the eventual fix should benchmark current main with realistic long transcripts before choosing the exact pagination or virtualization design.
- The REST/SSE session-history cursor endpoint is a likely reuse path, but wiring WebChat to it needs session-switch, auth-scope, streaming, and scroll-position coverage.
- Raising the render limit without virtualization can make the original DOM and synchronous markdown cost worse on very long sessions.
