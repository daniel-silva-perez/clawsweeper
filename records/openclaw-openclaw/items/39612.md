## Summary

Current main still permits sanitized provider/runner failure details to be sent to external direct chats when `/verbose` is on. The Anthropic/Bedrock thinking-block corruption path has partial shipped mitigation through #45010, but that does not satisfy the central request to never expose raw LLM/API failure details to WhatsApp or Telegram users.

## What This Changes

The issue requests framework-level handling for LLM failures so messaging-channel users receive generic or silent fallback behavior instead of raw provider/API errors, plus follow-up recovery for repeated failures and corrupted thinking-block sessions.

## Best Possible Solution

Keep this open for a maintainer decision on external-channel failure policy. The likely product direction is generic direct-chat fallback copy, silent non-direct chat behavior, detailed provider/runner errors only in logs or operator surfaces, and focused regression coverage for WhatsApp and Telegram verbose LLM failures; session migration and multi-agent escalation can be split into separate follow-ups.

## Reproduction Assessment

Yes. The current test fixture in `agent-runner-execution.test.ts` gives a high-confidence reproduction of the central behavior: a mocked runner rejection in a WhatsApp direct context with `resolvedVerboseLevel: "on"` is expected to return the provider error text in the final reply.

## Solution Assessment

No. Current main's documented `/verbose` behavior is useful for diagnostics, but it is not the best fit for the issue's external-chat safety goal because it still sends provider/runner details to end users; a safer path is to keep those details on operator-only surfaces and send generic or silent fallbacks externally.
