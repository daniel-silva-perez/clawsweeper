## Summary

Keep open. The PR addresses a real config-validation gap, and the locked dependency already recognizes `google-vertex`, but the enum-only patch would expose configs that still fail provider-level inline model resolution and can miss Google replay-policy handling for model-level declarations.

## What This Changes

This PR adds `google-vertex` to `MODEL_APIS` and the generated base config-schema enum arrays for provider-level and model-level model API configuration.

## Best Possible Solution

Land an end-to-end Vertex config support change that keeps the enum and generated schema in sync, preserves `api: "google-vertex"` through provider-level and model-level configured model assembly, routes Google replay/policy handling consistently, and adds focused tests for both provider-level and model-level declarations.

## Reproduction Assessment

Yes. A source-level reproduction is the PR body’s provider-level config: `buildInlineProviderModels` drops `entry.api: "google-vertex"`, and `resolveExplicitModelWithRegistry` ignores inline models without `api`. A second source path is a custom provider with model-level `api: "google-vertex"`, which validates after this PR but is not recognized by the current Google fallback/policy lookup.

## Solution Assessment

No. Adding only enum values is not the best fix because it moves validation ahead of runtime propagation and policy handling; the narrower maintainable solution is to add the enum plus the existing normalization/policy seams and tests that prove configured Vertex models actually route as Google Vertex.
