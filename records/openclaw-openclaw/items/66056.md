## Summary

Keep open. Current main still has English-only runtime failure text in the sanitizer and reply-runner paths, and the shared messages config/schema surface still has no error-locale field. Existing Control UI localization is browser-local UI text, while plugin message hooks can rewrite outbound content but do not expose stable runtime error keys or locale context for this surface.

## What This Changes

The issue requests a lightweight locale-aware system for user-facing runtime failure messages, including an error locale config, keyed English fallback strings, and translated coverage for sanitizer and reply-runner errors.

## Best Possible Solution

Keep this open until OpenClaw has a stable runtime localization contract: keyed user-facing error identifiers with English fallback, documented locale resolution, wiring through sanitizer and reply-runner failures, and focused tests for fallback plus at least one non-English locale. If localization is delegated to plugins, core should first expose structured error keys and locale context so plugins are not matching English literals.

## Reproduction Assessment

Not applicable as a normal bug reproduction: this is a feature/API request. The current-main verification path is high-confidence source and test inspection showing English-only sanitizer and reply-runner outputs with no locale-aware config field.

## Solution Assessment

Unclear. The proposed lightweight keyed catalog with English fallback is plausible, but the best maintainable solution needs maintainer direction on locale source, config shape, and whether the structured error contract belongs in core or plugin APIs.
