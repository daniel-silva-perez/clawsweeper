## Summary

Keep open. Current main still has English-only runtime failure text in the sanitizer, reply-runner, assistant error formatting, and failover paths, and the shared messages config/schema has no locale or error-locale setting. The existing Control UI i18n is browser-local UI text only, while plugin message hooks can rewrite outbound strings but do not expose stable error keys or locale context for these runtime failures.

## What This Changes

Request to add locale-aware user-facing runtime error messages with a config locale, keyed locale dictionaries, English fallback, and coverage for common rate-limit, timeout, context-overflow, billing, transport, and run-failure text.

## Best Possible Solution

Add a stable runtime localization contract for user-facing failure text: structured error keys with English fallback, documented locale resolution, wiring through sanitizer/reply-runner/assistant error paths, and focused regression tests for fallback plus at least one non-English locale. If localization should be plugin-owned, core should first expose structured error keys and locale context so plugins do not match English literals.
