## Summary

Keep this issue open. Current main still lacks a locale-aware user-facing error system: runtime failure text and sanitizer rewrites are hardcoded English, there is no `messages.locale` or `messages.errorLocale` config/schema surface, Control UI i18n is browser-local, and current plugin message hooks expose only outgoing content plus routing metadata rather than structured error keys and locale context.

## Best Possible Solution

Keep this issue open for maintainer product/API design. The likely path is a keyed user-facing error catalog with English fallback, a documented `messages.locale` or `messages.errorLocale` config/schema/help surface, locale-aware formatting through sanitizer and reply-runner failure paths, and focused regression tests. If maintainers prefer plugin-owned localization, first add structured error keys and locale context to the relevant plugin hook or reply-dispatch surface so plugins do not have to match English literals.
