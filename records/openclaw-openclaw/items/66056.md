## Summary

Keep open. Current main still does not provide locale-aware gateway/runtime error messages: the sanitizer and reply runner take no locale input, continue to construct English user-facing failures directly, and the `messages` config surface has no locale field. The existing Control UI localization is browser-local UI text only, and the available plugin message hook can rewrite strings but does not expose structured error keys or locale context.

## What This Changes

The issue requests a lightweight locale system for user-facing runtime error messages, including a `messages` locale config field, English fallback catalogs, localized formatting, and wiring through sanitizer and runner failure paths.

## Best Possible Solution

Keep this open for maintainer product/API design. A solid end state would add stable user-facing error keys with English fallback, a documented locale config or locale-resolution policy, wiring through sanitizer and reply-runner failure paths, and focused regression tests; if localization is delegated to plugins, core should first expose structured error keys and locale context so plugins do not have to match English literals.
