## Summary

Keep open. Current main still has English-only user-facing runtime failure text in the sanitizer and reply-runner paths, and the shared `messages` config/schema surface has no error-locale setting. Control UI localization is browser-local UI text, while plugin hooks can rewrite outbound strings but do not expose stable error keys or locale context for this runtime error surface.

## What This Changes

The issue requests locale-aware user-facing runtime error messages through a message locale config, keyed locale dictionaries, English fallback, and sanitizer/reply-runner wiring.

## Best Possible Solution

Keep the issue open until OpenClaw has a stable runtime localization contract: keyed user-facing error identifiers with English fallback, documented locale resolution, wiring through sanitizer and reply-runner failures, and focused tests for fallback plus at least one non-English locale. If localization should be plugin-owned, core should first expose structured error keys and locale context so plugins are not matching English literals.
