## Summary

Keep open. Current main still contains the exact hardcoded `Agent response timeout (120s)` race in the Synology Chat webhook path, and that race wraps the downstream agent dispatch before the configured `agents.defaults.timeoutSeconds` budget can complete.

## Best Possible Solution

Keep the issue open and fix the Synology Chat plugin path: remove the hardcoded outer 120s race or derive it from the same configured agent runtime budget, passing config/runtime timeout into the webhook handler as needed. Add a focused webhook/inbound-turn regression test with `agents.defaults.timeoutSeconds` set above 120 seconds.
