## Summary

Keep open: current main and the latest release still race Synology Chat webhook delivery against a hardcoded 120s timeout, so a longer configured `agents.defaults.timeoutSeconds` can still be cut off. Related PR #39745 only clears timer handles while preserving the 120s behavior, and the earlier direct removal in #38360 was closed unmerged.

## What This Changes

The issue asks for Synology Chat/channel delivery to honor `agents.defaults.timeoutSeconds` instead of failing after a hardcoded 120 seconds.

## Best Possible Solution

Fix this in the Synology Chat plugin by removing the outer 120s watchdog or deriving it from the same resolved agent runtime timeout, while preserving immediate webhook ACK behavior and adding regression coverage for a delivery that exceeds 120 seconds under a larger configured timeout.

## Reproduction Assessment

Yes. A focused unit test can set a delivery promise that resolves after 120 seconds and advance fake timers; current `processAuthorizedSynologyWebhook` will reject with `Agent response timeout (120s)` before the configured agent runtime budget can complete.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned because the hardcoded cap lives in `extensions/synology-chat`; core already exposes the configurable agent timeout contract used by downstream dispatch.
