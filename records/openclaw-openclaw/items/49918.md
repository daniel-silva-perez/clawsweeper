## Summary

Keep this issue open. Current `main` still matches the reported regression: TUI `/new` creates a `tui-<uuid>` session through local `setSession(...)` and does not call the backend reset path that emits `command:new` hooks. The linked PR #49945 is still open and unmerged, so it is an implementation candidate rather than a reason to close this issue before merge.

## Best Possible Solution

Review and land #49945, or apply an equivalent fix on `main`: keep the `tui-<uuid>` isolation for TUI `/new`, but route the new key through `client.resetSession(uniqueKey, "new")` or an equivalent `sessions.reset` path so `command:new` hooks fire. Add/update the TUI command-handler test to assert `/new` emits the reset path while `/reset` continues to reset the current session.

## Evidence
