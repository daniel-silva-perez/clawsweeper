## Summary

Keep this issue open. Current main and the latest release still route TUI /new through local setSession(tui-<uuid>) without the gateway sessions.reset path that emits command:new hooks, and the open PR #49945 is an unmerged implementation candidate with closing syntax.

## Best Possible Solution

Review and land PR #49945, or apply the equivalent on main: keep TUI /new's unique tui-<uuid> session key, but route that key through client.resetSession(uniqueKey, "new") or the equivalent sessions.reset path so command:new hooks fire. Add/update the TUI command-handler regression test to assert /new uses the reset path with reason new while /reset still resets the current session with reason reset.
