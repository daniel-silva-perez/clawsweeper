## Summary

Keep open. Current main still lacks Feishu recalled-message handling, and the PR is useful active implementation work, but the latest diff still misses two acceptance criteria: it can answer a recalled non-thread reply target through the existing direct-send fallback, and it only tracks text/card outputs while media and streaming-card reply outputs remain unmapped.

## What This Changes

The PR adds a Feishu recalled-message event handler, an in-memory user-message to bot-reply map with TTL and retry retention, send-result reply-path metadata, and text/card reply registration for recall cleanup.

## Best Possible Solution

Revise the Feishu plugin so recall state is enforced at the reply boundary, not only in the deletion handler: skip responses once the source message has been recalled, avoid direct-send fallback for recalled targets, register every reply-path bot output that should be deleted, and cover the behavior with focused Feishu unit tests. Keep broader cross-channel recall/edit cancellation tracked separately in #55691.

## Reproduction Assessment

Yes: the source-backed path is to simulate a Feishu reply target that is recalled or withdrawn before response; current send.ts still falls back to a direct send for non-thread replies, and the PR head does not call isMessageRecalled before that send. Media and streaming-card paths are also reproducible by following their replyToMessageId calls, which do not call registerBotReply.

## Solution Assessment

No: the PR is the right owner boundary, but the current implementation is not the narrowest complete fix because it adds tracking only to text/card sends and leaves recall suppression disconnected from the shared reply fallback path. A safer solution is to centralize reply-path registration/suppression across text, card, media, and streaming helpers, or explicitly document and test any excluded surfaces.
