## Summary

Keep open. Current main still lacks a WhatsApp group-creation agent tool, message action, active-listener method, or send API wrapper, and the existing plugin/runtime surface does not expose Baileys `groupCreate` as a stable external extension point. Because the requested tool would invite phone-number participants through the operator's linked WhatsApp identity, the next step is maintainer product/security review, not conservative cleanup.

## What This Changes

Request to add a `whatsapp_create_group` agent tool with `groupName` and `participants` parameters that uses Baileys `groupCreate` for WhatsApp listing workflows.

## Best Possible Solution

If maintainers accept the product/security tradeoff, implement this inside the WhatsApp plugin or first add a narrow documented WhatsApp runtime seam. The end state should be an owner-only or otherwise trusted `whatsapp_create_group` capability that normalizes participant phone numbers to WhatsApp JIDs, calls the active linked session, redacts identifiers in logs, records outbound activity, adds focused WhatsApp tests, and updates the public WhatsApp docs.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high-confidence: exact searches for `whatsapp_create_group` and `groupCreate` are empty, and inspected WhatsApp tool/action/listener/send surfaces expose login, send, poll, reaction, and composing only.

## Solution Assessment

Unclear until maintainer product/security review. A WhatsApp-owned, gated tool or a narrowly documented runtime seam is the maintainable direction; adding an ungated generic tool or broad core helper would not be the best way to solve this.
