## Summary

Keep open. Current main still lacks a WhatsApp group-creation agent tool, message action, active-listener method, or send/runtime API, and this is not a clean ClawHub-only close because an external tool cannot create groups through the already-linked WhatsApp session without a WhatsApp-owned runtime surface. The dependency appears to support `groupCreate`, but exposing participant invites from the operator's linked identity needs maintainer product/security approval first.

## What This Changes

The issue requests a WhatsApp-owned `whatsapp_create_group` agent tool that creates a WhatsApp group from a group name and participant phone-number list via the linked WhatsApp session.

## Best Possible Solution

If maintainers approve the product/security tradeoff, implement this inside the WhatsApp plugin as a gated owner or trusted-requester capability: normalize participant numbers to WhatsApp JIDs, add a narrow active-listener/send API method that calls the live Baileys `groupCreate`, redact identifiers in logs, record outbound activity, add focused WhatsApp tests, and update WhatsApp docs. If maintainers reject agent-created group invites, document that limitation explicitly.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high-confidence: code and docs searches plus inspection of the WhatsApp tool, action, listener, send, runtime, and docs surfaces show login/send/poll/reaction/typing only, with no group-create path.

## Solution Assessment

Unclear until maintainer product/security review. The maintainable path is a WhatsApp-owned gated tool or a narrow additive runtime seam; a broad core helper or ClawHub-only close would not give external tools access to the existing linked WhatsApp session.
