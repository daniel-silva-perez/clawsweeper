## Summary

Keep this open. Current main has useful subagent delivery hardening, but it still contains terminal announce retry-limit/expiry give-up paths, failed delivery bookkeeping, timeout-as-terminal behavior, and docs that describe announce-back as best-effort; the April 29 reproduction also reports the same retry-limit warning on a current release, and open PR #45200 only targets one notification symptom.

## What This Changes

The issue asks OpenClaw to stop silently losing subagent completion results by retrying or surfacing failed announces, handling subagent timeouts with restart or notification, and verifying file-creation claims before reporting completion.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit delivery model that separates child completion, parent-session recording, external channel delivery, retry durability, timeout/restart policy, duplicate prevention, and result/file verification, with targeted regression coverage for fast responders and Telegram-style async delivery.

## Reproduction Assessment

Yes. The April 29 comment gives a concrete fast-responder sessions_spawn reproduction for Pattern 1, and current code/tests show finite retry-limit and timeout outcome paths; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Current main is a partial hardening, not the complete solution, because it still permits terminal undelivered states and timeout-only outcomes; #45200 may help one notification symptom but should not close the broader delivery-semantics issue.
