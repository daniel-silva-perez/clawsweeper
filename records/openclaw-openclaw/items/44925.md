## Summary

Keep this open. Current main has useful subagent announce hardening, but the reported failure family still exists as finite retry-limit/expiry give-up, timeout-as-terminal cleanup without restart, and best-effort delivery semantics; the latest discussion also reports the same retry-limit warning on a current release line.

## What This Changes

The issue requests reliable async subagent completion handling: retry or notify on announce failure, restart or notify on timeout, and verify claimed file artifacts before surfacing completion.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit completion and delivery model that separates child completion, parent-session recording, external channel delivery, durable retry, timeout/restart policy, duplicate prevention, and verifiable artifacts/results, with regression coverage for fast responders and Telegram-style async delivery.
