## Summary

Keep this open. Current main has meaningful subagent delivery hardening, but the reported failure family is not fully implemented away: announce delivery still has finite retry-limit give-up, completion dispatch can still return undelivered, timeouts finalize as terminal task state rather than restart, and docs still describe sub-agent announce as best-effort/lost after gateway restart. The latest issue discussion also reports the same retry-limit warning on v2026.4.26 with a concrete fast-responder reproduction.

## What This Changes

The issue requests reliable subagent completion handling: retry or surface announce failures, notify or restart on subagent timeout, and verify claimed file/artifact results before announcing completion.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit, durable completion and delivery model that separates child completion, parent-session recording, external channel delivery, retry persistence, timeout/restart policy, deduplication, and artifact/result verification, with regression coverage for fast responders and Telegram-style async delivery.

## Reproduction Assessment

Yes. The April 29 report gives a concrete two-agent `sessions_spawn` fast-responder path for the retry-limit failure, and current main still contains the finite retry/give-up path that matches it. Live Telegram proof was not run in this read-only review.

## Solution Assessment

No. Current main is partial hardening rather than a complete fix, and open PR #45200 covers only one symptom; the maintainable solution needs a clear durable completion/delivery policy before a narrow repair PR can safely cover the whole issue.
