## Summary

Keep open. Current `main` has useful subagent announce hardening, but it still has finite retry-limit and expiry give-up paths, timeout-as-terminal behavior without restart, and docs that describe announce-back as best-effort; the latest discussion also reports the same retry-limit failure on a current release line.

## What This Changes

The issue requests reliable subagent completion handling: retry or surface announce failures, notify or restart timed-out subagents, and verify file/result claims before reporting completion.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit completion and delivery state model that separates child completion, parent-session recording, external channel delivery, retry durability, timeout/restart policy, duplicate prevention, and verifiable artifacts/results, with regression coverage for fast responders and Telegram-style async delivery.

## Reproduction Assessment

Yes. The April 29 reproduction gives a concrete two-agent `sessions_spawn` fast-responder path for the retry-limit failure, and current code/tests still show the finite retry and terminal give-up behavior; live Telegram proof was not run during this read-only sweep.

## Solution Assessment

No. Current `main` is partial hardening rather than a complete fix, and the open notify-on-give-up PR covers only one symptom; the maintainable solution needs a clearer durable completion/delivery model before broad repair work.
