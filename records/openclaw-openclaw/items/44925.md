## Summary

Keep this open. Current main has useful subagent delivery hardening, but it still has terminal retry-limit/expiry give-up paths, failed delivery bookkeeping, timeout-as-terminal behavior, and docs that describe announce-back as best-effort; the latest issue discussion also reports the same retry-limit warning on a current release line.

## What This Changes

The issue asks for reliable subagent completion handling: retry or surface failed announces, notify or restart timed-out subagents, and verify file-result claims before announcing success.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit delivery model that separates child completion, parent-session recording, external channel delivery, retry durability, timeout/restart policy, duplicate prevention, and result/file verification, with targeted regression coverage for fast responders and Telegram-style async delivery.

## Reproduction Assessment

Yes. The April 29 comment gives a concrete fast-responder `sessions_spawn` reproduction for Pattern 1, and current code/tests show finite retry-limit and timeout outcome paths; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Current main is partial hardening rather than the complete solution, and open PR #45200 only addresses one notification symptom; the maintainable fix needs a clearer completion/delivery state model before broad repair work.
