## Summary

Keep this open. Current main still has terminal announce give-up and timeout paths, and the April 29, 2026 discussion reports the same `Subagent announce give up (retry-limit)` failure on a current release with a fast-responder reproduction. The open related PR only targets one notification symptom, not the broader reliable completion/delivery semantics described here.

## What This Changes

The issue asks OpenClaw to make subagent completion reliable by retrying or notifying on failed announce delivery, handling timeouts with restart or visible notification, and verifying file/result claims before announcing completion.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit delivery model that separates child completion, parent-session recording, external channel delivery, retry durability, timeout or restart policy, duplicate prevention, and result/file verification, with regression coverage for fast responders and Telegram-style async delivery.
