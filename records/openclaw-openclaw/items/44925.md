## Summary

Keep this open. Current main still has finite announce retry give-up, terminal cleanup after failed delivery, timeout-as-outcome behavior, and tests that assert undelivered completion paths; the fresh April 29 reproduction also reports the same retry-limit warning on a current release. Recent code adds useful fallback delivery paths, but it does not fully address durable completion delivery, timeout policy, user-visible confirmation, duplicate prevention, or file/result verification.

## What This Changes

The issue requests reliable async subagent completion handling so announce failures, subagent timeouts, and claimed file outputs do not silently disappear.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit delivery model that separates child completion, parent-session recording, external channel delivery, retry durability, timeout or restart policy, duplicate prevention, and result/file verification, with regression coverage for fast responders and Telegram-style async delivery.

## Reproduction Assessment

Yes. The April 29 comment gives a concrete fast-responder sessions_spawn reproduction for Pattern 1, and current tests/code show the terminal retry-limit and timeout outcome paths; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Current main is a partial hardening, not the best complete solution, because it still permits terminal undelivered states and timeout-only outcomes; PR #45200 may be useful for one notification symptom but should not close the broader delivery-semantics issue.
