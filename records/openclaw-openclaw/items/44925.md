## Summary

Keep this open. Current main still has a finite retry-limit give-up path for subagent completion announce delivery, and the latest discussion reports the same warning on v2026.4.26 with a concrete fast-responder reproduction. The linked open PR is useful but explicitly covers only one symptom, while this report also covers timeout handling, user-visible delivery confirmation, and result/file verification.

## What This Changes

The issue requests reliable subagent completion handling so announce failures, timeouts, and unverifiable result claims are retried, surfaced, restarted, or validated instead of being silently lost.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit completion/delivery state machine that separates child completion, parent-session recording, external channel delivery, timeout policy, deduplication, and result verification, with regression coverage for fast responders and Telegram-style async delivery.
