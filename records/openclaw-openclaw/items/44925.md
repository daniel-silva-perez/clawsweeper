## Summary

Keep this open. Current main still has a finite subagent announce retry-limit give-up path, best-effort completion delivery can still return delivered:false, timeout lifecycle handling still finalizes a timed-out run rather than restarting or recovering it, and the latest issue discussion reports the same warning on v2026.4.26 with a fast-responder reproduction. Existing hardening and open PR #45200 address part of Pattern 1, but they do not close the broader durable-delivery, timeout-policy, dedupe, and result-verification request.

## What This Changes

The issue asks OpenClaw to make subagent completion handling reliable so announce failures, timeouts, and file/result mismatches are retried, recovered, or surfaced instead of silently losing async results.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit completion and delivery state model that separates child completion, parent-session recording, external channel delivery, timeout policy, deduplication, and result/file verification, with regression coverage for fast responders and Telegram-style async delivery.
