## Summary

Keep open. Current main still does not implement the PR's intended WhatsApp per-agent ack emoji fallback: WhatsApp ack reactions still read only the WhatsApp object `emoji`, and an object without `emoji` resolves to an empty value that suppresses the ack. The PR remains a plausible implementation candidate, but maintainer review is still needed around the WhatsApp-specific resolver, public SDK export shape, and typed test cleanup.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is a WhatsApp-specific resolver that preserves no reaction when no WhatsApp ack config exists, skips legacy `messages.ackReaction`, honors account-level precedence, falls back to the responding agent identity only when WhatsApp ack config is intentionally present without an emoji, preserves empty-emoji disable semantics, and covers those cases with typed tests.
