## Summary

Keep open. Current main still does not implement this PR's intended WhatsApp per-agent ack emoji fallback: WhatsApp ack reactions still use only the configured WhatsApp object emoji, and the current generic resolver still treats account/channel ack reactions as strings. The PR remains a focused implementation candidate, with maintainer review still needed around the WhatsApp-specific resolver/export shape and typed test cleanup.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best implementation should preserve WhatsApp's no-global-ack behavior, use the responding agent identity emoji only when WhatsApp ack config is intentionally present, honor account-level precedence, cover object-without-emoji and empty-emoji edge cases with typed tests, and treat any plugin SDK export as an intentional public contract.

## Risks / Open Questions

- The PR tests should avoid `as any` / `typescript/no-explicit-any` suppression so the object-shaped config contract remains protected by types.
