## Summary

Keep open. Current main has useful mitigations for session listing, model-catalog caching, startup diagnostics, stuck-session recovery, and runtime-deps churn, but the reporter reproduced the degraded behavior on v2026.4.29, which already shipped several of those fixes, and the only inspected post-release relevant code change is a narrow runtime-deps reinstall-loop mitigation rather than proof that the combined Gateway event-loop/session/model slowdown is fixed.

## What This Changes

The issue reports a post-2026.4.23 Gateway regression where `sessions.list`, `models.list`, and related Control UI/Gateway surfaces become very slow and saturate the event loop until rollback to 2026.4.23 restores stability.

## Best Possible Solution

Keep this as an active regression until maintainers choose a canonical tracker or run a targeted bisect from 2026.4.23 to current with a large session store, then land focused fixes for the remaining hot path and link or close the overlapping trackers afterward.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction shape: run the same large local state on a post-2026.4.23 build, measure `sessions.list`, `models.list`, liveness diagnostics, and CPU, then compare against rollback to 2026.4.23; a checked-in synthetic fixture would still improve maintainer verification.

## Solution Assessment

Unclear. The current mitigations are directionally useful, but because v2026.4.29 already contained several of them and still reproduced for the reporter, the best fix needs maintainer triage and a narrower root-cause split before implementation.
