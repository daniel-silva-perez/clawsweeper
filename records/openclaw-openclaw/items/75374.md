## Summary

The issue should stay open because current main still has a launchd-specific path where durable state-dir `.env` values are converted into managed-key metadata and removed before the LaunchAgent env file is generated. The result matches the report: the wrapper can source a file that lists managed keys without exporting their values.

## What This Changes

The issue reports that gateway service-env regeneration records managed environment key names but omits the corresponding exported values, causing launchd gateways to start without env-backed auth tokens or other managed secrets.

## Best Possible Solution

Add a focused macOS launchd service-env fix so durable state-dir `.env` and config-managed env values listed in `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` are actually exported into the owner-only LaunchAgent env file, while keeping secrets out of plist metadata and keeping the systemd behavior unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: state-dir `.env` values become durable environment entries, `writeManagedServiceEnvKeysToEnvironment` removes those entries and records only the key list, and the LaunchAgent writer serializes only the remaining passed entries into the sourced env file.

## Solution Assessment

Yes, the best fix is in the launchd service-env regeneration/handoff path, not in gateway auth resolution. The safer direction is to mirror systemd's source-file behavior or otherwise keep the durable values available to the owner-only launchd env file without reintroducing plaintext plist environment metadata.
