## Summary

Keep this open. Current main has the generic unsupported-keyword sanitizer and passes selected model compat into tool schema normalization, but the bundled Fireworks Kimi K2.5 Turbo catalog row does not declare any unsupported tool-schema keywords, and no duplicate or fixing PR was found during the review.

## What This Changes

The issue asks OpenClaw to mark the bundled Fireworks Kimi K2.5 Turbo model as incompatible with JSON Schema keywords such as `not` so tool schemas are stripped before Fireworks requests.

## Best Possible Solution

Add provider-owned Fireworks schema compatibility metadata through the existing model `compat.unsupportedToolSchemaKeywords` contract, cover the Fireworks catalog/default-model path with a regression test, and avoid adding Fireworks-specific branching in core tool normalization.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Live provider reproduction is unavailable without Fireworks credentials, but the source-level path is clear: a schema containing `not` is preserved unless the selected model supplies `unsupportedToolSchemaKeywords`, and the Fireworks default model currently supplies none.

## Solution Assessment

Is this the best way to solve the issue? Yes: using the existing manifest/model compat hook is the narrowest maintainable fix and keeps the provider-specific behavior in the Fireworks plugin instead of core.
