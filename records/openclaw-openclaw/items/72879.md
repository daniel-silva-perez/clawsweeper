## Summary

Keep this open: current main still has a native Google transport gap for the reported Gemini missing-`thoughtSignature` case. The code can replay a signature that is already present on a native Google tool-call block, but it does not recover one when native streaming/replay lost it, while the reporter provided current 2026.4.27 live reproduction evidence for the exact 400.

## What This Changes

Fix a native Google Generative AI regression where Gemini tool-calling turns can replay `functionCall` parts without the required `thoughtSignature` and fail with HTTP 400.

## Best Possible Solution

Keep the issue open and fix the native Google provider path so Gemini tool-call signatures are retained or recovered for same provider/model current-turn replay. The fix should be provider-scoped, should not weaken cross-provider stripping, and should add native Google regression tests for missing/chunked signatures, already-signed calls, parallel calls, and cross-provider boundaries.

## Reproduction Assessment

Yes. The reporter provided a current 2026.4.27 native Google CLI reproduction and exact 400, and current-main source inspection shows the serializer can still emit unsigned native `functionCall` parts when the stored tool-call block lacks `thoughtSignature`.

## Solution Assessment

Yes, the best direction is a narrow native Google transport/replay fix rather than closing against the earlier sanitizer or same-part round-trip work. A safe implementation should mirror the compat-path reinjection only when the signature source is same provider/API/model and trustworthy.
