## Summary

Keep this open. Current main still has the native Google path that can serialize Gemini `functionCall` parts without a `thoughtSignature` when the in-memory tool-call block lacks one, while the issue discussion includes a recent native-Google Gemini 3 reproduction and Google's contract requires current-turn function-call signatures or documented transfer dummy signatures.

## What This Changes

The issue requests fixing a Gemini native Google provider regression where tool-calling turns in thinking mode can replay `functionCall` parts without required `thought_signature` metadata.

## Best Possible Solution

Add a narrow native Google replay repair that preserves real same-model signatures byte-for-byte, recovers or merges signatures from the same streamed response when available, and uses Google's documented dummy-signature behavior only for current-turn transferred or deterministic function-call history that cannot have a real signature. Keep the existing cross-provider real-signature stripping boundary and add regression tests for signed replay, missing native signatures, transferred/fallback history, and parallel function calls.

## Reproduction Assessment

Yes. The issue includes a concrete native Google Gemini 3 thinking/tool-calling reproduction, and current source inspection shows the same unsigned `functionCall` payload shape can still be built when the block lacks `thoughtSignature`.

## Solution Assessment

Yes for keeping this open and routing a focused fix PR. The narrowest maintainable solution is provider-scoped native Google signature repair, not a broad transcript sanitizer or cross-provider signature reuse.
