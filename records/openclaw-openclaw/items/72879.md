## Summary

Keep this open: current main preserves native Google `thoughtSignature` only when it was already captured on the tool-call block, but the reported missing-signature replay/fallback case can still serialize unsigned Gemini `functionCall` parts. The issue discussion also includes a current native Google reproduction, and Google's documented Gemini 3 contract makes this a real provider-boundary bug rather than a stale duplicate.

## What This Changes

The issue reports a Gemini native Google provider regression where tool-calling turns can replay `functionCall` parts without required `thought_signature` metadata after multi-turn or fallback history handling.

## Best Possible Solution

Keep the issue open and add a native Google replay repair for current-turn `functionCall` parts that lack a trustworthy signature. The fix should preserve real same-model signatures byte-for-byte, keep cross-provider stripping of real signatures, and handle transferred/fallback history using Google's documented dummy-signature path or an equivalent provider-scoped sanitization that prevents Gemini thinking mode from receiving invalid unsigned current-turn tool calls.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path from the discussion: native Google Gemini 3 with thinking enabled, multi-turn tool use, and an unsigned current-turn `functionCall` replay produces the documented 400; current-main source inspection shows native serialization can still emit that unsigned shape.

## Solution Assessment

Yes for keeping this open and fixing native Google replay narrowly. The best implementation is not another broad sanitizer or blind cross-provider signature reuse; it should be a provider-scoped native Google policy for missing current-turn tool-call signatures, with tests for real signatures, dummy/transfer handling, parallel calls, and cross-provider boundaries.
