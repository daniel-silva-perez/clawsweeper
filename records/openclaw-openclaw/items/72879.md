## Summary

Keep this open: current main still has a native Google transport gap for the reported missing-signature case, and the latest release comments provide concrete 2026.4.27 reproduction evidence. The existing implementation handles already-present tool-call signatures, but it does not repair or re-inject a missing native Google `thoughtSignature` the way the OpenAI-compatible Google path does.

## What This Changes

The issue reports that native Google Gemini tool-calling requests can still lose required `thoughtSignature` data in 2026.4.25 through 2026.4.27, causing 400 errors during thinking-mode tool turns.

## Best Possible Solution

Keep the issue open and fix the native Google provider path so Gemini tool-call signatures are preserved or recovered for same provider/model replay without leaking provider-specific signatures across model/provider boundaries. The fix should be covered by native Google transport tests for missing/chunked signatures, normal signed function calls, parallel calls, and cross-provider stripping.

## Reproduction Assessment

Yes: the reporter provided a current 2026.4.27 CLI command and exact 400 error, and current-main source inspection shows the same missing-signature path is still possible when a native Google tool-call block lacks `thoughtSignature`. I did not run a live Gemini request because no Google key is configured here and this review is read-only.

## Solution Assessment

Yes, the maintainable direction is a narrow Google provider transport/replay fix rather than closing against the previous same-part round-trip work. A safer repair should preserve exact same-provider signatures and add a native equivalent of the compat-path reinjection only where the signature source is trustworthy.

## Repair Work Prompt

Fix the native Google Gemini `thoughtSignature` loss reported in https://github.com/openclaw/openclaw/issues/72879. Current main captures native Google tool-call signatures only from `part.thoughtSignature` in `extensions/google/transport-stream.ts` and serializes only `block.thoughtSignature`; the OpenAI-compatible Google path has `injectToolCallThoughtSignatures`, but native `google-generative-ai` has no equivalent recovery. Keep the change inside the Google provider transport/replay boundary unless a small generic seam is clearly needed. Preserve exact signatures for same api/provider/model replay, preserve Google's parallel function-call behavior, and keep cross-provider/model stripping intact. Do not add dependencies, broad core migrations, startup config repair, or dummy skip signatures unless a maintainer explicitly approves that behavior. Add focused unit tests in `extensions/google/transport-stream.test.ts` for the failing missing/chunked native signature case, the existing already-signed case, no-signature absence, parallel function calls, and cross-provider/model replay. Update `docs/reference/transcript-hygiene.md` only if the behavior contract changes, and add a changelog fix entry crediting @M00zyx and @yuxiaoyang2007-prog if changelog policy applies.
