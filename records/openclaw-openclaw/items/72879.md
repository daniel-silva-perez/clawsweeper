## Summary

Keep this open. Current main and the latest shipped tag still preserve native Google `thoughtSignature` only when the tool-call block already has one; they do not repair a missing current-turn Gemini function-call signature or apply Google's documented transferred-history dummy-signature path. The issue discussion includes a recent native-Google Gemini 3 reproduction, and the source/tests still leave that missing-signature case uncovered.

## What This Changes

The issue asks OpenClaw to fix native Google Gemini tool-calling replay so `functionCall` parts are not sent without the required `thought_signature` after thinking/tool-use turns or fallback history.

## Best Possible Solution

Add a native Google replay repair for Gemini current-turn tool calls that lack a trustworthy signature, preserving real same-provider signatures byte-for-byte, keeping cross-provider signature stripping, and using Google's documented transferred-history behavior or an equivalent provider-scoped sanitization for unsigned transferred calls.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path from the discussion: native Google Gemini 3 with thinking enabled and multi-turn tool use produces the documented 400, and current-main source inspection shows that an unsigned same-route tool-call block can still be serialized unsigned.

## Solution Assessment

Yes for keeping this open and fixing the native Google path narrowly. The best fix is not a broad sanitizer or cross-provider signature reuse; it should be a Google-owned replay/serialization repair with tests for real signatures, missing signatures, transferred history, parallel calls, and cross-provider boundaries.
