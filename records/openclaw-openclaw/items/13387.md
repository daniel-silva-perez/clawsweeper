## Summary

Current main still does not implement the requested replay-context pruning for consecutive assistant `NO_REPLY` / silent-reply turns. The embedded runner sanitizes, validates, heartbeat-filters, limits history, and repairs tool-use/result pairing, but there is no pass that collapses consecutive silent replies. The linked implementation PR #13388 is useful prior art but is closed and unmerged.

## Best Possible Solution

Keep this issue open and implement a focused in-memory replay-context prune in the embedded runner history pipeline. The pass should collapse only consecutive assistant silent replies, keep at most one by default, reuse shared silent-token detection, handle legacy string/text-block/JSON-envelope and empty-content silent stop shapes, and add focused replay-context tests. Treat closed #13388 as prior art, not as merged work.

## Clownfish Work Prompt

Implement a focused fix for this issue: collapse consecutive assistant silent replies in the in-memory replay context sent to the model, keeping at most one consecutive silent reply by default. Work in the embedded runner/replay-history boundary; do not rewrite persisted transcripts and do not add broad startup/load-time migrations. Reuse shared silent-token detection from `src/auto-reply/tokens.ts` (`isSilentReplyPayloadText` / `SILENT_REPLY_TOKEN`) rather than ad hoc matching. Handle assistant content shapes seen in transcripts: legacy string content, text blocks, JSON `{"action":"NO_REPLY"}` envelopes, and legitimate empty-content silent stop turns. Preserve isolated silent replies, mixed-content assistant replies, non-assistant messages, and tool-use/tool-result adjacency. Prefer a small helper such as `src/agents/pi-embedded-runner/no-reply-pruning.ts` plus unit tests, wired into `src/agents/pi-embedded-runner/run/attempt.ts` after replay validation/heartbeat filtering/history limiting and before final model state assignment. Closed #13388 is prior art but should not be copied blindly; incorporate the known review feedback about robust detection and config edge cases. Avoid adding a public config knob unless the surrounding config schema/help/docs/generated artifacts are updated. Add a user-facing changelog entry if the behavior ships, crediting `Thanks @Masha-L`.
