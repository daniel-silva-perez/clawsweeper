## Summary

Keep open. Current main still lacks the requested replay-context pruning pass for adjacent assistant silent replies, and the linked implementation PR is closed/unmerged, so there is remaining narrow product behavior to implement rather than a cleanup close.

## What This Changes

The issue requests automatic pruning of adjacent assistant `NO_REPLY` transcript entries so replay context keeps at most one consecutive silent assistant reply.

## Best Possible Solution

Add documented in-memory replay-history hygiene in the embedded runner that collapses only adjacent assistant silent replies, keeps one by default, reuses the shared silent-reply payload detector, handles legacy string/text-block/strict JSON silent envelopes and known empty silent-stop shapes, and leaves stored JSONL unchanged unless maintainers explicitly choose storage-level cleanup.

## Reproduction Assessment

Yes. A replay history containing adjacent assistant turns with `NO_REPLY` or strict `{"action":"NO_REPLY"}` silent envelopes will pass through the current sanitize/validate/heartbeat-filter/limit flow without being collapsed unless it happens to match the separate heartbeat prompt/ack filter.

## Solution Assessment

Yes. The narrow maintainable path is a replay-context pruning helper wired into the existing embedded-runner history hygiene flow with focused tests and docs; the closed prior PR is useful prior art, but a replacement should avoid adding a broad config surface unless maintainers explicitly want one.
