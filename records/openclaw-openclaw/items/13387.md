## Summary

Keep open. Current main still lacks the requested replay-context pruning pass for adjacent assistant silent replies: the embedded runner sanitizes, validates, heartbeat-filters, limits, and repairs history, but no step collapses consecutive `NO_REPLY` or strict JSON silent-envelope assistant turns. The linked implementation PR is closed and unmerged, so it remains useful prior art rather than shipped behavior.

## What This Changes

Add transcript/context hygiene that prunes consecutive assistant `NO_REPLY` messages from session replay context while preserving isolated silent replies.

## Best Possible Solution

Add documented in-memory replay-history hygiene in the embedded runner that collapses only adjacent assistant silent replies, keeps one by default, reuses the shared silent-reply payload detector, handles legacy string/text-block/strict JSON silent-envelope and known empty silent-stop shapes, and leaves stored JSONL unchanged unless maintainers explicitly choose storage-level cleanup.

## Reproduction Assessment

Yes. A replay history containing adjacent assistant turns with `NO_REPLY` or strict `{"action":"NO_REPLY"}` silent envelopes will pass through the current sanitize/validate/heartbeat-filter/limit flow without being collapsed unless it happens to match the separate heartbeat prompt/ack filter.

## Solution Assessment

Yes. The narrow maintainable path is a replay-context pruning helper wired into the existing embedded-runner history hygiene flow with focused tests and docs; the closed prior PR is useful prior art, but a replacement should avoid adding a broad config surface unless maintainers explicitly want one.
