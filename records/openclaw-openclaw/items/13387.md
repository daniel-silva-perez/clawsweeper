## Summary

Current main still does not implement replay-context pruning for consecutive assistant NO_REPLY/silent-reply turns. The linked implementation PR #13388 was useful context but is closed and unmerged, and current source still only has narrower heartbeat filtering, UI/history suppression, delivery suppression, and provider replay hygiene.

## Best Possible Solution

Keep this issue open and implement a focused in-memory replay-context prune in the embedded runner history pipeline. The pass should collapse only consecutive assistant silent replies, keep at most one by default, reuse shared silent-token detection, handle legacy string/text-block/JSON-envelope and empty-content silent stop shapes, and add focused replay-context tests. The closed #13388 is useful prior art but should be revived with the review feedback incorporated rather than treated as merged.

## Clownfish Work Prompt

Implement replay-context pruning for consecutive assistant silent replies for openclaw/openclaw. Start from the observable bug in this issue and prior art in closed PR #13388, but do not copy its stricter detector blindly. Add a focused helper that identifies assistant-only silent replies using existing shared token semantics (`isSilentReplyPayloadText` / `SILENT_REPLY_TOKEN`) and also accounts for current replay shapes such as legacy string content, text-block arrays, JSON `{"action":"NO_REPLY"}` envelopes, and legitimate empty-content `stopReason:"stop"` silent turns where appropriate. Integrate the pass in the embedded runner history pipeline, likely after `filterHeartbeatPairs` / `limitHistoryTurns` and before assigning `activeSession.agent.state.messages`, while preserving isolated silent replies, user messages containing NO_REPLY, mixed-content assistant replies, non-text/tool content, tool-call/tool-result pairing, and provider-specific replay validation. Keep at most one consecutive assistant silent reply by default. If you add a public config knob, update config types, zod schema/help/generated docs, user docs, and include a single-line changelog entry thanking @Masha-L; otherwise document the in-memory default behavior in transcript/session pruning docs. Do not change channel delivery suppression, UI history projection, heartbeat filtering semantics, or unrelated transcript repair behavior.
