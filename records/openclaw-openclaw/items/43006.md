## Summary

Keep open. Current main still implements pre-compaction memory flush as one scalar threshold, one boolean gate, one flush turn, and one per-compaction-cycle metadata marker. It does not implement progressive 80%/90%/95% stages, stage-specific prompts/results, persisted per-stage dedupe state, or a forced-summary handoff. The related RFC PR #43102 was closed unmerged, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open for maintainer design. A complete implementation likely belongs across the Gateway memory-flush runner and memory-core plugin contract: add staged threshold configuration, a staged decision/result type, per-stage persisted dedupe state, stage-specific prompts, a defined forced-summary or compaction handoff, config schema/help/docs updates, and focused tests for multi-stage behavior.

## Risks / Open Questions

- A progressive implementation would add more silent model turns, so maintainers need to decide acceptable latency, cost, duplicate durable-memory writes, and observability.
- The current memory plugin API is too narrow for a third-party plugin to express staged flushes cleanly; this likely needs an additive core/plugin contract rather than plugin-local work only.
