## Summary

Keep open. Current main has a pre-compaction memory flush, including fixes for prior timing/undercount concerns, but it is still a single-plan, single-threshold, once-per-compaction-cycle mechanism. The requested progressive 80%/90%/95% staged reminders, stage-specific prompts, persisted per-stage state, and forced-summary handoff are not implemented. The related docs-only PR #43102 was closed unmerged and does not supersede the feature request.

## Best Possible Solution

Keep this issue open for maintainer design. A complete path likely belongs in the gateway memory-flush pipeline plus an additive memory-plugin contract update: staged threshold configuration, stage-specific prompts, persisted per-stage completion/dedup state, and a defined forced-summary or compaction handoff, with config schema, docs, and tests updated together.

## Risks / Open Questions

- A progressive flush design would add extra model turns and needs maintainer product judgment for latency, prompt noise, duplicate durable-memory writes, per-stage persistence, and the final forced-summary/compaction handoff.
- Current plugin and config contracts appear too narrow for a third-party plugin to implement the requested staged behavior without a core/API extension.
