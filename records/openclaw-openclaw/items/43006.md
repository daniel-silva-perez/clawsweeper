## Summary

Keep open. Current main still has pre-compaction memory flush, but it remains a single scalar threshold and a single silent flush turn per compaction cycle. The issue’s requested progressive 80%/90%/95% stages, stage-specific prompts/results, and forced-summary handoff are not implemented. The related RFC PR #43102 was closed unmerged, so it does not supersede the remaining product/API work.

## Best Possible Solution

Keep this issue open for maintainer design. A complete implementation likely belongs across the Gateway memory-flush runner and memory-core plugin contract: staged threshold config, staged plan/result types, per-stage persisted dedupe state in sessions.json, stage-specific prompts, defined forced-summary or compaction handoff semantics, docs/schema/help updates, and focused tests for multi-stage behavior.

## Risks / Open Questions

- A progressive implementation would add more background model turns near compaction, so maintainers should explicitly decide latency, cost, duplicate durable-memory writes, failure handling, and observability semantics.
- The current memory plugin API returns one flush plan, so a clean implementation likely needs an additive core/plugin contract rather than plugin-local workarounds.
