## Summary

Keep open. Current main still implements memory flush as a single scalar threshold and one silent embedded Pi flush per compaction cycle. It does not have progressive 80%/90%/95% stages, stage-specific prompts, persisted stage state, or a forced-summary handoff. The related RFC PR was closed unmerged and does not supersede this issue.

## Best Possible Solution

Keep this issue open for maintainer design. A complete fix likely belongs at the Gateway plus memory-core plugin contract boundary: add staged threshold configuration, a staged decision/result type, per-stage persisted dedupe state, stage-specific prompts, a defined forced-summary or compaction handoff, and matching config schema, docs, and focused tests.

## Risks / Open Questions

- A progressive implementation would add extra silent model turns, so maintainers need to decide acceptable latency, cost, duplicate durable-memory writes, and user-visible observability.
- The current memory plugin API is too narrow for a third-party plugin to express staged flushes cleanly; this likely needs an additive core/plugin contract rather than only plugin-local work.
