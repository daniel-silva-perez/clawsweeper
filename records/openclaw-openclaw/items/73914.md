## Summary

Current main implements the WhatsApp post-408 app-silence recovery path described by the report, with regression coverage for a reconnected listener that keeps receiving transport frames while application messages stay silent. The fix is proven on current main only; no local release tag contains the proof commit, so this should close as implemented on main rather than shipped in a known release.

## What This Changes

The issue reports a WhatsApp Web regression where post-408 reconnects log as listening but do not restore inbound message delivery, especially in dual-account/group setups.

## Best Possible Solution

Keep the current WhatsApp plugin-owned reconnect recovery and group-inbound fallback on main, then let the reporter upgrade to the next release that includes it; any remaining live failure should come back with fresh current-main or post-release logs.

## Reproduction Assessment

Yes. The issue gives a concrete dual-account WhatsApp 408/app-silent sequence, and current main has a focused regression test that recreates the key post-408 silent-app-delivery condition with continued transport frames.

## Solution Assessment

Yes. The fix is in the WhatsApp owner module, keeps core generic, and targets the failure mode by reopening the listener when app delivery stays silent after a reconnect rather than relying on credential wipes or broad gateway restarts.
