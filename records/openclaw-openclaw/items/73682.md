## Summary

Keep open. Current main still matches the reported failure shape: Claude CLI credential caching is keyed only by the credentials-file path, so a `allowKeychainPrompt: false` runtime read can cache `null` and a later keychain-capable external-sync read can reuse that `null` within the TTL. If the expired `anthropic:claude-cli` profile is then resolved explicitly, `claude-cli` has no OAuth refresh adapter and the resolver still throws the exact `No credentials found for profile "anthropic:claude-cli"` error. The reporter also added live macOS evidence that the cache-poisoning sequence is plausible in the cron lane.

## Best Possible Solution

Keep this issue open. The narrowest product/code path is to separate Claude CLI credential cache entries by keychain access intent or source capability, then add a focused regression where a no-keychain null read is followed by an external-sync/keychain-capable read within the TTL and by an isolated cron/provider-auth path with expired `anthropic:claude-cli` stored credentials. The runtime should either adopt the fresh Claude CLI keychain credential before returning null or emit a clear diagnostic explaining why the bootstrap was skipped/refused.

## Risks / Open Questions

- No live macOS Keychain cron reproduction was run in this read-only Linux checkout; the reporter's follow-up supplies the live-environment evidence, but maintainers should still capture it in an automated regression or a targeted macOS smoke before landing a fix.
- Changing the cache key or bootstrap behavior needs care so background cron does not unexpectedly trigger keychain prompts while still allowing keychain-capable sync paths to see fresh credentials.
