## Summary

Keep this issue open. Current main still derives Feishu group routing only from the normalized `chat_type`, so the reported `chat_type: "p2p"` topic/open-chat event shape still takes the direct-message routing path; the supplied GitHub context also shows open PR #52342 explicitly linked as an unmerged implementation candidate.

## What This Changes

Feishu/Lark topic-group messages reported as `p2p` should be classified as group/topic messages so replies route back to the originating Feishu topic instead of the web/main direct session.

## Best Possible Solution

Keep the tracker open and review/land PR #52342 if it safely handles the reported event shape, or replace it with a focused Feishu plugin patch that centralizes chat classification using reliable topic evidence such as `thread_id`, `root_id`, and/or verified chat metadata. The end state should update bot routing and mention-forward classification with regression tests for p2p topic/open-chat events, true p2p DMs, and card-action contexts.

## Reproduction Assessment

Yes. A synthetic Feishu message event with `chat_type: "p2p"`, an `oc_...` chat id, and topic context would set `isGroup` to false on current main, skip `resolveFeishuGroupSession`, and route as a direct peer.

## Solution Assessment

Unclear for the currently open implementation candidate because the supplied PR summary uses an `oc_` prefix heuristic, while existing history says bare `oc_` targets can be ambiguous. The safest maintainable solution should use topic/root evidence or verified chat metadata and cover true p2p regressions.
