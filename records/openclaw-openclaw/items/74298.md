## Summary

Keep open. Current main does not show an OpenClaw path that strips Telegram or Discord token lines from the state-directory `.env`, but the requested migration hardening is still not implemented in the migration guide or as a targeted doctor preflight.

## What This Changes

The issue requests investigation and hardening for migrations where Telegram and Discord bot token lines are missing from the copied state-directory `.env`.

## Best Possible Solution

Document a migration-time state `.env` verification step in the public migration guide, recommend `openclaw status --all` or channel probes after cutover, and only extend doctor where OpenClaw has evidence that a specific env token is expected, such as an explicit env SecretRef or managed service env source.

## Reproduction Assessment

No for the reported rsync stripping: the issue gives a production incident, but current source inspection does not expose an OpenClaw mutation path and no local reproduction was feasible under the read-only review constraints. Yes for the hardening gap: current migration docs and doctor/status docs show no dedicated copied-`.env` token integrity checklist.

## Solution Assessment

Unclear for the requested doctor preflight as written, because warning on absent optional channel env vars would be noisy. The safer path is docs plus targeted diagnostics only when config or service metadata proves the token should exist.
