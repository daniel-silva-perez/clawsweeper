## Summary

Keep open. Current main does not show an OpenClaw path that strips Telegram or Discord token lines from the trusted state-directory `.env`, but the migration-hardening request is still not fully covered: the migration guide lacks an explicit `.env` token-integrity check, and doctor has channel-specific/generic diagnostics rather than a migration preflight for expected copied credentials.

## What This Changes

The issue requests investigation plus migration hardening for cases where `TELEGRAM_BOT_TOKEN` and `DISCORD_BOT_TOKEN` are missing after copying the OpenClaw state directory `.env`.

## Best Possible Solution

Keep the issue open for scoped migration hardening: add a migration checklist that verifies state-dir `.env` presence/key integrity and recommends post-cutover `openclaw status --all` or channel probes, and only add doctor logic where OpenClaw can infer a configured/enabled credential is expected without warning on every optional absent env var.

## Reproduction Assessment

No, we do not have a high-confidence way to reproduce the reported rsync line stripping against current main. The high-confidence verification here is source/docs inspection: OpenClaw currently accepts these token keys when present, while the migration documentation and preflight coverage remain incomplete.

## Solution Assessment

Unclear for the broad requested doctor preflight. A targeted docs update plus diagnostics for configured/enabled credential paths would be maintainable, while a blanket missing-env-var warning would be noisy and could mislead operators.
