## Summary

Keep open. Current main does not show an OpenClaw code path that strips Telegram or Discord token lines from the state-dir `.env`, but the requested migration hardening is still not fully covered: the migration guide lacks a token-integrity checklist and doctor only has generic/conditional channel diagnostics, not a migration preflight for copied env keys.

## What This Changes

The issue asks OpenClaw to investigate missing Telegram/Discord bot-token lines after rsync migration and add doctor plus migration-documentation safeguards.

## Best Possible Solution

Keep the issue open for a scoped migration-hardening decision: document state-dir `.env` verification and post-cutover `openclaw status --all` or channel probes in the migration guide, and only add doctor logic where OpenClaw can infer an expected credential from configured/enabled channel state or an explicit SecretRef without warning on every absent optional env var.

## Reproduction Assessment

No, we do not have a high-confidence reproduction for the reported rsync line stripping. The high-confidence check here is source/docs inspection: current main accepts the token keys when present, while the migration docs and preflight coverage remain incomplete.

## Solution Assessment

Unclear for the broad requested doctor check. A targeted docs update plus narrowly scoped diagnostics for configured credentials would be maintainable; a blanket missing-env-var doctor warning would be too noisy and could mislead users.
