## Summary

Keep this issue open. Current main has substantial fixes for parts of the duplicate-inbound family, including Telegram accepted-offset tracking, Slack same-ts suppression, and replay-unsafe inbound dedupe poisoning after provider/runtime failures, but the issue discussion also documents a subagent announce path where the same logical completion can be redelivered after run remap. Current main still builds subagent announce idempotency from `childSessionKey + childRunId`, which is exactly the unstable identity called out in the later reproduction comments.

## Best Possible Solution

Keep this issue open as the higher-signal umbrella until maintainers either prove the subagent announce replay is fixed on current main or land a stable logical completion id for announce idempotency. The already-fixed Telegram offset and #69303 provider-error paths can be acknowledged in a maintainer update, then the remaining scope should be narrowed to durable inbound/announce replay protection and a regression test that simulates orphan/remap with the same childSessionKey producing only one external delivery.
