## Summary

Keep open. Current main has partial duplicate-inbound hardening for Telegram, Slack, and generic `MessageSid` dispatch, but the umbrella report is not proven fixed: the subagent completion announce path still keys idempotency by `childSessionKey + childRunId`, matching the reported resume/remap bypass class.

## What This Changes

Ensure the Gateway injects each external inbound message or logical subagent completion into a session only once across channel retries, restarts, and run remaps.

## Best Possible Solution

Keep this open as the canonical gateway duplicate-injection umbrella until maintainers either prove current main covers every reported path or land a narrower fix for the remaining announce/remap identity gap. The desired end state is durable or otherwise restart-safe idempotency for inbound channel events and a stable logical completion identity for subagent announces, with channel-specific tests for Telegram, Slack, and the subagent resume/remap flow.

## Reproduction Assessment

Yes for the affected releases: the issue discussion includes concrete Telegram `message_id` repeats, Slack transcript repeats, and subagent run-remap timing evidence. I did not run a live current-main reproduction in this read-only review, but source inspection still shows the announce key shape matching one reported bypass.

## Solution Assessment

Unclear. The current partial mitigations are necessary, but the remaining subagent announce identity appears to need a stable logical completion key rather than another `childRunId`-scoped cache; maintainers should confirm the intended identity boundary before a fix PR.
