## Summary

Keep open. Current main has several later fixes for plan-only GPT/Codex behavior and async exec delivery, but the latest concrete reproduction in the discussion still matches current code: `heartbeat.every = "0m"` disables the heartbeat runner before `exec-event` completion wakes can dispatch a follow-up turn.

## What This Changes

Fix a regression where Codex/OpenAI GPT coding-agent runs acknowledge work but fail to complete or report final results, especially after long-running background exec work yields.

## Best Possible Solution

Implement a focused heartbeat/exec-event dispatch fix: targeted exec completion wakes should run a one-shot completion relay even when the periodic heartbeat interval is disabled, while preserving the global heartbeat enable switch, channel visibility rules, and normal no-heartbeat behavior for ordinary interval wakes.

## Reproduction Assessment

Yes. A high-confidence source reproduction is: set `agents.defaults.heartbeat.every` to `0m`, start an exec that backgrounds after `yieldMs`, let it finish with output, and observe that the completion event requests `exec-event` but the heartbeat runner skips dispatch as disabled before processing it.

## Solution Assessment

Yes, the narrowest maintainable direction is to let `exec-event` wakes bypass the per-agent cadence gate without re-enabling scheduled heartbeats. Restoring a separate codex-cli backend or changing commentary/final-answer handling would be broader and would not address the narrowed disabled-heartbeat failure.
