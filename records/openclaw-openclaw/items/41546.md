## Summary

Keep open. Current main still has the reported Overview input behavior, the same-author linked bug #41545 is still open, and this PR is an implementation candidate rather than obsolete. The exact patch needs maintainer review or rework because it would preserve a gateway token across arbitrary WebSocket endpoint changes, conflicting with the current selected-gateway token scope.

## Best Possible Solution

Keep this PR open for maintainer review or author rework. The best path is a focused UI fix that separates draft URL editing from confirmed endpoint changes: preserve the typed token for same-target or whitespace-only edits, but clear or require explicit confirmation/token entry when the normalized gateway target changes, with regression coverage for both cases and no backend auth or storage schema change.

## Risks / Open Questions

- Preserving `settings.token` for every URL edit can reuse a credential entered for one gateway against another endpoint if the user changes the WebSocket URL and clicks Connect.
- The current code/docs/tests treat dashboard tokens as scoped to the selected gateway URL, so the PR's exact behavior would regress that boundary even though the reported UX bug is real.
- The best fix needs to avoid clearing tokens on harmless in-progress edits while still clearing, confirming, or requiring token re-entry when the normalized gateway target actually changes.
