## Summary

Keep open. The PR targets a real Control UI typing-lag path, but current main still has reactive draft state and the proposed approach is incomplete against the current chat input code because other textarea paths still request host updates on normal input.

## What This Changes

The PR removes Lit `@state()` from `OpenClawApp.chatMessage` and adds explicit draft-sync `requestUpdate()` calls around abort, failed-send restore, and history recall paths.

## Best Possible Solution

Rework the fix on current main so draft typing is decoupled from full app renders, slash-menu/accessibility updates only request a host refresh when their visible state actually changes, and non-reactive draft sync points cover send, abort, history recall, command insertion, and failure restore with focused UI tests.

## Reproduction Assessment

Yes. The code-level reproduction path is clear on current main: typing triggers `handleInput`, which calls `updateSlashMenu`, which unconditionally invokes the host update callback, and `chatMessage` is still reactive.

## Solution Assessment

No. Removing `@state()` is a reasonable direction, but this PR is not the best current-main fix as-is because it leaves another per-keystroke host-update path intact and must be adapted to the newer history-key controller flow.

## Review Findings

Full review comments:

- **[P2] Gate slash-menu host updates:** `ui/src/ui/views/chat.ts:1099-1100`
  - body: Removing `@state()` from `chatMessage` will not stop current-main typing from re-rendering the host: every textarea input still calls `updateSlashMenu(...)`, and that helper unconditionally calls `props.onRequestUpdate` even for plain text. Gate that request to actual slash-menu state changes, or the reported lag path remains after this patch.
  - confidence: 0.88
- **[P2] Refresh recalled drafts on the current history path:** `ui/src/ui/views/chat.ts:1059-1066`
  - body: After `chatMessage` becomes non-reactive, current-main history recall must explicitly refresh the host. The current `props.onHistoryKeydown` branch mutates the draft through the history controller and returns without calling `requestUpdate`, so ArrowUp/ArrowDown recall can leave the textarea value stale after this PR is rebased.
  - confidence: 0.82
