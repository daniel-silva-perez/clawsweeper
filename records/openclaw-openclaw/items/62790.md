## Summary

Keep this PR open. Current main still has `chatMessage` as a Lit `@state()` property, and the chat input path still writes every textarea input into that property, so the PR’s central performance fix is not implemented on main. The provided PR diff is narrow UI code with no security-sensitive file changes, so the right maintainer action is review/land or request focused tests rather than close it as obsolete.

## Best Possible Solution

Keep this PR open as the active implementation candidate. The best path is a focused Control UI change that makes chat draft edits non-reactive or otherwise avoids full `OpenClawApp` renders on every keystroke, keeps explicit DOM refreshes for abort, failed send restore, and input-history recall, and adds a small regression/performance-oriented test or manual proof for those paths.
