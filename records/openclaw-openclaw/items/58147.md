## Summary

Keep open. Current main still formats Control UI/WebChat timestamps from browser/runtime locale helpers and does not expose or consume `agents.defaults.timeFormat` in the UI session data path. The report is not strong enough to close as implemented or cannot-reproduce, but the exact expected output is partly missing because the issue body is truncated.

## What This Changes

Request that Control UI/WebChat session duration and timestamp display honor `agents.defaults.timeFormat` rather than relying only on browser/default time formatting.

## Best Possible Solution

Maintainers should first decide the UI time-format contract: keep browser-locale behavior and document it, or add a supported UI/Gateway preference that can drive chat/session timestamp renderers. If the config should apply, expose the preference through a typed UI-facing payload, use a shared formatter that matches the existing 24-hour leading-zero behavior, and cover the affected chat/session timestamp surfaces with focused UI tests.
