## Summary

Current main still appears to ignore `agents.defaults.timeFormat` for Control UI chat timestamps, so this should not be stale-closed. The setting is implemented for prompt/current-time formatting, but the browser UI timestamp formatter uses locale defaults directly and the `sessions.list` defaults payload does not expose `timeFormat` to the UI.

## What This Changes

Request that Control UI/WebChat session duration and timestamp displays honor `agents.defaults.timeFormat`.

## Best Possible Solution

Expose a browser-safe time-format value to the relevant UI surface, then apply explicit 12h/24h formatting to Control UI/WebChat timestamp and duration renderers; if maintainers want `timeFormat` to remain prompt-only, update docs or open a separate UI preference issue instead.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the report because the expected output is truncated and this read-only pass did not launch the UI. Source inspection does give high confidence that the current Control UI chat timestamp path cannot honor `agents.defaults.timeFormat`.

## Solution Assessment

Unclear until maintainers confirm whether `agents.defaults.timeFormat` should govern UI display as well as prompt text. If yes, the narrowest maintainable fix is to thread the setting through the gateway UI payload and update the UI formatter/tests rather than hardcoding another local preference.
