## Summary

Keep this PR open. Current main has the handler registry and SDK export pieces, but it still does not implement the PR's central behavior: suppressing transient network uncaught exceptions such as ENETUNREACH/ENOTFOUND at the CLI entrypoints and routing Telegram polling uncaught network failures through the restart path. The PR discussion also includes fresh production corroboration for Telegram polling stalls and Discord voice ENOTFOUND crashes, and the linked bug report #60515 remains open.

## Best Possible Solution

Keep this PR open for maintainer review, rebase, or a direct current-main port. The best path is to preserve fatal hooks, add narrowly reviewed ENETUNREACH/ENOTFOUND-class uncaught suppression where appropriate, wire Telegram polling recovery through registerUncaughtExceptionHandler as well as registerUnhandledRejectionHandler, add regression tests for Telegram uncaught polling HttpError recovery and CLI ENETUNREACH/ENOTFOUND uncaught suppression, and resolve the duplicate legacy-handler logging concern before landing.

## Risks / Open Questions

- The PR needs a current-main rebase or careful port because current main now runs fatal-error hooks for truly fatal uncaught exceptions, and those hooks must remain on fatal paths.
- Broad top-level transient network suppression can hide unrelated uncaught bugs if an outer fatal error contains a nested transient-looking network cause; maintainers should choose between direct top-level suppression and narrower scoped handlers deliberately.
- The Greptile duplicate-handler concern remains relevant if both src/index.ts and runCli install equivalent uncaughtException suppression on the legacy CLI path.
- No supply-chain or CI security regression was visible in the PR file list, but the runtime exception policy change is security-sensitive because it changes when the process continues after an uncaught exception.
