## Summary

Keep this PR open. Current main still awaits startup catch-up before the first cron timer is armed, and persistence failures in that path can still escape before `armTimer(state)` runs. The PR remains focused candidate work for a real core cron scheduler resilience gap. Security review did not find supply-chain-sensitive file changes in the provided PR file list; the remaining concern is correctness of subtle cron recovery behavior.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it onto current main and either land it after targeted validation of startup catch-up failure resilience, one-shot interruption semantics, timeout-disabled jobs, and persistence-failure paths, or replace it with an equivalent main-branch fix that guarantees `start()` arms the cron timer even when startup catch-up or follow-up persistence fails.
