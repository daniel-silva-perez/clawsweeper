## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main has deliberately moved Telegram setup to numeric-only `allowFrom` entries, while this branch still advertises and tests `@username` setup resolution; the remaining target-resolution hardening is not implemented on main and depends on a shared string-only resolver contract decision.

## What This Changes

The PR widens Telegram setup allowFrom and runtime target resolver inputs to tolerate null, undefined, and non-string values, normalizes tokenFile inspection with `normalizeOptionalString`, and adds a setup-surface test for malformed allowFrom entries.

## Best Possible Solution

A mergeable fix should preserve numeric-only Telegram allowFrom setup, keep legacy username repair in doctor/fix paths, and either validate malformed resolver inputs at the shared caller boundary or add narrowly tested Telegram-only coercion without widening the public SDK contract accidentally.

## Reproduction Assessment

Yes for the stale setup behavior: current main can be checked by calling `telegramSetupWizard.allowFrom.resolveEntries` with `entries: ["@user"]`, which the existing test expects to return unresolved without fetching. For non-string target inputs, reproduction is unclear because the supported CLI/core resolver path trims and passes strings.

## Solution Assessment

No, this PR is not the best way to solve the issue as-is. It mixes useful defensive hardening with obsolete setup username resolution; the safer path is a rebase that preserves numeric-only setup and makes an explicit resolver-contract decision.
