## Summary

Keep this PR open. Current main already superseded the setup `allowFrom` username-resolution part with numeric-only Telegram setup, but it has not implemented the PR's defensive target-resolution coercion or the tokenFile status normalization; deciding whether to widen those guards despite the current string-only resolver contract needs maintainer review rather than conservative cleanup.

## What This Changes

The PR widens Telegram setup allowFrom and target resolver inputs to tolerate null, undefined, and non-string values, replaces one tokenFile trim with `normalizeOptionalString`, and adds a setup-surface test for malformed allowFrom entries.

## Best Possible Solution

A mergeable fix should preserve numeric-only Telegram allowFrom setup, drop the stale setup username-resolution pieces, and either keep the resolver contract string-only with validation at the caller boundary or add explicit, tested defensive coercion in Telegram without accidentally widening the public SDK contract.
