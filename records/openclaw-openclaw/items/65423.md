## Summary

Keep this PR open. Current main still builds auth profile candidates in a deterministic order for embedded subagent runs, so the requested thundering-herd mitigation is not implemented or superseded, but the proposed patch needs a rebase/revision before merge because it does not account for current provider-preferred profile ordering and its test validates a local helper copy rather than production behavior.

## What This Changes

The PR adds a Fisher-Yates shuffle helper, uses `params.spawnedBy` to randomize auth profile candidate order for subagent embedded runs, and adds a unit test file for the shuffle helper.

## Best Possible Solution

Keep the PR open and revise it into a production-owned candidate-ordering change that disperses automatic spawned embedded runs while preserving locked-profile behavior, provider-preferred profile precedence, cooldown handling, and auth-order eligibility semantics.

## Reproduction Assessment

Yes for the source-level ordering bug: a focused embedded-run candidate-selection test with multiple eligible profiles and `spawnedBy` set should show current main starts from the same ordered candidate list. The live 429 symptom itself is not high-confidence without provider-specific concurrency credentials.

## Solution Assessment

No, not as currently proposed. The direction is narrow and useful, but the maintainable fix should be rebased onto current main, preserve provider-preferred ordering, and test production candidate selection rather than a copied shuffle helper.
