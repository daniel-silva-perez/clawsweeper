## Summary

Keep open. Current main still lacks the PR's dispatcher-idle barrier before queued followup execution, and the related #65761 path is closed unmerged rather than a shipped replacement. The PR is useful directionally but not merge-ready because the new pre-run wait is not abort-aware, so a followup can still reply after a user stop request if it is already dequeued and parked behind dispatch idle.

## What This Changes

The PR adds a dispatcher-idle waiter to reply options, wraps generic reply dispatch in a dispatch-ready promise, and makes queued followups wait before starting and after dispatcher-based followup delivery.

## Best Possible Solution

Keep the PR open as a useful but incomplete implementation candidate. The desired end state is a turn-scoped, abort-aware followup dispatch barrier that waits only for the final reply delivery that caused the followup, tracks followup outbound sends until they drain, preserves `/stop` cancellation before embedded-run startup, and has regression coverage for dispatch ordering, stop during the wait window, and multiple queued followups.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor stop while waiting for dispatch idle:** `src/auto-reply/reply/followup-runner.ts:140`
  - body: The new pre-run wait runs before the followup creates a reply operation or starts `runEmbeddedPiAgent`. `drainNextQueueItem()` has already handed this item to `run()`, so `/stop` can clear the queue and abort active embedded runs without cancelling this parked followup; once the dispatcher drains, it can still start and reply after the stop request. Make the idle wait abort-aware or keep the dequeued followup registered with the stop path until it either starts or is cancelled.
  - confidence: 0.9
