## Summary

Keep open. Current main still lacks the PR's intended dispatcher-idle barrier before followup runs, and the related #65761 replacement was closed unmerged. The PR remains a plausible implementation candidate, but the review discussion identifies ordering and abort-handling regressions that should be fixed before merge.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to rework the implementation into a turn-scoped dispatch barrier: wait only for the dispatcher/final delivery that produced the queued followup, re-track followup outbound sends until they drain, make /stop cancel followups parked before embedded run startup, and add regression coverage for newer-turn isolation, multiple queued followups, and abort during the drain window. If a later canonical PR lands that complete fix, close this PR as superseded after that merge.
