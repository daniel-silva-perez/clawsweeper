## Summary

Keep open. Current main does not implement the proposed Credential Provider Plugin capability: the SDK has no credential-provider registration method, captured registration has no credential-provider collection path, and the shipped secrets docs still describe SecretRef as static eager snapshot resolution. The RFC is also not a ClawHub-only item because it asks for a missing cross-plugin/core capability and trust boundary, which the project docs explicitly route through plugin API design when a plugin cannot build the feature alone.

## What This Changes

The issue proposes a new Credential Provider Plugin capability with SDK registration, a credential resolution engine, CredentialRef config, lifecycle hooks, session scoping, and bridges to existing SecretRef and model-provider auth flows.

## Best Possible Solution

Keep this as an RFC until maintainers decide the credential trust model and scope. If accepted, implement it as a shared capability: core types and registry, SDK registration, runtime resolution helper, config schema, one proof provider plugin, consumer migration points, contract tests, and operator docs.

## Reproduction Assessment

Not applicable: this is an architecture RFC rather than a bug report. The relevant verification is source inspection, which shows the proposed SDK API and credential resolution engine are not present on current main.

## Solution Assessment

Unclear: the direction fits the documented capability model, but the RFC is too broad and security-sensitive for an automated fix. The narrowest maintainable path is a maintainer-approved phased design before any code PR.
