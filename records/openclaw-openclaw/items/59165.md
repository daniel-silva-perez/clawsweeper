## Summary

Keep this issue open. Current main has static SecretRef support and provider-specific auth hooks, but it does not implement the RFC's proposed CredentialProvider plugin capability, CredentialRef config, cross-plugin credential registry, requestCredential-style runtime helper, or lifecycle engine. The discussion also points to useful future implementations, including an AWS Secrets Manager provider offer tied to PR #64318, but that comment explicitly depends on the core interface landing first.

## Best Possible Solution

Keep the RFC open as the canonical design thread for an additive core credential capability. The best path is to define a narrow CredentialProvider contract in core, expose api.registerCredentialProvider plus a runtime request helper, add CredentialRef/config/schema/docs, preserve SecretRef for static injection, and land one reviewed provider implementation as proof. ClawHub is the right home for additional providers after the core API exists.

## Risks / Open Questions

- The design is security-sensitive: any implementation should include an explicit trust model, consumer authorization, redaction boundaries, per-session scoping rules, and lifecycle failure behavior before plugins can share credentials broadly.
