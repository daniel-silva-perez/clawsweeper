## Summary

Keep open. Current main still implements SecretRef as static, eager snapshot resolution over env/file/exec sources and does not expose a credential-provider plugin registration or credential-resolution engine. This is not a ClawHub-only cleanup close because the request is for a missing shared core/plugin capability and trust boundary that external plugins cannot implement cleanly without SDK/runtime work.

## What This Changes

The issue proposes a first-class Credential Provider Plugin capability for lifecycle-managed credential acquisition, refresh, rotation, health checks, per-session scoping, and cross-plugin consumption alongside existing SecretRef behavior.

## Best Possible Solution

Keep this as an active RFC until maintainers decide the credential trust model and phase plan. If accepted, implement it as a shared capability with core types, SDK registration, runtime helper, config/schema/docs, contract tests, and at least one proof provider plugin before migrating consumers.

## Reproduction Assessment

Not applicable: this is an architecture RFC rather than a bug report. Source inspection is the verification path, and it shows the proposed SDK registration and credential-provider runtime are not present on current main.

## Solution Assessment

Unclear. The direction fits OpenClaw's documented capability model, but the proposal is too broad and security-sensitive for an automated fix; the best next step is maintainer/secops architecture review before code.
