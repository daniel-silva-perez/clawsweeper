## Summary

Keep this issue open. Current main still has validation, temp-file rename, rejected/clobbered recovery, and last-known-good restore safeguards, but I did not find a lock or queue that serializes OpenClaw-owned config mutations across the full read/merge/write transaction; the only related fix PR is closed unmerged, and the newer related issue is a distinct partial-snapshot writer problem.

## What This Changes

The issue requests serializing concurrent OpenClaw-owned writes to the user config file so parallel config mutations cannot produce concatenated invalid JSON or lose a user turn before recovery.

## Best Possible Solution

Land a focused core config fix that makes OpenClaw-owned config mutations a single serialized transaction, using the existing lock infrastructure plus any needed same-process queue, while preserving the current validation/recovery semantics and adding a deterministic concurrent-writer regression test.

## Reproduction Assessment

Unclear: the reporter supplied plausible live audit evidence and quarantined concatenated snapshots, and the current code shape still exposes an unsynchronized mutation window. This read-only review did not run a race harness, and the repo does not appear to contain a deterministic concurrent config-writer regression test yet.

## Solution Assessment

Yes for the proposed direction: the narrow maintainable fix is to serialize the full config mutation transaction, not only the final disk replacement. A safer implementation should cover `replaceConfigFile`, `mutateConfigFile`, direct `writeConfigFile`, include writes, and runtime refresh/notification as one ordered write path.
