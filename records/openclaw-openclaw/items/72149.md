## Summary

Keep open. Current main still forwards `temperature` into native Anthropic Messages payloads for non-thinking requests, while the PR applies a narrow model-specific guard and a focused regression test for Claude Opus 4.7. I found no blocking correctness or security finding in the proposed patch; the remaining action is maintainer review and validation, not cleanup closure.

## What This Changes

The PR normalizes Claude Opus 4.7 model-id matching, skips `temperature` injection for that model in the native Anthropic transport, and adds a regression test asserting the payload omits `temperature`.

## Best Possible Solution

Land this PR or an equivalent native Anthropic transport patch that strips `temperature` for Claude Opus 4.7 while preserving current behavior for other Anthropic models, then run the targeted transport test and the normal changed gate before merge.

## Reproduction Assessment

Yes. Current main has a direct source-level reproduction path: pass `claude-opus-4-7` with `options.temperature = 0.2` and no reasoning, and `buildAnthropicParams` adds `temperature`; related #72123 supplies live logs showing the matching Anthropic 400 response.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable solution for this native transport request-shape bug: remove the unsupported field at Anthropic payload construction for Opus 4.7 only. A broader sampling-parameter filter is not needed in this PR because current native Anthropic code does not appear to forward `top_p` or `top_k`.
