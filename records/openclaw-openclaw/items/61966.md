## Summary

Keep open. The reported autocomplete gap still exists on current main, but this PR is not ready as-is because it reads plugin commands from the TUI process singleton; in gateway mode, plugin commands are registered in the Gateway process and are exposed through the existing commands.list catalog instead.

## What This Changes

The PR imports listPluginCommands() into src/tui/commands.ts and appends registered plugin command names/descriptions to getSlashCommands() autocomplete after built-in and gateway command entries.

## Best Possible Solution

Route terminal TUI autocomplete through the same command catalog used by the Gateway and Control UI for gateway mode, add a local embedded catalog path if needed, preserve local-command precedence and de-dupe behavior, and add focused regression coverage for plugin command autocomplete.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: register or mock a Gateway plugin command, connect the terminal TUI in gateway mode, and observe that current autocomplete is built from local getSlashCommands() rather than the Gateway commands.list result that contains plugin entries.

## Solution Assessment

No. The proposed local listPluginCommands() loop is too narrow because it reads the TUI process registry, while the normal Gateway-connected TUI needs command data from the Gateway process; a backend-backed command catalog is the safer maintainable fix.
