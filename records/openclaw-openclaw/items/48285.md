## Summary

Current main still does not implement this PR's intended gateway behavior. Device-less node-role loopback sessions are still rejected, backend self-pairing still excludes auth.mode="none", docs/tests continue to describe node clients as using normal device identity and pairing, and the proposed change remains security-sensitive because it broadens auth/pairing bypass behavior.

## Best Possible Solution

Keep this PR open for maintainer review or ask for a rebased, narrower replacement tied to the confirmed runtime rejection path. A strong fix should prove the actual cron, sessions_spawn, and ACP caller identities, cover auth.mode="none" if that remains broken, and scope any loopback node-role exemption to non-Control-UI direct-local sessions with regression tests for valid auth, failed auth, and remote rejection.
