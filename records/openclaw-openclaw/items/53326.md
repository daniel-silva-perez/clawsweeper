## Summary

Keep open. Current main and release v2026.4.27 still write workspace setup state under the nested dot-prefixed path, so the compatibility cleanup is still relevant, but the PR head should not merge as-is because the fallback, migration, filename-collision, and test surfaces still need revision.

## What This Changes

The PR changes `src/agents/workspace.ts` so workspace setup state is written at `<workspace>/workspace-state.json`, legacy state is fallback-read from `<workspace>/.openclaw/workspace-state.json`, and writes no longer create the dot-prefixed state directory.

## Best Possible Solution

Revise this PR around the same goal: keep workspace setup state out of dot-prefixed workspace entries, choose a collision-resistant non-dot filename or guard against foreign files, make legacy dot-path probes best-effort, promote successfully loaded legacy state to the new location, and add focused tests for first-run, legacy migration, collision handling, and dot-path I/O failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` now probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-ENOENT errors. On the dot-restricted filesystems this PR targets, that can make status checks and initialization fail before the new root state is created; catch and ignore legacy-probe I/O errors in the fallback paths.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber a project file with the same generic name. If that file parses without OpenClaw markers, the setup flow treats it as empty state and later overwrites it, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates `<workspace>/workspace-state.json`. Existing workspaces keep depending on the dot-prefixed directory and can lose completion markers if that directory is unavailable; persist fallback-loaded state to the new path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, release, package metadata, or secrets-handling changes were found, but the generic root state filename creates a concrete local data-integrity risk.

Concerns:

- **[medium] Generic root state file can clobber workspace data:** `src/agents/workspace.ts:207`
  - body: Moving OpenClaw state to `<workspace>/workspace-state.json` without namespacing or a foreign-file guard can overwrite an existing workspace file with the same name, which is a data-integrity regression for user-managed workspaces.
