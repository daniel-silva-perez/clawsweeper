## Summary

Keep open: current main and the latest release still use the nested `.openclaw/workspace-state.json` state path, so the compatibility goal is not obsolete, but the proposed patch has blocking data-integrity and migration issues before merge.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the dot-prefixed state directory.

## Best Possible Solution

Revise and rebase this PR around the same compatibility goal: keep workspace setup state out of dot-prefixed workspace entries, use a collision-resistant non-dot filename or a foreign-file guard, treat legacy dot-path probes as best-effort, promote successfully loaded legacy state to the new location, and add focused coverage for first-run setup, legacy migration, foreign-file collision handling, and dot-path I/O failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy fallback reads best-effort:** `src/agents/workspace.ts:269`
  - body: The new fallback probes `.openclaw/workspace-state.json` whenever the root state has no markers, including first run. Because `readWorkspaceSetupState()` still rethrows non-`ENOENT` I/O errors, the dot-restricted filesystems this PR targets can still abort setup or status before the new root state is written.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber a project file with the same generic name. A foreign JSON file with no OpenClaw markers is treated as empty state and later overwritten, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace()` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on the dot-prefixed directory and can lose completion markers if that directory is unavailable.
  - confidence: 0.86
