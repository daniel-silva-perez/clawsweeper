## Summary

Keep open. Current main and the latest release still store workspace setup state under the nested `.openclaw/workspace-state.json` path, so the PR is addressing a real unresolved compatibility cleanup, but the proposed patch has blocking migration and data-integrity issues before merge.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the dot-prefixed state directory.

## Best Possible Solution

Revise this PR around the same compatibility goal: keep workspace setup state out of dot-prefixed workspace entries, choose a collision-resistant non-dot filename or detect foreign files before writing, treat legacy dot-path probes as best-effort, promote successfully loaded legacy state to the new location, and add focused tests for first-run, legacy migration, collision handling, and dot-path I/O failures.

## Reproduction Assessment

Yes. The blocking issues are reproducible from the PR diff and current code path: first-run fallback probes the legacy dot path and rethrows non-`ENOENT` errors, a root `workspace-state.json` without OpenClaw markers can be overwritten later, and fallback-loaded legacy state does not mark the new path dirty.

## Solution Assessment

No. The goal is valid, but this patch is not the safest maintainable fix until legacy probes are best-effort, legacy state is promoted, and the root state filename cannot clobber unrelated workspace data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` now probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-`ENOENT` errors. On dot-restricted filesystems, setup and status checks can still fail before the new root state is written.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber an existing project file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so use a safer name or guard foreign files.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on the dot-prefixed directory and can lose markers if that path is unavailable.
  - confidence: 0.86
