## Summary

Keep this PR open, but do not merge it as-is. Current main still has the nested dot-prefixed workspace-state behavior the PR is trying to fix, while the PR head has blocking correctness and local data-integrity issues that need author or maintainer revision.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the `.openclaw` state directory.

## Best Possible Solution

Revise and rebase this PR around the same goal: keep setup state out of dot-prefixed workspace entries, choose a collision-resistant non-dot filename or detect foreign files before writing, make legacy dot-path probes best-effort, promote successfully loaded legacy state to the new location, and update focused workspace tests plus any fixtures that encode the state path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` now probes `.openclaw/workspace-state.json` whenever the new root state lacks markers, but `readWorkspaceSetupState()` still rethrows non-ENOENT errors. On the dot-restricted filesystems this PR targets, that can make status checks or initialization fail before the new root state is written; catch and ignore legacy-probe I/O failures.
  - confidence: 0.92
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber a project file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.89
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates `<workspace>/workspace-state.json`. Existing workspaces keep depending on the dot-prefixed directory and can lose completion markers if that directory is unavailable; persist fallback-loaded state to the new path.
  - confidence: 0.87
- **[P2] Update state-path tests for the new resolver:** `src/agents/workspace.ts:207`
  - body: The branch changes writes to `<workspace>/workspace-state.json`, but `src/agents/workspace.test.ts` still reads `<workspace>/.openclaw/workspace-state.json` after `ensureAgentWorkspace()`. Those existing tests will fail with ENOENT and miss coverage for the new path and legacy migration.
  - confidence: 0.83
