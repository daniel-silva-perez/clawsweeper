## Summary

Keep open. Current main and the latest release still write workspace setup state under the nested dot-prefixed path, so the compatibility problem remains real, but this PR should not merge as-is because the branch still has blocking fallback, migration, collision, and test-coverage issues.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the dot-prefixed state directory.

## Best Possible Solution

Revise this PR around the same goal: keep workspace setup state out of dot-prefixed workspace entries, choose a collision-resistant non-dot filename or guard against foreign files, make legacy dot-path probes best-effort, promote successfully loaded legacy state to the new location, and add focused tests for first-run, legacy migration, collision handling, and dot-path I/O failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-ENOENT errors. On the dot-restricted filesystems this PR targets, first-run and status checks can still fail before the new root state is created; catch and ignore legacy-probe I/O failures.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber a project file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Persist fallback-loaded state to the new path:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates `<workspace>/workspace-state.json`. Existing workspaces keep depending on the dot-prefixed directory and can lose markers when that directory is unavailable; write the migrated state to the new path.
  - confidence: 0.86
- **[P2] Update tests to follow the new state path:** `src/agents/workspace.test.ts:32-39`
  - body: At PR head, the workspace test helper still reads `<workspace>/.openclaw/workspace-state.json` while the branch writes `<workspace>/workspace-state.json`, so the existing workspace tests will fail and will not cover the new path or fallback migration behavior.
  - confidence: 0.83
