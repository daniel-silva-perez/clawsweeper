## Summary

Keep this issue open. Current `main` still marks fatal isolated cron/provider failures as error runs, but keeps an `isError: true` delivery payload and passes it through the normal announce dispatch path. The exact raw provider JSON is now more likely to be formatted as `cron isolated run failed: ...`, but failed runs can still be delivered to user-facing announce channels instead of being limited to run history and the separate failure-alert path.

## Best Possible Solution

Keep this issue open. Add a guard at the isolated cron finalize or `dispatchCronDelivery` boundary so fatal/error-status runs skip normal `delivery.mode: "announce"` delivery, while still recording the failed run and sending configured failure alerts through the separate failure-notification path. Add focused regression coverage for provider/run-level errors with announce delivery and `bestEffort: false` asserting no outbound announce send occurs, plus coverage that explicit failure alerts still send sanitized messages.
