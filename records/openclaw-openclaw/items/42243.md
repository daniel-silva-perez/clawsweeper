## Summary

Keep open. Current main may sanitize the exact raw provider JSON now, but fatal cron/provider failures are still converted into `isError: true` delivery payloads and passed through normal `announce` delivery. The related implementation PR #46236 was closed unmerged, and the current cron path still lacks a failed-run announce suppression guard.

## Best Possible Solution

Keep this issue open. Add a guard at the isolated cron finalize or `dispatchCronDelivery` boundary so fatal/error-status isolated cron runs skip normal `delivery.mode: "announce"` delivery, while still recording the failed run and sending configured failure alerts through the separate failure-notification path. Add focused regression coverage for provider/run-level errors with announce delivery and `bestEffort: false` asserting no outbound announce send occurs, plus coverage that explicit failure alerts still send sanitized messages.

## Risks / Open Questions

- The exact raw JSON leakage appears partly mitigated by newer sanitization paths; the remaining live issue is still user-facing delivery of failed-run error text through normal announce delivery.
- A fix needs to preserve run history, `lastRunStatus: "error"`, retry/backoff accounting, and configured failure notifications while suppressing only ordinary announce delivery.
