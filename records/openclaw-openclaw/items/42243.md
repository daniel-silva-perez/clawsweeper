## Summary

Keep this issue open. Current main still turns run-level cron/provider failures into `isError: true` delivery payloads and passes them through normal `announce` delivery. The exact raw JSON may now be formatted as `cron isolated run failed: ...`, but the central behavior reported here remains: failed cron runs can still post error text to user-facing channels instead of only recording the failed run and using the separate failure-alert path.

## Best Possible Solution

Keep this issue open. Add a guard at the isolated cron finalize or `dispatchCronDelivery` boundary so fatal/error-status isolated cron runs skip normal `delivery.mode: "announce"` delivery, while still recording the failed run and sending configured failure alerts through the separate failure-notification path. Add focused regression coverage for provider/run-level errors with announce delivery and `bestEffort: false` asserting no outbound announce send occurs, plus coverage that explicit failure alerts still send sanitized messages.
