## Summary

Keep this issue open. Current main still returns `raw: null` when raw redaction cannot safely round-trip, the Control UI still treats that as Raw mode unavailable and forces Form mode, and the current docs/tests explicitly encode that behavior. The related #59330 tracks a likely false-positive cause for the fallback, but this issue's requested read-only Raw display/editability split remains unresolved; the direct fix PRs #67010 and #67212 were closed unmerged in the provided context.

## Best Possible Solution

Keep this open unless maintainers explicitly decide the current force-Form behavior is the intended policy. A durable fix should separate raw display availability from raw editability in the `config.get`/Control UI contract: preserve a safely redacted raw string for read-only viewing when possible, expose an explicit unsafe-round-trip/editability reason, keep save/apply blocked while unsafe, and update gateway, UI, docs, and regression tests. The narrower false-positive round-trip cause in #59330/#59336 can be handled alongside this, but it does not fully replace the read-only Raw display request here.

## Risks / Open Questions

- Preserving redacted raw for display while blocking edits changes the current security/product contract; maintainers should explicitly decide what redacted raw text may be shown to browser clients and expose editability separately from display availability.
- A one-line removal of `redactedRaw = null` would conflict with current docs and tests and could imply raw edits are safe when the gateway cannot round-trip the snapshot.
