## Summary

Keep open. Current main has backend pending-approval list/resolve primitives and event-driven Control UI approval modals, but it still lacks the requested discoverable Web dashboard or CLI queue for finding and acting on missed exec approvals. The action path can approve live host exec/plugin operations, so the remaining surface needs maintainer design rather than conservative cleanup closure.

## What This Changes

The issue requests a Web GUI and/or CLI surface to list pending exec approvals and approve or deny them when popup prompts are missed.

## Best Possible Solution

Add a first-class Pending approvals surface in the Control UI that loads existing `exec.approval.list` and `plugin.approval.list` records on connect/reconnect, updates from approval events, resolves through the existing scoped `*.approval.resolve` RPCs, shows command/plugin/session/agent/expiry/risk metadata, and clears stale entries. If maintainers want CLI recovery too, add narrowly scoped `openclaw approvals pending|approve|deny` commands that reuse the same RPCs and document both workflows.

## Reproduction Assessment

Yes for the central missing surface: current-main inspection shows pending-list backend RPCs exist, while the CLI is policy-only and the Control UI only renders event-driven modal prompts. The intermittent missed-popup delivery itself is unclear without channel logs or a live reproduction.

## Solution Assessment

Yes in direction, but the maintainable fix should reuse the existing `exec.approval.*` and `plugin.approval.*` pending RPCs rather than adding another pending store. The exact Web/CLI scope, authorization UX, audit behavior, and timeout handling need maintainer review first.
