## Summary

Keep open. Current main has backend pending-approval list/resolve RPCs and a transient Control UI modal for live approval events, but it still lacks the requested discoverable Web dashboard or CLI queue for finding and acting on missed exec approvals. The discussion correctly separates this live pending-approval queue from the existing persisted approval policy editor, and the approve/deny path is security-sensitive enough to need maintainer product/security direction before automation attempts a fix.

## What This Changes

Request a discoverable Web GUI and/or CLI queue for pending exec approval requests, with approve/deny actions, so users are not dependent on transient popups.

## Best Possible Solution

Add a first-class Pending approvals surface in the Control UI that fetches `exec.approval.list` and `plugin.approval.list` on connect, updates from approval events, shows command/plugin/session/agent/risk metadata, resolves through the existing `*.approval.resolve` RPCs under `operator.approvals`, and clears stale prompts. If maintainers want CLI recovery too, add narrowly scoped `openclaw approvals pending|approve|deny` commands and document both surfaces.

## Reproduction Assessment

Yes for the central missing surface: current main can be inspected to show the CLI is policy-only and the Control UI only renders a modal from live approval events. The intermittent missed-popup delivery itself is unclear without channel logs or a live reproduction.

## Solution Assessment

Yes in direction, but the maintainable solution should reuse the existing `exec.approval.*` and `plugin.approval.*` pending RPCs rather than adding a second pending store. The exact Web/CLI scope, authorization UX, audit behavior, and timeout handling need maintainer review first.
