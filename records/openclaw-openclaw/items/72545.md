## Summary

Keep open. Current main has backend pending-approval list/resolve RPCs and a transient Control UI modal, but it still lacks the requested discoverable Web dashboard or CLI queue for finding and acting on missed exec approvals; the issue discussion correctly distinguishes this live queue from the existing persisted approval policy editor.

## What This Changes

The issue requests a Web GUI or CLI surface that lists pending exec approvals and lets an operator approve or deny them when popup delivery is missed.

## Best Possible Solution

Add a first-class Pending approvals surface in the Control UI that fetches `exec.approval.list` and `plugin.approval.list` on connect, updates from approval events, shows command/plugin/session/agent/risk metadata, resolves through the existing `*.approval.resolve` RPCs under `operator.approvals`, and clears stale channel prompts. If maintainers want CLI recovery too, add narrowly scoped `openclaw approvals pending|approve|deny` commands and document both surfaces.

## Reproduction Assessment

Yes for the central missing surface: current main can be inspected to show the CLI is policy-only and the Control UI only renders a modal from live approval events. The intermittent channel-popup failure is still unclear without logs or a live channel reproduction.

## Solution Assessment

Yes in direction, but the implementation should reuse the existing `exec.approval.*` and `plugin.approval.*` pending RPCs rather than adding a second store. The exact Web/CLI scope, audit model, timeout handling, and authorization UX need maintainer product/security review first.
