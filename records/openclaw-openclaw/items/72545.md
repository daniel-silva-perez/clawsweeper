## Summary

Keep open. Current main has backend pending-approval list/resolve primitives and event-driven Control UI approval modals, but it still does not provide the requested discoverable Web dashboard or OpenClaw CLI queue for finding and acting on missed exec approvals. Because this would approve or deny live host exec/plugin requests, the remaining scope and security semantics need maintainer design review.

## What This Changes

The issue requests a Web GUI, and optionally CLI, surface to list pending live exec approvals and approve or deny them without relying on popup delivery.

## Best Possible Solution

Add a first-class Pending approvals surface in the Control UI that loads `exec.approval.list` and `plugin.approval.list` on connect/reconnect, updates from approval events, shows command/plugin/session/agent/expiry/risk metadata, resolves through the existing `*.approval.resolve` RPCs under `operator.approvals`, and clears stale entries. If maintainers want CLI recovery too, add narrowly scoped `openclaw approvals pending|approve|deny` commands and document both surfaces.

## Reproduction Assessment

Yes for the central missing surface: current-main inspection shows the backend queue exists, but the OpenClaw CLI is policy-only and the Control UI only renders event-driven modal prompts. The intermittent missed-popup delivery path remains unclear without live channel logs or a focused reproduction.

## Solution Assessment

Yes in direction, but the maintainable path should reuse current `exec.approval.*` and `plugin.approval.*` pending RPCs rather than adding a second pending store. The exact Web/CLI scope, authorization UX, audit behavior, and timeout rules are still maintainer design decisions.
