## Summary

Keep open. Current main still hardcodes the lane wait diagnostic warning default at 2,000 ms for queue entries without an explicit override, and the diagnostics config/schema/docs only expose stuck-session tuning, not lane wait warning tuning. The related implementation PR is closed unmerged, while the broader session concurrency issue is separate and does not supersede this focused logging-control request.

## What This Changes

The issue requests a configurable lane wait warning threshold to reduce diagnostic log noise from legitimately long-running queued cron jobs.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` field that defaults to the existing 2,000 ms behavior, supplies the default for future queue entries when `opts.warnAfterMs` is omitted, preserves explicit overrides, handles config reload without a gateway restart, updates schema/types/help/generated docs, and leaves per-lane/adaptive timeout or session concurrency work to separate product follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue work behind an active lane entry without passing `opts.warnAfterMs`, wait more than 2,000 ms, and current main emits `lane wait exceeded`; adding `diagnostics.laneWaitWarnMs` is also not accepted or documented because schema, types, labels, help, and docs lack that field.

## Solution Assessment

Yes. A narrow global diagnostics threshold is the best maintainable fix for this issue; per-lane thresholds, auto-yield, or session concurrency are broader product work and are not required to stop the reported log flood.

## Repair Work Prompt

Fix the lane wait diagnostic threshold requested by https://github.com/openclaw/openclaw/issues/14747. Current main hardcodes `opts?.warnAfterMs ?? 2_000` in `src/process/command-queue.ts`, which floods logs for legitimate long-running cron jobs. Add a positive millisecond `diagnostics.laneWaitWarnMs` config field that defaults to 2,000 ms and supplies the default for future `enqueueCommandInLane` entries when `opts.warnAfterMs` is omitted. Treat reload as a no-restart/default-update behavior and add/adjust regression coverage for queue behavior, gateway lane/config reload behavior, and cron override behavior. Update config schema/types/labels/help/generated schema docs, `docs/gateway/configuration-reference.md`, `docs/concepts/queue.md`, and add a user-facing changelog entry with proper attribution. Use PR #16366 only as prior context; do not copy its unrelated bundled identity/session/model-catalog changes. Leave per-lane/adaptive timeout and session concurrency/auto-yield work to #53023.
