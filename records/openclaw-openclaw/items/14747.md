## Summary

Keep open: current main still lacks a configurable lane wait diagnostic threshold. The queue warning still depends on a per-entry threshold with a 2,000 ms fallback, diagnostics config/types/docs expose `stuckSessionWarnMs` but no lane wait setting, the cron path has a separate explicit 5,000 ms override to handle deliberately, and the related implementation PR is closed unmerged while the broader concurrency issue does not supersede this narrow request.

## What This Changes

Add a configurable diagnostics lane wait warning threshold so long-running cron or session workloads can reduce `lane wait exceeded` log noise without disabling diagnostics.

## Best Possible Solution

Land a focused core change adding documented `diagnostics.laneWaitWarnMs` as a positive millisecond config value, defaulting to the existing 2,000 ms behavior when unset, preserving explicit per-call overrides unless deliberately changed, wiring reload/current-config behavior for future enqueues, and updating schema/types/help/generated docs, queue/config docs, tests, and changelog. Keep #53023 as separate broader concurrency and auto-yield work.
