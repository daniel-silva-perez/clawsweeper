## Summary

Keep open. Current main still hardcodes the lane wait diagnostic default for queue entries that omit `warnAfterMs`, and the diagnostics config/schema/docs only expose stuck-session tuning. Related PR #16366 is closed unmerged, while #53023 is broader session concurrency work and does not supersede this focused logging-control request.

## What This Changes

Request to make the lane wait diagnostic warning threshold configurable so long-running cron jobs do not flood gateway logs with hardcoded short-wait warnings.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` field that defaults to the existing 2,000 ms behavior, supplies the default for future queue entries when `opts.warnAfterMs` is omitted, preserves explicit per-call overrides, handles config reload without a gateway restart, updates generated config surfaces and docs, and leaves per-lane/adaptive timeout or session concurrency work to separate follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue work behind an active lane entry without passing `opts.warnAfterMs`, wait at least 2,000 ms, and current main emits `lane wait exceeded`; adding `diagnostics.laneWaitWarnMs` is also unsupported because schema, types, labels/help, and docs lack that field.

## Solution Assessment

Yes. A narrow global diagnostics threshold is the best maintainable fix for this issue; per-lane thresholds, adaptive timeout logic, session concurrency, and auto-yield are broader product work and are not required to stop the reported log flood.

## Repair Work Prompt

Implement the diagnostics lane wait warning threshold requested by https://github.com/openclaw/openclaw/issues/14747. Current main hardcodes omitted queue warning thresholds as `opts?.warnAfterMs ?? 2_000` in `src/process/command-queue.ts`, while the diagnostics config surface only supports stuck-session warning tuning. Add a positive millisecond `diagnostics.laneWaitWarnMs` config field that defaults to 2,000 ms when unset and is used for future `enqueueCommandInLane` entries when `opts.warnAfterMs` is omitted. Preserve explicit per-call overrides, including the manual cron `warnAfterMs: 5_000` path. Make config reload/no-restart behavior explicit in the reload plan or runtime wiring so changing the config affects future queue entries without a gateway restart. Update schema/types/labels/help/generated config docs plus `docs/gateway/configuration-reference.md`, `docs/concepts/queue.md`, and `CHANGELOG.md` with proper user credit. Add focused tests for the command queue default, explicit override preservation, gateway lane/reload behavior, and cron behavior. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, auto-yield, or unrelated changes from PR #16366; leave #53023 as separate product work.
