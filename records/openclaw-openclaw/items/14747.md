## Summary

Keep open. Current main still hardcodes the default lane wait warning threshold at 2,000 ms when callers omit `warnAfterMs`, and the diagnostics config schema/types/help/docs expose stuck-session tuning but no lane wait warning threshold. The closed related PRs did not merge, while #53023 is broader concurrency/auto-yield work rather than a replacement for this focused logging-control request.

## What This Changes

Request to add a configurable lane wait diagnostic warning threshold so long-running cron or background jobs do not flood logs with non-actionable `lane wait exceeded` warnings.

## Best Possible Solution

Add a documented `diagnostics.laneWaitWarnMs` positive millisecond config field that defaults to the current 2,000 ms behavior when unset, applies to future queue entries without explicit overrides, preserves per-call `warnAfterMs`, covers reload/no-restart behavior, updates generated config surfaces and public docs, and leaves concurrency or auto-yield behavior to #53023.
