## Summary

Keep open. Current main still stores the lane wait warning threshold on each queued entry and falls back to a hardcoded 2,000 ms default when callers do not pass `warnAfterMs`; the diagnostics config/schema/docs only expose stuck-session warning tuning. The closed related PR tried to add the same knob but did not merge, and the open broader concurrency/auto-yield request does not supersede this narrower log-noise control.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so long-running cron or background jobs do not flood logs with non-actionable `lane wait exceeded` warnings.

## Best Possible Solution

Add a documented `diagnostics.laneWaitWarnMs` positive millisecond config field that defaults to the current 2,000 ms behavior when unset, applies to future queue entries without explicit overrides, updates schema/types/help/generated docs, covers reload/no-restart behavior, and leaves broader session concurrency or auto-yield work to #53023.
