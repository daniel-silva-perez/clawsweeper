## Summary

Current main still does not provide a user-configurable lane wait diagnostic threshold. The queue warning default remains `opts?.warnAfterMs ?? 2_000`, scheduled cron agent work inherits that default through the embedded-runner lane path, manual cron enqueue still has its own fixed `5_000` ms threshold, the strict diagnostics config schema exposes no lane-wait field, and the related implementation PR #16366 is closed unmerged. This is a narrow, valid fix-PR candidate.

## Best Possible Solution

Keep this issue open and land a focused core patch that adds a documented `diagnostics.laneWaitWarnMs` positive-ms setting, keeps the default at 2000ms for compatibility, applies it to `enqueueCommandInLane` when no per-call `warnAfterMs` is provided, preserves explicit per-call overrides, updates gateway start/reload application, covers cron lane behavior, updates generated config metadata/docs/changelog, and adds regression tests.
