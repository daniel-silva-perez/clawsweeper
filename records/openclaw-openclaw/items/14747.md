## Summary

Keep open. Current main still emits lane wait diagnostics from a per-entry threshold and defaults omitted thresholds to a hardcoded 2,000 ms; the public diagnostics config only exposes stuck-session warning tuning. The linked implementation PR is closed unmerged and the broader concurrency request does not replace this narrower log-noise control.

## What This Changes

Add a configurable diagnostics lane-wait warning threshold so long-running cron or agent work can avoid repeated hardcoded two-second warnings without disabling diagnostics.

## Best Possible Solution

Expose a documented positive millisecond diagnostics.laneWaitWarnMs config field that defaults to the current 2,000 ms behavior, feeds the default for future queue entries when opts.warnAfterMs is omitted, preserves explicit per-call overrides, treats config reload as no-restart/default-update behavior, and leaves broader lane concurrency or auto-yield work to #53023.
