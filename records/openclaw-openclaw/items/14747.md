## Summary

Keep open. Current main still stores the lane wait warning threshold per queued entry and defaults omitted thresholds to a hardcoded 2,000 ms, while the diagnostics config/schema/docs expose only stuck-session warning tuning; the related implementation PR is closed unmerged and the broader concurrency issue does not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable diagnostics threshold for lane wait warnings so long-running cron and background jobs do not flood logs under the hardcoded short default.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` config field that defaults to the existing 2,000 ms behavior, supplies the default for future queue entries when `opts.warnAfterMs` is omitted, preserves explicit per-call overrides, treats config reload as no-restart/default-update behavior, and leaves session concurrency or auto-yield behavior to the broader tracking issue.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue work behind an active lane entry without passing `opts.warnAfterMs`, wait more than 2,000 ms, and the current code emits `lane wait exceeded`; adding `diagnostics.laneWaitWarnMs` is not accepted or documented because the schema and types lack that field.

## Solution Assessment

Yes, the narrow config field is the best maintainable direction for this issue. Per-lane or adaptive timeout work can be considered separately, but this report only needs a global diagnostics threshold that preserves existing defaults and explicit overrides.
