## Summary

Keep open. Current main still hardcodes the default lane wait warning threshold at queue enqueue time, and the public diagnostics config surface only exposes stuck-session warning tuning. The closed related PRs do not land the requested control, while #53023 is broader concurrency/auto-yield work rather than a substitute for this focused logging knob.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so long-running cron or external-tool jobs can avoid repeated hardcoded 2 second wait warnings.

## Best Possible Solution

Add a documented `diagnostics.laneWaitWarnMs` positive millisecond config field that defaults to the current 2,000 ms behavior when unset, applies to future queue entries without explicit overrides, updates schema/types/help/generated docs, preserves explicit per-call overrides, covers reload/no-restart behavior, and leaves broader concurrency or auto-yield behavior to #53023.
