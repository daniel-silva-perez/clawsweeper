## Summary

Keep open. Current main still defaults lane wait warnings to a hardcoded 2,000 ms per queued entry, and the diagnostics config/schema/docs expose stuck-session tuning but no lane wait warning threshold; the related implementation PR is closed unmerged, and the broader concurrency request does not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable diagnostics threshold for lane wait warnings so legitimate long-running cron and background jobs do not flood gateway logs under the hardcoded short default.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` config field that defaults to the existing 2,000 ms behavior, supplies the default for future queue entries when `opts.warnAfterMs` is omitted, preserves explicit per-call overrides, treats config reload as no-restart/default-update behavior, and leaves session concurrency or auto-yield behavior to the broader tracking issue.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue work behind an active lane entry without passing `opts.warnAfterMs`, wait more than 2,000 ms, and current main emits `lane wait exceeded`; adding `diagnostics.laneWaitWarnMs` is not accepted or documented because the schema and types lack that field.

## Solution Assessment

Yes, the narrow global config field is the best maintainable direction for this issue. Per-lane or adaptive timeout work can be considered separately, but this report only needs a diagnostics threshold that preserves existing defaults and explicit overrides.
