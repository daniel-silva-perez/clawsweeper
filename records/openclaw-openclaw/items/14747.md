## Summary

Keep open. Current main still defaults omitted lane wait warning thresholds to a hardcoded 2,000 ms value, while the diagnostics schema, generated config metadata, reload planning, and public docs only expose stuck-session warning tuning. The related implementation PR is closed unmerged, and the broader concurrency/auto-yield issue does not supersede this narrower log-noise control request.

## What This Changes

Add a configurable lane wait diagnostic warning threshold so long-running cron or session work can avoid repeated non-actionable queue wait warnings without disabling diagnostics.

## Best Possible Solution

Expose a documented positive millisecond diagnostics.laneWaitWarnMs config field that defaults to the current 2,000 ms behavior, feeds the default for future queue entries when opts.warnAfterMs is omitted, preserves explicit per-call overrides, treats reload as no-restart/default-update behavior, and leaves broader lane concurrency or auto-yield changes to #53023.
