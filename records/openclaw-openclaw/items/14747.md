## Summary

Keep open. Current main still resolves omitted lane wait warning thresholds to a hardcoded 2,000 ms value, and the diagnostics config schema, typed config surface, labels/help, and public docs still expose only stuck-session warning tuning. The related implementation PRs are closed unmerged, and the broader concurrency/auto-yield issue does not supersede this focused diagnostics log-noise request.

## What This Changes

Requests a configurable lane wait diagnostic warning threshold so long-running cron-style work can avoid repeated non-actionable `lane wait exceeded` warnings.

## Best Possible Solution

Add a documented `diagnostics.laneWaitWarnMs` positive millisecond field that defaults to the current 2,000 ms behavior, feeds the default used by future `enqueueCommandInLane` calls when `opts.warnAfterMs` is omitted, preserves explicit per-call overrides, updates reload planning as a no-restart dynamic/default update, and leaves concurrency or auto-yield behavior to #53023.
