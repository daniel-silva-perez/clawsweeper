## Summary

Keep open. Current main still has the lane wait diagnostic threshold as a hardcoded queue-entry default, while the diagnostics config/schema/docs only expose stuck-session warning tuning. The related implementation PR is closed unmerged, and the broader concurrency/auto-yield issue does not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable diagnostic threshold for lane wait warnings so long-running cron or lane work does not flood gateway logs.

## Best Possible Solution

Add a documented `diagnostics.laneWaitWarnMs` positive millisecond config field that defaults to the current 2,000 ms behavior when unset, applies to future queue entries without explicit overrides, updates schema/types/help/generated docs, covers reload/no-restart behavior, and leaves broader session concurrency or auto-yield work to #53023.
