## Summary

Keep this PR open. The proposed UI-only performance guard is still relevant: current main still requests and renders a 200-message initial Control UI chat window, while the PR wants 80. The PR should not be closed as implemented because main has not shipped the behavior, and it should not be closed as obsolete because the intended fix is narrow and plausible. It does need a current-main refresh: active rendering now lives in `ui/src/ui/chat/build-chat-items.ts`, while the PR diff still updates the older view-local render path.

## Best Possible Solution

Keep this PR open and ask for a current-main refresh. The best implementation is to define one shared initial chat history window constant near the existing chat history limits, use it from `ui/src/ui/controllers/chat.ts` and `ui/src/ui/chat/build-chat-items.ts`, decide whether `ui/src/ui/chat/input-history.ts` should intentionally share the same limit, then update `ui/src/ui/controllers/chat.test.ts` and add direct coverage in `ui/src/ui/chat/build-chat-items.test.ts`. No Gateway protocol, server, dependency, or security-sensitive change appears necessary.
