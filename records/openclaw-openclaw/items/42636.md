## Summary

Keep this PR open. Current main still uses a 200-message initial Control UI chat hydration request and a 200-message render window, so the PR’s intended 80-message performance guard is not implemented. The prior ClawSweeper review remains accurate: the PR is useful, but stale against current main because active render slicing now lives in the extracted chat item builder path.

## Best Possible Solution

Refresh the PR on current main by introducing one shared initial chat history/render window constant near the existing `ui/src/ui/chat/history-limits.ts` surface, using it from `ui/src/ui/controllers/chat.ts` and `ui/src/ui/chat/build-chat-items.ts`, deciding whether `ui/src/ui/chat/input-history.ts` should follow it, and adding direct coverage for both the controller request limit and `buildChatItems()` truncation notice. No gateway protocol, server, dependency, CI, or security-sensitive change appears necessary.
