## Summary

Current main does not implement this PR's intended 80-message Control UI hydration/render window. The chat controller still requests 200 history messages, the active chat item builder still renders the last 200 messages, and current tests still encode the old hydration limit. The PR remains plausible, but it needs a refresh because current render slicing lives in `ui/src/ui/chat/build-chat-items.ts`, not only the older view-local path shown in the PR diff.

## Best Possible Solution

Refresh this PR against current main by introducing or reusing one shared initial chat history/render window constant, wiring it through `loadChatHistory()` and `buildChatItems()`, deciding explicitly whether `input-history.ts` should stay aligned, and adding direct coverage in `ui/src/ui/controllers/chat.test.ts` plus `ui/src/ui/chat/build-chat-items.test.ts`. No gateway protocol, dependency, CI, or security-sensitive change appears necessary.
