## Summary

Keep open. Current main still has the inline skill tool-policy bypass, and this PR is pointed at the right security boundary, but the branch needs a current-main rebase/adaptation before merge so the coding-tool policy path receives the full inline context and preserves the current lazy runtime boundary.

## What This Changes

The PR changes inline skill `command-dispatch: tool` handling to select tools from `createOpenClawCodingTools` instead of raw OpenClaw tools, removes the standalone owner-only filter, and adds regression coverage for profile-filtered exec dispatch.

## Best Possible Solution

Rebase or adapt the PR on current main so inline skill tool dispatch resolves through a lazy coding-tools runtime wrapper around `createOpenClawCodingTools`, removes the standalone owner-only fallback, forwards the current route/session/group/sender/model/config context, and keeps focused regression tests for profile-disallowed tools plus policy-critical argument forwarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Forward the full inline dispatch context:** `src/auto-reply/reply/get-reply-inline-actions.ts:198-224`
  - body: The new `createOpenClawCodingTools` call does not carry over current inline-only context such as `allowGatewaySubagentBinding` and the sender/group identity fields used by group tool policy. On current main those values feed plugin subagent binding and policy resolution, so inline tool-dispatch skills can lose existing session behavior or evaluate policy with incomplete inputs after this change. Please pass the same route/session/sender/group context that normal coding sessions and the current inline tool path provide.
  - confidence: 0.87
- **[P3] Keep coding tools behind a runtime loader:** `src/auto-reply/reply/get-reply-inline-actions.ts:2`
  - body: Current main keeps inline action tool loading behind a runtime import to avoid pulling the full tool stack into the inbound inline-actions module. Replacing the raw-tools import with a static `pi-tools` import reintroduces that eager load; when rebasing, add an equivalent coding-tools runtime wrapper or lazy loader.
  - confidence: 0.78
