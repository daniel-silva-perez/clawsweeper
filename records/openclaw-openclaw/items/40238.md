## Summary

Keep open. Current main still has the inline skill `command-dispatch: tool` policy gap, but this PR is not merge-ready because the latest diff drops gateway subagent binding and still passes only partial sender identity into the group tool-policy path.

## What This Changes

The PR changes inline skill `command-dispatch: tool` lookup to use `createOpenClawCodingTools` and adds regression tests for non-owner messaging-profile dispatch.

## Best Possible Solution

Revise or replace the branch so inline skill tool dispatch uses the central coding-tool policy factory with the same complete runtime context as normal command tool inventory: gateway subagent binding, message target/thread, account, group/session fallback, sender id/name/username/E.164, model/provider, and focused regression tests for restricted non-owner and group `toolsBySender` cases.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main dispatches inline skill tools through `createOpenClawTools` plus owner-only filtering, while the PR diff can be checked by asserting the missing factory arguments and a group `toolsBySender` deny case.

## Solution Assessment

No. Routing through `createOpenClawCodingTools` is the right seam, but this patch is incomplete because it drops gateway binding and omits sender identity fields used by group policy; the safer solution is to thread the full existing runtime context through the central policy factory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve gateway subagent binding:** `src/auto-reply/reply/get-reply-inline-actions.ts:198`
  - body: The old inline dispatch path passed `allowGatewaySubagentBinding: true`. This `createOpenClawCodingTools` call omits that option, and the factory only forwards it when supplied, so plugin-backed command-dispatch tools that rely on gateway subagent or node-runtime binding can become unavailable.
  - confidence: 0.88
- **[P2] Pass the full sender policy context:** `src/auto-reply/reply/get-reply-inline-actions.ts:212-213`
  - body: This call now enforces group tool policy, but it only passes `senderId`. Group policy also matches `senderName`, `senderUsername`, and `senderE164`, so username/E.164/name-specific `toolsBySender` allow or deny rules can be skipped for inline skill dispatch.
  - confidence: 0.86
