## Summary

Keep open. The PR addresses a real security-sensitive policy gap: current main still selects inline skill-dispatch tools from `createOpenClawTools` with only owner-only filtering, while normal command sessions use the fuller `createOpenClawCodingTools` policy pipeline. It is not ready to merge as written because the patch omits policy context that the normal path forwards and conflicts with the current lazy tool-loading boundary.

## What This Changes

The PR changes inline skill `command-dispatch: tool` handling to resolve tools through `createOpenClawCodingTools` and adds regression tests for non-owner messaging-profile dispatch.

## Best Possible Solution

Inline skill tool dispatch should resolve callable tools through the same policy-resolving factory as normal command sessions, behind a lazy runtime wrapper, while forwarding the resolved target policy session key, route, sender identity, group/spawned metadata, model/provider context, and gateway subagent-binding context with focused regression coverage.

## Reproduction Assessment

Yes. A focused reproduction is to configure a narrow tool profile such as `messaging`, register a skill command that dispatches to a tool outside that profile or to a policy-filtered plugin tool, and verify current main can still select it because inline dispatch only applies owner-only filtering.

## Solution Assessment

No, not as written. The direction is right, but the maintainable fix should be rebased onto current main and should mirror `resolveCommandsSystemPromptBundle`'s policy context instead of passing a smaller inline-only argument set or importing the full tool stack directly.

  - confidence: 0.78
