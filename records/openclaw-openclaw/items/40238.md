## Summary

Keep open. Current main still has the inline skill tool-policy bypass, and this PR targets the right security boundary, but the patch is not merge-ready because the new coding-tool dispatch path still drops context that current main uses for group/sender policy and plugin subagent binding, and it eagerly imports the coding tool stack in a path that is currently lazy-loaded.

## What This Changes

The PR changes inline skill `command-dispatch: tool` handling to select tools from `createOpenClawCodingTools` instead of raw OpenClaw tools plus owner-only filtering, and adds regression coverage for messaging-profile non-owner tool filtering.

## Best Possible Solution

Inline skill `command-dispatch: tool` should resolve callable tools through the same policy-resolving factory used by normal command sessions, behind a lazy runtime wrapper, while forwarding target-session, route, sender, model, group, and plugin-binding context and keeping focused regression tests for profile-disallowed exec-style tools.

Those fields feed plugin subagent binding and group tool policy, so inline tool-dispatch skills can see a different tool set than the session that exposed the command.

  - body: Current main keeps inline action tool loading behind a runtime import so the inbound inline-actions module does not eagerly load the full tool stack. Replacing the runtime load with a static `pi-tools` import reintroduces that eager dependency; add a small runtime wrapper or lazy loader for the coding-tools factory when rebasing.
  - confidence: 0.78
