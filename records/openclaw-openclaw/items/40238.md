## Summary

Keep open. Current main still uses the narrower inline skill dispatch path, and this PR moves in the right direction but remains incomplete in a security-sensitive authorization path because it does not preserve gateway subagent binding or pass the full group/sender policy context into the central tool-policy factory.

## What This Changes

The PR changes inline skill `command-dispatch: tool` lookup to use `createOpenClawCodingTools` and adds regression tests for non-owner messaging-profile dispatch.

## Best Possible Solution

Revise this branch or replace it narrowly so inline skill tool dispatch calls `createOpenClawCodingTools` with the same complete context as normal message tool inventory: gateway subagent binding, message target/thread, account, target/group fallback, sender name/username/E.164, model/provider, session policy, and focused tests for restricted non-owner and group-policy cases.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main dispatches inline skill tools through `createOpenClawTools` plus owner-only filtering, while existing `createOpenClawCodingTools` tests show the messaging profile filters `exec`; the remaining PR regressions can be reproduced by asserting the full factory arguments and a group `toolsBySender` deny case.

## Solution Assessment

No. The chosen seam is the right maintainable direction, but this patch is incomplete because it drops existing gateway binding and omits sender identity fields used by group policy; the safer fix is to thread the full existing runtime context through the central coding-tool policy factory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve gateway subagent binding for plugin tools:** `src/auto-reply/reply/get-reply-inline-actions.ts:198`
  - body: The old inline dispatch path opted plugin tools into gateway subagent binding with `allowGatewaySubagentBinding: true`. This new `createOpenClawCodingTools` call does not pass that option, and the coding-tool factory only forwards it when present, so plugin-backed command-dispatch tools that use the gateway subagent or node runtime can become unavailable outside the Gateway.
  - confidence: 0.88

  - body: This call is now responsible for enforcing group tool policy, but it only passes session group fields and `senderId`. Group policy resolution also uses ctx-derived group fallback plus sender name, username, and E.164 values; omitting them can miss a group-specific sender deny/allow rule.
  - confidence: 0.84
