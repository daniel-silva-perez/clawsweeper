## Summary

Keep open. Current main still has the reported inline skill tool-policy bypass, so the PR is addressing a real security-sensitive gap, but the proposed patch is not merge-ready because its central policy call drops existing gateway subagent binding and does not pass the full group/sender policy context used by normal tool inventory paths.

## What This Changes

The PR switches inline skill `command-dispatch: tool` lookup from the OpenClaw tool factory plus owner-only filtering to `createOpenClawCodingTools` and adds regression tests for non-owner messaging-profile dispatch.

## Best Possible Solution

Revise this branch or replace it narrowly so inline skill tool dispatch uses `createOpenClawCodingTools` with the same complete context as normal message tool inventory: gateway subagent binding, message route, target/group fallback, sender identity fields, model/provider, session policy, and focused regression tests for restricted non-owner and group-policy cases.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main dispatches inline skill tools through `createOpenClawTools` plus owner-only filtering, while `createOpenClawCodingTools` tests show the messaging profile filters `exec`; the remaining PR regressions can be reproduced by asserting the arguments passed to the new factory call.

## Solution Assessment

No. The direction is the right maintainable seam, but this patch is incomplete because it drops existing gateway binding and passes incomplete group/sender policy context; the safer solution is to thread the full existing runtime context through the central coding-tool policy factory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve gateway subagent binding for plugin tools:** `src/auto-reply/reply/get-reply-inline-actions.ts:198`
  - body: The old inline dispatch path opted plugin tools into gateway subagent binding with `allowGatewaySubagentBinding: true`. This new `createOpenClawCodingTools` call does not pass that option, and the coding-tool factory only forwards it when present, so plugin-backed command-dispatch tools that use the gateway subagent or node runtime can become unavailable outside the Gateway.
  - confidence: 0.88
- **[P2] Pass the full group policy context:** `src/auto-reply/reply/get-reply-inline-actions.ts:208-213`
  - body: This call is now responsible for enforcing group tool policy, but it only passes `sessionEntry?.groupId` and `senderId`. Group policy resolution also uses ctx-derived group fallback plus sender name, username, and E.164 values, which normal tool inventory paths already pass; omitting them can miss a group-specific deny/allow rule.
  - confidence: 0.84
