## Summary

Keep this PR open. Current main still dispatches inline skill `command-dispatch: tool` through raw OpenClaw tools plus owner-only filtering, so the broader coding-tool policy fix proposed by this PR is not implemented on main.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase or adapt it on current main so inline skill `command-dispatch: tool` resolves tools through `createOpenClawCodingTools`, removes the standalone owner-only fallback, preserves the lazy runtime boundary, forwards the full route/session/group/sender/model/config context, and keeps focused regression tests for profile-disallowed tools plus policy-critical argument forwarding.

## Risks / Open Questions

- Current main still has a security-policy divergence: inline skill `command-dispatch: tool` can select raw OpenClaw/plugin tools after only owner-only filtering instead of the configured profile/provider/group/sandbox/subagent policy pipeline.
- The PR should be rebased or adapted to current main without regressing lazy runtime boundaries; a small `coding-tools.runtime` wrapper is likely cleaner than a direct `pi-tools.js` import in the inline-actions module.
- The rebase must preserve newer inline dispatch context such as `requesterAgentIdOverride`, `allowGatewaySubagentBinding`, route/thread fields, sender identity, owner status, model/provider, config, and group context.
