## Summary

Keep open. Current main still dispatches inline skill `command-dispatch: tool` through `createOpenClawTools` plus owner-only filtering, so the reported policy gap remains, but this PR is not ready to merge because the patch still does not mirror the normal command-session policy context and it regresses the lazy tool-loading boundary.

## What This Changes

The PR changes inline skill `command-dispatch: tool` handling to resolve tools through `createOpenClawCodingTools` and adds regression coverage for non-owner messaging-profile dispatch.

## Best Possible Solution

Inline skill tool dispatch should resolve callable tools through the same policy-resolving factory as normal command sessions, behind a lazy runtime wrapper, while forwarding target session, resolved policy session key, route, sender identity, group/spawned metadata, model/provider, and subagent-binding context with focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Forward full inline policy context:** `src/auto-reply/reply/get-reply-inline-actions.ts:198-224`
  - body: The new coding-tools call still does not mirror the normal command path: it passes the wrapper `sessionKey` directly and omits `allowGatewaySubagentBinding` plus sender name, username, and E164. Those fields feed group, plugin, and subagent policy, so inline skill tool dispatch can still resolve a different tool set than the surrounding command session.
  - confidence: 0.88

  - body: Importing `pi-tools` from inline action parsing pulls the full coding-tool stack into this hot path. Current main keeps tool loading behind a runtime import, so this should be rebased through a small runtime wrapper before merge.
  - confidence: 0.78
