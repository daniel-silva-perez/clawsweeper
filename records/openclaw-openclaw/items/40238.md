## Summary

Keep open. Current main still has the inline skill tool-dispatch policy gap, and this PR is useful source work, but the proposed patch is not merge-ready because its coding-tool factory call does not yet forward the full policy context used by normal command sessions.

## What This Changes

The PR changes inline skill `command-dispatch: tool` handling to resolve tools through `createOpenClawCodingTools` and adds regression coverage for non-owner messaging-profile dispatch.

## Best Possible Solution

Inline skill tool dispatch should resolve callable tools through the same policy-resolving factory as normal command sessions, behind a lazy runtime wrapper, while forwarding target session, route, sender identity, group/spawned metadata, model/provider, and subagent-binding context with focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Forward full inline policy context:** `src/auto-reply/reply/get-reply-inline-actions.ts:198-224`
  - body: The new coding-tools call still does not mirror the normal command path: it uses the wrapper `sessionEntry`, omits target-session lookup, and does not pass `allowGatewaySubagentBinding` or sender name/username/E164. Those fields feed group, plugin, and subagent policy, so inline skill tool dispatch can still resolve a different tool set than the surrounding command session.
  - confidence: 0.88
- **[P3] Preserve the lazy tool-loading boundary:** `src/auto-reply/reply/get-reply-inline-actions.ts:2`
  - body: Importing `pi-tools` at module load time pulls the full coding tool stack into inline-action parsing. Current main keeps tool loading behind a runtime import; rebase this through a small runtime wrapper so command parsing does not eagerly load the heavy tool runtime.
  - confidence: 0.78
