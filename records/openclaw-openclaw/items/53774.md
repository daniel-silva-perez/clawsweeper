## Summary

Keep this issue open. Current main still renders the TUI footer from a fixed in-code field list and exposes no footer template, footer field config, or plugin status-provider surface consumed by the TUI. Open related work such as #64086 and PR #72557 is narrower and still open; it would at most provide part of the plugin API prerequisite, not the requested cost/balance/rate-limit footer behavior.

## Best Possible Solution

Keep this issue open as the broad TUI footer customization tracker. The maintainer path is to add a real footer customization contract across `src/tui/tui.ts`, `src/tui/tui-types.ts`, CLI/config/schema/docs, and focused tests, while deciding whether provider-specific cost, balance, and cooldown data should come from existing usage/cost surfaces, a reviewed plugin status-provider API such as #64086/#72557, or both. PR #72557 can be reviewed as a possible prerequisite, but a follow-up still needs to wire the TUI render path and define template/order/truncation/security behavior.
