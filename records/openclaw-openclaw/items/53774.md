## Summary

Keep this issue open. Current main still renders the TUI footer from a hard-coded field list, exposes no footer template/config/custom field surface, and has no plugin status-provider seam consumed by the TUI. Existing `/status`, `/usage`, transcript `usage.cost`, and provider usage-window data are useful inputs, but they do not satisfy the requested persistent configurable footer for cost, balance, or rate-limit state.

## Best Possible Solution

Keep this issue open as the broader tracker. The maintainer path is to implement a real TUI footer customization surface in `src/tui/tui.ts`, `src/tui/tui-types.ts`, CLI/config/schema/docs, and focused tests, or split it into a canonical footer-template issue plus a reviewed plugin status-provider API that the TUI render path actually consumes. Use existing `/status`, `/usage`, `usage.cost`, provider usage-window, and burn0-style data sources as inputs rather than treating them as the implementation.

## Risks / Open Questions

- Open PR #72557 could become a useful prerequisite if merged, but it remains only part of the requested capability and does not implement footer rendering or cost/balance/cooldown fields.
- Balance and rate-limit display need product/security decisions around secret-backed provider calls, refresh cadence, truncation, and provider-specific quota semantics.
- A ClawHub/plugin implementation may be appropriate for provider-specific data sources, but current main still lacks the core TUI/API surface needed to consume those fields in the footer.
