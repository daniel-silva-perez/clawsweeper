## Summary

Keep open. Current main still rejects the reported Firecrawl self-hosted `baseUrl` paths before any network call, the tests and public docs still lock Firecrawl to HTTPS on `api.firecrawl.dev`, and the open implementation PRs for this behavior remain unmerged.

## What This Changes

Allow the Firecrawl plugin to use a local or self-hosted Firecrawl endpoint over HTTP/private host configuration instead of requiring `https://api.firecrawl.dev`.

## Best Possible Solution

Land one canonical Firecrawl-plugin implementation that explicitly supports trusted self-hosted endpoints, keeps public endpoints HTTPS-only, uses the existing private-network validation pattern where HTTP is allowed, and updates Firecrawl docs plus endpoint tests. After that lands, close or supersede the competing open PRs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `resolveEndpoint("http://host.openshell.internal:3002/v1", "/v2/search")` throws the HTTPS error, and `resolveEndpoint("https://host.openshell.internal:444/v1", "/v2/search")` throws the host allowlist error before any Firecrawl service is contacted.

## Solution Assessment

No, current main is not the best solution because it makes the documented `baseUrl` override unusable for self-hosted Firecrawl. The maintainable fix should stay inside the Firecrawl plugin, choose an explicit self-hosted/private-endpoint policy, and preserve the strict default for the official cloud API.
