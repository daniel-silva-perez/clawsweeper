## Summary

Keep this issue open. Current main still sends `sessions_send` traffic into the resolved existing target session, then runs reply-back and announce steps against the existing requester and target sessions. The available isolated path is `sessions_spawn`, which is child/subagent oriented and does not provide the requested peer-to-peer ephemeral `sessions_send` mode.

## Best Possible Solution

Keep the issue open for maintainer product/API design. A conservative fix would add an explicit `sessions_send` option or `session.agentToAgent` policy for isolated peer exchanges, create a temporary no-prior-context target session, return or wake the requester without writing follow-up turns into either main session, enforce the existing A2A allowlist/visibility checks, and document/test cleanup or retention behavior including Feishu/channel delivery.
