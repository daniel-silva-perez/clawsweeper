## Summary

Keep this issue open. Current main still models the reported phone-number and phone-identifier channel fields as plain strings or string/number arrays, the canonical SecretRef target surface does not include those paths, and an open PR now specifically closes this issue but has not merged.

## What This Changes

The issue requests SecretRef support for channel phone-number and phone-identifier fields so PII can be kept out of plaintext OpenClaw config.

## Best Possible Solution

Keep this issue open while maintainers review PR #73822. The desired end state is a backward-compatible, channel/plugin-owned implementation that accepts plain strings and SecretRef objects for the affected phone-number and sender-identity fields, declares those targets in the canonical SecretRef surface, resolves them before routing/allowlist checks, and updates generated metadata, docs, redaction/status output, and focused tests.
