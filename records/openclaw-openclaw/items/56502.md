## Summary

Current main still does not implement SecretRef support for phone-number or phone-identifier channel fields. WhatsApp, Signal, and voice-call still model these fields as plain strings/string arrays, and the SecretRef runtime/credential surface remains field-gated to declared credential paths such as voice-call Twilio authToken, not phone numbers. The issue also has recent maintainer validation that it remains valid plus active contributor follow-up, so it should stay open for implementation/review.

## Best Possible Solution

Keep this issue open. The best implementation is a backward-compatible channel/plugin-owned change: allow plain strings to keep working while adding SecretInput/SecretRef support for phone-number and PII peer fields such as WhatsApp allowFrom/defaultTo/groupAllowFrom, Signal account/defaultTo/allowFrom/groupAllowFrom, and voice-call fromNumber/toNumber/allowFrom. Update exported types, schemas, generated config metadata/help, the canonical SecretRef credential surface, secrets configure/audit/apply target discovery, redaction/status output, and runtime resolution before routing/outbound/auth checks consume these values. Any migration from plaintext should be an explicit doctor or secrets workflow, not an implicit startup rewrite.
