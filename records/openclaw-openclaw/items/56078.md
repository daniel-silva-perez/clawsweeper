## Summary

Keep this issue open. Current main still defaults new isolated agent-turn cron jobs from the Control UI and backend/CLI creation paths to announce delivery without best-effort, so a local/webchat-only setup with no configured delivery channel can still hit the reported "Channel is required (no configured channels detected)" path after the run. The related PRs mentioned in the thread are closed and unmerged, and the related best-effort discussion only helps callers that explicitly enable best-effort delivery.

## Best Possible Solution

Keep this issue open and fix the cron product path directly. A conservative fix would make Control UI-created local/webchat-only isolated cron jobs without a resolvable delivery channel default to delivery.mode="none", or downgrade unresolved implicit announce delivery non-fatally while preserving hard failures for explicit delivery targets. Add regression coverage for a Control UI-created isolated cron job in a no-channel config, backend-created isolated agentTurn defaults, and the documented workaround.

## Risks / Open Questions

- Changing the default from announce to none can affect users who rely on implicit chat delivery, so the fix should distinguish Control UI/local no-channel jobs from explicit announce targets and active-chat-created reminders.
- The broader first-class file-output routing request is separate from this regression; this issue should stay focused on the no-channel cron delivery failure path.
