## Summary

Keep open. Current main still renders the full Control UI credentials gate for every disconnected state, and the gateway host still clears `connected` immediately on reconnect start and active WebSocket close. PR #72522 is an open, unmerged closing PR for the same bug, so this issue should remain open until that PR is reviewed and merged, closed, or replaced.

## What This Changes

Prevent Control UI WebChat from flashing the full credentials/login gate during transient reconnects after a prior authenticated connection.

## Best Possible Solution

Review and land PR #72522 or an equivalent narrow Control UI fix that keeps the chat UI mounted after a prior successful gateway hello during transient transport reconnects, while restoring the credentials gate for explicit auth, identity, origin, and pairing failures with focused tests.

## Reproduction Assessment

Yes. The deterministic current-main path is enough: after a successful session, any active WebSocket close invokes the host `onClose`, clears `connected`, and `renderApp` immediately returns the full credentials gate for `connected === false`.

## Solution Assessment

Yes. A sticky prior-successful-session/reconnecting state is the narrow maintainable direction because it models the missing UI state directly; a timing-only debounce would reduce the flash but would not distinguish transient reconnects from real auth failures.
