## Summary

Keep open. Current main still has the reported two-state Control UI behavior, and the open closing PR has not merged; the issue should stay open until PR #72522 lands, is closed, or is replaced.

## What This Changes

The issue requests that Control UI WebChat preserve the authenticated chat UI across transient reconnects instead of flashing the full credentials gate whenever `connected` becomes false.

## Best Possible Solution

Review and land PR #72522, or an equivalent narrow Control UI fix, so the app distinguishes a prior successful authenticated session from a never-authenticated or explicitly rejected session and keeps chat mounted during transient transport reconnects.

## Reproduction Assessment

Yes. The deterministic code path is enough: after a successful session, a recoverable WebSocket close calls `onClose`, clears `connected`, and `renderApp` immediately returns the credentials gate before the reconnect completes.

## Solution Assessment

Yes. A prior-successful-session or reconnecting state is the narrow maintainable solution; a timing-only debounce could reduce the flash but would not model the auth/reconnect distinction as directly.
