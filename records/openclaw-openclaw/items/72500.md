## Summary

Keep open. Current main still has the reported Control UI reconnect behavior, and open PR #72522 is the active but unmerged implementation path with an unresolved auth-revocation concern called out in review.

## What This Changes

The issue requests that Control UI WebChat distinguish transient reconnects from first-connect or explicit auth failures so the credentials gate does not flash on tab refocus.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that keeps chat mounted after prior successful sessions during recoverable transport closes, restores the credentials gate for first connect and explicit auth, pairing, origin, or device-identity failures, and includes focused gateway/render tests for both paths.

## Reproduction Assessment

Yes. The deterministic code path is enough: a recoverable WebSocket close invokes `onClose`, `app-gateway` clears `connected`, and `renderApp` immediately selects the credentials-gate branch before reconnect completes.

## Solution Assessment

Yes, the sticky prior-successful-session/reconnecting model is the narrow maintainable direction, provided it also clears on explicit auth or device revocation. A timing-only debounce would hide the flash without modeling the auth-versus-transient-disconnect distinction.
