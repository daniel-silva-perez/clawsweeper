## Summary

Keep open. Current main and the latest shipped release still render the Control UI credentials gate for every disconnected state, while the linked PR #72522 remains open as the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests that Control UI WebChat keep the chat UI mounted or show a reconnecting state during transient disconnects after a prior authenticated session, instead of flashing the credentials gate.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that tracks a prior successful session, keeps chat mounted during recoverable transport reconnects, restores the credentials gate only for first-connect or explicit auth failure states, and covers those transitions with focused tests.

## Reproduction Assessment

Yes. The deterministic reproduction path is visible in current main: after a successful connection, a recoverable WebSocket close sets connected=false and renderApp immediately selects the credential-gate branch before reconnect completes.

## Solution Assessment

Yes. A sticky prior-successful-session or reconnecting state is the narrow maintainable solution; a timing-only debounce would hide the flash but still conflate transient transport disconnects with first-connect and auth-failure states.
