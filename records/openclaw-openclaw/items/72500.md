## Summary

Keep this open. Current main still switches the whole Control UI back to the credentials gate whenever `connected` becomes false, and the gateway still sets that state immediately on reconnect start and WebSocket close. PR #72522 is an open, unmerged closing PR for the same bug, so this issue should stay open until that PR is reviewed and either merged or closed.

## What This Changes

Prevent Control UI/WebChat from flashing the full credentials/login gate during transient reconnects after a previously successful authenticated session.

## Best Possible Solution

Review and land PR #72522 or an equivalent narrow fix that adds a prior-successful-session/reconnecting state in the Control UI, keeps chat mounted across transient closes, clears back to the credentials gate on explicit auth or pairing failures, and adds tests for both paths.

## Reproduction Assessment

Yes. The deterministic code path is enough to reproduce: after any successful session, an active WebSocket `onClose` sets `connected = false`, and `renderApp` immediately renders the credential gate for that state.

## Solution Assessment

Yes, with review. A sticky prior-successful-connection state is a better fit than a timing-only debounce because it models the missing state directly, as long as auth and pairing close details reset it.
