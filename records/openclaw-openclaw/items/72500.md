## Summary

Keep open. Current main still has the reported Control UI two-state reconnect behavior, and the open unmerged PR #72522 is the active closing implementation path, so this issue should remain open until that PR is reviewed, landed, closed, or replaced.

## What This Changes

Report requests that Control UI WebChat avoid flashing the credentials gate during transient WebSocket reconnects after a successful authenticated session.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that preserves a prior-successful-session/reconnecting state, keeps chat mounted across recoverable transport closes, and restores the credentials gate only for first connect or explicit auth, pairing, origin, or device-identity failures with focused tests.

## Reproduction Assessment

Yes. The deterministic code path is enough: after a successful session, a recoverable WebSocket close invokes the active `onClose`, clears `connected`, and `renderApp` immediately selects the credential-gate branch.

## Solution Assessment

Yes. A sticky prior-successful-session/reconnecting model is the narrow maintainable fix; a timing-only debounce could hide the flash but would not model the auth-versus-transient-disconnect distinction cleanly.
