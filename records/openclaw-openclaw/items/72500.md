## Summary

Keep open. Current main still has the reported Control UI two-state behavior: transient disconnects clear `connected`, and any disconnected render returns the full credentials gate. PR #72522 is an open, unmerged closing PR for the same bug, so this issue should remain open until that PR is reviewed, landed, closed, or replaced.

## What This Changes

The issue requests that Control UI WebChat preserve the authenticated chat UI, or show a reconnecting state, during transient WebSocket reconnects instead of flashing the credentials gate after a successful session.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that adds a prior-successful-session/reconnecting state, keeps chat mounted across recoverable transport closes, and restores the credentials gate for real auth, device identity, origin, or pairing failures with focused tests.

## Reproduction Assessment

Yes. The deterministic code path is enough: after a successful session, a recoverable WebSocket close invokes the active `onClose`, clears `connected`, and `renderApp` immediately renders the credential gate for that disconnected state.

## Solution Assessment

Yes. A prior-successful-session/reconnecting model is the narrow maintainable fix; a timing-only debounce may reduce the flash but would not model the auth-vs-transient-disconnect distinction as clearly.
