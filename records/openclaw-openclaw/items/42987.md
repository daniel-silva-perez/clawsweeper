## Summary

Current main still does not implement cron procedural playbook memory, so this PR is not obsolete as implemented-on-main. It remains a plausible contribution, but it needs maintainer product direction and rework before merge because the proposed wiring records only the legacy `executeJob` path, misses scheduled/startup/manual execution paths, loses classified recovery data, and introduces durable cron error memory that needs privacy/retention handling.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as-is. Best path: rebase onto current main, split the autonomy roadmap docs from the runtime change, decide whether procedural cron memory belongs in core or a ClawHub/plugin implementation, and then either add the generic cron hook fields needed by plugins or implement core recording in the shared outcome path with `lastErrorKind`, serialized writes, privacy/retention docs, and tests covering scheduled, startup catch-up, and manual runs.

## Risks / Open Questions

- Functional readiness: the proposed `executeJob`-only wiring would not record the real scheduled, startup catch-up, or manual cron execution paths on current main.
- Functional correctness: recovery counts can be silently dropped unless classified error kind is persisted and propagated with the prior error.
- Product boundary: this may belong as a plugin/ClawHub implementation, but current public cron hook data likely needs an additive `errorKind` or equivalent generic classified-error field first.
- Privacy/data handling: durable memory of job ids, job names, and raw error text needs explicit retention, redaction, and user-facing docs before core adoption.
- Persistence concurrency: current cron supports `maxConcurrentRuns`, so per-signal load/save memory writes need a process-local lock or shared-state strategy.
