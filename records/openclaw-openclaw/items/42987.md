## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, and the PR remains a plausible contribution, but the proposed implementation is not ready to merge as-is: it records only the legacy `executeJob` path, misses scheduled/startup/manual cron runs, drops classified recovery data, and adds durable cron error memory that needs explicit privacy, retention, and concurrency handling. Security review did not show supply-chain or CI-sensitive changes in the provided PR file list, but the durable persistence of job names/ids/raw errors is a product/security concern that needs maintainer design review.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as-is. Best path: rebase onto current main, split the autonomy roadmap docs from the runtime change, decide whether procedural cron memory belongs in core or a ClawHub/plugin implementation, and then either add the generic cron hook fields needed by plugins or implement core recording in the shared outcome path with `lastErrorKind`, serialized writes, privacy/retention docs, and tests for scheduled, startup catch-up, and manual runs.

## Risks / Open Questions

- Functional readiness: the proposed hook point would miss scheduled, startup catch-up, and manual cron execution paths on current main.
- Correctness: recovery counting can be silently wrong unless classified error kind is persisted and propagated with the previous error.
- Privacy/data handling: durable memory of job ids, job names, and raw error text needs redaction, retention, and user-facing documentation before core adoption.
- Concurrency: cron supports `maxConcurrentRuns`, so per-signal load/save memory writes need serialization or a shared-state strategy.
- Product boundary: this may belong as a ClawHub/plugin implementation, but the public cron hook currently lacks `errorKind` or an equivalent classified-error seam.
