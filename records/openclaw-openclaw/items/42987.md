## Summary

Current main still does not implement cron procedural playbook memory. The PR remains a plausible implementation candidate, but it needs revision for current cron execution paths, recovery classification state, persistence/privacy semantics, and the newer cron plugin-hook surface before maintainers should merge or close it.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to rebase it onto current main, decide explicitly whether the feature should be core or plugin-backed via cron_changed and prompt hooks, and if it stays core, record signals from a shared cron outcome path covering scheduled, startup catch-up, and manual runs; persist and propagate lastErrorKind; serialize memory writes; document privacy/retention behavior; and keep the autonomy roadmap docs separate from the runtime change if that helps review.

## Risks / Open Questions

- Functional readiness: wiring only executeJob would miss scheduled ticks, startup catch-up, and manual cron runs on current main.
- Functional correctness: recovery counts can be dropped unless lastErrorKind is persisted and propagated with lastError.
- Concurrency: current cron supports maxConcurrentRuns, so per-signal load/save needs serialization or a shared state strategy.
- Privacy/data handling: durable memory of job ids, job names, and raw error text needs clear local retention and redaction semantics.
- Product boundary: current main now has cron_changed plus prompt hooks, so maintainers should decide whether this belongs in core or as a plugin built on the newer hook surface.
