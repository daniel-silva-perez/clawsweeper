## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, and the PR is not obsolete as implemented-on-main. It remains a plausible contribution, but it needs maintainer product direction and rework because current main has plugin hooks that may support this outside core, while the PR's proposed core wiring would miss important cron execution paths and has an `errorKind` recovery gap already identified in review.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as-is. Best path: rebase onto current main, split the autonomy roadmap docs from the runtime change, decide whether procedural cron memory belongs in core or a ClawHub/plugin implementation, and then either add the small generic hook fields needed by plugins or implement core recording in the shared outcome path with `lastErrorKind`, serialized writes, privacy/retention docs, and tests covering scheduled, startup catch-up, and manual runs.

## Risks / Open Questions

- Functional readiness: the proposed `executeJob`-only wiring would not record the real scheduled, startup catch-up, or manual cron paths on current main.
- Functional correctness: recovery counts can be silently dropped unless classified error kind is persisted and propagated with `lastError`.
- Product boundary: current hooks and plugin storage make a plugin/ClawHub implementation plausible, but the public cron hook may need a small generic `errorKind` field first.
- Privacy/data handling: durable memory of job ids, job names, and raw error text needs explicit retention, redaction, and docs before core adoption.
- Persistence concurrency: current cron supports `maxConcurrentRuns`, so per-signal load/save memory writes need serialization or another shared-state strategy.
