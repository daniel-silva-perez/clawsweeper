## Summary

Keep open. Current main has improved cron delivery routing through explicit `delivery.accountId` and agent/channel bindings, but it still does not implement the reported `agentId` to `accountId` fallback. The current resolver still lets a session-derived `lastAccountId` win before bindings, and tests explicitly cover that precedence, so the Feishu/Telegram multi-account wrong-sender class is still plausible when `--account` is omitted.

## Best Possible Solution

Keep this issue open and implement a focused cron delivery identity fix in `src/cron/isolated-agent/delivery-target.ts`: preserve explicit `delivery.accountId` as highest priority, then resolve the job agent's channel-bound account for cron-owned delivery before trusting stale session account state, with a carefully scoped exception for chat-created/current-session routes that intentionally preserve the live account. Add regression tests for Feishu/Telegram-style multi-account cron jobs, explicit `--account`, binding-derived accounts, and stale `lastAccountId` cases; update the cron CLI/docs if the intended operator-facing contract is bindings rather than `agentId === accountId`.

## Risks / Open Questions

- A naive `accountId = agentId` fallback may be wrong for installations where agent ids and channel account ids differ; the safer fix likely needs to use account-scoped bindings or a documented explicit mapping before any name-equality fallback.
- Changing precedence away from session-derived `lastAccountId` could break reminders intentionally created from an active chat, so the fix needs targeted cron tests for explicit delivery, `channel: "last"`, session-target delivery, and `--account` precedence.
- Until fixed, multi-account cron jobs without explicit `accountId` can still send via a stale or default account, which matches the security concern raised in the Telegram reproduction comment.
