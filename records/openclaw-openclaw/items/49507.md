## Summary

Keep open. Current main still does not implement the reported missing cron `agentId` to `accountId` inference, and the risk identified in the Feishu and Telegram reports remains plausible: cron delivery can prefer explicit `delivery.accountId`, then session-derived `lastAccountId`, then agent/channel bindings, but there is no fallback that treats the job agent identity as the outbound account when `accountId` is omitted. The related PRs that attempted the direct fallback were closed unmerged, and the related Slack identity issue fixed agent presentation, not this multi-account routing class.

## Best Possible Solution

Keep this issue open. The best fix is a focused cron delivery identity change in `src/cron/isolated-agent/delivery-target.ts`: keep explicit `delivery.accountId` highest priority, resolve the job agent's channel-bound outbound account before trusting stale session account state for cron-owned explicit delivery, and only consider `agentId === accountId` as a documented compatibility fallback if maintainers want that contract. Add regression tests for Feishu/Telegram-style multi-account cron jobs, explicit `--account`, binding-derived accounts, stale `lastAccountId`, and chat/session routes that intentionally preserve live account context.

## Risks / Open Questions

- Cron jobs without explicit `accountId` can still send through a stale session account or default account in multi-account Feishu/Telegram setups, matching the identity/confidentiality concern in the issue discussion.
- A naive `accountId = agentId` fallback may be wrong where agent ids and channel account ids do not match; the safer fix likely needs binding-based resolution first plus a documented fallback contract.
- Changing precedence away from `lastAccountId` could break cron jobs intentionally created from an active chat/session, so tests need to distinguish explicit channel/account delivery from `channel: "last"` or chat-created delivery flows.
