## Summary

Current main still does not implement the reported cron `agentId` to `accountId` inference. The cron delivery resolver uses explicit `delivery.accountId`, then session-derived `lastAccountId`, then agent/channel bindings; tests still assert session account precedence and `undefined` when no session or binding exists. The two direct fix PRs are closed unmerged, and the related Slack identity issue fixed agent presentation rather than this outbound account-routing gap.

## Best Possible Solution

Keep this issue open. The best fix is a focused cron delivery identity change in `src/cron/isolated-agent/delivery-target.ts`: keep explicit `delivery.accountId` highest priority, resolve the job agent's bound outbound account before trusting stale session account state for cron-owned explicit delivery where appropriate, and decide deliberately whether `agentId === accountId` should be a documented compatibility fallback. Add regression tests for Feishu/Telegram-style multi-account cron jobs, explicit `--account`, binding-derived accounts, stale `lastAccountId`, and session-bound routes that intentionally preserve live account context.

## Risks / Open Questions

- Cron jobs without explicit `accountId` can still send through stale session account state or a channel default in multi-account Feishu/Telegram setups, which matches the reported identity and confidentiality concern.
- A naive `accountId = agentId` fallback may be wrong for deployments where agent ids and channel account ids intentionally differ; the safer behavior should prefer explicit account, then a correct agent/channel binding, and only use an agent-id compatibility fallback if maintainers document that contract.
- Changing precedence without tests could regress active-chat reminders or session-pinned cron jobs that intentionally preserve live session account context.
