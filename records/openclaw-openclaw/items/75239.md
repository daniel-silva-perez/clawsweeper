## Summary

Keep open. Current main contains the earlier WebChat duplicate-message mitigations, but the report targets a release that already had those fixes and I did not find a newer source, test, or release-note change proving the reported post-final race is fixed. The Control UI still has an immediate same-session `session.message` history reload path once `chatRunId` is cleared, while the regression coverage does not cover the reported final-then-`session.message`/`sessions.changed` ordering.

## What This Changes

The issue asks OpenClaw to fix a Control UI WebChat regression where assistant replies still render as duplicate bubbles in 2026.4.27 after the earlier duplicate-message fixes.

## Best Possible Solution

Make Control UI WebChat reconciliation idempotent across `chat.final`, `session.message`, and `sessions.changed` ordering so a normal assistant final can never produce a second visible bubble, while preserving required history refreshes for silent finals, tool output, errors, aborts, and external transcript updates.

## Reproduction Assessment

Unclear. The issue gives a high-signal manual reproduction path on 2026.4.27, but I did not live-run the browser flow and the current tests only cover `session.message` before a renderable final, not the reported post-final race.

## Solution Assessment

Unclear. The existing #66875 direction is still the right surface, but the safest next step is a focused event-ordering regression test before choosing between a run-scoped post-final suppress, cooldown, or tighter history-merge guard.
