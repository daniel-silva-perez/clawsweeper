## Summary

Keep open. Current main and v2026.4.27 contain the startup mitigation for the pricing-refresh hang, but the shutdown side of the report is still not fully handled: the model-pricing stop callback only marks the starter stopped and clears an existing timer, while an already-running refresh can still continue, rely only on its own timeout signal, and schedule the next ref'ed refresh timer after shutdown has begun.

## What This Changes

The issue reports that npm-global OpenClaw 2026.4.24 on Windows 11 hangs during `openclaw gateway` startup when OpenRouter and LiteLLM pricing fetches time out, leaving the dashboard unreachable and Ctrl+C ineffective.

## Best Possible Solution

Keep the shipped startup fix, then make the model-pricing refresh lifecycle stop-aware: abort in-flight pricing fetches on gateway shutdown, prevent stopped refreshes from scheduling another timer, and unref any background pricing refresh timer so the process can exit cleanly. After that, verify a Windows npm-global gateway start and Ctrl+C under forced OpenRouter/LiteLLM timeouts, then close this issue as implemented.

## Reproduction Assessment

Yes. The original report gives a concrete released-package repro path, and current source supports a focused unit reproduction: start the pricing refresh with a hanging or timed-out fetch, call the returned stop callback before it settles, and assert no in-flight fetch or ref'ed follow-up timer survives shutdown.

## Solution Assessment

No for the current state. The v2026.4.27 startup mitigation is the right narrow fix for dashboard reachability, but the best complete solution also needs shutdown-owned cancellation and timer cleanup for the Ctrl+C behavior described in the report.
