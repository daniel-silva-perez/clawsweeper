## Summary

Keep open. Current main has useful mitigations for the startup side of the pricing-refresh regression, including deferred startup work and a config opt-out, but the shutdown/hung-process part remains source-backed: in-flight OpenRouter and LiteLLM fetches are not lifecycle-abortable, and refresh completion can still schedule a referenced 24-hour timer after stop.

## What This Changes

Fix a 2026.4.24 Windows/npm global regression where Gateway startup hangs when OpenRouter and LiteLLM pricing fetches time out, leaving the dashboard unreachable and Ctrl+C ineffective.

## Best Possible Solution

Keep the existing startup mitigations, then make model-pricing refresh shutdown-aware: pass a lifecycle abort signal into both pricing catalog fetches, abort in-flight refresh work on Gateway stop, prevent cache mutation or refresh scheduling after stop, unref long background timers, and cover the behavior with focused pricing/runtime shutdown tests.

## Reproduction Assessment

Yes. The issue provides concrete released-package steps, and current main supports a focused source-level reproduction by starting startGatewayModelPricingRefresh with hanging fetches, calling the returned stop callback during the in-flight refresh, and asserting the fetches abort with no post-stop timer scheduled.

## Solution Assessment

No. Current main is a partial mitigation, and PR #73214’s timeout/opt-out direction is useful for restricted networks but does not address the narrower shutdown lifecycle bug that can leave in-flight fetch work or timers alive after stop.
