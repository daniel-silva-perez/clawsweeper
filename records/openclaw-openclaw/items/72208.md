## Summary

Keep open. Current main and the latest release contain useful startup-side mitigations for model-pricing refresh work, but the remaining Ctrl+C/hung-process part is still source-backed: the pricing refresh stop hook cannot abort in-flight OpenRouter/LiteLLM fetches, and successful or retained-cache paths can still schedule the next refresh without an explicit stopped-state guard.

## What This Changes

The issue reports that OpenClaw 2026.4.24 can hang during `openclaw gateway` startup on Windows when OpenRouter and LiteLLM pricing fetches time out, leaving the dashboard unreachable and Ctrl+C unresponsive.

## Best Possible Solution

Keep the shipped startup optimization and config opt-out, then make the model-pricing refresh lifecycle shutdown-aware: pass a lifecycle abort signal into both pricing fetches, abort in-flight refresh work on gateway stop, prevent all post-stop scheduling or cache mutation, unref background timers, and lock the behavior with focused gateway pricing/runtime shutdown tests.

## Reproduction Assessment

Yes. The issue has concrete released-package steps, and current main supports a focused source-level reproduction by starting `startGatewayModelPricingRefresh` with hanging fetches, calling its stop callback, and asserting the fetches abort and no refresh timer remains scheduled.

## Solution Assessment

No. Current main is a partial mitigation: deferring startup work and offering `models.pricing.enabled: false` help affected users, but the narrow maintainable fix for the reported unresponsive shutdown belongs in the model-pricing refresh lifecycle rather than only shortening timeouts or relying on manual opt-out.

## Repair Work Prompt

Fix the remaining model-pricing shutdown lifecycle bug tracked by https://github.com/openclaw/openclaw/issues/72208. Current main already defers the initial refresh and has `models.pricing.enabled: false`; do not replace that with a broad Gateway rewrite.
