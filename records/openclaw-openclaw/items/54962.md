## Summary

Keep open: the linked Telegram callback bug is valid and current main still sends generic callback payloads raw, but this PR’s stale one-line fallback would also prefix callback payloads that current main must preserve as command/native callback input.

## What This Changes

The PR changes Telegram’s callback fallback to forward inline button payloads to the agent as `callback_data: ${data}` instead of the raw callback string.

## Best Possible Solution

Update the Telegram plugin so decoded native callbacks and command-like slash callbacks keep their command semantics, while only raw generic callback payloads are sent to the agent as `callback_data: <value>` with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. A focused current-main check is the callback_query path with `data: "cmd:option_a"`: the docs expect `callback_data: cmd:option_a`, while the source and existing test still route/assert only the raw payload.

## Solution Assessment

No. Prefixing non-command callback payloads is the right direction, but the submitted one-line fallback is not the best current-main implementation because it would erase native/command callback routing; the safer fix branches before prefixing.
