## Summary

Keep open. Current main has adjacent pieces: env-based provider API-key rotation, rate-limit classification, profile cooldown/backoff, and per-profile/model cooldown ordering. It still does not implement the central request: explicit Gemini project/quota-bucket metadata, bucket-level cooldown state, or failure fanout that cools every key/profile in the same project bucket before selecting a different bucket. The linked Keymanager sidecar and aashari fork comment are useful prior art; the aashari comment also notes bucket/project grouping is not native there.

## Best Possible Solution

Keep this open for maintainer design. The likely product path is to add explicit quotaBucket/quotaGroup metadata for provider auth profiles or provider-owned profile metadata, persist bucket-level health alongside profile health, update failure handling and auth-profile selection so Gemini 429/RESOURCE_EXHAUSTED cools every profile in the same project bucket, and add Gemini multi-project tests plus docs. If maintainers prefer a plugin route, first expose a supported provider hook/API that gives plugins the failing auth profile identity and lets them update shared bucket health without log tailing or rewriting config files.
