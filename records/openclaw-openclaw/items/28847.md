## Summary

Keep open. Current main has adjacent support for Gemini/API-key lists, rate-limit classification, profile cooldowns, model-scoped cooldowns, and rotation caps, but it still does not implement the central requested quota-bucket/project grouping for Gemini: there is no explicit quotaBucket/quotaGroup metadata, no bucket-level health state, and no fanout that cools all keys/profiles in the same project after one 429/RESOURCE_EXHAUSTED. The linked Keymanager sidecar and aashari fork comment are useful prior art but do not make this implemented in OpenClaw core.

## Best Possible Solution

Keep this open for maintainer design. The best path is to add an explicit provider-owned quotaBucket/quotaGroup field for auth profiles or stored credential metadata, persist bucket-level health alongside existing profile usage state, and update Gemini rate-limit handling so a 429/RESOURCE_EXHAUSTED cools every key/profile in the same project bucket before selecting a different bucket. The sidecar and fork are useful prior art, but core should not depend on log tailing or auth-file rewrites for this behavior.
