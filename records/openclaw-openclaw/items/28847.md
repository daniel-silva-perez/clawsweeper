## Summary

Keep open. Current main has adjacent provider key-list rotation, rate-limit classification, per-profile/model cooldowns, and rotation caps, but it still lacks the central requested Gemini quota-bucket model: explicit project/bucket metadata, bucket-level cooldown state, and fanout from one failed key/profile to all keys/profiles in the same quota bucket. The aashari fork comment and linked Keymanager sidecar are useful prior art, but they do not make this implemented in OpenClaw core.

## Best Possible Solution

Keep this open for maintainer design. The best path is to add an explicit provider-owned quotaBucket/quotaGroup field for auth profiles or stored credential metadata, persist bucket-level health alongside profile usage state, and update Gemini rate-limit handling so a 429/RESOURCE_EXHAUSTED cools every key/profile in the same project bucket before selecting another bucket. The linked Keymanager sidecar and aashari fork are useful prior art, but OpenClaw should not rely on log tailing or auth-file rewrites for this behavior.
