## Summary

Keep open. The PR addresses a real memory/wiki fallback problem and several earlier review comments appear to be handled, but the current head is not merge-ready because its plugin tool registry fallback can temporarily drop all plugin tools during an equivalent in-flight load, and the branch is currently diverged and unmergeable against main.

## What This Changes

The PR adds direct memory-wiki fallback paths for memory-core tools, changes plugin tool resolution to use fresh non-activating snapshot loads, exports a memory-wiki API subpath, preserves more memory plugin state during snapshot loads, and adds targeted memory/plugin regression coverage plus a local pre-PR script.

## Best Possible Solution

Keep the PR open, rebase it onto current main, preserve current memory tool behavior, and change the in-flight plugin tool path so compatible active registries remain available while a fresh snapshot load is already running.

## Reproduction Assessment

Yes. The PR's own mocked in-flight test exercises the relevant branch and expects an empty tool list, while current main shows the safer active-registry compatibility check happens before the in-flight short-circuit.

## Solution Assessment

No. Fresh non-activating snapshot loads are a reasonable direction for stale tool closures, but dropping all plugin tools during concurrent equivalent loads is not the narrowest maintainable behavior; use a compatible active registry as the fallback when one exists.
