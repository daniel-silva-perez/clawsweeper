## Summary

Keep open. The PR addresses a real gap that current main still has: memory-core still returns empty/null when wiki corpus supplements are missing, but the proposed plugin tool registry change should not merge as-is because it drops all plugin tools for requests racing an equivalent in-flight plugin load even when a compatible active registry is available.

## What This Changes

The PR adds a direct memory-wiki fallback for memory-core search/get, changes plugin tool resolution to use fresh non-activating plugin loads, exports a memory-wiki API subpath, adds memory-wiki bridge/public-artifact fallback work, and expands targeted regression coverage plus a local memory pre-PR script.

## Best Possible Solution

Keep the memory-wiki fallback direction, but rework the tool registry path so fresh non-activating snapshot loads do not make concurrent requests lose tools. The narrow fix is to reuse a compatible active or gateway-bindable registry when an equivalent load is in flight, then rebase and validate the memory/plugin-loader surfaces together.

## Reproduction Assessment

Yes. The PR's own tools.optional.test.ts in-flight case reproduces the behavior by mocking isPluginRegistryLoadInFlight=true and expecting no plugin tools; adding a compatible active registry to that scenario demonstrates the regression directly.

## Solution Assessment

No. The fallback work is plausible, but the proposed registry solution is not the best fix because it trades stale-tool avoidance for nondeterministic tool disappearance under concurrent gateway/tool-catalog traffic; the safer implementation should preserve a compatible active-registry fallback before degrading to empty.

## Review Findings

- **[P1] Reuse the active registry before dropping plugin tools:** `src/plugins/tools.ts:66-67`

Keep the in-flight guard, but fall back to the compatible active registry first.
