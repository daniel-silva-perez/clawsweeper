## Summary

Keep open. This PR has the protected `maintainer` label and the latest head still has a blocking channel startup regression: no-repair mode is applied at the parent `channels` policy, so operational channel commands can fail before their handlers run when bundled runtime deps are missing.

## What This Changes

The PR changes CLI/channel/plugin inspection and Gateway startup loading so selected paths avoid implicit bundled runtime-dependency repair, adds explicit `openclaw plugins deps --repair` and `plugins inspect --runtime`, memoizes prepared bundled runtime roots, and updates docs, tests, and changelog.

## Best Possible Solution

Keep the PR open and narrow no-repair mode to exact static/read-only channel surfaces, or leave runtime dependency repair enabled for operational channel commands that load channel runtimes before doing useful work. The Gateway pre-stage and explicit repair direction can remain, but the CLI policy needs a tighter boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a packaged install with a configured bundled channel plugin whose staged runtime deps are missing, then running `openclaw channels login --channel <id>` or `openclaw channels logout --channel <id>`; preaction inherits the parent `channels` no-repair policy and runtime loading fails before the handler can repair or proceed.

## Solution Assessment

No, not yet. The explicit repair and Gateway pre-stage direction is maintainable, but applying `installBundledRuntimeDeps: false` at the parent `channels` policy is broader than the stated read-only boundary; exact per-command policy is the safer fix.
