## Summary

Keep open: the latest PR head still leaves the sync bundled-runtime staging repair path able to no-op after writing the generated manifest, so the branch does not fully fix interrupted runtime-deps recovery. The PR is also protected by the maintainer label, but the blocker is narrow and matches the `clawsweeper:autofix` lane.

## What This Changes

This PR refactors bundled plugin runtime-dependency planning and repair across plugin loading, Gateway startup/hot reload, CLI channel/plugin commands, docs, changelog, scripts, and regression tests to use package-level plans, materialization checks, and package-manager repair.

## Best Possible Solution

Keep the package-level plan/materialization direction, but make every repair path that runs after a generated manifest is written force package-manager convergence consistently, including the sync staging helper used by lazy plugin loads. Add a focused regression test for an interrupted tree with package sentinels, then rerun targeted runtime-deps/Gateway/CLI tests and the normal changed gate before maintainer landing.

## Reproduction Assessment

Yes. A deterministic static reproduction exists: create an incomplete install root with `node_modules` package sentinels but no generated install manifest, then load a bundled plugin through `prepareBundledRuntimeLoadRootForPlugin`; the helper writes the manifest and calls the sync installer without `force`, allowing the second materialization check to return before npm/pnpm runs.

## Solution Assessment

No, not as-is. The package-level async repair path is the right maintainable direction, but the sync loader fallback must pass `force: true` or delegate through the same forced repair path before this PR fully solves interrupted bundled runtime-deps recovery.
