## Summary

Keep open. The protected `maintainer` label requires explicit maintainer handling, and the latest diff still has a blocking runtime-deps repair gap: the new sync staging helper bypasses the PR's own `force: true` repair path, so CLI/lazy plugin loads can still no-op on an incomplete generated install tree. The earlier review concern about disabling repair for all `channels/*` paths appears addressed by the latest command policy split.

## What This Changes

This PR refactors bundled plugin runtime-dependency planning and repair across plugin loading, Gateway startup/hot reload, CLI plugin/channel/deps commands, install helpers, docs, changelog, and regression tests.

## Best Possible Solution

Keep the package-level plan/materialization direction, but make every repair path that runs after a generated manifest is written force package-manager convergence consistently, including the new sync staging helper used by `loadOpenClawPlugins`. Then refresh the branch and run targeted runtime-deps/Gateway/CLI tests plus the normal changed/build gate before maintainer landing.

## Reproduction Assessment

Yes. A deterministic static reproduction exists: create an incomplete install root with `node_modules` and a generated package manifest, then load a bundled plugin through the new sync `prepareBundledRuntimeLoadRootForPlugin` path; the helper writes/uses the generated manifest and calls `installBundledRuntimeDeps` without `force`, so the new materialization guard can still no-op.

## Solution Assessment

No, not as-is. The package-level async repair path is the right maintainable direction, but the sync loader fallback must pass `force: true` or delegate through the same forced repair plan before the PR fully solves interrupted/incomplete runtime-deps recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Force the sync runtime-deps installer during fallback repair:** `src/plugins/bundled-runtime-staging.ts:52-62`
  - body: The new helper passes a custom `installDeps` callback, so `ensureBundledPluginRuntimeDeps` no longer reaches its default `installBundledRuntimeDeps({ force: true })` path. When a generated manifest has already been written for an incomplete tree, `installBundledRuntimeDeps` rechecks materialization and can return without running npm/pnpm, leaving CLI or lazy plugin loads stuck behind the same package sentinel this PR is meant to recover from. Pass `force: true` here or avoid the second materialization short-circuit.
