## Summary

Keep open. The protected `maintainer` label rules out cleanup closure, and the latest PR head still has a concrete P1 regression in deferred startup plugin loading: it refreshes HTTP plugin routes after the post-bind load but leaves the channel registry pinned to the pre-runtime registry.

## What This Changes

This PR refactors bundled plugin runtime dependency resolution into an explicit planning/materialization flow across Gateway startup, CLI plugin-dependency commands, doctor, config hot reload, channel/plugin inspection policy, docs, and regression tests.

## Best Possible Solution

Keep the PR open and update the deferred runtime replacement so the loaded startup plugin registry becomes the source for both HTTP route lookup and channel lookup, with a focused regression test proving a post-bind startup channel is visible before sidecars start.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: create Gateway runtime state before deferred startup plugin load, then load a startup channel plugin post-bind and resolve a channel through `getActivePluginChannelRegistry()`; the PR leaves that lookup on the pre-runtime registry.

## Solution Assessment

No. The PR direction is close, but it is not mergeable as-is because the registry replacement only preserves the HTTP route surface; the narrower maintainable fix is to repin the channel registry at the same replacement point or defer initial channel pinning until startup runtime load completes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Repin the channel registry after deferred startup load:** `src/gateway/server.impl.ts:1002`
  - body: The branch creates Gateway runtime state before startup plugins load, so the channel registry is pinned to the pre-runtime registry. When the post-bind load finishes, `replaceAttachedPluginRuntime` repins HTTP routes only, while channel/outbound loaders read `getActivePluginChannelRegistry()`. A startup channel plugin loaded after bind can therefore remain invisible for outbound/channel lookup. Repin the channel registry with the loaded registry or defer the initial channel pin until the loaded startup registry is available.
  - confidence: 0.9
