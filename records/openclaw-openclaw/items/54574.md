## Summary

Keep this PR open. Current main still unconditionally blocks Docker sandbox `network: "host"`, lacks the proposed `dangerouslyAllowHostNetwork` config/type/schema/help surface, and existing tests/docs still describe host networking as blocked. The PR is coherent, but it deliberately changes a Docker sandbox isolation boundary and needs maintainer/secops review rather than cleanup closure.

## What This Changes

The PR adds a Docker sandbox `dangerouslyAllowHostNetwork` boolean across network validation, config schema/help/types, Docker create-args tests, and audit remediation so `network: "host"` can pass when explicitly enabled.

## Best Possible Solution

Keep this PR open for maintainer/secops review of a rebased version or equivalent follow-up. A merge-ready implementation should keep the escape hatch limited to Docker sandbox config, thread the flag through runtime validation, config resolution, Zod/public/generated schema, help/labels/docs, Docker create args, dangerous-flag auditing, and targeted sandbox/config/security tests.
