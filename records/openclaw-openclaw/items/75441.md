## Summary

Keep open. This PR addresses real current-main config recovery observability gaps, but it has the protected `maintainer` label, overlaps the open #70515 backup-restore audit fix, and still leaves failed backup restores audit-valid as written.

## What This Changes

The PR adds warning logs for config health-state write failures and failed backup restore copies, adds async/sync regression tests in `src/config/io.observe-recovery.test.ts`, and updates `CHANGELOG.md`.

## Best Possible Solution

Land one canonical config recovery fix that keeps this PR's health-state write-failure logging, preserves focused async/sync regression coverage, and uses #70515-style failed-restore audit semantics so failed backup copies are not recorded as valid recoveries.

## Reproduction Assessment

Yes. Current main can be reproduced by injecting failing health-state writes or failing `copyFile`/`copyFileSync` during suspicious-read recovery; the PR's new tests encode those failure paths, though I did not run them in this read-only review.

## Solution Assessment

No, not as submitted. The health-state logging is narrow and useful, but the failed backup-copy path should also update observe audit validity and error details or be combined with #70515 before merge.
