## Summary

Keep open. The report is fresh, concrete, and current main still has only adjacent mitigations: memory flush tracks one flush per compaction cycle, compaction duplicate pruning covers a short retry window, and queued follow-up dedupe defaults to message id rather than same prompt content. ## What This Changes

The issue requests that OpenClaw stop re-delivering a prior user message after memory flush and hook-timed compaction complete.

## Best Possible Solution

Add a regression-backed idempotency guard at the memory-flush, compaction, transcript-rotation, and follow-up queue boundary so maintenance or hook-triggered compaction cannot turn the prior user prompt into a fresh user turn, while preserving legitimate repeated prompts and queued follow-ups.
