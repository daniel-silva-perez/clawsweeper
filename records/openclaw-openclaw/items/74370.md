## Summary

Keep open. Current main still has the memory-flush, compaction-hook, transcript-rotation, and follow-up queue surfaces involved here, but the only current duplicate-user-turn pruning I found is scoped to exact long user repeats inside a 60 second retry window; the reported repeat happened roughly nine minutes later after memory flush plus hook-triggered compaction.

## What This Changes

The issue reports duplicate delivery of a single user message after memory flush and hook-triggered compaction.

## Best Possible Solution

Add a regression-backed idempotency guard across the memory flush, hook-triggered compaction, session rotation, and follow-up queue boundary so maintenance compaction cannot re-surface the previous user prompt as a fresh turn, while preserving intentional queued follow-ups and legitimate later repeated prompts.

## Reproduction Assessment

Unclear. The report gives a plausible high-signal sequence and log anchors, but the referenced transcript is private; current main has enough exposed seams for a synthetic regression around memory flush, compaction hooks, session rotation, and queued follow-ups.

## Solution Assessment

Unclear, but a narrow boundary guard is the most maintainable direction. The existing 60 second compaction duplicate pruning is too narrow for this report, while global prompt dedupe would be riskier because users can intentionally repeat a prompt later.
