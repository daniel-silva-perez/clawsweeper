## Summary

Keep open. The report is current, concrete, and current `main` still shows only adjacent duplicate guards: compaction transcript pruning is limited to a 60-second same-content window, the queued-follow-up path defaults to message-id dedupe, and the memory-flush path does not appear to use the existing transcript hash field in production. That leaves the reported nine-minute, hook-timed same-prompt redelivery unresolved enough for a focused fix PR.

## What This Changes

The issue reports that one inbound user message can be delivered again as a fresh user turn after memory flush, hook-timed compaction, and session rotation.

## Best Possible Solution

Add a regression-backed idempotency guard at the memory-flush, compaction, transcript-rotation, and queued-follow-up boundary so maintenance or hook-triggered compaction cannot turn the prior user prompt into a new user turn, while preserving legitimate queued follow-ups and normal session rotation.

## Reproduction Assessment

Yes for a focused synthetic regression: the report gives enough timing and metadata to model one user message, memory flush, hook-triggered compaction, session rotation, and same-content redelivery. Live reproduction remains unclear without a sanitized trajectory or transcript export.

## Solution Assessment

Yes, a narrow idempotency fix in the runtime boundary is the best direction. The current 60-second compaction prune and message-id queue dedupe are narrower than the reported nine-minute same-content duplicate, so this does not need a broader product redesign first.
