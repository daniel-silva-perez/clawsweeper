## Summary

Keep open. The report is current against v2026.4.26 and describes a concrete memory-flush plus compaction timing bug; current main still has the relevant memory flush, compaction hook, and follow-up queue surfaces, but I did not find a current-main guard that clearly prevents the same user prompt from being run again after a hook-triggered compaction several minutes later. The related ClawSweeper items are adjacent memory-flush/threshold/feature reports, not a canonical duplicate for this duplicate-delivery behavior.

## What This Changes

The issue reports that one user message can be delivered a second time about nine minutes after pre-compaction memory flush and auto-compaction, especially when the second compaction is hook-related and files changed.

## Best Possible Solution

Add a regression-backed idempotency guard across the memory flush, compaction, and follow-up queue boundary so maintenance or hook-triggered compaction cannot re-surface the prior user prompt as a fresh user turn, while preserving normal queued follow-ups and session rotation.
