## Summary

Keep open. The report is current and concrete, and current main still has the same memory-flush, compaction-event, transcript-rotation, and follow-up queue boundary where a hook-timed compaction could resurface a prior user prompt. I found adjacent protections, but they do not cover the reported nine-minute, new-entry duplicate-delivery shape.

## What This Changes

The issue reports that after memory flush and hook-timed compaction, OpenClaw re-delivered one user prompt as a fresh user message about nine minutes later.

## Best Possible Solution

Add a regression-backed idempotency guard at the memory-flush, compaction, and follow-up queue boundary so maintenance or hook-triggered compaction cannot turn the prior user prompt into a fresh user turn, while preserving legitimate queued follow-ups and session rotation.

## Reproduction Assessment

Yes for a synthetic regression: the issue gives enough timing and metadata to model one user prompt, memory flush, hook-triggered compaction, transcript/session rotation, and same-content re-delivery. ## Solution Assessment

Yes, a focused idempotency fix is the best direction. The current 60-second compaction duplicate window and message-id-only follow-up dedupe are narrower than the reported nine-minute same-content duplicate, so broad product redesign is not needed before a targeted repair.
