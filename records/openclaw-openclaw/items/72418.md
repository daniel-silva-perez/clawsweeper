## Summary

Keep this open. Current main still contains the reported backend self-pairing exception for loopback/shared-secret gateway-client backend connections, the exception is documented as intentional gateway trust-model behavior, and open PR #72419 is an unmerged closing candidate that still needs maintainer security/product review.

## What This Changes

The issue requests hardening the gateway WebSocket backend self-pairing path so a loopback caller cannot self-declare `gateway-client`/`backend` with shared token or password auth and bypass device pairing.

## Best Possible Solution

Maintain this as an open security/product review item until maintainers either explicitly accept the documented backend exception or replace it with a narrower backend identity boundary, such as a dedicated bootstrap/backend credential plus focused tests for spoofed loopback backend clients without registered devices.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: a declared `gateway-client` backend on loopback, without browser Origin, with valid token/password shared auth makes `shouldSkipLocalBackendSelfPairing` return true.

## Solution Assessment

No for cleanup closure. Current main intentionally preserves the behavior and open PR #72419 is still unmerged, so the maintainable path is explicit maintainer security review rather than closing this issue as fixed.
