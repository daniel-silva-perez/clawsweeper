## Summary

Keep this open. Current main and the latest release still preserve the reported loopback backend self-pairing exception, the behavior is now documented as part of the gateway trust model, and open PR #72419 is an unmerged closing candidate, so this needs maintainer security/product handling rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent loopback clients authenticated with a shared gateway token from self-declaring the backend gateway-client identity and bypassing device pairing.

## Best Possible Solution

Maintain an explicit security/product decision for the backend self-pairing boundary: either keep the documented trusted-operator exception and close after maintainer acceptance, or land/rework #72419 into a narrowly tested change that binds backend pairing bypass to a dedicated backend credential or server-side identity signal without regressing internal gateway-client control-plane calls.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: a loopback/no-Origin caller with valid token/password auth can declare `client.id: gateway-client` and `client.mode: backend`, producing a true backend self-pairing skip and preserving the missing-device path.

## Solution Assessment

Unclear. Current main is not a cleanup fix because it preserves and documents the behavior; #72419 is the only linked implementation candidate, but it remains unmerged and needs maintainer review against the intended gateway trust model.
