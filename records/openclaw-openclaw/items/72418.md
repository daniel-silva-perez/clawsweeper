## Summary

Keep this open. Current main still preserves the reported loopback backend self-pairing exception, the docs frame that exception as intentional gateway trust-model behavior, and open PR #72419 is an unmerged closing candidate, so the issue should remain open for maintainer security/product review rather than cleanup closure.

## What This Changes

The issue reports that a loopback client with a valid shared gateway token can self-declare the backend gateway-client identity, skip device pairing, and retain self-declared operator scopes.

## Best Possible Solution

Keep this as an explicit maintainer security review item until maintainers either accept the documented backend exception as the intended trust model or replace it with a narrower backend identity boundary, with focused tests for spoofed loopback backend clients and scope preservation.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare `client.id: gateway-client` and `client.mode: backend` on loopback without an Origin header, authenticate with a valid token/password, and `shouldSkipLocalBackendSelfPairing` returns true; the message-handler then preserves scopes and bypasses not-paired approval on that path.

## Solution Assessment

No for cleanup closure. Current main intentionally preserves and documents the behavior, and PR #72419 is still open/unmerged, so the maintainable path is maintainer security review or a merged trust-boundary change, not closing this issue now.
