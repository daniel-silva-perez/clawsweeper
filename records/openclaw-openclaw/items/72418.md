## Summary

Keep open. Current main still preserves the reported loopback backend self-pairing path, documents it as an intentional gateway trust-model exception, and the linked closing PR remains the active unmerged candidate rather than shipped behavior.

## What This Changes

The issue asks OpenClaw to prevent a loopback shared-token/password client from self-declaring the backend `gateway-client` identity and skipping device pairing with operator scopes.

## Best Possible Solution

Maintain this as an open gateway security/product review item. If hardening is accepted, bind backend self-pairing to a server-side backend credential or identity and add focused regression coverage for spoofed loopback backend clients without registered devices.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: a declared backend `gateway-client` on loopback, without a browser Origin, with valid token/password shared auth makes `shouldSkipLocalBackendSelfPairing` return true. A live WebSocket run was not performed because this review was read-only.

## Solution Assessment

No for closure. Current main intentionally preserves the behavior, and the active PR path still needs maintainer security/product review before this can be considered solved.
