## Summary

Keep this issue open. Current main still contains the reported shared-token loopback backend self-pairing path, current docs describe it as an intentional gateway trust-model exception, and the same contributor still has open, unmerged closing PR #72419, so this needs maintainer/security review rather than cleanup closure.

## What This Changes

The issue reports that a loopback client with a valid shared gateway token can self-declare `gateway-client`/`backend`, skip device pairing, and preserve self-declared operator scopes.

## Best Possible Solution

Keep this open while maintainers review #72419 and decide the trust-model direction. If the behavior is accepted, close with an explicit security rationale; if hardening is desired, bind backend self-pairing to a server-side backend credential or identity and add focused regression coverage for spoofed loopback backend clients with no registered device.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: a declared gateway backend client on loopback, without a browser Origin, with valid token/password shared auth causes `shouldSkipLocalBackendSelfPairing` to return true; a full live WebSocket run was not performed because this review was read-only.

## Solution Assessment

No for cleanup closure. Current main preserves the reported path and the linked PR is not a complete fix for the shared-token/password condition, so the best route is maintainer security review followed by either an explicit trust-model rejection or a targeted hardening change.
