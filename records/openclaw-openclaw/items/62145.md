## Summary

Keep this open: current main still replaces the whole Control UI with the login gate whenever the Gateway connection drops, so the reported workflow problem remains valid and narrow enough for a focused UI fix.

## What This Changes

The issue requests that the Control UI preserve prior messages after a Gateway disconnect by disabling active controls and offering a reconnect path instead of replacing the whole view with the login gate.

## Best Possible Solution

Keep the initial login gate for first load and credential entry, but after a successful Control UI connection render the normal dashboard/chat shell during transient disconnects with mutating controls disabled, prior messages still visible, a clear disconnected status, and an explicit reconnect action.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: a Gateway WebSocket close sets `connected` false in `app-gateway.ts`, and `renderApp` immediately returns the login gate, which prevents the existing disconnected chat UI from rendering.

## Solution Assessment

Yes. The narrowest maintainable solution is to relax the top-level render gate only for a previously connected session and reuse the existing disabled chat controls plus a reconnect/status affordance, rather than changing Gateway protocol or auth behavior.
