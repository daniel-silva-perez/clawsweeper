## Summary

Keep open. Current main still exposes ACPX `permissionMode` and `nonInteractivePermissions` only as global plugin config, while the agent runtime config, ACP binding schema, configured-binding materialization, persistent binding metadata, and ACP runtime startup contract do not carry the requested scoped permission policy. The prior ClawSweeper comment remains consistent with current main, and the remaining work changes ACP permission semantics, so it should stay with maintainer review rather than conservative cleanup close.

## What This Changes

The issue requests `permissionMode` and related non-interactive permission policy overrides for ACP sessions at binding and agent-runtime scope, with global ACPX plugin config as the fallback.

## Best Possible Solution

Add a typed scoped ACP permission contract for both agent runtime config and ACP binding config, persist and rehydrate it with configured bindings, pass the effective policy into ACP runtime startup, and document/test precedence as binding > agent runtime > global ACPX plugin config for both `permissionMode` and `nonInteractivePermissions`.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the current global ACPX permission policy can over-permit lower-trust ACP sessions when any session needs broader access.

Concerns:

- **[medium] Global ACPX policy over-permits mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
  - body: The ACPX service passes one plugin-level `permissionMode` and `nonInteractivePermissions` into the runtime, while binding and agent runtime config cannot override them. In a mixed-trust gateway, using `approve-all` for one ACP agent broadens the default policy for the others.
