## Summary

Keep open. Current main still exposes ACPX permission policy as plugin-global config only; the typed agent runtime config, strict ACP binding schema, configured-binding resolution, persisted binding metadata, ACP runtime ensure input, and docs do not carry per-binding or per-agent `permissionMode` or `nonInteractivePermissions`. The closed related reports address global ACPX permission/config behavior, and the open ACP-everywhere RFC is broad architecture work rather than an implementation of this scoped permission request.

## What This Changes

Request scoped ACPX permission policy overrides on `bindings[].acp` and `agents.list[].runtime.acp`, falling back to `plugins.entries.acpx.config`.

## Best Possible Solution

Define a typed scoped ACP permission contract for both agent runtime config and ACP binding config, include `permissionMode` and `nonInteractivePermissions`, validate and persist those fields, compute effective policy as binding > agent runtime > plugin global, pass that policy through runtime startup or isolated runtime instances, and document/test the precedence.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a config using `bindings[].acp.permissionMode` or `agents.list[].runtime.acp.permissionMode` is rejected by strict schemas today, and even a bypassed value would be dropped before binding persistence and ACP runtime startup.

## Solution Assessment

Yes in direction, but it should not be a plugin-only patch. The maintainable solution crosses config validation, binding materialization, persisted metadata, ACP runtime API boundaries, ACPX runtime construction, docs, and regression tests so lower-trust sessions are not silently over-permitted.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the current plugin-global ACPX permission policy can over-permit lower-trust ACP sessions when any session needs broader access.

Concerns:

- **[medium] Global ACPX policy over-permits mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
  - body: The ACPX service passes one plugin-level `permissionMode` and `nonInteractivePermissions` into the runtime, while binding and agent runtime config cannot override them. In a mixed-trust gateway, using `approve-all` for one ACP agent broadens the effective policy for the others.
