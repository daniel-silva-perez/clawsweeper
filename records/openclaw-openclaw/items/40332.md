## Summary

Keep open. Current main still only supports ACPX `permissionMode` and `nonInteractivePermissions` as global plugin config, while `agents.list[].runtime.acp`, `bindings[].acp`, configured binding materialization, and ACP runtime session initialization do not carry the requested scoped permission policy. The newer `/acp permissions` runtime control is session-level `approval_policy` metadata and does not solve durable per-binding/per-agent ACPX startup policy or `nonInteractivePermissions`.

## What This Changes

The issue requests per-binding and per-agent ACPX permission policy overrides, with binding config taking precedence over agent runtime defaults and global ACPX plugin config.

## Best Possible Solution

Add an explicit scoped ACP permission contract: typed and strict `permissionMode` plus `nonInteractivePermissions` fields on `agents.list[].runtime.acp` and `bindings[].acp`, generated schema/help/docs updates, configured binding persistence/rehydration, ACP manager ensure input/runtime metadata, and ACPX resolution with binding > agent runtime > global plugin precedence plus regression tests.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because today one ACPX permission policy applies to all ACP sessions on the gateway.

Concerns:

- **[medium] Global ACPX permission policy over-permits mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
  - body: The ACPX service passes one plugin-level `permissionMode` and `nonInteractivePermissions` into the runtime, while binding and agent runtime config cannot override them. In a mixed-trust deployment, raising the global policy for one privileged ACP agent also raises it for lower-trust sessions.
