## Summary

Keep open. Current main still implements ACPX permission policy only as global plugin config; the agent runtime config, ACP binding schema, configured binding materialization, persistent binding metadata, and ACP runtime startup contract do not carry the requested per-binding or per-agent permission policy. The related closed reports cover adjacent global-config problems and point back here for scoped permissions rather than superseding this issue.

## What This Changes

The issue requests `permissionMode` and `nonInteractivePermissions` overrides for ACP sessions at `bindings[].acp` and `agents.list[].runtime.acp`, falling back to global ACPX plugin config.

## Best Possible Solution

Add a typed scoped ACP permission contract for both agent runtime config and ACP binding config, persist and rehydrate it with configured bindings, pass the effective policy into ACP runtime startup, and document/test precedence as binding > agent runtime > global ACPX plugin config for both `permissionMode` and `nonInteractivePermissions`.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the requested keys are absent from the strict agent-runtime and binding schemas, absent from configured-binding persistence, and absent from the runtime startup input, so a config using the proposed scoped fields cannot currently work on main.

## Solution Assessment

Yes in direction, but not as a plugin-only patch. The maintainable solution needs a typed cross-boundary ACP policy contract through config validation, binding materialization, persistence, runtime startup, ACPX runtime construction, docs, and regression tests.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the current global ACPX permission policy can over-permit lower-trust ACP sessions when any session needs broader access.

Concerns:

- **[medium] Global ACPX policy over-permits mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
