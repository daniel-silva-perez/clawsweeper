## Summary

Keep open. Current main still exposes ACPX permission policy only through plugin-global config, while the typed agent runtime config, strict binding schema, configured-binding resolution, persisted binding metadata, ACP runtime startup input, and docs omit the requested per-binding and per-agent policy fields. The related closed reports cover adjacent global ACPX permission/config problems and do not supersede this scoped permission request.

## What This Changes

The issue requests scoped ACPX `permissionMode` and `nonInteractivePermissions` overrides at `bindings[].acp` and `agents.list[].runtime.acp`, with the existing plugin-global ACPX config as fallback.

## Best Possible Solution

Add a typed scoped ACP permission contract for both agent runtime config and ACP binding config, persist and rehydrate it with configured bindings, pass the effective policy into ACP runtime startup or into clearly isolated runtime instances, and document/test precedence as binding > agent runtime > global ACPX plugin config for both permission fields.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a config using the proposed scoped fields is rejected or ignored because the strict schemas, configured-binding persistence, and runtime startup input do not include those keys.

## Solution Assessment

Yes in direction, but not as a plugin-only patch. The maintainable path needs a cross-boundary ACP policy design through config validation, binding materialization, persistence, runtime startup, ACPX runtime construction, docs, and regression tests.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the current global ACPX permission policy can over-permit lower-trust ACP sessions when any session needs broader access.

Concerns:

- **[medium] Global ACPX policy over-permits mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
  - body: The ACPX service passes one plugin-level `permissionMode` and `nonInteractivePermissions` into the runtime, while binding and agent runtime config cannot override them. In a mixed-trust gateway, using `approve-all` for one ACP agent broadens the default policy for the others.
