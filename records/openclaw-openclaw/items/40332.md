## Summary

Keep open. Current main still exposes ACPX `permissionMode` and `nonInteractivePermissions` only as global plugin config; the per-agent runtime schema, per-binding schema, configured binding materialization, persistent binding metadata, and ACP runtime startup path do not carry scoped permission policy.

## What This Changes

Request scoped ACP permission policy so `bindings[].acp` and `agents.list[].runtime.acp` can override the global ACPX plugin permission defaults.

## Best Possible Solution

Add an explicit scoped ACP permission contract with typed strict fields on `agents.list[].runtime.acp` and `bindings[].acp`, persist and rehydrate those fields for configured bindings, pass them through ACP manager/runtime initialization, resolve ACPX policy as binding > agent runtime > global plugin config, and update schema/help/docs plus regression tests.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because the current global ACPX permission policy can over-permit mixed-trust ACP sessions.

Concerns:

- **[medium] Global ACPX policy over-permits mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
  - body: The ACPX service passes one plugin-level `permissionMode` and `nonInteractivePermissions` into the runtime, while binding and agent runtime config cannot override them. If one ACP agent needs `approve-all`, less-trusted sessions on the same gateway inherit that broader policy.
