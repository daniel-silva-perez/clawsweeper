## Summary

Keep open. Current main still treats ACPX permission policy as plugin-global: the agent runtime config, ACP binding schema, configured binding resolver, persisted binding metadata, ACP runtime ensure input, ACPX service construction, and docs do not carry per-binding or per-agent `permissionMode` or `nonInteractivePermissions`. The related closed reports only cover global ACPX permission/config behavior, and the open ACP-everywhere RFC is broader design work rather than an implementation of this scoped permission request.

## What This Changes

Request scoped ACPX `permissionMode` and `nonInteractivePermissions` overrides on `bindings[].acp` and `agents.list[].runtime.acp`, falling back to `plugins.entries.acpx.config`.

## Best Possible Solution

Define a typed scoped ACP permission contract for both agent runtime config and ACP binding config, validate and persist those fields, compute effective policy as binding > agent runtime > plugin global, pass the effective policy through ACP runtime startup without silently reusing a broader policy for lower-trust sessions, and document/test the precedence.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: config containing `bindings[].acp.permissionMode` or `agents.list[].runtime.acp.permissionMode` is rejected by strict schemas today, and bypassed values would be dropped before binding persistence and ACP runtime startup.

## Solution Assessment

Yes in direction, but it should not be a plugin-only patch. The maintainable solution crosses config validation, binding materialization, persisted metadata, the ACP runtime API boundary, ACPX runtime construction, docs, and regression tests.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the current plugin-global ACPX permission policy can over-permit lower-trust ACP sessions when any session needs broader access.

Concerns:

- **[medium] Global ACPX policy over-permits mixed-trust sessions:** `extensions/acpx/src/service.ts:68`
  - body: The ACPX service passes one plugin-level `permissionMode` and `nonInteractivePermissions` into the runtime, while binding and agent runtime config cannot override them. In a mixed-trust gateway, using `approve-all` for one ACP agent broadens the effective policy for the others.
