## Summary

Current main still does not implement per-binding or per-agent ACPX permission policy. `permissionMode` and `nonInteractivePermissions` remain ACPX plugin-level config, while ACP agent runtime defaults, configured ACP bindings, persistent binding metadata, and ACP runtime session initialization still carry only agent/backend/mode/cwd/label-style fields. The prior ClawSweeper discussion correctly kept this open, and the same gap remains on current main.

## Best Possible Solution

Keep this issue open and implement it as a core ACP/runtime contract change: add `permissionMode` and `nonInteractivePermissions` to `agents.list[].runtime.acp` and `bindings[].acp` schema/types/help/docs, persist them in configured ACP binding specs/metadata, thread them through ACP session initialization or a versioned runtime ensure option, and make ACPX resolve precedence as binding override, then agent runtime default, then global `plugins.entries.acpx.config`. Add validation, precedence, binding materialization/rehydration, configured binding initialization, and ACPX runtime wiring tests.

## Risks / Open Questions

- Mixed-trust ACP deployments still need one global ACPX permission policy, so enabling `approve-all` for a privileged ACP agent can over-permit lower-trust ACP sessions on the same gateway.
- A correct fix changes public config schema/help/docs and ACP runtime/plugin permission semantics, so it needs focused maintainer API and security review plus regression tests.
