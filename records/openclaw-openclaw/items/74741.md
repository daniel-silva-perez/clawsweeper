## Summary

Keep open. Current main still matches the reported Signal SSE timeout path: the Signal client keeps a 10s default deadline for event streams, the reconnect loop calls it without a long-lived-stream override, and the latest released Signal files are unchanged from current main on this surface. The issue is also paired with open fixing PR #74744, with #73086 as an overlapping alternative, so it should stay open until a PR lands or maintainers choose a replacement path.

## What This Changes

The Signal channel should keep the `signal-cli` `/api/v1/events` SSE consumer alive for inbound messages instead of repeatedly reconnecting after the 10s HTTP deadline and leaving `lastInboundAt` unset.

## Best Possible Solution

Review and land one narrow Signal-plugin fix, preferably one of the existing open PRs, that disables the short deadline only for the long-lived receive stream while preserving bounded RPC/check deadlines, reconnect behavior, parsing limits, and access-control behavior.
