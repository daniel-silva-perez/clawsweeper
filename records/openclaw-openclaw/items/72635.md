## Summary

Keep open. The linked Docker bug is valid and this PR moves toward the right user-visible outcome, but the current patch is not mergeable because its missing-`python3` path falls back inside the low-level pinned helper to a path-based writer, bypassing the helper contract that currently rejects symlink-parent writes. That is a security-sensitive boundary regression for fs-safe/write paths, and it should be resolved before merge.

## What This Changes

The PR adds spawn-error detection around `runPinnedWriteHelper()` and new tests so missing or unexecutable `python3` falls back to the existing local writer for buffer and stream inputs.

## Best Possible Solution

Handle missing `python3` while preserving pinned-directory guarantees: either implement a Node fallback with equivalent no-follow descriptor semantics, or let `fs-safe` catch helper spawn failures and use the already-hardened legacy write/copy paths with regression coverage for symlink-parent, retarget-race, buffer, and stream-copy cases.

## Reproduction Assessment

Yes. The original bug has a concrete #72362 Docker reproduction, and the introduced regression can be reasoned from the existing symlink-parent test plus the PR's mocked spawn-ENOENT fallback path: that combination would route a case that currently rejects into the path-based fallback writer.

## Solution Assessment

No. Falling back inside `runPinnedWriteHelper()` to `runPinnedWriteFallback()` is not the safest maintainable fix because it bypasses the helper's current pinned parent-directory semantics; the fallback should preserve those semantics or occur at the higher fs-safe layer through an already-tested legacy path.
