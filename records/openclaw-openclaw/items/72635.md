## Summary

Keep this PR open because it targets a valid current-main bug from #72362, but the proposed helper-level fallback should not merge as-is: it broadens a direct local writer fallback on a security-sensitive fs-safe path and leaves a stream-consumption data-loss edge case. A related open implementation, #73791, appears to attack the same bug at the fs-safe caller layer and is a better comparison point for maintainer review.

## What This Changes

This PR adds spawn-error detection to the pinned write helper and a new spawn-fallback test so missing `python3` falls back to the local writer for buffer and stream writes.

## Best Possible Solution

Adopt one narrow missing-Python fix that detects helper spawn failures before misleading normalization while preserving the write boundary guarantees as much as possible, ideally at the `fs-safe.ts` caller layer where buffer data can be reused and copy sources can be reopened. Non-spawn Python helper errors should continue to flow through the pinned-helper path.

## Reproduction Assessment

Yes. #72362 gives a high-confidence Docker reproduction using `node:lts-bookworm-slim` without `python3`, and current main still has the matching `python3` helper path plus generic `normalizePinnedWriteError` wrapping.

## Solution Assessment

No, not as written. The intended fallback is useful, but putting it inside `runPinnedWriteHelper` broadens the direct local writer and creates a consumed-stream fallback path; a fs-safe-level fallback or a pinned-safe Node implementation is the safer direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep fallback on a verified write path:** `src/infra/fs-pinned-write-helper.ts:216`
  - body: The new missing-Python branch calls `runPinnedWriteFallback`, whose implementation writes and renames through ordinary paths. If a parent path is swapped to a symlink after the initial target resolution, this fallback can follow it and only fail post-write verification after clobbering outside the root. Handle spawn failures at the fs-safe layer through the existing verified legacy write/copy paths, or add equivalent pinned validation before expanding this fallback.
  - confidence: 0.84
- **[P2] Avoid fallback after consuming stream input:** `src/infra/fs-pinned-write-helper.ts:239-241`
  - body: The late spawn-error check runs after `pipeline(params.input.stream, child.stdin)` has completed. If that path is ever reached for a stream input, the fallback receives an already-drained stream and can atomically replace the target with empty content; throw for stream inputs in this late path or restructure so fallback happens before consumption.
  - confidence: 0.77

## Security Review

Status: needs_attention

Summary: The diff changes a security-sensitive filesystem write boundary and expands a fallback that lacks the pinned helper's dir_fd and `O_NOFOLLOW` protections.

Concerns:

- **[high] Fallback bypasses pinned write protections:** `src/infra/fs-pinned-write-helper.ts:216`
  - body: The added spawn-failure path sends normal Linux writes through `runPinnedWriteFallback`, which uses ordinary path joins, temp writes, and rename. That is weaker than the Python helper's dir_fd/O_NOFOLLOW parent walk and can turn a missing `python3` environment into a broader symlink-race exposure before post-write verification detects the final path.
  - confidence: 0.84
