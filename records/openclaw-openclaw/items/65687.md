## Repair Work Prompt

Fix the Discord voice Deepgram follow-up regression reported in openclaw/openclaw#65687. Observable bug: when the active `mediaUnderstandingProviders` registry contains providers such as OpenAI/MiniMax but omits configured Deepgram, `resolvePluginCapabilityProviders({ key: "mediaUnderstandingProviders", cfg })` returns the active list early and never loads the bundled Deepgram compat provider from `tools.media.audio.models`, leading Discord voice STT to throw `Media provider not available: deepgram` on follow-up turns.

Expected boundary: update `src/plugins/capability-provider-runtime.ts` and `src/plugins/capability-provider-runtime.test.ts`. Add a regression test with a partial active media-understanding registry plus config using `tools.media.audio.models: [{ provider: "deepgram", model: "nova-3" }]`; expect the resolved provider ids to include the active providers and Deepgram, deduped by provider id. Preserve scoped compat loading and avoid unrequested broad bundled provider fanout. Do not change Discord audio capture, Deepgram HTTP behavior, plugin manifests, startup activation policy, or unrelated provider generation behavior.

Because this is a user-facing bug fix, add a single-line changelog Fixes entry in the active version section with `Thanks @OneMintJulep`. Run the focused resolver test and targeted formatting locally, then run `pnpm check:changed` in Testbox before handoff.
