## Summary

Keep open. Current main still leaves the reported Discord voice Deepgram follow-up failure plausible: media-understanding provider resolution returns a partial active non-speech registry before loading configured compat providers, while Discord voice STT uses that shared media transcription path and still throws the reported missing-provider error when Deepgram is absent.

## What This Changes

The issue reports Discord voice follow-up turns failing after one successful exchange when Deepgram STT is configured through `tools.media.audio`, because provider resolution can omit Deepgram from a partially populated active media registry.

## Best Possible Solution

Extend capability provider resolution so configured/requested audio media-understanding providers from `tools.media.audio.models` are merged into a partial active registry by provider id, then cover the regression with a focused resolver test for active OpenAI/MiniMax media providers plus configured Deepgram audio STT.
