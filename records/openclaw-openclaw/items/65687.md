## Summary

Keep open. Current main still leaves the reported Discord voice Deepgram follow-up failure actionable: Discord voice uses the shared media-understanding audio path, and `resolvePluginCapabilityProviders` still returns a partial active `mediaUnderstandingProviders` registry before loading configured/compat Deepgram providers.

## What This Changes

The issue requests that Discord voice STT continue resolving configured Deepgram media providers across follow-up turns when the active media provider registry is only partially populated.

## Best Possible Solution

Extend capability provider resolution so configured `tools.media.audio.models` providers are merged into a partial active `mediaUnderstandingProviders` registry by provider id, preserving the active-registry fast path where it is safe and covering the Deepgram case with a focused regression test.

## Reproduction Assessment

Yes. The issue provides a concrete Discord voice plus Deepgram config and logs, and current main still has the static path that can reproduce the missing-provider condition when active media providers exist but Deepgram is only available through the compat/config-shaped load.

## Solution Assessment

Yes, with a narrow implementation. The maintainable fix is not Discord-specific; it should add media-audio requested-provider reconciliation to the shared capability resolver and prove active OpenAI/MiniMax plus configured Deepgram returns the union.
