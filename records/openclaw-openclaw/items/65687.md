## Summary

Current main still appears to leave the reported Discord voice Deepgram follow-up failure open: media-understanding provider resolution returns a partial active registry before compat fallback, while Discord voice STT uses the shared media transcription path that throws the reported provider-missing error.

## What This Changes

Discord voice follow-up transcription configured with Deepgram should keep resolving the compat/bundled Deepgram media provider after an initial turn instead of dropping it when other media providers are already active.

## Best Possible Solution

Extend capability provider resolution so requested tools.media.audio media-understanding providers are merged into a partial active registry by provider id, then add a regression test for active OpenAI/MiniMax media providers plus configured Deepgram audio STT.

## Repair Work Prompt

Fix the Deepgram Discord voice follow-up provider-resolution regression tracked by openclaw/openclaw#65687. In `src/plugins/capability-provider-runtime.ts`, preserve the existing active-registry fast path where safe, but when `key === "mediaUnderstandingProviders"` and a cfg is supplied, collect requested provider ids from `tools.media.audio.models` and shared `tools.media.models` entries that apply to audio, remove ids already present in the active registry, load only the bundled compat owners for missing requested providers, and merge those providers by id. Do not broadly load unrelated bundled media providers and do not change Discord voice capture/playback behavior. Add a focused regression test in `src/plugins/capability-provider-runtime.test.ts` where the active media registry contains OpenAI/MiniMax, cfg requests Deepgram via `tools.media.audio.models`, the loaded compat registry contains Deepgram, and the resolved provider ids include Deepgram without adding unrelated providers. Changelog is likely required because this is a user-facing Discord voice/STT bug fix; credit the issue reporter rather than Codex.
