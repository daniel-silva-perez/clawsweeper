## Summary

Keep this issue open. Current main supports a single run-level `heartbeat.model` plus interval-based `HEARTBEAT.md` tasks, but tasks do not carry a model/escalation field and all due tasks are folded into one heartbeat prompt before one reply call. The related reports in the provided context cover the run-level `heartbeat.model` bug, not per-task model selection.

## Best Possible Solution

Keep open for maintainer product/API design. A complete implementation should add a documented optional `model` or escalation field to structured `HEARTBEAT.md` tasks, define mixed-model execution semantics, update heartbeat docs/config help, and add tests covering cheap/default tasks, primary/escalated tasks, skipped tasks, timestamp updates, and delivery behavior.

## Risks / Open Questions

- A heartbeat implementation needs an explicit mixed-model contract: choose one model for the whole tick, group due tasks into multiple model calls, or re-run only escalated tasks after a cheap pass.
- Timestamp updates, delivery behavior, failure handling, and shared-memory context need regression coverage if due tasks can run in multiple model calls.
