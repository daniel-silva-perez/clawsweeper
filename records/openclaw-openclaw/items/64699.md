## Summary

Keep open. Current main still rejects a `sessions_send` payload that contains both `sessionKey` and `label` before resolving the target, which matches the reported failure mode. The latest release source still has the same guard, and open PR #74009 plus fork PR BingqingLyu/openclaw#2317 are active implementation paths rather than shipped fixes.

## What This Changes

The issue reports that `sessions_send` can receive an unintended `label` alongside an intended `sessionKey`, causing the tool's mutual-exclusion validation to reject an otherwise valid send.

## Best Possible Solution

Land one canonical `sessions_send` fix that treats an explicit `sessionKey` or session-id-shaped `sessionKey` as authoritative when redundant `label` or `agentId` hints are present, preserves label-only resolution and A2A visibility checks, updates the tool description, and adds focused regression tests for redundant and conflicting selector cases.
