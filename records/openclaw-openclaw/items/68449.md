## Summary

Keep this issue open. Current main has partial Dreaming hygiene fixes, but the remaining report is still actionable: direct cron-prompt fallback still preserves paired assistant narration, Dreaming still lacks an operator-facing session/cron exclusion config, REM legacy tag filtering is narrow, and light-dream narrative themes still come from raw persisted conceptTags.

## What This Changes

The issue reports that memory-core Dreaming ranks stopwords and transcript markers as themes and ingests cron/NO_REPLY session noise into its corpus.

## Best Possible Solution

Keep this issue open until memory-core has one coherent Dreaming cleanup path: preserve the current source-side stopword and archive/session classification improvements, add a shared concept-tag noise filter used by REM and light narrative inputs, cover decimal/template/legacy tags, and handle cron/session exclusions through spoof-safe session metadata or documented operator config coordinated with #72611.

## Reproduction Assessment

Yes. The original deployment report is backed by current source and fixtures: unclassified direct cron prompts can still leave assistant narration in exported session text, while REM and light paths still consume persisted conceptTags without a broad read-path noise filter.

## Solution Assessment

No, current main is not yet the best complete solution. The safest direction is a shared read-path tag sanitizer plus metadata-backed or configurable session exclusion; a regex that drops the next assistant line after any cron-looking user text would conflict with the existing spoofing regression guard.
