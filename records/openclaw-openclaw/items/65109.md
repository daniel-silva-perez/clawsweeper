## Summary

Keep open. Current main still displays cache read/write tokens in `/status`, but the cost estimate path passes only input/output tokens into the shared estimator; the cache-aware estimator and docs already support the missing fields. Open PR #65381 targets this exact runtime fix, so this issue should remain open until that PR, or an equivalent fix, lands.

## What This Changes

Request that `/status` include `cacheRead` and `cacheWrite` token pricing in the displayed cost estimate when cache token counts are present.

## Best Possible Solution

Land a narrow runtime fix that forwards `cacheRead` and `cacheWrite` into the existing `/status` cost estimate, treats cache-only usage as billable usage, and adds focused regression coverage for session-entry and transcript-derived cache usage. PR #65381 is the current candidate for that outcome.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: build a status message with nonzero cache read/write usage and API-key pricing, then observe that current main's cost object excludes those cache fields even though the cache line displays them.

## Solution Assessment

Yes, the best fix is to reuse the existing cache-aware `estimateUsageCost` contract from the `/status` caller rather than adding a second pricing path. The open PR direction appears aligned, but it still needs normal maintainer review before this issue can close.
