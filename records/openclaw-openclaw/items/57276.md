## Summary

Keep this PR open. Current main still does not implement managed system-scoped systemd gateway lifecycle support: Linux gateway status, runtime reads, stop/restart, uninstall, and the update restart helper all remain wired to user-scoped systemd paths. The PR is a plausible focused fix for the reported WSL system-unit problem, but the existing review comments identify real follow-up needed around sudo use, uninstall permissions, and test mocks before it should merge.

## Best Possible Solution

Keep this PR open and ask for a focused revision: add a shared user-vs-system systemd service context for the managed gateway path, use direct `systemctl` for read-only system-scope status where possible, reserve sudo for mutating operations and handle root/no-sudo cases, fix system-unit uninstall permission handling, and add regression tests for both user-scoped and system-scoped units plus update restart-helper generation.

## Risks / Open Questions

- Merging the PR as-is could make read-only status/show checks require passwordless sudo even though systemd often allows those unprivileged.
- The generated restart helper in the PR uses `sudo -n systemctl` for system units even when running as root or where sudo is unavailable, which can break post-update restart.
- The PR's system-unit uninstall path appears to remove `/etc/systemd/system/...` with direct `fs.unlink`, so non-root permission failures can be misreported as a missing service and leave the unit installed.
- Existing tests that mock `fs.access` broadly need path-specific updates once a system-unit probe is introduced.
- Security pass found no supply-chain or CI expansion, but the new privileged service-control path needs tighter command/privilege behavior before landing.
