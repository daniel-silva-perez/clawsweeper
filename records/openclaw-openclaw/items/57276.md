## Summary

Current main has not implemented the PR's intended system-scoped systemd gateway support. The Linux daemon adapter still resolves the managed unit under the user systemd directory and uses `systemctl --user` for status, runtime reads, uninstall, stop, and restart; the update restart helper also always emits `systemctl --user restart`. The PR addresses a real missing behavior, but it still needs maintainer review and fixes for the review-commented test and privileged-operation edge cases before it can land.

## Best Possible Solution

Keep this PR open for focused maintainer review. The best path is to update the PR so the daemon has a shared user-vs-system systemd service context, uses direct `systemctl` for read-only system status where appropriate, uses privileged operations only where necessary, handles system-unit uninstall errors accurately, updates the restart helper with root/non-root behavior, and adds regression coverage for both user-scoped and system-scoped gateway units.

## Risks / Open Questions

- The PR touches privileged service control behavior by adding system-scope `systemctl` and `sudo -n systemctl` paths; review should confirm root, non-root, passwordless-sudo, and sudo-absent environments before merge.
- Existing review comments identify likely test regressions from broad `fs.access` mocks and a misleading system-unit uninstall failure path; those should be fixed in the PR rather than closing it as obsolete.
- Current main's deep systemd service scanning could make the feature look partially present, but it does not repair the managed `gateway status` or `gateway restart` paths this PR targets.
