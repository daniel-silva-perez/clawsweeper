## Summary

Keep this PR open. Current main still does not implement the PR's intended managed system-scoped systemd gateway behavior: the Linux daemon adapter reads, checks, stops, restarts, and uninstalls the user unit, and the update restart helper still generates `systemctl --user restart`. The PR is a plausible focused fix, but review comments identify test and privileged-service-control issues that should be resolved before merge.

## Best Possible Solution

Keep this PR open and ask for a focused revision: introduce a shared user-vs-system systemd service context, use direct `systemctl` for read-only system-scope status where possible, reserve privilege escalation for mutating operations, handle system-unit uninstall permissions explicitly, update the restart helper for root/non-root/no-sudo cases, and add regression coverage for both user-scoped and system-scoped gateway units.

## Risks / Open Questions

- Merging the PR as-is could require passwordless sudo for read-only systemd status/show checks that normally work unprivileged.
- The generated restart helper in the PR uses `sudo -n systemctl` for system units even when already root or when sudo is absent.
- The PR's system-unit uninstall path appears to use direct file deletion for `/etc/systemd/system`, which can hide permission failures and leave the unit installed.
- Existing user-scope tests with unconditional `fs.access` mocks likely need path-specific updates if the system-unit probe is introduced.
