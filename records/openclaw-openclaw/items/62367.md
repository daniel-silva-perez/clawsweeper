## Summary

Keep open. Current main clarifies and tests direct OpenAI `openai/gpt-5.4-nano`, and the Codex OAuth provider now has explicit support/suppression for several GPT-5.4 Codex routes, but I did not find an implementation, explicit rejection, or regression test for the exact reported `openai-codex/gpt-5.4-nano` behavior. The remaining gap is still actionable: Codex nano should either be a real preserved route or fail before normal fallback with a precise unsupported-model error.

## Best Possible Solution

Keep this issue open and decide the Codex OAuth product contract for `openai-codex/gpt-5.4-nano`. If upstream supports nano, add a Codex resolver/catalog path plus tests proving the runtime preserves `gpt-5.4-nano`. If it is unsupported, add a manifest suppression and runtime/list tests so the model fails with a precise `Unknown model: openai-codex/gpt-5.4-nano` explanation before unrelated provider fallback.
