## Summary

Keep open. Current main restored the Codex OAuth mini route but still lacks a first-class Codex OAuth nano route or an explicit unsupported-model guard for nano; direct OpenAI nano support does not prove the `openai-codex` runtime behavior is fixed.

## What This Changes

The issue reports that `openai-codex/gpt-5.4-nano` does not remain a real nano Codex runtime route and may normalize to `gpt-5.4` or fall through to another provider.

## Best Possible Solution

Make the OpenAI plugin own the decision: if Codex OAuth currently accepts nano, add `openai-codex/gpt-5.4-nano` with dynamic resolution, catalog augmentation, runtime preference, docs, changelog, and focused regression tests matching the mini path; if it is unsupported, add a precise unsupported-model error/suppression so it cannot silently normalize or fall through.

## Reproduction Assessment

Yes for a source-level reproduction: current main has no `gpt-5.4-nano` branch in the OpenAI Codex provider, while direct OpenAI nano and Codex mini are covered. A live provider reproduction is still unavailable in this environment because usable Codex OAuth credentials were not present.

## Solution Assessment

Unclear until maintainers confirm upstream Codex OAuth nano availability. The narrow maintainable solution belongs in `extensions/openai` provider-owned model resolution and catalog metadata, not in a generic core fallback policy change unless the remaining fallthrough proves to be broader.
