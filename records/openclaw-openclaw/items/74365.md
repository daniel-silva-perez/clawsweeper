## Summary

Keep open. Current main has a shared CLI progress helper and several command families already use it, but the broad loading-state request is not fully implemented: skills search/install/update and plugin install/update paths still rely on direct ClawHub/npm/archive work, ordinary logger lines, or lifecycle tracing instead of a consistent progress wrapper.

## What This Changes

Add visible loading/progress indicators to slow OpenClaw CLI commands so users can distinguish active work from a hang.

## Best Possible Solution

Use the existing shared progress helper as the standard UI contract, then scope the slow command families and add command-boundary progress labels/tests/docs while preserving JSON/plain/non-TTY behavior.

## Reproduction Assessment

No high-confidence runtime reproduction is available because the report does not identify exact commands or timings. Source inspection does verify the remaining coverage gap: several skills and plugins flows still lack the shared progress wrapper on current main.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable solution is a scoped rollout of the existing progress helper at selected slow command boundaries, not blanket loaders across every CLI action.
