## Summary

Keep open. Current main has a shared CLI progress helper and coverage on several slow command families, but the broader loading-state request is still not complete because ClawHub skill search/install/update and plugin install/update paths still run through direct log lines or lifecycle tracing rather than a consistent progress wrapper.

## What This Changes

Request to add visible loading/progress states to slow OpenClaw CLI commands so users can tell work is progressing.

## Best Possible Solution

Keep this as a scoped CLI UX tracking item: choose the slow command families and delay/TTY/JSON/plain-output semantics, then apply the existing progress helper at those command boundaries with focused regression tests for clean machine-readable output.

## Reproduction Assessment

No for the broad report: it does not provide exact commands or timings to reproduce a user-visible blank pause. Static inspection does give high confidence that several plausible slow paths still lack the shared progress wrapper.

## Solution Assessment

Unclear until maintainers scope it. The right direction is to reuse the existing progress helper, but the best implementation should target selected command boundaries and preserve JSON/plain/non-TTY behavior rather than adding loaders everywhere.
