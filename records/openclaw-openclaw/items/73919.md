## Summary

Keep this PR open, but do not merge it as-is. Current main still only covers part of the one-shot bundled MCP cleanup path, while this PR is the right narrow replacement for #68450; however, the modified session runtime manager still has a concrete concurrent replacement race in the stale in-flight path.

## What This Changes

The PR threads a one-shot bundled MCP cleanup flag through cron, heartbeat, queued follow-up, and reply-runner paths, removes the isolated cron rollover cleanup skip, hardens session MCP runtime in-flight disposal, and adds regression tests plus a changelog entry.

## Best Possible Solution

Revise this PR as the focused implementation candidate: keep the cleanup flag propagation and isolated rollover coverage, add a post-await recheck or equivalent per-session single-flight guard in the runtime manager, preserve the #68450 credit and narrow scope, then validate the touched MCP, cron, heartbeat, and follow-up tests before the changed gate.

## Reproduction Assessment

Yes. The source PR provides a high-confidence live A/B repro for the underlying leak, current main source inspection shows the missing cleanup paths, and the remaining review finding can be reproduced by interleaving a second getOrCreate for the same session while the first stale in-flight replacement path is awaiting the old promise.

## Solution Assessment

No as-is. The cleanup direction and scope are right, but the runtime-manager race fix should re-read the session runtime/in-flight maps after awaiting the stale promise before creating another replacement runtime.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Recheck the session after stale in-flight disposal:** `src/agents/pi-bundle-mcp-runtime.ts:508`
  - body: In the different-fingerprint path, this deletes the old in-flight entry and awaits its promise before creating the replacement. During that await, another `getOrCreate` for the same session can install or cache a newer runtime; this continuation then creates another runtime and can overwrite or leak the newer one. Re-read `runtimesBySessionId` / `createInFlight` after the await and reuse the newer entry before creating a replacement.
  - confidence: 0.78
