## Summary

Keep this PR open. Current main still has the unguarded exec-event heartbeat wake behavior in both targeted call sites, the linked same-author issue remains open, and the PR is a plausible narrow implementation candidate rather than cleanup-close material; the main open question is whether the latest cron-run reproduction should be folded into this PR before merge.

## What This Changes

The PR imports `isSubagentSessionKey`, skips `requestHeartbeatNow` for subagent session keys in `maybeNotifyOnExit` and `emitExecSystemEvent` while preserving `enqueueSystemEvent`, and adds an `emitExecSystemEvent` regression test for subagent keys.

## Best Possible Solution

Keep the contributor PR in review and land a narrow exec-runtime guard that suppresses redundant heartbeat wakes for non-user-facing isolated sessions while preserving system-event enqueueing. The final patch should include regression coverage for both `emitExecSystemEvent` and notify-on-exit, and maintainers should decide whether cron-run keys belong in this PR or a linked follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: call `emitExecSystemEvent` or complete a notify-on-exit background exec with `agent:main:subagent:<id>`, and the code still calls `requestHeartbeatNow` with that scoped key; `resolveHeartbeatSession` then routes forced subagent keys to the main session.

## Solution Assessment

Yes for the originally reported subagent bug: guarding the wake call while preserving `enqueueSystemEvent` is the narrowest maintainable fix. The final merge scope is unclear because the latest comment reports the same pattern for cron-run keys, which may warrant adding `isCronRunSessionKey` coverage before merge.
