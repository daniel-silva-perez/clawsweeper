## Summary

Keep this PR open. The reported status bug is still present on current main, and the PR moves the fix to the right formatter layer, but the branch is stale/dirty and would regress current signal-labeled stopped-service diagnostics unless it is rebased carefully.

## What This Changes

The PR changes `src/daemon/runtime-format.ts` to hide `lastExitStatus` and `lastExitReason` from formatted daemon runtime output when `runtime.status` is `running`.

## Best Possible Solution

Land a rebased formatter-level fix that hides last-exit status and reason only in human status output for running services, preserves raw service runtime fields and signal labels for non-running states, and adds focused `runtime-format` regression tests for both suppression and preservation.

## Reproduction Assessment

Yes. A runtime object with `status: "running"`, `state: "active"`, `lastExitStatus: 1`, and `lastExitReason: "exit-code"` will still render stale exit details on current main because `formatRuntimeStatus` appends those fields unconditionally; live systemd is useful but not required to verify the formatter bug.

## Solution Assessment

Yes in direction, but not as a merge-ready branch. Formatter-level suppression is the narrowest maintainable solution because restart and onboarding diagnostics can still consume the raw runtime fields; the branch should be rebased to wrap the guard around `formatLastExitStatus(...)` and add regression coverage.
