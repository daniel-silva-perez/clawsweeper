## Summary

Keep this PR open. Current main still does not implement the proposed provider-usage TTL cache or `usage.cacheTtlMs` config, and status surfaces still invoke `loadProviderUsageSummary` directly. The PR remains a plausible implementation candidate for the linked Anthropic 429 problem, but it needs rebase/rework against the current loader contract before it can be considered obsolete or merge-ready.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a rebase or replacement patch against current main. The best path is to add the TTL cache in or near `src/infra/provider-usage.load.ts`, preserve the current provider-usage loader options and plugin/auth contracts, wire any accepted `usage.cacheTtlMs` setting through config types/schema/help/labels/generated metadata, and add focused regression tests for cache correctness and boundary cases.

## Risks / Open Questions

- Replaying the PR as-is may regress current loader behavior because current main now carries `config`, `env`, `workspaceDir`, and `skipPluginAuthWithoutCredentialSource` through provider usage resolution.
- A provider-usage cache needs careful key boundaries for provider list, agent/workspace/config/env, and explicit auth so one caller does not receive another caller’s usage summary.
- Regression coverage should include cache hit, TTL expiry, disabled cache, provider-list boundaries, workspace/config/env boundaries, explicit-auth bypass, and non-caching of transient provider failures such as 429s or timeouts.
