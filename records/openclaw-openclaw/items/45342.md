## Summary

Keep this PR open. Current main still does not implement the provider-usage TTL cache or the proposed `usage.cacheTtlMs` config, and the status surfaces still call `loadProviderUsageSummary` directly. The PR remains a plausible implementation candidate for the Anthropic OAuth 429 problem, but it needs rebase/rework against the current provider-usage loader contract before it can be considered obsolete or merge-ready.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a rebase or replacement patch against current main. The best path is to add the TTL cache in or near `src/infra/provider-usage.load.ts`, preserve the current loader options and plugin/auth contracts, wire any accepted `usage.cacheTtlMs` setting through config types/schema/help/labels/generated metadata, and add focused regression tests for cache hit, TTL expiry, disabled cache, provider-list boundaries, workspace/config/env boundaries, explicit-auth bypass, and transient-failure retry behavior.

## Risks / Open Questions

- The PR as described may need rebase/rework because current main's loader contract now carries `config`, `env`, `workspaceDir`, and `skipPluginAuthWithoutCredentialSource` through provider usage resolution.
- Cache key boundaries need care: provider list, agent/workspace/config/env, and explicit auth must not let one caller receive another caller's usage summary.
- Transient provider failures such as 429s and timeouts should not poison the cache for the full TTL.
- Security review of the provided PR file list found no workflow, dependency, install script, package publishing, artifact download, or secret-handling changes; the main security-adjacent concern is avoiding cross-account/provider data leakage through an overly broad in-memory cache key.
