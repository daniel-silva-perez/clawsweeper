## Summary

Current main still does not provide the requested single supported local bootstrap path for daemon install/refresh, dashboard auth, Telegram bot setup, and non-interactive Telegram owner DM access. The building blocks exist, and Telegram username resolution has improved since the issue was filed, but onboarding/channel CLI surfaces still require separate steps and do not expose Telegram owner/dmPolicy/allowFrom automation.

## Best Possible Solution

Keep this open as a first-run CLI/product gap. The best implementation is a first-class local bootstrap surface, either by extending `openclaw onboard --non-interactive` or adding a dedicated bootstrap command that composes daemon install/refresh, dashboard URL output with auth bootstrap, Telegram token setup, Telegram username-to-ID resolution, `dmPolicy: "allowlist"`, numeric owner `allowFrom`, and `commands.ownerAllowFrom`, with public docs and automated tests. If unattended DM pairing remains unsupported, document that explicitly and route automation through allowlist/preapproval rather than bypassing the trust model.

## Risks / Open Questions

- A partial implementation could blur the explicit DM pairing and command-owner trust model; the safer target is allowlist/preapproval automation that writes numeric `allowFrom` and `commands.ownerAllowFrom` deliberately.
- The requested user experience starts from the public one-line installer, so a complete fix may need coordination with installer docs/scripts in addition to this repo's CLI and Telegram plugin code.
- Current main already has a Telegram resolver; duplicating username lookup instead of reusing the channel resolver would risk drift.
