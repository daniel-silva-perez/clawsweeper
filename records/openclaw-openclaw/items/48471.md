## Summary

Keep this issue open. Current main has installer/onboarding, daemon install, dashboard token URL handling, Telegram setup, numeric allowlist setup, and a Telegram username resolver, but it still does not ship the requested single supported local bootstrap path that installs/refreshes the daemon, opens or prints dashboard auth, configures Telegram, and non-interactively sets the intended Telegram owner DM access. The Ryce comment and Keep My Claw link are useful state-preservation/workaround context, not evidence that OpenClaw core has implemented the requested bootstrap surface.

## Best Possible Solution

Keep this issue open as a first-run CLI/product gap. The best implementation is a first-class local bootstrap surface, either by extending `openclaw onboard --non-interactive` or adding a dedicated bootstrap command that composes daemon install/refresh, dashboard URL output with auth bootstrap, Telegram token setup, Telegram username-to-ID resolution, `dmPolicy: "allowlist"`, and numeric owner `allowFrom`, with docs and automated tests. The trust model should stay explicit: if unattended DM pairing remains unsupported, the supported path should be allowlist/preapproval automation rather than silent pairing bypass.
