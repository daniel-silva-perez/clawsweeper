## Summary

Current main has several pieces of the requested flow, including installer/onboarding, daemon install, dashboard token URL handling, Telegram setup, and a Telegram username resolver. It still does not ship the requested single supported one-line local bootstrap that installs/refreshes the daemon, opens/prints dashboard auth, configures Telegram, and fully configures owner DM access non-interactively. The related comment about a bootstrap state manifest is useful follow-up context, not a reason to close the issue.

## Best Possible Solution

Keep this issue open as a product/CLI onboarding gap. The best path is a first-class scripted bootstrap surface, likely by extending `openclaw onboard --non-interactive` or adding a dedicated bootstrap command that composes daemon install, dashboard URL output, Telegram token setup, username-to-ID resolution, `dmPolicy: "allowlist"`, and numeric owner `allowFrom`, with docs and tests. The existing Telegram resolver and dashboard token URL handling should be reused as building blocks.
