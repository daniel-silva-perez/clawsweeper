## Summary

Keep open. Current main still has the separate installer, non-interactive onboarding, dashboard, channel add, Telegram resolver, and Telegram access-control pieces, but it does not provide the requested single supported local bootstrap path that configures the daemon, dashboard auth, Telegram bot token, Telegram DM allowlist, and command-owner access non-interactively.

## What This Changes

Request to add a supported one-line local bootstrap flow that installs or refreshes OpenClaw, starts the local gateway service, opens or prints dashboard auth, configures Telegram, and safely pre-authorizes the intended Telegram owner.

## Best Possible Solution

Add a first-class local bootstrap surface that composes daemon install or refresh, dashboard auth URL output, Telegram bot token setup, Telegram username-to-ID resolution where possible, durable numeric `allowFrom`, `dmPolicy: "allowlist"`, and `commands.ownerAllowFrom`, with docs that explicitly say unattended DM pairing is not bypassed and tests covering the supported automation path.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. There is a high-confidence static verification path: inspect `openclaw onboard --non-interactive`, `OnboardOptions`, non-interactive local setup, `channels add`, Telegram resolver, and the docs, which shows the requested composed bootstrap surface is still missing.

## Solution Assessment

Unclear because there is no current patch to evaluate. The narrowest maintainable direction is to reuse the existing Telegram resolver and access-control adapters while adding a deliberate bootstrap/onboarding surface rather than weakening pairing or duplicating lookup logic.
