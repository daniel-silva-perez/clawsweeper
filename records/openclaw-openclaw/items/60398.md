## Summary

Keep open. Current main still writes and bootstraps the macOS Gateway LaunchAgent from the resolved HOME under `~/Library/LaunchAgents`, with no boot-volume fallback for homes on external APFS volumes. The generic `launchctl bootstrap` failure path would still surface error 5 as a raw bootstrap failure, so the reporter’s external-volume case remains actionable.

## Best Possible Solution

Implement a shared macOS LaunchAgent plist location resolver for the Gateway service that detects when the resolved HOME/LaunchAgents directory is not launchd-bootstrappable and stages the managed plist on a boot-volume per-user LaunchAgents path, then use it consistently in install, restart/handoff, status/doctor inspection, docs, and the macOS app snapshot path. Preserve existing profile labels and permissions, avoid leaving `--force` users without their previously working service on bootstrap failure, add focused unit coverage for external HOME path selection, and validate manually on macOS with an external APFS home.
