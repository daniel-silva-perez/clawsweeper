## Summary

Keep open. Current main still resolves the macOS Gateway LaunchAgent plist under the resolved HOME directory and passes that same path to `launchctl bootstrap`, so the reported external-APFS-home failure remains actionable and is not superseded by a linked fix PR.

## What This Changes

The issue asks the macOS gateway installer to handle external APFS home directories by staging the LaunchAgent plist on a boot-volume per-user LaunchAgents path instead of the resolved external HOME path.

## Best Possible Solution

Add a shared macOS LaunchAgent plist-location resolver for the Gateway service that detects a non-bootstrappable external HOME LaunchAgents directory and uses a boot-volume per-user LaunchAgents path consistently across install, stage, restart handoff, repair/status/doctor inspection, uninstall cleanup, app snapshot/log discovery, and docs. Preserve existing labels, permissions, profile behavior, and avoid leaving `--force` users with a working service unloaded if the replacement bootstrap fails.

## Reproduction Assessment

Yes, we have a high-confidence code-path reproduction: set `HOME` to an external-style path such as `/Volumes/MainDataDrive/Users/name` and current main will still write and bootstrap `/Volumes/.../Library/LaunchAgents/ai.openclaw.gateway.plist`. A live external APFS macOS run would still be needed to validate the final repair against launchd itself.

## Solution Assessment

Yes, a narrow shared resolver plus consistent lifecycle use is the best maintainable direction; a docs-only workaround would leave app and doctor/restart flows broken. The fallback should be implemented as a generic macOS launchd path seam rather than hard-coding one-off behavior in the install command.
