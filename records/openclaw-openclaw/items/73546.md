## Summary

Keep open. Current main still lacks a persisted last-active TUI session resume path: a fresh launch without `--session` resolves to the configured main/global key, while history is loaded only for that selected key. The older #41168 overlaps, but this report is focused, recent, and useful as the repair candidate for the TUI continuity bug.

## What This Changes

The issue asks for `openclaw tui` reconnects or fresh relaunches to resume the previous conversation session, or visibly warn/carry context when starting a new one.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key in scoped local state, update it on successful sends and explicit session switches, use it on startup only when no explicit `--session` was supplied and the key is valid for the current agent/scope, and show a visible resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to launch TUI with no `--session`, switch or create a `tui-<uuid>` session, exit, then relaunch with no `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing session listing/history APIs plus local scoped state; changing gateway reset/archive semantics would be broader than needed for the observed continuity gap.
