## Summary

Keep this issue open. Current main still reloads history for the currently selected in-memory TUI session during a WebSocket reconnect, but a fresh `openclaw tui` process derives its session from `--session`/defaults and has no persisted last-active TUI session resume path. The reporter's `tui-<uuid>` observation is also consistent with the intentional `/new` isolation behavior, not a restart resume contract.

## Best Possible Solution

Keep open and implement an explicit TUI resume contract: persist the last active TUI session key per gateway/agent or add a clear `/resume` flow, load that key on fresh `openclaw tui` startup, warn when starting a fresh session instead, and add targeted coverage for process restart plus transient WebSocket reconnect. Coordinate the shared gateway/session semantics with related #68162 where the WebChat reconnect contract overlaps.
