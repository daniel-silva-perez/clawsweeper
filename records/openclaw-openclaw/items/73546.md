## Summary

Keep open. Current main still distinguishes in-process WebSocket reconnect recovery from fresh `openclaw tui` startup: reconnect reloads history for the already-selected in-memory key, but a new TUI process still chooses from `--session` or the configured main/global default and has no persisted last-active TUI session resume path.

## What This Changes

The issue asks for `openclaw tui` to resume the previous conversation session after disconnect/restart, or clearly warn/carry context when it starts fresh.

## Best Possible Solution

Add an explicit TUI resume contract: persist the last active TUI session key in scoped local state, use it on startup only when no explicit `--session` was supplied, visibly report resume/fallback/fresh-session creation, and keep `/new` as an intentional isolated-session command.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a fresh `runTui` with no `opts.session` resolves to main/global and `loadHistory` then reads only that selected key, so a prior non-default TUI session is not resumed by startup logic.

## Solution Assessment

Yes for the proposed direction. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing `listSessions`/history APIs and scoped local state, not in a broad gateway reset or transcript migration path.
