## Summary

Keep this issue open. Current main preserves the in-memory TUI session key during a WebSocket reconnect, but a fresh `openclaw tui` process still derives its session from CLI/default config only and does not persist or resume the last active TUI session. The existing `/new` behavior also intentionally creates `tui-<uuid>` session keys, which explains part of the reporter's observed surface but does not provide a restart/reconnect resume path.

## Best Possible Solution

Keep open and implement an explicit TUI resume contract: persist the last active TUI session key per gateway/agent or add a clear `/resume` flow, load that key on fresh `openclaw tui` startup, warn when a new session is selected instead, and add targeted regression coverage for process restart plus transient WebSocket reconnect. Coordinate with the open WebChat reconnect work where the gateway/session contract overlaps.
