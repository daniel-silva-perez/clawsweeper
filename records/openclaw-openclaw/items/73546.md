## Summary

Keep open. Current main still has a real TUI session-continuity gap: it can recover an already-selected in-process WebSocket reconnect, but a fresh `openclaw tui` startup has no persisted last-active TUI session selection and falls back to the explicit `--session` value or the configured default. The issue is recent, source-backed, and narrow enough for a focused repair PR.

## What This Changes

The issue requests that TUI reconnect or restart resume the previous conversation session, or visibly warn when it starts a fresh session instead of silently resetting conversation continuity.

## Best Possible Solution

Add an explicit TUI resume contract: persist the last active TUI session key in scoped local state, use it on startup only when no explicit `--session` was supplied, visibly report resume/fallback/fresh-session creation, and keep `/new` as an intentional isolated-session command.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a fresh `runTui` with empty `opts.session` resolves to main/global, and `loadHistory` reads only that selected key, so a prior `agent:<id>:tui-<uuid>` session is not resumed by startup logic.

## Solution Assessment

Yes, the narrow maintainable direction is to fix the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state. A gateway transcript migration or automatic `.reset` recovery would be broader and would risk changing intentional reset semantics.
