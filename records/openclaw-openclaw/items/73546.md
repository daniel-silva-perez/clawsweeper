## Summary

Keep this open. Current main covers in-process TUI WebSocket reconnect by reloading history for the existing in-memory session key, but a fresh `openclaw tui` process still derives its initial session only from `--session`/defaults and does not persist or resume the last active TUI session. The observed `tui-<uuid>` keys are also intentional `/new` isolation behavior, not a restart-resume mechanism.

## What This Changes

The issue requests that TUI reconnect or restart resume the previous conversation session, or clearly warn/provide a resume path when starting a fresh session.

## Best Possible Solution

Define an explicit TUI resume contract: remember the last active TUI session key per relevant gateway/agent scope, use it on fresh `openclaw tui` startup when no explicit session was provided, make fallback/new-session behavior visible, and keep the current `/new` unique-session isolation intact.
