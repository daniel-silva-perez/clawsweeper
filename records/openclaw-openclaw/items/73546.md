## Summary

Keep open. Current main still separates in-process WebSocket reconnect recovery from fresh TUI process startup: reconnect reloads history for the already-selected key, but `openclaw tui` without `--session` still selects the configured main/global key and has no persisted last-active TUI session resume path. The issue is recent, source-backed, not protected, and no open PR directly references it.

## What This Changes

The issue requests that TUI reconnects or relaunches resume the previous conversation session, or clearly warn/carry context when a fresh session is selected.

## Best Possible Solution

Add an explicit TUI resume contract: persist the last active TUI session key in scoped local state, restore it on startup only when no explicit `--session` was supplied and the key is valid for the current gateway/agent/scope, display resumed/fallback/fresh-session status, and leave `/new` as the intentional isolated-session command.

## Reproduction Assessment

Yes for the narrowed current-main bug: source inspection shows a fresh `runTui` without `opts.session` defaults to main/global, reapplies that startup input on agents refresh, and loads history only for that key. The exact reset-archive symptom is not fully reproduced without live logs, but the session-continuity gap is high-confidence.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing `listSessions` and history APIs plus scoped local state; reading `.reset` archives or changing generic gateway reset behavior would be broader and riskier unless a focused regression proves that is also required.
