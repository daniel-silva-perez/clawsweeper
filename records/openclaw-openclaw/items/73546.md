## Summary

Keep open. Current main handles in-process WebSocket reconnect by refreshing agents and reloading history for the already-selected session, but a fresh TUI process still derives the session only from `--session` or defaults and has no persisted last-active TUI session resume path.

## What This Changes

The issue requests that `openclaw tui` resume the previous TUI conversation session after reconnect/restart, or clearly warn when it starts a fresh session.

## Best Possible Solution

Add a TUI resume contract that records the last active TUI session key in a scoped local state record, uses it on startup only when no explicit `--session` was supplied, visibly reports whether it resumed or started fresh, and keeps `/new` as an intentional isolated-session action.

Add a narrow resume contract for `openclaw tui`: persist the last active TUI session key in a scoped local state record when the user switches sessions, sends/loads a selected session, or uses `/new`; on fresh startup, use that persisted key only when no explicit `--session` was supplied. Preserve explicit `--session` behavior, global-session scope behavior, workspace-based agent selection, and `/new` isolation (`tui-<uuid>` must remain an intentional new isolated session).
