## Summary

Keep this open. Current main still has no persisted last-active TUI session resume path for a fresh `openclaw tui` launch, so the reported conversation reset remains a valid, focused TUI session-continuity bug.

## What This Changes

The issue requests that the TUI resume the previous conversation after reconnect or restart, or visibly warn/carry context when it must start a fresh session.

## Best Possible Solution

Add an explicit TUI resume contract: persist the last active TUI session key in scoped local state, use it on startup only when no explicit `--session` was supplied and the key is still valid for the current agent/scope, show a visible resumed/fallback/fresh-session status, and keep `/new` as intentional isolated-session creation.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: launch TUI with no `--session`, create or switch to a `tui-<uuid>` session, exit, relaunch with no `--session`, and current startup resolves back to main/global before loading history for that selected key.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; gateway reset archive migration would be a broader and less direct fix.
