## Summary

Keep open. Current main still lacks a persisted last-active TUI session resume path for a fresh `openclaw tui` launch, so restarting without `--session` still reselects main/global rather than the prior `tui-<uuid>` conversation.

## What This Changes

The issue requests that TUI reconnects or restarts resume the previous conversation, or visibly warn/carry context when a fresh session is unavoidable.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key in scoped local state, update it on session selection/switching, use it on startup only when no explicit `--session` was supplied and the key is valid for the current agent/scope, and show a visible resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: start TUI with no `--session`, create or switch to a `tui-<uuid>` session, exit, relaunch with no `--session`, and observe current startup resolving back to main/global before loading history for that selected key.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; changing gateway reset/archive semantics would be broader than needed unless a regression proves the TUI boundary is insufficient.
