## Summary

Keep open. Current main still distinguishes in-process WebSocket reconnect recovery from a fresh `openclaw tui` process: reconnect reloads history for the already-selected session key, but startup without `--session` still resolves to the configured main/global key and has no persisted last-active TUI session resume path.

## What This Changes

The issue requests that `openclaw tui` resume the prior TUI session after reconnect or restart, or visibly warn/carry context instead of silently starting with a fresh or different session.

## Best Possible Solution

Add an explicit TUI resume contract: store the last active TUI session key in scoped local state, use it on startup only when no explicit `--session` was supplied and the key is still valid for the current agent/scope, show a visible resumed/fallback/fresh-session status, and keep `/new` as an intentional isolated-session command.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a fresh `runTui` with no `opts.session` resolves empty input to main/global, `applyAgentsResult` reapplies that startup input, and `loadHistory` fetches only the selected key.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing `listSessions` and history APIs plus scoped local state, not in gateway reset handling or reset-archive transcript migration.
