## Summary

Keep open. Current main still does not provide equivalent startup diagnostics for exec/sandbox/elevated configuration drift, but this PR is not safe to merge as-is because it can block gateway startup, emits at least one warning from raw rather than effective runtime config, misses per-agent effective settings, and includes incorrect security guidance.

## What This Changes

This PR adds a config-conflict detector with tests and calls it during gateway startup to log warnings or errors for exec, sandbox, elevated, and exposed no-auth gateway combinations.

## Best Possible Solution

Keep the feature direction, but refresh the implementation so startup diagnostics run after effective runtime config is known, stay non-throwing, evaluate each agent's resolved sandbox/exec/elevated posture, reuse security-audit semantics where possible, and replace the exec.security guidance with wording that matches the documented deny/allowlist/full model.

## Reproduction Assessment

Yes. The blocking issues have high-confidence static reproduction paths: the PR branch calls the throwing sandbox resolver from startup, logs before runtime bind/auth overrides are applied, and reads global/default settings while current runtime code resolves exec and sandbox per agent.

## Solution Assessment

No. The current PR is a useful sketch, but the safest maintainable solution is a refreshed detector built around effective runtime/per-agent state and existing audit semantics rather than raw config checks at the early auth-bootstrap point.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep warning-only diagnostics from throwing at startup:** `src/config/detect-conflicts.ts:24`
  - body: `detectConfigConflicts` runs during gateway startup and calls `resolveSandboxConfigForAgent(config)`, which normalizes SSH sandbox settings and throws for a non-absolute `sandbox.ssh.workspaceRoot`. That makes a warning-only feature capable of preventing the gateway from booting for existing configs.
  - confidence: 0.91
- **[P2] Use effective runtime bind for no-auth exposure checks:** `src/gateway/server.impl.ts:502`
  - body: `logConfigConflicts(cfgAtStart)` runs before `resolveGatewayRuntimeConfig` applies CLI/runtime bind, host, auth, and tailscale overrides. A safe invocation such as overriding a file `gateway.bind="lan"` to loopback can still emit a critical unauthenticated exposure warning from stale raw config.
  - confidence: 0.88
- **[P2] Evaluate conflicts for agent overrides:** `src/config/detect-conflicts.ts:24-29`
  - body: The detector reads only the default sandbox mode and global `config.tools.*` values, but runtime exec and sandbox policy resolve `agents.list[]` overrides per agent. This misses conflicts where defaults are safe but a configured agent has active sandbox plus `tools.exec.ask="off"`, `host="gateway"`, or elevated overrides.
  - confidence: 0.87
- **[P2] Remove the insecure exec.security=full recommendation:** `src/config/detect-conflicts.ts:77-79`
  - body: The info message says `exec.security=full` provides maximum restriction, but OpenClaw documents `full` as allowing everything/equivalent to elevated, while `deny` and `allowlist` are stricter. This guidance can lead operators to weaken their exec posture.
  - confidence: 0.93
