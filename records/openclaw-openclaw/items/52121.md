## Summary

Keep this PR open. Current main still does not implement the PR's intended startup diagnostics for conflicting exec/sandbox/elevated configuration. Main has related docs, `openclaw security audit` checks, runtime exec errors, and hard gateway-auth refusal for exposed unauthenticated binds, but there is no equivalent `detectConfigConflicts` module or gateway startup warning path. The PR is still potentially useful, though it needs a maintainer refresh before merge because the proposed implementation can miss effective per-agent config, use raw pre-runtime bind/auth config, and introduce misleading security guidance.

## Best Possible Solution

Keep this PR open for maintainer review or a refresh. The best implementation would be a non-throwing startup diagnostic that runs after effective runtime config is known, reuses existing security-audit semantics where possible, evaluates per-agent exec/elevated/sandbox combinations, treats the already-enforced no-auth bind case consistently, fixes the `exec.security` guidance, and adds regression coverage for per-agent overrides, CLI/runtime bind overrides, SSH sandbox config, and the original channel-session debugging path.

## Risks / Open Questions

- Merging the PR as-is could produce false negatives because the proposed detector reads global/default config and does not evaluate per-agent effective exec/elevated/sandbox overrides.
- Merging the PR as-is could produce high-severity false positives because it logs from raw config before CLI/runtime bind and auth overrides are applied.
- The proposed warning path calls sandbox resolution during startup; current sandbox SSH normalization can throw on invalid `workspaceRoot`, so a warning-only feature could become a boot blocker for existing configs.
- The PR's near-high-safety recommendation points users toward `exec.security=full`, while current docs and audit code treat `full` as broad trust, not maximum restriction.
