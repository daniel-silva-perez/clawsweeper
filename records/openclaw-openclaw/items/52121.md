## Summary

Keep this PR open. Current main still does not implement equivalent startup diagnostics for conflicting exec/sandbox/elevated configuration. Main has docs, `openclaw security audit` findings, runtime exec routing errors, and a hard refusal for exposed unauthenticated gateway binds, but no non-throwing gateway startup conflict detector for the PR’s central behavior. The PR is still potentially useful, though it needs a maintainer refresh before merge because the proposed implementation has unresolved correctness and security/operational risks.

## Best Possible Solution

Keep this PR open for maintainer review or a refreshed implementation. The best path is a non-throwing startup diagnostic that runs after effective runtime config is known, reuses existing security-audit semantics where possible, evaluates per-agent exec/elevated/sandbox combinations, treats the already-enforced no-auth bind case consistently, fixes the `exec.security` guidance, and adds regression coverage for per-agent overrides, CLI/runtime bind overrides, SSH sandbox config, and the original channel-session debugging path.

## Risks / Open Questions

- Merging the PR as-is could miss real conflicts when the effective configuration comes from `agents.list[].tools.exec`, `agents.list[].tools.elevated`, or `agents.list[].sandbox` overrides instead of global/default config.
- Merging the PR as-is could produce high-severity false positives by warning from raw file config before CLI/runtime bind and auth overrides are applied.
- The PR’s warning-only startup path calls `resolveSandboxConfigForAgent`; current SSH sandbox normalization throws for a non-absolute `workspaceRoot`, so diagnostics could become a boot blocker for existing configs.
- The PR’s `exec.security=full` guidance conflicts with current docs/audit semantics, where `full` is broad trust and `deny`/`allowlist` are stricter postures.
