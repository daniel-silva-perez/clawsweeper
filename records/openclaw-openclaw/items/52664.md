## Summary

Keep open. Current main does not expose `rawBody` on the targeted plugin hook events, so the PR is still useful, but the proposed implementation needs one correctness fix before merge because it forwards `rawBody` for heartbeat/cron-style system-event runs even though the new hook contract documents those sources as `undefined`.

## What This Changes

The PR adds optional `rawBody` to `before_prompt_build`, legacy `before_agent_start`, and `agent_end` plugin hook events, threading it through immediate, deferred, and collect follow-up reply runs.

## Best Possible Solution

Land a rebased version that exposes `rawBody` only for user-originated channel runs across immediate, follow-up, and collect paths, keeps non-channel sources undefined, documents the hook field for plugin authors, and adds narrow tests for channel raw text, collect batching, and heartbeat/cron omission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Omit rawBody for system-event runs:** `src/auto-reply/reply/get-reply-run.ts:403-404`
  - body: This computes the new hook value for every `runPreparedReply` call and then forwards it into the hook pipeline. Heartbeat and cron wakeups also enter this path with `CommandBody`/`RawBody` populated, while the new event contract says `rawBody` is undefined for heartbeat, cron, CLI, and other non-channel sources. Gate the value on a user/channel-originated run before storing or forwarding it so plugins do not receive scheduler prompts as user input.
  - confidence: 0.86
