## Summary

Keep this PR open. Current main still does not expose `rawBody` on the relevant plugin hook events, so the contribution is not obsolete, but the latest diff still has a blocking contract bug: it computes and forwards `rawBody` for all reply runs even though the PR promises it is undefined for CLI, heartbeat, cron, and other non-channel sources. The earlier queued follow-up and collect-mode concerns appear to have been addressed in the PR discussion; the remaining issue is the system-event boundary.

## What This Changes

The PR adds optional `rawBody` to `before_prompt_build`, legacy `before_agent_start`, and `agent_end` hook event types and threads it through reply, embedded-runner, follow-up, and collect queue paths with generated SDK baseline hash updates.

## Best Possible Solution

Land a rebased version that exposes `rawBody` only for user-originated channel runs across immediate, deferred follow-up, and collect paths, keeps CLI/heartbeat/cron/system-event runs undefined, documents the hook field for plugin authors, and adds focused tests for channel raw text plus non-channel omission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Omit rawBody for system-event runs:** `src/auto-reply/reply/get-reply-run.ts:403-404`
  - body: The new hook contract says `rawBody` is undefined for CLI, heartbeat, cron, and other non-channel sources, but these lines compute it from `CommandBody`/`RawBody`/`Body` for every run and later forward it into the hook pipeline. Heartbeat and cron callers already populate those fields, so plugins would receive scheduler prompts as user raw text. Gate this value on a user/channel-originated run before storing or forwarding it.
