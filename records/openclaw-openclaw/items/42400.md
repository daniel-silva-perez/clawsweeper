## Summary

Keep open. Current main still does not implement a `neverReply` config or equivalent no-LLM group suppression path, and the linked plugin approach is relevant but not currently a complete replacement because unbound `inbound_claim` hooks are not broadcast from dispatch. The PR is useful but not merge-ready: several changed channel handlers still claim pending-history accumulation while returning before any history write/read path, so the patch needs author fixes plus maintainer architecture review.

## What This Changes

The PR adds a channel/account/default `neverReply` config, docs/schema/SDK updates, tests, and per-channel group-message gates intended to suppress LLM calls and outbound replies while retaining passive history where supported.

## Best Possible Solution

Keep the PR open until maintainers choose the architecture. If core `neverReply` is accepted, tighten it behind a shared resolver contract, align docs to the exact channels that retain history, and add representative channel tests proving no LLM call, no outbound reply, DMs unaffected, and pending-history write/read/clear behavior. If the plugin path is preferred, first land the generic hook-broadcast seam and move listen-only behavior into a plugin such as `flowolforg/openclaw-listen-only`.

## Reproduction Assessment

Yes. Static diff inspection gives a high-confidence reproduction for the PR regressions: configure `neverReply` on Discord, LINE, Matrix, or Mattermost group traffic and those handlers return before writing pending history, while main has no `neverReply` behavior to compare against. I did not run tests because this was a read-only review and repo tests may write caches/artifacts.

## Solution Assessment

No, not as currently written. The product direction may be valid, but this implementation repeats per-channel gates and already drifts across history semantics; the safer solution is either a carefully tested core helper pattern or a maintainer-approved plugin hook seam with the behavior implemented outside core.
