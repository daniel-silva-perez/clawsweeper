## Summary

Keep open. Current main still has the Windows restart false-cleanup risk this PR targets: loose listener classification still treats any command containing "openclaw" as a gateway, and the daemon restart path still enables unknown-listener fallback before terminating `staleGatewayPids`. The PR is useful source work, but it is stale and currently unmergeable, so it needs a rebase/reconciliation rather than closure or direct merge.

## What This Changes

The PR tightens Windows gateway restart handling by removing unknown-listener stale cleanup from the service-managed restart path, narrowing gateway listener classification, adding restart diagnostics and PID identity warnings, and updating related CLI tests.

## Best Possible Solution

Rebase or replace the PR on current main, keep the stricter listener classification, preserve the current restart-health contract, and ensure every restart cleanup caller only terminates positively identified gateway PIDs while still surfacing unknown listeners as diagnostics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: Windows daemon restart enables unknown-listener fallback, `inspectGatewayRestart()` includes unknown listener PIDs in `staleGatewayPids`, and lifecycle cleanup terminates that list; the existing restart-health tests also encode unknown listeners becoming stale when the fallback is enabled.

## Solution Assessment

No, not as currently submitted. The direction is right, but the branch is stale/unmergeable and should be reconciled with current main's version/plugin/channel restart checks; the safer implementation is to layer the Windows safety change onto the current restart snapshot contract and cover daemon plus update restart cleanup paths.
