## Summary

Keep this PR open. Current main still blocks the safe cache-trace patch reported in #72883, and the proposed diff is a narrow guard change with focused regression coverage; I did not find a blocking correctness or security issue in the patch.

## What This Changes

The PR lets the agent-facing gateway config mutation guard allow `diagnostics.cacheTrace.enabled` and content-capture flag changes only when the resulting cache trace config has `includeMessages`, `includePrompt`, and `includeSystem` all set to `false`, with regression tests for safe and unsafe `config.patch` cases.

## Best Possible Solution

Land a narrow gateway-tool guard like this PR after normal maintainer review and CI, keeping raw cache-trace content fields protected while allowing operators to enable decision/cache observability with content capture disabled.

## Reproduction Assessment

Yes. Current main can be checked at source level: the changed path collector will include `diagnostics.cacheTrace.enabled` and the allowlist has no matching entry, producing the protected-path rejection reported in #72883.

## Solution Assessment

Yes. The proposed post-merge `nextConfig` guard is the narrowest maintainable fix because it allows only the safe cache-trace fields and only when the resulting config disables all raw content capture fields, instead of broadly allowlisting the diagnostics subtree.
