## Summary

Keep open. Current main still persists `lastGood` but explicitly does not use it for auth-profile ordering, and both session override selection and embedded-run profile rotation consume that precomputed round-robin order. The shared timeout-budget symptom appears partly mitigated now because each embedded attempt creates its own abort timer, and slug generation now honors the general agent timeout, but the central lastGood/failover-ordering request and a slug-specific model override remain unresolved.

## What This Changes

Request auth-profile failover to prefer the last successful profile after rate-limit/auth failures, avoid timeout-budget cascade failures across profile rotations, and allow lightweight embedded slug generation to use a cheaper model override.

## Best Possible Solution

Add failover-aware auth-profile selection that can prefer the most recently successful usable profile after rate-limit/auth/billing failures while preserving normal round-robin behavior for routine scheduling. Cover resolveAuthProfileOrder, resolveSessionAuthProfileOverride, and embedded-runner rotation with focused tests, update the failover docs, and split the slug-generator model override/cooldown-isolation work into a narrower follow-up if maintainers want it handled separately.

## Reproduction Assessment

Yes for the main remaining ordering bug: source inspection gives a high-confidence reproduction path by setting multiple profiles, recording lastGood for the backup, and observing resolveAuthProfileOrder/session/embedded rotation continue through lastUsed round-robin order. The original shared-attempt timeout cascade is not reproduced from current code because each runEmbeddedAttempt now schedules its own attempt timer.

## Solution Assessment

No; current main is only a partial mitigation. The maintainable fix is not to replace round-robin globally, but to introduce explicit failover-aware lastGood/MRU selection for failure recovery and keep normal scheduling semantics intact.
