## Summary

Keep this issue open. Current main still has the specific `update.run` continuation producer gap described in the maintainer discussion: the restart-sentinel consumer can enqueue and drain continuations when present, and the restart tool can create them, but `update.run` still neither accepts `continuationMessage` nor writes `payload.continuation`. The latest shipped release has the same gap, and the linked #71573 implementation PR is closed unmerged rather than proof of a fix.

## Best Possible Solution

Keep this issue open and land a narrow additive `update.run` continuation fix: accept optional `continuationMessage` in the update protocol, parse/forward it through the agent gateway tool, write `buildRestartSuccessContinuation({ sessionKey, continuationMessage })` into successful update restart sentinels only when a restart is actually scheduled, and update the tests that currently assert no continuation. After that lands, maintainers can decide whether graceful drain and post-reset bridge behavior stay here or move to narrower follow-ups such as #73918.
