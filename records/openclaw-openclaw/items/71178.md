## Summary

Keep open. Current main and the latest release still lack the narrow `update.run` restart-continuation producer needed for an agent-initiated update to resume a turn after the gateway restarts, and the open PR #74362 is the active implementation candidate rather than proof that the issue is fixed.

## What This Changes

The issue reports that running `openclaw update` during an active Telegram or Discord-backed agent turn can restart the gateway and lose buffered assistant output without a continuation or bridge message.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that adds `continuationMessage` to the `update.run` protocol, forwards it from the agent gateway tool, and persists `buildRestartSuccessContinuation({ sessionKey, continuationMessage })` on successful update restart sentinels; then split or explicitly track any remaining drain/reset recovery behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main rejects `continuationMessage` for `update.run`, writes update restart sentinels without `continuation`, and has tests asserting that absence; a live Telegram or Discord restart was not run in this read-only review.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract is the smallest maintainable fix; closing now would be premature because current main and `v2026.4.27` still omit the producer field.
