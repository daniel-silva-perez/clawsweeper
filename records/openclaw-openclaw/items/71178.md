## Summary

Keep open: current main and the latest shipped release still have the `update.run` continuation producer gap, and the open implementation PR has not merged. Existing restart-continuation consumers are useful groundwork but do not make `update.run` persist a continuation today.

## What This Changes

The issue reports that running `openclaw update` during an in-flight Telegram or Discord agent turn restarts the gateway and can drop buffered assistant output without a continuation, drain, or bridge recovery.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix: add `continuationMessage` to the `update.run` protocol, parse and forward it from the agent gateway tool, write `buildRestartSuccessContinuation({ sessionKey, continuationMessage })` into successful update restart sentinels, and update the tests that currently assert no continuation. Then decide whether remaining drain/retry/reset recovery belongs here or in narrower follow-ups such as #73918 and #74424.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, invoking `update.run` with a session key writes a restart sentinel with no `continuation`, and the existing tests explicitly assert that absence; a live channel restart was not run in this read-only review.

## Solution Assessment

Yes for the narrow producer gap. The maintainable fix is to reuse the existing restart-continuation contract for `update.run`; broader startup retry or graceful drain behavior is a separate product/reliability follow-up rather than a substitute for this missing field path.
