## Summary

Keep open. Current main still has the narrow `update.run` continuation producer gap: restart continuations are consumed when present, but `update.run` neither accepts `continuationMessage` nor persists a `continuation`, and the tests still assert that absence.

## What This Changes

The issue requests reliable delivery or recovery when `openclaw update` restarts the gateway while an agent turn is still in progress in Telegram or similar messaging channels.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that threads `continuationMessage` through the update protocol, agent gateway tool, and successful update restart sentinel. Then decide whether graceful drain, post-reset bridge behavior, and related coalescing behavior such as #74424 belong in separate follow-ups.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The source-level reproduction is high confidence: on current main, an `update.run` request with a session key writes a restart sentinel without `continuation`, and existing tests assert that absence; a live Telegram/Discord update was not run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge handling are adjacent recovery work, not substitutes for the missing producer field.
