## Summary

Keep open. Current main still has the narrow `update.run` continuation producer gap: restart continuations are consumed when present, but `update.run` neither accepts nor persists one, and a related open PR is the implementation candidate rather than proof of a merged fix.

## What This Changes

The issue asks OpenClaw to avoid losing messaging-channel replies when an `openclaw update` restart interrupts an in-flight agent turn.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix: add `continuationMessage` to the `update.run` protocol, forward it from the agent gateway tool, persist `buildRestartSuccessContinuation({ sessionKey, continuationMessage })` on successful update restart sentinels, and then explicitly split or close the remaining drain/reset recovery scope.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, invoking `update.run` with a session key writes a restart sentinel without `continuation`, and existing tests assert that absence; a live channel restart was not run in this read-only review.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge behavior are adjacent reliability follow-ups, not substitutes for the missing producer field.
