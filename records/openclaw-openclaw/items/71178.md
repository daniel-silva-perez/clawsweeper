## Summary

Keep open. Current main still has the narrow `update.run` continuation producer gap: restart continuations are consumed when present, but `update.run` does not accept or persist one, and the latest shipped release has the same behavior. The open PR #74362 is the right implementation candidate to review rather than closing this issue early.

## What This Changes

The issue asks OpenClaw to avoid losing in-flight Telegram/Discord assistant replies when `openclaw update` restarts the gateway during an active agent turn.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix: add `continuationMessage` to the `update.run` protocol, parse and forward it from the agent gateway tool, persist `buildRestartSuccessContinuation({ sessionKey, continuationMessage })` on successful update restart sentinels, and update the tests that currently assert no continuation. Then split or explicitly scope the remaining drain/reset recovery behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, invoking `update.run` with a session key writes a restart sentinel without `continuation`, and existing tests assert that absence; a live channel restart was not run in this read-only review.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge behavior are adjacent reliability follow-ups, not substitutes for the missing producer field.
