## Summary

Keep open. Current main still lacks the `update.run` continuation producer: restart continuations are consumed when present, but `update.run` does not accept `continuationMessage`, does not persist `payload.continuation`, and tests still assert that absence. Open PR #74362 is the active implementation candidate, so this issue should remain open until that or an equivalent fix lands.

## What This Changes

Issue requests preventing Telegram/Discord message loss when `openclaw update` restarts the gateway while an agent turn is still in progress.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that threads `continuationMessage` through the update protocol, agent gateway tool, and successful update restart sentinel. Then split any remaining graceful-drain, post-reset bridge, or coalesced-restart recovery work into focused follow-ups.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, an `update.run` request with a session key writes a restart sentinel without `continuation`, and the existing tests assert that missing field.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge handling are adjacent recovery work, not substitutes for the missing producer field.
