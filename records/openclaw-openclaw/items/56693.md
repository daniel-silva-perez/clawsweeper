## Summary

Keep this open. Current main still delegates OpenAI Codex OAuth account selection to the upstream helper, stores the returned OAuth credential without active-workspace validation, and does not classify the reported `deactivated_workspace` marker on the failover path. The related open PR #55893 is useful but partial and not merged; its own scope says it only adds error classification, not OAuth login, auth storage, fallback ordering, or network recovery.

## What This Changes

The issue requests that OpenAI Codex OAuth avoid or recover from binding to a deactivated ChatGPT workspace when an account also has an active workspace.

## Best Possible Solution

Implement provider-owned resilience in layers: land or replace the narrow `deactivated_workspace` classification coverage, preserve a valid `accountId` when OAuth returns one, and add post-login or one-shot runtime recovery only if the upstream Codex OAuth/API surface can reliably identify active workspaces. If that surface is missing, coordinate the account-selection fix upstream and add clear OpenClaw remediation/doctor guidance meanwhile.
