## Summary

Keep this PR open. Current main still lacks the proposed stale-age expiry and direct reply dedupe behavior, but the branch is not merge-ready because its outbound coalescing crosses caller/recovery boundaries, its recovery path is stale against current claim/re-read safeguards, and the diff includes unrelated Control UI formatting churn.

## What This Changes

The PR branch adds 10-minute stale queued-outbound expiry during recovery, a 1.5-second direct reply/mirror outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting changes.

## Best Possible Solution

Prepare a narrow outbound-only replacement: add age-based recovery expiry after claiming and re-reading the current queue entry, preserve all stored delivery metadata, keep reconnect/live claim protections intact, and split or drop direct-reply dedupe unless its per-call security and callback semantics are fully specified.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by per-call delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits session, gatewayClientScopes, abortSignal, and callback identity. A cache or inflight hit can return another request's result without running delivery for the second caller, skipping session-scoped media roots, authorization-sensitive adapter behavior, hooks, and callbacks.
  - confidence: 0.9
- **[P1] Bypass direct dedupe for recovery replays:** `src/infra/outbound/deliver.ts:644`
  - body: Recovery replays call deliverOutboundPayloads with skipQueue: true for each stored entry. Since the key does not include the queued entry id, two legitimate queued reply-shaped entries with the same target and payload can coalesce, and the second entry can be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-198`
  - body: The recovery loop expires entries from the startup snapshot directly. Current main re-reads each queue file under an in-process claim before replaying or failing it, so expiry must run under that same guard to avoid racing reconnect or live delivery drains.
  - confidence: 0.87
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main's recovery replay passes stored replyToMode and session through to delivery. This branch's buildRecoveryDeliverParams omits both, so recovered messages can route, hook, or scope media differently from the original send.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces direct outbound coalescing and stale recovery movement in security-sensitive delivery paths without preserving every caller/session boundary.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:375`
  - body: The direct outbound cache key omits gatewayClientScopes and session while delivery passes both into adapters, media scoping, and hooks. A hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:644`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path with no queue entry identity in the key. Duplicate queued records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks the current ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the startup snapshot unless it is rebased into the current claim and re-read flow, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84
