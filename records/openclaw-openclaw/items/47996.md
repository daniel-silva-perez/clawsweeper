## Summary

Keep open. The requested outbound reliability work is not implemented on current main, but this PR head is stale and should not merge as-is: it mixes unrelated UI formatting churn with low-level delivery semantics and introduces dedupe paths that can collapse distinct sends or queued recovery replays.

## What This Changes

The PR branch adds 10-minute stale queued-outbound expiry during recovery, a 1.5s direct reply/mirror outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting changes.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds age-based recovery expiry after re-reading and claiming pending entries, preserves all current queued metadata and reconnect-drain protections, and splits any direct-reply dedupe into a separate design only if it can preserve per-call delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by per-call delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key includes target/payload/mirror fields but not `session`, `gatewayClientScopes`, `abortSignal`, or per-call callbacks. A cache or inflight hit can return another request's result without running `deliverOutboundPayloadsCore`, skipping session-scoped media roots, hooks, authorization-sensitive side effects, and callback delivery for a distinct send.
  - confidence: 0.88
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:386-418`
  - body: Recovery replays call `deliverOutboundPayloads` with `skipQueue: true` for each stored entry. Since the key only records `skipQueue` and not the queued entry identity, two legitimate queued reply-shaped entries with the same target and payload can coalesce inside the 1.5s window, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:176-199`
  - body: The recovery loop acts on the startup snapshot directly; current main re-reads each queue file under an in-process claim before replaying or failing it. Without that guard, startup recovery can race reconnect/live delivery drains and move or ack an entry after another path has taken ownership.
  - confidence: 0.86
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main's recovery replay passes stored `replyToMode` and `session` through to delivery. This branch's `buildRecoveryDeliverParams` omits both, so recovered messages can route, hook, or scope media differently from the original send.
  - confidence: 0.84
