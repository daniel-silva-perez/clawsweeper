## Summary

Keep open. Current main still does not implement the PR's requested stale-age expiry or direct outbound dedupe, and the latest PR review context matches local source checks showing merge-blocking outbound recovery and coalescing regressions.

## What This Changes

The PR branch adds 10-minute stale queued-outbound expiry during recovery, a 1.5-second direct reply/mirror outbound dedupe cache, outbound tests, one channel setup type loosening, and broad Control UI render formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only revision or replacement: add age-based recovery expiry after claiming and re-reading the current queue entry, preserve all queued metadata and reconnect-drain protections, and split direct-reply dedupe into a separate design unless it can preserve per-call delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by caller context:** `src/infra/outbound/deliver.ts:368-411`
  - body: The dedupe key omits `session`, `gatewayClientScopes`, `abortSignal`, and per-call callbacks, but cache or inflight hits return before core delivery runs. A distinct caller can therefore skip its own media scoping, hooks, abort handling, adapter scope behavior, and callback delivery.
  - confidence: 0.9
- **[P1] Skip direct dedupe for recovery replays:** `src/infra/outbound/deliver.ts:641-648`
  - body: Recovery replays call delivery with `skipQueue: true`, and the new wrapper still enters direct dedupe without queue entry identity in the key. Two legitimate queued reply-shaped entries with the same target and payload can coalesce, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Expire only after claiming the current entry:** `src/infra/outbound/delivery-queue-recovery.ts:190-198`
  - body: The stale-expiry check moves entries based on the startup snapshot. Current main claims and re-reads each queue file before retry or failure decisions; expiry needs the same guard or it can race reconnect/live drains that already own or removed the entry.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive outbound coalescing and recovery movement without preserving all caller/session and queue ownership boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:368`
  - body: The direct outbound cache key omits `gatewayClientScopes` and `session` while delivery passes both into adapters, media access scoping, and hooks. A hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:641`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path with no queue entry identity in the key, so duplicate stored records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks current ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:190`
  - body: The stale-expiry path moves entries based on the initial recovery snapshot rather than the claimed and re-read queue entry, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84
