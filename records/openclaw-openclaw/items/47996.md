## Summary

Keep this PR open, but do not merge it as-is. Current main does not contain the proposed stale-expiry or direct-dedupe feature, so the PR is not obsolete; however, the branch is stale against current outbound recovery semantics and introduces concrete delivery, recovery, and security-sensitive coalescing regressions.

## What This Changes

The PR branch adds 10-minute queued-outbound stale expiry during recovery, a 1.5-second direct reply/mirror outbound dedupe cache, outbound tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Prepare a narrow outbound-only replacement or revision: add age-based recovery expiry only after claiming and re-reading the current queue entry, preserve all queued metadata and reconnect-drain protections, and split direct-reply dedupe into a separate design unless it can preserve per-call security and callback semantics.

## Review Findings

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe to the full delivery context:** `src/infra/outbound/deliver.ts:368-397`
  - body: The dedupe key omits `session`, `gatewayClientScopes`, `abortSignal`, and per-call callbacks, but cache or inflight hits return before `deliverOutboundPayloadsCore` runs. A distinct caller can therefore skip its own media scoping, authorization-sensitive adapter behavior, hooks, abort handling, and callback delivery.
  - confidence: 0.9
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:641-648`
  - body: Recovery calls `deliverOutboundPayloads` with `skipQueue: true`, and this wrapper still enters the new direct dedupe path. Because the key does not include the queued entry id, two legitimate queued reply-shaped records with the same payload can coalesce and the second entry can be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:193-198`
  - body: The stale-expiry check moves entries from the startup snapshot directly. Current main claims and re-reads each queue file before deciding to replay or fail it; expiry needs the same guard or it can race reconnect/live drains that already own or removed the entry.
  - confidence: 0.87
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-80`
  - body: `buildRecoveryDeliverParams` omits stored `replyToMode` and `session`, while current main replays both. Recovered messages can route replies, hooks, and media scoping differently from the original queued send.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive outbound coalescing and stale recovery movement without preserving all caller/session and queue ownership boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:368`
  - body: The direct outbound cache key omits `gatewayClientScopes` and `session` while delivery passes both into adapters, media access scoping, and hooks. A hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:641`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path with no queue entry identity in the key, so duplicate stored records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks current ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the initial recovery snapshot rather than the claimed and re-read queue entry, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84

## Risks / Open Questions

- Direct outbound dedupe can cross session, gateway scope, abort, and callback boundaries, causing the second caller to reuse another send result without executing its own delivery path.
- Queued recovery replays can be coalesced by the new dedupe path, allowing distinct stored entries with identical reply payloads to be acknowledged without a real second send.
- The stale-expiry implementation is based on startup snapshots instead of the current claimed queue entry, risking races with reconnect drains or live sends.
- The PR branch is stale relative to current main's queued metadata replay and active-claim protections, so a clean merge would need careful conflict resolution rather than mechanical acceptance.
