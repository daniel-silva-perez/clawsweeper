## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested stale-age recovery expiry and direct-reply dedupe behavior, but the PR branch regresses current outbound recovery ownership, queued replay metadata, and caller-scoped delivery semantics; prior ClawSweeper review comments in the discussion identified the same security-sensitive blockers.

## What This Changes

The PR adds 10-minute stale queued-outbound expiry during recovery, a 1.5-second direct reply/mirror outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI render formatting churn.

## Best Possible Solution

Prepare a narrowed outbound-only replacement or revision: add age-based recovery expiry only after claiming and re-reading the current queue entry, preserve all queued replay metadata, and split direct-reply dedupe into a separate design unless it can maintain per-call delivery semantics and recovery queue identity.

## Reproduction Assessment

Yes for the PR regressions: the source-level path is clear for two identical queued reply-shaped entries replayed with skipQueue hitting the PR dedupe cache, and for stale startup snapshots being moved before claim/re-read. Current main tests around reconnect drains and active claims provide the focused regression shape to extend.

## Solution Assessment

No. The stale-expiry idea is plausible, but this PR combines it with a security-sensitive direct-send dedupe cache and unrelated UI churn; the maintainable path is a smaller outbound-only change that preserves current queue ownership and replay metadata first.

## Security Review

Status: needs_attention

Summary: The PR introduces outbound coalescing and recovery movement that do not preserve all caller/session and queue ownership boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:368`
  - body: The direct outbound cache key omits gatewayClientScopes and session while delivery passes both into adapters, media access scoping, and hooks. A cache or inflight hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:641`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path without queue entry identity in the key, so duplicate stored records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the initial recovery snapshot rather than the claimed and re-read queue entry, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84
