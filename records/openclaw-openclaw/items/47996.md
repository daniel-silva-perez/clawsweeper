## Summary

Keep open. Current main does not implement the requested stale-expiry/direct-dedupe behavior, but this branch is not merge-ready because it conflicts with newer outbound recovery safeguards, introduces security-sensitive coalescing across per-call contexts, and carries unrelated UI churn.

## What This Changes

The PR branch adds 10-minute queued outbound recovery expiry, a 1.5-second direct reply/mirror outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting churn.

## Best Possible Solution

Replace or heavily narrow the branch so outbound recovery expiry is added after claim and re-read, all queued metadata remains preserved, and any direct-reply dedupe is handled separately only after its per-call security and callback semantics are fully specified; drop unrelated UI formatting churn from the outbound fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by per-call delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key omits session, gatewayClientScopes, abortSignal, and callback identity. A cache or inflight hit can return another request's result without running delivery for the second caller, skipping session-scoped media roots, authorization-sensitive adapter behavior, hooks, and callbacks.
  - confidence: 0.9
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:386-418`
  - body: Recovery replays use skipQueue: true for each stored entry. Since the dedupe key does not include the queue entry id, two legitimate queued reply-shaped entries with the same target and payload can coalesce, and the second entry can be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:190-198`
  - body: The stale-expiry check acts on the startup snapshot. Current main claims and re-reads each queue file before replaying or failing it, so expiry must run under that guard to avoid racing reconnect or live delivery drains that already own the entry.
  - confidence: 0.87
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main stores and replays replyToMode, formatting, mirror, session, and gatewayClientScopes for queued deliveries. The branch's recovery params do not preserve all of those fields, so recovered messages can route, hook, or scope media differently from the original send.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces direct outbound coalescing and stale recovery movement in security-sensitive delivery paths without preserving every caller/session boundary.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:375`
  - body: The direct outbound cache key omits gatewayClientScopes and session while current delivery passes both into adapters, media scoping, and hooks. A hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:416`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path with no queue entry identity in the key. Duplicate queued records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks the current ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the startup snapshot unless it is rebased into the current claim and re-read flow, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84
