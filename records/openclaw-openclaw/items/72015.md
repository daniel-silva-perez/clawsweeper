## Summary

Keep this issue open. Current main has partial mitigations, but the central reliability/defaults problem remains: Active Memory is still a blocking pre-reply embedded agent run, still inherits the active/session or primary model by default, still lacks a nonblocking/deadline/concurrency mode, and QMD boot can still eagerly arm multiple agents when memory search is broad or explicit.

## Best Possible Solution

Keep this open and either implement or split focused reliability work: make Active Memory opportunistic/nonblocking by default with a short pre-prompt deadline, add per-agent/session recall concurrency or in-flight dedupe/drop behavior, warn on or avoid inheriting slow primary chat models, and add QMD boot controls such as max concurrent startup updates, jitter, lazy initialization, or default-agent/active-agent-only policy. Status/doctor/docs should surface costly Active Memory and broad QMD-on-boot configurations.

## Risks / Open Questions

- Closing now would lose tracking for user-visible reply latency caused by a blocking pre-prompt Active Memory run.
- Multi-agent operators with broad default or explicit memory-search config can still arm multiple QMD managers at startup.
- The docs still recommend inherited/session models as the simplest/safest setup, which conflicts with the report's heavy-model reliability concern.
- Status/doctor visibility for expensive Active Memory and broad QMD-on-boot configurations still appears unimplemented.
