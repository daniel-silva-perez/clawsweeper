## Summary

Keep open. Current main has useful mitigations for the reported reliability path, but it still keeps Active Memory in the pre-reply blocking hook, still inherits the session or primary model when no recall model is pinned, and the issue now has an open draft PR with closing syntax, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue requests safer Active Memory and QMD defaults so memory recall does not delay normal replies or overload multi-agent gateways at startup.

## Best Possible Solution

Keep the issue open while maintainers review #73667 or replace it with a focused PR that bounds or removes pre-reply blocking recall, adds recall concurrency/drop or dedupe behavior, gives operators safer recall-model defaults or warnings, and adds explicit QMD startup controls for broad multi-agent deployments.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: enable Active Memory for an eligible chat session, use a slow or setup-heavy recall run, and the before_prompt_build hook waits on maybeResolveActiveRecall before the main reply can proceed; tests already simulate timeout and hard-deadline paths.

## Solution Assessment

Unclear. Current main is not the full fix because it keeps blocking recall and model inheritance; #73667 is the right active implementation candidate to review first because it directly references and closes this issue, but maintainers should verify whether its narrower scope is enough.
