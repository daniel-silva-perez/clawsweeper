## Summary

Keep open. Current main partially mitigates the report with a hard Active Memory timeout and narrower implicit QMD boot initialization, but the core reliability/defaults concern remains: Active Memory is still a blocking pre-reply embedded run, still inherits the active or primary model by default, has no nonblocking/deadline or recall concurrency mode, and QMD boot work can still fan out when memory search is explicitly/default-enabled across agents.

## Best Possible Solution

Keep this issue open and either implement or split focused reliability work: make Active Memory opportunistic/nonblocking by default with a short deadline, add per-agent/session recall concurrency or dedupe/drop behavior, warn or choose a safer dedicated recall model instead of silently inheriting heavy chat models, cache or warm embedded-run plugin/model/auth setup for recall subagents, and add QMD boot controls such as max concurrent boot updates, jitter, lazy initialization, or active/default-agent-only policy. Status/doctor/docs should also surface expensive Active Memory and broad QMD-on-boot configurations.

## Risks / Open Questions

- Closing now would lose tracking for user-visible reply latency caused by a blocking pre-prompt Active Memory run with a 15s recommended/default-style budget unless operators tune it down.
- Multi-agent operators who explicitly enable memory search broadly can still arm multiple QMD managers and boot updates at gateway startup.
- The newest timing data suggests a second unresolved hot path: repeated plugin/model/auth setup inside embedded recall runs, including synchronous CLI/keychain probing in some auth configurations.
