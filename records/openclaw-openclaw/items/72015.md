## Summary

Keep open. The issue is still paired with open draft PR #73667, which uses closing syntax, and current main still runs Active Memory recall from the blocking pre-reply hook while inheriting the session or primary model when no plugin model is configured. Main has added useful mitigations around hard timeouts, circuit breaking, and QMD startup defaults, but the remaining maintainer decision is whether #73667 or a replacement fully closes the reliability-defaults gap.

## What This Changes

Request safer Active Memory and QMD defaults for multi-agent gateways by bounding or avoiding pre-reply blocking recall, using safer recall-model behavior, enforcing hard budgets/concurrency, and controlling QMD boot work.

## Best Possible Solution

Keep this issue open while maintainers review #73667 or replace it with a focused PR that makes pre-reply Active Memory recall safe under load, keeps recall budgets strict, gives operators clear fast-model defaults or warnings, adds recall drop/dedupe/concurrency behavior, and preserves QMD startup as lazy or explicitly bounded for broad multi-agent deployments.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: enable Active Memory for an eligible chat session with no pinned recall model and a slow or setup-heavy inherited model, then the before_prompt_build hook awaits recall before the main reply can proceed; current tests simulate the timeout side of that path.

## Solution Assessment

Unclear. Current main is not the complete solution because it still uses blocking recall and inherited-model defaults; #73667 is the active implementation candidate to review first, but maintainers should verify whether its narrower deadline/jitter scope satisfies the full reliability request.
