## Summary

Keep this open. Current main has useful partial mitigations for QMD startup laziness and Active Memory timeout backoff, but Active Memory is still a blocking pre-reply hook, still inherits the active/session model by default, and the paired draft implementation PR remains open and unmerged.

## What This Changes

The issue requests safer Active Memory and QMD defaults so multi-agent gateways do not block replies or overload memory backends during startup.

## Best Possible Solution

Keep the issue open until #73667 or a replacement lands a maintainable default profile: pre-reply recall should have a strict user-visible budget or fail-open/nonblocking path, inherited model behavior should be made safe or clearly warned, stacked recall work should be bounded, and QMD startup should stay lazy or be explicitly concurrency-limited for multi-agent gateways.

## Reproduction Assessment

Yes. A source-level reproduction exists by enabling Active Memory for an eligible interactive session without config.model and using a slow inherited session or primary model: before_prompt_build awaits maybeResolveActiveRecall before prompt construction completes. The QMD side is source-reproducible with opt-in startup refresh and multi-agent memory search defaults, where current tests still initialize all QMD agents.

## Solution Assessment

No. Current main is only a partial mitigation: QMD startup is lazy by default and Active Memory has a circuit breaker, but the blocking hook, inherited-model default, broad timeout plus setup grace, and absence of nonblocking/deadline/concurrency config remain. The narrow active path is to review and either land or supersede #73667.
