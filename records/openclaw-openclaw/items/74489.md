## Summary

Keep this PR open. It targets a real Discord plugin bug, but the current patch reuses startup-grade Discord REST retry/cooldown behavior inside status and capabilities probe paths, so a caller-provided probe timeout can be exceeded by minutes; the PR also has the protected `maintainer` label, so it needs maintainer handling before merge.

## What This Changes

The PR updates Discord REST and gateway metadata handling to sanitize Cloudflare/Error 1015 HTML 429 bodies, honor Retry-After or apply a fallback cooldown, derive parseable application IDs before REST lookup, add focused tests, and add a changelog entry.

## Best Possible Solution

Keep the Discord plugin-owned 429 classification and sanitizer direction, but make status/capabilities probes use a probe-specific retry policy or no cooldown sleep so their caller-provided timeout remains a real wall-clock budget, then land with the focused Discord tests and changed-gate proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor probe timeouts for application metadata:** `extensions/discord/src/probe.ts:48-52`
  - body: `fetchDiscordApplicationMe` now calls `fetchDiscord` with the default 429 retry policy, including the new 60s fallback cooldown. This path is used by `probeDiscord(..., { includeApplication: true })` for channel status/capabilities probes, where callers pass 2.5-30s timeout budgets; repeated Cloudflare 429s can therefore make those probes appear hung for minutes instead of respecting the probe timeout. Use a probe-specific retry policy or disable cooldown retries here.
  - confidence: 0.91
