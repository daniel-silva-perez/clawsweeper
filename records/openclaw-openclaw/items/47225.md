## Summary

Keep open. The feature is still useful and current main does not implement result-side tool schema selection, but this PR is not ready to merge because the proposed shared contract is only wired into the PI runner, treats an explicit empty override as absent, and does not give context engines live tool schemas to construct the override as advertised.

## What This Changes

The PR adds an optional `AssembleResult.toolsOverride` field plus PI runner stream wrapping that filters model-visible tool schemas by tool name.

## Best Possible Solution

Define an additive per-turn tool visibility API for context engines with explicit empty-subset semantics, a usable live schema or name-selection contract, PI and Codex harness enforcement, docs, and focused tests, while keeping executor authorization unchanged.

## Reproduction Assessment

Yes. A high-confidence source review reproduces the issues: the PR diff ignores empty arrays, current main only passes `availableTools` names to `assemble()`, and the Codex harness still sends `toolBridge.specs` unchanged.

## Solution Assessment

No. The proposed direction is reasonable, but the current patch is not the narrow complete fix because it changes a shared `AssembleResult` type without implementing the semantics in every harness or giving plugins a practical way to derive the override from live tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor explicit empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The guard only wraps the stream when `toolsOverride.length > 0`, so a context engine cannot intentionally hide all tool schemas for a pure-chat turn. Check for field presence instead, or clearly document and warn that an empty array is a no-op.
  - confidence: 0.96
- **[P2] Apply the shared override in every harness:** `src/context-engine/types.ts:10-16`
  - body: `toolsOverride` is added to the shared context-engine result type, but the PR only consumes it in the PI runner. The Codex app-server harness also calls `assemble()` and continues passing `toolBridge.specs`, so the same engine would silently keep the full schema set there.
  - confidence: 0.88
- **[P2] Expose a live-tool selection contract:** `src/context-engine/types.ts:16`
  - body: The result type asks plugins to return `AnyAgentTool[]`, but `assemble()` currently receives only `availableTools?: Set<string>`. That does not support the PR body's `params.tools.filter(...)` example or let plugin authors build an override from the live schemas without an external registry.
  - confidence: 0.9
