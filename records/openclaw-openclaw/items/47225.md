## Summary

Keep this PR open. Current main still lacks a context-engine output or model-request boundary for selecting the visible tool schema subset, so the underlying feature request remains valid; however, the submitted branch is not merge-ready because the proposed public contract and runner wiring have concrete behavior gaps.

## What This Changes

This PR adds an optional `toolsOverride` field to `AssembleResult`, adds a stream wrapper that filters model-visible tools by name, and wires that wrapper into the PI embedded runner when a context engine returns an override.

## Best Possible Solution

Define a backwards-compatible per-turn tool visibility contract for context engines, honor explicit empty-subset semantics, pass sufficient live tool information or make the override intentionally name-based, apply it consistently at each model-request boundary without changing executor authorization, and document/test the PI and Codex harness behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The new guard treats `toolsOverride: []` the same as an absent override, so an engine cannot intentionally suppress all tool schemas even though the contract says that when the field is present only those tools are included. Check for `undefined` instead, or document and warn that an empty list is a no-op.
  - confidence: 0.96
- **[P2] Apply the shared override in every context-engine harness:** `src/context-engine/types.ts:10-16`
  - body: `toolsOverride` is added to the shared `AssembleResult`, but the PR only consumes it in the PI embedded runner. The Codex harness also calls context-engine `assemble()` and currently projects only messages and `systemPromptAddition`, so the same engine would silently keep sending the full tool schema set there.
  - confidence: 0.84
- **[P2] Expose a usable live-tool selection contract:** `src/context-engine/types.ts:10-16`
  - body: The new result type requires `AnyAgentTool[]`, but `assemble()` receives only the name set on current main and the PR does not pass live tool definitions. The documented `params.tools.filter(...)` example is therefore not backed by the API; either make the override name-based or provide the resolved tools to engines.
  - confidence: 0.88
