## Summary

Keep open. Current main still lacks a context-engine tool schema override, and this PR is useful source work, but the proposed shared contract is incomplete: it only wires the PI embedded runner, treats an explicit empty override as a no-op, and does not give context engines live tool definitions to build the advertised override from.

## What This Changes

The PR adds an optional `toolsOverride` field to `AssembleResult`, a PI runner stream wrapper that filters model-visible tools by name, and wiring that applies the wrapper when a non-empty override is returned.

## Best Possible Solution

Define a backwards-compatible per-turn tool visibility contract for context engines, decide whether the override is name-based or schema-based, honor explicit empty subsets, apply the behavior at every model request boundary that uses context engines, and document/test both PI and Codex harness semantics without broadening tool executor authorization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The new guard treats `toolsOverride: []` the same as an absent override, so an engine cannot intentionally suppress all tool schemas even though the contract says that when the field is present only those tools are included. Check for `undefined` instead, or document and warn that an empty list is a no-op.
  - confidence: 0.96
- **[P2] Apply the shared override in every harness:** `src/context-engine/types.ts:10-16`
  - body: `toolsOverride` is added to the shared `AssembleResult`, but the PR only consumes it in the PI embedded runner. The Codex app-server harness also calls context-engine `assemble()` and continues sending `toolBridge.specs`, so the same engine would silently keep the full tool schema set there.
  - confidence: 0.88
- **[P2] Expose a usable live-tool selection contract:** `src/context-engine/types.ts:16`
  - body: The new result type requires `AnyAgentTool[]`, but `assemble()` currently receives only `availableTools?: Set<string>`. The PR description's `params.tools.filter(...)` example is not backed by the API, so plugins cannot build the override from live schemas without their own separate registry.
  - confidence: 0.9
