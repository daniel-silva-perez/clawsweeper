## Summary

Keep open. Current main still lacks this tool-visibility API, and the PR is useful but not merge-ready because the shared context-engine contract would be incomplete across harnesses and ambiguous for empty subsets and live tool selection.

## What This Changes

The PR adds optional `AssembleResult.toolsOverride`, a PI runner stream wrapper that filters model-visible tool schemas by name, and PI wiring that applies the wrapper when the override is non-empty.

## Best Possible Solution

Define an additive per-turn tool visibility API for context engines with explicit empty-subset semantics, a practical live schema or name-selection input, PI and Codex harness enforcement, docs, and focused tests, while keeping executor authorization unchanged.

## Reproduction Assessment

Yes. Source review is enough to reproduce the review issues: the PR diff ignores empty override arrays, current main exposes only `availableTools` names to `assemble()`, and the Codex harness still sends `toolBridge.specs` unchanged.

## Solution Assessment

No. The direction is reasonable, but the current patch is not the best complete solution because it changes a shared result type without defining usable selection inputs or applying the semantics across active harnesses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor explicit empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The guard only wraps the stream when `toolsOverride.length > 0`, so a context engine cannot intentionally hide all tool schemas for a pure-chat turn. Check field presence instead, or document and warn that an empty array is a no-op.
  - confidence: 0.96
- **[P2] Apply the shared override in every harness:** `src/context-engine/types.ts:10-16`
  - body: `toolsOverride` is added to the shared context-engine result type, but only the PI runner consumes it. The Codex app-server harness also calls `assemble()` and still starts threads with full `toolBridge.specs`, so the same engine would behave differently there.
  - confidence: 0.88
- **[P2] Expose a live-tool selection contract:** `src/context-engine/types.ts:16`
  - body: The result type asks plugins to return `AnyAgentTool[]`, but `assemble()` currently receives only `availableTools?: Set<string>`. That does not support the PR body's `params.tools.filter(...)` example or let plugins build an override from live schemas without another registry.
  - confidence: 0.9
