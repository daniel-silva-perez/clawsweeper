## Summary

Keep this issue open. Current main confirms the key distinction raised in the discussion: `contextPruning.mode: "off"` disables the pruning extension, but compaction is a separate path. However, current main does not provide an explicit `compaction.mode: "off"`, and there is no exact regression proof for the reported Anthropic API-key + Opus 4.6 + `contextTokens: 1000000` + `reserveTokensFloor: 200000` configuration. Closing would risk treating a real config/runtime gap as a solved support question.

## Best Possible Solution

Keep the issue open and turn it into a focused config/runtime task: add a regression test for the reported Anthropic API-key configuration, document the independence of `contextPruning` and compaction in the user-facing config docs, and decide whether to add an explicit `compaction.mode: "off"` or document/support a hard-limit-only recipe such as zero reserve/floor. Also verify that Anthropic `context1m` and/or configured model context windows feed the same context budget used by Pi compaction.

## Risks / Open Questions

- Closing now would skip the unresolved product/API question of whether OpenClaw should support an explicit compaction-off or hard-limit-only mode.
- The exact reported configuration is not covered by a regression test, so the low-context compaction behavior could still reproduce for Anthropic Opus 4.6 users who expect `agents.defaults.contextTokens` to raise the runtime window.
- Docs explain Anthropic `context1m`, but current compaction-budget code still needs an explicit audit against the provider header/runtime model path before treating the 1M-context part of this report as fixed.
