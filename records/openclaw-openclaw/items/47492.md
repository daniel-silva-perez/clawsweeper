## Summary

Keep open. Current main has exec approvals, turn-source routing metadata, plugin approval hooks, and input-provenance annotations, but it still does not implement the requested gateway-enforced verified-human instruction-provenance policy for sensitive target patterns. The linked CaMeL work and #54508 are related security provenance efforts, but they do not make this issue obsolete because this issue is about clean-session/autonomous sensitive actions rather than untrusted-content taint alone.

## What This Changes

The issue requests a policy layer that lets operators mark sensitive side-effect targets and require those actions to trace back to a verified human instruction or go through a stricter approval flow.

## Best Possible Solution

Define a generic instruction-provenance contract and gateway/tool-call policy point for sensitive side-effect targets, then wire it into exec and other high-risk surfaces with config schema, approval UX, tests, and docs once the security semantics are agreed.

## Reproduction Assessment

Not applicable as a feature/security design request. The high-confidence verification path is static: source, schema, docs, and tests show annotations and approval guardrails, but no verified-human sensitive-target enforcement.

## Solution Assessment

Unclear for implementation; this should not be handled as a small autonomous patch. The maintainable path is a coordinated design with the CaMeL/taint work and the approval policy surfaces so taint, direct human intent, standing orders, and approval routing compose cleanly.
