## Summary

Keep open. Current main has exec approvals, turn-source routing metadata, plugin approval hooks, and input-provenance annotations, but it still does not implement a gateway-enforced verified-human instruction-provenance policy for sensitive target patterns. The related CaMeL and persistent-taint items are adjacent security work, not merged replacements for this clean-session/autonomous-action gating request.

## What This Changes

Add a gateway-enforced policy that marks sensitive side-effect targets and requires actions against them to trace to a verified human instruction, with denial or stricter approval fallback when they do not.

## Best Possible Solution

Define a generic instruction-provenance contract and a gateway/tool-call policy point for sensitive side-effect targets, then wire it into exec and other high-risk surfaces with config schema, approval UX, protocol changes, tests, and docs after maintainers agree on the security semantics.

## Reproduction Assessment

Not applicable as a feature/security design request. The high-confidence verification path is static: current source, schema, docs, and related issue context show annotations and approval guardrails, but no verified-human sensitive-target enforcement.

## Solution Assessment

Unclear for immediate implementation; a one-off exec-only patch would likely be too narrow. The maintainable solution is a coordinated core design with the approval, provenance, and CaMeL/taint surfaces so sensitive target policy applies consistently across side-effect tools.
