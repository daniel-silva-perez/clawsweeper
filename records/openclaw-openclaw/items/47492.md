## Summary

Keep open. Current main has exec approvals, turn-source routing metadata, input-provenance annotations, and plugin before-tool-call approval hooks, but it still does not provide a gateway-enforced verified-human instruction-provenance policy for sensitive target patterns. The related CaMeL and persistent-taint work is adjacent security hardening, not a merged replacement for this clean-session/autonomous-action gating request.

## What This Changes

Add a gateway-enforced policy layer that lets operators mark sensitive exec or side-effect targets and require those actions to trace back to a verified human instruction or stricter approval flow.

## Best Possible Solution

Define a generic instruction-provenance contract and a gateway/tool-call policy point for sensitive side-effect targets, then implement it across exec and other high-risk surfaces with config schema, protocol changes, approval UX, tests, and user-facing docs after maintainers agree on the security semantics.

## Reproduction Assessment

Not applicable as a feature/security design request. The high-confidence verification path is static: current source, schemas, docs, and related issue state show existing approval/provenance pieces but no verified-human sensitive-target enforcement.

## Solution Assessment

Unclear for immediate implementation. The requested direction is valid, but the best maintainable solution is a coordinated core design rather than a one-off exec-only matcher.
