## Summary

Keep open. Current main has exec approvals, owner-only tool gating, approval forwarding, plugin before-tool-call approval hooks, and inter-session provenance annotations, but it does not implement the requested gateway-enforced verified-human instruction-provenance policy for sensitive target patterns. The related CaMeL PR remains open and addresses taint from untrusted content, which is adjacent but distinct from this issue's clean-session/autonomous-action concern.

## Best Possible Solution

Keep this issue open for maintainer security/product design. The best path is to coordinate with #54508 and #39231, define a generic direct-human or verified-instruction provenance contract and a gateway/tool-call enforcement point for sensitive side-effect targets, then add config schema, target matching, approval UX, tests, and docs once the semantics are agreed.
