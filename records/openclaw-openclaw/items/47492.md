## Summary

Keep open. Current main has exec approvals, owner-only tool gating, approval forwarding, and input/inter-session provenance annotations, but it still does not implement a gateway-enforced verified-human instruction-provenance policy for sensitive target patterns. The related CaMeL work tracks untrusted-content taint and remains distinct from this issue's clean-session/autonomous-action concern.

## What This Changes

Add a gateway-enforced policy layer that requires sensitive side-effect actions or targets, especially exec targets, to trace back to a verified human instruction before running or requiring stricter approval.

## Best Possible Solution

Define a generic instruction-provenance contract and gateway/tool-call policy point for sensitive side-effect targets, then wire it into exec and other high-risk surfaces with config schema, approval UX, tests, and docs once the security semantics are agreed.

## Reproduction Assessment

Not applicable as a feature/security design request. The current-main verification path is static: source, schema, docs, and tests show provenance annotations and approval guardrails, but no verified-human sensitive-target enforcement.

## Solution Assessment

Unclear for implementation; this should not be auto-fixed as a small patch. The maintainable route is a coordinated security/product design with #54508 and #39231 so taint, human instruction provenance, and approval policy compose cleanly.
