## Summary

Keep open. Current main has exec approval, allowlist, plugin approval, and limited input-provenance machinery, but it does not implement the requested gateway-enforced policy that requires sensitive exec/tool targets to trace back to a verified human instruction. The related CaMeL PR remains open and tracks taint/untrusted-content provenance, while this request explicitly covers clean-session/autonomous actions where taint can be absent.

## Best Possible Solution

Keep this issue open as a core security design item. The best path is to define a first-class instruction-origin/provenance contract, then extend the gateway/tool-policy/exec-approval boundary so operators can mark sensitive targets and require a fresh verified-human instruction or stricter approval challenge, while coordinating with #39231 and #54508 for taint and risk-class integration.

## Risks / Open Questions

- Closing now would lose a real security/product design gap: clean autonomous actions can still reach sensitive targets under existing generic exec policy if the operator config permits them.
- A plugin-only implementation would be incomplete unless core first exposes a trustworthy instruction-origin/provenance contract at the tool-call boundary.
- This should be coordinated with the open CaMeL/taint work so taint provenance and verified-human intent provenance remain separate but composable.
