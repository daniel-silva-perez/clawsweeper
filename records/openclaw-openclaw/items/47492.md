## Summary

Keep open. Current main has exec approvals, owner-only tool gating, approval forwarding, and input/inter-session provenance annotations, but it does not implement the requested gateway-enforced verified-human instruction-provenance policy for sensitive target patterns. The related CaMeL work is still open and tracks taint from untrusted content, which is distinct from this issue's clean-session/autonomous-action concern.

## Best Possible Solution

Keep this issue open for maintainer security/product design. The best path is to coordinate with #54508 and #39231, define a generic provenance contract and gateway/tool-call policy point for sensitive side-effect targets, then add config schema, enforcement, approval UX, tests, and docs once the semantics are clear.

## Risks / Open Questions

- This is security-sensitive product design, not a narrow implementation cleanup; it needs a threat model for what counts as verified human provenance across chat, ACP, cron, heartbeat, subagents, replayed sessions, and standing orders.
- Closing as covered by CaMeL would lose the distinct clean-session/autonomous-workflow safety gap described in the issue.
- A naive implementation could break legitimate cron or standing-order automation, or incorrectly treat channel-authenticated messages as non-replayable human authorization.
- The design should be coordinated with #54508 and open PR #39231 so taint provenance, direct-human intent, and exec risk classes do not become overlapping policy systems.
