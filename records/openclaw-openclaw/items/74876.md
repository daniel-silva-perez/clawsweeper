## Summary

Keep open. Current main still defaults group/channel source turns to message-tool-only delivery, so the reported visible final-reply suppression remains a real current behavior; docs and config help now describe the all-channel setting, but the remaining upgrade-warning, migration/default, and release-note policy choices need maintainer judgment.

## What This Changes

The issue requests that OpenClaw address the v2026.4.27 group/channel visible-reply default change by warning users, documenting all affected channels, making the opt-out easy to apply, clarifying reload behavior, and considering legacy-preserving migration.

## Best Possible Solution

Keep the safety default only if the upgrade path is explicit: public release notes and docs should name all group/channel chats, the supported messages.groupChat.visibleReplies opt-out should be operator-accessible through documented config commands, and maintainers should intentionally choose between a startup/doctor/runtime warning, legacy-preserving migration, or no migration.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with empty config and ChatType group or channel, delivery resolves to message_tool_only and current dispatch tests verify no automatic final reply is sent until messages.groupChat.visibleReplies is set to automatic.

## Solution Assessment

Unclear. The current implementation is internally consistent as a safety default and now documented, but it does not fully solve the upgrade-surprise report; the narrowest maintainable next step depends on maintainer policy, with PR #75015 covering only the warning path.
