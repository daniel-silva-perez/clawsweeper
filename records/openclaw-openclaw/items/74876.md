## Summary

Keep open. Current main still defaults group/channel source turns to message-tool-only delivery, so the reported final-reply suppression remains real; docs and tests now document the all-channel behavior and the automatic opt-out, but the warning, changelog wording, config.patch allowlist, and default/migration policy are not fully resolved on main.

## What This Changes

The issue asks OpenClaw to address the v2026.4.27 change where group/channel final replies across all chat channels stay private unless messages.groupChat.visibleReplies is explicitly set to automatic.

## Best Possible Solution

Land a coordinated maintainer-approved path that keeps the intended group-room safety posture explicit: all-channel release-note wording, operator-accessible config for the automatic opt-out, accurate hot-reload versus restart guidance, and either a startup/runtime warning or an intentional migration/default decision.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with empty config and ChatType group or channel, source delivery resolves to message_tool_only and the current dispatch regression test verifies no automatic final reply is sent; setting messages.groupChat.visibleReplies to automatic restores final delivery.

## Solution Assessment

Unclear. The current implementation is maintainable as a safety default, but it does not fully solve the upgrade-surprise report; the best fix depends on maintainer policy across the warning, changelog, config.patch, and migration/default tradeoffs.
