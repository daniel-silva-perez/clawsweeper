## Summary

Keep open. Current main still defaults group/channel source turns to message-tool-only delivery, so the reported final-reply suppression remains a deliberate current behavior; docs now explain the safety default, but the upgrade warning, changelog wording, and migration/default policy are still unresolved and already have related open PRs.

## What This Changes

The issue asks OpenClaw to address a v2026.4.27 group/channel reply-delivery default change that makes final replies private unless `messages.groupChat.visibleReplies: "automatic"` is configured.

## Best Possible Solution

Keep the group/channel safety default only with an explicit upgrade path: release notes should name all group/channel chats, the supported `messages.groupChat.visibleReplies: "automatic"` opt-out should be clearly operator-accessible and hot-reloadable, and maintainers should intentionally choose warning, migration, or no-migration behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with empty config and `ChatType` of `group` or `channel`, current main resolves delivery to `message_tool_only`, and the dispatch test verifies no automatic final reply is sent until `messages.groupChat.visibleReplies: "automatic"` is configured.

## Solution Assessment

Unclear. The current implementation is internally consistent as a group-room leakage safety default, but it does not fully solve the upgrade-surprise report; the best fix depends on maintainer policy for warnings, changelog wording, and migration compatibility.
