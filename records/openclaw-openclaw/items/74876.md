## Summary

Keep open. Current main still intentionally defaults group/channel source turns to message-tool-only delivery, so the reported final-reply suppression remains reproducible; docs and config schema now make the behavior clearer, but the upgrade-warning, migration, and agent-facing config-guard policy still need maintainer judgment.

## What This Changes

The issue reports that upgrading to v2026.4.27 makes group/channel final replies private across channels by default and asks for clearer changelog/docs, easier configuration, reload guidance, and possible migration or warning behavior.

## Best Possible Solution

Keep the safety default only if the upgrade path is explicit: public docs and release notes should name all group/channel chats, the supported `messages.groupChat.visibleReplies: "automatic"` opt-out should be operator-accessible, and maintainers should intentionally choose between a startup/doctor warning, legacy-preserving migration, or no migration.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with empty config and `ChatType` of `group` or `channel`, delivery resolves to `message_tool_only`, and the dispatch regression test verifies no automatic final reply is sent until `messages.groupChat.visibleReplies: "automatic"` is configured.

## Solution Assessment

Unclear. The current implementation is internally consistent and documented as a safety default, but it does not fully solve the upgrade-surprise report; the narrowest maintainable next step depends on maintainer policy for warning, migration, and agent-facing config permissions.
