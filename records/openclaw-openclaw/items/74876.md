## Summary

Keep open. Current main still intentionally defaults group/channel source turns to message-tool-only delivery, so the reported visible final-reply suppression remains reproducible; main has partially addressed the surrounding docs/config claims, but the remaining upgrade-warning or migration decision needs maintainer product judgment.

## What This Changes

The issue asks for upgrade handling around the 4.27 group/channel visible-reply default: clearer all-channel release notes/docs, an easier config opt-out, restart guidance, and possible legacy-preserving migration.

## Best Possible Solution

Keep the safety default only if the upgrade path is explicit: shipped release notes and public docs should name all group/channel chats, the supported `messages.groupChat.visibleReplies: "automatic"` opt-out should be easy to apply, and maintainers should intentionally choose either a startup/doctor warning or a legacy-preserving migration for existing installs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: empty config plus `ChatType: "group"` or `"channel"` resolves to `message_tool_only`, and dispatch coverage shows the final reply is not sent until `messages.groupChat.visibleReplies: "automatic"` is configured.

## Solution Assessment

Unclear. The current implementation is internally consistent and tested as a safety default, but it does not fully resolve the upgrade-surprise report; the narrower maintainable next step is likely warning/docs/release guidance unless maintainers decide old installs should be migrated to legacy automatic replies.
