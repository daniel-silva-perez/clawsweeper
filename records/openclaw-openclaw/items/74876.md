## Summary

Keep open. Current main still intentionally defaults group/channel source turns to message-tool-only delivery, so the reported upgrade surprise remains reproducible; docs and config references now describe the behavior more clearly, but warning, migration, and agent-facing config-permission policy still need maintainer judgment.

## What This Changes

The issue requests warning, documentation, config-access, and migration follow-up for group/channel final replies becoming private by default after v2026.4.27.

## Best Possible Solution

The desired end state is an intentional upgrade path: all group/channel surfaces should have clear release-note and docs language, a supported way for operators to opt back into legacy automatic replies, and a maintainer-approved decision on whether existing installs get a warning, doctor check, or migration.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. With empty config and `ChatType` of `group` or `channel`, current main resolves delivery to `message_tool_only`, and the dispatch regression test verifies no automatic final reply is sent until `messages.groupChat.visibleReplies: "automatic"` is configured.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The current implementation is coherent as a safety default, but it does not fully solve the upgrade-surprise report; the narrowest maintainable next step depends on maintainer policy for warning, migration, and agent-facing config permissions.
