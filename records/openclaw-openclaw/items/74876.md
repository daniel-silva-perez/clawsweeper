## Summary

Keep open. Current main still defaults group/channel source turns to message-tool-only delivery, so the reported “processing but no visible final reply” behavior remains real by design rather than fixed; however, several surrounding claims have already been partially addressed or contradicted by current main, so the remaining action needs maintainer product judgment around upgrade compatibility versus the group-room safety default.

## What This Changes

The issue requests clearer upgrade handling for the 4.27+ change that makes normal final replies private by default in group/channel chats across all channels unless `messages.groupChat.visibleReplies` is set to `automatic`.

## Best Possible Solution

Keep the current safety default only if the upgrade path is explicit: public docs and release notes should describe all group/channel chats, the supported `messages.groupChat.visibleReplies: "automatic"` opt-out should be easy to apply, and any startup/doctor warning or legacy-preserving migration should be intentionally chosen and covered by shared auto-reply/config tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: empty config plus `ChatType: "group"` or `"channel"` resolves to `message_tool_only`, and the dispatch test shows the final reply is not sent while the `automatic` config restores visible delivery.

## Solution Assessment

Unclear. The current implementation is internally consistent and tested as an intentional safety default, but it does not resolve the upgrade surprise; a safer path than silently reverting the default is likely an explicit warning/docs/doctor path unless maintainers decide existing installs should be migrated to legacy automatic replies.
