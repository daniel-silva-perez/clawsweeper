## Summary

Keep open. Current main does not already contain the proposed `src/verify` module, and earlier review feedback on empty-list guards, order-agnostic standalone equality, and recursive pipeline return walking appears to have been addressed in later PR commits, but the PR is still not merge-ready because it targets stale tool-policy and catalog shapes and can still produce incomplete verification inputs.

## What This Changes

This PR adds a new parser-only `src/verify` module with README, TypeScript AST parsers, shared parsed-policy types, a Vitest suite, and a standalone runner for extracting tool catalog, alias, owner-only, subagent-deny, and policy-pipeline metadata for future SMT-LIB2 verification.

## Best Possible Solution

If maintainers want this verification lane in core, rebase the branch onto current main, update extraction for the current catalog, owner-only approval-class map, and subagent policy shapes, make every unsupported AST form fail closed, and either remove generator/Z3 documentation or add the generator in the same reviewed scope.

## Reproduction Assessment

Yes. A high-confidence reproduction path is the source comparison against current main: the PR parser searches for an absent owner-only symbol, while its fixtures assert 25 tools and old policy values that do not match the current 32-tool catalog and deny lists.

## Solution Assessment

No. Parser-only tooling is a reasonable incremental direction, but this implementation is not the best current fix until it targets the live contracts and refuses partial extraction; the README should also stay parser-only unless the SMT generator is added.
