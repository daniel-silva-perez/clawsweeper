## Summary

Keep open: this is a useful security-verification direction, but the PR is not mergeable against current main because its parser targets stale tool-policy shapes, its fixtures lag the current catalog, and some extraction helpers still fail open for unsupported AST forms.

## What This Changes

The PR adds a new `src/verify` parser module, README, Vitest test, and standalone runner for extracting tool catalog, aliases, owner-only policy, subagent deny lists, and policy-pipeline metadata for future SMT-LIB2 verification.

## Best Possible Solution

Rebase onto current main, update extraction for the 32-tool catalog, approval-class owner-only map, and current subagent policy, make unsupported AST forms fail closed, and either align the README with parser-only scope or add the documented generator as a maintainer-approved verification lane.

  - body: Current main no longer declares `OWNER_ONLY_TOOL_NAME_FALLBACKS`; it uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS` to map owner-only tools to approval classes. After rebase this parser will throw before it can extract the current owner-only security boundary.

- **[P1] Reject unparsed policy array entries:** `src/verify/parse-policies.ts:92-94`
  - body: `parseStringArray` drops unsupported AST entries and only returns the remaining string literals. If a deny list later uses a constant, spread, or template entry, verification can silently omit that denied tool and reason about a weaker policy than runtime enforces.
  - confidence: 0.88
- **[P1] Reject unparsed catalog profile entries:** `src/verify/parse-tools.ts:80-85`
  - body: `extractStringArray` filters profile entries it cannot decode and returns the truncated array. A catalog refactor through constants, spreads, or templates would silently remove profile memberships from the verification model instead of stopping extraction.
  - confidence: 0.84
- **[P2] Refresh current catalog expectations:** `src/verify/__tests__/parse-tools.test.ts:12-44`
  - body: Current main has 32 core tool definitions, but this test still asserts 25 and hard-codes the older id set. The parser fixtures need to be regenerated or derived from the current catalog after rebase.
  - confidence: 0.95
- **[P2] Remove commands for the absent SMT generator:** `src/verify/README.md:38-45`
  - body: The README documents `generate-smt.ts`, emitted model files, and Z3 commands, but this PR only adds parser code. Users following the new docs would hit missing-module or missing-output errors.
  - confidence: 0.9
