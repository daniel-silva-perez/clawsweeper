## Summary

Keep this PR open, but do not merge it as-is. Current main still has no `src/verify` implementation, so the idea is not already shipped; however, the PR is stale against current tool-policy sources, its tests encode older tool and policy shapes, and the proposed security-verification parsers still have fail-open fidelity risks.

## What This Changes

This PR adds a new `src/verify` parser module with README and tests to extract core tool catalog, tool aliases, owner-only/subagent policy data, and tool-policy pipeline metadata for later SMT-LIB2 verification.

## Best Possible Solution

Rebase the PR onto current main, update it to parse the current catalog and owner-approval-class policy shape, make every parser fail closed on unsupported AST forms, scope the README to the files actually included, and get maintainer agreement on how the verification tooling should be run and kept in sync.

Full review comments:

- **[P1] Target the current owner-only approval map:** `src/verify/parse-policies.ts:101-106`
  - body: Current main no longer declares `OWNER_ONLY_TOOL_NAME_FALLBACKS`; it uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS` as a map of tool names to approval classes. After rebasing, `parsePolicies` will throw before producing any verification input, so the parser is not reading the runtime source it is meant to verify.
  - confidence: 0.96
- **[P1] Reject unparsed policy array entries:** `src/verify/parse-policies.ts:73-79`
  - body: `parseStringArray` filters out every non-string AST element and only checks that the remaining list is non-empty. If a deny list later uses a constant, spread, or template element, verification can silently omit that denied tool and reason about a weaker policy than runtime enforces.
  - confidence: 0.88
- **[P1] Reject unparsed catalog profile entries:** `src/verify/parse-tools.ts:75-82`
  - body: `extractStringArray` drops profile entries it cannot decode and returns the truncated array. A small refactor of catalog profiles through constants or spreads would silently remove profile memberships from the parsed verification model instead of stopping generation.
  - confidence: 0.84
- **[P2] Refresh catalog expectations for current tools:** `src/verify/__tests__/parse-tools.test.ts:10-12`
  - body: Current main has 32 core tool definitions, but the added test still asserts 25 and hard-codes old ids, profiles, and group membership. The advertised parser test will fail once the branch is rebased onto current main.
  - confidence: 0.95
- **[P2] Remove commands for absent SMT generator:** `src/verify/README.md:29-39`
  - body: The README documents `generate-smt.ts`, `emit-smt/`, model outputs, and Z3 commands, but this parser-only PR does not add those files. Users following the new docs would hit missing-module errors.
  - confidence: 0.9
