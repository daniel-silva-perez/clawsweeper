## Summary

Keep open: this is useful parser-only verification work and the branch addressed some earlier review feedback, but it is not merge-ready against current main because it targets stale tool-policy/catalog contracts and can still produce incomplete security-verification inputs for unsupported AST forms.

## What This Changes

The PR adds a new parser-only `src/verify` module with AST parsers, shared parsed-policy types, README documentation, a Vitest suite, and a standalone runner for extracting OpenClaw tool catalog and policy metadata.

## Best Possible Solution

If maintainers want this verification lane in core, land it as a rebased parser-only change that reads the current catalog, owner-only approval-class map, and subagent policy shapes, fails closed on unsupported AST forms, and documents only generator/Z3 functionality that is actually included.

## Reproduction Assessment

Yes. A high-confidence reproduction path is the source comparison against current main: the PR parser searches for a retired owner-only symbol, while current main has a 32-tool catalog and different subagent/owner-only policy fixtures than the branch asserts.

## Solution Assessment

No. Parser-only tooling is a reasonable incremental direction, but this implementation is not the best current solution until it targets the live contracts, rejects partial extraction, and avoids documenting absent SMT generator workflows.
