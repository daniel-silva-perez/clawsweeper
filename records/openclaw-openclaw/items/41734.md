## Summary

Keep open: the PR is useful source work, but it is not merge-ready against current main because the live tool-policy/catalog contracts have drifted and the parser can still produce incomplete verification inputs for unsupported AST forms.

## What This Changes

This PR adds a parser-only `src/verify` module with README, TypeScript AST parsers, shared parsed-policy types, a Vitest suite, and a standalone runner for extracting tool catalog, alias, owner-only, subagent-deny, and policy-pipeline metadata for future SMT-LIB2 verification.

## Best Possible Solution

If maintainers want this verification lane in core, rebase it onto current main, update extraction for the current catalog, approval-class owner-only map, and subagent policy shapes, make unsupported AST forms fail closed, and keep the README parser-only unless generator/Z3 files are added in the same reviewed scope.

## Reproduction Assessment

Yes. A high-confidence reproduction path is the source comparison against current main: the PR parser searches for a retired owner-only symbol, while current main has a 32-tool catalog and different policy lists than the PR fixtures assert.

## Solution Assessment

No. Parser-only tooling is a reasonable incremental direction, but this implementation is not the best current solution until it targets the live contracts, refuses partial extraction, and avoids documenting absent generator/Z3 workflows.
