## Summary

Keep open. Current main does not already provide priority-aware command queue scheduling, and the proposed patch is useful but not merge-ready because its starvation guard still lets high-priority work bypass aged low-priority work indefinitely and the branch is stale against the current queue timeout API.

## What This Changes

The PR adds low/normal/high command queue priorities, priority-based in-lane selection, starvation promotion, and command queue tests for the new scheduling behavior.

## Best Possible Solution

Rebase or replace the branch on current main, preserve the existing command queue timeout/snapshot/reset and lane APIs, and implement priority selection with a starvation guard that eventually selects aged low-priority work even while high-priority entries continue arriving.

## Reproduction Assessment

Yes. A high-confidence reproduction follows from the PR code: enqueue a low-priority entry, let it age past the threshold, then keep enqueueing high-priority entries; the aged entry only becomes normal priority, so high-priority entries continue to win.

## Solution Assessment

No. In-place priority scanning fits the existing queue shape, but the current proposed promotion rule and stale option type are not the narrowest maintainable fix; the safer path is to preserve current queue options and use an age-based or above-high starvation score with regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guarantee starvation promotion over high-priority work:** `src/process/command-queue.ts:133-134`
  - body: After `STARVATION_PROMOTION_MS`, a low-priority entry is only promoted to normal priority. A lane with a steady stream of high-priority entries can still bypass that aged entry forever, so the starvation guarantee is not met; promote aged work above high priority or use an age-based score and add a repeated-high-arrival regression test.
  - confidence: 0.9
- **[P2] Preserve task timeout options when adding priority:** `src/process/command-queue.types.ts:6`
  - body: The branch adds `priority` to an older inline enqueue option shape and omits current main's `CommandQueueEnqueueOptions.taskTimeoutMs` path. Rebase this onto the current option type and keep storing `taskTimeoutMs`, otherwise stuck lane tasks lose their timeout escape hatch.
  - confidence: 0.86
