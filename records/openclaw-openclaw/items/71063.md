## Summary

Keep open: the bug is valid, but the proposed guard does not cover the current Linux missing-binary failure shape. The supervisor can shell-wrap the child, report `child.stdin.writable === true`, and then emit an async `EPIPE`, so this branch still leaves the reported gateway-crash path reachable.

## What This Changes

The PR changes `createChildAdapter` to require `child.stdin.writable` before the initial stdin write and adds a unit test for a stubbed non-writable stdin stream.

## Best Possible Solution

Handle the initial `params.input` write/end path as a managed pipe operation: attach or otherwise consume stdin stream errors before writing, protect synchronous write/end failures, preserve normal writable stdin behavior, and add regression coverage for both non-writable stubs and the shell-wrapped async `EPIPE` missing-binary shape.

## Reproduction Assessment

Yes. A read-only Node probe using the same `/bin/sh -c ... exec "$0" "$@"` wrapper from current main reproduced `child.stdin.writable === true` followed by an uncaught `EPIPE`, and the Gemini CLI path reaches this supervisor input write.

## Solution Assessment

No. The PR is pointed at the right file, but `child.stdin.writable` is not the narrowest sufficient contract because Node streams can emit pipe errors asynchronously after the guard passes; the safer fix is explicit stdin write/end error handling around the initial input path.
