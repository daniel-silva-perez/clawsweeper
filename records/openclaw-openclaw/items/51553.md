## Summary

Keep this PR open. Current main still has the exact stale-cache gap the PR targets: `updateLastRoute` enters the session-store lock and then calls `loadSessionStore(storePath)` without `{ skipCache: true }`, while adjacent locked session-store writers already bypass the cache. The PR is narrow and consistent with the existing pattern; the main follow-up is adding a focused regression test for `updateLastRoute`.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a current-main version of the `updateLastRoute` cache-bypass fix, using the same locked-writer style as the adjacent functions, and add a focused regression test that primes the session-store cache, simulates an external preserved-mtime write, then verifies `updateLastRoute` does not clobber that data.
