## Summary

Keep this PR open. Current main still has the stale-cache read in `updateLastRoute`: it enters `withSessionStoreLock` and then calls `loadSessionStore(storePath)` without `{ skipCache: true }`, while the neighboring lock-holding writers already bypass the cache inside the lock. The PR is still a focused implementation candidate rather than obsolete work.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a current-main version of the `{ skipCache: true }` change in `updateLastRoute`, preferably matching the neighboring writer pattern and adding a regression test that primes the session-store cache, simulates an external preserved-mtime write, calls `updateLastRoute`, and verifies unrelated external fields are not clobbered.
