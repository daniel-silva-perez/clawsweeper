## Summary

Current main still has only machine-oriented inter-session provenance for sessions_send. It can identify that a message came through sessions_send and often the source session key/channel, but it does not expose the requested human-readable sender/recipient identity for session history display or filtering.

## Best Possible Solution

Keep this issue open and implement an additive core session/provenance path: persist or resolve display-safe source and target identity for sessions_send, surface it in session history/UI filtering, preserve sender-side target metadata, and add tests for receiving-session display plus sender-session tool/result visibility.
