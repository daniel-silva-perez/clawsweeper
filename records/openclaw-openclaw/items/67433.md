## Summary

Keep open. Current main still has only the fire-and-forget `/hooks/agent` behavior and does not parse or return a blocking agent result; this PR remains a live implementation candidate for the open result-return tracker #49019. The branch is now conflicting and needs maintainer rebase/review because current main moved hook request handling and added newer hook event routing and announcement protections that the PR must preserve.

## What This Changes

The PR adds an optional `blocking` field to agent hook payloads, makes hook dispatch return `{ runId, outputText?, agentError? }`, updates HTTP responses/idempotency handling for blocking and non-blocking requests, and adds Gateway hook tests.

## Best Possible Solution

Rebase this PR onto current main’s `src/gateway/server/hooks-request-handler.ts` and `src/gateway/server/hooks.ts`, preserve the current hook security and announcement policies, document the accepted blocking/result contract, and run targeted Gateway hook tests. If maintainers want a different contract than `blocking` plus `text`, keep #49019 as the canonical product/API tracker and revise this PR around that shape.

Yes. I did not start a live Gateway because this review is read-only.

## Solution Assessment

Unclear as currently proposed. Awaiting the isolated agent run and returning text is a plausible narrow implementation, but the branch is stale and must be rebased to current handler files while preserving hook security, idempotency, announcement, docs, and timeout semantics.
