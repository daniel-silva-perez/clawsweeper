## Summary

Keep open. Current main still routes `/v1/chat/completions` through the normal agent ingress path with a generated session key, and the agent path still persists session metadata, skills snapshots, session file paths, and post-run usage/model data. The repo has a raw model-run concept for CLI/RPC, but the OpenAI-compatible HTTP handler does not expose a passthrough/no-persistence mode, and the config schema has no such option. Related issues cover broader ephemeral sessions, session bloat, health-check misuse, and REST session management, but none supersede this specific lightweight LLM gateway request.

## Best Possible Solution

Keep this issue open and treat it as a core Gateway/API design item. The likely fix is a narrowly scoped `gateway.http.endpoints.chatCompletions` passthrough/raw mode that uses provider routing/fallback/auth without creating a session key, writing `sessions.json`, resolving transcript files, loading skills/tools, or injecting normal agent context. The docs should also distinguish “new session key per request” from true non-persistent/stateless execution, and the work should coordinate with #48159 so the broader ephemeral-session design does not conflict with this HTTP proxy mode.
