## Summary

Keep open. Current main still treats `/v1/chat/completions` as a normal agent-run ingress surface: anonymous requests get generated agent session keys, the handler calls the agent pipeline, and that path persists session metadata, skills snapshots, transcript file paths, and run usage. The related reports are adjacent but do not supersede this specific lightweight no-persistence gateway mode.

## What This Changes

The issue requests a configurable lightweight passthrough mode for `/v1/chat/completions` that preserves auth, model routing, and fallback while skipping session persistence, skills/tools, memory, and transcript writes.

## Best Possible Solution

Design a core Gateway Chat Completions passthrough/raw mode that uses the existing provider selection, model allowlist, auth profile, fallback, streaming, and usage accounting contracts without creating session store rows or transcript files. Update the docs to distinguish “new session key per request” from true non-persistent execution, and coordinate the semantics with the broader ephemeral-session work in #48159.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: an anonymous Chat Completions request is assigned an `agent:<id>:openai:<uuid>` session key and sent through `agentCommandFromIngress`, whose normal path persists session metadata and transcript state. I did not run a live request because this review is read-only and the code path itself would write session artifacts.

## Solution Assessment

Unclear. The requested passthrough mode is directionally appropriate for the reported gateway use case, but the exact fix should be a deliberate Gateway/API design rather than the minimum anonymous-request skip, because callers may depend on explicit session keys, model overrides, auth, fallback, streaming, and operator-scope behavior.
