## Summary

Keep this PR open. Current main still lacks segment-boundary state or a Control UI consumer that can prevent already-committed pre-tool stream text from rendering again, and the PR branch itself appears to add only a gateway-side `message.segmentOffset` hint without the UI-side handling needed to fix the visible browser duplication.

## What This Changes

The PR adds per-run gateway `segmentOffsets`, emits optional `message.segmentOffset` hints on full-snapshot chat deltas, and threads the new map through abort, maintenance, runtime, request-context, and helper paths.

## Best Possible Solution

Keep this PR open for maintainer review. The best fix is a current-main revision that either makes the Control UI segment-aware, or otherwise prevents it from rendering already committed pre-tool text twice, while preserving full-snapshot `chat` deltas for generic WS, TUI, ACP, and node/session subscribers. Add focused regression coverage for text -> tool -> text runs in the gateway chat event path and the Control UI stream/tool rendering path.
