## Summary

Keep this issue open. Current main still does not recognize `.m2a` in the shared media MIME/extension maps, voice-compatible audio extension set, Telegram document placeholder/preflight path, WebChat audio embedding map, UI transcript audio extension map, or memory multimodal audio extension list. The related PR #23260 was closed unmerged, so it does not supersede the work.

## Best Possible Solution

Implement a focused media-handling fix on current main: add verified `.m2a` audio classification to shared MIME/filename helpers, pass preserved original filenames into MIME fallback where inbound buffer saves need extension-based detection, update Telegram document placeholder/preflight handling where filenames indicate audio, and add targeted regression tests for shared media, Telegram inbound handling, voice-compatible audio, WebChat/UI maps, and any memory/media extension list intended to classify user audio files.
