## Clownfish Work Prompt

Implement `.m2a` audio support for OpenClaw media handling.

Observable bug: a Telegram document named like `voice.m2a` with generic or missing MIME is treated as `<media:document>` and does not reliably enter audio preflight/media-understanding paths. Current main also omits `.m2a` from shared audio filename detection, voice-compatible audio extension checks, and WebChat/UI audio maps.

Expected fix boundary:
- Add a verified `.m2a` extension/MIME mapping in `src/media/mime.ts` so filename-based detection and `isAudioFileName` recognize it as audio.
- Ensure `saveMediaBuffer` can use `originalFilename` as a `detectMime` filePath fallback when header/sniffing are missing or generic, while preserving current precedence for reliable sniffed/header MIME types.
- Update Telegram inbound document handling so document uploads with an audio filename/content type can produce audio placeholder/preflight behavior without hard-coding Telegram-specific rules in core.
- Add `.m2a` to voice-compatible/WebChat/UI audio maps only where it is compatible with the existing `.mp3`/`.m4a` treatment; document any intentionally skipped surface in tests or comments.
- Do not change Telegram SSRF/download policy, access control, media size limits, provider behavior, dependencies, or lockfiles.
- Add a single-line user-facing changelog entry under the active version if the repo expects one for this feature/fix, crediting `Thanks @mayonaissse`.

Related refs: this issue and closed-unmerged PR #23260.

Validation should include focused unit tests for shared MIME detection, `saveMediaBuffer` original filename fallback, Telegram document `.m2a` placeholder/preflight behavior, media-understanding attachment kind detection, voice-compatible audio classification, and WebChat/UI audio handling where updated.
