## Summary

Keep open. Current main still lacks an equivalent root execution guard, and the linked DigitalOcean/root-state failure mode still maps to the current CLI startup, state-dir, and systemd-user-unit behavior. The PR is a plausible core fix, but it needs maintainer review because the root refusal, override env var, container bypass, and DigitalOcean docs impact are product/security policy decisions.

## What This Changes

The PR adds a CLI root guard module and tests, then wires the guard into the modern and legacy CLI entrypoints to refuse uid/euid 0 unless an explicit root override or container-forwarded invocation applies.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer approval, with both CLI entrypoints guarded before mutable state/config side effects, real and effective UID covered, container-forwarded CLI behavior preserved, and DigitalOcean/root-shell guidance aligned with the chosen policy.
