## Summary

Keep open. Current main and the latest release still lack an equivalent root guard in the modern and legacy CLI entrypoints, while the linked DigitalOcean root-state corruption bug remains open and current state/systemd behavior still matches the report. The PR is active and focused, but root execution refusal is product-visible policy and needs maintainer review plus docs/install alignment before merge.

## What This Changes

The PR adds a CLI root guard module and tests, then wires it into `src/entry.ts` and `src/index.ts` to block real or effective UID 0 unless an explicit root override or container-forwarded CLI hints are present.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off: guard both CLI entrypoints before mutable state/config side effects, check both real and effective UID, preserve intentional container forwarding, and update DigitalOcean/1-Click guidance to match the chosen root-command behavior.
