## Summary

Keep open. Current main still lacks a root-execution guard, and this PR is a focused implementation for the root-state corruption reported in #67478, but it should not merge as-is because the public DigitalOcean/VPS docs still run `openclaw` commands from a root SSH session that the new guard would reject.

## What This Changes

The PR adds a CLI root guard with real/effective UID checks and explicit/container-forwarding bypasses, wires it into the modern and legacy CLI entrypoints, and adds unit coverage.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with both CLI entrypoints guarded before mutable state/config side effects, real and effective UID coverage retained, intentional container forwarding preserved, and DigitalOcean/1-Click guidance updated to the supported non-root or wrapper workflow.

## Reproduction Assessment

Yes. The linked bug has deterministic reproduction steps on a DigitalOcean 1-Click/root SSH flow, and current main code confirms root invocations still reach mutable CLI startup with state defaulting under the effective home directory.

## Solution Assessment

No, not yet as submitted. A core guard is a plausible narrow fix, but the PR needs matching install/docs policy or a narrower scope because current public DigitalOcean setup instructions still invoke `openclaw` from a root shell.
