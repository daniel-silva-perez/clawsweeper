## Summary

Keep open. The PR addresses a real linked bug and current main still has no root-execution guard, but the branch changes all root CLI behavior while the DigitalOcean/VPS docs still tell users to SSH as root and run `openclaw` commands, so this needs maintainer review or a docs/install compatibility update before merge.

## What This Changes

The PR adds a CLI root guard with real/effective UID checks, explicit/container-forwarding bypasses, modern and legacy entrypoint wiring, and unit coverage.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with matching install/docs changes: guard both CLI entrypoints before mutable state/config side effects, check both real and effective UID, preserve intentional container forwarding, and document the supported non-root service-user or wrapper workflow for DigitalOcean/1-Click users.

## Reproduction Assessment

Yes. The linked bug has deterministic DigitalOcean root-SSH reproduction steps, and current main confirms root invocations still reach CLI startup with mutable state defaulting under the effective home directory.

## Solution Assessment

No, not yet as submitted. A core root guard is a narrow plausible fix, but the PR should be paired with DigitalOcean install/docs policy updates or deliberately scoped so documented root setup commands do not start failing unexpectedly.
