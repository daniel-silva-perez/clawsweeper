## Summary

Keep this PR open. Current main still has no root guard in either CLI entrypoint, and the linked root-state corruption report remains valid; the proposed guard is focused, but it still needs maintainer review because it changes root-execution policy and currently conflicts with DigitalOcean docs that run OpenClaw from a root SSH shell.

## What This Changes

The PR adds a CLI root guard with real/effective UID checks, explicit root and container-forwarding bypasses, entrypoint wiring in `src/entry.ts` and `src/index.ts`, and unit tests.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with both CLI entrypoints guarded before mutable state/config side effects, real and effective UID coverage retained, intentional container forwarding preserved, and DigitalOcean/1-Click guidance updated to the chosen non-root or explicit-override workflow.

## Reproduction Assessment

Yes. The linked bug has concrete root-shell reproduction steps, and current main's entrypoints still reach `runCli()` without any UID/EUID guard while default state resolves to the invoking user's `~/.openclaw`.

## Solution Assessment

No, not yet. The core guard is a narrow and plausible fix for the corruption path, but the PR should not merge until the documented root-based DigitalOcean flow is reconciled or the guard is scoped to the intended 1-Click wrapper behavior.
