## Summary

Keep this issue open. It is a fresh, specific regression report against v2026.4.26, and current main still does not prove that the Discord native exec approval card/buttons are delivered for the reported embedded Discord setup. The source has native Discord approval runtime support, but the current gateway/native-runtime split can keep an approval pending while delivery silently returns no card, and the tests do not cover an end-to-end Discord-originated approval card send for the reporter's multi-account shape.

## Best Possible Solution

Keep the issue open and add a focused regression path for Discord native exec approvals. The best fix is to simulate a Discord-originated exec approval with configured approvers, including the reported multi-account case, and assert that the native runtime sends a Discord component card to the expected DM/channel or emits an explicit unavailable/fallback state instead of leaving only manual `/approve` as the working path.
