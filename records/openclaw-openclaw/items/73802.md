## Summary

Keep open. The report is fresh, specific to release v2026.4.26, and current main still does not prove that Discord native exec approval cards/buttons are delivered when an approval client exists. The code has a Discord native card path, but gateway acceptance is still based on the presence of an approval client or route, not on successful Discord message creation, and the native runtime can return zero delivered entries without surfacing a card or fallback. Related open issue #73954 reports the same no-card symptom plus malformed manual approval output, so this remains active work rather than conservative cleanup.

## Best Possible Solution

Keep this issue open and fix it in the Discord/native approval delivery path. Add a focused regression that simulates Discord-originated exec approvals with configured approvers, proves a Discord card/button delivery attempt is recorded, and makes empty delivery/no-entry cases visible as a fallback or failure instead of leaving the user with no card. If cross-account approval routing is intended, define that policy explicitly before changing account matching.
