## Summary

Keep open. Current main still skips Discord native approval delivery when the request is bound to a different Discord account than the enabled native approval account, and the fallback notice only covers runtimes that planned targets and then failed delivery. The reporter’s multi-account config and the follow-up comment still match an unresolved routing/UX gap rather than the stale Gateway-client fix from #74472.

## What This Changes

The issue requests restored Discord native exec approval cards/buttons for a multi-account Discord setup where approval IDs are created and manual `/approve` works but no native approval card appears.

## Best Possible Solution

Keep this open until maintainers define the Discord multi-account approval routing contract. If cross-account native approval delivery is supported, add an explicit authorization rule and focused regressions; if it is not supported, document origin-account-only native delivery and ensure the same-chat/manual fallback is reliable for this configuration.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure Discord native approvals on `xavier`, originate an approval request with `turnSourceChannel: "discord"` and `turnSourceAccountId` for another account, and current main predicts that the `xavier` native runtime skips delivery while manual `/approve` can still resolve through the approver path.

## Solution Assessment

Unclear. The current fallback and #74472 are useful adjacent fixes, but the requested native-card behavior needs a maintainer decision on whether Discord approvals may cross account boundaries and what authorization metadata must be preserved.
