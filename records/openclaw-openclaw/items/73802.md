## Summary

Keep open. Current main has Discord native approval plumbing and now has a same-chat manual fallback notice when native delivery reaches no targets, but that fallback is not proof that the expected Discord approval card/buttons are delivered. The source still allows Discord native delivery to return zero pending entries when message creation fails or account routing declines the request, while the gateway keeps the approval pending based on approval-client/route presence rather than confirmed Discord card delivery.

## What This Changes

The issue reports that Discord exec approvals in v2026.4.26 create approval IDs and accept manual `/approve` commands, but do not deliver the expected native Discord approval card/buttons.

## Best Possible Solution

Keep the issue open and fix the Discord native approval delivery contract: add a focused regression for Discord-originated exec approvals with configured approvers, assert successful native card/button delivery is recorded, and make no-entry delivery failures explicit. If cross-account approval routing is intended, define and test account selection before changing the matcher; keep manual fallback notices as a backup path, not the primary UX.

## Reproduction Assessment

Yes for a code-level reproduction path: simulate a Discord-originated exec approval with native approvals enabled, then force Discord delivery to return null or bind the request to a different Discord account and assert no native pending card entry is recorded while manual `/approve` remains possible. A live Discord reproduction was not run in this read-only review.

## Solution Assessment

No, relying on the current fallback notice is not the best complete solution because the documented primary UX is the native Discord card/buttons. The safer fix is a narrow Discord/native approval delivery regression plus an explicit account-routing policy, with fallback text only when delivery genuinely fails.
