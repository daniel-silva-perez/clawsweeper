## Summary

Keep open. Current main still accounts-gates Discord native approval delivery to the request's originating Discord account, and the post-report fallback only covers attempted native deliveries that planned targets but reached none. That does not prove native card/button delivery for the reported multi-account shape where approvals are enabled on one account but the request may originate from another.

## What This Changes

The issue asks OpenClaw to restore Discord native exec approval cards/buttons when approval IDs are created and manual `/approve` works but no native approval message appears.

## Best Possible Solution

Maintain a clear Discord multi-account approval contract. If cross-account approval surfaces are supported, implement and test delivery from a non-enabled origin account to the configured approver account; if not, document origin-account-only behavior and emit an explicit same-chat fallback instead of silently skipping native delivery.

## Reproduction Assessment

Yes, at source level: configure approvals only on `xavier`, originate an exec approval with `turnSourceAccountId` for another Discord account, and the current gate predicts that the `xavier` native runtime will skip card delivery while `/approve` can still authorize through configured approvers. A live Discord run would still be useful before landing a behavior change.

## Solution Assessment

Unclear. The current fallback mitigation is useful for failed delivery attempts, but it is not the full native-card solution requested here unless maintainers decide that origin-account-only delivery is the intended product behavior.
