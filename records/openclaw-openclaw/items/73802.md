## Summary

Keep open. Current main has a useful post-report mitigation for silent Discord approval failures, but it still does not prove the requested native Discord approval card/button delivery for the reported multi-account shape; the current native runtime can skip a request whose turn-source account differs from the enabled approval account.

## What This Changes

Discord exec approvals should deliver the native approval card/buttons when an approval is created, instead of requiring the operator to type `/approve` manually.

## Best Possible Solution

Keep the issue open until maintainers decide and encode the intended Discord multi-account approval routing contract. If cross-account approval surfaces are supported, add a focused regression that creates an approval from a non-enabled Discord account and proves the configured approver account receives a native card/button or an explicit same-chat fallback; if not, document origin-account-only behavior and make the fallback message clear.

## Reproduction Assessment

Yes, at source level: configure Discord approvals only on `xavier`, originate an exec approval from another Discord account/session, and the current account gate predicts no `xavier` native card attempt while manual `/approve` can still resolve through the approver path. I did not run a live Discord transport test.

## Solution Assessment

Unclear. The current fallback fix is a good mitigation for silent failures, but it is not the full native-card solution requested here unless maintainers decide that origin-account-only delivery is the intended behavior.
