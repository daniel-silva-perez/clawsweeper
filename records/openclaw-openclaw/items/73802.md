## Summary

Keep open. Current main still gates Discord native approval delivery to the request's source Discord account, so the reported multi-account setup can still create an approval that manual `/approve` accepts while the configured approval account does not get a native card/button. Changing that behavior needs maintainer/product approval because cross-account approval delivery exposes command text and approval controls across Discord accounts.

## What This Changes

The issue reports that Discord exec approvals are created and can be manually approved, but native Discord approval cards/buttons are not delivered for a v2026.4.26 multi-account Discord configuration.

## Best Possible Solution

Define the Discord multi-account approval contract first. If cross-account approval surfaces are supported, route native cards to the configured approver account with explicit authorization and regression tests; otherwise document origin-account-only native delivery and make the manual fallback clear for this configuration.

## Reproduction Assessment

Yes. At source level, configure native Discord approvals only on one account and create an approval request whose `turnSourceAccountId` is another account; the current account gate predicts no native card for the configured approval account while manual `/approve` remains authorized for the approver.

## Solution Assessment

Unclear. The existing fallback is useful, but the best native-card fix depends on whether maintainers want cross-account Discord approval fanout or stricter origin-account-only behavior.
