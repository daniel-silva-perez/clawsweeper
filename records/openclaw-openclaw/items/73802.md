## Summary

Keep open. Current main contains a fallback for failed Discord native approval delivery, but it still does not prove the requested native Discord card/button delivery for the reported multi-account configuration; the Discord native approval path remains account-gated and can skip requests whose source account differs from the enabled approval account.

## What This Changes

The issue reports that Discord exec approval IDs can be manually approved, but native Discord approval cards/buttons are not delivered in a multi-account Discord configuration.

## Best Possible Solution

Keep this open until maintainers define the Discord multi-account approval routing contract. If cross-account approval surfaces are supported, route the native card to the configured approver account with focused regressions; if not, document origin-account-only behavior and make the same-chat/manual fallback explicit for this configuration.

## Reproduction Assessment

Yes, at source level: configure Discord approvals only on xavier, originate an exec approval from another Discord account/session, and the current account gate predicts no xavier native card attempt while manual /approve can still resolve through the approver path. I did not perform a live Discord delivery run.

## Solution Assessment

Unclear. The current fallback mitigation is useful, but the best native-card fix requires maintainer agreement on cross-account delivery and authorization semantics before changing Discord fanout behavior.
