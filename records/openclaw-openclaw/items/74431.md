## Summary

Close as superseded by the earlier open #74254 implementation candidate for the same Docker apt GPG validation bug. Current main still needs a fix, but this branch is not the best candidate because it counts `fpr` records rather than primary `pub` records, which would reject Docker's current legitimate key that includes a subkey fingerprint.

## What This Changes

The PR changes the Dockerfile Docker CLI install block to collect all `fpr` records from the fetched Docker apt key, fail unless exactly one is present, compare it to `OPENCLAW_DOCKER_GPG_FINGERPRINT`, and add a changelog entry.

## Best Possible Solution

Consolidate on #74254 or an equivalent narrow fix that counts exactly one primary `pub` key, permits legitimate subkeys, verifies the pinned Docker fingerprint, and adds focused Dockerfile coverage plus Docker build proof.

  - command: `zcat /usr/share/doc/gnupg/DETAILS.gz | sed -n '40,95p;110,150p;155,180p'`
- **Live Docker key shape:** A live read of Docker's Debian apt key returned one primary `pub`, one `sub`, and two `fpr` records, with the primary fingerprint matching the pinned value; this branch's count-one-`fpr` guard would fail that legitimate key.
  - command: `curl -fsSL https://download.docker.com/linux/debian/gpg | gpg --batch --import-options show-only --import --with-colons 2>/dev/null | awk -F: '$1 == "pub" { pub_count++ } $1 == "sub" { sub_count++ } $1 == "fpr" { fpr_count++; print "fpr=" toupper($10) } END { print "pub_count=" pub_count; print "sub_count=" sub_count; print "fpr_count=" fpr_count }'`
