## Summary

Keep open: this PR targets a real Docker apt key validation gap that is still present on current main, but the proposed count gate counts `fpr:` records instead of primary keys or armor blocks. Live inspection of Docker's current Debian apt key shows one primary key, one subkey, and two `fpr:` records, so the patch would reject legitimate builds that enable Docker CLI installation. The linked bug remains open and #74254 appears to be a parallel fix path, so this needs maintainer/security review rather than cleanup closure.

## What This Changes

The PR changes `Dockerfile` to collect all fingerprints from the fetched Docker apt signing key, fail unless the count is one, compare the result to `OPENCLAW_DOCKER_GPG_FINGERPRINT`, and adds a changelog entry for #74234.

## Best Possible Solution

Land one security-reviewed Dockerfile fix that verifies the fetched Docker apt file contains exactly one primary public key or one armored key block, compares the primary key fingerprint to `OPENCLAW_DOCKER_GPG_FINGERPRINT`, and only then dearmors it for apt. The maintainer path should either update this PR or choose #74254 as the canonical fix, with changelog credit kept aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Count primary keys before enforcing the single-key guard:** `Dockerfile:248-253`
  - body: `gpg --with-colons` emits an `fpr:` row for the primary key and another for each subkey. Docker's current Debian key has one `pub:` row but two `fpr:` rows, so this guard rejects the legitimate key whenever `OPENCLAW_INSTALL_DOCKER_CLI` is enabled. Count `pub:` records or armor headers, then compare the primary fingerprint instead of the raw `fpr` list.
  - confidence: 0.97
