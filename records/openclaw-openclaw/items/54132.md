## Summary

Keep open. Current main still lacks the requested Sessions-to-history shortcut, but this PR is not merge-ready because its standalone transcript viewer route would expose session history outside the established scoped history API and does not fully match current session-store/artifact behavior.

## What This Changes

The PR adds a Sessions-page History link and subtitle, standalone Control UI `/history` and `/history.js` transcript viewer routes with focused tests, and a `.vs/` gitignore entry.

## Best Possible Solution

Land a reworked history navigation design that keeps the Sessions shortcut basePath-safe, reuses the authenticated `/sessions/<key>/history` API or an equally scoped endpoint, respects configured session stores and archive artifacts, and includes focused gateway/UI tests for auth, basePath, transcript loading, and large-file boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate history transcripts behind gateway auth:** `src/gateway/control-ui.ts:551-552`
  - body: `/history.js` is served from the Control UI handler before the existing bootstrap authorization path, and it serializes local transcript files into JavaScript. Any client that can reach the static Control UI surface can read session history without the `operator.read` gate used by `chat.history` and `/sessions/<key>/history`, so transcript access needs to move behind scoped gateway auth or reuse the existing history endpoint.
  - confidence: 0.95
- **[P2] Resolve transcript history from configured stores:** `src/gateway/control-ui.ts:121-122`
  - body: `loadSessionHistoryFiles` uses `resolveSessionTranscriptsDirForAgent(agentId)`, which ignores `session.store` and the store lookup semantics used by `sessions.list`, `sessions.preview`, and `/sessions/<key>/history`. Deployments with a custom or discovered session store can show empty or wrong history even though the supported APIs resolve the correct store from config.
  - confidence: 0.9
- **[P2] Size-check transcripts before reading them:** `src/gateway/control-ui.ts:137-138`
  - body: The byte cap is applied after `fs.readFileSync(filePath)`, so a single oversized transcript is still loaded synchronously before the cap can exclude it. That defeats the intended 50 MB protection and can block the event loop or spike memory on large logs.
  - confidence: 0.88
- **[P2] Include archived transcript artifacts:** `src/gateway/control-ui.ts:129`
  - body: Filtering with `file.endsWith(".jsonl")` drops archive names such as `*.jsonl.reset.<timestamp>` and `*.jsonl.deleted.<timestamp>`, which current session artifact helpers create when sessions are reset or deleted. The viewer can therefore miss archived logs that users expect a history page to show.
  - confidence: 0.86
- **[P2] Keep history routes behind Control UI root checks:** `src/gateway/control-ui.ts:551-552`
  - body: The new dynamic routes are handled before the existing invalid/missing Control UI root checks. Environments where Control UI assets are unavailable can now return transcript data with 200 instead of the prior assets-unavailable response, which widens the data exposure path.
  - confidence: 0.84
