## Summary

Keep this PR open. Current main still does not implement direct `openclaw gateway restart` restart-sentinel notification, and the PR's current v2 approach writes the sentinel after `runServiceRestart` returns, which can miss the restarted gateway's startup sentinel check.

## Best Possible Solution

Keep this PR open or replace it with a maintainer patch that builds the CLI restart sentinel after preflight/no-op decisions but before the actual service restart or unmanaged SIGUSR1 emission, passes the daemon service environment when resolving the sentinel path, removes the sentinel if restart emission/restart fails, and adds focused tests for managed, unmanaged, no-op, failure, scheduled, and custom-state restart paths.

## Risks / Open Questions

- The PR's post-`runServiceRestart` sentinel write can miss the restarted gateway's startup sentinel check and leave the notice for a later unrelated startup.
- A sentinel written with the invoking CLI environment can land in the wrong state directory for service installs that use `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`.
- The PR has no focused tests for managed restart, unmanaged SIGUSR1 restart, no-op restart, restart failure cleanup, scheduled restart, or daemon-environment path resolution.
