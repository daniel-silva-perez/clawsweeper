## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement the narrow CLI restart-sentinel behavior for `openclaw gateway restart`, and the PR is not obsolete; however, the current v2 patch writes the sentinel after `runServiceRestart` returns, which can miss the gateway startup check that consumes/schedules restart sentinel delivery.

## Best Possible Solution

Keep this PR open or replace it with a maintainer patch that builds the CLI restart sentinel after config/no-op preflight but before the actual service restart or unmanaged SIGUSR1 emission, resolves the sentinel path using the daemon service environment when available, removes the sentinel if restart emission or scheduling fails, and adds focused lifecycle tests for managed, unmanaged, no-op, failure, scheduled, and custom-state-dir paths.

## Risks / Open Questions

- Post-`runServiceRestart` sentinel writes can miss the restarted gateway's startup sentinel check and leave the notification for a later unrelated startup.
- Writing with the invoking CLI environment can put `restart-sentinel.json` in the wrong state directory for service installs that use `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`.
- The PR has no focused tests for managed restart, unmanaged SIGUSR1 restart, no-op restart, restart failure cleanup, scheduled restart, or daemon-environment path resolution.
