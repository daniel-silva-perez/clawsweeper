## Summary

Keep this PR open. Current main still does not write a restart sentinel for `openclaw gateway restart`, but the supplied patch is not safe to close as obsolete or land as-is because it writes the sentinel after the restarted gateway may already have skipped startup sentinel scheduling, and it does not resolve the sentinel path with the daemon service environment.

## Best Possible Solution

Keep this PR open or replace it with a maintainer patch that performs restart no-op/config preflight first, resolves the sentinel path with the daemon service environment when available, writes the sentinel before managed restart scheduling or unmanaged SIGUSR1 emission, removes it if scheduling/emission/restart fails, and adds focused lifecycle tests for managed, unmanaged, not-loaded/no-op, failure cleanup, scheduled restart, and custom state-dir paths.

## Risks / Open Questions

- Landing the current patch as-is can miss startup sentinel scheduling because the sentinel is written after `runServiceRestart` returns.
- The patch can write `restart-sentinel.json` into the invoking CLI state directory instead of the daemon service state directory when `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` differs.
- Closing this PR would drop a plausible user-visible CLI gap because current main still lacks CLI restart notification behavior.
