## Summary

Keep open: the patch does not resolve the linked gateway pairing bypass and it expands the bypass to bootstrap-token handshakes, which touches a security-sensitive authentication boundary.

## What This Changes

The PR adds `bootstrap-token` as an auth method that can skip local backend self-pairing and adds helper-level expectations/comments around that bypass.

## Best Possible Solution

Revise the gateway auth design so backend self-pairing is bound to a server-verifiable backend identity or a dedicated credential, and keep bootstrap-token handshakes on the existing bootstrap pairing/handoff path unless maintainers explicitly approve a broader backend exception with regression coverage.

## Reproduction Assessment

Yes. The helper-level path is enough to reproduce the reported condition: a loopback backend `gateway-client` with no Origin, shared token/password auth, and `sharedAuthOk: true` makes the self-pairing skip true, and the message handler uses that skip to preserve self-declared scopes.

## Solution Assessment

No. The proposed patch is not the best fix because it leaves the shared-token/password bypass intact and adds bootstrap-token to the same bypass condition instead of tightening the backend identity check.
