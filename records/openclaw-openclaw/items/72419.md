## Summary

Keep open for maintainer/security review. The PR is paired with open issue #72418 using closing syntax, and the proposed diff does not remove the shared-token/password backend self-pairing bypass that the linked report challenges; it adds `bootstrap-token` as another bypass method instead.

## What This Changes

The PR adds `bootstrap-token` as a valid auth method for `shouldSkipLocalBackendSelfPairing` and adds helper-level tests/comments around the backend self-pairing bypass.

## Best Possible Solution

Keep this PR open and have maintainers decide the gateway trust model explicitly: either revise the fix to remove or tightly bind the shared-token/password backend self-pairing bypass to a server-verifiable backend identity or reject the linked report with documented security rationale. Bootstrap-token behavior should stay aligned with the existing bootstrap pairing/handoff path unless a maintainer approves a broader backend exception with regression coverage.

## Reproduction Assessment

Yes. The source and helper tests provide a high-confidence reproduction path: a loopback `gateway-client`/`backend` connection with no browser Origin, shared token/password auth, and `sharedAuthOk: true` makes `shouldSkipLocalBackendSelfPairing` return true; the message handler then uses that skip to avoid clearing self-declared scopes.

## Solution Assessment

No. This is not the best fix because it leaves the reported shared-secret backend bypass intact and adds another credential type to the bypass; a safer solution would bind backend bypass to a distinct server-side identity or dedicated credential and add an end-to-end regression for the spoofed backend client path.
