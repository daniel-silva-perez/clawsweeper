## Summary

Keep open. The issue discussion narrowed the symptom away from compaction toward restart-correlated persistence loss, and current main still leaves ordinary non-forum Telegram group messages on a no-last-route path where the inbound metadata write is background-only. The linked fix PR was closed unmerged, and no later current-main change touched the relevant Telegram/core session files.

## What This Changes

The issue requests durable persistence of Telegram group chat session history so multi-day group conversations do not collapse to a single short stored session after gateway restarts.

## Best Possible Solution

Implement a narrow Telegram/core session change that makes ordinary non-forum Telegram group inbound session metadata durable before dispatch continues, while preserving one session per regular group, separate forum-topic sessions, and the intentional no-last-route behavior for bound ACP topic routes.

## Reproduction Assessment

Unclear for the full live restart loss sequence because this read-only pass did not run Telegram or gateway restart tests. The source-level path is clear: a regular non-forum Telegram group message has no resolved thread id, produces no updateLastRoute, and returns from recordInboundSession without awaiting the metadata write.

## Solution Assessment

Unclear whether the closed PR's updateLastRoute-only approach is the best complete fix. A safer repair is to add regression coverage and either provide a safe regular-group route update or await the no-route metadata path without changing documented group/topic session keys.
