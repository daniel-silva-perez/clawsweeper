## Clownfish Work Prompt

Fix the Telegram regular non-forum group session persistence gap described by this issue. Current main builds `updateLastRoute` only for DMs or group messages with a resolved forum topic thread, while `recordInboundSession()` fire-and-forgets `recordSessionMetaFromInbound()` when `updateLastRoute` is absent. Implement the smallest safe fix so ordinary non-forum Telegram group inbound session metadata is durable before dispatch/restart-like store reloads can lose it. Likely approaches: safely supply a group `updateLastRoute` for regular groups, or make the no-route metadata write awaited in the shared session helper if that is the correct generic contract. Add regression coverage proving repeated regular group messages with no forum topic keep the same session key and persist metadata/history across a store reload or deleted/missing-entry recreation path. Preserve existing semantics: regular groups remain one session (`agent:<agentId>:telegram:group:<id>`), forum topics remain topic-scoped, binding-channel routes must not be overwritten, DM `mainDmOwnerPin` stays DM-only, and Telegram auth/group allowlist policy must not change. Include a user-facing changelog fix entry with reporter credit if behavior changes.

## Risks / Open Questions

- The reporter expected multiple sessions, but current docs define one session per regular group; a fix should preserve that contract and address missing persisted history instead of creating per-message or per-day group sessions.
- updateLastRoute is delivery metadata, while transcript JSONL persistence happens through the agent/session runtime; a focused fix should include a regression that proves repeated regular group turns survive store reload/restart-like behavior, not only that lastRoute fields are populated.
- Changing regular group last-route behavior can affect outbound fallback delivery for group sessions, especially for root groups versus forum topics and ACP-bound routes.
