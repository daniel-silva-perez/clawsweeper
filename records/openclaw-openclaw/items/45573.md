## Summary

Keep open. Current main still leaves regular non-forum Telegram group messages without a last-route update, so inbound session metadata remains on the background path; the reporter’s restart-correlated persistence loss is still plausible and the linked fix PR is closed unmerged.

## What This Changes

The issue requests durable Telegram group chat session persistence so group conversations survive gateway restarts and appear correctly in session listing/history tools.

## Best Possible Solution

Land a focused Telegram/core session fix that makes regular non-forum group inbound session metadata durable before dispatch continues while preserving one session per regular group, separate forum-topic sessions, and intentional bound-topic route behavior.

## Reproduction Assessment

Unclear for the full live Telegram restart data-loss sequence. The current-main code path that makes the report plausible is reproducible with a regular non-forum Telegram group message: no resolved topic means no updateLastRoute, and recordInboundSession returns without awaiting the metadata write.

## Solution Assessment

Unclear whether the old PR’s updateLastRoute-only approach is the complete fix. The safer solution is a narrow regression-tested change that either supplies a safe regular-group route update or awaits the no-route metadata path without changing session-key semantics.

## Repair Work Prompt

Fix the Telegram group session persistence issue. Regular non-forum Telegram group messages currently resolve no forum thread id, so `extensions/telegram/src/bot-message-context.session.ts` leaves `updateLastRoute` undefined and `src/channels/session.ts` does not await `recordSessionMetaFromInbound()` before dispatch continues. Make regular Telegram group inbound session metadata durable on the critical path, either by safely supplying a group `updateLastRoute` with `to: telegram:<chatId>` and no thread id, or by changing the no-route metadata path so it is awaited without regressing routes. Preserve the documented session model: one session per regular group, separate `:topic:<threadId>` sessions for forum topics, `mainDmOwnerPin` remains DM-only, and configured/bound ACP topic routes that intentionally skip last-route updates must keep their behavior. Add regression coverage for repeated regular non-forum group messages, with and without Telegram `message_thread_id`, proving the same session key is used and session metadata is persisted/awaited across a store reload or restart-like boundary. Keep the fix inside Telegram/core session surfaces and do not change unrelated channel behavior. Add a single-line user-facing changelog entry under the active version with `Thanks @kiki830621`.
