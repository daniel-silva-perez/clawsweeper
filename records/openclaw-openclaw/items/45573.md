## Summary

Keep open. Current main still leaves ordinary non-forum Telegram group messages without a last-route update, so inbound session metadata is not kept on the awaited dispatch path; the reporter's restart-correlated data loss remains plausible and the linked fix PR is closed unmerged.

## What This Changes

The issue requests durable persistence for Telegram group chat session history so group conversations survive gateway restarts and appear correctly in session listing/history tools.

## Best Possible Solution

Land a focused Telegram/session persistence fix that makes regular non-forum group inbound session metadata durable before dispatch continues while preserving the documented one-session-per-group model and separate forum-topic sessions.

## Reproduction Assessment

Unclear for the full live Telegram restart data-loss sequence, because this was a read-only review. The current-main code path that makes the report plausible is reproducible by constructing a regular non-forum Telegram group message: no resolved topic means no updateLastRoute, and recordInboundSession returns without awaiting the metadata write.

## Solution Assessment

Unclear whether enabling updateLastRoute alone is the whole fix. The safer solution is a narrow, regression-tested change that either supplies a safe regular-group route update or awaits the no-route metadata path, without changing Telegram session-key semantics or bound topic behavior.
