## Summary

Keep open. Current main still uses a module-local sandbox backend registry while plugin-facing registration is exported through the SDK and core resolves sandboxed turns through that registry; the same-author implementation candidate remains open and unmerged, so this should stay with maintainer/security review rather than be swept closed.

## What This Changes

A custom sandbox backend plugin registered during startup should remain visible to `requireSandboxBackendFactory` when a sandboxed TUI turn resolves `agents.defaults.sandbox.backend`.

## Best Possible Solution

Keep this issue open until #67623 or a maintainer-authored equivalent moves sandbox backend registration to a process-shared registry appropriate for plugin-loaded module instances and adds a regression test proving SDK-side registration is visible to core sandbox context resolution.

## Reproduction Assessment

Yes. The report gives a concrete integration path with a custom backend id, and current source still follows a module-local registry lookup that can lose plugin registrations across module instances; a focused regression test can simulate that split-module registration path.

## Solution Assessment

Yes on the direction, but not yet resolved on main. A shared backend registry plus split-module regression coverage is the narrow maintainable fix; because this is sandbox code, the open implementation candidate still needs maintainer and secops review.
