## Summary

Keep open. Current main still stores sandbox backend registrations in a module-local map while plugin-facing registration is exported through the SDK and core later resolves through that same registry lookup; the same-author implementation candidate remains open and unmerged, so cleanup should not close this before maintainer review.

## What This Changes

A plugin-registered sandbox backend should be discoverable by core `requireSandboxBackendFactory` when a sandboxed TUI or agent turn resolves the configured backend after gateway startup.

## Best Possible Solution

Keep this issue open and review #67623, or a maintainer-authored equivalent, to move sandbox backend registrations to a process-shared registry appropriate for mutable live state and add a regression test proving SDK-side plugin registration is visible to core sandbox context resolution.

## Reproduction Assessment

Yes. The report gives a concrete integration path: install a native sandbox backend plugin, configure `agents.defaults.sandbox.backend` to that id, restart the gateway, and send a sandboxed TUI turn; current source still follows the module-local registry path that can lose plugin registrations across module instances.

## Solution Assessment

Yes on direction, not yet on the open PR as-is. A shared registry plus split-module regression test is the narrow maintainable fix, but the candidate PR still needs maintainer and security-owner review before this issue is closed.
