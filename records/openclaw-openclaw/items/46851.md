## Summary

Keep open. Current main has adjacent liveness features, but not the requested configurable fixed-interval visible heartbeat with started/running/done/failed lifecycle, per-channel overrides, min duration, and rate-limit fallback semantics. The open related status-surface issue is adjacent rather than a clear duplicate because it explicitly distinguishes a durable editable status surface from fixed-interval heartbeat messages.

## Best Possible Solution

Keep this issue open unless maintainers explicitly fold it into #52640. The best implementation path is a core/channel-runtime opt-in progress-heartbeat contract layered on existing reply lifecycle events, with per-channel capability handling, min duration, fixed cadence, terminal cleanup, and rate-limit fallback behavior documented in the channel/streaming docs.
