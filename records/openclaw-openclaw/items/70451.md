## Summary

Keep open. Current main still routes the hooks CLI through full plugin diagnostics/runtime loading after command bootstrap, so the Windows/Node plugin-loading timeout remains a plausible hooks-specific bug even though the broader startup report was closed and newer Windows plugin-load performance work landed.

## What This Changes

Fix Windows timeouts for `openclaw hooks` commands by avoiding deadlocking plugin registry/runtime loading while still showing and toggling hook availability correctly.

## Best Possible Solution

Keep the issue open and make `openclaw hooks list/info/check/enable/disable` resolve plugin-managed hook visibility from manifest/config/workspace snapshot data or a bounded hook-specific metadata helper. The command should not import full plugin runtime modules merely to list or toggle hooks, and tests should lock that startup boundary down.

## Reproduction Assessment

Unclear for the live SIGKILL: no native Windows/Node 24 run was performed here. The source-level reproduction path is clear, though: every affected hooks command reaches `buildHooksReport`, which still calls full plugin diagnostics with module loading enabled.

## Solution Assessment

Yes, the narrow maintainable fix is to move the hooks CLI report path to metadata/snapshot data while preserving plugin-managed hook display and toggle restrictions. A broad Windows plugin-loader speedup alone is not the best terminal fix because the CLI control plane should not depend on runtime plugin imports.
