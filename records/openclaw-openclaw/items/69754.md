## Summary

Keep open. Current main has addressed the adjacent Feishu native-button problem through shared interactive payload rendering, but this issue is narrower: degraded approval copy still does not consistently explain why Feishu buttons are unavailable or give a clean exact fallback path, and the Feishu docs still do not document approval fallback behavior.

## What This Changes

The issue asks for clearer Feishu approval fallback guidance when interactive approval buttons are unavailable or approval falls back to manual text.

## Best Possible Solution

Keep the shared Feishu native-card implementation, then add narrow approval-specific fallback copy and docs so degraded Feishu surfaces explain why buttons are unavailable, show exact `/approve` commands, and identify whether approval can be completed in-channel or must move to Web UI/terminal UI.

## Reproduction Assessment

Yes. A source-level reproduction is available by tracing Feishu `sendPayload` to the document-comment fallback path and the shared forwarded approval text; both show degraded text behavior without the Feishu-specific fallback explanation requested here.

## Solution Assessment

Yes, the issue is best solved with narrow copy/rendering/docs changes around the existing shared approval and Feishu outbound paths, not by adding a new Feishu approval system or resurrecting a legacy `inlineButtons` config.
