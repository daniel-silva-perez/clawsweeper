## Summary

Keep open. Current main still prompts for `Where will the Gateway run?` before it reaches any `--section model` handling, so the reported configure path can still block before model/auth setup. Direct `openclaw models auth ...` commands are valid workarounds, but they do not fix the configure wizard path itself.

## Best Possible Solution

Keep this open and fix the configure path itself. The narrow code direction is to let section-filtered model/auth configuration reach auth setup without first requiring the gateway-location select when the chosen sections do not need gateway, daemon, or health work, plus an early actionable non-interactive TTY guard that points users to direct `openclaw models auth` commands. Update `docs/cli/configure.md` so direct auth commands remain documented as workarounds, not as the closing fix.

## Clownfish Work Prompt

Fix the configure-flow bug where `openclaw configure --section model` can be blocked by the initial `Where will the Gateway run?` prompt before model/auth setup. Keep the PR narrow: for section-filtered runs, skip the gateway-location select when the selected sections do not include `gateway`, `daemon`, or `health`, while preserving existing bare `openclaw configure` behavior. Ensure model/auth setup still persists needed config metadata safely. Add an early actionable guard for clearly non-interactive stdin/stdout before the first Clack prompt, pointing users to `openclaw models auth login --provider <id>`, `openclaw models auth setup-token --provider anthropic`, `openclaw models auth paste-token --provider anthropic`, and https://docs.openclaw.ai/cli/models. Do not reintroduce or invent an Anthropic browser OAuth flow, do not change provider auth internals, and do not broaden gateway config semantics beyond the configure wizard entry path. Update `docs/cli/configure.md` to describe when the gateway-location prompt appears and keep direct model-auth commands documented as workarounds. Add or update tests proving `sections: ["model"]` reaches `promptAuthConfig` without consuming the gateway-location select, while bare configure still prompts for gateway location. Add one user-facing changelog Fixes entry thanking @LevityLeads if behavior changes.
