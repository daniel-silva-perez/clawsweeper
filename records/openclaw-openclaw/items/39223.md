## Summary

Keep open. Current main still runs the Gateway-location Clack select before any `--section model` handling, so the reported configure path can still block before model/auth setup. The documented `openclaw models auth ...` commands are useful workarounds, but they do not fix the configure wizard path itself.

## Best Possible Solution

Keep the issue open and fix the configure path itself. The best narrow product/code direction is to make section-filtered model/auth configuration able to reach auth setup without first requiring the Gateway-location select, and to add an early, actionable TTY/non-interactive guard that points users to `openclaw models auth login/setup-token/paste-token` when the interactive wizard cannot run. Update `docs/cli/configure.md` so direct auth commands remain documented as workarounds, not as the closing fix.

## Clownfish Work Prompt

Fix the open configure-flow bug where `openclaw configure --section model` can be blocked by the initial `Where will the Gateway run?` prompt before model/auth setup. Keep the PR narrow: make section-filtered model/auth configuration able to proceed without first requiring the Gateway-location select when the selected sections do not need Gateway/daemon/health work. Preserve the existing full `openclaw configure` behavior unless the change is explicitly needed for the bug. Add an early actionable guard for clearly non-interactive stdin/stdout before the first Clack prompt, pointing users to `openclaw models auth login --provider <id>`, `openclaw models auth setup-token --provider anthropic`, or `openclaw models auth paste-token --provider anthropic` and https://docs.openclaw.ai/cli/models. Do not reintroduce or invent an Anthropic browser OAuth flow, do not change provider auth internals, and do not broaden Gateway config semantics beyond the configure wizard entry path. Update `docs/cli/configure.md` to describe when the Gateway-location prompt appears and keep direct model-auth commands documented as workarounds.
