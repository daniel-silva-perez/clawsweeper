## Summary

Keep open. Current main has partial overlap with this PR, especially lazy primary command registration and skipping plugin CLI registration for `plugins --help`, but it does not implement the PR’s central `openclaw <subcommand> --help` fast path or metadata-only plugin help path. `memory --help` still reaches the full plugin command loader and imports the full `memory-core` runtime entry.

## Best Possible Solution

Keep this PR open for maintainer/contributor rework rather than auto-closing. The best path is to rebase onto current main’s lazy primary-command and manifest-activation loaders, add a narrowly scoped metadata-only help path where it is compatibility-safe, keep memory runtime registration synchronous or explicitly awaited by a supported loader contract, preserve pairing channel help data, and add regression tests for `memory --help`, runtime-only third-party plugin help fallback, and pairing help channel descriptions.

## Risks / Open Questions

- The PR head, as described in review comments, could make memory runtime registration race or disappear if `memory-core.register()` becomes async while `loadOpenClawPlugins()` requires synchronous registration.
- A blanket metadata-only help path can regress third-party plugins that expose CLI commands only from their normal runtime entry.
- Skipping plugin loading for pairing help can make `--channel` help text incomplete because `registerPairingCli()` snapshots `listPairingChannels()` at registration time.
- The provided PR file list does not show workflow, lockfile, dependency-source, release-script, or package-publishing changes; the security pass did not find a supply-chain surface, but the plugin loader behavior change is still runtime-sensitive.
