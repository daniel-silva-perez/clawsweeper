## Summary

Keep open. Current main still emits numeric-only `nextAt`/`nextRunAtMs` values in the relevant cron timer, retry/backoff, and event surfaces, and #58908 is an open unmerged PR with closing syntax for this issue, so the issue should remain open until that implementation or an equivalent one lands.

## What This Changes

Request to add a human-readable companion timestamp for cron scheduler `nextAt`/`nextRunAtMs` values while preserving backward-compatible numeric fields.

## Best Possible Solution

Land #58908 after review and refresh, or make an equivalent narrow additive change that keeps the numeric fields stable, adds ISO-8601 companion fields to the cron timer/retry/event payloads, and locks the logging seam with focused tests.

## Reproduction Assessment

Yes. A high-confidence read-only reproduction path exists: arm any enabled cron job with debug logging and inspect the `cron: timer armed` payload; the current source shows it will contain numeric `nextAt` only.

## Solution Assessment

Yes, the additive companion-field approach is the safest maintainable direction because it preserves existing numeric consumers while improving operator readability. The open PR follows that shape but still needs maintainer review and refresh before it can resolve the issue.
