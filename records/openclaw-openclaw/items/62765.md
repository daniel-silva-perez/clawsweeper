## Summary

Current main still leaves the core MS Teams pairing UX bug unresolved: unpaired DM senders are recorded and the message is dropped, but no in-band Teams pairing challenge is sent and the structured log omits the code. The linked fix PR proposed the right shared-helper approach, but it was closed unmerged and its changes are not present in the checked-out main.

## What This Changes

The issue asks for MS Teams `dmPolicy: "pairing"` DMs from unpaired senders to log and send the generated pairing challenge instead of silently dropping the message behind HTTP 200.

## Best Possible Solution

Update the MS Teams DM pairing rejection path to use the existing shared pairing challenge helper, preserving the stored conversation reference and drop semantics while sending the standard Teams reply on first contact and logging sender, label, and code. Extend the focused authz test so this cannot regress back to a store-only silent drop.

## Reproduction Assessment

Yes. The current-main code path for a personal MS Teams message with `dmPolicy: "pairing"` and an unapproved sender reaches `pairing.upsertPairingRequest` and returns before any pairing reply; the existing unit harness can reproduce this by mocking `sendActivity` and the pairing-store result.

## Solution Assessment

Yes. Reusing `pairing.issueChallenge` is the narrowest maintainable fix because the shared helper already implements create-if-missing, first-created-only replies, code-bearing creation callbacks, and reply-error handling used by other channel plugins.
