## Summary

Keep this issue open. Current main loads local skills for runtime/status/list surfaces, but the requested registration or search-index behavior is still missing: `skills.search`, `openclaw skills search`, and the Control UI discovery path remain ClawHub-only, and there is no `skills.register` or `openclaw skills register <path>` surface.

## Best Possible Solution

Keep open. The best implementation is a core-owned skill discovery change: either extend `skills.search` and `openclaw skills search` to return source-aware local and ClawHub results backed by the existing local skill loaders, or add an explicit `skills.register` / `openclaw skills register <path>` flow. It should update the gateway protocol, CLI, Control UI, docs, and tests while keeping ClawHub install/update semantics distinct from local skill paths.

## Risks / Open Questions

- A combined local-plus-ClawHub search needs an explicit source discriminator so local skill names or paths are not treated as ClawHub slugs for install, update, or detail flows.
- A register or auto-index flow must preserve current realpath confinement, symlink/path trust boundaries, skill precedence, agent allowlists, and installer safety semantics.
