## Summary

Keep open. Current main still loads local skills for runtime/status/list surfaces, and Skill Workshop can create workspace skills with hot refresh, but the requested registration/search-index behavior is not implemented. Gateway `skills.search`, CLI `openclaw skills search`, and the Control UI discovery path remain ClawHub-only, with no `skills.register` or `openclaw skills register <path>` surface.

## Best Possible Solution

Keep open. The best implementation is a core-owned, source-aware skill discovery change: extend `skills.search` and `openclaw skills search` to include local results from the existing local skill loaders, or add an explicit `skills.register` / `openclaw skills register <path>` flow. It should update the gateway protocol, CLI, Control UI, docs, and tests while keeping ClawHub install/update/detail semantics distinct from local skill paths.

## Risks / Open Questions
