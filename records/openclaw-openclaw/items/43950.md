## Summary

Keep this issue open. Current main loads local skills for runtime/status surfaces, but the requested registration/search behavior is still missing: `skills.search`, the CLI search command, and the Control UI discovery flow remain ClawHub-only, and there is no `skills.register` API or `openclaw skills register <path>` command.

## Best Possible Solution

Keep this open. The best implementation is a core-owned skill discovery change: extend `skills.search` and `openclaw skills search` to return source-aware local and ClawHub results backed by the existing local skill loaders, or add an explicit `skills.register` / `openclaw skills register <path>` flow. It should update the gateway protocol, CLI, Control UI, docs, and tests while keeping ClawHub install/update semantics distinct from local skill paths.

## Risks / Open Questions

- A combined local-plus-ClawHub search needs a source discriminator so local names or paths are not treated as ClawHub slugs for install, update, and detail flows.
- A register or auto-index flow should preserve current skill precedence, agent allowlists, realpath confinement, symlink/path trust boundaries, and installer safety semantics.
- Maintainers still need to choose the product shape: extend `skills.search`, add an explicit `skills.register` flow, or expose a separate local search surface.
