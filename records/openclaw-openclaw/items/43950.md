## Summary

Current main still does not support registering local skills into the skill search index. Local/workspace skills are loaded for runtime, status, list, info, dependency checks, and prompt snapshots, but the gateway `skills.search`, CLI `openclaw skills search`, protocol schema, docs, and Control UI search path remain ClawHub-only with no local source/path/register shape.

## Best Possible Solution

Keep this issue open. The best implementation is core-owned and source-aware: either extend `skills.search`, `openclaw skills search`, and the Control UI to include local/workspace/managed results backed by `loadWorkspaceSkillEntries`, or add an explicit `skills.register` / `openclaw skills register <path>` flow. The change should update the gateway protocol schema, CLI/UI behavior, docs, and tests while preserving existing ClawHub install/update semantics.

## Risks / Open Questions

- A combined local-plus-ClawHub search should add an explicit source discriminator so local skill names are not treated as ClawHub slugs for install/update flows.
- A local register/index path needs clear path trust boundaries, symlink handling, precedence behavior, agent allowlist interaction, and compatibility with existing workspace/managed skill loading.
