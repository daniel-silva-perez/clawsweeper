## Summary

Keep open. The user-facing GitHub Copilot device-flow problem still exists on current main, and the PR has the right intent but is not merge-ready: current plugin-owned login code still polls before any interval delay and handles `slow_down` with a fixed offset, while the PR targets an obsolete `src/providers` path and does not wire the new `--wait` option through the current provider-auth flow.

## What This Changes

The PR changes GitHub Copilot device-flow polling/backoff behavior and adds a `login-github-copilot --wait` CLI flow that prints copyable URL/code text before polling.

## Best Possible Solution

Port the behavior into the current plugin-owned GitHub Copilot auth implementation: update `extensions/github-copilot/login.ts` to schedule polling from the issued-code timestamp, honor GitHub's `slow_down` interval, expose a typed/direct-login `--wait` option through the existing provider auth context, document the flag, and cover the timing with fake-timer/fetch tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main performs the first token request before sleeping and the direct CLI does not accept `--wait`; a focused fake-fetch/fake-timer unit test can reproduce both the immediate poll and `slow_down` backoff behavior without live GitHub credentials.

## Solution Assessment

No. The polling/backoff direction is correct, but the patch is not the best current implementation because it edits the old core provider file and does not reach the plugin-owned login path; a narrow port to the current extension and provider-auth option plumbing is safer.
