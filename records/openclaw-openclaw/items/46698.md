## Summary

Keep open. The PR targets a real GitHub Copilot device-flow/DX problem, but current main has moved the active implementation into the bundled `github-copilot` plugin, while the PR edits an obsolete core-era provider file and does not wire the new CLI flag through the current provider-auth context.

## What This Changes

The PR adds a `--wait` option to `models auth login-github-copilot` and changes GitHub device-flow polling/backoff timing in the old GitHub Copilot auth provider path.

## Best Possible Solution

Port the behavior into the current plugin-owned GitHub Copilot auth implementation: update `extensions/github-copilot/login.ts` to delay the first token request until the device-code interval has elapsed, track GitHub's returned `slow_down` interval, expose a typed/direct-login `--wait` option through the provider-auth command path, use the plugin prompter for copyable URL/code plus confirmation, then update docs and focused fake-timer/fetch tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main posts the first token request before sleeping and the direct CLI has no `--wait` flag; a fake-fetch/fake-timer unit test can reproduce the immediate poll and `slow_down` backoff behavior without live GitHub credentials.

## Solution Assessment

No. The polling/backoff direction is correct, but this PR is not the best current implementation because it edits the old core provider path and does not plumb `--wait` into the plugin-owned auth flow; the safer fix is a narrow port to the current extension plus typed CLI/provider-auth option plumbing.
