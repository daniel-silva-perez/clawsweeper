## Summary

Keep open. The reported GitHub Copilot device-flow timing bug and WSL/remote-terminal UX gap still exist on current main, but this PR is stale against the plugin-owned implementation: it changes a removed core provider file and does not thread the new wait option into the active provider auth path.

## What This Changes

The PR adds a `--wait` option to the GitHub Copilot login shortcut and changes device-flow polling to wait before the first token request and honor GitHub `slow_down` interval responses.

## Best Possible Solution

Port the polling contract and remote-terminal confirmation UX into `extensions/github-copilot/login.ts` and the current provider-auth command path, with focused unit coverage, provider docs, and a changelog credit while leaving token storage and unrelated providers unchanged.

## Reproduction Assessment

Yes. A static current-main check reproduces the core issue path: the active plugin polls before any delay and handles `slow_down` with a fixed `intervalMs + 2000`; #64319 provides matching WSL user steps for the stuck authorization symptom.

## Solution Assessment

No. The intended fix is valid, but this branch is no longer the best implementation because GitHub Copilot auth moved to the bundled plugin path and direct provider auth needs a current-main option forwarding seam.
