## Summary

Close. Current main already solves the PR's Feishu streaming-card duplicate-final problem through later dispatcher changes that keep finals in one active streaming card and suppress late final text after the card has closed; the draft branch is stale and its normalization approach is incomplete for the idle-close writer path.

## What This Changes

The PR adds a Feishu final-text fingerprint helper and uses it to dedupe formatting-equivalent final replies in the Feishu reply dispatcher, with one regression test for CRLF, zero-width characters, and trailing whitespace.

## Best Possible Solution

Keep the current main dispatcher behavior: coalesce streaming finals into one card during the turn and suppress late final text after closeout. If maintainers want formatting-equivalent dedupe for non-streaming Feishu fallback paths, track that as a separate narrow report with its own regression test.

## Reproduction Assessment

Yes. The current test surface provides a high-confidence code-level reproduction path: one test sends distinct final payloads before idle and asserts one streaming card, and another sends a late final after card close and asserts no duplicate text/card delivery. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR's normalization is no longer the best way to solve the streaming-card issue because current main uses a broader one-card-per-turn and closed-streaming-final guard; the branch also leaves the idle-close writer on raw `streamText`, which would create mixed dedupe keys if revived as-is.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize idle-close dedupe writes:** `extensions/feishu/src/reply-dispatcher.ts:376-379`
  - body: The branch changes duplicate-final lookups to use `finalTextFingerprint`, but `closeStreaming()` still records raw `streamText` when an idle streaming card is closed. If a partial stream contains CRLF, zero-width, or trailing-space artifacts, a later final will be checked with the normalized key and miss the raw Set entry, so the duplicate can still be sent. Store the fingerprint in the idle-close writer too and cover that path.
