## Summary

Keep this issue open. Current main still only exposes local `sessions_send` and `sessions_spawn` session-tool paths, the config/docs still model multiple gateways as isolated instances plus a single remote-client target, and the provided discussion shows open same-author PR #43656 explicitly tracks the native cross-gateway implementation with closing syntax. The related `xihe-jianmu-ipc` MCP bridge is useful workaround/context, but it does not replace native OpenClaw session history, gateway auth, or visibility semantics.

## What This Changes

The issue requests native cross-gateway routing for `sessions_send` and related session delegation so agents on separate gateway instances can communicate without SSH scripts or an external relay.

## Best Possible Solution

Keep this as the canonical native cross-gateway session messaging tracker. Maintainers should review, rework, land, or deliberately replace #43656 with an additive design that covers config schema/help, session-tool schemas, gateway auth and token resolution, remote transcript/history ownership, visibility policy, tests, and public docs.
