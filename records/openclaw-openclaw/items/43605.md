## Summary

Keep this issue open. Current main still does not implement native cross-gateway `sessions_send` or `sessions_spawn`, and same-author PR #43656 remains open, unmerged, and explicitly claims to close this issue.

## Best Possible Solution

Keep this issue open as the canonical native cross-gateway session messaging tracker. Maintainers should review, revise, land, or close PR #43656; if that PR is not the right shape, a replacement should add a documented, additive, secure peer/remote-gateway contract across config schema, gateway protocol, session tool schemas, auth/token resolution, local/remote history handling, tests, and user-facing docs.

## Risks / Open Questions

- Closing now would conflict with an open same-author PR that is an implementation candidate and explicitly claims to close this issue.
- Treating the external MCP bridge as a complete replacement would leave native session history, visibility, A2A policy, and gateway auth semantics unresolved.
- Any native implementation needs explicit security review for peer URL validation, SSRF boundaries, token and SecretInput handling, operator scopes, and local-versus-remote visibility rules.
- PR #43656 has unresolved review feedback around peer URL allowlisting, tokenless peers, idempotency, and spawn parameter parity, so the best path is maintainer review and hardening rather than cleanup closure.
