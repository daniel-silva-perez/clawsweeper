## Summary

Keep this issue open. Current main still treats `sessions_send` and `sessions_spawn` as local-gateway tools, the config surface has only a single `gateway.remote` client target rather than a peer registry, and an open implementation PR already uses closing syntax for this issue.

## What This Changes

Request native cross-gateway session messaging so agents and sessions on peer gateway machines can be targeted without SSH workarounds.

## Best Possible Solution

Keep this as the canonical native cross-gateway session messaging tracker until maintainers land, rework, or deliberately reject the paired PR with an additive design covering config schema/help, tool schemas, gateway auth, visibility, transcript ownership, tests, and docs.
