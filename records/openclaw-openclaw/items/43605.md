## Summary

Keep open. Current main still implements local-only session tools with no remote gateway peer registry or cross-gateway tool parameters, and open PR #43656 already uses closing syntax for this issue, so cleanup closure would be premature.

## What This Changes

Request native cross-gateway session messaging so `sessions_send`, `sessions_spawn`, and scripted agent calls can target agents on other gateway machines instead of failing local agent or session validation.

## Best Possible Solution

Ship an additive cross-gateway session routing design that covers the tool schemas, peer or explicit remote target configuration, remote auth resolution, visibility and transcript ownership rules, tests, and docs while preserving the current local behavior as the default.

## Reproduction Assessment

Not applicable as a traditional bug reproduction, but the current limitation is high-confidence from static source checks: the session tool schemas and CLI path still lack a remote gateway target and validate against local state first.

## Solution Assessment

No, current main is not the best solution because it does not implement the requested behavior. The existing PR direction may be a viable starting point, but it needs maintainer and security review before this tracker can close.
