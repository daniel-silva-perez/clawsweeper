## Summary

Keep this issue open. Current main still implements only local session targeting for `sessions_send` and `sessions_spawn`, the CLI still rejects unknown `--agent` ids before making the gateway call, and open PR #43656 is already the active closing implementation candidate. The related xihe-jianmu-ipc MCP relay is useful external context, but it does not replace the requested native Gateway/session-history routing behavior.

## What This Changes

Request native cross-gateway session messaging so agents on separate Gateway instances can send or spawn work on remote sessions without SSH scripts.

## Best Possible Solution

Keep this issue open until one additive, security-reviewed cross-gateway session routing design lands, preserving local defaults while covering remote target configuration, auth resolution, session/history ownership, tests, and docs.

## Reproduction Assessment

Yes for the current limitation, though it is a feature gap rather than a traditional crash reproduction. Source inspection shows the session tools have no remote target surface, and `openclaw agent --agent` still validates ids against local config before the Gateway request.

## Solution Assessment
