## Summary

Keep this issue open. Current main still implements `sessions_send` and `sessions_spawn` as local-gateway session tools only: there is no `gateway`, `gatewayUrl`, peer registry, or remote-agent config in the tool schema/protocol path, and the server-side send handler still requires a local session entry before dispatch. The issue is also paired with same-author open PR #43656, which is the implementation candidate and should be reviewed, revised, landed, or closed before the tracker is resolved. The related `xihe-jianmu-ipc` MCP bridge reports are useful workaround evidence, but they do not supersede native cross-gateway session routing with OpenClaw session history and policy semantics.

## Best Possible Solution

Keep this issue open as the canonical native cross-gateway session messaging tracker. Maintainers should review, revise, land, or close #43656; if that PR is not the right design, a replacement should add a documented, additive, secure peer/remote-gateway contract across config schema, gateway protocol, session tool schemas, auth/token resolution, local/remote history handling, tests, and user-facing docs.
