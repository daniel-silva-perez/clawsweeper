## Summary

Keep this issue open. Current main still exposes local-only `sessions_send` and `sessions_spawn` tool surfaces, `openclaw agent --agent` still rejects agent ids that are not in the local config before it calls the Gateway, and the issue is already paired with open PR #43656 using closing syntax. The xihe-jianmu-ipc MCP relay discussed in related issues is useful external context, but it does not replace the requested native Gateway/session-history routing behavior.

## What This Changes

Request native cross-gateway session messaging so `sessions_send` and related agent/session entry points can target agents managed by another Gateway without SSH scripts.

## Best Possible Solution

Keep this issue open until one additive, security-reviewed cross-gateway session routing design lands, preserving local defaults while covering remote target configuration, auth resolution, session/history ownership, tests, and docs.

## Reproduction Assessment

Yes for the current limitation, though it is a feature gap rather than a crash reproduction. Source inspection shows the session tools have no remote target surface, and `openclaw agent --agent` still validates ids against local config before the Gateway request.

## Solution Assessment
