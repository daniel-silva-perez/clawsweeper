## Summary

Keep this PR open. Current main still implements the old direct-write Dreaming promotion path, so the PR is not obsolete, but the proposed patch is not merge-ready because the staged apply flow can lose rollback capability on crash, configured daily-signal updates can be filtered out, and staged plan paths are trusted without a workspace-boundary check.

## What This Changes

The PR adds stage-first memory-core Dreaming maintenance artifacts, `/dreaming apply|rollback`, `dailySignalFiles` and maintenance config, docs/tests/a verification script, plus unrelated Amazon Bedrock Mantle provider baseUrl/token-provider changes.

## Best Possible Solution

Revise the PR into a focused memory-core change: keep stage-first Dreaming maintenance scoped to memory-core and its SDK/docs contract, persist rollback metadata before applying file changes, validate staged file paths against the workspace and expected maintenance targets, pass configured daily signal paths through the ingestion whitelist, and split the Bedrock Mantle provider change into a separate PR.

## Reproduction Assessment

Yes. The current-main behavior is verifiable by inspecting the deep-sweep path and its tests/docs, which still write or expect `MEMORY.md`; the PR regressions are directly reproducible from the diff paths noted above. I did not run tests because this was a read-only review.

## Solution Assessment

No. Stage-first Dreaming with apply/rollback is a plausible direction, but this patch is not the narrowest maintainable solution until rollback ordering, daily-signal ingestion, path safety, and the unrelated provider changes are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist the rollback record before applying file changes:** `extensions/memory-core/src/dreaming-maintenance.ts:1214-1218`
  - body: `applyDreamingMaintenance` writes each planned file change before saving `last-apply.json`. If the process exits after `MEMORY.md` or `current.json` is written but before the record is persisted, rollback has no source record and a later apply will conflict against the changed `beforeSha1`. Build `restoreFiles`, save the last-apply record first, then write the file changes.
  - confidence: 0.95
- **[P1] Pass allowPaths when recording daily-signal batches:** `extensions/memory-core/src/dreaming-phases.ts:1263-1272`
  - body: `recordShortTermRecalls` now drops non-date memory files unless the path is whitelisted in `allowPaths`, but the collected daily-signal batch call omits that whitelist. After the first fallback-created entry, `seedConfiguredDailySignalFallback` skips the tracked path, so edits to `memory/daily-log.md` can stop refreshing the short-term evidence chain.
  - confidence: 0.9
- **[P1] Constrain staged file-change paths to the workspace:** `extensions/memory-core/src/dreaming-maintenance.ts:1121-1122`
  - body: The staged plan is parsed from workspace JSON with only a `fileChanges` array check, and `writeFileChange` writes `path.join(workspaceDir, change.path)`. A tampered staged plan or rollback record with an absolute path or `..` segment can make `/dreaming apply` or rollback write outside the workspace. Validate and normalize every file-change path before hash checks or writes.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The new staged maintenance apply/rollback path can write outside the workspace if a staged plan or last-apply record contains an unsafe path.

Concerns:

- **[medium] Staged file paths are not workspace-bound:** `extensions/memory-core/src/dreaming-maintenance.ts:1122`
  - body: `readStagedPlan` trusts `report.fileChanges`, and `writeFileChange` joins `change.path` with the workspace without rejecting absolute paths or parent traversal. Because `/dreaming apply` and rollback consume workspace artifacts, a tampered artifact can cause writes outside the intended memory files.
  - confidence: 0.86
