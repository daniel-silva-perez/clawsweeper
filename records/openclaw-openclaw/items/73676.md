## Summary

Keep this issue open. Current main still has unguarded `process.cwd()` calls on CLI startup, and the supplied GitHub context shows an open, unmerged implementation PR with closing syntax for this issue, so the right maintainer action is to review/land or revise that PR rather than close the issue now.

## Best Possible Solution

Keep this issue open while PR #73700 is reviewed. The best fix is a small shared safe-cwd path for CLI startup: cover `isMainModule`, `shouldLoadCliDotEnv`, and `loadCliDotEnv`; on `ENOENT`/`uv_cwd`, either skip cwd-local dotenv loading or emit a clear early diagnostic asking the user to change to an existing directory. Add focused regression tests that mock `process.cwd()` throwing in the startup and dotenv paths, then close this issue after the merged PR ships.
