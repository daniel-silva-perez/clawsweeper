## Summary

Keep open. Current main still has unguarded `process.cwd()` calls on the CLI startup/de dotenv path, and an open PR with closing syntax plus a broader open startup PR are still unmerged, so this should not be closed ahead of maintainer review.

## What This Changes

The issue asks for OpenClaw CLI startup to handle a deleted current working directory `uv_cwd` failure with a clear diagnostic or safe fallback instead of crashing.

## Best Possible Solution

Ship one shared safe-cwd startup path used by `isMainModule`, CLI/infra dotenv loading, and required home-dir fallback, with targeted tests that mock `process.cwd()` throwing `ENOENT`/`uv_cwd` and confirm normal cwd-local `.env` precedence remains unchanged.

## Reproduction Assessment

Yes. The reported `mkdir`, `cd`, remove directory, then run `openclaw tui` path maps directly to Node throwing from `process.cwd()`, and current main still calls `process.cwd()` before command-specific handling.

## Solution Assessment

Unclear until the open PRs are reviewed, but a narrow one-call-site patch is not the best complete fix. The maintainable direction is a shared helper or startup seam covering the entrypoint, CLI dotenv, infra dotenv, and home fallback paths together.
