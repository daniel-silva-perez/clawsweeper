## Summary

Keep this issue open. Current main still has unguarded `process.cwd()` calls in the CLI startup path, and the issue is already paired with open implementation PRs that need maintainer review rather than another cleanup close or automated repair lane.

## What This Changes

The issue asks OpenClaw CLI startup to handle a deleted current working directory without crashing with `ENOENT: no such file or directory, uv_cwd`.

## Best Possible Solution

Ship one shared safe-cwd startup path covering main-module detection, CLI and infra dotenv loading, and required home-dir fallback, while preserving cwd-local `.env` precedence whenever the cwd exists and adding focused regression tests for `process.cwd()` throwing `ENOENT`/`uv_cwd`.

## Reproduction Assessment

Yes. The reported create-directory, enter it, remove it, then run `openclaw tui` flow maps directly to Node throwing from `process.cwd()`, and current main still calls `process.cwd()` before command-specific handling can recover.

## Solution Assessment

Unclear until the open PRs are reviewed, but current main is not the solution. The narrowest maintainable direction is a shared startup/helper seam rather than patching only one raw `process.cwd()` call.
