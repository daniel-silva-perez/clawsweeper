## Summary

Keep this issue open. Current main at 0f3a9d812b40cbc8e545abcb80a1f12a93946cf6 still has unguarded `process.cwd()` calls on the CLI startup path, and the live public GitHub API shows PR #73700 is still open, unmerged, and explicitly claims to fix this issue. Repository cleanup policy says the issue should stay open while that implementation candidate is reviewed or revised.

## Best Possible Solution

Keep this issue open while PR #73700 is reviewed. The best fix is a small shared safe-cwd path for CLI startup that covers `isMainModule`, `shouldLoadCliDotEnv`, and `loadCliDotEnv`; on `ENOENT`/`uv_cwd`, skip cwd-local dotenv loading or emit a clear early diagnostic asking the user to change to an existing directory. Add focused regression tests that mock `process.cwd()` throwing in the startup and dotenv paths, then close the issue after the merged fix ships.
