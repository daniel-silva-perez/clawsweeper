## Summary

Keep open. Current main still has unguarded `process.cwd()` calls in the CLI startup and dotenv/home fallback paths, and the issue is already paired with open implementation PRs, including #74062 with closing syntax and #74994 with a broader startup guard, so it should not be closed before maintainer review.

## What This Changes

Handle a deleted current working directory during CLI startup so commands such as `openclaw tui` do not crash with `ENOENT: no such file or directory, uv_cwd`.

## Best Possible Solution

Ship one shared safe-cwd startup path covering `isMainModule`, CLI dotenv loading, infra dotenv loading, and required home-dir fallback, with targeted tests that mock `process.cwd()` throwing `ENOENT`/`uv_cwd` while preserving normal cwd-local `.env` precedence.

## Reproduction Assessment

Yes. The reported create-enter-delete directory flow maps directly to Node throwing from `process.cwd()`, and current main still invokes `process.cwd()` in the startup path before command-specific handling can recover.

## Solution Assessment

Unclear which open PR is the best final patch until maintainer review, but a shared startup helper across the known call sites is the narrowest complete direction. A one-off guard in only `shouldLoadCliDotEnv` would leave entrypoint and infra fallback throw paths behind.
