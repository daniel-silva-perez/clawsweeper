## Summary

Keep open. Current main does not show compaction directly flipping TTS on, and docs/code keep auto-TTS off by default, but the agent-facing `tts` tool is still always registered and its description says audio is delivered automatically without an explicit-request guard. That leaves a plausible current-main path for the assistant to produce audio in the dashboard without the user asking, especially after a long/compacted prompt changes model behavior.

## Best Possible Solution

Keep this issue open and land a narrow behavior hardening PR: keep auto-TTS and explicit `/tts` workflows intact, but update the default agent-facing TTS tool/prompt guidance so the assistant only uses the `tts` tool when the user explicitly requests speech/audio/voice or an active TTS policy clearly calls for it. Add focused tests around the tool description/system prompt so this does not regress.

## Clownfish Work Prompt

Fix this issue by hardening the default agent-facing TTS behavior. Current main keeps Auto-TTS off by default, but `createOpenClawTools` always registers the `tts` tool and `src/agents/tools/tts-tool.ts` tells the model that audio is delivered automatically without saying the tool should only be used for explicit audio/speech/voice requests. Update the tool description and, if needed, the Voice/TTS system-prompt hint so normal dashboard text chats remain text unless the user explicitly asks for audio/voice/speech or an active TTS policy clearly requires it. Do not remove the `tts` tool, do not break `/tts on`, `/tts chat on`, `/tts audio`, `tts.convert`, `messages.tts.auto`, provider selection, or channel voice-note delivery.
