## Summary

Keep open. Current main still has the reported provider normalization behavior, so the PR is not obsolete, but the patch is incomplete: changing the shared provider normalizer without updating adjacent Kimi plugin compatibility and shared alias-policy tests introduces concrete regressions before merge.

## What This Changes

The PR branch changes `src/agents/provider-id.ts` to stop normalizing `kimi-coding` to `kimi` and updates three `src/agents/model-selection.test.ts` expectations to preserve `kimi-coding` in selected model refs.

## Best Possible Solution

Accept the provider namespace fix only after preserving the bundled Kimi plugin's legacy override path explicitly and updating the shared provider-alias contract/tests so tool policy behavior matches the new `kimi-coding` semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve legacy Kimi explicit-provider lookup:** `src/agents/provider-id.ts:17`
  - body: Removing `kimi-coding` from the shared normalizer means `findExplicitProviderConfig(..., "kimi")` in the bundled Kimi plugin no longer finds `models.providers["kimi-coding"]`. Existing users would lose their legacy base URL/header override, and `extensions/kimi-coding/implicit-provider.test.ts` has assertions for that behavior. Add an explicit compatibility lookup or a narrower alias helper for this plugin path before changing the global normalizer.
  - confidence: 0.88
- **[P2] Update alias-policy cases with the new provider identity:** `src/agents/provider-id.ts:17`
  - body: The shared `providerAliasCases` table still maps `kimi-coding` to `kimi`, while this change makes `normalizeProviderId("kimi-coding")` return `kimi-coding`. Tool-policy tests that build `tools.byProvider.kimi` and call with model provider `kimi-coding` will no longer match, so the shared helper and its expected policy semantics need to be updated with this provider-id change.
  - confidence: 0.9
