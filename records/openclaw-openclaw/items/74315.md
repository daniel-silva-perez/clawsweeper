## Summary

Keep open. The PR targets a real model-resolution bug from #74310 and current main still collapses `kimi-coding` into `kimi`, but the proposed patch is incomplete because it breaks the bundled Kimi provider's legacy explicit-provider override path and leaves shared provider alias policy tests/contracts inconsistent.

## What This Changes

The branch removes `kimi-coding` from the Kimi alias branch in `normalizeProviderId` and updates three `model-selection` expectations so `kimi-coding` remains a distinct provider id.

## Best Possible Solution

Keep the namespace fix, but implement it with a narrower compatibility plan: preserve exact `kimi-coding` custom catalog resolution, add an explicit legacy lookup for the bundled Kimi provider path, and update the shared alias/tool-policy contract and tests to reflect the new provider-id boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve legacy Kimi provider overrides:** `src/agents/provider-id.ts:17`
  - body: Removing `kimi-coding` from the shared normalizer makes `normalizeProviderId("kimi-coding")` return `kimi-coding`. The bundled Kimi catalog calls `findExplicitProviderConfig(..., "kimi")` and matches configured keys through this same normalizer, so existing `models.providers["kimi-coding"]` baseUrl/header overrides no longer match. Add an explicit compatibility lookup, or move this split out of the global normalizer before removing the alias.
  - confidence: 0.88
- **[P2] Update provider alias policy cases:** `src/agents/provider-id.ts:17`
  - body: The shared `providerAliasCases` table still maps `kimi-coding` to `kimi`, and provider tool-policy tests build `tools.byProvider.kimi` for a `kimi-coding` model. After this line changes normalization to `kimi-coding`, those lookups no longer match, so the alias table and expected policy semantics need to be updated with the provider-id split.
  - confidence: 0.9
