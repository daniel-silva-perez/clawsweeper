## Summary

Keep open. The latest branch fixes only helper-level provider-map lookup; current main and the PR still re-normalize exact `kimi-coding` refs in runtime and fallback paths, so the linked `kimi-coding/k2p5` configuration can still collapse to `kimi/k2p5`.

## What This Changes

The PR preserves the `kimi-coding` to `kimi` alias but adds case-insensitive exact provider-key lookup before alias fallback in `findNormalizedProviderValue` and `findNormalizedProviderKey`, plus unit tests for those helper semantics.

## Best Possible Solution

Preserve exact configured provider IDs as runtime/catalog identities after model resolution, while keeping Kimi alias normalization only where provider-family policy, auth, and compatibility checks need it. Add a regression that exercises `models.providers.kimi-coding` with `agents.defaults.model.primary: kimi-coding/k2p5` through default selection, fallback candidate construction, and status/session persistence.

## Reproduction Assessment

Yes. The linked issue gives a concrete config and expected `/status` behavior, and source inspection shows the exact provider can still be collapsed by `normalizeModelRef` after initial resolution.

## Solution Assessment

No. Exact-first lookup in provider-map helpers is useful, but it is not the complete fix because runtime default selection and fallback candidate construction still re-run the global alias normalizer.
