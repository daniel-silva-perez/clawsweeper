## Summary

Keep open. Current main has WSL2/Windows Chrome guidance and related browser timeout hardening, but this report is recent, has no direct repro/logs, and the observed SIGKILL could be either a real WSL browser hang or the agent exec watchdog killing a timed-out `openclaw browser` command. There is not enough proof to close it as implemented, duplicate, or stale.

## Best Possible Solution

Keep the issue open and ask for a direct shell reproduction: `openclaw browser status --json`, `openclaw browser doctor --deep`, the redacted `browser` config/profile, gateway logs around the failure, and whether the goal is Windows host Chrome or a Linux Chrome inside WSL. If the goal is host Chrome, guide the reporter through the documented remote CDP profile path; if the direct logs show the command is being killed by OpenClaw's exec watchdog, improve browser/exec diagnostics or timeout handling for that path.
