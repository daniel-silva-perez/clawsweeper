## Summary

Keep open. Current main still documents and preserves the static Browserbase CDP URL path, while the browser runtime connects directly to the configured CDP endpoint and has no Browserbase Sessions API flow; Browserbase’s current docs require creating a session and using the returned connectUrl. citeturn1view0

## What This Changes

Add Browserbase session lifecycle support so OpenClaw creates a Browserbase session, connects to its returned WebSocket URL, and updates the stale static-CDP documentation/config path.

## Best Possible Solution

Browserbase support should live in the browser plugin as explicit provider/session lifecycle handling: add safe Browserbase credential/session config, create a session before CDP attach, connect Playwright to the returned connectUrl, close or document keep-alive behavior on stop/release, update docs/config schema/help and tests, and preserve generic remote CDP behavior for Browserless, local Chrome, and other providers.

## Reproduction Assessment

Yes. The documented Browserbase profile still flows through cdpUrl normalization and direct CDP attach, while Browserbase’s current API requires `POST /v1/sessions` before connecting to `connectUrl`; a credential-free probe of the documented static endpoint returned 404.

## Solution Assessment

Yes. The session-API direction is the right maintainable fix, but it should be implemented as explicit Browserbase profile/session config and environment support rather than only substring-detecting `browserbase.com` inside arbitrary cdpUrl values.
