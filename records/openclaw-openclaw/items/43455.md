## Summary

Current main still has the reported Browserbase integration gap: OpenClaw documents a static Browserbase WebSocket URL and the browser runtime attaches directly to configured CDP URLs, while Browserbase's current API contract requires creating a session first and using the returned connectUrl. 

## What This Changes

The issue requests Browserbase-aware browser profile support that creates a Browserbase session through the REST API and connects to the returned connectUrl instead of relying on a static cdpUrl.

## Best Possible Solution

Browserbase support should live in the browser plugin as explicit provider/session lifecycle handling: add safe Browserbase credential/session config, create a session before CDP attach, connect Playwright to the returned connectUrl, close or document keep-alive behavior on stop/release, update docs/config schema/help and tests, and preserve generic remote CDP behavior for Browserless, local Chrome, and other providers.

## Reproduction Assessment

Yes. The current documented Browserbase profile flows through cdpUrl normalization, getChromeWebSocketUrl(), and chromium.connectOverCDP() without any Browserbase REST session request; Browserbase's current docs require creating a session first to obtain connectUrl. No Browserbase credentials were present in env or ~/.profile, so I did not live-hit the reported 404.

## Solution Assessment

Yes. The session-API direction is the right maintainable fix, but it should be implemented as explicit Browserbase profile/session config and env support rather than only substring-detecting browserbase.com inside arbitrary cdpUrl values.
