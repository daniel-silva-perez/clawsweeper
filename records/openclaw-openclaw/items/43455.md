## Summary

Current main still has the reported Browserbase gap: the user-facing browser docs recommend the obsolete static Browserbase WebSocket URL, while the browser runtime and config model only normalize and attach to configured CDP URLs directly. Official Browserbase docs now require creating a session first and using the returned `connectUrl`, so this remains an actionable browser plugin fix rather than a stale or superseded report.

## What This Changes

The issue requests Browserbase session lifecycle support because the documented static Browserbase CDP WebSocket URL reportedly now returns 404 and Browserbase requires creating a session to obtain a connect URL.

## Best Possible Solution

Browserbase support should live in the browser plugin as explicit provider/session lifecycle handling: accept safe Browserbase credential and session config, create a Browserbase session before CDP attach, connect Playwright to the returned `connectUrl`, handle cleanup or documented keep-alive semantics on stop/release, update docs and config schema/help, and preserve generic remote CDP behavior for Browserless, local Chrome, and other providers.

## Reproduction Assessment

Yes. The documented Browserbase profile still resolves to a static `cdpUrl` and flows through `getChromeWebSocketUrl()` plus `chromium.connectOverCDP()` without a Browserbase REST session request, while official Browserbase docs require creating a session first to obtain `connectUrl`. Live credentials would only confirm the provider-side 404 reported by the user.

## Solution Assessment

Yes, the session-API direction is the maintainable fix, but it should be implemented as explicit Browserbase profile/session config and env support rather than only detecting `browserbase.com` inside arbitrary `cdpUrl` strings.
