## Summary

Current main still has the reported Browserbase gap: the browser docs recommend the obsolete static Browserbase WebSocket URL, and the browser runtime still connects to configured `cdpUrl` values directly instead of creating a Browserbase session first. The closed remote-persistence report #58646 covers CDP connection reuse, not this remaining session-creation requirement.

## What This Changes

The issue requests Browserbase-aware browser profile handling that creates a Browserbase session through the REST API and connects to the returned ephemeral CDP URL instead of using a static `cdpUrl`.

## Best Possible Solution

Browserbase support should be implemented inside the browser plugin as explicit session lifecycle handling: accept safe credential/config inputs, create a Browserbase session before CDP attach, connect Playwright to the returned `connectUrl`, release or document timeout behavior when OpenClaw closes the connection, and update docs/tests while preserving generic remote CDP behavior for Browserless, local Chrome, and other providers.

## Reproduction Assessment

Yes. The documented profile config still flows through `getChromeWebSocketUrl()` and `chromium.connectOverCDP()` as a direct static WebSocket URL, while Browserbase's current docs require creating a session first to obtain `connectUrl`; live credentials would only confirm the provider-side 404 observed by the reporter.

## Solution Assessment

Yes, with a refinement: the issue's session-API direction is the maintainable fix, but the implementation should prefer explicit Browserbase profile/session config and env support over only detecting `browserbase.com` inside arbitrary `cdpUrl` strings.
