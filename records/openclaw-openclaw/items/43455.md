## Summary

Current main still matches the reported Browserbase failure: OpenClaw documents the static Browserbase WebSocket URL and the browser runtime treats configured CDP URLs as direct/discovered endpoints, while Browserbase's current Sessions API flow creates a session first and returns a per-session connection URL. This should stay open and is narrow enough for a focused browser plugin repair.

## What This Changes

The issue requests Browserbase-aware browser profile support that creates a Browserbase session, connects through the returned CDP `connectUrl`, and releases the session when finished.

## Best Possible Solution

Implement Browserbase session lifecycle support in the browser plugin, update the Browserbase docs away from the static-root WebSocket assumption, preserve generic `cdpUrl` and Browserless direct-WebSocket behavior, and cover create/connect/cache/release/error paths with mocked Browserbase API tests.

Current docs tell users to configure `cdpUrl: "wss://connect.browserbase.com?apiKey=<BROWSERBASE_API_KEY>"`, but Browserbase now requires creating a session through the Sessions API and connecting to the returned `connectUrl`. Implement this inside the browser plugin boundary. Preserve existing generic `cdpUrl`, Browserless, loopback, direct WebSocket, and `/json/version` discovery behavior. Keep API keys redacted in errors/logs/config snapshots, do not move provider-specific policy into core unless a typed config/schema/docs seam is required, and do not change unrelated browser providers. Update `docs/tools/browser.md` and add a user-facing changelog fix with `Thanks @HanenVit`.
