## Summary

Keep open. Current main still documents the static Browserbase CDP URL and still connects directly to the configured CDP endpoint without creating a Browserbase session, while Browserbase's current API documents `POST /v1/sessions`, a returned `connectUrl`, and optional release through the session update endpoint. ([docs.browserbase.com](https://docs.browserbase.com/reference/api/update-a-session))

## What This Changes

The issue asks OpenClaw browser profiles to manage Browserbase Sessions API lifecycle instead of relying on a static Browserbase `cdpUrl`.

## Best Possible Solution

Add explicit Browserbase provider/session handling inside the browser plugin: accept safe credential/session config, create a session before CDP attach, connect Playwright to the returned `connectUrl`, release OpenClaw-created sessions when appropriate, update docs/schema/help, and keep generic remote CDP behavior unchanged for local Chrome, Browserless, and other providers.

## Reproduction Assessment

Yes. The documented static Browserbase endpoint still flows through direct CDP attach on current main, and a live unauthenticated probe of that endpoint returns HTTP 404 while Browserbase documents create-session-then-connect behavior.

## Solution Assessment

Yes. The Sessions API direction is the maintainable fix, but it should be modeled as explicit Browserbase profile/session config and environment support rather than only substring-detecting `browserbase.com` inside arbitrary CDP URLs.
