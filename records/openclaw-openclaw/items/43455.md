## Summary

Current main still matches the reported Browserbase breakage: the docs advertise the static Browserbase WebSocket URL and say no session creation is needed, while the browser runtime only treats configured CDP URLs as direct/discovered endpoints and has no Browserbase Sessions API create/release lifecycle. This should stay open and is narrow enough for a focused browser plugin repair PR.

## What This Changes

The issue asks OpenClaw's browser integration to stop relying on Browserbase's static `connect.browserbase.com` WebSocket and instead create, connect to, and release Browserbase sessions through the Browserbase Sessions API.

## Best Possible Solution

Implement Browserbase session lifecycle support inside the browser plugin, update the Browserbase docs away from the static-root WebSocket assumption, preserve generic `cdpUrl` and Browserless direct-WebSocket behavior, and cover create/connect/reuse/release/error paths with mocked Browserbase API tests.

## Repair Work Prompt

Fix the Browserbase browser integration reported in https://github.com/openclaw/openclaw/issues/43455. Current docs/config rely on `wss://connect.browserbase.com?apiKey=...`, but Browserbase's current Sessions API requires creating a session first and connecting to the returned `connectUrl`. Implement the fix inside the browser plugin/runtime boundary, not core-wide provider logic. Preserve generic `cdpUrl` behavior and existing Browserless/direct WebSocket fallback behavior. Support a conservative compatibility path for Browserbase profiles, including API key resolution from the existing `apiKey` query and/or `BROWSERBASE_API_KEY`, optional project ID from `BROWSERBASE_PROJECT_ID` if needed, secret redaction, session creation before Playwright CDP connect, and release via Browserbase's request-release API when the cached connection/profile is closed, reset, or stopped. Add mocked tests for create/connect/reuse/release/error/rate-limit behavior; do not require live Browserbase credentials. Update `docs/tools/browser.md` and add a user-facing changelog entry with proper attribution. Do not change unrelated browser profile semantics, SSRF policy defaults, or bundled provider/channel behavior.
