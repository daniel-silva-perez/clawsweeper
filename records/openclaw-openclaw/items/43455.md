## Summary

Current main still matches the reported Browserbase breakage: the docs recommend the static `connect.browserbase.com` WebSocket URL while the browser runtime treats configured CDP URLs as direct endpoints and never creates or releases Browserbase sessions through the provider API. The issue should stay open and is narrow enough for a focused browser-plugin repair PR.

## What This Changes

Have OpenClaw browser profiles manage Browserbase sessions through the Browserbase Sessions API and connect to the returned `connectUrl` instead of relying on the obsolete static `cdpUrl`.

## Best Possible Solution

Implement Browserbase session lifecycle support inside the browser plugin, update the config/docs away from the static root WebSocket assumption, preserve generic `cdpUrl` and Browserless direct-WebSocket behavior, and cover create/connect/reuse/release/error paths with mocked Browserbase API tests.
