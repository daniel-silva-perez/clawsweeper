## Summary

Keep this open. Current main still lets shared-session heartbeat runs select the configured heartbeat model, persist that runtime provider/model into the normal session entry, and have `session_status` render those persisted fields as the selected model when no explicit session override exists.

## What This Changes

The issue reports that heartbeat or internal async-completion runs using `agents.defaults.heartbeat.model` can persist that runtime model into the user-facing session fields and make status report the heartbeat model as the normal session model.

## Best Possible Solution

Add a run-kind-aware boundary for session runtime metadata: heartbeat/internal runs should not overwrite the session fields used as the selected user-turn model, or their runtime identity should be stored and rendered separately with a clear label. Keep normal runtime/fallback diagnostics intact and treat `heartbeat.isolatedSession` as an optional mitigation rather than the only way to avoid misleading status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a shared-session heartbeat with `heartbeat.model`, `isolatedSession` unset, and no explicit session override; the heartbeat model is selected, persisted to `SessionEntry.modelProvider/model`, and then displayed by `session_status` as the session model.

## Solution Assessment

Yes. The narrowest maintainable fix is a run-kind-aware persistence or display split, not reverting runtime model visibility wholesale, because normal user-turn fallback diagnostics from #55425 are still useful.
