## Summary

Keep this open. Current main still has the reported behavior: shared-session heartbeat runs can select the configured heartbeat model, persist that runtime provider/model onto the same session entry, and have `session_status` display those runtime fields as the session card model when no explicit user override exists. The related #51706 remains open and overlaps the status display surface rather than resolving the heartbeat/internal runtime bleed.

## What This Changes

The issue asks OpenClaw to stop heartbeat or internal async-completion runs from making a normal user session status look selected onto the heartbeat runtime model, or to display that internal runtime identity separately.

## Best Possible Solution

Introduce a run-kind-aware boundary between selected/default user-turn model identity and internal heartbeat runtime identity. Either prevent heartbeat/internal runs from overwriting the session fields used as the selected status-card model, or persist heartbeat runtime metadata separately and label it separately in status.

## Reproduction Assessment

Yes. A source-level reproduction is clear: run a shared-session heartbeat with `heartbeat.model`, no explicit session override, and empty fallbacks; the heartbeat override is selected, persisted to the shared session entry, and then status treats those persisted fields as the card model. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is a run-kind-aware persistence/display split rather than treating every runtime model write as selected user-session model state; that preserves normal fallback/runtime diagnostics while preventing heartbeat/internal events from making the user session appear manually switched.
