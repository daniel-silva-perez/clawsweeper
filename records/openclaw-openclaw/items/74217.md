## Summary

Keep open. Current main still has the reported behavior: heartbeat runs can select the heartbeat model for a shared session, the actual runtime provider/model is persisted back onto the same session entry, and session status uses those persisted runtime fields as the visible card model when no explicit override is set. The related #51706 remains open and moves status further toward runtime-model display rather than separating heartbeat/internal runtime identity from normal selected session identity.

## What This Changes

The issue asks OpenClaw to stop heartbeat or internal async-completion runs from making a normal user session status look selected onto the heartbeat runtime model, or to display heartbeat runtime identity separately.

## Best Possible Solution

Introduce a run-kind-aware boundary between selected/default user-turn model identity and internal heartbeat runtime identity. Either prevent same-session heartbeat/internal runs from overwriting the `SessionEntry.modelProvider` and `SessionEntry.model` fields used as the selected status-card model, or persist heartbeat runtime metadata separately and teach status to label it separately from the normal session model.

## Reproduction Assessment

Yes. The source path gives a high-confidence reproduction: run a shared-session heartbeat with `heartbeat.model`, no explicit session override, and empty fallbacks; the heartbeat override is selected for the run, persisted to the shared session entry, and then status treats those persisted fields as the card model. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, the best direction is a narrow run-kind-aware persistence/display split rather than treating all runtime model writes as selected user-session model state. That preserves normal fallback/runtime diagnostics while preventing heartbeat/internal events from making the user session appear manually switched.

## Repair Work Prompt

Fix the heartbeat model bleed reported in this issue. Current main lets heartbeat/internal runs using `agents.defaults.heartbeat.model` persist their runtime provider/model into the shared user session, and status then reports that runtime as the selected session model. Implement a narrow fix that separates normal selected/default user-turn model identity from heartbeat/internal runtime identity. Prefer either preventing same-session heartbeat/internal runs from overwriting the `SessionEntry.modelProvider`/`model` fields used as the selected status-card model, or adding separate heartbeat/internal runtime metadata and displaying it distinctly. Coordinate with related #51706 and #74257, but do not land #51706's runtime-display behavior blindly. Preserve normal user-turn fallback/runtime visibility, per-session explicit model overrides, provider alias behavior, and existing heartbeat `isolatedSession` semantics. Add regression tests for a shared-session heartbeat with `heartbeat.model`, no explicit session override, and empty fallbacks so `/status`/`session_status` does not claim the user session selected the heartbeat model.
