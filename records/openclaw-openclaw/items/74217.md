## Summary

Keep this open. Current main still has the reported source-level path: shared-session heartbeats can select `heartbeat.model`, the generic run accounting path persists that runtime provider/model into `SessionEntry.modelProvider/model`, and `session_status` then renders those persisted runtime fields as the selected model when no explicit user override exists. The docs and overflow diagnostic currently describe `isolatedSession` as mitigation for heartbeat model bleed, not as a fix.

## What This Changes

The issue asks for heartbeat/internal async-completion runs using `agents.defaults.heartbeat.model` not to overwrite or be shown as the normal user-facing session model, or for status to label that internal runtime model separately.

## Best Possible Solution

Add a run-kind-aware boundary for session runtime metadata: heartbeat/internal runs should not overwrite the session fields used as the selected user-turn model, or their runtime identity should be persisted separately and labeled separately in status. Preserve normal user-turn fallback/runtime visibility from the status fixes while keeping `heartbeat.isolatedSession` as an optional cost/isolation mitigation rather than the only way to avoid misleading status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is visible on current main: run a shared-session heartbeat with `heartbeat.model`, no explicit session model override, and `isolatedSession` unset; the heartbeat-selected provider/model flows into generic session persistence and then into `session_status` as the selected card model.

## Solution Assessment

Yes, the narrowest maintainable direction is a run-kind-aware persistence/display split, not reverting the runtime-model status work. That keeps useful fallback/runtime diagnostics while preventing heartbeat/internal events from looking like user-selected model switches.
