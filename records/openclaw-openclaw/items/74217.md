## Summary

Keep open. Current main still lets a heartbeat model override flow into shared-session runtime metadata, and the status paths then read that persisted runtime identity. The related #55425 work is useful for normal-turn visibility, but it does not supersede this bug because heartbeat/internal-run telemetry still needs a separate persistence or display boundary.

## What This Changes

The issue asks OpenClaw to stop heartbeat or internal async runs from making a user-facing session status look like the normal selected model changed.

## Best Possible Solution

Add a run-kind-aware runtime metadata boundary: heartbeat, cron, and internal async completion runs should either avoid overwriting user-turn `model`/`modelProvider` on shared session rows or persist/display their runtime model in a separate internal diagnostic field, while normal user turns and fallback selections continue to update status as #55425 intended.

## Reproduction Assessment

Yes. Configure a shared-session heartbeat with `agents.defaults.heartbeat.model`, leave `isolatedSession` unset, let the heartbeat run, and inspect the session entry or status card; current main has an unguarded path from heartbeat model selection to persisted session runtime model fields.

## Solution Assessment

Yes. The narrowest maintainable repair is to propagate run kind or `isHeartbeat` into persistence/status accounting and keep internal-run model telemetry separate; relying on `heartbeat.isolatedSession` is only a workaround and reverting runtime-model status would regress normal debugging behavior.

## Repair Work Prompt

Fix the shared-session heartbeat model leak reported in https://github.com/openclaw/openclaw/issues/74217. Observable bug: when `agents.defaults.heartbeat.model` is set and `heartbeat.isolatedSession` is false/unset, the heartbeat run can persist its provider/model into the user-facing session `modelProvider`/`model`, making `/status` or `session_status` look like normal user turns switched to the heartbeat model. Expected boundary: preserve #55425 runtime-model status visibility for normal user turns and fallback selections, but do not let heartbeat/internal async runs overwrite the user-turn runtime model fields. Prefer a run-kind-aware persistence boundary, or separate internal-run runtime metadata/display, over reverting status to configured defaults. Likely files: `src/infra/heartbeat-runner.ts`, `src/auto-reply/get-reply-options.types.ts`, `src/auto-reply/reply/get-reply.ts`, `src/auto-reply/reply/agent-runner.ts`, `src/auto-reply/reply/session-usage.ts`, `src/agents/tools/session-status-tool.ts`, `src/status/status-text.ts`, `src/status/status-message.ts`, and focused tests. Add regression coverage for a shared-session heartbeat with `heartbeat.model` proving the stored user session model/status remains the normal selected runtime model, while normal runtime/fallback model status behavior still passes. Update `docs/gateway/heartbeat.md` only if the user-visible semantics change, and add a one-line `CHANGELOG.md` fix entry with reporter credit. Do not require users to enable `heartbeat.isolatedSession` as the fix and do not broaden into unrelated heartbeat output leakage issues.
