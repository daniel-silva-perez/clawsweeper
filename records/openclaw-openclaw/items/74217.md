## Summary

Keep open. Current main still has the reported path: shared-session heartbeat runs can select `agents.defaults.heartbeat.model`, persist that runtime provider/model onto the same session entry, and `session_status` then uses the persisted runtime identity as the primary status model when there is no explicit override.

## What This Changes

The issue asks OpenClaw to prevent heartbeat/internal async-completion runs from making a normal user session status look switched to the heartbeat model, or to display heartbeat runtime metadata separately from normal user-turn model selection.

## Best Possible Solution

Add a run-kind-aware runtime metadata boundary: heartbeat and other internal runs should not update the user-facing `sessionEntry.model` / `modelProvider` selected-runtime fields, or should write to separate internal-run diagnostic fields that status renders distinctly. Normal user turns and real fallback/runtime model changes should continue to appear in status.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a shared-session heartbeat with `agents.defaults.heartbeat.model`, let the heartbeat run with `isolatedSession` unset, then inspect the session entry or `session_status`; current main still routes that heartbeat model into the persisted session runtime fields that status consumes.

## Solution Assessment

Yes. The narrowest maintainable direction is to thread run kind through persistence/status and split internal-run diagnostics from user-turn runtime model state, rather than disabling heartbeat model overrides or undoing normal runtime-model status support.

Current main lets heartbeat/internal runs that use `agents.defaults.heartbeat.model` persist their provider/model into the shared user session's `model` / `modelProvider`, and `session_status` then reports that as the normal selected/runtime model when there is no explicit override. Implement a narrow run-kind-aware boundary so heartbeat/internal runs either skip updating those user-facing runtime model fields or persist their internal model to separate diagnostic metadata that status can render distinctly.
