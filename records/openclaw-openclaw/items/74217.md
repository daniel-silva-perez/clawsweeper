## Summary

Keep this open. Current main still has the reported source-level path: shared-session heartbeats can select `heartbeat.model`, persist the actual runtime provider/model into the normal session entry, and `session_status` then renders those persisted runtime fields as the selected model when no explicit user override exists.

## What This Changes

Prevent heartbeat or internal async-completion runs using `agents.defaults.heartbeat.model` from making a normal user session/status appear to use the heartbeat runtime model.

## Best Possible Solution

Add a run-kind-aware boundary for session runtime metadata: heartbeat/internal runs should not overwrite the session fields used as the selected user-turn model, or their runtime identity should be persisted separately and labeled separately in status. Preserve normal user-turn fallback/runtime visibility and keep `heartbeat.isolatedSession` as an optional mitigation rather than the only way to avoid misleading status.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: run a shared-session heartbeat with `heartbeat.model`, no session override, and `isolatedSession` unset; the selected heartbeat runtime model is persisted to `SessionEntry.modelProvider/model`, and `session_status` uses those fields as the card model. I did not run live tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, the narrowest maintainable fix is a run-kind-aware persistence/display split, not a broad status rollback. That keeps normal runtime/fallback diagnostics from #55425 while preventing heartbeat/internal events from looking like user-selected model switches.
