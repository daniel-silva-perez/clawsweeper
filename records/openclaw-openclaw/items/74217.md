## Summary

Keep open. Current main still lets heartbeat runs select `agents.defaults.heartbeat.model`, persists the actual heartbeat runtime provider/model into shared session fields, and then uses those fields as the session-status card model when no explicit override exists. The related #51706 remains open and points in the opposite status-display direction rather than fixing heartbeat/internal runtime bleed.

## What This Changes

The issue asks OpenClaw to stop heartbeat/internal async runs from making a normal user session status look selected onto the heartbeat runtime model, or to display heartbeat runtime identity separately.

## Best Possible Solution

Introduce a run-kind-aware boundary between selected/default user-turn model identity and internal heartbeat runtime identity. Either prevent same-session heartbeat runs from overwriting the selected `SessionEntry.modelProvider`/`model` fields, or persist heartbeat runtime metadata separately and teach status to label it separately from the normal session model.
