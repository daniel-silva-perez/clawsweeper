## Summary

Keep this PR open. Current main has implemented and improved the diagnostics portion with per-target Google Chat auth rejection logging and appPrincipal warnings, but it has not implemented the PR's issuer-claim matching for app-url/add-on JWTs or the sender email fallback when message.sender exists without an email. Security review found no workflow, dependency, lockfile, or script changes in the PR scope, but the auth relaxation needs rebase and upstream issuer-contract validation before it can land.

## Best Possible Solution

Keep this PR open for maintainer review or author update. The best path is to rebase it onto current main, drop the now-obsolete diagnostics rewrite while preserving current per-target logging, add focused tests for any confirmed iss-only Google Chat/add-on token variants, verify google-auth-library issuer/cert behavior against the real token contract, and land the low-risk event.user.email fallback with regression coverage.

## Risks / Open Questions

- The PR is stale relative to current main; a revived patch must preserve current per-target auth rejection logging and appPrincipal warning behavior instead of reintroducing the targets[0] logger pattern from the original diff.
- The issuer-based auth relaxation is security-sensitive and may be incomplete as written because google-auth-library's default verifyIdToken issuer list does not include the Chat service-account issuer. Maintainers should require real token evidence or an explicit verifier configuration/source-of-certs change before accepting iss as equivalent to email.
