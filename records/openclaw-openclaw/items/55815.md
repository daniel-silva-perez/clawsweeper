## Summary

Keep this PR open. The current main branch still lacks the warning-only classifier this PR proposes, and `handleToolExecutionEnd()` still treats `evt.isError` as a hard tool error without checking for warning-only results. The linked bug context is still coherent: Telegram reaction soft failures return warning payloads, while mutating message-tool errors are still surfaced despite `messages.suppressToolErrors`.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused subscriber fix that treats warning-only tool results as non-errors for both top-level and `result.details.warning` payloads, preserves hard-error classification, and includes regression coverage for the Telegram `jsonResult({ ok: false, warning })` shape plus the emitted `isError: false` handler path.
