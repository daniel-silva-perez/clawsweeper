## Summary

Keep this PR open. Current main and the latest release still read transcript usage without validating that the selected transcript's session header belongs to the requested session, while the PR is a focused fix with a direct regression test and no blocking review finding.

## What This Changes

The PR adds a first-line session-header ID check to `readLatestSessionUsageFromTranscript` and a regression test for mismatched stale transcript usage after reset.

## Best Possible Solution

Land this narrow guard after normal review, preserving legacy/headerless transcript behavior while rejecting parseable mismatched session headers before transcript usage is surfaced.

## Reproduction Assessment

Yes. A high-confidence reproduction is to create a transcript whose first line has session ID A and assistant usage, then call `readLatestSessionUsageFromTranscript` for session ID B with that file discoverable; current main will extract the stale usage, and the PR test encodes that path.

## Solution Assessment

Yes. The proposed first-line header check is the narrow maintainable fix because transcript writers put the session header first, and the code intentionally falls back for unparseable or headerless legacy/custom transcripts.
