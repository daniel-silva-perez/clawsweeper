## Summary

Keep open. Current main still has a real source-level gap for omitted cache pricing fields: some config/catalog paths normalize missing cache prices to zero, but the direct usage-format cost index can still carry `undefined`, causing cost estimation to return `undefined` and the OpenTelemetry cost counter to be skipped. The issue is also paired with an open linked fix proposal, so it should not be closed before that work is reviewed, landed, or replaced.

## What This Changes

The issue asks for omitted `cost.cacheRead` and `cost.cacheWrite` pricing fields to default to zero so model cost telemetry is still emitted.

## Best Possible Solution

Normalize missing model cost multipliers to zero at the shared pricing boundary, or defensively in `estimateUsageCost`, while preserving tiered pricing behavior. Add focused regression coverage for input/output pricing with omitted cache prices and verify that `costUsd` is emitted for diagnostics when the resulting estimate is finite.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass a cost object with finite `input`/`output` but omitted `cacheRead`/`cacheWrite` through `estimateUsageCost`; current main multiplies by undefined, returns `undefined`, and the OpenTelemetry cost counter is not recorded.

## Solution Assessment

Yes, defaulting omitted cache price fields to zero is the narrow maintainable solution. The safest boundary is shared cost normalization or estimator-side coercion, with tests proving direct config/models.json lookup and diagnostics emission do not regress.
