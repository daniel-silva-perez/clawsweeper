## Summary

Keep open. Current main has several adjacent Windows startup and lazy-loading improvements, but the specific reported failure is not conservatively resolved: the CLI respawn planner still returns `null` on `win32` before it can add any child `execArgv`, and there is no `--stack-size` mitigation in source, tests, or docs. The issue also has two independent Windows reports and an open Microsoft-platform tracker entry, not a fixing PR.

## What This Changes

Investigate and fix native Windows CLI startup crashes and OOMs on heavy commands by adding a Windows-specific V8 stack-size respawn or otherwise reducing the CLI ESM module graph.

## Best Possible Solution

Add a narrow Windows startup mitigation in the existing CLI respawn planner, or an equally narrow lazy-loading fix that prevents affected native-Windows commands from evaluating the oversized graph before startup. The fix should keep existing skip guards, avoid broad plugin/runtime refactors, and add focused unit coverage plus Windows package smoke proof for the reported commands.

## Reproduction Assessment

Yes for the affected releases: the issue includes concrete Windows install/run steps, crash signatures, and a stack-size controlled experiment, with a second user confirming related commands on a later 2026.4.x release. Current-main reproduction is not yet live-verified and should be checked on a Windows runner after a patch.

## Solution Assessment

Unclear but likely: the proposed `buildCliRespawnPlan()` stack-size change is the narrowest direct mitigation for the reported V8 stack limit, but the existing `win32` respawn short-circuit means the patch must prove Windows respawn is safe. Longer-term import-graph slimming can remain follow-up work if the startup flag fixes the crash.
