## Summary

Keep open. This issue is recent, concrete, and has two Windows reports. Current main has useful adjacent startup work, including cold plugin-list paths and the #62320 plugin-loader reentry guard, but I did not find source or test evidence that the reported Windows stack/OOM startup failure is definitively fixed: Windows CLI respawn is still disabled, no `--stack-size` mitigation is present, and `dashboard` still loads the maintenance registrar path called out in the report.

## Best Possible Solution

Keep this issue open. The best fix is either a safe Windows startup mitigation for the large ESM graph, or splitting the affected command registrars so `dashboard`, `plugins list`, `cron list`, and heartbeat/system status paths do not evaluate heavy runtime graphs before execution. Add Windows package/CI smoke coverage for the affected commands and update `src/entry.respawn.test.ts` or command-registration tests to lock in the chosen behavior.
