## Summary

Keep open. The PR addresses a real prompt-disclosure path, but the added substantial-echo guard still has a concrete bypass because it samples BOOT.md windows every 20 characters instead of checking every contiguous 80-character echo window.

## What This Changes

The PR wraps BOOT.md content in internal-runtime-context delimiters, strips internal runtime context from message-tool text fields, adds a per-session boot echo guard, and adds changelog plus unit coverage.

## Best Possible Solution

Keep the PR open and revise the echo guard so substantial-literal-echo detection covers every outbound substring at or above the chosen threshold, preferably with a complete sliding-window or fingerprint approach plus regression tests for unaligned 80-character slices.

## Reproduction Assessment

Yes. Register a boot prompt and send a message-tool text field containing an unaligned 80-character contiguous slice of that prompt; the current PR algorithm can miss it because it only checks every twentieth start offset plus the tail.

## Solution Assessment

No. The layered direction is sound, but the sampled-window algorithm is not reliable enough for prompt-disclosure hardening; the narrow safer alternative is complete window coverage or an equivalent hashed-substring guard.
