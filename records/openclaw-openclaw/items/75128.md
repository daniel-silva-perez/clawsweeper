## Summary

Keep open. Current main does not implement this fix, and the PR targets a real prompt-disclosure path from #53732, but the proposed marker-only suppression appears under-scoped for literal BOOT.md echoes that omit the new delimiter lines.

## What This Changes

The PR wraps BOOT.md content in internal-runtime-context delimiters, strips those blocks from top-level message-tool text fields, and adds changelog plus unit coverage for boot prompt wrapping and message-tool sanitization.

## Best Possible Solution

Extend or revise the PR so BOOT.md runs carry a deterministic suppression signal, such as a boot prompt fingerprint or substantial literal-echo guard, through final replies and message-tool delivery while preserving intentional BOOT.md-directed notifications. The runtime-context delimiters are useful as a defense layer, but should not be the only protection for this prompt-disclosure bug.

## Reproduction Assessment

Yes. A high-confidence mock reproduction is to create BOOT.md, force primary model failure, and have the fallback send the generated BOOT.md prompt or section through the message tool; current main source shows the prompt is embedded plainly and the message tool lacks an internal-context or boot-echo guard.

## Solution Assessment

No. The PR is a useful partial hardening, but relying on the fallback to copy the delimiter lines leaves literal BOOT.md echoes that start at the BOOT.md section or content unmarked and therefore unstripped; a boot-run fingerprint or comparable echo guard is the safer fix.
