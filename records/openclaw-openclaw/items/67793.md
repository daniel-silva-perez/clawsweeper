## Summary

Keep this issue open. Current main has the shared inbound debouncer and Feishu wiring that should batch same-key text messages, but the reporter reproduced the Feishu DM idle-burst case after switching to `messages.inbound.debounceMs`, and the current Feishu regression tests still cover the group-shaped helper path rather than the exact p2p/DM same-sender path. Related #54409 overlaps with active-run Feishu queue serialization, but it does not supersede this idle inbound debounce report.

## What This Changes

The issue reports that Feishu direct messages sent close together with collect/inbound debounce settings are processed as separate replies instead of one batched agent turn.

## Best Possible Solution

Add direct coverage for the Feishu p2p/DM idle-burst path and use the result to either fix the Feishu debounce key/text/effective-window path or add targeted diagnostics/docs for the remaining runtime bypass conditions. Keep active-run queue serialization work coordinated with #54409 instead of changing global queue semantics without a failing shared test.

## Reproduction Assessment

Yes for the reported environment: the reporter provided a concrete Feishu DM idle-burst reproduction after switching to `messages.inbound.debounceMs`, including stable text/chat/sender metadata. The repository does not yet have a high-confidence automated reproduction for that exact p2p path.

## Solution Assessment

Yes, the best next step is a focused Feishu p2p regression before behavioral changes. If that test fails, fix the Feishu handler or shared debouncer narrowly; if it passes, add diagnostics/docs and ask for runtime logs rather than changing queue semantics speculatively.
