## Summary

Keep this issue open. Current main has both the shared inbound debouncer and Feishu wiring that should batch same-key text messages, and the docs/config distinguish idle inbound debouncing from active-run `messages.queue.collect`; however, the reporter reproduced the Feishu DM idle-burst case after switching to `messages.inbound.debounceMs`, and current Feishu debounce tests still do not directly cover that p2p/DM same-sender path. Related #54409 overlaps only with active-run Feishu follow-up queueing and does not fully supersede this idle inbound report.

## What This Changes

The issue reports that rapid Feishu messages are processed as separate agent turns instead of one batched turn when `messages.queue.mode: "collect"` and then `messages.inbound.debounceMs` are configured.

## Best Possible Solution

Add direct coverage for the Feishu p2p/DM idle-burst path and use that result to either fix the Feishu debounce key/text/effective-window path or add targeted diagnostics and docs for the remaining runtime bypass conditions. Keep the active-run queue serialization work coordinated with #54409 instead of folding both behaviors into one unclear fix.
