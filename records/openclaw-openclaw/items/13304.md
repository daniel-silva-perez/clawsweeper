## Summary

Keep open. Current main still implements bindings as static route selectors, and the existing maintainer discussion correctly identifies unresolved routing-contract questions rather than an already-shipped implementation or a narrower superseding tracker.

## What This Changes

Requests optional binding content matchers, such as regex or prefix rules with optional prefix stripping, so inbound messages can route to different agents based on message text.

## Best Possible Solution

Define a first-class, validated binding content matcher with deterministic precedence, safe text normalization and regex constraints, content-aware or bypassed route caching, representative channel/plugin plumbing for normalized inbound text, documented stripping semantics, and regression coverage for static routing plus content-routing combinations.

## Risks / Open Questions

- Precedence needs maintainer design across static binding tiers, runtime /focus or conversation bindings, ACP bindings, mention gating, and the narrower open mention-routing request #38138.
- stripMatch needs an API decision on whether it rewrites only the agent-facing body or also command text, raw body, reply context, grouped history context, and channel-specific inbound metadata.
- Regex or prefix matching needs explicit normalization, length, and safe-regex rules to avoid accidental broad routing or ReDoS-style hazards across channel and plugin inputs.
- Content-dependent routing conflicts with the current static route cache unless the cache key includes normalized content, content rules bypass caching, or cache behavior is split by matcher type.
