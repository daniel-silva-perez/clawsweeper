## Summary

This issue should stay open. Current main still has only static binding match fields, strict config validation rejects a proposed match.content object, the route resolver/cache do not accept message text or return a rewritten body, and the related model-routing and mention-routing items do not cover the broader content-based agent-binding feature.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The best implementation would define a validated content matcher for bindings, decide precedence relative to peer/account/guild/team/role tiers, update strict Zod/schema/help/docs, make route caching content-aware or safely bypassed, pass normalized message text into representative channel/plugin routing calls, and define a safe BodyForAgent rewrite contract if stripMatch remains in scope.

## Risks / Open Questions

- Closing now would lose a valid feature request: match.content and stripMatch are still absent from type, schema, resolver, cache, docs, and representative inbound routing paths.
- A naive implementation could make route caching incorrect unless content is included in the cache key, content-sensitive routes bypass the cache, or matching is split into a separate non-cached stage.
- stripMatch needs explicit body transformation semantics before dispatch, especially for channel envelopes, mention stripping, media/non-text messages, and transcript preservation.
- Regex support needs the same safe-regex and validation posture as existing mentionPatterns to avoid unsafe patterns in config.
