## Summary

Keep open. Current main still supports only static binding selectors, rejects the proposed `match.content` shape under strict config validation, and resolves routes without message text or a rewritten-body result; the maintainer review already called out the same product/API decisions that remain unresolved.

## What This Changes

The issue requests optional content-pattern matching in `bindings` so messages can route to specialist agents by prefix or regex, with optional prefix stripping before the agent turn.

## Best Possible Solution

Define a first-class validated binding content matcher in the public config contract, specify deterministic precedence with existing static bindings and mention-gating behavior, make route resolution and caching content-aware, pass normalized inbound text through representative channel/plugin routing paths, and document exactly what optional body stripping changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add the proposed `match.content` config: the strict binding schema has no such key, and the route resolver cannot evaluate it because callers do not pass message text.

## Solution Assessment

Unclear. A content matcher on bindings is likely the right user-facing surface, but the exact proposal is not yet the best safe implementation until maintainers decide precedence, regex safety, cache behavior, content normalization, and `stripMatch` rewrite semantics.

## Risks / Open Questions

- A content matcher needs deterministic precedence relative to peer, parent-peer, wildcard, role, account, channel, default, and the narrower open mention-routing request #38138.
- Regex and prefix matching need normalization and safety rules before accepting arbitrary message-content patterns in public config.
- `stripMatch` needs a clear contract for whether it rewrites only agent-facing text or also stored raw body, command text, reply context, grouped history, and tool-visible fields.
- Route caching is static-metadata based today, so content-aware routing needs either content-aware cache keys or a documented cache bypass for content rules.
