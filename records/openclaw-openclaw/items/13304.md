## Summary

Keep open. Current main still has only static binding selectors, strict config validation does not admit a `match.content` object, and the resolver/cache contracts do not carry message text or return a rewritten agent body. The related open mention-routing request is narrower, while the older related reports are model-routing requests, so none supersedes this agent-binding feature.

## What This Changes

Add an optional content matcher to agent bindings so messages in the same channel or peer can route to different agents by regex or prefix and optionally strip the matched text before agent dispatch.

## Best Possible Solution

Define a first-class, validated binding content matcher in the agent binding contract, specify deterministic precedence with existing static tiers and mention behavior, make route caching content-aware or inapplicable for content rules, pass normalized inbound text through representative channel/plugin routing calls, and document exactly what `stripMatch` rewrites.

## Risks / Open Questions

- The `stripMatch` behavior needs a product/API decision: whether it rewrites only `BodyForAgent` or also command text, raw body, combined history context, and channel-specific inbound context.
- Content-dependent routing conflicts with the current static route cache unless the cache key includes normalized content, is safely bypassed, or is split by matcher type.
- Regex/prefix matching needs explicit safe-regex, length, and normalization rules to avoid ReDoS risk and accidental broad agent takeover.
