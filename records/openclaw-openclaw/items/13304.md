## Summary

Keep open. Current main still treats bindings as static route selectors, while existing mention-pattern support only gates whether a channel message wakes an agent; content-based agent routing remains unimplemented and needs maintainer API decisions before a safe fix.

## What This Changes

The issue requests optional binding content matchers, such as regex or prefix rules plus optional matched-text stripping, so inbound messages can route to different agents based on message text.

## Best Possible Solution

Define an additive, validated binding content matcher in the public config contract, specify deterministic precedence with existing route tiers and mention behavior, make the resolver/cache contract content-safe, plumb normalized inbound text through representative channel paths, document `stripMatch`, and add regression coverage for static and content-rule combinations.
