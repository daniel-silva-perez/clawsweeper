## Summary

Keep open. Current main still exposes bindings as static route selectors, rejects a `match.content` object in strict config validation, and resolves routes without message text or rewritten body output. The open mention-routing request is narrower, while the older related reports are about model routing or middleware, so none supersedes this feature.

## What This Changes

The issue requests a binding-level content matcher so inbound messages can route to different agents by regex or prefix and optionally strip the matched text before delivery.

## Best Possible Solution

Define a first-class validated binding content matcher with deterministic precedence, safe regex/prefix semantics, content-aware or bypassed route caching, representative channel/plugin body plumbing, documented `stripMatch` behavior, and regression coverage for static and content-routing combinations.

## Risks / Open Questions

- Precedence still needs a maintainer API decision across static binding tiers, mention gating, runtime conversation bindings, ACP bindings, focus/thread inheritance, and default routing.
- Regex or prefix matching needs explicit normalization, length, and safe-regex rules before accepting user-controlled message text in routing decisions.
- `stripMatch` needs a defined rewrite boundary for agent-facing body text, raw body metadata, command parsing, reply context, grouped history, and channel-specific envelopes.
- Route caching and representative channel/plugin plumbing must be updated together; a partial implementation could reuse stale route decisions or strip content on only some inbound surfaces.
