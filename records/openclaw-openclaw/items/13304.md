## Summary

Keep open. Current main still has only static binding selectors, strict config validation would reject a proposed `match.content` object, `resolveAgentRoute` has no message-body input or rewritten-body output, and representative channel call sites still resolve routes from static metadata only. The already-linked #38138 is a narrower mention-routing request, while #11220 and #10969 were about model routing, so none supersedes this broader binding feature.

## What This Changes

The issue requests optional content-pattern matching in `bindings` so messages can route to specialist agents by prefix or regex, with optional prefix stripping before the agent turn.

## Best Possible Solution

Define a first-class validated binding content matcher as part of the public config contract, specify deterministic precedence with existing static bindings and mention-gating behavior, make route resolution and caching content-aware, pass normalized inbound text through representative channel/plugin routing paths, and document exactly what optional body stripping changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add the proposed `match.content` config: the strict binding schema has no such key, and the route resolver cannot evaluate it because callers do not pass message text.

## Solution Assessment

Unclear. A content matcher on bindings is likely the right user-facing surface, but the exact proposal is not yet the best safe implementation until maintainers decide precedence, regex safety, cache behavior, content normalization, and `stripMatch` rewrite semantics.

## Risks / Open Questions

- A content matcher needs a deterministic precedence decision relative to peer, parent-peer, wildcard, role, account, channel, default, and the narrower open mention-routing request #38138.
- Regex and prefix matching need normalization and safety rules before accepting arbitrary message-content patterns in public config.
- `stripMatch` needs a clear contract for whether it rewrites only the agent-facing text or also stored raw body, command text, reply context, grouped history, and tool-visible message fields.
- Route caching is currently static-metadata based, so content-aware routing needs either content-aware cache keys or a documented cache bypass for content rules.
