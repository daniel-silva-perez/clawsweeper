## Summary

Keep this issue open. Current main still has only static binding match fields and representative channel call sites do not pass message text into the route resolver, so declarative content-based agent routing and optional prefix stripping remain unimplemented. The existing April 26, 2026 maintainer/Codex review also explicitly kept this open and treated #10969/#11220 as related model-routing context rather than superseding this agent-routing request.

## Best Possible Solution

Keep this issue open for explicit maintainer product/design work. The likely path is a core binding or pre-routing design that adds a validated content matcher to AgentBindingMatch, updates strict Zod validation, generated schema/help/docs, route precedence and cache keys, representative channel/plugin routing calls, and tests. If stripMatch remains in scope, the design should also define a safe BodyForAgent transformation contract before dispatch. Treat #10969 and #11220 as related model-routing context, not as canonical replacements.

## Risks / Open Questions

- Content-sensitive routing would need a deterministic precedence decision relative to existing peer, guild+roles, guild, team, account, and channel tiers.
- Regex/prefix matching needs validation and safety limits before accepting arbitrary patterns in gateway config.
- stripMatch implies a message-body transformation contract across channel/plugin call sites, not just a resolver field.
- The current route cache and channel call sites are static; adding message content without cache-key and call-site changes could misroute later messages.
