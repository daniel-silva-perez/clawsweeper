## Summary

Keep this issue open. Current main still only supports static binding match dimensions, and representative inbound route calls do not pass message text into the route resolver, so content-based agent routing and optional prefix stripping remain unimplemented. The existing maintainer/Codex comment also kept this open and treated #10969 and #11220 as related model-routing context, not as replacements for this agent-routing request.

## Best Possible Solution

Keep this issue open for explicit maintainer product/design work. A good implementation would add a validated content matcher to AgentBindingMatch, update strict Zod validation, generated schema/help/docs, route precedence and cache keys, representative channel/plugin routing calls, and focused tests. If stripMatch remains in scope, the design should define a safe BodyForAgent transformation contract before dispatch. Treat #10969 and #11220 as related model-routing context, not canonical replacements.

## Risks / Open Questions

- Content-sensitive routing needs a deterministic precedence decision relative to peer, parent peer, wildcard peer, guild+roles, guild, team, account, and channel tiers.
- Regex matching needs validation and safety limits before accepting arbitrary operator-configured patterns in gateway config.
- stripMatch implies a message body transformation contract across channel/plugin inbound paths, not only a resolver field.
- The current resolved-route cache is keyed only by static route metadata; adding content routing without cache redesign could misroute later messages.
