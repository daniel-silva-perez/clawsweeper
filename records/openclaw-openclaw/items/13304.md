## Summary

Keep open. Current main still treats bindings as static route selectors: the public binding type and strict schema do not accept a content matcher, the route resolver does not receive message text or return a rewritten body, and representative channel paths resolve the route from channel/account/peer metadata before body-specific handling. The existing related reports do not supersede this because #38138 is a narrower mention override and #10969/#11220 are model-routing requests.

## What This Changes

The issue requests an optional `bindings[].match.content` matcher with regex or prefix support and optional `stripMatch` so messages in the same channel/peer can route directly to different agents based on message text.

## Best Possible Solution

Define a first-class validated binding content matcher that is additive to the public config contract, specify deterministic precedence with existing route tiers and mention behavior, make route caching content-aware or bypassed for content rules, plumb normalized inbound text through representative channel/plugin routing paths, document `stripMatch`, and add regression coverage for static/content combinations.

## Risks / Open Questions

- Precedence still needs a maintainer API decision across static binding tiers, mention gating, runtime conversation bindings, ACP bindings, focus/thread inheritance, and default routing.
- Regex or prefix matching needs explicit normalization, length limits, and safe-regex rules before user-controlled message text is accepted in routing decisions.
- `stripMatch` needs a defined rewrite boundary for agent-facing body text versus raw body, command parsing, reply context, transcripts, broadcast history, and audit/debug data.
- Content-dependent routing needs an explicit cache strategy to avoid stale static-route hits or unbounded cache growth.
