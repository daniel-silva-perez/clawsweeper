## Summary

Keep open. Current main still only supports static binding selectors, strict config validation does not admit a content matcher, and the route resolver/cache/channel call sites do not carry message text or return a rewritten body. The related open mention-routing request is narrower, and the older related reports are about model routing, so none supersedes this feature.

## What This Changes

Requests adding a content matcher to agent bindings so inbound messages can route to different agents by regex or prefix and optionally strip the matched text before dispatch.

## Best Possible Solution

Define a first-class validated binding content matcher, specify deterministic precedence with existing static routing and mention behavior, make route caching content-aware or bypassed for content rules, pass normalized inbound text through representative channel/plugin routing paths, and document exactly what optional body stripping changes.

## Risks / Open Questions

- `stripMatch` needs a product/API decision on whether it rewrites only the agent-facing body or also command text, raw body, reply context, grouped history context, and channel-specific inbound metadata.
- Content-dependent routing conflicts with the current static route cache unless the cache key includes normalized content, content rules bypass caching, or cache behavior is split by matcher type.
- Regex or prefix matching needs explicit normalization, length, and safe-regex rules to avoid accidental broad routing and ReDoS-style hazards across channel/plugin inputs.
- Precedence needs maintainer design across existing static tiers, runtime `/focus` or conversation bindings, ACP bindings, mention gating, and the narrower open mention-routing request #38138.
