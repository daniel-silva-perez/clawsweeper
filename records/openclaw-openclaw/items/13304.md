## Summary

Keep open. Current main still treats bindings as static route metadata: the public binding type and strict config schema have no content matcher, the resolver has no message-text input or rewritten-body output, and representative channel paths resolve routes before content/mention handling. The maintainer discussion already kept this open for API/product decisions, and the narrower open mention-routing request does not supersede this broader content-pattern binding feature.

## What This Changes

Request to extend routing bindings so inbound message content patterns can select an agent and optionally strip the matched prefix before forwarding.

## Best Possible Solution

Define a first-class validated binding content matcher in the public config contract, specify deterministic precedence with existing static bindings and mention-gating behavior, make route resolution and caching content-aware, pass normalized inbound text through representative channel/plugin paths, and document exactly what optional body stripping changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add the proposed match.content config: the strict binding schema has no such key, and the route resolver cannot evaluate it because callers do not pass message text.

## Solution Assessment

Unclear. A content matcher on bindings is a plausible user-facing surface, but the exact implementation needs maintainer decisions on precedence, regex safety, cache behavior, content normalization, and stripMatch rewrite semantics before it is the safest fix.
