## Summary

Keep open. Current main still only supports static binding match fields, strict config validation rejects a proposed `match.content` object, and `resolveAgentRoute` has no message-content input or rewritten-body output. Related #38138 is a narrower mention-routing override, while #10969 and #11220 are model-routing requests, so none supersedes this broader content-based agent binding feature.

## What This Changes

Add content-based binding matchers, such as prefix or regex patterns with optional `stripMatch`, so inbound messages can route to different agents based on message text.

## Best Possible Solution

Keep this open for maintainer product/design work. A good implementation should define a validated content matcher for route bindings, decide precedence across existing static tiers and mention behavior, make route caching content-aware or safely bypassed, pass normalized message text into representative channel/plugin routing calls, define whether `stripMatch` rewrites `BodyForAgent` only or a broader inbound context, update config schema/help/docs/generated metadata, and add resolver plus validation coverage.

## Risks / Open Questions

- Precedence relative to peer, parent-peer, wildcard, guild/role, account, channel, mention gating, runtime conversation bindings, and ACP bindings needs maintainer product judgment.
