## Summary

Keep open: current main and the latest release still let `defaultProfile: "user"` select the synthesized Chrome MCP existing-session profile, so the top-level `browser.cdpUrl` does not control the selected runtime path for the reported config. The closed-unmerged #48068 proposed the right narrow area to fix, and shipped per-profile existing-session `cdpUrl` support is only a workaround, not a fix for this top-level config contract.

## What This Changes

The issue asks for top-level `browser.cdpUrl` to be honored as the browser connection target when `defaultProfile` is `"user"` and no explicit `profiles.user` entry is configured.

## Best Possible Solution

Update the browser config resolver so top-level `browser.cdpUrl` cannot be silently bypassed by the implicit `user` Chrome MCP profile. The safest end state is to preserve explicit `profiles.user` behavior, but when `profiles.user` is absent and top-level `browser.cdpUrl` is set, route the default to the CDP-backed `openclaw` profile or another documented CDP-backed profile, with a regression test for the reported config.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to resolve the reported config through `resolveBrowserConfig` and `resolveProfile`; current main selects `user`, resolves it as `existing-session`, and leaves its `cdpUrl` empty, which then routes through Chrome MCP.

## Solution Assessment

Yes, a #48068-style resolver fix is the narrowest maintainable solution. It should not remove explicit `profiles.user` or the shipped per-profile existing-session `cdpUrl` support; it only needs to prevent the implicit `user` profile from overriding the top-level CDP contract.
