## Summary

Current main still routes the reported config to the built-in `user` existing-session profile, not to the top-level `browser.cdpUrl`. The related fix PR was closed unmerged, and the latest release has the same resolver behavior as current main for this surface.

## Best Possible Solution

Keep this issue open. Revive or reimplement the essence of #48068 in `extensions/browser/src/browser/config.ts`: when top-level `browser.cdpUrl` is set and there is no explicit `profiles.user`, avoid silently routing `defaultProfile: "user"` to the synthesized Chrome MCP profile, or intentionally rewrite it to the CDP-backed `openclaw` profile. Add a regression test for the exact reported config and update browser docs if maintainers choose a different supported configuration shape.
