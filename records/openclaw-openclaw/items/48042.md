## Summary

Current main still routes the reported config to the synthesized `user` existing-session profile when `browser.cdpUrl` and `defaultProfile: "user"` are both set without an explicit `profiles.user`. That profile has no inherited top-level CDP URL, uses Chrome MCP, and can still surface the reported DevToolsActivePort failure. The focused fix PR #48068 was closed unmerged, and the latest release tag keeps the same resolver behavior.

## Best Possible Solution

Keep this issue open. Reimplement the core of #48068 in `extensions/browser/src/browser/config.ts`: when top-level `browser.cdpUrl` is set and there is no explicit `profiles.user`, avoid silently selecting the synthesized Chrome MCP `user` profile for `defaultProfile: "user"`, or intentionally rewrite that default to the CDP-backed `openclaw` profile. Add a regression test for the exact reported config and update the public browser docs if maintainers choose a different supported configuration shape.
