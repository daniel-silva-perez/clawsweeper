## Summary

Current main still appears to route the reported config to the built-in `user` existing-session profile when `browser.cdpUrl` and `defaultProfile: "user"` are both set. That profile does not inherit the top-level CDP URL and uses Chrome MCP, which matches the reported DevToolsActivePort fallback path. The focused fix PR #48068 was closed unmerged, and the latest release has the same resolver behavior for this surface.

## Best Possible Solution

Keep this issue open. Reimplement the core of #48068 in `extensions/browser/src/browser/config.ts`: when top-level `browser.cdpUrl` is set and there is no explicit `profiles.user`, avoid silently selecting the synthesized Chrome MCP `user` profile for `defaultProfile: "user"`, or intentionally rewrite that default to the CDP-backed `openclaw` profile. Add a regression test for the exact reported config and update `docs/tools/browser.md` if maintainers choose a different supported configuration shape.
