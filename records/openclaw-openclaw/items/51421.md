## Summary

Current main still has the counter-sync behavior this PR targets: after a memory-flush-triggered compaction, `runMemoryFlushIfNeeded` writes the post-increment `compactionCount` into `memoryFlushCompactionCount`, and the flush gate then treats the next eligible cycle as already flushed. The PR should stay open for maintainer review/landing rather than be closed as obsolete.

## What This Changes

The PR preserves the pre-increment memory flush compaction counter after flush-triggered compaction, updates the runner expectation, adds consecutive-cycle coverage, and adds a changelog entry.

## Best Possible Solution

Keep this PR open and land this narrow fix, or an equivalent maintainer patch, after required CI is green or explicitly accepted. The right boundary is to preserve the pre-increment `memoryFlushCompactionCount`, update the persisted-counter expectation from `2` to `1`, keep the consecutive-cycle regression test, and let #12590 close through the normal merge path.
