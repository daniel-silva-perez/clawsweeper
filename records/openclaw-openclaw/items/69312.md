## Summary

Keep this PR open: current main still has the reported indented-code MEDIA false-positive, and the branch is an active focused fix candidate, but the latest patch still introduces parser compatibility regressions that should block merge until addressed.

## What This Changes

The PR changes media and fence parsing to skip MEDIA directives in indented code blocks, preserve retained code indentation during cleanup, tighten fenced-code closing validation, make MEDIA token matching case-sensitive, and add media/fence regression tests.

## Best Possible Solution

Land a focused version that keeps the indented-code protection and code-aware whitespace cleanup, while preserving or deliberately updating MEDIA case semantics across splitMediaFromOutput, block-reply parsing, streaming buffering, docs, and tests. If the fence parser tightening remains, validate the shared markdown span consumers in the same review path.

## Reproduction Assessment

Yes. The reported bug has a source-level reproduction on current main: a 4-space or tab-indented MEDIA line outside a fenced block reaches the trimStart/toUpperCase MEDIA gate. The PR regressions are also source-backed by existing lowercase block-reply coverage, current streaming buffering, and the PR audio cleanup regex.

## Solution Assessment

No. The indented-code heuristic is the right direction, but the latest patch is not the best mergeable solution until lowercase MEDIA compatibility and audio-tag code-aware normalization are aligned with the rest of the directive pipeline.
