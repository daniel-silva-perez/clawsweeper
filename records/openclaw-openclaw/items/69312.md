## Summary

Keep open. Current main still has the reported indented-code MEDIA extraction bug, and this PR is useful source work, but the latest patch still has blocking parser compatibility regressions plus a required changelog gap before it should merge.

## What This Changes

The PR updates media and fence parsing plus tests to preserve indented-code MEDIA lines, make cleanup more code-aware, tighten malformed fence-closing handling, and change MEDIA matching to uppercase-only.

## Best Possible Solution

Land a focused repair that keeps the indented-code protection and fence validation, preserves existing lowercase MEDIA compatibility across splitMediaFromOutput, block replies, and streaming buffering, makes post-audio cleanup respect the first surviving code line, and adds the required Unreleased changelog entry.

## Reproduction Assessment

Yes. The main-branch bug is reproducible by passing splitMediaFromOutput text containing a blank line followed by a 4-space or tab-indented MEDIA line; the PR regressions are source-backed by current lowercase block-reply coverage and by the new post-audio leading-normalization regex.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right direction, but the uppercase-only directive change and non-code-aware audio cleanup are not the narrowest maintainable fix for the reported bug.
