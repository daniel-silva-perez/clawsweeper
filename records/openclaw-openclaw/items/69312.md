## Summary

Keep this PR open. Current main still has the reported indented-code MEDIA extraction bug, but the current PR branch is not mergeable as-is because it changes lowercase MEDIA directive semantics without updating the block-reply and streaming directive surfaces, and it still has a code-indentation cleanup gap after audio tag removal.

## What This Changes

The PR updates media parsing to skip MEDIA directives in indented code blocks, preserve retained code indentation, tighten fenced-code closing validation, make MEDIA token matching case-sensitive, and add media/fence regression tests.

## Best Possible Solution

Land a focused version that preserves the indented-code protection and code-aware whitespace cleanup, while keeping MEDIA directive case handling consistent across splitMediaFromOutput, block-reply parsing, streaming buffering, docs, and tests. If lowercase support is intentionally being removed, that should be a deliberate compatibility change across all directive surfaces in the same PR.

## Reproduction Assessment

Yes. The reported bug has a source-level reproduction on current main: a 4-space or tab-indented MEDIA line outside a fenced block reaches the trimStart/toUpperCase MEDIA gate. The PR regressions are also source-backed by existing lowercase block-reply coverage and the still-case-insensitive streaming buffer.

## Solution Assessment

No. The indented-code heuristic is the right direction, but the proposed case-sensitivity change and audio tag cleanup path are not the narrowest maintainable fix until they are aligned with the rest of the directive pipeline.
