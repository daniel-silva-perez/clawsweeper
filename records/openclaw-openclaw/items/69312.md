## Summary

Keep this PR open. Current main still has the MEDIA false-positive path the PR targets: `splitMediaFromOutput` protects fenced code only, then trims indentation before checking for `MEDIA:`, so 4-space or tab-indented code lines can still be parsed as media directives. The linked issue #69311 is closed, but current main does not implement this PR's intended fix.

## Best Possible Solution

Keep the PR open for maintainer review or a corrected maintainer-apply. The best path is to add indented-code protection in `src/media/parse.ts`, preserve retained code indentation during media cleanup, add focused media and fence regression tests, and explicitly settle MEDIA case semantics across `splitMediaFromOutput`, streaming directive buffering, and reply delivery tests before merging.

## Risks / Open Questions

- The PR changes shared `parseFenceSpans` behavior used by markdown code spans, auto-reply chunking, canvas rendering, and the embedded block chunker, so those consumers need maintainer review before merge.
- The PR's MEDIA case-sensitivity changes should be aligned with streaming directive buffering, where current main still uses case-insensitive MEDIA tail detection.
- The indented-code heuristic is intentionally approximate; list and blockquote container contexts may need explicit tests or documented tradeoffs.
