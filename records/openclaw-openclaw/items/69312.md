## Summary

Keep this PR open. Current main still has the false-positive path the PR targets: `splitMediaFromOutput` only protects fenced code, trims indentation before checking for `MEDIA:`, and then globally folds retained text, so 4-space/tab indented code blocks can still be stripped and treated as media. The PR is still a plausible implementation candidate, but it needs maintainer review around shared fence parsing and MEDIA case-sensitivity compatibility before merge.

## Best Possible Solution

Keep the PR open for maintainer review or a maintainer-applied version. The best path is to add indented-code protection in `src/media/parse.ts`, preserve retained code indentation during cleanup, add focused media and fence regression tests, and settle the lowercase `media:` compatibility question across `splitMediaFromOutput`, streaming directive buffering, and reply delivery tests before merging.

## Risks / Open Questions

- `parseFenceSpans` is shared by media parsing, markdown code-span indexing, auto-reply chunking, canvas shortcode extraction, and the embedded block chunker, so the closing-fence semantic change needs focused maintainer review across those consumers.
- The PR's MEDIA case-sensitivity changes appear to conflict with current lowercase-media tests and case-insensitive streaming buffering on main.
- The indented-code heuristic is intentionally approximate; list and blockquote container contexts may need explicit tests or a documented tradeoff before landing.
