## Summary

Keep this PR open. Current main still has the reported indented-code MEDIA false positive, and the PR is a useful active contributor fix, but the latest diff still appears to make MEDIA directive case handling inconsistent across parser, block-reply, and streaming paths.

## What This Changes

The PR updates media and fence parsing plus tests to skip MEDIA extraction in indented code blocks, preserve retained code indentation, validate fence closers more strictly, and make MEDIA matching case-sensitive.

## Best Possible Solution

Land a focused version that keeps the indented-code protection and code-aware whitespace cleanup, while preserving existing lowercase MEDIA compatibility or deliberately updating every directive surface, doc, and test in one coherent compatibility change.

## Reproduction Assessment

Yes. The main-branch bug is reproducible from source by passing splitMediaFromOutput text with a blank line followed by a 4-space or tab-indented MEDIA line; the PR regression is source-backed by existing lowercase block-reply coverage and the still-case-insensitive streaming buffer.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right direction, but the case-sensitivity change is not the narrowest maintainable fix unless block-reply parsing, streaming buffering, docs, and tests are aligned too.
