## Summary

Keep this PR open. It addresses a real current-main media-parser bug and the latest head fixes the earlier lowercase `media:` compatibility concern, but the current diff still introduces a parser regression for unquoted bare filenames containing spaces.

## What This Changes

The PR adds indented-code detection to `splitMediaFromOutput`, preserves retained code-line whitespace during media cleanup, tightens Markdown fence-closing validation, keeps MEDIA directives case-insensitive, and adds media/fence regression tests.

## Best Possible Solution

Land a focused parser fix after preserving existing spaced bare-filename behavior, keeping lowercase MEDIA semantics aligned across block replies and streaming, and validating the shared fence-parser consumers touched by the stricter closer rule.

## Reproduction Assessment

Yes. Current main can be reasoned from source to reproduce the original bug with a blank-line-preceded `    MEDIA: /tmp/example.png`, and the remaining PR regression can be checked with `MEDIA:My File.png` against the current fallback and the PR diff.

## Solution Assessment

No, not yet. The indented-code approach is a reasonable narrow fix, but the final payload-start gate should not remove the existing whole-payload bare-filename fallback for names with spaces unless that compatibility break is intentional and covered.

## Review Findings
