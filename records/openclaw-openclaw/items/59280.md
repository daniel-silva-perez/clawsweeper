## Summary

Keep open. Current main still exposes only `openclaw backup create` and `openclaw backup verify`, so this PR is not obsolete, and the linked OpenClaw Snapshot skill is a useful workaround rather than a replacement for the core CLI restore path. The PR direction is valid, but the current diff has blocking restore-safety and archive-integrity findings before it should merge.

## What This Changes

The PR adds `openclaw backup restore <archive>` with dry-run, force, and JSON modes, shared archive-reading helpers, path decoding utilities, and restore command/unit tests.

## Best Possible Solution

Keep a core `openclaw backup restore` implementation, but revise it against current main so restore planning validates archive integrity at least as strictly as verify, derives or cross-checks destinations from encoded payload paths, handles both file and directory assets, rechecks conflicts immediately before writes, documents the command, and adds focused regression coverage for config-only, empty-directory, tampered-manifest, duplicate-entry, and force-overwrite cases.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting the CLI registration/docs, and the key PR defects are reproducible by static path through the new restore code plus existing `--only-config` backup behavior.

## Solution Assessment

No. The proposed core command is the right product direction, but this diff is not yet the best maintainable solution because it leaves archive trust and file/empty-directory restore semantics unsafe; the safer path is to revise the PR around the existing backup archive contract and staging helpers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive restore targets from the encoded payload path:** `src/infra/backup-restore.ts:99-100`
  - body: `planRestore` trusts `manifest.assets[].sourcePath` as the write destination. A tampered archive can keep a valid payload path but rewrite `sourcePath`, causing restore or `--force` rename/write operations outside the archived source location. Decode the payload path and reject manifests where it does not match before writing.
  - confidence: 0.93
- **[P1] Restore file and empty-directory assets:** `src/infra/backup-restore.ts:173-183`
  - body: This path treats an empty staging directory as corruption and later always creates `asset.restorePath` as a directory. Valid backups can contain file assets from `backup create --only-config` and empty workspace directories, so restore will fail or use the wrong destination shape for archives produced by current backup create.
  - confidence: 0.9
- **[P2] Recheck conflicts before writing each asset:** `src/infra/backup-restore.ts:131-133`
  - body: `executeRestore` relies on the conflict bit captured earlier in `planRestore`. If a target appears after planning, the restore writes into it without `--force` and without a `.pre-restore-*` snapshot, breaking the advertised overwrite-safety contract.
  - confidence: 0.86
- **[P2] Reject duplicate normalized entries before restore:** `src/infra/backup-restore.ts:74-80`
  - body: Restore planning collapses normalized entries into a `Set` before validation. Unlike `backup verify`, this accepts duplicate archive paths and can restore whichever duplicate the tar reader applies last, so restore can accept archives that verification would reject.
  - confidence: 0.84
- **[P2] Reject empty asset manifests before success:** `src/infra/backup-restore.ts:97-121`
  - body: A manifest with an empty `assets` array returns a normal restore plan and formats as `No assets to restore`. `backup create` does not produce useful zero-asset archives, so a manifest-only or tampered archive should fail instead of reporting a successful no-op restore.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds a local restore command that writes filesystem state, and the current implementation still has concrete archive-trust and overwrite-safety concerns.

Concerns:

- **[high] Manifest-controlled restore destination:** `src/infra/backup-restore.ts:99`
  - body: Restore writes to `asset.sourcePath` from `manifest.json` without deriving or validating it against the encoded payload path. A modified archive can redirect restore writes or `--force` snapshot renames to another path writable by the user.
  - confidence: 0.93
- **[medium] Duplicate archive entries accepted during restore:** `src/infra/backup-restore.ts:74`
  - body: Restore planning converts normalized entries to a `Set` and does not duplicate-check before validation, so crafted duplicate payload entries can bypass the stricter verify behavior and restore non-deterministic content.
  - confidence: 0.84
- **[medium] Overwrite check can go stale before extraction:** `src/infra/backup-restore.ts:131`
  - body: Restore decides whether `--force` is required from the earlier plan rather than rechecking the target path at execution time, allowing a late-created path to be overwritten without a safety snapshot.
  - confidence: 0.86
