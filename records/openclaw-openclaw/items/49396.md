## Summary

Keep this PR open. Current main still uses the proportional-allocation path for Peak Error Hours and does not expose precise hourly or quarter-hour message counts, so the PR is not obsolete or implemented on main. The discussion is also active: a maintainer requested targeted changes, and the author replied that those were amended into the current head.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land this or an equivalent precise hourly aggregation fix that adds backend per-bucket message/error counts, uses them in `buildPeakErrorHours` for Local and UTC views, preserves the legacy fallback, and keeps the added backend/frontend regression tests green.
