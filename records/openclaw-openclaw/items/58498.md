## Summary

Keep this issue open: current main still displays auth labels from the session-selected profile while provider usage resolution only receives the provider and auth-store order, and an open unmerged PR (#59208) explicitly targets this issue with closing syntax.

## What This Changes

The issue asks for status/session usage quota to resolve against the same active `openai-codex` OAuth profile shown in the status card.

## Best Possible Solution

Land a narrow provider-usage fix that carries the session-selected OAuth profile into usage auth resolution, validates that the preferred profile belongs to the requested provider, and preserves existing provider-order fallback when no valid preferred profile is available.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by configuring two `openai-codex` OAuth profiles where the session has `authProfileOverride` set to the second profile while usage auth order resolves the first profile; current main has exactly that split path.

## Solution Assessment

Yes. Threading a validated preferred profile through `loadProviderUsageSummary` and `resolveProviderAuths` is the narrow maintainable fix because it aligns the usage credential with the already-rendered auth label without changing provider fallback behavior.
