## Summary

Keep open: current `main` still has the reported status/usage mismatch path, and there is an open closing PR, #59208, that specifically targets this issue. The source shows the auth label uses the session profile override, while provider usage resolution still resolves credentials only by provider order.

## What This Changes

The issue asks for the status/session usage quota line to use the same OpenAI Codex OAuth profile that the status card displays as active for the current session.

## Best Possible Solution

Review and land #59208 or an equivalent narrow fix that passes the current session OAuth profile into provider usage resolution, validates that the preferred profile belongs to the requested provider, preserves existing provider fallback order, and adds regression coverage for preferred-profile selection and mismatched-provider fallback.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure two OAuth profiles for the same provider, make the session override point at the second profile, and observe that the label path honors the override while the usage path currently resolves by provider order only.

## Solution Assessment

Yes, the approach described by the open PR is the narrow maintainable solution: reuse the existing `preferredProfile` auth-order seam and thread it through status usage loading. The issue should remain open until that PR or an equivalent fix merges.
