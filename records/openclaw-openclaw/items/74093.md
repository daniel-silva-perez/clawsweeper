## Summary

Keep this issue open. Current main still computes reset-triggered state but does not feed that state into the active-run queue policy, and the issue is already paired with open PR #74144 as the implementation candidate.

## What This Changes

The issue reports that reset-triggered `/new` and `/reset` turns can be queued behind an active run under `messages.queue.mode: steer` instead of interrupting and starting a fresh session promptly.

## Best Possible Solution

Review and land PR #74144 or an equivalent focused core reply-queue fix that makes effective reset-triggered turns bypass steer/followup enqueueing, clear queued session-lane work, abort the active run with interrupt semantics, and preserve bare reset acknowledgements. Split the later startup/health symptoms into a narrower follow-up if they remain after the queue fix.

## Reproduction Assessment

Yes. The static current-main path is high-confidence: under an active run, `/new` or `/reset` can set reset state, but active `steer`/followup queue policy still resolves from `resolvedQueue.mode` and can enqueue instead of interrupting.

## Solution Assessment

Yes. A narrow core queue-policy and runner change is the best maintainable direction; the open linked PR appears to target that boundary, so the issue should stay open while that PR is reviewed rather than spawning a competing repair.
