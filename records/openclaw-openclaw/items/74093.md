## Summary

Keep open. Current main still detects reset-triggered turns but routes active-run queue decisions through the configured steer/followup mode, and open PR #74144 is already the linked implementation candidate using closing syntax, so this issue should not be closed before that PR lands or is rejected.

## What This Changes

Reset-triggered `/new` and `/reset` turns should bypass steer/followup queueing and interrupt or clear active session work immediately.

## Best Possible Solution

Review and land #74144, or an equivalent focused core reply-queue fix, so effective reset-triggered turns use interrupt semantics, suppress steer/followup handling for that turn, clear queued lane work, abort the active run, and add regression coverage across steer, steer-backlog, followup, and collect. Split any remaining Feishu startup or session-index consistency symptoms into a separate issue only if they still reproduce on current main after that queue repair.

## Reproduction Assessment

Yes. The high-confidence static path is an active session run with `messages.queue.mode` set to `steer` or a followup-like mode, followed by an authorized `/new` or `/reset`; current main sets `resetTriggered` but still derives the active-run action from the configured queue mode.

## Solution Assessment

Yes for the proposed direction. Threading reset state through the existing queue-policy and prepared-run cleanup seams is the narrowest maintainable fix; broad Feishu startup changes should stay separate unless they are proven to share the same root cause.
