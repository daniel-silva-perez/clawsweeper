## Summary

Keep open. Current main still has the reset state needed for `/new` and `/reset`, but the active-run queue policy and both runner call sites still derive steer/followup behavior from the configured queue mode without a reset override. Open PR #74144 is the linked closing implementation candidate and should be reviewed or rebased instead of closing this issue early.

## What This Changes

Reset-triggered turns such as `/new` and `/reset` should interrupt active work and start a fresh session immediately instead of being steered or queued behind the active run.

## Best Possible Solution

Land a focused shared reply-queue fix, likely by reviewing and rebasing #74144, that treats reset-triggered turns with interrupt semantics, suppresses steer/followup handling for that turn, clears queued lane work, aborts the active run, and adds regression tests across steer, steer-backlog, followup, and collect. Split the Feishu startup/session-index symptoms into narrower follow-up issues only if they reproduce after the queue fix.

## Reproduction Assessment

Yes. The high-confidence static reproduction is an authorized `/new` or `/reset` while a session run is active with `messages.queue.mode` set to `steer`, `followup`, `collect`, or `steer-backlog`; current main sets `resetTriggered` but still routes the active-run policy from the configured queue mode.

## Solution Assessment

Yes for the queue portion. Threading reset state through the existing queue-policy seam and prepared-run cleanup is the narrow maintainable fix; broad Feishu startup changes should not be mixed into that patch unless separately proven to share the root cause.
