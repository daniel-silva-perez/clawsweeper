## Summary

Keep open. Current main still computes reset-triggered state but the active-run queue policy does not consume it, so reset-triggered turns can still follow the configured steer/followup enqueue path. The issue is also paired with open PR #74144 using closing syntax, so maintainer review of that PR is the right next step rather than closing early or creating a separate automated repair job.

## What This Changes

The issue requests that `/new`, `/reset`, and reset-hook turns interrupt or clear active queued work and start a fresh session immediately instead of waiting behind active work under `steer`, `followup`, `collect`, or `steer-backlog` queue modes.

## Best Possible Solution

Review and land #74144 or an equivalent core reply-queue fix that makes effective reset-triggered turns bypass steer/followup enqueueing, clear the session lane, abort the active run with interrupt semantics, preserve bare reset acknowledgements, and cover `steer`, `steer-backlog`, `followup`, and `collect` with regression tests. Any remaining Feishu startup/readiness symptoms should be split into narrower follow-ups after the reset scheduling bug is fixed.
