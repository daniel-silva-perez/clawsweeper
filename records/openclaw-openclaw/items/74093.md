## Summary

Keep open. Current main still has the reported reset state but does not feed it into the active-run queue decision, and the linked open PR #74144 is the implementation candidate for the exact queue/reset path rather than a reason to close before merge.

## What This Changes

Reset-triggered turns such as `/new` and `/reset` should interrupt or clear active queued work instead of being delayed by `steer`, `followup`, `collect`, or `steer-backlog` queue behavior.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that gives effective reset-triggered turns interrupt semantics, suppresses steering/followup handling for that turn, clears queued lane work, aborts the active run, and adds regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. If the later Feishu startup/session-index symptoms persist afterward, split them into a focused Feishu/gateway startup follow-up instead of expanding the queue fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is active work in a session, `messages.queue.mode` set to `steer` or another followup-producing mode, then an authorized `/new` or `/reset`; current main sets `resetTriggered` but still chooses the active-run queue action from the configured queue mode.

## Solution Assessment

Yes. Threading reset state through the existing queue-policy seam and prepared-run cleanup is the narrowest maintainable solution for the reported queue delay; the Feishu startup probe behavior should stay separate unless it remains after the queue repair.
