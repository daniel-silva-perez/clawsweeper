## Summary

Keep open. Current main still matches the reported bug: reset-triggered turns are detected, but the active-run queue decision still ignores reset state and follows the configured `steer`/followup mode; the linked open PR #74144 is the implementation candidate, not a reason to close before merge.

## What This Changes

The issue requests that `/new` and `/reset` reset-triggered turns bypass active-run steer/followup queueing and interrupt or clear current work immediately.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that gives effective reset-triggered turns interrupt semantics, suppresses steering/followup handling for that turn, clears queued lane work, aborts the active run, and adds regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. If the later Feishu startup/session-index symptoms persist afterward, split them into focused follow-up issues rather than expanding the queue fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active run in a session, `messages.queue.mode` set to `steer` or another followup-producing mode, and an authorized `/new` or `/reset`; current main sets `resetTriggered` but still chooses the active-run queue action from the configured queue mode.

## Solution Assessment

Yes. Threading reset state through the existing queue-policy seam and prepared-run cleanup is the narrowest maintainable fix for the reported delay; the Feishu startup readiness symptoms should stay separate unless they reproduce after the queue repair lands.
