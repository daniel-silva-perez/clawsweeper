## Summary

Keep open. Current main still carries reset-triggered state, but the active-run queue policy and both runner paths derive steer/followup behavior from the configured queue mode without a reset override, and the linked open PR #74144 is the implementation candidate that uses closing syntax for this issue.

## What This Changes

Reset-triggered chat commands such as `/new` and `/reset` should bypass steer/followup queueing and interrupt active work immediately.

## Best Possible Solution

Review and land a focused core reply-queue fix, likely #74144 if review validates it, that treats effective reset-triggered turns with interrupt semantics, suppresses steer/followup handling for that turn, clears queued lane work, aborts the active run, and adds regression coverage across steer, steer-backlog, followup, and collect. Keep any remaining startup model-warmup or Feishu identity-probe symptoms separate unless they remain after the queue repair.

## Reproduction Assessment

Yes. The high-confidence static reproduction path is an active run with `messages.queue.mode` set to `steer`, `followup`, `collect`, or `steer-backlog`, followed by an authorized `/new` or `/reset`; current main sets `resetTriggered` but still resolves active-run handling from the configured queue mode.

## Solution Assessment

Yes. Threading reset state through the existing queue-policy seam and prepared-run cleanup is the narrowest maintainable fix; the linked PR should be reviewed as the implementation candidate rather than creating a separate repair job.
