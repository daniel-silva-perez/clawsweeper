## Summary

Keep open. Current main still recognizes reset-triggered turns, but the active-run queue policy and runner paths do not use that reset state, so `/new` or `/reset` can still be handled as normal queued work under steer/followup modes. Open PR #74144 is the linked implementation candidate and should be reviewed rather than closing this issue early.

## What This Changes

Reset-triggered channel turns such as `/new` and `/reset` should bypass active-run steer/followup queueing and start a fresh session immediately.

## Best Possible Solution

Land a focused core reply-queue fix, likely #74144 if review validates it, that treats effective reset-triggered turns with interrupt semantics, suppresses steer/followup handling for that turn, clears queued lane work, aborts the active run, and adds regression tests for steer, steer-backlog, followup, and collect. Split the later startup/identity/prewarm symptoms only if they remain after that queue fix.

## Reproduction Assessment

Yes. The high-confidence static reproduction path is an active run with `messages.queue.mode` set to `steer` or a followup mode, then an authorized `/new` or `/reset`; current main sets `resetTriggered` but still derives the active-run action from the configured queue mode.

## Solution Assessment

Yes. Threading reset state through the existing queue-policy seam and prepared-run cleanup is the narrowest maintainable fix; broad Feishu startup changes should not be mixed into this queue-policy repair unless they are proven to share the same root cause.
