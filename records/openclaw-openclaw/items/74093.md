## Summary

Keep this issue open. Current main still computes reset-triggered state but does not feed it into the active-run queue policy, so reset-triggered `/new` or `/reset` turns can still take the configured `steer`/followup enqueue path. The issue is also paired with open PR #74144, which uses closing syntax for this issue, so the next step is maintainer review of that PR rather than early closure or a separate repair job.

## What This Changes

The issue requests that reset-triggered turns such as `/new` or `/reset` bypass steer/followup queue handling and interrupt active runs immediately.

## Best Possible Solution

Keep this issue open while #74144 is reviewed. The best fix is a focused core reply-queue change that makes effective reset-triggered model turns bypass steer/followup enqueueing, clear queued work for the session lane, abort the active run with interrupt semantics, preserve bare `/new` and `/reset` acknowledgements, and add regression tests for reset plus `steer`, `steer-backlog`, `followup`, and `collect`.
