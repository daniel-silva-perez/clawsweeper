## Summary

Keep open. Current main still lets OpenAI-compatible `image_url` inputs flow into `existingImages` and return before the image sanitizer when the prompt text has no image refs, and the streaming lifecycle-error path can still finalize with `[DONE]` before visible error content is written. The issue is also paired with open same-author PR #62070 using closing syntax, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

OpenAI-compatible `/v1/chat/completions` images should be sanitized/resized before provider handoff, and streaming provider errors should be surfaced instead of returning only `[DONE]`.

## Best Possible Solution

The desired end state is a server-side gateway/runner path that sanitizes every OpenAI-compatible image exactly once before provider handoff and emits one visible streaming error chunk before the terminal SSE marker, with focused regression tests covering both paths. Client-side image resizing can remain a separate iOS/mobile concern.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: send an OpenAI-compatible `image_url` part with ordinary text and no prompt image refs, or emit a lifecycle `error` before the async command path writes content; current main shows both affected paths still exist.

## Solution Assessment

Yes, the proposed fix direction is the narrow maintainable solution, but it is not implemented on current main. Review and land PR #62070 or an equivalent rebased fix that sanitizes `existingImages` on the no-ref path and handles lifecycle streaming errors inside the current finalizer state machine.
