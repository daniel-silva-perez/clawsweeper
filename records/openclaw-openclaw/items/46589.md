## Summary

Keep open. Current main still lacks the cron chaining fields and still has the hardcoded Control UI cron copy this PR tries to fix, so the branch is not obsolete. The proposed patch is not ready as-is because chaining is inconsistent across cron run entry points and the locale updates are incomplete for the current generated Control UI i18n setup.

## What This Changes

The PR adds cron Control UI i18n/test-registration fixes and success/failure cron job chaining across cron runtime, gateway protocol/generated clients, and the Control UI form.

## Best Possible Solution

Land or replace the low-risk Control UI i18n fixes with regenerated locale metadata, then handle chaining only through a shared post-run dispatch path that covers scheduled, queued manual, and direct service runs with protocol/docs/tests updated together; if durable multi-step workflow semantics are desired, route that product direction through Task Flow instead of cron-native fields.

## Reproduction Assessment

Yes. At PR head, configure onSuccessJobId or onFailureJobId and invoke the public CronService.run()/ops.run() path directly; inspection shows that path returns after finishPreparedManualRun(), while chain dispatch exists only in queued manual and timer completion paths.

## Solution Assessment

No. The i18n fixes are narrow, but the chaining implementation is not the best solution as-is because post-run behavior is split across entry points and misses the public direct service run path; a centralized post-run dispatcher or Task Flow routing is safer.
