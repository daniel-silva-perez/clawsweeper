## Summary

Keep open. Current main still lacks the PR's cron chaining fields and the narrow cron UI i18n fixes, so the branch is not obsolete; however, the proposed chaining implementation is not merge-ready because direct `CronService.run()` calls do not dispatch configured chain targets, and the i18n changes are stale against current main's generated-locale workflow.

## What This Changes

The PR fixes cron Control UI i18n/test-registration gaps and adds success/failure cron job chaining fields across cron runtime, gateway protocol/generated clients, and the Control UI.

## Best Possible Solution

Land or replace the low-risk Control UI i18n/test-registration fixes after rebasing onto the current generated-locale and Vitest layout, then handle cron chaining only through a shared post-run dispatch path covering scheduled, queued manual, and direct service runs with protocol/docs/tests updated together; alternatively keep durable multi-step orchestration in Task Flow instead of adding cron-native pipelines.

## Reproduction Assessment

Yes. At PR head, configure a job with `onSuccessJobId` or `onFailureJobId` and invoke `CronService.run()`/`ops.run()` directly; inspection shows that path returns after `finishPreparedManualRun()`, while chain dispatch exists only in queued manual and timer completion paths.

## Solution Assessment

No. The i18n fixes are narrow, but the chaining implementation is not the best solution as-is because post-run behavior is split across entry points and misses the public direct service run path; a centralized post-run chain dispatcher or Task Flow routing is safer.
