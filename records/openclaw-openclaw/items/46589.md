## Summary

Keep open. Current main has not implemented the PR's central changes, and the PR branch still has a blocking cron chaining consistency bug: queued/manual gateway runs dispatch chained jobs, but the direct CronService.run()/ops.run() path completes without dispatching the configured success or failure target.

## What This Changes

The PR fixes cron Control UI i18n and test-registration gaps, then adds success/failure cron job chaining fields across cron runtime, gateway protocol, generated Swift models, and the Control UI.

## Best Possible Solution

Land or replace the low-risk Control UI i18n and test-registration fixes independently if maintainers want them, then handle cron chaining only through a shared post-run dispatch path covering scheduled, queued manual, and direct service runs, with protocol/docs/tests updated together; alternatively keep durable multi-step orchestration in Task Flow instead of adding cron-native pipelines.

## Reproduction Assessment

Yes. At PR head, configure a job with onSuccessJobId or onFailureJobId and invoke CronService.run()/ops.run() directly; inspection shows that path returns after finishPreparedManualRun(), while chain dispatch exists only in enqueueRun() and timer-triggered paths.

## Solution Assessment

No. The i18n fixes are narrow, but the chaining implementation is not the best solution as-is because it splits post-run behavior across entry points and misses direct manual service runs; a centralized post-run chain dispatcher or Task Flow routing is safer.
