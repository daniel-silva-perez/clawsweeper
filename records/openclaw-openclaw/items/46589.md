## Summary

Keep open. Current main still does not implement the PR's cron chaining contract and still has the cron UI i18n/test-registration gaps, so the PR is not obsolete; however, the PR should not merge as-is because direct `CronService.run()` calls still bypass chain dispatch and the cron-native pipeline direction needs maintainer review against Task Flow.

## What This Changes

This PR adds cron success/failure job chaining across cron runtime, gateway protocol/schema, generated Swift models, and Control UI, while also fixing cron UI i18n strings and registering cron UI tests.

## Best Possible Solution

Split and land the low-risk cron UI i18n/test-registration fixes if desired, then either rebase and complete cron-native chaining consistently across scheduled, queued manual, direct service manual, and generated protocol/client surfaces, or route durable multi-step orchestration through Task Flow instead of adding another pipeline primitive to cron.

Yes. At PR head, configure a job with `onSuccessJobId` or `onFailureJobId` and call the public `CronService.run()`/`ops.run()` path directly; the code returns after `finishPreparedManualRun()` without dispatching the saved chain target.

## Solution Assessment

No, the PR is not the best solution as-is. The narrower UI fixes look useful, but cron chaining needs either a complete implementation across every run entry point or a maintainer decision to keep durable orchestration in Task Flow.
