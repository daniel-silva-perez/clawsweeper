## Summary

Keep open. Current main still lacks the PR's cron chaining contract and also still has the narrow cron UI i18n/test-registration gaps, so this is not obsolete. The branch contains useful source work, but it should not merge as-is because the direct manual `CronService.run()` path still completes without dispatching configured success/failure chain targets, and maintainers need to decide whether durable pipeline orchestration belongs in cron or Task Flow.

## What This Changes

This PR adds cron success/failure job chaining across cron job types, gateway protocol/schema, generated Swift models, runtime execution, and Control UI, while also fixing cron UI i18n strings and registering cron UI tests.

## Best Possible Solution

Split and land the low-risk cron UI i18n/test-registration fixes if desired, then either rebase and complete cron-native chaining consistently across scheduled, queued manual, direct service manual, and generated protocol/client surfaces, or route durable multi-step orchestration through Task Flow instead of adding another pipeline primitive to cron.
