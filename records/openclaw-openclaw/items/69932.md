## Summary

Keep open. Current main still has the BlueBubbles monitor typing start, restart, and stop paths calling `sendBlueBubblesTyping` without this PR's monitor-level Private API guard, and the monitor tests still lack the disabled-Private-API regression case. The related merged #65447 covered send/attachment Private API cache handling, while open #70799 overlaps the broader helper-level fallback area but has not superseded or implemented this monitor change.

## What This Changes

The PR wraps BlueBubbles monitor typing start, restart, and stop calls in the existing `privateApiEnabled` check and adds a monitor test for the disabled-Private-API case.

## Best Possible Solution

Rebase this PR, preserve current-main log sanitization while applying the monitor guards and disabled-path test, and make an explicit maintainer choice on `null` Private API cache semantics so this monitor-level fix stays consistent with any helper-level work from #70799.

## Reproduction Assessment

Yes. The source path is clear: on current main, a BlueBubbles account whose cached Private API status is false or unknown can still reach monitor `sendBlueBubblesTyping` calls; the PR's proposed unit path is to mock `isBlueBubblesPrivateApiEnabled` as false and assert the monitor does not call `sendBlueBubblesTyping`.

## Solution Assessment

Unclear. The PR is a narrow plausible fix for the noisy disabled-Private-API monitor callbacks, but the strict false-for-unknown behavior and overlap with #70799 need maintainer review before calling it the best final shape.
