## Summary

Keep open. Current main now covers explicit Slack Block Kit payloads, shared presentation blocks, and interactive controls, but the ordinary markdown-table/rich-output path discussed here is still incomplete: `markdown.tables: "block"` remains schema-valid while runtime resolution coerces it back to `code`, and plain Slack replies still go through mrkdwn text chunking unless blocks are explicitly supplied. The active follow-ups #57717 and #57708 are still the right maintainer path for the remaining umbrella scope.

## What This Changes

Issue requests Slack Block Kit support for richer agent messages, including structured sections, fields, dividers, buttons, select menus, context blocks, and better Slack-native rendering for structured replies.

## Best Possible Solution

Keep this as the Slack rich-output umbrella until maintainers land or explicitly replace the native markdown-table Block Kit send path, then move broader automatic markdown-to-rich_text conversion into #57708 with conservative text/code fallbacks and public Slack docs updates.

## Reproduction Assessment

Yes. Configure `channels.slack.markdown.tables: "block"` and send an ordinary Slack markdown table reply without explicit blocks; current source and tests show the resolver returns `code`, and the send path posts mrkdwn text chunks instead of native Block Kit table blocks.

## Solution Assessment

No, closing this as implemented is not the best current action. Raw Block Kit and shared presentation support are implemented, but the remaining native table/rich-output work should be resolved by reviewing #57717 and scoping #57708 rather than creating duplicate cleanup work.
