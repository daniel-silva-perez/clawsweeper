## Summary

Keep open. Current `main` has shipped several important Slack Block Kit slices, including authored raw blocks, message-tool block payloads, shared presentation rendering, and interactive controls, but the ordinary markdown reply path still does not render markdown tables as native Slack Block Kit table blocks. The discussion also points to #57717 and #57708 as active follow-ups, so closing this umbrella now would hide remaining user-visible rich-output work.

## What This Changes

Request richer Slack agent responses by allowing agents to send structured Slack Block Kit content such as sections, dividers, fields, buttons, selects, context metadata, and better table rendering instead of plain text-only replies.

## Best Possible Solution

Keep this as the Slack rich-output umbrella until maintainers land or explicitly replace the native markdown-table Block Kit send path, then move any broader Markdown-to-Slack rich_text conversion discussion to #57708. Any implementation should stay in the Slack plugin send/rendering layer, preserve raw-block and presentation behavior, retain safe text/code fallbacks, and update public Slack or presentation docs for user-visible behavior.

## Reproduction Assessment

Yes. For the remaining observable gap, current main resolves Slack `markdown.tables: "block"` back to `code` and the ordinary Slack reply path posts mrkdwn text chunks, so markdown tables still do not become native Slack table blocks without an explicit Block Kit payload.

## Solution Assessment

No, closing this as implemented is not the best current action. Raw Block Kit and interactive/presentation slices are implemented, but the table/rich-output umbrella remains active and is better resolved by reviewing #57717 and scoping the broader #57708 RFC rather than starting a duplicate automated repair.
