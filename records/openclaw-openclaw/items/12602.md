## Summary

Keep open: current main implements raw Slack Block Kit delivery, shared presentation rendering, message-tool blocks, and opt-in interactive controls, but the ordinary Slack markdown reply path still renders through text chunks and `markdown.tables: "block"` is still coerced to code rendering. The remaining native markdown-table/rich-output work is already associated with active follow-ups, so this should stay as an umbrella rather than be closed as implemented.

## What This Changes

Allow Slack agent messages to use richer Block Kit output, including structured blocks, buttons/selects, context/dividers, and native rendering for markdown tables or other rich reply content.

## Best Possible Solution

Keep this as the Slack rich-output umbrella until maintainers land native markdown-table Block Kit delivery or explicitly split that remaining work to #57717 and #57708; preserve the existing raw-block, presentation, and interactive-directive paths with tests and public docs for any send-path change.

## Reproduction Assessment

Yes for the remaining observable gap: current main resolves Slack `markdown.tables: "block"` to `code`, and ordinary Slack replies without explicit blocks flow through `markdownToSlackMrkdwnChunks`. That source path is enough to reproduce why markdown tables still do not become native Slack table blocks on the default reply path.

## Solution Assessment

No, closing this as implemented is not the best current action. The narrow raw-block and interactive slices are implemented, but the issue discussion’s table/rich-output use case remains open and is better resolved by reviewing the active follow-ups rather than starting a duplicate repair.
