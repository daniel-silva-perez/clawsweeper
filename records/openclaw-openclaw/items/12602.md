## Summary

Keep open. Current main already supports explicit Slack Block Kit payloads, shared presentation blocks, and interactive controls, but it still does not complete the ordinary markdown-table/rich-output path requested in the discussion: `markdown.tables: "block"` is accepted by schema but resolved back to `code`, and plain Slack replies still go through mrkdwn text chunking unless blocks are explicitly supplied. Active follow-ups #57717 and #57708 remain the right maintainer-owned path for the remaining umbrella work.

## What This Changes

The issue requests richer Slack agent messages through Slack Block Kit, including structured blocks, native table rendering, visual hierarchy, buttons, select menus, and context metadata.

## Best Possible Solution

Keep this as the Slack rich-output umbrella until maintainers land or explicitly replace the native markdown-table Block Kit send path, then move the broader markdown-to-rich_text conversion design into #57708. The implementation should stay in the Slack plugin send/rendering layer, preserve explicit raw blocks and shared presentation behavior, keep conservative text/code fallbacks, and update the public Slack docs for user-visible behavior.

## Reproduction Assessment

Yes. Configure `channels.slack.markdown.tables: "block"` and send an ordinary Slack markdown table reply without explicit blocks; current source and tests show the resolver returns `code`, and the send path posts mrkdwn text chunks instead of Block Kit table blocks.

## Solution Assessment

No, closing as implemented is not the best current action. Raw Block Kit and shared presentation support are implemented, but the remaining native table/rich-output work should be resolved by reviewing #57717 and scoping #57708 rather than opening duplicate automated work.
