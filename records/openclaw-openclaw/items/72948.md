## Summary

Keep open. Current main still has the unmanaged stop fallback, but the verifier rejects the reported single-token `openclaw-gateway` process shape before the opt-in gateway-binary check can match, and the Unix discovery path still returns no candidates when `lsof` is unavailable, matching the original sandbox log.

## What This Changes

The issue requests that `openclaw gateway stop` actually terminate verified foreground-launched gateway processes instead of only reporting the service-manager state.

## Best Possible Solution

Update unmanaged gateway-stop discovery so it can find foreground gateway listeners in container Linux, accept a standalone `openclaw-gateway` executable only under the existing `allowGatewayBinary` verification gate, reuse or adapt the existing `ss`-capable listener inspection when `lsof` is unavailable, and keep strict gateway-only process verification before sending SIGTERM.

## Reproduction Assessment

Yes. The reporter supplied a concrete foreground-gateway reproduction, and current source inspection shows both the `openclaw-gateway` argv shape and missing-`lsof` path can produce an empty verified PID list followed by the service-disabled message.

## Solution Assessment

Yes. A narrow verifier and listener-discovery fix is the maintainable path; a message-only or docs-only change would leave sandbox/container users manually killing verified foreground gateways.

## Repair Work Prompt

Fix the unmanaged foreground gateway stop bug tracked by this issue. When `openclaw gateway stop` has no loaded service manager, it must find and SIGTERM only verified OpenClaw gateway listener processes, including the foreground child-binary argv shape where `/proc/<pid>/cmdline` is effectively `openclaw-gateway\0`. Move or adapt the `allowGatewayBinary` check in `isGatewayArgv()` so standalone `openclaw-gateway` is accepted only when that option is true, while unknown executables remain rejected. Also cover the original no-`lsof` Linux/container path: reuse or adapt the existing `ss`-capable listener inspection rather than broadening unsafe PID matching. Add focused tests for single-token `openclaw-gateway`, unmanaged stop signaling through the verified listener path, and missing-`lsof` discovery if changed. Do not add pid-file semantics, do not kill arbitrary listeners on the port, and do not relax gateway-only verification. Add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @mayank6136`.
