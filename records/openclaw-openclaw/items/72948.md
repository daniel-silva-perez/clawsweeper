## Summary

Keep open. Current main still has the unmanaged-stop fallback, but the argv verifier still rejects the reported standalone `openclaw-gateway` child process before the opt-in gateway-binary check can match, so `gateway stop` can still fall through to the service-disabled path without signaling the foreground gateway.

## What This Changes

The issue requests that `openclaw gateway stop` stop a directly or foreground-launched gateway process when no service manager is loaded, instead of only reporting that the gateway service is disabled.

## Best Possible Solution

Update gateway process verification so an opted-in standalone `openclaw-gateway` executable is accepted before requiring a separate `gateway` token, keep strict process verification before SIGTERM, and add focused coverage for unmanaged stop with the foreground child-binary argv shape. If the no-`lsof` container path is in scope, reuse or adapt the existing `ss`-capable port inspection fallback rather than broadening unsafe PID matching.

## Reproduction Assessment

Yes. The reporter supplied a concrete foreground-gateway reproduction and current source inspection shows the verifier rejects the exact single-token `openclaw-gateway` argv shape needed for the unmanaged stop fallback.

## Solution Assessment

Yes. A narrow verifier/listener fallback fix is the maintainable path; a docs-only or message-only change would still leave container and sandbox users manually killing foreground gateways.

## Repair Work Prompt

Fix the foreground gateway stop bug tracked by https://github.com/openclaw/openclaw/issues/72948. Current `isGatewayArgv()` rejects a standalone `openclaw-gateway` argv token before the `allowGatewayBinary` executable check can pass, so `openclaw gateway stop` can fail to SIGTERM a directly launched gateway child and then print `Gateway service disabled.`. Keep the fix narrow: accept the opted-in standalone `openclaw-gateway` executable shape before requiring a separate `gateway` token, preserve strict OpenClaw-only verification before any SIGTERM/SIGUSR1, and do not add broad arbitrary-port-listener kills. Add focused tests for `isGatewayArgv(['openclaw-gateway'], { allowGatewayBinary: true })`, rejected behavior without the opt-in where appropriate, and unmanaged stop coverage through `src/cli/daemon-cli/lifecycle.test.ts` or `src/infra/gateway-processes.test.ts`. If you touch listener discovery for minimal containers without `lsof`, prefer reusing/adapting the existing `ss`-capable port inspection path instead of unsafe matching. Changelog: add a single-line user-facing fix entry with appropriate non-bot contributor credit. Validate with the targeted tests first, then run the changed gate in Testbox before handoff.
