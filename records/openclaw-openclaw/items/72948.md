## Summary

Keep open. Current main has an unmanaged stop fallback, but the verifier still rejects the reported standalone `openclaw-gateway` child-process argv shape before the opt-in gateway-binary check can pass, so `gateway stop` can still fall through to `Gateway service disabled.` without stopping the foreground process.

## What This Changes

Request that `openclaw gateway stop` stop a foreground-launched gateway in no-service-manager environments instead of reporting `Gateway service disabled` while leaving the process and port alive.

## Best Possible Solution

Update gateway process verification so an opted-in standalone `openclaw-gateway` executable is accepted before requiring a separate `gateway` token, keep strict OpenClaw-only signaling before sending SIGTERM, and add focused regression coverage for unmanaged stop in the reported foreground child-binary shape. If the target environment lacks `lsof`, reuse or adapt the existing `ss`-capable port inspection path so the fallback can still find verified listeners.

## Reproduction Assessment

Yes. The reporter provided a concrete foreground-gateway repro, and current source shows the verifier rejects the single-token `openclaw-gateway` argv shape before unmanaged stop can signal it.

## Solution Assessment

Yes. Keeping this open for a narrow verifier/listener fallback fix is the right path; a docs-only or message-only change would still leave container and sandbox users manually killing the foreground gateway.

## Repair Work Prompt

Fix the foreground gateway stop fallback for https://github.com/openclaw/openclaw/issues/72948. Current main rejects a listener whose `/proc/<pid>/cmdline` is a single `openclaw-gateway` token because `isGatewayArgv()` requires a separate `gateway` token before checking `allowGatewayBinary`; update the verifier so the opted-in standalone gateway binary is accepted without allowing arbitrary port listeners.
