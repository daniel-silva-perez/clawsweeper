## Summary

Keep open. Current main has an unmanaged gateway stop fallback, but the process verifier still rejects the reported standalone `openclaw-gateway` argv shape before it can accept the opted-in gateway binary path, so the foreground/container case remains a real narrow lifecycle bug.

## What This Changes

Request that `openclaw gateway stop` stop a foreground-launched unmanaged gateway process instead of only reporting that the managed service is disabled.

## Best Possible Solution

Update gateway process verification so an opted-in standalone `openclaw-gateway` executable is accepted before requiring a separate `gateway` token, preserve strict OpenClaw-only process signaling before SIGTERM, and add regression coverage for unmanaged stop with the reported child-binary argv shape. If listener discovery is touched, align the stop path with the existing `ss` fallback behavior for Linux containers where `lsof` is absent.

Yes. The reporter supplied concrete foreground-gateway steps and current source shows the verifier gap: a single-token `openclaw-gateway` argv fails before the gateway-binary allowance is evaluated.

## Solution Assessment

Current main has an unmanaged stop fallback, but `isGatewayArgv()` rejects a standalone `openclaw-gateway` argv token before the `allowGatewayBinary` check, so `openclaw gateway stop` can fall through to the service-disabled path instead of SIGTERMing the foreground child.
