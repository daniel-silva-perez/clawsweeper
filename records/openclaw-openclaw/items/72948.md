## Summary

Keep open. Current main has an unmanaged stop fallback, but the current process verifier still rejects the reported standalone `openclaw-gateway` argv shape before the `allowGatewayBinary` check can pass, so `gateway stop` can still fall through to the service-disabled path. The original no-service-manager/container case is valid and narrow enough for a focused fix PR.

## What This Changes

Fix `openclaw gateway stop` so it stops a directly launched foreground gateway process instead of only disabling a service-manager registration.

## Best Possible Solution

Update gateway process verification so an opted-in standalone `openclaw-gateway` executable is accepted before requiring a separate `gateway` token, keep the existing verified-process safety checks, and add regression coverage for unmanaged stop against the reported foreground child-binary shape. While there, verify whether the no-`lsof` Linux/container path should reuse existing `ss`/port-inspection fallback instead of silently returning no PIDs.

## Reproduction Assessment

Yes. The reporter supplied a concrete foreground-gateway repro and current source reproduces the verifier gap: a single-token `openclaw-gateway` argv fails the `normalized.includes("gateway")` guard before the gateway-binary allowance is evaluated.

## Solution Assessment

Yes, keeping this open and routing a narrow fix is the right direction. The suggested line-level fix is necessary, but the safest final solution should also preserve strict OpenClaw-process verification and account for minimal Linux containers where `lsof` is absent.

## Repair Work Prompt

Fix the foreground gateway stop bug tracked by https://github.com/openclaw/openclaw/issues/72948. Current main has `stopGatewayWithoutServiceManager`, but `isGatewayArgv()` rejects the reported `/proc/<pid>/cmdline` shape where argv is a single `openclaw-gateway` token because it requires `normalized.includes("gateway")` before checking `allowGatewayBinary`. Make the smallest safe change so opted-in gateway-binary verification accepts standalone `openclaw-gateway`/platform variants while still rejecting unrelated processes. Add regression tests for `isGatewayArgv(["/usr/local/bin/openclaw-gateway"], { allowGatewayBinary: true })` and for unmanaged stop discovering/signaling that shape. Also inspect the original `lsof`-missing Linux/container path: if the stop fallback still cannot discover a listener without `lsof`, reuse an existing safe listener-inspection fallback such as `ss`/`inspectPortUsage` rather than killing arbitrary port owners. Do not change managed launchd/systemd/schtasks semantics, do not signal unverified PIDs, and do not add broad startup migrations. Add a user-facing changelog fix entry crediting @mayank6136 unless maintainers decide the final diff is tests-only.
