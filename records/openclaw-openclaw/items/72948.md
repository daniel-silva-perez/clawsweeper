## Summary

Keep open. Current main still has the reported foreground gateway stop gap: unmanaged stop exists, but the verifier rejects a single-token `openclaw-gateway` listener before the opt-in binary check can match, and the Unix discovery path still returns no candidates when `lsof` is unavailable. Open PR #72224 appears to contain the right source-level fix, but it is not merged, so this issue should remain available as the stop-specific tracker.

## What This Changes

The issue asks for `openclaw gateway stop` to stop, or at least clearly report, directly launched foreground gateway processes when no service manager is loaded.

## Best Possible Solution

Land #72224 after rebase/review, or land a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing `allowGatewayBinary` gate and adds Linux listener discovery that works without `lsof`, with regression coverage for foreground stop.

## Reproduction Assessment

Yes. The reporter supplied a concrete foreground-gateway reproduction, and current source inspection shows the single-token `openclaw-gateway` listener and missing-`lsof` paths can both produce an empty verified PID list followed by the service-disabled fallback.

## Solution Assessment

Yes for the requested direction. A narrow verifier plus listener-discovery fix preserves gateway-only process verification while making `gateway stop` useful in containers; a message-only change would leave users manually killing a verified foreground gateway.
