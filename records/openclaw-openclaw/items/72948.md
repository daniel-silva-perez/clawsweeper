## Summary

Keep open. Current main still contains the reported foreground-stop failure path: unmanaged stop exists, but verified listener discovery can return no candidates when `lsof` is absent, and the argv verifier rejects a single-token `openclaw-gateway` process before the opt-in gateway-binary check can match. The linked #72224 PR appears to contain the relevant fix, but it is open and not merged.

## What This Changes

The issue asks for `openclaw gateway stop` to stop a verified foreground-launched gateway process instead of reporting a service-manager-only disabled state while the process keeps listening.

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing opt-in gate and adds Linux listener discovery that works without `lsof`, with regression coverage for foreground `gateway stop`.

## Reproduction Assessment

Yes. The reporter supplied concrete foreground-gateway steps, and current source inspection shows the same failure path: no verified listener PID leads to the service-disabled fallback instead of SIGTERM.

## Solution Assessment

Yes. A narrow verified-process fallback is the best fix because it preserves the gateway-only safety check before signaling; a message-only change would leave foreground gateway users manually killing a valid OpenClaw process.
