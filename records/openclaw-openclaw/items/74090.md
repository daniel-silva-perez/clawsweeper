## Summary

Keep this open. Current main has overflow diagnostics, bounded auto-compaction/retry, memory promotion preview/apply commands, rem-harness preview plumbing, and session archive/delete primitives, but it still does not wire repeated session overflow into a deduped promote-then-rotate lifecycle suggestion, operator notification, CLI prompt, or confirmation flow. The remaining work crosses memory writes and active session rotation, so it needs maintainer product review before an implementation PR.

## Best Possible Solution

Keep this issue open and ask maintainers to approve the first lifecycle design: track repeated overflow per session, emit a deduped structured suggestion with a candidate count from a public memory preview seam, surface it in `openclaw sessions` and `openclaw memory status`, and require explicit operator confirmation before promotion or session rotation.
