## Summary

Keep this open. Current main still has the overflow diagnostic, bounded auto-compaction path, memory promotion preview/apply primitives, rem-harness preview plumbing, and session archive/delete support, but it still does not connect repeated context-overflow pressure to a deduped promote-then-rotate lifecycle suggestion or an operator confirmation flow.

## What This Changes

The issue requests an operator-visible, deduped promote-then-rotate lifecycle suggestion after repeated session context overflows, with dry-run memory promotion candidates and explicit confirmation before promotion or rotation.

## Best Possible Solution

Approve a small lifecycle design first: persist per-session overflow pressure, emit a deduped structured suggestion with a stable id and dry-run candidate count, expose it through operator-visible CLI/UI surfaces, and require explicit confirmation before either promotion or session rotation.

## Reproduction Assessment

Yes. Existing overflow tests provide a high-confidence simulation path for repeated overflow, and current code inspection shows those attempts compact/retry or return `context_overflow` without any promote-then-rotate suggestion.

## Solution Assessment

Unclear pending maintainer design approval. The issue's opt-in suggestion and confirmation model is safer than automatic promotion/rotation, but the exact state storage, API boundaries, main-session rotation behavior, and UI surfaces should be settled first.
