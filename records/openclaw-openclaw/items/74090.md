## Summary

Keep this open. Current main has the underlying overflow diagnostics, bounded auto-compaction/retry, memory promotion preview/apply, REM preview, and session archive/delete pieces, but it still does not wire repeated overflow into a deduped promote-then-rotate lifecycle suggestion or operator confirmation flow.

## What This Changes

The issue requests an opt-in lifecycle suggestion that detects repeated session context overflow, previews memory promotion candidates, and lets an operator promote and rotate the session.

## Best Possible Solution

Add a maintainer-approved lifecycle path keyed by session that dedupes repeated overflow signals, obtains a bounded memory-promotion preview through a public seam, surfaces the candidate count in logs/CLI/operator UI, and requires explicit confirmation before applying promotions or archiving/rotating a session.

## Reproduction Assessment

Yes. The feature gap is reproducible by following the current overflow branch in `run.ts`: repeated context overflow produces diagnostics and compaction retries, then reset/new guidance, with no lifecycle suggestion or promotion preview hook.

## Solution Assessment

Unclear until maintainers choose the product contract. The requested direction is sensible, but the best implementation should build on existing compaction memory-flush, transcript-rotation, REM preview, and session mutation primitives without bypassing plugin/core boundaries.
