## Summary

Keep open. The PR is directionally useful because current `main` has SSE streaming endpoints without downstream keepalive bytes, but the proposed helper still has a concrete resource-exhaustion bug for non-finite interval overrides and the added fake-timer tests need cleanup before merge.

## What This Changes

The PR adds a reusable SSE heartbeat helper, tests it, and wires it into streaming `/v1/chat/completions` and `/v1/responses` responses.

## Best Possible Solution

Keep the SSE comment heartbeat direction, but require finite interval normalization, timer cleanup in the new tests, and targeted gateway validation before merge. Avoid adding a config surface unless maintainers explicitly want operator-tunable cadence.

## Reproduction Assessment

Yes. The original proxy-idle issue is reproducible from current-main behavior because the streaming endpoints can go quiet after initial SSE events, and the PR-introduced interval bug is directly reproducible in Node 22 with `NaN` or `Infinity` becoming a 1 ms timer delay.

## Solution Assessment

No, not yet. SSE comments are the right narrow transport fix and match existing session-history precedent, but the helper must defensively normalize `intervalMs` and the tests must restore real timers before this is the best mergeable version.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize heartbeat intervals before clamping:** `src/gateway/http-common.ts:147`
  - body: If a caller passes `NaN` or `Infinity`, `Math.max` preserves that non-finite value and Node schedules `setInterval` with a 1 ms delay. Since this helper exposes `intervalMs` as an override and is explicitly trying to avoid heartbeat-driven resource pressure, validate `Number.isFinite` and fall back to the default before applying the 1000 ms floor.
  - confidence: 0.93
- **[P3] Restore real timers after heartbeat tests:** `src/gateway/http-common.test.ts:303`
  - body: This new suite switches Vitest into fake-timer mode in `beforeEach` and never switches back, so later tests in the same file inherit fake timers. Add cleanup around the suite, for example clearing pending timers and calling `vi.useRealTimers()` in `afterEach`.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch has one concrete resource-exhaustion concern in the heartbeat interval override validation.

Concerns:

- **[medium] Non-finite interval can create 1 ms heartbeat loops:** `src/gateway/http-common.ts:147`
  - body: `Math.max(1000, options?.intervalMs ?? 30_000)` does not reject `NaN` or `Infinity`; Node coerces those timer delays to 1 ms. The current production call sites use the default, but the exported helper's override should be finite-checked before merge because this code runs per open stream.
  - confidence: 0.86
