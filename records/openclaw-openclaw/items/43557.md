## Summary

Keep open. Current main still has the same OpenClaw-owned gap: Gmail/gog setup and watcher support exist, but the bundled skill and public Gmail scheduling docs do not explain durable non-interactive auth resolution for the reported macOS cron failure, and the watcher still only classifies address-in-use errors. The broader Google/Gmail operability issue remains related context, not a superseding fix.

## What This Changes

Investigate intermittent `gog` Gmail OAuth resolution failures from macOS scheduled or cron runs and add guidance or diagnostics for durable non-interactive use.

## Best Possible Solution

Add focused non-interactive Gmail auth troubleshooting to the bundled `gog` skill and Gmail scheduling/webhook docs, and optionally add a small auth-specific diagnostic hint for `No auth for gmail` in the Gmail watcher path. Coordinate wording with the broader Google/Gmail operability issue #61649 without changing credential storage defaults or global cron environment behavior.

## Reproduction Assessment

Unclear. The live intermittent auth failure requires macOS cron, configured `gog` OAuth state, and keychain/session conditions, but the current documentation and diagnostic gap is reproducible by static inspection of the bundled skill, Gmail docs, and watcher error handling.

## Solution Assessment

Yes. The narrowest maintainable first step is docs plus an optional auth-error classifier/hint; changing token storage, migrating credentials, or globally altering cron environment propagation would overreach without a confirmed OpenClaw runtime defect.
