## Summary

Keep open. Current main still has the same OpenClaw-owned gap: the bundled `gog` skill and Gmail scheduling/webhook docs do not cover durable non-interactive Gmail auth for macOS cron, and the watcher diagnostics still do not classify the reported `No auth for gmail` failure.

## What This Changes

The issue asks for diagnosis, documentation, and setup guidance for intermittent `gog` Gmail auth failures during macOS cron or scheduled runs.

## Best Possible Solution

Add focused non-interactive Gmail/gog auth troubleshooting to the bundled `gog` skill and Gmail scheduling/webhook docs, with an optional small watcher diagnostic for `No auth for gmail`; keep credential storage defaults and global cron environment behavior unchanged unless upstream evidence proves a runtime defect.

## Reproduction Assessment

Unclear. The live intermittent auth failure requires macOS cron, configured `gog` OAuth state, and keychain/session conditions, but the missing OpenClaw documentation and diagnostic coverage is reproducible by static inspection of the current skill, docs, watcher, and tests.

## Solution Assessment

Yes. The narrowest maintainable path is docs plus an optional auth-error classifier/hint; changing token storage or broad cron environment handling would overreach without a confirmed OpenClaw runtime bug.
