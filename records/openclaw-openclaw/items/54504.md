## Summary

Keep this issue open. Current main still lacks a lightweight workspace/user `subagents/<label>.md` role-file convention for native `sessions_spawn`; the implementation only supports explicit spawn params, full agent config/defaults, inline attachments, and broad subagent tool policy.

## Best Possible Solution

Keep open for maintainer product/API design. A narrow implementation should add a native resolver in the `sessions_spawn`/`spawnSubagentDirect` path for workspace-then-user `subagents/<label>.md` files, with explicit precedence over user-scoped files and explicit spawn params over frontmatter. It should include docs and focused tests for lookup precedence, invalid or unsafe frontmatter, allowed model overrides, tool-policy narrowing vs broadening, run-mode and thread-mode spawns, and unchanged behavior when no role file exists.

## Risks / Open Questions

- A plugin could partially approximate this by rewriting `sessions_spawn` tool params, but that does not provide the requested documented lookup precedence or an atomic pre-launch per-role system context/model/tool-policy contract.
- Existing alternatives such as full `agents.list` entries, `agents.defaults.systemPromptOverride`, explicit `model`/`thinking`, inline attachments, and `tools.subagents.tools` do not solve the central per-label persistent role-file workflow.
