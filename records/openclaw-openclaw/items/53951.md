## Summary

Keep open for maintainer review. The Defuddle idea is useful, but current main now routes readable HTML extraction through manifest-declared `webContentExtractors`, while this PR adds Defuddle to the older `src/agents/tools/web-fetch-utils.ts` path and adds a root parser dependency instead of a plugin-owned dependency.

## What This Changes

This PR adds Defuddle as the primary `web_fetch` HTML extractor with Readability fallback, updates extractor tagging/config/docs/tests/scripts, and adds the `defuddle` dependency.

## Best Possible Solution

Implement Defuddle through the existing `webContentExtractors` contract as a dedicated extractor plugin, with the `defuddle` dependency scoped to that plugin, `autoDetectOrder` ahead of Readability only if maintainers approve the default behavior change, `useAsync: false`, aligned docs/config copy, and focused tests for Defuddle success plus Readability fallback.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The correctness issue is source-verifiable: current main dispatches through the plugin extractor runtime, while the PR modifies the older utility extraction path.

## Solution Assessment

No. The requested capability should use the current plugin extractor contract and plugin-owned dependency boundary instead of adding Defuddle directly to core utility code and root runtime dependencies.
