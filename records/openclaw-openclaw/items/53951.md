## Summary

Keep this PR open. Current main does not implement Defuddle-first web_fetch extraction: the shipped path still resolves bundled web content extractor plugins and the only default extractor is the bundled Readability plugin. The PR is useful but needs rework against the current extractor-plugin architecture and a careful dependency/security review before it can land.

## Best Possible Solution

Rework this PR against current main as either a bundled web content extractor with a lower autoDetectOrder than Readability, or first add a documented external web-content-extractor plugin API if maintainers want Defuddle to live outside core. Preserve the existing guarded web_fetch network model by disabling or routing any Defuddle async fetch behavior, update docs/config wording only after the runtime path is real, and keep Readability as fallback with targeted tests asserting the selected extractor.

## Risks / Open Questions

- The PR adds the new defuddle dependency and transitive parser/runtime packages on an untrusted HTML processing path; dependency behavior, lockfile provenance, and async network behavior need review before merge.
- The PR appears written against an older direct web-fetch-utils extraction layout, while current main has moved local extraction into bundled web-content-extractor plugins. Landing it as-is would likely fight the current architecture.
- Current web-content-extractor resolution is bundled-only, so a pure external ClawHub plugin is not yet enough unless OpenClaw first expands that extractor API to external plugins.
