## Summary

Keep open. The PR is useful and current main still does not implement this memoization, but the current patch leaves the active runtime web-provider path effectively unmemoized because it keys the web tool cache by an object returned from a getter that clones on every call.

## What This Changes

The PR adds a WeakMap-chained memoizeToolFactory helper with unit tests and wraps image, media generation, PDF, web search, and web fetch tool factory calls in createOpenClawTools.

## Best Possible Solution

Revise the web-tool wrappers to key on stable scalar runtime-provider fingerprints, or expose a stable readonly runtime metadata snapshot, then add focused createOpenClawTools coverage proving both provider-change invalidation and warm-turn cache hits when runtime web metadata is active.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a stable runtime web-tools fingerprint:** `src/agents/openclaw-tools.ts:257-268`
  - body: getActiveRuntimeWebToolsMetadata() returns a structured clone on every call, so using runtimeWebTools as a WeakMap ref gives the web wrappers a fresh cache path each active-runtime turn. That leaves createWebSearchTool and createWebFetchTool effectively unmemoized for the runtime-provider path this PR targets; key these wrappers on stable scalar provider metadata or a stable snapshot instead.
  - confidence: 0.91
