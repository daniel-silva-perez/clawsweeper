## Summary

Keep open: current main still builds these tool factories directly, and the live PR head still has a performance-correctness gap in the runtime web-tools path because cloned runtime metadata makes the web factory cache miss on every active-runtime turn.

## What This Changes

The PR adds a WeakMap-chain memoizeToolFactory helper with unit tests and wraps the image, media generation, PDF, and web tool factory calls inside createOpenClawTools.

## Best Possible Solution

Revise the web-tool cache key to use stable scalar runtime-provider fingerprints, or expose a stable readonly runtime metadata snapshot, then add focused createOpenClawTools coverage through the real runtime metadata getter so provider-change invalidation and warm-turn cache hits are both proven.

## Reproduction Assessment

Yes. With active runtime web metadata, getActiveRuntimeWebToolsMetadata returns a structuredClone on each call, so repeated createOpenClawTools calls pass distinct runtimeWebTools refs into the WeakMap chain and miss the web factory cache.

## Solution Assessment

No. The overall memoization direction is narrow and maintainable, but keying web tools on the cloned runtime metadata object is not the best fix because it preserves invalidation by sacrificing the warm-cache path for active runtime providers.
