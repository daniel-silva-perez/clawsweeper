## Summary

Keep this PR open. Current main still constructs the targeted media, PDF, and web tools directly, so the proposed performance work is not implemented or superseded; however, the PR patch still needs revision because the web-tool cache key uses a runtime metadata object that current main deliberately returns as a fresh defensive clone on every read, defeating cross-turn memoization for the web factories.

## What This Changes

The PR adds a WeakMap-backed `memoizeToolFactory` helper, unit tests for cache invalidation, and wraps the media, PDF, and web tool factory calls inside `createOpenClawTools`.

## Best Possible Solution

Revise the web-tool cache key to use a stable fingerprint of the active runtime web provider metadata, keep the bounded WeakMap cache shape, and add a focused test that exercises `createOpenClawTools` with the real runtime metadata getter so cross-turn web cache hits and provider-change invalidation are both covered before benchmarking the per-turn path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use a stable runtime web-tools fingerprint:** `src/agents/openclaw-tools.ts:257-260`
  - body: `getActiveRuntimeWebToolsMetadata()` returns a structured clone on every call, so using `runtimeWebTools` as a WeakMap ref makes both web factory wrappers miss on every turn whenever runtime metadata is active. That leaves `createWebSearchTool` and `createWebFetchTool` effectively unmemoized for the runtime-provider path this PR is trying to speed up; key on stable scalar provider metadata or expose a stable readonly snapshot instead.
