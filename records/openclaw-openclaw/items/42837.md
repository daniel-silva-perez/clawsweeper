## Summary

Keep this open. Current main still creates Feishu dynamic-agent direct-peer bindings without `match.accountId`, while Feishu DM and comment ingress resolve and re-resolve routes with the runtime account id; for named accounts, that can leave the generated binding invisible to routing and fall back to the default agent. The directly related fix PR #42886 was closed unmerged, so there is no implemented or shipped fix to close against.

## What This Changes

Fix Feishu `dynamicAgentCreation` so named-account DM users get account-scoped dynamic agent bindings and route to independent per-user sessions instead of `agent:main`.

## Best Possible Solution

Make Feishu dynamic-agent creation account-aware in the Feishu plugin: pass the normalized runtime `accountId` into the helper, include `match.accountId` when checking and writing dynamic bindings for DM and comment flows, keep account-less legacy bindings matching only default-account traffic, and cover named-account creation, missing-binding repair, default-account compatibility, and duplicate-binding avoidance with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: enable Feishu `dynamicAgentCreation` on a named account, send a new DM, observe that the helper writes an account-less binding, and then route resolution with that named `accountId` falls back because existing routing tests show account-less bindings match only the default account.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin’s dynamic-agent helper and its DM/comment call sites, not by weakening core routing semantics; current routing behavior is explicit and covered by tests.
