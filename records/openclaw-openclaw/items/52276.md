## Summary

Keep open. Current main still has global text-only embedded block-reply dedupe, while this PR’s route-aware fix remains useful but is not merge-ready because the head patch still has a compile-breaking resolver reference and incomplete Telegram thread matching.

## What This Changes

The PR makes embedded messaging-tool duplicate suppression route-aware by threading route context through runner/subscriber paths, widening send-target extraction, and adding focused dedupe tests.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that wires the existing suppression helper into streamed and message_end block-reply paths, defines or imports the routed-provider resolver, carries origin provider/target/account/thread context through the embedded subscription boundary, and aligns target/thread extraction with outbound delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P0] Define the route provider resolver before using it:** `src/agents/pi-embedded-runner/run/attempt.ts:383-386`
  - body: `runEmbeddedAttempt` now calls `resolveRouteMessageProvider(...)`, but the PR head has no production declaration or import for that symbol. This blocks compilation or throws before embedded tools and reply handling can be constructed.
  - confidence: 0.99
- **[P1] Carry the origin thread into suppression matching:** `src/agents/pi-embedded-runner/run/attempt.ts:1205-1207`
  - body: The subscription receives provider, origin target, and account, but not the origin `messageThreadId`/current thread. Telegram topic context is stored separately, so same-topic sends can still fail route matching and emit a duplicate final block reply.
  - confidence: 0.91
- **[P2] Prefer messageThreadId when deriving the send thread:** `src/agents/pi-embedded-subscribe.tools.ts:384-389`
  - body: The extractor records `threadId` before `messageThreadId`, while Telegram execution resolves `messageThreadId` first. A call containing both aliases can be delivered to one topic and tracked against another, causing incorrect suppression decisions.
  - confidence: 0.86
