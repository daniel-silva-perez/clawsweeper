## Summary

Keep open. Current main still uses text-only embedded block-reply suppression, while this PR’s route-scoped direction is useful but the branch is not merge-ready because it references an undefined resolver and does not carry enough thread context for Telegram topic matching.

## What This Changes

The PR makes embedded messaging-tool duplicate suppression route-aware by threading provider, target, account, and channel context through runner/subscriber paths and widening message-send target extraction.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that wires the existing suppression helper into streamed and message_end block-reply paths, defines the routed-provider resolver, carries origin provider/target/account/thread context through the embedded subscription boundary, and keeps send-target tracking aligned with outbound delivery semantics.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: the PR head calls an undefined resolver, and current main can be inspected to show both embedded block-reply paths still suppress by text only. The Telegram topic failure is also traceable from current origin/thread handling and the suppression helper’s thread parity requirement.

## Solution Assessment

No. The PR’s direction is the right narrow product fix, but this branch is not the best mergeable solution until the resolver, origin-thread propagation, and message-send alias precedence are corrected.
