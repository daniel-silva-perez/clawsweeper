## Summary

Keep this PR open. Current main still has text-only embedded block-reply duplicate suppression, so the intended route-scoped behavior is not implemented, but the PR head is not mergeable because it calls an undefined routed-provider resolver and still loses Telegram origin-thread context for suppression matching.

## What This Changes

The PR attempts to make embedded messaging-tool duplicate suppression route-aware by threading provider, target, account, and channel context through runner/subscriber paths, widening message-send target extraction, and adding related tests.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that wires the existing suppression helper into streamed and message_end block-reply paths, derives the real route provider for synthetic automation providers, preserves providerless same-route sends, and carries origin target/account/thread metadata consistently through the embedded subscription boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P0] Define the routed provider resolver before using it:** `src/agents/pi-embedded-runner/run/attempt.ts:383-386`
  - body: runEmbeddedAttempt now calls resolveRouteMessageProvider(...), but PR head has no production declaration or import for that symbol. This blocks compilation and would fail embedded runs before tools or reply handling are constructed.
  - confidence: 0.99
- **[P1] Carry the origin thread into suppression matching:** `src/agents/pi-embedded-runner/run/attempt.ts:1205-1207`
  - body: The subscription receives originatingTo: params.messageTo, but Telegram topic context is carried separately as messageThreadId/currentThreadTs. Because suppression requires thread parity when a sent target has a topic id, same-topic sends can still emit the duplicate final block reply.
  - confidence: 0.91
- **[P2] Prefer messageThreadId when tracking Telegram sends:** `src/agents/pi-embedded-subscribe.tools.ts:384-388`
  - body: extractMessagingToolSends records threadId before messageThreadId, while Telegram action execution resolves messageThreadId first. A call with both aliases can be delivered to one topic and tracked against another, causing wrong duplicate-suppression decisions.
  - confidence: 0.86
