## Summary

Keep open. Current main still uses global text-only embedded block-reply suppression, so the route-scoped behavior remains relevant, but this PR head is not mergeable because it references an undefined resolver and still loses Telegram topic context during suppression matching.

## What This Changes

The PR makes embedded messaging-tool duplicate suppression route-aware by threading provider, target, account, and current-route context through runner/subscriber paths and expanding message-send target extraction tests.

## Best Possible Solution

Land a corrected version that keeps route-scoped duplicate suppression, defines the routed-provider resolver, carries origin provider/target/account/thread context through the embedded subscription boundary, and aligns dedupe target extraction with outbound delivery semantics for normal, providerless, synthetic-provider, and Telegram topic sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P0] Define the routed provider resolver before using it:** `src/agents/pi-embedded-runner/run/attempt.ts:383-386`
  - body: `runEmbeddedAttempt` now calls `resolveRouteMessageProvider(...)`, but PR head has no production declaration or import for that symbol. This blocks compilation and would fail embedded runs before tools or reply handling are constructed.
  - confidence: 0.99
- **[P1] Carry the origin thread into suppression matching:** `src/agents/pi-embedded-runner/run/attempt.ts:1205-1207`
  - body: The subscription forwards only `originatingTo: params.messageTo`, while Telegram topic context is stored separately. Because Telegram suppression requires thread parity, same-topic message sends can still emit the duplicate final block reply.
  - confidence: 0.9
- **[P2] Match Telegram thread alias precedence:** `src/agents/pi-embedded-subscribe.tools.ts:385-389`
  - body: `extractMessagingToolSends` derives the dedupe thread from `threadId` before `messageThreadId`, but Telegram execution reads `messageThreadId` first. When both aliases are present, suppression can track the wrong topic.
  - confidence: 0.86
