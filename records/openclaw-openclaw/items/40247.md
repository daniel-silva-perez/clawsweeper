## Summary

Keep this PR open. Current main still does not implement the PR's server-side dual-bind behavior: a tailnet/custom resolved bind host is still listened on by itself, while local gateway callers still default to 127.0.0.1. The PR remains a focused implementation candidate, with overlap but not clear supersession by #50751.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either rebase and land the server-side dual-bind fix with specific-IP regression tests plus docs/security expectation updates, or explicitly choose the client-side strategy in #50751 and close this PR only after a merged implementation demonstrably covers internal calls, CLI probes, and sessions_spawn for gateway.bind=tailnet and gateway.bind=custom.

## Risks / Open Questions

- Dual-binding tailnet/custom hosts would intentionally change documented behavior: same-host loopback access would begin working where docs currently say it will not.
- The listener change affects gateway locality and pairing semantics because loopback/direct-local requests are treated differently from remote tailnet/LAN requests.
- #50751 overlaps with a client-side host-resolution strategy; maintainers should choose a coherent listener/target model before closing either implementation candidate.
- No tests were run because this review was explicitly read-only; evidence is from current source, tests, docs, history probes, and the supplied PR context.
