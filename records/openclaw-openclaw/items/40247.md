## Summary

Keep this PR open. Current main still does not implement the PR's server-side dual-bind behavior for tailnet/custom bind hosts, and local gateway callers still target loopback. The PR is a plausible focused fix candidate, but it overlaps with the open client-side host-resolution approach in #50751 and changes documented tailnet/locality expectations, so it needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to choose either this server-side dual-bind model or the client-side host-resolution model in #50751, then land one coherent fix with regression coverage for tailnet/custom bind internal calls, aligned docs/security expectations, and targeted live proof for gateway management and `sessions_spawn` paths.

## Risks / Open Questions

- Closing now would discard a focused implementation candidate while current main still lacks this PR's server-side dual-bind behavior.
- The PR changes tailnet/custom locality semantics: same-host calls through `127.0.0.1` can become direct local or shared-secret loopback local, while current docs say same-host tailnet/LAN connects are treated as remote.
- The PR overlaps with #50751's client-side host-resolution model; maintainers should choose one coherent approach before landing either path.
- If the server-side dual-bind model is accepted, docs and security expectation tests should be updated alongside the code.
