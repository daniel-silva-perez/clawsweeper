## Summary

Keep open. Current main still routes managed deep-dreaming promotions directly through the `MEMORY.md` apply path, and the existing storage setting only governs phase/report artifacts, not the final promotion destination. The request remains a valid memory-core product/design gap rather than an implemented feature, duplicate, or ClawHub-only item.

## What This Changes

The issue asks for a configurable deep-dreaming promotion destination so scored candidates can be staged in a review queue before being written to `MEMORY.md`.

## Best Possible Solution

Keep this open until maintainers choose an additive final-promotion destination contract for memory-core. The end state should preserve the current direct `MEMORY.md` behavior as an explicit compatible mode while allowing staged review artifacts, and then update the manifest schema, resolver types, apply path, docs, and targeted tests together.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior is source-backed: a managed deep sweep can be traced from ranked candidates to `applyShortTermPromotions`, which writes the promoted section to `MEMORY.md`.

## Solution Assessment

Unclear until maintainers decide the workflow. The narrow maintainable direction appears to be a separate final-promotion write-mode or approval contract, rather than silently broadening `dreaming.storage.mode` beyond phase/report storage.
