## Summary

Keep open. Current main still applies managed deep-dreaming promotions directly through the `MEMORY.md` write path, while `dreaming.storage.mode` only controls phase/report artifacts. The linked storage-mode, raw-snippet, metadata-contamination, and promote-then-rotate items are adjacent but do not provide a final-promotion review queue, and current GitHub search found no open PR tied to this issue or a `promotionWriteMode` fix.

## What This Changes

The issue requests an additive memory-core configuration or workflow that stages deep dreaming promotion output in a review queue before, instead of or in addition to, appending to `MEMORY.md`.

## Best Possible Solution

Keep this issue open until maintainers choose an additive final-promotion destination or approval contract, then implement it in memory-core with the existing direct `MEMORY.md` behavior preserved as an explicit mode and staged review artifacts available for approval or distillation before canonical long-term memory writes.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior is source-backed: managed deep dreaming calls `applyShortTermPromotions`, and that helper appends selected candidates to `MEMORY.md` without a review destination.

## Solution Assessment

Unclear until maintainers choose the workflow. The narrow maintainable direction is an explicit final-promotion destination or approval contract rather than expanding `dreaming.storage.mode` implicitly from phase/report storage to canonical memory writes.
