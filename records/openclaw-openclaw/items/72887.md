## Summary

Keep open. Current main still treats deep dreaming promotion as an automatic final write to `MEMORY.md`; `dreaming.storage.mode` only controls phase/report artifacts, and there is no configured review/off/both promotion-write mode. The related storage-mode fix and raw-snippet reports are adjacent but do not supersede this review-queue/product-design request.

## What This Changes

Request to add a configurable deep-dreaming promotion output mode so ranked candidates can be staged in a review queue instead of automatically appending to `MEMORY.md`.

## Best Possible Solution

Keep the issue open for a maintainer design decision, then implement an additive memory-core promotion destination contract that preserves the current direct-write default, can stage structured review artifacts before canonical memory writes, and updates the manifest schema, resolver, CLI/docs/tests, and any Dreams UI surface consistently.

## Reproduction Assessment

Not applicable as a feature/design request, but the current behavior is high-confidence from source and existing tests: ranked deep candidates flow directly into `applyShortTermPromotions`, which writes `MEMORY.md`. I did not run write-path tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear until maintainers choose the desired workflow. A narrow implementation should avoid treating `dreaming.storage.mode` as an overloaded final-promotion switch unless that is explicitly selected, because phase report storage and canonical-memory approval are separate product decisions.
