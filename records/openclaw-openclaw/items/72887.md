## Summary

Keep open. Current main still sends deep dreaming promotions straight into `MEMORY.md`; the existing separate storage mode only controls phase/report artifacts, and neither the schema nor resolver exposes a review/off/both promotion destination. The related storage-mode and raw-snippet items are adjacent, not superseding this product/API request.

## What This Changes

Request an additive memory-core workflow/config option so deep dreaming can stage promoted candidates in a review queue before, instead of or in addition to, appending them to `MEMORY.md`.

## Best Possible Solution

Keep the issue open for a maintainer design decision, then implement an additive memory-core promotion destination contract that preserves the current direct-write default while allowing structured staged review artifacts before canonical `MEMORY.md` writes.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior is high-confidence from source and tests: ranked deep candidates flow into `applyShortTermPromotions`, which writes `MEMORY.md`; no write-path tests were run because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear until maintainers choose the workflow. A maintainable fix should add an explicit final-promotion destination/approval contract rather than implicitly changing phase report storage semantics.
