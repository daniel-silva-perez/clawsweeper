## Summary

Keep open. Current main still writes deep dreaming promotions directly to `MEMORY.md`, while `dreaming.storage.mode` only controls phase/report artifacts and the schema/resolver do not expose a review/off/both promotion-write destination. The adjacent storage-mode and raw-snippet reports do not supersede this product/API question.

## What This Changes

Request an additive memory-core workflow/config option so deep dreaming can stage promoted candidates in a review queue before, instead of or in addition to, appending them to `MEMORY.md`.

## Best Possible Solution

Keep the issue open for a maintainer design decision, then implement an additive memory-core promotion destination contract that preserves the current direct-write default while allowing structured staged review artifacts before canonical `MEMORY.md` writes.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior is high-confidence from source and tests: ranked deep candidates flow into `applyShortTermPromotions`, which writes `MEMORY.md`; no write-path tests were run because this review was read-only.

## Solution Assessment

Unclear until maintainers choose the workflow. A narrow solution should avoid overloading `dreaming.storage.mode` unless that is explicitly selected, because phase report placement and final canonical-memory approval are separate product decisions.
