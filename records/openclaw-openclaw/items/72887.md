## Summary

Keep open. Current main still applies deep dreaming promotions directly to `MEMORY.md`, while separate storage only controls phase/report artifacts; the requested final-promotion review queue remains a real product/API gap distinct from the adjacent raw-snippet and storage-mode reports.

## What This Changes

Request an additive memory-core configuration or workflow that stages deep dreaming promotion output in a review queue before, instead of or in addition to, appending to `MEMORY.md`.

## Best Possible Solution

Keep this open until maintainers choose an additive final-promotion destination contract, then implement it in memory-core so the existing direct `MEMORY.md` behavior remains available while staged review artifacts can be approved or distilled before becoming canonical long-term memory.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior is source-backed: managed deep dreaming calls `applyShortTermPromotions`, and that helper appends selected candidates to `MEMORY.md` without a review destination.

## Solution Assessment

Unclear until maintainers choose the workflow. The narrow maintainable direction is an explicit final-promotion destination or approval contract rather than expanding `dreaming.storage.mode` implicitly from phase/report storage to canonical memory writes.
