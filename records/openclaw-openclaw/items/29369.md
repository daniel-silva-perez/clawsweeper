## Summary

Keep this issue open. Current main still does not expose or honor a spawn-time `tools` allowlist on `sessions_spawn`; the schema, native spawn params, ACP spawn params, docs, and tests still route tool restrictions through config/effective-policy layers instead of a per-call `tools` parameter.

## Best Possible Solution

Keep this open for maintainer design. If approved, implement a narrow `sessions_spawn.tools` policy object that is schema-validated, propagated through both native and ACP spawn paths, stored or passed as trusted runtime metadata, and applied as a final narrowing layer in the effective tool-policy pipeline with deny-wins behavior. Add focused tests and docs for precedence and failure behavior.

## Risks / Open Questions

- Spawn-time tool policy is a security boundary; allowing the tool call to broaden capabilities would bypass configured policy unless the design is explicitly narrowing-only with deny-wins semantics.
- Native subagent and ACP runtime paths have separate schemas and spawn params; implementing only one would create inconsistent `sessions_spawn` behavior.
- Docs and tests need to pin precedence against `tools.allow`, `agents.list[].tools`, provider/group policy, sandbox policy, and `tools.subagents.tools`.
