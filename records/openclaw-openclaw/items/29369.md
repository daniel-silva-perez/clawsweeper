## Summary

Keep this issue open. Current main still does not expose or honor a spawn-time `tools` allowlist on `sessions_spawn`; the schema, native spawn params, ACP spawn params, docs, and effective tool-policy pipeline still route tool restrictions through config/profile/sandbox/subagent policy layers rather than a per-call `tools` parameter.

## Best Possible Solution

Keep this open for maintainer design. If approved, add a narrow `sessions_spawn.tools` policy object that is schema-validated, propagated through both native and ACP spawn paths, stored or passed only as trusted runtime metadata, and applied as a final narrowing layer in the effective tool-policy pipeline with deny-wins behavior. Update docs and focused tests for precedence, invalid entries, empty allowlists, and native/ACP parity.

## Risks / Open Questions

- A spawn-time tool allowlist is a security boundary: if implemented as a grant instead of a narrowing layer, it could broaden child-agent capabilities beyond configured policy.
- Native subagent and ACP runtime paths have separate params and lifecycle storage, so a partial implementation could leave one runtime bypassing or ignoring the requested policy.
- The expected precedence needs maintainer design: deny-wins behavior and failure behavior for empty or unknown allowlists should be explicit before implementation.
