## Summary

Keep open. Current main still treats `:thread:` session keys as ordinary `sessions_send` targets after normal resolution and visibility checks, and then starts an inter-session agent run against the resolved key. Existing docs and tests confirm thread session keys are valid routed session shapes, but there is no guard or warning for using one as the direct A2A target.

## Best Possible Solution

Add a `sessions_send` guard in `src/agents/tools/sessions-send-tool.ts` after canonical session resolution/visibility and before any `agent` gateway call. Reject canonical targets with a thread suffix using the shared session-key/thread parser, return a clear `status: "error"` message telling agents to use the parent channel session key, and add focused Vitest coverage in the sessions tool tests while preserving existing threadId announce-delivery tests.
