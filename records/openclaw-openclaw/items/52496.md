## Summary

Current main still accepts a resolved thread-shaped session key as a normal `sessions_send` target. The tool resolves `sessionKey`/label inputs, checks visibility, then starts an internal `agent` run with `params.sessionKey = resolvedKey`; there is no refusal or warning for `:thread:` targets even though docs and shared parsers recognize Slack/Discord thread session keys.

## Best Possible Solution

Keep this issue open and add a focused `sessions_send` target guard after canonical resolution/visibility but before any gateway `agent` call. Use the shared session-thread parser to reject canonical thread targets with a clear error telling agents to use the parent channel session key, and add regression tests for raw `sessionKey`, label, and sessionId resolution while preserving existing thread-bound announce/completion delivery behavior.
