## Summary

Keep open. Current main has partial stale-session recovery, but it does not implement the exact reported no-active-run reconciliation: `sessions.abort` can still return `no-active-run` without clearing a persisted `running` row, restart recovery only handles marked/lock-backed main sessions, and a related lifecycle-error cleanup path remains open in the provided related-report context.

## What This Changes

The issue requests recovery for sessions that remain `running` after the underlying LLM run dies, including `sessions.abort` cleanup, gateway-restart reconciliation, and an automatic timeout guard.

## Best Possible Solution

Add owner-reviewed stale-session reconciliation that marks or resets only proven-dead `running` sessions after checking active chat abort controllers, embedded runtime state, and session-lane snapshots. Cover `sessions.abort` no-active-run cleanup, restart reconciliation, and lifecycle-error deferral with focused regression tests.

## Reproduction Assessment

Yes, at source level: seed a persisted session entry with `status: "running"` and no matching active run, then call `sessions.abort`; current code can return `no-active-run` without mutating the row. The full production race is timing-dependent, but the stale-row no-active-run path is directly traceable in current main.

## Solution Assessment

No, current main is not the complete solution. The partial lane timeout, diagnostic recovery, and lifecycle persistence help, but the reported abort/restart reconciliation remains incomplete and should be fixed with guarded stale-state checks rather than unconditional cleanup.
