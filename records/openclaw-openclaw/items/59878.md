## Summary

Close as implemented. Current main now has default-on stuck-session diagnostics that detect long-running processing sessions and conservatively release only stale session lanes when there is no active embedded run, reply operation, or lane task; this addresses the reported indefinite queue blockage from the 2026.3.24 era. The fix is present on main and is included in the v2026.4.29-beta.1 prerelease.

## What This Changes

The issue asks OpenClaw to recover dead session runs so stale `running` session lanes stop blocking follow-up messages after abort, restart, or a configurable stale-lane timeout.

## Best Possible Solution

Keep the conservative watchdog-based recovery in core as the shipped behavior: it should unblock stale lanes without aborting legitimate long-running work, while broader registry/transcript divergence can remain with the narrower open follow-up reports if it still reproduces separately.

## Reproduction Assessment

Yes. The source-level reproduction is a diagnostic session stuck in processing with no active embedded handle or active lane work; current tests exercise that path and verify the session lane is reset.

## Solution Assessment

Yes. The implemented recovery is the safest current shape because it checks active embedded runs, reply operations, and lane tasks before releasing a lane, avoiding the unsafe broad post-restart scan or unconditional abort behavior that could corrupt live runs.
