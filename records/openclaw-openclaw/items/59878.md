## Summary

Keep this issue open. Current main has adjacent lifecycle and restart-recovery hardening, but it still does not satisfy the report's central behavior: `sessions.abort` can return `no-active-run` without repairing a persisted stale `running` row, and restart recovery is scoped to lock-marked main sessions rather than the broader webchat, cron, home-node, and default session cases described here.

## Best Possible Solution

Keep this issue open as the umbrella for stale `running` session recovery unless maintainers explicitly consolidate it into a canonical follow-up. The concrete fix should add regression coverage for `sessions.abort` returning `no-active-run` against a persisted `status: "running"` row, then implement a conservative reconciliation path for stale session rows on abort and gateway restart. Cross-link #63819 for pending terminal lifecycle errors, #60542 for registry/transcript divergence, and #52747 for ACP lane timeouts rather than treating any one of them as a full replacement.

## Evidence

- **sessions.abort no-active-run path does not repair persisted state:** The handler delegates to `chat.abort`, returns `status: "no-active-run"` when no run id is aborted, and only emits `sessions.changed` when `abortedRunId` is present. There is no store update in the no-active-run branch that would clear a stale persisted `running` status.

- **Lifecycle persistence depends on terminal lifecycle events:** Session lifecycle state moves to `running` on lifecycle `start`, and terminal states are derived only from lifecycle `end` or `error`. If the run dies without a terminal lifecycle event, this helper does not reconcile the persisted row by itself.
