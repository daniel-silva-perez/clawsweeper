## Summary

Keep open. Current main still has the reported owner-downgrade chain for isolated cron announce awareness: cron delivery mirrors the result as an untrusted system event, reply/heartbeat paths convert untrusted pending text into non-owner context, and owner-only tool filtering then removes `cron`, `gateway`, and `nodes`. The related hardening PRs explain why the behavior exists, but they do not add a cron-author provenance exception for owner-created announce jobs.

## What This Changes

The issue asks OpenClaw to stop owner-created isolated cron `announce` delivery results from stripping owner-only tools on the next verified owner turn.

## Best Possible Solution

Add provenance-aware handling for isolated cron announce awareness so owner-created cron delivery results do not downgrade the next verified owner turn, while untrusted hook/webhook/exec and externally influenced system events still remove owner-only tools. The fix should include focused regression coverage for cron delivery dispatch, inbound reply draining, heartbeat preflight, command authorization, and final tool inventory filtering.

## Reproduction Assessment

Yes. The current-main reproduction is source-level deterministic: enqueue a `trusted:false` isolated cron awareness event into the main session, drain it into an owner reply, observe `senderIsOwner` become false, then observe `cron`, `gateway`, and `nodes` filtered from the tool list. Existing tests already lock in the individual links, though not the exact end-to-end cron-origin composite.

## Solution Assessment

Unclear. The report's first suggested fix is plausible but could weaken the deliberate #68210 trust label if applied broadly; a safer implementation would distinguish owner-created cron announce awareness from other untrusted system-event sources before bypassing the owner downgrade.
