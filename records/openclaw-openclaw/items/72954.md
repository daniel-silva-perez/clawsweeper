## Summary

Keep open. Current main still contains the reported cron-awareness-to-owner-downgrade chain, and the existing tests lock in the individual links rather than proving a cron-origin exception. The remaining fix changes owner-only tool authorization semantics, so it needs maintainer policy review before implementation.

## What This Changes

The issue asks OpenClaw to stop owner-created isolated cron `announce` awareness events from forcing the next verified owner DM onto the non-owner tool surface.

## Best Possible Solution

Add provenance-aware handling for isolated cron announce awareness so output from an owner-created cron job can be available to the next verified owner turn without stripping owner-only tools, while untrusted hook/webhook/exec and externally influenced system events continue to downgrade the run. The change should include regression coverage across cron delivery dispatch, system-event draining/heartbeat preflight, command authorization, and final tool inventory filtering.

## Reproduction Assessment

Yes. The issue has a live reproduction from `openclaw@2026.4.23`, and current main has a source-level reproduction path: delivered isolated cron awareness is queued as `trusted:false`, pending or drained untrusted events force non-owner context, and owner-only tool policy removes `cron`, `gateway`, and `nodes`.

## Solution Assessment

Unclear. The report's suggested cron-authorship fix is plausible, but the safest maintainable solution is a narrow provenance or audience-aware exception for owner-created cron awareness, not a general rule that verified owner identity always overrides untrusted event downgrades.
