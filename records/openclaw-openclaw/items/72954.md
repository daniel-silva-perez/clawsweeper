## Summary

Current main still contains the exact trust and owner-downgrade path described in the report, so this issue should stay open. The behavior is not covered by the older cron-announce delivery fixes in the related context, and no open PR referencing this issue was found in the read-only checks.

## What This Changes

Owner-created isolated cron `agentTurn` jobs with `announce` delivery should not cause the next verified owner DM to lose owner-only tools after the cron result is mirrored into the main session.

## Best Possible Solution

Add a targeted owner-trust policy for isolated cron announce awareness that preserves untrusted hook/webhook protections while preventing owner-created cron delivery results from downgrading the next verified owner turn. The fix should include regression coverage across cron delivery dispatch, heartbeat context construction, command authorization, and final tool inventory filtering.

## Reproduction Assessment

Yes. The current-main code path is deterministic: a delivered isolated cron announce queues `trusted: false`, heartbeat converts that pending event into `ForceSenderIsOwnerFalse`, command auth hard-overrides owner status, and the tool policy filters owner-only tools. A focused regression test can reproduce this without live Discord by composing those seams.

## Solution Assessment

Unclear. The report's first suggested fix is narrow, but the existing `trusted: false` behavior was added deliberately for security, so the best implementation needs maintainer trust-model review before changing cron awareness or `ForceSenderIsOwnerFalse` semantics.
