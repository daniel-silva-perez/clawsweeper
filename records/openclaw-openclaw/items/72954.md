## Summary

Keep open. Current main still contains the reported chain: isolated cron announce awareness is queued into the main session as untrusted, untrusted pending or drained events force non-owner execution context, and owner-only tools are filtered from that turn. The report is valid, but the repair is security-sensitive and overlaps the open #72201 event-audience direction.

## What This Changes

The issue requests a fix for isolated cron `announce` output being mirrored as an untrusted main-session event, which causes the next verified owner DM to lose owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so output from an owner-created cron job can inform the next verified owner turn without stripping owner-only tools, while externally influenced untrusted system events continue to force non-owner tool policy. The #72201 internal-audience approach is a plausible candidate if maintainers agree it is the right contract.

## Reproduction Assessment

Yes. The issue includes a concrete live trace from `openclaw@2026.4.23`, and current main has a source-level reproduction path through cron awareness queueing, untrusted event detection, owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting owner-created cron output is plausible, but the safest maintainable fix should be narrower than changing general owner resolution, likely by distinguishing cron awareness provenance or using the event-audience design in #72201.
