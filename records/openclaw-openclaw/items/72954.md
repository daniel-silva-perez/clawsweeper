## Summary

Current main still contains the reported cron-awareness trust and owner-downgrade path, so this should stay open for maintainer follow-up. The behavior is narrow and reproducible from code, but the repair changes owner-only tool authorization around untrusted prompt events and needs security/product review rather than automatic closure.

## What This Changes

The issue requests preventing isolated cron `announce` delivery awareness from downgrading a verified owner DM and stripping owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Add a targeted cron-awareness trust policy that distinguishes owner-created isolated cron announce results from other untrusted runtime events, while preserving owner downgrade for hook/webhook/exec content and adding regression coverage through delivery dispatch, heartbeat context, command auth, and final tool filtering.

## Reproduction Assessment

Yes. The current-main path is deterministic: delivered isolated cron announce output queues an untrusted main-session awareness event, heartbeat maps that pending event to `ForceSenderIsOwnerFalse`, command auth hard-overrides owner status, and owner-only tool filtering removes the affected tools.

## Solution Assessment

Unclear. The first suggested fix is plausibly narrow, but marking cron awareness trusted or weakening `ForceSenderIsOwnerFalse` changes security semantics; the safer path is a maintainer-reviewed policy that only exempts the owner-created cron awareness case.
