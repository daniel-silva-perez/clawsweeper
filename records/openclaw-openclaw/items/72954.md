## Summary

Keep open. Current main still contains the reported chain: isolated cron announce awareness is mirrored into the main session as `trusted: false`, untrusted pending or drained system events force non-owner execution context, and owner-only tools such as `cron`, `gateway`, and `nodes` are filtered for that turn. This is valid but security-sensitive because the repair must preserve the untrusted-event downgrade for hook/webhook/exec content.

## What This Changes

The issue requests that owner-created isolated cron `announce` output stop causing the next verified owner DM to lose owner-only tools.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so output from an owner-created cron job can inform the next verified owner turn without stripping owner-only tools, while externally influenced untrusted system events continue to force non-owner tool policy.

## Reproduction Assessment

Yes. The issue has a concrete live reproduction from `openclaw@2026.4.23`, and current main has a source-level reproduction path through cron awareness queueing, untrusted event detection, owner downgrade, and owner-only tool filtering.

## Solution Assessment

Unclear. The report's cron-authorship fix is plausible, but the safest maintainable fix is narrower than changing general owner resolution: distinguish owner-created cron awareness from other untrusted system-event sources, likely via provenance or the event-audience approach in #72201.
