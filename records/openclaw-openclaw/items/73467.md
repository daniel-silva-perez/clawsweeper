## Summary

Keep open. Current main has meaningful mitigations for reload deferral, task inspection, readiness diagnostics, and local log-tail fallback, but it does not prove the central CPU-hot gateway event-loop starvation is fixed: readiness still runs on the gateway event loop, plugin prompt hooks still execute in process, and the narrower hot-path report remains open.

## What This Changes

The issue asks OpenClaw to keep gateway readiness/control-plane, log following, and reload lifecycle responsive under orchestration load instead of stalling behind CPU-hot in-process agent/plugin work or stale active-run accounting.

## Best Possible Solution

Keep this as a core reliability tracker until maintainers either designate #74860 as the canonical hot-path implementation issue and link any separate reload/accounting follow-up, or land fixes that keep probes/operator commands responsive or explicitly degraded under CPU-hot plugin/model work, bound or offload plugin prompt/tool/provider-auth paths, and keep active-work blockers inspectable and reapable.

## Reproduction Assessment

Unclear for current main. The issue has strong operational evidence and a plausible load shape, but no deterministic local reproduction; a high-confidence repro would need live Discord/WebUI orchestration with memory-style prompt hooks, continuous /readyz probing, and event-loop CPU profiles.

## Solution Assessment

Not applicable as a patch review. The maintainable solution is likely a set of narrow hot-path and lifecycle fixes, plus longer-term control-plane/runtime separation, rather than treating the current mitigations as complete.
