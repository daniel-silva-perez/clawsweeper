## Summary

Keep open. Current main has meaningful mitigations for the reported 2026.4.26 failure shape, but the central CPU-hot Gateway event-loop starvation is not proven fixed: readiness diagnostics are still served by the same Node event loop, plugin/tool/auth hot paths remain in-process, and the narrower hot-path report is still open in the provided context.

## What This Changes

The issue asks for Gateway control-plane probes, Discord/WebUI/operator commands, log following, and reload accounting to stay responsive or degrade safely during orchestration-heavy agent/plugin work.

## Best Possible Solution

Keep this as a core reliability tracker until maintainers either designate #74860 as the canonical hot-path implementation ticket and split any separate reload/task-accounting follow-up, or land a combined fix that keeps control-plane probes and operator paths responsive or explicitly degraded under CPU-hot plugin/model work.

Unclear for current main. The report has strong live operational evidence, but reproducing it with high confidence requires a loaded Discord/WebUI orchestration setup plus memory-style prompt hooks and CPU/event-loop profiling rather than a deterministic unit or integration test in the repo.

## Solution Assessment

Not applicable as a patch review. The current main mitigations are directionally useful, but the maintainable solution should be narrower implementation work around the confirmed hot path and any remaining stale active-work accounting, followed by live load validation.
