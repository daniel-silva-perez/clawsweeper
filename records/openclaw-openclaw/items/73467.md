## Summary

Keep open. Current main has useful mitigations after the reported 2026.4.26 build, but it does not prove the central CPU-hot gateway event-loop starvation is fixed: `/readyz` diagnostics still run on the same Node event loop, plugin prompt/tool paths remain in-process, and the narrower event-loop hot-path report remains open.

## What This Changes

The issue reports gateway event-loop/control-plane stalls on 2026.4.26 under orchestration load, including loopback `/readyz` timeouts, delayed Discord/WebUI control traffic, fragile `logs --follow`, and reloads blocked by active run accounting.

## Best Possible Solution

Keep this as a core reliability tracker until maintainers land narrower fixes that keep control-plane probes and operator commands responsive or explicitly degraded under CPU-hot model/plugin work, reap stale active-work blockers, and bound or offload plugin prompt/tool/provider-auth hot paths. The longer-term architecture direction should remain aligned with the distributed control-plane/runtime separation tracked elsewhere.

## Reproduction Assessment

Unclear for current main. The issue has strong operational evidence and a plausible load shape, but there is no deterministic repo-local reproduction, and the environment includes live Discord/WebUI orchestration plus an external `memory-lancedb-pro` plugin.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but not a complete solution because the remaining bug class still spans gateway liveness, plugin scheduling/offload, active-work accounting, and the narrower open hot-path report.
