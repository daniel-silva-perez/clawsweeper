## Summary

Keep open. Current main contains several mitigations after the reported 2026.4.26 build, but I could not verify that the core CPU-hot gateway event-loop starvation or the broader reload/accounting failure mode is fixed. The remaining surface still spans same-process readiness handling, in-process plugin prompt/tool paths, active-work accounting, and the open narrower event-loop stall report #74860.

## What This Changes

The issue reports 2026.4.26 gateway event-loop starvation under orchestration load, causing `/readyz` timeouts, delayed Discord/WebUI control-plane operations, and reload/restart deferral behind stale active work accounting.

## Best Possible Solution

Keep this as a core reliability tracker until maintainers split and resolve the remaining work: land or explicitly supersede the #74860 synchronous tool/plugin-registry hot path, decide whether active reply/embedded/task accounting needs a stronger inspect/reaper path, and keep the longer control-plane/runtime separation discussion under #42026. The already-fixed `logs --follow` fallback should remain treated as a related symptom handled by #74583, not as closure proof for this incident.

## Reproduction Assessment

Unclear for current main. The issue has strong operational evidence and a plausible load shape, but no deterministic local reproduction; high-confidence verification would need live/load validation with Discord/WebUI, memory-style prompt hooks, frequent `/readyz` probes, and CPU/event-loop profiles during a stall.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful, but they are not yet the narrowest complete fix because in-process plugin hook/tool paths and active-work accounting still need scoped maintainer decisions.
