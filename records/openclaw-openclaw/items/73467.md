## Summary

Keep open. Current main has meaningful mitigations after the reported 2026.4.26 build, including readiness event-loop diagnostics, a default restart deferral cap, task-registry reconciliation, and a local logs fallback, but those do not prove the core CPU-hot gateway event-loop starvation is fixed. The remaining failure mode still crosses control-plane liveness, in-process plugin hooks, active task accounting, and the open narrower event-loop stall report #74860.

## What This Changes

The issue asks for gateway control-plane resilience under orchestration load so `/readyz`, Discord/WebUI traffic, reloads, plugin hooks, and active task accounting do not stall behind CPU-hot agent/plugin work.

## Best Possible Solution

Keep this as a core reliability tracker until maintainers split and land narrower fixes: one for the current synchronous tool/plugin-registry stall class, one for stale active-work/reload accounting, and a longer-term control-plane/runtime separation path aligned with #42026. The already-fixed `logs --follow` fallback should stay handled through #74583 rather than being the reason to close this broader incident.

## Reproduction Assessment

Unclear for current main. The issue contains strong operational evidence and a plausible load shape, but there is no deterministic repo-local reproduction here, and the exact environment includes live Discord/WebUI orchestration plus an external `memory-lancedb-pro` plugin.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but not the narrowest complete solution because the remaining bug class still needs scoped maintainer decisions across gateway liveness, plugin scheduling/offload, task accounting, and the open #74860 hot path.
