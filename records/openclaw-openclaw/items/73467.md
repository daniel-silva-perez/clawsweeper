## Summary

Keep open. Current main has relevant mitigations after the reported 2026.4.26 build, including event-loop health reporting in readiness, a bounded default restart deferral, task-registry reconciliation for lost backing state, and local log-tail fallback. Those do not prove the reported CPU-hot main-thread starvation is fixed because readiness, WebSocket/RPC traffic, agent prompt hooks, and plugin work still share the gateway Node event loop, and the remaining work spans core liveness, plugin hook scheduling/offload, reload lifecycle policy, and task accounting.

## What This Changes

The issue requests gateway reliability work so `/readyz`, Discord/WebUI/control-plane traffic, `openclaw logs --follow`, and config reload/restart remain responsive and inspectable during CPU-hot orchestration and plugin load.

## Best Possible Solution

Keep this as a core reliability follow-up: preserve the shipped mitigations, then define maintainer-owned scope for control-plane survivability when agent/plugin work saturates the main event loop, including inspectable/reapable active-work blockers and cancellable or offloaded plugin hook execution. Route the standalone log-follow symptom through #74583 and the larger runtime split through #42026 without treating either as a complete replacement for this incident.
