## Summary

Keep open. Current main and the latest release still enforce a single exact memory-flush write target, so `memory/daily/YYYY-MM-DD.md` remains blocked; the related merged subdirectory work covers recall/dreaming ingestion, not the flush write sandbox.

## What This Changes

The issue requests allowing memory flush writes to target validated subpaths under `memory/` rather than only the single root-level daily memory file.

## Best Possible Solution

Define a narrow memory-flush write policy that validates requested paths relative to the workspace, allows only intended flush targets under `memory/` when that is the chosen contract, appends to the validated requested path, rejects traversal and sibling prefixes, and updates the memory-core prompt/plan contract plus tests.

## Reproduction Assessment

Yes. The unit-level reproduction is to wrap a write tool with `relativePath: "memory/2026-03-24.md"`, then call it with `path: "memory/daily/2026-03-24.md"`; current code follows the same branch that the existing test uses to reject `memory/other-day.md`.

## Solution Assessment

No for current main, and unclear for the issue’s suggested prefix check. The maintainable fix should use the existing path resolver plus a segment-safe relative-path check and retarget the append operation, rather than a raw `startsWith` comparison.
