## Summary

Keep open. The linked blank-tool-name failure remains a valid agent-runtime bug, but this PR’s literal mutable sentinel approach introduces dispatch, streaming-recovery, and replay-sanitization regressions that should block merge until the design is revised.

## What This Changes

The PR changes agent tool-call normalization to replace blank or whitespace-only names with `_blank`, treats that sentinel as incomplete for unknown-tool loop classification, and updates focused tests to expect it.

## Best Possible Solution

Fix the blank-name loop without storing a dispatchable sentinel in the mutable message object. A narrower design should use a reserved non-dispatchable failure path or explicit invalid-tool representation, preserve later id-based recovery, and keep replay sanitization dropping malformed blank history.

## Reproduction Assessment

Yes. The linked report provides concrete blank-name loop evidence, and current main has focused unit coverage showing blank no-id calls remain blank. The PR regressions are reproducible from source with focused cases for same-object streaming deltas, an allowlisted `_blank` tool, and no-allowlist replay sanitization.

## Solution Assessment

No. Replacing blanks with the literal `_blank` inside the mutable tool-call object is not the best fix because it changes later normalization behavior and can collide with real tool names. A reserved, non-dispatchable error path would be safer and more maintainable.
