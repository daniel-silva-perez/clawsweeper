## Summary

Keep this PR open. Current main already covers the chat picker/provider-qualified model selection part, but it still does not implement the root `llm` shorthand config key, and the proposed config implementation has blocking correctness and credential-surface gaps before it could merge.

## What This Changes

The PR adds an optional top-level `llm` config object that expands into custom model provider/default model settings, and changes the chat model picker to submit provider-qualified model refs.

## Best Possible Solution

A maintainer should decide whether OpenClaw wants a new root shorthand at all. If yes, implement it as an explicit config-owner change with schema, generated schema, labels/help, docs, SecretInput/redaction coverage, provider/API/ref validation, non-destructive provider/model merging, fallback-preserving primary assignment, and regression tests; keep the current main chat picker implementation.

## Reproduction Assessment

Yes for the PR defects: the diff shows deterministic cases such as an existing provider plus a different `llm.model`, or an existing `agents.defaults.model.fallbacks` with no primary. For the current-main status, static source checks confirm the `llm` key is absent while picker behavior is covered by current tests.

## Solution Assessment

No. The branch mixes an already-solved UI picker fix with a new root config alias that needs product approval and several correctness/security fixes before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve fallbacks when seeding the primary model:** `src/config/defaults.ts:593`
  - body: When `agents.defaults.model` already contains `fallbacks` but no `primary`, this assignment replaces the whole model object with `{ primary }`. Adding `llm` would silently drop the configured failover chain on every load, so spread the existing model object before setting `primary`.
  - confidence: 0.93
- **[P2] Add the shorthand model before pointing primary at it:** `src/config/defaults.ts:576-578`
  - body: If `models.providers[providerId]` already exists, the PR preserves that provider unchanged but can still set `agents.defaults.model.primary` to `providerId/modelId`. When the existing provider lacks that model, the new default points at a model that cannot resolve.
  - confidence: 0.91
- **[P2] Reject provider IDs that cannot form model refs:** `src/config/zod-schema.core.ts:270`
  - body: `provider_id` is later used as the left side of a `provider/model` ref, but the schema allows values containing `/`. A value like `foo/bar` becomes a primary ref that parses as provider `foo`, while the provider was inserted under `models.providers["foo/bar"]`.
  - confidence: 0.89
- **[P2] Validate llm provider API values:** `src/config/zod-schema.core.ts:265`
  - body: `llm.provider` is accepted as any string and then cast into `ModelProviderConfig["api"]`. Typos pass validation and reach downstream transport selection as unsupported API ids; validate this against the supported shorthand values plus the existing model API enum.
  - confidence: 0.87
- **[P2] Cover the new secret path in generated config surfaces:** `src/config/zod-schema.core.ts:267`
  - body: The PR adds a credential-bearing `llm.api_key` path but does not update the generated config schema/help/labels or add UI/redaction coverage for that public path. Secret-bearing config aliases need the same generated and user-facing metadata as `models.providers.*.apiKey`.
  - confidence: 0.8
