## Summary

Keep open. Current main has partial guardrails around stale gateway service entrypoints, update-time service refresh, and newer OpenAI Codex transport/model handling, but it still does not prove the split user/system install policy or the newer-install `/v1/chat/completions` timeout is fixed. The issue is recent and has related open implementation work that needs maintainer review first.

## What This Changes

The issue asks OpenClaw to clarify and/or fix split user-vs-system gateway installs so upgrades do not leave the live service pinned to an older path or make `/v1/chat/completions` unreliable when promoting `openai-codex/gpt-5.5`.

## Best Possible Solution

Keep the issue open until OpenClaw has a documented split user/system install migration path, status/doctor output that shows the active service entrypoint/package root/version/scope, and a reproduced fix or live smoke proving that a managed Linux gateway service can move to the newer install and reliably serve `/v1/chat/completions` with the intended Codex model.

## Reproduction Assessment

No. The report is credible and includes rollback evidence, but this review found no current-main fixture, logs, or live service-path smoke that reproduces the newer-install timeout with high confidence.

## Solution Assessment

Unclear. The existing doctor/update/Codex transport changes are the right nearby surfaces, but the best fix depends on whether the remaining failure is service metadata drift, system/user unit ownership, or an actual gateway runtime regression.
