## Summary

Keep open. Current main has useful adjacent fixes for gateway service entrypoint drift, update-time service refresh, stale systemd restart loops, Codex app-server idle timeouts, and OpenAI Codex OAuth transport, but it still does not settle the split user/system install policy or prove that repointing the gateway to the newer install no longer makes /v1/chat/completions unreliable.

## What This Changes

The issue asks OpenClaw to make split user/system gateway installs understandable and safe during upgrades, including warning or migration support when the live service is pinned to an older install and diagnosing the newer-install /v1/chat/completions timeout.

## Best Possible Solution

Keep this issue open until OpenClaw has a documented split user/system install migration story, status/doctor output that clearly reports the active service entrypoint, package root, version, realpath, and systemd scope, and either a reproduced fix for the newer-install /v1/chat/completions timeout or enough logs to route it to update/doctor, systemd supervision, or the OpenAI-compatible runtime path.

## Reproduction Assessment

No. The host-state narrative is credible and current main has unit coverage for nearby drift/restart behavior, but there is no high-confidence reproduction path for the newer-install gateway timeout itself.

## Solution Assessment

Unclear. The existing guardrails are in the right areas, but the narrowest maintainable fix depends on whether #74920’s diagnostics and any new gateway logs show service metadata drift, system-scoped restart behavior, or an actual OpenAI-compatible endpoint runtime regression.
