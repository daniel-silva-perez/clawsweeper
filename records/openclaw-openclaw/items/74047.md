## Summary

Keep open. Current main has partial guardrails for stale gateway service entrypoints, staged package updates, duplicate-install installer warnings, and the OpenAI Codex OAuth transport, but it still does not fully resolve the split user/system install policy or prove that repointing the managed gateway to the newer install no longer causes `/v1/chat/completions` timeouts.

## What This Changes

The issue asks OpenClaw to clarify and fix split user-vs-system gateway installs so upgrades do not leave the live gateway pinned to an older path or make `/v1/chat/completions` unreliable when promoting `openai-codex/gpt-5.5`.

## Best Possible Solution

Keep this issue as the canonical tracker until OpenClaw has a documented split user/system install migration path, status/doctor output that exposes the active CLI and service entrypoint/package root/version/scope, and a live or fixture-backed proof that a managed Linux gateway can move to the newer install and reliably serve `/v1/chat/completions` with the intended Codex model.

## Reproduction Assessment

No. The issue has credible host-state and rollback evidence, but this read-only review found no current-main live service layout, gateway logs, or test fixture that reproduces the newer-install timeout with high confidence.

## Solution Assessment

Unclear. The existing updater, doctor, installer-warning, and Codex transport fixes are the right nearby surfaces, but the narrowest maintainable fix depends on whether the remaining failure is multi-root install policy, systemd unit scope/ownership, or a gateway runtime regression.
