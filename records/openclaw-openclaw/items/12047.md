## Summary

Keep open. Current main downloads Telegram document attachments and can extract text-like files/PDFs through the shared media path, but it still does not implement the full requested document extraction: Office-style DOCX/XLSX/PPTX MIME types are skipped as binary before extraction, the default input-file allowlist remains text/JSON/PDF only, and the bundled document-extract plugin still registers only a PDF extractor. Related DOCX work in #57612 remains open rather than merged.

## What This Changes

Request to extract text from Telegram document attachments so agents can read document contents instead of only seeing a media placeholder.

## Best Possible Solution

Keep this open and solve it in the shared document extraction path. The likely product/code direction is to review or adapt #57612, add selected Office document MIME support through the existing document-extractor contract and bundled document-extract plugin, allow those MIME types through only when a bounded extractor is available, preserve binary safety checks and untrusted-content wrapping, and cover Telegram attachments plus shared extraction paths with focused tests and docs.

## Risks / Open Questions

- Closing now would overstate current document support: Telegram can download attachments, but Office-style document text is still not extracted into prompt context by default.
- A Telegram-only parser would leave the same gap for WebChat, OpenResponses input_file, chat.send, and other shared media-entry paths.
- Office extraction parses untrusted ZIP/XML or legacy binary document content, so supported formats, dependency choices, decompression/XML limits, and output caps need maintainer review before broadening the binary MIME guard.
