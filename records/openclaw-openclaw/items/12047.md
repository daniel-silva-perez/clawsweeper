## Summary

Keep open. Current main has moved partway past the original report by downloading Telegram document attachments and extracting text-like files/PDFs through the shared media path, but the requested document extraction is still incomplete for Office-style attachments. DOCX/XLSX/PPTX MIME types are still treated as binary before extraction, the default input-file allowlist remains text/JSON/PDF, and the related DOCX implementation candidate remains open rather than merged.

## What This Changes

Extract readable text from Telegram document attachments and include it in the agent context instead of leaving the agent with only a media placeholder.

## Best Possible Solution

Route the remaining capability through the shared document extraction path: review or adapt #57612, add selected Office MIME support only when a bounded extractor is available, preserve binary safety checks and untrusted-content wrapping, and cover Telegram plus shared media/input-file behavior with focused tests and docs.

## Risks / Open Questions

- Closing now would overstate current support because Telegram-downloaded Office documents still do not become extracted prompt text by default.
- A Telegram-only fix would leave the same extraction gap for shared media surfaces such as WebChat, OpenResponses input_file, chat.send, and related attachment paths.
- Office extraction parses untrusted ZIP/XML or legacy binary document content, so supported formats, dependency choices, decompression/XML limits, and output caps need maintainer review.
