## Summary

Current main still downloads Telegram document attachments and forwards them into the shared media pipeline, but it does not fully implement the requested document text extraction. Plain text-like files and PDFs are handled; Office-style DOCX/XLSX/PPTX remain outside the default file MIME allowlist, application/vnd Office MIME types are skipped as binary before extraction, and the bundled document-extract plugin still registers only a PDF extractor. The existing maintainer/Codex discussion correctly kept this open, and related DOCX work such as #57612 is still open rather than merged.

## Best Possible Solution

Keep this open and solve it in the shared document extraction path. The likely product/code direction is to review or adapt #57612, add selected Office document MIME support through the existing document-extractor contract and bundled document-extract plugin, allow those MIME types through only when a bounded extractor is available, preserve binary safety checks and untrusted-content wrapping, and cover Telegram attachments plus shared extraction paths with focused tests and docs.

## Risks / Open Questions

- A Telegram-only parser would leave the same gap for WebChat, OpenResponses input_file, chat.send, and other shared media-entry paths.
- Office/DOCX/XLSX/PPTX extraction parses untrusted ZIP/XML or legacy binary document content, so supported formats, dependency choice, decompression/XML limits, and output caps need maintainer review.
- The current application/vnd.* binary guard is a safety boundary with explicit regression coverage; loosening it globally without a bounded extractor allowlist could reintroduce prompt input inflation or unsafe binary decoding.
