## Summary

Keep open. Current main handles Telegram document download and already extracts text/JSON/PDF through the shared file path, but the Office-document part of the request is still intentionally unsupported: default file MIME support stops at text/JSON/PDF, Office MIME types are skipped as binary before extraction, and the related DOCX implementation remains open rather than merged.

## What This Changes

The issue requests extracting text from Telegram document attachments so agents can read files such as DOCX, PDF, TXT, and Markdown instead of seeing only a media placeholder.

## Best Possible Solution

Finish this through the shared document extraction pipeline rather than a Telegram-only path: add bounded Office MIME and extractor support, preserve untrusted-content wrapping and binary-safety checks, and cover Telegram handoff plus shared media/input-file behavior with focused tests and docs.

## Reproduction Assessment

Yes. A Telegram DOCX/XLSX/PPTX document can be traced from msg.document into MediaPath/MediaType and then into applyMediaUnderstanding, where application/vnd.* Office MIME types are skipped before extraction; the current Office skip regression tests confirm the behavior.

## Solution Assessment

No for a Telegram-specific extractor. The narrow maintainable solution is a shared document-extraction capability, likely by reviewing or adapting #57612 with explicit parser, MIME-scope, decompression, and resource-limit decisions.
