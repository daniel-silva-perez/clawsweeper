## Summary

Keep open. Current main fixes the memory-core cron startup race and can clear stale diagnostic bookkeeping when no active work remains, but the core active stuck-session recovery requested here is still not implemented: diagnostics call recovery without active-abort policy, recovery remains warning-only for active embedded/reply/session-lane work, and public config/docs expose only a warning threshold.

## What This Changes

The issue requests automatic recovery for agent processing lanes stuck in `processing`, plus retry or readiness handling for memory-core managed dreaming cron registration after gateway restart.

## Best Possible Solution

Ship one canonical core recovery path that keeps the memory-core cron retry, adds a documented configurable active stuck-session abort threshold, maps diagnostics safely to active embedded/reply/lane state, aborts or force-clears only after that threshold, preserves queued-message semantics, and surfaces a clear terminal error/final state with regression coverage for hung model, tool, delivery, WebSocket-disconnect, and gateway-restart paths.

## Reproduction Assessment

Yes for the remaining missing behavior: current source and tests provide a focused reproduction that diagnostic recovery does not abort active embedded/reply/lane work by default. The full live gateway triggers are intermittent, but the reported logs plus current code are enough to verify the core recovery gap.

## Solution Assessment

No, current main is only a partial solution because it fixes memory-core cron retry and stale/no-active-work cleanup but leaves active stuck-session recovery warning-only. The safer path is maintainer review of #73243 or #39117, or an equivalent narrow core patch with config, docs, and tests.
