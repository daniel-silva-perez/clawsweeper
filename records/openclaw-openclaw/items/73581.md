## Summary

Keep this issue open. Current main partially addresses the report with memory-core cron retry/re-resolution and conservative stale-lane release, but the central active processing-lane recovery request remains intentionally unresolved: the watchdog does not enable active abort recovery, and the recovery code/tests/docs still skip active embedded runs, active reply work, and active lane tasks by default.

## Best Possible Solution

Keep this issue open for the remaining core lane-recovery design. Treat the memory-core cron startup race as fixed on main, then implement and test a configurable watchdog policy that can safely map a stuck diagnostic session to active embedded/reply/lane work, abort or force-clear only after a defensible threshold, inject a terminal synthetic error/final state, preserve queued follow-up semantics, and cover hung model/tool/final-delivery paths. Keep related issues such as #68649 and #72338 linked as narrower reproductions.
