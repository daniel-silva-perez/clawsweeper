## Summary

Keep this issue open. Current main fixes the memory-core dreaming cron startup race, but the broader agent processing-lane timeout/recovery request is still not implemented. Stuck-session diagnostics still warn and emit telemetry only, while active-run abort/reset remains tied to explicit queue modes, slash/reset/stop flows, or restart recovery rather than automatic watchdog recovery. The later comments add corroborating reports, so this is active product/debugging work rather than cleanup material.

## Best Possible Solution

Keep this issue open for the core agent processing-lane recovery work. The best implementation is a configurable watchdog path that resolves the stuck sessionKey to the active embedded/reply run, aborts or marks that run failed, injects a clear terminal error/final state, drains or preserves queued work according to queue policy, and avoids disrupting valid long-running turns. Treat the memory-core dreaming cron race as already fixed on main by e84ebeafbd67a420faf22c516562c67dc66b8e7c and note or split that portion separately.

## Risks / Open Questions

- A watchdog that blindly resets processing sessions could abort legitimate long-running turns or corrupt queued follow-ups; the fix needs to map sessionKey to the active run and apply explicit queue/recovery policy.
- The issue combines one fixed plugin race with a still-open core lane recovery bug, so maintainers may want to split the fixed memory-core portion before closing anything.
