## Summary

Keep this issue open. Current main partially addresses the report: memory-core dreaming cron startup reconciliation now retries, and gateway diagnostics can release stale session-lane bookkeeping. The central active processing-lane timeout/recovery request remains unresolved because the watchdog still invokes recovery without active-abort enablement, and the recovery code/tests/docs explicitly keep active embedded runs, active reply work, and active lane tasks warning-only by default.

## Best Possible Solution

Keep this issue open for the remaining core recovery work. Treat memory-core cron startup reconciliation as fixed, then either land a reviewed general watchdog/active-abort design from an implementation candidate such as #73243 or #39117, or build the same behavior directly in core: configurable threshold, safe mapping from stuck diagnostics to active run/reply/lane state, abort or force-clear only after a defensible timeout, synthetic terminal error/final state, preserved queued-message semantics, and regression coverage for hung model, tool, delivery, and WebSocket-disconnect paths.
