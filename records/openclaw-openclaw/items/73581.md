## Summary

Keep this issue open. Current main partially addresses the report: memory-core startup cron reconciliation now retries and re-resolves cron, and diagnostics can release stale session-lane bookkeeping when no active work remains. The central processing-lane recovery request is still unresolved because the watchdog does not enable active abort recovery and the recovery implementation intentionally skips active embedded runs, active reply work, and active lane tasks by default.

## Best Possible Solution

Keep this issue open for the remaining core lane-recovery work. The best path is to treat the memory-core cron race as fixed, then implement a configurable watchdog policy that safely maps a stuck diagnostic session to active embedded/reply/lane work, aborts or force-clears only after a defensible threshold, injects a synthetic terminal error/final state, preserves queued-message semantics, and adds regression coverage for hung model/tool/delivery paths. Related narrower issues can remain linked as specific reproductions.
