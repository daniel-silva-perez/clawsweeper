## Summary

Keep open. Current main partially addresses the report by adding memory-core cron retry behavior, but the core processing-lane recovery remains incomplete: diagnostics still invoke recovery without active-abort permission, and the recovery path intentionally leaves active embedded runs, reply work, and active lane tasks alone by default.

## What This Changes

The issue requests configurable timeout and automatic recovery for stuck agent processing lanes, plus retry or readiness handling for memory-core managed dreaming cron registration after Gateway restart.

## Best Possible Solution

Keep the memory-core startup retry, then land one canonical core recovery path that adds a documented configurable active stuck-session abort threshold, safely maps diagnostic state to active embedded/reply/lane work, aborts or force-clears only past that threshold, preserves queued-message semantics, and emits a clear terminal user-visible error with regression coverage for hung model, tool, delivery, WebSocket-disconnect, and restart-interruption paths.

## Reproduction Assessment

Yes for the remaining code-level issue: current main source and tests show diagnostic recovery does not abort or release active embedded, reply, or active lane work by default. The live triggers are intermittent, but the provided logs plus current recovery behavior give a high-confidence reproduction target.

## Solution Assessment

No, current main is not the complete solution because it implements memory-core cron retry and conservative stale-lane cleanup only. The narrower maintainable fix is to review and land one canonical active-recovery PR, such as #39117 or #73243 if it proves correct, with config/docs/schema and regression coverage.
