## Summary

Keep this issue open. Current main has fixed the memory-core dreaming cron restart race and added a deliberately conservative stale-session recovery path, but the main reported problem remains: the diagnostics watchdog still does not automatically abort or force-clear active embedded runs, active reply operations, or active lane tasks when a session stays in `processing` with queued work. The same-day issue discussion also adds multiple corroborating reports, including long `queueDepth=1` stalls and Windows/WebChat/TUI symptoms, so this is not stale or low-signal.

## Best Possible Solution

Keep this issue open for the remaining core processing-lane recovery work. The memory-core cron portion can be treated as fixed on current main, but the best product/code path is a separate, configurable watchdog policy that can safely identify genuinely wedged active runs, map the diagnostic session to active embedded/reply/lane work, abort or force-clear only after a defensible threshold, preserve queue semantics, and surface a synthetic terminal error so later messages can proceed. Related narrower issues such as #53008, #53889, #68649, #72810, and #72338 should remain useful subcase evidence rather than replacing this broader tracker.
