## Summary

Keep this issue open. Current main fixes the memory-core dreaming cron startup race, but the broader processing-lane recovery request is still current: stuck-session diagnostics warn and emit telemetry, while active-run abort/reset paths remain explicit user or lifecycle flows rather than automatic watchdog recovery. Later comments in the issue also corroborate the same stuck `processing`/`queueDepth=1` behavior, including a much longer 43-minute case, so this is not cleanup-close material.

## Best Possible Solution

Keep this issue open for the core agent processing-lane recovery work. The best fix is a configurable watchdog recovery path that resolves a stuck diagnostic `sessionKey` to the active embedded/reply run, aborts or marks that run failed after a safe threshold, clears or preserves queued work according to queue policy, and injects a clear synthetic terminal error/final state so later messages can proceed. Treat the memory-core dreaming cron race as already fixed on main by e84ebeaf/#73493 and, if maintainers want cleaner tracking, split that fixed half out from the remaining lane-recovery bug.

## Risks / Open Questions

- The report combines a fixed memory-core plugin race with unresolved core lane recovery work; closing the whole issue now would lose the remaining user-visible bug.
- Automatic stuck-lane recovery needs careful design because valid long-running turns exist. The implementation must map `sessionKey` to the active run, abort only the wedged work, preserve or clear queued messages according to queue policy, and surface a terminal error without corrupting transcripts.
