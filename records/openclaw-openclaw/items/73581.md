## Summary

Keep open. Current main has useful partial fixes for stale lane cleanup and memory-core cron retry, but the core request remains unresolved because active embedded runs, active reply work, and active session-lane tasks are still warning-only by default and there is no configurable active stuck-session abort threshold wired into diagnostics.

## What This Changes

The issue requests automatic timeout recovery for stuck agent processing lanes plus retry/backoff for memory-core managed dreaming cron reconciliation after Gateway restart.

## Best Possible Solution

Keep the memory-core retry path, then land one canonical core recovery design that adds a documented configurable active stuck-session abort threshold, safely maps diagnostics to active embedded/reply/lane work, aborts or force-clears only after that threshold, preserves queued-message semantics, and emits a clear terminal user-visible error with focused regression coverage.

## Reproduction Assessment

Yes for the remaining code-level gap: current source and tests show that diagnostics does not abort or release active embedded/reply/lane work by default. The full live triggers are intermittent, but the issue body and corroborating comments provide enough log evidence to validate the affected processing-lane state.

## Solution Assessment

No, current main is only a partial solution: stale lanes without active work can recover and memory-core startup cron now retries, but active stuck processing remains observe-only. The maintainable path is to review and land one canonical active-recovery PR or an equivalent narrow core patch with config, docs, and tests.
