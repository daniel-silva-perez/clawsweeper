## Summary

Keep open. Current main at 1b25dcf partially addresses the report: memory-core managed dreaming cron startup retry is present, and diagnostics can release stale session lanes when no active work remains. The main requested behavior is still missing because active embedded runs, reply operations, and active session-lane tasks remain warning-only by default, with no public configurable active stuck-session abort threshold or typed abort event. Open PRs #73243 and #39117 remain implementation candidates rather than proof that current main fixes the issue.

## Best Possible Solution

Keep this issue open. Treat the memory-core dreaming cron startup race as fixed on current main, then review and land one focused active stuck-session recovery path from #73243, #39117, or an equivalent core change: a configurable active-run abort threshold, safe mapping from diagnostics to active embedded/reply/lane state, abort or force-clear after a defensible timeout, a synthetic user-visible terminal error/final state, preserved queued-message semantics, and regression coverage for hung model, tool, delivery, WebSocket-disconnect, and gateway-restart paths.
