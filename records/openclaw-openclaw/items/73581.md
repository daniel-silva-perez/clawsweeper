## Summary

Keep open. Current main partially addresses the report by adding memory-core cron startup retry and stale-lane cleanup when no active work remains, but the main requested active stuck-session recovery is still missing: active embedded runs, active reply work, and active session-lane tasks remain warning-only by default, and config/docs expose only a warning threshold. Open PRs #73243 and #39117 are relevant implementation candidates, not merged proof.

## What This Changes

The issue requests configurable automatic recovery for stuck agent processing lanes and retry/delay behavior for memory-core managed dreaming cron registration during gateway restarts.

## Best Possible Solution

Ship one canonical core recovery path that keeps the current memory-core cron retry, adds a documented configurable active stuck-session abort threshold, safely maps diagnostics to active embedded/reply/lane state, aborts or force-clears only after that threshold, preserves queued-message semantics, and emits a clear user-visible terminal error/final state with regression coverage for hung model, tool, delivery, WebSocket-disconnect, and gateway-restart paths.

## Reproduction Assessment

Yes for the missing recovery behavior: current tests and source provide a focused reproduction that a stuck diagnostic recovery call does not abort active embedded/reply/lane work by default. The full live gateway triggers are intermittent, but the reported logs and current code path are enough to verify the remaining gap.

## Solution Assessment

No, current main is not the complete solution because it fixes only stale/no-active-work cleanup and the memory-core cron retry. The maintainable path is to review and land one of the open stuck-session recovery PRs, or an equivalent narrow core patch, with config/docs/schema and regression coverage.
