## Summary

Keep this issue open. Current main fixed the memory-core dreaming cron startup race and added a conservative stale-session-lane recovery path, but it explicitly does not auto-abort active embedded runs, active reply operations, or active lane tasks. That leaves the central processing-lane timeout/recovery request unresolved, especially for the corroborated stuck `processing` / `queueDepth=1` cases.

## Best Possible Solution

Keep this issue open for the remaining core processing-lane recovery work. The best path is to split or note the memory-core cron race as fixed, then implement a configurable watchdog policy that can safely map a stuck diagnostic session to the active embedded/reply run, abort or force-clear only after a defensible threshold, reset the session lane when appropriate, preserve queue semantics, and surface a clear synthetic terminal error so later messages can proceed.
