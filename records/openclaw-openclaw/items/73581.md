## Summary

Keep open. Current main fixes the memory-core dreaming cron startup race, but the broader agent processing lane recovery request is still unresolved: stuck-session diagnostics warn and emit telemetry, while active-run abort/reset remains tied to explicit interrupt, stop, reset, or restart flows rather than automatic watchdog recovery. The provided discussion also includes a later user reporting the same problem, so this is active bug-tracking material rather than cleanup material.

## Best Possible Solution

Keep this issue open for the core agent processing lane recovery work. The best fix is a configurable watchdog path that resolves a stuck sessionKey to the active run, aborts or marks that run failed, clears or preserves queued work according to queue policy, and emits a synthetic terminal error/final state so later messages can proceed. Treat the memory-core dreaming cron race as fixed on main by e84ebeaf/#73493, but do not close the combined report until the general lane recovery behavior lands.

- Automatic stuck-lane recovery is non-trivial: it needs to map a diagnostic sessionKey to the active embedded run, abort only the wedged run, preserve or clear queued work according to queue policy, and surface a clear terminal error without corrupting legitimate long-running turns.
