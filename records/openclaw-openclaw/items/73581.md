## Summary

Keep this issue open. Current main has fixed the memory-core dreaming cron startup race and added partial stale-session-lane recovery, but it still does not implement the requested automatic recovery for active processing-lane stalls. The current watchdog detects stuck `processing` sessions, but active embedded runs, active reply operations, and active lane tasks remain warning-only by default.

## Best Possible Solution

Keep this issue open for the remaining core processing-lane recovery work. The best path is a configurable watchdog recovery flow that maps a stuck diagnostic session key to the active embedded or reply run, aborts or marks the wedged run failed after a safe threshold, releases the session lane when cleanup drains or is force-cleared, and surfaces a synthetic terminal error/final state so later messages can proceed. Treat the memory-core dreaming cron race as already fixed on main and split that half out only if maintainers want cleaner tracking.
