## Summary

Keep open. Current main only partially addresses the report: memory-core now retries startup cron reconciliation and diagnostics can release stale lanes when no active work remains, but the requested active stuck-session timeout/recovery remains warning-only for active embedded runs, active reply work, and active session-lane tasks. The discussion has multiple corroborating reports across channels/platforms, and the related open PRs are implementation candidates rather than merged proof.

## What This Changes

The issue requests configurable timeout and automatic recovery for stuck agent processing lanes, plus retry/readiness handling for memory-core managed dreaming cron registration after Gateway restart.

## Best Possible Solution

Keep the memory-core cron retry work, then ship one canonical core recovery path that adds a documented configurable active stuck-session abort threshold, maps diagnostic state to active embedded/reply/lane work safely, aborts or force-clears only after that threshold, preserves queued-message semantics, and emits a clear terminal user-visible error with regression coverage for hung model/tool/delivery, WebSocket disconnect, and gateway-restart paths.

## Reproduction Assessment

Yes for the remaining code-level issue: current source and tests provide a focused reproduction that diagnostic recovery does not abort or release active embedded/reply/lane work by default. The full live triggers are intermittent, but the provided logs and current recovery path are enough to verify the remaining gap.

## Solution Assessment

No, current main is not the complete solution because it implements only conservative stale-lane cleanup and the memory-core cron retry. The maintainable path is to review and land one canonical active-recovery PR, or an equivalent narrow core patch, with config/docs/schema and regression coverage.
