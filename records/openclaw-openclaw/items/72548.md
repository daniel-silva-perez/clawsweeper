## Summary

Keep open. Current main does not implement plugin-owned SecretRef sources, and this PR is the active implementation attempt, but the proposed runtime path still has blocking activation and third-party plugin resolution defects before it should merge.

## What This Changes

This PR adds a manifest-declared secret-provider contract, a `plugin-sdk/secret-provider` subpath, open SecretRef config and Gateway wire schemas, lazy provider dispatch, bundled GCP and OS keyring secret-provider plugins, and related docs/tests/changelog/labeler updates.

## Best Possible Solution

Revise the PR so secret-provider resolution is activation-aware and works through the same plugin origins that the SDK/docs advertise, while keeping the additive SecretRef schema/protocol shape and leaving broader SecretRef field coverage to a separate tracked follow-up unless maintainers choose to expand scope here.

## Reproduction Assessment

Yes. A static reproduction path is clear from the PR diff: configure a `gcp` SecretRef while `secrets-gcp` is disabled or denied and the resolver still reaches the bundled artifact loader; configure a third-party `aws` source and the schema accepts it but the runtime bundled-only lookup cannot resolve it.

## Solution Assessment

No. The manifest-first seam is the right product direction, but this patch is not yet the best implementation because it bypasses existing plugin activation policy and does not support the installed/workspace plugin sources that the new SDK surface is meant to enable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor plugin activation before loading providers:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The secret-provider resolver loads any bundled manifest owner for a referenced source without checking `plugins.enabled`, `plugins.deny`, `plugins.entries.<id>.enabled`, or the plugin's default activation state. A disabled or denied bundled plugin such as `secrets-gcp` can still execute when a SecretRef references its source, bypassing the activation controls used elsewhere.
  - confidence: 0.9
- **[P2] Resolve installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The runtime lookup only uses the bundled contract registry and bundled public-artifact loader. A third-party plugin can declare `contracts.secretProviders` and the new schema will accept that source, but resolution still returns an unknown-provider error, so AWS/Vault/etc. providers still need core changes despite the advertised SDK seam.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a secret-provider code execution path that does not honor plugin disable/deny activation controls before loading bundled provider artifacts.

Concerns:

- **[medium] Secret providers bypass plugin activation controls:** `src/plugins/secret-provider-resolver.ts:41`
  - body: Secret provider artifacts can import cloud SDKs or spawn platform credential tools, but the new resolver loads them based only on manifest source ownership. Operators who disable all plugins, deny a plugin, or leave a default-off secret plugin disabled would still execute that plugin if a SecretRef references its source.
  - confidence: 0.86
