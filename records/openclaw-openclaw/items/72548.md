## Summary

Keep this PR open. The feature direction matches the secrets/plugin requests, and the Greptile follow-up fixes appear reflected in the head diff, but the current implementation is not merge-ready because secret-provider runtime loading bypasses plugin activation controls and only resolves bundled plugin manifests, which breaks the advertised third-party plugin seam.

## What This Changes

This PR adds a manifest-declared secret-provider contract, a `plugin-sdk/secret-provider` subpath, open SecretRef config and Gateway wire schemas, lazy secret-provider dispatch, bundled GCP and OS keyring secret provider plugins, and related docs/tests/changelog/labeler updates.

## Best Possible Solution

Revise the PR so secret-provider resolution honors effective plugin activation and source ownership for the enabled plugin set, including installed/workspace/config plugins where the SDK docs promise support. Keep the additive schema and protocol shape, but route loading through a generic, activation-aware plugin artifact path and leave the broader SecretRef field-coverage work tracked separately unless maintainers choose to include it here.

## Reproduction Assessment

Yes. A static reproduction path is clear: with `plugins.enabled: false`, `plugins.deny: ["secrets-gcp"]`, or no `plugins.entries.secrets-gcp.enabled`, a `{ source: "gcp" }` provider still reaches the bundled artifact loader because no activation state is passed; a third-party source such as `aws` cannot resolve because the lookup filters to bundled manifests only.

## Solution Assessment

No. A manifest-first secret-provider seam is the right direction, but this patch is not the narrowest maintainable solution until runtime loading reuses plugin activation and supports the same plugin origins that the SDK/docs advertise.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor plugin activation before loading providers:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The secret-provider resolver loads any bundled manifest owner for a referenced source without checking `plugins.enabled`, `plugins.deny`, `plugins.entries.<id>.enabled`, or the plugin's `enabledByDefault` state. That means a disabled or denied bundled plugin such as `secrets-gcp` or `secrets-keyring` can still execute whenever `secrets.providers` references its source, bypassing the plugin activation controls this repo uses elsewhere.
  - confidence: 0.9
- **[P2] Resolve installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The runtime lookup only calls the bundled contract registry and bundled public-artifact loader. A third-party plugin can declare `contracts.secretProviders` and the config schema will accept its source, but resolution will still return `Unknown secret provider source`, so AWS/Vault/etc. providers still require core changes despite the advertised SDK seam.
  - confidence: 0.88
