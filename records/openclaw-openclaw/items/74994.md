## Summary

Keep open. The PR is directionally useful, but the current diff still leaves the original deleted-current-directory `openclaw tui` path able to hit raw `process.cwd()` calls in PATH bootstrap and TUI startup after the new dotenv guard.

## What This Changes

The PR adds a `safeCwd()` helper, routes selected home-dir, dotenv, CLI dotenv, and main-module cwd lookups through it, and adds home-dir regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd or fallback through dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 provides a high-confidence repro by deleting the current directory before running `openclaw tui`, and source inspection shows the same command path still reaches unguarded cwd reads after this PR's guarded dotenv preflight.

## Solution Assessment

No. A shared safe cwd helper is a good building block, but this PR is not the narrow complete fix until it also covers PATH bootstrap and the TUI startup callers that the original repro exercises.
