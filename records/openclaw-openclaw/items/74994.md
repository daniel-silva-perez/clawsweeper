## Summary

Keep open. The updated PR fixes the earlier CLI dotenv preflight/loader gap, but it still does not cover the original `openclaw tui` deleted-current-directory path because CLI PATH bootstrap and TUI startup still read `process.cwd()` directly on current main and are not touched by the patch.

## What This Changes

The PR adds a `safeCwd()` helper and routes home-dir fallbacks, main-module detection, infra dotenv loading, CLI dotenv preflight, and CLI dotenv loading away from raw `process.cwd()` calls when the current directory has been deleted, with home-dir regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd fallback through main-module detection, dotenv loading, CLI PATH bootstrap, and TUI session/workspace initialization while preserving workspace `.env` precedence when cwd exists and skipping project-local PATH probing when cwd is unavailable.

## Reproduction Assessment

Yes. Related issue #73676 gives a deterministic `openclaw tui` deleted-current-directory repro, and current-main source inspection shows remaining raw `process.cwd()` calls on that exact command path.

## Solution Assessment

No. The PR is a good partial fix for early startup and dotenv callers, but it is not the best complete fix until it also guards PATH bootstrap and TUI startup or otherwise handles the original `openclaw tui` command path.
