## Summary

Keep this PR open. It usefully guards the early dotenv and home-dir startup callers, but it is still not merge-ready because the original deleted-current-directory TUI path can continue into raw `process.cwd()` calls in PATH bootstrap and TUI startup.

## What This Changes

The PR adds a `safeCwd()` helper and routes home-dir fallbacks, main-module detection, infra dotenv loading, CLI dotenv probing/loading, and home-dir tests through deleted-cwd-safe behavior.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value or fallback through main-module detection, dotenv loading, CLI PATH bootstrap, and TUI session/workspace initialization, with coverage for the `openclaw tui` repro and preserved cwd-local `.env` precedence when cwd is available.

## Reproduction Assessment

Yes. Related #73676 gives a deterministic `openclaw tui` deleted-current-directory repro, and source inspection on current main shows that command path still reaches raw `process.cwd()` calls after the PR's guarded dotenv path.

## Solution Assessment

No. The PR is a useful partial fix, but the narrow maintainable solution must also guard PATH bootstrap and TUI startup or otherwise handle the original `openclaw tui` command path end to end.
