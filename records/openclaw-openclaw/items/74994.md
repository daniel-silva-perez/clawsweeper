## Summary

Keep open. The PR improves several early deleted-cwd callers, but it does not cover the original `openclaw tui` path end to end: CLI startup can still reach PATH bootstrap and TUI code paths that call `process.cwd()` directly after the proposed dotenv guard.

## What This Changes

The PR adds a `safeCwd()` helper, routes home-dir, main-module, runtime dotenv, and CLI dotenv startup callers through it, and adds home-dir regression tests for deleted-cwd fallback behavior.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value through dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup, while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 gives a concrete `openclaw tui` deleted-current-directory repro, and current-main source inspection shows that path still reaches raw `process.cwd()` calls not covered by this PR.

## Solution Assessment

No. The `safeCwd()` helper direction is useful, but the patch is not the narrow complete fix until it also guards the PATH bootstrap and TUI startup paths and adds regression coverage for the original command.
