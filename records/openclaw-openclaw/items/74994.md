## Summary

Keep open. The force-pushed PR is now narrower and removed the unrelated Telegram behavior, but it still does not cover the CLI dotenv startup path that the linked deleted-CWD bug exercises, so the reported `openclaw tui` crash can remain.

## What This Changes

The PR exports `safeCwd()`, uses it in home-dir fallback, main-module detection, and infra dotenv loading, and adds home-dir regression tests for deleted-current-directory fallback.

## Best Possible Solution

Finish this as a narrow CLI startup fix: keep the useful `safeCwd()` direction, route both CLI dotenv preflight and CLI dotenv loading through it while preserving workspace `.env` precedence when cwd exists, and add focused regression coverage for `isMainModule`, `loadCliDotEnv`, and the `runCli` deleted-CWD startup path.

## Reproduction Assessment

Yes. Related issue #73676 gives deterministic deleted-current-directory steps, and source inspection shows the same `uv_cwd` failure path still exists in the CLI dotenv preflight/loader after this PR.

## Solution Assessment

No. The PR is a good partial fix for home-dir, main-module detection, and infra dotenv loading, but it is not complete for the actual CLI startup path until `src/cli/run-main.ts` and `src/cli/dotenv.ts` are guarded too.
