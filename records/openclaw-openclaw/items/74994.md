## Summary

Keep open. The branch handles some early deleted-cwd callers, but the linked `openclaw tui` repro is still not covered because the same startup path continues into PATH bootstrap and TUI code that still call `process.cwd()` directly.

## What This Changes

The PR adds a `safeCwd()` helper, routes selected home-dir, is-main, runtime dotenv, and CLI dotenv startup callers through it, and adds home-dir deleted-cwd regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value through CLI dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup, while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 provides a deterministic `openclaw tui` deleted-current-directory repro, and current-main source inspection shows that command path still reaches raw `process.cwd()` calls not covered by this PR.

## Solution Assessment

No. The `safeCwd()` helper direction is useful, but the patch is not the narrow complete fix until it also guards PATH bootstrap and TUI startup and adds regression coverage for the original command path.
