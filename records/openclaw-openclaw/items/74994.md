## Summary

Keep open. The PR addresses some deleted-CWD startup crashes, but it still does not cover the original `openclaw tui` repro path because PATH bootstrap and TUI startup retain raw `process.cwd()` calls after the newly guarded dotenv path.

## What This Changes

The PR adds a `safeCwd()` helper, routes home-dir, main-module detection, runtime dotenv loading, and CLI dotenv preflight/loading through it, and adds home-dir regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value through dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup, while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 gives a deterministic path: create and enter a directory, delete it externally, then run `openclaw tui`; current-main source inspection shows the command path still reaches raw `process.cwd()` calls after this PR's guarded dotenv preflight.

## Solution Assessment

No. The proposed helper and early-startup guards are useful, but this is not the best complete fix until the same safe-cwd handling reaches PATH bootstrap and TUI startup with an end-to-end `openclaw tui` regression.
