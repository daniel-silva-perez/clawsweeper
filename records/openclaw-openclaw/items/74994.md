## Summary

Keep open. The PR is aimed at a real deleted-current-directory startup bug and covers several early dotenv/home-dir call sites, but the current diff still leaves the original `openclaw tui` path able to hit raw `process.cwd()` calls through PATH bootstrap and TUI startup, so it is not merge-ready as the complete fix for related #73676.

## What This Changes

The PR adds a `safeCwd()` helper, routes home-dir fallback, main-module detection, runtime dotenv loading, and CLI dotenv preflight/loading through it, and adds home-dir regression tests for deleted-cwd behavior.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value or deliberate fallback through dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup while preserving cwd-local `.env` precedence whenever cwd exists.

## Reproduction Assessment

Yes. Related #73676 gives a deterministic `openclaw tui` deleted-current-directory repro, and current-main source inspection shows the same command can still reach raw `process.cwd()` calls after the PR's guarded dotenv path.

## Solution Assessment

No. The `safeCwd()` helper direction is useful, but this PR is not the best complete fix until it also guards PATH bootstrap and TUI startup or otherwise handles the original `openclaw tui` command path end to end.
