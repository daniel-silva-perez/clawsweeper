## Summary

Keep open. The PR improves several early dotenv/home-dir/main-module deleted-cwd paths, but it still does not cover the original `openclaw tui` path from #73676 because startup continues into PATH bootstrap and then TUI initialization, both of which still read `process.cwd()` directly on current `main`.

## What This Changes

The PR adds an infra `safeCwd()` helper, routes home-dir fallback, main-module detection, runtime dotenv loading, and CLI dotenv preflight/loading through it, and adds home-dir regression tests.

## Best Possible Solution

Land one consolidated deleted-cwd startup fix that carries a safe cwd value through dotenv loading, PATH bootstrap, TUI initial agent/workspace resolution, autocomplete setup, and local auth spawn setup, while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. Related #73676 gives a deterministic `openclaw tui` deleted-current-directory repro, and current-main source inspection shows that same command still reaches raw `process.cwd()` calls after the PR's guarded dotenv path.

## Solution Assessment

No. The proposed direction is useful, but this PR is not the best complete fix until it also guards PATH bootstrap and TUI startup or otherwise handles the original `openclaw tui` command path end to end.
