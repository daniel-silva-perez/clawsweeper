## Summary

Keep open. Current main still lacks the specific active reply-run guard described by the reporter and follow-up commenters: heartbeat skips busy command/session lanes, but it does not consult the active reply-run registry before starting a heartbeat turn. The discussion is also paired with open, unmerged closing PR #64963, so this issue should stay open while maintainers review, rebase, or land that implementation candidate.

## What This Changes

Ensure heartbeat or async system-event turns in Telegram topic sessions do not preempt an in-progress user reply, or otherwise resume/replay the interrupted reply safely.

## Best Possible Solution

Review and land #64963, or apply the same focused change directly on current main: after heartbeat preflight resolves sessionKey, skip with requests-in-flight when resolveActiveReplyRunSessionId(sessionKey) is truthy, then keep the regression test that simulates an active reply operation with an otherwise idle lane. Keep broader async completion routing work tracked separately.
