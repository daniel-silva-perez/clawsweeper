## Summary

Keep open. Current main still lacks the specific active reply-run guard described in the report and follow-up comments: heartbeat skips busy command/session lanes, but does not consult the active reply-run registry before starting the heartbeat turn. The issue is also paired with open, unmerged closing PR #64963, so maintainers should review or land that implementation rather than close this issue early.

## What This Changes

The issue reports that Telegram topic heartbeat and async system-event turns can preempt an in-progress user reply and leave the original answer unreplayed.

## Best Possible Solution

Review and land #64963, or apply the same narrow fix directly on current main: after heartbeat preflight resolves sessionKey, skip with requests-in-flight when resolveActiveReplyRunSessionId(sessionKey) is truthy, emit the normal heartbeat skipped event, and keep a regression test where an active reply operation exists while command and session lane queue sizes are zero. Keep broader async completion routing fixes tracked separately.
