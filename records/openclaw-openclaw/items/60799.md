## Summary

Keep open. Current main and the latest release still reject top-level plain-array cron stores before job extraction, so the reported zero-job load and first-write clobber path remains present; the direct fix PR was closed unmerged and its regression tests are not on main.

## What This Changes

The issue requests backward-compatible cron store loading so legacy top-level JSON array jobs are preserved across upgrade and subsequent cron writes.

## Best Possible Solution

Add a shared cron-store normalization helper that accepts both `{ version: 1, jobs: [...] }` and legacy top-level `[...]` stores, use it from async and sync loaders while preserving current split-state migration behavior, and cover plain-array load plus service-level add/save preservation with focused tests.

## Reproduction Assessment

Yes. A top-level array passes JSON parsing but fails the `!Array.isArray(parsed)` guard in both loaders, producing an empty store; then `cron add` appends to that empty in-memory list and persists it.

## Solution Assessment

Yes. The narrow maintainable fix is loader normalization plus regression coverage; broader read-merge-write protection belongs to the related destructive-write discussion unless it is needed to preserve the legacy array round trip.

## Repair Work Prompt

Fix the legacy plain-array cron store regression reported in this issue. Current `src/cron/store.ts` rejects top-level JSON arrays in both `loadCronStore()` and `loadCronStoreSync()`, causing legacy jobs to load as empty and allowing `cron add` to persist only the newly added job. Add a shared normalization seam that accepts both `{ version: 1, jobs: [...] }` and legacy top-level `[...]` stores, preserving current split `jobs.json`/`jobs-state.json` behavior, inline-state migration, schedule identity handling, and secure atomic writes. Add focused regression coverage for async load of a plain-array store, sync load of a plain-array store, and service-level load-add-save preservation so a legacy job plus a new add round-trips. Check whether doctor coverage should be added or adjusted now that `maybeRepairLegacyCronStore()` depends on `loadCronStore()`. Use PR #61113 as useful source context, but adapt the implementation to the current split-state store code. Do not broaden this into a general read-merge-write redesign or unrelated cron scheduler changes. Add a user-facing changelog fix entry with reporter credit, not bot credit.
