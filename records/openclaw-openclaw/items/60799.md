## Summary

Keep this issue open. Current main still treats a top-level plain-array cron store as an empty object, so `loadCronStore()`/`loadCronStoreSync()` return zero jobs for the reported legacy format. The add path then appends to that empty in-memory store and `saveCronStore()` rewrites `jobs.json` from memory, preserving the reported clobber risk. The referenced fix PR #61113 was closed unmerged, and its proposed `normalizeCronStore` helper/regression tests are not present on main.

## Best Possible Solution

Implement a small cron-store normalization seam in `src/cron/store.ts` that accepts both `{ version: 1, jobs: [...] }` and legacy top-level `[...]` stores, use it from both async and sync load paths, and add focused regression tests that verify plain-array load plus load-add-save preservation. Consider adding a doctor repair for top-level array stores, but the runtime loader must be backward-compatible before this issue closes.

## Risks / Open Questions

- Leaving this unresolved preserves a direct data-loss path for users with legacy top-level array cron stores: list/status shows zero jobs, and the next mutation can overwrite the store from the empty in-memory state.
- A narrow fix should cover both async and sync loaders and the load-add-save round trip; otherwise one caller path may remain able to drop legacy jobs.
