## Summary

Keep open. Current main still prefers the inherited/default account before consulting the calling agent's channel/account binding, and this focused PR directly targets the open linked bug with no concrete functional or security blocker found in the provided diff.

## What This Changes

The PR updates the message tool and outbound message-action runner so omitted accountId resolves as explicit param first, calling-agent binding second, inherited/default account last, with regression coverage for the combined default-plus-binding case.

## Best Possible Solution

Land this PR or an equivalent narrow change after CI confirms the targeted regression and typecheck lanes. The desired end state is that explicit message-tool accountId remains authoritative, omitted accountId prefers the calling agent's configured channel/account binding, and inherited/default account is only the final fallback.

## Reproduction Assessment

Yes. Current main provides a high-confidence static reproduction: pass both defaultAccountId and an agent/channel binding to runMessageAction with params.accountId omitted, and the existing line 936 priority selects the default before the binding; the PR's new table case captures that scenario.

## Solution Assessment

Yes. The proposed patch is the narrowest maintainable fix I found: it preserves explicit params, stops the wrapper from manufacturing an explicit accountId from inherited context, moves default fallback after binding lookup, and adds focused regression coverage.
