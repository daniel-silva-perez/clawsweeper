## Summary

Current main still does not implement this PR’s intended Feishu streaming-card late-final suppression. Main only suppresses exact duplicate final text via `deliveredFinalTexts`, while distinct late `final` text after a streaming card closes can still open another Feishu card. This PR remains an active implementation candidate for the linked Feishu bug, but review/security comments raise real lifecycle and content-integrity concerns that need maintainer resolution before merge.

## Best Possible Solution

Keep this PR open as the active implementation candidate for the linked Feishu duplicate-card bug. The best path is maintainer review of the focused Feishu dispatcher/test change, resolving the text-plus-media content-integrity question and scoping any closed-stream suppression to the active reply lifecycle before merge.

## Risks / Open Questions

- The PR intentionally suppresses distinct late final text while still allowing media delivery, so captions, warnings, attribution, or other context can be separated from attachments unless the streamed card is guaranteed to contain the complete safe text.
- The proposed `streamingClosedForReply` / `streamingCloseErroredForReply` booleans are mutable dispatcher state written after awaited Feishu closeout work, while shared reply lifecycle hooks are fire-and-forget; slow closeout, error recovery, or back-to-back replies can still create dropped-final races without a reply-scoped token or awaited lifecycle contract.
- The PR currently leaves `renderMode: "auto"` late plain-text finals outside the suppression guard, which may be acceptable for the narrow card-duplication bug but should be an explicit maintainer product decision.
