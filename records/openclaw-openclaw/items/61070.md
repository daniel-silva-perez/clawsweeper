## Summary

Keep open. Current main partially covers compaction tuning through token-budget knobs, but the requested subagent operating-principle and max-failure-attempt configuration is still absent. The paired PR #61021 was closed unmerged, and I found no renamed implementation in current source, docs, tests, or history.

## Best Possible Solution

Keep the issue open and narrow the design. The compaction part may be satisfied by documenting or refining existing `reserveTokens`/`keepRecentTokens` controls, but the subagent prompt-policy part still needs maintainer product judgment and an implementation in config types, zod schema, labels/help/generated schema, `buildSubagentSystemPrompt`, spawn threading, docs, and focused tests.

## Risks / Open Questions

- Adding switches that remove subagent prompt rules could weaken orchestration and safety behavior unless defaults stay conservative and validation is explicit.
- A ratio/message-count compaction API would overlap current reserve-token and recent-token semantics, so maintainers should avoid creating two competing ways to tune the same runtime threshold.
