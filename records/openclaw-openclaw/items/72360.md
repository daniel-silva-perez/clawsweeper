## Summary

Keep this issue open. Current main still lets store-targeted `/allowlist add|remove` skip the `resolveConfigWriteDeniedText` policy check before mutating the pairing allowlist store, and open PR #72361 is already the linked unmerged fix candidate.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store without honoring channel or account `configWrites=false`.

## Best Possible Solution

Review and land PR #72361, or an equivalent narrow patch, so the existing config-write authorization policy runs before every pairing-store allowlist mutation and focused tests cover both store add and store remove with `configWrites=false`.

## Reproduction Assessment

Yes. A high-confidence source-path reproduction exists on current main: an owner-authorized sender on a pairing-capable channel can send `/allowlist add dm --store <id>`, which skips the config-write denial branch and reaches the pairing-store update call.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText` or the same underlying `authorizeConfigWrite` policy before store mutation is the narrowest maintainable fix unless maintainers intentionally define and document a separate permission for pairing-store edits.
