## Summary

Keep this issue open. Current main still lets `/allowlist add|remove --store` skip the existing `configWrites` authorization path before mutating the pairing allowlist store, and open PR #72361 already references this issue with closing syntax, so the issue should stay open until that PR or an equivalent focused fix lands.

## What This Changes

Require store-targeted `/allowlist add|remove --store` mutations to honor channel/account `configWrites=false` before updating the persistent pairing allowlist store.

## Best Possible Solution

Apply the existing `resolveConfigWriteDeniedText`/`authorizeConfigWrite` policy to store-targeted allowlist mutations before `updatePairingStoreAllowlist(...)`, using the same origin channel/account and target channel/account semantics as config edits, and add focused regression tests for `configWrites=false` store add/remove cases.

## Reproduction Assessment

Yes. The current-main code path is enough to reproduce with an owner-authorized sender on a pairing-capable channel: `/allowlist add dm --store <id>` sets `shouldUpdateConfig=false`, skips the config-write denial check, and reaches the store update call.

## Solution Assessment

Yes, the narrow maintainable solution is to reuse the existing config-write authorization before any persistent allowlist store mutation. A separate permission or docs exception should only be considered if maintainers intentionally want pairing-store edits to be a distinct policy surface.
