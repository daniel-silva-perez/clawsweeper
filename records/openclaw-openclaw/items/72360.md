## Summary

Keep this issue open. Current main still routes `/allowlist --store` around the `configWrites` check: `parsed.target === "store"` skips the branch that calls `resolveConfigWriteDeniedText(...)`, then the store-only branch calls `updatePairingStoreAllowlist(...)`. The linked open PR #72361 uses closing syntax for this issue, so the right path is to review and land or close that PR rather than close the issue early.

## Best Possible Solution

Keep this issue open until #72361 or an equivalent focused fix is merged. The fix should apply the same `resolveConfigWriteDeniedText`/`authorizeConfigWrite` policy to store-only `/allowlist add|remove --store` mutations, using the target channel/account scope, and add regression coverage for `configWrites=false` on the origin/target account before `updatePairingStoreAllowlist(...)` can run.
