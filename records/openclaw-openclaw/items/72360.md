## Summary

Keep this issue open. Current main still lets `/allowlist add|remove --store` skip the `configWrites` authorization branch before mutating the pairing allowlist store, and open PR #72361 is already the focused fix candidate linked to this issue.

## What This Changes

Prevent store-targeted `/allowlist add|remove` from mutating the persistent pairing allowlist store when the channel or account `configWrites` policy denies writes.

## Best Possible Solution

Land PR #72361 or an equivalent narrow patch that enforces the existing config-write policy before every pairing-store allowlist mutation, with focused regression coverage for `configWrites=false` store add and remove paths.

## Reproduction Assessment

Yes. A focused source-path reproduction exists on current main: with an owner-authorized sender on a pairing-capable channel, `/allowlist add dm --store <id>` skips the config-write denial branch and reaches the pairing-store update call.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText` or the same underlying `authorizeConfigWrite` policy before store mutation is the narrowest maintainable fix unless maintainers intentionally define and document a separate permission for pairing-store edits.
