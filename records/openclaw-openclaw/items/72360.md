## Summary

Keep this issue open. Current main still has the reported source-path bypass, and the linked closing PR is open and unmerged, so the issue should remain open until that PR or an equivalent access-control fix lands.

## What This Changes

The issue asks OpenClaw to enforce channel/account `configWrites` policy before `/allowlist add|remove --store` mutates the persistent pairing allowlist store.

## Best Possible Solution

Review and land PR #72361, or an equivalent narrow patch, so the existing config-write authorization runs before every pairing-store allowlist mutation and regression tests cover both store add and store remove when `configWrites=false`.

## Reproduction Assessment

Yes. The current source path is enough to reproduce with high confidence: an owner-authorized `/allowlist add dm --store <id>` on a pairing-enabled channel sets `shouldUpdateConfig` false, skips the only `resolveConfigWriteDeniedText(...)` call, and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText` or the same underlying `authorizeConfigWrite` policy before store mutation is the narrowest maintainable fix unless maintainers intentionally introduce and document a separate permission for pairing-store edits.
