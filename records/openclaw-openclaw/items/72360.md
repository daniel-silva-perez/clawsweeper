## Summary

Keep this issue open. Current main still has the reported `/allowlist --store` policy bypass, and the linked closing PR is open and unmerged, so the issue should remain open for maintainer/security review rather than be closed by cleanup.

## What This Changes

The issue asks OpenClaw to enforce channel/account `configWrites` policy before `/allowlist add|remove --store` mutates the persistent pairing allowlist store.

## Best Possible Solution

Review and land PR #72361, or an equivalent narrow patch, so the existing config-write authorization policy runs before every pairing-store allowlist mutation and regression tests cover the store-only add/remove path with `configWrites=false`.

## Reproduction Assessment

Yes. The source-path reproduction is high confidence: `/allowlist add dm --store <id>` parses as target `store`, skips the config-write branch, and reaches the pairing-store mutation call on current main.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText` or the same `authorizeConfigWrite` policy before store mutation is the narrowest maintainable fix unless maintainers intentionally add and document a separate permission for pairing-store edits.
