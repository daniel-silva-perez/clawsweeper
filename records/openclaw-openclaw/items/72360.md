## Summary

Keep this issue open. Current main still has the reported `/allowlist --store` bypass: the store-only target skips the only config-write authorization call and then mutates the pairing allowlist store, while the docs still promise that `/allowlist add|remove` honors channel `configWrites`. PR #72361 is the linked focused fix, but it is open and unmerged, so this issue should stay open for maintainer/security review.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store even when the channel’s `configWrites` policy disables persistent config writes.

## Best Possible Solution

Review and land PR #72361 or an equivalent narrow patch that runs the same config-write authorization before every pairing-store allowlist mutation, with regression coverage for both store add and store remove when `configWrites=false`.

## Reproduction Assessment

Yes. The current source path is enough for high-confidence reproduction: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches the pairing-store mutation call.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText` or the underlying `authorizeConfigWrite` policy before store mutation is the narrowest maintainable fix unless maintainers intentionally add and document a separate permission for pairing-store edits.
