## Summary

Keep this issue open. Current main still allows store-targeted `/allowlist add|remove` to skip the existing `configWrites` policy check before mutating the pairing allowlist store, and open PR #72361 is already the focused implementation candidate for this security-sensitive fix.

## What This Changes

Store-targeted `/allowlist add|remove` should honor channel/account `configWrites` before mutating the persistent pairing allowlist store.

## Best Possible Solution

A narrow merged fix should enforce the existing config-write policy before any pairing-store allowlist mutation, add focused regression coverage for `--store` add/remove under channel/account `configWrites=false`, and keep the documented `/allowlist` permission contract intact unless maintainers intentionally introduce a separate store-write permission.

## Reproduction Assessment

Yes. Current-main code inspection gives a high-confidence reproduction path: with an owner-authorized sender on a pairing-capable channel and `configWrites=false`, `/allowlist add dm --store <id>` skips the config-write denial branch and reaches the store update call.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText`/`authorizeConfigWrite` before store mutations is the narrowest maintainable fix; the open PR #72361 or an equivalent patch should be reviewed for that exact behavior.
