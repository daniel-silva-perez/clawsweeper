## Summary

Keep open. Current main has provider/tool image sanitization, but direct chat/RPC attachments still reject oversized images before any resize path runs, and the shared iOS/macOS chat composer still rejects attachment data over 5 MB. Related #68524 and #67031 are useful adjacent trackers, but they do not fully supersede this general/server-side request.

## Best Possible Solution

Implement inbound image resize/compression before the parser's image byte rejection in `src/gateway/chat-attachments.ts`, reusing existing image operations/sanitization primitives where the dependency boundary fits. Replace or augment the current oversized-image rejection test with coverage that proves large screenshots are reduced and forwarded. Keep #68524 for iOS client-side preflight/memory work and #67031 for configurable media limits.

## Risks / Open Questions

- A server-side fix must avoid turning oversized base64 into unbounded CPU or memory work; it should keep an input byte/pixel ceiling before invoking image decoding/resizing.
- Changing the shared parser affects `agent.run`, `chat.send`, and node-event dispatch, so the regression coverage should exercise the shared helper and at least one gateway surface.
