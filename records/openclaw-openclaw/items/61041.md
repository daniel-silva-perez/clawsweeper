## Summary

Keep open. Current main still rejects oversized direct chat/RPC image attachments before any resize path can run. Existing provider/tool image sanitization is real but operates later on content blocks, not on the inbound attachment parser path described here. The adjacent iOS PR #73710 is open and explicitly treats server-side resize for this issue as out of scope.

## Best Possible Solution

Implement inbound image resize/compression inside `parseMessageWithAttachments` before the image byte rejection, reusing existing image operations or sanitizer primitives where the dependency boundary fits and preserving strict predecode limits. Replace the current oversized-image rejection test with coverage that proves large screenshots are reduced and forwarded. Keep #73710 focused on iOS client-side memory/preflight work and #67031 focused on configurable media limits.

## Risks / Open Questions

- A server-side resize implementation should keep hard input byte and pixel ceilings before decode to avoid turning oversized base64 into unbounded CPU or memory work.
- Changing the shared gateway parser affects `agent.run`, `chat.send`, and node-event dispatch, so coverage should exercise the shared parser and at least one gateway surface.
- #73710 may reduce failures from the iOS composer if merged, but it is complementary and does not address other clients or server-origin attachment paths.
