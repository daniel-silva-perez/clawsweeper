## Summary

Keep open. Current main still rejects oversized direct chat/RPC image attachments before any resize path can run; existing image sanitization only covers later provider/tool content blocks, and the open iOS PR is useful but only addresses the client-side half tracked by #68524.

## What This Changes

Request auto-resizing oversized image attachments instead of rejecting them at the gateway or chat-composer size limit.

## Best Possible Solution

Add a server-side image shrink step in the shared chat attachment parser before the MAX_IMAGE_BYTES rejection, using the existing media image-ops/sanitization primitives and preserving explicit failure for images that still cannot fit safely. Keep #73710 focused on iOS preflight/privacy work and #67031 focused on configurability.

## Reproduction Assessment

Yes. A valid image attachment whose estimated decoded size is above MAX_IMAGE_BYTES reaches the current parseMessageWithAttachments rejection path, and the existing test suite locks in that rejection behavior.

## Solution Assessment

Yes for a narrow repair direction. The maintainable fix is to resize/compress in the shared gateway parser before the hard image-byte rejection; client-side resizing is useful defense in depth, but it is not sufficient for this general server-side request.
