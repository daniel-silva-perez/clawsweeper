## Summary

Keep open. Current main still has the same material split the reporter described: default status paths trust the WebSocket probe result, while `gateway status --require-rpc` uses a separate `status` RPC path that can succeed even when the default probe reports timeout/unknown capability. The current shipped-looking fix only covers scope-limited `missing scope: operator.read` classification, not generic local timeout or `Capability: unknown` false negatives.

## Best Possible Solution

Keep the issue open. Reproduce the default local probe versus `--require-rpc` split, then add a regression test that simulates `probeGateway` returning timeout/unknown while a direct `status` RPC succeeds. The likely code direction is to separate handshake reachability from read/detail probe failure, or add a bounded status-RPC fallback for local status paths. If maintainers choose #48360, #64400, or #73524 as the canonical thread, move this issue's regression-window and control evidence there before closing as duplicate/superseded.
