## Summary

Keep open. Current main still lets a wake-provided non-subagent session key win before an explicit `heartbeat.session`, so the confirmed exec/system-event heartbeat pollution path remains present; Telegram private DMs also still default to the agent main session by documented and tested policy. The open PR #56092 uses closing syntax for this issue, so maintainers should review, land, or supersede that implementation instead of closing the report early.

## What This Changes

The issue requests stricter isolation between Telegram DMs, heartbeat runs, and system-event-triggered heartbeat wakes so normal user traffic and heartbeat/system traffic do not share or pollute `agent:main:main` or active DM sessions.

## Best Possible Solution

End state should make configured heartbeat sessions win for system-event-triggered wakes, preserve or migrate pending system events without cross-session pollution, and separately settle Telegram DM default behavior by either documenting intentional main-session sharing or changing the Telegram plugin to isolate DMs with matching tests and docs.

## Reproduction Assessment

Yes for the confirmed exec-completion subcase: `notifyOnExit` enqueues a system event on the originating session, the wake path forwards that session key, and current heartbeat resolution accepts it before `heartbeat.session`. The original Telegram-only pollution path remains unclear without a fresh minimal live repro.

## Solution Assessment

No for current main. #56092 appears to target the confirmed root cause, but maintainers still need to review whether its event migration and skip-restore behavior are the narrowest durable fix and whether Telegram DM default isolation should be handled separately.
