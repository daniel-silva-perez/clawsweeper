## Summary

Keep open. Current main still lets system-event heartbeat wakes carry an originating session key that is resolved before an explicit heartbeat session, and Telegram private DMs still default to `agent:main:main` under the documented/tested default. The open implementation PR #56092 uses closing syntax for this report, so the issue should wait for that PR to be reviewed, merged, or superseded.

## What This Changes

Request to prevent Telegram DMs and heartbeat/system-event runs from sharing or polluting `agent:main:main` when a dedicated Telegram or heartbeat session should be used.

## Best Possible Solution

Review, land, or supersede #56092 for the system-event heartbeat path, then explicitly settle the Telegram default DM policy with aligned Telegram tests and channel-routing docs so heartbeat/main context and direct-message context cannot accidentally mix.

## Reproduction Assessment

Yes for the confirmed exec-completion subcase: a background exec with `notifyOnExit` in a Telegram DM session enqueues a system event for that session, wakes heartbeat with that session key, and current `resolveHeartbeatSession` accepts it before `heartbeat.session`. The original long-running Telegram-only pollution path still lacks a minimal live repro.

## Solution Assessment

Unclear until #56092 is reviewed; its direction matches the confirmed system-event root cause, but current main is not the best end state because Telegram DM default routing still intentionally maps private DMs to `agent:main:main` under default config.
