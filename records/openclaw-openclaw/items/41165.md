## Summary

Keep open. Current main still has the confirmed system-event heartbeat routing problem, and the Telegram default DM behavior still maps private DMs to the agent main session by documented/tested policy. The linked open PR #56092 uses closing syntax for this issue, so the issue should wait for that PR to be reviewed, merged, or superseded rather than being closed early.

## What This Changes

Ensure Telegram direct messages and heartbeat/system-event turns stay session-isolated so direct-chat traffic cannot pollute `agent:main:main` or configured heartbeat sessions.

## Best Possible Solution

Review, land, or supersede #56092 for system-event-triggered heartbeat routing, then explicitly settle the Telegram DM default policy with aligned Telegram tests and channel-routing docs so heartbeat/main context and direct-message context cannot accidentally mix.

## Reproduction Assessment

Yes for the confirmed exec-completion subcase: a background exec with `notifyOnExit` enqueues a system event for the originating session, the wake scheduler forwards that session key, and current `resolveHeartbeatSession` accepts it before `heartbeat.session`. The original Telegram-only pollution report remains unclear without a fresh minimal live repro.

## Solution Assessment

Unclear until #56092 is reviewed. Its direction matches the confirmed root cause, but the durable solution should make configured heartbeat sessions win for wake-triggered runs, preserve or migrate pending system events safely, and separately decide whether Telegram DMs should continue sharing the main session by default.
