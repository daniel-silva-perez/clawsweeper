## Summary

Keep open. Current main still resolves a wake-provided session key before `heartbeat.session`, so exec/system-event-triggered heartbeats can run in the originating session, and the documented/tested Telegram DM default still maps private DMs to `agent:main:main`. The related open PR #56092 uses closing syntax for this report, so this issue should wait for that PR to be reviewed, merged, or superseded.

## What This Changes

Request to keep Telegram DMs and heartbeat/system-event traffic isolated so direct-message conversations cannot pollute `agent:main:main` or a heartbeat session.

## Best Possible Solution

Review, land, or supersede #56092 for the system-event heartbeat path, then explicitly settle the Telegram default DM policy with aligned Telegram tests and channel-routing docs so heartbeat/main context and direct-message context cannot accidentally mix.

## Reproduction Assessment

Yes for the confirmed exec-completion subcase: a background exec with `notifyOnExit` in a Telegram DM session enqueues a system event for that session, wakes heartbeat with that session key, and current `resolveHeartbeatSession` accepts it before `heartbeat.session`. The original Telegram-only pollution path remains unclear because it lacks a minimal live repro.

## Solution Assessment

Unclear until #56092 is reviewed; its direction matches the confirmed system-event root cause, but current main is not the best end state. A maintainable fix should make configured heartbeat sessions win for wake-triggered runs, safely migrate prompt-visible events, and separately decide whether Telegram DMs should continue sharing the main session by default.
