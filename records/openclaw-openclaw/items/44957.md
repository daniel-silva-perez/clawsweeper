## Summary

Keep open. Current main still has no `outputHistory` or `recentOutputs` cron behavior, so the PR is not obsolete; however, the proposed patch still has concrete correctness issues around storing silent/control outputs and malformed boolean normalization before it is safe to merge.

## What This Changes

The PR adds a per-job cron `outputHistory` flag, persists bounded `recentOutputs` in cron state, injects recent outputs into isolated agent prompts, updates cron normalization/tool recovery and Gateway schemas, and adds cron service/protocol/prompt tests.

## Best Possible Solution

Land a refined version of the PR that documents `payload.outputHistory`, stores only meaningful bounded agent outputs, filters silent/control tokens before persistence, preserves external-content boundaries, normalizes malformed tool booleans consistently with existing cron recovery, and updates cron service, isolated-agent prompt, tool, protocol, docs, and changelog coverage as appropriate.

## Reproduction Assessment

Yes. A focused unit path can reproduce the issues by enabling `payload.outputHistory`, returning a successful isolated run whose final text is `NO_REPLY` or `HEARTBEAT_OK`, and by normalizing an agent-turn payload with `outputHistory: "true"`; code inspection shows both would currently reach the problematic PR paths.

## Solution Assessment

No, not as currently proposed. The feature direction is reasonable, and several earlier review concerns were addressed, but the patch should filter non-substantive control replies and normalize or drop non-boolean `outputHistory` before strict schema validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Filter silent replies before recording history:** `src/cron/service/timer.ts:347-355`
  - body: `applyJobResult` now appends any successful `outputText`, but isolated delivery can return `status: "ok"` with `outputText` still set to `NO_REPLY`, `HEARTBEAT_OK`, or stale parent text after delivery suppression. Those control replies would be injected into the next run as prior output, replacing useful history with internal sentinel text.
  - confidence: 0.86
- **[P2] Normalize malformed outputHistory booleans:** `src/cron/normalize.ts:101-102`
  - body: The new normalization path only treats real booleans as `outputHistory` and leaves non-boolean values in the payload. Model/tool recovery often sees stringified booleans, so `{ kind: "agentTurn", message, outputHistory: "true" }` will survive normalization and fail the strict Gateway schema instead of being coerced or stripped like other recovered cron fields.
  - confidence: 0.78
