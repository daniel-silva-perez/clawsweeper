## Summary

Keep this PR open: current main still accepts the invalid `hooks.enabled: true` without `hooks.token` through config validation and only fails later in `resolveHooksConfig`, so the PR addresses a real bug, but its new remediation text points users at unsupported `openclaw config.patch` syntax.

## What This Changes

The PR adds `hooks.enabled`/`hooks.token` cross-field validation in config validation and expands the gateway hooks startup error with remediation guidance.

## Best Possible Solution

Land a corrected version that rejects `hooks.enabled: true` with a missing or blank `hooks.token` during merged config write validation, keeps the runtime guard as a safety net, uses supported CLI/RPC remediation syntax, and adds focused config and gateway tests.

## Reproduction Assessment

Yes. The source path is clear: current main keeps `hooks.token` optional in schema/config validation, `config.patch` validates the merged config through that path, and `resolveHooksConfig` throws only at startup when hooks are enabled without a token.

## Solution Assessment

No, not as written. The validation seam is the right narrow place to fix the bug, but the user-facing remediation command must be changed to supported `openclaw config patch` CLI syntax or the documented gateway RPC form before merge.
