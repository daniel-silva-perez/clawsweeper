## Summary

Keep this PR open. Current main still does not validate `hooks.enabled=true` plus missing `hooks.token` at `config.patch` validation time, and the runtime error remains the old generic message. The PR addresses a real remaining gap, but it should be revised before merge because its suggested recovery command is not a valid CLI command and the new cross-field validation lacks regression coverage.

## Best Possible Solution

Keep the PR open and ask for a revision: add the `hooks.enabled`/`hooks.token` cross-field validation on the merged config path, cover it with focused config validation or config.patch tests, and replace the remediation text with a valid OpenClaw command or docs-aligned guidance. After that, the PR can either land or be superseded by an equivalent maintainer patch on main.

## Risks / Open Questions

- The PR's new error text tells users to run an invalid `openclaw config.patch` command, which would fail during recovery.
- The suggested command places a hook bearer token directly in a shell command; the final wording should be careful about secret handling and should prefer the repo's existing documented config-edit paths.
- No regression test currently proves `config.patch` rejects `hooks.enabled=true` without a token while accepting the same patch when the merged config already has a token.
