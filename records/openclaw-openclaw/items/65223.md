## Summary

Keep open. Current main partially handles one concrete example by making `process.poll` abort-aware, and it also races the outer embedded prompt against run abort, but it still does not define or implement the broader tool-execution interruption contract requested here: no per-tool framework race, no transcript/partial-result policy, and no caller `AbortSignal` path for `sessions_send` gateway waits.

## What This Changes

The issue requests a design and implementation path for AbortSignal-aware tool execution, framework-level interruption, steering semantics, partial-result handling, and cancellable gateway-backed tools.

## Best Possible Solution

Define a core agent-runtime interruption contract first, then implement it consistently: framework fallback behavior for stuck tools, opt-in per-tool partial results, transcript handling for aborted tool calls, and cancellable gateway plumbing for tools such as `sessions_send` without killing resumable background processes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? This is primarily a design issue, but code inspection gives high-confidence reproduction of the remaining gaps: `sessions_send` cannot receive a signal, gateway requests are timeout-only, and the OpenClaw wrapper forwards signals without a per-tool execution race.

## Solution Assessment

Is this the best way to solve the issue? No current implementation fully solves it; the `process.poll` fix and prompt-level abort race are useful partial measures, but the maintainable solution is an explicit runtime contract before more piecemeal tool behavior changes.
