## Summary

Keep open. Current main still renders human `openclaw logs` timestamps as UTC by default unless `--local-time` is passed or the CLI process has a valid `TZ`, and the issue is already paired with open unmerged PRs that reference it with closing syntax.

## What This Changes

The issue asks for `openclaw logs` to show local-time timestamps by default instead of UTC for users outside UTC.

## Best Possible Solution

Review and consolidate #46782 and #47365, then land one narrow fix that makes human-readable `openclaw logs` output local time by default while preserving JSON/tooling output, the existing `--local-time` compatibility surface, and an explicit UTC opt-out.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: run `openclaw logs` without `--local-time` and without a valid `TZ`; the CLI leaves local-time mode off and the formatter emits UTC via `toISOString()` or the UTC formatter path.

## Solution Assessment

No for closing this now, and no for only switching raw field selection. Current main already prefers the top-level `time` field; the maintainable fix is to change human-output default/flag behavior while preserving explicit UTC and JSON behavior.
