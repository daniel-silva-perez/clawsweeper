## Summary

Keep this issue open. Current main still renders human `openclaw logs` timestamps as UTC by default unless `--local-time` is passed or the CLI process has a valid `TZ`, and the linked implementation PRs that use closing syntax remain open/unmerged rather than shipped.

## What This Changes

The issue requests that `openclaw logs` display local-time timestamps by default instead of UTC timestamps for human log output.

## Best Possible Solution

Review and consolidate #46782 and #47365, then land one narrow fix that makes human `openclaw logs` output default to local time while preserving explicit UTC output, JSON/tooling behavior, and updated tests/docs. Treat Docker console timestamp behavior as a separate scope decision unless the same formatter path clearly owns it.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: default `openclaw logs` leaves `localTime` false, `formatLogTimestamp` returns `toISOString()`, and current tests assert UTC output by default.

## Solution Assessment

Yes, the narrowest maintainable direction is to change the human logs default to local time with an explicit UTC opt-out. Merely preferring the raw `time` field is insufficient because current main already parses that field and then normalizes it to UTC when local-time mode is off.
