## Summary

Keep open. Current main still pins `@mariozechner/pi-tui` to `0.70.5`, OpenClaw still delegates TUI editor input to pi-tui with only a local backspace deduper, and the exact pinned pi-tui decoder still rejects printable Kitty/CSI-u events carrying Alt or Ctrl modifier bits. That leaves the reported German AltGr `@`/`€` path unresolved, even if some duplicate backspace/input work has landed around the same TUI area.

## Best Possible Solution

Keep this open and fix the TUI input layer, ideally upstream in pi-tui or via a narrow OpenClaw adapter while waiting for upstream. The implementation should decode AltGr-produced printable CSI-u sequences as text without breaking real Alt/Ctrl shortcuts, then add focused regression coverage for German-layout `@`/`€` and duplicated printable Kitty events before bumping or patching the dependency.

## Risks / Open Questions

- Terminals that emit AltGr characters as plain Unicode rather than Kitty/CSI-u with modifier bits may not reproduce the AltGr failure, so regression coverage should target the reported Kitty/CSI-u path directly.
- A fix needs to distinguish AltGr-generated printable text from intentional Alt/Ctrl shortcuts such as Alt+Enter, Alt+Up, Ctrl+C, Ctrl+D, and editor navigation bindings.
- The duplicated printable symptom and the AltGr failure may have separate causes; they should be reproduced and tested separately instead of assuming one decoder change fixes both.
