## Summary

Keep open. Current main partially moved forward by pinning `@mariozechner/pi-tui` to `0.70.6`, whose stdin buffer suppresses duplicate unmodified Kitty printable/raw pairs, but the exact printable CSI-u decoder still rejects events carrying Alt or Ctrl modifier bits. That leaves the reported German AltGr `@` and `€` path unresolved in the TUI editor input flow.

## What This Changes

Fix TUI/gateway-client input so German keyboard layout printable characters are inserted once and AltGr combinations such as `@` and `€` enter as normal printable text.

## Best Possible Solution

Keep this open and fix the TUI input layer at the narrowest boundary: preferably by consuming an upstream pi-tui release that decodes AltGr-produced printable CSI-u text correctly, or by adding a small OpenClaw adapter while preserving existing shortcut handling. The fix should include focused regression coverage for German-layout `@`/`€` and for non-backspace duplicate printable events before the issue is closed.

## Reproduction Assessment

Yes for the remaining AltGr/Kitty path: a synthetic printable CSI-u event carrying Alt/Ctrl modifier bits, such as Ctrl+Alt/AltGr `@` or `€`, is rejected by `decodeKittyPrintable()` in `@mariozechner/pi-tui@0.70.6` and then is not inserted by the editor. A live German keyboard run was not performed, so the duplicated-character half should still be tested separately.

## Solution Assessment

No current complete solution is on main. The `0.70.6` dependency bump looks like a partial fix for duplicate unmodified printable events, but the best maintainable fix still needs AltGr-aware printable decoding without broadening real Alt/Ctrl shortcut handling.
