## Summary

Keep open. Current main still lacks the requested TUI image-attachment flow, while the proposed branch is directionally useful but not merge-ready because it rewrites normal prompt text, can leak raw local image paths on attachment-only sends, and needs a rebase/design pass for the current TUI backend split.

## What This Changes

The PR adds a TUI image extraction helper and tests, a `/image` slash command/help entry, and attachment forwarding from TUI sends to Gateway `chat.send`.

## Best Possible Solution

Land a maintainer-reviewed TUI implementation that preserves prompt text unless image tokens are actually removed, hides raw local paths from transcript/model input, threads attachments through the shared `TuiBackend` and `GatewayChatClient`, deliberately supports or rejects local embedded mode, reuses the current Gateway media limits/validation expectations, updates the TUI docs, and adds focused command-handler/parser tests.

## Reproduction Assessment

Yes. The feature gap is reproducible by static inspection of current main: there is no `/image` command, no TUI image extraction module, and no attachment field in the TUI send path; the PR bugs are reproducible by following `extractImagePaths` and the `cleanText || text` send fallback on the proposed branch.

## Solution Assessment

No. The requested direction fits the existing Gateway attachment contract, but this PR is not the best mergeable form because it rewrites prompts, leaks raw `@path` text for image-only messages, and does not account for the current shared TUI backend/local-mode boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses repeated whitespace and trims the result, and `sendMessage` uses that result for every non-BTW message. A normal prompt with indentation, leading whitespace, or intentional double spaces is silently changed before it reaches the model even when no image reference was extracted.
  - confidence: 0.95
- **[P2] Avoid sending raw @path text for image-only prompts:** `src/tui/tui-command-handlers.ts:565`
  - body: For a message such as `@/tmp/screenshot.png`, `cleanText` is empty, so `message = cleanText || text` falls back to the original local path token. That displays and sends the raw path instead of sending only the attachment payload or a neutral placeholder.
  - confidence: 0.93
- **[P2] Handle local TUI mode before advertising /image:** `src/tui/tui-command-handlers.ts:513`
  - body: On current main the command handler uses the shared `TuiBackend`, but the embedded backend only forwards text fields to the local runner. After rebase, `/image` would appear to attach a file in `openclaw chat` while the local backend receives only the placeholder text unless local attachments are implemented or explicitly rejected.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds local-file upload behavior and currently has concrete path-leak and size-check concerns that should be resolved before merge.

Concerns:

- **[medium] Attachment-only prompts leak local file paths:** `src/tui/tui-command-handlers.ts:565`
  - body: The proposed `cleanText || text` fallback sends raw `@/local/path` text when an image reference is the whole prompt, exposing local path details to the Gateway/model transcript instead of stripping the reference.
  - confidence: 0.93
- **[low] File size guard races the read:** `src/tui/tui-image-extract.ts:88`
  - body: The proposed loader checks `fs.stat` before `fs.readFile`, so a file can be replaced or enlarged between the check and read; reading first and checking `buffer.byteLength` would avoid oversized local reads.
  - confidence: 0.78
