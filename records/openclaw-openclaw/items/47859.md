## Summary

Keep open. Current main still lacks TUI image attachment support, so the contribution addresses a real gap, but the proposed patch is not merge-ready because it rewrites prompt text, can leak raw local paths for image-only prompts, and does not account for the current shared TUI backend/local-mode boundary.

## What This Changes

The PR adds TUI `@path` image extraction, a `/image` slash command, TUI-to-Gateway attachment forwarding, and parser/loader tests.

## Best Possible Solution

Land a maintainer-reviewed TUI implementation that preserves prompt text unless image tokens are actually removed, never sends raw local paths to the transcript or model, threads attachments through the shared `TuiBackend` and `GatewayChatClient` contracts, deliberately supports or rejects embedded local mode, reuses current Gateway media limits/validation, updates TUI docs, and adds command-handler plus parser tests.

## Reproduction Assessment

Yes. The feature gap is reproducible by static inspection of current main: no `/image` command, no TUI image extraction helper, and no attachment field in the TUI send path. The PR defects are reproducible by tracing `extractImagePaths` and the `cleanText || text` fallback in the proposed diff.

## Solution Assessment

No. The requested direction fits the existing Gateway attachment contract, but this PR is not the best mergeable form because it rewrites prompts, leaks raw `@path` text for image-only messages, and bypasses the current shared TUI backend/local-mode design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses repeated whitespace and trims the result, and `sendMessage` uses that result for every non-BTW message. A normal prompt with indentation, leading whitespace, or intentional double spaces is silently changed before it reaches the model even when no image reference was extracted.
  - confidence: 0.95
- **[P2] Avoid sending raw @path text for attachment-only messages:** `src/tui/tui-command-handlers.ts:565`
  - body: For a message such as `@/tmp/screenshot.png`, `cleanText` is empty, so `message = cleanText || text` falls back to the original local path token. That displays and sends the raw path instead of sending only the attachment payload or a neutral placeholder.
  - confidence: 0.93
- **[P2] Handle local TUI mode before advertising /image:** `src/tui/commands.ts:113`
  - body: The command is added unconditionally, but current main uses the shared `TuiBackend` for both Gateway and embedded local mode. Embedded local sends currently forward only text, so `/image` would appear available in `openclaw chat` without a defined attachment path unless the backend contract is extended or the command is gated.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff adds local-file upload behavior and currently has concrete path-leak and size-check concerns.

Concerns:

- **[medium] Attachment-only prompts leak local file paths:** `src/tui/tui-command-handlers.ts:565`
  - body: The proposed `cleanText || text` fallback sends raw `@/local/path` text when an image reference is the whole prompt, exposing local path details to the Gateway/model transcript instead of stripping the reference.
  - confidence: 0.93
- **[low] File size guard races the read:** `src/tui/tui-image-extract.ts:88`
  - body: The proposed loader checks `fs.stat` before `fs.readFile`, so a file can be replaced or enlarged between the check and read; reading first and checking `buffer.byteLength` would avoid oversized local reads.
  - confidence: 0.78
