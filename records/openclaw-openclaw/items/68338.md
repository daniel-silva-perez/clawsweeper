## Summary

Keep open. Current main still does not parse a bare `MEDIA:` line followed by a media path, and the PR is a narrow implementation with focused tests; it should get normal maintainer review rather than cleanup closure. No blocking correctness finding was found in the final PR diff, but it is stale against several later media-parser hardening changes and overlaps the newly linked #75182 parser-hardening discussion.

## What This Changes

The PR adds continuation parsing for empty `MEDIA:` directives so the next line can supply the media path, plus tests that verify ordered text/media segments and single-line continuation consumption.

## Best Possible Solution

Keep the PR open and review it as a narrow parser enhancement: rebase or otherwise reconcile it with current main, add any required changelog/docs note if maintainers consider the accepted multiline form user-facing, and keep the trailing-JSON hardening tracked separately in #75182 unless the same parser change can safely cover both.

## Reproduction Assessment

Yes. A focused parser case such as `splitMediaFromOutput("Before\nMEDIA:\n/tmp/screenshot.png\nAfter")` is enough: current main keeps the empty `MEDIA:` line because no regex match exists, while the PR adds direct coverage for that behavior.

## Solution Assessment

Yes, mostly. The final PR approach is narrow and the earlier greedy-continuation bug was addressed by consuming only the immediate next line, but it still needs maintainer review against current main’s newer MEDIA parser safety rules before merge.
