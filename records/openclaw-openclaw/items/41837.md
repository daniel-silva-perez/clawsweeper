## Summary

Keep this issue open. Current main still does not implement active Mattermost transport liveness for a connected-but-deaf WebSocket: the Mattermost wrapper only reacts to open/message/close/error plus bot update_at polling, while the generic stale-socket health policy only works when a channel publishes lastTransportActivityAt. The related ping/pong implementation candidate #57621 and duplicate symptom issue #44160 are still open, so this remains actionable Mattermost plugin work rather than cleanup-close material.

## Best Possible Solution

Keep this issue open and review/land #57621 or an equivalent Mattermost-owned fix. The most direct implementation belongs in extensions/mattermost/src/mattermost/monitor-websocket.ts: add WebSocket ping/pong keepalive or another active transport-liveness check that terminates deaf sockets so runWithReconnect can reconnect, and add focused regression tests in extensions/mattermost/src/mattermost/monitor-websocket.test.ts. If maintainers prefer the generic health monitor path, Mattermost should also publish provider-proven lastTransportActivityAt rather than relying on app-message silence.
