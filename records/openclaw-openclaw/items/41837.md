## Summary

Keep open. Current main still does not actively detect a connected-but-deaf Mattermost WebSocket: the Mattermost plugin listens for open/message/close/error and polls the bot `update_at` timestamp, but it does not send WebSocket ping frames, listen for pong, or publish `lastTransportActivityAt` for the generic stale-socket health policy. The open related PR #57621 remains the clearest implementation candidate, while #44160 and #51104 show the same remaining symptom is still tracked elsewhere too.

## Best Possible Solution

Keep this issue open and review/land #57621 or an equivalent Mattermost-owned fix. The direct path is to extend `extensions/mattermost/src/mattermost/monitor-websocket.ts` with WebSocket ping/pong keepalive that terminates deaf sockets so `runWithReconnect` can reconnect, plus regression coverage in `extensions/mattermost/src/mattermost/monitor-websocket.test.ts`. If maintainers prefer the generic health-monitor route, Mattermost also needs to publish provider-proven `lastTransportActivityAt`; app-message silence alone should not drive stale-socket recovery.
