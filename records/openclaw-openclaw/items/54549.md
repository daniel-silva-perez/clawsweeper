## Summary

Keep open. Current main has background exec completion wake plumbing via `notifyOnExit`, but it does not implement this PR's explicit `onComplete="notify"` API, auto-background behavior, forced empty-success notification behavior, or no-poll response hint. Related #46926 remains open and overlaps the wake UX, but it tracks default background notification behavior rather than this explicit tool parameter, so it does not cleanly supersede this PR.

## Best Possible Solution

Keep this PR open for maintainer/contributor iteration. The best path is to decide whether OpenClaw wants an explicit `onComplete` API or prefers the #46926 default-notify direction; if pursuing this PR, narrow the schema to the supported literal, handle or reject `host=node`, gate the no-poll hint on a deliverable session route, add empty-success and no-session-key tests, and update `docs/tools/exec.md`.

## Risks / Open Questions

- The PR changes the exec tool, which is a code-execution surface. Review comments identify correctness risks: the no-poll hint can be shown without a routable `sessionKey`, node-host exec would not honor the new option, and unrecognized `onComplete` string values would be silently ignored.
- Current main's default `notifyOnExit` behavior is not a drop-in replacement for the proposed explicit `onComplete="notify"` contract because it does not auto-background a foreground call and still suppresses empty successful completions unless configured otherwise.
