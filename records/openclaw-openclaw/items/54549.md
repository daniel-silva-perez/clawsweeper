## Summary

Keep open. Current main still does not implement the PR’s requested per-call `onComplete="notify"` exec API: the schema, handler, docs, and tests only expose the existing `background`/`yieldMs` plus default `notifyOnExit` behavior, and successful empty-output completion wakes remain config-gated. ## What This Changes

The PR adds an exec tool `onComplete="notify"` option that auto-backgrounds a command, forces completion notifications, and wires explicit completion wake state through the bash process registry/runtime.

- The PR head promises that no polling is needed, but the proposed notification can be dropped when no `sessionKey` is configured, which can leave agent follow-up logic waiting for a wake that will not arrive.
- The proposed `onComplete` schema is an open string, so misspellings or unsupported values are silently ignored instead of being rejected or warned about.
- The proposed local exec handling does not implement or explicitly reject `onComplete="notify"` for `host=node`, where current main returns through `executeNodeHostCommand` before local background/yield handling.
- The PR branch appears stale against current main’s schema split: current main owns `execSchema` in `src/agents/bash-tools.schemas.ts`, while the provided PR diff adds the field in `src/agents/bash-tools.exec-runtime.ts`.
