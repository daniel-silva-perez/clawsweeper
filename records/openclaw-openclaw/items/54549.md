## Summary

Keep open: current main still does not implement the proposed `onComplete="notify"` exec API, and the PR is useful but not merge-ready. The proposed diff has concrete correctness issues around unroutable notification promises, unsupported values, and node-host semantics, and it overlaps the still-open default-notification direction in #46926.

## What This Changes

This PR adds an `onComplete="notify"` exec option that auto-backgrounds local commands, forces completion notifications, and threads explicit completion state through bash process sessions.

## Best Possible Solution

Maintain a single exec completion-wake design: either land a revised per-call API or the default-notification direction from #46926, with literal validation, routed and unrouted notification semantics, explicit node-host support or rejection, docs, and focused tests for quiet success plus wake delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate the no-poll hint on routable notifications:** `src/agents/bash-tools.exec.ts:1581-1583`
  - body: `onCompleteHint` is appended whenever `onComplete="notify"` is used, but `maybeNotifyOnExit` still returns without enqueuing anything when the session key is empty. Tool instances without `defaults.sessionKey` can tell callers not to poll even though no wake can be delivered.
  - confidence: 0.92
- **[P2] Reject or implement onComplete for node-host exec:** `src/agents/bash-tools.exec.ts:1248`
  - body: The patch treats `onComplete="notify"` as a background request, but `host=node` returns through `executeNodeHostCommand` before the local background/session path. Node executions therefore do not get the promised auto-backgrounding or per-call completion wake semantics.
  - confidence: 0.9
- **[P2] Constrain onComplete to supported values:** `src/agents/bash-tools.exec-runtime.ts:168-173`
  - body: The schema accepts any string while the handler only recognizes `"notify"`, so typos such as `"notfiy"` silently no-op. Expose a literal/enum schema or return a warning/error for unsupported values.
  - confidence: 0.86
