## Summary

Keep open. Current main has improved the misleading allowlist-warning path, but the routed-provider execution problem remains: `apply_patch` is still only constructed for exact `openai` or `openai-codex` provider IDs, so Tailscale/PPQ-style OpenAI-compatible routing can still lose the tool before policy filtering runs. Because this is a workspace-writing tool, the remaining direction needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

The issue asks for `apply_patch` to be recognized and usable as a supported built-in tool in agent-routed runs instead of being treated as unavailable or unknown during tool policy resolution.

## Best Possible Solution

Define the `apply_patch` capability contract for routed/OpenAI-compatible providers, then either add a guarded provider-alias or explicit capability override with tests and docs, or keep the strict provider gate and make diagnostics/docs explicit for routed-provider setups.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve coding tools with `tools.exec.applyPatch` enabled and a non-exact routed provider ID; current code omits `apply_patch` before policy filtering. I did not run a live routed gateway session in this read-only review.

## Solution Assessment

No. The workaround proposed in the issue targets `coreToolNames` classification, but current main shows the tool may never be constructed. The safer fix is an explicit routed-provider capability or alias contract, or clearer diagnostics/docs if strict gating is intentional.
