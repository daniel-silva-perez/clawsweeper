## Summary

Keep open. Current main no longer treats gated core tools as plugin-only in the warning text, but the reported routed-provider execution problem remains: `apply_patch` is still constructed only when the resolved provider is exactly `openai` or `openai-codex`, and routed OpenAI-compatible setups can still lose the tool before policy filtering. The issue discussion and related closed PR both point to this being a provider/model capability-gating decision rather than a finished policy-pipeline bug.

## What This Changes

The issue asks OpenClaw to make the built-in `apply_patch` tool available and recognized for eligible agent-routed runs instead of being reported unavailable or stripped during tool-policy resolution, especially through routed OpenAI-compatible provider chains.

## Best Possible Solution

Define an explicit `apply_patch` capability contract for routed/OpenAI-compatible providers, then either add a guarded provider capability or override path with tests and docs, or deliberately keep the strict OpenAI/OpenAI Codex gate and make the routed-provider diagnostic and documentation unambiguous.
