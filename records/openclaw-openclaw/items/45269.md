## Summary

Keep open. Current main has improved the misleading allowlist-warning path for gated core tools, but the reported routed-provider execution failure remains: `apply_patch` is still only constructed for exact `openai` or `openai-codex` provider IDs, so Tailscale/PPQ-style routed providers can omit the tool before policy filtering runs.

## What This Changes

The issue asks OpenClaw to recognize and expose `apply_patch` for eligible agent-routed runs instead of treating it as unavailable or unknown when routed/OpenAI-compatible providers are used.

## Best Possible Solution

Define the `apply_patch` capability contract for routed/OpenAI-compatible providers, then either add an explicit trusted provider-alias/capability opt-in with focused tests and docs, or keep the strict provider gate and document the limitation clearly.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve coding tools with `tools.exec.applyPatch` enabled and a non-exact routed provider ID; current main makes `applyPatchEnabled` false and omits `apply_patch` before policy filtering.

## Solution Assessment

No. The issue's local workaround targets `coreToolNames` classification, but current main shows the tool can be absent before the policy pipeline runs; the safer fix is an explicit provider capability/trusted alias contract or clearer strict-gate diagnostics.
