## Summary

Keep open. Current main has improved the misleading warning path for gated core tools, but it has not resolved the routed-provider execution problem: `apply_patch` is still only constructed for exact `openai` or `openai-codex` provider IDs, so Tailscale/PPQ-style OpenAI-compatible routing can still omit the tool before policy filtering runs.

## What This Changes

The issue requests that `apply_patch` remain available to eligible agent-routed runs instead of being stripped or reported as unknown when `tools.exec.applyPatch.enabled` is configured.

## Best Possible Solution

Define the `apply_patch` capability contract for routed/OpenAI-compatible providers, then either add a trusted provider-alias or explicit capability opt-in with focused tests and docs, or keep the strict provider gate and make diagnostics/docs explicit for routed-provider setups.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve coding tools with `tools.exec.applyPatch` enabled and a non-exact routed provider ID; current code makes `applyPatchEnabled` false and omits `apply_patch` before policy filtering. The current test suite already asserts the same strict provider boundary for OpenAI versus non-OpenAI providers.

## Solution Assessment

No. The workaround proposed in the issue targets `coreToolNames` classification, but current main shows the tool can be absent before the policy pipeline runs. The maintainable fix is an explicit routed-provider capability/trusted alias contract, or clearer diagnostics and docs if strict gating is intentional.
