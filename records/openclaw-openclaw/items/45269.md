## Summary

Keep open. Current main has improved the policy-pipeline diagnostics around unavailable core tools, but it does not prove the reported routed-provider execution problem is solved: `apply_patch` is still created only for exact `openai` or `openai-codex` provider IDs, and the related PR that tried to frame this as a policy classification fix was closed unmerged after discussion shifted the likely root cause back to provider/model capability gating.

## What This Changes

Recognize `apply_patch` as a supported core tool during allowlist resolution for eligible agent-routed runs instead of treating it as an unknown or plugin-only entry that prevents execution.

## Best Possible Solution

Clarify the `apply_patch` capability contract for routed providers, then either add a narrowly tested provider-alias/capability override for safe OpenAI-compatible routes or improve the warning/docs so unavailable gated core tools are clearly reported without implying plugin-only stripping.
