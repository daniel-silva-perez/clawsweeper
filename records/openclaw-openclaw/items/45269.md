## Summary

Keep open. Current main has improved the misleading policy warning path, but it still only constructs `apply_patch` for exact `openai` or `openai-codex` provider IDs, so the routed-provider execution problem described in the report and follow-up comments remains unresolved.

## What This Changes

The issue asks OpenClaw to stop treating `apply_patch` as unavailable/unknown in agent-routed runs when the runtime is configured to allow the built-in patch tool.

## Best Possible Solution

Define the `apply_patch` capability contract for routed providers, then either add a guarded provider-alias/capability override with focused tests and docs, or intentionally preserve the strict provider gate while making diagnostics and docs explicit for routed-provider setups.
