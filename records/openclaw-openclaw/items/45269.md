## Summary

Keep open. Current main has fixed the misleading plugin-only warning path, but it still constructs `apply_patch` only for exact `openai` or `openai-codex` provider IDs, so the routed-provider execution problem described in the discussion remains unresolved and the linked policy-pipeline PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to treat `apply_patch` as an available built-in tool for supported agent-routed runs instead of warning as unknown and preventing execution when provider/model gating hides it.

## Best Possible Solution

Define the `apply_patch` capability contract for routed providers, then either add a guarded provider-alias/capability override with focused tests and docs, or explicitly preserve the strict provider gate while making diagnostics and docs unambiguous for routed-provider setups.
