## Summary

Keep this issue open. Current main has adjacent improvements for model warmup and hook-specific diagnostics, but it still does not warn at gateway startup or in doctor when configured agent primary/fallback model refs are absent from a successfully loaded provider catalog. OpenRouter also still intentionally resolves arbitrary dynamic model IDs, so a retired OpenRouter slug can remain a synthetic runtime model until the provider rejects it.

## Best Possible Solution

Keep this issue open and implement non-fatal configured-model diagnostics. Gateway startup and openclaw doctor should collect configured model refs across defaults, fallbacks, media/tool refs, hooks, channel overrides, and per-agent overrides; when a provider catalog was successfully fetched and is authoritative enough to prove a ref is absent, warn with a migration hint while preserving OpenRouter/custom-provider dynamic pass-through when catalog authority is unavailable or intentionally incomplete. Release notes should call out OpenClaw-bundled default slug changes or known retired slugs when OpenClaw changes compatibility behavior.

## Risks / Open Questions

- Hard-failing every uncatalogued OpenRouter ID would regress intentional dynamic pass-through for newly published upstream models.
- A useful warning needs to distinguish authoritative successful catalog misses from unavailable, stale, incomplete, or user-declared custom catalog data to avoid false positives.
- The diagnostic should cover more than the default primary: configured fallbacks, media model refs, per-agent overrides, and alias-resolved refs can all drift after provider catalog changes.
