## Summary

Keep this issue open. Current main still has only adjacent model warmup and hook-specific diagnostics: startup prepares provider metadata but does not compare configured primary/fallback model refs against the loaded catalog, and doctor only validates `hooks.gmail.model`. OpenRouter also still intentionally resolves uncataloged dynamic IDs, so the remaining fix needs a non-fatal diagnostic rather than a hard block.

## What This Changes

The issue requests startup warnings, doctor validation, and release-note guidance when configured model IDs become invalid or retired after an OpenClaw npm update.

## Best Possible Solution

Add a shared configured-model diagnostics path used by both gateway startup and `openclaw doctor`: collect configured model refs, resolve aliases, compare them against authoritative loaded catalog data, emit non-fatal warnings with migration hints, and preserve dynamic/custom-provider pass-through when catalog authority is unavailable or intentionally incomplete.
