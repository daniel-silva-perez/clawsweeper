## Summary

Keep this issue open. Current main still prewarms configured primary model metadata and has narrow `hooks.gmail.model` catalog diagnostics, but it does not generically compare configured primary/fallback/per-agent/channel model refs against an authoritative provider catalog at startup or in `openclaw doctor`. OpenRouter also still supports uncataloged dynamic model IDs, so the remaining fix should be a non-fatal, authority-aware diagnostic rather than a hard validation failure.

## What This Changes

The issue requests startup warnings, doctor validation, and changelog migration guidance when configured model slugs become invalid or retired after an OpenClaw npm update.

## Best Possible Solution

Add a shared configured-model diagnostics layer that gateway startup and `openclaw doctor` both use: collect configured model refs, resolve aliases with runtime-equivalent semantics, compare against authoritative catalog evidence, emit migration-oriented warnings, and preserve dynamic/custom-provider pass-through when catalog authority is unavailable or intentionally incomplete.
