## Summary

Keep open. Current main still has the reported key-only follow-up queue cleanup, and PR #68839 remains the open unmerged implementation candidate linked with closing syntax, so this issue should stay open until that PR is reviewed and merged or explicitly closed.

## What This Changes

The issue requests guarding follow-up queue drain cleanup so a late drain cannot delete or clear callbacks for a replacement queue after `/stop` or session reset clears and re-enqueues the same key.

## Best Possible Solution

Review PR #68839 as the canonical implementation candidate. If it is correct, land the identity guard and regression test so GitHub closes this issue from the merged PR; if not, keep this issue open and request a revised guard for both map deletion and callback cleanup.

## Reproduction Assessment

Yes. Yes for the proposed direction in PR #68839: an identity guard around the late `finally` cleanup is the narrowest maintainable fix. Current main is not the best solution because it still clears by key without proving ownership of the current map entry.
