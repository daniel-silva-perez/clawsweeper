## Summary

Keep this issue open. Current main still rejects SecretRef objects in `sandbox.docker.env` at the config schema and public type layers, and the secrets runtime only collects sandbox SSH secret fields, not Docker env entries. The related PR #38310 is closed unmerged and remains useful implementation context rather than a superseding fix.

## Best Possible Solution

Implement SecretRef support end-to-end for Docker sandbox env in core: accept `SecretInputSchema` for `sandbox.docker.env` values, type unresolved settings as `Record<string, SecretInput>` while keeping resolved Docker config string-only, collect defaults and per-agent Docker env refs through the existing secrets runtime, add target-registry/docs/generated-schema coverage for audit/plan/configure, and add focused schema/runtime regression tests. The related PR #38310 can be used as design context, but an equivalent reviewed implementation still needs to land on main.

## Risks / Open Questions

- Adding this surface changes secret handling for Docker sandbox containers; resolved values will still become container environment variables and may be visible to users with Docker/container inspection access, matching the existing plaintext env exposure but worth explicit maintainer acceptance.
- The implementation needs clear active-surface semantics so unresolved refs only block startup/reload when Docker sandbox env is effectively active for enabled agents, and does not accidentally block inactive/off sandbox configurations.
- Resolved secret values must not be written back to config, generated schema/docs, logs, audit output, model snapshots, or Docker debug messages.
