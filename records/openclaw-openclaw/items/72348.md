## Summary

Keep open. The bug is real on current main and this PR uses the right extension-owned action seam, but the patch still mishandles shared reaction removal semantics, so it should not be merged as-is while #72679 is also open for the same root issue.

## What This Changes

The PR adds and registers a Nextcloud Talk message action adapter, adds adapter and reaction-sender tests, and adds a changelog entry for #70110.

## Best Possible Solution

Land this PR or an equivalent Nextcloud Talk-owned adapter after it explicitly implements or rejects reaction removal requests, preserves the existing SSRF-guarded sender behavior, keeps focused adapter/sender tests, and resolves the duplicate #72679 path before closing #70110.

## Reproduction Assessment

Yes. Against current main, dispatching a Nextcloud Talk `react` action has no plugin `actions.handleAction`, so the dispatcher returns null instead of reaching the existing reaction sender.

## Solution Assessment

No, not as currently proposed. Registering a channel action adapter is the narrow maintainable fix, but the adapter must handle or explicitly reject `remove: true` and empty-emoji removal before reporting success.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle reaction removals before sending:** `extensions/nextcloud-talk/src/message-actions.ts:60-62`
  - body: `remove` is part of the shared `react` schema, and the docs define `remove: true` as removing a specific emoji. This handler ignores that flag and always calls `sendReactionNextcloudTalk`, so `{ emoji: "👍", remove: true }` would add/report `added` instead of removing or rejecting the unsupported operation. Please either implement removal or reject removal requests before sending.
  - confidence: 0.9
