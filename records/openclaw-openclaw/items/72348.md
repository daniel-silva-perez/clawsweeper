## Summary

Keep this PR open, but do not merge it as-is. Current main still has the Nextcloud Talk reaction dispatch gap the PR targets, but the proposed adapter ignores the shared `remove` flag for `react`, so a removal request with a non-empty emoji would be sent as an add reaction.

## What This Changes

The PR adds and registers a Nextcloud Talk message action adapter that routes `react` actions to `sendReactionNextcloudTalk`, adds adapter and sender tests, and adds a changelog entry for #70110.

## Best Possible Solution

Land this PR or an equivalent Nextcloud Talk-owned fix after the adapter explicitly implements or rejects reaction removal semantics, preserves the existing SSRF-guarded sender behavior, keeps focused adapter/sender tests, and then closes #70110 while resolving the duplicate PR path.

## Reproduction Assessment

Yes. Against current main, dispatching a Nextcloud Talk `message` tool `react` action follows a plugin-action path, but the Nextcloud Talk plugin has no `actions.handleAction`, so the dispatcher returns unsupported instead of calling the existing reaction sender.

## Solution Assessment

No, not as currently proposed. The adapter registration is the right narrow seam, but the patch should handle or explicitly reject `remove: true` before merge because the shared `react` action exposes removal semantics.
