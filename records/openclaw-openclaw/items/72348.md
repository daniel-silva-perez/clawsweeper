## Summary

Keep open: current main still has the reported Nextcloud Talk reaction dispatch gap, and the PR uses the right channel-owned action adapter seam, but the current patch still has a merge-blocking behavior bug because `remove: true` reaction requests are sent as add-reaction POSTs.

## What This Changes

The branch adds and registers a Nextcloud Talk message action adapter, adds adapter and reaction sender unit coverage, and adds a changelog entry for the Nextcloud Talk reaction dispatch bug.

## Best Possible Solution

Land one canonical Nextcloud Talk-owned adapter fix for #70110 after the handler preserves the shared reaction contract by rejecting unsupported removal or implementing a tested DELETE/removal path, while keeping the existing SSRF-guarded sender contract and changelog coverage.

## Reproduction Assessment

Yes. On current main, dispatching a Nextcloud Talk `message` tool `react` action reaches a plugin with no `actions.handleAction`, so dispatch returns unsupported instead of calling `sendReactionNextcloudTalk`; the PR-specific bug is reproduced by calling the new handler with `{ remove: true, emoji: "👍" }`.

## Solution Assessment

No, not as currently proposed. Registering a plugin-owned `ChannelMessageActionAdapter` is the narrow maintainable seam, but the handler must not convert a removal request into an add request; explicit unsupported-removal rejection is the safer minimal fix if DELETE support stays out of scope.

## Review Findings
