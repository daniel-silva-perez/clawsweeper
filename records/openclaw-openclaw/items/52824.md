The PR changes subagent completion announce routing to preserve Telegram DM-topic `threadId`, prioritizes chat-level `to` for bound-route conversation lookup, adds Telegram success-log thread observability, and adds regression coverage.

## Best Possible Solution

Rebase and narrow the PR onto current main: keep the chat-target-first requester conversation lookup in `src/agents/subagent-announce-delivery.ts` with focused regression coverage for Telegram topic/multiple-binding behavior, decide whether to add accurate effective-thread Telegram success logs, and avoid re-landing stale-entry fixes already covered on main.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with a Telegram requester origin containing `to: "123"` and `threadId: 42`, current main builds the bound-route requester conversation from `"42"` before `"123"`, which can miss chat-level bindings when more than one binding is active.

## Solution Assessment

Unclear as-is. The chat-target-first lookup is a narrow plausible fix for the remaining routing gap, but the defensive merge guard overlaps current `resolveAnnounceOrigin` behavior and the logging change should report the effective thread after any Telegram threadless fallback.
