## Summary

Keep open. Current main still does not implement the PR's intended Telegram DM-topic completion routing change, and the PR needs changes before merge because its chat-target-first lookup still does not parse the actual `telegram:<chatId>` requester target shape used by Telegram inbound runs; the logging addition also reports the pre-fallback thread rather than the effective send parameters.

## What This Changes

The PR changes subagent completion announce routing to preserve Telegram DM-topic thread IDs, adds Telegram `message_thread_id` success-log observability, and adds announce regression coverage.

## Best Possible Solution

Rebase and narrow the branch onto current main, derive the requester conversation id with channel/plugin-aware parsing of the chat target before falling back to `threadId`, keep `threadId` as the delivery modifier, add coverage for actual `telegram:<chatId>` DM-topic requester origins with multiple active bindings, and either log the effective post-fallback Telegram thread or leave the success log unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists for a Telegram DM-topic requester where inbound context supplies `to: "telegram:<chatId>"` and `threadId: <topicId>`; the current and proposed lookup can still fall back to the topic id instead of the chat id, which misses exact bound-route matching when multiple bindings are active.

## Solution Assessment

No as-is. The chat-target-first direction is the right narrow fix, but the PR needs to parse Telegram-prefixed targets before falling back to `threadId`, and its logging change should report the effective send params after threadless fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Parse Telegram-prefixed requester targets before falling back to threadId:** `src/agents/subagent-announce.ts:639-642`
  - body: Actual Telegram runs pass requester targets as `telegram:<chatId>` and the topic separately as `threadId`. `resolveConversationIdFromTargets()` ignores that colon-prefixed target, so this priority swap can still fall back to the topic id and miss the chat-level binding in multi-binding cases. Parse or normalize the Telegram target before using `threadId`, and cover the real target shape.
  - confidence: 0.84
- **[P3] Log the effective Telegram thread after fallback:** `extensions/telegram/src/bot/delivery.send.ts:117-118`
  - body: The suffix is derived from `baseParams` before `sendTelegramWithThreadFallback` can retry without `message_thread_id`. When that retry succeeds, the success log would still report the original thread, making the new observability misleading.
  - confidence: 0.78
