## Summary

Keep open. The PR targets a real Telegram DM-topic announce routing bug, but the proposed routing change is still incomplete for actual Telegram requester origins shaped as `telegram:<chatId>` plus `threadId`, so it needs author or maintainer revision rather than cleanup closure.

## What This Changes

The PR changes subagent completion announce routing to preserve Telegram DM-topic thread IDs, adds Telegram sendMessage thread logging, and adds announce regression coverage.

## Best Possible Solution

Rebase and narrow the branch onto current main, parse Telegram-prefixed requester targets into the chat conversation id before falling back to `threadId`, keep `threadId` as a delivery modifier, add regression coverage for `to: "telegram:<chatId>"` with multiple active bindings and stale entry thread ids, and either log the effective post-fallback Telegram thread or leave the success log unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: Telegram inbound context supplies `to: "telegram:<chatId>"` plus `threadId`, `sessions_spawn` records both, and the PR’s lookup still skips the prefixed target before falling back to the topic id for bound-route matching.

## Solution Assessment

No as-is. Preferring the chat-level route before topic `threadId` is the right direction, but the PR must parse the actual Telegram target shape and avoid misleading fallback logging before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Parse Telegram requester targets before falling back to threadId:** `src/agents/subagent-announce.ts:639-642`
  - body: Actual Telegram requester origins use `to: "telegram:<chatId>"` and keep the topic in `threadId`. This change calls `resolveConversationIdFromTargets([to])`, which skips unknown colon-prefixed targets, then falls back to the topic id, so multi-binding bound delivery can still miss the chat-level requester binding. Normalize the Telegram target and cover that shape.
  - confidence: 0.84
- **[P3] Log the effective Telegram thread after fallback:** `extensions/telegram/src/bot/delivery.send.ts:117-118`
  - body: The new suffix is computed from `baseParams` before `sendTelegramWithThreadFallback()` can retry without `message_thread_id`. A successful threadless retry would still log the original thread id, making the added diagnostic misleading.
  - confidence: 0.78
