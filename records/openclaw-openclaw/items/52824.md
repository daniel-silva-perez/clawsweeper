## Summary

Keep open. Current main still has the PR's remaining routing delta: subagent completion lookup derives the requester conversation from `threadId` before `to`, and Telegram success logs still omit `message_thread_id`; the stale-entry thread guard mostly overlaps behavior already covered on main, so this needs a narrow rebase rather than closure.

## What This Changes

The PR changes subagent completion announce routing to preserve Telegram DM-topic thread IDs, adds Telegram `message_thread_id` success-log observability, and adds regression coverage.

## Best Possible Solution

Rebase and narrow the change onto current main: derive the requester conversation id from the chat target before `threadId`, add focused coverage for Telegram DM topics with multiple active bindings, and either log the effective Telegram thread after fallback or leave the success log unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists with `requesterOrigin` `{ channel: "telegram", to: "123", threadId: 42 }` and multiple active Telegram bindings: current main builds requester `conversationId` as `"42"`, while the router needs the chat-level `"123"` to exact-match the correct binding.

## Solution Assessment

Unclear as-is. The chat-target-first lookup is the narrow maintainable fix for the remaining routing gap, but the defensive merge guard is largely redundant on current main and the logging change should report the effective post-fallback thread state.
