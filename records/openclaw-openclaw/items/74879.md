## Summary

Keep open. Current main still puts shell-completion cache generation on the onboarding path, and that helper still runs a synchronous 30-second `completion --write-state` subprocess without the plugin-skip environment used by the update path; the related open PR #72910 is adjacent but not merged and may only lengthen the wait rather than remove onboarding's slow critical path.

## What This Changes

The issue reports that OpenClaw 2026.4.26+ onboarding on Docker/umbrelOS takes minutes and shell completion often times out, whereas 2026.4.14 completed in a few seconds.

## Best Possible Solution

Keep automatic onboarding and doctor completion work bounded: use the existing core-only completion-cache path or defer full plugin-command cache generation outside onboarding, while preserving explicit full plugin completions for manual `openclaw completion --write-state` runs.

## Reproduction Assessment

Yes for the source path: a fresh QuickStart with no shell completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and then the synchronous 30-second completion-cache spawn. The reported Docker timing was not live-measured in this read-only review.

## Solution Assessment

Unclear that the adjacent open PR is the best complete fix. Raising the timeout can reduce `ETIMEDOUT` on slow machines, but the narrower maintainable fix for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
