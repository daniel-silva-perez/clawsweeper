## Summary

Keep open. Current main still puts automatic shell-completion cache generation on the onboarding path, and that generation still uses a synchronous 30-second `completion --write-state` subprocess that loads plugin commands unless an environment skip flag is set. The adjacent open PR #72910 covers the same timeout family but is not merged and does not yet prove the onboarding critical path is fixed.

## What This Changes

The issue reports a regression where onboarding and shell completion became much slower in OpenClaw 2026.4.26+ than in 2026.4.14, especially on Docker/umbrelOS.

## Best Possible Solution

Keep automatic onboarding fast by using a bounded/core-only completion-cache path for onboarding or deferring full plugin-command cache generation outside the critical setup flow. Preserve explicit full `openclaw completion --write-state` behavior for users who request it manually.

## Reproduction Assessment

Yes for the source path: a fresh QuickStart with no shell completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and then the synchronous 30-second cache-generation subprocess. The exact reported Docker wall-clock duration was not live-reproduced in this read-only pass.

## Solution Assessment

No for current main, and unclear for the adjacent PR as a complete fix. Increasing the timeout may reduce `ETIMEDOUT`, but the narrower maintainable fix for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
