## Summary

Keep open. Current main still puts shell completion cache generation on the classic onboarding path, and the doctor/onboarding helper still spawns full `openclaw completion --write-state` synchronously with a 30-second timeout and normal plugin-command discovery. The related open PR #72910 is adjacent but not merged and appears to raise the timeout rather than remove full completion generation from the onboarding critical path.

## What This Changes

The issue reports a regression where OpenClaw onboarding and shell completion became much slower in versions 2026.4.26 and newer, especially on Docker/umbrelOS.

## Best Possible Solution

Automatic onboarding and doctor health work should stay bounded and core-only: reuse or centralize the update path's skip-plugin completion refresh for automatic setup paths, preserve explicit full plugin-command cache generation for manual `openclaw completion --write-state`, and coordinate that with #72910 so timeout/progress changes and the fast-onboarding fix do not compete.

## Reproduction Assessment

Yes for the code path: a fresh QuickStart run with no completion profile reaches `setupWizardShellCompletion`, which synchronously generates the completion cache through the 30-second doctor helper. The exact multi-minute Docker timing is not live-verified from this checkout.

## Solution Assessment

Unclear that the adjacent open PR is the complete solution: raising the timeout helps slow-but-successful cache writes, but does not make automatic onboarding fast. The narrower maintainable fix is to use a skip-plugin/core-only cache refresh or defer full plugin completion generation outside the onboarding critical path.
