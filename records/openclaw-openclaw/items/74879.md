## Summary

Keep open. Current main and the latest tagged release still put completion-cache generation on the interactive onboarding finish path, and the onboarding helper still uses the slower full `completion --write-state` path rather than the bounded core-only update path.

## What This Changes

The issue reports that onboarding in 2026.4.26+ on Docker/umbrelOS takes minutes and shell completion often times out, whereas 2026.4.14 completed in seconds.

## Best Possible Solution

Keep automatic onboarding completion work off the slow full plugin-command path: reuse a core-only completion-cache generation mode or defer full plugin-command cache generation outside onboarding, while preserving explicit full completions for manual `openclaw completion --write-state` runs.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and then the synchronous 30-second `completion --write-state` subprocess. The specific reported Docker timing was not live-measured during this read-only review.

## Solution Assessment

Unclear that the adjacent open PR is the best complete solution. Raising the timeout may reduce `ETIMEDOUT`, but the narrower maintainable fix for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
