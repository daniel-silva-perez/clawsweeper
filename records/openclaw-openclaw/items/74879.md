## Summary

Keep open. Current main still keeps shell-completion cache generation on the onboarding path, and the shared helper still runs a synchronous 30-second `completion --write-state` subprocess without the core-only plugin-skip environment used by the update path. The related open PR #72910 is adjacent, but it is not merged and does not clearly remove full completion generation from onboarding's critical path.

## What This Changes

The issue reports a regression where OpenClaw 2026.4.26+ Docker onboarding on umbrelOS/Debian takes minutes and shell completion cache generation often times out around 30 seconds.

## Best Possible Solution

Keep automatic onboarding and doctor completion work bounded: use the existing core-only completion-cache path or defer full plugin command cache generation outside onboarding, while preserving explicit full plugin completions for manual `openclaw completion --write-state` runs.

## Reproduction Assessment

Yes for the code path: a fresh QuickStart with no completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and then the synchronous 30-second completion-cache spawn. The reported Docker timing itself was not live-measured here because this was a read-only review.

## Solution Assessment

Unclear that the adjacent PR is the complete solution. Raising the timeout can reduce `ETIMEDOUT` on slow machines, but the narrower maintainable fix for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
