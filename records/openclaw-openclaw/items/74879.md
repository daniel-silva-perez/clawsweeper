## Summary

Keep open. Current main still runs shell-completion cache generation during onboarding, and the cache helper still synchronously spawns full completion generation with a 30-second timeout. The related open PR #72910 is relevant but unmerged, and it does not clearly remove full completion generation from onboarding’s critical path.

## What This Changes

The issue reports a regression where Docker/umbrelOS onboarding in 2026.4.26+ is much slower than 2026.4.14, with shell completion often timing out around 30 seconds.

## Best Possible Solution

Keep automatic onboarding and doctor completion work bounded: reuse or centralize the existing core-only skip-plugin cache refresh for automatic setup paths, or defer full plugin-command cache generation outside onboarding, while preserving explicit full plugin completion generation for manual openclaw completion --write-state runs.

## Reproduction Assessment

Yes for the code path: a fresh QuickStart with no shell completion profile reaches setupWizardShellCompletion, forces installation, and calls ensureCompletionCacheExists, which spawns completion --write-state with a 30-second timeout. The exact Docker wall-clock slowdown was not measured here.

## Solution Assessment

Unclear whether the adjacent timeout-only direction is the best fix. Raising the timeout may avoid ETIMEDOUT on slow hosts, but the narrower maintainable solution for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
