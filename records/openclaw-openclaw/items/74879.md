## Summary

Keep open. Current main still puts shell-completion cache generation on the onboarding path, and that helper still synchronously spawns full completion generation with a 30-second timeout. The adjacent open PR #72910 touches the timeout/progress area, but it is not merged and does not clearly remove full completion generation from onboarding’s critical path.

## What This Changes

The issue reports a regression where OpenClaw 2026.4.26+ onboarding and shell completion are much slower than earlier releases, especially in Docker/umbrelOS.

## Best Possible Solution

Keep automatic onboarding and doctor completion work bounded: reuse or centralize the existing core-only skip-plugin refresh for automatic setup paths, or defer full plugin-command cache generation outside onboarding, while preserving explicit full plugin completion generation for manual openclaw completion --write-state runs.

## Reproduction Assessment

Yes for the code path: a fresh QuickStart with no completion profile reaches setupWizardShellCompletion, which calls ensureCompletionCacheExists and then the synchronous 30-second completion-cache spawn. The exact Docker wall-clock slowdown was not live-measured here.

## Solution Assessment

Unclear that the adjacent PR is the complete solution. Raising the timeout can avoid ETIMEDOUT on slow machines, but the narrower maintainable fix for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
