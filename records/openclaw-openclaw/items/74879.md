## Summary

Keep open: current main and the latest release still route fresh onboarding through synchronous completion-cache generation, and that helper still runs the full `completion --write-state` path with a 30-second timeout. The linked PR #72910 is relevant but does not by itself prove the onboarding slow path is removed, so this needs maintainer coordination rather than cleanup closure.

## What This Changes

The issue requests fixing a regression where OpenClaw onboarding and shell-completion generation became much slower in versions 2026.4.26+ compared with 2026.4.14.

## Best Possible Solution

Keep full plugin-command completion generation available for explicit `openclaw completion --write-state`, but move automatic onboarding cache refresh onto a bounded core-only path or defer full cache generation outside the onboarding finish path.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists`, which synchronously spawns `completion --write-state` with a 30-second timeout. The specific Docker/umbrelOS multi-minute timing was not live-measured in this read-only pass.

## Solution Assessment

Unclear because this issue has no patch, and #72910 appears to be only a partial related fix. Raising the timeout may reduce `ETIMEDOUT`, but the narrower maintainable solution for onboarding speed is to avoid or defer full plugin-command completion generation on the automatic onboarding path.
