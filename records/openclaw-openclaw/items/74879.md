## Summary

Keep open. Current main and the latest tagged release still put shell-completion cache generation on the onboarding path, and the shared onboarding/doctor helper still performs a synchronous 30-second `completion --write-state` subprocess without the core-only plugin-skip environment used by the update path.

## What This Changes

The issue requests fixing a regression where onboarding and shell completion became much slower or timed out on 2026.4.26+ compared with earlier OpenClaw releases.

## Best Possible Solution

Keep automatic onboarding completion work bounded by reusing a core-only completion-cache generation path or deferring full plugin-command cache rebuilds outside onboarding, while preserving explicit full `openclaw completion --write-state` behavior for users who request it.

## Reproduction Assessment

Yes for the source-level reproduction path: a fresh QuickStart with no shell completion profile reaches `setupWizardShellCompletion`, which awaits `ensureCompletionCacheExists`, which runs the synchronous 30-second completion-cache subprocess. The exact reported umbrelOS wall-clock slowdown was not reproduced live in this read-only pass.

## Solution Assessment

Unclear whether the adjacent timeout-increase PR is the best complete fix. The narrower maintainable fix is to prevent automatic onboarding from doing full plugin-command completion generation, rather than making users wait longer on the same critical path.
