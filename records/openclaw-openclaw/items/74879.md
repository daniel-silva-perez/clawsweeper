## Summary

Keep open. Current main still puts onboarding shell-completion setup on the finalization path, and the automatic cache helper still invokes a synchronous `completion --write-state` child process with a 30-second timeout; the related open PR is adjacent but not merged and does not clearly remove full completion generation from onboarding’s critical path.

## What This Changes

The issue reports a regression where OpenClaw 2026.4.26+ onboarding takes minutes compared with 2026.4.14, with shell completion often timing out around 30 seconds.

## Best Possible Solution

Keep automatic onboarding and doctor completion work bounded: reuse or centralize the existing core-only skip-plugin refresh for automatic setup paths, or defer full plugin-command cache generation outside onboarding, while preserving explicit full plugin completion generation for manual `openclaw completion --write-state` runs.

## Reproduction Assessment

Yes for the code path: a fresh QuickStart with no completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and then the synchronous 30-second completion-cache spawn. The exact Docker wall-clock slowdown still needs environment proof from the reporter or a maintainer machine.

## Solution Assessment

Unclear that the adjacent PR is the best complete solution. Raising the timeout can avoid `ETIMEDOUT` on slower machines, but the narrower maintainable fix for onboarding speed is likely to avoid full plugin-command completion generation on the automatic onboarding path.
