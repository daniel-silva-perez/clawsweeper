## Summary

Keep open. Current main and the latest release still put completion-cache generation on the onboarding finish path, and that helper still runs the full `completion --write-state` subprocess with a 30-second timeout instead of the bounded core-only mode already used by the update path. The overlapping open PR #72910 may help timeout/progress behavior, but it does not by itself prove the onboarding slowdown is fixed.

## What This Changes

The issue reports a regression where OpenClaw 2026.4.26+ Docker onboarding takes minutes and shell completion often times out, whereas 2026.4.14 completed within seconds.

## Best Possible Solution

Move automatic onboarding and doctor completion-cache generation off the slow full plugin-command path: use the existing core-only completion mode or defer full plugin-command cache generation outside onboarding, while keeping explicit full `openclaw completion --write-state` available for manual refreshes.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no installed completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and then the synchronous 30-second `completion --write-state` subprocess. The specific reported Docker timing was not live-measured during this read-only sweep.

## Solution Assessment

Unclear that the adjacent timeout PR is the best complete solution. Raising the timeout may reduce `ETIMEDOUT`, but the narrower maintainable fix for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
