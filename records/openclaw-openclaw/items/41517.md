## Summary

Keep open. Current main still does not implement this PR’s intended same-provider prefix stripping for `resolveModel("google", "google/gemini-2.5-pro", ...)` against a bare configured Google model, and the overlapping #41398 is still open rather than shipped. The PR is plausible but needs maintainer consolidation because review comments identify functional risks around broader normalization and provider-native nested IDs.

## Best Possible Solution

Keep this PR open and consolidate it with #41398. The best implementation is a narrow resolver fix that strips a duplicated active-provider prefix only when it is truly redundant for matching and transport, preserves pass-through and nested provider-native model ids, adds the Google configured-model regression test from this PR, and closes the competing PR only after the selected fix ships.

## Risks / Open Questions

- Merging this exact patch may normalize more than redundant slash prefixes because `parseModelRef` runs provider model-id normalization, which can change bare IDs and error text.
- A review comment flags possible breakage for provider-native nested IDs whose first segment equals the provider namespace; NVIDIA catalog and tests show these ids are an active compatibility concern.
- Closing this PR before a chosen fix lands would leave the current-main Google OpenAI-compatible resolver path unhandled; #41398 is still open and unmerged.
