## Summary

Keep open. Current main still lacks the requested generic RPM/RPD-aware fallback cooldown contract: provider config and schema do not expose quota fields, auth-profile usage state has no minute/day quota counters, and fallback preflight only reacts to existing cooldowns or failures. Recent discussion already kept this open, and related #38775 is broader and still open rather than a merged canonical replacement.

## What This Changes

Request to add generic per-provider RPM/RPD quota settings and quota-derived auth-profile cooldown/fallback preflight behavior.

## Best Possible Solution

Define a generic quota contract, persist minute/day quota state with exact reset-time calculation, integrate pre-request fallback skipping with cooldownUntil under the auth-profile store lock, and update config schema/help/docs plus focused auth-profile and model-fallback tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source inspection gives high confidence that the requested contract is absent across config/schema, usage state, and fallback preflight.

## Solution Assessment

Unclear until maintainer review. Quota-aware cooldowns are a plausible direction, but the maintainable API shape should be chosen before implementation because it crosses public config/schema, persisted state, and provider/plugin routing semantics.
