## Summary

Keep open. Current main still has reactive auth-profile cooldowns, cooldown probing, and provider usage display, but not a generic provider-level RPM/RPD quota contract, persisted minute/day quota counters, or a pre-request quota-aware skip that writes exact cooldownUntil windows. Related #38775 is adjacent and broader, but it is still open and does not supersede this narrower RPM/RPD cooldown request.

## Best Possible Solution

Keep this open as a core model fallback/auth-profile enhancement. The best path is maintainer agreement on a generic provider quota contract, then implementation of persisted minute/day quota state, exact reset-time calculation, pre-request skip that writes cooldownUntil, schema/help/docs updates, and focused tests for RPM/RPD windows alongside existing cooldown/probe coverage. Coordinate the design with #38775 but do not close this as a duplicate unless maintainers choose that issue as the canonical umbrella.

## Risks / Open Questions

- A fix needs product/API agreement on where quota limits live, how they interact with existing provider usage snapshots, and whether local daily reset should use gateway-local time or configured provider/account time.

- The request overlaps with broader usage-aware routing in #38775, so the two designs should be aligned before implementation.
