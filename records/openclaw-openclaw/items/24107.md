## Summary

Keep open. Current main still lacks the requested generic RPM/RPD-aware fallback cooldown contract: provider config/schema do not accept quota fields, auth-profile usage state has cooldown metadata but no minute/day counters, and model fallback only skips or probes from already-recorded cooldowns rather than doing quota preflight.

## What This Changes

Add configurable per-provider RPM/RPD quota handling so fallback cooldowns use precise quota reset windows instead of generic failure-driven backoff.

## Best Possible Solution

Define a generic quota contract, persist exact minute/day quota state with reset-time calculation, integrate pre-request fallback skipping with cooldownUntil under the auth-profile store lock, and update config schema/help/docs plus focused auth-profile and model-fallback tests while coordinating scope with #38775.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. Source inspection gives high confidence that the requested contract is absent across config/schema, usage state, docs, and fallback preflight.

## Solution Assessment

Unclear until maintainer review. Quota-aware cooldowns are a plausible solution, but the maintainable API boundary should be chosen before implementation because it crosses public config, persisted auth state, provider/plugin semantics, and concurrency behavior.
