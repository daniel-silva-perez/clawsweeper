## Summary

Keep open. Current main has improved cooldown behavior since the original report, but it still does not implement the requested generic RPM/RPD-aware fallback contract: provider config has no quota fields, usage state has no minute/day quota counters, and fallback only reacts to already-recorded cooldowns or failures. The maintainer comment already reached the same keep-open conclusion, and #38775 is related but broader and still open rather than a clear canonical replacement.

## What This Changes

The issue requests provider-configurable RPM/RPD quotas that precompute exact cooldownUntil windows and skip fallback candidates before sending requests that would exceed those quota windows.

## Best Possible Solution

Define a generic quota contract, add persisted minute/day quota state with exact reset-time calculation, integrate pre-request fallback skipping with cooldownUntil under the existing auth-profile store locks, and update config schema/help/docs plus focused auth-profile and model-fallback tests.
