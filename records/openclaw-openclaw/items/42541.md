## Summary

Keep open. Current main still leaves the central Gemini OAuth refresh visibility problem unresolved in automatic provider resolution and provider-usage auth, even though parts of the report have been improved: direct profile refresh now wraps a useful error and runtime auth snapshots are updated after saves. The later user comments and the prior ClawSweeper review are consistent with the remaining code-level gaps, and the linked attempted fix PR was closed unmerged.

## What This Changes

The issue requests fixing Gemini CLI OAuth refresh so expired tokens are refreshed or fail visibly instead of being swallowed by auth resolution, stale runtime snapshots, and cooldown behavior.

## Best Possible Solution

Keep this open until Gemini CLI OAuth has covered regression tests for expired profiles through automatic embedded provider resolution and provider-usage auth, refresh failures are surfaced at an operator-visible level, and the Gemini-owned projectId, refresh hook, doctor guidance, and cooldown policy are explicitly settled.

## Reproduction Assessment

Yes for the source-level failure path: an expired google-gemini-cli OAuth profile that makes resolveApiKeyForProfile throw will be downgraded to debug-only logging in automatic provider auth and ignored in provider-usage auth. Live OAuth reproduction is not included here because this was a read-only review.

## Solution Assessment

Unclear. Logging the swallowed catches is necessary but probably not sufficient; the maintainable fix should also cover the Gemini provider-owned refresh/projectId guidance and decide whether cooldown behavior should differ for Gemini refresh failures.
