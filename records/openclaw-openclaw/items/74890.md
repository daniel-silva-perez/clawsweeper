## Summary

Keep this issue open. Current main still lets the macOS direct/local config writer replace the config before treating `gateway-mode-removed` as suspicious, and `ConfigStore.save` still falls back to that direct writer after gateway `config.set` failures. The browser Control UI and TypeScript gateway writer are already guarded, so the remaining actionable gap is the macOS Swift writer/fallback path.

## What This Changes

The issue asks Dashboard/macOS config writers to preserve required config envelope fields such as `meta` and `gateway.mode`, or reject partial root writes before replacing `openclaw.json`.

## Best Possible Solution

Direct/local macOS saves should share the gateway writer's pre-commit safety contract: merge against a fresh config or reject/preserve required envelope fields before replacing `openclaw.json`, with focused Swift regression coverage for `OpenClawConfigFile.saveDict` and the `ConfigStore.save` local fallback.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to start with an existing config containing `gateway.mode: local`, then call `OpenClawConfigFile.saveDict` with a root whose `gateway` object lacks `mode`; current main writes that payload before only auditing `gateway-mode-removed`.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the macOS direct writer and local fallback behavior while leaving the already guarded browser Control UI `config.set` base-hash path and TypeScript gateway rejection path intact.
