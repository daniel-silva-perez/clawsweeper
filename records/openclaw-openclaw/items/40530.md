## Summary

Keep open. Current main does not implement this Feishu thread follow-up mention policy, and this PR is not ready to merge because the latest predicate still uses `replyInThread` as a bypass signal, which current main sets for ordinary rooted replies; it also adds a public config key without updating the manifest/docs and overlaps with #48945's safer API direction.

## What This Changes

The PR adds a Feishu global and per-group `requireMentionInThread` option, wires thread context into Feishu reply-policy resolution, and adds policy tests for thread mention overrides.

## Best Possible Solution

Converge on one Feishu thread-follow-up policy that only auto-responds in bot-owned or explicitly topic-scoped follow-ups, then update bot logic, policy resolution, schema, plugin metadata, docs, and bot/policy tests together.

## Reproduction Assessment

Yes. A normal Feishu group event with `root_id`, no bot mention, `requireMention: true`, and `requireMentionInThread: false` would make current session resolution set both `threadReply` and `replyInThread`, so the PR predicate can bypass the mention gate for an ordinary quote reply.

## Solution Assessment

No. A broad boolean override is not the best current shape because it relies on ambiguous `replyInThread` semantics; a topic-scoped policy like #48945's `threadFollowUp` direction is safer if maintainers want this feature.
