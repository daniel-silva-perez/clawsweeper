## Summary

Keep this PR open. Current main does not implement a Feishu thread-specific mention policy, and the PR is useful source work, but the latest diff still introduces a concrete mention-gate bypass for ordinary quote replies and leaves public config docs/generated metadata out of sync. The overlapping #48945 proposes a different, safer product shape, so this needs maintainer reconciliation or author rework before merge.

## What This Changes

The PR adds a Feishu `requireMentionInThread` global/per-group config option, threads an `isThreadReply` flag into `resolveFeishuReplyPolicy`, and adds policy tests for relaxing and tightening mention requirements in thread replies.

## Best Possible Solution

Land one Feishu-owned thread follow-up policy that only treats intentional bot topic follow-ups as implicit mentions, preserves ordinary group quote replies behind the mention gate, reconciles the public config shape with #48945, updates schema/docs/generated config metadata together, and adds bot-level regression coverage for topic scopes, configured `replyInThread`, normal quote replies, top-level messages, and bidirectional override behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep quote replies behind the mention gate:** `extensions/feishu/src/bot.ts:421-425`
  - body: The added predicate includes `groupSession?.replyInThread`, but current main sets `replyInThread` true whenever an inbound Feishu group message has `root_id` or `thread_id`. With `requireMentionInThread: false`, ordinary quote replies in normal groups can therefore bypass `requireMention` and dispatch to the agent without an @mention.
  - confidence: 0.92

  - confidence: 0.84

## Security Review

Status: needs_attention

Concerns:

- **[medium] Mention-gate bypass for quote replies:** `extensions/feishu/src/bot.ts:421`
