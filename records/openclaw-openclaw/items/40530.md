## Summary

Keep open. Current main does not implement the Feishu thread-reply mention bypass, and this PR still needs rework before merge because its thread predicate can let ordinary rooted quote replies bypass the Feishu @mention gate. It also adds a public config key without updating the plugin metadata or docs, and it overlaps with the active #48945 API direction.

## What This Changes

The PR adds a Feishu global and per-group `requireMentionInThread` option, wires thread context into Feishu reply-policy resolution, and adds policy tests for thread mention overrides.

## Best Possible Solution

Converge on one Feishu thread follow-up policy that only auto-responds in bot-owned or explicitly topic-scoped follow-ups, then update bot logic, policy resolution, schema, plugin metadata, docs, and bot/policy tests together.

## Reproduction Assessment

Yes for the blocking PR bug. A normal Feishu group event with `root_id`, `requireMention: true`, and `requireMentionInThread: false` would have `threadReply` and `replyInThread` set by current session resolution, causing the PR predicate to bypass the mention gate without an @mention.

## Solution Assessment

No. The broad boolean override is not the best current shape because it relies on `replyInThread`, which current main uses for ordinary rooted replies too; a topic-scoped `active`/`topic`/`off` style policy like #48945 is a safer API candidate if maintainers want this behavior.
