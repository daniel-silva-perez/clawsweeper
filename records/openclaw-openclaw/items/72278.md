## Summary

Keep open. Current main still matches the reported SecretRef product/API gap: supported secret-provider sources are limited to env/file/exec, batching only happens inside one configured provider alias, and the public external-vault examples still show one exec provider per concrete secret with `id: "value"`. Related work is useful context, but #70952 was closed as a narrower 1Password duplicate-traffic item, #56217 and #55459 track separate 1Password/macOS symptoms, and the open plugin-provider seam work still needs maintainer review before it can be treated as the generic fix.

## What This Changes

The issue requests a first-class generic SecretRef integration model that batches and memoizes external vault/provider lookups across backend contexts instead of relying on one exec provider per secret.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The end state should provide a supported external-backend pattern where one provider represents a backend/account/context, SecretRef ids carry backend paths, startup resolution batches active refs per backend, repeated container/item/payload work is memoized in memory for that pass, and docs/configure/apply/tests steer users away from one-command-per-secret examples. A reviewed plugin-provider seam or successor to the open related PR is likely the right route if maintainers want external backends outside core.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: source, schema, protocol, resolver, and docs still only support env/file/exec providers, group by exact source/provider alias, and show 1Password/Vault examples as per-secret exec providers with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation. The requested generic direction is sound, but maintainers still need to choose the durable API across config schema, exec compatibility, plugin/provider boundaries, configure/apply UX, docs, and security guarantees.
