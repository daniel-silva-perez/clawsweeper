## Summary

Keep open. Current main still matches the reported product/API gap: SecretRefs and secret providers are limited to env/file/exec, batching only works within one configured provider alias, and the public external-vault examples still encourage one exec provider per secret with `id: "value"`. The related reports are useful symptom evidence, but #70952 is closed as a narrower 1Password duplicate-traffic report while #56217 and #55459 track separate 1Password/macOS failure modes rather than this generic SecretRef integration model.

## What This Changes

The issue requests a first-class, generic SecretRef external-backend pattern that batches and memoizes lookups across vault-style secret providers without becoming a 1Password-only workaround.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where one provider represents a backend/account/context, SecretRef ids carry backend paths, startup resolution batches active refs per backend, repeated container/item/payload work is memoized in memory for that pass, and docs/configure/apply/tests guide users away from one-command-per-secret examples.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the source/schema/docs only support env/file/exec providers, the resolver groups by exact source/provider alias, and the public 1Password/Vault examples still use per-secret exec providers with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation. The requested generic direction is sound, but maintainers still need to choose the durable contract across config schema, exec compatibility, plugin/provider boundaries, configure/apply UX, docs, and secret-handling guarantees.
