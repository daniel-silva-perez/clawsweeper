## Summary

Keep open. Current main still has the SecretRef gap described in the report: OpenClaw batches exact refs within one configured provider alias, but it does not yet provide a first-class external-backend provider model or docs/configure guidance that preserves shared backend identity across vault-backed secrets.

## What This Changes

The issue asks OpenClaw to add a generic, first-class external SecretRef provider pattern that batches and memoizes vault-backed lookups across backend providers.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where users configure one provider per backend/account/context, put backend secret paths in SecretRef ids, resolve active refs in one startup pass, memoize repeated backend entity lookups in memory only, and update configure/apply/docs/tests so the batchable pattern is the default guidance.

## Reproduction Assessment

Yes. Current-main source and docs provide a high-confidence static reproduction: only env/file/exec providers are supported, batching is source/provider scoped, and the public 1Password/Vault examples still use one exec command per secret with id "value".

## Solution Assessment

Unclear as an implementation choice. The generic direction is sound, but maintainers still need to choose the narrowest durable contract across config schema, plugin/provider APIs, exec compatibility, docs, and security review.
