## Summary

Keep open. Current main still exposes only the generic env/file/exec SecretRef model, batches at exact-ref plus source/provider granularity, and documents 1Password/Vault examples in the one-exec-provider-per-secret shape that this report asks OpenClaw to replace or steer away from. The related closed planning issue does not supersede this because the current checkout does not contain native external-vault provider sources or backend/entity memoization.

## What This Changes

Add a supported external-vault SecretRef integration pattern that preserves backend identity so all vault backends can batch and memoize lookups during startup resolution.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is one supported external-backend pattern where users configure one provider per backend/account/context, place backend secret paths in SecretRef ids, resolve active refs in one startup pass, memoize repeated backend entity lookups in memory only, and update configure/apply/docs/tests to make that batchable path the default guidance.

## Reproduction Assessment

Yes for the missing supported capability: current-main source and docs show only env/file/exec providers, exact-ref/provider grouping, and per-secret exec examples. No live external-backend traffic reproduction was run, but it is not necessary to prove that the requested first-class external-vault model is absent.

## Solution Assessment

Unclear as an implementation choice because this is a product/API design issue, not a patch. The report's generic direction is sound; a narrow 1Password workaround or plugin-only punt would not address the documented config shape, but maintainers still need to pick the maintainable contract.
