## Summary

Keep open. Current main still matches the reported gap: SecretRefs are limited to env/file/exec providers, batching only groups exact refs within one configured provider alias, and the public external-vault examples still model each external secret as its own exec command with `id: "value"`. The related 1Password reports are useful evidence, but none of them supersedes this broader generic SecretRef product/API design item.

## What This Changes

Request to add a first-class generic external SecretRef provider pattern that batches and pass-scoped memoizes backend lookups across vault-backed secret providers, with matching docs, configure/apply behavior, and regression coverage.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where users configure one provider per backend/account/context, put backend secret paths in SecretRef ids, resolve active refs in one startup pass, memoize repeated backend entity lookups in memory only, and update configure/apply/docs/tests so the batchable pattern is the default guidance.

## Reproduction Assessment

Yes. Current-main source and docs provide a high-confidence static reproduction: only env/file/exec providers are supported, resolver grouping is source/provider scoped, and the public 1Password/Vault examples still use one exec command per secret with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation. The requested generic direction is sound, but maintainers still need to choose the durable contract across config schema, exec compatibility, plugin/provider boundaries, configure/apply UX, docs, and security guarantees.
