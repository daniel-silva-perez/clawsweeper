## Summary

Keep open. Current main and latest release still lose non-refusal ACP backend error classifications at the ACP response boundary: gateway chat events can carry `errorKind`, but `AcpGatewayAgent` only maps `refusal` to ACP `stopReason: "refusal"`; `timeout`, `rate_limit`, `context_length`, and `unknown` still become plain `end_turn` with no response metadata. The previous ClawSweeper comment remains accurate and this is a narrow fix-PR candidate.

## Best Possible Solution

Keep this issue open and implement a narrow ACP bridge fix: keep ACP `stopReason` values compliant, map `refusal` to `refusal`, and preserve non-refusal backend classifications such as `timeout`, `rate_limit`, `context_length`, and `unknown` in `PromptResponse._meta` so clients can distinguish failures from normal `end_turn`. Update focused ACP translator tests, ACP bridge docs, and the changelog if the metadata becomes supported behavior.

## Repair Work Prompt

Fix the ACP bridge so structured gateway backend error classifications are not lost at the ACP `PromptResponse` boundary. In `src/acp/translator.ts`, keep ACP `stopReason` values compliant: `errorKind: "refusal"` should resolve `stopReason: "refusal"`; non-refusal gateway `errorKind` values (`timeout`, `rate_limit`, `context_length`, `unknown`) should continue to use `stopReason: "end_turn"` but be preserved in a documented, namespaced `PromptResponse._meta` field so clients can distinguish backend failures from normal completion. Do not emit non-standard ACP stop reasons. Update `src/acp/translator.error-kind.test.ts` and any necessary stop-reason tests to assert the metadata shape, keep existing refusal/cancelled/final behavior intact, update `docs/cli/acp.md` if this becomes supported bridge behavior, and add a user-facing changelog entry with proper attribution if behavior/docs change. Related context: this issue, PR #41187, and issue #55919 as a downstream classification example.
