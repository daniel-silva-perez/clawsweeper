## Summary

Current main still does not satisfy this issue. Gateway chat events now carry structured `errorKind`, but `AcpGatewayAgent` maps only `errorKind: "refusal"` to ACP `stopReason: "refusal"`; `timeout`, `rate_limit`, `context_length`, and `unknown` still resolve as plain `end_turn` with no response metadata for ACP clients to distinguish them from normal completion.

## Best Possible Solution

Keep this issue open and implement a narrow ACP bridge fix: preserve ACP-compliant `stopReason` values, map `refusal` to `refusal`, and carry non-refusal backend error classification through `PromptResponse._meta` so clients can distinguish `timeout`, `rate_limit`, `context_length`, and `unknown` from normal `end_turn`. Add targeted ACP translator tests, update the ACP bridge docs, and add a user-facing changelog entry if the response metadata becomes part of the supported bridge behavior.

## Clownfish Work Prompt

Fix the ACP bridge gap from this issue. Current main already carries structured Gateway chat `errorKind` values, but `AcpGatewayAgent` maps only `errorKind: "refusal"` to ACP `stopReason: "refusal"` and collapses all other backend error kinds to plain `end_turn` with no response metadata. Instead, preserve non-refusal backend classification in `PromptResponse._meta` under an OpenClaw-owned namespaced key so ACP clients can distinguish backend failures from normal completion. Update `docs/cli/acp.md` to document the OpenClaw-specific metadata, and add a single-line changelog entry under the active version with proper attribution if the metadata is user-facing. Preserve existing cancellation/final/max_tokens behavior and reconnect prompt lifecycle behavior.

## Risks / Open Questions

- A fix must not emit non-ACP `stopReason` strings; SDK 0.20.0 only allows `end_turn`, `max_tokens`, `max_turn_requests`, `refusal`, and `cancelled`.
- The new metadata shape is client-visible. It should be namespaced under an OpenClaw-owned `_meta` key and documented as OpenClaw-specific, not represented as an ACP standard field.
- Rejecting the ACP prompt promise for non-refusal backend errors would be a larger compatibility change than preserving a normal ACP response with structured metadata.
