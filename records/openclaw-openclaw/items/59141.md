## Summary

Keep open. Current main and the latest tagged release still use semantic-only LanceDB recall, so the linked stale-recall bug is not implemented on main. The PR is useful source work, but it should not merge as-is because it is stale against the current `before_prompt_build` hook contract and introduces an auto-capture regression for explicit remember requests phrased as questions.

## What This Changes

The PR adds freshness-intent detection, wider candidate recall, recency-weighted reranking, timestamp/provenance hints, and regression tests for freshness-sensitive `memory-lancedb` recall.

## Best Possible Solution

Land a rebased narrow fix on current main that keeps `memory-lancedb` on `before_prompt_build`, adds freshness-intent recall reranking and timestamp/provenance hints, preserves the current auto-capture cursor behavior, avoids suppressing explicit remember requests that are phrased as questions, and adds focused regression tests for both freshness ordering and remember-question capture.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction path: freshness-sensitive prompts go through `before_prompt_build` to `db.search(vector, 3, 0.3)`, and `MemoryDB.search()` returns semantic vector results only. A mocked LanceDB test with an older closer vector result and a newer lower-similarity result would reproduce stale ordering.

## Solution Assessment

No, not as-is. The PR's freshness-ranking direction is the right narrow product fix, but the implementation should be rebased onto current `before_prompt_build` behavior and should not globally reject question-marked explicit memory requests.
