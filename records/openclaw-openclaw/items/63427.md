## Summary

Keep this issue open. Current main still constructs and starts a fresh `GatewayClient` for every `probeGateway()` call, and targeted source/test search found no per-URL or module-level `DEVICE_IDENTITY_REQUIRED` short-circuit. The reporter's corrected RCA is still consistent with current code: the client pauses reconnect for one instance, but repeated short-lived CLI/status/cron probes can create new probe instances.

## Best Possible Solution

Keep this issue open until a source-level fix lands. The best path is a narrow `src/gateway/probe.ts` change modeled on #63446: track recent `DEVICE_IDENTITY_REQUIRED` probe failures per gateway URL, short-circuit after a small threshold/TTL without constructing `GatewayClient`, clear on successful connection, and add focused tests proving the fourth repeated device-required probe does not instantiate the mock client. If #68280 is chosen as the vehicle, it should explicitly cover this repeated device-required probe storm before this issue closes.

Implement a focused fix for the repeated CLI WebSocket device-required probe storm. Current main still constructs a fresh `GatewayClient` for every `probeGateway()` call, so repeated short-lived CLI/status/cron probes can keep opening WebSockets after `DEVICE_IDENTITY_REQUIRED` closes. In `src/gateway/probe.ts`, add a module-level per-URL cache for recent device-required probe failures, using #63446 as guidance: after 3 device-required failures within a 5 minute TTL, return a synthetic failed `GatewayProbeResult` without constructing `GatewayClient`; include close code 1008, reason `device identity required`, and a hint that the probe was short-circuited; clear the cache on successful connection; keep caches independent per URL. Coordinate with #68280 and avoid duplicating its local auth fail-fast work; if #68280 has already landed when this starts, re-evaluate the current main behavior first.
