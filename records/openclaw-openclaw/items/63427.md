## Summary

Keep this issue open. Current main still lacks the per-URL/module-level device-required short-circuit described by the reporter and by the closed-but-unmerged fix PR. GatewayClient does pause reconnect after DEVICE_IDENTITY_REQUIRED for a single client instance, but probeGateway still constructs a fresh GatewayClient and starts a WebSocket probe for each call, so the cross-session/cron burst case is not fixed on main.

## Best Possible Solution

Keep this issue open until a source-level fix lands. The most direct path is to revive the PR #63446 approach in src/gateway/probe.ts: track recent DEVICE_IDENTITY_REQUIRED probe failures per gateway URL, short-circuit after a small threshold/TTL without constructing another WebSocket, clear the cache on successful connection, and add focused regression tests proving the fourth repeated device-required probe does not instantiate GatewayClient. If PR #68280 is intended to supersede this, it should explicitly cover the repeated device-required probe storm with equivalent tests before this issue closes.

Current main still constructs a fresh GatewayClient for every probeGateway() call, so repeated short-lived CLI/status/cron probes can keep opening WebSockets after DEVICE_IDENTITY_REQUIRED closes. In src/gateway/probe.ts, add a module-level per-URL cache for recent device-required probe failures, using the PR #63446 shape as guidance: after 3 device-required failures within a 5 minute TTL, return a synthetic failed GatewayProbeResult without constructing GatewayClient; include close code 1008, reason 'device identity required', and a hint that the probe was short-circuited; clear the cache on successful connection; keep caches independent per URL; do not change auth resolution, gateway connect policy, generated dist files, or add a new health endpoint.
