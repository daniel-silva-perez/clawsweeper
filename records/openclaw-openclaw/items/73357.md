## Summary

Keep open. Current main still lacks graceful ENOSPC/inotify watcher handling, and this PR is the active implementation candidate, but the latest diff still lets message-only storage exhaustion text enter the global transient unhandled-rejection suppressor without requiring watcher context.

## What This Changes

The PR adds a chokidar error listener to the builtin memory-core watcher and adds/test-covers a new `isTransientFileWatchError()` path for ENOSPC/inotify watcher errors.

## Best Possible Solution

Land a focused fix that handles watcher `error` events at each active memory watcher seam and keeps the global transient classifier narrow: ENOSPC/no-space message matching should require watcher or inotify context, with regression tests for both watcher positives and message-only disk-full negatives.

## Reproduction Assessment

Yes for a high-confidence code-level reproduction: current main initializes the builtin watcher during non-transient memory manager creation and has no error listener, while Chokidar documents ENOSPC watcher errors on its `error` event. The PR's unit tests validate classification behavior, but they do not live-reproduce inotify exhaustion or cover watcher-event integration.

## Solution Assessment

No. The watcher-local error handler is the right seam, but the current classifier is still too broad for global unhandled-rejection suppression; QMD watcher parity also needs either implementation or an explicit maintained follow-up before the user-visible claim is complete.
