## Summary

Keep this PR open. Current main has overlapping Telegram polling hardening, including a configurable 120s getUpdates/API-liveness watchdog and non-getUpdates API activity suppression, but it still does not implement this PR's central active update-handler bypass or a pre-handler middleware hook. The branch is stale against the pluginized Telegram layout, so it needs transplant work rather than direct merge.

## Best Possible Solution

Keep this PR open for maintainer or author follow-up. The best path is to transplant the active update-handler watchdog bypass into the current Telegram plugin layout, likely by adding narrow handler-lifecycle tracking after sequentialization and before Telegram handler registration, then wiring that state into TelegramPollingLivenessTracker or TelegramPollingSession with a bounded max bypass and regression coverage for a long-running handler that performs no Telegram API call. If maintainers decide the shipped pollingStallThresholdMs/API-liveness behavior is the intended product answer, close manually with that product decision.
