## Summary

Keep open. The PR isolates useful Telegram watchdog work from #41883, but it still patches the retired `src/telegram` implementation while current main runs Telegram from `extensions/telegram/src`; current main has adjacent liveness and threshold recovery, but not the active-handler hook this branch adds, so this needs maintainer judgment rather than auto-close.

## What This Changes

The branch cherry-picks Telegram watchdog stability changes into the old `src/telegram` implementation, adding a pre-handler hook, active-handler bookkeeping, capped watchdog bypass behavior, logging cleanup, and focused tests.

## Best Possible Solution

The desired endpoint is a plugin-native Telegram watchdog policy: either add bounded active-update-handler tracking under `extensions/telegram/src` with matching tests and docs, or explicitly close this work as superseded by the current liveness/threshold model.

## Reproduction Assessment

Yes for the PR correctness issue: current main has no `src/telegram` tree, and the active `TelegramBotOptions` lacks the PR's hook. The original runtime false-stall behavior still needs a maintainer-confirmed live or plugin-level reproduction.

## Solution Assessment

No, this PR is not the best way to solve the issue as written because it updates retired paths. The safer maintainable path is either a plugin-port integrated with `TelegramPollingLivenessTracker`, or an explicit maintainer decision that the shipped threshold/liveness behavior supersedes the bypass.
