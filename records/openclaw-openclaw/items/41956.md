## Summary

Keep open. Current main has overlapping Telegram polling watchdog hardening, including configurable getUpdates/API-liveness thresholds, but it still does not implement this PR's central bounded active update-handler bypass or a pre-handler middleware hook. The PR is stale against the pluginized Telegram layout, yet the focused runtime idea remains a plausible transplant candidate, and the same author still has paired split PR #41957 open.

## Best Possible Solution

Keep this PR open for maintainer or author follow-up. The best path is to transplant the active update-handler watchdog bypass into extensions/telegram/src, either by integrating bounded active-handler state into TelegramPollingLivenessTracker or by adding a narrow bot-core middleware seam after sequentialization and before handler registration. Add regression coverage for a long-running update handler that makes no Telegram API call. If maintainers decide the shipped API-liveness threshold is sufficient, close manually with that explicit product decision.
