## Summary

Keep this PR open, but do not merge it as-is. The requested cancellation support is still absent from current main and is a useful narrow API improvement, but the branch makes aborted work look successful by returning `hasError: false` with skipped result slots left undefined.

## What This Changes

The PR adds an optional `signal?: AbortSignal` parameter to `runTasksWithConcurrency`, stops workers from picking up new tasks after abort, and adds two unit tests for pre-aborted and mid-run cancellation.

## Best Possible Solution

Land the cancellation feature only after the helper preserves its success contract: no-signal behavior stays unchanged, already-running tasks may finish, but an abort that skips pending work must be reported as cancellation through `hasError`/`firstError` or an explicitly versioned result shape, with tests and a changelog entry updated together.

## Reproduction Assessment

Yes. The PR's own deterministic `limit: 1` mid-abort scenario reproduces the problem: after abort, later tasks are skipped and result slots stay undefined while the proposed implementation still reports no error.

## Solution Assessment

No. Checking the signal before picking up the next task is the right scheduling point, but silently returning success on cancellation is not the narrowest maintainable API behavior; cancellation should be surfaced as a non-success outcome.
