## Summary

Keep this PR open, but do not merge it as-is. The requested AbortSignal support is a useful additive API, and current main has not implemented it, but the proposed patch makes cancellation return a success-shaped result with skipped tasks and sparse results, which conflicts with the helper’s existing `hasError` contract and a prior Codex review finding in the PR discussion.

## What This Changes

The PR adds an optional `signal?: AbortSignal` parameter to `runTasksWithConcurrency`, stops worker loops from picking up new tasks after abort, and adds unit tests for pre-aborted and mid-run abort behavior.

## Best Possible Solution

Keep the additive cancellation capability, but make aborted runs explicit non-success outcomes so callers cannot receive `hasError: false` with skipped tasks. The narrowest maintainable fix is to align with existing abort helpers by throwing or returning a typed abort state, then update the new tests to assert that pre-aborted and mid-run aborts do not look like successful full completion while preserving current behavior when no signal is provided.

## Reproduction Assessment

Yes. The PR’s own pre-aborted scenario reproduces the contract issue: pass an already-aborted signal with one task, observe that no task runs, and the proposed test still expects `hasError` to be false.

## Solution Assessment

No. The requested AbortSignal support is a reasonable direction, but the proposed implementation is not the best fix because it silently returns partial results as success; a safer implementation must surface cancellation explicitly.

## Review Findings

- **[P1] Surface aborted runs as non-success:** `src/utils/run-with-concurrency.ts:23-25`
