## Summary

Current main has not implemented this PR's intended behavior: boundary path resolution still rejects symlink targets outside the configured workspace root, and workspace bootstrap reads still treat guarded read failures as missing files. This PR is also paired with open issue #64472 by the same contributor, so it should remain open as an implementation candidate. However, the submitted diff needs a narrower security design before merge because it relaxes shared boundary checks beyond symlink traversal and lacks automated regression coverage.

## Best Possible Solution

Keep this PR and #64472 open, but require a revised implementation before merge. The best maintainer path is a narrow, tested exception for intended workspace-bootstrap symlink traversal only: derive and canonicalize the effective OpenClaw state root, gate the exception to workspace roots that are themselves under that state root or to an explicit boundary mode, preserve rejection for direct absolute reads of sibling state directories, and add positive symlink plus negative direct-path and sandbox/security regression tests.

## Risks / Open Questions

- The submitted diff broadens shared boundary enforcement from a symlink-only exception to direct absolute access under the OpenClaw state tree, which could expose credentials, sessions, or plugin state through APIs that expect workspace or sandbox root confinement.
- The fallback uses `os.homedir()/.openclaw` instead of the effective canonical OpenClaw state root, so it can be wrong with `OPENCLAW_STATE_DIR`, `OPENCLAW_HOME`, profiles, or symlinked state directories.
- The relaxed behavior has no automated positive or negative tests in the PR, despite changing a security-sensitive path enforcement module.
- Because `resolveBoundaryPath` is reused by workspace, sandbox, plugin, and guarded file-open paths, a global fallback could weaken non-workspace boundaries outside the stated bootstrap use case.
