## Summary

Keep open. Current main still rejects Feishu webhook requests that lack `x-lark-request-timestamp`, `x-lark-request-nonce`, or `x-lark-signature` before parsing the `url_verification` challenge, and the latest tagged release has the same behavior. The open fix PR #60485 appears to be the right item to review/land rather than closing this issue before the fix reaches main.

## What This Changes

The issue reports that Feishu/Lark webhook URL verification fails with `401 Invalid signature` when webhook mode uses `encryptKey` because Feishu's setup challenge can arrive without signature headers.

## Best Possible Solution

Review and land #60485 or an equivalent narrow Feishu plugin fix that detects/decrypts valid `url_verification` challenges before the normal delivery-event signature gate, while keeping unsigned non-challenge webhook events rejected and adding regression coverage for plaintext and encrypted setup challenges.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run the Feishu webhook monitor with `connectionMode: "webhook"`, `verificationToken`, and `encryptKey`, then POST a JSON `url_verification` body without Lark signature headers; current tests already encode that path as `401 Invalid signature`.

## Solution Assessment

Yes, if implemented phase-aware rather than as a broad unsigned-request bypass. The maintainable fix should accept only verified Feishu setup challenges, then preserve strict signature validation for normal event delivery.
