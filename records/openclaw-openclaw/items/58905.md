## Summary

Keep open. Current main still rejects Feishu webhook requests that lack `x-lark-request-timestamp`, `x-lark-request-nonce`, or `x-lark-signature` before JSON parsing and before `Lark.generateChallenge(...)`, so the unsigned encrypted URL verification flow described here remains blocked. The related fix PRs #58938 and #59990 were closed unmerged, and current tests cover signed encrypted challenges plus missing-signature rejection, not unsigned encrypted challenge acceptance.

## Best Possible Solution

Implement the phase-aware fix in `extensions/feishu/src/monitor.transport.ts`: after content-type/body-size guards but before ordinary signature rejection, parse the JSON envelope, allow only a valid encrypted `url_verification` challenge to pass through `Lark.generateChallenge(...)` with `encryptKey`, return the challenge response, and keep strict signature validation for all non-challenge events. Add regression coverage for unsigned encrypted URL verification and a guard that unsigned non-challenge events still return 401. The closed PRs #58938 and #59990 are useful references for the intended shape.
