## Summary

Keep open. Current main still rejects unsigned Feishu/Lark webhook URL verification requests before challenge parsing, and the linked focused fix PR #60485 remains open rather than merged; earlier fix attempts #58938 and #59990 were closed unmerged.

## What This Changes

Feishu/Lark webhook mode should accept setup-time `url_verification` challenges without `X-Lark-Signature` when `encryptKey` is set, decrypt/respond with the challenge, and keep normal event signature validation intact.

## Best Possible Solution

Land a current-main Feishu plugin fix that parses enough of `url_verification` requests to authorize matching-token plaintext/encrypted challenges before the strict signature gate, while preserving fail-closed signature validation for all non-challenge webhook events and adding positive plus wrong-token negative e2e coverage.

## Reproduction Assessment

Yes. The current e2e path already reproduces the reported failure by sending an unsigned `url_verification` request with webhook secrets configured and expecting `401 Invalid signature`; the issue also includes a raw Feishu verification POST without `X-Lark-Signature`.

## Solution Assessment

Yes, a phase-aware challenge exception is the right direction; a broad unsigned-request bypass would weaken webhook auth. PR #60485 matches the intended boundary, but should be ported to current main and extended with a wrong-token negative regression test.
