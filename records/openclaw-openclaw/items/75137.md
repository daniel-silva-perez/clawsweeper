## Summary

Keep open. This is a fresh, concrete `openclaw-tui` CPU report, current `main` does not contain a proven idle-CPU or orphaned-TUI fix, and the later #75289 report corroborates the same TUI-process surface while #73532 tracks a different Gateway/plugin-loader CPU path.

## What This Changes

The issue requests fixing `openclaw-tui` so local TUI sessions do not busy-loop at roughly 90-100% CPU while idle or during message turns.

## Best Possible Solution

Keep a TUI-process tracker open, capture `perf top`, `strace -f`, or a V8 CPU profile from an idle affected `openclaw-tui`, then fix the narrowest proven loop or shutdown path in `src/tui` or the `pi-tui` dependency with regression coverage for idle CPU and terminal-close cleanup.

Yes for the reported environment: the issue gives exact `openclaw chat` steps plus `top`/`ps` proof for the `openclaw-tui` process. It is not independently reproduced on current `main` yet, so profiling evidence is still needed before implementing a targeted fix.
