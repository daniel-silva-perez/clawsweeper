## Summary

Keep open. The report is fresh, concrete, and still matches current main: `chat` is local TUI mode, the TUI still has active render/timer and shutdown surfaces that have not been proven fixed, and the related #75289/#75294 work appears to address terminal-hangup shutdown rather than the idle open-window CPU loop reported here.

## What This Changes

The issue requests fixing `openclaw-tui` so local TUI sessions do not busy-loop at roughly 90-100% CPU while idle or during message turns.

## Best Possible Solution

Keep this as the active TUI idle-CPU tracker, collect `perf top`, `strace -f`, or a V8 CPU profile from an idle affected `openclaw-tui`, then fix the narrowest proven loop in `src/tui` or the `pi-tui` usage boundary. Review #75294 for the related shutdown/orphan path, but do not treat it as complete proof for the idle busy-loop until profiling or a regression test confirms that case.

Yes for the reported environment: the issue gives exact `openclaw chat` steps plus `top`/`ps` proof for the `openclaw-tui` process. It has not been independently reproduced on current main in this read-only review, so profiling evidence is still needed before a targeted code fix.

## Solution Assessment
