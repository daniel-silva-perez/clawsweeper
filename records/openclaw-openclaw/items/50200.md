## Summary

Keep this PR open. Current main still lacks the inbound Slack `files.info` fallback it proposes: `resolveSlackMedia` returns null when file URLs are absent, and the message/thread paths do not pass a Slack WebClient into media resolution. The explicit `download-file` action does fetch fresh metadata, which supports the PR rationale but does not cover automatic inbound DM attachment downloads. Related PR #62927 addresses a different Slack attachment-drop/logging path and does not supersede this work.

## Best Possible Solution

Keep this PR open for maintainer review or a corrected successor. The best path is to thread the Slack WebClient through direct message, forwarded attachment, and thread-starter media resolution; call `files.info` when event file URLs are missing; consider retrying with fresh metadata after stale URL failures; preserve the current URL/auth/SSRF protections; and add regression tests plus useful verbose logging for the affected inbound paths.

## Risks / Open Questions

- The PR as described only refreshes metadata when event URLs are missing; the review discussion notes it may still leave stale or expired event URLs unresolved unless the resolver retries with fresh `files.info` metadata after a failed download.
- A `files.info` fallback still consumes external URL metadata from Slack; the implementation should keep the existing `assertSlackFileUrl`, Slack-host allowlist, authenticated same-origin redirect behavior, and SSRF protections intact.
