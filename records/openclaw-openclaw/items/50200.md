## Summary

Keep this PR open. Current main still does not implement the PR’s intended inbound Slack `files.info` fallback for DM/file attachment media: inbound message, forwarded attachment, and thread-starter media resolution still depend on event-provided `url_private_download ?? url_private`, while the explicit `download-file` action is the only path that fetches fresh Slack file metadata first. The PR remains a plausible focused fix rather than obsolete or superseded by current main.

## Best Possible Solution

Keep this PR open for maintainer review or a corrected successor. The best implementation path is to thread the Slack WebClient through direct inbound message, forwarded attachment, and thread-starter media resolution; call `files.info` when Slack event file URLs are missing; consider a retry-with-fresh-metadata path after stale URL download failures; preserve the existing Slack URL/auth/SSRF protections; and add regression tests plus useful verbose logging for the affected inbound paths.
