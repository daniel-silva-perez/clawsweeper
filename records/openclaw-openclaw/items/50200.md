## Summary

Keep this PR open. Current main still lacks the inbound Slack `files.info` fallback for DM/file attachment media: the automatic message, forwarded attachment, and thread-starter paths still resolve only `url_private_download ?? url_private` and skip files when both URLs are missing. The existing explicit `download-file` action does fetch fresh metadata, which supports the PR rationale, and related PR #62927 addresses a different Slack attachment/drop path rather than superseding this work.

## Best Possible Solution

Keep this PR open for maintainer review or a corrected successor. The best path is to thread the Slack WebClient through direct message, forwarded attachment, and thread-starter media resolution; call `files.info` when event file URLs are missing; consider retrying with fresh metadata after stale URL download failures; preserve the current Slack URL/auth/SSRF protections; and add regression tests plus useful verbose logging for the affected inbound paths.
