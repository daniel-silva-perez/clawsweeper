## Summary

Keep this PR open. Current main still does not implement `sessions_delegate` or `sessions_delegate_batch`; the shipped session tool surface has `sessions_spawn` as a non-blocking background spawn and `sessions_send` as messaging to an existing visible session. The PR remains a plausible implementation candidate, but it needs maintainer product/API review, a rebase onto current subagent context-engine wiring, and security/cleanup hardening before it can land.

## Best Possible Solution

Keep this PR open as the implementation candidate for synchronous subagent delegation. If maintainers accept the core tool surface, rebase it onto current main, reuse the shipped context-engine preparation path, add delegate transcript redaction for inline attachments, make output capture and cleanup ordering race-safe, release cleanup gates in failure paths, and add focused docs/display/policy/tests for the accepted behavior.

## Risks / Open Questions

- This is a new built-in agent tool/API surface, so maintainers still need to decide whether synchronous delegation belongs in core session tools or behind a plugin/API seam.
- The PR diff accepts inline delegate attachment content; without extending transcript sanitization beyond `sessions_spawn`, persisted tool-call inputs could retain sensitive attachment payloads.
- The provided review context still flags an output-capture race: a very fast child run may complete before the delegate cleanup gate exists, allowing cleanup to delete the session/transcript before output is read.
- The branch is stale against current main because `prepareSubagentSpawn` wiring and rollback behavior already exist in a richer form on main.
