## Summary

Keep this PR open. Current main still does not implement `sessions_delegate` or `sessions_delegate_batch`; the existing `sessions_spawn` remains a non-blocking background-spawn primitive, and `sessions_send` only messages an existing visible session. The PR remains a plausible implementation candidate, but it needs maintainer API/product review, a rebase because current main already has stronger `prepareSubagentSpawn` wiring, and security cleanup around delegate attachment redaction and output-capture ordering before it can land.

## Best Possible Solution

Keep this PR open as the implementation candidate for synchronous subagent delegation. Maintainers should decide the product/API direction, rebase it onto current main, reuse or adapt the shipped `prepareSubagentSpawn` path, extend transcript sanitization for delegate attachments, make output capture robust against cleanup races, release any cleanup gate in failure paths, and add focused docs/display/policy/tests if the new core tool surface is accepted.

## Risks / Open Questions

- This is a new built-in agent tool/API surface, so maintainers still need to decide whether synchronous delegation belongs in core session tools or behind a plugin/API seam.
- The PR diff accepts inline delegate attachment content, but current transcript sanitization only redacts `sessions_spawn`, creating a concrete data-retention risk for persisted tool-call inputs unless fixed.
- The proposed output-capture gate is registered after spawn returns, so very fast child completion can still race cleanup and cause missing delegate output or early transcript deletion.
- The branch is partly stale against current main because `prepareSubagentSpawn` wiring and rollback behavior are already implemented differently and with more context metadata.
- No CI workflow, dependency source, lockfile, package publishing, or install-script changes were observed in the provided PR file list; the security-sensitive surface is the new tool execution path and transcript data handling.
