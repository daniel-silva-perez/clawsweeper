## Summary

Keep open. The legacy EasyClaw/browser-extension relay path is no longer the active implementation, but the Desktop-specific `user` profile failure remains on current main: the built-in existing-session profile still defaults to spawning `npx -y chrome-devtools-mcp@latest`, and the MCP stdio transport spawns that executable directly. The maintainer discussion also explicitly scoped this as separate from the broader existing-session brittleness and login handoff reports.

## What This Changes

The issue reports that Genspark Claw Desktop cannot reuse existing Chrome sessions because EasyClaw OAuth callback fails and the built-in `user` browser profile attempts to spawn `npx chrome-devtools-mcp` even though the desktop bundle lacks `npx`.

## Best Possible Solution

Make the packaged Desktop/browser runtime handle the built-in `user` profile without relying on ambient `npx`: either ship or resolve a pinned Chrome MCP runner across the app/package boundary, or fail early with a structured missing-runner diagnostic that points to the managed `openclaw` profile and an explicit `mcpCommand` override. Keep the removed EasyClaw extension relay path retired, and coordinate remaining UX overlap with #62288 and #63926.

## Reproduction Assessment

Yes for the remaining Desktop `user` profile path at source level: current main creates the built-in `user` profile as `existing-session`, builds default args for `npx -y chrome-devtools-mcp@latest`, and the upstream MCP transport directly spawns that executable. The EasyClaw relay half is obsolete on current main because the extension relay config is migrated away and `chrome-relay` is no longer auto-created.

## Solution Assessment

Unclear because there is no proposed patch here and the right fix requires a maintainer packaging strategy. The documented `mcpCommand` workaround is useful, but it is not the best complete solution for a built-in Desktop profile users expect to work without separately installing `npx`.
