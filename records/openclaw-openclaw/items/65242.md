## Summary

Keep this PR open. Current main still does not include the proposed completion-delivery gate or ACP child-session silent-wake behavior, but the PR is not safe to merge as-is because the patch has blocking completion ownership, cleanup, shadow-mode, and Discord thread-binding review findings, plus overlap with open PR #64416.

## What This Changes

The PR adds an env-flagged CompletionDeliveryGate, wires it into task-registry, subagent announce, and farewell paths, adds ACP silent-wake policy/tests, and changes ACP requester-origin plus Discord child-thread binding behavior.

## Best Possible Solution

Land a revised, narrower completion ownership design that coordinates task-registry banner/session fallback, subagent announce, and farewell delivery after actual delivery outcomes are known, preserves cleanup/retry behavior, keeps shadow mode observational, and splits or separately validates the Discord/requester-origin routing changes while reconciling #64416.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Claim announce delivery only after successful delivery:** `src/agents/subagent-announce.ts:231-235`
  - body: The new gate claim is taken before `runSubagentAnnounceFlow` has waited for output or delivered anything. This function can still return `false` later for deferred or failed announce paths, but the retained claim lets task-registry or farewell fallback paths treat the run as already delivered, leaving the requester with no completion message.
  - confidence: 0.9
- **[P1] Keep the dedupe short-circuit inside cleanup flow:** `src/agents/subagent-announce.ts:236-238`
  - body: When another path already claimed the completion, this `return true` exits before the function enters its `try`/`finally`. That skips cleanup such as label patching and `sessions.delete`, while callers interpret the result as a successful announce and do not retry.
  - confidence: 0.88
- **[P2] Keep shadow mode observational for farewells:** `src/agents/subagent-registry-completion.ts:82-83`
  - body: `isCompletionClaimed` returns true in shadow mode because claims are stored, so this branch can flip `sendFarewell` to false even though shadow rollout is supposed to log would-be blocks without changing user-visible behavior.
  - confidence: 0.86
- **[P1] Preserve parent-channel lookup for child bindings:** `extensions/discord/src/monitor/thread-bindings.manager.ts:641`
  - body: For child placement, `conversationId` can be the current thread when `parentConversationId` is absent. Assigning it directly to `channelId` makes `bindTarget` try to create a child thread under a thread, which Discord rejects; keep the existing thread-to-parent resolution fallback.
  - confidence: 0.84

## Risks / Open Questions

- The proposed gate can claim announce delivery before a user-visible completion is actually sent, allowing fallback paths to be suppressed after a failed or deferred announce.
- The duplicate short-circuit exits before the announce finally block, so cleanup such as sessions.delete can be skipped while callers treat the announce as successful.
- Shadow mode is meant to be observational, but the patch can still suppress subagent farewells when a claim exists.
- The PR mixes the completion gate with ACP requester-origin and Discord thread-binding routing changes, and the Discord parent-channel lookup change has an unresolved P1 correctness risk.
