## Summary

Keep open. Current main still lacks the PR’s combined intended behavior: tool-use output suppression remains phase-based, `sessions_send` still uses announce-mode A2A follow-up delivery, session delivery still preserves established external routes for inter-session turns, and the sanitizer does not implement the proposed self-instruction prelude stripping. The PR is also not merge-ready because the proposed route and sanitizer changes introduce concrete regressions.

## What This Changes

The PR suppresses unphased tool-use assistant output, adds self-instruction prelude stripping, removes `sessions_send` A2A announce delivery, and changes session delivery routing so `sessions_send` inter-session turns do not reuse external routes.

## Best Possible Solution

Split the work into narrower decisions: review or land a structured tool-use suppression fix such as #71589, make an explicit maintainer call on whether `sessions_send` should keep A2A announce semantics, and if current-turn outward delivery must be suppressed, do that without overwriting the stored external route. Any sanitizer prelude stripping should be gated by stronger structural leak signals and negative tests for valid replies.

## Reproduction Assessment

Yes. The current-main code and targeted fixtures provide a high-confidence reproduction path: model an unphased `stopReason: "toolUse"` assistant message in the embedded subscription harness, and exercise `sessions_send` plus `session-delivery` route resolution with an established Telegram/Discord route.

## Solution Assessment

No. The PR addresses real symptoms, but the combined patch is not the best way to solve them because it bundles separate policy decisions and introduces route persistence and sanitizer-regression risks; the safer path is narrower suppression plus explicit route-delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve stored external routes for sessions_send:** `src/auto-reply/reply/session.ts:558-561`
  - body: For every `sessions_send` inter-session turn this makes the route resolvers prefer `webchat` and `session:*` identifiers, and `initSessionState` then persists those values back into the target session. A Telegram/Discord-backed session can lose its stored external route until a real external message repairs it, breaking later announce/cron/system follow-ups. Suppress only the current turn's outward delivery while keeping the persisted external route intact.
  - confidence: 0.88
- **[P2] Gate monologue stripping on stronger leak signals:** `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts:418-424`
  - body: The marker-count branch returns only the final block whenever the prelude has broad phrases like "the prompt" and "I should" and the last block has none. Those phrases are common in legitimate multi-paragraph answers, so this can silently delete valid user-facing content. Require stronger structural evidence of a leaked internal prelude before replacing the whole reply.
  - confidence: 0.82
- **[P2] Remove the hardcoded personal-name bypass:** `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts:383`
  - body: The production exclusion regex includes the literal name `Bernhard`, so any block containing that name skips the internal-monologue filter even if it otherwise matches the leak pattern. This looks like a test/debug artifact and makes sanitizer behavior depend on one personal name.
  - confidence: 0.9
