## Summary

Keep open. Current main still does not expose a web_fetch private-network opt-in, and this PR is useful source work, but it should not merge as-is because it is stale against current main's SSRF policy surface and needs maintainer security review for the new private-network capability.

## What This Changes

The PR adds `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` to web_fetch config/schema/runtime/cache handling, generated config metadata, and SSRF tests for private and loopback access.

## Best Possible Solution

Land or replace this with one canonical implementation under `tools.web.fetch.ssrfPolicy` that preserves existing RFC2544 and IPv6 ULA exceptions, partitions cache entries by every SSRF policy flag, updates schema/types/generated metadata, adds public web_fetch docs and a changelog entry, and clearly documents the private-network threat boundary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current-main source inspection shows strict web_fetch config accepts RFC2544 and IPv6 ULA flags but rejects `dangerouslyAllowPrivateNetwork`, while the PR head inspection shows the proposed helper drops the current IPv6 ULA path.

## Solution Assessment

Is this the best way to solve the issue? No, not as written. The `ssrfPolicy` location and explicit dangerous flag are plausible, but the branch must be rebased to preserve current SSRF flags/cache keys and should add public docs/changelog plus maintainer security signoff.
