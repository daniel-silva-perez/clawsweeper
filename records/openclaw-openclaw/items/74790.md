## Summary

Keep open. Current main still lacks the requested web_fetch private-network config surface, but this PR should not merge as-is: it predates current main's shipped IPv6 ULA SSRF opt-in, is dirty against main, and adds a security-sensitive private-network capability that needs maintainer security review and public operator docs before becoming the canonical path.

## What This Changes

The PR adds a `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` opt-in across web_fetch runtime policy/cache handling, config schema/help/labels/generated metadata, and SSRF regression tests.

## Best Possible Solution

Adopt one canonical implementation for the web_fetch private-network opt-in that keeps the existing RFC2544 and IPv6 ULA fake-IP exceptions, partitions cache entries by every SSRF policy flag, updates generated config plus public docs and changelog, and leaves the broader web_search/image/gateway scope to #53469 unless maintainers intentionally fold that work in.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The chosen `tools.web.fetch.ssrfPolicy` location is plausible, but the PR needs a rebase that preserves all existing policy flags plus maintainer security/docs/changelog review before it is the narrow maintainable fix.
