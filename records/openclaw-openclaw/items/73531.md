## Summary

Keep open. Current main still performs two independent task reconciliation passes inside one status summary, and the linked implementation PR #73650 is open and unmerged with closing syntax, so this issue should remain open for maintainer review or landing of that PR.

## What This Changes

The issue reports that `openclaw status` reconciles the inspectable task registry twice through separate registry and audit summary helpers, adding avoidable latency on large task ledgers.

## Best Possible Solution

Review and land PR #73650 if its shared-snapshot approach is correct, or apply an equivalent narrow fix that reconciles tasks once inside `getStatusSummary()` and derives both registry and audit summaries from that same snapshot while preserving zero-argument helper behavior for other call sites.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `getStatusSummary()` calls both task summary helpers, and each helper currently invokes `reconcileInspectableTasks()` independently; no live 10K-record timing run was performed because this review is read-only.

## Solution Assessment

No for a module-level process cache as the first fix. A per-status-call shared reconciled snapshot is narrower, avoids cache invalidation and shared mutable-reference hazards, and matches the open PR direction.
