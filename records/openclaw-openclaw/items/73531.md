## Summary

Keep open. Current main still follows the reported duplicated reconciliation path, and open PR #73650 is an unmerged implementation candidate with closing syntax for this issue, so the issue should stay open until that PR lands or maintainers choose a different fix.

## What This Changes

The issue requests avoiding duplicate task reconciliation during `openclaw status`, preferably by reusing a reconciled task snapshot or cache across registry and audit summary calculations.

## Best Possible Solution

Review and land PR #73650 if its shared-snapshot approach is correct, or apply an equivalent narrow fix that reconciles tasks once inside `getStatusSummary()` and derives both registry and audit summaries from the same snapshot while preserving existing zero-argument helper behavior for other callers.

## Reproduction Assessment

Yes. The duplicate path is reproducible from current main by tracing `getStatusSummary()` through the two helper calls; no timing benchmark is needed to prove the deterministic double call.

## Solution Assessment

No for the issue’s module-level cache as the first implementation. A per-status-call shared reconciled snapshot is the safer maintainable fix because it avoids invalidation and mutable-reference hazards while removing the redundant work; PR #73650 appears to target that direction but remains unmerged.
