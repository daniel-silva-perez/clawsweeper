## Summary

Keep open. Current main still performs two independent task reconciliation passes during one status summary, and open PR #73650 is an unmerged implementation candidate with closing syntax, so the issue should remain open for maintainer review or landing of that PR.

## What This Changes

The issue requests eliminating redundant task reconciliation work during `openclaw status`, preferably by reusing the same reconciled task snapshot within one status invocation.

## Best Possible Solution

Reconcile tasks once inside the status summary path and pass that snapshot to both registry and audit summary builders while preserving zero-argument helper behavior for tasks and gateway call sites; cover both the helper contract and `getStatusSummary()` call count with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current `getStatusSummary()` calls both task summary helpers, and each helper independently invokes `reconcileInspectableTasks()`; no live timing run was performed in this read-only review.

## Solution Assessment

Yes for the shared-snapshot direction in PR #73650; it is narrower than a module-level cache and avoids cache invalidation or shared mutable-reference hazards. The reporter's module-level cache would need more lifecycle care than the status-only duplication requires.
