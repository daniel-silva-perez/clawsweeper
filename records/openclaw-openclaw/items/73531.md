## Summary

Keep open. Current main still matches the reported performance bug: `openclaw status` builds the task registry summary and audit separately, and both helpers independently call `reconcileInspectableTasks()`. The expensive `listTaskRecords()` clone/sort path is still reached for each helper, and the latest-release path has not been replaced by a cache or shared status snapshot.

## Best Possible Solution

Keep this issue open and fix the status/task maintenance path. The narrowest implementation is to compute the reconciled task list once during `getStatusSummary()` and derive both the registry summary and audit summary from that snapshot. If maintainers prefer a reusable cache, implement a documented in-process reconciliation cache with invalidation on task registry observer events and maintenance configuration changes. Add a focused regression test proving one status summary invocation does not call the expensive task-listing/reconciliation path twice.

## Risks / Open Questions

- A module-level cache must avoid exposing mutable cached task objects to callers that may mutate returned records; a per-status-call shared snapshot may be the safer first fix.
- Any cross-call cache would need explicit invalidation for restored/upserted/deleted task events and for `configureTaskRegistryMaintenance()` changes that alter cron recovery behavior.
