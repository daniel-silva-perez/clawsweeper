## Summary

Keep open. Current main still has the reported duplicate reconciliation path, and an open implementation PR (#73650) already references this issue with closing syntax, so this issue should remain open until that PR is reviewed, landed, or closed.

## What This Changes

The issue requests reducing `openclaw status` latency by avoiding two independent `reconcileInspectableTasks()` calls in one status summary invocation.

## Best Possible Solution

Review and land #73650, or an equivalent narrow change, so `getStatusSummary()` computes the reconciled task list once and derives both the registry and audit summaries from that same snapshot while preserving zero-argument helper behavior for other call sites.

## Reproduction Assessment

Yes. The current-main static path reproduces the bug with high confidence: one `getStatusSummary()` invocation reaches two helpers, and each helper independently calls `reconcileInspectableTasks()` and therefore `listTaskRecords()`.

## Solution Assessment

Yes, the open PR's per-status shared snapshot is a better first fix than a process-lifetime cache because it removes the duplicated work without adding cache invalidation or mutable-reference risks across unrelated callers.
