## Summary

Keep open. Current main still performs the duplicated task reconciliation path, and PR #73650 is an open, unmerged implementation linked with closing syntax, so this issue should remain open for review, revision, landing, or closure of that PR.

## What This Changes

The issue requests reducing `openclaw status` latency by avoiding two independent `reconcileInspectableTasks()` clone-and-sort passes during one status summary build.

## Best Possible Solution

Review and land PR #73650 if its per-status-call shared snapshot is correct, or merge an equivalent narrow fix that reconciles tasks once inside `getStatusSummary()` and derives both registry and audit summaries from that snapshot while preserving zero-argument helper behavior for other call sites.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main calls both status task helpers in one summary build, and each helper independently calls `reconcileInspectableTasks()` before reaching the clone-and-sort task list path.

## Solution Assessment

No for a module-level process cache as the first fix. A per-call shared reconciled snapshot is narrower, avoids invalidation and shared-reference hazards, and matches the open PR direction.
