## Summary

Keep open. Current main still does not ship the PR’s first-class `/interrupt <message>` command or persistent cross-session thread-context store, so the work is not obsolete, but the proposed patch is not merge-ready because it can misroute interrupt requeues and loses thread-context updates under concurrent cron deliveries.

## What This Changes

The PR adds a chat `/interrupt <message>` handler that aborts and requeues a running agent turn, plus a JSON-backed thread-context store wired into cron delivery and new-session prompt construction.

## Best Possible Solution

Refresh and likely split the PR: implement `/interrupt <message>` on the current command loader and registry/docs, preferably reusing `sessions.steer` or the existing abort/steer seams while preserving accountId, threadId, internal/webchat routing, and subagent cleanup; then land the thread-context store with provider-normalized keys, serialized writes, TTL/cap pruning, and focused channel/cron tests.

## Reproduction Assessment

Yes. The PR bugs are reproducible by static contract checks against current main: `/interrupt` requeues `agent` without accountId while current delivery resolution relies on `turnSourceAccountId`, and concurrent `saveThreadContext` calls use an unlocked read-modify-write path.

## Solution Assessment

No. The proposed direction is useful, but this is not the best merge shape yet; the safer solution is to refresh against current main, use the current session steering/delivery abstractions, register the command in the shared catalog, and make context persistence concurrency-safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve the interrupt delivery account:** `src/auto-reply/reply/commands-session-interrupt.ts:82-90`
  - body: This requeue passes `channel` and `to` but omits `accountId`. Current delivery resolution treats the supplied turn source as authoritative, so `lastAccountId` becomes undefined and `/interrupt` from a non-default Slack/Telegram/etc. account can resume on the wrong account or fail delivery. Include the originating account id in the requeued request or use a session-steering path that preserves it.
  - confidence: 0.88
- **[P1] Keep internal interrupts session-local:** `src/auto-reply/reply/commands-session-interrupt.ts:86`
  - body: The handler always sets `deliver: true` even when the command comes from the internal/webchat surface. On current Gateway code that either tries to choose an external message channel or errors after the active run has already been aborted, so webchat interrupts are not reliable session-local redirects. Gate delivery by surface or use the existing session steering RPC for internal clients.
  - confidence: 0.8
- **[P2] Register the command in the shared catalog:** `src/auto-reply/reply/commands-core.ts:203`
  - body: Adding only the text handler means the generated command catalog and native command deployment do not know about `/interrupt`. Discord/Telegram native commands and Slack slash-command setups are built from the shared registry, so users on those surfaces will not be offered this new command. Add a registry entry and docs/tests for the new command.
  - confidence: 0.86
- **[P2] Serialize thread-context store updates:** `src/sessions/thread-context-store.ts:137-141`
  - body: `saveThreadContext` loads the JSON store, adds one entry, prunes, and writes it back without any per-file serialization. Two cron or isolated-agent deliveries saving at the same time can both read the same snapshot and the later write drops the earlier context, making follow-up replies nondeterministically lose their prior context.
  - confidence: 0.9
- **[P2] Normalize Telegram chat ids before storing context:** `src/cron/isolated-agent/run.ts:943`
  - body: For explicit Telegram delivery targets, `resolvedDelivery.to` is the bare chat id, but inbound Telegram replies use `OriginatingTo` in `telegram:<chatId>` form for lookup. Saving the bare value here means the later `loadThreadContext` key does not match, so explicit Telegram threaded cron replies never receive the prior-context note.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff does not add supply-chain risk, but the new interrupt delivery path can misroute or fail replies across account/internal-channel boundaries.

Concerns:

- **[medium] Interrupt requeue can use the wrong delivery account:** `src/auto-reply/reply/commands-session-interrupt.ts:82`
  - body: The new `/interrupt` path requeues an agent run with explicit `channel`/`to` and `deliver: true` but no `accountId`. On multi-account channels this can route the redirected response through the default or wrong account, which is a concrete confidentiality and delivery-integrity risk for chat replies.
  - confidence: 0.84
