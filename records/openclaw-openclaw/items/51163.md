## Summary

Keep this PR open. Current main has adjacent interruption primitives, including queue mode `interrupt` and the Gateway `sessions.steer` RPC, but it still does not implement the PR’s two concrete changes: a chat-facing `/interrupt <message>` command and a persistent cross-session thread-context store for cron/isolated-agent replies. The PR discussion also contains material follow-up concerns around native command registration, account/thread preservation, internal/webchat delivery, Telegram key normalization, and serialized store writes, so this is still an implementation candidate rather than conservative cleanup closure material.

## Best Possible Solution

Keep the PR open and refresh it against current main. The best path is to build `/interrupt <message>` on top of the existing `sessions.steer`/abort seams, register it in the chat command registry and docs, and preserve accountId, threadId, internal/webchat delivery semantics, and subagent cleanup. For cross-session thread context, either integrate with the existing thread starter/history pipeline or add a documented persistent store with normalized provider keys, serialized writes, TTL/cap pruning, and focused tests for Slack threads, Telegram topics, webchat/internal routing, and concurrent cron deliveries.

## Risks / Open Questions

- Closing this PR would drop an unmerged implementation candidate for behavior current main does not yet ship.
