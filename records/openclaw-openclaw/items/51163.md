## Summary

Keep this PR open. Current main has adjacent interruption primitives, including `/queue interrupt`, Gateway `sessions.steer`, and a Control UI-only `/redirect`, but it still does not implement the PR’s two concrete changes: a Gateway/chat-facing `/interrupt <message>` command and a persistent cross-session thread-context store for cron or isolated-agent follow-up replies.

## Best Possible Solution

Keep this PR open and refresh it against current main. The best path is to either split the two features or update the PR so `/interrupt <message>` builds on `sessions.steer`/abort seams, is registered in the chat command registry and docs, and preserves accountId, threadId, internal/webchat routing, and subagent cleanup. For cross-session thread context, integrate with the existing thread starter/history pipeline or add a documented persistent store with normalized provider keys, serialized writes, TTL/cap pruning, and focused Slack, Telegram, webchat/internal, and concurrent cron-delivery tests.

## Risks / Open Questions

- Closing this PR would drop an open implementation candidate for behavior current main does not yet ship end to end.
- Merging the PR as-is could misroute redirected replies on multi-account, threaded, Telegram, Slack, webchat, or internal-chat surfaces unless accountId, threadId, and delivery mode are preserved through the requeue path.
- The proposed persistent thread-context store injects prior task summaries into future prompts; key normalization and serialized writes need careful review to avoid cross-conversation context leakage or nondeterministic context loss.
