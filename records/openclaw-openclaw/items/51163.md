## Summary

Keep this PR open. Current main has adjacent mechanisms, including queue mode `interrupt`, `/queue`, and the lower-level `sessions.steer` RPC, but it still does not ship this PR's two concrete changes: a chat-facing `/interrupt <message>` command and a persistent cross-session thread context store for cron/isolated-agent thread replies. The PR also still carries material review concerns around routing, native command registration, thread/account preservation, and serialized context writes, so the best maintainer path is a focused refresh rather than cleanup closure.

## Best Possible Solution

Keep the PR open and refresh it. The best path is to split or update the work: implement `/interrupt <message>` on top of the existing `sessions.steer`/abort seams while preserving accountId, threadId, internal/webchat delivery semantics, and native command registration; then handle cross-session thread context either through the existing provider thread starter/history pipeline or through a documented, normalized, serialized persistent store with focused tests for Slack, Telegram topics, webchat/internal routing, and concurrent cron deliveries.
