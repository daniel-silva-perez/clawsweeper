## Summary

Keep this issue open. Current main still has the reported Telegram hook metadata gap: Telegram derives reply and forwarded provenance for prompt/session context, but the canonical message hook payload sent to internal and plugin hooks only carries generic routing and sender metadata, and there is no current via_bot/viaBot path.

## What This Changes

Add Telegram-specific forward, via-bot, and reply-parent provenance metadata to inbound message-received hook events.

## Best Possible Solution

Add a Telegram-owned provenance builder that produces a small, bounded metadata object for forward, viaBot, and replyTo data, carry it through the canonical inbound hook context, and map it into both plugin and internal message-received events including the Telegram mention-skip ingest path. Cover it with focused mapper and Telegram tests, plus a short docs/changelog update.

## Reproduction Assessment

Yes. A source-level reproduction is clear: build or dispatch a Telegram inbound message with forward_origin or legacy forward_from*, via_bot, and reply_to_message, then inspect toPluginMessageReceivedEvent/toInternalMessageReceivedContext; current main emits only generic metadata.

## Solution Assessment

Yes. The additive metadata route is the narrowest maintainable fix because MessageReceivedHookContext already has a provider-specific metadata escape hatch and Telegram can keep the extraction local without adding cross-channel top-level SDK fields.
