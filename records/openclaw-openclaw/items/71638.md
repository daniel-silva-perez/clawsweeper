## Summary

Keep this issue open. Current main still exposes hook metadata as the right additive surface, but the canonical received-message context and both received-hook mappers only carry generic routing/sender fields, so Telegram forward, via-bot, and reply provenance still cannot reach `message:received` consumers.

## What This Changes

The issue requests additive Telegram received-hook metadata for forwarded-message, inline-via-bot, and reply-to provenance on inbound Telegram messages.

## Best Possible Solution

Add a Telegram-owned provenance normalizer, pass provider metadata through the generic canonical received-hook context, and merge it into plugin/internal received-hook metadata for both normal dispatch and mention-skip ingest. Keep core extension-agnostic, avoid new Telegram API calls, bound reply parent text, and cover the mapper plus Telegram paths with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a Telegram message containing `forward_origin`, `via_bot`, or `reply_to_message` reaches received-hook paths whose canonical context and mapper outputs currently have no field for those values.

## Solution Assessment

Yes. The existing `metadata` escape hatch is the narrow maintainable solution; the safer implementation is a Telegram-local normalizer plus a generic canonical metadata passthrough, not new Telegram-specific fields in core hook types.
