## Summary

Keep this issue open. Current main already extracts Telegram reply and forward provenance for the agent prompt path, but the `message:received` hook metadata path still drops the requested forward, via-bot, and reply fields, so the reported consumer gap remains actionable.

## What This Changes

The issue requests adding Telegram forward, via-bot, and reply context to inbound Telegram `MessageReceivedHookEvent.context.metadata`.

## Best Possible Solution

Add a narrow Telegram-owned provenance extractor and a generic hook metadata passthrough so both plugin and internal `message:received` hooks receive bounded `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` data without adding Telegram-specific policy to core.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to send a Telegram inbound message containing `forward_origin`, `via_bot`, or `reply_to_message` through the existing dispatch path and inspect `toInternalMessageReceivedContext`; current main maps only base sender/thread metadata. I did not run tests because this review is read-only, but the source path is direct.

## Solution Assessment

Yes. The requested additive `metadata` approach is the narrow maintainable fix, provided it is implemented through the Telegram plugin plus a generic hook metadata seam and reuses existing reply/forward visibility rules rather than exposing raw Telegram data unconditionally.
