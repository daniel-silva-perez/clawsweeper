## Summary

Keep this issue open. Current main still has the reported gap: Telegram already derives some reply and forward provenance for prompt/session context, but the shared `message:received` hook mappers do not pass that provenance into `context.metadata`, and there is no current `via_bot`/`viaBot` plumbing path at all.

## What This Changes

Request that Telegram inbound `message:received` hook events include forward, via-bot, and reply provenance under `MessageReceivedHookEvent.context.metadata`.

## Best Possible Solution

Add a small, additive Telegram-owned provenance payload under hook metadata, using existing reply/forward helpers where possible, adding via-bot extraction, capping reply text, and covering both the central dispatch hook path and the Telegram mention-skip ingest hook path with focused tests and docs/changelog updates.

## Reproduction Assessment

Yes. A targeted test can inject a Telegram message containing `forward_origin`, `via_bot`, and `reply_to_message`, then assert that the emitted internal `message:received` event lacks `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` on current main.

## Solution Assessment

Yes. The narrowest maintainable fix is additive metadata plumbing through the existing hook `metadata` escape hatch and shared canonical mapper, with Telegram-specific extraction kept inside the Telegram plugin rather than hard-coding Telegram fields into unrelated core paths.
