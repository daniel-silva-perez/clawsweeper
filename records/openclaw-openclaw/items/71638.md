## Summary

Keep this issue open. Current main still has the received-hook metadata escape hatch, but the Telegram inbound path does not carry forward, via-bot, or reply provenance into the canonical context or the plugin/internal received-hook mappers; the previous ClawSweeper keep-open review remains accurate against current main.

## What This Changes

Request to add Telegram forward, via-bot, and reply context metadata to `MessageReceivedHookEvent` without changing existing received-hook fields.

## Best Possible Solution

Implement an additive Telegram-owned provenance mapper, pass it through the canonical received-hook path, and cover both normal dispatch and mention-skip silent ingest with focused tests. Keep core generic, reuse existing Telegram forward/reply normalization where it fits, avoid new Bot API calls, and bound any parent-message text included in metadata.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: an inbound Telegram message with `forward_origin`, `via_bot`, or `reply_to_message` reaches received-hook mappers that currently have no field capable of carrying those values, and `via_bot` has no current code reference at all.

## Solution Assessment

Yes. Adding provider-specific metadata on the existing received-hook metadata escape hatch is the narrow maintainable fix; the safer implementation is a Telegram-local normalizer plus a generic canonical metadata passthrough rather than hardcoding Telegram fields into core hook types.
