## Summary

Keep this issue open. Current main still has the reported Telegram hook metadata gap: Telegram already derives reply and forward provenance for prompt/session context, but the received-message hook path only emits generic routing and sender metadata, and there is no current via-bot plumbing.

## What This Changes

Surface Telegram forward, via-bot, and reply-parent provenance on `MessageReceivedHookEvent.context.metadata` for received Telegram messages.

## Best Possible Solution

Add a Telegram-owned provenance payload under received-hook metadata, reusing existing reply/forward extraction where possible, adding via-bot extraction from the raw message, bounding reply text, and covering both normal dispatch and Telegram mention-skip ingest paths with focused tests and user-facing docs/changelog updates.

## Reproduction Assessment

Yes. A focused test can construct a Telegram inbound context containing forwarded, via-bot, and reply fields and assert that current main's plugin/internal received-message hook payloads omit `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo`.

## Solution Assessment

Yes. The additive metadata approach is the narrowest maintainable solution because it keeps Telegram-specific provenance in the Telegram/plugin hook surface instead of adding cross-channel top-level SDK fields.
