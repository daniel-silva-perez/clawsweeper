## Summary

Keep this issue open. Current main still has the reported Telegram hook metadata gap: Telegram derives reply and forward provenance for prompt/session context, but the canonical message hook metadata sent to internal and plugin hooks only includes generic routing/sender fields, and there is no current via_bot/viaBot plumbing path.

## What This Changes

The issue requests additive Telegram provenance metadata for forwarded, inline-bot, and reply-parent messages on MessageReceivedHookEvent.context.metadata.

## Best Possible Solution

Add a small Telegram-owned provenance payload under hook metadata, using existing reply/forward helpers where possible, adding via-bot extraction, capping reply text, and covering both the central dispatch hook path and the Telegram mention-skip ingest hook path with focused tests plus docs/changelog updates.

Yes. A targeted test can build or dispatch a Telegram inbound message containing forward_origin, via_bot, and reply_to_message and assert that current main emits message:received metadata without forward, viaBot, or replyTo.

## Solution Assessment
