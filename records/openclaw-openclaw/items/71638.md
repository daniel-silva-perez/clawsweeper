## Summary

Keep this issue open. Current main still has the right additive hook metadata surface, but the canonical received-message context and both received-hook mappers only emit generic routing/sender metadata; Telegram forward, inline-bot, and reply provenance still cannot reach `message_received` or `message:received` consumers.

## What This Changes

The issue requests additive Telegram `MessageReceivedHookEvent.context.metadata` entries for forwarded-message, inline-bot, and reply-to provenance.

## Best Possible Solution

Add a Telegram-owned provenance normalizer, pass generic provider metadata through the canonical inbound hook context, and merge it into both plugin and internal received-hook metadata. Keep the core seam provider-agnostic, avoid new Bot API calls, cap copied reply text, and cover both normal dispatch and mention-skip ingest paths with focused tests and docs.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a Telegram message with `forward_origin`, `via_bot`, or `reply_to_message` can reach the received-hook path, but current canonical context and mapper output have no field that can carry those values.

## Solution Assessment

Yes. The existing `metadata` escape hatch is the narrow maintainable solution; the safer implementation is a generic metadata passthrough plus a Telegram-local normalizer, not Telegram-specific top-level fields in core hook types.
