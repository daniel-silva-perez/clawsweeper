## Summary

Keep open. Current main still rejects the requested trusted-internal deployment shape: `gateway.auth.mode: "none"` is accepted as an auth mode, but non-loopback Gateway startup is blocked unless shared-secret auth or trusted-proxy auth is configured. The related PR that tried to allow this was closed unmerged, and the later RFC is broader design discussion rather than a shipped or narrower canonical replacement.

## Best Possible Solution

Keep this issue open for gateway/auth product work. The likely path is a narrowly scoped private-ingress/no-auth mode or explicit opt-in that updates `src/gateway/server-runtime-config.ts`, `src/cli/gateway-cli/run.ts`, focused runtime/CLI tests, security-audit warnings, and gateway docs. The broader #69066 service-identity RFC can inform internal-caller design, but it should not hide this separate deployment request.

## Risks / Open Questions

- Allowing non-loopback `auth.mode=none` would intentionally permit unauthenticated Gateway access on a network interface, so any implementation needs an explicit dangerous opt-in, strong startup warnings, docs/security-audit alignment, and preserved Control UI origin protections.
- Docs contain some private-ingress `none` references for HTTP APIs while current startup policy blocks non-loopback no-auth binds; fixing this should reconcile behavior and documentation together.
