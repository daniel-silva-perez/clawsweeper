## Summary

Keep open. Current main still has a Telegram preview-finalization path that can deliver final text by editing the preview message directly, while the `message_sending` hook is only wired through the normal `deliverReplies` send path; the related Discord and Slack hook PRs are closed/unmerged context, not a current-main Telegram fix.

## What This Changes

The issue reports that Telegram preview streaming final delivery bypasses `message_sending` plugin hooks when `channels.telegram.streaming` is enabled.

## Best Possible Solution

Telegram preview final delivery should share the same outbound hook contract as normal final delivery: run `message_sending` before any visible final preview edit, honor content rewrites and cancellation, and keep the existing preview/fallback duplicate-safety behavior intact.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the text-only preview-final path edits the preview and avoids `sendPayload`, while `message_sending` is only invoked inside `deliverReplies`.

## Solution Assessment

Yes. The narrowest maintainable solution is Telegram-owned: inject or wrap the same `message_sending` hook application around final preview edits, rather than adding a broad core migration or changing unrelated channel behavior.

## Repair Work Prompt

Fix the Telegram streaming `message_sending` hook bypass reported in https://github.com/openclaw/openclaw/issues/61550. On current main, final preview delivery can edit an existing Telegram preview through `createLaneTextDeliverer`/`editPreview` without going through `deliverReplies`, where `runMessageSending` currently lives. Keep the fix Telegram-owned and core-agnostic. Ensure final preview edits run the same `message_sending` contract before a visible final edit: pass the same useful routing context as normal Telegram delivery (`channelId: "telegram"`, accountId, conversation/chat id, thread id, reply target where available), apply rewritten content to the edited final text, and honor `{ cancel: true }` by skipping edit/send. Preserve existing duplicate-avoidance, long-lived-preview fresh-final fallback, archived-preview cleanup, and Telegram edit-error behavior. Add focused regression tests for an existing preview finalizing by edit: hook fires once, rewrite edits modified text, cancel suppresses edit/send, and the normal `deliverReplies` hook path still passes. Add a changelog fix entry if repo policy requires a user-facing bug-fix entry, crediting the reporter as `Thanks @Postmanplayx`; do not change unrelated channels or broad plugin hook semantics.
