## Summary

Keep open. Current main wires `message_sending` into Telegram `deliverReplies`, but the Telegram preview-stream finalization path can still edit or materialize the preview and return `preview-finalized` before `sendPayload`/`deliverReplies` runs. The existing tests even assert that `deliverReplies` is not called when a final answer stays in the streamed preview.

## Best Possible Solution

Keep this issue open and fix the Telegram preview-stream path explicitly. The likely implementation should run `message_sending` before preview finalization/edit/materialization, honor content rewrites and cancellation, avoid double-running hooks when falling back through `deliverReplies`, and add a regression test for `streaming: partial` final edits.

## Risks / Open Questions

- A naive fix can double-run `message_sending` on fallback paths because `deliverReplies` already invokes the hook.
- A final-only fix may still leave earlier preview edits visible before policy hooks run; maintainers should decide whether `message_sending` covers every streaming edit or only finalization.
