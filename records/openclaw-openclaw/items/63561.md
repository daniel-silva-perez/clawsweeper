## Summary

Current main still matches the reported false-unavailable path: systemctl user calls inherit the process environment unchanged, Permission denied bus failures are classified as systemd unavailable, and the status/install surfaces still print or throw that unavailable state. The exact env-derivation fix described in the related PR is not present on main.

## Best Possible Solution

Keep this issue open and implement the missing user-systemd spawn recovery in `src/daemon/systemd.ts`: preserve existing env, derive `XDG_RUNTIME_DIR` and `DBUS_SESSION_BUS_ADDRESS` for Linux user-scope `systemctl` calls when absent, pass that env into all `systemctl --user` spawns, and add focused regression coverage for `gateway status` and `gateway install` with a healthy user service but broken inherited bus env. Re-evaluate the narrower `not found` classifier from #63625 separately if unit-level not-found false positives remain in scope.

## Risks / Open Questions

- Deriving DBus env must preserve already-correct caller values and stay Linux-only so it does not regress sudo-to-root machine-scope handling or non-Linux service adapters.
- A raw `Permission denied` can be a genuine access-control failure on some hosts, so the fix should recover only missing/dropped bus env cases and keep clear diagnostics when access is actually denied.
- No live Ubuntu service reproduction was run because this review is read-only and restarting/installing user services would mutate host state.
