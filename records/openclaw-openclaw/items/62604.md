## Summary

Keep open. Current main has improved the surrounding attachment and send/draft paths, but the reported base64-text paste failure is not fully covered: the Control UI paste handler only intercepts clipboard items whose MIME type starts with image/ and otherwise lets the textarea default paste proceed, so a text/plain or text/html data:image/...;base64 payload can still land in the composer. I also found no chat paste regression test for this case.

## Best Possible Solution

Keep the issue open and implement a narrow Control UI fix in ui/src/ui/views/chat.ts: detect pasted text/plain and text/html data:image/...;base64 payloads, either convert supported image data URLs into ChatAttachment entries or prevent/reject them with a clear non-invasive UI response. Add regression coverage in ui/src/ui/views/chat.test.ts for image/* clipboard items, data:image text payloads, and ordinary text paste. Leave failed-send draft/attachment restoration in ui/src/ui/app-chat.ts, where current main already centralizes it, and separately document that hotpatching hashed dist Control UI bundles is unsupported if maintainers want to address that part of the report.

## Risks / Open Questions

- Closing this as implemented would likely drop a remaining paste-input bug for clipboard payloads that expose the image as text/plain or text/html data URLs rather than image/* files.
- The side-payload-store mitigation is present on current main, but the provided latest release is v2026.4.26 and that tag still embedded attachment data URLs directly in chat state, so shipped users may not have that part yet.
- The source-map/dist traceability concern remains a product/docs decision; current packaging still excludes dist source maps.
