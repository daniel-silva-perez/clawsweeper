## Summary

Keep open. Current main has the image-file paste happy path, but the reported `data:image/...;base64,...` text-paste case is not handled, and the linked current report #74960 shows the image paste/send pipeline is still failing in v2026.4.27 rather than being safely implemented.

## What This Changes

The issue asks to harden Control UI image paste so large inline data-image payloads become attachments or are rejected, keep failed-send draft restoration centralized in the send pipeline, and clarify installed hashed Control UI dist hotpatch/debug expectations.

## Best Possible Solution

Add focused Control UI and Gateway regression coverage for pasted image data: reject or convert `data:image/...;base64,...` text/html/plain paste payloads without inserting them into the composer, prove pasted image bytes survive through the payload store and `chat.send`, and document that local patching of installed hashed Control UI assets is unsupported unless maintainers decide to change package debug artifacts.

## Reproduction Assessment

Yes. A focused DOM test can paste a `text/plain` or `text/html` `data:image/...;base64,...` clipboard payload into the textarea and confirm current `handlePaste` does not prevent default; #74960 also gives current macOS Chrome steps for the related wrong-image send path.

## Solution Assessment

Yes, the best fix direction is narrow: keep draft restoration in `app-chat.ts`, harden the paste handler and serialization tests, then only touch Gateway attachment parsing if a byte-identity regression proves the failure crosses that boundary. Shipping source maps should be a separate maintainer decision, not bundled into the paste fix by default.
