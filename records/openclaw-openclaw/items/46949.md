## Summary

Keep open. Current main still has the max-concurrent ACP runtime-cache behavior this PR is trying to fix, and the initial Greptile race/stat feedback was addressed in the PR discussion, but the latest patch still has blocking lifecycle-accounting defects around status-probe timeouts and eviction revalidation under pressure.

## What This Changes

The PR updates the ACP session manager to close cached oneshot runtimes after startup identity reconciliation, reclaim dormant or stale runtime cache entries before max-concurrent-session rejection, and add ACP manager regression tests.

## Best Possible Solution

Repair or replace this PR on current main while keeping the scope limited to ACP runtime lifecycle accounting: release dormant oneshot handles, enforce hard status-probe timeouts, recheck capacity before and during pressure eviction, recompute blind idleness inside the actor lock, and cover those cases with focused manager tests.

## Reproduction Assessment

Yes. Current main has a focused manager-level reproduction path by filling `acp.maxConcurrentSessions` and then opening another ACP session; the PR defects can be reproduced with a `getStatus` promise that ignores `signal` and with cache capacity or idleness changing while stale eviction waits in the actor queue.

## Solution Assessment

No, not as written. The design direction is maintainable, but the patch needs hard timeout and eviction revalidation guards so reclaiming a slot cannot hang session initialization or evict usable context unnecessarily.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Add a hard timeout around status probes:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This awaits `cached.runtime.getStatus(...)` directly and relies on the adapter honoring the abort signal. Since the ACP runtime contract only passes `signal` as optional input, an adapter that ignores it can leave session initialization stuck forever whenever capacity pressure triggers this path. Race the probe against a timer so max-concurrency enforcement always progresses.
  - confidence: 0.88
- **[P2] Recheck capacity after dormant oneshot eviction:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After dormant oneshot eviction frees a slot, this still runs stale-pressure eviction before checking the cache size. A single new session can therefore evict an additional idle runtime even though pressure was already relieved, unnecessarily dropping session context.
  - confidence: 0.86
- **[P2] Stop stale eviction once capacity is relieved:** `src/acp/control-plane/manager.core.ts:1297-1298`
  - body: The actor-queue callback can run after another session has already closed and brought the cache below the configured limit. Without a size check immediately before clear/close, this path can evict a usable idle session even though the new session no longer needs a slot.
  - confidence: 0.81
- **[P2] Recompute blind idleness inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-`getStatus` fallback decides staleness from `candidate.idleMs`, which was captured before waiting for `actorQueue.run`. If another actor-scoped operation touched the runtime while this callback was queued, the PR can still evict it based on stale idle time.
  - confidence: 0.8
