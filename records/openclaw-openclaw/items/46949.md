## Summary

Keep open: current main still has the reported ACP capacity/counting behavior, and this PR is a real implementation candidate, but the PR head still has blocking lifecycle bugs in the new pressure-eviction path and is dirty against current main.

## What This Changes

The PR makes ACP session-limit enforcement asynchronous, releases cached oneshot runtime handles after startup identity reconciliation, evicts dormant oneshot and stale ACP runtimes under pressure, and adds ACP manager regression tests.

## Best Possible Solution

Repair or replace the PR on current main while keeping the scope limited to ACP runtime lifecycle accounting: release dormant oneshot handles, add a hard status-probe timeout, recheck capacity before each pressure eviction, recompute blind idleness inside the actor lock, and cover those cases with focused manager tests.

## Reproduction Assessment

Yes. A focused ACP manager test can reproduce the current-main behavior by filling `acp.maxConcurrentSessions` and observing the max-session rejection; the PR defects can be reproduced with a `getStatus` promise that ignores `signal` and with capacity/idleness changes while stale-pressure eviction is queued.

## Solution Assessment

No, the PR is not the best fix as written. The direction is right, but the maintainable solution needs hard timeouts plus capacity and idleness revalidation at the point of eviction so reclaiming a slot does not hang or discard usable session context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Add a hard timeout around status probes:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This awaits `cached.runtime.getStatus(...)` directly and relies on the adapter honoring the abort signal. Since the ACP runtime contract only passes `signal` as optional input, an adapter that ignores it can leave session initialization stuck forever whenever capacity pressure triggers this path. Race the probe against a timer so max-concurrency enforcement always progresses.
  - confidence: 0.88
- **[P2] Recheck capacity before probing stale sessions:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After dormant oneshot eviction frees a slot, this still runs stale-pressure eviction before checking the cache size. A single new session can therefore evict an additional idle runtime even though pressure was already relieved, unnecessarily dropping session context.
  - confidence: 0.86
- **[P2] Stop stale eviction once capacity is relieved:** `src/acp/control-plane/manager.core.ts:1297`
  - body: The actor-queue callback can run after another session has already closed and brought the cache below the configured limit. Without a size check immediately before clear/close, this path can evict a usable idle session even though the new session no longer needs a slot.
  - confidence: 0.81
- **[P2] Recompute blind idleness inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-`getStatus` fallback decides staleness from `candidate.idleMs`, which was captured before waiting for `actorQueue.run`. If another actor-scoped operation touched the runtime while this callback was queued, the PR can still evict it based on stale idle time.
  - confidence: 0.8
