## Summary

Keep this PR open. Current main still lacks the pressure-release behavior the PR targets, but the PR head remains unmerged, dirty against base, and has concrete ACP lifecycle correctness issues around stale-probe timeouts and unnecessary eviction under pressure.

## What This Changes

The PR changes the ACP session manager to release oneshot runtime handles after startup identity reconciliation, evict non-running oneshot and stale idle runtimes under max-concurrency pressure, and add ACP control-plane regression tests.

## Best Possible Solution

Rebase or replace the PR on current main, keep the fix scoped to ACP runtime lifecycle accounting, add a hard timeout around pressure status probes, recheck capacity and current idleness inside guarded eviction paths, and keep focused regression coverage for startup oneshot release and pressure reclaim behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Add a hard timeout around status probes:** `src/acp/control-plane/manager.core.ts:1282`
  - body: This awaits `cached.runtime.getStatus(...)` directly and relies on the adapter honoring the AbortSignal. The runtime contract only makes `signal` optional, so an adapter that ignores it can leave session initialization stuck forever whenever capacity pressure triggers this probe. Race the status promise against a timer so max-concurrency enforcement always progresses.
  - confidence: 0.88
- **[P2] Recheck capacity before stale eviction:** `src/acp/control-plane/manager.core.ts:1105`
  - body: After dormant oneshot eviction frees a slot, this still runs the stale-pressure path before checking the cache size. A single new session can therefore evict an additional idle/stale runtime even though pressure was already relieved, unnecessarily dropping session context. Recheck capacity after the oneshot phase and skip stale probing once below the limit.
  - confidence: 0.86
- **[P2] Recompute blind idleness inside the actor lock:** `src/acp/control-plane/manager.core.ts:1288-1289`
  - body: The no-getStatus fallback decides staleness from `candidate.idleMs`, which was captured before waiting for `actorQueue.run`. If another actor-scoped operation touched the runtime while this callback was queued, the PR can still evict it based on stale idle time. Re-read the current cache timestamp inside the actor callback before blind eviction.
  - confidence: 0.8
