## Summary

Keep this PR open. Current main still lacks the PR's intended task-level lane timeout or an equivalent automatic recovery path for never-settling active queue tasks, and the same-author linked bug remains open with this PR as an implementation candidate. The PR should be rebased and reworked before merge because its current fallback timeout can conflict with longer or disabled caller-level timeout policies and can allow overlapping work after a timed-out promise keeps running.

## Best Possible Solution

Keep this PR open for maintainer review and rework. The best path is to rebase it onto current main, implement queue-level recovery or an equivalent health mechanism in `src/process/command-queue.ts`, preserve existing agent and cron timeout policies when deriving any fallback timeout, define cancellation and overlap behavior for timed-out tasks, and keep regression coverage for lane unblock, generation reset handling, disabled/custom timeout behavior, and diagnostics. Once a corrected implementation lands, close #48488 as implemented.

## Risks / Open Questions

- The PR's `Promise.race` approach can unblock a lane while the underlying task continues running, so later work may overlap with still-mutating session or cron state unless cancellation/overlap semantics are explicit.
- A fixed queue-level default timeout can falsely fail legitimate long-running agent or cron work unless it is derived from caller-level budgets, including disabled timeouts.
- Closing this PR now would discard a focused implementation candidate while the linked lane-queue liveness bug remains open and current main still lacks automatic recovery for never-settling active lane tasks.
