## Summary

Keep this PR open. Current main still does not implement a task-level timeout or equivalent automatic recovery in the command lane queue, and the same-author linked bug remains open with this PR as the implementation candidate. The PR is not ready to close as obsolete or implemented, but it needs maintainer review and likely rework before landing.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rework before landing: rebase onto current main, implement queue-level recovery or an equivalent health mechanism in `src/process/command-queue.ts`, preserve caller timeout budgets when deriving any lane fallback timeout, document and test zombie-task/cancellation behavior, and keep regression coverage for lane unblock, reset generation handling, disabled/custom timeout behavior, and diagnostics. Once a corrected implementation lands, close #48488 as implemented.

## Risks / Open Questions

- Closing this PR now would discard a relevant implementation candidate while the paired lane-queue liveness bug remains open and current main still lacks automatic recovery for never-settling tasks.
- The PR's `Promise.race` approach can unblock a lane while the original task keeps running, so the final design needs explicit cancellation or overlap semantics to avoid concurrent mutation of shared session or cron state.
- A fixed default queue timeout could reject legitimate long-running agent or cron work unless it preserves caller-specific timeout budgets, including disabled or long-duration runs.
