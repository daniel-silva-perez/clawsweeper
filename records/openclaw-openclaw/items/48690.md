## Summary

Keep this PR open. Current main still does not implement the PR's intended task-level lane timeout or an equivalent automatic recovery in the command queue, and the same-author linked bug remains open with this PR as the implementation candidate.

## Best Possible Solution

Keep this PR open for maintainer review and rework. The best path is to rebase onto current main, implement queue-level recovery or an equivalent health mechanism in `src/process/command-queue.ts`, preserve caller timeout policies from agent and cron surfaces, define cancellation/overlap behavior for timed-out tasks, and keep regression coverage for lane unblock, reset generation handling, disabled/custom timeout behavior, and diagnostics. Once a corrected implementation lands, close #48488 as implemented.

## Risks / Open Questions

- The proposed `Promise.race` approach can unblock a lane while the original task keeps running, so the landing design needs explicit zombie-task/cancellation/overlap semantics for shared session and cron state.
- A fixed queue-level timeout can falsely fail legitimate long-running agent or cron work unless it preserves caller-specific timeout budgets, including disabled and long-duration runs.
- Closing this PR now would discard a relevant implementation candidate while the linked lane-queue liveness bug remains open and current main still lacks automatic recovery for never-settling active lane tasks.
