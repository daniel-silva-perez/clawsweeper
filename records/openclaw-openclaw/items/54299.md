## Summary

Keep open. Current main still has the reported Google/Gemini CLI OAuth behavior: an environment client ID can be used without a matching client secret, the token exchange only sends client_secret when present, and the redirect URI/listener are still fixed to localhost:8085. Existing manual fallback mitigates a busy callback port but does not make the redirect port configurable or document/test the fixed-port contract as intentional.

## Best Possible Solution

Keep this issue open for the bundled Google plugin owner. Make the Gemini CLI OAuth client-secret contract explicit in code, docs, and tests: either reject or warn when an env client ID is supplied without the matching secret if confidential-client mode is intended, or document and test public-client mode as intentional. Separately, add supported redirect URI/port configuration if compatible with Google OAuth registration, or document and regression-test fixed localhost:8085 plus manual paste fallback as intended behavior.

## Risks / Open Questions

- The issue's attacker-impact framing is stronger than the source alone proves because this OAuth flow uses PKCE and changing env vars generally implies local environment control; the concrete product problem is the ambiguous, untested client-secret contract.
- A dynamic redirect port may be constrained by Google OAuth redirect URI registration, so the eventual fix may need explicit redirect URI/port configuration or a documented, regression-tested fixed-port plus manual-fallback policy.
