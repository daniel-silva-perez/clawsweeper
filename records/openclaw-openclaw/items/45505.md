## Summary

Keep open. Current main has the earlier narrow late-compaction repair and newer token snapshot plumbing, but it still does not provide the generalized authoritative reconciliation boundary requested here; count, token freshness, session rotation, status, memory flush, and auth-profile side effects still converge through separate timing paths.

## What This Changes

Request to generalize post-timeout compaction completion reconciliation so all compaction-derived session state converges after a late successful compaction, not only the persisted compaction counter.

## Best Possible Solution

Define one idempotent compaction-success reconciliation boundary that owns durable compaction-derived session updates, then route count, post-compaction token freshness, rotated transcript identity, and dependent status/memory/auth state through it with late-success and duplicate-signal regression coverage.

## Reproduction Assessment

Yes. A focused code/test reproduction can simulate a successful compaction_end after the run finalizer has moved on and assert that current main's late helper only repairs compactionCount while richer session/status-derived state does not converge through the same path.

## Solution Assessment

No. The current narrow count-only repair is not the full solution requested; the maintainable direction is to promote the existing completion hook into a monotonic reconciliation helper/event rather than adding more scattered finalizer writes.
