## Summary

Keep open. Current main still has only the narrow late-success `compactionCount` repair plus adjacent token snapshot plumbing; it does not yet provide the generalized authoritative reconciliation boundary requested here for token freshness, rotated session identity, status-visible state, memory-flush state, and auth-profile side effects.

## What This Changes

The issue requests a generalized post-timeout compaction completion reconciliation path so all compaction-derived session state converges after a late successful compaction, not only the persisted compaction counter.

## Best Possible Solution

Define one idempotent compaction-success reconciliation boundary that owns durable compaction-derived session updates, then deliberately route count, post-compaction token freshness, rotated transcript identity, and dependent status/memory/auth state through it with late-success and duplicate-signal regression coverage.

## Reproduction Assessment

Yes. A focused code/test reproduction can simulate a successful `compaction_end` after run finalization has moved on and assert that current main's late helper repairs only `compactionCount` while richer session/status-derived state does not converge through the same path.

## Solution Assessment

No. The current narrow count-only repair is not the full solution requested; the maintainable direction is to promote the existing completion hook into a monotonic reconciliation helper/event rather than adding more scattered finalizer writes.
