## Summary

Keep open. Current main still has the narrow late-success `compactionCount` repair from the predecessor work, but it has not been generalized into an authoritative reconciliation boundary for token freshness, rotated session identity, status-visible context state, memory flush gating, or auth-profile state.

## What This Changes

The issue requests a generalized post-timeout compaction completion reconciliation model across durable session state, status/UI token snapshots, memory/auth-derived state, and related completion semantics.

## Best Possible Solution

Define one monotonic compaction-success reconciliation boundary that owns durable compaction-derived session updates, then route count, post-compaction token freshness, rotated session identity, and dependent status/memory/auth state through it with explicit late-success and duplicate-signal coverage.

## Reproduction Assessment

Yes. A focused test can simulate a successful compaction_end after run finalization has already moved on and assert that current main repairs compactionCount while richer token/session/status-derived state does not converge through the late path.

## Solution Assessment

No. The current count-only repair was the right narrow predecessor, but the maintainable full solution is a single idempotent reconciliation helper or event rather than more scattered finalizer writes.
