## Summary

Keep this issue open. Current main still uses a single shared gateway token for token-mode auth, while identity-bearing access remains limited to trusted-proxy or Tailscale Serve flows; the requested per-token identity and Control UI session attribution are not implemented.

## What This Changes

Add multiple gateway auth tokens with per-token label/scope metadata so multiple users can access one Control UI with isolated, attributable sessions while preserving local CLI and device-pairing behavior.

## Best Possible Solution

Define a backwards-compatible core gateway auth contract for multiple token entries, resolve the matched token into a server-authenticated identity, propagate that identity through HTTP/WebSocket auth and Control UI sender/session metadata, preserve existing single-token, password, trusted-proxy, Tailscale, and device-token behavior, and document/test the security semantics before shipping.

## Security Review

Status: needs_attention

Summary: No patch is under review, but this issue is auth-sensitive and needs explicit security design before implementation.

Concerns:

- **[medium] Multi-token matching must not leak secret state:**
  - body: A future token map needs constant-time comparison, redaction, non-enumerating errors, and clear rate-limit accounting so attackers cannot learn which token labels exist or which secret failed.
  - confidence: 0.86
- **[medium] Token labels and scopes become authorization data:**
  - body: If labels/scopes flow into sessions or operator permissions, the server must treat them as authenticated identity, not browser-local display metadata, and define their interaction with existing device and trusted-proxy scopes.
  - confidence: 0.82
