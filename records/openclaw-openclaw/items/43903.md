## Summary

Keep open. Current main still models gateway token auth as one shared secret, while trusted-proxy and Tailscale are the only identity-bearing gateway auth paths; the browser-local Control UI identity improves attribution but does not satisfy per-token credentials, scopes, or token-derived session isolation. The team-deployment comment makes this a concrete security/product gap rather than stale cleanup.

## What This Changes

Support multiple gateway auth tokens with per-token labels and scopes so shared Control UI access can map users to isolated identities and sessions.

## Best Possible Solution

Define a backwards-compatible core gateway auth contract for multiple token entries, resolve the matched entry into an authenticated identity with explicit scope semantics, propagate that identity where Control UI and session metadata need it, preserve the existing shared-secret and identity-bearing modes, and document/test the security model.
