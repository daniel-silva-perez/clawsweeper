## Summary

Keep open. Current main still models token-mode gateway auth as one shared secret and does not resolve token entries into server-authenticated labels, scopes, or principals. The existing Control UI personal identity work is browser-local attribution, not multi-user token isolation, and the related open gateway-auth/scope/RBAC items are adjacent rather than canonical fixes.

## What This Changes

Support multiple gateway tokens mapped to per-user identity and scope metadata so shared Control UI access can isolate users and sessions.

## Best Possible Solution

Define a backwards-compatible gateway auth contract for multiple shared-token entries, resolve the matched entry into a server-attested principal with explicit scope semantics, preserve the current single-token behavior, and propagate identity only through documented Control UI, session, and author surfaces after maintainer security review.

## Reproduction Assessment

Not applicable as a feature request. Source verification is high-confidence: config types/schema, resolved auth, token authorization, and the WebSocket auth schema all lack a multi-token map or token-principal metadata on current main.

## Solution Assessment

Unclear. The proposed tokens map is plausible, but the best solution needs maintainer/security design and coordination with RBAC, author-attribution, shared-auth scopes, and service-identity work before implementation.
