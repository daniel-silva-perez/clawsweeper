## Summary

Keep this issue open. Current main still models gateway token auth as one shared secret with no per-token identity, while the identity-bearing Control UI paths remain Tailscale Serve and trusted-proxy; the team-use comment in the discussion confirms this is an active product gap rather than a stale one.

## What This Changes

The issue requests a backwards-compatible `gateway.auth.tokens` map that assigns labels/scopes to multiple gateway tokens so Control UI users can get isolated, attributable sessions without switching to trusted-proxy.

## Best Possible Solution

Define and document a backwards-compatible core gateway auth contract for multiple token entries, resolve the matched token into a server-authenticated identity, propagate that identity through HTTP/WebSocket auth and Control UI sender/session metadata, preserve existing single-token/password/device/trusted-proxy/Tailscale behavior, and cover the security semantics with focused tests before shipping.

## Security Review

Status: needs_attention

Summary: No patch is under review, but this is a security-sensitive gateway authentication change that needs explicit design review before implementation.

Concerns:

- **[medium] Multi-token matching must not leak secret state:** `src/gateway/auth.ts:342`
  - body: A token map would need constant-time comparison, redaction, non-enumerating errors, and clear rate-limit accounting so attackers cannot infer which token labels exist or which secret failed.
  - confidence: 0.88
- **[medium] Token labels and scopes become authorization data:** `src/gateway/auth-resolve.ts:17`
  - body: If token labels/scopes flow into sessions or permissions, the server must treat them as authenticated identity and define their interaction with device tokens, trusted-proxy users, Tailscale identity, and shared-secret HTTP scopes.
  - confidence: 0.84
