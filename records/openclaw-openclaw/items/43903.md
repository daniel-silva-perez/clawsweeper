## Summary

Keep open. Current main still models token auth as a single shared secret and the implemented Control UI personalization is browser-local, while the issue discussion gives a concrete multi-user deployment need; the remaining work is security-sensitive gateway-auth design, not stale cleanup.

## What This Changes

Add support for multiple gateway token credentials with per-token labels/scopes so shared Control UI deployments can isolate users and session metadata without trusted-proxy setup.

## Best Possible Solution

Define a backwards-compatible core gateway auth contract for multiple token entries, resolve the matched entry into an authenticated identity with explicit scope semantics, propagate that identity only to the Control UI/session metadata surfaces that need it, preserve existing single-token behavior, and document/test the security model.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high-confidence: the strict gateway auth config and WebSocket auth schemas have no `tokens` map or token identity fields, and token authorization checks exactly one configured secret.

## Solution Assessment

Unclear. The proposed `tokens` map is a plausible product direction, but the best implementation needs maintainer/security design for principal resolution and scope authority before a code PR; browser-local identity from #70362 is not sufficient.
