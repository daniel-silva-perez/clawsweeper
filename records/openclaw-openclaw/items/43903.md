## Summary

Keep this open. Current main still has one shared gateway token with no token-entry map, no per-token identity, and no token-derived session metadata; the discussion also confirms a real team deployment gap, so this is not stale cleanup.

## What This Changes

The issue requests multiple `gateway.auth` token entries with labels/scopes so several Control UI users can share one Gateway while retaining isolated identities and sessions.

## Best Possible Solution

Define a backwards-compatible core gateway auth contract for multiple token entries, resolve the matched entry into an authenticated identity, propagate that identity through Control UI sender/session metadata, preserve existing single-token/password/device/trusted-proxy/Tailscale behavior, and cover the security semantics with focused tests and docs.

## Risks / Open Questions

- A token-map implementation needs constant-time matching, redaction, non-enumerating failures, and clear rate-limit accounting so attackers cannot infer which token labels exist.
- Token labels and scopes would become server-authenticated authorization data, so their interaction with device tokens, trusted-proxy users, Tailscale identity, HTTP auth, and WebSocket auth needs an explicit design before implementation.
