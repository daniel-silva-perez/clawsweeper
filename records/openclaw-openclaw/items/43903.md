## Summary

Keep open. Current main still implements token-mode gateway auth as one shared secret with no token-entry identity, labels, or scopes; the merged browser-local Control UI identity work helps display attribution locally but does not provide a server-authenticated per-token principal or multi-user session isolation. The issue discussion gives a concrete team deployment need, and related open work (#68353, #8081, #69066) shows the remaining design crosses gateway auth, author attribution, scopes, and security boundaries rather than being stale cleanup.

## What This Changes

Request support for multiple gateway auth tokens in token mode, with per-token labels/scopes so multiple Control UI users can be identified and isolated while retaining local CLI and device-pairing behavior.

## Best Possible Solution

Define an additive, backwards-compatible gateway auth contract for multiple token entries, resolve the matched entry into a server-authenticated principal with explicit scope semantics, preserve current single-token behavior, and propagate that identity only through documented Control UI/session/author surfaces after the trust model is settled.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high-confidence: config types/schema, resolved auth, token authorization, and the WebSocket auth schema all lack a multi-token map or token-principal metadata.

## Solution Assessment

Unclear. The proposed tokens map is a plausible direction, but the best implementation needs maintainer/security design and should coordinate with open author-attribution and RBAC discussions before code changes.

## Risks / Open Questions

- A multiple-token implementation could accidentally reveal token-entry existence through timing, error text, logging, or rate-limit behavior if matching is not uniform and redacted.
- Token labels/scopes need a clear authority model before they are used as session metadata, chat author attribution, or authorization inputs.
- The request overlaps with open #68353 and #8081; maintainers should decide how token-derived identity composes with per-message author metadata and broader RBAC rather than implementing conflicting identity surfaces.
