## Summary

Keep this issue open. Current main still does not support multiple gateway auth tokens with per-token label/scope identity, and the discussion describes a real team deployment gap rather than a stale or plugin-owned request.

## Best Possible Solution

Keep this open for a deliberate core gateway change: add a backwards-compatible multi-token auth contract with SecretInput-backed token entries and non-secret label/scopes metadata, resolve the matched token into HTTP/WebSocket auth identity, propagate that identity into Control UI sender/session metadata, preserve existing single-token/password/trusted-proxy/Tailscale/device-pairing behavior, and update config schema/help/docs plus gateway auth/session tests.

## Risks / Open Questions

- This is security-sensitive work: a future design needs SecretInput/SecretRef support, redaction, constant-time multi-secret matching, non-enumerating auth errors, and clear rate-limit semantics across multiple tokens.
- Per-token labels/scopes should be server-resolved auth identity, not browser-supplied display metadata, or it will not satisfy the requested session isolation and attribution model.
