## Summary

Keep this issue open. Current main still does not implement multiple gateway auth tokens with per-token label/scope identity. The issue discussion describes a real team-use gap, and the current implementation remains a single shared token/password model with identity-bearing alternatives limited to trusted-proxy or Tailscale Serve paths.

## Best Possible Solution

Keep this open for a deliberate core gateway change: add a backwards-compatible multi-token auth contract with SecretInput-backed token entries and non-secret label/scopes metadata, resolve the matched token into HTTP/WebSocket auth identity, propagate that identity into Control UI sender/session metadata, preserve existing single-token, password, trusted-proxy, Tailscale, and device-pairing behavior, and update config schema/help/docs plus gateway auth/session tests.

## Risks / Open Questions

- A future implementation is security-sensitive: it needs SecretInput/SecretRef support, redaction, constant-time matching, non-enumerating errors, and clear rate-limit semantics across multiple secrets.
- Token-backed labels/scopes must not be confused with browser-local display identity; only server-resolved auth identity would satisfy the requested isolation and attribution model.
