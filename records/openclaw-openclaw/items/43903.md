## Summary

Keep open. Current main still has single shared-token gateway auth with no token-entry map, no server-resolved token principal, and no authenticated label/scope metadata that could isolate Control UI users. The reporter and follow-up comment describe a real team-deployment need; the existing browser-local identity and `session.dmScope` controls help attribution/routing but are not a server-authenticated per-token user boundary.

## What This Changes

The issue requests token-mode gateway auth support for multiple configured tokens, each mapped to labels/scopes, so Control UI users can keep local CLI/device pairing workflows while getting isolated per-user sessions.

## Best Possible Solution

Define a backwards-compatible gateway auth contract for multiple shared-token entries, resolve the matched entry into a server-attested principal with explicit scope semantics, preserve existing single-token behavior, and propagate identity only through documented Control UI/session/author surfaces after maintainer security review.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path: config types/schema, resolved auth, token authorization, and the WebSocket handshake contract all lack multi-token entries or token-principal metadata on current main.

## Solution Assessment

Unclear. The proposed `tokens` map is plausible, but the best solution needs maintainer/security design across shared-token auth, trusted-proxy identity, device tokens, RBAC/scopes, and session attribution before coding.
