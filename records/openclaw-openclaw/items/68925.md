## Summary

Keep this PR open. Current main still lacks the intended approval UI relabeling, queue-removal/overview-refresh behavior, and durable approval-resolution logging. The PR discussion indicates the earlier `@state()` and `loadOverview({ refresh })` regressions were addressed, but the remaining event-log-buffer concern is still relevant before merge.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to preserve the focused approval UI polish if maintainers want it, but align the returned-id behavior with the actual approval RPC contract, write any synthetic resolution event through the canonical event buffer or rely on the gateway resolved broadcast, and add a targeted UI/state test for the resolution flow before merge.
