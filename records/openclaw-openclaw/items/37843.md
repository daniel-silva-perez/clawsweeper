## Summary

Keep open. Current main still treats any trimmed non-empty cron `payload.model` as an explicit model reference, so `model: "default"` is sent through normal model resolution instead of falling through to the agent/default model. The same non-empty check also changes fallback policy, CLI add/edit can persist the literal value, docs describe `--model` only as a concrete allowed model, and the related PRs were closed unmerged.

## Best Possible Solution

Keep this issue open and implement a small shared cron payload-model helper that trims `payload.model` and treats exact `default` as unset. Use it in both `resolveCronModelSelection` and `resolveCronFallbacksOverride`, add focused tests for default/empty/explicit/invalid/fallback behavior, and update cron CLI help/docs if `--model default` is public behavior.

## Repair Work Prompt

Implement cron support for `payload.model: "default"` / `--model default` as a runtime-default sentinel. Add a small shared helper in the cron isolated-agent area that reads an agentTurn payload model, trims it, and returns unset for empty/whitespace or exact lowercase `default`; preserve all other strings unchanged. Use that helper in `resolveCronModelSelection` so `default` falls through to stored session or agent/default selection exactly like an unset payload model, and in `resolveCronFallbacksOverride` so `default` does not count as a cron payload model override. Add focused tests for default/whitespace/empty/explicit/invalid/fallback cases, including regression coverage that `model: "default"` does not force an empty fallback override. Update cron CLI help and docs if `--model default` is public behavior. Do not change provider catalogs, generic model parsing, or unrelated cron scheduling behavior. Add a user-facing changelog fix entry if required by current repo policy, crediting @ramsaycute and @Sid-Qin rather than automation.
