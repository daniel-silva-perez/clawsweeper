## Summary

Keep this issue open. Current main still treats trimmed cron `payload.model: "default"` as a non-empty explicit model override, and the fallback policy also treats it as an override. The related fix PRs in the provided context were closed unmerged, and current docs/tests do not show this sentinel as shipped behavior.

## Best Possible Solution

Implement a small shared cron payload-model helper that trims `payload.model` and treats exact `default` as unset. Use it in both `resolveCronModelSelection` and `resolveCronFallbacksOverride`, add focused tests for default/empty/explicit/invalid/fallback behavior, and update cron CLI help/docs if `--model default` is intended to be public.
