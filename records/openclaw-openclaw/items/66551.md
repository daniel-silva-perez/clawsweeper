## Summary

Keep open. Current main still rejects the HTML/XML/CSS host-read case, but the latest PR head retains a security-sensitive bypass where XML-sniffed content can be accepted through the buffer-verified document allowlist without full text validation.

## What This Changes

The branch expands the host-read media guard to allow validated HTML, XML, and CSS local attachment documents, adds media/outbound regression tests, and updates the changelog.

## Best Possible Solution

Keep the validated-text approach, but route every text-like document MIME without reliable full-buffer verification through isValidatedHostReadText before allowing host reads. In practice, remove the new HTML/XML/CSS/XML entries, especially application/xml, from the buffer-verified document allowlist and add a regression for extensionless or .txt XML-prologue content with a binary tail.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep XML out of the buffer-verified allowlist:** `src/media/web-media.ts:106`
  - body: Adding `application/xml` to `HOST_READ_ALLOWED_DOCUMENT_MIMES` lets host reads with a `.txt` or extensionless filename return through the sniffed-document branch before `isValidatedHostReadText` runs. `file-type@22.0.1` sniffs `application/xml` from a `<?xml ` prefix, so an XML prologue plus binary tail can bypass the full-buffer text validation this PR relies on.
  - confidence: 0.9
