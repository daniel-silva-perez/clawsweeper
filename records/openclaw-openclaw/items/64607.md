## Summary

Keep open. Current main has real partial Control UI media support, but the umbrella request is not complete: remote markdown images are still intentionally flattened and blocked by CSP, automatic web/link previews are absent, direct tool-result image rendering remains tracked separately, and generated/raw audio still has inconsistent UI-normalization paths.

## What This Changes

The issue requests inline display in chat for generated or received images, audio, video, and web page previews instead of requiring external open commands.

## Best Possible Solution

Keep this as the umbrella unless maintainers split it. Preserve the shipped MEDIA directive, attachment player, and managed generated-image paths, then finish the remaining slices through narrower work: decide the remote-image policy for #40945/#41025, implement web/link previews with SSRF/CSP/privacy constraints, finish direct tool-result image rendering for #50779/#52025, and normalize generated audio/video into durable UI-renderable media references that survive history reloads.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a single feature umbrella, but yes for key remaining gaps: current tests assert remote markdown images flatten to alt text, CSP blocks remote images, and source search found no automatic link-preview implementation.

## Solution Assessment

Is this the best way to solve the issue? No single implementation path is best for the whole umbrella. The maintainable path is to keep this open for tracking while landing smaller, security-reviewed PRs for remote images, link previews, direct tool-result images, and durable audio/video media references.
