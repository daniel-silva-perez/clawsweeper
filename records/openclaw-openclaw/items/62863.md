## Summary

Keep open. Current main has partial edit-tool hardening, but it does not fully implement the requested four-part improvement: broad whitespace-collapsing fallback, explicit preflight mismatch reporting before the edit attempt, `read line=`/`keyword=` positioning, and docs guidance for repeated very small edits remain materially incomplete.

## Best Possible Solution

Keep this issue open as a focused enhancement. The safest path is to extend the edit pipeline with an unambiguous whitespace-tolerant fallback only when it can preserve original offsets, keep the existing exact/fuzzy behavior as the first path, add or explicitly reject `read line=`/`keyword=` semantics, and update public tool docs with the recommended edit workflow.

## Risks / Open Questions

- A broad whitespace-collapsing fallback can misapply edits unless it proves a unique match and can map normalized offsets back to original content without damaging formatting.
- Adding `read keyword=` may duplicate existing grep/find workflows unless the product design clearly defines when read should search versus when agents should use other discovery tools.
