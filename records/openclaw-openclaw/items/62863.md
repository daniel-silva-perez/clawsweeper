## Summary

Current main still only partially addresses the request. OpenClaw delegates edit/read semantics to the pinned pi-coding-agent package and wraps edit failures with recovery/mismatch hints, while upstream 0.70.6 adds line-ending normalization and a narrow fuzzy matcher; the broader whitespace-collapsing edit fallback, explicit preflight contract, read keyword positioning, and requested small-edit guidance remain unimplemented. The related open #67276/#66915 path is narrower and targets user-facing mismatch messaging rather than the full behavior requested here.

## What This Changes

Request broad edit/read stability improvements: whitespace-tolerant edit matching, preflight validation, read line/keyword positioning, and documentation recommending small edit chunks.

## Best Possible Solution

Keep the issue open until maintainers define the edit/read tool contract. A maintainable end state would implement whitespace-tolerant matching only with uniqueness and offset-mapping safeguards, decide whether read should grow line/keyword positioning, and update public tool guidance if the recommended edit workflow changes.

## Reproduction Assessment

Yes for the remaining gaps: source inspection shows an edit oldText that differs by collapsed ASCII spaces/tabs or blank-line differences will still miss unless it falls within the narrow upstream fuzzy set, and the read schema exposes only path/offset/limit. I did not run mutating reproduction tests because this review was read-only, but the dependency and wrapper code provide a high-confidence static reproduction path.

## Solution Assessment

Unclear: the request identifies real pain, but broad whitespace normalization is not obviously the safest fix without a uniqueness/offset-mapping design. The safer path is a product/API decision followed by an upstream or wrapper implementation that preserves duplicate detection, preview, and diff correctness.
