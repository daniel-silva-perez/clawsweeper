## Summary

Keep open. The PR addresses a real Firecrawl self-hosting bug, but it is not merge-ready because the trusted-vs-strict fetch path is chosen from a static hostname predicate, so HTTPS self-hosted names that resolve to RFC1918 addresses can still hit the strict DNS SSRF block. The network-policy change is also security-sensitive and overlaps other open Firecrawl endpoint PRs, so maintainers should choose one canonical shape.

## What This Changes

The PR relaxes Firecrawl baseUrl validation, allows custom HTTPS and private/local HTTP endpoints, routes private/local Firecrawl API calls through the trusted web-tools fetch guard, and expands endpoint plus SecretRef tests.

## Best Possible Solution

Land one canonical Firecrawl self-hosting fix that keeps the hosted default strict, makes self-hosted trust explicit, allows HTTP only for DNS-verified private or loopback targets, routes trusted private endpoints through the appropriate web-tools guard, preserves SecretRef coverage, updates Firecrawl/web_fetch docs and manifest/config schema as needed, and then resolves #63877 plus the overlapping PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve private DNS before choosing strict mode:** `extensions/firecrawl/src/firecrawl-client.ts:154-156`
  - body: The branch chooses trusted vs. strict fetching from a static hostname/IP suffix check. A configured HTTPS endpoint such as `https://firecrawl.internal.example` can resolve to RFC1918 addresses without matching this predicate, so it still falls through to `withStrictWebToolsEndpoint` and is rejected by the DNS SSRF guard before fetch. Reuse a DNS-backed classifier or explicit trusted-host policy before selecting the guard.
  - confidence: 0.89
