## Summary

Keep this PR open. Current main still blocks the self-hosted Firecrawl use case from #63877, and this branch is a plausible fix, but the patch is not merge-ready because its trusted-vs-strict fetch routing still misses private-network HTTPS hostnames that resolve through DNS and it changes an SSRF-sensitive path that needs maintainer review.

## What This Changes

The PR relaxes Firecrawl baseUrl validation, permits HTTP for private/local hosts, routes private/local Firecrawl endpoints through the trusted web-tools guard, and expands Firecrawl endpoint and SecretRef tests.

## Best Possible Solution

Land a single canonical Firecrawl self-hosting fix that keeps public endpoints on HTTPS and strict SSRF protection, allows HTTP only for verified private or loopback targets, routes private endpoints through the trusted guard only after DNS-backed validation or an explicit trusted-host policy, preserves SecretRef tests, updates Firecrawl/web_fetch docs, and then closes #63877 while resolving the overlapping PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve private DNS before choosing strict mode:** `extensions/firecrawl/src/firecrawl-client.ts:154-156`
  - body: The branch chooses trusted vs. strict fetching from a static hostname/IP suffix check. A self-hosted HTTPS endpoint such as `https://firecrawl.internal.example` can resolve to RFC1918 addresses without matching this predicate, so it falls through to `withStrictWebToolsEndpoint` and is rejected by the DNS SSRF guard before fetch. Reuse a DNS-backed private-network classifier or explicit trusted-host policy before selecting the guard.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The patch changes Firecrawl SSRF guard selection and should use a stronger private-network validation or explicit trust policy before merge.

Concerns:

- **[medium] Static private-host routing weakens SSRF reviewability:** `extensions/firecrawl/src/firecrawl-client.ts:154`
  - body: `withTrustedWebToolsEndpoint` is selected from a local string predicate rather than the DNS-backed private-network validation already exposed through the SDK. That is a security-sensitive bypass of strict resolved-address checks for matched local-looking hosts and should be reviewed or replaced with a clearer trusted endpoint policy.
