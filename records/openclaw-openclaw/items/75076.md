## Summary

Keep this PR open. Current main does not contain the Control UI provenance/checksum/readback-hardening work, and the PR is a concrete implementation, but the current head still introduces a false health-warning path for channels whose connection state is unknown rather than explicitly disconnected.

## What This Changes

The PR adds Control UI auth/CSP/media hardening, bootstrap build provenance and checksum artifacts, Slack/channel readback health fields and warnings, docs/scripts/workflow updates, and related browser/unit tests.

## Best Possible Solution

Keep this PR as the reviewable implementation, but revise the generic channel health summary so missing connected remains unknown and warnings are emitted only for explicit disconnected states or channel/account types that declare connection-state support. Preserve the Slack-specific mode-aware warnings and hardening work, then rerun the targeted Control UI, Slack, Gateway, script, and changed-gate validation before merge.

## Reproduction Assessment

Yes. Construct a ChannelsStatusSnapshot with a configured/running passive or probed channel whose channel/account status omits connected; PR head's resolveChannelHealthSummary emits a no-active-connection/readback warning even though current main treats that state as unknown, not disconnected.

## Solution Assessment

No. The hardening direction is sound, but this PR is not mergeable as-is because the generic warning collapses unknown connection state into failure; the safer fix is to gate that warning on connected === false or an explicit connection-state capability/mode.
