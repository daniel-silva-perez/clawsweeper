## Summary

Keep this PR open. Current main does not already contain the proposed Control UI provenance/checksum/status hardening, and the latest PR head still has a concrete P2 regression where the new generic channel health summary treats missing connection state as a disconnected/readback failure.

## What This Changes

This PR hardens Control UI bootstrap, media, CSP, build provenance, checksum artifacts, Slack/channel health status, related docs, scripts, workflows, and targeted tests.

## Best Possible Solution

Keep the PR as the reviewable implementation, but revise the generic channel health summary so it only warns on explicit disconnected states or on channel/account types known to publish connection state. Preserve the Slack-specific mode-aware warning and the Control UI hardening work, then rerun the targeted Control UI, Slack, Gateway, script, and changed-gate validation before merge.

## Reproduction Assessment

Yes. Construct a ChannelsStatusSnapshot with a configured and running channel whose channel/account status omits connected, such as a passive/probed channel summary; resolveChannelHealthSummary will emit the no-active-connection/readback warning even though the connection state is unknown.

## Solution Assessment

No. The hardening direction is reasonable, but the current PR head is not the best mergeable solution until unknown connection state is distinguished from explicit disconnected state; the safer fix is to gate the warning on connected === false or a declared connection-state capability/mode.
