## Summary

Keep this PR open. The hardening direction is useful and current main does not already contain the requested Control UI provenance/checksum/readback work, but the PR head still introduces a concrete false-positive channel health warning by treating an omitted `connected` field as a reported disconnection.

## What This Changes

The PR adds Control UI auth/CSP/media hardening, bootstrap build provenance, checksum artifacts in CI/release workflows, Slack/channel readback health status fields and warnings, docs/scripts, and targeted browser/unit coverage.

## Best Possible Solution

Keep the PR as the reviewable implementation, but change the generic channel health summary so it warns only on explicit disconnection or on channel/account types that declare connection-state support. Preserve the Slack-specific mode-aware warning and the Control UI hardening work, then rerun the targeted UI, Slack, Gateway, script, and changed-gate validation before merge.

## Reproduction Assessment

Yes. Construct a `ChannelsStatusSnapshot` with a configured and running passive/probed channel whose channel/account status omits `connected`; the PR head's `resolveChannelHealthSummary` emits a no-active-connection/readback warning even though the state is unknown, not disconnected.

## Solution Assessment

No. The hardening direction is reasonable, but this PR is not the best mergeable solution until unknown connection state is distinguished from explicit disconnection; the safer fix is to gate the generic warning on `connected === false` or a declared connection-state capability/mode.
