## Summary

Keep this draft PR open. The current head is not implemented on main and it addresses the earlier CSP/proxy and Slack card mode-awareness findings, but the newly added generic channel health summary still treats an omitted `connected` field as a broken connection, which will create false warnings for healthy passive or HTTP-mode channel accounts.

## What This Changes

The PR hardens Control UI bootstrap/media/CSP handling, adds build provenance and checksum artifacts, expands Slack/channel health and readback status surfaces, updates CI/release workflows, and adds targeted tests and docs.

## Best Possible Solution

Keep the PR as the reviewable implementation, but revise the generic channel health summary so it only warns on explicit disconnected states or on channel/account types known to publish connection state. Preserve the CSP/proxy and Slack card fixes, then run the targeted Control UI, Slack, Gateway, script, and changed-gate validation before maintainer review.

## Reproduction Assessment

Yes for the blocking finding: construct a `ChannelsStatusSnapshot` with a configured and running channel whose status/accounts omit `connected`, such as Slack HTTP mode or a passive channel summary, and `resolveChannelHealthSummary` will emit the no-active-connection warning because it treats unknown as false.

## Solution Assessment

No. The hardening direction is reasonable, but the current PR head is not the best mergeable solution until the generic health summary distinguishes explicit disconnects from unsupported or mode-specific connection state.

## Review Findings

Gate this on `connected === false` or a channel/account capability or mode that actually supports connection state.
