## Summary

Keep open. Current main still has the reported write-before-validation behavior for `plugins enable` and `plugins disable`, and the issue is paired with open PR #73554 using closing syntax, so maintainers should review, land, or replace that PR before closing this issue.

## What This Changes

Request that `openclaw plugins enable/disable <id>` reject nonexistent plugin IDs before writing `plugins.entries` and exit non-zero without mutating config.

## Best Possible Solution

Add a pre-write lookup in the `plugins enable` and `plugins disable` command handlers using the discovered plugin snapshot, resolve valid aliases to the canonical plugin id, and exit 1 with a clear `Plugin not found` message before any config write or registry refresh on a miss.

## Reproduction Assessment

Yes. I did not run the CLI because this review is read-only, but the current source path directly reproduces the reported behavior: both commands pass the raw id into config mutation helpers, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The narrow maintainable fix is to validate the operator-supplied id against the discovered plugin registry before mutation; existing stale-entry cleanup should remain a separate doctor/fix concern unless maintainers choose to expand scope.
