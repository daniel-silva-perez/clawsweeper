## Summary

Keep open. Current main still has the write-before-validation path for `plugins enable` and `plugins disable`, and #73554 is an open, unmerged implementation candidate that already targets this issue.

## What This Changes

The issue requests that `plugins enable <id>` and `plugins disable <id>` reject nonexistent plugin ids with a non-zero exit before mutating `plugins.entries` config.

## Best Possible Solution

Add a pre-write lookup in the `plugins enable` and `plugins disable` command handlers using the discovered plugin snapshot, resolve valid aliases to the canonical plugin id, and exit 1 with a clear `Plugin not found` message before config writes or registry refresh on a miss. Land #73554 if it satisfies that boundary and coverage; otherwise replace it with an equivalent narrow PR.

## Reproduction Assessment

Yes. I did not run the mutating CLI because this review is read-only, but current source directly shows both command handlers reach config mutation before any registry lookup, and the shared helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The narrow maintainable fix is the requested pre-write registry validation in the CLI command handlers; stale-entry repair for configs already damaged by older versions should stay in doctor/fix unless maintainers choose a broader follow-up.
