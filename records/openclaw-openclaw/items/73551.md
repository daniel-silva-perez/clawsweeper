## Summary

Keep open. Current main still writes plugin policy config before proving the requested enable/disable target exists, and two open PRs (#73554 and #73623) already reference this issue with closing syntax. Those PRs are implementation candidates, not a reason to close before one is reviewed and merged.

## Best Possible Solution

Keep this issue open and review/land one canonical fix from #73554 or #73623. The fix should resolve the requested id against the discovered plugin snapshot before `enablePluginInConfig` or `setPluginEnabledInConfig`, exit 1 with a clear `Plugin not found` message on a miss, avoid config writes and registry refreshes on failure, and add regression tests for both enable and disable.
