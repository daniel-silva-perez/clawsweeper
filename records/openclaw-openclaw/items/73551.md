## What This Changes

The issue requests that `plugins enable <id>` and `plugins disable <id>` reject nonexistent plugin ids before mutating `plugins.entries` config.

## Best Possible Solution

Add a pre-write lookup in the `plugins enable` and `plugins disable` command handlers using the discovered plugin snapshot, resolve valid aliases to the canonical plugin id, and exit 1 with a clear `Plugin not found` message before config writes or registry refresh on a miss. Land #73554 if it satisfies that boundary and coverage, otherwise replace it with an equivalent narrow PR.

## Reproduction Assessment

Yes. I did not run the CLI because this review is read-only, but the current source path directly reproduces the reported behavior: both commands pass the raw id to config mutation helpers, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The narrow maintainable solution is to validate the operator-supplied id against the discovered plugin registry before mutation; existing stale-entry cleanup should remain a separate doctor/fix concern unless maintainers choose to expand scope.
