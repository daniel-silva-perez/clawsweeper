## Summary

Current main still has the reported plugin policy mutation bug, and two open PRs (#73554 and #73623) already reference this issue as their implementation target. The issue should stay open until maintainers choose and land one canonical fix.

## What This Changes

`plugins enable` and `plugins disable` should reject nonexistent plugin ids before writing `plugins.entries.*`, exit non-zero, and avoid stale config mutations.

## Best Possible Solution

Keep this issue open and review/land one canonical fix from #73554 or #73623. The fix should resolve the requested id against the discovered plugin snapshot before `enablePluginInConfig` or `setPluginEnabledInConfig`, exit 1 with a clear `Plugin not found` message on misses, skip config writes and registry refreshes on failure, and add regression tests for both enable and disable.
