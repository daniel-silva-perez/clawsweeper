## Summary

This issue should stay open. Current main still mutates `plugins.entries.<id>` for arbitrary `plugins enable` and `plugins disable` arguments before any discovered-plugin validation, and PR #73554 is an open unmerged implementation candidate that explicitly closes this issue.

## Best Possible Solution

Keep this issue open and review PR #73554 as the direct implementation candidate. The desired fix is to resolve the requested plugin id or display name against the discovered plugin snapshot before any enable/disable config write, exit non-zero with a clear `Plugin not found` error on a miss, skip config writes and registry refreshes on failure, and add regression tests for both commands.
