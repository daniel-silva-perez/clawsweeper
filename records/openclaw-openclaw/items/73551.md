## Summary

Keep open. Current main still mutates plugin config from the raw requested id before any discovered-plugin lookup, so the reported stale-entry bug remains; an open linked implementation PR, #73554, should be reviewed or replaced before this issue is closed.

## What This Changes

The issue requests that `plugins enable` and `plugins disable` reject nonexistent plugin ids before writing `plugins.entries` config.

## Best Possible Solution

Add a pre-write lookup in the `plugins enable` and `plugins disable` handlers using the discovered plugin snapshot used by list/inspect, resolve the requested id to the canonical plugin id when valid, and on a miss exit non-zero with a clear `Plugin not found` message before config writes, slot selection, or registry refresh. Cover both commands with policy tests that assert no write or refresh occurs for an unknown id.

## Reproduction Assessment

Yes. The issue's command-line repro is supported by the current source path: enable/disable write through `enablePluginInConfig` or `setPluginEnabledInConfig` before any registry lookup, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: validate the operator-supplied plugin id against the discovered registry before mutating config. The open linked PR should be reviewed for alias preservation and no-write test coverage before landing.
