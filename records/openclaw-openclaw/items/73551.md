## Summary

Keep open. Current main still mutates plugin config from the raw requested id before any discovered-plugin lookup, so the reported stale-entry bug remains; the open linked implementation PR #73554 should be reviewed and landed or explicitly replaced before this issue is closed.

## What This Changes

The issue requests that `plugins enable` and `plugins disable` reject nonexistent plugin ids before writing `plugins.entries` config.

## Best Possible Solution

Review and land #73554 or an equivalent focused patch that validates the requested plugin id against the discovered registry before any enable/disable config mutation, with no-write/no-refresh regression tests; keep cleanup of historical stale entries as a separate doctor/fix concern.

## Reproduction Assessment

Yes. The reporter's command-line repro is supported by current source: enable/disable write through `enablePluginInConfig` or `setPluginEnabledInConfig` before any registry lookup, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The requested pre-write registry validation is the narrow maintainable fix; it belongs at the CLI mutation boundary where registry context exists, not inside the low-level config toggle helper.
