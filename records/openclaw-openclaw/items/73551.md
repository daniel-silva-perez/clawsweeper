## Summary

Keep open. Current main still has the reported write path: `plugins enable` and `plugins disable` mutate `plugins.entries.<id>` for arbitrary IDs before checking the discovered plugin registry, and the later validation path only warns about the stale entry. Related stale-plugin cleanup work does not supersede this pre-write validation bug.

## Best Possible Solution

Keep this issue open and fix the CLI policy path. Add a shared resolver that checks the requested plugin id or display name against `buildPluginSnapshotReport` before calling `enablePluginInConfig` or `setPluginEnabledInConfig`; on a miss, print a clear `Plugin not found` error, exit 1, and skip config writes and registry refresh. Add regression tests for missing IDs on both enable and disable, plus coverage that valid bundled aliases still work.
