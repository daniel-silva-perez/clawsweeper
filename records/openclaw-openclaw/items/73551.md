## Summary

Keep open. Current main still mutates plugin config from the raw `plugins enable` / `plugins disable` argument before any discovered-plugin lookup, and the linked implementation PR remains open rather than merged.

## What This Changes

The issue asks for `openclaw plugins enable <id>` and `plugins disable <id>` to reject nonexistent plugin ids before writing stale `plugins.entries.<id>` config.

## Best Possible Solution

Add a pre-write registry/snapshot lookup in `plugins enable` and `plugins disable`, resolve valid inputs to the canonical plugin id, and on a miss exit non-zero with a clear `Plugin not found` message before any config write, slot update, or registry refresh. Cover both commands with policy tests that assert the write and refresh seams are not called for unknown ids, then review and land #73554 or a narrow replacement.

## Reproduction Assessment

Yes. The report’s CLI reproduction is source-supported on current main: enable/disable mutate config through `enablePluginInConfig` or `setPluginEnabledInConfig` before any discovered-plugin lookup, and the toggle helper creates entries for arbitrary ids.

## Solution Assessment

Yes. The requested validate-before-write behavior is the narrow maintainable fix; the active PR should be reviewed for id/alias semantics and no-write test coverage before this issue is closed.
