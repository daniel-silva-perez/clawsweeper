## Summary

Keep open. Current main still has the reported write-before-validation behavior for `plugins enable` and `plugins disable`, and the issue is already paired with open PR #73554 using closing syntax, so it should stay open until that PR or a replacement lands.

## What This Changes

The issue requests that `openclaw plugins enable/disable <nonexistent-id>` reject unknown plugin ids before writing stale `plugins.entries.<id>` config.

## Best Possible Solution

Add a pre-write lookup in the `plugins enable` and `plugins disable` handlers using the discovered plugin snapshot used by list/inspect, resolve valid requests to the canonical plugin id, and on a miss exit non-zero with a clear `Plugin not found` message before config writes, slot selection, or registry refresh. Keep cleanup of already-stale entries as a separate doctor/fix concern.

## Reproduction Assessment

Yes. The issue's CLI reproduction is deterministic from current source: enable/disable mutate config from the raw requested id before any discovered-plugin lookup, and the shared helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The requested pre-write registry validation is the narrow maintainable fix; the best next step is to review and land PR #73554 or replace it with equivalent no-write coverage if that branch is not suitable.
