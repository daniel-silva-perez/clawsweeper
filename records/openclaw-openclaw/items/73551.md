## Summary

Keep open. Current main still mutates plugin config before validating the requested enable/disable id against the discovered plugin snapshot, and the two related open PRs (#73554 and #73623) are implementation candidates that reference this issue with closing syntax. The issue should stay open until one canonical fix is reviewed and merged, or both PRs are closed with a different maintainer decision.

## Best Possible Solution

Review and land one canonical fix from #73554 or #73623, then let the merged PR close this issue. The fix should resolve the requested id against the discovered plugin snapshot before `enablePluginInConfig` or `setPluginEnabledInConfig`, exit non-zero with a clear `Plugin not found` message on a miss, skip config writes and registry refreshes on failure, and add regression tests for both enable and disable.
