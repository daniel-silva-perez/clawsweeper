## Summary

Current main still does not restore Chrome Extension Relay support. The browser plugin keeps the relay startup helper as an intentional no-op, config and profile creation still reject `driver: "extension"`, docs and doctor migration point old configs to Chrome MCP `existing-session`, and related open issues do not fully supersede the requested extension-specific relay behavior.

## Best Possible Solution

Keep this open for browser-plugin maintainer review. The clean direction is either an explicit decision to restore and harden an optional Chrome Extension Relay driver, or a documented browser-driver/plugin API that lets an external relay plugin implement the old extension-specific capabilities, with migration docs explaining where Chrome MCP `existing-session` is not equivalent.

## Risks / Open Questions

- Restoring the feature would reintroduce a privileged Chrome debugger/CDP relay, WebSocket authentication/origin handling, packaged extension assets, and existing-browser control, so it needs explicit browser and security maintainer judgment.
- Chrome MCP `existing-session` is a partial migration path, but current docs list limitations and it does not provide the requested extension-specific auto-reattach, video-control, chat-automation, or integrated port-18792 relay behavior.
- A ClawHub/plugin route may be the right long-term shape, but current main exposes a closed browser driver set and no simple third-party browser-driver registration seam.
