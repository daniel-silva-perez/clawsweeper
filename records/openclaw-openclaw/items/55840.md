## Summary

This issue should stay open. Current main still treats the Chrome Extension Relay as removed, rejects `driver: "extension"`, migrates old configs to Chrome MCP `existing-session`, and documents Chrome MCP as the replacement path rather than restoring the integrated relay on port 18792 or the extension-specific debugger/auto-reattach behavior requested here.

## Best Possible Solution

Keep this issue open for browser-plugin maintainer follow-up. The best product path is either to intentionally restore and harden an optional Chrome Extension Relay driver in the bundled browser plugin, or to define a documented browser-driver/plugin API so an extension relay can live outside core with clear migration docs explaining where Chrome MCP `existing-session` is not equivalent.

## Risks / Open Questions

- Closing as implemented would conflate Chrome MCP `existing-session` migration with the requested Chrome Extension Relay restoration and its extension-specific debugger, video-control, chat-automation, and auto-reattach behavior.
- Restoring this path would reintroduce privileged existing-browser control, Chrome debugger/CDP authority, relay authentication, and packaged extension assets, so it needs explicit browser/security maintainer judgment rather than automated cleanup.
- A ClawHub/plugin answer may be viable only as a replacement browser plugin or after a narrower browser-driver API is designed; current main does not show a simple public driver-registration seam for this request.
