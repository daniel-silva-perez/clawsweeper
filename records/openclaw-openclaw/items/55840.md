## Summary

This issue should stay open. Current main still intentionally removes the Chrome Extension Relay path and migrates legacy `driver: "extension"` configs to Chrome MCP `existing-session`; that is a migration/replacement path, not the requested restoration of the integrated extension relay, bundled extension implementation, port 18792 relay, or extension-specific debugger and auto-reattach behavior.

## Best Possible Solution

Keep this issue open for browser plugin maintainer follow-up. The concrete paths are either to intentionally restore and harden an optional extension-relay driver in the bundled browser plugin, or to define and document a narrow browser-driver/plugin API so an extension relay implementation can live outside core with clear migration docs explaining where Chrome MCP `existing-session` is not equivalent.

## Risks / Open Questions

- Closing as implemented would conflate the Chrome MCP `existing-session` migration with the requested Chrome Extension Relay restoration and its extension-specific behavior.
- Restoring this path would reintroduce privileged browser control, Chrome debugger/CDP authority, relay authentication, and packaged extension assets, so it needs explicit browser/security maintainer judgment.
- A ClawHub/plugin answer may be viable only as a replacement browser plugin today, or after a narrower browser-driver API is designed; current main does not show a simple public driver-registration seam.
