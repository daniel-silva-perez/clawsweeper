## Summary

Current main still does not restore Chrome Extension Relay support. The browser plugin keeps the relay startup helper as an intentional no-op, config/schema/profile creation still exclude and reject `driver: "extension"`, doctor/docs migrate old extension config to Chrome MCP `existing-session`, and the remaining related item #19289 is a broader open plugin/API design issue rather than an implementation that supersedes this request.

## Best Possible Solution

Keep this open for browser-plugin maintainer review. The best path is either an explicit product/security decision to restore and harden an optional Chrome Extension Relay driver, or a documented browser-driver/plugin API that lets an external relay plugin implement the old extension-specific capabilities, with migration docs explaining where Chrome MCP `existing-session` is not equivalent.

## Risks / Open Questions

- Restoring the feature would reintroduce a privileged Chrome debugger/CDP relay, WebSocket auth/origin handling, packaged extension assets, and existing-browser control, so it needs explicit browser and security maintainer judgment.
- Chrome MCP `existing-session` is a partial migration path, but current docs list limitations and it does not provide the requested extension-specific integrated port-18792 relay, auto-reattach, video-control, or chat-automation behavior.
- Closing to ClawHub would be premature while current main lacks a documented browser driver registration seam for an external plugin to add an `extension` profile driver to the bundled browser tool.
