## Summary

Current main still does not restore Chrome Extension Relay support. The browser plugin keeps the old relay lifecycle as an intentional no-op, config/schema/profile creation reject `driver: "extension"`, doctor/docs migrate legacy configs to Chrome MCP `existing-session`, bundled extension assets are gone except icons, and the current plugin SDK does not expose an in-place browser driver registration seam. This should stay open for maintainer product and security review rather than being closed as implemented or ClawHub-only.

## Best Possible Solution

Keep this open for browser-plugin maintainer product/security review. The best path is either an explicit, opt-in, hardened restoration of the Chrome Extension Relay driver with config/schema/docs/tests/live smoke coverage, or a stable browser backend/plugin API that lets external relay implementations such as the ClawHosters relay integrate without private internals, plus migration docs that state where Chrome MCP `existing-session` is not equivalent.

## Risks / Open Questions

- Restoring this as-is would reintroduce a privileged Chrome extension/debugger relay plus WebSocket authentication, origin validation, token handling, and remote-browser-control surfaces that need explicit security review.
- Chrome MCP `existing-session` covers some signed-in-browser use cases, but it does not appear equivalent to the old extension-specific debugger/video-control/auto-reattach/chat-automation workflow reported here.
- Closing as ClawHub-only would be premature because current main does not provide a documented in-place browser profile driver/transport API for an external relay plugin; a full replacement browser plugin is not the same migration path.
