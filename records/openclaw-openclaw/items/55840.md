## Summary

Keep this issue open. Current main intentionally keeps the Chrome extension relay removed, rejects `driver: "extension"`, and migrates legacy configs to Chrome MCP `existing-session`, but that does not restore the requested extension-specific capabilities. The related #19289 browser-backend plugin interface is a relevant prerequisite/demand signal, not a complete replacement for this extension-relay request.

## What This Changes

Restore the removed Chrome Extension Relay as an optional browser driver, or provide an equivalent migration path and documentation for users relying on extension-specific browser control.

## Best Possible Solution

Keep this open for maintainer product and security review. The best path is to decide whether legacy Chrome Extension Relay should stay retired, be restored as a maintained optional browser backend, or be enabled through an additive browser-backend plugin API tracked by #19289, with clear docs for users migrating from 2026.3.13-era configs.

## Risks / Open Questions

- Restoring the legacy relay is security-sensitive: it would reintroduce a browser-control WebSocket path, Chrome extension assets, token/auth handling, and `chrome.debugger`-style control over signed-in browser sessions.
- The current `existing-session` migration is only a partial substitute for the reporter's requested behavior; current code documents several `existing-session` action/snapshot/download limitations and does not bring back the old extension driver.
- Related #19289 may be the right prerequisite for a third-party or ClawHub implementation, but closing this issue into that generic API request would lose the extension-relay-specific demand signal and migration concerns.
