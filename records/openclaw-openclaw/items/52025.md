## Summary

Keep this PR open. Current main still has adjacent managed-media and assistant-image support, but it does not implement this PR’s intended behavior for direct tool-result image blocks shaped as `{ type: "image", data, mimeType }`, and the default tool-output UI still keeps message images inside a collapsed disclosure.

## Best Possible Solution

Keep this PR open as a focused implementation candidate for #50779, but require an updated patch against current main. The best fix should cover direct and nested tool-result `{ type: "image", data, mimeType }` blocks, image-only results, accompanying text/captions, default visibility outside collapsed tool details, and live/history behavior, preferably by projecting trusted tool images into bounded managed media references with grouped-render regression coverage.

## Risks / Open Questions

- A renderer-only extraction change can still miss live tool-stream messages and history reloads because those paths text-flatten or size-bound tool content.
- Rendering raw base64 tool media inline needs explicit MIME, SVG/data-URL, and size safeguards so a UI fix does not bypass the managed media controls.
- The PR’s current test surface, as shown in the supplied diff, includes `ui/src/ui/views/chat.test.ts`, which mocks the grouped renderer and is not enough by itself to prove this renderer behavior.
