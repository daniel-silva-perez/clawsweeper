## Summary

Keep this PR open. Current main has a partial adjacent change: `sessions.list` now loads the gateway model catalog, but session row context-window resolution still does not use that catalog and `loadGatewayModelCatalog` still has no context-cache side effect. The PR remains a plausible implementation candidate for the linked Anthropic/Claude Control UI meter bug in #67530, but the unresolved review concerns around cache key collisions, blanket 200k fallback, and test reset behavior still need maintainer handling before merge.

## Best Possible Solution

Keep this PR open for maintainer review or supersede it with a focused implementation that resolves session row `contextTokens` from precise gateway catalog or persisted runtime metadata, preserves provider alias normalization, avoids synthetic cache-key collisions for slash-containing model IDs, does not default unknown windows to 200k, and adds regression coverage plus cache reset hygiene.

## Risks / Open Questions

- The PR's `fallbackContextTokens: DEFAULT_CONTEXT_TOKENS` can make session rows show a confident but wrong percentage before catalog data is available for models whose real context window is not 200k.
- Writing both raw catalog IDs and synthetic provider-qualified IDs into `MODEL_CONTEXT_TOKEN_CACHE` can collide for slash-containing model IDs such as OpenRouter entries and underreport native provider windows through the conservative min merge.
- Adding gateway catalog writes to a global context-token cache needs a reset path through existing gateway/model test helpers so context windows do not leak between tests.
- Current main partially loads the catalog in `sessions.list`, so a final fix should account for that existing behavior instead of adding a second disconnected workaround.
