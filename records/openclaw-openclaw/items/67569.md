## Summary

Keep this PR open. Current main still lacks the PR’s intended cache-population behavior, and the session-listing path still uses synchronous context-window lookups that do not run gateway model discovery. The PR remains a plausible implementation candidate for the closed linked bug, but unresolved functional review concerns need maintainer handling before merge.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused revision that resolves context windows for session rows from precise gateway catalog or persisted runtime metadata, preserves provider alias normalization, avoids defaulting unknown windows to 200k, keeps slash-containing model IDs out of the synthetic qualified-key namespace, and resets any new global context-cache side effects in tests.

## Risks / Open Questions

- The PR’s unconditional `DEFAULT_CONTEXT_TOKENS` fallback can make the Control UI show a confident but wrong percentage before catalog data is available for models whose real context window is not 200k.
- Writing both bare IDs and synthetic provider-qualified IDs into `MODEL_CONTEXT_TOKEN_CACHE` needs a collision-safe design for slash-containing model IDs such as OpenRouter entries.
- Adding gateway catalog writes to the global context-token cache should reset that cache through the existing gateway/model test reset path so tests do not leak model windows across cases.
