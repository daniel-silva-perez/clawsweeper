## Summary

Keep this PR open. Current main still does not populate synchronous session-row context windows from the gateway model catalog, so the underlying Anthropic/Claude meter bug is not fully solved on main; however the PR branch still has blocking correctness concerns around default 200k fallbacks, cache-key collisions, and reset behavior.

## What This Changes

The PR adds gateway model-catalog context-window cache population with bare and provider-qualified keys, adds session-row fallback context tokens, and adds gateway catalog cache tests.

## Best Possible Solution

Revise the PR so session rows use precise catalog or persisted model metadata for context windows, preserve provider alias normalization, avoid synthetic cache-key collisions with slash-containing model IDs, leave unknown windows unknown instead of defaulting to 200k, and reset any new global cache side effects through the existing gateway/model test reset path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep raw slash IDs out of the qualified cache namespace:** `src/gateway/server-model-catalog.ts:33-42`
  - body: The new cache population writes both `entry.id` and `${normalizedProvider}/${entry.id}` into the same `MODEL_CONTEXT_TOKEN_CACHE`. For catalogs containing an OpenRouter raw ID like `google/gemini-2.5-pro` and a native Google `gemini-2.5-pro`, those two semantics share `google/gemini-2.5-pro`; the conservative merge can then underreport the OpenRouter model or native provider window in session rows.
  - confidence: 0.86
- **[P2] Do not default unknown session windows to 200k:** `src/gateway/session-utils.ts:1356-1360`
  - body: When `sessions.list` runs before catalog data has populated the cache, this fallback turns an unknown context window into `DEFAULT_CONTEXT_TOKENS`. That makes the UI meter deterministically wrong for discovery-only models with 128k, 1M, or other non-200k limits instead of withholding the percentage until a precise window is known.
  - confidence: 0.88
- **[P2] Reset the context-token cache with the catalog cache:** `src/gateway/server-model-catalog.ts:49-52`
  - body: `loadGatewayModelCatalog` now mutates the global context-token cache, but the gateway catalog reset helper still resets only the model catalog. Tests or same-process reload paths that reset the catalog can keep stale context-window entries from an earlier catalog load unless the new side effect is reset with it.
  - confidence: 0.78
