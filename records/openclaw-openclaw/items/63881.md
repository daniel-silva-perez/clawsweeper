## Summary

Keep this PR open. Current main still does not enforce a default dangerous-tool deny list on the plugin-tools MCP stdio surface; the open PR remains a plausible security hardening candidate, but it needs rebasing onto the current MCP handler split and should keep package release-age policy separate from the MCP authorization fix.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase it onto current main, put the dangerous-tool policy in the active plugin-tools MCP handler/server flow, split universal dangerous tools from HTTP-only exclusions, apply gateway.tools.allow and gateway.tools.deny consistently for listTools and callTool, preserve ownerOnly and before_tool_call behavior, add focused MCP security-boundary tests, update docs if gateway.tools now applies to MCP, and remove or split the axios release-age policy change.

## Risks / Open Questions

- The PR context targets an older MCP server shape; current main routes listTools/callTool through src/mcp/plugin-tools-handlers.ts, so the implementation needs a rebase onto that active path.
- The shared createPluginToolsMcpHandlers helper is also used by the openclaw-tools MCP server, so the final implementation should avoid accidentally changing selected core-tool MCP behavior unless that is intentional and tested.
- The PR context mixes MCP authorization with pnpm minimumReleaseAgeExclude / axios package policy; supply-chain-sensitive package resolution changes should be split or removed from the security-boundary patch.
- If gateway.tools.allow and gateway.tools.deny semantics are broadened from HTTP to MCP, docs and config/help wording should be updated so operators understand the new surface.
