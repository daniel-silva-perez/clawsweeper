## Summary

Keep this PR open. Current main still does not enforce a default dangerous-tool deny list on the plugin-tools MCP stdio path; it only has HTTP /tools/invoke default denial. The PR remains a plausible security hardening candidate, but it needs a rebase onto the current MCP handler split and should keep dependency release-age policy separate from the MCP authorization fix.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase it onto current main, put the dangerous-tool policy in the active MCP plugin-tools handler/server flow, split universal versus HTTP-only deny constants, apply gateway.tools.allow and gateway.tools.deny consistently for both listTools and callTool, preserve ownerOnly and before_tool_call behavior, add focused MCP security-boundary tests, update docs if gateway.tools now applies to MCP, and remove or split any remaining axios release-age policy change.

## Risks / Open Questions

- The core security concern remains on current main: an enabled plugin tool with a dangerous name can still be listed and invoked through the plugin-tools MCP server unless another policy path blocks it.
- The PR needs to be rebased onto the current createPluginToolsMcpHandlers/createToolsMcpServer structure so the fix lands on the active listTools and callTool path while preserving ownerOnly and before_tool_call behavior.
- If gateway.tools.allow and gateway.tools.deny semantics are broadened beyond HTTP, docs and focused tests should make that operator-facing behavior explicit.
- The PR context mixes MCP authorization with dependency release-age policy; that supply-chain-sensitive change should not be bundled with this security boundary fix.
