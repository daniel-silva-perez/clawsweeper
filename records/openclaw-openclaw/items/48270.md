## Summary

Keep this PR open. Current main has related timeout and Gateway RPC improvements, but it still does not implement this PR’s intended drift-based event-loop readiness preflight for `gateway probe`, and the probe path still opens the WebSocket directly after setup. The PR discussion includes useful Windows reproduction data and the later PR commits address the early unbounded-wait review concern, so the remaining question is product/implementation review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land a bounded, tested readiness helper shared by `probeGateway` and `callGateway`, or explicitly close the PR only after Windows/Node reproduction shows current main’s lazy-loading, probe-budget, and one-turn RPC-yield changes fully solve the `gateway probe` timeout without the drift-based preflight.
