## Summary

Keep this PR open. Current main has adjacent Gateway timeout work, but it still does not implement this PR's intended drift-based event-loop readiness preflight for `gateway probe`; the probe path opens the WebSocket directly after arming its own timer, while the existing one-turn yield only covers `callGateway`.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land a bounded, tested event-loop readiness helper for `probeGateway` and any affected Gateway RPC paths, or explicitly close the PR only after Windows/Node reproduction shows current main's budget and one-turn-yield changes fully solve the reported probe timeout without the drift-based preflight.
