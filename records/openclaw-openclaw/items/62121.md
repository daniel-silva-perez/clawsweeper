## Summary

Keep open. Current main has useful partial fixes for phase-tagged commentary, final-answer preference, and Telegram cron announce delivery, but it still has no provider-agnostic final-output boundary for plain untagged DeepSeek preamble. A maintainer already reviewed the report and explicitly left it open for a design pass rather than a marker/emoji/filter workaround.

## What This Changes

The issue asks OpenClaw to prevent untagged DeepSeek preamble text from being included in Telegram cron fallback delivery after the 2026.3.13 to 2026.4.5 upgrade.

## Best Possible Solution

Define a generic isolated-cron final-output contract that channels can trust without model IDs, emojis, or prose guessing. It should preserve existing phase-aware suppression, explain how unphased provider output becomes final-only, and add regression coverage for DeepSeek-style preamble before a formatted Telegram cron report.

## Reproduction Assessment

Yes, for triage: the reporter provided a concrete version range, model, channel, cron mode, and later confirmed every DeepSeek cron run on 2026.4.5 leaked preamble. A high-confidence automated reproduction should be a synthetic isolated cron run where the assistant emits unphased preamble plus the intended report, then assert Telegram announce delivery excludes the preamble under the new contract.

## Solution Assessment

No, the suggested marker/emoji/outputStartsWith sanitizer is not the best general fix. The maintainer-design direction is better: pin the delivery-path regression and add a provider-agnostic final-output boundary so cron does not infer finality from model prose or report formatting conventions.
