## Summary

Keep this issue open. Current main still does not fully satisfy the reported Feishu text-command fast path: Feishu message events still enter the inbound debouncer before dispatch, same-key non-debounced items can wait behind an active flush, and Feishu sequential-key routing only sends abort text to a control lane while /status and /new remain on the base per-chat lane. The provided GitHub context also shows open, unmerged PR #43512 as an implementation candidate whose body says it closes this issue, which is an explicit keep-open condition until it merges or is replaced.

## Best Possible Solution

Keep this issue open and have maintainers review, land, or replace PR #43512 with a current-main Feishu fix that detects command-only Feishu messages before the same-key inbound debouncer/serialization path or otherwise guarantees /stop, /new, and /status bypass active same-chat runs while preserving Feishu authorization, mention gating, dedupe, and normal-message ordering. Add regression coverage proving all three commands execute during an active same-chat Feishu run.
