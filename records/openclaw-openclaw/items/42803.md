## Summary

Keep open. Current main still routes Feishu receive events through inbound debounce before command dispatch, and non-debounced command-only events with the same debounce key can wait behind an active key chain. The current Feishu sequential key only gives abort-style text and `/btw` dedicated lanes while `/status` remains on the normal chat key, and this issue is paired with open closing PRs #43512 and fork #484 that should be reviewed, refreshed, landed, or replaced before closure.

## What This Changes

The issue requests restoring Feishu plain-text command fast-path behavior so `/stop`, `/new`, and `/status` bypass active same-chat agent runs.

## Best Possible Solution

Review and land or replace #43512 with a current-main Feishu ingress fix that classifies command-only text before same-key inbound debounce serialization, gives `/stop`, `/new`, and `/status` an immediate authorized control path, and preserves existing Feishu auth, mention gating, dedupe, normal-message FIFO behavior, and focused regression coverage.
