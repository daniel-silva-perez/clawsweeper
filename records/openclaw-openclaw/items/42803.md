## Summary

Current main still does not satisfy the reported Feishu text-command fast path. Feishu message receive handling sends every parsed message through the inbound debouncer before dispatch, the debouncer serializes same-key non-debounced work behind an active flush, and Feishu's sequential key resolver only gives abort-like text a control lane while /status and /new remain on the normal per-chat lane. The provided GitHub context also shows an open, unmerged PR with closing syntax for this issue, so cleanup policy says to keep the issue open until that PR or an equivalent fix is resolved.

## Best Possible Solution

Keep this issue open and have maintainers review, land, or replace PR #43512 with a current-main Feishu plugin fix that detects command-only Feishu messages before same-key inbound debouncer serialization, or otherwise gives /stop, /new, and /status a true control fast path while preserving authorization, mention gating, and normal message ordering. Add regression coverage proving these text commands execute during an active same-chat Feishu run.
