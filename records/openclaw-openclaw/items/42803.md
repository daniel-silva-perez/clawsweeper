## Summary

Keep open. Current main still gives Feishu `/status` and `/new` the normal same-chat sequential key, so those command-only messages can wait behind an active run; only abort-style text such as `/stop` and `/btw` currently get out-of-band lanes. Open closing PRs already target this issue, so the issue should remain open until that work is reviewed, landed, or replaced.

## What This Changes

Restore immediate Feishu text-command handling for `/stop`, `/new`, and `/status` during active agent runs instead of serializing those command-only messages behind the chat queue.

## Best Possible Solution

Review and land or replace the existing Feishu queue-bypass PR work with a current-main fix that gives authorized command-only `/stop`, `/new`, and `/status` an immediate out-of-band path before same-chat serialization, while preserving FIFO ordering for normal Feishu messages and adding a busy-lane regression test.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: start a long Feishu task on the base chat sequential key, then send `/status` or `/new`; current main resolves those commands to the same base key, so the sequential queue waits for the active task to finish.

## Solution Assessment

No for current main. The partial `/stop` and `/btw` key split does not solve all commands named in the report; the open PR direction is plausible but needs maintainer review against current main and the intended command allowlist.
