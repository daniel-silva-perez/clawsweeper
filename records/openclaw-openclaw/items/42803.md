## Summary

Keep open. Current main still only gives Feishu abort-style text and `/btw` separate out-of-band sequential keys; `/status` and `/new` stay on the normal same-chat key and can wait behind an active Feishu run. Open PR #43512 and the mirrored fork PR already target this regression, so this issue should remain open until that work is reviewed, landed, or deliberately replaced.

## What This Changes

The issue asks for Feishu plain-text `/stop`, `/new`, and `/status` command-only messages to bypass Feishu same-chat queueing and execute immediately during active agent runs, restoring pre-2026.3.8 behavior.

## Best Possible Solution

Review and land or replace the existing Feishu queue-bypass PR with a current-main fix that gives authorized command-only `/stop`, `/status`, and any maintainer-approved reset/session commands an immediate out-of-band path before same-chat serialization, while preserving FIFO ordering for normal Feishu messages and adding a busy-lane regression test.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: start a long Feishu task on the base chat sequential key, then send `/status` or `/new`; current main resolves those commands to the same base key, so the sequential queue waits for the active task to finish.

## Solution Assessment

No for current main. The existing partial `/stop` and `/btw` split does not cover all commands named in the report; the safer fix is a narrow Feishu ingress classification reviewed against the intended command allowlist rather than a broad bypass for every slash command.
