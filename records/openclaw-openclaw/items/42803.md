## Summary

Keep open. Current main only partially addresses the reported Feishu fast path: `/stop` and `/btw` get out-of-band sequential keys, but `/status` is explicitly expected to stay on the base chat key and `/new` has no Feishu out-of-band key, so the same-chat queue can still delay two of the commands named in the issue. The issue is also paired with open closing PRs, so the right maintainer action is to review, refresh, land, or replace that work rather than close this issue now.

## What This Changes

Feishu text slash commands such as `/stop`, `/new`, and `/status` should bypass same-chat queue serialization during an active agent run.

## Best Possible Solution

Review and land or replace the existing Feishu queue-bypass PR work with a current-main fix that gives authorized command-only `/stop`, `/new`, and `/status` an immediate out-of-band path before same-chat serialization, while preserving FIFO ordering for normal Feishu messages and adding a busy-lane regression test.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: start a long Feishu task on the base chat sequential key, then send `/status` or `/new`; current main resolves those commands to the same base key, so the sequential queue waits for the active task to finish.

## Solution Assessment

No for current main; the partial `/stop` and `/btw` key split does not solve all commands named in this issue. The open PR direction is plausible, but it needs maintainer review against current main and the intended command allowlist before it is the best fix.
