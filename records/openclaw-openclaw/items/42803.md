## Summary

Keep open. Current main only partially addresses the Feishu bypass: abort-style text such as `/stop` gets a control lane, but `/status` is still explicitly tested on the normal chat lane and `/new` has no out-of-band Feishu key, so the reported active-run delay can still happen. The issue is also paired with open closing PRs, so maintainers should review, refresh, land, or replace that work rather than close this issue now.

## What This Changes

The issue requests restoring Feishu text command fast-path behavior so `/stop`, `/new`, and `/status` execute immediately during an active agent run instead of waiting behind the chat queue.

## Best Possible Solution

Review and land or replace #43512 with a current-main Feishu ingress fix that gives authorized command-only `/stop`, `/new`, and `/status` an immediate out-of-band path before same-chat serialization, while keeping normal Feishu messages ordered and adding focused regression coverage for the busy-lane case.

## Reproduction Assessment

Yes. A code-level reproduction is clear: start a long Feishu same-chat task on the base sequential key, then send `/status` or `/new`; current main resolves those commands to the same base key, and the sequential queue waits for the first task to finish.

## Solution Assessment

Unclear for the open implementation until maintainers review it against current main. The direction is likely right, but the shipped partial fix is not enough because it leaves `/status` and `/new` on the normal Feishu chat lane.
