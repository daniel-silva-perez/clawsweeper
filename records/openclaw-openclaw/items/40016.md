## Summary

Keep open. Current main still has the lower-level Gateway label patch path, but it does not expose the requested first-class `openclaw sessions rename`, `set-label`, or `set-subject` workflow, and the local sessions list output still cannot show a user-set label.

## What This Changes

The issue requests a CLI command to set a human-readable subject or label for sessions whose keys are otherwise hard to identify.

## Best Possible Solution

Add a focused sessions label workflow: a first-class CLI command that updates the existing session label field, plus sessions list text/JSON output that shows an effective human-readable label while keeping channel-provided metadata distinct.

## Reproduction Assessment

Yes. The gap is reproducible by source inspection: current CLI registration lacks a rename/set-label command, and current sessions rows do not project label/display fields into text or JSON output.

## Solution Assessment

Yes. The narrow maintainable path is to reuse the existing `sessions.patch` label contract and extend the local sessions CLI/docs/tests, rather than adding a parallel `subject` or `displayName` mutation path.
