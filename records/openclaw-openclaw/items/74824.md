## Summary

Keep this PR open, but the patch should not merge as-is. Current main does not yet implement the requested Google Meet access controls or active-conference ending action, but the PR has two correctness problems: URL/code inputs are posted directly to `spaces.endActiveConference` even though the API contract requires the server-generated space resource id, and the new access-policy flags can be silently ignored by the existing browser fallback when OAuth is unavailable.

## What This Changes

The PR adds Google Meet create-time `SpaceConfig` access controls, a `meetings.space.settings` OAuth scope, a CLI/tool action for `spaces.endActiveConference`, docs, and targeted tests.

## Best Possible Solution

Keep the feature in the Google Meet plugin, but update the PR so active-conference ending resolves Meet URLs/codes to the real API space resource before posting, and so access-policy options either require the API/OAuth creation path or fail clearly instead of falling back to browser creation.

## Reproduction Assessment

Yes. The PR head source shows `googlemeet end-active-conference https://meet.google.com/abc-defg-hij` constructs `/v2/spaces/abc-defg-hij:endActiveConference`, and the current no-OAuth create branch still uses browser creation where the new access flags cannot be applied.

## Solution Assessment

No. The requested plugin direction is appropriate, but the current implementation is not the narrowest safe fix until it resolves URL/code aliases through `spaces.get` or requires true `spaces/{id}`, and validates access-policy flags before falling back to non-API creation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve Meet URLs before ending conferences:** `extensions/google-meet/src/meet.ts:422-424`
  - body: The new command/tool advertises Meet URLs and meeting codes, but this implementation posts the normalized input directly to `:endActiveConference`. Google's discovery contract describes that endpoint's `name` as the server-generated `spaces/{space}` id, while `meetingCode` is only documented as an alias for getting a space, so common URL/code inputs can fail. Resolve the URL/code with `fetchGoogleMeetSpace` first, or restrict this action to real space resource names and update docs/tests.
  - confidence: 0.88
- **[P2] Reject access policy flags without API creation:** `extensions/google-meet/src/cli.ts:1357-1364`
  - body: When no Google Meet OAuth token/config is present, `create` still follows the browser fallback, so the newly added `--access-type` and `--entry-point-access` options are accepted but not applied. That can create a room with inherited/default admission behavior after the caller explicitly requested a policy. The CLI and `google_meet` create path should fail clearly, or force the OAuth/API path, when these policy fields are supplied without API creation available.
  - confidence: 0.82
