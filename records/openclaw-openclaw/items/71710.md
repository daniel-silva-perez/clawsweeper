## Summary

Keep open. Current main still has Gateway-side abort support for accepted agent RPC runs, but the `openclaw agent` CLI path does not wire OS SIGTERM/SIGINT from cron or coreutils `timeout` into `callGateway`, `chat.abort`, embedded fallback/local execution, or a bounded process exit; the public docs also do not document the `timeout -k` mitigation for OS cron wrappers.

## What This Changes

The issue asks OpenClaw to prevent cron-launched `openclaw agent` processes from surviving SIGTERM and accumulating hung process chains, either by implementing bounded CLI shutdown or by documenting supervisor escalation requirements.

## Best Possible Solution

Implement a focused `openclaw agent` shutdown path: install SIGTERM/SIGINT handling for one-shot CLI runs, propagate a caller abort signal through `callGateway` and `GatewayClient.request`, abort the accepted Gateway agent run via `chat.abort` or `sessions.abort` using the known run/session identifiers, propagate cancellation into embedded fallback and `--local`, preserve session-lock cleanup, and document OS-cron wrapper guidance including `timeout -k` for older versions and non-cooperative supervisors.

## Reproduction Assessment

Yes. The issue provides a concrete Linux cron/coreutils `timeout` reproduction, and current source gives a high-confidence unit-level reproduction shape: keep a Gateway `agent` request or embedded run pending, emit SIGTERM, and observe that the CLI has no bounded abort or exit path beyond session-lock cleanup.

## Solution Assessment

Yes. The requested primary fix is the narrowest maintainable direction because the Gateway already has abort machinery for agent RPC runs; the missing work is to bridge local process signals into that machinery and add compatibility docs, not to fold this into the broader exec-supervisor issue #66399.
