## Summary

Current main still leaves the reported OS-cron/SIGTERM problem actionable. The `openclaw agent` CLI path waits on a Gateway `agent` request without a process-signal handler, caller `AbortSignal`, or follow-up `chat.abort`/`sessions.abort` call; Gateway-side abort machinery exists for accepted agent runs, but the CLI does not invoke it when coreutils `timeout` sends SIGTERM. The public agent/cron docs also still lack the `timeout -k` compatibility warning requested as the minimal mitigation.

## Best Possible Solution

Keep this issue open and implement a focused `openclaw agent` shutdown path: install SIGTERM/SIGINT handling around the one-shot CLI run, cancel the in-flight Gateway request, abort the accepted Gateway agent run when a runId/session key is known, propagate cancellation into embedded fallback/`--local`, preserve session-lock cleanup, and exit with the conventional signal status after a bounded grace window. Add a short public docs note for OS cron/supervisor wrappers, including `timeout -k`, as compatibility guidance for older releases and non-cooperative supervisors.
