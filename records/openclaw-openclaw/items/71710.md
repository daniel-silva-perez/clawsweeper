## Summary

Keep open. Current main still lacks a CLI-owned SIGTERM/SIGINT cancellation path for `openclaw agent`; the Gateway server can abort an accepted agent run, but the one-shot CLI, Gateway call wrapper, and Gateway client request API remain timeout-driven rather than process-signal-driven. The report is concrete, recent, not protected, and still suitable for a focused availability fix.

## What This Changes

Make externally scheduled `openclaw agent` invocations exit on SIGTERM within a bounded grace window, or at minimum document `timeout -k`/supervisor escalation for cron-spawned agent runs.

## Best Possible Solution

Implement a one-shot CLI shutdown contract: SIGTERM/SIGINT should abort the pending Gateway request, stop the accepted Gateway agent run using the existing abort registry, pass the same signal into local and embedded fallback runs, preserve session-lock cleanup, enforce a bounded grace deadline, and document `timeout -k` as an external-wrapper mitigation.

## Reproduction Assessment

Yes. The issue provides concrete Linux cron/coreutils steps and process-tree evidence, and current source supports a focused unit reproduction by holding a Gateway agent request or local embedded run pending, emitting SIGTERM/SIGINT, and observing that no CLI-owned abort path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort and `AgentCommandOpts.abortSignal` seams is the narrowest maintainable root fix; `timeout -k` documentation is useful mitigation guidance but should not be the only fix.
