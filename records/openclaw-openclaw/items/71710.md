## Summary

Keep open. Current main still has Gateway-side abort support for agent RPC runs, but the one-shot `openclaw agent` CLI path does not bind OS SIGTERM/SIGINT from cron/coreutils `timeout` into `callGateway`, `GatewayClient.request`, `chat.abort`/`sessions.abort`, or the local embedded fallback path; the public docs also do not document the `timeout -k` mitigation for external cron wrappers.

## What This Changes

The issue requests bounded SIGTERM handling for cron-spawned `openclaw agent` processes, or at minimum documentation telling cron users to use a SIGKILL escalation such as `timeout -k`.

## Best Possible Solution

Implement a focused one-shot CLI shutdown path: create a SIGTERM/SIGINT-owned AbortController for `openclaw agent`, pass it through `callGateway` and `GatewayClient.request`, use the known idempotency key/session key to ask the Gateway to abort the accepted run, propagate cancellation into `--local` and embedded fallback, preserve session-lock cleanup, enforce a small grace window before process exit, and document `timeout -k` as compatibility guidance for cron users and older releases.

## Reproduction Assessment

Yes. The issue gives concrete Linux cron/coreutils `timeout` steps, and current source supports a high-confidence unit reproduction: hold the Gateway agent request or embedded run pending, deliver SIGTERM, and observe that no CLI-owned abort or bounded exit path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort machinery is the narrowest maintainable root fix; documenting `timeout -k` is useful as an interim/operator workaround but should not be the only product fix.
