## Summary

Keep open. Current main still lacks a CLI-owned SIGTERM/SIGINT cancellation path for `openclaw agent`: the one-shot Gateway call and embedded/local fallback can use timeouts, but they do not bridge process termination into the Gateway request, active Gateway run, or local `agentCommand` abort signal. The issue has detailed Linux cron/coreutils reproduction data and remains actionable.

## What This Changes

`openclaw agent` should handle SIGTERM from cron/coreutils `timeout` by canceling in-flight work and exiting within a bounded grace window, or at minimum document `timeout -k` mitigation for externally scheduled runs.

## Best Possible Solution

Implement a focused one-shot CLI shutdown path: create a SIGTERM/SIGINT-owned AbortController for `openclaw agent`, pass it through `callGateway` and `GatewayClient.request`, abort the accepted Gateway run through the existing agent RPC abort machinery, propagate cancellation into `--local` and embedded fallback, preserve session-lock cleanup, enforce a small graceful-exit deadline, and document `timeout -k` for external cron wrappers and older releases.

## Reproduction Assessment

Yes. The issue provides concrete Linux cron/coreutils `timeout` steps, and current source supports a focused unit reproduction by holding the Gateway agent request or embedded run pending, delivering SIGTERM, and observing that no CLI-owned abort or bounded exit path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort path is the narrowest maintainable root fix; documentation for `timeout -k` is useful operator guidance but should not be the only fix.

## Repair Work Prompt

Fix the `openclaw agent` SIGTERM/SIGINT shutdown path for externally scheduled one-shot runs. The observable bug is that `timeout 600 openclaw agent ...` under cron can leave long-lived agent process chains when SIGTERM is not converted into cancellation and bounded exit. Implement a narrow root fix: add a CLI-owned AbortController for `agentCliCommand`/`agentViaGatewayCommand`, register SIGTERM/SIGINT handlers that abort once, preserve existing session-lock cleanup behavior, and enforce a small graceful-exit deadline before exiting with the normal signal-derived code. Thread the signal through `callGateway` and `GatewayClient.request`; when the Gateway has accepted an `agent` run, use the existing `chat.abort`/`sessions.abort`/active-run abort machinery or an equivalent internal helper so the Gateway run is stopped instead of leaving work behind. Also pass the same signal into `agentCommand` for `--local` and embedded fallback paths. Add tests that hold a Gateway request and embedded run pending, emit SIGTERM/SIGINT through injected process-like handlers where possible, and assert the pending request/run is aborted, cleanup runs, and listeners are removed. Update public docs for external cron/coreutils wrappers to recommend `timeout -k <delay> <duration>` until/for older releases. Add a single-line user-facing CHANGELOG.md fix entry crediting `Thanks @nikolaykazakovvs-ux`. Do not broaden this into the exec-supervisor two-phase kill work tracked by #66399, do not change provider defaults, and do not alter Gateway protocol incompatibly.
