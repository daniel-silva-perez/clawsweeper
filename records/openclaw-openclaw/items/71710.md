## Summary

Keep open. Current `main` still has the server-side Gateway abort seam, but the one-shot `openclaw agent` CLI path does not bridge SIGTERM/SIGINT into the Gateway request, pending client request, active Gateway run, or local embedded fallback; the relevant public docs also do not document the `timeout -k` mitigation.

## What This Changes

The issue requests bounded SIGTERM/SIGINT handling for cron-spawned `openclaw agent` processes, or at minimum documentation requiring `timeout -k`/equivalent supervisor escalation to prevent hung process chains.

## Best Possible Solution

Implement a focused one-shot CLI shutdown path: create a SIGTERM/SIGINT-owned AbortController for `openclaw agent`, pass it through `callGateway` and `GatewayClient.request`, abort the accepted Gateway run through the existing agent RPC abort machinery, propagate cancellation into `--local` and embedded fallback, preserve session-lock cleanup, enforce a small graceful-exit deadline, and add operator docs for `timeout -k` on external cron wrappers and older releases.

## Reproduction Assessment

Yes. The issue gives concrete Linux cron/coreutils `timeout 600 openclaw agent ...` steps and current source supports a focused unit reproduction by holding the Gateway agent request or embedded run pending, delivering SIGTERM, and observing that no CLI-owned abort or bounded exit path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort path is the narrowest maintainable root fix; documentation for `timeout -k` is useful compatibility guidance but should not be the only fix.
