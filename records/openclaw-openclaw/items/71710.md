## Summary

Keep open. Current main still has Gateway-side abort support for agent RPC runs, but the one-shot `openclaw agent` CLI path does not wire cron/coreutils SIGTERM or SIGINT into the Gateway request, the pending `GatewayClient.request`, the active Gateway run abort API, or the local embedded fallback path; the public docs also do not mention the `timeout -k` mitigation for external cron wrappers.

## What This Changes

The issue requests bounded SIGTERM handling for cron-spawned `openclaw agent` runs, or at minimum documentation warning cron users to use a SIGKILL escalation such as `timeout -k`.

## Best Possible Solution

Implement a focused one-shot CLI shutdown path: create a SIGTERM/SIGINT-owned AbortController for `openclaw agent`, pass it through `callGateway` and `GatewayClient.request`, use the known idempotency/session key to abort the accepted Gateway run through the existing abort machinery, propagate cancellation into `--local` and embedded fallback, preserve session-lock cleanup, enforce a small graceful-exit deadline, and document `timeout -k` as operator guidance for cron wrappers and older releases.

## Reproduction Assessment

Yes. The issue gives concrete Linux cron/coreutils `timeout` steps, and current source supports a focused unit reproduction by holding the Gateway agent request or embedded run pending, delivering SIGTERM, and observing that no CLI-owned abort or bounded exit path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort path is the narrowest maintainable root fix; documenting `timeout -k` is useful compatibility guidance but should not be the only fix.
