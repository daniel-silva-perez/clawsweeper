## Summary

Keep open. Current main still lacks a CLI-owned SIGTERM/SIGINT cancellation path for `openclaw agent`: the Gateway request path, local `--local` path, and embedded fallback path are still timeout-driven or cleanup-only, not process-signal-driven. The report has concrete Linux cron/coreutils reproduction data and remains a narrow availability fix candidate.

## What This Changes

`openclaw agent` should handle SIGTERM from cron/coreutils `timeout` by canceling in-flight work and exiting within a bounded grace window, with `timeout -k` documented as external-wrapper guidance for older or unfixed releases.

## Best Possible Solution

Implement a focused one-shot CLI shutdown contract: SIGTERM/SIGINT should abort the pending Gateway request, stop the accepted Gateway agent run through the existing abort registry, pass the same signal into local and embedded fallback runs, preserve session-lock cleanup, exit with a bounded grace deadline, and add operator docs for `timeout -k` around external cron wrappers.

## Reproduction Assessment

Yes. The issue gives concrete cron/coreutils steps, and current source supports a focused unit reproduction by holding a Gateway agent request or embedded run pending, emitting SIGTERM/SIGINT, and observing that no CLI-owned abort path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort path is the narrowest maintainable root fix; `timeout -k` documentation is useful mitigation guidance but should not be the only fix.

## Repair Work Prompt

Fix the `openclaw agent` SIGTERM/SIGINT shutdown path for externally scheduled one-shot runs. The observable bug is that `timeout 600 openclaw agent ...` under cron can leave long-lived agent process chains when SIGTERM is not converted into cancellation and bounded exit. Implement a narrow root fix: add a CLI-owned AbortController for `agentCliCommand`/`agentViaGatewayCommand`, register SIGTERM/SIGINT handlers that abort once, preserve existing session-lock cleanup behavior, and enforce a small graceful-exit deadline before exiting with the normal signal-derived code. Thread the signal through `callGateway` and `GatewayClient.request`; when the Gateway has accepted an `agent` run, use the existing `chat.abort`/`sessions.abort`/active-run abort machinery or an equivalent internal helper so the Gateway run is stopped instead of leaving work behind. Also pass the same signal into `agentCommand` for `--local` and embedded fallback paths. Add tests that hold a Gateway request and embedded run pending, emit SIGTERM/SIGINT through injected process-like handlers where possible, and assert the pending request/run is aborted, cleanup runs, and listeners are removed. Update public docs for external cron/coreutils wrappers to recommend `timeout -k <delay> <duration>` until or for older releases. Add a single-line user-facing `CHANGELOG.md` fix entry crediting `Thanks @nikolaykazakovvs-ux`. Do not broaden this into the exec-supervisor two-phase kill work tracked by #66399, do not change provider defaults, and do not alter Gateway protocol incompatibly.
