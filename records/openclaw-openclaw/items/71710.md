## Summary

Keep open. Current `main` still has the reported source gap: `openclaw agent` waits on a Gateway agent RPC or embedded local run without a CLI-owned SIGTERM/SIGINT cancellation path, while the docs do not warn external cron users to use `timeout -k` as a mitigation.

## What This Changes

The issue requests that externally scheduled `openclaw agent` runs either exit gracefully on SIGTERM within a bounded window or document the required `timeout -k`/supervisor escalation workaround for cron use.

## Best Possible Solution

Add a focused one-shot shutdown path for `openclaw agent`: create a CLI-owned SIGTERM/SIGINT AbortController, pass it through `callGateway` and `GatewayClient.request`, use the known session/run id to abort the accepted Gateway run when possible, propagate the same signal into `--local` and embedded fallback, retain session-lock cleanup, enforce a short grace exit, and document `timeout -k` for cron users and older releases.

## Reproduction Assessment

Yes. The issue gives concrete Linux cron/coreutils `timeout` steps, and current source supports a high-confidence unit reproduction by holding a Gateway or embedded agent run pending, delivering SIGTERM, and observing that no CLI-owned abort or bounded exit path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort machinery is the narrowest maintainable root fix; documentation for `timeout -k` is useful compatibility guidance but should not be the only fix.

## Repair Work Prompt

Current `openclaw agent` one-shot runs launched by external cron/coreutils `timeout` do not bind SIGTERM/SIGINT into the pending Gateway request or local embedded run, so processes can survive after `timeout` sends SIGTERM unless the wrapper uses `timeout -k`. Implement a focused CLI shutdown path: create a SIGTERM/SIGINT-owned AbortController for `openclaw agent`, pass the signal through `callGateway` and `GatewayClient.request`, stop/settle the pending request on abort, and, when the Gateway has accepted the run, use the known `sessionKey`/idempotency key/run id path to call `chat.abort` or `sessions.abort` so the Gateway-side `agent` abort controller is triggered.
