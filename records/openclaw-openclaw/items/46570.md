## Summary

Keep open. Current main still matches the reported failure mode: memory files and sessions are indexed as separate sources, but the builtin CLI search does not pass a source/corpus filter and the hybrid result path still returns strict merged hits before considering relaxed keyword-backed memory hits. The existing discussion was useful and correctly distinguishes this from the closed snippet issue, so this is a narrow memory-core follow-up rather than stale cleanup.

## What This Changes

The issue requests that builtin `openclaw memory search` surface indexed `MEMORY.md` and `memory/*.md` matches in mixed memory/session hybrid searches instead of returning only session transcript results.

## Best Possible Solution

Keep the issue open and fix the builtin memory-core search path so default mixed-source hybrid search retains relevant keyword-backed memory-file hits even when session transcript vector hits also satisfy strict scoring. A `--corpus memory|sessions|all` CLI option can be useful, but it should not be the only fix because the default mixed search should still return relevant memory files.

## Reproduction Assessment

Yes. The report gives a concrete CLI reproduction, and the current source shows a focused test path: configure builtin hybrid search with `sources: ["memory", "sessions"]`, put an exact phrase only in a memory file, add session transcript chunks that produce strict vector hits, then assert `MemoryIndexManager.search` and the CLI-equivalent path return a memory-source result.

## Solution Assessment

Yes, a narrow memory-core manager/CLI regression and ranking fix is the best path. A CLI-only corpus filter would provide a workaround, but the maintainable fix is to make the default mixed-source hybrid selection retain exact memory-file keyword hits instead of letting unrelated strict session hits crowd them out.
