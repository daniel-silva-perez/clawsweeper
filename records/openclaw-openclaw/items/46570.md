## Summary

Keep open. Current main still has the mixed-source builtin hybrid search failure shape: memory and session chunks can be searched together, but `MemoryIndexManager.search` returns strict hybrid hits before relaxed keyword-backed memory hits are considered, so exact memory-file matches can still be crowded out by session results. The docs and config support memory files plus optional sessions, and there is no open PR currently tied to this issue.

## What This Changes

The issue asks for `openclaw memory search` to return indexed matches from `MEMORY.md` and `memory/*.md`, not only session transcript results, when memory and session sources are both enabled.

## Best Possible Solution

Adjust builtin memory-core search selection so default all-source hybrid search preserves relevant keyword-backed memory-file hits even when session chunks also produce strict hits. Add regression coverage for mixed `memory` plus `sessions` sources; a CLI `--corpus` filter can be additive, but should not be the only fix.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, and current source supports a deterministic test: enable builtin hybrid search with `sources: ["memory", "sessions"]`, seed an exact phrase only in a memory file plus strict-scoring session chunks, and assert a memory-source hit is returned.

## Solution Assessment

Yes. A narrow `MemoryIndexManager.search` ranking/selection fix with manager and CLI-equivalent regression coverage is the best path; a CLI-only corpus filter would be a workaround and would not fix the default mixed search behavior.
