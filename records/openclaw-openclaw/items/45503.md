## Summary

Keep open. Current main still addresses tool-output bloat only through automatic cache-TTL pruning, live/persisted size caps, whole-session compaction, and optional pre-model reducers; I did not find a manual user- or agent-addressable clear/archive/recover path for a specific completed tool result. The discussion also raises recoverability and checkpoint semantics, so this needs maintainer product/API design rather than stale cleanup.

## What This Changes

The issue requests a way to manually purge or archive specific tool results from active context once they are no longer needed.

## Best Possible Solution

Define a core-supported prompt-view clear/archive mechanism for completed tool results that preserves raw transcript history and provider pairing, exposes an intentional user/agent surface, supports recovery or clear placeholders, and is documented with coverage for /context reporting, pruning, compaction, and provider replay.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature gap. The gap is reproducible by source inspection on current main: /context has only inspect modes, /compact compacts the whole session, and the pruning path is TTL/policy-driven with no manual tool-result selector.

## Solution Assessment

Unclear. The requested CLI command, agent function, and clearAfterUse flag are plausible surfaces, but the safest maintainable direction appears to be a prompt-view archive/clear design that preserves transcript and replay invariants instead of deleting history directly.

- An agent-accessible clear function needs recovery semantics so IDs, config values, or search result handles are not lost after one answer.
