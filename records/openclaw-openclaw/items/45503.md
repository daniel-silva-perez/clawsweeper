## Summary

Keep open. Current main still has automatic cache-TTL pruning, live tool-result truncation, whole-session compaction, and bundled pre-model reducers, but no user- or agent-addressable clear/archive/recover surface for a specific completed tool result. The discussion raises recovery and checkpoint semantics, so this is a core product/API design item rather than stale cleanup.

## What This Changes

The issue requests a way to manually purge or archive specific completed tool results from active context once they are no longer needed.

## Best Possible Solution

Define a core-supported prompt-view clear/archive mechanism for completed tool results that preserves raw transcript history and provider pairing, exposes an intentional user and/or agent surface, supports recovery or clear placeholders, and is documented across /context reporting, pruning, compaction, and provider replay.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature gap. The gap is reproducible by source inspection on current main: /context has only inspect modes, /compact compacts the whole session, and the pruning path is TTL/policy-driven with no manual tool-result selector.

## Solution Assessment

Unclear. The requested CLI command, agent function, and clearAfterUse flag are plausible surfaces, but the safer maintainable direction appears to be a prompt-view archive/clear design that preserves transcript and replay invariants instead of deleting history directly.

## Risks / Open Questions

- The desired semantics are unresolved: delete, prompt-view clear, recoverable archive, and clear-after-use each have different UX and safety tradeoffs.
- Rewriting stored transcript history could break provider replay invariants around assistant tool calls and matching tool results unless the design preserves pairing with placeholders or uses an in-memory prompt projection.
- An agent-accessible clear function needs recovery semantics so IDs, config values, or search result handles are not lost after one answer.
