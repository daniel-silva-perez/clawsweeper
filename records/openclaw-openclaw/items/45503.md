## Summary

Current main has automatic cache-TTL pruning, live tool-result size caps, /compact, and /context inspection, but I found no manual per-tool-result purge/recover command, tool-call clearAfterUse flag, or context-engine API. This should stay open for a focused product/API design decision.

## Best Possible Solution

Keep this issue open and design a core-owned context/session primitive for manually removing or archiving specific tool results from active model context, likely exposed through chat/CLI and the context-engine lifecycle. The implementation should define recoverability, transcript persistence, tool-result identifiers, compaction interactions, and tests/docs around large-result workflows.

## Risks / Open Questions

- Manual clearing has product semantics to settle: hard delete could force re-fetches or lose recovery context, while a soft archive/recover path would need a clear retrieval contract.
- Closing now would conflate existing automatic pruning and compaction mitigations with the narrower user-requested manual purge control.
