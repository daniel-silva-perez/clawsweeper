## Summary

Keep this PR open. Current main still lacks the PR's specific Telegram restart/update recovery watchdog and systemd path guidance; the existing docs cover polling-stall recovery, retry/network tuning, and troubleshooting, but not the host-level recovery pattern proposed here. The PR is docs-only and the systemd `~` concern appears addressed in the current PR diff, but the `openclaw health --json` watchdog guidance should address the low-risk disclosure review before merge.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to preserve the focused Telegram recovery-hardening docs, keep the fixed systemd path wording, and add a short warning or sanitized extraction example for `openclaw health --json` before merge.
