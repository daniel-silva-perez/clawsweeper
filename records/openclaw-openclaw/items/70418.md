## Summary

Keep open. Current main has Anthropic prompt-cache retention, cache-ttl pruning, and heartbeat keep-warm behavior, but it still does not implement a dedicated silent Cache TTL Warmer or a `cacheWarmer` config/schema/docs/runtime surface. The proposal remains a maintainer design decision because the background provider-call path affects spend, rate limits, state boundaries, provider semantics, and core-vs-plugin API shape.

## Best Possible Solution

Keep this issue open for maintainer design. If approved, implement a narrow opt-in cache-maintenance path or explicit plugin API seam that can reuse the same request prefix as a real user turn through the cache-control boundary, append a bounded minimal marker, consume and discard the response, avoid tool execution and all transcript/channel/memory writes, document `cacheRetention` and cost behavior, and include focused scheduler/request-shape tests plus an explicit live Anthropic validation plan.

## Risks / Open Questions

- A silent background provider-call loop can create unexpected Anthropic spend, rate-limit pressure, and confusing failures unless opt-in defaults, telemetry, and failure handling are explicit.
- Reusing heartbeat directly would violate the requested boundaries because current heartbeat can read HEARTBEAT.md, process task blocks, handle `HEARTBEAT_OK`, send typing/outbound payloads, update heartbeat task state, and consume system events.
- The minimal Anthropic marker request needs live validation for accepted message shape, `max_tokens` behavior, cache TTL refresh semantics across marker positions, and behavior after compaction/session reset.
- The current plugin API does not clearly expose a safe external path to build/send the exact next-turn provider request while avoiding transcript and channel state, so maintainers need to decide core implementation vs a new plugin API seam.
