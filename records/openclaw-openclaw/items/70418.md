## Summary

Keep open. Current main has prompt-cache retention, cache-ttl pruning, heartbeat keep-warm guidance, and provider-specific cache refresh for Gemini cachedContents, but it does not implement the proposed dedicated Anthropic Cache TTL Warmer or a `cacheWarmer` config/schema/runtime/docs surface. This remains a maintainer design decision because it would add paid background provider calls with explicit cost, provider, scheduler, and no-session/no-channel state-boundary requirements.

## What This Changes

Request to add an opt-in per-agent Cache TTL Warmer that refreshes Anthropic prompt-cache TTL with minimal non-transcript requests independent of heartbeat runs.

## Best Possible Solution

Keep this issue open for maintainer design. If approved, add a narrow opt-in cache-maintenance path or explicit plugin/API seam that reuses the same stable prefix/tools/system/messages through the cache-control boundary, appends only a bounded marker, consumes and discards the provider response, avoids tool execution and all transcript/session/channel/memory writes, documents cost and `cacheRetention` behavior, and includes focused unit tests plus live Anthropic validation.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/design proposal. The absence check is high-confidence from source, schema, docs, runtime, and synonym searches showing no `cacheWarmer` surface on current main.

## Solution Assessment

Unclear because there is no patch and the best implementation boundary needs maintainer judgment. A dedicated warmer may be reasonable, but it should not be queued until maintainers decide whether core should own paid background model calls versus extending heartbeat behavior or exposing a plugin seam.
