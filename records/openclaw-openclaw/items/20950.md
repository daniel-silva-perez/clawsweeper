## Summary

Keep this issue open. Current main still lacks directory-backed cron job definitions: the config contract exposes only `cron.store`, the strict schema has no `storeDir` or `jobsDir`, Gateway startup builds a single `storePath`, and the cron store reads/writes one aggregate `jobs.json` plus a derived `jobs-state.json` sidecar. The related open PR #41206 remains the existing implementation candidate, while #53581 only shipped the complementary runtime-state split and #70531 was already closed as a duplicate into this request.

## What This Changes

Requests opt-in cron job definition storage as one JSON file per job in a `jobs.d`-style directory for declarative NixOS/GitOps setups while preserving existing `jobs.json` behavior.

## Best Possible Solution

Keep this as the canonical tracker and land or rework #41206, or an equivalent additive implementation, so existing `cron.store`/`jobs.json` users remain compatible while an explicitly documented directory-store mode deterministically loads per-job JSON definitions, preserves `jobs-state.json` runtime state, handles add/edit/delete safely, and updates config schema/help, doctor, docs, and focused cron store/Gateway tests.
