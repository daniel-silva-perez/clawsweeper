## Summary

Keep open. Current main still lacks directory-backed cron job definitions: config/schema expose only the aggregate `cron.store`, Gateway startup resolves one store path, and the cron store still reads/writes `jobs.json` plus `jobs-state.json`. The discussion and related context identify open PR #41206 as the active implementation candidate, so this issue should remain the tracker for maintainer review rather than be closed or queued as duplicate automation.

## What This Changes

The issue requests an additive cron directory store so each scheduled job can live in its own JSON file for declarative GitOps/NixOS management alongside or instead of the existing aggregate `jobs.json` store.

## Best Possible Solution

Ship an additive, documented cron directory-store mode that keeps existing `cron.store` and `jobs.json` users compatible, deterministically loads per-job JSON definitions, keeps runtime state out of declarative files, and updates config schema/help, doctor behavior, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is verifiable on current main by inspecting config/schema, Gateway startup, cron store code, and docs, all of which still model one aggregate definition file rather than a per-job directory.

## Solution Assessment

Yes for triage: leaving this issue open while maintainers review, land, rework, or replace #41206 is the right path. A separate repair job would duplicate an already-open implementation candidate instead of improving review flow.
