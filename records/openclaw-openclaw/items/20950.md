## Summary

Keep this issue open. Current main still only supports a single aggregate cron definition file plus a derived runtime-state sidecar, and the discussion already identifies open PR #41206 as the active implementation candidate while keeping this issue as the canonical tracker.

## What This Changes

The issue requests support for directory-backed cron job definitions so each cron job can live in an individual JSON file for declarative NixOS/GitOps setups.

## Best Possible Solution

Keep this as the canonical tracker and land or rework #41206, or an equivalent additive implementation, so existing `cron.store` and `jobs.json` users remain compatible while a documented directory-store mode deterministically loads per-job JSON definitions, preserves runtime state, handles add/edit/delete safely, and updates schema/help, doctor, docs, and focused cron store/Gateway tests.
