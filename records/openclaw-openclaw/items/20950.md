## Summary

Keep open. Current main still lacks directory-backed cron job definitions: config/schema expose only the aggregate `cron.store`, Gateway startup resolves one store path, and the cron store still reads/writes `jobs.json` plus `jobs-state.json`. The provided timeline also identifies open PR #41206 as the active implementation candidate, so this issue should remain the tracker for maintainer review rather than be closed or queued as a separate repair.

## What This Changes

The issue requests a `jobs.d` or per-job JSON cron definition directory for declarative NixOS/GitOps setups alongside or instead of the current single `jobs.json` store.

## Best Possible Solution

Ship an additive, documented cron directory-store mode that keeps existing `cron.store` and `jobs.json` users compatible, deterministically loads per-job JSON definitions, preserves the current runtime-state sidecar model, and updates config schema/help, doctor behavior, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a feature request. The absence is verifiable from current main by inspecting the cron config/schema, Gateway startup, store implementation, and docs, all of which still point at a single aggregate definition file.

## Solution Assessment

Yes for triage: keeping this issue open while maintainers review, rework, land, or replace #41206 is the right path. A separate repair job would duplicate the already-open implementation candidate instead of improving review flow.
