## Summary

Keep open. Current main still lacks a directory-backed cron job store: config and schema only expose the aggregate `cron.store` path, Gateway startup passes one resolved store path into `CronService`, and the store implementation reads/writes `jobs.json` plus `jobs-state.json` rather than scanning or writing one file per job. Open PR #41206 is already the active implementation candidate, so this should stay as the tracker for maintainer review instead of being closed or queued as a separate repair.

## What This Changes

Add support for a cron `jobs.d` or per-job JSON directory so NixOS/GitOps users can declaratively manage individual cron job definitions while preserving existing cron storage compatibility.

## Best Possible Solution

Ship an additive, documented cron directory-store mode that keeps existing `cron.store` and `jobs.json` users compatible, deterministically loads per-job JSON definitions, preserves the current runtime-state sidecar model, and updates config schema/help, doctor behavior, docs, and focused cron store/Gateway tests.
