## Summary

Keep this issue open. Current main still supports a single aggregate cron definitions file through `cron.store`/`~/.openclaw/cron/jobs.json`, with runtime state split into `jobs-state.json`; it does not expose `cron.storeDir`, `jobsDir`, `jobs.d`, directory scanning, or one-file-per-job load/save behavior. The discussion already identifies #41206 as the open implementation candidate, while #70531 was closed into this issue and #53581 handled only the complementary runtime-state split.

## Best Possible Solution

Keep this issue open as the canonical product request for GitOps/NixOS-friendly directory-backed cron definitions. The best path is to review and either land #41206 or replace it with an additive implementation that preserves existing `cron.store`/`jobs.json` compatibility, adds a documented directory store option, deterministically loads per-job JSON files, keeps runtime state in the current sidecar model or an explicitly compatible replacement, and updates config schema/help/docs/doctor plus focused cron store/gateway tests.
