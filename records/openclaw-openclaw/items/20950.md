## Summary

Keep open. Current main still lacks a directory-backed cron definition store: config/schema, Gateway startup, store load/save, and docs all continue to use a single `cron.store`/`jobs.json` file plus `jobs-state.json`. The provided issue context identifies open PR #41206 as the active implementation candidate, so this issue should remain the tracker for maintainer review instead of being closed or queued for a duplicate repair.

## What This Changes

The issue requests opt-in cron `jobs.d` or per-job JSON directory support so NixOS/GitOps users can manage individual cron job definitions declaratively while preserving existing cron behavior.

## Best Possible Solution

Keep this issue as the canonical tracker and land, rework, or replace #41206 with an additive directory-store mode that keeps existing `jobs.json` users compatible, deterministically loads per-job JSON definitions, keeps runtime state out of declarative files, and updates config schema/help, doctor behavior, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is high-confidence verifiable by source and docs inspection: current config/schema, Gateway startup, store load/save, and docs still use one aggregate definition file rather than a per-job directory.
