## Summary

Keep open. Current main still lacks directory-backed cron definitions: config/schema accept only `cron.store`, Gateway builds cron from one resolved store path, the store loader reads one aggregate `jobs` array, and docs still describe `jobs.json` plus `jobs-state.json`. The issue discussion also identifies open PR #41206 as the active implementation candidate, so this should remain the tracker for maintainer review rather than be closed or duplicated by a repair job.

## What This Changes

The issue requests directory-backed cron job definitions, with one JSON file per cron job for NixOS/GitOps-style declarative management.

## Best Possible Solution

Keep this issue as the canonical tracker and land, rework, or replace #41206 with an additive directory-store mode that keeps existing `jobs.json` users compatible, deterministically loads per-job JSON definitions, keeps runtime state out of declarative files, and updates config schema/help, doctor behavior, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is high-confidence verifiable by source and docs inspection: current config/schema, Gateway startup, store load/save, and docs still use one aggregate definition file rather than a per-job directory.

## Solution Assessment

Yes for triage. Keeping this issue open while maintainers review, land, rework, or replace #41206 is the narrowest maintainable path; a separate ClawSweeper repair job would duplicate an already-open implementation candidate.
