## Summary

Keep this issue open. Current main at ab5c8025c9d0a31873d311ae3e73f3664b0a1185 still supports cron definitions through the single aggregate `cron.store` file, defaulting to `~/.openclaw/cron/jobs.json`, with runtime state split into `jobs-state.json`. It does not expose `cron.storeDir`, `jobsDir`, `jobs.d`, directory scanning, or one-file-per-job load/save behavior. The related context points to #41206 as the open implementation candidate, #70531 was closed into this issue, and #53581 only handled the complementary runtime-state split.

## Best Possible Solution

Keep this issue open as the canonical product request for GitOps/NixOS-friendly directory-backed cron definitions. The best path is to review and either land #41206 or replace it with an additive implementation that preserves existing `cron.store`/`jobs.json` compatibility, adds a documented directory store option, deterministically loads per-job JSON files, keeps runtime state in the current sidecar model or an explicitly compatible replacement, and updates config schema/help/docs/doctor plus focused cron store/gateway tests.
