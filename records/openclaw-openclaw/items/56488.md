## Summary

Keep open. Current main still has the normal cron timer and manual due-run paths calling `collectRunnableJobs` without the completed-one-shot guard, while startup catch-up uses that guard; the issue is also paired with open PR #56509, so it should remain open until a focused fix is merged or that PR is superseded.

## What This Changes

The issue asks for one-shot `kind: "at"` cron jobs with `deleteAfterRun: true` to be deleted or otherwise prevented from re-running after their first completed execution.

## Best Possible Solution

Land a focused cron scheduler patch that applies the existing completed-one-shot guard to the normal timer and manual due-run scans, with regression coverage for a past-due `at` job whose terminal state is already recorded. If #56509 contains unrelated changes, split or replace it with a cron-only fix before merge.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to persist an enabled `kind: "at"` job with `deleteAfterRun: true`, terminal `lastStatus`, and a past `nextRunAtMs`, then exercise `onTimer` or `runDueJobs`; current main omits the guard on both paths.

## Solution Assessment

Yes, with scope control. Reusing the existing `skipAtIfAlreadyRan` guard at the missing call sites is the narrowest maintainable fix, provided the final PR adds focused regression tests and avoids unrelated runtime/security changes.
