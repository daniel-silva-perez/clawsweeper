## Summary

Keep this PR open for maintainer review. Current main still lacks the proposed standardized filesystem permission-denied surface and mutation payload guards, but the PR branch has concrete correctness blockers in the sandbox read and mount-boundary classification paths.

## What This Changes

This PR adds an agent filesystem permission-denied helper, wraps apply_patch and sandbox filesystem denials with structured reason metadata, adds write/edit text payload size and NUL-byte guards, and updates related tests.

## Best Possible Solution

Request changes on this PR, then land it only after mount escapes classify as workspace-boundary denials, missing sandbox reads preserve ordinary not-found semantics, payload guard limits are accepted, and focused regression coverage proves host plus sandbox write/edit/read behavior.

## Reproduction Assessment

Yes. The defects are reproducible by line inspection and focused tests: pass an 'escapes allowed mounts' error through the PR classifier, and read a missing file through SandboxFsBridge while asserting the original ENOENT-style code/semantics are preserved.

## Solution Assessment

No, not as currently proposed. Centralizing permission-denied metadata is the right direction, but the branch should preserve path-miss semantics and recognize the existing mount-boundary message shape before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve ENOENT for missing sandbox reads:** `src/agents/sandbox/fs-bridge-path-safety.ts:80-84`
  - body: `openReadableFile` now wraps every failed boundary open in `E_FS_PERMISSION_DENIED`, including `reason === "path"` misses. `openBoundaryFile` classifies ENOENT/ENOTDIR/ELOOP as path failures, and current coverage expects missing sandbox reads to stay ordinary not-found errors before any Docker command runs, so direct read/media callers lose errno-based diagnostics. Re-throw the original path error and only wrap validation/permission failures.
  - confidence: 0.91
- **[P2] Map mount escapes to workspace_boundary:** `src/agents/fs-permission-denied.ts:23-25`
  - body: The new classifier does not recognize `"Sandbox path escapes allowed mounts"`, but this PR wraps that exact message from the local and remote sandbox bridge paths. Those denials will be reported as `reason=unknown` instead of `workspace_boundary`, which defeats the standardization this PR introduces. Add the phrase to the boundary classifier and cover it in the new helper test.
  - confidence: 0.94
