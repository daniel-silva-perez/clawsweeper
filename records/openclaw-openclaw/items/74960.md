## Summary

Keep open. Current main still uses the same WebChat image-paste and attachment-send path as the reported v2026.4.27 surface, and the new report describes a byte-identity failure that the earlier paste-support closure did not verify. I found source coverage for the path but no regression test proving the pasted clipboard image bytes are the exact bytes sent to the Gateway/agent.

## What This Changes

The issue reports that images pasted into WebChat/Control UI with Ctrl+V or Cmd+V are sent to the agent as a different or stale image in v2026.4.26 and v2026.4.27.

## Best Possible Solution

Add a narrow regression around WebChat paste byte identity, then fix the smallest failing boundary so the same image bytes copied from the clipboard are the bytes serialized into chat.send and delivered through parsedImages. Keep the fix in the Control UI paste/payload/send path unless the test proves the Gateway parser mutates or substitutes the payload.

Yes. The issue provides a concrete reproduction path: copy a screenshot on macOS Chrome, paste it into the WebChat/Control UI composer, send, and compare the image the agent receives with the clipboard image; this review did not live-run it but identified the exact source path to instrument.

## Solution Assessment

Unclear. Closing as implemented is not the best solution because current code only proves paste support exists; the safer path is a focused regression that proves clipboard bytes survive handlePaste, attachment storage, buildApiAttachments, and chat.send delivery unchanged.
