## Summary

Keep open. Current main still supports image attachment creation and dispatch, but it does not prove or fix the reported stale/wrong-image byte mismatch across WebChat paste, the side payload store, `chat.send`, and Gateway dispatch; the newer cross-channel WeChat context also makes this distinct from the earlier paste-support issue.

## What This Changes

Report that WebChat/Control UI pasted images are delivered to the agent as a different or stale image in v2026.4.26 and v2026.4.27.

## Best Possible Solution

Add a regression that compares original clipboard/channel image bytes with the bytes stored in the UI payload store, serialized into `chat.send`, parsed by Gateway, and dispatched to the agent, then fix the smallest failing boundary in the generic attachment/media path.

## Reproduction Assessment

Yes. The issue gives concrete macOS Chrome WebChat paste steps and cross-channel stale-image behavior, and current main has a clear instrumentation path through `handlePaste`, the side payload store, `buildApiAttachments`, `parseMessageWithAttachments`, and `dispatchInboundMessage`; this review did not run the live browser/channel repro.

## Solution Assessment

Unclear because there is no proposed patch yet. The narrow maintainable solution is a byte-identity regression plus a focused fix at the failing boundary, not closing this as implemented because image attachment support exists.
