## Summary

Keep open. Current main still shows WebChat paste flowing through a browser side-payload store into `chat.send` and Gateway attachment parsing, but the existing implementation and tests do not prove that the exact pasted image bytes are the bytes dispatched to the agent; the newer comments also add a cross-channel cached-image symptom that is not resolved by the earlier paste-support closure.

## What This Changes

Pasted or channel-delivered images should preserve the original image bytes end to end instead of delivering a stale or different image in WebChat/Control UI and openclaw-weixin.

## Best Possible Solution

Add a regression that compares original clipboard/channel image bytes with the bytes stored in the UI payload store, serialized into `chat.send`, parsed by Gateway, and dispatched to the agent, then fix the smallest failing boundary in the shared attachment/media path. Keep external WeChat-specific changes out of core unless the regression proves a missing generic plugin/media contract.

## Reproduction Assessment

Yes. The reporter gives concrete macOS Chrome WebChat paste steps and a second openclaw-weixin image path, and current main has a clear instrumentation path through `handlePaste`, the side payload store, `buildApiAttachments`, `parseMessageWithAttachments`, and `dispatchInboundMessage`; this review did not run the live browser/channel repro.

## Solution Assessment

Unclear. The best direction is a narrow byte-identity regression plus the smallest code fix at the failing boundary; closing as implemented because paste UI support exists would not solve the reported stale/wrong-image delivery.
