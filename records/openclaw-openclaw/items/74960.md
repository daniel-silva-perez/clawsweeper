## Summary

Keep open. The report is a concrete current regression about stale or substituted image bytes, and current main still has the same Control UI paste handler, side payload store, chat.send serialization, and Gateway attachment parsing path without a regression that proves pasted bytes are preserved through dispatch. The prior paste-support closure only proves images can be attached; it does not cover the byte-identity failure described here.

## What This Changes

The issue reports that pasted WebChat images, and reportedly WeChat image messages, are delivered to the agent as stale or wrong image bytes in v2026.4.26 and v2026.4.27.

## Best Possible Solution

Add a focused byte-identity regression that follows one pasted image from Clipboard/FileReader through the side payload store, chat.send serialization, Gateway parsing, media persistence, and dispatch images, then fix the smallest failing shared boundary. Keep the prior paste-support work closed, but track this as a separate data-integrity bug.

Yes. The issue gives concrete macOS Chrome WebChat paste steps and a second channel path, and current source provides a clear instrumentation path through handlePaste, getChatAttachmentDataUrl, buildApiAttachments, and parseMessageWithAttachments; this review did not execute the live browser/channel repro.

## Solution Assessment

Yes for the triage direction: a byte-identity regression plus the smallest pipeline fix is the maintainable path. Closing as implemented because paste attachment UI exists would not solve the stale/wrong-image delivery described by the reporter.
