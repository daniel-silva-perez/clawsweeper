## Summary

Keep open for normal PR review. Current main still rejects arbitrary extra bootstrap basenames, so this PR is not obsolete, but the proposed patch has a concrete glob-detection bug and overlaps the broader open session-scoped bootstrap PR #74735.

## What This Changes

The PR removes the fixed bootstrap basename allowlist for the bundled bootstrap-extra-files hook, widens extra bootstrap file names to strings, updates docs/changelog copy, and adds focused tests for arbitrary workspace-contained extra files.

## Best Possible Solution

Land one coherent bootstrap-extra-files change that removes the basename allowlist while preserving workspace containment, realpath checks, budget handling, deterministic ordering, and subagent filtering; before merge, fix bracket-only glob detection and decide whether this narrow PR or #74735 should be the canonical implementation.

## Reproduction Assessment

Yes. Current main can be checked by configuring `bootstrap-extra-files` with an arbitrary basename such as `hooks/REGISTRY.yaml`, which source inspection shows is rejected as `invalid-bootstrap-filename`; the PR edge case is reproducible with a bracket-only Node glob such as `src/agents/[wx]orkspace.ts`.

## Solution Assessment

No, not as currently written. Removing the basename allowlist is the right narrow shape for #39120, but the PR should classify `[` as glob syntax or otherwise delegate glob detection consistently before merge, and maintainers should compare it with #74735.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Treat bracket-only patterns as globs:** `src/agents/workspace.ts:186-188`
  - body: `hasGlobPattern()` does not check for `[`, so a valid Node glob like `docs/[ab].md` is treated as a literal path unless it also contains `*`, `?`, or `{`. That silently skips bracket-only glob patterns while this PR is broadening extra bootstrap file loading.
  - confidence: 0.92
