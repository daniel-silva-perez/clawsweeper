## Summary

Keep open. Current main still matches the reported read-only `/workspace` behavior for `workspaceAccess: "none"`, and the remaining product question is not fully covered by the related writable-custom-binds feature request. The Docker setup path also still defaults sandbox-enabled installs to `workspaceAccess: "none"`, so this can affect the documented Docker sandbox flow.

## Best Possible Solution

Keep this open for sandbox maintainer triage. The best fix is to make the `workspaceAccess: "none"` contract explicit and consistent: either hide/disable write-capable tools and avoid noisy bootstrap write attempts, or make the per-scope sandbox workspace writable while still keeping the real agent workspace off-limits. For Docker gateway/DooD installs, also verify that setup or docs cover a host-visible `sandbox.workspaceRoot`/volume-parity path so sandbox containers do not bind an empty host path. Coordinate with #59718 for the narrower writable-custom-bind case instead of closing this issue prematurely.
