## Summary

Keep open. Current main still matches the reported Docker sandbox failure path: Docker sandbox bootstrap sets `workspaceAccess: "none"`, that mode uses an isolated sandbox workspace but mounts it read-only, and the filesystem bridges reject writes unless access is `"rw"`. Related #59718 covers writable custom binds and #65316 covered tool-surface mismatch; neither fully resolves this Docker `/workspace` permission report.

## What This Changes

The issue reports that Docker sandbox sessions with `workspaceAccess: "none"` cannot read/write expected `/workspace` files and produce read-only sandbox filesystem errors.

## Best Possible Solution

Resolve the sandbox contract explicitly: either make only the isolated per-scope sandbox `/workspace` writable for `workspaceAccess: "none"` while keeping the real agent workspace hidden, or keep `none` read-only and suppress/diagnose write-capable file tools for that mode. The fix should cover Docker/browser mounts, local and remote filesystem bridges, docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from current main: Docker sandbox setup selects `workspaceAccess: "none"`, `appendWorkspaceMountArgs` emits a read-only `/workspace` bind, and `fs-bridge` rejects writes unless access is `"rw"`; this matches the reporter's logged `Sandbox path is read-only` failure.

## Solution Assessment

Unclear. Making the isolated sandbox workspace writable is likely the closest fix for this report and related #37634, but it is a sandbox permission change that needs secops review; the safer alternative if maintainers keep `none` read-only is to hide mutating tools and document that contract clearly.
