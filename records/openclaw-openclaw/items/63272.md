## Summary

Keep open. Current main still lacks the requested `openclaw skills uninstall <slug>` CLI command and still has no current-main helper that removes both a workspace skill directory and its `.clawhub/lock.json` entry; the related closed request and closed prior PR do not implement this surface, and the referenced open gateway PR is a different RPC path rather than the requested CLI command.

## What This Changes

Add an `openclaw skills uninstall <slug>` command that safely removes a ClawHub/workspace skill directory and the matching `.clawhub/lock.json` entry, with dry-run, confirmation, JSON, and agent-workspace behavior.

## Best Possible Solution

Implement the CLI-facing inverse of ClawHub skill install/update in the skills subsystem: resolve the active agent workspace, plan and display the skill directory plus lockfile entry, safely remove the workspace skill directory when requested, delete only that slug from `.clawhub/lock.json`, support `--yes`, `--dry-run`, `--json`, and `--agent`, document the command, and reuse any landed backend primitive from #69319 if it matches these semantics.

## Reproduction Assessment

Yes. The missing feature is reproducible by static current-main inspection: the skills CLI registration and public docs list no uninstall command, and relevant source searches find no skills uninstall helper or gateway method.

## Solution Assessment

Yes, with one sequencing caveat. The proposed CLI/workspace scope is the narrow maintainable fix for the reported lockfile reappearing problem, but it should share the backend removal primitive with #69319 if that PR lands first.
