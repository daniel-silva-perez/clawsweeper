## Summary

Keep this PR open. Current main still does not surface SQLite FTS5 availability through `openclaw doctor`, while the PR addresses a real doctor-visibility gap for the still-open FTS5 issue. The branch is now stale against current main's skipped-probe fix, so it should not merge as-is without preserving that newer `skipped` behavior.

## What This Changes

The PR branch adds an optional `fts` status object to `doctor.memory.status`, carries it into the doctor memory probe, and emits a targeted `openclaw doctor` warning when gateway memory search reports FTS5 unavailable.

## Best Possible Solution

Port the PR's narrow FTS visibility change onto current main: expose sanitized `fts` status from `doctor.memory.status`, carry it through `GatewayMemoryProbe`, render one targeted doctor warning when FTS is enabled but unavailable, and keep current skipped-probe semantics and regression coverage intact.

Yes. A high-confidence unit reproduction is to mock the memory manager status with `fts: { enabled: true, available: false, error: "no such module: fts5" }` and verify `doctor.memory.status` plus `noteMemorySearchHealth` surface that warning; current main lacks that thread.

## Solution Assessment

No, not as-is. The implementation direction is the narrow maintainable fix for doctor visibility, but the branch should be updated on current main so `probeGatewayMemoryStatus` adds `fts` while preserving `payload.embedding.checked !== false` and the `skipped` discriminator.
