## Summary

Keep open. Current main still does not implement the reported large memory reindex resilience: transient transport failures such as `TypeError: fetch failed` are not classified as retryable in the memory embedding retry policy, async sync warnings still flatten errors with `String(err)`, and failed atomic reindexes still delete the temp database rather than preserving resumable progress. Related PR #56879 was closed unmerged; #44167 is an open implementation candidate for the retry/split subset, but it has not landed and does not cover the logging/resume pieces. #65782/#65783 are related memory indexing work, but they target surrogate-pair chunking rather than this transport/reindex rollback failure.

## Best Possible Solution

Keep this issue open. Review and land #44167 or an equivalent memory-core retry/split fix for transient embedding transport failures, then address the issue-specific diagnostics and progress-loss pieces: use redacted stack/cause-aware formatting for async memory sync warnings, and add a checkpoint/resume or temp-index preservation design that keeps atomic swap semantics safe. Add focused tests for `TypeError("fetch failed")` with nested transport causes and for the chosen failed-reindex recovery behavior.
