## Summary

Keep open. The related #56879 PR identified the same retry/logging gap but was closed unmerged, and current main still does not fully address the reported large reindex failure: transient transport errors like `TypeError: fetch failed` are not classified for memory embedding retry, async sync warnings still flatten errors with `String(err)`, and failed atomic reindexes still delete the temp database rather than preserving resumable progress. The later #65782/#65783 surrogate-pair work is related memory-indexing context but tracks a different root cause and explicitly leaves this retry-policy gap separate.

## Best Possible Solution

Keep this issue open as the canonical memory embedding transport-failure/reindex-resume bug. The likely fix belongs in the memory-core embedding policy and sync paths: add bounded retry classification for transient fetch/undici/network causes, log redacted cause or stack detail for async sync failures, and either add resumable checkpointing or preserve enough completed embedding work to avoid restarting a large failed reindex from zero. Add focused tests for `TypeError("fetch failed")` with nested transport causes and for the chosen temp-index recovery semantics.
