## Summary

Keep open. Current main still implements `gateway.bind=lan` as an IPv4-only wildcard listener, and the related closed report adds concrete `ss`/tcpdump evidence for the same IPv6 connection failure, so this remains valid core gateway work rather than stale cleanup.

## What This Changes

The issue requests IPv6 listening support when the OpenClaw Gateway is started in LAN bind mode.

## Best Possible Solution

Implement IPv6-capable LAN listener planning in the gateway core, with deterministic fallback when IPv6 is unavailable, while keeping non-loopback auth requirements and docs aligned with the new contract.

## Reproduction Assessment

Yes. Current main can be verified statically: `gateway.bind=lan` resolves to `0.0.0.0`, non-loopback listen hosts are not expanded to `::`, and existing tests assert that behavior; the related #63743 report describes the live `ss`/tcpdump failure path.

## Solution Assessment

Yes, this belongs in the core gateway bind/listen path rather than a plugin. The safer implementation is a listener-planning change with explicit IPv6 fallback semantics, not accepting raw `::` as a legacy `gateway.bind` alias.

## Repair Work Prompt

Implement IPv6 listening support for OpenClaw Gateway `gateway.bind=lan`. Current main resolves LAN to `0.0.0.0` and only starts a single IPv4 wildcard listener; related #63743 has live `ss`/tcpdump evidence of IPv6 connections failing. Keep the fix scoped to gateway bind/listen planning and docs. Prefer a deterministic listener plan that supports IPv6 LAN wildcard (`::`) while preserving IPv4 reachability, handles platforms where `::` also accepts IPv4 versus `ipv6Only` behavior without spurious EADDRINUSE failures, and falls back cleanly if IPv6 binding is unavailable. Preserve existing non-loopback auth requirements, Control UI origin checks, trusted-proxy behavior, Tailscale Serve/Funnel loopback constraints, and loopback `127.0.0.1`/`::1` behavior. Update focused tests and user-facing docs/schema help as needed. Add a changelog entry only if this changes shipped user-facing behavior, crediting `Thanks @SandyLing` and, if using #63743 evidence, `Thanks @xinghuoseo`; do not credit bots.
