## Summary

Keep this issue open. The adjacent #58520 registration/label-resolution and #51972 delivery-routing reports have fixes, but current main still has the specific TUI reconnect/backfill gap described here: the terminal TUI does one history load on reconnect, only reports sequence gaps, and does not use the session transcript subscription/backfill path available to Gateway clients.

## Best Possible Solution

Keep the issue open and implement TUI reconnect/backfill parity in `src/tui`: reload or subscribe/backfill the current session transcript after gateway sequence gaps and after reconnect, with a small delayed post-reconnect pass if needed. Add focused regression coverage for `onGap`, reconnect timing, and a post-restart heartbeat reply becoming visible without clobbering active runs.

## Risks / Open Questions

- A naive reload on every gap or terminal event could erase active TUI stream state or optimistic user messages; any fix should preserve the existing active-run and concurrent-run guards.
- If TUI adopts session transcript subscriptions, it should subscribe/unsubscribe on session changes and still handle reconnects without duplicate history rendering.
