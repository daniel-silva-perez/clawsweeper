## Summary

Keep this open. Current main has a related active-run reconnect watchdog recovery from #74224, but it still does not backfill the watched TUI session after a Gateway sequence gap or after a reconnect when no active run is being tracked, which is the vulnerable window described for a post-restart heartbeat reply that lands after the immediate history load.

## What This Changes

The issue requests that the TUI show the same post-restart heartbeat recovery reply that is already visible in Web UI after a Gateway restart.

## Best Possible Solution

Add TUI-owned reconnect/backfill parity for the currently watched session: on Gateway sequence gaps and after reconnect, safely backfill the current transcript or subscribe to `session.message` updates, with debouncing and active-run guards so missed heartbeat replies appear without disturbing in-flight local runs.

## Reproduction Assessment

Yes. A focused reproduction can drive a TUI reconnect or sequence gap where the heartbeat reply is persisted after the immediate reconnect `loadHistory()` but no matching live `chat` event reaches the TUI; current handlers show no recovery path for that no-active-run/gap case.

## Solution Assessment

No, current main is only a partial solution. The merged #74224 active-run watchdog path is useful, but the narrowest complete fix is a TUI backfill/subscription path for gaps and post-reconnect idle sessions.
