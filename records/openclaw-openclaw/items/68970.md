## Summary

Keep this open. Current main still has the vulnerable TUI reconnect/gap window described in the report: the TUI does one immediate history load after reconnect, only surfaces sequence gaps as status text, and does not subscribe to transcript updates that could backfill a heartbeat reply which lands after that first reload.

## What This Changes

The issue requests that the TUI show the same post-restart heartbeat recovery reply that is already visible in Web UI after a Gateway restart.

## Best Possible Solution

Add TUI-owned reconnect/backfill parity for the currently watched session: recover on Gateway sequence gaps and post-reconnect idle windows by safely reloading the transcript or subscribing to transcript events, while preserving active-run streaming and pending optimistic-message behavior.

## Reproduction Assessment

Yes. The issue body gives concrete restart steps, and current-main source gives a deterministic focused reproduction path: make a TUI reconnect or sequence gap, persist the heartbeat reply after the immediate reconnect `loadHistory()`, and drop the matching live `chat` event; there is no TUI recovery path for that window.

## Solution Assessment

Yes, the requested direction is the right boundary. This should be fixed in the TUI reconnect/gap backfill path rather than changing heartbeat scheduling or core session routing, because Web UI already sees the persisted reply and current TUI lacks the parity mechanism.

## Repair Work Prompt

Fix openclaw/openclaw#68970. The observable bug is that after a Gateway restart, a post-restart heartbeat/recovery reply can be persisted and visible in Web UI but never appear in the TUI session the user was watching. Current main only does one immediate `loadHistory()` on reconnect, `client.onGap` only reports the gap, and merged #74224 covers reconnect-stale active runs but not no-active-run post-reconnect heartbeat replies. Implement a TUI-owned recovery path in `src/tui`: on Gateway sequence gaps and after reconnect, safely backfill the current session transcript, or use the existing `sessions.subscribe` / `sessions.messages.subscribe` / `session.message` protocol if that is the cleaner local design. Debounce duplicate reloads and preserve active-run streaming state, pending optimistic user messages, local BTW handling, and the #74224 reconnect watchdog behavior. Do not change heartbeat scheduling, core session routing, or Gateway protocol unless a minimal additive TUI client seam is required. Add focused regression tests for `onGap` triggering backfill, reconnect with no active run receiving a reply after the immediate history load, and the existing active-run reconnect path not double-loading. Add a user-facing changelog fix entry with reporter credit (`Thanks @Silronin`) if behavior changes.
