## Summary

Keep open. Current main still has the resolver bug the PR is trying to fix, and the PR’s user-entry direction is focused, but the latest diff still has a blocking channel-kind semantics issue: it converts matched `channel` directory entries to `group` even though current contracts and outbound routing distinguish those kinds.

## What This Changes

The PR changes outbound target resolution to derive directory-match target kind from the matched entry, adds a helper, and adds regression coverage for `listGroups` returning a user entry plus a channel-entry normalization case.

## Best Possible Solution

Land the narrow resolver/test fix after preserving authoritative directory entry kind for `user`, `group`, and `channel`. If maintainers intentionally want `channel` entries normalized to `group`, that policy should be documented at the resolver/session-routing contract level and covered with routing tests for plugins that support both kinds.

## Reproduction Assessment

Yes. A focused unit scenario can reproduce the current-main bug by returning `{ kind: "user", id: "staff_bob", name: "Bob" }` from `listGroups` for bare input `Bob`; current main returns the guessed `group` kind. The PR’s remaining channel issue is likewise reproducible with a `{ kind: "channel" }` directory entry and a plugin that supports both `group` and `channel`, where the PR routes it as `group`.

## Solution Assessment

No, not as currently written. The PR is the right narrow surface for the user/group bug, but the helper should preserve `entry.kind === "channel"` rather than collapsing it to `group`, or a maintainer needs to explicitly approve and document that normalization across routing contracts.
