## Summary

Keep this PR open, but do not merge the current branch. The underlying compaction-chain history gap is real, yet the PR head is dirty against main and introduces blocking regressions around active-branch transcript reads, reset/restore epochs, checkpoint boundaries, and uncontained transcript path reads.

## What This Changes

The PR adds a chained transcript resolver in `src/gateway/session-utils.fs.ts` plus tests and a production-derived fixture so `readSessionMessages` can read main-session compaction checkpoint snapshots and the live transcript as one continuous history.

## Best Possible Solution

Keep the PR open as useful evidence, but require a rebase and redesign before merge: preserve current `SessionManager` active-branch reads, model checkpoint segments by boundary and reset/restore epoch, use the existing safe transcript path resolver/containment rules, and keep reset-archive opt-in work coordinated with #60409 and #73883 rather than folding it silently into default `chat.history`.

## Reproduction Assessment

Yes. The PR head includes explicit red-bar reset/restore tests, and the source path is clear: current reset/restore flows preserve checkpoint metadata while the PR loops over every `preCompaction.sessionFile` and then reads every selected JSONL line.

## Solution Assessment

No. The chained-read direction addresses a real gap, but this branch is not the best solution because it bypasses current active-branch filtering, replays stale epochs and full snapshots, and trusts raw checkpoint paths instead of the established transcript resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve active-branch transcript reads:** `src/gateway/session-utils.fs.ts:287-288`
  - body: Current main reads tree-format transcripts through `SessionManager.getBranch()` so abandoned rewrite branches stay out of `chat.history`. This replacement loops over every JSONL line in each selected file, which would reintroduce stale/duplicate abandoned entries and fail the current active-branch regression test.
  - confidence: 0.93
- **[P1] Stop replaying stale reset and restore epochs:** `src/gateway/session-utils.fs.ts:212-224`
  - body: This loop prepends every stored `preCompaction.sessionFile`. Reset keeps `compactionCheckpoints[]` on the new entry, and restore writes a fork of a checkpoint while preserving the checkpoint list, so this will resurface pre-reset messages and double-read restored content in default `chat.history`.
  - confidence: 0.92
- **[P1] Trim snapshots at checkpoint boundaries:** `src/gateway/session-utils.fs.ts:221-223`
  - body: Each checkpoint records `firstKeptEntryId` because the post-compaction transcript keeps an unsummarized tail after that boundary. Concatenating whole pre-compaction snapshots with the live transcript duplicates retained tail messages and gives incorrect sequence counts across normal multi-compaction chains.
  - confidence: 0.86
- **[P2] Constrain transcript paths before reading them:** `src/gateway/session-utils.fs.ts:197-199`
  - body: `pushFile` trusts paths from `sessions.json` with only `path.resolve()` and `existsSync()`. The existing transcript resolver rejects paths outside the sessions directory or recognized agent-session layouts, so corrupted/stale checkpoint metadata can make `chat.history` ingest unrelated readable JSONL files.
  - confidence: 0.9
- **[P2] Prefer the canonical matching store row:** `src/gateway/session-utils.fs.ts:140-144`
  - body: The store scan returns the first row with a matching `sessionId` and optional `sessionFile`. In stores with legacy alias duplicates, an earlier stale/non-main row can stop the scan before the canonical main entry with the checkpoint chain is seen, leaving the original truncation bug unfixed.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces concrete chat-history privacy risks by reading uncontained checkpoint paths and by silently replaying reset/restore-cleared transcript content.

Concerns:

- **[medium] Uncontained checkpoint transcript reads:** `src/gateway/session-utils.fs.ts:197`
  - body: The new chained path accepts `preCompaction.sessionFile` and `entry.sessionFile` with only `path.resolve()` plus existence checks, bypassing the existing sessions-directory containment logic used by transcript candidate resolution.
  - confidence: 0.9
