## Summary

Keep open. Current main contains watchdog and stale-final mitigations, but it still has no prompt content-complete or finalizing path; the TUI continues to present chat deltas as `streaming` until terminal chat/lifecycle handling or a timeout fallback. The merged related PR #73749 explicitly left this issue and #67053 open as remaining content-completion timing work.

## What This Changes

The issue asks for `openclaw tui` to stop showing `streaming` promptly when visible assistant content finishes, rather than waiting 30-120 seconds for delayed gateway finalization.

## Best Possible Solution

Add a compatible content-completion or finalizing path that lets the TUI stop presenting visible assistant answer generation as active `streaming` once assistant-visible content is complete, while still tracking the run until terminal final/error/abort cleanup finishes. Keep the watchdog as recovery, update typed protocol/docs if a new event or field is added, and cover delayed lifecycle finals, concurrent runs, hidden/tool-only output, local `/btw`, and reconnect fallback behavior.

## Reproduction Assessment

Yes. The source-level reproduction is an active TUI run that receives assistant `delta` events, then stops producing visible text while backend lifecycle terminal handling waits on flush/persistence work before `chat` final arrives; the status remains `streaming` until that terminal path or the watchdog fires.

## Solution Assessment

Yes. A narrow TUI/gateway state-machine change is the maintainable direction; a broad immediate idle shortcut would be riskier because it could hide active runs, tool-only work, or delayed final/error state.
