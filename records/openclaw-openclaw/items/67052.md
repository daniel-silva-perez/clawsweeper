## Summary

Keep open. Current main still has timeout and unbound-final mitigations for stale TUI streaming state, but it does not add the requested prompt content-completion or finalizing transition; visible assistant deltas still mark the TUI as `streaming` until a terminal chat/lifecycle path or the 30s watchdog recovery. The related merged PR #73749 explicitly left this timing work open, and the later same-author duplicate #67053 should not make this older report close.

## What This Changes

The issue asks for the TUI to stop showing `streaming` promptly after visible assistant response content finishes instead of waiting on the later gateway final lifecycle/chat event.

## Best Possible Solution

Add a compatible content-completion or finalizing path that lets the TUI stop presenting visible assistant text generation as active `streaming` once assistant-visible content is complete, while still tracking the run until terminal final/error/abort handling finishes. Preserve the watchdog as recovery, keep concurrent-run and local `/btw` safeguards, update typed protocol/docs if a new event or payload field is added, and cover delayed lifecycle finals, tool-only/hidden output, late text, reconnect fallback, and duplicate #67053 in tests.
