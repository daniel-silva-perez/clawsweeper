## Summary

Keep open. Current main has two mitigations for stale TUI streaming state, the 30s watchdog from #67401 and the unbound-final cleanup from #73749, but it still does not implement the requested prompt transition when visible assistant content is complete. Source and tests still show `streaming` is driven by chat deltas until a terminal chat/lifecycle path or watchdog recovery, while lifecycle terminal emission can wait for async flush work.

## What This Changes

The issue asks for `openclaw tui` to stop showing `streaming • elapsed` promptly after visible assistant response content finishes, instead of waiting on later lifecycle/chat-final completion.

## Best Possible Solution

Add a compatible content-completion/finalizing path that lets the TUI stop presenting visible assistant text generation as active streaming while the run may still be finalizing. Keep the watchdog as recovery, update the typed protocol/docs if a new event or state is added, and cover delayed lifecycle finals, concurrent active runs, tool-only or hidden deltas, and reconnect fallback behavior.

## Repair Work Prompt

Fix the TUI streaming indicator timing for #67052 and the duplicate/open sibling #67053. Observable bug: after the last visible assistant response content arrives, `openclaw tui` can continue showing `streaming • elapsed` until lifecycle/chat-final completion or the 30s watchdog. Implement one focused PR that decouples visible content completion from full run cleanup. Preferred direction: add a compatible content-complete/finalizing signal or state through the typed Gateway protocol and emit it when model-visible assistant content is complete, before async block/channel flush and lifecycle terminal work; then teach the TUI to leave the `streaming` busy state or show a distinct finalizing/idle state while still tracking the active run. Keep the existing watchdog as a recovery fallback. Preserve concurrent-run behavior, reconnect handling, local `/btw`, abort/error paths, tool-only output, hidden/reasoning-only deltas, and late tool events. Do not add plugin-specific policy or remove existing final/watchdog cleanup. If the wire protocol changes, update schema, docs, and tests. Add a user-facing changelog fix entry credited to @tokced unless maintainers decide this is internal-only.
