## Summary

Keep open. Current main still has stale-streaming mitigations, but it does not implement the requested prompt transition when visible assistant content is complete; the TUI still labels chat deltas as `streaming` until terminal chat/lifecycle handling or the watchdog path. The merged related PR #73749 explicitly left this issue and #67053 open as remaining content-completion/finalizing timing work.

## What This Changes

The issue asks for `openclaw tui` to stop showing the response as actively `streaming` promptly after the last visible assistant content token is delivered, instead of waiting through post-run finalization.

## Best Possible Solution

Add a compatible content-completion or finalizing path that lets the TUI stop presenting visible answer generation as active `streaming` once assistant-visible content is complete, while still tracking the run until terminal final/error/abort handling finishes. Preserve the existing watchdog as recovery, update typed protocol/docs if a new event or payload field is added, and cover delayed lifecycle finals, concurrent runs, hidden/tool-only output, local `/btw`, and reconnect fallback behavior.
