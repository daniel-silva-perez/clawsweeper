## Summary

Keep open. Current main still routes the reported first-run Windows onboarding screen through the Clack-backed interactive confirmation prompt, with no OpenClaw-side Windows/plain-input fallback. The report is sparse, but native Windows remains documented as supported, and the refreshed Microsoft/Windows tracker still lists this as unresolved.

## What This Changes

The issue reports that Windows 11 installation/onboarding can freeze at the first security acknowledgement prompt after installation via the shell installer path.

## Best Possible Solution

Reproduce first-run onboarding on Windows 11 in PowerShell/Windows Terminal and any reporter-relevant Git Bash/MSYS path, then choose the narrowest fix: a prompt-layer fallback if Clack input is broken on supported native Windows, or installer/docs guidance if the failure is limited to an unsupported shell path.

## Reproduction Assessment

No. Source inspection identifies the exact onboarding prompt path, but the report lacks concrete terminal details and I did not live-run the Windows 11 onboarding flow.

## Solution Assessment

Unclear. A Windows/plain-input fallback may be right if the supported native Windows prompt path is broken, but clarifying install.ps1 or WSL2 guidance is safer if the hang only occurs through an unsupported install.sh-on-native-Windows shell.
