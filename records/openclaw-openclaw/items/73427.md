## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI origin, the browser WebRTC transport posts SDP directly to OpenAI, the OpenAI browser-session payload only configures output voice, and the WebRTC data-channel handler drops OpenAI error and lifecycle events so the UI can fall through to a non-actionable live state.

## What This Changes

The issue requests fixing Control UI OpenAI Realtime Talk so CSP allows the browser WebRTC call path and the UI surfaces realtime audio/session errors and lifecycle status.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the existing Control UI CSP, OpenAI realtime provider, WebRTC transport, and Talk status paths: allow the required Realtime call origin through a tight provider-aware mechanism, include the needed browser audio/VAD session settings, surface OpenAI `error` and basic speech/response lifecycle events, and cover the behavior with targeted tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 steps, and current main statically reproduces the blockers by combining a direct browser SDP POST to OpenAI with a CSP that omits that origin; the silent-error path is visible because the data-channel handler returns on unrecognized OpenAI events while the UI can still render `Talk live`.

## Solution Assessment

Yes. The narrow maintainable fix is in the existing bundled OpenAI browser Talk path, with a tight CSP allowance or provider-declared connect-src seam and explicit realtime event/status handling; the broader configurable-CSP request in #47565 would not address the swallowed OpenAI runtime errors.
