## Summary

Keep open. Current main still has the reported OpenAI browser Talk failure modes: Control UI CSP does not allow the OpenAI HTTPS origin used by the WebRTC SDP POST, the browser data-channel handler ignores OpenAI error and lifecycle events, and the OpenAI browser-session payload still configures output voice only. The closed Google Live Talk work is a separate transport path and does not resolve this OpenAI WebRTC/CSP issue.

## What This Changes

The issue requests fixing Control UI OpenAI Realtime Talk so CSP permits the WebRTC SDP exchange, realtime audio errors and lifecycle status are surfaced, and browser sessions include explicit input audio settings.

## Best Possible Solution

Ship a narrow OpenAI browser Talk fix that keeps standard provider API keys server-side, either permits only the required OpenAI realtime HTTPS origin in Control UI CSP or proxies the SDP exchange through the Gateway, surfaces OpenAI data-channel errors and basic speech/response lifecycle status, and adds focused tests/docs without rolling in the broader configurable-CSP request or Google Live behavior.

Yes.
