## Summary

Keep open. Current main still has both reported OpenAI browser Talk failure surfaces: the Control UI CSP does not allow the OpenAI Realtime SDP POST origin, and the OpenAI WebRTC data-channel handler drops provider error and lifecycle events instead of surfacing an actionable status.

## What This Changes

Make Control UI Realtime Talk support the bundled OpenAI WebRTC provider by allowing the OpenAI SDP exchange, configuring browser session audio input/VAD, and surfacing realtime errors and lifecycle status.

## Best Possible Solution

Keep this issue open and land a focused Control UI/OpenAI provider fix that allows only the required OpenAI Realtime connection, aligns the browser session payload with the realtime audio input contract, and maps provider errors/lifecycle events into the existing Talk status UI.

## Reproduction Assessment

Yes. The CSP failure is reproducible from current source alone because `connect-src` omits `https://api.openai.com` while the WebRTC transport fetches that origin; the hidden-error half is reproducible by sending an OpenAI `error` or lifecycle event through the current data-channel handler and observing that it falls through with no status update.

## Solution Assessment

Yes, the narrowest maintainable fix is in the current owner surfaces: `src/gateway/control-ui-csp.ts`, `extensions/openai/realtime-voice-provider.ts`, and `ui/src/ui/chat/realtime-talk-webrtc.ts`, with targeted tests. A broad user-configurable CSP surface should remain out of scope for this bug.
