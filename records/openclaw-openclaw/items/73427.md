## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI Realtime origin, the browser WebRTC transport posts its SDP to that origin, the OpenAI browser session payload still only configures output audio, and the WebRTC data-channel handler still drops OpenAI `error` and lifecycle events. The related Google Live report is adjacent, not a duplicate of this OpenAI WebRTC path.

## What This Changes

The issue asks for Control UI OpenAI Realtime Talk to allow the OpenAI WebRTC call endpoint through CSP, configure browser audio input/VAD settings, and surface realtime error/lifecycle events instead of hiding them.

## Best Possible Solution

Land a focused OpenAI browser Talk fix: add a tight Control UI CSP allowance for the OpenAI Realtime WebRTC call origin, mint browser client secrets with the needed input audio/VAD/transcription settings, map OpenAI `error` and basic speech/response lifecycle events into Talk status, and cover those paths with targeted tests and a changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 reproduction steps, and current main statically reproduces the first blocker by combining a direct OpenAI SDP POST with a CSP that omits that origin; the silent-error path is visible in the current data-channel handler's default drop behavior.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the existing bundled OpenAI browser Talk path in CSP, provider session creation, and WebRTC event handling; official OpenAI docs support the SDP endpoint and server-event/VAD contract that the UI should account for. ([platform.openai.com](https://platform.openai.com/docs/guides/realtime-webrtc))
