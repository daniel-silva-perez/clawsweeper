## Summary

Keep open. Current main still exposes the reported OpenAI browser Talk failure surfaces: the Control UI CSP does not permit the direct OpenAI SDP POST, the WebRTC data-channel handler drops diagnostic/lifecycle events, and the browser-session payload only configures output voice. The closed Google Live Talk item is a different transport path and does not supersede this OpenAI WebRTC issue.

## What This Changes

The issue requests fixing Control UI's OpenAI Realtime Talk browser path by allowing or proxying the WebRTC SDP connection, adding explicit audio input/VAD session config, and surfacing realtime errors and lifecycle status.

## Best Possible Solution

Ship a narrow OpenAI browser Talk fix that keeps standard provider API keys server-side, permits only the required OpenAI realtime HTTPS origin or proxies the SDP exchange through the Gateway, surfaces OpenAI data-channel errors and basic speech/response lifecycle status, and adds focused regression tests without rolling in broader configurable-CSP or Google Live changes.

## Reproduction Assessment

Yes. The issue has concrete steps, and current source independently reproduces the failure path: Control UI sends a direct browser SDP POST to an origin absent from CSP, then ignores the diagnostic events that would explain post-connection failures.

## Solution Assessment

Yes, if implemented narrowly. The maintainable fix is to repair the bundled OpenAI browser Talk path and tests; a broad configurable CSP feature should remain separate, and a server-side proxy is the safer alternative if maintainers do not want to add `api.openai.com` to browser `connect-src`.
