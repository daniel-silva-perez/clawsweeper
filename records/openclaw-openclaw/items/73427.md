## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP does not allow the OpenAI WebRTC SDP endpoint, the OpenAI browser session payload lacks explicit input/VAD/transcription audio settings, and the WebRTC data-channel handler drops OpenAI error and lifecycle events that should inform the UI.

## What This Changes

Fix Control UI OpenAI Realtime Talk by allowing the OpenAI WebRTC endpoint in CSP, configuring browser realtime audio input/VAD settings, and surfacing OpenAI data-channel error and lifecycle events.

## Best Possible Solution

Land a focused OpenAI browser Talk fix: add a narrowly scoped CSP allowance for the OpenAI Realtime WebRTC call origin, mint browser client secrets with explicit input audio/VAD/transcription settings, and map OpenAI error/VAD/output lifecycle events into user-visible Talk status without changing broader CSP configuration policy.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 reproduction steps, and current main statically reproduces the first blocker because the WebRTC transport posts to OpenAI while CSP omits that origin; the silent-error path is also visible in the current event handler's default drop behavior.

## Solution Assessment

Yes. The narrow maintainable direction is to fix the bundled OpenAI browser Talk path in the CSP builder, OpenAI realtime provider, and WebRTC transport, with targeted tests; a broad configurable CSP surface would be a separate product/security-policy decision.
