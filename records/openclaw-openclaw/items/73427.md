## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: Control UI CSP omits the OpenAI origin, the browser WebRTC transport posts SDP directly to that origin, the OpenAI browser-session payload only configures output voice, and the data-channel handler still drops OpenAI error and lifecycle events so the UI can fall through to a non-actionable live state. The related Google Live report #73601 is adjacent browser Talk work, not a duplicate of this OpenAI WebRTC path.

## What This Changes

Fix Control UI Realtime Talk for the bundled OpenAI browser provider by allowing the OpenAI WebRTC origin, configuring browser realtime audio input/VAD/transcription, and surfacing OpenAI realtime errors and lifecycle status.

## Best Possible Solution

Land a focused OpenAI browser Talk fix: add a tight `connect-src` allowance for the OpenAI Realtime origin, include the needed browser session audio input/VAD/transcription settings, surface OpenAI `error` plus basic speech/response lifecycle events through Talk status, and cover the CSP, provider payload, transport event handling, and UI status behavior with targeted tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 steps, and current main statically reproduces the blockers by combining a direct browser SDP POST to OpenAI with a CSP that omits that origin; the silent-error path is visible because the data-channel handler returns on unrecognized OpenAI events while the UI can still render `Talk live`.

## Solution Assessment

Yes. The narrow maintainable fix is in the existing bundled OpenAI browser Talk path rather than a broad core CSP configuration feature: the official OpenAI docs confirm the WebRTC call endpoint, data-channel server events, VAD lifecycle events, and error event handling expected by this flow.
