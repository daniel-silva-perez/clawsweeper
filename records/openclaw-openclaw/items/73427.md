## Summary

Keep open. Current main still has the reported OpenAI browser Talk failure modes: the Control UI CSP does not allow the HTTPS origin used by the browser SDP POST, the WebRTC event handler still drops OpenAI error and lifecycle events, and the OpenAI browser-session payload still configures output voice only. I found no open superseding PR, and the linked Google Live issue is a separate transport path.

## What This Changes

Fix Control UI OpenAI Realtime Talk so CSP permits the OpenAI WebRTC SDP exchange, browser sessions include input audio configuration, and realtime errors and lifecycle events surface in the UI.

## Best Possible Solution

Ship a focused OpenAI browser Talk fix that keeps standard OpenAI API keys server-side, permits only the required OpenAI realtime SDP origin or proxies that exchange through the Gateway, surfaces realtime data-channel errors and basic speech/response lifecycle status, maps existing OpenAI voice/VAD config into the browser client-secret session, and adds targeted regression tests without folding in broader configurable-CSP or Google Live work.

Yes. The issue gives concrete steps, and current source confirms the browser posts SDP to OpenAI while CSP does not allow that origin; after that, the current data-channel switch drops the diagnostic event types named in the report.

## Solution Assessment

Yes, the requested direction is the right repair scope if implemented narrowly. The maintainable path is an exact-origin CSP allowlist or Gateway proxy plus OpenAI event/payload handling and focused tests, not a broad configurable browser-security policy feature.
