## Summary

Keep open. Current main still matches the reported failure modes: Control UI CSP omits the OpenAI HTTPS origin used by the browser SDP POST, the OpenAI WebRTC data-channel handler still drops error and lifecycle events, and the OpenAI browser-session payload still configures output voice only. No open superseding PR was found in the available GitHub/search context, so this remains a narrow source-confirmed bug rather than cleanup-close material.

## What This Changes

Fix Control UI OpenAI Realtime Talk so CSP permits the OpenAI WebRTC SDP exchange, browser sessions include input audio configuration, and realtime errors/lifecycle events surface in the UI.

## Best Possible Solution

Ship a focused OpenAI browser Talk fix that keeps standard OpenAI API keys server-side, permits only the required OpenAI realtime SDP origin or proxies that exchange through the Gateway, surfaces realtime data-channel errors and basic speech/response lifecycle status, maps existing OpenAI voice/VAD config into the browser client-secret session, and adds targeted regression tests without folding in the broader configurable-CSP request or Google Live transport work.

## Reproduction Assessment

Yes. The issue includes concrete steps, and current main source confirms the high-confidence path: the browser directly posts SDP to OpenAI while CSP does not allow that origin, then the data-channel switch drops the diagnostic event types named in the report.

## Solution Assessment

Yes, the requested direction is the right repair scope if implemented narrowly. The safer implementation is an exact-origin CSP allowlist or Gateway proxy plus focused OpenAI event/payload handling and tests, not a broad configurable security-policy feature.
