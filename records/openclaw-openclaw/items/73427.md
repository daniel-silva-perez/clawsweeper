## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI WebRTC SDP origin, the OpenAI browser session payload lacks explicit input/VAD/transcription audio settings, and the WebRTC data-channel handler still drops error and lifecycle events. The related Google Live issue #73601 is closed, but that fix covers a different transport path and does not resolve this OpenAI WebRTC path.

## What This Changes

Fix Control UI OpenAI Realtime Talk by allowing the OpenAI WebRTC endpoint in CSP, configuring browser realtime audio input/VAD settings, and surfacing OpenAI data-channel error and lifecycle events.

## Best Possible Solution

Ship a focused OpenAI browser Talk fix in the existing owner surfaces: add the narrow OpenAI Realtime call origin to Control UI CSP, mint browser client secrets with explicit input audio/VAD/transcription settings, and map OpenAI error/VAD/output lifecycle events into user-visible Talk status.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 steps, and current main statically reproduces the first blocker because the WebRTC transport posts to OpenAI while CSP omits that origin; the silent-error path is visible in the current event handler's default drop behavior.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the CSP builder, bundled OpenAI realtime provider, and WebRTC transport with targeted tests; a broader configurable CSP surface is a separate product/security-policy decision tracked elsewhere.
