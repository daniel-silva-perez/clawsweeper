## Summary

Keep open. Current main still appears to have the CLI fallback retry gap: the fallback loop can retry with the same session/store state, the CLI path reuses an existing backend session id and still sends prompt text, and the orphaned trailing-user repair remains embedded-runner-only. The issue is narrow and actionable, with enough code and test surface to queue a focused repair PR.

## What This Changes

Add a CLI runner fallback retry repair so retries against resumed CLI backend sessions do not create duplicate user turns after rate-limit or timeout fallback.

## Best Possible Solution

Add a CLI-owned fallback retry repair that prevents resubmitting the same prompt into an already-mutated resumed backend session while preserving the original task through a fresh/reseeded session or backend-confirmed transcript repair, then cover timeout and rate-limit fallback retries with focused CLI regression tests.

## Repair Work Prompt

Fix openclaw/openclaw#54192. Current main lets `runWithModelFallback` call `runAgentAttempt` multiple times with the same session entry/store state, so a CLI fallback retry can reuse an existing `cliSessionId` while still sending prompt text again. Implement a CLI-owned fallback retry repair in the CLI attempt/runner boundary so timeout or rate-limit fallback retries do not append duplicate user turns into an already-mutated resumed backend session. Do not implement a blanket prompt-skip for resumed CLI sessions; preserve the original task by clearing/reseeding/resetting the backend session or by using a backend-confirmed transcript repair path. Add focused regression coverage for retries with an existing CLI session id, including timeout/rate-limit style failure, and assert the fallback still receives the task without resubmitting it into the same mutated backend session. Keep the fix scoped to the CLI runner/attempt path and shared helpers needed for that path; do not change embedded runner semantics.
