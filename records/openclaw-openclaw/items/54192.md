## Summary

Keep this issue open. Current main still has an embedded-runner orphaned trailing-user repair, but the CLI runner path still resumes a stored CLI backend session and submits fallback retry prompt text; it only clears the CLI session for missing Claude transcript content or explicit `session_expired`, not for timeout/rate-limit fallback retries after a prompt may already have been appended to an existing backend session.

## Best Possible Solution

Keep this issue open. Implement a CLI-owned fallback retry repair in `src/agents/command/attempt-execution.ts` or the CLI runner layer: when `isFallbackRetry` would reuse a stored CLI backend session after a timeout/rate-limit failure, detect the unsafe trailing-user/orphaned-turn state and either repair/branch it where supported or clear the stored CLI session binding and start a fresh/reseeded CLI session. Add focused regression coverage in `src/agents/command/attempt-execution.cli.test.ts` or `src/agents/cli-runner.reliability.test.ts` for a resumed CLI session whose first attempt fails after prompt submission and whose fallback does not append the same user task again to the backend session.

## Risks / Open Questions

- A naive prompt-skip fix could lose the task when the first CLI attempt failed before the backend actually appended the user turn.
- CLI backends have different resume semantics, so the repair likely needs backend-aware reset, branch, or reseed behavior rather than one generic argv/stdin change.
