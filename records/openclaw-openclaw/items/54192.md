## Summary

Keep open. Current main still has the CLI fallback retry gap: fallback retries can keep using the same reusable CLI backend session while still sending prompt text, and the explicit orphaned trailing-user repair remains embedded-runner-only.

## What This Changes

The issue requests a CLI-runner fallback retry repair so resumed CLI backend sessions do not get duplicate user turns after timeout or rate-limit fallback.

## Best Possible Solution

Add a CLI-owned fallback retry repair that avoids reusing a backend session after the first attempt may have appended the prompt, while preserving the original task through a fresh/reseeded CLI session or a backend-confirmed transcript repair. Cover timeout/rate-limit retries with focused regression tests.

## Repair Work Prompt

Fix the CLI fallback retry duplicate-user-turn bug tracked by openclaw/openclaw#54192. In the agent command / CLI runner path, when a fallback retry after timeout or rate-limit would reuse a stored CLI backend session id, do not resume that same backend session and submit the same prompt again. Prefer a CLI-owned repair that clears the reusable CLI session binding and reseeds a fresh CLI session from the OpenClaw transcript, or only skips/resumes when the backend-specific transcript check proves the prompt is already the last user turn. Preserve existing `session_expired` clearing behavior, Claude CLI missing-transcript handling, and the Claude CLI fallback prelude for cross-provider fallback. Add focused regression coverage in `src/agents/command/attempt-execution.cli.test.ts` and/or `src/agents/cli-runner.reliability.test.ts` for a resumed CLI session whose first attempt fails with timeout/rate-limit after prompt submission and whose fallback does not pass the same `cliSessionId` or append a duplicate backend user turn. Add a single-line changelog Fixes entry with `Thanks @mitchmcalister` unless maintainers classify this as internal-only. Do not change unrelated provider fallback policy or embedded runner orphan repair behavior.
