## Summary

Keep open. Current main still lacks the proposed failover hook surface, so this PR is not obsolete, but the proposed patch needs changes before merge: the terminal failure hook exposes the raw failure message without conversation-access gating, and two new fail-open tests instantiate the hook runner in fail-closed mode.

## What This Changes

The PR adds typed `model_failover` and `model_failure_terminal` plugin hooks, hook-runner methods, failover and terminal failure emission points, and focused tests for the new hook plumbing.

## Best Possible Solution

Keep the narrow core hook API if maintainers accept the product direction, but revise the terminal failure payload and gating first, add public plugin-hook documentation, fix the new tests to match the runner failure policy, and keep broader fallback notices or circuit-breaker behavior in the related follow-ups.

## Reproduction Assessment

Yes for the API gap and patch issues. Current main can be checked by searching the hook catalog/runner/emission points, and the PR diff shows both the new terminal payload emission and the fail-open tests that conflict with the runner defaults.

## Solution Assessment

No, not as currently written. The narrow hook direction is plausible, but the safest implementation should either avoid exposing raw terminal messages or require conversation-access opt-in, and the tests should explicitly enable fail-open behavior when asserting rejected handlers are swallowed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate terminal failure payloads behind conversation access:** `src/auto-reply/reply/agent-runner-execution.ts:1546`
  - body: This sends the raw terminal `message` to `model_failure_terminal`. For fallback summaries that message is built from per-attempt error text, while the PR does not add this hook to the conversation-access gated hook list, so non-bundled plugins could observe provider failure details without `allowConversationAccess`. Please either emit only sanitized/minimal fields or gate and document this hook like the existing conversation hooks.
  - confidence: 0.88
- **[P2] Enable fail-open mode in hook error tests:** `src/plugins/hooks.model-failover.test.ts:105-108`
  - body: `createHookRunnerWithRegistry` forwards no options here, and `createHookRunner` defaults `catchErrors` to false. A rejected handler will therefore make `runner.runModelFailover(...)` reject, so the new `.resolves.toBeUndefined()` fail-open test will fail unless it creates the runner with `catchErrors: true` or uses the global runner setup.
  - confidence: 0.92

## Security Review

Status: needs_attention

Concerns:
