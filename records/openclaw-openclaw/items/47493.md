## Summary

Keep open for maintainer review. Current main still prints one static doctor update note and calls the update runner without progress callbacks, while this PR is a focused one-file change that reuses the existing update progress controller and addresses the earlier TTY/cleanup review comments.

## What This Changes

The PR changes `src/commands/doctor-update.ts` so the interactive doctor pre-update flow creates shared update progress callbacks, passes them to `runGatewayUpdate`, gates spinner output on stdout TTY, and stops the controller in a `finally` block.

## Best Possible Solution

Land this focused doctor progress wiring after maintainer review, with narrow regression coverage for progress handoff, non-TTY behavior, and thrown-error cleanup while leaving the shared update runner and progress helper contracts unchanged.

## Reproduction Assessment

Yes. In a git checkout on current main with interactive stdin, answer yes to the doctor update prompt; the current code prints only the static update note and calls `runGatewayUpdate` without `progress`, so no per-step spinner callbacks can run.

## Solution Assessment

Yes. Reusing `createUpdateProgress` is the narrowest maintainable fix because the update runner already exposes progress hooks and `openclaw update` already uses the same controller; focused tests are the main hardening gap.
