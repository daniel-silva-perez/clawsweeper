## Summary

Keep this PR open for maintainer review. Current main still has the silent doctor pre-update path, while the PR is a narrow reuse of the existing update progress hook and addresses the earlier review comments about stdout TTY gating and spinner cleanup.

## What This Changes

The PR wires `createUpdateProgress()` into `maybeOfferUpdateBeforeDoctor`, passes progress callbacks to `runGatewayUpdate`, gates spinner output on stdout TTY, and stops the controller in a `finally` block.

## Best Possible Solution

The desired end state is for doctor's interactive git-update path to pass the existing update-runner progress callbacks through the shared progress controller, while preserving non-TTY output cleanliness and guaranteed spinner cleanup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. In a git checkout on current main with interactive stdin, answering yes to the doctor update prompt reaches code that prints only the static update note and calls `runGatewayUpdate` without `progress`, so no per-step spinner callbacks can fire.

## Solution Assessment

Is this the best way to solve the issue? Yes. Reusing `createUpdateProgress()` and the existing `runGatewayUpdate` progress contract is the narrowest maintainable fix; the remaining improvement is focused validation rather than a different design.
