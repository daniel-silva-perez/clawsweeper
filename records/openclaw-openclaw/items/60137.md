## Summary

Keep this PR open. Current main still has the two gaps this PR targets: Slack request-wrapper errors with an empty/no diagnostic message are not classified as transient, and null/undefined unhandled rejection reasons still fall through to the fatal process-exit path. The PR is narrow and security-sensitive only in the behavioral sense that it changes global fatal-rejection policy; it does not add dependency, CI, release, secret, or supply-chain changes.

## Best Possible Solution

Keep the PR open and review a scoped version of the fix. The best path is to add regression coverage for the empty Slack request-wrapper case and the undefined/null Socket Mode rejection case, then land either the PR’s core handler change with a clear rationale or a narrower Slack-owned normalization/catch if maintainers want to avoid broadening global rejection suppression.
