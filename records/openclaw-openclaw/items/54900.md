## Summary

Keep open. Current main still has the stale replay behavior this PR targets, but the PR is not ready to merge because its stale predicate is tied to `retryCount === 0` and misses never-attempted entries after recovery-budget deferral mutates them into retry entries.

## What This Changes

The PR adds a `maxEntryAgeMs` startup-recovery stale guard, `skippedStale` recovery accounting, and recovery tests for old first-attempt delivery-queue entries.

## Best Possible Solution

Rebase the branch over current main and make stale protection depend on true attempted-delivery state, or preserve never-attempted status across recovery-budget deferrals. Add a regression test where an old never-attempted entry is budget-deferred once and is still skipped as stale on the next startup, while fresh crash-recovery entries still replay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:218-219`
  - body: The stale gate only matches `retryCount === 0` entries with no `lastAttemptAt`, but recovery-budget deferral currently calls `failDelivery`, which increments `retryCount` and sets `lastAttemptAt` without any send attempt. An old never-attempted entry can be budget-deferred once and then bypass this new guard on the next startup, so stale replay can still happen.
