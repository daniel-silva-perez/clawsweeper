## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks any delivery-queue stale-entry TTL, and the related user report remains valid; however, the PR's retry-count-only stale guard misses old never-attempted entries after recovery-budget deferral mutates them into retry entries.

## What This Changes

The PR adds a `maxEntryAgeMs` startup-recovery stale guard, `skippedStale` recovery accounting, and recovery tests for old first-attempt delivery-queue entries.

## Best Possible Solution

Rebase over current main and make stale protection depend on true attempted-delivery state, or preserve never-attempted status across recovery-budget deferrals. Add a regression test where an old never-attempted entry is budget-deferred once and is still skipped as stale on the next startup, while fresh crash-recovery entries still replay.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create an old first-attempt entry, force `maxRecoveryMs` budget deferral so `failDelivery` increments it, then run startup recovery again and observe that the PR's `retryCount === 0` stale predicate no longer matches.

## Solution Assessment

No. The requested stale-entry protection is the right direction, but this patch is not the best implementation until it distinguishes actual delivery attempts from budget deferrals or otherwise preserves first-attempt state.
