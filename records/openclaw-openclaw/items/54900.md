## Summary

Keep open. Current main still lacks the stale first-attempt recovery guard, so the PR is not obsolete, but the proposed predicate misses a real existing recovery-budget path that can turn never-attempted stale entries into retry entries before the stale check sees them.

## What This Changes

The PR adds a startup-recovery staleness filter for pending delivery entries that have not yet been attempted, plus recovery summary accounting and recovery tests.

## Best Possible Solution

Keep the PR open as useful source work, but update the implementation so stale protection is based on actual delivery attempts rather than `retryCount` alone, or preserve first-attempt status for recovery-budget deferrals. Add a regression test where an old never-attempted entry is budget-deferred once and then skipped as stale on the next startup, then rebase over current main's recovery claim/re-read logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:216-217`
  - body: Entries deferred by the recovery time budget are updated via `failDelivery`, which increments `retryCount` even though no send was attempted. Because this new stale gate only treats `retryCount === 0` entries as first attempts, an old never-sent entry can be budget-deferred once and then bypass stale filtering on the next startup, leaving the out-of-context replay path in large backlogs.
  - confidence: 0.9
