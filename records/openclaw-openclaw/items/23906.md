## Summary

Keep this issue open. Current main still lacks a public per-call sessions_spawn tool selector or tool-less mode, while lower-level embedded-run primitives already have disableTools/toolsAllow that are not plumbed through the spawn API. The related #15032 and #62695 remain open, and the prior maintainer-facing review on this issue also explicitly kept it open.

## Best Possible Solution

Keep the issue open and let maintainers choose the canonical umbrella across #15032, #62695, and this issue. The eventual fix should add a backward-compatible sessions_spawn tool policy field, validate and document it, carry it through SpawnSubagentParams and the gateway agent RPC, map tool-less spawns to embedded disableTools:true, map subsets to toolsAllow, and cover core tools, plugin/MCP tools, config precedence, sandbox/security behavior, and local-model no-tools runs.

## Risks / Open Questions

- The API shape is still unsettled: tools:"none", toolsAllow, disableTools, allow/deny object, and plugin-tool naming all imply different public contracts.
- Tool restrictions are security-sensitive; a partial implementation could create a false isolation guarantee for prompt-injection or read-only subagent use cases.
- Plugin and bundle MCP tools need explicit contract coverage so per-spawn allowlists do not regress the plugin-tool use case raised in the comments.
- There are multiple overlapping open issues (#15032, #62695, and this issue), so maintainers should pick a canonical tracking path before implementation.
