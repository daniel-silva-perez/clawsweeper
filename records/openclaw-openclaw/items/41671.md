## Summary

Current main still contains the reported Telegram retry behavior. Explicit Telegram network config is respected when the initial transport decisions are built, but the direct fallback chain still adds forced IPv4-only dispatcher attempts and can promote them as sticky after retryable fetch errors. The related open PR #42059 is still an implementation candidate for this behavior, so this issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open. Review and land #42059 if it is still the right patch, or apply an equivalent narrow fix in `extensions/telegram/src/fetch.ts` so runtime IPv4 fallback is not armed when Telegram `network.autoSelectFamily` or `network.dnsResultOrder` was explicitly configured. Add regression coverage in `extensions/telegram/src/fetch.test.ts` and the inbound media path for `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"`.
