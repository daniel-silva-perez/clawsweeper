## Summary

Current main still appears affected. Telegram network config is honored for the initial dispatcher, but direct transports still build sticky IPv4-only fallback attempts after retryable fetch errors, and inbound Telegram media downloads consume those dispatcher attempts. The related open PR #42059 explicitly proposes skipping runtime IPv4 fallback when `channels.telegram.network` is configured, so this issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open. Review and land #42059 if it is still the right patch, or apply an equivalent narrow fix in `extensions/telegram/src/fetch.ts` so sticky IPv4 fallback is not armed when Telegram `network.autoSelectFamily` or `network.dnsResultOrder` was explicitly configured. Add regression coverage in `extensions/telegram/src/fetch.test.ts` and the inbound media path for `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"`.
