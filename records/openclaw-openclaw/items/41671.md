## Summary

Current main still matches the reported Telegram media/network bug. Telegram config is honored when the initial dispatcher decision is built, but the runtime fallback path still adds forced IPv4-only dispatcher attempts and can promote them after retryable fetch errors. Inbound media downloads use those same dispatcher attempts. The related open PR #42059 remains an implementation candidate, so this issue should stay open until that or an equivalent fix lands.

## Best Possible Solution

Keep this issue open. Review and land #42059 if it is still the right patch, or apply an equivalent narrow fix in extensions/telegram/src/fetch.ts so runtime IPv4 fallback is not armed when Telegram network.autoSelectFamily or network.dnsResultOrder was explicitly configured. Add regression coverage in extensions/telegram/src/fetch.test.ts and the inbound media path for autoSelectFamily true plus dnsResultOrder "verbatim".
