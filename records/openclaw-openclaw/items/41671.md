## Summary

Current main still matches the reported Telegram media/network regression. Telegram network config is resolved and source-tracked, but `resolveTelegramTransport` still creates forced IPv4 fallback dispatcher attempts whenever sticky fallback is allowed, without guarding explicit `channels.telegram.network.autoSelectFamily` or `dnsResultOrder` config. Inbound media downloads still use those dispatcher attempts through `fetchRemoteMedia`, and current tests still assert the forced IPv4 fallback behavior. The related open PR #42059 remains the implementation candidate, so this issue should stay open until that or an equivalent fix lands.

## Best Possible Solution

Keep this issue open. Review and land #42059 if it is still the right patch, or apply an equivalent narrow fix in `extensions/telegram/src/fetch.ts` so runtime IPv4 fallback is not armed when Telegram `network.autoSelectFamily` or `network.dnsResultOrder` was explicitly configured. Add regression coverage in `extensions/telegram/src/fetch.test.ts` and the inbound media path for `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"`.
