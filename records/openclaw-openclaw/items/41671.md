## Summary

Current main still matches the reported Telegram media/network regression. Explicit Telegram network config is parsed and tagged as config-sourced, but the transport resolver still arms sticky IPv4 fallback based only on dispatcher mode and still constructs fallback attempts that force IPv4. The inbound media path still passes those fallback attempts through to fetchRemoteMedia, and the related open PR #42059 remains an implementation candidate rather than a merged fix.

## Best Possible Solution

Keep this issue open. Review and land #42059 if it is still the right patch, or apply an equivalent narrow fix in extensions/telegram/src/fetch.ts so sticky IPv4 fallback is not armed when Telegram network.autoSelectFamily or network.dnsResultOrder was explicitly configured. Add regression coverage in extensions/telegram/src/fetch.test.ts for autoSelectFamily: true plus dnsResultOrder: "verbatim", and keep media-path coverage proving fetchRemoteMedia no longer receives forced IPv4 fallback attempts in that explicit-config case.
