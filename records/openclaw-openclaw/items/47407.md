## Summary

Keep open. Current main still lacks the edit-time cron model warning, so this PR is not obsolete, but the proposed warning still runs in the local CLI context and diverges from runtime model resolution; merging it as-is can produce misleading warnings or miss the runtime failure it is meant to predict.

## What This Changes

The branch adds a non-blocking warning in `cron edit --model` by loading config/catalog in the CLI and checking the parsed model against `buildAllowedModelSet` before sending the cron update.

## Best Possible Solution

Implement the warning in the gateway/runtime context or expose a gateway-side validation method, reuse `resolveAllowedModelRef`/`resolveCronModelSelection` semantics exactly, and add focused CLI or RPC coverage for disallowed models plus valid aliases, shorthand refs, configured defaults, auth-profile suffixes, and remote-gateway behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Validate against the target gateway config:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: `cron edit` can target a remote gateway via the shared gateway client options, but this warning reads `loadConfig()` in the local CLI process. When local and remote `agents.defaults.models` differ, the command can warn for a model that will run or stay quiet for one the gateway will reject, so the new diagnostic does not reliably predict execution behavior.
  - confidence: 0.93
- **[P2] Match runtime model resolution before warning:** `src/cli/cron-cli/register.cron-edit.ts:220-228`
  - body: The warning builds the allowlist with hard-coded defaults and compares a `parseModelRef(model, DEFAULT_PROVIDER)` key, while cron execution resolves configured defaults, aliases, bare refs, and auth-profile suffixes through `resolveAllowedModelRef`. Valid cron model inputs can therefore get a false warning, and the check can disagree with the runtime error path it is trying to preview.
  - confidence: 0.94
