## Summary

Keep open. The requested early warning is still not present on current main, but this PR should not merge as-is because its CLI-side warning can disagree with the gateway runtime that will actually execute the cron job.

## What This Changes

The PR adds a non-blocking `cron edit --model` warning by loading local config and model catalog data in the CLI and checking the parsed model against the allowed model set before `cron.update`.

## Best Possible Solution

Move the prediction into the gateway/runtime context, or expose a gateway-side validation RPC, so `cron edit --model` can reuse the target gateway's `resolveCronModelSelection` and `resolveAllowedModelRef` semantics for disallowed refs, aliases, bare refs, configured defaults, auth-profile suffixes, and remote-gateway usage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by editing a remote gateway whose `agents.defaults.models` differs from local config, or by using a valid runtime alias, bare ref, or auth-profile-suffixed ref that the PR warning path evaluates differently from cron execution.

## Solution Assessment

No. The PR addresses the right bug class, but checking locally loaded config before the gateway mutation is not the narrowest maintainable fix because cron execution validates in the target gateway with richer model-resolution semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate against the target gateway config:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: `cron edit` can target a remote gateway via `--url` and `--token`, but this warning reads local config before sending `cron.update`. When local and remote allowlists differ, the command can warn for a model that will run or stay quiet for one the gateway will reject.
  - confidence: 0.93
- **[P2] Reuse the runtime model resolver before warning:** `src/cli/cron-cli/register.cron-edit.ts:220-229`
  - body: The warning builds an allowlist and parses the raw CLI model in the CLI path, while cron execution resolves configured defaults, aliases, bare refs, and auth-profile suffixes through `resolveAllowedModelRef`. Valid runtime model refs can therefore receive false warnings, and rejected refs can be predicted inconsistently.
  - confidence: 0.94
