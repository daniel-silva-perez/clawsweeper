## Summary

Keep open. The user problem is still real on current main, but this PR should not merge as-is because the new warning can disagree with cron execution: it reads local CLI config even when editing a remote gateway, and it does not reuse the runtime resolver semantics for configured defaults, aliases, bare refs, or auth-profile suffixes.

## What This Changes

The PR adds a non-blocking `cron edit --model` warning in `src/cli/cron-cli/register.cron-edit.ts` by loading local config/model catalog data and warning before `cron.update` when the parsed model is outside the allowed model set.

## Best Possible Solution

Move the prediction into the gateway/runtime context, or expose a gateway-side validation RPC, so `cron edit --model` can reuse `resolveCronModelSelection` and `resolveAllowedModelRef` semantics for the target gateway with focused coverage for disallowed refs, aliases, shorthand refs, configured defaults, auth-profile suffixes, and remote-gateway usage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: edit a remote gateway whose `agents.defaults.models` differs from local config, or use a valid runtime alias/bare ref/auth-profile-suffixed ref; the PR warning path can disagree with the execution-time resolver in those cases.

## Solution Assessment

No. The PR addresses the right bug class, but checking a locally loaded config before the gateway mutation is not the narrowest maintainable fix because cron execution validates in the target gateway with richer model-resolution semantics.
