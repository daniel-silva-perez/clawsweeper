## Summary

Keep open. Current main still lacks an edit-time allowlist warning, so the PR targets a real CLI feedback gap, but the proposed patch is not merge-ready because it predicts cron execution from local CLI config and simplified model parsing instead of the target gateway/runtime resolver.

## What This Changes

The PR adds a non-blocking stderr warning in `src/cli/cron-cli/register.cron-edit.ts` when `cron edit --model` is outside the locally loaded model allowlist.

## Best Possible Solution

Add a gateway/runtime-backed validation path for `cron edit --model` so the warning uses the same target-gateway config and `resolveAllowedModelRef` semantics as cron execution, including aliases, bare refs, configured defaults, auth-profile suffixes, and remote gateway operation.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by comparing the PR's local `loadConfig()` and `parseModelRef` warning path with cron execution's target-gateway `resolveAllowedModelRef` path, especially for remote gateways or valid alias, bare, and auth-profile-suffixed refs.

## Solution Assessment

No. The warning is useful, but this implementation is not the best fix because it predicts runtime failure from local CLI state and simplified parsing instead of reusing the gateway/runtime validation contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Validate against the target gateway config:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: `cron edit` can target a remote gateway via `--url` and `--token`, but this warning reads local config before sending `cron.update`. When local and remote allowlists differ, the command can warn for a model that will run or stay quiet for one the gateway will reject, making the new feedback unreliable for remote edits.
  - confidence: 0.93
- **[P2] Reuse runtime model resolution before warning:** `src/cli/cron-cli/register.cron-edit.ts:220-229`
  - body: The warning parses the raw CLI model and checks `allowedKeys` directly, while cron execution uses `resolveAllowedModelRef` to handle aliases, bare refs, auth-profile suffixes, and configured defaults. Valid runtime model refs can therefore receive false warnings, and rejected refs can be predicted inconsistently.
  - confidence: 0.94
