## Summary

Keep this PR open. Current main still does not implement the PR's central behavior: there is no root `--log-file` option, no `OPENCLAW_LOG_FILE` file-sink override, and docs still describe only config-backed `logging.file`. The PR is still a plausible implementation candidate, but the review comments about routed-command fast paths and daemon-status environment scoping remain material before merge.

## Best Possible Solution

Keep this PR open as the implementation candidate. The best path is to revise it so root `--log-file` is parsed and promoted before both Commander and route-first execution, add an explicit `OPENCLAW_LOG_FILE` resolver with precedence over `logging.file`, pass an env parameter through log-path resolution for daemon status, and keep the focused CLI/logger/status/docs tests. Maintainers should still confirm that a dedicated root CLI flag is the desired product surface beyond the config-backed `logging.file` fix already shipped on main.

## Risks / Open Questions

- The PR's current preAction-only propagation can silently miss route-first commands unless the override is promoted before `tryRouteCli`.
- Daemon status can misreport the effective file sink unless log-file resolution accepts the inspected service environment instead of reading only the CLI process environment.
- The selected log file path is a local file-write surface. That is not a supply-chain concern in the provided diff, but it should remain scoped to the existing trusted operator/config model.
