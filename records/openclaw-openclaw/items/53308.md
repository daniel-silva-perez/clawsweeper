## Summary

Keep open. Current main still routes Signal `groupAllowFrom` through sender matching and still falls back to `requireMention: true` when no explicit group/default value is configured. Open PR #73538 is linked as the implementation candidate but is unmerged, and it appears to cover the group-id allowlist half while the mention-default change still needs maintainer policy review.

## What This Changes

The issue requests that Signal group allowlists match inbound Signal group IDs and that explicitly configured Signal groups not require a mention by default unless configured to do so.

## Best Possible Solution

Keep this issue open until the Signal-local group-id matching fix is merged or deliberately replaced, then resolve the `requireMention` default as an explicit group-policy decision with focused regression tests and user-facing docs aligned to the chosen behavior.

## Reproduction Assessment

Yes. Static reproduction on current main is high-confidence: with `channels.signal.groupPolicy="allowlist"` and `groupAllowFrom` containing a Signal group id, the handler extracts `groupId` but the access callback only compares the sender identity. For the mention half, an explicit group entry without `requireMention` reaches the final `return true`.

## Solution Assessment

Unclear for the full issue. A Signal-local group-id match like PR #73538 is the narrowest maintainable direction for the allowlist bug, but the mention-default request changes policy semantics and should be accepted or rejected explicitly with tests and docs.
