## Summary

Keep this issue open. Current main still blocks Signal group messages through the sender-based `groupAllowFrom` gate before the Signal `groups` entry for the actual group ID can admit the message. The `requireMention` default also still falls through to true, but that aligns with current group-safety docs and should be handled as an explicit maintainer product/security decision rather than cleanup closure.

## Best Possible Solution

Keep this issue open and fix the bundled Signal plugin. The likely path is to make Signal group admission consult `resolveChannelGroupPolicy` for the Signal group ID before sender authorization can drop the event, while preserving `groupAllowFrom` as sender authorization inside admitted groups. Add regression tests for a configured Signal group ID under allowlist mode, empty sender `groupAllowFrom`, sender allowlist accept/reject inside configured groups, and the explicit `requireMention: false` workaround. Decide separately whether an explicit group entry without `requireMention` should default to false or whether docs/config should continue requiring an explicit override.
