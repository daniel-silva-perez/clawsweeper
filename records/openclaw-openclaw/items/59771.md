Keep open: current main still closes the active Feishu streaming card from the dispatcher error path, so the central bug is not already implemented, but this PR should not merge as-is because the bundled streaming-card sanitizer can corrupt incremental code-span snapshots and the new async cleanup still races the core dispatcher’s non-awaited error hook.

## What This Changes

The PR changes Feishu reply dispatching to clear and reuse an active streaming card after transient reply errors, adds regression coverage, and bundles sibling Feishu streaming markdown/reasoning cleanup changes.

## Best Possible Solution

Land a focused Feishu fix that keeps the active streaming card through transient reply-delivery errors, clears dispatcher and session buffers before a recovered final can merge stale content, and handles markdown sanitization separately or in a way that does not mutate merge state. Keep #49381 open until that focused fix is merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Defer markdown sanitation until after streaming merge:** `extensions/feishu/src/streaming-card.ts:399`
  - body: `sanitizeCardKitMarkdown` appends synthetic closing backticks before `update()` merges streaming snapshots. If a later real snapshot supplies the natural closing backtick, the stored sanitized text no longer cleanly prefixes the new text, so `mergeStreamingText` falls back to concatenation and the card can show duplicated or corrupted code-span content. Sanitize only the outbound Card Kit payload, not the merge input/state.
  - confidence: 0.92
- **[P2] Serialize error cleanup before retry delivery:** `extensions/feishu/src/reply-dispatcher.ts:494`
  - body: The shared reply dispatcher invokes `onError` without awaiting it, so this async `clearStreamingProgress()` can still be clearing queued partials while a recovered final is delivered. That leaves the final path able to merge against stale streaming buffers, which is the failure mode this PR is meant to fix. Put the cleanup into the delivery chain or make the dispatcher await error cleanup before later deliveries run.
  - confidence: 0.82
