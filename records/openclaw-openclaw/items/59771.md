## Summary

Keep open. The PR targets a real Feishu duplicate-reply path, but the branch is not ready to merge: it carries unrelated sanitizer work from sister PRs, current main has newer Feishu streaming closeout changes that this branch does not reflect, and the proposed sanitizer/error-cleanup paths introduce concrete regressions.

## What This Changes

The PR adds Feishu streaming-session `clearText()` recovery, changes reply `onError` handling to clear streaming buffers without closing the active card, adds regression coverage, and bundles Feishu streaming markdown/reasoning sanitation changes.

## Best Possible Solution

Adapt the recovery idea onto current main as a focused Feishu change: keep one streaming card through transient reply errors, reset stale text without corrupting subsequent snapshots, preserve the newer idle/final dedupe behavior, and keep markdown sanitizer work either out of this PR or implemented without mutating canonical merge state.

## Reproduction Assessment

Yes for the blocking review finding: call `FeishuStreamingSession.update()` with an incremental code span such as `Here is \`co` followed by `Here is \`code\`` on the PR head; the sanitizer changes the first snapshot before merge and the fallback concatenates. The broader Feishu duplicate-reply path has log-backed reproduction in #49381, but I did not run live Feishu tests in this read-only pass.

## Solution Assessment

No. Keeping the card open and clearing stale buffers is a plausible direction, but this implementation is not the narrowest maintainable fix until cleanup is serialized with the core dispatcher, closed/stale session references are handled, and sanitizer changes are separated or made merge-safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not sanitize snapshots before merging:** `extensions/feishu/src/streaming-card.ts:399-400`
  - body: `sanitizeCardKitMarkdown()` appends synthetic closing backticks to partial snapshots before they become `currentText`. A later real snapshot such as `Here is `code`` no longer starts with the mutated prior value `Here is `co``, so `mergeStreamingText()` falls back to concatenation and corrupts the visible reply. Keep canonical streaming state unsanitized, or sanitize only the outbound Card Kit payload.
  - confidence: 0.9
- **[P1] Reset stale streaming refs after failed closes:** `extensions/feishu/src/reply-dispatcher.ts:314-315`
  - body: `clearStreamingProgress()` leaves `streaming` installed after calling `clearText()`. If the original failure came from `close()` after the session set `closed = true`, `clearText()` returns early and the stale session blocks `startStreaming()` from opening a replacement, so the recovered final falls back to a separate visible send path.
  - confidence: 0.85
- **[P2] Serialize error cleanup before retry delivery:** `extensions/feishu/src/reply-dispatcher.ts:494`
  - body: The regression test awaits `options.onError()` directly, but production core invokes `onError` without awaiting its returned promise. A retry/final queued right after a failed delivery can run before `clearStreamingProgress()` finishes draining partial updates and clearing buffers, leaving the stale merge race in place.
  - confidence: 0.82
