## Summary

Keep this PR open. Current main still resolves mixed `command` plus non-empty `url` MCP server configs as stdio before it ever considers HTTP transport, while this PR adds the narrow early rejection that the linked bug and current docs call for. The patch is focused, has targeted tests, has successful PR check-runs on the latest head, and I found no blocking functional or security issue in the diff.

## What This Changes

This PR adds an early mixed `command`+`url` guard in `src/agents/mcp-transport-config.ts` plus four resolver tests so ambiguous MCP server configs are skipped with a diagnostic instead of silently resolving to stdio first.

## Best Possible Solution

Land this focused resolver guard and tests, or apply the same change on main, so OpenClaw rejects mixed stdio/HTTP MCP server configs at config-resolution time with a clear sanitized diagnostic while preserving pure stdio and pure HTTP behavior.

## Reproduction Assessment

Yes. A direct current-main check of `resolveMcpTransportConfig("probe", { command: "node", url: "https://mcp.example.com/http", transport: "streamable-http" })` follows the stdio path because `command` is present; the runtime timeout path then matches the linked issue's observed error.

## Solution Assessment

Yes. The PR places the guard at the narrow resolver point where both fields are visible, before stdio precedence can mask the HTTP config, and the tests cover the main edge cases without changing lower-level stdio or HTTP helpers.
