## Summary

Keep this issue open. Current main has useful pieces around chat streaming, live tool cards, session listing, and subagent run status, but it does not implement the requested per-session heartbeat/status model for active turns. The session API lacks heartbeat endpoints or status fields like awaiting_model/streaming/tool_exec/tool_wait, the Control UI sessions table has no live health badge/stall state, and existing session change events refresh stored session rows rather than emitting periodic liveness updates. The related ClawSweeper matches are adjacent gateway/channel health work, not a canonical duplicate for per-session turn health in Control UI.

## Best Possible Solution

Implement this as a core Gateway and Control UI feature: add a typed per-session turn-health state owned by the active run/session lifecycle, expose it through an additive sessions status/heartbeat RPC or extended sessions.list row, push updates over the existing authenticated WebSocket subscription, and render compact live badges/tooltips in the Sessions list. Tests should cover state transitions, stale/no-heartbeat detection, permissions, and UI rendering. Related gateway/channel health issues should stay separate unless they converge on a shared generic health contract.

## Risks / Open Questions

- A naive heartbeat could add noisy gateway broadcasts or leak cross-session activity details; the implementation should respect existing operator scopes and session event subscriptions.
- Existing live tool/chat streaming should be reused where possible so this does not create a second, divergent status pipeline.
