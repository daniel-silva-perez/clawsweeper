## Repair Work Prompt

Fix the duplicate inbound/message delivery regression tracked by https://github.com/openclaw/openclaw/issues/72176. Current main only dedupes retries that reuse a stable provider MessageSid or chat.send idempotency key; retries that regenerate those ids can append the same user message again and trigger duplicate assistant replies. Use #72290 as source context but do not blindly copy it: make the inbound fallback reachable when MessageSid is absent or regenerated, scope it by provider/route/session/account/thread/peer plus safe message/timestamp context, and add transcript idempotency/messageId metadata only where it actually prevents duplicate session-history events without hiding intentional repeated messages. Keep the fix in Gateway/auto-reply owner modules, avoid core channel ids or plugin-specific policy, and do not edit external WeChat plugin code. Add a user-facing CHANGELOG.md fix entry crediting the reporter and any carried-forward contributor work. Validate with focused tests and run changed-gate in Testbox before handoff if runtime files change.
