## Summary

Keep this issue open. Current main still only dedupes the reported Gateway/WebChat path when retries reuse the same request idempotency key or stable MessageSid, while the closed follow-up PR is unmerged and previously found to leave the fallback ineffective.

## What This Changes

The issue asks OpenClaw to stop intermittent duplicate user-message and assistant-message delivery that began in 2026.4.24 across WebChat, WeChat, and session history.

## Best Possible Solution

Land a narrow Gateway/auto-reply fix that makes retry dedupe work when inbound retries lack a stable MessageSid or regenerate request ids, and add transcript/session-history idempotency only where it actually enforces duplicate suppression without hiding legitimate repeated messages.

## Reproduction Assessment

Yes for a focused code-level reproduction path: current main dedupes chat.send retries only when the same MessageSid/idempotency key is reused, and inline transcript updates without a messageId can become new session-history events with fresh seq values. The exact live intermittent setup was not rerun.

## Solution Assessment

No current shipped solution fully covers the reported path. The best direction is a narrow replacement for the useful parts of #72290, with the dedupe fallback made reachable and tested before merging.
