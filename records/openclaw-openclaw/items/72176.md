## Summary

Keep open. The issue has an open, unmerged fixing PR (#72290) with closing syntax, and current main still lacks the two concrete guards that PR proposes: a fallback inbound dedupe key when MessageSid is unavailable and idempotency on the webchat user transcript append path.

## Best Possible Solution

Review #72290 as the canonical implementation candidate. A good fix should add a conservative fallback inbound dedupe key for unstable retry IDs, add `clientRunId` idempotency to user transcript updates, and include regression tests that prove duplicate user messages with the same timestamp/content do not produce duplicate transcript entries or assistant replies while legitimate repeated messages still pass.
