## Summary

Keep open. Current main still only dedupes this path when a stable message id or client run id is present, while the reported regression and the open fixing PR both describe retries that arrive with unstable IDs; closing now would hide an active, unmerged implementation candidate.

## What This Changes

Fix an intermittent 2026.4.24 regression where duplicate inbound user messages across WebChat and WeChat create duplicate transcript entries and assistant replies.

## Best Possible Solution

Review #72290 as the canonical implementation candidate, then land or revise a narrow fix that adds conservative inbound fallback dedupe and user transcript idempotency with regressions proving retries are suppressed while intentional repeated messages still pass.

## Reproduction Assessment

Yes for a focused code-level reproduction path: current main will not build an inbound dedupe key without MessageSid, and separate chat.send requests with different idempotency keys append user transcript messages without a transcript idempotency key. The live intermittent channel repro was not run in this read-only pass.

## Solution Assessment

Unclear until #72290 is reviewed. Adding clientRunId idempotency to user transcript updates is narrow, but the fallback content-based key needs maintainer review for false-positive boundaries across channels.
