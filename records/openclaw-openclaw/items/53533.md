## Summary

Keep this PR open for maintainer follow-up. Current main still does not surface Feishu WebSocket/webhook lifecycle state into gateway health snapshots, so the PR is not implemented on main. The branch contains useful monitor-side work, but it is incomplete for the normal gateway path because `extensions/feishu/src/channel.ts` still would need to pass a status sink into `monitorFeishuProvider`; it is also stale against current main's async Feishu WebSocket client and heartbeat/reconnect repair. Related PR #39489 is the closer canonical implementation path, but it is still open, so this needs consolidation or a focused rebase rather than cleanup closure.

## Best Possible Solution

Keep this PR open and ask for maintainer consolidation with #39489 or a focused rebase. The best implementation should wire Feishu channel startup to a gateway status sink, propagate that sink through `monitorFeishuProvider`, `monitorSingleAccount`, and both transports, report WebSocket/webhook connected/disconnected/reconnect/error state plus inbound event and transport activity timestamps, preserve the current #72411 WebSocket heartbeat/reconnect behavior, and include focused Feishu monitor/channel tests plus a user-facing changelog entry if it lands.
