## Summary

Keep this PR open. Current main still lacks config-backed per-agent/default command lane isolation: foreground agent runs use an explicit request lane when supplied and otherwise fall back to the shared main lane. The PR is a relevant implementation candidate for the still-open request in #43235, but review comments identify correctness and security-sensitive lane-policy issues that need maintainer follow-up before merge.

## Best Possible Solution

Keep this PR open as the active implementation candidate for #43235. The best path is to revise it rather than spawn a duplicate PR: add the typed config-backed lane resolver and docs, thread server-resolved lanes through gateway and auto-reply paths, decide/limit explicit request-lane overrides at the network boundary, fix the `maxConcurrent` inheritance and resolved-config fallback issues, then validate targeted tests, generated config docs, and the changed gate.
