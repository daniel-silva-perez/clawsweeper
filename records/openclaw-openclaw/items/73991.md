## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main still lacks config-backed per-agent/default foreground command lanes, so the PR targets a real #43235 gap, but the patch preserves an untrusted gateway lane override and misapplies default lane concurrency to unrelated per-agent lanes.

## What This Changes

The PR adds `agents.defaults.commandLane` and `agents.list[].commandLane` config/schema/docs, wires those lanes into gateway and auto-reply agent execution, registers configured lane concurrency, adds tests, updates generated config baseline hashes, and adds a changelog entry.

## Best Possible Solution

Keep the implementation direction, but make gateway ingress lane selection server-controlled or allowlisted, fix the command-lane resolver so default `maxConcurrent` applies only to the default lane unless a per-agent cap is explicit, align `ResolvedAgentConfig` with the default fallback contract, then validate the config schema/docs baseline and targeted gateway/agent tests before landing or replacing the branch.

## Reproduction Assessment

Yes. The main user problem is reproducible from #43235's Feishu @all setup and current main's shared `main` lane path; the PR defects are also high-confidence static reproductions from the new resolver and ingress code without needing a full runtime test.

## Solution Assessment

No, not as proposed. The PR is aimed at the right core boundary for routed foreground agent runs, but the safest maintainable solution should not let network callers bypass configured lanes with arbitrary lane names and should not carry a default lane concurrency cap onto a different per-agent lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Block untrusted lane overrides at ingress:** `src/gateway/server-methods/agent.ts:1083-1084`
  - body: The new command-lane config is bypassed whenever a gateway client supplies `request.lane`. `AgentParamsSchema` accepts any string and `enqueueCommandInLane` creates lanes on demand, so an authenticated client can send unique lane names and run outside the configured per-agent/default caps. For network ingress, ignore `request.lane` or validate it against a server-owned allowlist.
  - confidence: 0.86
- **[P2] Keep default lane concurrency on the default lane:** `src/config/agent-command-lanes.ts:54`
  - body: If defaults set `{ id: "inbound", maxConcurrent: 3 }` and an agent sets only `{ id: "agent:heavy" }`, this fallback applies `3` to `agent:heavy`. That cap belongs to the default lane, so only inherit `defaults.maxConcurrent` when the agent did not override the lane id; otherwise leave the custom lane at its own explicit cap or the queue default.
  - confidence: 0.93
- **[P3] Resolve commandLane through agent defaults:** `src/agents/agent-scope-config.ts:125`
  - body: `ResolvedAgentConfig` is the merged per-agent view, but this field stores only `entry.commandLane`. Agents relying on `agents.defaults.commandLane` will look unconfigured to any consumer of the resolved config; use `entry.commandLane ?? agentDefaults?.commandLane` or do not expose it there.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff preserves and tests network-facing `request.lane` precedence over server-configured lanes, leaving configured isolation bypassable by arbitrary client-provided lane strings.

Concerns:

- **[high] Untrusted lane override bypasses configured caps:** `src/gateway/server-methods/agent.ts:1084`
  - body: Gateway `AgentParamsSchema` accepts `lane` as an arbitrary string, and the command queue creates state for new lane names. Because the PR resolves `request.lane` before the configured per-agent/default lane, an authenticated client can choose unique lanes to avoid configured concurrency caps.
  - confidence: 0.86
