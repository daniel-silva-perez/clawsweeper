## Summary

Keep open. Current main has a documented and tested `auth.cooldowns.rateLimitedProfileRotations` escape hatch that matches the reporter's workaround, but the default still allows only one same-provider rate-limit profile rotation before model fallback. That means the requested behavior of trying all remaining Codex profiles before fallback is not implemented by default.

## What This Changes

The issue requests that a rate-limited `openai-codex` auth profile advance through the remaining configured auth profiles for the same model before using the model fallback chain.

## Best Possible Solution

Decide the intended default policy explicitly. If Codex should try every eligible same-model auth profile before fallback, change the rate-limit rotation default or derive the cap from eligible profile count, update docs/config reference, and add a four-profile Codex regression test proving profile1 -> profile2 -> profile3 -> default before fallback. If the one-rotation cap is intentional, document that Codex multi-profile users should raise `auth.cooldowns.rateLimitedProfileRotations` when they prefer profile exhaustion over fast model fallback.

## Reproduction Assessment

Yes. Current main has a high-confidence simulated reproduction path: the multi-profile fallback E2E test proves the default makes only two same-provider attempts before fallback, while a high `rateLimitedProfileRotations` value makes more attempts. A live Codex OAuth repro would add provider-specific confidence but is not needed to see the current policy gap.

## Solution Assessment

Unclear. The existing knob is a maintainable workaround, but it is not the requested default; changing the default would contradict current docs and regression tests, so the best fix depends on maintainer policy rather than a purely mechanical code change.
