## Summary

Keep open. Current main has real Codex/auth-profile rotation improvements and a working `auth.cooldowns.rateLimitedProfileRotations` knob, matching the reporter's workaround, but it still intentionally defaults to only one same-provider rate-limit rotation before model fallback. With four profiles, the requested default behavior of trying all remaining profiles before fallback is still not implemented.

## Best Possible Solution

Keep the issue open and resolve it with a maintainer decision on the intended default. If the desired behavior is to try all eligible Codex profiles before model fallback, update the rate-limit rotation policy and add a regression test with at least four `openai-codex` profiles plus a fallback model proving profile1 -> profile2 -> profile3 -> default happens before fallback. If the one-rotation cap is intentional, close only after documenting that policy clearly for Codex users and pointing them to `auth.cooldowns.rateLimitedProfileRotations` as the supported opt-in.
