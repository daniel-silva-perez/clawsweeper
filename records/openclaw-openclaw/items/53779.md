## Summary

Current main still does not implement this PR’s intended provider-level isolation. The PR targets a real models.json all-or-nothing failure, reinforced by the related #21584 report and a later user confirmation on OpenClaw 2026.4.10, but the patch needs maintainer rework before merge because it is based on an older validation path, logs config-derived text directly, and does not cover the strongest missing-apiKey semantic failure enforced by PI’s ModelRegistry.

## Best Possible Solution

Keep this PR open, but revise before merge. The best fix should isolate bad provider entries at the actual models.json/PI boundary or another maintainer-approved models config boundary, preserve valid provider baseUrl/model overrides, route diagnostics through sanitized config/log formatting, and add focused coverage for mixed valid/invalid providers, all-invalid providers, log sanitization, and the missing-apiKey custom-provider path from #21584.
