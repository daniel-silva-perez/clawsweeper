## Summary

Keep open. The linked bug is real-looking and current main still routes runtime model discovery through PI's all-or-nothing `models.json` loader, but this PR does not reach the reported missing-`apiKey` semantic failure and adds an unsanitized warning path.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-validation pass in `src/config/validation.ts` plus a new Vitest file for valid, mixed-invalid, warning, and non-provider-invalid config cases.

## Best Possible Solution

Implement provider isolation where OpenClaw writes or loads `models.json`, using PI-equivalent per-provider semantic validation. Valid providers should keep base URL, auth, headers, model metadata, overrides, and capabilities; invalid providers should be skipped with sanitized diagnostics and regression coverage for mixed, all-invalid, missing-`apiKey`, invalid API enum, and generated implicit-provider cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure one valid custom provider with a base URL and one non-built-in custom provider with `models` but no `apiKey`; PI `validateConfig` throws and `loadCustomModels` returns an empty custom-model result.

## Solution Assessment

No. The proposed fix validates only against OpenClaw's Zod schema before runtime generation, while the reported failure happens later in PI semantic validation; the warning path also bypasses existing terminal sanitization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recover from PI semantic provider failures:** `src/config/validation.ts:251`
  - body: This pre-pass only calls `ModelProviderSchema.safeParse`, where `apiKey` is optional, and it runs before generated or merged providers are loaded by PI. A non-built-in provider with custom `models` but no `apiKey` can still pass here, then PI rejects the whole `models.json` and drops unrelated valid overrides.
  - confidence: 0.9
- **[P2] Sanitize provider warning text:** `src/config/validation.ts:264`
  - body: The warning interpolates the provider key and validation reason directly from user-editable config into `console.warn`. Existing config diagnostics sanitize path and message first, so this can allow crafted provider ids or issue text to inject newlines or terminal control characters into logs.
  - confidence: 0.88

## Security Review
