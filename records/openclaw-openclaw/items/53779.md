## Summary

Current main still does not implement this PR’s intended provider-level isolation for `models.json`, and the PR needs maintainer rework before merge because it targets config Zod validation rather than the PI `ModelRegistry` failure path, misses the missing-`apiKey` semantic failure, and emits config-derived warning text directly to the terminal.

## What This Changes

The PR adds a `models.providers` pre-validation pass in `src/config/validation.ts` plus tests so invalid provider entries are warned about and skipped while valid providers continue loading.

## Best Possible Solution

Keep this PR open, but revise before merge. The best fix should isolate invalid providers at the generated/merged `models.json` boundary or another maintainer-approved models config boundary, use PI-compatible semantic validation for custom-provider requirements, preserve valid provider `baseUrl` and model overrides, emit sanitized diagnostics, and add focused tests for mixed valid/invalid providers, all-invalid providers, missing-`apiKey` custom providers, and warning sanitization.
