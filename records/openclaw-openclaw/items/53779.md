## Summary

Keep this PR open as useful source work for a real-looking models.json failure mode, but do not merge it as-is. Current main still lacks generic per-provider isolation for PI models.json semantic failures, and the PR’s proposed pre-pass misses the reported missing-apiKey case while adding an unsanitized config-derived warning path.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-validation pass in `src/config/validation.ts` plus a new Vitest file covering schema-invalid provider entries being dropped while valid providers continue to load.

## Best Possible Solution

Implement provider isolation at the models.json generation or runtime discovery boundary using PI-equivalent per-provider semantic validation. Valid providers should preserve baseUrl, auth, headers, models, overrides, and metadata; invalid providers should be skipped with sanitized diagnostics and regression coverage for mixed providers, all-invalid providers, missing-apiKey custom providers, invalid API enums, and generated implicit providers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recover from PI semantic provider failures:** `src/config/validation.ts:251`
  - body: The pre-pass only checks `ModelProviderSchema`, where `apiKey` is optional. PI still rejects a non-built-in provider that has custom `models` without `apiKey` and then drops the whole custom-model result, so the reported repro can still discard unrelated baseUrl overrides.
  - confidence: 0.9
- **[P2] Sanitize provider warning text:** `src/config/validation.ts:264`
  - body: The new warning interpolates the provider key and Zod issue text directly from user-editable config into `console.warn`. Existing config diagnostics sanitize path and message first; this path can allow crafted provider ids or messages to inject newlines or terminal control characters into logs.
  - confidence: 0.88
