## Summary

Keep open. Current main does not implement this PR's intended provider-level isolation for the reported models.json failure class: OpenClaw config validation still parses the whole schema at once, and the pinned PI registry still drops all custom models and overrides when one provider fails semantic validation such as missing apiKey. The PR is directionally useful, but it needs maintainer review because its current pre-pass catches Zod schema failures, not the strongest PI missing-apiKey repro, and it introduces a raw config-derived console warning path.

## What This Changes

The PR adds a sanitizeInvalidProviders pre-validation pass in src/config/validation.ts plus a new Vitest file so provider entries failing ModelProviderSchema.safeParse are warned about and removed before whole-config validation.

## Best Possible Solution

Keep the provider-isolation behavior, but implement it at the generated/merged models.json boundary or another maintainer-approved models config boundary, using PI-compatible semantic validation per provider. Valid providers should keep their baseUrl, model, auth, and modelOverride data; invalid providers should be skipped with sanitized diagnostics and focused regression tests for mixed providers, all-invalid providers, missing-apiKey custom providers, invalid api enum recovery, and warning sanitization.
