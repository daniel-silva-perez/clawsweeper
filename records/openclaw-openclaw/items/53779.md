## Summary

Keep this PR open. Current main still lacks general per-provider recovery for broken `models.providers` entries, and the related user report remains plausible; however, the PR needs maintainer review because its Zod pre-pass does not cover the strongest confirmed PI `models.json` semantic failure and it introduces an unsanitized warning path for config-derived text.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-validation pass in `src/config/validation.ts` plus a new Vitest file so Zod-invalid `models.providers` entries are warned about and dropped while other providers continue validating.

## Best Possible Solution

Keep the provider-isolation behavior, but implement it at a maintainer-approved models config boundary that matches PI semantics. Valid providers should preserve `baseUrl`, auth, models, overrides, and model metadata; invalid providers should be skipped with sanitized diagnostics and regression coverage for mixed providers, all-invalid providers, missing-apiKey custom providers, invalid API enum recovery, and warning sanitization.
