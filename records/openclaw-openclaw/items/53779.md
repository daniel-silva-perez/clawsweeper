## Summary

Keep this PR open. Current main has not implemented the PR’s intended per-provider validation isolation: `models.providers` is still validated as one schema record, so a provider-level schema failure still rejects the whole raw config path. The PR is also tied to a concrete user report and follow-up confirmation, but the patch needs revision before it is clearly safe to land because it logs untrusted provider keys/Zod messages through raw `console.warn`.

## Best Possible Solution

Keep the PR open and ask for a current-main revision: implement per-provider isolation in the relevant config/models-json validation path, log through sanitized config logging, use parsed provider data when keeping entries, and add regression coverage for mixed valid/invalid providers plus the all-invalid edge case. Then review/land it against the linked user-visible provider-disappearance behavior rather than closing it as already implemented.

## Risks / Open Questions

- The PR as posted emits untrusted config-derived provider keys and validation messages through raw `console.warn`; that should be routed through the existing sanitized config logger before landing.
- The PR body’s missing-`apiKey` example no longer exactly matches current `ModelProviderSchema`, where `apiKey` is optional; tests should target current failing shapes such as strict-schema unknown fields, missing `baseUrl`, or malformed provider objects.
- The current main has newer model-provider normalization that can fill some missing `apiKey` cases, so the PR should be rebased and validated against the current generated `models.json` path, not only raw config validation.
