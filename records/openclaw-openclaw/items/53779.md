## Summary

Keep open. The reported bug is real-looking and corroborated by related discussion, but this PR does not fully recover the failing path: it only strips providers that fail OpenClaw Zod validation, while the reported missing-apiKey custom-provider case can still survive that pre-pass and fail later in PI model-registry loading. The PR also adds an unsanitized config-derived warning path.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-validation pass in `src/config/validation.ts` and a new Vitest file for provider-level validation sanitization.

## Best Possible Solution

Implement provider isolation at the models.json generation or runtime discovery boundary with PI-equivalent semantic validation. Valid providers should keep baseUrl, auth, headers, models, overrides, and metadata; invalid providers should be skipped with sanitized diagnostics and regression coverage for mixed providers, all-invalid providers, missing-apiKey custom providers, invalid API enums, and generated implicit providers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recover from PI semantic provider failures:** `src/config/validation.ts:251`
  - body: The pre-pass only calls `ModelProviderSchema.safeParse`, where `apiKey` is optional. The related repro fails later in PI when a non-built-in provider has custom `models` but no `apiKey`, so this patch can still leave the bad provider in `models.json` and PI can reject the whole custom catalog, dropping unrelated baseUrl overrides.
  - confidence: 0.9
- **[P2] Sanitize provider warning text:** `src/config/validation.ts:264`
  - body: This warning interpolates the provider key and validation reason directly from user-editable config. Existing config diagnostics sanitize path and message text before terminal output; without that, crafted provider ids or issue text can inject newlines or terminal control characters into logs.
  - confidence: 0.88
