## Summary

Keep open. Current main still uses the generic missing-register diagnostic in both the full plugin loader and CLI metadata loader, so the requested diagnostic improvement is not obsolete. The PR is an active but not-ready implementation: it preserves the debug module-shape path after the latest update, but its new wrong-loader check only handles `bundled-channel-entry`, while setup-runtime entries use the separate no-register `bundled-channel-setup-entry` contract, and the branch also adds a root `TODO.md` review-plan artifact.

## What This Changes

The PR adds bundled channel missing-field warnings, special missing-register errors in the full and CLI plugin loaders, regression tests for missing-register diagnostics, and a root TODO.md implementation-plan file.

## Best Possible Solution

Land a narrow diagnostics patch that preserves `formatMissingPluginRegisterError` for normal plugins, emits actionable diagnostics for both bundled channel entry and bundled setup-entry wrong-loader cases, keeps the field-level bundled contract warnings, removes PR-only artifacts, and covers both full and CLI loader paths with focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: route a bundled setup entry or malformed bundled channel entry through the legacy plugin loader and current main reaches the generic missing-register path; the PR adds one malformed-entry test but not the setup-entry case that matches the setup-runtime wording.

## Solution Assessment

No, not yet. The proposed direction is appropriate, but the current branch handles only `bundled-channel-entry`; the safer fix is a shared bundled wrong-loader predicate/message that includes `bundled-channel-setup-entry` and keeps the existing debug formatter for ordinary plugins.
