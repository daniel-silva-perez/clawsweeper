## Summary

Keep open. Current main does not implement deadline-aware or state-conditional cron intervals: cron schedules are still static `at`, fixed `every`, or cron-expression schedules, with one persisted `nextRunAtMs` derived from those fields plus retry backoff. The request is valid product/API work, not stale enough for insufficient-info closure and not superseded by the related scheduling/capacity reports in the provided context.

## What This Changes

The issue requests cron job definitions that can switch from a base interval to a shorter interval when a scheduler-visible state condition, such as an approaching deadline or queue threshold, is true.

## Best Possible Solution

Define a first-class, backward-compatible cron scheduling contract for conditional intervals only after maintainers choose the predicate model. A maintainable implementation would extend the cron type/schema/normalizer/store identity, compute next-run timing from a restricted declarative condition source, document the behavior, and cover CLI/API/UI plus scheduler tests without adding unrestricted expression evaluation.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection: current main accepts only `at`, `every`, and `cron` schedules and computes one static next run from those fields.

## Solution Assessment

No for the issue’s literal YAML shape as-is. The desired outcome belongs in core cron scheduling, but the safest solution is a maintainer-approved declarative predicate or scheduler hook contract rather than a freeform `state...` expression string accepted directly by the Gateway.
