## Summary

Keep open. Current main still has only static cron scheduling and no deadline-aware or state-conditional interval contract; the request is a real core scheduler/API feature, not stale enough for insufficient-info cleanup and not fully covered by the related scheduling or capacity reports.

## What This Changes

The issue requests deadline-aware cron scheduling so a job can use a shorter interval when a state condition is true.

## Best Possible Solution

Define a backward-compatible core cron scheduling contract after maintainer review chooses the predicate model. The maintainable path is to add a restricted declarative condition or approved scheduler hook, extend the cron type/schema/validators/store identity and next-run computation, then update CLI, UI, docs, and scheduler tests without adding unrestricted expression evaluation.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection, which shows current main accepts only at, every, and cron schedules and has no conditional interval field or state predicate hook.

## Solution Assessment

No for the issue’s literal free-form YAML shape as written. The desired capability belongs in the core cron scheduler, but the safer solution is a maintainer-approved declarative predicate or hook contract rather than evaluating arbitrary state expression strings directly.
