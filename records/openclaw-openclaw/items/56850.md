## Summary

Keep open. Current main still exposes fixed cron schedules only, and the requested state-aware conditional interval is neither implemented nor clearly superseded. The related closed zeroclaw PR is useful prior art, but it did not land in OpenClaw and uses a narrower date-range design in another codebase.

## What This Changes

The issue requests cron job definitions that can shorten their next run interval when a state-based condition, such as deadline proximity or queue size, is true.

## Best Possible Solution

Define an additive, safe scheduling contract before coding: either a typed core conditional schedule with explicit condition sources and bounded evaluation, or a deliberate plugin/API extension that lets external schedulers request earlier OpenClaw wakes without bypassing cron ownership. Any accepted core path should update the cron type/schema, normalization, next-run computation, CLI/API/UI surfaces, docs, and regression tests together.

## Reproduction Assessment

Not applicable as a runtime bug report. There is a high-confidence static verification path: current main’s docs, CLI parser, protocol schema, and next-run computation only accept `at`, `every`, and `cron` schedules, so the requested `conditional_interval` shape would be rejected or ignored before scheduling.

## Solution Assessment

Unclear; the requested direction is valid, but the YAML example’s free-form `state.*` predicate is not specified enough to be the safest maintainable implementation. A narrower typed-condition design, or an explicit plugin scheduler API if maintainers prefer extension ownership, should be decided first.
