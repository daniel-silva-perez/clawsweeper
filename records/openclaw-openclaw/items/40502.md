## Summary

Keep open. Current main still does not implement dynamic low default temperatures for tool-enabled embedded runs, so the PR is not obsolete, but the branch should not merge as-is because the current diff has two functional blockers and mixes the agent sampling change with unrelated security-sensitive/stale package-surface changes.

## What This Changes

The PR adds embedded-runner dynamic default temperatures for tool-enabled turns, lower defaults for execution-like tool names, focused runner tests and a changelog entry, while also carrying file-secret allowInsecurePath and lockfile importer changes.

## Best Possible Solution

Land a focused agent-only change that preserves explicit user temperatures and plain-chat omission, skips low-temperature injection only when actual provider generation thinking is active, threads the tool-name heuristic through both normal attempts and compaction, and drops the unrelated secrets, lockfile, and stale changelog hunks.

## Reproduction Assessment

Yes. A high-confidence code reproduction exists: on the PR head, a tool-enabled run with no explicit temperature, no active provider thinking config, and reasoningLevel set to on will skip the new default because reasoning visibility is treated as active thinking; compaction can be checked by following the setup path and seeing no availableToolNames hint reaches extra params.

## Solution Assessment

No. The requested direction is useful, but this branch is not the narrowest maintainable fix until the reasoning gate is corrected, compaction is wired, and unrelated security/package changes are split out or dropped during rebase.
