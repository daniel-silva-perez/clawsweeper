## Summary

Keep this PR open. Current main only partially covers the requested CLI help cleanup: `cron add --at` mentions offset-less datetimes and `--tz`, but `cron edit --at` still does not, and both `--tz` help strings still describe cron expressions only. The PR is narrow and security-benign, but its proposed `(default: UTC)` wording conflicts with current cron behavior, where cron schedules without `tz` use the Gateway host timezone.

## Best Possible Solution

Keep the PR open and ask for a narrow wording refresh: make `cron edit --at` mention offset-less datetimes using `--tz`, and make `--tz` help clarify both supported uses without claiming a UTC default for cron schedules. The best final wording should distinguish cron's host-local default from offset-less one-shot datetime behavior, then validate with the targeted cron CLI tests or rendered `openclaw cron add --help` / `openclaw cron edit --help` output.
