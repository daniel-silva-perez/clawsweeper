## Summary

Current main still does not implement eligibility-based skillsSnapshot invalidation, and this PR is not complete enough to close as obsolete or implemented. The remaining work spans reply sessions, CLI agent sessions, and isolated cron snapshots, and the proposed signature does not reflect all runtime inputs used by skill eligibility checks.

## Best Possible Solution

Keep this PR open for maintainer follow-up or supersede it with a canonical fix. The best implementation is a shared deterministic eligibility/cache signature used by reply sessions, CLI agent sessions, and isolated cron; it should reflect the same runtime state evaluated by shouldIncludeSkill/evaluateRuntimeEligibility, persist safe metadata on SessionSkillSnapshot, and add regression tests for bin, env/API-key, config, and remote eligibility changes without a SKILL.md watcher bump.

## Risks / Open Questions

- Closing or merging this as handled would leave stale snapshots when binaries are installed or removed within an unchanged PATH, env/API-key availability changes, or requires.config paths change without a SKILL.md watcher bump.
- The submitted patch only updates the reply-session path; CLI agent and isolated cron sessions still reuse cached snapshots using missing/version/filter checks.
- Security pass did not find supply-chain or CI/dependency changes, but the proposed signature hashes cfg.skills, whose entries can include env/apiKey config. A final design should avoid persisting secret-derived cache material when a narrower eligibility-state signature is sufficient.
