## Summary

Keep this PR open. Current main still does not implement eligibility-based skill snapshot invalidation, and the submitted change is not proven complete because the active cache keys only cover version/filter state while skill eligibility depends on runtime bin/env/config/remote checks across multiple snapshot paths.

## Best Possible Solution

Keep this PR open or supersede it with a canonical fix. The best direction is a shared deterministic eligibility/cache signature used by reply sessions, openclaw agent, and isolated cron snapshots; it should reflect the same inputs used by shouldIncludeSkill and evaluateRuntimeEligibility, persist the metadata on SessionSkillSnapshot, and add regression tests for existing snapshots where env/bin/config/remote eligibility changes without a SKILL.md watcher bump.
