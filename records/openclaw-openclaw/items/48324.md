## Summary

Current main still does not implement eligibility-based skillsSnapshot invalidation, so this PR is not obsolete or implemented-on-main. The PR is a plausible implementation candidate, but it still needs maintainer follow-up because the submitted patch covers only the reply-session path and the review discussion correctly identifies remaining coverage and signature-design gaps.

## Best Possible Solution

Keep this PR open for maintainer review or supersede it with a shared implementation. The best fix is a deterministic, secret-safe eligibility/cache signature used by reply sessions, CLI agent sessions, and isolated cron snapshots, backed by regression tests for bin/env/config/remote eligibility changes that occur without a `SKILL.md` watcher bump.

## Risks / Open Questions

- Closing this PR as implemented would leave stale skills snapshots when runtime eligibility changes without a watcher version bump or skillFilter change.
- The submitted patch is scoped to reply sessions; `openclaw agent` and isolated cron snapshots still need the same invalidation semantics or a shared helper.
- A cache signature must represent actual eligibility outcomes for binaries, env/API-key availability, config paths, and remote eligibility without persisting secret-derived material from skill config.
- Current tests cover version/filter snapshot behavior but do not show a main-branch regression test for eligibility changes without a `SKILL.md` watcher bump.
