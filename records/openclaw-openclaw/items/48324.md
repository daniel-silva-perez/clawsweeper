## Summary

Current main still does not implement the PR’s intended eligibility-based skillsSnapshot invalidation. The shipped cache paths refresh on missing snapshot, snapshot version, and skillFilter changes, while runtime eligibility also depends on bins, env/API-key state, config paths, and remote eligibility. The PR is therefore not obsolete or implemented-on-main, and it still has unresolved design coverage across reply, CLI-agent, and isolated-cron sessions.

## Best Possible Solution

Keep this PR open for maintainer follow-up, or supersede it with a canonical shared fix. The best implementation is a deterministic eligibility/cache signature used by reply sessions, CLI agent sessions, and isolated cron snapshots; it should reflect the actual `evaluateRuntimeEligibility` inputs or outcomes for bin, env/API-key, config, and remote eligibility without persisting secret-derived material, and add regression tests for eligibility changes that happen without a `SKILL.md` watcher bump.

## Risks / Open Questions

- Closing this PR as implemented would leave stale snapshots when a required binary is installed or removed in an unchanged PATH directory, env/API-key availability changes, or a `requires.config` path changes without a skills watcher/version bump.
- The submitted patch only updates the reply-session path; CLI agent sessions and isolated cron snapshots still use missing/version/filter checks.
- A final signature design should avoid persisting secret-derived material from `skills.entries.*.apiKey` or other sensitive config fields.
