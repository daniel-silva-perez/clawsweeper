## Summary

Keep open. Current main improves diagnostics for manifest-declared command aliases, but the reported lossless/lcm path is not proven fixed: the latest published lossless-claw manifest still lacks `commandAliases`, the core ownerless fallback can still recommend adding an unresolved command token to `plugins.allow`, and maintainer discussion explicitly left this issue open while treating #64732 as the broader adjacent tracker.

## What This Changes

The issue asks OpenClaw to stop giving contradictory `plugins.allow` guidance for the `lossless-claw` `/lossless` or `lcm` command surface under restrictive plugin allowlists.

## Best Possible Solution

Keep this issue open until the lossless command path is verified end to end. The desired end state is for `lossless-claw` to publish `commandAliases` for its `lcm` and runtime slash surfaces, and for OpenClaw core to harden the ownerless fallback tracked by #64732 so unresolved command names are never presented as valid `plugins.allow` plugin IDs.
