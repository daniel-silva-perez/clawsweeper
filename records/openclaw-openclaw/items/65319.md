## Summary

Keep this issue open. Current main has useful manifest-backed command-alias diagnostics, but it still falls back to telling users to add an unresolved command token to `plugins.allow`, config validation only redirects aliases that are known from manifest metadata, and the latest published `@martian-engineering/lossless-claw` manifest inspected from npm does not declare `commandAliases` for `lcm` or `lossless`. The maintainer discussion also explicitly left this issue open and treated #64732 as adjacent, not canonical.

## Best Possible Solution

Keep this issue open until the lossless-claw command path is verified end to end. The best fix is either for `lossless-claw` to publish `commandAliases` for its `lcm` and runtime slash surfaces in the documented manifest shape, or for OpenClaw core to harden the ownerless fallback and slash-token normalization so unresolved command-like tokens never receive invalid `plugins.allow` guidance. Coordinate with #64732 only if maintainers make it the canonical tracker for the remaining fallback behavior.
