## Summary

Keep open. Current main has better diagnostics for manifest-declared command aliases, but the reported contradiction class still exists when the command token is unresolved or slash-prefixed: the CLI fallback can still tell users to add the command token to `plugins.allow`, while config validation treats command tokens as stale plugin ids.

## What This Changes

The issue asks OpenClaw to stop giving contradictory `plugins.allow` guidance for lossless-claw command surfaces such as `/lossless` and `lcm` under a restrictive plugin allowlist.

## Best Possible Solution

Keep `plugins.allow` plugin-id-only, normalize command-alias lookup for slash-prefixed tokens, and harden the missing-command fallback so unresolved command names are never recommended as `plugins.allow` entries. If current `lossless-claw` lacks `commandAliases` for `lcm` or `lossless`, coordinate the plugin manifest update as a follow-up while preserving the core diagnostic guard.

## Reproduction Assessment

Yes. The source-level path is high confidence: with a restrictive `plugins.allow` containing `lossless-claw`, a token such as `/lossless` that does not resolve to a manifest alias reaches the fallback at `src/cli/run-main-policy.ts:155` and gets a message recommending the command token itself.

## Solution Assessment

No current solution on main is complete. The existing manifest-backed alias handling is the right direction, but the narrow maintainable fix still needs slash-token normalization and an ownerless fallback that avoids contradictory `plugins.allow` advice.

## Repair Work Prompt

Fix the contradictory `plugins.allow` diagnostics for plugin-owned command names reported in openclaw/openclaw#65319, coordinating with #64732 only where the generic fallback overlaps. Keep `plugins.allow` plugin-id-only. In core, update command-alias handling so slash-prefixed command tokens such as `/lossless` can resolve to manifest aliases declared without a leading slash, and harden `resolveMissingPluginCommandMessage` so an unresolved command token is not presented as a valid `plugins.allow` entry when a registry is available and the token is not a known plugin id. Preserve correct guidance for known plugin ids like `browser`, known CLI aliases like `wiki`, and runtime slash aliases like `dreaming`. Add focused tests for `/lossless`-style leading slash alias normalization and for the ownerless fallback avoiding `Add "<command>" to plugins.allow` for unknown command tokens. Also add config validation coverage if alias normalization affects `plugins.allow` warnings. Do not vendor or modify the external `@martian-engineering/lossless-claw` package in this repo; if its manifest lacks aliases, mention that as a follow-up. Add a changelog fix entry with reporter attribution if behavior changes user-facing CLI/config output.
