## Summary

Keep this issue open. Current main has a documented password fallback for trusted-proxy local-direct clients and an explicit same-host proxy opt-in, but it still fails the originally reported headerless loopback internal-connection shape unless a gateway password is configured. There is also an open implementation candidate referenced in the timeline with closing syntax, so this should remain open for maintainer review rather than be swept closed now.

## Best Possible Solution

Keep this issue open while maintainers decide whether the current password fallback is sufficient for the reported deployment, review or close the open loopback implementation candidate, and use #69066 as the canonical place for longer-term internal service identity work.
