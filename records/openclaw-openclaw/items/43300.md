## Summary

Keep this issue open. Current main has tactical trusted-proxy improvements: local-direct password fallback and explicit same-host loopback proxy support, but the original headerless internal self-connection shape is still intentionally fail-closed unless the operator configures a gateway password or routes through an explicitly allowed loopback proxy with real identity headers. The supplied GitHub context also shows an open fork PR with closing syntax for this issue, so it should remain open until that candidate is reviewed, merged, or explicitly superseded.

## Best Possible Solution

Keep this issue open while maintainers decide the intended trusted-proxy internal-caller boundary. The next concrete step is to review, merge, or explicitly supersede BingqingLyu/openclaw#2181 and decide whether #69066 should own the longer-term service-identity work. If maintainers decide the current password fallback plus allowLoopback docs are the accepted resolution, close only after the open closing PR is resolved or marked superseded.
