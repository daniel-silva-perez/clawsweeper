## Summary

Keep this issue open. Current main still does not provide native per-peer outbound-only WhatsApp messaging: per-direct-peer config only supports prompt overrides, inbound DM admission is governed by dmPolicy/allowFrom or pairing approvals, bindings only choose an agent route, and outbound target authorization still reuses allowFrom.

## Best Possible Solution

Keep the issue open and implement this in the bundled WhatsApp plugin surface. Add a direction-specific per-direct-peer inbound policy, enforce it immediately after WhatsApp DM normalization/admission and before enqueue/session/agent/reply work, preserve outbound authorization for notification targets, then update the WhatsApp config types, strict schema, generated config docs/help, public WhatsApp docs, and focused tests showing an allowlisted peer can receive outbound messages while inbound messages from that peer are ignored.

## Risks / Open Questions

- This is messaging access-control behavior, so implementation should preserve secure defaults for dmPolicy, allowFrom, pairing-store approvals, same-phone/self-chat handling, wildcard allowFrom, read receipts, and outbound echo protection.
- A generic binding-level inbound flag would affect all channels; the narrower low-risk path is likely a WhatsApp-owned per-peer inbound policy unless maintainers intentionally want a cross-channel routing contract.
