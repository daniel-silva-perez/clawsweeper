## Summary

Keep open. Current main still does not provide native per-peer outbound-only WhatsApp messaging: WhatsApp direct-chat config has no direction-specific peer controls, bindings only route admitted messages, inbound admission still uses dmPolicy plus allowFrom/pairing, and outbound authorization still reuses allowFrom.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin surface: add a direction-specific per-direct-peer inbound policy, enforce it after WhatsApp DM normalization/admission checks and before read receipts, enqueue/session metadata, hooks, agent execution, or replies; keep outbound target authorization working for notification-only peers; update the WhatsApp config types/schema, generated config help, public WhatsApp docs, and focused tests for an allowlisted peer whose inbound messages are ignored while outbound sends still succeed.

## Risks / Open Questions

- A fix changes messaging access control, so it needs to preserve secure defaults for dmPolicy, pairing approvals, wildcard allowFrom, dmPolicy=disabled, same-phone/self-chat handling, read receipts, and outbound echo filtering.
- A generic binding-level inbound flag would affect all channels; a WhatsApp-owned direct-peer policy is likely narrower unless maintainers intentionally want a cross-channel routing contract.
- The current no-model routing-sink workaround can suppress replies, but it still admits inbound messages into routing/session machinery, so the native fix should filter before enqueue, session metadata, hooks, agent execution, and fallback behavior.
