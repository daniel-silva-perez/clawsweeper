## Summary

Keep open. Current main still does not provide native per-peer outbound-only WhatsApp messaging: WhatsApp DM admission is still governed by account-level `dmPolicy` plus `allowFrom`, admitted inbound DMs are routed to an agent and processed, and outbound target authorization still reuses `allowFrom` rather than a separate outbound-only permission surface.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin surface: add a direction-specific per-direct-peer inbound policy, enforce it immediately after WhatsApp DM admission/normalization and before enqueue/session/agent/reply work, keep outbound target authorization working for notification-only peers, and update the WhatsApp schema, generated config docs/help, public WhatsApp docs, and focused tests for an allowlisted peer whose inbound messages are ignored while outbound sends still succeed.

## Risks / Open Questions

- A fix touches messaging access control, so it should preserve secure defaults for `dmPolicy`, pairing approvals, same-phone/self-chat handling, wildcard `allowFrom`, read receipts, and outbound echo protection.
- A generic binding-level inbound flag would affect all channels; a WhatsApp-owned bundled plugin surface is likely the narrower path unless maintainers intentionally want a cross-channel routing contract.
