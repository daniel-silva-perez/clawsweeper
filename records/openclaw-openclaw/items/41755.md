## Summary

Current main still leaves this Control UI refresh bug open: a refreshed tab loses the active run identity, so the chat toolbar cannot render Stop/Queue for the in-flight stream. The prior ClawSweeper analysis remains accurate against the current main SHA.

## Best Possible Solution

Keep this issue open. Add an authoritative active-run hydration path for Control UI on page load/reconnect, likely via hello/bootstrap or chat.history, returning the active run id plus enough buffered stream state for the visible session. Then restore chatRunId, chatStream, and chatStreamStartedAt before rendering controls, with regression coverage for refresh during active streaming and terminal cleanup.
