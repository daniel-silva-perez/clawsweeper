## Summary

Keep this open. Current main still contains the WebChat exclusion reported here, and v2026.4.27 contains the same resolver behavior; however, the nearby tests and routing guards intentionally treat WebChat as an internal/admin surface, so this needs maintainer product/security review rather than an automatic one-line change.

## What This Changes

The issue asks WebChat direct messages to respect `session.dmScope: per-channel-peer` and derive a WebChat channel-peer session instead of falling back to `agent:main:main`.

## Best Possible Solution

Maintainers should explicitly decide whether WebChat direct turns participate in secure DM mode. If yes, add a scoped WebChat runtime-policy/session-key path with regression coverage that preserves established external delivery routes; if no, document WebChat as an internal/admin exception in the WebChat, session, and security docs.

## Reproduction Assessment

Yes. A direct WebChat context on `agent:main:main` with a sender id reaches `resolveRuntimePolicySessionKey`; `resolvePolicyChannel` returns `undefined` for `webchat`, and the current unit test asserts the result stays `agent:main:main` and unsandboxed.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the obvious local change, but current tests and route guards intentionally preserve WebChat internal behavior, so the best fix depends on the intended WebChat trust boundary and peer identity semantics.
