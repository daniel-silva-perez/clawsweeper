## Summary

Keep open. Current `main` and the latest release still contain the WebChat runtime-policy exclusion, and existing tests intentionally preserve the reported main-session fallback; the remaining question is the intended WebChat trust boundary, not whether the reported behavior exists.

## What This Changes

The issue asks for WebChat messages to honor `session.dmScope: per-channel-peer` instead of falling back to the agent main session.

## Best Possible Solution

Maintainers should explicitly choose the WebChat session-isolation model. If WebChat should participate in secure DM/runtime-policy isolation, add a scoped WebChat session-key path with regression coverage that preserves external delivery-route guards; if it remains an operator/internal exception, document that exception in WebChat, session, routing, and security docs.

## Reproduction Assessment

Yes. A direct WebChat context on `agent:main:main` with sender metadata reaches `resolveRuntimePolicySessionKey`; current source drops `webchat` as the channel and the current unit test asserts the result stays `agent:main:main` and unsandboxed. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the narrow local edit, but current tests and delivery guards intentionally preserve internal WebChat behavior, so the best fix depends on maintainer product/security judgment.
