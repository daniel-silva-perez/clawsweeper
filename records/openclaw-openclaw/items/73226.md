## Summary

Keep this open. Current main still has the WebChat exclusion reported here, and the latest stable tag contains the same resolver/test behavior; however, adjacent tests and docs treat WebChat as an internal/operator surface in ways that make a one-line change a product and security-boundary decision rather than safe cleanup.

## What This Changes

The issue asks for WebChat direct messages to honor `session.dmScope: per-channel-peer` and derive a WebChat peer session key instead of falling back to `agent:main:main`.

## Best Possible Solution

Maintainers should explicitly decide the WebChat trust model. If WebChat participates in secure DM/runtime-policy isolation, implement a scoped resolver/session-key path with regression coverage that preserves external delivery-route guards; if WebChat remains an internal/operator exception, document that exception in WebChat, session, routing, and security docs.

## Reproduction Assessment

Yes. A direct WebChat context on `agent:main:main` with sender metadata reaches `resolveRuntimePolicySessionKey`; the current resolver drops `webchat` as a channel and the current unit test asserts the result stays `agent:main:main` and unsandboxed.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the narrow local change, but current tests intentionally preserve WebChat main-session/internal behavior, so the maintainable fix depends on a maintainer decision about WebChat isolation semantics.
