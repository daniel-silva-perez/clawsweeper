## Summary

Keep open. Current main still excludes WebChat from runtime-policy channel derivation, and current tests assert that direct WebChat main-session runs remain on `agent:main:main`; the report is concrete, but changing it affects WebChat’s operator/internal trust boundary and should get maintainer product/security review before a code change.

## What This Changes

The issue reports that direct WebChat turns ignore `session.dmScope: per-channel-peer` because runtime-policy channel resolution drops `webchat`, leaving them on `agent:main:main` instead of a per-channel peer policy session.

## Best Possible Solution

Maintainers should define the WebChat trust model explicitly. If WebChat should participate in secure DM/runtime-policy isolation, update the resolver and tests to derive a scoped WebChat policy key while preserving external delivery-route guards; if WebChat remains an internal/operator exception, document that exception in WebChat, session, channel-routing, and security docs.

## Reproduction Assessment

Yes. A direct WebChat context on `agent:main:main` with sender metadata reaches `resolveRuntimePolicySessionKey`; current source drops `webchat` as the channel, and the current unit test asserts the result stays `agent:main:main` and unsandboxed.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the narrow local edit, but current tests and adjacent route guards intentionally preserve special WebChat/internal behavior, so the best fix depends on the intended WebChat security and operator boundary.
