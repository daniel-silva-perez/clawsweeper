## Summary

Keep this open. Current main still has the reported WebChat exclusion in `resolveRuntimePolicySessionKey`, and the latest release tag contains the same behavior; however, nearby tests and routing guards also show that WebChat main-session behavior is intentional in at least some UI/admin paths, so this needs maintainer product judgment rather than an automatic close or a blind one-line removal.

## What This Changes

The issue reports that `session.dmScope: "per-channel-peer"` is ignored for WebChat direct messages and asks OpenClaw either to treat WebChat like other channels or document an intentional exception.

## Best Possible Solution

Maintainers should decide whether secure DM mode is supposed to include WebChat operators as isolated direct peers. If yes, implement a scoped WebChat dmScope path with regression coverage that preserves external-route protections; if no, document WebChat as an explicit internal/admin exception and make the UI/config wording less misleading.

## Reproduction Assessment

Yes, at source level: a direct WebChat context on `agent:main:main` reaches `resolveRuntimePolicySessionKey`, `resolvePolicyChannel` returns `undefined`, and the helper returns the main session key. The existing unit test currently locks in that behavior rather than failing on it.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` alone is not clearly the best fix because the current test suite and `chat.send` route guards intentionally treat WebChat as internal in some paths; the safer solution needs a maintainer decision and targeted tests for WebChat dmScope semantics.
