## Summary

Keep this open. Current main and v2026.4.29 still contain the WebChat runtime-policy exclusion reported here, and existing tests assert that direct WebChat main-session runs remain on `agent:main:main` and unsandboxed; however, adjacent WebChat routing guards and docs show this crosses an internal/operator trust-boundary decision rather than a safe automatic one-line cleanup.

## What This Changes

The issue asks WebChat direct turns to respect `session.dmScope: per-channel-peer` by deriving a WebChat channel-peer runtime policy/session key instead of falling through to `agent:main:main`.

## Best Possible Solution

Maintainers should explicitly decide the WebChat trust model. If WebChat should participate in secure DM/runtime-policy isolation, implement a scoped WebChat resolver/session-key path with regression coverage that preserves external-route guards; if WebChat remains an internal/operator exception, document that exception in WebChat, routing, session, and security docs.

## Reproduction Assessment

Yes. A direct WebChat context on `agent:main:main` with sender metadata reaches `resolveRuntimePolicySessionKey`; current source drops `webchat` as the channel, and current unit coverage asserts the result remains `agent:main:main` and unsandboxed. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the narrow local edit, but current tests and delivery guards intentionally preserve special WebChat internal behavior, so the best fix depends on the intended WebChat security and operator boundary.
