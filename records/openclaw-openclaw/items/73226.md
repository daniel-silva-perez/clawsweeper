## Summary

Keep this open. Current main still excludes WebChat from runtime policy session-key derivation, and the latest release tag contains the same exclusion, so the reported behavior is not implemented or obsolete. Adjacent tests and route guards also show intentional WebChat internal/admin behavior, so a one-line removal is not a safe automatic cleanup without maintainer product judgment.

## What This Changes

The issue requests that `session.dmScope: per-channel-peer` apply to WebChat direct contexts instead of falling back to the main session key.

## Best Possible Solution

Maintainers should define the intended WebChat dmScope contract. If WebChat should participate in secure DM mode, add scoped WebChat session-key derivation with regression coverage that preserves existing external-route protections; if it should remain an internal/admin exception, document that exception in the WebChat and session docs and make the configuration wording clear.

## Reproduction Assessment

Yes. A source-level reproduction is a direct WebChat context on `agent:main:main` with a sender id; the current resolver returns the main session key, and the existing unit test asserts that behavior rather than failing on it.

## Solution Assessment

Unclear. The requested one-line removal would address the immediate key derivation symptom, but it is not clearly the best fix because current WebChat route-protection and internal-delivery tests encode different semantics that need maintainer confirmation first.
