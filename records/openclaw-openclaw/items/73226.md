## Summary

Keep open. Current main still contains the WebChat runtime-policy exclusion and a regression test that asserts WebChat direct turns on the main session remain unsandboxed; the unresolved question is whether that is the intended WebChat/operator trust boundary, so this should not be auto-closed or auto-fixed as a one-line change.

## What This Changes

The issue reports that `session.dmScope: per-channel-peer` is ignored for WebChat because WebChat contexts resolve back to the main session instead of a per-channel peer session.

## Best Possible Solution

Maintainers should explicitly define the WebChat isolation model. If WebChat should participate in DM/runtime-policy isolation, add a WebChat-aware session-policy path with regression coverage for main-session and external-route cases; if WebChat is intentionally an operator/internal exception, document that exception in WebChat, session, routing, and security docs.

## Reproduction Assessment

Yes. A direct WebChat context passed to `resolveRuntimePolicySessionKey` on `agent:main:main` returns `agent:main:main`, and the current unit test asserts that the result remains unsandboxed under non-main sandbox mode. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the narrow local edit, but the surrounding WebChat delivery guards and tests intentionally preserve internal/operator routing, so the best fix requires maintainer security/product judgment first.
