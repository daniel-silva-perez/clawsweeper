## Summary

Keep this open. Current main and the latest release still exclude WebChat from runtime policy session-key derivation, so the reported source-level behavior is real; however current tests and routing guards also preserve WebChat as an internal selected-agent/main-session UI path, so the suggested one-line removal needs maintainer product judgment.

## What This Changes

The issue asks for WebChat direct turns to respect isolated DM scoping instead of falling back to `agent:main:main` when `session.dmScope` is configured.

## Best Possible Solution

Maintainers should decide whether WebChat operator turns are part of secure DM isolation or an explicit internal/admin exception. If isolation is intended, implement a scoped WebChat policy/session path with tests that preserve external-route protections; if not, document the exception in WebChat and session-routing docs.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `resolveRuntimePolicySessionKey` with `SessionKey: agent:main:main`, `Provider` or `OriginatingChannel: webchat`, `ChatType: direct`, and a sender id returns `agent:main:main` because `resolvePolicyChannel` drops WebChat.

## Solution Assessment

Unclear. The suggested deletion is plausible but not clearly the best fix because existing tests and WebChat route guards intentionally treat some WebChat paths as internal main-session operator traffic; a safer change needs an explicit maintainer decision and targeted tests.
