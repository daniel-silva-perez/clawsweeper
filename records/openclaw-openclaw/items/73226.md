## Summary

Keep this open. Current main and the latest release still contain the WebChat exclusion reported here, and the unit coverage still asserts that direct WebChat main-session runs stay on `agent:main:main`; however, adjacent routing guards and docs treat WebChat as an internal/operator surface, so the fix needs maintainer product/security review rather than automatic cleanup.

## What This Changes

The issue asks for WebChat direct messages to honor `session.dmScope: per-channel-peer` by deriving a WebChat peer session key instead of falling back to `agent:main:main`.

## Best Possible Solution

Maintainers should explicitly decide the WebChat trust model. If WebChat participates in secure DM/runtime-policy isolation, implement a scoped WebChat resolver/session-key path with regression coverage that preserves established external delivery-route guards; if WebChat remains an internal/operator exception, document that exception in WebChat, routing, session, and security docs.

## Reproduction Assessment

Yes. A direct WebChat context on `agent:main:main` with sender metadata reaches `resolveRuntimePolicySessionKey`; current source drops `webchat` as the channel and the current unit test asserts the result remains `agent:main:main` and unsandboxed.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the narrow local edit, but current tests intentionally preserve WebChat main-session/internal behavior, so the best solution depends on the intended WebChat security and operator boundary.
