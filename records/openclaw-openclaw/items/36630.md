## Summary

Keep this PR open. Current main still does not implement the central bidirectional Signal quote-reply behavior: Signal sends do not emit signal-cli quote RPC parameters, Signal outbound paths do not pass reply/quote author metadata, monitor replies are sent unquoted, and inbound Signal quote context still lacks structured ReplyToId. The PR is still a plausible implementation candidate, but the provided PR context also shows unresolved merge-conflict markers, extension-boundary violations, and attribution-sensitive review findings that need maintainer follow-up rather than cleanup closure.

## Best Possible Solution

Keep this PR open and rework it into a narrower, Signal-owned quote-reply change: add plugin-safe quote timestamp/author plumbing through the SDK or a Signal-local facade, pass reply metadata through formatted/direct/monitor text and media sends, surface inbound Signal quote ids as ReplyToId with quote body/sender, remove conflict markers and core deep imports, and keep focused regression coverage for group quote-author lookup, formatted sends, monitor replies, media/chunk/sendPayload consumption, debounce, and queued recovery.
