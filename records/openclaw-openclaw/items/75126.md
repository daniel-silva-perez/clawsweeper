## Summary

Keep open. Current main does not already implement strict tool mode, and the PR is useful directionally, but the proposed patch has blocking integration gaps: the new config key is not propagated into the generated config schema surfaces, and Google provider runs do not receive the strictness option through the real runtime path.

## What This Changes

The PR adds an opt-in strict/warn/off tool strictness mode with repair diagnostics across embedded agent runs, provider transport replay, config types, OpenAI-compatible gateway responses, tests, and docs.

## Best Possible Solution

Keep the PR open and fix the integration gaps before merge: regenerate the base config schema and config docs baseline for the new config key, route TransportStrictnessOpts through the Google provider-owned stream path, and validate the config, transport, gateway, and embedded-runner strictness lanes while preserving default permissive behavior.

## Reproduction Assessment

Yes for the review findings. The config drift is reproducible by the missing generated schema/baseline diff, and the Google gap is reproducible by following the runtime path from createBoundaryAwareStreamFnForModel through provider registration, where no strictness argument reaches createGoogleGenerativeAiTransportStreamFn.

## Solution Assessment

No. The proposed direction is maintainable, but this is not the best mergeable form until all public config schema surfaces are regenerated and every claimed provider transport path receives the strictness option through the real runtime, not only direct unit-test helpers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Regenerate the base config schema:** `src/config/zod-schema.ts:350-355`
  - body: Adding `toolStrictness` to `OpenClawSchema` is not enough because `openclaw config schema` is served from `GENERATED_BASE_CONFIG_SCHEMA`. Without regenerating and committing `src/config/schema.base.generated.ts` and the config docs baseline, the parser accepts the new key while schema/UI/docs consumers do not expose it, and the config schema/docs checks should drift.
  - confidence: 0.91
- **[P2] Pass strictness through the Google provider runtime:** `src/agents/provider-transport-stream.ts:87-90`
  - body: The Google transport factory now accepts `strictness`, but the real provider-owned path still constructs the Google stream without it: core only passes `ctx?.strictness` to Anthropic, and Google provider registration still calls `createGoogleGenerativeAiTransportStreamFn()` with no argument. Embedded runs using Google will therefore keep repairing replayed JSON-string tool args even when `toolStrictnessMode` is `strict`.
  - confidence: 0.89
