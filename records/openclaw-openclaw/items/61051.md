## Summary

Keep this PR open. Current main still does not implement the target-owned approval routing behavior, and the branch remains relevant, but it has blocking route-selection bugs and mixes an unrelated guarded-fetch change into the approval work.

## What This Changes

The PR changes exec/plugin approval routing, Telegram native approval authorization and delivery, approval docs/tests, deterministic prompt state propagation, and guarded-fetch undici dispatcher handling.

## Best Possible Solution

Implement the routing policy on top of current main's shared approval request handler: compute an approval-kind-aware route descriptor before broadcasting, include agentId/sessionKey/turnSourceTo/account inputs in the turn-source decision, suppress origin broadcasts only when forwarding owns the route, keep Telegram delivery targets separate from approval authority, and handle guarded-fetch changes in a separate security-reviewed thread.

## Reproduction Assessment

Yes. The current-main gap is reproducible by configuring targets-mode forwarding to Telegram and triggering an exec/plugin approval from an origin with approval clients; the code broadcasts before forwarding ownership is known. The PR findings are also reproducible by using `approvals.exec.agentFilter` with an approval that has agentId but no sessionKey, or by triggering targets-only exec approvals from Web UI/TUI origins.

## Solution Assessment

No. The direction is useful, but this branch is not the best merge shape because it targets older per-handler routing, misses two targets-only suppression cases, and includes an unrelated guarded-fetch change; the safer fix is a rebased, narrower shared-handler patch with focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass agentId through targets-only filtering:** `src/infra/exec-approval-surface.ts:92-97`
  - body: Targets-only suppression is filtered with only `{ sessionKey }`. If an exec approval has `agentId` but no `sessionKey` while `approvals.exec.agentFilter` is configured, this returns false, `hasApprovalTurnSourceRoute` stays true, and the gateway still broadcasts an origin approval prompt after forwarding accepted the request. Pass `agentId` through the route helper and include it in the filter request.
  - confidence: 0.9
- **[P2] Run targets-only suppression before Web UI/TUI returns:** `src/infra/exec-approval-surface.ts:115-128`
  - body: The early return marks Web UI and TUI sources as enabled before the targets-only check runs. In `approvals.exec.mode = "targets"`, forwarded approvals from those origins still get origin approval broadcasts instead of target-only routing. Apply the targets-only suppression before returning enabled for these origins.
  - confidence: 0.87
