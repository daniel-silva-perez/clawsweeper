## Summary

Keep this PR open. Current main has related native-approval infrastructure, but it does not implement the PR’s central behavior: target-owned approval routing without duplicate origin/operator broadcasts, approval-kind-aware turn-source routing, and Telegram target delivery that is separate from approval authority. The provided review discussion also surfaced real P1/P2 routing gaps, so this should be rebased and reviewed rather than auto-closed.

## Best Possible Solution

Keep the PR open, but ask for a rebase onto current main and a narrower patch. The best path is to implement generic approval route selection in the shared gateway handler, add exec and plugin tests for targets-only forwarding, same-origin suppression, webchat/TUI behavior, and agent/session filters, and keep Telegram delivery targets separate from approval authority. Any already-shipped or unrelated fetch-guard changes should be removed from this PR or handled in a separate security-reviewed thread.
