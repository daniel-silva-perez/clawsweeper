## Summary

Keep open: current main still returns all movable static keys for named-account channel configs when a channel has no `namedAccountPromotionKeys`, and the doctor plus `channels add` migration paths still delete those top-level keys into `accounts.default`. The report is valid for root-inheritance channels such as Slack and Discord, while the final fix needs channel-aware review because some channels intentionally inherit shared defaults from `accounts.default`.

## What This Changes

The issue asks to prevent multi-account channel promotion from moving shared access and routing defaults into `accounts.default` for channels that do not define `namedAccountPromotionKeys`.

## Best Possible Solution

Audit bundled channel account-resolution semantics, then add explicit promotion contracts or a conservative shared fallback with per-channel opt-outs so root-level shared policy/delivery defaults remain shared where named accounts inherit from the root. Add regression coverage for at least Slack/Discord-style root inheritance and preserve tests for channels that intentionally inherit from `accounts.default`.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction path: a channel with named accounts and no `namedAccountPromotionKeys` will have common keys selected by `resolveSingleAccountKeysToMove`, then doctor or non-default `channels add` will remove those keys from the root and write them into `accounts.default`.

## Solution Assessment

Unclear as a single generic change. The requested direction is correct for Slack/Discord-style root inheritance, but current WhatsApp behavior shows some channels intentionally use `accounts.default` as shared inheritance, so the best fix is channel-aware rather than a blunt global fallback.
