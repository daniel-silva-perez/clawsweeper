## Summary

Keep open. Current main still matches the reported failure mode: for channels with named accounts and no `namedAccountPromotionKeys`, the shared promotion helper returns all common movable keys, including shared policy keys, and the doctor migration then moves those keys into `accounts.default`. The related merged PRs in the issue body fix Telegram and Matrix-specific surfaces, but they do not cover Slack, WhatsApp, Discord, or the other bundled plugins that still lack a named-account promotion filter.

## Best Possible Solution

Keep this issue open. The best path is a narrow, source-backed fix to the setup-promotion contract: either make missing `namedAccountPromotionKeys` preserve shared policy/delivery keys when named accounts already exist, or add explicit `namedAccountPromotionKeys`/`configPromotion` surfaces to the affected bundled plugins. Cover at least Slack or WhatsApp plus the doctor and `channels add` promotion paths, and update the setup/config docs if the default contract changes.

## Risks / Open Questions

- A fix changes shared config migration semantics used by `doctor --fix`, `channels add`, and setup helpers, so it needs targeted regression coverage for both legacy single-account promotion and mixed named-account repair.
- Some current tests and docs describe the existing default of moving promoted account-scoped values into `accounts.default`; maintainers should decide whether to make the undefined `namedAccountPromotionKeys` fallback conservative or add explicit promotion filters to each affected bundled plugin.
