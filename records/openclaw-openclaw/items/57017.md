## Summary

Keep this issue open. Current main still reads cron run-log JSONL files with unbounded `fs.readFile` in the paged read paths, while pruning remains write-path-only. There is also an open implementation candidate, #57088, whose body uses closing syntax for this issue, so the issue should not be closed before that PR is reviewed/merged or closed.

## Best Possible Solution

Keep the issue open and use #57088 as the active implementation candidate unless maintainers decide to close that PR and replace it. The best fix is a race-safe bounded read for `readCronRunLogEntriesPage` and `readCronRunLogEntriesPageAll`, preferably tail-reading only the last configured byte window without mutating the file; if pruning on read is chosen instead, it should be serialized through `writesByPath`. Add regression coverage for oversized files, append/read races, prune failure resilience, custom `cron.runLog` options, and the all-jobs history path.

There is no `stat`, max-byte cap, tail read, or read-time prune in this path.

## Risks / Open Questions

- Current main can still allocate memory proportional to an oversized cron run-log file during `cron runs`/status-style reads.
- A read-time prune fix must be race-safe with `writesByPath`; the issue comment correctly flags that unsynchronized mutation from the read path can lose fresh appended entries.
- The all-jobs run-history path needs the same treatment, otherwise a fix limited to one job leaves a similar aggregate memory path open.
