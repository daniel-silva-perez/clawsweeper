## Summary

Keep this issue open. Current main still reads cron run-log JSONL files with unbounded `fs.readFile` in both the per-job and all-jobs paged read paths, while pruning remains append-path-only. There is also an open implementation candidate, #57088, whose body uses GitHub auto-close wording for this issue, so the issue should stay open until that PR is reviewed and merged or closed.

## Best Possible Solution

Keep this issue open and use #57088 as the active implementation candidate unless maintainers decide to replace it. The best fix is a race-safe bounded read for both `readCronRunLogEntriesPage` and `readCronRunLogEntriesPageAll`, preferably tail-reading a configured byte window without mutating the log; if read-time pruning is used, it should be serialized through `writesByPath` and respect configured `cron.runLog` options. Add regression coverage for oversized files, all-jobs history reads, prune failure resilience, and append/read race safety.

## Risks / Open Questions

- Current main can still allocate memory proportional to an oversized per-job cron run-log file during paged reads.
- The all-jobs run-history path can allocate across every JSONL run-log file, so a fix limited to the single-job path would leave similar aggregate memory risk.
- A read-time prune fix must be serialized through `writesByPath` or replaced with a non-mutating bounded tail read; the prior issue discussion correctly flags that unsynchronized read-path mutation can race with appends.
