## Summary

Keep open. Current main still has the explicit-only model merge behavior this PR targets, and the PR is paired with the open same-author Bedrock vision issue, but the current patch introduces a concrete false-positive capability inference for a known text-only Venice model and should be revised before merge.

## What This Changes

The PR adds merge-time `input: ["text", "image"]` defaults for explicit-only model entries whose IDs match selected vision-capable families, plus regression tests for Claude, GPT, Gemini, Llama, and brand-new provider paths.

## Best Possible Solution

Keep this PR open and ask for a narrower capability-inference strategy: preserve the Bedrock Claude repair, add negative coverage for known text-only IDs such as Venice `openai-gpt-52`, and either anchor provider-specific model families conservatively or move capability repair into provider-owned catalog/doctor paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on current main, call `mergeProviderModels({ models: [] }, { models: [{ id: "us.anthropic.claude-opus-4-7" }] })` and the returned model has no `input`, which `modelSupportsImages` treats as false.

## Solution Assessment

No, not as currently written. The targeted merge repair is plausible, but the broad `gpt-5` regex marks a known text-only Venice model as image-capable, so a narrower pattern or provider-owned metadata repair is safer.
