## Summary

Keep open. The PR targets a real current-main bug in explicit-only model catalog merging, but the proposed core-level regex default is still too broad and would mark a known text-only Venice model as image-capable, which should block merge until narrowed.

## What This Changes

The PR adds merge-time `input: ["text", "image"]` defaults for explicit-only model entries whose IDs match selected vision-capable families, plus regression tests for Claude, GPT, Gemini, Llama, and new-provider merge paths.

## Best Possible Solution

Keep the PR open and narrow the repair to a conservative provider-aware capability strategy: preserve the Bedrock Claude explicit-only fix, keep explicit `input` overrides authoritative, and add negative coverage for known text-only IDs such as Venice `openai-gpt-52` before merge.

## Reproduction Assessment

Yes. On current main, `mergeProviderModels({ models: [] }, { models: [{ id: "us.anthropic.claude-opus-4-7" }] })` leaves `input` unset, and `modelSupportsImages` then returns false; on the PR head, the `/gpt-5/` regex reproduces the false-positive match for `openai-gpt-52`.

## Solution Assessment

No, not as currently written. The targeted merge repair is plausible, but the unqualified GPT-5 pattern is not the narrowest maintainable fix; provider-owned catalog repair or a tighter exact-family allowlist is safer.
