## Summary

Keep open. Current main still drops the Discord-provided filename when saving inbound attachments, and the agent context is built from saved media path/type fields with no later filename recovery. PR #59783 is already open with closing syntax for this issue, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

Preserve Discord inbound attachment original filenames in saved media and agent-visible metadata instead of UUID-only names.

## Best Possible Solution

Land a narrow Discord plugin fix that threads attachment.filename, and the generated sticker candidate filename where applicable, into the existing saveMediaBuffer originalFilename parameter, with focused tests proving the saved path preserves the source name and the inbound context exposes that preserved path.

## Reproduction Assessment

Yes. A Discord attachment payload with filename reaches resolveMediaList; current main uses that filename only as the fetch hint and then saves without originalFilename, so the stored inbound media name falls back to the UUID-only path.

## Solution Assessment

Yes. The best fix is the narrow channel-owned change already sketched by the open PR: use the existing saveMediaBuffer originalFilename seam and update Discord tests, without adding core migration or new API surface.
