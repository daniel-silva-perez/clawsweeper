## Summary

Keep open. Current main still emits `gateway.trusted_proxies_missing` for loopback Control UI configs, and the existing test still expects that warning. This PR is not implemented on main; the submitted diff is also stale because the active gateway audit logic now lives in `src/security/audit-gateway-config.ts`, while the PR edits the older inline path in `src/security/audit.ts`.

## Best Possible Solution

Keep this PR open and request a revision against `src/security/audit-gateway-config.ts`. The fix should classify `gateway.trusted_proxies_missing` from effective gateway exposure and explicit reverse-proxy context, stop warning for definitely local-only loopback-resolving configs, preserve warnings for real reverse-proxy cases, and add regression coverage for loopback, LAN, custom loopback/custom exposed, auto/container behavior, and trusted-proxy auth interactions.

## Risks / Open Questions

- Merging the submitted patch unchanged would not update the active current-main gateway audit implementation in `src/security/audit-gateway-config.ts`.
- A raw `bind !== "loopback"` rule would continue to score configured bind mode rather than effective exposure, leaving custom/auto loopback-resolving cases and fallback cases misclassified.
- Suppressing the warning for every literal loopback config could hide same-host reverse-proxy deployments where current docs still require `gateway.trustedProxies` for forwarded-client-IP handling.
