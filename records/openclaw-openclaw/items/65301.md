## Summary

Keep open. Current main still has the broad send-side poll guard that this PR targets, and the PR is a focused implementation against the right helper and runner paths; the remaining work is maintainer review of the intended metadata semantics plus normal validation, not a close or replacement repair job.

## What This Changes

The PR adds a stricter real-poll-intent helper, switches `message` send validation to it, and updates regression tests so metadata-only poll params are allowed while non-empty poll question/options still reject on `send`.

## Best Possible Solution

Land this stricter action-scoped send guard, or an equivalent implementation, if maintainers agree that only non-empty poll question/options should block `send`. Keep `poll` action validation strict and preserve tests for both allowed metadata-only sends and rejected real poll payloads on `send`.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: `runMessageAction` with `action: "send"` and metadata-only values such as `pollDurationSeconds: "60"` or `pollPublic: "true"` reaches `hasPollCreationParams` and throws the poll-action error; the current send-validation test encodes that behavior.

## Solution Assessment

Yes for the central bug, with one maintainer semantic decision remaining. A separate `hasRealPollIntent` helper used only by the send guard is narrower than weakening `hasPollCreationParams`, because actual `poll` handling can keep its existing required question/options validation.
