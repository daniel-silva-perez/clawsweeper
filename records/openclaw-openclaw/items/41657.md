## Summary

Keep this issue open. Current main does not strip Slack files entirely: it downloads accessible Slack files, adds `[Slack file: ... (fileId: ...)]` placeholders, and exposes downloaded media through legacy `MediaPath`/`MediaPaths`/`MediaTypes`. The requested structured inbound file metadata contract for file id, name, MIME type, and size is still missing, and the related open PR #62792 is a narrower Slack media hydration fix rather than a canonical metadata implementation.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata contract to `MsgContext` and the plugin SDK, populate it from Slack `message.files` while preserving current `BodyForAgent` placeholders and `MediaPath`/`MediaPaths`/`MediaTypes` behavior, document the new field, and cover it with Slack plus shared channel contract tests. Review #62792 separately as the narrower Slack media hydration fix.

## Risks / Open Questions

- Closing now would leave agents without the requested structured file id/name/MIME/size metadata and preserve the current placeholder-only metadata workaround.
- A Slack-only field would create channel contract drift; the durable fix should be a shared additive inbound attachment model populated by Slack and other media-capable channels.
- Attachment names, MIME types, and sizes are provider/user-controlled metadata. Any implementation should mark them untrusted and avoid exposing Slack private URLs or bearer-token-derived download URLs to agents.
