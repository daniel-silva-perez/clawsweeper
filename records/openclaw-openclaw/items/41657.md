## Summary

Keep this issue open. Current main still has Slack file id/name/MIME/size available at ingestion, but agent delivery is still placeholders plus legacy MediaPath/MediaPaths/MediaTypes; there is no shared structured inbound attachment metadata contract carrying the requested fields. The related open Slack PR is narrower media hydration work, not a replacement for this contract.

## Best Possible Solution

Keep this issue open and implement a backward-compatible shared inbound attachment metadata contract in MsgContext and the plugin SDK. Populate it from Slack message.files with provider, file id, file name, MIME type, size, downloaded/local path state, and safe placeholder text while preserving existing BodyForAgent placeholders and MediaPath/MediaPaths/MediaTypes behavior. Add Slack and shared channel contract tests, update Slack/plugin docs, and review #62792 separately as the narrower Slack media hydration fix.

## Risks / Open Questions

- Closing this now would leave agents dependent on filename/fileId placeholders, MediaPaths, or active download-file calls rather than passive structured id/name/MIME/size metadata.
- A Slack-only ad hoc field could drift from Discord and Telegram; the lower-risk path is an additive shared inbound attachment model in MsgContext/plugin SDK.
- Attachment names, MIME types, and sizes are provider/user-controlled metadata and should be treated as untrusted; Slack private URLs should not be exposed directly to agents.
