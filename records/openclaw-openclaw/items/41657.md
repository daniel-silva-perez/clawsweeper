## Summary

Keep this issue open. Current main still has the Slack file metadata at ingestion, but the delivered agent context is still placeholders plus legacy MediaPath/MediaPaths/MediaTypes. There is no passive structured attachment metadata list carrying file id, file name, MIME type, and size, and the related open Slack PR is narrower media hydration work rather than a replacement for this contract.

## Best Possible Solution

Keep this issue open and implement a backward-compatible shared inbound attachment metadata contract in MsgContext and the plugin SDK, then populate it from Slack message.files while preserving existing BodyForAgent placeholders and MediaPath/MediaPaths/MediaTypes behavior. Add Slack and shared channel contract tests, document the new field, and review #62792 separately as the narrower Slack media hydration fix.

## Risks / Open Questions

- Closing this now would leave agents dependent on filename/fileId placeholders, MediaPaths, or active download-file calls instead of receiving passive file id/name/MIME/size metadata.
- A Slack-only ad hoc field could drift from Discord and Telegram behavior; the lower-risk implementation is an additive shared inbound attachment model in MsgContext/plugin SDK.
- Attachment names, MIME types, and sizes are provider/user-controlled metadata and should be treated as untrusted; Slack private URLs should not be exposed directly to agents.
