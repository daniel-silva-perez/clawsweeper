## Summary

Keep this issue open. Current main still observes Slack file id/name/MIME/size in the raw Slack event type, but inbound delivery to agents is still legacy media paths plus text placeholders; there is no structured inbound attachment metadata contract carrying file id, name, MIME type, and size. The related open PR #62792 is scoped to Slack media hydration reliability and does not supersede this metadata/API request.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata contract to MsgContext and the plugin SDK, populate it from Slack message.files while preserving BodyForAgent placeholders and MediaPath/MediaPaths/MediaTypes, document the new field, and cover it with Slack plus shared channel contract tests. Review #62792 separately as a narrower Slack media hydration fix.

## Risks / Open Questions

- Closing now would leave agents dependent on filename/fileId placeholders, legacy media paths, or an active download-file action instead of receiving passive structured file metadata.
- A Slack-only metadata field would likely create channel contract drift; the cleaner path is a shared additive inbound attachment model usable by Slack, Discord, Telegram, and other media-capable channels.
- Attachment names, MIME types, and sizes are provider/user-controlled and should be treated as untrusted; private Slack URLs should not be exposed directly to agents.
