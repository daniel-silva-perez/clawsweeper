## Summary

Keep this issue open. Current main still has Slack file metadata at the event boundary and can expose fileId placeholders or downloaded media paths, but it still does not pass a passive structured attachment metadata list with id/name/MIME/size through the final agent context; the related open Slack media PR is adjacent hydration work, not a replacement for this contract.

## What This Changes

Request that Slack inbound message delivery include file attachment metadata such as Slack file id, file name, MIME type, and size when Slack `message.files` is present.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata contract to `MsgContext` and the plugin SDK, populate it from Slack `message.files`, preserve existing placeholders and `MediaPath`/`MediaPaths` behavior, and document/test the new contract with Slack as the first implementation.

## Reproduction Assessment

Yes. A Slack file event with `files: [{ id, name, mimetype, size }]` follows the current prepare path into placeholders and legacy media fields only; the shared context type has no structured inbound attachment field to carry the full metadata.

## Solution Assessment

Yes, the best direction is an additive generic metadata contract populated by Slack first. Appending more provider-specific text to `BodyForAgent` would help humans but would not give agents a stable cross-channel API.
