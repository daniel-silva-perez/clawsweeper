## Summary

Keep this issue open. Current main partially handles Slack files by downloading accessible media and adding filename/fileId placeholders, but it still does not pass the requested passive structured attachment metadata to agents: file id, name, MIME type, and size are available in the Slack event type, then flattened into legacy text/media fields. The related open PR #62792 is a narrower Slack media hydration fix and does not supersede this metadata/API request.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata contract to MsgContext and the plugin SDK, populate it from Slack message.files while preserving BodyForAgent placeholders and MediaPath/MediaPaths/MediaTypes, document the new field, and cover it with Slack plus shared channel contract tests. Review #62792 separately as the narrower Slack media hydration fix.

## Risks / Open Questions

- Closing this now would leave agents dependent on filename/fileId placeholders, legacy media paths, or an active download-file action instead of receiving passive structured file metadata.
- A Slack-only ad hoc metadata field would likely create channel contract drift; Discord and Telegram have similar attachment concepts and should share an additive inbound attachment model.
- Attachment names, MIME types, and sizes are provider/user-controlled and should be treated as untrusted metadata; private Slack URLs should not be exposed directly to agents.
