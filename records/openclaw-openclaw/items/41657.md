## Summary

Keep this issue open. Current main still handles Slack files through placeholders plus legacy media paths, not through a structured inbound attachment metadata contract carrying file id, name, MIME type, and size. The related open PR #62792 is about Slack media hydration reliability, not this metadata API.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata contract to MsgContext and the plugin SDK, populate it from Slack message.files while preserving BodyForAgent placeholders and MediaPath/MediaPaths/MediaTypes, document the field, and cover it with Slack plus shared channel contract tests. Review #62792 separately as a narrower Slack media hydration fix.

## Risks / Open Questions

- Closing now would leave agents dependent on Slack filename/fileId placeholders or an active download-file action instead of receiving the requested passive metadata.
- A Slack-only fix would create channel contract drift; this should be a shared additive inbound attachment model usable by Slack and other media-capable channels.
- Attachment names, MIME types, and sizes are provider/user-controlled and should be treated as untrusted; private Slack URLs should not be exposed to agents.
