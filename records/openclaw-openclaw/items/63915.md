## Summary

Keep open. Current main still routes the bundled ZAI image provider through the generic OpenAI-compatible streaming image path with `glm-4.6v`; I found no merged ZAI vision-specific endpoint, payload, streaming, or abort fix, and the related route PR remains an open/unmerged candidate for a related 404 variant rather than proof that this abort report is fixed.

## What This Changes

The issue reports that `zai/glm-4.6v` fails as `agents.defaults.imageModel` through the Telegram image tool with duplicate `Request was aborted` errors, while direct Z.ai API calls with the same image and key succeed.

## Best Possible Solution

Keep this issue open as the canonical ZAI `glm-4.6v` abort report. The best end state is a focused regression for `zai/glm-4.6v` through the image tool using the reported coding-endpoint configuration, followed by the narrowest ZAI-owned fix in `extensions/zai` or a documented provider-facing image payload/transport seam if the failure proves generic across OpenAI-compatible providers.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path from two corroborating reports: configure ZAI with the coding endpoint, set `agents.defaults.imageModel` to `zai/glm-4.6v`, and send an image through Telegram or the image tool. I did not run the live API in this read-only review, but current main still follows the reported code path.

## Solution Assessment

Unclear. The currently suggested route-only approach in #70915 is plausible for related 404 reports, but it is not yet proven to solve this issue's immediate abort symptom; a live or mocked regression should pin the abort cause before choosing endpoint routing, payload transformation, streaming settings, or timeout handling.
