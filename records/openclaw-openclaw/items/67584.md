## Summary

Keep this PR open. Current main still resolves memory-wiki `vault.path` once at plugin registration and does not implement ctx-aware templating, while the PR remains the active scoped implementation candidate for the per-org/per-agent vault isolation problem discussed in #66003. Broader #63829 is relevant product-direction context, but it has not shipped an equivalent replacement on main.

## Best Possible Solution

Keep this PR open for normal maintainer review. The best path is to review the current head, validate the memory-wiki test lane and CI, then either land this scoped plugin-owned ctx-aware `vault.path` implementation or explicitly redirect the design to #63829 if maintainers want a broader per-agent plugin configuration model instead.

## Risks / Open Questions

- The PR intentionally changes where memory-wiki reads and writes data when `vault.path` is templated; maintainer review should focus on unresolved/unknown placeholders, `..` normalization, relative or CWD-backed paths, and recursive directory creation on write paths.
- The PR intentionally skips memory prompt/corpus supplements when `vault.path` is templated. That is a plausible safety boundary, but it changes `memory_search`/`memory_get` availability for templated deployments and needs maintainer product judgment.
- Open #63829 may become a broader per-agent plugin configuration direction, but current main has not shipped that replacement, so closing this PR now would drop the only concrete implementation candidate shown in the provided context.
