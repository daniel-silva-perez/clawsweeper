## Summary

Keep this PR open. Current main still advertises inline attachment fields on the shared `send` schema, but the active shared hydration, send runner, Telegram action runtime, and Telegram transport do not deliver `buffer`/`filename`/`contentType` for Telegram `message.send`. The PR is stale and needs retargeting, but it is not obsolete; #41779 remains the canonical open bug and #63578 only overlaps part of the shared hydration path.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate for #41779. The best next step is to refresh it against the pluginized Telegram code, add focused coverage for `send` with buffer-only payloads and data-URL normalization, coordinate shared hydration with #63578, and enforce a pre-decode base64 size limit before constructing a `Buffer`.
