## Summary

Keep this PR open. Current main still does not implement inline buffer/filename/contentType delivery for Telegram message.send, and this PR remains a relevant implementation candidate for #41779. It is stale against the pluginized Telegram paths and still needs a stronger pre-decode base64 size guard before it is mergeable.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate for #41779. A mergeable version should retarget the active pluginized Telegram paths in extensions/telegram/src/action-runtime.ts, extensions/telegram/src/channel-actions.ts, and extensions/telegram/src/send.ts; coordinate shared send hydration with #63578; add focused regression tests for buffer-only sends and data-URL normalization; and enforce an encoded-length bound before decoding inline base64.
