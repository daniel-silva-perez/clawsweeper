## Summary

Keep open. Current main still drops `allowUnsafeExternalContent` in the direct `POST /hooks/agent` normalization path, so the downstream dispatch and cron runner never see the request-body flag even though they already honor it when present.

## What This Changes

Direct `POST /hooks/agent` requests should honor `allowUnsafeExternalContent: true` so trusted hook dispatches can opt out of external-content wrapping when explicitly requested.

## Best Possible Solution

Make an explicit maintainer decision on the direct hook API contract. If per-request unsafe bypass is intended, add `allowUnsafeExternalContent?: boolean` to the direct hook payload normalization path, preserve it through dispatch and relevant replay scope, add regression coverage for direct `/hooks/agent`, and update hook/security docs with the same warnings as other bypass flags. If it is not intended, reject or ignore it deliberately and document the supported mapping/Gmail/cron bypass surfaces.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: a direct `/hooks/agent` request body can contain `allowUnsafeExternalContent: true`, but `normalizeAgentPayload()` drops it, the handler dispatches only `normalized.value`, and the cron runner wraps external hook content unless the agentTurn payload contains the flag.

## Solution Assessment

Unclear pending maintainer/security confirmation. The narrow code repair is straightforward, but because the flag disables external-content safety wrapping and direct endpoint docs omit it, the best final fix depends on whether maintainers want this as caller-controlled `/hooks/agent` API surface.
