## Summary

Keep open. Current main still has only adjacent cron hardening: stale acknowledgement retry, exact NO_REPLY suppression, denial markers, run-level failure handling, delivery-target checks, and mutating tool-error warnings. It does not implement the requested fail-closed cron/session delivery policy for fabricated final assistant text, marker phrases like "placeholder"/"mock data"/"simulate", missing required-tool evidence, or unsupported success claims.

## Best Possible Solution

Keep this issue open for maintainer design. The best implementation is a narrow unattended cron/session delivery policy, likely opt-in first, that suppresses or cleanly fails delivery when required resources or tool calls failed, final output contains self-referential draft/fabrication markers, or success claims are unsupported by turn/run evidence. Add focused regression tests around resolveCronPayloadOutcome, dispatchCronDelivery, and the embedded-run payload/error path while preserving existing NO_REPLY, stale-ack, denial-token, run-level-error, and --no-deliver behavior.
