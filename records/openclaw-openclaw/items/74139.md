## Summary

Current main still contains the false-positive allowlist guard path this PR targets: `disableTools: true` produces no runtime tools, explicit allowlist sources are still collected, and `buildEmptyExplicitToolAllowlistError` still returns the "tools are disabled for this run" error. The linked bug #74019 remains open, and this PR is a narrow, relevant implementation candidate rather than obsolete work.

## Best Possible Solution

Keep this PR open and review/land the narrow guard change if CI passes. The fix should treat `disableTools: true` as an intentional no-tool run while preserving the existing fail-closed error for missing or unavailable explicitly allowed tools.
