## Summary

Keep open. Current main still lacks a durable path-preserving Mintlify locale switcher, and the prior standalone-script approach was already rejected because docs-root JavaScript is not proven to deploy or stay synchronized with the generated locale source of truth.

## What This Changes

The issue requests that the Mintlify docs language switcher preserve the current page path when switching locales instead of navigating to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the boundary: either get Mintlify to preserve page paths upstream, or wire a repo-owned workaround through a supported docs publish/config mechanism that derives locale mappings from GENERATED_LOCALES and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue gives concrete live-docs steps, the earlier #34021 report corroborates the same behavior, and current source/live asset checks show no deployed path-preserving handler; final fix validation still needs an interactive browser switch from /install/docker to /zh-CN/install/docker.

## Solution Assessment

Unclear. The previously attempted standalone docs-root JavaScript file is not the best fix because deployment is unproven and mappings would drift; the maintainable solution is a verified Mintlify-supported hook, publish-pipeline integration, or upstream Mintlify fix.
