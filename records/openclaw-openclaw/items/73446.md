## Summary

Keep open. Current main still has no durable locale path-preservation hook for the Mintlify language switcher, and the related standalone-script PRs were closed unmerged because the repo has no verified custom-script deployment path and the locale mapping is generated by the docs publish pipeline.

## What This Changes

Preserve the current OpenClaw docs page path when switching languages in the Mintlify language dropdown instead of navigating to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose either an upstream Mintlify fix or a repo-owned workaround wired through a supported docs publish/custom-script mechanism, with locale mapping derived from `GENERATED_LOCALES` and verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue gives concrete live-docs steps and prior PRs corroborate the behavior; current source and live asset checks show there is still no deployed path-preserving handler. Final acceptance should still use an actual browser switch from `/install/docker` to `/zh-CN/install/docker`.

## Solution Assessment

Unclear for a repo workaround, and no for the previous standalone root JavaScript approach. The maintainable path is a verified Mintlify-supported hook or publish-pipeline integration that stays synchronized with `GENERATED_LOCALES`, or an upstream Mintlify platform fix.
