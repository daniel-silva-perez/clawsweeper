## Summary

Keep open. Current main still lacks a durable path-preserving locale switcher or a verified Mintlify custom-script deployment hook, and the issue is recent with concrete reproduction plus prior closed PR attempts that explain why a standalone docs-root script is too fragile.

## What This Changes

The issue asks for the Mintlify docs language switcher to preserve the current page path when switching locales instead of navigating to each locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the boundary: either get Mintlify to preserve locale paths upstream, or wire a repo-owned workaround through a supported docs publish/custom-script mechanism that derives locale mapping from `GENERATED_LOCALES` and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue and prior report provide concrete live-docs steps, and current source/live asset checks show no deployed path-preserving handler; final fix validation should still use a real browser switch from `/install/docker` to `/zh-CN/install/docker`.

## Solution Assessment

No for the previously attempted standalone docs-root JavaScript approach. The maintainable path is a verified Mintlify-supported hook or publish-pipeline integration that stays aligned with `GENERATED_LOCALES`, or an upstream Mintlify platform fix.
