## Summary

Keep this open because the report is current, concrete, and not implemented on current main. The source repo still has no locale path-preservation handler, live docs still do not serve the existing root docs JS/CSS assets, and the docs i18n flow is split across the publish pipeline, so a durable fix needs maintainer confirmation of the Mintlify-supported deployment hook or an upstream platform fix.

## What This Changes

The issue asks for the Mintlify docs language switcher to preserve the current page path when switching locales instead of navigating to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the product boundary. If OpenClaw owns a workaround, it should use a Mintlify-supported deployment hook, derive locale mappings from `GENERATED_LOCALES` or generated publish config, and be verified against a Mintlify preview or live docs deployment; otherwise the right outcome is an upstream Mintlify bug report for path-preserving language switching.
