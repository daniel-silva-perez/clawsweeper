## Summary

Keep open. Current main and the live docs still reproduce the path-loss behavior, and the repository still lacks a durable path-preserving locale switcher or verified Mintlify custom-script deployment hook. The prior standalone-script PRs were closed unmerged after maintainer review because docs-root assets are not served directly and the locale mapping would drift from the generated docs locale source.

## What This Changes

The issue requests that the Mintlify docs language switcher preserve the current docs page path when changing locales instead of returning to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the product boundary. The desired end state is either an upstream Mintlify language-switcher fix, or a repo-owned workaround wired through a supported docs publish/custom-script mechanism, deriving locale mapping from `GENERATED_LOCALES` and verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The live browser checks reproduce both directions: `/install/docker` to Simplified Chinese lands on `/zh-CN`, and `/zh-CN/install/docker` to English lands on `/`. The DOM also confirms the picker items are `DIV` elements without `href`, matching the report.

## Solution Assessment

No for the previously attempted standalone docs-root JavaScript file, because live docs do not serve those root assets and the hard-coded locale list can drift from `GENERATED_LOCALES`. The safer path is a verified Mintlify-supported hook or publish-pipeline integration, or an upstream Mintlify platform fix.
