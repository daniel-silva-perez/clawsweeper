## Summary

Keep open. Current main still has the active-subagent pruning gap, and the related user bug #56034 remains unresolved, but this PR is not mergeable as-is because its new active-key helper only reads the in-memory registry while existing cross-process subagent queries intentionally merge persisted registry state.

## What This Changes

The PR adds protected session-key plumbing so active subagent child sessions are skipped by stale-entry pruning, entry-count capping, and disk-budget eviction.

## Best Possible Solution

Merge a corrected version that computes active subagent child session keys from the same persisted snapshot seam used by other subagent registry readers, then threads those keys through current main's maintenance preservation path and disk-budget eviction path with targeted regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to seed a store with an old or over-limit active subagent child session, expose its run only through the persisted subagent registry snapshot, and trigger enforce-mode session maintenance from a different active session; current main and the PR helper's in-memory-only lookup can still prune the child.

## Solution Assessment

No. The direction is right, but the best fix must read `subagentRegistryDeps.getSubagentRunsSnapshotForRead(subagentRuns)` before building protected keys; otherwise the protection does not match the existing cross-process registry contract.
