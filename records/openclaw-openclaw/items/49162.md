## Summary

Keep open. Current main still has adjacent timeout/cache/retry behavior in memory-related paths, but the requested schema-backed `agents.defaults.memorySearch.query.timeoutMs`, `cacheTtlMs`, and `retry` controls are still absent from the core config schema, resolved per-agent runtime config, docs/help metadata, and builtin memory-search execution path.

## What This Changes

Add durable, schema-backed per-agent memory search query reliability controls for timeout, query-result cache TTL, and retry policy.

## Best Possible Solution

Implement the controls at the memory-search config/runtime boundary: add typed and zod-validated `memorySearch.query.timeoutMs`, `cacheTtlMs`, and `retry`; resolve them per agent; apply them to builtin memory search with graceful fallback; update generated schema/help and memory docs; keep the linked durability/provider issues as adjacent context rather than substitutes.

## Reproduction Assessment

Not applicable as a runtime bug report. The high-confidence verification path is static source inspection: the proposed keys are absent from the strict schema, resolved config, generated help, and memory-search execution path on current main.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix. Adding schema-backed controls to the existing memory-search query boundary is better than treating QMD-only or Active Memory plugin-local settings as substitutes.
