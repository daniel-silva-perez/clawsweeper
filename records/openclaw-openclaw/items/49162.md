## Summary

Keep this issue open. Current main still has adjacent memory-search reliability behavior, including hard-coded/provider query embedding timeouts, batch embedding retries, Active Memory plugin timeout/cache settings, and QMD search timeout settings. It still does not expose the requested schema-backed `agents.defaults.memorySearch.query.timeoutMs`, query-result `cacheTtlMs`, or query retry policy through the normal per-agent memory search config path.

## Best Possible Solution

Implement the requested controls in the memory-core/config boundary: add typed and zod-validated `memorySearch.query.timeoutMs`, `memorySearch.query.cacheTtlMs`, and `memorySearch.query.retry`; resolve them in `src/agents/memory-search.ts`; wire them into the memory search execution path with graceful timeout, retry, cache invalidation, and fallback behavior; add focused schema/runtime tests; and document the public options in the memory configuration reference. Keep #49056 and #49233 as adjacent durability/provider context rather than replacements for this query reliability surface.

- Do not close this as implemented based on QMD-only `memory.qmd.limits.timeoutMs`; the issue asks for schema-backed `agents.defaults.memorySearch.query` controls in the normal per-agent config path.
