## Summary

Keep this PR open. Current main still does not implement the PR’s intended `session.mainKey` routing behavior: runtime materialization overwrites custom values, central route builders hard-code `DEFAULT_MAIN_KEY`, current outbound and bundled Telegram/WhatsApp paths do not thread the configured key, and the remaining main-session delivery guard concern from review still applies.

## Best Possible Solution

Keep this PR open as an implementation candidate, but rebase or port it onto current main. The complete fix should preserve configured `session.mainKey` through runtime materialization, thread the normalized key through central route/session-key builders and plugin SDK callers, update current outbound/Telegram/WhatsApp paths, make delivery guards recognize configured custom main keys and legacy aliases, align docs/generated config help, and add focused regression coverage for config preservation, route resolution, outbound/plugin routing, and delivery-continuity behavior.

## Risks / Open Questions

- Merging the stale PR as-is would likely miss current refactored paths under `extensions/telegram`, `extensions/whatsapp`, and `src/infra/outbound/base-session-key.ts`.
- Honoring custom main keys only in some builders could split inbound, outbound, heartbeat, cron, and delivery-continuity state across different session keys.
- Without updating main-session guards, internal or webchat turns for a custom main key can be treated as non-main and may reuse stale external delivery context.
