## Summary

Keep this PR open. Current main still resolves explicit heartbeat targets through deliverable-channel normalization before plugin-only channel ids can be bootstrapped into the active registry. Telegram is now covered as a bundled channel, but the documented/plugin contract still allows custom plugin channel ids, and that remaining load-order gap is not implemented on main.

## Best Possible Solution

Keep this PR open or replace it with a focused successor that bootstraps or reads plugin channel metadata before deliverable-channel recognition, retries normalization after registry initialization, and adds regression coverage for `agents.defaults.heartbeat.target` pointing at a plugin-only channel while the active registry is initially empty. The security review should keep the bootstrap side effect narrow and avoid broadening plugin code execution beyond configured/enabled plugin discovery.

## Risks / Open Questions

- The PR’s current shape moves plugin bootstrap into a normalization helper. That may be the right seam, but reviewers should explicitly check startup/load side effects because plugin runtime loading can execute configured plugin code earlier than before.
- There is no regression coverage in the PR diff for a plugin-only heartbeat target with an initially empty active registry. Existing tests cover bundled Telegram and mocked plugin targets, but not the failing load-order path.
- The Telegram example is now less representative because Telegram is bundled on current main. A regression should use a custom/plugin-only fixture rather than relying on Telegram.
