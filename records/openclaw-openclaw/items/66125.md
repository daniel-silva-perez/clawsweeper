## Summary

Keep open. Current main includes related OpenAI-compatible proxy hardening, but it does not prove the reported fallback proxy turn now completes. The maintainer comments explicitly kept this issue open after those fixes because the remaining claim is broader: a selected fallback candidate still falls through without the exact real-agent request/response needed to identify the next sanitizer, strict-minimal mode, or logging change.

## Best Possible Solution

Keep this issue open until the reporter or a maintainer captures the failed fallback request body plus proxy response/status on current main or latest release. Then implement the narrowest fix in the OpenAI-compatible completions path, likely a provider/model compat option for strict-minimal text-only payloads or another targeted payload sanitizer, and add a regression test around the actual incompatible field.
