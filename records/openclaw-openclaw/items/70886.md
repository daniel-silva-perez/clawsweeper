## Summary

Keep open. Current main still routes ZAI image understanding through the configured provider base URL without a ZAI vision-model rewrite, and open PR #70915 uses closing syntax for this issue, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw's image tool to analyze images with ZAI vision models such as `zai/glm-4.6v` when ZAI is configured with Coding Plan endpoints instead of returning a provider 404.

## Best Possible Solution

Review and either land #70915 or replace it with an equivalent narrow ZAI-owned fix that routes only known ZAI vision models from Coding Plan base URLs to the matching standard multimodal endpoint, while keeping text/coding models on the configured Coding Plan endpoint and adding regression coverage for both global and CN routes.

## Reproduction Assessment

Yes for maintainers with a ZAI key: configure ZAI with `zai-coding-global` or `zai-coding-cn`, set `agents.defaults.imageModel` to `zai/glm-4.6v`, and call the `image` tool. This review is source-backed rather than live-backed because no ZAI credential was available for a provider request.

## Solution Assessment

Yes, keeping this issue open is the right triage outcome because #70915 is the active open implementation candidate. The code fix should stay in the ZAI provider/media-understanding boundary or a narrowly tested routing seam, not a broad core migration.
