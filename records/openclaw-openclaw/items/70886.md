## Summary

Keep open. Current `main` still has the ZAI image route described by the report, and there is an open, unmerged closing PR specifically targeting this issue, so the issue should remain open until maintainers land or close that implementation path.

## What This Changes

The issue requests fixing ZAI vision image-tool calls so `zai/glm-4.6v` and related ZAI vision models use a working multimodal endpoint instead of returning 404 from the Coding Plan route.

## Best Possible Solution

Keep this issue open while maintainers review and either land #70915 or replace it with an equivalent narrow fix. The end state should be provider-local ZAI routing that sends only vision models such as `glm-5v-turbo`, `glm-4.6v`, and `glm-4.5v` from Coding Plan base URLs to the same-region standard multimodal endpoint, with regression coverage proving text/coding models keep their Coding Plan URL.

## Reproduction Assessment

Yes for maintainers with a ZAI key: configure ZAI with `zai-coding-global` or `zai-coding-cn`, set `agents.defaults.imageModel` to `zai/glm-4.6v`, and call the `image` tool. In this read-only review, the high-confidence path is source-backed rather than live-backed because no ZAI key was available.

## Solution Assessment

Yes, keeping this issue open is the right triage outcome because an open closing PR is the active implementation candidate. The code fix should stay in the bundled ZAI provider/media-understanding boundary or model catalog routing, not in a broad core startup migration.
