## Summary

Keep open. Current main has addressed the OAuth-scope regression by preserving the upstream Codex OAuth authorize URL, and it has better Cloudflare/HTML error classification, but the remaining headless-container Cloudflare challenge is still on the active `openai-codex` runtime route and was explicitly left open by maintainer triage.

## What This Changes

The issue reports that `openai-codex` requests fail in Docker with Cloudflare challenge HTML from ChatGPT backend routes, alongside a missing OAuth-scope symptom and an onboarding auth-store concern.

## Best Possible Solution

Keep this issue open as the Cloudflare/headless-container tracker. The best end state is either a maintainer-approved runtime path that works for Codex OAuth in headless Docker without violating provider constraints, or clear docs/doctor guidance that steers affected users to supported Codex device-code/native harness/API-key routes while leaving the already-fixed OAuth-scope regression closed under #64713.

## Reproduction Assessment

Yes. The issue gives concrete Docker, auth, model, and log steps, and current main still routes `openai-codex` traffic to the same ChatGPT backend family; live reproduction would require a real Codex OAuth profile and the affected network/container path.

## Solution Assessment

Unclear. The removed scope mutation is the right narrow fix for the OAuth-scope half, but bypassing Cloudflare challenges or switching subscription OAuth traffic to public API scopes is not a safe automated code change without maintainer product/security review.
