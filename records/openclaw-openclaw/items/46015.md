## Summary

Keep open. Current main has the fallback diagnostics and general per-agent TTS config path, but automatic TTS still treats the selected ElevenLabs provider as the first provider in a fallback chain unless callers explicitly set `disableFallback`. The remaining provider-stickiness or fail-closed contract called out in the issue discussion is not implemented, and the docs/latest release still describe fallback as expected behavior.

## Best Possible Solution

Keep this issue open as the canonical remaining decision for auto-TTS provider stickiness/fail-closed behavior. A maintainer should choose the contract first: either explicit `messages.tts.provider` should fail closed for automatic replies, a separate strict/no-fallback option should be added and wired through auto TTS and `/tts audio`, or fallback should remain intentional and the docs/status wording should make that explicit.
