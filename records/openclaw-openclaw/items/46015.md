## Summary

Keep open. Current main has addressed fallback observability and now has a generic per-agent TTS config path, but automatic TTS still treats the configured ElevenLabs provider as the first provider in a documented fallback chain rather than a locked provider, which leaves the provider-stickiness policy from the issue discussion unresolved.

## What This Changes

The issue asks Discord automatic TTS to honor an explicitly configured ElevenLabs provider, avoid silent fallback to Edge TTS, and support per-agent voice IDs like Telegram.

## Best Possible Solution

Keep this as the canonical maintainer decision for auto-TTS provider stickiness: either document fallback as intentional for explicitly configured providers, or add a narrow opt-in contract that makes automatic replies fail closed when a provider, model, or voice is explicitly selected while preserving fallback for existing users.

## Reproduction Assessment

Yes for the remaining fallback behavior: current main shows automatic replies call textToSpeech without disableFallback, and synthesizeSpeech walks the resolved fallback provider list. A live ElevenLabs account is not required to reproduce that contract from source.

## Solution Assessment

Unclear. Forcing fail-closed automatic TTS would satisfy this report, but the current docs and tests preserve fallback as expected behavior, so the best fix depends on a maintainer choosing a provider-stickiness policy or a new opt-in setting.
