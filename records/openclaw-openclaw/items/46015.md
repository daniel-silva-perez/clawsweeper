## Summary

Keep open. Current main has the partial fixes discussed in #57953/#57954: per-agent TTS config and fallback diagnostics are present, but automatic reply TTS still builds a provider fallback chain and there is no config-level strict provider/stickiness policy for auto TTS.

## Best Possible Solution

Keep this issue open for the remaining product/code decision: add or explicitly reject a strict TTS provider policy for automatic replies, then cover Discord auto-TTS with a regression test that proves configured ElevenLabs either stays strict or reports fallback according to the chosen policy. If maintainers prefer #52186 as the canonical provider-stickiness tracker, close this only after linking that thread and confirming the Discord-specific context is represented there.
