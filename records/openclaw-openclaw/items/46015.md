## Summary

Keep open. Current main has the diagnostics and per-agent TTS config path, but automatic TTS still treats the configured ElevenLabs provider as the primary member of a fallback chain, which is the remaining provider-stickiness behavior called out in the maintainer discussion.

## What This Changes

The issue requests that Discord automatic TTS respect a configured ElevenLabs provider without silently falling back to Edge/Microsoft, and that Discord support per-agent voiceId configuration like Telegram.

## Best Possible Solution

Keep this as the canonical maintainer decision point for automatic TTS provider stickiness: either document fallback as intentional for explicitly configured providers, or add a narrow opt-in contract that makes automatic replies fail closed when a provider/model/voice is explicitly pinned while preserving fallback for existing users.

## Reproduction Assessment

Yes for the remaining fallback behavior. Current main shows automatic replies call textToSpeech without disableFallback, and synthesizeSpeech walks the resolved fallback provider list, so a live ElevenLabs account is not needed to reproduce that contract from source.

## Solution Assessment

Unclear. Forcing fail-closed automatic TTS would satisfy this report, but current docs, types, and tests preserve fallback as expected behavior, so the best fix needs a maintainer product decision or an opt-in provider-stickiness setting.
