## Summary

Keep open. Current main has resolved the diagnostics and per-agent TTS configuration parts, but the remaining provider-stickiness behavior is still intentionally not implemented: automatic TTS treats the configured ElevenLabs provider as the primary member of a fallback chain rather than as an exclusive, fail-closed provider.

## What This Changes

The issue asks Discord automatic TTS to use the configured ElevenLabs provider, surface fallback errors, and support per-agent TTS voice/provider overrides.

## Best Possible Solution

Keep this as the canonical maintainer decision point for automatic TTS provider stickiness: either document fallback as intentional for explicitly configured providers, or add a narrow opt-in strict-provider contract that fails closed when a provider/voice is pinned while preserving existing fallback behavior by default.

## Reproduction Assessment

Yes for the remaining fallback behavior. Current main shows automatic replies call textToSpeech without disableFallback, and synthesizeSpeech walks the resolved fallback provider list, so the remaining contract can be verified from source without a live ElevenLabs account.

## Solution Assessment

Unclear. Forcing fail-closed automatic TTS would satisfy this report, but current docs, types, and tests preserve fallback as expected behavior, so the maintainable fix needs a product decision or an opt-in strict-provider setting.
