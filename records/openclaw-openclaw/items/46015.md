## Summary

Keep open. Current main has fallback diagnostics and a generic per-agent TTS config path, but automatic TTS still treats the selected ElevenLabs provider as the first provider in an automatic fallback chain rather than a locked provider; the remaining provider-stickiness/fail-closed contract still needs maintainer judgment.

## What This Changes

The issue requests Discord auto TTS to respect the configured ElevenLabs provider, surface fallback errors, and support per-agent voiceId overrides.

## Best Possible Solution

Keep this as the canonical maintainer decision for auto-TTS provider stickiness: either document fallback as intentional for configured providers, or add a narrow opt-in/contract that makes explicit provider selection fail closed for automatic replies while preserving existing fallback users.

## Reproduction Assessment

Yes for the remaining fallback behavior: the current source path shows automatic replies call `textToSpeech` without `disableFallback`, and `synthesizeSpeech` proceeds through the resolved fallback provider list. A live ElevenLabs account is not required to reproduce that contract from current main.

## Solution Assessment

Unclear. A forced fail-closed change may satisfy this report, but the current docs advertise fallback as expected behavior, so the best implementation depends on a maintainer choosing whether explicit `messages.tts.provider` should lock the provider or whether a separate config flag/directive should do that.
