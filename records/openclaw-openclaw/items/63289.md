## Summary

Keep this issue open. Current main still has the gog calendar skill, tool catalog plumbing, tool-call normalization, tool-result repair, Anthropic tool_choice forwarding, and OpenResponses client-tool forcing, but I did not find a deterministic guard that prevents a final answer to an explicit calendar or raw shell-output request when no matching tool call was emitted.

## What This Changes

The issue requests that factual, time-sensitive calendar/raw-output requests require a fresh authoritative tool invocation and never be answered from memory or prior context without a matching tool trace.

## Best Possible Solution

Keep this issue open and have a maintainer define a narrow contract for explicit freshness or raw-output requests, such as calendar checks and instructions like “run this command and paste raw output.” A good fix likely belongs around the Pi embedded runner, prompt/tool policy boundary, or provider request policy, with regression coverage proving the assistant cannot finalize those explicit turns without a matching tool event. Avoid turning every arbitrary factual question into mandatory tool use.
