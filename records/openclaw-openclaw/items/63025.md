## Summary

Keep this PR open. Current main still has the empty embedded-dispatch completion path: an undefined/no-payload resolver becomes `replies = []`, no fallback is queued, counts are read, processing is recorded as completed, and the function returns with zero final/tool/block output. The PR is not obsolete, but it should be revised before merge to respect delivery suppression and the canonical chat-type contract.

## Best Possible Solution

Keep this PR open and review/update it against current main. The best fix is a narrow dispatch-level fallback after final queued counts are known, guarded by `!suppressDelivery`, direct-style conversation policy using `normalizeChatType` or an intentional canonical contract extension, no routed-origin delivery, no queued final/tool/block output, no block-stream/TTS output, and no intentional handled/no-text or plugin-handled branch. Add focused tests for the direct empty-output path and the suppressed/no-op cases.

## Risks / Open Questions

- The provided PR patch places the fallback after the normal `!suppressDelivery` block and does not include `!suppressDelivery` in the fallback condition, so a direct/private empty run under `sendPolicy: "deny"` could still enqueue an outbound error reply.
- The PR’s local `direct|dm|private` helper can drift from the canonical `normalizeChatType` contract and config/schema behavior unless `private` is deliberately added and tested.
- Regression coverage should include sendPolicy deny, routed-origin delivery, block-stream/TTS output, group/channel silence, and intentional handled-without-text no-op cases before merge.
