## Summary

Keep open. Current main still has the zero-output embedded dispatch path this PR targets, but the proposed patch is not merge-ready because its fallback can bypass explicit delivery suppression for direct-like sessions.

## What This Changes

The PR adds a static error fallback in `dispatchReplyFromConfig` when direct/private embedded dispatch returns no final, tool, or block output, plus regression tests for the fallback and channel no-op path.

## Best Possible Solution

Land a narrow dispatch-level empty-output fallback after final queued counts are known, but guard it with delivery policy, direct-style conversation policy, no routed-origin delivery, no queued final/tool/block output, no block-stream/TTS output, and no intentional handled/no-text branch. Use `normalizeChatType` unless `private` is intentionally added to the canonical contract with tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor delivery suppression before sending the fallback:** `src/auto-reply/reply/dispatch-from-config.ts:1050-1058`
  - body: The fallback is evaluated after normal final delivery but outside the existing `!suppressDelivery` guard. For a direct/private session with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"` and an empty resolver result, counts remain zero and this branch still queues `sendFinalReply`, bypassing the no-delivery contract protected by existing tests. Gate the fallback on `!suppressDelivery` and add a deny/message-tool-only regression before merging.
  - confidence: 0.91
