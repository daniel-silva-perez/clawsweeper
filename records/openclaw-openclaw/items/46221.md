## Summary

Keep open. The underlying delivery gap still exists on current main, and this PR is useful source work, but the branch is dirty and the proposed fallback has active-agent isolation and non-main-session scope concerns that need maintainer review before merge.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into non-main session drains and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side merge onto current main using the active session's agent id for the fallback main key, then either restrict fallback draining to approved direct-message scopes or explicitly document and test the broader non-main-session behavior, including multi-agent and group/channel/thread cases.

## Reproduction Assessment

Yes. Enqueue a system event under resolveMainSessionKey(cfg), then drain formatted events for an active per-channel-peer session key; current main drains only the active key, so the main-key event remains queued.

## Solution Assessment

No. The receiver-side merge is the right direction, but this patch resolves the fallback from the configured default agent and applies it to every non-main session; deriving the fallback from params.sessionKey and constraining or fully covering the intended scopes is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default or first agent's main key, not the agent for the active session. A turn in `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback main key from `params.sessionKey` before merging.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom session keys. That lets an unrelated non-DM turn consume and clear a main-session wake event, so restrict this fallback to approved scopes or add explicit maintainer-approved coverage for the broader behavior.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff can route untrusted system-event context across agent or conversation boundaries until fallback key derivation and delivery scope are constrained.

Concerns:

- **[medium] Cross-agent system-event drain:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: Using `resolveMainSessionKey(params.cfg)` for fallback draining lets a non-default active agent drain the configured default agent's main queue while missing its own agent-scoped queue. Because wake hooks enqueue untrusted text, this can inject context into the wrong agent/session prompt.
  - confidence: 0.86
- **[low] Over-broad non-main session drain:** `src/auto-reply/reply/session-system-events.ts:83`
  - body: The fallback applies to every non-main session shape, including group, channel, thread, and custom sessions. That can move main-session wake context into unrelated conversations and clear it before the intended session sees it.
  - confidence: 0.76
