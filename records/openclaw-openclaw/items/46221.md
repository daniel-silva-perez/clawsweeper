## Summary

Keep open. Current main still has the reported delivery gap, but this PR is not merge-ready because the proposed fallback drains the config-default main queue instead of the active agent's main queue, and the intended scope for non-main sessions still needs maintainer approval.

## What This Changes

This PR merges the canonical main-session system-event queue into non-main session drains, sorts the merged events by timestamp, and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side merge onto current main using the active session's agent id for the fallback main key, then either restrict the fallback to approved direct-message scopes or explicitly document and test the broader non-main-session behavior, including multi-agent isolation and group/channel/thread cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Derive the fallback main key from the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: The patch calls resolveMainSessionKey(params.cfg), which chooses the configured default or first agent. In an active session such as agent:ops:telegram:direct:..., this drains agent:main:main while leaving agent:ops:main queued; cron already enqueues agent-scoped main-session events, so derive the fallback from params.sessionKey before merging.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The mainSessionKey !== params.sessionKey guard matches every non-main key, including group, channel, thread, and custom sessions. That can consume and clear a main-session wake event in an unrelated conversation, so restrict this to approved DM isolation scopes or add explicit maintainer-approved coverage for the broader behavior.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff can route untrusted system-event context across agent or conversation boundaries until the fallback key and delivery scope are constrained.

Concerns:

- **[medium] Cross-agent system-event drain:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: Using resolveMainSessionKey(params.cfg) for fallback draining means a non-default active agent can drain the default agent's main queue while missing its own agent-scoped queue. Because wake hooks enqueue untrusted text, this can inject context into the wrong agent/session prompt.
  - confidence: 0.86
- **[low] Over-broad non-main session drain:** `src/auto-reply/reply/session-system-events.ts:83`
  - body: The fallback applies to every non-main session shape, including group, channel, thread, and custom sessions. That can move main-session wake context into unrelated conversations and clear it before the intended session sees it.
  - confidence: 0.76
