## Summary

Keep open. Current main still has the reported main-key-to-active-session delivery gap, and this PR is useful source work, but the proposed patch is not merge-ready because it drains the configured default agent's main queue instead of the active agent's queue and applies that fallback to every non-main session shape.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into non-main session drains and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side merge onto current main using the active session's agent id for the fallback main key, then either restrict fallback draining to approved direct-message scopes or explicitly document and test the broader non-main-session behavior, including multi-agent and group/channel/thread cases.

## Reproduction Assessment

Yes. The source-level reproduction is to enqueue a system event under resolveMainSessionKey(cfg), then run drainFormattedSystemEvents for an active per-channel-peer key; current main drains only the active key, so the main-key event remains queued.

## Solution Assessment

No. The receiver-side approach is the right direction, but this patch resolves the fallback from the configured default agent and drains it for every non-main session; deriving the fallback from params.sessionKey and constraining or fully testing the intended scope is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Derive the fallback key from the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default agent's main key, not the agent for the active session. A turn in `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback from the active session's agent id before merging.
  - confidence: 0.91
- **[P2] Limit fallback draining to approved session scopes:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event, so restrict the fallback to approved scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff can route untrusted system-event context across agent or conversation boundaries until the fallback key and delivery scope are constrained.

Concerns:

- **[medium] Cross-agent system-event drain:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: Using resolveMainSessionKey(params.cfg) for fallback draining lets a non-default active agent drain the configured default agent's main queue while missing its own agent-scoped queue. Because wake hooks enqueue untrusted text, this can inject context into the wrong agent/session prompt.
  - confidence: 0.86
- **[low] Over-broad non-main session drain:** `src/auto-reply/reply/session-system-events.ts:83`
  - body: The fallback applies to every non-main session shape, including group, channel, thread, and custom sessions. That can move main-session wake context into unrelated conversations and clear it before the intended session sees it.
  - confidence: 0.76
