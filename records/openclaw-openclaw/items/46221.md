## Summary

Keep open: current main still has the reported main-key drain gap, but this PR should not merge as-is because the fallback drain is keyed to the config default agent and the intended scope for non-main sessions still needs maintainer approval.

## What This Changes

The PR changes formatted system-event draining to also merge the canonical main-session queue and adds a regression test for per-channel-peer Telegram DM delivery.

## Best Possible Solution

Port the receiver-side fix onto current main using the active session's agent id for the fallback main key, then either restrict the fallback to approved direct-message scopes or document and test the broader non-main-session behavior explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default agent's main key, not the active session's agent. For a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so agent-specific events can cross agent boundaries or remain stuck. Derive the fallback from `params.sessionKey` instead.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, and thread sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event, injecting that context outside the intended conversation. Restrict this fallback to the approved DM scopes or add explicit maintainer-approved coverage for the broader behavior.
  - confidence: 0.78
