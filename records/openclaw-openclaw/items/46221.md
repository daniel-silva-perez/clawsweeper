## Summary

Keep open. The PR targets a real current-main delivery gap, but the proposed fallback drain is not merge-ready because it derives the fallback queue from the configured default main session instead of the active agent, and it broadens delivery to every non-main session shape without maintainer-approved scope and coverage.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into non-main active sessions and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side merge onto current main using the active session's agent id for the fallback main key, then either restrict fallback draining to approved direct-message scopes or explicitly document and test the broader non-main-session policy across multi-agent, group, channel, and thread cases.

## Reproduction Assessment

Yes. Enqueue a system event under `resolveMainSessionKey(cfg)`, then drain formatted events for `agent:main:telegram:direct:<peer>` under `per-channel-peer`; current main drains only the active key, so the main-key event remains queued unless a main-session turn drains it separately.

## Solution Assessment

No. Receiver-side merging is the right direction, but this patch should derive the fallback key from `params.sessionKey` and should not broaden delivery to every non-main session without an explicit maintained policy and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` selects the configured default or first agent, not the agent encoded in `params.sessionKey`. A turn for `agent:ops:telegram:direct:...` can drain `agent:main:main` while still missing `agent:ops:main`, so derive the fallback key from the active session's agent id before merging.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` guard also matches group, channel, thread, and custom session keys. That lets an unrelated non-DM turn consume and clear main-session wake context, so restrict this fallback to approved scopes or add maintainer-approved coverage for the broader behavior.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff can route untrusted system-event context across agent or conversation boundaries unless fallback key derivation and delivery scope are constrained.

Concerns:

- **[medium] Cross-agent system-event drain:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: Using `resolveMainSessionKey(params.cfg)` for fallback draining lets a non-default active agent drain the configured default agent's main queue while missing its own agent-scoped queue; `/hooks/wake` enqueues untrusted text, so this can inject context into the wrong prompt.
  - confidence: 0.86
- **[low] Over-broad non-main session drain:** `src/auto-reply/reply/session-system-events.ts:83`
  - body: The fallback applies to every non-main session shape, including group, channel, thread, and custom sessions, which can move main-session wake context into unrelated conversations and clear it before the intended session sees it.
  - confidence: 0.76
