## Summary

Keep open. Current main still has the stop-timeout stale-store behavior this PR targets, and the PR is relevant to #70024/#71412, but the patch is not merge-ready because releasing the task slot lets the old task's unguarded callbacks later mutate replacement runtime state.

## What This Changes

The PR changes `stopChannel` timeout handling to delete `store.aborts`/`store.tasks`, mark the account stopped, record `lastStopAt`, and updates the gateway channel test to expect restart after timeout.

## Best Possible Solution

Keep the PR open and revise the channel task lifecycle so timeout cleanup can permit recovery without allowing stale task-owned callbacks to write current account state. A task identity or generation guard should cover error writes, final stopped writes, restart bookkeeping, and cleanup; the final fix should include the delayed-settle regression and a changelog entry if this user-facing gateway fix lands.

## Reproduction Assessment

Yes. Current main has a deterministic unit-test shape for the timeout no-restart bug, and a stronger reproduction for the PR race is to let the timed-out task settle after a successful replacement start and assert that the replacement remains running with no stale error or restart bookkeeping.

## Solution Assessment

No. The PR addresses the stale map entry but is not the narrowest maintainable fix because it releases the slot before guarding task-owned callbacks; the safer approach is release-plus-identity/generation guards, with regression coverage for delayed old-task settlement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Guard stale task callbacks before releasing the slot:** `src/gateway/server-channels.ts:579`
  - body: Deleting `store.tasks` in the timeout branch lets a replacement start while the original `trackedPromise` is still pending. When that old task later settles, its existing `.catch`, `.finally`, and restart chain can still write runtime state or restart bookkeeping for the same account without checking that it owns the current task, so it can mark the replacement stopped or write stale errors. Add a task identity/generation guard around those writes and cover a timed-out task that settles after restart.
  - confidence: 0.9
