## Summary

Keep open. Current main still emits and tests the compact Gateway startup log shape, and I found no source, docs, or config surface that restores the requested structured auth/server/model/plugins/log-file/service-status startup summary. The related #14861 is adjacent subsystem outcome logging, not a full replacement for this formatted startup display request.

## What This Changes

Request structured Gateway startup log output showing auth mode/source, server URL, agent model, loaded plugins, log file, and health/heartbeat status instead of the current compact lines.

## Best Possible Solution

Add a redacted, deterministic Gateway-owned startup summary on the existing startup logging path, using resolved runtime state for auth, bind URL, model, sorted plugin list, log file, and coarse health/heartbeat readiness. Keep granular subsystem failure reporting as separate follow-up work under #14861 unless this implementation can share a small generic status seam cleanly.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current `logGatewayStartup` and its focused test assert compact output, and the post-attach and scheduled-service paths do not add the requested structured summary. I did not run a live Gateway because this review is read-only.

## Solution Assessment

Yes, a narrow extension of the existing Gateway startup logging surface is the best maintainable direction. A new config switch should be added only if maintainers want to preserve both minimal and structured modes as product behavior.
