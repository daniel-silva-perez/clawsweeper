## Summary

Keep open. Current main still emits the compact Gateway startup shape and does not provide the requested structured startup snapshot with auth mode, server URL, model, plugins, log path, and startup service status; #14861 is adjacent for subsystem outcome logging but does not fully cover this formatted startup display request.

## What This Changes

The issue requests restoring structured Gateway startup output with auth, server URL, agent model, loaded plugins, log file, health monitor status, and heartbeat status.

## Best Possible Solution

Add one redacted, deterministic Gateway startup summary in the Gateway-owned startup path, reusing existing logger behavior and runtime state, with focused tests and docs for the operator-visible fields; avoid a new config switch unless maintainers explicitly want both minimal and structured modes.

Yes. A high-confidence source-level reproduction exists: the current logGatewayStartup implementation and its test assert a compact startup log and omit the requested auth, server URL, and service-status sections. I did not run a live Gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable direction is to extend the existing Gateway startup logging surface and pass in the already-resolved auth/service state as needed; restoring an exact old emoji banner or adding a config option should be a PR-level product decision, not a prerequisite for keeping this actionable.
