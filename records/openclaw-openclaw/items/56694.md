## Summary

Keep open. Current main does not support the report's original alias-accepted execution premise, but the read-tool contract is still inconsistent enough that closing would lose useful maintainer follow-up: diagnostics accept `file_path`, execution requires canonical `path`, and adjacent display/fingerprint code still recognizes legacy path aliases.

## What This Changes

The issue asks for the embedded read-tool missing-path warning to treat `file` and `filePath` as valid path parameters when models use those aliases.

## Best Possible Solution

Set one coherent read-tool parameter contract across schema, execution validation, diagnostics, display, fingerprinting, docs, and tests: either restore `path`, `file_path`, `file`, and `filePath` aliases end to end, or enforce canonical `path` everywhere and remove stale alias assumptions from non-execution surfaces.
