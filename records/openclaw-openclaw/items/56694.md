## Summary

Keep open. Current main still has the read tool-start diagnostic checking only `path` and `file_path`, so the requested `file` and `filePath` handling is not implemented. The surrounding runtime now explicitly validates read calls as canonical `path` only, so this needs a maintainer contract decision rather than a narrow cleanup close.

## What This Changes

The issue asks for read tool-start diagnostics to recognize `file` and `filePath` path aliases so valid read calls do not emit false “without path” warnings.

## Best Possible Solution

Settle the read-tool parameter contract in one coherent direction: either support `path`, `file_path`, `file`, and `filePath` across schema, validation, execution, diagnostics, display, and tests, or enforce and document canonical `path` only while removing stale alias assumptions from diagnostics/display surfaces.
