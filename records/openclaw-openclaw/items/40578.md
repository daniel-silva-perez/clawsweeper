## Summary

Keep this draft PR open for maintainer review: the underlying per-agent memory isolation gap still exists on current main, but the proposed patch is stale against the current memory-lancedb hook path and has blocking data-isolation correctness issues before it can merge.

## What This Changes

The PR adds an opt-in memory-lancedb `agentScoping` setting, an `agentId` LanceDB column/migration, and context-aware tool/hook filtering so scoped agents read and write only their own memories plus legacy global rows.

## Best Possible Solution

Keep the contribution as useful source work, but require a maintainer-guided rebase/design pass that applies scoping to the current prompt-build and tool paths, validates or encodes agent IDs consistently before every read/write/delete, preserves legacy global rows deliberately, logs migration failures, and adds regression coverage for agent A/B isolation, duplicate detection, forget scoping, default unscoped behavior, and live tool factories.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate agent IDs before scoped memory access:** `extensions/memory-lancedb/index.ts:425`
  - body: When `agentScoping` is enabled, the patch stores `ctx.agentId` raw but strips disallowed characters before search/delete filters. IDs such as `a/b` and `ab` can collapse, letting one agent match another agent's rows while the original agent loses access to its own memories. Validate/reject or encode the ID once and use the same value for writes, reads, and deletes.
  - confidence: 0.93
- **[P2] Scope the current prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, but this patch scopes the legacy `before_agent_start` hook. After rebasing, the automatic recall path would remain unscoped unless the agent filter is applied to the current prompt-build hook.
  - confidence: 0.86
- **[P3] Unwrap factory tools in live coverage:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch converts registered memory tools to factories, but the live memory-lancedb smoke still expects each registered tool to be an object with `.execute`. Unwrap factory tools there too, or the live store/recall/forget test will fail before exercising the feature.
  - confidence: 0.84
- **[P3] Log agentId migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: The migration catch block swallows all `schema`/`addColumns` failures even though scoped reads depend on the new column. Operators enabling `agentScoping` need a warning when the existing table cannot be migrated.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes memory data isolation and currently lets raw and sanitized agent IDs diverge across scoped writes, reads, and deletes.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store the raw `ctx.agentId`, while scoped reads/deletes filter with a stripped version of that ID. Different IDs can collapse to the same filter value, causing cross-agent recall/delete matches or making an agent's own stored rows unreachable.
  - confidence: 0.9
