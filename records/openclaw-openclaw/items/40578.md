## Summary

Keep open. The per-agent memory isolation request is still valid and current main still uses one unscoped memory-lancedb table, but this draft PR should not merge as-is because it scopes the old `before_agent_start` path instead of current main's `before_prompt_build` hook and uses inconsistent agent ID handling across scoped writes, reads, and deletes.

## What This Changes

The PR adds an optional `agentScoping` setting to `memory-lancedb`, adds an `agentId` LanceDB column/migration/filtering path, and converts memory tools/hooks toward agent-context-aware scoping.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation that scopes `memory_recall`, `memory_store`, `memory_forget`, `agent_end`, and the active `before_prompt_build` recall path; uses one validated or encoded owner key consistently for write/read/delete; preserves legacy shared rows deliberately; logs migration failures; documents the config; and adds two-agent isolation regression coverage.

## Reproduction Assessment

Yes. Current main can be checked statically: one shared `MemoryDB`, no `agentId` schema/config, and unscoped `db.search`/`db.store` calls mean two agents using memory-lancedb still share the same memory pool.

## Solution Assessment

No. This PR is directionally useful, but the current patch is not the best mergeable solution because it targets a legacy hook path and has inconsistent scoped-agent key handling; the safer path is a rebase onto current main with a single explicit storage and agent-ID contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use one validated agent key for scoped memories:** `extensions/memory-lancedb/index.ts:425-455`
  - body: When `agentScoping` is enabled, the patch stores raw `ctx.agentId` values but reads and deletes with a stripped value. IDs containing characters such as `/` or spaces can become unrecoverable or collapse into another scope, so validate/reject or encode once and use that same key for writes, reads, and deletes.
  - confidence: 0.92
- **[P2] Scope the active prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-601`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, and the tests assert `before_agent_start` is no longer registered. This patch scopes the legacy hook path, so automatic recall would remain unscoped after rebase.
  - confidence: 0.9
- **[P3] Instantiate factory tools in live coverage:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch registers memory tools as factories, but the live smoke path discussed in review still treats registered tools as objects and calls `.execute` directly. Unwrap factory tools with a context before invoking them or live store/recall/forget coverage will throw.
  - confidence: 0.86
- **[P3] Log agent ID migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: `migrateAgentIdColumn` catches all schema or `addColumns` failures without warning. If migration fails, operators can enable scoping while existing tables remain effectively unscoped with no diagnostic, so surface a warning through the plugin logger.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes a memory isolation boundary and currently lets scoped writes and scoped reads/deletes use different agent ID values.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads and deletes filter a stripped value. Unexpected IDs can become unrecoverable or collide with another normalized ID, undermining the isolation guarantee this feature is meant to provide.
  - confidence: 0.9
- **[medium] Migration failures hide disabled isolation:** `extensions/memory-lancedb/index.ts:119`
  - body: The best-effort `agentId` column migration swallows errors without logging, so a failed schema update can leave existing tables unscoped while operators receive no warning.
  - confidence: 0.82
