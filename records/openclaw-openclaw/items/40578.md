## Summary

Keep open. Current main still has no per-agent memory-lancedb scoping, and this draft PR addresses a real multi-agent memory isolation problem, but the proposed patch is not mergeable because it has scoped read/write consistency problems, is stale against the active prompt-build recall hook, and needs maintainer agreement on the storage and agent-ID contract.

## What This Changes

This PR adds an optional `agentScoping` setting to `memory-lancedb`, adds an `agentId` LanceDB column/migration/filtering path, and refactors memory tools and hooks toward agent-context-aware scoping.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation that scopes `memory_recall`, `memory_store`, `memory_forget`, `agent_end`, and the active `before_prompt_build` recall path; uses one validated or encoded owner key consistently; preserves legacy shared rows deliberately; logs migration failures; updates config metadata and docs; and adds two-agent isolation regression coverage.

## Reproduction Assessment

Yes. Current main can be checked statically: one shared `MemoryDB`, no `agentId` schema/config, and unscoped `db.search`/`db.store` calls mean two agents using memory-lancedb still share the same memory pool; the PR-specific key mismatch also reproduces with an agent ID containing characters such as `/` or spaces.

## Solution Assessment

No. The PR is directionally useful, but it is not the best mergeable solution because it is stale against current main's hook path and does not define a consistent agent-ID storage/filter contract; a rebased patch with explicit isolation semantics is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use one validated agent key for scoped memories:** `extensions/memory-lancedb/index.ts:425-452`
  - body: When `agentScoping` is enabled, the patch stores raw `ctx.agentId` values but reads and deletes with a stripped value. IDs containing characters such as `/` or spaces can become unrecoverable or collapse into another scope, so validate/reject or encode once and use that same key for writes, reads, duplicate checks, and deletes.
  - confidence: 0.92
- **[P2] Scope the active prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-596`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, and the tests assert `before_agent_start` is no longer registered. This patch scopes the legacy hook path, so automatic recall would remain unscoped after rebase.
  - confidence: 0.9
- **[P3] Instantiate factory tools in live coverage:** `extensions/memory-lancedb/index.test.ts:372-378`
  - body: The patch registers memory tools as factories, but the live smoke path still treats registered tools as objects and calls `.execute` directly. Unwrap factory tools with a context before invoking them or live store/recall/forget coverage will throw.
  - confidence: 0.86
- **[P3] Log agent ID migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: `migrateAgentIdColumn` catches schema or `addColumns` failures without warning. If migration fails, operators can enable scoping while existing tables remain effectively unscoped with no diagnostic, so surface a warning through the plugin logger.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies, workflows, or secret handling, but it changes a memory isolation boundary and currently has concrete scoped-access correctness concerns.

Concerns:

- **[high] Agent ID normalization can break memory isolation:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads and deletes filter on a stripped value. Unexpected IDs can become unrecoverable or collide with another normalized ID, undermining the isolation guarantee this feature is meant to provide.
  - confidence: 0.9
- **[medium] Migration failures can silently disable scoping:** `extensions/memory-lancedb/index.ts:119`
  - body: The best-effort `agentId` column migration swallows errors without logging, so a failed schema update can leave existing tables unscoped while operators receive no warning.
  - confidence: 0.82
