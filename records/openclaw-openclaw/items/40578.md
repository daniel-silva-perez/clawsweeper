## Summary

Keep open. Current main still has the shared memory-lancedb storage model, so the requested per-agent isolation is not implemented; however, this draft PR is not ready to merge because the proposed patch has correctness and security issues around the active hook lifecycle and agent ID scoping contract.

## What This Changes

The PR adds an opt-in `agentScoping` setting to `memory-lancedb`, adds an `agentId` LanceDB column/migration, and attempts to scope memory recall, store, forget, and auto hooks by the calling agent.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation that scopes the active `before_prompt_build` recall path plus tool and auto-capture paths, uses one validated or encoded owner key consistently for writes/reads/deletes, preserves legacy shared rows deliberately, logs migration failures, documents the config, and adds two-agent isolation regression coverage.

## Reproduction Assessment

Yes. Current main can be checked statically: one shared `MemoryDB`, no `agentId` schema/config, and shared `db.search` calls mean two agents using memory-lancedb still read and write the same pool.

## Solution Assessment

No. This PR is directionally relevant, but the current patch is not the best mergeable solution because it scopes a stale hook path and lets raw and filtered agent IDs diverge across scoped writes, reads, and deletes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate agent IDs before scoped memory access:** `extensions/memory-lancedb/index.ts:425-455`
  - body: When `agentScoping` is enabled, the patch stores raw `ctx.agentId` values but reads and deletes with a filtered value. IDs containing characters such as spaces or slashes can become unrecoverable or collapse into another scope, so validate/reject or encode once and use that value everywhere.
  - confidence: 0.92
- **[P2] Port scoping to the prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-601`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build` and tests assert the plugin no longer registers `before_agent_start`. This patch scopes the legacy hook path, so the active automatic recall path would remain unscoped after rebase.
  - confidence: 0.9
- **[P3] Unwrap factory tools in live coverage:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch changes registered memory tools into factories, but the live smoke path still treats registered tools as objects and calls `.execute` directly. Instantiate factory tools with a context before invoking them or the live store/recall/forget check will throw.
  - confidence: 0.86
- **[P3] Log agentId migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: The migration catch block swallows schema or `addColumns` failures even though scoped reads depend on the new column. Surface a warning so operators can diagnose why enabled scoping is not isolating existing tables.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes a memory isolation boundary and currently lets scoped writes use raw agent IDs while reads/deletes use filtered agent IDs.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads and deletes filter a different value. Unexpected IDs can become unrecoverable or collide with another normalized ID, undermining the isolation guarantee this feature is meant to provide.
  - confidence: 0.9
