## Summary

Keep open. Current main still does not implement per-agent LanceDB memory isolation, so this PR is not obsolete, but the draft branch is stale against the current prompt-build recall path and has blocking scoped-data correctness/security defects before it can be merged.

## What This Changes

The PR adds an opt-in memory-lancedb `agentScoping` setting, an `agentId` LanceDB column/migration, and context-aware tool/hook filters for per-agent memory recall, store, and forget operations.

## Best Possible Solution

Keep the contribution as useful source work, but require a maintainer-guided rebase/design pass that scopes the current prompt-build and tool paths, validates or encodes agent IDs consistently before every read/write/delete, preserves legacy shared rows deliberately, logs migration failures, and adds regression coverage for agent A/B isolation, duplicate detection, forget scoping, default unscoped behavior, and live tool factories.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate agent IDs before scoped memory access:** `extensions/memory-lancedb/index.ts:425`
  - body: When `agentScoping` is enabled, the patch stores `ctx.agentId` raw but strips disallowed characters before search/delete filters. IDs such as `a/b` and `ab` can collapse, and IDs containing disallowed characters can store rows that are never recalled. Validate/reject or encode the ID once and use the same value for writes, reads, and deletes.
  - confidence: 0.93
- **[P2] Port scoping to the prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588`
  - body: Current main moved LanceDB auto-recall to `before_prompt_build`, and tests now assert the legacy `before_agent_start` hook is not registered. Rebasing this patch as-is would leave the automatic recall path unscoped or reintroduce the stale hook, so apply the agent filter to the current prompt-build hook instead.
  - confidence: 0.88
- **[P3] Unwrap factory tools in live coverage:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch converts registered memory tools to factories, but the live memory-lancedb smoke still treats `registeredTools[i].tool` as an object with `.execute`. Invoke the factory with a tool context before calling `.execute`, or the live store/recall/forget test will fail before exercising the feature.
  - confidence: 0.86
- **[P3] Log agentId migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: The migration catch block swallows `schema()` or `addColumns()` failures even though scoped reads depend on the new column. Surface a warning through plugin logging so operators can diagnose why enabled scoping is not isolating existing tables.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff changes memory data isolation and currently lets raw and filtered agent IDs diverge across scoped writes, reads, and deletes.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store the raw `ctx.agentId`, while scoped reads/deletes filter with a stripped version of that ID. Different IDs can collapse to the same filter value, and malformed IDs can make an agent’s own stored rows unreachable, undermining the isolation boundary.
  - confidence: 0.9
