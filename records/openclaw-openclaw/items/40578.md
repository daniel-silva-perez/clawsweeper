## Summary

Keep open. Current main still lacks per-agent isolation for memory-lancedb, but this draft PR is not merge-ready because it is stale against the active hook path and introduces inconsistent agent-ID handling across scoped writes, reads, and deletes.

## What This Changes

The PR adds an optional `agentScoping` setting to memory-lancedb, adds an `agentId` LanceDB column/migration/filtering path, and converts memory tools and hooks toward agent-context-aware scoping.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation that scopes `memory_recall`, `memory_store`, `memory_forget`, `agent_end`, and the active `before_prompt_build` recall path; uses one validated or encoded owner key consistently for write/read/delete; preserves legacy shared rows deliberately; logs migration failures; documents the config; and adds two-agent isolation regression coverage.

## Reproduction Assessment

Yes. Current main can be checked statically: one shared `MemoryDB`, no `agentId` schema/config, and unscoped `db.search`/`db.store` calls mean two agents using memory-lancedb still share the same memory pool; the PR-specific key mismatch also reproduces with an agent ID such as `agent/123`.

## Solution Assessment

No. The PR is directionally useful, but the current patch is not the best mergeable solution because it targets a legacy hook path on its branch and uses different agent-ID values for scoped storage and scoped filters; a rebase with an explicit storage and agent-ID contract is safer.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use one validated agent key for scoped memories:** `extensions/memory-lancedb/index.ts:425-452`
  - body: When `agentScoping` is enabled, the patch stores raw `ctx.agentId` values but reads and deletes with a stripped value. IDs containing characters such as `/` or spaces can become unrecoverable or collapse into another scope, so validate/reject or encode once and use that same key for writes, reads, and deletes.
  - confidence: 0.92
- **[P2] Scope the active prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-596`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, and the tests assert `before_agent_start` is no longer registered. This patch scopes the legacy hook path, so automatic recall would remain unscoped after rebase.
  - confidence: 0.9
- **[P3] Instantiate factory tools in live coverage:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch registers memory tools as factories, but the live smoke path still treats registered tools as objects and calls `.execute` directly. Unwrap factory tools with a context before invoking them or live store/recall/forget coverage will throw.
  - confidence: 0.86
- **[P3] Log agent ID migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: `migrateAgentIdColumn` catches all schema or `addColumns` failures without warning. If migration fails, operators can enable scoping while existing tables remain effectively unscoped with no diagnostic, so surface a warning through the plugin logger.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes a memory isolation boundary and needs attention because scoped writes and scoped reads/deletes can use different agent IDs; no new dependency, workflow, or package supply-chain surface was added.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads and deletes filter on a stripped value. Unexpected IDs can become unrecoverable or collide with another normalized ID, undermining the isolation guarantee this feature is meant to provide.
  - confidence: 0.9
- **[medium] Migration failures hide disabled isolation:** `extensions/memory-lancedb/index.ts:119`
  - body: The best-effort `agentId` column migration swallows errors without logging, so a failed schema update can leave existing tables unscoped while operators receive no warning.
  - confidence: 0.82
