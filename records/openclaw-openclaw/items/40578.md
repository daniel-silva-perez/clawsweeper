## Summary

This PR should stay open rather than be closed: current main still lacks per-agent memory-lancedb isolation, but the draft patch is not merge-ready because it is stale against the current hook path and has a concrete agentId normalization bug in the scoped data boundary.

## What This Changes

The PR adds an opt-in `agentScoping` setting to memory-lancedb, stores an `agentId` column, scopes recall/store/forget operations by agent plus legacy global rows, updates plugin metadata, and adjusts tests for tool factories.

## Best Possible Solution

Keep this PR as useful source work, but require a rebase and a tighter memory-lancedb design: apply scoping on `before_prompt_build`, validate or encode agent IDs consistently before every read/write/delete, preserve legacy global rows deliberately, surface migration failures, and add regression coverage for agent A/B isolation, default unscoped behavior, duplicate detection, forget scoping, and live tool factories.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate agent IDs before scoped memory access:** `extensions/memory-lancedb/index.ts:425`
  - body: When `agentScoping` is enabled, the patch stores `ctx.agentId` raw but strips disallowed characters before search/delete filters. IDs such as `a/b` and `ab` can collapse, letting one agent match another agent's rows while the original agent loses access to its own memories. Validate/reject or encode the ID once and use the same value for writes, reads, and deletes.
  - confidence: 0.93
- **[P2] Port auto-recall scoping to before_prompt_build:** `extensions/memory-lancedb/index.ts:588`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build`, but this patch scopes the legacy `before_agent_start` hook. After rebasing, the automatic recall path would remain unscoped unless the agent filter is applied to the current prompt-build hook.
  - confidence: 0.86
- **[P3] Keep live tests compatible with tool factories:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch converts registered memory tools to factories, but the live memory-lancedb smoke path still expects each registered tool to be an object with `.execute`. Unwrap factory tools in live coverage as well, or the live store/recall/forget test will fail before exercising the feature.
  - confidence: 0.84
- **[P3] Log agentId migration failures:** `extensions/memory-lancedb/index.ts:119`
  - body: The migration catch block swallows failures while the feature depends on the new `agentId` column. If `addColumns` fails, operators get no signal explaining why scoped recall is broken or unavailable. Thread a logger or diagnostic result through this path.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch changes memory data isolation and currently lets raw and sanitized agent IDs diverge across scoped writes and reads.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store the raw `ctx.agentId`, while scoped reads/deletes filter with a stripped version of that ID. Different IDs can collapse to the same filter value, causing cross-agent recall/delete matches or making an agent's own stored rows unreachable.
  - confidence: 0.9
