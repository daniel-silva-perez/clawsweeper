## Summary

Keep this draft PR open: current main still does not implement per-agent LanceDB memory isolation, but the proposed patch is not merge-ready because it is based on the legacy auto-recall hook path and introduces an inconsistent agentId isolation boundary.

## What This Changes

The PR adds opt-in per-agent scoping to the memory-lancedb plugin by adding an agentScoping config flag, an agentId LanceDB column/migration, scoped store/search/delete behavior, and tool factory registration.

## Best Possible Solution

Land a rebased implementation that scopes the active before_prompt_build auto-recall path plus all tool and auto-capture paths, uses one validated or encoded agent owner key consistently for writes, reads, and deletes, documents the new config, preserves legacy shared rows deliberately, logs migration failures, and adds two-agent isolation regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate agent IDs before scoped memory access:** `extensions/memory-lancedb/index.ts:425-455`
  - body: When agentScoping is enabled, the patch stores ctx.agentId raw but strips characters before search/delete filters. IDs such as agent/123 would be written under one value and searched under another, making rows unrecoverable or colliding with another agent scope. Validate/reject or encode once and reuse the same value everywhere.
  - confidence: 0.92
- **[P2] Port scoping to the prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-601`
  - body: Current main moved memory-lancedb auto-recall to before_prompt_build and tests forbid before_agent_start in bundled plugins. This patch scopes the legacy hook, so a rebase would leave the active automatic recall path unscoped unless the new logic is ported.
  - confidence: 0.9
- **[P3] Unwrap factory tools in live coverage:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch changes registered memory tools into factories, but the live smoke still reads registeredTools[i].tool and calls .execute directly. Instantiate factory tools with a context before invoking them or the live store/recall/forget test will throw immediately.
  - confidence: 0.86
- **[P3] Log agentId migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: The migration catch block swallows schema/addColumns failures even though scoped reads depend on the new column. Surface a warning so operators can diagnose why enabled scoping is not isolating existing tables.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes a memory data-isolation boundary and currently lets raw and filtered agent IDs diverge across scoped writes, reads, and deletes.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw ctx.agentId, while scoped reads/deletes filter using a stripped value. Unexpected IDs can become unrecoverable or collapse into another agent's allowed ID, undermining the isolation boundary this PR is meant to add.
  - confidence: 0.9
