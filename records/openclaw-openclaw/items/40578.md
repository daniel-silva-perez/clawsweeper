## Summary

Keep open. Current main still has one unscoped memory-lancedb table and no `agentScoping` config, so the requested per-agent memory isolation is not implemented; this draft PR remains relevant but is not ready to land because the branch is stale against current hook/config behavior and has a concrete agentId normalization issue in the proposed data isolation boundary.

## What This Changes

The PR adds an opt-in `agentScoping` setting to memory-lancedb, adds an `agentId` column, filters recall/store/forget paths by the calling agent plus global rows, and updates plugin metadata and tests.

## Best Possible Solution

Keep this PR open as a source candidate, but require a rebase and a tighter design inside `extensions/memory-lancedb`: use the current `before_prompt_build` hook, validate or parameterize agent IDs consistently before both reads and writes, preserve legacy global rows intentionally, and add focused regression coverage for agent A/B isolation, global legacy visibility, duplicate detection, forget/delete scoping, and default unscoped behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate agent IDs before filtering scoped memories:** `extensions/memory-lancedb/index.ts:425`
  - body: When `agentScoping` is enabled, the patch stores `ctx.agentId` raw but strips characters from the ID before search/delete filters. An ID like `a/b` is stored as `a/b` but later queried as `ab`, so it cannot recall its own rows and can match another agent's `ab` rows. Validate once and reject or safely encode unsupported IDs before every read and write.
  - confidence: 0.91
- **[P2] Move scoped auto-recall onto the current hook:** `extensions/memory-lancedb/index.ts:588`
  - body: The PR scopes auto-recall in the legacy `before_agent_start` hook, but current main has moved memory-lancedb recall to `before_prompt_build`. After rebasing, the scoped filter needs to be applied to the current prompt-build path or automatic recall will remain unscoped.
  - confidence: 0.84
- **[P3] Unwrap tool factories in live coverage:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch changes registered memory tools from static objects to factories, but the live smoke path still calls `registeredTools[i].tool.execute(...)` directly. With the new factory shape that value is a function, so the live memory-lancedb test will throw before exercising store/recall/forget.
  - confidence: 0.82
- **[P3] Surface agentId migration failures:** `extensions/memory-lancedb/index.ts:119`
  - body: The best-effort migration swallows every error when adding the `agentId` column. If an installed LanceDB version cannot add the column, operators can enable scoping and still get unscoped behavior with no warning, which makes the isolation failure hard to diagnose.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes memory data access scope and currently has an agentId normalization mismatch that can cross-match another agent's scoped rows.

Concerns:

- **[high] Agent ID normalization can cross scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped reads/deletes strip disallowed characters from `ctx.agentId`, while writes store the raw value. Distinct IDs such as `a/b` and `ab` can collapse in filters, letting one scoped agent recall or delete another agent's rows and losing access to its own stored rows.
  - confidence: 0.9
