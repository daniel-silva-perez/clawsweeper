## Summary

Keep open: current main still lacks per-agent LanceDB memory isolation, so the PR is not obsolete, but the proposed patch is not mergeable because it misses the active prompt-build recall path and has an agent ID consistency problem in the scoped isolation boundary.

## What This Changes

The PR adds an optional `agentScoping` setting to `memory-lancedb`, adds an `agentId` LanceDB column/migration, converts memory tools to factories, and filters scoped memory reads/writes/deletes by the calling agent context.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation that scopes the active `before_prompt_build` auto-recall path plus tool and auto-capture paths, uses one validated or encoded owner key consistently for writes/reads/deletes, deliberately preserves legacy shared rows, logs migration failures, documents the config, and adds two-agent isolation regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate agent IDs before scoped memory access:** `extensions/memory-lancedb/index.ts:425-455`
  - body: When `agentScoping` is enabled, the patch stores `ctx.agentId` raw but reads and deletes with a stripped value. IDs such as `agent/123` would be written under one value and queried under another, making rows unrecoverable or potentially collapsing scopes; validate/reject or encode once and reuse that value everywhere.
  - confidence: 0.92
- **[P2] Port scoping to the prompt-build recall hook:** `extensions/memory-lancedb/index.ts:588-601`
  - body: Current main moved memory-lancedb auto-recall to `before_prompt_build` and tests assert this bundled plugin no longer registers `before_agent_start`. This patch scopes the legacy hook, so a rebase would leave the active automatic recall path unscoped unless the logic is ported.
  - confidence: 0.9
- **[P3] Unwrap factory tools in live coverage:** `extensions/memory-lancedb/index.ts:351`
  - body: The patch changes registered memory tools into factories, but the live smoke still treats `registeredTools[i].tool` as an object and calls `.execute` directly. Instantiate factory tools with a context before invoking them or the live store/recall/forget test will throw.
  - confidence: 0.86
- **[P3] Log agentId migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: The migration catch block swallows schema/addColumns failures even though scoped reads depend on the new column. Surface a warning so operators can diagnose why enabled scoping is not isolating existing tables.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes a memory isolation boundary and currently lets raw and filtered agent IDs diverge across scoped writes, reads, and deletes.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:425`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads/deletes filter using a stripped value. Unexpected IDs can become unrecoverable or collapse into another allowed ID, undermining the isolation boundary this PR is meant to add.
