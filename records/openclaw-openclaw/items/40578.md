## Summary

Keep this PR open. Current main still lacks per-agent scoping for memory-lancedb, and this PR remains a plausible implementation candidate, but it is not merge-ready because the proposed data-scope change has unresolved correctness and observability/security review concerns.

## Best Possible Solution

Keep this PR open as an implementation candidate for bundled memory-lancedb per-agent isolation. The maintainer path is to rebase onto current main, keep the change inside extensions/memory-lancedb, consistently use trusted ctx.agentId from tool factories and hook contexts, resolve LanceDB filter/query ordering and agentId validation semantics, surface migration failures, and add regression coverage for agent A/B isolation, legacy shared memories, duplicate detection, forget/delete scoping, and default unscoped behavior. Then either land it or intentionally supersede it with a canonical replacement PR.

## Risks / Open Questions

- Current main still allows memory-lancedb auto-recall, auto-capture, explicit recall, store, forget, and duplicate detection to operate on one shared pool across agents.
- The proposed PR changes a data isolation boundary and interpolates ctx.agentId into LanceDB SQL filters; review comments flag remaining correctness issues around sanitization consistency and filtered recall quality.
- The proposed migration catch in the patch is best-effort and silent, so an incompatible LanceDB table could leave scoping ineffective without an operator-visible warning.
