## Summary

Keep open. Current main still accepts a literal `secretref:/...` QQ Bot `clientSecret` as a normal non-empty string and can pass that value to the QQ token endpoint; the related fix PR #74772 is open but not merged and still needs review for the literal-marker policy.

## What This Changes

The issue reports that QQ Bot `clientSecret` values using `secretref:/...` stopped resolving after an upgrade and can be sent to QQ as the literal secret marker.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, docs surface updates, and an explicit policy for literal legacy markers so raw SecretRef-looking strings cannot be sent to QQ.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current main treats that non-empty string as available plaintext and can post it as the QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this issue open is the right path because current main has not implemented the QQ Bot SecretRef contract/runtime fix; the narrow maintainable solution belongs in the QQ Bot plugin plus the shared SecretRef validation and credential-surface policy.
