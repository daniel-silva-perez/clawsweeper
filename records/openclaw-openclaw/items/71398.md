## Summary

Keep open. Current main still has the reported QQ Bot SecretRef gap: a literal `secretref:/...` value is accepted as a normal configured string, QQ Bot resolves `clientSecret` through that shared string path, and the raw value can reach the QQ access-token request; the linked fix PR #74772 is still the right implementation path rather than a reason to close this issue before merge.

## What This Changes

Resolve QQ Bot `clientSecret` SecretRefs after upgrade so `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret` do not send raw SecretRef-looking markers to the QQ token endpoint.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, credential-surface docs, and an explicit policy for raw SecretRef-looking marker strings so they cannot be sent to QQ.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current main treats that as an available plaintext string and can post it as QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this issue open is the best current action because current main has not landed the QQ Bot SecretRef contract/runtime fix, and the narrow maintainable implementation belongs in the QQ Bot plugin plus shared SecretRef policy coverage.
