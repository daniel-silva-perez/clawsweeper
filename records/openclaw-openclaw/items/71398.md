## Summary

Keep open. Current main still accepts a literal `secretref:/...` QQ Bot `clientSecret` as a normal non-empty string, can pass that value to the QQ token endpoint, and still lacks QQ Bot channel secret-contract coverage; the focused fix PR #74772 is open but not merged.

## What This Changes

The issue requests fixing QQ Bot `clientSecret` SecretRef handling after upgrade so `secretref:/...` values are resolved or rejected instead of being sent raw to QQ.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds secret-contract targets and runtime materialization for top-level and per-account `clientSecret`, updates the generated SecretRef surface/docs, and adds an explicit validation or migration policy so raw legacy markers cannot reach QQ.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current main treats that as an available string and can post it as the QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this issue open is the right path because the current implementation is incomplete and the open focused PR is the narrowest maintainable route; a parallel ClawSweeper repair job would duplicate that work.
