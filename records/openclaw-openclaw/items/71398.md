## Summary

Keep open. Current main still has the QQ Bot SecretRef gap: `secretref:/...` can be treated as a non-empty plaintext `clientSecret`, QQ Bot has no channel secret-contract artifact, and open PR #74772 targets the fix but has not landed.

## What This Changes

The issue requests fixing QQ Bot `clientSecret` handling so SecretRef-backed secrets are materialized instead of sending raw SecretRef markers to QQ OpenAPI.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds channel secret-contract/runtime materialization for top-level and per-account `clientSecret`, updates generated SecretRef credential-surface artifacts, adds regression tests, and prevents raw `secretref:/...` markers from reaching QQ as secrets.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.*.clientSecret` as `secretref:/QQBOT_...`; current main treats that as a non-empty string and can pass it into the QQ token request body.

## Solution Assessment

Yes for the direction, but no implemented solution is present on main. The narrow maintainable fix belongs in the QQ Bot plugin plus the shared SecretRef contract/docs surface, with explicit handling for literal `secretref:/...` values.
