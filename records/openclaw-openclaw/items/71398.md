## Summary

Keep open. Current main still accepts a literal `secretref:/...` QQ Bot `clientSecret` as a non-empty string, can pass that value to the QQ token endpoint, and still lacks QQ Bot channel secret-contract coverage in the canonical SecretRef surface; the focused fix PR #74772 is open but not merged.

## What This Changes

The issue requests fixing QQ Bot `clientSecret` handling so `secretref:/...` configuration does not get passed raw to the QQ access-token endpoint after upgrading.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds the channel secret contract, runtime assignment/materialization for top-level and account `clientSecret`, generated SecretRef surface docs, and an explicit legacy-marker validation or migration policy so raw markers cannot reach QQ.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to set `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` to `secretref:/QQBOT_...`; current main treats that as a configured string and can post it as the QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this issue open is the right path because current main has not implemented the necessary QQ Bot SecretRef contract/runtime fix; the narrow maintainable solution belongs in the QQ Bot plugin plus the shared SecretRef surface.
