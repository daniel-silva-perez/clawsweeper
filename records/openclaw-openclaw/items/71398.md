## Summary

Keep open: current main and the latest release still do not provide QQ Bot channel SecretRef runtime materialization, while QQ Bot docs say `clientSecret` accepts SecretRef input. A literal `secretref:/...` remains a non-empty string to the QQ Bot resolver and can still reach the QQ token request as the client secret.

## What This Changes

The issue requests that QQ Bot resolve SecretRef-backed `clientSecret` values instead of sending `secretref:/...` to QQ OpenAPI after upgrading to v2026.4.23.

## Best Possible Solution

Add a QQ Bot channel secret contract that covers `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, materialize supported structured SecretRefs before QQ Bot runtime starts, update the canonical SecretRef surface/docs, and make the literal `secretref:/...` case either a precise migration path or a validation/doctor error so it cannot be sent to QQ as a secret.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure QQ Bot `clientSecret` as `secretref:/QQBOT_...`; current main treats that as a normal non-empty string and the token manager posts it directly as `clientSecret`.

## Solution Assessment

No current solution is in place. The narrow maintainable fix is in the QQ Bot plugin plus the shared channel SecretRef contract/validation surface, not in the QQ token request itself.
