## Summary

Keep open. Current main and the latest release still let QQ Bot treat a literal `secretref:/...` `clientSecret` as an ordinary string, and the focused open PR #74772 is the right path to resolve the QQ Bot SecretRef contract gap after review and validation.

## What This Changes

The issue requests fixing QQ Bot `clientSecret` resolution so SecretRef-backed secrets are materialized instead of sending the raw marker to QQ OpenAPI.

## Best Possible Solution

Review and land PR #74772, or an equivalent QQ Bot-owned fix, after adding QQ Bot secret contract/runtime materialization, generated docs matrix coverage, regression tests, and explicit handling for literal `secretref:/...` markers so raw markers cannot be sent as QQ `clientSecret` values.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure QQ Bot `clientSecret` as `secretref:/QQBOT_...`; current main treats it as a non-empty string and passes that value into the token request body.

## Solution Assessment

Yes for the current direction, but it is not solved on main yet. The narrow maintainable fix belongs in the QQ Bot plugin plus the shared SecretRef contract/validation surface, not a broad core special case.
