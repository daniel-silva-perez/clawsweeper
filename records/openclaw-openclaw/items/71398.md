## Summary

Keep open. Current main still leaves the reported QQ Bot SecretRef path unresolved: `secretref:/...` is accepted as a plain configured string, QQ Bot can pass that value to the QQ token endpoint, the canonical SecretRef surface still omits QQ Bot `clientSecret` targets, and the focused fix PR #74772 is still an open/unmerged implementation path.

## What This Changes

The issue reports that QQ Bot `clientSecret` values using `secretref:/...` stopped resolving after upgrade and are sent to QQ as invalid credentials.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds the channel secret contract, runtime assignment/materialization coverage, generated SecretRef credential-surface docs, and explicit validation or migration policy so raw `secretref:/...` cannot reach QQ as `clientSecret`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current main accepts it as a non-empty string and can post that exact value to the QQ token endpoint.

## Solution Assessment

Yes. The best path is to keep this issue open and finish the focused QQ Bot SecretRef contract/runtime fix in PR #74772 or an equivalent branch; starting a parallel repair job would duplicate the active implementation lane.
