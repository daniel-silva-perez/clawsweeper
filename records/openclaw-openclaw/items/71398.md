## Summary

Keep open. Current main and the latest release still let QQ Bot treat a literal `secretref:/...` `clientSecret` as a normal string, QQ Bot still lacks channel SecretRef contract coverage, and an open focused PR (#74772) is the right implementation path but has not landed.

## What This Changes

The issue requests that QQ Bot resolve SecretRef-backed `clientSecret` values instead of sending `secretref:/...` to QQ OpenAPI after upgrading to v2026.4.23.

## Best Possible Solution

Review and land PR #74772, or an equivalent QQ Bot-owned fix, after aligning the generated SecretRef credential matrix/docs and the literal `secretref:/...` policy so the raw marker cannot be sent as a QQ `clientSecret`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure QQ Bot `clientSecret` as `secretref:/QQBOT_...`; current main treats it as a non-empty string and the token manager posts that value as `clientSecret`.

## Solution Assessment

No current solution is on main. The best direction is the focused QQ Bot plugin plus shared SecretRef contract/validation path represented by PR #74772, with generated docs matrix and legacy-marker policy completed before merge.
