## Summary

Keep open. Current main still routes gateway-executed plugin message sends through an explicit 10,000 ms timeout, the CLI media send path has no timeout override, and the supplied context shows the closing PR remains open rather than merged.

## What This Changes

The report requests fixing a regression where `openclaw message send --channel wecom --media <about-10MB-file>` times out after 10,000 ms instead of completing the file send.

## Best Possible Solution

Keep this issue open until #74169 is revised, landed, or replaced with a targeted generic fix for gateway-executed plugin message sends. The fix should cover the explicit message-action timeout path, especially media sends, and add focused regression coverage for CLI gateway plugin sends that take longer than 10 seconds.

## Reproduction Assessment

Yes for the core timeout boundary. The provided CLI command plus current code path shows a gateway-executed WeCom plugin send waits on `message.action` under a 10,000 ms caller timeout; a delayed gateway plugin action or the reporter’s 10 MB upload would exercise the same failure path.

## Solution Assessment

No for the described open PR if it only changes the generic gateway-call default. The safer fix is to adjust the explicit message-action/media-send timeout boundary in the runner or an intentional config/CLI seam, then prove that gateway-executed plugin sends no longer inherit the 10-second limit.
