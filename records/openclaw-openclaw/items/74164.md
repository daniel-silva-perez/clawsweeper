## Summary

Keep this open. Current main still has a generic gateway-executed plugin message-action path with an explicit 10,000 ms timeout, the CLI media-send path does not expose or forward a longer timeout, and the linked closing PR remains open rather than merged.

## What This Changes

The issue requests that `openclaw message send --channel wecom --media <large-file>` complete successfully instead of failing with `gateway timeout after 10000ms`.

## Best Possible Solution

Keep the issue open and resolve it with a generic core fix for gateway-executed plugin message actions, likely by making the message-action/media-send timeout longer or configurable at the correct boundary, with focused regression coverage for CLI media sends through plugin-owned channels.

## Reproduction Assessment

Yes for the core timeout boundary: a gateway-owned channel plugin `send` action that takes longer than 10 seconds will hit the same explicit caller-side timeout. The reporter's WeCom command provides a plausible live reproduction, but the core bug can be verified from the dispatch path without WeCom credentials.

## Solution Assessment

No for the currently linked PR as described: raising the generic gateway default alone is unlikely to fix this path because `message-action-runner` passes an explicit 10,000 ms timeout. The safer fix is targeted at gateway-executed message actions or media sends, with tests proving the explicit timeout boundary changed.
