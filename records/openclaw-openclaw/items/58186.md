Keep open. Current main still lacks hook mapping systemPrompt support, and the branch remains useful, but the latest PR diff is stale against the moved hook handler and current cron executor paths, so it would not reliably deliver Gmail system instructions after rebase. ## What This Changes

The PR adds optional literal `hooks.mappings[].systemPrompt` support, a default Gmail preset system prompt, hook-to-cron payload plumbing, config/docs metadata, tests, and a changelog entry.

## Best Possible Solution

Land a rebased repair that adds literal-only hook mapping `systemPrompt` end to end through the current config schema, Gmail preset, current mapped hook handler, cron job payload, current isolated cron executor, current public docs page, changelog credit, and focused tests while keeping webhook-derived email content in the user-message/external-content path.

## Reproduction Assessment

Yes. The source path is enough to reproduce the missing behavior: current main has no Gmail preset `systemPrompt`, no mapped dispatch field, no cron payload field, and no `extraSystemPrompt` forwarding in the current executor. A mapped Gmail hook test that expects a literal systemPrompt in the cron payload would fail until those current-main surfaces are wired.

## Solution Assessment

No, not as currently rebased. The literal-only systemPrompt direction is the right security boundary, but this PR is patching stale files and a redirect docs page; the maintainable fix is to port the same behavior to `src/gateway/server/hooks-request-handler.ts`, `src/cron/isolated-agent/run-executor.ts`, and the current docs/config-baseline surfaces.
