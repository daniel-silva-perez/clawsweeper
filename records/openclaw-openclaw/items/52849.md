## Summary

Keep open. Current main has a trusted bundled pre-model tool-result middleware path, but it still does not expose the requested public typed hook for external plugins. The existing public hook catalog has `before_tool_call`, observational `after_tool_call`, and sync persistence hooks; the requested same-turn external plugin seam remains missing.

## Best Possible Solution

Keep this open. Maintainers should either design a public, documented, versioned plugin hook for pre-model tool-result mutation, or define a trust-gated external policy for `registerAgentToolResultMiddleware`. The design should align hook types, SDK docs, registry enforcement, Pi and Codex runtime paths, failure semantics, bounds, provenance metadata, and contract tests; #62123 is useful related context if maintainers want one canonical API design thread.

## Risks / Open Questions

- Opening this seam to external plugins is security-sensitive because it can rewrite high-trust tool output before the model sees it; maintainers need explicit trust policy, fail-open/fail-closed behavior, bounds, and audit/provenance semantics.
- There are two open related design threads, this issue and #62123. Without consolidation, API design could fragment between same-turn model protection and transcript-write wording.
