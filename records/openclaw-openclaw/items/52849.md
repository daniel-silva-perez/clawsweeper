## Summary

Keep open. Current main has a trusted/bundled pre-model tool-result middleware path, but it does not expose the requested public typed hook for external plugins. The ordinary plugin hook catalog still has `before_tool_call`, observational `after_tool_call`, and sync persistence-time hooks; `before_tool_result_emit` and its synonym searches are absent.

## Best Possible Solution

Keep the issue open and design a public, documented, versioned plugin contract for pre-model tool-result mutation, or explicitly decide on a trust-gated external policy for `registerAgentToolResultMiddleware`. The implementation should align hook types, SDK docs, registry enforcement, Pi and Codex runtime paths, failure semantics, and contract tests; #62123 is useful related context if maintainers choose a single canonical API design thread.

## Risks / Open Questions

- Opening this seam to external plugins is security-sensitive because it can rewrite high-trust tool output before the model sees it; maintainers should define trust policy, fail-open/fail-closed behavior, result bounds, and Pi/Codex parity before implementation.
- The existing bundled tokenjuice-style middleware may look like an implementation at first glance, but docs and registry enforcement show it does not solve the external plugin hook request.
