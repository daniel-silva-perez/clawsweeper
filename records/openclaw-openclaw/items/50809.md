## Summary

Keep open. Current main still lacks the requested Android `sms.read` invoke command and the Android-node-mediated SMS channel ingress path. The linked implementation PR is closed unmerged, and the remaining work crosses Android SMS permissions, node command policy, gateway event routing, channel registration, and sender allowlist/privacy decisions, so it needs maintainer product/security review before implementation.

## What This Changes

The issue requests adding an Android `sms.read` invoke command plus an allowlist-gated SMS messaging channel that ingresses Android SMS events and replies through `sms.send`.

## Best Possible Solution

Keep the issue open and split the design into maintainer-approved pieces: first define a narrow, security-reviewed Android `sms.read` command contract, then separately decide the SMS ingress channel boundary and any generic plugin/node-event API needed before routing texts to agents.

## Risks / Open Questions

- Inbound SMS routing would expose private phone messages to agents and outbound replies through a paired Android device, so Android permissions, sender normalization, allowlist semantics, logging, and audit/privacy posture need explicit maintainer review.
- The issue bundles a narrow Android read command with a broader channel product decision; implementing both in one change may cross too many ownership and security boundaries.
- It remains a product decision whether Android-node SMS ingress should be bundled core behavior, plugin-backed behavior with a new generic node-event seam, or left outside OpenClaw core.
