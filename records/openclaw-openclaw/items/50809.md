## Summary

Keep open. Current main still lacks both requested pieces: Android exposes `sms.send` and `sms.search` only, and the gateway has no Android-node SMS ingress event or SMS channel registration. The linked implementation PR #50808 is closed unmerged, and the request crosses private SMS access, inbound agent-trigger authority, outbound SMS sending, allowlist semantics, Android permissions, and channel/plugin boundaries, so it needs maintainer product and security review rather than automatic cleanup.

## What This Changes

The issue requests an Android `sms.read` invoke command for message/thread reads and an allowlist-gated SMS messaging channel mediated by an Android node with inbound `sms.received` events and outbound replies via `sms.send`.

## Best Possible Solution

Keep the issue open and split the work into approved design pieces: first define a narrow Android `sms.read` command contract, then separately decide the SMS ingress channel boundary and implement it with explicit permission UX, allowlist normalization, node-event policy, reply controls, docs, and targeted tests.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence absence check. Current-main source inspection and exact searches show no `sms.read` enum/spec/handler, no `RECEIVE_SMS`/`SMS_RECEIVED` receiver, no `sms.allowFrom`, and no gateway `sms.received` event case.

## Solution Assessment

No, the combined issue request is not yet the best implementation path for an automated fix. The maintainable path is to split command-read behavior from inbound SMS channel routing and get maintainer security/product approval before code changes.
