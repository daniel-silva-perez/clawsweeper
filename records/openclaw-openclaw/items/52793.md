## Summary

Keep open. Current main still exposes `tools.fs.workspaceOnly` as one coarse boolean shared by read/write/edit/apply_patch, with only a separate `tools.exec.applyPatch.workspaceOnly` knob for apply_patch. There is no typed or runtime-validated per-operation fs policy that can allow reads outside the workspace while keeping write/edit/apply_patch workspace-contained.

## Best Possible Solution

Keep this open and implement a backward-compatible fs policy extension in core: preserve the existing boolean as the coarse default, add explicit per-operation policy or read-only external roots, update config types/schema/help/generated docs, thread the richer policy through `src/agents/tool-fs-policy.ts`, `src/agents/pi-tools.ts`, plugin tool context, media/image/pdf local-root handling as needed, and add regression tests for read/search outside workspace while write/edit/apply_patch stay confined.

## Risks / Open Questions

- This is a security-boundary feature: a fix must preserve existing safe defaults and prove that write/edit/apply_patch remain workspace-contained when reads are widened.
- Backward compatibility matters because existing deployments use `tools.fs.workspaceOnly` as a boolean; schema, generated config help, docs, and runtime migration/validation need to accept the old form.
- The issue mentions `search`, but current core fs tooling does not expose a separate filesystem search tool; maintainers should define whether this applies to a future fs search operation, exec-based search, or read-like external roots.
