## Summary

Keep open. The report is recent, detailed, and still maps to current main: external plugin CLI startup still goes through plugin command registration, command narrowing still falls back to full plugin loading when no manifest command owner is found, and the exact recipes/Kitchen implementation is not present in this repository to prove the macOS regression is fixed or non-reproducible.

## What This Changes

The issue reports a severe OpenClaw CLI subprocess startup regression that causes launchd workflow tick fan-out, multi-minute worker runtimes, and downstream Kitchen subprocess cache poisoning on timeout.

## Best Possible Solution

Reproduce on macOS with the external recipes/Kitchen plugins installed, profile the single-call CLI startup path, add or fix narrow command ownership/activation metadata for the recipes command if that is the missing load hint, and harden plugin-facing timeout/JSON handling so partial stdout from a killed subprocess cannot be cached as success. Launchd jitter can be a mitigation after the startup and timeout semantics are understood.

## Reproduction Assessment

Unclear. The issue provides concrete commands, timings, and cache-poisoning observations, but a high-confidence reproduction needs macOS launchd plus the external recipes/Kitchen plugins that are absent from this checkout.

## Solution Assessment

Unclear. There is no proposed patch; the maintainable direction is likely a combined external-plugin startup fix plus a generic core timeout/parse-safety contract rather than closing the report from current source inspection.
