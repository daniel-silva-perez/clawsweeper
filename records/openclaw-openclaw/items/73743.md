## Summary

Keep open. Current main has relevant mitigations for plugin CLI laziness, subprocess timeout metadata, and runtime-deps event-loop pressure, but it does not prove the reported macOS `recipes` CLI startup and launchd fan-out regression is fixed; the named recipes and ClawKitchen implementations are not present in this repository, and raw plugin subprocess callers can still misclassify timeouts unless they check the newer termination metadata.

## What This Changes

The issue asks OpenClaw to investigate and fix slow external plugin CLI startup under macOS launchd fan-out, plus prevent subprocess timeout partial output from poisoning ClawKitchen recipe caches.

## Best Possible Solution

Keep this as an active performance and reliability investigation: profile `openclaw recipes list` and `recipes workflows worker-tick` on macOS under idle and fan-out conditions, move the external recipes CLI path onto descriptor-backed or gateway-backed lightweight execution where feasible, and make subprocess timeout results impossible for plugin callers to cache as successful partial output.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path with exact environment details, idle and fan-out `time openclaw recipes list` comparisons, process snapshots, and cache observations; this checkout could only verify that the relevant core surfaces still exist, not remeasure macOS timing.

## Solution Assessment

Unclear as a single fix. The suggested directions are sound, but the maintainable solution likely spans core plugin CLI startup/timeout semantics plus external recipes/ClawKitchen cache handling, so current main is not enough evidence for closure.
