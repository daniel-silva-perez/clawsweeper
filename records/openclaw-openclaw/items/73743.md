## Summary

Keep open. Current main has relevant mitigations for plugin CLI laziness, runtime-deps startup pressure, and subprocess timeout metadata, but the exact macOS `recipes`/ClawKitchen regression is not proven fixed; part of the named implementation is outside this checkout, and the core timeout result can still be misclassified by callers that only look for a numeric exit code.

## What This Changes

The issue reports slow `openclaw recipes` CLI subprocess startup on macOS, launchd fan-out causing multi-minute workflow ticks, and downstream ClawKitchen cache poisoning after subprocess timeouts.

## Best Possible Solution

Keep this as an active reliability investigation: profile plugin CLI startup on macOS for the external recipes command, move that path toward descriptor-backed or gateway-backed lightweight execution where possible, and harden core subprocess timeout results so partial output cannot be cached as success by plugin callers.

Yes. The issue includes a high-confidence reproduction path with exact environment details, idle and fan-out timing comparisons, process/RSS snapshots, and cache observations; this checkout can verify the relevant core surfaces but cannot remeasure the external recipes/Kitchen flow.
