## Summary

Keep open. Current main does not show a core implementation that resolves the reported startup/fan-out problem: external `recipes`/`ClawKitchen` sources are not present in this repository, unknown plugin CLI commands still pass through plugin command registration/loading, and raw `runCommandWithTimeout` can still surface a timeout with `code: null`, which matches the cache-poisoning shape described by the report.

## What This Changes

The issue reports slow OpenClaw CLI subprocess startup on macOS, launchd workflow tick fan-out, and downstream ClawKitchen subprocess-cache poisoning after timeouts.

## Best Possible Solution

Treat this as a real open performance/reliability investigation: profile current-main plugin CLI startup on macOS with the reported fan-out, move read-only plugin CLI paths toward lightweight metadata or gateway-backed execution where feasible, and split the external recipes/kitchen timeout-cache behavior into the owning plugin package while keeping core subprocess timeout semantics unambiguous.

## Reproduction Assessment

Yes. The issue gives a concrete macOS launchd reproduction path with idle and fan-out `time openclaw recipes list` comparisons, and current source confirms the relevant unknown-plugin CLI registration path still exists.

## Solution Assessment

Unclear as a single fix. The suggested investigation areas are directionally sound, but the maintainable path is likely split between core plugin CLI startup/timeout semantics and the external recipes/ClawKitchen plugin behavior.
