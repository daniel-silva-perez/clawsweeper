## Summary

Keep this issue open. The report is recent and detailed, and current main still has the implicated core/plugin-runtime surfaces: plugin CLI startup still depends on manifest command ownership metadata to narrow loading, plugins still receive the raw timeout runner, and timeout results can still carry partial stdout with a nullable code unless callers use the safer helper correctly. The exact recipes/kitchen implementation is external to this checkout, so this needs maintainer investigation rather than cleanup closure.

## Best Possible Solution

Keep the issue open and investigate on macOS with the reported external plugin set installed. Profile openclaw recipes list and recipes workflows worker-tick through the CLI startup trace, verify the recipes plugin declares command ownership/activation metadata and lightweight cli-metadata descriptors, and harden the plugin-facing subprocess API or documented helper path so timeouts and invalid JSON cannot be cached as successful output. Treat launchd jitter as a secondary mitigation after the single-command startup cost and timeout semantics are understood.
