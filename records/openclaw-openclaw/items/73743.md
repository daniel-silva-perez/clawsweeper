## Summary

Keep this issue open. The report is recent, detailed, and still maps to current core/plugin-runtime surfaces on main: external plugin CLI startup is still metadata-dependent, the exact recipes/kitchen implementation is not bundled here, and the plugin-facing raw timeout runner is still exposed separately from the safer helper. Current main has some nearby CLI startup work after the report, but it does not prove the reported recipes/launchd/kitchen failure is fixed.

## Best Possible Solution

Keep this issue open and investigate with the reported macOS launchd setup and external recipes/kitchen plugins installed. Profile openclaw recipes list and recipes workflows worker-tick with startup tracing, verify the recipes plugin declares command ownership or activation metadata plus lightweight CLI descriptors, and harden the plugin-facing subprocess API or documented helper path so timeouts and invalid JSON cannot be cached as successful output. Treat launchd jitter as a mitigation after the single-command startup cost and timeout semantics are understood.
