## Summary

Keep open. Current main still has the reported config-load path: loading config materializes model defaults, provider defaults resolve bundled provider policy public artifacts, and that provider-policy loader still does not pass the existing runtime-deps opt-out. This remains a narrow, source-confirmed regression for sealed deployments rather than a stale or duplicate cleanup item.

## What This Changes

The issue reports that v2026.4.27 fails during config loading in sealed Docker deployments because provider-policy/default materialization attempts to create `/var/lib/openclaw/plugin-runtime-deps/...` before gateway startup.

## Best Possible Solution

Patch the provider-policy public-artifact path so config validation/default materialization can load lightweight provider policy artifacts without creating or repairing bundled runtime-dependency roots. Runtime startup, plugin activation, doctor, and explicit dependency repair should continue staging runtime deps where runtime code actually needs them.

## Reproduction Assessment

Yes. The issue provides concrete sealed Docker reproduction steps and an exact stack, and current main source confirms that the config-load path still reaches provider-policy public-artifact loading without the available runtime-deps opt-out.

## Solution Assessment

Yes. The narrow maintainable solution is to pass `installRuntimeDeps: false` for provider-policy public artifact loading used by config/default control-plane work and add regression coverage; globally disabling bundled runtime-deps staging would be too broad and would risk real plugin startup behavior.
