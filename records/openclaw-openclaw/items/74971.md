## Summary

Keep open. Current main still routes config loading through model-default materialization, provider-policy public-artifact loading, and bundled public-surface runtime-deps preparation unless the caller explicitly opts out; the provider policy loader still does not pass that opt-out. The existing ClawSweeper comment and reporter follow-up are consistent with the current code path, so this remains a valid narrow regression rather than stale cleanup.

## What This Changes

The issue reports that sealed Docker gateway deployments fail config loading on v2026.4.27 because provider-policy/default materialization tries to create `/var/lib/openclaw/plugin-runtime-deps`.

## Best Possible Solution

Keep the issue open and patch the provider-policy public-artifact control-plane path so config validation/default materialization can load lightweight provider policy artifacts without creating or repairing bundled runtime-dependency roots. Startup, plugin activation, doctor, and explicit dependency repair should continue to stage runtime deps where those runtime paths actually need them.

## Reproduction Assessment

Yes. The issue supplies a concrete sealed Docker reproduction and exact stack, and current main source confirms the same config-load path still reaches provider-policy public-artifact loading without the runtime-deps opt-out.

## Solution Assessment

Yes. The narrow maintainable fix is to make provider-policy public artifact loading pass `installRuntimeDeps: false` for config/default control-plane work and add regression coverage; globally disabling runtime-deps staging would be broader and risk startup/plugin activation behavior.
