## Summary

Keep open. The issue is concrete, recent, and still matches current `main`: the config-load path can resolve bundled provider policy artifacts and the provider policy caller still uses the public-artifact loader without opting out of bundled runtime-dependency staging, so a sealed `/var/lib/openclaw/plugin-runtime-deps` can still fail before normal gateway startup.

## What This Changes

The issue reports that upgrading a sealed Docker Desktop/WSL2 gateway from `v2026.4.23` to `v2026.4.27` makes config load fail with `EACCES` while creating `/var/lib/openclaw/plugin-runtime-deps/...`.

## Best Possible Solution

Keep the issue open and fix the control-plane boundary: config validation/default materialization should load lightweight provider policy artifacts without creating or repairing bundled runtime-dependency roots, while runtime-deps staging remains in startup, activation, doctor, and explicit repair paths.

## Reproduction Assessment

Yes. The report gives concrete deployment constraints and an exact stack, and current `main` source confirms the same path from config load through provider policy public-artifact loading into runtime-deps preparation.

## Solution Assessment

Yes. The narrowest maintainable repair is to pass an explicit runtime-deps opt-out for provider policy public artifacts during config/default loading and add regression coverage proving config materialization does not mkdir or install bundled runtime deps for those lightweight artifacts.
