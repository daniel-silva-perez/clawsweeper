## Summary

Current main still has the reported debounce gap: the shared inbound debounce predicate bypasses only `hasControlCommand()`, while the abort-aware helper already recognizes standalone abort phrases. The issue is valid, narrow, and not blocked by product strategy.

## What This Changes

The issue requests that bare abort phrases such as `stop`, `abort`, and `wait` bypass inbound channel debouncing the same way slash stop commands do.

## Best Possible Solution

Use abort-aware command detection for shared and plugin-local debounce bypasses, preserve the BlueBubbles coalescing exception, and add regression tests proving bare abort phrases flush immediately while ordinary text still debounces.

## Repair Work Prompt

Fix the inbound debounce control-command gap tracked by https://github.com/openclaw/openclaw/issues/48684. Current main still uses `hasControlCommand()` in `src/channels/inbound-debounce-policy.ts`, so bare abort phrases such as `stop`, `abort`, and `wait` are delayed or merged by debounced channels. Update the shared predicate to use abort-aware detection (`isControlCommandMessage()`), and audit bundled plugin-local debounce predicates that call `hasControlCommand()` directly: Feishu, Microsoft Teams, Mattermost, WhatsApp, and BlueBubbles. Preserve the documented BlueBubbles `coalesceSameSenderDms` opt-in exception where same-sender DM command coalescing is intentional. Add focused regression tests proving bare abort phrases bypass debounce immediately while normal text still debounces and `/status` behavior remains unchanged. Treat closed PR #48916 as useful prior source context, but implement against current main. Do not change debounce timing defaults, queue semantics, command authorization policy, or unrelated channel behavior. Changelog is likely user-facing because this fixes channel abort behavior; add a single-line Fixes entry with proper reporter/contributor attribution if the repo’s active changelog section expects it.
