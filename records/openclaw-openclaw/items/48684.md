## Summary

Keep open. Current main still treats standalone abort phrases as debounceable text because the shared inbound debounce predicate calls `hasControlCommand()` instead of the abort-aware helper, and the related fix PR is closed unmerged.

## What This Changes

Bare abort phrases such as `stop`, `abort`, and `wait` should bypass inbound channel debounce immediately, matching slash-command control behavior.

## Best Possible Solution

Make the shared debounce predicate and bundled plugin-local debounce gates use abort-aware control-command detection so standalone abort phrases flush immediately, while preserving normal text/media debounce behavior and the documented BlueBubbles coalescing exception.

## Reproduction Assessment

Yes. A direct source-level check of `shouldDebounceTextInbound({ text: "stop", cfg })` on current main reaches `hasControlCommand("stop") === false` and returns `true`, so the abort phrase is held for debounce instead of flushing immediately.

## Solution Assessment

Yes, with scope adjustment. Replacing the shared predicate with `isControlCommandMessage()` is the narrow core fix, but the maintainable solution should also update plugin-local debounce predicates that repeat `hasControlCommand()` directly.
