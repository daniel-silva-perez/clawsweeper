## Summary

Keep open. Current main still has the reported debounce gap: the shared inbound debounce predicate bypasses only slash/text commands through `hasControlCommand()`, while existing abort-aware helpers, tests, and docs recognize standalone abort phrases such as `stop`, `abort`, and `wait`. The linked fix PR #48916 is closed unmerged, so there is no merged implementation to close against.

## What This Changes

Bare abort phrases such as `stop`, `abort`, and `wait` should bypass inbound message debouncing across debounced chat channels.

## Best Possible Solution

Use abort-aware command detection for shared and plugin-local debounce bypass decisions, keep normal text/media debounce behavior unchanged, preserve BlueBubbles’ documented coalescing opt-in, and add focused regression tests proving bare abort phrases flush immediately.

## Repair Work Prompt

Fix the bare abort debounce bug tracked by https://github.com/openclaw/openclaw/issues/48684. Current main still uses `hasControlCommand()` in `src/channels/inbound-debounce-policy.ts`, so bare abort phrases such as `stop`, `abort`, and `wait` can be delayed or merged by inbound debouncing. Change the shared debounce predicate to use abort-aware control detection (`isControlCommandMessage`) while preserving existing blank text, media, `/status`, bot-username, and normal text behavior. Add focused regression tests proving `stop`, `abort`, and `wait` bypass debounce immediately and normal text still debounces.
