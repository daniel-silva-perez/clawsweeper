## Summary

Keep open. Current main still has the reported debounce gap: the shared inbound debounce predicate bypasses only slash/text commands through `hasControlCommand()`, while the abort-aware helper already recognizes standalone abort phrases such as `stop`, `abort`, and `wait`. The related fix PR #48916 is closed unmerged, and the current debounce policy test still lacks bare-abort regression coverage.

## What This Changes

The issue requests that bare abort phrases such as `stop`, `abort`, and `wait` bypass inbound channel debouncing the same way slash control commands do.

## Best Possible Solution

Use abort-aware command detection for shared and plugin-local debounce bypasses, preserve the documented BlueBubbles coalescing opt-in, and add focused regression tests proving bare abort phrases flush immediately while normal text still debounces.

## Risks / Open Questions

- A fix that only changes `src/channels/inbound-debounce-policy.ts` would leave bundled plugins with local debounce predicates still using `hasControlCommand()` directly.
- BlueBubbles has an explicit documented `coalesceSameSenderDms` exception where DM commands may intentionally wait inside the debounce window; that behavior should be preserved.
- `isControlCommandMessage()` is intentionally broader than `hasControlCommand()`, so the change should stay scoped to debounce bypass paths rather than auth or mention-gating decisions that still need slash-command semantics.
