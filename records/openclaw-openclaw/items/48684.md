Keep open. Current main still routes shared inbound debounce bypass through `hasControlCommand()`, so standalone abort triggers remain debounceable text, and the only linked fix PR is closed unmerged.

## What This Changes

Bare abort phrases such as `stop`, `abort`, and `wait` should bypass inbound channel debounce immediately, matching slash-command control behavior.

## Best Possible Solution

Use abort-aware command detection for shared and plugin-local debounce bypasses, preserve normal text/media debounce behavior and the documented BlueBubbles coalescing exception, then add focused regression tests proving standalone abort phrases flush immediately.

## Repair Work Prompt

Fix the inbound debounce control-command gap for standalone abort phrases. Current main still uses `hasControlCommand()` in `src/channels/inbound-debounce-policy.ts`, and several bundled plugin-local debounce predicates do the same, so `stop`, `abort`, `wait`, and localized abort triggers can be delayed or merged instead of flushing immediately. Use the existing abort-aware `isControlCommandMessage()` API where appropriate. Preserve normal text/media debounce behavior, command normalization options such as Telegram bot usernames, and BlueBubbles' documented `coalesceSameSenderDms` opt-in where command-like DM text may intentionally wait to coalesce split-send payloads. Treat closed PR #48916 as useful reference work, but make the final fix against current main and include bundled plugin-local predicates as needed. Add regression coverage proving bare abort phrases bypass debounce while normal text still debounces. Add a user-facing changelog Fixes entry if repo convention requires it for this behavior change, crediting the reporter and any source PR author whose approach is reused; do not touch unrelated queueing, command authorization, or tool-abort behavior.
