## Summary

Keep open. Current main still has the reported debounce gap: the shared inbound debounce policy bypasses only slash/text commands through `hasControlCommand()`, while the existing abort-aware helper recognizes bare `stop`, `abort`, and `wait`; related PR #48916 is closed unmerged and the regression coverage is still missing.

## What This Changes

The issue requests that bare abort phrases bypass inbound message debouncing across chat channels so abort intent flushes immediately instead of being delayed or merged away.

## Best Possible Solution

Use abort-aware command detection for debounce bypass, add focused regression tests for bare abort phrases, and keep documented channel-specific debounce exceptions intact.

## Repair Work Prompt

Fix openclaw/openclaw#48684. Current main still debounces bare abort phrases because `src/channels/inbound-debounce-policy.ts` uses `hasControlCommand()` instead of abort-aware command detection. Update the shared debounce predicate to use `isControlCommandMessage()` so bare `stop`, `abort`, `wait`, and documented standalone stop phrases bypass debouncing like `/stop`. Add focused regression tests in `src/channels/inbound-debounce-policy.test.ts` for bare abort triggers and normal text behavior. Audit bundled plugin-local debounce predicates that directly call `hasControlCommand()` in Feishu, WhatsApp, Microsoft Teams, and Mattermost, and switch them to the abort-aware runtime command helper where immediate control-command bypass is expected. Preserve documented exceptions and special batching behavior, especially BlueBubbles `coalesceSameSenderDms`, Telegram forwarded/media batching, and media/location/reply exclusions. Treat #48916 as reference work only; do not broaden unrelated command parsing. Add a user-facing changelog fix entry if repo policy expects one, crediting the reporter/source contributor with `Thanks @sherly-techtree` and, if the prior PR is used, `Thanks @javery556`.
