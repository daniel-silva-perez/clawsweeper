## Summary

Keep open. Current main still routes standalone abort phrases through the inbound debounce path because the shared predicate uses slash-command detection, while the existing abort-aware helper and trigger set are separate and not used at that boundary.

## What This Changes

The issue asks for standalone abort phrases such as `stop`, `abort`, and `wait` to bypass inbound message debounce across debounced chat channels instead of being delayed or merged.

## Best Possible Solution

Use abort-aware command detection at shared and plugin-local debounce gates, preserve the documented BlueBubbles coalescing exception, and add focused regression tests proving bare abort phrases flush immediately while normal text and media behavior stay unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is direct: on current main, `shouldDebounceTextInbound({ text: "stop", cfg })` reaches `hasControlCommand("stop") === false` and returns `true`, so the abort phrase is held for debounce instead of flushing.

## Solution Assessment

Yes. The narrow maintainable fix is to use `isControlCommandMessage()` or the equivalent runtime command helper at debounce boundaries, with tests for `stop`, `abort`, and `wait`; a shared-only patch is incomplete because several bundled plugins have local predicates.
