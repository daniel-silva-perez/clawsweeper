## Summary

Keep open. Current main still lacks the requested standalone Telegram processing-placeholder message lifecycle and does not accept the proposed global or Telegram-specific config keys; the implemented adjacent paths are typing cues, ack/status reactions, and live preview/tool-progress edits. The issue also has explicit maintainer signal to keep it open plus an overlapping open waiting-tips PR, so this remains a product/config decision rather than conservative cleanup closure.

## What This Changes

The issue requests an opt-in Telegram processing indicator that sends a visible placeholder while a request is running, deletes it after a successful reply, and edits or reports an error on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after eligible inbound acceptance, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.
