## Summary

Keep open. Current main still lacks the requested standalone Telegram processing-placeholder message lifecycle and does not accept the proposed global or Telegram-specific config keys. The maintainer discussion already calls out that this should remain open, and the overlapping #40421 waiting-tips PR is still open while the direct processing-indicator attempts #43021 and #43720 were closed unmerged.

## What This Changes

Add an opt-in Telegram processing indicator that sends a visible placeholder while processing, deletes it after a successful reply, and edits or replaces it on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after eligible inbound acceptance, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed config is not accepted, and Telegram dispatch inspection shows no stored placeholder message lifecycle.

## Solution Assessment

Unclear whether the minimal placeholder or the broader waiting-tips direction is the best product solution. The narrow code path is identifiable, but maintainers should resolve the config and UX direction before another repair PR is queued.
