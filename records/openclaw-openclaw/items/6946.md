## Summary

Current main still does not implement the requested Telegram standalone processing-indicator message lifecycle. The Telegram plugin has adjacent typing, ack/status reaction, and send/edit/delete primitives, but no accepted processingIndicator/waitingTips config and no inbound placeholder message ID carried through dispatch cleanup. The item should stay open for maintainer product review, especially because the open #40421 waiting-tips PR overlaps with the same user-visible gap.

## Best Possible Solution

Keep this issue open as the canonical bundled Telegram processing-indicator request. The best implementation would be an opt-in Telegram plugin change with typed config/schema/help/docs, normal inbound dispatch wiring that sends a standalone placeholder after an eligible message is accepted, carries the sent message id through dispatch, deletes it after successful final delivery, and edits or replaces it on failure. Maintainers should first decide how that relates to #40421.
