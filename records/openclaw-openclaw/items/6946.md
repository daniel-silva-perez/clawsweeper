## Summary

Keep open. Current main still lacks the requested Telegram processing-placeholder lifecycle and does not accept the proposed global or per-Telegram config keys; adjacent typing cues, ack/status reactions, native-command progress placeholders, and generic error replies do not cover the ordinary inbound-turn behavior. The open waiting-tips PR overlaps the UX/config decision, while the direct processing-indicator PRs were closed unmerged, so this needs maintainer product direction before another automated fix PR is queued.

## What This Changes

The issue requests an opt-in Telegram processing indicator that sends a visible placeholder message while an inbound request is running, deletes it after a successful reply, and edits or replaces it on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after an eligible inbound message is accepted, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.
