## Summary

Current main still lacks a standalone Telegram processing-placeholder lifecycle and does not accept the proposed config surface. Existing typing cues, ack reactions, status reactions, preview streaming, and generic error replies are adjacent but do not send a visible message that is later deleted or edited, and the overlapping waiting-tips PR remains open while the direct implementation PRs were closed unmerged.

## What This Changes

Request an opt-in Telegram processing indicator that sends a visible placeholder while a message is processed and removes or updates it when the turn finishes.

## Best Possible Solution

Choose the intended bundled Telegram UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after an eligible inbound message is accepted, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.
