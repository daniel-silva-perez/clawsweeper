## Summary

Keep open. Current main still lacks the requested standalone Telegram processing-placeholder message lifecycle and does not accept the proposed config keys. Existing typing cues, ack reactions, status reactions, native-command progress placeholders, preview cleanup, and generic error replies are adjacent but do not send a visible ordinary-turn message that is later deleted or edited. The overlapping waiting-tips PR remains open in the supplied context, while the direct implementation PRs were closed unmerged, so this needs maintainer product/config direction rather than cleanup closure.

## What This Changes

The issue requests an opt-in Telegram processing indicator message that is sent while a request is being processed, deleted after a successful reply, and edited or replaced on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after an eligible inbound message is accepted, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.
