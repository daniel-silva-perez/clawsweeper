## Summary

Keep open. Current main still does not expose the requested Telegram processing-indicator config or lifecycle, and the related discussion shows this is an unresolved product/config choice rather than a cleanup-close item.

## What This Changes

Add an opt-in Telegram processing indicator that sends a visible placeholder while a request is being processed, then deletes it on success or edits/replaces it on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after eligible inbound acceptance, deletes it after successful final delivery, and edits or replaces it on failure with focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed config is not accepted, and Telegram dispatch inspection shows no stored placeholder message lifecycle.

## Solution Assessment

Unclear whether the narrow placeholder or broader waiting-tips design is the best product solution. The maintainable path is to resolve the UX/config shape first, then implement it in the Telegram plugin and shared config surfaces only as needed.
