## Summary

Keep open. Current main still does not expose a Telegram processing-placeholder config or lifecycle, and the existing Telegram feedback paths are typing cues, ack reactions, status reactions, streaming edits, and fallback error replies. The related waiting-tips PR is still open while two direct processing-indicator PRs were closed unmerged, so this needs maintainer UX/config direction rather than conservative cleanup closure.

## What This Changes

Request an opt-in Telegram processing indicator that sends a visible placeholder message while a turn is running, deletes it after a successful reply, and edits or replaces it on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after eligible inbound acceptance, deletes it after successful final delivery, and edits or replaces it on failure with focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed keys are not accepted, and Telegram inbound/dispatch inspection shows no stored placeholder message lifecycle.

## Solution Assessment

Unclear whether the narrow placeholder design or the broader #40421 waiting-tips design is the best product solution. The maintainable path is a maintainer UX/config decision first, followed by a Telegram plugin implementation without broad core changes unless the chosen design needs a shared channel helper.
