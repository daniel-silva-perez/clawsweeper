## Summary

Keep open. Current main still lacks a Telegram processing-placeholder message config and lifecycle; it only has typing cues, ack reactions, status reactions, streaming edits, and fallback error replies. The related waiting-tips PR remains open while two direct processing-indicator PRs were closed unmerged, so this needs maintainer UX/config direction rather than cleanup closure.

## What This Changes

Request to add an opt-in Telegram processing indicator message that is sent while an inbound message is being processed, deleted after a successful reply, and edited or replaced on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after eligible inbound acceptance, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed config is not accepted, and Telegram inbound/dispatch inspection shows no stored placeholder message lifecycle.

## Solution Assessment

Unclear whether the narrow placeholder design or the broader waiting-tips design is the best product solution. The maintainable path is for maintainers to decide the UX/config shape first, then implement it in the Telegram plugin and shared config surfaces only where needed.
