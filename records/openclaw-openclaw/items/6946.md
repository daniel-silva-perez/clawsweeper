## Summary

Keep open. Current main still lacks the requested standalone Telegram processing-placeholder message lifecycle and does not accept either proposed config surface; adjacent typing cues, ack/status reactions, preview streaming, and generic error replies do not send a visible placeholder immediately after inbound acceptance and then delete or edit that same message on completion. The issue also overlaps an open waiting-tips PR and two prior direct attempts closed unmerged, so this needs maintainer product/config direction rather than cleanup closure.

## What This Changes

Requests an opt-in Telegram processing indicator that sends a visible placeholder message while an inbound turn is running, deletes it after a successful reply, and edits or replaces it with an error on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after an eligible inbound message is accepted, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.
