## Summary

Current main still does not implement this PR's intended WhatsApp document filename fallback. The active WhatsApp plugin send path still uses bare `"file"`, tests still assert that behavior, and the PR patch targets the old `src/web` layout rather than the current `extensions/whatsapp` plugin layout.

## Best Possible Solution

Keep this PR open and ask for a rebase/port to the current `extensions/whatsapp` plugin. The best implementation is a small shared WhatsApp document filename resolver near `outbound-media-contract.ts` that preserves names with extensions, appends `extensionForMime(mimetype)` for extensionless fallbacks, uses the plugin-facing `openclaw/plugin-sdk/media-mime` helper, and updates both send API and auto-reply tests to expect `file.pdf` and a dynamic MIME case such as `file.csv`.
