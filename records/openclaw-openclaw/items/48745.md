## Summary

Current main does not implement this PR's intended WhatsApp document filename fallback. The WhatsApp code now lives under the bundled plugin, and the active outbound send API still falls back to bare "file" for extensionless document sends; the current tests also assert that behavior. The PR is still useful, but it needs to be rebased/adapted to the current `extensions/whatsapp` layout and should use the plugin-facing media MIME helper rather than old core-relative imports.

## Best Possible Solution

Keep the PR open and ask for a rebase/port to `extensions/whatsapp`. The best implementation is to add a small shared WhatsApp document filename resolver, probably near `outbound-media-contract.ts`, that preserves names with extensions and appends `extensionForMime(mimetype)` for extensionless fallbacks. Then update `extensions/whatsapp/src/inbound/send-api.ts`, auto-reply coverage, and `extensions/whatsapp/src/inbound/send-api.test.ts` to assert `file.pdf` and a dynamic type such as `file.csv`.
