## Summary

Keep this PR open. Current main and the latest release still keep the old 32-character generic token guard, but the proposed patch should not merge as-is because it weakens narrow-terminal protection for ordinary 33-128 character tokens and makes the new punctuation-based copy-sensitive exemption too broad; related open PR #39565 overlaps dotted-token preservation but does not supersede this broader change.

## What This Changes

The PR changes the TUI render sanitizer threshold from 32 to 128 characters, adds a code-like punctuation exemption for copy-sensitive tokens, and updates sanitizer tests accordingly.

## Best Possible Solution

Revise before merge: keep the generic narrow-terminal wrapping guard for ordinary long tokens, add a constrained copy-sensitive classifier for real code-like or file-like fragments, and add regression tests for both preserved copy-sensitive examples and generic overflow protection. Coordinate with #39565 if its narrower dotted-token classifier becomes the canonical path.

## Reproduction Assessment

Yes. Against the PR diff, a 90-character ordinary token no longer matches the 129+ regex and passes through unchanged, while a >128-character token with incidental punctuation reaches the new copy-sensitive branch and skips `chunkToken`; current main tests show the old `b`.repeat(90) <= 32 guard.

## Solution Assessment

No. The PR solves some copy/paste corruption by weakening the broad layout guard; the safer solution is a narrower classifier that preserves specific copy-sensitive code/entity/file tokens while leaving generic long-token wrapping intact.
