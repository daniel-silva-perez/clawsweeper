## Summary

Keep open. The PR addresses a real remaining privacy bug, but current main has since moved audio upload filename construction into the shared transcription form-data helper, so this branch is stale and should not merge as-is. Current main already avoids the missing-filename crash, but Windows-style host paths can still be preserved in multipart filenames.

## What This Changes

Adds a local OpenAI-compatible audio upload filename resolver plus unit tests for POSIX path basename stripping and missing runtime filenames.

## Best Possible Solution

Update `resolveAudioTranscriptionUploadFileName` in `src/media-understanding/shared.ts` to strip both POSIX and Windows path segments while preserving the current AAC/MIME extension normalization, then add focused tests for POSIX paths, Windows paths, missing or whitespace-only filenames, and AAC remapping.

Yes. Reading the current helper plus a focused Node `path.basename` check gives a high-confidence reproduction for Windows-style filenames leaking through on POSIX; the missing-runtime-filename crash no longer reproduces because the shared helper accepts an optional filename and falls back to `audio`.

## Solution Assessment

No, not as currently proposed. The logic direction is right, but the PR modifies the pre-refactor local helper instead of the current shared upload filename resolver, so it should be rebased or replaced with a narrow patch to `shared.ts`.
