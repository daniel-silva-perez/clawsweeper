## Summary

Keep this PR open. Current main partially addresses the original failure mode by routing OpenAI-compatible audio uploads through the shared `buildAudioTranscriptionFormData` helper, which safely falls back to `audio` when `fileName` is missing. It does not fully implement the PRâ€™s intended path-leak fix: the shared resolver still uses platform `path.basename`, so Windows-style paths are preserved on POSIX hosts. The PR remains useful, but it should be rebased onto the current shared helper rather than reintroducing provider-local filename handling.

## Best Possible Solution

Keep the PR open, but update it to patch `resolveAudioTranscriptionUploadFileName` in `src/media-understanding/shared.ts` using POSIX and Windows basename normalization, preserving the AAC-to-M4A behavior. Add focused tests for POSIX paths, Windows paths, missing/empty filenames, and AAC remapping so the shared SDK helper is covered for OpenAI-compatible, bundled, and third-party STT providers.
