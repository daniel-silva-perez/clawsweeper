## Summary

Keep open. The PR targets a valid Telegram bug: current main exposes `channels.telegram.textChunkLimit`, but `sendMessageTelegram` still hardcodes `4000` in `buildChunkedTextPlan`. The production diff follows the existing resolver seam, but the added regression omits required `cfg`, so the PR should be amended before merge rather than closed or replaced.

## What This Changes

The PR updates Telegram send chunk planning to use the configured text chunk limit capped at 4096 and adds a send-path regression for the 4096 case.

## Best Possible Solution

Amend this PR in place: keep the Telegram-owned resolver/clamp implementation, pass the configured object through `cfg` in the new regression, optionally add one lower configured-limit regression, then run the targeted Telegram send test, extension test typecheck, and the changed gate before landing.
