## Summary

Keep open. The PR targets a real Feishu card-action data-loss gap, but the branch only updates the direct handler type/path and direct handler tests. Current main still normalizes live `card.action.trigger` callbacks to `action.value` and `action.tag` before the handler runs, and the decoder/fallback path only consumes `action.value`, so real form/input/select fields would still be dropped. The discussion also shows a paired same-author issue and overlapping Feishu PRs, so this needs maintainer consolidation rather than cleanup closure.

## What This Changes

The PR extends Feishu card-action action typing and direct handler tests so `input_value`, `form_value`, `option`, `options`, and `name` are forwarded as structured JSON when form/select fields are present while preserving plain button command extraction.

## Best Possible Solution

Implement one Feishu-owned card-action callback contract end to end: preserve `form_value`, `input_value`, `name`, `option`, and `options` in the raw parser, expose them in typed event/decoder surfaces, serialize them in a backward-compatible way, and add parser-to-handler regression tests for live `card.action.trigger` callbacks plus existing plain `text`/`command` buttons.
