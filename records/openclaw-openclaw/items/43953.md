## Summary

Keep open. Current main still does not pass Feishu form/input/select callback fields through, and this PR is a plausible but incomplete fix because it reads the new fields in the handler after the production callback parser has already discarded them. The same-author issue #43202 is still open, and overlapping Feishu callback-field PRs remain unresolved or closed unmerged, so this needs author/maintainer repair rather than cleanup closure.

## What This Changes

The PR extends Feishu card-action typing and handler behavior to serialize form/input/select callback fields into a JSON synthetic message, with direct handler tests for the new paths.

## Best Possible Solution

Implement one Feishu-owned card-action callback contract end to end: preserve `form_value`, `input_value`, `name`, `option`, and `options` at the raw parser boundary, expose them through the typed event/decoder surfaces, keep existing plain button command extraction compatible, and add parser-to-handler regression tests using realistic `card.action.trigger` payloads.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a raw `card.action.trigger` payload containing `action.form_value` reaches `parseFeishuCardActionEventPayload`, which rebuilds `action` with only `value` and `tag`, so the handler cannot see the field. The PR's new tests bypass that production parser path.

## Solution Assessment

No. The PR's handler-only approach is not the best complete fix because live callbacks are normalized before the handler runs; the safer path is to preserve the fields in the parser/decoder contract and test the raw callback path through dispatch.
