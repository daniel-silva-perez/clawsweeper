## Summary

Keep open. The requested Feishu callback-data support is valid, but this PR is not merge-ready against current main because live `card.action.trigger` payloads are normalized before the handler and the current parser/decoder path still drops the sibling callback fields before the new handler branch could see them. The paired issue #43202 and overlapping PR #42908 should stay open until one Feishu-owned end-to-end fix is reviewed and landed.

## What This Changes

The PR extends Feishu card-action event typing and handler tests so `form_value`, `input_value`, `option`, `options`, and `name` can be serialized into a structured synthetic message while preserving plain button command extraction for name-only actions.

## Best Possible Solution

Implement one Feishu-owned callback-field pipeline end to end: preserve `form_value`, `input_value`, `name`, `option`, and `options` at the raw parser boundary, expose them through the typed event and decoder/fallback surfaces, keep plain button command extraction compatible, and add regression tests that start from realistic raw `card.action.trigger` payloads.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: route a raw `card.action.trigger` payload containing `action.form_value` through `parseFeishuCardActionEventPayload`; current main returns only `action.value` and `action.tag`, so the handler cannot receive the submitted field.

## Solution Assessment

No. The handler-only approach is not the best complete fix because live callbacks are normalized before the handler runs; the safer solution is to preserve and test these fields from parser through decoder and dispatch.
