## Summary

Keep open. The requested Feishu callback fields are still absent from current main, and this PR is not merge-ready because it only updates the direct handler surface while current main's live `card.action.trigger` path normalizes the raw payload before handling and drops the sibling callback fields the PR later tries to read.

## What This Changes

The PR extends Feishu card-action handler typings and direct handler tests to serialize `input_value`, `form_value`, `option`, `options`, and `name` into a structured JSON synthetic message when those fields are present.

## Best Possible Solution

Implement one Feishu-owned card-action callback contract end to end: preserve `form_value`, `input_value`, `name`, `option`, and `options` in the raw parser, expose them in typed decoder/event surfaces, serialize them in a backward-compatible format, and add parser-to-handler regression tests for live `card.action.trigger` callbacks plus existing plain `text`/`command` buttons.

## Reproduction Assessment

Yes. A current-main source check reproduces the loss: a raw callback with `action.form_value` reaches `parseFeishuCardActionEventPayload`, which rebuilds `action` with only `value` and `tag`, so the later handler and decoder cannot see the form/select fields.

## Solution Assessment

No. The PR's direct-handler branch is not the narrowest complete fix for current main because live callbacks are normalized before the handler; the safer solution is to preserve the fields at the parser boundary and then define one handler/decoder serialization path with tests.
