## Summary

Keep open. The PR addresses a real Feishu card-action data-loss problem, but the branch only changes the handler-level type/path and tests that call the handler directly. Current main still normalizes live `card.action.trigger` payloads through a parser that rebuilds `action` with only `value` and `tag`, so real Feishu callbacks would still drop `form_value`, `input_value`, `name`, `option`, and `options` before this PR's new handler branch can see them.

## What This Changes

The PR extends Feishu card-action handling and handler tests so `form_value`, `input_value`, `name`, `option`, and `options` can be forwarded in a structured synthetic message when those fields are present.

## Best Possible Solution

Consolidate this with #43202 and #42908 into one Feishu-owned fix that preserves `form_value`, `input_value`, `name`, `option`, and `options` from the raw callback parser through the event type, decoder/fallback, and handler, with parser-to-handler tests that lock in both interactive-field delivery and existing text/command behavior.
