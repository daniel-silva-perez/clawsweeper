## Summary

Keep open. The reported Feishu callback-field bug is valid, but this PR is not complete enough to merge because it only teaches the handler/tests about the extra fields while the live `card.action.trigger` parser on current main still reconstructs `action` with only `value` and `tag`, dropping the data before the handler can use it.

## What This Changes

The PR extends Feishu card-action event typing and handler behavior to serialize form/input/select callback fields into a JSON synthetic message, with handler-level tests for `input_value`, `form_value`, `option`, and name-only button behavior.

## Best Possible Solution

Implement one Feishu-owned callback-field contract end to end: preserve `form_value`, `input_value`, `name`, `option`, and `options` at the raw parser boundary, expose them through the typed event and decoder/fallback surfaces, keep plain button command extraction compatible, and add regression tests that start from realistic raw `card.action.trigger` payloads.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass a raw `card.action.trigger` payload with `action.form_value` through `parseFeishuCardActionEventPayload`; current main returns only `action.value` and `action.tag`, so the handler cannot see the submitted form field.

## Solution Assessment

No. The handler-only approach is not the best complete fix because live callbacks are normalized before the handler runs; the safer solution is to preserve and test these fields from parser through decoder and dispatch.
