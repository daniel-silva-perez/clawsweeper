## Summary

Keep open. Current main still gives the shared trusted web-tools endpoint a private-network-allowing SSRF policy, the shared web-search helpers and SDK exports still route through that endpoint, and the linked implementation PR remains separate from main. Because this is SSRF-sensitive and plugin-facing, it should stay open for maintainer/security review.

## What This Changes

The issue asks to split web-tools SSRF policy so private-network access is explicit for self-hosted providers instead of inherited by all bundled web-search providers and SDK helpers.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search path, an explicit self-hosted private-network opt-in for SearXNG-style providers, additive SDK exports, and tests covering public-provider private redirects plus SearXNG private HTTP success.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: any provider using withTrustedWebSearchEndpoint or postTrustedWebToolsJson reaches withTrustedWebToolsEndpoint, whose policy sets dangerouslyAllowPrivateNetwork and causes the SSRF resolver to skip private-address checks.

## Solution Assessment

Yes. The narrow maintainable direction is a strict default shared helper plus an explicit self-hosted/private-network opt-in, with SearXNG moved to that path and plugin SDK exports added for third-party self-hosted providers.
