## Summary

Keep open. Current main still routes shared web-search/web-tools requests through a trusted SSRF policy that allows private-network targets, and the paired implementation PR remains open rather than merged. Because this is SSRF-sensitive and touches plugin-facing helpers, it should go through maintainer/security review instead of cleanup closure.

## What This Changes

The issue requests splitting the web-tools SSRF policy so private-network access is strict by default and explicit only for self-hosted providers such as SearXNG.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default shared web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, SDK exports that preserve compatibility, and tests proving public providers block private redirects while SearXNG remains usable.

## Reproduction Assessment

Yes. The current static path is high-confidence: `withTrustedWebToolsEndpoint` always passes `dangerouslyAllowPrivateNetwork`, shared web-search helpers use it, and existing SSRF tests show private redirect blocking only when that permissive policy is absent.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: make the shared trusted helper strict by default and add an explicit self-hosted/private-network endpoint for SearXNG and future self-hosted providers. Because it touches security policy and SDK surface, the implementation should be reviewed rather than auto-applied.
