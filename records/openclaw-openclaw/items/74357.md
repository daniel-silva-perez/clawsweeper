## Summary

Keep open. Current main still routes the shared trusted web-tools/web-search helpers through an SSRF policy with `dangerouslyAllowPrivateNetwork: true`, and the linked implementation path, PR #74360, remains an open paired fix rather than a merged change. Because this is SSRF-sensitive and affects plugin-facing SDK helpers, it needs maintainer/security review before cleanup.

## What This Changes

The issue requests splitting the shared web-tools SSRF policy so private-network access is strict by default and only explicitly enabled for self-hosted providers such as SearXNG.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always passes a private-network-allowing policy, and the shared web-search helpers used by bundled providers call that endpoint.

## Solution Assessment

Yes for the requested direction, but the exact implementation still needs review. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it preserves SearXNG while removing broad private-network access from public provider endpoints.
