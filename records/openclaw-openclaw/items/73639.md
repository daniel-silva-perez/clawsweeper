## Summary

Current main still has the reported gap: non-interactive onboarding only accepts token/password gateway auth, while trusted-proxy is already supported by the core gateway config, runtime auth path, docs, and interactive gateway setup. The issue should stay open because there is an open same-author draft implementation PR (#49107) plus a duplicate-looking same-author issue (#73638), so maintainer triage should choose the canonical implementation path before closing anything.

## What This Changes

Request non-interactive `openclaw onboard` flags and config writes for `trusted-proxy` gateway authentication behind a reverse proxy.

## Best Possible Solution

Review, land, or replace PR #49107 with a focused core onboarding change that accepts `--gateway-auth trusted-proxy`, writes `gateway.trustedProxies` and `gateway.auth.trustedProxy.*` safely, covers non-loopback Control UI origins where needed, updates the public onboarding docs, and adds targeted tests for valid config writes plus missing or unsafe security fields. After that path is settled, deliberately dedupe #73638 if it is not the canonical issue.

## Reproduction Assessment

Yes. The high-confidence reproduction path is `openclaw onboard --non-interactive --gateway-auth trusted-proxy ...`; current-main source rejects any non-interactive `--gateway-auth` value other than `token` or `password` before writing config.

## Solution Assessment

Yes, the requested direction belongs in core onboarding rather than a plugin because trusted-proxy is already a core gateway auth mode and the gap is the CLI setup surface. The safest implementation should be narrow and explicit about all security-critical trusted-proxy fields instead of accepting raw config blobs.
