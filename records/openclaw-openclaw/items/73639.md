## Summary

Keep open. Current main still leaves the reported onboarding gap: `trusted-proxy` is a supported gateway auth mode at runtime and in interactive gateway setup, but the non-interactive onboard path and CLI option type still accept only `token` and `password`. The issue is also paired with same-author open draft PR #49107, so cleanup should wait for maintainer review of that implementation or a maintainer-directed replacement.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring `trusted-proxy` gateway authentication and its required reverse-proxy trust settings through typed CLI flags.

## Best Possible Solution

Add a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, exposes typed flags for the trusted proxy source and identity constraints, writes the existing gateway config schema, updates user-facing docs, and preserves current fail-closed runtime validation.

## Reproduction Assessment

Yes. The source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits because only `token` and `password` are accepted.

## Solution Assessment

Yes. The maintainable solution belongs in core onboarding because `trusted-proxy` is already a core gateway auth mode and interactive setup already configures it; raw config editing or a plugin would bypass the typed onboarding contract and security prompts.
