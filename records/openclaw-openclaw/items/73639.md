## Summary

Keep open. Current main and the latest release still reject `trusted-proxy` in non-interactive onboarding, while the runtime and interactive gateway setup already support that auth mode; same-author draft PR #49107 targets the gap but remains open with security-sensitive review findings, so this issue should stay open until that PR is landed, replaced, or the duplicate cluster is resolved.

## What This Changes

The issue asks for non-interactive onboarding flags that can configure gateway `trusted-proxy` authentication for scripted reverse-proxy deployments.

## Best Possible Solution

Land or replace PR #49107 with a current-main-compatible core onboarding change that accepts `--gateway-auth trusted-proxy`, writes trusted-proxy config through typed flags, clears incompatible token state, handles loopback and Control UI origins explicitly, updates docs, and adds focused non-interactive onboarding tests plus gateway-auth regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits because only `token` and `password` are accepted.

## Solution Assessment

Yes for the requested direction, but not via raw config editing or a plugin. Because trusted-proxy is already a core gateway auth mode and interactive onboarding can configure it, the maintainable fix belongs in the core non-interactive onboarding surface, with security review for the exact flags and transitions.
