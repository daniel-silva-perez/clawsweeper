## Summary

Keep open. Current main still has the reported gap: `trusted-proxy` is a supported gateway auth mode at runtime and in the interactive gateway setup, but `openclaw onboard --non-interactive` still accepts only `token` or `password`. The issue is also paired with open draft PR #49107 and same-author duplicate/context issue #73638, so cleanup should wait for maintainer/security review rather than closing this item. Reviewed item: https://github.com/openclaw/openclaw/issues/73639

## What This Changes

Request non-interactive `openclaw onboard` flags and config writing support for `trusted-proxy` gateway auth behind reverse proxies.

## Best Possible Solution

Add a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, exposes typed non-interactive flags for trusted proxy IPs, identity header, required headers, allowUsers, loopback opt-in, and Control UI allowed origins as needed, writes the existing config schema, updates docs, and preserves current fail-closed runtime validation.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and rejects the auth mode before config can be written.

## Solution Assessment

Yes. Core onboarding parity is the right direction because trusted-proxy is already a core gateway auth mode and interactive setup already supports it; raw config editing or plugin work would bypass the typed onboarding/security contract.
