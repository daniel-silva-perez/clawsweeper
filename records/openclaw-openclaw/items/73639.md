## Summary

Keep open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding even though trusted-proxy is already a core gateway auth mode with interactive setup, runtime validation, and public docs; closing would be premature because same-author draft PR #49107 targets the exact gap and duplicate-looking #73638 is still open.

## What This Changes

Add non-interactive onboarding support for selecting `trusted-proxy` gateway auth and writing the required reverse-proxy auth configuration for scripted deployments.

## Best Possible Solution

Review, land, or replace PR #49107 with a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, writes explicit `gateway.trustedProxies` and `gateway.auth.trustedProxy.*` fields, handles non-loopback Control UI origins where required, updates onboarding docs, and adds targeted tests for valid config writes plus missing or unsafe security-critical inputs; then dedupe #73638 if it is not the canonical thread.

## Reproduction Assessment

Yes. The source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy ...`; current main exits from `applyNonInteractiveGatewayConfig` because non-interactive gateway auth only accepts `token` or `password`.

## Solution Assessment

Yes. This belongs in core onboarding rather than a plugin because trusted-proxy is already a core gateway auth mode; the maintainable fix is explicit first-class flags that preserve existing runtime validation rather than raw config injection.
