## Summary

Current main still has the reported gap, so this issue should stay open. Non-interactive onboarding still only accepts token/password gateway auth, while trusted-proxy already exists in the interactive setup, config contract, runtime auth path, docs, and tests; the open same-author draft PR #49107 is the active implementation candidate, and same-author issue #73638 should be triaged as part of the same cluster.

## What This Changes

The issue requests non-interactive `openclaw onboard` support for configuring trusted-proxy gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Review, land, or replace PR #49107 with a focused core onboarding change that adds explicit validated trusted-proxy flags, writes `gateway.auth.trustedProxy` and `gateway.trustedProxies` safely, handles Control UI allowed origins for non-loopback deployments, updates public onboarding docs, and adds targeted tests for valid writes plus missing/unsafe security-critical fields.

## Reproduction Assessment

Yes. The high-confidence reproduction path is `openclaw onboard --non-interactive --gateway-auth trusted-proxy ...`; current-main source rejects that value before any trusted-proxy config can be written.

## Solution Assessment

Yes. The requested direction is the right product boundary because trusted-proxy is already a core gateway auth mode and the missing piece is CLI onboarding parity; the safest implementation is an explicit flag surface rather than raw config injection or manual post-edit instructions.
