## Summary

Keep open. Current main still has no separate gateway service-identity path: gateway clients authenticate through the existing token/password/bootstrap/device credential payload, while trusted-proxy local-direct handling remains tied to loopback rejection plus password fallback. The issue also has fresh deployment confirmations and maintainer design feedback, so this is active security-sensitive RFC work rather than cleanup closure.

## What This Changes

The issue requests separating internal gateway service identity from user-facing gateway auth modes, especially for local/internal callers in trusted-proxy deployments.

## Best Possible Solution

Keep the RFC open for maintainer auth/security design review, agree on the local service-auth trust model and Phase 0 sequencing, then implement the approved service-identity surface as small additive PRs covering config schema, SecretRef-backed token storage, GatewayClient injection, auth result/principal handling, docs, doctor/migration behavior, and targeted regression tests.

## Reproduction Assessment

Yes for the current behavior: current tests show local-direct trusted-proxy token attempts reject with trusted_proxy_loopback_source and only password fallback succeeds. The issue comments also provide real nginx/oauth2-proxy and Traefik/authentik deployment confirmations.

## Solution Assessment

Unclear until maintainer security review. nickytonline's comment points to a narrower local SecretRef-backed service token for direct loopback calls, not treating localhost as trusted-proxy traffic, which should shape any implementation.
