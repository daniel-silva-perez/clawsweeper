## Summary

Keep open. Current main still couples internal/local Gateway clients to the existing user-auth decision tree: trusted-proxy auth rejects loopback by default, the only local-direct fallback is password-based, GatewayClient sends shared/device credentials, and config/docs expose no separate service-identity axis. The issue is also a fresh, security-sensitive RFC with active deployment confirmations in the comments, so it needs maintainer auth/design review rather than cleanup closure.

## What This Changes

The issue proposes separating internal Gateway service identity from user-facing Gateway auth modes so child processes, backend clients, and local operator tools no longer rely on trusted-proxy/password/local-direct fallback behavior.

## Best Possible Solution

Keep the RFC open for maintainer auth/security design review, agree on the trust model and phasing, then implement any approved service-identity surface as small additive PRs covering config schema, auth resolution, GatewayClient injection, scoped roles, docs, doctor/migration behavior, and targeted regression tests.
