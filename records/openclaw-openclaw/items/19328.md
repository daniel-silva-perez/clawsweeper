## Summary

Keep this PR open. Current main still builds the gateway agent session writeback patch from a cached `loadSessionEntry` result, including `modelOverride` and `providerOverride`, then merges that patch over the fresh store row loaded inside `updateSessionStore`; the PR is a focused fix for that remaining race.

## What This Changes

This PR removes cached session-entry fields from the gateway agent writeback patch and adds regression tests for preserving fresh session-store fields when `loadSessionEntry` is stale.

## Best Possible Solution

Land this PR or an equivalent focused change that keeps cached `entry` fields out of the agent session writeback patch while preserving fresh store values, then handle the broader delivery-field follow-up separately in #70575.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a deterministic stale-cache/fresh-store unit shape: make `loadSessionEntry` return a cached row without overrides, make `updateSessionStore` receive a fresh row with overrides, and observe current main merging the stale patch over the fresh row.

## Solution Assessment

Yes, for the reported modelOverride/providerOverride race. The narrowest maintainable fix is to stop putting cached override fields into the writeback patch so `mergeSessionEntry` preserves the fresh store row; broader stale delivery metadata should stay separate unless maintainers intentionally expand scope.
