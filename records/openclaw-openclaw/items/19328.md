## Summary

Keep this PR open. Current main still has the stale-cache writeback pattern the PR targets: the agent handler reads a cached session entry, copies `modelOverride`/`providerOverride` from that entry into `nextEntryPatch`, then merges that patch over the freshly loaded store row. The provided PR context also shows the current PR is focused to `agent.ts` and `agent.test.ts`, with no security-sensitive workflow, dependency, lockfile, package, or release-script changes.

## Best Possible Solution

Keep this PR open and review or land it, or land an equivalent focused fix in `src/gateway/server-methods/agent.ts` that prevents cached `entry` fields such as `modelOverride` and `providerOverride` from being merged over the fresh store row. Keep targeted regression coverage for a stale `loadSessionEntry` result racing with a fresh store row containing model override fields.
