## Summary

Keep this PR open. Current main still constructs an agent session patch from the cached `loadSessionEntry` result and includes store-owned fields such as `modelOverride` and `providerOverride`; that patch is then merged over the fresh store row, so the stale-cache race this PR targets is still present. Security pass found no supply-chain or CI risk in the provided PR scope: it only changes the gateway agent handler and its colocated tests.

## Best Possible Solution

Keep this PR open and review or land it, or land an equivalent focused fix in `src/gateway/server-methods/agent.ts`: ensure `nextEntryPatch` contains only request-derived values, or build/merge store-owned fields from the fresh row inside `updateSessionStore`. Keep #70575 as the separate narrower delivery-field follow-up, and keep regression coverage for stale cached `loadSessionEntry` data racing with a fresh store row that contains `modelOverride`/`providerOverride` and other store-owned fields.
