## Summary

Keep this PR open. Current main still contains the stale cached session-entry writeback pattern the PR targets: `agent` reads `entry` before the locked store update, copies `modelOverride` and `providerOverride` from that cached entry into `nextEntryPatch`, then merges that patch over the fresh store row. The linked maintainer discussion already keeps the narrower model-override fix separate from the residual delivery-field race in #70575.

## Best Possible Solution

Keep this PR open and review or land it, or land an equivalent focused fix in `src/gateway/server-methods/agent.ts` that stops cached `entry` fields such as `modelOverride` and `providerOverride` from being merged over the fresh store row. Keep the regression tests that simulate stale `loadSessionEntry` data racing with a fresh store row; leave the narrower delivery-field follow-up to #70575.
