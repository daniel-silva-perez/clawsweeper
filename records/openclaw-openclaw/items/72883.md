## Summary

Keep open: current main still rejects the reported safe cache-trace config mutation, and PR #73098 is an open, unmerged implementation candidate with closing syntax rather than a reason to close the issue before merge.

## What This Changes

Request that the agent-facing gateway `config.patch`/`config.apply` path allow enabling `diagnostics.cacheTrace` only when message, prompt, and system content capture remain disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so the gateway tool permits cache-trace enablement only when the resulting config has all three content-capture flags disabled, with regression coverage for safe enablement and unsafe attempts.

## Reproduction Assessment

Yes. Source inspection of current main is a high-confidence reproduction path: the reported patch changes `diagnostics.cacheTrace` leaves, and the mutation guard rejects any changed leaf absent from the allowlist.

## Solution Assessment

Yes. A conditional allow path based on the post-merge resulting config is the best direction; unconditionally allowing the subtree is less safe because the content-capture fields default to `true`.
