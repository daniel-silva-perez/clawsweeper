## Summary

Current main still rejects the reported safe cache-trace config patch, and PR #73098 is an open, unmerged implementation candidate with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Allow agent-facing gateway config.patch/config.apply to enable diagnostics.cacheTrace only when message, prompt, and system content capture are disabled.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so the gateway tool permits cache-trace enablement only when the resulting config has all three content-capture flags disabled, with regression tests for safe enablement and unsafe attempts.

## Reproduction Assessment

Yes. Static inspection of current main is enough: the reported patch changes diagnostics.cacheTrace leaves, those leaves are absent from ALLOWED_GATEWAY_CONFIG_PATHS, and assertGatewayConfigMutationAllowed rejects them before any write.

## Solution Assessment

Yes. The best direction is a conditional allow based on the resulting cacheTrace config; unconditionally allowing the subtree is less safe because all three content-capture flags default to true.
