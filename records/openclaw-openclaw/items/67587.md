## Summary

Keep this issue open. Current main still rewrites Discord outbound @name text only when the process-local, account-scoped directory cache already contains a matching handle; unknown handles are intentionally preserved as plaintext. I did not find a deterministic config-backed alias map, normal inbound-message cache seeding path, or first-class outbound helper that would make known agent/team mentions reliable without a prior directory lookup.

## Best Possible Solution

Implement this in the Discord plugin rather than closing the issue. A conservative fix would add an explicit account-scoped, preferably guild-aware mention alias map such as channels.discord.mentionAliases or channels.discord.accounts.<id>.mentionAliases, consult it in rewriteDiscordKnownMentions before the transient directory cache, and optionally seed trusted inbound author/member names into the cache. Cover configured aliases, account isolation, unknown-name preservation, reserved mentions, code spans/fences, normal sends, webhook sends, and structured sends.

## Risks / Open Questions

- A fix that rewrites plaintext @names can create real Discord notifications, so it should preserve @everyone, @here, unknown handles, and code spans/fences unless an alias is explicitly trusted.
- Discord usernames, display names, and nicknames can collide; deterministic aliases should be account-scoped and likely guild-aware.
- Adding the alias surface should update the Discord config type, zod schema/generated metadata, docs, UI/config hints if applicable, and regression tests together.
