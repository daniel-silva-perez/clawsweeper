## Summary

Keep open: current main still does not mirror Discord Components v2 outbound component text into the target session transcript, so the PR addresses a real bug, but the proposed patch is not merge-ready because it broadens shared media transcript behavior and introduces an unbounded, lower-trust adapter-metadata-to-assistant-transcript path.

## What This Changes

The PR adds Discord component transcript summarization, returns that summary through outbound delivery result metadata, changes shared delivery mirroring to prefer adapter transcript text, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a narrow enriched-transcript implementation that mirrors Discord component text and interactive labels through one idempotent write path, gates text-plus-media transcript merging to explicit enriched adapter summaries, preserves ordinary captioned-media mirror semantics, and bounds or labels UI-derived transcript text before it becomes assistant session memory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in `resolveMirroredTranscriptText` runs for every caller, not only adapter-enriched component sends. Current main has delivery mirror coverage expecting a captioned media send to persist `report.pdf`, not `caption\nreport.pdf`; this patch would change ordinary media transcripts across channels and duplicate assistant text. Keep the old media-first behavior unless the caller explicitly passes enriched adapter transcript text.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff introduces adapter result metadata into assistant transcript persistence without a clear size or trust boundary.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: `deliverOutboundPayloadsCore` would join `meta.transcriptText` from delivery results and persist it as assistant transcript text without a character, line, or result cap. Because channel adapters/plugins can populate result metadata, this should be bounded before it can inflate session files or downstream context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:816`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but it should be explicitly labeled or constrained so UI-derived text is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
