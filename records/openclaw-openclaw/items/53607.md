## Summary

Keep open. The PR targets a real Discord Components v2 context-loss bug that current main still does not solve, but the proposed patch is not mergeable as-is because it broadens media transcript behavior across all channels and leaves the adapter-provided transcript metadata trust/size boundary unresolved.

## What This Changes

The PR adds Discord component transcript summarization, returns that summary as outbound result metadata, changes shared delivery mirroring to prefer enriched adapter text, adjusts text-plus-media mirror resolution, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a refreshed, narrow enriched-transcript implementation that mirrors Discord component text and interactive labels through one idempotent write path, gates text-plus-media merging to explicit enriched adapter summaries, preserves ordinary captioned-media filename semantics, and bounds or labels UI-derived text before it becomes assistant session context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate text/media merging to enriched transcript paths:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new `trimmed && mediaUrls.length > 0` branch in `resolveMirroredTranscriptText` runs for every captioned media send, not just adapter-enriched Discord component sends. Current main has coverage expecting a captioned media delivery to persist `report.pdf`; this patch would persist `caption\nreport.pdf` across ordinary channels and increase transcript noise. Pass an explicit enriched-text signal or branch at the caller so non-enriched media mirrors keep existing semantics.
  - confidence: 0.92
- **[P2] Deduplicate base text in mixed enriched batches:** `src/infra/outbound/deliver.ts:813`
  - body: In mixed result batches, prepending `params.mirror.text` to `adapterParts` can duplicate content when the enriched component transcript already includes the same base message, such as `Pick a color` plus `Pick a color\n[Red]`. Merge or dedupe the base text before appending adapter summaries so a single mirrored assistant entry does not repeat itself.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff promotes generic adapter metadata into persisted assistant transcript text without a clear size or trust boundary.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: `deliverOutboundPayloadsCore` would join `meta.transcriptText` from delivery results and persist it as assistant transcript text without a character, line, or result cap. Because channel adapters/plugins can populate result metadata, this should be bounded before it can inflate session files or downstream model context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:816`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but the persisted text should be explicitly labeled or constrained so UI-derived content is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
