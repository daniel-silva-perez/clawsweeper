## Summary

Keep open: current main still has the Discord Components v2 transcript gap, but this PR is not merge-ready because the proposed shared transcript helper change would regress ordinary captioned-media mirroring and the adapter-provided transcript metadata still needs an explicit size/trust boundary before persistence.

## What This Changes

The PR adds Discord component transcript summarization, returns that summary as outbound result metadata, updates shared delivery mirroring to prefer enriched adapter text, changes text/media mirror resolution, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a refreshed, narrow enriched-transcript implementation that mirrors Discord component text and interactive labels through one idempotent write path, preserves existing filename-only media mirror semantics for non-enriched sends, deduplicates mixed enriched/plain batches, and bounds or labels UI-derived transcript text before it becomes assistant session context.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main registers Discord component entries and routes clicks back to the session, but the component send path does not add enriched outbound component text to the transcript and the shared mirror ignores result metadata.

## Solution Assessment

No. The direction is right, but this branch broadens shared media transcript behavior and lacks a clear trust/size contract for adapter transcript metadata; a safer fix should pass an explicit enriched-text signal, dedupe merged summaries, and constrain UI-derived text before persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in `resolveMirroredTranscriptText` runs for every caller, not just Discord component results. Current main has coverage expecting a captioned media send to persist `report.pdf`; this patch would persist `caption\nreport.pdf` for ordinary media sends across channels. Keep the old media-only behavior unless the caller explicitly passes enriched adapter transcript text.
  - confidence: 0.92
- **[P2] Deduplicate base text in mixed enriched batches:** `src/infra/outbound/deliver.ts:813`
  - body: In mixed result batches, prepending `params.mirror.text` to `adapterParts` can duplicate content when the enriched component transcript already includes the same base message, such as `Pick a color` plus `Pick a color\n[Red]`. Merge or dedupe the base text before appending adapter summaries so one mirrored assistant entry does not repeat itself.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff promotes adapter-provided transcript metadata into persisted assistant session context without an explicit size or trust boundary.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: `deliverOutboundPayloadsCore` would join `meta.transcriptText` from delivery results and persist it as assistant transcript text without a character, line, or result cap. Because channel adapters/plugins can populate result metadata, this should be bounded before it can inflate session files or downstream model context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:816`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but the persisted text should be explicitly labeled or constrained so UI-derived content is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
