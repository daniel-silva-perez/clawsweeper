## Summary

Keep this PR open. Current main still allows same-provider cross-context message sends by default, so the intended hardening is not obsolete, but this branch should not merge as written because it imports a stale Telegram core path, leaves a topic-bound Telegram escape to the base chat, and does not update the generated config schema help.

## What This Changes

This PR changes outbound `message` cross-context policy to default-deny same-provider target hops, updates tests/config help/changelog, adds Telegram topic handling, and also includes unrelated failover retry and Control UI scroll changes.

## Best Possible Solution

Refresh and narrow the branch to the outbound message-policy hardening: default-deny same-provider forwarding, preserve the legacy override, compare provider/thread identity through plugin-owned seams, require Telegram topic identity when a session is topic-bound, regenerate generated config schema help, and keep the changelog/docs migration clear.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction is `enforceCrossContextPolicy` or `runMessageAction` with a bound Slack-like `toolContext.currentChannelId`, `currentChannelProvider` set to the same provider, and a different explicit target while `allowWithinProvider` is omitted; current main returns early because the default is permissive.

## Solution Assessment

No. The default flip is the right direction, but this branch is not the best fix as written because it deep-imports a stale Telegram core path, still allows topic-bound Telegram escapes to the base chat, leaves generated schema help stale, and carries unrelated changes.
