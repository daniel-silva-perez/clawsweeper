## Summary

Keep this PR open for maintainer follow-up: current main still has the same-provider default-allow gap, but the PR is not merge-ready because it crosses the Telegram plugin boundary, leaves a topic-bound Telegram escape, has stale generated config help, and misses required changelog attribution.

## What This Changes

The PR changes outbound `message` policy to default-deny same-provider cross-context sends from bound sessions, updates config help/tests/changelog, and also includes unrelated failover retry and Control UI scroll changes.

## Best Possible Solution

Refresh and narrow the branch to the outbound message-policy hardening only: default-deny same-provider forwarding, preserve the legacy override, compare provider/thread identity through a supported channel/plugin seam, require Telegram topic identity when a session is topic-bound, regenerate config schema artifacts, add required changelog attribution, and split unrelated failover/UI work out.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is `enforceCrossContextPolicy` or `runMessageAction` with default config, matching `currentChannelProvider`, a bound `currentChannelId`, and a different explicit same-provider target; current main returns before comparing the target.

## Solution Assessment

No. The default flip is the right direction, but this PR is not the best implementation as written because it uses a stale Telegram core import, leaves a topic-bound Telegram escape, leaves generated config surfaces stale, and includes unrelated changes.
