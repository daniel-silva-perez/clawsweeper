## Summary

Keep open. Current main still allows the reported same-provider target hop, so this PR addresses a real remaining outbound-message bug, but the current patch has blocking correctness and security issues and should not merge as-is.

## What This Changes

The PR changes outbound `message` policy to default-deny same-provider cross-context sends, updates related config help/tests/changelog, adds Telegram topic handling, and also includes unrelated failover retry and Control UI scroll changes.

## Best Possible Solution

Refresh and narrow the branch to the outbound message-policy hardening only: default-deny same-provider forwarding, preserve the legacy override, compare provider/thread identity through a plugin-owned seam, require Telegram topic identity when a session is topic-bound, regenerate config schema artifacts, add required changelog attribution, and split unrelated failover/UI work out.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is `enforceCrossContextPolicy` or `runMessageAction` with a bound Slack-like `toolContext.currentChannelId`, matching `currentChannelProvider`, default config, and a different explicit target; current main returns early before comparing targets.

## Solution Assessment

No. The default flip is the right direction, but this PR is not the best implementation as written because it uses a stale Telegram core import, leaves a topic-bound Telegram escape, leaves generated config surfaces stale, and includes unrelated changes.
