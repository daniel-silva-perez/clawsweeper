## Summary

Keep this PR open because current `main` still allows same-provider cross-context message sends by default, so the security hardening remains relevant. The proposed branch is not merge-ready as-is because it is stale against current channel/plugin boundaries, leaves a Telegram topic-bound escape, misses generated config-schema drift, and mixes unrelated failover/UI changes into the data-scope change.

## What This Changes

The PR changes outbound `message` policy to default-deny same-provider cross-context sends, updates related config help/tests/changelog, adds Telegram topic handling, and also changes failover retry classification plus Control UI scroll fallback behavior.

## Best Possible Solution

Refresh this as a narrow outbound policy patch: default-deny same-provider forwarding, preserve the legacy override, compare channel/thread identity through a supported plugin-owned seam, require Telegram topic identity when the current context is topic-bound, regenerate config schema artifacts, and keep unrelated failover/UI work in separate PRs.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is `enforceCrossContextPolicy` or `runMessageAction` on current `main` with default config, matching `currentChannelProvider`, a bound `currentChannelId`, and a different explicit same-provider target; the policy returns early at the permissive default.

## Solution Assessment

No. The default flip is the right product direction, but this PR is not the best implementation as written because it reaches into a stale Telegram path, leaves a topic-bound Telegram escape, misses generated config output, and carries unrelated changes.
