## Summary

Keep open. Current main and the latest release still allow same-provider target hops by default, so the hardening is not obsolete; however this branch should not merge as-is because it has unresolved build/boundary, Telegram topic-isolation, and generated-schema issues plus unrelated changes.

## What This Changes

This PR changes outbound `message` cross-context policy to deny same-provider target hops unless `tools.message.crossContext.allowWithinProvider=true`, updates config help/tests/changelog, adds Telegram topic handling, and includes unrelated failover and Control UI scroll changes.

## Best Possible Solution

Refresh and narrow the branch to the outbound message-policy hardening: default-deny same-provider forwarding, preserve the legacy override, compare provider/thread targets through plugin-owned seams, require Telegram topic identity when a session is topic-bound, regenerate generated config schema help, keep the changelog/docs migration clear, and validate focused outbound plus Telegram coverage before the changed gate.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction is `enforceCrossContextPolicy` or `runMessageAction` with a bound Slack-like `toolContext.currentChannelId`, `currentChannelProvider` set to the same provider, and a different explicit target while `allowWithinProvider` is omitted; current main returns early because the default is permissive.

## Solution Assessment

No. The intended default flip is the right direction, but this branch is not the best fix as written because it deep-imports a missing/stale Telegram core path, leaves topic-bound Telegram escape behavior, and does not regenerate generated config help.
