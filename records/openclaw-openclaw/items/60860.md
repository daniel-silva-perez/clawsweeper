## Summary

Keep open. Current main has partial Google Vertex support, but it still does not fully cover this PR's intended dedicated runtime provider and service-account ADC path, and the PR remains tied to active Google Vertex follow-up work. The current PR branch is not merge-ready because the ADC token exchange drops the existing injected fetch path and introduces an unkeyed credential cache in an auth-sensitive path.

## What This Changes

The PR adds a dedicated `google-vertex` provider to the Google plugin, removes the legacy `google-vertex` alias from the Google AI Studio provider, adds Vertex region/catalog/ADC helpers and tests, and changes the Google transport to build Vertex project/location URLs with ADC-derived Bearer auth.

## Best Possible Solution

Land or replace this with a maintainer-approved Google plugin implementation that gives `google-vertex` exact provider ownership, keeps selection-time and runtime auth semantics aligned, supports authorized_user and service_account ADC with credential-keyed token caching, preserves injected fetch/proxy behavior, resolves project/location deterministically, and adds focused routing/auth tests plus docs for the supported setup.

## Reproduction Assessment

Yes for the PR review findings: the current main transport test path passes a token `fetch` mock through stream options, while the PR branch no longer threads that fetch into ADC token refresh. The linked Google Vertex issues also provide concrete ADC and wrong-endpoint failure configurations for the central feature.

## Solution Assessment

No, not as-is. The direction is maintainable and closer to the right product boundary than the alias-only path on main, but the branch should preserve the injected fetch seam and key ADC token caching by credential identity before it is a safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread the stream fetch into ADC token refresh:** `extensions/google/transport-stream.ts:287`
  - body: The branch now calls `resolveGoogleVertexAdcToken()` without passing the stream option's `fetch`, while current transport tests and proxy-aware callers provide a custom fetch for the OAuth token exchange. This makes Vertex ADC refresh use global `fetch`, bypassing the configured test/proxy seam and causing the existing token-refresh path to hit real OAuth instead of the supplied fetch implementation.
  - confidence: 0.88
- **[P2] Key the ADC token cache by credential source:** `extensions/google/vertex-adc.ts:151-152`
  - body: `resolveGoogleVertexAdcToken()` returns any unexpired cached token before reading the current ADC path or env. If `GOOGLE_APPLICATION_CREDENTIALS`, project env, or the underlying credential changes in the running process, later Vertex calls can reuse a token minted for a different account until expiry, causing cross-project auth failures or wrong-credential use.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR touches service-account ADC signing and token caching, and the unkeyed cache is a concrete auth-isolation concern.

Concerns:

- **[medium] Unkeyed ADC token cache can reuse prior credentials:** `extensions/google/vertex-adc.ts:151`
  - body: The new cache returns an unexpired token without checking the current credential path, refresh token, service account identity, or project env. In a long-running gateway where credentials rotate or agent/profile env changes, Vertex requests can be authorized with a previous account's token.
  - confidence: 0.82
