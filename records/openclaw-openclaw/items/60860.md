## Summary

Keep open. Current main now has partial Google Vertex support for authorized_user ADC, but it does not fully cover this PR's dedicated runtime provider, service-account ADC token minting, ADC project fallback, or exact provider ownership. The PR also still needs normal maintainer review because it overlaps recent main work and introduces concrete auth-cache/project-resolution issues.

## What This Changes

The PR adds a dedicated google-vertex provider path in the Google plugin with ADC token exchange, Vertex project/location URL construction, a small Gemini catalog, manifest auth metadata, and focused Vertex ADC/region/catalog tests.

## Best Possible Solution

Rebase or rework the PR against current main, keep the shipped authorized_user path that is already present, decide explicitly whether bundled Google Vertex should gain service-account ADC and exact runtime provider ownership, fix the cache and project-env issues, then land with focused auth/routing tests and maintainer security review or close it as superseded by a narrower maintainer implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Scope cached Vertex tokens to the ADC source:** `extensions/google/vertex-adc.ts:166-167`
  - body: `resolveGoogleVertexAdcToken` returns `cachedToken` before reading the current env or credentials path. In a long-running gateway, changing `GOOGLE_APPLICATION_CREDENTIALS` or project settings can make the next Vertex request reuse a Bearer token from the previous ADC file, sending work under the wrong GCP identity. Key the cache by credential path and credential identity/project, or invalidate it when the env source changes.
  - confidence: 0.87
- **[P2] Honor GCLOUD_PROJECT for Vertex routing:** `extensions/google/vertex-region.ts:86-88`
  - body: The current manifest and main transport accept `GCLOUD_PROJECT` as Google Vertex project evidence, but this resolver only checks `GOOGLE_CLOUD_PROJECT` and `GOOGLE_CLOUD_PROJECT_ID` before falling back to the ADC file. A setup with readable ADC, `GOOGLE_CLOUD_LOCATION`, and `GCLOUD_PROJECT` can be treated as configured yet fail to build the Vertex project URL unless the ADC file also carries a project.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds auth-token handling and has a concrete credential-isolation issue in the unscoped ADC token cache.

Concerns:

- **[medium] Vertex ADC token cache is not credential-scoped:** `extensions/google/vertex-adc.ts:166`
  - body: The PR caches a Bearer token globally without tying it to the ADC path, credential identity, refresh token, or project. If the gateway changes Google credential sources while the token is fresh, Vertex requests can be made with a stale service account or user token.
  - confidence: 0.86
