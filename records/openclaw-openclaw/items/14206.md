## Summary

Keep open. Current main still lacks a Feishu-owned outbound minimum-interval or pacing configuration, and the burst-prone reply, comment, message-tool, card, media, and streaming-card send paths still call Feishu APIs directly or only use unrelated block/streaming throttles. Related #13615 and #70879 are useful context, but they track broader cross-provider rate limiting and reactive retry/backoff rather than this proactive Feishu message-spacing request.

## What This Changes

Add a configurable Feishu channel minimum interval between outbound messages, ideally also influencing how long model responses are segmented into Feishu messages.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer resolved from top-level and per-account config, then apply it consistently around visible Feishu sends that can burst API calls. Keep broad cross-provider limit policy in #13615 and reactive retry/backoff behavior in #70879 unless maintainers intentionally expand the scope.
