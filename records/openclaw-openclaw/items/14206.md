## Summary

Keep open. Current main still lacks a Feishu outbound minimum-interval or pacing configuration, and the reply, comment, message-tool, media, and streaming-card delivery paths still call Feishu send APIs directly or only use unrelated block-streaming and typing/backoff controls. The related open trackers #13615 and #70879 are useful context, but they cover broader cross-provider rate limiting and reactive retry/backoff rather than this Feishu-specific proactive message spacing request.

## What This Changes

Request a configurable Feishu message rate limit or minimum interval that spaces outbound messages and ideally influences how long LLM responses are segmented into Feishu messages.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer resolved from top-level and per-account config, then apply it consistently around visible Feishu sends that can burst API calls. Keep broad cross-provider quota policy in #13615 and reactive retry/backoff behavior in #70879 unless maintainers intentionally expand the scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: send a Feishu reply or message-tool payload that chunks into multiple messages, and the code immediately iterates through sendMessageFeishu/sendStructuredCardFeishu/deliverCommentThreadText without any configured minimum interval.

## Solution Assessment

Yes. A Feishu plugin-owned configurable pacing gate is the narrow maintainable solution because the problem is Feishu-specific outbound delivery behavior; a core-wide limiter or retry-only fix would either overreach or fail to prevent the burst before Feishu rejects it.
