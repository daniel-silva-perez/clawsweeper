## Summary

This item remains valid on current main: Feishu still has no configurable outbound pacing or minimum-interval setting, and the reply, comment, message-tool, media, and streaming-card delivery paths can still issue consecutive Feishu API calls without a shared per-account pacing gate. The broader external API rate-limit tracker #13615 and Feishu retry-after-rate-limit tracker #70879 are related but do not supersede this narrower proactive Feishu send-spacing request.

## What This Changes

The issue requests a Feishu channel configuration that enforces a minimum interval between outbound messages and ideally influences how LLM responses are segmented before sending.

## Best Possible Solution

Add a Feishu-owned, default-off per-account outbound pacing layer resolved from top-level and account config, then wire it through every Feishu delivery path that can burst API calls. Document and expose the setting in the Feishu schema, plugin manifest, and public channel docs; leave broad cross-provider quotas to #13615 and retry-after-rate-limit recovery to #70879.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: configure a small Feishu textChunkLimit or send a long/message-tool/media response that splits into multiple Feishu sends, then inspect the current send loops to see consecutive Feishu API calls with no configured minimum interval.

## Solution Assessment

Yes. A Feishu plugin-owned pacing gate is the narrowest maintainable fix because the reported behavior is Feishu outbound delivery frequency; a core-wide limiter is broader than this issue, while retry-only handling would not prevent the initial burst.
