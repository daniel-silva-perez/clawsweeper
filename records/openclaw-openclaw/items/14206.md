## Summary

Keep open. Current main still lacks a Feishu outbound minimum-interval or pacing configuration, and the Feishu reply, comment, message-tool, media, and streaming-card paths still send directly or only throttle unrelated streaming updates. The related open trackers #13615 and #70879 are useful context, but they do not supersede this Feishu-specific proactive message-spacing request.

## What This Changes

Request a configurable minimum interval between outbound Feishu messages, ideally also affecting how long LLM responses are segmented into Feishu messages.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer resolved from top-level and per-account config, then apply it consistently around visible Feishu sends that can burst API calls. Keep broad cross-provider quota policy in #13615 and reactive retry/backoff behavior in #70879 unless maintainers intentionally expand the scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: send a Feishu reply or message-tool payload that chunks into multiple outbound messages, and the code immediately iterates through sendMessageFeishu, sendStructuredCardFeishu, sendMediaFeishu, or deliverCommentThreadText without a configured minimum interval.

## Solution Assessment

Yes. A Feishu plugin-owned configurable pacing gate is the narrow maintainable solution because the problem is Feishu-specific outbound delivery behavior; a core-wide limiter or retry-only fix would either overreach or fail to prevent bursts before Feishu rejects them.
