## Summary

Keep open. Current main still lacks a Feishu outbound pacing or minimum-interval setting, and the main Feishu reply, comment, message-tool, media, and streaming-card send paths still call Feishu APIs directly or only throttle unrelated streaming-card updates. The related broad rate-limit and Feishu retry/backoff trackers remain useful context, but they do not supersede this proactive Feishu message-spacing request.

## What This Changes

Add configurable Feishu outbound message rate limiting so bursty chunked LLM replies can be spaced out before Feishu rejects or temporarily bans API requests.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer resolved from top-level and per-account config, expose it in schema/manifest/docs, and apply it consistently around visible Feishu sends that can burst API calls. Keep broad cross-provider quota policy in #13615 and reactive retry/backoff handling in #70879 unless maintainers intentionally expand the scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: configure a small Feishu textChunkLimit or send a long/message-tool payload that splits into multiple sends, then the code immediately walks through sendMessageFeishu, sendStructuredCardFeishu, sendMediaFeishu, or deliverCommentThreadText without any configured minimum interval.

## Solution Assessment

Yes. A Feishu plugin-owned configurable pacing gate is the narrow maintainable solution because the reported failure is Feishu outbound delivery behavior; a core-wide limiter would be broader than needed, and retry-only handling would not prevent the initial burst.
