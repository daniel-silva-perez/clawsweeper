## Summary

Keep open. Current main still lacks a Feishu outbound minimum-interval/pacing configuration, and the relevant reply, comment, message-tool, media, and streaming-card send paths still call Feishu APIs directly or only throttle unrelated streaming-card updates. The broader #13615 and reactive retry/backoff #70879 trackers are related context, but they do not supersede this Feishu-specific proactive message-spacing request.

## What This Changes

The issue requests configurable Feishu message rate limiting, ideally with a minimum interval between outbound messages and behavior that also affects response chunking.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer resolved from top-level and per-account config, expose it through schema, manifest, and Feishu docs, and apply it consistently around visible Feishu sends that can burst API calls. Keep broad cross-provider quota policy in #13615 and reactive retry/backoff handling in #70879 unless maintainers intentionally expand this scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: configure a small Feishu textChunkLimit or send a long/message-tool/media payload that splits into multiple sends, then the code immediately walks through sendMessageFeishu, sendStructuredCardFeishu, sendMediaFeishu, or deliverCommentThreadText without any configured minimum interval.

## Solution Assessment

Yes. A Feishu plugin-owned configurable pacing gate is the narrow maintainable solution because the reported failure is Feishu outbound delivery behavior; a core-wide limiter would be broader than needed, and retry-only handling would not prevent the initial burst.
