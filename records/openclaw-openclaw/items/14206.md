## Summary

Keep open. Current main still lacks a Feishu outbound minimum-interval or pacing configuration, and the direct reply, comment, message-tool, media, and streaming-card send paths remain unpaced aside from unrelated webhook ingress guards, typing backoff, and streaming-card update coalescing.

## What This Changes

Add configurable message rate limiting for the Feishu channel so bursty chunked or multi-part outbound responses do not trip Feishu API limits.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, resolved from top-level and per-account config, with a default-off minimum interval. Wire it through all visible Feishu sends that can burst API calls, expose it in schema, manifest, and Feishu docs, and leave broad cross-provider quotas to #13615 and retry-after-failure behavior to #70879.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: configure a small Feishu textChunkLimit or send a long/message-tool/media payload that splits into multiple outbound sends, then the code immediately walks those chunks through sendMessageFeishu, sendStructuredCardFeishu, sendMediaFeishu, or deliverCommentThreadText with no configured minimum interval.

## Solution Assessment

Yes. A Feishu plugin-owned configurable pacing gate is the narrow maintainable direction because the reported failure is Feishu outbound delivery behavior; a core-wide limiter is broader than this issue, and retry-only handling would not prevent the initial burst.
