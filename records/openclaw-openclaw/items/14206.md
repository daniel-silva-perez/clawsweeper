## Summary

Keep open. Current main still does not provide a Feishu outbound minimum-interval or pacing setting, and the central reply, comment, message-tool, media, and streaming-card send paths still call Feishu APIs without a shared per-account outbound pacing layer. Related #13615 and #70879 are useful sibling trackers, but they do not supersede this narrower proactive Feishu pacing request.

## What This Changes

The issue requests configurable Feishu message rate limiting, including a minimum interval between outbound messages and ideally behavior that reduces or paces segmented LLM replies.

## Best Possible Solution

Add a Feishu-owned outbound pacing layer that resolves top-level and per-account configuration, gates every Feishu API path that can emit visible messages in bursts, documents the setting in the Feishu channel docs and manifest/schema, and keeps the broader cross-provider rate-limit and Feishu retry/backoff trackers separate unless maintainers explicitly merge those scopes.

## Repair Work Prompt

Implement proactive outbound message pacing for the Feishu plugin for this issue. Add a Feishu-owned per-account pacing helper and config, likely something like `outboundMinIntervalMs` or `messageMinIntervalMs`, resolved from top-level `channels.feishu` and `channels.feishu.accounts.<id>` with a conservative default. Apply it consistently before visible Feishu outbound API calls that can burst: normal chunked text replies, card replies, comment-thread replies, message-tool/direct outbound sends, media message sends after uploads, structured/presentation card sends, and streaming-card create/send/final-visible paths as appropriate. Keep this scoped to proactive Feishu pacing; do not implement the broader cross-provider rate-limit architecture from #13615 or the retry/backoff behavior from #70879 unless a tiny local seam is necessary. Update Feishu config schema, manifest metadata, docs, and focused tests. Add a single-line changelog entry because this is user-facing config/behavior, crediting @immars. Do not move Feishu-specific behavior into core.
