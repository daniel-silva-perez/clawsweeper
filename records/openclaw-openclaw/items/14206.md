## Summary

Keep open. Current `main` still has no Feishu-owned outbound pacing or minimum-interval configuration, and the visible reply, comment, message-tool, card, media, and streaming-card paths can still make consecutive Feishu API calls without a shared per-account pacing gate. The broader open rate-limit tracker #13615 and Feishu retry/backoff tracker #70879 are related, but they do not supersede this proactive Feishu message-spacing request.

## What This Changes

Add configurable outbound message rate limiting for the Feishu channel, ideally affecting both send cadence and how LLM responses are segmented into Feishu messages.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, resolved from top-level and per-account config, and apply it consistently around burst-prone visible outbound sends. Keep broad cross-provider rate-limit policy and retry-after-error recovery in their existing trackers unless maintainers intentionally expand scope.

## Repair Work Prompt

Implement configurable Feishu outbound message pacing for openclaw/openclaw. The observable issue is that long or segmented Feishu replies can emit many Feishu API calls back-to-back and trigger server-side flow control. Keep the fix Feishu-owned: add a conservative per-account outbound pacing/minimum-interval config resolved from `channels.feishu` and `channels.feishu.accounts.<id>`, expose it in `extensions/feishu/src/config-schema.ts`, `extensions/feishu/openclaw.plugin.json`, and `docs/channels/feishu.md`, and apply it consistently around visible outbound Feishu sends that can burst: reply dispatcher chunk sends, comment dispatcher chunk sends, message-tool/outbound text/card/media sends, media upload/send finalization where appropriate, and streaming-card start/update/close only where it affects Feishu-visible API cadence. Do not move this into core, do not implement the broad cross-provider tracker #13615, and do not fold in retry/backoff-after-error behavior from #70879 unless it is strictly necessary for tests. Add focused Vitest coverage for config parsing/merge and for pacing calls across reply, comment, outbound, send/media, and streaming-card paths. Add a user-facing changelog entry with reporter credit; use plugin terminology, not `extension`, in docs/changelog.
