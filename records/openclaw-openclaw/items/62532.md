## Summary

Keep open. Current main does not implement this PR's central change: `createOperatorApprovalsGatewayClient` still builds `GatewayClient` without resolving or passing any TLS fingerprint, while the regular gateway call path already has local/remote TLS fingerprint handling and `GatewayClient` enforces the option when present. The PR is still a relevant implementation candidate, but it touches a high-trust approval/TLS path and has review feedback about the remote-mode fallback case that should be resolved before landing.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land a focused fix in `src/gateway/operator-approvals-client.ts` that reuses or mirrors the established gateway TLS fingerprint resolution semantics, with targeted tests in `src/gateway/operator-approvals-client.tls.test.ts` covering local WSS, remote env/CLI override behavior, and the missing-remote-URL fallback case before merging.

## Risks / Open Questions

- Security-sensitive path: this PR changes certificate pinning behavior for operator approval clients, which are used for high-trust exec/plugin approval flows.
- The PR should address the existing review concern about `gateway.mode=remote` with missing `gateway.remote.url`, so a remote fingerprint is not pinned against a local fallback gateway certificate.
- Because current main lacks focused operator approvals TLS tests, landing should include coverage for local WSS, env URL override, CLI URL override, and remote-misconfigured fallback behavior.
