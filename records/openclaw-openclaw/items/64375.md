## Summary

Keep this PR open. Current main still has the `at`-only `deleteAfterRun` guard, while the linked bug remains open and this PR is a narrow implementation candidate with targeted regression coverage. The security pass found no unrelated workflow, dependency, lockfile, release, script, or secret-handling changes in the PR context; the open question is maintainer product/security judgment about allowing recurring jobs to self-delete.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for #63770. Review or rebase it, decide whether recurring `deleteAfterRun` is intended behavior, and land this guard change or an equivalent fix with regression coverage for successful `every` and `cron` deletion plus failed/skipped retention. Treat durable audit, tombstone, or run-log hardening as a separate cross-schedule follow-up if maintainers want stronger guarantees.
