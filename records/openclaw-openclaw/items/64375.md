## Summary

Keep this PR open. Current main still has the `at`-only `deleteAfterRun` guard, while the linked bug remains open and the PR is a narrow implementation candidate with relevant regression coverage. The security-sensitive pass found no unrelated CI, dependency, lockfile, script, release, or secret-handling changes in the PR context; the remaining question is maintainer product/security judgment about recurring jobs being allowed to self-delete.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for #63770. Review or rebase it, decide whether recurring `deleteAfterRun` is intended product behavior, and land this guard change or an equivalent with regression coverage for successful `every` and `cron` deletion plus failed/skipped retention. Treat durable audit, tombstone, or run-log hardening as a separate cross-schedule follow-up if maintainers want that stronger guarantee.

- Maintainers still need product judgment on whether `deleteAfterRun` should officially apply to recurring `every` and `cron` schedules or remain one-shot-oriented in CLI/docs/UI copy.

- The PR is an implementation candidate, not shipped behavior: current main at `5828dcdb05aad587bff831467e7fa51f73312ff8` still contains the old guard.
