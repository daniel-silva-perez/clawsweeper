## Summary

Keep this issue open. Current main still only expands loopback IPv4 to an IPv6 loopback listener; LAN/container exposure remains `0.0.0.0`, tailnet bind selection still prefers IPv4, and `customBindHost` still rejects IPv6 literals, so the reported dual-stack Gateway listening gap remains real core work.

## Best Possible Solution

Keep the issue open and implement this as a core Gateway networking change. Extend `resolveGatewayBindHost`/`resolveGatewayListenHosts` and the config contract so LAN/container auto/tailnet/custom can intentionally support IPv6 or dual-stack binding; add regression coverage for `::1`, `::`, LAN/container behavior, and IPv6 custom hosts; update gateway configuration/security docs and status/diagnostic messaging.
