## Summary

Keep this issue open. Current main has a partial IPv6 improvement for loopback only, but the reported dual-stack/LAN listening problem remains: `lan` and container `auto` still resolve to `0.0.0.0`, `tailnet` still prefers IPv4, and `customBindHost` rejects IPv6 literals.

## Best Possible Solution

Keep the issue open and implement IPv6 as a core Gateway networking change. The likely path is to extend `resolveGatewayBindHost`/`resolveGatewayListenHosts` and the config contract so LAN/auto can bind IPv4 and IPv6 intentionally, allow validated IPv6 custom hosts where appropriate, add regression coverage for `::1` and non-loopback dual-stack behavior, and update the gateway configuration docs.

## Evidence
