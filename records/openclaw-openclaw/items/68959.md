## Summary

Keep this issue open. Current main still renders a combined chat session selector grouped by agent, but it does not expose a separate Control UI agent switcher backed by all configured agents, so a configured agent with no existing session row is still not directly selectable from chat.

## What This Changes

The issue requests a Control UI chat agent switcher that lists configured agents, switches between them, and can start or target a chat session for any configured agent.

## Best Possible Solution

Add a first-class chat Agent selector backed by agents.list, use it to switch to that agent's most recent session or buildAgentMainSessionKey({ agentId }) when none exists, keep a separate session picker filtered to the selected agent, and preserve the existing model/thinking sessions.patch behavior. Coordinate that with #64121 and the open #68926 implementation work instead of treating the current grouped session dropdown as sufficient.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with agents.list containing a non-default agent that has no session row, resolveSessionOptionGroups will not add that agent because it only adds rows from sessions.list plus the current sessionKey, and the chat header/mobile UI render only that combined session select.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: a configured-agent selector plus an agent-scoped session selector. The current main implementation is only a grouped session picker, while #68926 points in the right direction but needs rebase and preservation of current thinking/model behavior before merge.
