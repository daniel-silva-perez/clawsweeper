## Summary

Keep this issue open. Current main still has the same bootstrap failure shape: clean workspaces seed BOOTSTRAP.md and docs expect the first run to complete it, but the embedded runner strips BOOTSTRAP.md from injected Project Context and relies on model tool-calling/read access. The only post-report local history I found references this issue in an inter-session prompt fix, but does not change the bootstrap read/fallback path.

## Best Possible Solution

Keep this issue open. Reproduce the clean-install Control UI path on current main with qwen3.5/Ollama or another affected local model, then fix the first-run bootstrap path with a safe fallback or clear diagnostic when model tool-calling/read access is unavailable or unreliable. Add regression coverage for bootstrap-pending routing, BOOTSTRAP.md stripping, and the fallback behavior.
