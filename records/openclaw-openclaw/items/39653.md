## Summary

Keep this PR open. Current main still accepts arbitrary non-IP `ws://` hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, and several gateway/onboarding tests still assert that behavior. The PR is a narrow security hardening candidate; the remaining maintainer work is to decide whether to land the fail-closed hostname behavior or design explicit private-DNS support.

## Best Possible Solution

Keep this PR open for maintainer security review. The best path is to land or adapt the narrow validator change so the break-glass flag accepts loopback/private IP literals but rejects unresolved hostnames, then update the gateway net, client, connection-details, and onboarding tests to match the chosen contract. If private DNS hostname support is required, track it separately with an explicit allowlist or DNS-resolution design.

## Risks / Open Questions

- Current main still permits arbitrary non-IP `ws://` hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, which can expose credentials and conversation data over plaintext if a remote URL or DNS target is user-influenced.
