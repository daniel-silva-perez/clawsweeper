## Summary

Keep this PR open. Current main still accepts arbitrary non-IP `ws://` hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, tests still assert that behavior, and the related bug report/prior PR are closed without a merged fix. The PR is a narrow security hardening candidate, but it likely needs rebasing/adaptation because downstream gateway client, connection-details, and onboarding tests on current main also encode hostname acceptance.

## Best Possible Solution

Keep this PR open for maintainer security review. The best path is to merge or adapt the narrow validator change so `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` accepts loopback/private IP literals but rejects unresolved hostnames, then update the gateway net, client, connection-details, and onboarding tests to match the intended contract. If private DNS hostname support is desired, track it separately with an explicit allowlist or DNS-resolution design.
