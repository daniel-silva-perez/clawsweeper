## Summary

Keep this PR open. Current main at 69fb7455c6223682a45dd8b8fb720bc3231bfa43 still accepts arbitrary non-IP `ws://` hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, and tests still assert that behavior. The linked issue #37177 is closed stale and the prior similar PR #37883 was closed unmerged, so this PR is not implemented or superseded.

## Best Possible Solution

Keep this PR open for maintainer security review. The best path is to merge or adapt the narrow validator change so `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` accepts loopback/private IP literals but rejects unresolved hostnames, then update the gateway net, client, connection-details, and onboarding tests to match the intended contract. If private DNS hostname support is desired, track that separately with an explicit security design.
