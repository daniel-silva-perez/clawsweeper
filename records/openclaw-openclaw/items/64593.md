## Summary

Keep open. Current main still matches the report: the bundled browser plugin launches and tracks only the direct Chromium `ChildProcess`, and the stop/restart paths do not reap or supervise Chromium/crashpad descendants after normal browser use. The closed startup-failure item, closed browser process-group PR, open gateway-orphan PR, and broader sandbox PID-1 item are related context but do not supersede this specific normal-exit managed-browser leak.

## What This Changes

The issue asks OpenClaw to reap managed Chromium and chrome_crashpad child processes after normal browser exits so PID-1 container gateways do not exhaust process slots.

## Best Possible Solution

Fix the bundled browser plugin lifecycle so managed Chromium is container-safe across normal exits, explicit stops, restarts, and startup retry cleanup. The desired end state is a supervised helper/init/subreaper or other safe tree/reaping strategy, focused unit coverage, a Docker/browser regression that repeats managed Chromium exits and checks for no defunct chromium or chrome_crashpad descendants, and docs for any required container option or workaround.

## Reproduction Assessment

Yes. The issue gives a high-confidence production reproduction and diagnostic path: run browser-heavy OpenClaw container gateways as PID 1, then count defunct chromium/chrome_crashpad children whose PPID is the OpenClaw process; this review did not run that live scenario, but current source still matches the reported lifecycle gap.

## Solution Assessment

Partly. The requested browser-service boundary is correct, but waiting on only the direct Chromium process is unlikely to cover crashpad/adopted descendants; the safer solution is a scoped browser lifecycle supervisor or subreaper strategy rather than a naive process-group kill.

## Repair Work Prompt

Fix the managed browser zombie-process leak described in https://github.com/openclaw/openclaw/issues/64593. The observable bug is that long-running Linux/Docker gateways running as PID 1 can accumulate defunct Chromium and chrome_crashpad descendants after normal browser use until child_process.spawn starts failing with EAGAIN. Keep the implementation scoped to the bundled browser plugin/runtime lifecycle unless a very small generic process-runtime seam is unavoidable. Start from extensions/browser/src/browser/chrome.ts, extensions/browser/src/browser/server-context.availability.ts, and extensions/browser/src/browser/server-lifecycle.ts. Do not use a naive Unix process-group kill that can signal the Gateway; account for the warning in src/process/kill-tree.ts about shared process groups. Prefer a supervised helper/init/subreaper or another safe tree/reaping strategy that handles normal exits, explicit openclaw browser stop, restart/retry cleanup, and Gateway teardown without broad core policy changes. Add focused unit tests plus a Docker/browser smoke or equivalent regression that repeatedly starts/exits managed Chromium and asserts no defunct chromium or chrome_crashpad descendants remain under the OpenClaw process. Update browser/Linux container docs if the fix requires or recommends a supported init/subreaper/container option. Add a user-facing changelog fix entry crediting Thanks @theandyroberts. Do not change provider surfaces, global process-kill semantics, unrelated sandbox PID-1 handling, or unrelated browser behavior. Related context: #8611 is startup failure only, #41750 is an older browser stale-port report that may share lifecycle symptoms, #38247 was a closed unmerged process-group attempt, #47776 is adjacent gateway-orphan cleanup, and #68691/#74083 are broader sandbox init/reaping work rather than this managed-browser normal-exit leak.
