## Summary

Keep this issue open. The report is recent, concrete, and still matches current main: the bundled browser plugin launches a managed Chromium process, tracks only the direct ChildProcess, and stops only that direct process. The current startup retry/singleton-lock cleanup does not cover the reported normal-exit/container PID-1 descendant zombie accumulation involving Chromium and chrome_crashpad processes.

## Best Possible Solution

Fix this in the bundled browser plugin/runtime lifecycle, not as stale cleanup. The maintainer path should make managed Chromium container-safe across normal exits, stop/restart, and startup retry cleanup, likely through a supervised helper/init/subreaper or a safe process-group/tree strategy that cannot signal the Gateway. Add focused unit coverage plus a Docker/browser smoke that repeats managed Chromium exits and verifies no defunct chromium or chrome_crashpad descendants, then document any supported container requirement or workaround.

## Risks / Open Questions

- Closing now would drop a plausible production reliability bug: PID exhaustion can break all child_process.spawn-backed tools in long-running container gateways.
- A fix needs careful process-lifecycle design. Existing process-tree comments warn that naive Unix process-group killing can terminate the Gateway when a child shares the Gateway process group.
- The remaining verification gap is a Docker/browser regression that repeatedly starts and exits managed Chromium and confirms no defunct chromium or chrome_crashpad descendants remain.
