## Summary

Keep this issue open. The report is recent, concrete, and still matches current main: managed Chromium is launched as a direct ChildProcess, runtime state tracks only that direct process, and the normal stop/exit paths do not supervise or reap Chromium descendants such as crashpad processes in a PID-1 container. The closed startup-failure issue #8611 and the current startup retry cleanup do not cover the reported normal-exit accumulation path.

## Best Possible Solution

Fix this in the bundled browser plugin/runtime lifecycle. The best path is to make managed Chromium container-safe across normal exits, stop/restart, and startup retry cleanup, likely through a supervised helper/init/subreaper or a safe process-group/tree strategy that cannot signal the Gateway. Add focused unit coverage plus a Docker/browser smoke that repeatedly starts and exits managed Chromium and verifies no defunct chromium or chrome_crashpad descendants remain; document any supported container requirement or workaround.

## Risks / Open Questions

- A live Docker reproduction was not run because this review was read-only, so the exact descendant-zombie mechanism still needs an automated regression test.
- Naive process-group killing is risky here; existing process-tree code documents that group-killing a non-detached child can signal the Gateway itself.
- Leaving the issue unresolved risks long-running container gateways exhausting PID limits and breaking all spawn-backed tools.
