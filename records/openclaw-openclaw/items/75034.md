## Summary

Keep this PR open for maintainer/security review, but do not merge it as-is. The reported gap is real for the agent-facing gateway tool, yet the proposed fix expands the model-to-operator config-write trust boundary to a group/channel visibility safety setting; direct admin `config.patch` already validates and writes config separately from this agent allowlist.

## What This Changes

The PR adds `messages.visibleReplies` and `messages.groupChat.visibleReplies` to the agent gateway tool's `config.apply`/`config.patch` mutation allowlist.

## Best Possible Solution

Keep direct operator/admin configuration of visible reply modes supported, but require an explicit maintainer decision before adding these privacy-affecting delivery settings to the agent gateway tool allowlist. If accepted, add focused regression coverage for both visible-reply paths and update the security/tool docs to name the expanded allowlist.

## Reproduction Assessment

Yes. A code-level reproduction is clear on current main: patching `messages.groupChat.visibleReplies` through the agent gateway tool produces a changed path outside `ALLOWED_GATEWAY_CONFIG_PATHS`, so the guard rejects it as protected before forwarding `config.patch`.

## Solution Assessment

No. The diff is mechanically narrow, but it lets a model persistently flip group/channel final replies from message-tool-only to automatic; the safer path is direct operator/admin config writes or a maintainer-approved, documented expansion of the agent allowlist.
