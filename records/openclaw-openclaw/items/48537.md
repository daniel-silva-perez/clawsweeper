## Summary

Current main still appears to have the shared Telegram polling-cycle abort behavior this PR targets. `createTelegramBotCore` applies `fetchAbortSignal` to every wrapped Telegram fetch, while `TelegramPollingSession` passes one per-cycle controller into bot creation and aborts it on polling restart/stop paths. Later liveness work reduced false restarts, but it did not add a `getUpdates`-only abort boundary, and a recent production comment reports the same abort bleed on recent releases.

## Best Possible Solution

Keep this PR open for maintainer follow-up or a contributor rebase. The best current fix is a Telegram-plugin change on current `bot-core.ts`/`polling-session.ts` that applies polling restart/shutdown aborts to `getUpdates` only while preserving hard cleanup for stuck long polls, with regression tests proving `getUpdates` still aborts and `sendMessage`/`sendChatAction` survive polling recovery.
