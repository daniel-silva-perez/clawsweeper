## Summary

Keep this PR open. Current main still wires the polling session's shared fetchAbortSignal into every Telegram client fetch, and polling restarts still abort that shared signal. Recent liveness work reduces some false restarts during non-getUpdates activity, but it does not implement the PR's intended behavior: preserving in-flight sendMessage/sendChatAction requests when a polling restart aborts getUpdates.

## Best Possible Solution

Keep the PR open and ask for an update against current main. The best fix is a Telegram-plugin change that scopes polling-restart aborts to getUpdates, or otherwise separates polling-restart abort from full shutdown semantics, with focused regression tests proving getUpdates is aborted during restart while sendMessage/sendChatAction survive.

## Risks / Open Questions

- The PR is stale against current main because the relevant Telegram bot creation logic now lives in bot-core.ts and polling restart orchestration remains in polling-session.ts.
- A rebased fix must preserve hard getUpdates timeout/restart cleanup and true gateway shutdown behavior while separating polling-restart aborts from ordinary outbound sends.
- If the fix uses grammY API middleware, it should include an integration-level assertion that getUpdates abort reaches the fetch layer through apiThrottler and that sendMessage/sendChatAction are not aborted by the polling restart path.
