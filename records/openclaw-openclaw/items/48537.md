## Summary

Keep open. Current main and the latest release still route the Telegram polling-cycle abort signal through the shared client fetch wrapper for all Telegram API methods, so a polling restart can still abort outbound sends. The PR direction is valid and supported by production comments, but the branch is stale against the current bot-core refactor and needs a rebase or narrow replacement.

## What This Changes

The PR scopes Telegram polling shutdown abort handling to getUpdates-only middleware and updates abort regression tests so sendMessage is not canceled by polling restart cleanup.

## Best Possible Solution

Land a current-main Telegram plugin fix that separates polling-restart getUpdates cancellation from outbound Telegram API sends, with regression coverage proving getUpdates still aborts on restart while sendMessage and sendChatAction survive polling recovery.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: current polling-session aborts the shared fetchAbortSignal during stall restart, bot-core forwards that signal into every fetch, and the existing fetch-abort test still asserts broad abort behavior.

## Solution Assessment

No, not as-is. The concept is the right direction, but the branch needs to be moved from the old bot.ts implementation into the current bot-core/polling-session structure before it can solve the live bug.
