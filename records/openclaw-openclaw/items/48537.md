## Summary

Current main still has the behavior this PR is trying to change: the Telegram polling-cycle abort signal is passed into the shared client fetch wrapper, and restart paths still abort that controller. Later liveness work reduces some restart cases, but it does not create a getUpdates-only abort boundary for polling restarts, so the PR remains useful after a rebase.

## Best Possible Solution

Keep this PR open for maintainer follow-up or a contributor rebase. The best fix is a Telegram-plugin change in the current bot-core/polling-session structure that scopes polling restart aborts to getUpdates only, with focused tests proving getUpdates is still aborted on restart/shutdown while non-polling outbound sends survive polling recovery.
