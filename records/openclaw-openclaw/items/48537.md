## Summary

Current main still has the behavior this PR is trying to fix. The Telegram polling session creates one shared abort controller for a polling bot, passes that signal into bot creation, and bot-core forwards that signal into every Telegram client fetch rather than scoping it to getUpdates. Polling recovery paths still abort that same controller, so a polling restart can still cancel unrelated outbound Telegram API sends. The PR remains useful, but it needs a rebase into the current bot-core/polling-session structure and should address the existing review concerns.

## Best Possible Solution

Keep this PR open for maintainer follow-up or contributor rebase. The best path is a Telegram plugin change in the current bot-core/polling-session structure that scopes polling restart aborts to getUpdates only, with focused regression coverage proving getUpdates is still aborted on restart/shutdown while non-polling outbound sends survive polling recovery. Address the existing review comments during the rebase.

## Risks / Open Questions

- The PR patch targets the older bot.ts implementation shape; current main delegates bot construction through extensions/telegram/src/bot-core.ts, so the fix needs a rebase rather than closure.
- A rebased middleware-based fix should include integration proof that getUpdates aborts still reach the fetch layer through @grammyjs/transformer-throttler, while sendMessage/sendChatAction do not inherit polling-restart aborts.
- The implementation must separate polling-restart cleanup from true gateway shutdown and preserve hard getUpdates timeout behavior.
