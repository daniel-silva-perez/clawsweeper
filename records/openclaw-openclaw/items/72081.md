## Summary

Keep open. Current main still strands headless/WSL users when the token-bearing dashboard URL cannot be copied or opened automatically, and two open PRs already reference this issue with closing syntax, so the issue should remain open until one fix lands.

## What This Changes

The report asks `openclaw dashboard` to provide an authentication path when the printed dashboard URL does not include the gateway token in a WSL/headless run.

## Best Possible Solution

Review and land one narrow CVE-safe dashboard fix that emits a manual authentication hint only when clipboard and browser delivery both fail, keeps the token value and `#token=` out of runtime logs, and adds regression coverage for the combined-failure and `--no-open` cases.

## Reproduction Assessment

Yes. Current-main code inspection provides a high-confidence reproduction path: configure a literal gateway token, make clipboard copy fail, and run in a headless/SSH/WSL path where browser open is unavailable; the command logs the bare URL and SSH/no-open hint without a manual token recovery hint.

## Solution Assessment

Yes, the direction in the linked PRs is the right narrow fix: keep the secure non-tokenized terminal log, but add a recovery hint naming `OPENCLAW_GATEWAY_TOKEN` or `gateway.auth.token` and the fragment key when automatic delivery fails.
