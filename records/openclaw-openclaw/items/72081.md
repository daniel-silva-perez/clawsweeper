## Summary

Keep open. Current main still preserves dashboard token redaction but does not give stranded headless/WSL users a manual token recovery hint when both clipboard and browser delivery fail, and the provided context shows #72802 is an open PR with closing syntax for this issue.

## What This Changes

The issue reports that `openclaw dashboard` prints a bare dashboard URL without token guidance when tokenized clipboard/browser delivery is unavailable.

## Best Possible Solution

Review and land #72802 or an equivalent narrow dashboard command change that adds a manual token recovery hint only when automatic tokenized delivery fails, while keeping runtime logs non-tokenized and covered by regression tests.

## Reproduction Assessment

Yes. The deterministic source-level path is a literal `gateway.auth.token` with `copyToClipboard` returning false and browser open unavailable, or `--no-open` plus clipboard failure; current main then logs the bare URL/fallback hint without telling the user where to retrieve or supply the token.

## Solution Assessment

Yes. The best fix direction is the narrow condition represented by #72802: preserve non-tokenized logs, but emit a CVE-safe manual authentication hint when `!copied && !opened && includeTokenInUrl`.
