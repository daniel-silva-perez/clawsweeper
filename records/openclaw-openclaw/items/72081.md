## Summary

Keep open. Current main still preserves dashboard token redaction but does not give a stranded WSL/headless user a manual token recovery hint when tokenized delivery via clipboard and browser is unavailable; the linked #72802 remains the open implementation candidate, while #73227 was closed unmerged.

## What This Changes

The issue reports that `openclaw dashboard` prints a non-tokenized dashboard URL on WSL2, leaving gateway token authentication unavailable when automatic token delivery fails.

## Best Possible Solution

Review and land #72802, or an equivalent narrow dashboard command change, so automatic token delivery stays redacted while failed clipboard/browser delivery prints a safe manual recovery hint that points to `OPENCLAW_GATEWAY_TOKEN` or `gateway.auth.token` without logging the token value.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is a plaintext `gateway.auth.token` with clipboard copy returning false and either browser open unavailable or `--no-open`; current main then logs only the bare URL/fallback hint without telling the user where to retrieve or supply the token.

## Solution Assessment

Yes. The best fix direction is narrow and owner-local: keep the tokenized URL limited to clipboard/browser paths, preserve log redaction, and add manual auth guidance only when automatic delivery fails.
