## Summary

Keep this PR open. Current main still enumerates raw iMessage account ids for gateway startup, and this PR filters only `listEnabledIMessageAccounts()`, which is not the monitor startup enumeration path. The PR also has concrete blocking review issues around disabled named accounts and normalized/effective default runtime lookup, while the linked bug and alternate implementation PR remain open.

## What This Changes

The PR adds iMessage account filtering and tests intended to skip a fallback-only `default` account when named accounts exist while preserving `default` when it has distinct runtime fields.

## Best Possible Solution

Land one iMessage-owned fix that changes the account list actually used for gateway monitor startup, dedupes only fallback-only `default` watchers covered by an enabled named account on the same local backend, preserves genuinely distinct default runtimes using normalized/effective account config, and adds focused regression tests for the reported config, disabled-placeholder accounts, and distinct-runtime cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wire the filtered list into monitor startup:** `extensions/imessage/src/accounts.ts:89-92`
  - body: This filters `listEnabledIMessageAccounts()`, but gateway startup still calls the plugin config adapter's `listAccountIds`, which remains wired to the raw `listIMessageAccountIds` helper on main. A config with both `default` and a named account will still start both monitors, so the reported duplicate watcher bug is not fixed unless this filtered enumeration is used by startup or an equivalent startup dedupe is added.
  - confidence: 0.95
- **[P2] Compute named accounts after enabled resolution:** `extensions/imessage/src/accounts.ts:84-87`
  - body: `hasNamedAccounts` is computed from every configured id before resolving account enabled state. If a config keeps a disabled named account alongside an enabled fallback `default`, this drops `default` and the later enabled filter drops the named account too, leaving no iMessage monitor running.
  - confidence: 0.9
- **[P2] Resolve default runtime through merged account lookup:** `extensions/imessage/src/accounts.ts:68-78`
  - body: `hasDistinctDefaultIMessageRuntime` reads `accounts.default` directly, so it misses normalized keys such as `Default` and runtime fields inherited by the effective default account from channel-level config. Those valid distinct default runtimes can be incorrectly filtered out when a named account exists.
  - confidence: 0.86
