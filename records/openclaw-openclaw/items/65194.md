## Summary

Keep open. The reported iMessage duplicate-monitor bug is still real on current main, but this PR is not merge-ready because its new filtering helper is not used by the gateway startup account enumeration path and the patch introduces additional edge-case regressions around disabled named accounts and normalized default keys.

## What This Changes

The PR adds fallback-default suppression to `listEnabledIMessageAccounts()` and extends the iMessage account tests for fallback-only, distinct-runtime, and inherited-processing default-account configurations.

## Best Possible Solution

Land one iMessage-owned fix that changes the account list actually used for gateway monitor startup, dedupes only fallback-only `default` watchers covered by an enabled named account on the same backend, preserves distinct default runtimes, handles normalized account keys, and includes focused regression tests for reported, disabled-placeholder, and distinct-runtime configs.

## Reproduction Assessment

Yes. The issue has a concrete config shape, and the current source path deterministically enumerates both `default` and the named account before starting one monitor per id; this review did not run tests because the task is explicitly read-only.

## Solution Assessment

No. The proposed direction is narrow, but the implementation is not the best fix yet because it filters a helper that startup does not call and computes suppression before enabled/default-key normalization is fully respected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wire the filtered account list into startup:** `extensions/imessage/src/accounts.ts:80`
  - body: `listEnabledIMessageAccounts` now contains the fallback-default filtering, but gateway startup still calls `plugin.config.listAccountIds(cfg)`, and the iMessage adapter still wires that to `listIMessageAccountIds`. The reported config will therefore still start both `default` and the named account monitors unless the filtered/deduped list is used by startup.
  - confidence: 0.94
- **[P1] Ignore disabled named accounts before dropping default:** `extensions/imessage/src/accounts.ts:82-85`
  - body: `hasNamedAccounts` is computed from every configured id, before resolving whether those accounts are enabled. A disabled named account plus an enabled fallback `default` would cause the filter to drop `default`, then the later enabled filter drops the disabled account too, leaving no iMessage monitor running.
  - confidence: 0.9
- **[P2] Resolve default through normalized account lookup:** `extensions/imessage/src/accounts.ts:68`
  - body: `hasDistinctDefaultIMessageRuntime` reads `accounts.default` directly, while the rest of account resolution accepts normalized/case-insensitive keys. A config using `Default` with a distinct runtime can be misclassified as fallback-only and the monitor can be dropped when another named account exists.
  - confidence: 0.86
