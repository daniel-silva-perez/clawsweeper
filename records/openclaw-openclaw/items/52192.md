## Summary

This PR should stay open because current main still has the Feishu sendMedia duplicate-send path described in the PR: companion text is sent before the mediaUrl check, and the no-media fallback can send the same text again. The proposed change is targeted and security-neutral, but as an active contributor implementation PR for a valid channel bug it needs normal maintainer review rather than auto-closure or a replacement repair job.

## What This Changes

The PR updates the Feishu outbound sendMedia handler so the companion text send is guarded by mediaUrl, with the adjacent comment changed to say text is only sent before media when media is present.

## Best Possible Solution

Land the minimal mediaUrl guard in the Feishu outbound sendMedia handler and add or request a focused regression test proving that text plus no mediaUrl sends exactly one Feishu text message, while leaving broader Feishu upload-failure behavior for a separate follow-up if maintainers want it.
