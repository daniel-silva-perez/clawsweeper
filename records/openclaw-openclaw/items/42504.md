## Summary

Keep this issue open. Current main can render configured agent avatars through `/avatar/<agentId>` and documents a browser-local assistant avatar override, but it still does not provide the requested Agents-tab upload flow that stores image bytes under `workspace/avatars/` and updates `IDENTITY.md`. The earlier ClawSweeper comment reached the same conclusion, and this refreshed shell check against `17811480dae20f65b526454d75e971ae8424ec10` did not find a new implementation.

## Best Possible Solution

Implement a first-party Control UI agent-avatar upload path: add an admin-scoped typed gateway method or authenticated endpoint for image bytes and metadata, store validated files under the selected agent workspace `avatars/` directory, update `IDENTITY.md` through the existing identity helpers, wire a picker/click action into the Agents tab, refresh avatar metadata after upload, and add focused gateway/UI regression coverage. The related avatar-display work such as #17298 should remain separate because it solved rendering configured avatars, not uploading them from the Agents tab.

## Risks / Open Questions

- Adding this remains security-sensitive: validate MIME/type, extension, and size; generate safe filenames; reject traversal, symlink, and hardlink escapes; and write only inside the selected agent workspace `avatars/` directory.
- The generic `agents.files.set` text-file editor is not an adequate substitute for binary avatar upload because it writes UTF-8 strings and lacks avatar-specific validation, identity refresh, and metadata semantics.
