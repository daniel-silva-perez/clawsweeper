## Summary

Current main resolves the reported Claude CLI/chat versus PI/subagent transport split through the canonical `agentRuntime.id` runtime selector. The official Anthropic CLI setup now writes `agentRuntime.id: "claude-cli"`, the agent runner routes canonical Anthropic models with that runtime to the CLI path instead of PI, and `sessions_spawn` enters the same Gateway agent execution path for the child session. This shipped in `v2026.4.27`.

## What This Changes

Route subagent and PI-harness agent executions for Claude CLI-backed Anthropic configurations through the same Claude CLI transport used by chat, avoiding direct API billing on the same OAuth credential.

## Best Possible Solution

Keep the shipped `agentRuntime.id` design as the maintained fix: Claude CLI auth/setup should select canonical Anthropic models plus `agentRuntime.id: "claude-cli"`, while `cliBackends` remains the command/fallback backend configuration and doctor handles legacy `claude-cli/*` migrations.

## Reproduction Assessment

Yes. The original failure mode is reproducible conceptually from the affected config shape, and current-main code plus regression coverage gives a high-confidence check that canonical Anthropic models with `agentRuntime.id: "claude-cli"` now use the CLI path for child agent execution instead of PI/direct API.

## Solution Assessment

Yes. The shipped solution is narrower and more maintainable than teaching PI to infer billing-sensitive transport from OAuth credentials or bare `cliBackends`: it separates model provider from execution runtime and applies uniformly through the existing agent dispatch path.
