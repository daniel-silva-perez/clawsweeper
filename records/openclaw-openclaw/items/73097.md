## Summary

Keep open. Current main still supports Claude CLI routing through explicit `claude-cli/...` provider refs or `agentRuntime.id: "claude-cli"`, but it does not implement the issue's requested `cliBackends`-only inference for PI/subagent dispatch. The remaining choice is a maintainer transport and billing contract decision, not a cleanup-close candidate.

## What This Changes

The issue requests that PI, subagent, and raw model execution honor configured Claude CLI backends so Claude OAuth users do not split chat and subagent calls across CLI and direct Anthropic API transports.

## Best Possible Solution

Define one supported contract and make it consistent: either strengthen docs/doctor checks so Claude CLI primary usage requires `agentRuntime.id: "claude-cli"`, or add an explicit opt-in policy that routes eligible Anthropic PI/subagent/model paths through the configured Claude CLI backend with focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With `agents.defaults.model.primary` set to `anthropic/...`, `agents.defaults.cliBackends.claude-cli` configured, and no `agentRuntime.id`, current source keeps execution on the canonical provider and falls through to embedded PI.

## Solution Assessment

Unclear. Automatically inferring CLI transport from `cliBackends` alone may surprise existing canonical Anthropic configs; a maintainer should choose between explicit docs/doctor enforcement and an opt-in routing change.
