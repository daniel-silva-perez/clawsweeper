## Summary

Keep open. Current main supports Claude CLI execution when the selected provider is `claude-cli` or canonical `anthropic/*` is paired with `agentRuntime.id: "claude-cli"`, but it still does not infer CLI execution from `agents.defaults.cliBackends` alone. The remaining question is a transport and billing contract decision, not a cleanup-close candidate.

## What This Changes

The issue requests that PI/subagent dispatch honor configured Claude CLI backends so eligible Anthropic OAuth calls use the same CLI transport as chat instead of direct API billing.

## Best Possible Solution

Define one explicit contract and make it consistent: either add docs/doctor warnings for Claude CLI configs that set only `cliBackends` without `agentRuntime.id`, or add a deliberate opt-in runtime policy that routes eligible Anthropic PI/subagent/model paths through the configured Claude CLI backend with focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With `agents.defaults.model.primary` set to `anthropic/...`, `agents.defaults.cliBackends.claude-cli` configured, and no `agentRuntime.id`, current source keeps the execution provider as `anthropic` and uses embedded PI; a live billing-account reproduction is not needed to verify that branch.

## Solution Assessment

Unclear. Automatically inferring CLI transport from `cliBackends` alone may surprise existing configs because current docs use `agentRuntime.id` as the execution selector; a safer path may be an explicit opt-in or doctor warning, depending on maintainer intent.
