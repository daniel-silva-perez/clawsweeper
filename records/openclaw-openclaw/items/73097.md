## Summary

Keep open. Current main supports Claude CLI execution when the model provider is explicitly `claude-cli` or when `agentRuntime.id` selects `claude-cli`, but it still does not infer Claude CLI transport from `agents.defaults.cliBackends` alone for canonical `anthropic/*` PI or subagent dispatch. The remaining question is a maintainer-level transport and billing contract decision, not a conservative cleanup close.

## What This Changes

The issue requests that PI/subagent execution honor configured Claude CLI backends for OAuth-backed Anthropic credentials so spawned-agent and chat paths use the same transport.

## Best Possible Solution

Define one supported contract and make it consistent: either keep Claude CLI selection explicit through `agentRuntime.id: "claude-cli"` plus docs/doctor checks, or add a deliberate opt-in policy that lets eligible Anthropic PI/subagent/model paths route through a configured Claude CLI backend with regression coverage for auth forwarding, fallback behavior, and subagent spawn.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With canonical `anthropic/*`, a configured `agents.defaults.cliBackends.claude-cli`, and no `agentRuntime.id`, current source leaves `cliExecutionProvider` as `anthropic` and falls through to embedded PI for native subagent execution.

## Solution Assessment

Unclear. Inferring CLI transport from `cliBackends` alone is not obviously the safest fix because `cliBackends` is currently backend configuration, while `agentRuntime` is the documented executor-selection contract; a maintainer should choose explicit docs/doctor enforcement or an opt-in routing change.
