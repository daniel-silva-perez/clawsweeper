## Summary

Keep open. Current main has a supported Claude CLI route when the selected provider is `claude-cli` or when canonical `anthropic/*` is paired with `agentRuntime.id: "claude-cli"`, but the exact `cliBackends`-only routing contract described here is still not implemented for PI/subagent dispatch.

## What This Changes

The issue requests that PI/subagent model dispatch honor configured Claude CLI backends so spawned agents and chat use the same transport for CLI-backed OAuth credentials.

## Best Possible Solution

Define one explicit contract and make it consistent: either document and doctor that Claude CLI primary usage requires `agentRuntime.id: "claude-cli"` or legacy `claude-cli/...`, or add a deliberate opt-in that routes eligible Anthropic PI/subagent/raw model paths through the configured CLI backend with focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw routing behavior: with `agents.defaults.model.primary` set to `anthropic/...` and only `agents.defaults.cliBackends.claude-cli` configured, the inspected source path keeps the selected provider as `anthropic` and falls through to embedded PI. A live billing-account reproduction was not needed to verify that transport decision.

## Solution Assessment

Unclear. Automatically inferring CLI transport from `cliBackends` alone may not be the best fix because current docs separate provider-id backend configuration from explicit `agentRuntime.id` execution selection; a maintainer should choose between stronger docs/doctor checks and an explicit opt-in routing change.
