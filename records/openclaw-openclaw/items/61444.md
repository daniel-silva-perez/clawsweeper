## Summary

Keep this PR open. Current main still does not implement `openclaw agents import`, and there is no local history evidence of a merged/superseding implementation. The PR is a plausible core CLI feature candidate rather than ClawHub scope, but it still needs rework before merge because the proposed importer trusts archive metadata for workspace paths, can leave workspace files copied before config commit succeeds, still extracts the archive twice, and only tests early error paths.

## Best Possible Solution

Keep the PR open and ask for a focused rework: define a strict `agent.json` schema, ignore or constrain archive-provided workspace paths unless an explicit trusted override is added, validate all optional fields before use, avoid partial imports by staging/copying after a successful config plan with rollback or safe ordering, extract once, and add tests for successful import, workspace copy, overwrite/force/non-interactive behavior, reserved ids, malformed metadata, and config-write failure behavior. This belongs in core if accepted because it mutates built-in agent config and workspace state; it is not just a plugin-discovery or optional integration feature.

## Risks / Open Questions

- Merging the PR as-is could let an archive choose an arbitrary workspace destination via `agent.json.workspace`, then copy files there during import.
- The proposed implementation copies workspace files before the config write succeeds, so config hash conflicts or write failures can leave a partial import on disk.
- The PR still extracts the same archive twice, increasing I/O and leaving two extraction paths to keep behaviorally consistent.
- Tests do not yet cover the success path or the higher-risk overwrite, malformed metadata, and partial-import cases.
