## Summary

Keep open. Current main still routes the iOS chat sheet through the shared SwiftUI chat view, where the message list and composer are sibling views in the same vertical layout. The non-macOS composer uses a focused `TextEditor` with min/ideal/max height and focus-driven compaction, so input-bar height changes still participate in the main chat layout path. I did not find a bottom-inset-only or overlay-based fix, a matching changelog/release entry, or an iOS regression test for this freeze scenario.

## Best Possible Solution

Keep this issue open and fix it in the shared `OpenClawChatUI` layout used by iOS: decouple composer height changes from the message-list layout, likely by giving the scroll view a stable viewport and reserving/overlaying composer space via a measured bottom inset or equivalent. Add an iOS-focused regression smoke/performance test or manual profiling note that covers multiline composer growth and focus compaction with a populated message list.
