## Summary

Keep this issue open. Current main still parses each Control UI WebSocket message as a complete JSON frame and silently drops malformed frames on parse failure; I did not find a source-backed fix or regression test proving long Chinese/CJK streaming content is preserved across the reported UTF-8 boundary failure. Related reports cover adjacent provider/SSE/tool-call JSON parsing paths, but none clearly supersede this Control UI transport/display issue.

## Best Possible Solution

Keep the issue open and add focused coverage around the Control UI gateway path: send a long CJK chat delta/final event through the Gateway/WebSocket browser client path and assert the UI receives complete text without JSON.parse errors, stream loss, or truncation. If malformed partial-frame input is possible, fix it at the transport boundary with explicit buffering/reassembly or clear reconnect/history recovery diagnostics rather than only swallowing parse errors.
