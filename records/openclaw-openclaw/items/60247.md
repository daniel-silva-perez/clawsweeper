## Summary

Keep open. Current main still has no source-backed fix or targeted regression for the reported Control UI CJK streaming parse failure; the browser client continues to parse each WebSocket message independently and drops malformed frames, while related parse-error fixes cover adjacent provider/SSE/channel paths rather than this Control UI transport/display path.

## What This Changes

The issue requests reliable Control UI webchat streaming for long Chinese/CJK responses without JSON.parse failures when UTF-8 multi-byte content crosses a transport boundary.

## Best Possible Solution

Add deterministic coverage for the Control UI Gateway/WebSocket chat path with long CJK content and a captured malformed-frame scenario if available, then fix the actual boundary that introduces malformed JSON. If browser WebSocket framing is not the failing layer, trace the provider-to-gateway-to-UI event path and preserve final history recovery with clear diagnostics rather than only swallowing parse errors.

## Reproduction Assessment

Unclear. The issue has credible manual steps and current source shows a plausible dropped-frame symptom path, but there is no deterministic current-main test, captured raw frame, or live reproduction proving that CJK UTF-8 boundary splitting reaches GatewayBrowserClient as malformed JSON.

## Solution Assessment

Unclear. A raw WebSocket buffering fix may not be the best solution because standard WebSocket text messages are delivered as complete decoded messages; the narrow maintainable path is to first prove where malformed JSON enters, then fix that transport/parser boundary with a focused regression.
