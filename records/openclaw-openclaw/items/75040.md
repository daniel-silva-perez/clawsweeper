## Summary

Keep open. Current main still has a credible framework-level collision path: provider wrappers can write `thinking`, then the generic OpenAI-compatible `extra_body` wrapper warns and overwrites the same key. The related DeepSeek fixes remain open, and the Ollama portion needs separate scoping because native Ollama uses top-level `think`, not the generic OpenAI-compatible `thinking` field.

## What This Changes

The issue asks OpenClaw to stop repeated `extra_body.thinking` collisions with provider-generated thinking payload fields and keep DeepSeek/Ollama disabled-thinking behavior reliable.

## Best Possible Solution

Keep this open and settle a canonical payload precedence rule: preserve `extra_body` as the final explicit override, avoid noisy no-op `thinking` warnings, normalize common disabled-thinking aliases, ensure disabled DeepSeek payloads cannot replay stale `reasoning_content`, and handle native Ollama `think` behavior in its provider path only if a focused reproduction proves an additional bug.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to use a DeepSeek V4 OpenAI-compatible request where the provider wrapper writes `payload.thinking` and model config also supplies `params.extra_body.thinking`; current main will warn and overwrite in the post-plugin wrapper.

## Solution Assessment

Unclear. The related DeepSeek PRs address concrete disabled-thinking pieces, but the broader framework fix needs maintainer scoping so it does not break the documented rule that `extra_body` wins on request-key collisions.
