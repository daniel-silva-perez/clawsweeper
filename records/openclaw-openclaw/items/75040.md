## Summary

Keep open. Current main still applies provider stream wrappers before the generic `extra_body` wrapper, and that wrapper still warns then overwrites any existing top-level request key, so a DeepSeek V4 request with both provider-generated `thinking` and configured `extra_body.thinking` remains a concrete collision path. The related DeepSeek issue/PR cover a narrower `reasoning_content` replay failure, while the Ollama evidence points at a separate native `think` path rather than proof that this framework collision is fixed.

## What This Changes

The issue asks OpenClaw to stop writing colliding `thinking` request fields through both provider/framework logic and `extra_body`, especially for DeepSeek and local Ollama thinking-disabled configurations.

## Best Possible Solution

Define a canonical payload-precedence rule that keeps explicit `extra_body` overrides predictable, normalizes disabled-thinking aliases and booleans before provider wrapping, suppresses noisy no-op `thinking` collisions, and leaves native Ollama `think` handling inside the Ollama plugin unless a focused Ollama reproduction proves a shared core gap.

## Reproduction Assessment

Yes for the core collision path: configure a DeepSeek V4 OpenAI-compatible model so the provider wrapper writes top-level `payload.thinking`, and also configure `params.extra_body.thinking`; current main will warn and overwrite that key in the later post-plugin wrapper. The broader all-provider timeout claim is less certain, but the source-backed collision is enough to keep the issue open.

## Solution Assessment

Unclear; there is no proposed patch on this issue, and the narrow DeepSeek PR discussed in the related context does not settle the framework-level `extra_body` precedence contract. The safest solution needs maintainer scoping before an automated repair changes shared request-shaping behavior.
