## Summary

Keep open. Current main still has a credible OpenAI-compatible `extra_body.thinking` collision path: provider wrappers can write `payload.thinking`, then the later `extra_body` wrapper warns and overwrites the same key. The DeepSeek disabled-thinking subproblem is covered by related open work, while the Ollama portion is a separate native `think` path that needs maintainer scoping rather than an automatic close.

## What This Changes

The issue asks OpenClaw to stop duplicate `thinking` request-payload writes and warning spam while making DeepSeek and Ollama disabled-thinking behavior reliable.

## Best Possible Solution

Define a canonical payload-precedence rule that keeps `extra_body` as the explicit final override, suppresses or downgrades noisy no-op `thinking` collisions, normalizes disabled-thinking aliases before provider wrapping, prevents disabled DeepSeek V4 payloads from replaying stale `reasoning_content`, and keeps native Ollama `think` behavior in the Ollama provider path unless a focused reproduction proves an additional bug.

## Reproduction Assessment

Yes for the core collision path: configure an OpenAI-compatible DeepSeek V4 request where the provider wrapper writes `payload.thinking` and `params.extra_body.thinking` is also configured, then inspect the composed payload after post-plugin wrapping. The live Ollama stuck-session portion is plausible but not fully reproduced by source inspection alone.

## Solution Assessment

Unclear. The related PRs address concrete DeepSeek subproblems, but the broader framework behavior needs maintainer scoping so the fix does not break the documented rule that `extra_body` wins on request-key collisions.
