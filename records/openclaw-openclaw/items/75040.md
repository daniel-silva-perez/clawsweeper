## Summary

Keep open. Current main still has the reported source-level collision path: provider stream wrappers can write `thinking`, and the generic OpenAI-compatible `extra_body` wrapper later warns and overwrites the same payload key. Some adjacent DeepSeek and Ollama thinking behavior is already covered in current code/tests, but the remaining framework-level warning/precedence behavior needs maintainer scoping rather than cleanup closure.

## What This Changes

This issue reports that `params.extra_body.thinking` collides with provider-generated `thinking` request fields, producing repeated overwrite warnings and possible DeepSeek/Ollama thinking-mode failures.

## Best Possible Solution

Define a canonical payload-precedence rule that keeps `extra_body` as the explicit final override, suppresses or avoids noisy no-op `thinking` collisions when the generated and configured values are semantically the same, and keeps DeepSeek replay repair and native Ollama `think` handling in their provider-owned paths with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a DeepSeek V4 OpenAI-compatible request where the provider wrapper writes `payload.thinking` and the model config also supplies `params.extra_body.thinking`; current main will warn and overwrite in the post-plugin wrapper.

## Solution Assessment

Unclear. The requested “write once” behavior conflicts with the documented rule that `extra_body` wins on collisions, so the safer solution is a collision-aware precedence/logging adjustment plus provider-specific thinking fixes where needed.
