## Summary

Keep open. Current main still has the reported collision path: provider stream wrappers can set `thinking`, then the generic OpenAI-compatible `extra_body` wrapper warns and overwrites the same key. The related DeepSeek work is still open and only covers part of the broader documented `extra_body` precedence problem; native Ollama uses a separate top-level `think` path that should not be folded into the generic OpenAI completions fix without maintainer scoping.

## What This Changes

The issue asks OpenClaw to avoid repeated `extra_body` collisions with provider-generated `thinking` request fields, especially across DeepSeek and Ollama thinking controls.

## Best Possible Solution

Choose one canonical fix that keeps `extra_body` as the final explicit override, avoids noisy no-op `thinking` collisions, normalizes common disabled-thinking aliases such as `false`, `disabled`, and `none`, and ensures a final disabled DeepSeek `thinking` payload cannot leave stale `reasoning_content` in replayed messages. Keep native Ollama top-level `think` behavior separate unless a focused Ollama reproduction proves an additional bug.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to use a DeepSeek V4 OpenAI-compatible request where the provider wrapper sets `payload.thinking` and the model config also supplies `params.extra_body.thinking`; current main will warn and overwrite in the post-plugin `extra_body` wrapper.

## Solution Assessment

Unclear. The related DeepSeek PRs address concrete disabled-thinking failures, but the broader fix should be maintainer-scoped so it does not break the documented rule that `extra_body` wins on request-key collisions.
