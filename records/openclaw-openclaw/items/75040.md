## Summary

Keep open. Current main still has the source-backed `extra_body` collision path: provider request wrappers can set a top-level `thinking` key, then the generic OpenAI-compatible `extra_body` wrapper logs a collision and overwrites that key by documented design. The Ollama timeout evidence points at a partly separate native `think` path, so the remaining work needs maintainer scoping rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop writing or colliding on the OpenAI-compatible `thinking` request field when `params.extra_body.thinking` and provider-specific thinking wrappers both shape the same payload.

## Best Possible Solution

Define a canonical request-payload precedence and normalization rule for thinking controls: preserve predictable explicit `extra_body` overrides, avoid noisy no-op collisions for equivalent disabled-thinking payloads, and keep native Ollama `think` behavior in the Ollama plugin unless a focused reproduction shows a shared core gap.

## Reproduction Assessment

Yes for the core collision path: a DeepSeek V4 OpenAI-compatible request with provider-generated `payload.thinking` and configured `params.extra_body.thinking` will hit the current warning and overwrite branch. The all-provider timeout impact is unclear without live logs or a narrower reproducer per provider.

## Solution Assessment

Unclear; there is no patch on this issue, and the narrow related DeepSeek work does not settle the shared `extra_body` precedence contract. A maintainer should choose the intended precedence/noise-suppression rule before an automated repair changes common request shaping.
