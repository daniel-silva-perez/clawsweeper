## Summary

Keep open. Current main still applies provider stream wrappers before the generic `extra_body` wrapper, so an OpenAI-compatible provider wrapper can set `thinking` and then `extra_body.thinking` can warn and overwrite it. Related DeepSeek PRs cover part of the failure mode, but they do not settle the broader override/no-op warning semantics, and native Ollama uses a separate top-level `think` path.

## What This Changes

The issue requests preventing `params.extra_body.thinking` from colliding with generated request-payload `thinking` fields and causing warning noise or inconsistent thinking state across providers.

## Best Possible Solution

Keep the issue open and choose a canonical fix that preserves documented `extra_body` final-precedence behavior while avoiding noisy no-op `thinking` collisions and preventing DeepSeek disabled-thinking replay inconsistency. The fix should leave native Ollama top-level `think` behavior intact and add focused regression coverage for DeepSeek plus generic OpenAI-compatible `extra_body` collision handling.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: use an OpenAI-compatible provider wrapper that sets `payload.thinking`, such as DeepSeek V4, and configure `params.extra_body.thinking`; the current wrapper order will warn before overwriting. I did not run a test because this sweep was explicitly read-only.

## Solution Assessment

Unclear for the whole issue. The open DeepSeek PRs address concrete parts of the inconsistent disabled-thinking state, but the broader framework-level solution should be scoped by maintainers so `extra_body` remains a deliberate final override without warning on harmless duplicate values.
