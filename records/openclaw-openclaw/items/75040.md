## Summary

Keep open. Current main still has the reported framework-level collision path: provider wrappers can set `payload.thinking`, then the generic OpenAI-compatible `extra_body` wrapper runs later, warns, and overwrites the same key. Recent DeepSeek and Ollama work covers parts of the surrounding behavior, but it does not settle the broader precedence/noisy-warning problem reported here.

## What This Changes

The issue asks OpenClaw to stop duplicate `thinking` request-field writes from causing `extra_body` overwrite warnings and provider thinking-state inconsistencies across OpenAI-compatible and Ollama paths.

## Best Possible Solution

Settle a canonical payload precedence rule: keep `extra_body` as the final explicit override, suppress or downgrade no-op duplicate `thinking` collisions, normalize common disabled-thinking aliases where config resolves thinking defaults, integrate or supersede the focused #74403 DeepSeek replay fix, and handle any native Ollama `think` bug in the Ollama plugin path only with a dedicated reproduction.

## Reproduction Assessment

Yes for the central warning collision: use a DeepSeek V4 OpenAI-compatible request where the provider wrapper writes `payload.thinking` and model config also supplies `params.extra_body.thinking`; current main will warn and overwrite in the post-plugin wrapper. The all-providers frequency and Ollama timeout impact still need separate live or targeted test proof.

## Solution Assessment

Unclear. A narrow fix should not simply block overwrites because docs promise `extra_body` wins; the safer direction is to preserve precedence, normalize disabled thinking consistently, and only suppress or repair collisions that are no-ops or known provider transcript hazards.
