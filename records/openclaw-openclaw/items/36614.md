## Summary

Keep open. Current main still has the write-side contamination path described here: `dmScope=per-channel-peer` resolves direct messages to scoped session keys, but several inbound DM recorders still pass the main session key to last-route persistence, and the shared recorder still writes `deliveryContext`/last-route fields to that supplied target. Merged #38418 mitigates a read-side WebChat inheritance symptom, but it does not remove the main-session state pollution this issue tracks; the two PRs that used closing syntax for this issue are both closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and finish the write-side fix. The most conservative path is to centralize the target decision in `recordInboundSession` or make every direct-message inbound writer use `resolveInboundLastRouteSessionKey`, so `lastRoutePolicy="session"` writes delivery metadata to the scoped inbound session while `lastRoutePolicy="main"` continues updating the main session. Add regression coverage for the central recorder plus iMessage and representative Slack, Signal, LINE, Matrix, Mattermost, and Discord component paths, while preserving existing Telegram and WhatsApp guarded behavior.
