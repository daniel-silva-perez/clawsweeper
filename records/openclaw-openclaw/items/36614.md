## Summary

Keep this issue open. Current main still has the write-side contamination path: `dmScope=per-channel-peer` resolves direct messages to scoped sessions, but the central recorder still writes `deliveryContext` and last-route fields to the caller-supplied target, and several inbound DM paths still pass the main session key. The supplied maintainer comments also explicitly distinguish this write-side issue from the merged read-side mitigation in #38418, while the two implementation PRs that named this issue are closed unmerged.

## Best Possible Solution

Keep this issue open and finish the write-side fix. The likely maintainer path is to make all direct-message inbound writers use `resolveInboundLastRouteSessionKey`, or centralize that decision in `recordInboundSession`, so `lastRoutePolicy=session` writes delivery metadata to the scoped inbound session while `lastRoutePolicy=main` continues updating the main session. Add regression coverage for iMessage plus representative Slack, Signal, LINE, Matrix, Mattermost, and Discord component paths, and preserve the existing Telegram/WhatsApp guarded behavior.
