## Summary

Keep open. Current main does not already implement this PR's intended deletion-aware cancellation for queued or already-collected cron executions. The PR is a plausible implementation path, but the discussion still contains unresolved correctness concerns around where the existence check runs, error handling in timer workers, and extra reload cost. The security review did not find supply-chain or secrets-surface expansion because the PR only changes cron service source and tests.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best implementation is a deletion-aware pre-execution guard for timed due jobs after collection and for manual runs after reservation but before execution, with existence-check failures handled as per-job outcomes, minimal reload overhead, and regression tests proving that deleting a scheduled task prevents queued-but-not-started delivery. If another PR lands the same behavior first, this PR can then close as superseded or implemented.

## Risks / Open Questions

- Current main can still execute a due cron job from an in-memory job object after the stored job was removed, then only discard the result after side effects have already happened.
- The PR should not land as-is unless the timer-side existence check handles reload/read errors as per-job outcomes inside the worker path, so one failed check cannot abort the whole tick and strand running markers.
- The proposed forced reload existence check may add avoidable disk I/O on hot cron paths; a final fix should minimize reload overhead while preserving deletion correctness.
