## Summary

Keep open. Current main still has the reported queue-boundary behavior, and the open PR is not ready to close as implemented because its timer-side existence check can fail outside the per-job error handling path and abort outcome application for the whole tick.

## What This Changes

The PR adds cron job existence checks and regression tests so deleting a scheduled task prevents queued timer or manual executions from running after removal.

## Best Possible Solution

Revise the PR so both timer-collected due jobs and queued manual runs check deletion after reservation but before any delivery side effect, handle existence-check reload failures as per-job outcomes, avoid unnecessary reload cost where possible, and keep regression coverage around deleting a job between queue/collection and execution.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to collect or enqueue a cron job, delete it before the worker starts, and observe that current main still executes the captured `CronJob` snapshot without a pre-execution store recheck.

## Solution Assessment

No. The PR’s direction is right, but the timer guard needs to be inside the per-job error path or otherwise converted into a normal skipped/error outcome so one reload failure cannot abort the whole timer tick.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Catch existence-check failures inside runDueJob:** `src/cron/service/timer.ts:724`
  - body: The new `jobExists` guard runs before `runDueJob` enters its `try/catch`. If the forced reload throws, this worker rejects, `Promise.all(workers)` aborts, and outcomes from other jobs in the same tick may never be applied, leaving running markers behind. Treat this check as a per-job skipped/error outcome instead of letting it reject the worker pass.
