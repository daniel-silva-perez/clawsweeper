## Summary

Keep this PR open. Current main still lacks the proposed cache-only Chrome MCP health probe, and the passive browser status/listing paths can still reach Chrome MCP session creation. This PR remains a focused implementation candidate rather than obsolete cleanup.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate. The likely best fix is the PR's cache-only Chrome MCP health probe wired into `GET /`, `GET /profiles`, and `GET /tabs`, with focused regression tests proving passive status/listing paths do not call the session factory or spawn Chrome MCP when the in-memory cache is empty.
