## Summary

Keep open. Current main still has passive Chrome MCP browser status/listing paths that can reach session creation and spawn `chrome-devtools-mcp`; this PR remains a focused implementation candidate rather than obsolete cleanup.

## What This Changes

This PR adds a cache-only Chrome MCP health probe and routes browser status, profile listing, and tab listing checks through it so passive probes do not spawn a Chrome MCP subprocess.

## Best Possible Solution

Review and land this PR or an equivalent narrow change that keeps passive `GET /`, `GET /profiles`, and `GET /tabs` Chrome MCP checks cache-only, while leaving explicit attach/start and broader orphan-process cleanup to their existing paths or separate issues.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: with a Chrome MCP existing-session profile and an empty in-memory session cache, passive status/listing routes call reachability helpers that can flow into `createEphemeralSession` and `StdioClientTransport` creation; the live macOS consent dialog was not reproduced in this read-only pass.

## Solution Assessment

Yes. The proposed cache-only health probe is the narrowest maintainable fix for the passive-spawn problem because it avoids `getSession` and pending-session awaits on read-only routes without changing CDP profiles or explicit attach/start behavior.
