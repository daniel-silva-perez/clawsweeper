## Summary

Keep open. Current main still lacks a cache-only Chrome MCP health probe, and passive browser status/listing paths can still reach Chrome MCP session creation; this PR remains a focused implementation candidate. I did not find a blocking correctness or security issue in the proposed diff, but it needs normal maintainer validation and a required changelog entry before merge.

## What This Changes

The PR adds a cache-only Chrome MCP health probe and rewires existing-session browser status, profile listing, and tab listing paths plus tests so passive reads do not spawn `chrome-devtools-mcp` when the in-memory cache is empty.

## Best Possible Solution

Keep this PR as the active implementation candidate, add the missing single-line changelog entry for the browser existing-session fix, then validate and merge if extension tests, extension typecheck, and maintainer review stay green. The broader subprocess/orphan cleanup should remain tracked separately under #53399 rather than expanding this small passive-probe fix.

## Reproduction Assessment

Yes. The high-confidence reproduction is code-level on current main: `GET /`, `GET /profiles`, and `GET /tabs` for a Chrome MCP existing-session profile can reach `isTransportAvailable` or `isReachable`, which reaches `ensureChromeMcpAvailable` or `listChromeMcpTabs`, and an empty cache falls through to `createEphemeralSession` / `createRealSession`.

## Solution Assessment

Yes, with a small policy follow-up. A Chrome MCP cache-only health probe owned inside `chrome-mcp.ts` and used only by passive status/listing routes is the narrowest maintainable fix; explicit attach/start paths remain unchanged, and broader process cleanup belongs in a separate issue.
