## Summary

Keep open. The PR fixes the legacy/raw JSONL reader path, but current v3 transcripts with tree entries still go through the active-branch reader and return before the new runtime-context handling runs, so the main path can still lose sender labels.

## What This Changes

The PR adds gateway session-history logic and regression tests to recover `senderLabel` from hidden `openclaw.runtime-context` custom message rows while keeping those rows out of `readSessionMessages()` output.

## Best Possible Solution

Keep the PR open and update `readSessionMessages()` so runtime-context custom rows are handled consistently in both the active-branch and raw JSONL readers, without reintroducing channel metadata into model-visible user text.

## Reproduction Assessment

Yes. Create a v3 transcript with `id`/`parentId` entries where a user message is followed by a hidden `openclaw.runtime-context` `custom_message`; `readSessionMessages()` takes the `SessionManager.getBranch()` path, ignores the custom row, and returns without attaching `senderLabel`.

## Solution Assessment

No. The raw JSONL recovery is useful but partial; the narrow maintainable fix is to share or duplicate the runtime-context handling in the active-branch loop and add a branch-path regression test.
