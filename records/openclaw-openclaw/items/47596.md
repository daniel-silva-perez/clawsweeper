## Summary

Keep open. Current main supports `streamTo="parent"` for ACP runs, but the parent relay still only surfaces assistant text and lifecycle notices. Existing agent event streams already carry tool, command-output, item, and thinking data elsewhere, but `startAcpSpawnParentStreamRelay()` drops every non-`assistant`/`lifecycle` stream, so the requested tool-call/tool-result/thinking visibility is not implemented. The related #67590 is a broader channel-level streaming request and is closed; it does not supersede this narrower ACP parent-relay API work.

## Best Possible Solution

Keep this issue open. The best implementation is a focused core change in `src/agents/acp-spawn-parent-stream.ts` that maps existing `tool`/`item`/`command_output` and, if product-approved, `thinking` events into concise parent system-event summaries. It should respect an explicit verbosity policy, avoid raw full args/results by default, reuse existing display/sanitization/truncation helpers where possible, and add relay tests for tool start, tool result, command output truncation, thinking gating, and lifecycle ordering.

## Risks / Open Questions

- Surfacing tool arguments, command output, and reasoning to a parent session can expose sensitive child-session data. The implementation should be explicitly gated, summarized, sanitized, and tested rather than forwarding raw event payloads by default.
- ACP bridge-mode tool streaming in `docs/cli/acp.md` is a different surface from ACP `sessions_spawn({ streamTo: "parent" })`; closing this issue based on bridge-mode support would conflate two separate runtime directions.
