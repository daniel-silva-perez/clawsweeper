## Summary

Keep this issue open. Current main still defers the generic Discord native-command wrapper and appends only already-registered plugin command specs during provider startup, while non-startup plugin aliases such as Codex remain plausibly absent until runtime plugin activation and `/vc` is a separate voice subcommand path.

## What This Changes

The issue reports that Discord slash commands registered by plugins can miss Discord's 3-second interaction acknowledgement deadline and fail with `Unknown interaction` or already-acknowledged errors.

## Best Possible Solution

Keep this open as a focused Discord regression investigation. Maintainers should separately verify non-startup plugin aliases such as `/codex` against Discord command deployment/startup timing, then verify `/vc` against Carbon subcommand defer and voice join behavior, while keeping any fix inside the Discord/plugin command boundary unless a generic SDK lifecycle seam is required.

## Reproduction Assessment

Unclear: the issue gives concrete live steps and logs against 2026.4.26, and current source still supports the reported activation-order hypothesis. I do not have a high-confidence local reproduction because this read-only pass did not run a live Discord gateway with the reported plugins and credentials.

## Solution Assessment

Unclear: the suggested immediate defer is already present for generic native command wrappers, so the best fix is likely narrower than the issue's full proposal. The maintainable path is to prove whether Discord-visible non-startup aliases should be eagerly prepared from manifest metadata, excluded until startup-loaded, or handled through a new plugin command lifecycle/responder seam, with `/vc` checked separately.
