## Summary

Keep open. Current main still has early defer for already-created Discord native commands and startup-time appending of already-registered plugin command specs, but there is no later targeted fix for the reported startup-lazy/plugin-registered command lifecycle. The prior ClawSweeper comment and current source both support leaving this as an active Discord regression investigation rather than closing as implemented.

## What This Changes

The issue reports that Discord slash commands owned by plugins can miss Discord's 3-second interaction acknowledgement deadline and fail with Unknown interaction or already-acknowledged errors.

## Best Possible Solution

Define a single Discord-native lifecycle for plugin-owned slash commands: either only deploy commands whose handlers are ready before gateway ready, or add a manifest-backed predeploy/activation contract that can acknowledge immediately and then load the owning plugin. Keep the /vc voice path as a separate verification against subcommand defer and voice-connect behavior.

## Reproduction Assessment

Unclear: the report includes concrete logs and steps, but this read-only review did not invoke Discord live. A high-confidence reproduction should run Discord with commands.native enabled and at least one startup-lazy runtime slash alias such as Codex, then record received-to-defer and command-registration timing.

## Solution Assessment

Unclear as a fix path because there is no proposed patch. Blanket eager plugin loading could fix timing but may violate the plugin laziness boundary; the safer direction is an explicit Discord-visible command readiness contract plus focused tests.
