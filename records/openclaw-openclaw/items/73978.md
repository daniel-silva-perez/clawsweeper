## Summary

Keep this issue open. Current main already defers the generic Discord native-command wrapper before command dispatch and appends already-registered plugin command specs during Discord provider startup, but those paths were already present in v2026.4.26, the version in the report. I did not find a later main-only fix for the reported late plugin registration / Unknown interaction behavior, and the examples mix at least two paths: plugin runtime slash aliases and the separate built-in Discord voice subcommand implementation.

## Best Possible Solution

Keep this issue open and narrow it into a focused Discord regression investigation: reproduce with a non-startup plugin command alias such as Codex plus a voice command, verify when plugin command aliases are registered versus deployed to Discord, keep the interaction acknowledgement before any runtime plugin loading, and either eagerly prepare Discord-visible runtime slash commands before gateway ready or clearly exclude non-startup aliases from native Discord registration. The `/vc` path should be checked separately against Carbon subcommand defer behavior and moved to an explicit defer/follow-up pattern if it can miss the 3-second deadline.
