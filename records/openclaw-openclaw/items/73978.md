## Summary

Keep open. Current main still has early defer for Discord native command objects and direct plugin-command dispatch after that defer, but Discord startup deployment still appends only plugin commands that are already present in the runtime `pluginCommands` registry. Startup-lazy manifest aliases such as Codex are not a handler-ready native command source, and the relevant current-main changes after the prior review do not close that lifecycle gap.

## What This Changes

The issue reports that Discord slash commands registered by plugins can miss Discord's 3-second interaction acknowledgement deadline when plugin command registration happens lazily during first invocation.

## Best Possible Solution

Define one Discord-native lifecycle for plugin-owned slash commands: either deploy only commands whose handlers are ready before Discord startup completes, or add a manifest-backed native-command contract that can acknowledge immediately and then activate the owning plugin without racing the interaction token. Keep `/vc` voice behavior verified separately from generic plugin command dispatch.

## Reproduction Assessment

Unclear: the issue includes concrete steps and logs, and current source supports the suspected lifecycle gap, but this review did not run a live Discord bot. A high-confidence reproduction should enable `commands.native`, include a startup-lazy runtime-slash plugin such as Codex, and capture received-to-defer plus plugin registration timing.

## Solution Assessment

Unclear because there is no proposed patch. Blanket eager plugin loading would likely fix timing but may violate the documented lazy plugin boundary; the safer direction is an explicit Discord-visible command readiness contract with focused tests.
