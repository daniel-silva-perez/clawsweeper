## Summary

Keep open. Current main still only gives the early Discord defer path to native command objects already present in the monitor command list; startup-lazy plugin aliases such as Codex are still metadata until the plugin runtime registers a handler, so there is no implemented-on-main close path after the latest commits.

## What This Changes

The issue asks for Discord plugin-registered slash commands to be registered or acknowledged early enough to meet Discord's 3-second interaction deadline instead of failing with Unknown interaction or already-acknowledged errors.

## Best Possible Solution

Define one Discord-native readiness contract for plugin-owned slash commands: either predeploy manifest-declared commands with a lightweight immediate-defer dispatcher that activates the owner plugin, or only deploy commands whose handlers are loaded before gateway readiness. Keep voice /vc behavior and the generic plugin command lifecycle covered by separate tests.

## Reproduction Assessment

Unclear: the report has concrete logs and plausible steps, but this read-only review did not run a live Discord bot. A high-confidence reproduction should enable Discord native commands with a startup-lazy plugin command such as /codex and capture received-to-defer plus command-registration timing.

## Solution Assessment

Unclear because there is no proposed patch and current main is not sufficient for startup-lazy plugin-owned native commands. The safer solution is an explicit Discord-visible command readiness contract rather than blindly eager-loading every plugin at startup.
