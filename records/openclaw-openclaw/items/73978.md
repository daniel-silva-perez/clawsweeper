## Summary

Keep this issue open. Current main still has the generic Discord native-command wrapper deferring immediately and appending already-registered plugin command specs during Discord provider startup, but those same central paths were already present in v2026.4.26, the reported release. I did not find a current-main fix that proves late plugin command registration, non-startup runtime slash aliases, or the separate `/vc` voice subcommand path are resolved.

## Best Possible Solution

Keep this open and narrow the investigation around two paths: non-startup plugin runtime slash aliases such as `/codex`, and the separate `/vc` voice command. Verify when each command is deployed to Discord versus registered in OpenClaw, ensure any native interaction is acknowledged before runtime plugin loading or voice work, and decide whether Discord-visible runtime slash aliases should be prepared from manifest metadata before gateway ready or excluded until their plugin is actually startup-loaded.
