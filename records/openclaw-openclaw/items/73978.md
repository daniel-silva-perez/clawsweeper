## Summary

Keep open. Current main still has an early defer for Discord native commands that already exist in the monitor, but plugin-owned native command specs are still sourced from the runtime plugin command registry; startup-lazy manifest aliases such as Codex are metadata, not registered handlers. The six commits after the prior ClawSweeper review did not change the Discord/native plugin command lifecycle, so there is no strong implemented-on-main close path.

## What This Changes

The issue requests a fix for Discord plugin-registered slash commands so they are registered or acknowledged early enough to meet Discord's 3 second interaction deadline instead of failing with Unknown interaction or already-acknowledged errors.

## Best Possible Solution

Define one Discord-native readiness contract for plugin-owned slash commands: either predeploy manifest-declared commands with a lightweight immediate-defer dispatcher that activates the owner plugin, or only deploy commands whose handlers are loaded before gateway readiness. Keep voice /vc validation separate from generic plugin command dispatch.

## Reproduction Assessment

Unclear: the report has concrete logs and plausible steps, but this read-only review did not run a live Discord bot. A high-confidence reproduction should enable Discord native commands with a startup-lazy runtime alias such as /codex and capture command registration plus received-to-defer timings.

## Solution Assessment

Unclear because there is no proposed patch, and current main is not sufficient for startup-lazy plugin-owned native commands. The safer solution is an explicit Discord-visible command readiness contract rather than blindly eager-loading every plugin at startup.
