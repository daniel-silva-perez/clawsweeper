## Summary

Keep this issue open. Current main still does not implement a custom TUI client display name: the TUI command has no `--client-name`, `--display-name`, or `--username` option, `TuiOptions` has no field to carry one, Gateway-mode TUI connects with only URL/auth options, and `GatewayChatClient` still hard-codes `clientDisplayName: "openclaw-tui"`. The Gateway protocol and presence path already support display names, so this remains a focused TUI wiring gap rather than a protocol request or duplicate of the already-implemented WebChat/Control UI display-name work.

## Best Possible Solution

Add a focused TUI implementation: parse `--client-name` or `--display-name` in `src/cli/tui-cli.ts`, extend `TuiOptions` and TUI relaunch argument building, pass a sanitized value through `runTui` into `GatewayChatClient.connect`, default to `openclaw-tui`, preserve the stable protocol client id, and update `docs/web/tui.md` plus focused CLI/backend tests.
