## Summary

Keep this PR open. Current main still uses the global TTS timeout for non-telephony synthesis and OpenAI speech config normalization still drops `timeoutMs`; the PR addresses that gap narrowly, and the follow-up commit resolves the only concrete review concern in the provided discussion.

## What This Changes

The PR preserves OpenAI TTS provider `timeoutMs`, resolves per-provider attempt timeouts for chat/non-telephony synthesis with explicit/provider/global precedence, and adds speech-core regression tests for provider timeout precedence.

## Best Possible Solution

Merge the narrow fix after normal maintainer review and gates, keeping the behavior in the speech-core synthesis loop plus the OpenAI provider normalizer rather than adding a new core API or provider-specific core special case.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `messages.tts.providers.openai.timeoutMs` higher than `messages.tts.timeoutMs`; chat synthesis still passes the global timeout from `extensions/speech-core/src/tts.ts`, and OpenAI normalization drops the provider timeout.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix: it preserves the provider field where OpenAI owns config normalization, resolves one attempt timeout in speech-core using explicit/provider/global precedence, and leaves telephony behavior unchanged.
