## Summary

Keep this issue open. Current main has partial protection for newly added ambiguous cron announce delivery, but WhatsApp target normalization can still accept a Telegram-prefixed numeric target and turn it into an E.164-like WhatsApp destination if runtime routing selects WhatsApp.

## What This Changes

The issue reports that cron error delivery can route a Telegram-targeted identifier through WhatsApp during config recovery, causing a Telegram chat ID to be interpreted as a phone number.

## Best Possible Solution

Harden channel target validation so provider-prefixed identifiers cannot cross channel boundaries. The likely fix is to reject non-WhatsApp prefixes in the WhatsApp target resolver, add cron/failure-alert regression coverage for WhatsApp-selected delivery with telegram:/tg: targets, and preserve Telegram prefix normalization only in Telegram-owned parsing.

## Reproduction Assessment

Yes. A focused unit reproduction can call the WhatsApp outbound target resolver with to="telegram:1234567890" and an empty allowFrom list, or drive cron delivery resolution with WhatsApp selected and that to value; current source shows it normalizes to an E.164-like WhatsApp target instead of failing.

## Solution Assessment

No current fix is sufficient. The existing cron validation is useful but only addresses ambiguous add/update cases; the narrower maintainable solution is channel-owned target-prefix rejection at the outbound resolver boundary plus cron regression tests for failure delivery.
