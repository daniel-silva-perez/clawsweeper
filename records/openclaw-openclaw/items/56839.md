## Summary

Keep this issue open. Current main still has the reported channel-boundary bug: WhatsApp target normalization can pass a Telegram-prefixed numeric target through generic E.164 normalization, and the WhatsApp outbound resolver accepts that normalized value when WhatsApp is the selected channel with an empty or wildcard allow list. Cron failure delivery still resolves through the selected channel target resolver, while current cron validation only checks configured channel selection, not whether `delivery.to` or `failureDestination.to` belongs to that channel.

## What This Changes

The issue asks OpenClaw to prevent Telegram-prefixed cron delivery identifiers from being interpreted as WhatsApp phone numbers during config recovery or failure notification delivery.

## Best Possible Solution

Harden the WhatsApp-owned target boundary so non-WhatsApp provider prefixes such as `telegram:` and `tg:` are rejected before generic E.164 normalization, while preserving valid WhatsApp E.164 numbers, WhatsApp user JIDs, group JIDs, LID forms, and `whatsapp:` prefixes. Add regression coverage at the WhatsApp normalizer/outbound resolver and cron failure-delivery surfaces so channel-selected delivery cannot coerce another provider’s identifier.

## Reproduction Assessment

Yes. A focused source-level repro is `normalizeWhatsAppTarget("telegram:1234567890")` followed by `resolveWhatsAppOutboundTarget({ to: "telegram:1234567890", allowFrom: [], mode: "explicit" })`; current code shows the target can normalize to a WhatsApp E.164-style destination and be accepted.

## Solution Assessment

Yes, the narrow maintainable solution is to fix the WhatsApp target resolver boundary and add cron regression coverage. Core cron validation may remain generic, but it should not be the only protection against provider-prefixed target coercion.
