## Summary

Keep this issue open. Current main still has a concrete WhatsApp target-boundary bug: the WhatsApp normalizer strips only WhatsApp prefixes, then calls a generic E.164 normalizer that strips any provider-style prefix, so a Telegram-prefixed numeric target can become a WhatsApp phone destination when cron/failure delivery selects WhatsApp. The existing cron channel validation is only partial and does not validate `delivery.to` or `failureDestination.to` against the selected channel.

## What This Changes

Prevent WhatsApp cron and failure delivery from interpreting Telegram-prefixed numeric recipients as WhatsApp phone numbers during config recovery or fallback routing.

## Best Possible Solution

Harden the WhatsApp target boundary so non-WhatsApp provider prefixes such as `telegram:` and `tg:` are rejected before generic E.164 normalization, while preserving valid WhatsApp E.164, JID, group JID, and `whatsapp:` forms. Add regression coverage for WhatsApp normalization/outbound resolution and cron failure delivery when WhatsApp is selected with Telegram-prefixed targets.

## Reproduction Assessment

Yes. A focused unit check can call `normalizeWhatsAppTarget` or `resolveWhatsAppOutboundTarget` with `to="telegram:1234567890"` and an empty or wildcard `allowFrom`; current source shows that this can normalize to an E.164-like WhatsApp target instead of failing.

## Solution Assessment

No current fix is sufficient. The existing cron channel guard helps ambiguous add/update cases, but the narrow maintainable fix belongs at the channel-owned WhatsApp target resolver boundary, with cron regression tests covering failure delivery.
