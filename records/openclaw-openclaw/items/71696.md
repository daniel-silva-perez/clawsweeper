## Summary

Keep this PR open. Current main still lacks Telegram chat-type-specific reply mode, so the work is not obsolete, but the branch is not merge-ready because the proposed schema default makes the new per-chat map override existing `replyToMode` and runtime fallbacks after config validation; it also lacks the required changelog entry for a user-facing Telegram feature/fix.

## What This Changes

The PR adds Telegram `replyToModeByChatType` config/schema/types/metadata, applies chat-type-specific reply mode in message and native-command delivery, and passes Telegram message ids into plugin command reply handling.

## Best Possible Solution

Keep the PR open and adjust the implementation before merge: make `replyToModeByChatType` optional in the schema/metadata, apply direct/group defaults only inside the Telegram resolver after per-chat, runtime fallback, and account `replyToMode` are absent, add validated-config regression tests for the fallback order, rebase onto current main, and add a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: validate a config with `channels.telegram.replyToMode: "first"` and no explicit `replyToModeByChatType`; the PR schema default inserts `{ direct: "off", group: "all" }`, and the PR resolver returns that per-chat value before the existing fallback/account mode.

## Solution Assessment

No. The PR touches the right Telegram/config/plugin surfaces, but defaulting the per-chat map in Zod is not the narrowest maintainable solution because it changes precedence for existing validated configs; the safer path is an optional schema plus resolver-owned defaults.
