## Summary

Keep open. Current main still lacks Telegram chat-type reply mode and the related message-id plumbing, but the PR has a blocking precedence regression: its schema default materializes per-chat values before resolver fallbacks, so existing runtime or account-level `replyToMode` settings can be ignored.

## What This Changes

The branch adds Telegram `replyToModeByChatType` schema/types/metadata, routes per-chat reply mode through message and native-command delivery, restores current-message reply directive plumbing, and exposes provider message ids to plugin command handlers.

## Best Possible Solution

Keep the PR open and fix the config/default precedence before merge: make the new schema key optional, let the resolver apply direct/group defaults only after per-chat, runtime fallback, and account `replyToMode` are absent, and add tests that use validated runtime config with both `replyToMode` and no explicit `replyToModeByChatType`.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists from the PR diff plus current config validation: `validateConfigObjectWithPlugins` applies defaults, so a parsed `channels.telegram: { replyToMode: "first" }` would receive default per-chat values that the PR resolver checks before the fallback/account mode.

## Solution Assessment

No. The PR targets the right Telegram/config/plugin surfaces, but defaulting `replyToModeByChatType` in the schema is not the safest way to solve it because it makes absent per-chat config indistinguishable from explicit per-chat config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve replyToMode fallback before per-chat defaults:** `src/config/zod-schema.providers-core.ts:234-237`
  - body: The resolver checks `replyToModeByChatType` before the runtime fallback and account `replyToMode`, but this schema default makes every parsed Telegram config look like it explicitly set `{ direct: "off", group: "all" }`. Existing configs such as `channels.telegram.replyToMode: "first"`, and runtime `createTelegramBot({ replyToMode: ... })` overrides when a Telegram config object exists, will be ignored after validation. Keep the schema optional and apply direct/group defaults only after the fallbacks are absent.
