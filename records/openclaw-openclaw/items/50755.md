## Summary

Keep open. Current main still only implements global session maintenance, so this PR is not obsolete; however, the proposed branch is not merge-ready because it exposes per-agent retention config while leaving normal runtime writes on the global policy and introducing unsafe cleanup behavior for shared or explicit stores.

## What This Changes

The PR adds an `agents.list[].maintenance` config surface, extracts a shared session maintenance Zod schema, merges per-agent overrides over global session maintenance, threads an agent id into session cleanup/save options, regenerates base config schema, and adds resolver tests.

## Best Possible Solution

Keep this PR open for maintainer review, but finish or replace it with an implementation that resolves effective per-agent maintenance from one normalized config snapshot, applies it consistently to normal runtime writes and loads, keeps shared/explicit-store cleanup on a safe policy, preserves legacy `pruneDays` precedence, and updates generated schema/docs/tests together.

## Reproduction Assessment

Not applicable as a feature PR, but the review findings have high-confidence source-level reproduction paths. Normal reply/session writes resolve global `session.maintenance`, while `--all-agents` and `--store` cleanup behavior can be traced through `resolveSessionStoreTargets()`.

## Solution Assessment

No. The PR is directionally useful, but as-is it is not the narrowest maintainable solution because it exposes per-agent config before the main enforcement paths and cleanup target semantics are safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Thread per-agent maintenance into runtime writes:** `src/config/sessions/store.ts:292`
  - body: Adding `agentId` to the save options is not enough unless normal write callers pass an effective per-agent policy. The reply/session, boot, and heartbeat paths still write session stores without selecting the agent override, so `agents.list[].maintenance` will not control the automatic maintenance path used during regular traffic.
  - confidence: 0.88
- **[P1] Avoid per-agent cleanup on shared stores:** `src/commands/sessions-cleanup.ts:311`
  - body: `--all-agents` dedupes identical store paths, but this PR applies the surviving target's per-agent policy to the whole JSON file. In a supported shared-store setup, one agent's stricter retention can prune sessions belonging to another agent; shared targets should fall back to the global policy or be split by entry ownership.
  - confidence: 0.9
- **[P2] Keep explicit store cleanup on global policy:** `src/commands/sessions-cleanup.ts:374`
  - body: For `--store`, the target resolver uses the default agent id only as a placeholder for the arbitrary path. Passing that synthetic id into maintenance resolution makes an explicit file cleanup inherit the default agent's override, which can delete entries from a store the default agent does not own.
  - confidence: 0.87
- **[P2] Normalize agent ids before matching overrides:** `src/config/sessions/store-maintenance.ts:139`
  - body: This lookup compares the raw config id to the requested id, while the rest of agent resolution normalizes both sides. Config entries such as `Ops` or `My Agent` will resolve elsewhere but miss their maintenance override here, silently falling back to the global policy.
  - confidence: 0.86
- **[P2] Resolve pruneAfter/pruneDays precedence before merging:** `src/config/sessions/store-maintenance.ts:157`
  - body: A spread merge can leave global `pruneAfter` and per-agent legacy `pruneDays` on the same object. Since the resolver prefers `pruneAfter`, a supported per-agent `pruneDays` override will not override the global age window.
  - confidence: 0.84
