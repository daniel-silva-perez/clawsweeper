## Summary

Current main does not yet solve the reported configuration shape. The code still applies `tools.profile: "coding"` before `agents.<id>.tools.allow`, so an agent allowlist containing only `browser` and Gmail plugin tools becomes an empty intersection after the coding profile filters those tools out. Current docs point users to `alsoAllow` for additive profile exceptions, and current main has a fail-closed precheck for empty explicit allowlists, but the issue’s core user problem still needs a maintainer decision or targeted UX/config fix.

## Best Possible Solution

Keep this issue open. The best maintainer path is to decide the intended contract for `tools.profile` plus per-agent tool overrides, then either make additive intent work for plugin/browser tools at the profile stage or add a targeted config/CLI diagnostic that explains the empty intersection and points users to `agents.list[].tools.alsoAllow` for this exact setup.
