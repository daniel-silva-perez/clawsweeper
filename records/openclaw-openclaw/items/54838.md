## Summary

Keep this PR open. Current main still does not expose a user-facing Control UI default-expand preference, and related issues were closed into this PR as the canonical implementation thread, but the branch is not ready to merge because its render patch controls `<details open>` directly and would collapse manually opened tool sections on rerender. Current main now has a better `autoExpandToolCalls`/`syncToolCardExpansionState` seam that this work should be rebased or replaced onto.

## What This Changes

The PR adds a persisted browser-local `chatExpandToolCalls` setting, desktop/mobile chat toolbar buttons, render plumbing, i18n copy, and fixture updates so Control UI tool-call output can expand by default.

## Best Possible Solution

Land a refreshed implementation that adds the browser-local preference and toolbar control while driving the existing stateful `autoExpandToolCalls` path. Keep manual per-tool expansion stable across rerenders, use accurate disabled-state copy, and leave any gateway-wide config option as a separate explicit product decision if maintainers want it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve manual expansion state:** `ui/src/ui/chat/grouped-render.ts:565`
  - body: Binding `open` directly to the default setting makes Lit remove the `open` attribute whenever the setting is false, which is the default. A user who manually opens a tool section during streaming or a history refresh will have it collapsed again on the next render, regressing existing disclosure behavior; initialize expansion state once instead of forcing `open=false` every render.
  - confidence: 0.89
- **[P3] Use accurate disabled state for the expand toggle:** `ui/src/ui/app-render.helpers.ts:312-324`
  - body: The new button uses the onboarding-disabled tooltip when tool calls are manually hidden, and it stays enabled during onboarding because `showToolCalls` is forced true there. Align the disabled condition with the neighboring chat controls and use a separate hidden-tools tooltip so the button state matches why it is unavailable.
  - confidence: 0.78
