## Summary

Keep open. Current main still lacks a persisted Control UI preference or toolbar control for expanding tool-call output by default, and the PR is useful but needs refresh against the stateful expansion path now on main. The blocking issue is that the PR drives raw disclosure open state instead of the existing expansion-state seam, which can collapse manually opened tool output again on rerender when the preference is off.

## What This Changes

The PR adds a browser-local `chatExpandToolCalls` setting, desktop/mobile Control UI chat toolbar controls, render plumbing, English copy, and fixture/test updates so tool-call output can expand by default.

## Best Possible Solution

Land a refreshed implementation that adds the browser-local preference and toolbar control, wires it into `autoExpandToolCalls` plus `syncToolCardExpansionState`, preserves manual expand/collapse state across streaming and history rerenders, and uses accurate disabled-state copy for onboarding versus hidden tool calls.

## Reproduction Assessment

Yes. The code-level reproduction is clear from the diff: with the preference false, manually open a tool disclosure and trigger a Lit rerender during streaming or history refresh; the PR's bound `?open=${false}` path removes the open state again.

## Solution Assessment

No. The feature direction is valid, but the patch should seed current main's stateful `autoExpandToolCalls`/`syncToolCardExpansionState` path instead of binding raw disclosure `open` state, and it should align disabled-state copy with the neighboring controls.
