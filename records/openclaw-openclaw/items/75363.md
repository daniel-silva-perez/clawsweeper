## Summary

Keep this PR open. The Discord hardening work is useful and is opted into automerge, but the current diff introduces a blocking reconcile regression by filtering real option localization fields before comparing desired and live Discord commands.

## What This Changes

The PR adds Discord REST priority lanes with stale request handling, gateway send/intents/reconnect validation, command metadata normalization, one-off component wait helpers, and related internal tests.

## Best Possible Solution

Keep the hardening direction, but narrow the command comparison normalization so it ignores Discord live-only singular localized fields while preserving plural localization maps on options and subcommands, with a regression test proving reconcile patches changed option translations.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord command option or subcommand whose `name_localizations` or `description_localizations` changes; the PR's comparator removes those fields under `options`, so reconcile mode can skip the needed PATCH.

## Solution Assessment

No. The broad `path.includes("options")` filter is not the best fix; the safer solution is to filter only live-only fields and fields Discord omits by default, while retaining user-authored localization maps that the serializer and Discord API both support.
