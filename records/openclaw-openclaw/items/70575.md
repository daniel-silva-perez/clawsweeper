## Summary

Keep open. Current main still builds the gateway agent session writeback patch from a cached `entry` and carries cached fallbacks for `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel`; that patch is later merged over the fresh session-store row, so the reported stale-writeback race remains real.

## What This Changes

The issue asks OpenClaw to stop stale cached `lastChannel`/delivery/channel values from clobbering fresher session-store values in the gateway agent handler session patch.

## Best Possible Solution

Keep cached `entry` fallback values out of the prebuilt agent-handler patch, derive any necessary fallback route fields from the fresh row inside the `updateSessionStore` mutator, and add stale cached-entry/fresh-store regression coverage for the delivery and `channel` fields.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a source-level stale-cache/fresh-store unit shape: make `loadSessionEntry` return a cached row without the delivery/channel field, make the locked `updateSessionStore` mutator receive a fresh row with that field, and observe current main merging the stale patch over the fresh row.

## Solution Assessment

Yes. The issue’s proposed direction is the narrowest maintainable fix: stop writing cached fallback fields into the patch, or compute the fallback from the fresh store row inside the locked mutator. The open question is branch coordination with #19328, not a different technical approach.
