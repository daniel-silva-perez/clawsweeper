## Summary

Keep open. Current main still builds the agent session-store patch from the cached `entry` and then merges that patch over the fresher store row, so the reported stale writeback class remains present for `lastChannel`/`lastTo`/`lastAccountId`/`lastThreadId` and `channel`. Related PR #19328 is open and unmerged, and its live diff still leaves these residual delivery/channel writebacks in place.

## Best Possible Solution

Keep the issue open and address it either by expanding PR #19328 before merge or by landing a focused follow-up. The fix should keep cached `entry` fallback values out of `nextEntryPatch`, derive any fallback delivery/channel values inside the `updateSessionStore` mutator from the fresh store row when needed, and add regression coverage for the four delivery fields plus `channel` using the stale cached-entry/fresh store pattern.
