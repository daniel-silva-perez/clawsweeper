## Summary

Keep open. Current main still has restart preflight, drain, update-restart health checks, stale-listener cleanup, and post-restart diagnostics, but the ordinary restart path closes the active Gateway before a replacement is proven healthy, so the requested no-disconnect rollback behavior remains unsolved.

## What This Changes

The issue requests an atomic Gateway restart handoff that starts and verifies a replacement instance before stopping the currently running instance.

## Best Possible Solution

Design and implement a transactional, supervisor-aware Gateway restart path: validate changes before mutation, stage a replacement or equivalent readiness probe, prove the replacement is reachable, switch ownership of the connectable endpoint and channel transports, and stop the old Gateway only after the new instance is healthy. If startup fails, keep the old Gateway reachable and report the failure through existing diagnostics and, where feasible, the active channel.

## Reproduction Assessment

Yes. The source-level reproduction path is a SIGUSR1/config/daemon restart where replacement startup fails: current main closes the active Gateway before restart handling, and the subsequent startup-failure path logs diagnostics without restoring the old listener.

## Solution Assessment

Unclear. The no-disconnect outcome is the right product target, but the literal start-new-first proposal needs maintainer design for same-port binding, process supervisors, config mutation timing, and channel ownership before it is the narrowest maintainable fix.
