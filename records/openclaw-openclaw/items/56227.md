## Summary

Keep open. Current main has meaningful restart mitigations, including config preflight, drain-before-restart, stale-listener cleanup, and post-restart diagnostics, but it still does not implement the requested atomic handoff: the active Gateway is closed or restarted before a replacement is proven healthy.

## What This Changes

The issue requests atomic Gateway restart behavior so a failed replacement startup leaves the previous Gateway instance running and able to notify users through existing channels.

## Best Possible Solution

Keep the issue open and design a conservative restart handoff: validate config first, start a replacement through an alternate port/socket or supervisor-aware handoff, probe readiness, switch the advertised/connectable endpoint, and stop the old Gateway only after the replacement is healthy. If replacement startup fails, leave the old Gateway serving and deliver a failure notice through the still-active channel when possible.

## Reproduction Assessment

Yes. The source-level reproduction path is a required SIGUSR1/config restart or managed service restart: current main closes or restarts the active Gateway before replacement health is known; no live destructive restart was run because this review was read-only.

## Solution Assessment

Unclear. The requested no-disconnect outcome is the right target, but the literal start-new-first approach needs maintainer design for same-port binding, supervisor behavior, and channel ownership before it is the narrowest maintainable fix.
