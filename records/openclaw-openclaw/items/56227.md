## Summary

Keep open. Current main still drains and closes the active Gateway before replacement startup is proven, while daemon/service restart paths add preflight, health checks, stale-listener cleanup, and diagnostics rather than an old-instance-preserving handoff. The prior ClawSweeper comment reached the same conclusion, and the relevant files have not materially changed since that review.

## What This Changes

The issue requests gateway restarts to keep the previous instance alive until a replacement is started and proven healthy, avoiding total disconnect on failed restarts.

## Best Possible Solution

Design and implement a transactional, supervisor-aware Gateway restart model: validate changes before mutation where possible, stage or probe a replacement Gateway, prove it is reachable, transfer endpoint/channel ownership, and stop the old instance only after the replacement is healthy. If startup fails, leave the old Gateway reachable and report diagnostics through existing surfaces and, where feasible, the still-active channel.

## Reproduction Assessment

Yes. The high-confidence source reproduction is a SIGUSR1/config/daemon restart where replacement startup fails after `server.close`; current main has no path that restores the prior listener afterward.

## Solution Assessment

Unclear. The requested no-disconnect outcome is the right product target, but the literal start-new-first approach needs maintainer design for same-port handoff, process supervisors, persisted config timing, and channel ownership before it is clearly the narrowest maintainable fix.
