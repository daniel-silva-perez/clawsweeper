## Summary

Keep open. Current main still has restart drain, preflight, update-restart health checks, stale-listener cleanup, and diagnostics, but ordinary gateway restarts still close the active Gateway before any replacement is proven healthy, so the requested no-disconnect rollback behavior remains unresolved.

## What This Changes

Gateway restart should keep the previous gateway instance reachable until a replacement has started and passed a health check, so failed restarts do not disconnect all user access paths.

## Best Possible Solution

Design and implement a transactional, supervisor-aware Gateway restart path: validate changes before mutation, stage or probe the replacement, prove the replacement is reachable, switch ownership of the endpoint and channel transports, and stop the old Gateway only after the new instance is healthy. If startup fails, keep the old Gateway reachable and report diagnostics through existing surfaces and, where feasible, the active channel.

## Reproduction Assessment

Yes. The source-level reproduction path is a SIGUSR1/config/daemon restart where replacement startup fails after `server.close`; current main has no path that restores the prior listener afterward.

## Solution Assessment

Unclear. The no-disconnect outcome is the right product target, but the literal start-new-first proposal needs maintainer design for same-port binding, process supervisors, config mutation timing, and channel ownership before it is the narrowest maintainable fix.
