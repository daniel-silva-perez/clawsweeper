## Summary

Keep open. Current main has restart preflight, drain, respawn health checks for update restarts, stale-listener cleanup, and post-restart diagnostics, but the normal restart path still closes the active Gateway before a replacement is proven healthy, so the requested no-disconnect handoff remains unsolved.

## What This Changes

The issue requests an atomic Gateway restart handoff that keeps the previous instance serving until a replacement starts and passes health checks.

## Best Possible Solution

Keep the issue open and design a conservative Gateway restart handoff: validate config before mutation, start or stage a replacement through a supervisor-aware path, prove readiness, switch the advertised/connectable endpoint, and stop the old Gateway only after the replacement is healthy. If startup fails, preserve the old Gateway and send a failure notice through the still-active channel when feasible.

## Reproduction Assessment

Yes. The source-level reproduction path is a SIGUSR1/config/daemon restart where replacement startup fails: current main closes the active Gateway before restart handling, then logs recovery diagnostics with no old listener to fall back to. I did not run a destructive live restart because this review is read-only.

## Solution Assessment

Unclear. The requested no-disconnect outcome is the right product target, but the literal start-new-first plan needs maintainer design for port binding, process supervisors, and channel ownership before it is the narrowest maintainable fix.
