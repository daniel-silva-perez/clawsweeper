## Summary

Keep open for maintainer/security follow-up. Current main intentionally blocks `OPENCLAW_CONFIG_PATH` from workspace `.env` through both an explicit key block and the fail-closed `OPENCLAW_` prefix policy, while this PR would allow that runtime-control key from an untrusted workspace file.

## What This Changes

The PR removes `OPENCLAW_CONFIG_PATH` from the workspace dotenv blocklist, precomputes the global dotenv path before workspace loading, and updates dotenv tests to allow project-local config while keeping `OPENCLAW_STATE_DIR` blocked.

## Best Possible Solution

Keep the current fail-closed workspace dotenv policy unless maintainers explicitly approve a safer project-local config selection mechanism. If project-local config selection is still desired, design it as a documented trusted opt-in with tests proving it cannot redirect global `.env`, state directories, credentials, plugin roots, or other runtime controls.

## Reproduction Assessment

Yes. Current-main source and tests give a high-confidence reproduction path: a workspace `.env` containing `OPENCLAW_CONFIG_PATH=...` is loaded through `loadWorkspaceDotEnvFile`, `loadDotEnv`, or `loadCliDotEnv`, and the key remains unset. I did not run tests because this review was read-only.

## Solution Assessment

No. This PR is no longer the best way to solve the linked bug because it creates an exception to a later shipped fail-closed `OPENCLAW_*` workspace dotenv policy; a safer solution needs explicit maintainer/security product direction.
