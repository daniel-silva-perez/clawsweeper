## Summary

Keep open. Current main does not implement this PR’s intended behavior: workspace `.env` still blocks `OPENCLAW_CONFIG_PATH`, tests assert that it remains unset, and the changelog documents a later stricter policy that blocks all `OPENCLAW_*` keys from untrusted workspace `.env` files. The PR is small and focused, but it touches a security-sensitive dotenv path and now conflicts with current fail-closed workspace dotenv policy rather than being already implemented.

## Best Possible Solution

Do not auto-close this PR as implemented. Maintainers should decide whether project-local config selection via workspace `.env` is still desired after the stricter workspace dotenv policy. If yes, implement a narrowly documented safe mechanism with tests proving no global `.env` or state-dir redirect; if no, close the PR and linked bug with the security rationale and supported alternatives such as shell env or trusted global `.env`.
