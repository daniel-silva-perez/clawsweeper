## Summary

Keep open. Current main still lacks both requested platform paths: cron has no zero-LLM direct execution payload, and skill env injection does not expose a foreground reply route for later background sends. The reporter reconfirmed the SignalRadar ClawHub workaround on 2026-04-29, so this is a current product/security decision rather than stale cleanup.

## What This Changes

Request a zero-LLM background notification path for skills, either through a directExec cron payload with announce delivery or foreground reply-route env vars that system crontab jobs can persist and reuse.

## Best Possible Solution

Expose one explicit zero-LLM notification contract that preserves channel permissions: either scoped foreground reply-route injection for user-authorized skill runs, or an opt-in directExec cron payload restricted to approved installed-skill commands and wired through existing cron delivery, logging, timeout, and failure-notification semantics.

## Reproduction Assessment

Yes. Static current-main inspection is enough: directExec is absent from the cron type/schema/CLI/agent-tool surfaces, isolated cron execution rejects non-agentTurn payloads, and skill env injection does not expose OPENCLAW_REPLY_* route variables.

## Solution Assessment

Unclear until maintainers choose the contract. Option A is the more general scheduler fix but needs host-exec security controls; Option B is smaller but still needs consent, account/thread scoping, persistence, and revocation semantics.
