## Summary

Keep open. Current main still lacks both proposed platform paths: cron payloads are limited to `systemEvent` and `agentTurn`, non-main cron jobs require `agentTurn`, and isolated cron execution enters the agent runner before delivery. Skill environment injection exists for configured skill env/API keys and CLI MCP loopback metadata, but there is no `OPENCLAW_REPLY_CHANNEL` / `OPENCLAW_REPLY_TARGET` foreground injection or direct zero-LLM cron execution mode. The SignalRadar ClawHub skill and webhook workaround are useful context, but this report identifies a missing core scheduler/delivery surface rather than a skill-only implementation gap.

## Best Possible Solution

Keep this issue open for a focused core design/implementation. The conservative path is an additive scheduler/delivery API: either an opt-in `directExec` payload with strict execution policy and announce semantics, or foreground route env injection with clear docs/tests so skill-owned crontab jobs can call `openclaw message send` without LLM turns. The existing SignalRadar ClawHub skill and webhook workaround should be acknowledged as prior art, but they do not remove the missing OpenClaw platform surface.

## Risks / Open Questions

- A `directExec` cron mode would run arbitrary commands from scheduler state; it needs an explicit opt-in, audit trail, command/skill ownership boundary, and sandbox/exec-policy story before implementation.
- Route env injection must avoid leaking chat IDs, account IDs, or channel context into unrelated child processes or long-lived global process state.
