## Summary

Keep open. Current main still lacks both requested platform paths: cron has no zero-LLM direct-execution payload, and skill execution does not inject foreground reply-route environment variables. The SignalRadar ClawHub context is a concrete use case, but the remaining work needs maintainer product/security direction before a safe implementation can be queued.

## What This Changes

The issue requests either a zero-LLM cron direct-execution payload that can announce output back to the chat route, or foreground reply-route environment injection for skills so system crontab jobs can send later notifications.

## Best Possible Solution

Add one explicit platform contract that preserves zero-LLM execution without weakening channel permissions: either scoped reply-route injection for foreground skill runs, or an opt-in directExec cron payload restricted to approved installed-skill commands and wired through existing cron delivery, run logs, timeout, and failure-notification semantics.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: a `directExec` payload is absent from the type/schema/tool/CLI/UI surfaces and would fail current cron validation, while `git grep` finds no `OPENCLAW_REPLY_*` reply-route env injection path.

## Solution Assessment

Unclear until maintainer direction. Option A is the more general cron solution but has command-execution security implications; Option B is narrower but still needs explicit consent, target scoping, and persistence semantics for reply routes.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive even without a patch because both proposed contracts expose powerful background delivery or host execution behavior.

Concerns:

- **[high] Scope direct command execution before implementation:**
  - body: A directExec cron payload would run commands as the Gateway process user, so it needs explicit opt-in, source restrictions, timeout/resource limits, auditability, and clear interaction with existing exec/sandbox policies.
  - confidence: 0.9
- **[medium] Define reply-route persistence permissions:**
  - body: Reply-route env injection would let skills capture channel/account/thread targets for later background sends, so maintainers need to define consent, lifetime, account/thread scope, and revocation behavior before exposing it broadly.
  - confidence: 0.86

## Risks / Open Questions

- A directExec cron mode would execute configured commands as the Gateway process user unless gated by explicit operator opt-in, installed-skill source restrictions, timeouts, audit logs, and delivery policy.
- Foreground reply-route env vars are privacy and permission sensitive because a skill could persist channel/account/thread targets and send later messages outside the original foreground context.
- The issue intentionally offers two viable solution paths, so implementation scope is not settled until maintainers choose the product contract.
