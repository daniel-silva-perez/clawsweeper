## Summary

Keep open. Current main still lacks both requested platform paths: cron has no zero-LLM direct execution payload, and skill execution does not inject foreground reply-route environment variables. The SignalRadar ClawHub use case is concrete and recently reconfirmed, but the remaining work needs maintainer product/security direction before it is a safe automated fix lane.

## What This Changes

Add zero-LLM background delivery for skills, either through a security-gated directExec cron payload or through foreground reply-route environment variables that skills can persist for later sends.

## Best Possible Solution

Add one explicit platform contract that preserves zero-LLM background checks without weakening channel permissions: either scoped reply-route injection for foreground skill runs, or an opt-in directExec cron payload restricted to approved installed-skill commands and wired through existing cron delivery, run logs, timeout, and failure-notification semantics.

## Reproduction Assessment

Yes. Static current-main inspection is enough: a directExec payload is absent from the type/schema/tool/CLI/UI surfaces and would fail cron validation, while repo search finds no OPENCLAW_REPLY_* route injection path for skills.

## Solution Assessment

Unclear until maintainer direction. Option A is the more general cron solution but needs explicit host-exec security controls; Option B is narrower and lower risk but still needs consent, target scoping, and persistence semantics.
