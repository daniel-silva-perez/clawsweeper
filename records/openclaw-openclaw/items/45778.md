## Summary

Current main still lacks both proposed platform paths. Cron supports only `systemEvent` and `agentTurn`, detached cron execution still runs through the isolated agent runner, and skill env injection still only applies configured skill env/apiKey values rather than foreground chat reply-route variables. The SignalRadar ClawHub context shows a real use case, but the directExec option needs maintainer security/product review before implementation.

## Best Possible Solution

Keep this open for maintainer design review. The likely product path is a narrow platform seam: either foreground reply-route injection with clear channel/account/thread scoping and permission semantics, or an opt-in directExec cron payload that runs only approved installed-skill commands and reuses existing cron delivery, run-log, timeout, and failure-notification behavior without invoking a model.

## Risks / Open Questions

- A directExec cron mode would execute configured commands as the Gateway process user unless scoped by explicit operator opt-in, source restrictions, timeouts, audit logs, and delivery policy.
- Foreground reply-route env vars are privacy and permission sensitive because skills could persist channel/account/thread targets and later send messages outside the original foreground context.
- The issue intentionally offers two different solution paths, so an implementation needs maintainer product direction before a narrow automated fix is safe.
