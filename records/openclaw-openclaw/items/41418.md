## Summary

Current main still does not implement a global `openclaw agent --dry-run` mode. The agent CLI, Gateway request schema, embedded agent options, docs, and Pi tool-definition adapter all lack a request-scoped dry-run path, and managed tools still execute for real while client tools are delegated as pending rather than returned as structured `dry_run` previews.

## Best Possible Solution

Keep this issue open as core agent execution work. Review the reporter's linked implementation branch as a starting point, but implement it as request-scoped plumbing through `openclaw agent`, Gateway `AgentParamsSchema`, embedded `AgentCommandOpts`, and both Pi tool-definition adapter paths. Add redacted stderr previews, structured `status: "dry_run"` results for managed and client tools, docs, and focused adapter/protocol/CLI regression coverage. Keep the narrower command-specific dry-run issues separate unless maintainers choose to create an explicit umbrella.

## Risks / Open Questions

- A process-global dry-run toggle would be unsafe for Gateway or concurrent embedded runs; this should be request-scoped through the CLI, Gateway protocol, and agent command options.
- Preview logging must reuse existing redaction and console sanitization before printing tool parameters to stderr.
- Dry-run interception should stay after policy and `before_tool_call` hooks so preview mode does not bypass safety policy.
- Client-delegated tools need explicit semantics: returning `dry_run` previews must not accidentally trigger the current `onClientToolCall` delegation path.
