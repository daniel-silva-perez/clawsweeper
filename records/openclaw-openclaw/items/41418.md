## Summary

Current main still does not implement a global `openclaw agent --dry-run` mode. The agent CLI, Gateway agent request schema, embedded agent options, docs, and Pi tool-definition adapter all lack request-scoped dry-run plumbing; managed tools still call `tool.execute(...)`, and client tools still return `status: "pending"` instead of structured dry-run previews. The related dry-run items are narrower command-specific requests, so this should stay open for core agent execution work.

## Best Possible Solution

Keep this issue open as core agent execution work. Review the reporter's linked implementation branch as useful context, but implement the feature with request-scoped plumbing through `openclaw agent`, Gateway `AgentParamsSchema`, embedded `AgentCommandOpts`, and both Pi tool-definition adapter paths. Add redacted stderr previews, structured `status: "dry_run"` results for managed and client tools, docs for `openclaw agent --dry-run`, and focused adapter/protocol/CLI regression coverage. Keep the narrower command-specific dry-run issues separate unless maintainers choose to create an explicit umbrella.

## Risks / Open Questions

- A process-global dry-run toggle would be unsafe for Gateway or concurrent embedded runs; this should be request-scoped through CLI, Gateway protocol, and agent command options.
- Preview logging must reuse existing redaction and console sanitization before printing tool parameters to stderr.
- Dry-run interception should remain after policy and `before_tool_call` hooks so preview mode does not bypass safety policy.
- Client tools need an explicit structured preview path; the current `pending` delegated result is not equivalent to a dry-run preview.
