## Summary

Keep open. Current main still does not implement the PR’s shared own-line `NO_REPLY` handling across reply, directive, gateway, and subagent paths, but the PR is not mergeable as written because it has blocking correctness findings and overlaps newer current-main cron suppression work.

## What This Changes

The PR adds an `isSilentTokenOnOwnLine` helper, applies own-line `NO_REPLY` suppression across reply normalization, directive parsing, cron, subagent follow-up, gateway streaming/history paths, and adds focused regression tests.

## Best Possible Solution

Consolidate the overlapping `NO_REPLY` work into one maintained fix: add a cached, case-insensitive own-line/trailing silent-intent helper, apply it consistently to reply normalization, directive/outbound planning, cron delivery, subagent follow-up, gateway live projection, and chat history, while preserving media, interactive, JSON-envelope, and channelData payload semantics with focused regression tests.

## Reproduction Assessment

Yes. Current main has a static reproduction path through `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })`, which existing tests still expect to deliver the preamble; the PR’s own inline directive test also fails by static trace because inline tokens are not stripped in `parseReplyDirectives`.

## Solution Assessment

No. The direction is correct, but this PR is not the best mergeable form because it needs case-insensitive matching, parser/test alignment, channelData preservation, regex caching in a hot path, and a rebase around current-main cron and JSON silent-reply changes.

  - body: The added test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but this patch only clears text for exact or own-line silent replies. Inline text still returns unchanged, so the new test fails unless the parser also strips inline tokens or the expectation is corrected.
  - confidence: 0.97

  - body: Existing silent-token helpers and tests accept mixed-case forms such as `no_reply`, and current cron coverage includes `No_Reply`. The new own-line regex has no `i` flag, so `summary\nNo_Reply` can still leak on paths that rely on this helper.
  - confidence: 0.91
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line `NO_REPLY` text as `isSilent` makes outbound planning treat channelData-only payloads as silent. Group/internal allow-policy paths then drop the entry, and direct rewrites can replace the intended empty-text card/button payload, so channelData delivery semantics regress.
  - confidence: 0.84
- **[P3] Cache the own-line regex:** `src/auto-reply/tokens.ts:64`
  - body: This helper compiles a new `RegExp` on every call while the PR adds it to streaming and polling hot paths. Cache it like the neighboring silent-token regex helpers to avoid avoidable per-chunk allocation.
  - confidence: 0.79
