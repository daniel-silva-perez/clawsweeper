## Summary

Keep open. The PR targets a real remaining silent-reply gap on current main, but the branch is not merge-ready because it has blocking correctness findings and overlaps newer silent-reply fixes that landed after the branch was opened.

## What This Changes

The PR adds an own-line `NO_REPLY` detector and applies it across reply normalization, directive parsing, cron/subagent delivery, gateway streaming, chat history filtering, and regression tests.

## Best Possible Solution

Consolidate the overlapping `NO_REPLY` work into one maintained patch: use a cached, case-insensitive shared helper for own-line silent intent, apply it consistently to normalization, outbound planning, cron, subagent follow-up, live projection, and history display, and preserve media, interactive payloads, direct rewrite policy, JSON envelopes, and `channelData` semantics with targeted regression coverage.

## Reproduction Assessment

Yes. Static current-main reproduction is high confidence: the existing normalizer test still expects `text\n\nNO_REPLY` to deliver the preamble, and the parser/gateway paths still use exact-token or lead-fragment checks rather than own-line silent detection.

## Solution Assessment

No. The direction is correct, but this PR is not the best mergeable form until it is rebased around newer silent-reply fixes and corrected for case-insensitive matching, parser/test alignment, and `channelData` preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Align the inline-token test with parser behavior:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The added test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but this patch only clears text for exact or own-line silent replies. Inline text still returns unchanged, so the test fails unless the parser strips inline tokens or the expectation is corrected.
  - confidence: 0.97
- **[P2] Make own-line token matching case-insensitive:** `src/auto-reply/tokens.ts:64`
  - body: Existing silent-token helpers accept mixed-case forms such as `no_reply`, and current cron coverage includes mixed-case `NO_REPLY`. The new own-line regex has no `i` flag, so `summary\nNo_Reply` can still leak on paths that rely on this helper.
  - confidence: 0.92
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line text as `isSilent` makes outbound planning treat channelData-only payloads as silent because it only excludes media from silence. Group/internal allow-policy paths can drop the entry, and direct rewrite paths can replace the intended empty-text card/button payload.
  - confidence: 0.84
- **[P3] Cache the own-line token regex:** `src/auto-reply/tokens.ts:64`
  - body: This helper compiles a new `RegExp` on every call while the patch adds it to streaming and polling paths. Cache it like the neighboring silent-token regex helpers to avoid avoidable per-chunk allocation.
  - confidence: 0.79
