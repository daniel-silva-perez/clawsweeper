## Summary

Keep open. Current main still does not implement the PR's full own-line NO_REPLY suppression behavior across reply normalization, reply directives, and gateway live/history projection, while a later cron fix partially overlaps the cron portion. The PR also is not merge-ready as-is because the added inline-token test does not match the implementation and the directive-layer change can drop channelData-only deliverables.

## What This Changes

The PR adds an own-line NO_REPLY helper and applies it across reply normalization, reply directives, cron delivery/subagent follow-up, gateway streaming, and chat history filtering, with unit coverage for token parsing and reply normalization.

## Best Possible Solution

Keep the work open and consolidate it with #42873 and #42472: add a shared cached own-line silent-token check, route reply normalization/directives/gateway projection through that shared contract, rebase away the cron overlap, preserve non-text deliverables when text becomes silent, and cover the behavior with focused token, reply-directive, outbound-plan, gateway live/history, and cron follow-up tests.

  - body: The new test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but this patch only clears text when the reply is exact-silent or own-line silent.   - confidence: 0.97

  - body: Marking own-line text silent here makes outbound planning treat the payload as droppable whenever it has no media. That path does not preserve channelData-only replies, so cards/buttons/custom channel payloads can be lost instead of being sent with empty text.
