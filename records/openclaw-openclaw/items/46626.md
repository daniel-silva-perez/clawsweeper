## Summary

Keep open. Current main still has own-line `NO_REPLY` gaps outside the newer cron-specific hardening, and this PR is useful source work, but the proposed patch is not mergeable as-is because it introduces concrete parser/test, case-sensitivity, and channel-data regressions while overlapping newer silent-reply changes.

## What This Changes

The PR adds an own-line `NO_REPLY` detector and wires that suppression into auto-reply normalization, directive parsing, cron/subagent delivery, gateway live/history filtering, and related regression tests.

## Best Possible Solution

Consolidate the silent-reply contract in one maintained patch: use a cached, case-insensitive shared helper for own-line silent intent, apply it consistently to normalization, outbound planning, cron/subagent follow-up, live projection, and history display, and preserve media, interactive payloads, direct rewrite policy, JSON envelopes, and `channelData` semantics with targeted regression tests.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` is still expected by tests to return the preamble, and parser/live/history paths use exact or lead-fragment checks rather than own-line silent detection.

## Solution Assessment

No. The direction is correct, but this branch is not the best mergeable solution until it is rebased around newer silent-reply fixes and corrected for inline parser expectations, case-insensitive matching, and `channelData` preservation.

  - body: The added test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the patch only clears text for exact or own-line silent replies. Inline text returns unchanged, so this test fails unless the parser strips inline tokens or the expectation is corrected.
  - confidence: 0.97

  - body: Existing silent-token helpers match mixed-case variants, and current cron coverage includes mixed-case `NO_REPLY`. The new own-line regex lacks the `i` flag, so `summary\nNo_Reply` can still leak on paths that rely on this helper.
  - confidence: 0.92
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line text as `isSilent` makes outbound planning treat channelData-only payloads as silent because it only excludes media from silence. Group/internal allow-policy paths can then drop the payload instead of sending the channel-specific data with empty text.
  - confidence: 0.84
- **[P3] Cache the own-line token regex:** `src/auto-reply/tokens.ts:64`
  - body: This helper compiles a new `RegExp` on every call while the patch adds it to streaming and polling paths. Cache it like the neighboring silent-token regex helpers to avoid avoidable per-chunk allocation.
  - confidence: 0.79
