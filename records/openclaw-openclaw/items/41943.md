## Summary

Keep this PR open. Current main still does not implement Feishu/Lark Axios HTTP proxy tunneling for ambient HTTPS_PROXY/HTTP_PROXY on the Feishu SDK HTTP path, and the linked Feishu probe report provides concrete reproduction context. The PR is a plausible focused fix, but it is not safe to close as obsolete and should be revised before merge.

## Best Possible Solution

Keep this PR open and request a revision against current main: implement Feishu HTTP proxy support in extensions/feishu/src/client.ts with target-aware HTTPS_PROXY/HTTP_PROXY selection, preserve NO_PROXY through existing OpenClaw proxy helpers or ProxyAgent, handle proxy state deliberately in the Feishu client cache or evaluate it per request, bridge the Lark/Axios options type-safely, keep dependency ownership correct for packaged plugins, and add focused Feishu HTTP proxy plus bypass tests covering the linked probe path.

## Risks / Open Questions

- Merging the current PR shape could ignore NO_PROXY/no_proxy and tunnel Feishu requests that the operator expected to send directly.
- The proxy agent is captured when the cached Feishu client is created, so runtime proxy environment changes may not be reflected until the cache entry is invalidated.
- The PR assigns Axios-only options to Lark.HttpRequestOptions; current upstream types do not declare httpsAgent or proxy.
- Current main's Feishu package no longer declares https-proxy-agent directly, so a rebase should use the existing plugin SDK proxy helper or restore explicit runtime dependency ownership if direct import remains.
- There are no focused Feishu HTTP proxy or bypass regression tests on current main.
