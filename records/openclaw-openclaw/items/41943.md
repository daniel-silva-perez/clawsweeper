## Summary

Keep open. Current main still does not implement this PR's intended Feishu/Lark Axios HTTP proxy tunneling for ambient HTTPS_PROXY/HTTP_PROXY. The linked bug report has concrete Feishu probe reproduction context, and this PR is still a plausible implementation candidate, but it needs revision for NO_PROXY behavior, cache/proxy-state handling, dependency ownership, and focused HTTP proxy tests.

## Best Possible Solution

Keep this PR open and request a revision against current main: implement Feishu HTTP proxy support in extensions/feishu/src/client.ts using target-aware HTTPS_PROXY/HTTP_PROXY selection, preserve NO_PROXY through existing OpenClaw proxy helpers, handle proxy state deliberately in the Feishu client cache or evaluate it per request, keep the Lark/Axios option bridge type-safe, make runtime dependency ownership explicit, and add focused Feishu HTTP proxy and bypass tests.

## Risks / Open Questions

- Merging the current PR shape could route Feishu requests through a proxy even when NO_PROXY should bypass the target host.
- A proxy agent captured when the cached Feishu client is created can become stale if proxy environment variables change while the cache entry remains valid.
- The PR imports https-proxy-agent for Feishu runtime code, but extensions/feishu/package.json currently declares only @larksuiteoapi/node-sdk and typebox as runtime dependencies.
- Current main has no focused Feishu HTTP proxy or bypass regression tests, so a proxy fix could regress silently.
