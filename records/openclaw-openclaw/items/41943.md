## Summary

Keep open. Current main still does not implement this PR's intended Feishu/Lark Axios HTTP proxy tunneling for ambient HTTPS_PROXY/HTTP_PROXY, and this PR remains a plausible implementation candidate for a real Feishu channel gap. It needs revision before merge to preserve NO_PROXY behavior, handle cached client proxy state deliberately, keep option forwarding type-safe, and add focused Feishu HTTP proxy tests.

## Best Possible Solution

Keep this PR open and ask for a revision against current main: implement Feishu HTTP proxy support in extensions/feishu/src/client.ts with target-aware HTTPS_PROXY/HTTP_PROXY selection, preserve NO_PROXY behavior via the existing proxy-env helpers, deliberately handle proxy state in the Feishu client cache or evaluate it per request, keep the Axios/Lark option bridge type-safe, avoid undeclared runtime dependencies, and add focused Feishu HTTP proxy and bypass tests.

## Risks / Open Questions

- Merging the current PR shape could route requests through a proxy even when NO_PROXY should bypass the Feishu/Lark target host.
- A proxy agent captured when the cached Feishu client is constructed can become stale if proxy environment variables change while the cache entry remains valid.
- The current extension package does not declare https-proxy-agent directly; a revision should either use an existing OpenClaw SDK helper or make runtime dependency ownership explicit.
- No focused Feishu HTTP proxy or bypass regression tests exist on current main.
