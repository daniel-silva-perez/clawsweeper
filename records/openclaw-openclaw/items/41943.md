## Summary

Keep open. Current main still does not implement this PR's intended Feishu/Lark Axios HTTP proxy tunneling for HTTPS_PROXY/HTTP_PROXY, and the proposed one-file PR is still a plausible implementation candidate for a real bundled Feishu plugin gap. It needs revision before merge to preserve NO_PROXY behavior, handle cached client proxy state deliberately, keep Axios option forwarding type-safe, declare or avoid direct runtime dependencies, and add focused Feishu HTTP proxy tests.

## Best Possible Solution

Keep this PR open and ask for a revision against current main: implement Feishu HTTP proxy support in extensions/feishu/src/client.ts with target-aware HTTPS_PROXY/HTTP_PROXY selection, preserve NO_PROXY behavior via the existing proxy-env helpers, handle proxy state in the client cache deliberately or evaluate per request, keep Axios option forwarding type-safe, declare or avoid any new direct runtime dependency, and add focused Feishu HTTP proxy and bypass tests.

## Risks / Open Questions

- Merging the PR as-is could route Feishu/Lark HTTP requests through a proxy even when NO_PROXY should bypass the target host.
- A proxy agent captured at Feishu client construction can become stale if proxy environment variables change while the cached client remains valid.
- The local HTTP wrapper is typed as Lark.HttpRequestOptions, so Axios-only options such as httpsAgent and proxy need a deliberate type bridge rather than loose property assignment.
- If the implementation imports https-proxy-agent directly from @openclaw/feishu code, the Feishu package should declare the runtime dependency or use an existing SDK helper.
- No focused Feishu HTTP proxy or bypass regression tests exist on current main.
