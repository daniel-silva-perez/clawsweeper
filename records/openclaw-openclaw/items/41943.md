## Summary

Keep this PR open. Current main still does not implement Feishu/Lark Axios HTTP proxy tunneling for ambient HTTPS_PROXY/HTTP_PROXY, and the linked Feishu probe bug has concrete reproduction context. The PR is a plausible focused fix, but it needs revision before merge for type safety, NO_PROXY behavior, dependency ownership, cache/proxy-state handling, and focused tests.

## Best Possible Solution

Keep this PR open and request a revision against current main: implement Feishu HTTP proxy support in extensions/feishu/src/client.ts with target-aware HTTPS_PROXY/HTTP_PROXY selection, preserve NO_PROXY through existing OpenClaw proxy helpers or an equivalent ProxyAgent path, handle proxy state deliberately in the Feishu client cache or evaluate it per request, bridge the Lark/Axios options type-safely, declare/mirror any runtime dependency ownership, and add focused Feishu HTTP proxy plus bypass tests covering the linked #41820 probe path.

## Risks / Open Questions

- Merging the current PR shape could proxy Feishu requests even when NO_PROXY should bypass the target host.
- The proxy agent is captured when the cached Feishu client is created, so runtime proxy environment changes may not be reflected while the cache entry remains valid.
- The PR appears to assign Axios-only fields to Lark.HttpRequestOptions, whose upstream type does not declare httpsAgent or proxy.
- The Feishu plugin package does not declare https-proxy-agent as a runtime dependency or mirrored root dependency, so packaged plugin installs could fail or depend on undeclared hoisting.
- Current main has no focused Feishu HTTP proxy or bypass regression tests, so the intended fix could regress silently.
