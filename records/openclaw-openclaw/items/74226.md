## Summary

Keep open. The PR targets a real Telegram edit-message length gap, but the current diff still has a correctness bug: it decides to truncate on raw rendered HTML length while computing the truncation footer from raw source length, so entity-expanded or markup-expanded edits can be cut without the promised marker.

## What This Changes

The PR adds `truncateTelegramEditPayload`, applies it to `editMessageTelegram` HTML and plain fallback calls, and adds Telegram send tests for long and under-limit edit payloads.

## Best Possible Solution

A corrected Telegram-plugin edit guard should cap both HTML and plain fallback edit payloads below Telegram's `editMessageText` limit, decide truncation using the Telegram-counted text rather than raw markup length, preserve entity boundaries where possible, always append a visible marker when content is actually cut, and add a regression test for raw text at or below 4000 characters whose rendered HTML exceeds 4096 characters.

## Reproduction Assessment

Yes. Current main can be exercised through `editMessageTelegram` with an over-limit edit or through the native-command progress-result path; the PR's remaining bug is reproducible by calling `truncateTelegramEditPayload` with raw/plain text at or below 4000 characters and rendered HTML above 4096 characters.

## Solution Assessment

No, not as submitted. The Telegram plugin edit path is the right boundary, but the helper should base both the truncation decision and footer reservation on the payload length Telegram actually counts, then cover the entity-expansion case in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Base truncation on Telegram-counted text length:** `extensions/telegram/src/send.ts:215-218`
  - body: The helper truncates whenever `htmlText.length > 4096`, but Telegram documents this limit after entities parsing and `omittedChars` is derived only from `rawText.length - 4000`. For escaped/entity-expanded input such as many `&` characters, raw text can be well under the limit while rendered HTML exceeds it; this code then cuts content and may append no footer at all. Base the guard and omitted-count calculation on the Telegram-counted/plain length, or otherwise ensure every actual cut gets a marker.
  - confidence: 0.93
