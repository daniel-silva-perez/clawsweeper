## Summary

Keep open. Current main still cannot distinguish terminal run-abort reasons from retryable provider failures in the model fallback layer, and the PR addresses that real gap, including the distinct HTTP client-disconnect case not covered by the cron-only related PRs. The branch currently needs maintainer review and rebase because GitHub reports it as non-mergeable against current main.

## What This Changes

The PR adds optional abort-signal propagation through model fallback callers and tags HTTP disconnect aborts so run-budget timeouts, cron timeout strings, and client disconnects stop the fallback chain instead of retrying another model.

## Best Possible Solution

Land a rebased, minimal version of the terminal-abort contract: keep retryable provider failures in the fallback chain, but propagate explicit run-budget and client-disconnect abort reasons immediately, with focused model-fallback tests and caller coverage for agent, cron, auto-reply, and HTTP gateway paths.

## Reproduction Assessment

Yes. Current main source provides a high-confidence static reproduction path: a run timeout or cron timeout aborts the caller signal, but runWithModelFallback cannot inspect that signal and therefore continues through its normal timeout/failover handling.

## Solution Assessment

Yes, conceptually. Inspecting AbortSignal.reason at the fallback boundary is a narrow maintainable seam, and the PR's later updates cover the important normalization and string-reason edge cases; it still needs rebase and maintainer coordination with the cron-specific PRs.
