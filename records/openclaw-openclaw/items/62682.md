## Summary

Keep open. The branch is useful and the earlier review issues were addressed, but it still does not fix the embedded runner timeout path described by #60388: the PR passes the caller abort signal into fallback, while `scheduleAbortTimer` aborts a private per-attempt controller inside `runEmbeddedPiAgent`. The PR also needs the required user-facing changelog entry before it is merge-ready.

## What This Changes

The PR adds terminal abort detection to model fallback, threads optional abort signals through agent/auto-reply/cron callers, tags HTTP client disconnect aborts with `ClientDisconnectError`, and adds unit coverage for terminal abort reasons.

## Best Possible Solution

Revise the PR so the actual embedded-run timeout path exposes an explicit terminal marker to `runWithModelFallback` without conflating provider timeouts, keep the client-disconnect tagging if maintainers want that behavior, and add the required Unreleased changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection gives a focused path: run an embedded agent attempt with a short `timeoutMs` and multiple fallbacks, then assert that the embedded runner's own `scheduleAbortTimer` stops after the first candidate; the current PR unit tests only cover a synthetic external `AbortSignal`.

## Solution Assessment

Is this the best way to solve the issue? No. The external-signal mechanism is a good fit for HTTP disconnects and cron wrapper timeouts, but it does not carry the embedded runner's private run-budget timeout into the fallback layer, so the central #60388 path needs a more direct contract.
