## Summary

Keep open. Current main does not implement this PR's terminal-abort handling: `runWithModelFallback` still has no `abortSignal` parameter, `shouldRethrowAbort` still treats timeout-shaped aborts as retryable fallback failures, and HTTP client disconnects still call `abortController.abort()` without a typed reason. The PR remains a plausible implementation candidate for open issue #60388, with adjacent but unmerged cron work in #42482 and #52365.

## Best Possible Solution

Keep this PR open for maintainer review or land an equivalent focused fix: thread the caller abort signal into `runWithModelFallback`, recognize terminal run-budget and client-disconnect reasons before failover normalization, tag HTTP disconnect aborts, and add model-fallback regression coverage. Coordinate the cron-specific pieces with #42482 and #52365 rather than closing this PR as already implemented.
