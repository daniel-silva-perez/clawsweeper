## Summary

Keep open. Current main and v2026.4.27 contain relevant stale device-token cleanup, recovery docs, and service token-drift diagnostics, but that is not enough to close this fresh report: the reported Linux systemd/control-UI failure reproduced through v2026.4.26, has not been verified on v2026.4.27, and the active code can still emit the exact token/device-token rejection path when the gateway token source, browser cached device token, or pairing record is stale.

## What This Changes

The issue reports a regression where the browser Control UI WebSocket handshake is rejected with `token_mismatch` and `device_token_mismatch` on Ubuntu Kylin 20.04 after clean installs of v2026.4.23 through v2026.4.26.

## Best Possible Solution

Reproduce the Control UI connect on a Linux systemd install using v2026.4.27 or current main, compare the dashboard URL token, resolved `gateway.auth.token`, effective service environment, stored browser device token, and paired-device record, then fix only the narrow drift or cache-repair gap that is proven without weakening gateway auth.

## Reproduction Assessment

No high-confidence reproduction is available yet. The issue gives a plausible step sequence and current source confirms the matching auth rejection paths, but the missing logs/token-source comparison and lack of a live Ubuntu Kylin 20.04 run leave the root cause unproven.

## Solution Assessment

Unclear. The v2026.4.27 stale-device-token cleanup is a plausible partial fix, but a best fix for this issue needs maintainer confirmation of whether the failing credential is the shared gateway token, the systemd service token source, the browser cached device token, or the paired-device record.
