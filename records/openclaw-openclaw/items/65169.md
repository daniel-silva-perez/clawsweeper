## Summary

Keep this PR open for maintainer/security review. Current main still exposes the reported pre-approval node command path and the PR is the linked implementation candidate for the still-open #65168, but the patch needs an amendment because approved commands would remain absent from a live node session after approval until reconnect.

## What This Changes

The PR changes node connection reconciliation to publish an empty command list for unpaired nodes and updates gateway tests to pre-approve invoke fixtures and assert `node.invoke` is blocked before node pairing exists.

## Best Possible Solution

Land a fail-closed node command gate that hides commands before node-pair approval, then makes approved commands available through a deliberate post-approval path such as updating the live node session or having `node.invoke` and `node.list` consult approved pairing state. The regression tests should cover pre-approval blocking, post-approval availability, and the existing approved-node invoke flows.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from current main source and tests: connect a device-paired node with `system.run` before node-pair approval, observe the live command list contains the declared commands, then invoke through an `operator.write` session.

## Solution Assessment

No, not as submitted. The fail-closed line is the right boundary for the bypass, but the patch should also preserve post-approval usability for an already-connected node or explicitly add a direct paired-state check in `node.invoke` with matching tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Refresh live commands after node pairing approval:** `src/gateway/node-connect-reconcile.ts:84`
  - body: With this change, the first unpaired connection is registered with `commands: []`. `node.pair.approve` only writes the paired record and broadcasts `node.pair.resolved`, while `node.list` and `node.invoke` keep using the live `NodeSession.commands`, so approving the pending request leaves that connected node unable to expose or run the newly approved commands until it reconnects. Please update the live session or have the invoke/list paths consult the approved pair, and add a post-approval assertion.
  - confidence: 0.89
