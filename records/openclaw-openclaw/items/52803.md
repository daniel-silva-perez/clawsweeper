## Summary

Keep open. Current main has useful backend/session primitives, `/tasks` chat status, and `/kill`/`/steer` command controls, but it still lacks the requested Control UI orchestration board: no dedicated hierarchy/tree route, no active/failed/completed UI filter, no virtualized large-session surface, and no bulk kill/steer/retry/cleanup controls beyond deleting selected sessions.

## Best Possible Solution

Keep this issue open and implement a dedicated Control UI orchestration surface that consumes existing `sessions.list` hierarchy/status/model/runtime fields plus task records where useful. The likely product path is a tree or virtualized list grouped by agent -> main session -> subagent/task, with active/failed/completed filters, a history toggle, and multi-select actions wired to scoped controls such as `chat.abort`, `sessions.steer` or command equivalents, `sessions.delete`, and `tasks cancel` once retry semantics are explicitly defined.

## Risks / Open Questions

- The request is broad enough that maintainers may want to split it into smaller UI, backend, and UX issues before implementation.
- Bulk kill/steer/retry controls need careful authorization and session-scope handling so the Control UI does not expose cross-session or cross-agent actions beyond the operator's allowed scope.
- Retry semantics are not defined by the existing sessions UI; maintainers should decide whether retry means restart a child session, re-run a task record, or steer an active run before adding controls.
