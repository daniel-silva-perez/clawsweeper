## Summary

Keep this PR open, but do not merge it as-is. Current main still has the order-dependent LAN address selection that the PR is trying to fix across gateway status/dashboard, setup-code QR pairing, and the bundled device-pair plugin, but the PR head introduces a TypeScript compile blocker by calling `fs.readFileSync` in `src/gateway/net.ts` without importing `node:fs`.

## What This Changes

The PR adds LAN IPv4 selection that prefers default-route or physical/default-looking interfaces over Docker/bridge/container/CNI interfaces, wires it into gateway LAN URL resolution and setup-code pairing, mirrors the behavior in the device-pair plugin, and adds regression tests.

## Best Possible Solution

Land this PR or an equivalent fix after adding the missing `node:fs` import, rebasing onto current main, and validating the touched gateway/setup-code/device-pair paths. The mirrored plugin logic is acceptable for this narrow fix because bundled plugins follow the third-party plugin boundary and should not import core internals.

## Reproduction Assessment

Yes. Current main can be reproduced with a network-interface snapshot where a `br-*` private IPv4 appears before a real LAN interface and neither `en0` nor `eth0` is present; the current helpers return the first matching address on the dashboard/status, setup-code, and device-pair paths.

## Solution Assessment

No, not in the current PR head. The product direction is the right narrow fix, but the patch must first import `node:fs` in `src/gateway/net.ts` and then be rebased and validated before it is mergeable.
