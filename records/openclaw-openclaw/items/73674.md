## Summary

Keep this PR open. Current main still has the QMD-on-Windows gap: both Windows `.cmd` shim resolvers only accept quoted `%~dp0`-relative targets, QMD availability still passes the stale `packageName: "qmd"`, and current tests do not cover quoted absolute JavaScript entrypoints. The PR is a focused implementation candidate, not obsolete cleanup, but it needs maintainer review because it broadens a generic public Windows spawn helper used outside QMD.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is to land a narrowly reviewed fix that supports quoted Windows absolute and UNC JS/EXE shim entrypoints without shell execution, keeps both resolver copies synchronized, updates all QMD package fallback call sites that should use `@tobilu/qmd`, and includes portable regression coverage for the generic resolver plus the QMD availability/runtime paths.

## Risks / Open Questions

- The PR touches a generic Windows spawn parser that is exported through the plugin SDK and reused by ACP/Codex/ACPX/Lobster-style paths, so maintainers should review whether accepting any existing quoted absolute JS/EXE token can select the wrong quoted argument in unusual wrappers.
