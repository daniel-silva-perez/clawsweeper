## Summary

Keep open. Current main still has the reported Windows QMD `.cmd` wrapper gap and still uses the stale unscoped QMD package fallback, while this PR is an active, focused implementation candidate. I did not find a definite blocking correctness or supply-chain issue in the diff, but the Windows command-resolution path needs maintainer review and targeted validation before merge.

## What This Changes

The PR extends QMD Windows `.cmd` shim parsing for quoted absolute entrypoints, switches QMD fallback resolution to `@tobilu/qmd`, adds focused resolver and memory-core test coverage, and changes the memory-host Windows spawn helper from a re-export into a local implementation.

## Best Possible Solution

Land a narrow, rebased fix after maintainer and security validation: accept existing quoted absolute JS/EXE targets from Windows `.cmd` wrappers without shell execution, use `@tobilu/qmd` consistently for QMD fallback resolution, and preserve shared source/package resolver behavior so command policy does not drift.

## Reproduction Assessment

Yes. A static current-main check gives a high-confidence reproduction path: create or inspect a wrapper containing `node "E:\\...\\@tobilu\\qmd\\dist\\cli\\qmd.js" %*`; the parser has no `%~dp0` match and reaches the existing unresolved-wrapper error.

## Solution Assessment

Mostly yes. Extending direct wrapper parsing and using the scoped QMD package name is the narrow maintainable fix for the reported behavior; the safer merge shape should keep the memory-host package path delegated to the central resolver or otherwise prove the two implementations stay aligned.
