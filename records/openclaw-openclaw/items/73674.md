## Summary

Keep this PR open. Current main still has the reported QMD Windows gap: both Windows `.cmd` shim resolvers only handle quoted `%~dp0`-relative targets, QMD availability still uses the stale `packageName: "qmd"`, and tests do not cover quoted absolute JavaScript entrypoints. The PR remains a focused implementation candidate, but it should get maintainer review because it changes a public generic Windows spawn helper and appears to miss the extension runtime QMD call site.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is to land a narrow fix that supports quoted Windows absolute and UNC JS/EXE shim entrypoints without shell execution, keeps both resolver copies synchronized, updates all QMD package fallback call sites that should use `@tobilu/qmd`, and adds portable regression coverage for the generic resolver plus QMD availability/runtime paths.
