## Summary

Keep this PR open. Current main still only unwraps quoted `%~dp0`-relative `.cmd` shim targets and still uses `packageName: "qmd"` on QMD spawn paths, while the PR describes a concrete Windows `qmd.cmd` wrapper with a quoted absolute `@tobilu/qmd` JavaScript entrypoint. The patch is focused and relevant, but it needs maintainer review because the proposed test looks non-portable on Linux CI and one QMD execution call site still appears to pass the old package name.

## Best Possible Solution

Keep the PR open and ask for a focused revision or maintainer port: support quoted Windows absolute/UNC JS and EXE entrypoints without shell execution, update all QMD package fallback call sites that need `@tobilu/qmd`, keep the duplicated SDK/package resolvers in sync, and make the regression test portable across Linux CI and Windows runtime semantics.

## Risks / Open Questions

- The added test in the PR constructs the entrypoint with the host `path.join(...)`; on Linux CI that is a POSIX absolute path, but the proposed `isWindowsAbsolutePath` only accepts drive-letter and UNC paths, so the test may fail unless the test or helper is adjusted.
- The patch broadens cmd-shim parsing to quoted absolute paths. Maintainers should verify it cannot accidentally choose a quoted argument path instead of the intended executable or JS entrypoint.
- The package-name update appears limited to availability checks in the diff; the QMD manager execution path still passes `packageName: "qmd"` on current main and may need the same correction.
