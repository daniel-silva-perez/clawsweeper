## Summary

Keep this issue open. Current main still has the reported bundled Signal installer hardening gap: the release asset downloader uses a private raw `node:https` request helper with recursive redirects and direct streaming to disk, while the guarded fetch/proxy/timeout seam exists but is not used there.

## Best Possible Solution

Keep this issue open and fix it in the bundled Signal plugin. Replace the private `node:https` downloader with `fetchWithSsrFGuard` from `openclaw/plugin-sdk/ssrf-runtime`, pass explicit timeout and redirect limits, decide whether trusted env-proxy mode is appropriate for this GitHub release flow, enforce both declared and streamed archive byte caps, sanitize the temporary archive filename, clean partial downloads on failure, and add Signal installer tests for timeout, redirect SSRF blocking, size caps, and cleanup.

## Risks / Open Questions

- The release asset request can still hang if the server stalls because the raw `node:https` request has no explicit timeout or abort wiring.
- Redirect chains from GitHub release asset URLs are still followed without validating each redirected target through the shared SSRF guard.
- The archive response streams directly to disk without declared or incremental byte limits, so an oversized response can consume local disk during setup.
- The eventual fix should also choose a conservative Signal archive size cap, sanitize the archive filename before joining it into the temp directory, remove partial downloads on failure, and add focused regression tests.
