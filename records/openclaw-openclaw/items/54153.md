## Summary

Keep open. Current main partially moved the Signal installer off the original raw `node:https` downloader and now uses the shared SSRF guard plus a streamed 256 MiB cap, but it still does not pass an explicit timeout to `fetchWithSsrFGuard`, lacks focused downloader regression tests, and open PR #73489 is already the active fix candidate linked for this issue.

## What This Changes

Request hardening the Signal plugin’s signal-cli release downloader with timeout, SSRF redirect validation, download size limits, and proxy-aware guarded fetching.

## Best Possible Solution

Review and land #73489 or an equivalent Signal plugin fix that completes the downloader hardening on top of current main: explicit fetch timeout, declared and streamed size checks, conservative proxy-mode choice, partial-download cleanup, safe archive path handling, focused Signal installer tests, and a changelog entry if the final change is user-facing.
