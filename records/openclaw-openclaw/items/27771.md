## Summary

Keep open. The requested Signal config-directory feature is valid and still missing from current main, but this PR is not merge-ready because it wires the obsolete `src/signal` runtime while current main uses the bundled Signal plugin under `extensions/signal`.

## What This Changes

This PR adds optional Signal `configPath` schema/type/help/docs/configured-state entries and legacy `src/signal` runtime/tests to pass a trimmed `--config` argument when autostarting `signal-cli`.

## Best Possible Solution

Port optional top-level and per-account Signal `configPath` support through the current bundled plugin end to end: strict schema/types, generated channel metadata, UI hints, docs, account resolution, monitor autostart propagation, daemon argv construction before `daemon`, blank-value filtering, focused tests, and changelog credit for the source contribution.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: a `channels.signal.configPath` setting is absent from strict Signal config validation/types, and the current Signal monitor/daemon cannot pass a config directory to `signal-cli`.

## Solution Assessment

No. The feature direction is sound, but this PR edits obsolete runtime paths; the maintainable solution is a narrow replacement patch against `extensions/signal` and the current generated config/docs surfaces.
