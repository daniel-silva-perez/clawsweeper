## Summary

Keep open. The requested Signal behavior is valid and upstream signal-cli supports a global `--config` option, but this PR is not merge-ready against current main because it ports runtime behavior into obsolete `src/signal/*` files while the current bundled Signal plugin, strict schema, generated metadata, and docs still omit `configPath`. The author has said they cannot continue, so the useful path is a narrow maintainer or ClawSweeper replacement PR rather than closing the work.

## What This Changes

The PR adds optional `channels.signal.configPath` config/docs/schema/help/type entries and wires that value into the old Signal daemon/monitor path so signal-cli can be started with `--config`.

## Best Possible Solution

Port optional `configPath` support through the current bundled Signal plugin end to end: schema/types, generated channel metadata, config UI hints/help, top-level and per-account account resolution, monitor autostart propagation, daemon argv construction before the `daemon` command, focused tests, docs, and a changelog credit for the source contribution.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the gap: a config using `channels.signal.configPath` is not represented in the strict Signal schema/types and cannot reach the current Signal daemon argv builder.

## Solution Assessment

No. The PR implements the right concept in an obsolete runtime path; the maintainable solution is a replacement patch against `extensions/signal` and the current generated config surfaces.
