## Summary

Keep open. The requested Signal `configPath` support is valid and narrow, but this PR is not merge-ready because it wires the setting through obsolete `src/signal/*` paths while current main runs Signal from the bundled Signal plugin under `extensions/signal`. Current main still lacks the schema/type/docs/metadata/runtime propagation needed for the feature, and the author has said they cannot continue, so this is a good replacement-PR candidate rather than a close.

## What This Changes

The PR adds an optional Signal `configPath` setting to config docs/schema/help/types and passes it as `--config` when the old Signal daemon autostarts.

## Best Possible Solution

Port optional `configPath` support through the current bundled Signal plugin end to end: strict schema/types, generated channel metadata, config UI hints/help, top-level and per-account account resolution, monitor autostart propagation, daemon argv construction before the `daemon` command, focused tests, docs, and a changelog credit for the source contribution.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: a config using `channels.signal.configPath` is not accepted by the strict Signal schema/types and cannot reach the current Signal daemon argv builder.

## Solution Assessment

No. This PR implements the right concept in obsolete runtime files; the maintainable solution is a narrow replacement patch against `extensions/signal` and the current generated config surfaces.
