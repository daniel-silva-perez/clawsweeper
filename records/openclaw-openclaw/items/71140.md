## Summary

Keep open. Current main still supports Telegram media sends and native reaction primitives, but the normal outbound reply path still does not parse `[[react...]]` directives, does not synthesize `channelData.telegram.reaction`, and does not execute reaction-only replies; the issue is also paired with open draft PR #71141 using closing syntax, so it should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue requests that Telegram outbound reply normalization parse reaction and media directives into structured payloads so native Telegram reactions and GIF/media sends happen instead of leaking or dropping control text.

## Best Possible Solution

The desired end state is a shared reply directive parser that emits structured reaction metadata, outbound normalization that preserves it in channel data, and Telegram delivery/adapter code that applies the native reaction to the resolved reply target before optional text/media delivery, with focused parser, payload, delivery, and adapter tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `[[react_to_current:...]]` is not recognized by the inline directive parser and reaction-only Telegram replies still reach the no-text/no-media rejection path instead of native reaction handling.

## Solution Assessment

Yes. The shared-parser plus Telegram-specific delivery/adapter wiring described in the issue and PR #71141 is the narrow maintainable fix; a Telegram-only string special case would leave outbound normalization inconsistent.
