## Summary

Keep open. Current main still has native Telegram media and reaction primitives, but the normal outbound reply path still does not parse `[[react...]]` directives into structured reaction data or execute them; the issue is also paired with open draft PR #71141 using closing syntax, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

Telegram outbound replies should consume reaction and media directives by applying native Telegram reactions and GIF/media sends instead of leaking or dropping directive text.

## Best Possible Solution

The end state should be a shared reply directive parser that emits structured reaction metadata, outbound normalization that preserves it in Telegram channel data, and Telegram delivery/adapter code that applies the native reaction to the resolved reply target before optional text/media delivery, with focused parser, payload, delivery, and adapter tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `[[react_to_current:...]]` is not recognized by the inline directive parser, remains visible text, and reaction-only Telegram replies hit the no-text/no-media rejection path instead of native reaction handling.

## Solution Assessment

Yes. The issue’s proposed shared-parser plus Telegram-specific delivery/adapter wiring is the narrow maintainable fix; a Telegram-only string special case would leave the shared outbound normalization contract inconsistent.
