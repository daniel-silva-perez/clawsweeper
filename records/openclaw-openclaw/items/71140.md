## Summary

Current main still has the reported Telegram reaction-directive gap: media directives are handled, native Telegram reactions exist, but `[[react...]]` is not parsed into structured outbound data and Telegram delivery/adapter paths do not execute it. The issue is also paired with open draft PR #71141 using closing syntax, so it should remain open until that PR is reviewed, merged, or replaced.

## What This Changes

Request to make Telegram outbound replies consume reaction and media directives as native Telegram reactions/GIFs instead of leaking or dropping directive text.

## Best Possible Solution

The end state should extend the shared reply directive parser to emit structured reaction metadata, preserve it through outbound normalization, and have Telegram delivery and sendPayload apply the native reaction to the resolved reply target before optional text/media delivery, with focused parser, payload, delivery, and adapter tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `[[react_to_current:...]]` is not recognized by the inline directive parser, remains visible text, and reaction-only Telegram payloads still hit the no-text/no-media rejection path.

## Solution Assessment

Yes. The shared-parser plus Telegram delivery/adapter wiring described here is the narrow maintainable fix; closing this issue before the linked implementation lands would leave the normal assistant delivery path broken.
