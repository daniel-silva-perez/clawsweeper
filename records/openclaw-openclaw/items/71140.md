## Summary

Keep open. Current main still has Telegram media/GIF delivery and native reaction primitives, but the normal outbound reply path still does not parse `[[react...]]` directives, does not synthesize `channelData.telegram.reaction`, rejects reaction-only Telegram replies before channel data can be applied, and does not execute reaction data in the outbound adapter. The issue is also paired with open draft PR #71141 using closing syntax, so the issue should remain open until that PR or a replacement fix lands.

## What This Changes

The issue requests parsing outbound Telegram reaction/media directives into structured reply data, accepting reaction-only replies, and applying native Telegram reactions/GIF/media sends instead of leaking or dropping directive text.

## Best Possible Solution

The end state should be a shared directive parser that emits structured reaction metadata, outbound normalization that preserves it in channel data, and Telegram delivery/adapter code that applies native reactions to the resolved reply target before optional text/media delivery, with focused parser, payload, delivery, and adapter tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `[[react_to_current:...]]` is not recognized by the directive parser, remains visible text, and a reaction-only Telegram payload reaches the no-text/no-media rejection path instead of native reaction handling.

## Solution Assessment

Yes. The shared-parser plus Telegram-specific delivery/adapter wiring described in the issue and maintainer comment is the narrow maintainable fix; a Telegram-only string special case would leave the cross-surface outbound normalization gap unresolved.
