## Summary

Keep this open. Current main still implements Telegram media extraction/native media sends, but the reaction directive path is missing, and the linked open draft PR is the active implementation candidate.

## What This Changes

The issue asks OpenClaw to parse Telegram outbound reaction directives, preserve them as structured channel data, allow reaction-only replies, and apply native Telegram reactions instead of leaking directive text.

## Best Possible Solution

Ship a focused fix that extends shared reply directive parsing, carries Telegram reaction intent through outbound payload normalization, permits reaction-only Telegram replies when a reply target exists, and invokes the existing native Telegram reaction primitive with parser, adapter, and delivery tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `[[react_to_current:...]]` is not parsed by the inline directive parser, no reaction channel data is emitted, and a reaction-only Telegram reply still reaches the no-text/no-media rejection path.

## Solution Assessment

Yes. The shared-parser plus Telegram delivery/adapter wiring is the narrow maintainable direction; a Telegram-only string special case would leave outbound normalization inconsistent across reply paths.
