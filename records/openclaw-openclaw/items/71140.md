## Summary

Keep this issue open. Current main still has native Telegram media/GIF and reaction transport primitives, but the normal outbound reply path does not parse `[[react...]]` directives, does not synthesize `channelData.telegram.reaction`, and does not execute reaction-only Telegram payloads; the supplied discussion also shows an open draft PR with closing syntax, so the issue should remain open until that PR or a replacement fix lands.

## What This Changes

The issue requests shared outbound normalization and Telegram delivery changes so reaction directives and media directives become native Telegram reactions/GIF/media sends instead of leaking or being dropped.

## Best Possible Solution

Land PR #71141 after review, or replace it with a maintainer patch that extends shared directive parsing, preserves structured Telegram reaction channel data, permits reaction-only replies with a valid target, calls the existing Telegram reaction primitive before optional text/media delivery, and adds parser, payload-normalization, delivery, and adapter tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `[[react_to_current:...]]` is not recognized by the directive parser, so it remains text, and a reaction-only Telegram payload still falls through the no-text/no-media rejection path.

## Solution Assessment

Yes. The best fix is the shared-parser plus Telegram-adapter/delivery path described in the issue and maintainer comment, rather than a Telegram-only string special case, because the bug crosses shared outbound normalization and channel-specific execution.
