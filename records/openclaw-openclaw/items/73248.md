## Summary

Keep open. Current main still builds a flat `<available_skills>` catalog and asks the model to scan skill descriptions; recent compact-format, character-budget, truncation, sorting, and explicit allowlist work mitigates prompt size but does not implement the requested hierarchical, tagged, or lazy skill discovery design.

## What This Changes

The issue requests hierarchical, tagged, or lazy skill discovery so large skill installations do not inject a flat 7K-10K character skills catalog into every model prompt.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A good end state would add an additive, documented narrowing or discovery contract while preserving deterministic ordering, prompt budgets, explicit allowlists, plugin-bundled skills, and SKILL.md on-demand loading.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the remaining structural behavior: current source, docs, and tests show a flat `<available_skills>` catalog with compact/truncation limits. No for the reporter's exact token measurements because no concrete 80-skill fixture or logs were provided.

## Solution Assessment

Is this the best way to solve the issue? Unclear at the design level: the issue identifies a real limitation, but maintainers need to choose between tags/categories, hook-based narrowing, lazy retrieval, or a skill graph. The safest path is an additive contract with regression tests rather than a broad rewrite of skill loading.
