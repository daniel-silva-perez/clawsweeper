## Summary

Keep open. Current main still exposes skills through a flat `<available_skills>` prompt catalog and model-side description scanning; the compact-format, count, character-budget, truncation warning, and explicit allowlist mitigations reduce prompt size but do not implement the requested hierarchical, tagged, or lazy skill discovery design.

## What This Changes

The issue requests a lower-token skills discovery mechanism for large skill installations, such as hierarchical categories, lazy loading, or tag/group-based narrowing instead of injecting one flat XML skill list.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A good end state would add an additive, documented narrowing or discovery contract, such as tags/categories, hook-driven skill filtering, or lazy retrieval, while preserving deterministic ordering, prompt budgets, explicit allowlists, plugin-bundled skills, and SKILL.md on-demand loading.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the remaining structural behavior: current source, tests, and docs show skills are still represented as a flat `<available_skills>` catalog. No for the reporter's exact 80-plus-skill token measurements because no runnable fixture or logs were provided.

## Solution Assessment

Is this the best way to solve the issue? Unclear at the design level: the issue identifies a real limitation, but the narrowest maintainable fix needs a maintainer decision between config tags/categories, hook-based narrowing, and lazy retrieval. The safer path is an additive contract with tests and docs rather than a broad rewrite of skill loading.
