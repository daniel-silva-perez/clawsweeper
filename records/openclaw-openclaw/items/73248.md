## Summary

Keep open. Current main still exposes skills through a flat `<available_skills>` prompt block and asks the model to scan descriptions; the existing compact-format and prompt-budget controls reduce size but do not add hierarchical categories, tag-based discovery, or lazy skill selection. Related #68734 is an open implementation candidate, while #52861 and #74100 overlap on prompt-token overhead and skill graph ideas without superseding this specific design request.

## What This Changes

The issue requests reducing large flat skills prompt injection by adding hierarchical, tagged, or lazy skill discovery so agents avoid scanning 80-plus skill descriptions every turn.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A maintainable end state should add an additive, documented narrowing or discovery contract, such as tags/categories, hook-driven skill filtering, or lazy retrieval, while preserving deterministic ordering, explicit allowlists, prompt budgets, plugin-bundled skills, and on-demand SKILL.md loading.

## Reproduction Assessment

Yes for the structural behavior: current source, docs, and tests show skills still flow through a flat `<available_skills>` catalog. No for the reporter's exact token measurements because no runnable 80-plus-skill fixture or logs were provided.

## Solution Assessment

Unclear at the design level: the issue identifies a real limitation, but maintainers still need to choose between tag/category config, hook-based narrowing, lazy retrieval, or a skill graph. The safer path is an additive contract with focused tests and docs rather than a broad rewrite of skill loading.
