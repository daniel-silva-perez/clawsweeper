## Summary

Keep open. Current main still mitigates skills prompt pressure with compact formatting, prompt character/count budgets, truncation notices, deterministic ordering, and explicit allowlists, but it still exposes skills to the model as a flat `<available_skills>` catalog. The requested hierarchical, tagged, graph-based, or lazy discovery layer is not implemented, and the closest related implementation PR remains open and unmerged.

## What This Changes

The issue requests a scalable skills discovery mechanism that reduces large flat skills prompt injection and model-side selection noise when many skills are installed.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A good end state would add an additive, documented narrowing or discovery contract that reduces injected skill text for large installations while preserving explicit allowlists, prompt budgets, deterministic ordering, plugin-bundled skills, on-demand SKILL.md reading, and prompt-cache stability; coordinate with #68734, #52861, and #74100.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the remaining structural behavior: current source and docs still show a flat `<available_skills>` catalog with model-side scanning and documented per-skill overhead. No for the reporter's exact 7,000-10,000 character measurement because no runnable 80-plus-skill fixture or logs were provided.

## Solution Assessment

Is this the best way to solve the issue? Unclear at the design level: the issue identifies a real remaining limitation, but the implementation should be chosen by maintainers across tags/categories, hook-driven narrowing, graph expansion, or lazy retrieval. A narrow additive contract with prompt-size regression coverage is safer than a broad rewrite of skills loading.
