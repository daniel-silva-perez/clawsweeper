## Summary

Keep open. Current main still builds a flat `<available_skills>` catalog and asks the model to scan descriptions; existing compact-format, prompt-budget, truncation warning, and explicit allowlist controls mitigate size but do not implement hierarchical, tagged, or lazy skill discovery. Related PR #68734 is an open implementation candidate for hook-based narrowing, while #52861 and #74100 track overlapping token-overhead and skill-graph directions.

## What This Changes

The issue requests a lower-overhead skills discovery mechanism so large skill installations do not inject a flat 7K-10K character catalog into every system prompt.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A maintainable end state should add an additive, documented narrowing or discovery contract, such as tags/categories, hook-driven skill filtering, or lazy retrieval, while preserving deterministic ordering, explicit allowlists, prompt budgets, plugin-bundled skills, and on-demand SKILL.md loading.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the structural behavior: current source, docs, and tests show skills still flow through a flat `<available_skills>` catalog. No for the reporter's exact token measurements because no runnable 80-plus-skill fixture or logs were provided.

## Solution Assessment

Is this the best way to solve the issue? Unclear at the design level: the issue identifies a real limitation, but maintainers still need to choose between tag/category config, hook-based narrowing, lazy retrieval, or a skill graph. The safer path is an additive contract with focused tests and docs rather than a broad rewrite of skill loading.
