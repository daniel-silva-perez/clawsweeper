## Summary

Keep open. Current main still builds and injects a flat `<available_skills>` catalog and tells the model to scan skill descriptions; the existing compact-format, count, character-budget, truncation notice, and explicit allowlist controls reduce prompt size but do not implement the requested hierarchical, tagged, or lazy skills discovery layer.

## What This Changes

The issue requests a structured skills discovery mechanism that avoids large flat skills prompt injection and reduces model-side selection noise for large skill catalogs.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A good end state would add an additive, documented narrowing or discovery contract, such as tags/categories, hook-driven skill filtering, or lazy retrieval, while preserving existing prompt budgets, explicit allowlists, plugin-bundled skills, SKILL.md on-demand loading, and prompt-cache stability; it should be coordinated with #68734, #52861, and #74100 rather than closed prematurely.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the remaining structural behavior: current source and docs show the prompt is still a flat `<available_skills>` list with model-side scanning. No for the reporter's exact 80-plus-skill token measurements, because no runnable fixture or logs were provided.

## Solution Assessment

Is this the best way to solve the issue? Unclear at the design level: the issue identifies a real remaining limitation, but the best implementation needs maintainer choice between core config tags, hook-based narrowing, and lazy retrieval. The safest path is an additive contract with tests and docs, not a broad rewrite of skills loading.
