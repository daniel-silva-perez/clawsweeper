## Summary

Keep open. Current main mitigates large skills catalogs with budgets, compact fallback, truncation notices, and per-agent allowlists, but it still exposes a flat `<available_skills>` list and asks the model to scan descriptions or names before selecting a skill. The issue’s requested hierarchical, tagged, or lazy discovery layer remains a real core design question rather than already-shipped behavior.

## What This Changes

The issue requests reducing large skills-prompt overhead by replacing the flat skills catalog with hierarchical, tagged, or lazy skill discovery.

## Best Possible Solution

Keep this open for a core skills discovery design. The end state should preserve the existing budget caps, per-agent allowlists, plugin-bundled skills, and on-demand SKILL.md loading while adding a structured discovery layer such as categories/tags or lazy retrieval, with config schema, docs, and regression coverage proving lower prompt overhead and stable skill selection.

## Reproduction Assessment

Unclear. The code path is easy to inspect and unit-test with synthetic skills, but the report does not provide a current-main fixture with 80-plus installed skills or measured prompt/model-selection impact.

## Solution Assessment

Unclear. Compact fallback and allowlists are good mitigations, but they are not the best final solution for the reported hierarchy/lazy-discovery problem; a design pass should pick a maintainable tagged or retrieval-based discovery contract before implementation.
