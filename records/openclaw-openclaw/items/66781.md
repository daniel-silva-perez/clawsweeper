## Summary

Keep open. Current main narrows the Active Memory symptom and fresh local onboarding defaults to the `coding` profile, but the broader disabled-by-default / explicit-enable request is still unresolved: unset and `full` remain unrestricted, empty runtime allowlists preserve the full tool surface, bundled MCP/plugin tools remain reachable under permissive profiles, and docs still describe broad defaults for sub-agents.

## What This Changes

The issue requests least-privilege agent tool defaults so agents receive only explicitly enabled tools instead of broad built-in, plugin, and media/session tool access by default.

## Best Possible Solution

Define explicit default semantics for per-agent and embedded-run tool policy, preserve an intentional escape hatch for unrestricted runs, add migration or doctor warnings for broad existing configs, update schema/help/docs, and cover core, plugin/MCP, Active Memory, sub-agent, and runtime `toolsAllow` behavior with regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: inspect unset/`full` profile resolution, the no-policy matcher, and empty `toolsAllow` handling to see that broad tool access remains the documented and tested behavior.

## Solution Assessment

No complete solution is present yet. The current Active Memory allowlist and onboarding `coding` default are useful mitigations, but the narrowest maintainable fix needs an explicit maintainer decision on default policy semantics and compatibility handling.
