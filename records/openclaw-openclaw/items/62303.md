## Summary

Keep this issue open. Current main still gates Telegram DM-topic auto-renames on `isFirstTurnInSession`, computed from the session store before the rename block runs, which is the same failure mode the report identifies. I found the original shipped feature in the changelog and current implementation, but no follow-up fix or regression coverage proving that a newly created/provisional DM topic with prior bootstrap/session metadata is now renamed.

## Best Possible Solution

Keep the issue open and fix it in the Telegram plugin. The likely maintainer path is to track whether `autoTopicLabel` has already been attempted per DM topic, or to base the rename decision on topic/session label state instead of only `systemSent`, then add a regression test where a DM topic session entry already exists before the first meaningful user message and still gets `editForumTopic(...)` after a generated label.
