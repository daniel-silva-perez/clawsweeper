## Summary

Current main still lacks both intended guardrails: read-tool missing-path errors are not wrapped with actionable guidance, and the message tool still does not reject unsupported discovered channel actions before dispatch. The PR remains a focused implementation candidate, with one small review/polish issue around duplicate action names in the supported-action error text.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best path is to land a focused version that wraps read-tool ENOENT/ENOTDIR with actionable path guidance, rejects unsupported discovered message actions before dispatch while preserving send/broadcast and empty-discovery behavior, deduplicates the supported-action list, and keeps the new regression tests.
