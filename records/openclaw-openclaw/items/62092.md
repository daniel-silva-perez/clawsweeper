## Summary

Keep open. Current main still aborts auto provider selection unless an adapter explicitly implements shouldContinueAutoSelection, so the PR’s intended behavior is not already implemented. The PR is focused and not a supply-chain/security-sensitive change, but it should address the existing review concern and add regression coverage before merge.

## Best Possible Solution

Keep this PR open for review. The best path is to update the patch to use the transport-based guard, add focused memory-core regression tests for a remote adapter that omits shouldContinueAutoSelection and for a local-transport adapter that should not be softened, then merge or replace it with an equivalent main-branch fix.

## Risks / Open Questions

- The PR’s current guard uses adapter.id !== "local"; the existing review correctly notes adapter.transport !== "local" is the safer predicate for third-party local adapters with non-local ids.
- The PR adds behavior without adding or updating tests; current tests do not exercise the omitted-callback fallback path.
- A similar shouldContinueAutoSelection default-false helper exists in src/gateway/embeddings-http.ts, so maintainers should decide whether the same auth/config soft-failure behavior should apply there or remain memory-core-only.
