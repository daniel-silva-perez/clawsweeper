## Summary

Keep open. The PR usefully expands the parser/type/docs surface for per-agent `verboseDefault` and `elevatedDefault`, but it still leaves the normal reply directive path reading verbose/elevated defaults from the global `agents.defaults` object instead of the per-agent entry, so the validated per-agent values can still be ignored at runtime.

## What This Changes

The PR adds `verboseDefault` and `elevatedDefault` to per-agent config schema/types/generated metadata/docs, extends resolved agent config for elevated defaults, and adds parser plus agent-scope precedence tests.

## Best Possible Solution

Land one consolidated fix for #73680 that makes the strict per-agent schema, public config type, generated metadata, docs, normal reply directive resolution, directive status/current-level helper, and agent-scope resolver all prefer `agents.list[]` values before `agents.defaults`, with regression tests for both parsing and runtime precedence.

## Reproduction Assessment

Yes. The original validation failure is high-confidence from current main's strict schema, and the remaining PR defect is source-reproducible by following a normal reply path where `agentCfg` is `cfg.agents?.defaults` and verbose/elevated defaults are not read from `agentEntry`.

## Solution Assessment

No. Adding schema/type/docs is necessary, but the PR is not yet the best complete fix because the core reply directive resolution still needs to consume the accepted per-agent defaults and prove that precedence with tests.
