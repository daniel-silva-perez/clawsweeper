## Summary

Keep open. The branch addresses the schema/docs/type side of #73680 and includes useful tests, but it still leaves normal reply directive resolution and directive-only current-level reporting reading verbose/elevated defaults from `agents.defaults` instead of the per-agent entry, so the user-visible precedence bug is only partially fixed.

## What This Changes

The PR adds per-agent `verboseDefault` and `elevatedDefault` schema/type/generated metadata/docs, wires `elevatedDefault` into `resolveAgentConfig`, and adds parser plus agent-scope precedence tests.

## Best Possible Solution

Land one consolidated fix for #73680 that aligns `AgentEntrySchema`, public config types, generated metadata, docs, `resolveAgentConfig`, normal reply directive resolution, directive-only current-level reporting, and persistence/ack behavior around per-agent-before-global precedence, with regression tests for both parsing and runtime behavior.

## Reproduction Assessment

Yes. The original issue is source-reproducible from current main because strict `AgentEntrySchema` omits both keys while global defaults accept them, and the remaining PR defect is source-reproducible through the normal `getReply` path where `agentCfg` is `cfg.agents?.defaults`.

## Solution Assessment

No. The schema/type/docs changes are necessary, but this is not yet the best complete fix because accepted per-agent values still need to be consumed by reply directive resolution and current-level helpers before the documented precedence is true.
