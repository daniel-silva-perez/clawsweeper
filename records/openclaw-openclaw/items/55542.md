## Summary

Keep open. The dependency replacement is a real proposal, but the current PR head is not safe to merge: it regresses EXIF normalization format preservation, drops the existing sharp backend override value, and removes sharp build approval while sharp is still present in the resolved dependency graph. Current main has also moved the active image backend behind the media-understanding-core plugin runtime, so the migration needs a maintainer-owned rework against that boundary.

## What This Changes

The PR removes sharp install guidance and the root sharp dependency, adds @napi-rs/image with native build approval and lockfile entries, and ports image-processing code plus image-related tests from sharp to @napi-rs/image Transformer APIs.

## Best Possible Solution

Rework the migration against current main by moving or intentionally retaining the plugin-owned media-understanding-core backend, preserving the public plugin-sdk media contract, keeping backend override compatibility, preserving non-JPEG formats during EXIF normalization, and removing sharp docs/build approvals only after every remaining sharp runtime path is accounted for.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve source format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: `normalizeExifOrientation` now returns `transformer.rotate().jpeg(90)` for every non-sips input. The previous sharp path used `rotate().toBuffer()`, so PNG/WebP callers can now receive lossy JPEG bytes and lose alpha even though they only requested orientation normalization.
  - confidence: 0.94
- **[P2] Keep the legacy sharp backend override working:** `src/media/image-ops.ts:28`
  - body: Changing the non-sips override from `OPENCLAW_IMAGE_BACKEND=sharp` to only `napi-rs` makes existing Bun/macOS installs with the documented `sharp` override silently fall back to sips after upgrade. Keep `sharp` as a compatibility alias or provide an explicit migration path.
  - confidence: 0.87
- **[P2] Keep sharp build approval while sharp remains resolved:** `package.json:964`
  - body: The PR removes `sharp` from `pnpm.onlyBuiltDependencies`, but the PR head lockfile still resolves Baileys with `sharp@0.34.5`. That can leave a remaining sharp install path with its lifecycle scripts unapproved, so keep the approval until the graph is actually sharp-free.
  - confidence: 0.82
