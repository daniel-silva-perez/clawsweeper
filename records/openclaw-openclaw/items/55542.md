## Summary

Keep open. Current main still has sharp-backed image processing in the media-understanding-core plugin, so the requested dependency replacement is not implemented; the PR branch is also stale against the current image-runtime split and introduces concrete behavior regressions around EXIF normalization and the legacy backend override.

## What This Changes

The PR removes sharp install guidance and the root sharp dependency, adds @napi-rs/image with native build approval and lockfile entries, and ports image-processing code plus image-related tests from sharp to @napi-rs/image Transformer APIs.

## Best Possible Solution

Rework the migration against current main by replacing the media-understanding-core sharp implementation, preserving the public plugin-sdk media contract, keeping legacy backend override behavior compatible, preserving non-JPEG formats during EXIF normalization, and only removing sharp docs once no shipped runtime path still depends on sharp.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the input format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: `normalizeExifOrientation` now returns `transformer.rotate().jpeg(90)` for every input. The previous sharp path used `rotate().toBuffer()`, which preserved PNG/WebP inputs; this change silently converts non-JPEG images to lossy JPEG and drops alpha for SDK callers that only asked for orientation normalization.
  - confidence: 0.93
- **[P2] Keep the legacy sharp backend override working:** `src/media/image-ops.ts:28`
  - body: Changing the Bun/macOS escape hatch from `OPENCLAW_IMAGE_BACKEND=sharp` to only `napi-rs` makes existing installs with the documented `sharp` override fall back to the sips path after upgrade. Keep `sharp` as a compatibility alias or add an explicit migration before changing this condition.
  - confidence: 0.87
