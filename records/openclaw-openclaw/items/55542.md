## Summary

Keep open. Current main does not implement the PR’s intended dependency swap: the root package no longer has a direct sharp dependency, but image processing still delegates to the bundled media-understanding-core plugin, which owns sharp as a staged runtime dependency. The PR is also stale against that split and still has concrete image behavior and install-policy regressions before it could be considered for merge.

## What This Changes

The PR removes root sharp install guidance and dependency entries, adds @napi-rs/image and native build approval/lockfile entries, and ports image operations plus related tests to @napi-rs/image Transformer APIs.

## Best Possible Solution

Rework the migration, if maintainers still want it, in the owner module that now holds image processing: media-understanding-core should own any backend swap, preserve the plugin SDK media contract, keep existing backend override compatibility or document a migration, retain format/alpha behavior for orientation normalization, and update install docs/build approvals only after the resolved dependency graph is actually sharp-free.

## Reproduction Assessment

Yes for the reviewed regressions. Read-only inspection of the PR head, current main, and @napi-rs/image@1.12.0 upstream README/types gives high-confidence reproduction paths: call normalizeExifOrientation with a non-JPEG alpha image, call convertHeicToJpeg with HEIC input, use OPENCLAW_IMAGE_BACKEND=sharp on Bun/macOS, and inspect pnpm approval behavior while the lockfile still resolves sharp.

## Solution Assessment

No. The PR is not the best current fix because it edits the old core image backend shape instead of the current plugin-owned media-understanding-core runtime, and it removes sharp-related compatibility before proving all sharp-backed behavior and install policy are covered.
