## Summary

Keep this PR open. It targets a real Telegram customCommands menu capability that current main still does not provide, so it is not obsolete or implemented. The proposed branch is not merge-ready as-is because the new validated-index helper is imported from a surface that does not export it on current main, and configured callback payloads are still accepted without Telegram limit validation.

## What This Changes

The PR adds Telegram customCommand `menus`/`routes` config, registers menu-backed slash command handlers, routes configured callback queries to menu edits, passes validated custom-command indices through bot setup, and updates Telegram config schema/types.

## Best Possible Solution

Land a refreshed Telegram plugin implementation that adds typed `menus` and `routes` on the current config surfaces, validates button labels and `callback_data` against Telegram constraints, scopes callback routing to authenticated originating menu messages, updates generated config metadata and public Telegram docs, and adds focused Telegram config/native-command/callback tests.

## Reproduction Assessment

Yes for the review findings: current main’s `src/plugin-sdk/config-runtime.ts` does not export `resolveValidatedCustomCommandIndices`, so the PR import is a compile-time failure unless the helper is moved or re-exported. The callback validation gap is reproducible with a configured blank or over-64-byte `callback_data`, which the PR schema accepts as `z.string()` before sending to Telegram.

## Solution Assessment

No, not as currently proposed. The feature direction belongs in the Telegram plugin/config surface, but the branch should be refreshed onto the current command-config modules and should reuse or mirror the existing Telegram callback-data validation contract before merge.
