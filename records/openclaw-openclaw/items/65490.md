## Summary

Keep open. Current main has partial fallback/status/cron visibility work, but the central production-operations requests remain unresolved: primary-model startup reachability is still background and warning-only, the agent-visible diagnostics surface is not a structured model/fallback/last-provider-error contract, and cron creation still persists announce jobs before a delivery-target dry-run/preflight gate.

## What This Changes

Request production execution-discipline improvements for primary-model startup health, structured runtime diagnostics, cron delivery preflight/dry-run behavior, and clearer fallback/error visibility.

## Best Possible Solution

Keep this open as the production execution-discipline tracker, or split it into focused follow-ups for a configurable startup provider preflight/degraded-mode policy, a safe structured runtime diagnostics RPC or agent tool, and cron add dry-run/preflight delivery validation before persistence.

## Reproduction Assessment

Partially yes. I did not run the production disabled-provider scenario, but current source and tests provide a high-confidence reproduction of the missing gates: startup prewarm is non-blocking by design, diagnostics lacks the requested structured fields, and cron.add accepts announce delivery without resolving a destination first.

## Solution Assessment

No, current main is not the complete solution. The existing fallback notices, status text, health snapshots, and cron delivery previews are useful partial work, but they do not provide the requested startup reachability policy, structured agent-readable last-error/fallback state, or fail-closed cron add preflight.
