## Summary

Keep open. Current main only covers adjacent Signal behavior, while the PR still targets missing Signal capabilities; however, the proposed patch is not ready as-is because quoted media is added outside the standard media-ingestion fields and the remote attachment path needs a safer rebased design.

## What This Changes

The PR adds Signal reply-target media fields, structured Signal mention and reply-to-bot detection, and data URI attachment encoding for remote signal-cli sends.

## Best Possible Solution

Land a rebased Signal-only implementation that keeps local signal-cli file-path behavior, adds an explicit or reliably resolved remote data-URI mode, detects structured bot mentions and reply-to-bot quotes through current mention-gating APIs, and routes quoted media through the standard media context fields so model media understanding can consume it.

## Reproduction Assessment

Yes, at code level: current main can be traced to file-path-only Signal sends, regex-only Signal mention gating, and no quoted attachment type/ingestion path. The PR's main media defect is also reproducible by tracing ReplyToMediaPath to normalizeAttachments, which ignores that field.

## Solution Assessment

No, not as proposed. The direction is maintainable, but the patch should be rebased onto current main, use standard MediaPaths/MediaTypes for quoted media, pass resolved MIME data into data URI construction, and avoid URL-host-only remote detection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Route quoted media through MediaPaths:** `extensions/signal/src/monitor/event-handler.ts:216-217`
  - body: This stores quoted attachments only in ReplyToMediaPath/ReplyToMediaType, but the media-understanding pipeline normalizes attachments from MediaPath/MediaPaths/MediaUrl/MediaUrls. A user asking about a quoted image or audio file would still not give the model that media unless a custom prompt template reads these new ad-hoc fields.
  - confidence: 0.9
- **[P2] Check quote UUID independently:** `extensions/signal/src/monitor/event-handler.ts:658-667`
  - body: Using quote.author ?? quote.authorUuid means the UUID is never checked when both fields are present. UUID-only bot account configs can miss reply-to-bot implicit mentions when the phone author value is present but does not match the configured account, causing mention-gated group replies to be skipped.
  - confidence: 0.82
- **[P2] Use resolved content type for data URIs:** `extensions/signal/src/send.ts:172-190`
  - body: localFileToDataUri re-derives MIME from a short extension map instead of using the resolved contentType already returned by resolveOutboundAttachmentFromUrl. Staged files for supported types outside that map, such as audio/aac or office documents, will be sent as application/octet-stream even when OpenClaw already knows the real MIME type.
  - confidence: 0.78
- **[P2] Make remote attachment mode explicit:** `extensions/signal/src/send.ts:204-205`
  - body: The PR treats any non-localhost baseUrl as remote and any localhost URL as local, but remote signal-cli is often reached through SSH tunnels or reverse proxies on 127.0.0.1. Those setups will continue sending local filesystem paths that the daemon cannot read, so the advertised remote-compatibility fix will fail for common deployments.
  - confidence: 0.7
