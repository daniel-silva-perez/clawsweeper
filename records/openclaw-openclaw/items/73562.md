## Summary

Keep this open. Current main still has the closed-stdin prompt failure shape, and open PR #73566 is the explicit implementation path, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that `openclaw plugins uninstall <id>` handle closed confirmation stdin with a clean exit-1 CLI error while preserving piped `y`/`n` answers and `--force`.

## Best Possible Solution

Add EOF-aware behavior to the shared prompt primitive, convert that specific prompt-closed condition in `plugins uninstall` into exit 1 with interactive or `--force` guidance, and keep piped answers working. The stale-entry cleanup tracked by #73551 should stay separate unless maintainers intentionally broaden the same change.

## Reproduction Assessment

Yes. Current main still awaits the shared prompt in the non-forced uninstall path, and a read-only Node readline smoke reproduces exit 13 for closed input while piped `n` resolves normally.

## Solution Assessment

Yes at the design level. Handling EOF in `promptYesNo()` and catching that specific condition in `plugins uninstall` is narrower and safer than a blanket non-TTY gate because piped `y`/`n` answers should continue to work.
