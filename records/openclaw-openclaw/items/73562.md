## Summary

Keep this open. Current main still has the closed-stdin failure shape, the latest tagged release still contains the same prompt path, and PR #73566 is open and unmerged with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Make `openclaw plugins uninstall <id>` fail cleanly when confirmation input closes while preserving piped `y`/`n` answers and `--force`.

## Best Possible Solution

Review and land PR #73566 or an equivalent narrow fix that makes the shared prompt helper settle on readline close/EOF, has plugin uninstall convert that condition into exit 1 with interactive or `--force` guidance, and adds focused prompt plus uninstall regression tests. Keep the stale-entry cleanup in #73551 separate unless a maintainer intentionally broadens the same PR.

## Reproduction Assessment

Yes. Current main still awaits the shared prompt in the non-forced uninstall path, and a read-only Node `readline/promises` closed-input smoke reproduces exit 13 while newline-delimited `n` input resolves normally.

## Solution Assessment

Yes at the design level. Handling EOF in the shared prompt primitive and catching that specific condition in `plugins uninstall` is narrower than a blanket non-TTY gate because piped `y`/`n` answers should continue to work.
