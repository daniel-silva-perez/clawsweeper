## Summary

Keep this open. Current `main` still has the reported closed-stdin failure shape, and an open linked fix PR already uses closing syntax for this issue, so the right cleanup path is to review and land or close that PR rather than closing this issue early.

## What This Changes

Make `openclaw plugins uninstall <id>` fail cleanly when confirmation stdin closes, while preserving piped `y`/`n` answers and `--force`.

## Best Possible Solution

Review and land PR #73566 or an equivalent narrow fix: make the shared prompt helper settle on readline close/EOF with a distinct error/result, have plugin uninstall turn that into exit 1 with actionable `--force`/TTY guidance, and add focused prompt and uninstall regression tests. Keep the stale plugin-entry issue #73551 as a separate cleanup unless one PR intentionally covers both.

## Reproduction Assessment

Yes. The source path on current `main` still awaits `promptYesNo()` for non-forced uninstall, and the minimal `node:readline/promises` closed-stdin command reproduces the Node exit 13 behavior while piped `n` resolves normally.

## Solution Assessment

Yes at the design level. The linked PR direction is the narrow maintainable fix because it handles EOF in the shared prompt primitive and catches that specific condition in `plugins uninstall` without blocking working piped answers via a blanket non-TTY gate.
