## Summary

Keep this issue open. Current main still has both reported memory-wiki lint defects: wikilink extraction runs regexes over raw page markdown, and broken-link validation still accepts only extensionless relative paths and basenames. Related #64696 overlaps on reply-tag and relative-link noise but is broader bridge/import durability work, so it does not supersede this narrower root-cause lint issue.

## What This Changes

The issue asks for memory-wiki lint to ignore literal wikilink syntax in Markdown code/comment/frontmatter contexts and resolve wikilinks by frontmatter title or normalized aliases.

## Best Possible Solution

Fix this in the memory-wiki plugin by making link extraction Markdown-aware before lint sees linkTargets, then replacing the path/basename-only broken-link resolver with a normalized page-target index that includes relative path, basename, title, id, slugified title, and aliases with clear collision handling.

## Reproduction Assessment

Yes. A minimal vault page containing fenced or inline literal `[[reply_to_current]]`, plus a slugged source page whose frontmatter title is linked as `[[Title]]`, will follow the current raw-regex extractor and path/basename-only resolver paths deterministically.

## Solution Assessment

Yes. The requested root-cause fix is the narrow maintainable path: parse or strip Markdown contexts generically and resolve against normalized page identity data, rather than adding tag-specific allowlists for each directive shape.
