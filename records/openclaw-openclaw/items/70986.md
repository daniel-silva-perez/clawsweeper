## Summary

Keep open: current main still has the reported memory-wiki lint behaviors, and the open related work only partially covers them. The title/slug resolver half is represented by #73574 and open PR #73722, while the Markdown-aware extractor half remains unmerged after #68852 was closed.

## What This Changes

The issue asks for memory-wiki lint to ignore literal wikilink syntax in Markdown non-prose contexts and to resolve wikilinks by page title or normalized variants, not only slugified paths.

## Best Possible Solution

Keep this as the canonical combined memory-wiki lint bug until both halves are resolved: land or replace the scoped resolver normalization from #73722, then add a robust Markdown-aware extraction pass that skips fenced blocks, inline code, frontmatter, and HTML comments without losing real prose links. Add focused regressions in `markdown.test.ts` and `lint.test.ts`, plus a short docs note if accepted link forms remain intentionally constrained.

## Reproduction Assessment

Yes. A page with fenced or inline literal `[[reply_to_current]]` still flows through raw `extractWikiLinks`, and a page linked as `[[Example 配置清单]]` still misses the path-only `validTargets` set unless the filename/basename exactly matches.

## Solution Assessment

Yes. The generic extractor-plus-normalized-resolver direction is the narrow maintainable fix; tag-specific sanitization would only hide one bridge symptom, and #73722 is useful but only covers the resolver side.
