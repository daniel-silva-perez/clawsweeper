## Summary

Keep open: current main still has both reported memory-wiki lint gaps, and the discussion adds credible live-vault confirmation that this is noisy in real use. This is not superseded by #64696, which overlaps on reply-tag noise but does not cover frontmatter-title wikilink resolution.

## What This Changes

The issue requests a memory-wiki lint fix so wikilink extraction ignores Markdown code/comment/frontmatter contexts and broken-link validation resolves natural frontmatter titles and related aliases.

## Best Possible Solution

Fix this in the bundled memory-wiki plugin by making link extraction Markdown-aware before `linkTargets` are recorded, then sharing or mirroring the compile-side normalized lookup model in lint so paths, basenames, titles, ids, aliases, and slugified titles resolve deterministically.

## Reproduction Assessment

Yes. A focused vault with a fenced or inline literal `[[reply_to_current]]` plus a slugged source page titled `Example 配置清单` linked as `[[Example 配置清单]]` will flow through the current raw extractor and path-only validator into broken-wikilink warnings.

## Solution Assessment

Yes. The root-cause direction in this issue is the narrow maintainable fix; a reply-tag allowlist would only mask one bridge symptom and would leave natural-title wikilinks broken.

## Repair Work Prompt

Fix the memory-wiki lint false positives and title-resolution bug reported in https://github.com/openclaw/openclaw/issues/70986. Keep the change inside the bundled `memory-wiki` plugin. Make wikilink extraction Markdown-aware before `linkTargets` are recorded: skip fenced code blocks using backtick and tilde fences, inline code spans, YAML frontmatter, and HTML comments. Then update broken-link validation so it uses a deterministic normalized target index that preserves existing relative-path and basename behavior while also accepting page `title`, `id`, `aliases`, and slugified title/alias forms. Prefer reusing or extracting a local helper similar to compile-side `buildPageLookupKeys` rather than adding tag-specific allowlists. Add regression tests for literal `[[reply_to_current]]` or `[[wikilinks]]` inside fenced and inline code, and for `[[Example 配置清单]]` resolving to `sources/example-配置清单.md` with matching frontmatter title. Do not move behavior into core, do not weaken structure/provenance lint rules, and do not make #64696’s unrelated relative-link or bridge-cache fixes part of this PR. Because this is user-facing lint behavior, add a single-line `CHANGELOG.md` fix entry crediting @richardmqq and @M-Lietz.
