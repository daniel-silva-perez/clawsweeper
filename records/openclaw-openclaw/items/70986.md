## Summary

Current main still has the reported memory-wiki lint behavior: wikilink extraction scans raw page Markdown, and broken-link validation only indexes path-derived targets. Related #64696 overlaps on bridge reply-tag noise but does not supersede the title-aware resolver gap described here.

## What This Changes

Make `openclaw wiki lint` ignore literal wikilink syntax in non-link Markdown contexts and resolve wikilinks against frontmatter titles and related normalized page identifiers.

## Best Possible Solution

Fix this inside the bundled memory-wiki plugin: make wikilink extraction Markdown-aware before `linkTargets` are stored, then replace direct path-only validation with a deterministic normalized target index that includes relative paths, basenames, titles, ids, aliases, and slugified title variants.

## Reproduction Assessment

Yes. A minimal vault with a fenced or inline literal `[[reply_to_current]]` plus a slugged page whose frontmatter title is linked as `[[Example 配置清单]]` follows the current raw-extractor and path-only resolver code deterministically.

## Solution Assessment

Yes. The narrow maintainable solution is a generic Markdown-aware extractor plus title-aware normalized resolver; a tag-specific allowlist would only hide one bridge symptom and would not fix natural-title links.
