## Summary

Keep open: current main still has the reported memory-wiki lint behavior, and the issue is specific enough for a narrow plugin-owned repair. The existing discussion adds credible live-vault confirmation, while related #64696 overlaps only on some bridge-link noise and does not supersede the frontmatter-title resolver bug.

## What This Changes

Make memory-wiki lint ignore literal wikilink syntax in Markdown non-link contexts and resolve wikilinks against normalized frontmatter titles, ids, aliases, and slug variants.

## Best Possible Solution

Fix this inside the bundled memory-wiki plugin: make link extraction Markdown-aware before `linkTargets` are recorded, then use one deterministic normalized target index for lint that includes paths, basenames, titles, ids, aliases, and slugified title forms, with focused regression tests and a short docs note for the accepted link forms.

## Reproduction Assessment

Yes. The current code path deterministically reproduces it with a vault page containing fenced or inline literal `[[reply_to_current]]`, plus a slugged source file whose frontmatter title is linked as `[[Example 配置清单]]`; both flow through raw extraction and path-only validation.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus normalized title-aware resolver is the narrow maintainable fix; a tag-specific allowlist would only hide one bridge symptom and would not resolve natural-title wikilinks.
