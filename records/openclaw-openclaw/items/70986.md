## Summary

Keep open: current main still has both reported memory-wiki lint behaviors, and the issue is specific enough for a narrow plugin-owned repair. The related #64696 overlaps on some bridge-link noise, but it does not supersede the frontmatter-title resolver bug.

## What This Changes

The issue requests that memory-wiki lint stop extracting literal wikilink-shaped text from Markdown structure and resolve wikilinks against frontmatter titles, aliases, and normalized page identities.

## Best Possible Solution

Fix this inside the bundled memory-wiki plugin: make link extraction Markdown-aware before `linkTargets` are recorded, then use one deterministic normalized lint target index that includes paths, basenames, titles, ids, aliases, and slugified title/alias forms, with a short docs note for accepted link forms.

## Reproduction Assessment

Yes. The current code path deterministically reproduces it with a vault page containing fenced or inline literal `[[reply_to_current]]`, plus a slugged source file whose frontmatter title is linked as `[[Example 配置清单]]`; both flow through raw extraction and path-only validation.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus normalized title-aware resolver is the narrow maintainable fix; a tag-specific allowlist would only hide one bridge symptom and would not resolve natural-title wikilinks.
