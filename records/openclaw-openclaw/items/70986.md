## Summary

Keep open. Current main still has the reported memory-wiki lint behavior: wikilink extraction scans raw page text, and broken-link validation only accepts extensionless paths and basenames, while the discussion adds credible live-vault confirmation that this creates widespread false positives.

## What This Changes

Requests memory-wiki lint to ignore literal wikilink syntax in non-link Markdown contexts and resolve wikilinks against frontmatter titles and normalized page targets.

## Best Possible Solution

Fix memory-wiki lint at the plugin boundary: make extraction Markdown-aware before storing linkTargets, then replace direct path-only matching with a deterministic normalized target index that includes page paths, basenames, titles, ids, aliases, and slugified title or alias variants.

## Reproduction Assessment

Yes. The current source provides a deterministic reproduction path: a page with fenced or inline literal [[reply_to_current]] will still produce a link target, and a slugged page with frontmatter title Example 配置清单 will not satisfy a bare [[Example 配置清单]] link because lint only indexes path and basename.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus a title-aware normalized resolver is the narrowest maintainable fix; a tag-specific allowlist would leave natural-title links and future literal wikilink examples broken.
