## Summary

Keep open: current main still has the reported memory-wiki lint behavior. The extractor scans raw markdown for wikilink regex matches, and the broken-link resolver still accepts only extensionless paths and basenames, so the confirmed false positives and title-based resolution failures remain. Related #64696 overlaps on bridge/link noise but does not supersede this narrower root-cause lint issue.

## What This Changes

The issue asks for memory-wiki lint to ignore literal `[[...]]` in Markdown code/comment/frontmatter contexts and to resolve wikilinks by frontmatter titles or normalized aliases in addition to path-based targets.

## Best Possible Solution

Fix the memory-wiki plugin by making wikilink extraction Markdown-aware before `linkTargets` are stored, then replace the lint resolver's direct path set with a normalized target index that includes paths, basenames, page titles, ids, slugified titles, and aliases with deterministic ambiguity handling.

## Reproduction Assessment

Yes. A minimal vault with a fenced or inline literal `[[reply_to_current]]` plus a slugged page whose frontmatter title is linked as `[[Example 配置清单]]` follows the current raw-regex extractor and path-only resolver deterministically.

## Solution Assessment

Yes. The root-cause repair is to skip non-link Markdown contexts generically and resolve against normalized page identity data; tag-specific allowlists would only cover one current symptom and would not fix title-based links.
