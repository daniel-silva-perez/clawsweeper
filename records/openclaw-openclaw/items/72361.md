## Summary

Keep this PR open. Current main still has the reported configWrites bypass in the store-only `/allowlist` path, and the PR is a narrow, apparently correct security hardening change with focused regression coverage; it needs maintainer/security review and CI rather than cleanup closure.

## What This Changes

The PR adds a configWrites authorization guard before store-targeted `/allowlist` pairing-store mutations and adds allowlist command tests for channel-level denial, account-level denial, and the allowed store-edit case.

## Best Possible Solution

Land the narrow store-path configWrites guard with the regression tests after maintainer security review and changed-surface validation, leaving the linked issue open until the merge closes it normally.

## Reproduction Assessment

Yes. Static reproduction is high-confidence on current main: configure a pairing-capable channel with `commands.config=true` and `configWrites=false`, send `/allowlist add dm --store <id>` as an owner/authorized sender, and the handler skips the config-write gate before mutating the pairing store.

## Solution Assessment

Yes. Applying the same `resolveConfigWriteDeniedText` policy check immediately before the store-only mutation is the narrowest maintainable fix, and the PR's tests cover both denial paths plus the allowed case.
