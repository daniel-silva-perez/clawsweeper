## Summary

Keep this PR open. Current main still has the reported `/allowlist --store` `configWrites` bypass, and this PR is useful work, but the proposed guard still authorizes against the selected target channel rather than the actual command source, leaving a cross-channel store-mutation bypass. The prior Greptile comment correctly covers the same-channel improvement, while the existing ClawSweeper review identified the remaining source-vs-target gap and missing changelog entry.

## What This Changes

The PR adds a `configWrites` denial check before store-only `/allowlist add|remove` pairing-store mutations and adds focused allowlist command tests for blocked channel/account store edits plus an allowed store edit.

## Best Possible Solution

Use separate source and target scopes for allowlist authorization: derive the origin from the actual command source and context account, derive the target from the parsed allowlist channel/account, apply the policy before both config and store mutations, and cover same-channel, target-account, and cross-channel source-denied store cases with tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction for the linked same-channel bug, and the PR gap can be reproduced by sending `/allowlist add dm --store --channel discord 789` from a Telegram source with `channels.telegram.configWrites=false` while the Discord target allows writes.

## Solution Assessment

No. The PR adds the right kind of guard, but it wires the store-only authorization with the selected target channel as the origin; the maintainable fix is to pass distinct origin and target scopes and add regression coverage for `--channel` and account-targeted store edits.
