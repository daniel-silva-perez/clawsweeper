## Summary

Keep this PR open for maintainer review and merge consideration. The affected surface is text `/allowlist add|remove --store` in `src/auto-reply/reply/commands-allowlist.ts`, linked to https://github.com/openclaw/openclaw/issues/72360; current main still skips the `configWrites` guard on the store-only path, while this PR adds the missing guard and focused regression coverage.

## What This Changes

The PR adds a `configWrites` authorization check before store-targeted `/allowlist` pairing-store mutations and adds tests for channel-level denial, account-level denial, and the allowed store-write case.

## Best Possible Solution

Land the narrow policy check after normal maintainer and CI review so config-backed and pairing-store-backed `/allowlist` mutations consistently respect the same channel/account `configWrites` boundary without changing unrelated defaults or migrations.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: use a pairing-capable channel with `commands.config=true` and `configWrites=false`, then `/allowlist add dm --store <id>` follows the store-only branch that skips `resolveConfigWriteDeniedText` and writes to the pairing store.

## Solution Assessment

Yes. The PR uses the existing config-write authorization helper immediately before the only remaining store-only mutation point, with an explicit channel/account target and regression tests for both denial scopes plus the allowed case.
