## Summary

Keep this PR open. Current main still has the reported store-only `configWrites` bypass, and the PR targets the right code path, but the proposed guard is not complete because it passes the selected target channel as the write origin when `/allowlist --channel ... --store` is used.

## What This Changes

The PR adds a `configWrites` authorization guard before `/allowlist --store` pairing-store mutations and adds allowlist command tests for blocked and allowed store-targeted edits.

## Best Possible Solution

Use separate source and target scopes for `/allowlist` authorization: derive the origin from `params.command.channelId`/`params.command.channel` and the current context account, derive the target from the selected allowlist channel/account, apply that consistently to config and store edits, and cover same-channel, target-account, and cross-channel source-denied cases.

## Reproduction Assessment

Yes. Current-main source inspection provides a high-confidence path for the linked same-channel report, and the PR gap can be reproduced by sending `/allowlist add dm --store --channel discord 789` from a write-disabled Telegram source while the Discord target allows writes.

## Solution Assessment

No. The PR uses the right policy helper, but it wires the store-only branch with the target channel as the origin; the maintainable fix is to pass distinct origin and target scopes and add a regression test for `--channel`.
