## Summary

Keep open: current main still uses a single durable Telegram accepted-update watermark and restores it as a skip cutoff, so an accepted-but-unfinished polling update can still be dropped after a restart or abort once grammY has advanced Telegram-side polling offset. The issue is also paired with open, unmerged PR #73993 using closing language, so it should remain open until that implementation or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to stop silently dropping Telegram polling messages when update offset acknowledgement can advance past in-flight message processing.

## Best Possible Solution

Land PR #73993 or an equivalent Telegram plugin-local durability fix that separates accepted/fetched and completed offsets, restarts from a completed-safe watermark or explicitly reconciles in-flight ranges, preserves duplicate/replay guards, and adds coverage for a two-update interrupted polling batch.

## Reproduction Assessment

Yes for the restart durability race: current source, tests, and grammY runner behavior show an update can be accepted and persisted before its handler completes, then rejected from the restored accepted watermark after restart. The normal-operation no-crash variant remains unclear without logs or a narrower failing scenario.

## Solution Assessment

Yes, the best direction is a narrow Telegram polling durability fix in the plugin, not a core channel rewrite. Closing this issue before the linked implementation lands would be premature.
