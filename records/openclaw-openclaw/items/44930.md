## Summary

Keep open. The current issue is still valid because current `main` still has a single durable Telegram update offset that is restored as an accepted watermark on restart, and an open PR with closing language already tracks the narrower durability fix. The existing code improves at-most-once replay protection, but it does not persist a separate completed watermark or provide a fetched/in-flight recovery barrier for the crash/restart race described in the issue discussion.

## What This Changes

The issue requests a Telegram polling fix so messages are not silently dropped when update offset acknowledgement can get ahead of handler completion during busy processing or restart windows.

## Best Possible Solution

Land or deliberately replace the narrow Telegram polling durability fix: persist enough fetched/accepted/in-flight state before grammY can advance polling, keep a separate completed watermark for restart recovery, preserve existing duplicate protections, and add regression coverage for a two-update batch where the second update is accepted but the process restarts before handler completion.
