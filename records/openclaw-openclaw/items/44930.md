## Summary

Keep open: current main still has the accepted-watermark-only Telegram polling model, and the linked open PR with closing language is unmerged. Source, tests, issue discussion, and the grammY runner implementation still support the restart/interrupt race described in the report.

## What This Changes

Fix Telegram polling so direct-chat messages are not silently dropped when update offset confirmation can race ahead of OpenClaw handler completion during concurrent or restarted processing.

## Best Possible Solution

The Telegram plugin should land PR #73993 or an equivalent replacement that durably separates fetched/accepted and completed offsets, restarts from the completed watermark, logs accepted-but-unfinished recovery ranges, and covers a two-update batch interrupted before the second update completes.

## Reproduction Assessment

Yes for the restart durability race: current source and tests show an accepted update can be persisted, restored in a new process, and skipped before completion. The normal-operation no-restart variant is unclear without logs around dedupe, media-group coalescing, or gateway restarts.

## Solution Assessment

Yes, a Telegram plugin-local polling durability fix is the narrow maintainable direction because the race sits in the bundled Telegram polling handoff, not in core channel routing. Closing before the open implementation merges would be premature.
