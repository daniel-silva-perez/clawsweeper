## Summary

Keep open. Current main still restores a single Telegram accepted-update watermark while the pending, failed, and completed update state is process-local, so it does not prove recovery for an update accepted before handler completion and then interrupted. The linked PR #73993 is open and unmerged with explicit closing language for this issue, so this should stay open until that PR lands or is deliberately replaced.

## What This Changes

The issue reports Telegram polling/direct-chat messages being silently lost when update offset acknowledgement can advance past in-flight handler processing during active work or restart-like conditions.

## Best Possible Solution

Land or deliberately replace the narrow Telegram polling durability fix so fetched, accepted, in-flight, and completed update state cannot let Telegram-side offset confirmation outrun OpenClaw handler completion, while preserving existing duplicate guards and adding a regression for a two-update batch interrupted before the second update completes.
