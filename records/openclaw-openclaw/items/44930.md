## Summary

Keep open. Current main still restores a single Telegram accepted-update watermark while pending, failed, and completed update state is process-local, so the restart/interrupt path from the issue discussion is not fixed on main; PR #73993 is open and unmerged with explicit closing language for this issue.

## What This Changes

The issue reports Telegram polling sometimes losing direct-chat messages when update offsets advance before OpenClaw finishes processing in-flight updates.

## Best Possible Solution

Keep the issue open until the Telegram plugin has a durable polling handoff that prevents fetched, accepted, in-flight, and completed update state from letting Telegram-side offset confirmation outrun OpenClaw handler completion, with regression coverage for a two-update batch interrupted before the second update completes.

## Reproduction Assessment

Yes for the restart durability race: current tests and source show an accepted update can be persisted, restored in a new process, and skipped before completion. The normal-operation, no-restart variant remains unclear without logs around dedupe, media-group coalescing, or gateway restarts.

## Solution Assessment

Yes, a Telegram plugin-local polling durability fix is the narrow maintainable direction; the issue should not close as implemented while PR #73993 is still open and current main still has only an accepted-watermark restore path.
