## Summary

Keep open. Current main still persists a single accepted Telegram update watermark and keeps completed, pending, and failed update state in memory, so it does not prove recovery for an update accepted by grammY but not completed before a restart. The linked discussion's crash/restart race remains consistent with the code and `@grammyjs/runner` behavior, and PR #73993 is already the open implementation candidate with closing language for this issue.

## What This Changes

The issue requests a Telegram polling durability fix so messages are not silently lost when update offset acknowledgement advances before OpenClaw finishes processing accepted updates.

## Best Possible Solution

Land or deliberately replace the narrow Telegram polling durability fix: persist enough fetched, accepted, and in-flight state before runner offset confirmation can outrun handler completion, keep completed recovery distinct from accepted-watermark dedupe, preserve existing duplicate guards, and add a regression for a two-update batch where the second update is accepted but the process restarts before handler completion.
