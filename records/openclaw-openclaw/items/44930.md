## Summary

Keep open. Current main still restores a single accepted Telegram update watermark on restart, while the confirmed race requires durable recovery across accepted/fetched/in-flight updates and handler completion. An open PR already uses closing language for this issue, so maintainer follow-up should happen there rather than closing this issue early.

## What This Changes

The issue requests a Telegram polling fix so messages are not silently dropped when update offset acknowledgement gets ahead of in-flight message processing.

## Best Possible Solution

Land or deliberately replace the narrow Telegram polling durability fix: persist enough fetched, accepted, and in-flight state before grammY can advance polling, maintain a separate completed watermark for restart recovery, preserve existing duplicate protections, and cover a two-update batch where the second update is accepted but the process restarts before handler completion.
