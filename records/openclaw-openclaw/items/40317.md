## Summary

Keep this issue open. Current main still has strict root/schema validation and validated write paths, but it does not expose an operator-configurable strict/tolerant validation mode and startup/read behavior still fails or recovers via last-known-good/doctor paths rather than stripping unknown keys and continuing. The narrower tolerant-startup issue #40268 is still open, and the related implementation PR #46213 was closed unmerged.

## Best Possible Solution

Keep this open for maintainer product judgment. Either add and document a config validation policy in the schema/types/help/docs with strict write validation and well-defined tolerant read/startup/hot-reload behavior, or explicitly decide that only the narrower tolerant-startup behavior should remain tracked in #40268 and close this later as superseded.

## Risks / Open Questions

- Closing now would drop an unresolved product/API decision: whether OpenClaw should expose an operator-configurable validation policy or keep strict validation plus last-known-good/doctor recovery only.
- A tolerant read mode still needs precise semantics for root vs nested unknown keys, known-key type/value errors, plugin-local schema failures, logging level, hot reload, startup, and last-known-good recovery interactions.
- Any future implementation must preserve the current strict write rejection for config.set, config.apply, and config.patch so invalid keys are not persisted through OpenClaw-owned writes.
