## Summary

Keep this issue open. Current main upgrades Pi to 0.70.5, but it still retains the same native clipboard dependency path: the lockfile includes @mariozechner/clipboard-linux-x64-gnu via @mariozechner/clipboard, upstream Pi 0.70.5 still evaluates clipboard-native.js from the package top-level export path, and OpenClaw still has Gateway request/session and embedded-runner paths that import @mariozechner/pi-coding-agent before any OpenClaw-owned clipboard opt-out exists. The issue discussion also contains two concrete coredump reports and a confirmed TERMUX_VERSION workaround, so this remains actionable rather than stale.

## Best Possible Solution

Keep this issue open. The best fix is either an upstream Pi release that makes native clipboard loading explicit/lazy or an OpenClaw-owned opt-out that is applied before any top-level @mariozechner/pi-coding-agent import in Gateway request/session and embedded-runner paths. Add focused regression coverage proving the guarded Linux Gateway/background path does not require @mariozechner/clipboard, and document the supported workaround instead of relying on renaming node_modules files or co-opting TERMUX_VERSION.
