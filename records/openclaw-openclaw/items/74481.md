## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` refresh for the OpenAI Control UI picker; existing catalog/discovery plumbing and provider-specific precedents do not cover this configured-proxy case, and the related open PR #74488 is only a first implementation slice rather than a merged fix.

## What This Changes

Request adding opt-in dynamic model catalog discovery from configured provider `/v1/models` endpoints, with caching and merge/replace behavior surfaced in the Control UI model picker.

## Best Possible Solution

Have maintainers choose the catalog contract first, preferably by reusing or deliberately extending the existing provider `catalog` architecture, then implement OpenAI-owned configured-endpoint discovery with guarded fetch, endpoint/auth-scoped caching, clear `models.mode` or replacement semantics, Gateway/Control UI refresh behavior, tests, and docs.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence check is static current-main inspection: no OpenAI catalog hook or Control UI refresh path exists, so a configured OpenAI-compatible proxy still depends on bundled or manually configured model rows.

## Solution Assessment

Unclear. The direction matches existing provider-specific discovery patterns, but deciding whether to extend `ProviderPlugin.catalog`, add a new hook, and reuse `models.mode` versus a new setting needs maintainer API judgment before implementation.
