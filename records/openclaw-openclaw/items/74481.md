## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the same author has an open paired implementation slice in #74488 that should be reviewed before settling the API seam.

## What This Changes

The issue requests an opt-in dynamic model catalog refresh that discovers models from configured provider `/v1/models` endpoints and feeds the Control UI picker.

## Best Possible Solution

Decide the provider-catalog contract first, likely by reusing or intentionally extending the existing live `catalog.run` and explicit refresh/cache paths, then land OpenAI-compatible discovery with credential/baseUrl-scoped caching, deterministic merge/replace behavior, Gateway/Control UI refresh behavior, focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The current absence is high-confidence from source inspection: current main has static OpenAI catalog data and configured model-list plumbing but no OpenAI `/v1/models` refresh path.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but adding a new hook may not be the narrowest solution if the existing provider catalog discovery contract can be reused safely.
