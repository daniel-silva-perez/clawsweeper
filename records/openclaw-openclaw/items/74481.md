## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the OpenAI Control UI picker, and an open paired PR by the same author only adds a discovery primitive without wiring it into catalog resolution. The request is valid but needs maintainer API/product direction before cleanup automation should try to finish it.

## What This Changes

The issue requests an opt-in dynamic provider catalog refresh that discovers configured OpenAI-compatible models from `/v1/models`, maps provider metadata, caches results, and refreshes the Control UI picker.

## Best Possible Solution

Settle the provider-catalog contract first, preferably by reusing or deliberately extending the existing `catalog`/`ModelProviderConfig.models` path, then wire OpenAI-compatible configured-provider discovery through Gateway/Control UI catalog loading with scoped caching, merge/replace semantics, config-change invalidation, tests, and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `models.providers.openai` with a proxy `baseUrl` and API key but no explicit proxy model list; current main has no OpenAI `/models` discovery hook and the Control UI only requests the configured model view, so the picker cannot reflect the upstream catalog.

## Solution Assessment

Unclear. The user problem is real, but the proposed new `resolveDynamicCatalog` hook may not be the narrowest solution because the existing provider `catalog` hook already supports live provider-owned catalogs; maintainers should decide whether to reuse or extend that seam before implementation proceeds.
