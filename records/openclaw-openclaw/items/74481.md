## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the same author has an open paired implementation slice in #74488 that should be reviewed rather than closing this design issue early.

## What This Changes

Request to add opt-in dynamic model catalog refresh from configured OpenAI-compatible provider `/v1/models` endpoints, with cache, merge/replace semantics, and Control UI picker refresh behavior.

## Best Possible Solution

Decide the provider-catalog contract first, likely by reusing or intentionally extending the existing live `catalog.run` path, then land OpenAI-compatible discovery with credential/baseUrl-scoped caching, deterministic merge semantics, Gateway/Control UI refresh behavior, focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The current absence is high-confidence from source inspection: current main has static OpenAI catalog data and configured model-list plumbing but no OpenAI `/v1/models` refresh path.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but adding a new hook may not be the best solution if the existing provider `catalog.run` live-discovery contract can be reused safely.
