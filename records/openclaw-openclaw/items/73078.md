## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a per-session comparison against each resolved agent's own main session, and the public schema still accepts only `off`, `non-main`, and `all`. The docs now warn about the session-vs-agent pitfall and per-agent overrides work, but there is still no schema-backed main-agent-only policy for dynamic peer agents, so this needs maintainer/security product judgment rather than cleanup closure.

## What This Changes

Request a sandbox policy or clearer contract so only the trusted main agent can run off-sandbox while dynamic peer agents, including their main sessions, remain sandboxed.

## Best Possible Solution

Add a backward-compatible, schema-backed policy such as `main-agent-only` or an equivalent default-agent-aware selector, then update runtime status, config schema/types, sandbox explain output, docs, and tests that cover dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"` and a session key shaped like `agent:feishu-ou_x:main`, the runtime resolves agent id `feishu-ou_x`, resolves that same agent's main key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode looks safer than changing `non-main`, while the existing docs and per-agent override workaround do not remove the misconfiguration risk described here.
