## Summary

Keep open. Current main documents that `non-main` is session-scoped, but the runtime and config schema still have no single `main-agent-only` mode, and dynamically created Feishu agents inherit defaults while their own `agent:<id>:main` sessions compare equal to their per-agent main session and bypass sandboxing. This is a valid sandbox policy/API question rather than an implemented, duplicate, or stale report.

## What This Changes

The issue requests clearer or new sandbox mode semantics so only the default main agent can run on host while dynamic or peer agents, including their main sessions, stay sandboxed without verbose per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the common policy explicitly, likely an additive `main-agent-only` mode or equivalent default-agent-aware sandbox selector, then update `sandbox explain`, config docs, and tests for dynamic peer agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is clear: set `agents.defaults.sandbox.mode` to `non-main` and evaluate `resolveSandboxRuntimeStatus` for `agent:feishu-ou_x:main`; the resolver derives the same per-agent main session key and returns `sandboxed=false`.

## Solution Assessment

Unclear until maintainer/security review chooses the product contract. The narrowest maintainable direction appears to be an additive mode plus docs/tests, because renaming or documentation alone leaves the reported main-agent-only policy verbose and easy to misconfigure.
