## Summary

Keep open. Current main still implements `non-main` as a per-session comparison against each resolved agent's own main session, and the schema still exposes only `off`, `non-main`, and `all`; existing docs clarify the pitfall but do not provide the requested main-agent-only policy. Because this affects sandbox isolation for dynamic peer agents, the next step is maintainer/security product judgment rather than an automatic cleanup close.

## What This Changes

The issue requests a clearer or new sandbox mode that lets the trusted default main agent bypass sandboxing while dynamic peer agents remain sandboxed without verbose per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the reported policy, likely an additive `main-agent-only` mode or equivalent default-agent-aware selector, then update `sandbox explain`, config schema/docs, and tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"`, `resolveSandboxRuntimeStatus` for a dynamic peer route shaped like `agent:feishu-<peer>:main` resolves that same agent's main session key and returns `sandboxed=false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode appears safer than redefining `non-main`, while docs-only clarification would not remove the verbose and easy-to-misconfigure workaround.
