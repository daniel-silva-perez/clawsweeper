## Summary

Keep open. Current main still implements `non-main` as a per-session comparison against each resolved agent's own main session, the config schema still exposes only `off`, `non-main`, and `all`, and the docs only warn about the pitfall rather than providing the requested main-agent-only policy. Because this affects sandbox isolation for dynamic peer agents, the next step needs maintainer/secops product judgment rather than cleanup closure.

## What This Changes

The issue requests a clearer or new sandbox mode so only the main agent bypasses sandbox while dynamic peer-agent main sessions remain sandboxed.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express this policy, likely an additive `main-agent-only` mode or equivalent default-agent-aware selector, then update `sandbox explain`, config schema/docs, and tests covering dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"` and a session like `agent:feishu-ou_peer:main`, the runtime resolves the peer agent's own main session key and returns `sandboxed=false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode appears safer than redefining `non-main`; docs-only clarification explains the trap but does not remove the verbose per-agent override requirement.
