## Summary

Keep open. Current main still treats `non-main` as a session comparison against each resolved agent's own main session, and the schema/docs do not provide a single main-agent-only sandbox policy. Because this can affect sandbox isolation for dynamic peer agents, cleanup closure would be premature.

## What This Changes

Request a clearer or new sandbox policy that lets only the default main agent bypass sandboxing while dynamic peer agents remain sandboxed without per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the requested policy, likely an additive main-agent-only mode or equivalent selector. The implementation should update runtime resolution, sandbox explain output, config schema/generated docs, sandbox docs, and focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"`, a dynamic peer session shaped like `agent:feishu-ou_peer:main` resolves that peer agent's own main key and `shouldSandboxSession` returns false.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode appears safer than changing `non-main`, while documentation-only clarification does not remove the verbose and easy-to-misconfigure workaround.
