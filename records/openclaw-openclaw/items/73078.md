## Summary

Keep open. Current main still implements `non-main` as a per-session comparison against the resolved agent's own main session, the config schema still exposes only `off`, `non-main`, and `all`, and the docs clarify the pitfall without adding a single main-agent-only policy. Because this affects sandbox isolation for dynamic peer agents, the next step should be maintainer/security product judgment rather than cleanup closure.

## What This Changes

The issue requests a clearer or new sandbox mode so the main agent can bypass sandboxing while dynamic peer agents remain sandboxed without verbose per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express this policy, likely an additive `main-agent-only` mode or equivalent default-agent-aware selector. The change should update runtime resolution, `sandbox explain`, config schema/generated docs, sandbox docs, and focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"`, a direct dynamic peer route shaped like `agent:feishu-ou_peer:main` resolves that peer agent's own main session key and `shouldSandboxSession` returns `false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode appears safer than redefining `non-main`, while documentation-only clarification would not remove the verbose and easy-to-misconfigure workaround.
