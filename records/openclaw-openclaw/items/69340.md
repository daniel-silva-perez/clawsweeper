## Summary

Keep this PR open, but it should not merge as written. Current main still has the hyphen-splitting gap this PR is trying to address, but the proposed one-line guard is too broad because it preserves any long dash/hyphen token and bypasses the existing long-token chunking that protects narrow terminals.

## What This Changes

The PR changes `src/tui/tui-formatters.ts` so `isCopySensitiveToken` treats any `FILE_LIKE_RE` token containing an underscore or hyphen as copy-sensitive.

## Best Possible Solution

Preserve genuinely copy-sensitive hyphenated identifiers in TUI output without disabling generic long-token chunking for dash-only separators or long hyphenated prose, and add focused sanitizer regression tests for both preserved and still-chunked cases.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with the PR condition, a token such as `-`.repeat(60) or a long hyphenated prose fragment matches `FILE_LIKE_RE`, contains `-`, and returns before `chunkToken`, so it is no longer constrained to 32-character display segments.

## Solution Assessment

No as written. The narrow maintainable fix should classify only copy-sensitive hyphenated identifiers and should keep generic dash/prose tokens chunkable, ideally with tests similar to the overlapping formatter coverage in #39565.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep generic hyphen runs chunkable:** `src/tui/tui-formatters.ts:82`
  - body: This guard treats every long token containing `-` and only `[A-Za-z0-9._-]` as copy-sensitive. That includes strings like `-`.repeat(60) or long hyphenated prose, which currently get split by `normalizeLongTokenForDisplay` to protect narrow terminals; after this change they bypass `chunkToken` and can overflow the TUI. Please narrow the hyphen heuristic and add regression tests for dash-only/prose tokens that must still wrap.
  - confidence: 0.88
