## Summary

Keep open. Current main still lacks the requested post-startup provider-history catch-up for messages sent while the gateway is offline. Discord and Slack are still live-event driven, Telegram has a provider-specific polling offset guard, and the generic channel adapter does not expose a history catch-up contract that core could call across providers.

## Best Possible Solution

Keep this issue open and implement a bounded, configurable catch-up design. The likely maintainer path is an additive channel/plugin SDK contract for providers that can list recent inbound messages, then provider implementations starting with Discord: resolve a safe per-conversation anchor such as the last outbound bot message, fetch a small time-bounded window after startup/READY, filter bot/self/duplicates through durable keys, enqueue accepted messages through the existing inbound pipeline, and document which channels support catch-up. Related queue/drain persistence work can remain separate unless it supplies reusable dedupe or delivery primitives.
