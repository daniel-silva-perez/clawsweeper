## Summary

Keep open. Current main still lacks the requested broad post-startup inbound catch-up for messages sent while the gateway is offline: BlueBubbles has a provider-specific startup catchup path and Telegram polling has offset persistence, but Discord and Slack remain live-event driven and the channel plugin contract does not expose a generic history replay hook. The related #42157 and #51620 reports are open but cover narrower Telegram webhook persistence and broader queued/in-flight restart loss, so neither is a canonical replacement for this request.

## What This Changes

Request a configurable post-startup inbound message catch-up mechanism so channel providers replay messages sent while the gateway was offline.

## Best Possible Solution

Add an additive, opt-in channel catch-up contract that lets capable providers enumerate recent inbound messages for known conversations after startup, then implement bounded provider-specific replay for Discord and Slack first using durable per-message dedupe, self-message filtering, last-outbound or cursor anchoring, and docs for supported channels. Keep Telegram webhook durability and queued/in-flight restart persistence as companion work unless the same primitives can be shared cleanly.

## Reproduction Assessment

Yes. The issue gives a concrete downtime scenario, and current source shows Discord and Slack only process live delivered events after reconnect; a live credentialed E2E would prove the user-visible drop, but the code path already shows no startup replay exists.

## Solution Assessment

Unclear as proposed. Anchoring to last outbound is a useful starting point, but the maintainable fix needs an explicit provider opt-in contract, per-provider history semantics, durable dedupe, and a time/window limit rather than a core loop that assumes every channel can fetch history the same way.
