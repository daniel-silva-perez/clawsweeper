## Summary

Keep open. Current main still lacks the requested broad post-startup inbound catch-up for messages sent while the gateway was offline. BlueBubbles has a provider-specific startup catchup path, and Telegram long polling persists offsets, but Discord and Slack remain live-event driven and the channel SDK does not expose a generic history catch-up contract for core to call across providers. The related Telegram webhook issue #42157 is narrower and open, not a canonical replacement for this broader request.

## Best Possible Solution

Keep this issue open and design an additive, bounded channel catch-up surface. A conservative path is to add a plugin SDK contract for providers that can list recent inbound messages for active conversations, then implement provider-specific handlers starting with Discord and Slack: anchor to a safe last-outbound or durable cursor, fetch a small time-bounded window after startup/READY, filter self/bot/duplicates, enqueue through the normal inbound path, and document supported channels. Telegram webhook queue persistence can remain a narrower companion thread unless the same durable delivery primitives become reusable.

## Risks / Open Questions

- A generic catch-up implementation can duplicate or replay stale user messages unless it is bounded, configurable, provider-aware, and deduped with durable message keys.
- Telegram webhook loss is an ACK/persistence problem, while Discord/Slack history catch-up is a provider-history problem; solving one path may not solve the other without a deliberate shared design.
