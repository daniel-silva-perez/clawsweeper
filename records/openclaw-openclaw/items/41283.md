## Summary

Keep this issue open. Current main still intentionally returns `INVALID_REQUEST` / `unknown agent id` when `tools.catalog` or `tools.effective` receives an unconfigured agent ID, but the user-visible problem is broader than that error: a `tools.profile: "full"` setup should not reduce a normal runtime tool list to only messaging/session tools. The report still lacks the failing `agentId`, `sessionKey`, effective/catalog request payloads, redacted agent/tool config, provider/channel context, and gateway logs needed to separate stale local agent state from a real policy/runtime regression.

## Best Possible Solution

Keep the issue open and ask for a fresh reproduction on current OpenClaw with the failing `agentId`, active `sessionKey`, `tools.catalog` and `tools.effective` params/responses, `openclaw agents list`, redacted `agents.list` and `tools` config, selected provider/model/channel/plugin, web-search provider config, install method/path, and gateway logs. If the data shows stale local state, improve doctor or Control UI diagnostics; if a valid full-profile session still sees only messaging/session tools, trace `resolveEffectiveToolPolicy`, runtime tool creation, model/provider filtering, group/channel policy, and web-search provider readiness.
