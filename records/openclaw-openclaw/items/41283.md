## Summary

Keep this issue open. Current main still intentionally returns `INVALID_REQUEST` / `unknown agent id` when `tools.catalog` or `tools.effective` receives an agent ID that is not in the runtime config, and `tools.profile: "full"` is still the unrestricted baseline. The remaining question is whether the reporter hit stale local agent state, a UI/session mismatch, provider/search readiness, or a real runtime filtering bug; the issue still lacks the concrete request/config/session data needed to close it safely.

## Best Possible Solution

Keep the issue open and ask for a fresh reproduction on current OpenClaw with the failing `agentId`, `sessionKey`, `tools.catalog` and `tools.effective` params/responses, `openclaw agents list`, redacted `agents.list` and `tools` config, selected provider/model/channel/plugin, web-search provider config, install method/path, and gateway logs. If it is stale local state, improve doctor or Control UI diagnostics; if a valid full-profile session still sees only messaging/session tools, trace `resolveEffectiveToolPolicy`, runtime tool creation, provider/model filtering, group/channel policy, and web-search provider readiness.
