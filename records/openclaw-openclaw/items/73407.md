## Summary

Keep this open for maintainer follow-up: current main and the latest v2026.4.27 release still carry the WhatsApp plugin dependency path from @openclaw/whatsapp to Baileys to @whiskeysockets/libsignal-node, and the pinned upstream libsignal-node metadata declares GPL-3.0. The prior ClawSweeper comment reached the same keep-open result, and this remains a concrete published-plugin compliance/security question rather than a cleanup close.

## What This Changes

The issue asks maintainers to address the GPL-3.0 libsignal dependency reached through the WhatsApp plugin's Baileys runtime dependency because of license infection risk.

## Best Possible Solution

Maintain the WhatsApp plugin only with an approved license posture: either replace the Baileys/libsignal path with an acceptable dependency, isolate or redistribute it in a maintainer-approved way with clear documentation, or disable/remove the published WhatsApp plugin path until compliance is settled.

## Reproduction Assessment

Yes. This is not an interactive runtime bug; it reproduces by inspecting current main and v2026.4.27 dependency metadata in extensions/whatsapp/package.json, pnpm-lock.yaml, the WhatsApp session code, and the pinned upstream libsignal-node package metadata.

## Solution Assessment

Unclear. The report identifies a real dependency path, but the best fix is a maintainer/legal product decision rather than a narrow code change that ClawSweeper should choose automatically.
