## Summary

Keep open. Current main still has a real Google Chat Space/DM metadata gap, the reporter confirmed the Space behavior remains operationally relevant on April 30, 2026, and an open unmerged PR specifically references this issue with a proposed fix. The issue should not be closed before that PR is reviewed/landed or intentionally consolidated with the related Google Chat routing work.

## What This Changes

Google Chat Space/group webhook messages should create and process group sessions instead of returning HTTP 200 without agent handling while DMs continue to work.

## Best Possible Solution

Google Chat inbound handling should reliably distinguish DMs, group chats, and Spaces using current Google payload fields, create `googlechat:group` sessions for Space messages, and preserve Space/thread metadata through reply and follow-up delivery.

## Reproduction Assessment

Yes. The issue has concrete end-to-end steps and a same-day reporter confirmation, and current main can be source-checked against modern Google payload fields because it only reads deprecated `space.type`. I did not run a live Workspace replay in this read-only review.

## Solution Assessment

Unclear whether the already-open PR is the complete fix. Adding `spaceType` and `singleUserBotDm` support is the narrow first step, but maintainers should verify whether thread/space metadata preservation belongs in that PR or should be consolidated with #69422.
