## Summary

Keep this PR open. Current main still lacks fenced-code stripping in the shared markdown stripper, so the reported BlueBubbles/iMessage approval-message problem remains relevant, but the proposed regex has a blocking plain-text regression for valid same-line triple-backtick code spans.

## What This Changes

The PR adds a fenced-code-block removal pass to `stripMarkdown` and new tests for fenced blocks, approval-message formatting, ordering, and adjacent surrounding text.

## Best Possible Solution

Land a narrow shared `stripMarkdown` fix that strips block-style fenced code while preserving valid inline code spans and existing TTS/plain-text behavior. Keep the broader automatic outbound markdown stripping policy in #56283 separate unless that PR is reviewed and merged first.

## Reproduction Assessment

Yes. A focused check against current main can reproduce it with an exec approval payload or `stripMarkdown("```sh\ngh repo list\n```")`: the source shows fenced blocks are emitted and BlueBubbles calls the shared stripper, which currently has no fenced-block pass. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as written. The targeted helper is the right boundary for this focused bug, but the regex should be constrained to block fences or replaced with the existing fence parser so inline triple-backtick spans do not lose their contents.

## Review Findings
