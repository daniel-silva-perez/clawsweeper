## Summary

Keep open. Current main still has Telegram runtime/config/group status checks, but it does not inspect persisted Telegram JSON artifacts for readability, size, parse/schema health, or show a backup/reset recovery path; the linked Keep My Claw skill is useful backup coverage, not the requested status/recovery UX.

## What This Changes

The issue asks for Telegram status/doctor diagnostics and safe recovery guidance when persisted Telegram session or state JSON is oversized, malformed, unreadable, or schema-invalid.

## Best Possible Solution

Add a Telegram-owned read-only state diagnostic that feeds channel status and doctor output. It should report known state artifact paths, readability, file size, JSON parse/schema status, likely impact, and a backup-before-reset recommendation, while leaving destructive reset/quarantine behavior to a separate explicit design decision.

## Reproduction Assessment

Unclear for the original production wedge because the bad file was not attached. There is a high-confidence focused reproduction for the missing diagnostic path: create malformed or oversized Telegram state JSON at the known persisted paths and current loaders collapse it to null/empty or verbose logs while status has no field to report it.

## Solution Assessment

Yes, the narrow maintainable fix is non-destructive diagnostics first. A reset command or startup auto-quarantine may be useful later, but it carries data-loss policy and should not block the status warning work.
