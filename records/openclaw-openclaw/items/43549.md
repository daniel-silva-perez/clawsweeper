## Summary

Keep open. Current main still lacks Telegram-owned persisted-state health diagnostics in status or doctor, and the known Telegram JSON state readers still collapse malformed or unreadable local state to null, empty state, or verbose-only logging, so the reported recovery gap remains valid.

## What This Changes

The issue requests Telegram persisted-session state diagnostics and recovery guidance when local JSON state becomes malformed, oversized, or otherwise wedges the channel.

## Best Possible Solution

Add a Telegram-owned, read-only persisted-state health diagnostic that feeds channel status and doctor warnings. It should report known Telegram state artifact paths, existence/readability, size, JSON parse/schema health, likely impact, and backup-before-reset guidance, leaving destructive reset or startup quarantine to a separately approved follow-up.

## Reproduction Assessment

Unclear for the original production wedge because the bad file was not attached. There is a high-confidence focused check for the missing diagnostic behavior: malformed or oversized JSON at known Telegram state paths is treated as null/empty or verbose-only, while status and doctor have no state-health warning path.

## Solution Assessment

Yes, the narrowest maintainable first fix is non-destructive diagnostics in the Telegram plugin status/doctor surface plus docs. A reset command or auto-quarantine may be useful later, but it is a data-loss policy decision and should not block warning coverage.
