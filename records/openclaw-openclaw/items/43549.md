## Summary

Keep open. Current main has Telegram runtime status warnings for polling/webhook failures and generic session-store tolerance for malformed or oversized stores, but I did not find a Telegram-specific status diagnostic or backed-up reset flow for bad persisted Telegram/session JSON. The discussion already includes a useful manual workaround and the Keep My Claw backup skill as a safety net, but neither replaces the requested core diagnostic/recovery UX.

## What This Changes

The issue asks for Telegram status and recovery tooling to detect oversized, malformed, or unusable persisted session JSON and guide a safe reset path.

## Best Possible Solution

Add a Telegram-owned persisted-state diagnostic that is visible from `openclaw channels status --probe` and the broader status/doctor path. It should inspect the relevant Telegram/session JSON artifacts for existence, readability, size, JSON parse/schema validity, and account/state path, then recommend a backed-up reset/restart flow. A destructive or automatic quarantine should be opt-in and narrowly scoped.
