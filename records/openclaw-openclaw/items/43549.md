## Summary

Keep open. Current main still lacks Telegram persisted-state health diagnostics in status or doctor, and current Telegram state readers still collapse malformed/unreadable JSON to null, empty state, or verbose-only logging, so the reported operator recovery gap remains valid and narrow enough for a focused fix.

## What This Changes

The issue requests Telegram status/doctor diagnostics and recovery guidance for malformed, oversized, or otherwise poisoned persisted Telegram session/state JSON.

## Best Possible Solution

Add a Telegram-owned, read-only persisted-state health diagnostic that feeds `openclaw status`, `openclaw channels status`, and doctor warnings. It should report known Telegram state artifact paths, existence/readability, size, JSON parse/schema health, likely impact, and backup-before-reset guidance, with destructive reset or startup quarantine left to a separately approved follow-up.

## Reproduction Assessment

Unclear for the original production wedge because the bad file was not attached. ## Solution Assessment

Yes, the narrowest maintainable first fix is non-destructive diagnostics in the Telegram plugin status/doctor surface plus docs. A reset command or auto-quarantine may be useful later, but that is a data-loss policy decision and should not block the warning path.
