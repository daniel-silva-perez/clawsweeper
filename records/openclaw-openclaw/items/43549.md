## Summary

Keep open. Current main still lacks Telegram persisted-state diagnostics in status or doctor, and the existing state readers collapse malformed JSON to empty/null state or verbose-only logging, so the reported recovery gap remains valid.

## What This Changes

The issue asks Telegram status/doctor tooling to detect malformed, oversized, unreadable, or poisoned persisted Telegram session/state JSON and guide safe recovery.

## Best Possible Solution

Add a Telegram-owned read-only state-health diagnostic that feeds channel status and doctor output. It should report known Telegram state artifact paths, readability, size, JSON parse/schema health, likely impact, and backup-before-reset guidance, with destructive reset or startup quarantine left to a separately approved follow-up.

## Reproduction Assessment

Unclear for the original production wedge because the bad file was not attached. There is a high-confidence focused reproduction for the missing diagnostic path: place malformed or oversized Telegram state JSON at the known persisted paths and current main treats it as null/empty or verbose-only while status has no state-health warning.

## Solution Assessment

Yes, the narrowest maintainable fix is non-destructive diagnostics in the Telegram plugin status/doctor surface plus docs. A reset command or auto-quarantine may be useful later, but that is a data-loss policy decision and should not block the warning/diagnosis work.
