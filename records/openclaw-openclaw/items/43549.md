## Summary

Keep this issue open. Current main still does not inspect Telegram persisted state in `openclaw channels status` / `openclaw status`, and it still lacks a safe Telegram-specific backup/reset/quarantine recovery command. The linked Keep My Claw skill is useful backup mitigation for corrupted local state, but it does not replace first-party Telegram diagnostics and recovery UX.

## Best Possible Solution

Keep this issue open and implement the fix in the Telegram plugin/status path. Add a read-only inspector for the resolved Telegram update-offset JSON plus relevant shared `sessions.json` entries, reporting path, size, parse/schema status, mtime, and account. Surface failures through `ChannelStatusIssue` in `openclaw channels status` / `openclaw status`, and add a non-destructive recovery flow that backs up or quarantines only the affected Telegram artifact and tells the operator whether to restart the channel or gateway. Use a generic core seam only if multiple channel plugins need the same persisted-state diagnostic API.
