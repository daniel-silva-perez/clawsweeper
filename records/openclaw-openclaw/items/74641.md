## Summary

Keep open: current main still has the reported ACP resolver short-circuit, and this PR targets the right area, but the patch should be adjusted before merge because an unresolved explicit token can now continue into requester-session fallback instead of preserving the bad-target behavior after the thread-bound lookup fails.

## What This Changes

The PR changes ACP target resolution to fall through after an unresolved token, adds a regression test for thread-bound `/acp steer`, and adds an Unreleased changelog fix entry.

## Best Possible Solution

Update the resolver so a failed token lookup can still use the current thread binding for Discord's auto-filled thread id case, but preserve the bad-token error before requester-session fallback when a non-empty supplied token remains unresolved; keep the regression test, add coverage for the no-thread invalid-token case, and land after normal ACP validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve bad-token handling after thread fallback:** `src/auto-reply/reply/commands-acp/targets.ts:62-66`
  - body: After this change, an unresolvable non-empty token falls through not only to the bound-thread lookup but also to `resolveRequesterSessionKey`. A typo such as `/acp close bad-token` from an ACP requester session can therefore act on the requester session instead of returning the documented bad-target error. Keep the thread-bound fallback for Discord's auto-filled thread id, but restore `Unable to resolve session target` before requester fallback when a supplied token still has not resolved.
  - confidence: 0.87
