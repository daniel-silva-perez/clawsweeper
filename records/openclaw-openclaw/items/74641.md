## Summary

Keep this PR open: it addresses a real ACP resolver bug, but the current patch lets an unresolvable explicit token continue into requester-session fallback after thread-bound lookup fails, which can make a typo target the requester session instead of returning the documented bad-target error.

## What This Changes

The PR changes ACP target resolution so unresolved explicit session tokens fall through to existing fallbacks, adds a bound-thread `/acp steer` regression test, and adds an Unreleased changelog fix entry for #66299.

## Best Possible Solution

Keep the resolver change narrowly scoped: unresolved explicit tokens should try the thread-bound binding, then return `Unable to resolve session target` before requester-session fallback; requester fallback should remain available only when no explicit token was supplied. Add focused coverage for the no-thread invalid-token case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve bad-token handling before requester fallback:** `src/auto-reply/reply/commands-acp/targets.ts:62-69`
  - body: After this change, an unresolvable non-empty target falls through past the thread-bound lookup into `resolveRequesterSessionKey`. A typo such as `/acp close bad-token` from an ACP requester session can then act on the requester session instead of returning the documented bad-target error, so return `Unable to resolve session target` before requester fallback when a supplied token remains unresolved.
