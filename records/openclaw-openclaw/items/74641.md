## Summary

Keep this PR open and do not merge it as-is. The branch fixes the bound-thread success case, but it still lets an unresolvable explicit ACP target fall through into requester-session fallback when no thread binding resolves, which can make a bad token act on the current requester session instead of returning the bad-target error described in the linked report and docs.

## What This Changes

The PR changes ACP target resolution so unresolved explicit session tokens fall through to existing fallbacks, adds a bound-thread `/acp steer` regression test, and adds an Unreleased changelog entry for #66299.

## Best Possible Solution

Keep the narrow thread-bound fallback fix, but insert the explicit-token error guard after thread-bound resolution and before requester fallback. Requester fallback should remain available only when no explicit target token was supplied, with focused coverage for both the bound-thread success case and the no-binding bad-token case.

- **[P2] Preserve bad-token handling before requester fallback:** `src/auto-reply/reply/commands-acp/targets.ts:62-69`
  - body: When a non-empty `--session` token fails lookup and no thread binding resolves, this falls through into `resolveRequesterSessionKey`. In an ACP requester context that can return `CommandTargetSessionKey` or `params.sessionKey`, so a typo such as `/acp close bad-token` may act on the requester session instead of returning the bad-target error. Return `Unable to resolve session target` after thread-bound lookup when `token` is set.
