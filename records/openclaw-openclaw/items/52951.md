## Summary

Keep this PR open. Current main at 69fb7455c6223682a45dd8b8fb720bc3231bfa43 still does not implement the proposed `tools.fs.roots` contract: config types/schema remain `workspaceOnly`-only, the effective FS policy does not carry root metadata or access modes, host read/write/edit/apply_patch wiring uses workspace guards only, and media/outbound access still uses default/agent local roots rather than configured per-agent FS roots. The PR remains an active implementation candidate for a real missing capability, but because it changes a security-sensitive filesystem boundary across tools, media, outbound delivery, and channel adapters, it needs explicit maintainer review rather than automated cleanup closure.

## Best Possible Solution

Keep the PR open. Maintainers should decide whether `tools.fs.roots` is the desired core config contract; if yes, review and land a narrowed, well-tested implementation covering schema/docs, host FS read/write/edit/apply_patch enforcement, media/outbound propagation, sandbox bypass behavior, and the pinned-read/write/delete edge cases. If maintainers prefer a different filesystem sandboxing model, close only with an explicit product decision and a canonical follow-up.
