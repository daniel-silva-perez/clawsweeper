## Summary

Keep open. Current main at 955b4df0939f90b4b6de5e7e6f3149dd125e1e50 still only exposes the binary `tools.fs.workspaceOnly` filesystem policy. It does not implement the proposed `tools.fs.roots` type/schema, per-root access modes, multi-root host tool enforcement, root-scoped apply_patch behavior, or rooted media/outbound propagation. The PR remains an active implementation candidate for a real security-sensitive configuration gap, not cleanup-close material.

## Best Possible Solution

Keep this PR open. Maintainers should decide whether `tools.fs.roots` is the desired core contract. If yes, review and land a narrowed, well-tested implementation covering schema/docs, host read/write/edit/apply_patch enforcement, media/outbound propagation, sandbox bypass behavior, pinned read/write/delete safety, and generated API/config baselines. If maintainers prefer a different filesystem sandboxing model, close only with an explicit product/security decision and a canonical follow-up.
