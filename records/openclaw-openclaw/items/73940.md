## Summary

Keep this PR open for maintainer handling. Current main still lacks the Feishu post-start SDK retry-exhaustion recovery path, but this PR should not merge as-is because the supervisor backoff resets after any prior connection and can stay at the minimum delay during repeated brief connect/disconnect cycles; it also overlaps the open maintainer PR #73739 for the same #52618 recovery family.

## What This Changes

The PR adds Feishu WSClient lifecycle callback forwarding, supervisor-driven client recreation on SDK retry exhaustion or stalled reconnect state, bot identity preservation across replacement cycles, focused tests, and a changelog entry.

## Best Possible Solution

Use one maintainer-approved Feishu plugin transport implementation as the canonical #52618 fix: observe terminal SDK retry exhaustion through the Lark lifecycle contract, preserve bot identity only across recovery cycles, keep shutdown cleanup deterministic, and make supervisor backoff increase across rapid reconnect failures unless a connection has been stable long enough to reset safely. Decide explicitly whether getReconnectInfo-based stale-state detection belongs in the same PR or a follow-up.

## Reproduction Assessment

Yes. A focused reproduction is to simulate a WSClient whose start() succeeds, invoke the SDK onError callback with the published `WebSocket reconnect exhausted after N attempts` terminal error, and verify current main never recreates the client while this PR does; the PR finding is reproducible by repeating onReady/onError cycles and observing the delay stays at 1000ms.

## Solution Assessment

No, not as currently proposed. The general Feishu-owned supervisor direction is right, but resetting backoff after any prior connection is not the narrowest safe behavior; a stable-connection threshold, continued backoff across rapid failures, and coordination with #73739 would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reset backoff only after a stable connection:** `extensions/feishu/src/monitor.transport.ts:318-321`
  - body: This resets `attempt` whenever the cycle has ever reached `onReady` or `onReconnected`, so a flaky network that briefly connects and then exhausts SDK retries will recreate clients at the 1s minimum forever. That defeats the bounded backoff and can amplify WSClient churn and logs; keep the attempt increasing unless the connection stayed healthy for a meaningful interval, or add equivalent rate limiting/jitter.
  - confidence: 0.88
