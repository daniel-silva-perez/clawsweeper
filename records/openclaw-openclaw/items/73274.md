## Summary

Keep open. Current main still lacks a durable transcript append helper on the public plugin runtime surface, while the internal append helper and the reporter's linked plugin workaround show the use case is real; the remaining decision is an API/security design question, not conservative cleanup.

## What This Changes

Request to expose a durable plugin runtime API for appending assistant messages to another session transcript so cron/announce output can persist in target channel context instead of relying on ephemeral system events.

## Best Possible Solution

Design an additive, narrow plugin runtime transcript-write capability that wraps the existing session manager path, defaults to assistant-only delivery-mirror records, preserves idempotency and transcript update events, requires an explicit trusted-plugin capability or config opt-in, and updates SDK docs plus runtime/contract tests. Separately evaluate whether built-in isolated cron announce delivery should mirror into target channel sessions without a plugin.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/API request; the high-confidence static check is that current main lacks `api.runtime.agent.session` transcript append support while the internal durable helper and ephemeral `enqueueSystemEvent` fallback both exist.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A durable helper is plausible, but exposing the internal helper directly with arbitrary session keys and user-role writes is broader than the narrow, capability-gated assistant-mirror API that current code suggests.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would let plugin code persist content into other sessions' transcript history.

Concerns:

- **[medium] Gate durable transcript writes by plugin capability:** `src/plugins/runtime/types-core.ts:142`
  - body: Adding a writer under `api.runtime.agent.session` would create durable cross-session write authority. That should require an explicit trusted-plugin capability or config opt-in plus clear target scoping and audit semantics.
  - confidence: 0.88
- **[medium] Keep transcript writes assistant-only unless explicitly designed:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact append helper rejects non-assistant messages, but the requested shape includes `role?: "assistant" | "user"`. Allowing user-role writes would let plugins forge user turns unless maintainers define role limits, attribution, and transcript validation first.
  - confidence: 0.84

## Risks / Open Questions

- A broad transcript-write API would let trusted plugin code persist content into arbitrary sessions unless capability, target scoping, attribution, and audit semantics are designed first.
- The proposed `role?: "assistant" | "user"` shape is wider than the existing internal helper and could allow plugins to forge user turns if accepted as-is.
- Maintainers still need to decide whether isolated cron announce mirroring should be native behavior instead of, or in addition to, a generic plugin runtime helper.
