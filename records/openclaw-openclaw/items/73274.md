## Summary

Keep open. Current main still has no durable transcript append helper on the public plugin runtime surface, and the reporter's linked `openclaw-plugin-cron-context` workaround is good evidence that the existing in-memory system event path does not cover the requested cross-session persistence. There is a related durable next-turn injection API, but its docs explicitly frame it as one-turn context rather than permanent transcript history, so it does not fully implement this request.

## What This Changes

Expose a durable plugin runtime API that lets plugins append assistant output into another session's persistent transcript by session key.

## Best Possible Solution

Design an additive, narrow plugin runtime transcript-write API that wraps the host session manager path, defaults to assistant-only delivery-mirror semantics, requires explicit trusted-plugin capability or operator opt-in, preserves idempotency and transcript update events, runs the same redaction/guard hooks as other transcript writes, and adds SDK docs plus focused contract/runtime tests. Maintainers should also decide whether isolated cron announce delivery should mirror into target channel sessions without a plugin.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks `api.runtime.agent.session` transcript append support while the internal durable helper, ephemeral `enqueueSystemEvent`, and durable-but-not-permanent next-turn injection paths are present.

## Solution Assessment

Unclear. A persistent transcript helper is plausible, but exposing the internal helper directly with arbitrary session keys or user-role writes is not proven to be the narrowest secure API; an assistant-only, opt-in mirror helper is the safer starting point.
