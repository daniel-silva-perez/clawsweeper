## Summary

Keep open. Current main has the internal persistent transcript append helper and the existing in-memory system event workaround, but it does not expose a durable transcript append method through `api.runtime.agent.session`; the linked `openclaw-plugin-cron-context` workaround is useful evidence that the gap is real, while the public API shape needs maintainer/security design before implementation.

## What This Changes

Expose a persistent, sessionKey-targeted transcript append helper to plugin runtime so plugins can write cron or delivery output into another session history instead of relying on `enqueueSystemEvent`.

## Best Possible Solution

Design an additive, narrow plugin runtime transcript API that preserves the existing session manager/idempotency/update-event behavior, defaults to assistant-only durable mirror writes, requires an explicit trusted-plugin capability or config opt-in, documents the security model, and adds contract tests plus runtime mocks. In parallel, keep the core cron persistence bug tracked separately so built-in cron delivery works even without a plugin workaround.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence check is static: current main lacks the requested `api.runtime.agent.session` transcript append method, while the internal durable helper and the ephemeral system-event fallback are both present in source.

## Solution Assessment

Unclear. A persistent transcript helper is a plausible direction, but exposing the internal helper directly, especially with arbitrary target sessions and user-role writes, is not yet proven to be the narrowest secure API.

## Security Review

Status: needs_attention

Summary: The requested API would let plugins persist cross-session prompt/history content, so it needs explicit capability, attribution, and role constraints before implementation.

Concerns:

- **[medium] Gate persistent transcript writes by plugin capability:** `src/plugins/runtime/types-core.ts:142`
  - body: Adding this helper under `api.runtime.agent.session` would let plugin code write durable content into any session key. That is stronger than the current in-memory system event queue and should require an explicit trusted-plugin permission and audit path.
  - confidence: 0.88
- **[medium] Avoid role-flexible forged transcript entries:** `src/config/sessions/transcript.ts:157`
  - body: The current internal append helper writes assistant delivery-mirror messages. The proposed `role?: "assistant" | "user"` shape could let plugins forge user turns unless maintainers define role limits and transcript-shape validation first.
  - confidence: 0.82
