## Summary

Keep open. Current main still has the durable internal assistant transcript append helper and the in-memory system-event workaround, but it does not expose any durable transcript append method through `api.runtime.agent.session`; the linked `openclaw-plugin-cron-context` workaround is useful evidence that the gap is real, while the public API shape needs maintainer security and product design before implementation.

## What This Changes

Expose a durable Plugin Runtime API under `api.runtime.agent.session` so plugins can append assistant transcript messages to another session by `sessionKey`, preserving cron/channel output across gateway restarts.

## Best Possible Solution

Design an additive, narrow Plugin Runtime transcript API that reuses the session manager/idempotency/update-event path, defaults to assistant-only durable mirror writes with clear attribution, requires an explicit trusted-plugin capability or config opt-in, validates target session ownership/existence, documents the security model, and adds contract tests plus runtime docs. Keep any built-in cron delivery persistence bug tracked separately from the public plugin API design.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence check is static: current main lacks a durable transcript append method under `api.runtime.agent.session`, while the internal append helper and the ephemeral system-event queue both exist in source.

## Solution Assessment

Unclear. A persistent transcript helper is a plausible direction, but directly exposing the internal helper with arbitrary target sessions and `user` role writes is not yet the narrowest secure solution; an assistant-only, capability-gated API is safer.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would let plugin code persist cross-session prompt/history content.

Concerns:

- **[medium] Gate persistent transcript writes by plugin capability:** `src/plugins/runtime/types-core.ts:142`
  - body: Adding this under `api.runtime.agent.session` would let native plugin code write durable content into any session key, which is stronger than the current in-memory system event queue and should require an explicit trusted-plugin permission or operator opt-in.
  - confidence: 0.9
- **[medium] Keep transcript roles constrained and attributed:** `src/config/sessions/transcript.ts:224`
  - body: The existing internal helper writes assistant delivery-mirror messages only. The proposed role-flexible shape could let plugins forge user turns unless maintainers define role limits, validation, attribution, and audit semantics first.
