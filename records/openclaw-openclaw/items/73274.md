## Summary

Keep open. Current main still lacks a durable transcript append helper on the public plugin runtime surface, while the internal helper, delivery mirroring paths, and tests already exist. The reporter's linked openclaw-plugin-cron-context workaround is concrete evidence that the current in-memory system event path does not cover cross-session persistent context, but the requested API needs maintainer security and SDK design before implementation.

## What This Changes

Request to expose a public plugin runtime API for appending durable assistant transcript entries to another session, primarily so cron output delivered to a channel can also become channel session history.

## Best Possible Solution

Design an additive, host-owned plugin runtime transcript-write capability that preserves the internal helper's assistant-only delivery-mirror semantics, idempotency, session normalization, and transcript update events while requiring explicit trusted-plugin or operator authorization for durable cross-session writes. Maintainers should also decide whether isolated cron announce delivery should mirror directly into target channel sessions without a third-party plugin workaround.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks api.runtime.agent.session transcript append support while the internal durable helper and the ephemeral enqueueSystemEvent workaround are both present.

## Solution Assessment

Unclear. A persistent helper is plausible, but directly exposing the internal helper with arbitrary session keys or user-role writes is not proven to be the narrowest secure solution; an assistant-only, opt-in delivery-mirror helper is the safer starting point.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would grant plugin code durable cross-session transcript write authority.

Concerns:

- **[medium] Gate durable transcript writes by explicit trust:** `src/plugins/runtime/types-core.ts:142`
  - body: Adding a write helper under api.runtime.agent.session would let plugins persist content into other sessions. That authority should require an explicit capability or operator opt-in beyond the current conversation-hook gate.
  - confidence: 0.9

  - body: The internal exact helper rejects non-assistant messages, while the proposed role option includes user writes. Public exposure should preserve assistant-only delivery-mirror semantics unless maintainers define attribution and role limits.
  - confidence: 0.86
