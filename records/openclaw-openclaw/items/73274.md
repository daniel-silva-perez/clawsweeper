## Summary

Keep open. Current main still has the internal durable transcript append helper and the in-memory system-event workaround, but it does not expose a durable `api.runtime.agent.session` transcript append method for plugins. The linked `openclaw-plugin-cron-context` workaround is useful evidence that the gap is real, while the public API shape needs maintainer security/product design before implementation.

## What This Changes

Request to expose a plugin runtime API that lets trusted plugins durably append assistant transcript entries to another session by `sessionKey` instead of relying on ephemeral `enqueueSystemEvent` queues.

## Best Possible Solution

Design an additive, narrow plugin runtime transcript API that reuses the existing session manager, idempotency, dedupe, and transcript-update behavior; restrict it to attributed assistant delivery-mirror writes by default; require explicit trusted-plugin/operator opt-in for cross-session durability; document the security model; and add plugin runtime contract tests plus transcript helper coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence check is static: current main lacks the requested `api.runtime.agent.session` transcript append method, while both the internal durable helper and the ephemeral system-event fallback are present.

## Solution Assessment

Unclear. A durable transcript helper is a plausible solution, but directly exposing the internal helper with arbitrary target sessions and optional user-role writes is not yet proven to be the narrowest secure public API.

## Risks / Open Questions

- A durable cross-session transcript write API would let plugins persist prompt/history content into arbitrary sessions unless it is gated by explicit capability/config and an audit model.
- The proposed role-flexible shape is broader than the current internal assistant-only delivery mirror and could allow forged user turns unless maintainers define role and attribution limits first.
- Maintainers still need to decide whether the best product fix is a generic plugin runtime API, core cron delivery persistence, or both.
