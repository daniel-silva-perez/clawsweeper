## Summary

Keep open. Current main still lacks a durable transcript append helper on the public plugin runtime surface, and the reporter's linked openclaw-plugin-cron-context workaround is concrete evidence that the existing in-memory system event path does not cover cross-session persistence. The requested API is plausible, but it is security-sensitive and needs maintainer API design before implementation.

## What This Changes

The issue requests an additive public Plugin Runtime API that lets plugins persist assistant transcript messages into another existing session by sessionKey.

## Best Possible Solution

Design a narrow, additive plugin runtime transcript-write API that wraps the host-owned session manager path, defaults to assistant-only delivery-mirror semantics, requires explicit trusted-plugin capability or operator opt-in, preserves idempotency and transcript update events, and runs through the same redaction/guard path as other transcript writes. Maintainers should also decide whether isolated cron announce delivery should mirror directly into the target channel session without a plugin workaround.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks api.runtime.agent.session transcript append support while the internal durable helper, ephemeral system-event path, and durable-but-once next-turn injection path are all present.

## Solution Assessment

Unclear. A persistent transcript helper is plausible, but directly exposing the internal helper with arbitrary session keys or user-role writes is not proven to be the narrowest secure API; an assistant-only, opt-in mirror helper is the safer starting point.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would let plugin code persist content into other sessions' transcript history.

Concerns:

- **[medium] Gate durable transcript writes by explicit trust:** `src/plugins/runtime/types-core.ts:142`
  - body: Adding a write helper under api.runtime.agent.session would grant durable cross-session transcript authority to plugin code. This should require an explicit capability or operator opt-in beyond the current conversation and prompt hook gates.
  - confidence: 0.9
- **[medium] Keep public transcript writes assistant-only and guarded:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact helper rejects non-assistant messages, while the proposed role option would let plugins forge user turns. The public API should preserve assistant-only delivery-mirror semantics unless maintainers define attribution and role limits.
  - confidence: 0.86
- **[medium] Apply transcript redaction guards before exposure:** `src/config/sessions/transcript.ts:280`
  - body: Current main still has bare appendMessage call sites in transcript append paths. Exposing this through plugin runtime before resolving guard semantics could persist sensitive plugin-supplied content without the same protections as other transcript writes.
  - confidence: 0.82
