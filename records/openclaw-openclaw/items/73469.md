## Summary

Keep open. Current main still uses the same Feishu streaming-card path for this report: no relevant Feishu streaming, mention, send, or docs files changed since the prior review, and the live path still sends a placeholder Card Kit card before final agent text exists, then applies the final mention-bearing text through later Card Kit updates without notification-capable mention metadata. The related open #60103 is useful for syntax normalization, but it does not supersede this Card Kit notification-timing bug.

## What This Changes

The issue asks Feishu streaming-card replies to send push notifications when agent-authored response text includes `<at user_id="...">name</at>` mentions.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves mention notification semantics for agent-authored tags while making any streaming tradeoff explicit. The end state should cover agent-authored tags, existing inbound `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and a documented fallback when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, return text containing `<at user_id="...">...</at>`, then inspect that current main sends the initial card reference before final text and applies the mention-bearing text through later Card Kit updates.

## Solution Assessment

Unclear. The fix should stay in the Feishu plugin, but the narrowest maintainable implementation needs a Feishu-supported notification strategy and an explicit UX tradeoff rather than a core channel change.
