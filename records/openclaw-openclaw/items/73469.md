## Summary

Keep open. Current main still sends Feishu streaming replies as an initial Card Kit card reference before the final agent text is known, then applies mention-bearing content through later Card Kit updates/close calls. The related open syntax-normalization issue is useful context, but it does not supersede this notification-timing problem.

## What This Changes

The issue asks Feishu streaming card replies to trigger push notifications when agent-authored text contains Feishu `<at user_id="...">...</at>` mention tags.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves mention notification semantics for agent-authored tags. The end state should coordinate with #60103 for syntax normalization, but also explicitly handle the Card Kit first-notification timing with a documented fallback when Feishu cannot notify from streaming card updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `channels.feishu.renderMode: "card"` with streaming enabled and an agent final response containing `<at user_id="ou_xxx">Name</at>`; current main sends the card reference first and applies that text later through Card Kit content updates. A live Feishu tenant push-notification check would complete external verification.

## Solution Assessment

Unclear. Adding a `mentions` parameter to `close()` alone is unlikely to solve this because the current close path maps to Card Kit content/settings updates and the checked upstream schemas do not show a close-time mention-notification payload; a safer fix needs a Feishu-specific notification strategy.
