## Summary

Keep open. Current main still delivers Feishu streaming-card text, including mention-looking content, after the initial interactive card message has already been sent, so there is no implemented path that can reliably trigger Feishu mention push notifications for agent-authored `<at>` tags. The related open syntax-normalization issue #60103 is adjacent but does not cover the Card Kit notification timing problem.

## What This Changes

The issue requests Feishu streaming-card replies to send push notifications when an agent-authored response includes `<at user_id="...">...</at>` mention tags.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves notification semantics for agent-authored mentions. The end state should cover raw `<at user_id>` parsing, existing `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and a documented fallback when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, have the agent return a `<at user_id="...">...</at>` tag, and compare the current code path: the first Feishu message is only a card reference while the mention text is applied later by Card Kit updates.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` does not look sufficient because `close()` only updates Card Kit content/settings, so the maintainable fix likely needs a Feishu-owned notification strategy around when to stream, fall back, or send an additional notification-bearing message.
