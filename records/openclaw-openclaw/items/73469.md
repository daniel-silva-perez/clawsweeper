## Summary

Keep open. Current main still sends Feishu streaming-card replies before the final agent text is known, then applies mention-bearing final text through later Card Kit content/settings updates; the existing non-streaming card/text paths can include structured mention targets on initial send, but the streaming close path cannot currently make agent-authored `<at user_id="...">...</at>` tags notification-bearing.

## What This Changes

The issue requests Feishu streaming-card replies to trigger push notifications when the agent includes Feishu `<at>` mention tags in generated text.

## Best Possible Solution

Keep the fix in the Feishu plugin: detect agent-authored Feishu mention tags before final delivery, preserve existing `mentionTargets` behavior, and route mention-bearing replies through a notification-capable initial send or a documented Feishu-specific fallback when Card Kit streaming updates cannot notify.

## Reproduction Assessment

Yes. Configure Feishu with streaming cards enabled, have the agent return text containing `<at user_id="ou_xxx">Name</at>`, and verify the current OpenClaw path sends the initial card reference before applying that text through later Card Kit updates; a live Feishu tenant check should confirm the missing push notification externally.

## Solution Assessment

Unclear. Adding a `mentions` parameter to `streaming.close()` alone does not look sufficient because the current close path maps to Card Kit content/settings updates, and the checked upstream schemas do not expose a close-time mention notification payload.
