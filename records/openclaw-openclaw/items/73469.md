## Summary

Keep open. Current main still starts Feishu Card Kit streaming before final agent text is known, sends only a card reference initially, and writes mention-bearing text through later Card Kit content/close updates. The related #60103 syntax-normalization work overlaps but does not supersede this notification-timing problem.

## What This Changes

Request that Feishu streaming-card replies send push notifications when agent-authored `<at user_id="...">...</at>` mentions appear in the final response.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves mention notification semantics for agent-authored tags. The end state should cover tag parsing, existing `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and a documented fallback when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming cards (`renderMode: "card"` or markdown-triggered `auto`), have the agent return text containing `<at user_id="...">...</at>`, and inspect that OpenClaw sends the initial card reference before applying the mention text through later Card Kit updates; a live tenant push-notification check would complete external verification.

## Solution Assessment

Unclear. Adding a `mentions` parameter to `close()` alone is unlikely to be sufficient because the current close path maps to Card Kit content/settings updates, and the checked upstream schemas do not show a close-time mention notification payload. ([raw.githubusercontent.com](https://raw.githubusercontent.com/larksuite/lark-openapi-mcp/main/src/mcp-tool/tools/en/gen-tools/zod/cardkit_v1.ts))
