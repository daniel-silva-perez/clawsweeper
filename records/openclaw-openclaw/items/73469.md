## Summary

Keep open. Current main still sends Feishu streaming cards before the final agent text is known, then writes the mention-bearing response through later Card Kit updates/close; agent-authored `<at user_id="...">...</at>` text is not turned into notification-capable initial-send mention data. The open related #60103 work overlaps on syntax normalization, but it does not supersede this streaming-card notification timing problem.

## What This Changes

The issue requests that Feishu streaming-card replies notify mentioned users when an agent response includes `<at user_id="...">name</at>` tags.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves mention notification semantics for agent-authored tags while making any streaming tradeoff explicit. The end state should cover agent-authored tags, existing inbound `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and a documented fallback when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes for the OpenClaw delivery path: configure Feishu streaming with `renderMode: "card"` or markdown-triggered `auto`, return text containing `<at user_id="...">...</at>`, and inspect that the initial card send contains only the card reference while the final text is applied later. A live Feishu push-notification check would still be useful to confirm the external API behavior.

## Solution Assessment

Unclear. The fix should stay in the Feishu plugin, but the narrowest maintainable implementation needs product/maintainer choice because Feishu may not support re-notifying from Card Kit updates; a mention-aware non-streaming fallback or secondary notification pattern is safer than changing core first.
