## Summary

Keep open. Current main still sends Feishu streaming-card mention text after the initial Card Kit/card-reference delivery point, and the open related syntax-normalization issue does not cover the notification timing problem. The report is recent, concrete, and Feishu-specific, but the correct behavior needs maintainer product judgment because preserving streaming and preserving mention push notifications appear to conflict under the checked Card Kit flow.

## What This Changes

Feishu streaming cards should trigger user push notifications when an agent-authored response contains `<at user_id="...">...</at>` mention tags.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves mention notification semantics for agent-authored tags. The end state should cover tag parsing, existing `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and a documented fallback when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu streaming cards with `renderMode: "card"` or markdown-triggered `auto`, have the agent return `<at user_id="...">...</at>`, and verify from current code that the initial send contains only a card reference while the mention text is applied by later Card Kit updates; a live tenant push-notification check would complete external verification.

## Solution Assessment

Unclear. Adding a `mentions` parameter to `close()` alone does not look sufficient because current `close()` maps to Card Kit content/settings updates rather than a new initial Feishu message, so the maintainable fix likely needs a Feishu-owned send strategy around mentions and streaming mode.
