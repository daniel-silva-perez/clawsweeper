## Summary

Keep open. Current main still has the diagnostic gap, and this PR is useful directionally, but the proposed metadata-only change likely does not fix the default operator console logs described by the linked bug because compact/pretty console rendering does not show structured metadata.

## What This Changes

The PR changes `triggerInternalHook` to pass an `Error.stack` value and `{ type, action, sessionKey }` event metadata to `log.error` when an internal hook handler throws.

## Best Possible Solution

Revise the PR so hook failures keep the structured stack/event metadata for file and JSON logs while also making the stack visible on the intended operator console surface without logging full `event.context` or `messages`; add a focused regression test for the thrown-hook path and its expected log output.

## Reproduction Assessment

Yes. A source-level reproduction is to register an internal hook handler that throws and call `triggerInternalHook`; current main reaches the catch block that logs only the formatted message, and the PR's added metadata would still be hidden in default compact console output.

## Solution Assessment

No. Structured metadata is a useful part of the fix, but by itself it is not the narrowest complete solution for the reported Docker/operator log problem; using the existing `consoleMessage` path or otherwise including the stack in the emitted console message is the safer direction.
