## Summary

Keep open. Current main still lacks a language-aware TTS routing contract: `auto: "inbound"` only decides whether to synthesize audio, while provider/model/voice selection remains driven by preferences, persona, config, and directive overrides, with no preserved STT language metadata available to route from.

## What This Changes

Add optional language-aware TTS routing so inbound or reply language can select the TTS provider, model, and voice while preserving current auto-TTS and per-agent override behavior.

## Best Possible Solution

Design a backward-compatible `messages.tts.languageRouting` contract, carry optional detected language metadata through media understanding into reply/TTS context, apply route selection before speech synthesis, and document/test fallback behavior across global, per-agent, channel, account, preference, persona, and directive layers.

## Reproduction Assessment

Yes for the current limitation: source and docs show `auto: "inbound"` only gates audio synthesis and no input/reply language is passed into provider routing. No live STT/TTS credentials are needed to verify the missing routing path.

## Solution Assessment

Unclear for the exact proposed JSON shape. The direction is sound, but the maintainable fix needs maintainer API/product review because it changes public config, media metadata, routing precedence, and plugin-facing behavior.
