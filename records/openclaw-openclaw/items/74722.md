## Summary

Keep open. Current main does not implement language-aware TTS routing: the public TTS config has no language-routing field, `auto: "inbound"` only decides whether to synthesize audio, provider selection still comes from prefs/persona/config, and audio transcription currently preserves transcript text rather than detected language metadata. The related TTS reports are delivery/provider-specific issues, not a canonical tracker for this requested routing feature.

## What This Changes

Add optional language-aware TTS provider/model/voice routing so inbound voice replies can synthesize with a route matching the detected input or reply language.

## Best Possible Solution

Design a backward-compatible `messages.tts.languageRouting` contract in the speech-core/public config surface, preserve optional STT language metadata through media understanding and inbound context, apply the selected route before provider synthesis, and document/test fallback behavior across global, per-agent, channel, and account overrides.
