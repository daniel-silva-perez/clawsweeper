## Summary

Keep open. Current main still lacks language-aware TTS routing: the public TTS config/schema has no routing block, inbound auto-TTS only gates whether synthesis happens, provider/model/voice selection is still based on preferences/persona/config/directives, and STT preflight does not preserve detected language metadata for routing.

## What This Changes

The issue requests language-aware TTS provider, model, and voice routing based on detected input or reply language for inbound voice replies.

## Best Possible Solution

Design a backward-compatible `messages.tts.languageRouting` contract, preserve optional detected language metadata from STT/media understanding, apply route selection before speech synthesis, and document/test fallback behavior across global, per-agent, channel, account, preference, persona, and directive overrides.

## Reproduction Assessment

Yes for the current limitation: source and docs show `auto: "inbound"` only gates audio synthesis and no input/reply language is available to provider routing. No live TTS or STT credentials are needed to verify the missing routing path.

## Solution Assessment

Unclear for the exact proposed JSON shape. The direction is sound, but the best fix needs maintainer API review because a one-off provider-specific or prompt-only workaround would not solve the shared TTS/STT routing contract.
