## Summary

Keep open. Current main still lacks language-aware TTS routing: the TTS config/schema has no route map, inbound auto-TTS only gates whether audio is synthesized, provider selection remains prefs/persona/config/directive based, and STT preflight does not preserve detected language metadata for routing.

## What This Changes

The issue requests language-aware TTS provider/model/voice routing for inbound voice replies so synthesized replies can follow the detected input or reply language.

## Best Possible Solution

Design a backward-compatible `messages.tts.languageRouting` contract, preserve optional detected language metadata from STT/media understanding, apply route selection before speech provider synthesis, and document/test fallback behavior across global, per-agent, channel, and account TTS overrides.

## Reproduction Assessment

Yes for the current limitation: source and tests show `auto: "inbound"` only gates audio synthesis and provider selection has no input/reply language input. No live TTS/STT credentials are needed to reproduce that missing routing path.

## Solution Assessment

Unclear for the exact proposed JSON shape. The direction is reasonable, but the best fix should be settled through maintainer API review because it crosses public config, provider routing, STT metadata, docs, and plugin-facing contracts.
