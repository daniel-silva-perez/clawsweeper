## Summary

Keep open. Current main still appears to finalize the isolated cron run when the first yielded agent turn resolves: `sessions_yield` is surfaced as paused/yielded agent metadata, but cron has only terminal `ok/error/skipped` statuses, forwards the first isolated result as `ok`, clears `runningAtMs`, computes duration from that first promise, and appends a `finished` run-log row. The related closed reports addressed survival/resume behavior; this monitoring/duration variant remains. A same-title duplicate (#73816) is also open, but this earlier item is still a valid canonical tracker.

## Best Possible Solution

Keep this open and fix the cron ownership model for yielded isolated runs. Cron should either carry an explicit yielded/pending state or keep the run/task owned as running while descendant subagents and the resumed parent are still responsible for completion. Only emit the final `finished` event and persist run-log `durationMs` after the resumed parent reaches a real terminal outcome. Add a regression test for an isolated cron job that spawns a subagent, calls `sessions_yield`, resumes after child completion, and verifies cron state/log duration covers the full orchestration. Coordinate with #73816 as a duplicate report once one canonical fix lands.
