## Summary

Keep open. Current main still has the same cron-side gap: the embedded runner marks yielded attempts as paused, but isolated cron finalization still converts the first yielded run result into terminal `ok`, and the timer clears `runningAtMs`/computes `durationMs` when that promise resolves.

## What This Changes

The issue asks isolated cron jobs using `sessions_yield` to report status and duration for the full subagent plus resumed-parent orchestration, not only the parent turn before yield.

## Best Possible Solution

Make yielded isolated cron execution explicit in the cron lifecycle. The complete end state is that cron does not emit `finished` or compute final duration until descendant subagents and the resumed parent reach a real terminal outcome; if maintainers choose an incremental path, it should clearly expose yielded/nonterminal metadata while tracking the remaining duration gap.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current source inspection shows the same mechanism: runner metadata says yielded/paused, but cron finalization still maps the first resolved run to terminal `ok` and immediately computes duration.

## Solution Assessment

Unclear until maintainers choose the lifecycle direction. Deferring `finished` is the complete fix for status and duration; an additive yielded field is a smaller compatibility step but only a partial observability fix.
