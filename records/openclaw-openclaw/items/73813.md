## Summary

Keep open. Current main still treats the first isolated cron agent turn as terminal when `sessions_yield` ends the parent turn, so cron state and run logs can be finalized before the descendant subagent and resumed parent work finish. The related closed reports addressed survival/reactivation failures, but this observability/duration variant is not covered by current cron state handling.

## Best Possible Solution

Keep this open and fix the cron ownership model for yielded isolated runs. Plumb yielded/paused run metadata into the cron outcome contract, keep the cron job/task in a running or explicitly pending-yield state while descendant subagents and the resumed parent are still responsible for completion, then emit the final `finished` event and run-log duration only after the resumed parent reaches a real terminal outcome. Add a regression test for an isolated cron job that spawns a subagent, calls `sessions_yield`, resumes after child completion, and verifies cron `finished`/`durationMs` reflect the full orchestration.
