## Summary

Keep open. Current main still treats the first yielded isolated cron parent turn as a terminal successful cron outcome for bookkeeping, so the reported misleading `status: "ok"` and short `durationMs` remain plausible even though the embedded runner now marks yielded runs as paused metadata.

## What This Changes

The issue requests cron run status and duration to reflect the full isolated `sessions_yield` plus subagent orchestration instead of finalizing when the parent first yields.

## Best Possible Solution

Cron should treat yielded isolated agent turns as nonterminal for run bookkeeping: keep the cron task/run active while descendants and the resumed parent still own completion, then emit `finished` and compute duration from the true terminal parent outcome. If maintainers want a smaller first step, propagate explicit yielded metadata while keeping the remaining duration gap tracked.

## Reproduction Assessment

Yes. The issue provides a concrete isolated cron `agentTurn` scenario with `sessions_spawn`, `sessions_yield`, and a long-running child, and current source inspection shows cron finalization still follows the first yielded embedded result rather than a resumed terminal orchestration result.

## Solution Assessment

No. The current implementation is not the best way to solve the issue because it records `ok` without using yielded metadata; the narrowest maintainable direction is either an explicit yielded/pending cron outcome as an interim contract or, preferably, deferred cron finalization through the full resumed parent lifecycle.
