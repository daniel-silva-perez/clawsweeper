## Summary

Keep open. Current main still treats the first yielded isolated cron parent turn as a terminal successful cron outcome for bookkeeping: the embedded runner marks `sessions_yield` as paused/yielded metadata, but cron finalization does not consume that metadata and the timer emits `finished` with the pre-yield duration.

## What This Changes

The issue asks cron run history and monitoring to avoid reporting an isolated `sessions_yield` subagent workflow as `status: "ok"` with a short duration before the child and resumed parent actually finish.

## Best Possible Solution

Cron run bookkeeping should model yielded isolated agent turns as nonterminal until the resumed parent reaches a true terminal outcome, so `finished` status and `durationMs` reflect the whole orchestration. If maintainers want an incremental step first, add explicit yielded/pending metadata while keeping the full lifecycle fix tracked.

## Reproduction Assessment

Yes. The issue gives a concrete isolated cron `agentTurn` with `sessions_spawn`, `sessions_yield`, and a long-running child; current source inspection shows that yielded embedded metadata still flows into cron finalization as ordinary `ok`.

## Solution Assessment

Unclear until maintainers choose the cron contract. Deferred finalization best matches the reported expected behavior, while adding `yielded?: true` to `CronRunOutcome` would be a narrower interim observability improvement but would not make duration accurate.
