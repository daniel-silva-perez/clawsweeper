## Summary

Keep open. Current main still constructs exec foreground, background/process, node-host, and Codex dynamic-tool results from raw stdout/stderr-derived strings before an exec-owned secret-redaction boundary; newer downstream event/UI sanitization is useful but does not fully answer the reported tool-result/agent-context exposure. The related open PR #75033 is adjacent HTTP/logging redaction work and does not supersede this exec-output issue.

## What This Changes

The issue reports that exec stdout/stderr can be returned to agent/tool-result context without redacting secret-bearing command output.

## Best Possible Solution

Add an exec-owned output-redaction seam that sanitizes stdout/stderr-derived text and details before foreground results, process poll/log results, node-host results, Codex dynamic-tool responses, and approval follow-up summaries are returned, while preserving existing caps, status metadata, and a deliberate maintainer-reviewed escape hatch if verbatim output remains a supported mode.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints an env-style secret flows into outcome/session aggregated output and then into exec result text/details without an exec-boundary redaction call. I did not run a live secret-printing command because this review is read-only and should not create secret output.

## Solution Assessment

Yes. The issue’s proposed primary direction, redacting stdout/stderr before exec result construction and at shared process/node-host boundaries, is the narrowest maintainable fix; blocking specific secret-file reads could supplement it but would not replace output redaction.
