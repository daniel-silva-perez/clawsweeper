## Summary

Keep open. Current main has partial defense-in-depth redaction for PI event projection, transcript persistence, and provider replay, but the exec-owned output constructors still preserve stdout/stderr-derived text in foreground results, runtime outcomes, background process results, node-host formatting, and approval follow-up summaries before a dedicated exec-boundary redaction pass. The related open PR #75033 is about HTTP/logging redaction and does not supersede this exec output issue.

## What This Changes

Request to redact or block secret-bearing stdout/stderr from exec before tool results and related process outputs are delivered to agent/runtime surfaces.

## Best Possible Solution

Keep the issue open and patch the exec-owned output boundary. Add a shared exec output redaction helper that applies the configured tool-payload redaction to foreground content/details, runtime outcome aggregated/reason strings, process poll/log content, node-host formatted results, and notify/follow-up summaries, while preserving existing caps, truncation, binary-output cleanup, and provider-replay details stripping.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: any command that prints an env-style secret can flow through session.aggregated into buildExecExitOutcome and then buildExecForegroundResult or process/node-host output without an exec-boundary redact call; I did not run a live secret-printing command because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to redact stdout/stderr-derived text before exec result construction and at the shared process/node-host output boundaries; hard-blocking specific file reads could be an additional policy layer, but it would not replace output redaction.
