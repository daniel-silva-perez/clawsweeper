## Summary

Keep open. Current main still constructs exec foreground, background/process, node-host, and approval-follow-up outputs from raw stdout/stderr-derived strings before an exec-owned redaction pass; downstream event/UI redaction is useful but does not close the tool-result boundary described by the report. The related open PR #75033 is adjacent HTTP/logging redaction work and does not supersede this exec output issue.

## What This Changes

The issue requests redaction or blocking of secret-bearing stdout/stderr before exec tool results can put those values into agent/tool context.

## Best Possible Solution

Patch the exec-owned output boundary with a shared helper that applies tool-payload redaction to stdout/stderr-derived foreground results, runtime outcomes, process poll/log responses, node-host formatted results, and approval/notify summaries, while preserving existing caps, truncation behavior, and downstream event/UI sanitization.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints an env-style secret flows through `session.aggregated` into foreground `buildExecForegroundResult`, process poll/log results, or node-host formatting without an exec-boundary redaction call. I did not run a live secret-printing command because this review is read-only.

## Solution Assessment

Yes. The issue’s proposed primary direction, redacting stdout/stderr before exec result construction and at shared process/node-host boundaries, is the narrowest maintainable fix; blocking specific secret-file reads could supplement it but would not replace output redaction.
