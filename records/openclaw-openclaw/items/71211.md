## Summary

Keep open. Current main has downstream transcript/UI/event redaction, but the exec-owned foreground, background/process, and node-host result builders still assemble tool result text/details from raw stdout/stderr-derived strings before those later sanitizers run. The related open PRs are adjacent logging/tool-payload redaction work, not merged fixes for this exec result boundary.

## What This Changes

The issue asks OpenClaw to redact or otherwise guard secret-bearing stdout/stderr before exec tool results are returned to the agent/runtime.

## Best Possible Solution

Keep this issue open and add a shared exec/tool-output redaction boundary used by foreground exec result construction, runtime outcomes, process poll/log output, node-host formatting, and approval follow-up summaries, with focused tests for env-style assignments and known token formats while keeping downstream UI/transcript redaction as defense in depth.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints `TOKEN=...` can flow through `session.aggregated` into foreground, background/process, or node-host tool result content/details without an exec-owned redaction call. I did not run a live secret-printing command because this review is read-only.

## Solution Assessment

Yes. Redacting stdout/stderr at the exec-owned result boundary is the narrowest maintainable fix; blocking specific secret-file reads can supplement it, but cannot replace output redaction across all command sources.
