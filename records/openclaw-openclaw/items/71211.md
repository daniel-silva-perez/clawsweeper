## Summary

Keep open. Current main still constructs foreground, background/process, and node-host exec tool results from raw stdout/stderr-derived strings before the exec-owned boundary applies any redaction; the existing downstream UI/event sanitizer is useful defense in depth but does not close the raw tool-result path. The related PRs #75033 and #75230 are open and adjacent logging/tool-payload redaction work, not merged fixes for this exec boundary.

## What This Changes

The issue asks OpenClaw to redact or otherwise guard secret-bearing stdout/stderr before exec tool results are returned to the agent.

## Best Possible Solution

Add a shared exec/tool-payload redaction helper at the exec-owned output boundary: foreground result construction, runtime outcomes, process poll/log responses, node-host formatting, and approval follow-up summaries. Cover env-style assignments and known token formats in focused tests while keeping downstream UI/event redaction as defense in depth.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints `TOKEN=...` flows through `session.aggregated` into foreground, background/process, or node-host tool result content/details without an exec-boundary redaction call. I did not run a live secret-printing command because this review is read-only.

## Solution Assessment

Yes. The issue's primary direction, redacting stdout/stderr before exec result construction and at shared process/node-host boundaries, is the narrowest maintainable fix; blocking specific secret-file reads could supplement it but would not replace output redaction.
