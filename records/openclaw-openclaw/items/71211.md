## Summary

Keep this open. Current main still constructs foreground, background/process, and node-host exec tool results from raw stdout/stderr-derived strings before an exec-owned redaction boundary; the existing downstream event/UI sanitizer is useful defense in depth but does not close the raw tool-result path.

## What This Changes

The issue asks OpenClaw to redact or otherwise protect secret-bearing stdout/stderr before exec tool results are returned to agent context.

## Best Possible Solution

Add a shared exec/tool-result redaction boundary that sanitizes stdout/stderr-derived text and structured details before foreground exec, background process poll/log, node-host exec, and approval follow-up summaries reach agent context, while keeping downstream UI/event sanitization as defense in depth.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints an env-style secret such as `TOKEN=...` flows through raw aggregate fields into foreground, background/process, or node-host tool result content/details before an exec-owned redaction pass.

## Solution Assessment

Yes. The issue's direction, redacting output before tool result construction and covering shared process/node-host paths, is the narrowest maintainable fix. Blocking specific secret-file reads could supplement it but would not close generic secret-bearing stdout/stderr.
