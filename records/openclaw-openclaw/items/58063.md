## Summary

Keep open. Current main still has a credible Grok/xAI web_search abort path: the bundled xAI provider uses the shared 30 second timeout, sends it into the Responses API fetch path, and the agent tool adapter still rethrows AbortError instead of returning a structured tool error. The linked #58733 gives a focused current-main reproduction for the same provider path, while related #43609 has useful source context but is not directly mergeable as-is.

## What This Changes

The issue reports that the Grok/xAI-backed managed web_search tool aborts immediately after the 2026.3.28 Responses API path change.

## Best Possible Solution

Ship a narrow provider-owned xAI web_search fix: give Grok a longer default timeout only when the user has not configured one, convert internal xAI fetch timeout AbortErrors into ordinary structured tool errors, and harden the xAI Responses parser against null output/content/annotation entries. Keep the change inside the xAI plugin/runtime unless a shared web-search contract gap is proven.

## Reproduction Assessment

Unclear for the exact immediate-abort claim because this review could not run a live xAI request. Yes for the confirmed related path: #58733 gives a current-main Grok requestXaiWebSearch reproduction that aborts at the 30 second boundary and succeeds when only the timeout is raised to 60 seconds.

## Solution Assessment

No current-main solution is present yet. The best path is a narrow replacement patch based on the useful parts of #43609, ported to the live xAI runtime files and without unrelated status-command changes.
