## Summary

Current main still matches the actionable parts of the report: Grok-backed web_search still uses the shared 30 second timeout on the xAI Responses path, the request path forwards that timeout into the guarded fetch layer, and AbortError-shaped failures are still rethrown by the tool adapter. The linked #58733 is useful narrower reproduction evidence, but it does not fully supersede this issue because it explicitly covers the 30 second timeout case rather than the original immediate-abort symptom.

## What This Changes

The issue reports that Grok-backed web_search returns `This operation was aborted` after the xAI Responses migration despite a configured xAI key.

## Best Possible Solution

Keep this issue open while maintainers review #43609 or an equivalent focused replacement. The durable fix should live in the bundled xAI/Grok web_search runtime: preserve explicit timeout overrides, use a provider-appropriate default for Grok when unset, translate internal timeout AbortErrors into normal tool errors with useful diagnostics, and add focused xAI Responses regression coverage.

## Reproduction Assessment

Unclear for the full issue. #58733 gives a high-confidence reproduction for a related timeout-shaped Grok/xAI Responses abort, but this issue's original sub-millisecond abort still lacks enough trace data to prove it is the same failure.

## Solution Assessment

No patch is proposed on this issue. The best current path is a narrow current-main fix in the xAI plugin runtime, not closing this as stale or duplicate while the immediate-abort symptom remains only partially explained.
