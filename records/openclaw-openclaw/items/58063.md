## Summary

Keep open. Current main still has the credible Grok/xAI web_search abort path: the xAI provider uses the shared 30 second timeout, sends it to the Responses API fetch helper, and internal AbortError failures still bypass structured tool-error handling. The related #58733 report gives a focused current-main timeout-boundary reproduction, while #43609 is useful source context but targets a stale provider path and should not be merged as-is.

## What This Changes

The issue reports that the Grok/xAI-backed managed web_search tool aborts after the 2026.3.28 xAI Responses API migration instead of returning synthesized search results.

## Best Possible Solution

Ship a narrow xAI plugin fix that preserves explicit user timeoutSeconds, uses a Grok-specific longer default only when unset, converts internal xAI fetch timeout AbortErrors into structured web_search tool errors, and hardens the xAI Responses parser against null/non-object output, content, and annotation entries.

## Reproduction Assessment

Unclear for the exact immediate-abort claim because this read-only review could not run a live xAI request. Yes for the confirmed related path: #58733 gives a current-main requestXaiWebSearch reproduction that aborts at the 30 second boundary and succeeds when only the timeout is raised to 60 seconds.

## Solution Assessment

No current-main solution is present yet. The best fix is a narrow replacement patch against the live xAI runtime, using #43609 only as source context and leaving unrelated status-command changes out.
