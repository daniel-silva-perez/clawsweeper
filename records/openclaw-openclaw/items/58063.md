## Summary

Keep open. Current main still has credible Grok/xAI `web_search` abort paths: the bundled provider keeps the shared 30 second timeout, passes it into the xAI Responses fetch path, lacks null-entry parser guards, and the agent tool adapter still treats internal `AbortError` as a cancellation rather than a structured tool failure.

## What This Changes

The issue reports that `web_search` with `tools.web.search.provider: "grok"` aborts with `This operation was aborted` after the xAI Responses API migration.

## Best Possible Solution

Ship a narrow xAI plugin fix: use a longer Grok default timeout only when users have not configured one, convert provider-internal xAI timeout `AbortError`s into ordinary structured tool failures, and harden the xAI Responses parser against null or malformed output/content/annotation entries. Keep this provider-owned unless a separate generic web-search abort contract gap is proven.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the exact immediate-abort symptom because this review did not run a live xAI request, but yes for the confirmed related path: #58733 reports a current-main Grok `requestXaiWebSearch` abort at the 30 second boundary that succeeds when only the timeout is raised to 60 seconds.

## Solution Assessment

Is this the best way to solve the issue? No current-main fix is present. The best path is a narrow xAI plugin patch, using #43609 as source context but avoiding its obsolete/unrelated changes.
