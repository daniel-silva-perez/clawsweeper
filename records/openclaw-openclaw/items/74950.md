## What This Changes

The PR adds a true-legacy mtime fallback to bundled runtime-deps lock expiry and regression tests for stale/fresh no-starttime/no-createdAt locks plus createdAt-bearing live locks.

## Best Possible Solution

Land the narrow true-legacy fallback after maintainer review and CI if the intended scope is no-starttime/no-createdAt lock recovery. Keep createdAt-bearing stale Docker lock recovery as a separate follow-up unless maintainers choose a safe cross-platform disambiguation rule.

## Reproduction Assessment

Yes. The focused reproduction is a direct call to `shouldRemoveRuntimeDepsLock()` with a live PID, no `starttime`, no `createdAtMs`, and a stale lock mtime; current main returns held, while the PR adds tests expecting stale true-legacy locks to expire and fresh ones to stay held.

## Solution Assessment

Yes for the true-legacy shape. The PR is the narrowest maintainable change I found because it only uses mtime when both stronger owner fields are absent and preserves the existing safety invariant for createdAt-bearing live locks.
