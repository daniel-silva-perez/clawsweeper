## Summary

Keep open. Current main still holds true-legacy PID-alive runtime-deps locks that lack both starttime and createdAtMs, while this PR adds a narrow fallback plus focused tests and I did not find a blocking correctness issue in the patch.

## What This Changes

The PR adds a true-legacy mtime fallback to bundled runtime-deps lock expiry and regression tests that keep createdAt-bearing live-PID locks held.

## Best Possible Solution

Land the narrow true-legacy fallback after maintainer review and the targeted plugin runtime-deps test lane pass; keep any broader createdAt-bearing stale-lock recovery separate so active non-Linux installs are not expired by age alone.

## Reproduction Assessment

Yes. The focused reproduction is a direct shouldRemoveRuntimeDepsLock call with an alive PID, no starttime, no createdAtMs, and a lockDirMtimeMs older than 10 minutes; current main returns held and the PR's new test expects it to expire.

## Solution Assessment

Yes for the true no-starttime/no-createdAt legacy shape. The PR is the narrowest maintainable change because it reuses existing mtime and stale-threshold logic while preserving createdAt-bearing live locks.
