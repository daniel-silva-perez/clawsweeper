## Summary

Keep open. Current main still has the live-PID path that returns before any true-legacy mtime fallback, and the PR is an active focused fix for that remaining edge case rather than obsolete work. The code direction looks narrow for no-starttime/no-createdAt legacy locks, but the PR still needs maintainer review, validation, and a required changelog entry before merge.

## What This Changes

The PR adds a true-legacy mtime fallback to `shouldRemoveRuntimeDepsLock()` for live-PID lock owners missing both `starttime` and `createdAtMs`, with regression tests for stale legacy locks and createdAt-bearing locks.

## Best Possible Solution

Land the narrow true-legacy fallback with the missing changelog entry once targeted tests, changed-gate validation, and maintainer scope review are clean. Keep createdAt-bearing live-lock recovery out of this PR unless maintainers choose a separate safe discriminator for active non-Linux installs.

## Reproduction Assessment

Yes. A direct helper call on current main with `isAlive(pid) === true`, no `starttime`, no `createdAtMs`, and a stale `lockDirMtimeMs` reproduces the reported hold because the function returns `false` before mtime fallback logic.

## Solution Assessment

Yes for the true legacy lock shape, with one process issue to fix before merge. The PR’s fallback is gated to records missing both fields, preserving the current invariant that createdAt-bearing live-PID locks are not expired by age alone.
