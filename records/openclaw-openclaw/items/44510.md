## Summary

Keep open. Current main still lacks the lock-loss notification/write guard, so this PR targets a real session-corruption gap, but the latest branch should not merge as-is because its guard remains installed during teardown flushes and can skip cleanup after lock loss.

## What This Changes

The PR adds session-lock force-release callbacks and an `isSessionLockHeld` helper, installs a SessionManager write guard in embedded attempts to reject writes after lock loss and fsync session writes, and adds unit coverage.

## Best Possible Solution

Integrate lock-loss cancellation with the current attempt cleanup architecture: watchdog lock loss should abort the run/session, persistence should fail closed after ownership loss, and teardown should always dispose the session, release websocket/runtime resources, and release the lock even if final pending-tool-result flushing fails.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction for the PR defect is to lose the lock before teardown, then have `flushPendingToolResultsAfterIdle()` call SessionManager persistence while the write guard is still installed; the guard throws before the cleanup steps below it run.

## Solution Assessment

No as-is. The fix direction is plausible, and the earlier abort-controller review feedback was addressed, but the guard must be integrated with the current best-effort cleanup helper or disposed before final flushes before this can safely merge.
