## Summary

Keep this issue open. Current main still sends the built-in `profile="user"` existing-session browser path through Chrome DevTools MCP over the upstream stdio transport, and the maintained discussion explicitly says the Termux stdin parent-death failure is separate from the merged Chrome MCP command/args work.

## Best Possible Solution

Keep this issue open and fix the browser Chrome MCP lifecycle path: use an OpenClaw-owned stdio/process manager that preserves the child stdin contract on Android/Termux, or add a supported non-stdio transport for existing-session attach. Add a focused regression test that simulates Chrome MCP exiting after stdin end/parent-death and verifies `profile="user"` status/actions report, recover from, or avoid that failure.
