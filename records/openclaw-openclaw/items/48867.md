## Summary

Keep this issue open. Current main still routes the built-in `profile="user"` existing-session browser path through Chrome MCP over the upstream stdio transport, and the recent maintainer comment explicitly says the Termux stdin parent-death failure is separate from the merged #71560 command/args work.

## Best Possible Solution

Keep this issue open and fix it in the browser Chrome MCP lifecycle path: either route Chrome MCP through an OpenClaw-owned stdio/process manager that preserves the child stdin contract on Android/Termux, or add a transport alternative for existing-session attach. Add a focused regression test that simulates the Chrome MCP subprocess exiting after stdin end/parent-death and verifies `profile="user"` reports, recovers, or avoids that failure.
