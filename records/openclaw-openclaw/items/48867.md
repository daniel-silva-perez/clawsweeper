## Summary

Keep this issue open. Current main still routes the built-in `profile="user"` existing-session path through Chrome DevTools MCP over the upstream stdio transport, and the maintainer discussion explicitly says the Termux stdin parent-death failure is separate from the already-merged Chrome MCP command/args work in #71560.

## Best Possible Solution

Keep this issue open and fix the browser Chrome MCP lifecycle path: either route Chrome MCP through an OpenClaw-owned stdio/process transport that preserves child stdin correctly on Android/Termux, or add a supported non-stdio transport for existing-session attach. Add a focused regression test that simulates stdin EOF/parent-death during `profile="user"` status/actions and verifies the session is avoided, recovered, or reported with a useful error.
