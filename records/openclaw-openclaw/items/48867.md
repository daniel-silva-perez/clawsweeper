## Summary

Keep this issue open. Current main still routes the built-in `user` / `existing-session` browser profile through Chrome DevTools MCP using the upstream stdio transport directly, and the issue discussion already has MEMBER triage saying this Android/Termux stdin parent-death failure is separate from the merged Chrome MCP command/args work in PR #71560.

## Best Possible Solution

Keep this issue open and fix the browser Chrome MCP lifecycle path directly. The most concrete direction is to route Chrome MCP through an OpenClaw-owned stdio/process transport that preserves child stdin correctly on Android/Termux, or add a supported non-stdio attach transport for existing-session profiles. Add a focused regression test that simulates stdin EOF or parent-death during `profile="user"` status/actions and verifies recovery or a clear actionable error, then validate on a real Termux setup if possible.
