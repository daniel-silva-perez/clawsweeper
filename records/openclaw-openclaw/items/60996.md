## Summary

Keep open. Current main does not implement the PR's intended exec changes: `argPattern` remains Windows-only, `ExecAsk` still excludes `auto`, the gateway approval flow has no classifier path, and the public exec approvals docs do not document subcommand convenience syntax. The PR is also not safe to close as obsolete or merge as-is because its own review discussion identifies broken subcommand expansion and security-sensitive `ask="auto"` approval gaps.

## Best Possible Solution

Do not close as implemented. Rework this into a narrower, tested exec change: first enable cross-platform `argPattern` with explicit compatibility handling for argv-missing call paths and Linux/macOS/Windows tests. If `ask="auto"` remains desired, split it into a separate security-reviewed design that keeps unknown commands on a prompt path, maps prompt decisions to an actual approval flow in all security modes, updates every policy normalizer/resolver, and documents the final contract. The convenience `git add:*` syntax should either be fixed against the existing executable-plus-argv matching model or removed until it has reliable tests.

## Risks / Open Questions

- This PR touches the host exec approval path, so incorrect `ask="auto"` behavior could silently run commands that the feature says should prompt.
- The proposed classifier defaults unknown commands to safe, which is an unsafe default for code execution approval policy.
- The proposed subcommand convenience syntax is broader than the existing allowlist contract and needs precise argv/path semantics plus tests before it can be trusted.
- No local tests were run because this review was intentionally read-only and the user prohibited commands that may create artifacts.
