## Summary

Keep open. Current main does not implement the requested all-platform subcommand argPattern or ask="auto" behavior, and the PR branch is not merge-ready because it introduces blocking exec-approval correctness and security regressions in the new policy mode and convenience pattern expansion.

## What This Changes

The PR enables argPattern matching beyond Windows, adds subcommand-pattern expansion and docs, introduces static/LLM-named exec safety classification, and wires a new ask="auto" mode into gateway exec approvals.

## Best Possible Solution

Split the work: first land a narrow, tested all-platform argPattern change that preserves existing compatibility, then treat ask="auto" as a separate security/product design with fail-closed classification, full policy normalization, UI/docs/schema coverage, and explicit tests for gateway and node approval flows.

## Reproduction Assessment

Yes. The blocking issues are reproducible from source-level checks: the PR's convenience expansion does not match matchArgPattern's argv.slice(1) contract, ask="auto" is not carried through all policy validators/maxAsk, and requiresExecApproval does not prompt for on-miss under security="full".

## Solution Assessment

No. This is not the best way to solve the issue because it combines a useful argPattern portability change with a new security-sensitive auto mode whose default behavior is not fail-closed; a smaller compatibility-preserving argPattern patch is the safer first step.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Carry auto through every ExecAsk policy helper:** `src/infra/exec-approvals.ts:15`
  - body: Adding `auto` to `ExecAsk` here leaves the rest of the policy layer on the old value set: `sanitizeExecApprovalPolicy` and `isExecAsk` still reject it, and `maxAsk` uses `Record<ExecAsk, number>` without an `auto` entry. That means persisted/default host policies are stripped or mishandled, and the typecheck should fail once this union changes.
  - confidence: 0.94
- **[P1] Prompt elevated auto decisions in full security mode:** `src/agents/bash-tools.exec-host-gateway.ts:124-125`
  - body: Mapping the classifier's `prompt` result to `on-miss` does not guarantee a prompt: `requiresExecApproval` only prompts for `on-miss` when `security === "allowlist"` and the allowlist misses. In `security="full"`, an elevated command classified as prompt-worthy can run without confirmation, contradicting the new auto-mode contract.
  - confidence: 0.93
- **[P1] Default unknown commands to prompt instead of safe:** `src/infra/exec-safety-analysis.ts:227-228`
  - body: When no hard-coded safety regex matches, `analyzeArgvSafety` returns `safe`, and `decideApprovalAction` turns that into `allow` for ask="auto". Unrecognized commands such as `bash script.sh` or uncommon admin tools would be trusted by default, so unknowns need to fail closed to a prompt/elevated path.
  - confidence: 0.91
- **[P2] Build subcommand argPattern from arguments only:** `src/infra/exec-allowlist-pattern.ts:128-131`
  - body: For `git add:*`, this generates `^git add\s+.*$`, but allowlist matching tests argPattern against argv after the executable, e.g. `add file.txt`. The documented convenience syntax will silently deny commands users intended to allow until the generated regex drops the executable name.
  - confidence: 0.95
- **[P2] Expand subcommand patterns to the executable pattern:** `src/infra/exec-allowlist-pattern.ts:130-131`
  - body: Returning `pattern: "git add"` makes the allowlist entry target the executable as `git add` instead of the executable `git` plus argument pattern `add ...`. Current matching compares patterns against the resolved executable/basename, so the subcommand convenience entries do not match real commands.
  - confidence: 0.89
- **[P2] Preserve argv-missing compatibility outside Windows:** `src/infra/exec-command-resolution.ts:345-346`
  - body: Changing `useArgPattern` to unconditional means an entry with `argPattern` no longer matches when callers invoke `matchAllowlist` without argv. Existing non-Windows paths previously treated the path match as sufficient, so this can unexpectedly deny allowlisted commands in analysis paths that do not have argv available.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR changes exec approval behavior and introduces concrete fail-open paths in ask="auto" that need security review before merge.

Concerns:

- **[high] Elevated auto decisions can bypass prompts:** `src/agents/bash-tools.exec-host-gateway.ts:125`
  - body: The PR maps classifier prompt decisions to `on-miss`; with `security="full"`, the existing approval policy does not prompt for on-miss, so elevated commands can execute without confirmation.
  - confidence: 0.93
- **[high] Unknown commands are auto-allowed:** `src/infra/exec-safety-analysis.ts:228`
  - body: The static analyzer classifies any command that misses its dangerous/elevated regexes as safe, and auto mode converts safe to allow. That makes the new trust boundary depend on an incomplete denylist instead of a fail-closed classifier.
  - confidence: 0.91
