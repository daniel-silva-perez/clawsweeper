## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested Slack `channel-list` and `search` actions, so the contribution is aimed at a real remaining gap; however, the branch is stale against current Slack runtime/config boundaries and has concrete review blockers around search output formatting, action discovery, and channel-list truncation, with overlapping Slack search work still open in #65691.

## What This Changes

The branch adds Slack `channel-list` and `search` message actions, Slack action helpers and dispatch/advertisement plumbing, Slack action config/schema entries, generated config metadata, and a channel-action contract update.

## Best Possible Solution

Land one coherent Slack plugin implementation: keep `channel-list` in the Slack action surface with pagination or explicit truncation metadata, add Slack `search` using resolved account/config data and the shared `results.messages` output contract, update the current extension-local contract/tests/docs, and reconcile the search work with #65691 before merge.

## Reproduction Assessment

Yes for the PR blockers: the source-level checks reproduce the issues without a live Slack token. A successful Slack search payload from the PR would be returned as top-level `messages`/`total`, while the current formatter only renders `payload.results.messages`, and the ambient `loadConfig()` call is directly visible in the PR head.

## Solution Assessment

No: this branch is not the best mergeable form today because it targets the right feature but misses current runtime boundaries and shared output contracts. The safer path is a rebased Slack-plugin implementation that passes resolved config through the runtime, returns the shared search envelope, and coordinates with the existing Slack search PR.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Pass resolved Slack config into search helpers:** `extensions/slack/src/actions.ts:333-347`
  - body: `resolveUserToken()` calls ambient `loadConfig()` from a Slack action helper when no token override is passed. Current main's runtime-action guard rejects `loadConfig()` in channel action helpers, so the search path should use the already-resolved config/account data passed through `handleSlackAction()`.
  - confidence: 0.94
- **[P2] Return search hits under the results envelope:** `extensions/slack/src/action-runtime.ts:494`
  - body: `handleSlackAction()` returns Slack search data as top-level `messages`/`total`, but the shared CLI formatter renders search rows from `payload.results.messages`. Successful Slack searches will skip the search-results table unless this returns the shared `results` envelope.
  - confidence: 0.9
- **[P2] Advertise search only for the invokable account:** `extensions/slack/src/message-actions.ts:55-63`
  - body: Unscoped discovery advertises `search` if any Slack account has a user token and search enabled, but invocation without `accountId` resolves the default account. A non-default search-capable account can therefore expose an action the default invocation cannot run.
  - confidence: 0.84
- **[P2] Report channel-list truncation to callers:** `extensions/slack/src/actions.ts:302`
  - body: `channel-list` stops at the default 500-channel cap but returns only `channels`. Larger workspaces will be silently truncated, leaving callers unable to tell whether discovery is complete or needs another page/cursor.
  - confidence: 0.82
