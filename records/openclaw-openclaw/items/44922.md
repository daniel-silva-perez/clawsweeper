## Summary

Current main still matches the reported cron-to-heartbeat duplication path, and the focused implementation candidate is still open rather than merged. The issue discussion correctly identified that main-session cron injects the raw system event while heartbeat also wraps the same cron-tagged pending event as a scheduled reminder.

## What This Changes

The issue requests that main-session cron `systemEvent` jobs with `delivery.mode: "none"` inject only the system event and suppress the redundant scheduled-reminder wrapper.

## Best Possible Solution

Land PR #44973 or an equivalent narrow fix that marks main-session cron system-event injections and excludes only those already-injected events from cron reminder wrapper construction, with regression coverage for `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and `delivery.mode: "none"`.

## Reproduction Assessment

Yes. The report provides a concrete cron configuration, and source inspection on current main shows the same text can be both drained as a queued system event and wrapped by the cron heartbeat prompt because of the `cron:` context key.

## Solution Assessment

Yes. The proposed tag-and-skip approach is the narrowest maintainable fix because it deduplicates prompt construction for main-session injected cron events without changing delivery semantics or adding broad outbound dedupe.
