## Summary

Current main still matches the reported cron-to-heartbeat duplication path, so this should stay open. The issue discussion already identified the same root cause, and open PR #44973 appears to target the narrow fix, but it is not merged into the reviewed main checkout.

## What This Changes

Request that main-session cron `systemEvent` jobs with `delivery.mode: "none"` avoid adding the redundant scheduled-reminder heartbeat wrapper when the raw system event is already injected into the session.

## Best Possible Solution

Land PR #44973 or an equivalent narrow fix that marks main-session cron system-event injections and skips the redundant cron reminder wrapper for those events, while keeping existing delivery semantics and adding regression coverage for `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and `delivery.mode: "none"`.

## Reproduction Assessment

Yes. The issue comments provide a concrete cron configuration, and current code inspection shows the same text can be both drained into the prompt as a queued system event and wrapped by `buildCronEventPrompt` because of the `cron:` context key.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix: dedupe main-session injected cron events at heartbeat prompt construction rather than changing `delivery.mode` or adding broad outbound dedupe. It is not implemented on the reviewed main checkout yet.
