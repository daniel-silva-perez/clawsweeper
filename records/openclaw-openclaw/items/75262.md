## Summary

Keep open. The linked problem is valid and this PR is a plausible direction, but the current patch is not mergeable: the new flag is dropped before it reaches the cron ops layer, and the proposed cwd handling mutates process-wide gateway state during an async run.

## What This Changes

The PR branch adds an `--as-scheduled` option to manual cron runs and attempts to plumb it from the CLI through gateway cron handling into isolated cron agent execution.

## Best Possible Solution

Keep the PR open for author revision, but route `asScheduled` through the typed cron service contract/wrapper and implement scheduled semantics at a run-scoped execution boundary, passing explicit cwd/env/context policy into the agent runner rather than changing process-global cwd.

## Reproduction Assessment

Yes. The PR behavior can be reproduced by tracing `openclaw cron run --as-scheduled` through the CLI and `cron.run` handler to `CronService.enqueueRun`, where the new options object is not part of the current contract/wrapper and is not forwarded to `ops.enqueueRun`.

## Solution Assessment

No. The requested direction is reasonable, but this implementation is not the best fix because the flag is ineffective at the gateway service boundary and the cwd change is process-global instead of isolated to the cron run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Forward asScheduled through CronService:** `src/gateway/server-methods/cron.ts:411`
  - body: This new third argument is passed to `context.cron.enqueueRun`, but the concrete `CronService.enqueueRun` wrapper and `CronServiceContract` still accept only `id` and `mode` and forward only those two values to `ops.enqueueRun`. As a result, the gateway path for `cron run --as-scheduled` drops the option before it reaches the implementation, so the flag is a no-op or a typecheck failure. Please update the service contract and wrapper to carry the options through.
  - confidence: 0.93
- **[P2] Avoid mutating process cwd for cron runs:** `src/cron/isolated-agent/run.ts:1050`
  - body: Changing `process.cwd()` around an awaited agent run affects the entire gateway process, not just this cron execution. While this run is awaiting model/tool work, unrelated gateway requests, hooks, or cleanup code that uses relative paths can observe the agent workspace cwd. Keep cwd/env behavior run-scoped by passing it into the executor/tool runtime instead of using `process.chdir`.
  - confidence: 0.88
