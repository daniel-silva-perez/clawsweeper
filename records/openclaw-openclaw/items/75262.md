## Summary

Keep open. The latest PR now plumbs the flag through the cron service boundary, but the implementation still changes `process.cwd()` around an awaited cron agent run, which can leak the agent workspace cwd to unrelated Gateway work. The branch also adds a user-facing CLI/protocol option without the required docs/changelog coverage, and the linked maintainer-labeled issue still needs careful cron semantics review.

## What This Changes

The PR adds an `--as-scheduled` cron-run option and threads `asScheduled` from the CLI through gateway protocol/handlers, cron service APIs, timer execution, and isolated cron agent execution, where it changes the process cwd to the agent workspace.

## Best Possible Solution

Keep the flag direction only if cron scheduled-environment parity is implemented as explicit run-scoped execution policy: carry the option through typed cron contracts, apply cwd/env/context through the existing runner seams, document the new CLI behavior, add the active changelog entry, and cover manual-versus-scheduled parity with focused cron tests.

## Reproduction Assessment

Yes. The review can reproduce the PR issue by tracing the latest diff: `--as-scheduled` reaches `runCronIsolatedAgentTurn`, where the branch calls `process.chdir` before awaiting the runner, while current main already passes `workspaceDir` into the actual CLI and embedded agent runners.

## Solution Assessment

No. The requested direction is reasonable, but mutating process-global cwd is not the best fix; the safer path is to keep scheduled semantics scoped to the cron run through the existing workspaceDir/runner contract and add explicit env/context handling where needed.
