## Summary

Keep open. Current main still lacks the commands-system-prompt WARN logging path, so the PR is not obsolete, but the proposed patch should not merge as-is because it adds a new mocked import without updating the focused Vitest mock.

## What This Changes

The PR imports `makeBootstrapWarn` and a subsystem logger in `commands-system-prompt` and passes a WARN callback into `resolveBootstrapContextForRun` for command-built system prompts.

## Best Possible Solution

Merge a narrow version that wires `makeBootstrapWarn` into the commands system-prompt path and updates the focused test mock or adds a small assertion that the warning callback is forwarded; no broader bootstrap behavior change is needed.

## Reproduction Assessment

Yes for the PR defect: the focused static path is `commands-system-prompt.test.ts`, whose Vitest mock only returns `resolveBootstrapContextForRun`; after the PR's new import/call, that test file needs a mocked `makeBootstrapWarn` export. For the underlying runtime gap, current main source shows the command path omits `warn` while truncation warnings are emitted only through `opts.warn`.

## Solution Assessment

No, not as submitted. Passing `makeBootstrapWarn` is the right maintainable seam and matches existing attempt/CLI callers, but the patch needs the focused test mock and preferably explicit callback coverage before merge.
