## Summary

Keep this issue open. The report is paired with an open, unmerged PR that declares it would close this issue, and current main still only emits a bare future-version warning while downgrade repair for changed value formats remains unresolved.

## What This Changes

The issue asks for clearer downgrade detection and repair guidance when a 2026.4.x-written openclaw.json is read by 2026.3.2 and contains newer keys or value formats.

## Best Possible Solution

Keep the issue open while maintainers review the linked PR. The end state should be a documented downgrade/recovery policy: at minimum, actionable future-version guidance and exact manual fixes, and ideally doctor support for known safe downgrade rewrites without pretending every newer value format can be auto-converted.

## Reproduction Assessment

Yes. The issue gives concrete package-version steps, and current source confirms the key boundary: newer configs are stamped and warned on read, unknown keys can be stripped, but changed value formats such as channel streaming objects are not generically downgraded.

## Solution Assessment

Unclear. The linked PR appears to be a narrow guidance improvement, but it does not fully implement automatic downgrade-aware migration; maintainers should decide whether warning-plus-manual-guidance is acceptable or whether known downgrade rewrites belong in doctor.
