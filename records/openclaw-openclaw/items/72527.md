## Summary

Keep this issue open. Current main has some newer-config guardrails, but source inspection still shows the invalid downgrade-config path can bypass the terse future-version warning, doctor repair currently strips unknown keys only, and the issue is paired with open PR #72535 using closing syntax.

## What This Changes

The issue requests downgrade-aware detection and repair guidance when a newer OpenClaw writes `openclaw.json` fields or value shapes that older OpenClaw versions reject.

## Best Possible Solution

Keep the issue open until PR #72535 is reviewed, expanded, narrowed, or replaced. The end state should surface future-version remediation even when validation fails, preserve valid JSON on repair, and implement only downgrade rewrites maintainers consider safe for existing configs.

## Reproduction Assessment

Yes. The report gives concrete versions and invalid-config messages, and current source inspection confirms the key control-flow gap: invalid config branches can return or throw before the future-version warning path runs.

## Solution Assessment

Unclear. A warning-only PR may help the valid-config path, but the safer fix is to attach newer-config remediation to invalid-config reporting and doctor preflight, then add narrowly scoped repair rules only where downgrade rewriting is safe.
