## Summary

Keep open. Current main still routes Control UI/WebChat new-session actions through `chat.send` with `/new`, the reset command handler returns only the synthetic acknowledgement without an agent run, and the WebChat transcript append path stores that acknowledgement as an assistant-shaped `gateway-injected` row. Related #75235 overlaps on reset readiness, but this report adds a distinct WebChat display/transcript problem that is still actionable.

## What This Changes

Request that fresh WebChat sessions either produce a real startup greeting or render the reset acknowledgement as a clearly non-assistant session-control event.

## Best Possible Solution

Keep the documented no-model reset policy unless maintainers explicitly choose otherwise, but make WebChat distinguish reset acknowledgements from assistant turns, ideally by projecting exact reset acknowledgements as system/session-control events with focused UI and gateway history coverage. Also align any docs or prompt wording that implies a greeting on paths where OpenClaw intentionally does not invoke the model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger the Control UI New Session action or send `/new`, observe that it goes through `chat.send`, the reset command handler returns only `✅ New session started.`, and the WebChat append path can present that as a `gateway-injected` assistant-shaped transcript row.

## Solution Assessment

Yes for a narrow keep-open repair. The safest fix is not to reintroduce an automatic model greeting by default, but to classify the reset acknowledgement as a session-control/system event in WebChat display and history while preserving existing transcript parent-chain and replay-filter behavior.
