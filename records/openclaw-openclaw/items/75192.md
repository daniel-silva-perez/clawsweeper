## Summary

Keep open. Current main still has backend support for agent-scoped session creation, but the Control UI chat plus button only runs the `/new` reset path for the active session and there is no visible flow to choose a configured agent, create a session with that `agentId`, and switch to it. An open related PR, #61423, overlaps the new-session modal surface but explicitly scopes itself to the current agent and says the agent selection flow is unchanged, so this issue still has distinct remaining work.

## What This Changes

Request a WebChat/Control UI flow that lets users choose a configured agent when creating a new chat session, then switches to that new agent-scoped session.

## Best Possible Solution

Keep this issue open and implement the missing agent-aware creation flow in Control UI: present configured agents from `agents.list`, call `sessions.create` with the selected `agentId`, switch to the returned session key, refresh session options/history/avatar, and preserve existing model/thinking/session-switch controls. If #61423 remains active, the best path is to extend or supersede it with the agent-selection behavior rather than landing two conflicting new-session UIs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: configure multiple agents, open Control UI Chat, and use the visible plus/New session control; current main routes that action to `/new` on the active session and has no UI path that submits a selected `agentId` to `sessions.create`.

## Solution Assessment

Unclear for the related open PR because #61423 creates a named session only for the current agent and explicitly leaves agent selection unchanged. The best maintainable solution for this issue is narrower than a new Gateway API but broader than #61423: add an agent-aware create-session UI on top of the existing `agents.list` and `sessions.create` contracts.
