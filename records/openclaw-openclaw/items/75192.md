## Summary

Keep open. Current main has an agent-aware session dropdown and a backend `sessions.create` RPC, but the Control UI still lacks a visible New Chat flow that lets a user choose a configured agent and switch to the newly created session. The related closed reports cover switching and labeling, while the open sidebar request is adjacent rather than a complete duplicate of this create-session workflow.

## What This Changes

The issue requests a WebChat UI flow to create a new chat session for a selected configured agent.

## Best Possible Solution

Add a focused Control UI New Chat affordance that lists configured agents from `agents.list`, creates a session through existing `sessions.create` with the selected `agentId`, switches to the returned key through the existing session-switch path, refreshes sessions, and documents the user-visible behavior without changing Gateway protocol unless a concrete API gap appears.

## Reproduction Assessment

Yes. Static current-main reproduction is high confidence: the WebChat chat header renders only existing session options, `/new` resets the current session, and `ui/src` has no `sessions.create` call despite the backend RPC supporting `agentId`.

## Solution Assessment

Yes, a narrow Control UI change using the existing `sessions.create` contract is the best path. A new core API or plugin boundary is not needed based on the inspected schema, handler, tests, and docs.
