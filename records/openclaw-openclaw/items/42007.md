## Summary

Keep this PR open. Current main still does not implement Feishu text keyword mention triggers: the Feishu config schemas and docs have no `mentionTriggers` surface, group handling still drops non-@mentioned messages when `requireMention` is true, and broadcast dispatch still derives active reply behavior from native `ctx.mentionedBot`. The PR remains a focused implementation candidate, but the existing review comments identify real merge blockers around broadcast dispatch, `WasMentioned`, empty trigger validation, and config precedence.

## Best Possible Solution

Keep this PR open and request a Feishu plugin revision. A mergeable version should choose the final config contract, preferably aligning with or intentionally distinguishing shared `mentionPatterns`; validate or normalize empty trigger entries; define top-level/account/group/agent precedence; treat keyword matches as effective mentions for both single-agent and broadcast dispatch; update `activeAgentId` and `WasMentioned`; and add focused schema, policy, bot, broadcast, and Feishu docs coverage.

## Risks / Open Questions

- Closing this PR now would lose a focused implementation candidate for behavior that current main still lacks.
- Merging the PR as provided would likely still behave incorrectly in broadcast-configured Feishu groups because active reply dispatch and `WasMentioned` remain tied to native `ctx.mentionedBot`.
- The proposed `z.array(z.string())` accepts empty strings; substring matching an empty trigger would make every message satisfy the trigger and effectively bypass `requireMention`.
- The config contract is unsettled: a Feishu-specific `mentionTriggers` field overlaps with existing shared `messages.groupChat.mentionPatterns` / `agents.list[].groupChat.mentionPatterns`, and top-level/account/group precedence is not defined.
- The PR does not include docs or focused tests for schema acceptance, keyword matching, broadcast dispatch, empty trigger handling, or precedence.
