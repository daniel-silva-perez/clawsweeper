## Summary

Keep open. Current main still does not implement Feishu `mentionTriggers`, and the PR remains a focused implementation candidate for a bundled Feishu plugin gap. The PR should be revised before merge because the current diff does not fully handle broadcast dispatch semantics, empty-trigger validation, per-group trigger configuration, tests, or docs.

## Best Possible Solution

Keep this PR open and request a Feishu plugin revision. A mergeable version should add a normalized non-empty `mentionTriggers` contract, define precedence across top-level, account, wildcard-group, and exact-group config, treat a keyword match as an effective mention for single-agent and broadcast dispatch, update `activeAgentId` and `WasMentioned` semantics, and add Feishu tests plus docs/config-reference updates.

## Risks / Open Questions

- Merging the PR as-is can silently fail in Feishu broadcast groups because keyword-only messages would pass the new early gate while no active Feishu reply dispatcher is selected.
- The proposed `z.array(z.string())` plus substring matching accepts empty strings; `includes("")` would make every group message satisfy the trigger and bypass `requireMention` when configured.
- The user-facing contract remains incomplete without normalized non-empty triggers, per-group precedence, focused Feishu tests, and docs/config-reference updates.
- Security/supply-chain pass: the provided diff is limited to two Feishu plugin source files and does not touch workflows, dependency sources, lockfiles, install/build/release scripts, package publishing metadata, secrets handling, or downloaded artifacts; the security-sensitive concern is the empty-trigger access-control behavior.
