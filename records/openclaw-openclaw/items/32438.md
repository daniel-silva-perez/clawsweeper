## Summary

Current main still does not implement the requested formatted sidecar log file. The central logger resolves only the JSONL file path and max size, writes one redacted JSON payload to the active log file, and current docs/UI/CLI surfaces still describe or consume Gateway file logs as JSONL. Recent CLI and channel-log changes improve local fallback and rolling-file discovery, but they do not create a persistent human-readable file beside `logging.file`. The earlier duplicate #29378 is closed, and the paired implementation PR #32442 is closed unmerged, so there is no merged fix or canonical open PR to close against.

## Best Possible Solution

Keep this issue open and land a focused logging enhancement: derive a human-readable sidecar path from the active `logging.file` path, write redacted formatted lines beside the unchanged JSONL log, keep day-change and size-rotation behavior bounded, document the file, and add regression coverage for redaction, rotation, and unchanged JSONL consumers.

## Clownfish Work Prompt

Implement openclaw/openclaw#32438 as a focused logging enhancement. Current main writes only the configured JSONL file; add a human-readable formatted sidecar derived from the active JSONL log path by inserting `-formatted` before the extension, for example `openclaw.log` -> `openclaw-formatted.log` and dated rolling files analogously. Keep `logging.file` and all existing JSONL content/consumers unchanged; do not add a new config key unless unavoidable. Write the sidecar whenever the main file transport writes, using the same log level threshold and redaction policy. Format each line as local timestamp, level, subsystem/module when available, and flattened message. Keep day-change, pruning, and size-rotation behavior bounded so the sidecar cannot grow without the same lifecycle discipline as the JSONL file.
