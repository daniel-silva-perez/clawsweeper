## Summary

Keep open. Current main still writes Gateway file logs as a single redacted JSONL stream, and the CLI, Control UI, and docs continue to treat that file as JSONL; recent local CLI fallback and rolling-log discovery work improve adjacent access paths but do not create the requested durable formatted companion file. The earlier duplicate is closed and the same-author implementation PR was closed unmerged, so this remains valid follow-up work.

## What This Changes

Request to derive and write a human-readable formatted companion log file beside the unchanged JSONL Gateway file log whenever file logging is enabled.

## Best Possible Solution

Add a focused logger-layer enhancement that derives a readable companion path from the active resolved logging.file, writes already-redacted plain-text lines beside the unchanged JSONL stream, handles rolling day changes and rotation/retention deliberately, and updates operator docs plus regression coverage for redaction and unchanged JSONL consumers.

Yes. A source-level reproduction is high-confidence: enable file logging and emit a log event; current buildLogger appends one redacted JSONL payload to activeFile and no formatted companion path exists in settings, schema, docs, or UI. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. A logger-layer sidecar is the narrowest maintainable solution for offline host inspection when the Gateway is down; CLI local fallback and Control UI parsing are useful display-time improvements but do not create the requested durable plain-text file.
