## Summary

Current main still does not implement the requested formatted companion log file. The logger resolves one active JSONL file and appends only JSON-stringified records, the public settings/schema/docs still describe file logs as JSONL, and CLI/Control UI consumers remain on the configured JSONL path. The earlier duplicate report is closed and the paired implementation PR was closed unmerged; recent log-tail improvements are adjacent but do not create the persistent human-readable sidecar requested here.

## What This Changes

The issue requests a derived plain-text formatted log file beside the existing JSONL Gateway log so operators can inspect logs when the Control UI is unavailable.

## Best Possible Solution

Land a focused logger-transport enhancement that derives a human-readable companion path from the active `logging.file`, writes redacted formatted lines beside the unchanged JSONL file, mirrors day-change and rotation lifecycle safely, documents the path, and adds regression coverage for redaction, rotation, and unchanged JSONL consumers.

## Repair Work Prompt

Implement openclaw/openclaw#32438 as a focused logging enhancement. Current main writes only the configured JSONL file; add a human-readable formatted companion file derived from the active JSONL log path by inserting `-formatted` before the extension, for example `openclaw.log` -> `openclaw-formatted.log` and dated rolling files analogously. Keep `logging.file` and all existing JSONL content/consumers unchanged; do not add a new config key unless unavoidable. Write the companion whenever the main file transport writes, using the same log level threshold and the same redaction policy.
