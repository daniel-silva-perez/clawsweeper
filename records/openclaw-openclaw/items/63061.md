## Summary

Keep this PR open. Current main still suffixes CLI profile state directories under `OPENCLAW_HOME`, while the generic config resolver uses the unsuffixed `$OPENCLAW_HOME/.openclaw` default. The PR is not obsolete, but maintainer product judgment is still needed because current CLI help/docs describe named profiles as separate state directories.

## Best Possible Solution

Keep this PR open for maintainer review. If `OPENCLAW_HOME` is intended to define the physical state root, land a normalized version of this PR and update CLI help/docs plus any service-path helpers that should follow the same precedence. If named profiles must remain separate even inside `OPENCLAW_HOME`, reject the PR and document that profile suffixing wins for the combined case.

## Risks / Open Questions

- Landing the PR would make explicit `OPENCLAW_HOME` the physical isolation boundary and collapse named profiles to `.openclaw` inside that home, which differs from current profile-isolation help/docs for the combined case.
- The PR does not update CLI help or docs, so maintainers should decide and document precedence between `OPENCLAW_HOME` and `--profile`/`OPENCLAW_PROFILE`.
- Some daemon/service fallback helpers still derive profile-suffixed state when `OPENCLAW_STATE_DIR` is absent, so they should be reviewed with the same scenario before merging.
