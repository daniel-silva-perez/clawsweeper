## Summary

Keep this PR open for maintainer follow-up rather than closing as implemented. Current main has a separate Google Vertex ADC improvement for the `gcp-vertex-credentials` marker in the provider transport, but the PR branch is stale, targets the old `"<authenticated>"` sentinel, edits the pre-split runner path, and adds unrelated Copilot compaction auth behavior.

## What This Changes

The PR adds a google-vertex ADC placeholder runtime-auth guard, applies it to embedded run and compaction flows, adds focused model-auth coverage, and also changes GitHub Copilot token handling in compaction.

## Best Possible Solution

The maintainable end state is a current-main google-vertex auth path that handles the `gcp-vertex-credentials` marker consistently for run, compaction, and provider transport, preserves real API-key auth, covers authorized_user and any intended service-account/metadata ADC behavior with focused tests, and keeps unrelated Copilot token changes out of this fix.

## Reproduction Assessment

Yes for the authorized_user marker path: current tests and code show ADC evidence resolving to `gcp-vertex-credentials` and the Google Vertex transport converting that marker to a Bearer token. Reproduction is still unclear for generic service-account ADC because the current transport helper requires an authorized_user credentials file while the auth-evidence marker only checks that a local credentials file exists.

## Solution Assessment

No, this PR is not the best way to solve the issue on current main. It targets the old placeholder and legacy runner file, while the safer path is a rebased or replacement fix against `src/agents/pi-embedded-runner/run/auth-controller.ts`, `src/agents/pi-embedded-runner/compact.ts`, and the provider-owned Google transport, without the unrelated Copilot compaction change.
