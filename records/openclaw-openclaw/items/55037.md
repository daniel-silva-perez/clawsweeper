## Summary

Keep this PR open. Current main still only supports workspace selection at the agent/default level, not per route binding, so the PR is not obsolete or implemented. The current PR head remains an implementation candidate, but it needs integration work: reply execution and runtime plugin setup should consume the already resolved route/binding workspace rather than re-scanning bindings from a session-key suffix.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rework it so AgentRouteBinding and the strict/generated config schema expose optional workspace, resolveAgentRoute carries the matched binding workspace with the route, and dispatch/reply/runtime-plugin setup consumes that resolved route value. Add docs and regression tests for peer, parent-peer, account, team, guild/role scoped bindings, uppercase peer IDs, case-insensitive agent IDs, fallback to agents.list[].workspace or agents.defaults.workspace, and preservation of binding session.dmScope.

## Risks / Open Questions

- Current PR head re-derives the workspace from agentId plus peer-id suffix instead of consuming the matched route, which can pick the wrong workspace when channel, accountId, teamId, guildId, roles, or parent-peer scope differs.
- The patch reads opts.sessionKey even though current GetReplyOptions does not define that field, so it needs an API/type integration pass against main.
- Workspace selection controls cwd, hooks, media staging, plugin runtime setup, and workspace-local context; a partial implementation can mix data across worktrees.
