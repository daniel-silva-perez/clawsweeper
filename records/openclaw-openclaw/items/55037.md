## Summary

Keep open. Current main still does not accept or apply a route-binding workspace override, and the PR remains useful, but its current diff has merge-blocking integration issues around re-deriving the binding instead of consuming the already resolved route.

## What This Changes

The PR adds an optional `bindings[].workspace` config field, threads it into `ResolvedAgentRoute`, and tries to prefer that binding workspace over the agent workspace during reply execution.

## Best Possible Solution

Carry the matched binding workspace as part of the resolved route and pass that resolved value through dispatch/reply/runtime setup, while updating the config type, strict schema, generated help/labels, public routing docs, and focused tests for every binding tier and fallback path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Use the resolved route for workspace selection:** `src/auto-reply/reply/get-reply.ts:60-63`
  - body: `resolveBindingWorkspace` scans all bindings by agent and peer-id suffix, so it ignores channel/account/team/guild/role constraints and cannot apply non-peer binding workspaces. In multi-account or team-scoped setups this can run the agent in the wrong worktree; the workspace should come from the binding already selected by `resolveAgentRoute`.
  - confidence: 0.9
- **[P1] Add `sessionKey` to the reply options contract:** `src/auto-reply/reply/get-reply.ts:192`
  - body: The new call reads `opts.sessionKey`, but `GetReplyOptions` on current main does not declare that property and dispatch does not pass it. With strict TypeScript this is a type error, and at runtime the helper has no reliable session key to match.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds workspace selection logic, and the current lookup can cross route scopes, which is an isolation-sensitive regression.

Concerns:

- **[medium] Workspace override can cross binding scopes:** `src/auto-reply/reply/get-reply.ts:61`
  - body: The new helper selects a workspace by agent plus peer-id suffix without validating channel, account, team, guild, or role constraints. Because workspace controls cwd/tool roots and workspace-local runtime behavior, a reused peer ID can make one conversation run in another binding's directory.
  - confidence: 0.88
