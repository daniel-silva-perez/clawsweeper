## Summary

Keep this PR open. Current main still supports workspace selection at the agent level for normal route bindings, not per binding, and the PR remains a plausible core routing/config change. Its current head still needs integration work because reply execution should consume the already resolved route/binding instead of re-scanning bindings by session-key suffix.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rework it so AgentRouteBinding and the strict/generated config schema expose optional workspace, resolveAgentRoute carries the matched binding workspace together with session overrides, and reply/runtime setup consumes that resolved route value instead of re-scanning bindings. Add docs and tests for peer, account, team, guild/role scoped bindings, uppercase peer IDs, case-insensitive agent IDs, fallback to agents.list[].workspace or agents.defaults.workspace, and preservation of binding session.dmScope.

## Risks / Open Questions

- The current PR head can run an agent in the wrong directory if a peer-id suffix matches across different channel/account/team/guild/role scopes, because the workspace lookup is detached from the full route resolver.
- The PR appears to read opts.sessionKey without adding that field to GetReplyOptions, so it likely needs a type/API integration pass against current main.
- Runtime workspace selection controls tool cwd, hooks, media staging, plugin runtime setup, and workspace-local context. A partial implementation is an isolation and data-mixing risk.
- A new public config field needs generated schema/help labels, public docs, and regression tests for precedence, validation, fallback, case-insensitive agent IDs, uppercase peer IDs, and binding session.dmScope preservation.
