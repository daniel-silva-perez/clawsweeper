## Summary

Keep open. This is a fresh, actionable Docker/WebChat latency bug with detailed timings and two follow-up reproductions on v2026.4.26. Current main still has the same WebChat async dispatch, stuck-session diagnostics, ACP turn/session queue path, and cold `tools.effective` deferral that match the reported symptoms; I found no current-main source, tests, docs, or post-release history proving the 30-90s completion latency is fixed or superseded.

## Best Possible Solution

Keep this issue open as the active user-facing latency tracker. Maintainers should reproduce or collect traces on current main/current release, add timing around `chat.send` -> `dispatchInboundMessage` -> lane dequeue -> `runEmbeddedAttempt` start -> tool inventory/system prompt/MCP setup -> stream function creation -> model completion, then land targeted fixes for the measured bottleneck(s). Likely follow-ups include reworking or landing the `tools.effective` cold-miss behavior from #73540, fixing the config-param stripping tracked by #73607 so OpenAI SSE/warm-up toggles can be tested, and adding a regression/perf guard for pre-run latency on a small Docker profile.
