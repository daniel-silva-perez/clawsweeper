## Summary

Keep open. The report is fresh and actionable, a second commenter reproduced related slow-turn phases on v2026.4.26, the related tools.effective PR was closed unmerged, and current main still has the same asynchronous WebChat/ACP completion path with no source, test, release, or live proof that the 30-90s Docker/OpenAI WebChat completion latency is fixed.

## Best Possible Solution

Keep this issue open as the active Docker/WebChat OpenAI latency tracker. The next maintainer pass should capture current-main or next-release per-phase timings for chat.send, tools.effective, acp-dispatch, embedded run start, prompt-hook/memory work, streamFn wrapper creation, actual OpenAI call time, and final transcript/UI delivery. Then fix the implicated path directly, likely alongside the open config-params stripping report if transport override testing is still blocked, and add targeted regression coverage plus one Docker/OpenAI manual proof.
