## Summary

Keep open. This is a fresh, actionable Docker/WebChat latency bug with multiple independent reproductions on v2026.4.26, and current main at 955b4df still has the same relevant behavior: `chat.send` acknowledges immediately and dispatches asynchronously, the stuck-session/queue telemetry still matches the report, the ACP session actor still serializes turns until terminal completion, the embedded runner still performs substantial pre-model bootstrap/tool/MCP/LSP/system-prompt/session setup, and the closed related tools.effective PR is not present on main. The only central-path change since the earlier ClawSweeper review is a `chat.history` TUI startup optimization, not a fix for WebChat turn latency.

## Best Possible Solution

Keep this issue open as the active user-facing Docker/WebChat latency tracker. Maintainers should add timing around `chat.send` ack, `dispatchInboundMessage`, lane dequeue, ACP session actor entry, `runEmbeddedAttempt` start, bootstrap/context/tool inventory, bundle MCP/LSP creation, system-prompt assembly, resource loader/session creation, extra-param transport setup, and first provider I/O. Then land targeted fixes with a small Docker/WebChat regression or perf guard. Coordinate narrower follow-ups with #73477 and #73607, and either revive or replace the unmerged #73540 approach if tools.effective cold misses are confirmed to contribute.
