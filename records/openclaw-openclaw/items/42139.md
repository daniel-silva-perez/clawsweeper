## Summary

Keep open. Current main clarifies and implements ACP/acpx as a Gateway-host external harness path, but it still does not provide a first-class way for Discord `/acp spawn` to place Claude/Codex on a separate Mac node. The remaining work is product/security design, not conservative cleanup.

## What This Changes

The issue asks OpenClaw to clarify or support an official Discord ACP topology where the Gateway runs on a Raspberry Pi while Claude/Codex harnesses run on a remote Mac node.

## Best Possible Solution

Keep this as a maintainer product/design issue. The end state should either explicitly document, in the Discord ACP path, that bundled ACP/acpx harnesses run on the Gateway host today, or add first-class remote ACP harness placement with typed config, node selection, credential handling, approvals, auditability, Discord binding behavior, schema docs, and regression coverage.

## Reproduction Assessment

Unclear for the full live failure because the Pi/Mac/Discord topology was not run end to end. There is a high-confidence static reproduction of the product gap: current docs, strict config schemas, runtime inputs, and spawn code expose no remote ACP node selector.

## Solution Assessment

Unclear. A docs-only clarification is the safest narrow improvement for current behavior, while first-class remote Mac ACP harness execution requires a larger security and product design because it crosses node trust, remote process execution, credentials, approvals, and Discord-triggered workflow semantics.
