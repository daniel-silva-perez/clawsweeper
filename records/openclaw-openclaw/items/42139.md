## Summary

Keep open. Current main still treats ACP/acpx harness execution as a Gateway-host runtime, while remote node support is documented and implemented for exec/system.run rather than ACP process placement. The supplied discussion and prior ClawSweeper comment both identify an unresolved product/docs gap for Raspberry Pi Gateway plus remote Mac harness execution.

## What This Changes

The issue asks OpenClaw to clarify or support a Discord ACP topology where the Gateway runs on a Raspberry Pi and Claude Code/Codex harnesses run on a remote Mac node.

## Best Possible Solution

Keep this open for maintainer product/docs judgment. The short-term end state should clearly document that bundled Discord /acp spawn and acpx harnesses run on the Gateway host today, with practical workarounds such as running the Gateway/acpx on the Mac or using an operator-managed wrapper. If first-class support is desired, it should add a typed remote ACP runtime node selector and process-execution contract across config schema, ACP runtime SDK, acpx integration, node-host/RPC handling, Discord docs, and targeted tests.
