## Summary

Keep open. Current main still exposes ACP/acpx as a Gateway-host external harness path: the ACP docs, Discord binding examples, config types, spawn path, and runtime manager all carry agent/backend/mode/cwd-style options, with no node selector or remote harness placement contract. The related open PR #46740 may improve one `acpx exited with code 1` error-classification symptom, but it does not resolve the remote Mac harness placement question.

## What This Changes

The issue asks whether Discord `/acp spawn` can run Claude Code or Codex ACP harnesses on a remote Mac node while the Gateway runs on another host, and asks for an official configuration pattern or documentation clarification if unsupported.

## Best Possible Solution

Keep this as a maintainer product/design issue. The end state should either explicitly document that bundled Discord ACP/acpx harnesses run on the Gateway host today, or add first-class remote ACP harness placement with typed config, node selection, process execution, credentials, approvals, auditability, Discord binding behavior, schema docs, and regression coverage.

## Reproduction Assessment

Partially. There is a high-confidence static reproduction for the product gap because current docs, config types, ACP runtime input, and spawn code expose no remote ACP node selector; the exact `acpx exited with code 1` topology was not live-reproduced.

## Solution Assessment

Unclear. A narrow docs clarification would be safe, but first-class remote Mac ACP harness execution is not a small repair because it crosses node trust boundaries, remote process execution, credentials, approvals, and Discord-triggered workflow semantics.
