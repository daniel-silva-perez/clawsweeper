## Summary

Keep open. Current main has clearer ACP/acpx and Discord binding documentation, but the official implementation still runs ACP harnesses through the Gateway-host runtime and does not expose a first-class remote Mac node selector or node-host harness process contract. The prior ClawSweeper review also kept this open, and the stale label was removed afterward, so the current discussion points to maintainer product/docs follow-up rather than cleanup closure.

## What This Changes

The issue asks for official ACP/Discord support or clear documentation for running the Gateway on one host while Claude/Codex ACP harnesses execute on a remote Mac node, plus clearer guidance when normal Discord routes and ACP bindings overlap.

## Best Possible Solution

Keep the issue open for maintainer product/docs judgment. The short-term end state should explicitly document that bundled ACP/acpx harnesses run on the Gateway host today and recommend running the Gateway/acpx on the Mac or using an operator-managed wrapper for multi-machine setups. If first-class support is desired, add a remote ACP runtime node selector and process-execution contract across config schema, ACP runtime SDK, acpx integration, node-host RPC/process management, Discord `/acp spawn` docs, and targeted tests.
