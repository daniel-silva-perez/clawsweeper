## Summary

Keep open. Current main supports and documents ACP/acpx as a Gateway-host external harness path, and Discord bind/thread behavior is clearer than the original report, but there is still no first-class way for Discord ACP spawn to place Claude/Codex on a separate Mac node. That remaining capability crosses product and security boundaries, so it is not conservative cleanup.

## What This Changes

Request to clarify or implement the official Discord ACP topology where the Gateway runs on a Raspberry Pi while Claude/Codex ACP harnesses execute on a separate Mac mini node.

## Best Possible Solution

Keep this as a maintainer product/design issue. The end state should either document in the Discord ACP path that bundled ACP/acpx harnesses run on the Gateway host today, or add a first-class remote ACP harness placement model with typed config, node selection, credential handling, approvals, audit logging, Discord binding semantics, schema docs, and regression coverage.

## Reproduction Assessment

Unclear for the full live failure because the Pi/Mac/Discord topology was not run end to end. There is a high-confidence static reproduction of the product gap: current docs, strict schemas, runtime inputs, and spawn handoff expose no node or host selector for ACP harness placement.

## Solution Assessment

Unclear. A docs-only clarification is the narrowest safe improvement for current behavior, while actual remote Mac ACP harness execution requires product and security design before code changes.
