## Summary

Keep open. Current main has clearer ACP/acpx and Discord binding documentation, but the official contract still requires the external harness to be runnable on the Gateway host and exposes only Gateway-host ACP runtime options; remote node-host docs cover `exec host=node`, not launching ACP harness processes. The prior issue discussion also kept this open and the stale label was removed afterward, so this remains a maintainer product/docs follow-up rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to clarify or support a Discord ACP topology where the Gateway runs on a Raspberry Pi while Claude/Codex ACP harnesses execute on a remote Mac mini, including clearer behavior when Discord channels already have non-ACP route bindings.

## Best Possible Solution

Keep this open for maintainer product/docs judgment. The short-term end state should explicitly document that bundled Discord `/acp spawn` and acpx harnesses run on the Gateway host today, with practical workarounds such as running the Gateway/acpx on the Mac or using an operator-managed wrapper. If first-class support is desired, it should add a typed remote ACP runtime node selector and process-execution contract across config schema, ACP runtime SDK, acpx integration, node-host process/RPC handling, Discord docs, and targeted tests.
