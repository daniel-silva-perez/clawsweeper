## Summary

Keep open. Current main clarifies ACP/acpx as a Gateway-host external harness path and supports Discord ACP bindings, but it still has no first-class contract for selecting a remote Mac node as the ACP harness execution host.

## What This Changes

The issue asks whether Discord ACP sessions can officially run Claude/Codex harnesses on a separate Mac node while the Gateway runs on another host, and asks for documentation or feature tracking if not.

## Best Possible Solution

Keep this as a maintainer product/design issue. The near-term docs should explicitly state that bundled Discord `/acp spawn` and acpx harnesses execute on the Gateway host today, and first-class remote support should add a typed ACP runtime node selector plus process-execution, credentials, approval, audit, Discord, config-schema, and regression-test coverage.

## Reproduction Assessment

Partially. There is a high-confidence static reproduction for the product gap because current docs, schemas, and runtime types expose no remote ACP node selector; the exact `acpx exited with code 1` topology was not live-reproduced.

## Solution Assessment

Unclear. A narrow docs clarification would be safe, but first-class remote ACP harness execution is not a small fix because it crosses node trust boundaries, credentials, approvals, and Discord-triggered process execution.
