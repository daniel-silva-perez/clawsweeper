## Summary

Keep open. Current main documents ACP/acpx and Discord binding flows more clearly, but it still does not provide the requested official topology where Discord `/acp spawn` on a Raspberry Pi Gateway runs the ACP harness process on a remote Mac node. The remaining gap is a product/runtime contract, not just a stale support question.

## Best Possible Solution

Keep this issue open for maintainer product/docs judgment. Short term, document explicitly that bundled ACP/acpx harnesses run on the Gateway host today and recommend running the Gateway/acpx on the Mac or using an operator-managed wrapper for multi-machine deployments. If OpenClaw should support the requested topology, add a typed ACP runtime node selector and node-host process execution contract across config schema, ACP runtime types/manager, acpx integration, node protocol, Discord `/acp` docs, and regression coverage.
