## Summary

Keep open. Current main at 359d871293e801dc9e5506b5002a4bf545c42662 still has the transcript rewrite path that branches at the first replacement and re-appends the active-branch suffix, so the raw JSONL duplication mechanism described here remains present even though `sessions_history` itself is read-only and active-branch readers hide some abandoned entries.

## What This Changes

Fix transcript-only duplication where `sessions_history` or other large tool results trigger truncation/rewrite and cause prior transcript suffix entries to be re-appended with new ids while preserving the same message contents and response ids.

## Best Possible Solution

Keep this open until #69486, #72511, or an equivalent maintained fix lands and is verified against this `sessions_history` raw JSONL case. The end state should keep `sessions_history` as a bounded read while making transcript rewrite/truncation avoid low-value duplicate suffix replay or safely clean abandoned entries without breaking append-only branch semantics.

Yes. The issue includes concrete raw JSONL evidence, a follow-up comment reproduced the mechanism, and current-main source still contains the same large-tool-result truncation to suffix-rewrite path; this review did not create live transcript artifacts because it was read-only.
