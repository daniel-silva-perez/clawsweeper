## Summary

Keep open. Current main still has the reported raw transcript duplication mechanism: `sessions_history` is read-only, but large history results can feed the tool-result truncation path, which branches at the first rewritten entry and re-appends the remaining branch suffix. Active-branch history reduces visible duplicates, but it does not eliminate append-only JSONL replay entries.

## What This Changes

The issue reports that invoking `sessions_history` on an active session can indirectly cause tool-result truncation to duplicate prior raw transcript entries with fresh ids but identical content and assistant `responseId`s.

## Best Possible Solution

Keep this open until #72511, #69486, or an equivalent maintained fix lands and is verified against this `sessions_history`-triggered raw JSONL case. The end state should keep `sessions_history` as a pure bounded read while making transcript rewrite or cleanup avoid low-value duplicate suffix replay without breaking append-only branch semantics.

## Reproduction Assessment

Yes. The issue includes concrete raw JSONL evidence, the discussion reproduced the mechanism, and current-main code still contains the same truncation-to-rewrite suffix replay path; this review did not create live transcript artifacts because it was read-only.

## Solution Assessment

Unclear. Fixing `sessions_history` itself would be the wrong boundary; the maintainable solution belongs in transcript rewrite/truncation cleanup, but maintainers should choose between the existing open repair strategies before a new automated patch is queued.
