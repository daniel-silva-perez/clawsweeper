## Summary

Keep open. Current main confirms `sessions_history` itself is read-only and `chat.history` now reads active transcript branches, but the reported mechanism still exists in the tool-result truncation path: it plans replacements, branches at the first rewritten entry, then re-appends the entire remaining branch suffix, including unchanged entries. The linked #73234 was a separate WebChat/TTS assistant-media duplicate class and does not supersede this transcript rewrite issue.

## What This Changes

The issue asks OpenClaw to stop transcript-only duplicate branch writes that appear after large `sessions_history` results trigger persisted tool-result truncation, while preserving correct billing and model context behavior.

## Best Possible Solution

Keep this open and fix the maintenance rewrite path narrowly: preserve `sessions_history` as a pure read, keep active-branch history behavior, and adjust tool-result truncation/transcript rewrite so it avoids low-value deep suffix replays or rewrites only the necessary tail while preserving branch labels, compaction markers, and provider overflow recovery semantics.

## Reproduction Assessment

Yes. The issue includes concrete reproduction data, and a follow-up comment reproduced the mechanism on `upstream/main @ e84ebeafbd`; current main still contains the same `truncateOversizedToolResultsInExistingSessionManager` to `rewriteTranscriptEntriesInSessionManager` branch-and-reappend path.

## Solution Assessment

Unclear. The current implementation is not a complete solution because it still re-appends unchanged suffix entries; however, the best fix needs a maintainer choice between tail-only rewrites, skip thresholds, or a lower-level transcript rewrite because the session dependency is append-only by design.
