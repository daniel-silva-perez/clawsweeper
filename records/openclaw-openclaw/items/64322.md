## Summary

Keep this PR open. Current main still materializes Discord thread fallback matches with the parent channel ID, so sibling Discord threads under one configured parent can still share the same configured ACP session key. The PR is also paired with the open same-author bug report #64199, and the discussion narrowed the implementation toward a focused Discord fix rather than obsolete work.

## Best Possible Solution

Rebase and review this PR, or land an equivalent focused fix: make Discord parent-channel ACP matches materialize the runtime thread `conversationId` while preserving `parentConversationId`, keep registry support for channel-canonical remaps such as Feishu topic folding, and include regression coverage for sibling Discord threads under one configured parent plus reverse-lookup/reset behavior for thread-derived configured ACP keys. Keep #64199 open until this PR or an equivalent fix lands.

## Risks / Open Questions

- Changing Discord configured ACP session keys from parent-keyed to thread-keyed will start fresh persistent ACP state for existing threads that previously shared the parent session.
- The PR's reset fallback treats any syntactically configured-format ACP session key as configured when registry reverse lookup cannot recover the runtime thread ID; maintainers should confirm that is acceptable for reset and stateful-target behavior.
- The fix should remain surgical. The initial broad registry/type rewrite would have changed Feishu `group_topic_sender` topic-folding semantics, and the branch should be checked after rebase to ensure that remains reverted.
