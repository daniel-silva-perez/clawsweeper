## Summary

Keep this PR open. Current main still materializes Discord configured ACP parent-channel matches with the parent channel ID, so sibling Discord threads under one configured parent can still collide on the same persistent ACP session; the PR remains a relevant focused fix, and the discussion shows the earlier broad Feishu-breaking rewrite was reverted.

## What This Changes

The PR changes Discord configured ACP parent-channel matches to use the runtime thread conversation ID, adds regression coverage for distinct thread session keys and reset behavior, and includes small channel lint/type cleanups.

## Best Possible Solution

Land this PR after rebase, or land an equivalent current-main fix: update the Discord ACP parent fallback to materialize the runtime thread `conversationId` while preserving `parentConversationId`, keep registry materialization from channel-canonical matches for Feishu-style remaps, and include regression coverage for sibling Discord threads plus reset behavior for thread-derived configured ACP keys.
