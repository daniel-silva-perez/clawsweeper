## Summary

Keep this PR open. Current main still materializes Discord thread parent-fallback ACP bindings with the parent channel ID, so sibling Discord threads under one configured parent can still share the same configured ACP session key. The PR is paired with the open same-author bug report #64199 and remains an implementation candidate rather than obsolete work.

## Best Possible Solution

Rebase and review this PR, or land an equivalent focused fix: Discord parent-channel ACP matches should materialize the runtime thread `conversationId` while preserving `parentConversationId`, registry support for channel-canonical remaps such as Feishu topic folding should remain intact, and regression coverage should cover sibling Discord threads under one configured parent plus reset behavior for thread-derived configured ACP keys. Keep #64199 open until the fix lands.

## Risks / Open Questions

- Landing the fix changes configured Discord ACP session granularity from parent-keyed to thread-keyed, so existing threads that shared a parent persistent ACP process will start fresh thread-specific ACP state.
- Thread-derived configured ACP keys cannot be reverse-resolved from the one-way hash; the PR's reset fallback should be reviewed for any session-key-only configured-binding paths beyond reset.
- The initial broad registry/type rewrite would have changed Feishu topic-folding behavior; maintainers should confirm the rebased branch still keeps that reverted and focused on Discord.
