## Summary

Keep open. Current main still forwards `embedding.dimensions` to the direct OpenAI-compatible `/embeddings` request and has a test that expects that behavior, while the PR head makes a focused source/config/manifest/docs/test change for a valid provider-compatibility bug. The latest diff has no definite blocking review finding, but exact-head CI was still pending and the active contributor PR needs normal maintainer review rather than closure or a replacement repair branch.

## What This Changes

The PR changes memory-lancedb’s direct OpenAI-compatible embedding path to omit `dimensions` from upstream `/embeddings` requests, truncate returned vectors locally, validate configured dimensions as positive integers in runtime and manifest config, update regression tests, and clarify docs.

## Best Possible Solution

Land a narrow memory-lancedb fix after exact-head validation that preserves the direct `/embeddings` transport, proxy/timeout/retry behavior, and response normalization, while omitting non-portable upstream `dimensions`, locally truncating normalized vectors, and keeping runtime config, manifest metadata, docs, and regression tests aligned.

## Reproduction Assessment

Yes. Current main has a source-level reproduction because configured `dimensions` is added to the outbound `/embeddings` body and the current test expects it; the PR body also gives a concrete vLLM-compatible provider reproduction. The PR’s new focused tests cover the intended omitted-field, local truncation, and short-vector paths, though I did not run tests because this was a read-only review.

## Solution Assessment

Yes for the reported provider-compatibility bug. Omitting the non-portable `dimensions` field on the raw OpenAI-compatible path and truncating locally is a narrow maintainable fix; provider-specific cases that need upstream dimensionality control should be handled through provider adapters or a separate explicit contract rather than reintroducing unconditional forwarding.
