## Summary

Keep open. Current main still exposes the existing broad cron job creation/update path and does not implement the requested constrained agent-facing scheduling API with gateway-injected provenance, playbook allowlisting, per-agent quota, or recurring lifetime semantics. The issue is recent, security-sensitive, and has supporting operator context, so it needs maintainer API/security design rather than cleanup closure.

## What This Changes

The issue requests an agent-facing scheduling API that lets agents create constrained cron jobs through the gateway with non-forgeable gateway-injected provenance, playbook allowlists, quotas, audit entries, and bounded recurring lifetimes.

## Best Possible Solution

Keep this as a maintainer-owned security/API RFC and design an additive Gateway scheduling contract that preserves unrestricted operator cron while adding a constrained agent path with server-side actor/session provenance, registered playbooks, parameter validation, quotas, audit records, and explicit recurring lifetime semantics.

## Reproduction Assessment

Not applicable as a feature/security RFC. The current-state verification is high-confidence: source, docs, protocol schemas, tool schema tests, and gateway method lists on current main show the proposed constrained API and fields are absent.

## Solution Assessment

Unclear for implementation, but keeping the issue open is the right triage outcome. The requested direction is plausible, yet the provenance, permissions, quota, playbook, and expiration contracts need maintainer design before a safe PR boundary exists.
