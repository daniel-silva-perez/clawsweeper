## Summary

Keep open. Current main still exposes the general cron management surface, but not the requested gateway-mediated agent scheduling API with playbook allowlists, non-forgeable actor provenance, recurring-job lifetime controls, quotas, and schedule-create audit entries.

## What This Changes

Request a security/API RFC for agents to create constrained scheduled tasks through the Gateway, with server-injected provenance, playbook allowlists, quotas, recurring expiration or permanence semantics, and audit logging.

## Best Possible Solution

Keep this as a maintainer-owned security/API RFC and design an additive Gateway scheduling contract that preserves unrestricted operator cron while adding a constrained agent path with server-side actor/session provenance, registered playbooks, parameter validation, quotas, audit records, and documented recurring lifetime semantics.

## Reproduction Assessment

Not applicable as a feature/security RFC. We do have a high-confidence current-state check: source, protocol schemas, tests, and docs on current main show the proposed method and fields are absent.

## Solution Assessment

Unclear for implementation, but keeping the issue open is the right triage outcome. The requested direction is plausible, yet the activeUntil/permanent-job question, migration story, and permission/provenance model need maintainer design before a safe PR boundary exists.
