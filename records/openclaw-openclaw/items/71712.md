## Summary

Keep open: current main has an owner-only agent cron tool and gateway cron RPC, but it does not implement the narrower provenance-injected scheduling API requested here. The current cron schema lacks createdBy/playbook/activeUntil/quota fields, and the existing tests/docs cover owner-only filtering and scoped self-removal rather than non-forgeable schedule provenance; the production comment also raises a real design conflict with permanent recurring jobs and #71711.

## What This Changes

The issue requests an agent-facing scheduling API that lets agents create constrained cron jobs through the gateway with gateway-injected provenance, playbook allowlists, quotas, recurring-job expiry, and audit logging.

## Best Possible Solution

Define an additive gateway-owned scheduling surface that injects requester identity and creation/update timestamps from authenticated runtime context, accepts only operator-registered playbook ids with validated params, enforces per-agent quotas and reviewed expiry/permanent semantics, and records schedule-create/update events without weakening the existing operator cron CLI.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. Source and docs inspection gives a high-confidence verification path: current cron schemas, gateway RPC, agent tool, and docs do not contain the proposed playbook/provenance/quota/expiry API.

## Solution Assessment

Unclear for implementation details, but the direction is not solved by the current broad owner-only cron tool. A maintainer/security design pass should settle playbook registration, permanent-job semantics, provenance identity, and audit storage before code is attempted.
