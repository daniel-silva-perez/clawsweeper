## Summary

Keep open: current main has an owner-only/full-power cron tool and gateway cron.add surface, but it does not implement the proposed narrower agent-facing scheduling API with gateway-injected, non-forgeable provenance, playbook allowlists, active lifetime enforcement, per-agent quota, or creation audit semantics. The issue is recent, security-sensitive, and has concrete production-use comments, so it should get maintainer/product-security review rather than stale cleanup.

## What This Changes

Request a gateway-mediated agent scheduling API that lets agents create or modify constrained cron jobs with gateway-injected provenance, playbook allowlists, recurring-job lifetimes, quotas, and audit records.

## Best Possible Solution

Define a small core design/ADR for agent-scheduled cron jobs: a separate gateway RPC/tool with gateway-injected creator identity and timestamp, registered playbook IDs plus parameter schemas, explicit recurring-job lifetime policy, per-agent quotas, audit/task-ledger entries, migration semantics for existing jobs, and tests/docs that preserve the existing unrestricted operator cron CLI.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The current-main verification path is source/docs inspection: cron.add and the cron tool exist, but the proposed schedule_task/provenance/playbook/quota/lifetime contract does not.

## Solution Assessment

Unclear until maintainers choose the product/security model. The proposed direction is narrower and safer than exposing the existing full cron tool to agents, but it should be resolved as a designed core gateway capability rather than an automated one-off patch.
