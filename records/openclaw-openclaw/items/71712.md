## Summary

This issue should stay open. Current main still exposes the existing owner/operator cron management surface, but it does not implement the proposed gateway-mediated agent scheduling API with non-forgeable gateway-injected provenance, registered playbook allowlists, per-agent quotas, recurring activeUntil/permanent semantics, or schedule-create audit entries. The issue is recent, has production-use comments, and is security/API design work rather than stale cleanup.

## Best Possible Solution

Keep this open as a maintainer-reviewed security/API RFC. The best path is to design a gateway-mediated scheduling surface or constrained cron mode that injects actor/session provenance server-side, requires registered playbooks with parameter schemas, coordinates with #71711 on recurring expiration versus explicitly permanent jobs, adds per-agent quotas, records schedule-create audit entries, documents the boundary from unrestricted operator cron, and decides how existing jobs without provenance should be displayed or migrated.

## Risks / Open Questions

- This is security-sensitive API design. The wrong implementation could let prompt-injected agent content create persistent recurring work, blur owner-vs-agent provenance, or expose a broader scheduling capability than operators expect.
- The comments introduce a real product tension with #71711: some users want expiring recurring jobs, while AS76 reports evergreen production jobs that need an explicit permanent/null model. Maintainers should decide that contract before implementation.
- A one-off patch would likely be premature because the issue spans gateway protocol, cron persistence, agent tool policy, playbook registration/schema validation, quotas, audit semantics, docs, and migration behavior.
