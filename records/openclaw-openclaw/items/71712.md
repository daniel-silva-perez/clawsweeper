## Summary

Keep open: current main still has the existing operator/admin cron RPC plus an owner-only full-power agent cron tool, but it does not provide the proposed narrower agent-facing scheduling API with gateway-injected provenance, playbook allowlists, recurring-job lifetime policy, per-agent quotas, or creation audit semantics. The RFC is recent, security-sensitive, and the comments add real production context, so this needs maintainer product/security design rather than cleanup closure.

## What This Changes

Request to design and harden an agent-facing Gateway scheduling API that injects non-forgeable provenance, restricts jobs to allowlisted playbooks and validated params, enforces recurring-job lifetime/quota policy, and audits schedule creation or modification.

## Best Possible Solution

Define a small core design/ADR for agent-scheduled cron jobs: a separate Gateway RPC/tool with gateway-injected creator identity and timestamp, registered playbook identifiers plus parameter schemas, explicit recurring-job lifetime/permanent policy, per-agent quotas, task/audit-ledger entries for creation and updates, migration semantics for existing jobs, and docs/tests that keep the unrestricted operator cron CLI intact.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main has cron.add and the owner-only cron tool, but not the proposed schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear whether the exact proposed TypeScript shape is the final best API until maintainers choose the trust and product model. The requested direction is safer than exposing the full cron surface to non-owner agents, but it should land as a designed core Gateway capability instead of an automated one-off patch.
