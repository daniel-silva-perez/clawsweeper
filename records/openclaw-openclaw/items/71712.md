## Summary

Keep open: current main still exposes cron through the existing operator/admin cron surfaces and the owner-only full-power agent tool, but does not implement the proposed narrower agent-facing scheduling API with gateway-injected provenance, playbook allowlists, recurring-job lifetime policy, per-agent quota, or creation audit contract. The RFC is recent, security-sensitive, and has concrete production-use comments, so it needs maintainer product/security design rather than cleanup closure.

## What This Changes

The issue requests a gateway-owned agent scheduling API that lets agents create constrained cron jobs with non-forgeable provenance, playbook allowlists, lifetime limits, quotas, and audit entries.

## Best Possible Solution

Define a small core design/ADR for agent-scheduled cron jobs: a separate gateway RPC/tool with gateway-injected creator identity and timestamp, registered playbook IDs plus parameter schemas, explicit recurring-job lifetime/permanent policy, per-agent quotas, task/audit-ledger entries for creation and changes, migration semantics for existing jobs, and tests/docs that preserve the unrestricted operator cron CLI.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source/docs inspection: cron.add and the owner-only cron tool exist, but the proposed schedule_task/provenance/playbook/quota/lifetime contract does not.

## Solution Assessment

Unclear until maintainers choose the product/security model. The proposed direction is narrower and safer than exposing the existing full cron tool to agents, but it should be resolved as a designed core gateway capability rather than a one-off automated patch.
