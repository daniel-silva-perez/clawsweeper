## Summary

Keep open: current main still has operator/admin cron APIs and an owner-only full-power cron tool, but not the proposed narrowed agent-facing scheduling API with gateway-injected provenance, playbook allowlists, recurring lifetime policy, per-agent quotas, or creation audit contract. The RFC is recent, security-sensitive, and has production-use context in the discussion, so it needs maintainer product/security design rather than cleanup closure.

## What This Changes

The issue requests a gateway-owned API that lets agents create or modify constrained scheduled jobs with non-forgeable provenance, playbook allowlists, lifetime/quota guardrails, and task audit entries.

## Best Possible Solution

Write a small core design/ADR for agent-scheduled cron before implementation, then add an additive gateway scheduling method/tool that injects creator identity and timestamp from the authenticated session, accepts only registered playbook IDs plus validated params, enforces recurring lifetime and quota policy, records creation/update audit entries, and leaves the unrestricted operator cron CLI intact.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source/docs inspection: current main exposes cron.add and an owner-only cron tool, but not the proposed schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product/security model. The requested direction is narrower and safer than exposing the existing full cron tool to agents, but it should be designed as a core gateway capability rather than attempted as a one-off autonomous patch.
