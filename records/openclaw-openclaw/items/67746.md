## Summary

Keep open. Current main still leaves the reported cleanup gap: CLI logout clears WhatsApp auth locally without going through the live gateway logout path, and channel removal mutates config without stopping the running gateway session or invoking WhatsApp credential cleanup. The related implementation PR was closed unmerged and only covered the logout routing subset.

## What This Changes

The issue asks for WhatsApp logout/remove to stop the active gateway session, remove WhatsApp config, and clear managed WhatsApp credentials without manual file edits or process kills.

## Best Possible Solution

Define the channel lifecycle contract explicitly and implement it in the narrow owner paths: CLI logout should prefer the live gateway `channels.logout` RPC with a controlled local fallback, and `channels remove --delete` should stop the live account and invoke plugin-owned credential cleanup before deleting WhatsApp config, while preserving custom-auth-dir safety rules and documenting the behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction is high confidence: mock `runChannelLogout` with a reachable gateway and observe that current main never calls `channels.logout`; mock `channelsRemoveCommand --delete` and observe that it writes config without stopping the gateway or calling WhatsApp logout.

## Solution Assessment

No for current behavior. The narrow maintainable direction is to reuse the existing gateway logout and WhatsApp `logoutWeb` seams rather than inventing WhatsApp-specific cleanup in generic config deletion code.
