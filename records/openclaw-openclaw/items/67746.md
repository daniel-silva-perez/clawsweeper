## Summary

Keep open. Current main still does not fully address the reported WhatsApp cleanup problem: the CLI logout path calls the plugin's local `logoutAccount` helper directly instead of the live gateway `channels.logout` RPC, while `channels remove --delete` only mutates config through the plugin config adapter and does not stop the active gateway runtime or invoke WhatsApp credential cleanup. The related PR #67981 was closed unmerged and explicitly scoped itself to the logout/session mismatch, leaving remove/config semantics unchanged.

## Best Possible Solution

Implement a live-aware CLI logout path that prefers the gateway `channels.logout` RPC for the default local gateway, with a clear fallback only when the gateway is unreachable. Then define and test `channels remove --delete` semantics for WhatsApp: stop the active runtime and either call plugin-owned credential cleanup before deleting config, or explicitly document/remove-copy that deletion is config-only and must be preceded by logout for credential cleanup. Add focused regression tests in `src/cli/channel-auth.test.ts`, `src/commands/channels/remove.test.ts`, and the gateway channel logout tests so the live-session teardown and remove behavior cannot regress.

## Risks / Open Questions

- Privacy risk remains if users run CLI logout/remove while a local gateway continues handling the already-running WhatsApp socket.
- Remove semantics need product decision: either `remove --delete` should be a destructive disconnect that stops runtime and invokes plugin-owned credential cleanup, or docs/CLI copy should make clear it is config-only and requires logout first.
- Credential deletion must preserve current safety boundaries for custom auth directories and symlink-crossing paths.
