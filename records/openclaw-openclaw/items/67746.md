## Summary

Keep open. Current main still routes CLI logout directly to the plugin's local `logoutAccount` helper, so it can clear WhatsApp credentials without coordinating with the live Gateway session. The Gateway already has a `channels.logout` path that stops runtime first, and the linked #67981 fix attempted that routing, but that PR is closed unmerged and does not implement the fix on main.

## What This Changes

The issue asks for WhatsApp logout/remove to terminate the active Gateway session, remove the WhatsApp config block, and delete managed WhatsApp credentials.

## Best Possible Solution

Land a maintainer-reviewed fix that routes CLI logout through the existing Gateway `channels.logout` seam before falling back to local cleanup, and separately clarify or adjust `channels remove` so disabling/deleting an account cannot leave a running WhatsApp runtime behind while preserving documented config deletion semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock a configured WhatsApp plugin plus reachable Gateway and call `runChannelLogout`; current main calls the local plugin helper and never calls the live `channels.logout` RPC, while the Gateway handler already proves the runtime-stop path exists.

## Solution Assessment

Yes for the narrow logout/session bug: the existing Gateway `channels.logout` RPC is the best seam because it stops the runtime before plugin credential cleanup. The broader remove/config deletion behavior needs maintainer confirmation before changing default CLI semantics.
