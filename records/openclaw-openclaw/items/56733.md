## Summary

Keep open. Current main has meaningful diagnostics, readiness detail, Feishu HTTP timeouts/reconnect handling, active-memory timeout guardrails, and channel-level stale-socket restarts, but it still does not prove recovery from the reported process-alive/all-HTTP-timeout state and does not provide a systemd watchdog or notify contract for a frozen event loop.

## What This Changes

The issue requests Gateway reliability improvements for a process that remains active under systemd while the Node event loop or outbound HTTP path stops responding, causing provider and channel API calls to time out until manual restart.

## Best Possible Solution

Keep the issue open until maintainers choose a supported recovery contract: add documented external watchdog/probe guidance for Linux/WSL services, add first-class systemd notify/watchdog support, or identify the underlying Feishu/HTTP/event-loop blocker and cover it with targeted regression tests.

## Reproduction Assessment

No high-confidence local reproduction path is available from the repository alone. The issue provides detailed overnight field evidence, but reproducing it appears to require the reporter's WSL2, Feishu WebSocket, multi-agent, provider, and low-activity setup.

## Solution Assessment

Unclear. The current diagnostics and channel/plugin-level mitigations are useful but are not the narrowest proven solution for a process-alive/all-HTTP-timeout state; supervisor-level watchdog behavior would be a broader service policy decision.
