## Summary

Keep this PR open. Current main still lacks a focused stored-session explain command, so the proposed work is not obsolete; however, the PR branch has correctness issues in scoped lookup, explicit-store handling, legacy-key selection, parent flag propagation, and diagnostic reason reporting that should block merge until maintainer review or author follow-up.

## What This Changes

The PR adds an `openclaw sessions explain <key>` CLI subcommand with text/JSON output, optional `--tool` availability checks, effective model/workspace/tool-policy diagnostics, and sessions command/registration tests.

## Best Possible Solution

Land a session explain command only after it is wired through the canonical session-store target and freshest-entry lookup contracts, honors existing parent CLI flags and agent scoping, reports model/tool/subagent reasons from the same selected entry/store, and adds docs plus tests for multi-agent, custom-store, legacy-key, and provider/model override cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor parent --json for explain:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The parent `sessions` command owns `--json`, and sibling subcommands merge parent and child flags. Here `openclaw sessions --json explain <key>` passes `json: false` unless the flag appears after `explain`, so scripts using the shared flag position get text output instead of JSON.
  - confidence: 0.92
- **[P2] Honor --agent when resolving explain targets:** `src/commands/sessions.ts:119-123`
  - body: `--agent` is passed into `sessionsCommand`, but explain resolves `opts.explain` without applying the requested agent. For unqualified keys such as `main`, `openclaw sessions --agent work explain main` canonicalizes against the default agent and can report the wrong session or not found.
  - confidence: 0.91
- **[P2] Load explicit stores through the resolved path:** `src/commands/sessions.ts:118`
  - body: This loads `opts.store` directly before the normal target resolver can expand `~` or `{agentId}` via `resolveStorePath`. Store values accepted by list/cleanup can therefore miss the file in explain mode.
  - confidence: 0.88
- **[P2] Pick the freshest matching store key:** `src/commands/sessions.ts:126-127`
  - body: The target returns canonical plus legacy/case-variant keys, but `.find` picks the first existing key rather than the newest entry. Current gateway lookup uses the freshest matching entry, so explain can report stale model/workspace data when both keys exist.
  - confidence: 0.9
- **[P2] Base model reasons on the selected source:** `src/commands/sessions.ts:217-223`
  - body: `resolveSessionModelRef` prefers explicit model overrides before persisted runtime fields and ignores provider-only overrides, but these flags and reasons are based only on field presence. Sessions with override+runtime metadata or provider-only overrides will show sources as participating when they did not determine the effective model.
  - confidence: 0.89
- **[P2] Use the inspected store for subagent diagnostics:** `src/commands/sessions.ts:145-165`
  - body: Explain selects a concrete `entry` from the inspected store, but capabilities and tool inventory are recomputed with `target.canonicalKey` and config-derived stores. For `--store <custom>` or legacy-key matches, the reported role/control scope/tool groups can describe a different session than the one printed.
  - confidence: 0.87
