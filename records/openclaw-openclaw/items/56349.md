## Summary

Keep open. Current main has meaningful pieces of the requested boundary, including `message_sending` cancel/modify support in the generic outbound dispatcher and newer Telegram/auto-reply hook wiring, but it still does not prove an unbypassable pre-send guarantee across all user-visible outbound paths. The Gateway webchat/control UI paths can still construct and broadcast final assistant payloads directly without running `message_sending`, and related narrower work remains open.

## What This Changes

The issue requests a single enforceable pre-send outbound policy boundary that can validate, modify, or block every outbound message before any channel or user-facing delivery path sends it.

## Best Possible Solution

Define one documented outbound policy contract and add coverage that proves every user-visible delivery surface either goes through `deliverOutboundPayloads` or an equivalent shared pre-send helper. The end state should include webchat/control UI final delivery, streaming/final semantics, message tool sends, auto-replies, and channel/plugin adapters, with tests that fail when a new path bypasses the hook or policy boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register a `message_sending` hook and send through Gateway webchat/control UI: `server-chat.ts` and `server-methods/chat.ts` build and broadcast final chat payloads without consulting the hook runner, while external channel delivery goes through hook-aware paths.

## Solution Assessment

No current-main implementation is not enough to close this as solved. The safer direction is to reuse or extract a shared pre-send helper for webchat and other non-`deliverOutboundPayloads` sends, then add contract tests proving no normal outbound path bypasses it.
