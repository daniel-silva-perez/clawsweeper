## Summary

Keep open. Current main has partial outbound hook coverage, including `message_sending` rewrite/cancel support in the shared dispatcher and newer auto-reply/Telegram wiring, but source inspection still shows user-visible WebChat/control UI and sampled channel-plugin reply paths that can deliver without a guaranteed pre-send boundary. The issue comments add useful live Telegram evidence from 2026.4.15 plus related Slack/WebChat work, so this remains a real platform invariant gap rather than cleanup-close material.

## What This Changes

The issue requests a single enforceable pre-send outbound policy boundary that can validate, modify, or block every outbound message before channel or user-facing delivery.

## Best Possible Solution

Define one documented outbound policy contract and route every user-visible delivery surface through `deliverOutboundPayloads` or an equivalent shared pre-send helper. The end state should cover WebChat deltas/finals, `chat.send`/`chat.inject`, explicit message sends, auto-replies, and bundled/external channel-plugin dispatchers, with contract tests that fail when a new normal outbound path bypasses the boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register a `message_sending` hook and deliver an assistant reply through Gateway WebChat/control UI; current `server-chat.ts` and `server-methods/chat.ts` construct and broadcast final `chat` payloads without consulting the hook runner.

## Solution Assessment

No current-main implementation is not enough to solve the issue. The narrowest maintainable direction is to extract or reuse a shared async pre-send helper for WebChat and channel reply dispatchers, then add boundary coverage before deciding any stronger runtime policy/classification layer.
