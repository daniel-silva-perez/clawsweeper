## Summary

Keep this PR open. Current main still has the reported 20s browser file-chooser client timeout, and the PR is a focused fix for that path, but the proposed CLI upload change leaves explicit `--timeout-ms` values using the same deadline for the server operation and the client transport wrapper.

## What This Changes

The PR updates browser file-chooser arming to use the browser action timeout budget, raises the CLI upload default request timeout to 60s, and adds browser client timeout assertions.

## Best Possible Solution

Land the narrow timeout alignment after padding the CLI explicit-timeout path, preferably by reusing the same action-style timeout helper or equivalent `+5_000ms` transport slack, while keeping upload guardrails and Playwright semantics unchanged.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, `browserArmFileChooser` and the CLI upload wrapper use a 20s transport timeout, and the PR's added unit expectations directly exercise the missing request-timeout propagation; I did not run tests because this review is read-only.

## Solution Assessment

No, not quite as proposed. The browser client part is the right narrow fix, but the CLI explicit-timeout path should also get transport slack so the wrapper deadline cannot expire at the same boundary as the server operation.
