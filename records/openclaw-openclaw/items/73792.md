## Summary

Keep open. Current main still has the stale MCP session-runtime cache behavior the PR targets, but the PR head introduces a blocking lifecycle regression: its transport-close handler also fires during intentional startup-failure cleanup and can retire the whole session runtime instead of skipping only the failed MCP server.

## What This Changes

The PR wires MCP transport `onclose`/`onerror` callbacks into session MCP runtimes, routes close events through manager-level session disposal with an operator log line, and adds a SIGKILL-based regression test for stale stdio MCP child sessions.

## Best Possible Solution

Keep the PR open and update the patch so manager-level eviction runs only for unexpected transport closure. Intentional cleanup for failed startup or listing should mark the server session as removed before closing it, and the test suite should add multi-server partial-startup coverage proving one failed server does not retire healthy sessions.

## Reproduction Assessment

Yes. The stale-runtime bug is reproducible by materializing a session MCP stdio server, killing its child process, and observing current main keep returning the cached runtime; the PR regression is reproducible with a multi-server config where one server fails during startup/listing after the new `onclose` handler is installed.

## Solution Assessment

No, not yet. Installing the callback before `connectWithTimeout` matches the SDK chaining contract, but the implementation must guard intentional closes, for example by deleting or marking the session before `disposeSession(session)` and no-oping the handler when the session is already absent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard intentional transport closes before eviction:** `src/agents/pi-bundle-mcp-runtime.ts:277-279`
  - body: This handler also runs when OpenClaw intentionally closes a transport in the startup-failure catch path, because the pinned MCP `Transport` contract says `onclose` should fire from `close()` and `OpenClawStdioClientTransport.close()` waits for the child's `close` event. Since the catch deletes `sessions` only after `disposeSession(session)`, a failed server in a multi-server config can call `onTransportClosed`, retire the whole session runtime, and abort or remove healthy server sessions that current main would keep. Mark intentional cleanup before closing and have the handler no-op when the session is already gone.
  - confidence: 0.9
