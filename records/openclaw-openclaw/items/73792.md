## Summary

Keep this PR open. Current main still lacks transport-close eviction for cached session MCP runtimes, and the PR targets a real stale-runtime failure, but the proposed patch has a blocking correctness issue: its new `onclose` handler also fires during intentional cleanup after a failed MCP server startup, so a multi-server runtime that should keep healthy servers can be retired/disposed instead.

## What This Changes

The PR wires MCP transport `onclose`/`onerror` callbacks into session MCP runtimes, routes close events through manager-level session disposal with an operator log line, and adds a SIGKILL-based regression test for stale stdio MCP child sessions.

## Best Possible Solution

Update the patch so manager-level eviction runs only for unexpected transport closure, while intentional cleanup for failed startup/listing marks the server session as removed before closing it. Preserve the child-death stale-runtime regression test and add multi-server partial-startup coverage proving one failed server does not retire healthy sessions.

## Reproduction Assessment

Yes. The stale-runtime bug is reproduced by the PR's proposed probe-server SIGKILL test and is supported by current main's missing `onclose` eviction path; the PR regression is reproducible with a multi-server MCP config where one server fails during startup/listing and another should remain usable.

## Solution Assessment

No, not yet. Installing the callback before `connectWithTimeout` is the right shape for the SDK chaining contract, but the implementation must guard intentional closes, for example by deleting/marking the session before `disposeSession(session)` and no-oping the handler when the session is already absent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Guard intentional transport closes before eviction:** `src/agents/pi-bundle-mcp-runtime.ts:277-280`
  - body: This handler also runs when OpenClaw intentionally closes a transport in the existing startup-failure catch path, because the pinned MCP transport contract says `onclose` fires from `close()` and `OpenClawStdioClientTransport.close()` waits for the child's `close` event. In a multi-server config, a failed server cleanup can therefore call `onTransportClosed`, retire the whole session runtime, and make catalog materialization fail or close healthy server sessions that current main would keep. Mark intentional cleanup before closing and have the handler no-op when the session is already gone.
  - confidence: 0.88
