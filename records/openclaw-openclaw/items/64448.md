## Summary

Current main does not implement the requested Feishu outbound image idempotency. The duplicate-send report remains plausible because the image path still uploads media and sends the resulting image message without a stable retry key, and the generic outbound context does not carry one to Feishu.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin outbound media path. Add or propagate a stable outbound send idempotency key through the CLI/gateway/direct channel context into `sendMediaFeishu`, apply it to Feishu `im.message.create` and `im.message.reply` using the SDK-supported `uuid` field for image sends, and add regression coverage that simulates a duplicate retry/reconnect while still allowing a later intentional send of the same image.

## Risks / Open Questions

- The provider field name should be implemented against the pinned/current Feishu SDK contract; the SDK type surface points to `uuid` on message create/reply, not `idempotent_id` on image upload.
- A local in-flight lock keyed only by media URL could suppress intentional repeat sends; a stable per-send retry key propagated into the provider call is the safer design.
- The bug is timing-dependent, so regression coverage should simulate duplicate retry/reconnect behavior instead of relying on a live WebSocket race.
