## Summary

Keep open. Current main and the latest tagged release still have the paragraph-boundary and table-unaware streaming behavior this PR is trying to fix, but the branch is not merge-ready because its separator-preserving path can exceed `maxChars` and its new table detector misses valid pipe-table shapes.

## What This Changes

The PR updates `EmbeddedBlockChunker` to preserve newline/paragraph separators and avoid detected table splits, adds a new markdown table-span helper with tests, extends streaming integration coverage, and updates streaming docs plus the changelog.

## Best Possible Solution

Land one canonical shared `EmbeddedBlockChunker` fix that preserves reconstructed Markdown boundaries without exceeding channel caps, recognizes the relevant GFM pipe-table spans, and keeps the unit, integration, docs, and changelog coverage aligned with #42106, #66614, and #66605.

## Reproduction Assessment

Yes. Current main has a deterministic source/test reproduction: `flushOnParagraph` consumes paragraph separators before emitting, and `pi-embedded-block-chunker.test.ts` still asserts the stripped boundary behavior. The PR-specific `maxChars` regression is also reproducible from the diff by using a paragraph break whose preserved separator pushes the emitted chunk over the configured limit.

## Solution Assessment

No, not as-is. The direction is right because the fix belongs in the shared streaming chunker rather than in channel-specific joins, but the branch needs a narrow repair for size clamping and table-span completeness before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cap paragraph separators before emitting:** `src/agents/pi-embedded-block-chunker.ts:217-221`
  - body: The new branch checks `paragraphBreak.index - start <= paragraphLimit`, but it then emits through `paragraphBreak.index + paragraphBreak.length`. A blank-line separator with spaces or extra newlines can therefore make the emitted chunk exceed `maxChars`, contradicting the documented channel cap and the existing clamping behavior.
  - confidence: 0.9
- **[P2] Accept shorter table delimiters:** `src/markdown/table-spans.ts:49`
  - body: The delimiter regex requires three or more dashes per cell, so valid pipe tables with one- or two-dash delimiter cells are not recognized as table spans and can still split mid-table. The parser should accept hyphen-only delimiter cells with optional alignment colons.
  - confidence: 0.82
- **[P3] Recognize single-column pipe tables:** `src/markdown/table-spans.ts:44-49`
  - body: Both row and delimiter detection require at least two cells after trimming outer pipes. A one-column table such as `| Name |` / `| --- |` / `| Alpha |` is therefore left unprotected by the new table-safe chunking path.
  - confidence: 0.78
