## Summary

Current main now covers the reported WhatsApp reconnect-stall cluster: the later WhatsApp reliability fixes address repeated 408 reconnect recovery, group inbound recovery after reconnect-time metadata failures, transport liveness, and Baileys crypto-error reconnects. The fix is present on main and included in the v2026.4.29-beta.1 prerelease, while the older v2026.4.27 stable release predates it.

## What This Changes

The issue reports WhatsApp listener disconnect and stall loops that produce `No active WhatsApp Web listener`, missed inbound messages, group hydration failures, and Bad MAC errors.

## Best Possible Solution

Keep the WhatsApp plugin fixes on the plugin side: socket timing stays configurable, transport activity drives liveness, reconnect-time group metadata fallback is cached inside the WhatsApp runtime, and crypto/session faults force reconnect instead of core migrations or broad channel-policy changes.

## Reproduction Assessment

Yes for the code-level reproduction. Current tests simulate post-408 reconnects with continued transport frames but no app delivery, plus group inbound after reconnect-time metadata fetch failure; a live WhatsApp session was not replayed in this read-only pass.

## Solution Assessment

Yes. The implementation is the narrow maintainable direction because it fixes the WhatsApp-owned Baileys listener, watchdog, metadata, and crypto-reconnect behavior without adding bundled-channel assumptions to core.
