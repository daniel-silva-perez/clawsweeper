## Summary

Keep this issue open. Current main still validates the request-body OpenAI `model` as an OpenClaw agent target before dispatch, even when `x-openclaw-agent-id` has selected an agent. The latest release tree shows the same behavior, and the active issue comments describe a concrete production compatibility need.

## Best Possible Solution

Implement a narrow Gateway compatibility fix: when `x-openclaw-agent-id` selects the agent for an OpenAI-compatible chat-completions request, ignore the request-body `model` for agent-target validation and let the selected agent's configured model run. Preserve explicit `x-openclaw-model` override validation, add regression coverage for generic body models with the agent header, decide/document whether `/v1/responses` follows the same contract, and update the public OpenAI HTTP docs if supported.

## Clownfish Work Prompt

Fix the Gateway OpenAI-compatible agent-header model validation issue. Current main rejects generic request-body models such as `gpt-4` before dispatch even when `x-openclaw-agent-id` selected the target agent. Implement a narrow compatibility path so `/v1/chat/completions` ignores the request-body `model` for agent-target validation when the agent header is present and selected, then runs the selected agent's configured model. Preserve existing behavior when no agent header is present. Preserve explicit `x-openclaw-model` parsing, allowlist enforcement, and error messages. Because `resolveOpenAiCompatModelOverride` is shared by `/v1/responses`, either apply the same header-aware behavior there with tests and docs, or keep the change scoped with an explicit helper option and document the reason. Add regression coverage for `model: "gpt-4"` plus `x-openclaw-agent-id` succeeding and dispatching to the selected agent, and for `x-openclaw-model` validation still rejecting invalid/disallowed overrides. Update `https://docs.openclaw.ai/gateway/openai-http-api` source docs if the compatibility contract changes; add a changelog fix entry with a real reporter credit if the repo policy requires it. Do not change provider model allowlist semantics, bundled plugin boundaries, or unrelated HTTP auth behavior.
