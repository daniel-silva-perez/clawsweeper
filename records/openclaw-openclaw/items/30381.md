## Summary

Current main still validates the OpenAI-compatible request-body `model` before dispatch even when `x-openclaw-agent-id` has already selected the agent, so generic OpenAI SDK model ids such as `gpt-4` can still be rejected. The issue remains valid, narrow, and suitable for a focused Gateway compatibility fix.

## Best Possible Solution

Implement a narrow Gateway compatibility fix: when `x-openclaw-agent-id` or its `x-openclaw-agent` alias resolves the agent for OpenAI-compatible chat completions, do not reject a generic OpenAI request-body `model`; route to the selected agent and continue using the agent's configured model unless `x-openclaw-model` is explicitly supplied and allowed. Add regression coverage, make the shared `/v1/responses` behavior intentional, and update the public OpenAI HTTP docs.
