## Summary

Keep this issue open. Current main still validates the OpenAI-compatible request body `model` before dispatch even when `x-openclaw-agent-id` or `x-openclaw-agent` has already selected the agent, so generic OpenAI SDK model ids can still be rejected. The report is concrete, recently reaffirmed in the discussion, and narrow enough for a focused Gateway compatibility fix.

## Best Possible Solution

Implement a narrow Gateway compatibility fix: when `x-openclaw-agent-id` or `x-openclaw-agent` resolves the agent for OpenAI-compatible chat completions, do not reject a generic request-body `model`; route to the selected agent and use the agent's configured model unless `x-openclaw-model` is explicitly supplied and allowed. Add regression coverage, make the shared `/v1/responses` behavior intentional, and update the public OpenAI HTTP docs.

## Repair Work Prompt

Fix the Gateway OpenAI-compatible chat-completions behavior for this issue. Current `main` rejects generic request-body `model` values such as `gpt-4` through `resolveOpenAiCompatModelOverride` even when `x-openclaw-agent-id` or compatibility alias `x-openclaw-agent` has already selected an agent. Expected behavior: for `/v1/chat/completions`, when the agent header resolves an agent, do not reject the body `model`; dispatch to that agent and use the agent's configured model unless `x-openclaw-model` is supplied and passes existing parsing/allowlist checks. Preserve strict body-model validation when no agent header is present. Preserve `x-openclaw-model` validation and allowlist semantics. Make an explicit choice for `/v1/responses` because it shares the helper path; if matching chat completions, add coverage there too. Do not change `/v1/embeddings`, `/v1/models`, provider catalogs, or model allowlist policy unless required by tests. Add regression tests for `model: "gpt-4"` plus `x-openclaw-agent-id: beta` returning 200 and dispatching to `agent:beta` with no backend model override, and a test showing `x-openclaw-model` still overrides only when valid. Update `docs/gateway/openai-http-api.md` and, if `/v1/responses` follows the same behavior, `docs/gateway/openresponses-http-api.md`. Add a single-line changelog Fixes entry in the active changelog section with thanks to `@mr-slurpy-wildcard` and `@shanekovalsky`.
