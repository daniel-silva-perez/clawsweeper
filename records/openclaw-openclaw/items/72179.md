## Summary

Keep this PR open for maintainer/API review. The underlying cron observability request is valid, but current main already has the merged `cron_changed` hook for the same gateway cron lifecycle stream, so this PR should not add a second overlapping public hook name as-is.

## What This Changes

The PR adds a new typed `cron_lifecycle` plugin hook, gateway cron start/finish emissions, hook documentation, and focused plugin/gateway tests.

## Best Possible Solution

Rework the PR around current main’s `cron_changed` contract: either add accepted missing fields such as `agentId` there with docs/tests, or make an explicit maintainer decision to replace/deprecate that hook before introducing `cron_lifecycle`.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a runtime bug. The high-confidence verification path is static inspection of current main: `cron_changed` is present, documented, tested, and emitted by gateway cron handling, while `agentId` is still absent from the public cron job projection.

## Solution Assessment

No. Adding `cron_lifecycle` beside the shipped `cron_changed` hook is not the narrowest maintainable solution; extending or clarifying `cron_changed` avoids splitting the public plugin contract unless maintainers intentionally want a new hook name.
