## Summary

Keep open. Current main still matches the reported Telegram regression: plain replies with non-`off` `replyToMode` populate the native-quote candidate map, and the dispatcher still uses that map as a blanket reason to disable answer draft preview streaming. The focused source PR was closed unmerged, and the open related configuration PR is broader, unmerged work rather than a landed fix.

## What This Changes

The issue requests that Telegram `streaming.mode: "partial"` continue to show live preview edits for plain replies when `channels.telegram.replyToMode` is non-`off`, while still disabling previews for genuine selected Telegram quote replies.

## Best Possible Solution

Land a narrow Telegram dispatcher fix that treats only real selected quote replies as preview-stream incompatible, preserves `replyQuoteByMessageId` for final delivery fallback, and adds focused regression coverage for `replyToMode: "first"` plus `streaming.mode: "partial"` on a plain Telegram message.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `replyToMode: "first"`, `streamMode: "partial"`, and a plain message with text and `MessageSid`, the dispatcher populates `replyQuoteByMessageId`, makes `hasNativeQuoteReply` true, and therefore never creates the answer draft stream.

## Solution Assessment

Yes. The narrowest maintainable fix is to scope the disable guard to a genuine selected quote, such as `replyQuoteText != null && replyQuoteMessageId != null`, while leaving native quote candidates available for final delivery.
