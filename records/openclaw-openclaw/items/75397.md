## Summary

Keep open. The PR addresses two real provider-runtime regressions visible on current main, and the patch is narrowly aligned with the plugin manifest ownership and runtime-cache contracts; I found no blocking correctness or security findings.

## What This Changes

The PR extends provider owner resolution to manifest `modelCatalog` provider and alias keys, avoids caching provider-runtime misses while provider loading is in flight, and adds focused regression tests.

## Best Possible Solution

Land this focused fix after normal maintainer approval and required gates so targeted provider loading keeps its performance behavior while honoring manifest alias ownership and avoiding config-scoped negative-cache poisoning during nested loads.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path for both failures: `azure-openai-responses` is a manifest alias and hook alias but is omitted from provider owner lookup, and an in-flight lookup currently returns `[]` before the runtime cache stores `null`.

## Solution Assessment

Yes. The PR fixes the generic ownership source and cache-state bug without restoring broad provider scans, which is the narrowest maintainable direction for the targeted hook path.
