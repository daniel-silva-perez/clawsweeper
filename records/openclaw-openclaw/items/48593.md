## Summary

Keep open. Current main still misses the attributed and self-closing final-tag leak in delivery-specific cleanup, and this PR is useful but not ready as-is because it is dirty against main and its widened word-boundary regex can misclassify hyphenated custom tags.

## What This Changes

The PR widens final/thinking tag regexes in embedded streaming, final-tag cleanup, message-end fallback, shared reasoning-tag cleanup, and live-test leak checks, while adding self-closing reasoning-tag handling and regression coverage.

## Best Possible Solution

Land or replace this with a current-main patch that uses one exact reasoning-tag matcher/parser across streaming cleanup, user-facing sanitization, and live-test leak assertions, supporting attributed and self-closing `<final>`/`<think>` tags while rejecting hyphenated custom tag names.

## Reproduction Assessment

Yes. A focused reproduction is to stream or sanitize text containing `<final id="1">Visible</final>` or `<final/>` against current main; code inspection shows the delivery-specific regexes only match `final\s*>`, and current focused tests cover plain split tags but not attributed or self-closing delivery-path cases.

## Solution Assessment

No, not as-is. The PR is directionally right, but the best maintainable fix should use an exact tag-name delimiter such as whitespace, `/`, or `>` and be rebased onto the current sanitizer/streaming code before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require exact reasoning tag delimiters:** `src/agents/pi-embedded-subscribe.ts:25`
  - body: The widened `\b[^<>]*` scanners also match hyphenated custom tags such as `<final-result>` and `<thinking-box>`. In `enforceFinalTag` streaming this can treat non-final custom markup as a real final block and deliver text that should remain suppressed; use a delimiter that only accepts whitespace, `/`, or `>` after the tag name.
  - confidence: 0.88
