## Summary

Keep open. Current main still has the reported macOS direct `wss://` first-use TOFU path, the macOS direct remote config still does not pass `gateway.remote.tlsFingerprint` into that path, and the issue is paired with open unmerged PR #50643 rather than a shipped fix.

## What This Changes

The issue reports that macOS node direct `wss://` first-use TLS auto-trusts an unvalidated gateway certificate and lacks config-side `gateway.remote.tlsFingerprint` hardening.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal certificate trust succeeds or an explicit configured/confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into macOS direct remote session construction, and add focused Swift coverage for first-use TLS and configured fingerprint behavior.
