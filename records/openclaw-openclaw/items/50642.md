## Summary

Keep this issue open. Current main still enables macOS node TLS trust-on-first-use for a fresh wss:// host:port, the TLS delegate still accepts and stores the presented certificate before SecTrustEvaluateWithError, and the macOS direct remote config path still does not consume gateway.remote.tlsFingerprint. The issue is also paired with open PR #50643, so it should remain open until that PR or a maintainer replacement lands.

## What This Changes

The issue requests hardening macOS node wss:// gateway connections so first-use TLS fails closed unless normal trust evaluation succeeds or an explicit pinned/confirmed fingerprint is configured, including support for gateway.remote.tlsFingerprint in the macOS direct remote path.

## Best Possible Solution

Remove the stale label, review and land PR #50643 or a maintainer replacement, and make the macOS node wss:// path fail closed unless SecTrustEvaluateWithError succeeds or an explicit configured/confirmed fingerprint matches. The fix should honor gateway.remote.tlsFingerprint in the macOS direct remote path and add focused Swift coverage for first-use wss:// behavior plus configured fingerprint handling.
