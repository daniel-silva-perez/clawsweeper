## Summary

Keep this issue open. Current main still has the reported macOS node first-use TLS trust-on-first-use path, the configured `gateway.remote.tlsFingerprint` is still not plumbed into the macOS direct remote session, and the issue is paired with open PR #50643 rather than a merged fix.

## What This Changes

The issue reports that the macOS node first-use `wss://` path accepts and pins an untrusted gateway TLS certificate, allowing a rogue gateway to control the node.

## Best Possible Solution

Make the macOS node direct `wss://` path fail closed unless normal trust evaluation succeeds or an explicit configured/confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into that path, and add focused Swift coverage for first-use TLS and configured fingerprint behavior.
