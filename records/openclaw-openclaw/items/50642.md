## Summary

Keep open. Current main still contains the reported macOS direct `wss://` first-use TOFU path, the macOS remote config path still does not carry `gateway.remote.tlsFingerprint` into node session construction, and the issue is paired with open unmerged PR #50643 rather than a shipped fix.

## What This Changes

The issue reports that the macOS node client trusts and stores the first certificate seen for a new direct `wss://` gateway before normal trust evaluation, with no macOS config plumbing for `gateway.remote.tlsFingerprint`.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal certificate trust succeeds or an explicit configured/confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into the macOS direct remote session, and add focused Swift coverage for first-use TLS and configured fingerprint behavior.
