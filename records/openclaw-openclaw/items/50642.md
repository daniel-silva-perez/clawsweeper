## Summary

Keep open. Current main still has the reported macOS direct `wss://` first-use TLS TOFU path, the macOS remote config path still does not plumb `gateway.remote.tlsFingerprint` into node session construction, and the issue is paired with open unmerged PR #50643 rather than a shipped fix.

## What This Changes

The issue asks OpenClaw to stop the macOS node client from auto-trusting the first TLS certificate for a new remote gateway and to require normal trust evaluation, explicit fingerprint pinning, or another confirmed trust path.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal certificate trust succeeds or an explicit configured/confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into the macOS direct remote session, and add focused Swift coverage for first-use TLS and configured fingerprint behavior.
