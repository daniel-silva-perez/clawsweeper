## Summary

Keep this issue open. Current main still enables macOS node TLS trust-on-first-use for a fresh wss:// host:port, the TLS delegate still accepts and stores the presented certificate before SecTrustEvaluateWithError, and the macOS remote config path still does not read gateway.remote.tlsFingerprint. The issue is also paired with open PR #50643, so it should remain open until that PR or a maintainer replacement lands.

## Best Possible Solution

Remove the stale label, review and land PR #50643 or a maintainer replacement, and make the macOS node wss:// path fail closed unless normal SecTrustEvaluateWithError succeeds or an explicit pinned fingerprint is configured/confirmed. The fix should honor gateway.remote.tlsFingerprint in the macOS direct remote path and add focused Swift coverage for first-use wss:// behavior plus configured fingerprint handling.
