## Summary

Keep open. Current main still contains the macOS direct `wss://` first-use TOFU path, the macOS remote config still has no `gateway.remote.tlsFingerprint` plumbing into node TLS params, and the issue is paired with open unmerged PR #50643 rather than a shipped fix.

## What This Changes

The issue requests hardening macOS node direct `wss://` TLS so first-use connections cannot auto-trust a rogue gateway certificate and explicit TLS pinning can protect first connection.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal trust evaluation succeeds or an explicit configured/user-confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into the macOS direct remote session, and add focused Swift coverage for first-use TLS and configured fingerprint behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure macOS remote direct mode to a fresh `wss://` host with no stored keychain fingerprint, and current main sets `allowTOFU: true` then accepts and stores the presented certificate before `SecTrustEvaluateWithError`.

## Solution Assessment

No for current main; it still implements the vulnerable first-use path. The right fix is a narrow macOS TLS enrollment change plus config pin plumbing, with maintainer/security review of open PR #50643 or a replacement that covers both gaps.
