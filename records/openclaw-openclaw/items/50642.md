## Summary

Keep this issue open. Current main still has the reported macOS node first-use TLS trust-on-first-use path, the macOS direct remote config still does not plumb `gateway.remote.tlsFingerprint` into that path, and the issue is paired with open PR #50643 rather than a merged fix.

## What This Changes

The issue asks OpenClaw to stop the macOS node direct `wss://` path from silently trusting the first presented gateway TLS certificate and to provide a configured or confirmed fingerprint/trust path.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal trust evaluation succeeds or an explicit configured/confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into the macOS direct remote session, and add focused Swift coverage for first-use TLS and configured fingerprint behavior.
