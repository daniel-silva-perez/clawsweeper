## Summary

Keep open. Current main still has the reported macOS direct `wss://` first-use TLS trust path, still does not carry `gateway.remote.tlsFingerprint` into the macOS direct node session, and the related closing PR #50643 is open and unmerged rather than a shipped fix.

## What This Changes

The issue requests hardening macOS node direct `wss://` gateway connections so a fresh endpoint cannot be trusted on first use without normal TLS trust validation or an explicit configured/confirmed certificate fingerprint.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection cannot trust an arbitrary certificate, wire `gateway.remote.tlsFingerprint` through the macOS direct remote endpoint into TLS session construction, and add focused Swift/macOS IPC coverage for first-use TLS, configured pins, and stored-pin behavior before landing PR #50643 or a maintainer-chosen replacement.
