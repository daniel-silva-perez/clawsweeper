## Summary

Keep open. Current main still has the reported macOS node first-use TLS trust-on-first-use path, the macOS direct remote config still does not carry `gateway.remote.tlsFingerprint` into that path, and the issue is paired with open PR #50643 rather than a merged fix.

## What This Changes

The issue reports that the macOS node direct `wss://` path trusts a first-use TLS certificate before normal trust evaluation and does not honor configured `gateway.remote.tlsFingerprint`.

## Best Possible Solution

Harden the macOS node direct `wss://` path so first connection fails closed unless normal trust evaluation succeeds or an explicit configured/confirmed fingerprint matches, wire `gateway.remote.tlsFingerprint` into the macOS direct remote session, and add focused Swift coverage for first-use TLS and configured fingerprint behavior.
