## Summary

Keep this draft PR open. The branch addresses a real WebChat terminal-event gap, but the current timeout backstop would regress the existing chat.send timeout contract by applying a fixed 30-second cap to ordinary Control UI sends that currently inherit the configured agent timeout or explicit no-timeout behavior.

## What This Changes

The PR adds a WebChat-specific chat.send terminal timeout backstop, creates missing transcript files before eager user transcript updates, and adds focused gateway regression coverage for the timeout path.

## Best Possible Solution

Keep the terminal backstop concept, but derive its deadline from the same resolved chat.send timeout value used by the abort controller, including explicit coverage for omitted timeoutMs, timeoutMs: 0, and a positive override; the transcript-file initialization can remain as the narrow companion fix.

## Reproduction Assessment

Yes. Use a WebChat client to send chat.send with omitted timeoutMs or timeoutMs: 0 while dispatch remains in flight beyond 30 seconds; the PR timer would emit an error even though current main would use the configured agent timeout or timer-safe no-timeout value.

## Solution Assessment

No. The backstop is directionally reasonable, but the fixed 30-second fallback is not the narrowest maintainable solution because it bypasses the existing gateway timeout semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Preserve the chat.send timeout contract:** `src/gateway/server-methods/chat.ts:191`

Current chat.send resolves omitted timeouts through resolveAgentTimeoutMs, whose default is 48h and whose explicit timeoutMs: 0 means no timeout, and the Control UI does not send timeoutMs. This would abort normal long UI runs before the existing gateway timeout.
