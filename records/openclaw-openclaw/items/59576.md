## Summary

Keep open. Current main still consumes the `file_ids` array from the initial Mattermost WebSocket `posted` payload and does not re-fetch the post when that array is empty, while the linked fix PR was closed unmerged. This remains a valid, narrow Mattermost plugin bug with a clear implementation and test boundary.

## What This Changes

The issue requests that the Mattermost plugin recover incoming file attachments when a Mattermost WebSocket `posted` event arrives before `file_ids` is populated.

## Best Possible Solution

Implement a Mattermost-owned bounded post re-fetch before media resolution when an inbound WebSocket post has no file IDs, then pass recovered IDs into the existing media resolver and cover both file-only and text-plus-file race cases in targeted tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a `posted` event whose parsed post has `file_ids: []`; current main forwards that directly to `resolveMattermostMedia`, which returns no media, matching the reported bot behavior.

## Solution Assessment

Yes. A narrow Mattermost plugin fix using `GET /api/v4/posts/{post_id}` with short bounded retry is the maintainable direction; no core channel API change is needed, but the implementation should avoid using merged debounce post IDs incorrectly.

## Repair Work Prompt

Implement a narrow fix for Mattermost inbound attachments when WebSocket `posted` events arrive with empty `file_ids`. Current main passes `post.file_ids` directly to `resolveMattermostMedia`, so file-only messages can be dropped and text-plus-file messages reach the agent without media. Add a typed `fetchMattermostPost` helper in `extensions/mattermost/src/mattermost/client.ts` using the existing `MattermostClient.request` path for `GET /posts/{postId}` and validate/return `MattermostPost`. In `extensions/mattermost/src/mattermost/monitor.ts`, before resolving media, when the original inbound post has a real post id and empty/null `file_ids`, perform a short bounded delayed refetch/retry to recover populated IDs. Preserve existing debounce behavior; do not re-fetch merged debounced posts with a synthetic/last post id if that would attach the wrong files. Keep the fix inside the Mattermost plugin, do not add core channel IDs or migrations, and do not change dependency patches. Add regression tests for an empty initial `file_ids` WebSocket post that refetches to `['file-1']` before media resolution, plus a no-file/text-only case that keeps retry cost bounded. Add a client test for the GET path. Add a changelog Fixes entry with `Thanks @alexanderkreidich` and consider `Thanks @SeniorXie` if the maintainer wants to credit the follow-up confirmation.
