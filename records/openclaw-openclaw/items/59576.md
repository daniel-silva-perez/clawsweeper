## Summary

Keep open. Current main still resolves inbound Mattermost media directly from the WebSocket post's `file_ids`, returns no media for an empty array, and has no REST re-fetch fallback. The linked fix PR was closed unmerged, and a second user comment reports a similar large-file attachment failure, so this remains a concrete Mattermost plugin bug rather than cleanup that can be closed.

## What This Changes

The issue requests a Mattermost inbound attachment fallback that re-fetches a post over REST when a WebSocket `posted` event arrives with empty `file_ids`.

## Best Possible Solution

Implement the fallback inside the Mattermost plugin: add a typed post-fetch helper using the existing authenticated client, perform a bounded REST retry only when inbound `file_ids` is empty or null, pass the populated ids into media resolution, and cover the helper plus inbound handoff with focused regression tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: when a Mattermost `posted` event contains `file_ids: []`, current main passes that empty array into `resolveMattermostMedia`, which returns no media; the issue also includes concrete user steps and a second similar report.

## Solution Assessment

Yes. A Mattermost-local REST re-fetch is the narrowest maintainable repair because the race is plugin/provider-specific and the existing client/media seams already support the needed authenticated API call and file download path.
