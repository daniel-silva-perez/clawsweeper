## Summary

Keep this issue open. Current main still feeds the Mattermost WebSocket post's `file_ids` directly into media resolution, and the media resolver returns no attachments when that array is empty. The linked implementation PR was closed unmerged, and I did not find an equivalent REST re-fetch path on main.

## Best Possible Solution

Keep this issue open and implement the fix in the Mattermost plugin: add a typed `GET /posts/{postId}` helper, re-fetch with bounded delay/backoff when a posted WebSocket event has empty or missing `file_ids`, feed the refreshed IDs into `resolveMattermostMedia`, and add regression coverage for empty-then-populated file IDs, text-only posts, debounced merged posts, and the 8MiB media cap behavior.

## Risks / Open Questions

- The follow-up comment about files larger than 8MB may also involve the current Mattermost inbound media default cap of `8 * 1024 * 1024`; a fix should separate the linkage race from size-limit behavior.
- A REST re-fetch workaround could add latency or API load for text-only messages because an empty `file_ids` array can mean either no files or not-yet-linked files.
- The debounced multi-message path currently synthesizes `file_ids: []`; any implementation needs to avoid accidentally re-fetching the wrong post or dropping attachments in batched messages.
