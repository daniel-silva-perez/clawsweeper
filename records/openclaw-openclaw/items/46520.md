## Summary

Keep open: current main still returns `URL.origin` directly for custom schemes, so exact `gateway.controlUi.allowedOrigins` entries like `tauri://localhost` cannot match. The provided GitHub context also shows open PR #73511 as the active closing implementation candidate, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests exact `gateway.controlUi.allowedOrigins` matching for custom desktop URL schemes such as `tauri://localhost` and `electron://...`.

## Best Possible Solution

Land a narrow gateway origin parser fix that preserves exact `scheme://host` matching for hosted custom schemes with opaque `URL.origin`, keeps literal `Origin: null` invalid, and adds accepted/rejected `tauri://` and `electron://` regression tests.

## Reproduction Assessment

Yes. The issue reproduces on current main by checking Node URL behavior for `tauri://localhost` / `electron://app` and tracing that `null` origin through `parseOrigin` and the exact allowlist comparison.

## Solution Assessment

Yes. The active PR direction is the narrowest maintainable fix: normalize hosted custom-scheme origins before the existing exact allowlist comparison, without changing wildcard, host-header fallback, or loopback rules.
