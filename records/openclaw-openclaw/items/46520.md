## Summary

Keep open. Current main still uses `URL.origin` directly for browser-origin allowlist matching, and Node reports custom schemes such as `tauri://localhost` and `electron://app` with the opaque origin string `null`, so exact `gateway.controlUi.allowedOrigins` entries still cannot match. The related fix PRs mentioned in the discussion were closed unmerged, and the latest comment correctly describes current behavior.

## Best Possible Solution

Keep this issue open and land a narrow gateway fix: update `parseOrigin` or `checkBrowserOrigin` to reconstruct custom-scheme origins from `protocol` and `host` when `URL.origin === "null"` and a host exists, add accepted/rejected `tauri://` and `electron://` regression cases in `src/gateway/origin-check.test.ts`, and verify the WebSocket handshake path still rejects literal `Origin: null`.
