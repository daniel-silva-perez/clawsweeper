## Summary

Keep open. Current main still normalizes hosted custom-scheme browser origins through `URL.origin`, which Node serializes as `null`, so exact `gateway.controlUi.allowedOrigins` entries like `tauri://localhost` still cannot match. The GitHub context shows open PR #73511 uses closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Allow exact `gateway.controlUi.allowedOrigins` matching for desktop-wrapper custom URL schemes such as `tauri://localhost` and `electron://app` without falling back to wildcard origins.

## Best Possible Solution

Land a narrow gateway origin parser fix that preserves exact `scheme://host` matching only for hosted custom schemes with opaque `URL.origin`, keeps literal `Origin: null` invalid, and adds accepted/rejected `tauri://` and `electron://` regression tests.

## Reproduction Assessment

Yes. The issue reproduces from current main by checking Node URL behavior for `tauri://localhost` / `electron://app` and tracing that value through `parseOrigin` and `allowlist.has(parsedOrigin.origin)`.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix: normalize hosted custom-scheme origins before the existing exact allowlist comparison, without changing wildcard, host-header fallback, or loopback rules.
