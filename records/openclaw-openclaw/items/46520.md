## Summary

Keep open. Current main still normalizes hosted custom-scheme `Origin` values through `URL.origin` to `null` before exact allowlist comparison, and open PR #73511 is already the active closing implementation candidate.

## What This Changes

The issue requests exact `gateway.controlUi.allowedOrigins` matching for custom desktop-app URL schemes such as `tauri://localhost` and `electron://...`.

## Best Possible Solution

Land a narrow gateway origin normalization fix that preserves exact `scheme://host` matching for hosted custom schemes, keeps literal `Origin: null` and malformed origins invalid, keeps unlisted custom origins rejected, and adds focused `tauri://` / `electron://` regression tests.

## Reproduction Assessment

Yes. A read-only Node check reproduces `URL.origin === "null"` for `tauri://localhost` and `electron://app`, and current main passes that value directly into the exact allowlist comparison.

## Solution Assessment

Yes. The best fix is a small change in `src/gateway/origin-check.ts` before the existing allowlist check, with unit coverage, rather than broadening wildcard behavior or Host-header fallback.
