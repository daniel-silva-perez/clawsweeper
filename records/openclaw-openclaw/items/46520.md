## Summary

Keep open. Current main still normalizes custom-scheme browser origins through `URL.origin`, which becomes `null` for `tauri://localhost` and `electron://app`, so explicit `gateway.controlUi.allowedOrigins` entries cannot match. Open PR #73511 already references this issue with closing syntax and is the active implementation candidate; earlier related PRs #46523 and #46555 were closed unmerged.

## What This Changes

The issue asks for exact `gateway.controlUi.allowedOrigins` matching to accept configured custom desktop-app URL schemes such as `tauri://localhost` and `electron://app` without requiring wildcard origins.

## Best Possible Solution

Land PR #73511 or an equivalent narrow gateway parser fix that preserves exact `scheme://host` matching for non-special hosted schemes, keeps existing host-header/local-loopback behavior unchanged, rejects `Origin: null` and malformed origins, and adds focused `tauri://` / `electron://` regression tests.

## Reproduction Assessment

Yes. A read-only Node URL check reproduces `URL.origin === "null"` for `tauri://localhost` and `electron://app`, and current `checkBrowserOrigin` passes that value into exact allowlist comparison on the WebSocket Control UI path.

## Solution Assessment

Yes. The narrowest maintainable solution is to fix custom-scheme origin normalization in `src/gateway/origin-check.ts` with unit coverage, not to broaden wildcard behavior or rely on Host-header fallback.
