## Summary

Keep open. Current main still does not implement the Feishu WebSocket supervisor/stall recovery this PR is trying to add, and #52618 remains the canonical open bug for that behavior. The PR needs a current-main rebase and review, not maintainer cleanup closure.

## Best Possible Solution

Keep this PR open and ask for a rebase against current main. The fix should add an OpenClaw-owned Feishu WebSocket supervisor that observes SDK retry exhaustion, closes the dead client, recreates a fresh WSClient with bounded backoff, preserves bot identity across supervisor cycles, and keeps shutdown cleanup intact. It should include regression tests for retry exhaustion, successful reconnect without false eviction, boot-time failure backoff, abort cleanup, and current redacted logging behavior.

## Risks / Open Questions

- The PR is based on an older Feishu transport shape. Current main has async `createFeishuWSClient()`, proxy support, runtime transport API imports, and redacted error logging, so the PR should be rebased carefully before any merge decision.
- The current SDK version exposes public WSClient lifecycle callbacks. The best implementation should prefer those callbacks where sufficient, and only use private `wsConfig.getWSInstance().readyState` as a documented fallback if the public callbacks cannot distinguish exhausted retries from healthy reconnects.
- Do not regress current secret redaction when integrating this PR; avoid raw `String(err)` in Feishu WebSocket supervisor error logs.
