## Summary

Keep open. Current main still starts one Feishu WSClient, resets only start-time retry state, then parks on abort; it does not implement the supervisor/stall recovery this PR targets. The PR remains a plausible implementation candidate for the still-open Feishu WebSocket recovery bug, but it needs a current-main rebase and maintainer review.

## Best Possible Solution

Keep this PR open and ask for a rebase against current main. The best implementation is an OpenClaw-owned Feishu WebSocket supervisor that detects SDK retry exhaustion after `start()` returns, closes the dead WSClient, recreates a fresh client with bounded backoff, preserves bot identity across supervisor cycles, respects abort cleanup, uses current SDK lifecycle callbacks where possible, and includes regression tests for retry exhaustion, successful reconnect without false eviction, boot-time failure backoff, abort cleanup, bot identity preservation, and redacted logging.

## Risks / Open Questions

- The PR is based on an older Feishu transport shape. Current main has async `createFeishuWSClient()`, proxy support, local runtime API imports, and redacted WebSocket error logging, so integration needs a careful rebase.
- The current SDK version exposes public WSClient lifecycle callbacks. A refreshed fix should prefer public callbacks where they are sufficient and only use private `wsConfig.getWSInstance().readyState` as a documented fallback.
- Do not regress current secret redaction when integrating the supervisor; avoid raw `String(err)` in Feishu WebSocket supervisor error paths.
