## Summary

Keep open. Current main still has the fixed tool-schema and prompt overhead at the center of the report, and the discussion treats this as the canonical multi-step performance tracker with only partial adjacent work completed or proposed.

## What This Changes

The issue requests reducing fixed per-session tool schema token overhead through channel-aware schema filtering, lazy or slim tool loading, shorter tool descriptions, and splitting or pruning broad tool schemas.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes: remove or make optional duplicated Tooling text, avoid emitting definitions when no effective tools remain, prune high-footprint message fields by active channel and capability without breaking cross-channel cron or isolated runs, shorten long descriptions where behavior stays covered, and design a compatible slim or lazy schema mode for plugins and declarative tools.
