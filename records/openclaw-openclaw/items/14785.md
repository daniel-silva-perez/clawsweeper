## Summary

Keep open. Current main still contains the core fixed-overhead paths described in the report, and the discussion continues to treat this as the canonical multi-step performance tracker rather than a completed or superseded item.

## What This Changes

The issue asks OpenClaw to reduce fixed per-session tool-schema overhead through channel-aware schema pruning, lazy or slim tool loading, shorter tool descriptions, and splitting or compressing high-footprint tools.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes: remove or make optional duplicated Tooling text, avoid emitting tool definitions when no effective tools remain, prune high-footprint message fields by active channel and capability, shorten long descriptions where behavior remains covered elsewhere, and design a compatible slim or lazy schema mode for plugins and declarative tools.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path remains: inspect /context detail or the current prompt/report code, then compare the separate tool-schema JSON and prompt Tooling surfaces while current main still preserves full descriptions/parameters and broad message/cron schemas.

## Solution Assessment

No single automated patch is the best way to solve this issue. The maintainable path is maintainer-sequenced incremental work because the remaining scope crosses prompt policy, provider contracts, channel/plugin schemas, cron compatibility, and lazy/slim schema design.
