## Summary

Keep open. Current main still has the fixed tool-schema overhead paths described in the report, and the discussion treats this as the canonical multi-step performance tracker with only workarounds or partial adjacent proposals so far.

## What This Changes

The issue requests reducing fixed per-session tool schema token overhead through channel-aware message schema pruning, lazy or on-demand tool loading, shorter tool descriptions, and smaller or split tool definitions.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes: remove or make optional duplicated Tooling text, avoid emitting tool definitions when no effective tools remain, prune high-footprint message fields by active channel and capability, shorten long descriptions where behavior remains covered, and design a compatible slim or lazy schema mode for plugins and declarative tools.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path remains: inspect a session with effective tools or /context detail, then compare the separate Tool schemas JSON and Tool list prompt text while current main still sends full descriptions/parameters and broad message/cron schemas.

## Solution Assessment

No single automated patch is the best way to solve this. The maintainable path is maintainer-sequenced incremental work because the remaining scope crosses prompt policy, provider contracts, channel/plugin schemas, cron compatibility, and lazy/slim schema design.
