## Summary

Keep open. Current main still has the fixed tool-schema overhead paths described by the report, and the discussion treats this as the canonical multi-step performance tracker with only partial/workaround-level progress so far.

## What This Changes

The issue requests reducing fixed per-session tool schema overhead through channel-aware message schema pruning, lazy or slim tool loading, shorter descriptions, and splitting high-footprint tools.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes: remove or make optional duplicated Tooling text, avoid emitting tool definitions when no effective tools remain, prune high-footprint message fields by active channel and capability, shorten long descriptions where behavior remains covered, and design a compatible slim or lazy schema mode for plugins and declarative tools.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path remains: inspect or run a session with effective tools and `/context detail`; current main still builds broad message schemas, includes full tool descriptions/parameters in model-facing tool definitions, and emits tool summaries in the system prompt.

## Solution Assessment

No single automated repair is the best way to solve the issue. The maintainable path is maintainer-sequenced incremental work because the remaining scope crosses prompt policy, provider contracts, channel/plugin schemas, cron compatibility, and lazy/slim schema design.
