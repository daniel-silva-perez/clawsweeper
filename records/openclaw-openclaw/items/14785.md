## Summary

Keep open. Current main still has the fixed per-run tool-schema overhead paths described in the report, and the discussion treats this as the canonical multi-step performance tracker rather than a completed or superseded item.

## What This Changes

The issue requests reducing fixed per-session tool-schema token overhead through channel-aware message schema pruning, lazy or stubbed tool loading, shorter model-visible descriptions, and smaller high-footprint tool schemas.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes: remove or make optional duplicated Tooling text, avoid unnecessary tool payloads, prune high-footprint message fields by active channel and capability without breaking cross-channel cron or isolated runs, shorten long descriptions where tests preserve behavior, and design a compatible slim or lazy schema mode for plugins and declarative tools.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path remains: inspect /context detail/list reporting plus the current prompt, message-tool, cron-tool, and tool-definition code paths to confirm the model-facing overhead still exists. A live Telegram measurement would refine the savings estimate but is not required to verify the remaining fixed overhead.

## Solution Assessment

No single automated patch is the best way to solve the issue. The requested direction is valid, but the maintainable solution is maintainer-sequenced incremental work because schema pruning and lazy loading must preserve provider contracts, plugin/declarative tools, cron delivery, isolated runs, and prompt-cache behavior.
