## Summary

Keep this PR open. Current main still buffers suppressed NO_REPLY lead fragments during delta handling, then finalization explicitly disables lead-fragment suppression and can emit a lifecycle-ending fragment such as NO_ as the final Gateway chat message. The PR is still the narrow fix candidate, but the latest branch approach needs maintainer review because its suppression provenance is keyed by caller-provided clientRunId/idempotencyKey.

## Best Possible Solution

Keep this PR open as the canonical narrow Gateway fix candidate. Revise or confirm the suppression provenance so it is scoped to session/source run or another server-owned identity, keep the lifecycle-ending NO_ or NO_RE regression coverage plus the intentional uppercase NO preservation test, then run the targeted Gateway test and the normal changed gate before merge.
