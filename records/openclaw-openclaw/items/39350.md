## Summary

Keep this PR open. Current main still has the Gateway final-message gap: a suppressed `NO_REPLY` lead fragment can be stored in the chat buffer, then lifecycle finalization reprojects it with lead-fragment suppression disabled and emits it as the final assistant message. The PR remains a relevant implementation candidate, but its latest branch still needs maintainer review for the `clientRunId`-scoped suppression state noted in the security discussion.

## Best Possible Solution

Keep this PR open as the canonical Gateway implementation candidate. Adjust the suppression provenance so it is session/run scoped rather than keyed only by client-supplied `clientRunId`, keep regression coverage for lifecycle-end `NO_`/`NO_RE` suppression and intentional uppercase `NO` preservation, then run the normal Gateway changed gate before merge.
