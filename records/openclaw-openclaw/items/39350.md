## Summary

Keep this PR open. Current main still buffers suppressed NO_REPLY lead fragments and then finalizes with lead-fragment suppression disabled, so a lifecycle-ending fragment such as `NO_` or `NO_RE` can still be emitted in the final chat payload. The PR is the relevant narrow Gateway fix candidate, but the latest PR-head suppression state needs maintainer review because it is keyed only by `clientRunId`, which comes from the caller-provided `idempotencyKey`.

## Best Possible Solution

Keep this PR open as the canonical narrow Gateway fix candidate. Review or revise the suppression provenance so it cannot collide across sessions, keep the lifecycle-ending `NO_`/`NO_RE` regression coverage and the intentional uppercase `NO` preservation case, then run the Gateway changed gate before merge.
