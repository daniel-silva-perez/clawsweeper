## Summary

Keep this PR open. Current main still has the Gateway final-message gap the PR targets: suppressed `NO_REPLY` lead fragments can remain buffered, then finalization reprojects with lead-fragment suppression disabled and can emit a partial fragment as a final assistant message. The live PR head is still the relevant implementation candidate, but it needs maintainer review of the new `clientRunId`-scoped suppression state before merge.

## Best Possible Solution

Keep this PR open as the canonical Gateway fix candidate. Review or adjust the suppression provenance so it is scoped to the actual session/run or clearly justified by existing idempotency semantics, preserve intentional uppercase `NO` replies, keep lifecycle-end `NO_`/`NO_RE` regression coverage, then run the normal Gateway changed gate before merge.

## Risks / Open Questions

- Current main can still leak a buffered `NO_REPLY` lead fragment in Gateway final chat payloads when the lifecycle ends before the full control token arrives.
- The PR head keys new suppression provenance by `clientRunId`; because chat and agent runs derive that from client-provided idempotency keys, maintainers should either scope the state by session/run or prove the existing global idempotency contract prevents cross-session suppression side effects.
- The Control UI defense-in-depth filter only drops exact `NO_REPLY` finals; it does not cover partial final payloads such as `NO_` or `NO_RE` from the Gateway.
