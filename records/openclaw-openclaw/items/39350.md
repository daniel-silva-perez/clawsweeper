## Summary

Keep this PR open. Current main still has the Gateway final-message gap: suppressed `NO_REPLY` lead fragments can remain buffered, then finalization disables lead-fragment suppression and can emit the fragment as a visible final assistant message. The PR head is still a relevant implementation candidate, with a maintainer-pushed repair preserving contributor credit, but it needs maintainer review of the `clientRunId`-scoped suppression state before merge.

## Best Possible Solution

Keep this PR open as the canonical Gateway implementation candidate. Review or adjust the suppression provenance so it is scoped to the actual session/run rather than only a client-supplied idempotency key, preserve intentional uppercase `NO` replies, keep lifecycle-end `NO_`/`NO_RE` regression coverage, then run the normal Gateway changed gate before merge.
