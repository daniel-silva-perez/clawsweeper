## Summary

Keep open. Current main still accepts Google Chat webhook payloads with a string event type, but the registered processor returns immediately for anything other than MESSAGE. Google Chat has outbound reaction actions, but no inbound reaction event shape, reaction notification config, docs, or system-event/plugin routing. The prior ClawSweeper comment is still accurate: #9764 covers outbound/user-OAuth capability and #17840 covers broader reaction-triggered agent turns; neither supersedes this Google Chat ingress gap.

## Best Possible Solution

Keep this issue open and implement the Google Chat-owned ingress path: add typed reaction-created/deleted payload parsing for the actual Google delivery format, run it through Google Chat access policy, enqueue a normalized passive system event or typed plugin hook with targeted tests, and document the required Google Chat subscription/config. Keep #9764 scoped to outbound user OAuth/reaction creation and #17840 scoped to cross-channel reaction-triggered turns.

## Risks / Open Questions

- The exact Google delivery surface should be confirmed during implementation because Google Chat app interaction events and Workspace Events/SpaceEvent reaction payloads use different naming/shapes than the issue title's REACTION_ADDED/REACTION_REMOVED wording.
- Inbound reaction handling must enforce the same Google Chat DM, group allowlist, sender, and mention-gating policies before enqueueing any system event; several other channels have security-sensitive reaction ingress checks.
- Product shape should stay aligned with #17840: passive inbound system events can be implemented in the Google Chat plugin, while opt-in reaction-triggered agent turns remain a broader cross-channel policy decision.
