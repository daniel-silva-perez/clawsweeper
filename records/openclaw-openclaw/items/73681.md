## Summary

Keep open. Current main still authenticates and dispatches Google Chat non-message webhook payloads, then drops them in the Google Chat event processor because only `MESSAGE` events are routed; there is still no inbound reaction payload model, hook, system-event path, or docs coverage. Related #9764 and #17840 are adjacent scope, not superseding fixes for this Google Chat ingress gap.

## What This Changes

Request Google Chat inbound reaction webhook handling so reaction add/remove events reach OpenClaw instead of being acknowledged and dropped before plugins or system events can observe them.

## Best Possible Solution

Add a Google Chat-owned inbound reaction path that parses reaction created/deleted payloads, applies the existing authenticated webhook target and sender/space access policy, normalizes accepted reactions into passive system events or a documented typed hook, adds focused tests, and documents any required Google Chat event subscription setup while leaving outbound OAuth work to #9764 and reaction-triggered turns to #17840.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: an authenticated non-`MESSAGE` Google Chat event with a valid `space` is parsed and dispatched by `monitor-webhook.ts`, then returned by `processGoogleChatEvent` without downstream routing.

## Solution Assessment

Yes for the channel-owned ingress direction. The safest first fix is passive Google Chat reaction system-event handling through existing auth/access seams; a dedicated `googlechat.onReaction` hook should only be added if maintainers intentionally want that public plugin API.
