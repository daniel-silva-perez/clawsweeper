## Summary

Keep open. Current main still has the Google Chat inbound-reaction gap: the bundled Google Chat channel only starts an HTTP interaction webhook, accepts/dispatches only the current GoogleChatEvent shape, and the event processor returns before routing anything unless the event type is MESSAGE. The repo has outbound Google Chat reaction tools, but no inbound reaction payload model, reaction notification config/docs, Workspace Events/Pub/Sub ingress, or normalized system-event/hook path. Related #9764 and #17840 remain adjacent, not superseding trackers.

## Best Possible Solution

Keep this issue open and implement the Google Chat-owned inbound reaction path: add support for the real Google Workspace Events/SpaceEvent reaction created/deleted and batch payloads, route them through Google Chat access controls, normalize them into passive system events or a typed plugin hook, add focused tests for allowed and blocked reactions, and document the required Google Chat event subscription setup. Keep #9764 scoped to user OAuth/outbound reaction creation and #17840 scoped to opt-in reaction-triggered agent turns.

## Risks / Open Questions

- The issue title uses REACTION_ADDED / REACTION_REMOVED, but current Google docs name Workspace reaction events as google.workspace.chat.reaction.v1.created/deleted and batch variants; implementation should test against the exact upstream payload shape.
- Inbound reaction handling must reuse Google Chat DM/group allowlists, groupPolicy, sender allowlists, and mention-gating decisions so unauthorized users cannot inject passive system events into sessions.
- Google Workspace Events subscriptions deliver CloudEvents through Pub/Sub, which has a different envelope and auth model from the current HTTP interaction webhook; implementing only a branch in monitor-webhook.ts may not be sufficient.
