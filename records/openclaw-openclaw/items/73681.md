## Summary

Keep open. Current main still accepts authenticated non-message Google Chat webhook events, but the Google Chat event processor returns immediately for anything other than MESSAGE, so REACTION_ADDED and REACTION_REMOVED never reach session routing, system-event handling, or a plugin-visible path. The related OAuth and generic reaction-turn trackers are adjacent, not replacements for this Google Chat ingress bug.

## What This Changes

The issue requests inbound Google Chat reaction webhook support so REACTION_ADDED and REACTION_REMOVED events are parsed, routed, exposed to plugin or system-event handling, and documented.

## Best Possible Solution

Add Google Chat-owned inbound reaction handling: type and parse the reaction payload, apply the existing sender/space access policy, resolve the affected session, emit a durable reaction system event or existing plugin-visible channel event consistent with other channels, and document inbound semantics without folding in outbound OAuth work or the broader reaction-triggered-turn feature.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send a valid authenticated Google Chat webhook event with type REACTION_ADDED or REACTION_REMOVED and a space: the webhook parser can return 200, then processGoogleChatEvent exits before any routing or event emission.

## Solution Assessment

Yes, the best fix is a narrow Google Chat plugin change using existing channel reaction/system-event patterns. A new googlechat.onReaction hook should only be added if the existing plugin-visible contracts cannot represent the event cleanly.
