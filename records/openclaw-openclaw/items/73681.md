## Summary

Keep open. Current main still accepts authenticated Google Chat non-message webhook payloads and then drops them at the Google Chat event processor because only `MESSAGE` events are routed; there is no inbound reaction event model, config, hook, system-event path, or docs entry that would satisfy the requested behavior. Related #9764 and #17840 are adjacent but do not supersede this Google Chat ingress gap.

## What This Changes

Request Google Chat inbound reaction webhook handling so reaction add/remove events reach the OpenClaw runtime instead of being dropped before plugins or system events can observe them.

## Best Possible Solution

Add a Google Chat-owned inbound reaction path: parse reaction created/deleted payloads, route them through the existing authenticated webhook target and sender/space access policy, normalize them into passive reaction system events or a documented typed hook, add focused Google Chat tests, and document the required Google Chat event subscription/configuration while leaving outbound OAuth scope work in #9764 and cross-channel reaction-triggered turns in #17840.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: any authenticated non-`MESSAGE` Google Chat event with a valid `space` is parsed and dispatched by `monitor-webhook.ts`, then dropped by the `eventType !== "MESSAGE"` return in `monitor.ts`; no live Google Chat tenant was needed to confirm that code path.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Google Chat plugin, reusing existing webhook auth, route resolution, access checks, and system-event/runtime seams; a new core API should only be added if a typed plugin hook is intentionally chosen and documented.
