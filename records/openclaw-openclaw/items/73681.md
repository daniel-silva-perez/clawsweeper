## Summary

Keep open. Current main still drops Google Chat non-message inbound events before access checks, session routing, system-event emission, or any plugin-visible path, so the reported inbound reaction gap is still present. The exact Google event contract needs care because current Google docs describe reaction subscription events as Workspace Events rather than the issue’s `REACTION_ADDED` / `REACTION_REMOVED` names. 

## What This Changes

Add inbound Google Chat reaction handling so user-added and user-removed reactions can be parsed, routed, surfaced to plugins or system events, and documented.

## Best Possible Solution

Implement Google Chat-owned inbound reaction support against the real Google Workspace Chat event shape: parse created/deleted and batch reaction payloads, derive the affected space/message/user/emoji, apply the existing Google Chat access model or a reaction-specific equivalent, resolve the session, emit a durable session-scoped reaction system event consistent with other channels, and document the inbound subscription requirements without folding in outbound OAuth work or generic reaction-triggered turns.

## Reproduction Assessment

Yes for the current OpenClaw code path: an authenticated Google Chat webhook body with a valid `space` and a non-`MESSAGE` reaction-like event type can parse and then exits at `extensions/googlechat/src/monitor.ts:59` before routing. A live end-to-end reproduction still needs the real Google Workspace Events payload, whose documented reaction event names are `google.workspace.chat.reaction.v1.created/deleted`.

## Solution Assessment

Unclear on the issue’s proposed `googlechat.onReaction` hook as the first step. The narrowest maintainable fix is to use existing channel system-event/plugin-visible contracts first, adding a new hook only if those contracts cannot represent Google Chat reaction metadata cleanly.
