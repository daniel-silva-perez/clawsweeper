## Summary

Keep open. Current main still only processes Google Chat webhook events when the event type is MESSAGE, while the Google Chat plugin has outbound reaction actions but no inbound reaction event model, config, docs, or system-event routing. The related #9764 tracks user OAuth for the bot creating reactions/media uploads, and #17840 is a broader opt-in reaction-triggered-turn policy; neither supersedes this Google Chat ingress gap.

## Best Possible Solution

Keep this issue open and implement the Google Chat-owned ingress path: extend the Google Chat event types/parser for reaction created/deleted webhook payloads, route them through the same access policy as messages, enqueue a normalized reaction system event or typed plugin hook with tests, and document the Google Chat inbound reaction configuration. Keep #9764 scoped to user OAuth/outbound reaction creation and #17840 scoped to cross-channel reaction-triggered turns.

## Risks / Open Questions

- Reaction ingress must enforce the same Google Chat DM/group allowlist and mention-gating policy before enqueueing any system event; other channels have had security hardening in this area.
- The exact Google delivery payload should be verified during implementation because current upstream discovery names reaction events as SpaceEvent reaction created/deleted types, while the issue title uses REACTION_ADDED/REACTION_REMOVED.
- Product shape should align with #17840: passive system event support can be fixed in the Google Chat plugin first, while opt-in reaction-triggered agent turns remain a broader cross-channel policy decision.
