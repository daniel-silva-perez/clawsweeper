## Summary

Keep open. Current main still has the Google Chat ingress gap described in the issue: authenticated non-message webhook payloads can be parsed and acknowledged, but the channel processor drops every event whose type is not `MESSAGE`, and there is no Google Chat inbound reaction model, hook, config, test, or docs path that would satisfy the request. The related OAuth and cross-channel reaction-turn issues are adjacent, not superseding.

## What This Changes

The issue requests Google Chat inbound reaction webhook events be routed into a normalized reaction event or plugin-facing hook instead of being silently acknowledged and ignored.

## Best Possible Solution

Add a Google Chat-owned inbound reaction path after existing webhook authentication: parse the actual reaction event payload, preserve sender/space access checks, resolve the Google Chat route, enqueue a passive reaction system event or expose a documented typed hook, and add focused tests/docs while leaving outbound OAuth work in #9764 and reaction-triggered agent turns in #17840.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a valid non-`MESSAGE` event with `space` passes webhook parsing and is dispatched with a 200 response, then `processGoogleChatEvent` returns before any downstream route or event handling.

## Solution Assessment

Yes, with one implementation caveat. The narrowest maintainable fix belongs in the Google Chat plugin using existing webhook auth, route resolution, and access policy; only the actual Google payload shape should be verified before choosing the parser fields and event labels.
