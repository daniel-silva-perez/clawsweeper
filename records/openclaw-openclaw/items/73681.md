## Summary

Keep open. Current main still accepts authenticated non-message Google Chat webhook events but the Google Chat processor immediately returns for every event type except MESSAGE, so inbound reaction events do not reach session/system-event or plugin-visible handling; the related OAuth and generic reaction-turn issues track adjacent work, not this Google Chat ingress gap.

## What This Changes

The issue requests inbound Google Chat REACTION_ADDED and REACTION_REMOVED webhook events to be routed into OpenClaw instead of being dropped by MESSAGE-only Google Chat handling.

## Best Possible Solution

Add Google Chat-owned inbound reaction support: parse and type the reaction payload, apply the appropriate sender/space access checks, route to the resolved session, emit a durable reaction system event or existing plugin-visible hook path consistent with other channels, and document the inbound semantics while leaving outbound OAuth work and generic reaction-triggered turns to their existing trackers.

## Reproduction Assessment

Yes. A current-main code-level reproduction is to pass a valid authenticated Google Chat webhook event with type REACTION_ADDED or REACTION_REMOVED and a space: the webhook handler can return 200, then `processGoogleChatEvent` returns before any routing or event emission.

## Solution Assessment

Yes. The narrow maintainable fix is in the Google Chat plugin, using existing channel reaction/system-event patterns; adding a new `googlechat.onReaction` SDK hook would be broader API work and should only happen if existing plugin-visible contracts are insufficient.
