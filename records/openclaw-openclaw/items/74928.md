## Summary

Keep open. Current main still has a phase-only visible assistant text contract: signatures, provider replay/schema, UI extraction, and channel delivery paths do not parse or enforce a public/internal assistant audience boundary. The related commentary/phase leak reports are useful context, but they do not supersede this broader delivery-boundary contract request.

## What This Changes

Add an explicit public/internal assistant audience contract so internal assistant text and exact silent sentinels are not selected as user-visible output while preserving legacy phase-only compatibility where needed.

## Best Possible Solution

Define one shared assistant audience contract in the message/signature layer and provider-facing schemas, carry it through stream parsing, replay conversion, transcript/history projection, and channel delivery, then enforce internal suppression and invalid-audience fail-closed behavior at user-visible extraction boundaries while preserving documented legacy phase fallback.

## Reproduction Assessment

Yes. A synthetic assistant message with textSignature JSON containing v:1, id, phase:"final_answer", and audience:"internal" is still selected by current main because the parser ignores audience and visible extraction selects final_answer text by phase only.

## Solution Assessment

Unclear as a ready implementation because this issue proposes a direction rather than a patch, but the direction is sound. The safest solution is a maintainer-reviewed cross-boundary contract rather than a delivery-only sanitizer.
