## Summary

Keep open. Current main still has phase-aware assistant text handling but no explicit public/internal assistant audience contract in the shared signature parser, OpenResponses schema, replay conversion, UI projection, or message-end delivery boundary. The reporter's sanitized package-artifact evidence matches the current source-level gap, and the related leak/phase reports are adjacent rather than superseding this broader contract request.

## What This Changes

Request an explicit public/internal assistant audience contract that suppresses internal assistant text at visible delivery boundaries, fails closed on invalid audience values, suppresses exact silent sentinels, and preserves legacy phase-only fallback where needed.

## Best Possible Solution

Define a single AssistantAudience contract in the shared message/signature layer, carry it through provider stream parsing, replay conversion, transcript projection, OpenResponses surfaces, UI history extraction, and channel delivery, then enforce public-only selection at user-visible boundaries while preserving an explicit legacy phase-only fallback.

## Reproduction Assessment

Yes. A source-level synthetic assistant text block with textSignature JSON containing v: 1, an id, phase: "final_answer", and audience: "internal" follows the current parser/extractor path because audience is ignored and final_answer text is selected as visible.

## Solution Assessment

Unclear as an immediate patch, but the requested direction is sound. The safest solution is a maintainer-reviewed cross-boundary contract rather than adding another late delivery scrubber.
