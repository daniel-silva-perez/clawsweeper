## Summary

Keep open. Current main still has phase-aware assistant text handling but no explicit public/internal assistant audience contract in the shared signature parser, OpenResponses schema, replay conversion, UI projection, or message-end delivery boundary. The reporter's sanitized package-artifact evidence and the existing ClawSweeper review both match the current source-level gap, and the related leak/phase reports remain adjacent rather than superseding this broader contract request.

## What This Changes

The issue requests an explicit public/internal assistant audience boundary for assistant text signatures and visible delivery extraction.

## Best Possible Solution

Define one assistant audience contract in the shared message/signature layer, carry it through provider stream parsing, replay conversion, transcript projection, OpenResponses surfaces, UI history extraction, and channel delivery, then enforce public-only selection at user-visible boundaries while preserving an explicit legacy phase-only fallback.

## Reproduction Assessment

Yes. A source-level synthetic assistant message with textSignature JSON containing v: 1, an id, phase: "final_answer", and audience: "internal" reproduces the gap because current parsing ignores audience and visible extraction selects final_answer text.

## Solution Assessment

Unclear as an immediately safe patch, but the requested direction is sound. The safest solution is a maintainer-reviewed cross-boundary contract rather than adding one more late delivery scrubber.
