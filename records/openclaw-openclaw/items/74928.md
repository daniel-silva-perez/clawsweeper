## Summary

Keep open. Current main still implements a phase-only visible-text contract and does not parse, expose, or enforce a public/internal assistant audience boundary across shared signatures, provider replay/schema, UI projection, or channel delivery. The reporter's sanitized artifact evidence and the prior ClawSweeper review match the current source-level gap, while the linked phase/commentary leak reports are related but do not supersede this broader delivery-boundary contract request.

## What This Changes

Add an explicit public/internal assistant audience contract for visible assistant delivery so internal assistant text and exact silent tokens are not selected as user-visible output while preserving legacy phase fallback where needed.

## Best Possible Solution

Define a single shared assistant audience contract, likely public/internal, in the message/signature layer and provider-facing schemas; propagate it through stream parsing, replay conversion, transcript/history projection, and channel delivery; enforce internal suppression and invalid-audience fail-closed behavior at visible extraction boundaries while preserving documented legacy phase-only fallback for older data.

## Reproduction Assessment

Yes. A synthetic assistant message with textSignature JSON containing v:1, id, phase:"final_answer", and audience:"internal" will still be selected by current source because the parser ignores audience and the visible extractor selects by phase only.

## Solution Assessment

Unclear as a ready patch because there is no proposed implementation here, but the requested direction is the right boundary. The safer path is a maintainer-approved cross-boundary contract instead of only adding another delivery sanitizer.
