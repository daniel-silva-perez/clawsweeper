## Summary

Keep open. Current main and the v2026.4.29 tag still use phase-only assistant text signatures and delivery filtering, so the requested public/internal assistant audience boundary remains missing and security-sensitive.

## What This Changes

Requests an explicit public/internal assistant audience contract for visible assistant text parsing, replay/projection, and channel delivery, with fail-closed handling for invalid audience values and silent sentinels.

## Best Possible Solution

Define one shared assistant audience contract, propagate it through provider stream parsing, replay conversion, transcript/history projection, OpenResponses surfaces, UI extraction, and channel delivery, then enforce public-only visible selection while preserving an explicit legacy phase-only fallback.

## Reproduction Assessment

Yes. A synthetic assistant text block with textSignature JSON containing v: 1, an id, phase: "final_answer", and audience: "internal" follows the current parser/extractor path because audience is ignored and final_answer text is selected as visible.

## Solution Assessment

Unclear as a quick patch, but the requested direction is the maintainable one. The safest fix is a maintainer-reviewed shared contract across parsing, projection, and delivery rather than adding another delivery-only text scrubber.
