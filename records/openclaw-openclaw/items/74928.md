## Summary

Keep open. Current main still has phase-aware visible-text handling but no explicit public/internal assistant audience contract in the shared signature parser, provider stream encoders, replay/conversion path, or delivery extraction helpers, so the core request remains valid and is not superseded by the related phase/commentary leak reports.

## What This Changes

The issue requests an explicit public/internal assistant audience contract at visible-text extraction and channel delivery boundaries, including fail-closed handling for unknown audience values and silent suppression for exact `NO_REPLY` text.

## Best Possible Solution

Define a single assistant audience contract in the shared message/signature layer, carry it through provider stream parsing and replay conversion, and enforce it at all user-visible extraction and delivery boundaries while preserving documented legacy phase-only fallback for older transcripts.

## Reproduction Assessment

Yes. A source-level synthetic assistant message with `textSignature: { v: 1, id, phase: "final_answer", audience: "internal" }` is enough to reproduce the gap on current main because the parser ignores `audience` and the visible extractor selects `final_answer` text.

## Solution Assessment

Unclear as a ready implementation, but the requested direction is sound. The safest solution is a maintainer-reviewed cross-boundary contract rather than a narrow sanitizer-only patch, because public/internal audience needs to be encoded, parsed, persisted, replayed, and enforced consistently.
