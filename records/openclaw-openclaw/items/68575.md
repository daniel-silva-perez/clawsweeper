## Summary

Keep open. The linked Telegram/context-overflow problem is real and current main does not fully subsume this PR’s thrown-overflow reset path, but the proposed catch-branch reset is too broad to merge as written because it can discard a live session from an aggregate/mixed failure whose formatted message merely contains a context-overflow attempt.

## What This Changes

The PR updates `runAgentTurnWithFallback` to reset the active session after thrown context-overflow failures and final embedded overflow errors with no payload, with regression tests and mocked overflow classifiers.

## Best Possible Solution

Land a rebased, narrow runtime change that treats a pure unrecoverable context-overflow signal as session-poisoning, resets through the existing callback, reuses `buildContextOverflowRecoveryText`, and adds focused tests for thrown overflow, mixed fallback-summary no-reset, and embedded no-payload behavior.

## Reproduction Assessment

Yes. A focused unit test can mock `runWithModelFallback` to reject with an aggregate `FallbackSummaryError` whose message includes one context-overflow attempt plus a different terminal cause, then assert the reset callback is not called.

## Solution Assessment

No. Reusing the existing reset callback is the right direction, but this implementation should gate reset on a pure overflow/compaction classification rather than broad formatted-message matching.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

- **[P1] Restrict resets to pure overflow failures:** `src/auto-reply/reply/agent-runner-execution.ts:1464-1469`

Gate this on a pure overflow/compaction signal before calling `resetSessionAfterCompactionFailure`.
