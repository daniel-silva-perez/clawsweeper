## Summary

Keep this PR open. Current main still lacks the arbitrary recursive skill discovery proposed here and still has no generic gateway startup warning for a selected model provider with no usable auth; #69663 is a narrower open follow-up for the recursive skill subset, but it does not cover the auth-warning half. The PR should not merge as written because the diff still has correctness and security-sensitive startup/traversal risks.

## Best Possible Solution

Keep this PR open for maintainer rework or contributor split. The skill-scanning half should align with or defer to #69663 and preserve bounded traversal, symlink/realpath containment, watcher behavior, docs, and regression tests for workspace, managed, extra-dir, and plugin-bundle skill roots. The startup-auth warning should be a separate best-effort diagnostic that uses the same non-throwing, read-only runtime auth contract as `hasAvailableAuthForProvider`, or be explicitly dropped if maintainers decide the current Anthropic-default gating fully resolves the original user symptom.

## Risks / Open Questions

- Closing this PR now would lose the remaining generic startup-auth warning work, and the recursive scanning work is only partially overlapped by the focused open #69663 path.
- Merging the PR as written could put write-capable auth-store I/O and uncaught auth-store failures on the gateway startup path, turning a warning into a possible startup failure.
- Merging the auth warning as written could produce false warnings for valid auth sources supported by current runtime auth resolution, including custom provider keys, synthetic local provider auth, Bedrock/AWS SDK auth, and broader profile resolution.
- Merging the recursive scanner as written could traverse large non-skill trees without preserving `maxCandidatesPerRoot` as a scanned-directory budget, and its symlink traversal needs to preserve current realpath containment guarantees.
- Security review found no new dependency, CI workflow, lockfile, install script, or publishing metadata changes in the PR diff, but the filesystem traversal and startup auth-store access are security-sensitive runtime paths that need rework and tests.
