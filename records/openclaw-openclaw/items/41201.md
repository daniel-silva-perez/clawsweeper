## Summary

Keep open. Current main handles the local workspace-avatar path, but the reported remote `http(s)` avatar case remains unresolved: OpenClaw still accepts remote avatar values in CLI/config while Control UI intentionally filters them out under its CSP/privacy policy.

## What This Changes

The issue requests that avatars configured with `openclaw agents set-identity --avatar`, including remote URLs and local paths, render correctly in the Control UI.

## Best Possible Solution

Settle the avatar contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, MIME, size, redirect, caching, and failure-feedback rules, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config schema/help, docs, validation, and UI diagnostics while preserving local workspace, data, and blob avatar support.
