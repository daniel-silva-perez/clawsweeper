## Summary

Keep open. Current main appears to cover the workspace-local avatar path through the same-origin `/avatar/<agentId>` route, but the reported remote `http(s)` avatar case is still a real contract mismatch: CLI/config continue accepting remote avatar values while Control UI, CSP, docs, and tests intentionally block remote image rendering for security and privacy reasons.

## What This Changes

The issue requests that agent avatars configured with `openclaw agents set-identity --avatar <url/path>` render in Control UI instead of showing a broken image, especially for external `http(s)` URLs and local workspace paths.

## Best Possible Solution

Resolve the contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, MIME, size, caching, redirect, and failure-feedback behavior, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config schema/docs, validation, and UI diagnostics while preserving the current local avatar route.
