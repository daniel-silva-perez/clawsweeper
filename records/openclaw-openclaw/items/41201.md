## Summary

Keep open. Current main handles local workspace avatars through the authenticated same-origin avatar route, but the remote `http(s)` avatar case remains a real contract mismatch: CLI/config still accept remote avatar values while Control UI deliberately filters and CSP-blocks remote image rendering for privacy and security reasons. The related fix PR #41239 is closed and unmerged, and later hardening PRs make direct remote rendering a maintainer product/security decision rather than a safe cleanup close.

## What This Changes

Setting an agent avatar with `openclaw agents set-identity --avatar` should display in Control UI for configured remote URLs and local workspace paths instead of showing a broken image.

## Best Possible Solution

Settle the avatar contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, MIME, size, redirect, cache, and failure-feedback rules, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config schema/help, docs, validation, and UI diagnostics while preserving local workspace, data, and blob avatar support.

## Reproduction Assessment

Yes for the remaining remote-avatar problem: current source and tests show `http(s)` avatar values are accepted, emitted as remote metadata, then rejected by the Control UI render gate and CSP. The local workspace-path side no longer has a current-main reproduction path from the inspected route, resolver, UI, and tests.

## Solution Assessment

Unclear. The closed PR #41239 approach of redirecting to arbitrary remote avatar URLs is not the best safe fix under the current CSP/privacy posture; maintainers should first decide between a controlled same-origin proxy/cache and an explicit unsupported-remote contract.
