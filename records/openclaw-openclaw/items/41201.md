## Summary

Keep open. Current main has a working authenticated same-origin path for local workspace avatars, but the reported remote `http(s)` avatar case remains a real contract mismatch: the CLI/config path still accepts remote avatar values while Control UI intentionally refuses to render or fetch them under the current CSP/privacy posture.

## What This Changes

The issue asks for avatars configured with `openclaw agents set-identity --avatar`, including external URLs and local workspace paths, to display in the Control UI instead of showing a broken image.

## Best Possible Solution

Settle the avatar contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, MIME, size, redirect, caching, and failure-feedback rules, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config schema/help, docs, validation, and UI diagnostics while preserving local workspace, data, and blob avatar support.

## Reproduction Assessment

Yes for the remaining remote-avatar problem: current source and tests show `http(s)` avatar values are accepted, emitted as remote metadata, then rejected by the Control UI render gate and CSP. The local workspace-path side no longer has a clear current-main reproduction path from the inspected resolver, route, UI, docs, and tests.

## Solution Assessment

Unclear. The closed PR #41239 approach of redirecting to arbitrary remote avatar URLs is not the safest fit under the current CSP/privacy posture; maintainers should first choose between a controlled same-origin proxy/cache and an explicit unsupported-remote contract.
