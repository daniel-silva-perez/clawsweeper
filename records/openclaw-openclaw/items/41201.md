## Summary

Keep open. Current main appears to handle workspace-local Control UI avatars, but the reported remote `http(s)` avatar path remains a real contract mismatch: CLI/config still accept remote avatar values while Control UI intentionally refuses to render or fetch them under the current CSP/privacy posture.

## What This Changes

The issue asks for agent identity avatars set through `openclaw agents set-identity --avatar <url/path>` to display in Control UI instead of showing a broken image or not-found response.

## Best Possible Solution

Settle the Control UI avatar contract deliberately: either implement remote avatars as authenticated same-origin content with documented security limits, or make remote avatars clearly unsupported for Control UI by aligning CLI help, config docs/schema expectations, validation, and UI diagnostics while preserving the current local, data, and blob avatar support.

## Reproduction Assessment

Yes for the remaining remote-avatar problem: current source and tests show remote avatar values are accepted, emitted as metadata, then dropped by the Control UI render gate and blocked by CSP. The local workspace-path side no longer has a clear current-main reproduction path from the inspected resolver, route, UI, docs, and tests.

## Solution Assessment

Unclear. The closed #41239 approach of redirecting to arbitrary remote avatar URLs is not the safest fit under the current CSP/privacy posture; maintainers should first choose a same-origin proxy/cache design or an explicit unsupported-remote contract.
