## Summary

Keep open: current main fixes the local workspace avatar path, but the remote `http(s)` avatar case is still an unresolved product/security contract mismatch. Control UI intentionally blocks remote image rendering while CLI/config still accept remote avatar values.

## What This Changes

The issue asks for Control UI to display avatars configured with `openclaw agents set-identity --avatar`, including external URLs and local workspace paths.

## Best Possible Solution

Resolve the contract deliberately: either add remote avatar support through an authenticated same-origin proxy/cache with explicit CSP, privacy, size, MIME, and failure-feedback behavior, or make remote avatars clearly unsupported for Control UI by tightening CLI help, docs, validation, and UI diagnostics while keeping the current local avatar route.
