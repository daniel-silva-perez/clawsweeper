## Summary

Current main has solid local-workspace avatar handling, but it does not solve the issue's central remote `http(s)` avatar case. The backend exposes remote avatar metadata yet only serves local avatar bytes, the frontend deliberately drops remote avatar URLs, and the Control UI CSP/docs make that intentional. The CLI/docs still accept remote avatar values, so this remains a real product/security decision rather than a cleanup close.

## Best Possible Solution

Keep this open for maintainer product/security judgment. Either implement remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, size, MIME, and caching controls, or declare remote avatars unsupported in Control UI by tightening CLI help, docs, validation, and UI feedback while preserving the current local workspace avatar path.
