## Summary

Keep open. Current main preserves the local workspace-avatar path and documents the Control UI remote-image block, but the issue's explicit `http(s)` avatar case is still a contract mismatch: CLI/config accept remote avatar values while the Control UI intentionally refuses to render them under the current CSP/privacy model.

## What This Changes

The issue requests that avatars configured with `openclaw agents set-identity --avatar <url/path>` render in the Control UI instead of appearing as broken images.

## Best Possible Solution

Resolve the avatar contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, MIME, size, redirect, caching, and failure-feedback rules, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config schema/help, docs, validation, and UI diagnostics while preserving the current local avatar route.

## Security Review
