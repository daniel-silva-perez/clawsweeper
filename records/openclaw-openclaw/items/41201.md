## Summary

Keep open. Current main appears to have fixed the workspace-local avatar path, but the remote `http(s)` avatar case from the report remains an unresolved Control UI contract mismatch: CLI/config still accept remote avatar values while the Control UI intentionally refuses to render them for CSP and privacy reasons.

## What This Changes

The issue asks for agent identity avatars configured with `openclaw agents set-identity --avatar`, including remote URLs and local workspace paths, to render correctly in Control UI instead of showing a broken image or fallback.

## Best Possible Solution

Resolve the public contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit content, privacy, and network controls, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config docs/schema/help, validation, and UI diagnostics while preserving the current local avatar route.
