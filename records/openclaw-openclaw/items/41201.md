## Summary

Keep open: current main handles workspace-local Control UI avatars through the authenticated same-origin avatar route, but the reported remote `http(s)` avatar path remains a real contract mismatch. CLI/config still accept remote avatar values while Control UI deliberately refuses to render them under the current CSP and privacy posture.

## What This Changes

The issue requests that agent avatars configured with `openclaw agents set-identity --avatar <url/path>` display correctly in the Control UI dashboard instead of showing a broken or missing image.

## Best Possible Solution

Settle the Control UI avatar contract deliberately: either implement remote avatars as controlled same-origin content with documented security limits, or make remote avatars explicitly unsupported for Control UI by aligning CLI help, config validation/docs, UI diagnostics, and tests while preserving the current local/data/blob avatar behavior.

## Reproduction Assessment

Yes for the remaining remote-avatar problem: current source shows remote avatar values are accepted, emitted as metadata, then rejected by the Control UI render gate and blocked by CSP. No high-confidence current-main reproduction remains for ordinary workspace-local avatar paths because local avatars resolve to `/avatar/<agentId>` and are fetched into blob URLs with test coverage.

## Solution Assessment

Unclear. The previously proposed redirect-style fix in #41239 is not the safest default under the current CSP/privacy posture; maintainers should first choose a same-origin proxy/cache design or an explicit unsupported-remote contract.
