## Summary

Keep open. Current main covers the workspace-local avatar path through the authenticated same-origin avatar route, but the remote `http(s)` case from the report is still a real Control UI contract mismatch: config and CLI surfaces continue to accept remote avatar values while Control UI intentionally refuses to render them for CSP/privacy reasons.

## What This Changes

The issue requests that avatars configured with `openclaw agents set-identity --avatar <url/path>` display correctly in Control UI, including remote URLs and local workspace paths.

## Best Possible Solution

Resolve the public contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit privacy and content controls, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config docs/schema/help, validation, and UI diagnostics while preserving the current local avatar route.
