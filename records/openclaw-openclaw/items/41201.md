## Summary

Keep open. Current main fixes the local workspace-avatar side, but the remote `http(s)` avatar case in the report remains a real contract mismatch: CLI/config still accept remote avatar values, while Control UI intentionally blocks rendering them for CSP/privacy reasons.

## What This Changes

The issue requests that avatars configured with `openclaw agents set-identity --avatar <url/path>` render correctly in Control UI instead of showing broken images.

## Best Possible Solution

Resolve the contract deliberately: either add remote avatar support through an authenticated same-origin proxy/cache with explicit CSP, privacy, MIME, size, caching, and failure-feedback behavior, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config schema/docs, validation, and UI diagnostics while preserving the current local avatar route.
