## Summary

Keep open. Current main already shipped selected-model override persistence for spawned subagent sessions, but it has not adopted the PR's subagent-aware gateway fallback, reply-time helper, or global-subagent-default precedence. The PR branch also still has alias-resolution bugs in both new fallback paths, so this needs author/maintainer revision rather than cleanup closure.

## What This Changes

The PR changes subagent configured model precedence and adds gateway/reply runtime fallback helpers with tests, plus a qa-lab type export and config-validation cleanup.

## Best Possible Solution

Rebase the PR onto current main, keep the already-shipped selected override persistence, make any remaining gateway/reply fallback reuse the alias-aware subagent spawn resolver or a shared alias-aware helper, and explicitly settle the global subagent default versus agent primary precedence with matching tests and docs.

## Reproduction Assessment

Yes. The alias regression in the proposed patch is reproducible by configuring `agents.defaults.models` with an alias such as `gpt` and setting a subagent default to that alias; the PR fallback paths parse the raw alias as `<defaultProvider>/gpt` instead of resolving it to the configured provider/model.

## Solution Assessment

No. The direction is plausible, but the current patch is not the best solution because it bypasses the alias-aware spawn resolver and overlaps with current-main spawn override persistence; a smaller shared resolver plus an explicit precedence decision would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve aliases in the reply-time subagent default:** `src/auto-reply/reply/directive-handling.defaults.ts:56`
  - body: `resolveSubagentConfiguredModelSelection` can return a configured alias such as `gpt`; the existing spawn path resolves that through the alias index, but this helper feeds the raw value to `parseModelRef`. In that configuration a subagent can boot and persist `<parent-provider>/gpt` instead of the configured provider/model. Reuse the alias-aware spawn resolution path or the same alias index here.
  - confidence: 0.91
- **[P2] Resolve aliases in the gateway fallback:** `src/gateway/session-utils.ts:1149`
  - body: The new `resolveSessionModelRef` fallback has the same alias gap when `entry.model` is empty: a configured subagent alias is parsed as a bare model under the default provider rather than resolved through the alias index. That makes the race fallback disagree with the spawn-time model selection already covered by tests.
  - confidence: 0.88
