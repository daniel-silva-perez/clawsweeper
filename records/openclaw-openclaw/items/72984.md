## Summary

Keep open. Current main still has the old subagent runtime/default-model behavior, so the PR is not obsolete, but the proposed patch introduces an alias-resolution regression in the new subagent fallback paths and still needs maintainer sign-off on the user-visible precedence change.

## What This Changes

The PR changes subagent model selection so gateway/reply runtime fallbacks use configured subagent defaults, updates related model-selection tests, and includes small config-validation and qa-lab export adjustments.

## Best Possible Solution

Revise the PR so every subagent runtime/read fallback reuses the same alias-aware model resolution as the spawn path, add regression coverage for aliased subagent defaults in both fallback paths, and land only after the intended precedence order is documented or explicitly accepted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Resolve aliases in the reply-time subagent default:** `src/auto-reply/reply/directive-handling.defaults.ts:56`
  - body: `resolveSubagentConfiguredModelSelection` can return a configured alias such as `gpt`; the existing spawn path resolves that through the alias index, but this new helper feeds the raw value to `parseModelRef`. In that configuration, a subagent can boot as `<parent-provider>/gpt` and persist that bogus runtime model instead of the configured provider/model. Use the same alias-aware resolution as `resolveSubagentSpawnModelSelection`.
  - confidence: 0.91
- **[P2] Resolve aliases in the gateway fallback:** `src/gateway/session-utils.ts:1151`
  - body: The new `resolveSessionModelRef` fallback has the same alias gap when `entry.model` is empty: a configured subagent alias is parsed as a bare model under the default provider rather than being resolved through the alias index. That makes the race fallback disagree with the spawn-time model selection already covered by tests.
  - confidence: 0.88
