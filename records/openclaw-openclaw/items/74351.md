## Summary

Keep open. Current main still blocks IPv6 ULA DNS answers under web_fetch even when the existing RFC2544 fake-IP opt-in is enabled, and the only directly related fix PR is closed unmerged. Because the proposed change relaxes an SSRF/private-network boundary, it needs maintainer/security direction rather than automatic cleanup.

## What This Changes

The issue requests that fake-IP proxy compatibility cover IPv6 ULA fc00::/7 addresses as well as the existing RFC2544 IPv4 198.18.0.0/15 opt-in, or expose a separate opt-in for that IPv6 range.

## Best Possible Solution

Define the intended security contract first, then either add a separate default-off IPv6 ULA fake-IP opt-in with schema/types/help/docs/cache-key/audit/test coverage, or document that ULA remains intentionally blocked and users should use a broader explicit private-network opt-in only if maintainers approve one.
