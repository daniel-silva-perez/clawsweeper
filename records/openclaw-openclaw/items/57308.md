## Summary

Keep open. Current main still lacks a heartbeat wake after Slack block-action and modal lifecycle system events are queued, while the PR is a narrow implementation candidate and is not superseded by the still-open, partially overlapping #61374.

## What This Changes

The PR updates Slack block-action and modal lifecycle handlers to request an `exec-event` heartbeat only after `enqueueSystemEvent` succeeds, adds Slack interaction regression coverage, exports the wake helper through plugin SDK runtime subpaths, and updates SDK docs plus the changelog.

## Best Possible Solution

Land this PR or a combined Slack interaction fix that wakes the same routed Slack session only when the system event was actually queued, keeps unauthorized or deduped paths quiet, preserves the modal lifecycle coverage, and reconciles the block-action/thread-routing overlap tracked by #61374.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction: a Slack block action or modal lifecycle event can reach `enqueueSystemEvent` without any `requestHeartbeatNow` call, and the focused unit test can mock both functions to prove the missing wake.

## Solution Assessment

Yes, with one merge-path caveat. Gating `requestHeartbeatNow({ reason: "exec-event", sessionKey })` on the `enqueueSystemEvent` boolean is the narrowest maintainable fix for this liveness bug, but the block-action part should be reconciled with #61374 before landing.
