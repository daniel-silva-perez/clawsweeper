## Summary

Keep open. Current main still blocks the self-hosted Firecrawl baseUrl use case, and this PR addresses the right behavior, but the proposed patch still changes an SSRF-sensitive trust boundary in a way that can choose trusted fetch mode from a DNS preflight that is separate from the guarded fetch resolution. The branch is also dirty and overlaps with active sibling work, so maintainers should choose one canonical Firecrawl self-hosting implementation rather than stale-close this contribution.

## What This Changes

This PR changes Firecrawl search and scrape endpoint resolution to allow custom HTTPS baseUrl values and private/loopback HTTP baseUrl values, selects strict or trusted web-tools fetch mode, updates tests, and adds a changelog entry.

## Best Possible Solution

Land one canonical Firecrawl plugin change that keeps default and public endpoints on strict SSRF handling, permits explicit self-hosted HTTPS endpoints, allows cleartext HTTP only for private/loopback targets using fetch-time pinned resolution or explicit operator opt-in, and updates the Firecrawl docs, tests, and changelog before retiring sibling PRs.

## Reproduction Assessment

Yes. Current main can be reproduced from source and the linked #63877 report: Firecrawl baseUrl values using local HTTP fail at the https check, and HTTPS self-hosted hosts fail the api.firecrawl.dev allowlist check.

## Solution Assessment

No, not as currently proposed. The feature direction is right, but trusted mode should be derived from the same pinned resolution used by the actual guarded fetch or from an explicit private self-hosting opt-in, while public HTTPS endpoints remain strict.

  - confidence: 0.78
