## Summary

Keep open. Current main still rejects the self-hosted Firecrawl baseUrl use case, and this PR is not implemented or safely superseded by a merged canonical fix. The overlapping #63941/#63877 thread is still open, and this PR also has security-sensitive review context that should be resolved by maintainer review rather than conservative cleanup.

## Best Possible Solution

Keep this PR open for maintainer review alongside #63941/#63877. The likely product/code direction is to implement Firecrawl-owned validation similar to the SearXNG pattern: HTTPS allowed for configured public hosts, cleartext HTTP allowed only for private/loopback targets, public endpoints kept on strict web-tools fetch mode, and any trusted/private mode tied to the same guarded resolution used for the actual fetch. Update Firecrawl and web_fetch docs plus regression tests for public HTTPS, private HTTP, public HTTP rejection, and default strict behavior.
