## Summary

Keep open. Current main still rejects self-hosted Firecrawl base URLs, so the PR is not obsolete, but the latest patch still needs maintainer/security review because trusted fetch mode is chosen from a DNS preflight separate from the guarded request path.

## What This Changes

The PR changes Firecrawl search and scrape endpoint resolution to allow custom HTTPS baseUrl values and HTTP private/loopback baseUrl values, selects strict or trusted web-tools fetch mode, and adds tests plus a changelog entry.

## Best Possible Solution

Land one canonical Firecrawl plugin change that keeps the default and public endpoints on strict SSRF handling, allows explicit HTTPS self-hosted endpoints, permits HTTP only when the actual request target is pinned to private/loopback, and updates the Firecrawl docs/tests/changelog before retiring sibling PRs.

## Reproduction Assessment

Yes. Current main can be verified from source: `resolveEndpoint` rejects any Firecrawl baseUrl that is not HTTPS on `api.firecrawl.dev`, and the public docs describe the same restriction.

## Solution Assessment

No, not as currently proposed. The feature direction is right, but trusted mode should be bound to the same pinned resolution used by the actual guarded fetch, or to an explicit operator opt-in, rather than a separate DNS preflight.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Bind trusted mode to the guarded fetch resolution:** `extensions/firecrawl/src/firecrawl-client.ts:140-142`
  - body: `postFirecrawlJson` chooses `withTrustedWebToolsEndpoint` after `validateFirecrawlBaseUrl` performs a separate DNS check. The guarded fetch resolves the hostname again later; if the name resolved private during validation but public at request time, the request still runs under trusted policy with private-network redirects and env-proxy mode enabled. Derive the trust decision from the same pinned target used for the request, or require an explicit private self-hosted opt-in.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch changes SSRF trust boundaries for Firecrawl endpoints and still has a concrete DNS preflight versus fetch-time trust mismatch.

Concerns:

- **[high] Trusted mode can outlive private DNS classification:** `extensions/firecrawl/src/firecrawl-client.ts:140`
  - body: The PR classifies an endpoint as trusted before the guarded fetch performs its own lookup. Because trusted mode enables private-network allowance and trusted env-proxy routing, a DNS change between classification and fetch can give a no-longer-private target broader SSRF behavior than strict mode would allow.
  - confidence: 0.84
