## Summary

Keep open. Current main still blocks self-hosted Firecrawl baseUrl values, so the contribution is addressing a real remaining gap, but the PR should not merge as-is because its trusted-mode decision is made before the guarded fetch performs its own resolution on an SSRF-sensitive path.

## What This Changes

The PR updates Firecrawl search and scrape endpoint resolution to accept custom HTTPS baseUrl values and private/loopback HTTP endpoints, switches between strict and trusted guarded fetch modes, and adds endpoint validation tests plus a changelog entry.

## Best Possible Solution

Land one canonical Firecrawl plugin change that keeps the default and public endpoints on strict SSRF handling, allows explicit HTTPS self-hosted endpoints, permits HTTP only when the actual request target is proven private/loopback at dispatch time or explicitly opted into, and updates tests, docs, and changelog before retiring sibling PRs.

## Reproduction Assessment

Yes. Current main can be verified from source and docs: resolveEndpoint rejects non-HTTPS and any host other than api.firecrawl.dev, matching the reported self-hosted Firecrawl failure path.

## Solution Assessment

No, not as currently proposed. The feature direction is right, but trusted mode should be derived from the same pinned resolution used by the guarded fetch or from an explicit operator opt-in, not a separate DNS preflight.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Bind trusted mode to the guarded fetch resolution:** `extensions/firecrawl/src/firecrawl-client.ts:140-142`
  - body: `postFirecrawlJson` chooses trusted mode from `validateFirecrawlBaseUrl`, which performs a separate DNS/private-network check before the guarded fetch resolves the hostname again. If a hostname resolves private during validation but public at request time, the request still runs with private-network redirects and trusted env-proxy behavior enabled; derive this decision from the same pinned dispatch target or require explicit self-hosted opt-in.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch changes Firecrawl SSRF trust handling and leaves a concrete DNS preflight versus fetch-time trust mismatch.

Concerns:

- **[high] Trusted mode can outlive private DNS classification:** `extensions/firecrawl/src/firecrawl-client.ts:140`
  - body: The PR classifies the endpoint as trusted before the guarded fetch performs its own lookup. Because trusted mode enables private-network allowance and trusted env-proxy routing, a DNS change between classification and fetch can give a no-longer-private target broader SSRF behavior than strict mode would allow.
  - confidence: 0.84
