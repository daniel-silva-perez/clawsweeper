## Summary

Keep open. Current main still lacks a Baileys `messaging-history.set` subscription, reconnect-history config, or a history-sync filter path; the recent WhatsApp reconnect changes cover socket timing and group metadata fallback, not offline message history ingestion. Related message-store/offline-reply attempts are closed or unmerged, and #56088 is useful caution about unreliable on-demand history rather than a merged replacement.

## What This Changes

The issue asks OpenClaw to support WhatsApp reconnect history sync by handling Baileys `messaging-history.set` and making synced messages available through agent-visible history/context buffers.

## Best Possible Solution

Add a WhatsApp-plugin-owned, opt-in reconnect-history path that subscribes to typed `messaging-history.set`, configures the needed Baileys history-sync filter, normalizes and dedupes synced messages through existing inbound helpers, and populates DM/group context buffers without changing default auto-reply behavior for stale backlog.

## Repair Work Prompt

Implement opt-in WhatsApp reconnect history sync for this issue. Keep core extension-agnostic and do not add a broad persistent message-search store. First read the pinned Baileys 7.0.0-rc.9 source/types/docs for `messaging-history.set`, `syncType`, `syncFullHistory`, and `shouldSyncHistoryMessage`. In the WhatsApp plugin, add typed config for opt-in reconnect history sync (default off) and a documented sync-type policy. Wire the Baileys socket/filter only when enabled, attach and detach a `messaging-history.set` listener in the inbound monitor, normalize synced messages through existing inbound/access/history helpers, dedupe against live upserts, and populate DM/group context buffers without auto-replying to stale backlog by default. Update config types, strict zod schema, generated metadata/docs/help as required, WhatsApp docs, and focused regression tests. Changelog is expected because this is user-facing. Do not change unrelated reconnect watchdog behavior, persistent history/search tools, or default `append` auto-reply behavior.
