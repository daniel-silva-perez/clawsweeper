## Summary

Keep open. Current main still has no WhatsApp `messaging-history.set` listener, no reconnect-history config, and no Baileys history-sync filter path; the existing code only handles live/recent `messages.upsert` events and deliberately skips stale append backlog.

## What This Changes

Add WhatsApp reconnect history synchronization by processing Baileys `messaging-history.set` events and making missed messages available through the existing history/context buffers.

## Best Possible Solution

Implement an opt-in WhatsApp-plugin reconnect-history path that listens for typed `messaging-history.set`, enables the needed Baileys history-sync filter for the selected sync types, normalizes and dedupes synced messages through the existing inbound helpers, and feeds DM/group context without auto-replying to stale backlog by default.

## Repair Work Prompt

Fix this issue with a narrow WhatsApp-plugin-owned reconnect history implementation. Current main lacks a `messaging-history.set` listener, reconnect-history config, and `shouldSyncHistoryMessage` filter; stale `messages.upsert` append backlog is still skipped. Add opt-in config such as `channels.whatsapp.syncHistoryOnReconnect` plus a constrained sync-type setting, wire it through account resolution, configure Baileys history filtering as needed, subscribe to `messaging-history.set` with cleanup on close, normalize/dedupe synced messages through existing inbound helpers, and populate DM/group context buffers without enabling default auto-replies to stale backlog. Do not add a broad persistent message store/search feature, do not move owner-specific behavior into core, and do not depend solely on on-demand `fetchMessageHistory` because #56088 reports it may not respond. Update WhatsApp docs/config generated metadata and add a user-facing changelog entry crediting @rafaelmariano-glitch if behavior changes ship.
