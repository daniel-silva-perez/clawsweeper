## Summary

Keep open. Current main still drops non-allowlisted Mattermost group senders before pending history is recorded, and the linked bug report remains open, so this PR is not obsolete. The proposed one-file control-flow change looks directionally correct, but it still needs maintainer review for the privacy/docs contract, a focused regression test, and a rebase/CI refresh before it can land.

## What This Changes

The PR changes the Mattermost monitor so non-allowlisted group senders under `groupPolicy: "allowlist"` continue far enough to record pending history, then return without triggering a bot response.

## Best Possible Solution

Keep the PR open as the implementation candidate for #57607, require a focused Mattermost monitor regression test proving history is recorded without a response, clarify the docs/privacy contract if maintainers accept the behavior, then rebase and land or supersede it with an equivalent fix.

## Reproduction Assessment

Yes. Configure Mattermost with `groupPolicy: "allowlist"`, have a non-allowlisted user post in a group, then inspect current main’s monitor path: it returns at `GROUP_POLICY_NOT_ALLOWLISTED` before `recordPendingHistory()` is defined or called.

## Solution Assessment

Conditionally yes. The PR’s flag-and-return-after-history approach is the narrowest code change I found for preserving context without authorizing a response, but it should be paired with a regression test and maintainer confirmation of the context-visibility contract.
