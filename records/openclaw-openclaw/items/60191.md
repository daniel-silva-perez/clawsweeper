## Summary

Keep this issue open. Current main still has the nearby OpenRouter `Key limit exceeded` fix, but I found no current source, docs, tests, or history evidence that the reporter's exact `403 API key budget limit exceeded (monthly limit)` string is classified as billing when a 401/403 status is present. The static path still appears to land on generic auth unless the message is already billing-classified.

## Best Possible Solution

Add targeted coverage for OpenRouter 401/403 `API key budget limit exceeded (monthly limit)` and update the failover classifier so this provider-scoped budget-exhaustion wording becomes `billing`, then verify the model fallback chain advances to the next configured model instead of marking the provider as generic auth.

## Evidence

- **Reporter narrowed the remaining gap:** After a commenter pointed to the main-branch OpenRouter `Key limit exceeded` fix, the reporter clarified that their actual local error string was `403 API key budget limit exceeded (monthly limit). Contact your org admin.` and noted they could not confirm budget-limit coverage.
- **403 handling only preserves messages already classified as billing:** For HTTP 401/403, the classifier returns `billing` only when `messageReason === "billing"`; otherwise it falls back to `auth`.
