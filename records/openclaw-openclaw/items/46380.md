## Summary

Keep this PR open. Current main still sends MiniMax VLM requests through raw `fetch()` to a host derived from configurable inputs, so the SSRF/key-exfiltration concern is real; however, the submitted branch is stale and should not merge unchanged because it bypasses current proxy-compatible egress behavior, drops the current timeout shape, and lacks focused security regression tests.

## What This Changes

The PR replaces MiniMax VLM's raw `fetch()` call with `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))` and releases the guarded dispatcher after response processing.

## Best Possible Solution

Rebase or replace the patch with a current-main fix that routes MiniMax VLM through the proxy-aware guarded provider helper or equivalent, includes `auditContext: "minimax-vlm"`, preserves timeout behavior, releases dispatchers clearly, and adds regression coverage for blocked private/cloud-metadata hosts via `apiHost`, `MINIMAX_API_HOST`, and `modelBaseUrl` plus proxy-compatible egress.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve proxy-compatible MiniMax egress:** `src/agents/minimax-vlm.ts:80`
  - body: This uses `withStrictGuardedFetchMode` directly, but current MiniMax VLM calls explicitly support env-proxy egress and the shared guarded media helper auto-selects `trusted_env_proxy` when HTTP(S)_PROXY applies. In proxy-only deployments, this can make MiniMax image understanding fail instead of routing through the configured proxy.
  - confidence: 0.9
- **[P2] Preserve the MiniMax VLM request timeout:** `src/agents/minimax-vlm.ts:79-95`
  - body: The guarded call does not pass a timeout or signal. Current main accepts `timeoutMs` and applies `AbortSignal.timeout(timeoutMs)` for this request, so merging this branch as-is would drop the timeout guard for slow or hung MiniMax VLM calls.
  - confidence: 0.86
- **[P3] Add a MiniMax audit context:** `src/agents/minimax-vlm.ts:80-81`
  - body: `fetchWithSsrFGuard` will otherwise log blocked requests under the generic default context. Since this is a security-sensitive provider endpoint, pass a caller label such as `minimax-vlm` so blocked-host audit logs can be attributed.
  - confidence: 0.82
