## Summary

Keep open. The MiniMax VLM SSRF concern remains valid on current main, but this PR is stale against current proxy and timeout behavior and is not safe to merge as-is.

## What This Changes

The PR replaces the MiniMax VLM raw `fetch()` call with `fetchWithSsrFGuard(withStrictGuardedFetchMode(...))` and releases the guarded dispatcher in a `finally` block.

## Best Possible Solution

Revise or replace the PR with a current-main patch that sends MiniMax VLM through the shared guarded provider HTTP helper or an equivalent wrapper, preserves proxy-aware dispatch and request timeouts, adds a MiniMax-specific audit context, and covers private/link-local/metadata hosts for `apiHost`, `MINIMAX_API_HOST`, and `modelBaseUrl`.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure `apiHost`, `MINIMAX_API_HOST`, or `modelBaseUrl` to a blocked host such as `http://169.254.169.254`; current main builds that endpoint and raw-fetches it with the bearer token and image body.

## Solution Assessment

No. Direct strict guarded fetch addresses the obvious raw SSRF path, but it is not the best current-main solution because it bypasses the shared proxy-aware provider transport, omits the request timeout, and lacks provider-specific audit/test coverage.
