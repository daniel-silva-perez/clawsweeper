## Summary

Keep this PR open. Current main still contains the token-sensitive auth-epoch behavior that explains the linked Claude CLI session-reset bug, and this active contributor PR is a targeted implementation for that remaining work. I found no definite PR-introduced correctness or security finding in the diff; the remaining action is maintainer/CODEOWNER acceptance of the identity-only token contract and green CI.

## What This Changes

The PR changes CLI auth-epoch encoding to make Claude token-shaped credentials and token auth-profile hashes identity-only, adds a Claude null-read local fingerprint fallback, bumps `CLI_AUTH_EPOCH_VERSION` to 5, adds regression tests, and updates the changelog.

## Best Possible Solution

Land this PR after `@openclaw/openclaw-secops`/maintainer review, green relevant checks, and explicit acceptance of the identity-only token auth-epoch contract; keep parser hardening or narrower null-read semantics as a separate follow-up only if maintainers want that defense-in-depth split.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main can simulate token-shaped Claude credentials and token auth profiles through `setCliAuthEpochTestDeps`, and the current session reuse path demonstrably resets on auth-epoch mismatch; live macOS keychain timing was not required for this code-path proof.

## Solution Assessment

Yes, with a product-contract caveat. Identity-only hashing plus a version bump is the narrowest maintainable fix for rotation-induced resets; the null-read fallback is defensible as race hardening but should be consciously accepted because it keeps the Claude local epoch stable when a read returns no credential.
