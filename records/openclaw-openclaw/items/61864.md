## Summary

Current main still does not enforce the requested post-compaction token-state invariant for the memory-flush compaction path. Some later work persists `tokensAfter` when it is available, but the path described here can still increment `compactionCount` without passing a trustworthy post-compaction token snapshot or invalidating the old one.

## Best Possible Solution

Keep this issue open and fix the memory-flush/compaction-count paths so every successful compaction either persists a finite post-compaction token snapshot or explicitly invalidates prior `totalTokens`/`totalTokensFresh` before the next gate can trust them. Add focused regression coverage for memory-flush compaction with missing or non-finite `tokensAfter`.
