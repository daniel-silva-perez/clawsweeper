## Summary

Keep open. Current main contains a partial compaction-token snapshot fix, but the invariant requested by this issue is not enforced across all compaction-count paths: CLI post-turn compaction and late embedded compaction reconciliation can still advance `compactionCount` without persisting a trustworthy post-compaction total or invalidating the prior fresh total.

## What This Changes

After compaction, the session store should persist a trustworthy post-compaction token snapshot or invalidate stale totals so the next-turn compaction gate cannot reuse pre-compaction totals.

## Best Possible Solution

Enforce one compaction token-state invariant across embedded, CLI, manual, and late-reconcile paths: whenever a compaction count is persisted, also persist a valid post-compaction `tokensAfter` as the fresh context snapshot or explicitly invalidate/delete the prior total before any later compaction gate or status fallback can treat it as current.

## Reproduction Assessment

Yes. A deterministic source-level reproduction can use the existing CLI compaction test shape: start with `totalTokens=950`, `totalTokensFresh=true`, `contextTokens=1000`, compact to a smaller transcript, and observe that current main records `compactionCount` without changing the fresh token snapshot, so the next gate can still see 950.

## Solution Assessment

No. The current partial fix is in the right area for embedded run completion, but it is not the best complete solution because count-only compaction persistence remains; the safer fix is to make the token-state update part of every compaction-count persistence helper.
