## Summary

Keep open: current main still prepends full fenced Conversation info and Sender JSON metadata for direct Telegram/external-channel user turns, and #72749 is already the open unmerged closing implementation candidate. The related #62077 tracks the broader minimal-DM metadata request, but this Telegram-focused report remains useful until the linked fix or an equivalent implementation lands.

## What This Changes

Request compact or structured delivery of Telegram inbound metadata so short user messages are not preceded by large repeated inline JSON blocks.

## Best Possible Solution

Land #72749 or an equivalent narrow prompt-builder/config change that keeps rich metadata where it is needed while allowing direct Telegram-style chats to use a compact AI-facing metadata prefix and stripping that prefix from user-visible history.

## Reproduction Assessment

Yes. A source-level reproduction is clear: build inbound user context for a direct Telegram/external-channel message with message and sender metadata, then pass it through reply prompt assembly; current main prepends full fenced Conversation info and Sender JSON blocks before the user text.

## Solution Assessment

Yes for triage. Keeping this issue open while the linked closing PR is reviewed is the right handling; the implementation should stay in inbound prompt construction/config rather than relying on provider-specific metadata fields that are not portable across supported model providers.
