## Summary

Keep open. Current main still prepends full fenced Conversation info and Sender JSON blocks to direct Telegram/external-channel user messages, and #72749 remains the active open implementation candidate, so cleanup closure before that PR lands would be premature.

## What This Changes

Request to reduce Telegram inbound user-message prompt noise by moving or compacting the repeated Conversation info and Sender JSON metadata currently prepended to message text.

## Best Possible Solution

Review and land #72749, or an equivalent narrow config-backed prompt-builder change, so direct Telegram-style chats can opt into compact metadata while preserving full untrusted context for groups, replies, forwards, locations, and richer history surfaces.

## Reproduction Assessment

Yes. A direct Telegram message sets ChatType direct, OriginatingChannel telegram, OriginatingTo, message id, timestamp, and sender fields; current prompt assembly then prepends the full fenced Conversation info and Sender JSON blocks before the user body.

## Solution Assessment

Yes for triage: keeping this issue open while the paired closing PR is reviewed is the right handling. The implementation should stay narrow in shared inbound prompt construction/config and avoid losing richer metadata for non-direct or context-heavy messages.
