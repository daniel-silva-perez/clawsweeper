## Summary

Keep open: current main still builds the reported inline fenced JSON metadata before direct Telegram/external-channel user text, and #72749 is the active open PR with closing syntax for this issue. Closing before that PR, or an equivalent fix, lands would be premature.

## What This Changes

The issue requests reducing or restructuring verbose Telegram/direct-chat inbound metadata that is injected inline before short user message text.

## Best Possible Solution

Review and land #72749, or an equivalent narrow config-backed prompt-builder change, so direct Telegram-style chats can opt into compact metadata while preserving full untrusted context for groups, replies, forwards, locations, and richer history surfaces.

## Reproduction Assessment

Yes. A direct Telegram message populates direct chat/channel/message/sender fields, and current prompt assembly prepends the resulting Conversation info and Sender fenced JSON blocks before the user body.

## Solution Assessment

Yes for triage: keeping this issue open while #72749 is reviewed is the right handling. The implementation should stay in shared inbound prompt construction/config rather than relying on provider-specific metadata fields that are not portable across supported model providers.
