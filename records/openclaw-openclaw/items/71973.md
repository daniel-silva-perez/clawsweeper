## Summary

Keep open. Current main still matches the report: the stable system prompt only includes timezone metadata, but exact wall-clock `Current time:` text is still generated and injected into heartbeat, isolated cron, post-compaction refresh, and reset/startup prompt paths. The related #62809 remains broader cache-affinity work and does not supersede this narrower timestamp-placement issue.

## What This Changes

Request moving exact wall-clock `Current time:` injection out of cache-sensitive heartbeat, cron, and post-compaction prompt prefixes while keeping stable timezone metadata in the system prompt.

## Best Possible Solution

Make a provider-aware prompt composition change that keeps timezone metadata in stable system context, late-binds exact current time only for turns that need it, and ensures volatile timestamps do not land before repeated provider cache checkpoints. Coverage should span heartbeat, isolated cron, post-compaction refresh, reset/startup prompts, and Anthropic/OpenAI-family payload shaping.

## Reproduction Assessment

Yes. A read-only source check on current main follows the reported paths and finds `Current time:` injected into heartbeat, isolated cron, post-compaction refresh, and reset prompts, with existing tests asserting those lines.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is likely right, but choosing where exact time belongs relative to provider cache boundaries needs prompt/cache owner design because several flows intentionally depend on that timestamp.
