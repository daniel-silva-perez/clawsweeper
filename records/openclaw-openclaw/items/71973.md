## Summary

Keep open. Current main still matches the report: stable system-prompt time handling is timezone-only, but exact `Current time:` text is still generated and injected into heartbeat, isolated cron, post-compaction refresh, and session-reset prompt paths. This is a valid prompt-cache optimization issue, not an implemented or duplicate cleanup candidate.

## What This Changes

The issue requests moving volatile exact current-time injection out of cache-sensitive prompt prefixes while preserving stable timezone metadata.

## Best Possible Solution

Make a provider-aware prompt composition change that keeps stable timezone metadata in the system prompt, preserves exact wall-clock time only where a turn truly needs it, and prevents volatile timestamps from landing in provider cache checkpoints or repeated stable prefixes. Regression coverage should include heartbeat, isolated cron, post-compaction refresh, session reset, and Anthropic/OpenAI-family payload shaping.

## Reproduction Assessment

Yes. A read-only source check on current main follows the reported paths and finds `Current time:` injected into heartbeat, isolated cron, post-compaction, and reset prompts, with existing tests asserting those lines.

## Solution Assessment

Unclear for a mechanical fix. The requested direction is maintainable, but deciding which runtime turns retain exact time and where cache checkpoints move needs prompt/cache owner judgment because the timestamp is also used for startup and automation grounding.
