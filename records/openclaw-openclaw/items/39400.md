## Summary

Keep this issue open. Current main still exposes provider quota windows to `session_status` as optional human-readable status text only; successful tool details do not include a structured `usage` object, explicit unavailable reason, source metadata, or reset explanation. Related #8635 is adjacent broader session/token usage work, and #45342 may reduce provider polling failures, but neither supersedes this narrower cron-safe quota-window contract.

## Best Possible Solution

Implement an additive `session_status.details.usage` contract while preserving the existing human-readable status card. The structured payload should include provider/source, availability or unavailable reason, plan when known, normalized quota windows such as 5h/day/week with usedPercent/percentLeft/resetAt, and reset explanation only when upstream data supports it. Coordinate naming with #8635, but keep this issue focused on provider quota-window stability and explicit availability metadata.
