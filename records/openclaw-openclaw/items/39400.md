## Summary

Keep this issue open. Current main still exposes provider quota windows to `session_status` only as optional human-readable status text, while successful tool details contain no structured `usage` object, explicit unavailable reason, source metadata, or reset explanation. Related #8635 is open and adjacent broader token/session budgeting work, but it does not supersede this narrower cron-safe quota-window contract.

## Best Possible Solution

Implement an additive `session_status.details.usage` contract while preserving the existing human-readable status card. The structured payload should include provider/source, availability or unavailable reason, plan when known, normalized quota windows such as 5h/day/week with usedPercent/percentLeft/resetAt, and reset explanation only when upstream data supports it. Coordinate names with #8635, but keep this issue focused on provider quota-window stability and explicit availability metadata.
