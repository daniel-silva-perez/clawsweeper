## Summary

Current main still does not provide the structured, cron-safe `session_status` usage contract requested here. Provider quota windows are available only as optional human-readable status text, and unavailable/error states can still disappear from the status card rather than being returned as explicit structured metadata.

## Best Possible Solution

Keep this open and implement an additive `session_status` structured usage contract. Preserve the existing human-readable status card, but return `details.usage` with provider/source, availability or unavailable reason, normalized provider quota windows such as 5h/day/week with percent and resetAt, and reset explanation only when upstream data actually supports it. Treat #8635 as related broader token/session budgeting work, not as a replacement unless it explicitly covers provider quota-window availability and reset metadata.
