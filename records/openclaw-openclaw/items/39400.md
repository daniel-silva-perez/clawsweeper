## Summary

Keep open. Current main still exposes provider quota windows to `session_status` as an optional human-readable status-card line and returns only `statusText` in tool details; it does not provide the requested structured 5h/day usage fields, source, explicit unavailable state/reason, or reset metadata. Current main does include Codex Day/Week label fixes, but those only address part of the observed window-label instability and do not solve the cron-friendly contract requested here.

## Best Possible Solution

Keep this issue open and implement a narrow `session_status` structured usage contract. Preserve the existing human-readable status card, but add `details.usage` with provider/source, explicit availability or `unavailableReason`, normalized provider windows such as 5h/day/week with percent-left or used-percent plus `resetAt`, and reset explanation only when upstream data supports it. Coordinate naming with #8635, which is broader token/session budgeting work, but do not collapse this issue unless that work explicitly covers provider quota-window availability and reset metadata.
