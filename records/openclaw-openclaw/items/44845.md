## Summary

Keep open. Current main already handles the field-name aliases named in the report, but the Volcengine coding-plan path still has no provider-owned fallback for an Ark stream that returns no usable usage payload, and the discussion records that suggested config workarounds did not resolve the reporter's setup.

## What This Changes

Fix `/status` context usage for `volcengine-plan/ark-code-latest` so it reports actual context tokens instead of staying at `0/200k`.

## Best Possible Solution

Add a Volcengine-owned usage compatibility path backed by an Ark coding-plan stream fixture and `/status` coverage; if Ark cannot return usage for coding streams, persist a narrow context snapshot fallback without weakening generic OpenAI-compatible provider behavior.

## Reproduction Assessment

Partially yes: current code gives a high-confidence mechanistic reproduction if a Volcengine Ark stream omits `usage`, because OpenClaw keeps the initialized zero usage and has no fresh context total. A live Ark key/session would still be needed to confirm the exact upstream response shape.

## Solution Assessment

No: the field-mapping fix proposed in the issue body is not the best fix because current main already normalizes those aliases. The safer path is provider-specific compat/fallback behavior with fixtures proving what Ark returns.
