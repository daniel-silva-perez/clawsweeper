## Summary

Keep open. Current main already handles the field-name aliases named in the report, but it still does not provide a Volcengine-owned fallback for Ark coding streams that emit no usable usage payload, and the issue discussion records that the suggested config workarounds did not resolve the reporter's setup.

## What This Changes

The issue requests that Volcengine Ark coding-plan sessions report nonzero context usage in `/status` instead of showing `0/200k`.

## Best Possible Solution

Add a Volcengine-owned usage compatibility path backed by an Ark coding-plan stream fixture and `/status` coverage; if Ark cannot return usage for coding streams, persist a narrow fresh context snapshot fallback without weakening generic OpenAI-compatible provider behavior.

## Reproduction Assessment

Partially yes: current code gives a high-confidence mechanistic reproduction if a Volcengine Ark stream omits `usage`, because OpenClaw keeps zero or stale usage and has no fresh context total. A live Ark key/session or captured stream fixture is still needed to confirm the exact upstream response shape.

## Solution Assessment

No: the field-mapping fix proposed in the issue body is not the best solution because current main already normalizes those aliases. The safer direction is provider-specific compat or fallback behavior with fixtures proving what Ark returns.
