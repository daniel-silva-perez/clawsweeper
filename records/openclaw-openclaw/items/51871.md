## Summary

Keep this issue open. The issue discussion narrowed the regression to malformed persisted cron jobs with missing `schedule` or incomplete/missing `payload`, and current `main` still has multiple Control UI and cron service paths that dereference `schedule.kind` or `payload.kind` without guarding those persisted rows. The CLI list path has a malformed-row regression test, but the Control UI/RPC path is not equivalently hardened.

## Best Possible Solution

Keep open. Harden cron persisted-job handling so jobs missing `schedule` or valid `payload.kind` are normalized when inferable, otherwise skipped/quarantined with a clear diagnostic and doctor repair guidance. Then make the Control UI presenter/controller paths tolerate invalid rows and add regression tests for `cron.list` plus the cron tab rendering malformed persisted jobs.

## Risks / Open Questions

- Fixing only the Control UI would leave the scheduler/RPC layer able to crash or return malformed rows; the safer fix belongs at the cron store/list boundary with UI fallbacks as defense in depth.
- A repair path must avoid silently deleting operator jobs; malformed rows should be quarantined, surfaced through diagnostics/doctor, or displayed as invalid with enough detail to repair.
