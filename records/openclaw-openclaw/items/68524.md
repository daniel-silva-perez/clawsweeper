## Summary

Keep open. Current main still takes raw iOS chat composer image bytes, rejects anything over 5,000,000 bytes before any resize/compression, and base64-encodes accepted attachment bytes directly for send. The open PR #73710 uses closing syntax for this issue and targets the same iOS composer path, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue requests client-side iOS chat composer image resizing/compression, off-MainActor processing, pixel and byte budgets, and safer multi-image handling before gateway/model transport.

## Best Possible Solution

Review and land PR #73710, or a narrow replacement with the same boundary: process iOS composer image bytes before storing/sending pending attachments, reuse the shared ImageIO/JPEG transcode layer, strip source metadata, keep the final 5 MB safety gate, and cover the processor with Swift unit tests plus a real-device send check.

## Reproduction Assessment

Yes. Current main provides a high-confidence static reproduction: any PhotosPicker or paste image over 5,000,000 bytes hits the immediate `exceeds 5 MB limit` branch, while accepted images are sent from raw `att.data.base64EncodedString()` without client-side resize.

## Solution Assessment

Not applicable to a merged patch yet. The direction in PR #73710 appears to be the narrowest maintainable fix for this iOS composer report, but the issue should close only after that PR or an equivalent fix is reviewed, validated, and merged.
