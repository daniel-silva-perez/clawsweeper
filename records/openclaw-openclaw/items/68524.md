## Summary

Keep open. Current main still has the iOS chat composer taking raw PhotosPicker bytes, rejecting images above 5 MB before any resize, and base64-encoding accepted attachment bytes in the MainActor chat view model. There is an open, unmerged PR that explicitly targets this issue, so the correct cleanup is to review and land or close that PR rather than closing the issue now.

## What This Changes

The issue requests client-side iOS chat composer resizing/compression of large image attachments before transport, with off-MainActor processing, pixel/byte budgets, and multi-attachment safety.

## Best Possible Solution

Review and validate PR #73710 as the narrow implementation for this issue, keeping broader gateway/general image-size work in the already-linked separate issues such as #61041, #59913, and #67031.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction: select or inject an image Data payload over 5,000,000 bytes into the iOS chat composer path and OpenClawChatViewModel rejects it before any client-side resize; accepted attachments are later base64-encoded from the stored raw Data.

## Solution Assessment

Yes for direction, pending PR review. A chat-specific OpenClawKit processing adapter over the existing JPEGTranscoder is the narrowest maintainable boundary for the iOS composer problem, while server/gateway resizing remains separate work.
