## Summary

Keep open. Current main still contains the Feishu receive-id type fallback that can send a derived `user_id` for unrecognized non-empty targets, and the linked PR #53787 remains open and unmerged with closing syntax for this issue. The direct Feishu send call does pass `receive_id_type`, so the remaining work is target/type validation and confirming the Feishu request behavior, not closing as already implemented.

## What This Changes

The issue reports Feishu WebSocket message sends failing with a 400 `receive_id_type is required` response even though the plugin intends to pass `receive_id_type` to the Feishu message-create API.

## Best Possible Solution

Review and either land or replace PR #53787 as the canonical fix. The desired end state is a narrow Feishu plugin change that validates receive-id type derivation before direct send, media send, reply fallback, and streaming startup paths call Feishu, while preserving supported `oc_` chat IDs, `ou_` open IDs, and deliberately supported bare user IDs.

Unclear. The issue has concrete user steps, but the exact 400 needs a live Feishu app and target value; static inspection does provide a focused code path where unrecognized non-empty targets still resolve to `user_id` and are forwarded to Feishu.
