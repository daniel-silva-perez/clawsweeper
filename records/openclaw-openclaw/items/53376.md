## Summary

Keep open. Current main still has the permissive Feishu receive-id type fallback, and PR #53787 remains the open unmerged implementation candidate with closing syntax. The SDK call site does pass `receive_id_type`, so the unresolved area is target/type derivation and any Feishu SDK/request behavior around that derived value.

## What This Changes

Feishu WebSocket-mode outbound sends should reliably send messages instead of returning Feishu 400 `receive_id_type is required` errors.

## Best Possible Solution

Use PR #53787 as the canonical candidate, or replace it with a narrow Feishu plugin fix that validates receive-id type derivation before direct send, media send, reply fallback, and streaming startup paths call Feishu. The fix should preserve valid `oc_` group IDs, `ou_` open IDs, and any intentionally supported bare user IDs while rejecting malformed non-empty targets with focused regression coverage.
