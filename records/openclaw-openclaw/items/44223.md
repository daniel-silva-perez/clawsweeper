## Summary

Keep open. Current main still documents and implements Ctrl+C in the TUI as clear/warn/exit only; Esc and `/abort` are still the active-run abort paths. The PR’s intended behavior, abort priority for Ctrl+C, abort-pending/stale-run guards, stale exit-confirmation reset, and preserving partial assistant text on aborted runs, is not present on main. The diff appears limited to TUI interaction code and tests, with no obvious security-sensitive workflow, dependency, lockfile, publishing, or secret-handling changes.

## Best Possible Solution

Keep this PR open for maintainer review. Maintainers should first choose the TUI Ctrl+C contract: either land this PR or an equivalent focused patch that makes Ctrl+C abort active runs and updates `docs/web/tui.md`, or explicitly keep Ctrl+C as clear/exit and resolve the user confusion through the narrower hint path in #38502/#38661. If landing the abort behavior, keep the scope to `src/tui/tui.ts`, `src/tui/tui-session-actions.ts`, `src/tui/tui-event-handlers.ts`, and focused regression tests.
