## Summary

Keep open. Current main still lacks a first-class Project object, Projects dashboard route/view, Gateway `projects.*` protocol surface, project-to-home-session/workstream binding, multi-workspace project model, and project-scoped tool/skill/runtime policy. Existing sessions, agents, workspaces, skill/tool allowlists, and sandbox settings are adjacent primitives but do not satisfy the durable Project model requested here.

## Best Possible Solution

Keep this as the canonical core/product dashboard request. A good solution should design explicit Project storage and additive Gateway `projects.*` protocol methods, add a Projects list/detail/switching surface in Control UI, bind projects to a home session plus optional workstreams, associate one or more workspace paths and docs links, and surface project-scoped tool/skill/runtime constraints while reusing existing agents, sessions, workspaces, skill allowlists, sandbox settings, and sub-agent metadata where they fit. The port/service registry idea from the discussion should be considered in the project-detail design or split into a focused follow-up after the Project object exists.
