## Summary

Keep open. Current main still lacks the requested first-class Project storage/protocol/UI concept, and the recent discussion already treats this as the broader canonical product request rather than a duplicate of narrower session/workspace issues.

## What This Changes

Add a first-class Project concept to the Gateway dashboard, including project/session/workspace bindings plus project-scoped skill, tool, and runtime constraints.

## Best Possible Solution

Keep this as the canonical core/product request until OpenClaw has an explicit Project domain model, additive Gateway projects.* protocol methods, a Control UI Projects list/detail/switching surface, project-to-home-session and workstream bindings, workspace/doc links, and clearly labeled project-scoped constraint display or enforcement built on existing agents, sessions, workspaces, skills, sandbox, and subagent primitives.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug with runtime reproduction steps. The current-main check is high-confidence source/docs inspection showing the requested Project surface is still absent.

## Solution Assessment

Unclear until maintainer design: existing agents, sessions, workspaces, skill allowlists, and sandbox/tool policy are useful building blocks but do not solve the durable Project object request. The maintainable direction should be an additive Project model/protocol/UI design with explicit warn-only versus hard-enforcement semantics.
