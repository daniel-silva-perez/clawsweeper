## Summary

Keep this PR open. Current main still does not provide native outbound WhatsApp location pins through the message tool, so the PR is not obsolete; however, the proposed patch is incomplete against the current channel/gateway architecture and would still fail normal configured or non-dry-run delivery paths.

## What This Changes

The PR adds a `location` message action with coordinate fields, core location send plumbing, ChannelLocation types, and WhatsApp/Baileys sendLocation support with focused tests.

## Best Possible Solution

Revise the PR around the current architecture: advertise `location` in WhatsApp message-tool discovery, add a supported location outbound contract through the current plugin SDK and WhatsApp outbound base/channel-outbound paths, implement a gateway `location` RPC with schema/idempotency/validation, cover core/gateway/WhatsApp tests, and remove coordinate logging before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main has no `location` action/schema/RPC, and the PR diff’s non-dry-run path calls a gateway method that current main and the PR diff do not register.

## Solution Assessment

No. The direction is useful, but the proposed patch is not the best mergeable solution because it misses current WhatsApp discovery/outbound-base wiring, gateway protocol handling, and privacy-safe logging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Expose `location` in WhatsApp action discovery:** `src/agents/tools/message-tool.ts:123-130`
  - body: Adding coordinate fields to the generic message schema is not enough for configured agents. WhatsApp discovery still advertises only reactions and polls, so normal configured message tools build an action enum that rejects `action: "location"` before these new fields can be used.
  - confidence: 0.91
- **[P1] Wire location into the shipped WhatsApp outbound path:** `extensions/whatsapp/src/outbound-adapter.ts:88-106`
  - body: This adds `sendLocation` on one outbound adapter, but the current WhatsApp plugin builds its production outbound surface through `createWhatsAppOutboundBase()`/`whatsappChannelOutbound`, which still expose only text, media, and polls. Real delivery will still see no `plugin.outbound.sendLocation` and fail as unsupported.
  - confidence: 0.9
- **[P1] Register the gateway `location` method before calling it:** `src/infra/outbound/message.ts:426`
  - body: Every non-dry-run location send calls gateway method `location`, but the gateway send handlers and protocol schemas only define `message.action`, `send`, and `poll`. Without a matching schema and handler, real WhatsApp delivery fails with an unknown/unsupported method.
  - confidence: 0.94
- **[P2] Stop logging raw location coordinates:** `extensions/whatsapp/src/send.ts:182-184`
  - body: Precise latitude and longitude are sensitive location data. This structured log would put raw coordinates into log sinks, while nearby WhatsApp send/poll logging redacts identifiers and avoids logging message contents.
  - confidence: 0.88
