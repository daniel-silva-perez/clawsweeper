## Summary

Keep open. Current main has useful Google Meet hardening for setup, browser-state reporting, speech gating, and audio byte counters, but it still does not implement the central request: a supported reliable headless/listen-first Meet participant with documented identity semantics and end-to-end audio or live transcription health proof.

## What This Changes

The issue requests a reliable Google Meet headless/listen-first agent join path with documented identity handling, separate lobby/admission/provider/audio/transcript health states, and clear preflight failure behavior.

## Best Possible Solution

Keep this as Google Meet plugin product work. The desired end state is a documented identity and meeting-entry model, a chosen official Media API versus browser-participant strategy, status that separates browser/lobby/admission/provider/audio/transcript state, preflight checks that fail on missing real audio or transcript movement, and live validation proving actual Meet audio or transcription.

## Reproduction Assessment

Unclear. The issue provides concrete 2026.4.24 live failure data and current main still exposes the relevant join/status/setup/test-speech surfaces, but high-confidence reproduction now requires a real Meet, signed-in Chrome profile or paired node, realtime provider, and audio route that this read-only review did not run.

## Solution Assessment

No. The current partial hardening is useful, but it is not the complete or narrowest maintainable solution for the requested reliable headless/listen-first participant because the realtime path still depends on Chrome/profile/virtual-audio behavior and bridge-side byte counters.
