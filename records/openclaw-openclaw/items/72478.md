## Summary

Keep open. Current main has useful Google Meet hardening, but it still does not deliver the central request: a supported reliable headless/listen-first Meet participant with documented identity semantics and end-to-end audio or transcription health proof.

## What This Changes

The issue requests reliable Google Meet agent joining, including headless/listen-first mode, identity and lobby/admission semantics, audio/transcription health checks, and clearer failure handling for Chrome/Meet audio routing.

## Best Possible Solution

Keep this as Google Meet plugin product work. The desired end state is a documented identity and meeting-entry model, a chosen official Media API versus browser-participant strategy, status that separates browser/lobby/admission/provider/audio/transcript state, preflight checks that fail on missing real audio or transcript movement, and live validation proving actual Meet audio or transcription.

## Reproduction Assessment

Unclear. The issue provides concrete live failure data and current main still exposes the relevant join/status/test_speech surfaces, but high-confidence reproduction now requires a real Meet, signed-in Chrome profile or node, realtime provider, and audio route that this read-only review did not run.

## Solution Assessment

No. The current implementation is a useful partial hardening path, but it is not the narrowest complete solution for the requested reliable headless/listen-first participant because it still depends on Chrome/profile/virtual-audio behavior and bridge-side byte counters.
