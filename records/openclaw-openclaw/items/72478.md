## Summary

Keep open. Current main has meaningful Google Meet hardening, including the merged #73256 observe-mode and speech-health work, but it still does not satisfy the central request for a supported reliable headless/listen-first Meet participant with end-to-end audio or transcription proof.

## What This Changes

Request Google Meet plugin support for reliable headless or listen-first agent joins with explicit identity/admission state, realtime audio/transcription health checks, and clear failure guidance.

## Best Possible Solution

Keep the issue open as Google Meet plugin product work. The desired end state is a documented supported identity and meeting-entry model, a clear choice of official Media API versus hardened Chrome-node participation, status that separates browser/lobby/admission/provider/audio/transcript state, preflight/test checks that fail on missing real audio or transcript movement, and live/manual validation proving admission plus actual Meet audio or transcription.

## Reproduction Assessment

Unclear. The issue gives concrete live steps and health fields from OpenClaw 2026.4.24, and current main still exposes `googlemeet status` and `test-speech` surfaces for the same Chrome/audio path, but this read-only review did not run a live Meet/Chrome audio session.

## Solution Assessment

No. The current main implementation and merged #73256 are useful partial hardening, but the requested reliable headless/listen-first participant remains broader than observe-only browser control plus bridge-byte speech checks.
