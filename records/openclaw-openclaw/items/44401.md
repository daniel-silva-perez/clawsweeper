## Summary

Keep open. Current main still does not broadcast the restart-sentinel GatewayRestart event across recently active sessions; it wakes only the sentinel session, while adjacent shipped work handles narrower restart continuations and restart-aborted recovery. The PR remains meaningful but is dirty against current main and needs changes before it would be safe to merge.

## What This Changes

The PR adds a `wakeRecentlyActiveSessions()` helper to `src/gateway/server-restart-sentinel.ts` that enumerates agent session stores and enqueues GatewayRestart system events for the triggering session plus sessions updated within 60 seconds before the restart timestamp.

## Best Possible Solution

Implement the fan-out as a current-main restart-sentinel path that uses the existing startup task, durable session-delivery queue, and heartbeat wake plumbing. The selection should be deterministic, use the current session timestamp/status contract, skip pseudo and non-resumable session kinds, enqueue GatewayRestart plus `requestHeartbeatNow` for each selected session, and include focused tests for cross-session selection, dedupe, no-route behavior, and skipped session kinds.

## Reproduction Assessment

Yes. On current main, a restart sentinel only calls `enqueueRestartSentinelWake` for the sentinel session key; adding another recently active session row has no code path that enqueues GatewayRestart or requests a heartbeat for that second session.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the branch needs a current-main rebase and should pair every selected system event with a heartbeat wake while filtering non-resumable session kinds through the same contracts used by restart recovery.
