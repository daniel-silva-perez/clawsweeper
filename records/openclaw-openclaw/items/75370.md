## Summary

Keep open. Current main still has the reported canvas-only PDF image fallback behavior, and this PR is a relevant localized fix attempt, but the latest patch is not merge-ready because it shells out to a new external PDF renderer for untrusted inputs, bypasses the existing pixel budget in the fallback path, and lacks the required user-facing changelog/docs/test coverage.

## What This Changes

The PR changes `extensions/document-extract/document-extractor.ts` to check for `pdftoppm` and render selected PDF pages through temporary files when `@napi-rs/canvas` cannot load.

## Best Possible Solution

If maintainers accept Poppler as an optional fallback, keep the implementation inside the `document-extract` plugin but make it bounded: use argv-based asynchronous process execution, avoid shell/PATH surprises, enforce the existing pixel/output/time budgets exactly, document the optional system dependency, add regression tests for the canvas-missing path, and add the required changelog entry.

## Reproduction Assessment

Yes. The original bug is reproducible from current main by following a low/no-text PDF through extraction fallback until `loadCanvasModule()` fails and returns `images: []`; the linked issue supplies a concrete Ubuntu 24.04 setup and observed callback. The PR defects are statically visible in the latest head at the DPI floor and synchronous `execSync()` call.

## Solution Assessment

No. A Poppler fallback is a plausible owner-module fix, but this implementation is not the best way to solve it because it can exceed `request.maxPixels` and blocks the Gateway event loop while invoking an external renderer through the shell.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor the render pixel budget:** `extensions/document-extract/document-extractor.ts:175`
  - body: When `resolveRenderPlan()` chooses `scale < 1`, this `Math.max(72, ...)` still renders at full 72 DPI in the Poppler path. Large or low-budget pages can produce more pixels than `request.maxPixels` while the loop subtracts only `plan.pixels`, so derive the Poppler output size from the render plan without the 72-DPI floor.
  - confidence: 0.94
- **[P2] Run Poppler without blocking the event loop:** `extensions/document-extract/document-extractor.ts:188-190`
  - body: This uses `execSync()` for every rendered PDF page. A malformed or expensive PDF can block the Node event loop for up to 30 seconds per page, and the interpolated command still routes paths through shell parsing; use `execFile`/`spawn` with argv, timeout/abort handling, and output bounds instead.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `extensions/document-extract/document-extractor.ts:149`
  - body: This is a user-facing PDF extraction behavior change, but the PR only changes plugin source. Add an entry under the active `CHANGELOG.md` section and credit the contributor per repo policy.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces external PDF rendering for untrusted inputs and still needs safer process execution plus hard resource-bound enforcement before merge.

Concerns:

- **[medium] Synchronous shell execution of external PDF renderer:** `extensions/document-extract/document-extractor.ts:188`
  - body: `pdftoppm` is invoked through `execSync()` with an interpolated shell command and bare binary lookup. Even with a timeout, hostile or malformed PDFs can block the Gateway event loop per page, and argv-based execution would avoid shell parsing of temp paths.
  - confidence: 0.9
- **[medium] Pixel budget bypass amplifies untrusted PDF rendering:** `extensions/document-extract/document-extractor.ts:175`
  - body: The Poppler fallback keeps a 72-DPI floor even when the render plan chose a smaller scale to satisfy `maxPixels`, allowing oversized PNG renders and larger prompt/memory payloads for attacker-supplied PDFs.
  - confidence: 0.92
