## Summary

Keep open. The linked bug is still present on current main, but this draft PR is not merge-ready because the fallback path can exceed the render budget, invokes Poppler synchronously through a shell, and lacks the dependency docs, regression coverage, and changelog entry needed for a new external renderer path.

## What This Changes

The PR modifies `extensions/document-extract/document-extractor.ts` to detect `pdftoppm` and render PDF pages through temporary files when `@napi-rs/canvas` cannot load.

## Best Possible Solution

If maintainers accept Poppler as an optional fallback, keep the work inside the `document-extract` plugin but implement it as a bounded renderer: argv-based async process execution, no shell command construction, explicit timeout/output/pixel limits, temp-file cleanup, user-facing dependency docs, focused canvas-missing fallback tests, and the required changelog entry.

## Reproduction Assessment

Yes. The underlying issue has a high-confidence static reproduction on current main: a low/no-text PDF reaches image rendering, `loadCanvasModule()` throws when `@napi-rs/canvas` is absent, and the extractor returns no images.

## Solution Assessment

No. A Poppler fallback is plausible, but this draft is not the best implementation because it can exceed the render budget, blocks the Gateway event loop through shell execution, and lacks the docs/tests/changelog needed for a new external renderer contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Honor the render pixel budget:** `extensions/document-extract/document-extractor.ts:175`
  - body: When `resolveRenderPlan()` chooses `scale < 1`, this `Math.max(72, ...)` still renders through Poppler at full 72 DPI. Large or low-budget pages can produce more pixels than `request.maxPixels` while the loop subtracts only `plan.pixels`; derive Poppler output from the render plan without the 72-DPI floor.
  - confidence: 0.94
- **[P2] Run Poppler asynchronously without a shell:** `extensions/document-extract/document-extractor.ts:187-190`
  - body: This invokes `pdftoppm` via `execSync()` for every rendered page. A malformed or expensive PDF can block the Node event loop for up to 30 seconds per page, and the command string still goes through shell parsing; use `execFile` or `spawn` with argv, abort handling, and output bounds instead.
  - confidence: 0.9

  - body: The fallback only works when `pdftoppm` is installed and on PATH, but the PR does not update the PDF docs that currently describe the canvas-only image-rendering path. Add user-facing install and behavior notes for the optional Poppler renderer.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces external rendering of untrusted PDFs and still needs safer process execution plus hard resource-bound enforcement before merge.

Concerns:

- **[medium] Synchronous shell execution of external PDF renderer:** `extensions/document-extract/document-extractor.ts:187`
  - body: The patch invokes `pdftoppm` through `execSync()` with an interpolated shell command and bare binary lookup. Even with a timeout, hostile or malformed PDFs can block the Gateway event loop per page, and argv-based execution would avoid shell parsing of temp paths.
  - confidence: 0.9
- **[medium] Pixel budget bypass amplifies untrusted PDF rendering:** `extensions/document-extract/document-extractor.ts:175`
  - body: The Poppler fallback keeps a 72-DPI floor even when the render plan chose a smaller scale to satisfy `maxPixels`, allowing oversized PNG renders and larger memory or prompt payloads for attacker-supplied PDFs.
  - confidence: 0.92
