## Summary

Keep this PR open. Current main still has no Control UI WebView2 bridge module or lifecycle wiring, while the PR adds a narrow SPA-side `draft-text`/`ready` bridge with cleanup and colocated tests. I did not find a discrete blocking correctness issue in the patch, but the native-message trust boundary and end-to-end WebView2 behavior still need normal maintainer review and validation before merge.

## What This Changes

The PR adds a Control UI WebView2 native bridge module and tests, then wires `initNativeBridge` into `OpenClawApp` connect/disconnect so native `draft-text` messages update the chat draft and the SPA sends a `ready` handshake.

## Best Possible Solution

If maintainers want Windows WebView2 bridge support, keep the minimal `draft-text` inbound and `ready` outbound surface, pair it with the existing Windows host-side validation, run the targeted UI bridge test plus the changed UI gate, perform a real WebView2 smoke, and then land the PR if that validation passes.
