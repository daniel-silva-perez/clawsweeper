## Summary

Keep open. Current main does not implement this PR's intended transparent-proxy/no-env behavior or its nested error-cause diagnostics: trusted env-proxy mode still falls back to DNS-pinned dispatching when no applicable proxy env var is configured, and tool error formatting still returns only the top-level error message. The PR also should not merge as-is because the draft diff and review comments identify a security-sensitive SSRF policy-check regression plus smaller test and serialization issues.

## Best Possible Solution

Keep the PR open for review, but request a narrow revision or replacement patch: support transparent proxy setups for operator-controlled `web_search` endpoints without dropping SSRF hostname/policy validation, keep strict `web_fetch` behavior unchanged, add deterministic proxy-env tests, and land the nested error-cause diagnostics with safe cause serialization.

## Risks / Open Questions

- Merging the draft as written could widen the trusted-env-proxy path by skipping target hostname/policy checks when no proxy env vars exist; a replacement should preserve policy validation before using default fetch.
- The cause-chain formatting patch needs guarded serialization for circular objects and BigInt causes so error handling cannot throw while formatting an already-caught tool error.
- The no-proxy-env regression test must clear all upper/lower proxy env vars to avoid false failures on developer machines or CI runners with proxy env configured.
