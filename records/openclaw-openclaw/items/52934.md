## Summary

Keep open. Current main still resolves bare npm-safe plugin install specs through ClawHub before npm, including community packages, and the existing mitigations warn or verify artifacts without preventing trusted-name substitution. The issue discussion correctly identifies the remaining ClawHub name-squatting risk as a security/product design problem rather than cleanup-close material.

## Best Possible Solution

Keep this open for a security/plugin-install design fix. The best direction is to prevent community ClawHub packages from silently winning over trusted npm names, either by requiring explicit clawhub: for non-official packages, verifying trusted publisher/namespace/provenance before ClawHub can take precedence, adding a durable plugin trust/signature policy, or making install/update prompts fail closed until the operator explicitly accepts the source. Cover the chosen behavior in CLI and chat /plugin install tests plus public plugin security docs.
