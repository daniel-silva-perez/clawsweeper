## Summary

Keep open. Current main still prefers ClawHub for bare plugin specs, and the implemented mitigations are warnings, metadata/integrity checks, private-package rejection, bundled-plugin precedence, and an `npm:` escape hatch. Those reduce risk but do not solve the reported name-squatting/trusted-name substitution problem for community ClawHub packages.

## Best Possible Solution

Keep this open for a security/plugin-install design fix. The best direction is to prevent community ClawHub packages from silently winning over trusted npm names, either by requiring explicit `clawhub:` for non-official packages, verifying trusted publisher/namespace/provenance before ClawHub can take precedence, adding a durable plugin trust policy/signature path, or making the install/update prompt fail closed until the operator explicitly accepts the source. Cover the chosen behavior in CLI and chat `/plugin install` tests plus public plugin security docs.
