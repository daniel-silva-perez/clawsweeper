## Summary

Keep this PR open. Current main still performs gateway `config.set`, `config.patch`, and `config.apply` by reading a config snapshot, checking the caller `baseHash`, computing a replacement config, and later committing that precomputed config without serializing the full read/check/compute/write window. The PR remains a relevant implementation candidate for the lost-update bug, but it needs a rebase over current SecretRef/shared-auth behavior and the existing review gaps addressed before maintainers should land it.

## Best Possible Solution

Keep this PR open as a still-relevant implementation candidate. Rebase or reimplement it on current main so `config.set`, `config.patch`, and `config.apply` serialize the full read/base-hash-check/validate/write sequence, or pass the caller `baseHash` into a fresh lower-level mutation that recomputes from current state before writing. Preserve current SecretRef and shared-auth behavior, map command-lane rejection errors to explicit RPC failures, and add stale concurrent-write tests for all three methods.

## Risks / Open Questions

- The submitted patch was written against older handler code and must be rebased over current SecretRef preflight plus shared-auth generation/disconnect behavior.
- Queue-level command-lane rejection handling remains an open review issue; without explicit handling, a queued config RPC could reject without a normal RPC error response during gateway drain or lane clearing.
- The current PR is correctly scoped to gateway RPC writes, but the broader macOS Talk Mode report #64973 may still need separate handling for direct non-gateway config writers.
