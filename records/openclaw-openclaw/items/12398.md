## Summary

Keep open. Current main still lacks a cron postcondition contract or equivalent scheduler-owned verifier: the cron job types, persisted job shape, closed gateway protocol schemas, CLI/UI creation paths, run logs, and plugin hook surface all continue to model the agent runner outcome directly, so an `ok` isolated agent turn cannot be deterministically converted into failure, retry, or escalation.

## What This Changes

The issue requests an optional cron `postconditions` field that runs a deterministic post-run shell check for isolated agent-turn jobs and uses the result for retry and escalation.

## Best Possible Solution

Add a reviewed, additive cron post-run verification design with typed config/state/protocol support, bounded deterministic verifier execution after isolated agent completion and before state application, durable run-log/UI/CLI/docs projection, retry context injection, and integration with existing retry and failure-alert delivery. If maintainers prefer plugin ownership, first add a supported post-run hook that can return a scheduler-owned failure/retry/escalation decision.

## Reproduction Assessment

Not applicable as a normal runtime bug reproduction; this is a missing feature. The absence is high-confidence from current-main searches plus inspection of the cron types, gateway schemas, detached execution path, state application, CLI/UI creation paths, docs, and plugin hook contracts.

## Solution Assessment

Unclear. The proposed shell-check shape addresses the real gap, but the best maintainable solution needs maintainer design around core contract versus plugin seam and around host command-execution security before a narrow patch is appropriate.
