## Summary

Keep open. Current main still has no cron postcondition contract or equivalent scheduler-owned verifier: isolated cron returns the agent result directly, the typed/runtime/protocol/UI/CLI surfaces do not accept or persist postcondition fields, and the existing retry/failure-alert/plugin-hook paths cover execution errors or observation rather than successful-but-incomplete agent outcomes. The related local-command/ClawHub discussion is useful but complementary, not a replacement for deterministic post-run verification.

## What This Changes

Add optional cron-job postcondition checks that run deterministic verification after agent-turn cron executions and drive retry or escalation when the outcome check fails.

## Best Possible Solution

Add a reviewed, additive cron post-run verification design: typed config/state/protocol support, bounded deterministic verifier execution after isolated agent completion and before state application, durable run-log/UI/CLI/docs projection, retry context injection, and integration with existing retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported fail-capable cron post-run hook that can request scheduler-owned retry or escalation.
