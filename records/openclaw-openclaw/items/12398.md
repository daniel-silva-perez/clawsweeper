## Summary

Keep open. Current main still has no cron postcondition contract or equivalent scheduler-owned verifier that can convert a clean isolated agent run into deterministic failure, retry, or escalation, and the recent discussion already routes this to maintainer design follow-up rather than cleanup closure.

## What This Changes

The issue requests an optional cron `postconditions` contract that runs deterministic post-run checks for agent-driven cron jobs and triggers retry or escalation when the agent exits cleanly without completing the intended outcome.

## Best Possible Solution

Add a reviewed, additive cron post-run verification design: typed config/state/protocol support, bounded deterministic verifier execution after isolated agent completion and before state application, durable run-log/UI/CLI/docs projection, retry context injection, and integration with existing retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported fail-capable cron post-run hook that can request scheduler-owned retry or escalation.
