## Summary

Keep open. Current main still has no cron postcondition contract or equivalent scheduler-owned verifier that can turn a clean isolated agent run into deterministic failure, retry, or escalation; the recent issue discussion also already routes this to maintainer design follow-up rather than cleanup closure.

## What This Changes

Request to add optional deterministic postcondition checks for isolated agent-turn cron jobs, including verifier command execution, retry context injection, escalation, persisted state, and UI/CLI/protocol support.

## Best Possible Solution

Add a reviewed additive cron post-run verification design: typed config/state/protocol support, bounded deterministic verifier execution after isolated agent completion and before state application, durable run-log/UI/CLI/docs projection, retry context injection, and integration with existing retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported fail-capable cron post-run hook that can request scheduler-owned retry or escalation.
