## Summary

Keep open. Current main still has no cron postcondition contract or equivalent scheduler-owned verifier: cron types, persisted job shape, protocol schemas, CLI/UI surfaces, docs, and plugin hooks still model the agent runner outcome directly, so a clean isolated agent run cannot be converted into deterministic failure, retry, or escalation. Recent issue discussion also explicitly kept this active for maintainer design follow-up.

## What This Changes

Request an optional cron `postconditions` field that runs deterministic post-run shell checks, records results, retries isolated agent work with failure context, and escalates after failure.

## Best Possible Solution

Add a reviewed, additive cron post-run verification design with typed config/state/protocol support, bounded deterministic verifier execution after isolated agent completion and before state application, durable run-log/UI/CLI/docs projection, retry context injection, and integration with existing retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported fail-capable cron post-run hook that can request scheduler-owned retry or escalation.
