## Summary

Keep open. Current main still has no cron postcondition contract, no deterministic post-run checker, no durable postcondition state, no retry prompt injection, and no plugin hook that can convert a clean isolated agent run into scheduler-owned failure, retry, or escalation. Existing cron retry/failure-alert behavior covers execution errors, and the related local-command/plugin discussion is complementary rather than a replacement for deterministic outcome verification.

## What This Changes

Request an optional cron `postconditions` configuration that runs a deterministic shell check after isolated agent jobs and can drive retry or escalation from the check result.

## Best Possible Solution

Add a versioned, additive cron post-run verification design: typed config/state/protocol/schema support, bounded deterministic verifier execution after isolated agent completion and before state application, run-log/UI/CLI/docs projection, retry context injection, and integration with existing retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported cron post-run hook that can fail the run and request scheduler-owned retry or escalation.
