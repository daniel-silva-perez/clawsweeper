## Summary

Keep this issue open. Current main still lacks a first-class cron postcondition contract, deterministic post-run checker, persisted checker result state, or plugin hook/API that can synchronously turn an otherwise clean isolated cron run into failure, retry, or escalation. The related localCommand cron request (#61152) is complementary, but it does not cover scheduler-owned post-run verification.

## Best Possible Solution

Keep this open for maintainer/product design. The best path is an additive cron postcondition contract with protocol/schema/CLI/UI/docs support, deterministic bounded verifier execution after isolated agent completion and before applyJobResult(), durable verifier result fields, run-log projection, retry prompt injection, and integration with existing retry/failure-alert delivery. If this should be plugin-owned, first add a supported cron post-run hook/API that can fail the run and request scheduler-owned retry/escalation.
