## Summary

Keep this issue open. Current main still has no cron postcondition contract, no deterministic post-run checker, no durable postcondition state, no retry prompt injection, and no plugin hook that can turn an otherwise clean isolated agent run into scheduler-owned failure, retry, or escalation. The existing maintainer-side Codex comment reached the same conclusion, and the current main SHA still lacks the requested surface.

## What This Changes

The issue requests an additive cron postcondition contract that runs deterministic checks after agent-turn cron jobs, records the result, and can retry or escalate when the check fails.

## Best Possible Solution

Keep this open for maintainer/product design. The best path is an additive cron postcondition contract across runtime types, gateway protocol/schema, CLI/UI, docs, bounded deterministic verifier execution after isolated agent completion and before applyJobResult(), durable verifier result fields, run-log projection, retry prompt injection, and integration with existing retry/failure-alert delivery. If the capability should be plugin-owned, first add a supported cron post-run hook/API that can fail the run and request scheduler-owned retry/escalation.
