## Summary

Keep this issue open. Current main still lacks a first-class cron postcondition contract, deterministic post-run checker, persisted checker result state, or plugin hook/API that can synchronously convert an otherwise clean isolated cron run into failure, retry, or escalation.

## Best Possible Solution

Keep this open for maintainer/product design. The likely path is an additive cron postcondition contract with protocol/schema/CLI/UI/docs support, deterministic bounded verifier execution after isolated agent completion and before applyJobResult(), durable verifier result fields, run-log projection, retry prompt injection, and integration with existing retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported cron post-run hook/API that can synchronously fail a run and request scheduler-owned retry/escalation.
