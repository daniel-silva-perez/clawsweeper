## Summary

Keep open. Current main still has no cron postcondition contract or equivalent scheduler-owned verifier, and the recent steipete and ClawSweeper comments already route this to maintainer follow-up rather than cleanup closure.

## What This Changes

The issue requests optional cron job postconditions that run deterministic checks after isolated agent turns, persist verifier result state, inject retry context, and escalate when retries are exhausted.

## Best Possible Solution

Add a reviewed additive cron post-run verification design across typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery. If maintainers prefer plugin ownership, first expose a supported post-run hook that can return a scheduler-owned failure, retry, or escalation decision.

## Reproduction Assessment

Not applicable as a normal runtime bug reproduction; this is a missing feature. The absence is high-confidence from current-main searches plus inspection of cron types, closed gateway schemas, detached execution, state application, CLI/UI surfaces, docs, and plugin hooks.

## Solution Assessment

Unclear. The proposed shell-check shape addresses the real deterministic-verification gap, but the safest maintainable solution needs maintainer design around core contract versus plugin seam and around host command-execution security before implementation.
