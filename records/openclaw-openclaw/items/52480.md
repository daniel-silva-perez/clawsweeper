## Summary

Keep this PR open. Current main still does not implement the intended Discord transient retry behavior: the Discord retry runner only retries Carbon RateLimitError, and the delivery retry predicate only handles errors with HTTP status/statusCode 429 or >=500. Statusless network failures such as connection resets, DNS errors, and fetch timeouts still bypass those predicates. The PR is paired with open issue #52396 via closing syntax, so it should remain an implementation candidate until maintainers land a focused fix or close the paired path together. The live PR shape still needs maintainer review because broadening the shared Discord retry runner would affect non-idempotent POST send/thread paths and could duplicate visible side effects after ambiguous write failures.

## Best Possible Solution

Keep this PR open and review it together with #52396. The best fix is a focused Discord retry policy that retries rate limits plus safe transient failures, preserves retryAfter handling, and separates idempotent request retries from non-idempotent send/thread POST retries with tests for both retry and no-retry cases. If maintainers do not want this exact branch, close it only after landing or replacing it with a safer canonical fix.
