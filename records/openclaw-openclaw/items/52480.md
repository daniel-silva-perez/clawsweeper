## Summary

Keep open. Current main still does not implement the requested Discord transient-network retry behavior, and this PR is paired with open issue #52396 by the same author. The current PR diff also widens the shared Discord request retry predicate used by non-idempotent visible POST sends, so it needs maintainer review/revision rather than cleanup closure or an automated replacement.

## What This Changes

The PR widens Discord's retry runner to match transient error message patterns in addition to `RateLimitError` and adds retry unit coverage for rate-limit, transient, permanent, and exhausted cases.

## Best Possible Solution

Land a focused Discord retry policy that keeps `retry_after` handling, retries safe transient failures, and explicitly separates idempotent REST/lookup/edit/delete retry behavior from non-idempotent message/thread/component/poll/sticker/voice POST behavior, then close #52396 with tests for both retry and no-retry cases.
