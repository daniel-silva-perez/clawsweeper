## Summary

Keep open. The linked Discord retry bug is still present on current main, but this PR is not ready to merge: the new test imports an undeclared old Discord package, and the proposed retry predicate applies broad transient retries to the shared runner used by visible Discord message/thread POSTs, which can duplicate sends after ambiguous transport failures.

## What This Changes

The PR adds a Discord transient-error retry predicate and a new retry unit test file covering rate-limit, transient, permanent, and exhausted retry cases.

## Best Possible Solution

Update this PR or replace it with a focused Discord retry policy that uses the current internal `RateLimitError`, separates idempotent REST/lookup/edit/delete retry behavior from non-idempotent send/thread/component/poll/sticker/voice POST behavior, and tests both retry and no-retry cases.

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the current Discord RateLimitError in the new test:** `extensions/discord/src/retry.test.ts:1`
  - body: `@buape/carbon` is not a dependency of the Discord plugin or the current workspace lockfile, while the production retry predicate checks `RateLimitError` from `./internal/discord.js`. This added import will fail module resolution and would not exercise the same class the predicate checks, so the new retry test cannot validate the patch.
  - confidence: 0.93
- **[P2] Limit transient retries away from Discord send POSTs:** `extensions/discord/src/retry.ts:25-27`
  - body: This broadens `createDiscordRetryRunner` for every caller, but `createDiscordClient().request` is also used around non-idempotent `POST /channels/{id}/messages` and forum-thread creation. A timeout/reset after Discord accepts the first request can be retried and create duplicate visible messages or threads; scope this predicate to idempotent operations or add a send-safe runner.
  - confidence: 0.9
