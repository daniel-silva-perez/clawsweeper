## Summary

Keep open. Current main still retries the Discord REST runner only for Carbon rate limits, and the paired bug remains unresolved; the PR is a plausible implementation candidate but needs maintainer review because its current diff widens the shared runner without a separate idempotent vs non-idempotent send boundary.

## What This Changes

The PR changes `extensions/discord/src/retry.ts` to add a transient-error regex to the shared Discord retry runner and adds `extensions/discord/src/retry.test.ts` coverage for rate-limit, transient, permanent, and exhausted retry cases.

## Best Possible Solution

Land a focused Discord retry policy that preserves `retry_after`, retries safe transient failures, and explicitly separates idempotent REST/lookup/edit/delete retries from non-idempotent message/thread/component/poll/sticker/voice POST behavior, then close #52396 with tests covering retry and no-retry cases.
