## Summary

Keep this open. Current main has adjacent fallback observability through attempt arrays, structured subsystem logs, trajectory sidecars, lifecycle/status notice state, and docs, but it still does not persist the requested ordered session JSONL audit event before the fallback `model-snapshot`.

## What This Changes

The issue requests an ordered, sanitized session JSONL audit record explaining the failed primary model attempt before a successful fallback model switch.

## Best Possible Solution

Add a small transcript audit event at the fallback/reply-runner boundary that records the failed selected provider/model, chosen fallback provider/model, reason/class, status/code when available, chain position, and bounded sanitized or hashed error detail before the fallback model snapshot/retry, while preserving current routing, logs, lifecycle events, status state, trajectory sidecars, and user-visible notices.

## Reproduction Assessment

Yes. A focused reproduction can mock a primary failover followed by a successful fallback and inspect the session manager entries; current source inspection shows only attempts/logs/notice state plus the later `model-snapshot`, not the requested ordered transcript audit entry.

## Solution Assessment

Yes, the narrowest maintainable fix is a sanitized transcript custom event wired at the fallback/reply-runner boundary. The safer alternative to raw error persistence is to reuse the existing fallback observation sanitization/hash fields and avoid changing candidate selection or fallback eligibility.
