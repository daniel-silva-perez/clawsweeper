## Summary

Keep this issue open. Current main has partially moved past the old hardcoded 5 MB parser cap: `chat.send` now uses `resolveChatAttachmentMaxBytes(cfg)`, which defaults to 20 MB and honors `agents.defaults.mediaMaxMb`. That does not fully implement the requested configurable Control UI upload limit, because image uploads still hit a separate fixed 6 MiB `MAX_IMAGE_BYTES` rejection, the Control UI bootstrap contract does not expose a resolved upload limit, and the browser paste/file/drop paths still read and send files without prevalidation. The related contributor comments correctly narrow the remaining work; the older open #44814 is useful prior art but targets an older `webui`/`agent.request` surface rather than the current Control UI `chat.send` path.

## Best Possible Solution

Keep this open for a focused Control UI upload policy. The best path is to preserve the current shared `agents.defaults.mediaMaxMb` behavior or intentionally add a `gateway.controlUi` field, then make the effective image/file limits coherent, expose the resolved raw-byte and transport ceilings through the Control UI bootstrap contract, add browser-side prevalidation before `FileReader.readAsDataURL`, document the limit on the public Control UI/config docs, and add tests for default behavior, raised-limit acceptance, oversized rejection, and WebSocket/base64 ceiling behavior.

## Risks / Open Questions

- Raising only the shared server parser cap would not fix high-resolution image uploads until the fixed image hydration cap is addressed or documented as the effective ceiling.
- Without a resolved limit in the Control UI bootstrap/runtime state, the browser can still spend memory base64-encoding files that the Gateway will reject later.
- A raw file limit above roughly the WebSocket/base64 practical ceiling can fail at transport level unless the UI and docs account for `MAX_PAYLOAD_BYTES = 25 * 1024 * 1024`.
