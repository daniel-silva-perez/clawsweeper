## Summary

Keep this issue open. Current main has a configurable shared server parser cap through `agents.defaults.mediaMaxMb`, but the Control UI request is still only partially covered: image uploads are separately capped at 6 MiB, the bootstrap contract does not tell the browser the effective upload limit, browser paste/file/drop paths read files before size validation, and the public Control UI docs do not document the configured limit or WebSocket/base64 ceiling.

## What This Changes

Add a configurable Control UI upload size limit so browser chat uploads can accept larger media consistently with the agent/media limits.

## Best Possible Solution

Implement a focused Control UI upload policy: align the image parser/hydration cap with the approved configurable limit, expose the resolved raw-byte and practical WebSocket/base64 limits through `/__openclaw/control-ui-config.json`, reject oversize paste/file/drop inputs before `FileReader.readAsDataURL`, document the defaults and ceiling at https://docs.openclaw.ai/web/control-ui, and add targeted Gateway/UI tests for default rejection, raised-limit acceptance, bootstrap exposure, browser-side rejection, and transport-boundary behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `agents.defaults.mediaMaxMb` above 6, then attempt a Control UI image upload larger than 6 MiB: `chat.send` uses the raised parser cap, but `parseMessageWithAttachments` still rejects images above `MAX_IMAGE_BYTES`, and the browser will read/base64 the file before any configured-size check.

## Solution Assessment

Unclear until maintainer policy is chosen. Reusing `agents.defaults.mediaMaxMb` is the narrowest current implementation path because it already drives `chat.send`, but a separate `gateway.controlUi.maxUploadSizeMb` may be preferable if operators need browser-specific limits.
