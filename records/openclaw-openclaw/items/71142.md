## Summary

Keep open. Current main only partially satisfies the request: the server-side chat path now uses a shared configurable attachment cap, but image uploads still hit a separate 6 MiB image guard, the Control UI does not receive or prevalidate an effective upload policy, and the docs do not describe the interaction with the WebSocket payload ceiling.

## What This Changes

Request to make the Control UI chat upload size limit configurable and consistent with media-processing limits, with browser-side validation and documentation.

## Best Possible Solution

Choose one public upload policy, thread the resolved byte limit through Gateway parsing and the Control UI bootstrap contract, prevalidate paste/file/drop before reading data URLs, keep server-side MIME and byte checks authoritative, and document both the effective raw-byte cap and WebSocket/base64 ceiling on the Control UI docs page.

## Reproduction Assessment

Yes. Configure agents.defaults.mediaMaxMb above 6 MiB and submit an image larger than MAX_IMAGE_BYTES through Control UI; static source and test evidence show chat.send reaches the shared cap path but parseMessageWithAttachments rejects at the fixed image cap.

## Solution Assessment

Unclear because there is no patch yet and the public config shape is still unsettled. The narrowest maintainable path appears to be reusing the existing shared media cap unless maintainers explicitly want an independent gateway.controlUi override.
