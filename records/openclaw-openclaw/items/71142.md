## Summary

Keep open. Current main only partially addresses the request: `chat.send` now uses a configurable shared attachment cap through `agents.defaults.mediaMaxMb`, but image uploads still hit a separate 6 MiB server cap, the Control UI bootstrap payload does not expose an effective upload policy, the browser reads pasted/selected/dropped files before size validation, and the Control UI docs do not describe the effective upload or WebSocket/base64 ceiling. The follow-up discussion from mayank6136 and hclsys is useful and correctly narrows the remaining work, but both are still waiting on maintainer direction for the public config surface.

## What This Changes

Request to make Control UI chat upload limits configurable so larger images and media can be uploaded consistently with the Gateway media handling policy.

## Best Possible Solution

Choose the Control UI upload policy surface, then implement it end to end: resolve the server-side image limit consistently, expose raw-byte and practical WebSocket/base64 ceilings through the bootstrap contract, reject oversize paste/file/drop inputs before `FileReader.readAsDataURL`, document the behavior on the public Control UI docs page, and cover default and raised-limit behavior with focused Gateway/UI tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `agents.defaults.mediaMaxMb` above 6 and send a Control UI image larger than 6 MiB: `chat.send` passes the larger shared cap, but `parseMessageWithAttachments` still rejects the image against `MAX_IMAGE_BYTES`.

## Solution Assessment

Unclear until a maintainer chooses the public config surface. Reusing `agents.defaults.mediaMaxMb` is the narrowest existing path, but a Control UI-specific setting may be the cleaner product boundary if browser uploads should differ from channel or agent media limits.
