## Summary

Keep open. Current main only partially addresses the request: `chat.send` now uses a configurable shared attachment cap, but Control UI image uploads still hit the separate 6 MiB image cap, the browser has no bootstrap upload policy for prevalidation, and the docs do not describe the effective upload ceiling. The follow-up comments from mayank6136 and hclsys are useful because they narrowed the remaining work, but they are still waiting on maintainer direction for the public config surface.

## What This Changes

Request to make Control UI chat upload limits configurable so larger images and media can be uploaded consistently with Gateway media policy.

## Best Possible Solution

Choose the Control UI upload policy surface, then implement it end to end: resolve the server-side image limit consistently, expose raw-byte and practical WebSocket/base64 ceilings through the bootstrap contract, reject oversize paste/file/drop inputs before `FileReader.readAsDataURL`, document the behavior on the public Control UI docs page, and cover default plus raised-limit behavior with focused Gateway/UI tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `agents.defaults.mediaMaxMb` above 6 MiB and submit a Control UI image larger than `MAX_IMAGE_BYTES`; `chat.send` passes the larger shared cap, but `parseMessageWithAttachments` still rejects against the fixed image cap.

## Solution Assessment

Unclear until a maintainer chooses the public config surface. Reusing `agents.defaults.mediaMaxMb` is the narrowest existing path, but a Control UI-specific `gateway.controlUi` policy may be cleaner if browser uploads should differ from channel or agent media limits.
