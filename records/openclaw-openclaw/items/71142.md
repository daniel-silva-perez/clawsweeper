## Summary

Keep open. Current main still leaves the Control UI upload policy incomplete: server chat uses a configurable shared attachment cap, but image uploads are independently capped at 6 MiB, the browser is not given an upload limit, paste/file/drop paths read full files before validation, and public Control UI docs do not explain the effective upload or WebSocket/base64 ceilings.

## What This Changes

The issue requests a configurable upload size limit for Control UI chat uploads so larger media can be submitted consistently with OpenClaw media handling and channel limits.

## Best Possible Solution

Define a focused Control UI upload policy, align image acceptance with the approved configured limit, expose resolved raw-byte and practical transport limits through the bootstrap contract, reject oversize paste/file/drop inputs before reading them, document the defaults and ceilings at `https://docs.openclaw.ai/web/control-ui`, and add targeted Gateway/UI regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to set `agents.defaults.mediaMaxMb` above 6, then submit a Control UI image larger than 6 MiB: `chat.send` uses the raised shared cap, but `parseMessageWithAttachments` still rejects the image at `MAX_IMAGE_BYTES`.

## Solution Assessment

Unclear. Reusing `agents.defaults.mediaMaxMb` is the narrowest current implementation path, but a separate Control UI policy may be the better public product boundary if browser uploads should differ from channel or agent media handling.
