## Summary

Keep this issue open. Current main has moved the server-side attachment parser to the shared `agents.defaults.mediaMaxMb` limit, but the requested Control UI upload policy is still not implemented: there is no `gateway.controlUi.maxUploadSizeMb` or equivalent schema/docs key, the bootstrap response does not expose an effective upload limit to the browser, the browser reads pasted/selected/dropped files before any size check, and images above the fixed 6 MiB media hydration cap are still rejected even if the shared attachment limit is higher. The related open #44814 targets an older `webui`/`agent.request` shape and does not cover the current Control UI `chat.send` path.

## Best Possible Solution

Keep this open for a focused Control UI upload-policy change. Maintainers should choose whether the effective knob is the existing shared `agents.defaults.mediaMaxMb` or a new `gateway.controlUi.maxUploadSizeMb`, then expose the resolved raw-byte/practical transport limit through the Control UI bootstrap response, prevalidate paste/file/drop before `FileReader.readAsDataURL`, reconcile server-side image/file ceilings with the chosen policy, document the defaults and WebSocket/base64 ceiling on the public Control UI docs, and add targeted server/UI tests for default behavior, raised-limit acceptance, oversized rejection, and browser/runtime validation.

## Risks / Open Questions

- Closing now would drop a real remaining Control UI product gap: larger phone photos can still be rejected by the fixed 6 MiB image cap even though the generic attachment parser limit may be higher.
- Adding only a server-side knob would still let browsers base64-read and WebSocket-send files that the Gateway later rejects; the bootstrap contract and client prevalidation need to move with server enforcement.
- Any raised limit needs an explicit practical ceiling because the post-auth Gateway WebSocket payload cap is 25 MiB and base64 inflates raw files.
