## Summary

Keep open. Current main partially addresses the original hardcoded parser-cap concern through `agents.defaults.mediaMaxMb`, but the requested Control UI upload policy is still incomplete: image uploads have a separate 6 MiB cap, the browser is not told the effective limit, client-side paste/file/drop paths read files before validation, and the public Control UI docs do not document the effective upload and WebSocket/base64 ceilings.

## What This Changes

The issue requests a configurable upload size limit for Control UI chat uploads so larger phone images and other media can be accepted consistently with OpenClaw media capabilities and channel limits.

## Best Possible Solution

Implement a focused Control UI upload policy that aligns server image acceptance with the approved configured limit, exposes resolved raw-byte and practical transport limits through the Control UI bootstrap contract, rejects oversize paste/file/drop inputs before `FileReader.readAsDataURL`, documents the defaults and ceilings at `https://docs.openclaw.ai/web/control-ui`, and adds targeted Gateway/UI tests for default rejection, raised-limit acceptance, bootstrap exposure, browser-side rejection, and transport-boundary behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `agents.defaults.mediaMaxMb` above 6, then submit a Control UI image larger than 6 MiB: `chat.send` uses the raised parser cap, but `parseMessageWithAttachments` still rejects images above `MAX_IMAGE_BYTES` and the browser reads the file before any configured-size check.

## Solution Assessment

Unclear until a maintainer chooses the config surface. Reusing `agents.defaults.mediaMaxMb` is the narrowest current code path, but a separate Control UI knob may be the better product boundary if browser uploads need a different limit from channel or agent media handling.
