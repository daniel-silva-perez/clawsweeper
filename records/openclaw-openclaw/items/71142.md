## Summary

Keep open. Current main only partially covers configurable Control UI upload sizing: `chat.send` reads the shared `agents.defaults.mediaMaxMb` cap, but image attachments still hit a separate 6 MiB server-side image limit, the Control UI bootstrap/client do not expose or prevalidate an effective upload policy, and the docs do not explain the configured limit plus WebSocket ceiling. The later issue comments narrowed the same remaining work, and no merged/current-main change resolves it.

## What This Changes

Request to make the Control UI chat upload size limit configurable and consistent with media understanding/channel limits, with browser-side validation and documentation.

## Best Possible Solution

Pick one public upload policy, thread the resolved byte limit through Gateway parsing and Control UI bootstrap, prevalidate paste/file/drop before reading data URLs, keep server-side MIME and byte checks authoritative, and document the effective raw-byte and WebSocket/base64 ceilings on the Control UI docs page.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `agents.defaults.mediaMaxMb` above 6, then submit an image larger than `MAX_IMAGE_BYTES`: `chat.send` passes the shared cap, but `parseMessageWithAttachments` still rejects the image at the fixed 6 MiB guard, and the current test suite asserts that behavior.

## Solution Assessment

Unclear because there is no patch yet and the public config shape is still undecided. The narrowest maintainable path appears to be reusing the existing shared media cap unless maintainers explicitly want an independent `gateway.controlUi` override.
