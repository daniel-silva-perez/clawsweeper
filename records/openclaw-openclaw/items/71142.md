## Summary

Keep open. Current main only partially covers configurable upload sizing: chat attachment parsing reads a shared media cap, but images still hit a separate 6 MiB server limit and the Control UI client/bootstrap/docs do not expose or enforce an effective upload policy. The follow-up from mayank6136 and hclsys narrows the remaining work, but the public config surface still needs maintainer direction.

## What This Changes

Add a configurable Control UI upload limit so web chat uploads can handle larger media consistently with OpenClaw media capabilities.

## Best Possible Solution

Control UI should receive an explicit effective upload policy from the Gateway, reject oversized paste/file/drop inputs before reading them, and have server-side parsing/image hydration accept the same policy without silent drops; public docs should explain the configured raw-byte limit plus the practical WebSocket/base64 ceiling.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to set agents.defaults.mediaMaxMb above 6 and send a Control UI image larger than 6 MiB: the shared maxBytes check can pass, but parseMessageWithAttachments still rejects the image against MAX_IMAGE_BYTES, and current tests assert that behavior.

## Solution Assessment

Unclear. The current direction is not settled because maintainers still need to choose whether Control UI should reuse agents.defaults.mediaMaxMb or get its own gateway.controlUi upload knob; either path must also preserve the image hydration and WebSocket safety limits.

## Risks / Open Questions

- Raising image upload limits without aligning downstream image hydration could reintroduce silent dropped images, which the current MAX_IMAGE_BYTES rejection is explicitly preventing.
- Control UI uploads travel as base64 inside Gateway WebSocket frames, so any raised limit must account for the 25 MiB post-auth payload cap and memory pressure.
- The public config surface is unsettled: reusing agents.defaults.mediaMaxMb is narrow, while gateway.controlUi.maxUploadSizeMb would let browser uploads differ from other chat clients.
