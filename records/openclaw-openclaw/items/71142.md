## Summary

Keep this issue open. Current main partially improves the old server-side attachment cap by using `agents.defaults.mediaMaxMb` for `chat.send`, but it still does not provide a Control UI-specific configurable upload policy, still rejects images above the fixed 6 MiB image hydration cap, does not expose an effective limit to the browser, and does not prevalidate files before base64/WebSocket upload. The active contributor discussion is aligned with those remaining gaps, and the related open #44814 targets an older `webui`/`agent.request` shape rather than fully solving the current Control UI `chat.send` path.

## Best Possible Solution

Keep this open for a focused Control UI upload-policy change. The best path is to have maintainers choose whether the effective knob is the existing shared `agents.defaults.mediaMaxMb` or a new `gateway.controlUi.maxUploadSizeMb`, then expose the resolved raw-byte and practical transport limits through the Control UI bootstrap contract, prevalidate paste/file/drop before `FileReader.readAsDataURL`, make the server-side image/file ceilings coherent with the chosen policy, document the effective defaults and WebSocket/base64 ceiling, and add targeted tests for default behavior, raised-limit acceptance, oversized rejection, and browser/runtime validation.

## Risks / Open Questions

- Closing now would drop a real Control UI product gap: larger high-resolution images can still be rejected by the fixed 6 MiB image cap even when the shared parser cap is higher.
- Adding only a server-side parser knob would leave browsers reading/base64-encoding files that the Gateway later rejects, and could still collide with the 25 MiB WebSocket payload ceiling.
- A new upload-limit config surface needs docs, schema/help metadata, bootstrap contract, UI validation, server enforcement, and tests kept in sync.
