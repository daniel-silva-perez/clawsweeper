## Summary

Keep open. The reported Docker `.env` handling problem is still real, but this PR is stale against current `main`: the active setup implementation now lives in `scripts/docker/setup.sh`, while the PR modifies the old root script surface.

## What This Changes

The PR branch adds project `.env` loading and quote-preserving `.env` rewrites to root `docker-setup.sh`, ignores `docker-compose.override.yml`, and passes `OPENCLAW_INSTALL_BROWSER` into Docker builds.

## Best Possible Solution

Retarget the useful logic into `scripts/docker/setup.sh`: load managed project `.env` defaults before Docker variables are captured without executing arbitrary shell, keep caller environment precedence, write managed `.env` values in a sourceable form, propagate `OPENCLAW_INSTALL_BROWSER`, and add focused e2e coverage for quoted package lists and browser build args.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: put `OPENCLAW_DOCKER_APT_PACKAGES="jq ripgrep"` or `OPENCLAW_INSTALL_BROWSER=1` in root `.env` and run the active setup path; those values are not generally loaded, and managed whitespace values are still rewritten bare.

## Solution Assessment

No. The PR direction is useful, but the implementation is on the wrong surface after the setup-script refactor; the narrow maintainable fix is to update `scripts/docker/setup.sh` and its e2e tests.
