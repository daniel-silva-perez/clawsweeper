## Summary

Keep open. The PR still targets a real Docker setup problem, but the current active setup implementation lives in `scripts/docker/setup.sh`; this branch puts the dotenv loader and quote-preserving writer on the root shim, so it does not fix the path users now run.

## What This Changes

The PR branch modifies `.gitignore` and `docker-setup.sh` to add manual project `.env` loading, quote-preserving `.env` rewrites, `docker-compose.override.yml` ignore, and `OPENCLAW_INSTALL_BROWSER` Docker build-arg propagation.

## Best Possible Solution

Port the useful behavior into `scripts/docker/setup.sh`: non-executing dotenv loading with caller-env precedence, sourceable writes for managed `.env` keys, active `OPENCLAW_INSTALL_BROWSER` build-arg propagation, and focused Docker setup e2e coverage.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: `.env` values for `OPENCLAW_DOCKER_APT_PACKAGES` are not loaded by the active setup script, and active `upsert_env` still writes whitespace values bare.

## Solution Assessment

No. The PR’s direction is useful, but the implementation is on the wrong surface after the Docker setup refactor; the narrow maintainable fix is to keep the root shim thin and move the dotenv/read-write behavior into `scripts/docker/setup.sh`.
