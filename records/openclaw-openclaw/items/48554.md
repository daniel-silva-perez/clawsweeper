## Summary

Keep open. Current main still forwards proxy-fetch request bodies directly into undici without normalizing FormData, while the audio transcription path still builds a global FormData body and an open contributor PR already targets this proxy-specific failure with closing syntax.

## What This Changes

The issue requests that proxy-aware fetch wrappers preserve multipart FormData semantics for audio transcription and batch uploads when HTTP_PROXY or HTTPS_PROXY is configured.

## Best Possible Solution

Review, rebase, land, or replace #48556 with a narrow proxy-fetch fix that normalizes global/FormData-like bodies before undici dispatch, preserves existing undici FormData bodies, clears stale multipart content headers when rebuilding the body, and adds regression coverage for both proxy wrapper entry points.

## Reproduction Assessment

Yes. The issue includes a focused FormData/undici reproduction script, and current main still follows the same source path from global FormData through the custom proxy fetch wrapper into undiciFetch without normalization.

## Solution Assessment

Yes. Fixing the proxy-fetch wrapper is the narrowest maintainable boundary because callers are already correct to use standard FormData, and the wrapper is what changes the fetch implementation to undici under proxy configuration.
