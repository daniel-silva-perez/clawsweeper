## Summary

Current main still does not clearly implement compatibility for legacy OpenAI Codex OAuth field names in stored auth profiles. The loader normalizes `mode`, `apiKey`, and SecretRef-shaped static key/token values, but not OAuth aliases such as `access_token`/`refresh_token` or `accessToken`/`refreshToken`; downstream PI/runtime paths still require canonical `access`, `refresh`, and positive numeric `expires`.

## Best Possible Solution

Keep this issue open and make a narrow auth-profile compatibility decision: either normalize known legacy OpenAI Codex OAuth aliases on read with canonical-field precedence, or surface a clear doctor/models-status warning when an OAuth profile is present but unusable due to schema drift. The change belongs in the auth-profile loader/diagnostic owner path with regression tests for legacy `openai-codex` OAuth shapes and should not change Codex refresh or profile-rotation behavior.
