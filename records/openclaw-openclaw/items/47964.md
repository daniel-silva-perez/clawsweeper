## Summary

Keep open. Current main still has an auth-profile compatibility gap for OpenAI Codex OAuth drift: fresh login paths write canonical OAuth fields, but stored alias-shaped profiles are not normalized on load, and a stale config mode can still mask a stored OAuth profile. The newly linked open PR appears relevant for the masked-mode diagnostic, but it is not merged and does not cover the original alias-normalization question.

## What This Changes

The issue asks whether OpenAI Codex OAuth auth profiles with older or conflicting stored credential shapes should be normalized or diagnosed instead of silently falling back.

## Best Possible Solution

Keep this issue open until maintainers choose a narrow compatibility policy: either normalize a small documented set of legacy OpenAI Codex OAuth aliases on auth-profile load, or emit clear doctor/status/runtime diagnostics for present-but-unusable OAuth profiles. Review the linked PR as the likely fix for the stale api_key-mode masking subcase, and add regression tests around both the alias-only profile and the masked-mode conflict.

## Reproduction Assessment

Yes. A high-confidence source-traced reproduction is to load an openai-codex OAuth profile containing only legacy token aliases such as access_token/refresh_token or accessToken/refreshToken; current main leaves those aliases untouched and PI conversion rejects the profile as unusable. A second source-traced path is a stored OAuth profile paired with auth.profiles.<id>.mode set to api_key, which resolveAuthProfileEligibility filters as mode_mismatch before the generic missing-API-key error.

## Solution Assessment

Unclear pending maintainer policy. The linked PR looks like a narrow approach for the masked-mode diagnostic, but the original alias-normalization behavior still needs a deliberate policy and tests before this issue should close.
