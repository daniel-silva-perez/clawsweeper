## Summary

Keep open. Current main still appears to have the reported compatibility gap: OpenAI Codex OAuth login writes canonical `access`/`refresh`/`expires` credentials, but the auth-profile loader only normalizes `mode`/`apiKey` and static SecretRef fields, so alias-shaped OAuth profiles remain present but unusable downstream.

## What This Changes

The issue asks maintainers to decide whether older OpenAI Codex OAuth credential field names should be supported or migrated automatically when loading `auth-profiles.json`.

## Best Possible Solution

Adopt a narrow, documented auth-profile compatibility policy for OpenAI Codex OAuth: either normalize known legacy aliases on read with canonical fields winning conflicts, or add a clear doctor/models-status diagnostic for present-but-unusable OAuth profiles, with regression tests for the chosen behavior.

## Reproduction Assessment

Yes. A high-confidence source-traced reproduction is to load an `openai-codex` OAuth profile containing only legacy token aliases such as `access_token`/`refresh_token` or `accessToken`/`refreshToken`; current main leaves those aliases untouched, and downstream PI/auth eligibility paths reject the profile as unusable.

## Solution Assessment

Unclear pending maintainer policy. The narrowest maintainable fix belongs in the auth-profile loader or diagnostic path and should avoid changing Codex refresh, identity adoption, external CLI bootstrap, or profile rotation semantics.
