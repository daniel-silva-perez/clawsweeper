## Summary

Keep open. Current main still has the reported compatibility gap: stored OAuth profiles are loaded with only `mode`/`apiKey` and SecretRef static-field normalization, while Codex OAuth runtime paths continue to require canonical `access`, `refresh`, and a valid numeric `expires`.

## What This Changes

The issue asks maintainers to confirm or add compatibility for legacy OpenAI Codex OAuth credential field names in stored auth profiles, or otherwise surface a clear warning when those profiles are unusable.

## Best Possible Solution

Adopt a narrow, documented auth-profile compatibility policy: either normalize known legacy OpenAI Codex OAuth aliases on read with canonical fields winning conflicts, or add a clear doctor/models-status diagnostic for present-but-unusable OAuth profiles, with regression tests covering the chosen behavior.

## Reproduction Assessment

Yes. A source-traced reproduction is to load an `auth-profiles.json` OAuth profile containing only legacy token aliases such as `access_token`/`refresh_token` or `accessToken`/`refreshToken`; current loader leaves those aliases untouched, and downstream PI/auth eligibility paths reject the profile as unusable.

## Solution Assessment

Unclear pending maintainer policy. The narrowest maintainable implementation would live in the auth-profile loader or diagnostic path and should avoid changing Codex refresh, identity adoption, or profile rotation semantics.
