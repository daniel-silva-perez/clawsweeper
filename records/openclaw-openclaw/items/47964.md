## Summary

Keep open. Current main has several adjacent Codex auth-profile fixes, but the specific compatibility gap reported here still appears real: persisted auth profiles are normalized for `mode`/`apiKey` aliases only, while runtime credential usability still expects canonical OAuth fields such as `access`, `refresh`, and `expires`. A profile shaped with legacy OAuth names like `access_token`/`refresh_token` would be accepted as an OAuth entry but remain unusable or fall into refresh/fallback behavior rather than being normalized on load.

## Best Possible Solution

Keep this issue open and fix it in the auth-profile owner surface. The narrow path is to extend `src/agents/auth-profiles/persisted.ts` so persisted OAuth profiles normalize known legacy aliases (`access_token`/`accessToken`, `refresh_token`/`refreshToken`, `expires_at`/`expiresAt` when safe, and `account_id`/`accountId` with canonical precedence), then add a regression test in `src/agents/auth-profiles.ensureauthprofilestore.test.ts` proving an `openai-codex` legacy OAuth profile is usable after load. If maintainers decide not to support those aliases, the fallback should at least emit a clear warning for present-but-unusable OAuth credentials rather than silently falling through to model fallback.

## Evidence

- **Issue discussion identifies field-name drift:** The report and follow-up comments say OAuth login wrote data to `auth-profiles.json`, but runtime behaved unauthenticated until the stored profile was rewritten to canonical `access`/`refresh`/`expires`/`accountId` fields. That is more specific than the adjacent closed Codex auth-profile issues about provider injection, stale default profile ids, or profile rotation.
- **Persisted auth-profile normalizer does not map OAuth aliases:** `normalizeRawCredentialEntry` maps `mode` to `type` and `apiKey` to `key`, then normalizes secret-backed `key`/`token` fields. It does not map OAuth aliases such as `access_token`, `refresh_token`, `expires_at`, `accessToken`, or `refreshToken` to the canonical OAuth fields before `coercePersistedAuthProfileStore` accepts the credential into `profiles`.
