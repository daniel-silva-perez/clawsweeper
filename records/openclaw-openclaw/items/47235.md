## Summary

Keep open. Current main still runs a live-or-cached embedding readiness probe for `openclaw memory status --deep` and renders that provider/probe result as generic `Embeddings: ready|unavailable` next to persisted index counts, so the diagnostic ambiguity reported here remains. The shipped #71568 doctor-status change helps an adjacent slow-probe path but does not resolve the CLI wording/state separation for this issue.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index health from live embedding-provider probe failures so transient probe errors do not look like index/content corruption.

## Best Possible Solution

Update the memory diagnostics surface so persisted index counts, vector/FTS storage state, and embedding-provider readiness are separate, explicitly named states. A failed probe should read as a live or cached provider probe failure while leaving the populated index state visible, with regression coverage and docs/changelog follow-through.

## Repair Work Prompt

Fix the `openclaw memory status --deep` diagnostic ambiguity. Current main prints persisted `Indexed` counts next to generic `Embeddings: unavailable` / `Embeddings error` from `probeEmbeddingAvailability()`, which can make a transient provider probe failure look like an index/content failure. Keep the explicit `--deep` probe behavior, but change the human output and, if appropriate, JSON/type metadata so persisted index state, vector/FTS store state, and live/cached embedding-provider readiness are clearly separate. Add a regression test where the index is populated and `probeEmbeddingAvailability()` returns `{ ok: false, error: "fetch failed" }`, and assert the output identifies this as a provider/probe readiness failure rather than overall index unavailability. Preserve the #71568 doctor cached/skipped default behavior; do not change remote embedding payloads, proxy routing, provider auth, or indexing semantics. Update `docs/cli/memory.md` and add a single-line user-facing changelog fix entry crediting the reporter. Validate with targeted tests and formatter first, then run `pnpm check:changed` in Testbox before handoff if runtime behavior changed.
