## Summary

Keep open. Current main still makes `openclaw memory status --deep` run the embedding readiness probe and render that probe result as generic `Embeddings: ready|unavailable` next to persisted index counts, so the diagnostic ambiguity reported here remains despite adjacent doctor-status caching work.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index state from transient live embedding-provider readiness probe failures.

## Best Possible Solution

Update the memory diagnostics surface so persisted index presence/counts, source counts, vector/FTS state, embedding-cache state, and live or cached embedding-provider readiness are explicitly separate. A failed probe should read as provider/probe availability, not index corruption, with CLI regression coverage plus docs and changelog updates.

## Reproduction Assessment

Yes for the diagnostic ambiguity: current source shows `memory status --deep` runs `probeEmbeddingAvailability()` and renders the result as generic embeddings availability beside index counts. The exact network flapping is not reproducible without the reporter's private endpoint, but that is not the blocker for the UI/diagnostic bug.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix belongs in `memory-core` CLI rendering and shared probe metadata only if needed; the broad overlapping #55458 branch should be treated as context rather than the direct solution.
