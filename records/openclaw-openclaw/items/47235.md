## Summary

Keep open. Current main still makes `openclaw memory status --deep` run a live embedding probe and still renders that probe as generic `Embeddings: ready|unavailable` beside persisted index counts. The in-process 30s cache and docs clarify part of the behavior, but they do not fully address separate CLI invocation flapping or the live-probe-vs-populated-index ambiguity reported here.

## Best Possible Solution

Keep this open and land a focused memory diagnostics fix: make `memory status --deep` explicitly label the embeddings result as a live or cached provider probe, keep persisted index counts/source health separate, add failed-probe-with-populated-index regression coverage, and decide whether any persistent or strict-budget probe caching belongs in this command versus the gateway/doctor path.
