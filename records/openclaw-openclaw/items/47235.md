## Summary

Keep open. Current main has reduced adjacent doctor/status probe latency, but the explicit deep memory CLI still presents a transient embedding-provider probe as a generic embeddings state next to persisted index counts, so the reported diagnostic ambiguity remains.

## What This Changes

The issue asks for `openclaw memory status --deep` to distinguish persisted memory index state from live embedding-provider readiness when transient probe failures occur.

## Best Possible Solution

Update the memory diagnostics surface so persisted index counts, vector/FTS state, embedding-cache state, and live or cached embedding-provider readiness are explicitly separate. A failed provider probe should leave populated index state visible, use provider/probe wording, and have CLI regression coverage plus docs/changelog follow-through.

## Reproduction Assessment

Yes for the diagnostic ambiguity: current source shows `memory status --deep` runs `probeEmbeddingAvailability()` and renders the result as a generic embeddings state beside index counts. The exact network flapping is not reproducible without the reporter's private remote endpoint, but that is not the blocker for the UI/diagnostic bug.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix belongs in the memory-core status CLI and shared probe metadata only if needed; the broad #55458 branch is not the best direct solution for this issue.
