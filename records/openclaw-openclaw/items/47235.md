## Summary

Keep open. Current main still makes `openclaw memory status --deep` perform an embedding readiness probe and then renders that provider/probe result as generic `Embeddings: ready|unavailable` beside persisted index counts. The newer cached doctor path for #71568 is useful adjacent work, but it does not resolve the CLI diagnostic ambiguity reported here.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index health from transient live embedding-provider probe availability.

## Best Possible Solution

Update the memory diagnostics surface so persisted index presence/counts, vector storage state, and live or cached embedding-provider readiness are separate, explicitly named states. A failed probe should read like provider/probe availability, not like index corruption, and the behavior should be covered by CLI regression tests plus docs/changelog updates.
