## Summary

Keep open. Current main still makes `openclaw memory status --deep` run an embedding readiness probe and render that live or cached provider result as generic `Embeddings: ready|unavailable` next to persisted index counts, so the reported diagnostic ambiguity remains. The adjacent doctor/status caching fix for #71568 improves gateway status paths, but it does not resolve this memory CLI output.

## What This Changes

The issue asks for `openclaw memory status --deep` to distinguish persisted memory index health from transient live embedding provider probe readiness.

## Best Possible Solution

Update the memory diagnostics surface so persisted index presence/counts, source counts, vector/FTS state, embedding-cache state, and live or cached embedding-provider readiness are explicitly separate. A failed probe should read as provider/probe availability, not index corruption, with focused CLI regression coverage plus docs and changelog updates.
