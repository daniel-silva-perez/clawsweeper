## Summary

Keep open. Current main still makes `openclaw memory status --deep` run an embedding readiness probe and then renders that transient provider result as generic `Embeddings: ready|unavailable` beside persisted index counts. The cached-path work for `doctor.memory.status` reduces related slow live probes, but it does not resolve the CLI diagnostic ambiguity reported here.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index state from transient live embedding probe readiness and failures.

## Best Possible Solution

Update the memory diagnostics surface so persisted index health and live or cached embedding-provider readiness are separate, explicitly named states, with failure wording that points to provider/probe availability instead of implying index corruption.
