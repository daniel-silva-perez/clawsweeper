## Summary

Keep open. Current main still makes explicit `openclaw memory status --deep` obtain an embedding readiness result from `probeEmbeddingAvailability()` and renders it as a generic `Embeddings: ready|unavailable` line next to persisted index counts. The recent doctor/status caching work reduces unrelated default-path live probes, but it does not resolve this CLI diagnostic ambiguity.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index health from live embedding-provider readiness probe failures or flapping.

## Best Possible Solution

Update the memory diagnostics surface so persisted index counts, vector/FTS storage state, embedding cache state, and live or cached embedding-provider readiness are explicitly separate. A failed probe should read as provider/probe readiness failure while leaving populated index state visible, with CLI regression coverage and docs/changelog follow-through.

## Reproduction Assessment

Yes for the diagnostic ambiguity: current source shows `memory status --deep` awaits `probeEmbeddingAvailability()` and renders that provider result as `Embeddings: ready|unavailable` beside persisted index counts. The exact transient network flapping is not live-reproduced without the reporter's private remote endpoint, but the misleading status path is source-reproducible.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is in the memory-core status CLI output, with shared SDK/doctor payload changes only if machine-readable readiness metadata needs to change; the adjacent doctor cache/skipped work is helpful but not sufficient for this issue.
