## Summary

Keep open. Current main partially mitigates the repeated-run flap by caching embedding probe results for 30 seconds, and the related #71568 doctor/status work is reflected in the changelog. But `openclaw memory status --deep` still treats deep status as a live vector probe plus live embedding ping on a cold/expired cache, then renders that transient probe result directly as `Embeddings: ready` or `Embeddings: unavailable` alongside persisted index counts. The remaining user problem, clearer separation between index health and live provider/probe health, is still actionable.

## Best Possible Solution

Keep this issue open and refine the memory status contract: explicitly separate persisted index/store health, last successful index/sync state, cached probe metadata, and the current live embedding provider probe. Render cold/failed live probes as provider/probe warnings rather than generic embedding unavailability, and add focused CLI/JSON tests for populated index plus failing probe, cache hits, and TTL expiry.
