## Summary

Keep open. Current main still makes `openclaw memory status --deep` run an embedding readiness probe and render that result as generic `Embeddings: ready|unavailable` next to persisted index counts, so the reported diagnostic ambiguity remains.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index health from the live embedding readiness probe when the probe flaps or fails transiently.

## Best Possible Solution

Update the memory diagnostics surface so persisted index counts, vector/FTS storage state, embedding cache state, and live or cached embedding-provider readiness are explicitly separate states. A failed probe should read as provider/probe readiness failure while leaving populated index state visible, with CLI regression coverage and docs/changelog follow-through.
