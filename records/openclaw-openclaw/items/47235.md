## Summary

Keep open: current main still makes `openclaw memory status --deep` depend on a live embedding probe and still renders that probe as generic embeddings readiness next to persisted index counts, so the reported live-probe-vs-index-state ambiguity remains.

## What This Changes

The issue requests clearer, less flappy `memory status --deep` diagnostics that distinguish persisted memory index state from the live embedding readiness probe.

## Best Possible Solution

Keep this open and land a focused memory diagnostics fix: make `memory status --deep` explicitly label the embeddings result as a live or cached provider probe, keep persisted index/source counts separate, show probe failure as provider-readiness information rather than index corruption, and add regression coverage for a failed embedding probe while indexed files/chunks are present. A maintainer should also decide whether persistent or strict-budget probe caching belongs here or in the gateway/doctor path.
