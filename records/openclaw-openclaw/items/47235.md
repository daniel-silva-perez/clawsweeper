## Summary

Keep open. Current main still makes `openclaw memory status --deep` run an embedding readiness probe and prints that provider/probe result as generic `Embeddings: ready|unavailable` beside persisted index counts, so the diagnostic ambiguity remains. The shipped #71568 doctor change is adjacent but does not resolve this CLI output separation.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index health from transient live embedding-provider probe failures.

## Best Possible Solution

Update the memory diagnostics surface so persisted index counts, vector/FTS storage state, and live or cached embedding-provider readiness are separate, explicitly named states. A failed probe should read as provider/probe readiness failure while leaving populated index state visible, with regression coverage and docs/changelog follow-through.

## Repair Work Prompt

Fix the `openclaw memory status --deep` diagnostic ambiguity. Current main prints persisted `Indexed` counts next to generic `Embeddings: unavailable` / `Embeddings error` from `probeEmbeddingAvailability()`, which can make a transient provider probe failure look like an index/content failure. Keep the explicit `--deep` probe behavior, but change the human output and, if appropriate, JSON/type metadata so persisted index state, vector/FTS store state, and live/cached embedding-provider readiness are clearly separate. Add a regression test where the index is populated and `probeEmbeddingAvailability()` returns `{ ok: false, error: "fetch failed" }`, and assert the output identifies this as a provider/probe readiness failure rather than overall index unavailability. Coordinate with the adjacent #74544 hang report if adding a diagnostic probe budget, but do not change remote embedding payloads, proxy routing, provider auth, or indexing semantics. Preserve the #71568 doctor cached/skipped default behavior. Update `docs/cli/memory.md` and add a single-line user-facing changelog fix entry crediting @ningfeemic-dev, and @DanWebb1949 too if the PR also resolves #74544.
