## Summary

Keep open. Current main still renders a live or cached embedding-provider probe as a generic embeddings availability line beside persisted index counts, so the diagnostic ambiguity remains; the adjacent doctor caching fix for #71568 does not cover the explicit `memory status --deep` CLI path.

## What This Changes

The issue asks for `openclaw memory status --deep` to distinguish persisted memory index health from transient live embedding readiness probe failures.

## Best Possible Solution

Update the memory diagnostics surface so persisted index counts, vector/FTS storage state, embedding cache state, and live or cached embedding-provider readiness are explicitly separate. A failed probe should read as provider/probe readiness failure while leaving populated index state visible, with CLI regression coverage and docs/changelog follow-through.
