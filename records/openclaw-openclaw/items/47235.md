## Summary

Keep open. Current main still makes explicit `openclaw memory status --deep` run an embedding readiness probe and render the result as generic `Embeddings: ready|unavailable` next to populated index counts. The shipped #71568 doctor/status caching work reduces repeated live probes, but it does not resolve the CLI diagnostic ambiguity reported here.

## What This Changes

The issue asks for `memory status --deep` to separate persisted memory index health from transient live embedding readiness or probe failures.

## Best Possible Solution

Update the memory diagnostics surface so persisted index counts, vector/FTS storage state, embedding cache state, and live or cached embedding-provider readiness are explicitly separate states. A failed probe should read as provider/probe readiness failure while leaving populated index state visible, with CLI regression coverage and docs/changelog follow-through.
