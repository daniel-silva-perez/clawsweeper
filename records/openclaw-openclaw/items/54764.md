## Summary

Keep this PR open. Current main still has the cleanup shape the PR is trying to fix: killed runs are collected per run id, but final cleanup is then deduplicated by child session key, so only one run sharing that child session gets `completeCleanupBookkeeping`. Orphan pruning also still removes registry entries directly instead of routing through the lifecycle controller’s final cleanup/retry path. The PR is narrow and plausible, but it needs rebase/adaptation to the current file split plus maintainer review of the side-effect ordering noted by the bot review.

## Best Possible Solution

Keep this PR open and adapt it to current main: run final cleanup per `runId` in `src/agents/subagent-registry-run-manager.ts`, decide whether orphan pruning should invoke shared lifecycle bookkeeping or a smaller shared helper without regressing side-effect ordering, and add a regression test for multiple runs sharing one `childSessionKey` plus the deferred cleanup retry behavior.
