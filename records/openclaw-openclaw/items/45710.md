## Summary

Keep this PR open. Current main still does not provide the proposed VS Code debugging path, but the PR should not merge as-is because the one-click clean debug launch removes generated assets that `run-node` does not recreate.

## What This Changes

The PR adds VS Code launch/tasks configuration, an opt-in `OUTPUT_SOURCE_MAPS` tsdown sourcemap toggle, a `clean:dist` script, and VS Code debugging documentation for Gateway development.

## Best Possible Solution

Land a narrow DX improvement once the VS Code launch flow emits source maps through a supported build path that preserves required generated assets, with docs that distinguish the automatic launch from manual `OUTPUT_SOURCE_MAPS=1` builds.

## Reproduction Assessment

Yes for the introduced regression: use the PR's `Rebuild and Debug Gateway` path, then run `/export-session` during that debug session. Source inspection shows the path deletes `dist/`, rebuilds through `run-node`, and omits the template-copy step that `/export-session` requires.

## Solution Assessment

No. The opt-in sourcemap toggle is a plausible part of the solution, but deleting all of `dist/` before a narrower `run-node` rebuild is not the best maintainable launch path; a supported debug build profile or asset-preserving clean is safer.
