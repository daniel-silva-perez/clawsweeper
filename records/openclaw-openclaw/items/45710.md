## Summary

Keep this PR open. Current main still does not provide the proposed VS Code debugging path, and the PR remains useful, but the current patch has a concrete P2 regression: the one-click clean debug launch deletes `dist/` and then relies on `run-node`, which rebuilds TypeScript/runtime-postbuild artifacts without recreating `dist/export-html` templates used by `/export-session`.

## What This Changes

The PR adds VS Code launch/tasks configs, an opt-in `OUTPUT_SOURCE_MAPS` tsdown sourcemap toggle, a `clean:dist` script, and a VS Code section in the debugging guide.

## Best Possible Solution

Land the DX improvement after the one-click launch path uses a supported source-map build flow that preserves or recreates all required runtime assets, especially export-session templates, while keeping the manual `OUTPUT_SOURCE_MAPS=1 pnpm build` path documented clearly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the introduced regression: use the PR's `Rebuild and Debug Gateway` flow from a clean `dist`, then run `/export-session`; source inspection shows the prelaunch task deletes the templates and the follow-up `run-node` rebuild does not restore them.

## Solution Assessment

Is this the best way to solve the issue? No. The opt-in sourcemap toggle is reasonable, but deleting all of `dist/` before a narrower runtime rebuild is not the narrowest maintainable debug path; an asset-preserving clean, full debug build profile, or dedicated source-map build task would be safer.
