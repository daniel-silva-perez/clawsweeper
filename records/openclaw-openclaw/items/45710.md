## Summary

Keep this PR open. Current main still lacks the proposed VS Code debugging path, and the PR remains useful, but the advertised one-click clean launch can break runtime behavior because it deletes `dist/` and then relies on the narrower `run-node` rebuild path instead of the full asset-copying build pipeline.

## What This Changes

The PR adds VS Code Gateway debug launch/tasks files, an opt-in `OUTPUT_SOURCE_MAPS` tsdown source-map toggle, a `clean:dist` script, and a VS Code debugging section in the debugging docs.

## Best Possible Solution

Land the DX improvement after the launch flow uses a supported source-map build path that preserves or recreates required generated assets, with docs that clearly separate automatic one-click debugging from manual `OUTPUT_SOURCE_MAPS=1 pnpm build` workflows.

## Reproduction Assessment

Yes for the introduced regression: use the PR's `Rebuild and Debug Gateway` path, then run `/export-session` during that debug session. Source inspection shows the launch deletes `dist/`, and `run-node` does not recreate `dist/export-html` before the command reads those templates.

## Solution Assessment

No. The opt-in source-map toggle is a plausible part of the solution, but deleting all of `dist/` before a narrower `run-node` rebuild is not the narrowest maintainable debug launch path; an asset-preserving clean, full debug build profile, or purpose-built debug build task would be safer.

## Review Findings
