## Summary

Keep this issue open. Current main has a configured global sub-agent lane cap and a per-requester child limit, but it still does not expose an agent-readable capacity snapshot through `session_status`, a `sessions_capacity` tool, or a structured `sessions_spawn` capacity result. The related session-budget issue #57053 is adjacent but does not cover global sub-agent lane capacity, and #56850 is about cron scheduling cadence rather than spawn capacity.

## Best Possible Solution

Add a read-only capacity surface in core, either as structured fields on `session_status` or a dedicated `sessions_capacity` tool. It should report the configured sub-agent lane limit, active lane count, queued count, remaining runnable slots, and the requester’s `maxChildrenPerAgent` usage. If `sessions_spawn` continues queueing on global capacity, document that explicitly; if it rejects, return a structured `capacity_exceeded` result and add tests for both global lane capacity and per-requester child limits.

## Risks / Open Questions

- Closing now would lose an unimplemented agent-visible capacity surface that affects multi-cron/sub-agent orchestration.
- A future implementation should distinguish the per-requester `maxChildrenPerAgent` fan-out limit from the global `agents.defaults.subagents.maxConcurrent` lane capacity.
- Using only the sub-agent registry for capacity would miss queued-but-not-started lane work; the queue needs a small read-only snapshot API or equivalent source of truth.
