## Summary

Keep this issue open. Current main has partial sub-agent limits and status surfaces, but it still does not expose the global sub-agent concurrency budget or a capacity snapshot that lets an agent know whether a new `sessions_spawn` will run immediately versus queue behind the `subagent` lane.

## Best Possible Solution

Add a read-only capacity surface in core, either as structured fields on `session_status` or a dedicated `sessions_capacity` tool. It should report the configured sub-agent lane limit, active sub-agent lane count, queued count, remaining runnable slots, and the requester’s `maxChildrenPerAgent` usage. If `sessions_spawn` keeps queueing on global capacity, document that clearly; if it rejects, return a structured `capacity_exceeded` result with tests for both the global lane and per-requester child limit.

## Risks / Open Questions

- The implementation should distinguish the per-requester `maxChildrenPerAgent` fan-out limit from the global `agents.defaults.subagents.maxConcurrent` lane capacity so agents do not make decisions from the wrong budget.
- Exposing active/queued lane state may need a small read-only queue snapshot API; using only the sub-agent registry would miss queued-but-not-started lane work.
