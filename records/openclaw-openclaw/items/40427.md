## Summary

Keep open. Current main still emits config warnings directly in `createConfigIO().loadConfig()` and `writeConfigFile()`, so this PR's direct warning-dedupe behavior is not fully implemented on main; however, the PR branch should not merge as-is because it leaves stale dedupe state across non-warning exits and regresses the real multiline warning format already covered on main.

## What This Changes

The PR adds a module-level per-config-path warning fingerprint cache in `createConfigIO().loadConfig()` and a unit test for repeated warning loads and re-logging after config content changes.

## Best Possible Solution

Land one canonical config logging implementation, likely coordinated with #38295, that dedupes repeated warning and error diagnostics by config content plus formatted diagnostics across load, write, and reload paths; clears state on missing, invalid, and no-warning transitions; preserves real multiline warning output; and includes regression coverage for warning disappearance and reappearance.

## Reproduction Assessment

Yes. A warning-producing config followed by repeated `createConfigIO({ logger }).loadConfig()` calls exercises the original repeated warning path, and the PR-introduced stale-state bug follows from warning config A, then a missing or invalid config, then warning config A again. I did not run tests because this was a read-only review.

## Solution Assessment

No. The PR is directionally useful, but it is not the best mergeable fix as written: it only guards the direct load warning call site, misses non-warning exit cleanup, and conflicts with current main's multiline warning-format contract.
