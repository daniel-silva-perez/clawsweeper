## Summary

Keep open. The branch targets a real config-warning log-spam problem, but it is not merge-ready because the new warning fingerprint state can survive early-exit and non-warning load paths, and the same area overlaps broader maintainer work in #38295.

## What This Changes

The branch adds an in-memory per-config-path fingerprint cache around `createConfigIO().loadConfig()` warning logs and adds a focused warning-dedupe unit test.

## Best Possible Solution

Land one canonical config-warning dedupe design in the config IO/reload area that defines the intended scope for load, write, startup, and reload warning emitters, clears state across warning-to-non-warning transitions, and has regression coverage for repeated loads, writes, config changes, and warning disappearance/reappearance.
