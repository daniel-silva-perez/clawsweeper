## Summary

Keep open. The PR targets a real config-warning log-spam problem, but the branch has concrete correctness regressions and overlaps broader maintainer work in #38295; current main also only partially mitigates the user-facing spam through runtime snapshot/reload behavior rather than implementing this direct `createConfigIO().loadConfig()` dedupe.

## What This Changes

The PR adds a module-level per-config-path warning fingerprint cache to `createConfigIO().loadConfig()` and adds a focused unit test for repeated warning loads and config-content changes.

## Best Possible Solution

Land one canonical config logging design that dedupes warning/error payloads at the appropriate config IO and gateway reload surfaces, clears state when warnings disappear or loads transition through missing/invalid states, preserves multiline formatting, and covers load, write, reload, and warning reappearance tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear warning state on non-warning load exits:** `src/config/io.ts:767-773`
  - body: The new fingerprint cache is only cleared when validation succeeds with no warnings. If warning config A is followed by a missing, parse-failed, or invalid config and then returns to warning config A, the stale fingerprint suppresses the re-emerged warning even though the file changed.
  - confidence: 0.93
- **[P2] Preserve real-newline warning formatting:** `src/config/io.ts:773`
  - body: The changed log template uses `Config warnings:\\n`, which prints a literal backslash-n sequence. Current main has coverage requiring real multiline `Config warnings:` output, so the branch should keep the real newline when rebased.
  - confidence: 0.87
