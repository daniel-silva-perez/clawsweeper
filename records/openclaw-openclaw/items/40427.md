## Summary

Keep open. The PR targets a real config-warning log spam problem, but the proposed cache can suppress a warning after early-exit load states, and current main still lacks an equivalent low-level warning dedupe implementation.

## What This Changes

The PR adds an in-memory per-config-path fingerprint cache around `createConfigIO().loadConfig()` warning logs and adds a focused warning-dedupe unit test.

## Best Possible Solution

Land one canonical config-warning dedupe design in the config IO and reload area that defines the intended scope for load, write, startup, and reload warning emitters, clears state across warning-to-non-warning transitions, and has regression coverage for repeated loads, writes, config changes, and warning disappearance/reappearance.
