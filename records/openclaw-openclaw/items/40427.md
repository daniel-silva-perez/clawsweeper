## Summary

Keep open. Current main still lacks config-warning dedupe in the lower-level config IO path, and the PR remains relevant, but it is not merge-ready as-is because its fingerprint state can remain stale across non-warning early exits and it only covers the load warning emitter while the write path still logs warnings unconditionally.

## What This Changes

The PR adds a per-config warning fingerprint cache around the config load warning log and adds regression tests for repeated warning loads.

## Best Possible Solution

A single config-warning dedupe implementation should live in `src/config/io.ts`, cover both load and write warning emitters, clear state on all no-warning and early-exit transitions, re-log when raw config content or formatted warning details change, and carry focused regression tests for repeated load/write and warning-to-non-warning-to-warning transitions.
