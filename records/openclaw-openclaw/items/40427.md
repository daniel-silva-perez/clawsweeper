## Summary

Keep open. Current main still has no direct config-warning dedupe in config IO: `loadConfig()` and `writeConfigFile()` both emit `Config warnings:` unconditionally, and existing tests only cover newline formatting. This PR remains a relevant implementation candidate, but it needs revision for stale dedupe state and overlap with the broader open maintainer PR #38295.

## Best Possible Solution

Keep this PR open for maintainer review unless maintainers explicitly choose #38295 as the canonical implementation. The best fix is a centralized warning fingerprint helper in `src/config/io.ts` that covers both load and write warning emitters, clears state on every no-warning, failed, and early-return load path, re-logs when raw config content or formatted warning details change, and adds focused tests for repeated load plus warning -> no-warning/invalid/missing/parse-failed -> same-warning transitions.

## Risks / Open Questions

- The PR's module-level warning fingerprint can remain stale across missing, invalid, parse-failed, or other early non-warning load paths, suppressing a re-emerged identical warning after the file changed away and back.

- There is implementation overlap with open maintainer PR #38295, so maintainers should choose a canonical path or rebase/narrow this PR before merge.
