## Summary

Keep open. Current main still has direct `Config warnings:` emitters in config load and write paths, so the PR is not implemented on main; however, the proposed load-only fingerprint cache has concrete correctness issues, and the overlapping maintainer PR #38295 makes the canonical dedupe policy a maintainer decision rather than a cleanup close.

## What This Changes

The PR adds a module-level per-config-path warning fingerprint cache in `src/config/io.ts` for repeated `createConfigIO().loadConfig()` warnings and adds a focused unit test for repeated loads and config-content changes.

## Best Possible Solution

Land one canonical config logging design that defines warning/error dedupe scope across load, write, startup, and reload emitters, clears state across warning-to-clean and invalid/missing transitions, preserves real-newline formatting, and includes regression coverage for repeated loads, writes, config changes, and warning disappearance/reappearance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear warning state on early-exit load paths:** `src/config/io.ts:767-773`
  - body: The new fingerprint is cleared only when validation succeeds with no warnings. If a path logs warning A, then the file becomes missing, parse-failed, or invalid, and later returns to warning A, the stale fingerprint remains and suppresses the re-emerged warning even though the file changed.
  - confidence: 0.93
- **[P2] Preserve real-newline warning formatting:** `src/config/io.ts:773`
  - body: This changed line logs `Config warnings:\\n...`, which prints a literal backslash-n sequence. Current main has regression coverage for real multiline `Config warnings:` output, so the branch should keep the current formatting when rebased.
  - confidence: 0.87
