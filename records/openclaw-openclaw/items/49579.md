## Summary

Keep this PR open. Current main still has the routine WhatsApp banner-noise problem, so the PR is not obsolete or implemented. The submitted diff still needs narrowing because it suppresses every `WhatsApp gateway disconnected...` system event, while current main queues the same generic disconnect event before distinguishing retryable reconnect flaps from logged-out, session-conflict, and max-attempt-stop terminal states. Related #72043 is adjacent configurable system-event policy work, not a replacement for this focused WhatsApp transport-banner fix.

## Best Possible Solution

Keep this PR open and request a narrower implementation: suppress routine connect banners now, and suppress disconnect banners only when the close is proven retryable/routine, or first change the WhatsApp monitor to enqueue distinct high-signal terminal events for logged-out, conflict, and max-attempt-stop paths. Add regression tests for routine reconnect flaps and terminal outage paths. Treat #72043 as separate configurable system-event policy work.
