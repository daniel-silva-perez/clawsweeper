## Summary

Keep this PR open. Current main still queues routine WhatsApp connect/disconnect banners into prompt-bound system events, and the submitted PR is still not ready as-is because its blanket `WhatsApp gateway disconnected` filter would also hide terminal WhatsApp outage context from the next agent turn. The related #72043 config-recovery knob is adjacent policy work, not a replacement for this focused WhatsApp banner fix.

## Best Possible Solution

Keep this PR open and request a narrower implementation: suppress routine connect banners, and suppress disconnect banners only when the close is proven retryable/routine or after the WhatsApp monitor first enqueues distinct high-signal terminal events for logged-out, conflict, and max-attempt-stop paths. Add tests for routine reconnect flaps plus terminal outage paths. Treat #72043 as a separate configurable system-event policy discussion, not the canonical fix for this PR.
