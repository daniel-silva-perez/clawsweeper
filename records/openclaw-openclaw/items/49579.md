## Summary

Keep this PR open. Current main still queues routine WhatsApp connected/disconnected banners into prompt-visible system events, so the underlying noise problem is real. The submitted implementation is still too broad because filtering every `WhatsApp gateway disconnected...` line would also hide terminal outage context from prompt assembly; the detailed logged-out/conflict/max-attempt messages are currently runtime errors, not queued system events.

## Best Possible Solution

Keep this PR open and request a narrower implementation: suppress routine connect banners, and suppress disconnect banners only when the close is proven retryable/routine or after the WhatsApp monitor first enqueues distinct high-signal terminal events for logged-out, conflict, and max-attempt-stop paths. Add tests for routine reconnect flaps plus terminal outage paths. Treat #72043 as separate configurable system-event policy work.
