## Summary

Keep this PR open. Current main still queues routine WhatsApp connect/disconnect banners into prompt-bound system events, so the user problem is not implemented away. The submitted PR is also not ready as-is because its blanket `WhatsApp gateway disconnected` prefix filter would hide terminal WhatsApp close states from the next agent turn; earlier review comments correctly identified the same functional blocker. Security review of the submitted diff did not find CI, dependency, package, secret, artifact, or script changes.

## Best Possible Solution

Keep this PR open and request a narrower implementation: suppress routine connect banners, suppress disconnect banners only when the close is proven retryable/routine, or first have the WhatsApp monitor enqueue distinct high-signal terminal events before prompt filtering. Add tests for routine reconnect flaps and for logged-out, conflict, and max-attempt-stop terminal paths. Coordinate any generic operator knob with #72043 separately.
