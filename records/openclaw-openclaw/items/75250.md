## Summary

Keep open. Current main has useful mitigations for stale plugin runtime-deps roots, service entrypoint stability, and plugin-version-skew degradation, but it still intentionally runs macOS local mode from an external Node/global CLI layout; Homebrew Node, the OpenClaw package, LaunchAgent runtime path, and plugin cache are not updated as one atomic unit. The open #74630 overlaps the runtime/bootstrap regression family but does not fully cover this Homebrew Node ABI and service-runtime drift report.

## What This Changes

The issue asks OpenClaw to survive or clearly repair macOS update drift between Homebrew Node, the global OpenClaw package, the LaunchAgent runtime path, plugin SDK files, and plugin runtime-deps cache.

## Best Possible Solution

Maintainers should decide the macOS gateway runtime authority first. If Homebrew/global npm remains supported, doctor/status/update should explicitly report and validate CLI Node, service Node, OpenClaw package root/version, LaunchAgent command, plugin min-host requirements, and runtime-deps stage key, then repair or prune stale cache/service state before startup. If OpenClaw should own the service runtime, the macOS installer/service manager should move toward a pinned managed runtime/package layout.

## Reproduction Assessment

No. The issue includes concrete macOS/Homebrew logs that match current code paths, but I did not find a deterministic repo-local reproduction for the full brew ABI churn plus service/package/plugin-cache skew sequence.

## Solution Assessment

Unclear. A bundled or pinned runtime may solve the broadest class, but that requires maintainer platform judgment; a narrower safe first step would be maintainer-approved diagnostics and validation around the chosen runtime policy.
