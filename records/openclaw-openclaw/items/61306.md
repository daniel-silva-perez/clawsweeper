## What This Changes

This PR adds a Claw mission-control backbone across `src/claw`, gateway RPC/events/scopes, shared/config types, Control UI, Claw docs, browser/runtime hardening, workflows, Zalo Vitest splitting, and memory-core/browser test stabilization.

## Best Possible Solution

Keep the Claw mission-control work in active maintainer review, split unrelated browser/memory/Zalo/CI stabilization where practical, preserve current guarded-networking invariants, and land a mission backbone only after its relationship to Task Flow, gateway events, runtime recovery, Control UI behavior, and security-sensitive paths is explicitly reviewed and tested.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve HTTP/1-only pinned dispatchers:** `src/infra/net/ssrf.ts:430-431`
  - body: `createPinnedDispatcher` now constructs Undici dispatchers directly instead of using the `createHttp1*` helpers that force `allowH2: false`. Guarded fetch dispatcher overrides rely on HTTP/1-only behavior, so this can re-enable HTTP/2 on SSRF-guarded pinned paths and weaken the existing protection.
  - confidence: 0.88
- **[P1] Isolate recovery before mission cycles:** `src/claw/runtime.ts:76`
  - body: `drainClawRuntimeQueue` awaits recovery before running mission cycles and only marks recovery complete after success. If one bad mission throws during recovery, every wake exits through the shared catch and no queued or active mission can progress.
  - confidence: 0.86
- **[P2] Reload Claw dashboard only when needed:** `ui/src/ui/app-gateway.ts:424-425`
  - body: The gateway event handler reloads the full Claw dashboard for every `claw.*` event on every connected UI, even outside the Claw tab. Runtime cycles can emit several Claw events, causing avoidable repeated mission/audit/artifact RPCs in unrelated views.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a guarded-networking regression by bypassing HTTP/1-only Undici dispatcher helpers in the SSRF pinned dispatcher path.

Concerns:

- **[high] Pinned SSRF dispatchers may re-enable HTTP/2:** `src/infra/net/ssrf.ts:430`
  - body: Current main centralizes `allowH2: false` in the `createHttp1*` dispatcher helpers because guarded fetch overrides are not reliable on HTTP/2 paths. The PR's raw `Agent`, `EnvHttpProxyAgent`, and `ProxyAgent` construction in `createPinnedDispatcher` can drop that protection for direct and proxy pinned-dispatcher requests.
  - confidence: 0.88
