## Summary

Keep open. Current main still renders the chat delete confirmation as a static absolute popover above the trigger with no viewport flip or clamp logic, the latest release tag has the same relevant implementation, and the related placement-fix PRs in the supplied context were closed unmerged while the remaining open related PR covers listener cleanup rather than overflow placement.

## What This Changes

The issue asks for the chat delete confirmation popover to stay within the page bounds, with a separate suggestion to expand Chinese localization coverage.

## Best Possible Solution

Add viewport-aware placement for the Control UI chat delete confirmation: measure the trigger and popover after insertion, choose above or below based on available space, clamp horizontally within the viewport, preserve skip-confirm/cancel/delete/outside-click behavior, and cover top-edge, bottom-edge, and narrow-width cases in focused tests. Keep the broader localization request separate from this bug fix.

## Reproduction Assessment

Yes. Clicking a chat message delete button near the top or edge of the clipped chat thread maps directly to current main: the popover is absolutely positioned above the trigger inside the message layout with no viewport measurement or flip/clamp path.

## Solution Assessment

Yes. A narrow viewport-aware placement fix in `ui/src/ui/chat/grouped-render.ts` and `ui/src/styles/chat/grouped.css` is the maintainable solution; the reporter’s workaround of removing absolute positioning may avoid this overflow but risks putting the confirmation back into document flow and shifting chat layout.
