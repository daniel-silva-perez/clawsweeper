## Summary

Keep open. The account-scoped reload idea is valid and current main still does whole-channel planning for account-only channel config paths, but this PR is not merge-ready because the new account restart branch bypasses current active-work deferral and the new reload-plan field is not handled by all reload consumers.

## What This Changes

The PR adds `restartChannelAccounts` to `GatewayReloadPlan`, detects `channels.<channel>.accounts.<accountId>.*` paths, routes them through account-aware channel stop/start calls, and adds gateway reload tests.

## Best Possible Solution

Land a rebased account-scoped reload implementation that keeps full-channel precedence for mixed/global changes, applies the same active-work deferral to both account-scoped and whole-channel reloads, updates `secrets.reload` and typed reload-plan helpers, and adds focused regression coverage for hot reload and secrets reload paths.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: on current main, `channels.telegram.accounts.botA.token` matches Telegram's broad `channels.telegram` reload prefix and is added to `restartChannels`, while the channel manager already supports `startChannel` and `stopChannel` with an optional account id.

## Solution Assessment

No. The requested direction is the right narrow gateway fix, but this PR is not the best mergeable solution until it preserves reload deferral semantics and updates all consumers of the expanded `GatewayReloadPlan` contract.
