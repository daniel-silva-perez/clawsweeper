## Summary

Keep open. The account-scoped reload direction addresses a real current-main over-broad restart path, but the PR is not mergeable as-is because the new account restart branch bypasses current reload deferral semantics and the new reload-plan field is not handled by the secrets reload consumer.

## What This Changes

The PR extends gateway reload planning with account-scoped channel restart state, detects `channels.<channel>.accounts.<accountId>.*` paths, dispatches channel stop/start with an account id, and adds gateway reload tests.

## Best Possible Solution

Land a corrected account-scoped reload implementation that keeps full-channel precedence for global or mixed changes, applies the same active-work deferral to account-scoped and whole-channel reloads, updates every reload-plan consumer including secrets reload, and includes focused regression coverage for both hot reload and secrets reload.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: on current main, `channels.telegram.accounts.botA.token` matches Telegram's broad `channels.telegram` reload prefix and becomes a whole-channel `restartChannels` entry even though the channel manager already supports account-specific stop/start.

## Solution Assessment

No. The requested direction is the narrow maintainable fix, but this PR is not the best mergeable solution until the account restart path preserves existing reload deferral semantics and every `GatewayReloadPlan` consumer handles the new account-scoped state.
