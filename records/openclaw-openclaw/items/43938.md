## Summary

Keep open. The underlying gateway reload problem is valid and this PR is pointed in the right direction, but the proposed patch is not mergeable as-is because the new account-scoped restart path bypasses existing active-work deferral and does not update the separate secrets reload consumer of `GatewayReloadPlan`.

## What This Changes

The PR adds account-scoped channel restart state to gateway reload planning, detects `channels.<channel>.accounts.<accountId>.*` paths, dispatches `stopChannel`/`startChannel` with an account id, and adds gateway reload tests.

## Best Possible Solution

Land a corrected account-scoped reload implementation on current main: keep full-channel precedence for global or mixed changes, apply the same active-work deferral to account-scoped and whole-channel reloads, update secrets reload and all reload-plan fixtures/helpers, and cover hot reload plus secrets reload with focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: on current main, `channels.telegram.accounts.botA.token` matches Telegram's broad `channels.telegram` reload prefix and becomes a whole-channel `restartChannels` entry even though the channel manager already supports account-specific start/stop.

## Solution Assessment

No. The requested direction is the narrow maintainable fix, but this PR is not the best mergeable solution until account restarts preserve reload deferral semantics and every `GatewayReloadPlan` consumer handles the new account-scoped state.
