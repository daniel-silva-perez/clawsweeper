## Summary

Keep open. Current main still has the coarse account-config-to-whole-channel restart behavior, and this PR is aimed at the right narrow gateway fix, but the proposed patch has two merge-blocking gaps: account-scoped restarts bypass the existing active-work deferral and the expanded reload-plan contract is not handled by all consumers.

## What This Changes

The PR adds account-scoped channel restart planning for `channels.<channel>.accounts.<accountId>.*`, dispatches those restarts through account-aware channel stop/start calls, and adds gateway reload tests.

## Best Possible Solution

Land a corrected account-scoped reload implementation that keeps full-channel precedence for mixed/global changes, applies the same active-work deferral to account-scoped and whole-channel reloads, updates secrets reload and typed reload-plan helpers, and adds focused regression coverage for hot reload plus secrets reload.

## Reproduction Assessment

Yes. A source-level reproduction is enough: on current main, `channels.telegram.accounts.botA.token` matches Telegram's broad `channels.telegram` reload prefix and becomes a whole-channel restart even though the channel manager already supports account-specific start/stop.

## Solution Assessment

No. The requested direction is the narrow maintainable fix, but this PR is not the best mergeable solution until the account restart path preserves existing reload deferral semantics and every `GatewayReloadPlan` consumer handles the new account-scoped field.
