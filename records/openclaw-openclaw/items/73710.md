## Summary

Keep this PR open. The linked iOS composer bug is still real on current main, and the PR is useful, but the latest branch still routes a documented longest-edge policy through a width-only transcoder path, so portrait images can remain taller than the promised 1600 px cap. The branch also needs a current-main CI/rebase pass before maintainer review can treat it as ready.

## What This Changes

The PR adds chat-side JPEG processing for iOS attachments, wires `OpenClawChatViewModel` to process images off the MainActor before storing pending attachments, extends `JPEGTranscoder` with alpha flattening, and adds `ChatImageProcessorTests` coverage for resize, byte budget, metadata stripping, invalid data, and alpha handling.

## Best Possible Solution

Land a corrected version that clamps the true oriented long edge, keeps metadata stripping and alpha flattening centralized in the image transcoder path, adds portrait coverage for both tall images and already-narrow tall images, rebases onto current main, and validates the OpenClawKit Swift tests plus an iPhone composer/send round trip before closing the linked bug.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clamp the actual long edge:** `apps/shared/OpenClawKit/Sources/OpenClawKit/ChatImageProcessor.swift:54`
  - body: `maxDimension` is documented and tested as a longest-edge cap, but this passes it as `maxWidthPx`. Because `JPEGTranscoder` only scales from oriented width, portrait images can still return around 1600x2133, and tall images whose width is already <=1600 may avoid resizing entirely. Please compute the scale from `max(orientedWidth, orientedHeight)` or add a true long-edge transcoder mode with portrait coverage.
  - confidence: 0.96
- **[P3] Remove the unreachable 5 MB post-check:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:1160-1163`
  - body: `processForUpload` maps `JPEGTranscodeError.sizeLimitExceeded` into an error before returning, so `processed.count > 5_000_000` cannot be reached through the current code path. The comment also describes a best-effort return path that the PR does not implement, which can mislead future maintenance of the attachment gate.
  - confidence: 0.86
