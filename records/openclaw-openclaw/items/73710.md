## Summary

Keep open. Current main still rejects raw iOS chat image attachments before any client-side resize/re-encode path, and the PR is useful work toward #68524, but the latest patch still routes the chat “longest edge” policy through JPEGTranscoder’s width-only `maxWidthPx` contract, so portrait and narrow-tall images can exceed the promised 1600 px long edge. The PR also needs a small cleanup for the unreachable 5 MB post-check and a required changelog entry before merge.

## What This Changes

The PR adds iOS chat attachment preprocessing that converts selected images to resized JPEGs with source metadata stripped, runs that processing off the MainActor, adds alpha flattening to JPEGTranscoder, and adds Swift unit tests for the new image processor.

## Best Possible Solution

Land a corrected branch that gives chat uploads a true longest-edge cap, keeps metadata stripping and alpha flattening centralized in JPEGTranscoder, covers portrait and narrow-tall fixtures in tests, adds the required changelog entry, and preserves #68524 as the linked issue until the corrected PR merges.

## Reproduction Assessment

Yes. A source-level reproduction is any portrait or narrow-tall image whose oriented width is at or below 1600 while height is greater than 1600, or a 3000x4000 portrait where width-based scaling can produce about 1600x2133; the contributor’s posted device evidence also matches that width-only path.

## Solution Assessment

No, not yet. Reusing JPEGTranscoder off the MainActor is the right maintainable direction, but the patch needs an additive true-long-edge mode or equivalent adapter calculation, plus cleanup of the misleading safety-net comment, before it matches the stated behavior.
