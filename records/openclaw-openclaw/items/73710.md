## Summary

Keep this PR open. Current main still rejects iOS/shared chat image attachments over 5 MB before any resize or metadata stripping, then sends accepted attachment bytes as base64 from the MainActor path. The PR is a focused implementation candidate for the linked open iOS composer bug #68524, and its latest diff addresses the earlier bot-review concerns with alpha flattening, best-effort JPEG encoding, and metadata leak tests.

## Best Possible Solution

Review and refine this PR as the candidate fix for #68524: wire client-side resize/compression and metadata stripping into `OpenClawChatViewModel.addImageAttachment`, reconcile or intentionally separate it from `JPEGTranscoder`, keep the added processor tests, and merge only after the remaining ImageIO/memory and comment-consistency concerns are resolved.
