## Summary

Keep open. Current main still has the raw iOS chat attachment path, and this PR remains the active implementation for the linked large-image composer bug. The latest diff is close but has a blocking resize mismatch: the new chat policy says 1600 px long-edge, while it calls the existing width-only JPEGTranscoder cap, so common portrait photos can still exceed the documented long-edge limit.

## What This Changes

The branch adds ChatImageProcessor, routes chat image attachments through off-MainActor JPEGTranscoder processing before storage/send, adds alpha flattening to JPEGTranscoder, and adds Swift unit tests for resize, payload budget, metadata stripping, non-image rejection, and alpha flattening.

## Best Possible Solution

Land a corrected version that clamps the actual oriented long edge, keeps the metadata-stripping and alpha-flattening behavior, adds portrait-oriented coverage that would fail for 1600x2133-style output, and validates OpenClawKit Swift tests plus a real-device composer/send path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clamp the actual long edge:** `apps/shared/OpenClawKit/Sources/OpenClawKit/ChatImageProcessor.swift:54`
  - body: `ChatImageProcessor` documents `maxDimension` as the longest edge cap and the PR body/tests claim long-edge <= 1600, but this passes the value to `JPEGTranscoder.maxWidthPx`. That helper only scales when the oriented width exceeds the limit, so portrait photos such as 3000x4000 can still come out around 1600x2133, missing the intended memory/payload cap. Please compute the scale from `max(orientedWidth, orientedHeight)` or add a real long-edge transcoder mode and cover portrait inputs in the tests.
  - confidence: 0.88
