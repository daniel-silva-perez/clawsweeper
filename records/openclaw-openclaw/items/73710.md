## Summary

Keep this PR open. Current main still lacks the iOS chat attachment processing path: picker data is passed raw into `addImageAttachment`, the 5 MB gate runs before any resize/transcode, attachments are stored with original bytes and MIME type, and send still base64-encodes those bytes from the `@MainActor` view model. The live PR is open, draft, unmerged, explicitly closes the still-open #68524, and the author says post-review code still needs real-device validation. The PR diff is relevant rather than obsolete; the main remaining work is maintainer review, rebase/device validation, and checks around the shared `JPEGTranscoder` change.

## Best Possible Solution

Keep this PR open as the draft implementation candidate for #68524. The best path is to rebase onto current main, complete physical-device iOS validation, review the `JPEGTranscoder` alpha/metadata behavior and tests, run the targeted Swift checks, then merge or close based on maintainer review. #73711 can continue separately as the UI-side companion because it does not replace this image processing pipeline.
