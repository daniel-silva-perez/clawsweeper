## Summary

Keep this issue open. Current main has subagent concurrency, timeout, nesting-depth, active-child, auto-archive, and liveness controls, but I did not find a durable per-session total spawn cap equivalent to the requested `maxPerSession`/`maxTotal`. The closest existing control, `maxChildrenPerAgent`, is explicitly an active-children limit, so it would not stop the reported sequential pattern of many runs over one long session.

## Best Possible Solution

Keep the issue open and implement a real lifetime/session budget if maintainers want this behavior: add a documented `agents.defaults.subagents.maxPerSession` or `maxTotal` config field, validate it in the config schema/types/generated docs, track a durable count by requester session key across native and ACP spawn paths, reject further `sessions_spawn` calls with a clear tool error once the cap is reached, and add focused tests covering sequential completed children as well as concurrent active children.
