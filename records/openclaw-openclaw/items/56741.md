## Summary

Keep open. Current main still ignores Telegram topic/group `groupPolicy` overrides in native command authorization, while this PR's one-line change aligns native command handling with the existing regular-message policy path. I found no blocking correctness or security issue in the proposed patch; the main residual gap is missing regression coverage for the topic override case.

## What This Changes

This PR flips Telegram native command authorization to pass `useTopicAndGroupOverrides: true` when evaluating group policy access for native and plugin commands.

## Best Possible Solution

Land this patch or an equivalent current-layout change, ideally with a focused regression test that exercises a Telegram forum topic where `topicConfig.groupPolicy` or `groupConfig.groupPolicy` disables native commands while account-level policy would otherwise allow them.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure a Telegram forum topic with `groupPolicy: "disabled"` while the account fallback allows groups, then send `/reset`; current main's command path ignores the topic override because it passes `useTopicAndGroupOverrides: false`.

## Solution Assessment

Yes. Reusing the existing policy resolver with `useTopicAndGroupOverrides: true` is the narrowest maintainable fix because it matches the regular-message path and does not introduce a separate command-only policy implementation.
