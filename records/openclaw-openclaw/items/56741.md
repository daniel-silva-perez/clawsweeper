## Summary

Keep open: current main still has the Telegram command-policy bypass that this PR targets, and the one-line change is the narrowest apparent fix for making native/plugin slash-command auth use the same topic/group policy resolution as normal Telegram messages. The remaining action is maintainer review and landing, ideally with focused regression coverage and a decision on whether disabled-topic command attempts should be user-visible or silent.

## What This Changes

This PR changes Telegram native/plugin command authorization to call the group-policy resolver with topic/group overrides enabled for slash commands.

## Best Possible Solution

Land this patch or an equivalent current-layout change with a targeted regression test for a forum topic or group where the override sets `groupPolicy: "disabled"` while the account-level policy would otherwise allow commands. If maintainers want the linked issue's exact silent-ignore behavior, the same area should distinguish topic/group override-disabled results from global command-disabled results and drop only the former without creating a session.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a Telegram forum topic with `topicConfig.groupPolicy: "disabled"` while account/default policy allows groups, then send `/reset`; current main ignores the topic policy because the command path passes `useTopicAndGroupOverrides: false`.

## Solution Assessment

Yes for the authorization bypass: reusing the existing resolver with `useTopicAndGroupOverrides: true` is the narrowest maintainable fix and matches the regular-message path. The only unresolved review choice is whether the disabled-topic command response should remain user-visible or be silently dropped for the multi-bot scenario.
