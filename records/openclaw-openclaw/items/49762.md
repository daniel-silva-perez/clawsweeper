## Summary

Keep open. Current main still does not show response duration in the per-reply usage footer, and the PR is a focused attempt, but the submitted patch is stale against the current formatter split and has blocking behavior regressions around footer formatting, `/usage tokens`, and timing semantics.

## What This Changes

The PR adds an optional `durationMs` value to the response usage formatter and passes a single run duration from `runReplyAgent` into diagnostics and usage footer formatting.

## Best Possible Solution

Rework the feature on the current `agent-runner-usage-line.ts` surface, preserve existing output when no duration is shown, expose duration only in the intended usage mode, choose an accurate timing boundary or label, and add focused formatter/runner tests plus docs if the visible footer contract changes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main calls `formatResponseUsageLine` without duration, while the PR diff would add duration for every non-off usage mode and alter cost formatting even without duration.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best current patch because it is stale against the formatter split and needs narrower mode gating, preserved separators, clearer timing semantics, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the cost separator when duration is absent:** `src/auto-reply/reply/agent-runner-utils.ts:128-139`
  - body: The new `parts.join(" / ")` path changes existing cost output from ` · est ...` to `/ est ...` even when `durationMs` is omitted. That breaks the stated backward-compatible contract and can regress parsers or snapshots that depend on the existing footer shape.
  - confidence: 0.92
- **[P2] Gate duration out of the tokens-only footer:** `src/auto-reply/reply/agent-runner.ts:595`
  - body: `responseUsageMode !== "off"` covers both `tokens` and `full`, so passing `durationMs` here makes `/usage tokens` include stopwatch metadata. Docs describe that mode as tokens-only; pass duration only for the richer mode or add an explicit new mode.
  - confidence: 0.9
- **[P2] Measure or label the full wall-clock duration accurately:** `src/auto-reply/reply/agent-runner.ts:537-538`
  - body: The sampled value is presented as response duration, but it is taken after delivery-related waits in the runner path. Move the measurement before post-generation delivery drains or label it as end-to-end wall time so users do not read transport/tool-result drain time as model generation time.
  - confidence: 0.82
