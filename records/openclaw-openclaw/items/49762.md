## Summary

Keep open. Current main still does not add response duration to the per-reply usage footer, but this PR is stale against the moved formatter surface and introduces blocking review defects around footer compatibility, `/usage tokens` semantics, and timing accuracy.

## What This Changes

The PR adds an optional `durationMs` formatter parameter and passes a run duration from `runReplyAgent` into diagnostics and the per-reply usage footer.

## Best Possible Solution

Rework the feature on the current formatter helper, preserve existing output when no duration is shown, gate the stopwatch to the intended usage mode, choose an accurate measurement boundary or label, and add focused formatter/runner tests plus docs if the footer contract changes.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the review defects: the PR head changes the formatter delimiter, passes duration for every non-off usage mode, and computes that duration after delivery drains.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best implementation because it is stale against current main and needs clearer footer compatibility, mode gating, timing semantics, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the cost separator when duration is absent:** `src/auto-reply/reply/agent-runner-utils.ts:139`
  - body: The new `parts.join(" / ")` path changes existing cost output from ` · est ...` to `/ est ...` even when `durationMs` is omitted. That breaks the stated backward-compatible contract and can regress parsers or snapshots that depend on the existing footer shape.
  - confidence: 0.92
- **[P2] Gate duration out of the tokens-only footer:** `src/auto-reply/reply/agent-runner.ts:598`
  - body: `responseUsageMode !== "off"` covers both `tokens` and `full`, so passing `durationMs` here makes `/usage tokens` include stopwatch metadata even though docs describe that mode as tokens-only. Pass duration only for the richer mode or introduce an explicit new mode.
  - confidence: 0.9
- **[P2] Measure or label the full wall-clock duration accurately:** `src/auto-reply/reply/agent-runner.ts:540`
  - body: The sampled value is presented as response duration, but it is taken after block reply flushing and pending tool-result delivery drains. Move the measurement before those post-generation waits or label it as end-to-end wall time so users do not read delivery delay as generation time.
  - confidence: 0.82
