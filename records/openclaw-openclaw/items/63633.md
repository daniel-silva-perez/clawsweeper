## Summary

Keep this PR open. It targets a real documented Telegram SOCKS proxy media-download bug, but the branch is still unmergeable against current main and has blocking correctness issues: Telegram media downloads still hit the explicit-proxy SSRF guard before SOCKS dispatcher construction, and the patch advertises SOCKS schemes that undici rejects at runtime.

## What This Changes

The PR adds SOCKS scheme detection and `Socks5ProxyAgent` routing to the shared proxy fetch helper and Telegram explicit-proxy dispatcher, plus proxy-fetch unit coverage.

## Best Possible Solution

Rework the PR around current main: support only the undici-backed `socks5://` and `socks://` protocols for the documented Telegram explicit-proxy media path, allow those schemes through the guarded media policy without weakening proxy-host or Telegram-target checks, preserve the current Telegram dispatcher pool/TLS/fallback behavior, and remove unsupported SOCKS4/SOCKS5H claims unless they are implemented with a dependency that actually supports them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Allow SOCKS through the media guard:** `extensions/telegram/src/fetch.ts:276`
  - body: The reported Telegram media path calls `fetchRemoteMedia()` and then `fetchWithSsrFGuard()` before this dispatcher is used. Since the PR does not update the explicit-proxy guard, `socks5://...` still fails with `Explicit proxy URL must use http or https`, so the user-visible media bug remains.
  - confidence: 0.93
- **[P2] Restrict SOCKS schemes to undici-supported protocols:** `src/infra/net/proxy-fetch.ts:20`
  - body: This allowlist routes `socks4://`, `socks4a://`, and `socks5h://` into `Socks5ProxyAgent`, but undici 8.1.0 accepts only `socks5://` and `socks://`. The added mock tests cannot catch the production runtime rejection, so the PR should stop advertising those schemes or add real support.
  - confidence: 0.9
