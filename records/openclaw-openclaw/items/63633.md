## Summary

Keep this PR open. The linked Telegram SOCKS media bug is still present on current main because the media download path carries the explicit proxy policy into the SSRF guard, and that guard still rejects non-http(s) proxy URLs before a dispatcher can be used. The branch is useful but not merge-ready: undici 8.1.0 already routes `socks5:`/`socks:` through `ProxyAgent`, while the PR still claims unsupported `socks4:`, `socks4a:`, and `socks5h:` handling and does not update the guarded media path.

## What This Changes

The PR branch adds SOCKS scheme detection to the shared proxy fetch helper and Telegram explicit-proxy dispatcher, imports undici `Socks5ProxyAgent`, and adds unit coverage for SOCKS proxy fetch routing and metadata reuse.

## Best Possible Solution

Rework the PR around the current main architecture: support dependency-backed `socks5://` and `socks://` for the documented Telegram explicit-proxy media path by allowing those schemes through the SSRF guard, keep the existing proxy host validation and Telegram target allowlist, preserve the Telegram HTTP/1/pool/TLS/pinned lookup behavior, and remove or separately implement unsupported SOCKS4/SOCKS5H claims.
