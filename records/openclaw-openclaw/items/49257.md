## Summary

Keep open. Current main has post-connect Debug and Logs views plus better single connection error text, but it still lacks the requested session-only browser diagnostics mode for pre-auth WebSocket lifecycle, handshake, close/error history, and copyable logs; the linked implementation candidate is closed unmerged.

## What This Changes

The issue requests a Control UI session-only diagnostics mode that captures and displays WebSocket connection, handshake, close, and error details for troubleshooting failed gateway connections.

## Best Possible Solution

Keep this as the active tracker and implement a Control UI-owned ephemeral diagnostics surface integrated with GatewayBrowserClient/app-gateway. It should capture sanitized WebSocket open/error/close/retry events, connect.challenge/connect response metadata, structured connect errors, close codes/reasons, and post-auth logs.tail snippets when available, then expose them through a copyable drawer backed only by memory or session-scoped storage.

## Reproduction Assessment

Not applicable as a conventional bug reproduction because this is a feature request. We do have high-confidence source verification of the missing behavior: disconnected rendering gates the dashboard, connect.challenge is consumed internally, and Debug/Logs controllers require a connected client.

## Solution Assessment

Yes, the requested Control UI-owned diagnostics mode is the narrowest maintainable solution, provided it includes strict redaction. The safer alternative to avoid is relying only on logs.tail, because that cannot diagnose failures before auth/connect succeeds.
