## Summary

Keep this issue open. Current main has a post-auth Debug tab, a post-auth Logs tab, and better single-message connection errors, but it does not implement the requested ephemeral Control UI connection diagnostics mode. Pre-auth failures still cannot open `logs.tail`, the WebSocket client does not retain a browser-side lifecycle/handshake trace, `connect.challenge` is consumed internally rather than shown in the event log, and the UI has no copyable overlay/drawer for connection diagnostics. The linked PR #49274 was closed unmerged, and local history has no commit referencing this issue or that PR.

## Best Possible Solution

Keep this issue as the active tracker. The best fix is a Control UI-owned ephemeral diagnostics mode that records WebSocket open/error/close/retry events, connect challenge/connect response metadata, structured connect error details, close codes/reasons, and relevant post-auth `logs.tail` entries when available, stored only in memory or session-scoped storage and exposed through a copyable diagnostics drawer. A refreshed PR can reuse the closed #49274 intent, but should integrate with the current `GatewayBrowserClient` and existing Debug/Logs surfaces.

- The eventual implementation must avoid persisting handshake/auth diagnostics to localStorage or IndexedDB, and should redact gateway tokens, device tokens, signatures, and other auth material before copy/export.
