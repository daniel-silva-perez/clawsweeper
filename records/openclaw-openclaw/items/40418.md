## Summary

Keep open. Current main partially covers the reset-preservation problem with a bundled `session-memory` hook and startup-context preload, but it still stores bounded raw user/assistant turns in slugged Markdown files and does not implement the requested synthesis, linked transcript receipt, or compact continuation package. Open PR #50584 is still a live implementation candidate, and open #69943 makes the sanitization/privacy design important before expanding the feature.

## What This Changes

The issue requests automatic `/new` session-memory preservation with saved state, full transcript receipts, and synthesized daily memory for continuity across sessions.

## Best Possible Solution

Land or replace #50584 with an opt-in, bounded design that synthesizes durable decisions, TODOs, preferences, and receipts into compact startup memory; records links to full transcripts without injecting them by default; sanitizes unsafe role/tool/control-token artifacts; and updates tests/docs for dedupe-aware startup loading.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: trigger `/new` or `/reset` with `session-memory` enabled and current main will write recent raw user/assistant text to a slugged memory file, while startup context only reloads bounded notes. No live run was attempted because this was a read-only review.

## Solution Assessment

Unclear as written. The layered summary-with-receipts direction from the discussion is safer than unconditional full transcript promotion, and the current open PR or a maintainer-approved replacement should resolve the remaining design and sanitization choices.
