## Summary

Keep open. Current main still lacks agent-level suspension, and this PR remains the active implementation path, but the branch is not merge-ready because it still misses scheduler enforcement and introduces routing/JSON-output regressions. The author’s follow-up addressed the initial disabled-entry data-loss findings, so the right next step is a targeted branch refresh rather than closure.

## What This Changes

The PR adds an optional `agents.list[].enabled` flag, `openclaw agents disable/enable` commands, disabled status in agent list output/JSON, config schema/docs/baseline/changelog updates, and regression tests around agent filtering, routing, and config preservation.

## Best Possible Solution

Land a refreshed implementation that defines a typed, schema-validated `agents.list[].enabled?: boolean` contract, keeps disabled agents visible to management operations, excludes them consistently from routing and background execution, preserves machine-readable CLI output, updates generated docs, and adds regression coverage for config preservation plus scheduler suspension.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor disabled agents in cron and heartbeat:** `src/config/types.agents.ts:62-64`
  - body: The new contract documents `enabled: false` as fully suspending an agent, but the branch does not update `src/gateway/server-cron.ts` or `src/infra/heartbeat-runner.ts`. Both current paths iterate raw `cfg.agents.list`, so cron jobs and heartbeat settings for disabled agents continue running.
  - confidence: 0.9
- **[P2] Route empty enabled sets to the fallback:** `src/routing/resolve-route.ts:140-142`
  - body: After filtering disabled agents out of the lookup, `pickFirstExistingAgentId` still returns the requested ID when the lookup map is empty. If every configured agent is disabled and a binding targets one of them, routing continues to that disabled ID instead of falling back.
  - confidence: 0.87
- **[P2] Keep disable and enable JSON output parseable:** `src/commands/agents.commands.disable.ts:58-60`
  - body: `agents disable --json` and `agents enable --json` call `logConfigUpdated(runtime)` before `writeRuntimeJson`, which sends a human update line to stdout ahead of the JSON payload. Guard that log in JSON mode like existing agent commands do.
  - confidence: 0.88
