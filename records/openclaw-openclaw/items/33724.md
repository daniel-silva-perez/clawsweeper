## Summary

Current main still uses a fixed adjacent five-file `.bak` ring for config writes, with no typed config/schema/UI surface for backup directory, retention count, or trigger policy. The two direct implementation PRs were closed unmerged, and the overlapping open issue #20460 is narrower rather than a complete replacement for this request.

## Best Possible Solution

Keep this open for maintainer product decision or implementation. A complete fix likely belongs in core config ownership: add typed config/schema/help/docs/UI support for backup directory, retention count, and trigger policy; thread that policy through `src/config/backup-rotation.ts`, `src/config/io.ts`, `src/config/mutate.ts`, recovery/audit code, and tests; preserve current defaults and permission hardening. If maintainers prefer not to add this product surface, close only after an explicit maintainer rejection and document the cron/archive workaround.

## Risks / Open Questions

- Implementing this touches sensitive config copies: custom directories must preserve owner-only permissions, avoid accidental secret exposure, and keep cleanup bounded to OpenClaw-managed backup files.
- Recovery and audit code currently assume adjacent `.bak`; changing the backup location/count without updating recovery paths could make last-known-good restore unreliable.
- The cron plus `openclaw backup create --only-config` workaround can help power users, but it is a separate archive/scheduling workflow and does not solve automatic config-write backup retention or root clutter.
