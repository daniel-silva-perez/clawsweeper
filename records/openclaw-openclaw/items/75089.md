## Summary

Keep open. The PR addresses a real Gateway buffer-retention gap, but the proposed agent-side tool-start cleanup introduces a blocking regression: it can delete still-active tool start records by age alone, so a later matching tool end loses hook params, duration, and start-arg-dependent bookkeeping.

## What This Changes

The PR centralizes Gateway chat buffer cleanup through `ChatRunState.clearRun`, tracks `bufferUpdatedAt` for raw-only streamed buffers, adds stale raw-buffer maintenance cleanup, and bounds agent `toolStartData` with TTL/max-entry sweeping plus tests.

## Best Possible Solution

Keep the Gateway `clearRun`/`bufferUpdatedAt` direction, but revise the tool-start cleanup so active starts are preserved until their matching end event or until the owning run is known inactive; then cover a long-running active tool that exceeds the TTL before ending.

## Reproduction Assessment

Yes for the review finding. In the patched code, start one tool, advance time beyond 30 minutes, start any other tool so the global sweep runs, then end the first tool; `handleToolExecutionEnd` will no longer find `startData`.

## Solution Assessment

No. The Gateway buffer cleanup is a reasonable narrow fix, but the tool-start TTL is not the best way to solve unmatched starts because it cannot distinguish stale records from legitimate long-running active tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve active tool starts beyond the TTL:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:656`
  - body: The new sweep runs globally on every tool start and deletes any `toolStartData` older than 30 minutes. A legitimate long-running tool in another run can exceed that age; when its matching end event later arrives, `handleToolExecutionEnd` has no `startData`, so `after_tool_call` gets empty params/no duration and start-arg-dependent cron or messaging bookkeeping is skipped. Please only reap records for inactive/unmatched runs, or otherwise avoid deleting active starts before their end event.
  - confidence: 0.87
