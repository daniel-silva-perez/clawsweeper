## Summary

This should stay open. Current main and the latest release still configure the Google Gemini CLI backend with `--skip-trust` plus JSON output/prompt flags, and I did not find a current implementation that maps OpenClaw's YOLO exec policy to Gemini CLI's documented `--yolo` / `--approval-mode yolo` tool-approval mode. The reporter's follow-up is actionable because they say adding `--yolo` fixes the tools and memory symptoms; the separate 100% CPU/Matrix part still needs logs or a live reproduction before it can be called fixed.

## What This Changes

The issue reports a regression where an OpenClaw Matrix setup using the Google Gemini CLI backend on Node v25.9 hits 100% CPU, cannot use VDS tools, and leaves long-term memory files empty until the reporter manually adds Gemini CLI `--yolo`.

## Best Possible Solution

Add a provider-local Google Gemini CLI permission-mode normalizer that follows the same OpenClaw exec-policy boundary as the Claude CLI backend, preserves explicit user backend args, and then ask for Matrix CPU logs or a retest if resource exhaustion persists after tools and memory work again.
