## Summary

Keep this PR open for maintainer review rather than closing it as obsolete. Current main already covers the zero-valued poll-default slice and no longer exposes generic `components` on the main message tool schema, but the PR still changes poll intent detection more broadly than current validated behavior and puts Discord empty-modal cleanup in an old shared runner path instead of the live Discord parser/runtime path.

## What This Changes

The PR modifies shared message send handling, poll parameter detection, component sanitization, and related tests to strip auto-populated poll and Discord component fields from `action="send"` requests.

## Best Possible Solution

Use the useful context from this PR, but adapt the fix to current main: preserve the shipped zero-default poll behavior unless maintainers intentionally broaden send sanitization, handle empty Discord modal skeletons in the Discord parser/runtime or schema surface, and add focused regressions for #42820, #43015, and the current default-filled parameter reports.

## Reproduction Assessment

Yes. The poll regression is reproducible through existing send-validation tests on current main, and the Discord empty-modal path is source-reproducible by passing a non-empty `components` object with `modal.fields: []` into the Discord `sendMessage` runtime path.

## Solution Assessment

No. The proposed patch is useful source material, but it is not the best current solution because it weakens validated poll-intent behavior and applies Discord-shaped cleanup in shared core code after the generic message schema has moved to `presentation`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve non-question poll intent validation:** `src/poll-params.ts:52-57`
  - body: Changing `hasPollCreationParams()` to only look at `pollQuestion` means sends with `pollOption`, `pollDurationSeconds: "60"`, `pollPublic: true`, or other non-zero poll params pass the guard and are then stripped into a plain send. Current main intentionally rejects those cases, with tests covering the behavior, so keep zero-default tolerance without treating every non-question poll field as noise.
  - confidence: 0.86
- **[P2] Move empty modal cleanup into Discord parsing:** `src/infra/outbound/message-action-runner.ts:724`
  - body: `stripEmptyComponents(params)` runs in the old shared message runner, but current main's live empty-modal failure is in the Discord `sendMessage` runtime that reads `ctx.params.components` and calls `readDiscordComponentSpec`. Rebasing this core cleanup would not cover that direct Discord path and would mutate a generic `components` param before plugin dispatch, so the empty-skeleton handling should live in the Discord parser/runtime with focused tests.
  - confidence: 0.8
