## Summary

Keep this PR open for maintainer review. It contains useful context for #42820 and #43015, but current main only ships the narrower zero-default poll fix, while this branch still weakens non-question poll intent validation and puts Discord empty-component cleanup in shared message handling instead of the Discord parser/runtime path that still accepts native `components`.

## What This Changes

The PR changes message send handling to strip poll params and empty component skeletons on `action="send"`, changes poll intent detection to require `pollQuestion`, and updates outbound poll/component tests.

## Best Possible Solution

Keep the shipped zero-default poll behavior unless maintainers explicitly choose broader sanitization, and address the remaining Discord empty-skeleton problem in `extensions/discord/src/components.parse.ts` or the Discord runtime with focused tests. Shared rich sends should stay on the generic `presentation` surface rather than reintroducing generic native `components` cleanup in core.

## Reproduction Assessment

Yes. The poll regression is covered by current send-validation tests that reject non-zero poll params on `send`, and the Discord empty-modal path is source-reproducible through `handleDiscordMessageSendAction()` calling `readDiscordComponentSpec()` with `modal.fields: []`.

## Solution Assessment

No. The PR is useful source material, but it is not the best current fix because it broadens poll stripping beyond the shipped zero-default case and handles Discord-shaped component cleanup in shared core code instead of the Discord parser/runtime path that still consumes native components.

Full review comments:

- **[P2] Preserve non-question poll intent validation:** `src/poll-params.ts:54-57`
  - body: Changing `hasPollCreationParams()` to return true only for non-empty `pollQuestion` means sends with `pollOption`, `pollDurationSeconds: "60"`, `pollPublic: true`, or negative durations are treated as noise and then stripped into a plain send. Current main intentionally preserves those non-zero/true values as poll intent and has send-validation coverage for the corrective error, so keep zero/default tolerance without silently dropping explicit non-question poll fields.
  - confidence: 0.88
- **[P2] Handle empty Discord components in the parser:** `src/infra/outbound/message-action-runner.ts:724`
  - body: `stripEmptyComponents(params)` only runs in the shared message runner before generic send dispatch. The current failing Discord path still reads native `ctx.params.components` in `sendMessage` and calls `readDiscordComponentSpec`, so direct Discord actions and legacy Discord component payloads can still throw on an empty modal skeleton. Move the empty-spec tolerance into the Discord parser/runtime and leave shared sends on `presentation`.
  - confidence: 0.8
