## Summary

Keep this PR open for maintainer review. It addresses real poll/default-schema and Discord component pain, but the current branch would silently discard explicit non-question poll intent and fixes empty Discord component skeletons only in the shared send runner rather than the Discord parser/runtime paths that still throw.

## What This Changes

The PR strips poll parameters and empty component skeletons from `action="send"`, narrows poll-intent detection to `pollQuestion`, and updates poll/outbound message tests.

## Best Possible Solution

Keep the shipped zero-default poll tolerance, preserve corrective validation for explicit non-zero/true poll-shaped send inputs unless maintainers intentionally choose silent stripping, and move empty Discord component skeleton tolerance into the Discord parser/runtime with focused tests.

## Reproduction Assessment

Yes. The poll regression is reproducible from current-main helper and send-validation tests, and the Discord failure is source-reproducible by passing `components.modal.fields: []` through `readDiscordComponentSpec()` or native Discord `sendMessage`.

## Solution Assessment

No. The branch is useful context, but the safer fix is narrower: avoid weakening explicit poll-intent validation and handle empty Discord component specs at the Discord boundary so all consumers are covered.
