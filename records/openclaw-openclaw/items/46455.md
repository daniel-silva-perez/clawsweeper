## Summary

Keep this PR open for maintainer review. Current main only shipped the zero-valued poll-default fix, while this branch still weakens explicit non-question poll intent and handles empty Discord component skeletons in the shared send runner instead of the Discord parser boundary.

## What This Changes

The PR strips poll parameters and empty component skeletons from `action="send"`, narrows poll-intent detection to `pollQuestion`, and updates poll/outbound message tests.

## Best Possible Solution

Keep the shipped zero-default poll tolerance, preserve corrective validation for explicit poll-shaped sends unless maintainers intentionally choose silent stripping, and handle empty Discord component skeletons in the Discord parser/runtime with focused tests.

## Reproduction Assessment

Yes. The poll behavior is reproducible from current-main helper and send-validation tests, and the Discord failure is source-reproducible by passing an empty `components.modal.fields` payload through `readDiscordComponentSpec()`.

## Solution Assessment

No. The branch contains useful context, but the safer path is narrower: keep explicit poll-intent validation unless product direction changes, and move empty component tolerance to the Discord parser boundary so native Discord consumers are covered.
