## Summary

Keep this PR open for maintainer review. It addresses real model auto-population failures, but the branch still weakens current poll-intent validation and handles empty Discord component skeletons in a shared runner path instead of the Discord parser/runtime that consumes native component payloads.

## What This Changes

The PR changes message send handling to strip poll params and empty `components` skeletons on `action="send"`, narrows poll-intent detection to `pollQuestion`, and updates outbound poll/component tests.

## Best Possible Solution

Keep the shipped zero-default poll behavior, preserve corrective validation for explicit non-zero/true poll intent unless maintainers deliberately choose broader silent stripping, and move empty Discord component skeleton tolerance into `extensions/discord/src/components.parse.ts` or the Discord send runtime with focused parser/runtime tests.

## Reproduction Assessment

Yes. Current-main source and tests reproduce the poll side through send-validation cases for non-zero/true poll params, and the Discord side is source-reproducible by passing `components.modal.fields: []` into `readDiscordComponentSpec()` through the native Discord `sendMessage` path.

## Solution Assessment

No. The branch is useful context, but the safer solution is narrower: keep zero/default tolerance without dropping explicit non-question poll intent, and fix empty Discord components at the Discord parser/runtime boundary so every native component consumer gets the behavior.
