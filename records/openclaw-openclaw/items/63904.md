## Summary

Keep open. Current main still lacks outbound Slack `reply_broadcast` plumbing and does not surface inbound broadcast visibility metadata for agents to mirror, while the related implementation PR #64365 is still open and unmerged.

## What This Changes

Request support for detecting Slack broadcast thread replies inbound and mirroring that visibility on outbound Slack thread replies.

## Best Possible Solution

Add the capability in the Slack plugin: typed outbound `replyBroadcast` plumbing through Slack send/action/message-tool paths, Slack API payload emission only for threaded replies, inbound context metadata for `thread_broadcast`, focused regression tests, and Slack docs.

## Risks / Open Questions

- `reply_broadcast` expands Slack visibility to the parent channel, so any fix should default false and only send it on explicit true with a valid parent `thread_ts`.
- An outbound-only change would not satisfy the issue because agents also need inbound `thread_broadcast` visibility surfaced in context.
- If the shared message tool gains this knob, it needs provider-scoped discovery or docs so non-Slack channels are not advertised misleading unsupported behavior.
