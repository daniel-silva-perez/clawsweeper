## Summary

Keep open. Current main still does not implement the requested Slack `reply_broadcast` mirroring: outbound Slack sends only plumb `threadTs`/`thread_ts`, and inbound `thread_broadcast` is accepted as a user message but not surfaced as a broadcast visibility flag agents can mirror. The related PR #64365 is still open per the provided GitHub context, so it remains an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open and review/land PR #64365 or a successor Slack-plugin patch. The complete fix should add typed `replyBroadcast` plumbing through Slack send/action/generic message-tool paths, pass Slack `reply_broadcast` only for threaded replies, expose inbound `thread_broadcast` or equivalent visibility metadata in finalized context/tool context for mirroring, and add focused regression tests plus Slack docs.

## Risks / Open Questions

- Slack `reply_broadcast` widens message visibility to the parent channel; implementation should default false and only send it with an explicit true value plus a valid Slack parent `thread_ts`.
- An outbound-only patch would not fully resolve this issue because the request also needs inbound `thread_broadcast` visibility surfaced to agents/tool context for mirroring.
- Adding a Slack-only knob to the shared message tool needs careful schema/discovery scoping so non-Slack providers are not given misleading unsupported behavior.
