## Summary

Keep this issue open. Current main still lacks outbound Slack `reply_broadcast` plumbing and inbound broadcast visibility metadata, while related PR #64365 remains the implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open and review/land PR #64365 or a successor Slack plugin patch. The complete fix should add typed `replyBroadcast` plumbing through Slack send/action/generic message paths, pass Slack `reply_broadcast` only for threaded replies, expose inbound `thread_broadcast` or equivalent visibility metadata to agents/tool context, and add focused regression tests plus Slack docs.

## Risks / Open Questions

- Slack `reply_broadcast` widens message visibility to the parent channel, so the eventual fix should default false and only send it for explicit true values with a valid Slack `thread_ts`.
- An outbound-only patch would not fully resolve the report because agents also need inbound `thread_broadcast` visibility metadata to mirror the user's choice.
- Adding a Slack-specific broadcast option to shared message tools needs careful schema/discovery scoping so non-Slack channels are not advertised unsupported behavior.
