## Summary

Keep open. Current main still has no native exec-approvals denylist or command-content deny rule, so an allowed shell/interpreter path can still satisfy allowlist matching before any dangerous inner command is rejected. The CRE comments provide useful prior art, while #6615 is broader overlapping design work and #47848 was closed without merge.

## What This Changes

Request adding deny patterns that match the full exec command before allowlist checks so allowed shells cannot run blocked payloads such as gateway stop commands.

## Best Possible Solution

Define an additive exec approvals deny-rule model that is evaluated before broad allow decisions and consistently matches canonical command text, argv, and shell-wrapper inline payloads across gateway and node execution. The end state should update the protocol schema, approvals CLI, exec-policy CLI, Control UI, docs, and focused regression tests while coordinating acceptance criteria with #6615.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure allowlist mode with a shell path such as /bin/zsh allowed, then request a shell-wrapper payload like zsh -c "openclaw gateway stop"; current main has no denylist schema or deny-before-allow branch to reject that command content.

## Solution Assessment

Unclear. Full-command deny patterns evaluated before allowlist are directionally appropriate for the reported bypass, but the best implementation needs maintainer security design around normalization, matching grammar, shell payload extraction, and compatibility with broader denylist work.
