## Summary

Keep open. Current main still has allowlist-oriented exec approval policy with no native denylist, no deny-before-allow branch, and no full-command content denial surface across source, protocol schema, CLI, Control UI, docs, or tests. The linked CRE implementation and closed unmerged PR #47848 are useful prior art, while #6615 overlaps the broader denylist design but does not make this focused shell-wrapper bypass safely obsolete.

## What This Changes

Request to add exec approval deny patterns evaluated against full command content before allowlist matching so allowed shells cannot run blocked payloads.

## Best Possible Solution

Define an additive native exec approvals deny-rule model evaluated before broad allow decisions, consistently matching canonical command text, argv, and shell-wrapper inline payloads across gateway and node execution. Expose the policy through protocol schema, approvals CLI, exec-policy CLI, Control UI, docs, and focused tests, while coordinating the narrower bypass with #6615.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: allowlisting a shell path can satisfy allowlist matching before any inline command-content denial is considered, and there is no denylist config or decision branch to block a payload such as an OpenClaw gateway stop command invoked through that shell.

## Solution Assessment

Unclear. Full-command deny patterns evaluated before allowlist are directionally appropriate for the reported bypass, but the best implementation needs maintainer security design around canonicalization, match grammar, shell payload extraction, UI/CLI/schema compatibility, and overlap with #6615.
