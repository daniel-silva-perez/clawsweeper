## Summary

Keep open. Current main is still allowlist-oriented: there is no native denylist collection, no `security: "denylist"` or `ask: "on-match"` mode, and no pre-allow command-content denial path across policy resolution, protocol schema, CLI, Control UI, or docs. The linked CRE implementation and the closed unmerged PR #47848 are useful context, and #6615 overlaps as broader denylist design, but none of those make this focused shell-wrapper bypass implemented or safely obsolete.

## What This Changes

The issue requests pre-allowlist exec approval deny patterns that match full command strings so dangerous shell-wrapped commands cannot be allowed solely because the shell binary is allowlisted.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, consistently matches canonical command text plus argv and shell payload across gateway and node execution, exposes the policy in approvals CLI and Control UI, documents precedence and limitations, and coordinates the narrower bypass report with #6615.
