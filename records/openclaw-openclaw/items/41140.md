## Summary

Keep open. Current main still lacks native command-content deny rules for exec approvals: the policy model remains allowlist-centered, the gateway/node execution paths only evaluate allowlist satisfaction, and shell-wrapper payloads can still be satisfied by an allowed shell executable before any inner command denial is considered. The CRE comments are useful prior art, while open #6615 is broader and closed-unmerged #47848 does not implement this focused bypass fix.

## What This Changes

The issue requests deny patterns for exec approvals that match full command content before allowlist decisions, so dangerous payloads are blocked even when invoked through an allowed shell.

## Best Possible Solution

Define an additive native exec approvals deny-rule model evaluated before broad allow decisions, consistently matching canonical command text, argv, and shell-wrapper inline payloads across gateway and node execution. Expose the policy through protocol schema, approvals CLI, exec-policy CLI, Control UI, docs, and focused regression tests while coordinating the acceptance criteria with #6615.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: allowlisting a shell path can satisfy allowlist matching before any inline command-content denial is considered, and there is no denylist config or decision branch to block a payload such as an OpenClaw gateway stop command invoked through that shell.

## Solution Assessment

Unclear. Full-command deny patterns evaluated before allowlist are directionally appropriate for the reported bypass, but the best implementation needs maintainer security design around canonicalization, match grammar, shell payload extraction, UI/CLI/schema compatibility, and overlap with #6615.
