## Summary

Keep this issue open. Current main still does not expose Kimi, Kimi Coding, or Moonshot through the provider usage pipeline. The prior PR is useful implementation context but was closed unmerged and had a reported UI provider-aliasing bug, so it is not a shipped fix.

## Best Possible Solution

Implement this in the bundled provider boundary: add or generalize provider usage discovery for kimi, kimi-coding, and moonshot; add Kimi/Moonshot resolveUsageAuth and fetchUsageSnapshot hooks with verified billing/balance endpoint parsing; add focused tests for auth resolution, response normalization, status visibility, and provider alias separation; update https://docs.openclaw.ai/concepts/usage-tracking and the Moonshot/Kimi provider docs. Treat /quota as a separate product decision unless maintainers want a new chat command beyond the existing usage/status surfaces.

## Risks / Open Questions

- Kimi Coding and Moonshot may require different billing credentials and endpoint contracts than inference keys; implementation should verify upstream billing behavior before shipping.
- Reviving the closed prior implementation should avoid the reported z.ai/Kimi aliasing bug and keep provider rows distinct.
- Adding only provider hooks may still leave the feature invisible unless usage-provider discovery is extended or generalized for kimi, kimi-coding, and moonshot.
- A standalone /quota command is not currently documented in main; maintainers should decide whether to add it or satisfy the request through existing /status, openclaw status --usage, Control UI, and usage.status surfaces.
