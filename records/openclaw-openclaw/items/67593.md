## Summary

Keep this issue open. Current main still does not expose Kimi, Kimi Coding, or Moonshot through provider usage/balance surfaces. The bundled providers are present for model use, web search, streaming usage metadata, and cost estimates, but they are not part of the provider-usage snapshot pipeline that feeds /status, openclaw status --usage, models.authStatus, or the Control UI auth/usage card.

## Best Possible Solution

Implement this in the bundled provider boundary: add or generalize provider usage discovery for kimi, kimi-coding, and moonshot; add provider-owned resolveUsageAuth and fetchUsageSnapshot hooks with verified Kimi/Moonshot billing/balance endpoint parsing; add focused tests for auth resolution, response normalization, status visibility, model-auth/Control UI usage enrichment, and provider alias separation; update https://docs.openclaw.ai/concepts/usage-tracking and https://docs.openclaw.ai/providers/moonshot. Treat /quota as a separate maintainer product decision unless the intended outcome is fully covered by existing usage/status surfaces.

## Risks / Open Questions

- Kimi Coding and Moonshot may require different billing credentials and endpoint contracts than inference keys, so implementation should verify upstream billing/balance behavior before shipping.
- Reviving the closed prior PR should avoid the reported z.ai/Kimi aliasing bug and keep provider rows distinct.
- Adding plugin hooks alone is not enough if usage discovery remains tied to the closed UsageProviderId/usageProviders list; discovery must include or generalize kimi, kimi-coding, and moonshot.
- A standalone /quota command is still a product decision; current documented surfaces are /status, /usage, openclaw status --usage, models.authStatus, and Control UI usage/auth panels.
