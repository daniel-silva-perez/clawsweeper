## Summary

Keep open. Current main has manual session reset, one-shot `abortedLastRun` tracking, bounded retry/failover paths, and explicit timeout handling, but it does not persist a consecutive-abort count or automatically reset a same session after a configurable threshold. The related circuit-breaker items remain open or unmerged, so this is still real product work rather than conservative cleanup.

## What This Changes

The issue requests a configurable gateway circuit breaker that starts a fresh session after the same session is aborted N consecutive times.

## Best Possible Solution

Design a gateway/session-owned breaker that records a durable consecutive abort streak per session, resets the streak after a successful non-aborted turn, applies a clearly documented threshold/config default, and invokes the existing session reset service only for the maintainer-approved abort classes. It should include focused tests around abort persistence, next-turn behavior, reset side effects, and interaction with existing retry/failover handling.

## Reproduction Assessment

Yes, by source-level inspection: repeated explicit aborts currently only set the same `abortedLastRun` boolean, and the next prompt consumes and clears that flag without resetting the session. A high-confidence live reproduction would be a focused gateway/session test that aborts the same session past the proposed threshold and observes the unchanged `sessionId`.

## Solution Assessment

No current solution exists on main. The likely maintainable path is not to overload the one-shot `abortedLastRun` hint, but to add a separate durable counter and threshold policy that calls the existing reset service after the chosen abort classes are confirmed.
