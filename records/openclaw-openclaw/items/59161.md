## Summary

Keep open. Current main has one-shot abort markers, manual/session lifecycle resets, targeted recovery for context overflow and transcript corruption, and unrelated plugin/channel breakers, but it does not implement a durable per-session consecutive-abort counter, configurable threshold, or automatic reset trigger. Related PR #42933 and issues #62615/#45389/#52384 are still open, so they are coordination context rather than proof that this narrower request is fixed or superseded.

## What This Changes

The issue requests a gateway circuit breaker that resets a session after a configurable number of consecutive aborts.

## Best Possible Solution

Keep this as a core gateway/session reliability request and coordinate it with the open session-breaker work. A maintainable implementation would add a documented default/per-agent threshold, persist per-session abort streak state, increment it from terminal abort lifecycle paths, clear it on successful runs and explicit resets, use the existing session reset service when the threshold trips, and add focused tests for chat.abort, sessions.abort, and gateway agent runs.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. The missing-capability check is high-confidence: current abort paths only record one run or a one-shot abortedLastRun hint, and current config/session schemas do not contain a consecutive-abort threshold or breaker state.

## Solution Assessment

Yes on the direction, but the exact mechanism is not settled. The narrowest maintainable path is to reuse existing session reset and lifecycle event plumbing with persisted breaker state rather than adding another ad hoc retry loop; PR #42933 is relevant but focuses broader model-error breaker semantics and still needs maintainer review.
