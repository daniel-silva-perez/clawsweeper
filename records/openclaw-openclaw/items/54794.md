## Summary

Keep open: current main still lacks Telegram inline-query UX. The only current inline-query-related code is allowed-update metadata plus BotFather capability/status reporting; there is no inline_query/chosen_inline_result handler, no answerInlineQuery or InlineQueryResult path, and no config/docs/test surface for enabling or constraining inline mode. The related implementation PR #51963 is closed and unmerged, so it is reference material rather than shipped behavior.

## What This Changes

Add Telegram inline query support so users can invoke the Telegram bot as `@botname query` from any chat and return inline results through the Telegram plugin.

## Best Possible Solution

Keep this as the canonical Telegram inline-query tracker. The end state should be a reviewed Telegram plugin implementation with explicit config/schema/docs, conservative authorization and session semantics, bounded request handling, InlineQueryResultArticle responses, chosen-result handling if needed, and focused Telegram plugin tests; #51963 can be mined for reference but should not be treated as shipped.

- Inline queries can originate from arbitrary Telegram contexts, so the feature needs explicit sender authorization, privacy/session boundaries, rate limiting or debounce behavior, and bounded execution deadlines before routing into the agent flow.
- The existing inlineQueries status flag can make operators think OpenClaw answers inline queries when it currently only reports the Telegram bot account capability.
