## Summary

Keep this issue open. Current main still does not implement Telegram inline-query UX: there is no `inline_query` or `chosen_inline_result` runtime handler, no `answerInlineQuery`/`InlineQueryResult` path, and no public Telegram inline-query config or docs surface. The only current references are allowed-update metadata and a status flag that reports the BotFather `supports_inline_queries` capability. Closed PR #51963 remains useful reference material, but it was not merged and is not shipped behavior.

## Best Possible Solution

Keep this issue open as the canonical Telegram inline-query tracker. The likely fix belongs in the bundled Telegram plugin: add explicit config/schema/docs for inline queries, handle `inline_query` and `chosen_inline_result` updates, authorize senders conservatively, route bounded requests through the existing agent flow, answer with Telegram `InlineQueryResultArticle` results, and add focused Telegram plugin tests. Treat closed PR #51963 as reference material unless a new reviewed implementation lands.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram contexts, so a future implementation needs explicit sender authorization, privacy/session boundaries, rate limiting or debounce, and bounded execution deadlines before routing to the agent.
- The existing `inlineQueries=true/false` status flag can mislead operators because it reports the Telegram bot account capability, not OpenClaw's ability to answer inline queries.
