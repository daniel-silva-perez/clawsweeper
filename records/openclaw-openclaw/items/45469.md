## Summary

Keep open: current main still has an unbounded generic GatewayClient reconnect path for non-paused closes, while only narrower auth-failure pauses, timer cleanup, and request timeouts have landed. The direct cap PR was closed unmerged and the later TUI reconnect report was closed as a duplicate of this canonical thread, so cleanup closure would lose the remaining reconnect-policy work.

## What This Changes

The issue requests a bounded reconnect policy for `GatewayClient.scheduleReconnect()` so permanently unreachable gateways do not cause endless reconnect attempts.

## Best Possible Solution

Define an explicit GatewayClient reconnect policy that preserves long-lived recovery where required, gives supervisor-backed or interactive clients a finite terminal path when appropriate, and adds regression tests for retry exhaustion, reset after `hello-ok`, auth-pause behavior, and caller-specific defaults.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to fake a non-auth WebSocket close such as code 1006 or 1012, advance timers, and observe `scheduleReconnect()` repeatedly creating new sockets because current main has no retry counter or cap.

## Solution Assessment

Unclear. The direct “default cap” approach is not the best blanket fix because current node-host and ACP call sites depend on automatic recovery; the safer direction is an explicit/caller-specific reconnect policy or threshold callback with tests.
