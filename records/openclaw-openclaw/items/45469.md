## Summary

Keep open. Current main still has no generic max retry limit in `GatewayClient.scheduleReconnect()`: ordinary closes schedule another `start()` after capped exponential backoff, and only selected auth failures pause reconnect. Related fixes bounded request timeouts and auth retry loops, but the specific reconnect-ceiling request remains unimplemented; the related PR that proposed `maxReconnectRetries` is closed and unmerged.

## Best Possible Solution

Keep this open for maintainer product judgment and a targeted gateway-client fix. The strongest path is to add an explicit reconnect policy surface, likely with bounded attempts or give-up callbacks for supervisor-managed/interactive clients, while preserving long-lived recovery where infinite reconnect is intended. Add regression tests for repeated close/error cycles, listener cleanup, timer cleanup, and any configured max-attempt behavior.

## Risks / Open Questions

- A hard global retry cap could regress the intentional recovery case noted in discussion: a client should reconnect automatically when a gateway returns hours later. The fix should distinguish supervisor-managed node-host behavior, interactive TUI/browser clients, auth failures, and generic network outages.
- The listener/zombie-process claim is not fully proven from source alone, but the narrower reported absence of a max reconnect policy is directly current on main and in the latest release.
