## Summary

Keep open. Current main still has an unbounded generic GatewayClient reconnect path for ordinary WebSocket closes; existing hardening covers pending request timeouts, reconnect-timer cleanup, startup retry delays, and selected terminal auth pauses, but it does not add a retry ceiling or caller-specific reconnect policy.

## What This Changes

The issue asks for a maximum retry limit or equivalent policy for GatewayClient reconnect scheduling to avoid unbounded reconnect loops when the gateway stays unreachable.

## Best Possible Solution

Define an explicit reconnect policy for GatewayClient that preserves indefinite recovery where it is documented, gives UI or supervisor-backed clients a finite terminal path or threshold callback, and covers retry exhaustion, reset after hello-ok, auth-pause behavior, and caller defaults in focused tests and docs.

## Reproduction Assessment

Yes. A high-confidence reproduction is to fake an ordinary WebSocket close such as code 1006 or 1012, advance fake timers, and observe current main repeatedly creating replacement sockets because there is no retry counter or cap.

## Solution Assessment

Unclear. The issue identifies a real unbounded path, but the suggested blanket max-retry default is not clearly the best solution because node-host recovery intentionally relies on long-lived reconnects; a caller-specific policy or threshold callback is safer.
