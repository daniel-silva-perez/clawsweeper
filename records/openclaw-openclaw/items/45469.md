## Summary

Keep open. Current main still has an unbounded generic GatewayClient reconnect path for ordinary WebSocket closes; adjacent hardening covers request timeouts, timer cleanup, startup retry delays, and selected terminal auth pauses, but it does not add a retry ceiling or caller-specific reconnect policy. The directly linked cap PR is closed and unmerged, while the merged request-timeout PR explicitly left reconnect retry policy as follow-up work.

## What This Changes

The issue requests a maximum retry limit or reconnect policy for `GatewayClient.scheduleReconnect()` so permanently unreachable gateways do not trigger endless reconnect attempts.

## Best Possible Solution

Define an explicit GatewayClient reconnect policy that preserves indefinite recovery where it is intentional, gives UI or supervisor-backed clients a finite terminal path or threshold callback, resets budgets after `hello-ok`, keeps auth-pause behavior intact, and documents caller defaults with focused tests.

## Reproduction Assessment

Yes. A high-confidence reproduction is to use the existing GatewayClient socket stub and fake timers, emit repeated ordinary close events such as 1006 or 1012, and observe replacement sockets being created after each scheduled delay with no retry counter or exhaustion state.

## Solution Assessment

Unclear. The issue identifies a real unbounded path, but the suggested blanket max-retry default is not clearly the best solution because node-host recovery intentionally relies on long-lived reconnects; a caller-specific policy or threshold callback is safer.
