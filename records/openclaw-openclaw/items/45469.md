## Summary

Keep open. Current main still has an unbounded GatewayClient reconnect path for ordinary WebSocket closes, while the landed hardening only covers request timeouts, timer cleanup, and specific auth-failure pauses; the direct reconnect-cap PR was closed unmerged and the newer TUI report was closed as a duplicate of this remaining policy work.

## What This Changes

The issue requests a finite or caller-controllable retry limit for `GatewayClient.scheduleReconnect()` so unreachable gateways do not cause permanent reconnect loops.

## Best Possible Solution

Define an explicit GatewayClient reconnect policy that preserves indefinite recovery for long-lived node/backend clients where intended, gives UI or supervisor-backed callers a finite terminal path or threshold callback, and covers retry exhaustion, reset after `hello-ok`, auth-pause behavior, and caller-specific defaults in tests.

## Reproduction Assessment

Yes. A high-confidence reproduction is to fake an ordinary WebSocket close such as code 1006 or 1012, advance fake timers, and observe current main repeatedly creating replacement sockets because there is no retry counter or cap.

## Solution Assessment

Unclear. A direct default max-retry cap is not the safest blanket solution because documented node-host behavior relies on in-process recovery; the maintainable direction is an explicit or caller-specific reconnect policy with tests.
