## Summary

Keep this PR open. The underlying session-memory performance problem still exists on current main, but the branch is stale against the active memory-core plugin layout and still has correctness defects that should block merging as-is.

## What This Changes

The PR adds hash-based incremental indexing for session memory files, offset metadata/schema support, and session benchmark/integration tests under the old `src/memory` layout.

## Best Possible Solution

Port the incremental session-indexing idea into the current memory-core plugin and memory-host SDK boundaries, preserve existing keyword-only and Unicode chunking behavior, update `chunks`, vector rows, and `chunks_fts` consistently, and leave regular `MEMORY.md` indexing behavior unchanged.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the reported behavior: changed session files go through `buildSessionEntry`, then `indexFile`, then `clearIndexedFileData` before reinserting regenerated chunks.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current solution because it targets old `src/memory` files and introduces search/indexing regressions; a narrow replacement against `extensions/memory-core` and `packages/memory-host-sdk` is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve FTS-only session indexing:** `src/memory/manager-embedding-ops.ts:828-833`
  - body: The new session path returns as soon as there is no embedding provider. Current `indexFile` still writes FTS-only chunks in that mode, and session sync switches session files to this method, so users without an embedding provider would lose keyword-searchable session transcripts.
  - confidence: 0.91
- **[P2] Refresh FTS rows when metadata moves:** `src/memory/manager-embedding-ops.ts:930-933`
  - body: The hash-match branch updates line and offset metadata only in `chunks`. Keyword search reads line ranges from `chunks_fts`, so reused chunks after truncation or line shifts can still return stale citations unless the matching FTS row is replaced or updated too.
  - confidence: 0.9
- **[P2] Reuse existing chunk sizing semantics:** `src/memory/internal.ts:506`
  - body: The offset chunker counts raw UTF-16 length, while the current chunker uses weighted sizing and surrogate-safe splitting for CJK-heavy text. Switching session sync to this helper can produce different chunks and hashes for Unicode transcripts, defeating reuse and changing chunk boundaries.
  - confidence: 0.82
- **[P3] Correct the overlap offset rewind:** `src/memory/internal.ts:488-489`
  - body: After `flush()` advances by `currentChars - 1`, `carryOverlap()` subtracts an overlap count that still includes the synthetic newline. With overlap enabled, subsequent `startOffset` values can shift one character early, so persisted offsets no longer point at the chunk text.
  - confidence: 0.76
