## Summary

Keep this PR open, but do not merge it as-is. The underlying performance issue is still present on current main, while this branch is dirty against main, carries a large stale memory diff, and has correctness blockers in the incremental-session path.

## What This Changes

The PR attempts to add hash-based incremental indexing for session transcript files, chunk offset metadata/schema support, and benchmark/integration tests for reduced embedding work.

## Best Possible Solution

Build a narrow replacement against the current memory-core plugin and memory-host SDK: incrementally preserve unchanged session chunks, update chunks/vector/FTS rows consistently, keep MEMORY.md full reindex behavior unchanged, and add focused tests for append growth, FTS-only mode, metadata shifts, Unicode chunking, and retry/idempotence.

## Reproduction Assessment

Yes. A changed session transcript deterministically reaches indexFile on current main, and indexFile clears indexed rows before writing regenerated chunks. I did not run tests because this review was read-only, but the source path is enough to reproduce the full delete/reinsert behavior.

## Solution Assessment

No. The idea is the right direction, but this PR is not the best implementation because it targets stale paths and introduces correctness regressions. The safer path is a clean, narrow re-port into the current plugin-owned memory engine.

  - confidence: 0.91

  - body: The hash-match branch updates line and offset metadata only in chunks. Keyword search reads citation ranges from the FTS table, so reused chunks after truncation or line shifts can still return stale line ranges unless the matching FTS row is replaced or updated too.
  - confidence: 0.9

  - body: chunkMarkdownWithOffset counts raw string length, while the current chunker uses weighted sizing and surrogate-safe splitting for CJK-heavy text. Switching session sync to this helper can change Unicode chunk boundaries and defeat reuse for non-Latin transcripts.
  - confidence: 0.83
