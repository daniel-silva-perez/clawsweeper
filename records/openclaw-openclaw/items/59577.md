## Summary

Keep open. The underlying session-memory performance issue is still present on current main, but this PR should not merge as submitted because it targets stale memory paths and introduces correctness regressions in indexing recovery, FTS-only behavior, FTS metadata, and chunking semantics.

## What This Changes

The PR adds hash-based incremental session-file indexing, chunk offset schema/metadata, and benchmark/integration tests intended to preserve unchanged session chunks instead of fully re-embedding growing session transcripts.

## Best Possible Solution

Port the useful incremental-session idea into the active memory-core plugin and memory-host SDK: preserve unchanged session chunks and embeddings for append growth, keep normal memory files on the existing full-reindex path, and update chunks, vectors, FTS rows, file records, FTS-only mode, retry/idempotence, and Unicode chunking consistently.

## Reproduction Assessment

Yes. A changed session transcript deterministically reaches indexFile on current main, and that write path clears indexed rows before rewriting chunks, which reproduces the reported O(n) delete/reinsert behavior by source inspection.

## Solution Assessment

No. The direction is appropriate, but this PR is not the best implementation because it patches the old src/memory tree and leaves concrete correctness blockers; a clean current-main replacement is the safer solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Port the hook to current memory-core:** `src/memory/manager-sync-ops.ts:912-914`
  - body: This changes the session sync call in the old `src/memory` tree, but current main's active session sync lives under `extensions/memory-core/src/memory`. As submitted, the hook will not affect the active memory engine and should be ported to the current plugin path.
  - confidence: 0.9
- **[P2] Preserve FTS-only session indexing:** `src/memory/manager-embedding-ops.ts:828-833`
  - body: The incremental session method returns immediately when there is no embedding provider. Current `indexFile` still writes keyword-searchable `fts-only` session chunks in that mode, so users without embeddings would lose session transcript search.
  - confidence: 0.91
- **[P1] Embed replacements before pruning indexed rows:** `src/memory/manager-embedding-ops.ts:890-916`
  - body: The incremental path deletes old chunks before requesting embeddings for replacements. A provider timeout or rate limit after the delete can leave the session index partially pruned, whereas the existing full path computes embeddings before mutating indexed rows.
  - confidence: 0.87
- **[P2] Use an upsert for new incremental chunks:** `src/memory/manager-embedding-ops.ts:929-947`
  - body: New incremental chunks are inserted without `ON CONFLICT`, so a retry after a partial sync can hit a duplicate chunk id and fail instead of repairing the stale file record. Match the full-index upsert behavior for idempotence.
  - confidence: 0.86
- **[P2] Refresh FTS rows when metadata moves:** `src/memory/manager-embedding-ops.ts:930-933`
  - body: The hash-match branch updates line and offset metadata in `chunks` without refreshing `chunks_fts`. Keyword search reads citation ranges from the FTS table, so reused chunks after line shifts can still return stale ranges.
  - confidence: 0.9
- **[P2] Reuse existing chunk sizing semantics:** `src/memory/internal.ts:506`
  - body: `chunkMarkdownWithOffset` counts raw string length, while the current chunker uses weighted sizing and surrogate-safe splitting for CJK-heavy text. Switching session sync to this helper can change Unicode chunk boundaries and defeat reuse for non-Latin transcripts.
  - confidence: 0.83
