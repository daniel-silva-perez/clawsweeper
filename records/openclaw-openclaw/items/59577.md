## Summary

Keep open. Current main still routes changed session transcripts through the full `indexFile` path, so the linked performance bug remains real, but this branch is not merge-ready because it targets the old `src/memory` layout, GitHub reports it as not mergeable, and the latest diff still has concrete correctness issues in metadata refresh and offset-aware chunking.

## What This Changes

The PR adds hash-based incremental indexing for session memory files, offset-tracking chunk metadata/schema fields, and benchmark/integration tests under the old `src/memory` layout.

## Best Possible Solution

Port the incremental session-indexing idea into the current `extensions/memory-core` and `packages/memory-host-sdk` boundaries, share or extend the existing chunking logic so offsets do not fork behavior, keep chunks/vector/FTS metadata consistent, and cover the append, metadata-shift, Unicode, and provider-failure cases with focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Refresh FTS rows when metadata moves:** `src/memory/manager-embedding-ops.ts:930-934`
  - body: The metadata-refresh branch updates `chunks.start_line/end_line` and offsets for hash-matched chunks, but it never updates `chunks_fts`. Keyword search reads line ranges directly from the FTS table, so reused chunks after truncation or line shifts can still return stale citations even though the `chunks` row was corrected.
  - confidence: 0.9
- **[P2] Preserve current chunk sizing in the offset chunker:** `src/memory/internal.ts:506`
  - body: `chunkMarkdownWithOffset` uses raw `line.length` and `segment.length` accounting, while the current chunker uses weighted sizing and surrogate-safe splitting for CJK-heavy text. Since session sync switches to this helper, Unicode transcripts can be chunked differently from normal memory files; carry offsets through the existing chunker semantics instead of forking them.
  - confidence: 0.78
