## Summary

Keep this PR open. The performance problem from #40919 is still real on current main, but this patch is not mergeable as submitted: it changes the old `src/memory` layout while the active builtin memory engine now lives under `extensions/memory-core` and `packages/memory-host-sdk`, and the proposed incremental path has concrete correctness blockers around FTS-only indexing, failure ordering, retry idempotence, FTS metadata, and Unicode chunking.

## What This Changes

The PR adds hash-based incremental indexing for session files under `src/memory`, chunk offset schema/metadata, and benchmark/integration tests intended to preserve unchanged session chunks and embeddings during session growth.

## Best Possible Solution

Port the useful incremental-session idea into the active `memory-core` plugin and host SDK: preserve unchanged session chunks and embeddings for append growth, keep normal memory files on the existing full-reindex path, and update chunks, vectors, FTS rows, file records, retry/idempotence behavior, FTS-only mode, and Unicode chunking consistently.

## Reproduction Assessment

Yes. A changed session transcript on current main deterministically reaches `syncSessionFiles` and then `indexFile`, whose `writeChunks` call clears indexed rows before rewriting regenerated chunks; that source path reproduces the reported full delete/reinsert pattern without needing the PR branch.

## Solution Assessment

No. The direction is appropriate, but this PR is not the best implementation because it patches the old memory tree and introduces correctness regressions; a narrow current-main replacement in `extensions/memory-core` and `packages/memory-host-sdk` is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Port the hook to current memory-core:** `src/memory/manager-sync-ops.ts:912-914`
  - body: This changes the session sync call in the old `src/memory` tree, but current main's active session sync lives under `extensions/memory-core/src/memory`. As submitted, the incremental hook will not affect the active builtin memory engine and should be ported to the current plugin path.
  - confidence: 0.92
- **[P2] Preserve FTS-only session indexing:** `src/memory/manager-embedding-ops.ts:828-833`
  - body: The incremental session method returns immediately when there is no embedding provider. Current `indexFile` still writes keyword-searchable `fts-only` session chunks in that mode, so users without embeddings would lose session transcript search.
  - confidence: 0.91
- **[P1] Embed replacements before pruning indexed rows:** `src/memory/manager-embedding-ops.ts:890-916`
  - body: The incremental path deletes old chunks before requesting embeddings for replacements. A provider timeout or rate limit after the delete can leave the session index partially pruned, whereas the existing full path computes embeddings before mutating indexed rows.
  - confidence: 0.88
- **[P2] Use an upsert for new incremental chunks:** `src/memory/manager-embedding-ops.ts:929-947`
  - body: New incremental chunks are inserted without `ON CONFLICT`, so a retry after a partial sync can hit a duplicate chunk id and fail instead of repairing the stale file record. Match the full-index upsert behavior for idempotence.
  - confidence: 0.86
- **[P2] Refresh FTS rows when metadata moves:** `src/memory/manager-embedding-ops.ts:930-933`
  - body: The hash-match branch updates line and offset metadata in `chunks` without refreshing `chunks_fts`. Keyword search reads citation ranges from the FTS table, so reused chunks after line shifts can still return stale ranges.
  - confidence: 0.9
- **[P2] Reuse existing chunk sizing semantics:** `src/memory/internal.ts:506`
  - body: `chunkMarkdownWithOffset` counts raw string length, while the current chunker uses weighted sizing and surrogate-safe splitting for CJK-heavy text. Switching session sync to this helper can change Unicode chunk boundaries and defeat reuse for non-Latin transcripts.
  - confidence: 0.83
