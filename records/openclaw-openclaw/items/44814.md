## Summary

Keep this PR open. Current main has already removed the old hardcoded 5,000,000-byte parser calls from the relevant Gateway paths, but it did not implement this PR's proposed `gateway.webui.attachmentMaxBytes` config surface, and the concrete larger-image case still fails the separate 6 MiB image cap. The branch is useful prior art for #71142, but it needs a current-main refresh and maintainer API decision before it can merge.

## What This Changes

The PR adds a `gateway.webui.attachmentMaxBytes` config type/schema key and uses it in Control UI `chat.send` and node-event `agent.request` attachment parsing with a 5 MB fallback.

## Best Possible Solution

Keep this PR as useful prior art, but resolve the remaining work through #71142: choose the maintainer-approved upload policy surface, route `chat.send`, Gateway/node `agent.request`, image hydration, bootstrap upload policy, browser prevalidation, docs, and tests through one bounded effective limit.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `agents.defaults.mediaMaxMb` above 6 and submit a Control UI image larger than 6 MiB; the parser reaches the configured max path but still rejects at `MAX_IMAGE_BYTES`.

## Solution Assessment

No. The PR was reasonable for the older hardcoded call sites, but it is no longer the best current fix because it adds `gateway.webui`, bypasses the shared resolver now on main, and does not address the image cap or browser upload policy.
