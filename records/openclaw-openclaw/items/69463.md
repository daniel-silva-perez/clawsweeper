## Summary

Keep this PR open. Current main has adjacent observability through diagnostics events, OTEL export, plugin hooks, and trusted W3C `traceparent` propagation, but it still does not implement the PR's proposed opt-in `diagnostics.sessionTracing` config or configurable `x-session-id` / `x-session-name` style headers on outgoing LLM provider requests. The contribution remains relevant, but it needs maintainer review for the config contract, privacy, docs/changelog, and outbound-header hardening.

## What This Changes

This PR adds an opt-in `diagnostics.sessionTracing` config, a stream wrapper that injects configurable session ID/name headers into LLM provider requests, config schema/type updates, and focused tests.

## Best Possible Solution

Keep this PR open for maintainer review. If core-level session tracing headers are desired, land them behind an opt-in diagnostics config, reuse existing provider header normalization and forbidden/protected-header behavior, preserve trusted `traceparent` precedence, document the privacy tradeoff, add config docs/help/changelog, and keep focused tests for disabled-by-default behavior, caller-header precedence, invalid session keys, subagent/run IDs, and privacy-sensitive fallbacks. If maintainers decide this belongs outside core, route it explicitly to a provider or observability plugin/API design rather than closing it as already implemented.
