## Summary

Current main still does not expose trusted Gateway client identity to the agent runtime line. Web and macOS clients already send distinct structured connect metadata, and the Gateway stores it, but the chat path only maps a small subset into sender metadata while RuntimeInfoInput/buildRuntimeLine still have no client fields.

## Best Possible Solution

Keep this issue open and implement a narrow trusted Gateway client-context path in core: sanitize selected fields from client.connect.client, thread them through chat.send and queued/follow-up run params into RuntimeInfoInput, render deterministic fields such as client=openclaw-macos, client_mode=ui, and client_platform=macos in buildRuntimeLine, and add system-prompt/Gateway chat tests plus WebChat/Control UI docs for the privacy-safe contract.

## Risks / Open Questions

- A fix should avoid exposing raw userAgent, instanceId, or modelIdentifier to prompts unless there is an explicit privacy decision; sanitized client id, mode, platform, and device family are safer.
- Runtime-line changes alter prompt bytes and prompt-cache behavior, so output ordering should be deterministic and covered by prompt/runtime tests.
- Direct webchat currently may surface client id only as untrusted sender metadata, which is not a good contract for agents to branch behavior on.
