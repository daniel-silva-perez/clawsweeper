## Summary

Keep open. Current main still does not provide this CLI command, and the PR remains useful source work, but the branch is stale against the current prompt assembly path and would introduce prompt-preview output that diverges from real embedded runs.

## What This Changes

The PR branch adds an `openclaw agents view-system-prompt [agentId]` subcommand, exports its handler, assembles a local agent system prompt with optional model/channel overrides, and prints prompt metadata.

## Best Possible Solution

Land a rebased implementation that shares or closely mirrors the runtime prompt-preview assembly path, uses generic channel/plugin prompt APIs, applies provider prompt contributions and transforms, and adds targeted CLI docs plus parity tests for model, channel, workspace, heartbeat, provider, and sandbox-aware cases.

## Reproduction Assessment

Yes. Static comparison of the PR diff against current main reproduces the correctness issues: the branch imports modules absent from current main and assembles a prompt through a narrower path than `runEmbeddedAttempt` uses for real runs.

## Solution Assessment

No. The command direction is useful, but duplicating the runtime assembly logic in a CLI handler is not the best solution unless it is factored through a shared helper or kept in strict parity with the embedded runner path.

Full review comments:

- **[P1] Use current channel prompt seams:** `src/commands/agents.commands.view-system-prompt.ts:31-33`
  - body: The new command imports `../signal/*` and `../telegram/*`, but current main no longer has those core modules. A rebase will fail typecheck; this should use the generic channel/plugin prompt helpers used by runtime assembly.
  - confidence: 0.97
- **[P2] Apply provider prompt overlays before printing:** `src/commands/agents.commands.view-system-prompt.ts:164-187`
  - body: This prints the direct `buildEmbeddedSystemPrompt` result, while runtime resolves provider prompt contributions and then applies provider/plugin system-prompt transforms. Provider overlays and text replacements can be missing from the displayed prompt.
  - confidence: 0.91
- **[P2] Use the agent workspace as cwd:** `src/commands/agents.commands.view-system-prompt.ts:101`
  - body: Passing `process.cwd()` into `buildSystemPromptParams` makes repo/runtime fields depend on the shell launch directory. Embedded runtime passes the effective agent workspace for both `workspaceDir` and `cwd`, so this can preview a different prompt than a real run.
  - confidence: 0.92
- **[P2] Preserve runtime workspace notes:** `src/commands/agents.commands.view-system-prompt.ts:130-134`
  - body: Injecting `Loaded context files: ...` through `workspaceNotes` adds prompt text production does not send and misses the runtime BOOTSTRAP/local-mode reminders production does send. The file contents already flow through `contextFiles`, so this reduces preview fidelity.
  - confidence: 0.87
- **[P2] Gate heartbeat guidance like runtime:** `src/commands/agents.commands.view-system-prompt.ts:153-156`
  - body: This injects the default-agent heartbeat prompt whenever the agent is default, but current embedded runtime only includes heartbeat guidance when the heartbeat trigger and config policy allow it. Normal-run previews can therefore include instructions the agent would not receive.
  - confidence: 0.88
