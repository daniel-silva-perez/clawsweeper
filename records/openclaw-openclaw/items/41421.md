## Summary

Keep open. The requested command is not present on current main, and the PR branch still needs maintainer/author work before merge because it is stale against current channel/plugin prompt seams and duplicates only part of the runtime prompt assembly path.

## What This Changes

The PR branch adds an `openclaw agents view-system-prompt [agentId]` CLI subcommand, a new command handler that assembles and prints a local embedded system prompt with optional model/channel overrides, and an export from the agents command barrel.

## Best Possible Solution

Land a rebased implementation that shares or closely mirrors the runtime prompt assembly path, uses generic channel/plugin prompt APIs, applies provider prompt contributions and transforms, and adds focused CLI docs plus parity tests for model, channel, workspace, heartbeat, provider, and sandbox-aware cases.

## Reproduction Assessment

Yes. Static comparison against current main reproduces the blocking issues: the command is absent on main, and the PR branch imports modules absent from current main while assembling a narrower prompt than `runEmbeddedAttempt` sends in real runs.

## Solution Assessment

No. The command direction is useful, but duplicating runtime prompt assembly in a CLI handler is not the narrowest maintainable solution unless the preview path is extracted from or kept in strict parity with the real runtime builder.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use current channel prompt seams:** `src/commands/agents.commands.view-system-prompt.ts:31-33`
  - body: The new command imports `../signal/*` and `../telegram/*`, but those core modules do not exist on current main after channel behavior moved behind plugin/runtime API surfaces. A rebase will fail typecheck and should use the generic channel prompt helpers instead.
  - confidence: 0.97
- **[P2] Apply provider prompt overlays before printing:** `src/commands/agents.commands.view-system-prompt.ts:164-187`
  - body: This prints the direct `buildEmbeddedSystemPrompt` result, while runtime resolves provider prompt contributions and then applies provider/plugin system-prompt transforms. Provider overlays and text replacements can be missing from the displayed prompt.
  - confidence: 0.91
- **[P2] Use the agent workspace as cwd:** `src/commands/agents.commands.view-system-prompt.ts:105`
  - body: Passing `process.cwd()` into `buildSystemPromptParams` makes repo/runtime fields depend on the shell launch directory. Embedded runtime passes the effective agent workspace for both `workspaceDir` and `cwd`, so this can preview a different prompt than a real run.
  - confidence: 0.92
- **[P2] Preserve runtime workspace notes:** `src/commands/agents.commands.view-system-prompt.ts:130-134`
  - body: Injecting `Loaded context files: ...` through `workspaceNotes` adds prompt text production does not send and misses the runtime workspace reminders production may send. The file contents already flow through `contextFiles`, so this reduces preview fidelity.
  - confidence: 0.87
- **[P2] Gate heartbeat guidance like runtime:** `src/commands/agents.commands.view-system-prompt.ts:153-156`
  - body: This injects heartbeat guidance whenever the selected agent is the default agent, but current embedded runtime only includes heartbeat guidance when the heartbeat trigger and config policy allow it. Normal-run previews can therefore include instructions the agent would not receive.
  - confidence: 0.88
