## Summary

Keep open. Current main still does not provide the requested `openclaw agents view-system-prompt` command, and the PR is useful direction, but the proposed branch is not merge-ready because it imports moved channel internals and duplicates runtime prompt assembly in ways that can diverge from real agent runs.

## What This Changes

The PR adds an `openclaw agents view-system-prompt [agentId]` CLI subcommand, a new command handler that assembles and prints a local system prompt, model/channel overrides, and stderr metadata.

## Best Possible Solution

Land a rebased implementation that reuses or extracts a runtime prompt-preview helper from embedded runs, then add focused CLI docs and parity tests for model override, channel prompt capabilities, workspace cwd, heartbeat policy, provider contributions/transforms, plugin guidance, and workspace notes.

## Reproduction Assessment

Yes. Static inspection is enough: current main has no command, while the PR head imports non-existent current-main modules and assembles a prompt through a path that differs from `runEmbeddedAttempt`.

## Solution Assessment

No. The command direction is useful, but duplicating prompt assembly in the CLI handler is not the narrowest maintainable fix unless it shares the same runtime seams used by actual embedded runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use current channel prompt seams:** `src/commands/agents.commands.view-system-prompt.ts:31-33`
  - body: These imports target `src/signal` and `src/telegram` modules that no longer exist on current main, so the branch will fail after rebase. Use the generic channel/plugin prompt helpers instead of direct channel internals.
  - confidence: 0.97
- **[P2] Collect runtime channel prompt capabilities:** `src/commands/agents.commands.view-system-prompt.ts:78-80`
  - body: This calls only `resolveChannelCapabilities`, but real runs use `collectRuntimeChannelCapabilities`, which also merges plugin-owned prompt capabilities. Channel previews can omit capabilities the real prompt includes.
  - confidence: 0.9
- **[P2] Use the agent workspace as cwd:** `src/commands/agents.commands.view-system-prompt.ts:101-105`
  - body: Passing `process.cwd()` makes repo/runtime fields depend on the shell launch directory. Embedded runtime passes the effective agent workspace as `cwd`, so this can print a different prompt than a real run.
  - confidence: 0.92
- **[P2] Preserve runtime workspace notes:** `src/commands/agents.commands.view-system-prompt.ts:130-134`
  - body: Injecting `Loaded context files: ...` through `workspaceNotes` adds prompt text production does not send and misses the runtime BOOTSTRAP/embedded-mode reminders production may send. The file contents already flow through `contextFiles`.
  - confidence: 0.87
- **[P2] Gate heartbeat guidance like runtime:** `src/commands/agents.commands.view-system-prompt.ts:153-156`
  - body: This injects heartbeat guidance for the default agent regardless of trigger and heartbeat policy. Normal-run previews can therefore include heartbeat instructions the agent would not receive.
  - confidence: 0.88
- **[P2] Apply provider prompt overlays before printing:** `src/commands/agents.commands.view-system-prompt.ts:164-187`
  - body: The command prints the direct `buildEmbeddedSystemPrompt` result, while runtime resolves provider prompt contributions and then applies provider/plugin system-prompt transforms. Provider overlays can be missing from the displayed prompt.
  - confidence: 0.91
