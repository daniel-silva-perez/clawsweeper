## Summary

Keep open. Current main still lacks the requested auto-TTS text-delivery policy and still gives the model only operational TTS prompt guidance, so the reported privacy/UX and spoken-output issues remain valid.

## What This Changes

Request adds an auto-TTS voice-only or caption delivery mode and strengthens the auto-TTS system prompt so replies are written for natural spoken output.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with default current behavior, apply suppression only after successful auto-TTS voice/audio generation, preserve text fallback when synthesis or voice delivery fails, and update the auto-TTS prompt hint with concise speech-first writing guidance without implementing a full spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS with successful synthesis and `maybeApplyTtsToPayload()` returns the original visible reply plus audio metadata, after which voice-capable channel delivery uses the retained text as a caption or follow-up.

## Solution Assessment

Yes. A generic TTS delivery policy plus a stronger prompt hint is the narrow maintainable fix; channel-specific hacks or reviving the full spoken-rewrite proposal from #22028 would be broader than this issue requires.
