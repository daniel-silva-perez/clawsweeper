## Summary

Keep open. Current main still lacks both requested pieces: auto-TTS has no configurable visible-text delivery policy, and the system prompt hint remains operational rather than speech-first.

## What This Changes

The issue requests a voice-only or caption-only auto-TTS text delivery policy plus stronger prompt guidance for natural spoken replies.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with current behavior as the default, suppress or caption visible text only after auto-TTS audio is successfully generated and voice-capable delivery is selected, preserve text fallback on TTS or voice delivery failure, and add concise speech-first prompt guidance without taking on the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a final reply on a voice-capable channel and current `maybeApplyTtsToPayload()` returns the original text plus generated audio, while channel delivery code then captions or follows up with the text.

## Solution Assessment

Yes, with scope control. A generic TTS delivery policy plus a stronger auto-TTS prompt hint is the narrow maintainable direction; channel-only special cases or a full spoken-rewrite pass would be broader than this issue requires.
