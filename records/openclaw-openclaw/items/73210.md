## Summary

Keep open. Current main still lacks a configurable auto-TTS text-delivery policy and the auto-TTS prompt hint still tells the model how to operate TTS rather than how to write replies that will be spoken aloud; the explicit TTS-tool duplicate-text fixes and older spoken-rewrite request do not cover this narrower auto-TTS behavior.

## What This Changes

The issue requests an auto-TTS delivery mode that can suppress or caption visible text when voice generation succeeds, plus stronger speech-first system prompt guidance for natural spoken replies.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with current behavior as the default, apply suppression only after successful auto-TTS generation for voice-capable delivery, preserve visible text fallback on TTS or delivery failure, and strengthen the auto-TTS prompt hint with concise speech-first writing guidance without implementing the broader spoken-rewrite pipeline from #22028.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply on a voice-capable channel and `maybeApplyTtsToPayload()` attaches audio while retaining the original visible reply text, which channel senders then use as a caption or follow-up text.

## Solution Assessment

Yes, with scoped implementation. A generic TTS delivery policy plus stronger auto-TTS prompt wording is the narrow maintainable fix; channel-only special cases or a full spoken-rewrite pass would be broader than this issue requires.
