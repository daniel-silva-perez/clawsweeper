## Summary

Keep open. Current main still lacks a configurable auto-TTS text-delivery policy and the auto-TTS system prompt hint is still operational guidance rather than speech-first writing guidance; the closed explicit TTS-tool duplicate-text fix and the older spoken-rewrite request do not cover this narrower auto-TTS behavior.

## What This Changes

The issue requests an auto-TTS voice-only delivery mode, optional caption-only text delivery, fallback-to-text on TTS failure, and stronger prompt guidance for natural spoken replies.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with current behavior as the default, apply suppression only after successful auto-TTS generation for voice-capable delivery, preserve text fallback on TTS or voice delivery failure, and add concise speech-first prompt guidance without taking on the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply on a voice-capable channel and `maybeApplyTtsToPayload()` returns generated audio while retaining the original visible reply text; the config and docs expose no suppression setting.

## Solution Assessment

Yes, with scope control. A generic TTS delivery policy plus a stronger auto-TTS prompt hint is the narrow maintainable path; channel-specific one-offs or the full spoken-rewrite feature from #22028 would be broader than this issue requires.

## Repair Work Prompt

Implement the auto-TTS delivery policy requested by https://github.com/openclaw/openclaw/issues/73210. Add a backward-compatible `messages.tts` text-delivery policy, likely `textDelivery: "normal" | "suppress" | "caption"` with default `"normal"`, and decide whether local `/tts` prefs need a matching `/tts text ...` command using the existing prefs pattern. Apply the policy only to auto-TTS after successful audio generation. Preserve text fallback when synthesis fails or when the channel cannot deliver a native voice-note style payload. For `"caption"`, keep caption text where the channel supports it and avoid separate overflow text; do not suppress context on channels that only send audio as ordinary file attachments. Update `buildTtsSystemPromptHint()` with concise speech-first guidance that replies will be spoken aloud, should use natural sentences, and should avoid markdown/code/URLs when possible; do not implement the broader spoken rewrite pass from #22028. Update config types, zod schema, schema help/labels/generated docs as required by repo conventions, TTS docs, command help/status if prefs are added, and add focused tests for speech-core, config validation, command handling, and Telegram/WhatsApp/Matrix/Feishu delivery behavior. Add a changelog entry because this is user-facing. Do not move channel-specific behavior into core except through generic delivery policy fields already carried by reply payloads or typed channel contracts.
