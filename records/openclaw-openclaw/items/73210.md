## Summary

Keep open. Current main still lacks an auto-TTS text-delivery policy and the TTS system prompt remains operational guidance rather than speech-writing guidance; the explicit `/tts` audio path and the stale spoken-rewrite issue do not cover this requested auto-TTS behavior.

## What This Changes

Add an auto-TTS voice-only/text-delivery mode and strengthen the TTS system prompt so auto-spoken replies are written for natural speech.

## Best Possible Solution

Add a backward-compatible generic `messages.tts` text-delivery policy with current behavior as the default, apply suppression only after successful auto-TTS generation and native voice-capable delivery, preserve text fallback for synthesis failure or file-attachment-only channels, and extend the prompt hint with concise guidance for natural spoken prose.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS and inspect `maybeApplyTtsToPayload()`, which returns the visible text payload plus generated audio on success, while the current prompt helper and schema have no natural-speech or text-suppression path.

## Solution Assessment

Yes. A generic optional policy plus prompt-hint improvement is the narrow maintainable fix; channel-specific hacks or a full spoken-rewrite pipeline would be broader than this issue requires.
