## Summary

Keep open. Current main still lacks a configurable auto-TTS text-delivery policy, and the injected TTS system prompt remains operational rather than speech-writing guidance. The linked stale spoken-rewrite issue is adjacent but not a live canonical tracker, and the linked duplicate-text fix only covered the explicit TTS tool path.

## What This Changes

Requests an auto-TTS voice-only or caption-only delivery mode plus stronger system-prompt guidance for natural spoken output.

## Best Possible Solution

Add a backward-compatible `messages.tts` text-delivery policy with default current behavior, apply it only after successful auto-TTS generation and voice-capable delivery, preserve text fallback when synthesis fails or audio is only a file attachment, and update the TTS prompt hint to steer models toward natural spoken prose without implementing a separate rewrite pass.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with auto-TTS enabled, `maybeApplyTtsToPayload()` returns the original visible text plus generated audio, while the config schema and docs provide no text-suppression option.

## Solution Assessment

Yes, with one constraint: a generic optional TTS delivery policy plus prompt hint is the narrow maintainable direction, while channel-specific hacks or a full spoken-rewrite pipeline would be broader than this request.

## Repair Work Prompt

Implement the requested auto-TTS text-delivery policy for https://github.com/openclaw/openclaw/issues/73210. Add a backward-compatible `messages.tts.textDelivery` enum, likely `normal | suppress | caption`, with default `normal`; add matching local preference support if practical through `/tts text ...` without changing existing `/tts on/off` behavior. Apply the policy only after auto-TTS successfully generates audio. `suppress` should remove the visible text only when the result is a voice-style delivery (`audioAsVoice`/native voice note); if synthesis fails, if TTS is skipped, or if the channel only receives a normal audio file attachment, keep the text fallback. `caption` should keep caption text only where the voice transport supports it and avoid overflow follow-up text; document any channel-specific limitations. Also strengthen `buildTtsSystemPromptHint()` so models know replies will be spoken aloud and should use natural sentences, avoid bullets/code/URLs/markdown when possible, and put technical details in a separate text follow-up when needed. Update config types/schema/help/docs, focused tests for speech-core auto-TTS payload behavior and prompt output, slash-command tests if adding prefs, and channel tests for WhatsApp/Telegram/Matrix/Feishu behavior affected by the policy. Do not implement the larger spoken-rewrite LLM pass from #22028, do not change the explicit `tts` tool behavior fixed around #70092, and keep core behavior extension-agnostic. Add a user-facing changelog entry with `Thanks @altierac` if code/docs add the feature.
