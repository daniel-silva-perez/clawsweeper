## Summary

Keep open. Current main still lacks a configurable auto-TTS text-delivery policy and still uses an operational TTS prompt hint rather than speech-writing guidance; the explicit TTS tool duplicate-text fix and the stale spoken-rewrite issue do not cover this narrower auto-TTS request.

## What This Changes

The issue requests voice-only or caption-only delivery controls for successful auto-TTS replies and stronger prompt guidance so auto-TTS output is written for spoken delivery.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with default current behavior, apply suppression only after successful auto-TTS voice-capable audio generation, preserve text fallback on synthesis or native-voice delivery failure, and update the auto-TTS prompt hint with concise speech-first writing guidance without implementing the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a final reply with successful synthesis on a voice-capable channel, and `maybeApplyTtsToPayload()` returns visible text plus voice media while config/docs expose no suppression setting.

## Solution Assessment

Yes, with scope control. A generic TTS delivery policy plus a stronger prompt hint is the narrow maintainable solution; channel-specific hacks or a full rewrite pass would be broader than this issue requires.

## Repair Work Prompt

Implement the auto-TTS delivery and prompt improvements requested by https://github.com/openclaw/openclaw/issues/73210. Add a backward-compatible `messages.tts` delivery policy, likely `textDelivery: "normal" | "suppress" | "caption"` with `normal` as the default, and wire it through the public TTS type, strict zod schema, effective resolved config, and docs. Add a local preference command only if it fits the existing `/tts` prefs model, for example `/tts text normal|off|caption`; keep naming consistent with product docs. In `extensions/speech-core/src/tts.ts`, apply text suppression only after successful auto-TTS synthesis and only when the generated audio is intended for voice-capable delivery; keep normal visible text fallback when synthesis fails, when the reply already has media, or when the channel cannot deliver a native voice note and needs text context. Preserve the explicit `/tts audio` and `/tts latest` tool behavior from #70092. Update `buildTtsSystemPromptHint()` to say replies will be spoken aloud and to prefer conversational sentences over bullets, code blocks, URLs, dense markdown, and raw technical formatting. For `caption`, preserve text as a voice/media caption where the channel supports it and avoid overflow/separate transcript behavior according to the new policy; add channel tests for Telegram, WhatsApp, Matrix, and Feishu rather than changing provider output formats or implementing the broader #22028 spoken-rewrite pass. Update `docs/tools/tts.md` and add a user-facing changelog entry crediting `Thanks @altierac`. Do not change unrelated TTS providers, channel auth, provider selection, or voice-note transcoding behavior.
