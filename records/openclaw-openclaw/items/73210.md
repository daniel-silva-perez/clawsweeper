## Summary

Keep open. Current main still lacks the requested auto-TTS text-delivery policy and the auto-TTS prompt hint still provides operational TTS instructions rather than speech-first writing guidance, so the explicit TTS-tool duplicate-text fix and the older spoken-rewrite issue do not resolve this narrower feature gap.

## What This Changes

Request adds a configurable auto-TTS voice-only or caption-only text delivery mode and strengthens the auto-TTS system prompt so model replies are written for natural spoken output.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with current double-delivery as the default, suppress or caption visible text only after auto-TTS generation and voice-capable delivery succeed, preserve text fallback on TTS or delivery failure, and add concise speech-first prompt guidance without taking on the full spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply on a voice-capable channel and the speech-core auto-TTS path returns generated audio while retaining the original reply text for Telegram or WhatsApp delivery; config, docs, and commands expose no suppression control.

## Solution Assessment

Yes, with scope control. A generic TTS delivery policy plus a stronger auto-TTS prompt hint is the narrow maintainable direction; channel-only special cases or a full rewrite pass would be broader than this issue requires.
