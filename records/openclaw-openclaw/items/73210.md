## Summary

Keep open. Current main still lacks a configurable auto-TTS text-delivery policy for normal final replies, and the prompt hint still focuses on operating TTS rather than telling the model to write naturally for speech; the current audio-only handling only covers explicit TTS/audio and streamed-block supplemental media, not the requested `/tts on` behavior across voice-capable channels.

## What This Changes

The issue requests an auto-TTS voice-only/caption/normal delivery mode plus stronger speech-first system prompt guidance for natural spoken replies.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with current behavior as the default, apply suppression only after successful auto-TTS generation for voice-capable delivery, preserve visible text fallback on TTS or voice delivery failure, and add concise speech-first prompt guidance without implementing the broader spoken-rewrite pipeline from #22028.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply on a voice-capable channel and the TTS layer attaches audio while retaining the original visible reply text, which channel senders then use as a caption or follow-up text.

## Solution Assessment

Yes, with scope control. A generic TTS delivery policy plus stronger auto-TTS prompt wording is the narrow maintainable direction; channel-only special cases or the full spoken-rewrite pass from #22028 would be broader than this issue requires.
