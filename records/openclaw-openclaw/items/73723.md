## Summary

Keep open. Current main still has the reported failure path: native hook relay registrations are in-memory and per-run, missing relay ids still produce `native hook relay not found`, the gateway returns that as `INVALID_REQUEST`, and the CLI fail-closes Codex `PreToolUse` as `Native hook relay unavailable`. `openclaw hooks list` still reports only ordinary hook entry readiness, not active Codex native relay registrations. The reporter's follow-up that a fresh session works while the old Telegram session stays blocked is consistent with the current per-session/per-run relay lifecycle, and I did not find a merged current-main change that repairs or diagnoses that stale-session relay state.

## Best Possible Solution

Fix this in the Codex native hook relay lifecycle rather than closing it. The likely path is to add active relay health/diagnostics separate from ordinary hook readiness, detect missing or stale relay ids on Codex thread resume or gateway restart recovery, and rebind or invalidate the affected session/thread so the old session can recover without relying on a fresh `/new`. Keep the security-sensitive fail-closed behavior unless maintainers explicitly choose a different policy, and add a regression test for a stale relay id after restart/resume where `hooks list` is ready but native tool execution would otherwise remain blocked.
