## Summary

Close as implemented on current main. The reported stale Codex native hook relay path is covered by merged PR #74107, which added a stable per-session relay id, a local loopback relay bridge before the Gateway fallback, and regression coverage for relay reuse across resumed runs; that fix is present in the latest release v2026.4.29.

## What This Changes

The issue reports that a long-running Telegram/Codex session can keep blocking native shell/tool execution with `Native hook relay unavailable` after a gateway restart even though ordinary hooks report ready.

## Best Possible Solution

Keep the shipped stable relay id plus direct loopback bridge implementation from PR #74107. If maintainers still want `openclaw hooks list` or doctor to show active native relay registrations as a separate diagnostic, that should be tracked as a narrower follow-up rather than keeping this fixed relay outage report open.

## Reproduction Assessment

Yes for the root failure path: the merged regression tests cover a retained Codex hook command reusing the same session relay id across a resumed run and resolving the active registration. The exact old Telegram live session was not replayed in this read-only review.

## Solution Assessment

Yes. The shipped fix is the narrow maintainable solution because it both stabilizes the relay id across session runs and lets the relay command reach the registering process directly before falling back to Gateway RPC; the older global-registry PR is superseded by that approach.
