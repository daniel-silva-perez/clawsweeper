## Summary

Keep open. Current main still has the reported one-shot normal remote media fetch path, and this issue is now paired with open PR #74553 using closing syntax, so the right maintainer workflow is to review and land or close that PR before closing this issue. The follow-up discussion also correctly narrows the bug: `(0/1)` is an attachment success count, not a retry-attempt counter.

## What This Changes

The issue requests bounded retry/backoff for transient media-understanding remote fetch failures so Discord audio attachments are not dropped before transcription starts.

## Best Possible Solution

Land or replace #74553 after review if it preserves fail-fast behavior for 4xx, maxBytes, SSRF blocks, and caller aborts while adding focused retry tests and a changelog credit; then close this issue as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `MediaAttachmentCache.getBuffer` for a URL attachment with a transiently failing fetch implementation; current main calls `fetchRemoteMedia` once and rethrows before transcription can begin.

## Solution Assessment

Yes for the requested direction. A shared retry policy at the `fetchRemoteMedia` or `MediaAttachmentCache` boundary is the narrowest maintainable fix because it covers Discord preflight and normal media-understanding consumers without Discord-specific behavior.
