## Summary

Keep open. Current main still uses a one-shot remote media fetch for normal media-understanding URLs, and the open closing PR is the implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests bounded retry/backoff for transient remote media fetch failures before Discord audio/media-understanding transcription, while preserving fail-fast behavior for non-retryable failures.

## Best Possible Solution

Review and land PR #74553, or replace it with an equivalent narrow implementation at the fetchRemoteMedia/MediaAttachmentCache boundary that covers Discord preflight and normal media-understanding consumers with focused tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create a remote media attachment URL, make the shared fetch fail transiently, and current main reaches one MediaAttachmentCache.getBuffer call and one fetchRemoteMedia guarded attempt before surfacing MediaFetchError.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable fix because it covers Discord audio preflight and other media-understanding consumers without adding Discord-specific retry policy.
