## Summary

Keep open. Current main still has a single normal remote-media fetch through the shared media-understanding cache, and the issue is already paired with open PR #74553 using closing syntax, so the right cleanup is to review/land or close that PR rather than closing this issue before merge.

## What This Changes

The issue requests bounded retry with backoff for transient remote media fetch failures before audio transcription, especially for Discord voice-note URLs processed by media-understanding.

## Best Possible Solution

Review PR #74553 at https://github.com/openclaw/openclaw/pull/74553 and either land a bounded shared retry implementation at the fetchRemoteMedia/MediaAttachmentCache boundary or close/supersede that PR with an equivalent narrow fix. The end state should preserve fail-fast behavior for 4xx, SSRF blocks, aborts/timeouts, and maxBytes while covering transient network/read failures and 5xx responses with focused tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: drive Discord audio preflight with one remote audio URL and make the shared fetch fail transiently; current main passes through MediaAttachmentCache.getBuffer to one fetchRemoteMedia call with no retry loop for normal URLs.

## Solution Assessment

Yes, the shared fetch/cache boundary is the best place to solve this because it covers Discord preflight and the normal media-understanding path without adding Discord-specific retry policy. Since an open closing PR already targets that boundary, the next step is normal PR review and validation rather than a separate repair lane.
