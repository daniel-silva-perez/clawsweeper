## Summary

Keep open. Current main still routes Discord audio preflight through the shared media-understanding cache, where normal remote URLs get one guarded fetch attempt, and open PR #74553 already specifically references this issue as the implementation path.

## What This Changes

Add bounded transient retry/backoff for remote media fetches used by media-understanding so a single Discord audio fetch failure does not drop voice-note transcription.

## Best Possible Solution

Review PR #74553 and either land its bounded shared retry implementation or replace it with an equivalent narrow fix at the fetchRemoteMedia/MediaAttachmentCache boundary. The final behavior should cover Discord preflight and normal media-understanding consumers, keep existing safety carve-outs, include focused unit/seam tests, and add an appropriate user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: drive media-understanding or Discord preflight with one remote audio URL and make the shared fetch fail transiently; current main reaches one MediaAttachmentCache.getBuffer call and one fetchRemoteMedia attempt before failing.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable place to solve this because it covers Discord audio preflight and other media-understanding consumers without adding Discord-specific retry policy; the issue should remain open until the linked PR is reviewed and merged or superseded.
