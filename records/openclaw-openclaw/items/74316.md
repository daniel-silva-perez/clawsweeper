## Summary

Keep open. Current main still has the reported one-shot remote media fetch behavior, and an open linked PR already targets the shared retry fix, so the issue should remain open until that PR is reviewed and merged or rejected.

## What This Changes

Add bounded retry/backoff for transient remote media fetch failures on the media-understanding path used by Discord audio attachments.

## Best Possible Solution

Review and land or reject the linked retry PR; once a bounded retry policy is merged for the shared media-understanding remote fetch/cache path, this issue can close through the PR's closing reference.

## Reproduction Assessment

Yes. A focused reproduction can mock the remote media fetch/cache path so the first fetch fails with a transient network/read failure or 5xx and current main surfaces the failure without a second attempt.

## Solution Assessment

Yes. The narrowest maintainable solution is retry at the shared `fetchRemoteMedia` or `MediaAttachmentCache` boundary while preserving fail-fast behavior for 4xx, SSRF/policy blocks, caller aborts, and maxBytes.
