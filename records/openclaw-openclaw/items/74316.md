## Summary

Keep open. Current main still has the reported single normal remote-media fetch path, and this issue is already paired with open closing PR #74553, so the maintainable path is to review, land, or close that PR rather than closing the issue before merge.

## What This Changes

The issue asks OpenClaw to retry transient remote media fetch failures before media-understanding drops Discord audio attachments.

## Best Possible Solution

Review PR #74553 and either land its bounded shared retry implementation at the `fetchRemoteMedia`/`MediaAttachmentCache` boundary or close/supersede it with an equivalent narrow fix that preserves fail-fast behavior for 4xx, SSRF blocks, aborts/timeouts, and maxBytes failures.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to route a Discord audio attachment URL through preflight, make the first shared remote media fetch fail transiently, and observe that current main reaches one `MediaAttachmentCache.getBuffer` to `fetchRemoteMedia` call with no retry loop.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable place to fix this because it covers Discord preflight and other media-understanding consumers without adding Discord-specific retry behavior; the existing open PR already targets that boundary.
