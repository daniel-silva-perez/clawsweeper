## Summary

Current main still has a one-shot remote media fetch on the media-understanding path used by Discord audio preflight. The discussion also narrowed the report correctly: the `(0/1)` log is an attachment success count, not a retry counter, but there is still no bounded retry/backoff for transient fetch failures before transcription runs.

## What This Changes

Request a bounded retry-with-backoff policy for media-understanding remote attachment fetches so transient Discord audio fetch failures do not permanently drop voice notes.

## Best Possible Solution

Add the retry policy at the shared remote media fetch/cache boundary used by media-understanding, expose retry visibility without changing the existing `(X/Y)` attachment summary semantics, and cover both Discord audio preflight and generic URL fallback behavior with focused tests plus a user-facing changelog fix crediting the reporter.

## Repair Work Prompt

Fix the media-understanding remote attachment fetch path so transient fetch failures do not permanently drop Discord voice notes. Implement a small shared retry-with-backoff policy around `fetchRemoteMedia` / `MediaAttachmentCache` used by media-understanding, not a Discord-only workaround. Retry only transient network/read failures and recoverable 5xx responses with bounded backoff and jitter, and preserve fail-fast behavior for 4xx responses, maxBytes errors, SSRF policy blocks, and caller aborts. Keep the existing `audio: failed (X/Y)` attachment summary semantics stable; if retry visibility is added, use distinct metadata/log fields. Add focused tests covering cache/fetch retry and non-retry cases plus Discord audio preflight or URL fallback behavior. Add a user-facing changelog fix entry crediting @gabrielexito-stack. Do not change provider selection, transcription fallback order, Discord auth/proxy behavior, or unrelated media/provider APIs.
