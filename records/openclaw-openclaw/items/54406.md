## Summary

Keep open. Current main still does not expose the richer compaction start/end event contract, and the submitted branch is stale against the renamed `compaction_start`/`compaction_end` event path. The PR is directionally useful, but it needs rework before merge because it omits `hasResult` from emitted end events and raises routine compaction lifecycle logs to `warn`.

## What This Changes

The PR branch adds compaction start/end event metadata, structured compaction lifecycle logs, dispatcher forwarding for auto-compaction start events, and focused tests for compaction payload observability.

## Best Possible Solution

Rework the change on current main so the embedded compaction dispatcher passes the `compaction_start` event into the handler, start events expose a safe `reason`, end events expose `completed`, `hasResult`, `willRetry`, and sanitized `errorMessage`, and compaction count, retry, reconciliation, hook, liveness, and UI behavior remain unchanged except for the richer observability contract.

- **[P2] Emit `hasResult` in compaction end events:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:123-127`
  - body: The PR computes and logs `hasResult`, and the PR description promises it, but both emitted event payloads still omit it.   - confidence: 0.9

  - body: The patch raises normal compaction start and successful end lifecycle logs from debug to warn. Healthy compaction is expected behavior, so warning on every normal cycle would create production noise; reserve warn for retry, abort, or failure paths.
