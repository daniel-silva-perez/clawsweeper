## Summary

Keep open. Current main still lacks the proposed pre-attempt fallback stop/deadline seam, but the PR branch is not ready to merge because its timeout budget math and deadline plumbing conflict with current cron execution semantics and active adjacent timeout work.

## What This Changes

The PR adds a model-fallback pre-attempt stop hook, threads cron deadline metadata into isolated cron execution, skips low-budget fallback attempts, and adds cron/model-fallback regression tests.

## Best Possible Solution

Land a rebased, narrow fix that preserves current deferred isolated-timeout cleanup semantics, keeps the generic pre-attempt stop seam, derives the skip threshold from the prepared per-attempt execution budget, and preserves explicit stop errors when no real candidate failure exists yet.

## Reproduction Assessment

Yes. The PR body includes a concrete production log showing the primary consuming almost all cron budget and the fallback failing at 0ms; source inspection on current main confirms there is no pre-attempt stop hook and cron only checks the abort signal inside the selected candidate run callback.

## Solution Assessment

No. The direction is useful, but the current diff is not the best mergeable solution because it bases the budget on the cron safety timeout and starts the propagated deadline too early; the safer path is a rebase that uses `prepared.context.timeoutMs` and starts the deadline from the current `onExecutionStarted` lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Base the fallback guard on the attempt timeout:** `src/cron/isolated-agent/run.ts:986-990`
  - body: The new guard computes `fallbackMinRemainingMs` from `resolveCronJobTimeoutMs(params.job)`, which defaults agent-turn jobs without payload `timeoutSeconds` to the 60-minute safety ceiling. The actual model call uses `prepared.context.timeoutMs`, so jobs configured with a 5-10s `agents.defaults.timeoutSeconds` can incorrectly require 30s remaining and skip fallbacks that still fit the real attempt budget.
  - confidence: 0.88
- **[P2] Start the propagated deadline when execution starts:** `src/cron/service/timer.ts:89-93`
  - body: Current main defers isolated cron timeouts until `onExecutionStarted`, so lane wait does not consume execution budget. Computing `deadlineAtMs` before `executeJobCore` lets queue wait drain the propagated fallback budget and can skip fallbacks before the job has had real execution time.
  - confidence: 0.84
- **[P2] Preserve the stop reason when no candidate failed:** `src/agents/model-fallback.ts:915-916`
  - body: When `beforeAttempt` stops after only cooldown/auth skips, `lastError` is still undefined. This branch breaks and later throws a synthesized fallback summary, dropping the hook's explicit stop message that cron budget checks are supposed to surface.
  - confidence: 0.78
