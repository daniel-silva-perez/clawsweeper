## Summary

Keep open. Current main still lacks the proactive pre-attempt cron fallback stop hook and deadline-budget plumbing, so this PR is not obsolete; however the PR head remains divergent and still has blocking timeout-policy defects that should be fixed before merge.

## What This Changes

The PR adds a model-fallback pre-attempt stop hook, threads cron deadline metadata into isolated cron execution, skips low-budget fallback candidates, and adds cron/model fallback regression tests.

## Best Possible Solution

Land a rebased, narrow fix that preserves current deferred isolated-timeout cleanup semantics, keeps the generic pre-attempt stop seam, computes the skip threshold from the prepared per-attempt execution budget, preserves explicit stop errors, and coordinates the final abort/fallback contract with #42482 and #62682.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Base the fallback budget on the attempt timeout:** `src/cron/isolated-agent/run.ts:985-986`
  - body: This computes `fallbackMinRemainingMs` from `resolveCronJobTimeoutMs(params.job)`, which falls back to the 60-minute cron safety ceiling when `payload.timeoutSeconds` is unset. The actual model call uses `prepared.context.timeoutMs`, so jobs configured with a 5-10s `agents.defaults.timeoutSeconds` can incorrectly skip fallbacks whenever less than 30s remain even though a useful attempt still fits the real budget.
  - confidence: 0.88
- **[P2] Propagate the explicit stop error:** `src/agents/model-fallback.ts:913-914`
  - body: When `beforeAttempt` stops after earlier candidates were only skipped, `lastError` is still `undefined`. The loop breaks and later throws a synthesized fallback summary, dropping the hook's explicit low-budget stop message and returning a misleading error for this new path.
  - confidence: 0.82
- **[P2] Start the deadline when isolated execution starts:** `src/cron/service/timer.ts:89-93`
  - body: This derives `deadlineAtMs` before `executeJobCore` reaches the isolated runner. Current main deliberately starts isolated timeouts from `onExecutionStarted`, so queue wait can consume the propagated fallback budget and skip fallbacks before the job gets real execution time.
  - confidence: 0.84
