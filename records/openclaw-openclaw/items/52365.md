## Summary

Keep open. Current main still lacks the proactive pre-attempt cron fallback budget guard, so the PR addresses a real remaining runtime behavior, but the latest head should not merge as-is because the fallback skip threshold is derived from the outer cron safety timeout and the deadline threading needs to preserve current main's deferred isolated-timeout semantics.

## What This Changes

The PR adds a model fallback pre-attempt stop hook, threads cron deadline metadata through isolated cron execution, skips fallback candidates when little cron budget remains, and adds cron/model fallback regression tests.

## Best Possible Solution

Land a rebased, narrow fix that keeps the generic pre-attempt stop seam, computes the skip threshold from the real prepared execution budget, preserves current deferred isolated-timeout cleanup, and coordinates the final abort/fallback contract with #42482 and #62682.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Base the fallback budget on the attempt timeout:** `src/cron/isolated-agent/run.ts:988-992`
  - body: `resolveCronJobTimeoutMs(params.job)` falls back to the 60-minute agent-turn safety timeout when `payload.timeoutSeconds` is unset, while the actual model attempt uses `prepared.context.timeoutMs` from `resolveAgentTimeoutMs` and `agents.defaults.timeoutSeconds`. With a 5-10s agent default, this guard still requires 30s of remaining outer budget and skips fallbacks that could have completed within the real per-attempt timeout.
  - confidence: 0.88
- **[P2] Preserve the deferred execution timeout boundary:** `src/cron/service/timer.ts:89-93`
  - body: This captures `deadlineAtMs` before `executeJobCore` reaches the isolated runner. Current main starts isolated agent-turn timeouts only from `onExecutionStarted`; using an enqueue-time deadline would let queue wait consume the new fallback budget and can skip fallbacks before they get real execution time.
  - confidence: 0.84
