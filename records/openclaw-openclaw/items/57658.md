## Summary

Keep open. Current main has scoped operator tokens and durable device-pairing bounds, so the read-only/read-write access-control part is partly covered, but it still does not implement custom named roles, a `gateway.roles` config surface, or an `openclaw roles create` CLI flow.

## Best Possible Solution

Make a maintainer product decision on whether scoped operator device tokens are the intended RBAC model. If custom named roles are desired, implement them in core gateway auth: add a versioned `gateway.roles` config schema, role-to-scope resolution, CLI management/listing, pairing-time role assignment, protocol/docs updates, and tests proving limited read/write roles cannot gain admin, approval, or pairing scopes.

## Risks / Open Questions

- The existing scoped operator-token model overlaps with the reported read-only/operatorRW use case, so maintainers may decide the best product outcome is documentation rather than custom named roles.
- A real custom-role implementation would touch gateway auth and security-sensitive device pairing; it must preserve approved scope baselines, no-silent-upgrade behavior, and method-scope enforcement.
