## Summary

Keep open. Current main has scoped operator tokens and durable scope-upgrade protections, but it still does not implement custom named gateway roles, a `gateway.roles` config surface, an `openclaw roles` CLI, or pairing-time assignment to custom role names. Custom role strings are rejected during the WebSocket handshake before they can authorize gateway RPCs.

## Best Possible Solution

Make a maintainer product/security decision on whether scoped operator device tokens are the intended RBAC model. If custom named roles are desired, implement them in core gateway auth with a versioned `gateway.roles` config schema, role-to-scope resolution, CLI management/listing, pairing-time role assignment, protocol/docs updates, and regression tests for reconnect, token rotation, admin/pairing/approval denial, and read/write scope containment. If scoped operator tokens are the intended model, document that clearly and consider improving the pairing/rotation UX for read-only and read-write devices.

## Risks / Open Questions

- A real custom-role implementation would touch gateway authentication and device pairing, so it needs security-focused tests proving custom read/write roles cannot gain admin, approval, pairing, node, or token-rotation authority.
- There is overlap between the requested read-only/operatorRW use case and existing scoped operator tokens; maintainers may choose to document and improve that scoped-token model instead of adding full named RBAC.
