## Summary

Keep open. Current main still only registers whole-install `openclaw backup create` and `openclaw backup verify`; there is no restore subcommand, no `--agent` scope, and no `--as` clone path. The existing backup planner and archive manifest are global state/config/credentials/workspace oriented, while related #41111 and open #59280 cover generic restore rather than per-agent scoping and config-entry rewriting.

## What This Changes

Add per-agent backup, restore, and clone support to the backup CLI with `--agent <id>` and optional restore-as-new-agent `--as <id>`.

## Best Possible Solution

Keep this issue open as a valid feature request. The best path is to settle the generic backup restore command and archive safety model first, then add per-agent scoping for `backup create --agent <id>`, restore with optional `--as <id>`, scoped workspace plus `agents/<agentId>` payloads, config-entry rewrite, docs, and focused safety tests.

## Risks / Open Questions

- Per-agent restore/clone would copy or rewrite auth profiles, sessions, workspaces, and config entries, so the design needs explicit path-containment, overwrite/conflict, and sensitive-state policy before implementation.
- The current main branch lacks a generic restore command; implementing per-agent restore before resolving #41111/#59280 could duplicate or conflict with the restore safety model.
- Clone semantics require careful config updates for `id`, `name`, `workspace`, and `agentDir`, plus collision handling when target paths or agent ids already exist.
