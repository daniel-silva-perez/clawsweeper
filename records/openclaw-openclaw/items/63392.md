## Summary

Current main does not implement per-agent backup, restore, or clone. The backup CLI exposes only `create` and `verify`; the planner backs up whole state/config/credentials/workspace roots; there is no first-party restore command, `--agent`, `--as`, per-agent manifest scope, or clone-time config rewrite. Related #13616, #41111, and #59280 are adjacent or prerequisite work, but they do not fully supersede the per-agent granularity requested here.

## Best Possible Solution

Keep this open as a valid, narrower feature request. The best path is to settle the generic restore command design first, then add per-agent backup/restore/clone as an extension of the same manifest and safety model: `backup create --agent <id>`, restore with optional `--as <id>`, config entry rewrite for `id`/`name`/`workspace`/`agentDir`, scoped workspace and `agents/<agentId>` payloads, docs, and focused tests.

## Risks / Open Questions

- A per-agent restore/clone implementation would extract archive content and rewrite `openclaw.json`; it needs the existing traversal, no-overwrite, and path-containment protections preserved or extended.
- The requested archive includes auth profiles and session transcripts, so clone semantics need explicit product/security policy for sensitive credentials, provider state, and historical conversations.
- The first-party restore command is still open in adjacent work (#41111/#59280), so per-agent restore should be designed with or after the generic restore flow rather than as a separate incompatible archive format.
