## Summary

Keep open. Current main still exposes only whole-install `openclaw backup create` and `openclaw backup verify`; there is no backup restore command on main and no `--agent` or `--as` flow for per-agent backup, restore, or clone. The existing planner, archive manifest, tests, and docs are whole-state/workspace scoped. Related #41111 and #59280 are adjacent generic restore work, but they do not supersede this narrower per-agent granularity request.

## Best Possible Solution

Keep this issue open as a valid feature request. The best path is to settle the generic backup restore command and manifest safety model, then add per-agent scoping: `backup create --agent <id>`, restore with optional `--as <id>`, scoped workspace plus `agents/<agentId>` payloads, config-entry rewrite for `id`/`name`/`workspace`/`agentDir`, docs, and focused safety tests.

## Risks / Open Questions

- Per-agent restore/clone would extract archives, move state/workspace trees, and rewrite config, so it needs explicit traversal, containment, conflict, dry-run, and no-overwrite policy.
- The requested archive includes auth profiles and session transcripts; maintainers should decide how cloning handles sensitive credentials, provider auth, and session history before implementation.
- Current main lacks a generic restore command; building per-agent restore before settling #41111/#59280 could duplicate or conflict with the broader restore safety model.
