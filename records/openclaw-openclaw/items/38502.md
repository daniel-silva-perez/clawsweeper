## Summary

Keep open. Current main still routes TUI Ctrl+C through clear/warn/exit status handling with no active-run hint, and this PR is not ready because its hint is gated only on `activeChatRunId`, missing the reported quick submit window where `pendingOptimisticUserMessage` is true before the first chat event assigns the run id. The broader #44223 Ctrl+C behavior PR is still open, so this needs maintainer/author handling rather than cleanup closure.

## What This Changes

This PR appends an active-run `/abort` hint to the TUI Ctrl+C clear and exit-warning status messages in `src/tui/tui.ts` when `activeChatRunId` is set.

## Best Possible Solution

Choose one TUI policy and implement it consistently: either amend this PR so the hint covers both pending optimistic submissions and active run ids, with focused TUI tests and docs alignment, or supersede it with #44223's abort-first behavior if maintainers want Ctrl+C to stop runs directly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit a TUI message and press Ctrl+C before the first chat event arrives: `pendingOptimisticUserMessage` is true, `activeChatRunId` is still null, and this PR's hint condition does not fire while the run can continue.

## Solution Assessment

Unclear. A text-only hint is the narrowest maintainable fix if Ctrl+C should remain clear/exit, but this diff is too narrow for the reported timing; #44223 may be the better solution if maintainers want Ctrl+C to abort active runs.
