## Summary

Keep open. Current main still uses the old Ctrl+C clear/warn status text while active runs are aborted only through the separate Esc or `/abort` path, so this PR is not implemented on main. The patch is a narrow text-only UX clarification and I did not find a discrete correctness or security regression, but maintainers need to choose between this status-hint approach and the broader Ctrl+C abort semantics proposed in #44223.

## What This Changes

This PR appends an active-run hint to the TUI Ctrl+C clear and warn status messages in `src/tui/tui.ts` when `activeChatRunId` is set, without changing abort semantics or keybindings.

## Best Possible Solution

Keep one coherent TUI interaction policy: either land this narrow active-run status hint with focused test/manual proof and aligned PR text, or supersede it with #44223's broader Ctrl+C abort behavior and close the paired bug through the implementation that actually lands.

## Reproduction Assessment

Yes. The issue can be reproduced conceptually on current main by starting a TUI run, pressing Ctrl+C while `activeChatRunId` is still set, and observing that the status text says only clear/exit while the run continues unless Esc or `/abort` is used.

## Solution Assessment

Unclear. The patch is the narrowest maintainable fix if the desired policy is to preserve Ctrl+C as clear/exit, but #44223 may be the better product fix if maintainers want Ctrl+C to actually stop active runs.
