## Summary

Keep open: current main still has the reported TUI Ctrl+C UX mismatch, but this PR’s `activeChatRunId`-only hint misses the immediate post-submit state described in the linked bug. The overlapping open PR #44223 also means maintainers need to choose between a text-only hint and abort-first Ctrl+C behavior before landing this surface.

## What This Changes

The PR appends a `/abort` active-run hint to the TUI Ctrl+C clear and exit-warning status messages in `src/tui/tui.ts` when `activeChatRunId` is set.

## Best Possible Solution

Choose one TUI policy and implement it consistently: either keep Ctrl+C as clear/exit and make the status hint cover both pending optimistic submissions and active run ids, or supersede this text-only PR with #44223’s abort-first behavior. The chosen path should include focused TUI tests and keep the shortcut docs aligned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit a TUI message and press Ctrl+C before the first chat event arrives: `pendingOptimisticUserMessage` is true, `activeChatRunId` is still null, and the PR’s hint condition does not fire while the submitted run can continue.

## Solution Assessment

Unclear. A text-only hint is the narrowest solution if Ctrl+C should remain clear/exit, but this specific diff is too narrow for the reported timing; #44223 may be the better product direction if maintainers want Ctrl+C to stop active runs directly.
