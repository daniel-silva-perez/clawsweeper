## Summary

Keep this issue open. Current main still lacks generic bounded retry for ordinary transient OAuth refresh failures: the shared OAuth path calls the provider/plugin refresh once, the manager adds locking/timeouts/adoption fallbacks but not retry, and the only retry-like branch is specific to refresh_token_reused recovery. Both retry PRs found for this issue are closed and unmerged.

## Best Possible Solution

Keep this issue open and implement a bounded, tested retry policy around OAuth refresh. Prefer reusing src/infra/retry.ts with a provider-safe transient-error classifier; preserve the current lock/adoption/fallback/reused-token behavior; add focused auth-profile tests for success after a transient refresh failure, retry exhaustion preserving the final error, and non-retryable OAuth failures such as invalid_grant.

## Risks / Open Questions

- A retry fix needs a narrow transient-error classifier so invalid_grant, revoked refresh tokens, and other permanent OAuth failures are not retried as if they were temporary.
- Retry placement must preserve the current global refresh lock, hard refresh timeout, external/main-store credential adoption, provider-plugin hooks, and refresh_token_reused recovery.
- Provider plugins can own custom refreshOAuth behavior, so shared retry policy should avoid surprising double-retry semantics where a provider already retries internally.
