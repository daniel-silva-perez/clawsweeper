## Summary

Current main still matches the reported Microsoft Teams attachment bug. The `file.download.info` branch returns raw `content.downloadUrl`, while the Graph shares rewrite only applies to the separate `contentUrl` path. The latest release tag has the same behavior, and existing tests cover Graph shares only for reference/contentUrl attachments, not file.download.info downloadUrl attachments.

## Best Possible Solution

Keep this issue open. Patch `extensions/msteams/src/attachments/download.ts` so the `file.download.info` branch applies `tryBuildGraphSharesUrlForSharedLink(downloadUrl)` and uses the Graph shares URL when available. Add focused regression coverage where `content.downloadUrl` is a SharePoint/OneDrive URL and only the Graph shares endpoint succeeds with Graph-scoped auth retry. Keep the existing warning logger behavior for failed downloads.
