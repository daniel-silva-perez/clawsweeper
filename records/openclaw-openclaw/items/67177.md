## Summary

Keep open. Current main still matches the reported Microsoft Teams attachment bug: the `file.download.info` branch returns raw `content.downloadUrl`, while the existing Graph shares rewrite only runs on the later `contentUrl` path. The latest release tag has the same source behavior, and the current regression tests do not cover SharePoint/OneDrive `downloadUrl` rewriting for `file.download.info` attachments.

## What This Changes

The issue requests rewriting Microsoft Teams `file.download.info` attachment `content.downloadUrl` values through the Graph shares endpoint, plus better regression coverage for silent attachment download failures.

## Best Possible Solution

Patch the bundled msteams plugin so the `file.download.info` branch applies the existing Graph shares rewrite to `content.downloadUrl`, while preserving direct-download behavior for non-SharePoint/OneDrive URLs. Add focused tests for a SharePoint/OneDrive `downloadUrl` that succeeds only through the Graph shares endpoint and keep the current warning logger contract for download failures.
