## Summary

Keep open. Current main still performs the per-source realpath containment check and emits a warn-level skills log before skill-name merging, so a personal `~/.agents/skills` symlink to an already-bundled skill still matches the reported warning-noise behavior. The provided context also shows an open implementation candidate with closing syntax, so the issue should remain open until that PR or an equivalent fix is resolved.

## Best Possible Solution

Keep this issue open. Review the open implementation candidate or land an equivalent focused skill-loader change that preserves the realpath containment guard while suppressing or downgrading the warning when a personal `.agents/skills` symlink resolves outside its root but the same skill is already available from a trusted bundled or managed source. Add regression coverage for that duplicate trusted-source symlink case.
