## Summary

Keep open. Current main still performs the realpath containment check per skill source and emits a warn-level skills log before the later skill-name merge, so a personal `.agents/skills` symlink to an already-bundled skill can still produce the reported warning noise. The discussion also includes an open implementation candidate, so the right path is maintainer review or an equivalent focused fix, not closing this issue as already resolved.

## Best Possible Solution

Keep this issue open. Review the open implementation candidate or land an equivalent focused change in `src/agents/skills/workspace.ts` that preserves realpath containment while suppressing or downgrading warnings when a personal `.agents/skills` symlink resolves outside its root but the same skill is already available from a trusted bundled or managed source. Add a regression test for that duplicate trusted-source symlink case before closing.
