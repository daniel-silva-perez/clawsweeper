## Summary

Current main still does not implement remote reranker endpoint support for memory search. The issue is concrete, externally authored, unprotected, and not superseded by the provided related items; it should stay open for a focused design/implementation.

## Best Possible Solution

Keep this issue open for a focused memory-search design. The likely path is an additive remote reranker contract covering config types/schema/help/docs, SecretRef targets, SSRF/private-network policy, request timeout/fallback behavior, a Cohere-compatible HTTP client, builtin post-retrieval reranking tests, and an explicit QMD integration decision. If maintainers prefer plugin-first, first add a documented reranker provider API so Qwen/Cohere/Jina adapters can live outside core without replacing the whole memory backend.

## Risks / Open Questions

- Adding this feature introduces a new outbound HTTP surface carrying memory snippets, so the implementation needs explicit SSRF/private-network policy, timeout behavior, SecretRef handling, and redaction/logging safeguards.
- QMD integration may require either upstream QMD support for remote rerankers or a clear OpenClaw-side post-processing design, because current QMD search consumes qmd/mcporter-ranked results.
