## Summary

Keep open. Current main still scopes automatic 1M Claude context promotion and `params.context1m` handling to `anthropic` and `claude-cli`, and an existing test explicitly preserves the non-promotion behavior for `github-copilot` Opus 4.7 IDs. The configured provider-model context override path appears to exist, so the remaining maintainer question is whether Copilot 1M aliases should be recognized by name/opt-in or whether the reported 272k status is coming from a more specific config shape or persisted session value.

## What This Changes

The issue asks OpenClaw to stop reporting GitHub Copilot Claude Opus 4.x 1M model aliases as 272k when the model name or user config indicates a 1M-capable context window.

## Best Possible Solution

Define a provider-owned Copilot long-context policy: either support explicit `context1m` or `*-1m`/`*-1m-internal` Copilot aliases with focused resolver/runtime/status tests and docs, or document that Copilot long context must be expressed only through `models.providers.github-copilot.models[].contextTokens` and make sure persisted status cannot mask that configured override unexpectedly.

## Reproduction Assessment

Yes for the source-level gap: current main has a focused resolver test proving `github-copilot` Opus 4.7 IDs are not promoted to 1M. The exact 272k status report is only partially reproducible from the issue text because current code should honor a matched provider-model `contextTokens` override, so the remaining reproduction needs the full config shape and session-store entry.

## Solution Assessment

Unclear. Dropping the provider gate for every provider is broader than the current docs/tests support; the safer path is a narrow Copilot opt-in or suffix policy after confirming the provider contract, plus regression coverage for configured `contextTokens` precedence in status and runtime session persistence.
