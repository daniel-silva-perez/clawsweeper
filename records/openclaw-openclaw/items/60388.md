## Summary

Keep open. Current main still routes timeout-shaped aborts through the model fallback path, and the open, unmerged implementation candidate #62682 is explicitly paired with this issue.

## What This Changes

The issue requests that embedded agent runs fail immediately when their own timeout budget expires instead of spending remaining time on configured model fallbacks.

## Best Possible Solution

Review and land a narrow fix like #62682 that passes terminal abort context into the model fallback layer or otherwise classifies the run controller's timeout reason, with focused tests proving run-budget timeouts stop while provider timeouts still fall back.

## Reproduction Assessment

Yes. The source path is high-confidence: `scheduleAbortTimer` calls `abortRun(true)`, the abort reason is a `TimeoutError`, `isTimeoutError` keeps that AbortError in the fallback loop, and the assistant failover policy can escalate timed-out runs to `fallback_model`.

## Solution Assessment

Yes, the requested direction is the right maintainable boundary if implemented narrowly. The best fix is to distinguish run-budget abort context at the fallback boundary rather than disabling all timeout fallback behavior.
