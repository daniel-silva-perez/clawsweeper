## Summary

Keep open. Current main still has the stale Control UI session dropdown path: a fresh `sessions.list` response replaces the cached backend rows, but the chat session dropdown unconditionally re-adds the active `sessionKey`, so a deleted active direct session can remain visible. The related fix attempt is closed unmerged, and the remaining fix is narrow enough for a focused UI repair PR.

## What This Changes

The issue requests that the Control UI sessions dropdown stop showing deleted sessions after the Gateway sessions store no longer contains them.

## Best Possible Solution

Treat a fresh `sessions.list` result as authoritative for normal direct sessions, switch a missing active direct key to a valid fallback such as the Gateway main/default session, keep intentional ephemeral subagent and cron fallbacks visible, and add regression coverage for both paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: set the Control UI active `sessionKey` to a deleted direct session, return a fresh `sessions.list` result without that key, and `resolveSessionOptionGroups` still adds the missing active key to the dropdown.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Control UI selection/dropdown path; fetching fresh data alone is not sufficient while the renderer reinserts a missing active direct key.
