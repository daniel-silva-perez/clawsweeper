## Summary

Keep open. Current main still re-adds the active Control UI chat session key to the dropdown even when the fresh `sessions.list` result no longer contains it, so a deleted active Discord/direct session can remain visible. The linked attempted fix PR #48190 was closed unmerged, and I did not find an equivalent landed normalization path in current main.

## Best Possible Solution

Use the closed PR #48190 discussion as a starting point: when a fresh `sessions.list` omits the active normal direct session, normalize the active selection to a valid fallback and stop synthesizing that missing direct key into the chat dropdown. Preserve intentional ephemeral fallback behavior for subagent or cron sessions if product still wants it, and add focused regression tests for deleted active direct sessions after `sessions.list`/`sessions.changed`.
