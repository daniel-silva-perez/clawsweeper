## Summary

Keep open. Current main still rebuilds the Control UI chat/session dropdown from the fresh `sessions.list` response but then unconditionally adds the current `sessionKey`, so a deleted active direct session can still appear after the Gateway no longer reports it. The prior direct fix PR #48190 was closed unmerged, and no protected labels or open fixing PR are present.

## Best Possible Solution

Land a focused Control UI fix that treats a fresh `sessions.list` result as authoritative for normal direct sessions: stop rendering a missing active direct session, normalize `sessionKey` and `lastActiveSessionKey` to a valid fallback such as the Gateway main/default session after refresh, preserve missing-key fallback for ephemeral subagent and cron sessions, and add regression coverage for both paths.
