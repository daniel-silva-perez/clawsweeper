## Summary

Keep open. Current main still has the stale dropdown path: deletion triggers a session refresh, but the chat dropdown resolver unconditionally re-adds the active session key even when fresh Gateway rows no longer contain that deleted direct session; the direct fix attempt in #48190 is closed unmerged.

## What This Changes

The issue asks the Control UI chat sessions dropdown to stop showing deleted backend sessions after the Gateway session store no longer contains them.

## Best Possible Solution

Treat fresh `sessions.list` rows as authoritative for normal direct sessions, normalize a missing active direct key to a valid fallback such as the Gateway main/default session, preserve intentional missing subagent and cron fallbacks, and add regression coverage for both behaviors.

## Reproduction Assessment

Yes. A source-level reproduction is clear: set the Control UI active `sessionKey` to a deleted direct session, return a fresh `sessions.list` response without that key, and `resolveSessionOptionGroups` still adds that missing key to the dropdown.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Control UI session selection/dropdown path; fetching fresh data alone is insufficient while the renderer reinserts a missing active direct key.
