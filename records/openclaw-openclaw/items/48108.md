## Summary

Keep open. Current main still rebuilds Control UI session options from the fresh `sessions.list` result and then unconditionally adds the active `sessionKey`, so a deleted active direct session can remain visible even after the Gateway no longer reports it. The refresh path updates `sessionsResult` and checkpoint caches but does not normalize `sessionKey` or `lastActiveSessionKey` away from a missing normal session. The earlier direct fix PR #48190 is closed unmerged, so this remains a narrow valid bug.

## Best Possible Solution

Land a focused Control UI fix that treats a fresh `sessions.list` result as authoritative for normal direct/group/channel sessions, normalizes a missing active normal session to a valid fallback such as the Gateway main/default session, preserves the existing missing-key fallback for ephemeral subagent and cron sessions, and adds regression coverage for both paths.
