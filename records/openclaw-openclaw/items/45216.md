## Summary

Keep this PR open. Current main still has the top-level string `detail` parser gap, and the PR is a narrow formatter/test fix for the linked chat-visible raw JSON leak; it needs maintainer review and a rebase or port because the branch is currently dirty against main.

## What This Changes

The PR updates shared assistant error formatting to recognize top-level string `detail` JSON payloads, use `detail` as a fallback error message, and add exact-output formatter tests.

## Best Possible Solution

Port or rebase the focused string-`detail` parser support into the shared assistant error formatter, keep the exact-output regression tests, and leave provider credential-gating or unconfigured-provider suppression as follow-up work under #44910.

## Reproduction Assessment

Yes. The source path is high-confidence: current main rejects `{ "detail": "..." }` in `parseApiErrorPayload`, cannot extract it in `parseApiErrorInfo`, and then falls through to raw short text in `formatAssistantErrorText`/the sanitizer paths.

## Solution Assessment

Yes. For the parser leak covered by this PR, recognizing top-level string `detail` and using it as a fallback message is the narrowest maintainable fix; nested `detail` objects and provider probe suppression are valid follow-ups rather than blockers for this patch.
