## Summary

Keep open. Current main mitigates the original plain `openclaw status` path by skipping memory inspection there, but the remaining status memory inspection path still resolves the selected memory plugin in the CLI process and probes vector/status locally. Also, open PR #62643 is a same-author, unmerged implementation candidate with closing syntax for this issue, so this issue should stay open until that PR is reviewed, merged, or explicitly superseded.

## What This Changes

The issue asks that `openclaw status` avoid initializing active memory plugins in the CLI process and avoid reporting memory/vector state from a CLI-local runtime that can diverge from the live gateway.

## Best Possible Solution

The durable end state is for status memory inspection to either use the live gateway runtime as the source of truth or have an enforced passive status contract that prevents plugin startup side effects and false local vector state, with regression coverage for custom memory plugins.

## Reproduction Assessment

Partially yes. The original plain `openclaw status` path no longer reproduces on current main, but the remaining call path is source-reproducible by configuring a custom memory slot and running `status --json --all`, which still loads the memory runtime locally and probes vector state.
