## Summary

Keep open. Current main still lacks a per-channel or per-binding Discord identity override for ordinary route bindings. The existing webhook persona support is real, but it is scoped to stored Discord thread bindings, not one bot token serving multiple agent-bound guild channels with distinct visible names. The related closed #46351 is useful context for the webhook-based design, but it does not supersede this issue because the remaining route-binding/channel use case is still unimplemented.

## Best Possible Solution

Keep the issue open and implement this in the bundled Discord plugin. A conservative path is to add a documented Discord route-binding identity override, or reuse routed `agents.list[].identity`, then extend Discord outbound delivery to use per-channel webhooks where available with safe fallback to bot-token sends. Update config types/schema/help, Discord docs, and tests covering normal channel bindings, missing webhook permission, and webhook echo suppression.

## Risks / Open Questions

- Discord bot nickname mutation may be guild-scoped rather than channel-scoped, so an implementation should validate Discord API behavior and likely prefer per-message webhook identity where possible.
- Extending webhook persona delivery to ordinary channel bindings needs explicit fallback and echo-suppression behavior for missing Manage Webhooks permission, media/poll/silent sends, and `allowBots` configurations.
