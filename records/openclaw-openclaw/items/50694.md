## Summary

Keep this PR open. Current main and the latest tagged release still use the raw memory-flush threshold math, so oversized `reserveTokensFloor` values can still collapse the token gate; the PR branch is a focused implementation path that centralizes the threshold calculation and adds regression coverage for the reported edge cases.

## What This Changes

This PR adds a shared memory-flush threshold resolver, uses it in preflight and runtime memory-flush gates, adds regression tests for oversized reserve floors and small context windows, and records the fix in the changelog.

## Best Possible Solution

Land this PR or an equivalent narrow patch that uses one context-aware threshold helper for both preflight diagnostics and runtime flush decisions, preserves valid high reserve budgets, and keeps regression coverage for oversized reserves and small context windows. After merge, close #50611 as fixed and leave #51316 as superseded prior art.

Yes. A source-level reproduction is clear on current main: set `contextWindowTokens` to `200000`, `reserveTokensFloor` to `200000`, and any positive soft threshold; the threshold resolves to `0`, `resolveMemoryFlushGateState` returns `null`, and the memory flush token gate cannot fire.

## Solution Assessment
