## Summary

Keep open. Current main still has the reported timestamp-refresh path: compaction count writes explicitly set `updatedAt`, flush metadata persists through the default session merge that advances `updatedAt`, and freshness evaluation still falls back to `updatedAt` when legacy lifecycle fields and a readable session header are absent. The prior #49515 fix is useful precedent, but it only covered route metadata writes, not compaction/flush bookkeeping.

## What This Changes

The issue reports that compaction and memory-flush metadata writes refresh `updatedAt`, which can hide daily or idle reset expiry for legacy session rows lacking lifecycle timestamps.

## Best Possible Solution

Add an explicit preserve-activity path for compaction and memory-flush bookkeeping writes, keep real user/agent turns responsible for activity timestamps, and add focused regression coverage for legacy daily and idle rows without lifecycle fields.

## Reproduction Assessment

Yes. A high-confidence reproduction is a legacy session store row with no `sessionStartedAt` and, for idle, no `lastInteractionAt`, then a compaction or flush metadata write that moves `updatedAt` past the relevant boundary before `initSessionState` calls `evaluateSessionFreshness`. I did not run tests because this review was required to remain read-only, but the source path is concrete.

## Solution Assessment

Yes, a narrow preserve-activity fix is the best direction. Reusing the existing #49515 merge policy pattern for compaction/flush metadata is safer than changing reset evaluation broadly or making all session-store patches preserve activity by default.

## Repair Work Prompt

Fix the session reset race reported in openclaw/openclaw#75037. Compaction and memory-flush metadata writes must not refresh activity timestamps used by daily/idle reset decisions for legacy session rows. Reuse the existing preserve-activity merge pattern from the #49515 `updateLastRoute` fix where appropriate. Likely approach: add an explicit preserve-activity option/helper for `updateSessionStoreEntry` metadata-only callers, use it for the memory-flush metadata write in `src/auto-reply/reply/agent-runner-memory.ts`, and stop `incrementCompactionCount` in `src/auto-reply/reply/session-updates.ts` from advancing `updatedAt` for same-conversation compaction bookkeeping. Do not globally change all `updateSessionStoreEntry` callers without auditing them, and do not change daily/idle reset semantics for rows that have real `sessionStartedAt` or `lastInteractionAt`. Add regression tests covering: legacy daily row missing `sessionStartedAt` where compaction advanced `updatedAt` past the 04:00 boundary; legacy idle row missing both `lastInteractionAt` and `sessionStartedAt` where flush metadata advanced `updatedAt`; controls where real lifecycle timestamps remain authoritative. Since this affects released session behavior, add a concise changelog fix entry with reporter credit unless maintainers determine it is main-only.
