## Summary

Keep this PR open. Current main still treats valid all-zero provider usage as missing usage in the subscription, accumulator, persistence, and status paths, and the PR only relaxes one guard in an older subscribe path, so the existing review comments are still materially correct.

## What This Changes

The PR removes the `hasNonzeroUsage` import from `src/agents/pi-embedded-subscribe.ts` and changes `recordAssistantUsage()` to ignore only absent normalized usage rather than all-zero normalized usage.

## Best Possible Solution

Implement an observed-usage signal independent of nonzero counters, carry it through subscription aggregation, runner usage metadata, session persistence, and status/transcript fallback, then add focused regression coverage proving `{ input: 0, output: 0, total: 0 }` displays as fresh zero usage while absent usage still renders unknown or stale.

Yes. A high-confidence code-level reproduction is available: an all-zero normalized usage object currently fails the `hasNonzeroUsage()` gates in `resolveAssistantUsage()`, the runner accumulator, and persistence, and status tests show missing fresh totals render as `Context: ?/...`.

No. The PR changes only one guard in `recordAssistantUsage()` and does not update the current-main `resolveAssistantUsage()` nonzero filter, `getUsageTotals()`, runner metadata, session persistence, or tests, so it is not the narrow complete fix for the reported behavior.
