## Summary

Keep open. Current main still preserves managed-over-workspace hook precedence and still has the diagnostic gap reported here: eventless managed hooks can win the name collision, then be skipped by the runtime loader while hook status/check output still treats them as loadable because readiness does not consider empty event metadata.

## What This Changes

The issue asks for workspace hook collisions and eventless managed-hook replacements to be surfaced as actionable diagnostics instead of silently disabling the workspace hook logic.

## Best Possible Solution

Preserve the current trust boundary, but make the failed outcome visible: track ignored workspace hook collisions, mark enabled hooks with empty event metadata as not ready, expose clear reasons in text and JSON status/check/info output, and document rename/remove guidance for same-name workspace hooks.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to define a managed hook and workspace hook with the same name, leave the managed hook with no metadata events, and observe that current policy keeps the managed entry, the loader skips it, and status readiness still ignores the empty event list.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable fix; allowing workspace hooks to override managed hooks would be a larger trust-policy change and is not needed to make the reported failure actionable.
