## Summary

Keep open. Current main still preserves managed-over-workspace hook precedence, which makes the name-collision part expected, but the diagnostics gap remains: an enabled hook with empty event metadata is skipped by the runtime loader while status/list/check can still report it as ready, leaving the user without an actionable signal.

## What This Changes

Workspace-defined hooks that share names with managed OpenClaw hooks are rejected at startup, and same-named managed hooks without event metadata can leave the workspace hook logic inactive without actionable status output.

## Best Possible Solution

Keep the managed-over-workspace trust policy, but make the outcome visible and actionable: report ignored same-name workspace hooks in status/list/check output, mark enabled hooks with no metadata events as not ready, expose that reason in text and JSON output, and document rename/remove guidance for colliding managed hooks.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by defining same-name managed and workspace hook entries and observing that the managed entry wins; if that managed entry has empty event metadata, the loader skips it while status still computes it as loadable from enablement and requirements only.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; changing workspace override precedence would be a broader trusted-code policy change and should not be part of an automated cleanup fix.
