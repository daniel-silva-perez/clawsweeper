## Summary

Keep open. Current main still intentionally prevents workspace hooks from overriding managed hooks, so the name-collision behavior is expected policy, but the actionable diagnostics gap remains: an enabled managed hook with no metadata events is skipped by the loader while hook status/list/check can still report it as ready. A prior unmerged PR, #30154, attempted the no-events status alignment for older issue #24981, but that fix is not on main and the broader collision visibility from this issue is still missing.

## What This Changes

The issue requests that same-name workspace hooks not be silently shadowed by managed hooks that have no runnable events, or at least that OpenClaw surface the collision and no-events condition clearly in status/CLI diagnostics.

## Best Possible Solution

Preserve managed-over-workspace precedence, but make the failure mode actionable: capture ignored hook collisions in hook status diagnostics, mark enabled hooks with empty event metadata as not ready, expose both reasons in text and JSON output, and document renaming or removing the colliding managed hook as the user path.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by defining same-name managed and workspace hooks, observing the managed entry wins, then using a managed entry with no metadata events: the loader skips it while current status logic can still mark it loadable.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; allowing workspace override would change hook trust policy and should require explicit maintainer product/security review.
