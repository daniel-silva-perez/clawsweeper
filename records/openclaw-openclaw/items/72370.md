## Summary

Keep open. Current main still intentionally rejects workspace hooks that collide with managed hooks, and the remaining bug is the diagnostics gap: an enabled managed hook with empty `metadata.openclaw.events` is treated as ready by status/list/check while the runtime loader warns and skips registration.

## What This Changes

The issue reports same-name workspace hooks being shadowed by managed hooks and asks for actionable diagnostics or safer handling when the kept managed hook has no events.

## Best Possible Solution

Preserve the managed-over-workspace trust boundary, but make the shadowing actionable: carry collision information into hook status, mark enabled hooks with no events as not ready, surface this in `openclaw hooks list/info/check` text and JSON, and document that users should rename workspace hooks or remove/update the managed hook pack when a managed hook shadows local code.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists from current main: same-name managed and workspace hook entries resolve to the managed hook, and enabled hooks with empty events are skipped by the loader while status still considers them loadable if requirements pass.

## Solution Assessment

Yes, for a diagnostics-focused fix. Allowing workspace hooks to override managed hooks would change the trusted-code precedence model, so the narrow maintainable path is to keep precedence intact and make collisions plus empty-event hooks visible and not ready.
