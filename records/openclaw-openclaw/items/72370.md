## Summary

Keep open. Current main still enforces managed-over-workspace hook precedence, which makes the same-name override rejection expected, but the diagnostics gap remains: the runtime loader skips hooks with empty event metadata while status/list/check can still treat those hooks as ready and do not surface ignored workspace collisions.

## What This Changes

The issue asks OpenClaw to make same-name workspace hook collisions and eventless managed hook replacements visible or actionable instead of silently disabling the user's workspace hook logic.

## Best Possible Solution

Keep the managed-over-workspace trust policy, but make the outcome visible and actionable: track ignored same-name workspace hooks in status/list/check output, mark enabled hooks with no metadata events as not ready, expose that reason in text and JSON output, and document rename/remove guidance for colliding managed hooks.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by defining same-name managed and workspace hook entries, then making the kept managed entry eventless; current source shows the managed entry wins, the loader skips it, and status still computes readiness without considering events.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; allowing workspace hooks to override managed, bundled, or plugin hooks would be a broader trust-policy change.
