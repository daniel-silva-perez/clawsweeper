## Summary

Keep this PR open. Current main still lacks the requested `file` display alias in the TypeScript tool-summary path and still has the embedded read-start diagnostic checking only `path`/`file_path`, so the PR is not obsolete. The remaining decision is maintainer/product judgment about whether display and diagnostics should continue recognizing legacy aliases now that read/write/edit execution validation requires canonical `path`.

## What This Changes

The PR adds `file` as a read/write/edit display path candidate, extends the embedded read-start diagnostic guard to recognize `filePath` and `file`, and adds a focused read-summary assertion for `{ file: ... }`.

## Best Possible Solution

Review this PR together with #54882 and settle the path-alias contract explicitly: either land an updated display/diagnostic compatibility fix, including any intended native display surface, or standardize user-facing behavior around canonical `path` and update the linked report accordingly.
