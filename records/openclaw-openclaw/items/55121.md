## Summary

Keep this PR open. Current main still does not implement the proposed `file` alias handling in the shared tool-display resolver or embedded read-start diagnostic, and the linked #54882 remains an open implementation target. The PR is small and display/logging-only, but it should be refreshed against current main because runtime validation now intentionally rejects legacy aliases instead of normalizing them for execution.

## Best Possible Solution

Keep this PR open and ask for an update against current main. The maintainers should decide the alias policy explicitly: either accept display/diagnostic compatibility by adding `record.file` to `resolvePathArg()`, aligning the embedded read-start check, and keeping the focused regression test, or close the linked bug/PR after documenting that only canonical `path` is supported.
