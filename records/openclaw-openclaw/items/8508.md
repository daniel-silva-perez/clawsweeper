## Summary

Current main still only supports a single configured ack reaction string plus lifecycle status-reaction states. There is no `ackReactions` array, selector contract, random/contextual chooser, or pre-ack model hook, and the prior maintainer/Codex comment in the issue reached the same conclusion two days earlier.

## Best Possible Solution

Keep this issue open for a backward-compatible core/channel design. Preserve the existing `ackReaction` string contract, then add a clearly named `messages.ackReactions` array or selector/strategy contract once maintainers choose deterministic, random, rule-based, or model-driven behavior. Update exported config types, zod/generated schemas, help/metadata, public docs, SDK channel-feedback contracts if third-party channels need the seam, and focused bundled-channel tests for Slack, Discord, Telegram, WhatsApp, and BlueBubbles fallback behavior.
