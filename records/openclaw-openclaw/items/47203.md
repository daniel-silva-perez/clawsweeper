## Summary

Keep open. Current main still drops navigate-specific `loadState` and `timeoutMs` before the Playwright navigation call, so the reported Linux/VNC 20s timeout remains a valid narrow browser plugin bug; the reporter’s direct HTTP API evidence and a second Linux VPS confirmation make the product symptom credible.

## What This Changes

The issue requests that browser navigate honor navigation load-state and timeout controls instead of timing out at the fixed 20s tool/client boundary while Playwright waits for page load.

## Best Possible Solution

Add a focused browser plugin fix that validates navigate-specific load-state and timeout inputs, propagates them through the agent tool, CLI, browser client, proxy dispatch, `/navigate`, `navigateViaPlaywright`, and `gotoPageWithNavigationGuard`, gives the outer request budget slack over the navigation timeout, and preserves the existing SSRF and final-navigation-result guards.

## Reproduction Assessment

Yes. The issue gives concrete Linux VNC steps, and current-main static inspection gives a high-confidence unit-level reproduction: pass `loadState` or a custom `timeoutMs` through browser navigate and observe that both are dropped before `page.goto`.

## Solution Assessment

Yes for a narrow pass-through and timeout-budget repair. A broader browser-control rewrite or raw-CDP default is not necessary for this report; any default load-state change should be explicit, documented, and covered by route/guard tests.
