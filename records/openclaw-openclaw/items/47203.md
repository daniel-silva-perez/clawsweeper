## Summary

Keep open. Current main still drops navigate-specific `loadState` and `timeoutMs` before the Playwright navigation path, and `page.goto` still runs with only a timeout, so the reported 20s Linux/VNC timeout remains a valid narrow browser plugin bug.

## What This Changes

The issue asks for `browser navigate` to honor load-state and timeout controls so Playwright navigation does not race the fixed 20s tool timeout or report a misleading browser-control-service reachability error.

## Best Possible Solution

Add a focused browser plugin fix that validates navigate-specific load-state and timeout inputs, propagates them through the agent tool, CLI, browser client, node/proxy dispatch, `/navigate`, `navigateViaPlaywright`, and `gotoPageWithNavigationGuard`, gives the outer request budget slack over the navigation timeout, and preserves SSRF/navigation-result guards.

## Reproduction Assessment

Yes. The issue provides concrete Linux VNC steps, and current-main static inspection gives a high-confidence unit-level reproduction: pass `loadState` or a custom `timeoutMs` through browser navigate and observe that the current tool/client/route path drops both before `page.goto`.

## Solution Assessment

Yes for the narrow pass-through and timeout-budget fix. A broader browser-control rewrite is not needed; any default change from `load` to `domcontentloaded` or `commit` should be deliberate, documented, and covered by route/guard tests.
