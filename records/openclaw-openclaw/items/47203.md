## Summary

Keep open. Current main still drops navigate-specific load-state and timeout data through the browser tool, client, CLI, and `/navigate` route, then calls Playwright navigation without a `waitUntil` option while the caller remains capped at 20s. The issue discussion includes a detailed Linux VNC reproduction plus a second Linux confirmation and workable bypasses, so this remains a valid narrow browser plugin bug.

## What This Changes

The issue requests that browser navigate stop timing out on Linux VNC by forwarding load-state/timeout options and avoiding the 20s outer timeout racing Playwright navigation.

## Best Possible Solution

Implement a focused browser plugin fix that validates navigate-specific load state and timeout values, forwards them through the agent tool, CLI, client, node/proxy request, `/navigate` route, `navigateViaPlaywright`, and `gotoPageWithNavigationGuard`, gives the outer request timeout slack beyond the navigation timeout, and preserves the existing SSRF/navigation-result guards.

## Reproduction Assessment

Yes. The issue provides concrete Linux VNC steps, and current-main static inspection gives a focused unit-level reproduction: pass `loadState` or a custom `timeoutMs` through browser navigate and observe that the current tool/client/route path drops both before `page.goto`.

## Solution Assessment

Yes. The narrow maintainable solution is a browser plugin pass-through plus timeout-budget fix with route and guard coverage, not a core migration or browser-control rewrite. Defaulting navigate away from full `load` may be appropriate, but it should be explicit and covered by tests.
