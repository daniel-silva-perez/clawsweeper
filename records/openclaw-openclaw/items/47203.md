## Summary

Keep open. Current main still matches the reported failure path: browser navigate accepts only URL/targetId through the tool, client, CLI, and `/navigate` route, then calls Playwright `page.goto` with only a timeout and no `waitUntil`, while the outer browser request remains capped at 20s. The issue discussion includes a detailed direct-HTTP workaround and a second Linux confirmation, and I did not find an open PR already carrying the fix.

## What This Changes

The issue asks for browser navigation to avoid the 20s tool timeout race by honoring navigate load-state/timeout options and reporting navigation timeouts distinctly from browser-control reachability failures.

## Best Possible Solution

Implement a focused browser plugin fix that adds navigate-specific load-state validation, forwards load state and navigation timeout through the agent tool, CLI, client, gateway/node proxy, `/navigate` route, `navigateViaPlaywright`, and `gotoPageWithNavigationGuard`, adds outer request timeout slack beyond the navigation timeout, keeps SSRF/navigation-result guards intact, and updates the user-facing browser docs/changelog if the CLI/API surface changes.

## Reproduction Assessment

Yes. The issue provides a concrete Linux VNC reproduction, and current-main static inspection gives a focused unit-level reproduction: pass `loadState` or `timeoutMs` through browser navigate and observe that the current tool/client/route path drops both before `page.goto`.

## Solution Assessment

Yes, the best fix is a narrow pass-through and timeout-budget change in the browser plugin rather than a core migration or unrelated browser-control rewrite. Defaulting navigate to `commit` or `domcontentloaded` may be reasonable, but it should be paired with explicit validation and SSRF-preserving route coverage.
