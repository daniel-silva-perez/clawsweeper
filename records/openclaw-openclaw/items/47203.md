## Repair Work Prompt

Fix this issue in one focused browser plugin PR. Current main exposes top-level `loadState` and `timeoutMs` in the browser tool schema, but `action: "navigate"`, the browser client, CLI command, `/navigate` route, and Playwright helper only forward URL/targetId and then call `page.goto` without `waitUntil`, with a 20s request timeout racing the navigation timeout.

Expected boundary:
- Add a navigate-specific load-state type/normalizer supporting Playwright `page.goto` values `load`, `domcontentloaded`, `networkidle`, and `commit`.
- Forward `loadState` and `timeoutMs` through `extensions/browser/src/browser-tool.ts`, `extensions/browser/src/browser/client-actions-core.ts`, `extensions/browser/src/browser/client-actions.types.ts`, `extensions/browser/src/browser/routes/agent.snapshot.ts`, `extensions/browser/src/browser/pw-tools-core.snapshot.ts`, and `extensions/browser/src/browser/pw-session.ts` so Playwright receives `waitUntil`.
- Add CLI support in `extensions/browser/src/cli/browser-cli-actions-input/register.navigation.ts` for the same load-state option and make navigate respect caller timeout rather than forcing an equal 20s HTTP cap.
- Add request/proxy timeout slack so the client reports the real navigation error instead of aborting first with the generic browser-control-service timeout.
- Preserve existing SSRF/navigation policy checks, including redirect/subframe checks. Do not add an unguarded raw CDP fallback.
- For Chrome MCP/existing-session navigation, pass `timeoutMs` where already supported and either reject unsupported non-default `loadState` clearly or document that load-state only applies to Playwright-backed profiles.
- Add targeted regression tests for tool, client, route, and Playwright pass-through. Existing tests around navigation guard and SSRF should keep passing.
- Add a single-line user-facing changelog fix entry if behavior changes, with `Thanks @ldfandian` and `Thanks @astroclaw` attribution.

## Risks / Open Questions

- Changing the default navigate wait state from Playwright's `load` could affect scripts that implicitly expect fully loaded pages; explicit `loadState` support plus a documented default is safer than an unreviewed silent behavior shift.
- A raw CDP fallback or `commit` implementation could weaken current SSRF, redirect, and subframe navigation checks unless it stays inside the existing guard path or reuses the same policy checks.
- Chrome MCP / existing-session navigation has a different backend contract; any new `loadState` option should be rejected clearly or documented as Playwright-profile-only rather than silently ignored.
