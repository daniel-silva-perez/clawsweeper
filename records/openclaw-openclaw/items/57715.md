## Summary

Keep open. Current main has the lower-risk early filter/search/active/limit path before row construction, with regression coverage for avoiding transcript fallback on discarded limited rows, but the issue is not fully resolved. `buildGatewaySessionRow()` still resolves transcript fallback for returned rows with missing usage fields, and child-session resolution still scans the full store per built row instead of using the request-scoped child index requested in the report and in related PR #59436.

## Best Possible Solution

Keep this issue open until #59436, or an equivalent maintainer-reviewed patch, lands with a request-scoped child session index and any needed performance/regression coverage. The landed fix should preserve the current early filter/search/active/limit behavior, avoid per-row full-store child scans, and treat TTL caching as a separate follow-up decision if repeated unchanged reads still dominate.
