## Summary

Keep open. Current main has already implemented the lower-risk early filter/search/active/limit path before row construction, and there is regression coverage that `limit` avoids transcript fallback reads for discarded rows. But the report is not fully resolved: row construction still performs transcript fallback for returned rows with missing usage fields, `resolveChildSessionKeys()` still scans the full session store per built row, there is no request-scoped child index in current main, and related implementation PR #59436 is still open/unmerged in the provided context.

## Best Possible Solution

Keep this open until #59436, or an equivalent maintainer-reviewed patch, lands with a request-scoped child session index and any needed benchmark or regression coverage. The landed implementation should preserve the current early filter/limit behavior, avoid per-row full-store child scans, and decide separately whether a TTL/cache layer is warranted for repeated unchanged reads.
