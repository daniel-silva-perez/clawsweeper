## Summary

Keep open. Current main has Slack Block Kit plumbing for explicit `channelData.slack.blocks`, shared presentation blocks, and interactive replies, but ordinary Slack text is still converted to Slack `mrkdwn` text rather than automatically rendered as `rich_text` blocks. The broader #12602 remains open, but this RFC is a narrower actionable follow-up with unresolved Slack product/API questions, so it is not implemented or safely duplicate/superseded.

## Best Possible Solution

Keep this issue open as the focused Slack rich-text follow-up under #12602. The likely implementation belongs in `extensions/slack`: add a MarkdownIR-to-Slack-`rich_text` renderer, decide opt-in versus default behavior, skip conversion when explicit Slack blocks or shared presentation blocks are present, and cover normal sends, slash replies, streaming finalization, chunking, lists, quotes, links, inline styles, and fenced code blocks in Slack tests and docs.

## Risks / Open Questions

- Automatic rich_text conversion would affect Slack normal sends, slash replies, streaming preview/final edits, chunking, fallback text, explicit-block precedence, and Block Kit limit handling, so it needs Slack-specific design direction and regression coverage before implementation.
- Slack rich_text element shapes, limits, and syntax-highlighting behavior should be verified against Slack's current API/types during implementation.
