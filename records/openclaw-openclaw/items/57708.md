## Summary

Keep open. Current main has the Slack Block Kit plumbing described in the related merged PRs, but it still auto-renders ordinary Slack text through `mrkdwn`, not Slack `rich_text` blocks. The broader #12602 remains open, but this issue adds a narrower, actionable RFC with a specific `MarkdownIR` to `rich_text` mapping and unresolved product questions, so it should not be swept closed as implemented or duplicate yet.

## Best Possible Solution

Keep this as a focused Slack follow-up under the broader #12602 umbrella. The likely implementation belongs in `extensions/slack`: add a `MarkdownIR` to Slack `rich_text` renderer, decide opt-in versus default config, skip conversion when explicit Slack blocks or shared presentation blocks are present, and cover normal sends, slash replies/streaming, chunking, lists, quotes, links, inline styles, and fenced code blocks in Slack tests and docs.

## Risks / Open Questions

- A default rich_text conversion would touch Slack formatting, fallback text, streaming/slash replies, chunking, and explicit-block precedence, so it needs product direction and focused regression tests before landing.
- Slack rich_text limits and element shapes should be verified against Slack’s current API/types during implementation.
