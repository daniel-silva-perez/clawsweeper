## Summary

Current main still lacks the requested dedicated browser lane or per-channel/per-workload lane routing for inbound channel sessions. The low-level queue can run arbitrary named lanes when a caller supplies `lane`, but gateway config only wires built-in lane concurrency, docs still describe inbound runs as defaulting to process-wide `main`, and primary plus follow-up auto-reply paths still call `runEmbeddedPiAgent` without a lane. The newer issue comment reinforces the same workload-classified routing need rather than pointing to a shipped OpenClaw fix.

## Best Possible Solution

Keep this issue open. Implement an explicit custom-lane routing surface for inbound work: validate configured lane pools, apply their concurrency during gateway startup and reload, route channel/account/agent/workload-classified sessions into the selected lane before `runEmbeddedPiAgent` enters the global queue, and add focused coverage for both primary auto-reply and queued follow-up paths. Browser-tool auto-detection can be considered later, but pre-run routing is the core missing capability.
