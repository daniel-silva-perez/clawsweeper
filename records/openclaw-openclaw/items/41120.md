## Summary

Keep this issue open. Current main still has the low-level ability to enqueue arbitrary lane names, but inbound channel auto-reply runs still default into the process-wide main global lane and the config/docs only expose main, cron, cron-nested, nested, and subagent-style concurrency, not a dedicated browser lane or workload/per-channel lane routing.

## Best Possible Solution

Keep this issue open. The best implementation path is a core custom-lane routing surface: validate configured lane pools, apply their concurrency during gateway startup/reload, route inbound channel/account/agent or workload-classified sessions into the selected lane before runEmbeddedPiAgent enters the global queue, and cover both primary auto-reply and queued follow-up paths. Browser-tool auto-detection can be a follow-up, but explicit pre-run routing is the key missing capability.
