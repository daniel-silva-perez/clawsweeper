## Summary

Keep this issue open. Current main still lacks a configured browser/workload/per-channel lane-routing surface for inbound channel runs; the existing queue can accept arbitrary lane strings, but gateway startup only configures known lanes and the auto-reply paths still call the embedded runner without a lane, so they fall back to the process-wide main lane.

## Best Possible Solution

Keep this issue open. The best implementation path is a core lane-routing surface: add validated custom lane pools, apply them during gateway startup/reload, route inbound primary and follow-up runs into the selected lane before runEmbeddedPiAgent enqueues global work, and document/test the behavior. Browser-tool auto-detection can be a later enhancement, but explicit pre-run routing is the key missing capability.
