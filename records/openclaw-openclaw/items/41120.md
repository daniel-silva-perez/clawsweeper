## Summary

Current main still lacks a dedicated browser lane or per-channel lane routing for inbound channel sessions. The queue can preserve arbitrary lane strings only when a caller supplies one, but gateway startup config only applies the built-in main/cron/subagent lanes, and the primary auto-reply plus queued follow-up paths still call runEmbeddedPiAgent() without a lane. Browser-heavy channel turns therefore still share the process-wide main lane with quick replies.

## Best Possible Solution

Keep this issue open and implement an explicit scheduler/config path: add a validated custom-lane concurrency surface, apply it during gateway startup and reload, route inbound channel/account/agent runs to a chosen lane before runEmbeddedPiAgent enters the global queue, and cover primary auto-reply plus queued follow-up behavior in tests and docs. Browser-tool auto-detection can remain a later enhancement once explicit pre-run routing is in place.
