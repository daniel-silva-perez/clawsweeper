## Summary

Keep this issue open. Current main still lacks a configured browser/workload/per-channel lane-routing surface for inbound channel runs. The queue and gateway agent RPC can carry arbitrary lane strings, but gateway startup only configures fixed built-in lanes and the auto-reply plus follow-up paths still call the embedded runner without a lane, so browser-heavy inbound sessions continue to compete for the process-wide main lane.

## Best Possible Solution

Keep this open for a core lane-routing design. The most concrete path is to add validated custom lane pools, apply them during gateway startup and reload, route inbound primary and follow-up runs into the selected lane before runEmbeddedPiAgent enqueues global work, and document/test the behavior. Browser-tool auto-detection can follow later; explicit pre-run routing is the missing capability.
