## Summary

Current main still lacks the requested dedicated browser lane or per-channel lane routing for inbound channel sessions. The queue can run arbitrary named lanes when a caller supplies `lane`, but gateway config only wires built-in lanes and the primary inbound auto-reply plus follow-up paths still enter `runEmbeddedPiAgent` without a lane, so browser-heavy channel turns continue to share the process-wide main lane with quick replies.

## Best Possible Solution

Keep this issue open. Implement an explicit custom-lane scheduling surface: validate configured custom lanes, apply their concurrency during gateway startup and reload, route inbound channel/account/agent sessions to a selected lane before `runEmbeddedPiAgent` enters the global queue, and add coverage for both primary auto-reply and queued follow-up runs. Browser-tool auto-detection can be a later enhancement once pre-run routing exists.

## Risks / Open Questions

- Closing this now would drop a still-missing scheduler/config capability for separating long browser-heavy inbound channel turns from quick text replies.
- The eventual fix needs maintainer design for lane naming, validation, reload behavior, fairness, provider rate limits, and whether routing belongs at channel, account, agent, or explicit browser-workflow scope.
- Browser-tool auto-detection after a run starts would not protect the first long browser turn; explicit pre-run routing should be considered the primary path.
