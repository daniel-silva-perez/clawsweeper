## Summary

Keep this issue open. Current main still has the same core gap: inbound channel auto-reply and follow-up runs default into the process-wide main lane, while gateway startup only configures fixed built-in lanes. The arbitrary lane plumbing exists for explicit agent/plugin runs, but there is still no validated browser/workload/per-channel lane config surface for inbound channel routing.

## Best Possible Solution

Keep this issue open for a core lane-routing design. Add a validated custom lane-pool config surface, apply it during gateway startup and config reload, route inbound primary and follow-up runs into the selected lane before runEmbeddedPiAgent enqueues global work, and document/test precedence across channel, account, binding, agent, and workload classification. Browser-tool auto-detection can follow once explicit pre-run routing exists.
