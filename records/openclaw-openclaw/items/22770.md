## Summary

Current main still lacks the requested $VISUAL/$EDITOR external-editor compose mode for the TUI. The TUI has inline editor shortcuts, slash commands, selectors, local shell commands, and a suspended /auth flow, but no keybinding or command that writes the current composer text to a temp markdown file, launches an external editor, reloads edited content with editor.setText, or documents/tests that behavior. The related #55836 remains adjacent vi/vim in-place editing work, not a duplicate or shipped substitute.

## Best Possible Solution

Keep this issue open and implement the feature in the core TUI composer: add a deliberate shortcut or command, suspend the TUI, write editor.getExpandedText() to a temporary markdown prompt file, launch $VISUAL then $EDITOR and wait, load the edited file back with editor.setText without auto-submitting, show clear missing-editor/non-zero-exit/file-I/O status messages, document it in the public TUI guide, and add focused TUI/editor tests. Keep #55836 separate for in-place vi/vim input mode.

## Clownfish Work Prompt

Implement external editor compose mode for the OpenClaw TUI. The observable request is: while editing a TUI prompt, the user can invoke an external editor using $VISUAL, falling back to $EDITOR, edit a temporary markdown prompt file prefilled with the current composer text, exit the editor, and have the edited contents loaded back into the composer without auto-submitting. Keep #55836 out of scope; do not implement vi/vim in-place input mode.

Expected boundary: core TUI only. Likely touch `src/tui/components/custom-editor.ts`, `src/tui/tui.ts`, and preferably a small helper such as `src/tui/tui-external-editor.ts` for resolving editor env, temp file handling, child-process execution, and result mapping. Use existing TUI suspension patterns from `withTuiSuspended`. Use `editor.getExpandedText()` for the current content and `editor.setText()` for the returned content. Show clear status/system messages for missing editor env, editor spawn errors, non-zero exit/signal, temp file write/read failures, and successful load. Do not auto-send the message. Decide the shortcut deliberately because pi-tui currently maps Ctrl+E to cursor line end; if Ctrl+E is used, document the override and add tests that prove it routes to the handoff.

Security/safety constraints: create the temp prompt file under a fresh temp directory with safe permissions where feasible, clean it up after the handoff, avoid surprising shell execution, and handle editor env values with arguments conservatively. Do not change provider/channel/plugin behavior, gateway protocol, or general command execution policy.

Docs/changelog: update the public TUI guide (`docs/web/tui.md`) and CLI TUI reference if useful. Add a user-facing changelog entry under the active version with `Thanks @PartyPython` if the repo's changelog policy applies.

Validation: add focused unit tests for editor shortcut routing and helper behavior, plus a runTui-level or command-handler test where practical. Run targeted TUI tests and formatting, then run the changed gate in Testbox per repo policy before handoff.
