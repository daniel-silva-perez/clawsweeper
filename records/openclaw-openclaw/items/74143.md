## Summary

Keep open. Current main still has the Feishu streaming-card overlap bug: active card-mode block payloads are appended through the delta path, while partial replies use the snapshot merge path, and the current regression-shaped test still expects duplicated output. Open PR #74146 is already the focused closing-syntax implementation candidate, so this issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

Fix Feishu card-mode streaming so overlapping `deliver({ kind: "block" })` payload text merges with partial snapshots instead of being concatenated into duplicate card content.

## Best Possible Solution

Land a narrow Feishu plugin fix that routes card-mode block payloads through snapshot-style overlap merging, or an equivalent merge, while preserving block-only fallback delivery and exact-duplicate suppression. The tests should update the `hello` plus `lo world` case to close as `hello world` and add coverage for longer overlapping partial-snapshot/block sequences.

## Reproduction Assessment

Yes. The current unit-test harness already reproduces the problematic path by calling `onPartialReply` with `hello`, then delivering block text `lo world`, and current main expects the duplicated close text `hellolo world`.

## Solution Assessment

Yes. The narrowest maintainable solution is in the Feishu dispatcher and tests: make overlapping block payloads use the same overlap-aware merge semantics as partial snapshots, then validate that block-only fallback behavior still works.
