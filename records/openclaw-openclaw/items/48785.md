## Summary

Keep open. Current main still implements `tools.sessions.visibility` as a single string scope shared by `sessions_list`, `sessions_history`, `sessions_send`, and `session_status`; I found no structured per-capability object form for send/history/list/metadata. The related open #55420 is adjacent allowlist work, but it does not supersede this request because it scopes which sessions are visible rather than splitting capabilities so send can be allowed without transcript reads.

## Best Possible Solution

Keep this open as a core security/config enhancement. The best implementation path is a backwards-compatible `tools.sessions.visibility` object or equivalent capability policy that lets maintainers grant `sessions_send` independently from `sessions_history` and `sessions_list`, with clear interaction rules for `tools.agentToAgent` and sandbox clamping.
