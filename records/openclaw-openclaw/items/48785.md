## Summary

Keep open. Current main still exposes `tools.sessions.visibility` as one four-value string scope and routes send, history, list, status, and memory-session filtering through the same visibility guard. The related open allowlist issue and A2A-send PR are adjacent, but neither implements or supersedes the requested per-capability object model.

## What This Changes

Requests an additive, backward-compatible object form for `tools.sessions.visibility` with separate send, history, list, and metadata permissions.

## Best Possible Solution

Define one coherent session-tool capability model: preserve the existing string shorthands, add a validated object form for separate send/history/list/metadata permissions, keep sandbox and A2A restrictions fail-closed, update generated schema/help/docs, and add tests proving cross-agent send can be allowed without transcript history or broad listing.

## Reproduction Assessment

Not applicable as a feature/config request. The source check is high-confidence: current main still has a string-only schema and a single shared guard for the affected actions.

## Solution Assessment

Unclear. The requested direction is plausible and addresses the privacy gap, but the best final shape should be decided alongside A2A policy and allowlist work because it changes public authorization semantics.
