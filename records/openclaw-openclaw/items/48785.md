## Summary

Keep open. Current main still exposes `tools.sessions.visibility` as one string-valued scope and applies that single scope across send, history, list, and status access; the requested per-capability object form is not present in source, schema, tests, or docs. The request is valid, but it is a security-sensitive public config and permission-model design, so it should get maintainer review before an implementation PR is queued.

## What This Changes

The issue requests a backward-compatible `tools.sessions.visibility` object form that can independently control session send, history, list, and metadata visibility instead of using one shared visibility enum.

## Best Possible Solution

Define one backward-compatible session-tool capability model: keep the existing string shorthands, add a validated object form for separate send/history/list/metadata permissions, preserve sandbox and A2A restrictions, update schema/types/generated help/docs, and add tests proving cross-agent send can be allowed without exposing transcript history or broad listing.
