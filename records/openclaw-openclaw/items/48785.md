## Summary

Keep open. Current main still models `tools.sessions.visibility` as one string scope and applies that single scope across send, history, list, and status access; the requested per-capability object form is not implemented. The adjacent allowlist issue and A2A-send PR are related, but they do not supersede this broader permission-model request.

## What This Changes

The issue requests a backward-compatible object form for `tools.sessions.visibility` with separate send, history, list, and metadata controls instead of one all-purpose string scope.

## Best Possible Solution

Define one coherent session-tool capability model: keep existing string shorthands, add a validated object form for separate send/history/list/metadata permissions, preserve sandbox and A2A restrictions, update schema/help/docs, and add focused tests proving cross-agent send can be allowed without transcript history or broad listing.
