## Summary

Keep this issue open. Current main still has no sibling-agent roster in the session system prompt, and the existing `agents_list` tool is scoped to `sessions_spawn` allowlists rather than the requested peer collaboration surface with `identity.name`, `identity.theme`, and computed `agentToAgent` reachability.

## Best Possible Solution

Keep this issue open and implement a focused agent-facing roster surface. The conservative path is to extend `agents_list` or add a dedicated discoverable tool that returns peer `id`, `identity.name`, `identity.theme`, and an explicit `agentToAgentAllowed` value computed for the requester, filtered by current session/tool/sandbox policy. Update the system prompt to direct agents to that live roster instead of embedding a static list, then add tests for payload shape and policy filtering plus docs in the subagents or multi-agent pages.

## Risks / Open Questions

- A fix needs to avoid leaking agent metadata beyond what the requester can use; `subagents.allowAgents`, `tools.sessions.visibility`, sandbox clamping, and `tools.agentToAgent` are separate policy gates today.
- Injecting a full roster directly into every system prompt would add prompt-cache churn and can still become stale during long sessions; a live discoverable tool with prompt guidance may be safer.
- Broadening the existing `agents_list` response should preserve the current `sessions_spawn` discovery contract or version/add fields in a backwards-compatible way.
