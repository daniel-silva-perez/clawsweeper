## Summary

Keep this issue open. Current main has an `agents_list` tool and an `agents.list` Gateway RPC, but neither implements the requested sibling-agent awareness in the session prompt or a discoverable agent-facing roster with `identity.name`, `identity.theme`, and per-peer `agentToAgent` reachability. The KeepMyClaw comment is useful context about external snapshots/state divergence, but it is not a core implementation of this requested runtime surface.

## Best Possible Solution

Keep the issue open and implement a focused agent-facing roster surface. The conservative path is to extend or add a discoverable tool that returns configured peer ids, `identity.name`, `identity.theme`, and an explicit `agentToAgentAllowed` value computed for the requester, then update the system prompt to direct agents to that tool instead of embedding a potentially stale roster on every run. Add tests for the tool payload and policy filtering, plus docs in the multi-agent/subagents pages.

## Risks / Open Questions

- A fix should avoid exposing more cross-agent metadata than the requester is allowed to use; `tools.agentToAgent`, session visibility, sandbox restrictions, and subagent allowlists are separate policies today.
- Injecting a full roster directly into the system prompt may create prompt-cache churn when config changes; a live `agents_list` expansion may be the safer design if it includes the missing identity/theme/reachability fields.
