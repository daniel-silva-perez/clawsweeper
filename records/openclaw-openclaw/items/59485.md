## Summary

Keep this PR open, but do not merge it as-is. The branch targets a Paperclip env-plumbing problem, but the proposed patch does not add the fields to the current runner/exec contracts, does not forward them through the embedded attempt tool-construction path, and does not make the exec tool consume the new default.

## What This Changes

The PR adds Paperclip runtime auth env helpers and tests, attempts to pass `paperclipRuntimeEnv` from `runEmbeddedPiAgent` through `createOpenClawCodingTools` into exec defaults, and refactors several fetch test casts into a shared helper.

## Best Possible Solution

If Paperclip runtime auth is still a supported product path, the fix should be rebuilt against current main by adding an explicit typed runner-to-attempt-to-tool contract and merging the env at the exec invocation boundary. If the provider runtime auth redesign supersedes Paperclip env injection, the PR should be closed by a maintainer as stale/superseded with a link to the replacement design.

## Reproduction Assessment

Yes. Static inspection of the PR diff against current main is enough to reproduce the failure path: the PR uses `paperclipRuntimeAuth`/`paperclipRuntimeEnv` fields that are absent from current params/default types, and current exec env construction never consumes the proposed default.

## Solution Assessment

No. This is not the best current fix because it adds Paperclip-specific fields at only part of the pipeline; a maintainable fix would either use the current provider/harness runtime auth mechanism or add a fully typed, end-to-end exec env contract with tests at the attempt boundary.
