## Summary

Current main already solves the underlying runtime-auth handoff through the provider runtime/authStorage path instead of the PR's Paperclip-specific `PAPERCLIP_*` env plumbing, and that implementation is present in the shipped `v2026.4.27` tag. The proposed branch is also not a safe merge candidate because its added Paperclip fields are not part of the current runner parameter contract and the env value still would not reach `createOpenClawCodingTools` in the embedded-attempt path.

## What This Changes

The PR branch adds Paperclip runtime-auth env construction and attempts to pass `paperclipRuntimeEnv` through `createOpenClawCodingTools`/exec defaults, with regression tests and shared fetch-test helper cleanup.

## Best Possible Solution

Keep the shipped provider runtime/authStorage implementation and close this PR without merging the Paperclip-specific env path. Any remaining Paperclip-auth problem should be reopened with a current-main reproduction against the provider runtime hook or auth profile path, not by reviving process/env plumbing.

Not applicable as a user-facing repro; this is a stale PR review. The current-main verification is high-confidence source inspection plus shipped-tag inspection showing the old Paperclip env surface is gone and runtime auth now flows through authStorage.

## Solution Assessment

No, this PR branch is not the best way to solve the issue now. Current main's provider-owned runtime auth path is narrower and safer, while the branch still has type/forwarding defects and preserves an obsolete Paperclip-specific env contract.
