## Summary

Keep this open. Current main has additional mid-turn precheck work, but it still pins an upstream `SessionManager` implementation that whole-file reads/parses JSONL in `loadEntriesFromFile()`, and OpenClaw still reaches `SessionManager.open()` in embedded and compaction paths before any protection can shrink a very large transcript.

## What This Changes

The issue reports that large embedded session JSONL transcripts can crash the gateway because `SessionManager.open()` performs duplicate whole-file reads/parses before context rebuild.

## Best Possible Solution

Fix the embedded Pi/session boundary so opening a transcript either reads only the header once or reuses already parsed entries, then add focused stress coverage for a large JSONL with oversized tool results proving the gateway returns a controlled error or successfully recovers instead of exiting.

## Reproduction Assessment

Unclear. Source inspection gives a high-confidence reproduction of the duplicate whole-file parse path in the pinned dependency and OpenClaw call sites, but this read-only review did not run the reporter's 80 MB transcript or a synthetic gateway crash fixture.

## Solution Assessment

Unclear as a final implementation path. The issue identifies the right failure boundary, but current mid-turn precheck and byte-threshold compaction are mitigations, not the narrow durable fix for the initial `SessionManager.open()` double-load behavior.
