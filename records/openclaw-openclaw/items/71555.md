## Summary

Keep this open. Current main moved the Pi packages to 0.70.6, but the latest upstream `@mariozechner/pi-coding-agent` still reads and parses the full JSONL in `loadEntriesFromFile()`, and `SessionManager.open()` still calls that before constructing a `SessionManager` that loads the same file again. OpenClaw still reaches that path in embedded runs before the context rebuild and before any mitigation can shrink the initial large transcript.

## What This Changes

The issue reports that large embedded session JSONL transcripts can crash the gateway because `SessionManager.open()` performs duplicate whole-file parsing before prompt/context rebuild.

## Best Possible Solution

Fix this at the embedded Pi/session boundary so opening a transcript reads only the header once or reuses the parsed entries, then add focused stress coverage for a large JSONL with oversized tool results proving the gateway returns a controlled result instead of exiting. The byte-threshold compaction path should remain a mitigation, not the only protection before the first large-session open.

## Reproduction Assessment

Unclear. The duplicate full-file parse is reproducible by source inspection of the current pinned dependency and OpenClaw call path, but this review did not have the reporter's 80 MB transcript or a synthetic crash fixture to reproduce the gateway exit end to end.

## Solution Assessment

Unclear as a final implementation path. The issue points at the right failure boundary, but the best fix is likely an upstream Pi change or explicitly approved dependency patch/adapter rather than relying on `maxActiveTranscriptBytes`, which still cannot avoid the initial open path.
