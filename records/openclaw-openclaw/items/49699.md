## Summary

Keep this PR open. Current main still uses the legacy device-token contracts (`DeviceAuthToken | null` and `{ ok, reason }` deny results), and the PR’s intended structured error-object migration has not landed. The PR remains potentially useful, but its head needs focused rework before it is mergeable.

## Best Possible Solution

Keep the PR open and ask for a focused rework: preserve the existing `ok`/`reason` and token-or-null contracts or intentionally version the migration, add structured details only where callers can consume them safely, keep public rotate/revoke denial messages generic while logging detailed context server-side, fix the syntax errors, and update focused infra/gateway tests for scope escalation, missing device/role, and revoke-denial cases.

## Risks / Open Questions

- The PR head, as provided, is not safe to land until the malformed gateway TypeScript is fixed.
- Changing `ensureDeviceToken`, `rotateDeviceToken`, and `revokeDeviceToken` from `null` / `ok` / `reason` contracts to standalone error objects is a breaking migration unless compatibility or versioning is handled deliberately.
- Detailed revoke errors must not become public gateway responses because they can create a resource-enumeration side channel; detailed reasons should stay in server logs.
