## Summary

Keep this PR open. Current main still uses the legacy device-token contracts, and the PR head is not mergeable because it introduces malformed TypeScript, changes established token-result discriminators, and exposes granular revoke-denial messages to clients.

## What This Changes

This PR changes device pairing token issuance, rotation, and revocation APIs plus gateway callers/tests to return and handle structured error objects for scope escalation and missing device/token cases instead of null or legacy ok/reason denial results.

## Best Possible Solution

Keep the existing fail-closed token behavior, and if structured diagnostics are desired, add them in a backwards-compatible way: preserve the current ok/reason and token-or-null public contracts or deliberately version them, keep client-facing rotate/revoke denials generic, and put detailed scope/device context in server-side logs plus focused infra/gateway tests.

## Reproduction Assessment

Yes. The PR diff itself is a high-confidence reproduction path: the changed gateway files contain invalid TypeScript syntax, and current-main tests/source show the legacy null and ok/reason contracts the PR is trying to replace.

## Solution Assessment

No. A standalone error-object migration is not the narrowest maintainable fix here; the safer direction is to preserve existing discriminators or add structured details alongside them while keeping public denial messages stable.
