## Summary

Current main and the shipped npm package already fix the reported packaged memory-lancedb startup failure. The failing v2026.3.28 package had an inlined single `./package.json` read, but the current runtime keeps the dynamic manifest-candidate resolver and the published package includes the extension manifest while still omitting `dist/package.json`.

## What This Changes

This PR reorders memory-lancedb manifest candidates so the bundled extension `package.json` is checked before `dist/package.json` and adds a regression test for the missing-`dist/package.json` npm layout.

## Best Possible Solution

Keep the shipped dynamic resolver and package layout; this PR can close because the packaged startup failure is already fixed without needing the candidate-order-only workaround.

## Reproduction Assessment

Yes. The v2026.3.28 npm tarball reproduces the reported failure shape with no `dist/package.json` and an inlined `./package.json` read, while v2026.3.31 and v2026.4.27 preserve the dynamic resolver against the same missing-`dist/package.json` layout.

## Solution Assessment

Yes. The current implementation is the better maintained fix because it prevents a single manifest read from being baked into the bundle and keeps source, flat dist, and nested dist candidate layouts covered.
