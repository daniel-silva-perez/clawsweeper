## Summary

Keep this issue open. Current main still treats `exec` as a separate runtime tool from the filesystem tools: denying `write`/`edit` blocks those named tools and `apply_patch`, but it does not block an allowed `exec` tool from running shell commands that can write to the accessible filesystem. The docs also still show a “Safe execution (no file modifications)” profile that allows `exec` while denying `write`/`edit`/`apply_patch`, so the reported user problem is still actionable as either a product hardening change or a docs correction.

## Best Possible Solution

Keep the issue open and either implement a first-class read-only exec mode that enforces filesystem side-effect boundaries, or revise the docs and examples to state plainly that denying `write`/`edit`/`apply_patch` does not constrain shell side effects when `exec` remains allowed. The misleading “Safe execution (no file modifications)” example should be corrected either way.

## Risks / Open Questions

- Security-sensitive read-only agent configurations can still be misconfigured by allowing `exec` while denying file tools, especially because current docs include a profile that implies this prevents file modification.
- A pure command-content inspection fix would be brittle across shells and platforms; a safer direction is an explicit exec policy mode, sandbox/workspace access enforcement, or clear documentation that `exec` is a write-capable/RCE surface.
