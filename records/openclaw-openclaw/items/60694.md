## Summary

Keep this issue open. Current main still treats `exec` as a separate runtime tool from `write`/`edit`/`apply_patch`: denying filesystem tools removes those named tools, but it does not make an allowed shell command read-only. The docs still include examples that can be read as safe/read-only while allowing `exec`, so the reported security/product gap remains actionable.

## Best Possible Solution

Keep the issue open. The best maintainer path is either to add a first-class read-only exec/sandbox mode that enforces filesystem side-effect boundaries, or to revise the sandbox/tool-policy docs so they state plainly that allowing `exec` remains a write-capable command-execution surface even when `write`, `edit`, and `apply_patch` are denied. The current `Safe execution (no file modifications)` example should be corrected either way.

## Risks / Open Questions

- Users can still configure an agent that looks read-only by denying filesystem tools while leaving shell execution able to write within its accessible execution environment.
- A command-string inspection fix would be brittle across shells and platforms; this likely needs an explicit exec policy/sandbox boundary or clearer docs that `exec` is write-capable when allowed.
