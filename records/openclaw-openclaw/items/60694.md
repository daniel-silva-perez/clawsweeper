## Summary

Keep this open. Current main still treats tool policy as named-tool gating: denying `write`/`edit`/`apply_patch` does not deny an allowed `exec` tool, and the docs still include a “Safe execution (no file modifications)” profile that allows `exec`, so the reported security concern remains actionable.

## What This Changes

The issue asks OpenClaw to prevent or clearly document `exec`-based filesystem writes when security executor agents deny filesystem write/edit tools.

## Best Possible Solution

Define the read-only agent contract explicitly: either add an enforceable read-only exec/sandbox mode for filesystem side effects, or correct the docs/examples so `exec` is clearly treated as write-capable and must be denied for no-file-modification profiles.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: use the documented allow/deny profile, observe that `exec` remains allowed, and the exec implementation runs the supplied shell command without mapping filesystem side effects back to `write`/`edit` denial.

## Solution Assessment

Unclear. The report is valid, but the best fix should be chosen by maintainers because command-string deny lists are likely weaker than an explicit exec policy mode, sandbox filesystem enforcement, and documentation changes.
