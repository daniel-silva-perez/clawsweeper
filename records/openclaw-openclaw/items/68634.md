## Summary

Keep open. Current main appears to fix the Telegram Native Approvals retry/log loop portion, but the CLI scope-upgrade and persistent pending-device behavior is still present by design/code path: CLI-mode calls still request the full default operator scope bundle, while the gateway still treats broader requested scopes than the paired token baseline as a `scope-upgrade` requiring a new pending approval.

## Best Possible Solution

Keep this issue open and scope the remaining fix to CLI connect/auth behavior. The likely maintainer path is to make routine CLI commands use least-privilege scopes or reuse the approved stored-device-token scope baseline when appropriate, while preserving explicit approval for genuine role/scope upgrades and adding regression coverage for `openclaw gateway status`, `openclaw devices list`, and other read-only CLI calls after bootstrap-issued operator tokens.
