## Summary

Keep open. Current main has partial Discord safeguards, but not the requested bot-to-bot loop detector. `allowBots=true` still maps to accepting bot-authored Discord messages, including mention-gated bot messages, and I did not find a documented or implemented `(senderBotId, receiverBotId, channelId)` sliding-window/cooldown guard. The related teleconference RFC treats bot-loop detection as a future dependency rather than a superseding implementation.

## Best Possible Solution

Keep this issue open and implement the guard in the Discord channel inbound path, or in a shared channel-inbound helper if multiple bot-enabled channels need it. The concrete path should add a documented `botLoopProtection` policy, track bot-pair traffic by sender, receiver/mentioned bot, and channel over a short window, suppress during cooldown before dispatching an agent turn, and add focused Discord tests for `allowBots=true` plus mention-gated bot messages.

## Risks / Open Questions

- Operators who need `allowBots=true` for multi-agent Discord setups can still create runaway bot-to-bot exchanges when bots mention each other across distinct messages.
- The current mitigations are configuration guidance (`allowBots=false` by default, `allowBots="mentions"`, allowlists) rather than automatic suppression after repeated bot-pair traffic.
