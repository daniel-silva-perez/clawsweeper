## Summary

Keep open. Current main has Discord bot-message safeguards and documentation warnings, but it still lacks the requested bot-pair sliding-window/cooldown guard for `allowBots=true`; the related teleconference RFC treats this as a dependency rather than a superseding implementation, and the follow-up production comment corroborates the runaway-loop impact.

## What This Changes

Request a Discord gateway-level bot-to-bot loop detector for `allowBots=true`, with per sender/receiver/channel rate limits and cooldown suppression before agent dispatch.

## Best Possible Solution

Add a Discord-owned `botLoopProtection` policy and deterministic bot-pair/window/cooldown guard in the inbound path before an agent turn is dispatched, with docs, changelog, schema/help updates, and focused tests for `allowBots=true` plus mention-gated bot messages. Keep embedding or progress-scoring detection as a later enhancement unless maintainers explicitly choose that broader design.

## Reproduction Assessment

Yes. A focused preflight/handler test can configure Discord with `allowBots=true`, send alternating bot-authored messages that mention the gateway bot, and observe that current main accepts them because no pair/window/cooldown state is consulted.

## Solution Assessment

Yes for the narrow shape. A deterministic Discord inbound guard keyed by sender bot, mentioned/receiver bot, and channel is the most maintainable first step; semantic/progress scoring should not be required for the initial fix.
