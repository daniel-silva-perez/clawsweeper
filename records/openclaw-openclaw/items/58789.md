## Summary

Keep open. Current main still lacks the requested Discord bot-to-bot loop detector: `allowBots: true` still accepts bot-authored Discord messages except the gateway's own user, the Discord config surface has no `botLoopProtection`/pair-rate/cooldown fields, and the existing safeguards cover self messages, webhook echoes, ACP self-loop system messages, inbound replay dedupe, tool-call loops, or `sessions_send` ping-pong rather than alternating Discord bot mentions across distinct messages. The supplied discussion also includes a recent production confirmation and design notes, so this is not stale or superseded by the related teleconference RFC.

## Best Possible Solution

Keep this issue open and implement the guard in the Discord plugin's inbound path, likely around `extensions/discord/src/monitor/message-handler.preflight.ts` before enqueueing an agent turn. Add a documented `botLoopProtection` policy to the Discord config contract, track bot-authored traffic by sender bot, mentioned/receiver bot, and channel over a short sliding window, suppress during a cooldown with an operator-visible warning, and add focused Discord tests for `allowBots=true` and `allowBots="mentions"` bot-to-bot mention loops. If maintainers want this generalized later, extract a shared channel-inbound helper after the Discord behavior is proven.

## Risks / Open Questions

- Operators using `channels.discord.allowBots=true` for multi-agent Discord setups can still trigger runaway alternating bot replies until external rate limits, manual allowlists, or mention settings intervene.
- A future implementation should keep the first fix cheap and deterministic; semantic embedding or progress-based detection may be useful later but would add cost, privacy, and false-positive tradeoffs if enabled by default.
