## Summary

Keep open. Current main still lacks the intended descriptive `Avatar:` parsing, and this PR is a useful narrow fix direction, but the proposed extractor is not merge-ready because it can truncate valid direct avatar paths containing spaced hyphens or en/em dashes.

## What This Changes

The PR updates `parseIdentityMarkdown` to extract the rightmost URL, data URI, or path-like value from descriptive avatar lines and adds parser regression coverage for a descriptive `Avatar:` line ending in a workspace-relative path.

## Best Possible Solution

Revise the helper so direct HTTP URLs, data URIs, and workspace-relative avatar paths are preserved unchanged first, then extract a trailing avatar only when the value is clearly descriptive text followed by a valid avatar source. Add focused tests for both the descriptive format and direct filenames like `avatars/hawk - final.png` and `avatars/hawk—final.png`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve direct avatar paths with dashes:** `src/agents/identity-file.ts:54`
  - body: The separator scan runs for every avatar value, including values that are already direct paths. For `avatars/hawk - final.png` or `avatars/hawk—final.png`, the helper can split the valid path and return `final.png` because that tail still looks like an avatar path, so existing workspace-relative avatars resolve to the wrong source. Accept direct URL/data/path values unchanged before attempting descriptive extraction, and add regressions for those direct-path cases.
  - confidence: 0.92
