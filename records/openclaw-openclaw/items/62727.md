## Summary

Keep open. Current main still lacks the descriptive avatar extraction this PR is trying to add, so the work is not obsolete, but the proposed separator-based parser has a blocking regression for valid direct avatar paths containing dash-like characters.

## What This Changes

The PR updates `parseIdentityMarkdown` to extract a trailing avatar path, URL, or data URI from descriptive `Avatar:` lines and adds parser regression coverage for a trailing local avatar path.

## Best Possible Solution

Revise the parser so it first preserves direct URL, data URI, and avatar-path values unchanged, then extracts a trailing avatar only when the value is clearly descriptive text followed by a valid avatar source. Add focused parser tests for the descriptive format and dash-containing filenames before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve direct avatar paths with dashes:** `src/agents/identity-file.ts:54`
  - body: The new separator scan runs for every avatar value, including values that are already direct paths. For `- Avatar: avatars/hawk - final.png` or `avatars/hawk—final.png`, the helper splits the valid path and returns `final.png` because that tail still looks like an avatar path, so existing workspace-relative avatars with dash-like filename characters stop resolving to the intended file. Gate extraction on a clear descriptive prefix or accept direct URL/path values unchanged first, and add regressions for those direct-path cases.
  - confidence: 0.91
