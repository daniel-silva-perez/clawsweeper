## Summary

Keep open. Current main still has the browser CLI/docs mismatch: `--browser-profile` is only registered on the parent `browser` command, Commander positional options are enabled, and the WSL2 guide still shows the option after `open` and `tabs`. The gateway/systemd half has later service-audit mitigation for version-managed Node paths, but the open user-facing browser regression is still enough to keep this issue active.

## What This Changes

The issue reports a regression where `--browser-profile` no longer works after `browser tabs/open` despite WSL2 docs using that syntax, plus a `doctor --fix` gateway restart loop caused by a systemd unit pinned to an nvm Node path.

## Best Possible Solution

Restore backward-compatible `--browser-profile` handling for documented browser leaf commands such as `tabs` and `open`, keep the current parent-command syntax working, and add focused CLI tests. Treat the gateway half as already mitigated unless a maintainer can reproduce a remaining WSL2+nvm failed-unit loop on current main, in which case add a narrow doctor/service-audit test before changing behavior.

## Reproduction Assessment

Partially: yes for the browser regression, based on current command definitions, Commander positional-option behavior, and the WSL2 guide examples. Unclear for the gateway regression because current main now audits version-managed Node service paths, but I did not find live WSL2 systemd proof for the exact failed-unit cycling case.

## Solution Assessment

Yes for the browser half: a small compatibility option/normalization on the affected leaf commands is the safest user-facing fix because existing docs and scripts used that placement. For the gateway half, the current service-audit direction appears appropriate, and further changes should be driven by a focused current-main reproduction rather than a broad doctor rewrite.
