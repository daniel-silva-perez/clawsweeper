## Summary

Keep open. Current main still runs WhatsApp pairing access control before proving the Baileys upsert contains extractable inbound user content, and open PR #73823 explicitly references this issue with closing syntax. Per cleanup policy, the issue should remain open until that PR is reviewed and either merged or closed.

## Best Possible Solution

Keep this issue open and route maintainer attention to PR #73823. The best fix is in the WhatsApp plugin: add a pre-access-control gate for extractable inbound user content, preserve explicit interactive replies as user content, and add regression tests for protocol/receipt/no-content messages under dmPolicy pairing.
