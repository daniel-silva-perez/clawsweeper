## Summary

Keep open. Current main still uses the kickstart-based macOS LaunchAgent restart path before any explicit bootout/PID-wait/start sequence, and the focused tests preserve that healthy-path behavior. The adjacent merged supervision recovery work references this issue but only covers installed-but-unloaded LaunchAgent recovery, not the requested restart ordering change.

## What This Changes

Request to refactor macOS `restartLaunchAgent` into a two-phase stop/wait/start sequence so the old LaunchAgent process exits before `bootstrap`/`kickstart` starts the replacement.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract. If accepted, implement the stop/wait/start ordering in the launchd owner module while preserving managed restart UX, detached self-restart, installed-but-unloaded recovery, disabled-state recovery, and stale listener cleanup.

## Reproduction Assessment

Unclear for a high-confidence live race reproduction because the failure depends on macOS launchd timing. The acceptance gap is reproducible by inspection on current main: restart and handoff paths still use `kickstart -k` before any explicit old-service-exit-before-start sequence.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the reported race, but the maintainable route needs launchd owner approval plus unit and live macOS validation because it changes an intentionally kickstart-based restart contract.
