## Summary

Keep open. Current main still uses the macOS LaunchAgent `kickstart -k` restart contract before any explicit stop/wait/start sequence, and the focused unit tests still assert that the healthy restart path does not boot out or bootstrap the service. The adjacent shipped LaunchAgent recovery work references this issue but covers installed-but-unloaded supervision recovery, not the requested two-phase ordering change.

## What This Changes

Request a macOS LaunchAgent restart refactor so `restartLaunchAgent` stops the old service, waits for the prior process to exit, and only then bootstraps/kickstarts the replacement.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract. If accepted, implement the two-phase ordering in the launchd owner module and handoff path while preserving managed restart UX, disabled-service recovery, unloaded-service recovery, stale listener cleanup, and documented `gateway restart` semantics.

## Reproduction Assessment

Unclear for a high-confidence live race reproduction because the reported failure depends on macOS launchd timing. The acceptance gap is reproducible by inspection on current main: normal restart and detached handoff paths still start with `kickstart -k` rather than a stop/wait/start sequence.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but it changes launchd lifecycle semantics, so the maintainable solution needs owner approval plus focused unit and live macOS validation.
