## Summary

Keep open. Current main still preserves the kickstart-based macOS LaunchAgent restart contract, the focused tests still assert no healthy-path bootout/bootstrap sequence, and the related merged recovery work only re-registers unloaded services rather than implementing the requested old-process-exit-before-start ordering.

## What This Changes

Request that macOS `restartLaunchAgent` switch to a two-phase bootout/wait-then-bootstrap/kickstart sequence so the old LaunchAgent process exits before launchd starts a replacement.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract. If the two-phase direction is accepted, implement it in the launchd owner module with tests proving old-service exit-before-start ordering while preserving detached self-restart, installed-but-unloaded recovery, disabled-state recovery, and stale listener cleanup.

## Reproduction Assessment

Unclear for a live race reproduction because it depends on macOS launchd timing. The acceptance gap is reproducible by inspection on current main: the normal restart path and tests still show `kickstart -k` before any bootout/PID-wait/start sequence.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but it changes an intentional kickstart-based launchd contract, so the safer path is maintainer-approved design plus focused unit and live macOS proof.
