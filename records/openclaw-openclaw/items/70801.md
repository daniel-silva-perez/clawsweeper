## Summary

Keep open. Current main still uses the existing kickstart-based macOS LaunchAgent restart path, and the tests continue to assert the absence of a normal-path bootout/bootstrap sequence. The related merged supervision-recovery work references this issue but does not implement the requested stop/wait/start ordering.

## What This Changes

Request that macOS `restartLaunchAgent` switch to a two-phase bootout/wait-then-bootstrap/kickstart sequence so the old LaunchAgent process exits before launchd starts a replacement.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract. If the two-phase direction is accepted, implement it in the launchd owner module with tests proving old-service exit-before-start ordering while preserving detached self-restart, installed-but-unloaded recovery, and disabled-state recovery.

## Reproduction Assessment

Unclear for the live race itself because it requires macOS launchd timing. The acceptance failure is reproducible by inspection on current main: the normal restart path and tests still show `kickstart -k` before any bootout/PID-wait/start sequence.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but it reverses an intentional kickstart-based launchd contract, so the safer path is a maintainer-approved design plus focused unit and live macOS proof.
