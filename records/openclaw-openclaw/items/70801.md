## Summary

Keep open. Current main still preserves the kickstart-first macOS LaunchAgent restart contract, including a focused test that asserts no healthy-path bootout/bootstrap, while the merged adjacent recovery work only covers installed-but-unloaded supervision states and explicitly references this issue without fixing its requested ordering change.

## What This Changes

The issue requests changing macOS LaunchAgent restart from a kickstart-first flow to an explicit stop, wait-for-old-process-exit, then bootstrap/start sequence.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract. If accepted, implement the ordering change inside the launchd owner module with focused unit coverage and live macOS proof for normal restart, managed self-restart, installed-but-unloaded recovery, disabled-state recovery, and stale listener cleanup.

Unclear for a high-confidence live race reproduction because the reported failure depends on macOS launchd timing. The acceptance gap is reproducible by inspection on current main: the restart and handoff paths still use `kickstart -k` before any explicit old-service-exit-before-start sequence.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but it reopens launchd lifecycle tradeoffs that earlier commits intentionally moved away from, so the safest route needs launchd owner review plus live macOS validation.
