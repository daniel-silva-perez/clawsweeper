## Summary

Current main does not implement the requested two-phase LaunchAgent restart ordering. The normal `restartLaunchAgent` path still enables the service and runs `launchctl kickstart -k` directly, and the targeted unit test explicitly asserts no `bootout` or `bootstrap` occurs on that healthy restart path. The merged related PR #72616 addresses installed-but-unloaded/split-brain recovery and references this issue, but it did not replace the normal restart sequencing requested here.

## Best Possible Solution

Keep this issue open for maintainer judgment on the LaunchAgent restart contract. The likely implementation path is to redesign the normal macOS restart sequence in `src/daemon/launchd.ts` so the old service is confirmed stopped before a new process is started, while preserving installed-but-unloaded recovery and the detached self-restart handoff. Tests should explicitly cover launchctl call ordering, no regression for restart fallback, and the self-restart path; live macOS launchd proof would be valuable before landing.
