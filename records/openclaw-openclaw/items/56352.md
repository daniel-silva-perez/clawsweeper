## Summary

Keep this open. Current main still holds the Telegram reply turn inside the same awaited dispatch, model-fallback execution, and embedded per-session lane until the run and dispatcher settle; I did not find a post-visible-delivery lane-release boundary that would solve the reported blocked same-session follow-up.

## What This Changes

The issue requests releasing the same-session lane after a successful Telegram reply is delivered during failover, rather than keeping it occupied until fallback cleanup, finalization, or timeout completes.

## Best Possible Solution

Define a supported post-delivery lifecycle boundary for reply turns: either keep the lane only through a bounded finalization phase, or move fallback cleanup into a separate cleanup path that preserves transcript consistency, compaction state, fallback selection persistence, dispatcher idle tracking, and follow-up steering semantics.

## Reproduction Assessment

Unclear. The production report is concrete and current source confirms same-session serialization until the awaited run settles, but I did not find a deterministic Telegram/failover regression that reproduces visible delivery followed by delayed cleanup and a blocked same-session follow-up.

## Solution Assessment

No. Releasing the session lane immediately after the first visible reply is not proven to be the safest implementation; a bounded finalization or explicit cleanup-lane design is the narrower maintainable direction because remaining code still mutates session and transcript state.
