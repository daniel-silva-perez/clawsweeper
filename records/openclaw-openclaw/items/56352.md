## Summary

Keep this open. Current main still serializes Telegram auto-reply work through the shared channel turn, reply runner, model-fallback execution, and embedded per-session lane until the awaited run settles; the requested post-delivery lane-release boundary is not present.

## What This Changes

The issue asks OpenClaw to release a same-session lane after a visible Telegram reply is emitted during model failover, so follow-up inbound messages are not blocked by remaining failover cleanup or timeouts.

## Best Possible Solution

Define an explicit post-delivery lifecycle boundary for reply turns: either keep the session lane until a bounded finalization phase completes, or move failover cleanup into a separate cleanup phase that preserves transcript consistency, compaction state, delivery finalization, and follow-up steering semantics.

## Reproduction Assessment

Unclear. The production report is concrete and current source confirms same-session serialization until the awaited run settles, but I did not find a deterministic Telegram/failover test that reproduces delivered visible output followed by delayed cleanup and a blocked same-session follow-up.

## Solution Assessment

No. Releasing the lane immediately after first visible delivery is not yet proven to be the safest fix; a bounded finalization or explicit cleanup-lane design is the narrower maintainable direction because remaining code can still mutate session and transcript state.
