## Summary

Keep this issue open. Current main has adjacent failover and timeout improvements, but the specific requested behavior is not implemented: same-session inbound work is still intentionally serialized through the session lane until the embedded reply turn and its finalization path complete. The report is recent, concrete, and overlaps prior lane/failover incidents, so it should not be swept as stale without a maintainer decision on the session lifecycle boundary.

## What This Changes

The issue requests releasing a Telegram session lane immediately after a reply is delivered during model failover so later inbound messages are not blocked by remaining cleanup, timeout, or error propagation work.

## Best Possible Solution

Define an explicit post-delivery lifecycle boundary for block/final replies: either keep the session lane until all session-state mutations are complete within a bounded timeout, or move remaining failover/cleanup work to a separate cleanup lane with clear ownership of transcript, compaction, delivery finalization, and follow-up steering. The fix should preserve the documented one-active-run-per-session invariant unless maintainers intentionally change that contract.

## Reproduction Assessment

Unclear. The issue gives a plausible production incident path, and current source shows same-session work remains serialized, but a high-confidence reproduction still needs a synthetic test that emits a Telegram block/final reply, delays failover cleanup or error propagation, then sends a same-session follow-up.

## Solution Assessment

No. Releasing the session lane immediately on first reply delivery is not yet proven to be the safest fix because remaining cleanup may still touch session state; the maintainable solution needs an explicit cleanup phase or bounded finalization design rather than an unconditional early unlock.
