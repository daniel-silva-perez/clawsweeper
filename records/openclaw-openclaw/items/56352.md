## Summary

Keep this issue open. Current main has mitigations for provider fallback, idle timeout, and stale lane recovery, but the requested lifecycle boundary is still absent: Telegram inbound work awaits the shared reply turn, the reply turn awaits fallback execution, and the embedded runner holds the per-session lane until the whole run promise settles.

## What This Changes

The issue requests releasing a Telegram session lane immediately after a successful reply is delivered during model failover so later same-session inbound messages are not blocked by remaining cleanup, timeout, or error propagation work.

## Best Possible Solution

Define a deliberate post-delivery lifecycle boundary for block/final replies. Either keep the lane until all session-state mutations finish within a bounded timeout, or move remaining cleanup/failover finalization to an explicit cleanup phase that preserves transcript consistency, compaction state, delivery finalization, and follow-up steering semantics.

## Reproduction Assessment

Unclear. The issue gives a concrete production incident path and current source confirms same-session serialization, but a high-confidence reproduction still needs a synthetic Telegram/block-reply test with delayed failover cleanup followed by a same-session inbound message.

## Solution Assessment

No. An unconditional lane release immediately after first visible delivery is not proven to be the safest fix because the remaining path can still mutate session and transcript state; a bounded finalization or explicit cleanup-lane design is the safer maintainable direction.
