## Summary

Keep this PR open. The linked bug still appears valid on current main, but this patch is not ready to merge because its broad `OPENCLAW_SERVICE_` prefix would also strip a documented service-repair control used by `openclaw doctor`.

## What This Changes

The PR blocks `OPENCLAW_SERVICE_*` and `OPENCLAW_SYSTEMD_UNIT` in the host exec environment sanitizer, regenerates the macOS Swift policy, and adds a regression test for leaked gateway runtime markers.

## Best Possible Solution

Land a rebased fix that strips only internal service runtime markers from exec-spawned host environments, preserves documented operator controls such as `OPENCLAW_SERVICE_REPAIR_POLICY`, keeps the macOS generated policy in parity, and proves both marker stripping and clean CLI process exit for the reported workflow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid blocking documented service repair policy:** `src/infra/host-env-security-policy.json:140`
  - body: Adding `OPENCLAW_SERVICE_` to the inherited blocked prefixes also strips `OPENCLAW_SERVICE_REPAIR_POLICY=external`. Doctor reads that env to skip service install/start/restart repairs when an external supervisor owns the gateway lifecycle, so an exec-spawned `openclaw doctor --repair` would fall back to `auto`. Please block the specific runtime marker keys or explicitly preserve this documented key.
  - confidence: 0.91
