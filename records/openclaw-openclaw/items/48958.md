## Summary

Keep this PR open. Current main still lets hook/memory-prepended context reach prompt-image detection, so the intended fix is not implemented on main. The PR is a plausible implementation candidate, but it still needs the review feedback addressed and likely should cover the analogous CLI prompt-image path too.

## Best Possible Solution

Keep this PR open and either update it or replace it with an equivalent main-branch fix. The best implementation is a shared current-turn image-reference boundary used by both detectAndLoadPromptImages and the CLI prompt-image payload path, with focused regression tests proving that <relevant-memories>/<relevant_memories> blocks containing [media attached: ...] are ignored while real current prompt images outside those blocks still load. Address the existing review feedback before merge.

## Risks / Open Questions

- The proposed patch currently has unresolved review feedback: the relevant-memory fast-path guard is case-sensitive while the regex is case-insensitive, and one test assertion is vacuous.
- A fix limited to the embedded runner may leave the CLI external-harness image payload path exposed, because it also scans a hook-prepared prompt for image refs.
- Until fixed, stale [media attached: ...] paths in injected memory context can be interpreted as current-turn vision inputs, causing wrong model inputs and unintended local media reads within the existing prompt-image loading policy.
