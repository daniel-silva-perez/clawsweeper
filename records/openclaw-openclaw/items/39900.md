## Summary

Keep open. Current main has the OpenAI-compatible TTS base URL path and known OpenAI speech fields, but it still does not preserve or merge arbitrary `extraBody`/`extra_body` request fields for the bundled OpenAI speech provider, so the reported Kokoro/custom endpoint use case remains unsolved.

## What This Changes

Request to add `extraBody`/`extra_body` passthrough under the OpenAI-compatible TTS provider so custom `/audio/speech` backends can receive provider-specific request-body fields.

## Best Possible Solution

Implement the feature inside the bundled OpenAI speech provider: accept sanitized object-valued `extraBody` and `extra_body` config, preserve it through OpenAI TTS config resolution, merge it into both normal and telephony `/audio/speech` request bodies with explicit documented precedence, and cover the behavior in focused provider tests and TTS/OpenAI docs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a provider-level unit/probe using `messages.tts.providers.openai.extraBody` with a custom `baseUrl`: current source inspection shows the field is neither retained in normalization nor included in the serialized request body.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-owned OpenAI TTS passthrough; the core TTS provider schema already accepts object-valued provider settings, so a core config migration or broad plugin API change is unnecessary.
