## Summary

Keep open. Current main still has the helper-path gaps this PR is trying to fix, but the latest PR head still has a blocking cron/MCP tool-policy regression: canonical-provider cron runs moved to the CLI path get an owner-only grant, but they still do not receive the per-job runtime `toolsAllow` or message-tool policy that the embedded path enforces.

## What This Changes

The PR routes queued followups, model auth probes, LLM slug generation, and canonical-provider cron runs through `runCliAgent()` when the resolved runtime is CLI-backed, while adding CLI session/MCP routing metadata and focused regression tests.

## Best Possible Solution

Keep the PR active, but require the CLI cron alias branch to enforce the same runtime tool restrictions as the embedded branch before merge. The maintainable end state is a shared or CLI-native policy path that carries `toolsAllow`, message-tool disabling, explicit-target requirements, and owner-only grants consistently for bundled MCP tools.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: create an isolated cron job with a canonical provider that resolves to a CLI runtime and `toolsAllow: ["read"]`; the PR routes it through `runCliAgent()` while the CLI/MCP path lacks the runtime `toolsAllow` filter that the embedded branch applies.

## Solution Assessment

No. Routing helper paths through CLI dispatch is the right direction, but this PR is not yet the best implementation because the new cron CLI path preserves only owner-only grants, not the full per-job tool-policy contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve cron toolsAllow on CLI alias runs:** `src/cron/isolated-agent/run-executor.ts:170-206`
  - body: When a canonical cron provider resolves to a CLI runtime, this branch calls `runCliAgent()` without carrying the job's runtime `toolsAllow` policy. The embedded path still filters tools with `params.agentPayload.toolsAllow`, but the CLI loopback path only applies owner-only grants, so a job configured with a narrow `--tools` allowlist can see a broader bundled MCP surface after this change.
