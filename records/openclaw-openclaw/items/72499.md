## Summary

Keep open. Current main still has the helper-path gaps this PR targets, but the latest PR head still routes canonical-provider cron runs into the CLI path without preserving the full cron tool-policy contract, which is a correctness and security-sensitive MCP/tool-surface regression.

## What This Changes

The PR routes queued followups, model auth probes, LLM slug generation, and canonical-provider cron runs through CLI-backed execution, adding CLI session/routing metadata, MCP message-send tracking, and regression tests.

## Best Possible Solution

Update this PR so CLI-backed helper paths preserve the same auth, session, routing, cancellation, and cron tool-policy contracts as the embedded paths. For cron specifically, canonical-provider CLI alias runs should either enforce toolsAllow and message-tool policy in the CLI/MCP loopback path or remain on the embedded path until that enforcement exists.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure an isolated cron job with a canonical provider such as anthropic that resolves to a CLI runtime and set a restrictive --tools policy such as read; the PR routes that run through runCliAgent while the CLI params omit the allowlist/message-tool restrictions that the embedded branch receives.

## Solution Assessment

No. Routing these helper paths through runCliAgent is the right direction, but this PR is not yet the best fix because the new canonical cron CLI path is not policy-equivalent to the existing embedded path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Preserve cron tool policy on CLI alias runs:** `src/cron/isolated-agent/run-executor.ts:203-205`
  - body: This CLI branch only forwards an owner-only grant, while the embedded branch still receives `toolsAllow` plus the message-tool policy fields. A cron job configured with `--tools read` or with the message tool disabled can now enter the CLI alias path and get the default bundled MCP/message tool surface. Add equivalent CLI-side enforcement or keep these cron runs on the embedded path until the policy is preserved.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR can weaken per-job cron tool restrictions for canonical-provider runs that resolve to CLI-backed runtimes.

Concerns:

- **[medium] Cron CLI alias path can bypass tool restrictions:** `src/cron/isolated-agent/run-executor.ts:203`
  - body: The new CLI branch does not carry the runtime toolsAllow allowlist or message-tool disable/require flags that the embedded branch enforces. Because bundled MCP tools are exposed through the CLI loopback, restrictive cron jobs can receive a broader tool surface than configured.
