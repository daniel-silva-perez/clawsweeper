## Summary

Keep this issue open. Current main still does not make `openclaw tui` consult a trusted active Gateway runtime port before building its default local loopback URL, and the provided GitHub context shows open PR #73338 as an implementation candidate with closing syntax. Per cleanup policy, the issue should remain open until that PR or an equivalent fix merges.

## Best Possible Solution

Keep this issue open and review, land, or supersede #73338 with a small fix that records a trustworthy active local Gateway runtime port, makes default TUI loopback resolution consult it before falling back to env/config/defaults, and adds regression coverage proving an active Gateway on 48789 makes `openclaw tui` resolve `ws://127.0.0.1:48789` while explicit URL/env/remote config still win.

## Risks / Open Questions

- Closing this now would leave the default `openclaw tui` custom-port path unresolved for users who start the Gateway with `--port` and do not also pass `--url` or set `OPENCLAW_GATEWAY_PORT` manually.
- The eventual fix needs to preserve explicit `--url`, `OPENCLAW_GATEWAY_URL`, remote Gateway config, auth precedence, non-loopback `ws://` safety checks, and stale-lock safety.
