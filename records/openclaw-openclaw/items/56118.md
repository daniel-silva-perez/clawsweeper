## Summary

Keep this issue open. Current main has partial mitigation for higher-level browser-tool calls, but I did not find source, test, or history evidence that the exact direct remote-node `browser.proxy` `/tabs?profile=user` consent failure is fixed. The direct `openclaw nodes invoke` path still has shorter outer/default timeouts than Chrome MCP attach can need, and the `/tabs` Chrome MCP route still relies on the outer proxy boundary rather than a dedicated consent-aware attach budget.

## Best Possible Solution

Keep this open and fix it in the bundled browser plugin's node-proxy path. The likely patch is to use a longer existing-session-aware default for direct `browser.proxy` manage routes such as `/tabs`, align the outer `node.invoke`/gateway timeout with that budget, pass timeout and abort signal through Chrome MCP reachability/listing, and add a regression test for direct `browser.proxy` `GET /tabs` with `profile=user` where attach is delayed by consent.

## Risks / Open Questions
