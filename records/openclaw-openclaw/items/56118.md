## Summary

Keep this open. Current main still does not show a source, test, or docs-backed fix for the exact direct remote-node `browser.proxy` `GET /tabs?profile=user` consent/attach failure. The higher-level browser tool has a partial 45s existing-session timeout mitigation, but the direct `openclaw nodes invoke` path remains bounded by shorter node/proxy defaults while `/tabs` still enters Chrome MCP attach/listing without a dedicated consent-aware timeout budget.

## Best Possible Solution

Keep this issue open and fix it in the bundled browser plugin's node-host `browser.proxy` path. The likely maintainer path is to give direct manage routes such as `GET /tabs` an existing-session-aware default timeout, align the outer `node.invoke` timeout with the proxy/Chrome MCP budget, pass timeout and abort signal through Chrome MCP reachability/listing, and add a regression test for delayed `profile=user` attach via direct `browser.proxy` `/tabs`.
