## Summary

Keep open. Current main still leaves gateway disconnect as a status-only state while Ctrl+C/SIGINT use the normal double-press path and the exit path still has no hard fallback if terminal shutdown does not complete; the overlapping open PR #75294 covers terminal hangup cleanup for #75289 but does not reference or fully solve the disconnected Ctrl+C/SIGINT behavior here.

## What This Changes

Gateway WebSocket close should leave the TUI responsive to Ctrl+C, Ctrl+D, and SIGINT so users can exit without SIGKILL.

## Best Possible Solution

Keep this open until the TUI has a focused lifecycle fix for gateway-disconnected exit behavior: disconnected Ctrl+C/SIGINT and Ctrl+D should always reach bounded exit cleanup, with regression coverage for the WebSocket-close state rather than only generic signal shutdown.

## Reproduction Assessment

Yes. The issue gives concrete gateway-restart steps, and current main has matching disconnected-state, Ctrl+C/SIGINT, Ctrl+D, and requestExit code paths; this review did not perform the live macOS TUI run.

## Solution Assessment

Unclear. The proposed direction is mostly aligned, but the locked pi-tui drainInput already has default max/idle bounds, so the best fix is likely to combine disconnected-state key handling with an exit-only finally/timeout cleanup path and coordinate it with #75294.
