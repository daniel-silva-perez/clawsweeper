## Summary

Keep open. Current main still treats gateway disconnect as a status change only, while Ctrl+C continues through the normal double-press gate and SIGINT is mapped to that same path; Ctrl+D reaches the exit path, but that path has no exit-only finally/hard fallback if terminal shutdown rejects. The related open #75294 overlaps the shutdown cleanup piece for #75289, but it does not reference this issue and intentionally keeps normal Ctrl+C behavior unchanged, so this report should not be swept closed yet.

## What This Changes

The issue requests that the TUI remain exitable via Ctrl+C, Ctrl+D, or SIGINT after a gateway WebSocket close leaves it in a disconnected state.

## Best Possible Solution

Keep this issue open until the TUI has a focused lifecycle fix that covers disconnected-state exit behavior explicitly: Ctrl+D and signal shutdown should always complete through a bounded/finally-based exit path, and maintainers should decide whether disconnected Ctrl+C/SIGINT should bypass the normal double-press gate. That fix should include regression coverage for the gateway-disconnected state, not only generic shutdown cleanup.

## Reproduction Assessment

Yes. The report gives concrete gateway-restart steps and current main has matching code paths for gateway disconnect, Ctrl+C/SIGINT double-gating, and Ctrl+D requestExit; this review did not run the macOS live repro.

## Solution Assessment

Unclear. The suggested direction is mostly aligned, but the locked pi-tui drainInput already has default max/idle bounds, so the narrowest maintainable fix is likely disconnected-state key handling plus an exit-only finally/timeout fallback rather than assuming drainInput itself is unbounded.
