## Summary

Current main still has the core memory/performance behavior reported here. The session metadata store remains a monolithic `sessions.json` path loaded with `fs.readFileSync` and `JSON.parse`; update paths still re-read and rewrite the whole store; and central transcript/history paths still materialize full JSONL transcripts before limits or cursors are applied. Main has useful mitigations, including store maintenance/pruning and bounded title preview reads, and open PR #51946 directly targets the directory-backed metadata-store side. The issue should stay open as part of the active session-storage performance cluster rather than be closed prematurely.

## Best Possible Solution

Keep this issue open and route it into the active session-storage performance cluster. The best path is to review, land, or supersede the directory-backed/per-session metadata work from #51946 for `sessions.json`, then follow with a narrower change that replaces full-transcript history readers with cursor-aware or streaming readers while preserving current maintenance, migration safety, and transcript compatibility.
