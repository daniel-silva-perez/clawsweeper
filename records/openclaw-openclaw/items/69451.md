## Summary

Keep open. Current main has meaningful mitigations, including session maintenance, entry caps, disk-budget cleanup, bounded preview reads, and bounded chat-history response projection, but it still uses monolithic `sessions.json` reads/parses/writes and full-transcript materialization on several session-history paths. The cross-referenced #71227 and related open PR #51946 confirm this remains an active storage/performance problem rather than a stale report.

## Best Possible Solution

Keep this open and route it into the active session-storage performance cluster. The best fix is not another pruning-only change; it should either land or supersede the directory-backed/per-session metadata work, and separately replace history/usage endpoints that currently read whole JSONL transcripts with streaming or cursor-aware readers while preserving current maintenance and migration guarantees.
