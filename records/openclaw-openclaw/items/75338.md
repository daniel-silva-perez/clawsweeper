## Summary

Keep open. The latest head repaired the earlier raw-fetch boundary failure and CI checks are green, but the PR still introduces a narrow scheduler correctness bug where a queued request can be requeued after the queue has been cleared.

## What This Changes

This PR hardens Discord rate-limit handling across REST scheduling, Retry-After parsing, webhook and voice-upload error typing, voice upload retries, tests, and the changelog.

## Best Possible Solution

Keep the Discord-owned retry hardening, but make the scheduler's retry path respect queue clearing or abort state before requeueing and add a focused regression test for the clearQueue/429 race.

## Reproduction Assessment

Yes. A focused unit test can enqueue a Discord REST request, call `clearQueue()` while the request is active, then return a 429 and verify the original promise rejects or settles without requeueing and `queueSize` returns to zero.

## Solution Assessment

No. The raw network-boundary repair is now the right direction, but the scheduler retry path still needs a small cancellation/cleared-generation guard before this is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Skip rate-limit requeue after queue clearing:** `extensions/discord/src/internal/rest-scheduler.ts:350-352`
  - body: `clearQueue()` only rejects entries that are pending at that moment. If an active queued request then receives a 429, this new branch requeues it and returns before `queuedRequests` is decremented, so work can survive a shutdown/teardown clear and run later. Track a cleared/cancelled generation or otherwise reject instead of requeueing after the queue has been cleared.
  - confidence: 0.91

## Repair Work Prompt

Repair PR https://github.com/openclaw/openclaw/pull/75338 on the current branch. The remaining blocker is in the Discord REST scheduler: after `clearQueue()` or `abortPending()` rejects pending work, an active queued request that later throws `RateLimitError` can still be requeued by `runQueuedRequest` and keep `queuedRequests` non-zero. Add the smallest scheduler state needed, such as a cleared/cancelled generation on queued entries, so cleared or aborted queued requests reject/settle and are not requeued after the clear. Add a regression test in `extensions/discord/src/internal/rest.test.ts` that enqueues a request, calls `clearQueue()` while the first request is active, then returns a 429 and verifies no retry is scheduled, the promise rejects or settles appropriately, and `queueSize` returns to zero.
