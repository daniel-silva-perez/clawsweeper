## Summary

Keep open. Current main still contains the raw script-file preflight scanner that applies an all-caps `$VAR` regex across full Python/Node file contents without string, docstring, comment, or execution-context awareness. The exact no-approval gateway YOLO path is now documented and skipped, but the false-positive scanner remains active in other preflight paths, so the reported product bug is still actionable.

## Best Possible Solution

Keep this issue open. Update `src/agents/bash-tools.exec.ts` so script-file preflight is lexical-context aware for Python and Node, or narrow the `$VAR` detection to real execution/shell-leak contexts. Add focused regression coverage in `src/agents/bash-tools.exec.script-preflight.test.ts` for `$SIM` in Python docstrings/help text and dollar-formatted output, while retaining negative coverage for raw `$DM_JSON` code paths and the existing gateway YOLO skip behavior. The closed unmerged PRs #67644 and #67924 are reasonable references, but current main still needs the fix.

## Risks / Open Questions

- A fix must preserve the current rejection for actual leaked shell-variable syntax such as `payload = $DM_JSON` while allowing benign `$VAR` text inside strings, docstrings, comments, and help text.
- The implementation should keep the documented gateway `security=full` plus `ask=off` YOLO exemption intact and avoid reintroducing script-preflight rejection on that path.
- The issue body's lowercase f-string examples like `$cost` likely do not match the current uppercase-start regex; the concrete remaining false positive is the `$SIM`-class all-caps token inside Python literals.
