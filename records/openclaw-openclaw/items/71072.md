## Summary

Keep open. Current main still returns the correct empty registry for `onlyPluginIds: []`, but it does not implement the PR's intended O(1) fast-path: it still checks the registry cache, enters the plugin load guard, creates loader/registry/runtime scaffolding, discovers plugins, and loads the manifest registry before scope filtering. The live PR diff remains useful but is not ready as-is because it references the old `inFlightPluginRegistryLoads` symbol instead of the current `PluginLoaderCacheState` API.

## Best Possible Solution

Keep this PR open and ask for a small implementation update against current main: add the explicit empty-scope return before cache/discovery work, use the current `PluginLoaderCacheState` reentry API rather than the removed guard, preserve activation-side-effect clearing when `activate` is true, return `createEmptyPluginRegistry()`, and keep the focused test proving discovery and manifest registry loading are skipped for `onlyPluginIds: []`.
