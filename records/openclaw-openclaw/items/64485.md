## Summary

Keep this PR open. Current main still has the direct Docker Compose failure the PR is trying to address: both services use raw `${OPENCLAW_CONFIG_DIR}` / `${OPENCLAW_WORKSPACE_DIR}` bind sources, and `docker compose config` fails when those vars are unset. The setup script exports defaults, but docs still include direct `docker compose` manual flows, so the central user problem is not implemented on main. The PR is not a security/supply-chain drive-by based on the supplied diff, but it still needs normal review cleanup before merge.

## Best Possible Solution

Keep the PR open and review it as an implementation candidate. The best fix is a focused Docker Compose change that keeps host bind-source defaults usable for direct `docker compose` flows, keeps container runtime paths anchored at `/home/node/.openclaw` via container `HOME`, adds a small rendered-compose assertion if feasible, and fixes the changelog attribution before merge.
