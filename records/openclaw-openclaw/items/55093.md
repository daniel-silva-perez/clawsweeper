## Summary

Keep this PR open. The runtime-facade build stabilization remains useful, but the dirty local version behavior advertised by the PR is incomplete because the new marker is written to build metadata without being read or rendered by the version output path.

## What This Changes

The PR adds recursive tsdown entries for `src/**/*.runtime.ts`, refactors `scripts/write-build-info.ts` into testable helpers that infer a dirty display marker, and adds focused build-info and tsdown config tests.

## Best Possible Solution

Finish the display-version contract by reading the optional marker from build metadata, composing it only into human-facing `openclaw --version` output, and preserving compatibility/version APIs that should remain plain semver-like values.

## Reproduction Assessment

Yes. The code path is deterministic: the PR writes `displayVersionMarker`, while current `src/version.ts` and `src/entry.version-fast-path.ts` only resolve and print `VERSION` plus an optional commit, so a focused unit test with marker-bearing build-info would reproduce the missing output.

## Solution Assessment

No. The runtime facade entry approach is a reasonable narrow fix, but the dirty-version portion is not the best complete solution until the marker is wired into version display and covered by `src/version.test.ts` and `src/entry.version-fast-path.test.ts`.
