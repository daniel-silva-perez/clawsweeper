## Summary

Keep open. Current main still has the stale cron cache behavior, but this PR should not merge as-is because the proposed cache invalidation misses the runtime state sidecar and can record mtimes before the files are actually read.

## What This Changes

The PR adds an mtime-based cache check to `CronService.ensureLoaded()`, ignores non-file store paths, and adds Vitest coverage for external `jobs.json` edits before cron list/run operations.

## Best Possible Solution

Refresh the implementation on current main so cron cache freshness covers every file read by `loadCronStore()`, records observed mtimes after successful reads, preserves the non-file guard, and adds focused coverage for both `jobs.json` edits and sidecar-only edits before `cron.list` and `cron.run`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to populate `CronService` state, externally edit `jobs-state.json` or `jobs.json`, then call `cron.list` or `cron.run`; current main and the PR fast path can both serve cached state on non-force loads.

## Solution Assessment

No. Mtime-based invalidation is a reasonable direction, but this PR is not the best current fix because it watches only the config store path, trusts a pre-read mtime for future cache hits, and needs rebasing onto the current split-store implementation.

  - confidence: 0.92

If a write's mtime is observed before its contents are read, old data can be cached with the new mtime and future reloads skipped; read first, then record mtimes for all loaded files.
  - confidence: 0.78
