## Summary

Keep this PR open. Current main still has the stale live cron cache behavior for normal Gateway `cron.list` and `cron.run` paths; only scheduler timer force-reload paths cover adjacent external schedule edits. The PR remains a useful implementation candidate, but it needs refresh for the current split `jobs.json` / `jobs-state.json` store and the mtime ordering concern already raised in review.

## Best Possible Solution

Keep this PR open for a refreshed patch against current main: invalidate normal live cron cache reuse when either `jobs.json` or `jobs-state.json` changes, handle non-file store paths, capture file mtimes in an ordering that avoids permanent stale-cache pinning, and add focused regression tests for external deletion/list/run behavior plus sidecar runtime-state recovery.

## Risks / Open Questions

- The PR was written for the older single-file cron store; current runtime recovery state now lives in `jobs-state.json`, so watching only `jobs.json` would leave part of the reported recovery path stale.
- The Greptile review's stat-before-read ordering concern is still relevant to resolve before merge so cache invalidation cannot permanently pin stale contents around a concurrent write.
- Adding stat checks to normal cron operations affects a hot path; the final implementation should avoid unnecessary syscalls while still detecting external edits reliably.
