## Summary

Keep this PR open for maintainer review, but do not merge it as-is. The intended stale-cache bug still exists on current main, yet the patch only watches `jobs.json` and records the watched mtime before reading; current cron persistence now splits runtime fields into `jobs-state.json`, so the proposed fix misses the manual recovery path for stuck runtime markers.

## What This Changes

The PR adds mtime-based cache invalidation to `CronService.ensureLoaded()` and new Vitest coverage for manual cron recovery and externally deleted jobs.

## Best Possible Solution

Refresh the implementation so cron cache reuse invalidates on both `jobs.json` and the derived `jobs-state.json`, records observed mtimes after successful reads, preserves non-file handling, and adds focused coverage for external sidecar recovery before `cron.run` and `cron.list`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Watch jobs-state.json before returning cached cron data:** `src/cron/service/store.ts:34-36`
  - body: Current main stores runtime fields such as `runningAtMs` in the derived `jobs-state.json` sidecar. This fast path only stats `state.deps.storePath`, so clearing a stuck runtime marker in the sidecar can leave `jobs.json` unchanged and keep returning the stale in-memory job.
  - confidence: 0.92
- **[P2] Record cron mtimes after reading the store:** `src/cron/service/store.ts:42-48`
  - body: The new cache hit depends on `state.storeFileMtimeMs`, but the load path records an mtime captured before `loadCronStore()` reads the file. If an external write's mtime becomes visible before the new contents are read, the service can cache old contents with the new mtime and skip future reloads.
  - confidence: 0.78
