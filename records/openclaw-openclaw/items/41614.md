## Summary

Keep this PR open. Current main still reuses cached cron state for normal list/read/manual-run paths, so the reported stale live-store behavior remains real, but the proposed patch needs refresh for the current split cron store and the mtime ordering concern already raised in review.

## What This Changes

The PR adds mtime-based invalidation to `CronService.ensureLoaded()` for external `jobs.json` edits and adds regression tests for manual clearing/deletion before `cron.run` and `cron.list`.

## Best Possible Solution

Refresh the PR against current main so normal cron cache reuse is invalidated when either the definition file or runtime sidecar changes, with non-file path handling, robust read/mtime ordering, and focused regression coverage for external deletion/list/run behavior plus sidecar runtime-state recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Watch jobs-state.json before returning cached cron data:** `src/cron/service/store.ts:47-49`
  - body: Runtime fields now live in the derived `jobs-state.json` sidecar. This fast path only stats `state.deps.storePath`, so clearing `runningAtMs` in the sidecar can leave `jobs.json` unchanged and `ensureLoaded()` will keep returning the stale in-memory job, preserving the `already-running` failure this PR is meant to fix.
  - confidence: 0.92
- **[P2] Store mtimes after reading the cron files:** `src/cron/service/store.ts:47-49`
  - body: The new cache hit depends on `state.storeFileMtimeMs`, but the load path records the mtime before `loadCronStore()` reads the file. If an external write's mtime becomes visible before the new contents are read, the service can cache old contents with the new mtime and future non-force calls will skip reload indefinitely.
  - confidence: 0.78

## Risks / Open Questions

- The proposed fast path only watches `jobs.json`, while runtime recovery edits such as clearing `runningAtMs` now belong in `jobs-state.json`.
- The patch records mtime before reading the store; under a concurrent external write, stale contents could be cached with the newer mtime and then reused indefinitely until another write changes the mtime.
- Adding filesystem stats to normal cron operations affects a hot path, so the refreshed implementation should avoid redundant syscalls while still detecting both backing-file changes.
