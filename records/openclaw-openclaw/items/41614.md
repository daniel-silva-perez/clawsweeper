## Summary

Keep this PR open. Current main still reuses the in-memory cron store for normal service operations unless a caller explicitly force-reloads, so the PR's central stale-cache problem is not implemented on main. The work still looks relevant, but it needs refresh for the current split `jobs.json` / `jobs-state.json` store and should address the existing mtime ordering concern before merge.

## Best Possible Solution

Keep this PR open for maintainer review, but ask for an updated patch against current main: invalidate cached cron data for normal live operations when either `jobs.json` or `jobs-state.json` changes, handle non-file paths, capture freshness metadata in an order that cannot permanently pin stale contents, and keep regression coverage for external deletion/list/run behavior plus sidecar runtime-state recovery.

## Risks / Open Questions

- The PR's original recovery story was written before the current split-store model; clearing a stuck runtime marker now belongs in `jobs-state.json`, so watching only `jobs.json` would leave an important stale-cache path unsolved.
- The mtime invalidation approach should avoid recording a post-write mtime with pre-write contents, otherwise a concurrent write could pin stale data until a later edit.
- Adding stats to normal cron operations changes a hot path; the final patch should keep the syscall behavior intentional and covered by focused tests.
