## Summary

Keep this PR open for maintainer review, but it is not merge-ready. The stale-cache problem is real on current main, and the PR is useful source work, but the implementation only invalidates on `jobs.json` while current cron runtime state lives in `jobs-state.json`, and it also makes cache hits depend on an mtime captured before the store read.

## What This Changes

The PR adds mtime-based cache reuse checks to `CronService.ensureLoaded()`, ignores non-file store paths, and adds Vitest coverage for externally edited or deleted cron job storage.

## Best Possible Solution

Refresh the PR so cache reuse invalidates on both the main cron store and the derived runtime sidecar, records observed mtimes after successful reads, preserves the non-file handling, and adds focused coverage for sidecar edits before `cron.list` and `cron.run`.

## Reproduction Assessment

Yes. A focused reproduction is to load a cron job, externally clear or remove runtime state in `jobs-state.json` without changing `jobs.json`, then call `cron.list` or `cron.run`; current main and the PR fast path both have enough source evidence to explain the stale-cache behavior.

## Solution Assessment

No. Mtime-based invalidation is the right direction, but this implementation is not the narrowest correct fix because it ignores the sidecar file and stores the cache mtime before reading the store.

## Review Findings

- **[P2] Watch the cron state sidecar before cache hits:** `src/cron/service/store.ts:34-36`
  - body: Current main stores runtime fields such as `runningAtMs` in `jobs-state.json`, and `loadCronStore()` merges that sidecar while ignoring inline state when it exists. This fast path stats only `state.deps.storePath`, so clearing a stuck runtime marker in the sidecar can leave `jobs.json` unchanged and keep returning stale in-memory data for `cron.list` and `cron.run`.
  - confidence: 0.92
- **[P2] Record the mtime after reading cron storage:** `src/cron/service/store.ts:35`
  - body: The new cache hit compares the current stat to `state.storeFileMtimeMs`, but `ensureLoaded()` still records an mtime captured before `loadCronStore()` reads the store. If an external write's mtime becomes visible before the new contents are read, the service can cache old contents with the new mtime and skip future reloads.
  - confidence: 0.78
