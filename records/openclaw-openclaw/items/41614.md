## Summary

Keep this PR open, but do not merge it as-is. Current main still reuses the in-memory cron store on normal read/run paths, so the underlying stale-cache problem remains; however, the PR only watches `jobs.json` even though current cron loads runtime state from `jobs-state.json`, and the proposed cache hit trusts mtimes captured before the successful read.

## What This Changes

The PR adds an mtime-based `ensureLoaded()` cache check for the cron `jobs.json` file, ignores non-file store paths, and adds Vitest coverage for external `jobs.json` edits before list/run operations.

## Best Possible Solution

Refresh the implementation so cron cache freshness covers every file read by `loadCronStore()`, including the derived runtime sidecar, records observed mtimes after successful reads, preserves the non-file guard, and adds focused coverage for sidecar-only edits before `cron.list` and `cron.run`.

## Reproduction Assessment

Yes. Load a cron job into memory, externally clear or delete runtime state in `jobs-state.json` without changing `jobs.json`, then call `cron.list` or `cron.run`; the source paths show the proposed cache hit can still serve stale sidecar-backed state.

## Solution Assessment

No. Mtime-based invalidation is a reasonable direction, but this patch is incomplete because it watches only the config store path and relies on a pre-read mtime for future cache hits.

  - confidence: 0.92

  - body: The new cache hit trusts `state.storeFileMtimeMs`, but that value is still captured before `loadCronStore()` reads the files. A write whose mtime becomes visible before its contents are read can cache old data with the new mtime and skip later reloads.
  - confidence: 0.78
