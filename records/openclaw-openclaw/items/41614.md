## Summary

Keep this PR open. The underlying stale cron-store cache problem still exists on current main, but the branch is now stale against the current cron persistence code and the proposed mtime cache hit is incomplete because it watches only `jobs.json` while current cron loading also depends on the derived runtime sidecar.

## What This Changes

The PR adds an mtime-based cache check to `CronService.ensureLoaded()`, ignores non-file store paths, and adds Vitest coverage for external `jobs.json` edits before cron list/run operations.

## Best Possible Solution

Refresh the implementation on current main so cron cache freshness covers every file read by `loadCronStore()`, records observed mtimes after successful reads, preserves the non-file guard, and adds focused coverage for both `jobs.json` edits and sidecar-only edits before `cron.list` and `cron.run`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to load cron state into memory, externally edit `jobs-state.json` or `jobs.json`, then call `cron.list` or `cron.run`; current main and the PR fast path both show non-force loads can serve cached state.

## Solution Assessment

No. Mtime-based invalidation is a reasonable direction, but this PR is not the best current fix because it watches only the config store path, trusts a pre-read mtime for future cache hits, and needs rebasing onto the current split-store implementation.

  - confidence: 0.78
