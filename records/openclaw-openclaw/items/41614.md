## Summary

Keep this PR open. Current main still has the stale cron cache behavior for normal read and manual-run paths, but the proposed patch is not merge-ready because it only invalidates on the main job-definition file and relies on an mtime ordering that can bless stale reads.

## What This Changes

The PR adds mtime-based invalidation to `CronService.ensureLoaded()` for external cron store edits and adds regression tests for manual recovery before `cron.run` and `cron.list`.

## Best Possible Solution

Refresh the PR so cron cache reuse invalidates on both backing files in the split store, records mtimes in an order that cannot permanently bless stale reads, preserves non-file path handling, and adds focused regression coverage for sidecar runtime-state recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Watch jobs-state.json before returning cached cron data:** `src/cron/service/store.ts:48-50`
  - body: Runtime fields now live in the derived `jobs-state.json` sidecar. This fast path only stats `state.deps.storePath`, so clearing `runningAtMs` in the sidecar can leave `jobs.json` unchanged and keep returning the stale in-memory job, preserving the `already-running` failure this PR is meant to fix.
  - confidence: 0.92
- **[P2] Record cron mtimes after reading the store:** `src/cron/service/store.ts:48-50`
  - body: The cache hit depends on `state.storeFileMtimeMs`, but the load path records the mtime before `loadCronStore()` reads the file. If an external write's mtime becomes visible before the new contents are read, the service can cache old contents with the new mtime and future non-force calls will skip reload indefinitely.
  - confidence: 0.78
