## Summary

Keep this PR open, but it is not merge-ready. The stale in-memory cron cache bug still exists on current main, yet the PR only watches `jobs.json` while current cron runtime state is also loaded from `jobs-state.json`, and the new cache comparison trusts an mtime captured before the store read.

## What This Changes

The PR adds mtime-based cache checks to `CronService.ensureLoaded()`, ignores non-file store paths, and adds Vitest coverage for external cron store edits.

## Best Possible Solution

Refresh the implementation so cache reuse invalidates on every file read by `loadCronStore()`, including the derived runtime sidecar, records observed mtimes after successful reads, keeps the non-file guard, and adds focused coverage for sidecar edits before `cron.list` and `cron.run`.

## Reproduction Assessment

Yes. Load a cron job, externally clear or remove runtime state in `jobs-state.json` without changing `jobs.json`, then call `cron.list` or `cron.run`; the source paths show the cache can be reused without seeing the sidecar edit.

## Solution Assessment

No. Mtime-based invalidation is a reasonable direction, but this patch is incomplete because it watches only the config store path and relies on a pre-read mtime for future cache hits.

## Review Findings

- **[P2] Watch the cron state sidecar before cache hits:** `src/cron/service/store.ts:34-36`
  - body: `loadCronStore()` merges runtime fields from the derived `jobs-state.json`, and `saveCronStore()` can write only that sidecar when runtime state changes. This fast path stats only `state.deps.storePath`, so clearing a stuck marker in the sidecar can still leave `cron.list` and `cron.run` serving stale in-memory data.
  - confidence: 0.92
- **[P2] Record cache mtimes after successful reads:** `src/cron/service/store.ts:35`
  - body: The new cache hit trusts `state.storeFileMtimeMs`, but that value is still captured before `loadCronStore()` reads the store. A write whose mtime becomes visible before its contents are read can cache old data with the new mtime and skip later reloads.
  - confidence: 0.78
