## Summary

Keep open. The PR targets a real cron cache-staleness bug, but it should not merge as-is because current main now loads runtime state from a `jobs-state.json` sidecar while the patch only watches `jobs.json`, and the new cache hit trusts a mtime captured before the successful read.

## What This Changes

The PR modifies `src/cron/service/store.ts` to stat the cron store mtime before reusing cached state and adds `src/cron/service.external-store-reload.test.ts` coverage for external clearing of `runningAtMs` and job deletion.

## Best Possible Solution

Refresh the implementation on current main so cron cache freshness covers every file read by `loadCronStore()` (`jobs.json` plus the derived state sidecar), records observed mtimes after successful reads, preserves the non-file guard, and adds focused coverage for both config edits and sidecar-only edits before `cron.list` and `cron.run`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to populate a `CronService` in memory, externally edit `jobs-state.json` or `jobs.json`, then call `cron.list` or `cron.run`; current main and the PR fast path can both serve cached state on non-force loads.

## Solution Assessment

No. Mtime-based invalidation is a reasonable direction, but this patch is not the best current fix because it watches only the config store path and uses a pre-read mtime as the future cache validator.

  - confidence: 0.78
