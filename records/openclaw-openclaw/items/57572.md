## Summary

Keep this draft PR open for maintainer follow-up, but it is not ready to merge. Current main still lacks the requested restart-mode surface, while the proposed patch has blocking correctness gaps: the new config is not accepted by strict schemas, the graceful path still uses the public start path that resets restart state, and reload/account override behavior is incomplete.

## What This Changes

The PR adds a gateway-level and per-channel channel-health restart mode, threads the mode into health-monitor startup/reload code, and adds a ChannelManager accessor for per-channel healthMonitor config.

## Best Possible Solution

Define the intended health-recovery contract first, then implement it end to end: accepted config schema and generated metadata, docs, reload behavior, focused tests, and either a real provider reconnect/preserve-start seam or an explicit decision to rely on the shipped stale-socket mitigation plus healthMonitor.enabled opt-outs.

## Reproduction Assessment

Yes for the PR defects: static tracing shows the advertised graceful path still calls startChannel, which resets restart attempts, and the strict schemas do not accept the new keys. No live Anthropic/Discord prompt-cache reproduction was run because this was a read-only review.

## Solution Assessment

No. The patch is not the best current solution because it exposes a public config surface before schema/docs/tests and does not create a real reconnect or preserve-restart-attempt path; a narrower, tested ChannelManager/provider seam would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve restart state through startChannel:** `src/gateway/channel-health-monitor.ts:194`
  - body: `graceful` skips the explicit `resetRestartAttempts()` call, but it still calls the public `startChannel()`. That wrapper invokes `startChannelInternal()` with the default `preserveRestartAttempts = false`, so the restart-attempt map is deleted anyway and the new mode behaves like `stop-start` for the state it is meant to preserve.
  - confidence: 0.91
- **[P2] Accept the new config keys in the strict schemas:** `src/config/types.gateway.ts:464`
  - body: The PR adds `gateway.channelHealthRestartMode` and `healthMonitor.restartMode` to TypeScript types only. OpenClaw config is validated by strict zod schemas, so users who add either advertised key will have config validation fail before the health monitor can see it.
  - confidence: 0.94
- **[P2] Merge account overrides with channel restart mode defaults:** `src/gateway/server-channels.ts:247`
  - body: Returning an account-level `healthMonitor` object as soon as it exists makes `channels.discord.accounts.foo.healthMonitor.enabled` mask a channel-level `healthMonitor.restartMode`. Field-level fallback should preserve the channel restart mode when the account override does not set one.
  - confidence: 0.86
- **[P2] Add the new gateway key to the reload plan:** `src/gateway/server-reload-handlers.ts:103-104`
  - body: This threads `defaultRestartMode` through the reload handler, but `buildGatewayReloadPlan()` has no rule for `gateway.channelHealthRestartMode`. Changing the new global knob at runtime therefore will not restart the health monitor and the new mode will not take effect until a broader restart.
  - confidence: 0.88
- **[P2] Update ChannelManager fixtures for the new required method:** `src/gateway/server-channels.ts:151-158`
  - body: Making `getChannelHealthMonitorConfig` required on `ChannelManager` means existing typed test fixtures that return `ChannelManager` without that method no longer satisfy the interface. Add the method to the shared mocks or make the dependency narrower where only the health monitor needs it.
  - confidence: 0.82
