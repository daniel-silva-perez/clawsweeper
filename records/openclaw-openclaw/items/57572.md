## Summary

Keep this PR open. Current main still lacks the requested `channelHealthRestartMode` / per-channel `restartMode` config surface and the health monitor still performs stop, reset restart attempts, then start. The later #69833 mitigation reduces false stale-socket restarts by using transport activity, but it does not implement this PR’s configurable lightweight reconnect behavior.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a tighter design that either implements a real health-monitor restart mode with schema/docs/generated metadata/tests plus a ChannelManager/provider reconnect or preserve-attempts seam, or explicitly reframes the work if maintainers decide the shipped #69833 stale-socket mitigation and existing `healthMonitor.enabled` opt-outs are the intended product answer.

## Risks / Open Questions

- Merging the PR as described would likely not preserve restart attempts unless the public `ChannelManager.startChannel()` path gains a preserve/reconnect option or a true provider reconnect seam.
- The proposed config surface needs zod schema, generated metadata/help, docs, reload planning, and tests; type-only config additions would drift from strict runtime validation.
- A restart mode named `graceful` or `reconnect` should not imply Anthropic prompt-cache preservation without a provider/runtime contract that actually preserves the relevant session state.
- No supply-chain-sensitive changes were visible in the supplied PR file list, but the runtime change should still be reviewed for restart-limit bypass or masking genuinely unhealthy channels.
