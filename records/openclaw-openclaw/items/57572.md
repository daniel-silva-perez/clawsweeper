## Summary

Keep this PR open. Current main has a later mitigation for false stale-socket restarts (#69833), but it does not implement the PR’s requested `channelHealthRestartMode` / per-channel `restartMode` surface or any lightweight reconnect path. The health monitor still performs stop/reset/start for unhealthy channels, and the config schema/docs only expose interval, stale threshold, max restarts, and `healthMonitor.enabled` overrides.

## Best Possible Solution

Keep the PR open for maintainer review, but ask for a tighter design: either implement a real health-monitor restart mode with schema/docs/tests and a channel/provider reconnect or preserve-restart-attempt seam, or explicitly close/reframe the work if maintainers decide the shipped #69833 stale-socket mitigation plus existing `healthMonitor.enabled` opt-outs are the intended solution.

## Risks / Open Questions

- Closing as implemented would be premature: current main reduces false stale-socket restarts but still lacks the requested configurable lightweight reconnect behavior.
- The proposed PR design likely needs revision before merge because skipping the health monitor’s explicit reset does not bypass `startChannel()` resetting restart attempts, and it does not introduce an actual provider-level reconnect/resume contract.
- If this feature proceeds, schema/generated metadata/docs/tests need to be updated together with the runtime behavior so the new config is accepted and discoverable.
