## Summary

Keep open. Current main still appears to have the immediate ACP history gap: ACP spawn dispatches the initial task through the gateway, while transcript persistence for ACP turns happens only after the runtime turn completes. This contributor PR is relevant, but it is not merge-ready because it is stale against current main, reports task delivery after a best-effort seed path, misses required changelog credit, and mixes the ACP bug fix with security-sensitive workflow changes.

## What This Changes

The PR seeds ACP `sessions_spawn` transcripts with the initial task, forwards input provenance into the first ACP dispatch, dedupes the later replayed prompt, adjusts prompt-only SessionManager handling, adds tests and a changelog entry, and changes several CI/labeler workflows.

## Best Possible Solution

Land a narrow, rebased ACP-only fix that writes the initial spawn task to the child transcript before reporting successful task delivery, dedupes only the first seeded replay by marker/provenance, preserves prompt-only transcript normalization, adds focused regression coverage, and leaves workflow token/permission policy changes to a separate maintainer-owned PR if still needed.

## Reproduction Assessment

Yes. Spawn an ACP session with `sessions_spawn(runtime: "acp", task: ...)`, inspect `sessions_history` or `chat.history` before the first assistant reply, and current main's code shows that immediate history reads the transcript file while ACP spawn only dispatches the task and persists transcript text later.

## Solution Assessment

No as-is. The seed-and-dedupe direction is the right product fix, but the current PR should be rebased, narrowed to ACP runtime/session transcript behavior, report delivery only after the seed is actually persisted, and fix the changelog credit before merge.
