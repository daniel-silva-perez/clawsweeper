## Summary

Keep open. Current main still does not seed ACP `sessions_spawn` task text into the child transcript before dispatch, and the discussion includes useful production validation for the empty-history failure. The PR is useful but not ready as-is because it can report task delivery after a best-effort seed failure, misses required changelog attribution, and mixes the ACP fix with workflow token/permission changes that need maintainer review.

## What This Changes

The PR seeds ACP `sessions_spawn` transcripts with the initial task, forwards provenance into the first ACP dispatch, dedupes the later replayed prompt, adjusts prompt-only session-manager handling, adds tests and a changelog entry, and changes CI/labeler workflow behavior.

## Best Possible Solution

Land a narrow, rebased ACP-only fix on current main that writes the initial spawn task to the child transcript before reporting success, dedupes only the first seeded replay by marker/provenance, preserves prompt-only transcript normalization, and moves workflow policy/auth changes to a separate maintainer-owned PR if they are still needed.

## Reproduction Assessment

Yes. Spawn an ACP session with `sessions_spawn(runtime: "acp", task: ...)`, inspect `sessions_history` before the first assistant reply, and current main's code shows that history reads the transcript file while ACP spawn only dispatches the task and persists transcript text later.

## Solution Assessment

No as-is. The seed/dedupe direction is the right product fix, but the PR should report delivery only when the seed actually persists, credit the contributor in the changelog, rebase onto current main, and split unrelated workflow/security-policy changes.
