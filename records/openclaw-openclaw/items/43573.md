## Summary

Keep open: current main still lacks immediate ACP spawn-task transcript seeding, and the discussion plus validation comment show this is a real ACP delegation gap. The PR is useful but not merge-ready because it can falsely report task delivery after seed failure, misses required changelog attribution, and mixes the ACP fix with unrelated workflow/token-policy changes that need maintainer review.

## What This Changes

The PR seeds ACP `sessions_spawn` transcripts with the initial task, forwards input provenance, dedupes the first replayed ACP prompt, adjusts prompt-only transcript initialization, adds ACP transcript/spawn tests, updates the changelog, and changes several CI/labeler workflows.

## Best Possible Solution

Land a narrow, rebased ACP-only fix that writes the initial spawn task to the child transcript before returning success, dedupes only the seeded first replay by provenance or marker, reports delivery truthfully, preserves current main's newer ACP spawn behavior, and moves CI/labeler policy changes to a separate maintainer-reviewed PR if still needed.

## Reproduction Assessment

Yes. Spawn an ACP session with `sessions_spawn(runtime: "acp", task: ...)`, inspect `sessions_history` before the first assistant reply, and current main's source shows history reads the transcript file while ACP spawn only dispatches the task and persists the turn later.

## Solution Assessment

No as-is. The ACP seed/dedupe direction is the right narrow product fix, but this PR should be rebased, should not claim delivery when seeding failed, and should split unrelated workflow/security policy changes.
