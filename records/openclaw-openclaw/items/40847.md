## Summary

Keep open. The diagnostic goal is valid, but the PR patch is not mergeable as written because the no-provider warning is unreachable, all-skipped zero-output attempts still remain silent, and the new warnings bypass the existing OpenClaw warning logger.

## What This Changes

The PR adds warning logs in `src/media-understanding/runner.ts` when selected image attachments produce zero media-understanding outputs, including failed-attempt details and a no-provider message.

## Best Possible Solution

Implement the image zero-output diagnostics at the actual runner decision points: warn before the no-provider early return, warn when selected image attempts all skip or return empty output, preserve failed-attempt context, use `logWarn`, and add focused tests for each warning path without changing successful image routing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Move the no-provider warning before the early return:** `src/media-understanding/runner.ts:817-821`
  - body: When no providers resolve, `runCapability` returns at the earlier `resolvedEntries.length === 0` guard, so this branch is unreachable and operators still get no warning for selected images with no image-understanding provider.
  - confidence: 0.94
- **[P2] Warn when every image attempt is skipped:** `src/media-understanding/runner.ts:801-821`
  - body: If each resolved image provider returns `skipped`, `failedAttempts` is empty and `allAttempts` is non-empty, so neither warning branch runs. That leaves a zero-output path this PR targets silent.
  - confidence: 0.92
- **[P3] Use logWarn for the new diagnostics:** `src/media-understanding/runner.ts:811`
  - body: The new warnings call `console.warn` directly while existing media-understanding failures use `logWarn`, which also routes messages through OpenClaw logging. These diagnostics may be missed by operators reading normal gateway logs.
  - confidence: 0.82
