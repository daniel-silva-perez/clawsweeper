## Summary

Keep open. The PR is aimed at a real diagnostics gap, but the proposed block still misses key zero-output paths on current main: the no-provider warning is placed after an early return, and all-skipped attempts remain silent.

## What This Changes

The PR adds warning logs in `src/media-understanding/runner.ts` when image media-understanding selects attachments but produces zero outputs.

## Best Possible Solution

Implement the diagnostic in the runner control flow itself: warn before the no-provider early return, warn when all selected-image attempts are skipped, keep the existing failed-attempt warning path with useful provider/model/reason context, route warnings through `logWarn`, and add focused Vitest coverage for failed, no-provider, and all-skipped selected-image zero-output cases.
