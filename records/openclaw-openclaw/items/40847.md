## Summary

Keep open. The observability direction is useful, but the PR is not merge-ready because one warning branch is unreachable, all-skipped zero-output image attempts can still produce no warning, and the new diagnostics bypass the repository logger.

## What This Changes

The PR adds warning logs in `src/media-understanding/runner.ts` when selected image attachments produce zero media-understanding outputs, including failed-attempt details and a no-provider message.

## Best Possible Solution

Implement the diagnostic at the actual runner decision points: warn before the no-provider early return, warn when selected image attempts all skip or return empty output, keep failed-attempt context, route warnings through `logWarn`, and cover each path with focused media-understanding tests while preserving successful image handling and model selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Move the no-provider warning before the early return:** `src/media-understanding/runner.ts:817-821`
  - body: When no providers resolve, `runCapability` returns at the earlier `resolvedEntries.length === 0` guard, so this new branch is unreachable. Operators still get no warning for selected images with no image-understanding provider.
  - confidence: 0.94

  - body: If each resolved image provider returns `skipped`, `failedAttempts` is empty and `allAttempts` is non-empty, so neither warning branch runs. That leaves the zero-output path this PR targets silent.
  - confidence: 0.92
- **[P3] Use logWarn for the new diagnostics:** `src/media-understanding/runner.ts:811`
  - body: The new warnings call `console.warn` directly while existing media-understanding failures use `logWarn`, which also routes messages through OpenClaw logging. These diagnostics may be missed by operators reading normal gateway logs.
  - confidence: 0.82
