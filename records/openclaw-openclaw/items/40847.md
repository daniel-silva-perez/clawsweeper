## Summary

Keep open. Current main still lacks the requested diagnostics for selected image attachments that produce zero outputs, but this PR is not merge-ready because its warning block misses the no-provider and all-skipped paths called out in review.

## What This Changes

The PR adds post-processing warnings in `src/media-understanding/runner.ts` when image media-understanding selects attachments but produces no outputs.

## Best Possible Solution

Implement the diagnostic in the runner control flow itself: warn before the no-provider early return, warn when all selected-image attempts are skipped, preserve the existing failed-attempt warning path with sanitized provider/model/reason context, route warnings through `logWarn`, and add focused Vitest coverage for failed, no-provider, and all-skipped selected-image zero-output cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Move the no-provider warning before the early return:** `src/media-understanding/runner.ts:817-821`
  - body: When no media entries resolve, `runCapability` returns before this new block executes, so `allAttempts.length === 0` is never reached for the no-provider scenario it is meant to diagnose. Operators would still get no warning when selected image attachments cannot resolve any image-understanding provider.
  - confidence: 0.94
- **[P2] Warn when every selected image attempt is skipped:** `src/media-understanding/runner.ts:801-821`
  - body: If providers resolve but every attempt returns `skipped` due to empty output or a skip error, `failedAttempts` is empty and `allAttempts` is non-empty, so neither branch logs anything. That leaves a selected-image zero-output path silent, which is the core diagnostic gap this PR is trying to close.
  - confidence: 0.92
