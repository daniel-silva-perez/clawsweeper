## Summary

Keep open. Current main does not implement `x-openclaw-workspace` or an equivalent per-request workspace override on `/v1/chat/completions`. The OpenAI-compatible handler passes model, session, channel, prompt, image, and trust metadata into `agentCommandFromIngress`, but no workspace field; docs only list model/agent/session/message-channel headers. The only current workspace override path is internal spawned-run metadata, explicitly limited to spawned subagent runs. The recent KinthAI comment and earlier KeepMyClaw context support that this is an active multi-tenant deployment need, not a stale or duplicate cleanup candidate.

## Best Possible Solution

Keep this issue open for a deliberate core Gateway design. The likely path is an opt-in OpenAI-compatible HTTP workspace override with trusted-proxy/header-stripping guidance, strict absolute path and allowlist/tenant-root validation, tests for spoofing and session isolation, and docs under the public OpenAI chat completions API page. The implementation should pass the validated workspace into the existing ingress agent run path without weakening the internal spawned-subagent boundary.

## Risks / Open Questions

- A naive header-to-path implementation would be security-sensitive: it could let a caller choose arbitrary filesystem-backed memory/tool context unless gated by explicit config, trusted ingress semantics, path validation/allowlists, and clear interaction with sessions/transcripts.
- The existing internal spawned-run workspace override may look similar but is not an external HTTP contract; closing this as implemented would hide a real missing gateway feature.
