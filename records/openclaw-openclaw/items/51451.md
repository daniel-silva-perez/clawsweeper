## Summary

Keep this issue open. Current main still does not implement `x-openclaw-workspace` or an equivalent per-request workspace override for `/v1/chat/completions`; the nearby workspace override is internal spawned-subagent plumbing and is not an external HTTP contract. The KeepMyClaw and KinthAI comments add useful real-world multi-tenant context, including the spoofing concern, so this needs maintainer design rather than cleanup closure.

## Best Possible Solution

Keep this open for a deliberate core Gateway design. The likely fix belongs in the OpenAI-compatible HTTP ingress path: add an opt-in workspace override contract, validate/allowlist paths, document trusted proxy spoofing requirements, and add tests proving per-request workspace/session isolation. The current internal spawned-subagent workspace inheritance should remain internal unless maintainers intentionally promote it into a safe external HTTP contract.

## Risks / Open Questions

- A naive `x-openclaw-workspace` implementation would be security-sensitive: without explicit opt-in, trusted-proxy/header-stripping guidance, absolute-path validation, and tenant-root allowlisting, it could let one authenticated caller route a run into another tenant's workspace.
