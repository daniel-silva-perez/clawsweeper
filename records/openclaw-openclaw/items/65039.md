## Summary

Keep open. Current main still has the same pre-Ready Discord voice join failure surface: the Discord plugin forwards `daveEncryption=false` into `@discordjs/voice`, waits only for the Ready state, destroys the connection on `entersState` failure, and the existing DAVE receive recovery is post-Ready only. Related open reports still overlap, while the merged receive/reply recovery PR covers a later stage of the pipeline.

## What This Changes

The issue reports that Discord `/vc join` aborts before the voice connection reaches Ready when DAVE is disabled, after the voice WebSocket receives Hello but before UDP handshaking.

## Best Possible Solution

Make the Discord voice join policy explicit in the Discord plugin: either support DAVE-disabled joins reliably, reject or warn on unsupported non-DAVE joins before starting the handshake, or add a documented configurable Ready timeout/retry path. Keep any fix scoped to the Discord voice manager/config/docs/tests and treat post-Ready receive recovery as a separate surface.

## Reproduction Assessment

Yes. The issue includes concrete live steps and a high-signal trace, and current main still has a source-level reproduction path where an `@discordjs/voice` pre-Ready abort produces the reported `Failed to join voice channel` response.

## Solution Assessment

Unclear. There is no proposed patch, and a timeout knob alone may help #63098 but may not be the best answer for the DAVE-disabled Hello-before-Ready close; maintainers should first decide the supported behavior for non-DAVE joins.
