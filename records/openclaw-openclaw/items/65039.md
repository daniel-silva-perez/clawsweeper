## Summary

Current main still matches the reported Discord voice pre-Ready abort surface. The `/vc join` path forwards `daveEncryption` into `@discordjs/voice`, waits for `VoiceConnectionStatus.Ready` with a fixed 15s timeout, destroys the connection on any readiness failure, and only records a voice session after Ready succeeds. Existing DAVE receive recovery is post-Ready and does not address a voice WebSocket close after Hello before UDP handshaking. The related #63098 remains relevant, but the provided context does not show maintainer consensus that it is canonical for this more specific DAVE-disabled Hello-before-Ready trace.

## Best Possible Solution

Keep this issue open until the intended non-DAVE Discord voice behavior is decided. The likely implementation belongs in `extensions/discord/src/voice/manager.ts`: add explicit pre-Ready state/close diagnostics and regression coverage around `entersState` failure before `sessions.set`, then implement bounded retry or DAVE-enabled fallback, or reject/deprecate `channels.discord.voice.daveEncryption=false` with clear operator guidance if Discord now requires DAVE. If #63098 becomes the canonical tracker, copy this issue's Hello-before-Ready and DAVE-disabled trace there before closing this one as duplicate.
