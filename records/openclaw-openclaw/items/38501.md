## Summary

Keep open: current main still routes TUI Ctrl+C through clear/warn/exit handling without checking active runs, while Esc and `/abort` use the separate abort path. The issue also has an open focused closing PR plus a broader open Ctrl+C behavior PR, so it should remain open for maintainer review rather than be closed or queued as another repair.

## What This Changes

The issue asks for active-run-aware TUI Ctrl+C feedback that says the run is still active and points users to Esc or `/abort`, without changing run-cancellation semantics.

## Best Possible Solution

Choose the intended TUI Ctrl+C policy and land one matching patch with focused regression coverage. If Ctrl+C remains clear/exit, #38502 or equivalent active-run hint text should close this after merge; if Ctrl+C should abort active runs, #44223 needs maintainer product review plus aligned tests and docs.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: during an active run, Ctrl+C reaches `handleCtrlC` and `resolveCtrlCAction`, which ignore `activeChatRunId`, while Esc and `/abort` route through `abortActive`.

## Solution Assessment

Unclear pending maintainer policy. The narrowest maintainable fix for the reported UX mismatch is the focused status-text hint, while changing Ctrl+C to abort active runs is a broader behavior change that should be reviewed deliberately.
