## Summary

Keep open: current main still routes TUI Ctrl+C through clear/warn/exit feedback without checking the active run, while Escape and `/abort` use the separate abort path. The issue is paired with open focused PR #38502 and broader behavior PR #44223, so the right next step is maintainer review of those patches rather than cleanup closure or a new repair lane.

## What This Changes

Request clearer TUI Ctrl+C feedback during an active run so users know the run is still active and can stop it with Esc or `/abort`.

## Best Possible Solution

Choose the intended TUI Ctrl+C policy and land one matching patch with focused regression coverage. If Ctrl+C remains clear/exit, the status text should explicitly say the run is still active and point to Esc and `/abort`; if Ctrl+C should abort active runs, reconcile #44223 with docs and tests before landing.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: during an active run, Ctrl+C reaches `handleCtrlC`, which does not inspect `activeChatRunId` and only clears input, warns, or exits.

## Solution Assessment

Unclear pending maintainer policy. The narrowest maintainable fix for the reported UX mismatch is the text-only active-run hint, while #44223 intentionally changes keybinding semantics and needs product review.
