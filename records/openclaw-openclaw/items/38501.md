## Summary

Keep open: current main still routes TUI Ctrl+C through clear/warn/exit status handling without checking active run state, while Escape and `/abort` use the separate abort path. The issue is also paired with open closing PR #38502 and the broader open PR #44223, so it should remain open for maintainer review rather than be closed or queued as a separate repair.

## What This Changes

The issue asks for TUI Ctrl+C feedback during an active run to state that the run is still active and that Esc or `/abort` stops it, without changing backend abort semantics.

## Best Possible Solution

Choose the intended TUI Ctrl+C policy and land one matching patch with focused regression coverage. If Ctrl+C remains a clear/exit shortcut, the status text should explicitly mention the active run and the supported stop path; if Ctrl+C should abort, docs and tests should be updated to match that behavior.
