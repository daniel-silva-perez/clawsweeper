## Summary

Keep open: current main still treats TUI Ctrl+C as clear/warn/exit feedback without checking the active run state or showing the requested Esc/`/abort` hint. The issue is also paired with an open focused fix PR and a broader open behavior PR, so this should stay with maintainer review rather than be closed or queued as duplicate automation.

## What This Changes

The issue requests clearer TUI Ctrl+C status text when a chat run remains active, so users know Esc or `/abort` is required to stop it.

## Best Possible Solution

Choose the intended TUI Ctrl+C policy and land one matching patch with focused regression coverage. If Ctrl+C remains clear/exit, the focused status hint from #38502 or equivalent should close this after merge; if Ctrl+C should abort active runs, reconcile the broader #44223 behavior with docs and tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: during an active run, Ctrl+C routes through `handleCtrlC`, which only clear/warn/exits and never checks `activeChatRunId` or shows the requested hint.

## Solution Assessment

Unclear pending maintainer policy. The narrowest solution for the reported UX mismatch is the text-only active-run hint, while #44223 changes Ctrl+C semantics and needs explicit product review.
