## Summary

Keep open: current main still routes TUI Ctrl+C through clear/warn/exit feedback without active-run awareness, while Esc and `/abort` use the separate abort path. The issue is also paired with open closing PR #38502 and broader open behavior PR #44223, so maintainer review of those patches is the right path rather than cleanup closure or another repair lane.

## What This Changes

The issue requests clearer TUI Ctrl+C feedback during an active run so users know the run is still active and can use Esc or `/abort` to stop it.

## Best Possible Solution

Choose the intended TUI Ctrl+C policy and land one matching patch with focused regression coverage. If Ctrl+C remains clear/exit, #38502 or an equivalent text-only fix should include the active-run hint and align with the Esc and `/abort` docs; if Ctrl+C should abort active runs, #44223 needs product review plus matching docs and tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: during an active run, Ctrl+C reaches `handleCtrlC`, which does not inspect `activeChatRunId`, while Esc and `/abort` route to `abortActive`.

## Solution Assessment

Unclear pending maintainer policy. The narrowest solution for the reported UX mismatch is the focused status-text hint, while #44223 intentionally changes keybinding semantics and should be reviewed as a broader behavior change.
