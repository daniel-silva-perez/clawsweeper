## Summary

Keep this PR open. Current main already has the schema/config-validation half from #71247, but the embedded runner still does not enforce `agents.list[].contextTokens` in the runtime budget resolver, direct or queued compaction, context-pruning, or compaction-safeguard setup. The PR is useful source work but is incomplete as proposed because it only threads `agentId` through one run path and has an override-precedence bug called out in review discussion.

## What This Changes

The PR adds `agents.list[].contextTokens` schema support, updates `resolveContextWindowInfo` to accept an optional agent id, passes that id into the embedded run resolver, and adds two context-window guard tests for per-agent/default precedence.

## Best Possible Solution

Land a corrected, rebased implementation that treats `agents.list[].contextTokens` as the authoritative agent-level runtime cap, threads the resolved session agent id through every embedded run and compaction budget path, and leaves the already-shipped #71247 schema work intact.

## Reproduction Assessment

Yes. A code-level reproduction is clear: with `agents.defaults.contextTokens` set lower than an agent-specific value, the PR resolver falls through to the global default, and current compaction/extension call sites have no way to pass the active agent id at all.

## Solution Assessment

No. The direction is right, but the proposed patch is not the best complete solution because it updates only one runtime call site and does not make the per-agent override authoritative over the global default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread agent ids through all context budget paths:** `src/agents/pi-embedded-runner/run.ts:392`
  - body: Adding `agentId` only to this resolver call leaves compaction and extension factories on the old agent-blind path. Direct/queued compaction, context-pruning, and compaction-safeguard setup still call `resolveContextWindowInfo` without the active agent id, so the per-agent cap is ignored in the places that most need the budget.
  - confidence: 0.88
- **[P2] Treat per-agent contextTokens as authoritative:** `src/agents/context-window-guard.ts:50-52`
  - body: When an agent has `contextTokens` but that value is not lower than the model base window, this falls through and can still apply a stricter `agents.defaults.contextTokens`. For example, defaults=100k, agent=196k, model=180k still resolves to 100k, so the per-agent override cannot raise or disable a global cap. Resolve the matching agent value first, capped by the model/config base window, and skip the global default when present.
  - confidence: 0.9
