## Summary

Keep this PR open. Current main has already gained the schema/config half of per-agent `contextTokens`, but the runtime resolver is still agent-blind; the proposed branch moves in the right direction but would still ignore per-agent caps in compaction/extension budget paths and mishandles per-agent precedence over global defaults.

## What This Changes

The PR adds `agents.list[].contextTokens` schema support, extends the context-window resolver with an optional agent id, passes that id into one embedded runner resolver call, and adds resolver tests for per-agent/default precedence.

## Best Possible Solution

Land a corrected, rebased implementation that keeps the already-shipped schema support, treats `agents.list[].contextTokens` as the authoritative per-agent runtime cap before falling back to `agents.defaults.contextTokens`, and threads the resolved session agent id through main run, direct/queued compaction, context-pruning, and compaction-safeguard budget setup.

## Reproduction Assessment

Yes. A code-level reproduction is clear: with `agents.defaults.contextTokens` lower than an agent-specific value, the PR resolver can still return the global default; current main also shows compaction and extension budget paths have no agent-id input at all.

## Solution Assessment

No. The requested direction is maintainable, but the proposed patch is not the best complete fix because it updates only one runtime path and does not make the per-agent value authoritative over the global default.

  - confidence: 0.9
