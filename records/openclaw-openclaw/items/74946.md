## Summary

Keep open. The PR is aimed at a real linked runtime bug, but the patch only adds a model-catalog planning alias and does not make `moonshotai/<model>` resolve through the direct Moonshot runtime path on current main.

## What This Changes

The PR adds a `moonshotai` `modelCatalog.aliases` entry to the Moonshot plugin manifest and a manifest-planner test for provider-filtered alias catalog rows.

## Best Possible Solution

Amend this PR so `moonshotai` and likely `moonshot-ai` canonicalize to the existing `moonshot` provider in the shared runtime provider-id path, with direct model-resolution regression coverage. Keep `moonshot` as the canonical plugin/provider/auth/config id; the manifest catalog alias can remain as listing UX if maintainers want that surface too.

## Reproduction Assessment

Yes. The linked issue gives concrete config/logs, and current main statically reproduces the failure path: `moonshotai` is not normalized to `moonshot`, then runtime lookup searches the registry under the unchanged provider id and can emit `Unknown model: moonshotai/kimi-k2.6`.

## Solution Assessment

No. A manifest catalog alias is useful for `models list --provider moonshotai`, but it is not the best fix for the reported runtime failure; the narrow maintainable fix is provider-id canonicalization plus a runtime resolution test.
