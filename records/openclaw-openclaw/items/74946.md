## Summary

Keep open: the PR adds a useful manifest-catalog alias path, but it does not fix the reported direct runtime model resolution failure for `moonshotai/kimi-k2.6`. Current main still lacks a Moonshot alias in provider normalization, and the runtime resolver keeps the parsed provider id when looking up registry rows.

## What This Changes

This PR adds a Moonshot `modelCatalog.aliases.moonshotai -> moonshot` manifest entry and a manifest-planner unit test for alias-filtered catalog rows.

## Best Possible Solution

Amend this PR so `moonshotai/<model>` canonicalizes to the existing `moonshot` provider on the runtime provider-id path, with regression coverage for direct model parsing and model-registry resolution. Keep `moonshot` as the canonical plugin/provider/auth/config id, and keep the manifest catalog alias only if maintainers also want alias-filtered model listings.

## Reproduction Assessment

Yes. The linked issue gives concrete steps, and the current source confirms the static path: `parseModelRef` leaves `moonshotai` as the provider because `normalizeProviderId` has no Moonshot alias, then runtime resolution looks under that unmatched provider id.

## Solution Assessment

No. The PR solves only the manifest catalog-planner slice; the narrow maintainable fix is a runtime provider-id alias with tests that exercise direct model-ref parsing or `resolveModelWithRegistry`.
