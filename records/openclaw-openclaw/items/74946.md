## Summary

Keep open. The PR is a reasonable catalog-planning change, but it does not fix the reported direct runtime failure because current runtime provider normalization still leaves `moonshotai` as its own provider id, so `moonshotai/kimi-k2.6` can still resolve to `Unknown model`.

## What This Changes

The PR adds a `moonshotai -> moonshot` `modelCatalog.aliases` entry to the Moonshot plugin manifest and a manifest-planner unit test for provider-filtered alias catalog rows.

## Best Possible Solution

Amend the PR so `moonshotai/<model>` and likely `moonshot-ai/<model>` canonicalize to the existing `moonshot` provider on the runtime provider-id path, with regression coverage for direct model parsing/resolution. Keep `moonshot` as the canonical plugin/provider/auth/config id, and keep the catalog alias only if maintainers want alias-filtered model listings too.

## Reproduction Assessment

Yes. The linked issue gives concrete steps, and the current source confirms the static path: `moonshotai/kimi-k2.6` parses to provider `moonshotai`, `normalizeProviderId` does not map it to `moonshot`, and runtime resolution continues under that unmatched provider id.

## Solution Assessment

No. This PR solves only the manifest catalog-planner slice; the narrow maintainable fix is a runtime provider-id alias with tests that exercise direct model-ref parsing or `resolveModelWithRegistry`.
