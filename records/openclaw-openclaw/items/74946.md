## Summary

Keep open. The PR addresses the right user report, but the patch only adds a manifest model-catalog alias and planner coverage; the runtime path that throws `Unknown model: moonshotai/kimi-k2.6` still resolves by provider id and current Moonshot runtime/provider normalization does not alias `moonshotai` to `moonshot`.

## What This Changes

The PR adds a `moonshotai` `modelCatalog.aliases` entry to the Moonshot plugin manifest and a manifest-planner test for provider-filtered alias catalog rows.

## Best Possible Solution

Amend this PR so the direct runtime ref `moonshotai/<model>` canonicalizes to the existing `moonshot` provider without renaming the plugin/provider/auth/config surface. A focused `normalizeProviderId` alias with runtime-level regression coverage is the narrowest maintainable fix; the manifest catalog alias can remain only if maintainers want `models list --provider moonshotai` to show alias rows too.

## Reproduction Assessment

Yes. The linked issue provides concrete steps, and the current source path gives a high-confidence static reproduction: `moonshotai/kimi-k2.6` is parsed as provider `moonshotai`, current `normalizeProviderId` does not map it to `moonshot`, and `resolveModelWithRegistry` searches under that unchanged provider id.

## Solution Assessment

No. This PR solves a catalog-planner slice but not the reported runtime failure; the safer fix is to canonicalize `moonshotai` and likely `moonshot-ai` to `moonshot` in the shared provider-id normalization path, with a regression that exercises direct model resolution or the model-ref parsing path.
