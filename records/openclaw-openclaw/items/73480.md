## Summary

Keep open. Current main still distinguishes recovery-key backup usability from full Matrix identity trust, still has coverage for the accepted-no/backup-usable diagnostic state, and does not contain the proposed `/keys/upload` OTK-collision mitigation; related Matrix PRs remain implementation candidates rather than merged proof this regression is fixed.

## What This Changes

The issue reports Matrix E2EE recovery failing on OpenClaw 2026.4.26, where `verify device --recovery-key-stdin` leaves the device untrusted and `matrix encryption setup` hits a Matrix `/keys/upload` one-time-key collision.

## Best Possible Solution

Keep this open as the Matrix E2EE recovery umbrella until maintainers review and land or replace the focused Matrix fixes, then verify the reported `verify device --recovery-key-stdin` and `matrix encryption setup` sequence against an affected or disposable Matrix account. The durable end state should handle or clearly report `/keys/upload` OTK collisions and make recovery-key diagnostics distinguish rejected keys, usable backup material, and missing cross-signing trust.
