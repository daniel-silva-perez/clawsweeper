## Summary

Keep open. Current main still preserves the Matrix recovery-key versus full identity-trust split, still has an explicit unit test for the reported `recoveryKeyAccepted: false` plus `backupUsable: true` state, and does not contain the `/keys/upload` one-time-key collision mitigation described in active related PR #74529. Related PRs #74529 and #74509 are useful implementation candidates, but they are still open in the provided context rather than merged proof that this issue is fixed.

## What This Changes

The issue reports a Matrix E2EE recovery regression where `verify device --recovery-key-stdin` leaves the device without full identity trust and `matrix encryption setup` fails on a Matrix `/keys/upload` one-time-key collision.

## Best Possible Solution

Keep this issue open as the Matrix E2EE recovery umbrella until maintainers review and land or replace the focused Matrix fixes. The durable end state should handle or clearly report `/keys/upload` OTK collisions and make recovery-key diagnostics distinguish rejected keys, already-usable backup material, and missing cross-signing trust without sending the operator back through an ineffective recovery-key loop.

Unclear for a full live reproduction: the issue gives concrete commands and current main has unit-level coverage for the accepted-no/backup-usable diagnostic state, but this review did not run against a live affected Matrix account or homeserver.
