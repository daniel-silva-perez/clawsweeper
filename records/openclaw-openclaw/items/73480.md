## Summary

Keep open. This is a fresh Matrix regression report with concrete command output, and current main still has the same recovery-key/device-trust behavior plus no relevant recovery-path fix after v2026.4.26. The reported `Recovery key accepted: no` with `Backup usable: yes` is also an explicitly modeled state in current tests, not evidence that the user problem is resolved.

## Best Possible Solution

Keep the issue open for Matrix maintainer follow-up. Reproduce the Windows/Docker `verify device --recovery-key-stdin` and `matrix encryption setup` sequence on current main, then fix either the recovery-key acceptance logic, the CLI guidance around `recoveryKeyAccepted: false` with `backupUsable: true`, or the setup bootstrap handling for Matrix `keys/upload` one-time-key conflicts. If the current behavior is intended for invalid recovery keys plus previously cached backup material, the CLI should say that clearly and point to the next repair step instead of appearing to reject a usable recovery key.
