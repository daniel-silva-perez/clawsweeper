## Summary

Keep open. Current main still routes Matrix recovery-key device verification through the same behavior reported from v2026.4.26: the recovery key path can fail until full cross-signing identity trust is established, and the CLI can print the same retry guidance. The report is fresh, has concrete logs, and includes a potentially contradictory state (`Recovery key accepted: no` with `Backup usable: yes`) plus a Matrix `keys/upload` one-time-key conflict from `matrix encryption setup`; there is no current-main recovery-path fix or clearly canonical duplicate in the provided related context.

## Best Possible Solution

Keep the issue open for Matrix-owner triage. Reproduce the Windows/Docker `verify device --recovery-key-stdin` and `matrix encryption setup` sequence on current main, then fix the recovery flow or CLI guidance around the `recoveryKeyAccepted: false` / `backupUsable: true` state. Also inspect whether the `keys/upload` one-time-key conflict should be handled idempotently during `matrix encryption setup` or reported with a clearer repair path.
