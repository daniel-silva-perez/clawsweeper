## Summary

Keep open. Current main still preserves the Matrix recovery-key/device-trust split behind this report, including the `recoveryKeyAccepted: false` with `backupUsable: true` state, and I did not find a shipped or current-main fix for the reported setup bootstrap one-time-key conflict.

## What This Changes

The issue reports that `openclaw matrix verify device --recovery-key-stdin` cannot recover a Matrix device with a recovery key on OpenClaw 2026.4.26, and that `openclaw matrix encryption setup` then fails during Matrix crypto bootstrap with a one-time-key upload conflict.

## Best Possible Solution

Keep this with Matrix maintainers. Reproduce the reported `verify device --recovery-key-stdin` and `matrix encryption setup` sequence on current main, then make the narrowest fix in the Matrix plugin: either accept and persist a truly validated recovery key, give clearer guidance for the accepted-no/usable-yes state, or handle the Matrix one-time-key bootstrap conflict without leaving users in a recovery loop.
