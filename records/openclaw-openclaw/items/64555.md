## Summary

Keep this open. Current main and the latest release still expose the WhatsApp default web auth directory as an eagerly resolved value and still resolve account auth roots from ambient process env, while the target-repo fix PR is closed unmerged and the fork candidate has not landed in openclaw/openclaw.

## What This Changes

The issue reports that WhatsApp Baileys credentials can cross `--profile` boundaries by resolving under the default `~/.openclaw/credentials/whatsapp/default` directory instead of the active profile state directory.

## Best Possible Solution

Land or recreate a focused WhatsApp fix that makes the default web auth dir lazy, threads explicit env/state-dir resolution through account discovery/default/legacy auth-dir helpers, and adds a regression test proving the default Baileys auth path stays under `OPENCLAW_STATE_DIR` for named profiles. Doctor/docs guidance for `OPENCLAW_OAUTH_DIR` can follow after the credential isolation path is corrected.

## Reproduction Assessment

Yes. The issue gives concrete profile-gateway steps and logs, and current source inspection still shows the default and account auth paths resolving from ambient OAuth state rather than an explicit active profile state dir.

## Solution Assessment

Yes. A plugin-owned path-resolution fix with explicit env/state threading and a narrow regression test is the smallest maintainable correction; broad core migration or docs-only workaround would leave the credential isolation bug in place.
