## Summary

Keep this issue open. Current main still has the implicated WhatsApp default auth export resolved eagerly at module evaluation time and the account auth-dir helpers still resolve the OAuth root from ambient process env instead of an explicit profile env/state-dir seam. The target-repo fix PR is closed unmerged, and the open external fork candidate has not landed in openclaw/openclaw.

## What This Changes

Ensure WhatsApp Baileys credentials resolve under the active `--profile` state directory instead of the default OpenClaw credentials directory.

## Best Possible Solution

Land or recreate a focused WhatsApp fix that makes the default web auth dir lazy or otherwise runtime-bound, threads explicit env/state-dir resolution through account discovery/default/legacy auth-dir helpers, and adds a regression test proving `OPENCLAW_STATE_DIR` profile isolation for the default Baileys auth path. Doctor/docs guidance for `OPENCLAW_OAUTH_DIR` can follow after the core credential isolation path is corrected.

## Security Review

Status: needs_attention

Summary: This issue reports a concrete credential/session isolation risk, and current main still contains the implicated path-resolution patterns.

Concerns:

- **[high] WhatsApp auth can resolve outside the active profile:** `extensions/whatsapp/src/auth-store.ts:39`
  - body: `WA_WEB_AUTH_DIR` is still initialized from `resolveOAuthDir()` at module evaluation time, so a load before the intended profile env is authoritative can pin the default credentials path and undermine profile isolation.
  - confidence: 0.84

  - body: `listWhatsAppAuthDirs()` and default/legacy account auth-dir resolution still call `resolveOAuthDir()` without an explicit env or state dir, leaving discovery coupled to ambient process env instead of the active profile boundary.
  - confidence: 0.82
