## Summary

Keep this PR open. Current main shipped a different mitigation for the linked FD-exhaustion report, but it still does not implement this PR's explicit chokidar `depth` hardening, and the memory-side target moved from the old core file into the bundled `memory-core` plugin. The PR is stale as-is, but the underlying hardening question remains actionable.

## Best Possible Solution

Keep this PR open as a focused hardening candidate, or replace it with a rebased patch that updates `src/agents/skills/refresh.ts`, `extensions/memory-core/src/memory/manager-sync-ops.ts`, and possibly `extensions/memory-core/src/memory/qmd-manager.ts`. Maintainers should choose the intended depth and symlink contract, preserve or update the documented recursive extraPaths behavior, and add regression tests that assert the chokidar options.

## Risks / Open Questions

- Closing this PR now would lose a focused hardening candidate while current main still lacks the explicit `depth` behavior the PR proposes.
- The PR is stale against current main because the memory implementation moved into `extensions/memory-core`; applying it directly would miss the active memory watcher.
- A naive memory depth limit could violate the documented recursive `memorySearch.extraPaths` contract or silently leave deeply nested notes stale.
- Current tests cover watched paths, filters, debounce behavior, and watch-triggered sync, but they do not assert depth or symlink options for skills, builtin memory, or QMD watchers.
- Security review found no supply-chain-sensitive diff in this PR; the remaining security concern is availability hardening around recursive watchers, broad extraPaths, and symlink policy.
