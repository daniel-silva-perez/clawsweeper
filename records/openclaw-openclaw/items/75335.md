## Summary

Keep open. The PR is a narrow and plausible behavior change for `music_generate`, but it is not complete as submitted because it makes music completions ignore the global async media direct-send option while leaving config help/user-facing guidance saying that option applies to async music and video tasks. The `maintainer` label also means this should stay in the maintainer review path rather than cleanup close handling.

## What This Changes

The PR adds a per-media-lifecycle direct completion delivery mode, disables direct completion delivery for async music generation, and changes music background tests to expect agent-mediated completion even when global direct send is enabled.

## Best Possible Solution

Keep the behavioral split only after aligning the public/config contract: async music completions should remain agent-mediated, while `tools.media.asyncCompletion.directSend` should either be documented as video-only or described as applying only to media lifecycles that opt into direct completion delivery.

## Reproduction Assessment

Yes. Current main has a focused unit-test path: enable `directSend` in `music-generate-background.test.ts` and observe direct `sendMessage` delivery; the PR flips that same scenario to expect subagent announcement delivery instead.

## Solution Assessment

No, not as submitted. The code-level approach is narrow and preserves video's existing config-gated direct delivery, but the config help and user-facing contract need to be updated so users are not told the option still controls music direct sends.
