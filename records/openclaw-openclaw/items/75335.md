## Summary

Keep open. The code change is narrow and matches the intended music behavior, but it still leaves the public config/docs contract saying `directSend` controls async music completion, and the PR has the protected `maintainer` label so cleanup should not close it.

## What This Changes

The PR adds a per-media-lifecycle direct completion delivery setting, disables direct channel delivery for async music completions, and updates the music background test to expect agent-mediated delivery even when global direct send is enabled.

## Best Possible Solution

Keep the per-lifecycle delivery split, but align the public contract: async music completions should remain agent-mediated, while `tools.media.asyncCompletion.directSend` should be documented as video-only or as applying only to media lifecycles that opt into direct completion delivery.

## Reproduction Assessment

Yes. The focused current-main path is `music-generate-background.test.ts` with `directSend: true`: it expects direct `sendMessage` delivery today, while the PR changes that same scenario to avoid `sendMessage` and use the announcement wake path.

## Solution Assessment

No, not as submitted. The code seam is the narrowest maintainable way to preserve video direct sends while opting music out, but the config help, user docs, generated schema text, and changelog need to match the new contract.
