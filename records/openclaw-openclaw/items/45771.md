## Summary

Keep open. Current main has normalized provider usage snapshots, status visibility, and static subagent fan-out guards, but it still lacks the requested pace-aware policy: response-header capture, projected 5h/7d tiers, persisted per-profile pace state, config-driven prompt/status/session integration, and autonomous throttling. The related rate-limit, load-balancing, and failover issues overlap, but they do not fully supersede this autonomous-agent pace-control request.

## What This Changes

Request built-in pace-aware rate limiting for autonomous agents by capturing Anthropic usage windows, projecting consumption pace, surfacing status/session data, and throttling subagent or non-interactive work through config.

## Best Possible Solution

Keep this as a design and implementation tracker for a provider-aware pace model: structured usage source/freshness, per-profile persisted windows, observed request-cost calibration, status/session output, prompt context, notifications, and explicit subagent or non-interactive enforcement rules that keep Anthropic-specific behavior provider-owned where possible.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main source and docs inspection, which shows usage snapshots and static subagent limits exist but the requested pace policy surfaces do not.

## Solution Assessment

Unclear as a full proposal. Header capture plus status visibility is a reasonable first slice, but automatic blocking should be designed around provider windows, observed request cost, and careful autonomous-agent UX before becoming enforcement policy.
