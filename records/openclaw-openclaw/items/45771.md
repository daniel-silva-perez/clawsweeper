## Summary

Keep open. Current main has normalized provider usage snapshots, status visibility, and static subagent fan-out guards, but it still lacks the requested pace-aware autonomous-agent policy: Anthropic response-header capture, projected 5h/7d pace tiers, persisted per-profile pace state, config-driven prompt/status/session integration, notifications, and autonomous throttling. Related open issues #13615, #38775, and #71770 overlap on rate limits, usage-aware routing, and failover, but they do not fully supersede this specific pace-control design request.

## What This Changes

The issue requests built-in pace-aware rate-limit tracking for autonomous agents using Anthropic 5h/7d utilization data, projected tiers, status/session visibility, config, persistence, and subagent throttling.

## Best Possible Solution

Keep this as a maintainer design tracker for a provider-aware pace model: start with structured usage source/freshness and status/session prompt visibility, keep Anthropic-specific capture in the Anthropic provider path, and add explicit opt-in enforcement only after persistence, per-profile semantics, request-cost calibration, and subagent UX are specified.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source/docs inspection on current main, which shows usage snapshots and static subagent limits exist but the requested pace policy surfaces do not.

## Solution Assessment

Unclear for the full proposal. Header or usage snapshot capture plus status/session visibility is a reasonable first slice, but automatic blocking should be designed around provider windows, observed request cost, and autonomous-agent UX before becoming enforcement policy.
