## Summary

Current main has adjacent Claude/provider usage-window visibility, but it does not implement the requested pace-aware rate limiting: no Anthropic response-header capture, no pace tier/projection model, no persisted pace state, and no subagent/non-interactive throttle policy. The related open rate-limiting item covers broader throttling, but this issue still has a distinct Anthropic usage-window and agent self-regulation design surface.

## Best Possible Solution

Keep this issue open as a focused product/design item. A conservative implementation would extend the generic provider-usage model with pace/projection metadata, add Anthropic-specific capture or polling behind that generic seam, surface the result in status/session_status and prompt context, and coordinate any blocking/throttling policy with the broader rate-limiting work in #13615.
