## Summary

Keep open. Current main has normalized provider usage snapshots and subagent fan-out guards, but it still lacks the requested pace-aware autonomous-agent policy; the related rate-limit/load-balancing issues overlap, but they do not fully supersede this Anthropic/status/session/subagent request.

## What This Changes

The issue requests built-in pace-aware usage tracking for autonomous agents, including Anthropic utilization capture, projected pace tiers, status/session visibility, config, and autonomous throttling behavior.

## Best Possible Solution

Define a provider-aware pace model before implementation: structured usage sources and freshness, persisted per-profile windows, observed request-cost calibration, status/session output, prompt context, notifications, and explicit subagent/non-interactive enforcement rules that keep provider-specific Anthropic behavior in the provider-owned surface where possible.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source/docs inspection of current main, which shows usage snapshots and subagent limits exist but the requested pace policy surfaces do not.

## Solution Assessment

Unclear as proposed. Header capture plus status visibility is a reasonable first slice, but automatic blocking should be designed around provider windows plus observed request cost and a careful UX for autonomous work before it becomes enforcement policy.
