## Summary

Keep open. Current main has adjacent provider usage reporting for Anthropic OAuth usage windows and surfaces it in status/session_status, plus subagent fan-out guards, but it still does not implement the requested built-in pace-aware autonomous-agent policy: no Anthropic response-header capture into persistent runtime state, no projected pace tiers/actions, no agents.defaults.pace config, no pace prompt injection/notifications, and no proactive subagent or non-interactive throttling before model calls. Related open items #13615, #38775, and #71770 are useful design context but do not supersede this narrower Anthropic pace-aware autonomous-agent request.

## Best Possible Solution

Keep this open as a pace-aware autonomous-agent design item. The best path is to consolidate the design with #13615, #38775, and #71770, then decide whether pace state and enforcement are core-owned or exposed through a provider/plugin policy API. A complete design should cover Anthropic response headers versus usage polling, per-profile persistence, provider fallback behavior, status/session context, prompt injection, notifications, and explicit subagent/non-interactive throttling rules.
