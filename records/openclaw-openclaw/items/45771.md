## Summary

Keep open. Current main has adjacent provider usage-window reporting and subagent safety guards, but it still does not implement the requested pace-aware autonomous-agent policy: no Anthropic inference response-header capture, no persisted projected pace tiers, no `agents.defaults.pace` config, and no prompt or throttling policy for subagents/non-interactive work. The latest discussion also raises a real design concern that utilization percentage alone may miscalibrate pace across model and context-size costs.

## What This Changes

Request built-in pace-aware rate limiting for autonomous agents, including Anthropic usage header capture, projected 5h/7d pace tiers, status/session visibility, prompt context, notifications, and autonomous throttling controls.

## Best Possible Solution

Define a provider-aware pace design before implementation: combine Anthropic response headers and/or usage polling, per-profile persisted state, observed cost-per-request modeling, status and `session_status` display, agent prompt context, notifications, and explicit subagent/non-interactive enforcement rules without hard-coding provider policy into unrelated core paths.
