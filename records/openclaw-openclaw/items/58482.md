## Summary

Keep this PR open. Current main still has uncancelable memory batch polling for OpenAI, Voyage, and Gemini: the shared batch option types expose no AbortSignal, provider polling loops still use plain timers, and memory-core has no signal to pass into provider batch adapters. The PR is not implemented on main or clearly superseded, but it needs to be rebased onto the current provider-owned extension files and tightened so cancellation is wired through sleep, upload/create/status/download HTTP calls, and the memory-core call chain.

## Best Possible Solution

Keep the PR open and ask for a current-main rework: add an optional `AbortSignal` to the memory batch contract, propagate it from memory-core through provider adapters into OpenAI/Voyage/Gemini batch runners, use a cleanup-safe abortable sleep helper that handles already-aborted signals, pass the signal into every guarded HTTP request phase, and add focused tests for end-to-end abort behavior and listener cleanup. Keep unrelated UI or locale changes out of the branch.

## Risks / Open Questions

- The PR is stale against current main because provider batch implementations now live in provider plugins under `extensions/{openai,voyage,google}` rather than the package-level files shown in the original diff.

- Listener cleanup, already-aborted signals, and concurrency error semantics need regression tests before merge; otherwise the earlier review findings can reappear.
- Because this touches remote HTTP execution paths, maintainers should verify the final signal threading preserves `withRemoteHttpResponse` SSRF guards, retry cleanup, and release handling.
