## Summary

Keep open. Current main still routes OpenShell sandbox exec commands into the SSH `/bin/sh -c` wrapper without preflight for generated placeholders or malformed shell syntax, and generic same-tool/same-args loops still return warning-level results until the later global no-progress breaker. The related #58726 overlaps the loop-abort policy, but it does not cover the OpenShell command-validation gap.

## What This Changes

The issue asks OpenShell to reject obviously generated or malformed shell commands before execution and to hard-abort repeated failing or no-progress tool loops after the configured threshold.

## Best Possible Solution

Add a narrow OpenShell sandbox exec preflight before the SSH `/bin/sh -c` wrapper for obvious generated-command hazards, then escalate generic repeats to critical only when failed or stable no-progress evidence reaches the configured critical threshold. Keep broader loop policy discussion linked to #58726 if maintainers want a separate threshold/defaults decision.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: an OpenShell exec command like `workflow install <name>` or a command with an unclosed quote reaches `buildExecRemoteCommand` and `/bin/sh -c`; generic no-progress repeats at `CRITICAL_THRESHOLD` are currently asserted to stay warning-level.

## Solution Assessment

Yes. The issue’s proposed direction is the narrowest maintainable fix if the command validator is conservative and the loop change keys off failed or no-progress outcomes rather than raw repetition alone.
