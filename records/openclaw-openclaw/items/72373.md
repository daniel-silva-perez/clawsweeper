## Summary

Keep open. Current main still forwards OpenShell exec command text into the SSH `/bin/sh -c` wrapper without generated-command or malformed-shell preflight, and generic repeated tool calls remain warning-level until separate no-progress blockers apply. The related loop issue and open loop PRs overlap only the hard-abort half; they do not cover the OpenShell command-validation gap.

## What This Changes

The issue requests fail-fast OpenShell validation for generated or malformed shell commands plus hard-abort behavior for repeated identical tool-call loops.

## Best Possible Solution

Add a focused OpenShell/SSH exec command preflight before remote `/bin/sh -c` construction, with tests for the reported generated-command hazards, and reconcile the generic-repeat hard-abort behavior with the canonical #58726 loop-detection work rather than forking a separate policy direction.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a command such as `workflow install <name>` or an unclosed quote flows from `extensions/openshell/src/backend.ts` into `src/agents/sandbox/ssh.ts` and then `/bin/sh -c`, while enabled generic repeated calls stay warning-only in current loop tests below the global breaker.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-owned OpenShell/SSH exec preflight plus shared generic-repeat escalation coordinated through the existing loop-detection issue. A broad heuristic across all host exec modes or a separate default-on loop policy would be riskier and should be a maintainer decision.
