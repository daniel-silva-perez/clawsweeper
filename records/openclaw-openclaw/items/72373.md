## Summary

Keep open. Current main still forwards OpenShell exec commands into the SSH `/bin/sh -c` wrapper without malformed/generated-command preflight, and `generic_repeat` still remains warning-level at `criticalThreshold` until the separate global no-progress breaker. Related loop items and PR #60248 cover part of the loop-abort gap, but they do not cover the OpenShell command-validation gap, and the relevant PR has not merged.

## What This Changes

OpenShell should reject obviously malformed generated shell commands before shell execution and repeated identical failed or no-progress tool calls should hard-abort after the configured threshold.

## Best Possible Solution

Add a narrow OpenShell/SSH sandbox exec preflight before remote command construction for obvious generated-command hazards, and update loop detection so generic repeated failed or no-progress calls escalate to `critical` at `criticalThreshold` while progressing generic repeats remain warning-only. Keep broader defaults, exec approval semantics, and unrelated transcript repair out of this fix unless a maintainer explicitly expands scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a command such as `workflow install <name>` or an unclosed quote reaches `buildExecRemoteCommand` and `/bin/sh -c`, and current loop-detection tests expect generic no-progress repeats at `CRITICAL_THRESHOLD` to remain warning-level. No live OpenShell run was executed because this was a read-only review.

## Solution Assessment

Yes. The best path is a targeted OpenShell sandbox exec preflight plus no-progress-gated generic repeat escalation; broad heuristic filtering across every host exec path, enabling loop detection by default, or changing exec approval policy would be larger product/security changes than this report requires.

## Repair Work Prompt

Current main forwards OpenShell sandbox exec commands from `extensions/openshell/src/backend.ts` into `src/agents/sandbox/ssh.ts::buildExecRemoteCommand`, which wraps the raw body as `/bin/sh -c` without rejecting obvious generated or malformed commands. Add a conservative OpenShell/SSH sandbox exec preflight before remote command construction that rejects clearly non-runnable generated-command hazards such as literal angle-bracket placeholders (`<name>`, `<workflow-id>`), unclosed single/double quotes, trailing backslash escapes, and unterminated backticks or command substitutions where the local parser can detect them. Coordinate with #54559, #58726, and open PR #60248: if #60248 has landed, reuse/adapt that implementation; otherwise avoid duplicating broad changes from #62775. Do not enable loop detection by default, lower default thresholds, change exec approval semantics, or add unrelated transcript/error-turn stripping.

Add focused tests for OpenShell/SSH command preflight and loop detection/e2e blocking behavior. Update docs if the user-visible OpenShell/exec behavior changes. Add a changelog Fixes entry in the active version with appropriate reporter/source credit, including Thanks @Kaspnov and related loop reporters/authors when their issue/PR is covered; never credit @codex or @openclaw.
