## Summary

Keep open. Current main still lacks OpenShell/generated-command preflight before the SSH `/bin/sh -c` wrapper, and the current loop detector still leaves generic repeated same-tool/same-args calls at warning level until a later no-progress global breaker. The related #58726 overlaps the loop-abort part but does not cover malformed OpenShell command validation.

## What This Changes

Request to add OpenShell preflight validation for malformed/generated shell commands and to hard-abort repeated identical tool-call loops instead of allowing warning-only retries.

## Best Possible Solution

Add a shared sandbox exec preflight before `buildExecRemoteCommand` for obvious generated placeholders and malformed shell structure, then align loop policy so repeated failed or no-progress generic calls can escalate at the configured critical threshold while preserving warning-only behavior for progressing repeats. Keep #58726 as the broader loop-threshold tracking item if maintainers want to split that part.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: an OpenShell sandbox exec command containing `<workflow-id>` or unbalanced shell syntax flows into the SSH `/bin/sh -c` command body, and the loop-detection tests encode generic repeats as warning-only below the global breaker.

## Solution Assessment

No, current main is only a partial mitigation. The global no-progress breaker helps, but it fires later than `criticalThreshold` and does not prevent malformed OpenShell commands from reaching the shell; a safer fix is narrow command preflight plus failure/no-progress-gated escalation.
