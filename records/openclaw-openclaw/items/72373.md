## Summary

Keep open. Current main still lacks fail-fast generated-command validation on the OpenShell/SSH sandbox exec path, and the loop detector still keeps generic repeated same-tool/same-args calls at warning level until the later no-progress global breaker. The prior ClawSweeper comment correctly notes that #58726 overlaps the loop-abort half, but it does not cover malformed OpenShell command preflight.

## What This Changes

Request fail-fast validation for malformed/generated OpenShell exec commands and hard-abort behavior for repeated identical tool loops after the configured threshold.

## Best Possible Solution

Add a narrow shared sandbox exec preflight before the SSH /bin/sh -c wrapper for obvious generated-command hazards, then make repeated failed or no-progress generic tool calls escalate at the configured critical threshold while preserving warning-only behavior for progressing repeats.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: an OpenShell sandbox exec command containing a literal placeholder or unbalanced shell syntax reaches buildExecRemoteCommand and /bin/sh -c, and the loop tests encode generic repeats as warning-only below the global breaker.

## Solution Assessment

No, current main is only a partial mitigation. The safest repair is targeted command preflight plus failure/no-progress-gated loop escalation, with #58726 kept as the broader loop-threshold tracking item if maintainers want to split the work.
