## Summary

Keep open. Current main still has the reported gap: OpenShell and the generic SSH sandbox exec path pass generated command text into the shared remote `/bin/sh -c` wrapper without command-shape preflight, and generic repeated same-tool/same-args loops remain warning-only until the later global no-progress breaker. The loop-abort half overlaps #58726 and open PR #60248, but neither covers malformed OpenShell command preflight.

## What This Changes

The issue requests OpenShell command preflight validation for malformed generated shell commands and a hard abort for repeated identical tool loops.

## Best Possible Solution

Add a conservative shared SSH sandbox exec preflight before the `/bin/sh -c` wrapper used by OpenShell and the generic SSH backend, rejecting only obvious generated placeholders and parser-detectable malformed shell shapes. Then resolve the repeated-loop half through the no-progress-gated `generic_repeat` escalation already being developed in #60248 or a compatible narrow successor.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: an OpenShell sandbox exec command containing a literal generated placeholder or unbalanced shell syntax flows from `extensions/openshell/src/backend.ts` into `src/agents/sandbox/ssh.ts` and then `/bin/sh -c`, while generic repeated calls at `CRITICAL_THRESHOLD` are still asserted to remain warning-level in the unit tests.

## Solution Assessment

No. Current main is only a partial mitigation: known polling, ping-pong, unknown-tool, and later global no-progress loops can block, but the OpenShell command preflight is absent and generic repeats still do not escalate at the configured critical threshold before the global breaker.

## Repair Work Prompt

Implement fail-fast generated-command validation for the SSH/OpenShell sandbox exec path on current main. Add a shared conservative preflight before `buildExecRemoteCommand` hands command text to `/bin/sh -c`, covering both `src/agents/sandbox/ssh-backend.ts` and `extensions/openshell/src/backend.ts` through the shared helper/API. Reject obvious generated-command hazards from this report: literal angle-bracket placeholder tokens used as executable words or args such as `<name>`, `<workflow-id>`, and quoted `"<task>"`; unclosed single or double quotes; trailing backslash escapes; and unterminated backticks or command substitutions when they can be detected without a full shell parser. Keep the validation narrow: do not reject normal redirection (`< input.txt`, `2>file`), here-docs, ordinary comparison text, or valid shell syntax merely because it contains `<` or `>`.
