## Summary

Keep open. Current main still forwards OpenShell exec command text into the SSH `/bin/sh -c` wrapper without a generated-command or malformed-shell preflight, and the generic same-tool detector still returns warning-level results until the separate global no-progress breaker. The related #58726 overlaps the hard-abort half, but it does not cover the OpenShell command-validation gap.

## What This Changes

The issue requests OpenShell preflight rejection for malformed or generated-looking shell commands and hard-abort behavior for repeated identical tool-call loops.

## Best Possible Solution

Add a focused OpenShell/SSH sandbox exec preflight before building the remote `/bin/sh -c` command, covering the reported generated-command hazards with tests, and adjust loop detection so repeated identical failed or no-progress generic calls become critical at the configured threshold while progressing repeats remain non-blocking. Update the user-facing loop/exec docs and changelog for the changed runtime behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command like `workflow install <name>` or an unclosed quote flows from `extensions/openshell/src/backend.ts` into `buildExecRemoteCommand` and then `/bin/sh -c`, and existing loop tests assert generic repeats stay warning-only below the global breaker.

## Solution Assessment

Yes. The narrow maintainable solution is OpenShell/SSH sandbox exec command preflight plus no-progress-gated generic loop escalation; applying a broad heuristic to all host exec modes would have higher false-positive and policy risk.
