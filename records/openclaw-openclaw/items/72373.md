## Summary

Keep open. Current main still lacks an OpenShell command-string preflight for the reported generated/malformed command hazards, and generic same-tool repeats still remain warning-level until the separate global no-progress breaker. The open #58726 overlaps the loop-abort part, but it does not cover the OpenShell validation gap.

## What This Changes

OpenShell should reject clearly generated or malformed shell commands before remote shell execution and repeated identical failing tool loops should abort the turn after the configured threshold.

## Best Possible Solution

Add a focused OpenShell/SSH sandbox exec preflight before remote command construction, cover the reported generated-command hazards with unit tests, and make repeated identical failed or no-progress generic tool calls become critical at the configured threshold while preserving non-blocking behavior for progressing repeats.

## Reproduction Assessment

Yes. A static reproduction path is clear on current main: `workflow install <name>` or an unclosed quote flows from `extensions/openshell/src/backend.ts` into `src/agents/sandbox/ssh.ts` and then `/bin/sh -c`; loop tests also demonstrate generic repeats remain warn-only below the global breaker.

## Solution Assessment

Yes. The narrow maintainable fix is to validate OpenShell/SSH sandbox exec command strings before shell wrapping and adjust no-progress-gated generic repeat escalation, rather than adding a broad heuristic to all host exec modes.
