## Summary

Keep open. Current main still passes OpenShell exec command text through to the SSH `/bin/sh -c` wrapper without a generated-command or malformed-shell preflight. Loop detection has partial hard-abort coverage for known polling, ping-pong no-progress, unknown-tool repeats, and the later global no-progress breaker, but generic same-tool/same-args repeats still remain warning-level until the global breaker. Related #58726 overlaps the loop-abort half but does not cover the OpenShell validation gap.

## What This Changes

The issue requests OpenShell command preflight for generated or malformed shell text and stricter loop detection that blocks repeated identical failed or no-progress tool calls.

## Best Possible Solution

Add a narrow OpenShell sandbox exec preflight before the SSH `/bin/sh -c` handoff for obvious generated-command hazards, then change generic loop detection so repeated identical failed or no-progress calls escalate at the configured critical threshold while progressing generic repeats remain warning-only. Keep broader host exec heuristic filtering out of scope unless maintainers explicitly choose that policy.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a command such as `workflow install <name>` or an unclosed quote reaches `buildExecRemoteCommand` and `/bin/sh -c`, and current tests encode warning-only generic repeats below the global breaker.

## Solution Assessment

Yes. The narrow maintainable path is an OpenShell/SSH sandbox exec preflight plus no-progress-gated loop escalation; applying broad heuristic filtering to all host exec would be a larger policy change than this report requires.
