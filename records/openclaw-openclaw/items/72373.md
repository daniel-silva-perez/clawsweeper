## Summary

Keep open. Current main still lacks OpenShell command preflight before the SSH `/bin/sh -c` handoff, and generic repeated same-tool calls still remain warning-level until the separate global no-progress breaker. The related #58726 overlaps the loop-abort half, but it does not cover the OpenShell command-validation gap.

## What This Changes

The issue requests fail-fast OpenShell validation for malformed/generated shell commands and hard aborts for repeated identical failing or no-progress tool loops.

## Best Possible Solution

Add a narrow OpenShell sandbox exec preflight before building the SSH remote command, cover the obvious generated-command hazards from the report, and adjust loop detection so repeated identical failed or no-progress generic calls become critical at the configured threshold while progressing repeats remain warnings. Update focused tests and docs/changelog for the changed runtime behavior.

## Reproduction Assessment

Yes. A static reproduction path is high-confidence: `workflow install <name>` or an unclosed quote flows from `extensions/openshell/src/backend.ts` into `buildExecRemoteCommand` and then `/bin/sh -c`; current loop tests also assert generic repeats stay warning-only at the critical threshold.

## Solution Assessment

Yes. The narrow maintainable fix is OpenShell/SSH sandbox exec command preflight plus no-progress-gated generic loop escalation; a broad heuristic across all host exec would be a larger policy change with higher false-positive risk.
