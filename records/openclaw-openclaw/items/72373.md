## Summary

Keep open. Current main still forwards OpenShell exec command text into the SSH `/bin/sh -c` wrapper without generated-command or malformed-shell preflight, and generic repeated tool calls still remain warning-level until separate no-progress blockers apply. The related loop issue and open loop PRs overlap only the repeated-call hard-abort half; they do not cover the OpenShell command-validation gap.

## What This Changes

The issue requests OpenShell command preflight validation for malformed/generated shell commands and stricter hard-abort behavior for repeated identical tool-call loops.

## Best Possible Solution

Add a focused preflight in the OpenShell/SSH exec path that rejects the reported non-runnable generated placeholders and malformed shell syntax before remote shell invocation, then resolve the repeated-loop hard-abort policy through the existing loop-detection work so one canonical threshold behavior ships with tests and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `workflow install <name>` or an unclosed quote currently flows from `extensions/openshell/src/backend.ts` into `src/agents/sandbox/ssh.ts` and then `/bin/sh -c`; generic repeated calls are still warn-only in the current detector and tests until the global breaker.

## Solution Assessment

Yes, the maintainable fix direction is narrow OpenShell/SSH exec preflight plus coordinated loop-detector escalation rather than a broad shell-obfuscation filter across every exec mode. The loop portion should reuse or supersede the existing #58726/#60248/#62775 work instead of creating a parallel policy.
