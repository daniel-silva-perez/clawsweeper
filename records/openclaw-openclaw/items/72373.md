## Summary

Keep open. Current main still forwards OpenShell sandbox exec commands into the SSH `/bin/sh -c` wrapper without generated-command or malformed-shell preflight, and generic repeated same-tool/same-args loops remain warning-only until the later no-progress global breaker. The linked #58726 overlaps the loop-abort half, but it does not cover the OpenShell command-validation gap.

## What This Changes

The issue requests fail-fast validation for malformed/generated OpenShell exec commands and hard-abort behavior for repeated identical tool-call loops after a threshold.

## Best Possible Solution

Add narrow sandbox exec preflight before the SSH `/bin/sh -c` wrapper for obvious generated-command hazards, then make failed or no-progress generic repeats escalate at the configured critical threshold while preserving warning-only behavior for progressing repeats.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: an OpenShell exec command such as `workflow install <name>` or an unclosed quote reaches `buildExecRemoteCommand` and `/bin/sh -c`, and the loop test suite currently expects generic repeats at the critical threshold to remain warning-level.

## Solution Assessment

Yes. The maintainable path is targeted command preflight plus no-progress-gated loop escalation; broad host-exec heuristic filtering would be riskier and #58726 can remain the broader policy tracker if maintainers want to split the loop half.
