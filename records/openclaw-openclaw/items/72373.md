## Summary

Keep open. Current main still forwards OpenShell exec commands into the SSH `/bin/sh -c` wrapper without generated-command or malformed-shell preflight, and generic repeated same-tool/same-args loops still stay warning-level until the separate global no-progress breaker. The related #58726 overlaps the loop-abort half, but it does not cover the OpenShell command-validation gap, and no open PR appears to be paired with this report.

## What This Changes

The issue requests OpenShell command preflight rejection for malformed generated commands and hard-abort behavior for repeated failed or no-progress tool loops.

## Best Possible Solution

Add a narrow OpenShell sandbox exec preflight before the SSH `/bin/sh -c` handoff for obvious generated-command hazards, and adjust loop detection so repeated identical failed or no-progress generic calls escalate to a critical block at the configured critical threshold while progressing generic repeats remain warning-only.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a command such as `workflow install <name>` or an unclosed quote reaches `buildExecRemoteCommand` and `/bin/sh -c`, and the current loop-detection tests expect generic no-progress repeats at `CRITICAL_THRESHOLD` to remain warning-level.

## Solution Assessment

Yes. The maintainable fix is a targeted OpenShell/SSH sandbox exec preflight plus no-progress-gated loop escalation; broad heuristic filtering across all host exec would be a riskier policy change and should not be required for this report.
