## Summary

Keep open. Current main still routes OpenShell exec commands into the SSH `/bin/sh -c` wrapper without a generated-command or malformed-shell preflight, and the generic repeated-call detector remains warning-only until the later no-progress global breaker. The related #58726 overlaps the loop-abort half, but it does not cover the OpenShell command-validation gap.

## What This Changes

The issue requests OpenShell exec preflight rejection for malformed or placeholder-like generated shell commands, plus hard-aborting repeated identical failing or no-progress tool loops after the configured threshold.

## Best Possible Solution

Add a conservative sandbox exec preflight before the SSH `/bin/sh -c` wrapper for obvious generated-command hazards, then escalate failed or no-progress generic repeats at the configured critical threshold while preserving warning-only behavior for progressing repeats. If bundled examples are found to encourage copy-pasting placeholders, update those examples so they read as non-runnable templates.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: an OpenShell command such as `workflow install <name>` or a command with an unclosed quote reaches `buildExecRemoteCommand` and `/bin/sh -c`, and the current loop tests encode warning-only generic repeats below the global breaker.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it stays conservative. A broad shell-obfuscation filter across all host exec would be riskier than targeted OpenShell/sandbox preflight plus no-progress-gated loop escalation.
