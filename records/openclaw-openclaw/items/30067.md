## Summary

Current main still lacks the requested proactive first-run/setup heads-up about temporary provider API rate limits before the TUI/Web chat handoff. Existing handling is reactive error formatting plus model/quota docs, so the issue remains valid and narrow.

## Best Possible Solution

Keep this issue open and implement a small onboarding UX change: add a concise first-run heads-up in the classic finalizeSetupWizard path before the TUI/Web/Later handoff, and mirror equivalent guidance in Crestodian only if that path can lead directly into normal agent chat. Cover the copy with focused tests and avoid changing provider runtime/failover behavior.
