## Summary

Current main still matches the reported configured ACP session-key collision: inherited Discord thread routing materializes the parent channel as the configured ACP conversation, and the persistent ACP key hashes only that conversation id. The same-author fix PR #64322 remains open, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Discord threads inheriting an ACP parent-channel binding get thread-scoped persistent ACP session keys instead of sharing one parent-channel persistent ACP process.

## Best Possible Solution

Preserve parent-channel route inheritance, but materialize inherited Discord thread ACP bindings with the live thread conversation id so persistent configured ACP sessions are isolated per thread. Add regression coverage proving two thread ids under one parent resolve to different targetSessionKey values while direct parent-channel and direct thread bindings still behave correctly.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: resolve two different Discord thread conversation ids with the same parent-channel configured ACP binding, and current main will match/materialize the parent id before hashing the persistent session key, producing the same targetSessionKey.

## Solution Assessment

Yes, the narrow maintainable direction is to keep Discord parent binding inheritance but make the materialized configured ACP identity thread-scoped for inherited thread contexts. The open PR #64322 appears to target that boundary and should be reviewed rather than starting a separate repair lane.
