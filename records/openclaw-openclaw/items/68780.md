## Summary

Keep open. Current main still emits the empty-allowlist warning for workspace/global discovered non-bundled plugins, and a maintainer already reviewed the same behavior on main and explicitly kept it open as a UX/policy issue. Some pieces are better than the original report described, but the actionable first-run warning copy remains unresolved.

## What This Changes

Requested behavior: make plugin empty-allowlist and untracked-provenance warnings less alarming and more actionable for first-time users.

## Best Possible Solution

Keep this open and land a narrow UX/docs repair: keep the security warning, but make it explicitly actionable by showing the discovered plugin IDs, a `plugins.allow` snippet, and the relevant `openclaw plugins list --enabled --verbose` / `openclaw plugins inspect` discovery commands. Broader loading-policy or channel-alias changes should remain a separate maintainer decision.

## Reproduction Assessment

Yes. The current source tests provide a high-confidence reproduction path: a workspace/global discovered non-bundled plugin with `plugins.enabled: true` and an empty allowlist still produces the empty-allowlist warning, and a loaded global plugin without provenance still produces the untracked-provenance warning.

## Solution Assessment

Yes for the maintainer-scoped UX/docs direction, not for suppressing the warning or changing auto-load policy. The narrow maintainable fix is to improve remediation text and documentation while preserving the warning level and trust semantics.

## Repair Work Prompt

Fix the plugin empty-allowlist warning UX for https://github.com/openclaw/openclaw/issues/68780. Preserve the existing security/trust semantics: do not suppress the warning on first boot, do not change plugin discovery/loading policy, do not auto-load new plugin classes, and do not implement broad channel-id alias policy unless existing helpers already do it for bundled channels. In `src/plugins/loader-provenance.ts`, make `warnWhenAllowlistIsOpen` actionable for first-time users by including the discovered plugin IDs, an exact config shape such as `plugins: { allow: [...] }`, and commands for ID discovery (`openclaw plugins list --enabled --verbose`, `openclaw plugins inspect <id> --json`). Consider making `warnAboutUntrackedLoadedPlugins` similarly point at `plugins.allow` or install records without downgrading the warning. Update focused tests in `src/plugins/loader.test.ts` and keep `src/plugins/loader.cli-metadata.test.ts` proving metadata loads stay quiet. Update public docs (`docs/tools/plugin.md` and/or `docs/cli/plugins.md`) with the warning meaning and remediation. Add a single-line changelog Fixes entry crediting `Thanks @JIRBOY`. Do not touch unrelated duplicate-plugin precedence, installer policy, or generated baseline files unless the implementation genuinely requires it.
