## Summary

Keep open. Current main still emits the active-startup empty `plugins.allow` warning with only generic remediation text, so the first-time-user actionability problem remains. Some original false-positive pieces are improved, especially bundled-plugin provenance handling and metadata-load suppression, but the central UX issue is not implemented yet.

## What This Changes

Request clearer, less alarming plugin allowlist/provenance warnings for first-time users, with explicit plugin-id remediation and channel-vs-plugin ID guidance.

## Best Possible Solution

Keep the security warning, but make it actionable: show exact discovered plugin ids, give a small `plugins.allow` example or command path, point to plugin inspection commands/docs, and clarify plugin ids versus channel ids without changing plugin trust policy.

## Reproduction Assessment

Yes. A source-level reproduction is any active plugin registry load with plugins enabled, an empty `plugins.allow`, and at least one workspace/global discovered plugin; current code deterministically emits the generic warning on that path.

## Solution Assessment

Yes for a focused UX repair, not for suppressing the warning entirely. The narrow maintainable fix is to update warning text, diagnostics/docs, and regression tests while preserving existing allowlist and provenance semantics.
