## Summary

Keep open. The report is a recent, concrete Microsoft Teams regression with a second confirmation that rolling back to the April 15 build restores Teams DMs, and current main still routes installed Teams traffic through the same msteams webhook JWT gate without an exact fix or canonical duplicate for this specific Web Chat works / installed Teams app fails path.

## What This Changes

Messages sent from an installed Microsoft Teams app through an Azure Bot should reach the OpenClaw msteams webhook and receive replies, matching the working Azure Bot Test in Web Chat path.

## Best Possible Solution

Keep this open until maintainers capture the installed Teams app webhook request status and token-claim shape, then either adjust the msteams JWT validation to accept legitimate Teams-channel tokens using a documented app-binding claim, or document/doctor the exact Azure Bot or Teams app manifest fix if the request never reaches OpenClaw. Add regression coverage and visible auth diagnostics so Web Chat and installed Teams app paths are distinguishable.

## Reproduction Assessment

Yes, as a live integration path: create the Azure-backed Teams app with the Teams CLI, configure `channels.msteams`, verify Azure Bot Test in Web Chat works, then send a Teams DM and compare gateway/Caddy inbound activity. It is not locally reproducible without Azure/Teams access, but the steps and rollback confirmation are specific enough to guide reproduction.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on whether current main is rejecting a legitimate installed-Teams token or whether the Teams app/Azure channel never sends to the endpoint; any code change should preserve the existing cross-bot replay protection and add diagnostics/tests around the observed token shape.
