## Summary

Keep open. Current main still lacks the reasoning-prefaced silent-reply classifier and runner suppression this PR proposes, and the PR is not superseded because #69394 only covers a broader auto-reply policy and does not address the embedded-runner retry/warning path. The patch is not merge-ready as written because it introduces an `AssistantMessage` type reference without importing or defining that type.

## What This Changes

The PR adds a reasoning-prefaced `NO_REPLY` detector in auto-reply token classification and updates embedded-runner incomplete-turn guards/tests so stripped silent replies do not trigger retries or user-visible failure warnings.

## Best Possible Solution

Fix the missing runner type import or replace the new parameter type with an existing local type, then land the narrow end-to-end suppression with the added token and embedded-runner regression coverage while preserving the #19537 substantive-reply contract.

## Reproduction Assessment

Yes. The PR body provides the exact production payload, and current main source shows a high-confidence static reproduction: `isSilentReplyPayloadText` does not recognize `think\n...NO_REPLY`, while the runner warning path only treats non-empty `assistantTexts` as silent before evaluating empty/reasoning retry guards.

## Solution Assessment

No, not as written. The behavioral direction is narrow and maintainable, but the unresolved `AssistantMessage` type makes the patch fail typechecking; importing the type from `@mariozechner/pi-ai` or using `EmbeddedRunAttemptResult["lastAssistant"]` would be the safer merge-ready form.
