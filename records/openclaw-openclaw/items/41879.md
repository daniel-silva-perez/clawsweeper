## Summary

Current main still matches the reported defaults-only heartbeat bug. `agents.defaults.heartbeat` is documented as global heartbeat behavior, but the enablement helper and scheduler only treat per-agent `agents.list[].heartbeat` blocks as explicit scope and otherwise fall back to the resolved default agent. The related fix PRs in the provided context are closed without merge, so this issue remains the canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical tracker. The likely fix belongs in the heartbeat owner path: align `isHeartbeatEnabledForAgent()`, `resolveHeartbeatAgents()`, scheduler coverage, and summary behavior so `agents.defaults.heartbeat` applies to the configured agent set when no per-agent heartbeat blocks exist, while preserving explicit per-agent scoping when any `agents.list[].heartbeat` block exists. Add regression coverage for defaults-only multi-agent enablement and scheduling, then make a separate product/docs decision on zero-config heartbeat behavior.

## Risks / Open Questions

- Fixing defaults-only scope will increase heartbeat activity for users who configured `agents.defaults.heartbeat` with multiple agents but were unknowingly getting only the default agent scheduled.
- The zero-configuration expectation in the report conflicts with current docs, which describe a default heartbeat cadence; maintainers should decide that separately from the defaults-only multi-agent bug.
