## Summary

Keep this issue open. Current main still has the reported defaults-only heartbeat scope behavior: enablement checks only per-agent heartbeat blocks, scheduler resolution falls back to the default agent, and the regression test still expects the non-default agent to be disabled even when agents.defaults.heartbeat is configured. The defaults-only scope bug remains canonical here; the zero-config behavior needs explicit maintainer policy because current docs still describe default heartbeats as enabled unless disabled with 0m.

## Best Possible Solution

Keep this issue open as the canonical tracker for the defaults-only multi-agent heartbeat scope bug. Maintainers should choose the policy explicitly, then update src/infra/heartbeat-summary.ts, src/infra/heartbeat-runner.ts scheduler resolution, runtime skip behavior, status/health reporting, docs, and regression tests together. The narrower doctor-warning request can remain separate in #46101, and empty-object disabling can continue under #64293/#64540.

## Risks / Open Questions

- Changing defaults-only scope would increase heartbeat activity for multi-agent users who configured agents.defaults.heartbeat but were unknowingly only scheduling the default agent.
- The zero-config disable request conflicts with current public docs that describe heartbeats as enabled by default and disabled via every: "0m"; that part needs explicit product/docs alignment.
- A partial fix only in isHeartbeatEnabledForAgent would leave scheduler resolution, status/health summaries, and tests inconsistent.
