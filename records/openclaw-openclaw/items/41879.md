## Summary

Current main still matches the reported defaults-only heartbeat scope bug. `agents.defaults.heartbeat` is documented as global behavior, but enablement and scheduling still treat only per-agent heartbeat blocks as explicit scope and otherwise fall back to the default agent. The zero-config disable expectation also conflicts with current docs, so this needs maintainer product/docs judgment rather than cleanup closure.

## Best Possible Solution

Keep this issue open as the canonical tracker. Align heartbeat scope policy across `src/infra/heartbeat-summary.ts`, `src/infra/heartbeat-runner.ts`, scheduler resolution, runtime skip checks, docs, and regression tests: either defaults-only heartbeat applies to the configured agent set, or the docs should document the default-agent fallback. Separately decide whether zero-config heartbeat remains enabled for the default agent or becomes disabled by default.

## Risks / Open Questions

- Fixing defaults-only scope will increase heartbeat activity for users who configured `agents.defaults.heartbeat` with multiple listed agents but were unknowingly getting only the default agent scheduled.
- The zero-configuration disable expectation conflicts with current docs describing heartbeats as enabled by default with a default interval, so maintainers should decide that policy explicitly.
