## Summary

Current main still matches the reported defaults-only heartbeat bug: heartbeat enablement and scheduling only treat per-agent heartbeat blocks as explicit scope, then fall back to the default agent. Docs describe agents.defaults.heartbeat as global behavior, and the regression test still expects the non-default agent to be disabled under defaults-only config. The zero-config disable request conflicts with current docs and should be handled as a separate product/docs decision.

## Best Possible Solution

Keep this issue open as the canonical tracker. Fix the heartbeat owner path by making the heartbeat scope policy consistent across isHeartbeatEnabledForAgent(), resolveHeartbeatAgents(), scheduling, status/summary, and any duplicated gating helpers: per-agent heartbeat blocks should keep explicit opt-in scope; defaults-only heartbeat should apply to the configured agent set or default fallback. Update regression tests to cover defaults-only multi-agent scheduling and decide separately whether zero-config heartbeat should remain documented default behavior or change.
