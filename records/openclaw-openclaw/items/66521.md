## Summary

This PR should stay open. Current main still does not implement the requested early Signal group typing cue, and the latest PR revision improves the original patch by limiting it to groups and routing through typing callbacks, but it still starts an outbound Signal typing side effect before dispatch has resolved sendPolicy and typing suppression.

## Best Possible Solution

Keep this PR open and rework it into a policy-aware Signal implementation. The clean path is to start group typing earlier only after route/session sendPolicy and typingMode/suppressTyping have been resolved, or add an equivalent Signal-local guard that exactly matches those contracts before calling the channel typing callback. Keep the author's entry.isGroup scope fix and callback-based keepalive/TTL approach, then add targeted Signal tests for early allowed group feedback plus denied/typing-disabled sessions emitting no typing indicator.

## Risks / Open Questions

- Merging the latest PR head could emit a Signal typing indicator when sendPolicy is "deny", despite the current dispatch contract suppressing outbound replies, notices, and indicators in that mode.
- Merging the latest PR head could ignore session.typingMode "never", explicit suppressTyping, internal webchat, system-event, or heartbeat typing suppression because it starts the channel callback before shared typing policy is resolved.
- The PR is active and has a plausible focused UX fix, so closing it as obsolete would lose useful work; it needs rework rather than cleanup closure.
- No focused test currently proves both sides of the desired behavior: early Signal group feedback for allowed sessions and no typing signal for denied or disabled sessions.
