## Summary

Keep this PR open, but do not merge the current patch. Current main still lacks the requested Signal-specific early group typing cue before Signal dispatch work, and the latest PR revision improves the original scope by limiting it to groups and using the shared typing callback. The remaining blocker is that the callback is still invoked before dispatch resolves sendPolicy and typing suppression, so it can emit an outbound Signal typing indicator for sessions that current main is designed to keep quiet.

## What This Changes

The PR adds a Signal group-only `typingCallbacks.onReplyStart()` call immediately before inbound dispatch so group chats show a typing indicator during pre-dispatch work.

## Best Possible Solution

Keep the group-only Signal UX improvement, but move the early typing start behind the shared route/session sendPolicy and typing suppression resolution. A maintainable fix should start typing early enough to cover model/preflight latency, avoid starting for denied or typing-disabled sessions, and add targeted Signal tests for allowed and suppressed group turns.

## Reproduction Assessment

Yes. A focused unit reproduction can configure a Signal group turn with `session.sendPolicy: deny` or `typingMode: "never"`, mock the Signal typing callback, and show that the PR calls it before dispatch has a chance to suppress typing.

## Solution Assessment

No. Calling `typingCallbacks.onReplyStart()` directly from the Signal monitor is not the best solution because it bypasses dispatch-owned policy decisions; a safer implementation should use a policy-aware hook after `resolveSourceReplyVisibilityPolicy` and typing mode resolution.
