## Summary

Keep this PR open. Current main still has the sync stat-error mismatch the PR is trying to address, but the branch is stale against the shared allowlist helper split and the proposed readFileSync rethrow would preserve a sync/async parity bug rather than fix it.

## What This Changes

The PR changes two catch branches in the old sync pairing allowlist reader in `src/pairing/pairing-store.ts` to rethrow non-ENOENT stat and readFile errors instead of treating the allowlist as missing.

## Best Possible Solution

Refresh the useful stat-error part into `src/pairing/allow-from-store-file.ts`, add focused tests for non-ENOENT stat failures, and handle read-file failures consistently across sync and async readers only after deciding whether the shared fallback contract should change.

## Reproduction Assessment

Yes: a focused test can stub `fs.statSync` to throw `EACCES` and observe current main returning an empty allowlist while the async stat path rethrows. A second stub where stat succeeds and `readFileSync` throws `EACCES` reproduces the PR's new sync-only read behavior against the async fallback contract.

## Solution Assessment

No: rethrowing unexpected sync stat errors is the right direction, but this PR also rethrows sync read failures without changing the async reader and targets an old inline helper. The safer fix is to update the current shared helper and test the stat contract separately from any read-fallback policy change.
