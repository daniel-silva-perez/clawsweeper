## Summary

Keep this PR open. Current main still has the sync allowlist reader swallowing unexpected stat and read errors, but the branch is stale against the shared helper split and its readFileSync change would still disagree with the async read-file fallback contract.

## What This Changes

The PR changes two sync allowlist reader catch branches in `src/pairing/pairing-store.ts` to throw non-ENOENT stat and read errors instead of returning an empty missing allowlist.

## Best Possible Solution

Refresh the fix into `src/pairing/allow-from-store-file.ts`, add focused tests for non-ENOENT stat failures, and make an explicit maintainer decision on read-file error semantics before changing both sync and async paths consistently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep sync read failures aligned with async reads:** `src/pairing/pairing-store.ts:429`
  - body: This `throw err` runs after `statSync` succeeds, but the async path still delegates to `readJsonFileWithFallback`, which returns the fallback for non-ENOENT read errors. An unreadable allowlist would therefore crash sync callers while async callers see an empty fallback, so this does not provide parity unless both contracts change together.
