## Summary

Keep this PR open. Current main still does not implement optional Slack run-start timing for ack reactions: there is no `ackReactionTiming` config field/schema/help/doc surface, Slack monitor context/provider do not carry timing, and Slack still starts ack/status reaction work before `dispatchInboundMessage` rather than from `onAgentRunStart`. The PR discussion remains relevant because the Greptile scope-gate concern was addressed in the PR with `ackReactionAllowed`, but the PR is still an unmerged, non-mergeable implementation candidate that needs maintainer review/rebase rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review and rebase. The best path is to land a rebased Slack implementation that defaults to current receipt-time behavior, accepts `received | run-start` through strict config/schema/docs, defers the ack/status reaction from `onAgentRunStart` only when configured, and keeps the existing scope gate and cleanup semantics covered by targeted Slack tests. Coordinate the config shape with #45449 and #46258 before merging if maintainers want a shared cross-channel contract.
