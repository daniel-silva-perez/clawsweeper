## Summary

Keep this PR open. Current main still does not implement optional Slack run-start timing for ack reactions: the config type/schema do not accept `ackReactionTiming`, Slack context/provider do not carry a timing value, and the Slack message path still prepares the ack/status reaction during message preparation/dispatch rather than from `onAgentRunStart`. The PR discussion is useful: Greptile found a scope-gate bypass, and the contributor addressed it in the PR with `ackReactionAllowed`; the remaining path is maintainer review and rebase, not cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review and rebase. The best implementation is a Slack-scoped or intentionally shared config contract that defaults to current receipt-time behavior, accepts `received | run-start` through strict config/schema/docs, defers the ack/status reaction from `onAgentRunStart` only when configured, and keeps the existing scope gate and cleanup semantics covered by targeted Slack tests.
