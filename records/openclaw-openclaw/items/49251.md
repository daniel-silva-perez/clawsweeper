## Summary

Keep open. Current main still handles rate-limit exhaustion as a terminal visible failure after model fallback, and the existing queues are either in-process followup/command queues or a durable restart-continuation delivery queue. I found related open work (#62540, #49431, #51623, #51803), but none is merged or clearly supersedes the durable pending prompt/replay behavior requested here.

## Best Possible Solution

Keep this issue open. The best path is a core durable pending-turn queue for model-capacity/API-limit exhaustion, coordinated with the open rate-limit auto-retry and pending-inbound/followup persistence work. It should persist the original prompt and route, expose a waiting state, schedule cooldown-aware replay, and include duplicate-prevention tests across gateway restart/crash recovery.

## Risks / Open Questions

- Durable queued prompts can contain sensitive user data, so storage, retention, permissions, redaction, and cleanup need to align with transcript/session security rules.
- Replay must preserve session key, channel route, thread, account, auth-profile/model cooldown metadata, and idempotency so recovered prompts do not run twice or reply to the wrong target.
- The overlapping open items (#62540, #49431, #51623, #51803) need maintainer scope alignment before implementation to avoid incompatible pending-turn stores.
