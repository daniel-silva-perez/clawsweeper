## Summary

Keep open. Current main has in-process followup queues, command lanes, model fallback, auth-profile cooldowns, and visible rate-limit errors, but it still does not durably preserve an API-limit-blocked prompt or replay it once capacity returns. The existing durable session-delivery queue is for restart continuation/system-event/agent-turn delivery, not a general pending prompt queue for all-model API-limit exhaustion.

## Best Possible Solution

Keep this issue open. The best path is a core durable pending-turn queue for model-capacity/API-limit exhaustion, likely integrated with the existing session key, reply route, delivery context, auth-profile cooldown metadata, retry/backoff timing, and idempotency model. It should expose waiting state to users and include tests for all-model rate-limit exhaustion, gateway restart/crash recovery, cooldown-expiry replay, and duplicate prevention.

## Risks / Open Questions

- Durable prompt storage needs retention, privacy, and redaction rules aligned with transcript/session storage because queued prompts may contain sensitive user data.
- Replay needs idempotency, route preservation, cooldown awareness, and duplicate-delivery safeguards so recovered prompts do not run twice or post to the wrong channel/thread.
- An implementation should clarify user-visible waiting state across channel UIs, Control UI, and status surfaces so prompts are not silently parked.
