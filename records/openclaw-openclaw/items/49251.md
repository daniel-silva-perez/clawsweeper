## Summary

Keep open. Current main still turns all-model API-limit/cooldown exhaustion into a visible terminal failure after fallback rather than a durable waiting prompt that can resume automatically, and the existing queues are either in-process serialization or a narrower durable delivery queue.

## What This Changes

Request durable queuing, user-visible waiting state, and automatic replay for prompts that cannot be answered immediately because all configured models or fallbacks are blocked by API limits.

## Best Possible Solution

Add a core durable pending-turn queue for model-capacity/API-limit exhaustion that records the original prompt, route, session, model/fallback state, retry time, and idempotency key; expose a waiting state to the user; replay after cooldown or restart; and keep it distinct from the narrower session-delivery queue unless that storage is intentionally generalized.

## Reproduction Assessment

Yes. A source-level reproduction path is all configured model candidates or auth profiles becoming rate-limited/cooldowned: current main reaches the fallback-summary branch, fails the reply operation, and returns rate-limit text instead of persisting a pending prompt.

## Solution Assessment

Yes on the product direction, but the JSONL sketch in the discussion should be implemented through OpenClaw's core state/recovery conventions rather than as an isolated ad hoc queue. The safest solution should coordinate with the existing session-delivery queue and pending-inbound PRs while preserving duplicate prevention.
