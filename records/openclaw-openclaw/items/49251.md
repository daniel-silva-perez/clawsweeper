## Summary

Keep open. Current main has in-process busy-session queues, model fallback, auth-profile cooldowns, and user-visible rate-limit errors, but it does not preserve an API-limit-blocked prompt as durable queued work or automatically replay it when model capacity returns. The discussion’s durable JSONL-per-session direction remains unimplemented and is not superseded by the related closed reports.

## Best Possible Solution

Keep this issue open. The best path is a core durable pending-turn queue, likely by extending the existing session-delivery/session queue machinery or adding a dedicated prompt queue with session key, route, prompt/media metadata, status, retry/backoff/cooldown time, idempotency, and startup replay. It should expose waiting state in status/UI and include tests for all-model rate-limit exhaustion, gateway restart/crash recovery, cooldown expiry replay, and duplicate prevention.

## Evidence

## Risks / Open Questions

- Durably storing queued prompts needs clear retention, privacy, and redaction behavior aligned with transcript/session storage.
- Replay needs idempotency, route preservation, auth/profile cooldown awareness, and duplicate-delivery safeguards so recovered prompts do not post to the wrong channel or run twice.
