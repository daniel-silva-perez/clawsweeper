## Summary

Keep this PR open. The reset-hook direction is useful and addresses a real current-main gap for non-slash Discord reset triggers, but the current head still has blocking transcript path regressions around explicit sessionFile handling, including an unsafe outside-directory preservation path.

## What This Changes

This PR changes custom reset-trigger fallback hook emission, gateway reset transcript path rotation, and adds Discord bang-prefix reset-trigger tests.

## Best Possible Solution

Land a revised branch that emits hooks for every matched custom resetTrigger, rotates generated/default transcript files to a fresh sessionId, preserves only validated explicit custom transcript files, and adds coverage for bang triggers, in-dir custom filenames, and invalid outside sessionFile fallback.

## Reproduction Assessment

Yes. Current main can be traced from resetTriggers matching in initSessionState to the slash-only fallback in get-reply.ts, and the PR regression is covered by the existing custom-owned-child-transcript reset test shape.

## Solution Assessment

No. The hook-emission change is the right narrow direction, but the session reset path should validate preserved transcript paths and classify explicit custom filenames by generated/stale basename rather than only by directory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve explicit in-dir sessionFile paths:** `src/gateway/session-reset-service.ts:336-338`
  - body: This check only treats an explicit transcript as custom when its directory is outside defaultSessionsDir. Existing reset coverage seeds a spawned session with a custom sessionFile in the store directory and expects reset to keep that exact path; with this branch, it is rewritten to <nextSessionId>.jsonl, breaking pinned transcript paths.
  - confidence: 0.91
- **[P1] Validate custom transcript directories before writing:** `src/gateway/session-reset-service.ts:340`
  - body: For outside paths, this branch derives explicitSessionFileDir directly from currentEntry.sessionFile and later creates/writes next.sessionFile there. Existing path helpers intentionally fall back when sessionFile is outside known agent sessions dirs, so this bypass can create transcripts under arbitrary persisted paths if store metadata is stale or corrupt.
  - confidence: 0.86
