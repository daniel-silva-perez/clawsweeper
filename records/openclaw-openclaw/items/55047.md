## Summary

Keep open: current main still has a credible Control UI blank-screen crash path matching the report and the linked discussion. Malformed persisted cron rows can still reach UI state, and shared render plus Cron edit/list paths still dereference `job.payload` without a shape guard; the related focused PRs were closed unmerged and no open PR currently references this issue.

## What This Changes

The issue reports a Control UI regression where left-menu sections such as channels, instances, sessions, agents, and skills render as blank pages with only the page title visible.

## Best Possible Solution

Make the Control UI resilient to malformed persisted cron rows at both the data boundary and UI consumption sites: normalize, filter, or annotate invalid rows before `cron.list`/`cron.listPage` exposes them, guard shared render and Cron edit/list paths before reading `payload.kind`, and add focused regression coverage proving malformed Cron rows cannot blank unrelated Control UI sections.

## Repair Work Prompt

Fix the Control UI blank-screen regression tracked by this issue and related #54439. Current main can still accept malformed persisted cron rows into `cron.list`/UI state, and shared Control UI rendering plus Cron edit/list paths dereference `job.payload.kind` without validating row shape. Implement a narrow resilience fix: validate, filter, or safely normalize malformed cron rows before they can break `renderApp`; add null-safe guards in Control UI consumers that read cron payloads; preserve valid legacy cron migration behavior; do not add core owner-specific policy or broad UI refactors. Use #54552 only as context, not as a blind cherry-pick, because it covered only one render read. Add regression tests with a malformed cron row lacking `payload` proving the Control UI shared render path does not throw and Cron list/edit handling does not blank unrelated sections. Add a user-facing changelog fix entry with proper attribution unless maintainers decide to batch it elsewhere.
