## Summary

Keep this issue open. Current main still has the reported Anthropic OAuth/model-level beta-header gap, and the provided GitHub context shows an open same-author implementation PR with explicit closing intent, so conservative cleanup should wait for that PR to be reviewed, replaced, landed, or closed.

## What This Changes

The issue requests preserving or injecting Anthropic's OAuth-required `oauth-2025-04-20` beta when a `sk-ant-oat-*` OAuth token is used and `context-1m-2025-08-07` is configured through model-level headers instead of `params.context1m`.

## Best Possible Solution

Keep the issue open and review, land, replace, or close PR #41461. The eventual fix should preserve OAuth-required Anthropic beta tokens independently of `params.context1m`, handle model/provider/request-level `anthropic-beta` headers in both the plugin wrapper and direct transport paths, strip or ignore `context-1m-*` for OAuth tokens per current docs, and add regression coverage for OAuth plus model-level `anthropic-beta` without `params.context1m`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a direct Anthropic model with `headers: { "anthropic-beta": "context-1m-2025-08-07" }` and use a `sk-ant-oat-*` token; current main builds the OAuth beta first, then lets model headers overwrite it, while the plugin wrapper path is not composed when no extra-param beta exists.

## Solution Assessment

No for the narrow always-apply-wrapper idea by itself. That can address the wrapper path, but current main also has a direct transport merge path where model-level `anthropic-beta` overwrites `oauth-2025-04-20`, so the best fix should protect/merge Anthropic beta headers at the actual transport boundary too.
