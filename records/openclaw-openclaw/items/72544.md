## Summary

Keep this PR open. Current main still builds native and Gateway `pluginToolAllowlist` values from the unmerged profile policies while the downstream policy pipeline already uses the merged-with-`alsoAllow` variants, so the PR addresses a real remaining inconsistency. The related #51388 work overlaps the same tool-policy area, but it tracks the subagent `alsoAllow` propagation gap rather than proving this profile/provider-profile change obsolete.

## What This Changes

The PR changes native agent and Gateway scoped tool construction to build `pluginToolAllowlist` from `profilePolicyWithAlsoAllow` and `providerProfilePolicyWithAlsoAllow` instead of the unmerged profile policies.

## Best Possible Solution

Land this narrow profile/provider-profile fix with focused regression coverage, or fold the same two-call-site change into the broader #51388 tool-policy work so native and Gateway plugin tool creation consistently sees effective `alsoAllow` grants while deny precedence remains unchanged.

## Reproduction Assessment

Yes. A static reproduction is clear from current main: configure a restrictive profile plus `tools.alsoAllow` for an optional plugin tool, and the optional tool can be removed during `resolvePluginTools()` because the earlier `pluginToolAllowlist` is built from the unmerged profile policies.

## Solution Assessment

Yes for the profile/provider-profile path. Reusing the already-computed merged policies at both allowlist-construction sites is the narrowest maintainable fix; the safer alternative is only to land it as part of the broader #51388 coordination if maintainers want one consolidated tool-policy PR.
