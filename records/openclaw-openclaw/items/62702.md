## Summary

Current main now provides the requested core behavior: plugin authors can register typed hooks with per-handler `timeoutMs`, the hook runner races slow handlers against that budget, logs and skips timed-out non-critical handlers, continues to later handlers, and keeps security-sensitive `before_tool_call` fail-closed in the global runner. Active Memory also exposes user-tunable timeout config for the motivating memory-recall case.

## Best Possible Solution

Keep the shipped main-branch implementation: per-hook `timeoutMs` on `api.on`, default timeout/skip behavior for prompt-context hooks, fail-closed policy for security-sensitive tool hooks, and plugin-owned timeout config for memory recall. Any remaining desire for a different JSON config shape should be opened as a narrower API-design follow-up.
