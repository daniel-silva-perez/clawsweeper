## Summary

Keep open. Current main still does not implement a bundled Telegram `groupPolicy: "members"`; the public config contract, strict schema, generated metadata, Telegram inbound authorization, native-command authorization, security audit coverage, and docs still only cover `open`, `disabled`, and `allowlist`. The closed implementation PRs and the `openclaw-chat-only-trigger` community plugin are useful prior art/workarounds, but they do not satisfy the requested bundled behavior.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes new groups based on trusted participants instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Define and land a bundled Telegram `members` policy that updates config types, schema, generated metadata, inbound group-message and native-command authorization, fail-closed membership verification, security-audit semantics, docs, and focused tests while preserving existing `open`, `disabled`, and `allowlist` behavior.
