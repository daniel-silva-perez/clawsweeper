## Summary

Keep open. Current main still does not implement a bundled Telegram `groupPolicy: "members"`; the exported config type, strict schema, generated metadata, Telegram inbound authorization, native-command authorization, security audit, and docs all still model only `open`, `disabled`, and `allowlist`. The closed implementation PRs and the `openclaw-chat-only-trigger` community plugin are useful prior art/workarounds, but they do not satisfy the requested bundled access-control behavior.

## What This Changes

The issue requests a bundled Telegram `groupPolicy: "members"` mode that lets trusted user allowlists authorize newly created Telegram groups without allowlisting each group chat ID.

## Best Possible Solution

Land a bundled Telegram `members` policy that updates the config contract, schema, generated metadata, inbound group-message and native-command authorization, fail-closed membership verification, security-audit behavior, docs, and focused tests while preserving existing `open`, `disabled`, and `allowlist` behavior.
