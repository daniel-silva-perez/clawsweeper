## Summary

Keep open. Current main at e9d4cb2bb6dc15f8896b13b7e46ede199668c198 still lacks the requested bundled Telegram `groupPolicy: "members"` across config, generated metadata, inbound authorization, native-command authorization, docs, and security audit behavior. The closed unmerged implementation PRs and the `openclaw-chat-only-trigger` community plugin are useful prior art/workarounds, but they do not satisfy the requested bundled all-current-members access-control policy.

## What This Changes

Request a Telegram `groupPolicy: "members"` mode that lets bots reply in newly created groups when all current participants are trusted users, instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy that updates the config type, strict schema, generated metadata, Telegram inbound and native-command authorization, fail-closed membership verification, security-audit behavior, docs, and focused tests while preserving existing `open`, `disabled`, and `allowlist` behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The missing behavior is high-confidence from static current-main verification across exported config, schema, generated metadata, Telegram authorization helpers, inbound/native-command call sites, docs, and tests.

## Solution Assessment

Unclear until maintainers settle the authorization contract. The durable fix belongs in the bundled Telegram plugin and shared config/docs surfaces, but a repair worker should not choose the fail-closed membership semantics without maintainer review.
