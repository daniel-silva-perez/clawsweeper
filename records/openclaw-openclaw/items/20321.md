## Summary

Keep open. Current main still does not ship a bundled Telegram `members` group policy: the public config type, strict schema, generated metadata, Telegram inbound/native-command authorization helper, docs, and security audit all remain limited to `open`, `disabled`, and `allowlist`. The closed implementation PRs and the external chat-only-trigger plugin are useful prior art/workarounds, but they do not resolve the requested OpenClaw behavior.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes newly created groups based on all current group members being trusted users instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy that updates config types/schema/generated metadata, inbound group-message and native-command authorization, fail-closed Telegram membership verification, security-audit semantics, docs, and focused tests while preserving the existing `open`, `disabled`, and `allowlist` behavior.
