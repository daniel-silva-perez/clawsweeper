## Summary

Keep this issue open. Current main still exposes only `open`, `disabled`, and `allowlist` as group policies, and Telegram inbound/native-command authorization still has no all-current-members verification path; the closed implementation PRs and the chat-only-trigger community plugin are useful prior art/workarounds, not a shipped bundled implementation.

## What This Changes

Add a Telegram `groupPolicy: "members"` mode that lets new groups work when every current participant is trusted by user allowlist, without allowlisting each group chat ID.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy with updated config types/schema/generated metadata, Telegram inbound and native-command authorization, security-audit semantics, docs, and focused tests. The policy should fail closed on uncertain membership checks and preserve existing `open`, `disabled`, and `allowlist` behavior.
