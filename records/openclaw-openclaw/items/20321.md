## Summary

Keep this issue open. Current main still does not expose or enforce Telegram `groupPolicy: "members"`; the public config contract, strict schema, generated metadata, Telegram group authorization helper, inbound/native-command call paths, security audit, docs, and focused tests still only cover `open`, `disabled`, and `allowlist`. The closed related PRs and the chat-only-trigger community plugin are useful prior art/workarounds, not a bundled implementation.

## What This Changes

Add a Telegram `groupPolicy: "members"` mode that allows newly created groups when every current member is trusted via user allowlists instead of requiring every group chat ID to be allowlisted.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with an updated shared config contract, docs, generated metadata, inbound and native-command authorization, security audit handling, and focused tests. The implementation should verify trusted current members with bounded Telegram API calls, fail closed on uncertainty, and preserve existing `open`, `disabled`, and `allowlist` behavior.
