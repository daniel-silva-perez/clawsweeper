## Summary

Keep open. Current main still lacks the requested bundled Telegram `members` group policy across the public config contract, strict schema, generated metadata, Telegram inbound/native-command authorization, docs, and security audit logic. The closed implementation PRs and constansino's chat-only-trigger plugin are useful prior art and workarounds, but they do not ship the requested OpenClaw Telegram access-control behavior.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes new groups based on trusted current members rather than per-group chat IDs.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata support, Telegram inbound and native-command authorization, fail-closed all-current-members verification, security-audit behavior, documentation, and focused regression tests while preserving existing `open`, `disabled`, and `allowlist` semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The missing behavior is high-confidence from static current-main verification across exported config, strict schema, generated metadata, Telegram runtime authorization, native-command authorization, docs, and security audit code.

## Solution Assessment

Unclear until maintainers settle the authorization contract. The durable solution belongs in the bundled Telegram plugin and shared config/docs/security surfaces; the external chat-only-trigger workaround is not equivalent to the requested all-current-members policy.
