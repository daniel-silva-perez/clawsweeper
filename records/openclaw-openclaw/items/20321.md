## Summary

Keep this issue open. Current main still does not implement Telegram `groupPolicy: "members"`: the public config type, strict schema, generated metadata, Telegram runtime authorization helper, inbound/native command paths, security audit logic, docs, and focused tests still expose only `open`, `disabled`, and `allowlist`. The closed related PRs and the linked chat-only-trigger plugin are useful prior art/workarounds, not a bundled implementation.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes newly created Telegram groups based on trusted users rather than pre-allowlisting each group chat ID.

## Best Possible Solution

Keep this open for an explicit bundled Telegram policy plus shared config-contract update. A good fix should add `members` to the exported group-policy type, strict schema, generated metadata/help, Telegram docs/examples, inbound and native-command authorization, and security audit behavior; it should verify all current Telegram group members against trusted numeric IDs with bounded API calls, fail-closed errors, cache TTL, and join/leave invalidation. Treat #12045, #28351, #28392, #33551, and the chat-only-trigger plugin as prior art rather than close evidence.
