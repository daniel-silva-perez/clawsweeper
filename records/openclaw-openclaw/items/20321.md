## Summary

Keep open. Current main still lacks the requested bundled Telegram members policy across the public config contract, validation schema, generated metadata, inbound/native-command authorization, docs, and security audit behavior. The linked closed PRs and constansino's chat-only-trigger plugin are useful prior art/workarounds, but they do not ship the requested Telegram access-control policy.

## What This Changes

The issue requests a bundled Telegram `groupPolicy: "members"` mode that allows newly created groups when every current group member is trusted via `groupAllowFrom`, instead of allowlisting each group chat ID.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata support, Telegram inbound and native-command authorization, fail-closed all-current-members verification, security-audit behavior, documentation, and focused regression tests while preserving existing `open`, `disabled`, and `allowlist` semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The missing behavior is high-confidence from static current-main verification across exported config, strict schema, generated metadata, Telegram runtime authorization, native-command code/tests, docs, and security audit code.

## Solution Assessment

Unclear until maintainers settle the authorization contract. The durable solution belongs in the bundled Telegram plugin and shared config/docs/security surfaces, while the external chat-only-trigger workaround is not equivalent to the requested all-current-members policy.
