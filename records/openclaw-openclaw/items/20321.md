## Summary

Keep this issue open. Current main still exposes only `open`, `disabled`, and `allowlist` for group policy in public config, generated metadata, docs, and Telegram authorization paths; the closed implementation PRs and the chat-only-trigger plugin remain useful prior art/workarounds rather than a shipped bundled `members` policy. Because this changes Telegram group access-control semantics, the remaining work needs maintainer/security judgment before implementation.

## What This Changes

Request to add a Telegram `groupPolicy: "members"` mode that authorizes new groups based on all current participants being trusted users instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy with updated config types/schema/generated metadata, Telegram inbound and native-command authorization, fail-closed membership verification, security-audit semantics, docs, and focused tests while preserving existing `open`, `disabled`, and `allowlist` behavior.
