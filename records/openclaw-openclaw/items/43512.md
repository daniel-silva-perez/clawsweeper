## Summary

Keep open. The linked Feishu regression remains partly live on current main for non-abort text commands, and this PR is a useful contributor attempt, but the proposed patch is not merge-ready: it targets the old `monitor.account.ts` receive path, broadens the fast lane beyond current main’s abort/BTW policy, and introduces destructive debounce cleanup before command authorization.

## What This Changes

The PR adds a Feishu receive-handler fast path for text control commands, destructive-command debounce dropping, test mock updates, and a shared inbound debouncer `dropKey` API.

## Best Possible Solution

Rework the Feishu fix on top of current main’s receive-handler and sequential-key design, explicitly decide which command classes deserve an out-of-band lane, and ensure destructive cleanup only happens after authorization or under a documented fail-closed policy with focused tests for `/stop`, `/new`, `/reset`, `/status`, and `/btw`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Clear queued chat work before destructive commands:** `extensions/feishu/src/monitor.account.ts:438`
  - body: This direct-dispatch path drops only the debounce buffer before running `/stop`, `/new`, or `/reset`. Any normal Feishu message that already flushed into the per-chat queue still runs after the destructive command, so stale text can restart work immediately after a stop/reset and defeat the command's intent.
  - confidence: 0.84
- **[P2] Gate destructive drops on command authorization:** `extensions/feishu/src/monitor.account.ts:435`
  - body: `dropKey` runs from text classification before `handleFeishuMessage` computes `CommandAuthorized`. In a chat where the sender can speak but cannot run destructive slash commands, an unauthorized `/new`, `/reset`, or `/stop` can still discard that sender's pending debounced text.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff has an authorization-adjacent integrity concern because destructive debounce cleanup happens before the normal command authorization result exists.

  - body: The direct-dispatch path drops pending debounce data from command text alone. If the command is later rejected by normal Feishu command authorization, an unauthorized sender can still cause pending chat work to be lost.
