## Summary

Keep this PR open for maintainer handling, but do not merge it as-is. Current main has partially moved Feishu onto a safer sequential-key design for `/stop` and `/btw`, while this PR still proposes a broader direct-dispatch path and has blocking stale-queue and pre-authorization cleanup issues.

## What This Changes

The PR adds a Feishu text control-command fast path in `monitor.account.ts`, adds `dropKey` to the inbound debouncer, and updates Feishu test mocks for the new debouncer API.

## Best Possible Solution

Rework the fix on current main's Feishu sequential-key design: define the exact out-of-band command set, keep authorization-sensitive cleanup after command authorization, preserve normal same-chat FIFO behavior, and add focused tests for `/stop`, `/new`, `/reset`, `/status`, and `/btw` with debounce and already-queued work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: enqueue normal Feishu work behind an active run, then send `/stop`, `/new`, or `/reset`; this PR drops only the debounce buffer and directly dispatches the command, leaving already-queued same-chat work able to run afterward. A second path is an unauthorized destructive command after buffered text, where cleanup occurs before command authorization rejects it.

## Solution Assessment

No. The direct-dispatch plus pre-auth `dropKey` approach is not the best fix because it bypasses the current sequential-key architecture and performs destructive cleanup before authorization; a narrower lane-classification fix on current main is safer.
