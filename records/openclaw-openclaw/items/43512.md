## Summary

Keep open for maintainer review. Current main partially addresses the related Feishu active-run problem for abort-style `/stop` traffic, but it does not implement this PR's broader control-command bypass, and the proposed branch still has blocking ordering and authorization-adjacent defects before it can be merged.

## What This Changes

The PR branch adds a Feishu fast path that detects text control commands before debounce/serialization, dispatches them directly, and adds an inbound debounce `dropKey` API for destructive command buffering.

## Best Possible Solution

Keep the fast lane aligned with the current Feishu sequential-key model: explicitly classify only the commands that are meant to be out-of-band, keep destructive cleanup behind or inside authorization-aware command handling, preserve normal FIFO/debounce behavior, and add focused tests for buffered and already queued same-chat work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: queue normal Feishu work behind an active run, then send `/stop`, `/new`, or `/reset`; this PR can run the destructive command while older same-chat work remains queued to run afterward. A second path is buffered text followed by an unauthorized destructive command, where cleanup happens before command authorization rejects it.

## Solution Assessment

No. The direct-dispatch plus pre-authorization debounce drop is not the narrowest maintainable fix; extending current main's sequential-key policy with authorization-aware cleanup and explicit command classification is safer.
