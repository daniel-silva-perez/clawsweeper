## Summary

Keep open. Current main only partially supersedes the PR by shipping Feishu `/stop` and `/btw` out-of-band lanes, while the PR still proposes a broader direct-dispatch path for text control commands and introduces blocking ordering and authorization-adjacent defects before it can merge.

## What This Changes

The PR branch changes Feishu inbound handling so text control commands bypass debounce and the chat serializer, dispatches them directly, and adds a shared inbound debounce `dropKey` API for command-triggered buffer cleanup.

## Best Possible Solution

Rebase or replace the work on the current `createFeishuMessageReceiveHandler` and sequential-key design: explicitly classify only commands that should be out-of-band, keep destructive cleanup behind authorization-aware command handling, preserve normal FIFO/debounce behavior for ordinary commands, and add focused tests for buffered and already queued same-chat work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: on the PR branch, queue normal Feishu work behind an active run, then send `/stop`, `/new`, or `/reset`; older same-chat work already in the queue can still run afterward, and an unauthorized destructive command can drop buffered text before authorization rejects it.

## Solution Assessment

No. The direct-dispatch plus pre-authorization debounce-drop approach is not the safest maintainable fix; extending the current sequential-key policy with authorization-aware cleanup and explicit command classification is the better path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Clear queued chat work before destructive commands:** `extensions/feishu/src/monitor.account.ts:441`
  - body: This direct path only drops the debounce buffer before running `/stop`, `/new`, or `/reset`. Messages that already flushed into the per-chat queue still run after the destructive command, so stale text can restart work immediately after a stop/reset and defeat the command's intent.
  - confidence: 0.86
- **[P2] Gate destructive drops on command authorization:** `extensions/feishu/src/monitor.account.ts:438`
  - body: `dropKey` runs from text classification before `handleFeishuMessage` computes command authorization. In chats where a sender can speak but cannot run destructive slash commands, an unauthorized `/new`, `/reset`, or `/stop` can still discard pending debounced text.
  - confidence: 0.82
