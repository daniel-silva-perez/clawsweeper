## Summary

Current main still does not implement this PR's broader Feishu control-command queue bypass. Main now routes Feishu `/stop` and `/btw` onto out-of-band sequential lanes, but `/status`, `/new`, `/reset`, `/help`, `/model`, and other command-only messages still enter the inbound debouncer and then the normal per-chat sequential queue. The PR is stale against the current handler split and needs maintainer review, but it is not obsolete cleanup material.

## Best Possible Solution

Keep this PR open as a plausible implementation candidate for the linked Feishu regression. The best path is to rebase or rewrite it on the current `createFeishuMessageReceiveHandler` and `getFeishuSequentialKey` design, decide exactly which command classes should bypass the Feishu per-chat lane, preserve authorization before destructive side effects where feasible, and add regression tests for active-run immediacy plus ordering/drop behavior for `/stop`, `/new`, `/reset`, `/status`, and read-only commands.

## Risks / Open Questions

- The PR likely needs a rebase or rewrite because current main split the Feishu receive path into `monitor.message-handler.ts` plus `sequential-key.ts`, while the PR targets the older `monitor.account.ts` shape.
- Routing every detected text command around the same-chat lane can reorder command effects relative to buffered or queued user text; read-only commands and state-changing commands need separate semantics.
- The proposed destructive cleanup path is security-sensitive: unauthorized `/stop`, `/new`, or `/reset` attempts could discard pending buffered text before command authorization unless the implementation gates or defers that side effect.
- The shared debounce API addition proposed by the PR would need focused contract coverage because it changes a core utility used beyond Feishu.
