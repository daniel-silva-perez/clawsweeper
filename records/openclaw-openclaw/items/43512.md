## Summary

Keep open. This PR addresses a real Feishu queue regression, but current main only partially covers the requested behavior and the PR patch is not safe to merge: it drops destructive-command debounce buffers before command authorization, does not invalidate already queued same-chat tasks, and targets an older receive-handler shape that has since moved into `createFeishuMessageReceiveHandler`.

## What This Changes

The PR adds a Feishu text control-command fast path that bypasses the per-chat queue, drops debounce buffers for `/stop`, `/new`, and `/reset`, and extends the inbound debouncer plus Feishu mocks with `dropKey`.

## Best Possible Solution

Rework the fix on top of current main's `createFeishuMessageReceiveHandler` and sequential-key design, define exactly which Feishu text commands get out-of-band lanes, preserve normal same-chat FIFO behavior, and add focused tests for authorized and unauthorized `/stop`, `/new`, `/reset`, `/status`, and `/btw` interactions with debounce and already queued work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Clear queued chat work before destructive commands:** `extensions/feishu/src/monitor.account.ts:441`
  - body: This direct path drops only the debounce buffer before running `/stop`, `/new`, or `/reset`. Normal Feishu messages that already flushed into the per-chat queue still run after the destructive command, so stale text can restart work immediately after a stop/reset and defeat the command's intent.
  - confidence: 0.84
- **[P2] Gate destructive drops on command authorization:** `extensions/feishu/src/monitor.account.ts:438`
  - body: `dropKey` runs from text classification before `handleFeishuMessage` computes command authorization. In chats where a sender can speak but cannot run destructive slash commands, an unauthorized `/new`, `/reset`, or `/stop` can still discard that sender's pending debounced text.
  - confidence: 0.8

## Security Review
