## Summary

Keep open. This is a real Feishu regression area and the PR is a useful implementation thread, but it is not merge-ready against current main: current main has moved the receive path into `createFeishuMessageReceiveHandler`, only implements the narrower `/stop` and `/btw` out-of-band policy, and the PR introduces concrete stale-work and pre-authorization drop problems.

## What This Changes

The PR adds a Feishu text-control-command fast path, a destructive-command debounce drop path through a new shared `dropKey` API, and Feishu test mock updates.

## Best Possible Solution

Rework the fix on top of current main's Feishu receive-handler and sequential-key design, decide the intended fast-lane command classes explicitly, and add focused tests that preserve authorization, same-chat ordering, `/stop`, `/new`, `/reset`, `/status`, and `/btw` behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Clear queued chat work before destructive commands:** `extensions/feishu/src/monitor.account.ts:438`
  - body: This direct-dispatch path drops only the debounce buffer before running `/stop`, `/new`, or `/reset`. Any normal Feishu message that already flushed into the per-chat queue still runs after the destructive command, so stale text can restart work immediately after a stop/reset and defeat the command's intent.
  - confidence: 0.84
- **[P2] Gate destructive drops on command authorization:** `extensions/feishu/src/monitor.account.ts:435`
  - body: `dropKey` runs from text classification before `handleFeishuMessage` computes `CommandAuthorized`. In a chat where the sender can speak but cannot run destructive slash commands, an unauthorized `/new`, `/reset`, or `/stop` can still discard that sender's pending debounced text.
  - confidence: 0.8
