## Summary

Keep open. Current main only partially covers the linked Feishu queue regression: abort-like `/stop` now has a dedicated out-of-band lane, but this PR's broader `/new`, `/status`, and general control-command bypass is not implemented, and the proposed patch still has correctness and authorization-adjacent issues.

## What This Changes

The PR adds a Feishu text control-command fast path that bypasses the per-chat queue, drops debounce buffers for `/stop`/`/new`/`/reset`, and extends the inbound debouncer API and Feishu mocks with `dropKey`.

## Best Possible Solution

Rework the fix on top of current main's `createFeishuMessageReceiveHandler` and sequential-key design, explicitly define which Feishu text commands get out-of-band lanes, preserve normal same-chat FIFO behavior, and add focused tests for authorized and unauthorized `/stop`, `/new`, `/reset`, `/status`, and `/btw` interactions with debounce and already queued work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Clear queued chat work before destructive commands:** `extensions/feishu/src/monitor.account.ts:438`
  - body: This direct path drops only the debounce buffer before running `/stop`, `/new`, or `/reset`. Normal Feishu messages that already flushed into the per-chat queue still run after the destructive command, so stale text can restart work immediately after a stop/reset and defeat the command's intent.
  - confidence: 0.84
- **[P2] Gate destructive drops on command authorization:** `extensions/feishu/src/monitor.account.ts:435`
  - body: `dropKey` runs from text classification before `handleFeishuMessage` computes `CommandAuthorized`. In chats where a sender can speak but cannot run destructive slash commands, an unauthorized `/new`, `/reset`, or `/stop` can still discard that sender's pending debounced text.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff has an authorization-adjacent integrity concern because destructive debounce cleanup happens before normal Feishu command authorization is computed.

Concerns:

- **[medium] Avoid pre-auth destructive cleanup:** `extensions/feishu/src/monitor.account.ts:435`
  - body: The direct-dispatch path drops pending debounce data from command text alone. If the command is later rejected by normal Feishu command authorization, an unauthorized sender can still cause pending chat work to be lost.
