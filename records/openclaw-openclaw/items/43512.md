## Summary

Keep open for maintainer handling, but do not merge as-is. Current main has a safer Feishu sequential-key path for `/stop` and `/btw`, while this PR still broadens direct dispatch and has concrete stale-queue plus pre-authorization cleanup defects.

## What This Changes

The PR adds a Feishu control-command direct-dispatch path before debounce/queue handling, adds `dropKey` to the shared inbound debouncer, and updates Feishu test mocks for that API.

## Best Possible Solution

Rework any remaining Feishu command bypass on top of the current sequential-key design: define the exact out-of-band command set, keep authorization-sensitive cleanup after command authorization, preserve normal same-chat FIFO behavior, and add focused coverage for debounce plus already-queued work.

## Reproduction Assessment

Yes. A high-confidence static path exists: queue normal Feishu work behind an active run, then send `/stop`, `/new`, or `/reset`; the PR drops only the debounce buffer and directly dispatches the command, leaving already-queued same-chat work able to run afterward. A second path is an unauthorized destructive command after buffered text, where cleanup occurs before command authorization rejects it.

## Solution Assessment

No. The direct-dispatch plus pre-auth `dropKey` approach is not the narrowest maintainable fix; the current sequential-key lane classification is safer and should be extended only after the desired command scope is settled.
