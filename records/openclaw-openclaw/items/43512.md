## Summary

Keep open. Current main partially overlaps by routing Feishu abort-style text and `/btw` onto out-of-band sequential lanes, but this PR's broader direct-dispatch/dropKey approach still has blocking correctness and authorization-adjacent issues, and the linked Feishu command fast-lane scope remains a maintainer decision.

## What This Changes

The PR adds a Feishu command-only fast lane before debounce/serialization, dispatches detected control commands directly, and adds `dropKey` to inbound debounce so destructive commands can discard buffered messages.

## Best Possible Solution

Build any remaining Feishu command fast lane on the current sequential-key design: explicitly classify the out-of-band command set, keep authorization-sensitive cleanup after command authorization, and add focused tests for debounce buffers plus already queued same-chat work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: queue normal Feishu work behind an active run, then send `/stop`, `/new`, or `/reset`; the PR can dispatch the destructive command while older same-chat work remains queued to run afterward. A second path is buffered text followed by an unauthorized destructive command, where cleanup happens before command authorization rejects it.

## Solution Assessment

No. The PR's direct-dispatch plus pre-auth `dropKey` path is not the narrowest maintainable fix; extending current main's sequential-key policy with authorization-aware cleanup is safer.

  - body: This path drops only the debounce buffer before running `/stop`, `/new`, or `/reset`. Messages that already flushed into the per-chat queue still run after the destructive command, so stale text can restart work immediately after a stop/reset and defeat the command's intent.
