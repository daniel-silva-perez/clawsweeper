## Summary

Keep this PR open. Current main still does not implement the proposed HarmonyOS app, gateway client ID, platform docs, setup-code/auth compatibility, or mobile-node policy. The PR is a plausible implementation candidate, but it is XL, product-sensitive, and security-sensitive, with unresolved review findings around bootstrap auth, secure storage, connection state, DevEco/build configuration, and a shell-executing build wrapper.

## Best Possible Solution

Keep this PR open for explicit maintainer product and security review. If OpenClaw wants first-party HarmonyOS support, split or harden the change: add accepted gateway client/platform/mobile-node policy with tests and docs, align setup-code/bootstrapToken auth with iOS/Android, fix the secure storage and connection-state review findings, replace the hvigor wrapper with deterministic non-shell tooling, remove developer-local docs/config drift, add a HarmonyOS labeler/docs surface if accepted, and require DevEco plus real-device validation before merge. If first-party support is not desired, maintainers should say so and incubate the HarmonyOS app outside core until the platform contract is agreed.
