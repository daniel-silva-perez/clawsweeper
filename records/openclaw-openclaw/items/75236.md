## Summary

Keep open. Current main still lacks cursor pagination for both requested RPCs: `sessions.list` accepts only `limit` and filters and returns no `nextCursor` or total, while `agents.files.list` accepts only `agentId` and returns a fixed bootstrap/memory file set. The cursor helper and artifact pattern referenced by the issue are still in open PR #75208 rather than current main, and the `agents.files.list` half needs maintainer scope judgment before this becomes a safe automated repair.

## What This Changes

Add cursor pagination fields to `sessions.list` and `agents.files.list`, reusing the cursor helper and response pattern proposed for artifact RPCs in #75208.

## Best Possible Solution

Settle the shared cursor contract from #75208 first, then add cursor pagination to `sessions.list` across protocol schema, runtime, SDK/client types, docs, generated artifacts, and focused tests. For `agents.files.list`, maintainers should either explicitly opt into the same pagination fields for consistency or document why the fixed bootstrap/memory file list remains unpaginated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence API-shape reproduction: strict schemas reject `cursor` for both RPCs, `sessions.list` only slices by `limit`, and neither result exposes `nextCursor` or `total`.

## Solution Assessment

Unclear. Cursor pagination is a maintainable direction for `sessions.list`, but applying it to `agents.files.list` and choosing the cursor encoding should wait for maintainer sequencing around #75208 and the fixed-scope agent file surface.
