## Summary

Keep open. Current main still lacks cursor pagination for both requested RPCs, and the referenced artifact cursor work remains an open, unmerged PR; the `sessions.list` half is a valid protocol/API gap, while the `agents.files.list` half still needs maintainer scope because that method currently returns a small fixed bootstrap/memory file set rather than an arbitrary workspace listing.

## What This Changes

Request to add `cursor`, `limit`, `nextCursor`, and `total` pagination fields to `agents.files.list` and `sessions.list`, following the cursor pattern proposed for artifact listing.

## Best Possible Solution

Settle the shared cursor contract from #75208 first, then add cursor pagination to `sessions.list` across schema, runtime, SDK/client types, generated protocol artifacts, docs, and focused tests. For `agents.files.list`, either add the same additive fields for consistency after maintainer approval, or explicitly document why the fixed-scope bootstrap file list remains unpaginated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence API-shape reproduction: strict schemas reject `cursor` on both RPCs, and `sessions.list` only applies a one-shot `limit` slice without returning `nextCursor` or a pre-page `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a maintainable direction for `sessions.list`, but applying it to `agents.files.list` before #75208 settles and before maintainers confirm that fixed-scope method should paginate is not clearly the narrowest solution.
