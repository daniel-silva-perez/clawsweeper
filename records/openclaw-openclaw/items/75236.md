## Summary

Keep open. Current main still does not expose cursor pagination on either requested RPC, and the request is tied to a public Gateway protocol contract that needs maintainer sequencing because the referenced artifact cursor PR is still separate from main and `agents.files.list` is currently a fixed bootstrap-file surface.

## What This Changes

Request cursor pagination fields for `agents.files.list` and `sessions.list`, matching the cursor-style API proposed for artifacts listing.

## Best Possible Solution

Settle the shared cursor contract first, then add cursor pagination to `sessions.list` across schema, runtime behavior, SDK/client types, protocol-generated artifacts, docs, and focused tests. For `agents.files.list`, either apply the same fields for protocol consistency or explicitly document why its fixed bootstrap-file result stays unpaginated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence API-shape reproduction: strict schemas reject `cursor` on both RPCs, and the current implementations return full or limit-sliced arrays without `nextCursor` or `total`.

## Solution Assessment

Unclear for the exact requested scope. Cursor pagination is a maintainable direction for `sessions.list`, but `agents.files.list` should wait for maintainer scope and sequencing decisions because it is fixed-size today and the referenced cursor helper is still pending in #75208.
