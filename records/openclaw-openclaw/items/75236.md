## Summary

Keep open. Current main still lacks cursor pagination on both requested RPCs, and the referenced artifact cursor helper/pattern is not present in the checked-out main implementation. The sessions side is a real unbounded-list concern, while the agents files side needs maintainer scope judgment because it currently lists a fixed bootstrap/memory file set rather than arbitrary workspace files.

## What This Changes

Add cursor, limit, nextCursor, and total pagination semantics to the Gateway `agents.files.list` and `sessions.list` RPCs for API consistency with the proposed artifact pagination pattern.

## Best Possible Solution

Settle the shared cursor contract first, then add cursor pagination to `sessions.list` across protocol schema, runtime, SDK/client types, docs, generated artifacts, and focused tests. For `agents.files.list`, maintainers should either apply the same fields for protocol consistency or document why the fixed-scope bootstrap file list remains unpaginated.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the API-shape gap: strict schemas reject `cursor` for both RPCs, and `sessions.list` only slices by `limit` without pagination metadata.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is the maintainable direction for `sessions.list`, but `agents.files.list` should wait for maintainer scope judgment because the current implementation is fixed-size and the referenced artifact cursor contract has not landed on main.
