## Summary

Keep open: the report is fresh, detailed, and current main still contains a plausible silent Discord gateway READY/IDENTIFY failure path. There is also an open linked implementation candidate with closing syntax, so this should not be closed before that work is reviewed, landed, or rejected.

## What This Changes

Investigate and fix an intermittent Discord gateway startup race where the bot appears online but the READY state never makes the Discord channel responsive.

## Best Possible Solution

Land a focused Discord plugin fix that makes gateway startup and IDENTIFY failures observable and restartable, with regression coverage for a delayed identify limiter plus socket close; then close the issue after the fix is merged and, ideally, shipped.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can mock the shared identify limiter delay, feed HELLO, close the gateway socket before `identifyWithConcurrency()` resumes, and assert current main silently skips IDENTIFY/error/reconnect; the reporter also provided a detailed live Docker timeline and raw WebSocket control result.

## Solution Assessment

No, current main is not enough to close this. The existing timeout/restart path helps a stuck READY startup, but the gateway IDENTIFY path can still return silently after a limiter wait, and the linked PR should be reviewed against that concrete failure mode before merge.
