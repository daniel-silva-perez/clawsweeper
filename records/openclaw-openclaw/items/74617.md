## What This Changes

The issue asks OpenClaw to fix a Discord gateway startup race where the bot logs in and appears online, but READY is never observed and the Discord channel remains unresponsive.

## Best Possible Solution

Keep this issue open until the linked PR is reviewed and either merged, replaced with an equivalent fix in `extensions/discord/src/monitor/provider.lifecycle.ts`, or closed with a clear reason. A durable fix should add regression coverage for a websocket that is already open before lifecycle debug listener registration and should consider surfacing or retrying the silent IDENTIFY-drop path in `extensions/discord/src/internal/gateway.ts`.

## Reproduction Assessment

Unclear: the report includes detailed release logs and a raw WebSocket control from the same container, and source inspection identifies matching race surfaces. I did not run a live Discord bot reproduction or find a deterministic current-main test for the missed-open startup window.

Unclear: the linked PR's already-open socket guard is a narrow plausible fix, but it should be reviewed against current `extensions/discord` code and the silent IDENTIFY-return path. Current main's 15s startup wait and one restart are useful but do not fully cover the reported race.
