## Summary

Keep open. Current main still has the same unbounded provider-auth wait before `status --usage --json` can emit JSON, and the issue is paired with open fix PR #74185, which should be reviewed or replaced rather than closing this report early.

## What This Changes

`openclaw status --usage --json` should return machine-readable JSON from non-TTY subprocesses instead of hanging or failing before stdout/stderr is written.

## Best Possible Solution

Land a narrow fix that bounds provider usage auth resolution before JSON status output waits on it, returns parseable degraded JSON when auth resolution times out, and adds a regression test where auth resolution never settles.

## Reproduction Assessment

Yes. The issue includes a concrete non-TTY Node `spawnSync('openclaw', ['status','--usage','--json'])` reproduction, and current main still has a source path where JSON output waits on unbounded provider auth resolution.

## Solution Assessment

Yes for triage: keeping this issue open while PR #74185 or an equivalent narrow fix is reviewed is the right path. The current main implementation is not sufficient because the timeout only applies after auth resolution completes.
