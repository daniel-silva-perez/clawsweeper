## Summary

Keep open. Current main still lets `status --usage --json` wait on provider usage auth before JSON can be written, and the issue is already paired with open fix PR #74185, which references this issue with closing syntax and should be reviewed or replaced rather than closing the report early.

## What This Changes

The issue reports that `openclaw status --usage --json` hangs or fails from a non-TTY subprocess after upgrading to OpenClaw 2026.4.26.

## Best Possible Solution

Land a narrow fix that bounds provider usage auth resolution before JSON status output waits on it, returns parseable degraded JSON when auth resolution times out, and adds a regression test where auth resolution never settles. The active PR #74185 is the current implementation candidate for that path.

## Reproduction Assessment

Yes. The issue includes a concrete non-TTY Node `spawnSync('openclaw', ['status','--usage','--json'])` reproduction, and current main still has a source path where JSON output waits on unbounded provider auth resolution.

## Solution Assessment

Yes for triage: keeping this issue open while PR #74185 or an equivalent narrow fix is reviewed is the right path. The current main implementation is not sufficient because the timeout only applies after auth resolution completes.
