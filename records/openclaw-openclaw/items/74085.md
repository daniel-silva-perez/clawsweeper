## Summary

Keep open. Current main still waits for provider usage auth resolution before writing JSON, and the linked open PR #74185 explicitly tracks the narrow fix, so this issue should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue reports a regression where `openclaw status --usage --json` hangs or exits without output from non-TTY subprocesses, especially with OpenAI Codex OAuth auth resolution.

## Best Possible Solution

Land PR #74185 or an equivalent narrow fix that bounds provider auth resolution inside the provider usage loader, returns parseable degraded JSON instead of hanging or producing empty output, and adds a regression test where auth resolution never settles.

## Reproduction Assessment

Yes. The issue includes a concrete `spawnSync('openclaw', ['status','--usage','--json'])` reproduction, and current source gives a focused unit reproduction by making `resolveProviderAuths` never settle before JSON output is written.

## Solution Assessment

Yes. The narrowest maintainable fix is to bound provider auth resolution in `loadProviderUsageSummary`, where the unbounded await lives and where other usage callers also benefit, rather than adding a subprocess-specific wrapper around the CLI.
