## Summary

Keep this PR open. Current main still sends model-provider guarded fetches through strict DNS-pinned `fetchWithSsrFGuard`, while this PR applies the existing trusted env-proxy pattern only when standard HTTP(S) proxy env vars apply and no explicit provider dispatcher policy exists.

## What This Changes

The PR updates `buildGuardedModelFetch` to select trusted env-proxy guarded-fetch mode for provider model requests without an explicit dispatcher policy, and adds regression tests for env-proxy selection and explicit-policy preservation.

## Best Possible Solution

Land this PR or an equivalent narrow patch after maintainer review and CI, keeping the change in `buildGuardedModelFetch` with focused tests and adding a user-facing changelog entry if maintainers want release-note coverage for the proxy regression.

## Reproduction Assessment

Yes. The focused reproduction is to set `HTTPS_PROXY`, build a model-provider guarded fetch with no explicit transport override, and assert the guarded options enter trusted env-proxy mode; current main inspection shows that path still stays strict.

## Solution Assessment

Yes. Reusing the existing trusted env-proxy guard helper behind `shouldUseEnvHttpProxyForUrl` is the narrowest maintainable fix here, and preserving explicit dispatcher policies avoids overriding direct, TLS, env-proxy, or explicit-proxy provider transport settings.
