## Summary

Keep open. The PR targets a real heartbeat model-selection bug, but the proposed stale detector is incomplete because it relies on fallback notice fields that the runtime can clear while the automatic model override remains persisted.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing, passes heartbeat context into reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Track durable auto-fallback origin metadata, or another explicit changed-default marker, at the time the auto override is persisted so heartbeat can clear only stale direct auto overrides after the configured default changes while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session auto override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing the configured default; the PR's helper returns false and keeps the old fallback selected.

## Solution Assessment

No. The PR is narrow in the right area, but fallbackNoticeSelectedModel is transient display state rather than durable origin metadata, so it is not the best complete way to decide whether an auto override is stale.
