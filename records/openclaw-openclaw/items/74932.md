## Summary

Keep this PR open, but do not merge it as-is. Current main still has the stale heartbeat auto-fallback override path, and the PR is aimed at the right surface, but the new stale check only works while fallback notice metadata is still present; existing runtime code can clear that notice after a successful fallback run while leaving the auto override persisted.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing in reply model selection, threads heartbeat context through directive selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Revise this PR so heartbeat can detect a changed configured default even after the fallback notice UI state has been cleared, likely by keeping durable original selected/default metadata for auto fallback overrides or by another explicit changed-default marker. Keep the fix limited to heartbeat auto overrides, and preserve explicit `heartbeat.model`, user and legacy overrides, parent inheritance, channel selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. A focused model-selection reproduction is available: seed a direct session `modelOverrideSource: "auto"` override with no `fallbackNoticeSelectedModel`, run heartbeat selection after changing the configured default, and the proposed code keeps the stale fallback pinned; current fallback-state code explains how that missing-metadata state can happen after a successful fallback run.

## Solution Assessment

No. The proposed placement is narrow and correct when selected-default metadata exists, but it is not the best complete fix because the state it relies on is not durable for long-lived heartbeat sessions.
