## Summary

Keep this PR open. It targets the right heartbeat model-selection bug, but the proposed stale check depends on fallback notice fields that current main can clear while leaving the auto model override persisted, so the old fallback can still remain pinned after a default-model change.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice selected-model metadata, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata, or an equivalent explicit changed-default marker, when the fallback override is written. Heartbeat selection can then clear only stale direct auto overrides after the configured default changes while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing the configured default; the proposed stale detector returns false and keeps the old fallback selected.

## Solution Assessment

No. The PR changes the right area, but fallback notice fields are transient UI/status state rather than durable origin metadata, so they are not the narrowest reliable basis for deciding whether an auto override is stale.
