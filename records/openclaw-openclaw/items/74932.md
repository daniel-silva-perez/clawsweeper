## Summary

Keep open. The PR addresses a real heartbeat stale-model bug, but the proposed stale check still misses long-lived auto fallback overrides after normal runtime fallback-notice state has been cleared, so the linked heartbeat session can remain pinned to the old fallback model.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice selected-model metadata, threads heartbeat context into model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Revise the PR so heartbeat stale-default detection is based on durable original selected/default metadata, or another explicit changed-default marker, rather than transient fallback notice UI state. Add a regression for an auto override after the fallback notice has been cleared, and keep the behavior scoped to direct heartbeat auto overrides.

## Reproduction Assessment

Yes. Seed a direct session `modelOverrideSource: "auto"` override without `fallbackNoticeSelectedModel`, run heartbeat model selection after changing the configured default, and the PR helper returns false so the stale override remains active.

## Solution Assessment

No. The PR is correctly scoped when notice metadata exists, but it is not complete because the state it depends on is cleared by normal fallback-transition persistence for long-lived sessions.
