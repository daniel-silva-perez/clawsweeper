## Summary

Keep this PR open. It targets a real heartbeat stale auto-fallback problem, but the proposed stale check relies on transient fallback notice fields, so long-lived auto override entries can still stay pinned after the configured default changes.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice selected-model metadata, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Add durable auto-fallback origin metadata, or an equivalent explicit changed-default marker, and use it to clear only stale direct heartbeat auto overrides after the configured default changes while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing the configured default; the PR's stale detector returns false and keeps the old fallback selected.

## Solution Assessment

No. The PR changes the right area, but fallback notice fields are not durable origin metadata; a safer fix should persist or infer the original configured model for the auto override instead of depending on UI notice state.
