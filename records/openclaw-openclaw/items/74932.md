## Summary

Keep open. The PR targets a real heartbeat model-staleness bug linked to #74284, but the proposed stale check only works while fallback notice selected-model state is still present; current runtime can clear that notice while leaving the auto override persisted, so long-lived heartbeat sessions can still remain pinned after the configured default changes.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Revise the implementation so heartbeat stale-default detection uses durable auto-fallback origin metadata, or another explicit changed-default marker, rather than the transient fallback notice UI fields. The final behavior should clear only stale direct heartbeat auto overrides while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session auto override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing the configured default; the proposed code keeps the old fallback selected.

## Solution Assessment

No. The PR is narrow in the right area, but relying on fallbackNoticeSelectedModel is not the best complete fix because that state is cleared during normal successful fallback lifecycle while the auto override remains persisted.
