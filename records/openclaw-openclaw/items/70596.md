## Summary

Keep this PR open. Current main does not implement the intended builtin explicit-local memory prewarm; `startGatewayMemoryBackend` still exits for every non-QMD backend. The PR also has a substantive unresolved review concern: probing the cached default builtin manager during startup can leave a rejected provider initialization promise in the manager that later `memory_search` reuses instead of retrying.

## Best Possible Solution

Keep this PR open and request an update rather than closing it. The best fix is a narrow explicit-local builtin prewarm that does not poison the default cached manager on transient failure: either prewarm through a transient/non-default manager purpose or change provider initialization so rejected init promises are cleared and later real searches can retry. Keep the existing boundary of no prewarm for `provider = "auto"` or remote providers, add a regression test for startup prewarm failure followed by later retry, and separate the unrelated provider test-helper change if it is not required.
