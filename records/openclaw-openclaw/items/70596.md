## Summary

Keep this PR open. Current main still does not implement explicit-local builtin memory prewarm: the post-attach scheduler only schedules QMD startup work, and the startup-memory path still skips non-QMD backends. The PR tracks a real cold-start gap, but it needs an update for current main and for retry-safe prewarming so a transient local-provider startup failure does not poison the cached default manager.

## Best Possible Solution

Keep this PR open and request an update. The best path is a narrow explicit-local builtin prewarm that is scheduled from the current post-attach startup policy, preserves the no-`auto` and no-remote-provider boundary, and avoids poisoning the default cached manager on transient failure by using a transient purpose or by clearing rejected provider initialization promises. Add regression coverage for builtin scheduling, non-local skip behavior, prewarm failure followed by later retry, and keep the provider-runtime test-helper change separate unless it is required for the memory fix.
