## Summary

Keep open. Current main still does not prewarm builtin explicit-local memory embeddings, but this PR is not merge-ready because its prewarm path can poison the cached builtin manager after a transient local provider initialization failure and make later real searches reuse that failure.

## What This Changes

The PR adds gateway startup logic and tests to prewarm builtin `memorySearch.provider = "local"` embeddings, plus a test-helper change accepting OpenAI/Codex missing-auth model variants.

## Best Possible Solution

Update the branch to add a narrow explicit-local builtin prewarm that cannot poison the default cached manager on transient failure, preserves current QMD startup behavior, and includes a failure-then-later-retry regression test. Keep `auto` and remote providers out of startup prewarm unless a separate product decision changes that boundary.

## Reproduction Assessment

Yes. Use builtin memory with `memorySearch.provider = "local"`, make provider initialization fail during startup prewarm, then call `memory_search` in the same process after the provider would be ready; the current manager cache and retained rejected promise provide a high-confidence reproduction path.

## Solution Assessment

No. Prewarming explicit local embeddings is the right target, but doing it through the cached default manager is not the safest implementation; a transient prewarm manager or retry-safe provider initialization state is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Avoid poisoning the cached builtin manager on prewarm failure:** `src/gateway/server-startup-memory.ts:49`
  - body: This calls `probeEmbeddingAvailability()` on the default builtin manager. Prewarm through a transient manager or clear failed init state, and add a failure-then-retry regression test.
