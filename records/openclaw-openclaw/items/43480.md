## Repair Work Prompt

Fix openclaw/openclaw#43480. The macOS VoiceWakeOverlay can enter a SwiftUI render loop because `VoiceWakeOverlayView` directly calls `controller.updateWindowFrame(animate: true)` from `onChange(of: controller.model.attributed)`, and the frame path calls `measuredHeight()`, which mutates `model.isOverflowing` during layout calculation. Keep the change scoped to the macOS VoiceWakeOverlay owner files. Make `measuredHeight()` side-effect free or move the overflow-state update behind a guarded/coalesced main-actor update, and defer/coalesce attributed-text frame updates so SwiftUI view updates do not synchronously trigger more observed state mutations. Add focused regression coverage for the update/coalescing behavior if feasible, plus keep existing overlay lifecycle tests green. Do not change the unrelated menubar watchdog/WorkActivityStore work from PR #60990. Add a user-facing changelog fix entry crediting @gdiab and @m13v if the repair lands. Suggested validation: `swift test --package-path apps/macos --filter VoiceWakeOverlay`, `swift build --package-path apps/macos --product OpenClaw`, `pnpm format:swift`, and manual macOS overlay stress verification by triggering, updating, editing/reordering, and dismissing the voice wake overlay repeatedly with no pinwheel or repeated SwiftUI render-loop warnings. If changed lanes expand beyond targeted macOS proof, run `pnpm check:changed` in Testbox before handoff.
