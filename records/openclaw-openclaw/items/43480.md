## Summary

Keep open: current main still matches the reported synchronous VoiceWakeOverlay update cycle. The overlay view observes attributed text and calls frame measurement synchronously, while measurement still writes observable overflow state that the view reads, so the source-level root cause has not been removed.

## What This Changes

The issue reports an intermittent macOS VoiceWakeOverlay pinwheel caused by synchronous SwiftUI state mutation during overlay measurement and frame updates.

## Best Possible Solution

Fix the macOS app owner code so overlay measurement is pure or schedules any overflow-state update outside the SwiftUI render/update cycle, coalesces attributed-text frame updates, and adds focused regression coverage or documented manual macOS verification for the VoiceWakeOverlay update cycle.

## Repair Work Prompt

Fix the macOS VoiceWakeOverlay render-loop/pinwheel reported in https://github.com/openclaw/openclaw/issues/43480. Current main still has `VoiceWakeOverlayView` synchronously calling `controller.updateWindowFrame(animate: true)` from `onChange(of: controller.model.attributed)`, while `updateWindowFrame()` -> `targetFrame()` -> `measuredHeight()` synchronously writes observable `model.isOverflowing`. Break that SwiftUI update-cycle loop in the macOS overlay owner code. Keep the fix scoped to VoiceWakeOverlay behavior: make measurement pure or defer/guard/coalesce `isOverflowing` updates outside the view update cycle, and coalesce/defer attributed-text frame updates if needed. Preserve existing visual behavior and token/session semantics. Do not change the unrelated menubar watchdog work from PR #60990. Add focused Swift coverage where feasible; if SwiftUI render-loop behavior cannot be unit-tested, add a small deterministic helper test plus document manual macOS verification in the PR body. Because this is a user-facing app hang fix, add a single-line changelog Fixes entry with `Thanks @gdiab` unless maintainers decide the changelog is not needed.
