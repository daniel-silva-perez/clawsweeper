## Summary

Current main still matches the reported VoiceWakeOverlay render-loop path. Attributed text changes synchronously call updateWindowFrame(), targetFrame() synchronously calls measuredHeight(), and measuredHeight() still mutates observable isOverflowing state that the SwiftUI overlay view reads. The related open menubar PR is a different surface and does not supersede this macOS overlay bug.

## Best Possible Solution

Keep this issue open and fix it in the macOS app owner code. The likely fix belongs around VoiceWakeOverlayController and VoiceWakeOverlayView: remove or defer the synchronous observable-state write from measuredHeight(), guard or coalesce isOverflowing changes, defer or coalesce attributed-text frame updates, and add focused regression coverage or documented manual macOS verification for the overlay update cycle. The related menubar PR #60990 should be reviewed on its own merits but does not resolve this VoiceWakeOverlay render-loop report.
