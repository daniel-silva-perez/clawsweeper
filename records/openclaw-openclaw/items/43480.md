## Summary

Keep open: current `origin/main` still contains the reported VoiceWakeOverlay path where SwiftUI observes attributed-text changes, synchronously updates the window frame, and frame measurement mutates observable overflow state that the view reads. The related open menubar-freeze PR does not touch this overlay path, so it is not a superseding fix.

## What This Changes

The issue reports that the macOS VoiceWakeOverlay can enter a SwiftUI render loop because overlay layout measurement mutates observable overflow state while attributed text changes synchronously updates the window frame.

## Best Possible Solution

Fix the macOS overlay owner code so height measurement is pure or overflow-state changes are guarded and scheduled outside SwiftUI view-update handling, coalesce frame updates from attributed-text changes, and add focused regression coverage plus manual macOS verification for repeated VoiceWakeOverlay open/update/reorder cycles.

## Repair Work Prompt

Fix the macOS VoiceWakeOverlay render-loop risk reported in https://github.com/openclaw/openclaw/issues/43480. Current `origin/main` has `VoiceWakeOverlayView` calling `controller.updateWindowFrame(animate: true)` directly from `onChange(of: controller.model.attributed)`, while `VoiceWakeOverlayController+Window.measuredHeight()` mutates `model.isOverflowing`. Break the synchronous SwiftUI update cycle by making measurement pure or by guarding and scheduling/coalescing overflow-state updates outside view-update handling; also defer/coalesce attributed-change frame updates if needed. Keep the change inside the macOS overlay owner surface and avoid unrelated menubar watchdog/`WorkActivityStore` work from PR #60990. Add focused regression coverage for overflow/frame-update behavior where feasible, keep Swift formatting, and add a single-line `CHANGELOG.md` Unreleased Fixes entry with `Thanks @gdiab` if the user-facing freeze behavior changes. Do not change bundled provider/channel/plugin behavior.
