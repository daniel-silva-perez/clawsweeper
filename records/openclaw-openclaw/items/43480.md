## Repair Work Prompt

Fix the macOS VoiceWakeOverlay render-loop risk reported in https://github.com/openclaw/openclaw/issues/43480. Current main has `VoiceWakeOverlayView` calling `controller.updateWindowFrame(animate: true)` directly from `onChange(of: controller.model.attributed)`, while `VoiceWakeOverlayController+Window.measuredHeight()` mutates `model.isOverflowing`; the latest v2026.4.27 tag still contains the same path. Break the synchronous SwiftUI update cycle by making measurement pure or by guarding and scheduling/coalescing overflow-state updates outside view-update handling, and defer/coalesce attributed-change frame updates if needed. Keep the change inside the macOS overlay owner surface and avoid unrelated menubar watchdog or `WorkActivityStore` changes from PR #60990. Add focused regression coverage for overflow/frame-update behavior where feasible, keep Swift formatting, and add a single-line `CHANGELOG.md` Unreleased Fixes entry with `Thanks @gdiab` if the user-facing freeze behavior changes. Do not change bundled provider, channel, or plugin behavior.
