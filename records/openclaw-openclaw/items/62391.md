## Summary

Keep this issue open. Current main still formats delivery-recovery failures through the shared message-only formatter before persisting `lastError`, so Slack `missing_scope` structured fields such as `data.needed` and `response_metadata.scopes` are still dropped. The related Slack formatter PR is open, does not touch the delivery-queue recovery path, and still has a redaction blocker.

## What This Changes

Preserve Slack API structured error details in delivery queue recovery logs and queued `lastError` entries for `missing_scope` failures.

## Best Possible Solution

Add redacted Slack/API error formatting at the Slack outbound boundary, or a generic structured-error formatter if maintainers prefer that seam, so recovery persists actionable fields like `needed`, granted scopes, accepted scopes, code, and retry metadata without leaking tokens. Add a delivery-queue recovery regression test using a Slack-shaped `CodedError`, and keep any retry-policy change for `missing_scope` explicit and covered.

## Reproduction Assessment

Yes. A focused reproduction can enqueue a delivery, make the injected `deliver` reject with a Slack-shaped `Error` containing `data.needed` and `data.response_metadata.scopes`, then run `recoverPendingDeliveries`; current main will persist only `An API error occurred: missing_scope`.

## Solution Assessment

Not applicable as a patch review, but the requested direction is valid. The safest implementation is a redacted Slack-owned formatter/wrapper that feeds the existing core recovery path, or a generic structured formatter if it stays extension-agnostic and redaction-preserving.
