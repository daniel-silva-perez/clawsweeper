## Summary

Keep this issue open. Current main still sends delivery recovery failures through the generic `formatErrorMessage()` path, which only uses `Error.message` plus `.cause` messages for `Error` instances and does not serialize Slack Web API `data.needed` or `data.response_metadata.scopes`. The Slack send code can inspect those fields for the `chat:write.customize` fallback, which confirms the structured fields are available, but that detail is not preserved when recovery writes `lastError` or logs retry failure. The related Slack formatting PR #53966 is relevant and useful context, but it is still open and the current checkout does not contain its proposed Slack error formatter or recovery coverage.

## Best Possible Solution

Keep the issue open and fix the recovery path on main. The conservative path is to add a redacted, dependency-light formatter for Slack Web API/CodedError details, use it before `failDelivery` and recovery logging, add focused recovery tests that assert `needed` and granted/accepted scopes survive into `lastError`, and decide whether `missing_scope` should be classified as a permanent delivery error for queued Slack sends. If #53966 is revived, it should either include this recovery-path coverage or be paired with a small core formatter change that the delivery queue can call without deep-importing Slack internals.

## Risks / Open Questions

- Closing now would leave a real diagnostic gap in the core delivery queue path: Slack scope failures can still be persisted and logged as only `An API error occurred: missing_scope`.
- A fix needs care to avoid leaking tokens or broad response payloads when formatting provider-specific structured errors; it should whitelist safe Slack fields such as code, needed, provided/scopes, acceptedScopes, and retryAfter.
