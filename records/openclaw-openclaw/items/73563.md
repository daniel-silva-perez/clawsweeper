## Summary

Keep this PR open. Current main still has the transcript-redaction bypass the PR is trying to fix, but the latest patch is not ready to merge because it still misses active config propagation on production mirror paths, can report success when the guard blocks a write, and promotes security-sensitive transcript guard plumbing through a public SDK barrel without a clear maintainer API decision.

## What This Changes

The PR wraps several raw session transcript append paths with `guardSessionManager`, adds transcript update-mode and config plumbing, exposes the guard via `agent-harness-runtime`, and expands transcript redaction/idempotency tests.

## Best Possible Solution

Land a corrected, maintainer-reviewed security fix that routes every raw transcript write through the shared guard, preserves transcript update semantics, honors the active runtime config at each production caller, handles guard-blocked writes as non-success, and either keeps the guard on a narrow internal bundled seam or documents and versions it as supported Plugin SDK API.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: current main writes delivered/CLI/gateway/Codex transcript messages through raw `SessionManager.appendMessage`, and the PR findings can be reproduced by using a custom redaction config on outbound mirror paths or a blocking `before_message_write` hook on the newly guarded helper.

## Solution Assessment

No, not yet. The overall direction is the right narrow fix for the transcript sink family, but this patch still needs author fixes plus maintainer/security API review before it is the best way to solve the issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread config through mirror callers:** `src/config/sessions/transcript.ts:168`
  - body: Adding `config` to the transcript helper does not make production mirror writes honor it. `deliver.ts` and `outbound-send-service.ts` both have active `cfg`, but their append calls still omit it, so `logging.redactSensitive: "off"` and custom `logging.redactPatterns` are silently ignored on delivered-message transcript mirrors. Thread the active config through those callers as part of this security fix.
  - confidence: 0.88
- **[P2] Handle blocked guarded appends before returning success:** `src/config/sessions/transcript.ts:286-287`
  - body: After this helper wraps the session manager, `appendMessage` can return `undefined` when the guard drops a message or a `before_message_write` hook blocks it. The helper still returns `{ ok: true, messageId }`, which can report success with no persisted transcript entry and an undefined id. Check the append result and return a failure/no-op result instead.
  - confidence: 0.86
- **[P2] Keep transcript guard out of the public runtime barrel:** `src/plugin-sdk/agent-harness-runtime.ts:129`
  - body: Exporting `guardSessionManager` from `openclaw/plugin-sdk/agent-harness-runtime` makes internal SessionManager redaction, blocking, and transcript-update plumbing part of the public Plugin SDK surface. Either keep this on a narrow bundled/internal seam or document and version it as a supported security contract before syncing the baseline.
  - confidence: 0.82
