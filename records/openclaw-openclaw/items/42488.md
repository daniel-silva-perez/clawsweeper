## Summary

Keep this issue open. Current main still rejects the requested edit-tool alias forms: the runtime contract is canonical `path` plus `edits: [{ oldText, newText }]`, and tests explicitly assert that `old_text`/`newString` legacy-style payloads fail. The related implementation PRs in the provided context are all closed unmerged, so this remains an actionable compatibility/product decision rather than cleanup closure.

## Best Possible Solution

Keep this issue open for an explicit edit-tool compatibility decision. If maintainers want alias tolerance, implement it against the current `edits[]` schema by normalizing `old_text`/`new_text` and the reported `oldString`/`newString` variants into canonical `oldText`/`newText`, update provider/tool schema exposure as needed, and replace the legacy-alias rejection tests with acceptance coverage. If canonical-only behavior is intentional, close manually with that rationale.
