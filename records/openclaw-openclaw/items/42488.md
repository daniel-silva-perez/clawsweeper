## Summary

Keep this issue open. Current main does not implement `old_text`/`new_text` aliases for the edit tool; it has moved the edit contract to canonical `path` plus `edits: [{ oldText, newText }]`, and tests now explicitly reject legacy alias-style payloads including `old_text` and `newString`. The related PR attempts were closed unmerged, and the later user comment broadens the compatibility concern to `oldString`/`newString`, so this still needs maintainer product/API judgment rather than cleanup closure.

## Best Possible Solution

Keep this issue open for an explicit edit-tool compatibility decision. If maintainers want alias tolerance, implement it against the current `edits[]` schema by normalizing `old_text`/`new_text` and the reported `oldString`/`newString` variants into canonical `oldText`/`newText`, expose the accepted shape in provider schemas, and update the legacy-alias rejection tests. If canonical-only behavior is intentional, close manually with that rationale.
