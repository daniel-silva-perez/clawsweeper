## Summary

Keep open. Current main has finite per-run fallback chains, profile cooldowns, same-provider rotation caps, and usage reporting, but it still does not implement the central safety request: a hard cross-provider spend/rate guard or user-visible lockout after repeated auth failures. The issue is recent, concrete, and only broadly related to open rate/budget issue #13615, not superseded by it.

## Best Possible Solution

Treat this as an active safety/regression investigation in the model failover and auth-profile layers. Add a reproducible simulated case for repeated 401/auth failures across a configured fallback chain, then implement a hard per-session or per-time-window cross-provider failover budget, user-visible lockout/alert when all profiles or providers fail auth, and documented semantics for auth_permanent, billing, cooldowns, and spend/rate guards. Keep #13615 as the broader outbound rate/budget control umbrella, but do not close this concrete incident until the auth-failover path is fixed or explicitly merged into a maintainer-approved canonical thread.
