## Summary

Keep open. Current main has finite per-run fallback chains, profile cooldowns, and partial profile-rotation caps, but it still does not implement the central safety request: a spend/rate guard or user-visible lockout that stops autonomous cross-provider model calls after repeated auth failures. The broader rate/budget request in #13615 is related, but this issue remains a concrete recent auth-failover incident that should not be swept closed yet.

## What This Changes

The issue requests safeguards after v2026.3.28 reportedly allowed autonomous auth-failover loops to consume monthly budget across configured LLM providers without user action.

## Best Possible Solution

Treat this as an active billing-safety regression. The desired end state is a tested auth-failover guard that simulates repeated 401/auth failures across a configured fallback chain, enforces a per-session or per-time-window cross-provider failure budget, surfaces a clear user-visible lockout/alert when all relevant profiles fail auth, and documents how that interacts with broader API budget controls in #13615.

## Reproduction Assessment

Unclear: the issue provides enough concrete logs and configuration context to build a high-confidence simulated repro, but there is not a safe live reproduction path in this read-only review without paid provider accounts. A focused test can force repeated 401 FailoverError results across configured fallback providers.

## Solution Assessment

No: current main’s finite candidate chain and cooldowns are useful partial safeguards, but they are not the best complete solution to the reported spend failure. The safer solution is a targeted auth-failover lockout/alert coordinated with the broader rate and budget controls tracked in #13615.
