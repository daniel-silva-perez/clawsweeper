## Summary

Keep open. Current main has partial safeguards around finite fallback chains, profile cooldowns, and profile-rotation caps, but it does not implement the central safety request: a spend/rate guard or user-visible lockout that stops autonomous cross-provider model calls after repeated auth failures. The reported v2026.3.28 incident remains a concrete, recent safety/regression report rather than a closeable stale or duplicate item.

## Best Possible Solution

Treat this as an active safety/regression investigation. The best fix likely belongs in the agent/model failover and auth-profile layers: reproduce or simulate repeated 401/auth failures across a configured fallback chain, add a hard per-session or per-time-window cross-provider failover budget, add a user-visible lockout/alert when all profiles for a provider or all configured providers fail auth, consider whether repeated invalid-bearer 401s should escalate to auth_permanent/disabled behavior, and document the resulting cooldown/spend-guard semantics. #13615 can cover the broader outbound rate/budget controls, but this issue should remain as the concrete auth-failover incident until that path is fixed or explicitly merged into a canonical maintainer-owned thread.
