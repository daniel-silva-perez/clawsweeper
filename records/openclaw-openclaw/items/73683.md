## Summary

Keep this issue open. Current main still has no durable pre-dispatch user-turn append for WebChat `chat.send`, and the OpenAI-compatible HTTP path also enters the embedded runner with only the prompt text. The newer comment is useful evidence that this is not Z.AI-specific; strict providers such as Anthropic can hit the same empty-messages boundary. No open PR currently appears to reference this issue with a fix.

## Best Possible Solution

Fix this in the core Gateway-to-embedded-runner path so a new-session WebChat `chat.send` and `/v1/chat/completions` request cannot make a first provider call without at least the accepted current user turn in the active session/provider payload. Use `SessionManager.appendMessage` or a shared runner ingress seam that preserves Pi parentId semantics; keep provider-specific workarounds out of the core path. Add focused regression coverage with a strict fake provider/stream function that fails if the first payload has only system context or zero non-system messages.
