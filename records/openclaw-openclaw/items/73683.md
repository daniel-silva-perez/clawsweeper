## Summary

Keep open. Current main still emits the WebChat user turn as a transcript update event and immediately dispatches the agent run, but I did not find a source-backed guarantee that the accepted user message is durably appended before the first embedded provider submission. The reported Z.AI/GLM strict-provider failure remains plausible and needs a code fix plus regression coverage.

## Best Possible Solution

Fix this in the core WebChat/Gateway to embedded-runner path. The accepted `chat.send` user turn should be appended through `SessionManager.appendMessage` before dispatch, or the embedded runner should receive an equivalent guaranteed current user message before first provider submission. Add a focused regression test for a brand-new WebChat session with delayed transcript update behavior and a strict-provider payload assertion; keep provider-specific workaround logic out of the core path except as test coverage.
