## Summary

Keep open. Current main still lacks a source-backed guarantee that a new WebChat turn is durably appended or otherwise carried into the first strict OpenAI-completions provider request before dispatch. The report is recent and concrete, and the follow-up comment narrows the remaining uncertainty to the WebChat plus ZAI path rather than proving the bug fixed.

## What This Changes

The issue requests that new WebChat bootstrap/reset sessions never submit a ZAI GLM request without at least one non-system user or assistant message.

## Best Possible Solution

Fix the generic WebChat/Gateway to embedded-runner handoff so the accepted `chat.send` user turn is guaranteed to reach the first provider submission and, when persisted, uses the Pi session manager contract. Add a focused regression test that inspects the actual strict OpenAI-completions/ZAI-like payload for a new empty WebChat session.

## Reproduction Assessment

Yes for the reported released setup: the issue includes repeated ZAI GLM failures and direct endpoint checks showing system-only requests fail while system-plus-user requests succeed. Current main still needs a strict mocked or live ZAI payload reproduction because the trajectory field is a pre-prompt snapshot.

## Solution Assessment

Unclear for the exact implementation, but the requested direction is broadly right. The narrowest maintainable solution is to assert the real first provider payload, then guarantee the current user turn before dispatch/provider submission rather than adding a ZAI-only workaround.
