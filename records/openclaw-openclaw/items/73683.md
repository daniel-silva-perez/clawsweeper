## Summary

Keep this issue open. Current main still has no source-backed guarantee that the WebChat `chat.send` user turn is durably appended before agent dispatch, and there is no regression test for an empty new WebChat session against a strict provider. The `prompt.submitted.messages=[]` trajectory field appears to be a pre-prompt session-history snapshot, so maintainers should verify the exact provider payload during the fix, but that nuance is not enough to close the report.

## Best Possible Solution

Fix this in the core WebChat/Gateway to embedded-runner path. The accepted `chat.send` user turn should either be appended with `SessionManager.appendMessage` before dispatch, or the embedded runner should receive an equivalent guaranteed current user message before first provider submission. Add a focused regression test for a new empty WebChat session with a delayed transcript update and a strict provider payload assertion, and keep provider-specific workaround logic out of the core fix except as test coverage.
