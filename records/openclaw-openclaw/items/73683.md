## Summary

Keep open. Current main still lacks a source-backed guarantee that a new WebChat user turn is durably appended or otherwise carried into the first strict-provider request before dispatch, and the existing coverage only checks that a transcript update is emitted. The `prompt.submitted.messages=[]` evidence is not definitive because that event records pre-prompt session history, but the different HTTP-plus-Anthropic reproduction in the comments does not prove the reported WebChat-plus-ZAI path is fixed.

## What This Changes

The issue requests that new WebChat bootstrap/reset sessions never submit a provider request without at least one non-system message for strict ZAI GLM openai-completions models.

## Best Possible Solution

Fix the generic WebChat/Gateway to embedded-runner handoff so the accepted `chat.send` user turn is guaranteed to reach the first provider submission and, when persisted, goes through the Pi session manager contract. Add a focused regression test with a strict OpenAI-completions/ZAI-like payload assertion for a new empty WebChat session.

## Reproduction Assessment

Yes for the reported released setup: the issue includes repeated ZAI GLM-5.1/GLM-5-Turbo failures plus direct endpoint checks showing system-only requests fail while system-plus-user requests succeed. Current main still needs a strict mocked or live ZAI payload reproduction because the trajectory field is a pre-prompt snapshot.

## Solution Assessment

Unclear for the exact implementation, but the requested direction is broadly right: solve this at the WebChat/Gateway runner boundary, not with a ZAI-only workaround. The narrowest maintainable fix is to assert the real first provider payload and then guarantee the current user turn before dispatch or provider submission.
