## Summary

Keep this issue open. Current main still has the reported race shape: WebChat `chat.send` emits the user transcript update asynchronously, immediately starts the dispatch path, and the embedded runner records/submits from the current `activeSession.messages` before `activeSession.prompt(...)` runs. Existing coverage proves the user transcript update is eventually emitted, not that the first provider submission has a user turn. Z.AI is a bundled provider and `zai/glm-5.1` is documented as the default model, so this is not an external-only provider concern.

## Best Possible Solution

Fix this in the core WebChat/Gateway to embedded-runner path: make the accepted `chat.send` user turn durably appended or injected into the runner-visible history before the first provider submission. Add a targeted regression test with a new empty WebChat session and delayed transcript update that fails if `prompt.submitted` or the model call sees zero conversation messages.
