## Summary

Current main fixes the strict-provider cold WebChat reset path by giving bare `/new` and `/reset` turns a non-empty transcript prompt marker, while moving bootstrap/startup instructions into runtime context. That means the embedded runner submits a non-empty user prompt to the provider even when the session history is empty; the `prompt.submitted.messages=[]` trajectory field is the pre-prompt replay history, not the final OpenAI-compatible provider payload.

## Best Possible Solution

Keep the current-main non-empty reset marker and runtime-context split. The remaining product path is to ship this fix in the next release; no additional core issue is needed for the zero-current-user-message WebChat bootstrap/reset failure.
