## Summary

Keep open. The report is recent, concrete, and not protected by a paired fix PR; current main still lacks a WebChat plus strict ZAI/OpenAI-completions payload regression proving that the first request always contains a non-system message. Source and dependency inspection also show that `prompt.submitted.messages=[]` is a pre-prompt history snapshot rather than definitive provider-payload proof, so the next step should prove the actual outbound payload before changing behavior.

## What This Changes

The issue reports that a new WebChat session can dispatch a first strict ZAI/GLM provider request without any user or assistant message, producing a 400 error.

## Best Possible Solution

Add a focused regression that intercepts the outbound OpenAI-completions payload for a fresh WebChat session using a strict ZAI-like model and asserts at least one non-system message is sent on the first request. If that test fails, fix the generic WebChat/Gateway to embedded-runner handoff so the accepted `chat.send` user turn is carried into provider submission without relying on a ZAI-specific workaround or raw transcript writes.

## Reproduction Assessment

Yes for the released setup described by the reporter: the issue includes repeated ZAI GLM failures and direct endpoint checks showing system-only requests fail while system-plus-user requests succeed. For current main, reproduction is unclear because the available current-main instrumentation covers HTTP plus Anthropic, not the original WebChat plus ZAI/OpenAI-completions path.

## Solution Assessment

Unclear until the actual current-main ZAI/OpenAI-completions payload is asserted. The maintainable direction is to prove and, if needed, fix the generic WebChat/Gateway payload boundary rather than injecting a provider-specific synthetic message based only on the trajectory snapshot.
