## Summary

Keep this issue open. Current main still implements overflow compaction as bounded prompt retry, not as a stable resumed turn with durable inbound-message identity or a reply-obligation ledger. Recent mitigations reduce duplicate transcript growth and cap some retry loops, but they do not satisfy the reported cron/system prompt reinjection contract.

## Best Possible Solution

Keep this issue open and fix the embedded runner/subscriber compaction retry contract. Preserve a stable current-turn identity across compaction, carry retry attempt and original inbound message metadata into the resumed attempt, prevent cron/system prompt reconstruction from being treated as fresh user input, and add regression coverage for a cron tool-failure cascade that compacts without re-executing deterministic failed tool calls. Keep #59618 and #63216 cross-linked as related compaction-resume work; treat #66128 as only a trigger-path mitigation.
