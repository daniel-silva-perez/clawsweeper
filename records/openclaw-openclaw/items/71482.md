## Summary

Keep open. The reported data-loss path is credible, but the proposed guard is too broad because `loadSessionStore()` also returns `{}` for valid empty, missing, and invalid stores; a failure-aware implementation with regression tests is still needed before merge.

## What This Changes

This PR adds an empty-session-store guard in `extensions/memory-core/src/dreaming-narrative.ts` that warns and skips orphan cleanup for that agent.

## Best Possible Solution

Keep the data-loss protection, but make it failure-aware: skip orphan archival only when the session store read/parse was not trustworthy, while allowing a valid empty store to continue through stale dreaming transcript cleanup. Add regression coverage for corrupt/missing store protection and valid `{}` cleanup.

## Reproduction Assessment

Yes. Current main can be reasoned to reproduce the reported data-loss path with an invalid `sessions.json` plus old dreaming transcripts; the PR regression is reproducible with a valid `{}` store and an old unreferenced dreaming `.jsonl` file that should still be archived.

## Solution Assessment

No. The direction is right, but using `Object.keys(store).length === 0` is not the narrowest maintainable signal because `{}` is also a valid empty store result; the fix should expose or compute load trust explicitly.

## Review Findings
