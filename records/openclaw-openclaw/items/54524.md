## Summary

Keep open. Current main still allows the reported overlap, and the public docs still describe session transcript indexing and the bundled session-memory hook separately without warning that they can index the same conversation twice. Related PR #40618 is open and only a partial mitigation.

## What This Changes

Clarify or mitigate duplicate memory indexing when `memorySearch.experimental.sessionMemory` with `sources: ["memory", "sessions"]` is combined with the bundled `session-memory` hook.

## Best Possible Solution

Add public guidance in the memory search/config docs and session-memory hook docs explaining the overlap, recommending `sources: ["memory"]` when relying on the hook or disabling/redirecting the hook when indexing raw session transcripts. A later runtime change can add provenance-aware dedupe or configurable output without changing current defaults.

## Repair Work Prompt

Fix the docs/design gap for openclaw/openclaw issue #54524. Current behavior allows `agents.*.memorySearch.experimental.sessionMemory: true` with `sources: ["memory", "sessions"]` while the bundled `session-memory` hook writes session-derived Markdown into `<workspace>/memory/`, so the same conversation can be indexed through both the `sessions` and `memory` sources.
