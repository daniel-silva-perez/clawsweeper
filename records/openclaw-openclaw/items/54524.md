## Summary

Keep open. Current main still has the reported overlap: the session-memory hook writes session-derived Markdown into workspace memory, memory search can index both memory files and session transcripts, and search deduplication is keyed to chunk ids rather than shared session provenance. The public docs still describe both features separately without warning users not to combine them, and related PR #40618 is open, unmerged, and only a partial mitigation.

## What This Changes

The issue requests docs guidance or deduplication for duplicate memory indexing when the session-memory hook and memorySearch session transcript indexing are enabled together.

## Best Possible Solution

Add public docs guidance in the memory config and session-memory hook docs that these two mechanisms overlap: use `sources: ["memory"]` when relying on the session-memory hook, or disable/redirect the hook when indexing raw session transcripts. A later code fix can add provenance-aware dedupe or a configurable hook output directory without changing current defaults.

## Repair Work Prompt

Add a docs-only clarification for the session-memory and memorySearch overlap. Explain that the bundled `session-memory` hook writes session-derived Markdown into `<workspace>/memory/`, while `memorySearch.sources: ["memory", "sessions"]` with `experimental.sessionMemory: true` indexes both those memory files and raw session transcripts. Recommend `sources: ["memory"]` when relying on the hook, or disabling/redirecting the hook when indexing transcripts directly. Keep wording precise: summaries and transcripts overlap semantically but are not byte-identical, and users may still keep hand-written memory notes alongside transcript indexing. Update `docs/reference/memory-config.md`, `docs/automation/hooks.md`, and `src/hooks/bundled/session-memory/HOOK.md` if the bundled hook metadata should mirror the docs. Do not implement outputDir, query-time dedupe, config migrations, or behavior changes in this PR. Coordinate wording with related PR #40618 if it has landed by the time the repair runs. Changelog is not required for a docs-only clarification unless maintainers request it.
