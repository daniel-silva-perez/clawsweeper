## Summary

Keep open. Current main still allows the reported overlap: memory search can index both workspace memory files and session transcripts, the session-memory hook still writes session-derived Markdown into workspace memory, and search deduplication is keyed by chunk id rather than shared session provenance. The public docs still describe the two features separately without warning users not to combine them.

## What This Changes

The issue requests docs guidance or design changes to prevent duplicate indexing when the session-memory hook and `memorySearch.sources: ["memory", "sessions"]` are enabled together.

## Best Possible Solution

Add public docs guidance in the memory search/config docs and session-memory hook docs that these mechanisms overlap: use `sources: ["memory"]` when relying on the session-memory hook, or disable/redirect the hook when indexing raw session transcripts. A later code change can add provenance-aware dedupe or a configurable hook output directory without changing current defaults.

## Repair Work Prompt

Fix the docs for openclaw/openclaw issue #54524. Document that the bundled `session-memory` hook writes session-derived Markdown under `<workspace>/memory/`, while `memorySearch.experimental.sessionMemory` plus `sources: ["memory", "sessions"]` indexes raw session transcripts separately, so enabling both can duplicate or near-duplicate the same conversation in memory search results and embedding work.
