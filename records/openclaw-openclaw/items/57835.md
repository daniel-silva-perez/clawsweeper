## Summary

Keep open. Current main still does not apply this PR's intended gateway fallback: both gateway helper paths populate the top-level agent name from `agents.list[].name` only, while `identity.name` remains nested. The linked UI dropdown path has its own fallback, but that does not cover gateway API, CLI status, or usage/status consumers that read the top-level name.

## What This Changes

The PR adds an `identity.name` fallback to `listGatewayAgentsBasic` and `listAgentsForGateway`, with regression tests for fallback, explicit-name precedence, and missing-name cases.

## Best Possible Solution

Land a refreshed version of this PR if maintainers want gateway summaries to expose the same human-readable fallback as UI labels, preserving explicit `agents.list[].name` precedence and keeping the regression coverage on both helper paths.

## Reproduction Assessment

Yes. A minimal config with `agents.list[].identity.name` and no `agents.list[].name` statically exercises the current code path: main keeps the top-level gateway `name` undefined in both helper outputs.

## Solution Assessment

Yes. The proposed `configuredName ?? identity.name` behavior is the narrowest maintainable fix for the requested gateway summary contract, with explicit configured names still authoritative; the only follow-up is maintainer validation after rebase.
