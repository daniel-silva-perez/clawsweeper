## Summary

Keep open. Current main still defines and injects the exact reported pruned-image marker as ordinary model-visible text, WebChat still accepts and forwards that literal marker as normal user text, and the Kimi Coding legacy model path remains present. The adjacent multimodal-history work helps stale media refs, but it does not close this marker-recursion report.

## What This Changes

The issue requests that OpenClaw prevent Kimi Coding from recursively echoing the internal pruned-image marker when a user sends that literal text.

## Best Possible Solution

Keep the issue open and fix the replay/sanitization boundary so OpenClaw does not send the internal pruned-image marker as an ambiguous ordinary text token. The fix should add a focused regression for user-authored literal marker text and preserve the existing stale-media pruning contract.

## Reproduction Assessment

No, not end-to-end. Static inspection gives a high-confidence path showing the literal marker is accepted by WebChat and injected into provider replay, but the Kimi recursive output itself still needs a live Kimi run or a targeted mocked regression to prove the final behavior.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is in the agent replay/sanitization layer, not a broad provider-specific output filter or closing this under the broader #66702 stale-image issue.
