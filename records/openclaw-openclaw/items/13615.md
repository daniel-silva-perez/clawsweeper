## Summary

Keep open. Current main still has adjacent provider usage readouts, auth-profile cooldowns, rate-limit fallback, and gateway auth throttling, but it does not implement the requested configurable outbound API rate limits, pre-dispatch queue/block behavior, token or cost budget enforcement, warnings/alerts, or `openclaw limits` CLI. The issue discussion also has maintainer signal to keep this as the canonical broad tracker, with closed unmerged PR #13686 as prior art rather than shipped behavior.

## What This Changes

Add configurable rate limiting, throttling, retry/backoff, budget controls, observability, and CLI status/actions for external API calls across providers and integrations.

## Best Possible Solution

Keep this as the canonical core design tracker. The desired end state is a maintainer-scoped, opt-in outbound policy layer that starts with model-provider dispatch, explicitly defines inclusion criteria for tools/search/channels/plugins, and aligns schema, persistence, status/CLI UX, warning thresholds, 429 backoff behavior, token/cost budgets, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature/design request rather than a concrete runtime bug. The high-confidence gap check is current-main inspection of config types/schema, provider transport, CLI registration, docs, and related issue/PR context.

## Solution Assessment

Unclear. The requested direction is valid, but an all-surfaces limiter/budget implementation is broad; the safer path is maintainer-scoped core policy design first, with a narrow initial model-provider enforcement boundary and explicit follow-ups for other outbound surfaces.
