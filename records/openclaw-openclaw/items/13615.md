## Summary

Keep open. Current main has adjacent provider 429 classification, auth-profile cooldown/fallback handling, provider usage readouts, and gateway/control-plane rate limits, but it still lacks the requested configurable outbound API rate limits, queueing/blocking, token or cost budget enforcement, warning thresholds, alerts, and `openclaw limits` CLI. The discussion also has maintainer signal to keep this as the broad canonical tracker, while prior implementation work was closed unmerged.

## What This Changes

Request configurable rate limiting, throttling, status, alerts, and token/cost budget controls for outbound external API calls across model providers, web search, and messaging integrations.

## Best Possible Solution

Keep this as the canonical core design tracker. The desired end state is a maintainer-scoped, opt-in outbound policy layer that starts with the model-provider dispatch path, explicitly defines inclusion criteria for tools/search/channels/plugins, and aligns schema, persistence, status/CLI UX, warning thresholds, 429 backoff behavior, token/cost budgets, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature/design request rather than a concrete runtime bug. The high-confidence gap check is the current-main inspection of config types/schema, provider transport, CLI registration, docs, and related issue/PR context.

## Solution Assessment

Unclear. The requested direction is valid, but an all-surfaces limiter/budget implementation is broad; the safer path is maintainer-scoped core policy design first, with a narrow initial model-provider enforcement boundary and explicit follow-ups for other outbound surfaces.
