## Summary

Keep open. Current main still has adjacent reactive provider cooldown/fallback handling, provider usage/status readouts, gateway auth throttling, and channel-specific retry helpers, but it does not implement the requested configurable outbound API rate limits, pre-dispatch queue/block behavior, token or cost budget enforcement, warning/alert thresholds, or `openclaw limits` CLI.

## What This Changes

The issue requests configurable outbound API rate limiting, throttling/backoff, token or cost budgets, usage visibility, and `openclaw limits` CLI support for external API calls.

## Best Possible Solution

Keep this as the canonical core design tracker. The desired end state is a maintainer-scoped, opt-in outbound policy layer that starts with model-provider dispatch, explicitly defines which tool/search/channel/plugin calls are in scope, and aligns schema, persistence, status/CLI UX, warning thresholds, 429 backoff behavior, token/cost budgets, docs, and focused tests.

Not applicable as a feature/design request rather than a concrete runtime bug. The high-confidence gap check is current-main inspection of config types/schema, provider transport, CLI/docs surfaces, reactive fallback behavior, and related issue/PR context.

## Solution Assessment
