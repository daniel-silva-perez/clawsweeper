## Summary

Keep open. Current main still does not persist isolated cron terminal session status, so the PR targets a real bug, but the current PR head is merge-conflicted and the diff introduces correctness regressions around delivery failures, setup failures, and session-store write failures.

## What This Changes

The PR adds isolated cron session status updates for running/done/failed/timeout states and regression tests for success, fatal payload, thrown-error, and abort paths.

## Best Possible Solution

Land a rebased isolated-cron status persistence fix that records running only once the execution window is protected, derives the terminal stored status from the final run plus delivery outcome, and treats session-store status writes as logged best-effort side effects.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: isolated cron runs currently persist session metadata but never write terminal `sessionEntry.status`, matching the linked session-store report for rows stuck as running.

## Solution Assessment

No. The PR is in the right owner module, but it persists terminal status before delivery can change the run result and lets status-store write failures change the function contract. A small helper used after final outcome classification, with best-effort logging, is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist status after delivery outcome is known:** `src/cron/isolated-agent/run.ts:570-571`
  - body: The new `done`/`failed` write happens before fallback/direct delivery runs. When `dispatchCronDelivery` returns an error result, this function returns `status: "error"` while the session store has already been persisted as `"done"` for a non-fatal payload, so status-based monitoring will misclassify failed cron runs.
  - confidence: 0.88
- **[P2] Move running status inside the protected run window:** `src/cron/isolated-agent/run-session-state.ts:67`
  - body: Adding `status = "running"` to the pre-run helper makes that value persist before later setup work such as auth-profile resolution. If one of those setup steps throws before the run-level `try/catch`, the stored cron session can stay stuck as `running`.
  - confidence: 0.82
- **[P2] Guard terminal status writes from masking run results:** `src/cron/isolated-agent/run.ts:714`
  - body: The catch and timeout paths now await `persistSessionEntry()` before returning the original run outcome. If the session-store write fails, `runCronIsolatedAgentTurn` rejects and masks the structured `{ status: "error" }` result callers currently receive.
  - confidence: 0.8
