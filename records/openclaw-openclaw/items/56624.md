## Summary

Keep open. Current main still does not persist isolated cron run terminal status, and the PR targets the right runtime boundary, but the current PR branch is dirty/unmergeable and the patch has correctness gaps that can misclassify delivery failures or leave sessions stuck as running.

## What This Changes

The PR changes isolated cron session state handling to set a running status, persist done/failed/timeout terminal states, and add focused regression tests for successful, fatal-error, thrown-error, and timeout paths.

## Best Possible Solution

Land a rebased isolated-cron status persistence fix that records running only inside a protected execution window, derives the final stored status from the final run plus delivery outcome, and handles status-store write failures as logged best-effort side effects without changing the structured cron result.

## Reproduction Assessment

Yes. The code path is high-confidence: current main creates and persists isolated cron session metadata but never assigns a terminal `sessionEntry.status` in the success, fatal payload, abort, or thrown-error return paths. The linked report’s session-store inspection steps align with those source paths.

## Solution Assessment

No. The PR is aimed at the correct owner module, but this patch persists `done`/`failed` before delivery can fail and persists `running` before later setup failures are caught. The safer fix is a small status persistence helper used after final outcome classification, with best-effort logging around store writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist status after delivery outcome is known:** `src/cron/isolated-agent/run.ts:570-571`
  - body: The new `done`/`failed` write happens before fallback/direct delivery runs. When `dispatchCronDelivery` returns an error result, this function returns `status: "error"` but the session store has already been persisted as `"done"` for a non-fatal payload, so status-based monitoring will misclassify failed cron runs.
  - confidence: 0.88
- **[P2] Protect running status from setup failures:** `src/cron/isolated-agent/run-session-state.ts:67`
  - body: Adding `status = "running"` to the pre-run helper makes that value persist before later setup work such as auth-profile resolution. If one of those setup steps throws before the run-level `try/catch`, the stored cron session can stay stuck as `running`.
  - confidence: 0.82
- **[P2] Keep terminal status persistence best-effort:** `src/cron/isolated-agent/run.ts:714`
  - body: The catch path now awaits `persistSessionEntry()` before returning the original run error. If the session store write fails, `runCronIsolatedAgentTurn` rejects and masks the structured `{ status: "error" }` result callers currently receive.
  - confidence: 0.8
