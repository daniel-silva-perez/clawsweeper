## Summary

Keep open. The requested isolated-cron session status persistence is still absent from current main and the latest release, and this PR remains a useful source branch, but its current diff is not merge-ready because it can persist terminal status before the actual delivery outcome and can strand setup failures as running.

## What This Changes

The PR changes isolated cron session state/run code to set `running` and persist `done`, `failed`, or `timeout`, plus adds focused session-status regression tests.

## Best Possible Solution

Implement a rebased status persistence helper that records `running` only once setup is protected, derives the terminal status from the final run plus delivery outcome, and logs best-effort store-write failures without changing the structured cron run result.

## Reproduction Assessment

Yes. The linked bug has concrete session-store reproduction steps, and the current-main code inspection gives a high-confidence focused check: isolated cron pre-run and terminal paths persist session metadata but never assign terminal `sessionEntry.status`.

## Solution Assessment

No. The PR targets the right owner boundary, but the current diff writes status too early and does not cover all terminal/setup paths; the safer solution is to persist after final outcome classification with regression tests for delivery failure, timeout, thrown error, setup failure, and persistence failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Persist status after delivery outcome is known:** `src/cron/isolated-agent/run.ts:570-571`
  - body: The new `done`/`failed` write happens before fallback/direct delivery runs. When `dispatchCronDelivery` returns an error result, this function returns `status: "error"` but the session store has already been persisted as `"done"` for a non-fatal payload, so status-based monitoring will misclassify failed cron runs. Move the terminal status write to the final return paths or update it after delivery failure is known.
  - confidence: 0.88
- **[P2] Cover setup failures after running status is persisted:** `src/cron/isolated-agent/run-session-state.ts:67`
  - body: Adding `status = "running"` to the pre-run helper makes that value persist before later setup work such as auth-profile resolution. If one of those setup steps throws, `prepareCronRunContext` rejects before the run-level `try/catch`, leaving the stored cron session stuck as `running`; either move the running persist inside a protected execution window or persist `failed` for these setup errors.
  - confidence: 0.82
