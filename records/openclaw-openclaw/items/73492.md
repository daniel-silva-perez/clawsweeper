The PR adds optional sessionId and sessionKey fields to ProviderCreateStreamFnContext and forwards them from embedded attempt, compaction, and BTW provider stream registration paths.

## Best Possible Solution

Land the additive optional fields after correcting the sessionId wording, adding focused passthrough coverage for attempt/compaction/BTW, and confirming CI; then resolve the linked createStreamFn context issue.

Yes. Current-main source inspection shows sessionId/sessionKey are available in the conversation-aware runner paths but absent from the createStreamFn plugin context, and #73488 documents a concrete multi-conversation glueclaw transcript-bleed reproduction.

## Solution Assessment

Yes, with a small docs/test adjustment. Adding optional sessionId/sessionKey to the existing provider createStreamFn context is the narrowest generic core seam for custom provider transports; the related #54725 prepareExtraParams work is adjacent, not a replacement.
