## Summary

Keep open. Current main still has a plausible cron-ok/no-artifacts path: the managed dreaming sweep can catch workspace-level failures after or before artifact writes, log them, and still return a handled processed result. The related heartbeat decoupling and detached narrative fixes are useful context, but they do not prove the reported missing artifact/event invariant is fixed.

## What This Changes

The issue reports that the memory-core managed dreaming cron on 2026.4.24 reports successful runs with eligible REM/deep harness data but stops writing DREAMS.md, per-phase dreaming reports, and memory.dream.completed events.

## Best Possible Solution

Make the memory-core managed dreaming sweep enforce an observable success invariant: with eligible workspaces and enabled phases, expected report files and memory.dream.completed events should be written, and an all-workspace failure before artifact writes should surface as a failed or clearly diagnosable cron result rather than an ok-looking processed run.

Yes for the reported 2026.4.24 install path: the issue provides cron history, rem-harness counts, memory status, and artifact/event inspections. Current-main live reproduction is not yet high-confidence because this read-only review did not run the mutating cron path.

## Solution Assessment

No proposed patch is attached, so the exact solution is not applicable yet. The narrowest maintainable fix is a memory-core regression test around cron-triggered separate-storage artifact writes plus failure propagation when every workspace fails before any expected artifact/event is written.
