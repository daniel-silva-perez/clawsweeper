## Summary

Keep open. Current main still has a plausible memory-core path where the managed cron reports a handled processed run even if every workspace fails before artifact writes, while the docs and writers continue to define visible reports/events as expected outputs.

## What This Changes

The issue reports that on OpenClaw 2026.4.24 the memory-core managed dreaming cron reports successful runs with eligible REM/deep material, but no new `DREAMS.md`, `memory/dreaming/*` reports, or `memory.dream.completed` events are written after April 18.

## Best Possible Solution

Make the memory-core managed dreaming sweep enforce an observable success invariant: when dreaming is enabled and a workspace has eligible material, the run should write the expected phase report and completion-event artifacts or return a clear failure/diagnostic instead of a generic processed success.

## Reproduction Assessment

Yes. The issue includes a concrete install/version, enabled managed cron history, non-empty `rem-harness` and memory status output, and artifact/event inspections; a focused regression test can seed eligible memory and assert reports/events after the cron-triggered sweep.

## Solution Assessment

No current patch exists, and current main is not sufficient. The narrowest maintainable fix belongs in `extensions/memory-core` around cron-triggered artifact writes and all-workspace failure propagation, without changing core plugin APIs unless a generic seam is proven necessary.
