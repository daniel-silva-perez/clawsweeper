## Summary

Keep open. Current main still has only a short config-file debounce and does not retain the watcher event path for reload attribution; the broader scoped plugin/channel reload parts overlap adjacent open feature work rather than being implemented or fully superseded.

## What This Changes

The issue asks the gateway watcher to debounce rapid watched-file edits into one reload, avoid unnecessary full plugin/channel restarts, and log which file triggered each reload.

## Best Possible Solution

Define the reload scope first, then split the end state: add source-aware coalescing and trigger logging to the gateway config reloader, while keeping plugin runtime hot reload and targeted channel reload granularity in their dedicated feature tracks.

## Reproduction Assessment

Yes for the narrow config-reload portion: source inspection shows watcher events more than the current 300 ms debounce apart can schedule separate reload runs, and the event path is discarded before logging. The broader claim about non-config watched files causing full reloads is only partially established because current docs say plugin runtime code hot reload is unsupported and the gateway config reloader watches the active config file path.

## Solution Assessment

No, current main is not the best final solution for the operator problem. The maintainable path is a narrow source-aware config reload coalescing/logging change plus explicit maintainer decisions for plugin runtime hot reload and channel reload granularity.
