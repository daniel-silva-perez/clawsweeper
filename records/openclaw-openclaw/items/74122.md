## Summary

Keep open. Current main still has only a short config-file debounce, the watcher does not retain the changed file path for reload attribution, and broader plugin/channel reload scoping remains unresolved rather than superseded by the related open items.

## What This Changes

The issue requests debounced, source-aware watcher reloads for rapid multi-file edits plus narrower reload behavior for non-config watched files.

## Best Possible Solution

Keep the issue open and split the end state into maintainable reload work: add source-aware config reload coalescing and trigger logging, then align any plugin runtime or account/channel reload granularity with the existing adjacent issues instead of burying those product decisions in a debounce-only patch.

## Reproduction Assessment

Yes for the config-reload portion: source inspection shows events more than the current 300 ms debounce apart can schedule separate reload runs, and the event path is discarded before logging or plan evaluation. The broader non-config scoped reload portion is only partially reproducible from current main because the gateway reloader watches one active config path and plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No, current main is not the best final solution for the reported operator problem. The safer path is a narrow source-aware config reload coalescing/logging change plus explicit maintainer decisions for plugin runtime hot reload and channel/account reload granularity.
