## Summary

Keep open. Current main has a configurable config-reload debounce, but it is a 300ms config-file debounce and does not fully satisfy the reported multi-file edit storm, longer coalescing window, file-source log attribution, or non-config/plugin scoped reload requests.

## Best Possible Solution

Keep this open and have maintainers scope the intended watcher boundary first. A good fix would likely preserve the existing config reload planner, add explicit trigger-source aggregation/logging, clarify or tune debounce behavior for operator edit batches, and split plugin runtime hot reload into the canonical plugin hot-reload work if it requires a separate API.

## Risks / Open Questions

- Changing the default debounce from 300ms to a multi-second window could make ordinary hot reload feel delayed; maintainers should choose the latency/reload-storm tradeoff deliberately.
- The report spans multiple watcher concepts: config hot reload, dev watch-mode process restarts, skills watcher refresh, `$include` files, and plugin runtime reload. A one-file patch could solve only part of the user-visible problem.
- Plugin-dist scoped reload overlaps broader plugin hot-reload/API ownership work and should not be hidden inside config watcher debounce without a clear boundary.
