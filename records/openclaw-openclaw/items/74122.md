## Summary

Keep open. Current main has a configurable config-file debounce and scoped config reload planner, but it still defaults to a 300ms quiet window, watches only the active config path in this reloader, drops the changed file path from watcher callbacks, and does not cover the requested multi-file/operator edit batching or plugin/non-config scoped reload behavior.

## What This Changes

The issue asks OpenClaw to coalesce rapid watcher-triggered reloads, log the triggering file or reason, and avoid full plugin/channel reloads when only a narrower subsystem changed.

## Best Possible Solution

Keep this open while maintainers split the scope: preserve the existing config reload planner, add trigger-source aggregation/logging and a well-defined operator debounce contract for config watcher events, and route plugin dist/source hot reload work through the canonical plugin hot-reload design if it needs a separate runtime API.

## Reproduction Assessment

Yes, at code level: emit config watcher events spaced beyond the current 300ms `gateway.reload.debounceMs` and each schedules a separate reload, while the callback does not retain the changed path. A full live reproduction would use the reporter's gateway log scenario, but was not run in this read-only review.

## Solution Assessment

Unclear. A narrow config-reloader change can address debounce tuning and trigger attribution, but the requested scoped reload for plugin/non-config files should be designed against the plugin hot-reload work instead of being folded into config reload opportunistically.
