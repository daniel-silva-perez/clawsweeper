## Summary

Keep this issue open. Current main still gates browser device identity on `crypto.subtle` and still fingerprints browser device keys through WebCrypto, so the HTTP LAN / non-secure-context identity-generation request remains unresolved. The secondary `dangerouslyDisableDeviceAuth` scope-clearing and implied operator-scope pieces are implemented, but they do not solve the central browser identity problem.

## Best Possible Solution

Keep this issue open. Maintainers should make an explicit product/security call on non-secure LAN browser device identity. If supported, update `ui/src/ui/gateway.ts` to attempt identity without the `crypto.subtle` gate, replace `ui/src/ui/device-identity.ts` fingerprinting with an approved browser-safe hash path, add/update Control UI tests and docs, and retain the already-shipped bypass and scope-normalization safeguards. If maintainers want a single broader tracker, coordinate with #32473 rather than closing this concrete report as already fixed.
