## Summary

Keep open. Current main still gates browser device identity on `crypto.subtle` and still fingerprints browser device public keys through WebCrypto, so the central HTTP LAN/non-secure-context request remains unresolved. The secondary `dangerouslyDisableDeviceAuth` scope-clearing and implied operator-scope pieces are implemented, but the paired implementation PR is closed unmerged and the related broader tracker remains open.

## What This Changes

The issue requests that browser device identity work without Secure Context/WebCrypto, while preserving Control UI device-auth bypass behavior and implied operator-scope normalization.

## Best Possible Solution

Maintainers should decide whether OpenClaw supports browser device identity over non-secure LAN HTTP. If yes, the fix should remove the `crypto.subtle` gate, use an approved browser-safe SHA-256 path with explicit package metadata, update the Control UI tests and public docs, and keep the already-shipped bypass and scope-normalization safeguards. If no, the docs and related tracking should clearly explain the supported HTTPS/Tailscale/trusted-proxy/break-glass paths.
