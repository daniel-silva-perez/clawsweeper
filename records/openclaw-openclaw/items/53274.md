## Summary

Keep open. Current main still depends on `crypto.subtle` for browser device identity and explicitly skips identity generation in insecure browser contexts, while the secondary bypass and operator-scope normalization pieces are already implemented; the remaining Secure Context change needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

Request to allow Control UI browser device identity generation over non-secure LAN or HTTP reverse-proxy contexts by removing the Secure Context/WebCrypto dependency, while preserving device-auth bypass and implied operator-scope handling.

## Best Possible Solution

Maintainers should choose the supported product/security posture for non-secure LAN HTTP. If supported, implement a browser-safe SHA-256 path with explicit dependency metadata, remove or narrow the Secure Context identity-generation guard, update Control UI tests and public docs, and keep the already-shipped bypass and scope-normalization safeguards. If not supported, document the HTTPS/Tailscale/trusted-proxy/break-glass paths as the intended options and align #32473 around that decision.
