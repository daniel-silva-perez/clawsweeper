## Summary

Keep open. Current main still keeps browser Control UI device identity behind the Secure Context/WebCrypto gate and still documents remote plain-HTTP Control UI as rejected without device identity; only the bypass-scope and implied operator-scope portions of the report are already implemented. The remaining change is real but security-sensitive, so it needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to generate browser Control UI device identity without depending on Secure Context/WebCrypto, preserve scopes when the explicit device-auth bypass is enabled, and normalize implied operator scopes.

## Best Possible Solution

Maintainers should make an explicit call on non-secure LAN HTTP. If supported, add a browser-safe hashing implementation with explicit UI dependency metadata, narrow or remove the Secure Context identity-generation guard, update focused UI/gateway tests, and revise public Control UI docs. If not supported, keep the shipped bypass and scope-normalization fixes and route remaining UX/messaging work through #32473, #56830, and #71669.

## Reproduction Assessment

Yes. Current-main source and tests give a high-confidence reproduction path: simulate an insecure browser context so the UI skips loadOrCreateDeviceIdentity(), then connect as a remote Control UI client without device identity and observe the reject-control-ui-insecure-auth policy path.

## Solution Assessment

Unclear. The pure-JS hash and always-attempted device identity direction is plausible for the client-side coupling, but it also changes the documented remote plain-HTTP security posture, so the narrow maintainable solution depends on maintainer threat-model approval.
