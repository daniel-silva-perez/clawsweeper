## Summary

Keep open. Current main still does not implement the Feishu liveness/status plumbing proposed by this PR: Feishu sets only the gateway port, and no status callback is threaded through monitor startup, inbound handlers, or transport lifecycle. The overlapping #39489 and #41940 are still open implementation candidates, not merged or clearly selected as canonical replacements.

## Best Possible Solution

Keep this PR open for Feishu plugin follow-up. The best path is to pick one canonical implementation among this PR, #39489, and #41940; thread the gateway `setStatus` callback through `extensions/feishu/src/channel.ts`, `monitor.ts`, `monitor.account.ts`, and `monitor.transport.ts`; report `connected` only after WebSocket start or webhook listen succeeds; report disconnect/error state on abort, startup failure, runtime failure, and early-abort paths; update inbound event liveness for meaningful Feishu events; decide whether Feishu can safely publish transport activity separately from app events; and add focused lifecycle plus inbound regression tests.
