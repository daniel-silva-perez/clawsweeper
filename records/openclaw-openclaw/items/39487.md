## Summary

Keep open. Current main still does not implement this PR's Feishu liveness/status plumbing: the Feishu start path only records the webhook port, no status callback is threaded through monitor startup, inbound event handlers, or transport lifecycle, and the related implementation candidates #39489 and #41940 are still open rather than merged or clearly selected as canonical replacements.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to choose one canonical implementation among this PR, #39489, and #41940; thread a gateway status callback through the Feishu plugin start path, monitor options, single-account monitor, inbound handlers, and transport lifecycle; report connected status only after WebSocket start or webhook listen succeeds; report disconnect/error state on abort, startup failure, runtime failure, and early-abort paths; decide whether Feishu should publish transport activity separately from app events; and add focused lifecycle plus inbound regression tests.
