## Summary

Keep this open. Current main still sends bundled Grok web_search through the shared 30 second timeout on the xAI Responses path, and the related timeout-fix PR remains open rather than merged or shipped.

## What This Changes

The issue requests a longer or provider-owned default timeout for bundled Grok xAI Responses web_search so slower valid searches are not aborted at the shared 30 second cutoff.

## Best Possible Solution

Land or refresh a narrow xAI-owned fix that keeps the shared web_search default unchanged for other providers, gives Grok a longer fallback only when no user timeout is configured, and adds focused regression coverage for the default timeout path. If maintainers prefer an operator setting instead, align the xAI plugin schema and docs with that provider-specific configuration.

## Reproduction Assessment

Yes. The reporter provided a direct `requestXaiWebSearch(...)` reproduction with a working xAI key and timings at the 30 second boundary, and current source confirms that an unconfigured Grok xAI Responses request still receives the shared 30 second timeout.

## Solution Assessment

Yes. A provider-owned Grok fallback or explicit Grok webSearch timeout setting is the narrow maintainable solution; raising the shared default for every web_search provider would broaden the behavioral change unnecessarily.
