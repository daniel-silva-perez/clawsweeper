## Summary

Keep this open: current main still routes bundled Grok web_search through the shared 30 second timeout, and the linked timeout fix remains an open, unmerged PR rather than shipped behavior.

## What This Changes

The issue asks for the bundled Grok xAI Responses web_search path to use a longer provider-specific default timeout so slow but successful searches are not aborted at the shared 30 second boundary.

## Best Possible Solution

Land #43609 or an equivalent provider-owned xAI change that keeps user-configured timeoutSeconds honored, raises only the Grok default, converts internal timeout AbortErrors into structured web_search errors, and adds focused xAI web_search timeout/error tests.

## Reproduction Assessment

Yes. The issue gives a direct requestXaiWebSearch reproduction with tools.web.search.provider set to grok, and current main shows that path still falls back to a 30 second guarded-fetch AbortSignal; live latency validation requires a working xAI key and a slow query.

## Solution Assessment

Yes for the scoped timeout bug: a Grok-specific default is narrower than changing the shared web_search default for every provider. The open PR also bundles adjacent AbortError/null-shape hardening, so it should receive normal maintainer review before merge.
