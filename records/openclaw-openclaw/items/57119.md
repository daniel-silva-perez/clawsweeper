## Summary

Keep this issue open. Current main still treats `skills.entries.<key>.enabled: false` as skill eligibility, status, prompt visibility, and sandbox-sync filtering. I did not find a skill-directory access-control boundary wired into `read`, `write`, `edit`, `apply_patch`, or `exec`; the existing docs also describe tool policy and sandboxing as the hard enforcement controls, not skill disablement.

## Best Possible Solution

Keep open for maintainer product/security design. If `enabled: false` is intended as access control, implement a centralized disabled-skill realpath deny policy and enforce it before filesystem tools and exec path/workdir/script access, with regression coverage across host, sandbox `none`/`ro`/`rw`, managed/bundled/plugin/workspace skills, and link escapes. If maintainers want the toggle to mean visibility only, rename or document that contract clearly and direct operators to sandboxing, tool deny rules, workspace-only filesystem policy, OS permissions, or container isolation for enforcement.
