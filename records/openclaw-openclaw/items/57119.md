## Summary

Keep this issue open. Current main still implements disabled skills as eligibility/visibility filtering, not as a skill-directory access-control boundary for `read` or `exec`. There are broader controls for sandboxing, tool allow/deny, and workspace-only filesystem access, but no skill-scoped deny rule tied to `skills.entries.<key>.enabled: false`.

## Best Possible Solution

Keep open and decide the intended contract. If `enabled: false` is meant to be access control, implement a centralized skill-directory deny policy derived from resolved realpaths of disabled skills and apply it before `read`, `write`, `edit`, `apply_patch`, and `exec` path/workdir/script access in both host and sandbox modes, with regression tests for workspace skills, managed/bundled/plugin skills, rw sandbox workspaces, and symlink/hardlink escapes. If maintainers prefer the toggle to mean visibility only, rename/clarify the UI/docs and point operators to sandbox, tool deny, workspace-only FS policy, OS permissions, or container isolation for enforcement.
