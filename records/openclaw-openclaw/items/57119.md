## Summary

Keep this issue open. Current main still treats `skills.entries.<key>.enabled: false` as skill eligibility/status/prompt filtering and config snapshot invalidation, not as a filesystem or process access-control boundary. I found no disabled-skill realpath deny list wired into `read`, `write`, `edit`, `apply_patch`, or `exec`; the existing docs point operators to sandboxing, workspace-only filesystem policy, and tool allow/deny lists for hard enforcement.

## Best Possible Solution

Keep open for maintainer product/security design. If `enabled: false` is meant to be access control, add a centralized disabled-skill realpath deny policy derived from skill discovery/config and enforce it before filesystem tools and exec path/workdir/script access, with regression coverage across host, sandbox `none`/`ro`/`rw`, managed/bundled/plugin/workspace skills, and path alias escapes. If maintainers want the toggle to mean visibility only, rename or document that contract clearly and direct admins to tool deny rules, `tools.fs.workspaceOnly`, sandbox workspace access, OS permissions, or container isolation.

## Risks / Open Questions

- Security/product semantics remain unresolved: users may reasonably read a Disabled skill toggle as enforcement while current main primarily implements visibility and eligibility filtering.
- A partial fix would be easy to bypass unless it covers realpath normalization, symlink/hardlink aliases, host and sandbox modes, read/write/edit/apply_patch, exec command and workdir/script resolution, managed/bundled/plugin/workspace skills, and elevated/node execution paths.
- Related #17931 covers read-only sandbox skill mounts, but it does not fully supersede this issue because this issue asks for `enabled: false` enforcement across both filesystem and process tools.
