## Summary

Keep open. The PR targets a real open QMD scheduling complaint, but the proposed guard is not safe against current main: it treats the resolved default `embedIntervalMs` as explicit user intent and conflicts with the shipped lexical/BM25-only contract in tests and docs.

## What This Changes

The PR changes QMD embed execution and timer scheduling so a positive resolved `embedIntervalMs` bypasses the lexical search-mode vector guard, and adds a regression test for `searchMode: "search"` with `embedInterval: "5m"`.

## Best Possible Solution

Decide the QMD lexical-mode contract first. If explicit `embedInterval` should opt lexical setups into embedding, preserve an explicit-config flag during backend config resolution, use that flag in both embed guards, and update the conflicting tests plus memory docs/schema help. If lexical mode must remain BM25-only, keep the current contract and close the paired bug/PR with that behavior stated clearly.
