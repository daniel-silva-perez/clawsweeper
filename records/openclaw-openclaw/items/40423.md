## Summary

Keep this PR open. Current main at 16906780fd2c481308ec4b80f96eb3b93d0add46 still only injects the existing OPENCLAW_SHELL markers for exec, acp-client, and tui-local child process paths; it does not implement or document OPENCLAW_AGENT_ID or OPENCLAW_SESSION_KEY. The narrower related #72980 remains open and does not supersede this broader runtime proposal.

## Best Possible Solution

Keep this PR open for rebase and maintainer review. The best implementation is a shared or consistently mirrored runtime env resolver that injects stable OPENCLAW_AGENT_ID plus an approved optional OPENCLAW_SESSION_KEY, threads explicit agent/session context through exec direct/sandbox/PTY and gateway approval follow-up execution, updates the current ACP helper in src/acp/client-helpers.ts, passes current TUI agent/session state into createLocalShellRunner, and adds focused tests for explicit-agent precedence, session-key fallback, missing/legacy context, env override semantics, and session-key exposure policy.

## Risks / Open Questions

- Exposing OPENCLAW_SESSION_KEY through process environment is security-sensitive because env values can leak through process inspection, shell tracing, logs, crash reports, or subprocess inheritance; this needs explicit maintainer policy before landing.
- The submitted ACP patch is stale against current main: current spawn-env logic is in src/acp/client-helpers.ts, while the provided PR diff modifies the older helper location in src/acp/client.ts.
- The gateway approval follow-up path and explicit-agent precedence need focused tests so non-main agents are not mislabeled when session keys are missing, global, or legacy-shaped.
- Env override semantics need a clear contract: current OPENCLAW_SHELL is forced after env merging, while related discussion for session markers mentions caller-supplied env winning on conflict.
