## Summary

Keep open. Current main still lacks the requested doctor diagnostic: state integrity loads session entries for transcript/orphan checks, but it does not inspect cached skill snapshots for stale bundled-skill paths, and both CLI and gateway reuse persisted snapshots based on version/filter state rather than active runtime-root validation.

## What This Changes

Add an `openclaw doctor` diagnostic that detects persisted session skill snapshots containing stale absolute bundled-skill paths from inactive, missing, or temp-backed runtime roots and provides safe remediation guidance.

## Best Possible Solution

Add a focused doctor state diagnostic that resolves the active bundled skills directory, scans persisted `skillsSnapshot.prompt` and `skillsSnapshot.resolvedSkills[].filePath` values for inactive, missing, or temp-backed bundled runtime roots, reports cached-session metadata drift separately from live runtime health, and backs up the session store before invalidating affected snapshot fields in repair mode.

## Reproduction Assessment

Yes. A session store entry with `skillsSnapshot.prompt` or `skillsSnapshot.resolvedSkills[].filePath` pointing at an old bundled runtime root is enough to verify the gap against current main: doctor loads the store but never checks those fields, and runtime reuse refreshes only on missing/version/filter changes.

## Solution Assessment

Yes. A doctor/session-store diagnostic is the narrowest maintainable fix for the operator-facing drift; startup/load-time migrations would be broader and would mutate long-lived session state outside an explicit repair flow.
