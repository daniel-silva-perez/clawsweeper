## Summary

Keep open. Current main still lacks the requested `openclaw doctor` diagnostic: doctor state integrity reads session stores for transcript/orphan checks, while the CLI and gateway paths continue to reuse persisted `skillsSnapshot` values based on snapshot version and skill-filter state rather than validating cached bundled-skill paths against the active runtime root.

## What This Changes

Add an `openclaw doctor` check that detects persisted session skill snapshots containing stale absolute bundled-skill paths from inactive, missing, or temp-backed runtime roots and gives safe remediation guidance.

## Best Possible Solution

Add a focused doctor state diagnostic that resolves the active bundled skills directory, scans persisted `skillsSnapshot.prompt` and `skillsSnapshot.resolvedSkills[].filePath` values for inactive, missing, or temp-backed bundled runtime roots, reports cached-session metadata drift separately from live runtime health, and backs up the session store before invalidating only affected snapshot fields in repair mode.

## Reproduction Assessment

Yes. A session store entry with `skillsSnapshot.prompt` or `skillsSnapshot.resolvedSkills[].filePath` pointing at an old bundled runtime root can be inspected against current main: doctor loads the store but never checks those fields, and runtime reuse only refreshes on missing/version/filter changes.

## Solution Assessment

Yes. A doctor/session-store diagnostic is the narrowest maintainable fix for the reported operator-facing drift; startup/load-time migrations would be broader and risk mutating long-lived session state outside an explicit repair flow.
