## Summary

Keep open. Current main has conservative stale-lane recovery, but active embedded runs, active reply work, and active lane tasks still remain warning-only by default, and an open closing PR already targets the missing abort threshold/recovery behavior.

## What This Changes

The issue requests configurable diagnostics recovery that aborts sessions stuck in `processing` so the gateway can recover without an external restart.

## Best Possible Solution

Review and land or revise #73243, or an equivalent narrow patch, so diagnostics keeps the existing warning threshold, adds a separate conservative abort threshold, aborts only after that threshold using established run cleanup primitives, emits structured recovery diagnostics, and updates config/docs/tests.

## Reproduction Assessment

Yes. The issue has production logs, RayWoo's comment adds a deterministic v2026.4.25 Telegram restart-window reproduction, and current-main source inspection confirms active stuck work still remains warning-only.

## Solution Assessment

Yes in direction, but current main is not sufficient. A separate `stuckSessionAbortMs` path reviewed through #73243 is safer than overloading `stuckSessionWarnMs` or aborting immediately at the warning threshold.
