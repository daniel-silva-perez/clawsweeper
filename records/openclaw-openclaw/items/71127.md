## Summary

Keep open. Current main still has conservative stale-lane recovery, but it does not implement the requested configurable active-run abort path, so the reported stuck processing-session recovery gap remains real.

## What This Changes

The issue requests diagnostics-driven recovery that aborts sessions stuck in `processing`, releases their queue, and emits structured recovery diagnostics without requiring an external gateway restart.

## Best Possible Solution

Ship one diagnostics-owned recovery path that keeps the existing warning threshold, adds a separate conservative abort threshold, uses existing embedded/reply-run cleanup primitives, emits typed `session.aborted` diagnostics and OTEL coverage, and documents the config and long-running-work boundaries.

## Reproduction Assessment

Yes. The issue has production logs plus a deterministic v2026.4.25 Telegram restart-window reproduction, and current main has a code-level reproduction by marking a diagnostic session `processing` past `diagnostics.stuckSessionWarnMs` and observing that the heartbeat does not authorize active abort.

## Solution Assessment

Yes in direction, but not solved by current main. A separate abort threshold wired into existing cleanup primitives is the narrow maintainable fix; the default and cron/long-running-job interaction should be settled in the linked PR review.
