## Summary

Keep open. Current main has partial stale-lane recovery, but it still intentionally leaves active embedded runs, active reply work, and active lane tasks warning-only, and the issue is already paired with an open closing PR that should be reviewed before closure.

## What This Changes

The issue asks for stuck `processing` sessions to be aborted and released after a configurable diagnostics threshold instead of only warning and requiring an external gateway restart.

## Best Possible Solution

Review and either land or revise the linked stuck-session abort PR so diagnostics has a separate conservative abort threshold, uses existing reply/embedded-run abort and cleanup primitives, emits a structured recovery event/log, and preserves warning-only behavior for healthy long-running work until that threshold is crossed.

## Reproduction Assessment

Yes. The issue has production logs plus a deterministic v2026.4.25 Telegram restart-window reproduction in the discussion, and current main source confirms active stuck runs are still not aborted by diagnostics.

## Solution Assessment

No for current main as-is. The current conservative stale-lane release is useful but does not satisfy the requested self-healing abort contract; the linked PR is the right review path, subject to maintainer validation of thresholds and active-run cleanup semantics.
