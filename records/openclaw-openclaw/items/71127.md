## Summary

Keep open. Current main only has conservative stale-lane recovery; active embedded runs, reply work, and active lane tasks still remain warning-only by default, and the open closing PR has not landed yet.

## What This Changes

The issue requests a configurable diagnostics-driven abort/recovery path for sessions stuck in `processing` so queued messages can resume without restarting the gateway.

## Best Possible Solution

Review and land or revise #73243, or an equivalent narrow patch, so diagnostics keeps the warning threshold, adds a separate conservative abort threshold, aborts only after that threshold using established run cleanup primitives, emits structured recovery diagnostics, and updates config/docs/tests.

## Reproduction Assessment

Yes. The issue has production logs, RayWoo added a deterministic v2026.4.25 Telegram restart-window reproduction, and current-main source inspection confirms active stuck work is still warning-only by default.

## Solution Assessment

Yes in direction, but current main is not sufficient. A separate `stuckSessionAbortMs` path is safer than overloading `stuckSessionWarnMs` or aborting immediately at the warning threshold.
