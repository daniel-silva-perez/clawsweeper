## Summary

Keep open. Current main still does not implement the requested active stuck-session abort path: diagnostics warn and invoke conservative recovery, but active embedded runs, reply work, and active lane tasks remain observe-only/keep-lane by default. The issue is also paired with open closing PR #73243, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to abort and release sessions that remain in `processing` beyond a diagnostic threshold so the gateway can recover without an external restart.

## Best Possible Solution

Review and land or revise #73243, or an equivalent narrow patch, so diagnostics keeps the warning threshold, adds a separate conservative abort threshold, aborts active stuck work only after that threshold, emits structured recovery diagnostics, and updates config/schema/docs/tests.

## Reproduction Assessment

Yes. The issue has production logs, RayWoo added a deterministic v2026.4.25 Telegram restart-window reproduction, and current-main source/tests confirm active stuck work is still not aborted by default.

## Solution Assessment

Yes in direction, but current main is not sufficient. A separate `diagnostics.stuckSessionAbortMs` path is safer than overloading `stuckSessionWarnMs` or aborting immediately at the warning threshold.
