## Summary

Keep open. The linked data-loss bug is real on current main, but this PR's guard avoids deleting the nested workspace by skipping the entire state directory, which means a State + config uninstall can leave selected config and auth state behind.

## What This Changes

The PR adds a workspace-preservation option to state cleanup, passes workspace directories from uninstall when the workspace scope is not selected, and adds a cleanup-utils dry-run regression test.

## Best Possible Solution

Revise the uninstall cleanup so it preserves unselected workspace directories while still removing selected state/config data that can be safely removed, or refuses before confirmation with clear messaging when selective cleanup cannot be done safely. Add CLI-level coverage for the default nested workspace layout plus helper coverage for config/OAuth paths inside state.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: the default workspace is under `~/.openclaw`, state-only uninstall recursively removes the state directory, and workspace deletion is guarded by a separate scope.

## Solution Assessment

No. The PR prevents the workspace data loss, but skipping the whole state directory is not the narrowest maintainable fix because it leaves selected state/config/auth cleanup incomplete; selective pruning or an explicit pre-confirmation refusal is safer.

## Review Findings

- **[P2] Keep removing state data around preserved workspaces:** `src/commands/cleanup-utils.ts:116-122`
