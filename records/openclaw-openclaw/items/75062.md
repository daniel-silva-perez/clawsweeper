## Summary

Keep this PR open. It addresses a real data-loss bug in uninstall, but the proposed fix skips the entire state directory when an unselected workspace is nested inside it, which can leave selected state, config, and credentials data behind after the user chose State + config removal.

## What This Changes

The PR adds a workspace preservation option to state cleanup, passes unselected workspace directories from uninstall, and adds a dry-run regression test for nested workspaces.

## Best Possible Solution

Revise the uninstall cleanup so selecting State + config preserves any unselected configured workspace while still removing state data that is safe to remove, or refuse before confirmation with clear messaging when selective cleanup cannot be performed safely.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: default workspaces resolve under ~/.openclaw, uninstall treats state and workspace as separate scopes, and current state cleanup recursively removes the whole state directory.

## Solution Assessment

No. Skipping the whole state directory prevents workspace deletion, but it is not the narrowest maintainable fix because it leaves selected state/config/auth cleanup incomplete; selective pruning or an explicit pre-confirmation refusal is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep removing state data around preserved workspaces:** `src/commands/cleanup-utils.ts:116-122`
  - body: When a workspace is inside the state dir, this branch skips the whole state-dir removal. In the default layout, config and credentials also live inside that dir and the helper skips in-state linked paths afterward, so selecting State + config can leave the selected state/auth data behind. Preserve the workspace by pruning around it or aborting before confirmation, not by treating state removal as complete.
