## Summary

Keep this PR open. Current main still only detects Gemini schema cleaning from the provider id, so a Gemini model routed through a custom OpenAI-compatible proxy can still miss `cleanSchemaForGemini()`. The PR addresses a real gap, but it should be rebased onto the current `pi-tools-parameter-schema.ts` split and updated to preserve `modelCompat.unsupportedToolSchemaKeywords` stripping on the new modelId-based Gemini path.

## Best Possible Solution

Keep the PR open, rebase it onto current main, implement Gemini detection from normalized `modelId` in `src/agents/pi-tools-parameter-schema.ts`, and make `applyProviderCleaning()` compose `cleanSchemaForGemini()` with `stripUnsupportedSchemaKeywords()` when both Gemini detection and `modelCompat.unsupportedToolSchemaKeywords` apply. Add regression tests for custom provider + Gemini modelId, `patternProperties`/`const` stripping, non-Gemini custom providers, and the combined Gemini-plus-modelCompat path before landing.
