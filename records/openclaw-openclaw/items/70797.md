## Summary

Keep open. Current main still lacks a user-visible guard for bare OpenAI-style `tool_calls` JSON: the shared sanitizer only strips bracketed/XML tool scaffolding, the Telegram/direct partial-preview path can pass the cleaned stream text directly to `onPartialReply`, and the related fix PRs for this exact leak were closed unmerged.

## What This Changes

The issue reports that bare OpenAI-style `tool_calls` JSON can appear in user-visible Telegram/direct streaming replies during background exec/process polling flows.

## Best Possible Solution

Add a shared assistant-visible guard for standalone OpenAI-style tool-call JSON, use it in both final delivery/history sanitization and partial-preview streaming, and lock the behavior with regression tests for complete payloads, incomplete streaming fragments, ordinary JSON, and code examples.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: pass visible prose followed by `{"tool_calls":[...]}` through the current sanitizer/preview path and the bare JSON is not stripped before `onPartialReply` or final visible text handling.

## Solution Assessment

Yes. The narrow maintainable direction is to sanitize or withhold internal tool-call payload text at assistant-visible boundaries, not to add a tinyrouter-specific workaround or try to execute downgraded text as a recovered tool call.
