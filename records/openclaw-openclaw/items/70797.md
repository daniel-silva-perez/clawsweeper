## Summary

Keep open. Current main still appears to allow the reported bare OpenAI-style `tool_calls` JSON leak class through both final assistant-visible text cleanup and Telegram/direct partial-preview delivery; the two linked fix PRs are closed unmerged and no active canonical PR closes this issue.

## What This Changes

The issue requests preventing raw internal `tool_calls` JSON payloads from being rendered in user-visible Telegram direct replies during background exec/process polling flows.

## Best Possible Solution

Add a shared, streaming-aware assistant-visible guard for bare OpenAI-style tool-call JSON, apply it before both final visible delivery and partial preview callbacks, and preserve ordinary JSON plus inline/fenced examples through focused regression tests.

## Reproduction Assessment

Yes. The live channel timing is not fully minimized, but the reported payload shape can be traced through current main: the sanitizer early-exits on bare JSON and the partial-preview path can pass the resulting text to `onPartialReply`.

## Solution Assessment

Yes. The narrow maintainable fix is to sanitize or withhold internal tool-call payload text at assistant-visible boundaries, rather than adding provider-specific tinyrouter workarounds or trying to execute downgraded text as a tool call.
