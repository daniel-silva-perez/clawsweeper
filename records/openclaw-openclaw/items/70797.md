## Summary

Keep open: current main still appears to allow the reported bare OpenAI-style `tool_calls` JSON to pass through assistant-visible text and Telegram/direct partial-preview delivery. The source has diagnostic detection for tool-shaped text, but that path only logs and still treats the payload as text, and the two related fix PRs were closed unmerged.

## What This Changes

The issue requests preventing raw internal `tool_calls` JSON from becoming user-visible during Telegram direct background exec/process polling flows.

## Best Possible Solution

Implement a shared, streaming-aware guard for bare OpenAI-style `tool_calls` JSON at the assistant-visible boundary, reuse it for final text and partial preview delivery, and keep ordinary JSON plus fenced/inline code examples intact.

## Reproduction Assessment

Yes for the leak class: the reported bare JSON shape can be traced through current source because the sanitizer returns it unchanged and the partial-preview path can pass the resulting text to `onPartialReply`. The exact live Telegram/tinyrouter timing remains only sketched, not fully minimized.

## Solution Assessment

Yes: the best fix is a narrow core sanitizer/preview-path change, not a provider-specific workaround or an attempt to execute downgraded text as a tool call. The repair should only prevent internal payload text from reaching user-visible delivery surfaces.
