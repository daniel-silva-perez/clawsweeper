## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting guidance and has no Feishu channel proxy policy; the SDK HTTP token path delegates through the Lark default HTTP instance while the WebSocket path only has ambient proxy-agent wiring. The issue is also paired with open docs PR #75050, so it should stay open until maintainers decide whether docs-only guidance is enough or a runtime policy is needed.

## What This Changes

Feishu/Lark should support or document a channel-specific proxy policy so provider requests can keep using ambient proxy env while Feishu token and WebSocket traffic can bypass it when needed.

## Best Possible Solution

Keep this open while maintainers review #75050. If Feishu/Lark NO_PROXY guidance is accepted as the intended resolution, land that PR and close afterward; otherwise add a Feishu-owned proxy policy that applies consistently to SDK HTTP token/API calls and WebSocket startup, with tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, affected endpoint, and observed error, and current source plus the locked SDK show tenant_access_token requests still flow through the SDK HTTP instance while WebSocket proxy handling is separate.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is a low-risk mitigation, but a channel-level proxyMode affects network egress policy and should be chosen deliberately before automation implements runtime bypass behavior.
