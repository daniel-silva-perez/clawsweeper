## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy documentation and a first-class Feishu proxy policy, and the same author has an open docs PR that should be reviewed rather than closing this issue ahead of merge.

## What This Changes

Request Feishu/Lark channel proxy control or documentation so provider traffic can keep env proxy settings while Feishu HTTP token and WebSocket traffic can bypass that proxy when needed.

## Best Possible Solution

Review and, if acceptable, land #75050 as the immediate Feishu/Lark NO_PROXY troubleshooting path. If maintainers want a fuller fix, add a Feishu-owned proxy policy that applies consistently to Lark SDK HTTP calls and WebSocket startup, with focused config-schema and client tests, rather than changing global provider proxy behavior.

## Reproduction Assessment

Unclear. The issue gives a concrete Windows mixed-proxy scenario and endpoint error, but a high-confidence live reproduction needs Feishu/Lark credentials plus a proxy environment; current source review confirms the requested control surface is missing.

## Solution Assessment

Unclear for the full issue. The linked docs PR is a narrow low-risk partial solution, but a channel-level proxyMode remains a product/security decision because it would explicitly change Feishu/Lark egress routing.
