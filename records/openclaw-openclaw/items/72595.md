## Summary

Keep open. Current main still lacks both Feishu/Lark mixed-proxy troubleshooting guidance and a channel-level proxy policy, and the issue is paired with open docs PR #75050 rather than resolved on main.

## What This Changes

Allow Feishu/Lark channel traffic to use a direct or explicit per-channel proxy policy while model-provider traffic can continue using environment proxy settings.

## Best Possible Solution

Keep the issue open while maintainers review #75050. If docs-only NO_PROXY guidance is the intended resolution, land that PR and close this afterward; otherwise add a Feishu-owned proxy policy covering both SDK HTTP and WebSocket paths, with tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, proxy env vars, Feishu WebSocket mode, and the failing tenant_access_token endpoint; current source still leaves SDK HTTP and WebSocket routing on separate proxy paths. No live Feishu credentials were used in this read-only review.

## Solution Assessment

Unclear. The docs-only PR is a reasonable near-term mitigation, but a channel-level proxyMode needs maintainer judgment because direct-by-default could bypass enterprise proxy or egress controls.
