## Summary

Keep open. Current main still lacks either Feishu/Lark mixed-proxy troubleshooting guidance or a channel-level proxy policy, and the issue is now paired with open docs PR #75050 rather than resolved on main.

## What This Changes

Request a Feishu/Lark channel-specific proxy policy or documentation so provider traffic can keep using ambient proxy env while Feishu HTTP and WebSocket traffic can bypass that proxy when needed.

## Best Possible Solution

Keep the issue open while maintainers review #75050. If docs-only NO_PROXY guidance is accepted as the intended resolution, land that PR and close this issue afterward; otherwise add a Feishu-owned proxy policy at the SDK HTTP and WebSocket seams with tests and docs.

## Reproduction Assessment

Yes. The issue provides concrete proxy env, Feishu WebSocket mode, and failing tenant_access_token endpoint details, and current source plus the locked SDK show Feishu HTTP calls still flow through the SDK axios instance while WS proxy handling is separate.

## Solution Assessment

Unclear. The docs-only PR is a reasonable near-term mitigation, but a channel-level proxyMode needs maintainer judgment because direct-by-default is not clearly safer than ambient proxy or NO_PROXY guidance for enterprise networks.
