## Summary

Keep open. Current main still lacks a Feishu/Lark channel proxy policy or merged troubleshooting docs for this mixed-proxy setup: the Feishu SDK HTTP path wraps the Lark SDK default HTTP instance for timeout/User-Agent behavior only, while the WebSocket path separately injects an ambient proxy agent. The same author has an open docs-only follow-up in #75050, so the issue should remain open while maintainers decide whether docs are enough or runtime proxy policy is needed.

## What This Changes

Request a Feishu/Lark channel proxy bypass or documented NO_PROXY policy so model-provider traffic can keep using process proxy env while Feishu token and WebSocket traffic can go direct when needed.

## Best Possible Solution

Decide the supported product contract first: either land #75050 as the documented NO_PROXY workaround, or add a Feishu-owned proxy policy that applies consistently to SDK HTTP token/API calls and WebSocket startup. If runtime behavior changes, keep it inside the Feishu plugin, document the account/channel setting, and add focused tests for ambient proxy, NO_PROXY bypass, and direct mode without weakening the managed network-proxy boundary.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup and current source confirms the relevant paths: Feishu SDK HTTP token/API calls have no channel-level proxy policy while WebSocket creation has separate ambient proxy handling. I did not run a live Feishu credential test in this read-only review.

## Solution Assessment

Unclear. The docs-only path in #75050 is a useful low-risk workaround, but it may not satisfy the requested per-channel proxy policy; a runtime fix needs maintainer judgment because it changes network egress behavior.
