## Summary

Keep open. Current main does not yet provide a Feishu/Lark per-channel proxy policy: the WebSocket path intentionally consumes ambient proxy env, the HTTP client wrapper only injects timeouts into the Lark SDK default HTTP instance, and the Feishu schema/docs have no `proxyMode` or equivalent setting. The same author has an open docs PR, so this should stay open for maintainer judgment on whether docs-only `NO_PROXY` guidance is enough or whether a runtime channel option is needed.

## What This Changes

Request a Feishu/Lark-specific proxy policy or documentation so Feishu token/API/WebSocket traffic can bypass ambient proxies while model-provider traffic continues using env proxy settings.

## Best Possible Solution

Have maintainers review the open docs PR first; if docs-only guidance is not enough, add a Feishu-owned, per-account proxy policy such as `ambient`/`direct` that applies consistently to token/API HTTP calls and WebSocket setup, with focused config/client tests and no core-wide proxy migration.

## Reproduction Assessment

Yes for source-level reproduction: current main can be inspected to show ambient proxy use on the Feishu WebSocket path and no Feishu-specific HTTP/WebSocket bypass setting. The reporter also provided concrete Windows mixed-proxy symptoms, but live Feishu credentials/network reproduction was not performed here.

## Solution Assessment

Unclear. The docs-only PR is the narrowest workaround and may be enough for many users, but the complete maintainable fix would be a Feishu-owned proxy policy if maintainers want reliable per-channel behavior without relying on global env tuning.
