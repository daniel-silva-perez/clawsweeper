## Summary

Keep this PR open, but do not merge the current patch. The requested full-reply-in-thread behavior is not fully implemented on current main, but the PR head introduces a visible-channel delivery path that can bypass `ANNOUNCE_SKIP`, has inconsistent cross-channel provenance, and is currently reported as unmergeable against main.

## What This Changes

This PR changes the A2A `sessions_send` flow to send the final target reply directly to the announce target, run the announce step in the requester session, suppress fallback announce delivery after a full-reply send, and include `threadId` on fallback sends.

## Best Possible Solution

Define an explicit A2A delivery policy on current main: visible threads should receive the intended final post-ping-pong content only when external delivery is allowed, while preserving `accountId`/`threadId`, target-channel provenance, per-session lanes, transcript hygiene, and focused tests/docs for announce delivery and `ANNOUNCE_SKIP`.

## Reproduction Assessment

Yes. A focused unit-level reproduction is to run the PR head with an announce target, a non-empty `latestReply`, and an announce step that returns `ANNOUNCE_SKIP`; the PR sends at line 106 before the skip check at line 157, while current main only sends after the skip guard.

## Solution Assessment

No. The current PR is not the best fix because it treats full-reply delivery as an unconditional pre-announce side effect; a safer solution should ask or compute the delivery policy first, then perform exactly one external send with the correct target provenance.

  - body: The new full-reply send publishes `latestReply` to the announce target before the announce step can return `ANNOUNCE_SKIP`.   - body: This announce step marks the source session as the target session but keeps `sourceChannel` as the requester channel. When requester and target channels differ, downstream provenance labels forwarded target content as coming from the wrong channel, which can affect formatting and routing decisions.
  - confidence: 0.8

  - body: When the full-reply send succeeds, the function still runs the announce model call even though the later send branch is guarded by `!fullReplyPosted`. That adds latency/cost and records a synthetic announce turn in the requester session without any deliverable result.
  - confidence: 0.73
