## Summary

Keep open. Current main still delivers the target's A2A result through an announce step rather than posting the full target reply, so the PR is not obsolete; however the proposed patch changes external message delivery before preserving the existing announce suppression contract and has a provenance mismatch that should block merge.

## What This Changes

This PR changes the A2A `sessions_send` flow to post the post-ping-pong `latestReply` directly to the announce target, run the announce step in the requester session, suppress duplicate announce sends, and include `threadId` on the fallback send params.

## Best Possible Solution

Define and implement an explicit A2A delivery policy on current main: visible threads should receive the intended final post-ping-pong content only when suppression permits external delivery, while preserving `accountId`/`threadId`, provenance metadata, per-session lanes, and docs/tests for announce delivery and `ANNOUNCE_SKIP`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor ANNOUNCE_SKIP before full-reply delivery:** `src/agents/tools/sessions-send-tool.a2a.ts:106-117`
  - body: The new full-reply send publishes `latestReply` to the announce target before the announce step can return `ANNOUNCE_SKIP`. That changes the existing suppression contract from no external post to posting the full target reply anyway, which can leak content into a linked channel or thread.
  - confidence: 0.9
- **[P2] Use target-channel provenance for requester announce:** `src/agents/tools/sessions-send-tool.a2a.ts:148-149`
  - body: This announce step marks the source session as the target session but keeps `sourceChannel` as the requester channel. When requester and target channels differ, downstream provenance labels the forwarded content as coming from the wrong channel, which can affect formatting and routing decisions.
  - confidence: 0.78
