## Summary

Keep this PR open. Current main still does not implement the requested full target-reply delivery behavior, but the submitted patch is not merge-ready because it can publish target replies before announce suppression, uses inconsistent source-channel provenance in the new requester announce step, and is currently dirty against main.

## What This Changes

The PR changes the A2A `sessions_send` flow to send the target session's final reply directly to the announce target, run the announce step in the requester session, suppress fallback announce delivery after a full-reply send, and include `threadId` on fallback sends.

## Best Possible Solution

Define an explicit A2A delivery policy on current main: visible announce targets should receive exactly one allowed post with the intended final post-ping-pong content, while preserving `ANNOUNCE_SKIP`, target-channel provenance, `accountId`/`threadId`, per-session lanes, transcript hygiene, and focused tests/docs for the full-reply case.

## Reproduction Assessment

Yes. A focused unit-level reproduction is to run the PR head with an announce target, a non-empty `latestReply`, and an announce step that returns `ANNOUNCE_SKIP`; the PR sends at line 106 before the skip check at line 157, while current main only sends after the skip guard and has coverage asserting no gateway `send` call.

## Solution Assessment

No. The PR is not the best way to solve the issue because it treats full-reply delivery as an unconditional external side effect; a safer fix should decide the delivery policy first, then perform exactly one external send with the correct target provenance after suppression is honored.

  - body: This announce step marks the source session as the target session but keeps `sourceChannel` as the requester channel. When requester and target channels differ, downstream provenance labels forwarded target content as coming from the wrong channel, which can affect formatting and routing decisions.
  - confidence: 0.8

  - body: When the full-reply send succeeds, the function still runs the announce model call even though the later send branch is guarded by `!fullReplyPosted`. That adds latency/cost and records a synthetic announce turn in the requester session without any deliverable result.
  - confidence: 0.73
