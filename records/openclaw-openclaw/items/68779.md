## Summary

Keep open. Current main has partial mitigations for existing-session automation, but the reported stability problem is not fully resolved: the code still uses short attach/readiness probes, several Chrome MCP actions remain unbounded or reject timeout overrides, and there is no regression test proving a multi-step `profile="user"` flow survives session loss and refreshes cleanly.

## What This Changes

The issue requests hardening browser existing-session automation by widening timeout budgets, adding Chrome MCP session recovery, improving diagnostics, and allowing configurable timeout behavior for multi-step flows.

## Best Possible Solution

Keep this issue open and implement a scoped browser-plugin hardening plan: add bounded Chrome MCP attach/readiness budgets that can be configured or derived safely, recover or refresh sessions for safe existing-session management calls after transport loss, and add a focused regression test for a multi-step `user` profile workflow that refreshes the selected page/session instead of dropping the flow.
