## Summary

Keep open. Current main still has the browser existing-session stability gaps called out by the report and by the maintainer follow-up: Chrome MCP attach/reachability budgets are still short, transport errors still drop the cached session for the failing call rather than recovering in-place, and the tests/docs only cover partial mitigations rather than a proven multi-step user-profile recovery path.

## What This Changes

The issue requests more resilient browser existing-session automation by relaxing tight Chrome MCP timeouts, adding reconnection or session refresh, improving diagnostics, and supporting safer configuration/operation handling for multi-step flows.

## Best Possible Solution

Implement a scoped browser-plugin hardening path: add bounded and evidence-backed attach/readiness budgets, recover or refresh safe existing-session management calls after Chrome MCP transport loss, keep mutating-action retries conservative, and add a focused regression test proving a multi-step user-profile flow can refresh the selected page/session instead of dropping the workflow.
