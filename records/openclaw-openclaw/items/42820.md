## Summary

Keep open. Current main still exposes poll creation fields on the shared message tool schema and still throws before Feishu or Telegram send handling when non-empty poll-looking params accompany action="send". The only current mitigation covers zero-valued defaults, and PR #46455 remains an open implementation candidate that explicitly links this issue for closure.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #46455. The durable fix should either strip accidental poll creation params before the send guard or narrow the LLM-facing schema so poll fields are only visible for poll-capable actions/channels, with regression coverage for Feishu and Telegram send-with-attachment calls carrying non-empty auto-populated poll fields.
