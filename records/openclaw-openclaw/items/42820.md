## Summary

Keep open. Current main still exposes poll creation fields on the shared LLM-facing message tool and still rejects action="send" when non-empty poll-looking params are present. The shipped zero-default mitigation is narrower than the reported failure, and PR #46455 remains open as an implementation candidate that explicitly links this issue for closure.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #46455. The fix should either sanitize accidental poll creation params before the send guard or narrow the LLM-facing schema so poll fields are only visible for poll-capable actions/channels, with regression coverage for Feishu and Telegram send-with-attachment calls that include non-empty auto-populated poll fields.

## Risks / Open Questions

- Closing now would leave the current-main false positive for send or send-with-attachment calls whenever a model supplies non-empty or true poll-looking fields from the shared schema.
- PR #46455 also changes empty component stripping alongside poll-param handling, so maintainers should review that combined behavior rather than treating the issue as already shipped.
- The durable fix needs to preserve explicit poll validation for poll-capable channels such as Telegram and Discord while avoiding poll leakage into normal send/file workflows.
