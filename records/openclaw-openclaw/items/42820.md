## Summary

Keep open. Current main still exposes poll creation fields on the shared message tool and still rejects `action="send"` when non-empty poll-looking params are present. The shipped mitigation only covers zero-valued poll defaults, while open PR #46455 explicitly targets the remaining false-positive behavior and uses closing syntax for this issue.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #46455. The fix should either sanitize accidental poll creation params before the send guard or narrow the LLM-facing schema so poll fields are only visible for poll-capable actions/channels, with regression coverage for Feishu and Telegram send-with-attachment calls that include non-empty auto-populated poll fields.
