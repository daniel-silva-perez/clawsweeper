## Summary

Keep open. Current main still exposes poll parameters on the shared message tool and still throws before Feishu/Telegram send handling when non-empty poll-looking params accompany action="send". The only shipped/current mitigation I found covers zero-valued defaults, while open PR #46455 remains an unmerged implementation candidate that links this issue for closure.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #46455. The durable fix should either sanitize accidental poll creation params before the send guard or narrow the LLM-facing schema so poll fields are only visible for poll-capable actions/channels, with regression coverage for Feishu and Telegram send-with-attachment calls carrying non-empty auto-populated poll fields.
