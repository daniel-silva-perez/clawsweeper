## Summary

Keep open. Current main still exposes shared poll creation fields on the LLM-facing `message` tool and still rejects `action="send"` when those fields contain non-empty, true, or non-zero values before Feishu/Telegram send handling can run. The shipped zero-default mitigation narrows one false positive, but it does not solve the reporter’s auto-filled non-empty poll-field case. There is also an open implementation candidate, PR #46455, that uses closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #46455. The durable fix should either sanitize accidental poll creation params before the send guard or narrow the LLM-facing schema so poll creation fields are only visible for poll-capable actions/channels, while preserving explicit `action="poll"` validation and adding regression coverage for Feishu and Telegram send-with-attachment calls with non-empty auto-populated poll fields.
