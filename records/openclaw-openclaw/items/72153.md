## Summary

Keep open. Current main and the latest release still generate the unsupported Windows `Start-Process -LiteralPath` command, while the PR is a narrow, correct-looking fix for the linked Windows failure.

## What This Changes

This PR changes the Windows `resolveConfigOpenCommand` command and its regression test to use `Start-Process -FilePath` instead of `-LiteralPath` when opening the config file.

## Best Possible Solution

Land this PR or the same minimal patch, preserve the existing `execFile` launcher and PowerShell single-quote escaping, update the focused unit assertion, and add any required changelog attribution during maintainer handling.

## Reproduction Assessment

Yes. Current main deterministically builds the failing Windows command, and local PowerShell verification rejects `-LiteralPath` while accepting `-FilePath`; the linked Windows log matches that failure mode.

## Solution Assessment

Yes. Replacing only the unsupported PowerShell parameter is the narrowest maintainable fix for the observed failure, with no need to change path escaping or non-Windows behavior.
