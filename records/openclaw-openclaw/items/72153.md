## Summary

Keep this PR open. Current main and the latest shipped release still generate the unsupported Windows `Start-Process -LiteralPath` command, while the PR makes the narrow parameter-only fix and updates the focused regression expectation.

## What This Changes

The PR changes the Windows branch of `resolveConfigOpenCommand()` and its unit test to use `Start-Process -FilePath` instead of `Start-Process -LiteralPath`.

## Best Possible Solution

Land this PR or the same minimal fix: use `Start-Process -FilePath` in the Windows config opener, preserve the existing `execFile` call and single-quote escaping, update the focused unit test, and add any required changelog attribution separately.

## Reproduction Assessment

Yes. Current main deterministically generates the failing Windows command, and local PowerShell verification confirms `Start-Process` rejects `-LiteralPath` and accepts `-FilePath`; the linked Windows bug log matches that failure mode.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix for the observed failure because it only replaces the unsupported PowerShell parameter and updates the direct regression assertion while leaving escaping and non-Windows launch behavior untouched.
