## Summary

Keep this PR open. Current main still builds the Windows config opener as `Start-Process -LiteralPath ...`, the regression test still asserts that old command, and a local PowerShell contract check confirms `Start-Process` supports `FilePath` but not `LiteralPath`. The PR diff is narrow, aligned with linked bug report #69933, and does not introduce a new security-sensitive surface.

## Best Possible Solution

Keep this PR open for maintainer review and merge, or land the same minimal fix on main: change the Windows branch of `resolveConfigOpenCommand()` to use `Start-Process -FilePath ...`, preserve the existing single-quote escaping, and update `src/gateway/server-methods/config.test.ts` to assert `-FilePath`.
