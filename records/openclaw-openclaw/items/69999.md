## Summary

Keep this PR open. Current main still lacks a macOS wallboard shell, so the branch is not obsolete, but the proposed implementation has a packaged-app resource lookup blocker and a smaller re-open reload issue that should be fixed before merge.

## What This Changes

The branch adds a macOS `--wallboard` launch path, display resolver, singleton wallboard manager, borderless `WKWebView` window controller, bundled placeholder resource, package resource rule, and smoke tests.

## Best Possible Solution

Keep the wallboard shell work in the macOS app if maintainers want REL-433, but make the resource resolver use the existing main-bundle-first packaged-app pattern, avoid reloading the placeholder when re-presenting an existing controller, and have a macOS maintainer review the WebKit/private-KVC choices before validating with Swift tests and a packaged-app launch.

## Reproduction Assessment

Not applicable for the feature request itself; source inspection verifies current main lacks the proposed `--wallboard` surface and the PR adds it. The packaged-app resource bug has a high-confidence source-level path from the PR diff and existing `DeviceModelCatalog` contract, but I did not live-run a macOS packaged app in this read-only review.

## Solution Assessment

No, not as-is. The feature split is narrow and plausible, but the direct `Bundle.module` lookup should be replaced with the safer existing bundle-resolution pattern before this is the best maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Resolve the placeholder from the packaged app bundle first:** `apps/macos/Sources/OpenClaw/Wallboard/WallboardWindowController.swift:58-61`
  - body: `Bundle.module` can trap in packaged `.app` builds before this guard runs when SwiftPM cannot find its generated resource bundle. Current macOS resource code explicitly avoids that by checking `Bundle.main` first and only then falling back to `Bundle.module`; the wallboard placeholder should use the same resolver or `--wallboard` can crash in the packaged app.
  - confidence: 0.9
- **[P2] Do not reload the placeholder when re-showing the window:** `apps/macos/Sources/OpenClaw/Wallboard/WallboardManager.swift:13-15`
  - body: When a controller already exists, `open()` calls `present()` and then starts a fresh `WKWebView` navigation. Re-showing the cached wallboard should preserve the already loaded placeholder; otherwise users can see a blank/black flash on every reopen.
  - confidence: 0.82
