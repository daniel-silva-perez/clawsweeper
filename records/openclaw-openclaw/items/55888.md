## Summary

Keep this issue open. Current main supports the reported `agents.defaults.compaction` tuning knobs, but it still defaults to a 20,000-token compaction reserve plus a 4,000-token memory-flush buffer, so a 32,768-token local Ollama model still hits the reported ~8.7k preflight threshold by default. The remaining work is a product/docs enhancement for an opt-in high-memory local-model profile or clearer tuning guidance, not an already-implemented fix.

## Best Possible Solution

Keep this issue open as a concrete enhancement. The best maintainer path is to add a documented high-memory local Ollama compaction profile, or an explicit opt-in preset that lowers `reserveTokensFloor`, sets `keepRecentTokens`, and explains the 4,000-token `memoryFlush.softThresholdTokens` buffer. If product wants automation, implement hardware-aware compaction policy behind a conservative opt-in and add tests for the threshold calculation and config precedence.
