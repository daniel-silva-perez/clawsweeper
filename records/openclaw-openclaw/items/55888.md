## Summary

Keep this issue open. Current main still supports the reporter's `agents.defaults.compaction` workaround, but the default behavior remains the reported conservative threshold: for a 32,768-token model, the effective preflight/memory-flush threshold is still 32,768 - 20,000 - 4,000 = 8,768 tokens. The remaining work is a product/docs enhancement for an opt-in high-memory local Ollama profile or clearer tuning guidance, not a cleanup-close candidate.

## Best Possible Solution

Keep this issue open as a focused enhancement. The best path is to add a documented high-memory local Ollama compaction profile or explicit opt-in preset under `agents.defaults.compaction` that lowers `reserveTokensFloor`, sets `keepRecentTokens`, and explains the 4,000-token `memoryFlush.softThresholdTokens` buffer. If maintainers want automation, implement hardware-aware compaction policy conservatively behind opt-in config and cover the threshold math plus config precedence with tests.
