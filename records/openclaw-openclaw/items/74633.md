## Summary

Current main has targeted gateway availability fixes for the reported failure class: local discovery advertisement is bounded, model-pricing startup is kicked off asynchronously, stale model-catalog refreshes no longer sit directly on the request path, and the merged follow-up prevents empty catalogs from poisoning the gateway cache. I could prove the fix on current main, but not in a shipped release newer than the latest published v2026.4.27.

## What This Changes

The issue asks OpenClaw to stop healthy Gateway `/v1/*` requests from hanging behind Bonjour, Telegram polling, and model-pricing/model-catalog timeout cycles.

## Best Possible Solution

Keep the current main implementation and carry it into the next release; users on 2026.4.26 or the latest published v2026.4.27 should upgrade once a build containing 6689e414 is available, and reopen with fresh gateway logs or an event-loop profile only if the same hang persists there.

## Reproduction Assessment

Yes. The report gives concrete curl commands and gateway log signatures for the 2026.4.26 failure, and current tests cover the fixed discovery and model-catalog blocking paths; this review did not perform a live macOS Intel reproduction.

## Solution Assessment

Yes. The current approach is narrow and maintainable: bound the stuck discovery startup path, keep pricing startup asynchronous, serve usable model data while refreshes run, and avoid empty-catalog cache poisoning rather than adding a broad event-loop workaround.
