## Summary

Keep this PR open. Current main still implements the unsafe cron/session behavior the PR is trying to fix: `current` can persist a caller direct-chat session key, explicit `session:<direct-key>` targets are reused at runtime, and update normalization still lacks caller session context. The PR is a plausible core security hardening change, but the existing review comments about direct-key classification and named-session regressions still need maintainer resolution before landing.

## Best Possible Solution

Keep this PR open and either refresh it or replace it with a canonical core fix: add a shared cron session-target guard used by create/update normalization and runtime fallback, pass caller session context through gateway and tool update paths, and add tests for canonical direct keys, legacy `dm`/`direct` formats, account-qualified direct keys, safe group/channel reuse, ordinary named sessions containing direct-like text, and failure-notification routing.
