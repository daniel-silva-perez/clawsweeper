## Summary

Keep this PR open. Current main still has the cron direct-session isolation gap, but this PR is not merge-ready because the proposed guard imports a non-exported helper and its direct-key classifier can both miss real direct sessions and downgrade custom named sessions.

## What This Changes

The PR adds a cron session-target guard, threads caller session context through cron add/update paths, adds a runtime fallback for legacy direct session targets, and covers the behavior in cron, agent-tool, and gateway tests.

## Best Possible Solution

Land a refreshed core fix that compiles against current session utilities, uses a canonical chat-session parser instead of broad token heuristics, handles legacy `direct`/`dm` formats, preserves ordinary custom named sessions, and documents or tests the intended failure-notification routing behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create or update an agent-turn cron job with `sessionTarget: "current"` from a direct session, or run a persisted `session:agent:...:direct:...` target, and current main still resolves/reuses that direct session key.

## Solution Assessment

No. The PR's layered fix direction is right, but the current patch is not the best mergeable form because it has a compile-time import error and an imprecise direct-session classifier; a safer version should use the current session chat-type module and stricter session-shape checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the chat-type helper from its actual module:** `src/cron/session-target-guard.ts:1`
  - body: `session-key-utils.ts` on current main exports `parseAgentSessionKey` but not `deriveSessionChatType`; that helper lives in the session chat-type module. As written, this new file will fail TypeScript compilation before the cron guard can run.
  - confidence: 0.96
- **[P1] Classify direct markers before reusable group tokens:** `src/cron/session-target-guard.ts:17`
  - body: The guard relies on `deriveSessionChatType(...) === "direct"`, but the current helper returns `group` or `channel` first whenever those tokens appear anywhere in the key. Account-qualified direct keys with an account segment like `group` or `channel` can still be treated as reusable, bypassing the intended direct-message isolation.
  - confidence: 0.9
- **[P2] Guard legacy direct session formats:** `src/cron/session-target-guard.ts:14-15`
  - body: `isUnsafeCronSessionKey` returns false unless `parseAgentSessionKey` succeeds, so legacy unscoped direct keys such as `telegram:dm:123` are still accepted as safe even though the session chat-type helpers recognize `dm`/`direct` as direct conversations.
  - confidence: 0.88
- **[P2] Avoid downgrading custom agent-shaped session names:** `src/cron/session-target-guard.ts:17`
  - body: Because any `agent:<id>:...` value is parsed and then classified by tokens, a custom named target like `session:agent:billing:direct:daily` is downgraded even though `session:<custom-id>` is documented as a persistent named session. The guard needs to distinguish real channel chat keys from legal custom IDs.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff does not add supply-chain or workflow risk, but the proposed security hardening is incomplete for several direct-session key shapes and can regress custom session isolation semantics.

Concerns:

- **[medium] Direct-session guard can under-block real DMs:** `src/cron/session-target-guard.ts:17`
  - body: The guard depends on the current broad chat-type heuristic and an agent-scoped parse gate, so legacy `dm` keys and account-qualified direct keys can remain reusable direct sessions. That leaves part of the session-isolation issue the PR is meant to fix.
  - confidence: 0.9
- **[low] Custom named sessions can be downgraded unexpectedly:** `src/cron/session-target-guard.ts:17`
  - body: Documented custom session IDs that happen to look like `agent:<id>:direct:<name>` are treated as unsafe direct chats, which can silently move scheduled work to isolated cron sessions instead of the operator's persistent session.
  - confidence: 0.84
