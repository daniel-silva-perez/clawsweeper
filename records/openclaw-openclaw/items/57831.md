## Summary

Keep this PR open (https://github.com/openclaw/openclaw/pull/57831). Current main still allows cron `current` and explicit `session:<id>` targets to bind to direct-message session keys, and the PR targets a real cron/session isolation surface across `cron.add`, `cron.update`, the agent cron tool, and gateway runtime execution. It is not obsolete or implemented on main, but it still needs maintainer review for the existing direct-key classification and named-session regression concerns before landing.

## Best Possible Solution

Keep this PR open and either refresh it or replace it with a canonical core fix. The best path is a small, shared cron session-target guard used by create/update normalization and runtime fallback, with tests for canonical direct keys, legacy `dm`/`direct` keys, account-qualified direct keys, safe group/channel reuse, ordinary named sessions containing `direct`, and failure-notification routing semantics.

## Risks / Open Questions

- The PR is security-relevant because it changes which transcript/session a scheduled agent run can read and write; it should get maintainer/security review before landing.
- Existing review comments identify plausible bypass/regression cases in direct-key classification and custom named sessions, so landing the PR as-is could under-fix the isolation gap or break legitimate persistent named sessions.
- Current main has tests that use direct session targets for failure notification routing; a final fix should explicitly preserve or redesign that behavior rather than accidentally removing operator notifications.
