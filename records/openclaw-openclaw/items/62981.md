## Summary

Current main still leaves the reported Gateway-timeout-to-embedded-fallback session lock contention unresolved. The CLI can still time out while waiting for the final Gateway `agent` response, then immediately invoke the embedded runner with the same session selectors; the daemon-side accepted run can continue on the same session and hold the transcript lock.

## Best Possible Solution

Keep this issue open as the focused tracker for Gateway-timeout-to-embedded-fallback session lock contention. The fix should be in `src/commands/agent-via-gateway.ts`: after a Gateway `agent` request has been accepted, a client-side timeout should not immediately start embedded execution against the same session. Use `agent.wait`, abort-and-wait, or an explicitly fresh user-visible fallback session, and add regression coverage in `src/commands/agent-via-gateway.test.ts` for an accepted Gateway run timing out before fallback. Coordinate the broader external-integration timeout work with #71605, but do not close this focused issue yet.
