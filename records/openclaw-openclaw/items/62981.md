## Summary

Current main still matches the reported failure path. The Gateway-side agent request can be accepted and continue running on the resolved session, while the CLI-side fallback path still invokes the embedded runner with the same original session options after a Gateway timeout. That leaves this as a valid focused bug rather than an implemented or stale report.

## What This Changes

The issue asks for Gateway-timeout embedded fallback to avoid immediately reusing a session file still locked by the Gateway run.

## Best Possible Solution

Keep this issue open and fix the CLI/Gateway handoff so an accepted Gateway run that later times out cannot race an embedded fallback on the same transcript. The safest direction is to surface the accepted `runId` to the CLI and use `agent.wait` or abort-and-wait before any embedded fallback; if a fresh session is used, make that behavior explicit and visible. Add regression coverage for the accepted-then-timeout path and coordinate the broader dispatch-timeout behavior with #71605.

## Repair Work Prompt

Fix the Gateway-timeout embedded-fallback session lock bug tracked by this issue. Current main sends `openclaw agent` through the Gateway with `expectFinal: true`; the Gateway client suppresses the accepted ack, the outer call can time out, and `agentCliCommand` then calls embedded `agentCommand` with the original session options, racing the still-running Gateway attempt on the same transcript lock. Implement a narrow fix so accepted Gateway runs do not immediately fall back into the same session. Prefer surfacing the accepted `runId` to the CLI and using `agent.wait` or abort-and-wait before embedded fallback; only use a fresh session if it is explicit in metadata/output and does not silently fork user history. Add regression coverage in `src/commands/agent-via-gateway.test.ts` for an accepted Gateway run timing out before final response and prove the embedded fallback does not reuse the locked session. Coordinate behavior with #71605, but do not attempt broad Gateway dispatch-timeout architecture or generic session-lock rewrites. Changelog entry expected under the active Fixes section with reporter credit. Do not change unrelated provider fallback, MCP cleanup, or session lock stale-recovery policy.
