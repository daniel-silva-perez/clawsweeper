## Summary

Keep this issue open. Current main supports manual per-session auth profile pinning with `/model ...@<profileId>`, but the `/models` provider/model picker surfaces do not expose account/profile choices or persist an auth profile when a user taps a model button.

## What This Changes

Request `/models` UI/buttons to let users choose and see the active provider account while selecting GPT/Claude-backed models.

## Best Possible Solution

Keep the issue open and extend the existing model-picker path to reuse the current `authProfileOverride` pipeline: add safe, non-secret profile choices to shared picker data, surface them in supported channel UIs, and apply selections as `provider/model@profileId` or equivalent session state with docs and tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main gap: manual `/model ...@<profileId>` works, while `/models` picker data and channel callbacks do not carry account/profile choices.

## Solution Assessment

Unclear until maintainers settle the privacy and channel-scope policy. The narrow technical direction should reuse existing auth-profile resolution and session override fields rather than adding GPT/Claude-specific account state.
