## What This Changes

The issue asks for a decision and fix for PR-branch emotion-tag parsing so line-start `[label1][label2]` is not misclassified between adjacent emotion tags and markdown reference links.

## Best Possible Solution

Keep this as a PR-stack review tracker. If emotion mode proceeds, define the ambiguous line-start rule explicitly and update #72229 with tests for numeric references, named references with definitions, named references without definitions, and adjacent emotion pairs.

## Reproduction Assessment

Not on current main: the affected file and `/emotions` command are absent. On #72229's head, the branch-level reproduction is high-confidence by source inspection because the markdown-reference guards require non-whitespace before the pair, so `[fast][1]` at index 0 loses `[fast]`.

## Solution Assessment

Unclear until maintainers choose the product rule. The issue correctly rejects a label-only heuristic; the safer implementation direction is document-context reference-definition detection, or an explicit rule that line-start allowlisted pairs remain emotion tags when no definition exists.
