## Summary

Keep open. Current main and the latest release do not contain the reported `emotion-tags` parser or `/emotions` command, while the ambiguity is still present in the open emotion-mode PR stack and needs a maintainer semantics decision before that stack lands.

## What This Changes

The issue requests fixing or defining how the proposed emotion-tag stripper treats a line-start `[label1][label2]` sequence that could be either adjacent emotion tags or a Markdown reference link.

## Best Possible Solution

Keep this as a PR-stack review tracker. Before merging emotion mode, define the line-start rule explicitly and update #72229 with tests for numeric references, definition-backed named references, named references without definitions, and adjacent emotion pairs.

## Reproduction Assessment

No high-confidence reproduction exists on current main because the affected parser and command are absent. There is a high-confidence branch reproduction on #72229 head by source inspection: line-start `[fast][1]` fails markdown-reference protection because `hasNonWhitespaceBeforeOnLine` returns false at index 0.

## Solution Assessment

Unclear until maintainers choose the product semantics. The issue correctly rejects a label-only heuristic; the more maintainable path is either definition-aware Markdown reference detection or an explicit rule that line-start allowlisted pairs remain emotion tags when no definition exists.
