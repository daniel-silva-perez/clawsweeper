## Summary

Keep open. Current main and the latest release still do not contain the emotion-tag parser or `/emotions` surface, while the ambiguity is present in the open emotion-mode PR stack and needs a maintainer semantics decision before that stack lands.

## What This Changes

The issue asks maintainers to resolve how the unmerged emotion-tag sanitizer should distinguish line-start markdown reference links like `[fast][1]` from adjacent emotion tags.

## Best Possible Solution

Keep this as a PR-stack review tracker. Before merging emotion mode, define the line-start rule explicitly and update #72229 with tests for numeric references, definition-backed named references, named references without definitions, and adjacent emotion pairs.

## Reproduction Assessment

Not on current main: the affected file and `/emotions` command are absent. On #72229 head, source inspection gives a high-confidence branch reproduction because the line-start `[fast][1]` pair fails markdown-reference protection and the first label is stripped.

## Solution Assessment

Unclear until maintainers choose the product rule. The issue correctly rejects a label-only heuristic; the safer implementation direction is document-context reference-definition detection or an explicit rule that line-start allowlisted pairs remain emotion tags when no definition exists.
