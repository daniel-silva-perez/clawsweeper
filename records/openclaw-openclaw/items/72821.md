## Summary

Keep open. Current main and the latest release do not contain the emotion-tag parser or `/emotions` surface, while the ambiguity still exists in the open emotion-mode PR stack and needs a maintainer semantics decision before that stack lands.

## What This Changes

The issue asks maintainers to resolve line-start `[label1][label2]` ambiguity in the proposed emotion-tag sanitizer so markdown reference links such as `[fast][1]` are not stripped as adjacent emotion tags.

## Best Possible Solution

Keep this as a PR-stack review tracker. Before merging emotion mode, maintainers should choose the line-start rule explicitly and update #72229 with implementation and tests for numeric references, definition-backed named references, named references without definitions, and adjacent emotion pairs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not on current main because the affected parser and `/emotions` surface are absent; yes by source inspection on #72229 head, where line-start `[fast][1]` fails markdown-reference protection through `hasNonWhitespaceBeforeOnLine` and the branch lacks a covering test.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the product semantics. The issue correctly avoids a pure label-content heuristic; the safer direction is document-context reference-definition detection or an explicit rule that line-start allowlisted pairs remain emotion tags when no definition exists.
