## Summary

Keep open. Current main has the DeepSeek V4 `reasoning_content` payload repair, but it is only reachable through native/provider-specific wrappers such as `deepseek`, `opencode-go`, and Venice. Generic OpenAI-compatible proxy models still lack a documented per-model opt-in that would route prefixed DeepSeek V4 IDs through that shared repair path.

## What This Changes

The issue asks for a per-model compatibility setting so DeepSeek V4 `reasoning_content` replay and thinking payload fixes can apply to DeepSeek models served through non-`deepseek` providers or differently prefixed model IDs.

## Best Possible Solution

Add a documented model-level compat opt-in for OpenAI-compatible proxy models that applies the shared DeepSeek V4 replay wrapper to matching/prefixed V4 model IDs, preferably by reusing the existing `compat.thinkingFormat: "deepseek"` contract unless maintainers choose to add a compatibility alias. Keep provider-specific exceptions like Venice intact and add regression coverage plus config docs/changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a custom OpenAI-completions provider model with id like `deepseek/deepseek-v4-pro`: current main can preserve/render `compat.thinkingFormat`, but the stream wrapper guard and generic wrapper path show that model will not reach the DeepSeek V4 payload repair.

## Solution Assessment

Partly. A per-model opt-in is the right shape, but adding a second `compat.reasoningFormat` key may not be the narrowest solution because `compat.thinkingFormat: "deepseek"` already exists in the typed config/schema/catalog path.
