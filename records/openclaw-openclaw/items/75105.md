## Summary

Keep open. Current main has the native DeepSeek V4 `reasoning_content` repair and an existing model-level `compat.thinkingFormat` field, but arbitrary OpenAI-compatible proxy models still do not reach the shared DeepSeek V4 payload wrapper.

## What This Changes

The issue requests a model-level compatibility setting that lets custom OpenAI-compatible proxy providers opt DeepSeek V4 models into the `reasoning_content` payload repair.

## Best Possible Solution

Add a documented model-level opt-in for OpenAI-compatible proxy models that routes matching DeepSeek V4 ids through the shared DeepSeek V4 payload repair, preferably by reusing `compat.thinkingFormat: "deepseek"`; keep provider-specific behavior intact and add regression coverage for prefixed model ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a custom OpenAI-compatible provider model such as `deepseek/deepseek-v4-pro`: current main can read/render `compat.thinkingFormat`, but the stream wrapper path will not invoke the DeepSeek V4 payload repair outside the native provider hook.

## Solution Assessment

Yes, with one adjustment. A per-model opt-in is the right maintainable shape, but the narrowest solution is likely to reuse the existing `compat.thinkingFormat: "deepseek"` field rather than introduce a second public `compat.reasoningFormat` key.
