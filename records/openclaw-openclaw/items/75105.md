## Summary

Keep open. Current main has DeepSeek V4 `reasoning_content` repair for the native DeepSeek plugin and some bundled proxy plugins, but arbitrary OpenAI-compatible providers still cannot opt a model into the shared DeepSeek V4 payload wrapper from model-level compat config.

## What This Changes

The issue requests a per-model compatibility setting so DeepSeek V4 `reasoning_content` handling works through custom or proxy OpenAI-compatible providers with differently prefixed model IDs.

## Best Possible Solution

Add a generic model-compat path that reuses `compat.thinkingFormat: "deepseek"` to apply the shared DeepSeek V4 OpenAI-compatible wrapper for configured proxy/custom models, and document the custom-provider configuration without adding provider allowlists such as Requesty-specific IDs.

## Reproduction Assessment

Yes. A source-level reproduction is to configure a custom `openai-completions` provider model with `compat.thinkingFormat: "deepseek"` and a prefixed DeepSeek V4 model ID, then observe that the current wrapper chain never applies the shared DeepSeek V4 payload repair because the native wrapper only accepts provider `deepseek` with exact model IDs.

## Solution Assessment

Yes to the per-model compatibility direction, but the narrowest maintainable fix is to reuse the existing `compat.thinkingFormat: "deepseek"` field rather than add an overlapping `compat.reasoningFormat` key unless maintainers explicitly want a new public config key.
