## Summary

Keep open. Current main already has model-level `compat.thinkingFormat` and shared DeepSeek V4 payload repair helpers, but arbitrary custom/OpenAI-compatible providers still do not get a generic path that installs the DeepSeek wrapper from model compat, and the native DeepSeek guard remains provider/exact-id gated.

## What This Changes

Request to let model-level configuration trigger DeepSeek V4 `reasoning_content` compatibility for third-party or custom OpenAI-compatible providers, including differently prefixed model IDs.

## Best Possible Solution

Add a generic model-compat-driven DeepSeek V4 path: reuse `compat.thinkingFormat: "deepseek"` unless maintainers explicitly want a new alias, install the shared replay wrapper for matching custom OpenAI-compatible DeepSeek V4 models including slash-prefixed IDs, and document the custom provider configuration without hard-coding Requesty or another proxy provider.

## Reproduction Assessment

Yes. Configure a custom `openai-completions` provider with a model id such as `deepseek/deepseek-v4-pro` and `compat.thinkingFormat: "deepseek"`; source inspection shows current main will use the compat value for rendering but will not install the DeepSeek V4 request wrapper in the generic stream chain.

## Solution Assessment

Yes, the generic per-model compat direction is the maintainable fix, but the repository already has `compat.thinkingFormat`, so reusing that key is safer than adding a separate overlapping `reasoningFormat` unless maintainers decide to rename the public contract.
