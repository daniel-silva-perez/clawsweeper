## Summary

Keep open. Current main still only consumes Baileys `messages.upsert` events and has a narrow 60-second `append` grace path; it does not persist per-chat last-seen cursors, request a bounded missed-message window after reconnect, or document a WhatsApp backfill setting. The discussion's per-JID cursor/backfill and gap-warning direction remains relevant.

## Best Possible Solution

Keep this open and treat the current append handling as a partial mitigation. The best path is a WhatsApp-plugin design for bounded reconnect catch-up: persist last-seen message state per account/chat, validate the Baileys/WhatsApp Web history mechanism for fetching a gap, dedupe by message ID before enqueueing, limit replay to configured monitored chats and a capped window, and log a clear gap warning when reliable provider backfill is unavailable. Add focused tests around reconnect, append/history events, cursor persistence, and dedupe; update WhatsApp docs/config only if a user-facing setting is introduced.
