## Summary

Keep open. Current main still lacks WhatsApp missed-message backfill after reconnect: inbound handling remains event-driven, stale `append` history is skipped, socket full-history sync is disabled, and reconnect handling only drains pending outbound deliveries. The closed BlueBubbles catch-up item is useful related context but is channel-specific and does not supersede this WhatsApp gap.

## What This Changes

Request WhatsApp recovery for messages missed during Web connection outages by backfilling monitored chats after reconnect, or at least warning when the gap cannot be recovered.

## Best Possible Solution

Implement bounded reconnect recovery inside the WhatsApp plugin: persist last-seen state per account and eligible chat, validate a Baileys history mechanism for monitored chats, cap lookback and replay volume, dedupe by message ID before enqueueing, and emit a clear operator warning when provider backfill cannot be trusted or is unavailable.

## Reproduction Assessment

Yes for the current code-level behavior: a Baileys `messages.upsert` `append` event with a timestamp five minutes before connection is covered by tests and is not delivered to `onMessage`. A live WhatsApp reconnect experiment is still needed to validate the provider-side history mechanism and exact outage behavior.

## Solution Assessment

Unclear. The requested cursor-plus-history direction is plausible, but the safest solution needs maintainer validation of Baileys history behavior and product decisions around privacy scope, replay limits, read receipts, dedupe, and fallback warning behavior.
