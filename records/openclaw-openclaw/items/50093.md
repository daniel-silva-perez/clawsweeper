## Summary

Keep open. Current main still relies on Baileys inbound events plus a narrow 60-second `append` grace path; it does not persist per-chat recovery cursors, call a reconnect-time WhatsApp history request, or provide an operator warning when reliable backfill is unavailable. The discussion points to the right fix direction, but the Baileys/WhatsApp Web history semantics need maintainer validation before this can be treated as a routine cleanup close.

## What This Changes

Request WhatsApp reconnect recovery that fetches and replays messages missed while WhatsApp Web was disconnected for active monitored chats.

## Best Possible Solution

Implement this inside the WhatsApp plugin as a bounded reconnect recovery path: persist last-seen state per account/chat, validate and use the Baileys/WhatsApp Web history mechanism for eligible monitored chats, cap the replay window and per-run volume, dedupe by message ID before enqueueing, and emit a clear gap warning when provider backfill cannot be trusted.
