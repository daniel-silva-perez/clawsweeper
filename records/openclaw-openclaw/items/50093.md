## Summary

Keep open. Current main has a narrow WhatsApp reconnect mitigation for Baileys `append` upserts, but it does not implement the requested monitored-chat backfill/cursor after reconnect. The code only processes append/history messages that Baileys already emits within a 60-second grace window and explicitly skips older append history, so the issue’s two-hour/gap-period recovery request remains unimplemented.

## Best Possible Solution

Implement this in the WhatsApp plugin as a bounded reconnect catch-up path: persist last-seen message cursors per account/chat, fetch or request available WhatsApp/Baileys history for active monitored chats after reconnect, dedupe by message ID before enqueueing, and add operator logging when a gap is detected but provider backfill is unavailable. Add focused tests beside the existing append-upsert coverage and update WhatsApp docs/config if a user-facing setting is introduced.

## Evidence

There is no reconnect-time per-chat history query or cursor-driven backfill in this path; the only connect-time side task here hydrates participating group metadata, not messages.

## Risks / Open Questions

- A future implementation needs upstream Baileys/WhatsApp Web API validation before enabling explicit history fetches; if the provider cannot reliably fetch gaps, the interim product path should at least log a clear missed-message gap warning.
- Because Baileys can already emit some recent `append` messages, any broader backfill should keep the existing message-id dedupe path to avoid double-processing replayed messages.
