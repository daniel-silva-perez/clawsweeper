## Summary

Keep open. Current main still lacks WhatsApp missed-message backfill after reconnect: inbound handling is event-driven from Baileys upserts, stale `append` history is intentionally skipped, full socket history sync is disabled, and reconnect success only drains queued outbound deliveries.

## What This Changes

The issue requests bounded WhatsApp backfill for missed monitored-chat messages after a WhatsApp Web outage and reconnect.

## Best Possible Solution

Implement provider-validated, bounded reconnect recovery inside the WhatsApp plugin: persist last-seen state per account and eligible chat, request only monitored-chat history for a capped window, dedupe by message ID before enqueueing, and emit a clear warning when reliable backfill is unavailable.

## Reproduction Assessment

Yes for current code-level behavior: an `append` upsert with a timestamp five minutes before connect is covered by tests and is expected not to call `onMessage`. A live WhatsApp outage/reconnect experiment is still needed to validate the provider-side history path and exact missed-message behavior.

## Solution Assessment

Unclear. The requested cursor-plus-history direction is plausible, but the safest solution needs maintainer validation of Baileys history semantics plus product decisions around privacy, replay limits, dedupe, read receipts, and fallback warnings.
