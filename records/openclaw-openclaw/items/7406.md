## Summary

Keep open. Current main has the Telegram topic-name source and prompt metadata, but the topic name still is not carried into stored session display metadata, Gateway session rows, or the Control UI selector; recent discussion also treats this as the canonical tracker while the directly related implementation PRs are closed unmerged and the remaining open UI PR is only adjacent cleanup.

## What This Changes

The issue requests human-readable Telegram forum topic names in the session dropdown instead of raw topic-qualified session keys or numeric IDs.

## Best Possible Solution

Add an additive generic thread/topic display-label path in session metadata, have the Telegram plugin populate it from the cached forum topic name for topic sessions, expose it through sessions.list, and make the Control UI selector prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. Simulate or send a Telegram forum-topic message after the topic name has been learned: current source and tests show TopicName reaches the Telegram context and prompt metadata, while the stored session metadata, Gateway row, and UI selector still derive the visible label from existing label/displayName/group fields rather than the topic name.

## Solution Assessment

Yes. A generic session display-label or ThreadLabel propagation path with Telegram supplying the known topic name is the narrowest maintainable fix; a UI-only key formatter or Telegram-specific core display rule would leave other session consumers inconsistent.
