## Summary

Keep open. Current main has the Telegram topic-name source, but the learned name still stops at context/prompt metadata and is not persisted or projected as a session display label for the Gateway row or Control UI selector. The recent issue discussion also treats this as the canonical tracker; the related implementation PRs are closed unmerged, and the still-open UI cleanup PR is adjacent rather than a complete source-to-session-label fix.

## What This Changes

Show Telegram forum topic sessions in the session dropdown with human-readable group/topic names instead of raw topic-qualified session keys or indistinguishable group labels.

## Best Possible Solution

Add a narrow additive generic thread/topic display-label path in session metadata, have the Telegram plugin populate it from the cached forum topic name for topic sessions, expose it through `sessions.list`, and make the Control UI selector prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. Send or simulate a Telegram forum-topic message after the topic name has been learned: current tests and source show `TopicName` reaches the Telegram context, while stored session metadata and the UI selector still derive the visible label from group/key fields rather than the topic name.

## Solution Assessment

Yes. End-to-end generic label propagation with Telegram supplying the known topic label is the narrowest maintainable fix; a UI-only formatter or Telegram-specific core rule would leave persistence and other session consumers inconsistent.
