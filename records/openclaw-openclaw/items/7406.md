## Summary

Keep open. Current main learns Telegram forum topic names and exposes them to context/prompt metadata, but the name is not persisted or projected as a session display label for Gateway session rows or the Control UI selector. The recent discussion also treats this as the canonical tracker; the directly linked implementation PRs are closed unmerged, while #51124 is adjacent UI cleanup rather than an end-to-end topic-name propagation fix.

## What This Changes

Request to show Telegram forum-topic sessions in the session dropdown with human-readable group and topic names instead of raw `agent:main:telegram:group:<id>:topic:<id>` keys.

## Best Possible Solution

Add an additive generic thread/topic display-label path in session metadata, have the Telegram plugin populate it from the cached forum topic name for topic sessions, expose it through `sessions.list`, and make the Control UI selector prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. Send or simulate a Telegram forum-topic message after the topic name has been learned: current source and tests show `TopicName` reaches the Telegram context, while stored session metadata and the selector still derive the visible label from group/key fields rather than the topic name.

## Solution Assessment

Yes. End-to-end generic label propagation with Telegram supplying the known topic name is the narrowest maintainable fix; a UI-only formatter or Telegram-specific core rule would leave persistence and other session consumers inconsistent.
