## Summary

Keep open. Current main can learn Telegram forum topic names and expose them as `TopicName`, but the persisted session metadata, gateway `sessions.list` row, and Control UI selector still only consume `label`/`displayName`/group fields, so the user-visible dropdown gap remains. The earlier implementation PRs are closed unmerged, and open PR #51124 is adjacent UI cleanup rather than the missing end-to-end topic-name propagation.

## What This Changes

Request to display Telegram forum-topic sessions with human-readable group/topic names in the session dropdown instead of raw topic-qualified session keys.

## Best Possible Solution

Land a narrow end-to-end patch that adds an additive generic thread/topic display-label path, populates it in the Telegram plugin from the cached `TopicName` for forum-topic sessions, persists/exposes it through session metadata and `sessions.list`, and has the Control UI selector prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to send a message in a Telegram forum topic after OpenClaw has learned the topic name, then inspect session metadata or the Control UI selector: `TopicName` is available in context, but `displayName` is still derived from group/key fields.

## Solution Assessment

Yes. The narrowest maintainable fix is generic label propagation through the existing session/gateway/UI path, with Telegram only supplying the known topic label; the adjacent UI-only cleanup PR is not sufficient because it does not create or persist the missing source label.
