## Summary

Current main still does not persist or project the known Telegram forum `TopicName` into session labels. Telegram now learns and exposes topic names in inbound context, but session metadata, gateway session rows, and the Control UI selector still use `label` / `displayName` / generic group fields, so this issue remains valid and canonical. Related PRs #7407 and #43105 are closed unmerged, and open PR #51124 is adjacent UI cleanup rather than a complete topic-name metadata fix.

## Best Possible Solution

Keep this issue open as the canonical tracker. Implement a narrow Telegram/session metadata fix that reuses the existing cached `TopicName`, persists or derives a human thread/topic display label for topic-qualified sessions, exposes it through `sessions.list`, and updates the Control UI selector to prefer that explicit label. Keep the core seam generic and let the Telegram plugin supply Telegram-specific topic names.

## Repair Work Prompt

Fix the Telegram forum topic session label gap tracked by this issue. Current main already learns Telegram forum topic names as `TopicName`; the missing piece is carrying that known value into session metadata / `sessions.list` / Control UI labels for topic-qualified sessions.

Scope:
- In the Telegram plugin, when a forum `TopicName` is known, supply a generic thread/topic label to the session metadata path. Prefer an additive generic seam such as `ThreadLabel` / topic-aware display metadata over hard-coding Telegram formatting in core.
- In core session metadata, persist or derive a stable human label/display name for topic-qualified group sessions without regressing other channel group labels.
- In gateway session rows, expose the topic-aware label through `sessions.list` so non-UI consumers benefit too.
- In Control UI, prefer the explicit topic-aware label/displayName in the session selector and keep existing disambiguation behavior.
- Add focused regression coverage for Telegram topic name capture into session metadata and UI selector rendering.
- Coordinate with open PR #51124 if necessary; do not implement only a UI cleanup that leaves metadata unchanged.
- Changelog: this is user-facing, so add a single-line active-version fix entry with `Thanks @Diaspar4u` unless maintainers decide the eventual PR is test/internal only.

Do not:
- Add Telegram-specific owner ids/defaults to core.
- Break existing non-Telegram group display label formats.
- Change dependency versions or broad plugin loading behavior.
