## Summary

Keep open. Current main still has the same end-to-end gap: Telegram can learn and expose forum topic names as context metadata, but stored session metadata, gateway session rows, and the Control UI selector do not consume that topic name for the visible session label.

## What This Changes

The issue asks OpenClaw to show Telegram forum-topic sessions with human-readable topic names in the session dropdown instead of raw topic-qualified session keys.

## Best Possible Solution

Land a narrow additive thread/topic display-label path: persist a generic label in session metadata, have the Telegram plugin populate it from the cached topic name for forum-topic sessions, expose it through `sessions.list`, and make the Control UI selector prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to send or simulate a Telegram forum-topic message after OpenClaw has learned the topic name, then inspect session metadata or the Control UI selector: `TopicName` is present in context, while the stored/displayed session label is still derived from group/key fields.

## Solution Assessment

Yes. The best fix is end-to-end generic label propagation with Telegram supplying the known topic label; a UI-only formatter or Telegram-specific core display rule would leave persistence and other session consumers inconsistent.
