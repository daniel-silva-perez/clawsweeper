## Summary

Keep open. Current main still learns Telegram forum topic names and exposes them to context/prompt metadata, but the topic name is not persisted into session display metadata, projected through gateway session rows, or preferred by the Control UI session selector.

## What This Changes

The issue asks OpenClaw to show Telegram forum-topic sessions with human-readable topic names in the session dropdown instead of raw topic-qualified session keys.

## Best Possible Solution

Land a narrow additive thread or topic display-label path: persist a generic label in session metadata, have the Telegram plugin populate it from the cached topic name for forum-topic sessions, expose it through `sessions.list`, and make the Control UI selector prefer that explicit label while preserving existing non-Telegram fallbacks.

## Reproduction Assessment

Yes. Send or simulate a Telegram forum-topic message after OpenClaw has learned the topic name, then inspect stored session metadata or the Control UI selector: `TopicName` is present in context, while the visible session label is still derived from group/key fields.

## Solution Assessment

Yes. End-to-end generic label propagation with Telegram supplying the known topic label is the narrowest maintainable fix; a UI-only formatter or Telegram-specific core rule would leave persistence and other session consumers inconsistent.
