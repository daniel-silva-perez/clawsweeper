## Summary

Keep this open. Current main has added durable cron recovery before lost-marking, but cron lost reconciliation still falls back to the transient in-memory active-job set, and the related open PR describes current cron execution paths that still fail to mark jobs active while they run.

## What This Changes

The issue requests preventing cron task audit rows from being falsely marked `lost` after gateway restart when the only missing backing signal is the transient in-memory active job set.

## Best Possible Solution

Keep the issue open until cron task reconciliation either uses durable live/terminal cron state for all cron execution paths or the active-job symmetry gap is closed and any terminal lost-row cleanup is deliberately handled through #63196.
