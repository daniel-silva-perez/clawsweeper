## Summary

Current main still does not implement stale cron session-row recovery, so this PR is not obsolete. The PR remains a plausible fix direction, but the provided diff is incomplete because recovery is coupled to retention pruning, appears limited to run-alias rows, and uses a terminal status that conflicts with existing interrupted-cron semantics.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is a rebased cron recovery patch that runs independently of retention pruning, updates the stable base cron session row plus any legacy run-alias rows, chooses a terminal status consistent with interrupted-job semantics, and adds focused regression coverage for retention-disabled configs, stale versus active running rows, base-key and run-key shapes, and short-retention prune deferral.

## Risks / Open Questions

- Merging the PR as described would skip orphan recovery when `cron.sessionRetention` is `false`, because the proposed recovery sits behind the retention-disabled early return.
- A recovery pass limited to `cron:<job>:run:<id>` aliases can leave the stable base cron session row stuck as `status: "running"`, which is the row current session listing surfaces.
- Marking a gateway-crashed cron run as `done` may contradict existing cron restart semantics that classify interrupted jobs as `error`.
- The added test coverage in the PR context does not cover retention-disabled recovery, base-key recovery, or short-retention defer-before-prune behavior.
