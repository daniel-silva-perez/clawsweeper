## Summary

Keep open. The PR addresses a real cron session recovery gap, but the current patch still misses the user-visible stable cron session row and skips recovery entirely when retention pruning is disabled; prior Greptile and ClawSweeper comments identified the same blocking paths, and current main still has the underlying behavior.

## What This Changes

The PR adds a two-hour stale-running recovery pass to `sweepCronRunSessions`, returns an additive `orphansRecovered` count, logs recoveries, and adds a unit test for marking old cron `:run:` rows as done.

## Best Possible Solution

Land a rebased cron session recovery patch that runs independently of retention pruning, repairs the visible stable cron session row plus any legacy `:run:` alias, uses the maintainer-approved terminal status for interrupted work, and adds focused regression coverage for retention-disabled, base-key, stale-versus-active, and short-retention cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recover stable cron session rows too:** `src/cron/session-reaper.ts:108`
  - body: The recovery branch only runs after `isCronRunSessionKey` and writes back to `store[key]`, so it can flip a hidden `:run:` alias while leaving the stable `agent:<agent>:cron:<job>` row stuck as `running`. Current isolated cron persistence stores mutable lifecycle state under the stable key, and session listing filters run aliases, so the visible row can remain stale after the sweep.
  - confidence: 0.92
- **[P2] Run recovery when retention is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: When `cron.sessionRetention` is `false`, this early return skips the orphan scan entirely. That setting disables pruning, but stale crashed sessions still need status recovery, so users who keep full cron history would never get the new behavior.
  - confidence: 0.94
