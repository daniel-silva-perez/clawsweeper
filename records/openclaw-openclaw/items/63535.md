## Summary

Keep this PR open. Current main still lacks stale-running cron session recovery, but the proposed patch has blocking behavior gaps: it skips recovery when retention pruning is disabled and only repairs hidden `:run:` rows while current cron lifecycle state is persisted under the visible stable cron session key.

## What This Changes

The PR adds a two-hour stale-running cron run recovery pass to `sweepCronRunSessions`, returns an additive `orphansRecovered` count, and adds unit coverage in `src/cron/session-reaper.test.ts`.

## Best Possible Solution

Land a rebased cron session recovery patch that runs independently of retention pruning, repairs the visible stable cron session row plus any legacy `:run:` alias, uses the maintainer-approved terminal status for interrupted work, and adds focused regression coverage for retention-disabled, stable-key, stale-versus-active, and short-retention cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recover stable cron session rows too:** `src/cron/session-reaper.ts:108`
  - body: The recovery path only scans `isCronRunSessionKey` entries and writes the recovered status back to `store[key]`. Current isolated cron lifecycle state is persisted under the stable `agent:<agent>:cron:<job>` key, and session listing hides `:run:` aliases, so the visible session row can remain stuck as `running` after the sweep.
  - confidence: 0.92
- **[P2] Run recovery when retention is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: When `cron.sessionRetention` is `false`, this early return skips the orphan scan entirely. Disabling pruning should preserve history, but stale crashed sessions still need status recovery, so this supported config mode would never get the new behavior.
  - confidence: 0.94
