## Summary

Keep this PR open. Current main still does not implement stale running cron-session recovery, and the submitted patch is useful but incomplete against the current session model: it only updates cron `:run:` aliases, is skipped when retention is disabled, and does not settle the terminal status semantics for interrupted cron work.

## What This Changes

The PR adds a two-hour stale-running detection pass to the cron session reaper, returns an `orphansRecovered` count, and adds a unit test for marking old cron run rows as done.

## Best Possible Solution

Keep the PR open for a rebased cron session recovery patch that runs independently of retention pruning, repairs the stable base cron session row plus any legacy run aliases, uses the maintainer-approved terminal status for interrupted work, and adds focused regression coverage for the retention-disabled and base-key cases.

## Risks / Open Questions

- Merging the PR as-is would leave `cron.sessionRetention: false` configurations without orphan recovery because the new scan is still behind the retention-disabled early return.
- A recovery pass limited to canonical `:run:` keys can leave the stable base cron session row stuck as `running`, and that base row is the one shown by session listing.
- The proposed `done` status may conflict with current interrupted-run semantics, which record gateway-restart interruptions as cron errors and map error lifecycle events to failed session state.
- The added test does not cover short-retention defer-before-prune behavior, retention-disabled recovery, or base-key recovery.
