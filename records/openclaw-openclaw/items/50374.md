## Summary

Keep open. Current main still builds OpenAI-compatible Chat Completions payloads from the upstream converter without an outgoing strict message-key policy; existing compatibility hooks only flatten pure text content arrays and adjust unrelated request fields. The KeepMy Claw workaround in the discussion is useful external routing context, but it does not close the in-repo provider compatibility gap.

## What This Changes

The issue requests an OpenAI-compatible provider serialization policy that strips unsupported message keys before dispatch to strict Chat Completions implementations.

## Best Possible Solution

Add a provider/model-level OpenAI-compatible Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, documents interaction with supportsTools and requiresStringContent, and covers prior assistant tool calls plus tool-result history in tests.

## Repair Work Prompt

Fix the OpenAI-compatible provider message-key compatibility gap tracked by openclaw/openclaw#50374, with related context from #45706. Add an explicit provider/model-level outgoing Chat Completions message serialization policy for `api: "openai-completions"` that can strip or allow only configured message keys for strict providers. The policy must run after upstream `convertMessages` and before dispatch in both the direct transport path and the embedded OpenAI wrapper/payload path, and it must not mutate stored transcripts. Do not blanket-strip all OpenAI-compatible providers by default; preserve normal tool replay for providers that support standard Chat Completions tool keys. Update config types, Zod schema, schema help/generated docs as needed, and document how the new policy composes with `compat.supportsTools`, `compat.requiresStringContent`, `requiresToolResultName`, and prior assistant tool-call/tool-result history. Add focused tests covering assistant `tool_calls`, tool-result `tool_call_id`/`name`, `reasoning_details`, and composition with `requiresStringContent`. Include a changelog fix entry with reporter credit; do not add provider-specific core defaults unless a maintainer requests them.
