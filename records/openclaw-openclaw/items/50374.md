## Summary

Current main still does not provide a strict outgoing message-key sanitizer for generic `openai-completions` providers. The existing mitigations cover content flattening, tool-call-id replay hygiene, turn ordering, and plugin/provider stream wrappers, but the shared Chat Completions transport still sends the `@mariozechner/pi-ai` converted message objects with fields such as `tool_calls`, `tool_call_id`, `name`, and reasoning-related keys when history contains tool calls/results. The KeepMy Claw workaround mentioned in the issue discussion is useful external routing advice, not an OpenClaw core fix.

## Best Possible Solution

Keep this issue open and decide a narrow strict Chat Completions serialization policy. The safest implementation is probably opt-in per provider/model or provider-owned, applied after `convertMessages` and before dispatch in both the direct OpenAI-compatible transport and embedded wrapper paths, with tests for prior assistant tool calls and tool results and docs explaining the interaction with `supportsTools` and `requiresStringContent`.

## Risks / Open Questions

- A blanket sanitizer for every `openai-completions` provider could break tool-capable OpenAI-compatible backends that correctly require `tool_calls` and `tool_call_id` for replay.
- A role/content-only sanitizer may make prior tool-call and tool-result history semantically lossy, and may still need a policy for whether `role: "tool"` messages are converted, dropped, or rewritten when `compat.supportsTools` is false.
- Related #45706 reports another strict OpenAI-compatible 422 path, and #44086 covers null assistant content around tool calls; the eventual fix should avoid overlapping one-off compatibility knobs.
