## Summary

Current main still lacks a strict outgoing message-key sanitizer for generic OpenAI-compatible Chat Completions providers. The direct transport and embedded wrapper have related compatibility hooks, but they only flatten pure text content, strip some top-level proxy-only fields, and apply tool/replay hygiene; they do not expose or apply a provider/model policy that strips each converted message object to a strict accepted key set. The KeepMy Claw workaround linked in the discussion is useful external routing advice, but it is not an OpenClaw core fix.

## What This Changes

The issue asks OpenClaw to sanitize outgoing OpenAI-compatible Chat Completions messages so stricter providers do not reject extra keys in tool-call history.

## Best Possible Solution

Add a provider/model-level OpenAI-compatible Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, documents how it interacts with `supportsTools` and `requiresStringContent`, and tests prior assistant tool calls plus tool-result history.

## Risks / Open Questions

- The allowed-key policy needs to be opt-in or tightly capability-scoped; stripping all non-`role`/`content` keys for every `openai-completions` provider would break providers that accept standard tool-call replay, Google thought-signature replay, or other supported Chat Completions fields.
- Infomaniak's reported `values: ["role", "content"]` shape is stricter than the standard OpenAI Chat Completions tool-call schema, so the implementation should make the chosen mode explicit rather than silently treating all OpenAI-compatible providers the same.
