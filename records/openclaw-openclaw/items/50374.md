## Summary

Keep open. Current main has newer OpenAI-compatible replay hygiene for tool-call ids, turn ordering/validation, and string-only content backends, but I found no implementation or config/API surface that strips outgoing chat-completions message objects down to a strict provider-accepted key set for Infomaniak-style OpenAI-compatible servers. The existing KeepMy Claw workaround mentioned in the discussion is useful routing advice, but it does not close the core/provider compatibility gap reported here.

## Best Possible Solution

Keep this issue open and implement a narrowly scoped OpenAI-compatible message serialization policy. The likely direction is a model/provider compat flag or provider hook that sanitizes outbound `openai-completions` `messages` immediately after `convertMessages` and before request dispatch, with regression coverage for tool-call history against a strict key whitelist and docs under the provider transcript hygiene/OpenAI-compatible provider guidance.
