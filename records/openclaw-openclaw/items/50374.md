## Summary

Keep open. Current main still sends OpenAI-compatible Chat Completions messages from the upstream converter without a provider/model policy for strict accepted message keys; existing compatibility hooks only cover string-content flattening, store/request overrides, and replay hygiene. The KeepMy Claw workaround noted in the discussion is useful external routing context, but it is not an in-repo fix.

## What This Changes

The issue asks OpenAI-compatible Chat Completions payloads to strip or transform outgoing message object keys so strict providers such as Infomaniak accept histories with tool calls and tool results.

## Best Possible Solution

Add an explicit provider/model-level OpenAI-compatible Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, documents interaction with supportsTools and requiresStringContent, and covers prior assistant tool calls plus tool-result history in tests.
