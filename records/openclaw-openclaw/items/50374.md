## Summary

Keep open. Current main still has related OpenAI-compatible compatibility hooks, but they only handle string-content flattening, store stripping, request overrides, and tool/replay hygiene; there is no provider/model policy that strips outgoing Chat Completions message objects to a strict accepted key set. The KeepMy Claw workaround from the discussion is useful external routing context, but it is not an in-repo fix.

## What This Changes

The issue asks OpenAI-compatible providers to strip or transform outgoing Chat Completions message objects so strict backends do not reject extra keys from tool-call history.

## Best Possible Solution

Add a provider/model-level OpenAI-compatible Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, documents how it interacts with `supportsTools` and `requiresStringContent`, and covers prior assistant tool calls plus tool-result history in tests.
