## Summary

Keep open. Current main still forwards the upstream OpenAI Chat Completions message objects without a provider/model-level message-key policy, so strict OpenAI-compatible providers can still receive `tool_calls`, `tool_call_id`, `reasoning_details`, `name`, or injected `extra_content` keys. The KeepMy Claw gateway workaround in the discussion is useful external context, but direct OpenClaw provider calls remain unfixed.

## What This Changes

The issue requests sanitizing outbound `openai-completions` chat messages so strict OpenAI-compatible providers are not sent unsupported message keys such as tool-call metadata.

## Best Possible Solution

Add an opt-in provider/model compatibility setting for outbound OpenAI Chat Completions message serialization, applied after conversion and before dispatch in both direct transport and embedded payload paths, while preserving stored transcripts and documenting how it interacts with `supportsTools` and `requiresStringContent`.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a direct or embedded `openai-completions` request with prior assistant tool-call and tool-result history; current source plus the locked converter show those messages can include the strict-provider-rejected keys described in the report.

## Solution Assessment

Yes. The narrow maintainable direction is an explicit provider/model serialization policy rather than automatically stripping every `openai-completions` payload, because some compatible providers support standard tool replay fields.
