## Summary

Keep open. Current main still sends OpenAI-compatible Chat Completions messages from the upstream converter without a provider/model policy to strip strict-provider-rejected message keys; the existing compatibility paths flatten string content and shape request-level fields, but do not sanitize tool-call or tool-result message objects. The KeepMy Claw workaround mentioned in the discussion is useful external routing context, but it does not fix direct OpenClaw provider calls.

## What This Changes

Request to sanitize outbound `openai-completions` message objects so strict OpenAI-compatible providers receive only accepted message keys instead of converter/tool-replay extras.

## Best Possible Solution

Add an explicit provider/model-level OpenAI-compatible Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, defaults existing providers to current behavior, and documents how it interacts with `supportsTools` and `requiresStringContent`.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a direct or embedded `openai-completions` request with prior assistant tool-call and tool-result history: current code converts that history into `tool_calls` and `tool_call_id` message keys, matching the strict-provider 422 described in the issue.

## Solution Assessment

Yes, an opt-in provider/model message-key serialization policy is the narrowest maintainable fix. Automatically stripping all `openai-completions` messages would be riskier because some compatible providers accept and require the standard tool-replay keys.
