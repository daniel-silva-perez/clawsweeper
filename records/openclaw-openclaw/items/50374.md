## Summary

Keep open. Current main still builds OpenAI-compatible Chat Completions payloads from the upstream converter and only applies string-content flattening, tool-id cleanup, and replay hygiene; it does not provide a provider/model policy that strips outbound message object keys for strict role/content-only backends. The KeepMy Claw workaround in the discussion is useful external routing context, but it does not fix OpenClaw's direct provider path.

## What This Changes

The issue requests sanitizing `openai-completions` outgoing message objects so strict OpenAI-compatible providers such as Infomaniak do not reject extra keys from tool-call history.

## Best Possible Solution

Add an explicit OpenAI-compatible Chat Completions message serialization policy at the provider/model compat layer, run it after message conversion and before dispatch in both direct and embedded payload paths, preserve stored transcripts, document its interaction with `supportsTools` and `requiresStringContent`, and cover assistant tool-call plus tool-result history in tests.
