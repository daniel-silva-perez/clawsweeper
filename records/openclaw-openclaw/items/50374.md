## Summary

Current main still lacks an outgoing strict message-key sanitizer for generic `openai-completions` providers. The direct transport and embedded wrapper can flatten string content and apply replay/tool-history hygiene, but neither exposes or applies a provider/model policy that strips converted Chat Completions message objects down to a strict accepted key set. The KeepMy Claw routing workaround in the discussion is useful external advice, but it is not an OpenClaw core fix.

## What This Changes

The issue requests a strict outgoing Chat Completions message serialization option for OpenAI-compatible providers so strict backends can receive only accepted message keys after tool-call history.

## Best Possible Solution

Keep this issue open and implement a narrow, opt-in OpenAI-compatible Chat Completions message-key serialization policy. The policy should run after message conversion and before dispatch in both direct and embedded payload paths, preserve stored transcripts, document interaction with `supportsTools` and `requiresStringContent`, and include tests for prior assistant tool calls and tool-result history.
