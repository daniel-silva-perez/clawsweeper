## Summary

Keep open. Current main still sends OpenAI-compatible Chat Completions messages from the upstream converter without a provider/model policy for strict accepted message keys; the existing compat hooks cover string-content flattening, request fields, and replay hygiene, but not stripping message object keys. The KeepMy Claw workaround in the discussion is useful external routing context, but it does not fix OpenClaw's direct provider path.

## What This Changes

Request to sanitize OpenAI-compatible provider Chat Completions messages by stripping unsupported message-object keys before dispatch to stricter endpoints such as Infomaniak.

## Best Possible Solution

Add an explicit provider/model-level OpenAI-compatible Chat Completions message serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, documents interaction with supportsTools and requiresStringContent, and covers assistant tool-call plus tool-result history in tests.
