## Summary

Keep open. Current main still builds OpenAI-compatible Chat Completions payloads from the upstream converter without a provider/model message-key policy; the existing compat hooks flatten string content and adjust unrelated request fields, but do not strip keys such as `tool_calls`, `tool_call_id`, `reasoning_details`, or `name`. The KeepMy Claw workaround in the discussion is useful external routing context, but it does not resolve the in-repository direct provider path.

## What This Changes

Add OpenAI-compatible provider support to strip or transform outgoing Chat Completions message objects so strict providers receive only accepted message keys.

## Best Possible Solution

Add an explicit OpenAI-compatible Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, documents interaction with `supportsTools` and `requiresStringContent`, and covers prior assistant tool-call plus tool-result history in tests.

## Repair Work Prompt

Fix #50374, and consider whether the same fix resolves related #45706. Add an opt-in provider/model-level OpenAI-compatible Chat Completions message serialization policy that can strip outgoing `messages[]` objects down to the keys accepted by strict endpoints such as Infomaniak, including tool-call history cases. Apply the policy after `convertMessages` and before dispatch in both `buildOpenAICompletionsParams` and the embedded OpenAI payload wrapper path. Preserve stored transcripts and default OpenAI/tool-capable provider behavior; do not add a blanket strip-all policy for every `openai-completions` model.
