## Summary

Keep this issue open. The report is fresh, includes a credible rollback boundary from v2026.4.27 to 2026.4.23, and current main does not prove the WhatsApp/MiniMax/fallback privacy regression is fixed: the key MiniMax and WhatsApp reasoning guards either already existed in the reported release or only cover payloads that are already marked/prefixed as reasoning.

## What This Changes

The issue reports that upgrading a WhatsApp direct session from 2026.4.23 to v2026.4.27 made the gateway unstable, exposed thinking text as WhatsApp messages, and appeared to stick the session on MiniMax instead of the configured Codex primary model.

## Best Possible Solution

Treat this as a current release regression cluster: collect redacted v2026.4.27/current-main bad-state /status output, session fallback fields, runtime fallback attempts, gateway stall excerpts, and one sanitized leaked-message sample, then split confirmed fixes between WhatsApp delivery filtering, MiniMax thinking transport handling, fallback-state cleanup, and the #74860 event-loop stall path.

## Reproduction Assessment

No high-confidence reproduction is available yet. The rollback boundary is credible, but reproducing needs a bad-state run on v2026.4.27 or current main with redacted leaked text, fallback transition/status data, and gateway logs/profiles.

## Solution Assessment

Unclear. There is no patch under review, and the existing MiniMax and WhatsApp guards either already existed in v2026.4.27 or only cover structured/prefixed reasoning payloads, so maintainer-led artifact collection is the safest next step before a repair PR.
