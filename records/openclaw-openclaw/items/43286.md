## Summary

Keep this issue open. Current main still routes Slack thread replies into thread-suffixed sessions and has no `session.threadIsolation` or equivalent shared-thread config. The linked implementation attempts are useful design input, but they were closed unmerged after explicit cross-user/context-leakage concerns, so cleanup-close would be premature.

## What This Changes

The issue requests an opt-in Slack shared-thread session mode so thread replies can reuse the parent or main session while preserving Slack thread metadata and per-thread history handling.

## Best Possible Solution

Keep isolated defaults, then design an additive shared-session model with explicit trust boundaries. A Slack implementation should align inbound and outbound session keys, key thread history by Slack channel/thread identity rather than shared session key, track first-seen thread state independently from session existence, preserve `MessageThreadId`, document the leakage tradeoff, and update config types/schema/help/labels/docs plus focused routing tests.
