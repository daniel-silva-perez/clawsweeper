## Summary

Keep open. Current main still lacks a `session.threadIsolation` or `session.channelIsolation` config surface, Slack thread replies still flow through thread-suffixed session keys, and the linked implementation PRs were closed unmerged; the remaining work changes Slack context isolation and needs maintainer product/security direction rather than conservative cleanup.

## What This Changes

The issue requests an opt-in `session.threadIsolation: false` mode so Slack thread replies can share the parent or main session while preserving Slack thread metadata and per-thread history context.

## Best Possible Solution

Keep isolated behavior as the default, then design an additive opt-in shared-session model with explicit Slack trust boundaries. A durable implementation should align inbound and outbound session keys, key history by channel/thread identity instead of shared session key, track first-seen thread state independently from session existence, preserve `MessageThreadId` for reply targeting, document leakage/reset tradeoffs, and add focused routing/config/docs tests.

## Reproduction Assessment

Yes. A static reproduction is high confidence: Slack thread replies go through `resolveSlackRoutingContext`, which calls `resolveThreadSessionKeys` without `useSuffix: false`, and existing tests assert the resulting `:thread:<threadTs>` session key.

## Solution Assessment

Unclear. `session.threadIsolation: false` is technically plausible, but the issue and related #19929 describe a trust-boundary design choice; a safer final shape may need explicit channel/thread groups or single-identity scoping rather than one broad switch.
