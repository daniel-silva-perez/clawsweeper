## Summary

Keep this issue open. Current main still lacks an opt-in Slack shared-thread session setting, and the remaining work is not safely cleanup-closeable because the linked implementation PRs were closed unmerged after explicit context-leakage and trust-boundary concerns.

## What This Changes

The issue requests an opt-in `session.threadIsolation: false` setting so Slack thread replies can share a parent or main session while preserving Slack thread metadata and per-thread history behavior.

## Best Possible Solution

Keep isolated defaults, then design an additive shared-session model with explicit trust boundaries. A future Slack implementation should align inbound and outbound session keys, key thread history by Slack channel/thread identity, track first-seen thread state independently from session existence, preserve `MessageThreadId`, and update config types, schema/help/labels, docs, and focused regression tests.
