## Summary

Keep this issue open. Current main still lacks an opt-in Slack shared-thread session setting, the observable Slack routing and thread-context gates still match the reported limitation, and the linked implementation PRs were closed unmerged after concrete trust-boundary concerns rather than replaced by a shipped fix.

## What This Changes

The issue requests an opt-in Slack setting that lets thread replies reuse a parent or shared session instead of appending a `:thread:<thread_ts>` suffix, while preserving per-thread history and Slack reply metadata.

## Best Possible Solution

Keep isolated defaults, then design an additive shared-session model with explicit trust boundaries. A future Slack implementation should align inbound and outbound session keys, key thread history by Slack thread identity, track first-seen thread state independently from session existence, preserve `MessageThreadId`, and update config types, schema/help/labels, docs, and focused regression tests.
