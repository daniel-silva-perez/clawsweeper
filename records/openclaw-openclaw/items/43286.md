## Summary

Keep this issue open. Current main still lacks an opt-in Slack shared-thread session setting, and the two linked implementation PRs were closed unmerged after concrete trust-boundary concerns. The broader #19929 design tracker is relevant, but it has not implemented or explicitly superseded these Slack-specific routing and history requirements.

## What This Changes

The issue requests an opt-in `session.threadIsolation: false` mode so Slack thread replies can share the parent or main session while retaining Slack thread metadata and per-thread history context.

## Best Possible Solution

Design an additive shared-session model with isolated defaults and explicit trust boundaries, then implement the Slack-owned thread-routing pieces after that shape is settled. The implementation should use the existing `resolveThreadSessionKeys` `useSuffix` seam, keep inbound and outbound session keys aligned, key thread history by Slack thread identity, track first-seen thread state independently from session existence, preserve `MessageThreadId`, and update config types, schema/help/labels, docs, and focused regression tests.
