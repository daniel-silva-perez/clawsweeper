## Summary

Keep this issue open. Current main still does not provide an opt-in Slack shared-thread session setting: the generic thread-key helper still defaults to appending `:thread:<id>`, Slack inbound routing does not pass `useSuffix: false`, Slack thread history/starter injection is still gated by session existence, and the config schema/docs expose no `threadIsolation` or `channelIsolation` keys. The linked implementation PRs #43360 and #43459 are useful prior art, but both were closed unmerged after explicit trust-boundary concerns, while #19929 is a broader open design tracker rather than an explicit superseder for these Slack-specific requirements.

## What This Changes

The issue requests an opt-in `session.threadIsolation: false` setting so Slack thread replies can share the parent session while preserving Slack thread metadata and per-thread history context.

## Best Possible Solution

Design an additive shared-session model with isolated defaults and explicit trust boundaries, then implement the Slack-owned thread-routing pieces only after that shape is settled. The Slack implementation should use the existing `resolveThreadSessionKeys` `useSuffix` seam, keep inbound/outbound session keys aligned, key thread history by Slack thread identity instead of a shared session key, track first-seen thread state independently of session existence, preserve `MessageThreadId` for reply targeting, and update config types, schema/help/labels, docs, and focused regression tests.
