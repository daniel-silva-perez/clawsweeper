## Summary

Keep this issue open. Current main still lacks a Slack `session.threadIsolation` or equivalent shared-thread setting, the linked implementation PRs are closed unmerged, and the remaining decision is a product/security trust-boundary choice rather than conservative cleanup.

## What This Changes

Add an opt-in Slack session setting so Slack thread replies can reuse a shared or parent session while preserving thread metadata and per-thread history behavior.

## Best Possible Solution

Keep isolated defaults, then design an additive shared-session model with explicit trust boundaries. A Slack implementation should align inbound and outbound session keys, key thread history by Slack channel/thread identity rather than shared session key, track first-seen thread state independently from session existence, preserve `MessageThreadId`, document leakage tradeoffs, and update config schema/help/labels/docs plus focused routing tests.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: current Slack thread replies flow through `resolveSlackRoutingContext` into `resolveThreadSessionKeys` with no `useSuffix: false`, and existing tests assert the `:thread:<thread_ts>` session suffix. Live Slack is not needed to verify the missing config surface.

## Solution Assessment

Unclear. The requested flat opt-out flag is a plausible starting point, but prior PR discussion exposed cross-user and cross-context leakage concerns; a safer solution likely needs explicit verified channel/thread grouping or identity-bound opt-in rather than a broad shared-session switch.
