## Summary

Keep this issue open. Current main still does not provide `session.threadIsolation` or `session.channelIsolation`, Slack thread replies still route through `:thread:<threadTs>` session suffixes by default, and the thread starter/history behavior remains keyed off session existence. The linked implementation PRs #43360 and #43459 are closed unmerged, so they are useful prior art rather than shipped behavior; the broader #19929 is related but does not explicitly supersede this Slack-specific request.

## Best Possible Solution

Keep this issue open. The best maintainer path is to decide the product shape for opt-in shared Slack thread sessions, then implement it in Slack-owned routing using the existing `resolveThreadSessionKeys`/`buildThreadAwareOutboundSessionRoute` `useSuffix` seam plus config type/schema/help/docs updates. The fix should keep defaults isolated, align inbound and outbound session keys, preserve `MessageThreadId` for reply targeting, key thread history by Slack thread identity rather than a shared session key, replace session-existence gates with per-thread seen state, document the context-leakage tradeoff, and add focused regression tests. If maintainers want #19929 to become the canonical tracker, migrate these Slack-specific requirements there explicitly before closing this as a duplicate.

## Risks / Open Questions

- The requested behavior intentionally reduces Slack thread/channel isolation and can leak context across conversations, so any implementation should stay opt-in with clear privacy/security docs.
- Inbound routing, outbound delivery, `MessageThreadId`, history keys, and last-route metadata are separate paths; a partial implementation could create mismatched session keys or deliver replies to the wrong Slack thread.
- The closed PRs are useful design notes, but they predate later routing/binding changes and should be reviewed against current main before being revived.
