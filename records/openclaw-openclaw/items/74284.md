## Summary

Keep open. Current main still applies stored session model overrides to heartbeat runs unless an explicit `heartbeat.model` resolves, and the tests/docs still preserve auto-fallback override persistence until reset. An open PR, #74932, uses closing syntax for this issue, so the right cleanup path is to review, land, or supersede that PR rather than close the issue early.

## What This Changes

The issue reports that heartbeat sessions without an explicit heartbeat model keep using a persisted auto fallback model after `agents.defaults.model.primary` changes and the gateway restarts.

## Best Possible Solution

Land or supersede #74932 with a narrow heartbeat/default-model-change rule that clears only stale direct auto-sourced fallback overrides when the recorded selected model no longer matches the current configured default.

## Reproduction Assessment

Yes. The source path is high-confidence: heartbeat without `heartbeat.model` reaches reply model selection with `hasResolvedHeartbeatModelOverride` false, and current tests assert stored overrides still apply in that case.

## Solution Assessment

Yes, the narrow model-selection boundary described by the linked PR is the best known direction: clear stale direct heartbeat auto-fallback overrides after a default change while leaving normal fallback persistence and explicit/user/legacy selections intact.
