## Summary

Keep open. Current main still has the heartbeat-specific stale stored-override path: heartbeat ticks without an explicit `heartbeat.model` enter reply model selection with the current configured default, but `createModelSelectionState` then reapplies the stored session override. The linked open PR #74932 directly targets this issue and uses closing linkage, so the issue should remain open until that PR is reviewed and merged or superseded.

## What This Changes

The issue reports that the autonomous heartbeat session `agent:main:main` can keep using a persisted auto-fallback model from session creation after `agents.defaults.model.primary` is changed and the gateway restarts.

## Best Possible Solution

Review and land or supersede #74932 with a narrowly scoped heartbeat model-selection rule that clears only stale direct `modelOverrideSource: "auto"` overrides selected for a previous effective default, while keeping the documented persistence behavior for all other override sources.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the current model-selection seam: a heartbeat run without explicit `heartbeat.model` starts from the updated default, then reapplies the stored session override because `hasResolvedHeartbeatModelOverride` is false; existing tests assert that behavior.

## Solution Assessment

Yes. The best direction is the open PR's narrow stale-direct-auto-fallback rule, not closing this issue before merge or applying a broad reset-on-restart policy that would conflict with documented fallback persistence.
