## Summary

Keep open. Current main mitigates the original agent-blocking failure by swallowing memory search/bootstrap embedding failures, but it still does not implement the requested credential routing: memory embeddings are selected from memory embedding adapters and provider-specific embedding credentials, not from the active DeepSeek chat provider.

## What This Changes

The issue reports that v2026.4.25 made memory-core choose OpenAI-scoped embedding credentials instead of DeepSeek provider credentials, causing embedding 401s and blocking agent replies.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make embedding credential selection deterministic and documented: either require/prompt explicit `memorySearch.provider` and `remote` configuration with doctor guidance, or add a DeepSeek/OpenAI-compatible memory embedding adapter if that endpoint and credential reuse are supported.

## Reproduction Assessment

Yes for the remaining credential-selection behavior: configure a DeepSeek chat model with `memorySearch.provider` left as auto and an OpenAI key present, and current source can select the OpenAI memory adapter independently of the DeepSeek provider. The original full agent-blocking failure no longer has the same throw-through path on current main because search/bootstrap failures are caught.

## Solution Assessment

Unclear. The current non-fatal mitigation is the right narrow protection for replies, but it is not a complete credential-routing fix; the safer next step is maintainer policy on explicit embedding configuration versus adding a supported DeepSeek-compatible embedding adapter.
