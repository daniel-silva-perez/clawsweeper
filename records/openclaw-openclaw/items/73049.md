## Summary

Keep open. Current main mitigates the agent-blocking failure mode by making memory-search embedding/bootstrap failures non-fatal, but the requested credential-routing behavior is still unresolved: memory embeddings are selected through `agents.*.memorySearch`, OpenAI remains an auto-selectable embedding adapter, and the bundled DeepSeek plugin does not register a memory embedding provider.

## What This Changes

The issue reports that upgrading to v2026.4.25 caused memory-core embeddings to use OpenAI-scoped credentials instead of the active DeepSeek provider credentials, producing 401s and blocking agent message processing.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make memory embedding credential selection deterministic and documented. The maintainable end state is either explicit `agents.defaults.memorySearch.provider`/`remote` configuration with doctor guidance, or a provider-owned DeepSeek/OpenAI-compatible memory embedding adapter if maintainers decide that endpoint and credential reuse are supported.

## Reproduction Assessment

Yes for the credential-selection path: current source shows memory embeddings resolve through `memorySearch.provider`/remote config and provider-scoped auth, while the DeepSeek plugin does not register an embedding adapter. The original full 401 requires the reporter's live proxy/key setup, and the agent-blocking propagation is now mitigated.

## Solution Assessment

Unclear. The current mitigation is the right narrow fix for preventing memory failures from blocking replies, but the requested automatic reuse of DeepSeek chat credentials needs a maintainer credential-policy decision before it is clearly the best fix.
