## Summary

Keep open. Current main and the latest release mitigate the worst reported impact by making memory search/bootstrap embedding failures non-fatal, but the requested credential-routing behavior is still not implemented: auto memory embeddings can select the OpenAI adapter from OpenAI credentials independently of the active DeepSeek chat model, and the bundled DeepSeek plugin does not register a memory embedding provider.

## What This Changes

The issue asks memory-core to use DeepSeek provider credentials for embeddings when the active agent model is DeepSeek instead of resolving OpenAI credentials and blocking replies.

## Best Possible Solution

Keep the shipped non-fatal memory-search behavior, then make memory embedding credential selection deterministic and documented. The maintainable end state is either explicit `agents.defaults.memorySearch.provider`/`remote` configuration with doctor guidance, or a provider-owned DeepSeek/OpenAI-compatible embedding adapter if maintainers confirm endpoint and credential semantics are supported.

## Reproduction Assessment

Yes for the credential-selection path: current source shows auto memory embeddings can select the OpenAI adapter and resolve OpenAI-scoped credentials while the DeepSeek plugin has no memory embedding adapter. The original full agent-blocking failure is mitigated on current main and in `v2026.4.29`.

## Solution Assessment

Unclear. The current non-fatal handling is the right narrow mitigation for agent replies, but automatically reusing DeepSeek chat credentials for embeddings is not proven to be the best fix because memory embeddings are provider-adapter scoped and DeepSeek does not expose that adapter today.
