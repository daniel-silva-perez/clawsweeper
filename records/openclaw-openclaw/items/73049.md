## Summary

Keep open. Current main mitigates the original agent-blocking failure by making memory search and bootstrap sync failures non-fatal, but it still does not satisfy the specific credential-routing request: memory embeddings are selected from memory embedding adapters and OpenAI-scoped credentials, not from the active DeepSeek chat model/provider.

## What This Changes

The issue asks memory-core to use the appropriate embedding credentials for a DeepSeek-backed agent instead of selecting OpenAI environment credentials and failing with embedding 401s.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make embedding credential selection deterministic and documented. The maintainable end state is either explicit `memorySearch.provider` and `remote` guidance with doctor diagnostics, or a supported DeepSeek/OpenAI-compatible memory embedding adapter if maintainers confirm that endpoint and credential reuse are valid.

## Reproduction Assessment

Yes for the credential-selection path: current source shows `provider: "auto"` can select the OpenAI memory adapter from OpenAI credentials independently of the active DeepSeek chat model, and the bundled DeepSeek plugin does not register a memory embedding provider. Live 401 reproduction is not necessary to keep this open, though it would require the reporter's exact invalid key/proxy environment.

## Solution Assessment

Unclear. The current non-fatal mitigation is the right narrow protection for agent replies, but it is not the complete fix for deterministic embedding credential routing; the safer next step is a maintainer decision on supported memory embedding providers and explicit configuration behavior.
