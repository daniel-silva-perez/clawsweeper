## Summary

Keep open. Current main and v2026.4.27 mitigate the worst reported impact by converting memory-search embedding failures into a structured unavailable result instead of throwing through the tool, and search bootstrap sync failures are logged and swallowed. The specific credential-routing request is still not implemented: the OpenAI embedding adapter still resolves OpenAI credentials and base URL from memory/provider config, while the bundled DeepSeek plugin does not expose a memory embedding provider.

## What This Changes

The issue requests that memory-core stop breaking agent replies after v2026.4.25 by avoiding OpenAI embedding credential selection when the active agent model uses DeepSeek.

## Best Possible Solution

The best end state is to keep the non-fatal memory-search behavior and have a maintainer decide the credential policy: either document that memory embeddings require explicit `agents.defaults.memorySearch.provider`/`remote` configuration, add a doctor warning for mismatched ambient OpenAI credentials, or design a supported DeepSeek/OpenAI-compatible embedding adapter path without coupling memory-core to the active chat provider.

## Reproduction Assessment

Yes for the credential-selection path: source inspection shows `provider: "auto"` can select the OpenAI embedding adapter from OpenAI credentials, independent of an active DeepSeek chat model. Live 401 reproduction would require the reporter's specific invalid OpenAI-compatible key and endpoint, so only the static reproduction is high-confidence.

## Solution Assessment

Unclear. The v2026.4.27 mitigation is the right narrow fix for preventing agent-message blockage, but reusing DeepSeek chat credentials for embeddings is not proven to be the best fix because the bundled DeepSeek plugin does not currently expose a memory embedding provider.
