## Summary

Keep open. Current main mitigates the worst reported impact by converting memory-search embedding failures and bootstrap sync failures into non-fatal unavailable/logged paths, and that mitigation is present in v2026.4.29. The specific credential-routing request remains unresolved: OpenAI memory embeddings still resolve OpenAI-scoped credentials, while the bundled DeepSeek plugin does not register a memory embedding provider.

## What This Changes

The issue reports that memory-core should not select ambient OpenAI embedding credentials when an agent is configured to use DeepSeek chat credentials.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make memory embedding credential selection deterministic and documented. The maintainable end state is either explicit `agents.defaults.memorySearch.provider`/`remote` configuration with doctor guidance, or a supported DeepSeek/OpenAI-compatible embedding adapter path if maintainers confirm that credentials and endpoint semantics are safe to reuse for embeddings.

## Reproduction Assessment

Yes for the credential-selection path: current source shows `provider: "auto"` can select the OpenAI embedding adapter from OpenAI credentials independently of the active DeepSeek chat model, and no DeepSeek memory adapter is registered. Live 401 reproduction would require the reporter's exact invalid key/proxy setup, while the original agent-blocking failure is mitigated on current main.

## Solution Assessment

Unclear. The shipped mitigation is the right narrow fix for preventing memory embedding failures from blocking agent replies, but automatically reusing DeepSeek chat credentials for embeddings is not proven to be the best fix because the DeepSeek plugin does not expose a memory embedding adapter today.
