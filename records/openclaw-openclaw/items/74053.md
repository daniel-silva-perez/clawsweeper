## Summary

Keep this PR open. Current main still builds the heartbeat runtime context with `ctx.To` copied from `sender`, so the branch’s exact change is not implemented; however, current main also has a broader system-event session-store guard that already prevents heartbeat/cron/exec turns from rewriting persisted `lastTo`, which reduces the original data-corruption path. The proposed runtime change remains narrow and directionally correct, with only a low-severity test-precision finding and no security or supply-chain concern.

## What This Changes

The PR changes heartbeat runtime context construction so `ctx.To` uses the resolved `delivery.to`, adds a regression test for target-none heartbeat sender leakage into persisted `lastTo`, and adds an Unreleased changelog entry.

## Best Possible Solution

Land the narrow heartbeat context correction on current main while keeping the existing session-store system-event guard in place; the regression test should assert the exact target-none `ctx.To` value and the changelog should continue crediting the contributor.

## Reproduction Assessment

Yes for the narrowed runtime-context issue: current main visibly sets heartbeat `ctx.To` from `sender`, while `resolveHeartbeatSenderId` can fall back to `heartbeat` when no candidate resolves. The persisted `lastTo` corruption is partly mitigated on current main by the system-event session guard, so the remaining reproducible behavior is the incorrect in-turn `ctx.To` value.

## Solution Assessment

Yes, deriving `ctx.To` from `delivery.to` while leaving `From: sender` is the narrowest runtime-context fix and aligns the field with the actual delivery target. The existing broader session persistence guard should remain because it protects heartbeat, cron, and exec event turns from route rewrites beyond this specific sender sentinel case.
