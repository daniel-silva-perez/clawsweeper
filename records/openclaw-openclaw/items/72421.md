## Summary

Keep this issue open. Current main has partial queue timeout and stuck-session recovery plumbing, but it still preserves active-looking session lanes by default, heartbeat still defers behind a busy target session lane, and there is still no `openclaw status --lanes` operator surface.

## What This Changes

The issue requests recovery and diagnostics for Discord session lanes that can remain held after cron, heartbeat, or system-event runs complete, including stale-lane release and lane-state visibility.

## Best Possible Solution

Keep this as a core session-lane reliability item until OpenClaw can distinguish stale queue bookkeeping from legitimate active work, recover or abort stale Discord/cron/heartbeat session lanes under a reviewed policy, and expose read-only lane state for operators.

## Reproduction Assessment

No, there is not a high-confidence deterministic reproduction for the exact Discord stall. The reporter provided credible long-run observations, and current source/tests show a plausible active-lane shape that remains preserved, but no focused test reproduces the terminal race end to end.

## Solution Assessment

No, current main is only a partial solution. The maintainable fix should be liveness-aware lane recovery plus diagnostics, not an unconditional reset of active-looking session work.
