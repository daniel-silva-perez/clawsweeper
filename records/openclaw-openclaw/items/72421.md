## Summary

Keep open. Current main has added useful partial protection around command-lane timeouts and stuck-session diagnostics, but it still does not fully resolve the reported stale Discord per-session lane symptom: recovery intentionally keeps active lane tasks, heartbeat still skips when the target session lane is busy, and there is no implemented `status --lanes` diagnostic surface.

## What This Changes

The issue asks OpenClaw to ensure Discord channel per-session lanes are released or safely recovered after cron, heartbeat, or system-event runs complete, with a stale-lane watchdog and lane diagnostics.

## Best Possible Solution

Keep the issue open as a core session-lane reliability item. The maintainable end state is a liveness-aware recovery path that can distinguish a genuinely active run from stale bookkeeping, plus a focused regression around cron/heartbeat/system-event completion on a Discord session key and a read-only lane diagnostic surface for operators.

## Reproduction Assessment

No, we do not have a high-confidence reproduction path yet. The reporter provided concrete long-run observations, and source inspection confirms the relevant busy-lane paths, but there is no tight unit or live repro proving the exact terminal branch that leaves the lane stale.

## Solution Assessment

No, the current main behavior is not yet the best complete solution. The partial stuck-session recovery is appropriately conservative, but the issue still needs instrumentation or liveness metadata before safely auto-releasing active-looking session lanes.
