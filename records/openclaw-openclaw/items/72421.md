## Summary

Keep this issue open. Current main contains useful partial stale-lane recovery and lane-timeout work, but the reported active-looking stale session lane is still intentionally preserved by recovery, heartbeat still skips busy target session lanes, and there is no `openclaw status --lanes` diagnostic surface.

## What This Changes

The issue requests core recovery and diagnostics for intermittent Discord session lanes that remain held after an agent run appears complete.

## Best Possible Solution

Keep this as a core session-lane reliability item. The desired end state is liveness-aware recovery that can distinguish genuine in-flight work from stale lane bookkeeping, regression coverage for cron/heartbeat/system-event completion on channel session keys, and a read-only lane-state diagnostic surface for operators.

## Reproduction Assessment

No, there is not a high-confidence deterministic reproduction. The reporter provided concrete long-run observations and current source confirms a plausible active-lane blockage path, but no focused test or live repro isolates the exact missed terminal release branch.

## Solution Assessment

No, current main is not yet the best complete solution. The partial timeout and stuck-session recovery work is useful, but a safer fix needs active-run liveness metadata or instrumentation before automatically releasing an active-looking session lane.
