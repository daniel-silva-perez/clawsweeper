## Summary

Keep this issue open. Current main has partial queue timeout, lane snapshot, and diagnostic recovery plumbing, but it still intentionally preserves active-looking session lanes, heartbeat still waits behind a busy target session lane, and the requested lane inspection surface is absent.

## What This Changes

The issue reports intermittent Discord channel session lanes staying held after cron, heartbeat, or system-event runs complete, causing repeated `lane wait exceeded` diagnostics until manual session cleanup and gateway restart.

## Best Possible Solution

Add liveness-aware core recovery that can safely identify stale session-lane bookkeeping, abort or release only when active work is no longer real, cover the Discord cron/heartbeat delivery shape with focused tests, and expose read-only lane snapshots through a supported diagnostic command.

## Reproduction Assessment

No. The reporter provided credible long-run observations and current source/tests show a plausible active-lane shape, but there is no deterministic end-to-end reproduction for the exact Discord cron/heartbeat stall on current main.

## Solution Assessment

No. Current main is a partial defensive implementation; the safer path is a maintainer-approved liveness policy plus diagnostics, not unconditional active-lane reset.
