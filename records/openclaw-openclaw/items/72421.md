## Summary

Keep this issue open. Current main has useful queue timeout, lane snapshot, and stale-session recovery plumbing, but the reported active-looking Discord/cron/heartbeat stall is still not safely resolved: recovery intentionally preserves active session lanes, heartbeat still waits behind a busy target session lane, and the requested lane status diagnostic surface is absent.

## What This Changes

The issue requests investigation and recovery for intermittent Discord session-lane stalls after runs complete, plus easier lane-state diagnostics.

## Best Possible Solution

Keep this as a core session-lane reliability item until OpenClaw can distinguish stale queue bookkeeping from legitimate active work, recover or abort stale session lanes with tests for Discord cron/heartbeat delivery paths, and expose read-only lane state for operators.

## Reproduction Assessment

No, there is not a high-confidence deterministic reproduction for the exact Discord stall. The reporter provided concrete long-run observations, and current source/tests show a plausible active-lane shape that recovery intentionally preserves, but no focused test reproduces the terminal cron/heartbeat race end to end.

## Solution Assessment

No, current main is only a partial solution. The maintainable fix should be liveness-aware lane recovery plus diagnostics, not an unconditional reset of active-looking session work.
