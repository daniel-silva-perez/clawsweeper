## Summary

Current main does not implement the requested `assistantPrefix` contract. Cron `agentTurn` payloads, agent defaults/config validation, `openclaw cron add/edit`, `openclaw agent`, and provider payload wrappers all lack an assistant-prefix field or provider-prefill path. The adjacent cron model/fallback work is already present, but it does not solve this fixed-reply-start guarantee, so this should remain open for maintainer API/design review.

## Best Possible Solution

Keep this open for maintainer API/design review. If accepted, implement it as an additive core protocol/runtime feature: add a validated optional prefix field to cron `agentTurn` payloads, agent/default config, and `agent` RPC/CLI surfaces; carry it through cron and normal agent execution into a typed provider payload hook; support at least Anthropic and one OpenAI-compatible route with explicit thinking-mode behavior; define no-op/debug semantics for unsupported providers; and document/test streaming plus fallback-delivery behavior.

## Risks / Open Questions

- The requested provider-prefill semantics are cross-provider and not uniform; current Anthropic/Cloudflare logic strips assistant prefill when extended thinking is enabled, which conflicts with a blanket assistant-prefill implementation.
- Ollama `raw: true` or prompt-template bypassing could change tool/chat formatting behavior if implemented without a provider-owned contract.
- A fixed prefix interacts with block streaming, preview streaming, final fallback delivery, and direct `message` tool sends; the guarantee needs to define whether the first streamed chunk, final payload, or every channel message must start with the prefix.
