## Summary

Keep open. Current main does not implement automatic per-agent Bedrock requestMetadata injection, and the linked feature request remains open. The PR is not obsolete, but it needs rework: it mixes unrelated Feishu media changes into a Bedrock agents feature and still has unresolved provider-boundary, wrapper-precedence, metadata-validation, and test-coverage concerns.

## Best Possible Solution

Keep the PR open for maintainer review/rework. Best path: split the unrelated Feishu changes, add a generic agent identity/context seam for provider hooks if needed, implement sanitized Bedrock requestMetadata in the amazon-bedrock provider-owned wrapper, preserve existing wrapper precedence, merge rather than drop caller metadata, and add focused tests for zero-extra-params Bedrock calls, invalid/long agent names, non-Bedrock no-op behavior, and configured metadata precedence. Coordinate the product surface with open #60602 and broader config request #57360.

## Risks / Open Questions

- Raw agentId/agentName injection needs validation or skip/truncation before sending to Bedrock. Current agent config accepts id/name as strings, while pi-ai documents Bedrock requestMetadata key/value limits; invalid names could turn every Bedrock call for an agent into a validation failure and would also expose configured names in AWS logs/cost tooling.
- The PR currently mixes an unrelated Feishu channel media fix into a Bedrock provider feature, increasing review and test scope. That should be split before maintainer review.
- The PR’s wrapper-order change can alter precedence between explicit extra params and provider defaults. A refreshed implementation should preserve current precedence and add regression coverage.
- No workflow, dependency, lockfile, lifecycle-script, or package-publishing changes were observed in the PR diff; the security-sensitive part is outbound metadata content rather than supply-chain execution.
