## Summary

Keep this PR open, but do not merge it as-is. Current main still has the memory-lancedb contamination problem, and this PR is the right extension-local direction, but the latest PR head still misses current inbound metadata shapes and whitespace-tolerant sentinel handling, leaving concrete privacy-sensitive bypasses.

## What This Changes

The PR adds memory-lancedb envelope-sludge detection, sanitizes auto-captured user text before embedding/storage, filters recalled memories before prompt injection, strips media annotations during prompt formatting, overfetches recall results, and adds regression tests.

## Best Possible Solution

Keep the PR open and revise it so the memory-lancedb sanitizer and detector mirror the current inbound metadata contract without importing core internals: tolerate leading whitespace, cover Location and arbitrary structured untrusted metadata blocks, add focused tests for those cases, rebase onto current main, and then rerun the targeted extension tests plus the changed gate.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction path through raw auto-capture and raw recall, and a focused probe of the PR logic shows indented sentinels plus Location/structured payloads are still missed.

## Solution Assessment

No, not as-is. The extension-local filtering approach is the right boundary, but the implementation needs to fully match current inbound metadata shapes before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Accept indented metadata sentinel lines:** `extensions/memory-lancedb/index.ts:533`
  - body: The new detector and plaintext sanitizer only match sentinel labels with `^${escapedSentinel}`, but existing inbound metadata stripping treats a sentinel line as metadata after `line.trim()`. An indented `Sender (untrusted metadata):` block is still transport metadata, yet this patch misses it, so a trigger phrase in the remaining text can still be embedded and stored. Add optional leading whitespace to the sentinel regexes and cover indented cases.
  - confidence: 0.9
- **[P2] Reject all current inbound metadata blocks:** `extensions/memory-lancedb/index.ts:498-505`
  - body: The copied sentinel list omits `Location (untrusted metadata):` and arbitrary structured-context labels such as `WhatsApp contact (untrusted metadata):` that current main emits. Those blocks use keys like `latitude`, `longitude`, `source`, `type`, and `payload`, so `ENVELOPE_JSON_LINE_RE` will not catch them; a location or structured block followed by `I always...` can still be stored as contaminated memory. Add coverage for the current labels or a generic line-anchored untrusted-metadata block match plus tests.
  - confidence: 0.86
