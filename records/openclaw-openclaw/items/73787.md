## Summary

Keep open. The PR addresses a real memory-lancedb contamination bug and usefully supersedes the closed #67475 attempt, but the current head still leaves concrete metadata-retention bypasses in the new filter and the branch is dirty against current main.

## What This Changes

The PR adds memory-lancedb envelope metadata detection/sanitization, filters contaminated recalled memories, strips recalled media annotations, and extends the extension tests.

## Best Possible Solution

Revise the extension-local filter so it mirrors the current inbound metadata contract: tolerate leading whitespace like the canonical stripper, cover Location and arbitrary structured untrusted metadata labels, add focused regression tests for those cases, rebase onto current main, and then run the memory-lancedb tests plus the changed gate.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection gives a direct reproduction: feed the PR head sanitizer a user message with an indented Sender sentinel or a Location/structured metadata block followed by a memory trigger such as “I always prefer dark mode”; the metadata survives into capture/recall filtering.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. Keeping the fix inside memory-lancedb is the right boundary, but the detector and sanitizer need to fully track the core-generated inbound metadata forms before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Accept indented metadata sentinels:** `extensions/memory-lancedb/index.ts:533`
  - body: The new sludge detector and plaintext sanitizer only match sentinels with `^${escapedSentinel}`, but the canonical inbound metadata stripper treats a line as metadata after `line.trim()`. An indented `Sender (untrusted metadata):` block can therefore bypass the new guard and still be embedded/stored if trigger text remains. Allow optional leading whitespace and add indented-sentinel tests.
  - confidence: 0.9
- **[P2] Reject every current inbound metadata label:** `extensions/memory-lancedb/index.ts:498-505`
  - body: The copied sentinel list omits current `Location (untrusted metadata):` blocks and arbitrary structured labels such as `WhatsApp contact (untrusted metadata):`. Those payloads use fields like `latitude`, `longitude`, `source`, `type`, and `payload`, which the JSON-key fallback does not catch, so metadata followed by a memory trigger can still be persisted. Add generic line-anchored untrusted-metadata handling or sync the full producer contract with tests.
  - confidence: 0.86
