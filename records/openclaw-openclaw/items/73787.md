## Summary

Keep this PR open, but it is not merge-ready as-is. Current main still has the memory-lancedb sludge capture/recall gap, and the PR is a focused improvement over the closed noisy predecessor, but the final patch still misses current inbound metadata variants and a security review bypass already noted in discussion.

## What This Changes

The PR adds memory-lancedb envelope-sludge detection, sanitizes auto-captured user text before embedding/storage, filters recalled memories before prompt injection, strips recalled media annotations, and adds regression tests for capture, recall, and prompt formatting.

## Best Possible Solution

Update this PR so the memory-lancedb sanitizer and detector match the current inbound metadata contract: tolerate leading whitespace like the canonical stripper, cover Location and structured untrusted metadata blocks, add focused tests for those cases, rebase onto current main, and then rerun the memory-lancedb tests plus the changed gate.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection shows current main embeds/stores raw captured text and recalls raw top-three memories, and a focused regex probe against the PR logic shows indented sentinels and Location metadata are still missed; I did not run Vitest because this review was read-only.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The PR’s extension-local sanitizer/filtering approach is the right boundary, but it needs to fully mirror the current inbound metadata contract before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Allow indented metadata sentinels:** `extensions/memory-lancedb/index.ts:533`
  - body: The new sludge detector and plaintext sanitizer only match sentinels with `^${escapedSentinel}`, but the canonical inbound metadata stripper treats a sentinel line as metadata after `line.trim()`. An indented `Sender (untrusted metadata):` block is therefore still transport metadata, yet this PR misses it; if the remaining text contains a memory trigger it can still be embedded, stored, and recalled. Add optional leading whitespace to the sentinel regexes and cover indented cases.
  - confidence: 0.9
- **[P2] Reject all current inbound metadata blocks:** `extensions/memory-lancedb/index.ts:498-505`
  - body: The sentinel list omits `Location (untrusted metadata):` and arbitrary structured-context labels like `WhatsApp contact (untrusted metadata):` that current main emits from `buildInboundUserContextPrefix`. Those blocks use keys such as `latitude`, `longitude`, `source`, `type`, and `payload`, so `ENVELOPE_JSON_LINE_RE` will not catch them; a location or structured block followed by `I always...` can still be stored as contaminated memory. Add coverage for the current labels or a generic line-anchored untrusted-metadata block match plus tests.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, package publishing, generated/vendor, or install-script changes were found, but the PR leaves concrete metadata-retention bypasses in the new memory filter.

Concerns:

- **[medium] Indented sentinels bypass sludge filtering:** `extensions/memory-lancedb/index.ts:533`
  - body: The patch’s sentinel regexes require the sentinel at column zero, while core stripping treats trimmed sentinel lines as metadata. Leading whitespace can therefore bypass the new capture/recall guard and preserve transport metadata in long-term memory.
  - confidence: 0.88
- **[medium] Location metadata can remain in long-term memory:** `extensions/memory-lancedb/index.ts:498`
  - body: Current main emits `Location (untrusted metadata):` blocks with location payload fields, but the PR does not detect or strip that label or its JSON keys. Location context followed by a memory trigger can still be persisted and later recalled.
  - confidence: 0.84
