## Summary

Keep this PR open. Current main still writes `src/infra/device-identity.ts` key material with direct `fs.writeFileSync` calls, so the PR is not obsolete. The idea is valid, but the best path is to update it to reuse the existing hardened JSON atomic writer and add targeted regression coverage before landing.

## Best Possible Solution

Keep the PR open and ask for a small rework: replace the two device identity direct writes with the existing synchronous `saveJsonFile` helper or an equally hardened shared helper, remove the misleading `EEXIST` mkdir wrapper, and add focused tests for atomic replacement, failed-write preservation, cleanup, and file mode. After that, review/land this as a targeted fix rather than closing it as implemented.

## Risks / Open Questions

- The PR should not land as a bespoke weaker writer if it can reuse `saveJsonFile`; predictable temp names, non-exclusive temp creation, missing fsync, and directory-sync gaps matter more for private key material than for ordinary JSON state.
- The PR currently has no new regression tests for the stated failure modes: original identity preservation on failed write, temp cleanup, and `0o600` permissions.
