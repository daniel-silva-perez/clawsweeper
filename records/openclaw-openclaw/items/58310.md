## Summary

Keep open. Current main still writes device identity key material with direct `fs.writeFileSync` calls, while the latest PR diff applies the intended temp-file-plus-rename path with owner-only permissions and addresses the earlier Greptile security note. I did not find a blocking correctness or supply-chain issue; this needs normal maintainer review and validation rather than cleanup closure.

## What This Changes

The branch adds a sync `atomicWriteFileSync()` helper in `src/infra/device-identity.ts` and routes device identity creation and mismatched-device-id repair writes through a temp-file write followed by rename with `0o600` permissions.

## Best Possible Solution

Review and land this PR, or an equivalent maintainer patch, after targeted identity tests and the normal changed gate. A focused follow-up test for failed temp writes preserving the original identity file would make the data-integrity claim easier to maintain.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a PR hardening change rather than a concrete runtime bug report; the affected code path is clear from the two direct write sites on current main, but no existing test simulates disk-full or partial-write failure.

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow goal of preventing direct partial overwrites of device identity files. If maintainers want full power-loss durability, the safer variant would be a sync equivalent of `writeTextAtomic()` with temp-file and parent-directory fsync, but that is a stricter follow-up rather than a merge blocker.
