## Summary

Keep this PR open. Current main still hardcodes the Slack assistant thread status, does not expose a typed `typingStatuses`, `statusMessages`, or `loading_messages` config surface, and the PR remains a focused implementation attempt even though it is not merge-ready against the current Slack plugin layout.

## Best Possible Solution

Keep this PR open for maintainer review, or replace it with a current implementation under `extensions/slack`. The best path is a typed Slack-specific status/loading-message config surface, likely under `channels.slack` with account-level inheritance, wired through the active monitor context and dispatch pipeline with validation for non-empty strings and Slack API limits, tests for default and configured behavior, and matching docs/config metadata updates.

## Risks / Open Questions

- The branch is not merge-ready because it targets the removed `src/slack` layout instead of the active `extensions/slack` plugin implementation.
- The proposed `identity.typingStatuses` shape puts Slack-specific runtime behavior into core agent identity; #67129 frames the same feature as Slack-channel configuration using Slack status/loading-message semantics.
- The proposed schema accepts empty strings and does not document or enforce Slack-side limits, so a configured empty entry could clear the indicator or fail API validation.
- The PR body still has unchecked manual testing items and the patch does not add current-main tests, docs, config help, or generated metadata updates.
