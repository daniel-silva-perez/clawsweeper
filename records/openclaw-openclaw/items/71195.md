## Summary

Keep open. Current main already has Gateway and browser realtime Talk support, but native macOS Talk Mode still uses the legacy speech-recognition, chat-send/history-polling, and TTS playback loop. Maintainer discussion has also explicitly narrowed the remaining work to native macOS integration using the existing realtime session path where possible.

## What This Changes

Add native macOS Talk Mode support for realtime speech-to-speech audio while keeping the current STT/chat/TTS path as fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up. The desired end state is Swift-side realtime session integration, AVAudioEngine capture, streaming playback, overlay phase/cancel handling, barge-in behavior, and fallback to the existing talk.speak path for non-realtime providers.

## Reproduction Assessment

Not applicable as a feature request; the current behavior gap is high-confidence from source inspection. Native macOS Talk still uses SFSpeechRecognizer plus chat.send/history plus TTS, while Gateway/browser realtime Talk support exists on current main.

## Solution Assessment

Yes for the narrowed maintainer direction, but not for the original new talk.realtime.open/audio/close API as the default path. The maintainable route is to reuse talk.realtime.session and existing realtime transports first, adding new gateway PCM methods only if native macOS cannot use the current contract.
