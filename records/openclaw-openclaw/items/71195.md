## Summary

Keep open. Current main already has the Gateway and browser realtime Talk machinery, but native macOS Talk still uses the legacy speech-recognition, chat, history-polling, and TTS path; the issue also has explicit maintainer direction to keep it open and narrow the remaining work to native macOS integration.

## What This Changes

Add a native macOS Talk Mode realtime speech-to-speech path using the existing realtime voice infrastructure, while keeping the current STT/chat/TTS path as fallback.

## Best Possible Solution

Keep this as the canonical native macOS Talk Realtime follow-up. The desired end state is Swift/macOS support that consumes the existing realtime Talk contract, adds native audio capture/playback and overlay/cancel wiring, and preserves the current STT/chat/TTS path for non-realtime providers; the ElevenLabs repeated-403 behavior should be split into its own narrower Talk TTS issue or PR.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Native macOS Talk still follows SFSpeechRecognizer -> `chat.send` -> `chat.history` -> TTS, while Gateway/browser realtime Talk is already implemented.

## Solution Assessment

No for the original new-method API sketch as the primary plan, because current main already has `talk.realtime.session` plus browser and Gateway relay transports. The best maintainable path is to reuse that contract for macOS and add native client/audio pieces unless maintainers choose a provider-agnostic backend bridge explicitly.
