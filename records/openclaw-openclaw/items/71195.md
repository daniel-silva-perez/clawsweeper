## Summary

Keep open. Current main has the Gateway and browser realtime Talk machinery, including `talk.realtime.session` and Gateway relay transports, but native macOS Talk still follows the legacy SFSpeechRecognizer -> `chat.send` -> `chat.history` -> `talk.speak`/TTS path. The issue discussion also has explicit maintainer direction to keep it open and narrow the remaining work to native macOS integration rather than adding a parallel backend API.

## What This Changes

Add native macOS Talk Mode realtime speech-to-speech support using the existing realtime voice/Talk session path while keeping the current STT/chat/TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS Talk Realtime follow-up. The desired end state is a macOS client implementation that consumes the existing `talk.realtime.session` contract, adds native session/audio capture/playback plumbing, updates overlay phase and cancellation behavior, and falls back cleanly to the current STT/chat/TTS path; any remaining ElevenLabs repeated-403 behavior should be handled in a separate Mac Talk TTS issue or PR.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Native macOS Talk still follows SFSpeechRecognizer -> `chat.send` -> `chat.history` -> `talk.speak`/TTS, while realtime Talk is implemented for Gateway/browser surfaces.

## Solution Assessment

No for the original new-method API sketch as the primary plan, because current main already has `talk.realtime.session` plus browser and relay transports. The best maintainable solution is to reuse that contract for macOS and add only the native client/audio pieces unless maintainers decide a new provider-agnostic backend bridge is required.
