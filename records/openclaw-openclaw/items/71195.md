## Summary

Keep open. Current main has the Gateway and browser realtime Talk surface, but native macOS Talk Mode still uses the legacy local speech-recognition, chat-send/history polling, and TTS playback loop. Maintainer discussion already narrowed the remaining work to native macOS integration on top of the existing realtime session path, not a new backend voice stack.

## What This Changes

The issue requests native macOS Talk Mode support for OpenAI Realtime speech-to-speech by integrating Swift audio capture/playback with the existing realtime Talk session flow while preserving the current TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up. The target state should add Swift-side realtime session/relay or WebRTC integration, AVAudioEngine capture, streaming PCM playback, overlay phase/cancel handling, fallback to the current talk.speak path for non-realtime providers, and focused docs/tests plus a live macOS smoke.

## Reproduction Assessment

Not applicable as a feature request; the current behavior gap is high-confidence from source inspection. Native macOS Talk still runs SFSpeechRecognizer to chat.send/history to TTS, while Gateway/browser realtime Talk support already exists on current main.

## Solution Assessment

Yes for the narrowed maintainer direction, but no for the original new talk.realtime.open/audio/close API sketch as the primary path. The maintainable route is to reuse talk.realtime.session and existing realtime transports first, adding new gateway PCM methods only if native macOS cannot use the current contract.
