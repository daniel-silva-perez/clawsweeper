## Summary

Keep open. Current main already has the Gateway/browser realtime Talk session and Gateway relay pieces, but native macOS Talk still uses the legacy SFSpeechRecognizer to chat.send/chat.history to talk.speak loop. The issue also has explicit maintainer direction to keep it open and narrow the remaining work to native macOS integration on top of the existing realtime session flow.

## What This Changes

Add native macOS Talk support for realtime speech-to-speech using the existing Gateway realtime Talk session/relay contract while preserving the current STT/chat/TTS fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up. The target state should add Swift wrappers for talk.realtime.session/relay, native capture/playback/cancel integration in TalkModeRuntime, overlay phase updates, fallback to the current talk.speak path for non-realtime providers, and focused docs/tests plus a live macOS smoke.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection: native macOS Talk still uses SFSpeechRecognizer, chat.send/history polling, and talk.speak playback while Gateway/browser realtime Talk already exists.

## Solution Assessment

No for the original new-method API sketch as the primary path; current main already has talk.realtime.session plus browser and relay transports. The narrower maintainable solution is to reuse that contract for macOS and add only native client/audio pieces unless maintainers explicitly choose a different backend bridge.
