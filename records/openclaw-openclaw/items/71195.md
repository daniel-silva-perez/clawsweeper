## Summary

Keep open. Current main has the shared Gateway/browser realtime Talk path, including `talk.realtime.session` and relay support, but native macOS Talk Mode still runs the legacy SFSpeechRecognizer to `chat.send`/history to `talk.speak` playback loop. The issue also has explicit maintainer direction to keep it open and narrow the work to native macOS integration on top of the existing realtime session flow.

## What This Changes

Request native macOS Talk Mode support for realtime speech-to-speech using the existing realtime voice plumbing while preserving the current STT/chat/TTS fallback path.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up. The target state should add Swift-side wrappers for the existing realtime Talk session/relay contract, native capture/playback/cancel integration in `TalkModeRuntime`, overlay phase updates, fallback to the current `talk.speak` path for non-realtime providers, and docs/tests for the changed macOS behavior.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Native macOS Talk still runs local speech recognition and `talk.speak` playback, while Gateway/browser realtime Talk support already exists on current main.

## Solution Assessment

No for the original new-method API sketch as the primary path; current main already has `talk.realtime.session` plus browser and relay transports. The narrower maintainable solution is to reuse that contract for macOS and add only the native client/audio pieces unless maintainers explicitly choose a different backend bridge.
