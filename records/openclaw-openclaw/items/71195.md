## Summary

Keep this issue open. Current main has the shared realtime Talk backend, browser Control UI realtime Talk, OpenAI browser-session support, and a Gateway PCM relay, but native macOS Talk still uses the legacy SFSpeechRecognizer -> chat.send -> chat.history -> talk.speak/local playback loop and has no realtime session client wrapper or native WebRTC/relay audio flow. The issue discussion also has explicit maintainer direction to keep it open and narrow it to native macOS integration.

## Best Possible Solution

Keep this issue open as the canonical native macOS Realtime Talk follow-up. The best path is to reuse the existing talk.realtime.session contract, decide whether macOS should consume direct OpenAI WebRTC-SDP sessions or the authenticated Gateway relay PCM transport, then add a native macOS realtime session object, GatewayConnection wrappers, AVAudioEngine/WebRTC or relay audio capture/playback, Talk overlay phase/cancellation wiring, and a clean fallback to the existing STT -> chat -> TTS path. Split any remaining ElevenLabs repeated-403 behavior into a separate, smaller Mac Talk TTS issue or PR.
