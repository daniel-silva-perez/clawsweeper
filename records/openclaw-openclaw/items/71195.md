## Summary

Keep open. Current main already has the Gateway/browser realtime Talk surface, but native macOS Talk Mode still uses the legacy local speech recognition, chat send/history polling, and `talk.speak` playback loop. The maintainer comments in the issue explicitly narrow the remaining work to native macOS integration on top of `talk.realtime.session`, with the ElevenLabs PCM 403 subthread split separately.

## What This Changes

Add native macOS Talk support for OpenAI Realtime speech-to-speech by reusing the existing realtime Talk session/relay flow while preserving the current STT/chat/TTS path as fallback.

## Best Possible Solution

Keep this as the canonical native macOS realtime Talk follow-up. The target state should add Swift-side wrappers for the existing realtime session/relay contract, native capture/playback/cancel integration in `TalkModeRuntime`, overlay phase updates, fallback to the current `talk.speak` path for non-realtime providers, and focused docs/tests for the changed macOS behavior.

## Reproduction Assessment

Not applicable as a feature request, but the behavior gap is high-confidence from source inspection. Native macOS Talk still runs local speech recognition and `talk.speak` playback, while Gateway/browser realtime Talk support already exists on current main.

## Solution Assessment

No for the original new-method API sketch as the primary path; current main already has `talk.realtime.session` plus browser and relay transports. The narrower maintainable solution is to reuse that contract for macOS and add only the native client/audio pieces unless maintainers explicitly choose a different backend bridge.
