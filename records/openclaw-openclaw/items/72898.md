## Summary

Keep this PR open. Current main still treats iMessage chat.db permission-denied output as ordinary stderr or a non-fatal probe error, while the monitor already has a fatal probe path that this PR uses appropriately. The change is narrow, useful, and not superseded by the broader open request about avoiding parent Node Full Disk Access entirely.

## What This Changes

The PR adds iMessage RPC chat.db permission-denied detection, fails pending RPC calls, stops the child process on fatal startup denial, and marks probe results fatal with a regression test.

## Best Possible Solution

Review and land this PR or an equivalent narrow iMessage plugin change, while keeping #43440 open for the separate process-context or upstream imsg architecture work.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to mock createIMessageRpcClient().request() to reject with an imsg permissionDenied error for Library/Messages/chat.db; current main returns a non-fatal probe result, while the PR expects fatal true.

## Solution Assessment

Yes for the bug this PR targets. The narrowest maintainable fix is in the iMessage RPC client/probe layer because the monitor already has a generic fatal-probe escape hatch, and core does not need iMessage-specific permission policy.
