## Summary

Keep open. Current main still allows builtin `memory_search` to query an existing index while dirty search sync runs in the background. The code only blocks for a forced bootstrap when there is no indexed content at all; if chunks/FTS rows already exist, it proceeds with the current committed state, which leaves the stale/transitional-index class reported here still plausible. The related open PR #59137 is useful memory reindex recovery work, but its own body says it does not change query behavior and does not close this issue.

## Best Possible Solution

Keep this issue open and fix it in `extensions/memory-core`: when builtin search sees dirty state, pending sync, or full-reindex-needed metadata, either await the sync/reindex before querying or return an explicit transient/indexing status instead of an empty result. Add focused regression coverage around existing populated SQLite/FTS data plus dirty or reindex-needed state so `memory_search` cannot fail closed until a manual forced reindex is run.

## Evidence

- **Builtin search only forces sync for an empty index:** `MemoryIndexManager.search()` calls `hasIndexedContent()` and only awaits `sync({ reason: "search", force: true })` when no indexed content exists. With existing chunks/FTS rows, it proceeds to query setup instead of waiting for a dirty or reindex-needed state to settle.
