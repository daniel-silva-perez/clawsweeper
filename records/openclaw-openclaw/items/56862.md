## Summary

Keep open. Current main still lets builtin `memory_search` query an existing populated index while dirty/reindex sync is only started in the background. The existing CLI status and force-reindex command are useful diagnostics/workarounds, but they do not implement the requested query-path guard. Related PR #59137 is still open and, by its own scope, does not change query behavior.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/memory-core`: before builtin search queries an existing index, detect dirty state, pending search/session sync, or metadata that requires a full reindex, then either await the sync/reindex or return an explicit transient indexing status instead of an empty result. Add focused regression coverage for a populated SQLite/FTS index with dirty or full-reindex-needed state so manual `openclaw memory index --force` is no longer required to recover search results.

## Evidence

- **Search only blocks when no indexed content exists:** `MemoryIndexManager.search()` checks `hasIndexedContent()` and only awaits `sync({ reason: "search", force: true })` when the index has no chunks/FTS rows. A stale but populated index therefore bypasses the synchronous bootstrap and continues toward query execution.
