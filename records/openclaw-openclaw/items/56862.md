## Summary

Keep open. Current main still only blocks builtin memory_search for an empty index; a populated but dirty or full-reindex-needed builtin index can still query immediately while search-triggered sync runs in the background. The CLI status and force-reindex path remain useful workarounds, and related PR #59137 is adjacent recovery work, but the requested query-path guard is not implemented.

## Best Possible Solution

Keep this issue open. The best fix belongs in extensions/memory-core: before builtin memory_search queries an existing index, detect dirty state, pending session/search sync, or metadata requiring a full reindex, then await the sync/reindex or return an explicit transient indexing status instead of an empty result. Add focused regression coverage for a populated SQLite/FTS index with dirty or full-reindex-needed state so manual openclaw memory index --force is no longer required to recover results.
