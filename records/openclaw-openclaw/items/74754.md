## Summary

Keep this issue open. Current main still has a credible Homebrew auto-update failure path, and an open unmerged PR already references this issue with closing syntax, so the issue should stay open until that PR or an alternate maintainer-approved fix lands.

## What This Changes

The issue asks for Homebrew-installed gateway auto-update to install a detected newer stable release after the configured delay and restart, instead of only recording update-check state.

## Best Possible Solution

Keep the issue open and resolve it through #74870 or a maintainer-approved alternate fix that either safely handles Homebrew-managed installs out of process or clearly documents/logs that Homebrew auto-apply is unsupported while preserving update detection.

## Reproduction Assessment

Yes, there is a high-confidence code-level reproduction path: run the gateway auto-update path from a service-marked environment after the stable due time, and the spawned `openclaw update` child inherits gateway service markers before the package updater refuses to run. A live Homebrew repro would still be useful to confirm the exact user-visible logs and jitter timing.

## Solution Assessment

Unclear whether the proposed direction is the best final product behavior. Skipping Homebrew auto-apply with a clear hint is a narrow safe fix for the silent failure, but it does not provide the auto-install behavior the issue expected, so maintainer policy should decide before closing.
