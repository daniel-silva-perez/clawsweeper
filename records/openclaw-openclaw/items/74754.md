## Summary

Keep this issue open. Current main still has a credible auto-update failure path: the gateway schedules auto-update checks and launches `openclaw update` from the gateway process, the child inherits gateway service markers, and the package updater refuses to run before it reaches the service-stop/restart path. The Homebrew-specific part also needs maintainer judgment because the current global package update model is npm/pnpm/bun-oriented rather than a documented `brew upgrade openclaw` flow.

## What This Changes

The reporter wants gateway auto-update to install a newer stable version after the configured delay on a macOS Homebrew install, instead of only recording availability in `update-check.json`.

## Best Possible Solution

Keep the issue open and decide the supported install boundary, then either add a safe out-of-process auto-apply flow that can stop and restart the managed gateway without inheriting service-mode blockers, or explicitly record/log that auto-apply is unsupported for Homebrew formula installs. Align docs and tests with the chosen behavior.
