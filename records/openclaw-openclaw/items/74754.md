## Summary

Keep this issue open. Current main still has a credible Homebrew/package auto-update failure path, and open PR #74870 already references this issue with closing syntax, so the safe maintainer outcome is to review or replace that implementation rather than close cleanup early.

## What This Changes

The issue requests that Homebrew-installed gateways with auto-update enabled actually apply a detected stable update after the configured delay instead of remaining on the old version.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or detect Homebrew-managed installs and skip with a clear `brew upgrade openclaw` hint plus aligned docs and tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the auto-update child inherits gateway service markers, then `openclaw update` refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a plausible narrow fix if maintainers choose skip-and-hint for Homebrew, but it does not implement the originally expected auto-install behavior.
