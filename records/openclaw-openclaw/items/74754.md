## Summary

Keep open. Current main still has a credible Homebrew/package auto-update failure path, and open PR #74870 already references this issue with closing syntax, so the right next step is review or replacement of that implementation rather than cleanup closure.

## What This Changes

The issue requests that Homebrew-installed gateways with auto-update enabled apply newer stable releases after the configured delay instead of only recording update availability.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or detect Homebrew-managed installs and skip with a clear brew upgrade openclaw hint, with aligned docs and regression tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the auto-update child inherits gateway service markers, then openclaw update refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a narrow plausible fix if maintainers choose skip-and-hint for Homebrew, but it does not implement the originally expected auto-install behavior, so product review is still needed.
