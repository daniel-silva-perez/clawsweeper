Keep open. Current main still has a credible package-managed gateway auto-update failure path, and open PR #74870 already references this issue with closing syntax, so the right action is maintainer review of that PR or a replacement product decision rather than cleanup closure.

## What This Changes

The issue asks for Homebrew/package-managed gateway auto-updates to install a detected stable release after the configured delay and restart, instead of only recording update-check.json state.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or detect Homebrew-managed installs and skip with a clear brew upgrade openclaw hint plus aligned docs and tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the auto-update child inherits gateway service markers, then openclaw update refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a narrow plausible fix if maintainers choose skip-and-hint for Homebrew, but it does not implement the originally expected auto-install behavior, so product review is still needed.
