## Summary

Keep open. The underlying successful fallback auth-profile persistence path is still missing on current main, but this PR is not safe to merge as-is because it is stale against the current agent-command structure, replaces the command re-export with an inlined implementation, and drops or bypasses the existing model-override authorization guard.

## What This Changes

The PR propagates a final auth profile id from embedded fallback runs and persists that winning profile back to the session auth override, with focused command tests, while also replacing `src/commands/agent.ts` with an inlined agent command implementation.

## Best Possible Solution

Apply the auth-profile persistence fix narrowly to the current modular implementation: return the final successful profile from the embedded runner, persist it in the current session-store update path while preserving a user-pinned source, and keep `src/commands/agent.ts` as the re-export entrypoint with the existing `allowModelOverride` guard intact.

## Reproduction Assessment

Yes. Source inspection on current main shows the successful embedded run path marks `lastProfileId` good/used but never exposes it as `finalAuthProfileId`, and the session-store updater has no path to write a successful rotated profile back to `authProfileOverride`.

## Solution Assessment

No. The intended fix direction is sound, but this branch solves it in the wrong place for current main by copying a stale command implementation and weakening the ingress model-override authorization contract.
