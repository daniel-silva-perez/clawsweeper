## Summary

Keep open: current main still rejects legacy `dreaming.storage` string values through memory-core manifest validation, and the available doctor repair path would quarantine the `memory-core` plugin config instead of preserving the user’s selected storage mode.

## What This Changes

The issue reports that upgrading with legacy `plugins.entries.memory-core.config.dreaming.storage` string values such as `"both"` breaks CLI config validation and needs a migration or reachable repair path.

## Best Possible Solution

Add a focused memory-core doctor/compat migration that recognizes legacy `inline`, `separate`, and `both` string values at `plugins.entries.memory-core.config.dreaming.storage`, rewrites them to the canonical object shape, keeps the manifest strict, and adds regression coverage for migration order before generic plugin quarantine.

## Reproduction Assessment

Yes. A config containing `plugins.entries.memory-core.config.dreaming.storage: "both"` has a high-confidence source-level reproduction path: the manifest requires an object, plugin validation emits an `invalid config` issue, and no current migration converts the string before quarantine.

## Solution Assessment

Yes. The best fix is a narrow memory-core legacy migration plus sequencing coverage; loosening plugin schema validation globally or relying on generic invalid-plugin quarantine would be broader and would not preserve the user’s selected storage mode.
