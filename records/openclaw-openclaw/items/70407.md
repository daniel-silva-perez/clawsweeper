## Summary

Keep open: current main still rejects the legacy `dreaming.storage` string shape and the existing repair path would quarantine the `memory-core` plugin config instead of preserving the user's selected storage mode.

## What This Changes

The issue requests an upgrade-safe migration or recovery hint for legacy `plugins.entries.memory-core.config.dreaming.storage` string values after the `memory-core` schema moved to an object shape.

## Best Possible Solution

Add a focused memory-core doctor compatibility migration that recognizes legacy `inline`, `separate`, and `both` string values and rewrites them to the canonical object while preserving the selected mode; keep the manifest strict and cover the migration before generic invalid-plugin quarantine can delete the config.

## Reproduction Assessment

Yes. A config containing `plugins.entries.memory-core.config.dreaming.storage: "both"` has a high-confidence source-level reproduction path: the manifest requires an object, plugin validation emits an `invalid config` issue, and no current migration converts the string before quarantine.

## Solution Assessment

Yes. The narrowest maintainable fix is a targeted doctor/legacy migration for this memory-core field, plus regression coverage; loosening plugin schema validation globally would be broader and less safe.
