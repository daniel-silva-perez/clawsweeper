## Summary

Keep open. Current main still has the reported crash class: the Gateway fatal fallback exits for unclassified unhandled rejections, while the browser plugin's registered handler only consumes the Playwright dialog-race shape and does not classify `Assertion error` rejections with `CRSession._onMessage` / `CRConnection._onMessage` stack markers.

## What This Changes

The issue requests that Playwright CDP session assertion failures from browser control be isolated so they do not crash the Gateway process or unrelated agent sessions.

## Best Possible Solution

Add a browser-owned stability boundary for Playwright CDP session assertions: classify direct and wrapped `Assertion error` rejections with `CRSession._onMessage` or `CRConnection._onMessage` stack markers, log/degrade browser control, reset cached Playwright CDP state where feasible, and keep unknown unhandled rejections fatal.

## Reproduction Assessment

Yes for a focused unit-style reproduction: synthesize an unhandled rejection with `Assertion error` and `CRSession._onMessage` / `CRConnection._onMessage` stack markers while the browser runtime handler is registered, then verify current main leaves it unhandled. A live Chrome trigger is less deterministic because it depends on receiving an unexpected CDP message.

## Solution Assessment

Yes. The best near-term fix is a narrow browser-plugin classifier and recovery path using the existing `registerUnhandledRejectionHandler` seam; broad subprocess isolation can remain a larger architecture discussion.
