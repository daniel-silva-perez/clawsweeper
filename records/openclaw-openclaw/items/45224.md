## Summary

Keep open. Current main still has the reported crash class: the browser plugin only suppresses the recently merged Playwright dialog-race rejection, while a `CRSession._onMessage` / `CRConnection._onMessage` `Assertion error` remains unclassified and will fall through the Gateway's fatal unhandled-rejection path.

## What This Changes

The issue asks OpenClaw to prevent Playwright CDP `CRSession._onMessage` assertion rejections from crashing the whole Gateway process.

## Best Possible Solution

Add a browser-owned stability boundary for the reported Playwright CDP session assertion: classify direct and wrapped `CRSession._onMessage` / `CRConnection._onMessage` assertion rejections, log them, evict or reset the affected browser connection when possible, and keep the core fatal fallback for unknown unhandled rejections.

## Reproduction Assessment

Yes for a focused unit-style reproduction: synthesize an unhandled rejection with `Assertion error` and CRSession/CRConnection stack markers while the browser runtime handler is registered, then verify current main leaves it unhandled. A live Chrome trigger is less deterministic because it depends on receiving an unexpected CDP message.

## Solution Assessment

Yes, the best near-term fix is a narrow browser-plugin classifier and recovery path using the existing runtime unhandled-rejection seam. A global `Assertion error` catch-all or blanket Playwright suppression would be less maintainable and could mask unrelated failures.
