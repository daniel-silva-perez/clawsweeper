## Summary

Keep open. Current main still has the reported crash class: Playwright CDP session assertion rejections that are not matched by a registered browser handler fall through the Gateway's global unhandled-rejection path and exit the process. The merged dialog-race fix is useful adjacent work, but it only covers `Page.handleJavaScriptDialog` / `No dialog is showing`, not `CRSession._onMessage` assertions.

## What This Changes

The issue asks OpenClaw to prevent Playwright CDP `CRSession._onMessage` assertion failures from crashing the entire Gateway process.

## Best Possible Solution

Add a browser-owned stability boundary for the narrowly identified Playwright CDP session assertion: classify direct and wrapped `CRSession._onMessage` / `CRConnection._onMessage` assertion rejections, log them, evict or reset the affected browser connection when appropriate, and preserve core's fatal fallback for unknown unhandled rejections.

## Reproduction Assessment

Yes for a focused unit-style reproduction: synthesize an unhandled rejection with the reported `Assertion error` plus CRSession/CRConnection stack markers while the browser runtime handler is registered, and verify current main leaves it unhandled. A live Chrome trigger is less deterministic because it depends on receiving an unexpected CDP message.

## Solution Assessment

Yes, the best near-term fix is a narrow browser-plugin classifier and recovery path, following the lifecycle pattern from the dialog-race fix. A broad global `Assertion error` suppression or generic nonfatal Playwright bucket would be less maintainable and could hide real corruption.
