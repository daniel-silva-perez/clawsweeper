## Summary

Keep open. Current main fixes the narrow pre-compaction memory-flush prompt leak, but it still lacks a consistent visible-system-message display contract across gateway history projection, Control UI rendering, and TUI history replay.

## What This Changes

The issue asks for internal/system-originated chat messages to render with a distinct UI treatment instead of looking like normal user or assistant messages.

## Best Possible Solution

Define a structural display contract for intentionally visible runtime/system messages, carry it through chat.history and session projection, render it with a dedicated Control UI system treatment, and make TUI live/history display use its system notice path while continuing to suppress runtime-only artifacts that should never be user-visible.

## Reproduction Assessment

Yes. A source-level reproduction is available on current main by projecting or rendering a visible role system or internal_system-provenance row: it is not mapped to a dedicated system display block in Control UI, and TUI history replay ignores ordinary system rows.

## Solution Assessment

Unclear for an automated fix lane. Distinct rendering is the right direction for intentionally visible runtime messages, but maintainers should first resolve the hide-versus-label policy with the open chat.history filtering work.
