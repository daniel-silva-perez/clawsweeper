## Summary

Keep open. The PR is aimed at a real daily-boundary inconsistency and prior review threads appear to have been addressed, but the current branch should not merge as-is because it regresses current-main session freshness semantics and has a timezone helper bug for spring-forward gaps in positive-offset zones.

## What This Changes

The branch adds a shared human-day helper and wires userTimezone-aware day/reset keys through session reset, session memory, usage cost buckets, memory flush paths, WhatsApp session snapshots, and targeted tests.

## Best Possible Solution

Keep the shared human-day direction, but rebase onto current main, thread sessionStartedAt and lastInteractionAt through the new timezone-aware freshness logic, fix DST-gap fallback selection for positive-offset zones, add coverage for that case, and update the session-memory docs before merge.

## Reproduction Assessment

Yes. The PR body provides concrete Asia/Shanghai repros for the intended fix, current main has executable tests for lifecycle timestamp semantics, and the DST fallback defect is reproducible with the PR helper algorithm using Australia/Sydney's 2026 spring-forward gap.

## Solution Assessment

No. The shared helper is a reasonable direction, but this branch is not the best mergeable solution until it preserves current-main lifecycle timestamp semantics and fixes the positive-offset DST fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve lifecycle timestamps in freshness checks:** `src/config/sessions/reset-policy.ts:72-76`
  - body: Current main passes sessionStartedAt and lastInteractionAt into evaluateSessionFreshness so daily reset is based on session start and idle reset is based on the last real interaction. This branch's signature and callers only use updatedAt, so route, system, or metadata updates that refresh updatedAt can keep stale sessions alive and will regress the existing session freshness tests.
  - confidence: 0.9
- **[P2] Avoid using wall-clock UTC as the DST fallback candidate:** `src/infra/format-time/human-day.ts:161-171`
  - body: For spring-forward gaps in positive-offset zones such as Australia/Sydney, wallClockMs is later than all real UTC candidates. Initializing latestCandidateMs to guessMs and then taking Math.max returns the fake wall-clock UTC time instead of the first real instant after the skipped local reset hour, making dailyResetAt hours late or future for those users.
  - confidence: 0.86
