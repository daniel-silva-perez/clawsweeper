## Summary

Keep this issue open. Current main still compares the raw existing config byte length against the canonical two-space JSON output, so a BOM/PowerShell-verbose `openclaw.json` can still look like a destructive size drop even when the parsed config is preserved. The `models auth login` flow still writes through that guarded config path, and the only linked open PR in the provided context is a maintainer tracker, not an implementation fix.

## Best Possible Solution

Implement a narrow config-writer fix that compares the shrink guard against a canonicalized previous config baseline, or otherwise proves that the shrink is formatting-only, while preserving rejection for real semantic drops such as missing `gateway.mode` or large key loss. Add a BOM plus verbose-indentation regression test and keep auth-login success reporting tied to a committed config update.
