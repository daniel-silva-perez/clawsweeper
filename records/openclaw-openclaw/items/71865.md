## Summary

Keep open. Current main still uses the raw existing config byte size as the size-drop baseline while writing canonical 2-space JSON, so a semantically equivalent PowerShell/BOM/verbose-format config can still be rejected before auth config updates are committed.

## What This Changes

The issue asks OpenClaw to let `models auth login` persist OAuth config updates when the existing `openclaw.json` was written in a PowerShell/BOM verbose format that shrinks substantially under canonical JSON formatting.

## Best Possible Solution

Update the config write safety check so formatting-only shrink is evaluated against a canonicalized previous config or a semantic diff, while keeping destructive clobber rejection intact. Add a regression test with a UTF-8 BOM plus PowerShell-style verbose formatting and keep auth-login completion messaging tied to a committed config update.

## Reproduction Assessment

Yes. The source path is high-confidence: the reported 59,662 to 19,829 byte change satisfies the current `< 50%` guard, and `models auth login` still flows through that guarded writer.

## Solution Assessment

Yes, a narrow config-writer fix is the best direction. An auth-specific bypass would leave the same false-positive guard for other OpenClaw-owned config writes and would risk weakening the clobber protection unevenly.

## Repair Work Prompt

Fix the config write size-drop false positive reported in https://github.com/openclaw/openclaw/issues/71865. Current main compares canonical 2-space JSON output against raw `snapshot.raw` bytes, so a PowerShell-created UTF-8 BOM / verbose-format `openclaw.json` can be rejected as `size-drop` even when the semantic config is preserved. Keep the fix in the config writer/safety layer rather than adding an auth-specific bypass. Preserve fail-closed rejection for real destructive writes such as missing `gateway.mode` or large key loss, and keep `.rejected.*` behavior for true clobbers. Add a regression test that starts from a semantically valid config with BOM plus verbose formatting, performs an auth/profile-style config update, and verifies the active config is committed. Add or adjust auth command coverage if needed so success logging happens only after config update commits. Include a user-facing changelog fix entry credited to `Thanks @balric-seo`. Do not change dependency patches, release metadata, or broad migration policy.
