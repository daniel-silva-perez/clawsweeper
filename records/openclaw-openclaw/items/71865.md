## Summary

Keep this issue open. Current main still bases the config write size-drop guard on raw file bytes versus canonical 2-space JSON output, so a BOM/verbose-indentation `openclaw.json` can still look like a destructive shrink even when the content is not being clobbered. The exact silent-success symptom is less certain on current main because the write path now throws/logs `CONFIG_WRITE_REJECTED`, but the core blocked-write condition remains present and lacks a PowerShell/BOM regression test.

## Best Possible Solution

Implement a narrow config-writer fix that compares size-drop risk against a canonicalized previous config baseline, or otherwise proves the shrink is formatting-only, while preserving rejection for real semantic drops. Add a PowerShell-style BOM/verbose-indentation regression test and keep auth-login success reporting tied to a committed config update.
