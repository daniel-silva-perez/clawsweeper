## Summary

Keep open. Current main still compares canonical two-space JSON output against the raw previous config byte size and blocks `size-drop` writes, while `models auth login` still reaches that shared writer through `updateConfig`; the latest release contains the same behavior, and related #70941 was closed in favor of this issue as the canonical tracker.

## What This Changes

The issue requests that auth login persist config changes when `openclaw.json` was previously written by PowerShell with BOM or verbose formatting, instead of being blocked by the config writer’s raw byte-size drop guard.

## Best Possible Solution

Fix the shared config writer so formatting-only shrink is judged against a canonicalized previous config or semantic diff while preserving fail-closed protection for real clobbers, then add regression coverage for BOM/verbose PowerShell formatting through the auth-login path and related config mutations.

## Reproduction Assessment

Yes. The source path is high-confidence: a valid raw config more than twice the canonical JSON size trips `size-drop`, and `models auth login` reaches that guarded writer through `updateConfig` and `replaceConfigFile`.

## Solution Assessment

Yes. A shared config-writer fix is the narrowest maintainable solution; an auth-only `allowDestructiveWrite` bypass would leave the same false positive in other config-changing commands and weaken protection unevenly.

## Repair Work Prompt

Fix the config write false positive tracked by https://github.com/openclaw/openclaw/issues/71865. Current main compares canonical two-space JSON output against the previous raw `openclaw.json` byte size, so a valid PowerShell/BOM/verbose-formatted config can shrink below the 50% threshold and be rejected as `CONFIG_WRITE_REJECTED` during `openclaw models auth login --provider openai-codex`. Keep the fix in the shared config writer (`src/config/io.ts`) rather than adding an auth-only destructive-write bypass. Compare the next write against a canonicalized previous config or semantic diff before treating shrink as destructive, while preserving blocking behavior for real clobbers such as dropped gateway mode or genuinely destructive shrink. Add focused regression tests for UTF-8 BOM plus verbose/PowerShell-style formatting in `src/config/io.write-config.test.ts`, and cover the auth-login path or shared `updateConfig` path enough to prove the login config update commits instead of leaving only a `.rejected.*` artifact. Related #70941 was closed in favor of this canonical issue; include a models-set/shared-writer case if it fits the same test seam. Add a user-facing changelog fix entry crediting `@balric-seo` and, if the models-set regression is covered, `@dy18501155150-blip`. Do not weaken the clobber guard globally, do not set `allowDestructiveWrite` for normal auth writes, and do not refactor unrelated config migration or plugin metadata code.
