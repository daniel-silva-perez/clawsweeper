## Summary

Keep open. Current main fixes the Telegram forum-topic `threadId` pass-through and adds inter-session provenance marking, but the visible-delivery fallback after `ANNOUNCE_SKIP` is still absent and requester-vs-target A2A announce routing remains an open policy/fix area covered by #57697 and #49223.

## What This Changes

The report asks `sessions_send` between Telegram forum topics to preserve target topic delivery, avoid losing visible handoffs when announce returns `ANNOUNCE_SKIP`, and prevent target replies from entering the requester transcript as fresh inter-session turns.

## Best Possible Solution

Keep the shipped `threadId` and provenance fixes, then land a coordinated A2A delivery policy that defines requester-vs-target announce routing, when a non-visible target reply may be reused after skip tokens, and how peer handoffs avoid transcript pollution across Telegram and other channels.

## Reproduction Assessment

Yes. The issue has a high-confidence code-level reproduction on current main: `runSessionsSendA2AFlow` suppresses the final gateway send when announce returns `ANNOUNCE_SKIP`, and `src/agents/openclaw-tools.sessions.test.ts` asserts zero visible sends in that path.

## Solution Assessment

No. Current main is a partial fix: it solves `threadId` pass-through and provenance marking, but the remaining requester routing and skip-token fallback need a maintainable cross-channel policy rather than closing this as fully implemented.
