## Summary

Keep open. Current main has the Telegram `threadId` pass-through and inter-session provenance mitigations, but the visible-delivery fallback after `ANNOUNCE_SKIP` and the broader requester-side A2A handoff policy remain unresolved and overlap open #57697 and #49223.

## What This Changes

The issue asks for `sessions_send` between Telegram forum-topic sessions to preserve target topic delivery, keep visible handoffs from disappearing when announce returns `ANNOUNCE_SKIP`, and avoid injecting topic-to-topic handoff turns back into the requester transcript.

## Best Possible Solution

Keep the shipped threadId and provenance fixes, then land a coordinated A2A delivery policy that defines requester-vs-target announce routing and when visible delivery may fall back after skip tokens, with Telegram forum coverage plus channel-agnostic regression tests.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction: `runSessionsSendA2AFlow` suppresses external send when announce returns `ANNOUNCE_SKIP`, and `src/agents/openclaw-tools.sessions.test.ts` asserts that no visible send occurs in that path; the peer A2A requester loop is also directly visible in the same flow.

## Solution Assessment

No. Current main is a partial fix, not the final solution; the maintainable path is to reconcile the requester routing work in #57697 with the skip-token delivery policy in #49223 before closing this issue.
