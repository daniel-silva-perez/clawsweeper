## Summary

Keep open. Current main has shipped the Telegram threadId pass-through and inter-session provenance mitigations, but the remaining visible-delivery and requester-side A2A handoff behavior is still present in the code and overlaps open #57697 and #49223.

## What This Changes

The issue requests that `sessions_send` preserve Telegram forum topic delivery, keep visible handoffs from disappearing after `ANNOUNCE_SKIP`, and avoid injecting topic-to-topic handoff traffic into the requester transcript.

## Best Possible Solution

Keep the shipped threadId and provenance fixes, then land one coordinated A2A policy that defines when visible delivery should fall back after skip tokens and when requester-side peer handoff turns should be suppressed, with Telegram forum coverage plus channel-agnostic regression tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction remains: current main suppresses the external send when announce returns `ANNOUNCE_SKIP`, and the existing sessions_send test asserts that no visible send occurs in that path; the peer A2A loop into the requester session is also directly visible in `runSessionsSendA2AFlow`.

## Solution Assessment

No. Current main is a partial fix, not the final solution; the maintainable path is to reconcile the requester routing work in #57697 with the skip-token policy in #49223 before closing this issue.
