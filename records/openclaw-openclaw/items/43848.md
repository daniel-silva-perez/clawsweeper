## Summary

Keep this open. Current main already preserves Telegram forum topic `threadId` for the visible announce send, but the remaining reported behavior is still present: `ANNOUNCE_SKIP` still suppresses the visible delivery even after a real target reply, and the A2A ping-pong path can still run the target reply back through the requester session as a fresh `inter_session` turn. The open related PR #57697 is unmerged and only addresses requester-target announce routing, while the earlier full-fix PRs #43869 and #44064 are closed unmerged.

## What This Changes

The issue asks OpenClaw to preserve Telegram forum topic delivery for `sessions_send`, avoid dropping visible topic handoffs when the announce step returns `ANNOUNCE_SKIP`, and prevent topic-to-topic A2A traffic from polluting the requester transcript.

## Best Possible Solution

Keep the shipped `threadId` preservation, review or land a corrected version of #57697 for requester-side announce routing if maintainers agree with that direction, and add focused coverage plus code for the two missing behaviors: visible Telegram topic fallback after `ANNOUNCE_SKIP` when a real target reply exists, and suppression or redesign of same-agent Telegram topic-to-topic ping-pong before it records new requester-side `inter_session` turns.
