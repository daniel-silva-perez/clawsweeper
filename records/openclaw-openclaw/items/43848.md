## Summary

Keep open. Current main has shipped the Telegram forum-topic threadId and inter-session provenance mitigations, but it still suppresses visible delivery when the announce step returns ANNOUNCE_SKIP and still runs normal peer A2A reply steps through the requester session; the remaining work overlaps open #57697 and open #49223.

## What This Changes

The issue requests that sessions_send preserve Telegram forum thread delivery, avoid dropping visible target-topic replies when announce returns ANNOUNCE_SKIP, and prevent topic-to-topic inter_session traffic from contaminating requester transcripts.

## Best Possible Solution

Keep the shipped threadId and provenance mitigations, then define a coordinated A2A delivery policy that decides when skip tokens suppress external delivery, when a real target reply should be delivered visibly, and when same-agent topic-to-topic handoffs should avoid requester-side injected turns.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: current main only sends when announceReply is not ANNOUNCE_SKIP, and the existing test path returns ANNOUNCE_SKIP and asserts no visible send; the remaining requester-side ping-pong path is also directly visible in source and tests.

## Solution Assessment

No. Current main is only a partial fix; the narrowest maintainable next step is to coordinate #57697 with the skip-token policy in #49223 instead of closing this issue as implemented.
