## Summary

Keep this issue open. Current main still treats `/new` as a reset-style session rollover on chat/gateway/UI paths, with only the TUI showing a narrower unique-session behavior. The requested cross-surface contract, where `/new` preserves the old session as a separately accessible session and `/reset` is the destructive reset path, is not implemented.

## Best Possible Solution

Keep this issue open for explicit product and implementation follow-up. The best path is to define the desired cross-surface contract, then update the chat/session store, gateway reset service, Control UI, ACP-bound behavior if applicable, docs, and regression tests so `/new` creates or switches to a new accessible session without destructively resetting the old one, while `/reset` remains the reset path.
