## Summary

Keep this issue open. Current main still treats `/new` as reset-style rollover across the auto-reply, gateway, Control UI, and docs paths; only the TUI has a narrower unique-session key behavior. The related PR #49595 was closed unmerged, and related #73450 covers a different `/new` empty-provider-request failure mode rather than preserving the old session as an accessible session.

## Best Possible Solution

Keep this issue open for explicit product and implementation follow-up. The best path is to define the cross-surface contract, then implement `/new` as a non-destructive create/switch flow, likely using or extending `sessions.create` plus session selection/history UI, while leaving `/reset` as the destructive reset path. Update gateway/chat handling, Control UI, ACP/thread binding behavior if applicable, docs, and regression tests together.
