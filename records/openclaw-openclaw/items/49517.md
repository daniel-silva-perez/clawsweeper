## Summary

Keep this issue open. Current main does not provide the requested general contract that `/new` preserves the old session as a separately accessible session while `/reset` is the destructive reset path. Several current surfaces still intentionally treat `/new` as a reset-style rollover or in-place reset, even though the TUI has a narrower unique-session behavior.

## Best Possible Solution

Keep this issue open for maintainer product judgment and implementation. The likely path is to decide whether `/new` should become a non-destructive session-spawn/switch command across chat/gateway/UI surfaces, then update `src/auto-reply/reply/session.ts`, `src/auto-reply/reply/commands-reset.ts`, gateway agent handling, Control UI command handling, docs, and regression tests so `/reset` remains the only destructive reset path.
