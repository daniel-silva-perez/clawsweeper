## Summary

Keep open. Current main still treats `agents.defaults.models` keys as canonical provider/model refs: suffixes are preserved during parsing, allowlist keys are built from the preserved model id, params are looked up by exact `provider/modelId`, and provider payloads send `model.id`. Related provider-prefix stripping does not cover this variant-key request, and naive `:` stripping would conflict with valid colon-bearing model IDs.

## Best Possible Solution

Keep open. Implement an explicit model-variant contract instead of arbitrary `:` stripping: let a unique `agents.defaults.models` entry carry a canonical target provider/model while retaining its own alias and params, then plumb the selected config key through `/model`, WebChat/session selection, extra-param resolution, and provider dispatch. Add regression tests for duplicate variants and for valid colon-bearing real model ids.

## Risks / Open Questions

- A naive delimiter-strip fix could break valid provider model IDs that use `:`, including OpenRouter `:free`, Ollama tags, and `hf:` model references.
- The implementation needs to preserve the selected config entry separately from the provider model id; otherwise variant-specific `params` may be dropped.
