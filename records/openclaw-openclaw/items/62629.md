## Summary

Keep open. Current main still treats configured model keys as exact provider/model refs, so a suffixed key remains a distinct model id through selection and runtime resolution; however, the proposed colon delimiter needs maintainer design because colons are already valid inside real model ids.

## What This Changes

Requests support for multiple `agents.defaults.models` entries that share one provider model by treating a custom key suffix as UI/config-only and sending the unsuffixed base model to the provider.

## Best Possible Solution

Add a documented, unambiguous model-variant mechanism rather than blind delimiter stripping, for example an explicit base/provider model field on an allowlist entry or another syntax that cannot collide with provider model ids. The selected variant should keep its alias and params for `/model` and webchat, while runtime resolution sends the configured base model id to the provider.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure `agents.defaults.models` with `openai-codex/gpt-5.3-codex:med`, select that entry, and the current parser and runner paths keep `gpt-5.3-codex:med` as the model id instead of resolving it to `gpt-5.3-codex`.

## Solution Assessment

Unclear as proposed. The desired outcome is valid, but stripping everything after `:` is not the safest solution because `:` is part of real model ids; an explicit variant/base-model contract is the safer maintainable direction.
