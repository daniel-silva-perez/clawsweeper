## Summary

Keep this PR open. Current main still lacks an interactive custom-provider metadata prompt, so the contribution remains relevant, but the branch is not ready to merge because the new lookup can miss provider-prefixed native model IDs and the Auto API choice cannot clear an existing model-level adapter override.

## What This Changes

The PR adds a custom-provider-only `openclaw configure --section model` wizard step plus helper tests for editing configured model metadata such as API adapter, reasoning, input modalities, context window, max output tokens, and token pricing.

## Best Possible Solution

Rebase onto current main, keep the prompt limited to custom provider setup, use shared model-ref parsing/canonicalization so native model IDs containing `/` are found, make Auto remove model-level API overrides back to provider inheritance, account for the current `contextWindow` versus `contextTokens` split, and add focused tests for prefixed model IDs, API clearing, and the current custom-provider metadata defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve provider-prefixed model ids before lookup:** `src/commands/model-metadata.ts:77-80`
  - body: `findPrimaryModelEntry` splits the primary key and searches for only the suffix. Current `modelKey` preserves native IDs that already start with the provider prefix, so a custom provider entry such as provider `openrouter` with model ID `openrouter/foo` gets primary key `openrouter/foo`; this code searches for model ID `foo` and silently skips the metadata prompt.
  - confidence: 0.87
- **[P2] Clear api when Auto is selected:** `src/commands/model-metadata.ts:131-132`
  - body: Selecting `Auto (inherit from provider)` leaves `result.api` unset, and the later merge spreads `...entry.model` before `...updated`, so any existing model-level `api` survives. Users can set an explicit adapter but cannot return to provider inheritance through this wizard.
  - confidence: 0.86
