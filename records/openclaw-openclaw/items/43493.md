## Summary

Keep this PR open. Current main still lacks an interactive metadata step for custom provider setup, so the contribution is not obsolete, but the branch is not merge-ready because the new lookup misses provider-prefixed native model IDs and the API adapter prompt cannot clear an existing model-level override.

## What This Changes

The PR adds a custom-provider-only configure wizard step and helper tests for editing configured model metadata such as API adapter, reasoning, input modalities, context window, max output tokens, and token pricing.

## Best Possible Solution

Rebase onto current main, keep the prompt limited to custom provider setup, use shared model-ref helpers for canonical/native model IDs, let Auto remove model-level API overrides back to provider inheritance, and add focused tests for prefixed model IDs, API clearing, and current contextWindow/contextTokens behavior. If maintainers want a broader first-run UX, supersede the branch through #48338 rather than losing the custom-provider metadata problem.

## Review Findings

  - body: `findPrimaryModelEntry` splits the primary key and looks up only the suffix. Current `modelKey` preserves native model IDs that already start with the provider prefix, so a custom provider entry like provider `openrouter` with model ID `openrouter/foo` gets primary key `openrouter/foo`; this code searches for model ID `foo` and silently skips the metadata prompt.
  - confidence: 0.87

  - confidence: 0.86
