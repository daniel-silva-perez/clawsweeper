## Summary

Keep this PR open. Current main still lacks an interactive metadata prompt for custom-provider setup, but this branch is stale against the current configure flow and still has merge-blocking correctness issues in the new metadata helper.

## What This Changes

The PR adds a custom-provider model metadata prompt to `openclaw configure --section model`, a new helper module for editing configured model definitions, and unit tests for the prompt/update flow.

## Best Possible Solution

Rebase the PR onto current main, integrate the prompt after the current custom-provider setup path, reuse the shared model-ref/modelKey helpers, preserve the full model metadata contract, define Auto API inheritance so it can clear overrides or relabel the option, and add focused tests for provider-prefixed ids, media modalities, and API override clearing.

## Reproduction Assessment

Yes. The current-main gap is reproducible by source inspection of the custom-provider configure path, and the PR defects are reproducible with small unit scenarios for provider-prefixed model ids, an existing model-level API override, and existing audio/video input modalities.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best mergeable solution yet because it reimplements model-ref parsing, narrows valid input modalities, and exposes provider API inheritance without clearing existing model-level overrides.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle provider-prefixed primary model IDs:** `src/commands/model-metadata.ts:77-80`
  - body: `findPrimaryModelEntry` splits the primary model key and searches only for the suffix. Current model refs can preserve native ids that already start with the provider prefix, so a custom entry like provider `openrouter` with model id `openrouter/foo` stores the primary as `openrouter/foo`, but this code searches for `foo` and silently skips the metadata prompt.
  - confidence: 0.87
- **[P2] Clear API overrides when Auto is selected:** `src/commands/model-metadata.ts:131-132`
  - body: Selecting `Auto (inherit from provider)` leaves `result.api` unset, and the later merge spreads the existing model before the update, so any existing model-level `api` value survives. The wizard offers provider inheritance but cannot actually remove a previous per-model adapter override.
  - confidence: 0.86
- **[P2] Preserve audio and video input modalities:** `src/commands/model-metadata.ts:140-149`
  - body: `ModelDefinitionConfig.input` accepts `video` and `audio`, but the prompt only offers `text` and `image` and then overwrites `entry.model.input` with the result. Editing a media-capable custom model can drop valid existing modalities when the user only meant to change token metadata.
  - confidence: 0.78
