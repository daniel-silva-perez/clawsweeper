## Summary

Keep this PR open. Current main still lacks the interactive custom-provider metadata prompt, the related custom-provider context UX issue remains open, and the patch is a plausible implementation path, but it needs a rebase plus fixes for model-ref lookup and API override clearing before it should merge.

## What This Changes

The PR adds a custom-provider-only configure wizard prompt for editing model metadata such as display name, API adapter, reasoning support, input modalities, context limits, max output tokens, and optional token costs.

## Best Possible Solution

Rebase the PR onto current main, keep the metadata prompt limited to the custom-provider setup path, reuse current model-ref parsing/resolution helpers, allow users to clear model-level API overrides back to provider inheritance, and add focused configure/model-metadata tests covering provider-prefixed native model IDs, API inheritance, contextWindow/maxTokens, contextTokens expectations, and the skip path. If maintainers prefer a different UX, supersede this with an equivalent fix for #48338 rather than dropping the user problem.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve prefixed model refs before lookup:** `src/commands/model-metadata.ts:77-80`
  - body: On current main, `modelKey` intentionally stores native IDs that already start with the provider prefix without duplicating it, for example `openrouter/hunter-alpha`. Splitting the primary ref on the first slash makes this lookup search for model ID `hunter-alpha`, so the metadata prompt silently skips configured models whose stored ID is `openrouter/hunter-alpha`. Use the existing model-ref parsing/resolution helpers or match against collected model keys.
  - confidence: 0.87
- **[P2] Let Auto clear model API overrides:** `src/commands/model-metadata.ts:131-133`
  - body: Selecting `Auto (inherit from provider)` leaves `result.api` unset, and the later `{ ...entry.model, ...updated }` merge preserves any existing per-model `api`. That means a user can set an explicit model adapter but cannot use the new wizard to return to provider-level inheritance, despite the option label saying it will inherit.
  - confidence: 0.86
