## Summary

Keep open. Current main does not implement the proposed portable `openclaw migrate export` / `openclaw migrate import <archive>` workflow: the existing `migrate` command is provider-based import from other agent systems, and `backup` only creates/verifies archives. The PR remains a product/security-sensitive restore/import design question, reinforced by maintainer discussion about restore attack surface and possible narrower `backup` target/convert alternatives.

## What This Changes

Adds `openclaw migrate export` and `openclaw migrate import` subcommands for portable cross-device OpenClaw state migration, including archive creation/import, selective components, path remapping, secret stripping, CLI registration, and tests.

## Best Possible Solution

Keep this PR open for maintainer product/security review. Maintainers should choose the intended scope first: either rework into narrower backup export/convert improvements, or design a carefully gated restore/import flow with archive safety helpers, preview/conflict preflight, explicit trust prompts, backup/rollback behavior, and focused tests. Current main does not already solve the PR's central request.
