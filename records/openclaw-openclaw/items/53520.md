## Summary

Keep open. Current main does not implement this archive export/import workflow, and the PR is broad, stale against the newer provider-based `openclaw migrate` surface, and security-sensitive. The discussion already has maintainer concern about automated restore risk, and the latest branch still has concrete import/export safety issues that should block merge until product scope and archive safety are resolved.

## What This Changes

The PR branch adds archive-based `openclaw migrate export` and `openclaw migrate import` commands for config, credentials, workspace, and session migration, with CLI registration, archive planning/import/export helpers, and tests.

## Best Possible Solution

Pick the recovery model first. If core should stay conservative, evolve `backup` with safer export targeting, conversion, and manual restore docs. If OpenClaw should own automated import, integrate it with the existing provider-based migrate namespace and require preview-first UX, strong target canonicalization, archive validation, rollback/conflict handling, and security-focused tests before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Canonicalize targets before destructive writes:** `src/infra/migrate-import.ts:328-333`
  - body: `assertTargetWithinRoot` only compares `path.resolve` strings, so an existing symlinked parent such as `~/.openclaw/agents` can pass validation while the later `fs.rm`/copy operates outside the OpenClaw state tree. Resolve existing target ancestors or reject symlinked target parents before any destructive import write.
  - confidence: 0.89
- **[P2] Do not export default workspace for unknown agents:** `src/commands/migrate-shared.ts:177-180`
  - body: When `--agents` contains an ID that is not in `agents.list`, this branch still adds the default workspace because the requested agent lacks an explicit workspace. That makes `openclaw migrate export --agents typo --include workspace` include unrelated workspace data despite matching no agent.
  - confidence: 0.88
- **[P2] Preserve existing migrate provider commands:** `src/cli/program/register.migrate.ts:45-58`
  - body: Current main already uses `openclaw migrate` for plugin-owned provider imports with `list`, `plan`, and `apply`. This registration defines the command as only archive `export`/`import`, so after rebase it would either conflict with or drop the existing Claude/Hermes migration surface.
  - confidence: 0.82
