## Summary

Keep open. Current main does not implement the PR's archive-based `openclaw migrate export` / `openclaw migrate import` workflow: `migrate` is still a plugin-provider import surface, `backup` still only creates/verifies archives, and machine-to-machine migration is documented as manual copy/extract. The PR also remains a product/security decision because the discussion explicitly questioned whether automated restore/import should exist at all.

## What This Changes

The PR branch adds archive-based `openclaw migrate export` and `openclaw migrate import` commands with selective components, secret stripping, path remapping, config merge, CLI registration, archive import/export infrastructure, and tests.

## Best Possible Solution

Choose the intended recovery model first. If core should stay conservative, rework this toward smaller `backup` export/target/convert improvements plus clear manual restore docs. If OpenClaw should own automated import, design it as an explicitly trusted, preview-first restore flow with strong archive validation, rollback/backups, conflict handling, and security-focused tests before merge.
