## Summary

Current main still does not implement the requested global/per-agent hard iteration budget. The schema and public docs lack `maxTurns`, `maxToolCalls`, `maxToolCallsPerTurn`, `maxToolCallsPerSession`, and `toolCallTimeout` for agent runs; existing loop detection is optional and pattern-based; and OpenResponses `max_tool_calls` is explicitly accepted-but-ignored. Keep this issue open as the canonical global/per-agent agent iteration limit request, separate from the open per-channel budget request #13271.

## Best Possible Solution

Keep this issue open as the canonical global/per-agent hard iteration budget request. A complete implementation should add typed strict config under `agents.defaults` and per-agent scope, enforce model-tool turn and total tool-call counters in the embedded/tool-call runtime before executing the next tool, define timeout-after-first-tool-call semantics, document precedence with existing loop detection/timeouts, and add focused tests for exhaustion and final-response behavior. Keep #13271 separate for per-channel budgets and auto-delegation.

## Risks / Open Questions

- A future fix must define precedence between a hard budget, `tools.loopDetection`, whole-run `agents.defaults.timeoutSeconds`, provider request timeouts, and ignored OpenResponses `max_tool_calls`.
- The exhaustion behavior is product-sensitive: block the next tool call, inject a model-visible warning, return the last useful tool result, or force a final response all have different channel and safety implications.
- Schema, generated help/labels, docs, and focused embedded-runner/tool-call tests need to land together so users can rely on the limit instead of prompt-only instructions.
