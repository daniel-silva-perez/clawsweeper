## What This Changes

Request a redacted doctor/status/gateway-status diagnostic when a non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` shadows configured `gateway.auth.token` in local gateway mode.

## Best Possible Solution

Keep this issue as the canonical report and resolve it by reviewing PR #74433 or a replacement that adds one shared, redacted diagnostic: parse the state-dir `.env`, compare non-empty `OPENCLAW_GATEWAY_TOKEN` with configured local `gateway.auth.token`, explain env precedence, and suggest removing or syncing the env key plus restarting the gateway without printing secrets.

## Reproduction Assessment

Yes. A high-confidence reproduction path is local gateway mode with `gateway.auth.token` set in `~/.openclaw/openclaw.json` and a different non-empty `OPENCLAW_GATEWAY_TOKEN` in `~/.openclaw/.env`; current main’s dotenv loading and credential precedence explain the observed auth disagreement.

## Solution Assessment

Yes for the requested direction: a warning is safer than deleting or rewriting a possibly intentional env credential. The maintainable implementation should use a shared helper that reads the state-dir `.env` source directly, rather than relying only on `process.env`, so doctor/status surfaces report the same redacted remediation.
