## What This Changes

Request to add redacted doctor/status diagnostics when a state-directory `OPENCLAW_GATEWAY_TOKEN` shadows a configured local `gateway.auth.token`.

## Best Possible Solution

Keep this issue as the canonical report and resolve it by reviewing PR #74433 or a replacement that adds one shared, redacted diagnostic: parse the state-dir `.env`, compare non-empty `OPENCLAW_GATEWAY_TOKEN` with configured local `gateway.auth.token`, explain env precedence, and suggest removing or syncing the env key plus restarting the gateway without printing secrets.

## Reproduction Assessment

Yes. A high-confidence reproduction path is local gateway mode with `gateway.auth.token` set in `~/.openclaw/openclaw.json` and a different non-empty `OPENCLAW_GATEWAY_TOKEN` in `~/.openclaw/.env`; current main’s dotenv loading and credential precedence coverage explain the observed auth disagreement.

## Solution Assessment

Yes for the requested direction. A redacted warning is safer than deleting or rewriting a possibly intentional env credential, and the maintainable implementation should share one helper across doctor/status-style surfaces rather than changing credential precedence as a side effect.
