## Summary

Keep open. Current main still has the local env-first credential precedence that can shadow `gateway.auth.token`, and the requested state-dir `.env` warning is not present in doctor/status/gateway status. The issue also has an open closing PR (#74433), so it should stay open until that PR is reviewed and either landed or replaced.

## What This Changes

Add a diagnostic warning when local gateway mode has `gateway.auth.token` configured but `OPENCLAW_GATEWAY_TOKEN` is also present in `~/.openclaw/.env` and may override or diverge from the config token.

## Best Possible Solution

Keep this issue open as the canonical user report and resolve it by reviewing PR #74433 or a replacement that adds a shared, redacted diagnostic for the state-dir env-token shadowing case across the relevant doctor/status/gateway-status surfaces.

## Reproduction Assessment

Yes. A high-confidence reproduction path is local gateway mode with `gateway.auth.token` in `~/.openclaw/openclaw.json` and a different non-empty `OPENCLAW_GATEWAY_TOKEN` in `~/.openclaw/.env`; current main’s credential precedence and dotenv loading explain why CLI/probe paths can disagree with the gateway service.

Unclear for the open PR, because the issue requests doctor/status/gateway-status coverage and state-dir `.env` source reporting, while the provided PR summary focuses on `noteSecurityWarnings()` and `security audit --deep`.
