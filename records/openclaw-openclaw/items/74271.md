## Summary

Keep open. Current main still has the reported split between env-first local CLI credential resolution and config-first gateway service token resolution, while doctor/status/gateway-status do not yet emit the requested state-dir `.env` shadowing warning. The linked PR #74433 is an open implementation candidate with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Request a redacted doctor/status/gateway-status warning when local `gateway.auth.token` is shadowed by a non-empty `OPENCLAW_GATEWAY_TOKEN` from the state-dir `.env`.

## Best Possible Solution

Land or revise the linked implementation so a shared, redacted diagnostic reads the trusted state-dir `.env`, checks only local gateway mode with configured `gateway.auth.token`, and surfaces consistent remediation in doctor/status/gateway-status without printing secret values.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is local gateway mode with configured `gateway.auth.token` plus non-empty state-dir `OPENCLAW_GATEWAY_TOKEN`; current tests and code show env-first CLI paths and config-first gateway service paths. No live command was run because this review is read-only.

## Solution Assessment

Yes, the requested warning is the maintainable direction: it preserves explicit env-based auth while making stale durable env credentials visible. Automatically deleting or rewriting the token would be riskier because env auth can be intentional.
