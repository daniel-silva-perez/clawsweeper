## Summary

Keep open. Current main still has the env-first local CLI credential path and state-dir dotenv loading, but the inspected doctor, status, and gateway-status surfaces do not yet warn when a state-dir `OPENCLAW_GATEWAY_TOKEN` shadows a configured `gateway.auth.token`. PR #74433 is open and uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

Warn local gateway users in doctor/status/gateway diagnostics when a non-empty `OPENCLAW_GATEWAY_TOKEN` in the state-dir `.env` overrides a configured `gateway.auth.token`.

## Best Possible Solution

Ship a shared, redacted diagnostic that reads the trusted state-dir `.env`, only triggers in local gateway mode when `gateway.auth.token` is configured and state-dir `OPENCLAW_GATEWAY_TOKEN` is non-empty, and surfaces consistent remediation in doctor, status, and gateway status/deep output without deleting user credentials automatically.

## Reproduction Assessment

Yes. The report gives a concrete source-level reproduction state: local gateway mode, `gateway.auth.token` in config, and non-empty `OPENCLAW_GATEWAY_TOKEN` in the state-dir `.env`; current code inspection confirms that state-dir dotenv loading plus env-first local CLI auth makes the mismatch plausible on main.

## Solution Assessment

Yes, the warning-based direction is the best way to solve this. It preserves intentional env-based auth while explaining precedence, the source path, secret-safe remediation, and the need to restart the gateway; automatic deletion or migration would be riskier.
