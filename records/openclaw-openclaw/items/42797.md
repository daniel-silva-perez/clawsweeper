## Summary

Keep this PR open. Current main still does not implement the PR's central behavior: default gateway `models.list` is not narrowed to authenticated or otherwise available providers when no explicit allowlist is configured. Main has gained a `view: "configured"` path and some UI callers use it, but that is configured-model filtering, not auth/availability filtering; TUI and macOS gateway chat still call `models.list` without params and receive the default catalog behavior. The PR also remains unsafe to land as-is because review discussion correctly flagged missing Bedrock AWS SDK/default-chain auth and synthetic local-provider auth handling.

## Best Possible Solution

Keep this PR open for maintainer review, but require changes before merge. The implementation should use `hasAvailableAuthForProvider` or an equivalent shared gateway-safe provider-availability helper, preserve the full-catalog fallback for truly unconfigured installs, and add focused gateway/TUI/UI regression coverage for env keys, auth profiles, config keys, Bedrock AWS SDK/default-chain auth, and synthetic local-provider auth. If a broader canonical implementation such as #59693 lands first and covers this behavior, close this PR as superseded then.

## Risks / Open Questions

- Landing the submitted helper as-is could silently hide valid Amazon Bedrock models for users relying on AWS SDK credentials, AWS_PROFILE, or the default credential chain.
- Landing the submitted helper as-is could hide usable Ollama/local/LAN custom providers that rely on synthetic local auth rather than an explicit API key.
- The PR-local auth heuristic may drift from runtime credential resolution and profile eligibility semantics, including expired or invalid auth profiles.
