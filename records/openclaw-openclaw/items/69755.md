## Summary

Keep open. Current main still builds and appends a plain `Run: openclaw doctor --non-interactive` restart-sentinel hint for config/update restart paths, and the delivery path can surface that message through channel or heartbeat contexts where approval-capable execution may not be available. The related open PR #64684 is adjacent but not a complete resolution because it adds doctor summaries while leaving the actionability wording problem in place.

## What This Changes

Request that post-restart config/update messages distinguish `openclaw doctor --non-interactive` as a terminal or approval-capable follow-up instead of always presenting it as immediately actionable in the current heartbeat/channel context.

## Best Possible Solution

Keep the diagnostic follow-up, but make the restart-sentinel message surface-aware enough that channel/heartbeat notices do not imply the current context can execute the command. A narrow fix should update the shared hint wording or payload rendering to point users to a terminal or approval-capable Control UI, without weakening exec approvals or building a broad autonomous recovery loop.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: config/update restart flows set the doctor hint, `formatRestartSentinelMessage()` appends it, and `scheduleRestartSentinelWake()` can deliver that text through system events, heartbeat wake, and outbound channel notices; docs confirm headless/no-UI approval paths may not complete prompts.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution if implemented as restart-sentinel UX wording or metadata. Auto-running doctor everywhere or relaxing approvals would be broader and riskier; #64684 is adjacent but does not by itself fix the misleading `Run:` follow-up.
