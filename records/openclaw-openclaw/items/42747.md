## Summary

Keep open. Current main does not implement Slack OAuth token rotation: the Slack channel schema and docs expose static bot/app/user/signing-secret credentials, account resolution reads those from config or env once, and runtime passes the resolved bot/app tokens into Bolt without an oauth.v2.access refresh path. Because Slack is a shipped OpenClaw channel and the request is a security/major-channel capability, this should be implemented in the Slack plugin rather than closed as ClawHub cleanup.

## Best Possible Solution

Implement the feature in the existing Slack plugin: add typed/schema-backed tokenRotation config with sensitive fields and UI hints, use Slack oauth.v2.access to refresh before expiry, persist rotated bot/refresh/expires data through the existing config or credential-store mutation path, invalidate Slack clients or restart/reconnect the Slack account safely, expose status/doctor visibility, and document/test both Socket Mode and HTTP mode. Only add a generic core/plugin SDK seam if the Slack plugin cannot safely persist rotated sensitive credentials with current public APIs.

## Risks / Open Questions

- Implementation will touch sensitive Slack credentials and config mutation, so it needs careful redaction, SecretRef/credential-store handling, atomic persistence, and tests for both Socket Mode and HTTP mode.
- A naive in-process token swap may leave cached WebClient/Bolt clients using stale credentials; the design should explicitly handle client cache invalidation, reconnect/restart behavior, and status/doctor reporting.
