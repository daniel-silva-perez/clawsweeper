## Summary

Keep open. Current main still only supports static Slack bot/app/user/signing-secret credentials; it has no Slack token-rotation config, no oauth.v2.access refresh path, no setup/status command for refresh credentials, and expired tokens are still classified as non-recoverable auth failures rather than refreshed.

## What This Changes

Add native Slack token rotation support using Slack oauth.v2.access refresh-token flow, including config/schema fields, proactive refresh, credential persistence, setup/status tooling, and Socket Mode plus HTTP mode coverage.

## Best Possible Solution

Implement this in the bundled Slack plugin, not in core: add schema-backed sensitive rotation credentials or credential-store entries, refresh with oauth.v2.access before expiry, persist the new bot/refresh/expires values safely, reconnect or rebuild Slack clients when credentials change, expose non-secret status/doctor signals, and document/test Socket Mode and HTTP mode. Add a generic plugin SDK persistence seam only if the Slack plugin cannot safely update rotated sensitive credentials through existing public APIs.

## Reproduction Assessment

Not applicable as a feature request, but code inspection gives a high-confidence absence check: current main rejects a tokenRotation object through the strict Slack schema and the runtime only resolves static tokens from config/env.

## Solution Assessment

Yes in direction, but the exact config shape should be reviewed. Native support belongs in the existing Slack plugin because Slack is a bundled channel and the request concerns secure operation of that shipped channel; the safest implementation may use the credential store or SecretRefs rather than writing all refreshed secrets directly into openclaw.json.
