## Summary

Keep open. Current main still does not implement native Slack token rotation: Slack config/schema/docs only expose static bot/app/user/signing-secret credentials, runtime resolves those from config/env once, Bolt is constructed with fixed tokens, and `token_expired` is treated as a non-recoverable auth error rather than a refresh trigger. Because Slack is a bundled channel and the request is a security-relevant capability for that shipped channel, this should remain an OpenClaw Slack plugin follow-up rather than ClawHub cleanup.

## Best Possible Solution

Implement this in the bundled Slack plugin: add typed/schema-backed `tokenRotation` config with sensitive fields, bootstrap/status CLI or channel setup support, Slack `oauth.v2.access` refresh before expiry, durable persistence of rotated bot/refresh/expires values through the existing config or credential-store mutation path, safe Slack client/Bolt reconnect behavior, status/doctor visibility, docs, and tests for Socket Mode and HTTP mode. Add a generic plugin SDK seam only if the Slack plugin cannot safely persist rotated credentials through current public APIs.

## Risks / Open Questions

- Implementation will touch sensitive Slack credentials and config or credential-store mutation, so it needs explicit redaction, SecretRef handling, atomic persistence, and rollback/error behavior.
- A naive token refresh could leave cached WebClient/Bolt clients using stale credentials; the design should specify client invalidation, reconnect/restart behavior, and status/doctor reporting for both Socket Mode and HTTP mode.
