## Summary

Keep this PR open, but do not merge it as-is. The force-pushed fallback value is type-valid, but the one-line change would make unset Telegram streaming final-only while the resolver tests, Telegram docs, and config UI hints still define `partial` as the default; this is a product-visible default change that needs maintainer direction and aligned public surfaces.

## What This Changes

The PR changes Telegram preview streaming's unset fallback in `extensions/telegram/src/preview-streaming.ts` from `partial` to `off`.

## Best Possible Solution

Maintainers should choose the Telegram default explicitly: keep `partial` if the shipped in-place preview finalization is the intended answer, or accept final-only-by-default behavior by changing the fallback to `off` together with resolver tests, Telegram docs, config UI hints, and a user-facing changelog entry.

## Reproduction Assessment

Yes for the PR impact: current source and tests show unset Telegram streaming resolves to `partial`, while the PR diff changes that fallback to `off`. The linked ghost-message report has detailed external reproduction, but I did not perform a live Telegram verification during this read-only review.

## Solution Assessment

No as submitted. `off` is a valid resolver value, but changing only the fallback leaves the tested and documented contract contradictory; the maintainable fix needs either a product-approved default flip across all public surfaces or a narrower partial-mode delivery fix.
