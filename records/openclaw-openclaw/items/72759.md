## Summary

Keep this PR open, but do not merge it as-is. The force-pushed `off` fallback is type-valid, but it changes a documented Telegram product default while leaving the focused resolver test, Telegram docs, and config UI hints aligned to `partial`; maintainers still need to decide whether the default should change or whether current partial-mode finalization should remain the default behavior.

## What This Changes

The PR changes Telegram preview streaming's unset fallback in `extensions/telegram/src/preview-streaming.ts` from `partial` to `off`.

## Best Possible Solution

Maintainers should choose the desired Telegram default explicitly: keep `partial` as the documented default if current in-place preview finalization is the intended fix, or accept final-only-by-default behavior by changing the fallback to `off` together with resolver tests, Telegram docs, config UI hints, and a user-facing changelog entry.

## Reproduction Assessment

Yes for the PR impact: current source and tests show unset Telegram streaming resolves to `partial`, while the PR diff changes that fallback to `off`. The original ghost-message bug has detailed external reproduction in the linked issue, but I did not perform a live Telegram verification during this read-only review.

## Solution Assessment

No as submitted. `off` is a valid resolver value, but changing only the fallback leaves the tested and documented contract contradictory; the maintainable fix needs either a product-approved default flip with all public surfaces updated or a narrower partial-mode delivery fix.
