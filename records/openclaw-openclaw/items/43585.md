## Summary

Keep open. Current main still lacks deterministic `$include` NUL-byte rejection and raw/resolved include-path length validation, and this PR remains a focused security-hardening candidate rather than obsolete or implemented work. It should not be merged as-is until the open review feedback is handled.

## Best Possible Solution

Keep this PR open as the implementation candidate. Revise it to reject NUL bytes before path resolution, enforce raw and normalized/resolved include-path limits with `Buffer.byteLength(..., "utf8")`, make the exclusive fewer-than-4096-byte boundary explicit in errors and docs, add regression tests for NUL, raw limit, resolved limit, boundary acceptance, and multibyte byte-limit cases, then run the targeted config include test and changed gate before maintainer merge.

## Risks / Open Questions

- Merging the PR as-is would enforce a filesystem-style limit with UTF-16 code units rather than bytes, so multibyte include paths could still exceed the intended byte limit without deterministic validation.
- The current main behavior still lets malformed `$include` paths reach later Node/fs behavior instead of failing early with a stable include-path validation error.
- No supply-chain or CI permission expansion was observed in the PR diff; the remaining risk is correctness and completeness of the security hardening.
