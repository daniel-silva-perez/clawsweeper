## Summary

Keep open. The PR is a plausible narrow security hardening change, but current main does not already implement it and the proposed patch has a blocking correctness/security gap: its PATH_MAX-style cap uses JavaScript string length instead of UTF-8 byte length, so multibyte paths can exceed the intended filesystem byte boundary.

## What This Changes

The PR adds explicit `$include` path validation for NUL bytes and raw/resolved paths at or above 4096 string characters, with matching config include tests and gateway configuration docs.

## Best Possible Solution

Land a revised narrow hardening change that rejects NUL bytes before path resolution, enforces an explicit fewer-than-4096 UTF-8 byte limit on both raw and normalized/resolved include paths, and keeps docs/tests aligned around raw limit, resolved limit, near-boundary acceptance, and multibyte byte-limit cases.

## Reproduction Assessment

Not applicable as a hardening PR rather than a user bug report. The review has a high-confidence source/diff check plus a byte-length probe showing why the proposed `.length` cap does not enforce the documented filesystem-style byte boundary.

## Solution Assessment

No. The direction is right, but the implementation should measure UTF-8 bytes with `Buffer.byteLength(..., "utf8")` for both raw and resolved paths and make the exclusive boundary clear in errors, docs, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-207`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. A multibyte UTF-8 path can stay below 4096 code units while exceeding the PATH_MAX-style byte boundary the PR documents, so the hardening still falls back to filesystem behavior instead of deterministic validation. Use UTF-8 byte length for both checks and add a multibyte boundary test.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:199-209`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 is allowed. After switching to byte length, phrase the message and docs as an exclusive fewer-than-4096-byte bound.
  - confidence: 0.82
