## Summary

Keep open. Current main still lacks the proposed `$include` NUL-byte and path-length validation, so the PR is not obsolete, but the proposed hardening should not merge as-is because its PATH_MAX-style limit is measured with JavaScript string length rather than UTF-8 filesystem bytes.

## What This Changes

The PR adds NUL-byte rejection and raw/resolved 4096-character length checks for config `$include` paths, plus config include tests and gateway docs updates.

## Best Possible Solution

Revise this PR, or a narrow replacement, so `$include` validation rejects NUL bytes before path resolution, enforces an explicit fewer-than-4096 UTF-8 byte limit on both raw and normalized/resolved paths, and aligns docs/tests around raw limit, resolved limit, near-boundary acceptance, and multibyte byte-limit cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-213`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. A path of 2048 CJK characters is 6144 UTF-8 bytes, so it can pass the 4096 check while exceeding the PATH_MAX-style filesystem boundary the PR is trying to enforce; use `Buffer.byteLength(..., "utf8")` for both checks and cover a multibyte case.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:200-213`
  - body: The guards reject `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 is allowed. After the byte-limit fix, phrase the message and docs as an exclusive fewer-than-4096-byte bound.
  - confidence: 0.82
