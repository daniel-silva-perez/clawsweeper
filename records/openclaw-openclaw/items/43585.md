## Summary

Keep open. Current main and the latest release still lack deterministic NUL-byte and include-path length validation, and this PR is useful hardening work, but the proposed path cap is character-counted while the stated PATH_MAX-style contract is byte-based, so the patch should be revised before merge.

## What This Changes

The PR adds NUL-byte rejection and raw/resolved 4096-character `$include` path checks in config include resolution, with related tests and gateway docs updates.

## Best Possible Solution

Revise this PR, or a narrow replacement, so `$include` path validation rejects NUL bytes before resolution, enforces an explicit fewer-than-4096 UTF-8 byte limit on both raw and normalized/resolved paths, and aligns docs/tests around raw limit, resolved limit, near-boundary acceptance, and multibyte byte-limit cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-213`
  - body: The PR frames the 4096 cap as PATH_MAX-style hardening, but the new guards use `.length` on the raw and normalized strings. That counts UTF-16 code units, so multibyte UTF-8 paths can exceed 4096 filesystem bytes while still passing validation. Please use `Buffer.byteLength(..., "utf8")` for both checks and add a multibyte regression case.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:200-213`
  - body: The checks reject `>= MAX_INCLUDE_PATH_LENGTH`, but the error text says `maximum length (4096 characters)`, which reads as if 4096 is allowed. Please phrase the error as an exclusive bound, ideally matching the byte-based contract after the limit check is corrected.
  - confidence: 0.82
