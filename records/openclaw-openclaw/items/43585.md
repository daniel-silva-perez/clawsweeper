## Summary

Keep open. The PR is a useful narrow hardening direction, but the proposed patch is not correct as-is because its PATH_MAX-style limit is enforced with JavaScript string length, so multibyte paths can exceed the intended filesystem byte boundary while passing validation. Current main also still lacks the explicit NUL and length contract, so the work is not obsolete.

## What This Changes

The PR adds config `$include` guards for null bytes and paths at or over 4096 string characters, updates include tests, and documents the new constraints in gateway configuration docs.

## Best Possible Solution

Land a revised narrow hardening patch that rejects NUL bytes before path resolution, enforces a clearly documented fewer-than-4096 UTF-8 byte limit on both raw and normalized/resolved include paths, and adds aligned tests for raw limit, resolved limit, near-boundary acceptance, and multibyte byte-limit cases.

## Reproduction Assessment

Not applicable as a security-hardening PR rather than a reported runtime bug. The review has a high-confidence source/diff check plus a byte-length probe that reproduces the proposed guard’s mismatch with the documented filesystem-style limit.

## Solution Assessment

No. The direction is right, but the implementation should use UTF-8 byte length for both raw and resolved path checks and make the exclusive boundary clear in errors, docs, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-207`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. A multibyte UTF-8 path can stay below 4096 code units while exceeding the PATH_MAX-style byte boundary the PR documents, so the hardening still falls back to filesystem behavior instead of deterministic validation. Use UTF-8 byte length for both checks and add a multibyte boundary test.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:199-209`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 is allowed. After switching to byte length, phrase the message and docs as an exclusive fewer-than-4096-byte bound.
  - confidence: 0.82
