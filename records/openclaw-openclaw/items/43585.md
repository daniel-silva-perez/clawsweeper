## Summary

Keep open. Current main still lacks deterministic `$include` NUL-byte rejection and raw/resolved include-path length validation, and this PR is a focused implementation candidate, but it is not merge-ready because the proposed PATH_MAX-style guard counts UTF-16 string units instead of UTF-8 filesystem bytes.

## What This Changes

The PR adds NUL-byte rejection and raw/resolved 4096-character `$include` path checks in config include resolution, with related tests and gateway docs updates.

## Best Possible Solution

Revise this PR, or a narrow replacement, so `$include` path validation is centralized before resolution, rejects NUL bytes, enforces an explicit fewer-than-4096 UTF-8 byte limit on both raw and normalized/resolved paths, and aligns docs/tests around raw limit, resolved limit, near-boundary acceptance, and multibyte byte-limit cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:198-213`
  - body: The PR documents this as PATH_MAX-style hardening, but the new guards use `string.length` for both raw and normalized paths. That counts UTF-16 code units, so multibyte paths can exceed 4096 filesystem bytes while still passing the guard. Please use `Buffer.byteLength(..., "utf8")` for both checks and add a multibyte regression case.
  - confidence: 0.93
- **[P3] Clarify the exclusive length boundary:** `src/config/includes.ts:200-213`
  - body: The checks reject `>= MAX_INCLUDE_PATH_LENGTH`, but the error text says `maximum length (4096 characters)`, which reads as if 4096 is allowed. Please phrase the error as an exclusive bound, ideally matching the byte-based contract after the length check is corrected.
  - confidence: 0.82
