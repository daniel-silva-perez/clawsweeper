## Summary

Keep open. Current main still lacks the explicit `$include` NUL-byte and path-length guards, and this PR is useful security hardening, but the proposed length check is not merge-ready because it enforces JavaScript string length rather than the byte-based PATH_MAX-style limit described by the PR and docs.

## What This Changes

The PR adds `ConfigIncludeError` validation for `$include` paths containing NUL bytes or raw/resolved paths at or above 4096 characters, updates include tests, and documents the new gateway config include path restrictions.

## Best Possible Solution

Land a revised narrow hardening change that rejects NUL bytes before path resolution, enforces an explicit fewer-than-4096 UTF-8 byte limit on both raw and normalized/resolved include paths, and keeps tests/docs aligned around the raw limit, resolved limit, near-boundary acceptance, and multibyte byte-limit cases.

## Reproduction Assessment

Not applicable as a hardening PR rather than a user bug report. The review has a high-confidence source/diff check: current main lacks the new guards, and a byte-length probe demonstrates why the PR's `.length`-based path cap does not enforce the documented filesystem-style byte boundary.

## Solution Assessment

No. The direction is right, but the implementation should measure UTF-8 bytes with `Buffer.byteLength(..., "utf8")` for both raw and resolved paths and make the exclusive boundary clear in errors, docs, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:195-207`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. A path made of multibyte characters can pass these checks while exceeding the 4096-byte PATH_MAX-style boundary the PR documents, so the hardening falls back to filesystem behavior instead of deterministic validation. Use UTF-8 byte length for both checks and add a multibyte boundary test.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:197-209`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the errors say `maximum length (4096 characters)`, which reads as if 4096 is allowed. After switching to byte length, phrase the message and docs as an exclusive fewer-than-4096-byte bound.
  - confidence: 0.82
