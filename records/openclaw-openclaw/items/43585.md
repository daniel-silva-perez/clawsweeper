## Summary

Keep open. Current main still does not implement the proposed NUL-byte rejection or include-path length contract, and the PR is a useful security hardening direction, but the current patch has a blocking correctness/security issue because its PATH_MAX-style cap is measured with JavaScript string length instead of UTF-8 byte length.

## What This Changes

The PR adds ConfigIncludeError guards for `$include` paths containing NUL bytes or raw/resolved lengths at or above 4096, updates include tests, and documents the new path restrictions in gateway configuration docs.

## Best Possible Solution

Revise this PR, or land a narrow replacement, so `$include` validation rejects NUL bytes before path resolution, enforces an explicit fewer-than-4096 UTF-8 byte limit on both raw and normalized/resolved paths, and aligns docs/tests around raw limit, resolved limit, near-boundary acceptance, and multibyte byte-limit cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Measure include path limits in bytes:** `src/config/includes.ts:195-207`
  - body: The new guards compare `includePath.length` and `normalized.length`, which count UTF-16 code units. A path made of multibyte characters can pass these checks while exceeding the 4096-byte PATH_MAX-style boundary the PR documents, leaving the hardening to fall back to filesystem behavior instead of a deterministic `ConfigIncludeError`. Use `Buffer.byteLength(..., "utf8")` for both checks and cover a multibyte case.
  - confidence: 0.93
- **[P3] Clarify the exclusive path length boundary:** `src/config/includes.ts:197-209`
  - body: The guards reject values at `>= MAX_INCLUDE_PATH_LENGTH`, but the error messages say `maximum length (4096 characters)`, which reads as though 4096 is allowed. After switching to byte length, phrase the error/docs as an exclusive fewer-than-4096-byte bound.
  - confidence: 0.82
