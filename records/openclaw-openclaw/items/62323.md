## Summary

Keep open. Current main still feeds all registry entries into the model catalog and `/model status` groups those entries by provider, so the OpenRouter-qualified model can still be accompanied by the native `google` row. This PR is a focused implementation candidate for the still-open linked bug, but its diff has unresolved functional review risks and should be reviewed/reworked rather than closed as cleanup.

## Best Possible Solution

Keep this PR open for maintainer review alongside the linked bug and the other open implementation attempt (#62481). The best fix belongs in the model catalog/status path and should filter only entries proven to be duplicate aggregator-prefix artifacts while preserving real native provider discovery, normalizing provider aliases consistently, and keeping defensive field coercion with regression tests for all three cases.

## Risks / Open Questions

- The proposed filter can hide legitimate direct-provider models when that provider is discovered from auth/env/profile state but is not explicitly listed in `cfg.models.providers`.
- The PR diff has a raw-vs-normalized provider-prefix mismatch for aliased provider IDs, so some phantom rows would still bypass the filter.
- The PR diff removes defensive string coercion before `.trim()`, which can make malformed registry entries abort catalog processing instead of being skipped.
