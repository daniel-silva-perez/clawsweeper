## Summary

Keep this PR open for maintainer follow-up, but do not merge it as-is. Current main still does not implement the PR's mutex-based appendMessage serialization, while the submitted implementation would change guarded appendMessage from a synchronous string-returning operation into a Promise-returning operation and can track pending tool calls before a message is actually accepted for persistence.

## Best Possible Solution

Do not merge this PR as-is. Maintainers should either close it manually as obsolete if the current idle-before-flush implementation fully covers the linked synthetic-result reports, or request a rework that preserves the synchronous SessionManager.appendMessage contract, keeps pending tracking tied to successful message acceptance, and adds a current-main regression test through a real async boundary or explicit async append adapter.

## Risks / Open Questions

- Merging the PR as submitted would make guarded appendMessage asynchronous at runtime while existing TypeScript-facing and runtime call sites still treat it as synchronous.
- Moving pending tracking before write acceptance can synthesize tool-result repairs for assistant tool calls that were blocked by hooks or failed before persistence.
- Current main's idle-before-flush mitigation may already cover the linked user-visible reports, but it does not prove or implement the PR's broader concurrent appendMessage serialization claim.
