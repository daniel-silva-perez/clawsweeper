## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement the submitted appendMessage mutex or its concurrent regression tests, while the PR as submitted changes the synchronous SessionManager.appendMessage runtime contract and moves pending tool-call tracking ahead of write acceptance. Current main has a narrower idle-before-flush mitigation for synthetic tool-result cleanup, but that is not the same as this PR's serialization change.

## Best Possible Solution

Do not auto-close or merge this PR as-is. Maintainers should either close it manually if the current idle-before-flush implementation fully covers the linked synthetic-result failures, or ask for a narrower rework that preserves the synchronous `SessionManager.appendMessage` contract, tracks pending tool calls only after accepted persistence, and adds a current-main regression test through a real async boundary or explicit async append adapter.

- Current main may already mitigate some linked synthetic-result failures through idle-before-flush cleanup, but it does not cover or test a future genuinely async append adapter; a rework needs a real async-boundary regression test.
