## Summary

Keep this PR open for maintainer follow-up. Current main still does not include the submitted append serialization, but the patch is not mergeable as-is because it changes guarded `appendMessage` from a synchronous, id-returning operation into a Promise-returning microtask and records pending tool calls before the message has passed the write hook or persisted.

## What This Changes

The PR adds a promise-chain mutex around `installSessionToolResultGuard`, moves pending tool-call tracking before append, and adds three concurrent append regression tests.

## Best Possible Solution

Do not merge this branch unchanged. The maintainable end state is either to leave the current idle-before-flush mitigation in place if it fully covers the linked failures, or rework the fix so guarded appends preserve the synchronous `SessionManager.appendMessage` contract, track pending tool calls only after accepted persistence, and add a regression test with a real async append boundary if that boundary is supported.

## Reproduction Assessment

Unclear. The historical synthetic-result failures are credible and current main has idle-before-flush coverage for one race, but this PR's exact concurrent `appendMessage` race is not reproduced with high confidence because the pinned upstream `SessionManager.appendMessage` is synchronous and the new tests mostly schedule synchronous appends in microtasks.

## Solution Assessment

No. The proposed mutex is not the best fix as submitted because it silently changes a synchronous persistence contract and can synthesize tool results for assistant messages that were blocked or failed to persist; a narrower rework should preserve current contracts and test the real async boundary before changing guard semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve synchronous appendMessage behavior:** `src/agents/session-tool-result-guard.ts:310-312`
  - body: `appendMutex.run(...)` always returns a Promise and defers the append body to a microtask, but the cast keeps TypeScript believing `appendMessage` still returns the synchronous entry id. Current `SessionManager.appendMessage` is declared and implemented as `string`, and current guard callers can read state, emit ids, or flush immediately after append; this patch lets that code run before the append actually happens.
  - confidence: 0.93
- **[P1] Track pending calls only after write acceptance:** `src/agents/session-tool-result-guard.ts:282`
  - body: Recording `pendingState.trackToolCalls(toolCalls)` before `applyBeforeWriteHook(...)` and `originalAppend(...)` leaves pending ids behind when an assistant message is blocked or the append fails. The next non-tool message can then synthesize tool results for tool calls that were never persisted, corrupting the transcript pairing the guard is meant to protect.
  - confidence: 0.91
