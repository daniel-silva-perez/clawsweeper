Keep this PR open for maintainer judgment. Current main does not implement the submitted appendMessage mutex/serialization change, and the submitted patch still has contract risks that make it unsafe to auto-close or merge as-is.

## Risks / Open Questions

- Merging the submitted patch as-is would make guarded `appendMessage` asynchronous at runtime while the pinned dependency and current OpenClaw wrapper treat it as a synchronous string-returning operation.
- Moving pending tool-call tracking before `beforeMessageWriteHook` and `originalAppend` can leave pending IDs for assistant messages that were blocked or failed to persist.
- Current main may already mitigate some linked synthetic-result failures through idle-before-flush cleanup, but it does not implement or test the PR's broader concurrent append serialization claim.
