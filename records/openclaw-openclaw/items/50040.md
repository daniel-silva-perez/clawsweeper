## Summary

Keep this issue open. Current main still has adjacent polling recovery, safe-send retry, startup delivery recovery, and shared outbound queue pieces, but interactive Telegram final replies still go through direct `deliverReplies` delivery and are not persisted as pending/retryable/ambiguous work that can drain after polling recovery.

## What This Changes

The issue requests durable runtime recovery for Telegram outbound replies when polling stalls or transient Bot API send failures occur, with safe retry classification and stateful delivery entries.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct delivery, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and triggers eligible recovery during polling restart as well as startup where the shared queue contract fits.

## Reproduction Assessment

Yes. The issue includes production timelines and current-main code has a static reproduction path: make a Telegram final reply send fail in `deliverReplies` and the direct path rethrows without durable pending/retryable/ambiguous state.

## Solution Assessment

Unclear. The requested direction is correct, but the exact implementation needs maintainer design judgment because non-idempotent Telegram sends cannot be blindly retried without duplicate-message risk.
