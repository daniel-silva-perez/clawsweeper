## Summary

Current main still does not fully cover the reported Telegram reply-loss path. The shared outbound delivery queue and recovery/drain helpers exist, and the standalone Telegram send helper has conservative retry logic for safe pre-connect errors, but inbound Telegram final replies still flow through `bot-message-dispatch` -> `deliverReplies` -> `sendTelegramText` -> direct `bot.api.sendMessage`. That path emits failure hooks and rethrows on send failure, with no persistence, failure-state classification, or Telegram polling-restart drain for already-computed final chunks.

## Best Possible Solution

Keep this open and design a Telegram reply-delivery recovery path for direct inbound final/block replies. A good fix should persist or otherwise safely recover already-computed reply chunks before direct `bot.api.sendMessage`, classify failures as safe-to-retry, ambiguous, or permanent, retry only safe cases with bounded backoff, preserve ambiguous failures without blind duplication, and trigger a drain after Telegram polling recovery as well as startup.
