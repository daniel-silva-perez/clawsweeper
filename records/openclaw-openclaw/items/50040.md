## Summary

Keep this issue open. Current main still has the central gap described by the report: interactive Telegram final replies are sent through the direct `deliverReplies` path, failures are rethrown after a failed hook emission, and the existing durable queue/runtime recovery machinery only covers `deliverOutboundPayloads` callers or restart-continuation paths, not already-computed Telegram reply chunks during polling recovery.

## What This Changes

The issue requests durable runtime Telegram outbound delivery recovery for polling-stall windows, including safe, ambiguous, and permanent failure classification plus stateful retry handling for already-computed replies.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct send, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and hooks eligible recovery into polling restart as well as startup where the shared queue contract fits.
