## Summary

Keep this open. Current main has a narrow Telegram safe-send retry fix and a shared crash-recovery delivery queue, but the reported runtime gap still remains for already-computed Telegram final replies: that path still delivers directly, rethrows send failures, and does not persist pending/ambiguous chunks for recovery after polling restarts.

## What This Changes

The issue requests stronger Telegram outbound delivery recovery so polling stalls do not silently lose already-computed replies, with safe retries, ambiguous-failure handling, and runtime recovery without requiring a gateway restart.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct send, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and hooks eligible recovery into polling restart as well as startup where the shared queue contract fits.
