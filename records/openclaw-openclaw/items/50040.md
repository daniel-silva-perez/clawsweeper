## Summary

Keep open. Current main still has adjacent fixes for Telegram polling recovery, bounded safe-send retry, and the shared outbound queue, but the interactive Telegram final-reply path still calls direct delivery and does not persist failed reply chunks as pending, retryable, or ambiguous runtime work.

## What This Changes

The issue requests durable runtime recovery for Telegram outbound replies when polling stalls or transient Bot API send failures occur, including safe retry classification and preservation of ambiguous/non-idempotent sends.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct delivery, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and triggers eligible recovery during polling restart as well as startup where the shared queue contract fits.
