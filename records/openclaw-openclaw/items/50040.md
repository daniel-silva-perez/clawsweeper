## Summary

Keep this issue open. Current main still has adjacent Telegram polling recovery, bounded safe-send retry, and shared outbound queue recovery, but the interactive Telegram final-reply path still calls direct `deliverReplies` delivery and does not persist reply chunks as pending/retryable/ambiguous work that can drain after polling recovery.

## What This Changes

The issue requests durable runtime recovery for Telegram outbound final replies when polling stalls or restarts coincide with transient `sendMessage` failures.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct delivery, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and triggers eligible recovery during polling restart as well as startup where the shared queue contract fits.

## Reproduction Assessment

Yes. The issue includes production timelines, and current-main code gives a static reproduction path: make a Telegram final reply `sendMessage` fail in `deliverReplies`, and the direct path rethrows without durable pending/retryable/ambiguous state.

## Solution Assessment

Unclear. The requested direction is correct, but the exact fix needs maintainer design judgment because non-idempotent Telegram sends cannot be blindly retried without duplicate-message risk.
