## Summary

Keep this open. Current main has the shared outbound queue and a narrower Telegram safe-send retry fix, but interactive Telegram final replies still send through the direct `deliverReplies` path, rethrow delivery failures, and do not create pending or ambiguous durable work that can drain after polling recovery.

## What This Changes

The issue requests runtime Telegram outbound delivery recovery so polling stalls or transient `sendMessage` failures do not silently lose already-computed replies.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct send, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and hooks eligible recovery into polling restart as well as startup where the shared queue contract fits.
