## Summary

Keep this issue open. Current main has the shared outbound crash-recovery queue and a narrower Telegram safe-send retry fix, but interactive Telegram final replies still go through the direct `deliverReplies` path, rethrow delivery failures, and are not persisted as pending/ambiguous work that can drain after polling recovery.

## What This Changes

The issue asks for stronger Telegram outbound delivery recovery so polling stalls or restarts do not silently lose already-computed replies, with safe retries, ambiguous-failure handling, and runtime recovery without requiring a gateway restart.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct send, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and hooks eligible recovery into polling restart as well as startup where the shared queue contract fits.
