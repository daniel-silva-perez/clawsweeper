## Summary

Keep open. Current main has useful partial hardening: strict safe retries for Telegram non-idempotent sends, a shared write-ahead outbound queue, and startup recovery. It still does not implement the central runtime recovery path described here: Telegram assistant reply chunks are sent through `deliverReplies`, failures are rethrown after failure hooks, and Telegram polling restart/recovery does not drain pending outbound deliveries. The newer related #53895 was closed as a duplicate of this older canonical tracker, not as a fix.

## Best Possible Solution

Keep this issue open as the canonical tracker. Implement durable Telegram assistant-reply delivery recovery by either routing the `deliverReplies`/dispatcher path through the shared outbound queue or adding a Telegram-owned queue that preserves chat, topic/thread, reply target, account, session, media, buttons, and delivery metadata. Classify failures as safe retry, ambiguous, or permanent; retry only safe/rate-limited entries with backoff; preserve ambiguous entries for operator-visible recovery; and trigger a drain after Telegram polling restart/recovery. Add regression coverage for a multi-chunk final reply where a later `sendMessage` fails after the assistant has already computed the answer.

## Risks / Open Questions

- A future fix must avoid blind retries for ambiguous non-idempotent Telegram sends, or it can duplicate messages that Telegram actually received before the transport failed.
- Some polling-stall hardening has already shipped, so the remaining work should stay scoped to durable outbound reply recovery during Telegram recovery windows rather than reopening solved polling watchdog behavior.
