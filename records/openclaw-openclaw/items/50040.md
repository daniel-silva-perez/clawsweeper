## Summary

Keep this open. Current main has narrowed the safest Telegram send retry case and has a shared crash-recovery outbound queue, but the specific runtime gap reported here remains for already-computed Telegram final replies: direct inbound reply delivery still calls `deliverReplies`, rethrows failed sends, and does not persist pending/ambiguous chunks or drain them after a polling restart.

## What This Changes

The issue requests durable runtime recovery for Telegram outbound `sendMessage` failures during polling stalls, including safe retry classification, preserved ambiguous failures, and recovery without a gateway restart.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that preserves already-computed final/block reply chunks before or during direct send, retries only failures classified as safe, holds or surfaces ambiguous failures without blind duplication, and hooks recovery into Telegram polling restart as well as startup where the shared queue contract fits.
