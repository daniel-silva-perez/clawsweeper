## Summary

Keep this open. Current main still has the reported runtime-delivery gap: interactive Telegram final replies are sent through the direct `deliverReplies` path, failed direct sends are rethrown, and the shared write-ahead delivery queue/recovery path protects other outbound surfaces rather than already-computed Telegram reply chunks during polling recovery.

## What This Changes

The issue requests durable runtime recovery for Telegram outbound message delivery when polling stalls or restarts cause `sendMessage` failures.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct send, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and hooks eligible recovery into polling restart as well as startup where the shared queue contract fits.
