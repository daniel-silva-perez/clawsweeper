## Summary

Keep this issue open. Current main has Telegram polling hardening, a shared outbound write-ahead queue, and bounded safe-send retry for pre-connect failures, but it still does not provide the requested continuous runtime recovery or durable pending/ambiguous state for interactive Telegram final replies that fail during polling recovery.

## What This Changes

The issue asks for Telegram outbound delivery recovery so transient sendMessage failures during polling stalls do not silently lose already-computed replies, while ambiguous non-idempotent sends are preserved rather than blindly retried.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records final/block reply chunks before or during direct delivery, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and triggers eligible recovery during polling restart as well as startup where the shared queue contract fits.
