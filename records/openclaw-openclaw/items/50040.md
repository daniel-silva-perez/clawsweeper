## Summary

Keep this open. Current main partially addresses the narrower Telegram safe-send case from #74203 by retrying wrapped pre-connect grammY failures, but the broader durable runtime recovery requested here is still missing: direct inbound Telegram final replies still flow through `deliverReplies` and rethrow send failures without persisting pending/ambiguous reply chunks or draining them after polling recovery.

## What This Changes

The issue requests robust Telegram runtime outbound recovery so polling-stall `sendMessage` failures do not silently drop already-computed replies, with retry classification and stateful delivery entries.

## Best Possible Solution

Keep this open for a maintainer-reviewed Telegram reply-delivery recovery design. The best path is to preserve already-computed direct final/block reply chunks before or during send, classify failures as safe-to-retry versus ambiguous versus permanent, retry only safe cases with bounded backoff, make ambiguous failures operator-visible or held without blind duplication, and trigger recovery after Telegram polling restart as well as gateway startup. The implementation should reuse the shared queue where its contract fits, or add a narrow Telegram/plugin-owned seam where direct reply semantics require it.

## Risks / Open Questions

- A naive retry of ambiguous Telegram `sendMessage` errors can duplicate visible messages when Telegram received the request but the client lost the response.
- Direct final replies carry Telegram-specific reply, quote, preview, hook, and chunking semantics; moving them through the shared queue without a deliberate contract could regress those behaviors.
- Partial chunk delivery can still leave users with truncated replies when a later chunk fails after earlier chunks were delivered.
