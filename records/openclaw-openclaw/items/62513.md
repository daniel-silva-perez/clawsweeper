## Summary

Keep this issue open. Current main still calls `hasIndexedContent()` from the memory search preflight and that helper still performs SQLite probes; the issue is also already paired with an open implementation PR that uses closing syntax, so it should remain open until that PR lands or is replaced.

## What This Changes

Request caching the `memory-core` `hasIndexedContent()` result to avoid repeated SQLite probes during memory search.

## Best Possible Solution

Review and land PR #62597, or an equivalent narrow memory-core patch, that caches the positive indexed-content state and invalidates it on all index mutation/reset paths with focused tests covering search preflight behavior.

## Reproduction Assessment

Yes. A source-level check on current main shows each `search()` starts by calling `hasIndexedContent()`, and that helper performs SQLite `SELECT 1` probes against `chunks` and, when available, `chunks_fts`; no live performance benchmark was run in this read-only review.

## Solution Assessment

Yes, the requested cache with explicit invalidation is the narrow maintainable direction. The safest path is normal review of the existing open PR or a replacement with tests for insert, delete, reset, and empty-index transitions.
