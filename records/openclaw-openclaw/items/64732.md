## Summary

Keep open. Current main partially protects root help and the reserved `tools` namespace, but the missing-command fallback still treats any other unregistered primary token as a plugin id under restrictive `plugins.allow`, which preserves the reported `tool`/synthetic-command failure mode and the maintainer-designated canonical tracker remains valid.

## What This Changes

The issue asks OpenClaw to stop confusing core, framework, synthetic, or command-alias CLI tokens with plugin ids when `plugins.allow` is restrictive.

## Best Possible Solution

Add an explicit missing-primary classification before plugin allowlist diagnostics: framework/core commands should bypass plugin-id guidance, manifest aliases should point to the owning plugin id, runtime-only slash aliases should say they are chat commands, and unknown/stale roots should fall through to normal unknown-command guidance.

## Reproduction Assessment

Yes. The source path is deterministic: with a non-empty `plugins.allow`, an unregistered primary such as `tool` can pass through lazy plugin registration and still receive the generic plugin allowlist error from `resolveMissingPluginCommandMessage()`.

## Solution Assessment

Yes. The narrow maintainable fix is the command-classification layer described in the maintainer comment; adding synthetic ids like `tool`, `help`, or `subagents` to `plugins.allow` would preserve the config contradiction.

## Repair Work Prompt

Fix the CLI missing-command/plugin-allowlist misclassification tracked by https://github.com/openclaw/openclaw/issues/64732. Current main still sends non-plugin roots like `tool` through `resolveMissingPluginCommandMessage()` and can tell users to add synthetic command names to `plugins.allow`. Implement a small explicit classifier before emitting plugin allowlist guidance: core/framework/help commands must never recommend adding themselves to `plugins.allow`; manifest-backed CLI aliases should point to the owning plugin id; runtime-only slash aliases should say they are chat-only and suggest the related CLI command when manifest metadata provides one; unknown or stale roots should fall through to Commander/unknown-command style guidance instead of plugin allowlist guidance. Preserve existing behavior for real plugin ids and manifest-backed plugin CLI aliases such as `browser` and `wiki`. Do not add or restore a new `openclaw tool` CLI command unless existing product docs already require it; `/tool` is a chat slash-command surface. Add focused regression coverage in the CLI policy/exit tests for restrictive `plugins.allow` with `help`, `tool`, `tools`, a real plugin id, a manifest alias, and a runtime slash alias. Add a one-line changelog Fixes entry if the branch changes user-visible CLI diagnostics, crediting the original reporter with `Thanks @efe-arv`.
