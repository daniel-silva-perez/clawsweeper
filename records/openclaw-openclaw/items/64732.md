## Summary

Current main still has the reported CLI policy problem. Unresolved primary command tokens are still passed into the plugin allowlist diagnostic path, and that fallback can still tell users to add non-plugin names such as `tool`, `run`, or `subagents` to `plugins.allow`. The issue also has explicit maintainer signal making this the canonical tracker, with related duplicates closed into it and #65319 left adjacent.

## Best Possible Solution

Implement the maintainer-suggested command-classification layer before the plugin allowlist fallback: framework/core help and built-in command surfaces should bypass plugin allowlist guidance, manifest command aliases should point to the owning plugin ID, runtime slash aliases should explain chat-only or related CLI usage, and unknown or stale non-plugin names should fall through to normal Commander unknown-command handling instead of recommending synthetic `plugins.allow` entries.

## Clownfish Work Prompt

Implement the command-classification fix for the restrictive `plugins.allow` CLI fallback tracked by this issue. Current main still calls `resolveMissingPluginCommandMessageFromPolicy(primary, config, ...)` for unresolved primary tokens and can tell users to add non-plugin command names such as `tool`, `run`, or `subagents` to `plugins.allow`, even though `plugins.allow` is plugin-ID-only.

Expected boundary:
- Add a small explicit classification layer before the missing-plugin allowlist fallback.
- Framework/core help and built-in command surfaces must never suggest adding command names to `plugins.allow`; `openclaw help`, `openclaw help <command>`, and command help should behave like normal help/Commander handling.
- Manifest command aliases should continue to resolve to and message about the owning plugin ID.
- Runtime slash aliases should say they are chat-only or point to the related CLI command when available.
- Unknown or stale non-plugin command names should fall through to normal Commander unknown-command handling, not plugin allowlist guidance.
- Preserve the current useful error for real excluded plugin-backed commands such as `browser`.
- Do not broaden plugin loading or change plugin allow/deny semantics.

Likely tests:
- Add or extend focused tests in `src/cli/run-main.test.ts`, `src/cli/command-registration-policy.test.ts`, `src/cli/argv.test.ts`, and/or `src/cli/run-main.exit.test.ts` for restrictive `plugins.allow` covering `openclaw help`, `openclaw help <command>`, non-plugin `tool`, built-in/core names, real excluded plugin command `browser`, a manifest CLI alias, and a runtime slash alias.
- Add a user-facing changelog fix entry under the active version if behavior changes, crediting reporters such as @efe-arv and related duplicate reporters as appropriate. Do not credit Codex, OpenClaw, or steipete as reporter credit.

Related refs: this issue, #64859, #63907, #65997, #67901, and #65319.
