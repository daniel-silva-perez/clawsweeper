## Summary

Keep open. Current main still parses Feishu fetched quoted/root/thread message content from raw `body.content` without modeling or applying fetched-message mention metadata, while the live inbound path already normalizes Feishu mention placeholders. Open PR #48976 is paired with this issue using closing syntax, so the issue should stay open until that PR or an equivalent focused fix lands.

## What This Changes

The issue requests resolving Feishu quoted/replied-message mention placeholders like `@_user_N` into readable `<at user_id="...">name</at>` mention tags when fetched message content is used for reply context.

## Best Possible Solution

Review and narrow PR #48976, or land an equivalent Feishu-plugin-only patch that models fetched-message `mentions`, adapts fetched mention ids into the existing normalization helper, applies normalization after fetched content parsing for quoted/root/thread messages, and adds focused `extensions/feishu/src/send.test.ts` regression coverage for text and post/thread content containing `@_user_N` plus matching mention metadata.

## Reproduction Assessment

Yes. A mocked `getMessageFeishu()` response with `body.content` containing `@_user_1` and sibling `mentions` metadata would currently return the raw placeholder because `parseFeishuMessageItem()` ignores mentions; the same parser is used for quoted and thread context.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Feishu plugin fetched-message parser rather than core. The implementation should adapt the fetched mention shape and add regression tests; the already-open PR should be reviewed or narrowed instead of starting a duplicate repair lane.
