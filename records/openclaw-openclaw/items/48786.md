## Summary

Keep open. Current main still ignores fetched Feishu message mention metadata when building quoted/root/thread context, so raw `@_user_N` placeholders can pass through even though the live inbound path already normalizes them. Open PR #48976 is paired with this issue using closing syntax, so the issue should remain open until that PR or an equivalent focused fix lands.

## What This Changes

Resolve Feishu quoted/replied message context so fetched `@_user_N` mention placeholders are normalized with fetched message mention metadata.

## Best Possible Solution

Review and narrow PR #48976, or land an equivalent Feishu-plugin-only patch that models fetched-message `mentions`, adapts the fetched mention shape into the existing normalization helper, applies normalization after fetched content parsing for quoted/root/thread messages, and adds focused `extensions/feishu/src/send.test.ts` coverage for text and post/thread content containing `@_user_N` plus matching mention metadata.

## Reproduction Assessment

Yes. A mocked `getMessageFeishu()` response with `body.content` containing `@_user_1` plus sibling mention metadata would currently return the raw placeholder because `parseFeishuMessageItem()` ignores mentions; the same parser feeds quoted and thread context.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the Feishu plugin fetched-message parser rather than core, reusing the existing mention-normalization behavior with a small adapter and regression tests.
