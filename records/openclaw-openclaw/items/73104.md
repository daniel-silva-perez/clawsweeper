## Summary

Keep this PR open. Current main still lacks Telegram `edited_message` handling, so the branch is aimed at a real gap, but the proposed debounce path appears to drop real production updates after the timer fires because it reuses an update context that the Telegram update tracker has already finalized as handled.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-message debounce, a test-only debounce timing option, focused edited-message tests, and a test-harness idle callback cleanup.

## Best Possible Solution

Keep the PR open and repair the debounce/update-tracker interaction so the tracker dedupes the Telegram update before the handler returns, while the delayed final inbound dispatch does not re-check the same completed update key. Add a regression test that exercises the middleware/update-tracker path, add the required changelog entry, rerun the Telegram changed gate, then land this PR or an equivalent narrow replacement.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static: in the real bot pipeline the update-tracker middleware finalizes the update when the handler returns, then the timer calls `handleInboundMessageLike` with the same context and `shouldSkipUpdate` treats it as recent and skips dispatch.

## Solution Assessment

No. The handler shape and debounce idea are the right direction, but reusing the original update context for a delayed dispatch is not the narrowest maintainable implementation because it collides with the existing update tracker; the safer fix is to separate update-level dedupe from the delayed final message dispatch and test that production path.
