## Summary

Keep open: current main still does not provide the broad outbound markdown-stripping behavior, and the related feature request remains valid, but this PR is not merge-ready. The branch has concrete regressions in markdown-capable channel detection, WhatsApp formatting, account-scoped config handling, and fenced-code stripping.

## What This Changes

The branch adds a `markdown.strip` config option, expands `stripMarkdown()`, and wires outbound delivery to strip markdown after plain-text sanitization for channels detected as non-markdown-capable.

## Best Possible Solution

Keep the feature direction tied to #24943, but require a rebase and a narrower capability-aware implementation before merge: strip only truly plain-text delivery surfaces, preserve WhatsApp/lightweight-markdown channels, honor account-scoped `markdown.strip`, and coordinate the fenced-code behavior with #58948 rather than landing overlapping broken regex changes.

## Reproduction Assessment

Yes. The PR bugs have high-confidence static reproduction paths: `shouldStripMarkdown()` would auto-strip channels whose metadata does not advertise markdown, WhatsApp runs lightweight formatting through `sanitizeForPlainText()`, account markdown config already has an account-scoped resolver pattern, and the fenced-code regex was reproduced with a read-only Node probe.

## Solution Assessment

No. The requested behavior is useful, but this diff is not the best maintainable solution because it treats missing metadata as plain text, ignores account-scoped config, and expands a markdown stripper with brittle regexes; a safer fix should use explicit channel/plugin capability contracts and focused parser tests.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve markdown-rendering Matrix output:** `src/infra/outbound/sanitize-text.ts:85`
  - body: `shouldStripMarkdown()` defaults to stripping whenever channel metadata does not report markdown support. Matrix outbound uses `chunkerMode: "markdown"` and renders markdown to Matrix HTML, but its package channel metadata does not declare `markdownCapable`, so auto mode would downgrade Matrix replies. Make the capability source complete or restrict auto-stripping to an explicit plain-text set.
  - confidence: 0.88
- **[P1] Keep WhatsApp lightweight formatting intact:** `src/infra/outbound/sanitize-text.ts:85`
  - body: WhatsApp outbound text is normalized through `sanitizeForPlainText()`, which intentionally turns HTML into WhatsApp-style `*bold*`, `_italic_`, and backtick markers. Since WhatsApp is not marked markdown-capable, this new return path strips those markers right after they are produced, losing intended formatting. Exclude WhatsApp or convert markdown into its supported syntax instead of stripping it.
  - confidence: 0.9
- **[P2] Honor account-scoped markdown.strip overrides:** `src/infra/outbound/deliver.ts:370-371`
  - body: The resolver only reads `cfg.channels[channel].markdown`, but channel configs also support `channels.<channel>.accounts.<id>.markdown` and existing markdown table resolution checks account overrides first. Multi-account users cannot set different strip behavior per account unless this lookup accepts `accountId` and follows the same precedence.
  - confidence: 0.85
- **[P2] Consume full fenced-code fences:** `src/shared/text/strip-markdown.ts:9`
  - body: The fenced-code regex only accepts lowercase language tags and leaves the newline after the closing fence outside the match. Cases like a mid-message fence produce an extra blank line, and fences such as `c++` leak `++` into the delivered text. Match the full info string and consume the trailing newline before relying on this in outbound delivery.
  - confidence: 0.93
- **[P3] Add the required changelog attribution:** `CHANGELOG.md:112`
  - body: The new changelog entry has no `Thanks @...` credit. OpenClaw changelog rules require each added user-facing entry to include at least one credited GitHub username, so this should be fixed before merge.
  - confidence: 0.82
