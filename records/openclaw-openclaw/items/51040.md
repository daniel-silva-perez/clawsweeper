## Summary

Current main still does not make macOS Voice Wake respect the selected session target. The macOS forwarding defaults are still `sessionKey: "main"` and `channel: .webchat`, both Voice Wake send paths construct options with only `voiceWakeTrigger`, and the separate active WebChat session state is not consulted by the Voice Wake path. Gateway-level `voicewake.routing` exists and can route by wake trigger, but it is a separate configured routing layer rather than a binding to the macOS session picker described in this issue.

## Best Possible Solution

Keep this issue open. Fix the macOS Voice Wake send path to resolve the active selected session target at send time and pass the resulting `sessionKey`, `channel`, and `to` into `VoiceWakeForwarder.ForwardOptions`. Add a macOS unit test for a non-main selected session and, if the design uses Gateway `voicewake.routing`, document how the macOS picker updates that config rather than leaving it as a separate manual routing mechanism.
