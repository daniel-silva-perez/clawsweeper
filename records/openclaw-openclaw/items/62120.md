## Summary

Keep open. Current main still lets `openclaw channels login --channel openclaw-weixin` inherit the parent `channels` startup policy, so CLI bootstrap can broadly preload plugins before `runChannelLogin()` reaches the selected external plugin's QR login. The scoped channel resolver exists, but login/logout still need an exact no-preload startup policy and explicit selected-channel loading.

## What This Changes

Make `openclaw channels login --channel openclaw-weixin` reach the external plugin's QR login instead of hanging during host CLI preflight/plugin preload.

## Best Possible Solution

Implement a generic host CLI fix: make `channels login` and `channels logout` avoid broad pre-action plugin preload, then ensure channel auth explicitly loads only the requested or inferred auth-capable channel through the existing scoped channel setup/plugin resolution path. Keep OpenClaw core channel-agnostic and avoid hard-coded Weixin policy.

## Repair Work Prompt

Fix the host CLI channel-auth startup path for this issue. Observable bug: `openclaw channels login --channel openclaw-weixin --verbose` can hang before the external plugin's QR login output because `channels login`/`logout` inherit parent `channels` `loadPlugins: "always"` and broadly preload plugins during CLI bootstrap before `runChannelLogin()` runs. Implement a generic OpenClaw-core fix: add exact startup-policy/catalog handling so `channels login` and `channels logout` do not broad-preload plugins, then ensure `runChannelLogin()`/`runChannelLogout()` explicitly resolve and load only the requested or inferred auth-capable channel through the existing scoped channel setup/plugin resolution path. Preserve already-installed and catalog/trusted external channel behavior, including non-catalog trusted workspace plugins if supported by the current resolver. Add focused regression coverage in CLI startup-policy/path/execution tests and channel-auth/channel-plugin-resolution tests.
