## Summary

Keep open: current main still keeps Control UI/WebChat turns internal and prevents them from inheriting stored external routes, while Feishu inbound turns do record a Feishu route. The reported Feishu reply loss remains a real cross-surface routing/product decision rather than cleanup-close material.

## What This Changes

The issue reports that a Feishu-started conversation loses subsequent Feishu delivery after the user interrupts or continues the same session from the Web Control UI.

## Best Possible Solution

Define an explicit cross-surface delivery contract for Feishu-started sessions, then implement a permissioned route-inheritance or opt-in mirroring path that keeps WebChat's leakage guard intact and adds focused regression coverage for Feishu main-session delivery after a Control UI follow-up.

## Reproduction Assessment

Yes. The source path is high-confidence: Feishu records `OriginatingChannel=feishu`, Control UI sends `chat.send` with `deliver:false` as WebChat, and current tests assert WebChat/UI main-session sends do not inherit external routes.

## Solution Assessment

Unclear. Automatically inheriting the stored Feishu route would address the visible loss, but it is not clearly the best fix until maintainers define the product and security contract for cross-surface delivery from the Control UI.
