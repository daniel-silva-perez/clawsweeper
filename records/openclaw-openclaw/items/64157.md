## Summary

Current main still matches the reported failure mode rather than implementing the requested Feishu pushback. Normal Control UI chat sends are explicitly internal (`deliver: false`), the gateway refuses to inherit stored external routes for UI/main-session sends even with `deliver: true`, and Feishu inbound dispatch records the Feishu origin but does not change the Control UI send path into an external Feishu delivery path.

## What This Changes

The issue requests that assistant replies continue to be delivered to the original Feishu conversation when a user interrupts or continues the same conversation from the Web Control UI.

## Best Possible Solution

Keep this open and decide the intended product contract first. If Control UI should be able to continue a Feishu-started conversation and push the final answer back to Feishu, implement an explicit, permissioned route inheritance or opt-in mirroring mode that preserves the existing WebChat safety guards and adds Feishu main-session plus channel-scoped regression coverage.
