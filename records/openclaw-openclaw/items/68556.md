## Summary

Keep this PR open, but do not merge it as-is. Current main has not implemented the intended behavior: the active package-owned resolver still scopes explicitly named workspace-local custom QMD collections per agent. The proposed patch also needs a current-main rebase, a product/security decision about whether local explicit names may intentionally collapse agent isolation, and a scoped-to-unscoped QMD collection migration path before it is safe.

## What This Changes

The PR changes QMD custom collection resolution so explicit custom collection names are preserved verbatim, updates memory-core expectations from agent-suffixed names to explicit names, and keeps unnamed custom collections on the generated agent-scoped naming path.

## Best Possible Solution

If maintainers want explicit local QMD names to be authoritative, implement it in the package-owned resolver with collision-safe multi-agent semantics and a runtime migration from scoped names to the new names. Otherwise, keep the current rule: explicit names are preserved for genuinely shared external paths, while workspace-local/per-agent paths remain scoped.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep local custom names collision-safe across agents:** `src/memory-host-sdk/host/backend-config.ts:261-263`
  - body: This makes a default relative custom path such as `path: "notes", name: "workspace"` resolve to `workspace` for every agent even though each agent has a different workspace root. QMD collections are managed and searched by collection name, so one agent can rebind or search another agent's collection. Preserve scoping for per-agent/local paths, or add an explicit collision-safe policy before unscoping them.
  - confidence: 0.89
- **[P1] Add scoped-to-unscoped collection migration:** `src/memory-host-sdk/host/backend-config.ts:261-263`
  - body: Existing workspace-local explicit collections are already stored as `<name>-<agent>`, but after this change searches target `<name>`. The current migration only handles the opposite unscoped-to-scoped direction, and name-only QMD list output cannot identify the old slot for rebind, so upgraded installs can keep searching a missing collection. Add compatibility coverage before changing the resolved name.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No CI, dependency, or supply-chain surface changed, but the naming change can weaken multi-agent memory isolation by making different local paths share one QMD collection name.

Concerns:

- **[high] Potential cross-agent memory collection collision:** `src/memory-host-sdk/host/backend-config.ts:261`
  - body: Preserving the same explicit name for workspace-relative per-agent paths can collapse separate agents onto one global QMD collection key, causing memory search results or rebinds to cross workspace boundaries.
  - confidence: 0.82
