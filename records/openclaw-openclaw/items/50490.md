## Summary

Keep open. Current main still matches the Feishu-specific gap described in the discussion: `/activation mention` persists generic session activation state, but Feishu decides whether to admit non-mentioned group messages from config-only `requireMention` before that session activation state can affect routing.

## What This Changes

The issue reports that Feishu group `/activation mention` appears to succeed but later non-mentioned group messages still trigger responses when config mention gating was disabled.

## Best Possible Solution

Implement the behavior in the Feishu plugin so the persisted group activation for the admitted group session overrides config-derived mention gating before the hard mention check, while preserving group and sender allowlist blocks; update focused tests, Feishu/group docs, and the user-facing changelog.

## Repair Work Prompt

Fix the Feishu group activation regression reported at https://github.com/openclaw/openclaw/issues/50490. In Feishu group chats, `/activation mention|always` persists `sessionEntry.groupActivation`, but the Feishu pre-dispatch `requireMention` gate ignores that session override. Keep the fix scoped to the Feishu plugin unless a small generic helper is clearly needed. After group disabled/group allowlist/group sender allowlist checks pass, resolve the effective activation for the target group session before the `requireMention && !ctx.mentionedBot` return. Use config-derived `requireMention` as the fallback, and persisted `groupActivation` as the per-session override. Do not let activation state bypass `groupPolicy`, `groups.<id>.enabled=false`, group admission, or sender allowlists. Account for normal group sessions, topic-scoped sessions, and broadcast/multi-agent session keys. Add focused Feishu regression tests for `/activation mention` with config `requireMention:false`, `/activation always` with config `requireMention:true`, missing/invalid session-store fallback to config, and blocked group/sender cases. Update `docs/channels/feishu.md` and `docs/channels/groups.md` to state the supported Feishu behavior, and add a user-facing changelog fix entry crediting `Thanks @Cafeynman`. Do not change unrelated channel behavior or broad command semantics.
