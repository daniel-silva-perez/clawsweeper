## Summary

Keep open. Current main still matches the reported Feishu behavior: the Feishu group hard gate uses config-derived `requireMention` before the agent route/session is resolved, while `/activation mention` only persists `sessionEntry.groupActivation` for the generic reply path. Hollychou924's root-cause comment is consistent with the current source, and the remaining fix is narrow enough for a focused Feishu plugin PR.

## Best Possible Solution

Keep this open and fix it in the Feishu plugin: resolve the effective group activation for a Feishu group session as session override over Feishu config/default before the hard mention gate, preserve existing access-control and broadcast behavior, add Feishu regression coverage for `/activation mention`, and update docs/changelog if the supported Feishu surface changes.

## Clownfish Work Prompt

Fix openclaw/openclaw#50490. Current Feishu group handling lets `/activation mention` return success but still dispatches ordinary non-@ group messages when config-level `requireMention` is false, because `extensions/feishu/src/bot.ts` applies a hard config-derived gate before the route/session state is available and `src/auto-reply/reply/commands-session.ts` only stores `sessionEntry.groupActivation`. Implement this in the Feishu plugin boundary: resolve effective group activation as session `groupActivation` override over Feishu config/default before the hard mention gate, then use that effective mode for the non-mention drop decision. Preserve disabled groups, groupPolicy/groupAllowFrom admission, per-group sender allowlists, broadcast groups, topic/thread group session scopes, bot-mention detection, pending history behavior, and authorized control-command handling. Prefer existing runtime/session helpers already exposed through the Feishu runtime API; do not add bundled Feishu ids or owner-specific policy to core, do not mutate `openclaw.json`, and do not change unrelated channel behavior. Add Feishu regression tests proving that a persisted `groupActivation: "mention"` suppresses non-mentioned group dispatch even when Feishu config would otherwise allow all messages, that mentioned messages still dispatch, and that `groupActivation: "always"` or no session override preserves existing behavior. Update `docs/channels/feishu.md`, `docs/channels/groups.md`, and `CHANGELOG.md` only if the supported user-facing Feishu `/activation` surface changes; if adding a changelog entry, credit the reporter/commenter per repo rules.
