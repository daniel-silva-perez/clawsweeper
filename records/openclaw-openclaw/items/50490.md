## Summary

Keep open. Current main still matches the Feishu-specific activation gap: `/activation mention` persists generic group activation state, but Feishu's hard group-message gate still derives `requireMention` from configuration before route/session activation state is available, so unmentioned messages can continue into dispatch when config allows them.

## What This Changes

The issue asks Feishu group chat to honor `/activation mention` by replying only when the bot is mentioned after the command succeeds.

## Best Possible Solution

Implement this in the Feishu plugin: after group and sender allowlist checks, resolve the effective routed group session activation and use the persisted `/activation` value as the per-session override for mention admission, while keeping config-derived `requireMention` as the default. Update Feishu/group docs, focused tests, and changelog so the command confirmation reflects real runtime behavior.

## Reproduction Assessment

Yes. The issue steps are reproducible from source on current main: with Feishu config allowing unmentioned group messages, `/activation mention` persists `groupActivation`, but the next unmentioned message is admitted by a config-only Feishu gate before that session state is consulted.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-owned admission logic, not a core startup migration or prompt-only change, because other channel owners already bridge persisted activation into their own hard gates and Feishu currently lacks that bridge.
