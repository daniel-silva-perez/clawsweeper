## Summary

Keep open. Current main still sends `openclaw models set` through the shared config replacement path without any command-specific write policy, while the config writer still rejects raw payloads that shrink below half of the prior file size with `CONFIG_WRITE_REJECTED`; the prior ClawSweeper comment remains consistent with the current source.

## What This Changes

The issue reports that `openclaw models set` can fail to persist a legitimate default-model change because the config write safety guard treats the rewrite as a destructive size drop.

## Best Possible Solution

Add a regression test that drives a real `openclaw models set` write through the config writer with a valid config whose raw formatting or benign rewrite causes a large byte-size drop, then fix the shared write policy so intended small semantic config changes persist while actual clobbers remain blocked.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction class: use a valid existing `openclaw.json` over 512 bytes whose raw representation shrinks below half when rewritten, then run `openclaw models set <provider/model>`. The current path reaches `writeConfigFile` without `allowDestructiveWrite`, so it will reject with `CONFIG_WRITE_REJECTED`.

## Solution Assessment

No current-main solution is present. The safest direction is a semantic or normalized-size policy with regression coverage, not a blanket `allowDestructiveWrite` bypass for `models set`.
