## Summary

Keep open. Current `main` still routes `openclaw models set` through the shared config replacement path without command-specific write policy, while the config writer still blocks raw size drops with `CONFIG_WRITE_REJECTED`.

## What This Changes

The issue reports that `openclaw models set` can reject an intended default-model update when the rewritten config is less than half the previous raw file size.

## Best Possible Solution

Add regression coverage that drives a real `openclaw models set` write through the config writer with a valid config whose raw formatting or benign rewrite causes a large byte-size drop, then adjust the shared write policy so intended semantic config changes persist while actual clobbers remain blocked.

## Reproduction Assessment

Yes. A high-confidence reproduction class is a valid existing `openclaw.json` over 512 bytes whose rewritten JSON drops below half the prior raw size, followed by `openclaw models set <provider/model>`; the current path reaches `writeConfigFile` without `allowDestructiveWrite`.

## Solution Assessment

No current-main solution is present. The narrowest maintainable direction is a semantic or normalized-size policy with regression coverage, not a blanket destructive-write bypass for the models command.
