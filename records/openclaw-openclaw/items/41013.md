## Summary

Keep open. Current main and the latest release still lack `<session-recap>` handling in the shared inbound metadata sanitizer, so the user-visible leak reported in #40948 remains real. The PR is a focused fix in the right area, but the current patch still has blocking sanitizer bugs and the branch is currently reported unmergeable against main.

## What This Changes

The PR adds a `<session-recap>` fast-path/helper to `src/auto-reply/reply/strip-inbound-meta.ts` and colocated tests for stripping leading recap blocks while preserving mid-message tags.

## Best Possible Solution

Rebase or replace the focused sanitizer fix on current main so both full and leading-only user-visible cleanup strip only complete leading `<session-recap>` or `<session_recap>` blocks, preserve malformed and mid-message literals unchanged, retain existing timestamp/active-memory behavior, and land it as the implementation for #40948.

Full review comments:

- **[P1] Require a closing session-recap tag before stripping:** `src/auto-reply/reply/strip-inbound-meta.ts:131-135`
  - body: When the first nonblank line is `<session-recap>` but no closing tag exists, the helper walks to EOF and returns `lines.slice(i)`, erasing the entire message. Treat malformed or user-authored snippets as non-metadata unless a matching close tag is found.
  - confidence: 0.96

  - confidence: 0.91
