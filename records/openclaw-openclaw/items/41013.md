## Summary

Keep open. The linked user-visible leak still appears valid on current main because the shared inbound-message sanitizer has no `<session-recap>` fast path, but this PR should not merge as-is: the proposed helper can erase malformed/user-authored recap-like text and can trim content when no leading recap block was actually removed.

## What This Changes

The PR adds a session-recap fast path, a helper to strip leading `<session-recap>`/`<session_recap>` blocks, and colocated tests for the inbound metadata sanitizer.

## Best Possible Solution

Land a rebased sanitizer fix that strips only complete leading `<session-recap>` and `<session_recap>` blocks in both display paths, leaves malformed and mid-message literals untouched, preserves the current timestamp and active-memory stripping behavior, and adds regression coverage for malformed/no-close and no-strip whitespace cases before resolving #40948.

## Reproduction Assessment

Yes. The current-main leak is reproducible by passing a leading closed `<session-recap>` block without standard metadata sentinels through `stripInboundMetadata()` or the TUI leading-strip path; the PR regressions are source-level reproducible with a missing closing tag and with a non-leading recap marker plus leading/trailing whitespace.

## Solution Assessment

No. The direction is the right narrow fix, but the current implementation should track whether a complete leading block was actually stripped and return the original text when it was not, rather than consuming to EOF or trimming unchanged user content.
