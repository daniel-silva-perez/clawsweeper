## Summary

Keep open. Current main still lacks `<session-recap>` stripping in the shared inbound-message cleanup path used by Control UI and TUI, so the PR is not obsolete; however, the submitted patch has concrete correctness bugs and needs a small repair before it can land.

## What This Changes

The PR adds a `<session-recap>` fast-path/helper to `strip-inbound-meta.ts` and tests for leading, blank-line, underscore-variant, chained-sentinel, and mid-message cases.

## Best Possible Solution

Repair and rebase the focused sanitizer change on current main, stripping only well-formed leading recap blocks in both full and leading-only cleanup while preserving existing timestamp, active-memory, and non-stripped user text behavior; then land it as the implementation for #40948.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Require a closing session-recap tag before stripping:** `src/auto-reply/reply/strip-inbound-meta.ts:131-135`
  - body: When the first nonblank line is `<session-recap>` but no closing tag exists, the loop advances to the end and `return lines.slice(i)` drops the whole message. A pasted or malformed XML snippet would be erased in both cleanup paths; return the original lines unless a matching close tag is found.
  - confidence: 0.96
- **[P2] Preserve the original text when no recap block is removed:** `src/auto-reply/reply/strip-inbound-meta.ts:172-173`
  - body: A mid-message `<session-recap>` sets `hasSessionRecap`, the helper leaves the lines unchanged, and the no-sentinel branch rejoins plus trims leading/trailing blank lines. That mutates user-authored content even though no internal block was stripped; keep the original string when the helper makes no change.
  - confidence: 0.91
