## Summary

Keep open. Current main and v2026.4.27 still do not strip leading `<session-recap>` prefixes, so the reported UI/TUI leak remains real, but this PR should not merge as-is because the patch can erase malformed user text and mutate mid-message user content.

## What This Changes

The PR adds a `stripLeadingSessionRecapBlock` helper, a session-recap fast path, and colocated tests for stripping leading `<session-recap>` / `<session_recap>` blocks from user-visible inbound messages.

## Best Possible Solution

Land a rebased or replacement sanitizer fix on current main that strips only complete leading `<session-recap>` / `<session_recap>` blocks in both full and leading-only display paths, preserves malformed and mid-message literals unchanged, keeps current timestamp and active-memory behavior, and adds regression tests for the malformed/no-close and whitespace-preservation cases before resolving #40948.
