## Summary

Keep open. Current main still has no `<session-recap>` handling in the shared inbound-message sanitizer, and the PR targets the right user-visible leak, but the branch has two concrete content-loss regressions that should block merge until fixed.

## What This Changes

The PR adds a session-recap fast path, a helper for stripping leading `<session-recap>` or `<session_recap>` blocks from inbound user text, and colocated sanitizer tests.

## Best Possible Solution

Land a rebased, narrow sanitizer fix that strips only complete leading `<session-recap>` and `<session_recap>` blocks in both display paths, leaves malformed or mid-message literals untouched, preserves existing inbound metadata behavior, and adds regression coverage before resolving the linked leak report.

## Reproduction Assessment

Yes. A leading closed `<session-recap>` block with no standard inbound metadata sentinel bypasses both current-main sanitizer entry points; the PR-specific regressions are reproducible from the proposed loop and slow-path return behavior.

## Solution Assessment

No. The direction is the right narrow fix, but the current PR should track whether a complete leading recap block was actually removed, require a closing tag before stripping, and return the original text when no leading block was stripped.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Require a closing recap tag before stripping:** `src/auto-reply/reply/strip-inbound-meta.ts:131-135`
  - body: When the first nonblank line is a recap opener but no closing tag exists, this loop reaches EOF and `lines.slice(i)` deletes the whole message. Only strip after finding a matching close tag so malformed or pasted user XML remains visible.
  - confidence: 0.96

  - confidence: 0.91
