## Summary

Keep open. Current main still does not strip leading session-recap blocks, so the PR is not obsolete, but the proposed sanitizer change has blocking edge-case regressions that can drop malformed user text or alter untouched user content.

## What This Changes

The PR adds a session-recap fast path, a helper for stripping leading `<session-recap>`/`<session_recap>` blocks from inbound user text, and colocated sanitizer tests.

## Best Possible Solution

Land a rebased, narrow sanitizer fix that strips only complete leading `<session-recap>` and `<session_recap>` blocks in both Control UI and TUI display paths, leaves malformed or mid-message literals untouched, preserves existing inbound metadata behavior, and adds regression coverage before resolving the linked leak report.

## Reproduction Assessment

Yes. A leading closed `<session-recap>` block with no standard inbound metadata sentinel will bypass both current-main sanitizer entry points; the PR-specific regressions are source-level reproducible with a missing closing tag and with a non-leading recap marker plus user-authored surrounding whitespace.

## Solution Assessment

No. The direction is the right narrow fix, but the current PR branch should track whether a complete leading recap block was actually removed and return the original text when it was not; it also needs a rebase that preserves current sanitizer behavior.
