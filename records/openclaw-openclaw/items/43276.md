## Summary

Keep this PR open. Current main reduces the default user-visible failure by making group/channel visible replies message-tool-only, but it still supports legacy automatic group replies and that path still injects the same same-group message-tool prohibition the PR is trying to relax for attachments.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best fix is a narrow rebase that leaves the current `message_tool_only` default intact, changes the legacy `automatic` group/channel prompt to direct text replies normally while permitting `message(action=send)` for files/images/attachments to the same group, and updates tests around the current trusted/untrusted prompt structure.

## Risks / Open Questions

- Applying the PR as-is would be stale: it needs to preserve the current `message_tool_only` branch and update only the remaining `automatic` prompt path.
- The rebase should preserve the trusted/untrusted prompt split; group subjects, members, topic labels, and similar human-controlled metadata should stay in untrusted inbound context.
- The security-sensitive surface is prompt policy, not supply chain: wording should allow same-group message sends for files/images/attachments without encouraging unnecessary visible text sends in automatic mode.
