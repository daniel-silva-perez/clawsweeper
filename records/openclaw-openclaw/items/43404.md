## Summary

Keep open. Current main still has the original reminder-guard bug, and this PR fixes useful parts of it, but PR head still introduces a P2 call-site regression by deriving the default `main` agent id when `sessionKey` is absent or not an agent-shaped key.

## What This Changes

The PR expands reminder-guard cron matching to isolated jobs by agent id, preserves `successfulCronAdds` during compaction retry, and adds reminder-guard unit tests.

## Best Possible Solution

Land this PR or a narrow replacement after guarding the runner call site so isolated agent-id matching is used only when the run has reliable session identity, while keeping the compaction-counter preservation and the isolated-job matcher coverage.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original bug: isolated jobs cannot match the guard and compaction clears the successful cron-add counter. The PR regression is reproducible by invoking `runReplyAgent` without `sessionKey` while the cron store contains an enabled isolated `main` job with no `sessionKey`.

## Solution Assessment

No. The PR is the right narrow direction, but deriving an agent id from an absent or non-agent-shaped session key is not the safest implementation; the call site should pass an agent id only from a trusted parsed session identity.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Guard agent-id fallback when session key is absent:** `src/auto-reply/reply/agent-runner.ts:531`
