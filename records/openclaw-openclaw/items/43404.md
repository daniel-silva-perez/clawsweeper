## Summary

Keep open. Current main still exhibits the original reminder-guard gaps, and this PR is directionally useful, including the author's follow-up that restricts the helper fallback to `sessionTarget: "isolated"`. It still has a concrete call-site regression: omitted session keys are resolved to the default `main` agent, so an unrelated isolated main-agent cron can suppress the guard note.

## What This Changes

The PR expands reminder-guard cron matching to include isolated jobs by agent id, preserves `successfulCronAdds` across compaction retry, and adds reminder-guard unit tests.

## Best Possible Solution

Land this PR or a narrow replacement after guarding the runner call site so isolated agent-id matching is used only when the run has reliable session identity, while keeping the compaction-counter preservation and adding the omitted-session regression.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original bug: isolated jobs cannot match the guard and compaction clears the successful cron-add counter. The PR regression is reproducible by invoking `runReplyAgent` without `sessionKey` while the cron store contains an enabled isolated `main` job with no `sessionKey`.

## Solution Assessment

No. The PR is the right narrow direction, but the current call-site implementation is not the safest solution because it derives `main` from missing session identity; only pass an agent id when the runner can trust the session identity.
