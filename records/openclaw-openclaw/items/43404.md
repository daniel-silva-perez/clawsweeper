## Summary

Keep open. Current main still lacks both intended fixes, and this PR is useful source work, but the latest head still introduces an over-suppression path when `sessionKey` is absent because it default-resolves the agent id to `main`.

## What This Changes

This PR updates reminder-guard cron matching to include enabled isolated cron jobs by agent id, preserves `successfulCronAdds` across compaction retry, and adds helper-level reminder-guard tests.

## Best Possible Solution

Land this PR or a narrow replacement after the call site passes an agent id only when the runner has reliable session identity, while keeping isolated-job matching scoped to `sessionTarget: "isolated"`, preserving `successfulCronAdds` through compaction retry, and adding the missing `runReplyAgent` regression.

## Reproduction Assessment

Yes. Current main has a direct source-level reproduction for the original bug, and the PR regression is reproducible by omitting `sessionKey` while the cron store contains an enabled isolated `main` job with no `sessionKey`; the existing missing-session test shows the note should still be appended.

## Solution Assessment

No. The PR is directionally the right narrow fix, but the call-site `resolveAgentIdFromSessionKey(sessionKey)` is not safe when `sessionKey` is absent; the safer fix is to pass `agentId` only when a canonical session key or otherwise reliable agent identity is present.
