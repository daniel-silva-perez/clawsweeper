## Summary

Keep open. Current main partially addresses the request through the shared `channels.modelByChannel` surface, including channel/group-style conversations and the Discord channel use case from discussion, but ordinary DM matching is still not wired and the latest schema-export report is valid: the public generated config schema keeps only a UI hint for `channels.modelByChannel`, not the actual schema property.

## What This Changes

The issue asks for persistent model selection overrides for channel, group, and direct-message conversations in OpenClaw configuration.

## Best Possible Solution

Finish the generic `channels.modelByChannel` implementation instead of adding provider-specific nested `model` fields: add direct-message candidate resolution from finalized/persisted origin data, expose the shared property in the public config schema, document accepted DM keys, and keep session/runtime overrides higher precedence than config defaults.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: a direct conversation with `channels.modelByChannel.<provider>.<dm-key>` has no resolver candidate when `groupId` and `parentSessionKey` are absent, and the generated config schema exposes no `channels.modelByChannel` property under `channels.properties`.

## Solution Assessment

Yes. The best direction is to complete the existing generic `channels.modelByChannel` contract because it already covers channel/group overrides and avoids adding Slack/WhatsApp/Discord-specific config shapes; the current implementation is only partial.
