## Summary

Keep open. Current main has the shared `channels.modelByChannel` surface and covers the channel/group portion, including the Discord channel use case raised in the discussion, but ordinary direct messages still are not wired into model override resolution because the resolver and reply call site do not pass stable DM identifiers.

## What This Changes

Request to add persistent model selection overrides for specific channels, groups, and direct-message conversations through OpenClaw configuration.

## Best Possible Solution

Finish `channels.modelByChannel` as the generic config surface for direct conversations by deriving normalized DM candidate keys from finalized context, persisted session origin/delivery data, native provider ids, and current direct session keys, while preserving existing session override and channel/group precedence. Document the accepted DM key forms instead of adding provider-specific nested `model` fields.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve a direct-message turn with `channels.modelByChannel` configured but no `groupId` or parent group/channel key: `resolveDirectChannelModelMatch` returns `null`, and the reply path does not supply the DM identifiers that could match.

## Solution Assessment

Yes. Extending the shared resolver and its call sites is the narrowest maintainable path because current main already chose `channels.modelByChannel`; adding WhatsApp/Slack-specific nested `model` fields would duplicate config semantics and miss other channels such as Discord.
