## Summary

Keep open. Current main already covers the channel/group part through the shared `channels.modelByChannel` surface, including the Discord channel use case raised in comments, but ordinary DM matching is still missing because the resolver and reply call site do not pass direct-peer identifiers into override candidate construction.

## What This Changes

The issue requests persistent model selection overrides for channel, group, and DM conversations through configuration.

## Best Possible Solution

Finish the generic `channels.modelByChannel` implementation for direct conversations by deriving normalized DM candidate keys from finalized context, persisted session origin, native provider identifiers, direct delivery targets, and the current direct session key; document the accepted key forms without adding provider-specific nested `model` fields.
