## Summary

Keep this issue open. Current main has the generic `channels.modelByChannel` surface for persistent channel/group model pins, but the per-DM part of the request remains unimplemented: the resolver still only receives group/thread-ish identifiers, and normal direct conversations have no direct-key candidates to match.

## Best Possible Solution

Keep this open and finish the generic `channels.modelByChannel` implementation for direct conversations. Extend the resolver input and reply call site to include normalized direct-peer candidates from finalized context, persisted session origin, native direct identifiers, and current session key; document supported direct key forms and wildcard behavior; add focused Slack/WhatsApp/Discord DM regression coverage while preserving existing channel/group precedence.

## Clownfish Work Prompt

Fix persistent per-DM model overrides for the existing `channels.modelByChannel` config surface tracked by openclaw/openclaw#53638.

Observable bug: current main supports channel/group model pins, but direct conversations do not reliably match because `resolveChannelModelOverride` only receives channel/group/parent-session fields. Ordinary DMs have no `groupId` or `parentSessionKey`, so `resolveDirectChannelModelMatch` can return null even when the finalized context has direct identifiers.

Expected boundary:
- Extend `src/channels/model-overrides.ts` with typed direct-peer inputs and normalized candidate generation for DMs.
- Update `src/auto-reply/reply/get-reply.ts` to pass direct candidates from finalized context, active session key, session entry origin/delivery metadata where available, and native direct identifiers.
- Preserve precedence: explicit per-session `/model` or `session_status(model=...)` and stored user overrides still beat configured channel/direct defaults.
- Preserve current channel/group/thread behavior and wildcard matching.
- Document supported direct key forms and wildcard behavior in `docs/gateway/config-channels.md` if behavior changes are user-visible.
- Add focused regression tests for direct matching, including Slack-style `user:<id>`, WhatsApp phone/JID-style IDs where feasible, and Discord direct user/channel-style candidates if the shared resolver supports them.
- Do not bundle #60078 announce delivery behavior or #63425 context-token display behavior into this PR.
- No dependency changes, no generated baseline edits, and no broad config migration.

Likely files: `src/channels/model-overrides.ts`, `src/channels/model-overrides.test.ts`, `src/auto-reply/reply/get-reply.ts`, `src/auto-reply/reply/*get-reply*.test.ts`, `docs/gateway/config-channels.md`, and possibly channel finalization tests under `extensions/slack` / `extensions/whatsapp` / `extensions/discord` if direct identifier assumptions need locking down.

Validation: run targeted tests for the resolver and touched reply/channel tests, then use the changed gate in Testbox before handoff if code/docs changed broadly enough.
