## Summary

Keep open. Current main partially satisfies the request with the shared `channels.modelByChannel` surface for channel/group-style conversations, including the Discord channel use case raised in discussion, but the direct-message part is still not wired and the public generated config schema still omits the actual `channels.modelByChannel` schema property.

## What This Changes

The issue requests persistent per-channel, per-group, and per-DM model selection overrides in `openclaw.json`.

## Best Possible Solution

Finish the existing generic `channels.modelByChannel` contract: add stable direct-message candidate resolution from finalized context/session origin data, expose the shared key in the public config schema, document accepted DM key forms, and preserve current precedence where explicit session/runtime model overrides beat config defaults.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: a direct conversation with `channels.modelByChannel.<provider>.<dm-key>` has no DM candidate inputs in the resolver call, while `openclaw config schema` is backed by a generated schema whose `channels.properties` is empty.

## Solution Assessment

Yes. Completing `channels.modelByChannel` is the best direction because the shared config surface already exists for channel/group overrides and avoids adding provider-specific `channels.whatsapp.*.model` or `channels.slack.*.model` shapes.
