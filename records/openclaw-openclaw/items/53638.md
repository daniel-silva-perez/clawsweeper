## Summary

Keep this issue open. Current main has the shared `channels.modelByChannel` mechanism for persistent channel/group model routing, and that covers much of the channel side of the request, but the direct-message part is still not wired: the reply-time resolver is not given Slack or WhatsApp DM identifiers such as `OriginatingTo`, `To`, `From`, native direct user id, or the current direct session key.

## Best Possible Solution

Keep this issue open and finish the generic `channels.modelByChannel` implementation for direct conversations. Extend the resolver input and reply call site so direct peer candidates from session origin/context and current direct session keys can match provider entries, document the supported key forms and wildcard behavior, and add Slack/WhatsApp DM regression coverage while preserving existing channel/group precedence.

- Support and document practical key forms for Slack and WhatsApp DMs, including `user:<id>` for Slack and normalized phone/JID-style identifiers for WhatsApp; keep matching deterministic and conservative.
- Add focused tests for Slack DM and WhatsApp DM model selection, wildcard channel default behavior for DMs, existing group/channel precedence, and explicit session override winning.
