## Summary

Keep open. Current main still resolves compaction to a single provider/model and only retries summarization on that same model; additionally, open PR #74470 uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for compaction/summarization model fallback behavior so Azure content-filter 400s do not permanently retry the same compaction model.

## Best Possible Solution

Keep this issue open while #74470 is reviewed, then land a narrow implementation that tries documented fallback candidates for implicit compaction without mutating persisted session model fields and leaves the broader hard-escalation behavior coordinated with #64962.

## Reproduction Assessment

Yes. A focused regression can stub summarization to throw an Azure-style filtered 400 while model fallbacks are configured; current main resolves one compaction model and retries that same `params.model`, so the fallback path remains absent.

## Solution Assessment

Unclear until #74470 is reviewed. Reusing the session fallback chain for implicit compaction may be the narrowest maintainable repair, but maintainers still need to decide whether the requested explicit compaction fallback config should exist for operator-controlled summarization models.
