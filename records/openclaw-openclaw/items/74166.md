## Summary

Keep open. Current main still hardcodes the Gateway chat-delta broadcast throttle at 150 ms and does not contain the requested environment resolver; the linked implementation PR is closed unmerged, and the related Telegram request is channel-specific rather than a replacement for this shared Gateway WebSocket path. This is narrow code-wise, but it adds a supported core/operator tuning surface, so maintainer product/API approval should happen before automation queues a fix.

## What This Changes

Request to make the Gateway `chat` delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the existing 150 ms default.

## Best Possible Solution

If maintainers accept the knob, add a small centralized resolver with the 150 ms default, strict invalid/negative fallback behavior, and an explicit decision for `0`; then replace the hardcoded threshold, update the fixed-window comments/tests, and document the supported operator surface.

## Reproduction Assessment

Yes. Source inspection and the existing unit-test scenario provide a high-confidence reproduction path: emit two assistant deltas less than 150 ms apart and the later visible text is suppressed until a final/tool-boundary flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, but the best supported surface may be a documented config field plus env precedence rather than an env-only knob.
