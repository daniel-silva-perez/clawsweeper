## Summary

Keep open. Current main still hardcodes the shared gateway chat-delta broadcast throttle at 150 ms, has tests/comments built around that fixed window, and does not contain the requested environment resolver. The linked implementation PR is closed unmerged, and the Telegram streaming-throttle item is adjacent channel-specific work rather than a replacement for this gateway WebSocket surface.

## What This Changes

Request configurable gateway chat-delta broadcast throttling via `OPENCLAW_CHAT_DELTA_THROTTLE_MS`, preserving the 150 ms default and allowing `0` to disable throttling.

## Best Possible Solution

A maintained solution would centralize gateway chat-delta throttle resolution, keep the released 150 ms default, reject invalid/negative input, honor `0` if maintainers accept disabling the throttle, update the fixed-window comments/tests, and document the knob only if it is meant to be a supported operator control.

## Reproduction Assessment

Yes. Source inspection and existing unit tests give a high-confidence reproduction path: emit two assistant deltas less than 150 ms apart and the second delta is buffered rather than broadcast until a flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, but the best public surface for a new gateway tuning knob needs maintainer agreement before an automated fix PR should proceed.
