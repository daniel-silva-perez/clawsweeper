## Summary

Keep open. Current main still hardcodes the gateway chat-delta broadcast throttle at 150 ms, has tests/comments that rely on that fixed window, and does not contain the requested environment resolver. The linked implementation PR is closed unmerged, and the adjacent Telegram streaming-throttle request is channel-specific rather than a replacement for this shared Gateway WebSocket path.

## What This Changes

Request to make the gateway chat-delta broadcast throttle configurable with an optional `OPENCLAW_CHAT_DELTA_THROTTLE_MS` environment variable while preserving the 150 ms default.

## Best Possible Solution

If maintainers accept the operator knob, add a small centralized resolver with the 150 ms default, strict invalid/negative fallback behavior, and an explicit decision for `0`; then update the fixed-window comments/tests and document the knob if it is a supported configuration surface.

## Reproduction Assessment

Yes. Source inspection and existing unit tests provide a high-confidence path: emit two assistant deltas less than 150 ms apart and the second delta is buffered until a final or tool-boundary flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, but the best public surface for a new gateway tuning control needs maintainer agreement before implementation should proceed.
