## Summary

Keep this open. Current main still hardcodes the gateway chat-delta broadcast throttle at 150 ms, the requested `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver/config surface is absent, and the linked implementation PR is closed unmerged. The request is narrow and well evidenced, but it adds an operator-facing runtime knob, so it needs maintainer product/API approval before ClawSweeper should queue an automated fix.

## What This Changes

Request to make the shared gateway chat-delta broadcast throttle configurable through `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the current 150 ms default and honoring `0` as throttle-disabled.

## Best Possible Solution

If maintainers accept the surface, add a small gateway-owned throttle resolver with the 150 ms default, apply it only to the shared chat-delta broadcast path, update the fixed-window comments/tests, and document or log the knob only to the extent OpenClaw wants to support it as an operator control.

## Reproduction Assessment

Not applicable as a feature request, but we do have high-confidence verification of the missing behavior. Current-main source and tests still encode the fixed 150 ms throttle, and repository search confirms the requested env resolver is absent.

## Solution Assessment

Unclear until maintainer product/API review. The proposed env-only resolver is a narrow maintainable implementation, but maintainers should first decide whether this belongs as an env var, config field, documented operator knob, or some combination.
