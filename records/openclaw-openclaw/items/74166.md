## Summary

Keep this issue open. Current main still hardcodes the gateway chat-delta broadcast throttle at 150 ms, has fixed-window test/comment coverage, and does not contain the requested `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver. The linked implementation PR is closed unmerged, and the related Telegram request is adjacent rather than a replacement for the shared gateway WebSocket path.

## What This Changes

The issue requests an operator-configurable gateway chat-delta broadcast throttle via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the existing 150 ms default.

## Best Possible Solution

Keep this open for maintainer product/API review. If accepted, implement a narrow gateway change that preserves the 150 ms default, centralizes non-negative throttle resolution for `OPENCLAW_CHAT_DELTA_THROTTLE_MS`, honors `0`, updates the fixed-throttle comments/tests, and adds docs only if maintainers want the env var treated as a supported operator knob.
