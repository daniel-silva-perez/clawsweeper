## Summary

Keep open. Current main has auto-compaction, pre-compaction memory flush, optional successor transcript rotation, queued-followup retargeting, context-engine compaction hooks, and overflow fallback, but it still does not implement the requested first-class auto session split lifecycle with configurable token/window thresholds, active-run/tool-call boundaries, routing rollover, and continuity payloads for task or confirmation state.

## Best Possible Solution

Keep this issue open as the canonical core product/design tracker. The best implementation is a supported session.autoSplit-style policy built on current compaction, memory flush, successor transcript rotation, and queued-followup retargeting, with configurable token/window thresholds, active run/tool-call boundaries, session-key routing updates, continuity payload handling for summaries/tasks/pending confirmations, docs, and focused regression tests.

## Risks / Open Questions

- Existing compaction, memory flush, context-engine, and successor transcript machinery partially address the operational pain, so any fix should reuse those paths rather than building a parallel session mechanism.
- A context-engine plugin can customize assembly and compaction, but the requested invisible routing/session-key rollover and active-run/tool-call boundary behavior appears to be core session lifecycle work, not something already satisfied by a plugin alone.
