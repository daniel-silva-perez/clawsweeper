## Summary

Keep open. Current main still lacks the requested Gateway preflight surface: the Gateway CLI and service command registrars do not expose it, `gateway run` still proceeds into real server startup, and server startup still binds the listener. Existing fixes cover adjacent config validation, update staging, and the cited context-engine regression, but not a shared no-bind startup validation command.

## What This Changes

The issue requests `openclaw gateway preflight [--json]` to validate config, auth, plugins, context-engine startup paths, and port availability without binding the Gateway port or mutating session state.

## Best Possible Solution

Add a first-class `openclaw gateway preflight [--json]` command backed by a shared startup validation routine that can run the real config/auth/plugin/context-engine resolution paths in a no-bind, no-session-mutation mode, then document the command on the public Gateway CLI page and compose it into safe update flows later.

## Reproduction Assessment

Yes. The absence is reproducible by inspecting current command registration and docs: there is no `gateway preflight` subcommand, and the run path still reaches `startGatewayServer()` and `startListening()` rather than a no-bind report path.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is sound, but the maintainable solution is a refactored shared no-mutation startup validation path, not a parallel schema-only validator or an unsafe full-startup dry run.
