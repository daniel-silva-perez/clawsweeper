## Summary

Keep this PR open for maintainer handling. The config-validation idea is not fully present on current main, but the branch mixes it with already-shipped Slack behavior and a TTS parser change that would silently drop currently supported inline directives.

## What This Changes

The PR centralizes validation known-plugin-ID collection, adds plugin-validation tests, adds Slack Bolt socket-mode `ignoreSelf: false` coverage, and changes TTS directive parsing/tests to require standalone directive lines except compact text blocks.

## Best Possible Solution

Split the useful config-validation change into a narrow rebased PR that uses current manifest and bundled metadata contracts, keeps focused validation tests, preserves current inline TTS directive parsing, and drops the already-shipped Slack hunk unless maintainers want separate test coverage.

## Reproduction Assessment

Yes. A source-level reproduction is a reply like `Hello [[tts:provider=edge]] world` with provider overrides enabled: current main matches and strips the inline directive, while the PR's newline-bounded `directiveRegex` would leave it in the text and ignore the override.

## Solution Assessment

No. Centralizing known-ID resolution may be a reasonable direction, but this mixed branch is not the best implementation because it combines config validation with unrelated Slack and TTS behavior changes; a config-only rebase is safer.
