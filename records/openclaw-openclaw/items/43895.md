## Summary

Keep this PR open. The underlying status bug is still present on current main, but the branch is stale against the current renderer split: it updates the older inline `status.command.ts` Sessions row while current human status output is built through `status-overview-values.ts`.

## What This Changes

The PR derives status session defaults from each agent’s effective model, adds an optional `variesByAgent` marker to status and TUI summary types, renders `varies by agent`, and adds divergent/uniform regression coverage.

## Best Possible Solution

Update the PR against the current status summary/runtime shape: compute `sessions.defaults` across effective configured agents while preserving the lazy `status.summary.runtime` boundary, expose a backward-compatible variation marker through CLI JSON and TUI types, render it in `status-overview-values.ts` and the TUI summary, and cover divergent plus uniform defaults in summary and overview tests.

## Reproduction Assessment

Yes. A config with multiple agents whose `agents.list[].model` values resolve differently will still flow through the current single no-`agentId` summary resolver, while per-agent session rows already have an agent-aware resolver path.

## Solution Assessment

No, not as currently proposed. The intended per-agent default aggregation is the right direction, but the branch must be rebased to update the current overview helper and current runtime boundary before it is mergeable.
