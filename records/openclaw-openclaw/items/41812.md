## Summary

Keep open. The PR targets a real current-main gap, but the branch still has a blocking correctness issue: it adds catalog feedback while bypassing the conflict- and include-aware config mutation path that model commands currently use.

## What This Changes

The branch changes `models set` to load the model catalog, warn when the requested provider/model is absent, and then persist the selected default model.

## Best Possible Solution

Add catalog-miss feedback while preserving the existing `updateConfig`/`replaceConfigFile` mutation path, keep catalog misses as warnings for configured/custom models, and cover both the warning behavior and protected config-write behavior with focused tests.

## Reproduction Assessment

Yes. Current main's source reproduces the user-facing gap because `modelsSetCommand` never loads or checks the catalog; the PR blocker is reproducible by changing config during the awaited catalog load or by exercising an include-owned config section before the direct write.

## Solution Assessment

No. Warning on catalog misses is the maintainable direction because runtime still trusts configured custom models, but the implementation should preserve the base-hash and include-aware write path instead of writing the stale updated object directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the conflict-safe config write path:** `src/commands/models/set.ts:29`
  - body: The patch reads config, awaits catalog loading, then persists `updated` with `writeConfigFile(updated)`. That loses the existing `updateConfig`/`replaceConfigFile` base-hash path, so config edits made while the catalog loads can be projected away when the stale object is written. Keep the catalog check, but perform the mutation through the existing config mutation path or carry the original snapshot/hash into the write.
  - confidence: 0.88
