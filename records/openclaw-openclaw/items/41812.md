## Summary

Keep open. Current main still lacks any catalog feedback in `models set`, and this PR is still useful, but the proposed write path introduces a blocking config-persistence regression by bypassing the existing base-hash and include-aware mutation helper.

## What This Changes

The PR changes `models set` to load the model catalog, warn when the resolved provider/model is absent, and then persist the selected default model through a direct config write.

## Best Possible Solution

Add catalog-miss feedback while preserving the existing `updateConfig`/`replaceConfigFile` mutation path. Keep catalog misses as warnings unless maintainers explicitly choose to reject custom/configured models that runtime still treats as valid.

## Reproduction Assessment

Yes. Current main's source reproduces the original gap because `modelsSetCommand` never loads or checks the catalog; the PR regression is reproducible by changing config during the awaited catalog load or by using an include-owned config section that the direct write cannot route through `replaceConfigFile`.

## Solution Assessment

No. Warning on catalog misses is a reasonable direction, but this implementation is not the best fix because it bypasses the conflict-safe, include-aware config mutation path; the safer fix is to keep the catalog check and perform the final write through `updateConfig` or `replaceConfigFile` with the original snapshot/hash.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the conflict-safe config write path:** `src/commands/models/set.ts:29`
  - body: The new code loads config, awaits catalog loading, then persists the earlier `updated` object with `writeConfigFile(updated)`. That bypasses the existing `updateConfig`/`replaceConfigFile({ baseHash: snapshot.hash })` path, so concurrent config edits made while the catalog loads can be projected away from a stale object, and isolated include writes can be skipped. Keep the catalog warning, but perform the mutation through the existing config mutation helper or carry the original snapshot/hash into `replaceConfigFile`.
  - confidence: 0.88
