## Summary

Keep open. Current main still lacks catalog feedback for `openclaw models set`, so this PR is not obsolete and the linked bug report describes a real DX gap; however, the branch bypasses the existing conflict-safe and include-aware config mutation path, so it should not merge as-is.

## What This Changes

The PR changes `src/commands/models/set.ts` to load the model catalog during `openclaw models set`, warn when the requested provider/model is absent, and then persist the selected default model.

## Best Possible Solution

Add catalog-miss feedback while keeping the actual config mutation on the existing `updateConfig`/`replaceConfigFile` path, and cover both the warning behavior and protected config-write behavior with focused tests.

## Reproduction Assessment

Yes. Current main’s source shows the user-facing gap because `modelsSetCommand` never loads or checks the catalog; the PR blocker is reproducible by changing config between its initial load and later `writeConfigFile`, or by exercising an include-owned `agents` config section.

## Solution Assessment

No. Warning on catalog misses is the maintainable direction because runtime still trusts configured custom models, but the implementation should preserve the base-hash and include-aware write path instead of writing the stale updated object directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the conflict-safe config write path:** `src/commands/models/set.ts:29`
  - body: The patch reads config, awaits catalog loading, then calls `writeConfigFile(updated)` with an object derived from the earlier snapshot. That bypasses `updateConfig`/`replaceConfigFile`'s base-hash and include-aware write handling, so concurrent config edits can be overwritten and include-owned config sections can regress. Keep the catalog check, but perform the mutation through the existing config mutation path or carry the original snapshot/hash through the write.
  - confidence: 0.88
