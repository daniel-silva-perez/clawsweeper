## Summary

Keep open. Current main still lacks catalog feedback in `models set`, so this PR is not obsolete, but the current patch should not merge because it bypasses the existing conflict-safe config mutation path after an async catalog load.

## What This Changes

The PR changes `models set` to load the configured model catalog, warn when the selected provider/model is absent, and then persist the selected default model.

## Best Possible Solution

Add catalog feedback while keeping the existing `updateConfig`/`replaceConfigFile` mutation discipline or an equivalent snapshot/base-hash-aware flow, then cover invalid catalog feedback and concurrent-write safety with focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the conflict-safe config write path:** `src/commands/models/set.ts:29`
  - body: The patch loads config, awaits `loadModelCatalog`, then writes an `updated` object derived from that earlier snapshot with `writeConfigFile(updated)`. Because that write path rereads the current config and computes persistence against the supplied stale object, config changes made while the catalog is loading can be treated as deletions instead of conflicts. Keep this on `updateConfig`/`replaceConfigFile` or carry a validated base snapshot/hash through the catalog check.
  - confidence: 0.88
