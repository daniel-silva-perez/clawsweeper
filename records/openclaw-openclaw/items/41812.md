## Summary

Keep open. Current main still does not add catalog feedback to `models set`, so this contributor PR is not obsolete, but the patch should not merge as-is because it bypasses the existing conflict-safe config mutation path after an awaited catalog load and the reject-versus-warn behavior still needs maintainer decision.

## What This Changes

The PR changes `src/commands/models/set.ts` to resolve the target model, load the model catalog, warn when the provider/model is absent, and then persist that model as the default.

## Best Possible Solution

Implement catalog feedback inside the existing model-command config mutation discipline: preserve source-config and base-hash handling, keep configured/custom/stale-catalog allowlist semantics, decide warning versus rejection explicitly, and add focused tests for catalog and write-safety cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the conflict-safe config write path:** `src/commands/models/set.ts:29`
  - body: The patch reads config, awaits `loadModelCatalog`, then writes an `updated` object derived from that earlier snapshot with `writeConfigFile(updated)`. Because `writeConfigFile` rereads the current config and computes a patch from the latest runtime snapshot to this stale object, any config changes made while the catalog is loading can be treated as deletions. Keep this on the `updateConfig`/`replaceConfigFile` path or carry the validated snapshot/base hash through the catalog check.
  - confidence: 0.88
