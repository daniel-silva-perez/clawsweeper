## Summary

Keep this PR open. Current main still has the exact shared group-policy behavior the PR targets: any configured `groups` entry turns on allowlist mode even when `groupPolicy` is explicitly `"open"`, so an unlisted group can still be rejected through Telegram's shared access path.

## Best Possible Solution

Review and land this PR or an equivalent narrow patch: update `src/config/group-policy.ts` so `hasGroups` only implies allowlist mode when `groupPolicy` is not explicitly `"open"`, preserve implicit allowlist behavior when the policy is unset, add the channel-level and account-level regression tests from the PR, and clarify the public groups/Telegram docs around explicit-open per-group overrides.

## Risks / Open Questions

- The PR changes access-control-adjacent behavior by allowing unlisted groups for configs that explicitly set `groupPolicy: "open"` while also defining specific `groups`; that appears intentional and documented, but maintainers should confirm nobody depends on the older contradictory implicit allowlist behavior.
- Docs currently mix two statements: `groupPolicy: "open"` bypasses allowlists, while configured `groups` keys act as allowlists. The landed change should clarify that explicit open policy treats exact group entries as overrides, while unset or allowlist policy keeps the existing allowlist behavior.
