## Summary

Keep this PR open. Current main still has the shared group-policy behavior the PR targets: any configured `groups` entry turns on allowlist mode even when `groupPolicy` is explicitly `"open"`, so unlisted groups can still be rejected through Telegram's shared access path. The PR is a narrow implementation candidate, not obsolete.

## Best Possible Solution

Review and land this PR or an equivalent narrow patch: update `src/config/group-policy.ts` so `hasGroups` only implies allowlist mode when `groupPolicy` is not explicitly `"open"`, preserve implicit allowlist behavior when the policy is unset, add the channel-level and account-level regression tests, and clarify the public group/Telegram docs around explicit-open per-group overrides.

That still matches the reported failure for explicit `groupPolicy: "open"` plus unrelated `groups` entries.
