## Summary

Keep open: current main still rejects the reported explicit-open/unlisted-group case, so this PR is not obsolete. I would not auto-merge it as-is because the behavior change broadens group admission while current public docs still tell operators that configured `groups` entries are group allowlists.

## What This Changes

This PR changes shared group-policy resolution so explicit `groupPolicy: "open"` prevents `groups` entries from enabling allowlist mode, and adds channel-level and account-level regression tests.

## Best Possible Solution

Maintainers should choose and document one consistent contract. If explicit-open groups are overrides only, land this narrow helper change with matching docs updates for Groups, Telegram, and security guidance while preserving implicit allowlist behavior when `groupPolicy` is unset; otherwise keep the current behavior and clarify that `groups` is always an admission allowlist unless `"*"` is present.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction: with `groupPolicy: "open"`, a non-empty `groups` map makes `allowlistEnabled` true on current main, and an unlisted Telegram group reaches the `group-chat-not-allowed` path rendered as `This group is not allowed.`

## Solution Assessment

Unclear as-is. The code change is the narrowest implementation if maintainers want explicit open to override `groups` admission, but the best complete solution also needs public docs/security guidance updates or an explicit decision to keep `groups` as an allowlist.
