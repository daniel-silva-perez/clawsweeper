## Summary

Keep open: current main still has the reported behavior, and this PR is a narrow implementation candidate, but the patch is incomplete because it changes access-control semantics without updating the operator-facing group-policy and security guidance that still describes configured group entries as allowlists in some places.

## What This Changes

The PR changes shared group-policy resolution so explicit `groupPolicy: "open"` prevents `groups` entries from enabling allowlist mode, and adds channel-level and account-level regression tests.

## Best Possible Solution

Land a complete contract update: either accept explicit-open groups as overrides-only and update the shared docs, Telegram docs, and security guidance accordingly, or keep the current fail-closed interpretation and document that `groups` remains an admission allowlist unless a wildcard route is configured.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction: with `groupPolicy: "open"` and a non-empty `groups` map, current main sets `allowlistEnabled` to true, leaves an unlisted group without `groupConfig`, and Telegram maps that denial to `This group is not allowed.`

## Solution Assessment

No, not as submitted. The code change is the narrowest implementation if maintainers want explicit open to override group admission, but the best complete fix also updates the public group-policy/security docs or explicitly keeps the existing allowlist semantics.
