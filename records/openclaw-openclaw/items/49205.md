## Summary

Keep this issue open. Current main still has the reported gap: `chat.send` writes OpenClaw transcript state and broadcasts Gateway chat events, but I found no Open WebUI visible-history mirror path, no `chat-open-webui-sync` helper, and no code that updates Open WebUI `webui.db`/`chat_message` records after a Control UI send. The issue discussion includes a concrete local fix and verification, but that implementation is not present on main.

## Best Possible Solution

Keep the issue open for maintainer/product design. The best fix is an explicit, supported Open WebUI visible-history sync path after successful external `chat.send` into WebUI-backed UUID sessions, using client id to skip only the real `webchat-ui` source and not `openclaw-control-ui`. If maintainers do not want core to mutate Open WebUI storage directly, define a small supported plugin/API seam and implement the Open WebUI sync outside core, with regression coverage for both OpenClaw transcript context and Open WebUI visible history.
