## Summary

Keep this issue open. Current main still has no supported path to forward an arbitrary inbound OpenAI-compatible HTTP header, such as `custom-token`, into the agent command or expose it to a Python skill. The existing implementation supports fixed OpenClaw routing/control headers plus config-backed skill `env`/`apiKey` injection, which does not solve dynamic per-request credential passthrough.

## Best Possible Solution

Keep this issue open for maintainer product and security design. If OpenClaw should support dynamic caller credentials for skills, add an explicit allowlisted mapping from selected OpenAI-compatible inbound headers into per-run context or per-skill environment, with redaction, per-skill opt-in, sandbox semantics, docs, and gateway/skill tests. If dynamic header passthrough is intentionally unsupported, document the supported alternative: fixed credentials through `skills.entries.<skill>.env` or `skills.entries.<skill>.apiKey`.

## Risks / Open Questions

- Forwarding arbitrary inbound HTTP headers into skill execution is security-sensitive because it could leak caller credentials, gateway auth, cookies, proxy identity headers, or other secrets unless it is explicitly allowlisted, redacted, audited, and scoped per skill/run.
- The report names `/v1/completions`, while current main documents and implements `/v1/chat/completions` and `/v1/responses`; maintainers should clarify the intended compatibility endpoint before designing this feature.
- Current skill env injection is host-process scoped and sandboxed skills do not inherit host `process.env` by default, so any dynamic header-to-skill design needs explicit sandbox behavior and docs.
