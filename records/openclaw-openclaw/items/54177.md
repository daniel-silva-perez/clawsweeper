## Summary

Keep this issue open. Current main still has no supported path to forward an arbitrary inbound OpenAI-compatible HTTP header, such as `custom-token`, into an agent run or expose it to a Python-backed skill. The shipped behavior supports fixed OpenClaw routing/control headers and static config-backed skill `env`/`apiKey` injection, which does not solve dynamic per-request credential passthrough.

## Best Possible Solution

Keep this issue open for maintainer product and security design. If OpenClaw should support dynamic caller credentials for skills, add an explicit allowlisted mapping from selected OpenAI-compatible inbound headers into per-run context or per-skill environment, with redaction, per-skill opt-in, sandbox semantics, docs, and gateway/skill tests. If this is intentionally unsupported, document the supported alternative: fixed credentials through `skills.entries.<skill>.env` or `skills.entries.<skill>.apiKey`.

## Risks / Open Questions

- Forwarding arbitrary inbound HTTP headers into skill execution is security-sensitive: without an explicit allowlist/redaction model it could leak bearer tokens, cookies, trusted-proxy identity headers, or gateway auth material into prompts, logs, sandboxed commands, or child processes.
- The report names `/v1/completions`, while current main documents and implements `/v1/chat/completions` and `/v1/responses`; maintainers should clarify endpoint compatibility before designing the feature.
- Existing static provider/model request headers (`models.providers.*.headers` and `request.headers`) are outbound model-provider configuration and should not be confused with dynamic inbound caller credential passthrough to skills.
