## Summary

Keep open. Current main still trusts a successful native sqlite-vec load without proving that vec0 functions/modules were registered, so the issue's load-success/no-functions failure mode remains valid despite later runtime-deps and fallback-search fixes.

## What This Changes

The issue asks OpenClaw to restore sqlite-vec-backed memory vector indexing/search when sqlite-vec loads successfully but registers no usable vec0 APIs after the v2026.4.11 upgrade.

## Best Possible Solution

Add a narrow sqlite-vec post-load health probe in the memory host SDK loader, keep the source and package loader copies aligned, make memory-core base vector availability on that probe, record an actionable `vector.loadError` when vec0 APIs are absent, preserve existing JSON embedding writes and in-process cosine fallback, and add regression coverage for a native load that returns successfully but leaves vec0 unusable.

## Reproduction Assessment

Yes. The issue has concrete user steps plus two independent Linux confirmations, and current main can be unit-reproduced by making the sqlite-vec loader return ok while `vec0_version` or `CREATE VIRTUAL TABLE ... USING vec0` fails.

## Solution Assessment

No for the suggested dependency rebuild/pin as the first fix: rebuilding sqlite-vec or pinning SQLite is broader dependency work. The narrower maintainable fix is to probe sqlite-vec usability after load and degrade with a clear error when the probe fails.
