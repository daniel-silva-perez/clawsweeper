## Summary

Keep open. Current main still lacks a post-load sqlite-vec usability probe: a successful native load is treated as vector-ready before OpenClaw proves that vec0 functions/modules were registered, so the reported load-success/no-functions failure remains a valid narrow memory bug.

## What This Changes

The issue asks OpenClaw to restore sqlite-vec-backed memory vector search after v2026.4.11 degraded to non-vec behavior when sqlite-vec loads but registers no usable vec0 functions/modules.

## Best Possible Solution

Add a narrow sqlite-vec post-load health probe, base vector availability on that probe, convert missing vec0 functions/modules into an actionable load error, preserve the in-process cosine fallback, and add regression coverage for a load-success/no-functions case.

## Repair Work Prompt

Fix the sqlite-vec load-success/no-functions failure reported in https://github.com/openclaw/openclaw/issues/65156. Add a post-load health probe in the memory host SDK path so `loadSqliteVecExtension` only returns ok when sqlite-vec registered usable vec0 APIs, or returns `ok: false` with an actionable error when `vec0_version`, a temporary `vec0` table, or equivalent probe fails. Update memory-core so `ensureVectorReady` catches probe/table failures, records `vector.loadError`, returns false, preserves chunk writes with JSON embeddings, and lets search use the existing in-process cosine fallback instead of treating the native extension as ready. Keep the fix scoped to the memory host/plugin path; do not change sqlite-vec or SQLite dependency versions unless a maintainer explicitly approves that broader dependency decision. Add regression coverage for a native load that succeeds but leaves vec0 unavailable, plus warning/search behavior coverage as needed. No changelog entry is required unless the final PR changes user-visible behavior or messages beyond the bug fix.
