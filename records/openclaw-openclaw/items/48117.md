## Summary

Keep open. Current main and latest release still run the Windows PowerShell installer through a plain captured `npm.cmd install -g ... --no-fund --no-audit` path and fail on any nonzero npm exit, with no `UNABLE_TO_VERIFY_LEAF_SIGNATURE` handling, CA export, `cafile`, `NODE_EXTRA_CA_CERTS`, or scoped retry. The exact related PR #48416 is closed unmerged, while merged PR #51146 addressed a Linux/nvm startup TLS path and does not cover `scripts/install.ps1`.

## Best Possible Solution

Keep this issue open and implement a maintainer-reviewed Windows installer TLS recovery path in `scripts/install.ps1`. Prefer preserving strict verification with a scoped CA trust repair such as npm `cafile`, `NODE_EXTRA_CA_CERTS`, or Windows certificate export where feasible; only use a scoped `--strict-ssl=false` retry if maintainers explicitly accept that tradeoff. Update installer docs and prove the behavior with a Windows PowerShell smoke or a small mocked installer harness plus deployment verification for the hosted script.
