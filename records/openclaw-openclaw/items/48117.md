## Summary

Keep open. Current main and the latest tagged release still route the Windows PowerShell installer through a plain `npm.cmd install -g ... --no-fund --no-audit` call and fail on any nonzero npm exit, with no handling for the reported npm TLS certificate error. The exact related Windows PR is closed unmerged, while the merged TLS work covers Linux/nvm runtime startup rather than `install.ps1`.

## What This Changes

The issue requests that the Windows PowerShell installer recover from or clearly repair npm `UNABLE_TO_VERIFY_LEAF_SIGNATURE` failures during `iwr -useb https://openclaw.ai/install.ps1 | iex`.

## Best Possible Solution

Add a Windows installer recovery path that detects npm TLS verification failures and preserves certificate verification by default, preferably by guiding or applying a scoped CA trust repair such as `cafile` or `NODE_EXTRA_CA_CERTS`. If maintainers accept an insecure retry, keep it single-install scoped, loud in the output, and covered by installer tests, docs, and a Windows smoke path before closing.

## Reproduction Assessment

Partially. The issue gives a concrete Windows one-liner and npm error, and the failure branch can be reproduced with a focused `Install-OpenClawNpm` mock that returns `UNABLE_TO_VERIFY_LEAF_SIGNATURE`; this review did not live-reproduce the host-specific certificate mismatch on Windows.

## Solution Assessment

Unclear because there is no implemented solution on main. The closed PR’s scoped `--strict-ssl=false` retry is useful prior art, but the safer maintainable direction is to prefer a CA trust repair and reserve insecure retry behavior for explicit maintainer approval.
