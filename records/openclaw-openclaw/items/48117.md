## Summary

Keep open. Current main and the latest tagged release still route the Windows PowerShell installer through a plain `npm.cmd install -g ... --no-fund --no-audit` call with generic nonzero-exit failure handling; the related Linux CA-bundle work does not cover this installer path, and the Windows-specific PR that discussed a retry was closed unmerged.

## What This Changes

The issue requests a fix for the Windows PowerShell installer failing during global npm install with `UNABLE_TO_VERIFY_LEAF_SIGNATURE`.

## Best Possible Solution

Add a Windows installer recovery path that detects npm TLS verification failures and preserves certificate verification by default, preferably by guiding or applying a scoped CA trust repair such as `cafile` or `NODE_EXTRA_CA_CERTS`. If maintainers approve an insecure retry, make it single-install scoped, loud in installer output, documented, and covered by PowerShell installer tests plus a Windows smoke path.

## Reproduction Assessment

No, not end-to-end in this review. The issue provides a concrete Windows one-liner and npm error, and the failure branch can be exercised with a mocked `Invoke-NativeCommandCapture`, but the host-specific Windows certificate-store mismatch was not live-reproduced here.

## Solution Assessment

Unclear because there is no implemented solution on main. The safer maintainable direction is CA-preserving recovery first; the closed PR's scoped `--strict-ssl=false` retry is useful prior art but should need maintainer approval before adoption.
