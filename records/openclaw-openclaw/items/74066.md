## Summary

Keep this issue open. Current main still accepts any non-empty cron expression through the protocol schema, then calls `context.cron.add(...)` / `context.cron.update(...)` without catching Croner parse failures; the outer WS dispatcher still logs uncaught handler errors and maps them to `UNAVAILABLE`. Related PR #52619 was closed unmerged and targeted crash isolation, not the request-boundary `INVALID_REQUEST` behavior described here.

## Best Possible Solution

Keep open and land a narrow gateway-side validation fix: validate Croner-backed `schedule.kind === "cron"` expressions for `cron.add` and `cron.update` before calling the cron service, return `ErrorCodes.INVALID_REQUEST` with a sanitized human-readable `invalid --cron` message, and add regression tests proving invalid field-count and out-of-range expressions do not reach the WS catch-all or persist invalid jobs.
