## Summary

Keep open. Current main still implements the reported Gemini CLI OAuth client-config extraction path, still caches extracted values, and still uses those extracted values as the OAuth client config fallback when explicit env vars are absent. The latest release tag contains the same behavior. The existing warning/confirmation surfaces cover unofficial Gemini CLI OAuth account risk, but they do not specifically disclose or gate reading another Gemini CLI installation's source/bundle files for OAuth client ID and secret. The protect-mcp comment is useful credential-isolation design input, and #41991 is related, but that PR is explicitly diagnostics-only and does not resolve this issue.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The likely code path is the bundled Google plugin: remove automatic Gemini CLI OAuth client-config extraction by default, or gate it behind explicit extraction-specific opt-in/confirmation, then update extensions/google/oauth.credentials.ts, the OAuth flow/tests, provider prompt, and public Google/Gemini provider docs. Review #41991 separately as a diagnostics-only PR, and treat protect-mcp as useful credential-isolation design context rather than a substitute for hardening this shipped behavior.

## Risks / Open Questions

- Removing or gating automatic extraction could break Gemini CLI OAuth onboarding for users who rely on local Gemini CLI client-config discovery.
- The vulnerability classification needs maintainer security/product judgment because the extracted OAuth client config is third-party/user-controlled rather than OpenClaw-owned, but the behavior is bundled and shipped by OpenClaw.
- The protect-mcp comment is relevant credential-vault design input, but adopting a proxy/vault pattern would not by itself remove or disclose the current bundled extraction path.
