## Summary

Keep open. Current main still implements the reported Gemini CLI OAuth client-config extraction path, still caches extracted values for the process, and still uses those extracted values as the fallback when explicit OAuth client environment variables are absent. The latest release tag contains the same behavior. Existing prompts/docs warn about unofficial Gemini CLI OAuth/account risk, but they do not explicitly disclose or gate reading another Gemini CLI installation's source/bundle files for OAuth client ID and secret. The related #41991 PR is diagnostics-only and does not resolve this security/product question.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The likely fix belongs in the bundled Google plugin: remove automatic Gemini CLI OAuth client-config extraction by default, or gate it behind explicit extraction-specific opt-in/confirmation, then update extensions/google/oauth.credentials.ts, the OAuth flow/tests, provider prompt, and public Google/Gemini provider docs. Treat the protect-mcp comment as useful credential-vault design input, but not as a substitute for hardening this bundled behavior; review #41991 separately as a diagnostics-only PR.

## Risks / Open Questions

- Leaving this open is appropriate because the reported behavior is present on current main and the latest release tag checked here.
- Removing or gating automatic extraction could break Gemini CLI OAuth onboarding for users who rely on local Gemini CLI client-config discovery.
- The security classification may need maintainer judgment because SECURITY.md treats trusted installed plugins and third-party/user-controlled credentials differently from OpenClaw-owned secret exposure, but the requested hardening targets bundled OpenClaw behavior.
