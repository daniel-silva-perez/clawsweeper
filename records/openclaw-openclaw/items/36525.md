## Summary

Keep open. Current main still implements the existing sandbox workspace contract: `workspaceAccess: "rw"` selects the agent workspace for `/workspace`, non-rw modes use the sandbox workspace but keep managed writes read-only, and split non-`none` layouts can still expose the agent workspace at `/agent`. The related #46300 PR is narrower writable-`none` work and does not supersede this broader semantics refactor.

## Best Possible Solution

Keep this issue open for an explicit sandbox contract decision. If maintainers accept the refactor, update `ensureSandboxWorkspaceLayout`, Docker/browser/SSH/OpenShell mount builders, filesystem bridge write gates, tests, config/status surfaces, and public sandbox/security docs so `/workspace` consistently represents the session sandbox in sandbox mode and the main workspace is exposed only through documented explicit binds. Treat #46300 and #37634 as narrower writable-`none` follow-ups, not full replacements for this broader contract change.
