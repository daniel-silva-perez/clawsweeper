## Summary

Keep this open. Current main still has normal-path QA gateway cleanup and qa-channel readiness probing, but the specific interruption/orphan ownership and sleep-free restart readiness work requested here is not implemented, and the related implementation PR #65511 is closed unmerged with no open replacement found.

## What This Changes

The issue asks for qa-lab to reliably reap interrupted QA gateway children and temp roots, replace restart sleeps with readiness handshakes, gate qa-channel config on a healthy QA bus, and avoid live incident noise from stale QA runtimes.

## Best Possible Solution

Land a focused qa-lab fix that gives the host suite explicit SIGINT/SIGTERM ownership of spawned gateway process groups and temp roots, replaces restart settling sleeps with concrete gateway and transport readiness checks, and prevents qa-channel activation/config emission from proceeding when the QA bus for that run is unhealthy.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to start the host `pnpm openclaw qa suite` qa-channel lane, interrupt or detach it after the detached gateway child starts, and then inspect the `openclaw-qa-suite-*` temp root and child process group; source inspection shows the host suite lacks matching signal cleanup.

## Solution Assessment

Yes for the issue direction: fixing this inside `extensions/qa-lab` is the narrowest maintainable path because the bug is QA harness ownership/readiness behavior, not a core gateway policy change. The previous broad PR #65511 is not the best current vehicle because it was closed unmerged and mixed this with other timeout and channel work.
