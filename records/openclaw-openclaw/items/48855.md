## Summary

Keep open. Current main still defines Android as a companion-node platform rather than a Gateway host, and `openclaw gateway start` remains a managed service command whose backend registry excludes Android/Termux. The linked AndroidForClaw/HermesApp project is useful external context, but it does not make the OpenClaw platform decision or documentation gap obsolete.

## What This Changes

The issue requests support for running the OpenClaw Gateway directly on Android/Termux so mobile-only users can access the dashboard and remote connections without a separate PC or VPS.

## Best Possible Solution

Decide the platform boundary explicitly: either keep Android node-only and add clear Termux foreground/workaround documentation, or design a supported Android/Termux Gateway-hosting mode with documented lifecycle, auth, networking, dependency, and validation requirements. The current issue should remain the canonical place for that product decision.

## Reproduction Assessment

Yes for the reported service-command failure path: `openclaw gateway start` is registered as a managed service command and the current service resolver only supports darwin/linux/win32. Full live reproduction on Android/Termux was not performed in this read-only review, so device-specific runtime failures remain unverified.

## Solution Assessment

Unclear as an implementation direction. A narrow docs update could address workaround guidance, but actual Android Gateway hosting is a broader platform decision; adding a service adapter without a supported Android lifecycle and security model would be premature.
