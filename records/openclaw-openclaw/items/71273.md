Current main still has the Kimi Coding provider surface and no targeted merged fix for the reported repeated tool-call loop; the concrete report is recent, but the proposed stop-reason root cause is contradicted by current source and tests, so this should not be auto-closed or auto-fixed from that patch alone.

## What This Changes

The issue reports that the Kimi Coding provider repeats identical tool calls after receiving tool results instead of producing a final assistant answer.

## Best Possible Solution

Keep this open until maintainers capture or add a replay fixture for a Kimi Coding turn where tool results are followed by repeated identical tool calls. The end state should be a provider-owned Kimi stream or replay fix with focused regression coverage, or a generic loop-handling repair only if the fixture proves core is reusing stale calls.

Unclear: the issue gives concrete steps and log shape, but this review does not have a current live Kimi transcript or replay fixture proving the failing post-tool-result message shape. Current tests verify that ordinary final assistant text is not blanket-rewritten to tool use.

## Solution Assessment

No: the proposed stop-reason-only patch is not the best proven solution because current main already gates the rewrite on parsed tagged tool-call markup. A transcript-first regression for the Kimi provider is the safer maintainable path.
