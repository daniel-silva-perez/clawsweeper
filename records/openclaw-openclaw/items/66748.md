## Summary

Keep this issue open. Current main still has the reported unguarded exec-event heartbeat wake calls for background exec completion, forced subagent heartbeat routing still resolves to the main session, and the same-author implementation PR with closing syntax remains open rather than merged.

## What This Changes

Suppress exec-event heartbeat wakes from internal subagent and related isolated cron background exec completions so they do not wake the main session or surface spurious HEARTBEAT_OK messages.

## Best Possible Solution

Keep this issue open until #66749, a replacement PR, or the broader #69492 design lands a tested fix that prevents internal subagent and cron-owned exec completions from waking the main heartbeat while preserving user-facing exec completion delivery for normal sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: pass an agent-scoped subagent key through either exec completion path, observe requestHeartbeatNow receive an exec-event wake for that key, then observe resolveHeartbeatSession map that forced subagent key back to the main session.

## Solution Assessment

Unclear as currently proposed. Guarding isSubagentSessionKey is the narrowest fix for the original report, but the latest reporter context suggests isCronRunSessionKey or a more explicit event-audience/internal-session predicate may be needed to cover the same main-session wake-up class safely.
