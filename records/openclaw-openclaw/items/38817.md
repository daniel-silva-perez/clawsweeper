## Summary

Keep open. Current main still strips unsupported tool-schema keywords only when resolved model compat metadata declares them, and the Volcengine Ark/Kimi catalog rows still do not declare `compat.unsupportedToolSchemaKeywords`; the direct fix PRs are closed unmerged and the remaining open Kimi PR tracks a different schema-size/malformed-tool-call problem.

## What This Changes

The issue asks OpenClaw to prevent Volcengine-hosted Kimi and Ark models from receiving unsupported JSON Schema keywords such as `minLength` in tool parameter schemas.

## Best Possible Solution

Add provider-owned `compat.unsupportedToolSchemaKeywords` metadata to the affected Volcengine Ark/Kimi catalog rows, with tests proving the manifest metadata reaches tool normalization; include BytePlus rows only with confirmed matching behavior, and keep core normalization provider-agnostic.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to resolve a current Volcengine Ark/Kimi model row, normalize a tool schema containing `minLength`, and observe that current main leaves the keyword because the row has no compat metadata; the live provider rejection was reported but not rerun here.

## Solution Assessment

Yes. The narrow maintainable solution is to feed the existing generic compat seam from provider catalog metadata, rather than adding Volcengine/Kimi substring detection to core tool normalization.
