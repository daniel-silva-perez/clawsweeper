## Summary

Keep open. Current main still uses a generic compat-driven schema keyword stripping path, but the Volcengine Ark/Kimi catalog rows do not declare the `compat.unsupportedToolSchemaKeywords` metadata needed for that path, and the two direct fix PRs were closed unmerged while the remaining open Kimi PR targets a different schema-size problem.

## What This Changes

The issue requests that Volcengine/Ark-hosted Kimi tool schemas omit unsupported JSON Schema validation keywords such as `minLength` so provider validation does not reject tool calls.

## Best Possible Solution

Add provider-owned `compat.unsupportedToolSchemaKeywords` metadata to the affected Volcengine/Ark/Kimi catalog rows, and include sibling BytePlus rows only with confirmed matching backend behavior, while preserving the existing generic compat-driven normalization path in core.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to select a current Volcengine Ark/Kimi catalog row, normalize a tool schema containing `minLength`, and observe that current main leaves the keyword because the resolved model compat is undefined; the live provider rejection was reported but not rerun here.

## Solution Assessment

Yes. The narrow maintainable fix is provider catalog metadata feeding the existing generic compat seam, not adding Volcengine/Kimi string detection to core normalization.
