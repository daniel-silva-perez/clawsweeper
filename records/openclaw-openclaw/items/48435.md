## Summary

Keep open. Current main still lacks the requested Slack bot-message unfurl controls: the strict Slack config schema/type do not accept the proposed fields, the outbound `chat.postMessage` payload builder never sends Slack unfurl flags, docs/generated metadata do not describe them, and the prior related attempts in the provided context were closed unmerged rather than superseding this request.

## What This Changes

Add Slack config options such as `channels.slack.unfurlLinks` and `channels.slack.unfurlMedia` so bot messages can pass `unfurl_links` and `unfurl_media` through to Slack `chat.postMessage`.

## Best Possible Solution

Implement the feature in the bundled Slack plugin/config surface: add typed and schema-validated `unfurlLinks` and `unfurlMedia` at both top-level Slack config and per-account config, resolve them through the existing account merge path, pass them as `unfurl_links` and `unfurl_media` for normal `chat.postMessage` text/block/follow-up sends, and update Slack docs, generated config metadata/schema, tests, and changelog.

## Reproduction Assessment

Yes. This is a feature gap rather than a runtime crash, and the high-confidence reproduction path is static: the proposed config keys are absent from strict schema/types and the Slack `chat.postMessage` payload lacks the corresponding API arguments on current main.

## Solution Assessment

Yes. A config-driven bundled Slack implementation with top-level inheritance and per-account override is the narrowest maintainable fix; it preserves current behavior when unset and avoids a broader per-message tool/API design that the request does not require.
