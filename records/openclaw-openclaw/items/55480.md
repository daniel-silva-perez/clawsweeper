## Summary

Keep this PR open. Current main still has the reported Discord display-name path: `normalizeDiscordSlug` is ASCII-only, `displayChannelSlug` uses it directly, and a pure non-ASCII channel name can still fall through to guild-space/id labeling. However, this PR is not ready to close or merge as-is because its `GroupSpace` changes would replace stable guild IDs with slugs in policy contexts, while current docs and code support ID-keyed guild policy. A current-main follow-up also needs to account for the shared session display-name sanitizer, which is still ASCII-only.

## Best Possible Solution

Keep the PR open and revise it into a narrower fix: preserve Unicode at the display-label/session-display layer, add regression tests for pure and mixed non-ASCII Discord channel names, and remove or rework the `GroupSpace` slug-first hunks so ID-keyed guild policy remains authoritative.

## Risks / Open Questions

- Merging the PR as submitted could regress Discord guild authorization/tool policy for users whose `channels.discord.guilds` config is keyed by stable guild ID and has no matching `slug` field.
- The PR's preflight fallback alone may not solve the stated session display-name problem on current main because the shared display-name sanitizer still removes non-ASCII characters.
- There is no current regression test proving pure or mixed non-ASCII Discord channel names render correctly in session display names.
