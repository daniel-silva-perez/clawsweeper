## Summary

Keep this PR open. Current main already has raw `media://inbound` support in `loadWebMedia`, but the sandbox media bridge helper still resolves the raw URI through the sandbox bridge first, producing the malformed `media:/inbound` workspace path described by #74123. The PR targets a real, unmerged sandbox-specific gap, though the proposed fallback should get maintainer/security review because the diff appears to rewrite any `media://<authority>/<basename>` to an inbound fallback when a staged file exists.

## Best Possible Solution

Keep this PR open and review it as the focused implementation path for #74123. The best fix is an early `media://inbound/<id>` branch in `resolveSandboxedBridgeMediaPath` that avoids bridge path mangling, uses the staged `media/inbound/<id>` fallback when present, preserves raw URI pass-through only when downstream handling is actually safe, and adds focused helper/tool coverage without changing unrelated sandbox or media contracts.
