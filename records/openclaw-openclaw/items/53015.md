## Summary

Keep open. Current main still has the IRC send/NickServ problems this PR targets, but the PR is not mergeable as-is because its closed-send guard prevents graceful QUIT delivery and its delayed NickServ reclaim timers are not tied to client shutdown.

## What This Changes

The PR changes the bundled IRC plugin client, transient send path, and NickServ tests to surface empty or closed sends, delay transient disconnects, use account-qualified NickServ IDENTIFY, and reclaim fallback nicks with delayed GHOST/NICK commands.

## Best Possible Solution

Merge a repaired IRC plugin change that keeps the useful empty-send visibility, closed-socket detection, transient drain delay, account-qualified IDENTIFY, and fallback-nick reclaim behavior while preserving graceful QUIT delivery, tying delayed reclaim timers to client lifetime, and adding focused client/send coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QUIT when guarding closed sends:** `extensions/irc/src/client.ts:175-177`
  - body: The new `sendRaw` guard rejects whenever `closed` is true, but `quit()` sets `closed = true` before it calls `sendRaw("QUIT ...")`. Every graceful shutdown now throws inside `quit()`, swallows the error, and closes the socket without sending a QUIT frame. Move the closed transition until after QUIT is written, or let the shutdown frame bypass this guard.
  - confidence: 0.94
- **[P2] Tie reclaim timers to client shutdown:** `extensions/irc/src/client.ts:354-369`
  - body: The delayed NickServ GHOST/NICK `setTimeout` calls are not stored, cleared, or unref'd. Once scheduled, they can keep a one-shot IRC send alive after `quit()` and can fire against a client that has already been closed. Track these timers and clear or unref them as part of shutdown.
  - confidence: 0.86
