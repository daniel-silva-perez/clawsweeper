## Summary

Keep this PR open. Current main still has strict Anthropic setup-token validation, so the requested behavior is not implemented, but the proposed patch is not merge-ready because it bypasses length validation for legacy-prefixed tokens and targets a helper that has since moved behind the provider auth surface.

## What This Changes

The PR lowers Anthropic setup-token validation from 80 to 40 characters and accepts tokens without the legacy `sk-ant-oat01-` prefix.

## Best Possible Solution

Confirm the new Anthropic token format, then rework the change in `src/plugins/provider-auth-token.ts` so both accepted formats share an unconditional minimum-length guard, with regression tests for valid and too-short legacy/prefixless inputs and no unrelated auth behavior changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reject too-short legacy setup tokens:** `src/commands/auth-token.ts:33-38`
  - body: The new guard only checks length when the token lacks `sk-ant-oat01-`, so truncated legacy values like `sk-ant-oat01-x` would be accepted and stored instead of failing at paste/setup time. Keep the relaxed prefix handling separate from an unconditional minimum-length check.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The diff does not change CI, dependencies, lockfiles, or release scripts, but it weakens validation on an auth-token input path.

Concerns:

- **[low] Malformed legacy tokens bypass validation:** `src/commands/auth-token.ts:33`
  - body: Because length validation only applies to prefixless tokens, a too-short token with the legacy prefix would pass upfront validation and be stored as a credential, making auth failures harder to diagnose and weakening the existing guardrail.
  - confidence: 0.9
