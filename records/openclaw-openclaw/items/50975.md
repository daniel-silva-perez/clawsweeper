## Summary

Keep open for maintainer review: current main still requires the legacy Anthropic setup-token prefix and 80-character minimum, while this PR's relaxation introduces a concrete validation regression by letting too-short legacy-prefixed tokens pass. The prior related token-format discussion also shows maintainer caution about accepting unconfirmed new formats.

## What This Changes

The PR changes Anthropic setup-token validation to lower the minimum length from 80 to 40 and accept prefixless tokens in `src/commands/auth-token.ts`.

## Best Possible Solution

Confirm the accepted Anthropic setup-token formats first; if the relaxation is valid, update the shared validator in `src/plugins/provider-auth-token.ts` with a format-agnostic but unconditional minimum-length guard, keep the Anthropic runtime and live helper in sync, and add focused validator unit coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reject too-short legacy setup tokens:** `src/commands/auth-token.ts:33-38`
  - body: The new guard only checks length when the token lacks `sk-ant-oat01-`, so truncated legacy values such as `sk-ant-oat01-x` would be accepted and stored instead of failing at paste/setup time. Keep prefix relaxation separate from an unconditional minimum-length check.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The patch weakens validation on an auth-token input path by allowing malformed legacy-prefixed tokens through.

Concerns:

- **[low] Malformed legacy tokens bypass validation:** `src/commands/auth-token.ts:33`
  - body: Because length validation only applies to prefixless tokens, a too-short value with the legacy Anthropic prefix would pass upfront validation and be stored as a credential, making auth failures harder to diagnose and weakening the existing guardrail.
  - confidence: 0.9
