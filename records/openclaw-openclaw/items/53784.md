## Summary

Keep open. Current main still routes ordinary MP3/WAV/M4A-style Feishu sends as regular files unless explicit voice intent triggers transcoding, so this PR is not obsolete; however the branch is not merge-ready because it changes the default Feishu audio contract while leaving docs and lockfile/package state out of sync.

## What This Changes

The PR broadens Feishu outbound audio detection to treat common audio extensions and audio MIME types as playable audio messages, maps them to `file_type: "opus"`, adds `music-metadata` duration parsing, and updates Feishu media tests/package metadata.

## Best Possible Solution

Decide the Feishu audio contract explicitly. If direct playable bubbles for common audio formats are desired, land a rebased narrow change that updates routing, duration handling, tests, docs, package metadata, and lockfile together; otherwise keep the shipped explicit voice-transcode path and close or supersede this branch with that policy stated.

## Reproduction Assessment

Yes. Current main has a focused Feishu media test that sends `song.mp3` with `audio/mpeg` and expects `file_type: "stream"` plus `msg_type: "file"`; the routing code explains the same behavior by limiting native audio to Ogg/Opus.

## Solution Assessment

Unclear as-is. The PR's live Feishu API and community-plugin evidence make the direction plausible, but it is not the best merge shape until the default-audio policy, docs, lockfile, and rebase conflicts are resolved.
