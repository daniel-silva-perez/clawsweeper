## Summary

Keep open. Current main still preserves the behavior this PR is trying to change: ordinary MP3/audio-mpeg media routes as a Feishu file unless explicit voice intent triggers transcoding. The PR has useful live-API and community-plugin context, but it is not ready to merge because the new dependency is not lockfile-aligned, user-facing Feishu docs would become wrong, and maintainers still need to decide the default-audio contract.

## What This Changes

The PR changes Feishu outbound media routing so common audio extensions and audio MIME types upload as `file_type: "opus"` and send as `msg_type: "audio"`, adds `music-metadata` duration parsing, and updates Feishu media tests/package metadata.

## Best Possible Solution

Decide the Feishu outbound audio contract explicitly. If direct playable bubbles for common formats are desired, land a rebased narrow change that updates routing, duration handling, tests, docs, package metadata, and lockfile together; otherwise keep the shipped explicit `audioAsVoice`/`asVoice` transcode path and close or supersede this branch with that policy stated.

## Reproduction Assessment

Yes. Current main has both code and a focused Feishu media test showing `song.mp3` with `audio/mpeg` uploads as `stream` and sends as `file` unless `audioAsVoice` is set.

## Solution Assessment

Unclear as-is. The direct common-audio routing may be viable based on the PR's live API and community-plugin evidence, but the patch is not the narrowest maintainable merge until docs, lockfile, stale-branch state, and the default-audio policy are resolved.
