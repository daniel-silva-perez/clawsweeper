## Summary

Keep open for maintainer review. Current main still routes ordinary MP3/WAV/M4A-style Feishu audio as file attachments unless explicit voice delivery requests transcoding, so this PR is not obsolete; the author also addressed the Greptile naming feedback and supplied useful live-API/community-plugin context. The proposed branch still needs a maintainer decision on Feishu's default audio contract plus docs, lockfile, and rebase alignment before it is mergeable.

## What This Changes

The PR broadens Feishu media routing so common audio extensions and audio MIME types upload as `file_type: "opus"` and send as `msg_type: "audio"`, adds `music-metadata` duration parsing, and updates Feishu media tests/package metadata.

## Best Possible Solution

Decide the Feishu audio contract explicitly. If direct playable bubbles for common audio formats are desired, land a rebased narrow change that updates routing, duration handling, tests, docs, package metadata, and lockfile together; otherwise keep the shipped `audioAsVoice`/`asVoice` transcode path and close or supersede this branch with that policy stated.

## Reproduction Assessment

Yes. Current main has both source and a focused Feishu media test showing `song.mp3` with `audio/mpeg` is sent as `file_type: "stream"` and `msg_type: "file"` unless explicit voice intent is set.

## Solution Assessment

Unclear as-is. The PR's direction is plausible and supported by useful external evidence, but it is not the best merge shape until maintainers choose the default-audio policy and require docs, lockfile, and rebase alignment.
