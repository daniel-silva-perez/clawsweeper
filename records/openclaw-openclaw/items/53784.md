## Summary

Keep open. Current main still routes ordinary MP3/WAV/M4A-style Feishu media as file attachments unless explicit voice delivery requests transcoding, so this PR is not obsolete. It should not merge as-is because it changes the documented Feishu audio default, adds a runtime dependency without lockfile alignment, and needs a rebase/product decision against the newer Feishu package state.

## What This Changes

The PR broadens Feishu outbound audio routing to treat common audio extensions and audio MIME types as playable Feishu audio messages, maps them to `file_type: "opus"`, adds `music-metadata` duration parsing, and updates Feishu media tests/package metadata.

## Best Possible Solution

Decide the Feishu audio contract explicitly. If direct playable bubbles for common formats are desired, land a rebased narrow change that updates routing, duration handling, tests, docs, package metadata, and lockfile together; otherwise keep the shipped explicit voice-transcode path and close or supersede this branch with that policy stated.

## Reproduction Assessment

Yes. Current main has both code and a focused Feishu media test showing `song.mp3` with `audio/mpeg` uploads as `stream` and sends as `file` unless `audioAsVoice` is set.

## Solution Assessment

Unclear as-is. Mapping common audio formats to Feishu `file_type: "opus"` may be viable based on the PR's live API/community-plugin evidence, but it is not the narrowest maintainable merge until docs, lockfile, rebase conflicts, and the default-audio policy are resolved.
