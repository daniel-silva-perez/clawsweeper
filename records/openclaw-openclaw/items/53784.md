## Summary

Keep this PR open. Current main does not implement the PR's central behavior: ordinary MP3/WAV/M4A/FLAC/AAC media still routes as file attachments unless the caller explicitly requests voice delivery, and duration is not computed before Feishu upload. The current main path instead added a more conservative `audioAsVoice`/`asVoice` transcode flow, so this PR needs maintainer review or revision rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review or ask the contributor to rebase it onto the current `audioAsVoice` design. The best implementation path is likely to preserve ordinary attachments as files unless product owners decide otherwise, use the existing voice-intent transcode path for playable Feishu audio bubbles, and add duration computation through an approved shared media-runtime seam with focused Feishu tests for MP3/WAV/M4A/FLAC/AAC voice delivery.

## Risks / Open Questions

- The PR's approach uploads common formats as Feishu `file_type: "opus"` without transcoding, while current docs say native Feishu audio requires Ogg/Opus; maintainers should confirm whether that live API behavior is an accepted product contract before landing.
- Adding `music-metadata` to the Feishu plugin broadens the runtime dependency and parses untrusted media buffers; a revised PR should include the lockfile/importer update and dependency review.
- Current main already has a voice-intent transcode path, so a direct merge could conflict with the newer `audioAsVoice` design or change ordinary music/audio attachments into voice bubbles unexpectedly.
