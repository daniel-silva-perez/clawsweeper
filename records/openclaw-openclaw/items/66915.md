## Summary

Keep open. Current main still suppresses ordinary verbose-off `edit` mismatch details into the generic mutating-tool warning, while the lower edit layer has richer raw mismatch context that is not safe to surface directly. The config-side disable path already exists through `tools.deny`, and open PR #67276 is the explicit unmerged implementation candidate for the remaining user-facing warning fix.

## What This Changes

The issue requests clearer user-facing `edit` exact-text mismatch errors and a way for reliability-focused deployments to disable `edit` in favor of `write`.

## Best Possible Solution

Review PR #67276 as the focused fix path. The desired end state is a sanitized payload-layer summary for exact-text `edit` mismatches in verbose-off output, unchanged `edit` semantics, existing `tools.deny` policy as the disable-edit path, and broader whitespace normalization or read-positioning ideas handled separately under #62863.
