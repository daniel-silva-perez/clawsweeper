## Summary

Keep this PR open for maintainer review, but do not merge the current patch. Current main does not already implement the requested assemble hooks or AICM plugin, and the proposed diff has blocking correctness and security issues around unsafe transcript file access, non-chained hook transformations, and AICM build/runtime packaging.

## What This Changes

The PR adds `before_assemble` and `after_assemble` plugin hook types and runner methods, wires them into the Pi embedded runner prompt path, and adds a bundled `aicm` plugin package with tools, audit helpers, runtime files, and a lockfile importer entry.

## Best Possible Solution

If maintainers want this seam, split it into a narrow, typed, documented context-assembly hook with contract tests and request-scoped data flow. Keep AICM as an external ClawHub/plugin package or a separate owner-reviewed bundled-plugin proposal, and do not let plugin tools mutate transcript files by guessing filesystem paths.

## Reproduction Assessment

Yes. Static review against current main and the PR diff gives a high-confidence reproduction path for the blockers: main lacks the hooks, the diff shows the unsafe hard-coded transcript path, and the existing hook runner passes the original event to each modifying handler.

## Solution Assessment

No. A generic runtime hook may fit OpenClaw's plugin direction, but this patch combines public API design with a large bundled plugin and unsafe session-file mutation; the safer path is a smaller core hook contract first, with AICM reviewed separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope AICM file access to the active session:** `extensions/aicm/index.ts:28`
  - body: The plugin selects transcript files from a hard-coded `/workspace/projects/agents/main/sessions` directory and stores deletes under a shared `current` key. In multi-agent, non-default workspace, or packaged installs, an LLM-callable AICM tool can read or rewrite the most recently modified main session rather than the session invoking the tool. Use the current hook/session context and scoped storage instead of scanning a global path.
  - confidence: 0.93
- **[P2] Fix cascade IDs to match session message IDs:** `extensions/aicm/index.ts:185-193`
  - body: `SessionMessage.id` and `parentId` are typed as strings, but `findCascadeGroup` uses `number` and `Set<number>`, and `performDelete` passes `t.id` into it. With the new package's `strict` tsconfig, this does not typecheck; make the cascade id type match the parsed message ids before adding the extension.
  - confidence: 0.9
- **[P2] Pass before_assemble results to the next handler:** `src/plugins/hooks.ts:1326-1328`
  - body: This merge function only keeps the latest returned `messages`, while `runModifyingHook` still invokes every handler with the original event. If plugin A removes messages and plugin B returns a reorder based on the original list, plugin B can reintroduce plugin A's removals, so the promised sequential chaining does not work.
  - confidence: 0.88
- **[P2] Mark the AICM package as ESM:** `extensions/aicm/package.json:13`
  - body: The new package points at `dist/index.js` generated from ES2022 modules, but its nearest `package.json` does not declare `"type": "module"`. Packaged runtime loading will treat those `.js` files as CommonJS and fail on ESM syntax; existing bundled plugin packages all declare module type.
  - confidence: 0.78
