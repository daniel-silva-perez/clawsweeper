## Summary

Keep this PR open for maintainer handling, but do not merge it as-is. Current main does not already implement these hooks, and the linked AICM plugin is useful evidence of demand, but the proposed public hook API exposes transcript mutation and observation without the existing hook-policy gates and has concrete semantic bugs in the runner wiring.

## What This Changes

The PR adds typed `before_assemble` and `after_assemble` plugin hooks, wires them into the Pi embedded runner around context assembly, re-exports their event/result types, and leaves an `extensions/aicm` workspace importer in `pnpm-lock.yaml`.

## Best Possible Solution

Define the assembly-extension contract as a narrow, guarded plugin API: decide its relationship to the existing context-engine slot, classify transcript mutation/observation under explicit policy gates, document ordering and payload authority, add SDK/docs/contract tests, pass accumulated message state through sequential handlers, expose the actual model input for observation, and remove unrelated lockfile drift before merge.

## Reproduction Assessment

Yes. Static review is enough: current main lacks the hook names, and the PR diff shows the hooks being added without policy classification, with non-chained modifying-event input, and with `after_assemble` observing only messages rather than the full model submission.

## Solution Assessment

No. A guarded assembly extension point may be a reasonable core direction, but this patch is not the best implementation because it bypasses existing safety gates and does not accurately implement the stated sequential or final-context semantics.

Full review comments:

- **[P1] Gate assembly hooks through hook policies:** `src/plugins/hook-types.ts:106-107`
  - body: The new hooks expose or mutate full conversation messages, but the diff only adds them to the generic hook-name list. Because they are not classified with the prompt-injection and conversation-access hooks, a non-bundled plugin can register raw transcript hooks without the current `allowPromptInjection` and `allowConversationAccess` gates.
  - confidence: 0.93
- **[P2] Pass accumulated messages to each assembly handler:** `src/plugins/hooks.ts:1321-1324`
  - body: `runBeforeAssemble` promises chained transformations, but `runModifyingHook` calls every handler with the original event. A later plugin therefore sees the unmodified message list and can undo an earlier plugin's deletion or reorder when it returns its own `messages`.
  - confidence: 0.9

  - body: The `after_assemble` payload passes only `activeSession.messages`, while the request sent to the model also includes prompt text, system/runtime context, and optional images. Hook users will not observe the final context the PR says is about to be sent to the LLM.
  - confidence: 0.84

  - body: The force-pushed PR no longer contains an `extensions/aicm` package, but the lockfile still adds `extensions/aicm: {}` as a workspace importer. That leaves package metadata inconsistent with the workspace package set and should be removed from this PR.
