## Summary

Keep this PR open. The reported WhatsApp reconnect failure is still present on current main and in the latest stable tag, but the branch is stale: it edits the old `src/web` layout while the shipped WhatsApp reconnect code now lives under `extensions/whatsapp/src`.

## What This Changes

The PR changes the old WhatsApp Web reconnect loop to default to unlimited retries and use a degraded retry cooldown instead of permanently stopping after a retry cap.

## Best Possible Solution

Ship the reconnect change in the WhatsApp plugin: make the default retry policy long-lived, keep retryable disconnects in an explicit degraded retry mode after bounded caps, preserve terminal auth/conflict exits, and align tests plus config help around the chosen `maxAttempts: 0` semantics.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: use the default or any positive `web.reconnect.maxAttempts`, force retryable WhatsApp closes until the cap, and the controller returns `stop` while the monitor shuts down and breaks.

## Solution Assessment

No, not as written. The behavioral direction is useful, but the branch edits obsolete `src/web` files; the maintainable fix is to refresh the logic into `extensions/whatsapp/src` and update the plugin tests/config docs there.
