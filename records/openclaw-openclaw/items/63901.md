## Summary

Keep open. Current main still does not implement the central request: dynamic/session-aware provider request metadata for OpenRouter/OpenAI-compatible API calls. Static `extra_body` pass-through and native OpenAI Responses metadata exist, but provider extra-param/body hooks still do not receive `sessionKey` or `sessionId`, the embedded runner does not pass session identity into those hooks, and OpenRouter does not inject per-session `user` or `trace` fields. The supplied issue context also shows open related implementation candidates, including #54725 and #64813, so this issue should remain open for maintainer review rather than be closed as implemented.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review and either land or explicitly close #54725 and #64813, converging on a narrow opt-in provider attribution design with tests for `sessionKey`, `sessionId`, agent/static `extra_body` precedence, OpenRouter/LiteLLM-compatible `user`/`trace` behavior, and privacy-safe configuration semantics.

## Risks / Open Questions

- Static `extra_body` can already handle hard-coded provider metadata, so the remaining product gap should stay narrowly framed as dynamic/session-aware attribution and/or explicit `user` pass-through.
- Any automatic `user` or `trace` forwarding should be opt-in and documented because session keys may contain channel, peer, agent, tenant, or task identifiers that would be sent to third-party providers.
