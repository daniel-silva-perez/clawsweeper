## Summary

Keep open. Current main still does not implement session-aware provider request metadata: it supports static OpenAI-compatible `extra_body` pass-through and adjacent transport/header observability, but the provider extra-param and stream-wrapper contexts do not receive `sessionKey` or `sessionId`, and the OpenRouter wrapper does not inject dynamic `user` or `trace` request-body fields. Related open PRs #54725 and #64813 cover different pieces of the remaining design, so this should stay open for maintainer review rather than be swept closed.

## What This Changes

Request to add configurable or automatic provider request metadata, especially OpenRouter/OpenAI-compatible `trace` and `user` fields, so shared provider accounts can attribute costs by graph, task, or agent session.

## Best Possible Solution

Keep the issue open until maintainers choose a privacy-safe attribution design: likely an additive provider SDK/runtime seam for sanitized session attribution, OpenRouter/LiteLLM-compatible `user` and `trace` handling where appropriate, clear precedence against static `extra_body`, focused payload-shape tests, and public docs that make third-party metadata forwarding explicitly opt-in.

## Risks / Open Questions

- The feature is privacy-sensitive because session keys can encode channel, sender, task, or workspace identifiers; implementation needs explicit opt-in, sanitization, and precedence rules before forwarding that data to third-party providers.
- Static `extra_body` already covers hard-coded metadata, so the remaining work should stay focused on dynamic/session-aware attribution rather than duplicating existing pass-through behavior.
- The two related open PRs solve different slices of the problem; either one alone may still leave `trace`, OpenRouter-specific behavior, or provider SDK API shape unresolved.
