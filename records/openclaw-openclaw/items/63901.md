## Summary

Keep open. Current main still does not implement session-aware provider request metadata: it supports static `extra_body` pass-through for OpenAI-compatible request bodies, but the provider extra-param and stream-wrapper contexts do not receive `sessionKey`/`sessionId`, and the OpenRouter plugin does not inject per-session `user` or `trace` fields. The related open PRs #64813 and #54725 should be reviewed before closing this issue.

## What This Changes

Request support for passing provider request metadata such as `trace` or `user`, especially session-aware graph/task/agent attribution for OpenRouter and other OpenAI-compatible providers.

## Best Possible Solution

Keep the issue open until maintainers choose a privacy-safe attribution design. The likely end state is an additive provider SDK seam for session identity or a sanitized attribution object, plus OpenRouter/LiteLLM-compatible `user`/`trace` handling, focused tests for precedence and payload shape, and docs that make third-party metadata forwarding explicitly opt-in.

## Risks / Open Questions

- Static `extra_body` already supports hard-coded metadata, so the remaining work should stay focused on dynamic/session-aware attribution rather than broad arbitrary payload injection.
- Forwarding session keys can expose channel, peer, task, tenant, or agent identifiers to third-party providers, so any automatic `user` or `trace` behavior should be opt-in and documented.
- The two open PRs solve different parts of the request: #64813 covers configured `user` pass-through, while #54725 exposes session identity to provider hooks.
