## Summary

Keep open. Current main still only supports static `extra_body` pass-through and provider routing/thinking wrappers; it does not pass session identity into provider extra-param contexts or inject OpenRouter/OpenAI-compatible `user` or `trace` request metadata. The related open PRs cover overlapping parts of the design, so maintainer review should happen before this is swept closed or turned into another automated repair.

## What This Changes

Request to add provider request metadata support so OpenRouter/OpenAI-compatible calls can carry trace/user fields for per-graph, per-task, or per-agent cost attribution.

## Best Possible Solution

Keep the issue open until maintainers choose a privacy-safe attribution design: an additive provider SDK/runtime seam for sanitized session attribution, OpenRouter/OpenAI-compatible `user` and `trace` handling, explicit precedence with static `extra_body`, focused payload-shape tests, and docs that make third-party metadata forwarding opt-in.

## Reproduction Assessment

Yes. Static inspection of current main gives a high-confidence reproduction path: configure an OpenAI-compatible provider without static `extra_body` and there is no code path from `sessionKey`/`sessionId` into provider extra params, OpenRouter wrapping, or the request body.

## Solution Assessment

Unclear. Option B is a small implementation, but automatically forwarding raw session keys is privacy-sensitive; the safer solution is a maintainer-approved opt-in attribution seam with sanitization and precedence rules rather than hard-coding `user = sessionKey`.
