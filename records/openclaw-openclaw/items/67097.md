## Summary

Keep this issue open. Current main still generates `models.json` by resolving plugin-owned implicit provider catalogs before mode-specific handling, and the Codex and Arcee catalog paths named in the report are still present. Related #66957 covers the narrower `models.mode="replace"` ordering bug, but this issue also asks for a broader maintainer decision about whether generated manifests should include bundled provider catalogs without explicit onboarding or config intent.

## Best Possible Solution

Keep this issue open for maintainer judgment on the generated `models.json` contract. A concrete path is to fix or track the `models.mode === "replace"` short-circuit under #66957, then separately decide whether merge/default generated manifests should document provider-owned bundled catalogs as intentional or gate Codex/Arcee catalog hooks on explicit activation, onboarding, auth, or config intent.

## Risks / Open Questions

- Closing this as a duplicate of #66957 would lose the broader product/API question about merge/default generated `models.json` including bundled provider catalogs without explicit onboarding/config intent.
- A fix should keep generated `models.json` behavior separate from read-only `models list --all`, where docs now intentionally allow provider-owned static catalog rows.
- Codex and Arcee are not identical paths: Codex has synthetic app-server auth plus fallback models, while Arcee depends on Arcee/OpenRouter auth or configured catalog rows. A blanket removal could regress intended provider-plugin catalog behavior.
