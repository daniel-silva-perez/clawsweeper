## Summary

Keep open. Current main has better sessions_yield bookkeeping, but it still does not implement the requested parent-session progress/stream bridge. Native subagent spawns still ignore streamTo, sessions_yield ends the parent turn, Control UI clears the active stream on the terminal lifecycle event, and subagent-yielded runs are recorded as paused rather than forwarded as live progress to the requester session.

## Best Possible Solution

Keep the report open and implement a generic parent-session waiting/progress seam for yielded subagent orchestration: when a registered child is active or paused after sessions_yield, emit requester-scoped lifecycle/progress events that Control UI and TUI can render as an active waiting state, and separately decide whether native runtime=subagent should support a streamTo="parent"-style relay. The implementation should preserve the current paused-yield correctness from #73413 and avoid raw token-delta delivery to external channels.

## Risks / Open Questions

- Closing now would conflate task/session bookkeeping with the requested parent-session UX; users can inspect subagent/session state elsewhere, but the parent chat/TUI still has no generic live waiting indicator after sessions_yield terminates the parent run.
- Forwarding raw child token streams to external messaging surfaces would conflict with the repo's streaming guidance, so the fix should prefer a generic lifecycle/progress event or an explicit native subagent streamTo contract with surface-specific rendering.
