## Summary

Keep this PR open. Current main does not implement the proposed `update_plan` loop normalization, and the PR still has a concrete behavior regression: it collapses all non-empty plan step text to presence bits, so distinct plans with the same length/status pattern can hash as identical no-progress updates and trip the global breaker.

## What This Changes

The PR normalizes `update_plan` call parameters and outcome details before loop hashing, and adds regression tests for wording-only plan churn and structural status progress.

## Best Possible Solution

Keep a special `update_plan` loop signature, but distinguish material plan changes from cosmetic churn. A maintainable fix would preserve normalized or digested step content, or use a narrower canonicalization rule that collapses tiny formatting/explanation churn without making unrelated plans collide.

## Reproduction Assessment

Yes. A source-level reproduction is clear: record enough `update_plan` calls whose steps change text but keep the same status pattern, then call `detectToolCallLoop`; with the PR normalization those calls share the same args/result shape and can reach the global breaker.

## Solution Assessment

No. The proposed direction addresses the real wedge, but the current implementation over-collapses plan content; a safer solution should keep a content signal for each step while still ignoring purely cosmetic fields.
