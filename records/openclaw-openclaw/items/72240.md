## Summary

Keep open. Current main still preserves SIGKILL/timeout metadata internally but does not expose enough of it through foreground exec and process polling results for the macOS diagnostic gap reported here; adjacent open reports and PRs do not provide a merged current-main fix for this user-facing surface.

## What This Changes

The issue requests richer user-facing diagnostics when macOS exec/process commands are killed with SIGKILL, including timeout, cancellation, supervisor or OS cause metadata, output tails, and log correlation.

## Best Possible Solution

Add a focused diagnostics contract improvement that propagates existing supervisor/runtime metadata through foreground exec and background process results: exitSignal, failureKind, exitReason or terminationReason, timedOut, noOutputTimedOut, timeout/no-output limits when available, run/session/log correlation, and bounded output tails. Preserve current kill/escalation policy in this repair and leave graceful SIGTERM/SIGKILL behavior to #66399 unless maintainers explicitly combine the work.

## Reproduction Assessment

Unclear: the intermittent macOS SIGKILL incident is credible but not deterministic from the report. The missing user-facing metadata is source-reproducible by following simulated signal, timeout, no-output-timeout, cancellation, foreground-result, and process.poll paths on current main.

## Solution Assessment

Yes for the first repair step: enriching exec/process result metadata is the narrowest maintainable fix for the observable diagnostics gap. Broader signal escalation, OS memory diagnosis, or broad-scan guardrails should remain separate unless maintainers expand the scope deliberately.
