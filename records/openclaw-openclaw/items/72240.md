## Summary

Keep open. Current main still keeps useful SIGKILL, timeout, and failure-kind metadata inside the exec runtime and supervisor, but foreground exec results and process polling do not expose enough of it to solve the user-facing macOS diagnostic gap. The related open reports and PR are adjacent rather than a merged fix for this surface.

## What This Changes

The issue requests richer user-facing diagnostics when macOS exec/process commands are killed with SIGKILL, including timeout/cancellation cause, output tails, and log or run correlation.

## Best Possible Solution

Add an additive diagnostics contract improvement that propagates existing supervisor/runtime metadata through foreground exec and background process results: exitSignal, failureKind, exitReason or terminationReason, timedOut, noOutputTimedOut, relevant timeout limits, run/session/log correlation, and bounded output tail context. Keep graceful SIGTERM/SIGKILL policy changes in #66399 unless maintainers intentionally expand the scope.

## Reproduction Assessment

Unclear: the intermittent macOS SIGKILL incident is not deterministic from the report. The missing user-facing metadata is source-reproducible on current main by tracing simulated signal, timeout, no-output-timeout, cancellation, foreground-result, and process.poll paths.

## Solution Assessment

Yes for the first repair step: enriching exec/process result metadata is the narrowest maintainable fix for the observable diagnostics gap. Broader signal escalation, OS memory diagnosis, or broad-scan guardrails should remain separate unless maintainers deliberately combine the work.
