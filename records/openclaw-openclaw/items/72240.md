## Summary

Keep open. Current main still has a real user-facing diagnostics gap: exec/runtime and supervisor layers classify signal, timeout, and failure-kind data, but foreground exec results and process polling still drop key termination metadata that would distinguish macOS SIGKILL causes. The related open reports and PR are adjacent and useful, but they are not a merged fix for this issue’s broader result-metadata surface.

## What This Changes

The issue requests richer user-facing diagnostics when macOS exec commands are hard-killed with SIGKILL, including termination metadata, output tail context, and log/run correlation.

## Best Possible Solution

Add an additive diagnostics contract improvement that propagates existing supervisor/runtime metadata through foreground exec and background process results: exitSignal, failureKind, exitReason or terminationReason, timedOut, noOutputTimedOut, timeout limits, and bounded output tail context. Keep signal escalation policy and broad recursive-scan guardrails as separate follow-ups unless maintainers intentionally expand scope.

## Reproduction Assessment

Unclear for the intermittent macOS SIGKILL incident itself; the report does not provide a deterministic local repro. Yes for the user-facing bug class: current main can be source-traced through simulated signal, timeout, no-output-timeout, cancellation, foreground-result, and process.poll paths where available metadata is dropped from result details.

## Solution Assessment

Yes for the first repair step: enriching exec/process result metadata is the narrowest maintainable fix for the observable diagnostics gap. It avoids changing kill behavior while still giving users enough structured evidence to distinguish timeout, supervisor cancellation, and signal-killed outcomes.
