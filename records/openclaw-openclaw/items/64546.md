## Summary

Keep open. Current main is still affected, and this PR is the active linked security fix candidate, but the proposed fallback secret is random and process-local, so valid Mattermost button callbacks can fail across separate CLI/gateway processes or after a restart unless users preconfigure a persistent secret.

## What This Changes

The PR adds optional Mattermost interaction-secret config and changes interaction token signing from bot-token-derived HMAC secrets to configured or randomly generated per-account secrets across Mattermost code, tests, schema, and types.

## Best Possible Solution

Use a per-account interaction signing secret that is not derived from the bot token and is persisted/shared across CLI sends, gateway verification, and restarts, with SecretRef or credential-store support plus schema/types/docs/setup updates and regression coverage for forgery resistance and cross-process/restart verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Persist the fallback interaction secret across processes:** `extensions/mattermost/src/mattermost/interactions.ts:192-194`
  - body: `ensureInteractionSecret` now creates a random in-memory secret when `interactions.secret` is unset. `sendMessageMattermost` can sign buttons in one process while the gateway monitor verifies them in another, and restarts also lose the generated value, so valid callbacks will be rejected as invalid unless users preconfigure a secret. The fallback needs to be persisted/shared or the configured secret must be required for buttons.
  - confidence: 0.92

## Security Review

  - body: The PR replaces deterministic bot-token derivation with a random fallback, but that fallback is only stored in process memory. Because button signing and verification can occur in different processes or across restarts, the callback authorization boundary is not stable unless a persistent configured secret exists.
