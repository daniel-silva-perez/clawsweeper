## Summary

Close as implemented on current main. Active Memory now registers `before_prompt_build` with a hook timeout derived from its own `timeoutMs` config plus setup grace, and the generic hook runner now prefers per-registration `hook.timeoutMs` over the default 15s modifying-hook cap. The latest release tag still shows the reported 15s cap behavior, so this is a main-only fix until the next shipped release.

## What This Changes

The issue asks for the `before_prompt_build` outer hook timeout to be configurable or to respect Active Memory's plugin-level `timeoutMs` setting.

## Best Possible Solution

Keep the plugin-owned timeout wiring: Active Memory should use its own config to set both the recall budget and the registered prompt-hook budget, while the generic hook runner keeps a conservative default for plugins that do not opt into a longer handler budget.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction by comparing `v2026.4.27`, where Active Memory registers `before_prompt_build` without hook options and the runner uses only the 15s global cap, against current main where a configured Active Memory timeout controls the hook cap.

## Solution Assessment

Yes. The current main approach is the narrowest maintainable fix for this report because it uses the generic plugin hook registration contract instead of adding an Active Memory-specific core exception or a broader gateway config API before multiple plugins need it.
