## Summary

Keep this PR open. Current main still computes both preflight compaction and memory-flush thresholds from `reserveTokensFloor` only, while the config/Pi settings contract treats `reserveTokens` as the effective headroom with `reserveTokensFloor` as a minimum. The PR is a focused implementation candidate with no security-sensitive files in the live diff, so it needs normal maintainer review rather than cleanup closure.

## What This Changes

This PR adds an effective Pi compaction reserve resolver, wires preflight compaction and memory-flush gates to use that effective reserve, renames the gate parameter, adds focused regression tests, and records the fix in the changelog.

## Best Possible Solution

Land this PR or an equivalent narrow core fix that resolves one effective compaction reserve from configured `reserveTokens`, `reserveTokensFloor`, current Pi reserve when relevant, and the small-context cap, then uses that value consistently in preflight compaction and memory-flush threshold checks with focused regression coverage and contributor credit preserved.
