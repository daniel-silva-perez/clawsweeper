## Summary

Keep this PR open. Current main still computes both preflight compaction and memory-flush thresholds from `reserveTokensFloor`, while the config/Pi settings path treats `reserveTokens` as the effective reserve with `reserveTokensFloor` as a floor. The PR is a focused implementation candidate for the still-open #66830 and supersedes the closed, unmerged #69379 and #67297 attempts.

## Best Possible Solution

Keep this PR open and review/land it, or land an equivalent narrow core fix. The fix should resolve one effective compaction reserve from configured `reserveTokens`, `reserveTokensFloor`, and the small-context cap, then use that value consistently in preflight compaction and memory-flush gates with focused regression tests and contributor credit for #69379/#67297.
