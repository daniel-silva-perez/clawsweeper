## Summary

Keep open: current main still has the recursive state-directory deletion path that can remove a nested workspace, and this PR is a focused implementation for the linked data-loss bug with no blocking Codex review findings found.

## What This Changes

The PR changes uninstall state cleanup to preserve configured workspace directories during state-only removal, keeps workspace/all removal destructive, and updates uninstall tests and docs.

## Best Possible Solution

Land the focused uninstall cleanup fix after normal maintainer review and completed checks, preserving configured workspaces only for state-only uninstall while keeping `--workspace`, `--all`, and reset full-scope deletion intentionally destructive.

## Reproduction Assessment

Yes. Current main resolves the default workspace under `.openclaw/workspace`, then state-only uninstall removes the parent state directory recursively before any separate workspace-scope decision can preserve it.

## Solution Assessment

Yes. The PR uses the narrowest maintainable boundary: pass configured workspace paths into state cleanup only when workspace deletion is not selected, add filesystem-level regression coverage, and document both CLI and manual uninstall behavior.
