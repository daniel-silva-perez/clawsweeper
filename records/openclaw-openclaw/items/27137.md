## Summary

Keep open. Current main still lacks the requested cross-channel `authorizeIngress()` contract, and representative inbound surfaces continue to authorize through lower-level shared helpers or channel-local gates. The existing maintainer discussion already treats this as a security/design RFC, so cleanup closure would be premature.

## What This Changes

The issue requests a unified ingress authorization gate that centralizes sender, allowlist, DM/group policy, and event-kind authorization for inbound channel events.

## Best Possible Solution

Keep the RFC open until maintainers and secops approve an additive shared ingress authorization contract. The target end state should centralize policy decisions in a shared security/SDK seam while preserving channel-specific identity/context adapters, then migrate representative message, reaction, pin, interaction, callback, slash, voice, and media paths with contract tests.
