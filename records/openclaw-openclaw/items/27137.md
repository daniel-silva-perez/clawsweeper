## Summary

Keep open. Current main still lacks the requested unified ingress authorization contract; the new channel turn kernel is useful adjacent infrastructure, but it is scoped to message-turn lifecycle and representative non-message/channel paths still authorize locally around lower-level shared helpers. The existing maintainer and ClawSweeper comments correctly route this as security/design follow-up rather than cleanup closure.

## What This Changes

The issue requests a unified `authorizeIngress()` security contract so channel/event authorization decisions are centralized instead of reimplemented per plugin surface.

## Best Possible Solution

Keep the RFC open until maintainers and secops approve an additive shared ingress authorization contract. The target end state should centralize policy decisions while preserving channel-specific identity/context adapters, then migrate representative message, reaction, pin, interaction, callback, slash, voice, and media ingress paths with contract tests.

## Reproduction Assessment

Not applicable as a single runtime reproduction because this is a security architecture RFC. We do have high-confidence structural verification from source inspection: the requested gate is absent and representative ingress paths still authorize locally or through narrower helpers.

## Solution Assessment

Unclear until maintainer/secops design review approves the contract shape. A shared additive gate is plausible, but the maintainable solution must preserve plugin SDK compatibility and channel-specific normalization instead of forcing one unsafe generic adapter.
