## Summary

Keep open. Current main still lacks the requested unified ingress authorization contract, and representative channel/event paths continue to authorize through lower-level shared helpers or channel-local gates. The discussion already treats this as a security/design RFC, so this needs maintainer and secops design review rather than cleanup closure.

## What This Changes

The issue requests a shared `authorizeIngress()` gate in `src/security/ingress-auth.ts` to centralize channel/event authorization and migrate per-channel auth paths onto it.

## Best Possible Solution

Keep this RFC open until maintainers approve an additive ingress authorization contract. The desired end state is a single shared policy decision point with channel-specific identity/context adapters, followed by representative migrations and contract tests for messages, reactions, pins, interactions, callbacks, slash commands, voice, and media ingress.
