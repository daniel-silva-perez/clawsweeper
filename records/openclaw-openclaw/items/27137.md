## Summary

Keep open. Current main still lacks the requested cross-channel `authorizeIngress()` contract, and representative ingress paths continue to authorize through lower-level shared helpers or channel-local gates. The issue discussion already has maintainer/bot reviews keeping this open as a security/design RFC, so cleanup closure would be premature.

## What This Changes

The issue requests a unified shared ingress authorization gate that centralizes DM, group, allowlist, sender, account, channel, and event-kind policy decisions across channel plugins.

## Best Possible Solution

Keep the RFC open until maintainers and secops approve an additive shared ingress authorization contract. The end state should centralize policy decisions while preserving channel-specific identity/context adapters, then migrate representative message, reaction, pin, interaction, callback, slash, voice, and media ingress paths with contract tests.

## Reproduction Assessment

Not applicable as a single runtime reproduction; this is a security architecture RFC. Current-main source inspection provides high-confidence verification of the structural gap because the requested gate is absent and representative paths still authorize locally.

## Solution Assessment

Unclear until maintainer/secops design review approves the API shape. A shared additive gate is a plausible direction, but the maintainable solution must preserve plugin SDK compatibility and channel-specific normalization rather than forcing one unsafe generic adapter.
