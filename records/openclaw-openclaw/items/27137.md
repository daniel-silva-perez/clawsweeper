## Summary

Current main still lacks the requested unified ingress authorization gate, and the remaining work is a cross-channel security architecture decision rather than a closeable cleanup item. Existing shared primitives and several hardened channel-local helpers reduce drift in some places, but they do not provide the proposed single `authorizeIngress()` contract, and the existing maintainer review already kept the RFC open for security design follow-up.

## What This Changes

The issue requests a unified ingress authorization gate that centralizes channel/account/event context, sender identity, chat context, allowlist merging, DM/group policy evaluation, command authorization, and standardized denial results across message, reaction, pin, interaction, callback, slash, voice, and related ingress surfaces.

## Best Possible Solution

Keep the RFC open until maintainers approve an additive, versioned core/security or SDK-backed ingress authorization contract. The end state should preserve existing channel-specific identity adapters while centralizing the policy decision and adding contract tests that prove representative message, reaction, pin, interaction, callback, slash, voice, and media ingress paths cannot bypass the shared gate.
