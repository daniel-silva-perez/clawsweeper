## Summary

Keep open. Current main still lacks the requested cross-channel ingress authorization contract, and representative channel/event paths continue to assemble authorization locally around lower-level shared helpers. The existing maintainer and bot discussion already points this toward security/design follow-up rather than cleanup closure.

## What This Changes

Request to add a shared `authorizeIngress()` security/SDK contract that centralizes ingress authorization decisions across channel and event adapters.

## Best Possible Solution

Keep the RFC open until maintainers and secops approve an additive shared ingress authorization contract. The target end state should centralize policy decisions while preserving channel-specific identity/context adapters, then migrate representative message, reaction, pin, interaction, callback, slash, voice, and media ingress paths with contract tests.

## Reproduction Assessment

Not applicable as a single runtime reproduction because this is a security architecture RFC. We do have high-confidence structural verification from source inspection: the requested gate is absent and representative ingress paths still authorize locally.

## Solution Assessment

Unclear until maintainer/secops design review approves the API shape. A shared additive gate is a plausible direction, but the maintainable solution must preserve plugin SDK compatibility and channel-specific normalization instead of forcing one unsafe generic adapter.
