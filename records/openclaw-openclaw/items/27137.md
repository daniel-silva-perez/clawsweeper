## Summary

Keep open. Current main still lacks the requested unified ingress authorization gate, and representative channel/event paths continue to assemble authorization through lower-level helpers or plugin-local gates. The issue discussion also already treats this as a security/design RFC, so it is not a conservative cleanup-close candidate.

## What This Changes

Request to add a shared `authorizeIngress()` contract that centralizes inbound channel authorization while letting channel adapters provide identity and context.

## Best Possible Solution

Keep the RFC open until maintainers approve an additive ingress authorization contract. The target end state should centralize policy evaluation behind a shared gate, preserve channel-specific identity/context adapters, and migrate representative message, reaction, pin, interaction, callback, slash, voice, and media ingress paths with contract and bundled-plugin tests.
