## Summary

Keep open. Current main still has the reported Discord `components` gap, and the PR is useful, but the current branch is dirty/unrebaseable and its edit-runtime hunk targets the old monolithic Discord messaging runtime instead of the split files on main.

## What This Changes

The PR adds a Discord shared message-tool `components` schema, forwards top-level `params.components` through Discord send/edit actions, and adds focused Discord source tests for those paths.

## Best Possible Solution

Rebase this PR onto current main and adapt the implementation to the split Discord messaging runtime: keep the schema aligned with the documented OpenClaw component-spec object, forward top-level `components` for send, wire component edits through `runtime.messaging.messages.ts` plus `runtime.messaging.runtime.ts`, and keep targeted tests for schema, send bridge, and edit runtime behavior.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: call the Discord shared `message` tool with `action: "send"` and a top-level `components` object; the local bridge never reads `params.components`, so the lower component sender receives no component spec.

## Solution Assessment

No, not as currently proposed. The direction is correct, but the branch needs rebase/adaptation because its component edit support is implemented in the old monolithic runtime file instead of the split current-main handlers.
