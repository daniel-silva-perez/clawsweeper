## Summary

Keep open. Current main still manages the macOS gateway lifecycle through the user LaunchAgent path and GUI launchd domain only; status has a diagnostic system-service scan, but stop/uninstall do not have a system scope and cannot manage the reported LaunchDaemon case. The recent maintainer/bot discussion also kept this work active alongside the broader LaunchDaemon design in #51860.

## What This Changes

The issue requests macOS gateway lifecycle support for system-level LaunchDaemons under `/Library/LaunchDaemons`, including accurate detection, stop/uninstall guidance, and possible explicit system-scope flags.

## Best Possible Solution

Define one explicit macOS launchd scope model, then update lifecycle, status, docs, and regression tests so user LaunchAgents and system LaunchDaemons are detected accurately, stop/uninstall never report user-scope success while a matching system daemon remains active, and privileged actions require explicit operator intent.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with a default-label LaunchDaemon loaded in the `system` domain and no matching user LaunchAgent, current main still checks `gui/<uid>/<label>` and `~/Library/LaunchAgents`, while the CLI has no lifecycle scope flag.

## Solution Assessment

Unclear. The requested direction is valid, but the best implementation should be chosen after maintainers settle the shared LaunchDaemon scope contract with #51860 rather than adding ad hoc stop/uninstall behavior alone.
