## Summary

Keep open: current main still lacks managed macOS system LaunchDaemon lifecycle support for gateway stop/uninstall, and the existing deep status scan is diagnostic rather than a system-scope lifecycle path.

## What This Changes

Request that gateway stop, uninstall, and status detect system-level macOS LaunchDaemons under `/Library/LaunchDaemons` and avoid reporting user LaunchAgent success while a matching system daemon remains active.

## Best Possible Solution

Define one explicit macOS launchd user/system scope model, then update lifecycle, status, docs, and regression tests so user LaunchAgents and system LaunchDaemons are detected accurately, stop/uninstall never report user-scope success while a matching system daemon remains active, and privileged actions require explicit operator intent.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with only `/Library/LaunchDaemons/ai.openclaw.gateway.plist` loaded, current managed lifecycle code still checks `~/Library/LaunchAgents` and `gui/<uid>/<label>`, while stop/uninstall have no system target.

## Solution Assessment

Unclear. The requested direction is valid, but the best exact fix needs maintainer agreement on the privileged LaunchDaemon scope contract before an automated PR chooses sudo behavior or daemon ownership semantics.
