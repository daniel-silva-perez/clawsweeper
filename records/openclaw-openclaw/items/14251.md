## Summary

Keep open. Current main still manages macOS gateway lifecycle through the user LaunchAgent path and GUI launchd domain only; `status --deep` is diagnostic discovery, and it does not provide stop/uninstall handling for the reported system LaunchDaemon case.

## What This Changes

Improve macOS gateway stop/uninstall/status behavior so system-level LaunchDaemons in `/Library/LaunchDaemons` are detected and either explicitly managed or reported with actionable privileged-command guidance.

## Best Possible Solution

Define one explicit macOS launchd scope model, then update lifecycle, status, docs, and regression tests so user LaunchAgents and system LaunchDaemons are detected accurately, stop/uninstall never report user-scope success while a matching system daemon remains active, and privileged actions require explicit operator intent.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with only `/Library/LaunchDaemons/ai.openclaw.gateway.plist` loaded, current stop/uninstall still check `~/Library/LaunchAgents` and `gui/<uid>/<label>`, and the CLI has no lifecycle scope flag.

## Solution Assessment

Unclear. The requested direction is valid, but the exact implementation should be chosen after maintainers settle the shared LaunchDaemon scope contract with #51860 rather than adding ad hoc stop/uninstall behavior alone.
