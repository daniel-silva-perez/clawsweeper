## Summary

Keep open. Current main still manages the macOS gateway as a user LaunchAgent only for stop/uninstall, while the issue asks for detection and lifecycle handling of matching system LaunchDaemons. The request is valid, but it crosses privileged launchd behavior and overlaps the broader macOS LaunchDaemon product decisions in #51860 and #55208.

## What This Changes

The issue requests that gateway stop, uninstall, and status detect `/Library/LaunchDaemons/ai.openclaw.gateway.plist`, avoid false user-scope success, and provide an explicit safe system-scope path or guidance.

## Best Possible Solution

Define one explicit macOS launchd scope model, then update lifecycle, status, docs, and regression tests so user LaunchAgents and system LaunchDaemons are detected accurately, stop/uninstall do not report user-scope success while a matching system daemon remains active, and privileged actions require explicit safe operator intent.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: with the gateway plist in `/Library/LaunchDaemons`, `openclaw gateway stop` and `uninstall` still check `gui/<uid>/<label>` and `~/Library/LaunchAgents`, so the system daemon path described by the issue is outside the managed lifecycle path.

## Solution Assessment

Unclear. The requested direction is correct, but the best exact solution needs maintainer design for the privileged LaunchDaemon boundary rather than an automatic narrow fix with guessed flag names or sudo behavior.
