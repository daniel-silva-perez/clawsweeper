## Summary

Keep open. Current main still models macOS gateway lifecycle as a user LaunchAgent for install, stop, uninstall, runtime reads, and loaded checks. `status --deep` has a best-effort extra-service scan, but that is diagnostic only and does not give stop/uninstall a system LaunchDaemon target, so the reported lifecycle gap remains valid.

## What This Changes

The issue requests detecting and managing macOS system-level `/Library/LaunchDaemons/ai.openclaw.gateway.plist` entries in gateway stop, uninstall, and status paths instead of only user LaunchAgents.

## Best Possible Solution

Define one explicit macOS launchd scope model, then update lifecycle, status, docs, and regression tests so user LaunchAgents and system LaunchDaemons are detected accurately, stop/uninstall never report user-scope success while a matching system daemon remains active, and privileged actions require explicit operator intent.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: with only `/Library/LaunchDaemons/ai.openclaw.gateway.plist` loaded, the managed lifecycle path still checks `gui/<uid>/<label>` and `~/Library/LaunchAgents`, while stop can at most signal an unmanaged listener and uninstall never removes or unloads the system daemon.

## Solution Assessment

Unclear. The requested direction is correct, but the best exact fix needs maintainer agreement on the privileged LaunchDaemon UX and shared scope naming before an automated narrow PR guesses sudo behavior or daemon ownership semantics.
