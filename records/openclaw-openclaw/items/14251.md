## Summary

Keep open. Current main still manages macOS gateway stop/uninstall through the user LaunchAgent path only, while `status --deep` is only diagnostic; the related discussion keeps the broader LaunchDaemon scope work active with #51860 rather than resolving this lifecycle gap.

## What This Changes

Improve macOS gateway stop/uninstall so system-level LaunchDaemons in `/Library/LaunchDaemons` are detected and either managed explicitly or reported with actionable elevated-command guidance.

## Best Possible Solution

Define one explicit macOS launchd scope model, then update lifecycle, status, docs, and regression tests so user LaunchAgents and system LaunchDaemons are detected accurately, stop/uninstall never report user-scope success while a matching system daemon remains active, and privileged actions require explicit operator intent.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with only `/Library/LaunchDaemons/ai.openclaw.gateway.plist` loaded, current stop/uninstall still target `~/Library/LaunchAgents` and `gui/<uid>/<label>`, and the CLI has no lifecycle scope flag.

## Solution Assessment

Unclear. The requested direction is valid, but the best exact fix needs maintainer agreement on the privileged LaunchDaemon scope contract and alignment with #51860 before choosing `--system`/`--daemon`, sudo behavior, and ownership semantics.
