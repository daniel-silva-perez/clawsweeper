## Summary

Keep this PR open. Current main still implements `${VAR}` substitution as string substitution only, while this PR proposes a real behavior change to the global config substitution contract; it needs maintainer review around scope, write-back preservation, whitespace opt-in semantics, docs/changelog coverage, and failing checks before merge.

## What This Changes

The PR changes config env substitution so a config string that is exactly `${VAR}` can resolve to a `string[]` when the env value is a JSON string array, and adds boundary tests for that behavior.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers want the feature, decide whether coercion should be global or limited to known list-typed config paths, define strict vs whitespace-tolerant opt-in semantics, update write-back preservation for authored whole-env-reference arrays, add docs/changelog coverage, and get the targeted config tests plus changed gate green.

## Risks / Open Questions

- This changes the global config substitution contract: any config string exactly matching `${VAR}` could become an array, not only known list-typed paths such as allowlists.
- Whole-env-reference array coercion can interact poorly with config write-back preservation because current restoration logic treats string/array type mismatches as intentional changes.
- The proposed placeholder match tolerates surrounding whitespace, so `"  ${VAR}  "` also opts into array coercion unless maintainers decide that is intended.
- The PR adds tests but no docs or changelog entry for a user-visible config behavior change.
- The PR head currently has failing checks.
