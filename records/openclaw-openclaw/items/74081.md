## Summary

Current main still implements config env substitution as string substitution only, so this PR is not obsolete or implemented. The PR is narrow and relevant, but it changes the global config substitution contract and needs maintainer review for write-back preservation, whitespace-trigger behavior, docs/changelog coverage, and failing checks before merge.

## What This Changes

This PR adds JSON string-array coercion for whole-value `${VAR}` config placeholders and adds boundary tests for the new env-substitution behavior.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers want the feature, decide whether coercion should be global or limited to known list-typed config paths, update env-ref write-back preservation for whole-env-ref arrays, document the behavior, add a changelog entry if accepted, resolve the whitespace behavior, and require targeted config tests plus the changed gate before merge.

## Risks / Open Questions

- Global coercion would change the contract for every exact `${VAR}` string, not only known list-typed config paths, so maintainers need to decide whether this belongs globally or behind schema/path-aware handling.
- The current write-back preservation path does not restore an original string env ref when the incoming resolved value becomes an array, so config writes could persist the literal array instead of preserving `${VAR}`.
- The PR intentionally treats surrounding whitespace around the placeholder as opt-in, which may be surprising unless maintainers confirm and document that behavior.
- Docs and changelog coverage are missing for a user-visible config/env behavior change, and the provided context reports non-green checks on the PR head.
- Security review found no supply-chain or execution-surface expansion in the diff, but allowlist-style config values are security-relevant enough that the type-coercion contract should be reviewed deliberately.
