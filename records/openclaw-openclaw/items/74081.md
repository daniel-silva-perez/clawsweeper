## Summary

Keep open. Current main still implements config env substitution as string substitution only, and this PR changes a global config contract that needs maintainer product judgment before merge, especially around path scoping, whitespace semantics, write-back preservation, docs, and changed-gate validation. The security pass did not find workflow, dependency, lockfile, install script, release, publishing, downloaded-artifact, or permission changes.

## What This Changes

The PR adds whole-value `${VAR}` JSON string-array coercion in `src/config/env-substitution.ts` and boundary tests in `src/config/env-substitution.test.ts`.

## Best Possible Solution

If maintainers want this capability, define the config contract first: either make whole-env-reference JSON string arrays a documented global substitution rule or limit it to known list-typed paths, then update env substitution, write-back preservation, docs/changelog if user-facing, and focused regression coverage before running the changed gate.

## Risks / Open Questions

- Global string-to-array coercion would change resolved config types for every string path, including plugin-owned config, unless maintainers choose a schema/path-gated contract.
- The proposed whitespace-tolerant whole-placeholder match may coerce values like `  ${ALLOW_FROM}  ` even when surrounding whitespace was accidental.
- Config write-back preservation currently handles matching string/string or array/array shapes, so an authored string `${VAR}` that resolves to an array needs an explicit round-trip policy.
