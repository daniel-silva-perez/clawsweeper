## Summary

Keep this PR open. Current main does not implement JSON-array coercion, but the branch is not ready to merge because it changes the global env-substitution contract, can unexpectedly turn any exact env-backed string into an array, and does not preserve authored env references on config write-back for the intended array case.

## What This Changes

The PR changes config env substitution so a whole-value `${VAR}` can resolve to a `string[]` when the env value is a JSON string array, and adds boundary tests for that behavior.

## Best Possible Solution

If maintainers want this capability, make the coercion schema- or path-aware for list-typed config fields, or introduce an explicit list-ref syntax. Preserve authored env references through write-back, document the resulting contract, add a changelog entry, and cover both read-time coercion and write-back round trips in focused tests.

## Reproduction Assessment

Yes for the review findings. The high-confidence static path is `resolveConfigForRead` substituting before validation, followed by `writeConfigFile` calling `restoreEnvVarRefs`; a parsed string `${ALLOW_FROM}` that becomes an incoming array currently hits the mismatched-type fallback and is kept as the resolved array.

## Solution Assessment

No. The proposed syntax-only global coercion is narrower than parsing all strings, but it is not the safest maintainable contract because exact `${VAR}` is already documented as normal string substitution for any string-valued field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve env refs after array coercion:** `src/config/env-substitution.ts:200`
  - body: Returning a `string[]` here means the intended config value no longer has the same type as the parsed source string `${ALLOW_FROM}` that `writeConfigFile` gives to `restoreEnvVarRefs`. That helper only restores string/string or array/array leaves, so a read/write round trip would persist the resolved array and lose the authored env reference. Add preservation coverage for whole-ref arrays or keep the resolved type compatible with the parsed source.
  - confidence: 0.91
- **[P2] Avoid coercing every exact env-backed string:** `src/config/env-substitution.ts:197-200`
  - body: `SINGLE_VAR_REFERENCE.test(value)` has no path or schema guard, so any existing string field authored as `${VAR}` becomes an array when the env value is JSON array text. Fields such as channel `systemPrompt` are validated as strings, so an env-backed JSON prompt value can now fail validation or change type unexpectedly. Gate this on known list-typed config paths/schema metadata, or use an explicit syntax.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/config/env-substitution.ts:191`
  - body: This is a user-facing config feature, and OpenClaw policy requires user-facing `feat` changes to add a `CHANGELOG.md` entry. The branch only touches config substitution code and tests, so release notes would miss the new config contract.
  - confidence: 0.88
