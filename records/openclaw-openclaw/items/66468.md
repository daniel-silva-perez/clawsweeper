## Summary

Keep open. Current main still does not emit a live provider/model-attempt event from the model fallback loop, and the CLI TUI still renders the footer from sessionInfo without consuming any fallback/model-switch stream. The prior ClawSweeper comment remains accurate on current main; adjacent web fallback status support exists, but it is not wired to the production fallback loop or the TUI footer path reported here.

## Best Possible Solution

Implement a narrow typed fallback-attempt status path in the owner code: emit runId/sessionKey/provider/model/attempt/total immediately before each actually attempted model fallback candidate, broadcast it through the existing agent event path, have the CLI TUI consume it with correct session/run filtering before updating state.sessionInfo.modelProvider and state.sessionInfo.model, and add regression coverage for both fallback emission and TUI footer state updates.
