## Summary

Keep open. Current main still has a Feishu per-chat sequential queue that waits for the full message dispatch promise to settle before starting later same-chat messages, and there is no timeout or release guard on that queue. The existing Feishu HTTP client timeout covers stalled Lark API calls, but not an agent dispatch that remains in-flight for minutes. The related #54409 discussion is useful context but explicitly tracks a different collect-mode batching failure.

## Best Possible Solution

Keep this issue open and implement a narrow Feishu queue starvation fix: add a bounded release/timeout guard for same-chat message dispatch tasks, preserve the existing `/stop` control and `/btw` out-of-band lanes, and add regression tests proving a stuck first task cannot indefinitely block later P2P messages. Coordinate with #54409 if the implementation also changes collect-mode behavior, but do not treat it as the same acceptance target.
