## Summary

Keep open. Current main still serializes ordinary Feishu same-chat messages behind the previous dispatch promise, and I did not find a timeout, abort, or release guard for a normal P2P dispatch that hangs before `dispatch complete`. The related #54409 and #43512 context is useful but does not supersede this starvation report.

## What This Changes

The issue asks for Feishu P2P per-chat queue handling to recover when one dispatch hangs so later messages in the same chat are not blocked indefinitely.

## Best Possible Solution

Add a Feishu-owned starvation guard for ordinary per-chat dispatches that uses existing abortable dispatch semantics where possible, releases the per-chat lane after a bounded timeout, logs the timeout clearly, and preserves FIFO behavior plus the existing `/stop` and `/btw` out-of-band lanes. Cover it with focused queue and message-handler tests, and coordinate any broader followup-queue redesign with #54409 rather than treating that issue as the canonical fix.

## Reproduction Assessment

Yes. The issue provides timestamped Feishu logs showing one dispatch occupying a P2P chat for several minutes while later messages log as queued, and the current source has a deterministic seam: make the first `handleMessage` promise never settle and a second same-key message will wait behind `createSequentialQueue`.

## Solution Assessment

Unclear because there is no patch yet. The requested timeout/release guard is the right direction, but the maintainable fix should abort or settle the underlying dispatch path instead of only letting another message start while the original hung run can still emit replies.

## Repair Work Prompt

Fix Feishu P2P per-chat queue starvation from https://github.com/openclaw/openclaw/issues/70133. Current main still awaits `createSequentialQueue()` around normal `handleMessage` dispatch in `extensions/feishu/src/monitor.message-handler.ts`, and the queue only releases when the prior promise settles. Add a bounded timeout/release guard for ordinary Feishu per-chat dispatches so one hung task cannot block later same-chat messages indefinitely. Prefer an implementation that aborts or otherwise settles the underlying dispatch via existing abortable reply/turn semantics; do not merely `Promise.race` and leave the old run able to emit late replies. Preserve normal FIFO ordering when dispatches complete normally, preserve existing `/stop` and `/btw` out-of-band lanes in `extensions/feishu/src/sequential-key.ts`, and do not change unrelated channels or broad gateway queue policy unless a tiny generic seam is clearly required. Add focused tests that simulate a first same-key Feishu dispatch that never resolves, prove the queue releases or times out, prove the next message is no longer stuck forever, and prove no unhandled rejection or processing-claim leak occurs. Coordinate wording with related #54409, but do not close or solve collect batching unless the same narrow change naturally fixes it. Add a user-facing changelog fix entry for Feishu queue starvation with `Thanks @loyur` if behavior changes.
