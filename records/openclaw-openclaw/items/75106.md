## Summary

Keep open. The PR targets a valid `memory status --deep` diagnostic bug from #74544, but the current diff still leaves the inner 600-second embedding batch timeout alive after the 8-second race, suppresses verbose non-TTY progress output, and misses the required changelog credit, so it should not merge as-is.

## What This Changes

This PR adds an 8-second budget around the memory-core deep embedding probe, renders a distinct timeout state, exposes `timedOut` on memory probe result types, and updates tests, memory CLI docs, and the changelog.

## Best Possible Solution

Move the diagnostic budget into a path that avoids or cancels the normal batch embedding timeout, such as a manager/provider-level short-budget probe or a cancellable probe API. Keep non-TTY progress visible with the `log` fallback for all piped status runs, preserve the longer batch timeout for indexing, and add the required changelog credit plus tests that exercise the real manager timer behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a cache-miss `memory status --deep` awaits `probeEmbeddingAvailability()`, which runs `embedBatchWithRetry(["ping"])` through the local 600-second batch timeout path; the reporter's #74544 follow-up shows the same 600-second timeout in real output.

## Solution Assessment

No. The proposed top-level `Promise.race` is not the best fix because it does not cancel or avoid the inner manager probe and its ref-counted batch timeout; a manager-level short-budget or cancellable status probe is safer and matches the command-runtime requirement.
