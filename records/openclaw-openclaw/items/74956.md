## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` hook context, even though the shared dispatch path has already resolved both values before emitting the hook. The reported WAL artifact is not present in this source checkout, but the generic hook-contract gap is source-backed and narrow enough for a focused fix.

## What This Changes

The issue asks for `message:received` hook events to carry per-agent routing context (`workspaceDir` and/or `agentId`) so WAL protocol hooks can write state to the correct agent workspace.

## Best Possible Solution

Expose optional `agentId` and `workspaceDir` on the internal `message:received` context, populate them from already-resolved routing/session data at each emitter, update docs and focused tests, and keep WAL-specific behavior out of core.

## Reproduction Assessment

Yes for the root cause: current-main source and docs show `message:received` lacks `agentId`/`workspaceDir` while `dispatchReplyFromConfig` has already resolved both. The exact WAL write-to-main-workspace symptom is plausible but not directly reproducible here because the reported dist hook artifact is not in the source checkout.

## Solution Assessment

Yes, the narrow maintainable solution is an additive hook-contract enrichment with optional routing fields. Reconstructing WAL policy in core would be the wrong boundary; core should only provide generic per-agent context to hooks.
