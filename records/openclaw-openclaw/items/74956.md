## Summary

Keep open. Current `main` still exposes the internal `message:received` hook without `agentId` or `workspaceDir`, while the inbound dispatch path has already resolved both values before emitting the hook. The reported packaged WAL hook artifact is not present in this source checkout, but the generic hook context gap is source-backed and narrow enough to fix directly.

## What This Changes

The issue reports that the internal `message:received` hook context lacks per-agent routing fields needed by hook handlers to write agent-scoped state.

## Best Possible Solution

Add additive `agentId` and `workspaceDir` fields to the internal `message:received` context wherever the emitter has resolved per-agent routing data, update the docs and focused tests, and keep the change generic rather than adding WAL or active-memory-specific behavior to core.

## Reproduction Assessment

Yes for the root cause: inspect the current inbound dispatch path and `message:received` hook context, and it lacks `agentId` and `workspaceDir` even though `dispatchReplyFromConfig` has already resolved them. The exact WAL file-write symptom is unclear because the named packaged hook artifact is absent from the source tree.

## Solution Assessment

Yes, enriching the internal hook event context is the narrow maintainable fix because it preserves the existing hook model and gives hook authors the same per-agent routing data command hooks already receive. The safer boundary is to add only additive routing fields, not a hook-specific migration or active-memory special case.
