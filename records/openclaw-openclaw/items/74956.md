## Summary

Keep open. Current main still exposes `message:received` internal hook context without `agentId` or `workspaceDir`, while the shared inbound dispatch path has already resolved both values before emitting the hook. The exact `wal-protocol` dist artifact named in the report is not present in the source tree, but the generic hook context gap is source-backed and narrow enough for a focused fix.

## What This Changes

The issue requests adding per-agent routing context to `message:received` hooks so hooks can write state into the correct agent workspace instead of the default workspace.

## Best Possible Solution

Additive hook-contract support should expose optional `agentId` and `workspaceDir` on internal `message:received` context, populate them from already-resolved routing data, update docs/types/tests, and keep the change generic rather than adding WAL- or active-memory-specific behavior to core.

## Reproduction Assessment

Yes for the root cause: current-main code inspection shows `dispatchReplyFromConfig` resolves `sessionAgentId` and `workspaceDir` before emitting `message:received`, but the emitted context omits them. The downstream WAL write symptom is plausible but not directly reproducible from this checkout because the named dist hook artifact is absent.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: enrich the existing internal hook context with optional per-agent fields. The safer boundary is a generic hook API addition plus emitter coverage, not reconstructing the WAL hook or adding active-memory policy in core.
