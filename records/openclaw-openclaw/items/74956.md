## Summary

Keep open. Current main still has the reported hook-contract gap: the internal `message:received` context and mapper omit `agentId` and `workspaceDir`, while the shared dispatcher has already resolved `sessionAgentId` and `workspaceDir` before emitting the hook. The reported WAL artifact is not present in this source checkout, but the generic per-agent routing gap is source-backed and narrow.

## What This Changes

The issue asks for the internal `message:received` hook context to include per-agent routing data so hooks can write state to the correct agent workspace.

## Best Possible Solution

Expose optional `agentId` and `workspaceDir` on the internal `message:received` context, populate them from already-resolved routing/session data at each emitter, update docs and focused tests, and keep WAL-specific write policy out of core.

## Reproduction Assessment

Yes for the root cause: current-main source and docs show `message:received` lacks `agentId`/`workspaceDir`, while `dispatchReplyFromConfig` has already resolved both values. The exact WAL file write symptom remains unaudited because the reported generated hook artifact is absent from the source checkout.

## Solution Assessment

Yes, an additive hook-contract enrichment is the narrowest maintainable fix. Rebuilding WAL behavior in core would cross ownership boundaries; core should provide generic per-agent context and leave hook-specific state handling to the hook/plugin.
