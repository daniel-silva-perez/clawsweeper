## Summary

Keep open. Current main still forwards OpenAI-compatible `input_type` values but does not expose or send Jina's separate `task` field, and the same author has an open, unmerged PR that explicitly closes this issue. Per policy, the issue should stay open until that PR is reviewed, landed, or closed.

## What This Changes

Request support for configuring Jina Embeddings v5 task-specific adapter parameters on OpenAI-compatible memory embedding requests, preferably with separate query and document task fields.

## Best Possible Solution

Review #74444 as the implementation candidate. The desired end state is typed query/document task configuration carried through memorySearch schema, config resolution, SDK/provider options, inline and batch OpenAI-compatible embedding payloads, cache identity, docs, tests, and changelog without weakening existing `input_type` behavior.

## Reproduction Assessment

Yes. The focused current-main check is to inspect `MemorySearchSchema` and the OpenAI inline/batch embedding request builders: they accept/forward `input_type` but have no `task` config or payload path.

## Solution Assessment

Yes. A typed query/document task seam is the narrowest maintainable direction because memory search already distinguishes query and document embedding requests; arbitrary embedding-body passthrough should remain a separate maintainer product decision.
