## Summary

Keep open. Current main still has Discord presence support through static config, auto-presence, and the gated `setPresence` message action, but it still does not expose the requested code-plugin helper at `api.runtime.channel.discord.updatePresence()`. The remaining decision crosses a public plugin-runtime API boundary, so this is not conservative cleanup.

## Best Possible Solution

Keep this open for Discord/plugin-runtime owner review. The best path is either a narrow Discord-owned, documented runtime helper that reuses the existing gateway registry, action gate, validation, account selection, and tests, or an explicit maintainer decision that the existing gated `setPresence` message action is the supported code-plugin path.

## Risks / Open Questions

- A new helper would mutate the live Discord gateway session, so it needs the same operator opt-in, account resolution, connected-gateway checks, validation, and errors as the existing `setPresence` action.
- Auto-presence and static startup presence can also call `gateway.updatePresence(...)`; plugin-driven updates need clear precedence so they are not unexpectedly overwritten.
- The proposed `api.runtime.channel.discord.*` shape touches a public plugin-runtime boundary while current docs/types describe `api.runtime.channel` as a generic helper surface.
- Implementation should re-check the `@buape/carbon/gateway` 0.16.0 `GatewayPlugin.updatePresence` and `UpdatePresenceData` contract before exposing a stable SDK helper.
