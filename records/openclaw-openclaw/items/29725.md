## Summary

Keep open. Current main has adjacent Discord presence support through static config, auto-presence, and a gated `setPresence` message action, but it still does not expose the requested code-plugin helper `api.runtime.channel.discord.updatePresence()`. The remaining work needs a maintainer decision on the public runtime API shape, because the generic runtime channel surface is intentionally core-generic while the request asks for a Discord-specific mutation helper.

## Best Possible Solution

Keep this open for Discord/plugin-runtime owner review. The best path is either a narrow Discord-owned, documented `updatePresence` helper that reuses the existing gateway registry and presence validation path, or an explicit docs/API decision that the existing gated `setPresence` message action is the supported code-plugin equivalent. The fix should stay in the Discord plugin/runtime surface unless maintainers decide a generic plugin-runtime capability is needed.

## Risks / Open Questions

- A runtime helper would mutate the active Discord gateway session, so it must preserve the existing opt-in `channels.discord.actions.presence` gate, account selection, gateway registration, connection checks, validation, and clear error messages.
- Static presence and auto-presence can also update the same Discord gateway state; a new helper needs explicit interaction rules so plugin-driven updates are not immediately overwritten in surprising ways.
- The requested `api.runtime.channel.discord.*` shape crosses a public plugin-facing runtime boundary while `PluginRuntimeChannel` is documented as generic core helpers only, so maintainers should choose whether to add this helper, expose a generic runtime-context capability, or document the existing `setPresence` message action as the supported code-plugin path.
