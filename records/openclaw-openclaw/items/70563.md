## Summary

Keep this issue open. Current main still has a Kimi Coding-specific replay gap: native `sessions_spawn` can carry a thinking override into the child run, and the Kimi provider enables top-level thinking, but its payload wrapper does not preserve or synthesize `reasoning_content` on replayed assistant tool-call messages. The related #50178 and open Moonshot PR #74846 are useful context, but they do not cover the bundled `kimi`/Kimi Coding provider path reported here.

## What This Changes

The issue reports that native `sessions_spawn` fails for Kimi thinking-enabled subagent runs because replayed assistant tool-call messages lack `reasoning_content`.

## Best Possible Solution

Keep the fix provider-owned in `extensions/kimi-coding`: when Kimi thinking is enabled, preserve existing provider reasoning metadata or add a Kimi-accepted `reasoning_content` placeholder on replayed assistant tool-call messages, with focused regression coverage for Kimi tool-call continuation and a subagent thinking override path.

## Reproduction Assessment

Yes. The issue gives a concrete reproduction path: use Kimi/Kimi Coding with thinking enabled and spawn a native subagent that performs a tool call, forcing assistant tool-call history replay.

## Solution Assessment

No. Current main is not the best solution yet because it only sets top-level Kimi `thinking`; the narrow maintainable solution is a Kimi provider payload/replay wrapper analogous to the existing DeepSeek reasoning-content repair, not a core subagent workaround.
