## Summary

Keep open. Current main has prompt/docs guidance and optional loop detection for repeated running exec outcomes, but it still does not provide the default runtime guard this report asks for: loop detection is disabled by default, the exec tool still returns a running session and relies on the model to switch to process, and the covered exec loop detector only blocks after repeated no-progress history when explicitly enabled.

## What This Changes

The issue requests a runtime guard against repeatedly launching identical long-running exec commands after an exec result reports `status: "running"` and tells the agent to use `process`.

## Best Possible Solution

Add a conservative exec-owned guard that detects an identical in-flight backgrounded command in the same agent/scope and returns the existing process session or a hard instruction to use `process`, while preserving distinct exec calls and leaving broad provider-routing policy out of the first fix.

## Reproduction Assessment

Yes for the runtime gap: a focused test can start a long-running exec with `yieldMs` or `background`, receive `status: "running"`, then issue the same exec parameters again and verify current main starts another session. The Xiaomi/MiMo trigger remains model-dependent, but the missing core guard is reproducible without that provider.

## Solution Assessment

No, current main is not the best solution because optional loop detection is off by default and only intervenes after many repeated outcomes. A narrower and safer fix is to guard exact duplicate in-flight exec launches at the exec/process runtime boundary.

## Repair Work Prompt

Fix openclaw/openclaw#62432. Current main returns `status: "running"` from `exec` with a process `sessionId`, but if the model calls `exec` again with the same long-running command, the runtime can launch another background process instead of forcing/coalescing to `process`. Implement a conservative exec-owned guard: when the same agent/scope already has an in-flight backgrounded exec with the same effective command context, do not spawn another process; return the existing session id with clear `process` follow-up guidance or otherwise coalesce safely. Keep the match exact enough to preserve legitimate parallel work: distinguish workdir, host/node/sandbox target, env-affecting params, pty, security/approval context, and other meaningful exec options. Do not solve this by broad Xiaomi/MiMo provider routing, enabling global loop detection by default, or changing unrelated tool-loop policy. Add focused tests for repeated identical running exec, distinct exec params still launching separately, and process visibility/poll follow-up. Update docs/changelog only if the user-visible behavior changes materially.
