## Summary

Keep open. Current main has the intended `exec` -> `process` guidance and an opt-in loop detector that can normalize repeated running `exec` outcomes, but it does not implement the requested default runtime guard against relaunching the same in-flight `exec` after a `Command still running` result. The related reports in the issue body are adjacent, but they do not supersede this Xiaomi/MiMo-specific cross-session guardrail request.

## Best Possible Solution

Keep the issue open and implement a provider-agnostic exec guard in the agent tool runtime: after a background `exec` returns `status: "running"`, repeated identical in-flight `exec` calls in the same agent/session scope should be coalesced to the existing session or rejected with a structured result directing the model to `process(action="poll")`. Add focused regression coverage for default config behavior, plus a test that volatile session ids/pids do not let repeated relaunches fan out.
