## Summary

Keep open. Current main still has no catalog-derived fallback option, and this PR is useful, but the proposed patch is not mergeable as-is because it would drop existing shared model timeout support and does not update the embedded runner gate that decides whether model fallback is configured.

## What This Changes

The PR adds an opt-in `agents.defaults.model.fallbacksFromModels` config path with schema/types/docs/tests to derive fallback candidates from `agents.defaults.models`, and preserves fallback settings when switching the default model.

## Best Possible Solution

Rebase the branch onto current main, keep the existing shared `timeoutMs` schema/type behavior, update the shared configured-fallback helper so catalog-derived candidates participate in embedded runner failover decisions, add coverage for that runner path, and then run the targeted agent/config tests plus the changed gate.

## Reproduction Assessment

Yes. The patch defects are reproducible from source: parse an `imageGenerationModel.timeoutMs` config after the PR's shared schema split, and run an embedded fallback-worthy prompt/assistant/retry-limit path with `fallbacksFromModels: true` but no explicit `fallbacks`; the candidate builder may derive candidates while `fallbackConfigured` remains false.

## Solution Assessment

No, not as currently patched. The opt-in config direction is reasonable, but the safest implementation must preserve existing shared model fields and make the same derived fallback state visible to both candidate construction and the embedded runner failover gate.
