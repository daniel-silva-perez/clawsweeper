## Summary

Keep open. Current main does not implement the PR's intended gateway `chat.history` filter, and the proposed branch is a narrow, source-consistent fix for transcript-only `delivery-mirror` and `gateway-injected` assistant rows. I did not find a definite blocking correctness or security issue in the diff, though direct `chat.history` regression coverage would make the change stronger.

## What This Changes

The PR promotes the transcript-only OpenClaw assistant-message predicate into shared chat content helpers, reuses it in pi-embedded subscription handling, filters matching rows from gateway `chat.history`, and adds shared predicate unit tests.

## Best Possible Solution

Keep the fix at the gateway history boundary for remote/WebChat consumers, add a focused `chat.history` regression with transcript-only rows, and decide whether the same shared predicate should also be applied to the embedded local TUI history pipeline for parity.

## Reproduction Assessment

Yes. A source-level reproduction is clear: persist an assistant row tagged `provider: "openclaw"` with `model: "delivery-mirror"` or `"gateway-injected"`, then request `chat.history`; current main projects that row as a normal assistant message because no transcript-only filter runs in the handler.

## Solution Assessment

Yes for gateway `chat.history`: a server-side filter before slicing/sanitizing is the narrowest maintainable fix for WebChat and GatewayChatClient consumers. The safer follow-up is to add direct handler coverage and consider reusing the predicate in the embedded local TUI history path if that surface must share the same display contract.
