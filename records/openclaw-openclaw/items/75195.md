## Summary

Keep open. Current main still sends `delivery-mirror` and `gateway-injected` assistant transcript entries through `chat.history`, while existing agent/replay paths already treat them as transcript-only. The PR’s gateway-boundary filter is consistent with that contract, and I found no blocking correctness or security findings in the diff.

## What This Changes

The PR exports a shared predicate for transcript-only OpenClaw assistant messages, reuses it in pi-embedded subscriber handling, filters those messages from `chat.history`, and adds predicate coverage in shared tests.

## Best Possible Solution

Land a gateway-side `chat.history` filter backed by a shared predicate and regression tests, so TUI, web chat, and future chat clients receive the same display-normalized transcript view without duplicating client-side filters.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a real assistant reply followed by an assistant row tagged `provider: "openclaw"` and `model: "delivery-mirror"` or `"gateway-injected"` into a session transcript and call `chat.history`; current main reads and projects those rows without a transcript-only filter.

## Solution Assessment

Yes. Filtering once at the gateway boundary before slicing, sanitization, and byte caps is the narrowest maintainable fix for all `chat.history` consumers, and the shared predicate preserves the existing pi-embedded subscriber behavior.
