## Summary

Current main still appears to miss the requested runtime wiring. The direct subagent announce path calls the gateway `agent` method with the requester session and channel hints, but `agentCommandInternal` resolves the execution model from the agent default plus stored/explicit overrides only. The normal auto-reply/status paths do use `resolveChannelModelOverride`, and docs say `channels.modelByChannel` should apply when no session model override exists, so this issue remains actionable rather than stale, duplicate, or already implemented.

## Best Possible Solution

Keep this issue open and fix the shared gateway/agent ingress model-selection path. The implementation should resolve `channels.modelByChannel` from the requester session/channel context when there is no stored or explicit model override, preserve `explicit > stored > channel > default`, use the same alias and allowlist checks as existing model selection, and add targeted regression coverage for direct subagent announce delivery plus explicit/stored override precedence.

## Risks / Open Questions

- Users with channel-pinned models can still get announce runs on the agent default model, causing fallback latency or failures when the default is unsuitable for announce prompts.
- A fix should preserve override priority and model allowlist behavior; applying channel overrides too late or without alias/allowlist checks could regress explicit `/model` or trusted per-run overrides.
