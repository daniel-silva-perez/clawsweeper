## Summary

Current main still appears to have the reported bug. The direct subagent announce path forwards channel/session routing into the Gateway, but the shared agent ingress model-selection path still chooses only the agent default, then stored session override, then explicit run override. It does not call the existing channel model override resolver, while normal reply and status paths do. The prior ClawSweeper review that kept this open at c0ec58 remains valid against e476523.

## Best Possible Solution

Keep this issue open and fix the shared Gateway/agent ingress model-selection path. The implementation should resolve channels.modelByChannel from the requester session and channel context for announce/direct ingress runs when no explicit or stored session model override exists, preserve explicit > stored > channel > default precedence, reuse the existing alias and allowlist checks, and add targeted regression coverage for direct subagent announce delivery plus explicit/stored override precedence.

## Risks / Open Questions

- Users with channel-pinned models can still get subagent announce runs started on the agent default model, causing avoidable fallback latency or failures when the default is unsuitable.
- A fix should preserve explicit and stored session override precedence, alias resolution, and agent model allowlist behavior; applying channel config as an unchecked per-run override could regress model authorization boundaries.
