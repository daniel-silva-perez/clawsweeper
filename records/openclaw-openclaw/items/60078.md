## Summary

Keep open. Current main still misses the requested runtime wiring: subagent announce delivery calls the Gateway `agent` method with channel/session routing context but no provider/model override, and the shared ingress model-selection path still chooses default, stored session override, then explicit run override without consulting `channels.modelByChannel`. The existing channel override resolver is used by normal reply/status paths and docs promise that contract, so this remains a valid, narrow bug rather than stale, duplicate, or implemented.

## What This Changes

Subagent announce delivery should use the parent/requester session channel's `channels.modelByChannel` model when no explicit or stored session model override exists.

## Best Possible Solution

Add channel-model resolution to the shared Gateway/agent ingress model-selection path, using requester session/channel context when no explicit run override or stored session model override exists. Preserve `explicit > stored session > channel config > agent default`, reuse existing alias and allowlist handling, and add focused regression tests for direct subagent announce plus explicit/stored precedence.

## Reproduction Assessment

Yes. The reporter's configured Discord channel-pin scenario has a high-confidence static reproduction path on current main: direct subagent announce enters `agentCommandFromIngress` without provider/model, and that path never calls the channel override resolver before running the model fallback chain.

## Solution Assessment

Yes. Fixing the shared ingress model-selection path is the narrowest maintainable solution because it covers direct announce and other Gateway agent ingress calls consistently without adding announce-specific model injection that could drift from reply/status behavior.
