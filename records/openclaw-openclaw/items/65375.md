## Summary

Keep this open. Current main still blocks the LINE webhook HTTP 200 on event processing in the reusable Node handler, the Express-style middleware, and the current shared-path gateway route; tests also preserve the blocking/error-before-ACK behavior. The related fix PR was closed unmerged, so there is no implementation to close against.

## What This Changes

The issue requests that LINE webhooks acknowledge verified requests with HTTP 200 immediately, then process events asynchronously so slow replies can be delivered through push/fallback handling instead of holding the webhook response open.

## Best Possible Solution

Implement the fix inside the LINE plugin: after raw body read, signature validation, target resolution, and JSON parsing succeed, send the 200 response immediately and run event processing fire-and-forget with runtime error reporting. Preserve pre-auth limits, signature-integrity behavior, account disambiguation, in-flight limiter semantics, and existing reply/push fallback behavior, then update tests to prove ACK-before-processing for all three handler paths.

## Reproduction Assessment

Yes. A signed POST whose `handleWebhook` or `onEvents` promise does not resolve reproduces the source-level issue: current tests already show headers are not sent until processing completes, and failures currently produce 500 instead of an already-sent 200.

## Solution Assessment

The requested direction is the right narrow fix, but the literal two-location patch in the issue is incomplete for current main because `extensions/line/src/monitor.ts` has a shared-path route that also awaits event processing before ACK. A complete fix should cover the reusable Node handler, Express middleware, and monitor route together.
