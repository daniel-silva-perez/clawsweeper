## Summary

Keep this open. Current main has the adjacent SDK Retry-After wait cap from #68474 and per-provider request timeouts, but it still does not expose a typed per-provider or per-candidate model API retry-count budget. The proposed agent-level and provider-level fields would be rejected by strict config schemas, and the OpenAI/Azure SDK client construction still forwards timeout/fetch only, not maxRetries.

## Best Possible Solution

Keep this issue open and implement a typed model-provider retry budget, preferably under a versioned provider request config such as models.providers.<id>.request.retry.maxRetries or a clearly documented models.providers.<id>.maxRetries. Plumb it into OpenAI/Azure and Anthropic Vertex SDK construction where supported, define no-op or explicit behavior for direct fetch transports, update config schema/help/docs for model failover and retry behavior, and add regression tests proving a low retry budget advances fallback quickly on 429/529-style failures.

## Risks / Open Questions

- Provider SDKs do not all expose identical retry-count APIs or semantics, so an implementation should verify OpenAI/Azure, Anthropic Vertex, direct Anthropic/Google transports, and OpenAI-compatible custom providers before standardizing the config shape.
- OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS partially addresses long Retry-After stalls and could be confused with retry-count control; docs should clearly separate wait caps from retry budgets.
- Direct fetch transports may currently have no SDK-internal retry loop, so a provider-level option should avoid accidentally adding retries where the transport is intentionally single-attempt.
