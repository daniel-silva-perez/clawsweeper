## Summary

Keep open. Current main has adjacent failover improvements and a Stainless SDK Retry-After wait cap, but it does not expose the requested configurable per-provider or per-candidate model API retry budget. The strict model-provider config schema has no maxRetries field, and the OpenAI/Azure SDK construction path only passes timeout/fetch options, not retry-count options.

## Best Possible Solution

Keep the issue open and implement a typed model-provider retry budget, likely under models.providers.<id>.request or a dedicated models.providers.<id>.retry field. Plumb it through the OpenAI/Azure SDK client options and any provider-owned transports that perform retries, document the interaction with OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS and model fallback, and add schema plus transport tests proving low maxRetries advances fallback quickly on 429/529-style failures.

## Risks / Open Questions

- Current retry-after caps and auth-profile rotation settings partially reduce the same symptom, so a fix should avoid conflicting semantics and clearly distinguish SDK retry budget from fallback/profile rotation policy.
- Provider SDKs do not necessarily share one retry option shape; implementation should verify the exact upstream APIs for the OpenAI/Azure, Anthropic/Stainless, Google, and any custom transport paths before standardizing config names and defaults.
