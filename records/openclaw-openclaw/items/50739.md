## Summary

Current main still does not provide a priority or queue-bypass path for `openclaw system event`; the CLI, wake protocol, gateway handler, cron wake path, and heartbeat runner continue to route system events through the best-effort heartbeat/session-lane flow. The KeepMyClaw comments are useful recovery context, but external snapshots do not solve the requested immediate in-session alert path.

## What This Changes

Add a high-priority or bypass-queue mode to `openclaw system event --mode now` so operator alerts can surface promptly during provider outages or saturated session lanes.

## Best Possible Solution

Design an explicit high-priority alert contract that can notify the active operator immediately, preferably separating UI/operator alert delivery from prompt-visible transcript mutation unless a safe ordered injection protocol is added and covered across CLI, Gateway protocol, heartbeat/session runtime, docs, and security-sensitive formatting.

## Risks / Open Questions

- A direct prompt-visible bypass could race an active streaming turn or corrupt transcript ordering; current heartbeat code explicitly avoids busy session lanes.
- Because the wake schema allows unknown metadata, callers can send a `priority` field today and have it validate while current main silently ignores it.
- System events become trusted prompt-visible `System:` lines by default, so any bypass path needs spoofing and prompt-injection review.
- The product shape is unresolved: immediate operator/TUI notification may need to be separate from prompt-visible transcript injection.
