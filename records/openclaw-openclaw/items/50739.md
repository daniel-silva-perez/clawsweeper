## Summary

Keep open. Current main still has no priority or bypass mode for `openclaw system event`; the CLI sends only `{ mode, text }` to the wake RPC, the wake path enqueues the event and requests a heartbeat, and the heartbeat runner intentionally skips both the main queue and the resolved session lane when requests are in flight. Ryce's KeepMyClaw comments are useful context for snapshot/recovery during outages, but they do not replace the requested in-session alert delivery path.

## Best Possible Solution

Keep this issue open for maintainer design. The likely product direction is an explicit high-priority alert contract that can notify the active TUI/operator immediately without unsafely mutating the session transcript; prompt-visible system-event injection can still respect session-lane ordering unless a safe append/injection protocol is designed and tested.

## Risks / Open Questions

- A direct transcript bypass could race an active agent run or corrupt session state, which is why the current heartbeat path deliberately avoids interrupting a busy session lane.
- The wake schema accepts unknown fields for forward compatibility, so callers can send `priority` today and have it silently ignored unless the CLI/RPC/runtime contract is made explicit.
- System events can become prompt-visible `System:` or `System (untrusted):` lines, so any immediate alert path needs to preserve trust downgrades and avoid creating a spoofing or prompt-injection shortcut.
