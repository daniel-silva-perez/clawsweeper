## Summary

Keep open. Current main still does not implement a priority or queue-bypass mode for `openclaw system event`; the CLI sends only mode/text, the wake RPC ignores any extra metadata, and `mode: "now"` still wakes the heartbeat path that deliberately skips busy main/session lanes. The KeepMyClaw discussion is useful recovery context, but external snapshots do not solve the requested immediate in-session alert path.

## What This Changes

Request adding a high-priority or queue-bypass mode to `openclaw system event --mode now` so urgent system alerts can surface reliably during congested session lanes.

## Best Possible Solution

Design an explicit high-priority alert contract that can notify the active operator immediately, preferably separating UI/operator alert delivery from prompt-visible transcript mutation unless a safe ordered injection protocol is added and tested across CLI, Gateway protocol, heartbeat/session runtime, docs, and security-sensitive formatting.

## Risks / Open Questions

- A direct prompt-visible bypass could race an active streaming turn or corrupt transcript ordering; current heartbeat code explicitly avoids that by skipping busy session lanes.
- Because the wake schema allows unknown metadata, a caller can send a `priority` field today and have it validate but be ignored unless the protocol and handler make priority explicit.
- System events become trusted prompt-visible `System:` lines by default, so any bypass path needs a spoofing and prompt-injection review before implementation.
- The product shape is unresolved: immediate operator/TUI notification may need to be separate from prompt-visible transcript injection.
