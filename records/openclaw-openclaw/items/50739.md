## Summary

Keep open. Current main still routes `openclaw system event --mode now` through the wake RPC, queued system events, and heartbeat delivery; it has no CLI flag, protocol field, gateway handling, or session-stream bypass for urgent alerts. The KeepMyClaw discussion is useful recovery context, but snapshots do not provide the requested immediate in-session alert surface.

## What This Changes

The issue requests a high-priority or bypass-queue mode for `openclaw system event --mode now` so degraded-system alerts can appear immediately during congested session lanes.

## Best Possible Solution

Define a first-class urgent system alert contract that delivers immediate operator-visible notifications, with transcript or prompt mutation handled through a separately ordered and trust-reviewed path. The implementation should align the CLI, Gateway protocol, wake/cron path, heartbeat/session handling, TUI behavior, docs, and regression tests.

## Risks / Open Questions

- A prompt-visible bypass could race an active streaming turn or corrupt transcript ordering; current heartbeat code explicitly skips busy session lanes to avoid that case.
- The existing wake schema accepts unknown metadata, so callers can send a `priority` field today and have it validate while current main silently ignores it.
- System events become trusted `System:` prompt-visible lines by default, so an urgent bypass needs explicit spoofing and prompt-injection review.
- The product shape is unresolved: immediate operator/TUI notification may need to be separate from transcript or prompt mutation.
