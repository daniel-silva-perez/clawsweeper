## Summary

Keep open. Current main still lacks a priority or queue-bypass mode for `openclaw system event`; the command sends only `mode` and `text`, the wake path enqueues a system event, and heartbeat delivery still intentionally skips busy main/session lanes.

## What This Changes

Add a priority or bypass-queue mode for `openclaw system event --mode now` so in-session alerts can surface reliably during congested or failing LLM request lanes.

## Best Possible Solution

Keep this open for maintainer design. The likely direction is an explicit high-priority alert contract that can notify the active TUI/operator immediately without unsafely mutating an active transcript, with prompt-visible system-event injection remaining lane-ordered unless a safe append/injection protocol is designed and tested.

## Risks / Open Questions

- A direct transcript bypass could race an active streaming turn or corrupt session ordering; current heartbeat code deliberately skips busy session lanes to avoid that.
- The wake schema accepts unknown metadata, so callers can send a `priority` field today and have it silently ignored unless the protocol and handler make priority explicit.
- System events can become prompt-visible `System:` lines, so any immediate alert path must preserve trusted/untrusted formatting and avoid creating a spoofing or prompt-injection shortcut.
- There may be two product shapes here: immediate operator/TUI notification and prompt-visible transcript injection. The issue needs maintainer direction on whether those should be separate contracts.
