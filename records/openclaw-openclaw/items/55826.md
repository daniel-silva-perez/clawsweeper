## Summary

Keep open. The audit logging goal is valid and still not implemented on current main, but this PR is not merge-ready because the proposed pipeline log can misattribute deny-list removals to allow-list config keys and the sandbox audit log is emitted from a shared formatter path that can run more than once for the same blocked call.

## What This Changes

The PR adds INFO-level audit logs for tool-policy pipeline filtering and sandbox blocked-tool decisions, plus unit coverage for pipeline audit logging.

## Best Possible Solution

Implement audit logging at the policy decision boundary with matched allow/deny rule and source-key attribution, and keep user-facing error-formatting helpers side-effect-free or explicitly deduped per blocked call.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a default pipeline step with tools.deny can be logged under a tools.allow label, and an unknown-tool sandbox error rendered through formatAssistantErrorText can reach the formatter-side audit path repeatedly.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because attribution must come from the matched rule/source and sandbox audit emission should happen once at enforcement time or through an explicit dedupe path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Report the actual allow or deny key:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The audit message interpolates `step.label`, but default policy steps are labeled `*.allow` even when a `deny` list removes a tool. With a normal `tools.deny` rule, the new log can point operators at `tools.allow`, so derive or log the matched rule key instead.
  - confidence: 0.91
- **[P2] Move sandbox audit logging out of formatting:** `src/agents/sandbox/runtime-status.ts:160-162`
  - body: `formatSandboxToolPolicyBlockedMessage` is reached from shared error-rendering paths. Logging there can produce repeated identical INFO audit entries for one blocked tool call, so emit the audit at the enforcement boundary or dedupe it per run/tool.
  - confidence: 0.87
