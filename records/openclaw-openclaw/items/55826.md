## Summary

Keep open. Current main still lacks tool-policy rule attribution, and this PR addresses a real gap linked to #55801, but it should not merge as-is because the pipeline audit can point operators to the wrong allow/deny key and the sandbox audit is emitted from an error formatter that has multiple rendering call sites.

## What This Changes

The PR adds INFO-level audit logs for tool-policy filtering and sandbox-blocked tool calls, plus unit coverage for pipeline removal logging and spy cleanup.

## Best Possible Solution

Keep the PR open and revise it so pipeline audit messages derive the actual allow/deny source for each removed tool, while sandbox audit logging is emitted once at the policy-enforcement event boundary or deduped per run/tool. Add focused tests for default `tools.deny` attribution and for avoiding repeated sandbox audit lines through repeated error formatting.

## Reproduction Assessment

Yes. A source-level reproduction is clear: run the default pipeline with `tools.deny` removing `browser` and the PR would log through the generic `tools.allow` label; separately, the sandbox formatter is reachable from several `formatAssistantErrorText` call sites for the same assistant error.

## Solution Assessment

No. The requested audit logging direction is maintainable, but this implementation is not the best form because it misattributes some pipeline removals and places sandbox audit side effects in display formatting rather than a single enforcement path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Report the actual allow or deny key:** `src/agents/tool-policy-pipeline.ts:171`
  - body: The new pipeline audit line uses `step.label`, but the default steps are labeled `tools.allow`/`agents.<id>.tools.allow` even when a `deny` list removes the tool. For a config like `tools.deny: ["browser"]`, the log would point operators at `tools.allow`, defeating the rule-attribution goal. Derive the matched allow/deny key for removed tools, or use a neutral policy-step label plus rule details.
  - confidence: 0.91
- **[P2] Move sandbox audit logging out of the formatter:** `src/agents/sandbox/runtime-status.ts:158-160`
  - body: `formatSandboxToolPolicyBlockedMessage` is called from shared assistant-error rendering and failover paths, so adding `logInfo` here can produce multiple identical audit entries for one blocked tool call. Emit the sandbox audit at the enforcement boundary, or dedupe it per run/tool, so audit counts stay meaningful.
  - confidence: 0.87
