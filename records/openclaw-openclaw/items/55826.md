## Summary

Keep this PR open. Current main still lacks tool-policy audit logging, and the PR addresses the linked operator need, but the sandbox audit log is added inside a user-facing error formatter that is called from multiple rendering/failover paths, so one blocked tool can produce duplicate INFO audit records. The prior Greptile mock teardown note was addressed by the PR's second commit.

## What This Changes

This PR adds INFO-level tool-policy audit logging for pipeline tool removals and sandbox blocked-tool guidance, with pipeline tests for deny, allow, and no-removal cases.

## Best Possible Solution

Land a scoped audit feature that emits rule attribution exactly once per policy decision: keep the pipeline removal attribution, and move or dedupe the sandbox blocked-tool audit so it is tied to a single enforcement/evaluation point with regression coverage for duplicate emission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Move sandbox audit logging out of the formatter:** `src/agents/sandbox/runtime-status.ts:161-163`
  - body: The new `logInfo` call lives in `formatSandboxToolPolicyBlockedMessage`, but that helper is reached through `formatAssistantErrorText` from runner, payload, lifecycle, and failover rendering paths. A single sandbox-blocked unknown-tool error can therefore emit several identical INFO audit records, making the audit trail and counts misleading. Emit this once from a policy/filtering/enforcement path, or dedupe it per run/tool.
