## Summary

Keep open. Current main still lacks the requested tool-policy audit logging and the PR is a useful implementation candidate for open #55801, but the patch is not merge-ready because the pipeline audit can misattribute deny-list removals to allow-list keys and the sandbox audit log is emitted from shared error formatting rather than a single policy-enforcement boundary.

## What This Changes

The PR adds INFO-level audit logs when tool-policy pipeline filtering removes tools and when sandbox policy blocks a tool, plus unit coverage for pipeline audit logging and logger spy cleanup.

## Best Possible Solution

Keep the PR open and revise it so pipeline audit entries report the actual matched allow or deny source for each removed tool, while sandbox audit logging is emitted once at the policy decision boundary or deduped per run/tool. Add regression coverage for default `tools.deny` attribution and for avoiding repeated sandbox audit lines through repeated error formatting.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with the PR applied, a default pipeline step using `tools.deny: ["browser"]` would log through the generic `tools.allow` label, and the sandbox formatter is reachable from multiple `formatAssistantErrorText` call sites for the same assistant error.

## Solution Assessment

No. The requested audit logging direction is maintainable, but this implementation is not the best fix because it misattributes some pipeline removals and puts sandbox audit side effects in display formatting rather than a single policy-enforcement event path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Report the actual allow or deny key:** `src/agents/tool-policy-pipeline.ts:171`
  - body: The new pipeline audit message uses `step.label`, but the default global, provider, agent, and group steps are labeled as `*.allow` even when their `deny` list removes a tool. With `tools.deny: ["browser"]`, operators would be pointed at `tools.allow`, which defeats the rule-attribution goal. Derive the matched allow/deny key for removed tools, or log a neutral step label plus rule details.
  - confidence: 0.91
- **[P2] Emit sandbox audit logs outside error formatting:** `src/agents/sandbox/runtime-status.ts:158-160`
  - body: `formatSandboxToolPolicyBlockedMessage` is called from shared assistant-error rendering paths, including runner, payload, failover, and lifecycle handling. Logging inside this formatter can produce repeated identical INFO audit entries for one blocked tool call, so emit the audit at the enforcement boundary or dedupe it per run/tool.
  - confidence: 0.87
