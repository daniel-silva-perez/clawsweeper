## Summary

Keep open. Current main still lacks rule-attributed tool-policy audit logging, and this open PR is a plausible implementation path for #55801, but the patch is not merge-ready because it can misattribute deny-list removals to allow keys and emits sandbox audit logs from a reusable error formatter.

## What This Changes

The PR adds INFO-level audit logs for tool-policy pipeline filtering and sandbox blocked-tool decisions, plus unit coverage for pipeline audit logging.

## Best Possible Solution

Implement audit logging at the policy decision boundary with matched allow/deny rule and source-key attribution, then keep user-facing formatting helpers side-effect-free or explicitly dedupe audit emission per run/tool.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass a default pipeline step with `tools.deny: ["browser"]` and the PR logs through the hard-coded `tools.allow` label; call `formatAssistantErrorText` more than once for the same sandbox unknown-tool error and the formatter path can re-emit the audit log.

## Solution Assessment

No. The audit direction is useful, but this patch is not the best fix because attribution should come from the matched rule/source and sandbox audit emission should happen once at enforcement time or through an explicit dedupe path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Report the actual allow or deny key:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The new pipeline audit message uses `step.label`, but default global, provider, agent, and group steps are labeled as `*.allow` even when a `deny` list removes the tool. With `tools.deny: ["browser"]`, operators would be pointed at `tools.allow`, defeating the rule-attribution goal. Derive the matched allow/deny key or log neutral step plus rule details.
  - confidence: 0.91
- **[P2] Move sandbox audit logging out of formatting:** `src/agents/sandbox/runtime-status.ts:160-162`
  - body: `formatSandboxToolPolicyBlockedMessage` is reached from shared assistant-error rendering paths. Logging here can produce repeated identical INFO audit entries for one blocked tool call, so emit the audit at the enforcement boundary or dedupe it per run/tool.
  - confidence: 0.87
