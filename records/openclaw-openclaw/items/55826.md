## Summary

Keep open. The PR targets a real gap tracked by open #55801, and current main still lacks tool-policy audit logging, but the proposed patch is not merge-ready because it can misattribute deny-list removals and can duplicate sandbox audit lines from shared error formatting paths.

## What This Changes

The PR adds INFO-level audit logs for tool-policy pipeline filtering and sandbox blocked-tool decisions, plus unit coverage for pipeline audit logging.

## Best Possible Solution

Implement audit logging at the policy decision boundary with matched allow/deny rule and source key attribution, then keep formatting helpers side-effect-free or explicitly dedupe audit emission per run/tool. Add regression coverage for deny-key attribution and repeated error formatting.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with a default pipeline step containing `tools.deny: ["browser"]`, the PR would log through the generic `tools.allow` label, and repeated `formatAssistantErrorText` calls for one unknown-tool error would re-enter the sandbox formatter.

## Solution Assessment

No. The audit logging direction is maintainable, but this patch is not the best fix because attribution should come from the matched rule/source and sandbox audit emission should happen once at enforcement time or through an explicit dedupe path.

## Review Findings

  - confidence: 0.87
