## Summary

Keep open. Current main still has no trust-tier implementation, so the PR is not obsolete, but the proposed patch has a concrete security-facing correctness bug: omitted-format native plugins would be displayed as `content` trust even though the loader normalizes them to `openclaw` and treats them as native-capable. The branch is also dirty against current main and needs maintainer product/security review before this taxonomy becomes public CLI/docs metadata.

## What This Changes

The PR adds a `PluginTrustTier` taxonomy and resolver, stores trust tiers on plugin records, prints them in plugin CLI output, and documents/tests content, sandboxed, and native plugin tiers.

## Best Possible Solution

Land a rebased version only if maintainers want this public taxonomy, with trust tiers computed from the same normalized effective plugin format used in `PluginRecord`, docs that separate current capabilities from reserved future behavior, and tests for native, bundle, omitted-format fallback, loader records, and CLI list/info output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve trust tiers from the normalized plugin format:** `src/plugins/loader.ts:849-853`
  - body: The loader stores missing manifest formats as `openclaw`, but this new call passes raw `manifestRecord.format` into `resolveTrustTier`. For manifests without an explicit format, the record can show `format: openclaw` with `trustTier: content`, under-reporting a native-capable plugin in CLI/security metadata. Use the same effective format at both loader call sites and align the undefined-format docs/tests with that behavior.
