## Summary

Keep open: current main still has no trust-tier model, so the PR is not obsolete, but the proposed patch has a correctness and security-metadata issue because it resolves the new tier from the raw manifest format while plugin records normalize missing formats to `openclaw`. The PR needs maintainer product/security review plus a branch fix before merge.

## What This Changes

This PR adds plugin trust-tier types and resolution, stores the resolved tier on plugin records, shows it in plugin CLI output, and adds trust-tier documentation and tests.

## Best Possible Solution

If maintainers want this taxonomy, land a rebased implementation that computes tiers from the normalized effective plugin format, documents current capabilities separately from reserved future behavior, and covers native, bundle, omitted-format fallback, plugin list/inspect output, and loader record paths in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve trust tiers from the normalized plugin format:** `src/plugins/loader.ts:849-853`
  - body: The loader records omitted manifest formats as `openclaw`, but this new call passes raw `manifestRecord.format` into `resolveTrustTier`. For manifests without an explicit format, the registry can show `format: openclaw` with `trustTier: content`, under-reporting a native-capable plugin in CLI/security metadata. Use the same effective format at both loader call sites and update the undefined-format docs/tests to match.
  - confidence: 0.92
