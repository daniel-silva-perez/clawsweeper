## Summary

Keep open. Current main does not contain a trust-tier model, and the PR still contains a blocking security-facing metadata bug: omitted-format plugins are normalized to native format in records today, while the PR resolves the new tier from the raw undefined manifest format and reports content.

## What This Changes

The PR adds plugin trust-tier types and resolution, stores a trust tier on plugin records, prints it in plugin CLI output, and adds trust-tier docs and tests.

## Best Possible Solution

Land a rebased version only if maintainers want this public taxonomy, with tiers computed from the same normalized effective plugin format used in PluginRecord, docs that distinguish current capabilities from reserved future behavior, and tests covering native, bundle, omitted-format fallback, loader record paths, and CLI list/info output.

## Reproduction Assessment

Yes for the blocking PR bug. A manifest without format flows through current loader record creation, createPluginRecord normalizes it to openclaw, and the PR's raw resolveTrustTier call would classify that same record as content.

## Solution Assessment

No, the proposed patch is not the best way to solve the issue as written. The safer narrow fix is to resolve trust from the effective normalized format at both loader call sites and align the undefined-format docs/tests with that contract before landing security-facing metadata.
