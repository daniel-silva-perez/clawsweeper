## Summary

Keep open. Current main has no trust-tier implementation, and the PR introduces security-facing plugin metadata with a concrete mismatch: omitted-format plugins are represented as native `openclaw` records today, but the PR resolves their new trust tier from the raw undefined manifest format and would report them as `content`.

## What This Changes

The PR adds plugin trust-tier types and resolution, stores a tier on plugin records, prints it in plugin CLI output, and documents content, sandboxed, and native tiers.

## Best Possible Solution

Land a rebased version only if maintainers want this public taxonomy, with tiers computed from the same normalized effective plugin format used in `PluginRecord`, docs that distinguish current capabilities from reserved future behavior, and tests covering native, bundle, omitted-format fallback, loader record paths, and CLI list/info output.

## Reproduction Assessment

Yes for the blocking PR bug. A manifest without `format` flows through current loader record creation, `createPluginRecord` normalizes it to `openclaw`, and the PR's raw `resolveTrustTier` call would classify that same record as `content`.

## Solution Assessment

No, the proposed patch is not the best way to solve it as written. The narrow fix is to resolve trust from the effective normalized format at both loader call sites and align docs/tests with that contract before landing the security-facing metadata.
