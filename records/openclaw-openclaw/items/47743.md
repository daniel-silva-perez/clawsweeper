## Summary

Keep open: current main does not implement plugin trust tiers, so this PR is not obsolete. The branch should not merge as-is because the proposed trust metadata can under-report missing-format native plugins as content-tier, and the public security taxonomy needs maintainer review before becoming CLI/docs contract.

## What This Changes

This PR adds plugin trust-tier types and resolution, stores the tier on plugin records, prints it in plugin CLI output, and adds trust-tier docs and tests.

## Best Possible Solution

If maintainers want this taxonomy, land a rebased version that computes tiers from the normalized effective plugin format, defines the supported public CLI/JSON/status contract, documents only current capabilities plus clearly reserved future behavior, and covers native, bundle, missing-format fallback, list/info output, and inspect/status behavior in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve trust tiers from the normalized plugin format:** `src/plugins/loader.ts:849`
  - body: The loader stores missing formats as `openclaw` via `createPluginRecord`, but this new call passes raw `manifestRecord.format` into `resolveTrustTier`. For manifests without an explicit format, the resulting record would show `format: openclaw` with `trustTier: content`, under-reporting a native-capable plugin in CLI/security metadata. Use the same effective format at both call sites and align the undefined-format docs/tests with that behavior.
  - confidence: 0.92
