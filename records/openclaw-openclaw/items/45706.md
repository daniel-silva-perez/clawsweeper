## Summary

Current main still leaves this strict Mindie/OpenAI-compatible 422 report actionable. The supported generic compat fields exist and explicit streaming-usage overrides are preserved, but the Chat Completions payload builder still sends `stream_options.include_usage` on every streaming request, so strict backends cannot opt out of that request parameter.

## What This Changes

The issue requests a fix for Mindie-hosted DeepSeek OpenAI-compatible requests failing with HTTP 422 parameter validation errors when OpenClaw sends unsupported request fields.

## Best Possible Solution

Make the OpenAI Chat Completions transport honor `compat.supportsUsageInStreaming: false` by omitting `stream_options`, preserve usage reporting for native or explicitly compatible endpoints, and keep additional strict-provider handling behind generic compat/schema-backed request shaping rather than Mindie-specific IDs.
