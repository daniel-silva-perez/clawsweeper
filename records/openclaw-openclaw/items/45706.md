## Summary

Keep open: current main still has the reported strict OpenAI-compatible provider failure path. The config schema and normalizer now support generic compat flags, but the Chat Completions payload builder still sends `stream_options.include_usage` unconditionally, so a Mindie-style backend has no effective opt-out for that request parameter.

## What This Changes

The issue reports a local Mindie DeepSeek OpenAI-compatible endpoint failing OpenClaw agent turns with HTTP 422 because strict request-parameter validation rejects OpenClaw's payload.

## Best Possible Solution

Add generic OpenAI-compatible request shaping in the Chat Completions transport: omit `stream_options` when `compat.supportsUsageInStreaming` is explicitly false, keep streamed usage for known or explicitly compatible providers, and document the supported compat key for strict local backends. Leave message-key sanitization to #50374 unless the same patch needs a small shared helper.

## Reproduction Assessment

Yes for the OpenClaw-side bug: construct a custom `openai-completions` model with `compat.supportsUsageInStreaming: false` and inspect `buildOpenAICompletionsParams()`, which still includes `stream_options.include_usage`. The exact remote 422 requires a strict Mindie-style backend, but the reported logs and current payload builder align.

## Solution Assessment

Yes, a generic compat-gated request-shaping fix is the narrowest maintainable solution. Provider-specific Mindie hardcoding or a blanket removal of `stream_options` would either put owner-specific policy in core or regress compatible local/proxy endpoints.
