## Summary

Keep open. Current main still leaves the Mindie/OpenAI-compatible 422 report actionable because the config schema can express strict-backend compatibility, and normalization preserves `compat.supportsUsageInStreaming: false`, but the Chat Completions transport still sends `stream_options.include_usage` on every streaming `openai-completions` request.

## What This Changes

The issue reports a Mindie-hosted DeepSeek/OpenAI-compatible local model failing with HTTP 422 because the backend rejects unsupported request parameters from OpenClaw.

## Best Possible Solution

Honor the existing generic compat surface in the Chat Completions transport: omit `stream_options` when `compat.supportsUsageInStreaming` is explicitly false, keep streamed usage enabled for known or explicitly compatible backends, and document the supported compat key for strict local providers. Leave broader message-key sanitization to #50374 unless a small shared helper is needed.

## Reproduction Assessment

Yes for the OpenClaw-side bug: construct a custom `openai-completions` model with `compat.supportsUsageInStreaming: false` and inspect `buildOpenAICompletionsParams()`, which still includes `stream_options.include_usage`. The exact remote 422 requires a strict Mindie-style backend, but the reported logs and current payload builder align.

## Solution Assessment

Yes. A generic compat-gated request-shaping fix is the narrowest maintainable solution; hardcoding Mindie or globally removing `stream_options` would either put owner-specific policy in core or regress compatible local/proxy endpoints.
