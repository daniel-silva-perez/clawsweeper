## Summary

Current main still leaves this Mindie/OpenAI-compatible 422 report actionable. OpenClaw now has schema-backed generic compat flags and preserves explicit streaming-usage overrides, but the Chat Completions request builder still sends `stream_options.include_usage` on every streaming request, so a strict backend cannot opt out of that unsupported parameter. The related #50374 is relevant strict-provider cleanup, but it tracks message-key sanitization rather than this request-parameter shaping gap.

## What This Changes

Mindie/OpenAI-compatible local model requests return HTTP 422 when the strict backend rejects OpenClaw request parameters it does not support.

## Best Possible Solution

Add generic OpenAI-compatible request shaping in the Chat Completions transport: honor `compat.supportsUsageInStreaming: false` by omitting `stream_options`, keep streamed usage for known or explicitly compatible providers, and rely on existing `compat.supportsTools: false` or a documented schema-backed compat flag for tool-related payload changes without hardcoding Mindie IDs.
