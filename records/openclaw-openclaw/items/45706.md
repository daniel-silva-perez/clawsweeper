## Summary

Current main still leaves this Mindie/OpenAI-compatible 422 report actionable: the config schema and normalizer support generic compat flags, but the Chat Completions request builder still emits `stream_options.include_usage` unconditionally for streaming requests. The related #50374 is relevant strict-provider context, but it tracks extra message keys rather than this streaming/tool-choice request-shaping gap, so this issue should remain open for a focused runtime fix.

## What This Changes

Report a Mindie-hosted DeepSeek OpenAI-compatible Chat Completions backend returning HTTP 422 because the backend rejects unsupported OpenAI request parameters, with a prior workaround rejected by config validation.

## Best Possible Solution

Add a generic OpenAI-compatible request-shaping fix in the OpenAI Chat Completions transport: honor `compat.supportsUsageInStreaming: false` by omitting `stream_options`, keep streamed usage for providers that explicitly or manifest-detect support it, and ensure the strict-endpoint tool mitigation is covered by the existing `compat.supportsTools: false` path or a documented schema-backed compat flag without hardcoding Mindie IDs.

## Repair Work Prompt

Fix this issue as a narrow OpenAI-compatible Chat Completions request-shaping repair. Current main resolves `compat.supportsUsageInStreaming`, and config/model normalization preserve explicit false, but `buildOpenAICompletionsParams()` still always sends `stream_options: { include_usage: true }`. Update the generic transport so `compat.supportsUsageInStreaming: false` omits `stream_options`, while known/manifest-compatible providers and explicit true overrides still include usage.
