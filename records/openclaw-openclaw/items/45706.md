## Summary

Keep open: current main still leaves a concrete OpenAI-compatible request-shaping gap for strict Mindie-style backends. The schema and model normalizer accept generic compatibility flags, but the Chat Completions request builder still sends `stream_options.include_usage` on every streaming request, so `compat.supportsUsageInStreaming: false` cannot suppress the unsupported parameter.

## What This Changes

The issue asks OpenClaw to prevent Mindie/OpenAI-compatible runs from failing with HTTP 422 by avoiding unsupported request parameters through generic compatibility handling.

## Best Possible Solution

Make the OpenAI Chat Completions transport honor `compat.supportsUsageInStreaming: false` by omitting `stream_options`, keep streamed usage for known or explicitly compatible providers, and keep any tool-related mitigation on schema-backed generic compat flags rather than Mindie-specific IDs.
