## Summary

Keep open: current main still sends `stream_options.include_usage` unconditionally for OpenAI-compatible streaming requests, so strict Mindie-style backends do not have a working opt-out for this reported 422 path. The related #50374 is relevant strict-provider cleanup, but it tracks message-key sanitization rather than request-parameter shaping.

## What This Changes

The issue asks OpenClaw to make strict Mindie/OpenAI-compatible local DeepSeek deployments stop failing with HTTP 422 request-parameter validation errors.

## Best Possible Solution

Add generic OpenAI-compatible request shaping that lets an explicitly configured strict backend omit `stream_options` without hardcoding Mindie, while preserving streamed usage for native, known-compatible, and default local/proxy endpoints that rely on #68746. Document the supported compat key and cover the opt-out with regression tests.
