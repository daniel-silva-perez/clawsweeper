## Summary

Keep open: current main still has the reported strict OpenAI-compatible provider failure path. OpenClaw now accepts generic `compat.supportsUsageInStreaming` and `compat.supportsTools`, but `buildOpenAICompletionsParams()` still sends `stream_options.include_usage` on every streaming Chat Completions request, so a Mindie-style backend has no schema-backed opt-out for that unsupported request parameter. The related #50374 is adjacent strict-provider cleanup for message keys, not this top-level request-parameter issue.

## What This Changes

The issue requests support for a local Mindie/DeepSeek OpenAI-compatible provider that rejects OpenClaw requests with HTTP 422 because unsupported OpenAI request parameters are still sent.

## Best Possible Solution

Add generic OpenAI-compatible request shaping in the Chat Completions transport: honor `compat.supportsUsageInStreaming: false` by omitting `stream_options`, preserve streamed usage for known or explicitly compatible providers, and document the supported compat key for strict backends. Keep this out of Mindie-specific code and leave message-key sanitization to #50374.

## Reproduction Assessment

Yes for the OpenClaw-side bug: construct a custom `openai-completions` model with `compat.supportsUsageInStreaming: false` and inspect `buildOpenAICompletionsParams()`, which still includes `stream_options.include_usage`. The exact 422 requires a strict Mindie-style backend, but the reported logs and current payload builder align.

## Solution Assessment

Yes, a generic compat-gated request-shaping fix is the narrow maintainable solution. Provider-specific Mindie hardcoding or blanket removal would either add owner-specific policy to core or regress compatible local/proxy endpoints.
