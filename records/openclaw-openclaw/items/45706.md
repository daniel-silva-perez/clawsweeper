## Summary

Keep open. Current main still has an actionable OpenAI-compatible/Mindie request-shaping bug: the config schema can express strict-backend compatibility and normalization preserves an explicit streaming-usage opt-out, but the Chat Completions transport still sends `stream_options.include_usage` on every streaming `openai-completions` request. The issue discussion and related reports point to strict OpenAI-compatible backends rejecting unsupported request fields, and no open PR matching this issue or the related strict-provider work was found in the current public issue search.

## What This Changes

The issue requests a fix for Mindie/local OpenAI-compatible `openai-completions` runs failing with HTTP 422 because OpenClaw sends request parameters that the backend rejects.

## Best Possible Solution

Honor the existing generic compat surface in the Chat Completions transport: omit `stream_options` when `compat.supportsUsageInStreaming` is explicitly false, keep streamed usage enabled for known or explicitly compatible backends, and document the supported compat key for strict local providers. Leave broader transcript/message-key sanitization to #50374 unless the implementation naturally shares a small generic helper.

## Reproduction Assessment

Yes for the OpenClaw-side bug: construct a custom `openai-completions` model with `compat.supportsUsageInStreaming: false` and inspect `buildOpenAICompletionsParams()`, which still includes `stream_options.include_usage`. The exact remote 422 requires a strict Mindie-style backend, but the reported logs and current payload builder align.

## Solution Assessment

Yes: a generic compat-gated request-shaping fix is the narrowest maintainable solution. Hardcoding Mindie or globally removing `stream_options` would either put owner-specific policy in core or regress compatible local/proxy endpoints that use streamed usage accounting.
