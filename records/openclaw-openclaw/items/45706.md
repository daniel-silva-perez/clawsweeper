## Summary

Current main still leaves this Mindie/OpenAI-compatible 422 path unresolved: the config schema does not accept the suggested `supportsStreamOptions`/`supportsToolChoice` keys, `buildOpenAICompletionsParams` still sends `stream_options.include_usage` unconditionally, and proxy-like completions endpoints still default `tool_choice` to `auto` when tools are present.

## What This Changes

Fix a Mindie/DeepSeek OpenAI-compatible provider regression where strict request validation rejects OpenClaw’s outgoing Chat Completions payload with HTTP 422.

## Best Possible Solution

Keep this issue open and fix the generic OpenAI-compatible request-shaping path. The narrow first fix is to make `buildOpenAICompletionsParams()` respect `compat.supportsUsageInStreaming: false` by omitting `stream_options`, while preserving streamed usage for native and known-compatible endpoints and explicit true overrides. For Mindie-style `tool_choice` rejection, either add a narrowly documented compat flag with schema/tests or document `compat.supportsTools: false` as the supported mitigation.
