## Summary

Keep open. The current main branch still has an actionable OpenAI-compatible request-shaping gap: schema and normalization can express `compat.supportsUsageInStreaming: false`, but the Chat Completions payload builder still sends `stream_options.include_usage` on every streaming request. The issue discussion already narrowed this to strict Mindie/OpenAI-compatible endpoints rejecting unsupported fields, and #50374 tracks a related but different message-key sanitization problem.

## What This Changes

Fix HTTP 422 failures when a Mindie-hosted DeepSeek OpenAI-compatible backend rejects unsupported OpenAI request parameters during OpenClaw agent runs.

## Best Possible Solution

Honor the existing generic compat contract in the Chat Completions transport: omit `stream_options` when `compat.supportsUsageInStreaming` is explicitly false, preserve usage chunks for native or explicitly compatible endpoints, and document the supported compat key for strict local providers without adding Mindie-specific branching.

## Reproduction Assessment

Yes for the OpenClaw-side bug: construct a custom `openai-completions` model with `compat.supportsUsageInStreaming: false` and inspect `buildOpenAICompletionsParams()`, which still includes `stream_options.include_usage`. The exact remote 422 still needs a strict Mindie-style backend or its rejected-parameter response to prove end-to-end.

## Solution Assessment

Yes: a generic compat-gated payload fix is the narrowest maintainable solution. Globally removing `stream_options` would regress compatible local/proxy providers, while hardcoding Mindie would put provider-specific policy into the core transport.
