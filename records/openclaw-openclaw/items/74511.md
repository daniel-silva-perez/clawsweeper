## Summary

Keep open. Current main still has the stale configured inline-row bypass, and this PR targets the right resolver path, but the proposed check is incomplete because it does not pass the provider config/baseUrl context required by conditional manifest suppressions.

## What This Changes

The PR adds a suppression check before returning configured inline models with `api` in `resolveExplicitModelWithRegistry` and adds a unit test for configured `openai-codex/gpt-5.4-mini`.

## Best Possible Solution

Revise this PR so inline rows with `api` use the same context-aware suppression call as the existing resolver path, add a regression case for a conditional suppression such as Qwen Coding Plan, and keep catalog visibility changes coordinated with #73280.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass suppression context for inline matches:** `src/agents/pi-embedded-runner/model.ts:596-602`
  - body: The new inline-row check calls `shouldSuppressBuiltInModel` with only `provider` and `id`. Manifest suppressions can depend on configured provider API/baseUrl, so inline rows with `api` can still bypass conditional suppressions such as Qwen Coding Plan. Mirror the existing check below by passing `baseUrl: providerConfig?.baseUrl` and `config: cfg`.
  - confidence: 0.89
