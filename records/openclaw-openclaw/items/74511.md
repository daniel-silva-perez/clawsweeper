## Summary

Keep open. The PR addresses a real current-main bug, but the branch should not merge as-is because the new inline-path check treats an omitted config/baseUrl suppression query as an unconditional-only query. The manifest contract allows API-conditional suppressions, and the current matcher skips `providerConfigApiIn` when config is absent, so this patch can over-suppress valid configured inline rows for plugins that use provider-API-only conditions.

## What This Changes

The branch adds an inline-model suppression check in `resolveExplicitModelWithRegistry`, a regression test for configured `openai-codex/gpt-5.4-mini` rows with `api`, and a changelog entry.

## Best Possible Solution

Revise the fix so inline rows are rejected only for truly unconditional manifest retirements, or correct the matcher so missing config fails `providerConfigApiIn`; then cover both the stale OpenAI Codex inline row and a provider-API-only conditional suppression case while preserving the existing Qwen explicit-inline override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use an explicit unconditional suppression query:** `src/agents/pi-embedded-runner/model.ts:596-599`
  - body: The new inline-path check calls `shouldSuppressBuiltInModel({ provider, id })` without config/baseUrl and treats any match as a global retirement. Manifest suppressions can be conditional only on `when.providerConfigApiIn`, and the matcher currently skips that condition when config is missing, so valid explicit inline rows can be rejected for plugins that use an API-only conditional suppression. Please query only suppressions with no `when`, or make missing config fail `providerConfigApiIn` and add coverage for that conditional case.
  - confidence: 0.86
