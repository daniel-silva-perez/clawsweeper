## Summary

Keep open. Current main has partial protection for exact `${VAR}` templates on supported SecretRef-backed `openclaw.json` targets, but it does not implement the issue’s full requested behavior: there is no `ENV_REF` audit classification, `$VAR` shorthand is not accepted as a SecretRef, and related cleanup PR #64344 is still open rather than shipped. The earlier comment pointing at #61207 is useful context, but that PR is closed unmerged and its described guard is not present in current `collectEnvPlaintext()`.

## Best Possible Solution

Keep this issue open and route it through #64344 or a successor fix. The maintainer path is to decide the supported syntax (`${VAR}` only versus both `${VAR}` and `$VAR`), then update `src/config/types.secrets.ts`/`src/secrets/audit.ts`, add focused regression coverage in `src/secrets/audit.test.ts`, and align `docs/cli/secrets.md` plus `docs/reference/secretref-credential-surface.md` with the chosen audit classification.

## Risks / Open Questions

- Closing this as implemented would hide the still-unresolved `$VAR_NAME` shorthand request and the missing `ENV_REF`/safe-pattern audit distinction.
- The current code partially handles `${VAR}` as an inline SecretRef, so any eventual fix should avoid regressing that path or weakening plaintext detection for mixed strings such as `Bearer ${TOKEN}`.
