## Summary

Keep open. Current main still does not implement the requested safe env-reference classification: audit codes have no `ENV_REF` equivalent, `${VAR}` can be resolved before audit sees the authored value, `$VAR` is explicitly outside current config/env-template parsing, and related fixes remain open rather than shipped.

## What This Changes

The issue requests that `openclaw secrets audit --check` classify `$VAR` and `${VAR}` references in secret-bearing config/env surfaces as safe env references instead of `PLAINTEXT_FOUND`.

## Best Possible Solution

Keep this issue open until a canonical fix lands. The desired end state is a source-aware audit classification that distinguishes exact env-reference values from real plaintext, documents the supported syntax, preserves strict plaintext detection for mixed strings, and adds focused regression tests for config, auth-profile, and `.env` audit paths.

## Reproduction Assessment

Yes. The source path is high-confidence: with an authored `${OPENAI_API_KEY}` on an audited config target and that env var set, config loading substitutes the value before audit classification; with `$OPENAI_API_KEY`, current parsing leaves it as an ordinary string, so it remains eligible for `PLAINTEXT_FOUND`.

## Solution Assessment

No, current main is not the best solution because the observable false-positive class is still present and the report contract lacks a safe env-ref classification. The safer path is to review and land a focused source-config-aware fix through #64344, #54141, or a combined successor.
