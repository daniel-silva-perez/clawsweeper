## Summary

Current main still keeps the Control UI/WebChat pending image attachment payload as a base64 data URL in Lit reactive state before sending. The downstream Gateway `chat.send` RangeError fixes cited by related reports do not address this browser-side add/preview path, so this issue remains actionable.

## Best Possible Solution

Keep this open as a Control UI/WebChat attachment-add bug. The likely fix belongs under `ui/src/ui`: keep large pending File/Blob payloads out of Lit `@state`, render previews with object URLs or another non-reactive payload store, convert to base64 only when building `chat.send`, clean up object URLs, and add regression coverage for selecting, pasting, and dropping a large image.
