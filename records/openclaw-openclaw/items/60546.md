## Summary

Keep open. Current main still lacks Anthropic routing or an onboarding guardrail in the built-in microsoft-foundry provider, and the linked guardrail PR is open and unmerged, so closing this before that PR lands would drop the active fix path.

## What This Changes

The issue reports that the built-in microsoft-foundry provider can offer Claude/Anthropic Azure AI Foundry deployments but still configures, tests, and runs them through OpenAI-compatible Foundry routes.

## Best Possible Solution

Keep this issue open while maintainers review PR #72446 or an equivalent microsoft-foundry change. The narrow short-term endpoint is to filter or warn on Anthropic deployments during onboarding; full anthropic-messages routing can be handled as a separate provider enhancement if maintainers want native support.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the discovery path can select any succeeded deployment, then the persisted model config, runtime auth base URL, and connection test all resolve to OpenAI-compatible Foundry routes rather than Anthropic Messages routes.

## Solution Assessment

Yes for the immediate bug boundary. The linked guardrail is the narrowest maintainable short-term solution because native Anthropic Foundry support would require a larger runtime/auth/API branch beyond preventing the misleading successful setup.
