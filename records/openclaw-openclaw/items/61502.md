## Summary

Current main still has the reported Slack interaction routing gap: block-action payloads preserve `threadTs`, but the system-event session key is resolved without it, and Slack production code does not request a wake after enqueueing the event. The related narrow fix PR was closed unmerged, while the open broader wake PR has not made current main fixed.

## What This Changes

Slack interactive button clicks should reliably wake the correct agent session and preserve the originating Slack thread context.

## Best Possible Solution

Keep this open and land a focused Slack-owned fix that carries `threadTs` through system-event session resolution, wakes the correct thread-scoped session for actionable block actions, and preserves delivery context for replies to the same Slack thread. Broader heartbeat scheduler refactors should wait for separate evidence after the deterministic Slack routing failure is covered by tests.

## Reproduction Assessment

Yes. The supplied 2026.4.14 repro is concrete, and current main has a static reproduction path: simulate a Slack `block_actions` payload with `container.thread_ts` and observe that `enqueueSystemEvent` receives a base session key rather than the expected `:thread:<threadTs>` key.

## Solution Assessment

No current-main solution is present. The narrow maintainable direction is to extend the Slack system-event resolver and block-action enqueue path for thread-aware routing plus a targeted hook wake, using the same thread-key helper as inbound Slack messages rather than a broad core rewrite first.

## Repair Work Prompt

Fix Slack interactive reply/block-action clicks so they route to and wake the originating thread-scoped session. Current main parses `threadTs` from Slack block actions and includes it in the event payload, but `enqueueSlackBlockActionEvent` calls `resolveSlackSystemEventSessionKey` without it and the resolver cannot append `:thread:<threadTs>`. Implement the smallest Slack-owned fix: add an optional `threadTs` parameter to `SlackMonitorContext.resolveSlackSystemEventSessionKey`; resolve the same base route as today through bindings/fallback; when `threadTs` is present, apply `resolveThreadSessionKeys` with the same normalization used by inbound Slack message routing. Forward `parsed.threadTs` from block actions and any shared authorized system-event context path that handles thread-capable Slack events. Ensure actionable Slack reply actions request a hook wake for the resolved session if current main still lacks one, and attach/merge delivery context as needed so heartbeat/system-event turns can reply to the same Slack thread. Do not move Slack-specific behavior into core or add a broad heartbeat scheduler refactor unless a focused failing test proves it is required. Use PR #74336 as useful source context and check open PR #67751 for overlap before choosing the final wake approach. Add regression tests proving a `block_actions` payload with `container.thread_ts` enqueues/wakes `agent:...:thread:<ts>`, preserves absent/empty thread behavior, respects binding-routed DM/channel sessions, and does not change plugin interactive handler/exec approval behavior. Changelog only if the final change is user-facing release-note material; credit the non-bot repro/fix contributors per repo rules.
