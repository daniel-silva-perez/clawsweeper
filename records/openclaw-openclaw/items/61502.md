## Summary

Keep this issue open. Current main still does not reliably route Slack interactive reply clicks to the originating thread session: the handler carries `threadTs` in the event payload, but resolves the system-event session without `threadTs`, then only enqueues a passive system event. Existing tests still assert that non-thread-aware resolver call, and related open PRs #61374 and #67751 remain active candidate fixes rather than shipped behavior.

## Best Possible Solution

Keep this issue open as the canonical Slack interactive-reply routing tracker. The fix should make Slack block-action clicks resolve the correct thread-aware session key, trigger exactly one agent turn or wake for that session, preserve delivery context for replies back to the same Slack thread, and add focused regression coverage for reply buttons, non-reply block actions, duplicate clicks, and busy-session retry behavior. Review or close the related open PRs #61374 and #67751 as part of that decision.
