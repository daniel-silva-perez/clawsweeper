## Summary

Keep open. Current main still does not expose the concrete OpenRouter model or provider-reported cost for `openrouter/auto`: the transport initializes assistant metadata with the selected model, parses only token counters into local estimated cost, and the response footer accepts no resolved-model/provider-cost metadata. OpenRouter's docs say auto responses expose the actual model in the response `model` field and usage accounting includes response cost details, so the upstream data exists but is not threaded through OpenClaw yet. 

## What This Changes

Request to expose the concrete model and provider-reported cost chosen by OpenRouter when the selected model is `openrouter/auto`.

## Best Possible Solution

Capture OpenRouter's returned model and provider usage cost at the OpenAI-compatible transport boundary, preserve selected-versus-resolved model metadata through assistant/session records, and render the resolved auto label plus provider-reported cost on response usage surfaces with focused OpenRouter regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed an OpenAI-compatible OpenRouter final stream chunk containing a top-level resolved `model` and numeric `usage.cost`: current parsing ignores those fields and the footer can only render local token estimates.

## Solution Assessment

Yes. Keeping this open and fixing the transport-to-footer metadata path is the narrowest maintainable solution; the closed PR #43026 was useful prior art but did not land, and the related open issues track adjacent cost/model visibility rather than a shipped fix.
