## Summary

Keep this PR open. Current `main` still has the reported audit false-positive paths, and the PR is a relevant targeted fix, but the latest patch still leaves one dual-field auth-profile case misclassified as plaintext.

## What This Changes

The PR adds `$VAR` shorthand parsing for env SecretRefs, changes auth-store audit to skip plaintext findings for inline env refs, and adds parser/audit coverage for `$VAR`, `${VAR}`, and explicit-ref-plus-plaintext cases.

## Best Possible Solution

Update this PR so auth audit classifies the inline credential value independently from the explicit sibling ref, preserving `PLAINTEXT_FOUND` for real leftover plaintext while treating `$VAR` and `${VAR}` as env refs in all auth-profile states; coordinate separately with #64344 for config-summary/reporting behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Classify inline env refs before plaintext fallback:** `src/secrets/audit.ts:303-305`
  - body: `resolveSecretInputRef` returns `inlineRef: null` when `keyRef`/`tokenRef` is present, so this guard only skips the plaintext fallback for env refs without an explicit sibling ref. A profile with `keyRef` plus `key: "$OPENAI_API_KEY"` or `"${OPENAI_API_KEY}"` still reaches `isNonEmptyString` and reports `PLAINTEXT_FOUND`, leaving the same false positive for dual-field auth stores. Check whether `entry.value` itself is an env ref before treating it as plaintext, while keeping the explicit-ref-plus-real-plaintext case covered.
  - confidence: 0.88
