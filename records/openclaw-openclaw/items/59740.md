## Summary

Keep open. Current main still has grouped CI orchestration for boundary guards, not the requested unified/config-driven architecture-check framework. The previous ClawSweeper comment remains accurate on the current SHA: the runner delegates to individual package scripts, package.json still exposes many bespoke guard entrypoints, and representative checks still contain per-rule scanner logic.

## Best Possible Solution

Keep this issue open as the canonical cleanup request unless maintainers split it into narrower tracked work. The best implementation path is to build on the existing shared guard helpers, introduce a documented unified runner/config or equivalent lint-rule framework, migrate a meaningful first batch of boundary checks, update CI/package scripts, and provide compatibility wrappers or a clear deprecation plan for existing guard entrypoints.
