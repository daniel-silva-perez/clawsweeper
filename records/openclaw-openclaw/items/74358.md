## Summary

Keep open. Current main still has a source-level path where the reporter's object-only Slack streaming config resolves to partial/native defaults, but top-level DMs without a thread do not initialize draft preview streaming and do not wire the partial-text callback; the issue also overlaps open PR #56544, so it should be reconciled rather than closed.

## What This Changes

The issue requests restoring Slack in-flight answer previews when `streaming.preview.toolProgress` is `false`, especially for verbose multi-tool turns in Slack DMs.

## Best Possible Solution

Slack should make `streaming.preview.toolProgress: false` mean answer-preview-on and tool-lines-off wherever Slack preview streaming is enabled, including the reporter's object-only DM path, with tests covering default mode/nativeTransport resolution, flat DM preview eligibility, and no separate generic progress chatter. The final docs and config help should match that contract, and #56544 should either be updated to carry this regression or superseded by a focused replacement.

## Reproduction Assessment

Yes. The reporter provided a concrete Slack DM config and current source shows the object-only config defaults to partial/native streaming while default top-level DMs have no thread target, which disables both draft stream creation and `onPartialReply` wiring.

## Solution Assessment

Unclear for the issue as filed because there is no patch here, and the proposed line-991-only fix would be incomplete. The maintainable solution should reconcile the flat-DM preview gate, native transport fallback behavior, and `toolProgress:false` suppression semantics with regression tests.
