## Summary

Keep open. Current main still has a credible Slack silent-preview path for the reporter’s object-only `streaming.preview.toolProgress: false` config, and the issue overlaps but is not resolved by the open flat-DM preview PR. The remaining work needs maintainer coordination around Slack DM preview eligibility, native streaming fallback, and docs rather than closure.

## What This Changes

The issue requests Slack live answer previews to continue during verbose multi-tool turns when `streaming.preview.toolProgress` is false.

## Best Possible Solution

Slack should treat `streaming.preview.toolProgress: false` as answer-preview-on and in-preview-tool-lines-off whenever Slack preview delivery is active, with regression tests for object-form config in top-level DMs and updated Slack docs/config help that match the shared streaming contract.

## Reproduction Assessment

Yes, for a high-confidence source-level reproduction path. The reporter provided a stable object-only Slack config, and current source shows that config resolves to partial/native streaming while top-level DMs remain preview-ineligible and `onPartialReply` is not wired for that path; I did not perform a live Slack socket-mode run.

## Solution Assessment

No, the initial suggested line-991-only fix is not sufficient. The better fix must also reconcile the flat-DM preview gate and native-streaming callback path, likely by updating or replacing the overlapping #56544 approach with coverage for `toolProgress: false`.
