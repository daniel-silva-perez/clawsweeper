## Summary

Keep open. Current main and the latest release still preserve the Slack paths that can make the reporter's object-only `streaming.preview.toolProgress: false` configuration silent in top-level DMs, and the overlapping flat-DM preview PR is still open rather than merged or linked with closing syntax.

## What This Changes

Restore Slack in-flight reply preview when `streaming.preview.toolProgress` is false, so disabling tool-progress lines does not make verbose Slack turns silent until the final reply.

## Best Possible Solution

Slack should treat `streaming.preview.toolProgress: false` as answer-preview-on and in-preview-tool-lines-off wherever Slack preview delivery is active, including the object-only config path; the fix should reconcile the flat-DM/native-transport split, add regression coverage, and align Slack docs/config help with the shared streaming contract.

## Reproduction Assessment

Yes, for a high-confidence source-level reproduction path. The reporter confirmed the exact object-only config, and current source resolves it to partial/native streaming while flat top-level DMs remain preview-ineligible and `onPartialReply` is not wired for the native-streaming path; no live Slack run was performed.

## Solution Assessment

No, the issue body's initial line-coupling fix is not sufficient by itself. The maintainable fix needs to address both disabled tool-progress handling and Slack's flat-DM/native-streaming preview gating, likely by updating or replacing the overlapping #56544 approach with this regression coverage.
