## Summary

Keep open. Current main still routes Telegram delete-message API failures through the shared API logger/rethrow path, while the same file already has local fail-soft precedents for analogous Telegram no-op errors; the related fix PRs in the provided context are closed unmerged with no open replacement reported.

## What This Changes

The issue requests that Telegram message deletion soft-fail for benign Bot API 400 responses instead of logging an ERROR and throwing.

## Best Possible Solution

Patch the Telegram plugin surface only: add a narrow benign-delete classifier, suppress shared ERROR logging for those exact no-op outcomes, return a warning result from deleteMessageTelegram, surface `deleted: false` through the action runtime, and add focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue's CLI path maps to current main, and the same behavior can be reproduced without live Telegram credentials by injecting an API whose deleteMessage rejects with one of the reported benign 400 strings.

## Solution Assessment

Yes, a narrow Telegram-plugin fix is the maintainable approach. The issue's proposed direction is right, but the implementation should use the existing shouldLog seam and avoid blanket soft-failing permission or authorization failures.

## Repair Work Prompt

Fix the Telegram delete-message fail-soft bug for https://github.com/openclaw/openclaw/issues/73726. Current main still logs and throws benign Telegram deleteMessage 400s from `extensions/telegram/src/send.ts`. Keep the fix scoped to the Telegram plugin: add a conservative benign-delete classifier, pass it as `shouldLog` into `requestWithDiag` so `withTelegramApiErrorLogging` does not emit ERROR logs for benign delete no-ops, catch those errors and return `{ ok: false, warning }` from `deleteMessageTelegram`, and update `extensions/telegram/src/action-runtime.ts` to surface `{ ok: false, deleted: false, warning }` instead of always returning `{ ok: true, deleted: true }`. Add regression coverage for `message to delete not found`, `message can't/cannot be deleted`, and `MESSAGE_ID_INVALID`; only include `MESSAGE_DELETE_FORBIDDEN` if dependency/source checks show the exact variant is a stale/non-deletable-message no-op, and keep auth, permission, credential, chat access, network, and unknown errors hard. Do not change core channel abstractions or shared retry semantics. Add a user-facing CHANGELOG.md fix entry with credit to @Avicennasis.
