## Summary

Keep open. Current main still propagates benign Telegram delete-message failures through the Telegram API logging wrapper, and the related fix PRs for this issue are closed unmerged rather than superseding the report.

## What This Changes

The issue asks for Telegram message deletes to fail-soft on benign Bot API 400 responses instead of logging recurring hard errors.

## Best Possible Solution

Patch this in the Telegram plugin, not core: add a narrow benign-delete classifier, suppress API error logging for those benign delete outcomes, return a warning result from `deleteMessageTelegram`, surface `deleted: false` with the warning in the action runtime, and add focused regression coverage for benign and non-benign delete errors.

## Reproduction Assessment

Yes. The issue's CLI path maps directly to current main: a stale or non-deletable Telegram message makes `api.deleteMessage` reject, `requestWithDiag` logs and rethrows, and no delete-specific catch handles it.

## Solution Assessment

Yes, a narrow Telegram-plugin fix is the right approach, but it should also use the existing `shouldLog` hook so benign delete failures do not hit the ERROR logger before the warning result is returned.

## Repair Work Prompt

Fix openclaw/openclaw#73726 in the Telegram plugin only. Current main lets `deleteMessageTelegram` propagate benign Telegram Bot API delete failures and logs them at ERROR level through `withTelegramApiErrorLogging`. Add a narrow helper in `extensions/telegram/src/send.ts` for benign delete outcomes such as `message to delete not found`, `message can't be deleted` / `message can’t be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`; keep auth, permission, routing, rate-limit, network, and unknown errors throwing. Use `requestWithDiag(..., "deleteMessage", { shouldLog: err => !isBenignTelegramDeleteError(err) })` or an equivalent existing seam so benign delete failures do not hit the API error logger before being caught. Widen `deleteMessageTelegram` to return `{ ok: true } | { ok: false; warning: string }`, log only verbose detail for benign skips, and update `extensions/telegram/src/action-runtime.ts` so the `deleteMessage` action returns `deleted: false` with the warning instead of unconditional `deleted: true`. Add focused tests in the Telegram test suite for all benign strings, for a non-benign error that still rejects, and for the action-runtime warning JSON. Do not move behavior into core and do not broaden Telegram error suppression beyond delete-message no-ops. Add a user-facing changelog fix entry with `Thanks @Avicennasis` if the active changelog policy requires one.
