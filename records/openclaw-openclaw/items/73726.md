## Summary

Keep open. Current main still hard-fails Telegram delete-message API errors through the shared API logging/rethrow path, the action runtime still reports every successful call as `deleted: true`, and the related implementation PRs were closed unmerged with no open replacement found.

## What This Changes

The issue asks for `deleteMessageTelegram` to treat benign Telegram Bot API delete-message 400 responses as soft failures rather than logging/rethrowing them as runtime errors.

## Best Possible Solution

Patch this inside the Telegram plugin: classify only true already-gone or not-deletable no-op delete states, reuse that classifier both to suppress shared API error logging and to return a warning result, surface `deleted: false` through the action runtime, and cover benign and hard-failure paths in focused tests.

## Reproduction Assessment

Yes. A stale or non-deletable Telegram message makes grammY reject `api.deleteMessage`; current main routes that rejection through `requestWithDiag`, logs through `withTelegramApiErrorLogging`, rethrows, and has no delete-specific catch.

## Solution Assessment

Yes. The narrow maintainable fix is in `extensions/telegram`: add a delete-specific benign classifier used by both `shouldLog` and the catch, while keeping authorization, token, permission, and network failures hard.

## Repair Work Prompt

Fix the Telegram delete-message fail-soft bug tracked by https://github.com/openclaw/openclaw/issues/73726. Work only in the Telegram plugin/runtime surface unless tests prove a generic seam is required. In `extensions/telegram/src/send.ts`, add a narrow helper that classifies only benign/idempotent delete failures such as `message to delete not found`, `message can't be deleted`, and `MESSAGE_ID_INVALID`. Do not classify `MESSAGE_DELETE_FORBIDDEN`, token/auth, permission, config, or network failures as benign. Use the classifier both in `requestWithDiag(..., { shouldLog })` so benign delete failures do not hit the shared ERROR logger, and in a catch that returns a warning result instead of throwing. Widen `deleteMessageTelegram`'s return type accordingly. In `extensions/telegram/src/action-runtime.ts`, stop reporting soft no-op deletes as `deleted: true`; surface `deleted: false` and the warning while preserving hard failures. Add focused tests in `extensions/telegram/src/send.test.ts` and `extensions/telegram/src/action-runtime.test.ts` for successful delete, each benign string, no ERROR logging for benign deletes, and hard rethrow/log behavior for `MESSAGE_DELETE_FORBIDDEN` and auth-like failures. Update `CHANGELOG.md` because this is user-facing. Do not change core plugin SDK contracts, Telegram auth/permission policy, or unrelated delete paths.
