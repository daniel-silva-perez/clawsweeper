## Summary

Keep open. Current main still routes Telegram delete-message failures through the shared log-and-rethrow path, while adjacent Telegram helpers already have narrow fail-soft handling for benign API responses; the related fix PRs in the provided context were closed unmerged, so there is still a narrow plugin-local bug to fix.

## What This Changes

The issue requests that Telegram `deleteMessageTelegram` treat benign delete no-op 400s as soft failures with a warning instead of throwing and producing recurring error logs.

## Best Possible Solution

Patch only the Telegram plugin: add an exact benign-delete classifier, use the existing `shouldLog` seam so those no-op deletes do not emit shared ERROR logs, return a structured warning from `deleteMessageTelegram`, surface `deleted: false` through the action runtime, and cover both soft and hard failure cases in focused tests plus a user-facing changelog entry.

Yes. The report’s CLI path maps to the Telegram `deleteMessage` action, and current source shows an injected `api.deleteMessage` rejection with any reported benign string will pass through `withTelegramApiErrorLogging`, log, and rethrow.

## Solution Assessment

Yes. A narrow plugin-local fail-soft path is the best solution; the safer implementation is exact delete-error classification plus tests proving non-benign errors still rethrow, not a broad Telegram 400 catch.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73726 in the Telegram plugin only. Current `deleteMessageTelegram` in `extensions/telegram/src/send.ts` hard-fails and logs benign Telegram delete no-op errors such as `message to delete not found`, `message can't be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`. Add a narrow classifier for those delete-specific benign errors, pass a `shouldLog` predicate to the existing `requestWithDiag`/`withTelegramApiErrorLogging` path so those no-op errors do not emit shared ERROR logs, catch them and return `{ ok: false, warning: string }`, and keep non-benign errors rethrowing. Update `extensions/telegram/src/action-runtime.ts` so the `deleteMessage` action surfaces soft delete results as `deleted: false` with the warning instead of always reporting `deleted: true`. Add focused regression coverage in `extensions/telegram/src/send.test.ts` and `extensions/telegram/src/action-runtime.test.ts`; extend proxy coverage only if the type/behavior change requires it. Add a single-line user-facing `CHANGELOG.md` fix entry credited to `@Avicennasis`. Do not move this into core, do not broaden all Telegram 400 handling, do not suppress auth/token/chat-not-found/rate-limit/server/network errors, and do not change unrelated Telegram delete paths that are not part of this message action helper.
