## Summary

Keep open. Current main still hard-fails Telegram delete-message API errors through the shared API logging/rethrow path, and the related implementation PRs for this report are closed unmerged with no open replacement found.

## What This Changes

The issue requests Telegram delete-message handling that soft-fails benign already-gone or stale-message Telegram 400 responses instead of logging and propagating them as runtime errors.

## Best Possible Solution

Patch this inside the Telegram plugin: add a narrow benign-delete classifier, pass it into `requestWithDiag` as `shouldLog` to suppress the shared ERROR log for benign delete outcomes, return a warning result from `deleteMessageTelegram`, surface `deleted: false` through the action runtime, and add focused tests for benign and hard failures.

## Reproduction Assessment

Yes. The issue's CLI path maps directly to current main: a stale or non-deletable Telegram message makes `api.deleteMessage` reject, `requestWithDiag` logs and rethrows, and no delete-specific catch handles it.

## Solution Assessment

Yes. A narrow Telegram-plugin fix is the maintainable solution; it should tighten the proposed regex by keeping `MESSAGE_DELETE_FORBIDDEN` and other authorization/config failures hard.
