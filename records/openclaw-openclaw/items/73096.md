## Summary

Keep this issue open. Current main still gives pinned harness ids a hardcoded `fallback: "none"`, so config-backed `fallback: "pi"` can be bypassed on pinned/plugin-runtime paths; one adjacent `claude-cli` alias path has been improved, but that does not resolve the pinned-policy gap.

## What This Changes

The issue asks OpenClaw to either honor configured agent harness fallback for pinned runtime selection or document the fail-closed behavior and billing implications more clearly.

## Best Possible Solution

Define the runtime contract for pins explicitly: same-scope `agentRuntime.fallback: "pi"` should either be honored for configured explicit runtimes with a clear warning, or the docs/schema should state that persisted pins always fail closed. Add focused coverage for `agentHarnessId` plus configured fallback and keep CLI backend aliases out of embedded-harness pinning.

## Reproduction Assessment

Yes. A focused code path is clear: call selection with a non-auto `agentHarnessId` and config fallback `pi` while the harness is unregistered, and current main reaches `resolvePinnedAgentHarnessPolicy` first, sets fallback to `none`, and throws the disabled-PI-fallback error.

## Solution Assessment

Unclear. Option A is plausible but not automatically safest because the issue correctly identifies a billing-sensitive fail-closed brake; a maintainable fix likely needs a maintainer decision plus tests and warning/docs updates, not only a mechanical parameter pass-through.
