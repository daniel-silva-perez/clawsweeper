## Summary

Keep this open. Current main still resolves an explicit or persisted harness pin before the config-backed fallback policy and still assigns pinned non-auto ids `fallback: "none"`, while the public config docs describe same-scope `agentRuntime.fallback: "pi"` as the opt-in for missing explicit plugin runtimes. Because changing that behavior removes a fail-closed brake with billing-sensitive routing implications, this needs maintainer policy review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to resolve the mismatch between pinned agent harness selection and configured fallback policy, either by honoring fallback configuration for pins with clear billing warnings or by documenting/enforcing strict fail-closed semantics.

## Best Possible Solution

Define the contract for explicit and persisted harness pins, then implement it consistently. Either honor same-scope fallback for pins with a prominent warning/diagnostic when PI/direct-API routing is used, or codify that pins always fail closed and update docs/schema/error messaging; in either case add focused tests for pinned missing harnesses with `fallback: "pi"` and `fallback: "none"`.

## Reproduction Assessment

Yes. A high-confidence static reproduction on current main is `selectAgentHarness` with `agentHarnessId: "codex"`, no registered `codex` harness, and config `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`; the pinned resolver creates `fallback: "none"` and the selection path throws the disabled-PI-fallback error before consulting config.

## Solution Assessment

Unclear. The proposed pass-through is mechanically narrow, but the issue correctly identifies that it changes fail-closed behavior in a billing-sensitive runtime path, so the best fix needs maintainer policy review plus tests and docs or warning updates.
