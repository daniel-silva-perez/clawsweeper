## Summary

Keep this open. Current main still resolves an explicit or persisted harness pin before the config-backed runtime policy and assigns the pin `fallback: "none"`, so a configured `agentRuntime.fallback: "pi"` is still bypassed for pinned plugin harness ids.

## What This Changes

The issue asks OpenClaw to reconcile pinned embedded agent harness selection with the configured runtime fallback policy, without losing the fail-closed billing safety around PI/direct-API fallback.

## Best Possible Solution

Define the runtime contract for persisted and explicit harness pins. Either make pins honor same-scope `agentRuntime.fallback: "pi"` with a clear gateway warning when PI/direct-API fallback is selected, or codify that pins always fail closed and update docs, schema/help text, and error messages. In either direction, add focused coverage for pin plus fallback, persisted session pins, configured plugin runtime pins, and CLI backend aliases.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to call selection with `agentHarnessId: "codex"`, no registered Codex harness, and config fallback `pi`; current main resolves the pin first, hardcodes `fallback: "none"`, and throws the disabled-PI-fallback error before consulting config.

## Solution Assessment

Unclear. A mechanical pass-through of configured fallback would fix the config mismatch, but the issue correctly identifies that this changes a billing-sensitive fail-closed brake, so the narrowest maintainable fix needs an explicit maintainer policy decision plus tests and docs or warning updates.
