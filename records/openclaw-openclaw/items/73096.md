## Summary

Keep open. Current main still gives persisted or explicit harness pins precedence over the config-backed fallback policy, and a pinned non-auto harness id still resolves to `fallback: "none"`; the remaining decision is billing-sensitive, so this should stay with maintainer policy review.

## What This Changes

The issue asks to resolve the mismatch where pinned agent harness selection ignores configured `agentRuntime.fallback`, causing missing plugin harnesses to fail closed even when `fallback: "pi"` is configured.

## Best Possible Solution

Define the contract for explicit and persisted harness pins, then implement it consistently: either honor same-scope `fallback: "pi"` for pins with a clear warning/diagnostic for PI direct-API routing, or codify that pins always fail closed and update docs, schema help, and errors. Add focused coverage for pinned missing harnesses with both `fallback: "pi"` and `fallback: "none"`.

## Reproduction Assessment

Yes. A high-confidence static reproduction on current main is `selectAgentHarness` with `agentHarnessId: "codex"`, no registered `codex` harness, and config setting `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`; the pinned resolver assigns `fallback: "none"` before the config-backed policy is consulted.

## Solution Assessment

Unclear. The suggested plumbing change is narrow, but it changes a fail-closed billing-sensitive runtime route, so the best fix needs explicit maintainer policy plus tests and docs or warnings.
