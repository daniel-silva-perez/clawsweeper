## Summary

Keep this open. Current main still resolves explicit or persisted harness pins before the config-backed fallback policy and assigns pinned non-auto ids `fallback: "none"`, while docs and schema present same-scope `agentRuntime.fallback: "pi"` as the opt-in for missing explicit plugin runtimes. The remaining choice is billing-sensitive, so this should stay with maintainer policy review rather than conservative cleanup closure.

## What This Changes

The issue asks OpenClaw to resolve the mismatch between pinned agent harness selection and configured fallback policy, either by honoring the fallback setting for pins with billing warnings or by documenting/enforcing strict fail-closed pin semantics.

## Best Possible Solution

Define the contract for explicit and persisted harness pins, then implement it consistently. Either honor same-scope fallback for pins with a prominent warning/diagnostic when PI/direct-API routing is used, or codify that pins always fail closed and update docs/schema/error messaging; in either case add focused tests for pinned missing harnesses with both `fallback: "pi"` and `fallback: "none"`.

## Reproduction Assessment

Yes. A high-confidence static reproduction on current main is `selectAgentHarness` with `agentHarnessId: "codex"`, no registered `codex` harness, and config `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`; the pinned resolver creates `fallback: "none"` and the selection path throws the disabled-PI-fallback error before consulting config.

## Solution Assessment

Unclear. The suggested pass-through is mechanically narrow, but it changes a fail-closed, billing-sensitive runtime route, so the maintainable fix needs explicit maintainer policy plus tests and docs or warnings rather than only an argument plumbing change.
