## Summary

Keep this issue open. Current main still resolves pinned harness ids before the config-backed runtime policy and assigns pinned ids `fallback: "none"`, so the documented same-scope `agentRuntime.fallback: "pi"` opt-in is bypassed for persisted or explicit embedded harness pins.

## What This Changes

The issue asks OpenClaw to reconcile pinned embedded agent harness selection with the configured runtime fallback policy, or document that pins always fail closed.

## Best Possible Solution

Define the contract for persisted and explicit harness pins: either honor same-scope `agentRuntime.fallback: "pi"` with a clear warning for PI/direct-API routing, or codify that pinned runtime ids always fail closed and update docs/schema/messages accordingly. Add focused coverage for pin plus fallback, and keep CLI backend aliases out of embedded harness pinning.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to call selection with `agentHarnessId: "codex"`, no registered `codex` harness, and config fallback `pi`; current main reaches `resolvePinnedAgentHarnessPolicy` first and throws the disabled-PI-fallback error before consulting config.

## Solution Assessment

Unclear. Honoring fallback for pins is plausible, but the issue correctly identifies a billing-sensitive fail-closed brake, so the best fix needs maintainer policy review plus tests and docs/warning updates rather than only a mechanical argument pass-through.
