## Summary

Keep open. Current main still lacks the requested env-gated abort-source diagnostic, while the centralized embedded-run abort path and the separate abort triggers remain present; related PR #74110 is closed and unmerged, so this is still a maintainer operability/product decision rather than shipped behavior.

## What This Changes

Request to add an opt-in `OPENCLAW_LOG_ABORT_SOURCES=1` diagnostic that logs one structured `[abort-source]` line from embedded-run `abortRun()` with trigger classification and caller stack frames.

## Best Possible Solution

If maintainers accept the surface, implement a narrow default-off helper in the embedded runner, call it once from `abortRun()`, classify the existing abort sources without changing run behavior, and add focused helper tests plus debugging docs that explain stack-frame sensitivity.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. The source check is high-confidence: current main has the abort paths described by the report but no requested env gate, marker, classifier, stack capture, or test file.

## Solution Assessment

Unclear until maintainers accept the diagnostic surface. Technically, a single helper called from `abortRun()` is the narrowest maintainable shape, but the final design should consider stack redaction and whether to use the existing diagnostics flag mechanism.
