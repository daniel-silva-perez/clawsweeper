## Summary

Keep open. Current main still lacks the requested env-gated abort-source diagnostic, while the embedded runner still has the shared abortRun path and the named abort triggers. The related implementation PR is closed and unmerged, so this remains a maintainer operability/security decision rather than cleanup-close material.

## What This Changes

Request an opt-in structured embedded-run abort diagnostic, gated by OPENCLAW_LOG_ABORT_SOURCES, that logs abort source classification and caller stack frames from abortRun().

## Best Possible Solution

If maintainers accept the diagnostic surface, add a narrow default-off helper in the embedded runner, call it once from abortRun(), classify the existing abort sources without changing run behavior, and document stack-frame sensitivity with focused helper tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. The current-main source check is high-confidence: the abort paths described by the report exist, but the requested env gate, marker, classifier, stack capture, and test file do not.

## Solution Assessment

Unclear until maintainers approve the core operator diagnostic surface. Technically, a single helper called from abortRun() is the narrowest maintainable shape, but using the existing diagnostics flag system and documenting/redacting stack output may be safer.
