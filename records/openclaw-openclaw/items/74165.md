## Summary

Keep open. Current main still lacks the requested env-gated abort-source diagnostic, the related implementation PR is closed unmerged, and the existing embedded-run abort paths still converge on an uninstrumented `abortRun()` closure; this remains a maintainer operability/product decision because it adds a new core diagnostic env var and opt-in stack logging.

## What This Changes

Request adds `OPENCLAW_LOG_ABORT_SOURCES=1` to emit a structured `[abort-source]` log line with abort-source classification and caller stack frames for embedded runner aborts.

## Best Possible Solution

If maintainers accept the diagnostic surface, implement a narrow default-off helper in the embedded runner, call it once from `abortRun()`, classify the existing abort sources without changing run behavior, and add focused helper tests plus debugging docs that call out stack-frame sensitivity.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. The source check is high-confidence: current main has the abort paths described by the report but no requested env gate, marker, classifier, stack capture, or test file.

## Solution Assessment

Unclear until maintainer acceptance of the new diagnostic surface. Technically, a single helper called from `abortRun()` is the narrowest maintainable shape, but the design should explicitly handle stack-frame sensitivity and documentation.
