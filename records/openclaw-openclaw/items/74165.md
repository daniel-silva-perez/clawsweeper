## Summary

Keep open. Current main lacks the requested env-gated abort-source diagnostic while the centralized embedded-run abort path and separate abort triggers still exist; related PR #74110 is closed unmerged, so this remains a maintainer operability/product decision rather than shipped behavior.

## What This Changes

Request to add an opt-in `OPENCLAW_LOG_ABORT_SOURCES=1` diagnostic that logs a structured `[abort-source]` line from embedded-run `abortRun()` with trigger classification and stack frames.

## Best Possible Solution

If maintainers accept the operability surface, implement a narrow default-off helper in the embedded runner, call it once from `abortRun()`, classify the existing abort sources without changing run behavior, and add focused helper tests plus debugging docs for the new diagnostic and stack-frame sensitivity.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. The source check is high-confidence: current main has the abort paths described by the report but no requested env gate, marker, classifier, stack capture, or test file.

## Solution Assessment

Unclear until maintainer acceptance of the new diagnostic surface. Technically, a single helper called from `abortRun()` is the narrowest maintainable shape, but the final design should consider stack redaction and whether to use the existing diagnostics flag mechanism.
