## Summary

Keep this open. Current main and the latest release still lack the requested abort-source diagnostic, while the embedded runner continues to funnel external aborts, idle timeout, run timeout, compaction timeout, and explicit cancel through the same uninstrumented abort path. The related implementation PR is closed unmerged, so it remains useful prior art rather than shipped behavior.

## What This Changes

Request to add an opt-in `OPENCLAW_LOG_ABORT_SOURCES` diagnostic that classifies embedded-run abort sources and logs stack context from the central `abortRun()` path.

## Best Possible Solution

Keep the issue open for maintainer acceptance. If accepted, implement the diagnostic as a narrow embedded-runner helper called once from `abortRun()`, test default-off/env-gated behavior and all five classifications, and document or constrain the stack logging so operators understand what it can expose.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/observability request. The missing behavior is high-confidence verifiable by exact symbol search plus inspection of the current `abortRun()` and trigger paths on main and `v2026.4.27`.

## Solution Assessment

Unclear pending maintainer product review. The proposed helper plus one central `abortRun()` call appears to be the narrowest maintainable implementation if the diagnostic surface is accepted, with stack-log privacy as the main design detail to settle.
