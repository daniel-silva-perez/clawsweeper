## Summary

Keep open. Current main still lacks the requested env-gated abort-source diagnostic, while the centralized embedded-run abort path and the named abort triggers remain present; the related implementation PR is closed and unmerged, so this is a maintainer operability/product decision rather than shipped behavior.

## What This Changes

Add an opt-in `OPENCLAW_LOG_ABORT_SOURCES=1` diagnostic that logs classified embedded-run abort sources and caller stack frames from the centralized `abortRun()` path.

## Best Possible Solution

If maintainers accept the surface, add a narrow default-off helper in the embedded runner, call it once from `abortRun()`, classify the existing abort sources without changing run behavior, and document the stack-frame sensitivity alongside focused helper tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. The current-main source check is high-confidence: the abort paths described by the report exist, but the requested env gate, marker, classifier, stack capture, and test file do not.

## Solution Assessment

Unclear until maintainers accept adding this core diagnostic surface. Technically, a single helper called from `abortRun()` is the narrowest maintainable shape, but the final design should consider whether to use the existing diagnostics flag mechanism and how to handle stack redaction.
