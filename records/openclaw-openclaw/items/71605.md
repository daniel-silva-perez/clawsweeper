## Summary

Keep this open. Current main has fixed the narrower #62981 timeout-fallback lock race by using a fresh `gateway-fallback-*` session, and it has an accepted-ack flush mitigation, but the broader external-integration problem remains: the CLI still waits on the final `agent` response with `expectFinal: true`, the initial accepted runId is still hidden from that path, and non-timeout Gateway failures still fall back to embedded execution without a daemon/state-dir guard.

## What This Changes

The issue requests reliable external `openclaw agent`/Gateway WS invocation by fixing Gateway `agent` dispatch timeouts, preventing embedded fallback from contending with the running daemon's session locks, or documenting the limitation.

## Best Possible Solution

Keep this open until the external-agent path is explicitly reliable: the CLI should get the fast accepted `runId`, wait through `agent.wait` or an equivalent result-bearing terminal RPC, avoid embedded fallback after the Gateway has accepted a run, and keep a clear fallback/error story for pre-accept failures while preserving the fresh-session lock mitigation already merged for #62981.

## Reproduction Assessment

Unclear. The issue has concrete reproduction steps and current source still contains the final-response wait/fallback mechanics, but this read-only review did not run a live daemon or reproduce the intermittent 60s fresh-session timeout on macOS.

## Solution Assessment

No, current main is only a partial solution. The fresh `gateway-fallback-*` timeout fallback is the right narrow fix for #62981, but it does not solve the broader accept-vs-final wait behavior or define a safe non-timeout embedded fallback contract while a daemon owns the state directory.
