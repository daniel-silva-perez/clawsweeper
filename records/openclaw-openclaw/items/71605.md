## Summary

Keep open. Current main has partial mitigations for the accepted-ack starvation and timeout-fallback lock race, but the broader external-agent path is still not fully settled: the CLI still waits for the final Gateway response, accepted `runId` frames are suppressed for that caller, `agent.wait` still returns terminal metadata rather than the reply payload the CLI prints, and non-timeout Gateway failures still fall through to embedded execution without a daemon/state-dir guard.

## What This Changes

The issue asks OpenClaw to make external `openclaw agent`/Gateway WS invocation reliable by fixing Gateway `agent` dispatch timeouts, preventing embedded fallback from contending with the daemon’s session locks, or documenting/refusing the unsafe mode clearly.

## Best Possible Solution

Define one reliable Gateway-backed external invocation contract: expose a quick accepted `runId` path that clients can safely wait on, make the wait path capable of returning the final reply payload needed by `openclaw agent`, and only use embedded fallback when the Gateway definitely has not accepted ownership or when a fresh/isolated state contract is in force.

## Reproduction Assessment

Unclear. The issue provides concrete commands and logs, and current source still contains the relevant wait/fallback mechanics, but this review did not run a live daemon to reproduce the intermittent 60-second timeout after the newer partial mitigations.

## Solution Assessment

No. Current main is only a partial solution: it improves accepted-frame scheduling and timeout fallback lock isolation, but it has not yet made the external CLI/Gateway invocation path explicit and safe across accepted runs, wait responses, and non-timeout Gateway failures.
