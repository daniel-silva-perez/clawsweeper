## Summary

Keep this open. Current main has a real partial mitigation for the timeout-fallback lock race, but it still does not solve the broader external-agent path: the CLI waits for the final Gateway `agent` response, accepted `runId` frames remain suppressed for that caller, `agent.wait` does not currently return the reply payload the CLI prints, and non-timeout Gateway failures still fall through to embedded execution without a daemon/state-dir guard.

## What This Changes

The issue requests reliable external `openclaw agent` invocation while a daemon is running by fixing Gateway WS `agent` timeout behavior and embedded fallback lock contention, or by making the limitation explicit.

## Best Possible Solution

Make the external-agent path explicit and reliable: have the CLI resolve quickly once the Gateway accepts a run, wait through `agent.wait` or an equivalent terminal RPC that can provide the final reply payload, avoid embedded fallback after the Gateway has accepted ownership of a run, and keep the #62981 fresh-session mitigation for true timeout fallback cases.

## Reproduction Assessment

Unclear. The issue gives concrete reproduction steps and current source still contains the relevant wait/fallback mechanics, but this read-only review did not run a live daemon to reproduce the intermittent 60-second timeout on macOS.

## Solution Assessment

No. Current main is only a partial solution: the timeout fallback uses a fresh session, but the main CLI still hides the accepted `runId` behind `expectFinal: true` and still lacks a clear safe fallback contract for non-timeout Gateway failures while a daemon may own the same state directory.
