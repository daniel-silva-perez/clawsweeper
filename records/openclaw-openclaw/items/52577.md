## Summary

Keep open. Current main has stronger Discord gateway lifecycle handling and an inbound worker queue, but that queue only starts after a Discord MESSAGE_CREATE event reaches OpenClaw. The specific post-boot/reconnect gap remains: there is no Discord-owned catch-up path that fetches recent channel history on READY/reconnect and replays missed messages through the existing preflight/worker pipeline. Related ClawSweeper context for #51116 and #55763 covers stale socket/reconnect recovery, not this narrower missed-inbound startup window.

## Best Possible Solution

Keep this issue open and implement the fix inside the Discord plugin. The likely path is a Discord-owned READY/reconnect catch-up pass: persist or maintain a per-account/channel last-seen message cursor, fetch recent messages via Discord REST after the cursor when the gateway becomes ready, dedupe against live MESSAGE_CREATE deliveries with the existing replay guard/key shape, and replay eligible messages through createDiscordMessageHandler/preflight so current authorization and routing behavior stays intact.

## Risks / Open Questions

- A reconnect catch-up implementation could double-process messages that were already delivered through the live gateway unless it uses account/channel/message-id dedupe across both live and replay paths.
- Discord history replay must preserve the existing DM/guild allowlist, mention-gating, bot/self filtering, thread binding, and media hydration behavior by routing replayed messages through the same preflight and worker path.
