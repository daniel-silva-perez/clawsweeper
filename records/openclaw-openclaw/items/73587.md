## Summary

Keep open. Current main and the latest release still pass `logging.file` through unchanged before logger directory creation, and the exact fix PR was closed unmerged. The immediate logger crash is valid and narrow; the reporter's broader workspace-path concern is useful follow-up context, but `agents.defaults.workspace` already routes through `resolveUserPath` on the central resolver path.

## What This Changes

The issue reports a gateway startup crash when `openclaw.json` sets `logging.file` to a leading-tilde path because the logger passes the unexpanded path to filesystem directory creation.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and `fs.mkdirSync`, reusing the existing home-expansion helper and adding regression coverage for plain and rolling `logging.file` values. Treat wider config-path normalization as a follow-up audit unless it falls out naturally from the same helper without changing unrelated startup behavior.

## Reproduction Assessment

Yes. The source path is deterministic: current main reads `logging.file` raw, returns non-rolling log paths unchanged from `resolveActiveLogFile`, and then calls `fs.mkdirSync(path.dirname(activeFile))` on the literal tilde path.

## Solution Assessment

Yes. The narrow logger-side expansion used by the closed PR is the maintainable fix because it handles the filesystem boundary directly and avoids a broader startup-time config migration; `agents.defaults.workspace` already uses `resolveUserPath` in the central resolver.
