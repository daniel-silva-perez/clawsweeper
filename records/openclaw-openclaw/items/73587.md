## Summary

Keep open. Current main and the shipped v2026.4.29 tag still pass raw `logging.file` values into the logger filesystem path without leading-tilde expansion, and the only linked fix PR is closed unmerged.

## What This Changes

Fix a gateway startup crash where `logging.file` values beginning with `~` are used literally when the logger creates its log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands leading `~` via the existing home helper before rolling-path detection and directory creation, with regression coverage for normal and rolling log filenames; track any broader config-path audit separately if maintainers want that scope.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: a `logging.file` value of `~/.openclaw/logs/gateway.log` flows from `readLoggingConfig()` into `resolveSettings()`, then through `resolveActiveLogFile()` unchanged before `fs.mkdirSync(path.dirname(activeFile))`.

## Solution Assessment

Yes. Expanding leading `~` at the logger filesystem boundary is the narrowest maintainable fix for the boot crash; rewriting config loading or auditing every path-like setting is useful follow-up work but not required to stop this failure.
