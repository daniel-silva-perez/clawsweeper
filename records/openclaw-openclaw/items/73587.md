## Summary

Keep open. Current main still lets `logging.file` bypass the existing tilde-normalization path: the logger reads the raw logging block, keeps `cfg.file` unchanged, and creates the directory from `resolveActiveLogFile`, which returns non-rolling `~/...` paths verbatim. The linked fix PR is closed and unmerged, so it is useful source work but does not supersede the issue.

## What This Changes

The issue requests restoring leading-tilde expansion for `openclaw.json` `logging.file` paths so gateway startup does not crash when creating the log directory.

## Best Possible Solution

Fix the logger boundary itself: expand a leading `~` on `logging.file` before rolling-path detection and directory creation, reuse the existing home-path helper, and add regression tests for both non-rolling and rolling log file patterns. Treat the broader config-derived path audit as follow-up unless the logger fix exposes a shared missing helper.

## Reproduction Assessment

Yes. The source path is deterministic: a config `logging.file` of `~/.openclaw/logs/gateway.log` is read unchanged, `resolveActiveLogFile` returns it unchanged, and `fs.mkdirSync` receives the literal `~/.openclaw/logs` directory. I did not run a live gateway or artifact-creating test because this review was read-only.

## Solution Assessment

Yes, the narrowest maintainable fix is to normalize the configured log file path inside the logging path boundary before filesystem calls. A broad startup migration would be riskier and unnecessary for the immediate crash loop.
