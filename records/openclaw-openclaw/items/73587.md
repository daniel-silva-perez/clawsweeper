## Summary

Keep open. Current `main` and the latest release still leave `logging.file` unexpanded before logger directory creation, and the linked fix PR was closed unmerged, so the central gateway crash remains actionable.

## What This Changes

The issue requests restoring `~` expansion for `logging.file` and auditing config-derived filesystem paths so gateway startup does not crash on literal tilde paths.

## Best Possible Solution

Land a narrow logger-boundary fix that expands a leading `~` before rolling-path detection and `fs.mkdirSync`, with focused regression tests for non-rolling and rolling `logging.file` values. Keep any broader config-path audit as a follow-up unless the logger fix naturally reuses an existing shared helper without changing unrelated startup behavior.

## Reproduction Assessment

Yes. The source path is deterministic: `logging.file` is read raw, non-rolling paths are returned unchanged by `resolveActiveLogFile`, and `fs.mkdirSync(path.dirname(activeFile))` therefore targets literal `~/.openclaw/logs` for the reported config.

## Solution Assessment

Yes, the best fix is the narrow approach proposed by the linked closed PR: reuse the existing home-expansion helper at the logger filesystem boundary and add regression coverage. A broad migration or startup-time config rewrite would be riskier than fixing the path before the logger touches `node:fs`.
