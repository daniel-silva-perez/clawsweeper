## Summary

Keep open. Current main and the latest shipped v2026.4.29 release still pass raw `logging.file` values into the logger filesystem path without leading-tilde expansion, while the linked fix PR is closed unmerged and no open replacement PR was found.

## What This Changes

Expand leading `~` in configured `logging.file` paths so gateway startup does not crash when file logging uses a home-relative path.

## Best Possible Solution

Land a focused logger-boundary fix that reuses the existing home-expansion helper before rolling-path detection and directory creation, adds regression coverage for normal and rolling log filenames, and leaves the broader config-path audit as a follow-up unless it is explicitly scoped into the same change.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: set `logging.file` to `~/.openclaw/logs/gateway.log` with file logging enabled, and current main passes the literal path through `resolveActiveLogFile()` into `fs.mkdirSync(path.dirname(activeFile))`.

## Solution Assessment

Yes. The narrow maintainable fix is to expand leading `~` at the logger filesystem boundary using the existing helper; rewriting config loading or auditing every path-like config field is not required to stop the boot crash.
