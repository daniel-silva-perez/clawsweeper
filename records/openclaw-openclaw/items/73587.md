## Summary

Keep open. Current main and the latest shipped v2026.4.27 release still pass `logging.file` through the logger without home expansion before `fs.mkdirSync`, while the linked fix PR was closed unmerged, so the gateway startup crash remains a narrow actionable bug.

## What This Changes

The issue requests restoring leading-tilde expansion for `logging.file` so gateway startup does not crash when the logger creates the configured log directory.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for both normal and rolling log filenames. Keep any all-config-path audit as a follow-up unless it falls out naturally from reusing an existing helper.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: configure `logging.file` as `~/.openclaw/logs/gateway.log`, enable file logging, and the logger still passes the literal tilde path through `resolveActiveLogFile` into `fs.mkdirSync(path.dirname(activeFile))`.

## Solution Assessment

Yes for the immediate crash. The narrowest maintainable fix is to expand the configured log file at the logger filesystem boundary, preferably by reusing the existing home-expansion helper, rather than rewriting startup config loading or broad owner-specific path behavior.
