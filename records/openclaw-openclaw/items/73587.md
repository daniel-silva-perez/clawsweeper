## Summary

Keep open. Current main still reads `logging.file` as a raw config string and passes it to logger directory creation without expanding a leading `~`. The issue also has an open, unmerged PR with closing syntax, so the right next step is to review and land or close that PR rather than closing this issue early.

## Best Possible Solution

Keep this issue open. Review PR #73614 as the immediate implementation candidate, or apply the same fix directly: expand `logging.file` through the existing home-relative path helper before rolling-path detection and before any `fs.mkdirSync`/write path. Add focused regression tests for both non-rolling and rolling `~/...` log paths, then separately audit other config-derived filesystem paths if maintainers want to broaden the fix.
