## Summary

Current main still matches the reported bug: `generic_repeat` has no `criticalThreshold` path, warning-level loop detections do not block tool execution, and `process` with `action: "list"` is still outside known-poll handling. The issue is recent, actionable, unprotected, and not implemented on main.

## Best Possible Solution

Keep this issue open until main changes. The narrow fix belongs in `src/agents/tool-loop-detection.ts`: add a `generic_repeat` critical branch before the warning branch, update unit and before-tool-call e2e tests that currently assert warn-only behavior, and align docs/config help if the intended contract changes. Maintainers should review, land, or replace #60248, and separately decide whether `process` `action: "list"` should join known poll/no-progress handling.

## Risks / Open Questions

- Adding critical escalation to `generic_repeat` could block legitimate repeated idempotent calls when users configure low thresholds, so the fix should preserve threshold ordering and include focused regression coverage.
- Whether `process` `action: "list"` should be treated as known poll/no-progress behavior is a related product decision; the central missing critical path for `generic_repeat` is independently actionable.
- The broader tool-only-turn circuit breaker discussed in #34574 may still need separate design after `generic_repeat` critical escalation is fixed.
