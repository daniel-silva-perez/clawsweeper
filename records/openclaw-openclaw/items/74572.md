## Summary

Keep open. Current main still has the reported Active Memory / Memory Core composition path: Active Memory starts a lightweight embedded recall run with a memory-only runtime allowlist, Memory Core owns the requested tools as plugin tools, and the embedded runner applies the runtime allowlist only after tool construction. Open PR #74592 already references this issue with closing syntax, so this issue should stay open until that PR is reviewed and merged or rejected.

## What This Changes

Active Memory’s lightweight embedded recall run should compose with Memory Core plugin tools without resolving its memory tool allowlist to an empty set or persisting a placeholder failed assistant turn.

## Best Possible Solution

Review and land or close #74592. The desired end state is a generic, extension-agnostic fix where explicitly allowlisted plugin tools can be materialized for embedded runs before the final runtime allowlist filter, and unavailable memory tools do not leave a persistent user-visible placeholder in transcripts.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to enable active-memory and memory-core, trigger the Active Memory pre-summarizer, and observe that current main applies the memory toolsAllow after tool construction while that runtime allowlist does not participate in plugin tool materialization.

## Solution Assessment

Yes for the triage route. The issue should stay open behind #74592, and the fix should remain generic to plugin-tool materialization and gateway subagent binding rather than hardcoding Memory Core behavior in core.
