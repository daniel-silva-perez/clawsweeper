## Summary

Keep open. Current main still matches the reported Active Memory / Memory Core composition failure: Active Memory runs a lightweight embedded recall with a memory-only runtime allowlist, Memory Core provides the requested names as plugin tools, and the embedded attempt applies `toolsAllow` only after tool construction while plugin-tool materialization does not receive that runtime allowlist. Open PR #74592 already references this issue with closing syntax, so the issue should remain open until that PR is reviewed and either merged or closed.

## What This Changes

Active Memory should compose with Memory Core so lightweight embedded recall runs can use the explicitly allowlisted `memory_search` and `memory_get` plugin tools without persisting a stream-error placeholder turn.

## Best Possible Solution

Review and land or close #74592. The desired end state is a generic, extension-agnostic fix where runtime `toolsAllow` can participate in plugin-tool materialization for embedded runs, Active Memory can bind the gateway subagent when needed, and unavailable memory tools do not leave persistent user-visible placeholder transcript turns.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to enable `active-memory` and `memory-core`, trigger Active Memory’s pre-summarizer, and observe the embedded run resolving the memory-only runtime allowlist after plugin tools were not materialized for that allowlist.

## Solution Assessment

Yes. The best route is the linked generic plugin-tool materialization fix, preserving the final runtime allowlist filter and avoiding hardcoded Memory Core behavior in unrelated core paths.
