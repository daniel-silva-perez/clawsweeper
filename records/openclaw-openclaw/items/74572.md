## Summary

Keep open. Current main still matches the reported Active Memory / Memory Core composition failure, and open PR #74592 already references this issue with closing syntax, so the issue should remain open until that implementation candidate is reviewed and either merged or closed.

## What This Changes

Active Memory lightweight embedded recall should compose with Memory Core plugin tools and avoid leaving persistent placeholder assistant turns when the memory-tool allowlist resolves to no callable tools.

## Best Possible Solution

Review and land or close #74592. The desired end state is a generic, extension-agnostic tool materialization path where runtime toolsAllow can make matching plugin tools available for embedded runs without hardcoding Memory Core behavior in unrelated core code.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable active-memory and memory-core, trigger Active Memory's pre-summarizer, and observe that the memory runtime allowlist is applied after plugin tools are materialized from policy allowlists that do not include runtime toolsAllow.

## Solution Assessment

Yes. The best fix direction is the linked generic plugin-tool materialization change, because it preserves the final runtime allowlist filter and avoids special-casing Memory Core or Active Memory in core paths.
