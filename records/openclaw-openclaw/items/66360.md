## Summary

Keep open. Current main still has no configurable per-transcript `.jsonl` size or line cap under `session.maintenance`; the existing maintenance path covers session-entry pruning, entry caps, reset/deleted archive cleanup, and whole sessions-directory disk-budget eviction, but it does not rotate or tail-trim an active referenced transcript before it grows large.

## What This Changes

The issue requests session maintenance controls that cap or rotate individual transcript `.jsonl` files so long-running topic-bound sessions cannot grow without bound and overload the Gateway.

## Best Possible Solution

Add a first-class transcript maintenance path that caps active `.jsonl` transcripts without deleting the live session: schema/help/docs fields such as `transcriptRotateBytes` plus an optional retained-line limit, a hot-path check scoped to the active transcript, an explicit cleanup/bulk path, safe archive naming and permissions, and regression tests for header preservation, rollback, concurrent writers, disabled defaults, and docs/schema drift.

## Repair Work Prompt

Fix the unbounded session transcript `.jsonl` growth reported in openclaw/openclaw#66360. Current `session.maintenance` only handles session-entry pruning/capping and sessions-directory disk budget; it does not cap active referenced transcripts. Add a narrow first-class transcript maintenance feature, preferably new optional fields such as `session.maintenance.transcriptRotateBytes` and `session.maintenance.transcriptMaxLines` rather than reusing deprecated `rotateBytes`. Use closed PR #66546 only as source context, not as a blind port.
