## Summary

Keep open. Current main still uses static GitHub Copilot model catalog/fallback behavior, so the PR is not obsolete, but the proposed dynamic discovery implementation is not merge-ready because it drops agent-scoped auth context, caches one account's model capabilities process-wide, and regresses current xhigh fallback behavior when live discovery is unavailable.

## What This Changes

The PR adds runtime GitHub Copilot `/models` discovery with capability mapping, process caching, catalog augmentation, dynamic-model preparation, thinking-profile resolution/tests, and small SDK/test cleanup changes.

## Best Possible Solution

Land a revised GitHub Copilot plugin implementation that fetches authenticated model capabilities through provider-owned hooks, preserves current conservative fallback behavior when discovery fails, scopes cached model data to the relevant credential/baseUrl/agent context, and includes focused tests for agent-local auth profiles, multi-agent/profile switching, cache invalidation, and fallback thinking levels.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths: an agent-local Copilot auth profile can pass catalog preflight but return no dynamic models because the shared fetch helper omits `agentDir`, and a discovery failure for `gpt-5.4` falls back to a profile without current-main `xhigh`.

## Solution Assessment

No, not as written. Dynamic `/models` discovery is the right direction, but the narrow maintainable fix must pass the credential context into discovery, key or invalidate the cache by credential/baseUrl/agent context, preserve known-model fallback thinking behavior, and rebase onto current main before validation.
