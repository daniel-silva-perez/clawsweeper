## Summary

Keep open. Current main still implements Android node location as foreground-only: the settings model has only `off` and `whileUsing`, legacy `always` values are migrated down to `whileUsing`, the settings UI exposes no `Always` option, and `location.get` returns `LOCATION_BACKGROUND_UNAVAILABLE` whenever the app is backgrounded. The related closed #71015 covered a different metadata-delivery idea, not this foreground-service/background-permission request.

## What This Changes

Request Android support for `location.enabledMode: always` using background location permission and a foreground service so scheduled `nodes location get` calls can succeed while the Android app is backgrounded.

## Best Possible Solution

Make an explicit Android product/privacy decision, then implement `Always` as a first-class Android location mode with background-location permission handling, a `foregroundServiceType="location"` foreground service for short-lived fixes, preserved `whileUsing` background denial, Android unit tests, and updated public docs.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence repro path: with `isForeground = false`, `LocationHandler.handleLocationGet` returns `LOCATION_BACKGROUND_UNAVAILABLE`, and the settings/model code has no reachable `Always` state.

## Solution Assessment

No implemented solution exists on main. The proposed foreground-service path is plausible for Android, but it is not a routine cleanup because it expands background location collection and app-store policy surface.
