## Summary

Current main already implements the requested core behavior: file logs rotate at `logging.maxFileBytes`, keep bounded numbered archives, and continue writing after rotation instead of suppressing diagnostics at the size cap. The related PR #66457 was closed unmerged, but equivalent behavior is present and documented on main and recorded in the v2026.4.26 changelog.

## Best Possible Solution

Keep the built-in rotation implementation that shipped on main and in v2026.4.26. If maintainers want compressed archives or retention tuning beyond the current bounded archives, track that as a separate focused enhancement rather than keeping this broad log-rotation request open.
