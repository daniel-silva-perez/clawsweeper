## Summary

Keep open. Current main has not implemented the requested removal of locked disk re-reads, and the related open report #70857 corroborates the long-held `sessions.json.lock` symptom. The PR still needs maintainer review because its current change would route locked updates through the default cache-hit path, which clones/parses the full cached store under the lock.

## What This Changes

The PR removes `{ skipCache: true }` from the locked `updateSessionStore` and `updateSessionStoreEntry` session-store loads in `src/config/sessions/store.ts` and updates the adjacent comment.

## Best Possible Solution

Rebase the branch onto current main and give the locked session-store update helpers a coherent no-disk-read, no-full-clone path while preserving the existing lock and write-through cache semantics. Add a focused regression or performance guard that would fail if large cached stores are parsed inside the write lock.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid cloning the cached store inside the lock:** `src/config/sessions/store.ts:427`
  - body: This call uses `loadSessionStore`'s default cache-hit behavior, which goes through `readSessionStoreCache` and `cloneSessionStoreRecord`; that still `JSON.parse`s the full cached `sessions.json` while the session-store lock is held. For the large stores described here, the event-loop stall can remain, so the locked update helpers need a no-clone cache path or equivalent.
