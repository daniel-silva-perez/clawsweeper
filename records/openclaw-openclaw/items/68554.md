## Summary

Keep this PR open: it targets a plausible outage-class session-store performance problem, and the related discussion in #70857 corroborates long-held `sessions.json.lock` symptoms. The current patch is not ready to merge because current main still depends on the forced locked disk read for a stale-cache guard, and the proposed default cache path still clones/parses the full cached store under the lock.

## What This Changes

The PR removes `{ skipCache: true }` from the two locked `loadSessionStore` calls in `src/config/sessions/store.ts` and updates the adjacent comment.

## Best Possible Solution

Rebase this PR onto current main and implement a locked session-store load path that avoids both the synchronous disk re-read and the full cached-store clone/parse, while either preserving the tested external-writer freshness guarantee or deliberately updating that contract with matching tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve the stale-cache bypass for external writes:** `src/config/sessions/store.ts:650`
  - body: Current main has a regression test that primes the cache, changes `sessions.json` on disk while preserving mtime, then expects `updateSessionStoreEntry` to merge with that disk-only change. Replacing the locked read with default `loadSessionStore(storePath)` lets a stat-matching stale cache win, so the save can clobber the external update unless the rebase adds an equivalent freshness guard or updates the contract and test.
  - confidence: 0.82
- **[P2] Avoid cloning cached stores inside the write lock:** `src/config/sessions/store.ts:427`
  - body: On a cache hit, `loadSessionStore` calls `readSessionStoreCache`, which returns `cloneSessionStoreRecord` via `JSON.parse(serialized ?? JSON.stringify(store))`. The new default call therefore still parses the whole large `sessions.json` under the session-store lock, leaving much of the reported event-loop stall in place; the locked path needs a no-clone/no-parse cache read or equivalent.
  - confidence: 0.86
