## Summary

Keep this PR open. Current main still forces a cache-bypassing session-store load in both locked update paths, and the PR is tied to a plausible outage-class symptom also reported in #70857. The visible patch is not merge-ready as-is because switching to `loadSessionStore(storePath)` still goes through the cache clone path, which performs a full `JSON.parse` under the lock instead of the near-zero-cost cache read described by the PR.

## What This Changes

The PR removes `skipCache: true` from the locked `updateSessionStore` and `updateSessionStoreEntry` session-store loads in `src/config/sessions/store.ts` and updates the adjacent comment.

## Best Possible Solution

Rebase the PR onto current main and make the locked session-store update paths avoid both the disk re-read and the full cached-store clone/parse, while preserving the existing lock and write-cache semantics. A focused regression or performance guard around large cached stores would make the fix easier to trust.
