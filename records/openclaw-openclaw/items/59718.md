## Summary

Keep this issue open. Current main still carries custom Docker bind writability through path resolution, but sandbox write authorization still has a global `workspaceAccess === "rw"` gate, so the requested least-privilege writable-bind mode is not implemented. The related open bind-source and sandbox workspace reports are adjacent, not canonical replacements for this mount-level authorization request.

## What This Changes

Allow normal sandbox file tools to write only to explicitly `:rw` custom Docker binds while `workspaceAccess` remains `"none"`, preserving read-only binds and keeping the main workspace isolated.

## Best Possible Solution

Implement a mount-aware sandbox write authorization contract in the filesystem bridges: allow mutations only when the resolved target is on a writable mount, keep `workspaceAccess: "none"` from granting broad workspace writes, preserve `:ro` bind failures, and add focused regression coverage for `write`, `edit`, `mkdir/remove/rename`, and `apply_patch` behavior on `:rw` and `:ro` custom binds.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is to configure `workspaceAccess: "none"` plus a `:rw` Docker bind and call `write` or `edit` on the bind target; current resolution can mark the bind writable, but `ensureWriteAccess` rejects before mutation because `allowsWrites("none")` is false.

## Solution Assessment

Yes, the requested mount-aware authorization direction is the narrow maintainable fix, but it should be implemented with explicit sandbox security review and apply_patch policy coverage rather than by broadly treating `workspaceAccess: "none"` as writable.
