## Summary

Keep this issue open. Current main still resolves custom Docker binds and records their per-bind writability, but the normal sandbox filesystem write path still has a global `workspaceAccess === "rw"` gate. With `workspaceAccess: "none"`, a target on a `:rw` bind can resolve as writable and still be rejected before `write`, `edit`, or `apply_patch` can mutate it. The related open bind-source issue #57271 covers one adjacent setup blocker, but it does not cover this mount-level write authorization feature.

## Best Possible Solution

Keep this issue open and implement a mount-aware write authorization path in the sandbox filesystem bridges. The best fix would allow `write`, `edit`, and `apply_patch` only when the resolved target is on a writable mount, keep the default sandbox workspace and `:ro` binds read-only under `workspaceAccess: "none"`, add regression tests for `:rw` and `:ro` custom binds in that mode, and coordinate with #57271 for any remaining external bind-source usability work.

## Risks / Open Questions

- The fix is security-sensitive: authorization should depend on the resolved mount's writability without making the isolated workspace or `:ro` binds writable by accident.
- Docker, local bridge, and remote/OpenShell bridge behavior need to stay aligned, or operators could see different least-privilege semantics by backend.
