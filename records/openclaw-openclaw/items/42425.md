## Summary

Current main still does not implement this PR's intended fix. Gateway startup loads internal hooks once from the default workspace, the hook loader still discovers one workspace hooks directory per call, and the multi-agent workspace enumeration helper is not wired into startup hook loading. The PR remains a plausible fix candidate, but it needs a rebase against the current `server-startup-post-attach.ts` path and reconciliation with the current explicit opt-in/no-override workspace hook policy.

## Best Possible Solution

Keep this PR open as a fix candidate. The best path is a rebase or replacement patch that wires multi-workspace hook loading into `src/gateway/server-startup-post-attach.ts`, enumerates workspaces with `listAgentWorkspaceDirs`, loads shared hooks once, loads explicitly enabled workspace-local hooks once per configured workspace, scopes local handlers to the matching event/session/agent workspace, isolates bad workspace failures, preserves the current no-override contract, and adds focused Gateway startup regression coverage.

## Risks / Open Questions

- The PR is stale against current main because the startup integration point is now `src/gateway/server-startup-post-attach.ts`; `src/gateway/server-startup.ts` is only a re-export wrapper.
- Loading hooks from every configured workspace broadens trusted local code execution at Gateway startup, so a refreshed patch needs explicit opt-in, scope guards, boundary checks, and per-workspace failure isolation.
- The PR discussion's earlier shared/local override behavior needs reconciliation with current main's documented and tested policy that workspace hooks cannot override bundled, managed, or plugin hooks.
