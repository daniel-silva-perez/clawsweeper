## Summary

Keep this issue open. Current main still has no direct zero-LLM cron exec payload: cron payload types and Gateway validators only accept systemEvent or agentTurn, job validation rejects non-agentTurn isolated/current/session jobs, detached cron execution still calls the isolated agent runner, and CLI/UI/docs/tool surfaces expose only the agent-mediated path. The related implementation PR #51276 is closed unmerged, while several duplicate shell/exec cron requests were consolidated into this thread rather than superseding it.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The best implementation is an additive cron payload kind for direct command execution that updates cron types, Gateway protocol/schema validation, job normalization/validation, runtime dispatch, run history, retry/failure accounting, delivery, CLI add/edit/run surfaces, Control UI, docs, and focused tests, with explicit secops review of the host-command security policy.

## Risks / Open Questions

- A direct cron exec runner is security-sensitive because persisted schedules would be able to launch host commands; the design needs explicit policy for approvals, sandbox/host routing, shell selection, cwd/env handling, output truncation, logs, cancellation, timeouts, retries, and delivery.
- Closing this now would orphan the most concrete canonical tracker after several duplicate shell/exec cron requests were consolidated here and the direct implementation PR was closed unmerged.
- The existing exec tool and exec-approval surfaces do not by themselves implement cron payload persistence, Gateway protocol validation, scheduler dispatch, run history, or channel delivery for zero-LLM scheduled command execution.
