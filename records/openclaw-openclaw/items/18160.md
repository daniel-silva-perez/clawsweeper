## Summary

Keep open. Current main still exposes only agent-mediated cron payloads for isolated/current/session jobs, the related implementation PR is closed unmerged, and the remaining design is security-sensitive enough to need maintainer/secops policy before implementation.

## What This Changes

Add a direct cron execution payload that runs a configured command on schedule, captures stdout/stderr and exit status, updates cron run state, and optionally delivers output without starting an LLM agent turn.

## Best Possible Solution

Keep this as the canonical core tracker and land an additive, owner-reviewed cron payload kind for deterministic command execution, covering types, Gateway schema, normalization/validation, scheduler dispatch, run logs and error accounting, delivery, CLI, Control UI, docs, tests, and an explicit secops policy for scheduled host commands.
