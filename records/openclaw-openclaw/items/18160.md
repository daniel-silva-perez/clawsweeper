## Summary

Keep open. Current main still has no zero-LLM cron exec payload: the public cron contract only accepts systemEvent and agentTurn, isolated/current/session jobs are still validated as agentTurn-only, and detached runtime execution still enters the isolated agent runner. The discussion also shows this is the canonical core tracker after duplicate shell/exec cron requests and a closed unmerged implementation PR, with several recent production datapoints and a useful external shim workaround that does not replace OpenClaw cron run history, retry accounting, or delivery.

## What This Changes

Add a direct cron exec payload and CLI/UI mode that runs scheduled shell commands without an LLM turn while preserving cron tracking and delivery.

## Best Possible Solution

Keep this as the canonical core cron tracker and land an additive, owner-reviewed command payload that updates the cron type/schema contract, validation, scheduler dispatch, run history/accounting, delivery, CLI, Control UI, docs, and tests, with a documented host-command security model before merge.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to submit or validate a cron job with payload.kind="exec" on current main; the source contract rejects that payload, isolated job validation requires agentTurn, and detached execution has no direct exec branch.

## Solution Assessment

Yes for the requested direction, but not as an unsupervised quick patch. An additive core cron payload is the maintainable solution because ClawHub plugins or the posted ollama-exec-shim workaround cannot own OpenClaw's persisted scheduler, run history, retry accounting, and delivery contract; the exact design needs maintainer and secops review.
