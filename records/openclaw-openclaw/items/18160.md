## Summary

Keep this issue open. Current main still has no direct cron exec payload: cron payload types and Gateway schemas only accept `systemEvent` or `agentTurn`, job validation still requires isolated/current/session jobs to be `agentTurn`, runtime dispatch still sends detached cron work through `runIsolatedAgentJob`, and CLI/UI/docs expose only agent-mediated cron creation. The prior implementation PR #51276 was closed unmerged, while several duplicate exec/shell cron requests have been closed toward this issue, so this remains the right canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical core direct-exec cron tracker. The best implementation path is an additive `exec` or shell payload across cron types, Gateway schema/validators, normalization, runtime dispatch, CLI add/edit, Control UI, agent-facing cron tooling, docs, and focused tests, with an explicit security policy for host command execution and integration with existing run history, retries, failure alerts, timeouts, cancellation, and delivery behavior.

## Risks / Open Questions

- A native cron exec runner would add a privileged host-command surface and needs maintainer design for approvals/sandbox policy, shell selection, environment handling, cwd, output capture/truncation, timeout/cancellation, logging, retries, and delivery semantics.
- Closing this now would orphan the canonical tracker after duplicate direct-exec cron issues were already closed toward it and the implementation PR was closed unmerged.
- Recent cron reliability fixes reduce some agent-turn failure modes but still leave deterministic shell jobs dependent on an LLM session, so they do not satisfy the zero-LLM stdout-to-channel use case.
