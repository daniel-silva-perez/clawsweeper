## Summary

Keep open. Current main still has no zero-LLM cron exec payload in the type contract, Gateway schema, validation, runtime dispatch, CLI, Control UI, or docs, and the latest tagged release still shows the same limitation. The discussion has multiple production datapoints, a closed unmerged implementation PR, and an external shim workaround, but no merged OpenClaw core implementation.

## What This Changes

Request direct exec cron payloads that run scheduled shell commands without creating an LLM agent turn while preserving run history and channel delivery.

## Best Possible Solution

Keep this as the canonical core tracker and land an additive, owner-reviewed cron command payload with a documented host-command security model, Gateway protocol/schema updates, scheduler dispatch, run history/accounting, delivery behavior, CLI and Control UI support, docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence current-main check is to attempt or validate a cron job with `payload.kind: "exec"`; the public schema rejects it, `assertSupportedJobSpec` rejects non-`agentTurn` isolated jobs, and the detached runtime has no direct execution branch.

## Solution Assessment

Yes, the requested additive exec payload is the right core-level direction for deterministic scheduled shell work because external cron loses OpenClaw run history and delivery. The implementation should wait for maintainer/secops policy rather than be treated as a narrow automated fix.
