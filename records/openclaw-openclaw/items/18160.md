## Summary

Keep this issue open. Current main still lacks a direct zero-LLM cron exec payload: the cron types, Gateway schema, validators, runtime dispatch, CLI, Control UI state, and docs all remain limited to `systemEvent` and `agentTurn`. The attempted implementation PR is closed unmerged, and the discussion already treats this issue as the canonical tracker for the remaining work.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The best path is an additive `exec` or shell cron payload across cron types, Gateway schemas, normalization, validation, runtime dispatch, CLI add/edit, Control UI, agent-facing cron tooling, docs, and focused tests, with an explicit security policy for host command execution and integration with existing run history, retries, failure alerts, timeouts, cancellation, and delivery behavior.

## Risks / Open Questions

- A native cron exec runner would add a privileged host-command surface and needs explicit design for approvals, sandbox policy, shell selection, environment/cwd handling, output capture/truncation, timeout/cancellation, logging, retries, alerts, and delivery semantics.
- Closing this now would orphan the canonical tracker after duplicate direct-exec cron requests were closed toward this issue and the implementation PR was closed unmerged.
- Recent cron reliability and delivery improvements reduce some agent-turn failure modes but do not satisfy the zero-LLM stdout-to-channel use case described by the reporter and commenters.
