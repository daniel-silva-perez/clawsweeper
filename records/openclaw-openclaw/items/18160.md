## Summary

Keep this issue open. Current main still does not implement a direct cron exec payload: the cron type contract, Gateway schema, job validator, runtime dispatch, CLI, Control UI, agent-facing cron tool, and public docs all expose only `systemEvent` and `agentTurn`. The closed implementation PR #51276 was not merged, and the provided discussion makes this issue the active canonical tracker for duplicate direct-exec cron requests.

## Best Possible Solution

Keep this issue open as the canonical core direct-exec cron tracker. The best path is an additive `exec` or shell payload across cron types, Gateway schema/validators, normalization, runtime dispatch, CLI add/edit, Control UI, agent-facing cron tool guidance, docs, and focused tests, implemented with a controlled process runner integrated with existing run history, failure alerts, delivery behavior, timeouts/cancellation, and exec security policy.

## Risks / Open Questions

- A native cron exec runner would add a privileged host-command surface and needs explicit maintainer design for exec approvals, sandbox/security policy, shell selection, environment handling, working directory, output capture/truncation, timeouts, cancellation, logging, retries, and delivery semantics.
- Closing this now would orphan the canonical tracker after several duplicate direct-exec cron requests were already closed toward it and the only implementation PR was closed unmerged.
- Recent cron reliability fixes reduce some symptoms but still route deterministic shell work through `agentTurn`, so they do not satisfy the zero-LLM stdout-to-channel use case.
