## Summary

Keep open. Current main and the latest tagged release still only support `systemEvent` and `agentTurn` cron payloads; direct zero-LLM shell execution is missing from the type contract, Gateway schema, validation, scheduler dispatch, CLI, Control UI, and docs. The discussion provides strong production evidence, a closed unmerged implementation PR, and an external shim workaround, but no merged OpenClaw core implementation.

## What This Changes

The issue requests an additive cron payload mode that runs a configured command directly, captures stdout/stderr, records run status, and optionally delivers output without starting an LLM agent turn.

## Best Possible Solution

Keep this as the canonical core tracker and land an additive, owner-reviewed cron command payload with a documented host-command security model, Gateway protocol/schema updates, scheduler dispatch, run history/accounting, delivery behavior, CLI and Control UI support, docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence current-main check is to validate or create a cron job with `payload.kind: "exec"`; the public schema has no such variant, `assertSupportedJobSpec` rejects non-`agentTurn` isolated jobs, and detached runtime dispatch has no direct execution branch.

## Solution Assessment

Yes, the requested additive core cron payload is the right direction for deterministic scheduled shell work because current plugin/external cron workarounds cannot satisfy the existing OpenClaw cron store, run history, retry, and channel delivery contract without core schema and runtime changes. The implementation should wait for maintainer and secops policy review rather than being treated as a narrow automated fix.
