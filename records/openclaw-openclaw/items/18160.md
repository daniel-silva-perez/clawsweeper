## Summary

Keep this issue open. Current main still does not implement a zero-LLM cron exec payload: the core cron payload types, Gateway schema, job validator, runtime dispatch, CLI, Control UI, and docs remain limited to `systemEvent` and `agentTurn`. The attempted implementation PR #51276 is closed unmerged, while duplicate direct-exec cron requests were already closed toward this issue, making this the canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The likely product path is an additive `exec` or shell cron payload spanning cron types, Gateway schemas, normalization/validation, runtime dispatch, CLI add/edit, Control UI, agent-facing cron tools, docs, and focused tests, with a clear host-command security policy and integration with existing run history, retries, failure alerts, timeout/cancellation, and delivery behavior.

## Risks / Open Questions

- A native cron exec runner would introduce privileged host command execution and needs an explicit policy for approvals, sandboxing, shell selection, environment/cwd handling, output truncation, timeouts, cancellation, run logs, retries, alerts, and channel delivery.
- Closing this issue now would orphan the canonical tracker after duplicate direct-exec cron issues were already consolidated here and the implementation PR was closed unmerged.
- Recent cron reliability and delivery improvements may reduce some agentTurn pain, but they do not satisfy the zero-LLM stdout-to-channel use case requested here.
