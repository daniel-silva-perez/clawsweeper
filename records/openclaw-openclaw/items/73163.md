## Summary

Keep open. The PR is directionally the right fix for the paired operator-warning request, and exact-head checks are green, but the latest patch still has a P2 correctness bug: the warning predicate reads raw startup config while the gateway has already resolved runtime auth and Control UI overrides.

## What This Changes

The PR adds a gateway startup warning for remote plain-HTTP Control UI secure-context rejection, suppression coverage, regression tests, and a changelog entry.

## Best Possible Solution

Thread the resolved Control UI enabled state and resolved gateway auth mode from runtime config through the post-attach startup logging boundary, keep the existing auth rejection behavior unchanged, and add focused override quiet-case tests before landing the warning for #71669.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: start with config that would warn, then use runtime overrides such as controlUiEnabled: false or auth mode trusted-proxy; the PR warning still reads cfgAtStart instead of the resolved runtime values.

## Solution Assessment

Mostly yes. Startup logging is the narrowest maintainable place for this operator guidance, but this implementation should use resolved runtime state so the warning matches the gateway that actually started.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use resolved startup state for the warning:** `src/gateway/server-startup-log.ts:66-72`
  - body: `logGatewayStartup` derives `controlUiEnabled` and `auth.mode` from `cfg`, but startup already resolved server/CLI overrides before this log runs. A gateway started with an override like `controlUiEnabled: false` or `--auth trusted-proxy` can still emit the insecure Control UI warning because `startGatewayPostAttachRuntime` passes only `cfgAtStart`; thread the resolved state into the logger and cover those quiet cases.
