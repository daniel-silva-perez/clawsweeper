## Summary

Keep this PR open. The core warning is useful and current main does not implement it yet, but the proposed predicate would warn for supported trusted-proxy Control UI deployments where current auth policy already permits operator access without device identity.

## What This Changes

The PR adds a gateway startup warning and tests for non-loopback plain-HTTP Control UI access, while also changing package-acceptance workflow assertions and unit-fast test routing.

## Best Possible Solution

Land a scoped startup warning whose predicate mirrors the actual Control UI auth policy: non-loopback plain HTTP should warn only when device identity will really be required, with trusted-proxy covered as a quiet exception, focused regression tests, and a user-facing changelog entry.

## Reproduction Assessment

Yes. For the PR bug, configure non-loopback HTTP with Control UI enabled and gateway.auth.mode="trusted-proxy"; the PR helper would warn even though current connect-policy and tests allow trusted-proxy operator Control UI without device identity.

## Solution Assessment

No. Adding the warning in server-startup-log is the right boundary, but the predicate needs to include trusted-proxy semantics and the branch should either justify or remove the unrelated test-harness changes.
