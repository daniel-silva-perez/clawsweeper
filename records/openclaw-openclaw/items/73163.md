## Summary

Keep this PR open, but do not merge it as-is. The warning belongs in gateway startup logging, but the current patch derives the warning from static config instead of the resolved runtime startup state, so runtime overrides can produce incorrect operator warnings.

## What This Changes

The PR adds a gateway startup warning for non-loopback plain-HTTP Control UI access that requires secure browser context auth, plus focused startup-log tests and a changelog entry.

## Best Possible Solution

Keep the startup warning, but thread the resolved Control UI enabled state and resolved gateway auth mode through the post-attach logging boundary so the message reflects the gateway that actually started.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: use config that would warn, then start with runtime overrides such as controlUiEnabled: false or trusted-proxy auth; the PR helper still reads cfgAtStart and can warn anyway.

## Solution Assessment

Mostly yes, but not in this exact form. Startup logging is the narrowest maintainable place for this operator guidance, but it should consume resolved runtime state rather than re-deriving from static config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87
