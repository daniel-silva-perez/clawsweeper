## Summary

Keep this PR open. Current main does not implement the PR’s intended startup behavior: `tools.exec.pathPrepend` is still applied to spawned exec environments, while the binary-detection paths named in the PR still read `process.env.PATH` directly. The PR remains a plausible fix candidate, but it needs maintainer review for global PATH side effects, reload semantics, and docs/test alignment.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is either to land a config-aware implementation that reuses the existing path-prepend helper, adds focused config-load/reload tests, and updates docs/help for the broader contract, or to choose a narrower design where `hasBinary`/media discovery receives an explicit effective PATH without globally mutating `process.env.PATH`.

## Risks / Open Questions

- Security review: the PR changes `tools.exec.pathPrepend` from an exec-environment setting into a global process PATH mutation. That could affect binary discovery and subprocesses outside the exec tool, so maintainers should decide whether this expanded trust boundary is intended and document it if accepted.
- The PR’s latest commit addresses the Greptile trim/dedupe review, but the later review comment about config reloads remains material: reducing or clearing `tools.exec.pathPrepend` in-process could leave previously prepended directories in `process.env.PATH` until restart unless the implementation tracks and removes owned entries.
