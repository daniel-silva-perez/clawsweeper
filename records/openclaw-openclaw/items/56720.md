## Summary

Keep open. The PR is cleanly applicable to current main and addresses most of the Gmail watcher regression, but it still conflicts with its own documented empty-list behavior: `excludeLabels: []` omits the flag, while upstream `gog` only disables exclusions when the flag is present with an empty value. Current main still lacks the regression fix, and the linked bug remains open, so this PR needs one more branch update before merge.

## What This Changes

This PR adds `hooks.gmail.excludeLabels` config/runtime plumbing, passes default Gmail watcher excluded labels, updates sanitized log expectations, generated config metadata, and Gmail watcher docs/tests.

## Best Possible Solution

Land one canonical fix that restores default `SPAM,TRASH,DRAFT,SENT` watcher exclusions, preserves explicit empty-list opt-out by passing an empty `--exclude-labels` value, keeps schema/docs/tests/generated metadata aligned, and then closes the linked Gmail regression while superseding overlapping open fix PRs.

Yes. A high-confidence source-level reproduction exists: current main omits `--exclude-labels` from the watcher serve args, and upstream `gog` defaults an omitted flag to `SPAM,TRASH`, so SENT and DRAFT are not excluded.

## Solution Assessment

No, not as-is. The typed config/runtime approach is the right narrow direction, but the builder must preserve the documented `excludeLabels: []` semantics by sending an explicit empty flag value or the docs/tests must stop promising that behavior.
