## Summary

Current main still does not apply `cron.sessionRetention` to subagent or ACP session rows. The cron reaper remains cron-run-key-only, subagent/ACP key predicates exist but are not used by that retention path, and the default native subagent cleanup path still keeps run/session-mode rows outside the `sessions.delete` archive branch. The related implementation PR #51861 was closed unmerged, while the closed duplicate reports add scope rather than superseding this canonical issue.

## Best Possible Solution

Keep this issue open as the canonical tracking item. Implement a restart-safe session-store-level retention path for completed native subagent and ACP sessions, either as a generic ephemeral-session reaper or documented subagent/ACP retention setting. The fix should preserve active and intentionally persistent sessions, use durable terminal markers such as `endedAt`, `abortedLastRun`, or `cleanupCompletedAt`, delete/archive via existing session deletion helpers, run on startup/store-write or another persistent catch-up path, and add focused tests for expired cron runs, completed subagent/ACP runs, running subagents, keep/session-mode behavior, and transcript cleanup.

## Risks / Open Questions

- A naive expansion of the cron reaper could delete active, resumable, thread-bound, or intentionally persistent keep/session-mode rows unless it uses durable terminal-state and cleanup-policy markers.
- Relying on generic `session.maintenance` count/age cleanup is a blunt fallback and can still miss the reported short retention window or evict the wrong sessions under pressure.
