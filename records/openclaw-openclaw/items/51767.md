## Summary

Keep open. Current main still applies `cron.sessionRetention` only through the cron-run reaper, while default keep-mode subagent sessions and ACP session rows have separate lifecycle cleanup that does not provide an equivalent durable short-retention sweep. The related reports are closed or folded into this one, and the direct fix PR was closed unmerged, so this remains the canonical tracker.

## What This Changes

Apply session retention or an equivalent durable cleanup policy to completed subagent and ACP sessions instead of only pruning isolated cron-run session entries.

## Best Possible Solution

Keep this as the canonical tracker and design a restart-safe retention path for completed native subagent and ACP sessions at the session-store or lifecycle boundary. The finished behavior should preserve active and intentionally persistent sessions, use existing transcript archival/deletion helpers, cover startup or store-write catch-up, and document the chosen retention knob.

## Reproduction Assessment

Yes for the code-path reproduction: current main still has a cron reaper guard that only accepts `isCronRunSessionKey`, and the existing test explicitly preserves an old non-cron row. I did not live-reproduce the reported RSS growth, but the cleanup exclusion is directly visible in source and tests.

## Solution Assessment

No, simply broadening the key guard is not clearly the best fix because subagent keep/session modes and ACP lifecycle semantics have separate ownership and terminal markers. The safer path is a lifecycle-aware durable retention sweep with targeted tests for completed keep-mode subagents, active/thread-bound preservation, ACP parent-owned sessions, and transcript cleanup.
