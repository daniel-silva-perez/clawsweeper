## Summary

Keep this open. Current main still has a plausible cron scheduler recovery gap, and an open same-author PR already targets the reported runtime timer-chain stall with closing syntax.

## What This Changes

The issue asks for cron runtime scheduling to recover when automatic timer ticks stop after long Gateway uptime, leaving `nextWakeAtMs` stale until restart.

## Best Possible Solution

Land PR #73355 or an equivalent narrow cron-service fix that makes rejected timer ticks self-recover, adds any needed independent watchdog lifecycle, and includes regression coverage for the stalled-chain path.

## Reproduction Assessment

Unclear. The report gives concrete runtime evidence, including stale `nextWakeAtMs` and missing run-log entries, but a high-confidence live reproduction would require the reported multi-day macOS Gateway uptime scenario or a focused unit test that forces the rejection path.

## Solution Assessment

Unclear. Re-arming after timer-tick rejection is a narrow maintainable direction, but maintainers should decide whether the added interval watchdog is necessary beyond catch recovery and verify its lifecycle behavior.
