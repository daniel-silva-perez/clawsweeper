## Summary

Keep open: the requested migration still appears valid, but this PR is not ready to merge because its validation helper accepts explicit `gateway.bind="auto"` for Tailscale Serve/Funnel and the branch is stale against the current main migration layout.

## What This Changes

The PR branch adds a legacy config rule, migration, shared loopback helper, validation rewrite, and tests to normalize literal non-loopback `gateway.bind` values to `loopback` when Tailscale Serve/Funnel is configured.

## Best Possible Solution

Refresh the fix on current main as a doctor/shared runtime legacy migration plus a source-literal legacy rule for `lan`, `tailnet`, and non-loopback `custom` under `serve` or `funnel`; keep explicit `auto` rejected by validation and cover ordering before the existing `allowedOrigins` migration with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main rejects non-loopback and explicit `auto` Serve/Funnel binds, while the PR helper makes explicit `auto` pass validation; the original legacy `lan` plus `serve` crash-loop path remains plausible from the current missing migration evidence.

## Solution Assessment

No. The migration direction is appropriate, but this implementation is not the best fix because it relaxes the validation contract and edits stale migration files; the safer alternative is a current-main runtime migration that does not change validation semantics.
