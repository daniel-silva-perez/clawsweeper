## Summary

Keep open: the reported Tailscale Serve/Funnel migration bug is still a valid gap on current main, but this PR head is not safe to merge because it relaxes explicit `gateway.bind="auto"` validation and targets an obsolete migration file layout.

## What This Changes

The PR adds config legacy rules, a legacy migration, shared bind-resolution helpers, and tests to normalize literal Tailscale Serve/Funnel non-loopback `gateway.bind` values to `loopback`.

## Best Possible Solution

Refresh the fix on current main by adding a doctor/shared runtime legacy migration and source-literal legacy rule for `lan`, `tailnet`, and non-loopback `custom` under `serve` or `funnel`; preserve the current rejection of explicit `auto`; and cover migration ordering before `allowedOrigins` seeding with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main rejects `gateway.bind="lan"` with `tailscale.mode="serve"`, while PR head's shared helper returns false for explicit `auto`, causing validation to accept a config current main rejects.

## Solution Assessment

No. The requested migration direction is appropriate, but this PR is not the best current implementation because it relaxes `auto` validation and edits a migration file that no longer exists on main; the safer path is to rebase into the doctor/shared runtime migration layout while keeping validation behavior unchanged.
