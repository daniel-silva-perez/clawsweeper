## Summary

Keep open. The underlying config/gateway bug remains valid on current main, but this PR is not merge-ready because its helper makes explicit `gateway.bind="auto"` pass the Serve/Funnel loopback check and the migration still targets files that no longer exist in current main's doctor/shared migration layout.

## What This Changes

The PR branch adds a legacy config rule, migration, shared loopback helpers, and tests to normalize literal non-loopback `gateway.bind` values under Tailscale Serve/Funnel to `loopback`.

## Best Possible Solution

Refresh the fix on current main by adding a doctor/shared runtime legacy migration plus a source-literal legacy rule for `lan`, `tailnet`, and non-loopback `custom` under `serve` or `funnel`; preserve explicit `auto` rejection in validation; and cover ordering before `allowedOrigins` seeding with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main rejects non-loopback Serve/Funnel binds, while PR head routes validation through a helper that treats explicit `auto` as safe; no live launchd startup run was needed for this read-only review.

## Solution Assessment

No. The migration direction is appropriate, but this implementation is not the best fix because it relaxes a security-sensitive validation path and edits obsolete migration files; the safer path is a current-main doctor/shared migration that preserves the loopback-only contract.
