## Summary

Keep open. The underlying gateway/config bug is valid and current main still lacks the requested Serve/Funnel bind migration, but this PR head is not merge-ready because it weakens the loopback-only validation for explicit `gateway.bind="auto"` and targets a migration file that no longer exists on current main.

## What This Changes

The PR branch adds a legacy config rule, migration, shared loopback bind helpers, and tests to normalize literal non-loopback `gateway.bind` values under Tailscale Serve/Funnel to `loopback`.

## Best Possible Solution

Refresh the fix on current main by adding a doctor/shared runtime legacy migration plus a source-literal legacy rule for `lan`, `tailnet`, and non-loopback `custom` under `serve` or `funnel`; keep explicit `auto` rejected by validation; and cover migration ordering before `allowedOrigins` seeding with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main rejects non-loopback Serve/Funnel binds, while PR head routes validation through a helper that treats explicit `auto` as safe; no live startup run was needed for this read-only review.

## Solution Assessment

No. The migration direction is appropriate, but this implementation is not the best fix because it relaxes a security-sensitive validation path and edits an obsolete migration file; the safer path is a current-main doctor/shared migration that preserves the existing loopback-only contract.
