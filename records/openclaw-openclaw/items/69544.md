## Summary

Keep open. Current main still derives default group mention regexes from both `identity.name` and `identity.emoji`, and Matrix room mention gating consumes those regexes without a Matrix `enableEmojiMention` or per-agent `emojiMention` switch. The existing workaround is still to override `mentionPatterns`, not the requested boolean control.

## Best Possible Solution

Keep this issue open and implement a backward-compatible opt-out for emoji-derived mention patterns. The best fix is likely a generic per-agent group-chat setting plus a Matrix channel-level default, with default `true`, explicit `mentionPatterns` remaining authoritative, and native Matrix mentions continuing to work.

Implement the feature requested in https://github.com/openclaw/openclaw/issues/69544: allow Matrix users to disable agent `identity.emoji` as a derived mention trigger while preserving the current default behavior.

Expected boundary:

- Do not hard-code Matrix-specific policy into generic core unless the helper/seam is generic and safe for other channels.

- Update public config types, zod schema/help/generated config metadata, Matrix docs, shared group mention docs if needed, and the changelog with reporter attribution.

Likely implementation idea:
- Extend the shared mention-pattern derivation to accept options that can exclude `identity.emoji` while still deriving the name pattern.
- Wire Matrix's mention regex construction to pass the effective setting from channel config plus per-agent override.
- Add regression coverage proving a room message containing only the agent emoji no longer triggers when disabled, while name/native mentions still trigger and default behavior still triggers on emoji.

Do not change unrelated channel mention behavior, plugin loading, Matrix auth, Matrix outbound mention metadata, or broad group policy defaults.
