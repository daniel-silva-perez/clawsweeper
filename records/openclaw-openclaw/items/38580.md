## Summary

Current main still does not implement on-demand Telegram pinned-message reads. The bundled Telegram plugin advertises and handles send/poll/react/delete/edit/sticker/topic actions, but it does not expose `list-pins`, `pins`, or `/getpinned`; the config/schema has no Telegram `pins` gate; the existing `getChat` helper only returns chat ids; and docs/tests still list pin reads for Discord/Slack/Matrix but not Telegram. Related PRs #38613 and #38718 remain useful prior art, but both were closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and land a focused bundled Telegram plugin change that opts Telegram into the existing `list-pins`/`pins` message action backed by Bot API `getChat.pinned_message`. Add a `channels.telegram.actions.pins` config/schema gate, normalize the returned pinned message into the shared `pins` payload shape, document the chat-level limitation, and add regression tests for present pin, no pin, API error, disabled gate, account/target resolution, and forum-topic target handling. Treat #38613 and #38718 as prior art, not canonical fixes.

## Risks / Open Questions

- Adding this feature changes Telegram data access scope through the agent/tool surface: it should use the existing bot token only, return only the chat-level `getChat.pinned_message`, and include an explicit `channels.telegram.actions.pins` gate.
- Telegram Bot API exposes one current chat-level pinned message through `getChat`; topic-specific pins or pin history should not be promised unless Telegram exposes them separately.
- A native `/getpinned` command would intersect Telegram command authorization and group/topic routing; the lower-risk first implementation is the existing generic `message pins` / `list-pins` action with tests.
