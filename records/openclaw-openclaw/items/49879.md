## Summary

Keep open. Current main still treats `responseUsage` as session-scoped state only: `/usage` writes `SessionEntry.responseUsage`, the reply runner reads only the active session/session store, Gateway session rows expose that session value, and docs explicitly say the footer setting persists per session. The config surface does not define `agents.defaults.responseUsage`: the `AgentDefaultsConfig` type and strict `AgentDefaultsSchema` omit the key, and `getSessionDefaults()` returns model/context/thinking defaults without a usage-footer default. The related reports in the provided context are adjacent global-default requests, but none is a canonical tracker for this specific usage-footer default.

## Best Possible Solution

Keep this issue open for product/implementation work. A conservative implementation would add `agents.defaults.responseUsage` to `AgentDefaultsConfig`, `AgentDefaultsSchema`, generated config help/docs, and session/default resolution; initialize or resolve new sessions from that default without overriding explicit per-session `/usage` state; then add config validation plus reply-runner/session-default tests covering `off`, `tokens`, and `full`.
