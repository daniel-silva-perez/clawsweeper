## Summary

Keep open. Current main still matches the reported split-provider auth bug: runtime auth, session auto-selection, and status labels all consult canonical provider auth-profile order before the selected provider entry's configured credential on the normal path.

## What This Changes

The issue requests that split provider entries sharing one upstream auth provider use the credential intended by the selected provider entry instead of an unrelated profile from canonical `auth.order`.

## Best Possible Solution

Define a documented provider-entry auth-profile binding that preserves existing `apiKey` credential semantics, then apply that rule consistently in runtime auth resolution, session auto-selection, status labels, schema/docs, and focused regression tests for multiple provider entries sharing one upstream auth provider.

## Reproduction Assessment

Yes. The issue provides a concrete split-OpenRouter configuration, and current-main source inspection shows the deterministic path that selects from canonical profile order before provider-entry credentials unless the provider entry explicitly sets `auth: "api-key"`.

## Solution Assessment

No for a narrow `apiKey` overload as-is. The safer fix is an explicit, documented provider-entry profile binding or equivalent contract, then wiring that single rule through runtime auth, session state, and status display.
