## Summary

Keep open. Current main still routes split provider entries through canonical provider auth-profile order before falling back to the selected provider entry's `apiKey`, while the documented/type contract treats that `apiKey` as credential material rather than an auth-profile binding.

## What This Changes

The issue requests provider-entry-aware auth selection so split OpenRouter provider entries sharing the same upstream provider use the entry-intended key instead of an unrelated profile from canonical `auth.order.openrouter`.

## Best Possible Solution

Define an explicit split-provider auth contract, preferably a dedicated provider-entry auth-profile reference or other documented binding that preserves existing `apiKey` credential semantics, then centralize that rule across runtime auth, session auto-selection, status labels, docs/schema, and focused regression tests for multiple OpenRouter-style entries sharing one auth provider.
