## Summary

Keep open. Current main still has manual selected-session deletion and generic session-store maintenance, but it does not implement the requested Dashboard Sessions prune workflow: no Prune button, 48-hour default age input, orphan candidate preview, or Telegram/Discord channel/topic existence checks. Related #50310 remains open and is narrower Discord thread-session cleanup work, not a replacement for this broader Dashboard feature.

## Best Possible Solution

Keep this issue open. The best path is an additive gateway-backed prune workflow: add a preview RPC that returns stale orphan candidates with reasons, expose safe plugin-owned Telegram/Discord reachability checks through a generic seam, revalidate during apply, and wire Dashboard Sessions to show a 48-hour default age field, candidate preview, and explicit Prune action. Keep #50310 as the narrower Discord cleanup thread rather than treating it as a duplicate.

## Risks / Open Questions

- A future prune implementation must distinguish confirmed deletion from transient Telegram/Discord API failures, auth failures, permission changes, and rate limits before deleting sessions.
- Preview and apply should share one gateway-side eligibility contract and revalidate candidates during deletion so Dashboard, CLI, tests, and plugin-owned checks do not drift.
- Fan-out channel existence checks could be expensive for large session stores; implementation should bound concurrency and cache only where it cannot hide fresh deletion state.
