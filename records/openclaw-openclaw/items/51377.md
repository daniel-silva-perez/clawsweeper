## Summary

Keep open. Current main only partially covers the broader WebChat attachment side; the central OpenResponses `input_file` request is still not implemented for Excel or generic binary receipts. Excel MIME types remain outside the default `input_file` allowlist, unsupported `input_file` MIME types still fail, and accepted files are decoded into prompt context rather than persisted as durable workspace artifacts with metadata and a stable path for tools or skills.

## Best Possible Solution

Keep this issue open as the canonical Gateway/OpenResponses file-ingest request. The best implementation is a documented `input_file` contract for Excel and allowed generic binaries that fail closed, preserve current SSRF and size controls, materialize accepted binaries into managed media or workspace storage, record metadata such as filename, MIME, sha256, and size, and expose a stable receipt/path to tools and skills. The existing WebChat attachment work should remain as the chat-specific path or be generalized only through a clear shared media-ingest seam.
