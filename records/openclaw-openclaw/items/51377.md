## Summary

Keep open. Current main partially addresses the WebChat side by accepting non-video files, preserving `.xlsx` MIME from filenames, and routing non-image chat attachments through managed media paths. It still does not implement the central OpenResponses `input_file` request: Excel MIME types are not in the default allowlist, unsupported file types fail, and accepted `input_file` content is decoded into prompt context rather than persisted with a stable binary receipt/path for tools or skills.

## Best Possible Solution

Keep this issue open as the canonical Gateway/OpenResponses file-ingest request. The best implementation would add an explicit Excel/generic-binary `input_file` contract with documented MIME allowlists and size limits, fail-closed errors, durable managed-media/workspace persistence, metadata such as name/sha256/mime, and a stable receipt/path that tools and skills can consume. Existing WebChat non-video attachment support should remain as the chat-specific path, not as a substitute for `input_file`.
