## Summary

Keep open. Current main still has the exact local-media allowlist rejection, but the delivery-queue permanent-error classifier does not match it, so this issue remains a valid narrow policy fix. The broader local-media regression cluster has had related work, but this specific retry-classification gap is still present. https://github.com/openclaw/openclaw/issues/66489

## What This Changes

The issue requests that deterministic delivery failures for local media paths outside allowed directories be classified as permanent errors instead of consuming the normal retry budget.

## Best Possible Solution

Add a focused permanent delivery classification for the exact local-media allowlist rejection in the delivery-queue policy, with policy and recovery tests proving queued entries move directly to `failed/`. Leave enqueue-time validation, `lastErrorClass`, and Telegram overlong-message classification as follow-ups unless they are separately verified in the same PR.

## Reproduction Assessment

Yes. The source path is high-confidence: a queued Telegram payload with a relative local `mediaUrls` entry reaches `loadWebMedia`, `assertLocalMediaAllowed` throws the reported message, and the recovery classifier currently routes that message through retry/backoff rather than immediate failure.

## Solution Assessment

Yes for the local-media allowlist case. The narrow maintainable fix is to add an exact permanent-error classification plus tests in the delivery-queue policy/recovery layer; enqueue-time validation is useful but broader, and overlong Telegram message handling needs separate proof.
