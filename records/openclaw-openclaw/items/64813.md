## Summary

Keep open. Current main still lacks direct `params.user` passthrough, and the PR targets the right extra-params path, but the patch is unsafe as written because it injects `user` into every provider payload without the API-family guard used by neighboring payload patches.

## What This Changes

The PR adds a Pi embedded runner extra-param wrapper that normalizes `params.user` and injects it into provider request payloads via `streamWithPayloadPatch`.

## Best Possible Solution

Land a guarded static `params.user` passthrough only for provider API families that support a top-level `user` request field, with regression tests for supported injection and unsupported-provider no-op behavior. Keep broader dynamic trace/session metadata under #63901.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate user payload injection to supported APIs:** `src/agents/pi-embedded-runner/extra-params.ts:329-348`
  - body: `createUserFieldWrapper` always calls `streamWithPayloadPatch`, and `applyPostPluginStreamWrappers` installs it whenever `effectiveExtraParams.user` is present. A global `agents.defaults.params.user` would therefore add a top-level `user` field to Anthropic, Google, and other non-OpenAI payloads that do not define it, which can make those requests fail. Add an API-family guard before patching the payload.
  - confidence: 0.95
