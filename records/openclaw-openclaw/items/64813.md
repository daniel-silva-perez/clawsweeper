## Summary

Keep open. Current main still lacks direct `params.user` payload passthrough, and this branch remains useful source work, but it is unsafe as-is because the proposed wrapper would mutate every provider payload without the supported-API guard used by neighboring payload patches. The broader #63901 request is still open and does not supersede this focused PR yet.

## What This Changes

The PR adds a `user` extra-param wrapper in the Pi embedded runner so configured agent/model params are injected into provider request payloads.

## Best Possible Solution

Land a narrow, guarded `params.user` payload passthrough only for provider API families that actually support a top-level `user` request field, with regression tests for supported injection and unsupported-provider no-op behavior. Keep the dynamic trace/session metadata design under #63901 rather than expanding this PR into a broader metadata system.
