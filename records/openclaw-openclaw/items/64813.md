## Summary

Keep open. Current main still does not implement direct `params.user` passthrough, and the PR targets the right area, but the branch is not safe to merge because it injects the configured field into every provider payload instead of only API families that support that top-level request field.

## What This Changes

The PR adds a Pi embedded runner wrapper that normalizes `ctx.effectiveExtraParams.user` and writes it into outgoing provider payloads via `streamWithPayloadPatch`.

## Best Possible Solution

Land a narrow static `params.user` passthrough that only mutates payloads for supported OpenAI/OpenRouter-compatible API families, with regression tests for supported injection and unsupported-provider no-op behavior. Keep dynamic trace/session metadata design in #63901.

## Reproduction Assessment

Yes. A focused unit case can configure `agents.defaults.params.user`, call the PR wrapper with an `anthropic-messages` or `google-generative-ai` model, and observe the added top-level `user` payload key; I did not execute tests because this review is read-only.

## Solution Assessment

No. The requested capability is reasonable, but the current implementation is not the best fix because it applies the field globally; the maintainable path is a provider/API-family guard plus tests modeled on the existing `parallel_tool_calls` coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate user payload injection by supported API:** `src/agents/pi-embedded-runner/extra-params.ts:329-348`
  - body: With a global `agents.defaults.params.user`, this wrapper patches every provider payload. Non-OpenAI-family routes such as `anthropic-messages` and native Gemini payloads will receive an unexpected top-level `user` key, which can leak an attribution identifier and cause provider request rejection. Return the underlying stream unchanged unless the model API supports this field.

## Security Review

Status: needs_attention
