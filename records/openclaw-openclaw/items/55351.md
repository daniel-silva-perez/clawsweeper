## Summary

Keep open. The underlying Open WebUI UUID session split is still valid on current main, but this PR should not merge as-is because it mixes the session fix with unrelated credentialed Codex proxy/auth/routing work and still has concrete session-alias regressions.

## What This Changes

The PR preserves Open WebUI UUID-backed session keys across gateway store resolution and Control UI URLs, threads agent context into transcript lookup, adds regression tests, and also adds Codex proxy transport, compaction-auth, model-auth, and outbound routing changes.

## Best Possible Solution

Land a narrow Open WebUI UUID session fix that preserves bare UUIDs without losing agent ownership, normalizes legacy aliases for any default agent, resolves configured main aliases before UUID preservation, and keeps history/live subscriptions on one canonical identity; split Codex proxy/auth/routing changes into separate owner-reviewed PRs.

## Reproduction Assessment

Yes. The related bug has concrete Open WebUI-to-Control-UI reproduction steps, and current main source inspection shows a bare UUID-like key is still routed through default-agent canonicalization while Control UI URL handling leaves session aliases unchanged.

## Solution Assessment

No. The narrow UUID preservation direction is appropriate, but this branch is not the best solution because it combines unrelated security-sensitive runtime changes with the session fix and introduces alias/canonicalization regressions; a smaller targeted PR is safer.

## Review Findings

Full review comments:

- **[P1] Preserve agent scope for UUID keys in combined stores:** `src/gateway/session-utils.ts:643-644`
  - body: Returning bare UUIDs from canonicalizeSessionKeyForAgent also runs while per-agent stores are merged for gateway listings. A UUID row from an ops store can be listed as an unscoped key, so later get/patch/delete resolves through the current default agent store and can miss or mutate the wrong history.
  - confidence: 0.87
- **[P2] Normalize UUID aliases for non-main default agents:** `ui/src/ui/app-settings.ts:51-52`
  - body: normalizeChatUrlSessionKey strips only agent:main:<uuid>, but the old canonicalizer used whichever agent was default. Installs with ops or another default keep agent:<id>:<uuid> in URL/settings and can continue the split identity this PR is meant to converge.
  - confidence: 0.86
- **[P2] Resolve main-key aliases before UUID preservation:** `src/gateway/session-utils.ts:668-670`
  - body: The UUID fast path runs before the existing session.mainKey alias handling. If an operator configured a UUID-shaped main key, this now returns the bare UUID instead of the canonical main session key, splitting clients that use the configured main alias.
  - confidence: 0.8
- **[P2] Canonicalize UUID aliases for message subscriptions:** `src/gateway/session-utils.ts:668-670`
  - body: Bare UUID and agent:<id>:<uuid> forms remain different canonical keys, while session-message subscriptions and transcript fan-out use exact string keys. A legacy client subscribed on the prefixed form can miss live messages emitted under the bare UUID form.
  - confidence: 0.78
- **[P2] Send the system prompt only once:** `src/agents/pi-embedded-runner/codex-proxy-transport.ts:722-723`
  - body: buildRequestBody sets instructions from context.systemPrompt, while convertResponsesMessages also injects that prompt into input messages. That duplicates system instructions on every Codex proxy call and can change behavior or consume context unnecessarily.
  - confidence: 0.88
