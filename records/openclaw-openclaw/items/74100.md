## Summary

Keep open. Current main has bounded metadata-only skill prompting and on-demand reading, but it still lacks the requested skill dependency contract, dependency-depth expansion, graph/DOT CLI, and `skills.maxDepth` configuration surface; the feature needs maintainer product/API design rather than cleanup closure.

## What This Changes

Request a skill dependency graph that can parse skill-to-skill references, expand only needed dependency layers at runtime, expose DOT graph output, and configure maximum expansion depth to reduce token usage.

## Best Possible Solution

Design a backward-compatible skill dependency metadata contract, then add a graph builder that respects existing skill eligibility, allowlists, visibility policy, and prompt budgets; expose optional DOT output and runtime depth semantics without confusing skill dependencies with `agents.defaults.subagents.maxSpawnDepth`.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of current main: source, docs, and tests show metadata-only on-demand skill loading, not dependency graph traversal.

## Solution Assessment

Unclear as a concrete implementation because no patch exists and the requested behavior changes public skill metadata, runtime prompt semantics, CLI, config schema, and docs. The safest direction is a maintainer-approved dependency contract plus narrow graph/runtime expansion rather than parsing ad hoc `sessions_spawn` mentions as the only source of truth.
