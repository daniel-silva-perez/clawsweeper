## Summary

Keep this issue open. Current main has improved TUI activity handling for user-originated and chat-delta-visible runs, but the reported system-injected start gap still exists: terminal TUI only adopts an unknown run from matching `chat` events, while unknown `agent` lifecycle/tool events are dropped before they can mark the status line as running. The issue discussion’s diagnosis is still directionally correct: the TUI needs a server-side run/lifecycle subscription path for non-local triggers such as bridge/webhook/cron work.

## Best Possible Solution

Implement a targeted TUI run-start path: either have the terminal TUI subscribe to session-scoped lifecycle/run-state events, or allow `handleAgentEvent` to adopt a session-matching lifecycle `start` when no active run is tracked. Add a regression test for an unknown `agent` lifecycle start with the current session key setting `activeChatRunId` and `running`, then clearing on terminal lifecycle/chat events without affecting concurrent local or BTW runs.

## Evidence

That does not cover a lifecycle start before the first assistant delta.
