## Summary

Keep this issue open. Current main still only adopts unknown runs from session-matching chat events; an unknown session-matching agent lifecycle start is rejected before it can set the TUI status to running. Gateway agent events already carry sessionKey and lifecycle start is also mirrored as sessions.changed, but the terminal TUI neither pre-registers those agent events nor subscribes to the session lifecycle stream. Open PR #66400 is a relevant unmerged implementation candidate, not proof that main is fixed.

## Best Possible Solution

Keep this issue open and either review/land #66400 or implement an equivalent targeted TUI fix: accept sessionKey-matching agent lifecycle/tool events before the known-run gate, set activeChatRunId and running on lifecycle start, and add regression coverage for externally started runs while preserving cross-session and /btw isolation.
