## Summary

Keep open. Current main still lacks the requested schema-driven tool security classification contract; existing allow/deny policy, owner-only filtering, plugin approval hooks, display metadata, provider payload parallelism, and mutation heuristics are adjacent controls but do not provide the proposed public per-tool metadata surface or consumers.

## What This Changes

Add optional per-tool security classification metadata for read-only, destructive, concurrency, interrupt, open-world, and classifier-input behavior, then use it for safety prompts, scheduling, abort handling, and conformance checks.

## Best Possible Solution

Define a backwards-compatible public tool-security metadata contract, align concurrency semantics with upstream AgentTool executionMode, preserve metadata through core tools, plugin registration, adapters, inventories, docs, and conformance snapshots, then wire consumers behind conservative fallbacks that keep existing approval, policy, sandbox, and redaction controls in force.
