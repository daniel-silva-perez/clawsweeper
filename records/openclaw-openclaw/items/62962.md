## Summary

Keep open. Current main still lacks the requested schema-driven tool security classification contract; it has adjacent controls such as allow/deny policy, owner-only filtering, plugin approval hooks, tool display metadata, provider-level parallel_tool_calls configuration, and an argument-sensitive mutation heuristic, but no public per-tool metadata surface for the six requested properties or consumers that use them.

## What This Changes

Add optional tool security classification metadata to tool definitions and use it for approval prompts, parallel scheduling, interrupt behavior, open-world policy, classifier input, and conformance checks.

## Best Possible Solution

Define a backwards-compatible public tool-security metadata contract, align concurrency semantics with upstream AgentTool executionMode, preserve the metadata through core tools, plugin registration, adapters, inventories, docs, and conformance snapshots, then wire consumers behind conservative fallbacks that keep existing approval, policy, sandbox, and redaction controls in force.
