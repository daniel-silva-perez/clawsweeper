## Summary

Keep open. Current main still lacks the requested six-property per-tool security classification contract and consumers; the newer trusted policy and plugin tool metadata seams are adjacent, but they only cover runtime policy hooks and display/risk catalog metadata. The related RFC and draft SDK follow-up explicitly do not close this broader tool capability/provenance proposal.

## What This Changes

Add optional per-tool security classification metadata for read-only, destructive, concurrency, interrupt, open-world, and classifier-input behavior, then use it for safety prompts, scheduling, abort handling, and conformance checks.

## Best Possible Solution

Define a backwards-compatible public tool-security metadata contract, preserve it through core tools, plugin registration, adapters, inventories, catalog/docs, and conformance snapshots, then wire conservative consumers that do not bypass existing allow/deny policy, owner-only filtering, sandboxing, host approvals, or plugin approval hooks.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The current-main verification is high-confidence: exact field searches are empty and the central tool, adapter, policy, conformance, plugin metadata, and catalog paths do not expose the requested contract.

## Solution Assessment

No current or linked implementation is the complete solution. The adjacent trusted-policy and display/risk metadata seams are useful foundations, but the broader per-tool classification contract still needs maintainer design before a narrow fix PR is safe.
