## Summary

Keep this PR open. Current main still lacks Telegram custom emoji reaction support, and the PR contains useful source work, but it is not merge-ready because it targets obsolete files and the proposed action path still validates the standard emoji field before the new custom emoji id can be used.

## What This Changes

The PR adds Telegram custom emoji reaction handling for inbound message_reaction updates, outbound reactMessageTelegram payloads, Telegram react action plumbing, and bot/send tests.

## Best Possible Solution

Port the narrow reaction feature into the current Telegram plugin surfaces: add customEmojiId to the shared message-tool schema and parser, allow either emoji or customEmojiId for add calls, send Telegram custom_emoji reaction payloads from reactMessageTelegram, emit inbound custom emoji reaction system events after the existing authorization and notification gates, and update focused tests plus user-facing docs.

## Reproduction Assessment

Yes. The blocking PR bug is reproducible by calling the proposed Telegram react action with chatId, messageId, and customEmojiId but no emoji; readReactionParams still requires the emoji key before the PR reads customEmojiId. Current main verification also shows no customEmojiId schema, send option, or inbound custom_emoji handling.

## Solution Assessment

No. The feature direction is valid, but this patch is not the best merge path because it targets obsolete current-main files and leaves the custom-only action path blocked by validation; a small port to the current runtime files and shared message-tool contract is safer.
