## Summary

Keep this PR open. The feature is valid and current main still lacks Telegram custom emoji reaction support, but the proposed branch is not merge-ready because the action path still validates `emoji` before accepting `customEmojiId`, and the branch targets older Telegram files that have since moved on main.

## What This Changes

The branch adds Telegram custom emoji reaction handling to inbound `message_reaction` updates, outbound `reactMessageTelegram`, Telegram `react` action plumbing, and bot/send tests.

## Best Possible Solution

Port the narrow reaction feature into the current Telegram plugin surfaces: add `customEmojiId` to the shared message tool schema and parser, allow either `emoji` or `customEmojiId` for add calls, send Telegram `{ type: "custom_emoji", custom_emoji_id }` payloads from `reactMessageTelegram`, emit inbound custom emoji reaction system events after the existing authorization and notification gates, and update focused tests plus user-facing docs.

## Reproduction Assessment

Yes. The blocking PR bug can be reproduced by calling the Telegram `react` action with `chatId`, `messageId`, and `customEmojiId` but no `emoji`; `readReactionParams` requires `emoji` before the proposed code reads the custom emoji id.

## Solution Assessment

No. The proposed direction is useful, but the current patch is not the best merge path because it targets obsolete files and leaves the custom-only action path blocked by validation; a small port to the current runtime files and shared message tool contract is safer.
