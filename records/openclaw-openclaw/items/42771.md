## Summary

Keep open. Current main has not implemented the PR's intended Docker behavior: Playwright browsers still install under `/home/node/.cache/ms-playwright`, there is no default `/opt/playwright` path, and the docs still describe the hidden cache path. The PR is also paired with the same-author open bug #42819 using closing syntax, so the right path is maintainer review/merge or explicit rejection rather than cleanup closure.

## Best Possible Solution

Keep this PR open for contributor and maintainer follow-up. A complete fix should rebase on current main, set Docker's Playwright browser path to a stable non-hidden location, create it with node ownership for default runtime installs, recursively chown after optional preinstall, remove apt-list deletion from the BuildKit cache mount, create the Chromium symlink only after a successful install, update the current Docker/browser-control docs, and add focused Dockerfile assertions for path, ownership, symlink, and cache behavior.

## Risks / Open Questions

- Merging the PR as-is can leave `/opt/playwright` root-owned in default builds where `OPENCLAW_INSTALL_BROWSER` is unset, causing runtime browser installs as `USER node` to fail with `EACCES`.
- Merging the PR as-is can degrade Docker rebuild caching by deleting apt metadata inside a named BuildKit cache mount.
- The docs patch needs a rebase: current main's Docker Playwright guidance is in `docs/install/docker.md` and `docs/tools/browser-control.md`, while the PR edits `docs/tools/browser.md`.
- The Dockerfile change affects a runtime command path through `/usr/local/bin/chromium`; no new supply-chain source was observed, but the symlink and install ownership behavior should be corrected before merge.
