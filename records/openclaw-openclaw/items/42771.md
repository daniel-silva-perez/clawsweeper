## Summary

Keep this PR open. Current main still installs Docker Playwright browsers under `/home/node/.cache/ms-playwright`, does not set a default `/opt/playwright` browser path, and does not expose a stable `chromium` symlink. The same-author bug #42819 remains open and this PR uses closing syntax for it, so the right maintainer path is to review, update, merge, or explicitly close the implementation candidate rather than auto-close it. The patch is plausible in-repo Docker/docs work, but it still needs Dockerfile corrections and a docs rebase before merge.

## Best Possible Solution

Keep this PR open for contributor and maintainer follow-up. A complete fix should rebase on current main, set the Docker Playwright browser path to a stable non-hidden location, create it with node ownership before runtime installs, recursively chown after optional browser preinstall, remove apt-list deletion inside the BuildKit cache mount, ensure the Chromium symlink is created only after a successful install, update `docs/install/docker.md` and `docs/tools/browser-control.md`, and add focused Dockerfile assertions for path, ownership, symlink, and cache behavior.

## Risks / Open Questions

- Merging the PR as-is can leave `/opt/playwright` unwritable by `USER node` in default builds where `OPENCLAW_INSTALL_BROWSER` is unset.
- Merging the PR as-is can degrade Docker rebuild caching by deleting apt metadata inside a named BuildKit cache mount.
- The PR edits `docs/tools/browser.md`, but current main’s Docker Playwright install guidance is in `docs/tools/browser-control.md`, so the docs patch needs a rebase/refresh.
- The Dockerfile change touches image build/runtime command paths; security review found no new secrets, workflow permissions, package sources, or release hooks, but the symlink and install failure behavior need correction before merge.
