## Summary

Keep open. Current main mitigates the original default 600s reproduction by using a 48-hour agent runtime timeout, but the core bug remains: non-compaction run timeouts are still treated as assistant timeout failover signals, and there is no tool-execution timeout cause wired into the embedded runner or failover policy.

## Best Possible Solution

Add phase-aware timeout state in the embedded runner, preferably a timeoutCause enum or active-tool timeout flag derived from the existing tool item lifecycle. When the global run timer fires during active tool execution, surface a run/tool timeout or partial-progress result without rotating auth profiles or falling back models. Keep model idle/request timeouts on the existing failover path, and add focused tests for resolveRunFailoverDecision plus an embedded attempt where a timeout fires with an active tool item.
