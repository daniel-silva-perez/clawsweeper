## Summary

Keep this PR open. Current main still does not implement the proposed `before_identity_resolve` hook or an equivalent dynamic direct-message peer identity seam. The PR remains a plausible implementation candidate, but the review discussion raises real follow-up work around secondary routing paths, blank hook results, hook short-circuiting, and session-isolation semantics.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused additive core/plugin contract: add a typed identity-resolution hook, expose a narrow SDK helper, thread an optional resolved peer id through route and session-key construction, integrate all direct-message and secondary routing paths for Slack, MS Teams, and WhatsApp, and add contract/channel tests for normalization, blank results, identityLinks fallback, multi-plugin priority or short-circuit behavior, and session-isolation safety.

## Risks / Open Questions

- Merging the PR unchanged could split direct-message history in secondary Slack system-event and WhatsApp broadcast routing paths that still rebuild session keys from raw sender ids.
- A blank or whitespace `canonicalPeerId` from a higher-priority hook could suppress lower-priority valid hook results unless normalized before first-result-wins precedence.
- Running all registered identity hooks after a winning result can add avoidable latency and side effects if plugins call external identity services.
- The feature changes session-isolation semantics: a buggy or compromised identity plugin could collapse distinct users into the same session, so the contract and tests need to make that risk explicit.
