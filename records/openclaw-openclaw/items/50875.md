## Summary

Keep this PR open. Current main at c0ec58f4b6906ad06fd4c6f265d43be053b2d193 still does not implement `before_identity_resolve`, `resolveInboundPeerIdentity`, a `canonicalPeerId` hook result, or any equivalent dynamic direct-message identity seam. The PR remains a plausible implementation candidate, but the review discussion identifies real follow-up work around secondary routing paths, blank hook results, hook short-circuiting, and session-isolation semantics.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused additive core/plugin contract: add a typed identity-resolution hook, expose a narrow SDK helper, thread an optional resolved peer id through route and session-key construction, integrate all direct-message and secondary routing paths for Slack, MS Teams, and WhatsApp, and add contract/channel tests for normalization, blank results, identityLinks fallback, multi-plugin priority or short-circuit behavior, and session-isolation safety.

## Risks / Open Questions

- Merging the PR unchanged may split history for secondary Slack system-event paths and WhatsApp broadcast paths that still rebuild session keys from raw sender ids.
- A blank or whitespace `canonicalPeerId` from a higher-priority hook could suppress a lower-priority valid result unless normalized before first-result-wins precedence.
- Running every registered identity hook after a winning result can add latency and side effects if plugins call external identity services.
- The feature changes session-isolation semantics: a buggy or compromised identity plugin could collapse distinct users into the same direct-message session, so the contract and tests need to make that explicit.
