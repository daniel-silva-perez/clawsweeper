## Summary

Keep this PR open. Current main does not implement the proposed `before_identity_resolve` hook or an equivalent dynamic peer-identity seam: the typed hook registry and handler map omit it, the hook runner exposes no identity-resolution phase, the routing SDK exports no helper, and route/session-key construction still depends on static `session.identityLinks`. The PR remains a plausible implementation candidate, but it still needs maintainer review because the discussion identifies real session-correctness follow-ups around blank canonical IDs, secondary Slack/WhatsApp routing paths, and multi-plugin execution behavior.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused additive plugin contract: add a typed identity-resolution hook, expose a narrow SDK helper, thread a `resolvedPeerId` override through route/session-key construction, integrate all relevant direct-message and secondary routing paths for Slack, MS Teams, and WhatsApp, and add contract/channel tests for normalization, blank results, identityLinks fallback, multi-plugin priority, and session-isolation behavior.

## Risks / Open Questions

- Merging unchanged could split direct-message history when secondary Slack system-event paths or WhatsApp broadcast paths rebuild session keys from raw sender IDs instead of the plugin-resolved canonical peer ID.
- A blank or whitespace `canonicalPeerId` from a higher-priority hook could suppress a lower-priority valid result unless the merge semantics normalize empty values before first-result-wins precedence.
- Executing every registered identity hook after the first result wins can add avoidable latency and side effects if plugins call external identity systems.
- The feature changes session isolation semantics: plugin-provided canonical peer IDs need a clear contract because a bad or compromised plugin can intentionally or accidentally collapse distinct users into one session.
