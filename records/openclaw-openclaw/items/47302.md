## Summary

Keep open. Current main still lacks `doctor --check-config`, and this PR is the active implementation attempt, but the latest PR head still has blocking correctness issues, including a non-exported TTS import and diagnostic checks that can either mutate user state or report success for broken config paths.

## What This Changes

The PR adds a `doctor --check-config` flag, a new deep config validation module for schema/plugin/model/catalog/fallback/TTS/channel checks, and formatter-focused tests for the new output.

## Best Possible Solution

Land or replace this with a rebased doctor deep-config mode that uses registry-backed TTS/model APIs, keeps validation read-only, treats active unresolved credential inputs as failures when they break enabled subsystems, validates fallback refs against runtime catalog/provider availability, documents the CLI surface, and adds behavior tests for subsystem checks and non-zero exits.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: PR head imports `TTS_PROVIDERS` from current `src/tts/tts.ts`, and current main does not export that symbol. The false-success cases are also traceable from PR head logic and current env substitution, TTS provider, model catalog, and model-selection helpers.

## Solution Assessment

No. The requested direction is useful for the linked silent config-degradation reports, but this implementation is not the narrowest maintainable solution until it is updated to current registry APIs, read-only catalog loading, and runtime-equivalent failure semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use the registry-backed TTS provider list:** `src/commands/doctor-check-config.ts:18`
  - body: `TTS_PROVIDERS` is not exported by current `../tts/tts.js`, so this new module will fail typecheck/build as soon as it is included. Use current registry-backed APIs such as `resolveTtsProviderOrder(configuredProvider, cfg)` or `listSpeechProviders(cfg)` instead of importing the old static list.
  - confidence: 0.98
- **[P2] Fail unresolved active config warnings:** `src/commands/doctor-check-config.ts:68-73`
  - body: Snapshot warnings are emitted as `warn`, and `hasFailures` ignores warnings. With an unset `${OPENAI_API_KEY}` in an enabled provider config, env substitution preserves the non-empty placeholder, so this command can still report success and exit 0 while the runtime credential is unusable.
  - confidence: 0.9
- **[P2] Keep catalog checks read-only:** `src/commands/doctor-check-config.ts:183`
  - body: Calling `loadModelCatalog({ config: cfg })` uses the default non-read-only path, which can call `ensureOpenClawModelsJson` and create or rewrite model catalog state. A doctor validation mode intended for CI gating should pass `readOnly: true` so checking config does not mutate user state.
  - confidence: 0.88
- **[P2] Reject unparseable qualified primary refs:** `src/commands/doctor-check-config.ts:152-154`
  - body: When `agents.defaults.model` contains a slash but cannot be parsed, such as `anthropic/`, `parseModelRef` returns null and this branch falls through to reporting the resolved fallback default as healthy. That misses a broken explicitly configured model ref.
  - confidence: 0.86
- **[P2] Validate fallback refs against availability:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` only proves the fallback string can be parsed. A fallback like `missing-provider/model` can still be counted as validated even though it is not present in the runtime catalog/provider set, leaving the requested fallback validation gap uncaught.
  - confidence: 0.9
