## Summary

Keep open. Current main still lacks the requested doctor config-validation flag, and the PR branch is useful but not mergeable as submitted because it uses a stale TTS export and misses several runtime-equivalent validation cases.

## What This Changes

The PR adds `openclaw doctor --check-config`, a new `doctor-check-config` module, and formatter tests for schema/plugin/model/catalog/fallback/TTS/channel diagnostics.

## Best Possible Solution

Land a rebased deep-config doctor mode that uses current registry-backed TTS APIs, keeps catalog/config inspection read-only, treats active unresolved config inputs as failures when they break enabled subsystems, validates primary and fallback model refs against runtime-equivalent availability, documents the new CLI flag, and adds behavior tests beyond output formatting.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: PR head imports `TTS_PROVIDERS` from `../tts/tts.js`, while current main does not export that symbol. Additional false-success paths are traceable from current env-substitution warnings, model resolution fallback behavior, and catalog-loading semantics.

## Solution Assessment

No. The requested direction is useful for #45552 and #20909, but this implementation is not the best current fix until it uses the current TTS/provider contracts, avoids write-capable catalog loading, and validates the same failure modes the runtime will hit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use the registry-backed TTS provider list:** `src/commands/doctor-check-config.ts:18`
  - body: `TTS_PROVIDERS` is not exported by current `../tts/tts.js`, so this new module fails typecheck/build when included. Use the registry-backed helpers already exported by the TTS facade, such as `resolveTtsProviderOrder`, or the speech provider registry instead of the removed static list.
  - confidence: 0.98
- **[P2] Fail active unresolved config inputs:** `src/commands/doctor-check-config.ts:72`
  - body: Snapshot warnings are emitted as `warn`, and `hasFailures` only considers `fail`. With an active provider key left as an unresolved `${ENV_VAR}` placeholder, later subsystem checks can still treat the non-empty placeholder as configured and exit 0, missing the broken config this mode is meant to catch.
  - confidence: 0.9
- **[P2] Keep catalog validation read-only:** `src/commands/doctor-check-config.ts:183`
  - body: `loadModelCatalog({ config: cfg })` uses the default writable path, which can call `ensureOpenClawModelsJson` and create or rewrite catalog state. A doctor validation mode intended for CI gating should pass `readOnly: true` so checking config does not mutate user state.
  - confidence: 0.9
- **[P2] Reject unparseable primary refs:** `src/commands/doctor-check-config.ts:152-154`
  - body: When `agents.defaults.model` is a malformed qualified ref such as `anthropic/`, `parseModelRef` returns null and this branch falls through to reporting the resolved fallback default as healthy. That misses a broken explicitly configured primary model.
  - confidence: 0.86
- **[P2] Validate fallbacks against availability:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` only proves the fallback string can be parsed. A fallback like `missing-provider/model` can still be counted as validated even though it is not present in the runtime catalog/provider set, leaving the requested fallback-validation gap uncaught.
  - confidence: 0.88
