## Summary

Keep open. Current main still does not provide `openclaw doctor --check-config`, and the linked deep config-validation problem is a valid CLI/doctor feature request, but this PR head is not mergeable as-is because it imports a stale TTS export and its checks can still report success while missing runtime-relevant config failures or mutating model catalog state.

## What This Changes

The PR adds `openclaw doctor --check-config`, a new deep config validation module for schema/plugin/model/catalog/fallback/TTS/channel diagnostics, and formatter tests for the new output.

## Best Possible Solution

Land a rebased doctor deep-config mode that uses current registry-backed TTS/model APIs, keeps validation read-only, fails active unresolved credential/config refs when they break enabled subsystems, validates primary and fallback refs against runtime-equivalent provider/catalog availability, documents the CLI surface, and adds behavior tests for subsystem checks and non-zero exits.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: PR head imports `TTS_PROVIDERS` from `../tts/tts.js`, while current main does not export that symbol; the false-success cases are also traceable through current env-substitution, model-resolution, and catalog-loading code paths.

## Solution Assessment

No. The requested direction is useful for #45552 and #20909, but this implementation is not yet the narrowest maintainable fix because it uses stale TTS APIs, performs a non-read-only catalog load, and still misses runtime-equivalent validation failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use current speech provider APIs:** `src/commands/doctor-check-config.ts:18`
  - body: `TTS_PROVIDERS` is not exported by current `../tts/tts.js`, so this new module will fail typecheck/build as soon as it is included. Use the registry-backed helpers already exported by the TTS facade, such as `resolveTtsProviderOrder` or `listSpeechProviders`, instead of the old static list.
  - confidence: 0.98
- **[P2] Fail unresolved active config refs:** `src/commands/doctor-check-config.ts:68-73`
  - body: Snapshot warnings are emitted as `warn`, and `hasFailures` only considers `fail`. With an enabled provider configured as an unset `${OPENAI_API_KEY}`, env substitution preserves a non-empty placeholder, so this command can still exit 0 while the runtime credential is unusable.
  - confidence: 0.9
- **[P2] Keep catalog validation read-only:** `src/commands/doctor-check-config.ts:183`
  - body: Calling `loadModelCatalog({ config: cfg })` uses the default writable path, which can run `ensureOpenClawModelsJson` and create or rewrite model catalog state. A doctor validation mode intended for CI gating should pass `readOnly: true` so checking config does not mutate user state.
  - confidence: 0.9
- **[P2] Reject unparseable qualified primary refs:** `src/commands/doctor-check-config.ts:152-154`
  - body: When `agents.defaults.model` contains a slash but cannot be parsed, such as `anthropic/`, `parseModelRef` returns null and this branch falls through to reporting the resolved fallback default as healthy. That misses a broken explicitly configured model ref.
  - confidence: 0.86
- **[P2] Validate fallbacks against availability:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` only proves the fallback string can be parsed. A fallback like `missing-provider/model` can still be counted as validated even though it is not present in the runtime catalog/provider set, leaving the requested fallback validation gap uncaught.
  - confidence: 0.88
