## Summary

Keep open. The requested deep config validation command is still useful and not implemented on current main, but this PR is not merge-ready because the branch has drifted from the current TTS provider API and still misses several of the configuration failure modes it is meant to catch.

## What This Changes

The PR adds a `doctor --check-config` option, a new deep config validation module for schema/plugin/model/fallback/TTS/channel checks, and formatter-focused tests for the new result output.

## Best Possible Solution

Land or replace this with a core doctor deep config validation mode that is rebased onto the current plugin/provider registry APIs, fails active unresolved configuration inputs, validates fallback models against the same catalog/provider contract users rely on at runtime, documents the new CLI surface, and adds behavior tests for subsystem checks and non-zero exits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current TTS provider registry:** `src/commands/doctor-check-config.ts:18`
  - body: `TTS_PROVIDERS` is no longer exported by current main's `../tts/tts.js`, so this new module will fail typecheck/build when merged. Use the registry-backed API, such as `resolveTtsProviderOrder(configuredProvider, cfg)` or `listSpeechProviders`, instead of importing a static provider list.
  - confidence: 0.98
- **[P2] Validate fallback models beyond parsing:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` succeeds for any syntactically valid `provider/model`, so a fallback like `missing-provider/model` is counted as validated even when it is not in the catalog or configured providers. That leaves the #20909 failure mode uncaught; check each resolved fallback against the runtime catalog/provider status before reporting success.
  - confidence: 0.92
- **[P2] Fail active unresolved env-backed config:** `src/commands/doctor-check-config.ts:72`
  - body: Missing env substitutions are added as `warn`, and `hasFailures` ignores warnings. With an active TTS API key like `${OPENAI_API_KEY}` unset, the placeholder remains a non-empty string, the provider can look configured, and the command exits 0 instead of catching the broken runtime setup.
  - confidence: 0.89
- **[P2] Fail invalid qualified primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: When the configured primary model contains `/` but cannot be parsed, this branch only warns if `parseModelRef` returns a mismatched ref. A value such as `anthropic/` returns `null`, falls through, and reports the fallback default as a healthy primary model, missing a broken configured model ref.
  - confidence: 0.84
