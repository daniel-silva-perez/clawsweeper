## Summary

Keep open. Current main still lacks the requested doctor flag, so the contribution is not obsolete, but the PR is not mergeable as submitted because it still has blocking static and diagnostic correctness issues against current main.

## What This Changes

The PR adds an `openclaw doctor --check-config` option, a new deep config validation module for schema/plugin/model/fallback/TTS/channel checks, and formatter-focused tests for the new result output.

## Best Possible Solution

Land or replace this with a rebased doctor deep-config mode that uses current registry-backed TTS/model APIs, treats active unresolved env/SecretRef-backed values as failing when they would break an enabled subsystem, validates fallback refs against runtime catalog/provider availability, documents the CLI surface, and adds behavior tests for subsystem checks and non-zero exits.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: PR head imports `TTS_PROVIDERS` from `../tts/tts.js`, and current main does not export that symbol, so typecheck/build would fail after merge. The remaining false-pass cases are traceable from the branch logic and current runtime helpers without mutating the checkout.

## Solution Assessment

No. The requested direction is appropriate for the linked config-degradation reports, but this implementation should be updated to current registry APIs and should validate/fail the same runtime conditions it claims to diagnose before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current TTS provider registry:** `src/commands/doctor-check-config.ts:18`
  - body: `TTS_PROVIDERS` is not exported by current `../tts/tts.js`, so this new module fails typecheck/build when merged. Use registry-backed APIs such as `resolveTtsProviderOrder(configuredProvider, cfg)` or `listSpeechProviders(cfg)` instead of importing a static provider list.
  - confidence: 0.98
- **[P2] Fail active unresolved env-backed config:** `src/commands/doctor-check-config.ts:72`
  - body: Snapshot warnings are emitted as `warn`, and `hasFailures` only considers `fail`. With an active value like an unset `${OPENAI_API_KEY}`, the command can still exit 0 even though the enabled subsystem will run with missing credentials.
  - confidence: 0.9
- **[P2] Validate fallback refs against runtime availability:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` only proves the fallback string can be parsed. A fallback like `missing-provider/model` will still be counted as validated even though it is not usable at runtime, leaving the requested fallback validation gap uncaught.
  - confidence: 0.92
- **[P2] Reject unparseable qualified primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: When `agents.defaults.model` contains a slash but cannot be parsed, such as `anthropic/`, `parseModelRef` returns null and this branch falls through to reporting the resolved fallback default as a healthy primary model. That misses a broken explicitly configured model ref.
  - confidence: 0.84
