## Summary

Keep open. The requested deep config validation is still absent from current main, and this PR is a useful active implementation attempt, but the latest PR head is not mergeable as-is because it imports a removed TTS export and still misses several runtime-fidelity validation cases.

## What This Changes

The PR adds an `openclaw doctor --check-config` option, a new deep config validation module for schema/plugin/model/fallback/TTS/channel checks, and formatter-focused tests for the new result output.

## Best Possible Solution

Land or replace this with a rebased core doctor deep-config mode that uses current TTS registry APIs, classifies active unresolved secret/env-backed config as failing when it would break an enabled subsystem, validates fallback refs against the same model catalog/provider status used at runtime, documents the CLI surface, and adds behavior tests for subsystem checks plus non-zero exits.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: PR head imports `TTS_PROVIDERS`, while current main's `src/tts/tts.ts` does not export that symbol, so typecheck/build would fail after merge. The remaining diagnostic false-pass cases are also traceable from the branch logic and current runtime helpers without needing to mutate the checkout.

## Solution Assessment

No. The requested direction is appropriate for the linked config-degradation reports, but this implementation should be updated to current registry-backed TTS/model APIs and should test the subsystem behavior it claims to validate before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the current TTS provider registry:** `src/commands/doctor-check-config.ts:18`
  - body: `TTS_PROVIDERS` is no longer exported by current `../tts/tts.js`, so this new module fails typecheck/build when merged. Use registry-backed APIs such as `resolveTtsProviderOrder(configuredProvider, cfg)` or `listSpeechProviders(cfg)` instead of importing a static provider list.
  - confidence: 0.98
- **[P2] Validate fallback refs against runtime availability:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` only proves the fallback string can be parsed. A fallback like `missing-provider/model` will still be counted as validated even though it is not usable at runtime, leaving the fallback validation request uncaught.
  - confidence: 0.92
- **[P2] Fail active unresolved env-backed config:** `src/commands/doctor-check-config.ts:72`
  - body: Missing env substitutions are emitted as warnings, and `hasFailures` ignores warnings. With an active API key like `${OPENAI_API_KEY}` unset, the placeholder can still look configured to the subsystem check and let the command exit 0 for a broken runtime setup.
  - confidence: 0.89
- **[P2] Reject unparseable qualified primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: When `agents.defaults.model` contains a slash but cannot be parsed, such as `anthropic/`, `parseModelRef` returns null and this branch falls through to reporting the resolved fallback default as a healthy primary model. That misses a broken explicitly configured model ref.
  - confidence: 0.84
