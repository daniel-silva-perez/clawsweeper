## Summary

Keep open. The report is still tied to open fix PRs with closing syntax, including a same-author PR, so the issue should not be closed before a canonical fix merges. Current main also does not show a clear catalog-level filter for provider entries inferred from OpenRouter-style slashed model IDs; the `/model status` path can still group raw catalog entries by `entry.provider` in the no-allowlist flow.

## What This Changes

The issue asks OpenClaw to stop `/model status` from showing native-provider groups inferred from provider-like prefixes inside OpenRouter model IDs.

## Best Possible Solution

Land or rework one canonical fix that filters only registry-inferred aggregator shadow entries at the catalog or directive boundary while preserving explicitly configured native providers, then close this issue through the merged PR.

## Reproduction Assessment

Yes. The issue includes concrete configuration steps, and the source path supports the failure mode by accepting provider/id catalog entries and grouping `/model status` by provider; current tests do not cover this exact OpenRouter phantom-provider case.

## Solution Assessment

Yes, a catalog/directive-level suppression of inferred aggregator shadows is the narrow maintainable direction. A rendering-only skip for missing-auth provider groups would be broader and could hide real configuration problems.
