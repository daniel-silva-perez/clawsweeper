## Summary

Keep open. Current main still has the reported failure shape: cross-provider fallback drops the original auth profile, later runs under the auto-selected fallback provider can resolve a `google-gemini-cli` auth profile, and CLI session reuse still invalidates when stored and current auth profile ids differ. The recent Gemini CLI auth-epoch work helps account-change detection but does not remove the auth-profile mismatch path described here.

## What This Changes

The issue requests that `google-gemini-cli` fallback runs keep a stable provider-appropriate auth profile identity so CLI sessions are reused instead of resetting with `reason=auth-profile`.

## Best Possible Solution

Keep the issue open and fix the auto-reply CLI fallback path so a Gemini CLI fallback run resolves, stores, and compares a stable provider-appropriate identity for session reuse while preserving cross-provider auth isolation. Add regression coverage for fallback-created bindings and stale bindings where the Gemini CLI auth profile is missing on one side.
