## Summary

Keep open. Current main still has the reported fallback/auth-profile shape: cross-provider fallback scopes the run auth profile to undefined, the auto-reply CLI path forwards that scoped value into `runCliAgent`, and CLI session reuse still invalidates when the stored and current auth profile ids differ. The issue body and follow-up comment include concrete logs and a stale-binding case, and I found no current-main implementation that resolves a provider-appropriate Gemini CLI auth profile before binding/reuse.

## What This Changes

The issue asks OpenClaw to preserve CLI session context when `google-gemini-cli` is reached as a fallback provider with an OAuth auth profile, instead of resetting every turn with `reason=auth-profile`.

## Best Possible Solution

Fix the auto-reply CLI fallback auth path so a `google-gemini-cli` fallback run resolves and stores a stable Gemini-appropriate auth profile identity for CLI session binding and reuse, while preserving cross-provider auth isolation. Add regression coverage for both fallback-created bindings and stale `cliSessionBindings` where the Gemini CLI auth profile identity is missing on one side.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path and logs, and current main still has a source-level path that can produce the same missing-versus-present auth-profile mismatch; I did not run a live OAuth fallback session during this read-only review.

## Solution Assessment

Unclear. The requested direction is narrow, but the exact fix should be chosen by a maintainer because treating missing auth profile ids as compatible could weaken profile isolation; resolving a provider-appropriate Gemini CLI auth profile before binding looks safer than reusing the primary provider profile.
