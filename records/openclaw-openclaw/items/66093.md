## Summary

Keep open. Current main still drops the run auth profile on cross-provider fallback before the CLI path, and CLI session reuse still invalidates missing-versus-present auth profile ids, so the reported Gemini CLI fallback reset remains plausible and unfixed. The issue discussion includes concrete logs and a stale binding follow-up, and I found no open PR that references this item as a fix.

## What This Changes

The issue asks OpenClaw to preserve CLI session context when `google-gemini-cli` is used as a fallback provider with OAuth auth profiles.

## Best Possible Solution

Keep this open and fix the auto-reply CLI fallback auth path so a Gemini CLI fallback run resolves and stores a stable provider-appropriate auth profile identity for session binding and reuse, while preserving cross-provider auth isolation. Add regression coverage for fallback-created bindings and stale bindings where Gemini CLI auth profile identity is missing on one side.
