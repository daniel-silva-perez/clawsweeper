## Summary

Keep open. Current main still drops the session auth profile on auto-reply cross-provider fallback and still invalidates CLI session reuse when the stored and current auth profile ids differ, so the reported `google-gemini-cli` fallback reset is not fixed. Existing coverage protects the #1156 cross-provider drop but does not cover resolving a provider-appropriate CLI fallback auth profile or stale Gemini CLI bindings.

## What This Changes

The issue requests preserving CLI session context when `google-gemini-cli` is reached as a fallback with OAuth auth profiles, instead of resetting every call due to auth-profile mismatch.

## Best Possible Solution

Keep the issue open and fix the auto-reply CLI fallback auth path so a `google-gemini-cli` fallback run uses stable, provider-appropriate auth profile identity for CLI session binding and reuse, while retaining cross-provider auth isolation. Add regression coverage for fallback-created and stale `cliSessionBindings` with missing/present Gemini CLI auth profile ids.
