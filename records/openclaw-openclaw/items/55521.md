## Summary

Keep open. The feature is not implemented on current main, but the proposed patch would create misleading Feishu typing reactions on messages that current policy intentionally ignores and would bypass the existing typingIndicator toggle.

## What This Changes

The PR adds an immediate Feishu `Typing` message reaction in `handleFeishuMessage()` after receipt logging and before routing or policy checks.

## Best Possible Solution

Settle the Feishu typing direction with the related native-typing tracker, then implement any immediate acknowledgement through the existing Feishu typing lifecycle after reply eligibility is known and behind the same typingIndicator configuration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move ack reaction after reply eligibility checks:** `extensions/feishu/src/bot.ts:327-336`
  - body: The new `Typing` reaction runs before the disabled-group, group allowlist, sender allowlist, and mention-required early returns later in `handleFeishuMessage()`. In mention-required or allowlisted groups, messages the bot is supposed to ignore would still get a visible processing reaction with no reply, creating noisy and misleading UX.
  - confidence: 0.93
- **[P2] Honor the typingIndicator setting:** `extensions/feishu/src/bot.ts:330-334`
  - body: This call unconditionally sends a `Typing` reaction, but the existing Feishu typing path skips typing when `account.config.typingIndicator` is false. Users who disabled typing indicators would still get these reaction calls, so the new path should share the same toggle semantics.
  - confidence: 0.88
