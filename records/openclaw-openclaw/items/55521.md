## Summary

Keep open. The Feishu UX idea is understandable, but the PR diff is not mergeable as written because it emits a persistent Typing reaction before reply eligibility gates, bypasses the existing Feishu typing lifecycle and config toggle, and overlaps an open product/API question about using Feishu's native typing surface.

## What This Changes

The PR adds an immediate Feishu `im.messageReaction.create` call in `handleFeishuMessage()` to attach a `Typing` emoji reaction right after an inbound message is logged, before routing and policy checks.

## Best Possible Solution

If immediate Feishu feedback is still desired, integrate it with the existing typing lifecycle after reply eligibility is known, respect `typingIndicator`, keep and delete the reaction id, and add focused tests for silent-drop gates and disabled typing while resolving the native-typing direction in #69572.

## Reproduction Assessment

Yes. Applying the PR diff to current main would call `messageReaction.create()` immediately after the inbound log, while current main later returns for blocked groups, blocked senders, unmentioned mention-required groups, unauthorized DMs, and empty no-media messages.

## Solution Assessment

No. A direct early reaction call is not the narrowest maintainable solution; the safer path is to reuse or extend the existing Feishu typing pipeline after eligibility checks and cleanup hooks.

## Review Findings

  - body: The new path unconditionally sends a Typing reaction, while the existing Feishu typing path skips reaction calls when `account.config.typingIndicator` is false. Users who disabled typing indicators would still get this API call and visible reaction.
  - confidence: 0.9
