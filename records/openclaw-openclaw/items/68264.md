## Summary

Keep open. Current main still treats `canvas present` and `canvas navigate` as node-backed control commands that return JSON success, while chat-visible visual confirmation remains a separate `canvas snapshot`, `browser snapshot`, or `browser screenshot` action. The prior ClawSweeper comments are still consistent with the current source and docs, so this needs maintainer product-contract triage rather than cleanup closure.

## What This Changes

The issue requests reliable chat-visible confirmation or rendering after canvas/browser navigation or presentation commands instead of node/validation errors or invisible state changes.

## Best Possible Solution

Define the intended chat-visible contract for canvas and browser state-changing actions, then either add an explicit preview-producing result or automatic optional snapshot after successful present/navigate, or tighten tool descriptions, docs, and no-node error guidance so users know to select/pair a canvas node and call snapshot/screenshot for visual confirmation.

## Reproduction Assessment

Partially yes: the `node required` branch is reproducible from the current resolver logic and unit test when no default node can be selected. The full chat-visibility failure still needs live node/Control UI confirmation because this review was source-only.

## Solution Assessment

Unclear. There is no proposed patch, and the narrowest maintainable fix depends on maintainer choice between auto-previewing state changes and documenting/enforcing explicit preview actions.
