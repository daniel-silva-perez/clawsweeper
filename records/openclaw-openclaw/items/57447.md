## Summary

Keep open. Current main still blocks cross-agent `sessions_send` under non-`all` session visibility before an A2A allow policy can authorize it, and the sandboxed label-resolution path still rejects or scopes cross-agent targets. The target repo also has open PR #57483 with closing syntax for this issue, so cleanup closure would be premature until that candidate is merged, replaced, or explicitly rejected.

## What This Changes

Allow `sessions_send` to reach A2A-allowed cross-agent sessions without granting cross-agent list, history, or status visibility.

## Best Possible Solution

Review and update PR #57483, or replace it with a narrow current-main PR, so `createSessionVisibilityChecker` lets cross-agent `send` fall through to `tools.agentToAgent` policy while read/status/list actions remain visibility-gated. The fix should add focused regression coverage for A2A-allowed sends under `tree`, A2A-denied sends, unchanged read/status/list denial, and the sandboxed label/session resolution case if that remains in scope.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: configure `tools.sessions.visibility="tree"` with `tools.agentToAgent.enabled=true` and an allowlist containing requester and target agent IDs, then `sessions_send` to an `agent:other:*` session still hits the non-`all` visibility rejection before the A2A allow check.

## Solution Assessment

Yes for the core authorization model: send should be gated by A2A policy rather than transcript visibility. The current open PR’s direction is plausible, but it needs rebasing to the current plugin SDK guard and should either handle sandboxed cross-agent label resolution or explicitly leave that as a canonical follow-up.
