## Summary

Keep this open. Current main still does not expose `--profile-id` or an equivalent operator-selected OAuth profile alias on `openclaw models auth login`; the related OpenAI Codex email-derived profile fix addresses one overwrite path but not the broader explicit alias feature, and open PR #49315 remains the active implementation candidate.

## What This Changes

The issue requests adding `--profile-id <id>` to `openclaw models auth login` so OAuth logins can persist under an operator-selected auth profile ID.

## Best Possible Solution

Keep this issue open as the tracker for explicit OAuth profile aliases. Review and land PR #49315 or equivalent work that adds `--profile-id <id>` to `models auth login`, threads it through the login/persistence path, validates that the requested ID belongs to the selected provider, handles collisions and multi-profile auth results safely, and updates CLI docs plus focused parser/persistence/provider tests.

## Reproduction Assessment

Yes. The missing feature is reproducible by inspecting current main's command registration and docs: `models auth login` has no `--profile-id` option, while `paste-token` does; no live OAuth run is needed to verify that CLI surface gap.

## Solution Assessment

Yes. A generic login-level `--profile-id` with validation and single-profile ambiguity handling is the narrow maintainable fix; a provider-specific OpenAI Codex alias flag would not cover the issue's broader OAuth-provider use case.
