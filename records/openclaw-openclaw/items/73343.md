## Summary

Keep open. Current main still drops channel sender identity before the generic `before_tool_call`, exec environment, and bundle MCP execution surfaces; the existing native plugin-tool context is only a partial path. The related open PRs are useful implementation context, but they are not merged and they cover different slices of a security-sensitive public SDK design.

## What This Changes

Expose channel sender identity, channel/account context, or an opt-in env/metadata provider to skill, shell exec, and MCP tool execution so downstream integrations can attribute actions to the triggering channel user.

## Best Possible Solution

Add an additive, opt-in SDK design that carries normalized sender, channel, account, conversation, and run identity into the relevant tool execution context, with explicit exec-env contribution and MCP caller-metadata paths, reserved-key policy, docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists: follow `SenderId` from `get-reply-run` into `createOpenClawCodingTools`, then inspect `PluginHookToolContext`, exec env construction, and bundle MCP `client.callTool` forwarding to see where identity is dropped.

## Solution Assessment

Unclear pending maintainer design. The issue's plugin-driven opt-in direction is safer than automatic global env injection, but the maintainable fix should cover hook context, exec env, and MCP caller metadata together with explicit privacy controls.

## Security Review

Status: needs_attention

Summary: The requested feature intentionally exposes channel user identity to executable and third-party tool surfaces, so the design needs explicit privacy and supply-chain guardrails.

Concerns:

- **[medium] Design identity exposure as explicit opt-in:**
  - body: Sender/channel identifiers may be personal or tenant data; putting them into process env or MCP metadata can expose them to logs, subprocesses, node hosts, sandbox runtimes, and third-party MCP servers. Any implementation should require explicit opt-in, documented fields, and reserved-key or metadata policy.
