## Summary

Keep open. Current main still has no sender/channel identity path into `before_tool_call` results or exec/skill shell environments; the only adjacent support is native plugin-tool context, which does not cover the requested shell/MCP/skill execution path. The request is valid but changes a public SDK/core security boundary, so it needs maintainer design rather than cleanup closure.

## What This Changes

Expose trusted channel user and channel identity to plugin, skill, and tool execution, likely through an opt-in hook-provided environment contribution.

## Best Possible Solution

Add an additive, opt-in plugin SDK/core seam that carries normalized sender, channel, account, and conversation identity into tool-hook context and provides a typed exec-environment contribution path through existing env validation. Document the privacy model and cover hook context, env merging, native plugin tools, and skill/script execution with focused tests.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it asks OpenClaw to expose channel identity to executable environments.

Concerns:

- **[medium] Design identity env injection as explicit opt-in:**
  - body: Passing sender/channel identity into process environments can disclose user or tenant identifiers to arbitrary commands, logs, node hosts, sandboxed processes, and third-party skill/MCP code. The SDK/API should require explicit opt-in, auditable variable declarations, and existing env filtering rather than automatic broad injection.
