## Summary

Keep open. Current main carries sender identity into the agent run and some plugin/message-tool contexts, but the generic tool hook contract, exec environment construction, and bundle MCP call path still do not expose sender/channel identity to shell or MCP tool execution. The linked open PRs are useful related implementation work, but the remaining decision is a public SDK/runtime and privacy/security design choice, not conservative cleanup.

## What This Changes

Expose channel sender and related channel identity to plugin/skill tool execution so downstream shell and MCP integrations can attribute requests to the channel user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK/runtime design that carries normalized requester identity into tool execution context, supports explicit exec-env contribution and MCP caller metadata, validates reserved env keys, documents privacy/redaction behavior, and includes focused regression coverage for hooks, exec hosts, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by following MsgContext.SenderId into run.senderId and createOpenClawCodingTools, then observing that before_tool_call, exec env construction, and SessionMcpRuntime.callTool do not carry it further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global OPENCLAW_* env injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it asks to expose channel requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
