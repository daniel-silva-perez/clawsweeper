## Summary

Keep open. Current main still drops sender identity at the generic tool-execution boundary: the tool hook contract has no sender or env output, exec env construction only uses process/request/sandbox inputs, and bundle MCP calls forward only tool arguments. Because the request changes public SDK behavior and exposes user identity to executable/MCP surfaces, it needs maintainer design rather than cleanup closure.

## What This Changes

Expose channel sender and routing identity to plugin, skill, shell-tool, or MCP execution through an opt-in mechanism such as hook context, exec env contribution, or MCP caller metadata.

## Best Possible Solution

Add an additive, opt-in SDK seam that carries normalized sender, channel, account, conversation, and run identity into tool execution context, with explicit exec-env contribution and MCP caller-metadata paths, reserved-key filtering, public docs, and focused regression tests across hooks, exec hosts, and bundle MCP.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: trace MsgContext.SenderId into get-reply-run, then inspect createOpenClawCodingTools, runBeforeToolCallHook, exec env construction, and bundle MCP call forwarding to see where identity is not exposed.

## Solution Assessment

Unclear pending maintainer design. The plugin-driven opt-in direction is safer than broad built-in env injection, but the maintainable fix should cover hook context, exec env contribution, and MCP metadata together rather than landing a shell-only path.
