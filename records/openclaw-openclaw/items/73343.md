## Summary

Keep open. Current main still lacks a sender/channel identity path into `before_tool_call`, exec environment construction, and bundled MCP tool calls; the existing `requesterSenderId` support only covers native plugin tool factories and some channel-owned message tooling. The request is valid, but it changes public SDK and executable/MCP privacy boundaries, so it needs maintainer API/security design rather than cleanup closure.

## What This Changes

Expose channel sender and channel identity to plugin, skill, shell exec, and MCP tool execution so downstream systems can attribute actions to the triggering channel user.

## Best Possible Solution

Add an additive, opt-in plugin SDK seam that carries normalized sender, channel, account, conversation, and run identity into tool-hook context, then provide explicit exec-env contribution and MCP caller-metadata paths with reserved-key filtering, public docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence code inspection path exists: start from `MsgContext.SenderId`, follow `createOpenClawCodingTools` and `runBeforeToolCallHook` into exec and bundle MCP execution, and observe that sender/channel identity is omitted before those execution surfaces.

## Solution Assessment

Unclear pending maintainer design. The plugin-driven opt-in direction is safer than automatic broad env injection, but a complete solution should cover `before_tool_call` context, exec env, and MCP metadata together rather than landing a narrow env merge without privacy and reserved-key policy.
