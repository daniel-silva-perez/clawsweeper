## Summary

Keep open. Current main still does not carry sender/channel identity into the generic tool hook context, shell exec environment construction, or bundle MCP tool calls. Native plugin tools have a partial `requesterSenderId` path, and the related open PRs are useful context, but this remains a public SDK and privacy/security design item rather than cleanup closure.

## What This Changes

Expose trusted channel sender and channel identity to plugin, skill, shell exec, and MCP tool execution so downstream integrations can attribute actions to the initiating channel user.

## Best Possible Solution

Add an additive, opt-in plugin SDK seam that carries normalized sender, channel, account, conversation, and run identity into tool execution context, with explicit exec-env contribution and MCP caller-metadata paths, reserved-key filtering, public docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence code inspection path exists: start at `MsgContext.SenderId`, follow `get-reply-run` into `createOpenClawCodingTools`, then inspect `before_tool_call`, exec env construction, and bundle MCP `callTool` forwarding to see where identity is dropped.

## Solution Assessment

Unclear pending maintainer design. The plugin-driven opt-in direction is safer than broad automatic env injection, but the best fix should cover hook context, exec env, and MCP metadata together rather than landing only a post-sanitization env merge.
