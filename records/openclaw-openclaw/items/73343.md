## Summary

Keep open. Current main still drops trusted sender/channel identity before the generic tool-hook context, shell exec environment construction, and bundle MCP calls; native plugin tools only cover a narrower requesterSenderId path. The two related open PRs are useful partial implementations, but the remaining work is a public SDK and privacy/security design decision rather than cleanup closure.

## What This Changes

Expose trusted channel sender and channel identity to plugin, skill shell exec, and MCP tool execution so downstream integrations can attribute calls to the initiating channel user.

## Best Possible Solution

Add an additive, opt-in plugin SDK seam that carries normalized sender, channel, account, conversation, and run identity into tool execution context, with explicit exec-env contribution and MCP caller-metadata paths, reserved-key filtering, public docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence code inspection path exists: follow MsgContext.SenderId through get-reply-run into createOpenClawCodingTools, then inspect before_tool_call context construction, exec env construction, and bundle MCP callTool forwarding to see where trusted identity is dropped.

## Solution Assessment

Unclear pending maintainer design. The plugin-driven opt-in direction is safer than broad automatic env injection, but the best fix should cover hook context, exec env, and MCP metadata together rather than landing only a post-sanitization env merge.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would expose channel user identity to executable and third-party integration surfaces.

Concerns:

- **[medium] Design identity exposure as explicit opt-in:**
  - body: Passing sender or channel identifiers into process environments or MCP metadata may expose personal or tenant data to logs, subprocesses, node hosts, sandbox runtimes, and third-party servers. Any implementation should require explicit opt-in, documented variables or metadata declarations, and reserved-key filtering or an equivalent policy.
