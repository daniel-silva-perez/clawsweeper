## Summary

Keep open. Current main carries sender identity into the agent run and some message/plugin tool contexts, but it still does not expose sender/channel identity through the generic before_tool_call contract, exec environment construction, or bundle MCP tool calls. The related open PRs are relevant adjacent work, but this remains a public SDK/runtime and privacy/security design decision rather than conservative cleanup.

## What This Changes

Expose channel sender and channel identity to plugin, skill, shell exec, and MCP tool execution so downstream systems can attribute actions to the triggering user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK/runtime contract that carries normalized requester identity into tool execution context, supports explicit exec-env contribution and MCP caller metadata, validates reserved environment keys, documents privacy/redaction behavior, and includes focused regression coverage across hooks, exec hosts, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction follows MsgContext.SenderId into run.senderId and createOpenClawCodingTools, then observes that before_tool_call, exec env construction, and SessionMcpRuntime.callTool do not carry it further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in hook/context is safer than automatic global OPENCLAW_* injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.
