## Summary

Keep open. Current main still drops sender/channel identity before the generic tool-hook, shell exec, and bundle MCP call surfaces, while native plugin tools only have a partial `requesterSenderId` path. The related open PRs cover useful pieces, but this remains a public SDK plus privacy/security design item rather than cleanup closure.

## What This Changes

Expose channel sender and channel identity to skill/tool execution, including shell exec environments, generic plugin tool hooks, and MCP tool calls.

## Best Possible Solution

Add an additive, opt-in plugin SDK contract that carries normalized requester, channel, account, conversation, and run identity into tool execution context, with an explicit exec-env contribution path, MCP caller metadata support, reserved-key filtering, public docs, and regression tests across hook, shell exec, and bundle MCP surfaces.

## Reproduction Assessment

Yes. A high-confidence code inspection path exists: `MsgContext.SenderId` is captured into the run, but current main omits it from `PluginHookToolContext`, exec env construction, and bundle MCP `client.callTool` requests.

## Solution Assessment

Unclear pending maintainer design. The issue's plugin-driven opt-in direction is safer than automatic global env injection, but the maintainable fix should cover hook context, exec env, and MCP caller metadata together with explicit privacy controls.

## Security Review

Status: needs_attention

Summary: The requested feature intentionally exposes channel user identity to executable and third-party tool surfaces, so the design needs explicit privacy and supply-chain guardrails.

Concerns:

- **[medium] Design identity exposure as explicit opt-in:**
  - body: Sender/channel identifiers may be personal or tenant data. Passing them into process environments or MCP metadata can expose them through subprocesses, logs, node hosts, sandbox runtimes, and third-party servers, so any implementation should require explicit opt-in, documented fields, reserved-key filtering, and tests for non-leakage by default.
