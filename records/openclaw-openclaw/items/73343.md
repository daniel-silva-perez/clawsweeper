## Summary

Keep open. Current main still carries sender identity into the run and some message-tool/plugin discovery paths, but the generic `before_tool_call` contract, exec environment construction, and bundle MCP `callTool` path still do not expose sender/channel identity. The linked open PRs are relevant implementation context, but the remaining work is a public SDK and privacy/security design decision rather than a cleanup close.

## What This Changes

Expose channel sender and channel identity to plugin hooks and exec/MCP tool execution so downstream tools can attribute calls to the initiating channel user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK design that carries normalized requester identity into tool execution context, supports explicit exec-env contribution and MCP caller metadata, validates reserved env keys, documents privacy/redaction behavior, and includes focused regression coverage for hooks, exec hosts, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by following `MsgContext.SenderId` into `run.senderId` and `createOpenClawCodingTools`, then observing that `before_tool_call`, exec env construction, and `SessionMcpRuntime.callTool` do not carry it further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global `OPENCLAW_*` env injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it asks to expose channel user identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant data. Passing them through env vars or MCP metadata can expose them to subprocess logs, node hosts, sandbox runtimes, and third-party MCP servers, so any implementation should require explicit plugin/admin opt-in, documented fields, reserved-key validation, and redaction guidance.
