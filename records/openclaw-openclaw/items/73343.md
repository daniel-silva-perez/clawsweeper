## Summary

Keep open. Current main still preserves sender identity into the run and some native plugin/message-tool paths, but it drops that identity before the generic `before_tool_call` context, exec environment construction, and bundle MCP `callTool` request. The linked open PRs are relevant implementation context, but this remains a public SDK and privacy/security design decision rather than a cleanup-close candidate.

## What This Changes

Expose the trusted channel sender and channel identity to plugin hooks, shell/skill execution, and MCP tool calls so downstream integrations can attribute requests to the initiating user.

## Best Possible Solution

Add an additive, opt-in SDK design that carries normalized sender, channel, account, conversation, and run identity into tool execution context, with explicit exec-env contribution and MCP caller metadata paths, reserved-key policy, documentation, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by following `sessionCtx.SenderId` into `run.senderId` and `createOpenClawCodingTools`, then inspecting the generic hook context, exec env construction, and bundle MCP `client.callTool` forwarding where identity is absent.

## Solution Assessment

Unclear pending maintainer design. The issue's plugin-driven opt-in direction is safer than automatic global env injection, but the maintainable fix should cover hook context, exec env, and MCP metadata together with privacy controls.

## Security Review

Status: needs_attention

Summary: The requested feature intentionally exposes channel user identity to executable and third-party tool surfaces, so it needs explicit privacy and supply-chain guardrails.

Concerns:

- **[medium] Design identity exposure as explicit opt-in:**
