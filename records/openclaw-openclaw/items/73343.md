## Summary

Keep open. Current main still carries sender identity into the agent run and some plugin/message-tool discovery paths, but it does not expose that identity through the generic `before_tool_call` contract, exec environment construction, or bundle MCP tool-call metadata; the linked open PRs are useful implementation context, but this remains a public SDK and privacy/security design decision.

## What This Changes

Expose channel sender and channel identity to plugin hooks and exec/MCP tool execution so downstream tools can attribute calls to the initiating channel user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK design that carries normalized sender, channel, account, conversation, and run identity into tool execution context, with explicit exec-env contribution and MCP caller metadata paths, reserved-key validation, manifest or config transparency, docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by following `MsgContext.SenderId` into `run.senderId` and `createOpenClawCodingTools`, then observing that `before_tool_call` context/result, exec env construction, and `SessionMcpRuntime.callTool` do not carry it further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global `OPENCLAW_*` env injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue itself is security-sensitive because it asks to expose channel user identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant data. Passing them through env vars or MCP metadata can expose them to subprocess logs, node hosts, sandbox runtimes, and third-party MCP servers, so any implementation should require explicit plugin/admin opt-in, documented fields, reserved-key validation, and redaction guidance.
