## Summary

Keep open. Current main partially addresses the Slack thread case with a shipped `channels.slack.thread.requireExplicitMention` option, but it does not implement the requested general strict mention mode and still has Discord thread-bound routing that disables `requireMention` before the shared mention gate runs.

## What This Changes

The issue requests an opt-in strict mention mode so implicit reply, thread, or session participation does not satisfy `requireMention` unless the bot is explicitly mentioned or a configured mention pattern matches.

## Best Possible Solution

Keep the issue open and define an additive mention policy that maps to the existing shared `allowedImplicitMentionKinds` seam, then wire it through the affected channel plugins starting with Discord bound threads and document how it relates to Slack's existing `thread.requireExplicitMention` option.

## Reproduction Assessment

Yes. A source-level reproduction exists for the remaining Discord path: create a guild thread-bound session with configured `requireMention: true`, then send an unmentioned follow-up; current main sets `bypassMentionRequirement` from the bound thread and disables the gate before it can skip.

## Solution Assessment

No. The current Slack-specific fix is useful but not the best complete solution for this issue because the same policy is not exposed consistently across Discord and other implicit-mention paths; the safer direction is an explicit opt-in policy backed by the shared mention gate.
