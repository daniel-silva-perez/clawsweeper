## Summary

Keep open. Current main still does not accept `api_key` as an alias in `auth-profiles.json`; only `apiKey` is normalized to canonical `key`, while eligibility and runtime resolution require `key` or `keyRef`. Docs now show the correct `{ provider, key }` shape, but the reported silent failure remains for `api_key`-only profiles. Related PR #57418 targeted the exact alias fix, but the provided GitHub context shows it was closed unmerged.

## Best Possible Solution

Patch `src/agents/auth-profiles/persisted.ts` so `normalizeRawCredentialEntry` maps `api_key` to canonical `key` when `key` is absent, add a regression test beside the existing `mode/apiKey` alias cases, and consider a warning or doctor check for unrecognized auth-profile credential fields so future config mistakes are visible.
