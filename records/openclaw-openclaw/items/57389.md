## Summary

Keep open. Current main still does not accept `api_key` as an alias for auth-profile API key material: persisted auth-profile normalization maps only `apiKey` to `key`, while runtime resolution and eligibility look only at `key`/`keyRef`. The docs now state the canonical shape is `{ provider, key }`, but the reported silent `api_key` failure remains in the runtime path.

## Best Possible Solution

Patch `src/agents/auth-profiles/persisted.ts` so `normalizeRawCredentialEntry` accepts `api_key` as a backwards-compatible alias for `key` when canonical `key` is absent, add a regression test beside the existing `mode/apiKey` alias tests, and consider a warning or doctor check for unknown auth-profile credential fields so similar mistakes are visible.
