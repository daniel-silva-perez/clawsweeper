## Summary

Keep this open. Current main proves the canonical in-process `edit` payload works, but it does not cover the reported DeepSeek/OpenAI-compatible transport failure where a native `edits` array appears to reach validation in a malformed item shape.

## What This Changes

The issue reports that the `edit` tool rejects both stringified and native-array `edits` payloads in a DeepSeek/OpenAI-compatible environment, making the tool unusable for that provider path.

## Best Possible Solution

Add a focused current-main regression for the DeepSeek/OpenAI-compatible tool-call path that exercises `edit` with a native `edits` array. If the captured payload confirms item-wrapped array entries, add a narrow transport-artifact normalizer before edit validation while keeping `oldText`/`newText` as the public contract.

## Reproduction Assessment

Unclear. The reporter provides concrete payloads, version, provider, model, and error strings, while current tests prove only the canonical in-process path; this read-only review did not produce a high-confidence current-main reproduction of the DeepSeek transport failure.

## Solution Assessment

Unclear. The maintainable direction is not to broadly accept arbitrary edit shapes; first capture the raw provider/tool payload, then normalize only the confirmed wrapper artifact or fix the upstream boundary that creates it.
