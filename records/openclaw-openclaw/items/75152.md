## Summary

Keep this open. Current main still proves only the canonical in-process `edit` payload, while the reported DeepSeek/OpenAI-compatible path can still plausibly fail before or during validation with a malformed `edits` element shape.

## What This Changes

The issue reports that the `edit` tool rejects both stringified and native `edits` array payloads through the DeepSeek/OpenAI-compatible route, making the tool unusable in that environment.

## Best Possible Solution

Add a focused regression that exercises the DeepSeek/OpenAI-compatible tool-call path with a native `edits` array, capture the raw parameters received by OpenClaw, then fix the narrow boundary that creates or receives the wrapper while preserving `{ oldText, newText }` as the public edit contract.

## Reproduction Assessment

Unclear. The reporter provides concrete payloads, provider/model, version, and errors, but current-main inspection only proves canonical in-process edit calls and does not reproduce the DeepSeek transport failure with captured raw parameters.

## Solution Assessment

Unclear. The maintainable direction is to capture and normalize only the confirmed transport artifact, or fix the upstream adapter that creates it; broadly accepting edit-shape variants is not yet justified by the evidence.
