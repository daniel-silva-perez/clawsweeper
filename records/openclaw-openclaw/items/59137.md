## Summary

Keep this PR open. Current main still lacks the two central recovery guarantees described here: direct embedding batches are cached only after all batches finish, and safe reindex rollback restores the original DB without preserving new cache progress or an explicit full-session retry marker. The PR is useful, but current main has since refactored memory-core helpers, so it likely needs maintainer review/rebase or a replacement rather than cleanup closure.

## What This Changes

The PR updates memory-core reindex rollback logic to preserve sync retry state, track pending full-session retries, mirror successful embedding-cache writes to the committed DB during safe reindex, and add focused regression tests.

## Best Possible Solution

Land or replace this with a current-main memory-core patch that persists each successful direct embedding batch, mirrors safe-reindex cache writes into the committed DB until rollback/commit is decided, restores dirty/session retry state after failed safe and unsafe reindexes, and adapts the tests to the current helper split.

## Reproduction Assessment

Yes. The PR body gives concrete manual and mocked reproduction paths, and current-main source confirms the direct-batch cache write happens only after all batches complete, so a later batch failure can discard earlier successful cache work.

## Solution Assessment

Yes in direction, but unclear for the exact branch. The proposed state snapshot, full-session retry marker, and cache mirroring are a maintainable fix shape, but the implementation should be reviewed against current-main helper modules before merge.
