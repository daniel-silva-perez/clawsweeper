## Summary

Keep open. Current main still watches only the active root config path, while `$include` support and an include-path scanner already exist. The older #41587 was closed in favor of this more concrete report, #59632 was closed unmerged, and #74122 is broader debounce/scoped-reload work rather than a substitute for include-file watching.

## What This Changes

The issue asks for config hot-reload to watch files referenced by `$include` so edits to included config files reload without a gateway restart.

## Best Possible Solution

Add dynamic include-watch synchronization inside the gateway config reloader, using the existing include scanner after valid snapshots are read, while preserving current recovery, in-process write, debounce, and restart-planning behavior. The finished behavior should watch the root config plus the current include graph, drop removed include paths, and document the user-facing hot-reload behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `openclaw.json` with a valid `$include`, start the gateway, then edit only the included file; current main has no watcher on that included path, so no reload event is scheduled.

## Solution Assessment

Yes. The issue’s proposed direction is the narrowest maintainable fix: reuse `collectIncludePathsRecursive()` and chokidar `add`/`unwatch` from `startGatewayConfigReloader`, with tests covering include-only changes and stale include removal.

## Repair Work Prompt

Fix openclaw/openclaw#59616. Current main only watches `opts.watchPath` in `src/gateway/config-reload.ts`, so direct edits to files referenced by `$include` do not trigger gateway config reload. Use the existing `collectIncludePathsRecursive()` helper from `src/config/includes-scan.ts` to maintain a dynamic set of include watch paths inside `startGatewayConfigReloader`: after startup/initial valid snapshot reads and after each successful valid snapshot apply, collect include paths from the root authored/parsed config, diff against the current include watch set, call `watcher.add()` for new paths and `watcher.unwatch()` for removed paths, and keep failures best-effort so scanner or watcher errors do not break the reload loop. Ensure include-file change events schedule reloads even when the root config hash is unchanged, including after in-process writes; do not reintroduce the root-hash dedupe bug discussed on #59632. Preserve existing recovery, restart planning, skill invalidation, and in-process write semantics. Add focused tests in `src/gateway/config-reload.test.ts` for startup include watching, include-only change reload, unwatching removed includes, nested or missing include scan behavior, and the hash-dedupe include-only regression. Update docs/changelog if behavior text changes; this is user-facing, so any changelog entry should credit @alkor2000 and likely @christopherflora. Do not change extension/plugin ownership boundaries or add bundled plugin-specific logic.
