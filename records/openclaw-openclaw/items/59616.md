## What This Changes

The issue requests that config hot-reload watch files referenced through `$include` so edits to included config files trigger a reload.

## Best Possible Solution

Land a narrow gateway reloader change that keeps the active watcher set synchronized with the current valid `$include` graph, with tests for startup discovery, add/remove diffing, nested includes, included-file change events, and nonfatal sync failures.

## Repair Work Prompt

Fix openclaw/openclaw#59616. Current main's `startGatewayConfigReloader` only watches `opts.watchPath`, so edits to files referenced by `$include` do not trigger reload. Implement a narrow reloader-side include-watch sync: after startup/initial successful snapshot and after each accepted external snapshot, compute include paths from the valid authored config snapshot, diff them against a maintained include-watch set, call Chokidar `watcher.add()` for new paths and `watcher.unwatch()` for removed paths, and keep sync failures best-effort/nonfatal. Preserve existing debounce, missing-config retries, in-process write hash dedupe, last-known-good promotion, restart planning, and source-config reload planning. Do not solve #74122 debounce/scoped reload in this PR. Be careful not to watch paths that the normal include resolver/security guards would reject. Add focused tests in `src/gateway/config-reload.test.ts` for initial include watch registration, included-file change triggering reload, include removal unwatching, nested include discovery, and graceful sync/read failures. Add a one-line `CHANGELOG.md` Fixes entry with `Thanks @alkor2000` if behavior changes.
