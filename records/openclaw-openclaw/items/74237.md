## Summary

Keep this open. Current main still lacks a Feishu-owned repair or quarantine path for stale/corrupt Feishu runtime state and direct channel session entries, and the issue is now paired with open PR #74397 using closing syntax, so policy says the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request a Feishu-owned guard or doctor repair so stale/corrupt Feishu local runtime state and direct channel session entries cannot drive a gateway hot loop.

## Best Possible Solution

Land or replace #74397 with a scoped Feishu plugin repair path that detects corrupt Feishu JSON state and stale direct Feishu session entries, archives only Feishu-owned runtime/session artifacts, preserves credentials and non-Feishu/ACP sessions, logs the affected keys/files, and adds focused Feishu doctor tests plus nearby session repair coverage.

## Reproduction Assessment

Unclear for the full sustained CPU hot loop because the private stale/corrupt state files are not available. The report gives enough detail to build a high-confidence synthetic repair test using corrupt Feishu dedupe JSON plus stale `agent:*:feishu:direct:*` session entries.

## Solution Assessment

Yes, the best direction is a Feishu-owned doctor/repair path rather than a core startup migration. PR #74397 appears aligned with that boundary, but it is unmerged and still needs maintainer review.
