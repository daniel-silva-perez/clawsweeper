## Summary

Keep this issue open. The report is fresh, detailed, and still matches current main: Feishu startup warms persistent dedupe state and starts the WebSocket monitor, but current code does not quarantine corrupt Feishu dedupe/runtime state, does not isolate `agent:...:feishu:...` session rows before dispatch, and the documented repair paths are generic rather than Feishu-state-specific.

## Best Possible Solution

Keep this open and investigate in the Feishu plugin first. The best fix is likely a plugin-owned defensive path that detects corrupt Feishu dedupe/runtime state, quarantines or rebuilds it with a clear warning, and optionally adds a doctor repair that archives only Feishu-scoped session rows/artifacts while preserving credentials. Add targeted Feishu tests around corrupt dedupe JSON and stale `agent:*:feishu:*` session entries before considering any shared session-store seam.
