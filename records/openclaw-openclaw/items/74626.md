## Summary

Current main and the shipped v2026.4.27 release already implement the requested behavior: `sessions_spawn.model` is persisted onto the child session as a provider/model override before the child run starts, and the agent runtime resolves that stored override before selecting the model. The report was against v2026.4.25, and the relevant fix shipped after that build.

## What This Changes

Make `sessions_spawn` honor an explicit model parameter for subagents instead of falling back to the configured primary model.

## Best Possible Solution

Keep the shipped child-session override implementation and have users on v2026.4.25 upgrade to v2026.4.27 or newer; if the same symptom still appears on that release, it should be reported with fresh gateway logs and the exact session entry/status output.

## Reproduction Assessment

Yes. The original repro path is clear: set a different primary model, call `sessions_spawn` with an explicit model, then inspect the child session/runtime model. Current-main source and regression tests show that path now persists and consumes the requested provider/model before the child run starts.

## Solution Assessment

Yes. Persisting the override on the child session is the narrow maintainable fix because it keeps runtime execution, status, and stored session state aligned; the closed direct-forwarding PR is redundant for the shipped path.
