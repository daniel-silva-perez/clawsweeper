## Summary

Keep open. Current main and the latest release still clear self-declared operator scopes for ordinary device-less shared-token/password API clients, while preserving scopes only for the documented direct-loopback internal backend path; that leaves the reported external orchestration use case unresolved pending maintainer product/security judgment.

## What This Changes

The issue requests restoring shared-auth preservation of requested operator scopes for headless operator API clients that connect without device identity.

## Best Possible Solution

Define the supported external headless API-client contract explicitly: either add a narrow, documented auth path that binds operator scopes without browser device identity and covers it with gateway protocol tests, or document that external API clients must use device identity/device tokens while keeping the current backend exception internal-only.

## Reproduction Assessment

Yes. Current tests reproduce the reported class: shared-token/password operator connects with `device: null` are accepted, their scopes are cleared, and scoped RPCs fail with `missing scope`.

## Solution Assessment

Unclear. The issue's direct `sharedAuthOk` preservation would restore the old behavior, but it is not clearly the safest fix because it broadens scope preservation for every device-less shared-secret API client; a narrower explicit API-client/device-token contract may be safer.
