## Summary

Current main does not implement the issue's requested sharedAuthOk-style preservation for generic shared-token/password API clients without device identity. The gateway now has a narrower direct-local `gateway-client`/`backend` exception, but ordinary device-less shared-auth operator connects still clear scopes and tests assert that behavior, so the reported external orchestration/API-client use case remains actionable unless maintainers intentionally decline or document a different supported path.

## Best Possible Solution

Keep the issue open for maintainer product/security judgment. The concrete path is to either add and document a supported external headless API-client auth model that binds operator scopes without browser/device identity, or explicitly document that external API clients must use device identity/device tokens and that the current direct-local `gateway-client` backend exception is internal-only. Any implementation should update the gateway protocol docs and add regression tests for the chosen external-client behavior.
