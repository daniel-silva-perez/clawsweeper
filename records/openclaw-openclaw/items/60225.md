## Summary

Current main still does not implement sharedAuthOk-style scope preservation for generic device-less shared-token/password operator API clients. The code and tests preserve scopes only for the narrow direct-local internal `gateway-client`/`backend` path; ordinary device-less shared-auth operator connects still clear scopes and can hit the reported `missing scope` behavior.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. The best path is either to add and document a supported external headless API-client auth model that can bind operator scopes without browser/device identity, or to explicitly document that external API clients must use device identity/device tokens and that the current direct-local `gateway-client` backend exception is internal-only. Either direction should update gateway protocol docs and add regression tests for the chosen behavior.

## Risks / Open Questions

- Preserving scopes for every device-less shared-secret API client would broaden an auth-sensitive path; the current implementation appears intentionally narrowed to direct-local internal backend RPCs.
- Docs around explicit scopes and device-less exceptions should be clarified if maintainers decide generic external API clients must use device identity/device tokens instead of retaining shared-auth scopes.
