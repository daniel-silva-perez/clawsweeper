## Summary

Current main removed the stale visible browser Speech API controls and added Control UI Talk, but it does not implement the requested self-hosted `messages.tts` read-aloud route or a batch `tools.media.audio` STT bridge for WebChat. The related Talk work resolves realtime voice visibility, not this gateway-backed self-hosted TTS/STT provider request.

## Best Possible Solution

Keep the issue open. The best product path is to separate the concerns explicitly: if per-message read-aloud remains desired, wire it to gateway `tts.convert` with authenticated playback and browser SpeechSynthesis only as fallback; for mic input, add MediaRecorder capture plus an authenticated Gateway RPC/HTTP endpoint that validates audio size/type and routes through `transcribeAudioFile`/`tools.media.audio`. Update Control UI docs/tests to distinguish realtime Talk from gateway-backed read-aloud and server-side STT.
