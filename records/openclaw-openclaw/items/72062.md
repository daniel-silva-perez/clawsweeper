## Summary

Keep this issue open. Current main still has per-agent tool-name allow/deny and positive exec allowlist/script-path matching, but no typed, documented, or enforced per-agent negative policy for shell wrapper or script paths, so the reported exec-policy gap remains valid.

## What This Changes

The issue requests a per-agent policy for denying specific shell wrapper or script paths invoked through `tools.exec`, even when the interpreter itself is otherwise allowed.

## Best Possible Solution

Add a typed per-agent negative exec script-path policy under the exec policy surface, reuse the existing shell-wrapper/script candidate resolution, apply it before approval prompts and durable approval reuse, make deny win over allowlists and approvals, cover gateway and node exec, and document the behavior with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: configure a per-agent `tools.deny` for an MCP namespace, allow `sh` or `bash` through exec allowlist, then invoke `sh <sensitive-wrapper>`; current tool policy sees only the `exec` tool name while exec approval code evaluates allowlist state without a wrapper-path deny.

## Solution Assessment

Unclear. A negative script-path policy addresses the real gap, but the exact API name, nesting, interpreter coverage, and error semantics should be decided by maintainers because this changes core exec security behavior.
