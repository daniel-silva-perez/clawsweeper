## Summary

Keep open. Current main still has no typed, documented, enforced per-agent deny policy for exec wrapper or script paths; tool denial remains tool-name based, while wrapper script paths are resolved only for positive allowlist and approval matching. This is a valid core exec-policy/security request, but the API shape and enforcement semantics need maintainer review before an automated repair is safe.

## What This Changes

Request to add a per-agent exec wrapper/script-path deny policy, proposed as `agents.list[].tools.deny_wrappers`, so allowed shell interpreters cannot reach disallowed wrapper scripts.

## Best Possible Solution

Add a documented per-agent negative exec script-path policy in the core exec security surface, reuse the existing wrapper/script candidate resolution, make deny win before allowlist satisfaction and approval reuse, and cover both gateway exec and node `system.run` with focused schema, policy, and runtime regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `agents.list[main].tools.deny` for an MCP namespace, allow `sh` or `bash` through exec allowlist/approval, and invoke `sh <sensitive-wrapper>`; current main filters only the `exec` tool name while wrapper script paths are not denied.

## Solution Assessment

Unclear. A negative script-path policy is the right class of fix, but the exact API, interpreter scope, path resolution rules, and precedence need maintainer security/product judgment before implementation.
