## Summary

Keep open. Current main still lacks a typed, documented, and enforced per-agent negative policy for exec wrapper or script paths; existing per-agent tool denies are tool-name filters, while exec allowlist and approval paths only use resolved script paths as positive allowlist or approval inputs.

## What This Changes

Request a per-agent exec wrapper/script-path deny policy, such as `agents.list[].tools.deny_wrappers`, that blocks resolved interpreter script operands before exec approval or execution.

## Best Possible Solution

Add a documented per-agent negative exec script-path policy in the core exec security surface, reuse existing shell-wrapper/script candidate resolution, make deny win before allowlist satisfaction, durable approval reuse, allow-always persistence, and approval prompts, and cover both gateway and node exec with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure per-agent `tools.deny` for an MCP namespace, allow `sh` or `bash` through exec allowlist, then invoke `sh <sensitive-wrapper>`; current main filters only the exec tool name while exec policy has no script-path deny check.

## Solution Assessment

Unclear. A negative script-path policy is the right class of fix, but the API name, nesting, interpreter coverage, and precedence semantics should be decided by maintainers because this changes core exec security behavior.
