## Summary

Keep this issue open. Current main still has per-agent tool-name allow/deny plus positive exec allowlist/script-path matching, but no typed, documented, or enforced per-agent negative policy for shell wrapper or script paths, so the reported exec-policy gap remains valid.

## What This Changes

The issue requests a per-agent policy to deny specific shell wrapper or script paths invoked through `tools.exec`, preventing `tools.deny` from being bypassed by allowed interpreters running shared scripts.

## Best Possible Solution

Implement a typed and documented per-agent exec path deny policy that reuses the existing shell-wrapper/script candidate resolution, applies before approval prompts and durable approval reuse, makes deny win over allowlist and approval paths, covers gateway and node exec, and includes regression tests plus public docs.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure `agents.list[main].tools.deny` for an MCP namespace, allow `sh`/`bash` through exec allowlist, and invoke `sh <script>`; current tool policy sees only the `exec` tool name while exec approval code evaluates positive allowlist state without any wrapper-path deny.

## Solution Assessment

Unclear. The requested negative path policy addresses the right core gap, but the exact API name and nesting should be decided by maintainers because it touches exec security semantics and should probably live with the existing `tools.exec` policy fields.
