## Summary

Keep this PR open. Current main still does not implement automatic macOS idle-sleep prevention for the Gateway, and the related broader PR #28317 was closed without merging, so the contribution is not obsolete. The proposed one-file patch remains useful but should not merge as-is because the existing review discussion is correct: `spawn()` failures are delivered through the child `error` event, the helper is started outside the cleanup-protected lock scope, and the new startup command should avoid PATH-based resolution.

## What This Changes

The PR adds a macOS-only Gateway run-loop helper that spawns `caffeinate -i -w <pid>` and kills it during shutdown cleanup.

## Best Possible Solution

Land this only after the Gateway lifecycle and command-execution edges are tightened: use `/usr/bin/caffeinate` or a validated absolute resolver, attach `error` and useful `exit` handling before `unref()`, start the helper inside the same cleanup-protected scope as the Gateway lock, preserve the non-macOS no-op path, add focused run-loop tests, and document the macOS power-management behavior if it becomes the default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so the surrounding `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken `caffeinate` launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate inside the cleanup scope:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but cleanup is only registered in the later `finally`. If lock acquisition throws, the helper can be left running outside the intended cleanup path; start it after the lock is acquired or wrap lock acquisition and helper startup in the same cleanup scope.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a Gateway startup command-execution path that should avoid PATH-based executable resolution before merge.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment decide which executable runs during Gateway startup. Because wrappers, dev shells, and launchd/service environments can affect PATH, this fixed macOS helper should use `/usr/bin/caffeinate` or another validated absolute path with explicit error handling.
  - confidence: 0.8

## Risks / Open Questions

- The patch currently lacks a child-process `error` listener, so a failed `caffeinate` spawn can crash Gateway startup instead of degrading cleanly.
- The helper starts before Gateway lock acquisition but is cleaned up only by the later `finally`, leaving a lifecycle gap if lock acquisition fails.
- Default macOS idle-sleep prevention is a user-visible power-management choice and should be accepted deliberately, with docs updated if maintainers want it on by default.
- The new startup command is resolved through PATH, which is avoidable for a fixed macOS system binary.
