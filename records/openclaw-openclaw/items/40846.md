## Summary

Keep open. Current main still does not implement automatic macOS idle-sleep prevention for the Gateway, and the existing docs only describe manual 24/7 workarounds. The PR remains a plausible narrow implementation, but it is not merge-ready because the helper process error path is unhandled, the command is resolved through PATH, the branch is dirty against current main, and there is no focused run-loop coverage for the new macOS lifecycle behavior.

## What This Changes

The PR adds a macOS-only Gateway run-loop helper that spawns `caffeinate -i -w <pid>` and calls its cleanup function from the run-loop `finally` block.

## Best Possible Solution

Keep the work open for a rebase or replacement that fits the current Gateway lifecycle. The ideal implementation is a small macOS-only helper that uses a trusted `caffeinate` executable path or validated resolver, attaches `error` and useful `exit` handling, starts only inside a cleanup-protected lifecycle scope, stops on Gateway shutdown, and includes focused run-loop tests plus any docs/config decision maintainers want for default power behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable via an asynchronous `error` event, so this `try/catch` does not protect Gateway startup. Add an `error` listener before unref so a broken `caffeinate` launch logs and degrades instead of crashing the process with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Keep caffeinate inside the cleanup scope:** `src/cli/gateway-cli/run-loop.ts:64-65`
  - body: The helper starts before `acquireGatewayLock()`, but cleanup is only registered in the later `finally`. If lock acquisition throws and the caller retries in the same process, the caffeinate process can be left running until process exit and each retry can spawn another helper. Start it after the lock is acquired or wrap both startup and lock acquisition in the same cleanup scope.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a Gateway startup process execution path through PATH; no dependency, workflow, secret, or permission-file changes were found.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment choose which executable runs at Gateway startup. Because wrappers and service environments can affect PATH, this should use `/usr/bin/caffeinate` or another validated absolute path, with explicit error handling.
  - confidence: 0.78
