## Summary

Keep this PR open. Current main still does not implement automatic macOS idle-sleep prevention for the Gateway, and the docs still describe `caffeinate` only as a manual 24/7 workaround. The proposed patch is not merge-ready because it adds a Gateway startup child-process path without handling asynchronous spawn failures, starts the helper outside the guaranteed cleanup scope, and needs maintainer judgment on default macOS power behavior.

## What This Changes

The PR modifies `src/cli/gateway-cli/run-loop.ts` to spawn a macOS-only `caffeinate -i -w <pid>` helper when the Gateway run loop starts and stop it during shutdown cleanup.

## Best Possible Solution

Land a small macOS-only helper only after the lifecycle and security edges are tightened: use `/usr/bin/caffeinate` or a validated resolver, attach `error` and useful `exit` handling, start it inside a cleanup-protected Gateway lifecycle scope, stop it on shutdown, preserve non-macOS no-op behavior, and add focused run-loop tests plus docs if maintainers approve this as default behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so the surrounding `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken `caffeinate` launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Keep caffeinate inside the cleanup scope:** `src/cli/gateway-cli/run-loop.ts:64-65`
  - body: The helper is started before `acquireGatewayLock()`, but cleanup is only registered in the later `finally`. If lock acquisition throws or the caller retries in the same process, the helper can be left running outside the intended cleanup path; start it after the lock is acquired or wrap lock acquisition and helper startup in the same cleanup scope.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a Gateway startup command-execution path through PATH and needs tightening before merge.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment choose which executable runs at Gateway startup. Because wrappers and launchd/service environments can affect PATH, this should use `/usr/bin/caffeinate` or another validated absolute path with explicit error handling.
  - confidence: 0.8
