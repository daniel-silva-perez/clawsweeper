## Summary

Keep open for maintainer review. Current main still does not implement automatic macOS idle-sleep prevention for the Gateway, so this PR is not obsolete, but the proposed patch is not merge-ready: it adds a Gateway startup child-process path with async spawn-error and PATH-resolution gaps, starts the helper before lock ownership, and is stale against the current run-loop lifecycle shape. The author already acknowledged the missing spawn error handler in review discussion and planned to address it in a later rebase cycle.

## What This Changes

The PR modifies the Gateway run loop to spawn a macOS-only `caffeinate -i -w <process.pid>` child process and clean it up when `runGatewayLoop` exits.

## Best Possible Solution

Revise before landing: create the macOS sleep-prevention helper only after Gateway lock ownership is established and inside a cleanup-protected lifecycle, use `/usr/bin/caffeinate` or a validated absolute resolver, attach a child `error` listener before `unref()`, preserve the non-Darwin no-op path, add focused tests for spawn failure and lock-conflict cleanup, and document the default macOS power behavior if maintainers approve it.

## Reproduction Assessment

Yes for the patch-level defects: the PR diff shows the helper starts before lock acquisition and lacks a child `error` listener, current main retries supervised lock conflicts in the same parent process, and a Node probe confirms missing-executable spawn failures are asynchronous. The actual overnight idle-sleep improvement was not reproduced here.

## Solution Assessment

No. The direction may be useful, but this PR is not the best mergeable solution until command resolution, async spawn-error handling, cleanup scoping, tests, docs, and maintainer approval of the default macOS power behavior are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so the surrounding `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate after locking:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but if lock acquisition throws under supervised recovery, the run-loop never reaches the later cleanup path. That can leave `caffeinate -w <parent pid>` running while this process is only retrying or yielding to an existing Gateway, so create it only after lock ownership is established.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff introduces a new Gateway startup command-execution path that uses PATH-based executable resolution and lacks explicit child-process error handling.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment decide which executable runs during Gateway startup. Because dev shells, launchd wrappers, or service environments can alter PATH, this fixed macOS helper should use `/usr/bin/caffeinate` or another validated absolute path with explicit error handling.
  - confidence: 0.82
