## Summary

Keep open. Current main and the latest release still lack automatic macOS idle-sleep prevention for the Gateway, so the PR is not obsolete, but the proposed patch has unresolved correctness and command-execution concerns before it should merge. The prior broader PR #28317 is closed and unmerged, and the author acknowledged the missing child-process error handling in review discussion.

## What This Changes

The PR adds a Darwin-only Gateway helper that spawns `caffeinate -i -w <pid>` from `src/cli/gateway-cli/run-loop.ts` and attempts to kill it when `runGatewayLoop` exits.

## Best Possible Solution

Revise before landing: use `/usr/bin/caffeinate` or a validated absolute resolver, attach child `error` handling before `unref()`, start and stop the helper inside one cleanup-protected lifecycle scope, preserve non-Darwin behavior, add focused run-loop tests, and document the macOS power behavior if maintainers want it enabled by default.

## Reproduction Assessment

Yes for the patch-level defects: the PR diff shows the helper starts before lock acquisition, and Node's child_process docs establish that failed spawn is reported via an `error` event. The long-duration macOS idle-sleep benefit itself has only author-reported verification, not live proof in this review.

## Solution Assessment

No. The product direction may be useful, but this patch is not the best mergeable form until command resolution, async spawn-error handling, cleanup scoping, tests, and maintainer approval of default macOS power behavior are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so the surrounding `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate inside the cleanup scope:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but cleanup is only registered in the later `finally`. If lock acquisition throws, the helper can be left running outside the intended cleanup path; start it after the lock is acquired or wrap lock acquisition and helper startup in the same cleanup scope.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a new Gateway startup command-execution path that should avoid PATH-based executable resolution before merge.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment decide which executable runs during Gateway startup. Because dev shells, wrappers, and service environments can affect PATH, this fixed macOS helper should use `/usr/bin/caffeinate` or another validated absolute path with explicit error handling.
  - confidence: 0.82
