## Summary

Keep open for maintainer review. Current main and the latest release still lack automatic macOS idle-sleep prevention for the Gateway, but this PR is not mergeable as-is because the new helper can crash on async spawn failure, can outlive failed supervised lock attempts, and adds a startup command-execution path that needs tighter resolution.

## What This Changes

The PR adds a macOS-only Gateway run-loop helper that spawns `caffeinate -i -w <gateway pid>` and calls its cleanup from the run-loop `finally` block.

## Best Possible Solution

Revise before landing: create the macOS sleep-prevention helper only after Gateway lock ownership is established and within a cleanup-protected lifecycle, use `/usr/bin/caffeinate` or a validated absolute resolver, attach an `error` listener before `unref()`, preserve the non-Darwin no-op path, add focused run-loop tests for spawn failure and lock-conflict cleanup, and document the macOS power behavior if maintainers want it enabled by default.

## Reproduction Assessment

Yes for the patch-level defects: the PR diff shows the helper starts before lock acquisition and lacks a child `error` listener, current main retries supervised lock conflicts in the same parent process, and a local Node probe confirms missing-executable spawn failures are asynchronous. The actual overnight idle-sleep benefit is not reproduced in this review.

## Solution Assessment

No. The direction may be useful, but this patch is not the best mergeable solution until command resolution, async spawn-error handling, cleanup scoping, tests, docs, and maintainer approval of the default macOS power behavior are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so the surrounding `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate after locking:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but cleanup is only reached later in the run-loop lifetime. In supervised lock recovery, lock conflicts are retried in the same parent process, so failed attempts can leave `caffeinate -w <parent pid>` children running while this process is not the active Gateway.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a new Gateway startup command-execution path that should avoid PATH-based executable resolution and include explicit child-process error handling before merge.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment decide which executable runs during Gateway startup. Because dev shells, launchd wrappers, or service environments can alter PATH, this fixed macOS helper should use `/usr/bin/caffeinate` or another validated absolute path with explicit error handling.
  - confidence: 0.82
