## Summary

Keep open. Current main still does not prevent macOS idle sleep from the Gateway run loop, and the earlier related PR #28317 was closed without merge, so the requested behavior is not implemented or superseded. The PR is a useful implementation candidate, but its current diff is not merge-ready because it adds a Gateway startup helper with unhandled child-process error behavior, a cleanup-scope hole, and PATH-based command resolution.

## What This Changes

The PR adds a Darwin-only Gateway helper that runs `caffeinate -i -w <pid>` from `src/cli/gateway-cli/run-loop.ts` and attempts to kill it when the run loop exits.

## Best Possible Solution

Revise before landing: use `/usr/bin/caffeinate` or a validated absolute resolver, attach child `error` handling before `unref()`, start and stop the helper inside the same cleanup-protected lifecycle scope, preserve the non-Darwin no-op path, add run-loop tests, and document the macOS power behavior if maintainers want it enabled by default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so the surrounding `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate inside the cleanup scope:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but cleanup is only registered in the later `finally`. If lock acquisition throws, the helper can be left running outside the intended cleanup path; start it after the lock is acquired or wrap lock acquisition and helper startup in the same cleanup scope.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a Gateway startup command-execution path that should avoid PATH-based executable resolution before merge.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment decide which executable runs during Gateway startup. Because dev shells, wrappers, and service environments can affect PATH, this fixed macOS helper should use `/usr/bin/caffeinate` or another validated absolute path with explicit error handling.
  - confidence: 0.82
