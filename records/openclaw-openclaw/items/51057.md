## Summary

Keep open. Current main has mitigations around idle timeouts and assistant-side fallback, but it still explicitly excludes timeout failures from persisted auth-profile failure/cooldown handling, and a focused test locks in that behavior. That means the central request in this issue is not implemented on main.

## Best Possible Solution

Keep this issue open for a targeted failover fix. The best path is to add a conservative timeout policy that counts consecutive model/provider/profile timeouts during embedded runs and triggers a short model-scoped cooldown or bounded primary-provider failover budget, while preserving the current guard against poisoning shared auth health for isolated transport timeouts and external aborts. Add focused tests for multi-profile timeout sequences reaching configured model fallback quickly and for one-off timeouts not permanently disabling a profile.

## Risks / Open Questions

- Changing this naively could poison shared auth-profile health for one-off network stalls or user aborts, which the current policy intentionally avoids.
- Closing as implemented would hide a still-real product decision: whether repeated timeouts should create a short, model-scoped circuit breaker or be handled by a separate primary-provider failover budget.
