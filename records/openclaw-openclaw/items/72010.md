## Summary

Keep open. Current main does not implement the requested progress/backoff behavior, and the latest PR head still has a blocking generic-dispatch coverage gap: several real progress callbacks remain passed through without refreshing diagnostic session activity.

## What This Changes

The PR adds diagnostic session progress tracking and repeated stuck-warning backoff, wires progress marks into ACP and generic reply dispatch paths, updates diagnostics config docs/generated schema text, adjusts read-only channel plugin loader typing, and updates related tests.

## Best Possible Solution

Land this PR or an equivalent patch after making the diagnostic progress marker cover every non-typing, real work/reply progress callback in generic dispatch, preserving ACP progress gating by `diagnostics.enabled`, keeping repeated stuck-warning backoff, and keeping docs/generated config text aligned with the no-progress semantics.

## Reproduction Assessment

Yes. Current main has a high-confidence fake-timer reproduction by marking a session `processing`, advancing beyond `diagnostics.stuckSessionWarnMs`, and observing `session.stuck`; the PR-specific gap is source-reproducible with a resolver that only invokes callbacks like `onPartialReply`, `onItemEvent`, or `onCommandOutput`.

## Solution Assessment

No. The design is at the right layer, but the current PR implementation is incomplete because it only refreshes diagnostic activity for a subset of generic dispatch progress callbacks; the safer fix is to wrap all non-typing progress callbacks consistently.
