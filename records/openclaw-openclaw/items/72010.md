## Summary

Keep open. Current main still lacks the PR's no-progress diagnostic semantics, and the closed predecessor #71969 is superseded by this active PR rather than by a merged fix. The proposed patch is on the right code path, but review found one remaining correctness gap in the generic dispatch progress wiring before it should land.

## What This Changes

The PR adds diagnostic session progress marking and repeated stuck-warning backoff, wires progress marks into ACP and generic reply dispatch paths, updates diagnostics config docs/generated schema, adjusts read-only plugin loader typing, and updates affected tests.

## Best Possible Solution

Land this PR or an equivalent implementation after tightening the generic dispatch progress coverage: diagnostic state should reset on every real reply/work progress signal that can show an active turn, repeated `session.stuck` warnings should back off while unchanged, typing keepalive should remain excluded, ACP runtime progress should stay gated by diagnostics config, and docs/generated config surfaces should match the behavior.

## Reproduction Assessment

Yes. Current main can be reproduced with the existing fake-timer heartbeat path: mark a session `processing`, advance beyond `diagnostics.stuckSessionWarnMs`, and it emits `session.stuck` from age alone. The PR-specific gap is also source-reproducible by a resolver that only calls callbacks such as `onPartialReply` or `onItemEvent`; those callbacks are not wired to `markProgress()` in the diff.

## Solution Assessment

No. The PR is aimed at the right maintainable layer, but the proposed generic dispatch wiring is incomplete because several real progress callbacks still do not refresh diagnostic activity. The safer fix is to keep the same diagnostic marker/backoff design and wire it consistently to all non-typing progress callbacks.
