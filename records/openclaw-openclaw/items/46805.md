## Summary

Keep open. The PR addresses a real Feishu sender-attribution gap, but it is not merge-correct against current main: current main resolves sender names through the split `bot-sender-name.ts` helper, while this PR still changes the older in-file resolver shape in `bot.ts` and does not include the current resolver file.

## What This Changes

The PR adds Feishu `senderUserId` context plumbing, changes sender display-name lookup to prefer `user_id` before `open_id` and fallback IDs, and adds Feishu bot tests for priority and fallback behavior.

## Best Possible Solution

Port the ordered lookup behavior onto the current split resolver: add optional `senderUserId` context, pass both IDs from parsing into `handleFeishuMessage`, implement per-candidate cache and API lookup in `bot-sender-name.ts`, and keep focused regression tests for priority, fallback, cache, and permission-error behavior.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: use an inbound Feishu event with both `sender_id.user_id` and `sender_id.open_id`; the parser sees both, but sender-name lookup receives only `ctx.senderOpenId`, so `user_id` cannot be attempted first.

## Solution Assessment

No. The direction is sound, but this branch is no longer the best landing shape because current main moved sender-name resolution into `bot-sender-name.ts`; the maintainable fix is to adapt the behavior to that resolver boundary.
