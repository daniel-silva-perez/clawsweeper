Keep open: the Feishu sender-name fix is still relevant, but this PR is not merge-ready against current main because the active resolver now lives in a split `bot-sender-name.ts` path and the branch does not update that boundary. The remaining work needs maintainer sequencing with adjacent Feishu display-name work rather than automatic closure.

## What This Changes

The PR adds Feishu `senderUserId` plumbing, changes sender display-name lookup to prefer `user_id` before `open_id` and fallback IDs, and adds Feishu bot tests for priority and fallback behavior.

## Best Possible Solution

Port the ordered Feishu sender lookup into the current resolver boundary: retain optional `senderUserId` in `FeishuMessageContext`, pass both IDs from `handleFeishuMessage`, implement per-candidate cache/API lookup in `bot-sender-name.ts`, and keep regression coverage for priority, fallback, cache, and permission-error behavior.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: send an inbound Feishu event with both `sender_id.user_id` and `sender_id.open_id`; current code reads both but calls the sender-name resolver with only `ctx.senderOpenId`, so `user_id` cannot be preferred.

## Solution Assessment

No. The proposed behavior is the right direction, but this branch is no longer the best landing shape because current main moved sender-name resolution into `bot-sender-name.ts`; the safer fix is to adapt the change to that resolver boundary.
