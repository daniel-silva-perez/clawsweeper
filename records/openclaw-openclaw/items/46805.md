## Summary

Keep open. Current main still does not implement the central Feishu sender-name behavior: inbound events can expose both `sender_id.user_id` and `sender_id.open_id`, but the resolver is still called with only the historical `senderOpenId` value and performs a single lookup. The PR remains useful, but it is now stale against main because sender-name resolution has been split into `bot-sender-name.ts`, so maintainers should rebase/adapt it before merge.

## What This Changes

The PR changes Feishu sender-name resolution to prefer `sender_id.user_id` before `sender_id.open_id`, adds `senderUserId` to the Feishu message context, and adds sender lookup priority, fallback, and cache regression tests.

## Best Possible Solution

Adapt the PR onto current main by adding `senderUserId` to the Feishu message context, moving the ordered candidate lookup into `extensions/feishu/src/bot-sender-name.ts`, and keeping the cache priority, fallback-after-error, client-construction, cross-ID cache, and cache-poisoning regression coverage focused in the Feishu plugin tests. No core/plugin SDK change appears necessary.
