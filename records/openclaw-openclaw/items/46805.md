## Summary

Keep open: current main still resolves Feishu sender names through a single ID path, and this PR contains useful source work but is stale against the current split sender-name resolver. The branch should be rebased and ported into the active resolver boundary, with maintainer coordination around the adjacent open Feishu display-name PR #73399.

## What This Changes

The PR adds Feishu `senderUserId` plumbing, changes sender display-name lookup to prefer `sender_id.user_id` before `open_id` and fallback IDs, and adds Feishu bot tests for priority and fallback behavior.

## Best Possible Solution

Port the ordered Feishu sender lookup into the current resolver boundary: retain optional `senderUserId` in `FeishuMessageContext`, pass both IDs from `handleFeishuMessage`, implement per-candidate cache/API lookup in `bot-sender-name.ts`, and keep regression coverage for priority, fallback, cache, and permission-error behavior while coordinating with #73399.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: send an inbound Feishu event with both `sender_id.user_id` and `sender_id.open_id`; current code reads both but calls the sender-name resolver with only `ctx.senderOpenId`, so `user_id` cannot be preferred.

## Solution Assessment

No. The requested behavior is the right direction, but this branch is no longer the best landing shape because current main moved sender-name resolution into `bot-sender-name.ts`; the safer fix is to adapt the change to that resolver boundary and reconcile it with #73399.
