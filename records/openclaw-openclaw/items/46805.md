## Summary

Keep open. The requested Feishu behavior is still missing on current main, and the PR contains relevant work, but the branch is dirty and targets the pre-split in-file resolver instead of the current `bot-sender-name.ts` implementation.

## What This Changes

The PR carries `senderUserId` through Feishu message context, changes sender display-name lookup to prefer `user_id` before `open_id` with fallback/cache handling, and adds Feishu regression tests.

## Best Possible Solution

Port the intended ordered sender-name lookup onto the current split resolver: add an optional `senderUserId` to `FeishuMessageContext`, pass both IDs from `parseFeishuMessageEvent` into `handleFeishuMessage`, implement ordered per-candidate lookup and cache behavior in `extensions/feishu/src/bot-sender-name.ts`, and keep focused Feishu regression tests for priority, fallback, cache, and permission behavior.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: create an inbound Feishu event with both `sender_id.user_id` and `sender_id.open_id`; current code parses both but calls the name resolver with only `ctx.senderOpenId`, so `user_id` cannot be attempted first.

## Solution Assessment

No. The intended direction is sound, but this branch is no longer the best landing shape because current main moved sender-name lookup into `bot-sender-name.ts`; the safer solution is to adapt the same behavior onto that resolver surface.
