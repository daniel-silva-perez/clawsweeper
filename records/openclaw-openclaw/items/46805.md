## Summary

Keep open. The underlying Feishu sender-name problem still exists on current main, and the review discussion shows useful focused work with cache/fallback issues already iterated on, but the PR branch now targets the pre-split in-file resolver while current main resolves sender names through `bot-sender-name.ts`. That makes the patch not merge-ready without rebase/adaptation.

## What This Changes

The PR carries `senderUserId` through the Feishu plugin message context, changes sender display-name lookup to try `user_id` before `open_id` and fallback IDs with cache/error handling, and adds Feishu tests for priority and fallback behavior.

## Best Possible Solution

Port the intended `user_id`-first sender-name lookup onto current main by adding an optional `senderUserId` to `FeishuMessageContext`, passing both IDs from `parseFeishuMessageEvent` into the handler, implementing ordered candidate lookup and cache semantics in `extensions/feishu/src/bot-sender-name.ts`, and keeping regression coverage focused on Feishu sender-name priority, fallback, cache, and permission behavior.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: an inbound event with both `sender_id.user_id` and `sender_id.open_id` is parsed, but the handler calls `resolveFeishuSenderName` with only `ctx.senderOpenId`, so the resolver cannot attempt `user_id` first.

## Solution Assessment

No. The intended direction is maintainable, but this branch is no longer the best way to land it because current main moved sender-name lookup into `bot-sender-name.ts`; the safer path is to adapt the same behavior onto that split resolver surface.
