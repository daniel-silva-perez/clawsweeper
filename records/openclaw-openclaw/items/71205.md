## Summary

Keep open. Current main still has no `OPENCLAW_SKIP_STATE_DIR_HARDEN` escape hatch, and config writes still call the state-dir hardening helper that chmods group/world-accessible state dirs to `0o700`. The PR is a narrow implementation for a plausible documented Fly/shared-state deployment surface, with no blocking code-review finding from this read-only pass, but it still needs normal maintainer/security/docs review and fresh validation before merge.

## What This Changes

The PR adds a truthy `OPENCLAW_SKIP_STATE_DIR_HARDEN` guard with accepted-env logging before config state-dir chmod hardening, adds a POSIX regression test preserving a setgid `0o2770` state directory, and records an Unreleased changelog entry.

## Best Possible Solution

Land a maintainer-approved version if OpenClaw wants this operational exception: keep default state-dir hardening unchanged, allow the skip only through trusted operator environment, log accepted use, document the multi-uid shared-volume case, and retain tests for both default `0o700` hardening and setgid preservation with the opt-out enabled.

## Reproduction Assessment

Yes. The current-main path is reproducible by creating a default state/config dir with group or other permission bits and calling `writeConfigFile`, which invokes the helper that chmods it to `0o700`; the PR's proposed test mirrors the multi-uid `0o2770` case with the skip env set.

## Solution Assessment

Yes for the code shape: an opt-in guard before stat/chmod is the narrowest change that preserves current defaults and fixes the reported shared-volume case. The remaining question is product/security policy and documentation, not a clearer implementation bug in the diff.
