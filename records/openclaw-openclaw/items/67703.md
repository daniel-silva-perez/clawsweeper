## Summary

Keep open. Current main still matches the reported failure shape: local probe auth resolves no shared secret under `auth.mode="none"`, `probeGateway` sends no device identity for a fresh diagnostic probe, `method: "none"` is not treated as shared auth, and the non-Control-UI CLI probe can still fall through to `1008: device identity required`. The related #69066 RFC is relevant design context, but it remains open and unshipped.

## What This Changes

Allow Gateway-owned internal RPC probes and exec approval followup dispatches to keep working when `gateway.auth.mode` is `none` instead of being rejected for missing device identity.

## Best Possible Solution

Add a narrowly scoped internal-service identity path, or an equivalently constrained loopback-only Gateway-owned caller exception, for internal probe/followup/backend RPC clients under `gateway.auth.mode="none"`. Keep device identity fail-closed for remote clients, browser-origin clients, node-role clients, and non-internal operator clients, with regression coverage for the probe and approval/followup path.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="none"`, no token/password, and a fresh CLI with no cached operator device token follows a clear source path: local probe auth resolves empty, `probeGateway` disables device identity, and the non-Control-UI operator connection reaches the 1008 `device identity required` branch.

## Solution Assessment

No current fix is present on main. The token-mode workaround is operationally valid, but the maintainable fix should be an internal-only service identity or tightly scoped internal caller path rather than widening `auth.mode="none"` globally.
