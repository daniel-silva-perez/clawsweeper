## Summary

Keep this issue open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding: the CLI and onboarding option types expose token/password only, and the non-interactive gateway config path rejects other modes. Runtime config and the interactive gateway flow already support `trusted-proxy`, so this remains a real core onboarding gap rather than plugin work. Related same-author PR #49107 and issue #73639 are implementation/tracking candidates, not shipped fixes.

## Best Possible Solution

Keep this issue open and implement the feature in core onboarding. Accept `--gateway-auth trusted-proxy`, add explicit non-interactive flags for `gateway.trustedProxies` and `gateway.auth.trustedProxy.*` including `userHeader`, `requiredHeaders`, `allowUsers`, and an intentional `allowLoopback` choice, account for non-loopback Control UI allowed origins, update CLI docs, and add focused tests for successful config writes plus invalid/missing required fields. Review or replace PR #49107 as the implementation candidate.

## Risks / Open Questions

- A fix is security-sensitive: it should validate proxy IPs, user identity headers, required headers, allowUsers, allowLoopback, Control UI allowed origins, and clear mutually exclusive token/password fields when switching to trusted-proxy.
