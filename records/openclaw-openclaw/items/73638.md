## Summary

Keep this issue open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding, while runtime config, docs, and the interactive gateway flow already support trusted-proxy auth. The related same-author draft PR #49107 is an implementation candidate, not a merged fix; duplicate-looking issue #73639 should not make this older tracking issue close before the feature lands.

## Best Possible Solution

Keep this issue open and implement the feature in core onboarding, preferably by reviewing or replacing draft PR #49107. The fix should accept `--gateway-auth trusted-proxy`, add explicit non-interactive flags for `gateway.trustedProxies` and `gateway.auth.trustedProxy.*`, handle Control UI allowed origins for non-loopback deployments, update CLI docs, and add focused tests for valid writes plus invalid/missing security fields.

## Risks / Open Questions

- Closing this now would leave scripted reverse-proxy deployments without a supported non-interactive path for a runtime-supported gateway auth mode.
- The eventual implementation is security-sensitive: it should validate trusted proxy IPs, identity and required headers, allowUsers, allowLoopback, non-loopback Control UI allowed origins, and clear mutually exclusive token/password fields when switching to trusted-proxy.
