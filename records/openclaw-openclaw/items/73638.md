## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though the runtime config and interactive gateway flow support it; same-author draft PR #49107 remains the open implementation candidate, while #73639 is a newer duplicate.

## What This Changes

Add non-interactive onboarding support for selecting and configuring trusted-proxy gateway auth for scripted reverse-proxy deployments.

## Best Possible Solution

Implement the feature in core onboarding by landing or replacing #49107: accept `--gateway-auth trusted-proxy`, add explicit flags for trusted proxy IPs, user header, required headers, allowed users, loopback opt-in, and Control UI allowed origins, clear incompatible token state when switching modes, and update focused tests plus the public onboarding docs.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: the CLI forwards `gatewayAuth`, `OnboardOptions` excludes `trusted-proxy`, and `applyNonInteractiveGatewayConfig` deterministically exits for anything other than `token` or `password`.

## Solution Assessment

Yes, the requested direction is the right surface: trusted-proxy auth is already a core gateway runtime and interactive onboarding mode, so non-interactive parity belongs in core onboarding rather than a plugin or manual config editing workaround.
