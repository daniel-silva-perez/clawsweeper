## Summary

Keep open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding even though trusted-proxy is a supported core runtime mode and the interactive gateway flow can configure it; #49107 is an open draft implementation candidate and #73639 appears to be a newer duplicate rather than the canonical tracker.

## What This Changes

Add non-interactive onboarding CLI support for configuring `trusted-proxy` gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Implement narrow core onboarding parity for the existing runtime mode: accept `--gateway-auth trusted-proxy`, add explicit non-interactive flags for trusted proxy IPs, identity header, required headers, allowUsers, allowLoopback, and Control UI allowed origins, clear incompatible token/password state when switching modes, then update focused tests and public docs.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig` and the current guard exits with `Invalid --gateway-auth (use token|password).`

## Solution Assessment

Yes on the product boundary: this belongs in core onboarding because trusted-proxy is already a core gateway runtime mode, not a plugin-only integration. The exact branch should still be reviewed for security-sensitive defaults and compatibility with existing token/password onboarding behavior.
