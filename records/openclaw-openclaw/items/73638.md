## Summary

Keep open. Current main still lacks the requested non-interactive trusted-proxy onboarding path: the CLI, option types, and non-interactive config helper only handle token/password, while runtime and interactive configuration already support trusted-proxy auth. The open same-author draft PR #49107 is the active implementation candidate, so this should stay open for maintainer/security review rather than be closed or auto-repaired.

## What This Changes

The issue requests scripted `openclaw onboard --non-interactive` support for selecting and configuring `gateway.auth.mode: "trusted-proxy"` for reverse-proxy deployments.

## Best Possible Solution

Review, land, or replace PR #49107 with a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, adds explicit typed flags for the trusted-proxy fields, clears incompatible shared-token state, preserves fail-closed runtime defaults, documents the scripted deployment path, and adds focused tests for valid and unsafe inputs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits because only token and password are accepted.

## Solution Assessment

Yes for the requested direction. Core non-interactive onboarding is the right surface because trusted-proxy is already a core gateway auth mode and interactive onboarding can configure it; the exact patch still needs security-focused maintainer review.
