## Summary

Keep open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding even though trusted-proxy is a supported core gateway runtime mode and the interactive gateway flow can configure it. The same-author draft PR #49107 is an open implementation candidate, while #73639 appears to be a newer duplicate, so this issue should remain the canonical tracker until a fix merges.

## What This Changes

The issue requests `openclaw onboard --non-interactive` support for selecting `trusted-proxy` gateway auth and writing the related reverse-proxy security fields for scripted deployments.

## Best Possible Solution

Implement the feature in core onboarding by landing or replacing #49107: accept `--gateway-auth trusted-proxy`, add explicit non-interactive flags for `gateway.trustedProxies`, `gateway.auth.trustedProxy.userHeader`, required headers, `allowUsers`, `allowLoopback`, and `gateway.controlUi.allowedOrigins`, clear incompatible token/password state when switching modes, and update focused tests plus public docs.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `register.onboard.ts` passes the raw `--gateway-auth` value, `OnboardOptions` excludes `trusted-proxy`, and `applyNonInteractiveGatewayConfig` deterministically exits unless the value is `token` or `password`.

## Solution Assessment

Unclear for the existing draft PR, but the issue direction is correct: this belongs in core onboarding rather than a plugin because trusted-proxy auth is already a core gateway mode. The safest solution is a narrow onboarding change that mirrors runtime fail-closed requirements and interactive setup, with explicit security-oriented flags and regression tests.
