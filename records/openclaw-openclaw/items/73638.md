## Summary

Keep open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding even though runtime and interactive gateway setup support trusted-proxy auth. The issue also has a same-author draft implementation candidate in #49107, so the right next step is maintainer/security review of that path rather than cleanup closure.

## What This Changes

Add a non-interactive `openclaw onboard` path for configuring trusted-proxy gateway auth in scripted reverse-proxy deployments.

## Best Possible Solution

Review, land, or replace #49107 with a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, adds typed flags for trusted proxy source and identity settings, clears incompatible token/password state, keeps trusted-proxy runtime defaults fail-closed, updates onboarding docs, and adds focused non-interactive tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is calling `applyNonInteractiveGatewayConfig` with `opts.gatewayAuth = "trusted-proxy"`, or running the equivalent non-interactive onboarding path; current main exits at the token/password-only validation.

## Solution Assessment

Yes for the requested direction. Trusted-proxy is already a core gateway auth mode and interactive setup supports it, so core non-interactive onboarding is the narrow maintainable surface; the exact flags and defaults need security review.
