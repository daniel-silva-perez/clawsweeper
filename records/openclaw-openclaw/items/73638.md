## Summary

Keep this issue open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding, even though runtime config, docs, and the interactive gateway setup path already support trusted-proxy auth. The same-author draft PR #49107 is an implementation candidate, not a merged fix, and the duplicate-looking #73639 should not close this older tracking issue before the feature lands.

## Best Possible Solution

Keep this issue open and implement the feature in core onboarding, preferably by reviewing or replacing draft PR #49107. The fix should accept `--gateway-auth trusted-proxy`, add explicit non-interactive flags for `gateway.trustedProxies` and `gateway.auth.trustedProxy.*`, support Control UI allowed origins for reverse-proxy deployments, update `docs.openclaw.ai/cli/onboard`, and add focused tests for valid config writes plus invalid or missing security fields. Treat #73639 as the newer duplicate rather than closing this issue.

## Risks / Open Questions

- Closing now would leave scripted reverse-proxy deployments without a supported non-interactive path for a runtime-supported gateway auth mode.
- The eventual implementation is security-sensitive: it should validate trusted proxy IPs, identity and required headers, allowUsers, allowLoopback, Control UI allowed origins for non-loopback deployments, and cleanup of mutually exclusive token/password fields.
