## Summary

Keep open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding even though runtime, schema, docs, and interactive configuration support trusted-proxy auth; the same-author draft PR #49107 is the active implementation candidate, so this should remain open for maintainer/security review rather than be closed or queued as a separate repair.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring gateway `trusted-proxy` auth in scripted reverse-proxy deployments.

## Best Possible Solution

Land or replace PR #49107 with a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, adds explicit typed flags for the trusted-proxy fields, clears incompatible shared-token state, keeps unsafe trust choices explicit, updates docs, and adds focused tests for valid and unsafe inputs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits because only token and password are accepted.

## Solution Assessment

Yes for the requested direction. Core non-interactive onboarding is the correct surface because trusted-proxy is already a core gateway auth mode and interactive onboarding can configure it; the exact patch still needs security-focused maintainer review.
