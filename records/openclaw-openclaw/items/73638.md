## Summary

Keep open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding, while the runtime config schema, docs, and interactive gateway setup support trusted-proxy auth. The same-author draft PR #49107 remains an open implementation candidate, so this issue should stay open until a fix lands or maintainers choose a different design.

## Best Possible Solution

Keep this issue open and implement the feature in core onboarding, preferably by reviewing or replacing draft PR #49107. The fix should accept `--gateway-auth trusted-proxy`, add explicit non-interactive flags for `gateway.trustedProxies` and `gateway.auth.trustedProxy.*`, handle Control UI allowed origins for non-loopback deployments, update CLI docs, and add focused tests for valid writes plus invalid or missing security fields.

## Risks / Open Questions

- Closing now would leave scripted reverse-proxy deployments without a supported non-interactive path for a runtime-supported gateway auth mode.
- The eventual fix is security-sensitive and should validate trusted proxy IPs, identity and required headers, allowUsers, allowLoopback, non-loopback Control UI allowed origins, and stale token/password fields when switching modes.
