## Summary

Keep open. Current main still lacks the requested non-interactive trusted-proxy onboarding surface, while runtime and interactive configuration already support trusted-proxy auth. The issue is also paired with open same-author draft PR #49107, so the best maintainer action is to review or replace that implementation rather than close or queue a separate repair.

## What This Changes

The issue requests non-interactive `openclaw onboard` support for selecting and configuring trusted-proxy gateway authentication.

## Best Possible Solution

Review, land, or replace PR #49107 with a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, adds explicit typed flags for trusted proxy source and identity settings, prevents incompatible token/password state from being left behind, preserves fail-closed trusted-proxy runtime defaults, updates onboarding docs, and adds focused non-interactive tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is calling `applyNonInteractiveGatewayConfig` with `opts.gatewayAuth = "trusted-proxy"`, or running the equivalent non-interactive onboarding command path; current main exits at the token/password-only validation.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy is already a core gateway auth mode and interactive setup can configure it; the implementation should stay narrow and security-reviewed rather than moving to a plugin.
