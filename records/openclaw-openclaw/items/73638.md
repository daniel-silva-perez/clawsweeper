## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though runtime and interactive gateway configuration support that auth mode, and the same author has an open draft implementation candidate in #49107.

## What This Changes

Add a non-interactive onboarding path for configuring `gateway.auth.mode="trusted-proxy"` and its trusted proxy settings from CLI flags.

## Best Possible Solution

Keep this issue open until PR #49107 is reviewed, landed, or replaced with a narrow core onboarding change that adds typed non-interactive trusted-proxy flags, updates docs, and adds focused tests without weakening gateway auth defaults.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass `opts.gatewayAuth = "trusted-proxy"` through `applyNonInteractiveGatewayConfig`; current main exits at the token/password-only validation, matching the equivalent non-interactive CLI path.

## Solution Assessment

Yes for the requested direction. Core onboarding is the right surface because trusted-proxy auth is already a core gateway auth mode and interactive setup supports it; the safer implementation path is a narrow, security-reviewed CLI/docs/test change rather than a plugin.
