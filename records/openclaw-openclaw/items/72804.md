## Summary

Current main already solves the reported workaround: `browser.extraArgs` is a validated config surface, explicit Chrome proxy flags are appended to managed Chrome launch args, and `--no-proxy-server` is omitted when a proxy-control arg is present. The same behavior and docs are present in release `v2026.4.25`, matching the related #8079 cleanup conclusion.

## What This Changes

Requests a user-facing browser proxy configuration path so managed Chrome can use a proxy without manually editing compiled JavaScript to remove `--no-proxy-server` after installs.

## Best Possible Solution

Keep the existing explicit Chrome-flag model: users who need a browser proxy should set `browser.extraArgs` with `--proxy-server=...` or `--proxy-pac-url=...`, while OpenClaw preserves the default direct-launch and SSRF-sensitive ambient-env behavior.

## Reproduction Assessment

Yes for the observable workaround. A config object with `browser.extraArgs: ["--proxy-server=..."]` validates, and the launch-arg helper appends that flag while suppressing the default `--no-proxy-server`.

## Solution Assessment

Yes. The existing `browser.extraArgs` surface is the narrow maintainable solution because it uses Chromium’s native proxy flags and avoids making provider `HTTP_PROXY`/`NO_PROXY` environment variables silently change managed-browser SSRF posture.
