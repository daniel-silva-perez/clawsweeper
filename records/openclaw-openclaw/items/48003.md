## Summary

Keep this open. Current main still matches the reported failure: steer injection is guarded by streaming state, active non-streaming Pi/reply handles reject queued steer messages, and Discord same-session inbound work still serializes behind the active run before it can reach the reply pipeline. The active path is to review or extend the open implementation PR rather than close this issue now.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` does not inject same-session Discord/main-session messages into an active turn at tool boundaries and instead queues them until the turn completes.

## Best Possible Solution

Use #73365 as the canonical implementation path and make the final fix cover both halves of the regression: lifecycle-aware steering for active non-streaming runs, plus a Discord delivery path that lets steer-mode same-session inbound messages reach the steering queue before the per-session run queue blocks them. Add focused tests for Pi/reply queue acceptance, `runReplyAgent` steer gating, and Discord same-session steer bypass behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with `messages.queue.mode: "steer"`, send a Discord message during an active main-session tool-execution window; current main either keeps the inbound job behind the same session queue or, if it reaches reply handling while active but non-streaming, rejects steering via the `isStreaming` checks.

## Solution Assessment

Unclear whether the currently open PR is sufficient until maintainers verify it covers both the lifecycle gate and the Discord delivery path. The narrow maintainable direction is to preserve run serialization for normal messages while allowing only steer-mode messages targeting an active run to bypass or reach the steering path safely.
