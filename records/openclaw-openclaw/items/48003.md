## Summary

Keep open. Current main still matches the reported failure path: steering only runs when the active run is streaming, active non-streaming Pi/reply handles reject queued steer messages, and Discord same-session inbound work still enters a per-session serialized run queue before it can reach the reply pipeline. The active path is maintainer review of the open implementation PR rather than closing this issue.

## What This Changes

The issue requests that `messages.queue.mode: "steer"` inject same-session Discord/main-session messages into an active agent turn at tool boundaries instead of waiting for the current turn to finish.

## Best Possible Solution

Use #73365 as the canonical implementation path and make the final fix cover both halves of the regression: lifecycle-aware steer acceptance for active non-streaming Pi/reply runs, plus a Discord delivery path that lets steer-mode same-session inbound messages reach the steering queue without waiting behind the active run. Add focused tests for Pi/reply queue acceptance, `runReplyAgent` steer gating, and Discord same-session steer bypass behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with `messages.queue.mode: "steer"`, send a Discord message during an active main-session tool-execution window; current main either holds the inbound job behind the same session queue or rejects steering through the streaming-only guards when the run is active but not streaming.

## Solution Assessment

Unclear whether the currently open implementation PR is sufficient until maintainers verify it covers both the lifecycle gate and Discord delivery path. The narrow maintainable direction is to preserve normal run serialization while allowing only steer-mode messages targeting an active run to reach the steering path safely.
