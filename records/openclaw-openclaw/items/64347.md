## Summary

Keep open. Current main still has no native warm-reuse path for `sessions_spawn`: native subagent spawning creates a fresh UUID-backed child session key, the tool/config surfaces expose no native reuse key or pool selector, and docs/tests explicitly keep `resumeSessionId` ACP-only. Related merged PRs #64468 and #72331 addressed adjacent bootstrap-marker and ACP-field behavior, not this native subagent reuse contract.

## What This Changes

Add native warm-reuse, pooling, or stable session-key support for `sessions_spawn` so repeated delegations can avoid creating a fresh UUID-backed cold subagent session every time.

## Best Possible Solution

Define an explicit native subagent reuse contract first, then implement it across the tool schema, config/types, spawn key selection, docs, and focused tests. The end state should make the isolation/history tradeoff explicit and preserve safe defaults while allowing deliberate warm reuse for repeated delegated work.

## Risks / Open Questions

- The correct native reuse semantics are still undefined: opt-in stable key, automatic pooling, main-session routing, thread-bound reuse, and archive cleanup each have different isolation and history-sharing tradeoffs.
- Automatic pooling would need concurrency/liveness locking so a reused session is not selected while another subagent run is still active.
- A reuse contract could accidentally mix sensitive transcript history across delegated tasks if it is not explicit about isolation, requester identity, target agent, and cleanup boundaries.
