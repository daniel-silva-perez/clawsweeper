## Summary

Keep open. Current main still creates native subagent sessions with fresh UUID-backed keys and keeps resume behavior ACP-only; no tool, config, docs, or tests expose native warm reuse, pooling, or a reuse key. Related merged PRs #64468 and #72331 fixed adjacent cache-marker and ACP-field behavior, but they do not implement this requested native reuse contract.

## What This Changes

The issue requests a warm-reuse or pooling path for native `sessions_spawn` subagent sessions so repeated delegation to the same target agent can avoid repeated cold-start context/cache cost.

## Best Possible Solution

Define an explicit native subagent reuse contract before implementation: choose opt-in reuse versus automatic pooling, model isolation/history tradeoffs, add schema/types/docs around the chosen API, and cover native `runtime="subagent"` behavior with focused unit tests around spawn key selection, concurrency/liveness, cleanup, and context preparation.

## Risks / Open Questions

- A naive native reuse key could mix unrelated child histories unless the contract is explicit, scoped, and documented.
- Automatic pooling needs allocation and locking semantics so concurrent spawns do not reuse an active child session unsafely.
- Archive cleanup, thread-bound session mode, follow-up routing, and context-engine preparation need defined behavior for reused native child sessions.
