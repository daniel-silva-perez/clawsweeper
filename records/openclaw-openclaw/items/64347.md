## Summary

Keep open. Current main still creates native subagent sessions with fresh UUID-backed keys, keeps resume behavior ACP-only, and exposes no native reuse key, warm pool, or stable child-session selector in tool, config, docs, or tests. The related merged PRs #64468 and #72331 fixed adjacent bootstrap-marker and ACP-field behavior, but not this native warm-reuse contract.

## What This Changes

Add native warm-reuse, pooling, or stable session-key support for `sessions_spawn` so repeated delegations can avoid a fresh cold subagent session each time.

## Best Possible Solution

Define an explicit native subagent reuse contract, then implement it through the tool schema, config/types, child-session key selection, docs, and focused tests. The desired end state should keep isolated fresh sessions as the safe default while allowing deliberate warm reuse with clear history-sharing semantics.

## Risks / Open Questions

- The correct native reuse semantics are still undefined: opt-in stable key, automatic pooling, main-session routing, and thread-bound reuse each carry different isolation and history-sharing tradeoffs.
- Automatic pooling would need concurrency and liveness locking so an active child session is not selected for another run.
- A reuse contract could mix transcript history across delegated tasks if requester identity, target agent, cleanup, and isolation boundaries are not explicit.
