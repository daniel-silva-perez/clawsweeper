## Summary

Keep open. Current main still limits Telegram configured ACP bindings to group/forum-topic conversation IDs: the Telegram adapter parses only topic-shaped IDs and rejects non-negative chat IDs, while the docs still say Telegram persistent ACP bindings are scoped to forum topics. The report is concrete, current, and narrow enough for a focused fix PR.

## What This Changes

Support top-level ACP configured bindings for Telegram direct-message peer IDs so DMs can create or resume persistent ACP harness sessions instead of falling through to the native runtime.

## Best Possible Solution

Teach the Telegram configured-binding adapter to accept canonical positive DM peer IDs as direct conversations while preserving the existing negative group/forum-topic path, then document the Telegram DM binding shape and add regression coverage for compile-time and inbound matching.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure a Telegram ACP binding with `match.peer.kind: "direct"` and a positive numeric peer id, then the current adapter returns null during configured-binding compilation because the id is not a topic and has no parent conversation.

## Solution Assessment

Yes. The narrowest maintainable fix is Telegram-owned: adjust `extensions/telegram/src/channel.ts` binding normalization/matching and add Telegram tests/docs, without changing core ACP persistence semantics or the shared registry unless the existing adapter contract proves insufficient.
