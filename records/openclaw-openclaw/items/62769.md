## Summary

Keep open. Current main still filters Telegram ACP configured bindings through a topic-only adapter that rejects direct-message peer ids, while the schema accepts the reported `peer.kind: "direct"` binding shape. The related #70177 work fixed a separate Claude CLI transcript-resume failure, but it does not add configured ACP routing for Telegram DMs.

## What This Changes

Support top-level `bindings[]` entries with `type: "acp"` for Telegram direct-message conversations so dedicated Telegram bot DMs can create or resume persistent ACP sessions.

## Best Possible Solution

Add direct-message support in the Telegram plugin’s configured ACP binding adapter, keeping the generic configured-binding registry unchanged, preserving existing forum-topic behavior, and documenting whether Telegram DM bindings should use the sender user id or chat id.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the reported direct peer id reaches `resolveConfiguredBindingRoute()` as a DM `conversationId`, then the Telegram adapter returns null because it only accepts topic parses with negative group chat ids.

## Solution Assessment

Yes. The narrowest maintainable solution is a Telegram plugin change that recognizes direct peer ids for `peer.kind: "direct"`, plus targeted Telegram ACP binding tests and docs; a generic ACP/runtime rewrite is not needed.
