## Summary

Keep this PR open. The underlying config-validation bug is still present on current main, and the PR is a useful narrow direction, but the branch is stale against the current validation pipeline and its backfill currently masks malformed explicit provider values that the schema should reject.

## What This Changes

The PR adds pre-schema config normalization to infer missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids and adds regression tests for inferred and explicit auth-profile providers.

## Best Possible Solution

Keep the pre-schema normalization approach, but implement it in current main after deprecated-key stripping and before legacy/schema validation; infer only omitted or blank-string providers from well-formed profile ids, preserve malformed explicit providers for Zod rejection, and cover omitted, blank, explicit, malformed, and malformed-id cases with focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: current main sends a profile such as `{ auth: { profiles: { "qwen-portal:default": { mode: "oauth" } } } }` into a schema that requires `provider`, so validation still fails before any provider-id inference exists.

## Solution Assessment

No, not as currently proposed. The direction is the narrow maintainable fix, but the PR needs to be rebased into the current validation flow and changed so non-string explicit provider values are not overwritten before schema validation.
