## Summary

Keep this PR open: it targets a real current-main gap, but the proposed normalization currently masks malformed explicit provider values that the schema should reject, and the branch needs adaptation to the newer config validation pipeline before it can merge.

## What This Changes

The PR adds pre-schema config normalization that infers missing `auth.profiles[*].provider` values from `<provider>:<name>` profile keys and adds regression tests for inferred and explicit providers.

## Best Possible Solution

Rebase the fix onto current main as a narrow normalization step after deprecated-key stripping and before legacy/schema validation, fill only absent or blank string providers from the profile id, preserve malformed explicit provider values for Zod to reject, and add a regression test for the non-string explicit-provider case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve invalid provider values for schema validation:** `src/config/validation.ts:250`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then writes an inferred provider from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would pass even though current schema rejects non-string providers; only fill when `provider` is missing or a blank string.
  - confidence: 0.88
