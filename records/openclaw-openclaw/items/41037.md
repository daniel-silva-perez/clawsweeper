## Summary

Keep this PR open. Current main still has the reported config-validation gap, and the PR is a useful narrow implementation for #38913, but the proposed backfill currently masks malformed explicit `provider` values and the branch is marked dirty against main, so it needs author or maintainer follow-up before merge.

## What This Changes

The PR adds pre-schema config normalization that infers missing `auth.profiles[*].provider` values from `<provider>:<name>` profile keys and adds regression coverage for inference and explicit-provider precedence.

## Best Possible Solution

Land the narrow normalization in current `src/config/validation.ts` after deprecated-key stripping and before legacy/schema validation, but only backfill when `provider` is missing or an intentionally blank string. Preserve explicit valid providers, leave invalid explicit provider types for Zod to reject, and keep focused regression coverage in `src/config/config.schema-regressions.test.ts`.

## Review Findings

  - body: The backfill treats every non-string `profileValue.provider` as absent and overwrites it from the profile id.
