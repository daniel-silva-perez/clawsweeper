## Summary

Keep this PR open. Current main and the latest release still lack auth-profile provider inference, so the underlying bug remains valid, but the proposed patch should not merge as-is because it can overwrite malformed explicit provider values before schema validation rejects them.

## What This Changes

The PR adds pre-schema config normalization that infers missing `auth.profiles[*].provider` values from `<provider>:<name>` profile keys and adds regression tests for inferred and explicit-provider paths.

## Best Possible Solution

Land the narrow normalization in `src/config/validation.ts` after deprecated-key stripping and before legacy/schema validation, but only backfill when `provider` is absent or a deliberately blank string. Preserve explicit invalid provider values for Zod to reject, add a regression test for non-string `provider`, then rebase and run the targeted config test, formatter check, and changed gate in Testbox.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve invalid provider values for schema validation:** `src/config/validation.ts:247-248`
  - body: The backfill treats every non-string `profileValue.provider` as absent and overwrites it from the profile id. That makes malformed config like `{ provider: 123, mode: "oauth" }` validate successfully when today's schema rejects it. Only backfill when `provider` is missing or an intentionally blank string; leave other explicit values for Zod to reject.
  - confidence: 0.87
