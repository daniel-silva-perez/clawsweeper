## Summary

Keep this PR open, but it is not merge-ready. The underlying config bug is still real on current main, while the PR as written can mask malformed explicit provider values and needs rebasing onto the current validation pipeline.

## What This Changes

The PR adds pre-schema config normalization to infer missing auth profile provider values from `<provider>:<name>` keys and adds regression tests for inferred and explicit provider behavior.

## Best Possible Solution

Keep the normalization approach, but integrate it after deprecated-key stripping in the current validation flow, infer only omitted or blank-string providers from the profile id, and leave malformed explicit provider values untouched for schema validation to reject.

## Reproduction Assessment

Yes. The linked report gives a concrete config shape, and current main still sends that shape to a schema that requires `auth.profiles[*].provider` to be a string.

## Solution Assessment

No, not as currently proposed. The general fix direction is narrow and maintainable, but the implementation should preserve invalid explicit values for Zod and be rebased onto current `validateConfigObjectRaw()` structure.

## Review Findings
