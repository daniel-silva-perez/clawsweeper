## Summary

Keep open. Current main still lacks auth-profile provider backfill, so the PR addresses a real config validation gap linked from #38913, but the proposed patch has a blocking correctness bug: it overwrites non-string explicit provider values before Zod can reject them. The branch is also stale against the current validation pipeline.

## What This Changes

The PR adds pre-schema config normalization that infers missing `auth.profiles[*].provider` values from `<provider>:<name>` profile keys and adds regression tests for inferred and explicit providers.

## Best Possible Solution

Rebase the normalization onto current main after deprecated-key stripping and before legacy/schema validation, infer only absent or blank string providers from the profile id, preserve malformed explicit provider values for Zod to reject, and add regression coverage for the non-string explicit-provider case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve invalid provider values for schema validation:** `src/config/validation.ts:250`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then writes an inferred provider from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would pass even though the current schema rejects non-string providers; only fill when `provider` is missing or a blank string.
