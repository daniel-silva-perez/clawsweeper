## Summary

Keep this PR open. Current main still rejects auth profiles that omit `provider`, so the PR is not obsolete, but the proposed patch should be rebased and adjusted so malformed explicit provider values still fail validation.

## What This Changes

The PR adds a pre-schema-validation normalization step that backfills missing `auth.profiles[*].provider` values from `<provider>:<name>` profile IDs and adds regression tests for inference and explicit-provider precedence.

## Best Possible Solution

Land the narrow normalization in current `src/config/validation.ts` after deprecated-key stripping and before legacy/schema validation, preserve explicit valid providers, let invalid explicit provider types continue to fail Zod validation, and keep focused regression coverage in `src/config/config.schema-regressions.test.ts`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve invalid provider values for schema validation:** `src/config/validation.ts:247-248`
  - body: The backfill treats every non-string `profileValue.provider` as absent and overwrites it from the profile id. That makes a malformed config like `{ provider: 123, mode: "oauth" }` validate successfully when today's schema rejects it. Only backfill when `provider` is missing or an intentionally blank string; leave other explicit values for Zod to reject.
  - confidence: 0.87
