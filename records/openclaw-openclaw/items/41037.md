## Summary

Keep open. Current main still lacks auth-profile provider inference, so the user-facing bug remains, but this PR should not merge as-is because its backfill overwrites malformed explicit provider values before Zod can reject them and it needs to be adapted to the current validation pipeline.

## What This Changes

The PR adds pre-schema config normalization to infer missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids and adds config schema regression tests for inferred and explicit providers.

## Best Possible Solution

Keep the pre-schema normalization approach, but apply it in the current validation pipeline after deprecated-key stripping and before legacy/schema validation; infer only omitted or blank-string providers from well-formed profile ids, preserve malformed explicit providers for Zod, and cover omitted, blank, explicit, and invalid-provider cases with focused regression tests.

## Reproduction Assessment

Yes. Source inspection is enough: current main validates `auth.profiles` against a schema that requires `provider` before any provider-id inference exists, so `{ auth: { profiles: { "qwen-portal:default": { mode: "oauth" } } } }` still fails validation.

## Solution Assessment

No, not as currently proposed. The direction is the narrow maintainable fix, but the implementation should preserve non-string explicit provider values for schema rejection and be rebased into the current validation flow.
