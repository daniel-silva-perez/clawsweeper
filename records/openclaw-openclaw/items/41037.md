## Summary

Keep open. The linked config bug still exists on current main, and this PR is a useful narrow fix direction, but the proposed patch should not merge as-is because it masks invalid explicit provider values and needs adaptation to the current validation flow.

## What This Changes

The PR adds pre-schema config normalization to infer missing `auth.profiles[*].provider` values from `<provider>:<name>` profile ids and adds regression tests for inferred and explicit provider behavior.

## Best Possible Solution

Keep the pre-schema normalization approach, but apply it to the current validation pipeline after deprecated-key stripping, infer only omitted or blank-string providers from well-formed profile ids, preserve malformed explicit provider values for schema validation, and cover omitted, blank, explicit, and invalid-provider cases with focused regression tests.

## Reproduction Assessment

Yes. The linked report gives a concrete config shape, and current main still validates that shape against a schema that requires `auth.profiles[*].provider` to be a string before any provider inference exists.

## Solution Assessment

No, not as currently proposed. The direction is the narrow maintainable fix, but the implementation should preserve invalid explicit provider values for Zod and be rebased onto the current `validateConfigObjectRaw()` structure.
