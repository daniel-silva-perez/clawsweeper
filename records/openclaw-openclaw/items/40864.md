## Summary

Keep open. Current main has a working `sessions_send` A2A path and a separate durable session-delivery queue, but the requested persistent at-least-once inbox for `sessions_send` is not wired: timeouts still return as timeouts, announce delivery failures are logged, and the durable queue is currently used for restart-continuation recovery rather than ordinary inter-session sends.

## What This Changes

The issue requests a persistent per-agent message queue for `sessions_send` so agent-to-agent messages survive timeouts, failures, restarts, and duplicate delivery risk.

## Best Possible Solution

Add a deliberate core design for reliable A2A delivery: persist accepted `sessions_send` messages before dispatch, retry or drain them on startup/heartbeat, expose/record stable idempotency and acknowledgement state, and cover timeout/restart/duplicate cases in the session-tool and gateway tests plus docs.
