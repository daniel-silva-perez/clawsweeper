## Summary

Keep open. Current main still accepts and dispatches `sessions_send` through the gateway with process-local dedupe and A2A follow-up handling, but I did not find the requested durable per-agent inbox, acknowledgement, or retry/recovery path for failed or timed-out `sessions_send` delivery. The existing durable session-delivery queue is wired to restart continuation recovery, not to cross-session message injection.

## Best Possible Solution

Keep this issue open as a core session-tool reliability design item. The likely fix belongs around `sessions_send`/gateway `agent` dispatch: persist an accepted delivery record before dispatch, use stable sender-provided idempotency keys, add acknowledgement or terminal failure state, drain/retry on startup or heartbeat, and add regression tests for timeout and gateway restart around cross-session sends. If maintainers prefer plugin participation, first add an explicit core extension point for durable cross-session send acknowledgement; the current plugin/ClawHub surfaces alone do not satisfy this guarantee.

## Risks / Open Questions

- Existing restart-continuation and outbound delivery queues are easy to confuse with the requested A2A inbox, but they cover different delivery surfaces.
- A real fix needs careful idempotency, replay, and policy design so recovery does not duplicate agent turns or bypass cross-agent visibility rules.
