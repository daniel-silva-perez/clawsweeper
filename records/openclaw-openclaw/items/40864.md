## Summary

Keep open. Current main still dispatches ordinary `sessions_send` directly through the Gateway `agent` method and reports timeout/error without durable queue, acknowledgement, or retry state; the durable session-delivery queue exists, but it is wired to restart-continuation recovery rather than general A2A inbox delivery.

## What This Changes

Request a persistent message queue for `sessions_send` and A2A delivery so failed or timed-out messages can be retried, acknowledged, and recovered after restarts.

## Best Possible Solution

Define a core reliable A2A delivery contract: persist accepted `sessions_send` messages before dispatch, drain pending entries on startup or heartbeat, record durable acknowledgement and idempotency state, and document/test timeout, restart, duplicate-delivery, permission, and transcript-provenance behavior across the session tool and Gateway agent flow.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: ordinary `sessions_send` calls Gateway `agent` directly, its timeout path returns no durable queue metadata, and the queue enqueue path is scoped to restart-continuation recovery in the inspected production code.

## Solution Assessment

Unclear as an implementation path until maintainer design. A persistent queue is the right class of solution, but a file-only BOOT polling inbox would be weaker than a core contract covering acceptance, retry, dedupe, recovery, permissions, and provenance.
