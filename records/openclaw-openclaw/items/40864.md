## Summary

Keep open. Current main has a durable session-delivery queue primitive and restart-continuation recovery, but ordinary `sessions_send` still starts a Gateway `agent` run directly, reports timeout/error without a persisted delivery or ack state, and treats the A2A announce delivery as best-effort. The issue is still a real core reliability gap rather than something already implemented or safely closeable as duplicate scope.

## What This Changes

The issue requests a persistent at-least-once delivery queue for agent-to-agent `sessions_send` messages so timed-out or failed sends can be retried and recovered.

## Best Possible Solution

Define a core reliable A2A delivery contract: persist accepted `sessions_send` messages before dispatch, drain them on startup or heartbeat, record stable acknowledgement and idempotency state, and document/test timeout, restart, duplicate-delivery, and security-boundary behavior across the session tool, Gateway agent method, queue recovery, and docs.
