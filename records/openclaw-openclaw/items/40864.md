## Summary

Keep open. Current main still does not implement the requested durable agent-to-agent inbox or at-least-once acknowledgement/retry path for `sessions_send`. The current path starts an internal gateway `agent` run with a fresh random idempotency key and process-local dedupe; the durable queues I found cover outbound channel delivery and restart-continuation recovery, not cross-session message injection reliability. This matches and updates the prior ClawSweeper review against current `main` at 17811480dae20f65b526454d75e971ae8424ec10.

## Best Possible Solution

Keep this issue open as a core gateway/session reliability design item. A good implementation would persist an accepted cross-session delivery record before dispatch, use stable caller/sender idempotency keys, record acknowledgement or terminal failure, drain and retry on gateway startup or heartbeat, and add regression coverage for timeout and gateway-restart behavior around `sessions_send`. If maintainers want plugins involved, first add an explicit durable cross-session delivery/ack extension point; the current plugin/outbound queue surfaces do not provide this guarantee.

## Risks / Open Questions

- Existing durable delivery code can be mistaken for this request: `src/infra/outbound/*` is outbound channel delivery, and `src/infra/session-delivery-queue-*` is currently restart-continuation recovery rather than a target-agent `sessions_send` inbox.
- A real fix needs careful replay and idempotency semantics so restart recovery does not duplicate agent turns or bypass session visibility and `tools.agentToAgent` policy.
