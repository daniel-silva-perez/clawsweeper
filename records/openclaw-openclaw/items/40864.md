## Summary

Keep open. Current main still starts ordinary `sessions_send` by directly calling the Gateway `agent` method and returns timeout/error states without durable queue, acknowledgement, retry, or recovery metadata; the durable session-delivery queue in the tree is scoped to restart-continuation recovery, not a general A2A inbox path.

## What This Changes

Add reliable persistent agent-to-agent delivery so `sessions_send` messages survive timeouts, failures, and restarts and can be retried or acknowledged by the target agent.

## Best Possible Solution

Define a core reliable A2A delivery contract: persist accepted `sessions_send` messages before dispatch, expose durable acknowledgement/retry state, drain pending entries on startup or heartbeat, and cover timeout, restart, duplicate-delivery, permission, and transcript-provenance behavior with focused source and integration tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: ordinary `sessions_send` calls Gateway `agent` directly, its timeout/error paths return no durable queue metadata, and the only durable session-delivery enqueue path found in production is restart-continuation recovery.

## Solution Assessment

Unclear until maintainer design. A persistent queue is the right class of solution, but the proposed file-only BOOT polling inbox is weaker than a core contract covering acceptance, retry, dedupe, recovery, permissions, and provenance.
