## Summary

Keep open. Current main has durable session-delivery primitives for restart-continuation recovery, but ordinary `sessions_send` still dispatches a Gateway `agent` run directly and reports timeout/error without a persisted inbox entry, durable acknowledgement state, or retry handle; the A2A announce path is also still best-effort.

## What This Changes

The issue requests persistent, at-least-once agent-to-agent delivery for `sessions_send`, including durable inbox state, retries, startup or heartbeat polling, and idempotency.

## Best Possible Solution

Define a core reliable A2A delivery contract: persist accepted `sessions_send` messages before dispatch, drain pending entries on startup or heartbeat, record stable acknowledgement and idempotency state, and document/test timeout, restart, duplicate-delivery, and security-boundary behavior across the session tool, Gateway agent method, delivery queue, and public docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: ordinary `sessions_send` calls Gateway `agent` directly, its timeout branch returns no durable queue metadata, and `rg` finds `enqueueSessionDelivery` wired only through restart-continuation production paths rather than the session-send tool.

## Solution Assessment

Unclear as an implementation path until maintainer design. A persistent queue is the right class of solution, but the best maintainable fix needs an explicit core contract for acceptance, retry, dedupe, recovery, permission checks, and transcript provenance rather than a narrow file-based inbox patch.

## Risks / Open Questions

- Generalizing the restart-continuation queue to ordinary A2A messages could duplicate inter-session instructions unless acknowledgement, idempotency, and transcript provenance are specified together.
- Persistent cross-session delivery changes security and permission boundaries because queued messages may run after the original sender context, tool policy, or target route has changed.
- The existing queue storage is adjacent but not yet a public delivery contract; maintainers need to decide storage format, retry/backoff semantics, failure visibility, and docs before implementation.
