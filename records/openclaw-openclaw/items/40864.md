## Summary

Keep open. Current main still has durable session-delivery primitives, but ordinary `sessions_send` dispatches a Gateway `agent` run directly and returns timeout/error without a persistent inbox entry, durable acknowledgement state, or retry handle; the A2A announce path also remains best-effort.

## What This Changes

The issue requests persistent, at-least-once agent-to-agent delivery for `sessions_send`, including durable inbox state, retries, startup or heartbeat polling, and idempotency.

## Best Possible Solution

Define a core reliable A2A delivery contract: persist accepted `sessions_send` messages before dispatch, drain pending entries on startup or heartbeat, record durable acknowledgement and idempotency state, and document/test timeout, restart, duplicate-delivery, and permission behavior across the session tool, Gateway agent method, queue, and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: ordinary `sessions_send` calls Gateway `agent` directly, its timeout branch returns no durable queue metadata, and queue enqueueing is only wired to restart-continuation paths in the inspected production source.

## Solution Assessment

Unclear as an implementation path until maintainer design. A persistent queue is the right class of solution, but a file-only BOOT polling inbox would be weaker than a core contract covering acceptance, retry, dedupe, recovery, permissions, and transcript provenance.

## Risks / Open Questions

- Generalizing the restart-continuation queue to ordinary A2A messages could duplicate inter-session instructions unless acceptance, acknowledgement, idempotency, and transcript provenance are specified together.
- Persistent cross-session delivery changes security boundaries because queued messages may run after the sender context, tool policy, target visibility, or route has changed.
- Maintainers still need to decide whether reliable delivery belongs in the session tool, Gateway `agent` method, or a broader session-delivery service before implementation.
