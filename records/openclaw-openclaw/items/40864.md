## Summary

Keep open. Current main still handles ordinary `sessions_send` as a direct Gateway `agent` dispatch and returns timeout/error states without a durable queue id, acknowledgement, or retry handle; the durable session-delivery queue now present in the tree is still wired to restart-continuation recovery, not a general A2A inbox path.

## What This Changes

Request a persistent message queue for `sessions_send` and A2A delivery so failed or timed-out messages can be retried, acknowledged, and recovered after restarts.

## Best Possible Solution

Define a core reliable A2A delivery contract: persist accepted `sessions_send` messages before dispatch, drain pending entries on startup or heartbeat, record durable acknowledgement and idempotency state, and document/test timeout, restart, duplicate-delivery, permission, and transcript-provenance behavior across the session tool and Gateway agent flow.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: ordinary `sessions_send` calls Gateway `agent` directly, its timeout path returns no durable queue metadata, and the only durable queue enqueue path found in production is restart-continuation recovery.

## Solution Assessment

Unclear until maintainer design. A persistent queue is the right class of solution, but the proposed file-only BOOT polling inbox would be weaker than a core contract covering acceptance, retry, dedupe, recovery, permissions, and provenance.
