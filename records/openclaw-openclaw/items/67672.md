## Summary

Keep open. Current main still lacks an awaited Slack SocketModeClient disconnect in the provider shutdown path, and the supplied PR is a plausible narrow fix for the reported stale-socket leak, but it needs maintainer review because the current main cleanup helper has moved and the PR still lacks focused regression coverage.

## What This Changes

The PR adds a Slack SocketModeClient accessor/disconnect helper, awaits it before Bolt app stop during Slack cleanup, and removes a redundant SlackProbe cast.

## Best Possible Solution

Land a narrow Slack-owned cleanup on current main that awaits the underlying `SocketModeClient.disconnect()` before Bolt `app.stop()` in the shared Slack cleanup helper, keeps the removed `stopAccount` hook out, preserves the SlackProbe cast cleanup, and adds a regression test for disconnect call order/awaiting during abort or stale-socket restart cleanup.

## Reproduction Assessment

No high-confidence live reproduction was run for the full OOM/leaked-socket production symptom. There is a high-confidence static reproduction path for the cleanup gap: health monitor restart calls `stopChannel`, Slack cleanup reaches `gracefulStopSlackApp`, and current main only awaits Bolt `app.stop()` while the locked receiver stop wrapper does not await the client disconnect.

## Solution Assessment

Yes, directionally: awaiting the Slack `SocketModeClient.disconnect()` in Slack-owned cleanup is the narrow maintainable fix. It should be rebased onto the current `provider-support.ts` helper and covered by a focused regression test before merge.
