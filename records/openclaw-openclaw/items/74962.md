## Summary

Keep open: current main still has the bundled-only web_fetch provider lookup reported by #74915, and this active PR is a focused implementation that aligns non-sandboxed fetch provider resolution with the plugin contract while retaining the sandbox restriction. ## What This Changes

The PR updates web_fetch provider resolution and tests so non-sandboxed explicit or runtime-selected providers can come from non-bundled plugins, while sandboxed fetch stays bundled-only.

## Best Possible Solution

Land this PR or an equivalent narrow fix after maintainer review and targeted validation, preserving bundled-only provider resolution for sandboxed fetch while allowing non-sandboxed web_fetch to honor external plugin provider contracts.

## Reproduction Assessment

Yes. The linked issue gives concrete external-plugin setup steps, and current main has a source-level reproduction because resolveWebFetchDefinition filters candidates to bundled providers before explicit or runtime-selected provider ids can match.

## Solution Assessment

Yes. The PR uses the existing runtime-provider resolver for the non-sandboxed runtime-preferred path and removes the non-sandboxed bundled-origin filter for explicit provider lookup, which is narrower than adding a new plugin API or special-casing a provider id.
