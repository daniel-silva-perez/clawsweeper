## Summary

Keep this issue open. Current main still clears Control UI/WebChat active-run state on reconnect hello, the hello snapshot/protocol has no active chat-run recovery field, and the browser client rejects in-flight RPCs on socket close rather than replaying or reattaching. There is an open implementation candidate, #41067, that appears to target the same active-run reconnect recovery gap, so this issue should stay open while that PR is reviewed/refreshed rather than being closed as implemented.

## Best Possible Solution

Keep this issue open as the canonical WebChat/Control UI active-run reconnect recovery report. The best fix is to review and refresh #41067 or implement the same narrow product outcome: expose active chat-run state in an additive gateway hello/snapshot or resubscribe contract, preserve/reconcile `chatRunId` and streaming/busy state across reconnect, avoid clearing active state until the run is known terminal, replay or reconcile missed terminal state, and add regression coverage for WebSocket close/reconnect during an active `chat.send` run. Keep #45952 separate for outbound message queue/ACK loss.
