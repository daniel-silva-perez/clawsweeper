## Summary

Current main still does not recover an active Control UI/WebChat chat run across a WebSocket reconnect. The browser reconnect hello path clears active run state, the hello snapshot protocol still has no active chat-run recovery field, and an open PR, #41067, is still unmerged and explicitly targets the missing active-run recovery behavior.

## Best Possible Solution

Keep this issue open as the canonical Control UI/WebChat active-run reconnect recovery report. The best path is to review and refresh #41067 or land an equivalent narrow fix: add an additive active chat-run recovery contract in the gateway hello/snapshot or a dedicated reconnect RPC, preserve/reconcile `chatRunId` and busy/streaming state across reconnect, avoid clearing active state until the run is known terminal, handle reconnect-time `chat.send` rejection without losing the run, and add regression coverage for WebSocket close/reconnect during an active `chat.send`. Keep #45952 separate for outbound message queue/ACK loss.
