## Summary

Keep this PR open for maintainer handling. It is a MEMBER-authored PR with the protected `maintainer` label, and the latest head still introduces a blocking accessibility conformance issue by assigning `role="combobox"` to a native `textarea`.

## What This Changes

The PR adds stable slash-command listbox and option IDs, textarea menu relationship attributes, active-descendant/live-status announcements, composer wrapper CSS, focused UI tests, and a changelog entry for Control UI chat accessibility.

## Best Possible Solution

Revise the composer accessibility pattern so the native textarea keeps conforming textbox semantics while the slash menu still has stable option IDs, allowed relationship attributes, and live announcements; then update the tests and land after focused UI checks and exact-head CI are clean.

## Reproduction Assessment

Yes. Static review of PR head `41faecbd82a996c8f8d7e4e0efb0fd89eeb8e371` reproduces the issue at `ui/src/ui/views/chat.ts:1221`, and the W3C ARIA-in-HTML table confirms `textarea` cannot take `role="combobox"`.

## Solution Assessment

No. The stable IDs and live announcement direction are useful, but assigning `combobox` to the textarea is not the best fix because it violates the textarea role contract; a conforming textbox/autocomplete pattern or separate conforming combobox control is safer.
