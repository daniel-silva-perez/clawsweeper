## Summary

Keep this PR open. Current main still does not implement the PR’s central behavior: media understanding resolves one env-proxy fetch before it knows the configured audio/video backend baseUrl, then passes that fetch into provider calls. Main has newer target-aware proxy helpers, but they are not wired into this runner path, and the current tests still assert env-only fetch selection rather than a NO_PROXY-covered backend bypass.

## Best Possible Solution

Keep this PR open and adapt it to current main: resolve the media provider execution context before choosing `fetchFn`, reuse or deliberately extend the shared `proxy-env.ts` target-aware matcher, then add focused runner/proxy tests for a NO_PROXY-covered self-hosted audio backend and for private/internal backends that must still receive the proxy fetch when not excluded.

## Risks / Open Questions

- The PR changes provider network egress routing, so review should verify it never broadens direct egress beyond loopback and operator-declared bypasses.
- The branch likely needs adaptation to current main because `src/infra/net/proxy-env.ts` now contains shared target-aware NO_PROXY logic; duplicating a separate matcher in `proxy-fetch.ts` risks semantic drift.
- CIDR-style NO_PROXY support is broader than Undici’s native `EnvHttpProxyAgent` behavior and should be an explicit maintainer decision with focused tests for both bypassed and still-proxied private backends.
