## Summary

Keep open. Current main still does not expose the requested read-only REST endpoints for sessions, agents, status, or session detail. The Gateway has WS RPC handlers for `sessions.list`, `agents.list`, `sessions.get`, `status`, and `health`, plus existing HTTP probes and separate HTTP routes such as `/v1/*`, `/tools/invoke`, `/sessions/:key/history`, and `/sessions/:key/kill`, but no `/api/v1/sessions`, `/api/v1/agents`, `/api/v1/status`, or equivalent common-query REST surface. The linked implementation PR was closed unmerged, and no matching route/file is present on main.

## Best Possible Solution

Keep this open unless maintainers explicitly close it in favor of #20934. The best implementation path is a small authenticated, read-only Gateway HTTP API that maps GET endpoints for sessions, agents, status, and session detail onto the existing RPC/query helpers, with tests for auth, 404/method handling, and docs under the Gateway HTTP API surface.

## Risks / Open Questions

- A REST implementation would expose private session and agent metadata over HTTP, so it should reuse the existing Gateway HTTP auth helpers and `operator.read` scope checks rather than adding an unaudited shortcut.
- The endpoint shape should be reconciled with the broader open REST session-management discussion tracked in #20934 if maintainers want one canonical API design thread.
