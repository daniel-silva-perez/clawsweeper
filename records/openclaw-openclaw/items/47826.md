## Summary

Current main still does not expose the requested read-only REST endpoints for sessions, agents, session detail, or combined status. The underlying Gateway WebSocket RPC methods exist and are read-scoped, but the HTTP router/docs only cover the existing probes, OpenAI-compatible endpoints, tools invoke, session kill/history helpers, plugin routes, managed media, and Control UI. The related implementation PR #47863 is closed unmerged, while #20934 is adjacent broader session-management work rather than a full duplicate.

## Best Possible Solution

Keep this open for a maintainer-owned Gateway HTTP API design decision. The best path is to add a small authenticated read-only REST surface for session list, agent list, combined status, and session detail; reuse the existing `listAgentsForGateway`, `listSessionsFromStore`, and `sessions.get` semantics; enforce `operator.read` consistently with Gateway method scopes; add route/auth/schema tests; and document the public endpoint namespace on https://docs.openclaw.ai/gateway. Coordinate the contract with #20934 so read-only queries and broader session-control endpoints fit together.

## Risks / Open Questions

- Adding these endpoints expands the public Gateway HTTP API and needs an explicit namespace/versioning decision: the issue asks for `/api/*`, the related PR proposed `/api/v1/*`, and current first-party HTTP surfaces mostly use `/v1/*`, `/tools/invoke`, and `/sessions/*` helper paths.
- Auth and scope behavior needs maintainer design: the read endpoints should consistently require `operator.read`, and shared-secret HTTP auth should align with the current Gateway HTTP trust model.
- There is real overlap with #20934, so read-only query endpoints and future mutating session-management endpoints should share one coherent contract instead of diverging.
