## Summary

Keep open. Current main still lacks the requested read-only Gateway REST endpoints, while the underlying WebSocket RPC methods and read-scope authorization already exist; the direct implementation PR is closed unmerged and the related session-management issue is broader rather than a complete duplicate.

## What This Changes

Request to add lightweight authenticated Gateway HTTP REST endpoints for read-only sessions, agents, session detail, and status-style queries so web apps can avoid CLI cold-start and WebSocket overhead.

## Best Possible Solution

Define a small authenticated read-only Gateway REST surface for session list, agent list, combined status, and session detail; reuse the existing RPC helpers and `operator.read` scope checks; add route/auth/schema tests; and document the contract on https://docs.openclaw.ai/gateway while coordinating with #20934.

## Reproduction Assessment

Yes. A source-level reproduction path is high confidence: current main has no registered route or handler for the requested `/api` paths, Control UI excludes `/api/sessions`, and unhandled HTTP requests fall to 404.

## Solution Assessment

Unclear until a maintainer confirms the public API shape. The narrowest maintainable direction appears to be an authenticated read-only REST layer over the existing RPC helpers, not re-enabling command parsing on the HTTP chat path or relying on the closed unmerged PR as-is.
