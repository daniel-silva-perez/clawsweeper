## Summary

Keep this issue open. Current main still lacks the requested lightweight read-only Gateway REST endpoints for sessions, agents, session detail, or combined status. The underlying WebSocket RPC read methods exist and are scoped as `operator.read`, but the HTTP router and docs only expose the existing probe, OpenAI-compatible, tools, session kill/history, media/plugin, and Control UI surfaces. The related implementation PR #47863 is closed unmerged, while #20934 is adjacent broader HTTP session-management work rather than a full duplicate.

## Best Possible Solution

Keep this open for a maintainer-owned Gateway HTTP API decision. The strongest path is a small authenticated read-only REST surface for session list, agent list, combined status, and session detail, implemented in the Gateway HTTP router, backed by existing `sessions.list`, `sessions.get`, and `agents.list` semantics, enforced with `operator.read`, covered by route/auth/schema tests, and documented on https://docs.openclaw.ai/gateway. Coordinate the exact namespace with #20934.

## Risks / Open Questions

- Adding a new public Gateway HTTP API changes the authenticated operator surface; docs currently describe HTTP bearer credentials as effectively all-or-nothing operator access, so route scope, token behavior, and trusted-proxy semantics need maintainer review.
- Endpoint namespace should be coordinated with #20934 so read-only query endpoints and broader session-management endpoints do not diverge or create incompatible REST shapes.
- The previous implementation PR #47863 closed unmerged, so a replacement should not blindly replay that branch without reviewing why it did not land.
