## Summary

Keep open. Current main still lacks the requested read-only Gateway REST endpoints, while the existing WebSocket RPC methods and read-scope authorization are already present; the linked implementation PR is closed unmerged and the related session-management issue is broader rather than a complete duplicate.

## What This Changes

Request to add lightweight authenticated Gateway HTTP REST endpoints for common read-only session, agent, status, and health queries so integrations can avoid CLI cold-start and WebSocket overhead.

## Best Possible Solution

Define a small authenticated read-only Gateway REST surface over the existing `agents.list`, `sessions.list`, and `sessions.get` behavior; enforce the existing operator read-scope boundary; add route/auth/schema tests; document the public contract on the Gateway docs page; and coordinate naming with #20934 before accepting an implementation.

## Reproduction Assessment

Yes. A source-level reproduction path is high confidence: current main has no registered route or handler for the requested `/api` read endpoints, Control UI excludes `/api/sessions`, and unhandled HTTP requests fall through to 404.

## Solution Assessment

Unclear until a maintainer confirms the public API shape. The narrow maintainable direction appears to be an authenticated read-only REST layer over existing RPC helpers and scopes, not re-enabling command parsing on chat HTTP paths or relying on the closed unmerged PR as-is.
