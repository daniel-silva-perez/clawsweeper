## Summary

Keep this issue open. Current main still does not implement the requested protection for every persisted `status: "running"` session during session maintenance. Save-time enforcement only preserves the caller-supplied active session key, stale pruning and entry-count capping ignore `SessionEntry.status`, load-time cleanup has no preserve set, and disk-budget eviction only skips one active key. The related PR #59932 remains an implementation candidate in the provided GitHub context, not something already merged on main.

## Best Possible Solution

Keep this issue open as the tracking bug until a focused core fix lands. The best implementation is to build a shared protected-session-key set from the caller `activeSessionKey` plus all currently active or persisted `status: "running"` session rows, then thread that set through save-time stale pruning, maxEntries capping, load-time maintenance, and disk-budget eviction. Review and land #59932 if it still applies cleanly, or replace it with an equivalent patch plus regression tests covering multiple running sessions under age, count, and disk-budget pressure.

## Risks / Open Questions

- Concurrent running subagent sessions can still be evicted when a different session save triggers enforce-mode maintenance and only that caller's active key is preserved.
- Oversized-store load-time cleanup can still remove persisted running sessions before runtime reconciliation has a chance to recover them.
- Disk-budget cleanup can delete running session entries and their artifacts if they are oldest and not the single caller active key.
