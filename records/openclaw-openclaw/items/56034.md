## Summary

Keep this issue open. Current main still only protects one caller-supplied active session key during enforce-mode maintenance, while stale pruning, entry-count capping, load-time cleanup, and disk-budget eviction do not skip every persisted session with status "running". The related PR #59932 remains open and is not implemented on main.

## Best Possible Solution

Keep this issue open and use it as the tracking bug until a focused core fix lands. The best fix is to derive a shared protected-session-key set from the caller activeSessionKey plus all currently active/running sessions, then thread that set through stale pruning, maxEntries capping, load-time maintenance, and disk-budget eviction. Review and land a corrected #59932, or replace it with an equivalent patch plus regression tests for multiple running sessions under age, count, and disk-budget pressure.

## Risks / Open Questions

- Concurrent running subagent sessions can still be evicted when another session save triggers enforce-mode maintenance and only that caller's activeSessionKey is preserved.
- Oversized-store load/restart cleanup can still remove persisted running entries before later code has a chance to reconcile them.
- There is no current regression test proving multiple status:"running" sessions survive pruneAfter, maxEntries, and maxDiskBytes enforcement.
