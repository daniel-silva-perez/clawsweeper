## Summary

Keep this PR open. Current main still deletes Telegram commands before setting a changed non-empty command menu, and the PR remains a narrow, relevant follow-up rather than something already implemented or superseded. The main maintainer decision is whether to land it as-is after rebase/targeted validation or sequence it with the separate retry-after rate-limit PR because skipping the delete changes the failure mode when `setMyCommands` fails.

## What This Changes

The branch changes the Telegram command-menu sync helper and its colocated tests so non-empty command sync calls `setMyCommands()` without a preceding `deleteMyCommands()`, while empty-menu sync still deletes commands explicitly.

## Best Possible Solution

Keep the change scoped to the Telegram plugin: non-empty command-menu updates should use one `setMyCommands` call, empty-menu clearing should continue to use `deleteMyCommands`, and tests should lock both paths. If maintainers consider stale-menu persistence on failed set unacceptable, add a focused retry/fallback policy in the same helper without changing broader channel or plugin SDK contracts.
