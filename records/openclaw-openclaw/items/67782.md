## Summary

Keep this PR open for maintainer review. Current main still performs `deleteMyCommands()` before `setMyCommands()` for changed non-empty Telegram command menus, while this PR is a narrow Telegram plugin change that preserves explicit empty-menu clears and adds focused regression coverage.

## What This Changes

The PR changes Telegram command-menu sync so non-empty menus call `setMyCommands()` without a preceding `deleteMyCommands()`, while empty menus still call `deleteMyCommands()`, with colocated tests for both paths.

## Best Possible Solution

Keep the change scoped to the Telegram plugin: non-empty command-menu updates should use one `setMyCommands()` call, empty-menu clearing should continue to use `deleteMyCommands()`, and tests should lock both paths. If stale-menu persistence on failed set is unacceptable, add a focused retry or fallback policy in the same helper rather than changing broader channel or plugin SDK contracts.

## Reproduction Assessment

Yes. The current-main helper and unit test provide a high-confidence reproduction: a non-empty command sync calls `deleteMyCommands()` before `setMyCommands()`, and startup registration reaches that helper.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix for the stated behavior because it changes only the Telegram command-menu helper and tests, and the Telegram method contract supports direct non-empty replacement. The only maintainer judgment is whether to add a fallback for failed `setMyCommands()` or sequence with #73947.
