## Summary

Keep this issue open. Current main has fixed or narrowed several related provider-qualified selection problems, and it already supports some per-model/per-agent params, but it does not implement the central request: using `provider/name` as the internal model profile key while reserving `id` solely for the upstream API model id. Current code, schema, plugin docs, and picker paths still key configured models by `provider/id`, and `reasoning` remains a boolean model capability rather than a model-entry level string.

## Best Possible Solution

Keep this open as a core model/config design issue. The best path is a backward-compatible model-profile key design, likely with an explicit profile key or API model id separation, plus a migration plan for `agents.defaults.models`, session overrides, model picker values, provider plugin manifests, and docs. Treat already-fixed provider-qualified bugs as related history, and keep narrower open work like #41594/#47852 linked rather than using them to close this umbrella request.

## Risks / Open Questions

- Closing now would lose a real remaining design gap: multiple named profiles for the same upstream model id under one provider still are not represented as distinct internal keys across the current catalog/picker/config paths.
- The proposal bundles two separable changes, profile keying and `reasoning` level syntax; maintainers may want to split those before implementation.
- A migration from `provider/id` to another key shape would affect persisted sessions, allowlists, aliases, defaults, UI selection, fallback chains, plugin manifests, and docs, so it needs explicit compatibility design rather than conservative cleanup.
