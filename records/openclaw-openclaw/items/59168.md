## Summary

Keep open: current main still uses the provider-qualified model id as the canonical internal model key, while `name` remains display metadata. The secondary request is also not implemented because configured model `reasoning` still validates as a boolean-only capability flag, with reasoning levels configured through existing thinking defaults/params instead.

## What This Changes

The issue proposes using `provider/name` as the internal model key while preserving `id` as the upstream API model identifier, plus allowing reasoning levels on configured model entries.

## Best Possible Solution

Keep this open for maintainer design review of the model-ref contract, then implement one coherent migration that separates internal model/profile identity from upstream API model ids without breaking existing `provider/model` configs. The design should explicitly cover duplicate profiles for one API model, backward-compatible lookup, deprecation warnings, docs, UI/session persistence, and how reasoning/generation/context parameters attach to the selected profile.

## Reproduction Assessment

Yes for the source-level contract: current main can be inspected to see that configured model rows are keyed by `provider/id`, while `name` is label metadata and `reasoning` is boolean-only in the schema. A live provider reproduction is not necessary to verify this architectural gap.

## Solution Assessment

Unclear: the requested `provider/name` direction is plausible, but it is not obviously the safest final API because `name` currently behaves as display text. A maintainer should decide between `provider/name`, a new explicit profile/key field, or an alias/profile layer before coding.
