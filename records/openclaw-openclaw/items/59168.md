## Summary

Keep open. Current main still treats the selectable model reference as `provider` plus model `id`; `name` remains display metadata, and duplicate configured model IDs under one provider collapse in catalog/UI/runtime lookup paths. Some related provider-prefix bugs are covered by current parsing and session tests, but this broader profile-key migration is not implemented and needs maintainer design.

## What This Changes

The issue proposes using `provider/name` rather than `provider/id` as the internal model key so one upstream API model ID can have multiple local profiles.

## Best Possible Solution

Design a backwards-compatible model profile key contract that separates local profile identity from upstream API model ID, migrates existing `provider/model` references safely, and updates selection, catalog, session, UI, docs, and provider-runtime lookup together with focused regression tests for duplicate same-provider API IDs.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure two `models.providers.<provider>.models[]` rows with the same `id` and different `name`/params, then inspect the configured model view or runtime lookup: current main keys and dedupes by `provider/id`, so the profiles cannot remain distinct.

## Solution Assessment

Unclear. Using `name` as the internal key is plausible, but maintainers should compare it against an explicit profile `key` or `apiModelId` field because `name` currently also serves display semantics and the migration would affect public config and persisted state.
