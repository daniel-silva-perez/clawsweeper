## Summary

Keep this PR open. Current main still injects full task-completion result text into both internal runtime-context and plain prompt paths, so the PR addresses a real unbounded-payload gap; the only concrete cleanup I found is a low-severity provider test-helper weakening that should be rebased away now that current main already restores the Codex mini catalog route.

## What This Changes

The PR adds head/tail truncation for task-completion internal-event results with new agent internal-events tests, plus an unrelated provider-runtime catalog test/helper adjustment for `openai-codex/gpt-5.4-mini`.

## Best Possible Solution

Rebase the useful internal-event truncation change onto current main, keep the focused agent tests, and drop or reconcile the now-redundant provider catalog test changes so catalog tests keep exact cardinality. If maintainers want a user-configurable cap, track that as a follow-up to #57148 rather than blocking the bounded default.

## Reproduction Assessment

Yes. On current main, a task-completion event with a very large `result` flows through `formatAgentInternalEventsForPrompt` and `formatAgentInternalEventsForPlainPrompt` without truncation; the PR’s added tests exercise that deterministic formatter path.

## Solution Assessment

Mostly yes. Bounding at the internal-event formatting boundary is the narrowest place that covers both runtime-context and plain prompt delivery, but the unrelated provider-runtime test helper should stay exact and be rebased against current main’s Codex mini route restoration.
