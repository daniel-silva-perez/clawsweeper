## Summary

Keep this PR open. Current main still lacks the proposed cron webhook private-network opt-in: cron webhook POSTs call the SSRF-guarded fetch without a policy, the cron config type/schema do not accept `cron.webhookAllowPrivateNetwork`, and docs/tests only cover the default blocked behavior. The PR remains a focused implementation candidate, but because it intentionally widens an SSRF-sensitive path it should get maintainer/security review and be reconciled with the stricter open hardening direction in #49120.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants the opt-in, land this PR or a maintainer-authored equivalent with `CronConfig`, strict Zod schema, generated config metadata, help/labels, public docs, and targeted tests kept in sync; preserve default blocking; pass the policy through primary delivery, failure destination, failure alert, and legacy webhook fallback paths; and add clear operator-risk wording. If maintainers choose the stricter #49120 direction instead, close this PR as superseded by that security hardening path.

## Risks / Open Questions

- Enabling the proposed flag intentionally lets cron webhook POSTs reach loopback/private/internal targets; if cron job definitions are user-controlled or multi-tenant, this can reopen SSRF-style reachability.
- `allowPrivateNetwork` skips the guard's private-network checks and reserved-hostname blocking for names such as `localhost` and `.internal`, so the docs and review should describe the full operator risk, not only IP ranges.
- Open PR #49120 proposes stricter pre-persistence rejection for private/internal cron webhook URLs, so maintainers need to choose or reconcile the product/security direction before landing this opt-in.
