## Summary

Keep open. Current main still lacks this opt-in config/schema/docs/plumbing, and the PR addresses a real operator use case, but it deliberately widens cron webhook SSRF policy for private-network targets. Because #49120 proposes the opposite stricter posture, this needs maintainer/security review rather than cleanup closure or an automated replacement PR.

## What This Changes

The PR adds optional `cron.webhookAllowPrivateNetwork` config, schema/help/docs metadata, cron webhook policy forwarding for delivery/failure paths, and targeted gateway cron tests.

## Best Possible Solution

Maintainers should choose the cron webhook security posture explicitly: either land this opt-in after confirming the warning/docs/naming and all webhook paths are correct, or close it as superseded if the stricter reject-at-save direction in #49120 is preferred.
