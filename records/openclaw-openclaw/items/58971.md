## Summary

Keep open. Current main still lacks the proposed cron private-network opt-in, but the PR intentionally creates a security-sensitive SSRF exception and overlaps with open #49120's stricter reject-at-save direction, so it needs maintainer/security judgment rather than cleanup closure.

## What This Changes

The PR adds a `cron.webhookAllowPrivateNetwork` config/docs/schema flag, forwards `{ allowPrivateNetwork: true }` into cron webhook POST delivery and failure notification paths, and adds gateway cron tests for the default and opt-in behavior.

## Best Possible Solution

Maintainers should choose the cron webhook network policy explicitly: either land a tightly documented opt-in with final naming/warnings and coverage for all webhook paths, or supersede it with the stricter private/internal URL rejection direction in #49120.
