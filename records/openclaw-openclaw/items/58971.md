## Summary

Keep open. Current main still lacks the cron webhook private-network opt-in, and the PR is a focused implementation, but it intentionally widens a security-sensitive SSRF policy and needs maintainer/security judgment, especially against the stricter open #49120 direction.

## What This Changes

This PR adds `cron.webhookAllowPrivateNetwork` config, schema, generated help/docs plumbing, forwards `{ allowPrivateNetwork: true }` through cron webhook delivery and failure webhook POST paths, and adds gateway cron tests for default and opt-in policy forwarding.

## Best Possible Solution

Maintainers should choose the cron webhook network policy explicitly: either land a tightly documented operator-only opt-in with final naming, warnings, schema/docs/tests, and all webhook paths covered, or supersede it with the stricter reject-at-save behavior from #49120.
