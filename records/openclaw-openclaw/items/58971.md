## Summary

Keep open. Current main still lacks the cron private-network webhook opt-in, and the PR is a focused implementation rather than obsolete cleanup. The remaining blocker is maintainer/security policy judgment because #49120 tracks the opposite direction of rejecting private/internal cron webhook URLs before persistence.

## What This Changes

The PR adds a `cron.webhookAllowPrivateNetwork` config/docs/schema flag, forwards `{ allowPrivateNetwork: true }` into cron webhook delivery and failure webhook paths, and adds gateway cron tests for default and opt-in policy forwarding.

## Best Possible Solution

Maintainers should choose the cron webhook network policy explicitly: either land a tightly documented opt-in with final naming, warnings, and coverage for all webhook paths, or supersede it with the stricter reject-at-save direction in #49120.
