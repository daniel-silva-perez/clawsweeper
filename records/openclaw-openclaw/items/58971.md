## Summary

Keep open. Current main still lacks the proposed cron webhook private-network opt-in, and cron webhook POSTs continue to use the default SSRF guard. The PR diff is focused and earlier schema/baseline review issues appear addressed, but the remaining decision is maintainer/security policy: whether to allow an operator opt-in for private-network cron webhooks or pursue the stricter reject-before-persistence direction in #49120.

## What This Changes

This PR adds `cron.webhookAllowPrivateNetwork` to cron config/schema/docs, forwards `{ allowPrivateNetwork: true }` through cron webhook delivery and failure webhook paths, and adds gateway cron tests for default and opt-in policy forwarding.

## Best Possible Solution

Maintainers should explicitly choose the cron webhook network policy: either land a tightly documented operator-only opt-in with final naming, warnings, schema/docs/tests, and all webhook paths covered, or supersede this with the stricter reject-at-save behavior from #49120.
