## Summary

Keep this PR open. Current main still lacks the requested exec child-process env injection, so the PR is not obsolete; however, the proposed node-host implementation is incomplete because it forwards the new values as `system.run` env overrides that are filtered out for shell-wrapper node commands before the actual child process is spawned.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec child-process environments, updates gateway and node-host forwarding, and adds docs plus regression tests for override and empty-string semantics.

## Best Possible Solution

Revise the PR so `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` are injected as trusted runtime metadata after node-host shell-wrapper env filtering, consistently in both the TypeScript node host and macOS node runtime, then keep the docs' always-set empty-string semantics and add coverage that proves the final child process sees the values while user-provided env cannot spoof them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve trusted env vars through node-host sanitization:** `src/agents/bash-tools.exec.ts:1685-1686`
  - body: This adds `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` to `requestedEnv` for `host=node`, but node exec wraps commands with `/bin/sh -lc` or `cmd.exe`, and the node side filters shell-wrapper env overrides to TERM/LANG/LC/color keys before spawning. These `OPENCLAW_*` keys will be dropped, so node-host children still cannot read them despite the docs and tests claiming node coverage. Pass them through as trusted runtime env after sanitizer handling, or update the node-host allowlist in both TS and macOS paths.
