## Summary

Keep open. Current main treats text-shaped tool calls as assistant text and offers per-model `tool_choice` workarounds for local OpenAI-compatible backends, but it still does not provide a maintainer-approved strict/retry/routing policy that forces a real protocol-level tool invocation when a model emits simulated tool-call prose.

## What This Changes

The issue asks OpenClaw to detect or correct local-model replies that simulate tool calls in text instead of emitting real OpenAI-style `tool_calls`.

## Best Possible Solution

Define a maintainer-approved strict mode that preserves the invariant that only protocol-level tool calls execute, while optionally allowing provider/model-scoped retries with explicit `tool_choice` when configured. The implementation should include focused tests for text-shaped tool-call detection, vLLM/OpenAI-compatible payload behavior, hook enforcement, and docs that explain when strict forcing is safe.
