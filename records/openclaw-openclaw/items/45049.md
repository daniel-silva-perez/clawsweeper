## Summary

Keep open. Current main has diagnostics, display cleanup, an opt-in local-model `tool_choice` workaround, and an LM Studio-specific text promotion path, but it still does not provide the requested generic strict protocol-boundary policy, retry/forced tool-choice behavior, or deterministic routing for assistant text that only simulates tool use.

## What This Changes

The issue requests stricter agent-loop handling so assistant text that simulates tool use is not accepted as successful tool invocation, with possible retry, forced tool-choice, or deterministic routing safeguards.

## Best Possible Solution

Define a maintainer-approved strict protocol policy: generic executable tool activity should originate from structured tool events, simulated tool-call text should be surfaced or blocked according to an explicit mode, forced `tool_choice` retries should be opt-in and provider/model scoped, and provider-local text promotion exceptions should preserve hooks, approvals, provenance, docs, and focused tests.

## Reproduction Assessment

Yes. The exact LiteLLM/vLLM setup was not live-run here, but current main has a deterministic unit-level reproduction: pass an assistant message containing JSON tool-call-shaped text and no structured tool call into `handleMessageEnd`; it logs the warning and continues the normal reply path instead of retrying or forcing a tool call.

## Solution Assessment

No. Current main's logging plus per-model `tool_choice` override is a useful workaround, but it is not the best complete fix for this security-sensitive boundary because it leaves policy, retry behavior, and provider exceptions undecided.
