## Summary

Keep open. Current main no longer appears to execute generic text-shaped tool calls as real tools; it logs them as assistant text and documents per-model `tool_choice` workarounds for vLLM. The central request remains unresolved, though: there is no generic strict/retry/routing policy that forces a real protocol-level tool invocation when a local model emits simulated tool-call text.

## What This Changes

The issue asks OpenClaw to detect or correct local-model replies that simulate tool calls in text instead of emitting real OpenAI-style `tool_calls`.

## Best Possible Solution

Define a maintainer-approved strict mode that preserves the invariant that only protocol-level tool calls execute, while optionally allowing provider/model-scoped retries with explicit `tool_choice` when configured. Any implementation should include focused tests for text-shaped tool-call detection, vLLM/OpenAI-compatible payload behavior, hook enforcement, and docs that explain when strict forcing is safe.
