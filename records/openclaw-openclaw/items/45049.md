## Summary

Keep open. Current main has useful diagnostics and a proxy `tool_choice: "auto"` compatibility fix, but it does not implement the requested strict protocol-boundary behavior or any generic retry, forced tool-choice, deterministic routing, or delivery block for assistant text that merely looks like tool use. The discussion is also security-sensitive because changes here affect what can become executable tool activity.

## What This Changes

The issue requests stricter agent-loop handling for models that emit simulated tool calls in assistant text instead of real structured `tool_calls`, especially for local LiteLLM/vLLM deployments.

## Best Possible Solution

Define a maintainer-approved strict protocol policy: generic executable tool calls should come from structured tool events, simulated tool-call text should be detected and surfaced or blocked according to an explicit mode, forced `tool_choice` retries should be opt-in and provider/model scoped, and any provider-local text promotion exception should preserve hooks, approvals, provenance, docs, and focused tests.
