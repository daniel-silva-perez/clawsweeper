## Summary

Keep open. Current main has diagnostics, docs, a per-model `tool_choice: required` workaround, a narrow strict-agentic path for OpenAI/Codex GPT-5-family runs, and an LM Studio-specific text-promotion compatibility path, but it still does not implement the generic strict protocol-boundary enforcement, provider-scoped forced retry behavior, or deterministic routing requested for LiteLLM/vLLM local model tool-selection failures.

## What This Changes

The issue asks OpenClaw to prevent assistant prose from being accepted as tool activity and to enforce, retry, or route real structured tool invocations when a tool is clearly required.

## Best Possible Solution

Define a maintainer-approved strict protocol policy where generic executable tool activity originates only from structured tool events, simulated tool-call text is surfaced or blocked according to an explicit mode, forced `tool_choice` retries are provider/model scoped, and any provider-local promotion path still enters the same hook, approval, provenance, docs, and test surfaces as normal tool calls.

## Reproduction Assessment

Yes. The exact LiteLLM/vLLM setup was not live-run, but current main has a deterministic unit-level reproduction: pass an assistant message containing JSON tool-call-shaped text and no structured tool block into `handleMessageEnd`; it logs the warning and continues down the normal reply path instead of retrying or forcing a tool call.

## Solution Assessment

No. Current main's diagnostics, docs, per-model `tool_choice` override, strict-agentic GPT-5 path, and LM Studio exception are useful partial mitigations, but they are not the requested generic strict boundary for local OpenAI-compatible providers.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the fix changes the trust boundary between assistant text, structured tool calls, policy hooks, and executable side effects.

Concerns:

- **[medium] Decide strict text-to-tool boundary semantics:** `src/agents/pi-embedded-subscribe.tool-text-diagnostics.ts:75`
  - body: Current main logs simulated tool-call text and treats it as a reply. Any future block, retry, or forced-tool path needs an explicit policy so assistant prose cannot become execution outside the structured tool protocol.
  - confidence: 0.9
- **[medium] Preserve hook and approval guarantees:** `src/agents/pi-tool-definition-adapter.ts:232`
  - body: `before_tool_call` runs inside the structured tool adapter before `tool.execute`; any text-promotion or deterministic routing path must enter this adapter path rather than bypassing policy hooks and approval handling.
  - confidence: 0.88
- **[medium] Reconcile LM Studio text promotion:** `extensions/lmstudio/src/stream.ts:159`
  - body: LM Studio currently promotes standalone bracketed text for registered tools into structured tool-call blocks. A strict protocol policy should explicitly decide whether that provider-local exception remains allowed and under which constraints.
  - confidence: 0.82
- **[medium] Scope forced tool choice carefully:** `docs/gateway/local-models.md:239`
  - body: Docs warn `tool_choice: "required"` should only be used where every normal turn should call a tool. A generic retry mechanism could over-force tools on ordinary text answers unless scoped by provider, model, prompt, and tool availability.
  - confidence: 0.8
