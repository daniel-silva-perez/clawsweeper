## Summary

Keep open. Current main still has the reported Tailscale startup shape: after enabling Serve/Funnel, Gateway immediately performs a single hostname lookup via `tailscale status --json`, and verbose `runExec` still logs command failures before the caller suppresses the hostname error. The alternate `gateway.tailscale.binaryPath` escape hatch is also not present in config schema/types/docs.

## Best Possible Solution

Keep this issue open and fix the Gateway Tailscale startup path. Add bounded retry/backoff around the post-Serve/Funnel hostname/status lookup, preferably in the shared Tailscale status/hostname helper or the Gateway startup wrapper, and keep transient retry failures quiet until attempts are exhausted. Add regression coverage where the first `tailscale status --json` call fails after Serve setup and a later attempt succeeds. Treat `gateway.tailscale.binaryPath` as a separate schema/docs/product enhancement if maintainers want that operator control.
