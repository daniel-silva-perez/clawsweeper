## Summary

Keep open. Current main still matches the reported Tailscale startup race: Gateway Serve/Funnel startup immediately asks for the Tailnet hostname with a single `tailscale status --json` attempt, and verbose `runExec` still logs `Command failed: tailscale status --json` before the caller suppresses the hostname failure. The alternate `gateway.tailscale.binaryPath` escape hatch also is not present in schema/types/docs.

## Best Possible Solution

Keep this issue open and fix the Gateway Tailscale startup path. Add bounded retry/backoff around the post-Serve/Funnel hostname/status lookup, ideally in the shared Tailscale status/hostname helper or the Gateway startup path, and add regression coverage where the first `tailscale status --json` call fails and a later retry succeeds without producing a misleading verbose startup failure. Treat `gateway.tailscale.binaryPath` as a separate schema/docs/product enhancement if maintainers want explicit operator control.
