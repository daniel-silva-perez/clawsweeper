## Summary

Current main still matches the reported Gateway Tailscale startup race. `startGatewayTailscaleExposure` enables Serve/Funnel and immediately performs a best-effort single-shot hostname lookup, `getTailnetHostname` and `readTailscaleStatusJson` do not retry transient `tailscale status --json` failures, verbose exec logging can still emit the confusing failure line before the caller suppresses the error, and the suggested `gateway.tailscale.binaryPath` escape hatch is still absent from the strict config surface and docs.

## Best Possible Solution

Keep this issue open and fix the Gateway Tailscale startup path. Add bounded retry/backoff around the post-Serve/Funnel hostname/status lookup, preferably in the shared Tailscale status or hostname helper, and avoid verbose failure noise until retries are exhausted. Add regression coverage where the first post-Serve `tailscale status --json` call fails and a later attempt succeeds. Treat `gateway.tailscale.binaryPath` as a separate schema/docs/product enhancement if maintainers want that operator control.
