## Summary

Current main still matches the reported Gateway Tailscale startup race. The startup path runs Serve/Funnel setup and immediately performs a single `tailscale status --json` hostname lookup; the shared status helpers do not retry transient failures, verbose exec logging can still emit `Command failed: tailscale status --json`, and the suggested `gateway.tailscale.binaryPath` config escape hatch is still absent.

## Best Possible Solution

Keep this issue open and fix the Gateway Tailscale startup path. Add bounded retry/backoff around the post-Serve/Funnel hostname/status lookup, preferably in the shared Tailscale status or hostname helper, and suppress transient verbose failure noise until retries are exhausted. Add regression coverage where the first post-Serve `tailscale status --json` call fails and a later attempt succeeds. Treat `gateway.tailscale.binaryPath` as a separate schema/docs/product enhancement if maintainers want that operator control.
