## Summary

Keep open. Current main still matches the reported Gateway Tailscale startup race: after `tailscale serve`/`funnel`, the gateway immediately performs a single `tailscale status --json` hostname lookup, and verbose exec logging can still print `Command failed: tailscale status --json` before the caller suppresses the hostname error. The suggested `gateway.tailscale.binaryPath` escape hatch is also still absent from schema/types/docs.

## Best Possible Solution

Keep this issue open and fix the Gateway Tailscale startup path. Add bounded retry/backoff around the post-Serve/Funnel hostname/status lookup, preferably in the shared Tailscale status/hostname helper or Gateway startup wrapper, and keep transient retry failures quiet until attempts are exhausted. Add regression coverage where the first `tailscale status --json` call fails after Serve setup and a later attempt succeeds. Treat `gateway.tailscale.binaryPath` as a separate schema/docs/product enhancement if maintainers want that operator control.
