## Summary

Keep this PR open, but do not merge it as-is. The linked regression is still real on current main, yet the proposed branch is dirty/unmergeable and introduces definite build, session-lifecycle, policy, and unrelated cron regressions.

## What This Changes

The PR adds structured session-miss error codes, agent session bootstrap/start APIs and tool wiring, sessions_send message tracking/read-receipt behavior, generated Swift/tool-display updates, and cron scheduling fallback/force-run changes.

## Best Possible Solution

Land a focused fix for the missing-session regression that defines when an agent main session may be auto-created, preserves existing visibility/A2A/sandbox checks, updates protocol schemas/generated clients/tool catalogs/docs/tests together, and moves the cron scheduler changes to a separate review path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the declared typebox package:** `src/agents/tools/sessions-start-tool.ts:1`
  - body: The new tool imports `@sinclair/typebox`, but this repo depends on and imports `typebox` and the PR does not add a package or lockfile entry for `@sinclair/typebox`. This will fail module resolution before the tool can load.
  - confidence: 0.94
- **[P1] Don't redirect missing named sessions to main:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When `sessions.send` misses an agent-scoped key, this branch bootstraps the agent's main session and then continues with `bootstrap.canonicalKey`. A missing key like `agent:ops:incident-123` would be silently delivered into `agent:ops:main`, merging unrelated conversations instead of returning `SESSION_NOT_FOUND`.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrapping dead sessions:** `src/gateway/sessions-bootstrap.ts:117-122`
  - body: `applySessionsPatchToStore()` preserves an existing entry. If the dead main entry exists without `sessionId`, this patch does not allocate a new id before transcript creation, so bootstrap can fail and roll back the saved entry instead of recovering the session.
  - confidence: 0.89
- **[P2] Preserve unavailable bootstrap failures:** `src/gateway/server-methods/sessions.ts:416-420`
  - body: When transcript creation fails, `bootstrapAgentSession()` returns `UNAVAILABLE`, but this `sessions.send` path rewrites every non-`AGENT_NOT_FOUND` failure to `SESSION_NOT_FOUND`. Callers lose the distinction between a missing session and an infrastructure/storage failure.
  - confidence: 0.86
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:43-47`
  - body: `sessions_start` forwards any `agentId` directly to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. In restricted setups, a model could start or probe sessions it should not access.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal `operator.write` scopes will hit the default `operator.admin` requirement when calling the new API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, agents using normal coding or messaging profiles will have the recovery tool filtered out before execution.
  - confidence: 0.86
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. Existing regression coverage documents this 48h jump failure mode, so the fallback should stay anchored near the next day.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-543`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path catches those failures and records schedule errors, so force-run preflight should use the same recoverable behavior.
  - confidence: 0.84
- **[P2] Restore prior session entries on bootstrap failure:** `src/gateway/sessions-bootstrap.ts:138-140`
  - body: When transcript creation fails, the rollback unconditionally deletes the target store entry. In force or replacement flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82
- **[P2] Emit sessions.changed after sessions.start creates a session:** `src/gateway/server-methods/sessions.ts:889-899`
  - body: `sessions.start` can create a new session but returns success without emitting the same session-list refresh event used by `sessions.create`. Clients subscribed to session updates can stay stale until a later mutation happens.
  - confidence: 0.84
