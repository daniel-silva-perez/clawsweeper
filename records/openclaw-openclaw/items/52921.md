## Summary

Keep this PR open for maintainer-directed revision: the linked regression is still real on current main, but the proposed branch mixes the session recovery work with unrelated cron scheduler changes and has blocking correctness and security issues before it can merge.

## What This Changes

The PR adds structured session-miss error codes, `sessions.start`/`sessions_start`, `sessions_send` auto-bootstrap, message IDs and read receipts, generated Apple/tool-display updates, and cron scheduler fallback/force-run changes.

## Best Possible Solution

Split the work into a focused fix for #52875 that defines the missing-session recovery contract, preserves existing session visibility/A2A/sandbox policy, updates protocol schemas/generated clients/tool catalogs/docs/tests together, and moves the cron scheduler changes into a separate review path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the repo's TypeBox package:** `src/agents/tools/sessions-start-tool.ts:1`
  - body: The new tool imports `@sinclair/typebox`, but this repo depends on and imports `typebox` and the PR does not add a package or lockfile entry for `@sinclair/typebox`. This module will fail to resolve.
  - confidence: 0.94
- **[P1] Don't redirect missing named sessions to main:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When `sessions.send` misses any agent-scoped key, this bootstraps the agent's main session and continues with that canonical key. A missing key like `agent:ops:incident-123` would be silently delivered into `agent:ops:main`, merging unrelated conversations instead of returning `SESSION_NOT_FOUND`.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrapping dead sessions:** `src/gateway/sessions-bootstrap.ts:119-122`
  - body: `applySessionsPatchToStore()` preserves an existing store entry. If the dead main entry exists without `sessionId`, this patch does not allocate a new id before transcript creation, so bootstrap can fail and roll back the saved entry instead of recovering the session.
  - confidence: 0.88
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:39-45`
  - body: `sessions_start` forwards any `agentId` directly to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. In restricted setups, a model could start or probe sessions it should not be able to access.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal `operator.write` scopes will hit the default `operator.admin` requirement when calling the new API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, agents using normal coding or messaging profiles will have the recovery tool filtered out before execution.
  - confidence: 0.86
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. Existing regression coverage documents this 48h jump failure mode, so the fallback should stay anchored near the next day.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-543`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path records schedule errors instead, so forced runs should use the same recoverable behavior.
  - confidence: 0.84
- **[P2] Restore prior session entries on bootstrap failure:** `src/gateway/sessions-bootstrap.ts:138-140`
  - body: When transcript creation fails, the rollback unconditionally deletes the target store entry. In force or replacement flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82
- **[P2] Preserve UNAVAILABLE when auto-starting fails:** `src/gateway/server-methods/sessions.ts:416-420`
  - body: If bootstrap fails after agent lookup because transcript creation or storage is unavailable, this branch rewrites every non-`AGENT_NOT_FOUND` failure to `SESSION_NOT_FOUND`. Callers then lose the distinction between a bad session key and an infrastructure failure.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff needs security attention because `sessions_start` adds an agent-session lifecycle operation without the existing visibility, sandbox, and A2A policy gates.

Concerns:

- **[medium] Session-start policy bypass:** `src/agents/tools/sessions-start-tool.ts:43`
  - body: `sessions_start` accepts an arbitrary `agentId` and optional `force`, calls `sessions.start` directly, and returns session metadata. Unlike existing session tools, it does not first enforce `tools.sessions.visibility`, sandbox restrictions, or `tools.agentToAgent`, so restricted agents can probe or start sessions outside their allowed scope.
  - confidence: 0.92
