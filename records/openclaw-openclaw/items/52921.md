## Summary

Keep this PR open: the linked regression is still real on current main, but the proposed patch has blocking correctness issues and a concrete session-access policy bypass, and it also mixes unrelated cron scheduler changes into the session recovery work.

## What This Changes

The PR adds structured session-miss error codes, a `sessions.start` gateway method and `sessions_start` agent tool, `sessions_send` auto-bootstrap/message tracking/read-receipt behavior, generated client/tool-display updates, and cron scheduler fallback/force-mode changes.

## Best Possible Solution

Land a focused fix for the missing-session regression that defines the recovery contract, preserves existing session visibility/A2A/sandbox policy, keeps gateway method scopes, protocol schemas, generated clients, tool catalogs, docs, and tests aligned, and splits the cron scheduler work into a separate review path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve requested agent session keys on bootstrap misses:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When `sessions.send` misses an agent-scoped key, this branch bootstraps only the agent main session and then continues with the bootstrap canonical key. A missing key such as `agent:ops:incident-123` would be silently delivered into `agent:ops:main`, merging unrelated conversations instead of returning `SESSION_NOT_FOUND`.
  - confidence: 0.91
- **[P1] Allocate fresh session ids for replacement sessions:** `src/gateway/sessions-bootstrap.ts:117-122`
  - body: `applySessionsPatchToStore()` preserves an existing entry. If the dead main entry exists without `sessionId`, or `sessions.start({ force: true })` is meant to replace an existing session, this patch can reuse/preserve the stale entry instead of allocating a fresh transcript id, so recovery can fail or stay attached to the stuck session.
  - confidence: 0.89
- **[P2] Preserve storage failures as unavailable:** `src/gateway/server-methods/sessions.ts:416-420`
  - body: When transcript creation fails, `bootstrapAgentSession()` returns `UNAVAILABLE`, but this `sessions.send` path rewrites every non-`AGENT_NOT_FOUND` bootstrap failure to `SESSION_NOT_FOUND`. Callers lose the distinction between a missing session and an infrastructure/storage failure that should be retried or diagnosed differently.
  - confidence: 0.86
- **[P2] Enforce session visibility before sessions_start:** `src/agents/tools/sessions-start-tool.ts:43-47`
  - body: `sessions_start` forwards any `agentId` directly to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. Restricted agents could start or probe sessions outside their allowed scope.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:72`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal operator write scopes will fall through to the default admin requirement and fail to call the new API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:266-268`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, agents using the normal coding or messaging profiles can have the recovery tool filtered out before execution.
  - confidence: 0.86
- **[P2] Emit sessions.changed after sessions.start creates a session:** `src/gateway/server-methods/sessions.ts:889-899`
  - body: `sessions.start` can create a new session but returns success without emitting the same session-list refresh event used by `sessions.create`. Clients subscribed to session updates can stay stale until a later mutation happens.
  - confidence: 0.84
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. Existing regression coverage documents this 48h jump failure mode, so the fallback should remain anchored near the next day instead of two days out.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-544`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path catches those failures and records schedule errors, so force-run preflight should use the same recoverable behavior.
  - confidence: 0.84
- **[P2] Avoid persisting nextRunAtMs before force-run reservation:** `src/cron/service/ops.ts:542-546`
  - body: In force mode this advances `job.state.nextRunAtMs` before the manual run is reserved and persisted. If the process dies after that reservation persist but before completion, recovery can preserve the future timestamp and drop a past-due forced run.
  - confidence: 0.78
- **[P2] Restore prior session entries on bootstrap failure:** `src/gateway/sessions-bootstrap.ts:138-140`
  - body: When transcript creation fails, the rollback unconditionally deletes the target store entry. In force or replacement flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82
