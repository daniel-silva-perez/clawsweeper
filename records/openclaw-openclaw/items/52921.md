## Summary

Keep this PR open because the linked regression still appears valid on current main, but the proposed patch is not merge-ready: it changes gateway protocol, session lifecycle, agent tool policy, generated clients, and unrelated cron behavior while leaving blocking correctness and security issues.

## What This Changes

The PR adds structured session-miss error codes, `sessions.start`/`sessions_start` bootstrapping, `sessions_send` auto-start/messageId/read-receipt behavior, generated Swift/tool-display updates, and cron scheduler fallback/force-mode changes.

## Best Possible Solution

Land a focused fix for #52875 that defines the missing-session recovery contract, preserves existing session visibility/A2A/sandbox policy, updates gateway protocol schemas/generated clients/tool catalog/docs/tests together, and moves the cron scheduler changes into a separate review path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Don't redirect missing named sessions to main:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When `sessions.send` misses an agent-scoped key, this branch bootstraps only the agent's main session and continues with `bootstrap.canonicalKey`. A missing key like `agent:ops:incident-123` would be silently delivered into `agent:ops:main`, merging unrelated conversations instead of returning `SESSION_NOT_FOUND`.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrapping dead sessions:** `src/gateway/sessions-bootstrap.ts:119-122`
  - body: `applySessionsPatchToStore()` preserves an existing store entry. If the dead main entry exists without `sessionId`, this patch does not allocate a new id before transcript creation, so bootstrap can fail and roll back the saved entry instead of recovering the session.
  - confidence: 0.88
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:39-45`
  - body: `sessions_start` forwards any `agentId` directly to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. In restricted setups, a model could start or probe sessions it should not be able to access.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal `operator.write` scopes will hit the default `operator.admin` requirement when calling the new API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, agents using the normal coding or messaging profiles will have the recovery tool filtered out before execution.
  - confidence: 0.86
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. The existing regression coverage documents this 48h jump failure mode, so the fallback should stay anchored near the next day instead of two days out.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-543`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path catches those failures and records schedule errors, so the forced run path should use the same recoverable behavior.
  - confidence: 0.84
