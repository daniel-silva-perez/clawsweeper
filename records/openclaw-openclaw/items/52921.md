## Summary

Keep open. The linked regression is still real on current main, but this branch has blocking correctness and security issues before it can merge: it can fail to build, redirect missing named sessions into the main session, bypass existing session-tool policy, misclassify the new gateway method's scopes, and reintroduce cron scheduling regressions.

## What This Changes

The PR adds structured session-miss error codes, session auto-bootstrap and `sessions.start`/`sessions_start`, `sessions_send` message tracking/read receipts, generated Apple/tool-display updates, and cron scheduler fallback/force-run changes.

## Best Possible Solution

Land a focused fix for #52875 that defines the missing-session recovery contract, preserves existing session visibility/A2A/sandbox policy, updates gateway protocol schemas/generated clients/tool catalogs/docs/tests together, and moves the cron scheduler changes into a separate review path.

## Reproduction Assessment

Yes. The linked bug has a concrete path: on current main, sending to a configured agent with no active persisted `sessionId` reaches `handleSessionSend` and returns `INVALID_REQUEST`/`session not found` instead of bootstrapping or returning a structured session-miss code.

## Solution Assessment

No. The PR targets the right regression, but the current implementation is not the narrowest maintainable fix because it changes protocol, tool policy, generated clients, session bootstrap behavior, and cron scheduling while leaving blocking correctness and security gaps.

Full review comments:

- **[P1] Use the repo's TypeBox package:** `src/agents/tools/sessions-start-tool.ts:1`
  - body: The new tool imports `@sinclair/typebox`, but this repo depends on `typebox` and the PR does not add `@sinclair/typebox` to the package metadata. This will fail module resolution when the file is loaded.
  - confidence: 0.94
- **[P1] Keep missing named sessions from falling back to main:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When a missing agent-scoped key is requested, this branch bootstraps the agent's main session and continues with that canonical key. A missing key like `agent:ops:incident-123` would be delivered into `agent:ops:main`, merging unrelated conversations instead of reporting the missing session.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrap recovery:** `src/gateway/sessions-bootstrap.ts:119-122`
  - body: `applySessionsPatchToStore()` preserves an existing entry, so bootstrapping a dead main entry that lacks `sessionId` does not allocate a transcript id before transcript creation. The recovery path can fail and roll back instead of repairing the session.
  - confidence: 0.88
- **[P2] Preserve unavailable bootstrap failures:** `src/gateway/server-methods/sessions.ts:416-420`
  - body: The auto-start path rewrites every non-`AGENT_NOT_FOUND` bootstrap failure to `SESSION_NOT_FOUND`. If transcript creation or storage is unavailable, callers lose the distinction between a bad key and an infrastructure failure that may be retryable.
  - confidence: 0.84
- **[P2] Enforce session visibility before starting sessions:** `src/agents/tools/sessions-start-tool.ts:39-45`
  - body: `sessions_start` forwards arbitrary `agentId` values to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. Restricted agents can probe or start sessions outside their allowed surface.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves the method unclassified. Non-admin clients with normal write scopes will hit the default admin requirement instead of being able to use the new write API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, normal coding or messaging profiles can filter out the recovery tool before execution.
  - confidence: 0.86
- **[P1] Keep cron fallback anchored near the next run:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. Existing regression coverage documents this 48h jump failure mode, so the fallback should stay near the next day instead of two days out.
  - confidence: 0.87
- **[P1] Catch cron schedule errors in force preflight:** `src/cron/service/ops.ts:542-543`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path catches and records schedule errors, so forced runs should use a recoverable path too.
  - confidence: 0.84
- **[P2] Restore prior entries on bootstrap rollback:** `src/gateway/sessions-bootstrap.ts:138-140`
  - body: When transcript creation fails, rollback unconditionally deletes the target store entry. In force or replacement flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82
