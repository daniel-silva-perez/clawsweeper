## Summary

Keep this PR open. Current main still does not expose or consume `gateway.mcpLoopback.port`, so the PR's intended operator-configurable stable MCP loopback URL is not implemented. Main does have related low-level port support and a separate `mcpResumeHash` mitigation for session reuse, but that does not replace the requested public gateway config field.

## Best Possible Solution

Keep this PR open and ask for a rebase against current main. If maintainers still want operator-configured stable MCP URLs, add `gateway.mcpLoopback.port` to the typed config, strict Zod schema, help/labels/generated schema, and public gateway configuration docs, then pass `cfg.gateway?.mcpLoopback?.port ?? 0` into the current lazy `ensureMcpLoopbackServer(configuredPort)` path in `src/agents/cli-runner/prepare.ts`. Add tests for config validation, prepared backend startup on a configured port, bind-failure diagnostics, and preservation of loopback-only bearer-token behavior. If maintainers decide `mcpResumeHash` fully supersedes the product need, record that explicit decision or link a canonical follow-up before closing.

## Risks / Open Questions

- The PR branch targets the older eager `src/gateway/server.impl.ts` startup path, while current main starts bundled MCP lazily in `src/agents/cli-runner/prepare.ts`; it likely needs a rebase and runtime retargeting.
- The provided latest PR patch declares `configuredMcpLoopbackPort` inside a `try` block and references it in `catch`, which appears to need a TypeScript compile fix during rebase.
- A fixed local port can collide on restart or with another local service, so the final implementation should include clear configured-port bind-failure diagnostics.
- Because MCP loopback is an access path to gateway tools, the final implementation must explicitly preserve 127.0.0.1 binding, bearer-token isolation, and current owner/non-owner scoped tool behavior.
