## Summary

Keep open. Current main still lacks the requested between-turn Talk Mode idle timeout: wake-triggered Talk can start directly, but the runtime only finalizes non-empty transcripts after the short silence window and then resumes listening after replies or failures.

## What This Changes

Add a configurable Talk Mode idle timeout that auto-deactivates wake-triggered Talk sessions after a period with no user interaction.

## Best Possible Solution

Add an additive Talk lifecycle setting that is validated in config, included in the gateway Talk protocol/config payload, documented on the Talk docs page, and honored by macOS Talk so idle sessions return to wake listening while silenceTimeoutMs remains scoped to transcript-finalization pauses.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable Voice Wake with Trigger Talk Mode, wake into Talk, complete a turn, then remain silent; current code ignores empty between-turn silence and resumes recognition after replies or errors.

## Solution Assessment

Unclear as proposed. The direction is narrow and maintainable, but a maintainer should define the public Talk lifecycle contract before implementation chooses naming, defaults, limits, disabled behavior, and reset points.
