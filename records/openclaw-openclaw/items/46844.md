## Summary

Keep open. Current main still lacks the requested between-turn Talk Mode idle timeout: Voice Wake can activate Talk Mode directly, but Talk only finalizes non-empty transcripts on `silenceTimeoutMs` and resumes recognition after replies or failures. The related heartbeat and session-cleanup reports are broader lifecycle requests and do not cover this Talk-specific wake path.

## What This Changes

Add a configurable Talk Mode idle timeout that auto-deactivates a Voice Wake-triggered Talk session after a period with no user interaction.

## Best Possible Solution

Add an additive Talk lifecycle setting that is validated in config, exposed through `talk.config`, documented on https://docs.openclaw.ai/nodes/talk, and honored by macOS Talk so idle sessions return to wake listening while `silenceTimeoutMs` remains scoped to transcript-finalization pauses.

## Reproduction Assessment

Yes for a source-level reproduction path: enable Voice Wake plus Trigger Talk Mode, wake into Talk, complete one turn, then remain silent. Current code shows empty between-turn silence is ignored by `checkSilence`, Talk resumes listening after replies or errors, and no idle-timeout config field is available.

## Solution Assessment

Unclear as proposed. The direction is narrow and maintainable, but a maintainer should define the public Talk lifecycle contract before implementation chooses naming, defaults, limits, disabled behavior, and whether the timer resets on user speech, assistant reply, or both.
