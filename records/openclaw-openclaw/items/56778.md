## Summary

Keep this issue open. Current main still has the reported cron list isolation gap: the CLI exposes no `--agent` option, the RPC schema has no accepted exact agent filter, the Gateway handler forwards no agent field, the service returns all matching jobs when unfiltered, and the agent cron tool's list action does not infer the caller agent even though add does.

## What This Changes

The issue requests an `openclaw cron list --agent <id>` filter plus agent-session auto-scoping so multi-agent cron lists do not expose other agents' jobs.

## Best Possible Solution

Add a backward-compatible exact `agentId` filter to the cron list protocol and service, expose it as `openclaw cron list --agent <id>`, and have agent-tool list calls default to the caller's session agent while preserving unfiltered list behavior for admin callers.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current main has no CLI option, schema field, Gateway forwarding, service predicate, or agent-tool inferred list scope for `agentId`, so unscoped `cron.list` will continue returning globally visible enabled jobs.

## Solution Assessment

Yes. The narrow maintainable fix is an additive list-filtering seam plus agent-tool default scoping and focused tests/docs; broader cron ownership or declarative agent-cron work is related but not required for this bug.
