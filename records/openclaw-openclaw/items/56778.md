## Summary

Keep this issue open. Current `main` still has no native `cron.list` agent filter: the CLI only sends `includeDisabled`, the RPC schema rejects unknown list parameters, the Gateway handler forwards no agent field, and the cron tool's list action does not auto-scope from the caller session even though add does infer `agentId`. The stale label does not make this obsolete, and the related reports in the provided context cover adjacent cron-add or declarative-agent-cron work rather than this list-isolation gap.

## What This Changes

Request an agent-scoped `cron list` path so multi-agent setups can list only the requesting agent's cron jobs, with unassigned jobs visible only to the default agent.

## Best Possible Solution

Add a backward-compatible exact agent filter to `cron.list` across the protocol, service, CLI, and agent tool. Unfiltered list behavior should remain unchanged for admin callers, while scoped agent-tool calls should default to the caller's session agent and include unassigned jobs only when that agent is the configured default.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current `main` has no `--agent` option, no `CronListParamsSchema.agentId`, no handler forwarding, and no service exact-agent predicate, so the reported global list behavior follows from the code path.

## Solution Assessment

Yes. The maintainable fix is a narrow list-filtering seam plus tests and docs, not a broader cron ownership migration; agent-owned declarative cron and cron-add warnings are adjacent but separate issues.
