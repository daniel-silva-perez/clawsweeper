## Summary

Keep this PR open, but do not merge it as-is. The queue-wide dedupe and context-key preservation changes are directionally useful, but the linked heartbeat-failover cascade involves fresh approval/run IDs; current main includes those IDs in approval text and exec event keys, so the proposed `(text, contextKey)` equality check can still miss the reported duplicate prompt path.

## What This Changes

The branch changes `src/infra/system-events.ts` to scan each session queue for duplicate `(text, contextKey)` entries, preserve `lastContextKey` on skipped or contextless enqueues, remove the old `lastText` state, and add four system-event regression tests.

## Best Possible Solution

Implement a stable intent-level dedupe path for exec approval retries, likely by carrying a command/session-level dedupe key from the exec approval or heartbeat failover layer into system events, while keeping channel reactions and other context-specific events distinct. The final patch should include a regression where the command is the same but approval/run IDs differ.

## Reproduction Assessment

Yes. We have a high-confidence unit-level reproduction for the current system-event bug by enqueuing an interleaved duplicate, and a focused review path showing the PR still misses fresh ID-bearing retries; full live heartbeat verification remains pending with the reporter.

## Solution Assessment

No. Queue-wide `(text, contextKey)` dedupe is useful but not the narrowest complete fix for the reported cascade because the problematic IDs are embedded in both text and keys; a stable dedupe key or ID normalization is needed before this can be considered the best solution.
