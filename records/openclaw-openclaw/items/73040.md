## Summary

Keep this PR open, but do not merge it as-is. The patch is a useful partial cleanup for queue-local duplicate suppression, but the linked heartbeat-failover cascade is driven by fresh approval/run IDs that change both the event text and context key, so exact `(text, contextKey)` matching does not fully solve the reported behavior.

## What This Changes

The PR changes `src/infra/system-events.ts` to remove `lastText`, scan each session queue for exact duplicate `(text, contextKey)` events, preserve `lastContextKey` only after non-duplicate contextful enqueues, and add four regression tests.

## Best Possible Solution

Implement a stable intent-level dedupe path for exec approval retries, likely by carrying a command/session-level dedupe key from the exec approval or heartbeat failover layer into system events, while keeping channel reactions and other context-specific events distinct. The final patch should include a regression where the command is the same but approval/run IDs differ.

## Reproduction Assessment

Yes. A high-confidence focused reproduction exists by enqueuing or generating the same exec command under different approval/run IDs and showing the PR's exact `(text, contextKey)` scan still accepts both; full live heartbeat verification remains pending with the reporter.

## Solution Assessment

No. Queue-wide exact `(text, contextKey)` dedupe is a good local cleanup, but it is not the best complete fix for the linked cascade because the problematic retries intentionally carry fresh ID-bearing text and keys; a stable dedupe key or ID normalization is the safer boundary.

## Review Findings

Full review comments:

- **[P2] Use a stable key for approval retries:** `src/infra/system-events.ts:117-120`

Please derive a stable command/session dedupe key or normalize the ID-bearing fields, then add a regression with different IDs.
