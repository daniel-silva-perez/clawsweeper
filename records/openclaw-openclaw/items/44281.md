## Summary

Keep this PR open. Current main has some generic Telegram reply normalization and some agent-final paths pre-strip reply tags before media parsing, but it still does not implement the PR's core parser-order/direct-delivery fix: `parseReplyDirectives()` extracts `MEDIA:` before stripping `[[reply_to:*]]`, and the media parser only recognizes lines whose trimmed start is `MEDIA:`.

## Best Possible Solution

Refresh the PR against the current `extensions/telegram` layout and keep the fix narrow: strip reply tags before `MEDIA:` extraction for the extraction path, without normalizing unrelated text or code-block whitespace. Add a parser/outbound regression for `[[reply_to:<id>]] MEDIA:...` and a Telegram delivery test proving the media is sent with the reply target.
