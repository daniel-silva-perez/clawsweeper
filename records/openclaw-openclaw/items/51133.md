## Summary

Keep open. Current main has useful adjacent mitigations, but I did not find a hard fix for the reported Feishu/MiniMax loop where repeated `message` sends are driven by `sessions_list` returning no current-session row. The existing comments are useful: KeepMyClaw was offered as an external session-state recovery workaround, and Artyomkun narrowed the likely root cause to the model misreading current-session visibility. Related ClawSweeper reports cover other loop classes, not this exact `message` + `sessions_list` Feishu path.

## Best Possible Solution

Keep this issue open and implement a targeted fix around the actual failure mode: make current-session status unambiguous to the model, preferably by steering current-session checks to `session_status` or by having `sessions_list` include/annotate the current session consistently, and add a regression test for a Feishu/MiniMax-style `message` + `sessions_list` loop. A second useful hardening layer would be a per-run duplicate outbound-message guard or enabling a narrow mutating-tool loop breaker for repeated identical `message` sends to the same target.
