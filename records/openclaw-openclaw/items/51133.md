## Summary

Keep open. Current main still has adjacent mitigations, but I did not find a hard fix for the reported Feishu/MiniMax loop where `sessions_list` can return no current-session row and the model repeatedly sends duplicate outbound `message` tool calls.

## Best Possible Solution

Keep this issue open. Make current-session state unambiguous to the model, preferably by steering self-session checks to `session_status` or by making `sessions_list` consistently include or explicitly annotate current-session absence. Add a regression test for a Feishu/MiniMax-style sequence where `sessions_list` returns zero and the model attempts repeated identical `message` sends. A narrow per-turn duplicate outbound-message guard for identical sends to the same target would be useful hardening.
