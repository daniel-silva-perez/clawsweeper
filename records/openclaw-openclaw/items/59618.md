## Summary

Keep open. Current main has partial compaction-retry handling, retry-state UI tracking, and timeout guards, but I did not find source, tests, docs, or release notes proving the reported mid-turn compaction liveness and final-delivery failure is fixed. The discussion also includes a second concrete reproduction where work continues after compaction but the user-visible TUI/WebSocket delivery disappears, which is not covered by the current targeted tests I found.

## Best Possible Solution

Keep this open and implement a focused compaction-turn liveness contract: preserve the active turn/reply obligation across both in-attempt compaction and overflow retry, ensure final assistant/tool events use the same journal and WebSocket/TUI delivery path as a fresh turn, and add regression coverage for both variants described here. Coordinate the design with open related #66126 so retry re-injection, infinite loops, and missing delivery are handled as one compaction-retry state-machine problem.
