## Summary

Keep open. Current main has partial compaction-retry and delivery safeguards, but it does not prove the reported long-context TUI/gateway delivery loss is fixed; the relevant tests cover retry waits, lifecycle state, and notices rather than the deterministic post-compaction output-disappears scenario, and related #66126 remains open on the same retry-state boundary.

## What This Changes

The issue requests that automatic compaction during an in-progress agent turn preserve and resume the turn so the final assistant output is delivered instead of silently disappearing.

## Best Possible Solution

Implement or land a focused turn/reply-obligation fix that survives local auto-compaction and verifies that post-compaction tool events and the final assistant reply traverse the normal gateway/WebSocket/TUI delivery path. If #66126 becomes canonical, fold this issue into that fix and add a regression for this issue's TUI-delivery variant.

## Reproduction Assessment

Unclear for current main. The issue and comment provide credible logs plus a deterministic high-context vLLM trigger, but there is no inspected in-repo regression that reproduces the active-client delivery loss after compaction.

## Solution Assessment

No closeable solution is proven yet. The current retry-wait, liveness, and notice plumbing are useful mitigations, but the safer solution is a regression-backed fix around turn obligation and client delivery rather than treating the issue as implemented.
