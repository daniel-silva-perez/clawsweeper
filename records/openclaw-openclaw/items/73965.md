## Summary

Keep open. Current main still lacks this PR's intended fail-closed Discord startup behavior: Discord bot identity lookup errors are caught and converted into an undefined bot id, malformed/no-id identity responses are not rejected, and the guild mention-required drop still depends on `botId`. The PR is a narrow implementation candidate for the open security-labeled #42219 and carries forward the same focused fix direction from #46856 and #49218.

## Best Possible Solution

Keep this PR open for maintainer review and land it, or land an equivalent focused Discord plugin fix. The fix should make `fetchDiscordBotIdentity` throw on rejected or malformed `fetchUser("@me")`, keep cleanup before lifecycle startup intact, remove the redundant `botId` guard from the mention-required drop branch, and retain regressions for rejected identity fetch, no usable id, and configured mention-pattern gating when bot id is absent.
