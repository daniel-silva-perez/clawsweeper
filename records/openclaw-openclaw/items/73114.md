## Summary

Keep open. Current main has meaningful partial guardrails for task cleanup, cron-run session pruning, and manual/session-store maintenance, but the reported default-case memory bloat is still not resolved: a 471-entry, 26-day store remains under the default age/count/high-water thresholds, startup still reads and parses the full store, disk-budget cleanup is opt-in, and skills snapshots remain duplicated inline per session.

## What This Changes

The issue requests bounded gateway memory and storage behavior for accumulated session records, task records, repeated skills snapshots, stale running tasks, and eager session-store loading.

## Best Possible Solution

Keep the shipped task-ledger and cron-run reaper improvements, then make an explicit session-storage policy change: add a default live-entry or disk budget that covers under-500 but large stores, avoid eager full-store materialization for historical metadata, and replace inline per-session skillsSnapshot copies with a compatible interned or versioned reference model.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline, parses the full sessions.json before cleanup, and would not prune a 471-entry store whose rows are 26 days old under the 30-day, 500-entry, and 550 high-water defaults. The precise heap-growth number was not reproduced live in this read-only sweep.

## Solution Assessment

Unclear. The current task TTL, lost-task reconciliation, cron-run reaper, and cleanup command are necessary partial fixes, but the complete solution needs maintainer-chosen session retention, disk-budget, deduplication, or lazy-loading policy rather than a narrow mechanical patch.
