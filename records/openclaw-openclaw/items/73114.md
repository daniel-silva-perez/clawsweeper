## Summary

Keep open. Current main has useful task-ledger cleanup and some session-store maintenance, but the reported default-case memory bloat is still not fully addressed: a 471-entry, 26-day store remains below the default age/count cleanup triggers, startup still reads and parses the whole sessions.json, disk-budget cleanup is opt-in, and skillsSnapshot remains stored inline on each session row.

## What This Changes

The issue requests bounding gateway memory growth by pruning or archiving historical session/task records, deduplicating per-session skill snapshots, and reconciling zombie running tasks.

## Best Possible Solution

Keep the existing task-ledger and manual cleanup guardrails, then make an explicit session-storage change: bound live sessions by a default byte or lower live-entry policy that covers under-500 large stores, avoid eager full-store materialization for historical metadata, and replace inline per-session skillsSnapshot copies with a compatible interned or versioned reference model.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline, parses the full sessions.json before cleanup, and would not prune a 471-entry store whose rows are 26 days old under the default 30-day, 500-entry, and 550 high-water behavior. The precise heap-growth number was not reproduced live because this was a read-only review.

## Solution Assessment

Unclear. The current task TTL, lost-task reconciliation, cron-session pruning, disk-budget option, and cleanup command are necessary partial fixes, but the complete fix needs a maintainer-chosen session retention and storage-schema direction rather than a narrow mechanical patch.
