## Summary

Keep open. Current main has useful partial guardrails for task cleanup and session-store maintenance, including task TTL pruning, lost-task reconciliation, and optional session disk-budget cleanup, but it still does not resolve the reported memory shape by default: the concrete 471-entry, 26-day store remains below the default age/count thresholds, disk-budget cleanup is opt-in, the full sessions JSON is still parsed into memory before cleanup, and each session entry can still persist its own inline skillsSnapshot.

## What This Changes

The issue requests bounded session/task retention and reduced gateway memory growth from accumulated session rows, task records, duplicated skills snapshots, and stale running tasks.

## Best Possible Solution

Keep the task-ledger fixes, then decide the session-store storage policy explicitly: either lower/default the retention or disk-budget behavior to cover stores like the reported one, add compatible skillsSnapshot interning or lazy snapshot lookup, and avoid startup paths that must parse all historical session metadata before deciding what is live.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline per session, parses the full sessions.json at load, and would not prune the reported 471-entry, 26-day store under the default 30-day/500-entry thresholds. No for the exact heap number, because this read-only sweep did not replay the reporter's local workload.

## Solution Assessment

Unclear. The current task TTL and lost-task reconciliation are necessary and well-targeted, but they are not the full solution for this issue because the session-memory side still needs maintainer-chosen retention, disk-budget, deduplication, or lazy-loading policy.
