## Summary

Keep open. Current main has important new guardrails for session-store maintenance and task-ledger cleanup, but the specific memory-bloat report is not fully resolved: the reported 471-entry, 26-day session store would still sit below the default session count and age thresholds, and skill snapshots are still persisted inline on each session entry rather than deduplicated or lazy-loaded.

## What This Changes

Request to reduce Gateway memory growth by bounding accumulated session and task records, deduplicating per-session skill snapshots, detecting zombie CLI/subagent tasks, and avoiding eager historical loading.

## Best Possible Solution

Keep the issue open for maintainer review of the remaining storage design: preserve the current task/session sweepers, then decide whether to add compatible skill-snapshot interning, stricter detached-session retention, lazy session metadata loading, or a disk-budget default that addresses the reported memory shape without breaking transcript, trajectory, and skill-runtime consumers.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline per session, and the default 30-day/500-entry session maintenance thresholds would not prune the reported 471-entry, 26-day store. No live heap reproduction was run in this read-only sweep.

## Solution Assessment

Unclear. Current main's task TTL and zombie reconciliation are necessary partial fixes, but they are not enough to close the broader memory-bloat request because session-store duplication and default retention behavior remain materially relevant.
