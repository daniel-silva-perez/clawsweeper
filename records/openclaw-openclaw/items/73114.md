## Summary

Keep open. Current main has partial task-ledger cleanup, but the reported session-store memory path is still unresolved: the default thresholds would not prune the reported 471-entry, 26-day store, disk-budget cleanup remains opt-in, startup still parses the full session JSON, and skills snapshots are still stored inline per session entry.

## What This Changes

The issue requests bounded retention and memory-conscious storage for persisted session/task records to prevent gateway heap growth.

## Best Possible Solution

Keep the shipped task-ledger sweeper, then make a maintainer-owned session-storage policy decision that covers the default case: stricter detached-session retention or a default disk/live-entry budget, compatible skillsSnapshot interning or lazy lookup, and a startup path that can decide what is live without materializing all historical session metadata first.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline per session, parses the full sessions.json at load, and would not prune the reported 471-entry, 26-day store under the default 30-day/500-entry thresholds. The exact heap-growth number was not reproduced in this read-only sweep.

## Solution Assessment

Unclear. The current task TTL and lost-task reconciliation are necessary partial fixes, but the broader issue needs maintainer-chosen session retention, disk-budget, deduplication, or lazy-loading policy before it can be closed.
