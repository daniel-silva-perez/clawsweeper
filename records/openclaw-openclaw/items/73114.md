## Summary

Keep open. Current main has meaningful partial guardrails for task cleanup and session-store maintenance, but the specific memory-bloat report is still not resolved by default: the reported 471-entry, 26-day session store is below the default age/count thresholds, disk-budget cleanup is opt-in, startup still parses the full session store, and skills snapshots remain embedded per session entry.

## What This Changes

The issue requests bounded session/task retention and reduced Gateway memory growth from accumulated session rows, task records, duplicated skills snapshots, and stale running tasks.

## Best Possible Solution

Keep the shipped task-ledger sweeper, then make an explicit session-storage policy decision that covers the reported default case: stricter detached-session retention or a default disk/live-entry budget, compatible skillsSnapshot interning or lazy lookup, and a startup path that does not need to materialize all historical session metadata before deciding what is live.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline per session, parses the full sessions.json at load, and would not prune the reported 471-entry, 26-day store under the default 30-day/500-entry thresholds. The exact heap-growth number was not reproduced in this read-only sweep.

## Solution Assessment

Unclear. The current task TTL and lost-task reconciliation are necessary partial fixes, but the issue is broader than the task ledger and needs maintainer-chosen session retention, disk-budget, deduplication, or lazy-loading policy before it can be closed.
