## Summary

Keep open. Current main has important partial guardrails for task cleanup, cron-run session pruning, and session-store maintenance, but it still does not solve the reported default case: a 471-entry, 26-day `sessions.json` remains below the 30-day and 500-entry thresholds, startup still reads and parses the full store, disk-budget cleanup is opt-in, and `skillsSnapshot` is still stored inline per session. Target item: https://github.com/openclaw/openclaw/issues/73114

## What This Changes

The issue requests bounded gateway memory and storage behavior for accumulated session records, task records, repeated skills snapshots, stale running tasks, and eager session-store loading.

## Best Possible Solution

Keep the task-ledger and cron-run reaper improvements, then make a maintainer-owned session-storage policy change: add a default live-entry or disk budget that covers the reported under-500 store, avoid parsing all historical metadata before deciding what is active, and replace inline per-session skillsSnapshot copies with a compatible interned or versioned reference model.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline, reads and parses the full sessions.json before maintenance, and would not prune a 471-entry store whose rows are 26 days old under the 30-day and 500-entry defaults. The precise heap-growth number was not reproduced live.

## Solution Assessment

Unclear. The current task TTL, lost-task reconciliation, cron-run reaper, and session cleanup command are necessary partial fixes, but they are not the narrowest complete solution for default gateway memory bloat because the remaining session-store policy and schema choices need maintainer design judgment.
