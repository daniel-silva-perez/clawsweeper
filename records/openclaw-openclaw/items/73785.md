## Summary

Keep open. The PR targets a real current-main gap, but it should not merge as-is because the new system-event path downgrades heartbeat-token handling to message mode and would still deliver short no-op summaries ending in HEARTBEAT_OK, which is one of the leak shapes described in the related issue discussion. The branch is also dirty against newer main and needs to preserve the current relayable-exec-completion logic before landing.

## What This Changes

The PR branch adds shared no-op sentinel detection, applies it in heartbeat reply normalization, switches system-event heartbeat-token stripping to message mode, and adds regression tests for exact sentinel-only replies and exec handoffs.

## Best Possible Solution

Land a narrow core fix that keeps the shared exact-sentinel detector, preserves heartbeat-mode `ackMaxChars` suppression for no-op cron and metadata-only system-event summaries, allows meaningful relayable exec output summaries through, and adds regression coverage for exact sentinels plus short summary-with-HEARTBEAT_OK replies.

## Reproduction Assessment

Yes. The related issue discussion gives a concrete reproduction trace, and the PR branch code path can be checked directly: a system-event reply like `Watchdog clean. HEARTBEAT_OK` enters `message` mode and becomes deliverable text instead of using heartbeat-mode short-remainder suppression.

## Solution Assessment

No. The centralized exact-sentinel helper is a good direction, but switching every system event to message-mode token stripping is not the narrowest correct fix; the safer path is to keep heartbeat-mode suppression for no-op system events and only exempt genuinely relayable exec summaries.
