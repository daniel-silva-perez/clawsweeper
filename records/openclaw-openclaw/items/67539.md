## Summary

Keep open. Current main still has only the generic TTS system prompt and the speech-provider contract has no provider-owned prompt-hint hook, so agents still cannot learn active-provider expressive syntax at runtime. The Gemini TTS path and docs support the premise: tags are preserved and documented for humans, but not injected through a provider-specific runtime hint.

## What This Changes

Request to add an optional provider-owned, model-aware TTS prompt-hint hook and append only the active speech provider's short hint when TTS is enabled.

## Best Possible Solution

Add a backwards-compatible provider prompt-hint hook to the speech-provider contract, have speech-core append only the active provider's short hint when TTS is enabled, let the Google provider own Gemini audio-tag guidance, and cover active-provider-only plus policy-gated behavior in tests and docs.

## Repair Work Prompt

Implement the requested provider-specific TTS prompt-hint support as a narrow additive change. Add an optional provider-owned prompt hint hook to SpeechProviderPlugin and exported speech SDK types with context for the resolved provider config, model/voice when available, modelOverrides policy, and synthesis target. Update buildTtsSystemPromptHint and its call sites to resolve only the active speech provider and append that provider's concise hint when TTS is enabled; do not load or advertise inactive providers. Keep the generic speech-core TTS hint provider-neutral. Add a Google/Gemini hint for square-bracket expressive cues inside spoken text, gated so disabled/off-policy controls are not advertised. Add focused tests for hook absence, active-provider-only injection, Google hint presence, and policy-gated omission. Update relevant provider/TTS/plugin SDK docs and add a single-line changelog entry credited to @barronlroth. Do not change provider selection, fallback semantics, provider auth, or hard-code Gemini guidance in core.
