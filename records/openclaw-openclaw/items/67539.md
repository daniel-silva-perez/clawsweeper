## Summary

Keep open. Current main still has only the generic TTS system prompt and the speech-provider contract still has no provider-owned prompt-hint hook, while the Google Gemini TTS provider and docs show the motivating expressive-tag behavior exists only as human-facing guidance.

## What This Changes

Add provider-specific, model-aware TTS prompt hints so the active speech provider can teach agents which expressive controls are valid at runtime.

## Best Possible Solution

Add a backwards-compatible provider prompt-hint hook to the speech-provider SDK, have speech-core append only the active provider's short hint when TTS is enabled, and let the Google provider own Gemini audio-tag guidance with active-provider-only and policy-gated tests.

## Reproduction Assessment

Yes. A high-confidence reproduction is source inspection on current main: enable TTS and build the system prompt, and it can only include the generic buildTtsSystemPromptHint text because no provider hook or provider context is present.

## Solution Assessment

Yes. A provider-owned hook is the narrowest maintainable solution because it keeps Gemini and future vendor syntax in the owning provider instead of hard-coding provider rules in speech-core or relying on optional skills/docs.

Implement provider-specific TTS prompt hints for openclaw/openclaw#67539. Add a backwards-compatible optional hook to SpeechProviderPlugin and the public speech SDK so a speech provider can return a short runtime prompt hint using resolved provider config, resolved model/voice where available, target, and modelOverrides policy. Update speech-core so buildTtsSystemPromptHint keeps the existing generic Voice (TTS) guidance and appends only the selected active provider's hint when TTS is enabled.
