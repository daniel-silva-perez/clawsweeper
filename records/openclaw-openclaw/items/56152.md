## Summary

Keep open. Current main still uses the single-canonical-path local media root check, so the PR's macOS `/tmp` versus `/private/tmp` false-negative fix is not implemented on main. The PR also has unresolved security review findings around symlink escape and workspace-sibling containment, so the right cleanup is revision, not closure.

## Best Possible Solution

Revise the PR rather than close it: keep canonical media-path enforcement when `fs.realpath(mediaPath)` succeeds, use the lexical media path only when realpath fails, allow configured roots to match the needed lexical/canonical aliases, preserve the workspace-sibling guard against canonical variants, and add regression tests for `/tmp`↔`/private/tmp`, symlink escape outside the allowed root, and workspace-sibling symlink cases.

## Risks / Open Questions

- The proposed lexical-plus-realpath cross-product can allow a path that is lexically under an allowed media root even when its canonical target is outside that root.
- The proposed `resolved` value changes the workspace-sibling restriction from canonical-path behavior to lexical-path behavior.
- Current main may still falsely reject valid macOS `/tmp` and `/private/tmp` local media paths, so the user-facing bug likely remains without a safer revision.
