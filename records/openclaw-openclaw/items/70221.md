## Summary

Keep open. Current main still implements Control UI hosted embeds as a height-only presentation path: width is not parsed into the canvas preview shape, not carried through UI/sidebar types, and the iframe remains forced to container width.

## What This Changes

Add a width parameter to the Control UI `[embed]` directive so iframe previews can render at a requested, bounded width instead of always filling the chat message column.

## Best Possible Solution

Ship a bounded `preferredWidth` presentation hint end to end for hosted Control UI embeds, with responsive constraints that keep chat layout usable and without changing sandboxing, external URL policy, or non-web channel behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `[embed ref="cv_demo" title="Demo" height="600" width="900" /]`: current main parses and applies height, but width is ignored by `previewFromShortcode`, dropped by UI preview types, and overridden by the iframe `width: 100%` CSS.

## Solution Assessment

Yes. The narrowest maintainable fix is to add a bounded width/preferredWidth field through the existing canvas preview contract, renderer, sidebar path, prompt/docs, and focused tests rather than accepting arbitrary inline style or changing iframe security policy.
