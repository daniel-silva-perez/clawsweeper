## Summary

Keep this PR open. Current main still has the Android TLS shape the PR is trying to fix: the app targets SDK 36, applies a domain-specific network security config, and the active gateway WebSocket TLS path still supplies anonymous X509TrustManager instances without X509ExtendedTrustManager socket/engine overloads. The related SDK 36 issue describes the same error, but the fix is not present on main.

## Best Possible Solution

Review and land this PR or an equivalent Android-owned fix: update both GatewayTls.kt custom trust managers to X509ExtendedTrustManager with socket/engine overloads that preserve hostname-aware default trust, keep fingerprint/TOFU behavior intact, remove or audit the redundant domain-config entries, and add the narrowest feasible regression coverage plus SDK 36 device validation.
