## Summary

Keep open. Current main still has an ACP runtime-handle cache that can reuse a cached runtime for the same backend, agent, mode, cwd, identity, and live status without comparing the runtime config or a restart generation; Discord inbound jobs also carry a preflight config snapshot into queued processing. The related exec override issue remains open, but this report describes an additional ACP/runtime-cache restart boundary rather than something already implemented or safely duplicate-closable.

## What This Changes

The issue requests that Discord-bound ACP sessions stop reusing stale runtime handles or config after gateway restarts and exec config changes.

## Best Possible Solution

Add a generic ACP runtime cache invalidation boundary: either store a stable runtime config or restart-generation fingerprint in `CachedRuntimeState` and compare it in `ensureRuntimeHandle()`, or explicitly clear ACP cached handles during gateway/plugin-service restart. Handle Discord queued config snapshots only if jobs can survive the restart boundary with old exec policy, and coordinate any `/exec` precedence change with #56597 rather than hiding it in Discord-specific code.

## Reproduction Assessment

Yes from source-level inspection. A cached ACP handle can be reused when backend, agent, mode, cwd, metadata, and runtime status match, and no current-main production path found during this review clears the singleton ACP manager cache on the inspected restart/plugin lifecycle boundary; Discord jobs also carry a queued `cfg` snapshot.

## Solution Assessment

Unclear. The report's fingerprint and forced-reinitialize options are both plausible, but the narrow maintainable fix should be chosen as a generic ACP lifecycle/cache contract while keeping the persisted `/exec` policy decision aligned with #56597.
