## Summary

Keep open. Current main still matches the reported stale ACP runtime-handle shape: cached handles are reused based on backend, agent, mode, cwd, persisted handle identity, and backend status, but not on the current runtime-affecting config. The linked persisted `/exec` directive issue is adjacent and still open, but it does not supersede this ACP cache/restart invalidation path.

## What This Changes

The issue reports that Discord-bound ACP sessions can reuse stale runtime handles after gateway restarts, ignoring later `tools.exec.security` and `tools.exec.safeBins` config changes.

## Best Possible Solution

Add a deterministic ACP runtime-cache validity contract that covers runtime-affecting OpenClaw config, especially exec policy and safe-bin settings, and invalidate cached handles when that contract changes or the active backend runtime is re-registered. Preserve persistent ACP session identity/resume semantics and add focused regression coverage for Discord-bound ACP dispatch across an in-process restart/config change.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create a cached ACP runtime handle for a Discord-bound session, change runtime-affecting config, then call `ensureRuntimeHandle()` again with the same backend/agent/mode/cwd; current main can reuse the cached handle because no config fingerprint is checked. I did not run a live Discord gateway restart because this sweep is read-only.

## Solution Assessment

Unclear. The issue's config-fingerprint direction is plausible, but the best fix should be security-reviewed to define which config fields and backend runtime identities invalidate ACP handles, and whether SIGUSR1 restart should explicitly clear the ACP manager cache.
