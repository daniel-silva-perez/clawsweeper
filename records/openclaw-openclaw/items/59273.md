## Summary

Keep open. Current main still has the core stale-handle shape described by the report: `AcpSessionManager.ensureRuntimeHandle()` reuses cached ACP runtime handles based on backend, agent, mode, cwd, persisted handle identity, and backend status, but it does not compare any fingerprint of the current `OpenClawConfig` or the active registered runtime object. Discord ACP turns flow through this same manager with the Discord handler's captured config, and in-process gateway restarts reset command lanes but do not reset the ACP session manager singleton. Related cleanup reports do not supersede this one; #56597 is adjacent persisted `/exec` directive behavior, and #57896 covered a different restart activation path.

## Best Possible Solution

Keep this issue open and fix it in the ACP control plane: add a deterministic runtime-config fingerprint, at least covering exec policy/safe-bin/runtime-affecting config and the active backend runtime registration identity, to `CachedRuntimeState` and `ensureRuntimeHandle()` reuse checks. Also consider clearing ACP runtime cache on in-process gateway restart/plugin runtime re-registration, with focused regression coverage for Discord-bound ACP sessions after `tools.exec.security` or `tools.exec.safeBins` changes.
