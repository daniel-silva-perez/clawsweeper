## Summary

Keep open. Current `main` still contains the unbounded full-transcript read and regex split paths reported here, and gateway callers still reach those paths before applying response limits. The direct fix PR was closed unmerged, so there is no shipped or main-branch implementation to close against.

## What This Changes

The issue reports that very large session JSONL transcripts can block the gateway event loop when transcript readers materialize the whole file and run regex line splitting.

## Best Possible Solution

Replace full-file transcript materialization with bounded, decoder-safe line or entry iteration in the gateway transcript readers, preserving branch traversal, compaction markers, sequence metadata, malformed-line tolerance, cross-agent transcript resolution, and usage aggregation semantics. Session maintenance caps, doctor warnings, and health-probe UX can be follow-up work after the reader no longer blocks the gateway.

## Reproduction Assessment

Yes. The issue includes concrete `sample`, `lsof`, file-size, and recovery evidence, and current `main` still exposes a static reproduction path: create or retain a very large session JSONL and hit a gateway path such as `chat.history` or session event sequencing that calls `readSessionMessages`.

## Solution Assessment

Yes, the narrow transcript-reader fix is the best first solution. It directly removes the event-loop blocking mechanism without making a broader product decision about rotation, maintenance caps, or health-monitor policy.
