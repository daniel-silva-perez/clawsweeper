## Summary

Current main and the shipped v2026.4.29 tag already normalize parameter-free MCP/OpenAI tool schemas on the native OpenAI provider path, including the stricter `{ type: "object", properties: {}, required: [], additionalProperties: false }` shape needed by GPT-5.4-family strict tools. The report is valid for v2026.4.27, but the remaining user action is to upgrade rather than keep this issue open.

## What This Changes

The issue reports a v2026.4.27 regression where OpenAI GPT-5.4 rejects parameter-free MCP tool schemas such as `flux-mcp__get_flux_instance` because the outbound schema lacks `properties`.

## Best Possible Solution

Close this as fixed in current main and v2026.4.29; users still on the affected v2026.4.27 image should upgrade to v2026.4.29 or newer so the provider-owned OpenAI tool-schema normalization is present.

## Reproduction Assessment

Yes. The report includes a concrete version, model, MCP tool, provider path, and OpenAI 400 log; current-main verification maps that failing `{}` schema case onto the covered OpenAI normalization path instead of relying on a live Flux/OpenAI run.

## Solution Assessment

Yes. The implemented fix is the maintainable boundary: generic empty-schema normalization remains in agent tooling, while stricter native OpenAI schema shaping is provider-owned through OpenAI plugin hooks rather than hard-coded broadly into core.
