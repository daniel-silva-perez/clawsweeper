## Summary

Current main already fixes the reported OpenAI rejection path: parameter-free tool schemas are normalized through the shared OpenAI schema/transport layer before OpenAI Responses/WebSocket registration, and the fix is covered by runtime contract tests and included in v2026.4.29.

## What This Changes

The issue asks OpenClaw to keep parameter-free MCP tool schemas, such as `flux-mcp__get_flux_instance`, normalized before sending them to `openai/gpt-5.4` after a reported v2026.4.27 regression.

## Best Possible Solution

Keep the shared OpenAI transport/schema normalization path and its runtime contracts; no flux-specific workaround is needed because all parameter-free OpenAI tool payloads now pass through the same object-schema normalization before registration.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the existing schema-normalization contract: construct a parameter-free tool with `openai/gpt-5.4` on `openai-responses` and verify the HTTP Responses and WebSocket payloads contain the normalized object schema; the exact live flux/OpenAI call was not rerun in this read-only sweep.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it normalizes at the provider/transport boundary shared by MCP, plugin, and built-in tools instead of special-casing `flux-mcp__get_flux_instance`.
