## Summary

Current main already implements the requested normalization in both the OpenAI provider hook and the final OpenAI HTTP/WebSocket tool payload conversion. The fix first shipped in v2026.4.10, and the v2026.4.27 tag named in the report plus the latest v2026.4.29 release both contain the same normalization path, so the reported v2026.4.27 behavior no longer matches the shipped source behavior.

## What This Changes

The issue reports that OpenClaw v2026.4.27 sends a parameter-free MCP tool schema to `openai/gpt-5.4` without the required object `properties` field, causing OpenAI to reject `flux-mcp__get_flux_instance`.

## Best Possible Solution

Keep the shipped provider-owned and transport-level normalization. If a fresh v2026.4.29 or newer official image still reproduces this, the next actionable report should include the raw outbound OpenAI tool payload or a minimal MCP server fixture showing a path that bypasses the normalized tool conversion.

## Reproduction Assessment

Yes for the code-level path: a parameter-free tool with `parameters: {}` is covered by current runtime contract tests for both OpenAI HTTP Responses and WebSocket payload conversion. The live third-party MCP/OpenAI deployment was not rerun in this read-only pass.

## Solution Assessment

Yes. The current solution is the narrow maintainable boundary: provider-owned OpenAI/Codex hooks normalize native route schemas, and the final OpenAI transport conversion defensively normalizes tool parameters before submission.
