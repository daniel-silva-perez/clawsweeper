## Summary

Keep this PR open for maintainer/security review. The reported invalid-input timing hardening target is real in current main, and the added edge-case tests are useful, but the latest diff replaces the current padded byte comparison with a fixed-key HMAC digest comparison of candidate credentials, conflicting with the recent CodeQL remediation and active GitHub Advanced Security comments on the PR.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs, compare fixed-key HMAC-SHA256 digests with `timingSafeEqual`, and add null/undefined-versus-empty regression cases.

## Best Possible Solution

Keep the padded timing-safe byte comparison as the equality primitive, add a reviewed invalid-input path that performs comparable dummy work before returning false, and retain the new null/undefined-versus-empty regression coverage.

## Reproduction Assessment

Yes. Current main visibly returns before buffer allocation and `timingSafeEqual` for non-string inputs, while the string path performs byte conversion and timing-safe comparison; this is a high-confidence source-level reproduction of the reported timing shape.

## Solution Assessment

No. The PR addresses a real hardening target and adds useful tests, but replacing the CodeQL-remediated padded byte comparison with fixed-key HMAC digest comparison is not the safest maintainable solution.
