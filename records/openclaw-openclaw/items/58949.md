## Summary

Keep this PR open for maintainer changes, but do not merge it as-is. Current main has not implemented the null/undefined timing hardening, yet this branch conflicts with the fresh CodeQL remediation by replacing padded byte equality with SHA-256 digest equality for candidate secrets.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs to empty strings, hash both normalized values with SHA-256, compare the hashes with `timingSafeEqual`, and then require both original inputs to be strings.

## Best Possible Solution

Keep the current padded timing-safe byte comparison as the equality primitive, then add a reviewed invalid-input path that performs comparable dummy work before returning false. Add regression coverage for null/undefined against empty strings so normalized-empty false positives cannot slip in.

## Reproduction Assessment

Yes for the reported fast-fail path: on current main, `safeEqualSecret(null, "secret")` returns before the buffer and `timingSafeEqual` work that valid strings perform. This review did not measure timing, but the branch difference is visible in source.

## Solution Assessment

No. The requested hardening is narrow, but hashing normalized inputs conflicts with current main's CodeQL-remediated byte comparison; the safer implementation should retain padded raw-byte comparison and only remove or mask the invalid-input fast path.
