## Summary

Keep this PR open for maintainer review. The reported invalid-input fast path still exists on current main, so the work is not obsolete, but the proposed implementation should not merge as-is because it replaces the current padded byte comparison with SHA-256 digest comparison, which main recently removed as part of CodeQL remediation and documented in the changelog.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs to empty strings, compare SHA-256 digests with `timingSafeEqual`, bitwise-combine the result with original type checks, and add edge-case tests for null/undefined against empty strings.

## Best Possible Solution

Keep the padded timing-safe raw byte comparison as the equality primitive, then adjust the invalid-input path so it performs comparable dummy work before returning false. Add regression coverage for null/undefined against empty strings and both-invalid inputs so normalized-empty false positives cannot regress.

## Reproduction Assessment

Yes for the source-level behavior: on current main, `safeEqualSecret(null, "secret")` exits at the type guard before the buffer and `timingSafeEqual` work that valid strings perform. This review did not run a timing benchmark.

## Solution Assessment

No. The PR addresses the fast-fail by hashing normalized inputs, but that conflicts with the current CodeQL-remediated padded-byte implementation; the safer path is to retain the current comparison primitive and mask only the invalid-input branch.
