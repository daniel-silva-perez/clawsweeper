## Summary

Keep this PR open. Current main still has the hardcoded 5 MiB media-store and sandbox-staging cap, so the requested behavior is not implemented; however the submitted branch is stale against current main and still needs correctness work around startup wiring, generated config metadata, chat sandbox preflight, and existing caller-provided media limits before it is mergeable.

## What This Changes

The PR adds optional `media.maxBytes` config schema/type/help metadata, mutable media-store max-byte accessors initialized from gateway startup, and switches media storage, staging, and serving paths from the fixed 5 MiB constant to the configured limit.

## Best Possible Solution

Refresh or replace this work on current main with a documented, generated, bounded `media.maxBytes` config contract that is initialized in the current gateway startup path, used by media store defaults, sandbox staging, and chat sandbox preflight, and that preserves explicit per-caller/channel/tool limit semantics where those are intentionally narrower or broader.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move startup wiring to the current startup path:** `src/gateway/server.impl.ts:464`
  - body: The setter is added in `server.impl.ts` next to an old `initSubagentRegistry()` location, but current main initializes that registry from `src/gateway/server-startup-plugins.ts`. After refresh, this hunk will not apply to the live startup path; if the setter is dropped during conflict resolution, `media.maxBytes` will parse but never affect the effective limit.
  - confidence: 0.88
- **[P2] Preserve explicit media-store source limits:** `src/media/store.ts:244-245`
  - body: The patch changes source downloads and local source reads to compare against the module-global `mediaMaxBytes`. Current main has an explicit `saveMediaSource(..., maxBytes)` contract used for generated/outbound media limits, so resolving this branch that way would reintroduce 5 MiB rejections for callers that intentionally pass a larger configured cap.
  - confidence: 0.86
- **[P2] Use the configured cap in chat preflight:** `src/auto-reply/reply/stage-sandbox-media.ts:82`
  - body: Switching `stageSandboxMedia` to `getMediaMaxBytes()` is not enough for `chat.send`: current main rejects sandboxed non-image offloaded attachments above `MEDIA_MAX_BYTES` before staging runs. A 10 MiB PDF would still fail even with `media.maxBytes` raised unless that preflight uses the same effective limit.
  - confidence: 0.84
- **[P2] Regenerate the config schema metadata:** `src/config/zod-schema.ts:476-482`
  - body: The PR adds `media.maxBytes` to the hand-written schema, labels, and help text, but it does not update the generated schema metadata consumed by config surfaces. Without regenerating `schema.base.generated.ts`, the public config schema/help output will drift and omit the new key.
  - confidence: 0.82
