## Summary

Keep this PR open. The reported media-size configurability gap is real and still present on current main, but the submitted branch is stale and incomplete: the config key is not effectively wired through startup/current sandbox preflight paths, PR-head call sites use an unimported getter, and one changed store path would ignore explicit caller/channel caps.

## What This Changes

The PR branch adds a root `media.maxBytes` config field, mutable media-store max-byte helpers, config labels/help/types/tests, and switches several media store and staging call sites from the hardcoded constant to the new getter.

## Best Possible Solution

Ship a refreshed media-limit contract that wires one effective sandbox/store limit through config parsing, media store defaults, sandbox staging, and `chat.send` sandbox preflight; preserves intentional lower caller/channel caps; regenerates config schema metadata; and documents how this root limit differs from media-understanding and image-specific limits.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence static reproduction path: configure a sandboxed agent and send a non-image attachment between 5 MiB and the higher chat attachment cap; `chat.send` rejects it against `MEDIA_MAX_BYTES` before staging.

## Solution Assessment

No. The requested direction is plausible, but this submitted patch is not the best mergeable solution because it is stale, misses effective startup/preflight wiring, and does not consistently preserve caller-specific media caps.
