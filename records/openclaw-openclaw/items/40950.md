## Summary

Keep this PR open. The configurable media-size gap is still real on current main, but the submitted branch is stale and incomplete: the latest diff calls new helpers without importing them, does not wire the config value through gateway startup or the current chat sandbox preflight path, and weakens caller/channel cap semantics in the store path.

## What This Changes

The PR adds a root `media.maxBytes` config field, mutable media-store max-byte helpers, config help/labels/types/tests, and switches media store/serving/staging call sites from the hardcoded 5 MiB constant to a configurable getter.

## Best Possible Solution

Ship a refreshed media-limit contract that wires one effective sandbox/store limit through config parsing, media store defaults, sandbox staging, and `chat.send` sandbox preflight; preserves intentional lower caller/channel caps; regenerates config schema metadata; and documents how this root limit differs from media-understanding and image-specific limits.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence static reproduction path: configure a sandboxed agent and send a non-image attachment between 5 MiB and the higher chat attachment cap; `chat.send` rejects it against `MEDIA_MAX_BYTES` before staging.

## Solution Assessment

No. The requested direction is plausible, but this submitted patch is not the best mergeable solution because it is stale, misses effective startup/preflight wiring, and does not consistently preserve caller-specific media caps; a refreshed narrow patch is safer.
