## Summary

Keep this PR open because current main still does not implement the requested `media.maxBytes` configuration surface, but the submitted branch is not merge-ready. The current diff is stale against main, misses startup wiring, leaves changed call sites with unresolved `getMediaMaxBytes()` references, and does not update all current sandbox preflight paths that still enforce the fixed 5 MiB cap.

## What This Changes

The PR adds optional top-level `media.maxBytes` config schema/type/help metadata, introduces mutable media-store max-byte accessors, and switches some media staging/serving paths from the hardcoded constant to `getMediaMaxBytes()`.

## Best Possible Solution

Refresh the work onto current main and implement `media.maxBytes` as a documented, generated, bounded config contract that is initialized at gateway startup and consistently used by the media store, sandbox staging, chat sandbox preflight, and current attachment/offload paths while preserving explicit channel/caller limit semantics where intended.
