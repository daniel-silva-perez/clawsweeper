## Summary

Keep open. The underlying hardcoded media staging limit is still present on current main and the linked bug remains open, but this PR is stale, dirty against current main, and currently introduces blocking compile/runtime issues plus incomplete limit wiring.

## What This Changes

The PR adds a root `media.maxBytes` config key, media-store getter/setter helpers, config help/labels/types coverage, and switches media staging/store call sites toward a configurable media byte limit.

## Best Possible Solution

Ship a refreshed, current-main patch that defines one effective sandbox/store media limit, initializes it from config during gateway startup/reload, uses it consistently in store, media serving, sandbox staging, and chat.send preflight, preserves lower caller/channel caps, and updates generated config metadata plus focused tests/docs.

## Reproduction Assessment

Yes. Current main gives a high-confidence static reproduction path: configure a sandboxed agent and send a non-image attachment larger than 5 MiB but below the chat attachment cap; `chat.send` rejects it against `MEDIA_MAX_BYTES` before staging.

## Solution Assessment

No. The requested direction is plausible, but this submitted PR is not the best mergeable solution because it is stale, does not wire the effective limit through current startup/preflight paths, and breaks current imports/cap semantics.
