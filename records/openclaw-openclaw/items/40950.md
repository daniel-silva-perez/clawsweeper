## Summary

Keep this PR open. The underlying media-size configurability bug still exists on current main, but the submitted branch is stale and has blocking correctness issues: missing imports for the new getter, no gateway startup wiring for the new setting, omitted generated schema metadata, and a change to a media route file that current main has since removed.

## What This Changes

The PR adds a root `media.maxBytes` config field, mutable media-store limit helpers, and updates media store/staging/config help, labels, types, and tests to use or document the limit.

## Best Possible Solution

Ship a refreshed media-limit contract that threads the effective limit through media store defaults, sandbox staging, and `chat.send` sandbox preflight; preserves intentional lower caller/channel caps; regenerates config metadata; and documents how this root limit relates to existing media-understanding and channel-specific limits.

## Reproduction Assessment

Yes. Static inspection against PR head reproduces the build/runtime breakages from the missing imports and missing setter call, and current-main source shows the 5 MiB cap remains in the store and sandbox preflight paths.

## Solution Assessment

No. The requested direction is valid, but this submitted patch is not the narrowest maintainable fix until it is rebased onto current main and the config surface is wired through every effective limit path without regressing existing caller-specific caps.
