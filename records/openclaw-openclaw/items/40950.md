## Summary

Keep open. The underlying hardcoded media staging/store limit is still present on current main and the linked bug remains open, but this PR is stale against current main and still has blocking compile/runtime and limit-contract issues before it can merge.

## What This Changes

The PR adds a `media.maxBytes` config field and media-store getter/setter plumbing, then switches media store, route, sandbox staging, and config metadata surfaces toward a configurable inbound media byte limit.

## Best Possible Solution

Ship a refreshed current-main patch that defines one bounded effective sandbox/store media limit, initializes it from config during gateway startup/reload, uses it consistently in media storage, sandbox staging, and chat preflight, preserves lower caller/channel caps, and updates generated config metadata plus focused tests/docs.

## Reproduction Assessment

Yes. Current main gives a high-confidence static reproduction path: configure a sandboxed agent and send a non-image attachment larger than 5 MiB but below the chat attachment cap; `chat.send` rejects it against `MEDIA_MAX_BYTES` before staging.

## Solution Assessment

No. The requested direction is plausible, but this submitted PR is not the best mergeable solution because it is stale, does not wire the effective limit through current startup/preflight paths, and breaks import/cap semantics.
