## Summary

Keep this PR open, but it is not merge-ready. Current main still does not expose `media.maxBytes`, so the user problem remains valid; however the PR head has missing imports, no current startup wiring for the setter, stale media-route assumptions, and does not cover the current chat sandbox preflight or explicit per-caller media-store limits.

## What This Changes

The PR adds a `media.maxBytes` config field, media-store getter/setter state for the max-byte limit, and updates media storage, serving, staging, schema, type, label, and help surfaces to use the configured limit.

## Best Possible Solution

Refresh or replace the work on current main with a documented, generated, bounded `media.maxBytes` config contract that is initialized during the current gateway startup path, used consistently by media-store defaults, sandbox staging, and chat sandbox preflight, and that preserves intentional explicit per-caller/channel/tool limit semantics.
