## Summary

Keep this issue open. Current main and the latest tagged release still resolve one global memory-wiki plugin config at registration, do not accept per-agent plugin overrides, and route wiki reads/writes through the captured vault path; the related implementation PR #67584 is open and unmerged, so the requested isolation is still active work.

## What This Changes

Add per-agent or context-aware memory-wiki vault configuration so multiple agents on one gateway can keep isolated wiki vaults instead of sharing one global `plugins.entries.memory-wiki` vault.

## Best Possible Solution

Keep this as the active memory-wiki vault-isolation tracker. Land #67584 or an equivalent memory-wiki-owned change that resolves the vault root from invocation context or explicit per-agent mapping across all wiki read/write, gateway, CLI, prompt, and corpus surfaces, preserves the single-vault default, updates docs/schema, and adds two-agent regression coverage.

## Reproduction Assessment

Yes. The source-backed reproduction is two agents on one gateway with intended separate wiki vaults, then writing via one agent and searching or reading via the other; current main has no accepted per-agent plugin config and uses the same captured `config.vault.path` for wiki reads and writes. I did not run a live test because this review was required to stay read-only.

## Solution Assessment

Unclear. The issue's broad agent-level plugin override option would be a larger core config seam, while the narrower context-aware vault path or explicit memory-wiki mapping approach appears more maintainable and is already represented by open PR #67584; it still needs maintainer review to ensure every surface resolves the same agent-scoped vault.
