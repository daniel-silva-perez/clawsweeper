## Summary

Keep open. The reporter added a concrete reproduction signal: browser hosting works after install, then `openclaw browser --browser-profile openclaw status` reports the gateway closed after Feishu pairing. Current main still routes browser status through `browser.request` and the browser control service enablement gate, I found no Feishu/browser regression coverage, and the linked fix PR #52057 is closed unmerged rather than shipped.

## Best Possible Solution

Keep this issue open until maintainers reproduce fresh install plus browser setup plus Feishu pairing on current main, identify whether Feishu setup changes plugin allowlists, gateway auth/pairing state, or browser config/control-service readiness, and land a focused fix with regression coverage for `openclaw browser status` after channel setup. If the gateway is healthy but browser control is disabled or unauthorized, the CLI should report that precise condition instead of implying the gateway itself is closed.
