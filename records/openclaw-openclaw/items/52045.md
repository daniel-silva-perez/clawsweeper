## Summary

Keep open. The reporter supplied a concrete follow-up: browser hosting works after install, then `openclaw browser --browser-profile openclaw status` reports the gateway closed after Feishu pairing. Current main still routes browser status through the same `browser.request` gateway method and status endpoint, and I did not find a landed fix or regression test proving the Feishu-pairing mismatch is resolved. The directly linked fix PR #52057 was closed unmerged, so it is not a shipped or canonical resolution.

## Best Possible Solution

Keep this issue open until maintainers reproduce fresh install plus browser setup plus Feishu pairing on current main, identify whether pairing rewrites browser/plugin/gateway auth config or leaves browser control stale, and land a focused fix with regression coverage around `openclaw browser status` after channel setup. If the root cause is browser control being disabled, the CLI should report an actionable browser-control/plugin/config error rather than implying the gateway itself is closed.
