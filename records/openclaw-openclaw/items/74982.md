## Summary

Keep open. Current main still drops the Telegram `audioAsVoice` delivery hint before it reaches Telegram's `asVoice` sender option, and this PR appears to patch the missing legacy and registered `sendPayload` paths without a blocking correctness or security finding.

## What This Changes

The PR wires Telegram outbound `audioAsVoice` through to `asVoice` in the legacy outbound adapter and registered plugin `sendPayload` path, and adds legacy adapter regression coverage.

## Best Possible Solution

Review and land this Telegram plugin fix after targeted validation so the existing `audioAsVoice` delivery hint reaches Telegram's existing `asVoice` option inside the plugin boundary, with no core Telegram special cases.

## Reproduction Assessment

Yes. A voice-marked TTS/media reply reaches shared outbound delivery with `audioAsVoice`, but current main drops that hint in both Telegram outbound adapters before `send.ts`, where `sendVoice` only happens when `asVoice` is true.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix: map an existing outbound context flag to an existing Telegram sender option in the owning Telegram plugin paths; adding a registered-path regression test would strengthen proof but does not change the implementation approach.
