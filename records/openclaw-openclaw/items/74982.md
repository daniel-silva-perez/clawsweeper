Keep open for maintainer review. Current main still drops the Telegram `audioAsVoice` hint in both the legacy adapter and registered-plugin `sendPayload` path, while the PR targets those missing owner-local mappings and appears functionally correct from code review.

## What This Changes

The PR threads Telegram `audioAsVoice` into legacy and registered send options as `asVoice` and adds outbound-adapter tests for propagation and defaulting.

## Best Possible Solution

Review and land this Telegram plugin fix after targeted validation. The desired end state is for the existing `audioAsVoice` delivery hint to reach Telegram's existing `asVoice` option in both legacy and registered outbound paths without adding core Telegram special cases.

## Reproduction Assessment

Yes. A TTS/media reply with `[[audio_as_voice]]` reaches shared delivery with `audioAsVoice`, but current main drops that hint before Telegram `sendMessageTelegram`; `send.ts` only calls `sendVoice` when `asVoice` is true.

## Solution Assessment

Yes. The patch is the narrowest maintainable fix for the observed bug because it maps an existing outbound context flag to an existing Telegram send option inside the Telegram plugin boundary; adding a registered-path test would strengthen proof but does not change the implementation direction.
