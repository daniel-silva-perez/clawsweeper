## Summary

Keep open. Current main still drops Telegram `audioAsVoice` before it reaches the low-level sender in both the legacy outbound adapter and the registered plugin `sendPayload` path, while this PR wires the existing flag to Telegram's existing `asVoice` option without crossing the plugin boundary.

## What This Changes

The branch threads Telegram `audioAsVoice` through legacy outbound media/payload sends and the registered plugin payload send options as `asVoice`, with legacy adapter regression tests.

## Best Possible Solution

Review and land this Telegram plugin fix after targeted validation so the existing `audioAsVoice` delivery hint reaches the existing Telegram `asVoice` option in both outbound paths, without adding Telegram-specific behavior to core.

## Reproduction Assessment

Yes. A TTS/media reply containing `[[audio_as_voice]]` is normalized into `audioAsVoice` and delivered with that override on current main, but both Telegram outbound paths omit it before calling the sender that only uses `sendVoice` when `asVoice` is true.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix because it maps an existing outbound context flag to an existing Telegram send option inside the Telegram plugin boundary; a registered-path regression test would improve proof but not change the fix direction.
