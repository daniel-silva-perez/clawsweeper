## Summary

Keep open for maintainer review. Current main still drops the Telegram `audioAsVoice` hint before it reaches `sendMessageTelegram` in the legacy adapter and registered plugin payload path, while this PR maps that existing hint to Telegram's existing `asVoice` option without moving behavior out of the Telegram plugin boundary.

## What This Changes

The PR threads Telegram `audioAsVoice` through legacy `sendMedia`/`sendPayload` and the registered `sendPayload` path as `asVoice`, with legacy outbound adapter regression tests.

## Best Possible Solution

Review and land this Telegram plugin fix after targeted validation. The desired end state is for `[[audio_as_voice]]` and `audioAsVoice` delivery hints to consistently reach Telegram's `asVoice` option in both legacy and registered outbound paths, without adding Telegram-specific policy to shared core delivery code.

## Reproduction Assessment

Yes. A TTS or media reply containing `[[audio_as_voice]]` reaches shared outbound delivery with `audioAsVoice`, but current main drops that hint before Telegram `sendMessageTelegram`; `send.ts` only selects `sendVoice` when `asVoice` is true.

## Solution Assessment

Yes. The patch is the narrowest maintainable fix for the observed bug because it maps an existing shared outbound hint to an existing Telegram send option inside the Telegram plugin boundary; a registered-path regression test would strengthen merge proof.
