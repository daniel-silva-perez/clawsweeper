## Summary

Keep open. Current main still lacks the proposed per-field hook agent metadata limits, and the PR is not obsolete. The hardening is useful, but as written it only protects direct `/hooks/agent` normalization while mapped `/hooks/<name>` agent dispatch can still carry the same oversized metadata fields.

## Best Possible Solution

Keep this PR open and revise it into a shared hook-agent metadata validator used by both direct `/hooks/agent` payload normalization and mapped `/hooks/<name>` agent dispatch. Add focused tests for direct and mapped over-limit rejection, exact-boundary acceptance for all six fields including `agentId` and `sessionKey`, and unchanged behavior for the overall HTTP body-size limit.
