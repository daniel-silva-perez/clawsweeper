## Summary

Keep open. Current main still lacks the requested hook-agent metadata length limits, and this PR is not obsolete, but the direct-only validation would leave mapped hook agent dispatch uncovered.

## What This Changes

The branch adds hook metadata length constants plus `normalizeAgentPayload()` validation and unit tests for oversized and boundary metadata values.

## Best Possible Solution

Use a shared hook-agent metadata length validator for both direct `/hooks/agent` payloads and mapped `/hooks/<name>` agent actions before replay-key construction, with tests covering direct and mapped over-limit rejection plus exact-limit acceptance for all six capped fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a mapped hook whose `name`, `sessionKey`, `to`, `model`, or `thinking` renders from an oversized payload value and POST to `/hooks/<name>`; current mapped dispatch does not pass those fields through `normalizeAgentPayload()` before use.

## Solution Assessment

No. Direct-only validation is not the best fix for this security hardening because mapped hooks expose the same metadata surface; the narrow maintainable approach is one shared validator called from both dispatch paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Validate mapped hook metadata too:** `src/gateway/hooks.ts:25-30`
  - body: The new guards only run inside `normalizeAgentPayload()`, which protects direct `/hooks/agent` requests. Mapped endpoints render or transform the same `name`, `sessionKey`, `to`, `model`, and `thinking` fields and dispatch them without that normalizer, so callers can still reach the downstream replay/logging/storage paths with oversized metadata. Move these checks into a shared validator and call it for mapped agent actions before replay-key construction.
  - confidence: 0.93
