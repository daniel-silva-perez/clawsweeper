## Summary

Keep open. Current main still lacks an opt-in macOS LaunchDaemon/system service mode for `openclaw gateway install`, and managed install/start/stop/restart/uninstall/status still target the per-user LaunchAgent path and GUI launchd domain. The issue discussion is useful and current: the reporter supplied a tested LaunchDaemon workaround, a later comment confirmed the stale TMPDIR failure on OpenClaw 2026.4.7, and the existing Codex review correctly identified only partial adjacent fixes on main.

## Best Possible Solution

Keep this issue open and design an explicit macOS system-service mode. The likely product path is `openclaw gateway install --system` or equivalent, generating `/Library/LaunchDaemons/ai.openclaw.gateway.plist` with safe ownership/permissions, `UserName`, stable service env, and matching status/start/stop/restart/uninstall/doctor support for the system launchd domain, coordinated with #14251 and #55208.

## Risks / Open Questions

- A correct LaunchDaemon implementation is security-sensitive: it needs root-owned plist permissions, explicit `UserName`, durable env/state paths, careful secret handling, system-domain launchctl operations, and safe uninstall/doctor behavior.
- Closing now would lose the central boot-without-login product request; related #14251 and #55208 are adjacent open threads, but neither clearly supersedes this broader install/status request.
