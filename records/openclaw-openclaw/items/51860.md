## Summary

Keep open. Current main still does not implement the requested macOS system LaunchDaemon mode for `openclaw gateway install` or lifecycle/status management. It has partial adjacent improvements: durable Darwin TMPDIR handling and `gateway status --deep` discovery for system-level services, but managed install/start/stop/restart/uninstall still targets the per-user LaunchAgent and GUI launchd domain.

## Best Possible Solution

Keep this issue open. The best product path is an explicit macOS system-service design: add opt-in `openclaw gateway install --system` or equivalent, generate `/Library/LaunchDaemons/ai.openclaw.gateway.plist` with safe ownership/permissions and `UserName`, use durable service env/state paths, teach status/start/stop/restart/uninstall/doctor to target the system launchd domain, and coordinate the lifecycle details with #14251 and #55208.

## Risks / Open Questions

- A correct LaunchDaemon implementation is security-sensitive: it needs root-owned plist permissions, explicit `UserName`, durable env/state paths, system-domain launchctl operations, and safe uninstall/doctor behavior.
- Closing now would lose the central boot-without-login product request; current main only fixes part of the TMPDIR failure mode and adds best-effort discovery for external services.
