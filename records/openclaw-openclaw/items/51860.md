## Summary

Keep open. Current main still treats the macOS gateway service as a per-user LaunchAgent: install writes under the user LaunchAgents directory, lifecycle/status use the GUI launchd domain, and the CLI has no `--system` or `--daemon` selector. `gateway status --deep` can discover extra LaunchDaemons and TMPDIR is now durable for LaunchAgent installs, but neither implements the requested boot-without-login LaunchDaemon install/status/lifecycle support.

## Best Possible Solution

Keep this issue open and implement an explicit macOS system-service mode, likely `openclaw gateway install --system` or `--daemon`, while preserving the current per-user LaunchAgent default. The implementation should generate `/Library/LaunchDaemons/ai.openclaw.gateway.plist` with safe ownership/permissions, `UserName`, durable environment/state paths, and matching status/start/stop/restart/uninstall/doctor support for the system launchd domain, with docs and tests coordinated with #14251 and #55208.

## Risks / Open Questions

- A correct LaunchDaemon implementation is security-sensitive: it needs root-owned plist permissions, explicit `UserName`, stable env/state paths, careful secret handling, system-domain launchctl operations, and safe uninstall/doctor behavior.
- The overlap with #14251 and #55208 should be coordinated so install, status, stop, restart, uninstall, and doctor do not diverge between LaunchAgent and LaunchDaemon modes.
