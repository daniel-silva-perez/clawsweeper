## Summary

Keep this issue open. Current main still only manages the macOS Gateway as a per-user LaunchAgent in the GUI launchd domain. It has partial fixes for the related stale TMPDIR symptom and can discover LaunchDaemons with deep status scans, but it does not provide `gateway install --system`, `/Library/LaunchDaemons` plist generation, `UserName` handling, or system-domain status/lifecycle management.

## Best Possible Solution

Keep this issue open. The best product path is an explicit macOS system-service design: add an opt-in `openclaw gateway install --system`, generate `/Library/LaunchDaemons/ai.openclaw.gateway.plist` with safe ownership/permissions and `UserName`, use durable service env/state paths, teach status/start/stop/restart/uninstall/doctor to target the system launchd domain, and document the operational and security tradeoffs. Coordinate with #55208 and #14251 rather than closing this report prematurely.

## Risks / Open Questions

- A correct LaunchDaemon implementation is security-sensitive: it needs root-owned plist permissions, explicit `UserName`, stable state/TMP handling, system-domain launchctl operations, and safe uninstall/doctor behavior.
- Closing as implemented would lose the central boot-without-login request; current main only fixes part of the TMPDIR failure mode and adds discovery for external services.
