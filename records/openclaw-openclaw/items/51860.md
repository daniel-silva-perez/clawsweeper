## Summary

Current main partially fixes the related stale macOS TMPDIR symptom, but the core request remains open: macOS gateway service management is still per-user LaunchAgent/gui-domain only, with no `gateway install --system` LaunchDaemon mode and no lifecycle/status support for a managed system-domain LaunchDaemon.

## Best Possible Solution

Keep this issue open. If maintainers want first-class boot-without-login support on macOS, implement an explicit system-service mode: add `openclaw gateway install --system`, generate `/Library/LaunchDaemons/ai.openclaw.gateway.plist` with safe ownership/permissions and `UserName`, use the durable service TMPDIR/state path, teach status/start/stop/restart/uninstall/doctor to target the system launchd domain, and document the operational and security tradeoffs. If maintainers decide OpenClaw should not manage system LaunchDaemons, close by explicit product decision rather than as already implemented.

## Risks / Open Questions

- The remaining implementation is security-sensitive: a system LaunchDaemon path would need root-owned plist permissions, explicit `UserName` handling, correct `OPENCLAW_HOME`/state ownership, system-domain launchctl status, and safe uninstall/doctor behavior.
- Closing as implemented would lose the central boot-without-login request; current main only addresses the related durable TMPDIR failure.
