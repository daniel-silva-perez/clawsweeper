## Summary

Keep open. Current main still turns Feishu inbound thread metadata into an effective thread reply despite disabled config; the reporter's compiled-file truthiness diagnosis is not the current TypeScript boundary, but the observable bug remains. Open PR #39507 overlaps this behavior and is still unmerged, so this should not be closed as implemented.

## What This Changes

The issue asks for Feishu `replyInThread: "disabled"` to prevent `reply_in_thread: true` on quoted/thread replies so bot replies stay inline and mentions are preserved.

## Best Possible Solution

Make the Feishu runtime semantics match the config contract: if disabled is a hard off switch, stop deriving `replyInThread` or dispatcher thread mode from inbound `root_id`/`thread_id` under disabled config, preserve explicit enabled/topic-session behavior, and update the current regression tests. If maintainers intentionally want implicit thread following, update schema and docs to say so instead.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: configure a Feishu group or account with `replyInThread: "disabled"`, receive a group event with `thread_id` or `root_id`, and current main resolves and dispatches `replyInThread: true` without needing live Feishu delivery.

## Solution Assessment

No for the reporter’s suggested current-source fix. The send helpers already take a boolean; the safer fix belongs in `extensions/feishu/src/bot-content.ts` and `extensions/feishu/src/reply-dispatcher.ts`, with tests covering disabled account/group config and inbound thread metadata.
