## Summary

Keep this PR open for maintainer review. Current main has already superseded the early startup-log wording, but the branch still contains unmerged channel-summary and Telegram/Feishu recovery work for the linked Windows startup report; as diffed, it needs a rebase and fixes for current webhook-confirmation and Feishu timeout semantics.

## What This Changes

This PR updates gateway startup/readiness logs, adds a per-channel sidecar health summary, and changes Telegram webhook cleanup plus Feishu bot-identity startup probing behavior and tests.

## Best Possible Solution

Rebase or split the PR on current main, keep the already-shipped gateway listening semantics, add the channel health summary only with wording that matches handoff-state reality, and land the Telegram/Feishu hardening with current webhook confirmation, Feishu startup timeout configurability, typed grammY AbortSignal usage, and focused regression tests.

## Reproduction Assessment

Yes. The review findings have source-level reproductions: the Feishu regression follows by setting `OPENCLAW_FEISHU_STARTUP_PROBE_TIMEOUT_MS` above 10000 and observing the PR's explicit 10s override, while the Telegram regression is covered by current tests that mock `deleteWebhook` failure plus a non-empty or unavailable `getWebhookInfo` response.

## Solution Assessment

No, not as currently diffed. The product direction is reasonable, but the implementation should preserve current main's webhook-confirmation behavior and Feishu startup timeout contract, and should avoid new `any` suppressions because the pinned grammY API already exposes an AbortSignal parameter.
