## Summary

Keep this PR open but do not merge it as-is. The PR usefully follows up the Windows startup/channel-recovery report and the author addressed earlier Greptile notes about background retry timeout and deterministic summary ordering, but the latest diff still regresses current Telegram webhook-confirmation behavior and bypasses Feishu's startup-specific timeout contract.

## What This Changes

This PR updates gateway startup/readiness logs, adds a per-channel sidecar health summary, and changes Telegram webhook cleanup plus Feishu bot-identity startup probing behavior and tests.

## Best Possible Solution

Rebase or split the PR on current main, keep the shipped `http server listening (...)` gateway milestone, add the channel health summary with wording that matches handoff-state reality, and land Telegram/Feishu hardening only while preserving webhook confirmation, Feishu startup timeout configurability, typed grammY AbortSignal usage, and focused regression coverage.

## Reproduction Assessment

Yes. The Telegram regression is reproducible from the existing mock path where `deleteWebhook` fails and `getWebhookInfo` reports a non-empty or unavailable webhook; the Feishu regression is reproducible by setting `OPENCLAW_FEISHU_STARTUP_PROBE_TIMEOUT_MS` above 10000 and observing the PR's explicit 10s override.

## Solution Assessment

No. The product direction is reasonable, but this diff is not the narrowest maintainable solution because it drops a documented Telegram safety check and overrides Feishu's startup-specific timeout; preserving those contracts while adding the useful logging and transient-failure handling is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep webhook confirmation before polling:** `extensions/telegram/src/polling-session.ts:256`
  - body: Returning `ready` for any recoverable `deleteWebhook` failure skips the current `getWebhookInfo` confirmation. When Telegram still has a webhook URL, current main retries cleanup before polling; this path would start polling while the webhook may still be active, regressing the covered recovery behavior.
  - confidence: 0.88
- **[P2] Preserve the Feishu startup timeout override:** `extensions/feishu/src/monitor.ts:78`
  - body: Passing `FEISHU_PROBE_REQUEST_TIMEOUT_MS` here forces monitor startup back to the generic 10s probe budget. `fetchBotIdentityForMonitor` already defaults to the startup-specific 30s/env-configurable timeout, so this bypasses `OPENCLAW_FEISHU_STARTUP_PROBE_TIMEOUT_MS`.
  - confidence: 0.9
