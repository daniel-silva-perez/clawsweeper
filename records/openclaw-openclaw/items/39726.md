## Summary

Keep this PR open. Current main still requires `tools.exec.notifyOnExitEmptySuccess` to be explicitly true before a successful no-output background exec sends a completion wake, while this PR remains a focused implementation candidate for the chat-session default and has addressed the earlier opt-out/provider-label review concerns.

## Best Possible Solution

Review and either land or intentionally decline this PR based on the chat-surface notification policy. If accepted, keep the current narrow shape: default empty-success notifications only for real reply-capable delivery contexts with a channel and target, preserve explicit `notifyOnExitEmptySuccess: false`, keep generic provider labels silent, and retain the regression tests plus changelog credit.
