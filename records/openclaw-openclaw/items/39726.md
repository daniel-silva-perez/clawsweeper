## Summary

Keep this PR open. Current main still requires `notifyOnExitEmptySuccess` to be explicitly true, so reply-capable chat background execs that exit 0 with no stdout/stderr remain silent by default. The PR is not obsolete; its latest diff is a focused implementation plus regression coverage and changelog entry, and the earlier opt-out/generic-provider review concerns are addressed in the current head.

## Best Possible Solution

Keep this PR open for maintainer review and likely landing if the product behavior is accepted. The best implementation path is the current narrowed PR shape: default empty-success notifications only for real reply-capable chat delivery contexts when the config value is unset, preserve explicit `false`, keep generic provider labels such as `openai` silent, update docs/changelog as accepted, and run the repo’s changed gate before merge.
