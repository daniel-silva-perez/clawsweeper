## Summary

Keep this PR open. Current main still preserves the explicit-only `notifyOnExitEmptySuccess` behavior, so reply-capable chat sessions do not get quiet-success background exec completion wakes by default. The PR remains a focused implementation candidate, and the provided latest head diff shows the earlier opt-out and generic-provider concerns were repaired rather than superseded on main.

## Best Possible Solution

Review and either land or intentionally decline this PR based on the chat-surface notification policy. If accepted, keep the narrow repaired shape: default empty-success notifications only for real reply-capable delivery contexts with a channel and target, preserve explicit `notifyOnExitEmptySuccess: false`, keep generic/non-chat provider labels silent, and retain the regression tests plus changelog credit.
