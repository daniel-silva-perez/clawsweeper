## Summary

Keep this PR open. Current main still has the Telegram reaction path as generic emoji-only system-event enqueueing, with no `reactionSemantics` config, no stable `reaction_key`, no Telegram `custom_emoji` handling, and no heartbeat wake from mapped reactions. The PR is also explicitly paired with the still-open #47677 request and relates to the still-open #64752 wake bug, so it remains a live implementation candidate rather than cleanup to close.

## Best Possible Solution

Keep this PR open as the active implementation candidate for #47677 and related wake report #64752. Maintainers should review the Telegram-owned config/runtime design, decide whether the wake helper belongs in a narrower SDK subpath or the existing plugin runtime API, and then land a refined version with schema/docs/tests covering standard emoji, Telegram custom emoji, `wake|queue|ignore`, authorization, duplicate suppression, and queue-only generic notifications.
