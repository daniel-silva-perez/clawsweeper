## Summary

Current main still has the reported stale-display path: heartbeat-aware runners know the run is a heartbeat, but session usage persistence has no heartbeat preservation mode and still writes the just-used model/modelProvider into the session entry that display surfaces resolve. The direct fix PR #62987 is closed unmerged, so this remains valid and narrowly repairable.

## What This Changes

The issue requests that heartbeat turns use their configured per-turn model without overwriting the session entry's persisted model/modelProvider used by UI and history display surfaces.

## Best Possible Solution

Add a heartbeat-aware preservation mode to the session usage persistence boundary so heartbeat turns keep the existing persisted session model identity while still recording normal usage, token, context, cost, system-prompt, and non-heartbeat updates. Cover both persistence branches with a regression test that proves display resolution remains on the session/default model after a heartbeat run that executed on a different heartbeat model.

## Repair Work Prompt

Fix the heartbeat session-display persistence bug tracked by #62954. Current main lets heartbeat turns execute on agents.defaults.heartbeat.model, but persistSessionUsageUpdate still writes the heartbeat run's modelUsed/providerUsed into SessionEntry.model/modelProvider, which makes resolveSessionModelRef, sessions.list, chat.history-backed UI, and model picker defaults show the heartbeat model until a normal turn runs. Add an isHeartbeat or preserveSessionModelIdentity parameter to persistSessionUsageUpdate and persistRunSessionUsage, pass opts?.isHeartbeat from runReplyAgent and createFollowupRunner, and when true preserve entry.model and entry.modelProvider in both the usage/fresh-context branch and the model/context-only branch. Keep normal non-heartbeat persistence unchanged, and do not change estimatedCostUsd, token totals, contextTokens, systemPromptReport, CLI session binding, or heartbeat cost policy unless required for the narrow fix. Treat closed PR #62987 as useful source context but implement against current main. Add focused regression coverage, preferably in src/auto-reply/reply/session.test.ts, that seeds a session with a default/main model, persists a heartbeat usage update with a different heartbeat model/provider, and asserts stored model/modelProvider and resolveSessionModelRef/list display remain on the original session/default model; cover the no-usage/model-only branch too. Also keep or run existing heartbeat execution selection coverage so heartbeat.model still drives the actual model call. Add a user-facing changelog fix entry if this changes shipped behavior, crediting Thanks @tumbleweedlabs and any adapted prior work appropriately; keep the reporter's CLI binding and cost attribution questions as separate follow-up scope.
