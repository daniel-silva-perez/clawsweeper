## Summary

Keep open. Current main still has the reported persistence/display path: heartbeat-aware runners compute the runtime heartbeat model, but session usage persistence has no heartbeat preservation signal and still writes that runtime model/provider into the session entry consumed by display surfaces. The direct fix PR is closed unmerged, so this remains a narrow valid bug.

## What This Changes

Heartbeat turns should stop overwriting the persisted session `model` and `modelProvider` fields used by Control UI, `chat.history`, and other session model display surfaces.

## Best Possible Solution

Add a heartbeat-aware preservation mode at the session usage persistence boundary so heartbeat turns keep the existing persisted session model/provider identity, while normal turns continue updating runtime model metadata and all usage, token, cost, context, and system-prompt fields keep their current semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: invoke `persistSessionUsageUpdate` for an existing session with a heartbeat `modelUsed`/`providerUsed`, then resolve the display model through `resolveSessionModelRef`; current main will surface the heartbeat model from the persisted runtime fields.

## Solution Assessment

Yes. The narrowest maintainable fix is to thread heartbeat state, or an explicitly named preservation flag, into `persistSessionUsageUpdate`/`persistRunSessionUsage` and preserve only `model` and `modelProvider` for heartbeat turns; the adjacent cost and CLI binding questions should stay out of scope unless maintainers intentionally broaden the work.
