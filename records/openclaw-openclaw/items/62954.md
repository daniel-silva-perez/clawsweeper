## Summary

Current main still appears to have the stale heartbeat display bug. The session usage persistence layer has no heartbeat preservation parameter and still writes the just-used model/modelProvider into the session entry. Both the main reply runner and follow-up runner know when a run is a heartbeat, but neither passes that state into persistence. Gateway chat/session display paths still resolve model identity from those persisted fields. The direct fix PR #62987 is closed unmerged, and the later cost snapshot fix in the same area did not address model/modelProvider preservation.

## Best Possible Solution

Keep this issue open and land a narrow persistence fix: add an isHeartbeat or preservation-mode parameter to persistSessionUsageUpdate/persistRunSessionUsage, pass it from runReplyAgent and createFollowupRunner, and preserve entry.model and entry.modelProvider during heartbeat turns in both persistence branches. Add a focused regression test showing a heartbeat run can execute on the heartbeat model without changing the session model reported by resolveSessionModelRef/chat.history.
