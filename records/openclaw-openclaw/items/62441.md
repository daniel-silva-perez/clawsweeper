## Summary

Keep this draft PR open. Current main does not implement the proposed sender-name sanitizer, and the PR still looks unsafe as a merge candidate because it mutates `senderName` on paths that current main uses for group tool policy. The linked bug remains plausible, but the current code/dependency path does not yet prove that Telegram display names are emitted as OpenAI chat `messages[].name`; the fix should wait for a captured failing outbound payload or failing regression test.

## Best Possible Solution

Keep the PR draft/open for rework. The best fix is to capture the actual rejected outbound payload for #61192, add a failing regression test, preserve raw `senderName` for `toolsBySender` and plugin policy matching, and sanitize only the proven provider-bound field at the serializer/transport boundary.

## Risks / Open Questions

- Functional risk: the PR can break existing `toolsBySender` `name:<displayName>` policies because names like `John Doe` become `John_Doe`.
- Security/authorization risk: sanitized display-name collisions can route a sender into the wrong allow/deny policy if policy matching sees the sanitized value instead of the raw identity.
- Root-cause risk: the inspected current main and pinned pi-ai serializer do not prove the reported OpenAI `messages[].name` rejection path, so the PR may not fix the actual 400 while still changing policy behavior.
