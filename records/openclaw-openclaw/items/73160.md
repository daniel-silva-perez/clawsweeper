## Summary

Keep open. Current main has partial lock-based main-session restart recovery, but the timeout path still reports failure without a transcript-backed import or explicit usable-history fallback; the workspace-first workaround in the discussion is useful operational context, but it does not solve the core recovery gap.

## What This Changes

Request restart-aborted main sessions to retry, fall back, or reconstruct/import context from persisted JSONL transcripts when the Gateway resume RPC times out instead of orphaning the prior session context.

## Best Possible Solution

Fold this into the Gateway graceful-restart recovery track: keep the old main-session binding recoverable while Gateway readiness settles, retry with explicit recovery state, and add a bounded transcript-backed recall/import path or clear user-visible failure result when resume cannot be accepted.

## Reproduction Assessment

Yes at source level: create a running main-session row with `abortedLastRun: true` and a resumable transcript tail, then make `callGateway` time out; `recoverRestartAbortedMainSessions` returns `failed` and does not attach the transcript to a replacement session. A live reproduction should use the reported Windows/Telegram restart timing.

## Solution Assessment

No, current main is not the complete fix. The synthetic resume RPC is a useful partial recovery path, but the durable context-loss problem needs retry/readiness handling plus a transcript-backed fallback or explicit recovery UX.
