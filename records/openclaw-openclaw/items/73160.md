## Summary

Keep open. Current main has partial lock-based main-session recovery and bounded retry scheduling, but a Gateway `agent` RPC timeout still counts recovery as failed without transcript-backed import, active-history reattachment, or a user-visible usable-history fallback; the broader graceful-restart design remains open in #57425.

## What This Changes

The issue asks OpenClaw to preserve or reconstruct main-session context after a Gateway restart when restart recovery times out instead of orphaning the existing transcript and starting a fresh session.

## Best Possible Solution

Fold this into the Gateway graceful-restart recovery track: keep the old main-session binding recoverable while Gateway readiness settles, retry with explicit recovery state, and add a bounded transcript-backed recall/import path or clear user-visible recovery failure when resume cannot be accepted.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a `sessions.json` row with `status: "running"`, `abortedLastRun: true`, and a resumable transcript tail calls `callGateway` with a 10 second timeout; if that call times out, recovery returns `failed` and does not attach the transcript to a replacement session.

## Solution Assessment

No. Current main's synthetic resume RPC and retry loop are useful partial recovery, but they do not solve durable context loss when the RPC keeps timing out or the old transcript becomes detached from active session history.
