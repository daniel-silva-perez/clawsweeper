## Summary

Keep open. Current main has partial lock-based main-session restart recovery, but a Gateway timeout still leaves the recovery result failed without a transcript-backed import, replacement-session attachment, or user-visible usable-history fallback; the broader graceful-restart issue is related but this newer timeout path has concrete, distinct evidence worth preserving.

## What This Changes

The issue asks for main-session restart recovery to retry or fall back to durable transcript history when a post-restart resume attempt times out, instead of orphaning the old context and starting fresh.

## Best Possible Solution

Fold this into the Gateway graceful-restart recovery track: keep the old main-session binding recoverable while Gateway readiness settles, retry with explicit recovery state, and add a bounded transcript-backed recall/import path or clear user-visible failure result when resume cannot be accepted.

## Reproduction Assessment

Yes at source level: create a running main-session store row with abortedLastRun true and a resumable transcript tail, then make callGateway time out; recoverRestartAbortedMainSessions returns failed and does not attach the transcript to a replacement session. A live reproduction should still validate the reported Windows/Telegram restart timing.

## Solution Assessment

No, current main is not a complete solution. The synthetic resume RPC is a useful partial recovery path, but the durable context-loss problem needs readiness-aware retry plus a transcript-backed fallback or explicit recovery UX.
