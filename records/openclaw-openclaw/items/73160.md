## Summary

Keep open. Current main has partial restart-aborted main-session recovery, but the timeout path still returns a failed recovery without a transcript-backed import or explicit durable context handoff, and related graceful-restart work remains open.

## What This Changes

The issue requests reliable main-session recovery after a Gateway restart timeout so existing transcript context is retried, imported, or otherwise made accessible instead of being orphaned.

## Best Possible Solution

Fold this into the Gateway graceful-restart recovery design: preserve the old main-session binding while recovery is pending, retry after Gateway readiness, and if resume cannot be accepted, surface a clear recovery result plus a bounded transcript-backed history/import path that cannot inherit stale approvals or auto-execute old work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a running main-session row with `abortedLastRun: true` and a resumable transcript tail, then make the `agent` RPC timeout; current recovery reports failure without importing the transcript into a replacement session. A full live repro should still use the reporter's Windows/Telegram restart timing.

## Solution Assessment

No. The current implementation is a useful partial resume path, but it is not the best completed solution for durable context recovery after timeout; the safer path is a maintainer-designed fallback aligned with the broader restart-recovery track.
