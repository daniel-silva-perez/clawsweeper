## Summary

Keep open. Current main has partial lock-based main-session recovery and bounded retry scheduling, but a Gateway `agent` RPC timeout still leaves the recovery counted as failed without a transcript-backed import, active-history attachment, or user-visible usable-history fallback; the broader graceful-restart work remains open in #57425.

## What This Changes

The issue requests reliable main-session recovery after gateway restart timeouts, including retry/readiness handling and a fallback that preserves or imports the old transcript context.

## Best Possible Solution

Fold this into the Gateway graceful-restart recovery track: keep the old main-session binding recoverable while Gateway readiness settles, retry with explicit recovery state, and add a bounded transcript-backed recall/import path or clear user-visible recovery failure when resume cannot be accepted.

## Reproduction Assessment

Yes. At source level, a `sessions.json` row with `status: "running"`, `abortedLastRun: true`, and a resumable transcript tail will call `callGateway` with a 10 second timeout; if that call times out, recovery returns `failed` and does not attach the transcript to a replacement session.

## Solution Assessment

No. Current main's synthetic resume RPC and retry loop are useful partial recovery, but they do not solve durable context loss when the RPC keeps timing out or the old transcript becomes detached from active session history.
