## Summary

Keep open. Current main has added partial main-session restart recovery, including stale-lock marking, bounded retries, and a synthetic resume message, but the reported timeout path still ends as a failed recovery with no transcript-backed attach/import or user-visible durable context fallback. The filesystem/workspace-first workaround in the discussion is useful operational context, but it does not resolve the OpenClaw recovery gap.

## What This Changes

The issue requests durable main-session recovery after Gateway restart timeouts, including retry or transcript-backed context restoration instead of silently losing the previous conversation context.

## Best Possible Solution

Fold this into the graceful Gateway restart recovery design: keep the interrupted main-session binding recoverable while Gateway readiness settles, make retry and failure state explicit, and add a bounded transcript-backed attach/import or clear user-visible recovery failure path when live resume cannot be accepted.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a running main session marked `abortedLastRun: true` with a resumable transcript tail calls Gateway `agent` with a 10 second timeout; repeated timeout returns `failed` without attaching/importing the old transcript into the active replacement session.

## Solution Assessment

No. The current RPC-only recovery path is a useful partial mitigation, but it is not the best complete solution because it can still give up after bounded Gateway timeouts and leave durable transcript history detached from the active session.
