## Summary

Keep open. Current main has partial restart recovery for interrupted main sessions, but it still depends on a 10s Gateway `agent` RPC, lacks a transcript-backed import/fallback path for main sessions after repeated timeout, and related graceful-restart work remains unresolved.

## What This Changes

The issue requests robust recovery of interrupted main-session context after Gateway restart timeouts, including retries or transcript-backed context import instead of silently moving to a fresh session.

## Best Possible Solution

Treat this as part of the Gateway graceful-restart recovery track: preserve the old main-session binding while recovery is pending, retry after Gateway readiness, and if resume cannot be accepted, surface an explicit recovery result plus a bounded transcript-backed history/import path instead of silently losing usable context.

## Reproduction Assessment

Yes, at source level: create a `running` main-session row with `abortedLastRun: true` and a resumable transcript tail, then make `callGateway` time out; `recoverRestartAbortedMainSessions` returns `failed=1` and does not import the transcript into a replacement session. A full live reproduction should use the Windows/Telegram restart timing from the report.

## Solution Assessment

No for closing as solved. The current implementation is a useful partial retry path, but the report’s durable context-recovery problem still needs a main-session fallback or explicit failure UX, preferably aligned with the broader #57425 restart-recovery design.
