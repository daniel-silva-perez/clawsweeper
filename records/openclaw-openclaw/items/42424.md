## Summary

Keep open. Current main still has the core Discord self-message gap: bot identity lookup can return no bot user id, the own-bot filters only run when that id is present, and replay/history context still does not carry trusted Discord author ids needed to re-filter contaminated history. Current main does partially address the `/new` persistence symptom by clearing session queues on rollover, but that does not solve failed-open ingress or replay/followup filtering.

## Best Possible Solution

Keep this issue open and implement a Discord-owned defense-in-depth fix: make bot identity reliable or fail closed using the already-resolved application-id/token fallback where appropriate, carry trusted Discord author/bot metadata through pending history and queued followups, re-apply own-bot filtering before replay/followup execution, and add regression tests for `fetchUser("@me")` failure, self-authored history replay, followup queue replay, and session rollover queue cleanup.

## Risks / Open Questions

- A fix must preserve intentional `allowBots: true` and `allowBots: "mentions"` behavior for other bots while always rejecting OpenClaw's own Discord bot account.
- Replay filtering should use trusted Discord ids or explicit channel metadata, not display labels, to avoid false positives with renamed bots, relay bots, and PluralKit-style identities.
- `HistoryEntry` is exported through the plugin SDK, so adding metadata should be backward-compatible and covered by SDK-facing tests.
