## Summary

Keep open. Current main still only blocks Discord own-bot messages when `botUserId` was resolved; `fetchDiscordBotIdentity` still returns an undefined bot id on `fetchUser("@me")` failure, and replay/history data still lacks trusted Discord author identity for re-filtering contaminated history. Current main partially addresses `/new` persistence by clearing queues on session rollover, but the core self-message/replay gap remains.

## Best Possible Solution

Keep this issue open and implement the Discord-owned defense-in-depth fix: make bot identity reliable or fail closed using the existing application-id/token fallback where appropriate, carry trusted Discord author/bot metadata through pending history and queued followups, re-apply own-bot filtering before replay/followup execution, and add regression tests for `fetchUser("@me")` failure, self-authored history replay, and session rollover queue cleanup.

## Risks / Open Questions

- A fix must preserve intentional `allowBots: true` and `allowBots: "mentions"` behavior for other bots while always rejecting OpenClaw's own Discord bot account.
- Replay filtering should use trusted Discord ids or explicit channel metadata, not display labels, to avoid false positives with renamed bots, relay bots, and PluralKit-style identities.
- `HistoryEntry` is exported through the plugin SDK, so metadata additions should be backward-compatible.
