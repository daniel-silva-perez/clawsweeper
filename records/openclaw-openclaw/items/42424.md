## Summary

Keep open. Current main partially addresses the `/new` queue-cleanup concern, but the core Discord self-message gap remains: bot identity lookup can still fail open, live self-message filters are still conditional on a resolved `botUserId`, and replay/history data still lacks trusted Discord author identity needed to re-filter contaminated history. The related implementation PR #46338 is closed unmerged, so there is no shipped full fix to close against.

## Best Possible Solution

Keep this issue open and implement the remaining Discord-owned fix: make bot identity reliable or fail-closed, carry trusted Discord author/bot metadata through pending history and queued followups, re-apply own-bot filtering before replay/followup execution, and add regression tests for failed `fetchUser("@me")`, self-authored pending-history replay, and the already-partial `/new` queue cleanup behavior.

## Risks / Open Questions

- A fix must preserve intentional `allowBots: true` and `allowBots: "mentions"` behavior for other bots while fail-closing OpenClaw's own Discord bot account.
- Replay filtering should use trusted Discord ids or explicit channel metadata, not display labels, to avoid false positives with renamed bots, relay bots, and PluralKit-style identities.
- `HistoryEntry` is exported through the plugin SDK reply-history surface, so any metadata addition should be backward-compatible.
