## Summary

Current main still has the reported Discord self-message gap: bot identity lookup can fail open, the live ingress/preflight own-message filters remain conditional on a resolved `botUserId`, and replay/followup data still does not carry trusted Discord author/bot identity needed to re-filter contaminated history. The related implementation PR #46338 is closed unmerged, so there is no shipped fix to close against.

## Best Possible Solution

Keep this issue open and implement a Discord-owned fix: make bot identity reliable or fail-closed, carry trusted Discord author/bot metadata through pending history and queued followups, re-apply own-bot filtering before replay/followup execution, and add regression tests for failed `fetchUser("@me")`, pending-history replay, and `/new` queue cleanup.

## Risks / Open Questions

- A fix needs to preserve intentional `allowBots=true` and `allowBots="mentions"` behavior for other bots while fail-closing OpenClaw's own Discord bot account.
- Replay filtering should use trusted Discord ids or explicit channel metadata, not display labels, to avoid false positives with renamed bots, relay bots, and PluralKit-style identities.
- `HistoryEntry` is exported through `openclaw/plugin-sdk/reply-history`, so adding metadata should be backwards-compatible for plugin consumers.
