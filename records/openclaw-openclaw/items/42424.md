## Summary

Keep open. Current main still has the core bug: Discord own-message filtering depends on a resolved `botUserId`, that identity still becomes `undefined` when `fetchUser("@me")` fails, and replay/followup state still lacks trusted Discord bot/author identity needed to re-filter contaminated history. The newer reset cleanup clears queued runtime state during session rollover, but it does not prevent a self-authored Discord message from entering ingress/history/followup paths first. The related implementation PR #46338 is closed unmerged, so it is not a shipped fix.

## Best Possible Solution

Keep this issue open. Implement a Discord-owned fix that makes bot identity reliable or fail-closed, carries trusted Discord author/bot identity through pending history and queued followup metadata, re-applies own-bot filtering before replay/followup execution, and keeps the current `/new` queue cleanup covered by regression tests for failed `fetchUser("@me")`, pending-history replay, and queued followups.

## Risks / Open Questions

- A fix must preserve intentional `allowBots=true` and `allowBots="mentions"` behavior for other bots while fail-closing OpenClaw's own Discord bot account.
- Replay filtering should use trusted Discord ids rather than display labels to avoid false positives with renamed bots, relay bots, or PluralKit-style identity handling.
- Extending shared history/followup metadata should remain backwards-compatible for plugin SDK consumers because `HistoryEntry` is exported through `openclaw/plugin-sdk/reply-history`.
