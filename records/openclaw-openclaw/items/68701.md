## Summary

Keep this PR open, but do not merge it as-is. Current main partially addresses WebChat external-route inheritance at the `chat.send` boundary, but the PR's normalization guard still preserves arbitrary structured non-WebChat recipients such as `telegram:...` or `channel:...`, which undercuts the stated wrong-recipient fix.

## What This Changes

The PR changes `normalizeDeliveryContext` to drop unstructured `to` values for `webchat` delivery contexts while preserving colon-prefixed structured targets, and adds regression coverage for a valid WebChat target and a phone-number-shaped target.

## Best Possible Solution

Adapt the fix to the current route-normalization path and preserve only WebChat-owned targets, or clear `to` for WebChat contexts if there is no supported WebChat direct-target contract. Add focused coverage for valid `webchat:...`, phone-number-shaped stale values, and structured non-WebChat values such as `telegram:...` or `channel:...`.

## Reproduction Assessment

Yes for the PR defect: from the supplied diff, `normalizeDeliveryContext({ channel: "webchat", to: "telegram:123" })` would keep the recipient because it only checks for a generic structured prefix. Current-main `extractDeliveryInfo` then shows why any preserved `channel + to` context remains routable downstream.

## Solution Assessment

No. The proposed approach is close but too broad because it distinguishes only structured from unstructured targets, not WebChat-owned targets from stale cross-channel targets; the safer fix should validate the target prefix against the normalized channel or the current channel-route contract.
