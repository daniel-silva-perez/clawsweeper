## Summary

Keep this PR open for maintainer handling rather than cleanup closure. Current main still has no Linux, Windows, or unified desktop companion app, and the vision/platform docs still describe those apps as planned first-party platform work; however, this branch is not mergeable as-is because the submitted app scaffolds have build, runtime, packaging, and security blockers and overlap with the active desktop follow-up in #73315 plus canonical #75.

## What This Changes

The PR adds separate Tauri v1 Linux and Windows app scaffolds under `apps/linux` and `apps/windows`, including Cargo/npm manifests, Tauri configs, Rust command/gateway/pairing modules, and a small React/Vite UI only for Linux.

## Best Possible Solution

Consolidate the Linux/Windows companion work around the canonical platform request and the active unified desktop app direction, then land one maintained implementation with workspace and lockfile integration, committed app assets, secure Tauri defaults, real gateway/pairing behavior, docs, labeler/release wiring, and Linux plus Windows build validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Move the Tauri build scripts to crate roots:** `apps/linux/src/build.rs:1-3`
  - body: Cargo only auto-discovers `build.rs` at the package root unless `[package].build` points elsewhere. Both new crates put `tauri_build::build()` under `src/build.rs` and omit a build override, so the Tauri build script is skipped and generated context setup is not run. Move both build scripts to the crate roots or declare the build path in both manifests.
  - confidence: 0.94
- **[P1] Add the missing Windows Vite entry files:** `apps/windows/package.json:6-8`
  - body: The Windows package exposes `npm run dev` and `npm run build` through Vite, but this PR does not add `apps/windows/index.html`, `apps/windows/src/main.tsx`, or an app component. A clean Windows app build therefore has no frontend entry to serve or bundle.
  - confidence: 0.93
- **[P1] Commit the bundle icon assets referenced by Tauri:** `apps/linux/tauri.conf.json:31-37`
  - body: Both Tauri configs reference `icons/32x32.png`, `icons/128x128.png`, `icons/icon.icns`, and `icons/icon.ico`, but the PR adds no `icons` directories or files. Release packaging will fail or depend on untracked local files until those assets are committed or the references are removed.
  - confidence: 0.91
- **[P2] Return from gateway connection setup after the handshake:** `apps/linux/src/gateway.rs:36-42`
  - body: `connect()` awaits the WebSocket receive loop inline, so the method does not return while the socket stays open. Any Tauri command wired to this helper would hang instead of responding to the frontend; move the receive loop into a spawned task and keep explicit connection state handling.
  - confidence: 0.88
- **[P2] Replace stubbed gateway commands with real wiring:** `apps/linux/src/commands.rs:22-25`
  - body: The Tauri commands report success without using the `GatewayConnection`, the provided URL, or the API key. The UI can display a connected state even when no gateway connection or pairing occurred, which makes the submitted app non-functional for the feature it claims to add.
  - confidence: 0.86
- **[P2] Set a restrictive CSP before enabling a desktop WebView:** `apps/linux/tauri.conf.json:40-42`
  - body: Both Tauri configs set `"csp": null`. For a desktop app that talks to the gateway, shipping with no injected Content Security Policy unnecessarily widens the WebView injection surface; add a policy scoped to the app and expected gateway origins.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a new desktop WebView and dependency surface with disabled CSP and missing reproducibility/release integration, so it needs security review before any merge path.

Concerns:

- **[medium] CSP disabled for new desktop WebViews:** `apps/linux/tauri.conf.json:42`
  - body: Both `tauri.conf.json` files set `security.csp` to `null`; Tauri's schema describes CSP injection as an important WebView security control. A first-party gateway companion should not ship without a restrictive policy for app resources and gateway WebSocket/HTTP origins.
  - confidence: 0.86
- **[medium] New package graphs lack repo integration:** `apps/linux/package.json:10`
  - body: The PR adds new npm and Rust dependencies for first-party apps but does not add root workspace registration, `pnpm-lock.yaml`, Cargo lockfiles, or CI/release wiring. That leaves dependency resolution and package provenance outside the repo's normal review and reproducibility paths.
  - confidence: 0.82
