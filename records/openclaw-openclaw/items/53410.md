## Summary

Keep open. Current main still sends update checks through the generic SemVer comparator, which treats numeric `-N` suffixes as prerelease identifiers, so the PR addresses a real remaining correction-release mismatch. The branch needs maintainer adaptation because current main moved the parser into shared ClawHub-facing code, and the numeric-suffix comparison should be made precision-safe before merge.

## What This Changes

This PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Keep the PR open and adapt it to current main by applying correction-release ordering only to OpenClaw update-version comparisons, or by adding an explicit mode to the shared comparator without changing generic ClawHub/plugin SemVer semantics. Regression coverage should include correction-vs-stable, correction-vs-beta, revision ordering, affected update/status callers, and precision-safe numeric suffix comparison, with a changelog credit if maintainers accept the behavior.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.78

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. For registry-provided versions above `Number.MAX_SAFE_INTEGER`, adjacent revisions such as `2026.3.23-9007199254740992` and `2026.3.23-9007199254740993` can collapse to the same JS number and produce the wrong update ordering. Keep the suffix as a normalized string, use BigInt, or bound the accepted length before comparing.
  - confidence: 0.78
