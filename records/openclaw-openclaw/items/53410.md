## Summary

Keep open. The requested update-version behavior is still not implemented on current main, and the PR is useful source work, but it is stale against the current shared comparator layout and should address the numeric precision concern before merge.

## What This Changes

This PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and prereleases.

## Best Possible Solution

Keep the PR open and adapt it to current main by applying correction-release ordering to OpenClaw update-version comparisons, or by adding an explicit mode to the shared comparator, without changing generic ClawHub/plugin SemVer semantics. Use precision-safe numeric suffix comparison and cover correction-vs-stable, correction-vs-beta, revision ordering, and affected update/status callers.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. Registry-provided versions above `Number.MAX_SAFE_INTEGER` can collapse adjacent revisions to the same JS number and produce the wrong update ordering; keep the suffix as a normalized string, use `BigInt`, or bound the accepted length before comparing.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch introduces a low-severity update-ordering concern by parsing unbounded numeric revision suffixes into JavaScript numbers.

Concerns:

- **[low] Avoid lossy numeric revision parsing:** `src/infra/update-check.ts:405`
  - body: Registry-provided version strings can contain very large numeric suffixes; coercing those suffixes with `Number.parseInt()` can lose precision and make update gating choose the wrong ordering.
  - confidence: 0.78
