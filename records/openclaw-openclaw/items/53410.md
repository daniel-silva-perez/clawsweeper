## Summary

Keep open. Current main and the latest release still route update checks through the generic SemVer comparator, so the false update-available behavior for OpenClaw correction releases remains. The PR is useful source work, but it needs maintainer adaptation because current main has extracted the comparator into a shared helper used by ClawHub/plugin compatibility checks, and the proposed numeric revision path has a precision concern.

## What This Changes

The PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Keep the PR open and adapt it on current main so OpenClaw package update/status/startup/update-runner/update-command comparisons use correction-release ordering, while generic ClawHub/plugin SemVer comparisons keep standard prerelease semantics unless maintainers explicitly add a mode. The adapted implementation should compare numeric suffixes precision-safely and include focused regression coverage for correction-vs-stable, correction-vs-beta, revision ordering, and affected update callers.

## Reproduction Assessment

Yes. Current source shows `compareSemverStrings()` delegates to the generic SemVer helper, and the current regression tests still encode numeric `-N` as prerelease ordering; from that path, `2026.3.23-1` compares older than `2026.3.23`, which explains the false update-available banner.

## Solution Assessment

Unclear as-is. The requested behavior is the right direction for OpenClaw package versions, but the PR targets the older update-check-local parser shape; on current main the safer solution is an OpenClaw-specific wrapper or explicit comparator mode plus precision-safe numeric suffix comparison.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.78

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. Registry-provided versions above `Number.MAX_SAFE_INTEGER` can collapse adjacent revisions to the same JS number and produce the wrong update ordering; keep the suffix as a normalized string, use `BigInt`, or bound the accepted length before comparing.
  - confidence: 0.78
