## Summary

Keep open. Current main still has the reported update-version ordering gap, but this PR needs maintainer adaptation before merge because the comparator now lives in a shared SemVer helper used by ClawHub/plugin range checks, and the proposed revision path compares unbounded numeric suffixes through lossy JavaScript numbers.

## What This Changes

The PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Adapt the change on current main so OpenClaw update/status/startup/update-runner/update-command paths use correction-release ordering for OpenClaw package versions, while generic ClawHub/plugin SemVer comparisons keep standard prerelease semantics unless maintainers intentionally add an explicit mode. The adapted implementation should compare numeric suffixes precision-safely and include focused regression coverage for correction-vs-stable, correction-vs-beta, revision ordering, and affected update callers.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. Registry-provided versions above `Number.MAX_SAFE_INTEGER` can collapse adjacent revisions to the same JS number and produce the wrong update ordering; keep the suffix as a normalized string, use `BigInt`, or bound the accepted length before comparing.
  - confidence: 0.78
