## Summary

Keep open. Current main still uses the generic SemVer comparator for update checks, so the reported false update-available path remains plausible, and the PR is useful prior work. It needs maintainer adaptation before merge because current main moved parsing into `src/infra/semver-compare.ts`, that comparator is shared with ClawHub/plugin range checks, and the proposed numeric revision parsing has a concrete precision concern raised in review discussion.

## What This Changes

This PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Adapt the PR to current main by making update-version comparison use OpenClaw correction-release semantics without unintentionally changing generic ClawHub/plugin SemVer range behavior. The implementation should compare numeric suffixes without precision loss and cover update/status callers plus revision, stable, beta, and same-base correction cases.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.78

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. For registry-provided versions above `Number.MAX_SAFE_INTEGER`, adjacent revisions can collapse to the same JS number and produce the wrong update ordering. Keep the suffix as a normalized string, use `BigInt`, or bound the accepted length before comparing.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a low-severity update-ordering concern by parsing unbounded numeric revision suffixes into JavaScript numbers.

Concerns:

- **[low] Avoid lossy numeric revision parsing:** `src/infra/update-check.ts:405`
  - body: Registry-provided version strings can contain very large numeric suffixes; coercing those suffixes with `Number.parseInt()` can lose precision and make update gating choose the wrong ordering.
  - confidence: 0.78
