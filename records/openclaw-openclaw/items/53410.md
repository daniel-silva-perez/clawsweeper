## Summary

Keep open. Current main still uses the generic SemVer comparator for update availability, so numeric-only `-N` suffixes are still treated as prerelease identifiers in the update/status path rather than OpenClaw correction revisions. The PR remains useful, but it needs maintainer adaptation to the current comparator split and should address the low-severity precision concern before merge.

## What This Changes

This PR changes update-version comparison and regression tests so numeric-only hyphen suffixes rank as OpenClaw correction revisions newer than the same base stable release and prereleases.

## Best Possible Solution

Adapt the fix on current main so OpenClaw update/status/startup/update-runner comparisons use correction-release ordering for OpenClaw package versions without accidentally changing generic ClawHub/plugin SemVer range semantics. Compare numeric suffixes with a precision-safe representation, and cover correction-vs-stable, correction-vs-beta, revision ordering, and affected update/status callers.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. Registry-provided versions above `Number.MAX_SAFE_INTEGER` can collapse adjacent revisions to the same JS number and produce the wrong update ordering; keep the suffix as a normalized string, use `BigInt`, or bound the accepted length before comparing.
