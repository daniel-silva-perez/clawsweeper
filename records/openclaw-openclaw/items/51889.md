## Summary

Keep open. Current main does not already provide `/restore`, and the PR is useful, but the branch still introduces blocking correctness and transcript-confidentiality problems around restore destination paths, keyless archive ownership, filtered archive scanning, and duplicate reset hook execution.

## What This Changes

The PR adds a built-in `/restore [number]` chat command, archived transcript listing/restoration helpers, sessionKey transcript headers, docs, and focused session restore tests.

## Best Possible Solution

Revise the branch so restore moves the selected archive into the exact `resetResult.entry.sessionFile`, validates that the archive belongs to the current sessionKey before listing or restoring, handles legacy keyless archives through an explicit private recovery policy, scans until it has enough matching archives under a bounded strategy, and lets one reset path own hook emission. Add tests for persisted sessionFile destinations, thread/topic sessions, keyless/corrupt/long headers, multi-channel isolation, and filtered scan ceilings.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create a session entry whose reset preserves `entry.sessionFile`, or create keyless/corrupt/long-header archives from another sessionKey; the PR target derivation and `null` header filter then restore to the wrong file or list another session's archive.

## Solution Assessment

No. A `/restore` command is a reasonable feature direction, but this implementation is not the narrowest maintainable fix until it reuses the reset service's persisted destination and enforces an explicit archive ownership contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Restore into the persisted reset session file:** `src/gateway/session-restore.ts:231-235`
  - body: `performGatewaySessionReset` returns the active `entry.sessionFile`, and current reset code can preserve the prior transcript path. This derives a fresh `{newSessionId}.jsonl` path instead, so restore can move the archive to a file the session store does not read and then report success. Use and validate `resetResult.entry.sessionFile` as the destination.
  - confidence: 0.94
- **[P1] Require owned archive headers before listing or restoring:** `src/gateway/session-restore.ts:188`
  - body: `readSessionKeyFromHeader` returns `null` for missing keys, parse failures, truncation, and read errors, and this condition accepts `null` for any requested sessionKey. That can expose or restore another channel's archived transcript. Distinguish safe legacy archives from unreadable headers, or skip keyless archives unless ownership is explicit.
  - confidence: 0.92
- **[P2] Scan until enough matching archives are found:** `src/gateway/session-restore.ts:178-180`
  - body: The scan ceiling is applied before the sessionKey filter, so newer archives from other channels can consume the whole window and make valid current-channel history disappear. Bound total work if needed, but apply `limit` after filtering or continue scanning until enough matches are found.
  - confidence: 0.86
- **[P2] Avoid emitting reset hooks twice during restore:** `src/auto-reply/reply/commands-restore.ts:90-98`
  - body: This handler emits reset command hooks and then calls `performSessionRestore`, which immediately calls `performGatewaySessionReset` and triggers the reset hook path again. Hook consumers can see duplicate side effects or inconsistent pre-reset snapshots; let one reset path own hook emission for restore.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR can cross the sessionKey confidentiality boundary for archived chat transcripts.

Concerns:

- **[high] Cross-channel transcript restore via keyless headers:** `src/gateway/session-restore.ts:188`
  - body: Archive header parsing collapses missing sessionKey, parse failures, truncation, and read errors to `null`, and `listRecentArchives` treats `null` as visible to any sessionKey. In a multi-channel instance, an authorized user in one channel can be offered another channel's archived transcript if that archive is legacy, corrupt, or has a long header.
