## Summary

Keep this PR open because current main still does not implement `/restore [number]`, archive browsing, or reset/deleted transcript restoration. The branch is still a relevant implementation candidate, but the current diff has blocking correctness and transcript-isolation issues that should be resolved before merge.

## What This Changes

The PR adds a built-in `/restore [number]` chat command, session archive listing/restoration helpers, sessionKey transcript headers, docs, and focused gateway tests.

## Best Possible Solution

Rework the PR into a first-party restore flow that restores into the active reset entry's resolved `sessionFile`, enforces sessionKey isolation with a clear legacy-archive policy, preserves normal reset and hook semantics without duplicate side effects, and adds tests for multi-channel isolation, corrupt/long headers, persisted sessionFile paths, thread/topic sessions, and archive selection drift.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Restore into the reset entry's transcript path:** `src/gateway/session-restore.ts:246-252`
  - body: `performGatewaySessionReset` returns the session entry that runtime will read next, including `entry.sessionFile`. This patch ignores that and derives a new path from `newSessionId`, so sessions with an existing persisted `sessionFile` can report a successful restore while the active store entry still points at the reset-created transcript. Move the archive to `resetResult.entry.sessionFile` after validating it is inside the sessions directory.
  - confidence: 0.9
- **[P1] Require a matching archive sessionKey:** `src/gateway/session-restore.ts:199-203`
  - body: The current filter treats `null` from `readSessionKeyFromHeader` as a match for every channel, and that same `null` is returned for missing keys, unreadable files, and JSON parse failures. An authorized user in one channel can therefore be offered or restore another channel's keyless/corrupt archive. Distinguish legacy-no-key from read errors and avoid cross-channel matches without an explicit safe migration path.
  - confidence: 0.88
- **[P2] Scan until enough matching archives are found:** `src/gateway/session-restore.ts:191-193`
  - body: `scanCeiling` is applied before the per-session `sessionKey` filter. In a multi-channel deployment, many newer archives from other channels can consume the first 50 candidates and make `/restore` report no history even when older matching archives exist. Apply `limit` after filtering, or use a bounded but correctness-preserving scan policy.
  - confidence: 0.82
- **[P2] Avoid firing reset hooks twice:** `src/auto-reply/reply/commands-restore.ts:103-113`
  - body: `/restore` calls `emitResetCommandHooks` and then immediately calls `performGatewaySessionReset`, which also triggers the internal command hook and the before-reset plugin hook for the same reset. Hook users can see duplicate side effects, and the first call often receives no `previousSessionEntry` for the active transcript. Let one reset path own hook emission with the correct active-session snapshot.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete transcript isolation risk by allowing keyless or unreadable archive headers to match any sessionKey during archive listing and restore.

Concerns:

- **[high] Cross-channel transcript restore via keyless archives:** `src/gateway/session-restore.ts:201`
  - body: `listRecentArchives` includes archives when `readSessionKeyFromHeader` returns `null`, but `null` also covers missing `sessionKey`, truncated headers, parse failures, and read failures. That can expose or restore another channel's transcript into the current channel, which is a confidentiality boundary for chat history.
  - confidence: 0.88
