## Summary

Keep this PR open. Current main at 891c7d9f1c1f5e0f52935fadfeacf695b0ef1eb8 still has reset/new transcript archiving and a checkpoint-only compaction restore path, but it does not implement the proposed `/restore [number]` command, archive browser, or reset/deleted archive restore workflow. The provided discussion shows active author follow-up and prior review work, so this remains a plausible implementation candidate rather than obsolete cleanup.

## Best Possible Solution

Keep this PR open for maintainer review as an implementation candidate, or replace it with an equivalent first-party `/restore [number]` flow. The shipped path should list recent reset/deleted archives for the active session boundary, restore into the active entry's resolved transcript path, preserve normal reset semantics and hooks, and include tests for multi-channel isolation, topic/thread sessions, legacy archives, corrupt headers, and restore selection races. The related open `/resume` request can be cross-linked as product context, but it does not supersede this PR while no implementation has landed.
