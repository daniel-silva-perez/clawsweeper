## Summary

Keep open. Current main still lacks a user-facing `/restore` command, but the latest PR head still has blocking restore-path and transcript-isolation issues, so this should remain in author/maintainer review rather than close or land as-is.

## What This Changes

The PR adds a built-in `/restore [number]` chat command, archive listing/restoration helpers, sessionKey transcript headers, docs, and focused gateway tests.

## Best Possible Solution

Rework the branch into a first-party restore flow that restores into the active reset entry's validated `sessionFile`, enforces a clear and private legacy-archive policy, lets one reset path own hook emission, and adds coverage for multi-channel isolation, corrupt or long headers, persisted sessionFile paths, thread/topic sessions, and archive-list selection drift.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore into the reset entry transcript path:** `src/gateway/session-restore.ts:248-252`
  - body: `performGatewaySessionReset` has already persisted the active entry and may preserve an existing `entry.sessionFile`; deriving `targetPath` from `newSessionId` can move the archive into a file the session store does not read, so `/restore` can report success while the next turn sees a fresh transcript. Use and validate `resetResult.entry.sessionFile` instead.
  - confidence: 0.92
- **[P1] Require a matching archive sessionKey:** `src/gateway/session-restore.ts:201`
  - body: `readSessionKeyFromHeader` returns `null` for keyless, unreadable, or invalid headers, and this condition accepts `null` for every channel. That can list or restore another channel's archive in the current session. Distinguish legacy no-key from read errors and avoid cross-channel matches without an explicit safe migration policy.
  - confidence: 0.9
- **[P2] Scan until enough matching archives are found:** `src/gateway/session-restore.ts:193`
  - body: The scan ceiling is applied before sessionKey filtering, so 50 newer archives from other channels can make current-channel history disappear. Enforce `limit` after filtering, or use a bounded scan policy that still finds enough matching archives.
  - confidence: 0.84
- **[P2] Avoid firing reset hooks twice during restore:** `src/auto-reply/reply/commands-restore.ts:104-113`
  - body: This calls `emitResetCommandHooks` and then `performSessionRestore` calls `performGatewaySessionReset`, which also runs command and before-reset hooks. Hook consumers can see duplicate or inconsistent reset side effects; let one reset path own hook emission with the active session snapshot.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The PR needs security attention because archive filtering can cross the sessionKey confidentiality boundary for chat transcripts.

Concerns:

- **[high] Cross-channel transcript restore via keyless archives:** `src/gateway/session-restore.ts:201`
  - body: `readSessionKeyFromHeader` collapses missing keys, parse failures, truncated headers, and read errors to `null`, and `listRecentArchives` accepts `null` for any requested sessionKey. An authorized user in one channel can therefore be offered or restore another channel's archive when that archive is keyless or unreadable.
  - confidence: 0.9
