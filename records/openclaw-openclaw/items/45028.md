## Summary

Keep open. Current main includes a partial route-preservation fix, but the ordinary Control UI composer still sends non-delivery WebChat turns and the dispatcher intentionally avoids routing those replies to the stored Telegram route; the related open PR #46356 targets this broader operator/WebChat mirroring behavior and needs maintainer review or splitting before this issue can close.

## What This Changes

The issue requests that Control UI replies sent while viewing a Telegram-backed session be delivered back to Telegram instead of remaining on WebChat.

## Best Possible Solution

Define a narrow operator-reply path for externally backed sessions that intentionally uses the stored channel route when the Control UI is acting as an operator surface, while preserving WebChat-native sessions, stale-route safeguards, and explicit-delivery semantics; split or review #46356 if it is the chosen vehicle.

## Reproduction Assessment

Yes. The high-confidence static reproduction is: Control UI calls `chat.send` with `deliver: false`, gateway stamps the run as a WebChat non-delivery turn, and reply dispatch refuses to route that turn to the stored external route; current tests assert the same policy.

## Solution Assessment

Unclear for an automatic route-preservation fix. The requested outcome is valid, but the safest maintainable solution is an explicit operator-reply or mirroring policy rather than a Telegram-specific adapter workaround or blanket removal of the WebChat inheritance guard.
