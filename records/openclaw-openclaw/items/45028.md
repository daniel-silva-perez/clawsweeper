## Summary

Keep open. Current main still treats ordinary Control UI/WebChat composer turns as internal WebChat traffic and suppresses delivery to an established Telegram-style external route; the related #46356 implementation candidate is still open and has not shipped.

## What This Changes

The issue asks for Control UI replies sent while viewing a Telegram-backed session to route back to Telegram instead of staying on the internal WebChat surface.

## Best Possible Solution

Define an explicit operator-reply or WebChat mirroring policy for externally backed sessions that uses the stored external route only when the Control UI is acting as an operator surface, with regression coverage for Telegram-style sessions, WebChat-native sessions, explicit delivery, and stale-route leakage.

## Reproduction Assessment

Yes. The high-confidence static path is Control UI `chat.send` with `deliver: false`, gateway origin resolution to internal WebChat, and dispatch suppression of route-reply delivery for non-explicit WebChat turns.

## Solution Assessment

Unclear for the one-line fallback suggested in discussion, because current main intentionally blocks WebChat route inheritance for route-safety reasons. A safer solution is an explicit operator-reply or mirroring path, likely by reviewing or splitting #46356.
