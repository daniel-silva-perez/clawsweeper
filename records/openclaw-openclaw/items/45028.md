## Summary

Keep open. Current main preserves external route metadata in the session store, but ordinary browser Control UI composer sends still stay on WebChat: the UI sends `chat.send` with `deliver: false`, the Gateway resolves non-explicit sends as internal WebChat turns, and the dispatcher suppresses external route delivery unless explicit delivery is set. Open PR #46356 targets the broader mirroring behavior, so this needs maintainer policy and PR review rather than cleanup closure.

## What This Changes

The issue requests that replies typed in the Control UI while viewing a Telegram-backed session route back to the Telegram chat instead of remaining on WebChat.

## Best Possible Solution

Add a narrow, explicit Control UI operator-reply path or routing policy for externally backed sessions that can deliver to the stored channel route without weakening existing WebChat route-leakage safeguards. The end state should cover browser Control UI behavior, Gateway route resolution, dispatcher explicit-delivery handling, and regression tests for Telegram-style channel-scoped sessions plus WebChat-native sessions.
