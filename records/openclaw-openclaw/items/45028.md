## Summary

Keep open. Current main still sends ordinary Control UI composer turns as non-delivering WebChat traffic and suppresses external route-reply delivery unless explicit delivery is set; the existing route-preservation code prevents stored Telegram-style routes from being overwritten, but it does not make Control UI replies go back to Telegram. The related #46356 implementation candidate is open and unshipped.

## What This Changes

Control UI replies sent while viewing a Telegram-backed direct session should route to the Telegram delivery target instead of staying on WebChat.

## Best Possible Solution

Define an explicit operator-reply or WebChat mirroring policy for externally backed sessions that preserves WebChat-native safety while allowing Control UI replies to intentionally use the stored external delivery route, with regression coverage for Telegram-style sessions, WebChat-native sessions, explicit delivery, and stale-route leakage.

## Reproduction Assessment

Yes. The high-confidence static path is Control UI `chat.send` with `deliver: false`, Gateway origin resolution to internal WebChat, and dispatch suppression of route-reply delivery for non-explicit WebChat turns; current tests also assert this non-delivery policy.

## Solution Assessment

Unclear. The one-line fallback suggested in discussion would conflict with existing route-safety tests, so the safer solution is an explicit operator-reply or mirroring path, likely by reviewing or splitting #46356.
