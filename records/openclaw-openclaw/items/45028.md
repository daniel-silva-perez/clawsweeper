## Summary

Keep open. Current main preserves external session route metadata when WebChat/Control UI touches an externally backed session, but the ordinary Control UI composer still sends a non-delivery WebChat turn and the dispatcher intentionally suppresses routing those replies to the stored Telegram route. The related open PR #46356 targets this broader WebChat/operator-reply behavior, so this is not cleanup-closeable yet.

## What This Changes

The issue asks for Control UI replies sent while viewing a Telegram-backed session to route back to the underlying Telegram conversation instead of staying on WebChat.

## Best Possible Solution

Define a narrow operator-reply or WebChat-mirroring path for externally backed sessions that uses the stored channel route only when the UI is intentionally acting as an operator surface, while preserving WebChat-native behavior and the existing stale-route safeguards.
