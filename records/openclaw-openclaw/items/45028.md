## Summary

Keep open. Current main preserves stored external session metadata in some paths, but the ordinary Control UI composer still starts a non-delivery WebChat turn and the dispatcher intentionally does not route those replies to the stored Telegram route; the related open PR #46356 is the active broader implementation candidate and needs maintainer review or splitting before this can close.

## What This Changes

The issue requests that Control UI replies sent while viewing a Telegram-backed direct session route back through the session’s Telegram delivery target instead of remaining on the internal WebChat surface.

## Best Possible Solution

Define a narrow operator-reply or mirroring policy for externally backed sessions that deliberately uses the stored delivery route when the Control UI is acting as an operator surface, while preserving WebChat-native sessions, explicit-delivery semantics, stale-route safeguards, and route-leakage tests.

## Reproduction Assessment

Yes. The high-confidence static reproduction path is: Control UI calls `chat.send` with `deliver: false`, the gateway stamps the turn as internal WebChat without an explicit delivery route, and reply dispatch suppresses routing to the stored external channel; current tests assert that policy.

## Solution Assessment

Unclear for the one-line fallback suggested in discussion, because current main intentionally blocks WebChat route inheritance to prevent cross-channel leakage. A safer solution is an explicit operator-reply or mirroring path, likely by reviewing or splitting #46356 rather than changing Telegram-specific code.
