## Summary

Keep open. Current main still treats ordinary browser Control UI composer turns as internal WebChat non-delivery, and reply dispatch suppresses routing those replies to a stored Telegram route; the shipped route-preservation work only prevents metadata overwrite, while #46356 remains the active broader implementation candidate.

## What This Changes

Route replies sent from the Control UI while viewing a Telegram-backed session back to Telegram, or provide an explicit operator choice, instead of keeping them on WebChat.

## Best Possible Solution

Define a narrow operator-reply or WebChat mirroring policy for externally backed sessions that deliberately uses the stored delivery route only when the Control UI is acting as an operator surface, with regression coverage for Telegram-style sessions, WebChat-native sessions, explicit delivery, and stale-route leakage.

## Reproduction Assessment

Yes. The high-confidence static reproduction path is Control UI `chat.send` with `deliver: false`, followed by gateway origin resolution to internal WebChat and dispatch suppression of external route delivery; current tests assert those branches.

## Solution Assessment

Unclear for the one-line fallback suggested in discussion, because current main intentionally blocks WebChat route inheritance for route-safety reasons. A safer fix is an explicit operator-reply or mirroring path, likely by reviewing or splitting #46356.
