## Summary

Keep open: current main still treats the browser Control UI as a WebChat client, sends Control UI composer turns with `deliver: false`, and has Gateway/dispatcher guards that keep those turns on WebChat unless explicit external delivery is requested. The session-delivery fixes preserve external route metadata, but they do not make normal browser Control UI replies deliver back to Telegram.

## What This Changes

The issue requests that replies typed in the Control UI while viewing a Telegram-backed direct session be delivered through Telegram instead of staying on WebChat.

## Best Possible Solution

Add a narrow, explicit Control UI operator-reply affordance or routing policy for externally backed sessions that can deliver to the stored channel route without weakening existing WebChat route-leakage safeguards. The change should cover browser Control UI behavior, Gateway route resolution, dispatcher explicit-delivery handling, and regression tests for Telegram-style channel-scoped sessions plus WebChat-native sessions.
