## Summary

Keep this PR open, but do not merge it as-is. Current main still has the scheduled-run bundled-tool leak, and this PR targets the active path, but its exact-name final filter would likely drop bundled MCP tools when `toolsAllow` uses the supported `bundle-mcp` plugin key.

## What This Changes

The PR re-applies runtime `toolsAllow` filtering to the combined embedded-runner tool list after bundled MCP/LSP tools are materialized.

## Best Possible Solution

Enforce runtime `toolsAllow` after bundled tool materialization with plugin-aware semantics, preserving exact tool-name allowlists and plugin/group keys such as `bundle-mcp`, then add focused regression coverage for cron-style exec-only filtering and bundled MCP/LSP cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: schedule an `agentTurn` with `toolsAllow: ["exec"]` while bundled MCP/LSP tools are configured, and current main still appends those bundled tools after the early base-tool filter.

## Solution Assessment

No. The repair direction is right, but applying the exact-name helper to the final combined list is not the best fix because the existing bundled-tool policy uses plugin metadata and supports plugin-key allowlist semantics.
