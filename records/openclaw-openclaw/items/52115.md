## Summary

Keep this open. Current main still leaves the default builtin long-running gateway path able to query stale rows on the first search after dirty memory files, because dirty search sync is started in the background rather than awaited. The newer main/release code adds an empty-index bootstrap, but that does not cover the reported create/delete-after-start case.

## What This Changes

The issue requests that agent-facing `memory_search` refresh its long-lived gateway memory index before returning results so newly created files appear and deleted or moved paths stop surfacing.

## Best Possible Solution

Make builtin `MemoryIndexManager.search()` freshness-aware when `dirty` or `sessionsDirty` is set by awaiting the relevant pending or newly-started sync before querying, while keeping clean searches fast and adding create/update/delete first-search regression coverage.
