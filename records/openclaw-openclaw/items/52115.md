## Summary

Current main still appears to have the reported default builtin gateway `memory_search` freshness gap. The gateway/tool path uses a cached default `MemoryIndexManager`; builtin search still fires dirty `onSearch` sync in the background before querying existing rows, so the first search after a create/delete/rename can still observe stale or ghost results. QMD has an awaited dirty-search path, but QMD is opt-in and does not cover default builtin users.

## Best Possible Solution

Keep this issue open and fix the builtin/default `MemoryIndexManager` path used by gateway `memory_search`. The likely maintainer change is to make agent-facing builtin search await pending or newly triggered sync when `dirty` or `sessionsDirty` is set, then add regression coverage for a cached default manager where the first search after memory file create, delete, and rename reflects disk state and returns no nonexistent paths.

## Risks / Open Questions

- A broad fix that awaits every full sync could add latency to ordinary `memory_search`; the safer product shape is to await only when builtin memory is dirty, sessions are dirty, or a freshness sync is already pending.
- QMD has fresher dirty-search semantics, but it is opt-in and cannot be treated as resolving the default builtin gateway behavior.
