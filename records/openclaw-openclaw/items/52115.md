## Summary

Keep this issue open. Current main still has the reported freshness gap: the default agent-facing `memory_search` path reuses the long-lived builtin manager, and dirty search sync is still started asynchronously before the existing index rows are queried. The CLI comparison remains valid because CLI search uses a transient manager purpose.

## What This Changes

The issue requests that agent/tool-chain `memory_search` refresh the long-running gateway memory index so newly created, moved, or deleted memory files are reflected without a gateway restart and stale paths are not returned.

## Best Possible Solution

Make the builtin `MemoryIndexManager` freshness-aware for agent-facing searches: when memory or enabled session state is dirty, or a relevant sync is already pending, await the sync before reading rows; keep fast searches unchanged when the index is clean, and add regression coverage for first-search-after-create/update/delete behavior.

## Repair Work Prompt

Fix the current builtin memory-core freshness bug reported in https://github.com/openclaw/openclaw/issues/52115. Agent-facing `memory_search` uses a cached default `MemoryIndexManager`; when memory/session state is dirty or a freshness sync is pending, the first search should not read stale index rows before the sync completes. Update the builtin manager/search helper so dirty searches await the relevant sync before querying, while preserving the clean-index fast path and avoiding QMD behavior changes unless a shared typed seam is clearly needed. Add regression tests that cover a long-lived manager seeing a memory file create/update/delete or move on the first search after the dirty event, and adjust the existing async-search unit test that currently asserts search sync is not awaited. Keep the fix inside `extensions/memory-core` unless a generic config/type seam is required. Add a single-line changelog fix entry with `Thanks @FicheallADa` if the repo's active changelog requires user-facing bug fixes.
