## Summary

Keep this issue open. Current main still has the reported default gateway freshness gap: the agent-facing `memory_search` path uses a cached builtin manager, and builtin `search()` still triggers dirty `onSearch` sync asynchronously before querying the existing index rows. The CLI comparison remains meaningful because CLI memory search asks for a transient `purpose: "cli"` manager.

## What This Changes

The issue asks for gateway/agent `memory_search` to refresh its long-lived builtin memory index before returning results, so newly created files appear and deleted or renamed files stop surfacing without a gateway restart.

## Best Possible Solution

Make the builtin `MemoryIndexManager` freshness-aware for agent-facing searches: when memory or session state is dirty, or a relevant freshness sync is already pending, await that sync before reading rows; preserve the fast path when the index is clean and keep QMD behavior separate unless a shared interface is needed.

## Repair Work Prompt

Current behavior: the agent/tool path uses a cached default `MemoryIndexManager`, and builtin `search()` starts dirty `onSearch` sync fire-and-forget before querying rows, so the first search after memory file create/delete/rename can miss new files or return ghost paths.
