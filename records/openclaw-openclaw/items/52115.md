## Summary

Current main still appears to leave the reported builtin/default gateway freshness gap open. The default memory backend is still builtin, gateway/tool memory search uses the cached default manager, and builtin `search()` still schedules dirty `onSearch` sync fire-and-forget before querying existing rows. That means the first search after create/delete/rename can still observe stale rows or ghost paths, while the CLI path uses a transient manager.

## Best Possible Solution

Keep this issue open and fix the builtin/default `MemoryIndexManager` path used by gateway `memory_search`. The likely maintainer change is to make agent-facing builtin search await pending or newly triggered sync when memory is dirty or sessionsDirty, then add regression coverage for a cached default manager where the first search after memory file create, delete, and rename reflects disk state and returns no nonexistent paths.

## Risks / Open Questions

- A fix that awaits every full sync could add latency to ordinary memory_search calls; the narrower fix should await only when builtin memory is dirty, sessionsDirty, or an already-pending freshness sync exists, or expose an explicit freshness mode.
- QMD has a separate awaited dirty-search path, but QMD is opt-in and does not cover default builtin users.
