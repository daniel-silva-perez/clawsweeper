## Summary

Keep open. Current main still leaves the default builtin gateway memory manager able to query stale indexed rows on the first search after dirty memory files, and the only current-main delta since v2026.4.27 in this area is search warning logging. PR #72748 is open and appears to target the same dirty-search sync path, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that agent-facing `memory_search` refresh the long-lived gateway memory index before returning results so new files appear and deleted or moved paths are not returned.

## Best Possible Solution

Land PR #72748 or an equivalent focused memory-core fix that awaits dirty builtin search sync before querying, adds first-search create/update/delete and ghost-path regression coverage, preserves clean-search speed, and leaves QMD's already-awaited dirty search path unchanged.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the default builtin manager after indexed content exists: mark memory dirty via create/change/unlink, call agent-facing `memory_search`, and observe that current `search()` starts sync in the background before querying the old rows.

## Solution Assessment

Yes. Awaiting dirty builtin sync before querying is the narrowest maintainable fix because the dirty flag is already set by watchers and stale-path pruning already happens inside sync; QMD already follows the awaited-sync model.
