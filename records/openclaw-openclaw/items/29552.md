## Summary

Keep this issue open. Current main has shipped and main-only work around updater split-brain detection, future-config guards, and clearer gateway transport errors, but the reported pairing failure still maps to a live ordinary local Gateway auth path: local calls still resolve credentials once, env credentials still win over gateway.auth config for ordinary local call/probe paths, and AUTH_TOKEN_MISMATCH retry remains limited to trusted cached device-token recovery rather than retrying once with config-first gateway.auth credentials. The closed fallback PR #29676 was not merged, and open PR #72535 only targets warning wording for newer-config/downgrade cases.

## Best Possible Solution

Keep this open for maintainer auth-path judgment. The best fix is either to explicitly keep env-first local call precedence and document stale-token recovery more directly, or add a narrowly guarded local-only retry after AUTH_TOKEN_MISMATCH that retries once with config-first gateway.auth token/password only when there is no remote mode, URL override, explicit token/password, or unresolved SecretRef. Keep #72535 as warning-message cleanup and treat any auth recovery as a separate Gateway-auth change with focused tests.
