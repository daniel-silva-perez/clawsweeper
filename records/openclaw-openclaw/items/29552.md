## Summary

Keep this issue open. Current main still has the plausible unresolved local gateway pairing failure described in the report: local gateway calls can prefer a stale OPENCLAW_GATEWAY_TOKEN over gateway.auth.token, callGatewayWithScopes sends that selected credential set once without a config-auth retry, and the existing AUTH_TOKEN_MISMATCH retry path is scoped to cached device-token recovery. The related fallback PR #29676 was closed unmerged, while open PR #72535 only improves the newer-config warning text and does not address the pairing failure.

## Best Possible Solution

Keep this open and split the work into a clear maintainer decision: either intentionally keep env-first local call precedence and document the manual token-drift recovery path, or add a narrowly guarded local-only retry after AUTH_TOKEN_MISMATCH that retries once with config-first gateway.auth token/password when there is no remote mode, URL override, explicit token/password, or unresolved SecretRef. Track warning-message cleanup separately through #72535 and already-shipped update/version-drift fixes separately through the closed update-drift items.
