## Summary

Keep this issue open. Current main has several adjacent fixes for update/install drift and setup health checks, but the reported pairing failure can still occur on the ordinary local Gateway call path: local calls still resolve env credentials ahead of gateway.auth config, callGatewayWithScopes sends one credential set, and AUTH_TOKEN_MISMATCH retry remains scoped to trusted cached device-token recovery rather than a config-auth retry. The closed fallback PR #29676 was not merged, and open PR #72535 only targets newer-config warning wording.

## Best Possible Solution

Keep this open for maintainer auth-path judgment. Either intentionally keep env-first local call precedence and document stale-token recovery more directly, or add a narrowly guarded local-only retry after AUTH_TOKEN_MISMATCH that retries once with config-first gateway.auth token/password when there is no remote mode, URL override, explicit token/password, or unresolved SecretRef. Keep warning-message cleanup in #72535 separate from any auth recovery change.
