## Summary

Keep this issue open. Current main has session/subagent relationship metadata and flat inspection surfaces, but it still lacks a user-facing `openclaw tree`, `openclaw sessions tree`, or `openclaw agents tree` command with hierarchical text/JSON/watch/session/resource modes.

## Best Possible Solution

Keep this open for a core inspection feature. A conservative implementation would add a sessions/agents tree command that reuses existing session and subagent metadata (`spawnedBy`, `parentSessionKey`, `childSessions`, status, runtime, token/cost fields), supports text plus `--json` first, and defers `--watch` and CPU/RSS resource accounting until the runtime data contract is explicit.

## Risks / Open Questions

- The requested `--resources` mode likely needs a new explicit data contract for per-agent CPU/RSS ownership; current surfaces expose runtime, tokens, and estimated cost, not reliable process resource attribution.
- Command naming still needs product judgment: `openclaw tree`, `openclaw sessions tree`, and `openclaw agents tree` are all plausible and should avoid overlapping confusingly with existing flat `sessions`, `tasks`, and `/subagents list` surfaces.
