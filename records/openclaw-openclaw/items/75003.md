## Summary

Keep open: this is a maintainer-labeled MEMBER PR, so the cleanup policy requires explicit maintainer handling. The patch is narrow and still relevant because current main and the latest release both create default TTY spinners without checking raw stdin, while the Crestodian status path reaches the shared CLI progress helper.

## What This Changes

The PR adds a raw-stdin spinner eligibility helper in CLI progress, suppresses the default readline-backed spinner while stdin is raw, and adds focused progress tests for raw TUI input and normal TTY behavior.

## Best Possible Solution

Keep this PR open and, if maintainers agree with the raw-stdin boundary, land the small progress guard with its focused tests so ordinary terminal commands retain spinner behavior while active TUI input is not taken over by clack/readline.

## Reproduction Assessment

Yes. The source path is high-confidence: Crestodian status reaches statusCommand, statusCommand uses withProgress, and current main's default TTY spinner path has no raw-stdin check before creating the clack spinner.

## Solution Assessment

Yes. The proposed approach is the narrowest maintainable fix I found: guard only default/readline-backed spinner selection when process.stdin is already raw, while preserving explicit line/log/none modes and normal TTY spinner behavior.
