## Summary

Keep this open. Current main still has a strict threshold-only deep ranking path: candidates below minScore are dropped before ranking output and apply rechecks the same score gate, with no wildcardRatio or random sub-threshold sampling surface. The request is actionable, but the default/random durable-memory behavior needs maintainer product judgment before implementation.

## What This Changes

Request adding a wildcard sampling lane to memory-core dreaming deep ranking so sub-threshold but hard-gated short-term candidates can occasionally surface for promotion.

## Best Possible Solution

Decide the memory-policy semantics first, then implement a narrow, testable wildcard lane only after hard gates and before the composite-score cutoff, with deterministic injection for tests, clear promote-explain/status visibility, and aligned config/docs if the setting is user-facing.

## Reproduction Assessment

Not applicable as a feature request rather than a bug. Current-main inspection gives a high-confidence behavior check: sub-threshold candidates are filtered in ranking and filtered again before apply.

## Solution Assessment

Unclear whether the proposed default random 0.15 sampling is the best policy. A safer maintainable solution would make wildcard selection deterministic/testable and explicitly approved for durable memory writes before exposing or defaulting it.
