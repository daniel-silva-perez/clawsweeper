## Summary

Current main does not implement the requested wildcard candidate behavior. Deep promotion still ranks only candidates that pass signal, diversity, age, and `minScore` gates, returns the sorted top slice, and `applyShortTermPromotions()` re-enforces the same score threshold before writing. The issue is recent, non-maintainer-authored, has no protected labels, and no related item in the provided context clearly supersedes this exact serendipitous sub-threshold sampling request.

## Best Possible Solution

Keep this open for maintainer/product review. A good implementation would add an explicit, bounded wildcard policy in `memory-core` deep promotion, preserve existing hard safety gates, make the randomness deterministic or auditable enough for tests and logs, and update both ranking and apply semantics plus docs/config if the option becomes user-facing.

## Risks / Open Questions

- A wildcard promotion path would change memory-selection behavior and may affect determinism, auditability, and user trust if random sampling is not bounded and explainable.
- The apply path currently enforces the same score gate as ranking, so the implementation needs a deliberate contract for which sub-threshold candidates are eligible and how they bypass or satisfy apply-time validation.
