## Summary

Keep open. Current main has meaningful Bonjour mitigations, including a bounded discovery-advertise startup timeout and a sliding-window cap for flapping advertiser restarts, but it still preserves the broader default-on bundled plugin policy that the reporter and later comments identify as the remaining upgrade/startup problem.

## What This Changes

The issue reports that upgrading through the 2026.4.26 plugin set can leave many bundled plugins enabled or inspected by default, with Bonjour probing/watchdog churn and bundled runtime-dependency work making gateway startup unusable until the operator trims to a minimal plugin set.

## Best Possible Solution

Settle and implement a maintainable default startup policy: either make optional bundled services like Bonjour opt-in/configured-only after upgrade, or keep defaults while proving startup, status, health, and runtime-deps repair only load or stage the configured operational set on constrained hosts. Operator recovery docs should describe a minimal profile/safe-mode path and provider recheck steps.

## Reproduction Assessment

Yes for the remaining policy shape: current main statically reproduces Bonjour as default-on and default-enabled bundled plugin/runtime-deps inclusion through manifest, resolver, and test evidence. The exact resource-constrained Linux restart churn is environment-specific and was not live-reproduced here.

## Solution Assessment

Unclear until maintainers choose the product policy. The landed Bonjour restart cap and discovery timeout are good narrow mitigations, but they are not the best complete solution for the broader upgrade/default plugin surface reported here.
