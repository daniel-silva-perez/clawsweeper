## Summary

Keep open. Current main contains useful mitigations for Bonjour failure handling and bundled runtime-dependency staging, but it still ships Bonjour as enabled-by-default startup behavior and still repairs/loads default-enabled bundled plugin surfaces unless users explicitly constrain them. The issue also has an open narrow Bonjour PR (#74242) and a related open CPU/startup regression report (#74328), so the remaining work needs maintainer policy and triage rather than cleanup closure.

## What This Changes

The issue requests that upgrades avoid enabling or initializing a broad set of optional bundled plugins, especially Bonjour, in a way that can block gateway startup or normal operation.

## Best Possible Solution

Keep this issue open. Maintainers should first choose the intended default policy: make Bonjour and other optional bundled services opt-in/minimal after upgrade, or keep them default-on while proving they cannot delay readiness, restart the gateway, or stage unused dependencies on constrained hosts. A narrow Bonjour watchdog fix such as #74242 can land separately, but the broader solution likely belongs in Bonjour manifest/default policy, gateway startup plugin planning, bundled runtime-dependency selection, and operator recovery docs or a safe/minimal plugin profile.
