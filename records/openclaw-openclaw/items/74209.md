## Summary

Keep open. Current main contains meaningful Bonjour guardrails, but it still keeps Bonjour and other default-enabled bundled plugins in the default startup/runtime-dependency path, while the reporter, a maintainer comment, and a later macOS data point all identify the unresolved problem as broader default bundled-plugin/minimal-profile policy rather than only the Bonjour watchdog loop.

## What This Changes

Requests that upgrades avoid enabling or initializing a broad default bundled plugin set, especially Bonjour, in a way that can block gateway startup, and suggests a configured-only, minimal-profile, or safe-mode recovery path.

## Best Possible Solution

Define and implement a maintainable default startup policy for upgrades: load and stage only configured or explicitly enabled operational plugins by default, preserve intentional core exceptions, keep explicit disables authoritative, and document a minimal-profile/safe-mode recovery path for operators. The existing Bonjour timeout and restart caps should remain as defensive guardrails, not as the whole fix.

## Reproduction Assessment

Yes for the remaining default-policy shape: current main statically reproduces Bonjour as default-on/default-startup and tests assert default-enabled sidecars and runtime deps at idle. The exact constrained Linux restart churn is environment-specific and was not live-reproduced here.

## Solution Assessment

No for closing this as already solved. The current implementation is a useful narrow mitigation for Bonjour and discovery blocking, but it does not resolve the broader upgrade/startup policy that keeps default-enabled bundled plugins in the normal path.
