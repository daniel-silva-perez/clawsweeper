## Summary

Keep open. Current main has a narrow Bonjour restart-loop mitigation, but the broader reported problem remains: Bonjour is still default-enabled startup behavior, default-enabled bundled plugin/runtime-deps policy remains broad, and maintainer/user follow-up explicitly says the minimal-profile/default-plugin policy is unresolved.

## What This Changes

The issue asks OpenClaw to avoid enabling, loading, or staging broad optional bundled plugins after upgrades in a way that can block gateway startup, especially when Bonjour is unused or stuck in mDNS probing.

## Best Possible Solution

Define and implement a maintainer-approved upgrade/startup policy that keeps optional bundled plugins out of the critical path unless configured or explicitly allowed, keeps Bonjour either opt-in or provably non-blocking, and gives operators a safe minimal-profile recovery path with docs and targeted tests.

## Reproduction Assessment

Yes. The issue includes concrete Linux npm-global upgrade steps, gateway/systemd logs, repeated Bonjour probing/watchdog lines, and recovery evidence after trimming plugins; current main still matches the default-on startup surfaces needed to reproduce the policy risk.

## Solution Assessment

No. The current main mitigation is a good narrow fix for one Bonjour restart loop, but it is not the best complete solution for the reported upgrade failure class because default plugin activation, startup staging, and minimal-profile recovery remain policy-level work.
