## Summary

Keep open. Current main has useful Bonjour guardrails and bounded discovery startup, but it still treats Bonjour and other default-enabled bundled plugins as part of normal startup/runtime-dependency policy; the issue discussion also has maintainer and operator comments saying the remaining problem is the broader default bundled-plugin/minimal-profile policy, not only the Bonjour restart loop.

## What This Changes

The issue asks OpenClaw to prevent upgrades from enabling or initializing a large unused bundled-plugin set, especially Bonjour, in a way that can block or destabilize gateway startup.

## Best Possible Solution

Define a durable upgrade/startup policy that loads only configured, explicitly enabled, or intentionally core-required plugins by default, preserves explicit disables and allowlists, keeps Bonjour defensive caps/timeouts, and documents a minimal-profile or safe-mode recovery path for operators.

## Reproduction Assessment

Yes for the remaining policy shape: current main statically reproduces Bonjour as default-enabled/on-startup, default bundled startup inclusion, and default-enabled runtime-deps staging. The exact small-host Linux watchdog/restart churn remains environment-specific and was not live-reproduced in this read-only review.

## Solution Assessment

No, current main is not a complete solution for this issue. The Bonjour restart cap and discovery timeout are useful narrow mitigations, but the requested fix needs a broader compatibility/product decision about default bundled-plugin startup scope and upgrade recovery.
