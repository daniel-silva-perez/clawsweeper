## Summary

Keep open. Current main still does not implement a user-facing configurable exec approval timeout: the typed config and strict runtime schemas lack `tools.exec.approvalTimeoutMs` or `approvals.timeoutMs`, and the active exec approval paths still use the fixed 30-minute default. The PR remains a relevant implementation candidate, but it needs maintainer review for config shape, default alignment, bounds, propagation tests, and the security-sensitive timeout behavior.

## Best Possible Solution

Keep this PR open for maintainer review, or explicitly supersede it under #51287 if maintainers choose a different config shape. A complete fix should choose the public config key, add bounded validation plus schema/help/docs updates, thread the resolved timeout through gateway and node exec approval registration and transport waits, align the documented default with current main, and add focused propagation tests.

## Risks / Open Questions

- The proposed timeout is security-sensitive: a positive-integer-only setting can make approvals expire unusably fast or keep approval and transport waits open for a very long time unless maintainers choose explicit lower and upper bounds.
- The PR text/help describes a 120000 ms default, while current main uses and documents a 1,800,000 ms default.
- The provided PR context does not show focused tests for config parsing, bounds, precedence, gateway/node propagation, expiry behavior, or transport-timeout alignment.
