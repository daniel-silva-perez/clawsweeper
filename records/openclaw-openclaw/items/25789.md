## Summary

Keep this PR open. Current main still does not implement a user-facing configurable exec approval timeout, and the existing maintainer review explicitly kept the PR open for follow-up. The latest PR head is more complete than the original bot review, but it remains security-sensitive and needs maintainer choices around config shape, bounded validation, default alignment, and tests before landing or superseding under #51287.

## Best Possible Solution

Keep this PR open for maintainer review, or explicitly supersede it under #51287 if maintainers choose a different config shape. A complete path should choose tools.exec.approvalTimeoutMs versus approvals.timeoutMs, add bounded validation and schema/help/docs updates, thread the resolved timeout through gateway and node exec approval registration plus transport waits, align the documented default with current main, and add focused propagation tests.

## Risks / Open Questions

- The proposed timeout is security-sensitive: positive-integer-only validation can make approvals expire unusably fast or keep approval and transport waits open for a very long time unless maintainers set explicit lower and upper bounds.

- The provided PR file list does not include focused tests for config parsing, precedence, gateway/node propagation, expiry behavior, or transport-timeout alignment.
