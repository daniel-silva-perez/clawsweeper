## Summary

Keep open. Current main still exposes the reported mismatch: the tool/docs contract says `sessionKey="current"` means the caller session, but `session_status` only accepts the `resolveSessionReference` branch when it resolved through a session-id path and can still fall through to `Unknown sessionKey`. Two open PRs explicitly target this issue, so cleanup should happen by reviewing and landing or closing those PRs rather than closing the issue first.

## What This Changes

The issue asks for `session_status` to resolve explicit `sessionKey="current"` to the requester session for channel-plugin driven agents, while preserving literal `current` session-key/session-id precedence.

## Best Possible Solution

Keep the issue open until one canonical fix PR lands. The implementation should add a narrow explicit-current requester fallback in `session-status-tool.ts` after literal `current` key/session-id resolution, keep visibility checks treating the target as self, and add a focused channel-style requester regression test.

## Reproduction Assessment

Yes. The report gives a concrete channel-style requester key and log sequence, and current main still has a static reproduction path where explicit `sessionKey="current"` can miss store/gateway resolution and fall through to `Unknown sessionKey`.

## Solution Assessment

Yes. A final, literal-`current` fallback in `session-status-tool.ts` after the existing literal-session paths is the narrowest maintainable fix; closing this issue before #74142 or #72306 merges would be premature.
