## Summary

Keep open. Current main still has the reported `session_status` resolution gap, and the provided timeline shows open linked implementation PRs (#74142 and #72306) with closing syntax for this bug, so this issue should remain open until a fix merges or maintainers choose a different canonical path.

## What This Changes

The issue requests that `session_status` treat `sessionKey="current"` as the caller's own session for channel-plugin-driven agents without breaking literal `current` session key or sessionId precedence.

## Best Possible Solution

Land one narrow fix that adds an explicit `sessionKey="current"` fallback to the requester session only after literal `current` key/sessionId resolution has failed, with regression coverage for a channel-style requester and preservation of the existing visibility guards.

## Reproduction Assessment

Yes. The issue includes concrete gateway logs, and current main still has a static reproduction path: an unstored channel-style requester plus explicit `sessionKey: "current"` falls past the session-id-only branch and reaches the unknown-session throw.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix if it is placed after literal-current resolution and keeps the existing visibility checks; the immediate next step is to review and land one of the linked PRs rather than closing this issue early.
