## Summary

Keep open. Current main still matches the reported failure shape, and there are open closing PRs targeting this exact bug, so the issue should remain open until maintainers review and land one fix or choose a different implementation.

## What This Changes

Fix `session_status` so explicit `sessionKey="current"` resolves to the requester session for channel-plugin agents instead of throwing and forcing retry latency.

## Best Possible Solution

Keep the issue open while maintainers review the linked PRs. The end state should be a narrow `session_status` fix that treats explicit `sessionKey="current"` as the requester after literal-current key/sessionId and gateway resolution paths miss, with a regression test for an unstored channel-style requester.

## Reproduction Assessment

Yes. The issue gives concrete gateway log steps, and the current code path can be reproduced with a focused unit case: channel-style requester key, empty session store, and `session_status({sessionKey: "current"})` should currently reach the `Unknown sessionKey: current` throw.

## Solution Assessment

Yes, with review. A final `session_status`-local requester fallback after literal-current resolution is the narrowest maintainable direction; expanding the first-party client alias set would blur client-label semantics and risk changing other session tools.
