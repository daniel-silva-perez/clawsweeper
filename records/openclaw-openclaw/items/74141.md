## Summary

Keep open. Current `main` still has the reported `session_status` resolution shape, and PR #74142 is open and explicitly closes this issue. The tool/docs contract tells agents to use `sessionKey="current"`, but the current resolver only treats first-party client labels as aliases and `session-status-tool` only follows the `resolveSessionReference` fallback when it resolved through a session ID. The best cleanup path is to review, validate, and either merge or close the linked PR rather than closing this issue first.

## Best Possible Solution

Keep this issue open and use PR #74142 as the canonical implementation candidate. Review or replace that PR with a narrow `session-status-tool` fix that falls back from literal `sessionKey="current"` to the requester session only after literal-current store lookup and gateway resolution fail, while preserving visibility guards and existing literal-current behavior. Then close the issue through the merged PR.
