## Summary

Keep open. Current main still supports explicit model-provider env proxy configuration, but it does not emit the requested startup warning when proxy environment variables are present and remote providers lack `request.proxy`; the direct implementation PR #68023 is closed and unmerged.

## What This Changes

Request a non-breaking gateway startup warning when proxy environment variables are detected but remote model providers are not explicitly configured to use `models.providers.<provider>.request.proxy.mode = "env-proxy"`.

## Best Possible Solution

Add a gateway startup diagnostic that reuses the existing proxy-env helpers, inspects configured model providers for remote HTTP(S) targets without explicit `request.proxy`, emits one actionable remediation message, and covers warning plus suppression cases in focused tests.

## Repair Work Prompt

Implement the warning requested by this issue without changing provider request routing semantics. Add a gateway startup diagnostic in `src/gateway/server-startup-log.ts` that detects configured proxy env (`HTTP_PROXY`, `HTTPS_PROXY`, lowercase variants as appropriate), inspects configured `models.providers` entries, identifies remote HTTP(S) provider targets that do not have any explicit `request.proxy`, and logs one actionable warning recommending `models.providers.<provider>.request.proxy.mode = "env-proxy"`. Suppress the warning when no proxy env is present, all remote providers have `request.proxy`, or providers are local/loopback/private/LAN-style targets that should not be routed through env proxy by default. Reuse existing proxy-env helpers where possible; add a small helper only if needed for remote-target classification.
