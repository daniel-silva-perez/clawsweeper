## Summary

Keep open. Current main still lacks the requested startup warning: provider request proxy support exists and env proxy helpers exist, but `logGatewayStartup` does not inspect proxy environment variables or configured remote providers, and model-provider guarded fetches still use explicit dispatcher policy rather than ambient proxy environment by default. The directly related implementation candidate, #68023, is closed and unmerged.

## What This Changes

Add a non-breaking gateway startup warning when proxy environment variables are present but remote model providers lack explicit `models.providers.<provider>.request.proxy.mode = "env-proxy"` configuration.

## Best Possible Solution

Add a warning-only gateway startup diagnostic that reuses existing proxy-env helpers, identifies configured remote HTTP(S) model providers without any explicit request proxy override, emits one actionable remediation message, and covers warning plus suppression cases in focused startup-log tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to set `HTTPS_PROXY`, configure a remote model provider without `request.proxy`, and call the current startup logger: the inspected source has no proxy/provider warning branch and the tests have no proxy observability coverage.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix because the capability already exists and model-provider routing should remain explicit; a safer implementation should not alter dispatcher selection or default proxy behavior.
