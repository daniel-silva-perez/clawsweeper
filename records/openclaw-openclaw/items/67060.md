## Summary

Keep open. Current main still has explicit model-provider env-proxy support, but the gateway startup path does not emit the requested warning when proxy environment variables are present and remote providers lack explicit request proxy configuration; the directly related implementation PR, #68023, is closed and unmerged.

## What This Changes

The issue requests a warning-only gateway startup diagnostic for proxy-required environments when HTTP_PROXY/HTTPS_PROXY is set but remote model providers do not explicitly enable `request.proxy.mode = "env-proxy"`.

## Best Possible Solution

Add a gateway startup diagnostic that reuses the existing proxy-env helpers, inspects configured model providers for remote HTTP(S) targets without explicit request proxy configuration, emits one actionable remediation message, and covers warning and suppression cases in focused tests.

## Repair Work Prompt

Implement the warning-only startup observability fix for the provider env-proxy report. In `src/gateway/server-startup-log.ts`, add a check during `logGatewayStartup` that detects configured proxy environment variables and warns once when one or more configured model providers target remote HTTP(S) endpoints that would need env proxy but do not explicitly configure `models.providers.<provider>.request.proxy`. Reuse existing helpers from `src/infra/net/proxy-env.ts` where possible, including NO_PROXY-aware behavior. Do not auto-enable env proxy, do not change provider request routing semantics, and do not broaden core defaults. Suppress warnings for local/non-remote providers such as loopback, localhost, `.local`, LAN/private/tailnet/self-hosted targets, providers bypassed by NO_PROXY, and providers with any explicit request proxy configuration. Add focused tests in `src/gateway/server-startup-log.test.ts` for: warning with proxy env plus remote unconfigured provider; no warning with no proxy env; no warning when all remote providers have request proxy configured; no warning for local/non-remote providers; and NO_PROXY-bypassed remote targets if the helper path supports it cleanly. Treat #68023 as useful context but do not copy it blindly. Add a single-line changelog entry in the active version section if current repo policy treats this user-visible startup warning as changelog-worthy; credit @wnlen and prior implementation context where appropriate, never bots.
