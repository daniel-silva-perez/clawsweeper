## Summary

Keep this issue open. Current main still does not implement a built-in L1-L4 command risk classifier for host exec approvals. The exec path remains security/ask policy, allowlist, durable approval, safeBins, and strict inline-eval based. ACP has a separate readonly tool auto-approval classifier, but it explicitly treats exec-capable tools as non-auto-approved and does not solve host exec command classification.

## Best Possible Solution

Keep this issue open for maintainer product/security design. If pursued in core, implement an explicit opt-in exec policy layer with a documented threat model, UI/config affordances, and regression coverage for safe reads, pipelines, shell wrappers, interpreters, Windows/POSIX parsing, and dangerous-command denial. If maintainers prefer plugin-first classification, first add or document an exec-specific plugin API for classify/block/require-approval/auto-allow decisions without weakening host approval boundaries.

## Risks / Open Questions

- A broad built-in safe/dangerous command classifier would materially change host exec security posture and needs explicit maintainer design for shell parsing, wrappers, interpreters, pipelines, PATH resolution, Windows/POSIX differences, aliases, env, redirection, and destructive-pattern semantics.
- Closing as ClawHub is premature unless maintainers first expose or document an exec-specific plugin policy hook that can classify, block, require approval, and auto-allow while preserving existing host exec approval boundaries.
