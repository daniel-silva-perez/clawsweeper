## Summary

Keep open because current main still does not implement a built-in L1-L4 command risk classifier for exec approvals, and the newer discussion adds security design concerns rather than proof of shipped behavior. The request would change host-exec auto-allow/auto-deny semantics, so it needs maintainer/security direction before a repair PR is safe to queue.

## What This Changes

The issue requests a built-in L1-L4 command risk classifier for exec approvals, including auto-approval for safe commands and auto-denial for dangerous commands.

## Best Possible Solution

Keep the issue open for maintainer/security design. A maintainable end state would be an explicit opt-in exec policy layer with a documented threat model, parsed argv or shell-AST semantics, UI/config controls, and regression coverage across gateway and node hosts; if plugin-first is preferred, add or document an exec-specific extension seam that can deny, require approval, or narrowly allow without bypassing the host approval boundary.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows allowlist, safe-bin, strict-inline-eval, and durable approval behavior but no built-in L1-L4 classifier.

## Solution Assessment

Unclear; the requested static L1/L4 classification is not yet proven to be the best solution because it changes security-sensitive host exec behavior. A safer path is maintainer-approved design around parsed command shapes and opt-in policy, or a first-class plugin seam, rather than broad command-name heuristics.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the requested classifier would change when host commands are auto-approved or force-denied.

Concerns:

- **[medium] Heuristic auto-approval can weaken host exec boundaries:**
  - body: Broadly classifying commands like cat, grep, curl, sed, awk, xargs, find, or git as safe by text pattern can miss argv context, shell composition, redirects, PATH hijacking, wrappers, aliases, platform differences, and file operands; any implementation needs an explicit threat model and maintainer/security review.
  - confidence: 0.9
