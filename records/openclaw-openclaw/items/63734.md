## Summary

Keep this open because current main still has no built-in L1-L4 command risk classifier for exec approvals, and the new discussion adds a possible L4 prototype rather than evidence that the core behavior has shipped. The remaining work changes the host-exec security model, so it needs maintainer/security design before an implementation PR is safe to queue.

## What This Changes

Request to add a built-in command risk classifier for exec approvals, including L1 safe auto-approval and L4 dangerous auto-denial behavior.

## Best Possible Solution

Keep this issue open for maintainer and security design. If core accepts it, implement an explicit opt-in exec policy layer with a documented threat model, UI/config affordances, and regression coverage across gateway and node hosts; if the preferred path is plugin-first, first add or document an exec-specific plugin API that can classify/block/require-approval/auto-allow without weakening the host approval boundary.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is source and docs inspection: the exec approval contract, gateway path, node path, and docs all show allowlist/safe-bin/durable-approval behavior rather than a built-in L1-L4 classifier.

## Solution Assessment

Unclear; the requested direction is not yet proven to be the best fix because broad command heuristics are security-sensitive. A safer maintainable path is an explicit opt-in design or plugin API review first, with any L4 dangerous-command detector evaluated as one component rather than merged as an ad hoc preflight layer.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because a built-in exec command classifier could change when host commands are auto-allowed or force-denied.

Concerns:

- **[medium] Heuristic auto-approval can weaken host exec boundaries:**
  - body: Classifying broad commands like cat, grep, curl, sed, awk, xargs, or find as safe by text pattern alone can miss shell parsing, wrappers, aliases, PATH hijacking, file operands, interpreters, and platform differences; this needs maintainer/security design before implementation.
