## Summary

Keep this issue open. Current main has useful cron state tracking, run history, failure alerts, and generic `/health`/`/readyz` probes, but it still does not satisfy the central requests: structured per-attempt cron execution logging, warning logs for skipped/non-throwing failures, cron-specific health/readiness status, or `openclaw cron list` annotations for overdue/repeatedly failing jobs. Related PR #46501 was directly relevant but closed unmerged, and open PR #44363 overlaps enough that maintainers should decide whether to harvest it or pursue a narrower cron-observability PR.

## Best Possible Solution

Keep this issue open. The best path is a focused cron-observability change: add structured logging for scheduled timer runs, startup catch-up/missed runs, and manual runs; log warnings for skipped and non-throwing error outcomes; expose cron health either as `/health/cron` or an explicit component in the existing readiness/health model; and add `openclaw cron list` annotations for overdue and repeatedly failing jobs. Review #44363 and the closed #46501 for prior art, but do not treat either as shipped.

## Risks / Open Questions

- Adding per-attempt logs for frequent cron jobs could increase log volume; fields should stay structured and avoid leaking prompts, channel targets, tokens, or delivery payloads.
- Cron health should integrate with existing `/health`, `/readyz`, run-history, task, failure-alert, and backoff semantics without duplicating alerts or changing scheduling behavior.
- A cron-specific public health route must avoid exposing sensitive job names, prompts, recipients, or model/provider details to unauthenticated infrastructure probes.
- Open PR #44363 overlaps cron diagnostics work, so maintainers should coordinate before starting a duplicate implementation lane.
