## Summary

Keep open. Current main still sends Gemini CLI OAuth token exchange and project-discovery requests through the default strict guarded-fetch path, which intentionally ignores environment proxy settings unless trusted proxy mode or managed proxy lifecycle is enabled. The linked #47518 PR is an open implementation candidate for this exact plugin-owned path, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue reports that google-gemini-cli OAuth token exchange fails behind a macOS HTTP(S) proxy because the guarded fetch path does not respect proxy routing.

## Best Possible Solution

Review and land #47518 or a narrower replacement that keeps the fix inside the Google plugin OAuth helper: use trusted env-proxy routing only for the known first-party Google OAuth/project endpoints when the target will actually use env proxy, preserve strict DNS-pinned defaults elsewhere, add focused tests for `HTTP_PROXY`/`HTTPS_PROXY` plus `NO_PROXY`, and clarify whether macOS system proxy without env vars is supported or requires OpenClaw proxy configuration.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: the OAuth token exchange and project discovery call `extensions/google/oauth.http.ts`, which currently invokes `fetchWithSsrFGuard` without trusted proxy mode, while the shared guard tests assert strict mode ignores env proxy by default.

## Solution Assessment

Unclear as written. The reporter's `trusted_env_proxy` direction is the narrow maintainable fix for `HTTP_PROXY`/`HTTPS_PROXY` or managed proxy environments, but the macOS system-proxy-without-env boundary should be settled while reviewing #47518.
