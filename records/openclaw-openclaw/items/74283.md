## Summary

Keep open. Current main still allows the reported approval-followup failure shape: unmatched `resultText` is parsed as unstructured text, generic-sanitized, and sent directly to the originating channel when the follow-up `agent` session resume fails. The discussion correctly distinguishes this from the already-closed restart delivery-recovery path, and the remaining work is a security-sensitive delivery-boundary hardening plus producer trace.

## What This Changes

The issue requests hardening of exec approval follow-up recovery so raw internal handler/source-like text cannot be delivered directly to a chat channel after session resume fails.

## Best Possible Solution

Add a fail-closed boundary in the exec approval direct fallback so only verified structured exec completion or denial payloads are eligible for direct channel delivery; malformed/source-like payloads should be dropped or replaced with generic unavailable copy. Then trace and fix the producer so handler-source-like state cannot become `resultText`, with regression tests for the reported JavaScript-like snippets.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, rejected `callGatewayTool("agent", ...)`, and JavaScript-like `resultText`; current main parses it as `kind: "other"` and sends the sanitized raw text through direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal. The Layer 1 fail-closed direct-delivery guard is the narrow immediate containment, but the best complete fix should also identify and repair the producer path that can populate `resultText` with internal handler state.
