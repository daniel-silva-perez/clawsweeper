## Summary

Keep open. Current main still has the reported approval-followup leak shape: the session-resume failure path can fall back to direct channel delivery, and malformed or source-like `resultText` that does not match a structured exec result is only passed through the generic sanitizer before send. The post-release main change around this path adjusted direct delivery and prefixing but did not add a fail-closed guard or regression coverage for raw handler-source-like payloads.

## What This Changes

Prevent the `/approve` post-approval follow-up fallback from sending raw internal handler/source-like text to Telegram when automatic session resume fails.

## Best Possible Solution

Add a fail-closed boundary for direct exec approval follow-ups: only verified structured exec completion/denial payloads should be eligible for channel fallback, malformed or source-like payloads should become a generic unavailable-status message or be dropped, and the producer should be traced so handler-source-like text cannot become `resultText`. Cover the session-resume failure path with a regression test using the reported raw JavaScript-like shapes.
