## Summary

Keep open. Current main still has the approval-followup leak boundary described here: unrecognized `resultText` is parsed as `kind: "other"`, generically sanitized, and sent directly to the originating channel when the follow-up agent session resume fails. The comments correctly distinguish this from the older restart recovery path; this remaining path needs security-sensitive delivery-boundary hardening and producer tracing.

## What This Changes

Harden exec approval post-approval follow-up recovery so source-like or otherwise unverified `resultText` cannot be sent directly to Telegram after session resume fallback.

## Best Possible Solution

Add a fail-closed boundary in the exec approval direct fallback so only recognized structured completion/denial payloads, or otherwise verified safe user-facing text, can be delivered directly after session resume failure. Then trace and repair the producer that can place handler-state/source-like text into `resultText`, with regression coverage for the reported `rams.resultText.trim()` and `const sessionOnlyOriginChannel` shapes.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, Telegram route, rejected `callGatewayTool("agent", ...)`, and JavaScript-like unstructured `resultText`; current main parses it as `kind: "other"` and sends the sanitized raw text through direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal. The Layer 1 fail-closed delivery guard is the narrow immediate containment, but the best complete solution should also repair the upstream producer so internal handler state cannot become `resultText`.
