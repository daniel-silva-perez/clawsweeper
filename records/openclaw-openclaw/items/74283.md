## Summary

Keep open. Current main has a dedicated exec-approval followup path and a sanitized direct fallback, but the fallback still accepts non-structured `resultText` through the generic user-facing sanitizer. I did not find a fail-closed guard or regression test for the reported raw JS/internal-handler-source slices, so the issue is not safely closeable as implemented.

## Best Possible Solution

Keep this open and treat it as approval-followup leak hardening. The best fix is to make `sendDirectFollowupFallback` fail closed unless `resultText` is a verified exec followup shape, replace malformed/source-like payloads with a generic unavailable-status message, and then trace why the producer ever supplied handler-source-like text. Add targeted regression coverage for session-resume failure plus raw JS-like `resultText`.
