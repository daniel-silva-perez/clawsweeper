## Summary

Keep open. Current main still has the reported approval-followup leak shape: session-resume failure can fall back to direct channel delivery, and unmatched or source-like result text is sanitized and sent rather than verified or fail-closed. The recent main-side changes adjusted direct mode and prefix behavior, but did not add the guardrail or regression coverage described in the report and follow-up comment.

## What This Changes

The issue asks OpenClaw to stop `/approve` post-approval follow-up fallback from sending raw internal handler or source-like `resultText` to Telegram or other channels when session resume fails.

## Best Possible Solution

Add a fail-closed boundary for direct exec approval follow-ups: only verified structured exec completion or denial payloads should be eligible for channel fallback, malformed or source-like payloads should become a generic unavailable message or be dropped, and the producer should be traced so handler-source-like text cannot become `resultText`. Add regression coverage for the session-resume failure path using the reported JavaScript-like snippets.
