## Summary

Keep open. Current main still permits the reported approval-followup leak shape: unrecognized `resultText` is treated as raw text, sanitized generically, and then sent through the direct channel fallback after session resume fails. The issue discussion also has an independent confirmation that #72184 covered the restart recovery path, not this `/approve` follow-up path.

## What This Changes

The issue requests hardening the `/approve` exec approval follow-up path so session-resume fallback does not send raw internal handler source or local-state text to Telegram.

## Best Possible Solution

Add a fail-closed boundary for direct exec approval follow-ups: only verified structured exec completion or denial payloads should be eligible for channel fallback, malformed/source-like unstructured payloads should be dropped or replaced with a generic unavailable status, and the upstream producer should be traced so internal handler fragments cannot populate `resultText`.

## Reproduction Assessment

Yes for the code path: mock `callGatewayTool` to reject, provide a deliverable Telegram/Discord route plus a session key, and pass a source-like unstructured `resultText`; current main will format it as `kind: "other"` and send it through `sendMessage`. Live Telegram was not exercised in this read-only review.

## Solution Assessment

Yes in direction, but it needs maintainer scoping. The contained Layer 1 guard proposed in the discussion is the narrowest first fix, while the producer-side root cause should be tracked or handled separately so the delivery boundary is not the only defense.
