## Summary

Keep this issue open. Current main still has no `tools.exec.shell` contract: shell selection is derived inside the exec runtime, the public exec config type and strict zod schema do not accept `shell`, and the exec docs list the current PowerShell/SHELL behavior without a shell override. The linked implementation PR #39413 remains open, so the next step is maintainer review of that command-execution contract rather than stale cleanup or a duplicate fix lane.

## What This Changes

Request adding a configurable `tools.exec.shell` override, especially so Windows users can run exec commands through WSL or another bash-compatible shell instead of the built-in PowerShell default.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable/argument semantics, startup or first-use validation behavior, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, docs/schema metadata, and targeted exec tests, then land it through PR #39413 or a maintainer-directed replacement.
