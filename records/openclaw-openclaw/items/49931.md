## Summary

Keep this issue open. Current main still does not expose a supported `tools.exec.shell` contract: shell selection is resolved inside the exec runtime, the public exec config type and strict schema omit `shell`, and the exec docs describe fixed platform defaults rather than a configurable override. The discussion and linked PR show this is a real command-execution contract question, not stale cleanup.

## What This Changes

The issue requests a supported `tools.exec.shell` configuration option so users can override the exec tool's default shell, especially for Windows/WSL bash workflows.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable-plus-argument semantics, validation behavior, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, schema/help/docs updates, and targeted exec tests, then land it through PR #39413 or a maintainer-directed replacement.
