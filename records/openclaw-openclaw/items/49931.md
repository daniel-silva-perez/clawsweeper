## Summary

Keep this issue open. Current main still does not expose a `tools.exec.shell` contract: local exec shell selection is resolved by the helper and runtime, while the public config type, strict schema, config threading, and exec docs do not accept or document the requested key. The related implementation PR #39413 remains open, and the discussion raises real command-execution security/product questions, so this needs maintainer review rather than stale cleanup or a duplicate automated fix lane.

## What This Changes

Add a global and per-agent `tools.exec.shell` option so exec commands can run through an operator-selected shell such as WSL bash, Git Bash, zsh, or PowerShell instead of the current platform defaults.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable/argument semantics, validation behavior, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, schema/help/docs updates, and targeted exec tests, then land it through PR #39413 or a maintainer-directed replacement.
