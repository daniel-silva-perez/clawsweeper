## Summary

Keep this issue open. Current main still lacks a supported `tools.exec.shell` contract: the exec runtime derives the shell internally, config types/schema do not accept the requested key, docs describe only the existing `SHELL`/PowerShell behavior, and the related implementation PR remains open for maintainer review.

## What This Changes

The issue requests a configurable `tools.exec.shell` option so exec tool invocations can use a user-selected shell such as WSL bash, bash, zsh, `pwsh`, or another executable.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable-plus-argument semantics, validation behavior, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, schema/help/docs updates, and targeted exec tests, then land it through PR #39413 or a maintainer-directed replacement.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `getShellConfig()` has no config input, the runtime calls it directly, the strict schema omits `shell`, and the exec docs do not document `tools.exec.shell`.

## Solution Assessment

Unclear. The requested config direction is plausible, but the best implementation needs maintainer review because shell selection is a command-execution policy boundary rather than a simple string passthrough.
