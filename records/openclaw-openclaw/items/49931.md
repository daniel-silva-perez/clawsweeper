## Summary

Keep this issue open. Current main still does not expose a supported `tools.exec.shell` contract: shell selection is resolved internally, the public exec config type and strict runtime schema omit `shell`, and the exec docs describe fixed platform defaults rather than a configurable override. The related implementation PR #39413 remains open, and the command-execution policy implications need maintainer review rather than stale cleanup.

## What This Changes

The issue requests a `tools.exec.shell` configuration option to override the default shell used by exec calls, especially for Windows users who want WSL or bash-style command parsing.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable-plus-argument semantics, validation behavior, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, schema/help/docs updates, and targeted exec tests, then land it through PR #39413 or a maintainer-directed replacement.
