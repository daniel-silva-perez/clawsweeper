## Summary

Keep this issue open. Current main still lacks a supported `tools.exec.shell` contract: shell selection is resolved inside the exec helper/runtime, while the public config type, strict schema, generated config metadata, config threading, and exec docs do not accept or document the requested key. The linked implementation PR #39413 is still open, and the discussion identifies real command-execution policy questions that need maintainer review rather than stale cleanup.

## What This Changes

Add a global and per-agent `tools.exec.shell` option so exec commands can run through an operator-selected shell such as WSL bash, Git Bash, zsh, or PowerShell instead of the current platform defaults.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable-plus-argument semantics, validation behavior, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, schema/help/docs updates, and targeted exec tests, then land it through PR #39413 or a maintainer-directed replacement.
