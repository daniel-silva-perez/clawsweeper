## Summary

Keep this issue open. Current main still lacks a supported `tools.exec.shell` contract, and the related implementation PR remains open, so this is valid follow-up work rather than stale, duplicate, or implemented cleanup.

## What This Changes

The issue requests a supported `tools.exec.shell` configuration option so users can override the default shell used by exec tool invocations, especially on Windows/WSL.

## Best Possible Solution

Define a supported exec shell override contract with global and per-agent precedence, executable-plus-argument semantics, validation behavior, Windows/WSL cwd and PATH expectations, approval and safe-bin interactions, schema/help/docs updates, and targeted exec tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the shell helper has no config input, the runtime calls it directly, the strict config schema rejects unknown exec keys, and the docs do not document `tools.exec.shell`.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation needs maintainer review because shell selection is a command-execution policy boundary rather than a simple string passthrough.
