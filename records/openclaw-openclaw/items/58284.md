## Summary

Keep open. Current main still only exposes and wires TTL-gated `cache-ttl` pruning, so continuous active sessions can avoid pruning until the cache touch ages past the TTL. The related open PR #43238 is a useful implementation candidate for rolling pruning, but it is not merged and should be reviewed, refreshed, or replaced rather than using it as a close reason now.

## What This Changes

Request native turn-age or rolling context pruning so active sessions trim old tool results every turn instead of only after cache-TTL inactivity.

## Best Possible Solution

Add a native rolling or turn-age pruning path in the existing context-pruning hook: update config types, Zod/generated schema, docs, and tests; register the hook without cache-TTL provider eligibility for the rolling mode; preserve current `cache-ttl` behavior; and use #43238 as the source PR or replace it with a current-main PR if the branch cannot be refreshed.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with `contextPruning.mode: "cache-ttl"` and a fresh `lastCacheTouchAt`, `extension.ts` returns before `pruneContextMessages`, and the existing test proves a second pass is skipped after the TTL window resets.

## Solution Assessment

Unclear as written. Native context-pruning support is the right boundary because plugins cannot currently change prompt assembly here, but maintainers should decide whether the shipped knob should be a new `rolling` mode, the issue's nested `rollingPrune.keepLastTurns`, or both for compatibility.
