## Summary

Keep this issue open. Current main still does not accept the requested `agents.defaults.memorySearch.dimensions` or per-agent `memorySearch.dimensions` key, and the OpenAI-compatible inline and batch embedding paths still do not forward a `dimensions` request parameter. Existing `outputDimensionality` support is narrower provider-specific behavior for Gemini/Bedrock-style providers, while the LanceDB plugin has a separate `embedding.dimensions` surface that does not satisfy this built-in memory-search request.

## Best Possible Solution

Keep this issue open. Maintainers should choose the public config contract (`dimensions`, `outputDimensionality`, or an alias pair), then update schema/types/help/docs, default and per-agent merge logic, OpenAI-compatible inline and batch request builders, provider cache-key data, and reindex identity. Add focused tests for schema acceptance/rejection, config resolution, inline request body, batch request body, cache-key separation, and reindex behavior.

## Risks / Open Questions

- A naive unconditional `dimensions` pass-through could regress OpenAI-compatible providers or models that reject that field; the implementation should be provider/model-capability aware or clearly scoped to compatible endpoints.
- Changing embedding vector dimensions must invalidate embedding cache and reindex identity. Current memory-core provider keys hash runtime `cacheKeyData`, while OpenAI cacheKeyData does not include any dimension setting.
- The existing LanceDB plugin `embedding.dimensions` setting may be useful for some users, but it is a separate plugin configuration surface and should not be treated as resolving the requested built-in `memorySearch` default/per-agent contract.
