## Summary

Keep this issue open. Current main still does not implement the requested `agents.defaults.memorySearch.dimensions` config key, and the OpenAI-compatible memory embedding paths still do not forward an OpenAI `dimensions` request parameter for inline or batch memory indexing. Current support is a narrower `outputDimensionality` field documented and wired for Gemini/Bedrock-style providers, not the OpenAI-compatible behavior requested here.

## Best Possible Solution

Keep the issue open. Maintainers should choose the public config contract (`dimensions`, `outputDimensionality`, or an alias pair), then update schema/types/help/docs, default and per-agent merge logic, OpenAI-compatible inline and batch request builders, provider cache-key data, and reindex identity. Add focused tests for config acceptance, resolution, inline request body, batch request body, cache-key separation, and reindex behavior.

## Risks / Open Questions

- A naive unconditional `dimensions` pass-through could break OpenAI-compatible providers or models that reject the field; the implementation should be provider/model-capability aware.
- Changing embedding vector dimensions must affect embedding cache identity and trigger or require reindexing so stored vectors of incompatible sizes are not mixed.
