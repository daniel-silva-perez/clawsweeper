## Summary

Keep this issue open. Current main still does not accept the requested `agents.defaults.memorySearch.dimensions` or per-agent `memorySearch.dimensions` key, and the OpenAI memory embedding inline and batch paths still omit a `dimensions` field. The only adjacent implementation is the gateway's OpenAI-compatible `/v1/embeddings` endpoint, which can map a request payload `dimensions` into `outputDimensionality`; that does not solve the requested memory-search config contract or OpenAI provider pass-through.

## Best Possible Solution

Keep this issue open. The best fix is to define the public memory-search dimension contract, update schema/types/help/docs, merge defaults and per-agent overrides, pass the selected value through OpenAI-compatible inline and batch embedding request bodies, include it in provider cache-key/reindex identity, and add focused tests for schema acceptance, config resolution, inline request bodies, batch request bodies, cache separation, and reindex behavior.

## Risks / Open Questions

- A naive unconditional `dimensions` pass-through could regress OpenAI-compatible providers or models that reject that field; the contract should be provider/model-capability aware or explicitly scoped.
- Changing vector dimensions must update embedding cache identity and reindex identity so old vectors with a different length are not reused.
- Maintainers still need to choose whether the public config key is `dimensions`, `outputDimensionality`, or an alias pair, then document the compatibility behavior.
