## Summary

Keep this issue open. Current main still does not persist or replay followup queue items across a process restart, the restart sentinel remains a single restart-notice/continuation payload rather than a pending inbound queue, and the drain window still rejects new command-lane work with `GatewayDrainingError`. The provided thread context also shows open, unmerged implementation PRs with closing syntax for this issue, so the issue should remain open until those candidates are reviewed, merged, or closed.

## Best Possible Solution

Keep this issue open. Maintainers should review and either land or close one of the phase-1 PRs, then decide whether drain-window arrivals and provider downtime catch-up stay here or are split into #55792. The best implementation path is a durable, typed pending-inbound/followup handoff owned by gateway/auto-reply, with channel-specific ACK or history-catch-up hooks where provider contracts support them.
