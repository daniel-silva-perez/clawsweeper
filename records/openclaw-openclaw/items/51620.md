## Summary

Keep this issue open. Current main still keeps followup queues in process memory, does not have a persisted pending inbound replay file/path, and the drain path still rejects new gateway lane enqueues with `GatewayDrainingError` rather than durably handing them off. There are also open implementation PRs using closing syntax for this issue, so the issue should remain open until those PRs are reviewed, merged, or closed.

## Best Possible Solution

Keep this issue open. Maintainers should review and either land or close the linked phase-1 PRs, then decide whether the remaining drain-window and provider catch-up behavior belongs here or should be split against the broader #55792 tracker. The likely code path is a durable, typed pending-inbound handoff/replay mechanism owned by gateway/auto-reply with channel-specific ACK or history-catch-up hooks where providers support them.
