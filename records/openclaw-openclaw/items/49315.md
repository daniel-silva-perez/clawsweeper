## Summary

Keep this PR open. Current main still does not support `--profile-id` on `openclaw models auth login`, the related feature request #40402 remains open, and the PR's generic post-login profile rewrite still has a real provider side-effect risk for providers that persist credentials during their own auth flow. PR #68477 is closed unmerged, so it does not supersede this work on main.

## Best Possible Solution

Keep this PR open for maintainer review and revision. The safer implementation is to add the `--profile-id` CLI surface, pass the requested profile target through the provider auth context/options, make OpenAI Codex consume it when constructing its OAuth result, avoid generic post-login profile rewrites, update docs, and add focused tests that cover both the CLI plumbing and provider-owned profile selection. Link the final implementation to #40402.

## Risks / Open Questions

- Merging the PR as written could duplicate or misprioritize credentials for providers that write auth profiles/config before returning a `ProviderAuthResult`, as the GitHub Copilot provider currently does.
- A generic post-login profile-id rewrite does not let provider-owned login code choose the target account/profile during the OAuth/device flow; providers that support named profiles need an explicit provider-facing option.
- Current main has no shipped implementation for the requested CLI/docs behavior, so closing this PR as implemented would strand the open feature request #40402.
