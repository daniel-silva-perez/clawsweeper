## Summary

Keep this PR open. Current main still does not implement the intended `--profile-id` surface for `openclaw models auth login`, and the related feature request #40402 remains open. The PR is relevant, but its generic post-login profile rewrite still needs maintainer revision because provider-owned auth flows should receive the requested profile before they persist credentials.

## Best Possible Solution

Keep this PR open for revision or maintainer review. The best implementation is to add the CLI flag, pass the requested profile target through the provider auth context/options, make OpenAI Codex honor it while preserving identity-derived fallback ids, avoid generic post-login profile rewrites, update the public docs, and add focused tests for CLI plumbing plus provider-owned profile selection. Link the final implementation to #40402.

## Risks / Open Questions

- Merging the PR as written could create duplicate auth profiles for providers that persist credentials during their own auth flow before returning a `ProviderAuthResult`.
- A post-result rename does not let OpenAI Codex or other provider-owned OAuth/device-code code choose the target profile during login; the safer contract is to pass the requested profile through provider auth context/options and have supported providers consume it intentionally.
