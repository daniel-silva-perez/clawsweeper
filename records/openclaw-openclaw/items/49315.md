## Summary

Keep this PR open. Current main still does not implement or document a named-profile option for `openclaw models auth login`, and the related tracker #40402 remains open. The PR is a real implementation candidate, but the proposed generic post-login profile rewrite needs maintainer revision because provider-owned auth flows can persist credentials before returning their `ProviderAuthResult`. Security review found no CI, dependency, workflow, or package-resolution changes in the provided PR diff; the main risk is credential/profile misrouting in auth persistence.

## Best Possible Solution

Keep this PR open for maintainer revision or replacement. The best implementation is to add the login flag, pass the requested profile target through provider auth context/options, validate that the requested id belongs to the selected provider, let supported providers such as OpenAI Codex honor it before persistence, avoid generic post-login profile rewrites, define collision and multi-profile behavior, and update public docs plus CLI/provider regression tests. Link the final implementation to #40402.
