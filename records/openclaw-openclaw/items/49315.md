## Summary

Keep this PR open. Current main still does not implement the intended `--profile-id` surface for `openclaw models auth login`, and the open tracker #40402 still covers the same operator-chosen OAuth profile alias feature. The PR is relevant, but the proposed generic post-login profile rewrite remains risky for provider-owned auth flows that persist credentials during login.

## Best Possible Solution

Keep this PR open for maintainer revision or review. The best implementation is to add the login flag, pass the requested profile target through provider auth context/options, validate that the requested id belongs to the selected provider, let supported providers such as OpenAI Codex honor it before persistence, avoid generic post-login profile rewrites, define collision and multi-profile behavior, and update public docs plus CLI/provider regression tests. Link the final implementation to #40402.

## Risks / Open Questions

- Merging the PR as written could duplicate or misprioritize credentials for providers that persist their own auth profile before returning `ProviderAuthResult`, with GitHub Copilot as a concrete current-main example.
- The proposed helper-level test covers post-result remapping but does not prove CLI parser plumbing, provider-owned profile selection, prefix validation, overwrite/collision handling, or OpenAI Codex behavior.
- Closing this PR now would drop an active implementation candidate for #40402 while the requested operator-chosen OAuth profile alias remains absent from current main.
