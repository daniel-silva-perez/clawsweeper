## Summary

Keep this PR open. Current main still does not implement the PRâ€™s intended `--profile-id` surface for `openclaw models auth login`; the flag exists only for `paste-token`, the login command does not pass a requested profile through to provider auth, and the OpenAI Codex login paths still derive profile names from OAuth identity metadata. The related feature request #40402 is still open, and the PR also needs revision because its generic post-login profile rewrite can conflict with provider-owned auth flows that persist credentials during login.

## Best Possible Solution

Keep this PR open for maintainer review and revision. The safer path is to add the CLI flag, pass the requested profile target through the provider auth context/options, make OpenAI Codex consume it while constructing its OAuth/device-code auth result, avoid generic post-result profile rewrites, update the public docs, and add focused tests for CLI plumbing plus provider-owned profile selection. Link the final implementation to #40402.

## Risks / Open Questions

- Merging this PR as written could duplicate or misprioritize credentials for providers that persist profiles during their own auth flow before returning a `ProviderAuthResult`.
- A generic post-login profile rewrite does not let provider-owned OAuth/device-code code choose the target account/profile during login; providers need an explicit provider-facing option.
- Current main now has stable OpenAI Codex identity-derived fallback ids, so any revision should rebase carefully and preserve that behavior while adding explicit named-profile targeting.
