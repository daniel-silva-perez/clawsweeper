## Summary

Keep open. Current main still rejects Exa `webSearch.baseUrl`, the Exa runtime still uses the fixed Exa endpoint, and open PR #54939 already claims the issue for implementation review.

## What This Changes

Request support for `plugins.entries.exa.config.webSearch.baseUrl`, or explicit schema/docs guidance if Exa custom endpoints remain unsupported.

## Best Possible Solution

Land a reviewed Exa-plugin implementation that adds `webSearch.baseUrl` to manifest/UI metadata, resolves and validates the endpoint in the runtime, sends requests through the trusted web-search path, includes endpoint isolation in cache keys, and updates Exa/web-search docs and tests. If maintainers reject custom Exa endpoints, document that provider capability gap and improve the validation guidance instead.
