## Summary

Keep open. Current main still lacks Exa `webSearch.baseUrl` support and does not provide provider-specific unsupported-field guidance; the issue is also paired with open, unmerged PR #54939, which explicitly targets this work and should be reviewed or replaced before the issue is closed.

## What This Changes

The issue requests support for `plugins.entries.exa.config.webSearch.baseUrl`, or clearer Exa-specific schema/docs guidance when that field is unsupported.

## Best Possible Solution

Land a reviewed Exa-plugin implementation that adds `webSearch.baseUrl` to the manifest/UI hints, resolves and validates it in the Exa runtime, uses it through the trusted web-search endpoint path, includes the resolved endpoint in cache keys, and updates Exa/web-search docs and tests. If maintainers choose not to support custom Exa endpoints, document that capability difference and improve the validation guidance instead.
