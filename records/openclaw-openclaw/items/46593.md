## Summary

Keep open. Current main still auto-selects Gemini CLI for audio after provider/local audio backends fail, and the docs still list Gemini CLI in the default audio auto-detect order. The PR’s narrow defensive behavior is not implemented on main.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best path is to land an equivalent `capability === "audio"` guard in `resolveGeminiCliEntry`, preserve Gemini CLI auto-detect for image/video, add the regression test proving provider audio wins when `gemini` is on PATH, and update the audio docs to match.
