## Summary

Keep open. Current main still has the reported operator gap: `channels.start` exists as an admin Gateway RPC and the channel manager has internal start/stop primitives, but the `openclaw channels` CLI exposes only list/status/capabilities/resolve/logs/add/remove/login/logout and there is no typed `channels.stop` or `channels.restart` RPC surface.

## What This Changes

The issue requests non-destructive CLI lifecycle controls to start, stop, or restart an existing linked channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Add additive, admin-scoped lifecycle controls: keep the existing `channels.start`, add typed `channels.stop` and `channels.restart` Gateway methods backed by the channel manager, expose matching `openclaw channels start|stop|restart --channel <name> [--account <id>]` commands, and document this as non-destructive recovery for linked accounts while leaving health-monitor retry policy separate.

## Reproduction Assessment

Yes for the requested CLI/API gap: current main can be checked by source inspection and shows no `openclaw channels start`, `stop`, or `restart`, with only `channels.start` and `channels.logout` registered as RPCs. The transient network wedge itself has reporter logs but was not live-reproduced in this read-only review.

## Solution Assessment

Yes. Exposing generic admin lifecycle RPC/CLI controls through the existing channel manager is the narrowest maintainable fix for operator recovery; WhatsApp-specific recovery logic or health-monitor backoff redesign would be a broader follow-up.
