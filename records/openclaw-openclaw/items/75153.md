## Summary

Keep open. Current main still lacks the requested operator-facing channel lifecycle CLI, and the gateway protocol only exposes `channels.start` plus logout/status rather than stop or restart. The report is current, source-backed, and narrow enough for a focused admin-scoped recovery PR; the broader health-monitor retry-budget behavior should stay separate.

## What This Changes

Expose non-destructive `openclaw channels start`, `stop`, and `restart` commands so linked channel accounts can recover from a stopped runtime without QR re-pairing or container restart.

## Best Possible Solution

Add additive, admin-scoped generic channel lifecycle controls: a CLI wrapper for existing `channels.start`, new typed `channels.stop` and `channels.restart` gateway methods backed by the existing channel manager, focused tests, and docs/runbook updates that present this as non-destructive recovery for linked accounts. Leave automatic backoff/notification changes for a separate issue.

Yes for the requested CLI/API gap: current main can be verified by reading the channel CLI registration, gateway method registry/schema, and docs. The transient DNS/Baileys stopped-state was not live-reproduced, but the missing recovery command is source-reproducible without that environment.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix for the operator problem: expose generic admin lifecycle seams rather than adding WhatsApp-specific recovery logic. The health-monitor give-up/backoff policy should be handled separately so this change stays focused.
