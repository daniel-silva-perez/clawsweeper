## Summary

Keep open. Current main still has a gateway `channels.start` method and internal start/stop manager hooks, but the `openclaw channels` CLI exposes only status/list/capabilities/resolve/logs/add/remove/login/logout, and there is no `channels.stop` or `channels.restart` gateway method. The linked restart-hang report is separate context, not a duplicate or an implementation of this operator recovery path.

## What This Changes

The issue requests non-destructive CLI channel lifecycle commands so an already-linked but stopped channel can be started, stopped, or restarted without QR re-pairing or restarting the container.

## Best Possible Solution

Add additive, admin-scoped generic channel lifecycle controls: standalone `openclaw channels start/stop/restart --channel <name> [--account <id>]`, typed `channels.stop` and `channels.restart` gateway methods backed by the existing channel manager, focused tests, CLI/protocol/health docs, and a user-facing changelog entry. Keep the longer-term health-monitor backoff/notification behavior as a separate follow-up.

## Reproduction Assessment

Yes for the CLI/API gap: current-main source and docs show no standalone lifecycle CLI commands and no stop/restart RPCs. The exact transient network wedge was not live-reproduced, but the missing recovery surface is reproducible from source inspection.

## Solution Assessment

Yes. An additive admin-scoped CLI/RPC wrapper around existing generic channel lifecycle hooks is the narrowest maintainable fix for the operator problem; WhatsApp-specific recovery logic or health-monitor retry-policy changes should stay out of this focused repair.

## Repair Work Prompt

Implement non-destructive channel lifecycle controls for this issue. Add `openclaw channels start --channel <name> [--account <id>]`, `openclaw channels stop --channel <name> [--account <id>]`, and `openclaw channels restart --channel <name> [--account <id>]` for existing configured/linked accounts. Keep the implementation channel-generic: no bundled channel ids or WhatsApp-specific recovery policy in core.

Expected boundary: add typed gateway protocol params/results for `channels.stop` and `channels.restart`; register those methods in the gateway method list and admin scope; implement handlers in `src/gateway/server-methods/channels.ts` using the existing `context.startChannel` and `context.stopChannel` hooks; add CLI command wrappers under the existing channels CLI/commands pattern; update CLI/protocol/health docs and CHANGELOG with credit to @Jason-Vaughan. `restart` should stop then start the resolved account. Choose conservative, documented behavior for already-running start and already-stopped stop, and keep stop-timeout behavior consistent with `stopChannel` rather than adding new teardown policy.

Do not address the separate container restart hang or the broader health-monitor backoff/notification proposal in this PR.
