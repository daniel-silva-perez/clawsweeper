## Summary

Keep this issue open. Current main still has a Telegram sendChatAction circuit breaker that is explicitly scoped to 401/unauthorized failures, with a regression test asserting non-401 failures do not suspend the handler. There is also an open fork PR that references this issue with closing syntax, so maintainer cleanup should leave the issue open while that candidate is reviewed, ported, or superseded.

## Best Possible Solution

Keep this issue open. Maintainers should review, port, or explicitly supersede the open fork PR, then add a Telegram-owned regression test for repeated non-401 sendChatAction failures. The fix should treat typing indicators as best-effort, cap or suspend network/429/529 sendChatAction retries with backoff, and avoid resetting the relevant failure breaker while an existing typing keepalive cycle is already active or tripped.
