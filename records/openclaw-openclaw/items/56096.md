## Summary

Keep this issue open. Current main still bounds Telegram sendChatAction backoff/suspension only for 401 Unauthorized errors, while the reported network/529-style sendChatAction failures remain outside the circuit breaker. The shared typing callback also resets its start guard on each reply-start cycle, so a higher-level typing keepalive can keep rearming the failing sendChatAction path. There is also an open fork PR referencing this issue with closing syntax, so the issue should remain open while that candidate is reviewed, ported, or superseded.

## Best Possible Solution

Keep this issue open. Review or port the candidate fix into the Telegram plugin: classify non-critical sendChatAction failures such as network/429/529 separately from 401 auth failures, cap or suspend retries with backoff, avoid resetting the typing failure breaker while an existing keepalive cycle is active or already tripped, and add a regression test covering repeated higher-level typing starts after non-401 sendChatAction failures.
