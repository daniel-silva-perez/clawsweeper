## Summary

Current main still appears to match the reported authorization gap: session-key-based Gateway RPCs and session event subscriptions are enforced by operator method scope, but the target session is resolved by key without a caller-device/client-to-session authorization check. I did not find a canonical superseding item or an implementation on main that closes the object-level session access issue.

## Best Possible Solution

Keep this issue open and implement a Gateway session-authorization seam. Define what makes a caller authorized for a session, persist or derive the needed owner/share metadata, filter `sessions.list`, gate `sessions.get`/preview/compaction reads, validate `sessions.subscribe` and `sessions.messages.subscribe`, and enforce the same check before `chat.send` and session mutations with a deliberate `operator.admin` or explicit-share bypass. Add regression tests with two paired operator devices/scopes proving cross-session list/read/subscribe/send/mutate denial and documented admin behavior.

## Risks / Open Questions

- This is an authorization and privacy risk for deployments with multiple paired operator devices or differentiated operator scopes: a read-scoped client can still learn session keys and read/subscribe to unrelated session content if it can obtain a key.
- A fix needs careful product/API design because current Control UI and multi-device workflows may intentionally share some sessions; admin or explicitly shared sessions likely need a documented bypass while ordinary read/write clients get object-level filtering.
- Session event delivery, list filtering, transcript reads, chat sends, and session mutations need to be updated together; patching only one path would leave an IDOR-style bypass through another key-based method.
