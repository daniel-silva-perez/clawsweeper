## Summary

Keep open. Current main has partial protection: raw `Agent failed before reply` boilerplate is suppressed in group/channel contexts, and Slack filters messages from the same bot account. But the reported OAuth/auth cascade is still plausible because current main still emits specific gateway auth guidance to non-direct chats, and Slack with `allowBots: true` still accepts messages from other bot accounts. The related #37842 remains a different sessions_send loop class, not a canonical replacement for this Slack gateway-error path.

## Best Possible Solution

Add a dedicated guard for gateway-generated error replies in non-direct bot-visible channels: either suppress/tag them so Slack inbound ignores them across sibling OpenClaw bot accounts, or add per-channel/per-agent error dedupe and auth-failure cooldowns. Cover the OAuth refresh failure path specifically with a Slack channel test using `allowBots: true` and two distinct bot identities.
