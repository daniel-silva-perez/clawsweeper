## Summary

Current main still only partially protects this path: generic runner boilerplate is silenced in non-direct chats, but OAuth refresh failures still produce visible gateway auth guidance, and Slack with allowBots enabled still accepts messages from other bot accounts when routing/mention policy lets them through.

## Best Possible Solution

Add a dedicated guard for gateway-generated error replies in non-direct bot-visible Slack channels. The most robust path is to tag or classify OpenClaw gateway error replies so sibling OpenClaw bot accounts ignore them, then add per-channel/per-agent repeated-error dedupe and an auth-failure cooldown as defense in depth. Cover the OAuth refresh failure path with a Slack test using allowBots=true and two distinct bot identities.
