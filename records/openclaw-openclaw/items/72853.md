## Summary

Keep this PR open. Current main still has the id-less cron-job state collision, and the PR is a narrow implementation candidate with no blocking review findings from this read-only sweep; it needs maintainer review and CI validation rather than cleanup closure.

## What This Changes

This PR backfills and persists stable IDs for loaded cron jobs missing both `id` and legacy `jobId`, guards outcome application for invalid job IDs, and updates cron tests, docs, and the changelog.

## Best Possible Solution

Land a focused cron fix after maintainer review and CI, keeping the load-time stable-ID backfill, the invalid-outcome guard, and the operator-facing docs/changelog note together.

## Reproduction Assessment

Yes. #72849 gives a concrete repro with multiple persisted cron jobs missing `id` and `jobId`, and current main source confirms the loaded rows can remain id-less while outcome application compares `undefined` IDs.

## Solution Assessment

Yes. The proposed load-time backfill is the narrowest maintainable fix because it makes hand-authored persisted jobs satisfy the existing `CronJob.id` contract, while the timer guard prevents any remaining invalid outcome from mutating the wrong row.
