## Summary

Keep this PR open. Current main still has the id-less cron outcome collision described by the linked bug, and the PR is a focused implementation candidate with no blocking correctness finding from static review, existing bot review context, or public exact-head check-run inspection.

## What This Changes

The PR backfills and persists stable IDs for loaded cron jobs missing both `id` and legacy `jobId`, guards outcome application against invalid job IDs, and adds cron tests, docs, and a changelog entry.

## Best Possible Solution

Land this focused cron fix after maintainer review and exact-head CI validation, keeping the load-time ID backfill in the cron store seam, the defensive outcome guard in the timer path, and the operator-facing docs/changelog note.

## Reproduction Assessment

Yes. The linked bug gives a high-confidence path: create multiple persisted cron jobs without `id` or `jobId`, restart the gateway, and let any cron run produce an outcome; current main can then compare `undefined` to `undefined` in the outcome lookup.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable approach I found: make persisted hand-authored rows satisfy the existing `CronJob.id` contract at load time, then reject any remaining invalid outcome before it can mutate the wrong row.
