## Summary

Current main still has the reported Telegram null-media path: getFile() failures can return null, the single-message warning only runs for thrown media errors, and the inbound body builder can still emit a media placeholder when no attachment was resolved. The issue also has open implementation PRs (#41048 and #42059), so it should remain open until maintainers land, split, or close that PR path.

## Best Possible Solution

Keep this issue open and resolve it through the focused Telegram plugin fix path, preferably by reviewing and landing #41048 or extracting the equivalent minimal behavior from #42059. The fix should make getFile null/failure distinct from 'no media', send a user-visible warning for failed media resolution, avoid misleading placeholder-only bodies when no attachment exists, preserve caption text, and update the Telegram resolver/body tests.

## Risks / Open Questions

- A naive fix that throws for every null media result could drop useful captions or text; the implementation should distinguish failed media resolution from messages that genuinely have no downloadable media.
- Repeated user-visible warnings for attacker-triggered getFile failures could spam a chat unless the final patch keeps the warning bounded and accurate.
- The broader #42059 batch touches several Telegram behaviors, so maintainers may prefer a narrow split of the #40991 fix rather than landing the whole batch.
