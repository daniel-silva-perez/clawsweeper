## Summary

Keep open. Current main still has the Feishu HTTP server cleanup race: both the targeted stop path and webhook abort path call `server.close()` and immediately remove state references, and the latest release tag contains the same behavior. The issue is also paired with open PR #48588, which explicitly targets this cleanup bug, so it should remain open until that PR or an equivalent replacement lands.

## What This Changes

The issue requests that Feishu monitor shutdown await HTTP server close and clean up state only after server resources are actually released.

## Best Possible Solution

Land or replace the existing Feishu-specific fix so HTTP server teardown is awaitable, active or idle connections are drained where appropriate, cleanup has timeout/error handling, and focused regression coverage checks targeted, global, and abort cleanup. Keep the cross-plugin ResourceManager discussion in #48793/#49804 as separate architecture work.

## Reproduction Assessment

Yes. The code-level reproduction path is high confidence: start Feishu webhook monitoring, trigger account stop or abort/reload, and current main calls `server.close()` before immediately deleting or clearing the state maps. Live Feishu credentials would verify the user-visible port or memory symptom, but the race is visible without them.

## Solution Assessment

Yes. The best immediate solution is a narrow Feishu cleanup fix like PR #48588 or an equivalent maintainer-owned replacement; the broader ResourceManager proposal is useful follow-up but should not block the shipped plugin bug fix.
