## Summary

Keep open. Current main still has the Feishu HTTP server cleanup race in both the explicit stop path and the webhook abort cleanup path, and the latest release tag contains the same behavior. The issue is also paired with open PR #48588, so the issue should remain open until that PR or an equivalent maintainer replacement lands.

## What This Changes

The issue asks for Feishu monitor HTTP server shutdown to wait for server close or connection drain before clearing `httpServers` and related monitor state.

## Best Possible Solution

Land or replace the Feishu-specific fix so HTTP server teardown is awaitable, active connections are drained or bounded by timeout where appropriate, and regression coverage checks targeted stop, global stop, and webhook abort cleanup. Keep the broader ResourceManager discussion in #48793/#49804 as separate architecture work.

## Reproduction Assessment

Yes. The code-level reproduction path is high confidence: start Feishu webhook monitoring, trigger account stop or abort/reload, and current main calls `server.close()` before immediately deleting or clearing state maps. Live Feishu credentials would verify the user-visible port or memory symptom, but the race is visible without them.

## Solution Assessment

Yes. The best immediate solution is a narrow Feishu plugin cleanup fix like PR #48588 or an equivalent maintainer-owned replacement; a cross-plugin ResourceManager may be useful follow-up but should not block the plugin bug fix.
