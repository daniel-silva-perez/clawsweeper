## Summary

Keep open. Current main and the latest release still have the reported Feishu HTTP server cleanup race, and the issue is paired with open PR #48588, which explicitly targets this bug. The broader ResourceManager discussion in #48793/#49804 is useful follow-up, but it does not replace the narrow Feishu cleanup fix needed here.

## What This Changes

The issue asks for Feishu monitor HTTP server shutdown to await or force-drain server close before clearing `httpServers` and bot state, avoiding restart leaks and `EADDRINUSE` failures.

## Best Possible Solution

Land or replace the existing Feishu-specific fix so HTTP server teardown is awaitable, force-drains active connections where appropriate, has timeout/error handling, and includes focused regression coverage for targeted and global cleanup. Keep the cross-plugin ResourceManager work in #48793/#49804 as separate architecture follow-up.

## Reproduction Assessment

Yes. The issue has a high-confidence code-level reproduction path: Feishu webhook startup stores servers in `httpServers`, and both stop/abort paths currently call `server.close()` before immediately deleting or clearing the state maps. A live Feishu restart loop would verify the user-visible leak or port-binding symptom, but the cleanup ordering itself is directly visible on current main.

## Solution Assessment

Yes. The best immediate solution is the narrow Feishu fix already represented by PR #48588 or an equivalent maintainer-owned replacement; the broader ResourceManager proposal should not block fixing this shipped plugin cleanup bug.
