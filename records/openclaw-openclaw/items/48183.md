## Summary

Keep open. Current main still closes Feishu HTTP servers and immediately deletes or clears monitor state without awaiting the close callback, the latest release tag contains the same behavior, and open PR #48588 still references this issue with closing syntax.

## What This Changes

The issue asks for Feishu monitor HTTP server cleanup to wait for server shutdown before clearing `httpServers` and related state so repeated restarts do not leak resources or hit port binding races.

## Best Possible Solution

Land or replace #48588 with a narrow Feishu plugin fix that makes HTTP server teardown awaitable, preserves the existing WebSocket cleanup behavior, drains or bounds active HTTP connections, and adds regression coverage for targeted stop, global stop, and webhook abort cleanup. Keep the broader ResourceManager ideas in #48793 and #49804 as separate architecture work.

## Reproduction Assessment

Yes. The code-level reproduction path is high confidence: start Feishu webhook monitoring, trigger account stop or abort/reload, and current main calls `server.close()` before immediately deleting or clearing state maps; live Feishu credentials would only be needed to measure the external port or memory symptom.

## Solution Assessment

Yes. A narrow Feishu-specific async HTTP cleanup fix is the best immediate solution; a cross-plugin ResourceManager may be useful follow-up, but it should not block fixing this plugin bug.
