## Summary

Current main still does not implement the requested prompt/heartbeat race UX. The Control UI queues prompts only when it already owns a local chat run, while backend follow-up queuing only preserves execution order. A heartbeat-active run is not surfaced as the UI busy state before submit, and heartbeat deltas from another run are ignored once the prompt owns chatRunId. Related #49252/#49261 addressed the optimistic-message part, but does not supersede this remaining heartbeat-active queue/visibility path.

## Best Possible Solution

Keep this open and implement an explicit session-scoped active-run and queued-follow-up surface shared by Gateway chat events and Control UI. When a heartbeat is active and a prompt arrives, the UI should keep rendering heartbeat/tool progress, show the submitted prompt as queued or pending with cancel/steer/inject controls, and drain it after the heartbeat reaches a safe boundary. Add focused tests for app-chat, app-gateway, controllers/chat, server-chat/chat.send, and the reply-runner follow-up path.
