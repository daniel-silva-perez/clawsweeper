## Summary

Current main has backend follow-up queuing for non-heartbeat turns when another run is active, but it does not implement the issue's requested prompt/heartbeat conflict UX. The Control UI only treats its own `chatRunId`/`chatSending` as busy, so an already-running heartbeat does not make a newly submitted prompt enter the visible queue. While that prompt run is active, chat events from the heartbeat run are ignored except for final events, so the heartbeat still is not shown immediately in the race case.

## Best Possible Solution

Keep this issue open and implement an explicit session-scoped active-run/queued-followup surface for Control UI and Gateway chat events. When a heartbeat is active and a prompt arrives, the UI should keep rendering heartbeat/tool progress, mark the submitted prompt as queued/pending with cancel or steer/inject controls, and drain it after the heartbeat reaches the next safe boundary. Add targeted tests around `app-chat`, `app-gateway`, `controllers/chat`, and the Gateway `chat.send` plus reply-runner path.
