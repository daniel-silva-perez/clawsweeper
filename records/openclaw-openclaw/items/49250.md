## Summary

Current main still does not implement the requested prompt/heartbeat race UX. The UI only queues when it already owns a local chat run, backend follow-up queuing is execution-order only, and heartbeat/other-run deltas can still be ignored while a prompt-owned run is active. Related #49252 and closed-unmerged PR #49261 do not cover the remaining visible queued-prompt/cancel-inject behavior.

## Best Possible Solution

Keep this issue open and implement a session-scoped active-run plus queued-follow-up surface shared by Gateway chat events and Control UI. When a heartbeat is active and the user submits a prompt, the UI should keep rendering heartbeat/tool progress, show the prompt as queued or pending with cancel/steer/inject controls, and drain it after the heartbeat reaches a safe boundary. Add focused tests around app-chat, app-gateway, controllers/chat, server-methods/chat.send, and the reply-runner follow-up path.
