## Summary

Keep this PR open for maintainer review. Current main has the newer Gateway wizard session RPC that covers GUI-driven setup execution, but it still does not expose the static `configure surface --json-out` discovery/export surface proposed here; the PR is therefore not already implemented or clearly superseded. The branch also introduces concrete correctness and security-sensitive concerns around raw argv parsing and active plugin loading, so it should not be merged without revision.

## What This Changes

The branch adds a `configure surface` CLI subcommand, setup-surface descriptor/export types, provider auth surface metadata, command descriptor updates, generated Plugin SDK baseline changes, and targeted tests.

## Best Possible Solution

Settle whether GUI setup discovery should be a CLI file export, Gateway RPC, or both; if the static export remains desired, implement it as a documented, versioned, non-activating setup-surface contract that uses manifest/setup-entry metadata where possible and has schema, CLI, docs, and plugin-boundary tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep surface export on a non-activating plugin load:** `src/setup/surface-export.ts:297`
  - body: `configure surface` is advertised as a read-only export, but this call loads plugins with `activate: true`. That can clear/restore global plugin state, run activation-only registrations, and stage bundled runtime dependencies while an external UI is only asking for metadata. Use the existing snapshot/setup-entry path instead of a full active load for this export.
  - confidence: 0.84
- **[P2] Use parsed options for surface sections:** `src/cli/program/register.configure.ts:95`
  - body: The action ignores Commander’s parsed `opts.section` and re-parses `program.rawArgs`. A valid invocation with an earlier argument value named `configure` can anchor the scan before the real command and drop `--section channels`, causing the machine-consumed JSON to include the default provider and channel sections instead of the requested subset.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds a read-only CLI export but builds it through active plugin loading, which broadens code-execution and bundled runtime dependency staging behavior for a metadata command.

Concerns:

- **[medium] Read-only export activates plugin code:** `src/setup/surface-export.ts:297`
  - body: `exportSetupSurface` passes `activate: true` to `loadOpenClawPlugins`. In current loader semantics, active loads can run activation-only plugin registrations and bundled runtime dependency staging, so a metadata export can trigger code execution and dependency side effects beyond the stated read-only purpose.
  - confidence: 0.84
