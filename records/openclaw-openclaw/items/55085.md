## Summary

Keep open. Current main still lacks the proposed static `configure surface` CLI export, and the related #55557 wizard protocol solves interactive setup execution rather than provider/channel discovery. The PR is useful directionally but is not ready to merge because the export path activates plugins for a read-only metadata command and still has concrete machine-output correctness bugs.

## What This Changes

The branch adds a `configure surface` CLI subcommand, setup-surface descriptor/export types, provider auth surface metadata, command descriptor updates, generated Plugin SDK baseline updates, and targeted tests.

## Best Possible Solution

Settle whether GUI setup discovery should be a CLI file export, Gateway RPC, or both; if a static export remains desired, implement it as a documented, versioned, non-activating setup-surface contract that uses manifest/setup-entry metadata where possible and has schema, CLI, docs, and plugin-boundary tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep surface export on a non-activating plugin load:** `src/setup/surface-export.ts:297`
  - body: `configure surface` is advertised as a read-only metadata export, but this call uses `activate: true`. Current loader semantics can clear/restore global plugin state, run activation-only registrations, and stage bundled runtime deps, so an external UI asking for metadata can trigger plugin code and dependency side effects. Build the export from manifest/setup-entry metadata or a non-activating load.
  - confidence: 0.84
- **[P2] Use parsed options for surface sections:** `src/cli/program/register.configure.ts:95`
  - body: The action ignores Commander’s parsed `opts.section` and reparses `program.rawArgs`. A valid invocation with an earlier argument value named `configure` can anchor the scan before the real command and drop `--section channels`, causing the JSON export to include the default provider and channel sections instead of the requested subset.
  - confidence: 0.78
- **[P2] Preserve disabled state for wizard-backed channels:** `src/setup/surface-export.ts:121-125`
  - body: This path passes `adapter.getStatus(...)` through without considering `entry.enabled`. The shared adapter status contract does not receive an enabled flag, so disabled installed channels with setup wizards can look active/configured in the exported JSON. Wrap the status and selection hint with the disabled state before returning the descriptor.
  - confidence: 0.74
- **[P2] Preserve installed state for catalog-only channels:** `src/setup/surface-export.ts:267-268`
  - body: Catalog entries include discovered plugin manifests as well as fallback catalogs, but this branch hardcodes `installed: false` for every catalog-only entry. If an installed plugin is absent from `registry.channelSetups` after a load or config failure, the export marks it installable and can send UIs toward reinstall instead of repair.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The diff adds a read-only CLI export but builds it through active plugin loading, which broadens code-execution and bundled runtime dependency staging behavior for a metadata command.

Concerns:

- **[medium] Read-only export activates plugin code:** `src/setup/surface-export.ts:297`
  - body: `exportSetupSurface` passes `activate: true` to `loadOpenClawPlugins`. In current loader semantics, active loads can run activation-only plugin registrations and bundled runtime dependency staging, so a metadata export can trigger code execution and dependency side effects beyond the stated read-only purpose.
