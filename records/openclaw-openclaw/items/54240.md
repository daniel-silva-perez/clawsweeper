## Summary

Current main still does not guarantee that a Control UI model dropdown change is persisted before the next chat.send. The picker awaits sessions.patch only inside its own change handler, but no shared pending model-switch state reaches handleSendChat or the send controls, and the Gateway can process sessions.patch and chat.send concurrently.

## Best Possible Solution

Keep this issue open. Implement a shared Control UI model-switch barrier: store the pending sessions.patch for the active session, disable the model picker and send/queue controls while it is pending, have handleSendChat and queued-send flushing await it before chat.send, roll back the visible selection and suppress send if the patch fails, and add a regression test with an unresolved sessions.patch followed by an immediate send attempt.

## Risks / Open Questions

- Users can still visually select a model and immediately send while the session patch is unresolved, allowing chat.send to start from the prior session model depending on request scheduling.
- Backend live model-switch handling may recover some active-run switches, but it does not provide the requested UI-level guarantee that the next send waits for the selected model to persist before the run begins.
