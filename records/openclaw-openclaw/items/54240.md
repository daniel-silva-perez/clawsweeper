## Summary

Keep open. Current main still has the Control UI ordering gap: the model picker fires an async `sessions.patch` and updates local selection optimistically, but the send path and send buttons do not share any pending model-switch state to await or disable on, and the `chat.send` payload cannot carry a one-turn model override.

## What This Changes

The issue requests that Control UI persist or await a model dropdown session override before the next chat send so an immediate send uses the selected model.

## Best Possible Solution

Add a Control UI-side per-session model-switch barrier: store the pending `sessions.patch` operation/status in shared UI state, disable the active model picker and send/queue controls while it is pending, await it before dispatching chat sends, suppress the waiting send and roll back the optimistic selection on patch failure, and keep the Gateway `chat.send` protocol unchanged unless maintainers choose a broader API design later.

## Repair Work Prompt

Fix the Control UI model-selection race described by this issue. Implement a narrow UI-side barrier so a model selected from the chat header dropdown is persisted through `sessions.patch` before the next chat send can start. Track a per-session pending model-patch operation/status in shared Control UI state, have the model picker set/clear it around `switchChatModel`, disable the active model picker and send/queue controls while the active session has a pending model switch, and make `handleSendChat` await the pending operation before normal `chat.send` dispatch. If the model patch fails, roll back the optimistic picker state as today, surface the error, and do not send the message that was waiting on that failed switch. Keep the Gateway `chat.send` protocol unchanged and do not add a one-turn model override field. Decide in tests whether `/btw`, queued follow-ups, compact/new/reset sends, and steer sends share the same barrier. Add regression coverage that leaves `sessions.patch` unresolved, immediately attempts a send, verifies `chat.send` is not called until the patch resolves, and verifies patch failure suppresses send and restores the picker state. This is UI runtime behavior, so add a changelog fix entry crediting @betamod.
