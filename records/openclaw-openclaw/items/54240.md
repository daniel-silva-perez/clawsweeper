## Summary

Keep open. Current main still has the reported Control UI ordering gap: model selection persists through an async `sessions.patch`, while normal send, queued send, and run controls do not share a pending model-switch barrier before `chat.send` starts.

## What This Changes

Ensure a Control UI model dropdown change is committed before the next chat send starts, or block/delay send until the session patch finishes.

## Best Possible Solution

Add a Control UI-side per-session model-switch barrier: track the pending `sessions.patch`, disable model/send/queue controls while it is pending, await it before dispatching sends, restore the draft and roll back the optimistic selection on patch failure, and keep the Gateway `chat.send` protocol unchanged unless maintainers choose a broader API design later.

## Repair Work Prompt

Fix the Control UI model-switch/send race for this issue. Current main updates `chatModelOverrides` optimistically in `ui/src/ui/chat/session-controls.ts` while awaiting `sessions.patch`, but `handleSendChat` in `ui/src/ui/app-chat.ts` and the send/queue controls do not observe that pending operation. Add a per-session pending model-switch barrier in shared UI state, disable the active model picker and send/queue controls while the patch is pending, have `handleSendChat` await the relevant pending patch before dispatching normal sends, and on patch failure roll back the picker and avoid sending the message while preserving/restoring the draft and attachments. Keep the Gateway `chat.send` protocol unchanged. Decide and cover whether `/btw`, queued follow-ups, `/compact`, `/new`/`/reset`, and queued steer sends should share the same barrier. Add focused regression tests in the Control UI test files. Add a single-line user-facing changelog Fixes entry with `Thanks @betamod`. Do not change provider defaults, model catalog filtering, or protocol schema unless a maintainer explicitly asks for a broader design.
