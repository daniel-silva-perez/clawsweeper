## Summary

Keep open: current main and the latest release still leave the repo-root cache parent behind, so the cleanup is still relevant, but the PR’s added `rmdirSync(tempParent)` can race with another isolated retry and fail that retry with `ENOENT`. This needs author or maintainer revision before merge.

## What This Changes

The PR adds best-effort removal of `.openclaw-config-doc-cache` after isolated config-surface retry cleanup and adds a regression assertion that the cache parent is absent after a successful retry.

## Best Possible Solution

Revise the cleanup so the retry stops leaving repo-root residue without deleting a shared parent that another retry may be about to use. A good end state would preserve repo-root dependency resolution, remove the cache parent only when concurrency-safe or make `mkdtempSync` retry after recreating it, keep the regression assertion, skip a changelog entry as internal tooling hygiene, and validate the targeted config-surface test plus the normal changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid removing the shared retry parent during setup:** `scripts/load-channel-config-surface.ts:177`
  - body: The new `rmdirSync(tempParent)` can delete `.openclaw-config-doc-cache` after another process has run `mkdirSync(tempParent)` but before it calls `mkdtempSync(...)`. In that interleaving the other isolated retry throws `ENOENT`, so concurrent config/doc invocations can fail instead of recovering. Use a per-run parent under the repo root or recreate/retry the parent before `mkdtempSync`.
  - confidence: 0.86
