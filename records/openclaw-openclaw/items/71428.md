## Summary

Keep open. Current main still enforces conversation-content typed-hook access from per-plugin host config only, the native manifest and manifest registry do not model `permissions.conversationAccess`, and an open implementation PR already references this issue as its closing target.

## What This Changes

The issue requests an additive plugin-manifest `permissions.conversationAccess` field, with host-config backward compatibility, so external plugins can declare conversation-hook access needs without patching OpenClaw config schemas.

## Best Possible Solution

Define `permissions.conversationAccess` as an additive manifest contract with explicit trust semantics, preferably request metadata plus documented operator approval and deny precedence. Parse it in `src/plugins/manifest.ts`, preserve it through `PluginManifestRecord`, expose it where status/install surfaces need disclosure, and update hooks/config/manifest docs plus loader and manifest-registry tests.

## Reproduction Assessment

Yes. The reproduction path is code-level and test-backed: a non-bundled plugin registering `llm_input`, `llm_output`, `before_agent_finalize`, or `agent_end` is rejected unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`, and current manifest parsing ignores `permissions.conversationAccess`.

## Solution Assessment

Unclear for the exact issue sketch. A manifest field that grants access by itself is not the safest trust model; a manifest request field with explicit host/operator approval, as the linked open PR describes, is the narrower maintainable direction.
