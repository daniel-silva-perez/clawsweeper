## Summary

Keep open. Current main still has host-config-only conversation hook access, the native manifest contract still lacks `permissions.conversationAccess`, and open PR #72690 is already the linked implementation candidate rather than a reason to close before merge.

## What This Changes

The issue requests an additive native plugin manifest field for conversation-access needs, replacing Smarter-Claw's parallel schema patches while preserving backward compatibility with the existing host config flag.

## Best Possible Solution

Review and land PR #72690, or an equivalent narrow patch, that adds manifest-level conversation-access request metadata, preserves it through manifest registry/status surfaces, keeps explicit operator config as the effective grant or deny, and updates manifest, hooks, config docs, and focused loader/registry tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a non-bundled plugin declaring only `permissions.conversationAccess: true` in `openclaw.plugin.json` and registering `agent_end`; current main ignores that manifest field and still blocks the hook unless `plugins.entries.<id>.hooks.allowConversationAccess=true` is present.

## Solution Assessment

No for the RFC sketch if the manifest value grants access by itself. The safer maintainable solution is a manifest request/disclosure field plus explicit host approval, which is the direction described by the linked implementation PR.

## Security Review

Status: needs_attention

Summary: This issue changes metadata and approval semantics for non-bundled plugin access to raw conversation-content hooks, so maintainer security review is still needed.

- **[medium] Preserve operator approval for transcript access:** `src/plugins/registry.ts:1956`

The implementation should keep the manifest as request/disclosure metadata or otherwise require explicit host approval with clear deny precedence.
