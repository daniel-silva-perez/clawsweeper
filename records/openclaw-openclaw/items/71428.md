## Summary

Keep open. Current main still has only host-config-based effective access for conversation-content typed hooks, the native plugin manifest contract does not include `permissions.conversationAccess`, and open PR #72690 is already the linked implementation candidate for maintainer review.

## What This Changes

Request an additive native plugin manifest field for declaring conversation-access needs, while keeping the existing `plugins.entries.<id>.hooks.allowConversationAccess` config flag for compatibility and precedence.

## Best Possible Solution

Review and land PR #72690 or an equivalent narrow patch that adds manifest-level conversation-access request metadata, preserves it through manifest registry/status surfaces, keeps explicit operator approval as the effective grant, and updates manifest/hooks/config docs plus loader and registry tests.

## Reproduction Assessment

Yes. A code-level reproduction is a non-bundled plugin registering `llm_input`, `llm_output`, `before_agent_finalize`, or `agent_end`: current tests and registry code show those hooks are blocked without host config and accepted only with `plugins.entries.<id>.hooks.allowConversationAccess=true`, while manifest parsing ignores `permissions.conversationAccess`.

## Solution Assessment

Unclear for the RFC sketch as written. A plugin-owned manifest value should not grant raw conversation access by itself; the safer maintainable solution is manifest request/disclosure metadata plus explicit host approval and documented precedence.
