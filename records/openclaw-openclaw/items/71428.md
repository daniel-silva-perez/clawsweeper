## Summary

Keep this issue open. Current main still enforces conversation-content typed-hook access from `plugins.entries.<id>.hooks.allowConversationAccess` only, while the native plugin manifest and manifest registry do not model `permissions.conversationAccess`. The linked open PR uses closing syntax for this issue, so this should stay open until maintainers review and land or close that implementation path.

## What This Changes

The issue requests an additive `permissions.conversationAccess` plugin-manifest field for conversation-content hook access, while preserving backward compatibility with the existing host config flag.

## Best Possible Solution

Define the security semantics first, then add a documented additive manifest contract that is parsed and preserved by the native manifest and registry. The safest endpoint is likely manifest request metadata plus explicit operator approval precedence, with tests for manifest-only, config-only, both-present, explicit deny, default rejection, and invalid values.

## Reproduction Assessment

Yes. The reproduction path is code-level and test-backed: a non-bundled plugin that registers `llm_input`, `llm_output`, `before_agent_finalize`, or `agent_end` is rejected unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`; current manifest parsing cannot alter that outcome.

## Solution Assessment

Unclear. The exact issue proposal would let plugin-owned manifest metadata grant raw conversation access, while the open PR's request-metadata-only approach preserves operator approval; maintainers should choose the narrower trust model before implementation lands.
