## Summary

Keep open. Current main still enforces conversation-content typed hooks from per-plugin host config only, the native plugin manifest/registry path has no `permissions.conversationAccess` metadata, and open PR #72690 is already the paired implementation candidate for maintainer review.

## What This Changes

The issue requests an additive plugin manifest field such as `permissions.conversationAccess` so plugins can declare conversation-hook access needs without carrying host config schema patches, while preserving backward compatibility.

## Best Possible Solution

Review and land PR #72690 or an equivalent narrow patch that adds manifest-level request metadata, preserves it through manifest registry/status surfaces, keeps effective raw conversation hook access behind explicit operator approval, and updates hooks/config/manifest docs plus loader and manifest-registry tests.

## Reproduction Assessment

Yes. A non-bundled plugin registering `llm_input`, `llm_output`, `before_agent_finalize`, or `agent_end` is blocked without host config and allowed with `plugins.entries.<id>.hooks.allowConversationAccess=true`; current manifest parsing and registry records ignore `permissions.conversationAccess`.

## Solution Assessment

Unclear for the exact RFC sketch because a plugin-owned manifest field should not grant transcript access by itself. The safer direction is manifest request/disclosure metadata plus explicit host approval, which is the direction described by the linked open PR.
