## Summary

Keep open. Current main still lacks a native manifest `permissions.conversationAccess` contract and still gates conversation-content typed hooks from per-plugin host config; open PR #72690 is already the linked implementation path and should be reviewed rather than spawning a separate cleanup close.

## What This Changes

The issue requests an additive plugin manifest field for conversation-hook access disclosure so plugin authors can avoid parallel host config schema patches while preserving existing host config compatibility.

## Best Possible Solution

Keep this issue open and use PR #72690 as the active review path. The desired end state is a documented manifest request field that is parsed and preserved through the manifest registry/status surfaces, with explicit host/operator approval or deny precedence still controlling actual raw conversation hook access.

## Reproduction Assessment

Yes. The reproduction path is code-level and test-backed: a non-bundled plugin registering `llm_input`, `llm_output`, `before_agent_finalize`, or `agent_end` is blocked unless host config sets `plugins.entries.<id>.hooks.allowConversationAccess=true`, and current manifest parsing ignores `permissions.conversationAccess`.

## Solution Assessment

Unclear for the issue sketch as written. A manifest field that grants access by itself is not the safest model; the narrower maintainable solution is manifest request metadata plus explicit host approval, which is the direction described by PR #72690.

## Security Review

Status: needs_attention

Summary: The request changes how plugins declare access to raw conversation-content hooks, so maintainer security review is still required.

Any implementation should treat the manifest value as request metadata or define explicit host approval/deny precedence.
