## Summary

Keep this issue open. Current main still does not expose configurable per-session lane concurrency, auto-yield timing, or configurable lane-wait warning defaults. There are adjacent queue-mode mitigations and an open related PR for the session-concurrency slice, but the requested umbrella behavior is not implemented on main.

## Best Possible Solution

Keep this issue open as the umbrella tracker. Review or rework #63864 for the per-session concurrency slice, including config schema/types, generated metadata/docs, startup/reload lane application, and queue/runner coverage. Keep #14747 as the warning-threshold tracker unless maintainers choose to fold it here. Design autoYieldAfterMs separately in the embedded runner/tool loop with explicit cancellation and transcript-ordering semantics.

## Risks / Open Questions

- Increasing per-session concurrency needs explicit design around transcript ordering, session mutation, and write-lock interactions because current docs rely on per-session serialization as a guarantee.
- Auto-yield is separate from the existing explicit sessions_yield tool and needs clear semantics for in-flight tool calls, abort/cancellation, and queued message delivery.
- Closing this umbrella before #63864 merges and before auto-yield is designed would drop still-missing requested behavior.
