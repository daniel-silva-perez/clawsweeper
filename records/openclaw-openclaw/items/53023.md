## Summary

Keep open. Current main still serializes each session lane at one active run, exposes only global and subagent concurrency defaults, keeps the default lane-wait warning at 2 seconds unless a caller overrides it, and implements yielding only through explicit `sessions_yield` calls. The related open PR #63864 covers only the session-concurrency slice, #14747 separately tracks the warning-threshold slice, and auto-yield still needs maintainer design.

## What This Changes

Request configurable per-session lane concurrency, configurable lane-wait warning thresholds, and automatic yielding of long tool-running agent turns so interactive messages do not queue behind a single session lane.

## Best Possible Solution

Keep this as the umbrella tracker: review or rework #63864 for the per-session concurrency slice, decide whether #14747 remains canonical for lane-wait diagnostics, and design auto-yield in the embedded runner/tool loop with explicit ordering and cancellation semantics before implementation.
