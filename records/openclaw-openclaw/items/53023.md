## Summary

Keep open. Current main still serializes each session lane at one active run, exposes only global and subagent concurrency in agent defaults, keeps the default lane wait warning at 2 seconds, and implements yielding only when the model explicitly calls `sessions_yield`. The related open PR #63864 is only a session-concurrency implementation candidate, while #14747 separately tracks the warning-threshold slice and neither covers auto-yield.

## What This Changes

The issue requests configurable per-session lane concurrency, automatic yielding after a configured tool-execution interval, and a configurable lane wait warning threshold.

## Best Possible Solution

Keep this as the umbrella tracker. Review or rework #63864 for the per-session concurrency slice, decide whether #14747 remains the canonical lane-wait-threshold tracker, and design auto-yield in the embedded runner/tool loop with explicit transcript, cancellation, and user-message ordering semantics before implementation.

## Risks / Open Questions

- Per-session concurrency needs a deliberate transcript-ordering and session-write-lock design; simply raising the lane cap could let multiple runs mutate one session concurrently.
- Auto-yield is not just a config key: it needs semantics for in-flight tool calls, cancellation boundaries, queued user messages, and whether it should reuse `sessions_yield` metadata.
- Closing this umbrella before #63864 merges and before the auto-yield slice is designed would drop requested behavior that remains missing on current main.
