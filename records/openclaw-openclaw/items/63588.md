## Summary

Keep this PR open. Current main still does not implement the Slack-owned CDN redirect-auth behavior the PR targets: Slack auth is retained for same-origin redirects only, while current tests still expect auth to be stripped on `files.slack.com` to `slack-edge.com` / `downloads.slack-edge.com` redirects. The PR is still a plausible implementation candidate, but it needs a current-main rework before landing because the head diff is dirty/stale and security-sensitive around bearer-token forwarding.

## Best Possible Solution

Keep this PR open as a focused implementation candidate, but require a current-main rebase or replacement Slack plugin patch. The best fix belongs in `extensions/slack/src/monitor/media.ts`: retain `Authorization` only across HTTPS Slack-owned redirect hosts, strip credentials for non-Slack redirects, preserve `fetchRemoteMedia` and SSRF protections plus the local `media.runtime.ts` dispatcher boundary, keep the HTML-login-page guard, and update Slack media tests for CDN auth retention, non-Slack stripping, HTTP downgrade rejection, and SSRF-blocked redirect targets.
