## Summary

Keep this issue open. Current main still serializes agent turns through per-session FIFO command lanes, and `sessions_send`/A2A traffic is distinguishable as `inputProvenance.kind === "inter_session"` but is not assigned a lower queue priority. The KinthAI comment usefully confirms the same production symptom and suggests the same two-tier queue shape; the provided related items do not supersede this queue-starvation report.

## Best Possible Solution

Keep the issue open and implement a conservative two-tier command-lane priority: preserve existing FIFO behavior for equal-priority work, demote `sessions_send`/`inter_session` agent-originated turns behind human/channel/control turns for the same target session, and add regression tests for queue ordering and sessions_send starvation. Treat budget throttling/back-pressure as a separate follow-up if needed.

## Clownfish Work Prompt

Fix human/channel turns being starved behind `sessions_send` agent-to-agent traffic in the agent command lanes. Implement a conservative two-tier queue priority in the command queue: default/foreground entries keep existing behavior, known agent-originated `inputProvenance.kind === "inter_session"` entries are background, and FIFO ordering is preserved within each tier. Apply the priority through `runEmbeddedPiAgent` so both the per-session lane and the global lane use the same classification; do not change `session.agentToAgent.maxPingPongTurns`, A2A prompts, channel routing semantics, budget throttling, or provider behavior. Add regression tests proving: foreground entries queued behind background entries in the same lane run first once the active task finishes; equal-priority entries remain FIFO; active tasks are not preempted; and a `sessions_send`/inter-session run is demoted relative to a later human/non-inter-session turn for the same target session. ## Risks / Open Questions

- A fix should preserve FIFO ordering within each priority tier and must not preempt an already-active run; only queued work should reorder.
- Classifying all non-`inter_session` work as foreground may also prioritize cron/system events unless the implementation is scoped carefully. The narrowest fix is to demote known agent-originated inter-session traffic or add an explicit queue priority at the gateway/channel boundary.
- The KinthAI comment also raises budget/back-pressure concerns. That is adjacent but should not be bundled into the first starvation fix unless maintainers explicitly want a larger policy change.
