## Summary

Keep open. Current main still does not implement Vapi as a bundled Voice Call provider, and recent maintainer discussion explicitly kept this as a product and ownership decision rather than a cleanup close. The external Clawdia bridge is useful context, but it is not first-class `provider: "vapi"` support in `@openclaw/voice-call`.

## What This Changes

The issue requests adding Vapi as a Voice Call plugin provider, including provider code, config schema, provider registration, tests, and docs.

## Best Possible Solution

The desired end state is an explicit ownership decision, followed by either a first-class bundled Vapi provider wired through Voice Call end to end or a documented external/ClawHub path backed by any missing public telephony-provider seam.

## Reproduction Assessment

Not applicable as a runtime bug report. The absence is high-confidence by source inspection: `provider: "vapi"` is not accepted by the schema, no provider module exists, and runtime provider resolution has no Vapi branch.

## Solution Assessment

Unclear until maintainers choose the product boundary. If Vapi belongs in the bundled Voice Call plugin, an end-to-end provider/config/runtime/manifest/docs/test implementation is the narrowest maintainable path; otherwise the better path is external bridge/plugin documentation plus any needed public seam.
