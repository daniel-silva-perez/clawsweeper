## Summary

Keep open. Current main still does not contain a Vapi Voice Call provider, and recent maintainer discussion explicitly kept this request open for product and ownership judgment rather than closing it as implemented, stale, or superseded.

## What This Changes

The issue requests adding Vapi as a first-class Voice Call plugin provider with provider implementation, configuration schema, registration, tests, and docs.

## Best Possible Solution

The desired end state is an explicit ownership decision, followed by either a first-class bundled Vapi provider wired through Voice Call end to end or a documented external/ClawHub path backed by any missing public telephony-provider seam.

## Reproduction Assessment

Not applicable as a runtime bug report. The current absence is high-confidence by source inspection: `provider: "vapi"` is not accepted by the schema and the runtime has no Vapi branch or provider module.

## Solution Assessment

Unclear until maintainers choose the product boundary. If Vapi belongs in the bundled Voice Call plugin, an end-to-end provider/config/runtime/manifest/docs/test implementation is the narrowest maintainable path; otherwise the better path is external plugin/ClawHub documentation plus any needed public seam.
