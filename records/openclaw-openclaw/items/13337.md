## Summary

Keep open. Current main still does not implement Vapi as a Voice Call provider, and the recent maintainer discussion kept this as a product/ownership decision rather than closing it as implemented, stale, or superseded. The existing Clawdia Vapi bridge is useful related context, but it is not first-class `provider: "vapi"` support in the bundled Voice Call plugin.

## What This Changes

The issue requests adding Vapi as a first-class Voice Call plugin provider with provider code, config schema, runtime registration, tests, and documentation.

## Best Possible Solution

The desired end state is an explicit ownership decision, followed by either a first-class bundled Vapi provider wired through Voice Call end to end or a documented external/ClawHub path backed by any missing public telephony-provider seam.

## Reproduction Assessment

Not applicable as a runtime bug report. The absence is high-confidence by source inspection: `provider: "vapi"` is not accepted by the schema and the runtime has no Vapi branch or provider module.

## Solution Assessment

Unclear until maintainers choose the product boundary. If Vapi belongs in the bundled Voice Call plugin, an end-to-end provider/config/runtime/manifest/docs/test implementation is the narrowest maintainable path; otherwise the better path is external plugin/ClawHub documentation plus any needed public seam.
