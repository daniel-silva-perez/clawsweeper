## Summary

Keep open. Current main still does not include Vapi as a bundled Voice Call provider: provider enums, config schema, runtime resolution, manifest metadata, provider exports, tests, and docs all remain limited to Telnyx, Twilio, Plivo, and Mock. The documented Clawdia Vapi bridge is useful related context, but it is an external HTTP bridge rather than `provider: "vapi"` support in `@openclaw/voice-call`, and recent maintainer discussion kept this open for product and ownership judgment.

## What This Changes

Requested adding Vapi as a Voice Call plugin provider, including provider implementation, config schema, runtime registration, tests, and docs.

## Best Possible Solution

Keep this open for maintainer product judgment. If Vapi should be first-party in the bundled Voice Call plugin, add `VapiProvider`, Vapi config/env/schema validation, runtime provider selection, manifest metadata/secret inputs, provider/config/runtime tests, docs at https://docs.openclaw.ai/plugins/voice-call, and a changelog entry. If maintainers prefer external ownership, document the existing Clawdia bridge path or first add a public Voice Call telephony-provider registration seam, then track Vapi as a ClawHub/plugin item.

## Risks / Open Questions

- Bundling Vapi would add a new external call-control API, credential fields, webhook parsing, and likely signature or shared-secret verification requirements, so it needs product/security ownership rather than an automatic sweep fix.
- Current code does not expose a public third-party telephony-provider registration seam for `voice-call`; a true external `provider: "vapi"` path may require a new documented plugin API, while the existing Clawdia bridge is a separate HTTP bridge approach.
- The issue includes hard-coded Vapi Assistant IDs and an earlier implementation-complete comment that did not land on main; a fix should avoid committing live/personal credentials and should be based on Vapi's current API contract.
