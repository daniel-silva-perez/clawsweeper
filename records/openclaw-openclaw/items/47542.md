## Summary

Keep this PR open. Current main still lacks composing-aware WhatsApp inbound debounce, and the PR is a useful focused feature direction, but the proposed patch has merge blockers around strict config validation, group debounce key matching, repeated presence subscriptions, and an unbounded LID mapping cache.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, passes it through account and auto-reply setup, subscribes to WhatsApp presence updates, and extends the shared inbound debouncer with touch helpers.

## Best Possible Solution

Land a focused update that makes `composingDebounceMs` a real accepted WhatsApp root/account config field, documents it, covers the debouncer touch APIs with tests, wires Baileys presence events to the same direct and group debounce keys already used by the monitor, deduplicates subscriptions, and bounds short-lived identity caches.

## Reproduction Assessment

Not applicable as a user bug reproduction, but the current-main absence is high-confidence from source inspection. The PR's schema blocker is reproducible by parsing a WhatsApp config containing `composingDebounceMs`, because the strict WhatsApp zod shape does not include that key.

## Solution Assessment

No, this PR is not yet the best way to solve the issue. The feature direction is reasonable, but the maintainable solution needs schema/metadata/docs/test coverage plus corrected presence keying and bounded subscription/cache behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the setting into strict config validation:** `src/config/types.whatsapp.ts:85`
  - body: The PR adds `composingDebounceMs` to the WhatsApp config type and then reads it at runtime, but it never adds the key to the strict WhatsApp zod shape. A user config that sets this option will be rejected as an unknown key before the feature can run.
  - confidence: 0.95
- **[P2] Match group buffers with the same sender key:** `extensions/whatsapp/src/inbound/monitor.ts:486-487`
  - body: The presence handler touches group buffers using an E.164 prefix, while the debounce key prefers the participant identity/JID before `senderE164`. In normal group traffic with a participant JID, composing events will not match the buffered message key, so the debounce window is not extended.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:400-402`
  - body: `sock.presenceSubscribe(remoteJid)` is called for every inbound message while the feature is enabled. Keep a per-chat subscription guard and retry on failure to avoid unnecessary WhatsApp presence traffic and rate-limit exposure.
  - confidence: 0.86
- **[P2] Expire the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:76`
  - body: `jidToE164Cache` records every distinct JID indefinitely, but the mapping only needs to bridge the short interval between an inbound message and its presence events. Add a TTL or max-size cap so long-running sessions do not grow this map without bound.
  - confidence: 0.82

## Security Review
