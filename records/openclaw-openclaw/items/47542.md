## Summary

Keep this PR open. Current main still does not implement composing-aware inbound debounce for WhatsApp: there is no `composingDebounceMs` config, no `presence.update` listener or `presenceSubscribe` path wired into inbound buffering, and the shared inbound debouncer still only returns `enqueue` and `flushKey`. The proposed feature remains plausible, but the PR has unresolved review blockers around schema wiring, group key matching, repeated presence subscriptions, and an unbounded LID-to-E164 cache.

## Best Possible Solution

Keep the PR open and ask for a focused update on the existing branch: add runtime schema/config metadata and tests for `composingDebounceMs`, add covered `touch`/`touchByPrefix` debouncer behavior, wire WhatsApp `presence.update` using direct and group key formats that match current debounce buffers, dedupe `presenceSubscribe`, bound or TTL the LID-to-E164 cache, correct the security-impact section, and validate the WhatsApp plugin tests plus the shared inbound debounce tests.
