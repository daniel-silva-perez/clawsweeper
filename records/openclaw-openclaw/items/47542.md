## Summary

Keep this PR open. Current main still does not implement composing-aware inbound debounce for WhatsApp, and the PR remains a plausible focused feature, but the branch is not ready to merge because the new config is not accepted by the strict runtime schema and the proposed presence wiring has concrete correctness and operational blockers.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, propagates it through account and auto-reply wiring, subscribes to WhatsApp presence updates, and adds `touch`/`touchByPrefix` helpers to the shared inbound debouncer.

## Best Possible Solution

Land a focused branch update that makes `composingDebounceMs` a real accepted WhatsApp config field at root and account scope, covers the debouncer touch APIs with tests, wires Baileys presence events to the same direct and group debounce keys that current main builds, dedupes subscriptions, bounds the LID mapping cache, and updates docs/metadata for the new user-facing option.

## Reproduction Assessment

Not applicable as a bug reproduction, but the current-main absence is high-confidence from source inspection. The PR's schema blocker is reproducible by parsing a WhatsApp config with `composingDebounceMs` because the strict `WhatsAppConfigSchema` does not include that key.

## Solution Assessment

No, the current PR branch is not yet the best way to solve the feature request. The direction is reasonable, but the maintainable fix needs schema/metadata/test coverage plus corrected presence keying and bounded subscription/cache behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the setting into strict config validation:** `src/config/types.whatsapp.ts:85`
  - body: The PR exposes `composingDebounceMs` in the config type and then resolves/passes it at runtime, but it never adds the key to `buildWhatsAppCommonShape`. Both WhatsApp schemas are strict, so any user who sets this option will have their config rejected before the feature can run.
  - confidence: 0.95
- **[P2] Match group buffers with the same sender key:** `extensions/whatsapp/src/inbound/monitor.ts:486-487`
  - body: The presence handler touches group buffers using an E.164 sender prefix, but the debounce `buildKey` prefers the participant identity/JID before `senderE164`. In normal group traffic with `participantJid`, composing events will not match the buffered message key, so the debounce window is not extended.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:400-402`
  - body: `sock.presenceSubscribe(remoteJid)` is called for every inbound message while the feature is enabled. Baileys needs a subscription to receive presence updates, but repeatedly subscribing for the same active chat adds avoidable network traffic and rate-limit risk; keep a per-chat guard and retry only after failure.
  - confidence: 0.86
- **[P2] Expire the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:76`
  - body: `jidToE164Cache` records every distinct JID indefinitely, but the mapping only needs to bridge the short interval between an inbound message and its presence events. Long-running WhatsApp sessions with many contacts can grow this map without bound; add a TTL or max-size cap.
  - confidence: 0.82
