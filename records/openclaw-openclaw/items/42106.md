## Summary

Keep this open. Current main still matches the reported block-streaming boundary-loss path: the paragraph separator is consumed by the chunker, text_end delivery can flush the stripped chunks as independent block replies, and the delivered payload/context still has no boundary metadata a downstream edit-in-place client could use to reconstruct the lost blank line.

## Best Possible Solution

Keep this open and fix the shared block-streaming pipeline. Preserve boundary semantics from EmbeddedBlockChunker through block reply delivery, either by carrying explicit paragraph/newline/hard-split metadata or by retaining consumed separators while normalizing dedupe/coalescing comparisons. Add a regression test that reconstructs visible text from multiple flushed text_end block deliveries and preserves Markdown such as `# Title\n\nFirst paragraph.`.

## Risks / Open Questions

- A naive fix that appends separators directly to emitted chunks could double-insert separators when multiple chunks are coalesced into one outbound payload.
- The regression should cover reconstruction from multiple flushed text_end block replies, not only chunker output, because the user-visible corruption happens at the delivery/reconstruction boundary.
