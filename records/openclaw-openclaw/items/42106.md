## Summary

Keep this issue open. Current main still strips paragraph/newline separators before emitting block chunks, and the downstream block-reply payload/metadata surface does not carry boundary semantics that edit-in-place clients could use to reconstruct a paragraph break across separate streamed deliveries. Existing tests still assert separator-stripped chunks rather than the reported reconstruction behavior.

## Best Possible Solution

Keep this issue open and fix the shared block-streaming pipeline. Preserve boundary semantics from EmbeddedBlockChunker through block reply delivery, either by retaining consumed separators while normalizing dedupe/coalescing or by carrying explicit paragraph/newline/hard-split metadata. Add a regression test that reconstructs visible text from multiple flushed block deliveries and preserves Markdown such as '# Title\n\nFirst paragraph.'.
