## Summary

Keep this issue open. Current main still matches the reported block-streaming boundary-loss path: paragraph separators are consumed by the chunker, text_end streaming can flush the stripped chunk as an independent block reply, and the delivered payloads still do not carry boundary metadata that a downstream visible-message reconstructor could use.

## Best Possible Solution

Keep this open and fix the shared block-streaming pipeline. Preserve boundary semantics from EmbeddedBlockChunker through delivery, either by carrying explicit paragraph/newline/hard-split metadata or by retaining consumed separators while normalizing only dedupe/coalescing comparisons. Add a regression test that reconstructs visible text from multiple flushed text_end block deliveries and preserves the Markdown paragraph break.

## Risks / Open Questions

- A naive fix that appends separators directly to emitted chunks could double-insert separators when multiple chunks are coalesced into one outbound payload.
- There is still no focused regression test that reconstructs visible text from at least two flushed text_end block replies and asserts that Markdown paragraph structure such as "# Title\n\nFirst paragraph." survives.
