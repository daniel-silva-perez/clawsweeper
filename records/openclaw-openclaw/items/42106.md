## Summary

Keep open. Current main still has the reported block-streaming boundary loss: the shared chunker consumes paragraph/newline separators before emitting chunks, and downstream block-reply delivery only carries plain text after a flush. The issue is also already paired with open PR #73981, which is an implementation candidate and should be reviewed or closed before this issue is resolved.

## What This Changes

Preserve Markdown paragraph boundaries when block streaming splits one assistant reply across multiple delivered updates.

## Best Possible Solution

Keep this issue open until the linked implementation PR merges or is replaced by an equivalent shared pipeline fix. The desired end state is boundary-preserving block streaming, either by retaining consumed separators in emitted chunks while normalizing dedupe or by carrying explicit paragraph/newline/hard-split metadata through delivery, with regression coverage for reconstructed multi-paragraph Markdown.

## Reproduction Assessment

Yes. The high-confidence source reproduction is the existing paragraph-preference harness: `First block line\n\nSecond block line` is delivered as two `onBlockReply` payloads without the blank-line separator, so a client concatenating delivered updates collapses the paragraph boundary.

## Solution Assessment

Yes, the right direction is a shared block-streaming pipeline fix rather than per-channel join rules, because flush boundaries can come from paragraph, newline, idle, coalescing, or size limits. PR #73981 is the active implementation candidate to review before closing this issue.
