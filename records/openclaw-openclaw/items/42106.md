## Summary

Keep this issue open. Current main at 6d323ee still matches the reported block-streaming separator-loss path: the chunker consumes paragraph/newline boundary text, text_end delivery can flush those chunks as separate block replies, and the delivered payload/context still has no boundary metadata for an edit-in-place client to reconstruct the lost blank line. A related Telegram message-chunking changelog entry covers Telegram HTML retry chunking, not this shared block-streaming pipeline.

## Best Possible Solution

Keep this issue open and fix the shared block-streaming pipeline. Preserve boundary semantics from EmbeddedBlockChunker through block reply delivery, either by carrying explicit paragraph/newline/hard-split metadata or by retaining consumed separators while normalizing dedupe and coalescing comparisons. Add a regression test that reconstructs visible text from multiple flushed text_end block deliveries and preserves Markdown like `# Title\n\nFirst paragraph.`.

## Risks / Open Questions

- A naive fix that always joins separate block deliveries with `\n\n` would corrupt non-paragraph splits such as size-based hard breaks or sentence/whitespace chunking.
- Appending separators directly to emitted chunks could double-insert separators when multiple chunks remain in one coalesced outbound payload; the fix needs normalized dedupe/coalescing behavior or explicit boundary metadata.
