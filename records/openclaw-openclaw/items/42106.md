## Summary

Keep this issue open. The existing discussion identified a shared block-streaming boundary bug, and current main still matches that diagnosis: paragraph/newline separators can be consumed by the chunker, text_end streaming can flush the stripped chunk as a separate delivery, and downstream block reply payloads do not carry boundary metadata for clients reconstructing one visible reply.

## Best Possible Solution

Keep this issue open and fix the shared block-streaming pipeline. Preserve boundary semantics from EmbeddedBlockChunker through delivery, either with explicit paragraph/newline/hard-split metadata or by retaining consumed separators while normalizing only dedupe comparisons. Add a regression test that reconstructs visible text from at least two flushed text_end block deliveries and preserves the Markdown paragraph break.

## Risks / Open Questions

- A naive fix that appends separators directly to chunks could double-insert separators when chunks are coalesced into one outbound payload.
- There is still no focused regression test for text_end block streaming reconstructing Markdown such as "# Title\n\nFirst paragraph." across separate flushed deliveries.
