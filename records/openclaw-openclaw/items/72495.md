## Summary

Keep this PR open. Current main still keys the guarded workspace bootstrap file cache on canonical path, device, inode, size, and mtime only, so the same-inode/equal-size/restored-mtime stale-cache gap remains. The PR is a narrow, relevant fix, and the latest patch addresses the earlier Greptile test flake by asserting observable reload behavior instead of direct ctime inequality.

## What This Changes

The PR adds `ctimeMs` to the workspace bootstrap file cache identity, adds same-inode/same-size/restored-mtime regression coverage, and records the fix in the changelog.

## Best Possible Solution

Land this PR or an equivalent maintainer patch that adds `ctimeMs` to the existing fstat-backed workspace file identity, keeps the focused restored-mtime regression test, and avoids broadening the bootstrap cache design.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: load a workspace `AGENTS.md`, rewrite it in place with equal-size content, restore the original mtime, and current main can reuse the old cache entry because the identity lacks ctime.

## Solution Assessment

Yes. Adding `ctimeMs` to the existing identity is the narrowest maintainable fix for this cache gap; hashing every read or removing the lower cache would be broader and would reopen the performance tradeoff documented around the related refresh work.
