## Summary

Keep this issue open. Current main has adjacent compaction checkpoint branch/restore RPCs and native subagent transcript forking, but it still does not provide arbitrary named session snapshots, `/session save`, `/session load`, `/session restore`, reusable snapshot templates, or restore from `.reset` / `.deleted` transcript archives. The latest maintainer-facing issue comment also explicitly kept this open for the same product gap.

## Best Possible Solution

Keep this issue open until maintainers either scope a deliberate native session snapshot surface or split it into smaller canonical issues. A complete path would add typed Gateway snapshot save/list/load/delete and archive-restore methods, `/session save|load|restore` command handling, UI/CLI controls, deterministic preservation of transcript/model/session metadata where feasible, reusable spawn-from-snapshot support, and a security-reviewed restore path for `.reset` / `.deleted` transcript archives.

## Risks / Open Questions

- The request bundles several product shapes: named checkpoints, rollback in the current session, restore from reset/deleted archives, reusable templates, and spawn-from-snapshot batch workflows. A single narrow implementation may not satisfy all of them without maintainer scoping.
- Session snapshots would persist and rehydrate full transcripts, model/session metadata, tool outputs, and possibly sensitive context, so storage location, permissions, redaction, retention, and restore authorization need deliberate design.
- The prompt-cache-sharing use case depends on byte-identical prefixes and system prompts; any fork metadata or dynamic session-specific prompt content could defeat that optimization even if transcript forking works functionally.
