## Summary

Keep open. Current main has adjacent recovery and branching primitives, but it still does not implement the requested named session snapshot workflow, reusable templates, or import/restore from reset and deleted transcript archives; the latest maintainer-facing discussion also kept the design open.

## What This Changes

Add explicit session snapshot save/load/fork/restore support so users can checkpoint, reload, branch, and reuse conversation state.

## Best Possible Solution

Define a native session snapshot model on top of SessionManager, session metadata, compaction checkpoint artifacts, reset/deleted transcript archives, and subagent fork support, then add coherent save/list/load/branch/import semantics across chat commands, Gateway API, CLI/Control UI where appropriate, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main gap is high-confidence from source and docs: `/session` only accepts `idle`/`max-age`, while checkpoint and fork primitives are narrower than named snapshot save/load/restore.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer decisions on restore versus branch semantics, retention/import behavior, permissions, and cache-preserving forks before a narrow fix can be chosen.
