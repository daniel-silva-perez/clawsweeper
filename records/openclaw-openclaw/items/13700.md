## Summary

Keep open. Current main has adjacent primitives, especially compaction checkpoint list/get/branch/restore RPCs and native `sessions_spawn` transcript forking, but it still does not provide the requested user-facing named `/session save` and `/session load` workflow, reusable session templates, or restore from `.reset`/`.deleted` transcript archives. A maintainer comment on 2026-04-26 already reached the same conclusion, and the current-main source still matches that assessment.

## Best Possible Solution

Keep this open and design a native session snapshot feature that builds on the existing SessionManager, session store, compaction checkpoint, and subagent fork primitives. The design should define explicit save/list/load/branch semantics, user-facing slash or Gateway/CLI surfaces, restore/import rules for `.reset` and `.deleted` archives, permissions and retention, docs, and focused tests.

## Risks / Open Questions

- The request combines several product slices: current-session rollback, named checkpoints, subagent fork/prompt-cache behavior, reusable batch templates, and restore from reset/deleted archives. A single implementation should be scoped before code work starts.
- Compaction checkpoint restore is already a mutating/admin-sensitive Gateway action tied to pre-compaction artifacts. Turning it into user-facing save/load needs explicit naming, retention, permission, interruption, and destructive-vs-branch semantics.
- Archived `.reset` and `.deleted` transcripts are native transcript files, but treating arbitrary exports, sanitized history, or trajectory bundles as restore inputs would risk corrupting the session DAG or reintroducing unsafe tool/approval state.
