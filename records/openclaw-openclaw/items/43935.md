## Summary

Keep open. Current main still represents channel hot reloads as whole-channel restart targets, so a Telegram account-scoped config edit still maps to a full Telegram channel restart; the linked open PR #43938 remains the canonical implementation path because it includes closing syntax for this issue.

## What This Changes

The issue requests account-scoped gateway reload planning so `channels.<channel>.accounts.<accountId>.*` changes restart only the affected channel account instead of the entire channel.

## Best Possible Solution

Review, land, or replace PR #43938 with a current-main implementation that adds account-scoped restart targets to `GatewayReloadPlan`, promotes mixed or channel-global changes to full-channel restarts, updates hot reload and secrets reload consumers, preserves active-work deferral, and adds focused planner and handler regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad reload prefix, the planner adds `telegram` to `restartChannels`, and the handler calls channel stop/start without an account id; the discussion also includes matching runtime logs.

## Solution Assessment

Yes. Account-scoped reload planning is the narrow maintainable fix for the reported blast radius; Telegram reply retry or drain hardening should be a separate follow-up only if failures still reproduce after reload granularity is fixed.
