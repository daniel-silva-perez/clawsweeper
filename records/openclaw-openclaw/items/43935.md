## Summary

Keep open. Current main still represents channel reloads as whole-channel targets and restarts Telegram without an account id, while an open linked implementation PR with closing syntax is already the right review path for this issue.

## What This Changes

Account-scoped channel config changes under `channels.<channel>.accounts.<accountId>.*` should restart only the affected account instead of the entire channel.

## Best Possible Solution

Land or replace PR #43938 with a rebased implementation that adds account-scoped restart targets to the reload plan, promotes mixed/channel-global changes to full-channel restarts, preserves active-work deferral and secrets reload semantics, and adds focused planner and handler regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad reload prefix, the planner adds `telegram` to `restartChannels`, and the handler calls `stopChannel/startChannel` without an account id; the discussion also includes matching timestamped Telegram failure logs.

## Solution Assessment

Yes. Account-scoped reload planning is the narrow maintainable solution for this issue; any additional Telegram delivery retry or drain hardening should be handled separately if it still reproduces after reload granularity is fixed.
