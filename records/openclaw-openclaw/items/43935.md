## Summary

Keep open. Current main still plans strict account-scoped channel config changes as whole-channel reloads, and an open implementation PR is explicitly linked as the fix path, so the issue should remain tracked until that PR is reviewed, landed, replaced, or closed.

## What This Changes

The issue requests account-scoped gateway hot reload for `channels.<channel>.accounts.<accountId>.*` config changes instead of restarting the entire channel.

## Best Possible Solution

Land or replace PR #43938 with a rebased implementation that represents account-scoped restart targets in the reload plan, promotes mixed or channel-global changes to full-channel restarts, preserves active-work deferral, updates reload-plan consumers, and adds focused planner and handler regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a path such as `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad `channels.telegram` reload prefix, enters `restartChannels`, and the hot reload handler restarts Telegram without an account id.

## Solution Assessment

Yes. Account-scoped reload planning is the narrow maintainable solution for this issue, provided full-channel precedence and active-work deferral remain intact; any remaining Telegram delivery retry or drain hardening should be tracked separately if failures persist.
