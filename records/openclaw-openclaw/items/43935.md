## Summary

Keep open. Current main still collapses strict account-scoped channel config changes into whole-channel reload targets, and the same-author implementation PR #43938 is open with closing syntax, so this issue should remain tracked until that PR is reviewed, landed, replaced, or closed.

## What This Changes

The issue requests gateway hot reload planning that restarts only affected channel accounts for strict `channels.<channel>.accounts.<accountId>.*` config changes while preserving full-channel restarts for global or mixed channel changes.

## Best Possible Solution

Land or replace PR #43938 with a rebased implementation that represents account-scoped restart targets in the reload plan, promotes mixed or channel-global changes to full-channel restarts, preserves active-work deferral, updates all reload-plan consumers, and adds focused planner and handler regression tests.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad `channels.telegram` prefix, is planned into `restartChannels`, and the hot reload handler calls channel-wide stop/start even though the channel manager accepts an optional account id.

## Solution Assessment

Yes, the requested account-scoped reload planning is the narrow maintainable direction, provided the implementation keeps full-channel precedence and preserves current active-work deferral. Any remaining Telegram reply retry or drain hardening should be handled separately if failures persist after reload granularity is fixed.
