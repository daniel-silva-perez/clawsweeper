## Summary

Keep open. The report is a current Windows/npm regression with concrete logs and attachments, and current main still contains the Telegram startup paths that match the user-visible failures: polling waits for `deleteWebhook` before the runner starts, recoverable cleanup failures retry rather than confirming an already-empty webhook, and native command sync still calls `deleteMyCommands` then `setMyCommands`. Current main does include a Windows native-Jiti guard that may address the adjacent module-loading symptom, and docs/doctor now diagnose one 404/apiRoot misconfiguration class, but that is only partial evidence and does not close the reported 2026.4.24+ Windows regression.

## Best Possible Solution

Keep this issue open as the broader Windows Telegram regression tracker. The best path is to review or land #47384 or an equivalent Telegram-owned cleanup fallback for recoverable `deleteWebhook` failures, then separately verify the npm Windows package on current main against the reporter's 2026.4.23-vs-2026.4.24+ scenario, including command sync and `bot-native-commands.runtime` loading. If the attached diagnostics show an accidental full `/bot<TOKEN>` `apiRoot`, route that through `openclaw doctor --fix`, but do not assume that is the only cause without reproducing the release regression.
