## Summary

Current main still matches the reported heartbeat contention path: `requests-in-flight` heartbeat skips are retried by the shared 1 second wake retry, tests assert that timing, and there is still no heartbeat-specific busy-session retry/backoff config in schema, types, docs, or generated config help. Because heartbeat turns still enter the normal session/global agent-run queues, a heartbeat that starts immediately after an active Telegram turn can still make later inbound messages wait behind it under the default collect queue mode.

## Best Possible Solution

Keep this issue open and fix the heartbeat busy-session retry path directly. A narrow implementation should add a heartbeat-specific `requests-in-flight` retry delay or backoff separate from generic wake error retry, choose a conservative maintainer-approved default, expose it under `agents.defaults.heartbeat` and per-agent heartbeat config, update schema/types/generated config help/public heartbeat docs, and add scheduler/wake regression tests. Leave #13744 for broader session-lock behavior and #14051 for activity-based heartbeat semantics.
