## Summary

Current main still has the reported heartbeat contention behavior: a `requests-in-flight` heartbeat skip is retried through the shared 1 second wake retry, tests still assert that timing, and there is still no heartbeat-specific busy-session retry/backoff config in schema, types, or docs. The latest release tag also still has the same 1 second path, while the direct fix candidate in #40629 was closed unmerged.

## Best Possible Solution

Keep this issue open and implement a narrow heartbeat busy-session retry policy. The fix should keep the no-drift behavior from #39182, add a separate configurable delay/backoff for `requests-in-flight` skips under `agents.defaults.heartbeat` and per-agent heartbeat config, update schema/types/generated help and https://docs.openclaw.ai/gateway/heartbeat, and add wake/scheduler regression tests. Leave #13744 for broader session-lock scope and #14051 for activity-based heartbeat semantics.

## Risks / Open Questions

- A naive longer retry could reintroduce the old heartbeat drift problem from #29762 if it advances `nextDueMs` or bypasses wake target coalescing incorrectly.
- Changing the shared `DEFAULT_RETRY_MS` would also affect thrown handler error retries; the fix should separate busy-session retry/backoff from generic wake errors.
- The default delay is product-sensitive: too short can keep blocking active Telegram conversations, while too long can delay real heartbeat alerts and main-session cron wakeups.
