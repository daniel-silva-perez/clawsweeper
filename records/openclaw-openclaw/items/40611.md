## Summary

Keep open. Current main still retries `requests-in-flight` heartbeat skips through the shared 1 second wake retry, tests still assert that timing, and the heartbeat config schema/docs do not expose a separate busy-session retry delay. The latest release tag has the same path, and the direct fix candidate in #40629 was closed unmerged.

## Best Possible Solution

Keep this issue open and implement a narrow heartbeat busy-session retry policy: keep the no-drift behavior from #39182, add a separate configurable delay/backoff for `requests-in-flight` skips under `agents.defaults.heartbeat` and per-agent heartbeat config, update schema/types/generated help/docs, and add wake/scheduler regression tests. Leave #13744 for broader session-lock scope and #14051 for activity-based heartbeat semantics.

## Repair Work Prompt

Fix the heartbeat busy-session retry regression described in this issue. Current main retries `{ status: "skipped", reason: "requests-in-flight" }` through `DEFAULT_RETRY_MS = 1_000` in `src/infra/heartbeat-wake.ts`, causing heartbeats to start immediately after active conversations release the lane. Implement a heartbeat-specific busy-session retry delay/backoff, defaulting to a conservative delay around the prior #40629 proposal (60s unless existing maintainer guidance says otherwise), without changing generic thrown-handler error retry behavior. Wire the setting through `agents.defaults.heartbeat` and `agents.list[].heartbeat` as a documented `retryDelayMs` or equivalent duration field, update zod schema/types/generated config help/labels/inventory as required by existing config patterns, and document it in `docs/gateway/heartbeat.md`. Preserve the #39182 no-drift behavior: do not advance `nextDueMs` on interval `requests-in-flight` skips. Do not broaden this PR into #13744 session lock scope or #14051 activity-based heartbeat semantics. Add focused regression coverage in `src/infra/heartbeat-wake.test.ts`, `src/infra/heartbeat-runner.scheduler.test.ts`, and config schema tests; update changelog with reporter credit if the repo policy requires a user-facing fix entry.
