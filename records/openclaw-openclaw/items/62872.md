## Summary

Keep open. Current main still runs the generic `sessions_send` A2A ping-pong plus announce flow for normal peer/persistent session sends, and the tool still has no per-call ping-pong disable or max-turn parameter. Recent inter-session `isUser=false` marking and the merged ACP parent-child skip reduce adjacent failure modes, but they do not satisfy the requested persistent session behavior.

## What This Changes

The issue requests stopping or controlling `sessions_send` A2A ping-pong/announce behavior so persistent session-to-session sends do not produce duplicate messages.

## Best Possible Solution

Keep this issue as the canonical tracker for persistent-session duplicate behavior. The desired end state is a context-aware `sessions_send` A2A lifecycle or explicit per-call ping-pong control that prevents persistent peer sends from generating unwanted follow-up turns, while preserving the requester-owned ACP child skip and any intentional peer announce workflow with regression coverage.
