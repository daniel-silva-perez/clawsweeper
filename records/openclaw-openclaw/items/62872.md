## Summary

Keep this issue open. Current main still only skips the A2A ping-pong/announce path for a requester sending to its own parent-owned oneshot ACP child; persistent ACP sessions and normal peer/persistent session sends still run the ping-pong loop and announce step. The merged #69817 work is relevant and shipped, but it explicitly covers a narrower ACP-child case rather than the persistent-session duplicate behavior reported here.

## Best Possible Solution

Keep this issue open as the canonical tracker for persistent-session sessions_send duplicate behavior. The likely fix belongs in src/agents/tools/sessions-send-tool.ts and src/agents/tools/sessions-send-tool.a2a.ts: make A2A context-aware for persistent peer-session sends or add an explicit per-call ping-pong max/disable option, while preserving the #69817 parent-owned oneshot ACP child skip and adding regression coverage for persistent peer-session duplicates.
