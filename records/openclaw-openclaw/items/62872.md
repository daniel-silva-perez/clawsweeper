## Summary

Keep open. Current main still preserves the generic peer/persistent `sessions_send` A2A ping-pong plus announce behavior, while the merged ACP fix only skips parent-owned background ACP targets. The report remains a valid product/API decision rather than an implemented cleanup candidate.

## What This Changes

The issue requests preventing `sessions_send` between persistent sessions from creating duplicate persisted messages through the automatic A2A ping-pong and announce flow.

## Best Possible Solution

Keep this as the canonical tracker for persistent-session duplicate behavior. The desired end state is an explicit, documented A2A policy or per-call control that prevents unwanted persistent peer follow-up turns while preserving intentional peer announce workflows and the parent-owned ACP background skip.

## Reproduction Assessment

Yes, at source/test level. A normal peer or persistent `sessions_send` with default config still starts the target run, then enters the bounded reply loop and announce step; current tests assert that peer behavior rather than suppressing it.

## Solution Assessment

Unclear. The requested per-call `maxPingPongTurns` or context-aware skip could solve the report, but choosing that API/default affects documented peer A2A semantics and needs maintainer product judgment before a narrow repair PR is safe.
