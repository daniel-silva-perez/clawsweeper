## Summary

Keep open. Current main still has no per-call `sessions_send` A2A opt-out, and normal non-skipped peer sends still run the default ping-pong loop plus announce step. Recent ACP parent-owned background handling narrows adjacent loops, but it does not cover the normal persistent peer-session behavior described in the report.

## What This Changes

The issue requests preventing `sessions_send` between persistent sessions from producing duplicate A2A ping-pong and announce messages, either through per-call ping-pong control or context-aware behavior.

## Best Possible Solution

Keep this as the canonical tracker for persistent-session duplicate behavior. The desired end state is a context-aware `sessions_send` A2A lifecycle, or explicit per-call A2A control, that prevents unwanted persistent peer follow-up turns while preserving intentional peer announce workflows and the existing parent-owned ACP background skip.
