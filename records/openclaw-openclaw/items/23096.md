## Summary

Keep this PR open. Current main still has no Bitwarden/Vaultwarden SecretRef integration, and the discussion has active maintainer direction to keep it as the focused Bitwarden candidate, but the branch is not merge-ready until maintainers choose the plugin-seam route versus a hardened exec-wrapper route and address the command-execution concern in the wrapper.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef documentation, a protocol-v1 Bitwarden CLI exec resolver script with TypeScript declarations, and tests for parsing, field extraction, grouping, and child-process resolution.

## Best Possible Solution

Keep the Bitwarden/Vaultwarden work focused, but land it only after the route is explicit: preferably a secret-provider plugin once the plugin seam is available, or a deliberately hardened exec-wrapper integration that keeps the secondary Bitwarden binary path validated and documented.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The absence on current main is source-verified, and the proposed behavior has a clear validation path with the focused resolver tests plus a live Bitwarden CLI smoke after the branch is refreshed.

## Solution Assessment

No, not as currently proposed. The exec-wrapper approach is plausible, but the wrapper must not resolve the Bitwarden CLI via unvalidated PATH, and maintainers still need to decide whether the plugin seam is the better long-term home.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Resolve the Bitwarden CLI without PATH search:** `scripts/bw-exec-resolver.mjs:27`
  - body: The configured exec provider validates only this wrapper script, but the wrapper then calls `execFile("bw", ...)` while the docs pass through `PATH`. A writable or overridden PATH can cause OpenClaw to run a different `bw` binary with the Bitwarden session/client env, bypassing the current exec-provider path hardening. Require an absolute/validated Bitwarden binary path, such as a `BW_BIN` override checked against trusted dirs, before spawning it.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a command-execution path that spawns the actual Bitwarden CLI through PATH after core has validated only the wrapper script.

Concerns:

- **[medium] PATH-resolved Bitwarden binary bypasses exec hardening:** `scripts/bw-exec-resolver.mjs:27`
  - body: The wrapper is the configured trusted exec command, but it invokes `bw` by name and the docs pass PATH into the wrapper. That lets the real secrets-reading binary be selected outside OpenClaw's command path validation, symlink policy, and trustedDirs checks.
