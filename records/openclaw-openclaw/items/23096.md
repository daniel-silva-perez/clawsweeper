## Summary

Keep this PR open. Current main still has only env/file/exec SecretRef transports and no Bitwarden/Vaultwarden docs or bundled resolver, while maintainer discussion points to a real remaining product choice between a future secret-provider plugin seam and a hardened exec-wrapper integration; the current PR also has a concrete command-path security issue before merge.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef documentation, a protocol-v1 Bitwarden CLI exec resolver script with TypeScript declarations, and tests for parsing, field extraction, grouping, and child-process resolution.

## Best Possible Solution

Land Bitwarden/Vaultwarden support only after the route is explicit: preferably as a secret-provider plugin once #72548 is available, or as a hardened bundled exec-wrapper/docs integration that validates the secondary Bitwarden binary path, documents trusted installation paths, and keeps secret/error redaction tight.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. For the blocking review finding, yes: the PR diff shows execFile("bw") and the docs pass PATH into the wrapper, so placing a different bw earlier in PATH would exercise the unsafe binary-selection path.

## Solution Assessment

No, not as currently proposed. The exec-wrapper approach is plausible, but the wrapper must not resolve the Bitwarden CLI through unvalidated PATH, and maintainers still need to decide whether the plugin seam is the narrower long-term home.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Resolve the Bitwarden CLI without PATH search:** `scripts/bw-exec-resolver.mjs:27`
  - body: The configured exec provider validates only this wrapper script, but the wrapper then calls `execFile("bw", ...)` while the docs pass through `PATH`. A writable or overridden PATH can cause OpenClaw to run a different `bw` binary with the Bitwarden session/client env, bypassing the current exec-provider path hardening. Require an absolute/validated Bitwarden binary path, such as a `BW_BIN` override checked against trusted dirs, before spawning it.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a command-execution path that spawns the actual Bitwarden CLI through PATH after core has validated only the wrapper script.

Concerns:

- **[medium] PATH-resolved Bitwarden binary bypasses exec hardening:** `scripts/bw-exec-resolver.mjs:27`
  - body: The wrapper is the configured trusted exec command, but it invokes `bw` by name and the docs pass PATH into the wrapper. That lets the real secrets-reading binary be selected outside OpenClaw's command path validation, symlink policy, and trustedDirs checks.
