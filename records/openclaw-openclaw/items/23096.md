## Summary

Keep this PR open, but do not merge it as-is. Current main still does not provide Bitwarden/Vaultwarden SecretRef documentation, a native bw source, or a Bitwarden wrapper, while the PR's exec-wrapper route introduces a security-sensitive PATH-resolved secondary binary that bypasses the current exec provider's command-path hardening.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef documentation, a protocol-v1 Bitwarden CLI exec resolver script with TypeScript declarations, and tests for parsing, field extraction, grouping, and child-process resolution.

## Best Possible Solution

Land Bitwarden/Vaultwarden support only after the route is explicit: preferably as a secret-provider plugin once #72548 provides the seam, or as a hardened bundled exec-wrapper integration that validates the secondary Bitwarden binary path, documents trusted installation paths, preserves error redaction, and coordinates with the related Bitwarden Secrets work in #29110.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. For the blocking finding, yes: the diff shows execFile("bw") and the docs pass PATH into the wrapper, so placing a different bw earlier in PATH would exercise the unsafe binary-selection path.

## Solution Assessment

No, not as currently proposed. The exec-wrapper route is plausible, but it must not resolve the secrets-reading Bitwarden CLI through unvalidated PATH, and maintainers still need to decide whether the plugin seam is the narrower long-term home.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Resolve the Bitwarden CLI without PATH search:** `scripts/bw-exec-resolver.mjs:29`
  - body: The configured exec provider validates only this wrapper script, but the wrapper then calls `execFile("bw", ...)` while the docs pass PATH into the wrapper. A writable or overridden PATH can cause OpenClaw to run a different `bw` binary with the Bitwarden session/client env, bypassing the current exec-provider path hardening. Require an absolute and validated Bitwarden binary path, such as a `BW_BIN` override checked against trusted dirs, before spawning it.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a command-execution path that spawns the actual Bitwarden CLI through PATH after core validates only the wrapper script.

Concerns:

- **[medium] PATH-resolved Bitwarden binary bypasses exec hardening:** `scripts/bw-exec-resolver.mjs:29`
  - body: The wrapper is the configured trusted exec command, but it invokes `bw` by name and the docs pass PATH into the wrapper. That lets the real secrets-reading binary be selected outside OpenClaw's command path validation, symlink policy, and trustedDirs checks while receiving Bitwarden credential env.
  - confidence: 0.86
