## Summary

Keep this PR open, but do not merge it as-is. Current main still has no Bitwarden/Vaultwarden SecretRef support, and the PR is the focused candidate discussed by maintainers, but the branch introduces a security-sensitive wrapper that selects the real secrets-reading `bw` binary through `PATH`; maintainers also still need to choose the exec-wrapper route versus the plugin-provider seam being tracked in #72548.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef documentation, a protocol-v1 Bitwarden CLI exec resolver script with TypeScript declarations, and tests for parsing, field extraction, grouping, and child-process resolution.

## Best Possible Solution

Land Bitwarden/Vaultwarden support only after the route is explicit: preferably as a secret-provider plugin once #72548 provides the seam, or as a hardened bundled exec-wrapper integration that requires an absolute validated Bitwarden binary path, keeps error redaction, documents trusted installation paths, and coordinates with #29110.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. For the blocking finding, yes: the PR docs pass `PATH` into the wrapper and the wrapper calls `execFile("bw")`, so putting a different `bw` earlier in `PATH` exercises the unsafe binary-selection path.

## Solution Assessment

No, not as currently proposed. The exec-wrapper route is plausible, but it must not resolve the secrets-reading Bitwarden CLI through unvalidated `PATH`, and maintainers still need to decide whether the plugin seam is the narrower long-term home.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Resolve the Bitwarden CLI without PATH search:** `scripts/bw-exec-resolver.mjs:29`
  - body: The configured exec provider validates only this wrapper script, but the wrapper then calls `execFile("bw", ...)` while the docs pass `PATH` alongside Bitwarden credential env. A writable or overridden PATH can make OpenClaw run a different `bw` binary with the Bitwarden session/client credentials, bypassing the current exec-provider path hardening. Require an absolute validated Bitwarden binary path, such as a `BW_BIN` override checked against trusted dirs, before spawning it.
  - confidence: 0.86
