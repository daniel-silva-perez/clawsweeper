## Summary

Keep this PR open. Current main still does not implement the intended fix: sensitive-looking keys configured only in `skills.entries.*.env` are still blocked unless the skill metadata declares the env var through `primaryEnv` or `requires.env`. The PR remains a useful candidate for #31583, but it still needs maintainer review around env scoping before merge.

## Best Possible Solution

Keep this PR open as an implementation candidate for #31583. The best revision is a focused fix that allows explicitly configured sensitive-pattern keys only for the eligible skill scope active in the run, preserves the hard dangerous-host-env blocks, rejects or isolates duplicate sensitive env names, and adds regression coverage for `GOG_KEYRING_PASSWORD`, `CUSTOM_SERVICE_TOKEN`, disabled/unselected skills, and duplicate configured keys.

## Risks / Open Questions

- Leaving this PR unmerged keeps #31583 unresolved for skills like `gog` that need a sensitive-pattern env var from `skills.entries.*.env`.
- Merging the current approach without tighter scoping could inject sensitive env vars for disabled, unselected, or prompt-ineligible skills into the run environment.
- Duplicate sensitive env names still appear to share process-wide first-wins state, so two skills configured with the same `*_TOKEN` or `*_PASSWORD` key could expose the wrong credential to subprocesses.
- The listed PR files do not add workflows, dependencies, lockfile changes, release scripts, or package metadata, but they do touch a security-sensitive host env injection path.
