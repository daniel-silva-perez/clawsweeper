## Summary

Keep open. Current main still has the underlying skills env bug tracked by #31583, but this PR is not safe to merge as-is because it broadens the sensitive env allowlist before the run has narrowed env injection to active/eligible skills and before duplicate sensitive env names are handled. The branch appears to address the earlier circular-import review concern by extracting active env state, but the remaining secret-scoping issues are merge-blocking.

## What This Changes

The PR widens skill env sanitization so sensitive-pattern keys from `skills.entries.*.env` pass through, adds active skill env state plumbing for eligibility isolation, and adds skills env regression tests.

## Best Possible Solution

Keep this PR as an implementation candidate for #31583, but revise it so explicitly configured sensitive-pattern env keys are admitted only for skills that are active/eligible for the run or snapshot, hard dangerous host env names stay blocked, duplicate sensitive env conflicts are rejected or isolated, and regression tests cover undeclared configured secrets, disabled/unselected skills, duplicate keys, and host exec inheritance.

## Reproduction Assessment

Yes. The source path is enough to reproduce: configure a skill env key like `GOG_KEYRING_PASSWORD` that is not declared in skill metadata, then `sanitizeEnvVars` blocks the sensitive pattern and `allowedSensitiveKeys` does not re-allow it on current main. The PR findings can be reproduced with two configured skills where one is disabled/unselected or where both configure the same sensitive env key.

## Solution Assessment

No. Allowing user-configured sensitive keys is the right direction, but the proposed implementation widens the bypass at the sanitizer level while env application still happens before eligible-skill scoping and while duplicate sensitive keys still share process-wide first-wins state. The narrower maintainable fix is to align env injection with the same active/eligible skill set used for the run and define conflict handling before admitting arbitrary sensitive names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate sensitive env overrides to included skills:** `src/agents/skills/env-overrides.ts:172`
  - body: Adding every configured env key to the sensitive allowlist means `*_TOKEN` and `*_PASSWORD` values from disabled, allowlist-blocked, or otherwise unselected skills can now pass sanitization. `applySkillEnvOverrides()` still runs before the run is narrowed to eligible skills, so those secrets can enter `process.env` and be inherited by exec subprocesses even when the skill is not part of the run. Scope the new bypass to the active/eligible skill set before injecting secrets.
  - confidence: 0.86
- **[P2] Reject duplicate sensitive env key conflicts:** `src/agents/skills/env-overrides.ts:172`
  - body: Once generic sensitive names such as `GITHUB_TOKEN` or `CUSTOM_SERVICE_TOKEN` are allowed, two configured skills can supply different values for the same key. The existing active env state keeps the first value and increments a count, so later subprocesses silently inherit the first credential. Reject conflicting duplicate sensitive keys or isolate env values by the selected skill/tool scope.
  - confidence: 0.78
