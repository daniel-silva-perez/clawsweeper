## Summary

Keep this PR open. Current main still blocks sensitive-pattern keys supplied only through `skills.entries.*.env`, so the PR’s intended fix for the linked exec/skill env bug is not implemented. The branch remains security-sensitive and needs maintainer review before merge.

## Best Possible Solution

Keep this PR open as a useful implementation candidate for #31583. The best path is a focused revision that allows explicitly configured sensitive-pattern keys only for the skill scope that is active for the run, preserves the hard host-env blocks, handles duplicate sensitive env keys deliberately, and adds regression tests for `GOG_KEYRING_PASSWORD`, `CUSTOM_SERVICE_TOKEN`, disabled/unselected skills, and duplicate configured env names.

## Risks / Open Questions

- Merging the current branch without additional scoping could allow disabled, unselected, or otherwise prompt-ineligible skill env blocks to inject sensitive-looking secrets into the run environment if they are loaded before eligibility filtering is finalized.
- Duplicate sensitive env names remain process-wide first-wins state today, so two skills configured with the same `*_TOKEN` or `*_PASSWORD` key could expose the wrong credential to a subprocess unless the fix isolates or rejects that case.
- Leaving the PR unmerged leaves the linked `GOG_KEYRING_PASSWORD` / exec subprocess issue unresolved on current main.
