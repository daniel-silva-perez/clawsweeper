## Summary

Keep this PR open. Current main still does not implement the PR's intended fix: sensitive-pattern keys supplied only through `skills.entries.*.env` are still blocked unless the skill metadata declares them through `primaryEnv` or `requires.env`. The PR remains useful for #31583, but the security review concerns around selected-skill scoping and duplicate sensitive keys still need maintainer handling before merge.

## Best Possible Solution

Keep this PR open as a useful implementation candidate for #31583. The best revision is a focused fix that allows explicitly configured sensitive-pattern keys only for the eligible skill scope active in the run, preserves the hard dangerous-host-env blocks, deliberately rejects or isolates duplicate sensitive env names, and adds regression tests for `GOG_KEYRING_PASSWORD`, `CUSTOM_SERVICE_TOKEN`, disabled/unselected skills, and duplicate configured env keys.

## Risks / Open Questions

- Leaving the PR unmerged leaves #31583 unresolved for skills like `gog` that need `GOG_KEYRING_PASSWORD` from `skills.entries.*.env`.
- Merging the current branch without tighter scoping could let disabled, unselected, or prompt-ineligible skill env blocks inject sensitive-looking secrets into the run environment.
- Duplicate sensitive env names still use process-wide first-wins state, so two configured skills with the same `*_TOKEN` or `*_PASSWORD` key could expose the wrong credential to subprocesses.
