## Summary

Keep open. Current main still blocks sensitive-pattern keys configured only through `skills.entries.*.env` unless the skill metadata declares them through `primaryEnv` or `requires.env`. The PR targets a real remaining bug tracked by #31583, but the submitted host-env injection change still needs maintainer review around active/eligible skill scoping and duplicate sensitive env names before it is safe to merge.

## Best Possible Solution

Keep this PR open as an implementation candidate for #31583. The best revision is a focused skills env fix that allows explicitly configured sensitive-pattern keys only for the active/eligible skill scope, preserves the hard dangerous-host-env blocks, rejects or isolates duplicate sensitive env names, and adds regression coverage for `GOG_KEYRING_PASSWORD`, `CUSTOM_SERVICE_TOKEN`, disabled/unselected skills, duplicate keys, and host exec inheritance.

## Risks / Open Questions

- Leaving the PR unmerged keeps #31583 unresolved for skills like `gog` that need undeclared sensitive-pattern env vars from `skills.entries.*.env`.
- Merging the current approach without tighter active/eligible skill scoping could inject sensitive env vars for disabled, unselected, or prompt-ineligible skills into the run environment.
- Duplicate sensitive env names still use process-wide first-wins state in `acquireActiveSkillEnvKey`, so two configured skills sharing a `*_TOKEN` or `*_PASSWORD` key could expose the wrong credential to subprocesses.
