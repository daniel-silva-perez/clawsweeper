## Summary

Keep this PR open. Current main still uses message-id-only Feishu dedupe at the early receive and final persistence gates, while the PR targets those gates with a Feishu-local media key and adds focused regression coverage; I found no blocking correctness or security issue in the patch.

## What This Changes

The PR changes Feishu inbound and final message dedupe to include media identifiers from `file_key` or `image_key` for media messages and adds an audio regression test.

## Best Possible Solution

Land the Feishu-owned media-aware dedupe fix after normal maintainer review, preserving message-id-only behavior for non-media events and true retries while allowing distinct media attachments with reused Feishu ids to reach the agent.

## Reproduction Assessment

Yes. Current main has a high-confidence trace reproduction: two Feishu audio events for the same account with the same `message_id` and different `file_key` values collide in the message-id-only early claim and final persistent dedupe paths.

## Solution Assessment

Yes. The PR is the narrow maintainable direction because it keeps the behavior inside the Feishu plugin and reuses Feishu media identity already parsed by `parseMediaKeys`; sharing the helper later could reduce duplication, but that is not a blocking correctness issue.
