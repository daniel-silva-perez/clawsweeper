## Summary

Keep open: current main still has the plausible abort-all Telegram polling behavior, but this branch is not merge-ready because its `bot-core.ts` changes target the older fetch wrapper and need to be rebased onto the current nested `runFetch` timeout/fallback path.

## What This Changes

The PR adds an internal Telegram fetch-abort method allow-list, passes `getupdates` from polling sessions, and adds regression coverage so polling restarts abort `getUpdates` without aborting overlapping `sendMessage` replies.

## Best Possible Solution

Rebase the branch onto current main, apply the method allow-list inside the current `runFetch` abort-forwarding wrapper, keep default all-method shutdown abort behavior for other callers, preserve the timeout fallback and polling timeout-floor changes already on main, and land with focused Telegram regression coverage.

## Reproduction Assessment

Yes for the PR's narrow failure path: current main creates one polling-cycle fetch abort signal and forwards it to every wrapped Telegram API fetch, so an overlapping `getUpdates` and `sendMessage` can both observe the polling restart abort. The broader production symptoms in #72867 still need live/session evidence to prove this is the only cause.

## Solution Assessment

Yes in design: scoping polling-cycle aborts to `getUpdates` is a narrow fix that avoids weakening the watchdog. The submitted branch is not the best merge form until it is rebased onto the current wrapper and preserves the newer timeout retry/floor behavior.
