## Summary

Keep this PR open. Current main still has the Telegram polling restart path abort the same fetch controller used by all wrapped Bot API calls, so the intended fix is not already implemented; however, the PR branch needs to be rebased because its `bot-core.ts` hunk no longer applies cleanly to current main after later wrapper changes.

## What This Changes

The PR adds an internal Telegram `fetchAbortMethods` allow-list, scopes polling-session aborts to `getUpdates`, and adds regression coverage that an overlapping `sendMessage` is not aborted during a polling restart.

## Best Possible Solution

Rebase the branch onto current main, wire the method allow-list into the current nested `runFetch` abort-forwarding wrapper, keep the option internal to Telegram polling, preserve default all-method abort behavior for other callers, and land it with focused regression coverage plus the Telegram changed gate.

## Reproduction Assessment

Yes for the PR's narrow failure: on current main, create a wrapped Telegram client fetch with `fetchAbortSignal`, start overlapping `getUpdates` and `sendMessage` requests, then abort the polling controller; the wrapper forwards that abort to both requests. The broader #72867 production report still needs live/session evidence to prove this is the only cause.

## Solution Assessment

Yes in design: scoping the polling restart abort to `getUpdates` is narrower than weakening the watchdog and matches grammY's method-in-URL fetch contract. It is not merge-ready as submitted until rebased over the current `bot-core.ts` wrapper.
