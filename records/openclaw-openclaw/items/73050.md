## What This Changes

The PR adds an internal Telegram fetch-abort method allow-list, scopes polling-session restart aborts to getUpdates, and adds regression coverage so overlapping sendMessage calls are not aborted during polling recovery.

## Best Possible Solution

Land a rebased version of the narrow Telegram-owned fix: keep fetchAbortMethods internal to the Telegram bot options, apply the allow-list inside the current runFetch abort-forwarding path without weakening request timeouts or fallback retry behavior, pass getupdates from polling sessions, add focused regression coverage, and include the required changelog credit.

Yes for the PR's narrow bug: create a Telegram client fetch wrapper with fetchAbortSignal, start overlapping getUpdates and sendMessage calls, then abort the polling controller; current main forwards that abort to both requests. ## Solution Assessment

Yes in design, but no as submitted. Scoping the polling restart abort to getUpdates is the narrow maintainable fix, but it needs to be rebased into the current bot-core runFetch wrapper and accompanied by the changelog entry before merge.
