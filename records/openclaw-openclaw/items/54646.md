## Summary

Keep this PR open. Current main still has the gateway-rpc timeout validation bug: the shared gateway-rpc option advertises a 30000 ms default, but the runtime still coerces the option with Number(...) and lets invalid values reach the gateway call path, where non-finite timeouts fall back to 10000 ms. The PR remains a plausible focused fix, with useful review follow-up for partial numeric strings, empty values, and cron timeout-seconds side effects.

## Best Possible Solution

Keep this PR open for maintainer review or deliberately supersede it with a coordinated timeout-validation PR. The best implementation is strict positive-integer validation at the gateway-rpc CLI boundary, preserving the omitted-option default at 30000 ms, rejecting explicit empty/whitespace and partial numeric strings, adding focused gateway-rpc regression tests, and auditing any shared parser changes so cron, status, health, devices, channels, gateway, and nodes timeout consumers behave consistently.
