## Summary

Keep this issue open. Current main still persists auto-enable-derived `enabled: true` flags during gateway startup and still treats auto-enable as a doctor repair mutation; the earlier restart-loop hash guard mitigates self-induced reload churn, but it does not implement the requested runtime-only canonicalization.

## What This Changes

The issue asks for plugin auto-enable to remain runtime-only so gateway startup and doctor do not persist inferred legacy `enabled` flags into `openclaw.json`.

## Best Possible Solution

Decide the canonical config contract, then either keep the existing persistent auto-enable behavior intentionally or split auto-enable into a runtime activation overlay plus read-only doctor diagnostics so inferred `enabled` flags are not written by startup or `doctor --fix`.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-backed: remove the reported `enabled` keys while leaving channel/provider config present, then the current startup and doctor paths still call `applyPluginAutoEnable()` and persist the returned config; existing tests also assert that startup writes `enabled: true`.

## Solution Assessment

No. The current main solution only suppresses the first self-write watcher event with a hash guard; it does not solve the issue's requested non-persistent, runtime-only auto-enable behavior.
