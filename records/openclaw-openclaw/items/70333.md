## Summary

Keep this issue open. Current main still persists auto-enable-derived `enabled: true` keys during gateway startup and still lets doctor treat auto-enable as a repair mutation; the already-merged #67557 hash guard mitigates the self-induced restart churn, but it does not implement the requested runtime-only canonical config behavior.

## What This Changes

The issue requests that plugin/channel auto-enable remain an in-memory runtime activation and stop rewriting legacy `enabled` keys into `openclaw.json` during gateway startup or `doctor --fix`.

## Best Possible Solution

Define the canonical config contract first. If runtime-only auto-enable is accepted, keep `openclaw.json` as the authored source, build an effective runtime overlay for gateway/plugin bootstrap, make doctor report auto-enable reasoning without marking it as a repair write, and update the persistence-oriented tests and docs accordingly.

## Reproduction Assessment

Yes. A source-backed reproduction path is clear: remove the reported `enabled` keys while leaving the channel/provider config present, then current startup and doctor paths still call `applyPluginAutoEnable()` and persist or mark the returned config for writing.

## Solution Assessment

No for current main. The hash-guard solution is useful for the restart-loop symptom, but it is not the narrow complete fix for the remaining runtime-only canonicalization request.
