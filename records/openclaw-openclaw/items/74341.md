## Summary

Keep this PR open. Current main still bypasses `supports()` when a plugin harness is forced or session-pinned, so the reported fallback misroute remains real; the proposed fix is useful but needs maintainer review because the Codex provider-id expansion can change the documented auto-mode boundary unless it is gated by `requestedRuntime`.

## What This Changes

This PR validates forced/session-pinned plugin harness selection with `supports()`, falls back to PI or throws on unsupported candidates, widens Codex harness provider ids, and adds selection/Codex harness/status tests plus a changelog entry.

## Best Possible Solution

Land a narrowed harness-selection fix that consults each plugin harness's `supports()` before forced or session-pinned selection, falls back to PI only when the active policy permits it, and reports a clear unsupported-harness error otherwise. Keep the Codex auto-mode support contract explicit, with tests and docs aligned to whichever provider-id set maintainers choose.
