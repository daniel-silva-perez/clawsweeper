## Summary

Keep open. The reported forced-harness bug is real on current main and the PR fixes the right selector path, but the Codex harness default-provider change introduces a blocking regression by making auto mode claim OpenAI/OpenAI-Codex routes that current docs and tests explicitly keep on PI unless Codex is forced.

## What This Changes

The PR adds a `supports({ provider, modelId, requestedRuntime })` check before selecting a forced or session-pinned plugin harness, adds PI fallback/error behavior for unsupported candidates, broadens the Codex harness default provider ids, updates tests, and adds a changelog entry.

## Best Possible Solution

Land the selector-side support check, but keep Codex auto mode conservative: accept `openai` and `openai-codex` only for explicit `requestedRuntime: "codex"` or an intentional documented contract change, while continuing to reject unsupported providers such as `9router` before they reach Codex app-server.

## Reproduction Assessment

Yes. Current main reproduces the reported selector bug by reading the forced branch at `src/agents/harness/selection.ts:108`, and the PR regression is statically reproducible because the widened provider set makes `supports({ provider: "openai-codex", requestedRuntime: "auto" })` true while the existing test expects false.

## Solution Assessment

No as written. The forced-branch `supports()` check is the right narrow fix, but broadening the Codex default provider set without checking `requestedRuntime` is not the best solution because it changes auto-mode routing outside the stated bug.
