## Summary

Keep this PR open for maintainer review. Current main still has the forced plugin harness path returning the registered harness without consulting `supports()`, so the underlying bug is not implemented on main. The PR is useful, but its Codex provider-id widening likely changes the documented auto-mode boundary unless it is narrowed or made conditional on `requestedRuntime`.

## What This Changes

The PR validates forced and session-pinned plugin harness selection with `supports()`, widens the Codex harness default provider ids, updates selection/status tests, adds Codex harness tests, and adds a changelog entry.

## Best Possible Solution

Review and land a narrowed fix that calls `supports({ provider, modelId, requestedRuntime })` before selecting forced or session-pinned plugin harnesses, falls back to PI only when policy allows, and throws a clear unsupported-harness error otherwise. Preserve the documented auto-mode boundary by keeping Codex provider support conditional on explicit `requestedRuntime: "codex"` or updating docs/tests if maintainers intentionally want a broader contract.
