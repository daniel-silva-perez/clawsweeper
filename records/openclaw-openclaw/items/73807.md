## Summary

Keep this PR open. Current main still lacks the JWT-expiry short-circuit for stale OpenAI Codex OAuth `expires` values, while the existing generic OAuth locks and `refresh_token_reused` recovery only cover the refresh race after refresh is attempted. I found no discrete blocking correctness or security finding in the proposed direction, but it needs normal maintainer review on top of the newer Codex auth changes.

## What This Changes

The PR adds an OpenAI Codex OAuth refresh short-circuit that trusts a still-valid cached access-token JWT `exp`, caps the corrected `expires` value at 30 minutes, and adds focused provider tests for JWT, opaque-token, and malformed-token cases.

## Best Possible Solution

Land a rebased provider-local fix that corrects stale OpenAI Codex OAuth `expires` values from the access-token JWT only when the JWT expiry is beyond `DEFAULT_OAUTH_REFRESH_MARGIN_MS`, keeps the generic OAuth manager locks and reused-token recovery unchanged, and records the user-facing reliability fix in the changelog.

## Reproduction Assessment

Yes. A deterministic unit path is enough: seed an `openai-codex` OAuth credential with stale `expires` and an access JWT whose `exp` is still outside the refresh margin; current main proceeds to HTTP refresh instead of returning the cached access token.

## Solution Assessment

Yes, with rebase and changelog follow-through. The proposed provider-local short-circuit is narrower than moving Codex JWT semantics into the generic OAuth manager, and it complements rather than replaces the existing cross-agent refresh serialization.
