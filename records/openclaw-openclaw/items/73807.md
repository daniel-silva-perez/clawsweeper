## Summary

Keep open. Current main still has the reported stale-`expires` refresh gap, and the PR is aimed at the right provider-owned boundary, but it needs a small repair over current main because the new short-circuit would bypass current JWT-derived Codex account metadata persistence and it lacks the required changelog entry.

## What This Changes

The branch updates the OpenAI Codex provider to skip HTTP OAuth refresh when the cached access-token JWT expiry is still beyond the refresh margin, caps the synthetic expiry, and adds unit tests for JWT, opaque-token, malformed-token, and boundary cases.

## Best Possible Solution

Land a rebased provider-local fix that decodes the cached Codex access-token JWT before HTTP refresh, uses `DEFAULT_OAUTH_REFRESH_MARGIN_MS`, caps synthetic rechecks, preserves JWT-derived `email`, `accountId`, and `chatgptPlanType` in the returned credential, adds focused regression tests, and records the user-facing reliability fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Seed an `openai-codex` OAuth credential with stale stored `expires` and an access-token JWT whose `exp` is still beyond the five-minute refresh margin; current main routes into provider refresh instead of reusing and correcting the cached access token.

## Solution Assessment

No, not as-is on current main. The provider-local short-circuit is the right maintainable direction, but it should also merge the current JWT identity metadata before returning and persisting the corrected credential.
