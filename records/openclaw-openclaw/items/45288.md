## Summary

Keep this PR open, but do not merge it as-is. Current main still accepts arbitrary `ws://`/`wss://` gateway-status targets, so the hardening target is real; however the patch leaves IPv4-mapped IPv6 metadata URLs unblocked and also rejects documented `.local` gateway targets by reusing the generic HTTP SSRF hostname block.

## What This Changes

The PR adds hostname/IP screening in `gateway-status` URL normalization and a new security test file for blocked metadata/internal hostnames plus allowed local/private gateway targets.

## Best Possible Solution

Keep the hardening effort, but implement a gateway-specific URL policy: parse the URL, preserve legitimate loopback/private-LAN and documented `.local` gateway targets, reject wildcard/link-local/cloud-metadata targets through the shared IP parsing path including IPv4-mapped and transition IPv6 forms, and cover both explicit and `gateway.remote.url` targets in tests.

## Reproduction Assessment

Yes. Current main can be checked by reading `normalizeWsUrl()`, and the PR bypass can be reproduced with Node URL normalization of `ws://[::ffff:169.254.169.254]:80`, which becomes `::ffff:a9fe:a9fe` and misses the added dotted `169.254.` prefix check.

## Solution Assessment

No. The direction is useful, but the narrow string-prefix implementation is not the best fix for SSRF-sensitive URL validation; it should reuse the shared IP/SSRF parsing helpers with a context-specific allowlist for legitimate gateway endpoints.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new check only blocks dotted-decimal hosts that equal `0.0.0.0` or start with `169.254.`. `ws://[::ffff:169.254.169.254]:80` is normalized by `URL` to `::ffff:a9fe:a9fe`, so it bypasses this branch while still reaching the link-local metadata address. Use the shared IP parser/SSRF helper and add IPv4-mapped/hex IPv6 coverage before merging.
  - confidence: 0.94
- **[P2] Preserve documented .local gateway targets:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: Calling the generic `isBlockedHostname()` helper rejects `gateway.local`, but OpenClaw docs support cleartext `ws://` Gateway URLs on `.local` LAN hosts and Bonjour discovery uses `local.`. This would make `gateway-status --url ws://gateway.local:18789` and matching configured remote URLs disappear instead of being probed, so the gateway-status policy needs a context-specific allowlist.
  - confidence: 0.84
