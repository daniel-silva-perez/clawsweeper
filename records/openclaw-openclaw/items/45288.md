## Summary

Keep open. The PR addresses a real gateway-status hardening gap, but the proposed mitigation remains bypassable for IPv4-mapped IPv6 metadata addresses and also blocks documented `.local` gateway routes, so it should not merge as-is.

## What This Changes

The PR adds hostname/IP screening to gateway-status WebSocket URL normalization and adds a new security test file covering metadata, internal, wildcard, localhost, LAN, public-hostname, and protocol cases.

## Best Possible Solution

Keep the hardening effort, but replace the string-prefix checks with a gateway-specific URL policy that parses IP literals through the shared IP/SSRF helpers, rejects wildcard and metadata/link-local targets including IPv4-mapped and transition IPv6 forms, preserves documented `.local` and private-LAN gateway routes under the existing transport-security rules, and covers both explicit URLs and `gateway.remote.url`.

## Reproduction Assessment

Yes. Current-main behavior is reproducible by source inspection of `resolveTargets()` and `probeGateway()`, and the PR regression is reproducible with a focused Node URL parsing check for IPv4-mapped metadata literals.

## Solution Assessment

No. The PR is directionally useful, but string-prefix hostname checks plus the generic blocked-hostname helper are not the narrowest maintainable gateway policy; reusing shared IP classification with gateway-specific `.local`/LAN semantics is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new block only checks dotted `169.254.` strings and `0.0.0.0`. `URL` canonicalizes `ws://[::ffff:169.254.169.254]:80` to an IPv6 hostname like `[::ffff:a9fe:a9fe]`, so the metadata endpoint bypasses this mitigation. Use the shared IP parser/SSRF helper and add IPv4-mapped and hex IPv6 cases before merging.
  - confidence: 0.94
- **[P2] Preserve documented .local gateway targets:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: `isBlockedHostname()` rejects `.local` for both `ws://` and `wss://`, but OpenClaw documents `.local` gateway routes for private LAN/Bonjour flows with TLS or explicit private-network opt-in. This would drop legitimate targets such as `wss://gateway.local:18789` instead of probing them.
  - confidence: 0.84
