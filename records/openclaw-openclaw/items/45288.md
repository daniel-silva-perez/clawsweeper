## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks host/IP validation for gateway-status probe URLs, so the contribution is directionally useful, but the proposed string-prefix filter leaves the metadata range reachable through IPv4-mapped IPv6 forms that the shared SSRF/IP helpers already know how to classify.

## What This Changes

The PR adds gateway-status WebSocket URL hostname checks and a new security test file covering metadata, internal/local hostname, wildcard, localhost, LAN, public hostname, and protocol cases.

## Best Possible Solution

Revise the gateway-status URL classifier to use shared IP parsing/SSRF primitives for canonical and embedded IP literals while preserving intentional localhost and private LAN gateway probes, then add regression coverage for IPv4-mapped/hex IPv6 link-local metadata addresses and any wildcard IPv6 forms the policy should reject.

## Reproduction Assessment

Yes. Current main accepts the reported explicit metadata-style URL because `normalizeWsUrl()` only checks the `ws://` or `wss://` prefix, and the PR's residual bypass is reproducible by Node normalizing `ws://[::ffff:169.254.169.254]:80` to `[::ffff:a9fe:a9fe]`.

## Solution Assessment

No. The PR is a useful start, but the best fix is a gateway-status-specific host/IP policy built on the shared IP normalization helpers rather than ad hoc string prefix checks that miss embedded IPv4 forms.
