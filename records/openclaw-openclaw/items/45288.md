## Summary

Keep open. The PR targets a real gateway-status hardening gap on current main, but the proposed mitigation is incomplete because IPv4-mapped and hex IPv6 forms of metadata/link-local addresses bypass the new string checks.

## What This Changes

The PR adds gateway-status WebSocket hostname filtering plus a new security test file for metadata, link-local, wildcard, internal/local, localhost, loopback, LAN, public-hostname, and non-WebSocket URL cases.

## Best Possible Solution

Land a current-main gateway-status hardening that parses each target URL once, uses shared hostname/IP classification or equivalent canonical IP-literal handling for mapped IPv6 and legacy forms, preserves the intended loopback/LAN diagnostic behavior, and adds regression coverage for mapped/hex IPv6 metadata URLs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only rejects exact `0.0.0.0`, dotted `169.254.*` hosts, and blocked hostname suffixes. Node canonicalizes `ws://[::ffff:169.254.169.254]:80` to `[::ffff:a9fe:a9fe]`, which bypasses those checks and is not a blocked hostname, so the metadata endpoint remains accepted. Use the shared SSRF/IP classifier or equivalent canonical IP-literal handling and add mapped/hex IPv6 cases.
  - confidence: 0.94
