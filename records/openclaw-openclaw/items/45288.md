## Summary

Keep this PR open. Current main still accepts explicit and configured gateway-status WebSocket targets after only trimming and ws/wss prefix checks, so the reported surface is real, but the proposed mitigation is incomplete because its string checks miss IPv4-mapped IPv6 forms of the metadata range that Node URL parsing canonicalizes away from the dotted 169.254 prefix.

## What This Changes

This PR adds gateway-status WebSocket hostname validation plus a new security test file for metadata, link-local, wildcard, internal/local, localhost, loopback, LAN, public hostname, and non-WebSocket URL cases.

## Best Possible Solution

Land a current-main gateway-status hardening that parses each target URL once, delegates hostname/IP threat classification to the shared SSRF utilities where possible, explicitly preserves the supported loopback/LAN gateway probing policy, and adds mapped/hex IPv6 metadata regression coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only rejects exact 0.0.0.0, dotted 169.254.* hosts, and blocked hostname suffixes. Node canonicalizes ws://[::ffff:169.254.169.254]:80 to [::ffff:a9fe:a9fe], which bypasses those checks and is not a blocked hostname, so the metadata endpoint remains accepted. Use the shared SSRF IP classifier or equivalent IP-literal handling and add mapped/hex IPv6 cases.
  - confidence: 0.94
