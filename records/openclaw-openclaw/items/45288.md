## Summary

Keep this PR open. The hardening direction is valid and current main still lacks gateway-status hostname/IP filtering, but the proposed patch is incomplete for its stated SSRF threat model and would reject documented `.local` gateway targets.

## What This Changes

The PR adds hostname/IP screening to `gateway-status` WebSocket URL normalization and adds security tests for metadata/internal hosts, wildcard addresses, and allowed local/private gateway targets.

## Best Possible Solution

Keep the hardening effort, but implement a gateway-specific URL policy: parse the URL, preserve legitimate loopback/private-LAN and documented `.local` gateway targets, reject wildcard/link-local/cloud-metadata targets through shared IP parsing that covers IPv4-mapped and transition IPv6 forms, and test both explicit URLs and `gateway.remote.url`.

## Reproduction Assessment

Yes. The PR bypass is reproducible with a focused Node URL parsing check for `ws://[::ffff:169.254.169.254]:80`, and current-main behavior is verified by reading `normalizeWsUrl()` in `src/commands/gateway-status/helpers.ts`.

## Solution Assessment

No. The PR is directionally useful, but string-prefix hostname checks are not the best maintainable fix for SSRF-sensitive URL validation; a context-specific gateway policy should reuse the shared SSRF/IP parsing helpers with explicit gateway allowlists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new check only blocks dotted-decimal hosts that equal `0.0.0.0` or start with `169.254.`. `ws://[::ffff:169.254.169.254]:80` is normalized by `URL` to an IPv6 hostname (`::ffff:a9fe:a9fe`), so it bypasses this branch while still representing the link-local metadata address. Use the shared IP parser/SSRF helper and add IPv4-mapped/hex IPv6 coverage before merging.
  - confidence: 0.94
- **[P2] Preserve documented .local gateway targets:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: Calling the generic `isBlockedHostname()` helper rejects `gateway.local`, but OpenClaw docs support cleartext `ws://` Gateway URLs on private LAN and `.local` hosts. This would make `gateway-status --url ws://gateway.local:18789` and matching configured remote URLs disappear instead of being probed, so the gateway-status policy needs a context-specific allowlist.
  - confidence: 0.84
