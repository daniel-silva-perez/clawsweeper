## Summary

Keep open. Current main still accepts gateway-status explicit and configured WebSocket targets with only trim/scheme checks, so the reported security gap is not implemented on main. The PR is directionally useful, but its mitigation is incomplete because IPv4-mapped IPv6 metadata addresses bypass the new string checks called out in the review discussion.

## What This Changes

This PR adds hostname validation to gateway-status WebSocket URL normalization and a new security test file for metadata, link-local, wildcard, internal/local, localhost, loopback, LAN, public hostname, and non-WebSocket URL cases.

## Best Possible Solution

Land a current-main gateway-status hardening that parses target URLs once, delegates hostname/IP threat classification to the shared SSRF utilities where possible, explicitly preserves the supported loopback/LAN gateway probing policy, and adds mapped/hex IPv6 metadata regression coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only rejects exact `0.0.0.0`, dotted `169.254.*`, and blocked hostname suffixes. Node canonicalizes `ws://[::ffff:169.254.169.254]:80` to `[::ffff:a9fe:a9fe]`, which bypasses those checks and is not a blocked hostname, so the metadata endpoint remains accepted. Use the shared SSRF IP classifier or equivalent IP-literal handling and add mapped/hex IPv6 cases.
  - confidence: 0.94

## Security Review

Status: needs_attention

Concerns:

- **[high] IPv4-mapped metadata URLs still bypass:** `src/commands/gateway-status/helpers.ts:103`
