## Summary

Keep this PR open for maintainer follow-up. Current main still lacks the central session-health visibility and remediation surfaces, while the PR branch remains too broad and has concrete cleanup-safety defects around transcript identity, archive retention, stale-archive age checks, and agent scope.

## What This Changes

The PR adds session-health collection and derivation, gateway health and Control UI visibility, sessions cleanup remediation dry-run/execution flags, docs/tests, and pre-commit hook gate changes.

## Best Possible Solution

Split the read-only session-health collector/cache/health/UI visibility from the manual remediation executor, fix the cleanup contract bugs before enabling deletion, and move the pre-commit gate behavior into a separate infra PR if maintainers want that workflow change.

## Reproduction Assessment

Yes. The current-main gap is reproducible by static inspection of the cleanup command, CLI registration, health types, gateway health refresh, scheduled services, and docs; the PR defects are reproducible by comparing the branch diff against current sessionFile and archive-retention contracts. No destructive cleanup command was executed because this was a read-only review.

## Solution Assessment

No. The visibility direction is plausible, but this combined PR is not the safest implementation as written because the remediation executor can act on the wrong transcript identity, wrong archive age source, wrong reset retention policy, or wrong agent scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Resolve orphans from stored sessionFile paths:** `src/infra/session-health-file-discovery.ts:115-117`
  - body: Session entries can persist a `sessionFile` that is not `${sessionId}.jsonl`, but this code only compares disk basenames to `sessionId`. A live transcript stored under a custom or migrated filename will be classified as orphaned and `archive-orphan-transcripts` can rename it out from under the active session.
  - confidence: 0.9
- **[P1] Use archive timestamps for stale deleted files:** `src/infra/session-health-file-discovery.ts:141`
  - body: The executor archives orphan transcripts with a fresh `.deleted.<timestamp>` name, but stale discovery checks the file mtime. Because `fs.rename` preserves the old transcript mtime, a tier-1 run can archive an old orphan and then immediately delete the newly archived file in the stale-deleted action, bypassing the archive-first safety model.
  - confidence: 0.88
- **[P1] Honor resetArchiveRetention for reset archives:** `src/infra/session-health-remediation-executor.ts:526-527`
  - body: Reset archive purging uses `snapshot.maintenance.pruneAfterMs`, but OpenClaw has a separate `session.maintenance.resetArchiveRetention` setting, including `false` to disable reset cleanup. This path can therefore delete `.reset` recovery artifacts earlier than configured or when cleanup is disabled.
  - confidence: 0.9
- **[P2] Execute remediation in the planned agent scope:** `src/infra/session-health-remediation-executor.ts:213-221`
  - body: The plan is built from `collectSessionHealth(cfg)`, which aggregates all configured agents, but execution always resolves and mutates only the default agent store. An action caused by a non-default agent can validate and then report as skipped/resolved while leaving the real target untouched, which makes operator execution reports misleading.
  - confidence: 0.84
- **[P2] Do not make temp artifacts reconciliation-critical:** `src/infra/session-health-collector.ts:337-338`
  - body: The collector sets `reconciliationRecommended` when any orphaned temp file exists, and `deriveIndexHealth` turns that flag into a critical index-health state. A single crashed atomic-write `.tmp` file should follow the orphan-temp warning path, not force the whole index health indicator to critical.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds local session-file deletion paths; no new third-party supply-chain source was found, but the cleanup executor can violate retention policy and delete recovery archives.

Concerns:

- **[medium] Remediation can delete reset archives against policy:** `src/infra/session-health-remediation-executor.ts:526`
  - body: The new executor purges reset archives using the general prune window instead of resetArchiveRetention, including configurations where reset cleanup is disabled, which risks deleting recovery data contrary to local policy.
  - confidence: 0.9
- **[medium] Archive-first cleanup can immediately delete fresh archives:** `src/infra/session-health-file-discovery.ts:141`
  - body: Stale archive discovery uses mtime rather than the archive timestamp in the filename, so a renamed orphan with old mtime can be purged in the same execution batch that archived it.
  - confidence: 0.86
