## Summary

Keep this PR open. Current main has shipped a broader write-ahead outbound delivery queue and a WhatsApp reconnect drain path, which addresses much of the persistent-loss cluster discussed in the comments. However, current main does not implement this PR's distinct configurable WhatsApp send retry surface: there is no `channels.whatsapp.retry` type/schema/docs entry, no WhatsApp outbound retry helper, and the direct WhatsApp send test still asserts that transient outbound send failures are not retried. The April 25 maintainer comment also explicitly treated configurable send retry as adjacent rather than identical to the queue work and said it still needs fresh review against current main.

## Best Possible Solution

Keep this PR open for maintainer review, but ask for a rebase against current main and a narrower design review that decides whether WhatsApp still needs configurable immediate send retry on top of the shipped write-ahead queue and reconnect drain. If kept, the implementation should add the config/type/schema/docs surface, wrap the actual production send paths after deps resolution, preserve per-account/default-account semantics, and include focused tests for retryable versus non-retryable send failures and duplicate-delivery safeguards.
