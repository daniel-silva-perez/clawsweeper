## Summary

Keep this PR open. Current main still lacks the proposed user-facing WhatsApp `retry` config surface, so the branch is not fully implemented on main, but the shipped write-ahead delivery queue and WhatsApp reconnect drain mean the retry knob is now adjacent to current behavior and needs maintainer product/architecture review before merge. The current PR diff also still has a concrete P2 correctness issue: its retry classifier misses the message-only reconnect-window failures described in the linked reports.

## What This Changes

The PR adds a WhatsApp `retry` config surface plus exponential-backoff retry wrapping for text, media, and poll sends, with config schema/generated-doc updates and retry-policy helper tests.

## Best Possible Solution

Decide whether configurable immediate WhatsApp send retry is still desired as a complement to the durable queue. If yes, rebase the PR onto current main, align the retry classifier with queue/reconnect-drain semantics, and document which failures retry immediately versus defer to queued recovery.
