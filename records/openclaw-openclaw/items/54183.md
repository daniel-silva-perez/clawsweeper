## Summary

Keep this PR open for maintainer review. Current main still lacks the proposed configurable WhatsApp retry surface, so the PR is not implemented or obsolete, but the shipped durable delivery queue/reconnect drain changes make the product scope adjacent rather than straightforward, and the PR still has a concrete classifier gap against the reconnect-window errors discussed in the thread.

## What This Changes

The PR adds a WhatsApp `retry` config surface and wraps WhatsApp text, media, and poll sends with a shared exponential-backoff retry helper, plus config schema, generated docs, and tests.

## Best Possible Solution

Decide whether configurable immediate WhatsApp send retry is still desired as a complement to the durable queue. If yes, rebase the PR onto current main, align the retry classifier with queue/reconnect-drain semantics, and document which failures retry immediately versus defer to queued recovery.

## Reproduction Assessment

Yes. A focused unit check around the PR helper with `new Error("Connection Closed")` or `new Error("No active WhatsApp Web listener")` would show the classifier returning false even though those are the linked reconnect-window failures the PR claims to cover.

## Solution Assessment

No, not as-is. The direction may still be useful, but the current patch is not the best merge shape until the retry classifier and documentation are reconciled with the shipped write-ahead queue and reconnect drain.
