## Summary

Keep open. Current main has strong in-memory replay repair for strict providers and a pre-load session-file repair hook, but the on-disk repair path does not apply the tool_use/toolResult pairing repair requested here, so orphaned toolResult corruption can still persist in the raw JSONL across restarts.

## What This Changes

The issue requests repairing orphaned or displaced tool_result/tool_use transcript corruption when session files are loaded and rewriting the repaired session state so corruption does not survive restarts.

## Best Possible Solution

Extend the existing pre-load session-file repair path with a storage-aware equivalent of repairToolUseResultPairing, backing up and rewriting the JSONL only when repaired message entries actually change, with regression coverage for orphaned, displaced, duplicate, missing, and aborted/error tool-result cases.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a syntactically valid session JSONL containing a user turn followed by an unmatched toolResult will pass repairSessionFileIfNeeded unchanged, while the same message shape is dropped by sanitizeToolUseResultPairing only later in replay memory.

## Solution Assessment

No, current main is not the complete solution for this issue. The narrow maintainable fix is to reuse the existing pairing semantics in the pre-load file repair layer rather than adding another provider-specific replay workaround.
