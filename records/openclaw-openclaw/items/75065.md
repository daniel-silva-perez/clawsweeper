## Summary

Keep open. Current main still lacks stable cron create ids and duplicate-id rejection, and this PR implements the intended behavior, but the proposed generated Swift model change introduces a source-compatibility break that should be fixed before merge.

## What This Changes

The PR adds an optional cron add `id` field across cron create types, gateway/CLI/tool normalization, docs, generated Swift models, and tests, plus duplicate-id rejection in `cron.add`.

## Best Possible Solution

Keep the additive cron stable-id behavior, but make the generated Swift public API source-compatible before merge by teaching the generator to default optional initializer parameters to `nil` or by preserving an overload, then regenerate both Swift model copies and the protocol artifacts.

## Reproduction Assessment

Yes. The compatibility issue is reproducible by source inspection: the PR adds a required `id:` argument to the public generated `CronAddParams` initializer, and the current generator emits optional properties without default values.

## Solution Assessment

No. The TypeScript cron service and gateway direction is narrow and maintainable, but the proposed generated Swift API shape is not the best final form because an optional wire field should remain source-compatible for Swift callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep Swift initializer source-compatible:** `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift:3619`
  - body: Adding `id: String?` as a non-defaulted parameter to the public generated `CronAddParams` initializer makes existing Swift callers of `CronAddParams(name:...)` fail to compile, even though the wire field is optional and the PR says this is not an API break. Please give the generated parameter a `nil` default or keep a compatibility overload, and keep the macOS generated copy in sync.
  - confidence: 0.88
