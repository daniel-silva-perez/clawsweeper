## Summary

Keep open. The report is fresh, corroborated, and still broader than the partial mitigations currently on main: Feishu streaming-card fallback, plugin/runtime-deps setup, model resolution, and embedded-run startup remain on the reported reply path, and there is no end-to-end timing proof that current main restores the 4.22/4.23 latency budget.

## What This Changes

The issue reports a 4.22/4.23 to 4.26/4.29 regression where Feishu/Lark reactions and replies become seconds to over a minute slower under the same configuration.

## Best Possible Solution

Keep this as the broad upgrade-regression tracker until maintainers collect live stage timings on the same Feishu/Lark setup and then split the dominant current bottleneck into a narrow PR, while preserving already-landed mitigations for runtime-deps, Active Memory timeout grace, and embedded-run diagnostics.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a same-host, same-config Feishu/Lark timing matrix across 4.22/4.23, 4.26, v2026.4.29, and current main with reaction, first visible reply, final reply, streaming-card start/fallback, runtime-deps, plugin-load, model-resolution, and embedded-run startup timings; this read-only review did not execute that live setup.

## Solution Assessment

No. A broad revert based on bundle/file size is not the best fix; the maintainable route is measured, layer-specific repair for the dominant current bottleneck, with Feishu streaming fallback and embedded-run/runtime-deps setup as first profiling targets.
