## Summary

Keep open. Current main still lacks the requested `tools.fs.allowedRoots` config, and this PR is the active implementation attempt, but the branch has blocking filesystem-boundary regressions against current main.

## What This Changes

This PR adds `tools.fs.allowedRoots` and threads extra filesystem roots through config, sandbox path checks, read/write/edit tools, media loading, `apply_patch`, and regression tests.

## Best Possible Solution

Land a corrected, rebased implementation that adds `allowedRoots` through one canonical filesystem policy path, keeps pinned root-contained helpers for all workspace-only mutations, resolves sandbox mount paths before root matching, and keeps config/help/docs/tests aligned.

## Reproduction Assessment

Yes. Source comparison against current main and raw PR head gives a high-confidence reproduction path: main lacks the config, while PR head shows the unpinned delete, pre-bridge allowed-root check, and changed apply_patch default at concrete lines.

## Solution Assessment

No. The feature direction is maintainable, but this branch should not merge until it restores pinned deletion, performs allowed-root checks after sandbox bridge/mount resolution, and preserves current default apply_patch availability.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned deletion for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: The workspace-only delete path validates with `assertSandboxPath()` and then calls `fs.rm()` on `sandboxResult.resolved`. Current main uses `removePathWithinRoot()`, so this reintroduces a check-then-use mutation path that can be redirected outside the matched root if a parent is swapped between validation and deletion.
  - confidence: 0.9
- **[P2] Resolve mounted sandbox paths before allowed-root checks:** `src/agents/pi-tools.read.ts:601`
  - body: In sandbox runs this wrapper compares the container path against host-configured `additionalRoots` before `SandboxFsBridge.resolvePath` maps bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` still rejects `/shared/file`, so the new setting fails for the Docker bind-mount case.
  - confidence: 0.87
- **[P2] Preserve default apply_patch enablement:** `src/agents/pi-tools.ts:360-361`
  - body: This branch requires `tools.exec.applyPatch.enabled` to be explicitly true before creating `apply_patch`. Current main keeps `apply_patch` available for OpenAI models unless it is explicitly disabled, so merging this would silently remove the tool from normal coding profiles with no `applyPatch` config.
  - confidence: 0.86
