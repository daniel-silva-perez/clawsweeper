## Summary

Keep open. Current main still does not expose `tools.fs.allowedRoots`, so the PR is not obsolete, but the branch is not ready to merge: it still has a security-sensitive unpinned delete path in `apply_patch`, rejects the Docker/sandbox bind-mount case it is meant to support, and has drifted behind current main's default `apply_patch` enablement behavior.

## What This Changes

This PR adds a `tools.fs.allowedRoots` config field and threads additional filesystem roots through FS guards, `apply_patch`, media/prompt-image path handling, schema/help text, and regression tests.

## Best Possible Solution

Land a corrected, rebased version that keeps one canonical filesystem policy path, preserves pinned fs-safe helpers for all root-contained mutations, resolves sandbox container paths through the bridge or mount table before comparing configured roots, preserves current `apply_patch` defaults, and keeps config/schema/help/tests aligned across read/write/edit, `apply_patch`, and media tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned removal for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: The workspace-only delete path validates with `assertSandboxPath()` and then calls `fs.rm()` on the resolved string. Current main uses `removePathWithinRoot()`, so this separates validation from deletion and can delete outside the matched root if a parent is swapped between the check and the operation. Use the pinned removal helper with `sandboxResult.matchedRoot` and `sandboxResult.relative`.
  - confidence: 0.9
- **[P2] Validate allowed roots after sandbox mount resolution:** `src/agents/pi-tools.read.ts:601`
  - body: In sandbox runs, this wrapper checks the raw/container path against host-configured `additionalRoots` before `SandboxFsBridge.resolvePath` maps bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` still rejects `/shared/file`, so the new setting fails for the Docker bind-mount case. Resolve through the bridge or mount table first, or compare against corresponding container roots.
  - confidence: 0.87
- **[P2] Preserve default apply_patch enablement:** `src/agents/pi-tools.ts:361`
  - body: This branch requires `tools.exec.applyPatch.enabled` to be explicitly true before creating `apply_patch`. Current main and its tests keep `apply_patch` available for OpenAI models unless it is explicitly disabled, so merging this stale branch would silently remove the tool from normal coding profiles with no `applyPatch` config.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff expands filesystem access policy and currently drops pinned deletion protection for workspace-only `apply_patch` deletes.

Concerns:

- **[high] Unpinned apply_patch deletion after path validation:** `src/agents/apply-patch.ts:311`
  - body: `apply_patch` delete now performs sandbox validation and then deletes `sandboxResult.resolved` with `fs.rm()`. That is a check-then-use mutation path, unlike current main's pinned removal helper, and can be redirected outside the intended root by a concurrent parent-directory alias swap.
