## Summary

Keep open. Current main still lacks `tools.fs.allowedRoots`, and this PR is the active implementation attempt, but the branch still has a blocking sandbox bind-mount gap: sandboxed read/write/edit checks compare container paths directly against host-configured roots before `SandboxFsBridge` can map bind mounts.

## What This Changes

This PR adds a `tools.fs.allowedRoots` config field and threads it through filesystem, media, prompt-image, and `apply_patch` path checks with regression tests.

## Best Possible Solution

The desired end state is a single path-policy implementation where read/write/edit, `apply_patch`, and media tools resolve sandbox container and host paths through the bridge or mount table before comparing against configured host roots, with schema/help/tests aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate allowed roots after sandbox mount resolution:** `src/agents/pi-tools.read.ts:601`
  - body: In sandbox runs, this wrapper checks the raw/container path against host-configured `additionalRoots` before `SandboxFsBridge.resolvePath` can map Docker bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` still rejects `read`/`write`/`edit` on `/shared/file`, so the new setting fails for the sandbox bind-mount case. Resolve through the bridge/mount table first, or compare against corresponding container roots.
  - confidence: 0.87
