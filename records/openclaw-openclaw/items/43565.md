## Summary

Keep open. Current main still lacks `tools.fs.allowedRoots`, the linked feature tracker was closed stale, and this PR remains the active implementation candidate, but the patch is not merge-ready because sandbox bind-mount paths are still validated against host allowed roots before mount resolution.

## What This Changes

This PR adds `tools.fs.allowedRoots` to config/schema/help and threads it through filesystem tools, `apply_patch`, sandbox/media/image/PDF paths, and regression tests.

## Best Possible Solution

Keep this PR open as the canonical implementation candidate, fix the sandbox bind-mount policy gap, rebase onto current main while preserving recent path-hardening behavior, then validate the filesystem/media/apply_patch surfaces with targeted tests before maintainer security review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate allowed roots after sandbox mount resolution:** `src/agents/pi-tools.read.ts:601`
  - body: In sandbox runs, this wrapper checks the raw/container path against host-configured `additionalRoots` before `SandboxFsBridge.resolvePath` can map Docker bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` still rejects `read`/`write`/`edit` on `/shared/file`, so the new setting fails for the sandbox bind-mount case. Resolve through the bridge/mount table first, or compare against the corresponding container roots.
