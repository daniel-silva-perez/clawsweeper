## Summary

Keep open. Current main still does not implement `tools.fs.allowedRoots`, so this PR is not obsolete, but the PR head has blocking filesystem-boundary regressions that need author or maintainer fixes before merge.

## What This Changes

This PR adds a `tools.fs.allowedRoots` config field and threads additional filesystem roots through FS path guards, `apply_patch`, media/prompt-image loading, schema/help text, and regression tests.

## Best Possible Solution

Land a corrected, rebased version that keeps one canonical filesystem policy path, preserves pinned fs-safe helpers for all root-contained mutations, resolves sandbox container paths through the bridge or mount table before comparing configured roots, preserves current `apply_patch` defaults, and keeps config/schema/help/tests aligned across read/write/edit, `apply_patch`, and media tools.

## Reproduction Assessment

Yes. Current-main source/schema inspection reproduces the missing `allowedRoots` feature, and the PR regressions are directly visible at PR head; targeted tests should cover the delete, bind-mount, and default-enable paths after fixes.

## Solution Assessment

No. The feature direction is useful, but this branch is not the best mergeable solution until it preserves pinned deletion, validates allowed roots after sandbox mount resolution, and keeps default `apply_patch` enablement unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned removal for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: The workspace-only delete path validates with `assertSandboxPath()` and then calls `fs.rm()` on the resolved string. Current main uses `removePathWithinRoot()`, so this separates validation from deletion and can be redirected outside the matched root if a parent is swapped between the check and the operation. Use the pinned removal helper with `sandboxResult.matchedRoot` and `sandboxResult.relative`.
  - confidence: 0.9
- **[P2] Validate allowed roots after sandbox mount resolution:** `src/agents/pi-tools.read.ts:601`
  - body: In sandbox runs, this wrapper checks the raw/container path against host-configured `additionalRoots` before `SandboxFsBridge.resolvePath` maps bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` still rejects `/shared/file`, so the new setting fails for the Docker bind-mount case.
  - confidence: 0.87
- **[P2] Preserve default apply_patch enablement:** `src/agents/pi-tools.ts:361`
  - body: This branch requires `tools.exec.applyPatch.enabled` to be explicitly true before creating `apply_patch`. Current main keeps `apply_patch` available for OpenAI models unless it is explicitly disabled, so merging this would silently remove the tool from normal coding profiles with no `applyPatch` config.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff expands filesystem access policy and currently drops pinned deletion protection for workspace-only `apply_patch` deletes.

Concerns:

- **[high] Unpinned apply_patch deletion after path validation:** `src/agents/apply-patch.ts:311`
  - body: `apply_patch` delete now performs sandbox validation and then deletes `sandboxResult.resolved` with `fs.rm()`. That is a check-then-use mutation path, unlike current main's pinned removal helper, and can be redirected outside the intended root by a concurrent parent-directory alias swap.
