## Summary

Keep open. Current main still lacks a cron-owned watcher or idle reload path for direct jobs.json writes after an empty startup, and the PR head is not merge-ready because its watcher only invalidates memory without waking the scheduler or handling self-write, missing-file, and rename-watch cases. The security pass found no dependency, CI, secret, release, or package-resolution changes; the remaining risks are localized scheduler correctness issues.

## What This Changes

The PR adds a cron jobs.json watcher helper, stores its cleanup callback in cron service state, starts it from cron startup, stops it during cron shutdown, and updates the test harness state shape.

## Best Possible Solution

Keep this PR open, but rework it before merge. The best fix is a cron-owned watcher or bounded idle poll that routes external jobs.json changes through the existing locked reload, recompute, persist-if-needed, and armTimer path; suppresses self-write invalidations from persist(); handles missing-file startup and rename-based atomic writes; cleans up on stop(); and adds focused tests for direct jobs.json writes after an empty startup.

## Risks / Open Questions

- Merging the PR as-is would still leave direct jobs.json writes after an empty startup unscheduled, because invalidating the in-memory store does not wake or re-arm the scheduler.
- The watcher callback mutates state outside the locked cron operation path and can race with internal persist()/armTimer() flows or self-write events from saveCronStore().
- A file-bound fs.watch can fail when jobs.json is missing at startup or stop tracking after atomic rename-based writes, so the feature can silently degrade without a retry or directory-level strategy.
