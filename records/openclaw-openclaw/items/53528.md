## Summary

Keep open. Current main still lacks a cron file watcher or wake/reload path for direct `jobs.json` writes, while the proposed patch still has blocking scheduler correctness issues: cache invalidation alone does not wake an idle scheduler, and the watcher can react to the service's own writes. The request is valid and narrow, but this PR needs changes before merge.

## What This Changes

The PR adds a cron service `fs.watch` helper for `jobs.json`, stores a watcher cleanup callback in cron service state and the test harness, and starts/stops the watcher with the cron service.

## Best Possible Solution

Implement this as cron-owned external-change handling: observe `jobs.json` robustly, suppress internal persistence events, serialize reload/recompute under the existing cron lock, wake or re-arm scheduling after external changes, clean up on stop, and add focused tests for empty-startup direct writes, missing `jobs.json` at startup, and rename/self-write watcher behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wake the scheduler when the watch fires:** `src/cron/service/store.ts:88-89`
  - body: The callback only clears the cached store. If cron started with no runnable jobs, `armTimer()` left no timer active, so an external write can fire this callback but no later path calls `ensureLoaded()` or re-arms the scheduler; the newly written job remains unscheduled until another cron operation or restart.
  - confidence: 0.94
- **[P1] Ignore internal store writes before invalidating state:** `src/cron/service/store.ts:88-89`
  - body: `saveCronStore()` rewrites `jobs.json`, so add/update/remove can trigger this watcher from the service's own persist. Clearing `state.store` between `persist()` and the caller's following `armTimer()` can leave the scheduler unarmed or make later optional store accesses silently no-op. Suppress self-write events and perform reload/re-arm work under the cron lock.
  - confidence: 0.9
- **[P2] Retry watching when jobs.json is absent or replaced:** `src/cron/service/store.ts:85-101`
  - body: A single file-bound `fs.watch(path)` is installed once and failures are swallowed. If `jobs.json` is missing at startup, or if rename-based persistence invalidates the watched inode, no directory watch, retry, or fallback wake is installed, so later external writes are never observed.
  - confidence: 0.86
