## Summary

Keep open. Current main still lacks a cron-owned watcher or idle reload path for direct jobs.json writes after an empty startup, and the PR head is not merge-ready because it only invalidates cached state without reliably waking the scheduler or handling self-writes, missing-file startup, and atomic rename writes. The security pass found no CI, dependency, secret, release, or package-resolution changes; the remaining concerns are localized cron scheduler correctness issues.

## What This Changes

The PR adds a cron jobs.json watcher helper, stores its cleanup callback in cron service state, starts and stops it with the cron service, and updates the cron service test harness state shape.

## Best Possible Solution

Implement a cron-owned external-change handler for jobs.json that runs through the existing locked reload, maintenance recompute, persist-if-needed, and armTimer path; it should suppress self-write invalidations, survive missing-file startup and atomic rename writes, clean up on stop(), and include focused tests for direct writes after an empty startup.

## Risks / Open Questions

- Merging the PR as-is would still leave direct jobs.json writes after an empty startup unscheduled, because invalidating the in-memory store does not wake or re-arm the scheduler.
- The watcher callback mutates cron state outside the locked operation path and can race with internal persist()/armTimer() flows or self-write events from saveCronStore().
- A single file-bound fs.watch can fail when jobs.json is missing at startup or stop tracking after atomic rename-based writes, so the feature can silently degrade without retry or directory-level handling.
