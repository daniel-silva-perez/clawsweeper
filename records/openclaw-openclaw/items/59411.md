## Summary

Keep open. Current main still lacks the proposed timezone-aware reset field and the `idleGuard` API, but the PR is not mergeable as-is: the patch targets the older reset implementation, has concrete timezone validation/offset bugs, and overlaps with the separate adaptive reset API in #71400.

## What This Changes

The PR adds optional `session.reset.timezone` and `idleGuard` fields, threads `agents.defaults.userTimezone` into reset-policy resolution, updates config schema/help/types, and adds session reset tests.

## Best Possible Solution

Choose the public session-reset API first, then port the implementation into `src/config/sessions/reset-policy.ts`: validate IANA timezone IDs, preserve host-local behavior unless a timezone is explicitly configured or intentionally migrated, handle real-world offsets and DST safely, and reconcile AND semantics with #71400 before updating generated schema, docs, tests, and changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate timezone IDs before evaluating resets:** `src/config/zod-schema.session.ts:21`
  - body: `timezone` accepts any string, but reset evaluation later passes it to `Intl.DateTimeFormat`. A typo such as `UTC+8` can throw a `RangeError` during message/session handling, so invalid IANA IDs should be rejected or normalized at config/policy resolution.
  - confidence: 0.9
- **[P1] Fix offset wrapping for UTC+13 and UTC+14 zones:** `src/config/sessions/reset.ts:273-278`
  - body: `computeTzOffsetMs` treats any difference above 720 minutes as a day wrap, but valid zones such as `Pacific/Kiritimati` and `Pacific/Chatham` exceed +12 hours. Those offsets get inverted, producing reset boundaries on the wrong day/hour.
  - confidence: 0.91
- **[P2] Reject idleGuard without an idle window:** `src/config/sessions/reset.ts:175-187`
  - body: `idleGuard` only switches to AND mode when `idleExpiresAt` exists. With `idleGuard: true` and no positive `idleMinutes`, the flag is silently ignored and the policy behaves as daily-only, which is misleading for a guard that depends on idleness.
  - confidence: 0.84
- **[P2] Preserve host-local behavior when timezone is omitted:** `src/auto-reply/reply/session.ts:382`
  - body: Passing `agents.defaults.userTimezone` whenever reset config lacks `timezone` changes existing daily reset timing for users who only configured their prompt timezone. That contradicts the PR's compatibility claim and should be explicit API behavior or documented as a migration.
  - confidence: 0.79
