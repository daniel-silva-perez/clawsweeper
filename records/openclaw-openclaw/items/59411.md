## Summary

Keep this PR open for maintainer review. Current main still lacks both timezone-aware session reset boundaries and daily-plus-idle AND semantics, but the proposed patch is not merge-ready: it has concrete runtime correctness bugs, conflicts with the current reset-policy split, and partly overlaps the separate adaptive-mode design in #71400.

## What This Changes

The PR adds optional `timezone` and `idleGuard` fields to session reset config, threads user timezone into reset policy resolution, updates config schema/help/types, and adds tests for timezone daily boundaries and AND-style daily-plus-idle freshness.

## Best Possible Solution

Choose the public API first, then port the implementation onto `src/config/sessions/reset-policy.ts`: validate IANA timezones at config/policy boundaries, handle real UTC offsets through +14, preserve current session freshness semantics, update generated config/docs/changelog as needed, and reconcile the AND behavior with #71400 rather than landing two competing config shapes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate timezone IDs before evaluating resets:** `src/config/zod-schema.session.ts:21`
  - body: `timezone` currently accepts any string, but the reset helper later passes it to `Intl.DateTimeFormat`. A typo such as `UTC+8` will throw `RangeError` during session freshness evaluation, so this should be rejected or normalized at config/policy resolution time.
  - confidence: 0.9
- **[P1] Fix offset wrapping for UTC+13 and UTC+14 zones:** `src/config/sessions/reset.ts:273-278`
  - body: The offset normalization treats any `diffMinutes > 720` as a day wrap and subtracts 1440, but valid zones such as `Pacific/Kiritimati` and `Pacific/Chatham` exceed +12 hours. Those users get an inverted offset and reset boundaries on the wrong day.
  - confidence: 0.91
- **[P2] Reject idleGuard without an idle window:** `src/config/sessions/reset.ts:175-187`
  - body: `idleGuard` only switches to AND mode when `idleExpiresAt` exists. With `idleGuard: true` and no positive `idleMinutes`, the guard is silently skipped and the config behaves as daily-only, which is misleading for a flag defined as an idle guard.
  - confidence: 0.84
- **[P2] Keep omitted timezone configs host-local:** `src/auto-reply/reply/session.ts:382`
  - body: Passing `agents.defaults.userTimezone` as the fallback changes existing daily reset behavior for users who configured prompt timezone but did not set `session.reset.timezone`. That contradicts the current host-local session docs and the PR's compatibility claim, so the behavior should be explicit or fully documented as a migration.
  - confidence: 0.79
