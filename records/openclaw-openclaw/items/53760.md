## Summary

Keep this issue open. Current main still lacks an enforced DM-level sender tool-policy contract for Signal/Telegram-style direct message channels. Telegram has typed direct config for `direct.*.tools` and `direct.*.toolsBySender`, but the enforced tool-filtering path is still group-scoped; Signal still exposes `toolsBySender` only under group config.

## Best Possible Solution

Keep this open and implement it as a core/channel-policy feature, not a ClawHub workaround. Add a generic direct-message sender tool-policy contract, shared config/schema/docs for DM-level `tools` and `toolsBySender`, verified direct sender identity resolution for Signal and Telegram, a direct-policy resolver parallel to the existing group resolver, integration with `createOpenClawCodingTools` and `applyFinalEffectiveToolPolicy`, and regression tests proving one allowlisted DM sender can retain tools while another is denied.
