## Summary

Keep this issue open. Current main still removes the in-process session lock from `HELD_LOCKS` before the asynchronous close/unlink cleanup finishes, while the same-PID/no-starttime orphan path still treats absence from `HELD_LOCKS` as reclaimable. The reporter's later comment correctly lowers practical severity because representative session-store callers await release, but the lock primitive itself has not been hardened or covered by a release-in-progress regression test.

## Best Possible Solution

Keep the issue open and harden `src/agents/session-write-lock.ts` so a releasing lock remains visible until close/unlink cleanup completes, or add an explicit releasing guard that `shouldTreatAsOrphanSelfLock` respects. Add a focused regression test for non-awaited release followed by same-session reacquire with a same-PID payload lacking `starttime`, proving old cleanup cannot delete the new lock file.

## Risks / Open Questions

- Closing now would leave a low-probability but real primitive-level race for callers that begin release cleanup and reacquire the same session before cleanup completes.
- The practical impact appears mitigated in representative session-store paths because they await release, so the fix should stay targeted rather than redesigning session locking broadly.
