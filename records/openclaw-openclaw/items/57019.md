## Summary

Keep open. Current main still deletes the in-process session lock entry before async lock-file cleanup finishes, and the same-PID/no-starttime orphan path can still reclaim that file while release is in progress. The reporter’s lower-severity follow-up is fair because representative call sites await release, but the lock primitive itself is not hardened yet.

## Best Possible Solution

Keep this issue open and harden `src/agents/session-write-lock.ts` so a lock remains visible until close and unlink cleanup complete, or add an explicit releasing guard that `shouldTreatAsOrphanSelfLock` respects. Add a regression test that calls `release()` without awaiting it, simulates a same-PID payload without `starttime`, reacquires the same session, and proves the old cleanup cannot remove the new lock file.

## Risks / Open Questions

- Closing now would leave the lock primitive vulnerable to a low-probability release-in-progress race on macOS, Windows, and Linux environments where process start time is unavailable.
- Current call sites mostly await release, so the practical severity appears low; the fix should be targeted and covered by a focused regression test rather than broad lock redesign.
