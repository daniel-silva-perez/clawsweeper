## Summary

Keep open. Current main still deletes the in-process `HELD_LOCKS` entry before lock-file cleanup finishes, and the orphan-self reclamation path still treats same-PID locks without `starttime` as reclaimable when that map entry is absent. Existing OpenClaw call sites mostly await `release()`, which lowers practical exposure, but the lock primitive itself still has the release-in-progress gap described here.

## Best Possible Solution

Keep this issue open and harden `src/agents/session-write-lock.ts`. The best fix is to consider the lock held until `close` and `rm` complete, or add an explicit releasing guard checked by `shouldTreatAsOrphanSelfLock`; then add a regression test that starts release without awaiting it, simulates a missing `starttime` payload, reacquires the same session, and proves the old release cannot delete the new lock file.
