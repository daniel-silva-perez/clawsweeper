## Summary

Keep this issue open. Current main still parses dotenv files with `dotenv.parse` and assigns the parsed values directly to `process.env`, so `$VAR` and `${VAR}` aliases remain literal. The closed implementation PR #38254 is useful prior art, and its review comments are relevant, but it was closed unmerged and the current loader has since gained additional workspace `.env` security constraints that any fix must preserve.

## Best Possible Solution

Keep this issue open and implement dotenv-level expansion in `src/infra/dotenv.ts` before assignment to `process.env`, after key normalization and workspace/global filtering. Support shell-style `${VAR}` and `$VAR` references using `process.env` plus earlier accepted dotenv entries, preserve process/CWD/global/`gateway.env` precedence and conflict warnings, define missing/cyclic/escaped-dollar behavior, and add focused regression tests in `src/infra/dotenv.test.ts` plus a short docs/changelog note if behavior changes user-facing setup.
