## Summary

Current main still loads `.env` values literally. `src/infra/dotenv.ts` parses with `dotenv.parse` and assigns parsed values directly to `process.env`, while the separate config substitution path only applies to config string values. The prior implementation PR #38254 was closed unmerged, so this remains actionable in the repo.

## Best Possible Solution

Keep this issue open and implement dotenv-level expansion in `src/infra/dotenv.ts` before assignment to `process.env`. Support `${VAR}` and `$VAR` using process env plus earlier accepted dotenv entries, preserve shell precedence and workspace blocklists, keep `gateway.env` conflict warnings intact, and add focused regression tests in `src/infra/dotenv.test.ts`. PR #38254 is useful prior art but needs rechecking against the current workspace `.env` security constraints.

## Risks / Open Questions

- Adding expansion must preserve the current precedence contract: process env first, then CWD `.env`, then global `.env`, then `gateway.env` fallback conflicts.
- Workspace `.env` expansion must not allow blocked runtime-control, credential, endpoint, or trust-root keys to influence accepted values indirectly.
- Expansion semantics need explicit tests for missing refs, same-file ordering, chained refs, cycles, literal dollar values, and cross-file fallback ordering.
