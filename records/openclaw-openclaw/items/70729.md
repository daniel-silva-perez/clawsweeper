## Summary

Keep open. Current main still accepts and authorizes the trusted-proxy user but does not preserve that server-authenticated identity on the WebSocket client or use it when WebChat builds the agent message context. The remaining work is real, but it needs maintainer judgment on the public identity contract because the fix crosses a trusted auth boundary.

## What This Changes

Surface the server-authenticated trusted-proxy WebChat user into agent message context so agents can identify the real human sender.

## Best Possible Solution

Persist the accepted trusted-proxy user on the server-side WebSocket client after handshake, derive WebChat agent identity from that stored value, and add regression coverage proving client-declared metadata cannot spoof proxy identity. If maintainers choose a distinct authenticated-user field, document and version that context contract alongside the sender-field behavior.

## Reproduction Assessment

Yes. The issue gives concrete proxy/config steps, and current-main source inspection reproduces the failure logically: auth returns a trusted-proxy user, the WebSocket client does not store it, and `chat.send` uses generic Control UI client metadata for sender fields.

## Solution Assessment

Unclear until maintainer review chooses the public identity contract. The implementation boundary should be server-side propagation from `authResult.user`; reusing sender fields may be right, but the alternative distinct authenticated-user field needs product/security agreement.
