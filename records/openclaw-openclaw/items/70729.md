## Summary

Current main still does not surface trusted-proxy identity into WebChat agent message context. Trusted-proxy auth extracts and authorizes the header user, but the WebSocket client state does not retain that user, and `chat.send` still populates sender fields from generic Control UI client metadata.

## Best Possible Solution

Keep this open. The best path is a gateway-owned identity-contract decision, then a narrow implementation that stores the accepted trusted-proxy user on the registered WebSocket client and uses that server-authenticated identity for WebChat `chat.send` sender context, with negative tests proving client metadata cannot spoof it.
