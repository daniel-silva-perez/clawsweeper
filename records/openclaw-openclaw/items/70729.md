## Summary

Keep open. Current main still resolves the trusted-proxy user for authentication and authorization, but the WebSocket client state does not persist that user identity and `chat.send` builds the agent `MsgContext` sender fields from generic Control UI client metadata instead of the `X-Forwarded-User` identity. The related trusted-proxy fallback/default-user report is separate, not a duplicate.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment on the message-context contract. A good fix would persist the trusted-proxy user on the authenticated WebSocket connection, then have WebChat-originated `chat.send` populate the chosen agent-visible identity fields from that stored trusted identity, with targeted tests around trusted-proxy handshake, Control UI/WebChat `chat.send`, and a negative case proving untrusted client metadata cannot spoof the sender.

## Risks / Open Questions

- The fix changes agent-visible identity semantics for WebChat; maintainers should decide whether to populate `SenderId`/`SenderName` or add an explicit authenticated-user field.
- The implementation must only use the user identity captured from successful trusted-proxy authentication, not a per-message client-supplied value or unauthenticated connect metadata.
- Session-index persistence is useful but should be scoped carefully so it does not turn into a broader session migration or audit-schema change.
- Targeted regression coverage is needed because current tests cover trusted-proxy authentication and WebChat send behavior separately, but not propagation of trusted-proxy user identity into `MsgContext`.
