## Summary

Keep open. Current main still authenticates and authorizes trusted-proxy users but drops that server-authenticated identity before WebChat `chat.send` builds the agent message context, so the existing ClawSweeper keep-open assessment remains accurate.

## What This Changes

The issue asks WebChat to surface the trusted-proxy `X-Forwarded-User` identity in agent message context for WebChat-originated messages.

## Best Possible Solution

Persist the accepted trusted-proxy user on the WebSocket client after the server-side handshake, derive WebChat agent sender context from that stored authenticated value, and add regression coverage proving client-supplied metadata cannot spoof the proxy identity. Any session-index metadata should be added only if the stored session schema remains backward-compatible.

## Reproduction Assessment

Yes. The issue gives a concrete reverse-proxy deployment reproduction, and the current-main source trace is enough to reproduce the behavior logically: auth returns a trusted-proxy user, the WebSocket client does not store it, and `chat.send` uses generic Control UI client metadata for sender fields.

## Solution Assessment

Unclear until maintainer review chooses the public identity contract. Persisting the server-authenticated user and using it for WebChat dispatch is the right implementation boundary, but maintainers should decide whether to reuse `SenderId`/`SenderName` or add a distinct authenticated-user field.
