## Summary

Keep open. Current main has restored the duplicate heartbeat suppression half, but the plugin-channel fallback gap is still present: `target: "last"` can recover a stored plugin route from session metadata and then drop it when heartbeat routing re-enters strict outbound resolution against an empty or unbootstrapped plugin registry. The linked implementation attempt, #46203, is closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open. Implement the remaining fallback in the heartbeat outbound target path: when `target: "last"` recovers a stored plugin route from session or turn-source metadata, preserve or bootstrap that route when strict outbound resolution fails only because the plugin registry is empty or unbootstrapped. Leave the restored duplicate suppression intact and add unmocked regression coverage for `resolveHeartbeatDeliveryTarget()` with an empty active plugin registry.

## Risks / Open Questions

- A fix should be scoped to trusted stored session or turn-source routes for `heartbeat.target: "last"`; bypassing outbound normalization for arbitrary explicit heartbeat target strings could widen delivery attempts to unsupported channels.
- Regression coverage should use the real registry-backed normalization path, or a focused unmocked test, because the existing target tests mock the failing seam.
- The remaining fix needs to preserve existing accountId validation, directPolicy blocking, and threadId behavior for plugin-owned group routes.
