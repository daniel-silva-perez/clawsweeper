## Summary

Keep this issue open. Current main has restored duplicate heartbeat suppression, but the plugin-channel heartbeat fallback gap is still present: a stored plugin route can be recovered from session metadata for `target: "last"`, then dropped when heartbeat routing re-enters strict outbound plugin resolution against an empty or unbootstrapped registry. The related implementation PR #46203 is closed and unmerged in the provided GitHub context, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open. Implement the remaining heartbeat fallback so a stored plugin route recovered for `target: "last"` is preserved or bootstrapped when strict outbound resolution fails only because the plugin registry is empty or unbootstrapped. Leave the restored duplicate suppression in place and add unmocked regression coverage for `resolveHeartbeatDeliveryTarget()` with an empty active plugin registry.

## Risks / Open Questions

- A fix should be scoped to trusted stored session or turn-source routes for `heartbeat.target: "last"`; bypassing outbound normalization for arbitrary explicit heartbeat target strings could widen delivery attempts to unsupported channels.
- Regression coverage should use the real registry-backed normalization path, or a focused unmocked test, because the current target tests mock the seam that fails.
- The remaining fallback must preserve existing accountId validation, `directPolicy` blocking, and plugin-owned group-route threadId behavior.
