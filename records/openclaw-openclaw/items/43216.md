## Summary

Keep this issue open. Current main has restored the duplicate-heartbeat suppression half, but the plugin-channel heartbeat fallback is still incomplete: `resolveHeartbeatDeliveryTarget()` can recover a stored route, then drops it when strict outbound plugin resolution fails against an empty or unbootstrapped registry. The related implementation PR #46203 is closed and unmerged, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open. Implement the remaining heartbeat fallback so a stored plugin route recovered for `target: "last"` is preserved or safely bootstrapped when strict outbound resolution fails only because the plugin registry is empty or unbootstrapped. Keep the restored duplicate suppression, and add an unmocked regression test for `resolveHeartbeatDeliveryTarget()` with an empty active plugin registry.

## Risks / Open Questions

- A fallback should be scoped to trusted stored session or turn-source routes for `heartbeat.target: "last"`; bypassing outbound normalization for arbitrary explicit heartbeat target strings could widen delivery attempts to unsupported channels.
- Regression coverage should exercise the real registry-backed normalization path, because the current outbound target tests mock the seam that rejects empty-registry plugin ids.
- The fix should preserve existing `accountId` validation, `directPolicy` blocking, and plugin-owned group-route `threadId` behavior.
