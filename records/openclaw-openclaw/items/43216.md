## Summary

Keep this issue open. Current main has restored duplicate heartbeat suppression, but the plugin-channel heartbeat fallback remains incomplete: `resolveHeartbeatDeliveryTarget()` can recover a stored plugin-style route and still drop it to `channel: "none", reason: "no-target"` when registry-backed outbound resolution cannot recognize/bootstrap the plugin channel. Related PR #46203 was closed unmerged, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open and narrow it to the remaining plugin-channel heartbeat fallback. The likely fix belongs in `src/infra/outbound/targets.ts` or the outbound channel-resolution seam: when `target: "last"` recovered a stored plugin route from session metadata, preserve or bootstrap that route when outbound normalization fails only because the plugin registry is empty/unbootstrapped. Leave the restored duplicate suppression unchanged and add an unmocked regression test for `resolveHeartbeatDeliveryTarget()` with an empty active plugin registry.

## Risks / Open Questions

- A fix should be scoped to trusted stored session routes for `heartbeat.target: "last"`; broadly bypassing outbound normalization for explicit configured targets could send to unsupported or unintended channels.
