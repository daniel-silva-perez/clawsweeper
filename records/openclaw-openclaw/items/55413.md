## Summary

Keep this PR open. Current main still lacks the requested canonical PluralKit inbound ID behavior, but the latest PR diff should not merge as-is because it can drop PK webhook messages before the lookup has a chance to classify them as PluralKit traffic.

## What This Changes

The PR changes Discord inbound handling to look up PluralKit data for webhook-authored messages, carry a canonical PK original ID through preflight, use it as `MessageSid`, preserve the Discord provider ID as `MessageSidFull`, and add regression tests.

## Best Possible Solution

Keep the narrow canonical-ID direction, but rework the preflight ordering so known OpenClaw bound-thread webhook echoes are filtered before external PK calls while every other PK-enabled message is looked up before bot and mention gates that depend on `sender.isPluralKit`; then thread `pluralkitInfo.original` into the current context-building path with regression coverage for default `allowBots=false`, webhook payloads, hydrated payloads, and echo suppression.
