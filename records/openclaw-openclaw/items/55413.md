## Summary

Keep this PR open. Current main still does not canonicalize Discord PluralKit webhook proxy copies to the PK `original` id, so the proposed fix is useful, but the PR head is not correct as-is because it moves the webhook-capable PK lookup after bot filtering and is stale relative to current main's preflight/context split.

## What This Changes

The PR changes Discord PluralKit inbound handling to look up PK data for webhook-authored messages, carry a canonical original id through preflight, use it as `MessageSid`, preserve the provider Discord id as `MessageSidFull`, and add regression tests.

## Best Possible Solution

Land a rebased Discord fix that suppresses known OpenClaw bound-thread webhook echoes before external PK calls, performs PK lookup for every other PK-enabled inbound message before bot and mention gates, threads a trimmed `pluralkitInfo.original` through the current preflight/context builder, and emits canonical `MessageSid` plus provider `MessageSidFull` with focused regression coverage.

## Reproduction Assessment

Yes. The current-main failure is reproducible by tracing a PK-enabled webhook-authored Discord message through `resolveDiscordPreflightPluralKitInfo`, which exits on `webhookId`, and the PR regression is reproducible by following the PR diff path where `author.bot` can return before the new lookup resolves `sender.isPluralKit`.

## Solution Assessment

No. Canonicalizing to the PK `original` id is the right product direction, but this PR head is not the best implementation because it loses the current pre-bot lookup ordering and targets stale context-building code; the safer fix is to adapt the current helper/context builder directly.
