## Summary

Keep open. Current main still does not implement the requested canonical PluralKit inbound identity split, and the PR targets a real Discord bug, but the proposed patch still has a blocking ordering regression: PluralKit identity resolution can happen after bot-message filtering, so default `allowBots=false` traffic can be dropped before PK identity is known.

## What This Changes

The PR changes Discord inbound handling to look up PluralKit data for webhook-authored messages, carry a canonical PK original ID through preflight, use it as `MessageSid`, preserve the provider Discord ID as `MessageSidFull`, and add regression tests.

## Best Possible Solution

Land a rebased Discord fix that suppresses known OpenClaw bound-thread webhook echoes before external PK calls, performs PK lookup for every other PK-enabled inbound message before bot and mention gates, threads a trimmed `pluralkitInfo.original` through the current preflight/context builder, and emits canonical `MessageSid` plus provider `MessageSidFull` with focused regression coverage.
