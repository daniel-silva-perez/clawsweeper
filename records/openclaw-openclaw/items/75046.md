## Summary

Keep open. Current main does not implement a Cursor SDK backend, and this PR is an active contributor implementation paired with the still-open feature issue #75044. The latest head fixes several earlier runner defects, but it still leaves Cursor timeout/abort handling incomplete and introduces a core runtime, root dependency, and security-policy boundary that need maintainer review before merge.

## What This Changes

The PR adds `@cursor/sdk` as a root dependency and wires a new `cursor-sdk` backend, config block, auth env var, dispatch branches, and unit/live tests into chat reply, agent command, and cron execution paths.

## Best Possible Solution

Decide the Cursor SDK ownership boundary first. If maintainers want this in core, land it only as a documented runtime integration that preserves OpenClaw cancellation, timeout, fallback, lifecycle hooks, status semantics, auth UX, and security policy; otherwise keep Cursor on the existing ACP/acpx or ClawHub/plugin path.

## Reproduction Assessment

Yes for the review findings: static inspection of the PR head and `@cursor/sdk@1.0.7` public types shows the timeout timer only covers streaming and that existing abort signals are not passed into the new Cursor runner. A live Cursor key is not needed to reproduce those control-flow gaps.

## Solution Assessment

No. The proposed PR is not the best merge path as-is because it bypasses the documented external-harness route and leaves core runtime contracts incomplete; the safer path is maintainer-approved runtime/plugin design followed by a narrower implementation that handles policy, abort, timeout, and supply-chain review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Enforce timeout around the whole Cursor run:** `src/agents/cursor-sdk-runner.ts:160-170`
  - body: `timeoutMs` is only started after `agent.send()` resolves and is cleared before `run.wait()`. If the SDK stalls while creating/sending the run or while waiting for the terminal result, this path can hang instead of cancelling/failing over; wrap send, stream, and wait in one timeout/cancel flow.
  - confidence: 0.87
- **[P2] Propagate run aborts into Cursor SDK:** `src/auto-reply/reply/agent-runner-execution.ts:1400-1412`
  - body: The existing CLI path passes the reply/cron abort signal, but the new Cursor branch calls `runCursorSdkAgent` without any abort signal and the runner has no signal parameter. User stops or cron aborts will not cancel an in-flight Cursor run until the SDK returns; pass the signal through and cancel/fail over on abort.
  - confidence: 0.82
