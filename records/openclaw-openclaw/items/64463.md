## Summary

Current main still does not implement the requested opt-in session wall-clock duration cap or cumulative per-session token cap. The existing surfaces cover daily/idle freshness, parent-fork context guards, storage cleanup, per-run cron timeouts, context display snapshots, and plugin/tool hook policy points, but not a first-party core session-budget kill/compact/reset mechanism. The related discussion remains useful: the implementation sketch identifies the likely check sites and actions, and the linked Helm-side docs are a workaround rather than a shipped OpenClaw core feature.

## Best Possible Solution

Keep this issue open and implement it as a core session-budget feature. Add optional config schema/types/docs for session duration and cumulative token limits, persist reliable per-session lifecycle and cumulative usage-budget state, enforce before LLM/tool work and on lifecycle checks, emit deterministic user/operator feedback, and add focused regression tests across normal chat, cron/heartbeat, provider-owned CLI/ACP sessions, compaction/reset interactions, and budget-exceeded actions.

## Risks / Open Questions

- Budget-exceeded behavior still needs maintainer product/API judgment: stop, warn, compact, reset, graceful memory flush, and channel/runtime-specific feedback semantics are not settled.
- A correct cumulative token cap likely needs new durable per-session usage accounting because current `totalTokens` is intentionally a context snapshot, not cumulative spend.
- Cron, heartbeat, always-on, ACP/provider-owned CLI, normal chat, compaction/reset, and tool-call paths need explicit coverage so caps trigger before costly work where possible.
