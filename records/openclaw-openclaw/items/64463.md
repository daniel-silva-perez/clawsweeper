## Summary

Current main still does not implement the requested opt-in session wall-clock duration cap or cumulative per-session token cap. The related discussion includes a useful implementation sketch and an external Helm documentation workaround, but the current OpenClaw config schema, public types, lifecycle checks, usage persistence, and cron/heartbeat docs still only cover daily/idle freshness, thread-binding max age, per-run timeouts, context snapshots, and store maintenance.

## Best Possible Solution

Keep this issue open and implement it as a core session-budget feature. Add optional config schema/types/docs for session duration and cumulative token limits, persist reliable per-session lifecycle and cumulative usage-budget state, enforce before LLM/tool work and on lifecycle checks, emit deterministic user/operator feedback, and add focused regression tests for normal chat, cron/heartbeat, provider-owned CLI/ACP sessions, compaction/reset interactions, and budget-exceeded actions.

## Risks / Open Questions

- Budget-exceeded behavior still needs maintainer product/API judgment: stop, warn, compact, reset, graceful memory flush, and channel/runtime-specific feedback semantics.
- A correct cumulative token cap likely needs new durable per-session usage accounting because current totalTokens is deliberately a context snapshot, not cumulative spend.
- Cron, heartbeat, always-on, ACP/provider-owned CLI, and normal chat paths need explicit coverage so caps are enforced before costly work and not only after a run persists usage.
