## Summary

Keep this PR open for maintainer follow-up. Current main has shipped a different overload failover model under `auth.cooldowns`, so this branch is stale against the current product surface, and the branch also carries live-test environment regressions that should block merge as-is.

## What This Changes

The PR raises embedded Pi overload failover pacing to a configurable 30s ceiling via `agents.defaults.embeddedPi.overloadBackoffMaxMs`, updates config schema/help/docs/tests, and adds live-test HOME/profile staging changes.

## Best Possible Solution

If overload pacing still needs product work, implement it against the shipped `auth.cooldowns` model with focused docs/schema/tests. The live-test HOME/profile staging changes should be split or rebased separately, with profile sourcing gated to live setup and config staging preserving include roots.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR findings: ordinary test setup calls `installTestEnv()`, and the PR's profile sourcing can import `LIVE` / `OPENCLAW_LIVE_TEST` before `.live.test.ts` module-level gates evaluate.

## Solution Assessment

Is this the best way to solve the issue? No. The overload config path in the PR no longer matches current main's shipped `auth.cooldowns` design, and the unrelated live-test staging changes need to be narrowed or fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate profile sourcing to live setup:** `test/test-env.ts:338`
  - body: The patch calls `loadProfileEnv(realHome)` for every `installTestEnv()` invocation. A developer profile that exports live flags or provider credentials can therefore turn ordinary non-live test workers into live/network runs before the isolated live setup has run.
  - confidence: 0.9
- **[P2] Refresh the live env snapshot after loading the profile:** `test/test-env.ts:336`
  - body: `liveEnvSnapshot` is captured before profile loading, but `stageLiveTestState()` uses that snapshot to choose `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. Live runs that rely on `~/.profile` for those paths will stage the wrong config/state.
  - confidence: 0.86
- **[P2] Preserve config include files when staging live HOME:** `test/test-env.ts:315-316`
  - body: The staging code copies only the active config file into the temp home. Config `$include` paths are resolved relative to the config file root, so valid real configs with sibling include files can fail after staging.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR's test-env changes can source developer profile secrets and live flags into ordinary test processes.

Concerns:

- **[medium] Profile sourcing can leak live credentials into unit tests:** `test/test-env.ts:338`
  - body: Because profile env loading runs outside the live-only path in the PR, non-live test setup can import provider keys, GitHub tokens, and live flags from a developer profile before isolation. That broadens secret exposure and can trigger expensive or flaky live network suites unexpectedly.
  - confidence: 0.88
