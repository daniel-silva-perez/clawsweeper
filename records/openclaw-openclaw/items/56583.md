## Summary

Keep this PR open for maintainer handling rather than conservative cleanup. The branch is stale against current main's shipped `auth.cooldowns` overload model, and the proposed live-test staging changes introduce concrete correctness and security regressions before merge.

## What This Changes

The PR raises embedded Pi overload failover backoff to a configurable 30s ceiling via `agents.defaults.embeddedPi.overloadBackoffMaxMs`, updates config schema/help/tests, and adds live-test HOME/profile staging behavior.

## Best Possible Solution

If overload pacing still needs product work, implement it against the shipped `auth.cooldowns` model with focused schema/docs/tests. The live-test HOME/profile staging work should be split or rebased separately and keep profile sourcing live-gated while preserving config include roots.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR findings: ordinary test setup calls `installTestEnv()`, the PR sources `~/.profile` unconditionally, and live test gates read `LIVE` / `OPENCLAW_LIVE_TEST` from `process.env` at module load.

## Solution Assessment

Is this the best way to solve the issue? No. The overload change targets an older `agents.defaults.embeddedPi` knob instead of current main's `auth.cooldowns` contract, and the unrelated live-test staging changes need to be narrowed or fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate profile sourcing to live setup:** `test/test-env.ts:338`
  - body: The patch calls `loadProfileEnv(realHome)` for every `installTestEnv()` invocation. A developer profile that exports live flags or provider credentials can turn ordinary non-live test workers into live/network runs before the isolated live setup has run.
  - confidence: 0.9
- **[P2] Refresh the live env snapshot after loading the profile:** `test/test-env.ts:336`
  - body: `liveEnvSnapshot` is captured before profile loading, but `stageLiveTestState()` uses that snapshot to choose `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. Live runs relying on `~/.profile` for those paths will stage the wrong config or state.
  - confidence: 0.86
- **[P2] Preserve config include files when staging live HOME:** `test/test-env.ts:315-316`
  - body: The staging code copies only the active config file into the temp home. Config `$include` paths are resolved relative to the config file root, so valid real configs with sibling include files can fail after staging.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete secret-exposure risk by sourcing developer profile environment into ordinary test setup.

Concerns:

- **[medium] Profile sourcing can leak live credentials into unit tests:** `test/test-env.ts:338`
  - body: Because profile env loading runs outside the live-only path, non-live test setup can import provider keys, GitHub tokens, and live flags from a developer profile before isolation. That broadens secret exposure and can trigger expensive or flaky live network suites unexpectedly.
  - confidence: 0.88
