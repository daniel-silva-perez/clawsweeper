## Summary

Keep this PR open for maintainer judgment, but it is not mergeable as-is. Current main and `v2026.4.27` already shipped a different overload policy under `auth.cooldowns`, while this branch still contains blocking live-test environment regressions around profile sourcing, staged config includes, and secret-bearing env import.

## What This Changes

The PR raises embedded Pi overload failover backoff behavior/configuration and adds schema/docs/tests, while also changing live-test HOME/profile staging behavior and documentation.

## Best Possible Solution

If maintainers still want overload pacing changes, implement them against the shipped `auth.cooldowns` model with focused docs/schema/tests. The live-test HOME staging work should be separated or rebased so profile sourcing is gated to live setup and staged configs preserve include roots.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR findings: `test/global-setup.ts` and `test/setup.shared.ts` call `installTestEnv()` during ordinary test setup, and the PR’s unconditional `loadProfileEnv(realHome)` can import profile-only live flags before `.live.test.ts` modules evaluate their module-level gates.

## Solution Assessment

Is this the best way to solve the issue? No. The backoff goal should align with current main’s `auth.cooldowns` design, and the test-env changes need to be narrowed or fixed before this branch can be considered safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate profile sourcing to live setup:** `test/test-env.ts:338`
  - body: The patch now calls `loadProfileEnv(realHome)` for every `installTestEnv()` invocation, including normal `pnpm test` setup. If a developer profile exports `LIVE`, `OPENCLAW_LIVE_TEST`, provider keys, or gateway live flags, those values are imported into non-live workers before `.live.test.ts` files evaluate their module-level gates, so ordinary test runs can unexpectedly execute live/network suites and expose local credentials to tests.
  - confidence: 0.9
- **[P2] Refresh the live env snapshot after loading the profile:** `test/test-env.ts:336`
  - body: `liveEnvSnapshot` is captured before `loadProfileEnv(realHome)`, but `stageLiveTestState()` later uses that snapshot to choose `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`. Live runs that rely on `~/.profile` for those paths will stage the wrong config/state, even though the docs say live runs source the profile for missing settings.
  - confidence: 0.86
- **[P2] Preserve config include files when staging live HOME:** `test/test-env.ts:315-316`
  - body: This copies only the active config file into `tempHome/.openclaw/openclaw.json`. Config `$include` paths are resolved relative to the config file’s directory and are blocked from escaping that root, so a valid real config such as `agents: { $include: "./agents.json5" }` will fail after staging because the included file was not copied alongside it.
  - confidence: 0.82
