## Summary

Keep open. Current main still resolves plugin text transforms through the createRequire-based active runtime helper, while both CLI and provider prompt paths consume that resolver; the related fix PR #65331 remains open and unmerged.

## What This Changes

The issue asks for plugin-registered `textTransforms` to be resolved from the active plugin registry and applied to CLI/provider prompts in the main gateway process.

## Best Possible Solution

Fix the runtime text-transform lookup so bundled main-process code can read the active plugin registry directly or through a proven generic seam, preserve or document any external-plugin fallback that is still needed, and add focused regression coverage for CLI and provider prompt application.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: a plugin can register `textTransforms`, but current main resolves them through a helper that only require-probes stable runtime filenames before the CLI/provider prompt paths consume the result.

## Solution Assessment

Yes as a direction, but the current open PR should be adapted to the present `active-runtime-registry` layout and paired with regression tests rather than landed unchanged.
