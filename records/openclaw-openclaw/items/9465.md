## Summary

Keep open. Current main still does not expose persisted cron lifecycle hooks or a pre-execution gate before model execution, and the issue discussion shows active, concrete demand for deterministic cron short-circuiting. The related open PR #45465 is the right implementation candidate to review or supersede, but the requested shell/webhook behavior remains product- and security-sensitive rather than cleanup-ready.

## What This Changes

The issue requests configurable cron job lifecycle hooks for pre-, post-, and error phases, including agent turn, system event, shell command, and webhook actions around scheduled execution.

## Best Possible Solution

Land or explicitly supersede #45465 with an additive cron hook lifecycle covering persisted job/config types, Gateway schemas and RPCs, scheduled/startup/manual execution paths, CLI, Control UI, docs, tests, and an operator-controlled security model for user-configured hooks.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. We do have a high-confidence verification path: source and schema inspection on current main shows no persisted hooks field and no hook phase before executeJobCoreWithTimeout.

## Solution Assessment

Unclear for the exact proposed schema because this is active product and security design work. The narrow maintainable path is to review #45465 or a successor design that covers every cron execution path and defines safe user-script/webhook semantics before merge.

## Security Review

Status: needs_attention

Summary: The requested shell/webhook cron lifecycle is security-sensitive and needs explicit design review before implementation.

Concerns:

- **[medium] Define user hook execution boundaries:**
  - body: Cron shell and webhook hooks would execute local commands or outbound network calls around scheduled jobs. The design needs clear permission, timeout, logging, skip, and failure semantics so hooks do not become an implicit privileged execution path.
