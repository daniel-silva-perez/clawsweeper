## Summary

Keep this issue open. Current main still lacks a persisted per-job cron hook lifecycle: no `hooks` field in cron job types or Gateway schemas, no pre/post/onError execution phases around scheduled/startup/manual runs, and no CLI or Control UI editing surface. Existing plugin hooks (`before_agent_reply`, `cron_changed`) are useful adjacent workarounds, but they are not the requested job-defined cron hook contract. Open PRs #45129 and #45465 remain implementation candidates that need maintainer/security review rather than using this issue as cleanup-close material.

## Best Possible Solution

Keep this issue open until maintainers either land an additive first-class cron hook lifecycle across persisted cron types, Gateway protocol/schema/RPC, scheduler/manual/startup execution paths, CLI, Control UI, docs, and tests, or explicitly supersede it with a canonical narrower tracker. The existing plugin hook surfaces should be documented as partial workarounds, not treated as a replacement for job-defined cron hooks.
