## Summary

Keep this issue open. Current main still lacks persisted per-job cron hooks: cron job types and Gateway schemas have no hooks field, scheduled/startup/manual execution paths call the core job executor directly, and CLI/Control UI only expose existing schedule, payload, delivery, timeout, tools, and failure-alert controls. The issue also has concrete zero-token pre-gate use cases and two open implementation PRs, so cleanup closure would be premature.

## Best Possible Solution

Keep this issue open until maintainers either land an additive first-class cron hook lifecycle across persisted cron job types, Gateway schema/RPC, scheduled/startup/manual execution paths, CLI, Control UI, docs, and tests, or explicitly supersede it with a narrower canonical tracker. Existing plugin hooks are useful adjacent workarounds, but they do not replace the requested job-defined cron hook contract.

- Shell hooks and webhook hooks would introduce user-configured code execution and outbound network behavior before scheduled model calls, so timeout, approval/sandbox posture, logging, secret exposure, and SSRF-style review need maintainer attention.
- The requested surface spans persisted cron data, Gateway protocol, scheduler run paths, startup/manual execution, CLI, Control UI, docs, and tests, so this is broader than a narrow cleanup patch.
- Open PRs #45129 and #45465 already attempt overlapping implementations; opening a separate automated fix lane would likely duplicate or conflict with maintainer review.
