## Summary

Keep open. Current main still lacks a persisted cron hook lifecycle and still routes scheduled, startup catch-up, and manual cron execution directly into the core executor; the related open implementation candidate #45465 plus shell/webhook execution semantics make this active product and security design work, not cleanup-ready stale work.

## What This Changes

Request adding cron job pre/post/onError hooks that can run agent turns, system events, shell commands, or webhooks around cron execution for skip gates, retries, escalation, and conditional workflows.

## Best Possible Solution

Land or explicitly supersede #45465 with an additive cron hook lifecycle covering persisted job/config types, Gateway schemas and RPCs, scheduled/startup/manual execution paths, CLI, Control UI, docs, tests, and an operator-controlled security model for user-configured hooks.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. We do have a high-confidence verification path: source and schema inspection on current main shows no persisted hooks field and no hook phase before executeJobCoreWithTimeout.

## Solution Assessment

Unclear for the exact proposed schema because the issue spans product and security decisions. The narrow maintainable path is to review #45465 or a successor design that covers all cron execution paths and defines safe user-script/webhook semantics before merge.

## Security Review

Status: needs_attention

Summary: The requested shell/webhook cron lifecycle is security-sensitive and needs explicit design review before implementation.

Concerns:

- **[medium] Define user hook execution boundaries:**
  - body: Cron shell and webhook hooks would execute local commands or outbound network calls around scheduled jobs. The design needs clear permission, timeout, logging, skip, and failure semantics so hooks do not become an implicit privileged execution path.
