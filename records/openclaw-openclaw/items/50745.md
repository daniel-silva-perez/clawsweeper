## Summary

Keep open. Current main still lacks the requested WhatsApp lookup API, and this PR remains a focused implementation candidate, but the branch is stale and the proposed implementation still misses current authDir-first LID resolution, clobbers existing changelog entries, and needs maintainer privacy/API review before exposing phone-number lookup to plugins.

## What This Changes

The PR adds a WhatsApp `lookupPnLidEntry` listener API/type, implements LID and phone lookup in the web monitor, updates listener mocks and monitor tests, and changes the unreleased changelog.

## Best Possible Solution

Rebase the PR on current main, add the contract in the current `inbound/types.ts` surface, implement lookup through the shared authDir-first `jidToE164`/`resolveJidToE164` semantics, preserve all existing changelog entries, add cold-cache/authDir fallback coverage, and document or gate the sensitive lookup if maintainers approve the API.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with a `lid-mapping-*_reverse.json` authDir mapping and missing or empty `lidLookup`, the PR returns `null` at the LID branch while current `jidToE164` tests show the same LID should resolve.

## Solution Assessment

No. The API may be useful, but this implementation is not the best merge shape until it reuses the existing resolver semantics, repairs the changelog, and gets an explicit maintainer decision on exposing phone-number lookup to plugins.
