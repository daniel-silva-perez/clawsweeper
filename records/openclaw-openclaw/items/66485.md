## Summary

Keep this PR open: current main already implements the `/usage` context-percentage alignment with safer freshness semantics, but the compaction-history refresh requested by the PR is still missing. The PR should not merge as-is because it would reintroduce a `/usage` freshness regression and can reload the visible chat for a sessionless, unrelated compaction event.

## What This Changes

The PR changes Control UI `/usage` context percentage calculation and reloads chat history after completed compaction agent events.

## Best Possible Solution

Rebase or replace the PR with a narrow Control UI change that preserves current `/usage` logic and adds a compaction-complete history refresh only when the event is clearly for the active session or active run, backed by app-gateway tests for matching, unrelated, and missing-sessionKey cases.

## Reproduction Assessment

Yes. A focused static/event-path reproduction exists: a session row with valid `totalTokens` but omitted `totalTokensFresh` shows the PR's `/usage` regression, and a completed compaction `agent` event with no `sessionKey` shows the proposed chat reload can fire for the visible chat without proving ownership.

## Solution Assessment

No. The proposed direction is close, but the best fix is narrower: keep the shipped `/usage` snapshot logic and add compaction refresh with the same session/run scoping discipline used elsewhere in Control UI event handling.
