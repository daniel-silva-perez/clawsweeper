## Summary

Keep open. Current main contains the e311ffd materialization-superset fix and an Unreleased changelog entry, but the latest discussion includes a same-host retest where the published 2026.4.29 dist was patched with that predicate change and still reproduced gateway starvation plus recurring gateway-owned `pnpm install` children; source review also shows the synchronous per-plugin install fallback path remains present when materialization is not accepted.

## What This Changes

Fix the gateway runtime-deps convergence regression so bundled runtime dependencies install once, remain converged after Telegram starts, and do not repeatedly spawn high-CPU `pnpm install` children or starve the event loop after readiness.

## Best Possible Solution

Keep this as the canonical stable-release runtime-deps reinstall-loop report and add a regression that models the post-prestage lazy activation path that still invokes package-manager work; the shipped behavior should treat a converged package-level runtime-deps root as stable and avoid synchronous package-manager installs on gateway hot paths after readiness.

## Reproduction Assessment

Yes. The issue provides high-confidence VPS reproduction evidence for 2026.4.29, including process trees, runtime root state, liveness warnings, and repeated gateway-owned `pnpm install` PIDs; the later patched-dist retest gives a focused reproduction that e311ffd alone is insufficient, although this read-only review did not run the live VPS scenario.

## Solution Assessment

No. The e311ffd predicate change is a relevant partial fix, but the reporter's patched-dist retest indicates it is not the complete solution; the safer direction is to find and cover the additional caller or materialization failure that still reaches the synchronous install path.
