## Summary

Keep open. The report is a fresh, detailed stable-release regression with repeated gateway-owned `pnpm install` children after readiness, and current `main` has not changed the relevant runtime-deps install/materialization paths relative to `v2026.4.29`; the closed predecessor was superseded by this cleaner issue rather than resolved.

## What This Changes

The issue reports that OpenClaw 2026.4.29 on an Ubuntu npm-global gateway repeatedly spawns high-CPU runtime-deps `pnpm install` processes after the gateway is ready, causing event-loop starvation, RPC timeouts, and WebSocket handshake failures.

## Best Possible Solution

Keep this issue open as the canonical stable-release runtime-deps convergence bug, then make the package-level runtime-deps root converge exactly once: after successful pre-stage or repair, later bundled plugin activations should prove the same generated manifest and full dependency set are satisfied without launching another package-manager install, and gateway hot paths should avoid blocking installs after readiness.

## Reproduction Assessment

Yes. The issue provides a high-confidence host reproduction path against OpenClaw 2026.4.29 with process-tree, cwd, manifest, package-file, liveness, and recurring PID evidence; this read-only review did not run the live VPS scenario but confirmed current `main` still contains the same install/convergence code paths.

## Solution Assessment

No current solution is present on `main`. The narrow maintainable direction is to fix runtime-deps convergence and add regression coverage around repeated package-level/plugin-level activation after pre-staging, rather than treating this as Telegram-specific behavior.
