## Summary

Current main has two relevant post-release runtime-deps improvements, but the issue should stay open because the reporter reproduced the gateway-owned `pnpm install` recurrence and event-loop starvation after applying the first main fix to the published dist, and the later merged fast-path PR explicitly did not verify the live Linux npm-global Telegram repro or change package-manager behavior. The remaining symptom overlaps open gateway/runtime-mirror starvation reports rather than being proven implemented on main.

## What This Changes

The issue reports that OpenClaw 2026.4.29 repeatedly launches gateway-owned runtime-deps `pnpm install` children after readiness, starving the gateway event loop and causing RPC and WebSocket timeouts.

## Best Possible Solution

Keep this issue open as the canonical stable-release runtime-deps install recurrence report until maintainers either verify current main on the same VPS-style npm-global Telegram path or land a follow-up that proves runtime-deps convergence after startup. The end state should make bundled runtime-deps staging converge without later package-manager children and keep gateway RPC/WebSocket responsiveness healthy after channel startup.

Yes. The issue has a high-confidence live reproduction path on OpenClaw 2026.4.29 with gateway logs, process tree, child `pnpm install` commands, runtime root details, liveness warnings, and a same-host patched-dist retest showing e311ffd alone was insufficient.

## Solution Assessment

No. Current main contains useful fixture-backed fixes, but the latest known live evidence says the first fix was not sufficient and the later fast-path PR was not validated against the live Telegram/npm-global reproduction; the safer path is a focused follow-up or maintainer verification before closing.
