## Summary

Current main implements the specific runtime-deps reinstall-loop fix: generated package-level runtime-deps manifests now count as converged when they are supersets of a later lazy plugin request, so the later activation path no longer relaunches/prunes via package-manager install after startup pre-staging. The fix is on main only; no shipped tag in this checkout contains it yet.

## What This Changes

The issue reports that OpenClaw 2026.4.29 can reach gateway ready, then repeatedly spawn gateway-owned runtime-deps `pnpm install` children and starve local RPC/WebSocket handling after Telegram starts.

## Best Possible Solution

Keep the main-branch convergence fix, ship it in the next release, and route any remaining no-`pnpm` gateway event-loop starvation to the broader runtime-mirror performance work rather than reopening this reinstall-loop report.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction against the released 2026.4.29 package with process-tree, runtime root, liveness, and recurring install-child evidence; this read-only review verified the main-branch code and regression test for the reinstall-loop cause rather than rerunning the live VPS scenario.

## Solution Assessment

Yes for the reported reinstall loop. Accepting generated package-level manifest supersets is the narrow maintainable fix because it preserves dependency/version satisfaction checks while preventing later narrower lazy activations from pruning and relaunching the same runtime-deps install.
