## Summary

Keep open. Current main contains only the e311ffd generated-manifest superset fix for this runtime-deps loop, while the issue discussion includes a later controlled dist-patch experiment showing the gateway-owned `pnpm install` recurrence and event-loop starvation still reappeared with that predicate change applied; there is not enough evidence to close this as implemented.

## What This Changes

The issue reports that OpenClaw 2026.4.29 repeatedly launches gateway-owned runtime-deps `pnpm install` children and starves the gateway event loop after startup.

## Best Possible Solution

Keep this as an active runtime-deps convergence regression and make the package-level install root provably converge across startup pre-staging, later lazy plugin activation, and post-ready channel/plugin loads, with regression coverage that fails if a satisfied root can relaunch package-manager installs.

## Reproduction Assessment

Yes. The released-package reproduction is high confidence because the issue includes exact version, OS, service command, liveness/RPC logs, runtime root state, process tree, child process commands, and recurrence samples; this read-only review did not run the live VPS scenario.

## Solution Assessment

No. The current-main e311ffd predicate change is a useful partial fix, but the later issue comments show it is not sufficient by itself; the maintainable solution needs to find the remaining caller or convergence predicate that still reaches package-manager install after readiness.
