## Summary

Keep open. Current main has useful loop detection, including critical blocks for unknown-tool repeats, known polling no-progress, ping-pong no-progress, and the global no-progress breaker, but generic identical tool calls still warn first and only become blocked through the separate global no-progress threshold. That does not satisfy the requested low-threshold hard abort for `session_status`-style repeated identical calls, and related open reports #73763 and #72373 corroborate that the remaining behavior is still live.

## What This Changes

Request a configurable runtime guard that hard-aborts an agent run after a small number of identical tool calls instead of only injecting warning text into tool results.

## Best Possible Solution

Add a per-run generic identical-call hard-stop path that is configurable, no-progress aware, and produces a clear terminal user-visible failure rather than another warning the model can ignore. Cover `session_status` or an equivalent generic tool in regression tests, keep docs/config/schema aligned, and coordinate broader malformed OpenShell command validation under #72373.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction: with loop detection enabled, a generic repeated tool fixture keeps resolving through repeated identical calls below the global breaker, while only the separate global no-progress threshold causes rejection.

## Solution Assessment

No. The current implementation is not the best solution for this report because generic identical calls still rely on warning text and a high global breaker; a narrower, configurable critical abort for repeated no-progress generic calls would better match the reported queue wedge without relying on model self-restraint.

## Risks / Open Questions

- Changing generic repeated calls from warning to abort can false-positive legitimate repeated status, read, or polling workflows unless the rule is no-progress gated and configurable.
- The requested default threshold and whether loop detection should be enabled by default are product/runtime policy decisions with broad behavioral impact.
- The desired recovery semantics are not fully specified: blocking one tool call, returning a terminal run error, forcing a stop reason, and killing the session have different queue and transcript effects.
