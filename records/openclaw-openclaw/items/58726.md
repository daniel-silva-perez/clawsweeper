## Summary

Keep open. Current main has opt-in loop detection and can block known polling, ping-pong, unknown-tool, and no-progress loops at higher thresholds, but the specific requested behavior for generic identical tool calls is still warn-only until the separate global no-progress breaker threshold. The open related report #72373 also tracks hard-aborting repeated tool loops, but it is broader OpenShell/runtime work rather than a merged implementation or a narrower canonical replacement.

## Best Possible Solution

Implement a generic repeated identical tool-call hard-abort path in the agent tool loop, with configurable warning/abort thresholds matching the intended product default, user-visible terminal failure semantics, and regression tests for session_status or another generic tool. Keep #72373 available for the broader OpenShell malformed-command work, or split it after this focused loop-abort fix lands.

## Risks / Open Questions

- Changing generic repeated calls from warning to abort can false-positive legitimate repeated status/read checks; the product decision needs a clear threshold and escape hatch.
- The requested queue-unblock behavior should be implemented as a real turn/session abort or terminal tool-loop failure, not only another tool warning that the model can ignore.
- A fix should include coverage for a generic non-poll tool such as session_status, because current tests mainly protect warning behavior and no-progress breaker behavior.
