## Summary

Keep this PR open. Current main still does not implement MCP readiness metadata or prompt gating, so the PR is not obsolete, but the proposed readiness check has a blocking precedence bug: an explicit skill-level non-callable record can still be bypassed by a callable server-level record.

## What This Changes

The PR adds MCP metadata parsing, reads `mcp-readiness.json`, filters MCP-backed skills out of the advertised skills prompt until readiness says they are callable, adds focused skills coverage and a smoke script, and also touches qa-channel, package-boundary, gateway startup, cache-trace, and test plumbing.

## Best Possible Solution

Keep the PR under review if maintainers want this as core safety behavior: make skill-level readiness records authoritative, add regression coverage for denial precedence and owner scoping, decide and document readiness freshness semantics, keep the smoke script opt-in, and split or rebase unrelated gateway/qa-channel/package-boundary changes before merge.

## Reproduction Assessment

Yes. On the PR head, a readiness file with `skills["mcp-memory"] = { declared: true, callable: false }` and `servers["memory"] = { declared: true, callable: true }` follows the code path at `src/agents/skills/workspace.ts:175-180` and returns `true` for the skill.

## Solution Assessment

No. The core gate is a plausible direction, but this implementation is not the best merge shape until explicit skill-level denials win, stale-readiness behavior is settled, and unrelated surfaces are split or rebased.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Short-circuit explicit skill readiness denials:** `src/agents/skills/workspace.ts:176-180`
