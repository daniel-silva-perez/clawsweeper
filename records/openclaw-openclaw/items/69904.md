## Summary

Keep this PR open. Current main does not contain the proposed stale transient system-event policy or the session-updated-at filtering path, and the PR is still unmerged. The work may still be useful, but it needs review/revision because the current branch discussion flags a real mixed-queue risk around non-completion exec events.

## Best Possible Solution

Keep the PR open, update it on current main, and revise the heartbeat/system-event filtering so only explicitly transient success-style notices are dropped after newer session activity. The best implementation should add stale-policy support in `src/infra/system-events.ts`, pass current session activity into both reply and heartbeat drain paths, preserve non-completion exec/cron/durable failure events, and add regression tests for stale exec success, exec denied/started, mixed cron plus stale exec, and delivery-context preservation.
