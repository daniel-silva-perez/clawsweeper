## Summary

Keep this open. Current main still has the reported parser and restart/announce fallback behavior: agent-prefixed Slack channel keys are not parsed as Slack by `resolveGroupSessionKey()`, and the restart-sentinel route recovery still merges base-session routing when thread metadata is incomplete. The provided timeline also shows open PR #49750 as the active implementation candidate, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests stable A2A Slack thread routing by parsing `agent:<id>:slack:...` session keys as Slack and preventing announce/restart fallback from replacing a valid target route with base-session routing.

## Best Possible Solution

Land #49750 or an equivalent narrow fix that makes group-session resolution use the shared agent-aware conversation parser, preserves explicit A2A target delivery context, and changes restart/announce recovery so base-session data only fills genuinely missing fields on the same route. Add focused regression coverage for agent-prefixed Slack thread metadata and restart/announce delivery routing.

## Reproduction Assessment

Yes. The issue gives concrete session-key examples, and current main provides a high-confidence static reproduction: call `resolveGroupSessionKey()` with `From=agent:builder:slack:channel:C123:thread:T`, `Provider=webchat`, and `ChatType=channel`; the resolver still reads only the first segment and can return webchat.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix: reuse the existing agent-aware session-conversation parser and tighten restart/announce fallback merging instead of adding Slack-specific special cases throughout core routing.
