## Summary

Keep open. Current main fixes the album path-preservation part, but the reported behavior is not fully resolved: Telegram still mirrors local media paths into `MediaUrls`, the prompt note still renders `path | path` when URL equals path, and media understanding still selects only the first image by default. The prior ClawSweeper comment and Ryce's impact note are consistent with the current code: this is still a message assembly/media-understanding policy issue, not a download failure.

## What This Changes

Telegram multi-image messages should include every image path in the media header once and should transcribe every image instead of describing only the first image and listing the rest.

## Best Possible Solution

Keep the issue open and make the intended Telegram album behavior explicit. The end state should preserve all `MediaPaths`, suppress duplicate URL rendering when `MediaUrls[i]` equals `MediaPaths[i]`, and either make Telegram albums opt into all-image understanding by default with regression coverage or document/configure `tools.media.image.attachments.mode = "all"` as the supported path.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by inspection: a Telegram album produces multiple `MediaPaths`, mirrors those paths into `MediaUrls`, the prompt note renders both values, and media understanding selects only the first image unless configured otherwise. I did not run a live Telegram album because this was a read-only review.

## Solution Assessment

Unclear. The current implementation is not the complete fix because it still duplicates path/URL text and only describes the first image by default; the narrow safe fix is header de-duplication, while all-image default transcription needs maintainer policy judgment.
