## Summary

Keep open. Current main still has the targeted cooldown-probe bookkeeping behavior: a transient primary probe can mark the provider as used for the run, and a later same-provider fallback can then be skipped. The PR’s two-code-path fix is narrow, matches the documented model-failover contract, and has no concrete blocking review finding, but it needs maintainer refresh/revalidation against current main.

## What This Changes

The PR changes model fallback cooldown-probe bookkeeping so first-candidate primary probes do not check or consume the per-provider transient probe slot, and adds regression tests for same-provider fallback probing and non-primary deduplication.

## Best Possible Solution

Refresh or reapply the narrow primary-probe exemption on current main, preserving the existing one-probe-per-provider rule for non-primary fallbacks, and land it with regression coverage for both the same-provider-after-primary-probe scenario and non-primary same-provider deduplication.

## Reproduction Assessment

Yes. The current-main code path is enough to reproduce the bug at the unit level: a first-candidate transient cooldown probe can set transientProbeProviderForAttempt, the failed attempt adds that provider to cooldownProbeUsedProviders, and a later same-provider fallback is skipped by the provider-set check.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix: add the isPrimary guard only around transient provider-slot checking/consumption, while leaving cooldown eligibility, primary probe throttling, and non-primary fallback deduplication intact.
