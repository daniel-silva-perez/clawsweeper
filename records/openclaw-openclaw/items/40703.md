## Summary

Keep this PR open. Current main still lacks the requested default-model auth guard before channel startup, so the work is not obsolete, but the branch should not merge unchanged because it targets the old startup layout and checks a narrower auth contract than runtime/model-listing code now uses.

## What This Changes

The PR adds a gateway startup auth guard and tests that fail before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

Integrate the guard in the current post-attach sidecar startup path before `startChannels()`, preserve `OPENCLAW_SKIP_CHANNELS` and `OPENCLAW_SKIP_PROVIDERS`, base eligibility on the same non-mutating auth availability contract used by current model visibility/runtime auth, and cover missing auth plus env/config/profile/AWS/local/plugin auth modes with focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use runtime auth availability before blocking startup:** `src/gateway/server-startup.ts:47-50`
  - body: The new startup gate calls `hasAuthForProvider` with only `provider` and `agentDir`, but that helper only checks env keys and auth-profile presence. Runtime auth also accepts config-backed provider keys, AWS SDK Bedrock auth, and synthetic local/plugin auth, so this can reject valid setups before channels start.
  - confidence: 0.9
- **[P3] Avoid suggesting the failing default model:** `src/gateway/server-startup.ts:61`
  - body: The error text always suggests `openclaw models set "openai-codex/gpt-5.4"`. When that is already the default model without auth, the remediation points users back to the same failing configuration.
  - confidence: 0.82
