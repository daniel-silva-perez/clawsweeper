## Summary

Keep this PR open. Current main still does not fail fast before channel startup when the default model provider lacks auth, but the proposed patch is not merge-ready because it uses a narrower helper than the runtime auth contract and is behind the current startup split.

## What This Changes

The PR adds a gateway startup auth guard and tests that throw before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

If maintainers want fail-fast behavior, add the guard in `server-startup-post-attach.ts` immediately before channel startup, honor both channel/provider skip flags, use `hasAuthForModelProvider` or the cached provider-auth checker with config/workspace/env/agent-dir context, avoid circular remediation text, and cover env, config, profile, AWS, local, and plugin synthetic auth modes.

## Reproduction Assessment

Yes for the PR defects. Static inspection of the diff and current auth helpers shows the proposed gate can reject auth modes accepted by current runtime auth resolution, and the hardcoded example becomes circular when the failing default is already `openai-codex/gpt-5.4`.

## Solution Assessment

No. The goal is plausible, but this implementation patches an obsolete startup surface and uses an auth helper that is intentionally narrower than the runtime provider-auth contract; a rebased post-attach guard using the existing provider-auth availability seam is safer.

## Review Findings
