## Summary

Keep this PR open. Current main still does not implement the proposed default-model auth fail-fast guard, and the PR remains a plausible bug-fix candidate for the linked startup/auth symptom. It should be revised before landing because startup moved into the post-attach module, recent main explicitly starts channels without waiting for model prewarm, and the PR's auth check is narrower than current runtime auth resolution.

## Best Possible Solution

Keep this PR open for revision rather than closing it. Rebase or rewrite the behavior into src/gateway/server-startup-post-attach.ts, decide explicitly how it should coexist with non-blocking channel startup, preserve OPENCLAW_SKIP_CHANNELS and OPENCLAW_SKIP_PROVIDERS bypasses, and base any guard on the same non-mutating runtime auth contract used by model execution, such as hasAvailableAuthForProvider or a safe resolveApiKeyForProvider probe with cfg/store/agentDir. Add coverage for missing auth, config-backed API keys, Bedrock/aws-sdk, local/plugin synthetic auth, skip bypasses, and non-circular remediation text.

## Risks / Open Questions

- The review discussion's circular remediation-text bug remains unresolved in the PR head.
- A maintainer product decision may be needed on whether missing default-model auth should block channel startup, warn during readiness, or be handled at first model turn, given recent main moved toward prompt channel readiness.
