## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested fail-fast default-model auth guard, so the PR is not obsolete, but the branch is stale against the split startup path and the proposed guard uses a narrower auth helper than runtime model resolution.

## What This Changes

The PR adds a gateway startup auth guard and tests that throw before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

If maintainers want the fail-fast behavior, implement the guard immediately before channel startup in server-startup-post-attach, honor both channel/provider skip flags, use hasAuthForModelProvider or createProviderAuthChecker with cfg/workspace/env/agent-dir context, avoid circular remediation text, and cover env, profile, config, AWS, local, plugin synthetic auth, and skip-flag cases.

## Reproduction Assessment

Yes for the PR defects. Static inspection shows the PR's helper rejects auth modes that runtime accepts, and the hardcoded remediation becomes circular when the failing default is already openai-codex/gpt-5.4.

## Solution Assessment

No. The target behavior is plausible, but this patch is not the best solution because it targets the old startup surface and uses a narrower helper than runtime auth resolution; a post-attach guard using the existing provider-auth availability seam is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use runtime auth availability before blocking startup:** `src/gateway/server-startup.ts:47-50`
  - body: The new gate calls hasAuthForProvider with only provider and agentDir, but that helper only checks env keys and stored profiles. Valid runtime auth through config/workspace keys, AWS SDK Bedrock auth, local synthetic auth, or plugin synthetic auth can be rejected before channels start.
  - confidence: 0.9
- **[P3] Avoid suggesting the failing default model:** `src/gateway/server-startup.ts:61`
  - body: The error text always suggests openclaw models set "openai-codex/gpt-5.4". When that is already the default model whose provider lacks auth, the remediation points users back to the same broken configuration.
  - confidence: 0.82
