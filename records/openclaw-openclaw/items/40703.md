## Summary

Keep this PR open. Current main still does not implement the requested fail-fast default-model auth guard, but the PR is not mergeable as-is because it targets the old startup file shape and checks a narrower auth helper than the current runtime auth contract, which would reject valid auth setups.

## What This Changes

The branch adds a gateway startup helper and tests to fail before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

If maintainers want this behavior, implement the guard in `server-startup-post-attach.ts` immediately before channel startup, honor both channel/provider skip flags, use `hasAuthForModelProvider` or the cached provider-auth checker with config, workspace, env, and agent-dir context, avoid circular remediation text, and cover missing auth plus env, config, profile, AWS, local, and plugin synthetic auth modes.

## Reproduction Assessment

Yes for the PR defects. Static inspection shows the PR's `hasAuthForProvider` call can reject auth modes accepted by current runtime auth checks, and the hardcoded example is circular when the failing default is already `openai-codex/gpt-5.4`.

## Solution Assessment

No. The goal is useful, but the proposed patch is not the best implementation because it patches the old startup surface and uses a helper that is intentionally narrower than the runtime auth contract; the safer path is a rebased post-attach guard using the existing provider-auth availability seam.

## Review Findings

Overall correctness: patch is incorrect

Valid setups using config or workspace keys, AWS SDK Bedrock auth, synthetic local provider auth, or plugin synthetic auth can be rejected before channels start.
  - confidence: 0.9

  - body: The error text always suggests `openclaw models set "openai-codex/gpt-5.4"`. When that is already the default model whose provider lacks auth, the remediation points users back to the same broken configuration.
  - confidence: 0.82
