## Summary

Keep this PR open as an implementation candidate, but it should not merge as written. Current main still lacks a fail-fast startup auth guard, while the PR branch is stale against the split startup surface and its proposed guard uses a narrower auth helper than the runtime/provider-auth contract, which can block valid configurations.

## What This Changes

The PR adds a gateway startup guard and focused tests that throw before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

If maintainers want fail-fast behavior, add the guard in `server-startup-post-attach.ts` immediately before channel startup, honor both channel/provider skip flags, use `hasAuthForModelProvider` or the cached provider-auth checker with config/workspace/env/agent-dir context, avoid circular remediation text, and cover env, config, profile, AWS, local, and plugin synthetic auth modes.

## Reproduction Assessment

Yes for the PR defects. Static inspection shows a config/workspace/AWS/local/plugin-auth setup can be accepted by runtime auth checks while the PR's `hasAuthForProvider` gate rejects it, and the hardcoded example becomes circular when the failing default is already `openai-codex/gpt-5.4`.

## Solution Assessment

No. The target behavior is plausible, but this patch modifies an obsolete startup file and uses a helper that is intentionally narrower than runtime provider-auth resolution; a rebased post-attach guard using the existing provider-auth availability seam is safer.
