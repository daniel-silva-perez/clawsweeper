## Summary

Keep this PR open. Current main still does not implement the PR's intended fail-fast startup guard before channel startup, but the proposed branch is not correct as-is because it targets the old startup file shape and uses a narrower auth helper than the current runtime auth availability contract.

## What This Changes

The PR adds a gateway startup helper and focused tests to error before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

If maintainers want this behavior, put the guard in the current post-attach sidecar path immediately before channel startup, honor both channel/provider skip flags, use the full `hasAuthForModelProvider` or cached equivalent with config, workspace, env, and agent-dir context, avoid circular remediation text, and add focused coverage for missing auth plus env, config, profile, AWS, local, and plugin synthetic auth modes.

## Reproduction Assessment

Yes for the PR defects. Static inspection shows configurations accepted by `hasAuthForModelProvider` such as local no-key providers or implicit Bedrock auth can be rejected by the PR's `hasAuthForProvider` call, and the circular remediation text follows directly when the missing-auth default is already `openai-codex/gpt-5.4`.

## Solution Assessment

No. The goal is useful, but this PR is not the best implementation because it patches the old startup file shape and checks a helper that is intentionally narrower than the current runtime auth contract; the safer path is a rebased guard in `server-startup-post-attach.ts` using the existing provider-auth availability seam.
