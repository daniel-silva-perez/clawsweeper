## Summary

Keep this PR open. Current main still lacks the proposed fail-fast default-model auth guard before channel startup, so the work is not implemented on main, but the branch is stale against the current split startup path and introduces a blocking false-negative auth check by using a narrower helper than current runtime/model auth availability semantics.

## What This Changes

The PR adds a gateway startup auth guard and focused tests intended to fail before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

Revise the guard into `src/gateway/server-startup-post-attach.ts` immediately before `startChannels()`, preserve `OPENCLAW_SKIP_CHANNELS` and `OPENCLAW_SKIP_PROVIDERS`, use the current non-mutating auth availability contract with config/workspace/agent context, avoid circular remediation text, and cover missing auth plus env/config/profile/AWS/local/plugin auth modes in focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use runtime auth availability before blocking startup:** `src/gateway/server-startup.ts:47-50`
  - body: The startup gate calls `hasAuthForProvider` with only `provider` and `agentDir`, but that helper only checks env keys and auth-profile presence. Current runtime auth also accepts config-backed provider keys, AWS SDK Bedrock auth, synthetic local auth, and plugin synthetic auth, so this can reject valid setups before channels start.
  - confidence: 0.9
- **[P3] Avoid suggesting the failing default model:** `src/gateway/server-startup.ts:61`
  - body: The error text always suggests `openclaw models set "openai-codex/gpt-5.4"`. When that is already the default model without auth, the remediation points users back to the same failing configuration.
  - confidence: 0.82
