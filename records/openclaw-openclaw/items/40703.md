## Summary

Keep this PR open. Current main still lacks the requested fail-fast default-model auth guard before channel startup, so the PR is not obsolete, but it should not land unchanged because it targets the old startup layout and uses a narrower auth check than runtime model execution accepts.

## What This Changes

The PR adds a gateway startup auth guard and tests that fail before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

Revise the PR into the current post-attach startup path before channel startup, preserve the `OPENCLAW_SKIP_CHANNELS` and `OPENCLAW_SKIP_PROVIDERS` bypasses, and base the guard on the same non-mutating auth availability contract used by runtime model execution. The finished change should cover missing auth, valid config/env/profile/AWS/local/plugin auth modes, and non-circular remediation text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use runtime auth availability before blocking startup:** `src/gateway/server-startup.ts:47-50`
  - body: The new startup gate calls `hasAuthForProvider` with only `provider` and `agentDir`, but that helper only checks env keys and auth-profile presence. Runtime auth also accepts config-backed provider keys, AWS SDK Bedrock auth, and synthetic local/plugin auth, so this can reject valid setups before channels start.
  - confidence: 0.9
- **[P3] Avoid suggesting the failing default model:** `src/gateway/server-startup.ts:61`
  - body: The error text always suggests `openai-codex/gpt-5.4`. When that is already the default model without auth, the remediation points users back to the same failing configuration.
  - confidence: 0.82
