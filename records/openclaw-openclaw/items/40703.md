## Summary

Keep this PR open. Current main still does not fail fast on missing default model auth before channel startup, but the proposed patch is not merge-ready because it targets the old startup file shape and uses a narrower auth helper than the current runtime/model-picker contract, which can reject valid config, AWS, local-provider, and plugin-auth setups.

## What This Changes

The PR adds a gateway startup guard and focused tests to error before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

If maintainers still want fail-fast startup behavior, place the guard in the current post-attach sidecar path immediately before channel startup, honor both channel/provider skip flags, use the full runtime auth availability contract with config, workspace, env, and agent context, avoid circular remediation text, and cover missing auth plus env, config, profile, AWS, local, and plugin auth modes in focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use runtime auth availability before blocking startup:** `src/gateway/server-startup.ts:47-50`
  - body: The new startup gate calls `hasAuthForProvider` with only `provider` and `agentDir`, but that helper only checks env keys and stored profiles. Valid setups using configured provider keys, AWS SDK Bedrock auth, local-provider markers, or plugin synthetic auth can be rejected before channels start.
  - confidence: 0.9
- **[P3] Avoid suggesting the failing default model:** `src/gateway/server-startup.ts:61`
  - body: The error text always suggests `openclaw models set "openai-codex/gpt-5.4"`. When that is already the default provider/model without auth, the remediation points users back to the same broken configuration.
  - confidence: 0.82
