## Summary

Keep open. Current main still gives regular Telegram messages the ack reaction and early typing path, but native Telegram slash commands are handled by separate `bot.command(...)` handlers that do not receive or call those feedback dependencies. The open fork PR linked from the issue is an implementation candidate, so this should stay open until a maintainer reviews, upstreams, or closes that work.

## What This Changes

Telegram slash commands such as `/new` and `/reset` should emit the same configured ack reaction and typing feedback as regular inbound Telegram messages.

## Best Possible Solution

Add a Telegram-plugin-local shared feedback helper or equivalent dependency threading into native command registration, emit ack and typing after skip/auth checks, honor `ackReactionScope` and thread IDs, and cover both built-in and plugin-backed native commands with focused tests plus a small docs/changelog update.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: regular Telegram messages build a feedback-capable context, while native Telegram slash commands execute in separate handlers with no ack or typing calls.

## Solution Assessment

Yes. The narrowest maintainable fix is to emit the same feedback signals in the Telegram native command path after skip/auth checks; routing commands through the full inbound message pipeline would couple command parsing, menus, and plugin command execution to regular message dispatch unnecessarily.
