## Summary

Keep open. Current main still has the reported Telegram behavior gap: regular message handling wires ack reactions and typing through the Telegram message context, while native slash-command handlers bypass that context and dispatch command replies directly. The provided GitHub context also shows an open, unmerged fork PR that explicitly tries to close this issue, so this should remain open for upstream review/implementation rather than maintainer cleanup closure.

## Best Possible Solution

Implement this in the bundled Telegram plugin: thread `ackReactionScope` and `sendChatActionHandler` into native command registration, add a shared helper that emits ack reaction and typing after skip/auth checks, call it from both built-in and plugin command handlers, and add focused tests for `/new`-style direct commands, group mention scopes, `off`/`none`, thread IDs, and API failure handling. The open fork PR can be reviewed or upstreamed as an implementation candidate; close this issue only after an upstream merge or equivalent main-branch fix.

## Risks / Open Questions

- A fix should emit feedback only after `shouldSkipUpdate` and authorization checks, so ignored or unauthorized command attempts do not get misleading acknowledgement signals.
- Both Telegram native built-in commands and plugin-backed native command handlers need coverage because they are separate paths.
- Telegram API failures for reactions or typing should stay fire-and-forget/logged so command execution is not made less reliable.
