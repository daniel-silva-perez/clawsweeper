## Summary

Keep open. Current main still routes Telegram native slash commands through command-specific handlers that do not receive or call the ack reaction or typing feedback dependencies, while regular inbound messages do. The linked fork PR is open and unmerged, so upstream review or an equivalent main-branch fix is still needed.

## What This Changes

Request Telegram slash commands such as `/new` and `/reset` to emit the same ack reaction and typing feedback as regular inbound messages under the configured Telegram reaction and typing settings.

## Best Possible Solution

Implement this inside the bundled Telegram plugin with a shared feedback helper or equivalent dependency threading into native command registration. It should preserve the current command architecture, honor `ackReactionScope` and thread IDs, cover both built-in and plugin-backed native commands, and add focused tests plus a short docs/changelog note.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: regular Telegram messages build a feedback-capable message context, but Telegram native slash commands execute in separate `bot.command(...)` handlers with no ack or typing hooks.

## Solution Assessment

Yes, the narrowest maintainable fix is to emit the same ack and typing signals in the Telegram native command path after skip/auth checks. Re-routing commands through the full inbound message pipeline would be riskier because it couples command argument parsing, menus, and plugin command execution to regular message dispatch.
