## Summary

Keep open. Current main still has the old Active Memory setup-grace timeout behavior and constructs the full embedded tool set before applying the runtime allowlist, so this PR is not obsolete; the patch targets the relevant paths and I found no discrete blocking correctness or security issue in the diff. It still needs maintainer review and CI, especially because it uses closing syntax for the broader open Active Memory latency issue.

## What This Changes

The PR threads runtime `toolsAllow` into embedded/core/plugin tool construction, skips plugin tool factories that cannot match the runtime allowlist, and updates Active Memory timeout handling/tests to remove the 30s setup grace.

## Best Possible Solution

Land this PR or an equivalent narrow patch if CI passes, while preserving the final runtime allowlist as the authorization boundary. If the broader linked issue still covers non-blocking Active Memory or cold-start readiness, keep that follow-up tracked separately rather than treating this startup fix as the whole product answer.

## Reproduction Assessment

Yes. The linked issue gives direct-chat Active Memory steps and timeout logs, and current main shows the two relevant mechanics: setup grace extends the blocking budget and runtime `toolsAllow` is applied only after full tool construction.

## Solution Assessment

Yes for this PR's narrow startup and timeout-grace problem. It is not necessarily the complete solution for the broader direct-chat reliability issue, where non-blocking recall or prewarm/readiness behavior may still be needed.
