## Summary

Current main still has the Feishu-facing presentation gap. The raw JSON envelope is parsed away, but provider-shaped `LLM error server_error: ...` text remains user-facing, post-stream error payloads are still preserved for delivery, and Feishu dispatch sends the resolved text without a channel-safe rewrite. Keep open; there is also overlapping open PR work that should be coordinated rather than duplicated.

## What This Changes

Feishu users should see generic channel-safe copy for provider `server_error` failures, raw provider details/request IDs should stay diagnostic-only, and retries should happen only before visible output is delivered.

## Best Possible Solution

Land one canonical fix, likely by extending or superseding the overlapping provider-error PR, that gives external chat channels generic copy for internal/server provider failures, keeps request IDs/provider payloads in diagnostics only, and suppresses or genericizes final assistant error payloads after visible assistant content while preserving the existing pre-output retry behavior.
