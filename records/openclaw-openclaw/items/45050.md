## Summary

Current main still has the Feishu-facing provider-error presentation gap. The raw JSON envelope is parsed away, but user-facing paths still preserve provider-shaped `LLM error server_error: ...` text, keep provider message details visible, retain post-stream error payloads, and send the resolved text through Feishu without a channel-safe rewrite. The pre-output retry path already exists, so the remaining work is a focused presentation/suppression fix.

## What This Changes

The issue requests Feishu-safe handling of transient provider `server_error` failures by hiding provider details from users and retrying only before visible output is delivered.

## Best Possible Solution

Add an external-chat error presentation policy so transient/internal provider failures become generic user copy in Feishu, provider details remain diagnostic-only, and final assistant error payloads after already-visible assistant content are suppressed or genericized while preserving the existing pre-output retry behavior.
