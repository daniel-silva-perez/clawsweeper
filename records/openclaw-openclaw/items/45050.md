## Summary

Keep open. Current main still formats Codex/provider JSON `server_error` payloads into provider-shaped user-facing text, can preserve a late error payload after visible streaming output, and the Feishu dispatcher sends the resolved payload text without a channel-specific generic rewrite. The existing pre-output transient HTTP retry is present, and #49401 overlaps, but that PR is still open and does not settle the post-visible-output Feishu path or request-ID policy.

## What This Changes

Feishu users should receive a generic safe failure message for provider `server_error` failures, with raw provider payloads and request IDs kept in logs and optional retry only before visible output is delivered.

## Best Possible Solution

Land one canonical fix that gives external chat channels generic copy for internal/server provider failures, keeps provider payloads and request IDs in diagnostics only, and suppresses or genericizes final assistant error payloads after visible assistant content while preserving the existing pre-output retry behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists through the shared formatter, embedded payload builder, post-stream payload filtering, and Feishu dispatch path; no live Feishu/provider outage is needed to see the current delivery behavior.

## Solution Assessment

No current fix is present on main. The narrow maintainable solution should coordinate with #49401, but must also cover late post-visible-output error payloads and decide that request IDs stay out of chat replies.
