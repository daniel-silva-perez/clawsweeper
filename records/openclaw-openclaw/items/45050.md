## Summary

Keep open. Current main still converts Codex/provider `server_error` payloads into provider-shaped user-facing text, still preserves late `isError` payloads after successful block streaming, and Feishu dispatch sends the resolved payload text without a channel-safe generic rewrite. The pre-output transient HTTP retry exists, and #49401 overlaps, but that PR is open, explicitly preserves request IDs in displayed copy, and does not touch Feishu dispatch or post-visible-output behavior.

## What This Changes

The issue requests Feishu-safe handling for provider `server_error` failures: generic user-facing copy, raw payload/request IDs retained only in diagnostics, and at most one retry before visible output.

## Best Possible Solution

Land one canonical fix that gives external chat channels generic copy for internal/server provider failures, keeps provider payloads and request IDs in diagnostics only, and suppresses or genericizes final assistant error payloads after visible assistant content while preserving the existing pre-output retry behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists through the shared formatter, embedded payload builder, post-stream payload filter, and Feishu dispatcher; no live Feishu outage is needed to verify the current delivery path.

## Solution Assessment

No current fix is present on main. The maintainable path should coordinate with #49401, but it also needs to cover late post-visible-output error payloads and settle that provider request IDs stay out of chat replies.
