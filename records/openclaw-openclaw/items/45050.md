## Summary

Keep open. Current main still has the Feishu-facing presentation gap: raw JSON envelopes are parsed away, but provider-shaped `LLM error server_error: ...` text remains user-facing, post-stream error payloads are still preserved for delivery, and Feishu dispatch sends the resolved text without a channel-safe rewrite. The overlapping open provider-error PR should be coordinated rather than duplicating work.

## What This Changes

The issue requests Feishu-safe handling for provider `server_error` failures so raw/internal payload details stay out of user-visible replies, with a safe retry only before visible output when possible.

## Best Possible Solution

Land one canonical fix that gives external chat channels generic copy for internal/server provider failures, keeps request IDs/provider payloads in diagnostics only, and suppresses or genericizes final assistant error payloads after visible assistant content while preserving the existing pre-output retry behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the Codex `server_error` shape is normalized to provider-shaped `LLM error server_error: ...`, `buildEmbeddedRunPayloads` emits it as an error payload, `buildReplyPayloads` preserves post-stream errors, and Feishu dispatch sends the resolved text.

## Solution Assessment

No current fix is present on main. The narrowest maintainable path is to adjust the shared formatter/payload filtering and Feishu-facing dispatch behavior, while coordinating with PR #49401 because that branch overlaps but does not satisfy the request as-is.
