## Summary

Keep this issue open. Current main fixes the worst raw JSON/request-id leak by parsing provider error payloads, but it still intentionally renders the reported Codex `server_error` shape as provider-shaped user text and still allows final error payloads after visible output paths instead of the requested generic Feishu-safe fallback.

## What This Changes

Feishu should sanitize provider `server_error` failures, keep raw details in logs, and only retry transient provider failures before any visible output has been delivered.

## Best Possible Solution

Add a focused shared reply/error policy that treats transient/internal provider failures as generic for external chat surfaces, keeps raw provider details in diagnostics, and suppresses or genericizes final assistant error payloads once user-visible content has already been delivered. Feishu should get coverage through the generic channel path plus a Feishu/originating-channel regression for the reported Codex `server_error` shape.
