## Summary

Keep this issue open. Current main fixes the worst raw JSON/request-id exposure by parsing provider error payloads, but the reported Feishu UX is still not implemented: `server_error` remains rendered as provider-shaped user text, final error payloads are preserved after visible output paths, and Feishu delivery forwards the resolved reply text without a channel-specific generic fallback.

## What This Changes

The issue requests Feishu to hide raw provider `server_error` payloads/request IDs behind generic failure copy and only retry transient failures before any visible output has been delivered.

## Best Possible Solution

Add a focused shared reply/error presentation policy for external chat surfaces: transient/internal provider failures should become generic user copy, raw provider details should stay in diagnostics, and final assistant error payloads after already-visible assistant content should be suppressed or genericized. Feishu should get a regression through its normal dispatcher path, with existing raw JSON sanitization preserved.
