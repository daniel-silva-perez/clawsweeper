## Summary

Keep this open. Current main has partial raw-JSON cleanup and the pre-output transient retry path exists, but Feishu can still receive provider-shaped `server_error` copy after visible assistant output because the shared formatter returns `LLM error server_error: ...`, embedded payload construction emits it as an `isError` reply item, and the Feishu dispatcher forwards resolved text without a channel-safe fallback.

## What This Changes

Request Feishu chat replies to convert transient/internal provider `server_error` payloads into generic user-safe failure copy, keep raw details in logs, and only retry before visible output is delivered.

## Best Possible Solution

Add a focused external-chat error presentation policy so transient/internal provider failures become generic user copy, raw provider details and request IDs remain diagnostic-only, and final assistant error payloads after already-visible assistant content are suppressed or genericized. Cover the shared formatter/payload behavior and the Feishu dispatcher path with regression tests while preserving existing diagnostics and pre-output retry behavior.
