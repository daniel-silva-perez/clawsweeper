## Summary

Keep open: current main still preserves session provenance internally but hard-codes every surfaced built-in `memory_search` hit to `corpus: "memory"`, and the latest release has the same mapper. The issue is also already paired with open fix PR #72886 using closing syntax and overlapping open PR #71898, so it should stay open until maintainers choose and land a focused fix.

## What This Changes

The issue asks for built-in `memory_search` session-transcript hits to surface `corpus: "sessions"` instead of being flattened to `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core fix that derives the surfaced built-in result corpus from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, or recall tracking.

## Reproduction Assessment

Yes: a high-confidence source-level reproduction is to mock `memory.manager.search` so it returns a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`. Current main’s mapper still returns `corpus: "memory"` for that surfaced hit.

## Solution Assessment

Yes: deriving the surfaced corpus from the existing `result.source` is the narrowest maintainable fix because the backend already records provenance and the public schema already accepts `corpus: "sessions"`. The KinthAI comment’s stricter index-time tagging model is useful context, but it is broader than needed for this contract bug.
