## Summary

Keep open: current main still preserves `result.source` but hard-codes every surfaced built-in `memory_search` hit to `corpus: "memory"`, so session transcript results remain mislabeled. Open PR #72886 uses closing syntax for this issue, and #71898 targets the same bug, so the right maintainer action is to review or choose one focused fix rather than close the report now.

## What This Changes

The issue requests that built-in `memory_search` surface session-transcript hits with `corpus: "sessions"` when the underlying memory result has `source: "sessions"`.

## Best Possible Solution

Land one focused memory-core change that widens surfaced built-in result typing to `corpus: "memory" | "sessions"`, derives corpus from `result.source` after filtering and decoration, and adds regression coverage for session-only, memory-only, and mixed result sets without changing visibility filtering, citation decoration, wiki supplement ranking, or recall tracking.

## Reproduction Assessment

Yes: mock `memory.manager.search` to return a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`. Current main’s mapper still returns `corpus: "memory"` for that hit.

## Solution Assessment

Yes: deriving the surfaced corpus from the existing `result.source` is the narrowest maintainable fix because the backend already records provenance and the public schema already accepts `corpus: "sessions"`. A broader index-time tagging redesign is not required for this bug.
