## Summary

Current main still has the reported bug: the built-in memory_search path preserves result.source but hard-codes every surfaced memory-manager hit to corpus "memory". The issue should stay open because open PR #72886 uses closing syntax for this issue, #71898 also targets the same fix, and #72926 was closed unmerged; maintainers should choose or land one focused fix rather than close the report now.

## What This Changes

The issue asks memory_search to surface session-transcript hits with corpus "sessions" when the underlying MemorySearchResult.source is "sessions".

## Best Possible Solution

Land one focused memory-core change that widens the surfaced built-in result corpus type to "memory" | "sessions" and derives corpus from result.source after filtering and decoration, with regression tests for session-only, memory-only, and mixed result sets while leaving visibility, recall tracking, wiki supplements, and ranking unchanged.

## Reproduction Assessment

Yes: mock memory.manager.search to return a MemorySearchResult with source "sessions", execute memory_search, and inspect details.results. Current main’s mapper at extensions/memory-core/src/tools.ts:281-284 will still return corpus "memory" for that hit.

## Solution Assessment

Yes: fixing the tool-surface mapping is the narrowest maintainable solution because the backend already records source provenance and the public schema already accepts corpus "sessions". A broader index-time retagging design is not required for this bug.
