## Summary

Current main still reproduces the reported provenance flattening: built-in `memory_search` results keep `source: "sessions"` internally but are surfaced with `corpus: "memory"`. This should stay open because open PR #72886 is explicitly paired with the issue and has not merged, while the narrower related PR #72926 was closed unmerged; the KinthAI discussion also reinforces that downstream consumers rely on strict result provenance.

## What This Changes

`memory_search` should surface session-transcript hits with `corpus: "sessions"` instead of flattening them to `corpus: "memory"`.

## Best Possible Solution

Land a focused `memory-core` change that widens the surfaced built-in result type to allow `corpus: "sessions"`, derives corpus from `result.source` after visibility filtering, and adds tests for session-only, memory-only, and mixed built-in result sets while leaving source filtering, citations, recall tracking, and wiki supplement behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence reproduction can be made by mocking `memory.manager.search` to return a `MemorySearchResult` with `source: "sessions"` and calling `memory_search`; current main’s final mapping at `extensions/memory-core/src/tools.ts:281` surfaces that hit with `corpus: "memory"`.

## Solution Assessment

Yes. Mapping the surfaced corpus from the already-preserved `result.source` is the narrowest maintainable fix because the backend and schema already distinguish `memory` and `sessions`; #72886 should be reviewed or replaced with a smaller equivalent patch if its branch is too broad.
