## Summary

Current main still flattens every built-in memory_search hit to corpus "memory" even when the retained source is "sessions". The issue is also paired with open PR #72886, while the narrower #72926 was closed unmerged, so this should stay open until a fix is reviewed and merged.

## What This Changes

The request is to have memory_search surface session-transcript hits as corpus "sessions" when MemorySearchResult.source is "sessions", while keeping durable memory-file hits as corpus "memory" and adding regression coverage.

## Best Possible Solution

Land a focused memory-core change that widens the surfaced built-in result corpus type to "memory" | "sessions" and derives corpus from result.source after visibility filtering, with tests covering session-only, memory-only, and mixed result sets. Keep session visibility, source filtering, and supplement/wiki behavior unchanged.

## Reproduction Assessment

Yes: a high-confidence reproduction can be made with a mocked memory.manager.search result whose source is "sessions" and then calling memory_search. Current main’s final mapping at extensions/memory-core/src/tools.ts:281 returns corpus "memory" for that mocked hit.

## Solution Assessment

Yes: mapping corpus from result.source at the tool-surface boundary is the narrowest maintainable fix because the backend already carries source provenance and the schema already accepts corpus "sessions". The practical next step is to review #72886 or replace it with the smaller #72926-style diff if the open branch is too broad.
