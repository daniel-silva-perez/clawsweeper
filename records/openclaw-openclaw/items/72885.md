## Summary

Keep open: current main still preserves session provenance internally but hard-codes every surfaced built-in `memory_search` result to `corpus: "memory"`, and the latest shipped release has the same mapper. Because open PR #72886 uses closing syntax for this issue and open PR #71898 targets the same bug, the report should remain open until maintainers review or choose one focused fix.

## What This Changes

The issue asks for transcript-backed `memory_search` results to surface `corpus: "sessions"` when the underlying hit has `source: "sessions"`.

## Best Possible Solution

Land one focused memory-core change that widens surfaced built-in result typing to `corpus: "memory" | "sessions"`, derives the public corpus from `result.source` after existing filtering and decoration, and adds regression coverage for session-only, memory-only, and mixed result sets without changing visibility filtering, citation decoration, ranking, or recall tracking.

## Reproduction Assessment

Yes: a high-confidence source-level reproduction is to mock `memory.manager.search` so it returns a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`. Current main’s mapper still returns `corpus: "memory"` for that surfaced hit.

## Solution Assessment

Yes: deriving the surfaced corpus from the existing `result.source` is the narrowest maintainable fix because the backend already records provenance and the schema already accepts `corpus: "sessions"`. The broader index-time tagging idea from the KinthAI comment is useful context, but it is not required to fix this specific contract mismatch.
