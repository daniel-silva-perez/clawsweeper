## Summary

Keep open: current main still preserves session provenance internally but flattens every surfaced built-in `memory_search` hit to `corpus: "memory"`. The issue is valid, and an open PR uses closing syntax for it while another open PR targets the same narrow fix, so the issue should remain open until maintainers choose and land one implementation.

## What This Changes

`memory_search` should label session-transcript hits as `corpus: "sessions"` instead of surfacing them as durable memory hits with `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core change that widens surfaced built-in result typing to `corpus: "memory" | "sessions"`, derives the public corpus from `result.source`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, supplement search, or recall tracking.

## Reproduction Assessment

Yes: a high-confidence source-level reproduction is to mock `memory.manager.search` so it returns a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`. Current main’s mapper still returns `corpus: "memory"` for that surfaced hit.

## Solution Assessment

Yes: deriving the surfaced corpus from the existing `result.source` is the narrowest maintainable fix because the backend already records provenance and the schema already accepts `corpus: "sessions"`. The broader index-time tagging idea in the KinthAI comment is useful context but not required for this specific contract mismatch.
