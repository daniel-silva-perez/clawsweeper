## Summary

Keep this open. Current main has useful pieces around manual backup, reset behavior, cron failure handling, and memory diagnostics, but the central upgrade-safety request remains unresolved: updates do not automatically snapshot user state, rollback is still code pinning rather than state restore, CLI session repair is not first-class, and legacy session overrides are intentionally preserved in a way that can still match the reported poisoned-session class.

## What This Changes

The issue requests safer upgrade handling for memory, workspace, session, auth/model override, cron, rollback, and diagnostics continuity after the 2026.4.10 update regression.

## Best Possible Solution

Add an opt-out pre-update backup using the existing backup planner, provide a supported restore/rollback path for state and workspace data, expose CLI session reset/delete/clear-overrides tooling, and extend doctor/status to flag stale session overrides plus missing root `MEMORY.md` when daily memory exists. Keep cron and model/auth migration validation as explicit post-update health checks rather than relying on manual JSON edits.

## Reproduction Assessment

Partly: the original two-host 2026.4.10 failure was not live-replayed, but the remaining source-level gaps are reproducible against current main by inspecting `openclaw update`, `openclaw sessions`, reset preservation, and doctor memory diagnostics. Those checks show the requested safety rails are still missing or partial.

## Solution Assessment

No: the current implementation is useful mitigation, not the best complete solution for this issue. The maintainable fix is to add targeted upgrade-safety and repair flows while preserving deliberate user model selections unless a diagnostic or explicit repair command clears them.
