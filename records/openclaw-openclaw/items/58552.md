## Summary

Keep open. Current main does not implement the central Apple Silicon utility/runtime work, and this PR is not merge-ready: inspection of the PR head shows blocking process-cleanup regressions, lowered exec buffer limits, compile/runtime errors, and a security-sensitive shell execution path.

## What This Changes

The PR adds Apple Silicon platform utilities, automatic process/shell/process-tree changes, hybrid runtime/tensor/thread-pool helpers, docs, and a verification script.

## Best Possible Solution

Narrow this into measured macOS ARM64 improvements that preserve existing process and shell contracts, keep Node's execFile defaults unless callers opt in, respect configured shells, retain detached-aware process-tree cleanup, and either remove or separately design the unintegrated hybrid/tensor/thread-pool work with focused tests and benchmark evidence.

## Reproduction Assessment

Yes for the blocking patch defects: static inspection of the PR head reproduces the lowered maxBuffer, non-async await syntax, missing thread-pool execution path, missing exports, and process-group logic. No high-confidence reproduction exists for the optimization claims because the PR does not include benchmark or live Apple Silicon proof.

## Solution Assessment

No. The current PR is broader than the stated optimization goal and changes core execution contracts without preserving current behavior; the safer path is a smaller benchmark-backed patch with Linux/Intel/macOS ARM64 regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Preserve runExec default maxBuffer capacity:** `src/process/exec.ts:115`
  - body: Object-options callers that omit `maxBuffer` now get `OPTIMAL_BUFFER_SIZE` instead of leaving the value undefined for Node's default. That drops the cap to 64 KiB or 128 KiB and can make existing timeout-only `runExec` calls fail with `ERR_CHILD_PROCESS_STDIO_MAXBUFFER`.
  - confidence: 0.97
- **[P1] Keep Unix process-group SIGKILL off Apple checks:** `src/process/kill-tree.ts:81-83`
  - body: On non-Apple-Silicon Unix hosts, `useProcessGroup` becomes false, so the grace-period force kill skips `process.kill(-pid, "SIGKILL")` and only targets the parent PID. Children can survive even though this helper is meant to terminate the tree.
  - confidence: 0.96
- **[P1] Clean up detached timeout process groups:** `src/process/exec.ts:252-253`
  - body: The Apple Silicon path sets `detached = true`, creating a new process group, but the timeout handler still calls `child.kill("SIGKILL")` on only the direct child. Grandchildren can survive timeout cleanup on the platform this PR is trying to improve.
  - confidence: 0.94
- **[P2] Make dynamic-import tests async:** `src/utils/platform.test.ts:54`
  - body: This test body and several later `it` callbacks use `await import()` inside non-async functions, which is a TypeScript syntax error. The new test file will not compile until those callbacks are marked `async` or rewritten.
  - confidence: 0.99
- **[P2] Replace CommonJS require in the ESM helper:** `src/utils/platform.ts:25`
  - body: This module is compiled as ESM in a `type: module`/NodeNext project, but it calls `require()` in function bodies. Those paths can throw `ReferenceError` at runtime and one of them also creates a platform-helper to shell-utils cycle.
  - confidence: 0.93
- **[P2] Respect configured SHELL before preferring zsh:** `src/agents/shell-utils.ts:61-66`
  - body: The new Apple Silicon branch returns zsh before the existing `process.env.SHELL` handling. Users who explicitly configured bash or another supported shell on macOS ARM64 would silently get zsh instead.
  - confidence: 0.92
- **[P1] Define thread-pool sizing helpers:** `src/utils/thread-pool.ts:25-29`
  - body: `getDefaultThreadPoolConfig()` calls `getPhysicalCpuCount()` and `getPerformanceCoreCount()`, but this file does not import or define either symbol. The new thread-pool module cannot typecheck as written.
  - confidence: 0.97
- **[P1] Preserve thread-pool task resolvers:** `src/utils/thread-pool.ts:181-184`
  - body: `run()` stores the caller's Promise handlers, then `submitTaskToWorker()` immediately overwrites the same active task entry with no-op callbacks. Completed tasks would not settle the Promise returned to callers.
  - confidence: 0.93
- **[P1] Add worker-side execute-task handling:** `src/utils/thread-pool.ts:175-179`
  - body: The main thread posts `execute-task` messages to workers, but the module never defines a worker-side `parentPort` message loop to run the task and emit completion or error messages. Dispatched work can remain pending forever.
  - confidence: 0.91
- **[P2] Stop calling missing thread-pool exports:** `src/utils/tensor-ops.ts:396-405`
  - body: The tensor helpers destructure `getOptimalParallelFactor` and `getOptimalBatchSize` from `./thread-pool.js`, but the new thread-pool module exports neither name. Calling these helpers will fail at runtime.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff changes command-execution paths and adds an environment-controlled shell command sink.

Concerns:

- **[medium] Avoid shelling out with raw SHELL:** `src/utils/platform.ts:166`
  - body: `getShellCommand()` interpolates `process.env.SHELL` into an `execSync` command string. Because `SHELL` is environment-controlled and may contain quotes or shell metacharacters, calling this helper could execute unintended shell syntax; use `fs.accessSync` or execFile-style arguments instead.
  - confidence: 0.88
