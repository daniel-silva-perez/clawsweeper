## Summary

Keep this issue open. Current main still uses an automatic daily session rollover as the default, still rotates to a new session ID on the next real message after the boundary, and does not provide a supported manual-only/off reset mode, pre-reset warning/cancel/postpone flow, or general restore path for daily reset archives. Later work fixed the opposite reset-not-firing regression, but it does not resolve the user-facing control and recovery concerns raised here.

## Best Possible Solution

Keep this open for maintainer/product handling. The likely fix belongs in the session reset contract and user surfaces: add a supported manual-only/off policy or a clearly discoverable equivalent, expose the active reset policy in setup/status/config UI, add a user-visible pre-rollover warning or confirmation path where channels can support it, and define whether daily reset archives can be restored or branched through CLI/Control UI.

## Risks / Open Questions

- Automatic resets are also used to bound context growth and provider cost, so a fix should make any manual-only mode explicit and visible rather than silently disabling safety behavior.
- A restore/rebind feature needs clear retention semantics because reset archives are currently subject to maintenance and disk-budget cleanup.
