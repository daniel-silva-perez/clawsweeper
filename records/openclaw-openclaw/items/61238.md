## Summary

Keep this open. Current main still defaults implicit session reset to a daily 4:00 AM rollover, still rotates on the next real message after the boundary, and still lacks a supported no-auto-reset mode, pre-reset warning/cancel/postpone flow, or general restore path for reset archives. Related open items cover parts of the problem, but not the full warning/control/recovery request end to end.

## What This Changes

The issue asks OpenClaw to prevent silent daily session rollover data loss by adding user warning/cancel/export controls, a discoverable opt-out/manual-only mode, and a way to restore prior reset archives.

## Best Possible Solution

Define a session reset contract that gives operators explicit control: support a real manual-only/no-auto-reset policy, surface the active implicit policy in setup/config/status, add channel-capable warning or confirmation behavior for automatic rollovers, and provide a documented restore or branch-from-archive path for reset transcripts.

## Reproduction Assessment

Yes. The current code and tests provide a high-confidence reproduction path: default config plus an existing session before the 4:00 AM boundary causes `initSessionState` to create a new session on the next real message after the boundary.

## Solution Assessment

No proposed patch is under review, and the current behavior is not the best solution for the reported UX. The maintainable fix should be a coordinated config contract and user-surface change, with archive restore handled through the gateway/session APIs rather than only docs.

## Risks / Open Questions

- Changing the default reset policy can affect context growth, provider cost, and safety bounds, so it needs an explicit product decision rather than a silent default flip.
- A true pre-reset warning/cancel flow is complicated by the lazy reset design: the rollover happens on the first real message after the time boundary, not from a proactive timer in every channel.
- Archive restore needs clear retention and conflict semantics because reset archives are subject to session maintenance cleanup.
