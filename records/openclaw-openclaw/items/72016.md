## Summary

Keep open. Current main still lacks a public doctor-check registration surface for plugins, and the requested repair/fix portion needs maintainer API and trust-policy decisions before it is safe to implement.

## What This Changes

Request a persistent plugin-style extension API for `openclaw doctor` so operators can add custom environment checks and optional fixes that survive npm upgrades.

## Best Possible Solution

Keep this as a plugin-SDK design item. Add an additive doctor contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, tests, docs, and a read-only check path first; add repair support only after the interactive and unattended fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives a high-confidence verification path: current main has no public `registerDoctorCheck` equivalent or plugin contribution registry for doctor health checks.

## Solution Assessment

Unclear until maintainer API and trust-policy direction lands. A narrow additive SDK seam for read-only doctor checks looks maintainable, but plugin-owned repair authority should not be added without an explicit consent and unattended-fix policy.

## Risks / Open Questions

- Plugin-owned doctor fixes would run near local config and host repair paths, so source attribution, prompting, logging, and unattended `--repair`/`--yes` policy need explicit design.
- Loading plugin checks from doctor could affect the existing cold-path performance and plugin loading boundary; the current docs note that non-interactive doctor skips eager plugin loading.
- The motivating multi-gateway overlap check may require host-wide config discovery, which needs a clear sandbox/privacy boundary before being bundled or exposed to third-party plugins.
