## Summary

Keep open. Current main still lacks the requested public doctor-check extension surface: doctor health contributions are core-only, the public plugin API has no doctor-check registration method, and the existing plugin doctor contract only covers compatibility/config repair. The issue is recent and source-anchored by the discussion, but the repair side is security-sensitive enough to need maintainer API and trust-policy direction first.

## What This Changes

Request a persistent plugin/extensibility API for `openclaw doctor` so operators can add custom environment checks and optional repair workflows that survive npm upgrades.

## Best Possible Solution

Keep this as a core/plugin-SDK design item. Add an additive doctor-check contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, tests, and docs; start with read-only checks, then add repair support only after interactive and unattended fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. The verification path is high-confidence source inspection: current main has no public way for a plugin or persistent local extension to register arbitrary doctor health checks.

## Solution Assessment

Unclear until maintainer direction lands. A narrow plugin-SDK doctor contribution seam appears maintainable, but repair authority, CLI controls, SDK location, and host-wide config access should be designed before a fix PR starts.
