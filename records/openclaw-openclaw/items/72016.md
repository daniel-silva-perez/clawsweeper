## Summary

Keep open. Current main still exposes a static core-only doctor health contribution list plus narrow plugin compatibility migrations, with no public plugin SDK registration API or CLI selection/listing for custom doctor checks. The discussion is recent, concrete, and source-anchored, so this remains a real core/plugin-SDK design gap.

## What This Changes

Request upgrade-safe extensibility for OpenClaw doctor checks and optional fixes, motivated by custom multi-gateway configuration-drift diagnostics.

## Best Possible Solution

Keep this as a core/plugin-SDK design item. Add a narrow, additive doctor-check extension contract with plugin source attribution, deterministic ordering, duplicate-id behavior, tests, and docs; start with read-only checks, then add repair support only after explicit prompting and unattended-mode semantics are settled.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is source inspection: the public plugin API and doctor command do not expose a way to register arbitrary doctor health checks.

## Solution Assessment

Unclear until maintainer direction lands. A narrow plugin-SDK doctor contribution seam is the right product direction, but fix authority, SDK location, CLI controls, and host-wide config access need an explicit design call before implementation.
