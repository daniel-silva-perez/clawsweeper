## Summary

Current main still lacks an upgrade-safe public extension point for arbitrary `openclaw doctor` checks or plugin-owned doctor fixes. The issue is recent, actionable, and the discussion correctly identifies a missing core/plugin-SDK contract rather than something already implemented, stale, duplicate, or ClawHub-only.

## Best Possible Solution

Keep this open as a core/plugin-SDK design item. Add a narrow, documented doctor-check extension contract with plugin source attribution, deterministic ordering, duplicate-id handling, focused tests, and docs. Start with read-only plugin checks, then layer fix support through the existing doctor repair prompting model. Use the multi-gateway/channel-overlap scenario as a proving case once maintainers decide whether that check belongs in core or as an external plugin after the API exists.

## Risks / Open Questions

- A plugin-owned doctor fix API would run near local config and state repair paths, so the design needs explicit source attribution, duplicate-id behavior, deterministic ordering, prompting, and unattended repair semantics.
- The motivating multi-gateway/channel-overlap check may require host-wide profile/config discovery, which is broader than current per-plugin compatibility repair contracts and should be designed deliberately.
