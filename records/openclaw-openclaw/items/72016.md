## Summary

Keep open. Current main still has a core-only doctor health contribution list, a narrow plugin doctor compatibility contract, and no public plugin-SDK API for registering custom doctor checks or check-specific fixes. The issue is recent, source-anchored by the discussion, and needs maintainer design/security direction before implementation.

## What This Changes

Request a persistent plugin/extensibility API for `openclaw doctor` so operators can add custom environment health checks and optional repair workflows that survive npm upgrades.

## Best Possible Solution

Keep this as a core/plugin-SDK design item. Add an additive doctor-check extension contract with plugin attribution, deterministic ordering, duplicate-id handling, tests, and docs; start with read-only checks, then add repair support only after interactive and unattended fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. The verification path is high-confidence source inspection: current main has no public way for a plugin or persistent local extension to register arbitrary doctor health checks.

## Solution Assessment

Unclear until maintainer direction lands. A narrow plugin-SDK doctor contribution seam is the maintainable direction, but repair authority, CLI controls, SDK location, and host-wide config access should be designed before a fix PR starts.
