## Summary

Keep open. Current main still lacks the requested public doctor-check extension surface: the doctor health contribution model and run loop are core-only, the plugin API has no doctor-check registration method, and the existing doctor/plugin contracts cover compatibility and config repair rather than arbitrary persistent health checks. The remaining work is a public SDK/trust-policy design item, not a stale cleanup candidate.

## What This Changes

Request a persistent doctor extensibility API so plugins or local extensions can add custom health checks and optional repairs that survive npm upgrades.

## Best Possible Solution

Keep this as a plugin-SDK design item. Add an additive doctor contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, focused tests, docs, and a read-only first step; add repair support only after interactive and unattended fix policy is explicit.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/API request. Source inspection gives a high-confidence verification path: current main has no public registerDoctorCheck-style API or equivalent plugin contribution registry for doctor health checks.

## Solution Assessment

Unclear until maintainer API and trust-policy direction lands. A narrow additive plugin-SDK doctor contribution seam appears maintainable, but plugin-owned repair authority and unattended fix behavior need explicit design before implementation.
