## Summary

Keep open. Current main still has the voice-call SecretRef mismatch the PR targets, and the functional patch shape is aligned with the existing SecretInput lifecycle, but the submitted diff adds a new bundled SecretInput manifest path without syncing the generated SecretRef credential matrix and supported-surface docs, which leaves a concrete CI-blocking docs/metadata regression.

## What This Changes

The PR widens voice-call Twilio auth-token config to SecretInput, resolves it strictly before Twilio provider construction, adds realtime provider API-key SecretInput metadata/docs, and extends focused config/runtime/collector tests.

## Best Possible Solution

Land this PR or an equivalent focused patch after updating the generated SecretRef credential matrix and supported-surface docs, while keeping provider construction strict so unresolved SecretRefs never reach Twilio or realtime provider initialization.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: a structured SecretRef at `plugins.entries.voice-call.config.twilio.authToken` is parsed by the voice-call config schema before runtime secret materialization, and that schema currently accepts only strings.

No, not as submitted. The functional approach is the narrow maintainable fix, but the manifest expansion must also update `docs/reference/secretref-user-supplied-credentials-matrix.json` and `docs/reference/secretref-credential-surface.md` before merge.
