## Summary

Keep open. Current main still lacks a custom-provider output modality/capability contract, and configured provider model rows are still surfaced as selectable/catalog models and resolved into the LLM streaming path. The input-modality side has improved since the original report, and user-selected fallback behavior is now stricter in docs/code, but the central media-output diagnostic/support gap remains.

## What This Changes

The issue requests that Poe image/video-output models configured through `models.providers` either work through media-generation routing or be explicitly rejected with clear unsupported-output diagnostics instead of malformed requests or fallback behavior.

## Best Possible Solution

Add an explicit text-output boundary for session/chat models: custom provider rows should either declare/detect supported output modalities and reject unsupported image/video-output selections with a clear diagnostic, or route true media generation through `image_generate` / `video_generate` provider/plugin surfaces with docs covering supported input/output combinations.

## Reproduction Assessment

Yes for the OpenClaw-side mismatch: current source accepts configured provider model rows, lacks output metadata, copies them into the model catalog, and resolves them as LLM stream models. The live Poe backend request was not rerun in this read-only review.

## Solution Assessment

No current-main solution exists yet. The narrowest maintainable direction is explicit unsupported-media-output handling and docs first; full Poe image/video generation should be separate provider/plugin work on the existing media-generation surfaces.
