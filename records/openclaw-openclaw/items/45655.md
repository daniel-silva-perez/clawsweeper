## Summary

Keep open. Current main still accepts custom provider model rows without any output modality contract and still normalizes those rows into the LLM streaming path, while image and video generation are separate tool/provider surfaces; the Poe media-output confusion remains actionable.

## What This Changes

The issue asks OpenClaw to either support Poe image/video-output models configured under custom model providers or reject them clearly instead of letting them fail or fall back at runtime.

## Best Possible Solution

Keep this issue open for an explicit text-output session boundary: document `models.providers` as chat/LLM-output configuration, add a supported way to declare or reject non-text output models with a clear diagnostic, and route real Poe image/video generation through the image/video generation provider APIs rather than the LLM stream adapter.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the OpenClaw-side mismatch: a custom provider model row is accepted without output metadata and is normalized into the inline LLM streaming path. The exact Poe backend response was not live-reproduced here.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution exists yet. The best short-term fix is explicit unsupported media-output handling plus docs; full Poe media generation support should be a separate provider/plugin implementation on the existing media-generation surfaces.
