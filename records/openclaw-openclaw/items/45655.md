## Summary

Keep this issue open. Current main has separate image/video generation provider APIs and docs, and it now accepts broader input modality labels, but custom `models.providers` entries still have no output modality/capability field and still route through the LLM stream path, so the reported Poe media-output confusion remains actionable.

## What This Changes

The issue requests that Poe image/video-output models configured through `models.providers` either work through the media-generation runtime or be explicitly rejected with a clear unsupported-output diagnostic instead of malformed chat requests or fallback.

## Best Possible Solution

Keep the issue open for an explicit text-output session boundary. The maintainable end state is to document `models.providers` as chat/LLM-output config, add a supported way to declare or reject non-text output models with a clear diagnostic, and route any real Poe image/video generation through image/video generation provider APIs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the OpenClaw-side mismatch: a custom `models.providers.poe` model row is accepted without output metadata and is normalized into the LLM stream path; the exact Poe HTTP response was not live-reproduced here.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution exists yet. The best short-term fix is explicit unsupported media-output handling and docs; full Poe media support should be a separate provider/plugin implementation on the existing media-generation surfaces.
