## Summary

Keep open. Current main has partially addressed the input-modality half of the report, but the core gap remains: custom `models.providers` rows still have no output modality/capability contract, are surfaced in the model catalog as chat models, and are resolved into the PI LLM streaming path rather than the media-generation provider registries. The issue discussion is still aligned with current code, and no open PR referencing this issue was found.

## What This Changes

The issue asks OpenClaw to reject or correctly route Poe image/video-output models that can be configured as custom providers but fail when used through the chat/session runtime.

## Best Possible Solution

Add an explicit text-output session boundary for custom provider models: support a declared or detected non-text output capability, reject unsupported image/video-output chat selections with a clear diagnostic, and route real Poe media generation through `image_generate` / `video_generate` provider or plugin surfaces with docs covering supported input/output combinations.

## Reproduction Assessment

Yes for the OpenClaw-side mismatch: current source accepts configured provider model rows, lacks output modality metadata, copies those rows into the model catalog, and resolves them as LLM stream models. The live Poe-specific 400/fallback behavior was not rerun in this review.

## Solution Assessment

No current-main solution exists yet. The safest short-term direction is explicit unsupported-media-output handling and docs; full Poe media generation should be separate provider/plugin work on the existing media-generation surfaces.
