## Summary

Keep open. Current main still has the reported product/API gap: custom `models.providers` entries can describe input modalities but cannot declare output modality, configured provider models are normalized into the LLM streaming path, and image/video generation is handled through separate `image_generate` / `video_generate` provider contracts rather than Poe-style chat model refs.

## Best Possible Solution

Keep this open for an explicit product/API decision. The short-term fix should document `models.providers` as a text-output LLM/chat model surface, add a supported way to declare or reject non-text output models, and emit a clear unsupported media-output diagnostic when such a model is selected. Full Poe image/video support should go through the existing image/video generation provider APIs, not the generic chat stream adapter.
