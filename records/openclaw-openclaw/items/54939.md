## Summary

Keep open. Current main still does not implement Exa `webSearch.baseUrl`: the manifest schema only permits `apiKey`, the runtime always posts to the fixed Exa endpoint, and the user docs still document only `apiKey`. The PR is useful and its later commits address the endpoint-resolution and cache-key feedback from review, but the branch is now stale against current main's Exa runtime split and generated-artifact policy, so it should be rebased or replaced rather than closed or merged as-is.

## What This Changes

This PR adds Exa `webSearch.baseUrl` schema/UI metadata, resolves Exa search requests through a configurable endpoint with cache-key isolation, and adds endpoint-resolution tests plus generated config metadata updates.

## Best Possible Solution

Port the feature onto current main's Exa plugin architecture: add `webSearch.baseUrl` to the manifest, resolve and cache by the effective endpoint in `exa-web-search-provider.runtime.ts`, keep the lazy wrapper/shared contract intact, update Exa docs and generated config hash artifacts, add a changelog credit, then land so #54928 can close through the merged implementation.
