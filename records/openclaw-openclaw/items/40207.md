## Summary

Keep open. Current main still derives macOS LaunchAgent stdout/stderr paths from the OpenClaw state directory and rewrites those paths during install and restart fallback, so the symlinked or external-volume state-dir failure remains source-backed. The issue discussion has two independent confirmations plus a working workaround, and the latest release tag still contains the same relevant behavior.

## What This Changes

The issue reports that the macOS gateway LaunchAgent exits with `EX_CONFIG` when generated plist log paths point under a symlinked or external-volume `.openclaw` state directory.

## Best Possible Solution

Give macOS LaunchAgent `StandardOutPath` and `StandardErrorPath` a launchd-safe destination that does not depend on the OpenClaw state directory symlink, such as `$HOME/Library/Logs/openclaw`, or canonicalize those two plist targets before writing. Preserve state/config directory semantics and non-macOS restart-log behavior, and update status/diagnostics/docs to point at the same macOS supervisor log paths.
