## Summary

Keep open. Current main still builds macOS LaunchAgent stdout/stderr paths from the OpenClaw state directory and writes them into the plist during install and restart-regeneration, so a symlinked or external-volume state directory can still hit the reported launchd EX_CONFIG path. The issue has independent confirmations, no protected labels, and no open PR found that specifically fixes this focused log-path bug.

## What This Changes

The issue requests a launchd-safe default or canonicalized path for generated macOS LaunchAgent StandardOutPath/StandardErrorPath, plus clearer install-time failure reporting.

## Best Possible Solution

Update the macOS LaunchAgent generation path so StandardOutPath and StandardErrorPath use a launchd-safe local user log location, such as ~/Library/Logs/openclaw, or canonicalize only those LaunchAgent log paths before plist write. Cover fresh install and restart-regeneration, keep restart diagnostics coherent, update user-facing log hints/docs, and add targeted tests for symlinked state-dir behavior.

## Repair Work Prompt

Fix the macOS LaunchAgent log-path bug tracked by https://github.com/openclaw/openclaw/issues/40207. Current main writes StandardOutPath/StandardErrorPath from resolveGatewayLogPaths(), which places them under $OPENCLAW_STATE_DIR/logs and can fail launchd with EX_CONFIG when ~/.openclaw is symlinked or on an external volume. Keep the fix focused on CLI-managed macOS LaunchAgent stdout/stderr paths: prefer a launchd-safe user log directory such as ~/Library/Logs/openclaw, or canonicalize only the LaunchAgent stdout/stderr directory before plist generation. Cover both fresh install/stage and restart-regeneration paths, and update status/runtime hints and user docs so the displayed log paths match. Do not broadly change non-macOS service behavior, gateway file logging, restart-log semantics, launchd restart policy, or unrelated supervisor recovery. Add or update targeted tests for profile-aware paths and a symlinked/external-state-dir-style case.
