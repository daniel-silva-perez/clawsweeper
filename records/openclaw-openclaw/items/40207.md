## Summary

Keep open. Current main and latest release v2026.4.27 still derive macOS LaunchAgent stdout/stderr paths from the OpenClaw state directory, and the issue discussion has two independent macOS confirmations that moving those plist paths away from a symlinked home `.openclaw` path avoids launchd exit 78. Related #46153 covers a separate launchd hang and does not supersede this narrower log-path crash.

## What This Changes

The issue asks for generated macOS Gateway LaunchAgent stdout/stderr paths to avoid symlinked or external-volume home `.openclaw` log paths so launchd no longer exits with EX_CONFIG after install or plist regeneration.

## Best Possible Solution

Add a macOS launchd-specific supervisor stdout/stderr path that does not depend on a symlinked OpenClaw state directory, or canonicalize those plist paths before writing. Keep state/config storage and non-macOS restart-log behavior intact, then align status hints, diagnostics, tests, docs, and changelog with the chosen LaunchAgent log destination.

## Reproduction Assessment

Yes. The issue has a concrete macOS reproduction using a symlinked home `.openclaw`, `openclaw gateway install`, and launchd status/exit-code checks; current-main source inspection shows generated plist stdout/stderr still follow the state directory.

## Solution Assessment

Yes. A focused fix in LaunchAgent log-path generation and the user-facing status/docs surfaces is the narrowest maintainable path; the gateway runtime and general config path normalization do not need a broad rewrite for this report.
