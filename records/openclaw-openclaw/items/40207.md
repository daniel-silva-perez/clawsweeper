## Summary

Keep open. Current main still generates macOS LaunchAgent stdout/stderr paths from the OpenClaw state directory, and the issue has two independent confirmations that moving those plist log paths away from a symlinked/external `.openclaw` path avoids launchd exit 78. Related #46153 covers a separate launchd hang and does not supersede this log-path crash.

## What This Changes

The issue reports that generated macOS Gateway LaunchAgent plist stdout/stderr paths under the OpenClaw state directory can cause launchd `EX_CONFIG` when `~/.openclaw` is symlinked or on an external volume, and asks for a resilient log destination or canonicalization plus better diagnostics.

## Best Possible Solution

OpenClaw should write macOS LaunchAgent `StandardOutPath` and `StandardErrorPath` to launchd-safe paths that do not depend on a symlinked OpenClaw state directory, while preserving normal state/config storage and non-macOS restart-log behavior. Status hints, diagnostics, tests, docs, and the changelog should be updated to match the chosen LaunchAgent log destination.

## Reproduction Assessment

Yes. The issue gives a concrete macOS reproduction with a symlinked home `.openclaw`, `openclaw gateway install`, and launchd exit-code checks; source inspection shows current main still emits plist stdout/stderr paths through the state directory.

## Solution Assessment

Yes. A focused macOS LaunchAgent log-path fix is the narrowest maintainable path; broad gateway runtime or general config normalization changes are not needed for this report.
