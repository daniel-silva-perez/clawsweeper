## Summary

Keep open. Current main still derives the macOS LaunchAgent stdout/stderr paths from the OpenClaw state directory and writes those paths directly into the plist, so a symlinked or external-volume state directory can still produce the reported launchd `EX_CONFIG` failure. The issue has two independent confirmations, the earlier same-author report was only a refile, the related preflight PR is closed unmerged, and the broader launchd hang tracker does not supersede this focused log-path bug.

## What This Changes

The issue asks for macOS gateway LaunchAgent log paths to avoid symlinked or external-volume `~/.openclaw/logs` targets that cause launchd `EX_CONFIG` startup failures.

## Best Possible Solution

Fix the CLI-managed macOS LaunchAgent generation path so StandardOutPath and StandardErrorPath use a launchd-safe, non-symlinked user log location such as ~/Library/Logs/openclaw, or canonicalize only the LaunchAgent stdout/stderr directory before writing plists. Cover both fresh install and restart-regeneration paths, update status/runtime/doc hints, and consider an install-time health check that surfaces launchd EX_CONFIG failures clearly.
