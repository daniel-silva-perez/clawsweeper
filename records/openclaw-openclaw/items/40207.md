## Summary

Current main still writes the macOS LaunchAgent stdout/stderr paths under the OpenClaw state directory, so a symlinked or external-volume `~/.openclaw` can still become the `StandardOutPath` and `StandardErrorPath` target. The issue remains concrete and actionable.

## Best Possible Solution

Keep this issue open. Fix the CLI-managed macOS LaunchAgent path in `src/daemon` by writing `StandardOutPath` and `StandardErrorPath` to a launchd-safe, non-symlinked user log location such as `~/Library/Logs/openclaw`, or by canonicalizing the generated log directory before writing the plist. Add regression coverage for symlinked state directories and plist regeneration, update runtime/status/docs hints, and consider an install-time health check that surfaces launchd EX_CONFIG failures clearly.

## Risks / Open Questions

- No live macOS launchd reproduction was run from this Linux checkout; the keep-open decision rests on the concrete issue confirmations plus current source/tests/docs still matching the reported failing path.
- Changing `resolveGatewayLogPaths` globally would also affect restart logs, diagnostics, runtime hints, and non-macOS service behavior; a LaunchAgent-specific stdout/stderr resolver may be the safer implementation boundary.
- Docs currently disagree in places about macOS gateway log locations, so the fix should update the user-facing docs alongside code and tests.
