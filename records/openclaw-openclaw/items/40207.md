## Summary

Keep open. Current main still generates and rewrites the macOS gateway LaunchAgent stdout/stderr paths from the OpenClaw state directory, so a symlinked or external-volume state directory can still feed launchd the same problematic log targets described in the report. The issue also has two independent confirmations and a working log-path workaround, and I found no open PR specifically referencing this issue.

## What This Changes

The issue reports that the macOS gateway LaunchAgent fails with EX_CONFIG when generated StandardOutPath and StandardErrorPath point under a symlinked or external-volume ~/.openclaw state directory.

## Best Possible Solution

Fix the CLI-managed macOS LaunchAgent generation path so StandardOutPath and StandardErrorPath use a launchd-safe, non-symlinked user log location such as ~/Library/Logs/openclaw, or canonicalize only the LaunchAgent stdout/stderr directory before writing plists. Cover both fresh install and restart-regeneration paths, update status/runtime/doc hints, and consider an install-time health check that surfaces launchd EX_CONFIG failures clearly.
