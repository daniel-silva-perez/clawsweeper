## Summary

Keep open. Current main and the latest release still route macOS LaunchAgent stdout/stderr paths through the OpenClaw state directory, while the issue discussion has two independent macOS confirmations that moving those plist paths off a symlinked `.openclaw` path avoids launchd exit 78.

## What This Changes

The issue requests fixing macOS LaunchAgent installation so generated `StandardOutPath` and `StandardErrorPath` do not make launchd fail when `~/.openclaw` is symlinked to an external or non-standard path.

## Best Possible Solution

Add a macOS launchd-specific supervisor log path that does not depend on the OpenClaw state directory symlink, or canonicalize the plist stdout/stderr targets before writing. Keep state/config and non-macOS restart-log behavior intact, then align status hints, tests, docs, and changelog with the chosen LaunchAgent log destination.

## Reproduction Assessment

Yes. The issue has a concrete macOS repro using a symlinked `~/.openclaw`, `openclaw gateway install`, and launchd status/exit-code checks; current-main source inspection shows the generated plist paths still follow that symlinked state directory.

## Solution Assessment

Yes. The narrow maintainable fix is in LaunchAgent log-path generation and the status/docs surfaces that report those paths, not in the gateway runtime or general config path handling.

## Repair Work Prompt

Fix the macOS LaunchAgent `StandardOutPath`/`StandardErrorPath` failure tracked by https://github.com/openclaw/openclaw/issues/40207. Current main routes these plist paths through `resolveGatewayLogPaths`, which points at `$OPENCLAW_STATE_DIR/logs` or default `~/.openclaw/logs`; when `.openclaw` is symlinked to an external volume, reporters see launchd exit 78 before the gateway starts. Implement a narrow macOS launchd fix: prefer a supervisor log path independent of the OpenClaw state-dir symlink, such as `$HOME/Library/Logs/openclaw`, or canonicalize the plist stdout/stderr targets before writing. Preserve state/config directory semantics, profile-aware naming, `OPENCLAW_LOG_PREFIX` behavior where appropriate, and non-macOS restart-log behavior. Update status/runtime hints, tests, docs, and changelog with reporter credit; do not attempt to solve the separate launchd module-evaluation hang from #46153 in this PR. Validate with targeted formatter/tests and `pnpm check:changed` in Testbox; when macOS is available, smoke `openclaw gateway install --force` with `~/.openclaw` symlinked and verify launchd no longer reports `EX_CONFIG` and `openclaw gateway status` is healthy.
