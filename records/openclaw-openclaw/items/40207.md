## Summary

Keep open. Current `main` and the latest release still generate macOS LaunchAgent stdout/stderr paths under the OpenClaw state directory, so a symlinked or external-volume state dir can still hit the reported launchd `EX_CONFIG` failure. The discussion has independent confirmations and a clear workaround, and the provided related context does not show a merged or open fix that supersedes this focused bug.

## What This Changes

The issue requests that macOS LaunchAgent stdout/stderr log paths stop pointing through a symlinked or external-volume `~/.openclaw/logs` state directory so launchd can start the gateway reliably.

## Best Possible Solution

Move only the macOS LaunchAgent `StandardOutPath` and `StandardErrorPath` defaults to a launchd-safe user log location such as `~/Library/Logs/openclaw`, or canonicalize those LaunchAgent log targets before plist write. Keep gateway state/config semantics unchanged, cover fresh install and restart-regeneration, preserve useful status diagnostics, and update docs/tests/changelog so users can find the new log path.
