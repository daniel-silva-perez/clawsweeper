## Summary

Keep open. Current main has per-account channel start/stop hooks, task abort tracking, synchronous runtime-context leases, and some SDK lifecycle helpers, but it still does not expose the proposed centralized PluginResourceManager-style async resource ownership API for channel plugins. The motivating Feishu HTTP server cleanup pattern is still visible, and related #48183, #48588, and #49804 are still open rather than merged/shipped superseders.

## What This Changes

Request to add a centralized PluginResourceManager-style account-scoped async lifecycle and cleanup contract for channel plugins, with gateway-owned tracking before restart/shutdown.

## Best Possible Solution

Keep this open for maintainer design review. The best path is to decide whether the public plugin SDK should grow an account-scoped async resource registration/lifecycle contract, then migrate one or two high-risk channel resources such as Feishu HTTP servers as contract-test examples. Handle the immediate Feishu bug through #48183/#48588 and coordinate the broader lifecycle design with #49804 rather than closing this as already implemented.

## Risks / Open Questions

- A centralized resource lifecycle manager would change public plugin SDK expectations for bundled and third-party channel plugins, so it needs design/versioning review before implementation.
- The immediate Feishu server.close race remains a narrower bug and should stay coordinated with #48183/#48588 rather than being blocked on the larger lifecycle API.
- Existing helper surfaces like channel-lifecycle and channel-runtime-context could be part of the answer, but they do not currently enforce consistent async cleanup across all channel plugins.
