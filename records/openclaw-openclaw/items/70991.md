## Summary

Keep this issue open. Current main still has the reported recovery gap: Control UI Stop visibility is tied to browser-local run tracking that is cleared on reconnect, the hello snapshot does not expose active abortable work, and gateway keepalive is still only an application-level tick unless one of the related open ping PRs lands.

## What This Changes

The issue reports that long Control UI tasks drop the gateway connection, then lose the Stop control and fail to stop reliably via `/stop` after reload or reconnect.

## Best Possible Solution

Add an additive Gateway recovery contract for authenticated Control UI clients to discover whether the current session has abortable work after reconnect/reload, keep a session-scoped Stop affordance available when the exact run id is unknown, and consolidate WebSocket protocol ping ownership through whichever of #56668 or #74783 lands as canonical.

## Reproduction Assessment

Yes for the Stop/reconnect loss: current source shows Stop depends on `chatRunId`, reconnect clears that value, and the hello snapshot has no active-run recovery field. The live Windows/Kimi intermediary drop itself was not reproduced in this read-only pass.

## Solution Assessment

Yes, a focused UI/Gateway recovery fix is the best path, with ping ownership coordinated against #56668 and #74783. Connected session-scoped `/stop` is a useful partial foundation but does not make the Stop control or disconnected recovery reliable after reload/reconnect.
