## Summary

Keep open. Current main only partially addresses the report: session-scoped `chat.abort` and `/stop` support exist, but the Control UI still hides Stop after local run state is lost, and the Gateway still lacks a landed protocol-level ping/connection-health implementation for the long-task disconnect side.

## What This Changes

The issue reports that long OpenClaw Control tasks can drop the Gateway connection, hide the Stop control after reload, and leave `/stop` unable to interrupt the still-running model.

## Best Possible Solution

Keep this as the canonical user-facing regression until one Gateway WebSocket keepalive/connection-health implementation lands and the Control UI derives abort availability from server/session active-run state after reconnect or reload, using session-scoped `chat.abort` when the local run id is unavailable.

## Reproduction Assessment

Unclear for the full connection-drop bug: the issue has plausible long-task steps but no logs or live Windows/Kimi proof. Yes for the Stop-button disappearance path: source inspection shows reconnect clears `chatRunId`, rendering only shows Stop when that local value exists, and a test covers the no-run disconnected abort no-op.

## Solution Assessment

No, current main is not the full solution. The backend session-scoped abort path is useful, but the UI still does not surface a session-level Stop control after local run-state loss, and the connection stability fix should come from one canonical Gateway ping/health implementation.
