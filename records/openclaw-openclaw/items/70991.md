## Summary

Keep open. Current main still has only a partial abort fix: the backend can abort active runs by session, but the Control UI still hides Stop after reconnect/reload when its local run id is cleared, and the gateway connection-drop side is still represented by open keepalive/telemetry PRs rather than a shipped main implementation.

## What This Changes

The issue reports Control UI gateway disconnects during long-running tasks, loss of the Stop button after reload, and ineffective `/stop` fallback on OpenClaw 2026.4.22 for Windows 11.

## Best Possible Solution

Keep this as the canonical user-facing regression until maintainers land one consolidated Gateway WebSocket keepalive/connection-health path and update Control UI chat controls to derive abort availability from active session state after reconnect/reload, using session-scoped `chat.abort` when the client run id is unavailable.

## Reproduction Assessment

Unclear for the full bug. The report has plausible long-task steps but no logs; static source inspection gives a high-confidence reproduction for the Stop-button disappearance because reconnect clears `chatRunId` and rendering only shows Stop when that local value is present.

## Solution Assessment

No, current main is not the full solution. Backend session-scoped abort and `/stop` support are useful, but the UI still does not surface a session-level Stop control after local run-state loss, and the connection stability fix is still split across open PRs.
