## Summary

Keep open. Current main still only partially addresses the report: backend session-scoped abort and `/stop` support exist, but the Control UI still hides Stop when the local `chatRunId` is lost after reconnect/reload, and the Gateway connection-drop side is still represented by open keepalive/connection-health PRs rather than a landed main implementation.

## What This Changes

The issue requests stable Gateway-to-Control UI connectivity during long tasks and a reliable Stop or `/stop` abort path after reconnect or reload.

## Best Possible Solution

Keep this as the canonical user-facing regression until one Gateway WebSocket keepalive/connection-health implementation lands and the Control UI derives abort availability from server/session active-run state after reconnect, using session-scoped `chat.abort` when the client run id is unavailable.

## Reproduction Assessment

Unclear for the full connection-drop bug because the report has plausible long-task steps but no logs or live Windows/Kimi proof. Yes for the Stop-button disappearance: source inspection shows reconnect clears `chatRunId` and rendering only shows Stop when that local value exists.

## Solution Assessment

No, current main is not the full solution. Backend session-scoped abort is useful, but the UI still does not expose a session-level Stop path after local run-state loss, and the connection stability fix needs one canonical Gateway ping/health implementation.
