## Summary

Keep this issue open. Current main still has the reported Control UI recovery gap: the Stop affordance is tied to browser-local `chatRunId`, reconnect clears that run state, the hello snapshot does not expose active abortable work, and the Gateway still appears to rely on application-level ticks rather than WebSocket protocol ping/pong. Backend session-scoped abort and connected `/stop` are useful partial fixes, but they do not make Stop or `/stop` reliable after reload/reconnect.

## What This Changes

The issue asks for long-running Control UI/Gateway connections to remain stable and for Stop or `/stop` to stay usable after reload/reconnect while a task is still running.

## Best Possible Solution

Add an additive Gateway recovery contract that exposes abortable active work for the current session after reconnect/reload, keep a session-level Stop affordance available when the run id is unknown, use session-scoped abort in that path, add WebSocket protocol keepalive, and cover the reconnect/reload behavior with focused UI and Gateway tests.

## Reproduction Assessment

Yes for the Stop/reconnect loss: current source shows `canAbort` depends on `chatRunId`, reconnect clears that value, and disconnected abort without a run id is a no-op. The live Windows/Kimi network drop itself was not reproduced in this read-only pass.

## Solution Assessment

Yes, a focused UI/Gateway recovery PR is the right path. The existing backend session abort is a partial foundation, but the best fix must also restore active abortability after reload/reconnect and add WS keepalive rather than only changing the button or only landing #56668.
