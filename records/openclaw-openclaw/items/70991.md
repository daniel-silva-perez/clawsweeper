## Summary

Keep this issue open. Current main has useful backend abort support, including session-scoped `chat.abort` and `/stop` documentation, but the reported user-visible failure is still plausible: the Control UI still hides Stop when the local `chatRunId` is lost after reconnect or reload, and the connection-health side overlaps active open keepalive/telemetry PRs rather than a shipped fix.

## What This Changes

The issue reports long-running Control UI sessions losing Gateway connection, hiding the Stop button after reload/reconnect, and failing to stop the active model run via `/stop`.

## Best Possible Solution

Keep the issue as the canonical user-facing regression until maintainers either land one consolidated Gateway connection-health path and add a Control UI session-level abort affordance after reconnect/reload, or split those into linked narrower follow-ups with clear ownership.
