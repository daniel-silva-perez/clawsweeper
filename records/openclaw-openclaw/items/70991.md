## Summary

Keep this issue open. Current main still has the Control UI reconnect-stop gap: reconnect handling clears the active chat run id, while Stop visibility is still derived from that id, so a reload/reconnect can hide Stop even if the gateway still has an abortable session run. The connection-drop half is also not fully settled because the related WebSocket ping/health PRs are still open and unmerged.

## What This Changes

The issue reports that long-running Control UI sessions on Windows lose the Gateway connection, then reload without a Stop button and fail to interrupt the still-running task with `/stop`.

## Best Possible Solution

Keep this as the canonical issue. The desired end state is one gateway-owned connection liveness implementation from #56668 or #74783, plus Control UI behavior that derives abortability from active session state after reconnect/reload, keeps a session-scoped Stop control available, and shows clear feedback when `/stop` or `chat.abort` finds or does not find an active run.

## Reproduction Assessment

Yes for the Control UI half: with an active run, a reconnect hello clears `chatRunId`, and rendering then hides Stop because `canAbort` is based only on that id. The exact reported Windows disconnect frequency still needs logs or a live long-task run to reproduce the transport drop itself.

## Solution Assessment

Yes for keeping this open. Current main has partial session-scoped abort support, but the UI does not reliably expose it after reconnect/reload; the safest fix is a narrow Control UI abort-state recovery/fallback while separately coordinating the ping ownership work already represented by #56668 and #74783.
