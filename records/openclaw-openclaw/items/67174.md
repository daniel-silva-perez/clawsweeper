## Summary

Keep open. Current main still lacks the split-tenant Microsoft Teams Graph configuration and token-provider path, so the PR is not obsolete; however, the latest diff still has concrete merge blockers around secret handling and changelog attribution.

## What This Changes

The PR adds `channels.msteams.graphTenantId` and `MSTEAMS_GRAPH_TENANT_ID`, wires Microsoft Teams Graph token acquisition through that tenant when configured, updates docs/plugin metadata/config schema, and adjusts thread Graph error handling tests.

## Best Possible Solution

Land a narrow Teams plugin change that keeps Bot Framework reply tokens on the bot tenant, uses `graphTenantId` only for Graph app-only token acquisition, avoids retaining raw client secrets in cache keys, and keeps config schema, docs, plugin metadata, tests, and changelog attribution aligned.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run in this read-only review; the static path is clear, though: a split-tenant Teams deployment on current main still mints Graph tokens through `tenantId`, while the PR author reports manual cross-tenant verification.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer review: the split-provider direction is narrow and consistent with Teams ownership, but the raw-secret cache key and changelog attribution should be fixed before this is merge-ready.
