## Summary

Keep this PR open. Current main partially addresses the inbound conversation-normalization side through channel-owned resolvers, but it still does not implement the PR's central contract: typed agent hook contexts and embedded/native call sites still omit conversationId and accountId. The PR also is not merge-ready as-is because its added after_agent_complete hook is absent from current main's hook catalog/runner and has unresolved streaming suppression semantics raised in review.

## Best Possible Solution

Refresh this PR against current main. Keep the channel-owned inbound resolver approach, add conversationId and accountId to src/plugins/hook-types.ts and src/agents/harness/hook-context.ts, wire params.messageTo and params.agentAccountId through embedded and native hook contexts, and add focused regression tests. Drop the after_agent_complete portion or redesign it explicitly around current before_agent_finalize, reply_dispatch, and block-streaming semantics with maintainer approval.

## Risks / Open Questions

- Merging the PR as-is risks compile/runtime failure because the proposed after_agent_complete runner method is not implemented in createHookRunner according to the existing review discussion.
- The proposed suppress/reinject semantics are unsafe with block streaming unless redesigned around the current block reply pipeline and final dispatch hooks.
- Closing this PR would discard focused contributor work for a still-current plugin hook identity gap, even though the linked issue is closed.
- Regression coverage is still needed for conversationId/accountId propagation across before_model_resolve, before_prompt_build, llm_input, llm_output, before_agent_finalize, and agent_end on embedded and native harness paths.
- Security/supply-chain pass did not show dependency, lockfile, CI, workflow, or package publishing changes in the provided diff, but the new hook would broaden plugin control over final-response delivery and should get maintainer design review.
