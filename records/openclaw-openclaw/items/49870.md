## Summary

Keep open. Current main has partially addressed the inbound conversation-normalization direction, but it still does not implement this PR's central agent-hook contract: `PluginHookAgentContext` and the embedded/native harness hook contexts still lack `conversationId` and `accountId`, and the hook call sites still omit those values. The PR itself also has unresolved review blockers around its added `after_agent_complete` design, so the right outcome is a current-main refresh rather than cleanup closure.

## Best Possible Solution

Keep this PR open and ask for a refresh against current main: preserve the channel-owned inbound resolver approach already present, add `conversationId` and `accountId` to `src/plugins/hook-types.ts` and `src/agents/harness/hook-context.ts`, wire `params.messageTo` and `params.agentAccountId` through the embedded runner hook contexts for before-model, prompt, LLM input/output, finalization, and agent-end phases, and add focused regression tests for those contexts. The PR should either drop `after_agent_complete` in favor of current `before_agent_finalize` and `reply_dispatch` surfaces or redesign it explicitly with maintainer approval and streaming-safe semantics.
