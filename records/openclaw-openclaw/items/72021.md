## Summary

Keep open. Current main and the latest release still compare `minRecallCount` against aggregate short-term signal count, and the issue is already paired with open PR #74114 using closing semantics. The terminal next step is maintainer review of that PR, not closing this issue first or spawning a duplicate repair lane.

## What This Changes

The issue asks memory-core short-term promotion to stop treating daily/session/grounded signal counts as real recall counts for the `minRecallCount` gate so real user recalls are not hidden by inflated aggregate signals.

## Best Possible Solution

Codify the promotion threshold contract in memory-core: decide whether `minRecallCount` means true user recalls only, aggregate evidence, or recall-plus-grounded evidence, then land one policy-compatible PR with updated tests and any needed docs before closing this issue.

## Reproduction Assessment

Yes for the mixed-signal gate: read-only source and test inspection shows a daily-only candidate with `recallCount: 0` and `dailyCount: 3` can clear default promotion gates. The exact reporter score calibration needs their private recall-store fixture.

## Solution Assessment

Unclear. A recall-only gate is the narrowest proposed fix for false daily-only candidates, but it must be reconciled with the currently tested daily-only and grounded promotion behavior before it is clearly the best maintainable solution.
