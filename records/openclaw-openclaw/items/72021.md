## Summary

Current main still compares `minRecallCount` against aggregate short-term signal count, and the latest release contains the same behavior. Because PR #74114 is open, unmerged, and explicitly closes this issue, the issue should stay open until maintainers land or reject that implementation path.

## What This Changes

The issue asks for short-term memory promotion to distinguish real user recall counts from daily/session-derived signals when applying recall gates and scoring promotion candidates.

## Best Possible Solution

Codify the promotion threshold contract: decide whether `minRecallCount` means true user recalls only, aggregate short-term evidence, or recall-plus-grounded evidence, then land one policy-compatible PR with matching tests and memory docs before closing this issue.

## Reproduction Assessment

Yes for the mixed-signal gate: current source and tests show `recallCount: 0` plus `dailyCount: 3` can clear default promotion gates. The exact reporter score values need their private recall-store fixture to reproduce numerically.

## Solution Assessment

Unclear. A recall-only gate addresses the core report, but the final fix must also reconcile existing daily/session and grounded promotion behavior rather than silently invalidating tested paths.
