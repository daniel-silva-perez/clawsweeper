## Summary

Keep this PR open. Current main still stores trusted safe-bin directories with exact path casing and compares them with exact-cased resolved executable directories, while doctor lowercases configured safeBins before resolving them. The reported false warning remains plausible on case-insensitive filesystems, and the PR is a focused implementation candidate, but its latest global temp-dir case-sensitivity probe still needs trust-boundary review.

## Best Possible Solution

Keep this PR open for maintainer review or a small revision. The best fix is a shared, testable safe-bin path comparison helper that compares trusted dirs and resolved executable dirs consistently on genuinely case-insensitive paths without collapsing distinct paths on case-sensitive or mixed-sensitivity volumes, plus deterministic regression coverage for doctor/runtime mixed-case matching and a trust-boundary test for case-sensitive siblings.

## Risks / Open Questions

- Current main still lacks a fix for mixed-case safeBinTrustedDirs matching on case-insensitive filesystems, so closing this PR would drop a plausible core exec-policy bug fix.
- The latest PR implementation may be wrong when os.tmpdir() is on a case-insensitive volume but a configured safeBinTrustedDirs path is on a case-sensitive volume, because distinct sibling directories can collapse to one lowercase trust key.
- The added PR tests mirror the runtime filesystem probe, so they do not deterministically prove the trust boundary for mixed-sensitivity volumes.
