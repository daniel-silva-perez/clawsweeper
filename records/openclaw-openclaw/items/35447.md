## Summary

Keep open. Current main still stores cron job agent/session metadata and the agent cron tool stamps it during add, but the cron service contract, gateway protocol, gateway handlers, and tool calls still do not carry or enforce trusted caller context for list/update/remove/run/enqueueRun. The linked implementation PRs are closed unmerged; the related open items are narrower leakage or broader policy discussions rather than replacements.

## Best Possible Solution

Keep this issue open and implement a core cron caller-context contract: derive trusted agentId/sessionKey or an explicit operator/admin override from the agent/gateway context, thread it through the gateway protocol/handlers and CronService operations, filter list/listPage, deny update/remove/run/enqueueRun across ownership boundaries, define legacy-job behavior, and add service, gateway, and cron-tool regression tests. Coordinate with #49175 for run-history name leakage and keep #49183 as the broader policy discussion.
