## Summary

Keep this issue open. Current main still stores cron job ownership metadata and stamps it on jobs created from the agent cron tool, but cron list/update/remove/run service operations, gateway handlers, protocol params, and tool calls still do not carry or enforce caller agent/session context. The related implementation PRs #36517 and #49174 are closed unmerged; #49175 is only the narrower cron.runs job-name leakage follow-up, and #49183 is a broader policy RFC rather than a replacement for this fix.

## Best Possible Solution

Keep this issue open and implement a core cron caller-context contract: thread trusted caller agentId/sessionKey or an explicit operator override through the agent tool/gateway protocol/handlers into CronService operations, filter list/listPage, deny update/remove/run/enqueueRun across ownership boundaries, define legacy-job behavior, and add focused service, protocol, gateway, and cron-tool regression tests.
