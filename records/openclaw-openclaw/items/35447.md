## Summary

Keep this issue open. Current main still has the exact ownership-metadata gap described: cron jobs carry `agentId` and `sessionKey`, and the agent cron tool stamps new jobs from the caller session, but the service contract, service operations, gateway protocol, gateway handlers, and cron tool list/update/remove/run calls still do not carry or enforce caller identity. The related implementation PRs #36517 and #49174 are closed unmerged; #49175 covers only the narrower `cron.runs` job-name leakage follow-up, and #49183 is a broader tool-policy RFC rather than a replacement for this cron isolation fix.

## Best Possible Solution

Keep this issue open and implement a core cron caller-context contract: thread caller `agentId`/`sessionKey` or an explicit trusted override from the agent cron tool/gateway context into the gateway protocol/handlers and `CronService` operations, filter list/listPage, deny update/remove/run/enqueueRun across ownership boundaries, define behavior for legacy jobs without ownership metadata, and add regression tests for service, protocol, gateway handler, and cron tool paths. Treat #49175 as a linked follow-up or include it in the same reviewed design if maintainers want one comprehensive cron visibility fix.
