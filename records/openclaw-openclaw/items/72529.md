## Summary

Current main already supports the requested behavior through the generic persistent-session cron target. A cron agent-turn job can use `sessionTarget: "session:<full-session-key>"` / `openclaw cron --session session:<full-session-key>`, and that covers subagent session keys such as `agent:<agentId>:subagent:<uuid>` without adding a separate `subagent` spelling.

## What This Changes

The issue requests a way for cron agent-turn jobs to route directly into an existing persistent subagent session instead of only the main session or a fresh isolated session.

## Best Possible Solution

Keep the existing generic persistent-session cron target and document/use `--session "session:<full-session-key>"` for subagent sessions rather than adding a subagent-specific alias. `--session-key` should remain a metadata/grouping key, not the routing mechanism for existing sessions.

## Reproduction Assessment

Yes. The current-main verification path is to create or inspect a cron agent-turn job with `sessionTarget: "session:agent:<agentId>:subagent:<uuid>"` and confirm the Gateway passes the stripped key to the runner while the runner does not force a new session unless the target is exactly `isolated`.

## Solution Assessment

Yes. The shipped generic `session:<id>` target is the narrowest maintainable solution because it covers subagents, ACP sessions, and other persistent session keys without baking subagent-specific routing into cron.
