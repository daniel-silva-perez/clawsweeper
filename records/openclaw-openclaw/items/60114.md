## Summary

Keep open. Current main does not implement persisted TUI prompt aliases, so the feature is still relevant, but the proposed branch has blocking issues: alias prompts are parsed after whitespace normalization, and unrelated Matrix body-formatting regression tests are deleted.

## What This Changes

The PR adds persisted TUI `/alias` and `/unalias` prompt shortcuts with autocomplete/state wiring and tests, while also rewriting `extensions/matrix/src/matrix/monitor/handler.body-for-agent.test.ts`.

## Best Possible Solution

Revise the PR rather than close it: keep the TUI alias feature, parse alias definitions from the raw command tail so prompts are preserved exactly, retain or relocate all existing Matrix body-for-agent tests, clarify alias list copy, update TUI docs, and then run targeted TUI/Matrix validation before the changed gate.

## Reproduction Assessment

Yes. The alias parsing defect is reproducible from the diff by following `/alias review "line1  line2"` through `parseCommand(raw)` into `parseTuiAliasArgs(args)`, where repeated whitespace is already collapsed. The Matrix regression is reproducible by comparing current main's named tests with the PR diff that deletes them.

## Solution Assessment

No. The requested feature is a reasonable TUI UX improvement, but the current PR is not the best merge shape because it mixes an unrelated Matrix test rewrite and loses exact prompt text before persistence; the safer path is a narrower TUI-focused patch with preserved Matrix coverage.

Parse the alias tail from the raw command string, or make the command parser preserve the original argument substring for this path.

Those are unrelated to TUI aliases and currently guard Matrix inbound prompt formatting, so keep or relocate them instead of deleting them here.
  - confidence: 0.88

  - body: Listing aliases as `/${name} -> ${prompt}` suggests users can invoke `/review` directly, but this handler only runs aliases through `/alias review`. Use copy such as `/alias ${name} -> ${prompt}` or add a header that explains the invocation form.
