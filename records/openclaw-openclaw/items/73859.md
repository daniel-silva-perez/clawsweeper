## Summary

Keep open. Current main still has no clear Windows/default-stack mitigation for the reported bundled plugin import failure, and the affected plugin entrypoints and Gateway load path remain present. The existing loader catch limits startup blast radius, but it does not make minimax, google, or talk-voice usable when module evaluation overflows the stack.

## Best Possible Solution

Keep this issue open and implement a focused packaging/runtime fix. Prefer making the minimax, google, and talk-voice built entrypoints shallow enough to import under default Windows Node stack settings; only use a Windows launcher stack-size guard if maintainers explicitly choose that product tradeoff. Add package smoke coverage that imports dist/extensions/minimax/index.js, dist/extensions/google/index.js, and dist/extensions/talk-voice/index.js with default Node stack, ideally in a Windows package-acceptance lane, while preserving the existing non-fatal plugin-load diagnostics.

## Clownfish Work Prompt

Fix the Windows/default-stack bundled plugin import failure reported for OpenClaw v2026.4.26: `minimax`, `google`, and `talk-voice` fail at Gateway startup with `RangeError: Maximum call stack size exceeded` when importing their built `dist/extensions/<id>/index.js` entries. Work in current main. Prefer reducing built plugin entry import depth and startup fanout for the affected bundled plugins, especially by lazily loading heavy provider/runtime helpers from `extensions/minimax/index.ts`, `extensions/google/index.ts`, and `extensions/talk-voice/index.ts`. Do not add a global `NODE_OPTIONS=--stack-size=...` launcher workaround unless the source-level import-depth fix is not feasible and the tradeoff is explicitly documented in the PR. Add or extend a package/build smoke that imports `dist/extensions/minimax/index.js`, `dist/extensions/google/index.js`, and `dist/extensions/talk-voice/index.js` with default Node stack; if possible, wire/describe Windows package-acceptance proof. Keep core extension-agnostic: no bundled ids in generic core paths except test/smoke inventories where explicit affected entries are the point. Update docs/changelog only if user-facing behavior or release notes policy requires it; this is likely a fix entry with reporter credit if a changelog is touched.
