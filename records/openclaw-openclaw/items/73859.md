## Summary

Keep open. Current main has an adjacent Windows native-require fast path, but it does not prove this reported default-stack built-plugin import overflow is fixed: gateway startup still loads the affected startup plugin entries through the normal loader, and a module-evaluation RangeError still leaves the plugin unavailable even though the loader records the failure and continues.

## What This Changes

The issue reports that bundled `minimax`, `google`, and `talk-voice` plugin entrypoints fail to load on Windows with `RangeError: Maximum call stack size exceeded` unless Node is launched with a larger stack.

## Best Possible Solution

Land a focused Windows/package reliability fix that either makes the affected built plugin entrypoints shallow enough to import under default Windows Node settings or adds a narrowly tested Windows stack-size launcher guard. Preserve nonfatal loader diagnostics, but add package-level Windows proof that `dist/extensions/minimax/index.js`, `dist/extensions/google/index.js`, and `dist/extensions/talk-voice/index.js` import successfully without user-provided `NODE_OPTIONS`.

## Reproduction Assessment

Yes for the reported released package: the issue gives Windows Server 2019, Node v24.14.1, OpenClaw v2026.4.26, exact plugin ids, log lines, and default-stack versus `--stack-size=65536` require commands. Current-main reproduction is not high-confidence until a Windows packaged smoke imports the built entries under the default Node stack.

## Solution Assessment

Unclear for current main. The existing native-require Windows fast path is a useful adjacent optimization, but the maintainable fix needs proof against the reported direct built-entry import overflow, not only faster Jiti avoidance.
