## Summary

Keep open. Current main still has the reported direct browser `/navigate` gap: adjacent managed/click and explicit wait/download flows are covered, but navigation to an attachment URL still goes through a URL-only Playwright path that rethrows non-policy `page.goto()` errors and returns no download metadata.

## What This Changes

The issue requests that CDP-backed browser navigation to attachment/download URLs save the file and return successful download metadata instead of surfacing Playwright's `Download is starting` abort or losing the download.

## Best Possible Solution

Implement a narrow browser-plugin fix in the Playwright/CDP navigation seam: observe or arm a download around guarded navigation, treat `Download is starting` as success only when a real download is captured and saved under the guarded downloads directory, return structured `download` metadata from `/navigate`, and preserve existing SSRF/navigation-policy behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `POST /navigate` to a `Content-Disposition: attachment` URL through the Playwright/CDP path: Playwright 1.59.1 throws `Download is starting` for `Page.navigate` downloads, and current OpenClaw rethrows that non-policy error.

## Solution Assessment

No. Current main is not the best solution because it fixes adjacent explicit/click download paths but leaves direct `/navigate` attachment downloads on a URL-only, error-throwing contract; the safer fix is localized to browser-plugin navigation/download handling plus regression coverage.
