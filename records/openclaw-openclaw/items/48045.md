## Summary

Keep open. Current main still lacks direct browser `/navigate` handling for attachment downloads over the Playwright/CDP path: the helper returns only URL metadata, the route forwards that URL-only result, non-policy `page.goto()` errors are rethrown, and no CDP download protocol hook or attachment-navigation regression coverage is present. Adjacent fixes cover managed/click/explicit download flows, but not this reported navigation-download failure.

## What This Changes

The issue asks OpenClaw's browser navigate path over Playwright/CDP to treat attachment downloads as successful, save the file, and return structured download metadata instead of surfacing Playwright's "Download is starting" abort.

## Best Possible Solution

Implement the missing browser-plugin navigation-download path: arm or observe a download around guarded navigation, treat the download-starting abort as success only when a real download is captured, persist the file through the guarded downloads directory, return structured download metadata from `/navigate`, and preserve current navigation policy checks and explicit waiter ownership.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `POST /navigate` to a `Content-Disposition: attachment` URL through the Playwright/CDP path: Playwright 1.59.1 throws `Download is starting` when `Page.navigate` reports `isDownload`, and current OpenClaw rethrows that non-policy error.

## Solution Assessment

No, current main is not the best solution for this issue because it only covers adjacent managed/click/explicit download flows. The narrow maintainable fix belongs in the browser plugin's guarded navigate/download seam, with route-contract and regression-test updates.

## Repair Work Prompt

Fix the browser plugin's direct `/navigate` handling for attachment downloads over Playwright/CDP. Observable bug: navigating to a URL that responds with `Content-Disposition: attachment` currently propagates Playwright's `Download is starting` error and returns no saved-file metadata. Expected behavior: `/navigate` succeeds only when a real download is captured, saves the file under the guarded OpenClaw downloads directory, and returns structured download metadata such as `{ download: { triggered: true, path, suggestedFilename, url } }` while preserving targetId behavior.

Scope and constraints:
- Work in `extensions/browser/src/browser/**`; do not move owner-specific browser behavior into core.
- Preserve the navigation SSRF/policy guard in `gotoPageWithNavigationGuard` and do not treat arbitrary navigation errors as downloads.
- Avoid double-saving or stealing events from explicit `/download` and `/wait/download` waiters; keep `downloadWaiterDepth` ownership semantics intact.
- Prefer a shared lower-level helper if save-path logic must be reused; do not reintroduce the old #48132 private-helper import cycle or call non-existent Playwright APIs.
- If using CDP-level handling, use a typed narrow adapter and keep failures best-effort unless needed to prove download success.
- Add/update focused regression coverage for `Content-Disposition: attachment` navigation and the route response shape.
- Add a changelog fix entry if behavior changes, crediting the reporter and any source PR context; do not credit bot accounts.

Related refs: this issue, #48132, #64558, and #38519.
