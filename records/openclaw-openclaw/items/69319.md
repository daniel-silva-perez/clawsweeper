## Summary

Keep this PR open. Current main still has `skills.install` and `skills.update` but no `skills.uninstall` registration, handler, schema, docs, or backend uninstall helper, so the PR is not obsolete or implemented on main. The PR is also not safely closeable as-is because the diff has a destructive filesystem-scope concern already called out in review: it can delete an existing workspace skill directory even when that directory is not ClawHub-tracked.

## Best Possible Solution

Keep this PR open for maintainer/contributor rework. The best path is to land a narrow `skills.uninstall` Gateway RPC only after it gates deletion on ClawHub provenance or lock membership, classifies invalid slugs as `INVALID_REQUEST`, uses the current request runtime config pattern, and keeps schema/docs/generated client artifacts plus backend and Gateway handler tests in sync.

## Risks / Open Questions

- The proposed backend can delete a manually managed `skills/<slug>` directory when no ClawHub origin or lockfile entry exists, which is broader than a ClawHub-scoped uninstall RPC should be.
- Malformed slug/path-safety failures are currently surfaced as `UNAVAILABLE` in the PR diff, which can make client input errors look like gateway outages.
- The proposed handler uses `loadConfig()` in the diff while current surrounding handlers use `context.getRuntimeConfig()`; this needs review for import/typecheck correctness and runtime-config consistency.
- This is an additive Gateway protocol change with generated Swift model updates, so schema, docs, method scopes, generated clients, and tests need to stay aligned before merge.
