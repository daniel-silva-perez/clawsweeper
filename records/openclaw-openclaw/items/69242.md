## Summary

Keep this open. Current main has useful lower-level pieces: the supervisor records exit signals and timeout/no-output-timeout state, the exec runtime classifies failures internally, and the Linux OOM victim-selection mitigation is documented. The remaining user-visible bug is still present because foreground exec details and process poll details do not consistently expose `exitSignal`, `failureKind`, `exitReason`/`terminationReason`, `noOutputTimedOut`, runtime, and output tail metadata needed to distinguish timeout, manual cancellation, OS SIGKILL, and other causes. Related #72240 is still open for the same diagnostics gap on macOS, while merged #70419 explicitly treated this Linux report as separate SIGKILL observability work.

## What This Changes

The issue requests clearer handling and diagnostics for Linux broad find/grep exec commands that intermittently terminate with SIGKILL despite no matching host OOM evidence.

## Best Possible Solution

Land a narrow generic exec/process observability fix: propagate existing supervisor/runtime termination data into foreground exec details and background process list/poll/log details, including exitSignal, failureKind, exitReason or terminationReason, timedOut/noOutputTimedOut, duration/runtime, and useful output tail. Cover related #72240 in the same fix, and leave resource limits or broad search guardrails for separate product work.
