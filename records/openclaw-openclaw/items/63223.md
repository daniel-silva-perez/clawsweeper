## Summary

Keep open. Current main still only partially addresses this report: Discord lifecycle status now tracks READY/transport health, but the operator-visible `logged in to discord...` line is still emitted only before the lifecycle wait, and the gateway/Discord recovery path remains a generic in-process channel restart rather than a certificate-expiry-aware process restart or documented watchdog hook. The Raspberry Pi/systemd follow-up also shows this is not only a macOS launchd CA-rotation report.

## What This Changes

The issue asks OpenClaw to recover from Discord TLS certificate-expiry reconnect loops and restore an operator-visible Discord READY login log line.

## Best Possible Solution

Keep this open as the active Discord gateway reliability and observability tracker, or split it into two canonical follow-ups: a narrow Discord lifecycle fix that logs bot identity when the gateway actually reaches READY after a pending startup, and a maintainer-approved recovery/docs path for repeated `certificate has expired` TLS failures, such as channel restart, supervised process restart, or an operator watchdog hook. Coordinate with #71546 and the closed #38596 history, but do not close this until both surfaces are implemented or explicitly split.
