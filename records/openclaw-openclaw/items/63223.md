## Summary

Keep open. Current main still leaves meaningful parts of the report unresolved: Discord startup can still emit only the pre-lifecycle `awaiting gateway readiness` identity line, certificate-expiry/TLS trust-store failures are not classified for a process-level recovery path, and the generic health monitor restarts channels in-process rather than respawning the Gateway process.

## What This Changes

The issue asks OpenClaw to recover a Discord gateway daemon stuck after certificate-expiry/TLS trust-store failures and to restore an operator-visible Discord READY login log line.

## Best Possible Solution

Implement or explicitly split the two remaining surfaces: add a lifecycle-backed Discord READY/RESUMED identity log after the gateway actually reaches READY, and define a maintainer-approved recovery path for repeated certificate-expiry TLS failures that can escape process-local trust-store state. Coordinate the Discord reliability work with #71546 and the closed #38596 history rather than closing this tracker prematurely.
