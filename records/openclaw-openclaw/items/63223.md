## Summary

Keep open. Current main still lacks a certificate-expiry/TLS-trust recovery class that can escape process-local state, and the Discord READY path still updates status without emitting a post-READY operator log line.

## What This Changes

The issue asks OpenClaw to recover the Discord gateway from long-running TLS certificate-expiry zombie states and restore a reliable Discord post-READY identity log.

## Best Possible Solution

Define a maintainer-approved process-level recovery policy for repeated Discord TLS certificate-expiry failures, then add a lifecycle-backed Discord READY/RESUMED identity log and focused tests/docs for both operational paths.

## Reproduction Assessment

Unclear for a full live CA-rotation reproduction in this read-only pass. The missing READY log and absent certificate-expiry classification are directly verifiable in current-main source and tests.

## Solution Assessment

No, current main is not the best solution because it only has in-process reconnect and channel health restarts. A maintainable fix should separate the low-risk READY log from the higher-risk fresh-process recovery decision.
