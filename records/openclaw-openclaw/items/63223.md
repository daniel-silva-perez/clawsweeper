## Summary

Keep open. Current main still has only partial Discord/gateway recovery: generic channel health restarts and READY status patches exist, but the operator-visible post-READY Discord login log is still not emitted after a pending startup becomes ready, and there is no Discord/gateway-specific recovery or docs path for repeated `certificate has expired` TLS failures. The Linux/Raspberry Pi follow-up keeps this from being only a macOS CA-rotation report.

## Best Possible Solution

Keep this open as an active Discord gateway reliability and observability issue. Land a narrow READY observability fix that logs the bot identity when `runDiscordGatewayLifecycle()` actually reaches READY after a pending startup, then make a maintainer-reviewed policy decision for repeated `certificate has expired` TLS failures: channel-only restart, supervised gateway process restart, or documented/operator-exposed watchdog hook. Coordinate with #71546 and the closed #38596/#50046 history, but do not close this issue until both the operator-visible READY signal and cert-error recovery policy are addressed or split into canonical follow-ups.
