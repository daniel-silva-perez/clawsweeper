## Summary

Keep open. Current main still leaves both central parts unresolved: Discord startup can still log only the pre-lifecycle “awaiting gateway readiness” identity line, and certificate-expiry/TLS trust-store failures are not classified into any fresh-process recovery path. The existing health monitor can restart the channel lifecycle in-process, but that does not address the process-local CA-cache failure mode described by the reporter and reinforced by the Debian/systemd follow-up.

## What This Changes

The issue asks OpenClaw to recover from Discord TLS certificate-expiry failures that require a process restart and to restore a Discord READY identity log after gateway readiness.

## Best Possible Solution

Split or explicitly approve the two surfaces: add a lifecycle-backed Discord READY/RESUMED identity log after actual readiness, and define a maintainer-approved recovery path for repeated certificate-expiry TLS failures that can escape process-local trust-store state.
