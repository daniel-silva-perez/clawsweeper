## Summary

Keep open. Current main has partial Discord/gateway recovery work, including READY gating, transport-activity health tracking, and health-monitor restart paths, but it still does not clearly restore the operator-visible READY log after an initially pending gateway becomes ready, and I found no Discord/gateway-specific handling or documentation for the reported `certificate has expired` process-local TLS failure. The additional Linux report also suggests the root cause may not be limited to macOS CA rotation.

## Best Possible Solution

Keep this open and treat it as an active Discord gateway reliability/observability issue. The best path is to split the work into a narrow, low-risk READY observability fix that logs the bot identity when `runDiscordGatewayLifecycle()` actually reaches READY, plus a maintainer-reviewed recovery policy for repeated `certificate has expired` TLS failures that decides whether to restart only the Discord channel, restart the gateway process under service supervision, or document/operator-expose a watchdog hook. The recovery policy should preserve TLS verification and existing health-monitor caps.

## Risks / Open Questions

- The current main behavior is partially fixed compared with 2026.4.5, so a fix should avoid undoing existing Carbon-owned reconnect and health-monitor recovery semantics.
- The reported TLS failure crosses Node trust-store behavior, Discord WebSocket reconnects, and service supervision; forcing `process.exit(1)` on a broad cert string could create restart loops or mask unrelated TLS misconfiguration if done without a precise policy.
- The additional Linux two-hour cycle suggests the original CA-rotation theory may be incomplete; maintainers should verify whether the failure comes from Carbon/Discord reconnect state, gateway metadata fetches, Node TLS cache behavior, or DNS/session renewal before choosing a hard recovery action.
