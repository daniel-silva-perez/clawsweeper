## Summary

Keep open. Current main still does not restore a reliable post-READY Discord identity log, and it still has no Discord-specific certificate-expiry/TLS trust-store recovery path that can escape a process-local CA-cache failure. The Debian/systemd follow-up reinforces that this is not only a launchd/macOS incident, while the existing in-process reconnect and channel health monitor paths do not address the fresh-process recovery requested.

## What This Changes

Fix Discord Gateway behavior where cached TLS certificate failures can leave the daemon unrecoverable without a fresh process, and restore a post-READY Discord identity log for operator health checks.

## Best Possible Solution

Split or explicitly approve the two surfaces: add a lifecycle-backed Discord READY/RESUMED identity log after actual readiness, and define a supervised recovery path for repeated certificate-expiry TLS failures that can escape process-local trust-store state, with docs for the chosen operational model.
