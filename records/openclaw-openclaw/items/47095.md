## Summary

Keep open. Current main still has the reported Discord voice prompt gap: the voice transcript path resolves Discord channel config for authorization, but the successful voice run never carries the per-channel `systemPrompt` into `agentCommandFromIngress`, while normal Discord text/native ingress already maps that config into `GroupSystemPrompt` and the shared reply runner forwards it as `extraSystemPrompt`.

## What This Changes

Discord voice transcript turns should apply the same per-channel Discord `systemPrompt` override that text-channel Discord ingress already applies.

## Best Possible Solution

Update the Discord plugin voice ingress so successful guild voice transcript runs reuse the trusted resolved channel config, build the same per-channel prompt used by text/native ingress, and pass it through the existing `extraSystemPrompt` field while keeping STT transcript text and channel metadata out of trusted system prompt content.

## Reproduction Assessment

Yes. Configure a Discord guild voice channel with `systemPrompt`, drive `processDiscordVoiceSegment` through the existing mocked voice manager path, and inspect the `agentCommandFromIngress` options; current main has no code path that can include the prompt.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Discord plugin: carry or re-resolve the trusted channel config for voice ingress, reuse `buildDiscordGroupSystemPrompt`, and pass the result via existing agent ingress plumbing rather than changing the shared agent runner.
