## Summary

Keep open. Current main still has the reported Discord voice gap: the voice transcript path resolves Discord channel config for authorization, but the successful agent run does not forward the resolved per-channel `systemPrompt`, while Discord text and native ingress already route that prompt through `GroupSystemPrompt` into `extraSystemPrompt`.

## What This Changes

The issue asks for Discord realtime voice transcript turns to honor the same per-channel `systemPrompt` override that Discord text-channel turns already use.

## Best Possible Solution

Carry the trusted resolved Discord channel config through successful voice authorization, derive the same trimmed group prompt used by text/native Discord ingress, and pass it as `extraSystemPrompt` to `agentCommandFromIngress` for authorized guild voice transcript runs while keeping STT transcript text as untrusted user content.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: configure a Discord guild voice channel with `systemPrompt`, process a voice segment, and inspect the `agentCommandFromIngress` options; current main has no path that can include that prompt.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing Discord channel config resolution and `buildDiscordGroupSystemPrompt` behavior for voice ingress, rather than adding a Discord-core special case or changing the shared agent runner.

## Repair Work Prompt

Fix the Discord voice transcript per-channel prompt gap from https://github.com/openclaw/openclaw/issues/47095. Current main resolves Discord channel config in `authorizeDiscordVoiceIngress` for voice authorization but drops it before `processDiscordVoiceSegment` calls `agentCommandFromIngress`, so configured `channels.discord.guilds.<guild>.channels.<voiceChannel>.systemPrompt` is ignored for realtime voice transcript turns. Keep the fix scoped to the Discord plugin: return or otherwise carry the trusted resolved channel config through successful voice authorization, build the same prompt via the existing Discord inbound prompt helper, and pass it as `extraSystemPrompt` to `agentCommandFromIngress` for guild voice runs. Do not trust STT transcript text or channel metadata as system prompt content, do not change broad voice/text routing or the open #53562 `sessionChannelId` design, and preserve existing owner/model/access behavior. Add focused tests that fail before the fix, likely in `extensions/discord/src/voice/access.test.ts` and `extensions/discord/src/voice/manager.e2e.test.ts`, asserting the resolved per-channel prompt reaches the voice agent command. Add a user-facing changelog fix entry with `Thanks @qearlyao` unless maintainers decide this release stream does not need one.
