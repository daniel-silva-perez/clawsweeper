## Summary

Keep open. Current main still has the reported Discord voice prompt gap: the voice segment path authorizes against resolved Discord channel config but does not pass the per-channel `systemPrompt` into the agent run, while the normal Discord text/native paths already convert that trusted config into `GroupSystemPrompt` and then `extraSystemPrompt`.

## What This Changes

Discord voice transcript turns should apply the same per-channel `systemPrompt` override that Discord text-channel ingress applies.

## Best Possible Solution

Update the Discord plugin voice ingress path so a successful guild voice transcript run reuses the resolved trusted channel config, builds the same per-channel prompt as text/native ingress, and passes it through the existing `agentCommandFromIngress` `extraSystemPrompt` field with focused regression coverage.

## Reproduction Assessment

Yes. Configure a Discord guild voice channel with `systemPrompt`, drive `processDiscordVoiceSegment`, and inspect the `agentCommandFromIngress` options; current main has no code path that can include the prompt.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Discord plugin: carry the resolved channel config from voice authorization, reuse `buildDiscordGroupSystemPrompt`, and pass the result via existing ingress plumbing rather than changing the shared agent runner.
