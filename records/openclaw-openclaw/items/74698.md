## Summary

Keep open. Current main still has a Telegram group message-path gate that can drop control commands before any visible reply when no effective command allowlist is configured, and PR #74894 is open and unmerged with closing syntax for this issue. Maintainers should resolve this through that PR or an equivalent reviewed Telegram fix, not by closing or starting a parallel repair lane.

## What This Changes

The issue reports Telegram group `/status` commands silently producing no reply on OpenClaw 2026.4.27 while ordinary group messages still reach agents.

## Best Possible Solution

Land PR #74894 if review confirms the authorization semantics, or replace it with an equivalent focused Telegram fix plus regression coverage showing authorized group `/status` replies in the originating group/topic while explicit group sender allowlists and owner-command restrictions still fail closed.

## Reproduction Assessment

Yes for the code-level failure class: route a group `/status` through `resolveTelegramInboundBody` with `useAccessGroups` enabled and no effective group command allowlist, then compare it with a normal group text message. The exact reporter config is still missing, so the fix should validate both native and message-path command delivery.

## Solution Assessment

Unclear whether the open PR is sufficient as-is because it changes an authorization gate without merged test proof in current main. The narrow maintainable solution is a Telegram-owned fix in the body/native command boundary with tests for no configured allowlist, explicit allowlist, explicit denial, and forum topic reply routing.
