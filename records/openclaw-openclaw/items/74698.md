## Summary

Keep open. The report is fresh against `v2026.4.27`, current `main` still has Telegram group command authorization paths that can block `/status` while normal group messages pass, and open PR #74894 explicitly closes this issue, so the issue should stay open until that PR is reviewed, merged, or superseded.

## What This Changes

Telegram group `/status` slash commands should produce a visible status reply in configured group sessions where normal non-slash messages already route successfully.

## Best Possible Solution

Review PR #74894 or replace it with an equivalent focused fix that lets authorized Telegram group `/status` commands reply in the originating group/topic when group/chat policy already permits the sender, while keeping explicit group allowlists, `commands.allowFrom`, owner-only commands, and unauthorized senders fail-closed.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a configured Telegram group where normal messages pass group policy but no effective command/group allowlist is configured; current command gating then treats `/status` as unauthorized and returns without a visible reply on the text path, while native command auth also rejects absent a configured authorizer.

Yes for the triage path: keeping this issue open for the linked implementation review is the narrowest maintainer action. Closing now would race an unmerged PR and there is no current-main or release proof that the reported behavior is fixed.
