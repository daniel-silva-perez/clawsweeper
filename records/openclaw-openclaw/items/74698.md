## Summary

Keep open. Current main still has a deterministic Telegram group command gate that can drop a command-only `/status` before it reaches the normal command reply path when no effective group command allowlist is configured, and PR #74894 is open with an explicit closing link for that path.

## What This Changes

Telegram group slash commands such as `/status` should send a visible Gateway reply in configured group sessions where normal Telegram group messages already work.

## Best Possible Solution

Land PR #74894 or an equivalent focused Telegram fix that lets configured group control commands reach the command handler when no command/group sender allowlist is configured, while preserving explicit sender allowlists, group chat allowlists, owner-only command checks, and thread/topic reply placement.

## Reproduction Assessment

Yes for a focused code-level reproduction: in a Telegram group command message with default `commands.useAccessGroups`, empty effective `groupAllowFrom`/`allowFrom`, and `/status` as a control command, `resolveTelegramInboundBody` returns `null` before command delivery. The reporter's full live configuration is still missing, so the PR should also cover explicit allowlist variants.

## Solution Assessment

Yes, keeping this issue open for the linked PR review is the right maintainer path. Closing now would race an open unmerged implementation candidate, and current main does not yet contain a source/test/release proof that this group `/status` no-reply path is fixed.
