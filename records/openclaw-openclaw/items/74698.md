## Summary

Keep open. Current main still has a code-level path where Telegram group text can route normally while a whole-message `/status` is treated as an unauthorized control command and dropped before a visible reply; the issue is also paired with open, unmerged PR #74894 using closing syntax, so it should remain open until that PR lands or is superseded.

## What This Changes

The issue requests restoring visible Telegram group replies for slash commands such as `/status` when normal Telegram group messages already route successfully.

## Best Possible Solution

Review and land PR #74894, or an equivalent focused Telegram fix, so configured group `/status` commands reply in the originating group/topic while explicit group sender allowlists, owner-only command checks, disabled groups/topics, and native-command authorization remain fail-closed.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a configured Telegram group where normal text routes but `effectiveGroupAllow.hasEntries` is false; sending `/status` makes the current control-command gate unauthorized and the group handler returns `null` before a visible status reply.

## Solution Assessment

Yes. The narrowest maintainable path is to adjust the early Telegram text-command gate rather than the shared command-gating contract or unrelated native command flow, with normal PR review focused on preserving group sender and owner-command authorization boundaries.
