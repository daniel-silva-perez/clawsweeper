## Summary

Current main does not expose `connected`, `rttMs`, or `lastHeartbeatAt` on the `health` RPC, and there is no `gateway.ping` method in the gateway method list. The existing health response is a cached service/channel/agent/session summary, while the WebSocket tick keepalive only feeds private client liveness state. This is a valid feature request; the fix should be additive and keep per-connection telemetry out of the shared health cache.

## Best Possible Solution

Add an additive connection telemetry surface for the current authenticated gateway connection. Prefer a `connection` block over loose top-level fields if it lands on `health`, and overlay it per response rather than caching it globally. A separate lightweight `gateway.ping` is also reasonable if maintainers want UI polling to avoid the heavier health snapshot path. Update protocol/docs/types/tests and keep channel-specific stale-socket issues separate.

## Clownfish Work Prompt

Implement gateway connection-level telemetry for this issue. Current main’s `health` RPC returns service/channel/agent/session health only. Add an additive surface that lets UI callers determine whether the current gateway connection is live, how fast it is, and when the last connection heartbeat was observed. Prefer adding a `connection` object to `health` with `connected`, `rttMs`, and `lastHeartbeatAt` if it can be overlaid per response; otherwise add a lightweight `gateway.ping` RPC and document why it is the better polling surface.
