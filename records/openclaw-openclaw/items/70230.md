## Summary

Current main still does not expose the requested current-connection telemetry on the Gateway `health` RPC, and there is no `gateway.ping` method. The existing implementation returns a cached/global service health snapshot, while liveness is limited to tick broadcasts and private client watchdog state.

## Best Possible Solution

Add an additive authenticated Gateway connection telemetry surface. Prefer a `connection` block on `health` with `connected`, `rttMs`, and `lastHeartbeatAt` overlaid for the current client and kept out of the cached global snapshot; a separate lightweight `gateway.ping` is also acceptable if maintainers prefer a polling-specific RPC. Update health/protocol docs and focused Gateway tests.

## Clownfish Work Prompt

Implement current-connection telemetry for the authenticated Gateway health surface requested by this issue.

Expected behavior: `health` should include an additive `connection` object for the requesting WebSocket connection with `connected: true`, `rttMs` when a recent measurement exists, and `lastHeartbeatAt` when a heartbeat/liveness signal has been observed. Overlay this object per response; do not mutate or cache it in the shared global `HealthSummary`. If the implementation uses a separate `gateway.ping` instead, document that choice and advertise the method, but prefer `health.connection` unless the existing protocol shape makes that unsafe.

Implementation boundary: keep this in core Gateway health/liveness code. Do not change channel/plugin stale-socket repair policy, and do not treat health probe `durationMs` as network RTT. If measuring `rttMs`, use an explicit WebSocket ping/pong or equivalent connection-level timing source and keep a small rolling mean per connection.
