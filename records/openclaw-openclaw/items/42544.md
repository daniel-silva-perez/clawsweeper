## Summary

Keep open. The underlying service-marker gateway cleanup bug is still valid on current main, but this PR is not safe to merge because its health gate depends on `curl`, probes only loopback, and does not stop startup after detecting a healthy existing gateway.

## What This Changes

The PR adds a `curl`-based `/health` probe before service-marker stale gateway cleanup in `src/cli/gateway-cli/run.ts` and skips cleanup when that probe reports healthy.

## Best Possible Solution

Implement the health gate in the current service-marker startup path with an in-process probe against the resolved gateway probe host. A healthy existing gateway should return successfully without cleanup or another bind attempt; stale cleanup should run only after the probe fails, with focused coverage for healthy, unhealthy, and custom-bind cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Return after detecting a healthy gateway:** `src/cli/gateway-cli/run.ts:271-275`
  - body: In the healthy branch, the patch only logs that stale cleanup was skipped and then continues into normal gateway startup. That can still contend with the already-running gateway instead of making service-mode startup a clean success path.
  - confidence: 0.9
- **[P2] Probe the resolved gateway host:** `src/cli/gateway-cli/run.ts:234-238`
  - body: The new check always probes `127.0.0.1`, but gateway startup can bind through custom, tailnet, LAN, or container-aware hosts. A healthy gateway that is not reachable on loopback can still be treated as stale and killed.
  - confidence: 0.88
- **[P2] Remove the curl subprocess probe:** `src/cli/gateway-cli/run.ts:234-238`
  - body: If `curl` is missing or shadowed in `PATH`, the probe resolves false and proceeds to stale cleanup. This safety gate should use an in-process HTTP probe so correctness does not depend on an external executable.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The PR adds a PATH-resolved `curl` subprocess to gateway service startup for a check that can be done in-process.

Concerns:

- **[medium] Avoid PATH-resolved curl in service startup:** `src/cli/gateway-cli/run.ts:234`
  - body: The health gate runs `curl` from `PATH` before deciding whether to kill gateway processes. That broadens the code-execution surface in service startup and can fall back to killing a healthy gateway when the executable is unavailable or replaced.
