## Summary

Keep open: the reported service-mode cleanup bug is still present on current main, but the submitted patch is not safe to merge because its health gate can still fall through into startup, probes only loopback, and adds a PATH-resolved curl dependency before deciding whether to kill gateway processes.

## What This Changes

The PR modifies `src/cli/gateway-cli/run.ts` to add a curl-based `/health` probe before service-marker stale PID cleanup and skip that cleanup when the probe reports a healthy gateway.

## Best Possible Solution

Implement the service-marker health gate on current main with an in-process probe against the resolved gateway health host. A healthy existing gateway should log and return successfully before stale cleanup or another bind attempt; failed probes should fall back to the existing stale cleanup and startup path, with focused regression tests for healthy, unhealthy, and custom or wildcard bind cases.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: set `OPENCLAW_SERVICE_MARKER`, run gateway startup while a healthy gateway already owns the port, and current main reaches `cleanStaleGatewayProcessesSync(port)` before any health probe.

## Solution Assessment

No. The PR is directionally correct, but the implementation is not the best fix because it depends on `curl`, hardcodes loopback instead of using the resolved health host, and does not return after detecting a healthy gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Return after detecting a healthy gateway:** `src/cli/gateway-cli/run.ts:271-275`
  - body: The healthy branch only logs that cleanup was skipped and then falls through to normal gateway startup. That can still contend with the already-running gateway and fail with a lock error instead of making service-mode startup a clean success path.
  - confidence: 0.9
- **[P2] Probe the resolved gateway host:** `src/cli/gateway-cli/run.ts:234-238`
  - body: The new check always probes `127.0.0.1`, but gateway startup can bind through custom, LAN, tailnet, or container-aware hosts. A healthy gateway that is not reachable on loopback can still be classified as stale and killed.
  - confidence: 0.88
- **[P2] Remove the curl subprocess probe:** `src/cli/gateway-cli/run.ts:234-238`
  - body: If `curl` is missing or shadowed in `PATH`, the probe resolves false and proceeds to stale cleanup. This safety gate should use an in-process HTTP probe so correctness does not depend on an external executable.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete supply-chain/code-execution concern by running a PATH-resolved `curl` subprocess in gateway service startup.

Concerns:

- **[medium] Avoid PATH-resolved curl in service startup:** `src/cli/gateway-cli/run.ts:234`
  - body: The health gate runs `curl` from `PATH` before deciding whether to kill gateway processes. That broadens the code-execution surface in service startup and can fall back to killing a healthy gateway when the executable is unavailable or replaced.
  - confidence: 0.86
