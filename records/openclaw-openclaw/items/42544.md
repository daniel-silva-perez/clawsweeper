## Summary

Keep open: current main still performs service-marker stale gateway cleanup before any health probe, so the user problem remains valid. The PR should not merge as-is because the proposed guard can still fail the healthy-gateway path and adds a PATH-resolved curl dependency in gateway startup.

## What This Changes

The PR changes `src/cli/gateway-cli/run.ts` to probe `GET /health` with `curl` before service-marker stale PID cleanup and skip that cleanup when the probe reports a healthy gateway.

## Best Possible Solution

Implement the service-marker health gate on current main using the existing in-process HTTP probe style against the resolved gateway health host. If the existing gateway is healthy, log and return successfully before stale cleanup or another bind attempt; if not healthy, preserve the existing stale cleanup and startup behavior with focused regression tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: set `OPENCLAW_SERVICE_MARKER`, have a healthy gateway already holding the resolved port, and current main reaches `cleanStaleGatewayProcessesSync(port)` before any health probe.

## Solution Assessment

No. The PR is directionally aimed at the right bug, but this is not the best implementation because it shells out to `curl`, probes a hardcoded loopback host, and does not return after detecting a healthy existing gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Return after detecting a healthy gateway:** `src/cli/gateway-cli/run.ts:271-275`
  - body: The healthy branch only logs that cleanup was skipped and then falls through to normal gateway startup. That can still contend with the already-running gateway and fail with a lock error instead of making service-mode startup a clean success path.
  - confidence: 0.9
- **[P2] Probe the resolved gateway host:** `src/cli/gateway-cli/run.ts:236`
  - body: The new check always probes `127.0.0.1`, but gateway startup can resolve to LAN, tailnet, auto, or custom bind hosts. A healthy gateway that is not reachable on loopback can still be classified as stale and killed.
  - confidence: 0.88
- **[P2] Remove the curl subprocess probe:** `src/cli/gateway-cli/run.ts:234-238`
  - body: If `curl` is missing or shadowed in `PATH`, the probe resolves false and proceeds to stale cleanup. This safety gate should use an in-process HTTP probe so correctness does not depend on an external executable.
  - confidence: 0.91
