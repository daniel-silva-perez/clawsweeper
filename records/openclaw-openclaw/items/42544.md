## Summary

Keep open: the underlying service-mode gateway cleanup bug is still present on current main, but this PR is not mergeable as written because the healthy-gateway path still falls through toward startup, probes the wrong host for non-loopback binds, and adds an external curl dependency to service startup.

## What This Changes

The PR adds a pre-cleanup gateway health probe in `src/cli/gateway-cli/run.ts` so service-marker startup skips stale PID cleanup when an existing gateway responds.

## Best Possible Solution

Implement the health gate in the current service-marker startup path with an in-process probe against the resolved gateway probe host. A healthy existing gateway should return successfully without cleanup or another bind attempt; stale cleanup should run only after the probe fails, with focused coverage for healthy, unhealthy, and custom/wildcard bind cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Return after detecting a healthy gateway:** `src/cli/gateway-cli/run.ts:271-275`
  - body: In the healthy branch, the patch only logs that stale cleanup was skipped and then continues into normal gateway startup. That can still contend with the already-running gateway instead of making service-mode startup a clean success path.
  - confidence: 0.9
- **[P2] Probe the resolved gateway host:** `src/cli/gateway-cli/run.ts:234-238`
  - body: The new check always probes `127.0.0.1`, but gateway startup can bind through custom, tailnet, LAN, or container-aware hosts. A healthy gateway that is not reachable on loopback can still be treated as stale and killed.
  - confidence: 0.88
- **[P2] Remove the curl subprocess probe:** `src/cli/gateway-cli/run.ts:234-238`
  - body: If `curl` is missing or shadowed in `PATH`, the probe resolves false and proceeds to stale cleanup. This safety gate should use an in-process HTTP probe so correctness does not depend on an external executable.
  - confidence: 0.91

## Security Review

Status: needs_attention

  - body: The health gate runs `curl` from `PATH` before deciding whether to kill gateway processes. That broadens the code-execution surface in service startup and can fall back to killing a healthy gateway when the executable is unavailable or replaced.
