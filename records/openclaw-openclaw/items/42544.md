## Summary

Keep this PR open. Current main still performs service-mode stale gateway cleanup before any health probe, so the central bug is not implemented on main. The PR targets a real gateway startup problem, but the submitted diff remains unsafe to merge as-is because it probes with a PATH-resolved curl subprocess, hard-codes loopback, falls through after detecting a healthy gateway, and adds no regression coverage.

## Best Possible Solution

Keep the PR open for maintainer review or contributor rework. The best fix belongs in gateway startup: resolve the effective bind/probe target before stale cleanup, probe the existing listener with built-in Node HTTP or fetch plus timeout handling against `/healthz` or `/health`, return successfully when the existing gateway is healthy, call `cleanStaleGatewayProcessesSync` only for nonresponsive listeners, avoid spawning `curl`, and add focused tests for healthy, unhealthy, and non-loopback service-marker paths.

## Risks / Open Questions

- Closing this PR now would leave current main able to kill a healthy service-mode gateway before startup reaches the existing supervised health recovery path.
- Merging this PR as-is would add a PATH-resolved `curl` subprocess to gateway startup, which is a portability and supply-chain-sensitive regression for a path that can be implemented with built-in Node HTTP/fetch APIs.
- Merging this PR as-is would still fail or misclassify important cases: the healthy branch falls through toward startup, and the probe is hard-coded to `127.0.0.1` instead of the effective bind/probe host.
- The PR adds no focused regression coverage for healthy service-marker preservation, unhealthy stale cleanup, custom/non-loopback bind behavior, or probe failure handling.
