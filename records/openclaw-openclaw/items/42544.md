## Summary

Keep open: current main still runs service-marker stale-PID cleanup before any gateway health probe, so the reported behavior remains valid. The PR targets the right startup path, but it is not safe to merge because the healthy path still falls through to startup, the probe is hard-coded to loopback, and the diff adds a PATH-resolved curl subprocess.

## What This Changes

The PR modifies `src/cli/gateway-cli/run.ts` to add a service-marker pre-cleanup health probe that skips stale gateway PID cleanup when `/health` responds.

## Best Possible Solution

Implement the service-marker health gate on current main with an in-process probe against the resolved gateway health host. A healthy existing gateway should log and return successfully without stale cleanup or another bind attempt; failed probes should fall back to the existing stale cleanup and continue startup, with focused tests for healthy, unhealthy, and custom/wildcard bind cases.

## Reproduction Assessment

Yes. The focused source check is high-confidence: with `OPENCLAW_SERVICE_MARKER` set, current main calls `cleanStaleGatewayProcessesSync(port)` before any `/healthz` probe, so a healthy listener can still be treated as stale by the cleanup helper.

## Solution Assessment

No. The PR is directionally correct but not the best fix because it uses `curl`, probes only `127.0.0.1`, and does not return after detecting a healthy gateway; the safer solution is to reuse or extract the existing in-process health probe and gate cleanup before startup continues.
