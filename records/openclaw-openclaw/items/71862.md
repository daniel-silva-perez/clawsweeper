## Summary

Keep open. Current main does not implement hook-selected auth profiles, and this PR is still the active implementation path, but the patch has a blocking queued-run auth-state gap plus a required changelog omission before merge.

## What This Changes

The PR extends plugin model-resolution hooks to return `authProfileOverride`, threads hook-selected auth profiles into embedded-runner and live-switch selection, and updates related docs, tests, and plugin SDK API baseline.

## Best Possible Solution

Keep the hook/API direction, but carry the legacy auth-profile compaction marker through every queued reply and embedded-runner entry path that can carry a session auth profile. Add regression coverage proving migrated automatic profiles remain overrideable by trusted model hooks while source-less user-pinned profiles stay locked, then add the active-version changelog entry and revalidate.

## Reproduction Assessment

Yes. A static reproduction is clear: use a migrated session entry with `authProfileOverride` plus `authProfileOverrideCompactionCount` and no source, enter the queued reply path, and the new helper receives no compaction count, classifying the profile as user-pinned.

## Solution Assessment

No. The proposed public hook direction is reasonable and narrow, but the implementation is incomplete until queued runs preserve or normalize legacy auth-profile source metadata before `resolvePreferredRunAuthProfile` runs.
