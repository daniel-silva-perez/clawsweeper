## Summary

Keep open. Current main still lacks hook-selected auth-profile support, so the PR is not obsolete; however, the branch still has a concrete functional blocker where the queued reply path drops legacy auto auth-profile compaction metadata, causing migrated automatic profiles to be treated as user-pinned and preventing hook-driven auth-profile selection.

## What This Changes

The PR extends `before_model_resolve` and legacy `before_agent_start` hooks to return `authProfileOverride`, threads that into embedded-runner auth-profile selection and live-switch comparison, and updates related plugin docs and tests.

## Best Possible Solution

Keep the hook/API direction, but thread the legacy auth-profile compaction marker through every queued reply and runner entry path that can carry a session auth profile. Add a regression proving migrated automatic profiles remain overrideable by trusted model hooks while source-less user-pinned profiles stay locked, then rebase and validate the branch.

## Reproduction Assessment

Yes. A static reproduction is clear: use a migrated session entry with `authProfileOverride` plus `authProfileOverrideCompactionCount` and no source, enter the queued reply path, and the new helper receives no compaction count, classifying the profile as user-pinned.

## Solution Assessment

No. The proposed API is a reasonable narrow direction, but the implementation is incomplete until queued runs preserve or normalize legacy auth-profile source metadata before `resolvePreferredRunAuthProfile` runs.
