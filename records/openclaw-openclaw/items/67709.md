## Summary

Keep this issue open. Current main still preserves the reported cold-start stall shape: Control UI refreshes chat on connect, chat refresh fetches history and models together, `models.list` synchronously awaits the gateway model catalog, and `chat.history` still awaits the same catalog when a session lacks a persisted `thinkingLevel`. The newer issue comment also adds a fresh reproduction on `openclaw@2026.4.25`, so this is not stale or limited to the original Windows/WebChat report.

## Best Possible Solution

Keep this issue open and fix the cold startup path. `chat.history` should be able to return stored history without waiting for full model-catalog discovery just to fill a missing thinking default. Control UI should not block initial chat readiness on `models.list`; use a background or timeout-bounded model picker refresh with cached/configured fallback. Add a regression test with a deferred `loadGatewayModelCatalog()` proving cold `chat.history` and initial chat UI readiness do not wait for catalog/pricing discovery, plus coverage for the CLI/TUI default timeout path.

## Risks / Open Questions

- The exact 352-second timing was not live-reproduced in this read-only review, but the current source and the fresh issue comment still match the cold equal-duration stall shape.
- A fix that lets `chat.history` return before full catalog discovery must preserve or deliberately redefine default `thinkingLevel` semantics for sessions without a persisted value.
- Adding only a client-side timeout would improve perceived responsiveness but would not remove the server-side catalog coupling that now also affects CLI/TUI cold `chat.history` calls.
