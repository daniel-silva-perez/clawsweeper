## Summary

Keep open. Current main has configurable provider-backed STT for inbound audio and voice-note media, including OpenAI-compatible transcription helpers, but interactive Talk, push-to-talk, and wake-word voice capture still instantiate Apple/Android platform speech recognizers directly and the Talk config only exposes TTS provider, locale, silence, and interruption settings. The prior GMTekAI comment matches the current code: this is a valid feature request, but it needs a provider seam/product scope decision rather than a small settings toggle.

## What This Changes

Add an opt-in ASR provider setting for custom HTTP/OpenAI-compatible transcription endpoints while keeping built-in platform speech recognition as the default for voice input.

## Best Possible Solution

Add a first-class interactive ASR provider boundary for Talk/voice capture, keep Apple/Android platform speech as the default provider, and reuse the existing media STT/OpenAI-compatible request helpers where practical. The implementation should include config schema/docs/UI, secure secret handling, explicit fallback behavior, and focused tests for provider selection plus platform-default compatibility.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high-confidence: source inspection of macOS/iOS/Android Talk and wake capture paths plus Talk config/docs confirms there is no custom ASR endpoint setting for interactive voice input.

## Solution Assessment

No current implementation solves the requested interactive custom ASR path. Existing `tools.media.audio` is the right reusable building block for inbound voice notes, but not a complete solution for live Talk/wake recognition without a provider abstraction and product-scope decision.
