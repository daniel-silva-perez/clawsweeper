## Summary

Keep open. The current WhatsApp inbound logger still drops the participant and reply fields even though the monitor computes them for the queued inbound message; the related fix PR was closed unmerged after a review that conflated queued message metadata with the log payload, and the latest release shows the same logger omission.

## What This Changes

The issue asks for WhatsApp gateway inbound log entries to include group participant identity and quoted reply context fields.

## Best Possible Solution

Add the already-computed WhatsApp sender and reply-context values to the structured inbound log payload in the WhatsApp plugin, with a focused regression test on the logger call and no changes to mention gating, delivery, access control, or core channel contracts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a group quoted-reply upsert through the existing monitor inbox test harness will produce a logger payload containing only `from`, `to`, `body`, media fields, and `timestamp` while the queued message contains the missing sender/reply metadata.

## Solution Assessment

Yes. The requested approach is the narrowest maintainable fix because it copies optional fields from values already computed in the same function into the existing structured log object, plus a regression assertion for that log call.
