## Summary

Current main and the v2026.4.27 release already fix the reported startup-prewarm stall. The prewarm path no longer imports or calls the PI model resolver fallback that could run provider runtime hooks, and channel startup is not held waiting for a slow or hung prewarm.

## What This Changes

Stop `prewarmConfiguredPrimaryModel` from blocking gateway startup for about 30 seconds when an `openai-codex/*` primary model misses the registry and falls into provider runtime discovery.

## Best Possible Solution

Keep the shipped nonblocking, metadata-only startup prewarm path. Remaining first-message provider factory or broader multi-provider latency should continue under the narrower related reports rather than reopening this startup-prewarm item.

## Reproduction Assessment

Yes. The reported old path is identifiable from the issue steps, and current source plus regression tests give a high-confidence check that startup prewarm no longer calls the model resolver fallback or blocks channel startup.

## Solution Assessment

Yes. The current fix is narrower and safer than canonicalizing `openai-codex` to `codex`, because the docs treat `openai-codex/*` and the bundled `codex` plugin as different routes while the implementation removes the blocking runtime-hook path.
