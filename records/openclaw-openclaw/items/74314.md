## Summary

Keep open. Current main still routes google-antigravity through the Google/pi-ai Cloud Code Assist transport, and the pinned published `@mariozechner/pi-ai` 0.70.6 source still defaults Antigravity to user-agent version `1.21.9`. The auth snapshot side is only partially mitigated: same-process saves update an active snapshot, and failure marking reloads disk under lock, but active runtime snapshots still take precedence over disk until the snapshot is explicitly replaced or reloaded.

## What This Changes

The issue reports that the google-antigravity provider uses a deprecated Antigravity user-agent version and that stale runtime auth snapshots can keep failing after credentials are refreshed.

## Best Possible Solution

Keep the issue open for a maintainer-reviewed provider/auth fix: choose whether OpenClaw should bump or override the Antigravity transport version locally versus wait for `pi-ai`, then add a targeted runtime auth snapshot refresh or invalidation path so successful credential refresh or explicit reload makes the active gateway observe fresh credentials without a restart.

## Reproduction Assessment

Partly yes. The Antigravity version problem is reproducible from current source and the published dependency; the stale snapshot behavior is source-traceable, but a full end-to-end reproduction still needs live google-antigravity OAuth credentials.

## Solution Assessment

Unclear. Updating the Antigravity version is necessary for the provider-side failure, but the auth half should be solved through a narrow runtime snapshot refresh/reload contract rather than broad cache clearing or startup-time migration.
