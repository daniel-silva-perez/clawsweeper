## Summary

Keep open. Current main still leaves both reported problems materially unresolved: google-antigravity can still route through the pi-ai Cloud Code Assist transport with the deprecated Antigravity user-agent default, and runtime auth snapshots still mask externally refreshed auth-profiles.json state until an internal save or secrets-runtime reload replaces the active snapshot.

## What This Changes

The issue asks OpenClaw to restore google-antigravity by using an accepted Antigravity client version and by invalidating or refreshing stale runtime credential snapshots after token refresh.

## Best Possible Solution

Keep this issue open until maintainers choose the provider-version strategy for Antigravity and add a generic auth snapshot refresh or invalidation path that lets externally refreshed auth-profiles.json credentials become active without restarting the gateway.

## Reproduction Assessment

Yes for the local code path: current main shows google-antigravity models still reach the pi-ai google-gemini-cli transport and active runtime snapshots still take precedence over disk state. Live endpoint reproduction still needs maintainer OAuth credentials for google-antigravity.

## Solution Assessment

No complete current-main solution exists. The same-process save snapshot fix covers part of the auth family, but this report still needs an Antigravity client-version decision plus a generic way to refresh or invalidate runtime auth snapshots after external credential updates.
