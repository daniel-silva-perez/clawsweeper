## Summary

Keep open. Current main still routes google-antigravity through the pinned Cloud Code Assist transport whose Antigravity user-agent default is 1.21.9, and the auth snapshot behavior only refreshes active snapshots for same-process saves or explicit secrets reloads, not arbitrary external credential refreshes after a failure.

## What This Changes

Fix google-antigravity so its Cloud Code Assist transport uses an accepted Antigravity version and the active gateway can observe refreshed OAuth credentials without a restart.

## Best Possible Solution

Keep the issue open until maintainers choose a provider-owned Antigravity version strategy and add a generic runtime auth snapshot refresh or invalidation path that lets refreshed auth-profiles.json state become active without restarting the gateway.

## Reproduction Assessment

Yes for the local code path: current main still uses the pinned pi-ai Cloud Code Assist transport with the 1.21.9 Antigravity default, and the snapshot masking path is visible by activating a runtime auth snapshot before changing auth-profiles.json externally. A live Google sandbox reproduction still needs maintainer OAuth credentials.

## Solution Assessment

No current-main solution is complete. The narrow maintainable direction is a Google-provider/dependency strategy for the Antigravity user-agent plus a generic auth snapshot reload/invalidation mechanism, not a core provider-specific migration hidden in startup.
