## Summary

Keep open. Current main still exposes the bundled Google Antigravity path through the Google plugin and the pinned pi-ai transport still defaults to `antigravity/1.21.9`; the auth side also still has only partial recovery because active runtime auth snapshots win over changed disk state until a same-process save or explicit secrets reload replaces them.

## What This Changes

The issue requests fixing the bundled `google-antigravity` provider after Google rejected the pinned Antigravity user-agent version and making refreshed credentials recover without a full gateway restart.

## Best Possible Solution

Keep the issue open for a provider/auth fix that chooses a supported Antigravity version strategy at the Google plugin or upstream pi-ai boundary, then makes fresh OAuth credentials observable by the active gateway without requiring a process restart.

## Reproduction Assessment

Yes for the source-level reproduction: current main routes Antigravity through pi-ai’s `google-gemini-cli` transport, and the current pinned/latest upstream package still emits `antigravity/1.21.9` by default. The live HTTP rejection path needs maintainer credentials to reproduce end to end.

## Solution Assessment

Unclear because there is no patch yet, and the best fix needs maintainer choice across an upstream transport default and runtime OAuth snapshot semantics.
