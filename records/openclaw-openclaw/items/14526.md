## Summary

Keep open. Current main has adjacent updater, staged package replacement, restart health verification, manual backup/verify, status/health, Control UI update status, and auto-update primitives, but not the integrated opt-in safe-update workflow requested here. The recent discussion and related open partials still treat the broader backup, smoke-check, rollback, retention, and restore behavior as remaining work.

## What This Changes

Requests an opt-in safe-update workflow that creates pre-update OpenClaw state backups, runs post-update health and smoke checks, and rolls back/restores on failure with retention and secret-safety controls.

## Best Possible Solution

Make a maintainer decision on whether safe update belongs as a first-class core updater mode. If accepted, implement an explicit opt-in orchestrator around existing update, backup, doctor, status/health, channel-status, Control UI, and package rollback primitives, with retention controls, secret-preserving backup behavior, gateway/channel/real-agent smoke checks, and controlled downgrade plus state restore semantics.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence verification path is source/docs inspection, and current main still lacks the requested safe-update CLI/API workflow.

## Solution Assessment

No, the current adjacent primitives are not the complete solution. The narrowest maintainable path is a designed core updater mode, or a maintainer split across explicit backup-restore, rollback, smoke-check, and Control UI follow-ups.
