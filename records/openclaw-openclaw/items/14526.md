## Summary

Keep open. Current main still has only adjacent primitives: staged package update, doctor, gateway restart/version health verification, manual backup create/verify, manual rollback docs, status/health probes, and related open partials. It does not yet provide the integrated safe-update workflow the discussion asks for, and maintainer-facing comments in the issue explicitly keep that broader scope alive.

## What This Changes

The issue requests an opt-in safe update workflow that creates a pre-update backup, applies the update, runs health and smoke checks, then rolls back or restores state if verification fails.

## Best Possible Solution

Make an explicit maintainer decision on whether safe update belongs in core. If accepted, implement it as a core updater mode that orchestrates existing update, backup, doctor, status/health, channel-status, Control UI update, and package rollback primitives; adds update-tied backups with retention controls; runs gateway, channel, and real-agent smoke checks; and defines controlled downgrade plus state restore behavior with secret-preservation guarantees.

## Security Review

Status: needs_attention

Summary: No patch was reviewed, but the requested feature is security-sensitive because it would package and restore local state, credentials, sessions, workspaces, and possibly downgrade code after migrations.

Concerns:

- **[medium] Define secret-safe backup retention and restore policy:** `docs/cli/backup.md:80`
  - body: Safe-update backups can include credentials, auth profiles, sessions, and workspace data, while current backup docs leave size and retention unmanaged. The design needs explicit permissions, storage, retention, deletion, and leakage controls before implementation.
  - confidence: 0.87
- **[medium] Constrain automatic downgrade and state restore:** `docs/install/updating.md:197`
  - body: The rollback docs are currently manual. Automating downgrade plus state restore can reintroduce older code against state or config touched by a newer version, so compatibility checks and confirmation rules should be designed before this is automated.
  - confidence: 0.84
