## Summary

Keep open. Current main has staged package updates, gateway restart/version verification, manual backup/verify, health/status/channel probes, manual rollback guidance, and auto-update scheduling, but it still does not provide the requested opt-in workflow that ties pre-update snapshots, retention, post-update channel or real-agent smoke checks, and automatic rollback plus state restore into one updater mode. Recent discussion also explicitly kept the broader request active, with narrower related items covering only partial pieces.

## What This Changes

The issue requests an optional safe-update workflow that creates a pre-update OpenClaw state backup, runs post-update health and smoke checks, and rolls back or restores state when checks fail.

## Best Possible Solution

Make a maintainer scope decision for safe update as a first-class core updater mode. If accepted, implement an explicit opt-in orchestrator around the existing update, backup, doctor, health/status, channel-status, Control UI, and package rollback primitives, with retention controls, secret-preserving backup behavior, gateway/channel/real-agent smoke criteria, and controlled downgrade plus state restore semantics.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence verification path is source/docs inspection, and current main still lacks a safe-update CLI/API workflow that ties backups, health checks, smoke checks, rollback, and restore together.

## Solution Assessment

No, the current adjacent primitives are not the complete solution. The narrowest maintainable direction is a designed core updater mode, or a maintainer-approved split across explicit backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.
