## Summary

Current main still does not implement the proposed opt-in safe-update workflow. OpenClaw has useful primitives: manual backup create/verify, update status/run surfaces, auto-update timing policy, doctor checks, restart health/version/plugin diagnostics, Control UI update triggering, and manual rollback docs. The missing center remains automatic pre-update backup, retention policy, channel or real-agent smoke verification, and automatic rollback/state restore. The Bortlesboat auto-update kit linked in the discussion is useful operator prior art/workaround, and #65011 is a broader open product-surface follow-up, but neither is a merged core implementation or superseding tracker.

## Best Possible Solution

Keep this issue open until maintainers either design and implement an opt-in core safe-update mode or explicitly decide the workflow should live outside core. A core solution should build on the existing update, backup, doctor, health, channel status, and Control UI primitives; add automatic pre-update backup creation, retention controls, post-update gateway/status/channel/real-agent smoke checks, and controlled downgrade plus state restore; and coordinate the UX with #65011 while treating Bortlesboat's openclaw-auto-update-kit as useful operator prior art.

## Risks / Open Questions

- A core implementation would touch secrets-bearing backups and must preserve permissions and avoid accidental archive leakage.

- Gateway health alone can miss lane/session failures like the #65035 scenario cited in the discussion, so health criteria need product design.
- This overlaps with the broader open Update Manager request in #65011 and should be scoped with maintainers before implementation.
