## Summary

Keep open. Current main has adjacent updater, auto-update, manual backup/verify, doctor, health/status, channel-status, and manual rollback primitives, but it still does not implement the requested opt-in safe-update workflow tying pre-update snapshots, retention, post-update gateway/channel/real-agent smoke checks, and controlled rollback/restore together. Recent discussion also explicitly kept this broader request active while pointing to partial related work rather than a superseding fix.

## What This Changes

The issue requests an optional safer self-update workflow with pre-update backups, health and smoke checks, retention policy, and automatic or prompted rollback/restore on failure.

## Best Possible Solution

Make a maintainer scope decision for safe update as a first-class updater mode. If accepted, implement an explicit opt-in orchestrator around the existing update, backup, doctor, status/health, channel-status, Control UI, and package rollback primitives, with retention controls, secret-preserving backup behavior, gateway/channel/real-agent smoke criteria, and controlled downgrade plus state restore semantics.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence verification path is source/docs inspection, and current main still lacks a safe-update CLI/API workflow tying backups, health checks, smoke checks, rollback, and restore together.

## Solution Assessment

No, the current adjacent primitives are not the complete solution. The narrowest maintainable direction is a designed core updater mode, or a maintainer-approved split across explicit backup-restore, rollback, smoke-check, retention, and Control UI follow-ups.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it proposes automated archiving of local state plus automated downgrade/restore behavior.

Concerns:

- **[medium] Define secret-safe backup retention and restore policy:** `docs/cli/backup.md:11`
  - body: Safe-update backups can include credentials, auth profiles, sessions, and workspace data, while current backup docs leave size and retention unmanaged. The design needs explicit permissions, storage, retention, deletion, and leakage controls before implementation.
  - confidence: 0.88
- **[medium] Constrain automatic downgrade and state restore:** `docs/install/updating.md:197`
  - body: Rollback is currently documented as a manual operator action. Automating downgrade plus state restore can run older code against state or config changed by a newer version, so compatibility checks and confirmation rules should be designed before automation.
  - confidence: 0.86
