## Summary

Keep open. Current main has adjacent update, staged package replacement, restart health verification, manual backup, manual rollback, Control UI update, and auto-update primitives, but it still does not implement the requested opt-in safe-update workflow with update-tied snapshots, retention, channel/real-agent smoke checks, or automatic rollback plus state restore. The recent discussion and related open items treat this as broader remaining work, not as already shipped or superseded.

## What This Changes

The issue proposes an opt-in safe-update workflow that snapshots critical OpenClaw state before updating, runs post-update health and smoke checks, and rolls back or restores state when checks fail.

## Best Possible Solution

Make a maintainer decision on whether safe update belongs as a first-class core updater mode. If accepted, implement it as an explicit orchestrator around existing update, backup, doctor, health/status, channel-status, Control UI, and package rollback primitives, with retention controls, secret-preserving backup behavior, gateway/channel/real-agent smoke checks, and controlled downgrade plus state restore semantics.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence verification path is source/docs inspection: current main lacks a safe-update CLI/API surface and only documents manual backup, health verification, and rollback steps.

## Solution Assessment

No, the current adjacent primitives are not the complete solution. The narrowest maintainable path is a designed core updater mode or a maintainer decision to split the request across explicit backup-restore, rollback, smoke-check, and Control UI follow-ups.

## Security Review

Status: needs_attention

Summary: The issue itself is security-sensitive because it proposes automated archiving of local state and automated downgrade plus restore after updates.

Concerns:

- **[medium] Define secret-safe backup retention and restore policy:** `docs/cli/backup.md:11`
  - body: Safe-update backups can include credentials, auth profiles, sessions, and workspace data, while current backup docs leave size and retention unmanaged. The design needs explicit permissions, storage, retention, deletion, and leakage controls before implementation.
  - confidence: 0.88

  - body: Rollback docs are currently manual. Automating downgrade plus state restore can run older code against state or config changed by a newer version, so compatibility checks and confirmation rules should be designed before automation.
