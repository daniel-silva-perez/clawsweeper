## Summary

Keep open. Current main still has only adjacent updater, backup, doctor, health/status, auto-update, and package-staging safety primitives; it does not provide the integrated safe-update workflow, and the recent discussion explicitly keeps the broader request alive rather than superseding it.

## What This Changes

The issue requests an opt-in safe update workflow with pre-update backups, post-update health and smoke checks, retention policy, and rollback or state restore on failure.

## Best Possible Solution

Make an explicit maintainer decision on whether safe update belongs as a first-class core updater mode. If accepted, implement it as an orchestrator around the existing update, backup, doctor, health/status, channel-status, Control UI, and package rollback primitives, with retention controls, secret-preserving backup behavior, gateway/channel/real-agent smoke checks, and controlled downgrade plus state restore semantics.
