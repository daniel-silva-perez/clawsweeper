## Summary

Keep open. Current main still has adjacent primitives for updating, doctor checks, gateway restart/version verification, manual backup create/verify, manual rollback docs, and status/health probes, but not the integrated opt-in safe-update workflow requested here. The issue discussion also includes production rollout evidence and maintainer-facing comments that explicitly keep the broader workflow active, while the related items are partial rather than superseding coverage.

## What This Changes

The issue requests an opt-in safe update workflow that creates a pre-update snapshot, applies the update, runs health and smoke checks, then rolls back or restores state if verification fails.

## Best Possible Solution

Make an explicit maintainer decision on whether safe update belongs in core. If accepted, implement it as a core updater mode that orchestrates existing update, backup, doctor, status/health, channel-status, Control UI update, and package rollback primitives, with update-tied backups, retention controls, gateway/channel/real-agent smoke checks, and controlled downgrade plus state restore behavior.

## Security Review

Status: needs_attention

Summary: No patch was reviewed, but the requested feature is security-sensitive because it would package and restore local state, credentials, sessions, workspaces, and possibly downgrade code after migrations.

Concerns:

- **[medium] Define secret-safe backup retention and restore policy:** `docs/cli/backup.md:80`
  - body: Safe-update backups can include credentials, auth profiles, sessions, and workspace data, while current backup docs leave size and retention unmanaged. The design needs explicit permissions, storage, retention, deletion, and leakage controls before implementation.
  - confidence: 0.87
- **[medium] Constrain automatic downgrade and state restore:** `docs/install/updating.md:197`
  - body: The rollback docs are currently manual. Automating downgrade plus state restore can reintroduce older code against state or config touched by a newer version, so compatibility checks and confirmation rules should be designed before automation.
  - confidence: 0.84
