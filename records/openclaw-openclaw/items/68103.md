## Summary

Keep this PR open. Current main has baseline NVIDIA API-key provider support, docs, manifest catalog metadata, and onboarding tests, but it still lacks the PR's dynamic NVIDIA NIM `/models` discovery and expanded model metadata; the branch contains useful unshipped work but is not merge-ready against the current provider SDK contracts.

## What This Changes

The PR expands NVIDIA provider docs and catalog data, adds a NVIDIA models helper with optional NIM `/models` discovery, and wires dynamic catalog/model resolution into the bundled NVIDIA provider.

## Best Possible Solution

Rework this on current main as a focused NVIDIA provider update: keep the existing manifest/onboarding/provider-auth contracts, add optional bounded NIM discovery with static fallback metadata, preserve NVIDIA model-prefix semantics, and cover config writes, discovery fallback, multimodal/reasoning metadata, and secret handling with targeted tests.

## Reproduction Assessment

Not applicable as a user bug report; this is a feature PR. The merge blockers are high-confidence from static inspection: the PR auth shape conflicts with `defineSingleProviderPluginEntry`, the config patch writes outside `models.providers`, and model lookup compares provider-stripped IDs against prefixed catalog IDs.

## Solution Assessment

No. The requested NVIDIA NIM discovery direction is useful, but the current patch is not the best implementation because it bypasses current provider auth/config contracts and needs to be rebased onto the merged NVIDIA provider/onboarding design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use auth option objects in the provider entry:** `extensions/nvidia/index.ts:24`
  - body: `defineSingleProviderPluginEntry` already wraps each `provider.auth` entry with `createProviderApiKeyAuthMethod()`. Passing the created auth method here gives the wrapper an object without option fields such as `methodId`, so NVIDIA setup can fail instead of registering the API-key choice.
  - confidence: 0.9
- **[P1] Write NVIDIA config under models.providers:** `extensions/nvidia/index.ts:37-44`
  - body: This `applyConfig` patch writes `cfg.providers`, but current OpenClaw config stores model providers under `models.providers`. The auth flow can return a config patch that does not configure the actual NVIDIA provider path and can fail type checking on `cfg.providers`.
  - confidence: 0.88
- **[P2] Normalize NVIDIA IDs before catalog lookup:** `extensions/nvidia/index.ts:89`
  - body: Dynamic resolution can receive provider-stripped IDs, while the new catalog stores NVIDIA-owned IDs such as `nvidia/nemotron-...`. Comparing only `ctx.modelId` misses known catalog rows, drops metadata, and can send a bare model ID to the runtime.
  - confidence: 0.84
