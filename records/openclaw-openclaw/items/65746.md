## Summary

Keep open. Current main still lacks the config fallback this PR is trying to add, so the reported PDF custom-provider bug remains real, but the branch is not ready because the new helper bypasses existing registry resolver behavior and only matches inline providers by exact provider id.

## What This Changes

The PR adds a PDF-only fallback from runtime registry lookup to `models.providers` inline models and adds a regression test for a registry-miss custom provider model.

## Best Possible Solution

Update this PR or a replacement so PDF model resolution preserves the current normalized registry lookup, then falls back to `models.providers` using the same normalized provider and provider-scoped model matching semantics as the embedded runner, with regression coverage for the local custom provider case, provider aliases, and provider-prefixed registry model ids.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve registry resolver behavior before config fallback:** `src/agents/tools/media-tool-shared.ts:417-419`
  - body: The new helper calls `modelRegistry.find(params.provider, params.modelId)` directly instead of preserving the existing `resolveModelFromRegistry` path. Current main normalizes refs and retries provider-prefixed custom model IDs, with a regression test for that behavior, so this swap can make registry-backed custom IDs fail before the config fallback is considered. Reuse the existing resolver for the registry phase, or duplicate its semantics before falling back to config.
  - confidence: 0.88
- **[P2] Match inline providers with normalized ids:** `src/agents/tools/media-tool-shared.ts:427`
  - body: The inline fallback only accepts exact provider ids, but `runWithImageModelFallback` has already normalized aliases such as `aws-bedrock` to `amazon-bedrock`. A config under `models.providers.aws-bedrock` can still miss here and throw `Unknown model`; mirror `findInlineModelMatch` so alias keys and provider-scoped model ids resolve consistently.
  - confidence: 0.94
