## Summary

Keep open. Current main still selects `agents.defaults.subagents.model` for isolated cron embedded runs, but the fallback resolver and embedded `fallbackConfigured` flag only consult payload, agent, or default model fallbacks, so the reported configured sub-agent fallback chain can still be skipped.

## What This Changes

The issue requests that isolated cron embedded-agent runs fall back from `kimi/kimi-code` to configured sub-agent model fallbacks when the primary model times out.

## Best Possible Solution

Make isolated cron/sub-agent fallback resolution use the same model source for the primary and fallback candidates: when the selected primary comes from `subagents.model`, honor that object’s `fallbacks`, preserve payload fallback precedence and explicit empty-list strictness, and make embedded failover observability report the effective fallback chain.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and the current source path selects Kimi while never reading the sub-agent fallback list.

## Solution Assessment

Yes. The best fix is generic fallback-policy plumbing shared by isolated cron and embedded failover observability, not a Kimi-specific workaround or a change to provider timeout classification.

Observable bug: an isolated cron `agentTurn` whose primary comes from `agents.defaults.subagents.model` or per-agent `subagents.model` can time out on the primary while `subagents.model.fallbacks` is ignored, causing `fallbackConfigured: false` and no fallback attempt. Expected boundary: update the generic cron/sub-agent fallback policy so payload `fallbacks` still have highest precedence, explicit `fallbacks: []` remains strict, per-agent sub-agent model fallbacks beat default sub-agent fallbacks when that source selected the primary, and normal agent/default model fallback behavior is preserved.
