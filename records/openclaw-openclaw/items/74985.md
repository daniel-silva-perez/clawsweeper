## Summary

Keep open. Current `main` still has the reported policy gap: isolated cron can select a primary from `subagents.model`, but the fallback override and embedded `fallbackConfigured` observation still use helpers that read normal agent/default `model` fallbacks instead of the fallback list attached to the selected sub-agent model object.

## What This Changes

The issue reports that isolated embedded-agent cron sessions using `subagents.model` fallbacks time out on `kimi/kimi-code` without failing over, even though fallback models are configured.

## Best Possible Solution

Add a generic sub-agent-aware fallback resolution path so the same selected sub-agent model config that supplies the primary also supplies its fallback chain when payload fallbacks are absent. Preserve payload fallback precedence, explicit `fallbacks: []` strictness, existing agent/default model semantics, and provider timeout classification.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current `main` has no code path that reads those sub-agent fallbacks for `fallbacksOverride` or `fallbackConfigured`.

## Solution Assessment

Yes. The narrow maintainable fix is fallback-policy plumbing for the sub-agent model selection source, not a Kimi-specific workaround or a change to timeout/error classification.
