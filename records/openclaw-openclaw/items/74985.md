## Summary

Keep open. Current main still appears to ignore sub-agent model fallbacks for isolated cron embedded runs: model selection can use `agents.defaults.subagents.model` as the primary, but the fallback override and `fallbackConfigured` paths only consult payload, agent, or default model fallbacks. The related closed fallback/timeout reports cover adjacent behavior, not this remaining sub-agent fallback source.

## What This Changes

The issue reports that isolated embedded-agent cron runs time out on `kimi/kimi-code` without failing over to configured sub-agent fallback models.

## Best Possible Solution

Make isolated cron/sub-agent fallback resolution use the same model source for both primary and fallback candidates: when the selected primary comes from `subagents.model`, honor that object’s `fallbacks`, while preserving payload `fallbacks` precedence and explicit empty-list strictness. The embedded `fallbackConfigured` observation should reflect the effective fallback chain used by the outer runner.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `agents.defaults.subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and observe that current fallback resolution has no path that reads those sub-agent fallbacks.

## Solution Assessment

Yes, the narrow maintainable solution is generic fallback-policy plumbing, not a Kimi-specific workaround. The fix should thread or compute the effective sub-agent fallback list once and use it for `runWithModelFallback` and embedded failover observability.

## Repair Work Prompt

Fix isolated cron embedded-agent fallback inheritance for sub-agent model defaults. Observable bug: when `agents.defaults.subagents.model` or `agents.list[].subagents.model` is configured as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, isolated cron `agentTurn` runs select the sub-agent primary but do not pass the sub-agent fallback list to `runWithModelFallback`, and embedded failover logs can show `fallbackConfigured: false`. Expected boundary: update cron/sub-agent model selection/fallback policy so the selected sub-agent model object contributes its `fallbacks` when no payload `fallbacks` override is present. Preserve existing semantics: payload `fallbacks` replaces configured fallback policy; payload `fallbacks: []` disables fallback; explicit payload `model` remains a cron job primary and should still use configured fallback policy unless payload fallbacks override it; do not change provider error classification, auth-profile rotation, or unrelated model catalog derivation. Add focused tests for `agents.defaults.subagents.model.fallbacks` and per-agent `subagents.model.fallbacks` flowing into isolated cron `fallbacksOverride`, plus a test that `fallbackConfigured`/effective fallback detection treats sub-agent fallbacks as configured if the implementation touches that helper. Add a changelog fix entry crediting @chrisgwynne if the repo’s active changelog section requires it.
