## Summary

Keep open. Current main still selects the isolated cron embedded-agent primary from `subagents.model`, but the fallback override and embedded `fallbackConfigured` checks continue to read only payload fallbacks plus the ordinary agent/default `model` fallback paths, so configured `subagents.model.fallbacks` can be skipped.

## What This Changes

The issue asks for isolated cron embedded-agent runs using a `subagents.model` primary to fail over to that model object's configured fallback chain when the primary times out.

## Best Possible Solution

Add a generic sub-agent-aware fallback resolution path so isolated cron uses the same configured model object for both the selected primary and fallback chain when payload `fallbacks` is absent. Preserve payload fallback precedence, explicit `fallbacks: []` strictness, existing timeout classification, and avoid folding in the broader precedence changes from #72984 unless maintainers intentionally choose that path.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and the current source path shows the primary is selected from `subagents.model` while fallback resolution never reads its `fallbacks`.

## Solution Assessment

Yes. The best fix is narrow fallback-policy plumbing shared by isolated cron and embedded failover observation, not a Kimi-specific workaround or a provider timeout-classification change.
