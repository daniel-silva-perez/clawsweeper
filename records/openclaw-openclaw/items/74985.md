## Summary

Keep open. Current main still has the reported fallback-policy gap: isolated cron can choose a primary from `subagents.model`, but both the cron fallback override and embedded `fallbackConfigured` paths still resolve fallbacks from payload fallbacks plus per-agent/default `model`, so `subagents.model.fallbacks` can be skipped.

## What This Changes

The issue requests that isolated embedded agent cron runs fail over to the fallback chain configured on `agents.defaults.subagents.model` or per-agent `subagents.model` when the selected primary model times out.

## Best Possible Solution

Add a sub-agent-aware fallback resolution path for isolated cron and embedded failover observation so the same configured sub-agent model object that supplies the selected primary also supplies its fallback chain when payload fallbacks are absent. Preserve payload fallback precedence, explicit `fallbacks: []` strictness, and existing provider timeout classification.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current main has no code path that reads those sub-agent fallbacks for `fallbacksOverride` or embedded `fallbackConfigured`.

## Solution Assessment

Yes. The narrow maintainable fix is generic fallback-policy plumbing for the sub-agent model selection source, not a Kimi-specific workaround or a change to timeout classification.
