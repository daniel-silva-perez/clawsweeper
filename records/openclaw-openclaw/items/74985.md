## Summary

Keep open. Current main still has the reported fallback-policy gap: isolated cron can select a primary from `subagents.model`, but the fallback override and embedded failover-observation paths still resolve fallback configuration from payload fallbacks plus regular agent/default `model` config, so `subagents.model.fallbacks` can be skipped.

## What This Changes

The issue requests that isolated embedded-agent cron runs use configured `subagents.model.fallbacks` when the selected sub-agent primary model times out.

## Best Possible Solution

Add a sub-agent-aware fallback resolution path so the same configured sub-agent model object that supplies the selected isolated-cron primary also supplies its fallback chain when payload fallbacks are absent. Preserve payload fallback precedence, explicit `fallbacks: []` strictness, and existing provider timeout classification.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current main has no code path that reads those sub-agent fallbacks for `fallbacksOverride` or embedded `fallbackConfigured`.

## Solution Assessment

Yes. The narrow maintainable fix is generic fallback-policy plumbing for the sub-agent model selection source, not a Kimi-specific workaround or a timeout-classification change.
