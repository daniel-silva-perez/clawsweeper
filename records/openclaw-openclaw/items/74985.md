## Summary

Keep open. Current main still has the reported sub-agent fallback-policy gap: isolated cron model selection can choose a primary from `subagents.model`, but the fallback override path and embedded failover observability still look at payload fallbacks, normal per-agent `model`, or `agents.defaults.model`, so `subagents.model.fallbacks` can be skipped.

## What This Changes

The issue requests that isolated cron embedded-agent runs fail over from `kimi/kimi-code` to the fallback models configured under the sub-agent model stack when the primary times out.

## Best Possible Solution

Add generic sub-agent-aware fallback resolution so the same `subagents.model` source that supplies an isolated cron primary also supplies its fallback chain when payload fallbacks are absent, while preserving explicit payload fallback overrides, explicit empty-list fallback disablement, and timeout/failover classification behavior.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and the current source path selects the sub-agent primary but does not read those sub-agent fallbacks for `fallbacksOverride` or embedded `fallbackConfigured`.

## Solution Assessment

Yes. The narrow maintainable fix is a shared resolver for the sub-agent model-selection source and its fallback chain, not a Kimi-specific timeout workaround or a broad change to provider failover semantics.
