## Summary

Keep open. Current main still has the reported sub-agent fallback gap: isolated cron can select a primary from `subagents.model`, but the cron fallback override and embedded `fallbackConfigured` paths still resolve fallback policy from payload fallbacks or normal agent/default `model`, so `subagents.model.fallbacks` can be skipped.

## What This Changes

The issue reports that isolated cron embedded agent runs using a sub-agent Kimi primary ignore fallbacks configured on `subagents.model`/model stack and time out without failing over.

## Best Possible Solution

Add a generic sub-agent-aware fallback resolution path for isolated cron and embedded failover observation, using the same configured sub-agent model object that supplies the selected primary to also supply its fallback chain. Preserve payload fallback precedence, explicit `fallbacks: []` strictness, and existing timeout/failover classification.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current main selects the sub-agent primary while resolving fallback policy from non-sub-agent model fields.

## Solution Assessment

Yes. The best repair is generic fallback-policy plumbing for the sub-agent model selection source, not a Kimi-specific timeout workaround or a change to provider error classification.
