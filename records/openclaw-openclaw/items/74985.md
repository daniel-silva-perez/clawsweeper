## Summary

Keep open. Current main at 52e2d4e still has the reported gap: isolated cron model selection can choose `agents.defaults.subagents.model` as the primary, but the fallback override and embedded `fallbackConfigured` paths still consult only payload, per-agent `model`, or `agents.defaults.model` fallbacks. The prior ClawSweeper comment remains accurate because the relevant files did not change between that review SHA and current main.

## What This Changes

The issue asks for isolated embedded-agent cron runs that select Kimi from sub-agent defaults to fail over to fallback models configured on the sub-agent model stack after timeout.

## Best Possible Solution

Add a sub-agent-aware fallback resolution path for isolated cron runs so the same configured model object that supplies the selected primary also supplies its fallback chain when payload `fallbacks` is absent. Preserve payload fallback precedence, preserve `fallbacks: []` strictness, and make embedded failover observability report the effective fallback state.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and inspect that current fallback resolution has no code path that reads those sub-agent fallbacks.

## Solution Assessment

Yes. The best fix is generic fallback-policy plumbing shared by model selection and failover observation, not a Kimi-specific workaround or a broad provider retry-policy change.

## Repair Work Prompt

Fix isolated cron embedded-agent fallback inheritance for sub-agent model defaults. Observable bug: when `agents.defaults.subagents.model` or `agents.list[].subagents.model` is configured as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, isolated cron `agentTurn` runs select the sub-agent primary but do not pass that fallback list to `runWithModelFallback`, and embedded failover logs can show `fallbackConfigured: false`. Expected boundary: update cron/sub-agent model selection or fallback policy so the selected sub-agent model object contributes its `fallbacks` when no payload `fallbacks` override is present. Preserve existing semantics: payload `fallbacks` replaces configured fallback policy; payload `fallbacks: []` disables fallbacks; explicit payload `model` remains a cron job primary and should still use configured fallback policy unless payload fallbacks override it; do not change provider error classification, auth-profile rotation, global sub-agent default precedence, alias resolution behavior from #72984, or unrelated model catalog derivation. Add focused tests for `agents.defaults.subagents.model.fallbacks` and per-agent `subagents.model.fallbacks` flowing into isolated cron `fallbacksOverride`, plus a test that embedded `fallbackConfigured` or the effective fallback detector treats sub-agent fallbacks as configured if that helper changes. If the fix also closes the broader sub-agent fallback gap in #44353, mention it in the PR body; otherwise keep the PR scoped to isolated cron. Add a single-line user-facing changelog fix entry if the active changelog policy requires it, crediting @chrisgwynne and any other directly covered reporter.
