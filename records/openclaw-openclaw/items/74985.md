## Summary

Keep open. Current main still has the reported fallback-policy gap: isolated cron can select a primary from `subagents.model`, but both the cron fallback override path and embedded failover observation still resolve fallback availability from payload/default/per-agent `model` fallbacks, so `subagents.model.fallbacks` can be skipped.

## What This Changes

Embedded agent sub-sessions in isolated cron runs should fail over to the fallback chain configured for the selected sub-agent model when the primary model times out.

## Best Possible Solution

Add generic sub-agent-aware fallback-policy plumbing so the same `subagents.model` object that supplies the isolated cron primary model also supplies its fallback chain when payload fallbacks are absent, while preserving strict explicit-disable and timeout classification behavior.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and the current source path selects the sub-agent primary but does not read those sub-agent fallbacks for `fallbacksOverride` or `fallbackConfigured`.

## Solution Assessment

Yes, the narrow maintainable direction is a generic fallback resolver for the sub-agent model-selection source, not a Kimi-specific timeout workaround or a change to provider timeout classification.
