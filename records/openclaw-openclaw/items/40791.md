## Summary

Keep this PR open. Current main still does not include the external rescue watchdog, Claude runner, systemd timer templates, or rescue-watchdog docs, and the adjacent health, status, Crestodian rescue, and manual rescue-bot surfaces do not implement the same out-of-process incident capture and repair handoff. The PR also still needs maintainer review for functional races and security-sensitive command execution before it can land or be superseded.

## Best Possible Solution

Keep the PR open for maintainer judgment. The best path is to decide whether OpenClaw wants this external script/systemd rescue path alongside the core watchdog-service proposal; if yes, rebase it, drop unrelated test-only churn, fix the cooldown/incident-ID/overlap issues, harden state and incident-file handling, add focused coverage or shell checks for the watchdog behavior, and document the security model for captured diagnostics and Claude runner permissions.

## Risks / Open Questions

- The PR executes an operator-configured rescue runner through a shell and launches Claude Code with edit permissions; this is deliberate but security-sensitive.
- The watchdog sources `state.env` and writes persistent incident state without evident owner/permission/symlink hardening in the proposed script.
- Incident bundles include health/status output, gateway log tails, git status, diff stats, and runner output; docs should warn operators that bundles may contain sensitive local diagnostics.
- Review comments identify actionable races: stale suppression after a healthy cycle, second-granularity incident ID collisions, and overlapping runs before suppression state is persisted.
- The PR overlaps with the separate open core-service watchdog approach, so maintainers need to choose whether an external shell/systemd path belongs in core or should be superseded by the core cron/service design.
