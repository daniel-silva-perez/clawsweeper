## Summary

Keep this PR open. Current main still does not include the external rescue watchdog script, Claude runner, systemd timer templates, or rescue-watchdog docs, and the adjacent health, channel-monitor, Crestodian rescue, and multiple-gateway paths do not provide the same out-of-process incident capture and runner handoff. The related core watchdog PR #46502 is still an open/unmerged product-direction candidate, not shipped evidence that this external shell/systemd PR is implemented or clearly superseded.

## Best Possible Solution

Keep this PR open for maintainer judgment. The best path is to decide whether OpenClaw wants this external shell/systemd rescue path alongside, instead of, or after the core watchdog-service proposal in #46502. If the external path is accepted, rebase it onto current main, remove unrelated test-only churn, harden locking/state/incident-file handling and permissions, add focused shell or unit coverage, and document the security model for captured diagnostics and Claude runner permissions. If #46502 or another core design becomes the accepted shipped implementation, close or mark this PR superseded after that merge with a concrete canonical link.

## Risks / Open Questions

- The proposed watchdog intentionally creates a new local command-execution surface through an operator-configured shell runner and Claude Code handoff; that needs explicit maintainer security approval before merge.
- Incident bundles may contain health/status snapshots, gateway log tails, git status/diff stats, and runner output, so docs and implementation should define storage permissions, redaction expectations, and operator handling clearly.
- The PR still appears to need correctness hardening for suppression reset, overlapping runs, incident ID uniqueness, and state persistence before runner execution.
- The PR overlaps with the open core-service watchdog design in #46502 and should not be auto-closed until maintainers choose a canonical direction or one path lands.
