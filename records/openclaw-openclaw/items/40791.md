## Summary

Keep this PR open. Current main still does not include an external rescue watchdog, Claude runner, systemd rescue timer, or equivalent rescue-watchdog documentation, and the overlapping core watchdog work in #46502 is still open and unmerged. The PR is also not ready to merge as-is because the proposed shell watchdog has concrete duplicate-suppression and incident-collision bugs plus security hardening questions around sourced state and diagnostic bundles.

## What This Changes

The PR branch adds an opt-in external shell rescue watchdog, a Claude Code runner, systemd user service/timer templates, rescue-watchdog docs, and several unrelated test type annotation adjustments.

## Best Possible Solution

Choose the canonical recovery architecture first. If the external watchdog is wanted, keep it scoped to the shell/systemd/docs surface, harden state and incident handling, fix the concurrency bugs, add focused shell or unit coverage, and align the docs with current automation pages. If #46502 becomes the accepted core design, close this PR only after that implementation is merged or maintainers explicitly mark the external shell approach superseded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reset duplicate suppression after healthy checks:** `scripts/rescue-watchdog.sh:175-177`
  - body: When the watchdog sees a healthy cycle it exits without updating state.env. A fail -> recover -> fail sequence within OPENCLAW_RESCUE_COOLDOWN_SEC will reuse the old LAST_FINGERPRINT and skip the new outage as a duplicate, so the runner and notification never fire for a fresh incident.
  - confidence: 0.86
- **[P2] Use collision-proof incident directories:** `scripts/rescue-watchdog.sh:196-198`
  - body: incident_id only has second-level granularity and mkdir -p reuses an existing directory. Two overlapping timer/manual runs in the same second can write into the same incident bundle, leaving mixed files or a partial capture instead of independent incidents.
  - confidence: 0.88
- **[P2] Write suppression state before running rescue commands:** `scripts/rescue-watchdog.sh:250`
  - body: state.env is saved only after RUNNER_CMD returns. During a long Claude rescue run, another watchdog invocation sees no updated fingerprint and can launch a duplicate runner/notification cycle for the same outage; persist state first or take a lock before invoking the runner.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The PR adds an opt-in local command-execution and diagnostic-capture path, and the proposed implementation needs hardening before it is a safe repo-shipped script.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:51`
  - body: load_state sources $STATE_FILE from an environment-configurable state directory without ownership, mode, symlink, or regular-file checks. A user timer that executes a sourced state file should fail closed on unsafe state paths, because replacing state.env turns the next watchdog run into shell execution.
  - confidence: 0.84
- **[medium] Warn about sensitive incident bundles:** `docs/automation/rescue-watchdog.md:35`
  - body: The docs list health snapshots, status output, gateway log tails, git status/diff stats, and runner output in the incident bundle, and the Claude runner passes the incident directory to the model. Operators need explicit guidance on permissions, retention, and possible secret/local diagnostic exposure before enabling this flow.
  - confidence: 0.82
