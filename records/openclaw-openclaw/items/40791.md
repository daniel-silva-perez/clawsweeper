## Summary

Keep open for maintainer review. Current main still does not provide the out-of-process rescue loop this PR is trying to add, but the proposed shell watchdog is not mergeable as-is because it has state/concurrency bugs, security hardening gaps, branch drift, and product overlap with #46502 and #44441.

## What This Changes

The PR adds an opt-in external shell watchdog, a Claude Code incident runner, systemd user timer templates, rescue-watchdog documentation, an auth-monitoring docs link, and several test-only typing adjustments.

## Best Possible Solution

Maintainers should choose the canonical recovery architecture first. If an external shell watchdog remains desired, keep it scoped to scripts/systemd/docs, fix the state and concurrency defects, harden state-file and incident-bundle handling, add focused scripted coverage, and split unrelated test typing changes; if #46502 is selected, salvage only the useful external setup docs or systemd prior art.

## Reproduction Assessment

Yes for the patch defects: the fail-recover-fail cooldown path, same-second overlapping invocation path, and long-runner overlap path are directly traceable through the proposed rescue-watchdog.sh control flow. I did not execute the PR scripts because this review was read-only.

## Solution Assessment

No. The external watchdog idea may be useful, but this PR is not the best mergeable form until the state/concurrency defects, command-execution hardening, incident-bundle guidance, branch drift, and overlap with #46502 are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Reset cooldown state after healthy probes:** `scripts/rescue-watchdog.sh:175-177`
  - body: When the watchdog sees a healthy cycle, it exits without clearing the previous fingerprint and timestamp. A fail -> recover -> fail sequence inside OPENCLAW_RESCUE_COOLDOWN_SEC can therefore suppress the fresh outage and skip incident capture, runner execution, and notification.
  - confidence: 0.86
- **[P2] Generate collision-proof incident directories:** `scripts/rescue-watchdog.sh:196-198`
  - body: incident_id has only second-level granularity and mkdir -p reuses an existing directory. Two overlapping timer/manual runs in the same second can write into the same bundle, producing mixed or partial incident artifacts instead of independent captures.
  - confidence: 0.88
- **[P2] Persist suppression state before the runner:** `scripts/rescue-watchdog.sh:250`
  - body: state.env is saved only after RUNNER_CMD returns. During a long rescue run, another watchdog invocation sees no updated fingerprint and can launch a duplicate runner or notification cycle for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR adds opt-in command execution and diagnostic capture without enough hardening for repo-maintained automation.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:51`
  - body: load_state sources $STATE_FILE from an environment-configurable state directory without regular-file, ownership, mode, or symlink checks. Replacing state.env would turn the next watchdog run into shell execution.
  - confidence: 0.84
- **[medium] Warn about sensitive incident bundles:** `docs/automation/rescue-watchdog.md:38`
  - body: The docs list health snapshots, status output, gateway log tails, git state, and runner output in the incident bundle, and the runner adds that directory to model context. Operators need explicit permissions, retention, and sensitive-data guidance before enabling this flow.
  - confidence: 0.82
