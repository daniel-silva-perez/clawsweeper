## Summary

Keep this PR open. Current main has an opt-in `compat.requiresStringContent` path for strict OpenAI-compatible Chat Completions backends, but it does not definitely implement this PR's intended NVIDIA/default-provider behavior: the bundled NVIDIA catalog and docs still do not opt NVIDIA models into that compatibility flag, and the linked NVIDIA bug remains plausibly unresolved for users following the shipped setup.

## Best Possible Solution

Keep this PR open for maintainer review and reconcile it with the shipped `compat.requiresStringContent` design. The likely best fix is either to add the compat flag to affected NVIDIA/strict provider catalog entries with NVIDIA-focused docs and regression tests, or to revise the wrapper into a generic core path that preserves current newline joining and metadata rules while matching maintainer intent for strict OpenAI-compatible backends.

## Risks / Open Questions

- Closing this PR as implemented would likely leave the linked NVIDIA default setup unresolved because current main only normalizes content after a model-level `compat.requiresStringContent` opt-in.
- Merging the PR as-is may be broader than current main's compatibility policy because it normalizes all `openai-completions` payloads instead of using the shipped model-level compat flag.
- The review discussion still raises behavior details, especially separator preservation and metadata-bearing text blocks, that should be reconciled before merge even though the latest PR text says annotated blocks are preserved.
