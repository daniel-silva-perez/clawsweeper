## Summary

Keep this open. Current main has stronger release validation, beta-first channel semantics, staged package verification, rollback guidance, and update status/dry-run surfaces, but it still does not expose the requested per-release production-ready, early-adopter, or known-issues indicator in release notes, release metadata, or update UI/CLI surfaces.

## What This Changes

The issue requests a per-release production-readiness indicator, such as production-ready, early-adopter, or known-issues, visible in release notes, metadata, or update surfaces.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one canonical source for each release, and expose it first in GitHub release notes; update/status surfaces can consume the same metadata later if maintainers want CLI visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: docs:list, targeted release/update docs, exact term searches, update CLI registration, changelog, release workflow, and related-item context show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release-process metadata rather than heuristic-only update warnings.
