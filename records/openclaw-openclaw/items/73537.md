## Summary

Keep this open. Current main has stronger release validation, beta-first channel semantics, update dry-run/status surfaces, staged package verification, and rollback guidance, but it still does not expose the requested per-release production-ready / early-adopter / known-issues indicator in release notes, release metadata, or update UI/CLI surfaces.

## What This Changes

The issue requests a visible production-readiness stability indicator on each release so production operators can decide whether to upgrade.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one canonical source for each release, and expose it first in GitHub release notes; update/status surfaces can consume the same metadata later if maintainers want CLI visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: docs:list, targeted release/update docs, exact term searches, update CLI registration, latest changelog section, CODEOWNERS, and related-item context show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the public taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release process metadata rather than a heuristic-only update command.
