## Summary

Keep open. Current main has stronger release validation, beta-first channel semantics, update dry-run/status surfaces, and post-publish/package checks, but it still does not provide the requested visible per-release production-ready / early-adopter / known-issues indicator in GitHub release notes, release metadata, or update UI/CLI surfaces. The related pre-upgrade risk PR is still open and unmerged, and it would not by itself define a release-owner readiness label.

## What This Changes

Request to add a visible production-readiness or stability indicator to each OpenClaw release, such as production-ready, early-adopter, or known-issues metadata in release notes or release surfaces.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, make release operators set that value from a single source of truth for each release, and expose it consistently in GitHub release notes first, then in update/status surfaces if maintainers want CLI visibility.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: docs:list, targeted release/update docs, exact term searches, CLI registration, latest changelog, and the v2026.4.29 release body show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start with release metadata/process policy rather than a heuristic-only update command.
