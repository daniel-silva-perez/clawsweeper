## Summary

Keep this issue open. Current main has beta-first release lanes, release validation, package acceptance, update status, dry-run, and update hardening, but it still does not provide the requested per-release production-readiness / early-adopter / known-issues indicator in GitHub release notes, release metadata, or update surfaces. The related open update-review PR and broader release-trust issues overlap, but they do not supersede this narrower release-label request.

## Best Possible Solution

Keep this issue open for release-process/product follow-up. A good solution would add a documented release-readiness enum, require release operators to set it for each GitHub release note or release metadata record, and optionally expose that value through update status/review surfaces. Coordinate broader update-risk UX with #61973, #65011, and #65302, but preserve this issue as the narrow request for a visible per-release stability indicator.
