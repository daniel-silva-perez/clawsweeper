## Summary

Keep this issue open. Current main has stronger release validation, beta-first channel policy, update hardening, dry-run/status surfaces, and post-publish checks, but it still does not provide the requested visible per-release production-ready / early-adopter / known-issues indicator in GitHub release notes, release metadata, or update surfaces. The related update-risk PR and broader update-trust issues overlap, but they do not supersede this narrower release-label request.

## What This Changes

Request a visible production-readiness or stability indicator on each OpenClaw release so operators can decide whether a production setup should update.

## Best Possible Solution

Define a documented release-readiness enum and release-process source of truth, require release operators to set it for each GitHub release note or release metadata record, and then expose it consistently in update/review surfaces where useful.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: docs:list, targeted terminology searches, release workflow inputs, update CLI registration, release docs, update docs, and the latest changelog show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release metadata/process policy before adding CLI or UI exposure.
