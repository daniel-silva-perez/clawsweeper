## Summary

Keep this issue open. Current main has release lanes, beta-first validation, package acceptance, update status, dry-run, and package-update hardening, but it still does not provide the requested visible per-release production-readiness / early-adopter / known-issues indicator in GitHub release notes, release metadata, or update surfaces. The related update-risk PR and broader release-trust issues overlap, but they do not supersede this narrower release-label request.

## What This Changes

Request a per-release production-readiness/stability indicator in release notes, release metadata, or update surfaces so production operators can decide whether to upgrade.

## Best Possible Solution

Add a documented release-readiness enum and source of truth, require release operators to set it for each GitHub release note or release metadata record, and optionally expose it through update status/review surfaces. Coordinate broader update-risk UX with #61973, #65011, and #65302 without losing this issue's narrow per-release indicator.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: docs, release workflow inputs, update CLI registration, and terminology searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the taxonomy and criteria. The requested direction is plausible, but the maintainable fix should start with release-process policy and metadata/source-of-truth design before adding CLI or UI exposure.
