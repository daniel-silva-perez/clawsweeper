## Summary

Keep open. Current main has release channels, beta-first validation, release checks, update status, dry-run, and package-update hardening, but it still does not provide the requested visible per-release production-readiness / early-adopter / known-issues indicator in GitHub release notes, release metadata, or update surfaces. The related update-risk PR and broader update-trust issues overlap, but none supersede this narrower release-label request.

## What This Changes

Request to add a visible production-readiness or stability indicator to each OpenClaw release, such as production-ready, early-adopter, or known-issues status.

## Best Possible Solution

Add a documented release-readiness enum and canonical source of truth, require release operators to set it for each GitHub release note or release metadata record, and optionally expose it through update status or update-review surfaces while coordinating with #61973, #65011, and #65302.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, channel docs, update docs, CLI registration, workflow inputs, and terminology searches show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the taxonomy and criteria. The requested direction is plausible, but the maintainable implementation should start with release-process policy and metadata/source-of-truth design before adding CLI or UI exposure.
