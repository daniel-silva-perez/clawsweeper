## Summary

Keep open. Current main has beta-first release lanes, release validation gates, update docs, dry-run/status support, and recent update hardening, but it does not provide the requested per-release production-readiness / early-adopter / known-issues indicator in release notes, release metadata, or update surfaces. Related #65302, #65011, and #61973 overlap with release/update trust, but they do not clearly supersede this narrower release-label request.

## Best Possible Solution

Keep this issue open for release-process/product judgment. A good implementation would add a small documented release-readiness enum, require release operators to set it for each GitHub release note, and optionally expose it through update status/review surfaces. Tie the policy to existing beta-first validation and package acceptance evidence, while keeping broader release-stability feedback coordinated with #65302 and UI/CLI update planning with #65011 and #61973.
