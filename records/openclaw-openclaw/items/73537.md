## Summary

Keep open. Current main has release lanes, beta-first validation, package acceptance, update status, dry-run, and package-install hardening, but it still lacks the requested visible per-release production-readiness indicator in release notes, release workflow metadata, or update surfaces.

## What This Changes

The issue requests a visible production-readiness/stability label for each release, such as production-ready, early adopter, or known issues.

## Best Possible Solution

Define a documented release-readiness enum and source of truth in the release process, require release operators to set it for each GitHub release or release metadata record, and then expose it in release notes and optionally update status/review surfaces. Coordinate with #61973, #65011, and #65302 without losing this narrower per-release indicator.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: targeted searches plus release docs, workflow inputs, and update CLI registration show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start with release-process policy and metadata before CLI or UI exposure.
