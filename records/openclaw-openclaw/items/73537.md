## Summary

Keep open. Current main and the latest GitHub release surface release lanes, validation, update status, and dry-run safety checks, but they still do not expose the requested per-release production-readiness, early-adopter, or known-issues indicator. Related update-trust work is open and overlapping, but it does not supersede this narrower release-label request.

## What This Changes

The issue requests a visible stability/readiness indicator on each OpenClaw release so production operators can decide whether to upgrade.

## Best Possible Solution

Define a documented release-readiness enum and decision criteria in the release process, require release operators to set it for each GitHub release or release metadata record, and optionally expose the same value in update status/review surfaces once the semantics are agreed.

## Reproduction Assessment

Not applicable as a feature request. The current-main check is a docs/source/release-surface inspection showing no per-release stability marker in release policy, update docs, update CLI output shapes, changelog format, or the latest release notes.

## Solution Assessment

Unclear until maintainers choose the label semantics. A narrow readiness metadata field plus release-note display is likely maintainable, but the criteria and promise level should be decided before an automated implementation PR.
