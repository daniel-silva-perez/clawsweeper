## Summary

Keep open. Current main still short-circuits live-PID runtime-deps locks before consulting `createdAtMs` or filesystem mtimes, so the reported legacy Docker/PID-reuse edge remains valid and narrow enough for a focused fix.

## What This Changes

The issue asks for legacy plugin runtime-deps locks that lack `starttime`/`pidStartTimeMs` and `createdAtMs` to expire from lock/owner-file mtimes instead of staying fresh forever when the recorded PID is alive.

## Best Possible Solution

Ship a narrow runtime-deps lock staleness change with regression tests: preserve start-time matching for active owners, but let legacy PID-alive locks with no usable start-time and stale `createdAtMs` or lock/owner-file mtimes be removed.

## Reproduction Assessment

Yes. A focused source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 1, lockDirMtimeMs: old, ownerFileMtimeMs: old }, now, () => true)`, which current main returns `false` before reaching the mtime fallback; the Docker restart path in the issue is the integration form of the same predicate failure.

## Solution Assessment

Yes, a narrow fix in `src/plugins/bundled-runtime-deps-lock.ts` plus colocated regression coverage is the best path. The safer implementation should add the legacy fallback after dead-PID/start-time checks and preserve existing active-lock and ownerless-lock behavior.
