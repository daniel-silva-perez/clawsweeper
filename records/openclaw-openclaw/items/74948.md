## Summary

Keep open. Current main still treats a live-PID bundled runtime-deps lock as fresh before consulting legacy creation-time or filesystem mtime fallback data, and the reporter has already linked open PR #74950 for the same lock/test surface.

## What This Changes

The issue requests expiring stale legacy bundled runtime-deps lock directories whose owner PID appears alive after a Docker restart but lacks usable start-time and/or creation-time evidence.

## Best Possible Solution

Land PR #74950 or an equivalent narrow change in the bundled runtime-deps lock predicate, with regression coverage for stale legacy same-PID locks while preserving verified active owner/start-time safety checks.

## Reproduction Assessment

Yes. A focused source-level reproduction is a call to `shouldRemoveRuntimeDepsLock()` with a live PID, no matching start-time evidence, and stale `createdAtMs` or lock/owner-file mtimes; current main returns `false` before those stale fallbacks can run.

## Solution Assessment

Yes. The narrowest maintainable solution belongs in `src/plugins/bundled-runtime-deps-lock.ts` with colocated tests in `src/plugins/bundled-runtime-deps.test.ts`; the already-open linked PR is the right implementation candidate to review first.
