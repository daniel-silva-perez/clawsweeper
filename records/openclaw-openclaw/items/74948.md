## Summary

Keep open. Current main still treats any live-PID bundled runtime-deps lock as fresh after only the optional start-time mismatch check, so the reported legacy Docker lock path remains valid; the same reporter has already linked open PR #74950 for the narrow fix, so this should wait for that PR or an equivalent maintainer-reviewed change.

## What This Changes

The issue requests expiring stale legacy bundled runtime-deps lock directories that have a recycled live PID but lack start-time or creation-time evidence, using creation-time or filesystem-mtime fallback checks.

## Best Possible Solution

Review and land PR #74950, or make the same narrow predicate change with regression coverage for live-PID legacy locks missing start-time and creation-time evidence while preserving active lock safety for matching start-time owners.

## Reproduction Assessment

Yes. A source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 7, lockDirMtimeMs: 1_000, ownerFileMtimeMs: 2_000 }, 1_000_000, () => true)`, which current main returns `false` because the live-PID branch exits before stale mtime fallback.

## Solution Assessment

Yes. The maintainable fix belongs in `src/plugins/bundled-runtime-deps-lock.ts` with focused tests in `src/plugins/bundled-runtime-deps.test.ts`; a broader lock implementation rewrite is not required to address this edge case.
