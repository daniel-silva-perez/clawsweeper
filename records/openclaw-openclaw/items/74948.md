## Summary

Keep open. Current main still treats any live-PID runtime-deps lock as fresh before consulting `createdAtMs` or filesystem mtime fallback data, and the reporter has already linked open PR #74950 for this exact legacy-lock edge case.

## What This Changes

Request to expire stale legacy bundled runtime-deps locks whose owner PID appears alive in Docker but whose owner record lacks start-time and creation-time evidence.

## Best Possible Solution

Land a narrow fallback in the runtime-deps lock predicate that preserves dead-PID and starttime mismatch behavior, expires abandoned legacy PID-alive locks from `createdAtMs` or the latest lock/owner-file mtime after the stale threshold, and adds focused regression coverage for the Docker legacy-lock case plus active-lock preservation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 1, lockDirMtimeMs: old, ownerFileMtimeMs: old }, now, () => true, () => null)`, which current main returns `false` through the live-PID branch.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/plugins/bundled-runtime-deps-lock.ts` with colocated tests; because PR #74950 is already linked for the same change, the next step is to review, land, or revise that PR rather than queue a separate repair.
