## Summary

Keep open. Current `main` still returns `false` for a live-PID runtime-deps lock before consulting `createdAtMs` or filesystem mtime fallbacks, and the reporter has an open paired repair PR, so this should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that legacy runtime-deps locks missing reliable start-time metadata expire using stale timestamp evidence instead of being treated as fresh forever when their recorded PID is alive.

## Best Possible Solution

Land PR #74950 or an equivalent narrow predicate change with regression coverage that preserves dead-PID and start-time mismatch cleanup while safely reclaiming stale legacy live-PID lock directories.

## Reproduction Assessment

Yes. A source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 1, lockDirMtimeMs: old, ownerFileMtimeMs: old }, now, () => true)`: current main reaches the live-PID branch and returns `false` before the stale mtime fallback.

## Solution Assessment

Yes, the narrowest maintainable direction is in `src/plugins/bundled-runtime-deps-lock.ts` with colocated tests. The implementation should be reviewed to preserve the active-lock invariant rather than broadly expiring all live-PID locks by age.
