## Summary

Keep open. Current `main` still treats a live-PID runtime-deps lock as fresh before the legacy `createdAtMs` or filesystem-mtime fallback can run, and the same reporter has already linked open PR #74950 as the implementation candidate for this edge case.

## What This Changes

The issue asks for legacy bundled runtime-deps lock files without process start-time or creation-time metadata to expire using filesystem mtime fallback instead of staying fresh forever when the recorded PID is alive.

## Best Possible Solution

Review and land PR #74950, or apply the same narrow predicate update with regression coverage for live-PID legacy locks missing both start-time and creation-time evidence while preserving active-lock safety for verified current owners.

## Reproduction Assessment

Yes. A source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 7, lockDirMtimeMs: 1_000, ownerFileMtimeMs: 2_000 }, 1_000_000, () => true)`, which current `main` returns `false` because it exits at the live-PID branch before stale mtime fallback.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `src/plugins/bundled-runtime-deps-lock.ts` with focused tests in `src/plugins/bundled-runtime-deps.test.ts`; a broader lock rewrite is not required for this edge case.
