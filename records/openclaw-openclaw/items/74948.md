## Summary

Keep open. Current main still treats any live-PID runtime-deps lock as fresh after the optional start-time check, so a legacy lock with no `starttime` and no `createdAtMs` never reaches the mtime staleness fallback; the reporter also linked an open paired fix PR that should be reviewed rather than closing this issue early.

## What This Changes

The issue requests expiring stale legacy bundled runtime-deps locks whose owner has a live PID but lacks start-time and creation-time metadata by falling back to lock and owner-file mtimes.

## Best Possible Solution

Review and land PR #74950, or an equivalent narrow patch, so legacy PID-alive locks without reliable start-time evidence can expire from stale creation time or lock/owner-file mtimes while existing active-owner protections remain intact.

## Reproduction Assessment

Yes. A focused source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 1, lockDirMtimeMs: old, ownerFileMtimeMs: old }, now, () => true)`, which current main returns `false` for before reaching the mtime fallback.

## Solution Assessment

Yes for the requested direction. The maintainable fix is localized to `shouldRemoveRuntimeDepsLock()` plus colocated regression tests, with careful gating so mtime fallback applies only when stronger PID/start-time evidence cannot prove the lock is active.
