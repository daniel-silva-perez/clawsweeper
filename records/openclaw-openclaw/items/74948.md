## Summary

Keep open. Current main still short-circuits live-PID runtime-deps locks before consulting legacy filesystem mtime fallback, and the reporter has linked an open, unmerged repair PR for the same issue, so this should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that legacy plugin runtime-deps lock files without `starttime`/`pidStartTimeMs` or `createdAtMs` can remain fresh forever when the recorded PID is alive in Docker PID-reuse scenarios.

## Best Possible Solution

Review and land PR #74950, or make an equivalent narrow change that preserves active owner/start-time checks while expiring stale legacy PID-alive locks with regression coverage in the bundled runtime-deps tests.

## Reproduction Assessment

Yes. A focused source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 1, lockDirMtimeMs: old, ownerFileMtimeMs: old }, now, () => true)`, which current main still returns `false`; the Docker restart case in the report is the integration form of that branch.

## Solution Assessment

Yes. The narrow fix belongs in `src/plugins/bundled-runtime-deps-lock.ts` with colocated tests; the open PR is the appropriate implementation candidate to review before closing this issue.
