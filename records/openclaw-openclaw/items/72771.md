## Summary

Keep open. Current main does not implement `agent.run.status`, and the PR is useful in direction, but it should not merge as-is because the latest metadata fix still writes model, usage, and tool-count data only on the synthetic terminal lifecycle event that normal embedded runs skip. Greptile’s original tool-count concern was acknowledged, but the follow-up does not fix the common terminal path.

## What This Changes

The PR adds an `agent.run.status` gateway broadcast event, registers it in gateway event discovery and scope guards, and attempts to propagate start/completion status, timing, model, usage, tool-call count, and exit reason from agent lifecycle events.

## Best Possible Solution

Revise the PR so the real terminal lifecycle event that reaches the gateway carries consistent startedAt, model, usage, and tool-count data; then add focused gateway/agent tests plus the required changelog entry for the new public event.

## Reproduction Assessment

Yes. A high-confidence static path is enough: normal embedded runs emit lifecycle `end`/`error` from `handleAgentEnd`, `agentCommandInternal` sets `lifecycleEnded = true`, and the PR’s enriched synthetic terminal event is skipped.

## Solution Assessment

No. The gateway broadcast approach is reasonable, but enriching only the fallback synthetic lifecycle event is not the narrow maintainable fix because it misses the common terminal lifecycle path.
