## Summary

Keep open. Current main does not already implement `agent.run.status`, and the PR is directionally useful, but the proposed metadata enrichment misses the common embedded-run completion path because normal lifecycle end/error events are emitted before `agent-command.ts` has result metadata.

## What This Changes

The PR adds a gateway `agent.run.status` event, registers it for gateway broadcasts, and attempts to include start/completion status, timing, model, usage, tool-call count, and exit reason from agent lifecycle events.

## Best Possible Solution

Revise the PR so the real embedded and ACP lifecycle events that reach the gateway carry consistent `startedAt`, model, usage, and tool-count data, then add focused gateway/agent tests for started, completed, failed, and interrupted status payloads.

## Reproduction Assessment

Yes. A static reproduction path is clear: follow a normal embedded run through `handleAgentEnd`; it emits lifecycle `end`/`error`, `agentCommandInternal` sets `lifecycleEnded = true`, and the new synthetic enriched terminal event is skipped.

## Solution Assessment

No. Adding the gateway broadcast is reasonable, but enriching only the fallback synthetic terminal lifecycle event is not the best fix because it misses the normal terminal lifecycle path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Populate metadata on the real terminal lifecycle event:** `src/agents/agent-command.ts:996-1019`
  - body: Normal embedded runs emit lifecycle `end`/`error` from `handleAgentEnd`, causing `agentCommandInternal` to set `lifecycleEnded = true` and skip the new block that adds `model`, `usage`, and `toolCallCount`.
