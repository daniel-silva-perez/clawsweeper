## Summary

Keep open. Current main still lacks a cron-owned structured outbound delivery log that ties each cron send attempt to the originating job/run, account, target, and platform message ID; existing cron run traces and generic outbound hooks cover adjacent diagnostics but not this attribution contract.

## What This Changes

Request to emit exactly one privacy-safe structured `cron.delivery.sent` log record for each outbound cron delivery success or error with cron job/run, agent, account, channel target, status, and platform message ID attribution.

## Best Possible Solution

Add a cron-owned, privacy-safe delivery telemetry layer at the direct cron outbound send boundary: carry the existing cron job/run identity and resolved account/channel/target into delivery, emit `cron.delivery.sent` after both success and error paths, map returned platform message IDs into channel-specific fields, and keep message bodies, prompts, tokens, and other sensitive content out of the log.

## Reproduction Assessment

Yes. The missing behavior is reproducible by source inspection or a focused unit test around `dispatchCronDelivery`: current main can deliver through `deliverOutboundPayloads`, but no logger receives a `cron.delivery.sent` event with cron job/run IDs and returned platform message IDs.

## Solution Assessment

Yes. The requested cron-owned telemetry is the narrow maintainable solution; generic `message_sent` hooks and bounded diagnostic events are either content-bearing, plugin-facing, or lack cron attribution, so they should not be treated as the forensic log requested here.
