## Summary

Keep open. Current main has cron run delivery traces and generic outbound message hooks, but it still does not emit a cron-owned `cron.delivery.sent` structured log record that ties each outbound attempt to `cronJobId`, `cronRunId`, `agentId`, `accountId`, and the platform message ID.

## What This Changes

Request to emit exactly one privacy-safe structured `cron.delivery.sent` log line for every outbound cron delivery attempt, covering success and error cases with cron/job/run/account attribution and platform message IDs.

## Best Possible Solution

Add a cron-owned delivery telemetry path around the direct outbound send boundary: thread `{ cronJobId, cronRunId, agentId }` plus resolved `{ accountId, channel, to }`, emit a structured privacy-safe `cron.delivery.sent` record after success and error paths, map returned platform message IDs into channel-specific fields, and keep message bodies, prompts, tokens, and other sensitive content out of the log.

## Reproduction Assessment

Yes. The missing behavior is reproducible by code inspection or a focused unit test around `dispatchCronDelivery`: current main calls `deliverOutboundPayloads` and records delivered state, but no logger receives a `cron.delivery.sent` event with cron job/run IDs and returned message IDs.

## Solution Assessment

Yes. The requested cron-owned telemetry wrapper is the narrow maintainable solution because generic `message_sent` hooks are content-bearing plugin/internal hooks and lack cron job/run attribution; adding the log at the cron direct delivery boundary keeps the privacy and attribution contract local to this feature.
