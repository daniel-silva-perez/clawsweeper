## Summary

Keep open. Current main has partial Chrome MCP hardening, but passive browser status and tab routes can still spawn or await `chrome-devtools-mcp` without the route's short budget, and sticky timeout cleanup is still incomplete. The Windows Defender comments are useful related context, but they do not resolve the original macOS/Node 25 `npx` hang report.

## What This Changes

The issue asks OpenClaw to stop browser control routes from hanging and leaking orphaned processes when the Gateway spawns `npx -y chrome-devtools-mcp@latest` for an existing-session browser profile.

## Best Possible Solution

Fix this in the browser plugin by making passive status/tab probes non-spawning, forwarding timeout and abort budgets through Chrome MCP reachability/listing/tab operations, closing failed sticky sessions immediately, and using a plugin-safe process-tree-aware stdio transport or local equivalent. Add regressions for hung default `npx`, hung `client.connect`, and hung `list_pages` through `GET /` and `GET /tabs`, including bounded response time and cleanup proof.
