## Summary

Keep open. Current main still has the Chrome MCP `npx` spawn path reachable from passive browser status/tab surfaces, route timeout budgets are not consistently forwarded into `listChromeMcpTabs`, and sticky-session timeout cleanup is weaker than ephemeral cleanup. The Windows Defender comments are useful adjacent context, but they do not resolve the original macOS/Node 25 hang report; open PR #73523 covers only the passive non-spawning slice and still needs maintainer coordination.

## What This Changes

The issue reports that the existing-session browser control server hangs when Gateway spawns `npx chrome-devtools-mcp@latest` through `StdioClientTransport`, causing browser HTTP requests to time out and orphaned processes to accumulate.

## Best Possible Solution

Fix this inside the browser plugin by making passive status/profile/tab probes cache-only and non-spawning for Chrome MCP profiles, forwarding timeout/abort budgets through reachability/listing operations, closing failed sticky sessions immediately, and adding regressions for hung `npx`, hung `client.connect`, and hung `list_pages` through `GET /` and `GET /tabs`.
