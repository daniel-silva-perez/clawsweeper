## Summary

Keep this issue open. Current main has Chrome MCP hardening and useful workarounds, but it still does not fully bound the reported existing-session browser-control hang path. The `/tabs` route still performs a 300 ms reachability probe, but the Chrome MCP branch drops that timeout, then awaits `listTabs()` with no route-level timeout. Sticky-session timeout cleanup also deletes the cache entry without immediately closing the session client in that catch path. The Windows Defender discussion is useful related context, but it does not resolve the original macOS/Node 25 `npx chrome-devtools-mcp` spawn/list path.

## Best Possible Solution

Keep this issue open and fix it in the browser plugin: propagate timeout/abort budgets through Chrome MCP reachability, `/tabs`, tab listing, and tab-management calls; close failed, timed-out, pending, and sticky sessions immediately; use a plugin-safe process-tree-aware stdio transport or browser-plugin-local equivalent for Chrome MCP; and add a regression that simulates a hung default `npx`/`client.connect`/`list_pages` path through `/tabs` with bounded response time and cleanup proof.
