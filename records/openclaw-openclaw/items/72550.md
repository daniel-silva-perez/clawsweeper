## Summary

Keep open: the report is recent and current main still has a plausible shared-core false-success path for channel plugins that return a failed raw send result instead of throwing. The exact WeChat `errcode -14` behavior lives in the external plugin, but that uncertainty argues against closing rather than for it.

## What This Changes

The issue asks cron announce delivery for `openclaw-weixin` to mark failed WeChat outbound sends as failed instead of recording `delivered: true`.

## Best Possible Solution

Keep this issue open and fix the shared outbound/plugin result contract so failed raw channel send results cannot be counted as successful cron delivery. WeChat token/session recovery should remain in the external plugin, and dynamic account resolution should stay with the separate accountId issue unless the same narrow patch needs a small generic seam.

## Reproduction Assessment

Yes for the core path: a focused test can register or mock a channel adapter whose text send returns `{ ok: false, messageId: '', error: 'session expired' }` and verify cron does not persist `delivered`. The live WeChat reproduction is plausible from the report but depends on external plugin behavior not present in this repo.

## Solution Assessment

Yes, the narrowest maintainable core fix is to make the outbound delivery boundary reject or filter failed raw send results before cron counts returned results. A WeChat-specific workaround in core would cross the documented plugin ownership boundary and should be avoided.
