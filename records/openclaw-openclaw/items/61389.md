## Summary

Keep this issue open. Current main already rejects stale thresholds below the effective health-check interval, so the fixed `.min(5)` part is not the right direction, but `gateway.channelMaxRestartsPerHour: 0` is still rejected by both the Zod schema and generated schema while the intended zero semantics remain ambiguous.

## What This Changes

The issue requests tightening gateway config validation so stale thresholds align with health-check cadence and `channelMaxRestartsPerHour` can be set to `0`.

## Best Possible Solution

Keep this issue as the canonical tracker, preserve the existing interval-relative stale-threshold validation, and update the schema, generated schema, docs/help text, and focused tests once the `channelMaxRestartsPerHour: 0` contract is explicitly chosen.

## Reproduction Assessment

Yes, for the remaining behavior: current schema source and generated schema both reject `gateway.channelMaxRestartsPerHour: 0` via minimum `1`. The stale-threshold half no longer reproduces as described because current validation/tests already reject values below the effective health-check interval.

## Solution Assessment

No, the literal suggested fix is not the best solution because a hard `.min(5)` would conflict with valid faster health-check intervals. The maintainable path is to keep the cross-field stale-threshold rule and separately decide/document whether a zero restart cap means zero allowed restarts or unlimited restarts.
