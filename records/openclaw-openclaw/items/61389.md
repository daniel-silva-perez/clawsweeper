## Summary

Keep open. Current main only partially addresses the report: stale thresholds below the effective health-check interval are rejected through a gateway-level Zod refinement and tests, but `gateway.channelMaxRestartsPerHour` still has `min(1)` in the schema and generated config metadata, so `0` is still invalid. The linked PR was closed unmerged, so it is not a canonical shipped fix.

## Best Possible Solution

Keep this issue open and resolve the contract explicitly. If maintainers want `gateway.channelMaxRestartsPerHour: 0` to mean unlimited, update the Zod schema/generated metadata, health-monitor cap logic, config tests, monitor tests, and docs. If `0` should mean disable restarts, document that contract and align runtime/tests; otherwise point users to `gateway.channelHealthCheckMinutes: 0` or per-channel `healthMonitor.enabled: false`.
