## Summary

Keep open. Current main still has the secure-context/device-identity rejection behavior and documentation, but it does not yet emit the requested gateway startup warning. An open linked PR, #73163, uses closing syntax for this issue, so the right cleanup path is to review that PR rather than close this issue early.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled but non-localhost HTTP access will be rejected because secure-context device identity is required.

## Best Possible Solution

Land a narrow gateway startup-log change that warns only when Control UI is enabled, TLS is off, at least one listen host is non-loopback, and device auth has not been explicitly disabled, with regression tests for both warning and quiet cases.

## Reproduction Assessment

Yes. A source-backed reproduction is a remote browser loading the Control UI over plain HTTP: the browser skips device identity because `crypto.subtle` is unavailable, and the gateway rejects the WebSocket connect with `CONTROL_UI_DEVICE_IDENTITY_REQUIRED`.

## Solution Assessment

Yes, the requested warning is the narrowest maintainable fix for the operator UX gap because the underlying rejection is intentional security behavior and already documented. The active PR appears to target the right startup-log boundary, but it needs normal review before merge.
