## Summary

Keep open. Current main still has the reported Control UI insecure-context rejection path and documents the workaround, but it does not emit the requested startup warning; the issue is also tied to an open, unmerged fixing PR, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue requests a gateway startup warning for Control UI deployments that will reject non-localhost plain-HTTP browser connections because device identity is required.

## Best Possible Solution

The desired end state is a narrow gateway startup-log diagnostic with regression coverage that explains the secure-context requirement while preserving the existing Control UI device-identity rejection policy.

## Reproduction Assessment

Yes. Current source gives a high-confidence reproduction path: open the Control UI over non-localhost plain HTTP so the browser skips device identity, then the gateway rejects the operator connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED while startup logging emits no targeted warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable solution for this issue because it adds operator guidance without weakening the current auth policy; the open PR direction appears aligned pending normal review.
