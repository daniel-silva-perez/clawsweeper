## Summary

Keep open. Current main still has the remote plain-HTTP Control UI rejection behavior and connection-time/docs guidance, but it does not emit the requested gateway startup warning. The linked implementation PR is still open and unmerged with closing syntax, so this issue should remain open until maintainers land, replace, or reject that PR.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled but remote plain-HTTP access will be rejected because device identity still requires a secure browser context.

## Best Possible Solution

Land a narrow gateway startup-log change, likely through #73163 after review, that warns only when resolved bind/TLS/Control UI/device-auth configuration makes remote plain-HTTP Control UI connections likely to fail. Preserve the current authentication policy and keep coverage in the startup-log tests.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main’s startup log has no Control UI secure-context warning, while the connection policy and tests still reject the described remote device-less Control UI path.

## Solution Assessment
