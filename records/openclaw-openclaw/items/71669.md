## Summary

Keep open. Current main still rejects remote plain-HTTP Control UI connections without device identity and documents the workaround, but it does not emit the requested operator-visible gateway startup warning; #73163 is an open, unmerged implementation PR with closing syntax, so this issue should remain open until that PR is reviewed, landed, replaced, or rejected.

## What This Changes

Request a gateway startup warning for Control UI deployments that will reject remote plain-HTTP browser connections because device identity is required and no secure context is available.

## Best Possible Solution

Land a narrow gateway startup-log change, likely through #73163 after normal review, that warns only when resolved bind/TLS/Control UI/device-auth configuration makes remote plain-HTTP Control UI connections likely to fail. Preserve the existing authentication enforcement and add focused startup-log coverage for warning and non-warning cases.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main's startup log has no Control UI secure-context warning, while the connection policy and tests still reject the described remote device-less Control UI path.

## Solution Assessment

Yes. A startup warning after runtime config and bind/TLS resolution is the narrowest maintainable solution for this issue; changing the secure-context/device-identity policy belongs to broader related issues, not this logging request.
