## Summary

Keep open. Current main still enforces and documents the Control UI secure-context/device-identity rejection path, but startup logging only emits generic startup and dangerous-config messages, not the requested warning. The issue is also paired with open, unmerged PR #73163 using closing syntax, so it should remain open until that PR lands, is replaced, or is closed.

## What This Changes

Add a gateway startup warning when Control UI is enabled in a configuration where remote plain-HTTP browser connections will be rejected for missing device identity.

## Best Possible Solution

Land PR #73163 or an equivalent narrow gateway startup-log change with focused regression tests, while preserving the existing Control UI device-identity rejection policy. Broader changes to allow device identity over non-secure HTTP should stay with #53274 or a dedicated security/product thread.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: open the Control UI from a non-localhost plain-HTTP browser so crypto.subtle is unavailable, the UI sends no device identity, and the gateway rejects the connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED while startup logging has no targeted warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for this issue because it improves operator guidance without weakening the current auth policy; the existing open PR appears aligned pending normal review.
