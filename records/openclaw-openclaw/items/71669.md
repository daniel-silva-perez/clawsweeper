## Summary

Keep open. Current main still rejects the reported Control UI insecure-context path and documents the workaround, but it does not emit the requested startup warning; the issue is also paired with open, unmerged PR #73163, so it should remain open until that PR lands, is replaced, or is closed.

## What This Changes

Request a gateway startup warning when Control UI is enabled and non-loopback plain HTTP access will be rejected because device identity is required.

## Best Possible Solution

Review and land PR #73163, or an equivalent narrow gateway startup-log change with regression tests, while preserving the existing device-identity rejection behavior and leaving broader remote HTTP policy changes to the related product/security threads.

## Reproduction Assessment

Yes. Current source gives a high-confidence reproduction path: open the Control UI over non-localhost plain HTTP so the browser skips device identity, then the gateway rejects the operator connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED while startup logging emits no targeted warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable fix for this issue because it adds operator guidance without weakening the current Control UI auth policy; the existing open PR appears aligned with that direction pending normal review.
