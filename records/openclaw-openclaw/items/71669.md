## What This Changes

The issue requests a gateway startup warning when Control UI is enabled and non-localhost plain HTTP clients will be rejected because device identity is unavailable outside a secure context.

## Best Possible Solution

Land a narrow gateway startup warning in the startup logging path, with regression tests for the warn and quiet cases, while leaving broader Control UI device-identity policy changes to the related product/security issues.

## Reproduction Assessment

Yes. Current source and tests show the browser skips device identity in non-secure HTTP contexts and the gateway rejects a Control UI operator connection without device identity using the secure-context error path.

## Solution Assessment

Yes, the linked PR direction appears to be the narrowest maintainable fix for this issue: add runtime guidance without weakening the existing auth policy. The issue should stay open until that PR is reviewed and merged or replaced.
