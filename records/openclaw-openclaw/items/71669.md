## Summary

Keep open. Current main still documents and enforces the Control UI secure-context/device-identity rejection path, but it does not emit the requested startup warning; the issue is also paired with open, unmerged PR #73163, so it should remain open until that PR lands, is replaced, or is closed.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled but non-localhost HTTP access will be rejected for lacking secure-context device identity.

## Best Possible Solution

Review and land PR #73163, or an equivalent narrow gateway startup-log change with regression tests, while preserving the existing Control UI device-identity rejection policy and leaving broader remote HTTP policy changes to their own product/security threads.

## Reproduction Assessment

Yes. Source and tests give a high-confidence path: load the Control UI over non-localhost plain HTTP so the browser has no crypto.subtle/device identity, then the gateway rejects the connection with CONTROL_UI_DEVICE_IDENTITY_REQUIRED while startup logging emits no targeted warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable solution for this issue because it adds operator guidance without weakening the current Control UI auth policy; the open PR appears aligned with that direction pending normal review.
