## Summary

Keep open. Current main still deduplicates Feishu inbound receive, debounce, and finalization by account plus `message_id` before media identity is incorporated, so the reported same-id/different-audio case can still drop the later audio. The issue is also paired with open PR #75078 using closing syntax, so maintainers should review and land or close that PR before closing this issue.

## What This Changes

Feishu should process distinct inbound audio attachments independently even when Feishu reuses the same `message_id` for different `file_key` values.

## Best Possible Solution

Review and land #75078, or an equivalent narrow Feishu-owned patch, so inbound audio dedupe computes a stable media-aware key before the early receive claim and final persistent record while preserving existing behavior for text messages and true retries with the same media key.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive payloads for the same account with identical `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide at the current early processing claim before media parsing. Live upstream ID reuse still needs reporter logs or Feishu-side confirmation.

## Solution Assessment

Yes. A Feishu plugin fix that resolves a media-aware dedupe key before receive/finalize is the narrowest maintainable solution; changing shared core dedupe or adding broad content hashing would be more invasive for this media-key-specific bug.
