## Summary

Keep open. Current main still deduplicates Feishu inbound audio by account plus bare `message_id` before media identity is incorporated, so the same-id/different-audio case can still drop the later attachment. The issue is already paired with open PR #75078, so maintainers should review, land, or close that implementation candidate rather than close this issue early.

## What This Changes

Fix Feishu inbound dedupe so distinct audio attachments are keyed by media identity instead of account plus `message_id` alone.

## Best Possible Solution

Land #75078 or an equivalent narrow Feishu-owned patch that derives one stable dedupe key for early receive claims, debounce suppression, persistent finalize, and suppressed-record writes, using media identity for audio while preserving existing behavior for text and true identical retries.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive payloads for the same account with the same `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide at the early claim before media parsing. The live Feishu ID-reuse part still depends on reporter logs or live confirmation.

## Solution Assessment

Yes. A Feishu-local media-aware dedupe key is the narrow maintainable fix; changing shared core dedupe or adding broad content-hash fallback would be larger than the reported same-id/different-audio failure requires.
