## Summary

Keep open. Current main still deduplicates Feishu inbound receive events by account plus `message_id` before media content is parsed, so the reported same-id/different-audio case can drop the later audio event. This is a narrow Feishu plugin bug candidate rather than a cleanup close.

## What This Changes

The issue asks for Feishu inbound audio dedupe to treat distinct audio attachments with reused `message_id` values as separate messages.

## Best Possible Solution

Keep the repair Feishu-owned: compute a normalized media-aware inbound dedupe key before the early receive claim and persistent finalize path, using account, `message_id`, message type, and `file_key` or `image_key` for media events, while keeping text and true duplicate retries deduped.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive payloads for the same account with identical `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values will hit the early message-id processing claim before media resolution. The live Feishu behavior is plausible from the report but lacks logs.

## Solution Assessment

Yes. The narrowest maintainable solution is a Feishu plugin dedupe-key fix with regression tests; changing shared persistent dedupe semantics or adding broad content-hash dedupe would be riskier and unnecessary for this specific bug.

Fix the Feishu inbound audio/media dedupe bug reported in https://github.com/openclaw/openclaw/issues/75057. Scope the change to the Feishu plugin. Current main claims and persists inbound messages by `message_id` only before parsing media, so two audio events with the same `message_id` but different `content.file_key` values can drop the second attachment. Implement a stable normalized dedupe key available before the early receive gate, likely using account namespace, `message_id`, message type, and normalized `file_key`/`image_key` for media messages.
