## Summary

Keep open. Current main still keys Feishu inbound receive claims, debounce suppression, and persistent finalize on the bare `message_id`, so same-id audio payloads with different `file_key` values can still collide. Open PR #75078 is already the implementation candidate and should be reviewed or landed before this issue is closed.

## What This Changes

Feishu audio messages with a reused `message_id` but different audio attachment identity should be processed independently instead of being dropped as duplicates.

## Best Possible Solution

Land #75078 or an equivalent narrow Feishu-owned patch that derives one stable media-aware dedupe key and uses it consistently for early receive claims, debounce dedupe, persistent finalize, and suppressed-record writes, with regression coverage for different audio `file_key` values and unchanged true-retry behavior.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive events for the same account with the same `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide before media parsing. The live upstream Feishu ID-reuse condition remains based on reporter and PR context rather than an independently run live Feishu test.

## Solution Assessment

Yes. A Feishu-local media-aware dedupe key is the narrow maintainable fix; changing shared core dedupe or adding broad content-hash fallback would be larger than this same-id/different-audio failure requires.
