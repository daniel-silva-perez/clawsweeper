## Summary

Keep open. Current main still deduplicates Feishu inbound audio by account plus raw `message_id` before media identity is part of the key, and open PR #75078 uses closing syntax for the same bug, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Feishu audio messages sent in quick succession can reuse `message_id` with different audio content, causing OpenClaw to drop the later audio as a duplicate before it reaches the agent.

## Best Possible Solution

Review and land PR #75078, or an equivalent narrow Feishu-owned patch, so the early receive claim, debounce filtering, suppressed-message recording, and persistent finalize path all use a stable audio media-aware dedupe key while preserving current behavior for text and true identical retries.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is two Feishu receive events for the same account with identical `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values; current main claims the first by account/message ID and drops the second before reading `file_key`.

## Solution Assessment

Yes. The narrow maintainable solution is a Feishu plugin dedupe-key fix that incorporates audio media identity at every current dedupe point; a broad core dedupe rewrite would be more invasive for this channel-specific report.
