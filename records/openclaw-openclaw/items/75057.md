## Summary

Keep open. Current main still keys Feishu inbound early and persistent dedupe on account plus message_id, so the same-id/different-audio case can still drop the later audio. The issue is also paired with open PR #75078 using closing syntax, so maintainers should review that PR rather than close this issue early.

## What This Changes

The issue requests that Feishu inbound audio dedupe process distinct audio attachments independently when Feishu reuses the same message_id.

## Best Possible Solution

Review and land #75078, or an equivalent narrow Feishu-owned patch, so inbound media dedupe computes a stable key before the early receive claim and final persistent record using account, message_id, message type, and file_key or image_key for media while retaining message-id-only behavior for text and identical retries.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive payloads for the same account with identical message.message_id, message_type "audio", and different JSON file_key values collide at the current early claim before media resolution. The live Feishu reuse behavior still needs reporter logs or live confirmation.

## Solution Assessment

Yes, the requested direction is best solved as a Feishu plugin dedupe-key fix. A shared core dedupe change or broad content-hash fallback would be more invasive for this same-id/different-media bug.
