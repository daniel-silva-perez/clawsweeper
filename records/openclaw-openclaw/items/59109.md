## Summary

Keep open. Current main covers part of the request through lower-level primitives, but it does not expose the requested open-agent-style SDK surface for `forkSession(id)` or `continue: true`; the supportive issue comment is useful product signal rather than evidence of an implemented behavior.

## What This Changes

The issue requests OpenClaw session APIs for forking an existing session, resuming by session ID, and continuing the most recent session without an ID.

## Best Possible Solution

Define a first-class App SDK/session contract that maps cleanly onto OpenClaw’s existing `sessionKey` and `sessionId` model: keep exact-session resume, add an explicit fork API that resolves IDs safely before using parent-session lineage, and add a most-recent-session helper only after the selection and authorization semantics are settled.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The source check is high-confidence: current main exposes partial primitives but no direct `forkSession(id)` or `continue: true` SDK surface.

## Solution Assessment

No, the current lower-level workaround is not the best final solution for this request. A small public API design pass is needed so SDK users do not have to manually stitch together `sessions.list`/`sessions.resolve`/`sessions.create({ parentSessionKey })` with ambiguous most-recent-session behavior.
