## Summary

Keep open. Current main has useful adjacent session capabilities, including explicit CLI resume by session id, Gateway session resolution by sessionId, ACP-only resumeSessionId, native sub-agent context="fork", and checkpoint branching. Those do not fully satisfy the requested open-agent-sdk-style feature set because there is still no general public forkSession(id) surface and no continue:true / continue-most-recent selector. The supplied related items cover restart resume or narrower fork bugs, not a canonical replacement for this broader feature request.

## Best Possible Solution

Keep this open for maintainer product/API design. A complete fix should add a first-class session-control surface, likely in the Gateway protocol plus CLI/SDK docs/tests, that can fork an arbitrary existing OpenClaw session, resume by key or sessionId, and deterministically continue the latest eligible session without the caller already knowing its id. Existing primitives such as `sessions.resolve`, `sessions.create`/`sessions.send`, `SessionManager` branching, and ordered `sessions.list` can be reused, but the behavior should be explicit and covered by protocol, CLI, and regression tests.
