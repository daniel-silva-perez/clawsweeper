## Summary

Keep open. Current main still resolves the A2A `sessions_send` announce target from the target session, so the PR addresses a real remaining behavior gap, but the branch is stale and would need rebase work to preserve current-main `threadId` forwarding and per-session nested lane behavior before merge.

## What This Changes

The PR changes A2A `sessions_send` announce delivery to resolve from the requester session, keeps separate target-side channel metadata for prompts/provenance, and adds requester-vs-target regression coverage.

## Best Possible Solution

Rebase and port the PR onto current main, keeping requester-side announce delivery when `requesterSessionKey` is available, the target-side metadata fallback for prompts/provenance, the target fallback for requester-less calls, current `threadId` forwarding, and `resolveNestedAgentLaneForSession(...)` for all A2A agent steps. Keep the requester-vs-target regression test and add/adjust assertions so the stale current-main protections cannot be dropped.

## Reproduction Assessment

Yes. Current main has a focused unit-test path: the existing A2A ping-pong announce test still expects target-side delivery, and the implementation resolves the final announce target from `targetSessionKey`; the PR adds a mocked requester-vs-target regression that would fail on current main.

## Solution Assessment

No, not as-is. The requester-side routing direction is the narrow maintainable fix, but the current branch is not the best merge vehicle until it is rebased and reconciled with newer current-main lane and thread delivery changes.
