## Summary

Keep open. Current main still routes an explicitly disabled `tools.web.fetch` object into web provider discovery, and this PR is a relevant active fix rather than obsolete work. I found no blocking correctness issue in the updated diff; the remaining work is maintainer review and choosing the canonical fix alongside #74924.

## What This Changes

The PR adds a credential-aware short-circuit in `resolveRuntimeWebTools` for explicitly disabled web search/fetch surfaces without plugin-scoped config and extends colocated tests for credential-free, non-credential nested, and credential-bearing disabled configs.

## Best Possible Solution

Land one canonical patch that skips provider discovery for explicitly disabled, credential-free web search/fetch surfaces while preserving inactive SecretRef warnings for credential-bearing disabled configs; track any remaining enabled-tool manifest discovery stall separately if it still reproduces.

## Reproduction Assessment

Yes. #74896 provides a concrete Linux/Docker config and startup trace, and current main shows the disabled fetch object still bypasses the secrets fast path and enters fetch provider resolution. I did not run the live Docker reproduction because this review is read-only.

## Solution Assessment

Yes for the reported escape hatch. The PR's resolver-level credential-aware skip is narrow and preserves disabled-surface SecretRef warning behavior; a broader `runtime.ts` fast-path can be considered, but should not drop plugin-scoped or credential-bearing inactive warnings.
