## Summary

Keep open. The PR is aimed at a real current-main startup regression, but the proposed fast path is too narrow: valid disabled web tool configs with nested non-credential options would still enter provider discovery and can hit the same pre-bind stall this patch is trying to avoid.

## What This Changes

The PR adds a disabled-surface fast path in `resolveRuntimeWebTools` plus runtime web-tool tests for disabled web search/fetch discovery behavior.

## Best Possible Solution

Revise the fast path so disabled web-search and web-fetch surfaces skip provider discovery unless there is an actual credential-bearing provider config or SecretRef warning to resolve. Add regression coverage for disabled local-only nested options such as `tools.web.fetch.ssrfPolicy` and `tools.web.search.openaiCodex`, while preserving inactive SecretRef warnings for real provider credential blocks.

## Reproduction Assessment

Yes. The linked bug has concrete config, version, trace boundary, and workaround, and current main shows a present `tools.web.fetch` object reaches provider discovery before bind. For this PR’s regression, code inspection shows disabled configs with nested non-credential objects would fail the new skip guard.

## Solution Assessment

No. The direction is right, but `hasNestedObjectFields` is too broad for the config contract; a safer solution should check known credential/provider fields or actual SecretRefs instead of treating every nested option object as requiring discovery.

## Review Findings

Check for actual credential-bearing provider fields or SecretRefs instead.
