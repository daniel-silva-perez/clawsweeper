## Summary

Keep open. Current main still takes the web-fetch provider discovery path when `tools.web.fetch` is present, and this PR is an active contributor fix for a concrete regression linked from #74896. I did not find a new blocking correctness issue in the final PR patch; the remaining work is maintainer review, exact-head CI, and choosing the canonical fix alongside open PR #74924.

## What This Changes

This PR adds a credential-aware disabled web search/fetch fast path in `resolveRuntimeWebTools` and colocated regression tests for disabled credential-free, non-credential nested, and credential-bearing web tool configs.

## Best Possible Solution

Land one canonical fix that lets explicitly disabled web fetch/search surfaces avoid unnecessary provider discovery while preserving inactive SecretRef warning behavior for credential-bearing config; track any remaining enabled-tool manifest discovery hang separately if it still reproduces.

## Reproduction Assessment

Yes. #74896 provides a concrete Linux/Docker startup trace and minimal config, and current main still shows the code path that enters web-fetch discovery when the fetch object is present. I did not run the live Docker reproduction because this review is read-only.

## Solution Assessment

Yes for the reported escape hatch. The PR's resolver-level short-circuit is narrow and preserves credential-bearing disabled-surface handling; the main maintainer question is whether to add or prefer a broader `runtime.ts` fast-path gate without losing inactive warning semantics.
