## Summary

Keep open. Current main partially counts runtime-observed compactions, but it still does not solve the reported third-party context-engine case where an `ownsCompaction` engine performs proactive/background compaction from `afterTurn` or maintenance without returning a compaction delta to core. `/status` and `session_status` still display only the persisted `SessionEntry.compactionCount`.

## Best Possible Solution

Keep this open and add an additive plugin-facing accounting path: either let context engines return/report successful compaction deltas from `afterTurn`/maintenance, or add an optional per-session stats API. Then persist those deltas through the existing `incrementRunCompactionCount`/session store path and add regression coverage proving an `ownsCompaction` engine's proactive compaction is reflected in both `/status` and `session_status`.

## Risks / Open Questions

- Counting every `afterTurn` call would overcount ordinary ingest/maintenance. The fix needs an explicit successful-compaction signal, per-session stats delta, or hook payload from the engine.
- Any stats API should be per-session and idempotent so retries, loop transforms, and background maintenance do not double-count compactions.
