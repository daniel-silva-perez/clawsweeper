## Summary

Keep this PR open. Current main still has direct OpenAI Responses, Azure OpenAI Responses, and OpenAI chat-completions SDK calls with no shipped `LLM_MAX_CONCURRENT` or equivalent provider request limiter. The PR remains relevant, but the submitted diff is incomplete and should not merge as-is.

## Best Possible Solution

Keep this PR open for revision or maintainer replacement. A mergeable implementation should add a tested provider request limiter at the OpenAI/Azure transport boundary, cover OpenAI Responses, Azure OpenAI Responses, and OpenAI-compatible chat completions, hold slots for the full streamed response lifetime, handle queued and active aborts, parse invalid config safely, and document the chosen configuration surface. If a stronger implementation lands elsewhere, close this PR as superseded by that merged work.

## Risks / Open Questions

- The PR should not merge as-is because it misses the chat-completions path promised by the PR body and has formatter-blocking indentation in the submitted patch.
- The proposed limiter wraps SDK stream creation, so a slot can be released before a long streamed response is consumed.
- Invalid `LLM_MAX_CONCURRENT` values can turn the cap into `NaN`, silently disabling throttling instead of falling back safely.
- Queued work does not reject promptly when the caller aborts while waiting for a limiter slot.
- The configuration surface is env-only in the PR and lacks docs, schema/help text, and regression tests.
