## Summary

Keep open: current main still has replay-time pruning, Gateway attachment offload, and display/update redaction, but the central session persistence path does not yet replace inbound/user image `data` blocks with durable media references before JSONL append. The linked open Gateway PR is adjacent display work, while the closed reference PRs were not merged, so the storage-level bug remains actionable.

## What This Changes

Request to stop storing Discord and other inbound image payloads as base64 in session transcripts by persisting compact media references and hydrating bounded image data only when building model context.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound/user image content: persist compact media reference and metadata instead of raw `image.data`, hydrate only bounded recent references during model-context construction, keep legacy inline-base64 transcripts readable, and reuse the existing media store/offload contract where possible.

## Reproduction Assessment

Yes. The issue discussion includes concrete Discord DM forensics, and current main has a focused code-level path: inbound media becomes base64 image content for `activeSession.prompt`, while the central append guard persists user messages without an image-data rewrite.

## Solution Assessment

Yes. A central persistence-layer claim-check is the narrowest maintainable fix; replay pruning, display redaction, and Gateway-only attachment offload reduce symptoms but do not remove duplicate inline base64 from stored session transcripts.
