## Summary

Keep open: current main still has mitigations around replay pruning, image sizing, Gateway attachment offload, and display/update redaction, but it does not implement the requested storage-level fix for user image blocks reaching the session transcript writer.

## What This Changes

The issue requests replacing inline base64 image data in session transcripts with compact media references and hydrating bounded images from disk when constructing model context.

## Best Possible Solution

Add a shared, backward-compatible transcript claim-check for inbound/user image content: persist compact media reference and metadata instead of raw image `data`, hydrate only bounded recent references when building model context, keep legacy inline-base64 transcripts readable, and align Gateway/WebChat history projection with the compact representation.

## Reproduction Assessment

Yes. The issue discussion includes concrete Discord DM forensics, and current main has a focused code-level reproduction path: a user message containing an image block that reaches the guarded `appendMessage` path is persisted after transforms that do not strip image data.

## Solution Assessment

Yes, a central persistence-layer claim-check is the narrowest maintainable fix. Gateway display redaction or replay-only pruning reduces symptoms but does not remove the duplicate inline base64 from the stored session transcript.
