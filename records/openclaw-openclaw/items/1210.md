## Summary

Keep open: current main mitigates image replay and display bloat, and it has a Gateway media claim-check path, but it still does not claim-check inbound image content before the session JSONL persistence path. Discord media paths are saved and surfaced as prompt context, then the embedded runner hydrates them back into image blocks for the upstream agent session, whose event persistence still appends those image-bearing user messages.

## What This Changes

The issue requests storing durable media references for inbound Discord images in session transcripts instead of persisting raw base64 image blocks.

## Best Possible Solution

Add a backward-compatible session transcript claim-check for inbound image content: persist compact media reference and metadata instead of raw image.data, hydrate only bounded recent references during model-context construction, keep legacy inline-base64 transcripts readable, and align the visible history projection with the adjacent chat.history work.
