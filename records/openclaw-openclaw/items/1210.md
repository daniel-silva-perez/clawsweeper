## Summary

Keep open: current main has replay pruning, Gateway/WebChat media-reference paths, and display redaction, but the storage bug remains in the central session append path because user image blocks are still not claim-checked before transcript persistence.

## What This Changes

The issue asks OpenClaw to store inbound Discord image references and metadata in session transcripts instead of embedding base64 image data, then hydrate bounded image data when building model context.

## Best Possible Solution

Add a shared, backward-compatible transcript claim-check for inbound image content: persist compact media reference and metadata instead of raw `image.data`, hydrate only bounded recent references when constructing model context, keep legacy inline-base64 transcripts readable, and align Gateway history/export projection with the compact representation.
