## Summary

Keep open. Current main has replay-time pruning, media-note prompting, and Gateway display/offload mitigations, but it still lacks the requested storage-level claim-check for session JSONL: user image blocks can still reach Pi as `{ type: "image", data }` and the guarded append path persists non-tool-result messages without replacing image data with durable media references.

## What This Changes

Replace inline base64 Discord image blocks in session transcripts with compact media reference metadata and hydrate bounded image data only when building model context.

## Best Possible Solution

Add a shared, backward-compatible transcript claim-check for inbound image content: persist compact media reference and metadata instead of raw `image.data`, hydrate only bounded recent references during context construction, keep legacy inline-base64 transcripts readable, and align Gateway history/export display with the compact representation.
