## Summary

Keep open: current main has replay pruning, prompt/display mitigations, and a chat.send-specific media-path rewrite, but the Discord/auto-reply path still hydrates saved media into `{ type: "image", data }` blocks and passes them to the embedded Pi session. The shared session append guard still persists user messages without replacing image blocks with durable media references.

## What This Changes

Store inbound Discord image references and metadata in session transcripts instead of raw base64 image payloads, then hydrate bounded recent images from disk when building model context.

## Best Possible Solution

Add a shared, backward-compatible transcript claim-check for inbound image content: persist compact media reference metadata instead of `image.data`, hydrate only bounded recent references during context construction, keep old inline-base64 transcripts readable, and align Gateway history/export display with the same compact representation.
