## Summary

Keep open: current main still has replay-time pruning, Gateway/WebChat attachment offload, and display cleanup, but it does not add the requested storage-layer claim-check before session JSONL persistence. A user image block can still be hydrated as `{ type: "image", data, mimeType }`, passed into the upstream session prompt, and persisted through the guarded append path without image-data rewriting.

## What This Changes

Store Discord and other inbound image references in session transcripts instead of inline base64 image data, then hydrate bounded image data from disk only when building model context.

## Best Possible Solution

Add a central, backward-compatible transcript claim-check for inbound/user image content: persist a compact media reference plus metadata instead of raw `image.data`, hydrate only bounded recent references during model-context construction, and keep legacy inline-base64 transcript replay readable.

## Reproduction Assessment

Yes. The discussion provides concrete Discord DM forensics, and the current-main code path still hydrates prompt images as inline base64 before upstream prompt/session persistence.

## Solution Assessment

Yes. A central persistence-layer claim-check is the narrowest maintainable fix because replay pruning and UI projection reduce symptoms but do not remove duplicate inline base64 from stored transcripts.
