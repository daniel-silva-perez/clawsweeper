## Summary

Keep open: current main still has replay-time pruning, Discord media download/storage, and Gateway display projection work, but it does not add the requested storage-layer claim-check before session JSONL persistence. The issue has concrete live forensics, and the current append path can still write user image blocks produced by prompt image hydration.

## What This Changes

Request to persist Discord/channel image content in session transcripts as compact media references instead of inline base64, then hydrate bounded image data when building model context.

## Best Possible Solution

Add a central, backward-compatible transcript claim-check for inbound/user image content: persist media reference and metadata instead of raw `image.data`, hydrate only bounded recent references during context construction, and keep legacy inline-base64 transcript replay supported.

## Reproduction Assessment

Yes. The discussion includes concrete Discord DM forensics from v2026.3.24, and current main still has a source-backed path where hydrated image blocks can reach `AgentSession.prompt` and then `SessionManager.appendMessage`.

## Solution Assessment

Yes. A central persistence-layer claim-check is the narrowest maintainable fix; replay pruning and UI projection reduce symptoms but do not remove duplicate inline base64 from stored transcripts.
