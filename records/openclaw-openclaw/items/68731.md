## Summary

Keep open. Current main still limits BlueBubbles Contacts enrichment to group participants, skips email handles in the resolver, and has no 1:1 inbound sender-name resolver or renamed config flag; the issue discussion also has maintainer signal asking for a PR rather than a cleanup close.

## What This Changes

Broaden the bundled BlueBubbles local Contacts enrichment so allowed 1:1 inbound DMs, including phone and email handles with unusable sender-name values, can resolve human names through the same resolver/cache currently used for group participants.

## Best Possible Solution

Implement the broadening inside the BlueBubbles bundled plugin: keep lookup after authorization/gating, reuse the existing Contacts/cache machinery, add phone and email handle support, preserve legacy config compatibility, and cover the new 1:1 path with focused tests and docs.

## Reproduction Assessment

Yes. The current-main reproduction is source-level: an allowed 1:1 inbound message reaches `processMessageAfterDedupe`, but the Contacts enrichment call is only under the group branch, and the resolver currently ignores email ids.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: extend the existing BlueBubbles-owned resolver and call site rather than moving channel-specific Contacts behavior into core or duplicating it in a third-party plugin.
