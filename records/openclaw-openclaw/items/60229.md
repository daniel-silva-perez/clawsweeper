## Summary

Keep open. Current main still has the Telegram serialization behavior the PR is trying to address, but the proposed patch is not merge-ready because it can acknowledge Telegram updates before the buffered busy flush runs, can race the existing long-text fragment combiner, and leaves sequential-key tests inconsistent with the new key format.

## What This Changes

The PR adds a `coalesceWhileBusy` option to the shared inbound debouncer, enables it for Telegram, and appends `message_id` to ordinary Telegram sequentialization keys so same-chat follow-up messages can reach collect-mode queueing while a run is active.

## Best Possible Solution

Keep the PR open and revise it into an awaitable busy-coalescing path that preserves Telegram update durability and long-text fragment assembly. The final fix should add focused debouncer tests, Telegram sequential-key tests, and an integration-style Telegram handler test showing collect-mode batching while an active run is still in progress.

## Reproduction Assessment

Yes. A high-confidence reproduction is a Telegram conversation with queue mode `collect`, one long-running active agent turn, and rapid same-chat follow-up messages; current main's chat/topic sequential key plus same-key debouncer chain means those follow-ups reach queue policy only after the active run finishes.

## Solution Assessment

No. The product direction is plausible, but this patch is not the best implementation as-is because it returns from `enqueue` before the coalesced busy flush completes and removes chat/topic serialization without protecting the text-fragment buffer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await the coalesced busy flush before returning:** `src/auto-reply/inbound-debounce.ts:207-213`
  - body: This branch schedules the busy-buffer flush with `void enqueueKeyTask(...)` and then returns, so callers awaiting `enqueue` can treat the Telegram update as complete before `runFlush(pending)` has executed. Telegram persists accepted update offsets around middleware completion, so a restart in that window can skip follow-up messages that were never flushed.
  - confidence: 0.91
- **[P2] Preserve fragment ordering before using per-message keys:** `extensions/telegram/src/sequential-key.ts:63-65`
  - body: Adding `:msg:${messageId}` lets adjacent same-chat messages enter the handler concurrently, but the long-text fragment combiner mutates a shared per-chat buffer and expects the first near-limit fragment to create the buffer before the next fragment arrives. Split Telegram paste fragments can therefore be emitted separately or out of order.
  - confidence: 0.86
- **[P2] Update sequential-key expectations with the new suffix:** `extensions/telegram/src/sequential-key.ts:63-65`
  - body: The new return value appends `:msg:{messageId}` for regular messages, but `sequential-key.test.ts` still expects base chat/topic keys for the default mock message ID. Those existing cases will fail until the expected strings and missing-message-id fallback coverage are updated.
  - confidence: 0.88
