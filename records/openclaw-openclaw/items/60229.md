## Summary

Keep this PR open. Current main does not implement the intended Telegram busy-run coalescing: Telegram still uses a chat/topic-level grammY sequential key, and the shared inbound debouncer has no coalesce-while-busy path. The PR is focused and relevant, but it needs changes before it can be merged because the proposed per-message sequential key conflicts with existing sequential-key tests and risks racing Telegram text-fragment assembly.

## Best Possible Solution

Keep the PR open and ask for a revised patch: add covered coalesce-while-busy behavior in the inbound debouncer, enable it for Telegram, update sequential-key tests if the key changes, and preserve/update regression coverage for Telegram text fragments and update-offset safety. If per-message grammY keys are still needed, protect the fragment assembler with a narrower lane before letting ordinary messages bypass chat/topic serialization.

## Risks / Open Questions

- Security review pass: the provided PR diff only touches TypeScript runtime files, not workflows, lockfiles, dependency sources, package publishing metadata, secrets handling, or downloaded/executed artifacts, so I do not see a supply-chain regression in the diff as provided.
- Functional risk: the proposed coalesceWhileBusy branch schedules a follow-up flush asynchronously; the existing Telegram update tracker persists accepted update IDs early, so the final implementation should prove queued buffered messages cannot be acknowledged or skipped before processing.
- Functional risk: appending message_id to the grammY sequential key can race shared Telegram text-fragment state unless that path is kept serialized or moved behind its own per-chat/topic/sender lane.
