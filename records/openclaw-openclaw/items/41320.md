## Summary

Keep open. Current main still has no Tirith or `tools.exec.commandSecurity` surface, and this PR is not mergeable as-is because the gateway scan can launch the configurable scanner before effective `security=deny` is enforced. This is a core exec/security-boundary change, so it needs author changes plus maintainer/security review rather than cleanup closure.

## What This Changes

The PR adds optional default-on Tirith pre-exec command scanning, wires `tools.exec.commandSecurity` through gateway, sandbox, and node exec paths, and adds command-security tests.

## Best Possible Solution

A mergeable version should enforce effective host deny before any scanner process can start, document the operator-facing config and fail-open model, add gateway/sandbox/node regression coverage, and settle the core-versus-plugin trust-model decision.

## Reproduction Assessment

Yes for the blocking PR regression: configure effective gateway exec policy to `security=deny` with command security enabled, then invoke a gateway exec. The PR diff calls `checkCommandSecurity` before the current main deny resolver in `processGatewayAllowlist`, so the scanner can start before denial; tests were not run because this review is read-only.

## Solution Assessment

No. The security-in-depth idea is plausible and earlier node-ordering/output-bounding feedback was addressed, but this placement is not the safest implementation because it runs a configurable scanner before the effective deny guard.
