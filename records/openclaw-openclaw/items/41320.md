## Summary

Keep open. Current main still lacks Tirith or `tools.exec.commandSecurity`, and the PR addresses a real exec security boundary, but it should not merge as-is because the added gateway scanner can spawn the configured scanner before existing `security=deny` policy rejects the exec.

## What This Changes

The PR adds default-on Tirith pre-exec command scanning, flows `tools.exec.commandSecurity` through config and agent tool creation, and adds command-security unit plus exec approval integration tests.

## Best Possible Solution

Keep this PR open for maintainer security review. A mergeable version should preserve existing deny semantics before any scanner child process runs, document the new config and operational model, add regression coverage for denied policies, and either justify core integration or redirect the capability to a plugin built on `before_tool_call`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor deny policy before launching the scanner:** `src/agents/bash-tools.exec.ts:405-422`
  - body: With `tools.exec.security` or the host approval file resolving to `deny`, current gateway exec rejects in `resolveExecHostApprovalContext`. This new block runs `checkCommandSecurity` before the gateway allowlist path reaches that denial, so a denied exec can still spawn the configured/PATH `tirith` binary. Move the scan after effective deny resolution or skip it when the effective policy is deny, and add a regression for gateway `security=deny`.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete security-boundary regression because a denied gateway exec can still launch the configured scanner binary before the existing deny policy is enforced.

Concerns:

- **[high] Scanner runs before gateway deny policy:** `src/agents/bash-tools.exec.ts:405`
  - body: The PR invokes `checkCommandSecurity` in the gateway path before `processGatewayAllowlist` calls the shared approval resolver that rejects `hostSecurity === "deny"`. Because `tirithPath` is a configurable/PATH-resolved executable, this can execute a local binary even when exec policy says the command should be denied.
