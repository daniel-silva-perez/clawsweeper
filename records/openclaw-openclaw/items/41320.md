## Summary

Keep open. Current main does not implement Tirith command-security scanning, and the PR still needs maintainer/security review because the diff runs the configurable scanner before the gateway effective-deny guard, so a `security=deny` gateway exec can still spawn Tirith. Earlier Greptile feedback appears addressed by later PR commits, but this blocking security-boundary issue remains.

## What This Changes

The PR adds optional default-on Tirith pre-exec command scanning, wires `tools.exec.commandSecurity` through gateway, sandbox, and node exec paths, and adds command-security and approval integration tests.

## Best Possible Solution

A mergeable version should enforce effective host deny before any scanner process can start, document the operator-facing config and fail-open model, add gateway/sandbox/node regression coverage, and settle the core-versus-plugin trust-model decision with secops review.

## Reproduction Assessment

Yes for the blocking PR regression: configure effective gateway exec policy to `security=deny` with command security enabled, then invoke a gateway exec. The PR diff calls `checkCommandSecurity` before `processGatewayAllowlist` reaches the current-main deny resolver; tests were not run because this review is read-only.

## Solution Assessment

No. The security-in-depth idea is plausible, and earlier node-ordering/output-bounding feedback appears addressed, but this placement is not the safest implementation because it starts a configurable scanner before the effective deny guard.
