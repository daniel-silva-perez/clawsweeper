## Summary

Keep this PR open. Current main still does not implement Tirith pre-exec scanning or the proposed `tools.exec.commandSecurity` config, and this is a focused security-hardening change for built-in exec rather than an obsolete PR. A plugin implementation may be viable through `before_tool_call`, but deciding whether command scanning belongs in core exec versus ClawHub/plugin space needs maintainer security judgment.

## Best Possible Solution

Keep this PR open for maintainer security review. The best path is either a narrowed core implementation after reviewing the external-binary trust model, warning/approval semantics, docs, and changed-gate coverage, or an explicit maintainer request to move Tirith scanning into a ClawHub/native plugin using `before_tool_call` plus any needed plugin API extension for node/sandbox parity.

## Risks / Open Questions

- The PR introduces configurable execution of an external `tirith` binary inside built-in exec handling; trust, PATH resolution, binary spoofing, timeout, and fail-open semantics need explicit maintainer security review.
- A ClawHub/native plugin using `before_tool_call` can plausibly scan normal exec tool calls, but may not exactly match the PR's node platform resolution, sandbox/elevated behavior, and approval warning semantics without further API/product decisions.
- Merging core behavior would change documented YOLO/no-extra-prefilter expectations and would require aligned docs, config help, and focused tests.
- The PR touches `.secrets.baseline` for line-number drift; that should be reviewed as generated/security baseline churn even though the shown patch appears limited to shifted line numbers.
