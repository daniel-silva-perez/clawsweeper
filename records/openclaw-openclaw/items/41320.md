## Summary

Keep this PR open. Current main still has no Tirith integration, no `tools.exec.commandSecurity` config surface, and no built-in content-level pre-exec scanner for gateway/node exec. The proposal is security-hardening work on the built-in exec path, so it needs maintainer security/product review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer security review. The best path is either a narrowed core implementation after validating the external-binary trust model, warning/approval semantics, docs, and changed-gate tests, or an explicit maintainer redirect to a ClawHub/code-plugin implementation using `before_tool_call` if optional external command scanners should stay out of core.

## Risks / Open Questions

- The PR introduces a security-sensitive external binary execution path (`tirithPath`) inside built-in exec handling; maintainers should review trust, PATH resolution, spoofing, timeout, and fail-open semantics before merge.
- The proposed behavior changes approval/blocking semantics for gateway, node, sandbox, and elevated exec paths. A plugin implementation may be viable through `before_tool_call`, but core integration could provide tighter built-in semantics; that scope call needs maintainer judgment.
- The PR touches `.secrets.baseline` for line-number churn. That may be generated fallout from config-label edits, but baseline changes are security-adjacent and should be reviewed, not auto-accepted.
