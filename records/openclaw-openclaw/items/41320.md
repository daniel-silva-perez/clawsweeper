## Summary

Keep open. Current main does not implement Tirith or the proposed `tools.exec.commandSecurity` surface, and the PR is not mergeable as-is because its gateway scan can launch the configurable scanner before effective `security=deny` is enforced. This is a core security-boundary change, so it needs author changes plus maintainer/secops trust-model review rather than cleanup closure.

## What This Changes

The PR adds optional default-on Tirith pre-exec command scanning, wires `tools.exec.commandSecurity` through exec config/runtime paths for gateway, sandbox, and node execution, and adds command-security tests.

## Best Possible Solution

A mergeable version should resolve effective host deny before any scanner process can start, document the operator-facing config and fail-open model, add focused gateway/sandbox/node regression coverage, and settle whether this belongs in core or behind the existing plugin/trusted-policy surfaces.

## Reproduction Assessment

Yes for the blocking PR regression: with this PR, configure effective gateway exec policy to `security=deny` and command security enabled; the diff calls `checkCommandSecurity` before current main's deny resolver in `processGatewayAllowlist`, so the scanner can be started before denial. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The security-in-depth idea is plausible, and the author addressed earlier node-ordering and output-bounding feedback, but the gateway placement is not the safest implementation because it runs a configurable scanner before the effective deny guard.
