## Summary

Keep open. Current main does not implement Tirith pre-exec scanning, so the PR is not obsolete; however, the proposed patch is not mergeable as-is because the gateway path starts the configurable scanner before the existing effective `security=deny` guard runs. This is a core exec security-boundary change and needs maintainer/secops review rather than conservative cleanup closure.

## What This Changes

The PR adds default-on optional Tirith command scanning, wires `tools.exec.commandSecurity` through gateway, sandbox, and node exec paths, adds approval warning plumbing, updates config labels/schema/types, and adds command-security integration tests.

## Best Possible Solution

A mergeable version should resolve effective gateway/node deny policy before invoking any scanner process, add a regression test proving `security=deny` does not start Tirith, document the operator-facing config and fail-open/default-on behavior, and get secops agreement that this belongs in core rather than as plugin or ClawHub tooling.

## Reproduction Assessment

Yes for the blocking PR regression: set effective gateway exec policy to `security=deny`, leave command security enabled/default-on, and invoke a gateway exec. The PR diff shows `checkCommandSecurity` runs before `processGatewayAllowlist` reaches the current deny resolver, so the scanner can start before denial; I did not run tests because this review is read-only.

## Solution Assessment

No. The security-in-depth direction is plausible and the earlier node-ordering/output-bounding feedback appears addressed, but the current placement is not the safest implementation because it launches a configurable scanner before the existing effective deny guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor deny policy before launching the scanner:** `src/agents/bash-tools.exec.ts:419-423`
  - body: With effective gateway exec policy resolving to `security=deny`, current main rejects inside `resolveExecHostApprovalContext` before local execution proceeds. Resolve effective deny before the scan or move the scan after the deny guard, and add a gateway deny regression.
