## Summary

Keep open. Current main has addressed important Telegram-specific pieces and the RFC2544 fake-IP case, but the broader report is still actionable for web_fetch: the web_fetch config/runtime only exposes allowRfc2544BenchmarkRange, so a fake-IP resolver returning loopback or other private addresses for ordinary external hostnames is still blocked before fetch. This issue is broader than the closed Telegram-only and RFC2544-only related reports.

## Best Possible Solution

Keep this issue open and fix the generic web_fetch/proxy surface. A conservative path is to extend tools.web.fetch.ssrfPolicy with an explicit private-network/fake-IP opt-in or a trusted operator-proxy mode, wire it through runWebFetch/fetchWithSsrFGuard, document the security tradeoff, and add regression tests for both 198.18.0.0/15 and loopback/private fake-IP DNS answers under proxy use. Telegram-specific behavior can remain as already implemented.

## Risks / Open Questions

- A broad implementation that trusts all proxied requests would weaken SSRF protections; the likely fix needs an explicit operator-controlled web_fetch policy or trusted-proxy mode with redirect and hostname checks preserved.
- WeChat is documented as an external plugin, so the core repo should focus on the shared SSRF/proxy API surface and web_fetch behavior; plugin-side behavior may need separate verification after a generic fix exists.
