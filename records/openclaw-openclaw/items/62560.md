## Summary

Keep open. Current main only partially addresses the report: Telegram media now has RFC2544 fake-IP support plus a Telegram-specific private-network opt-in, but general `web_fetch` still uses the strict guarded-fetch path, keeps DNS pinning even when proxy env vars are set, and exposes only RFC2544/IPv6-ULA opt-ins. That leaves the reported loopback fake-IP and broad web_fetch failure class unresolved, while WeChat remains external-plugin-owned rather than directly fixable in this repo without confirming the plugin contract gap.

## What This Changes

The issue asks OpenClaw to let external URL fetches work behind local fake-IP or transparent proxies across Telegram media, WeChat, `web_fetch`, and related skills without SSRF false positives.

## Best Possible Solution

Define an explicit, documented trusted-proxy DNS mode for the affected surfaces instead of silently bypassing SSRF for every proxied request: keep strict defaults for arbitrary URLs, add narrowly scoped operator opt-ins where appropriate, cover loopback/RFC2544/ULA fake-IP DNS in tests, and expose any needed plugin SDK seam so external channel plugins such as WeChat can opt in without importing core internals.

## Reproduction Assessment

Yes for the remaining `web_fetch` behavior: a mocked DNS lookup returning a private or loopback address for a public hostname follows the current guarded-fetch path and fails before fetch, and existing tests assert the private-DNS and HTTP_PROXY-pinning behavior. Live Clash/WeChat reproduction is not available in this read-only pass, but it is not needed to show the unresolved core web_fetch path.

## Solution Assessment

No: the issue's broad suggestion to trust the proxy for all requests is not clearly the safest fix for agent-controlled URLs. A safer direction is a maintainer-approved trusted-proxy mode with explicit config, proxy-host validation, redirect handling, docs, and SDK/plugin guidance.
