## Summary

Keep this PR open. Current main still lacks any active Gateway lock-port read path for default TUI connections, and the linked bug remains open; the PR is an active implementation candidate and I found no blocking correctness or security issue in the proposed diff.

## What This Changes

The PR records the resolved Gateway runtime port in lock metadata, has TUI default local connection resolution use a verified active lock port, updates TUI/Gateway docs and regression tests, and removes two stale awaits in active-memory plugin tests.

## Best Possible Solution

Keep the focused lock-metadata approach: persist the resolved runtime port in the per-config Gateway lock, read it only for verified active local Gateway owners, preserve explicit URL and remote-config precedence, and land the regression tests and docs once maintainers are comfortable with the stale-lock semantics.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the linked bug's manual flow: start the Gateway with a custom runtime port such as 48789, then run TUI without --url; current main has no code path for TUI to discover that runtime-only port and will build from env/config/defaults.

## Solution Assessment

Yes for the core bug. Recording the resolved runtime port in the existing lock is narrower and more process-local than trying to infer it from config, and the PR preserves the documented explicit URL and remote Gateway precedence; the only cleanup question is whether the unrelated active-memory test change should be split.
