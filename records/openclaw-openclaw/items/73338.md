## Summary

Keep this PR open. Current main still does not implement the intended TUI behavior: the default local TUI connection path does not read an active Gateway lock port, and the lock payload does not persist the resolved runtime port. The linked bug #42461 remains open, so this PR is still a focused implementation candidate rather than obsolete work.

## Best Possible Solution

Keep this PR open for maintainer review as the current implementation candidate for #42461. The best path is to land a focused fix that records the resolved Gateway runtime port in the active lock, reads that verified lock for default local `openclaw tui` connections, preserves explicit `--url` and remote-config precedence, and keeps the new regression coverage plus the user-facing docs note.
