## Summary

Keep open. Current main still does not expose `agentId` on typed message hook contexts and outbound typed hooks still do not receive session identity, so the PR is not obsolete. The proposed branch is useful but not merge-ready because it is dirty against current main and its mirror/session helper can publish a mixed `sessionKey`/`agentId` pair.

## What This Changes

The PR adds optional `sessionKey` and `agentId` to typed plugin message hook contexts, threads them through inbound and outbound message hook delivery, and adds mapper, dispatch, and delivery tests.

## Best Possible Solution

Land a rebased additive contract update in the current hook type files that exposes optional `ctx.agentId`, maps it from the active inbound session and the resolved outbound delivery identity, derives `agentId` from the same selected session key, documents the new field, and covers mirror, session-only, and unset paths in targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Keep mirror session identity together:** `src/infra/outbound/deliver.ts:414`
  - body: When `mirror.sessionKey` is present but `mirror.agentId` is absent, this falls back to `session.agentId`, so hooks can receive a mirror session key paired with the sender session's agent. That breaks the correlation contract for plugins; derive the agent from the selected mirror session key or omit it until that identity is available.
  - confidence: 0.74
- **[P3] Document the new plugin context field:** `src/plugins/types.ts:564`
  - body: This adds a public typed hook context field but leaves the plugin hooks docs' message context field list without `ctx.agentId`. Update `docs/plugins/hooks.md` alongside the API change so plugin authors can discover the new contract.
  - confidence: 0.67
