## Summary

Keep open. Current main still does not expose `agentId` on the typed plugin message hook context, so the requested API gap remains real, but this PR should not merge as-is because the branch is stale against the current split hook type files and the proposed outbound resolver can pair a mirror `sessionKey` with the sender session's `agentId`.

## What This Changes

The PR adds optional `sessionKey` and `agentId` to typed plugin message hook context plumbing for inbound and outbound message hooks, with mapper, dispatch, delivery, and test updates.

## Best Possible Solution

Land a rebased additive hook contract update in the current split type files that exposes optional `ctx.agentId` alongside the existing `ctx.sessionKey` for `message_received`, `message_sending`, and `message_sent`, derives both fields from the same selected session identity, and documents the new field for plugin authors.

## Reproduction Assessment

Yes. A source-level reproduction is clear: current main's typed message context has `sessionKey` but no `agentId`, and outbound typed hook calls do not pass session identity; the PR diff also reproduces the proposed-patch bug with a mirror session key falling back to a sender session agent id.

## Solution Assessment

No. The additive API direction is reasonable, but this branch is not the best mergeable solution until it rebases to the current hook type files, keeps mirror/session identity internally consistent, and updates the public hooks docs.
