## Summary

Keep this issue open. Current main has relevant Telegram hardening around token resolution, channel reloads, startup 401 handling, polling, and a scoped sendChatAction 401 backoff, but I did not find high-confidence evidence that the Windows-specific cluster of 401s, config changes not sticking after restart, webchat-to-Telegram sync failure, and persistent session/subagent looping is fixed end to end.

## Best Possible Solution

Keep the issue open and ask for a focused Windows repro bundle: redacted `openclaw config get channels.telegram.botToken` or token-source details, `openclaw channels status --probe` or gateway status output, relevant Telegram 401 logs, exact config path/profile/state-dir, whether `TELEGRAM_BOT_TOKEN` is set, and the session key involved in the repeated `sessions_send` or subagent loop. If reproduced, split the work by owner surface: Windows config/service diagnostics in gateway/doctor, Telegram token/probe behavior in `extensions/telegram`, and persistent session looping under sessions/subagent tooling.
