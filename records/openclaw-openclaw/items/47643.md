## Summary

Keep open. Current main has stronger Telegram 401 diagnostics, token-source handling, polling guards, config hot-reload behavior, and troubleshooting docs than the reported v2026.3.13 build, but the report still combines Windows-specific Telegram auth/startup failure, config visibility/reload behavior, multi-device routing, and sub-agent loop symptoms without enough logs to prove the whole observable problem is fixed or fully superseded. Newer reports such as #73323 and #74086 are better anchors for current Windows Telegram/network regressions, but they do not cover the older 401/config-not-sticking/sub-agent-loop mix completely.

## Best Possible Solution

Keep this open for maintainer triage and consolidation. Ask for a current-release retest with `openclaw channels status --probe telegram`, `openclaw logs --follow`, the redacted token source/config path shown by the CLI, Windows/Node/install details, and whether `openclaw sessions clear --all` or `/subagents kill all` clears the loop. If the current failure matches #73323 or #74086, consolidate there; otherwise split the config-reload/status and sub-agent-loop symptoms into narrower issues.
