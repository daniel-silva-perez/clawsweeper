## Summary

Keep this PR open, but do not merge it as-is. The PR addresses a real Discord autoThread duplicate-reply report, yet current main now defaults group/channel visible replies to message-tool-only delivery, so the proposed unconditional “do not call the message tool for the normal current conversation” guidance can suppress visible Discord replies under the default policy.

## What This Changes

The PR changes the changelog, the Codex embedded developer-instruction prompt, and the bundled Discord skill so normal current-conversation replies use the final assistant message while the `message` tool is reserved for explicit out-of-band Discord actions.

## Best Possible Solution

Rework the prompt and skill wording so it is visibility-policy aware: preserve message-tool delivery when the current context says visible group/channel output must use `message(action=send)`, but prevent duplicate same-conversation sends when automatic final delivery is active or when OpenClaw has already routed the final assistant reply to the auto-created thread.

## Reproduction Assessment

Yes. The linked bug has concrete 2026.4.26 Discord autoThread reproduction steps, and current-main code/tests give a high-confidence reproduction path for the PR regression: default guild/channel turns are `message_tool_only`, so final-only guidance can suppress visible output.

## Solution Assessment

No. A prompt-only fix is plausible, but the current PR is not the best form because it unconditionally tells the agent to avoid the message tool for normal current-conversation replies; the safer fix must account for `sourceReplyDeliveryMode` or equivalent current-context guidance.
