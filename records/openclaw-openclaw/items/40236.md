## Summary

Keep open. Current main still has adjacent cron safeguards, including payload-level agent timeout, one-shot deletion, retry/backoff, failure alerts, run-log pruning, session retention, and plugin cron-change observation, but it does not implement the requested persisted job-level lifecycle contract for timeout, retry/cleanup policy, or per-job lifecycle hooks. The related open PR #45465 may address part of the hook story, but it is unmerged and does not cover the timeout and cleanup portions.

## What This Changes

The issue requests cron job lifecycle configuration for job-level timeouts, retry/cleanup policy, and start/complete/failure hooks.

## Best Possible Solution

Treat this as a cron product/API design item. A complete solution should add an additive lifecycle contract across Gateway protocol schemas, stored job normalization/types, scheduler timeout and cleanup semantics, CLI/UI/docs, tests, and security boundaries while preserving existing cron timeout, delivery, alerting, retention, and plugin observation behavior.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is directly verifiable on current main by inspecting the cron protocol schema, persisted job types, timeout policy, cleanup path, hook dispatch, and docs.

## Solution Assessment

No automated narrow fix is the best path yet. The requested shape spans protocol, persistence, scheduler semantics, hook execution security, CLI/UI/docs, tests, and an overlapping unmerged hook PR, so maintainers should settle the API and trust model first.
