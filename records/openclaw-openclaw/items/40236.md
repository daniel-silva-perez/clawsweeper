## Summary

Keep open. Current main still has adjacent cron safeguards, including payload-level timeout support, one-shot cleanup, session/run retention, failure alerts, and plugin-level `cron_changed` observability, but it does not implement the requested persisted job-level lifecycle contract for `timeoutMs`, automatic cleanup by run/time/status, or per-job start/complete/failure hooks. The related open PR #45465 may address part of the hook story, but it is unmerged and does not cover the timeout and cleanup portions.

## What This Changes

Request to add cron job lifecycle management with job-level timeout configuration, automatic cleanup policies, retry controls, and start/complete/failure hooks.

## Best Possible Solution

Treat this as a cron product/API design item. A complete solution should add an additive lifecycle contract across Gateway protocol schemas, stored job normalization/types, scheduler timeout and cleanup semantics, CLI/UI/docs, tests, and security boundaries while preserving existing `payload.timeoutSeconds`, `deleteAfterRun`, retry/backoff, `failureAlert`, session retention, run-log pruning, and plugin `cron_changed` behavior.
