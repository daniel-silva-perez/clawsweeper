## Summary

Keep this issue open. Current main still lacks the requested job-level cron lifecycle contract: there is no persisted `lifecycle` block, no job-level `timeoutMs` separate from payload `timeoutSeconds`, no max-runs/TTL/status cleanup policy, and no cron start/complete/failure hook contract. The related hook-only work in #37855 and open PR #45465 does not cover the broader timeout and automatic cleanup portions.

## Best Possible Solution

Keep this issue open as a cron product/API design item. The best path is an additive lifecycle contract across the Gateway protocol schema, cron job types/store normalization, scheduler execution timeout handling, cleanup counters/TTL/status semantics, hook delivery, CLI/UI, docs, and regression tests. Coordinate the hook portion with #37855 and #45465 while preserving existing `payload.timeoutSeconds`, `deleteAfterRun`, failure alerts, run-log pruning, and session-retention behavior.

## Risks / Open Questions

- Closing now would lose the timeout and automatic cleanup requirements because the related open hook work only covers part of the requested surface.
- Running user-defined cron hooks is security-sensitive; any implementation should reuse existing hook/plugin trust, timeout, isolation, and permission boundaries rather than adding an ad hoc script runner.
