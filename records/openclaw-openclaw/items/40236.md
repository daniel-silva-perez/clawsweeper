## Summary

Keep open. Current main still lacks a persisted job-level cron lifecycle contract for timeout, retry/cleanup policy, and per-job start/complete/failure hooks. Existing code covers payload-level isolated-run timeout, one-shot deletion, failure alerts, run/session pruning, and plugin observation via `cron_changed`, but that is narrower than the requested lifecycle API and the overlapping PR #45465 remains open for maintainer review.

## What This Changes

Add cron job-level lifecycle configuration for maximum run time, retry and auto-cleanup policy, and start/complete/failure hooks.

## Best Possible Solution

Design one additive cron lifecycle contract across Gateway protocol schemas, stored job normalization, scheduler timeout semantics, cleanup policy, hook execution boundaries, CLI/UI/docs, and focused regression tests while preserving current payload timeout, delivery, failure-alert, retention, and plugin `cron_changed` behavior.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is directly verifiable on current main by inspecting the cron protocol schema, stored job types, timeout policy, cleanup path, hook dispatch, docs, and related open PR context.

## Solution Assessment

No, the proposed shape is not ready for an automated narrow fix. The hook portion needs API and security review, and #45465 overlaps part of the design while timeout and cleanup policy remain separate scheduler concerns.
