## Summary

Current main still does not implement the requested cron job-level lifecycle contract. The repo has adjacent behavior such as payload-level agent timeouts, one-shot retry/backoff, one-shot deleteAfterRun cleanup, failure alerts, run-log/session retention, and non-cron hook/plugin surfaces, but there is no per-job lifecycle block for timeoutMs, maxRetries, autoCleanup by runs/time/status, or configured onStart/onComplete/onFailure hooks.

## Best Possible Solution

Keep this issue open as a cron product/API design item. The best implementation is an additive lifecycle contract for cron jobs that defines job-level timeout semantics, bounded retries, cleanup policies such as max runs, TTL, and status-based cleanup, and start/complete/failure hook delivery through the existing hook/plugin architecture, with Gateway protocol/schema, cron store normalization, scheduler, CLI/UI, docs, and regression tests updated together. Coordinate the hook portion with #37855 and #45465, but preserve existing payload.timeoutSeconds, deleteAfterRun, failureAlert, run-log, and session-retention behavior.

## Risks / Open Questions

- Per-job cron hooks that run user-defined scripts are security-sensitive and should reuse existing hook/plugin trust, timeout, isolation, and permission boundaries rather than adding an ad hoc script runner.
- The request spans protocol schema, stored job normalization, scheduler semantics, config, CLI/UI, docs, and tests; a partial hook-only implementation would not close the timeout and automatic cleanup portions.
- Closing now would likely lose the broader API/product tracking because the open related PR only addresses lifecycle hooks and is unmerged on current main.
