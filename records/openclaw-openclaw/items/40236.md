## Summary

Keep open. Current main still has adjacent cron safeguards, including payload-level timeout support, one-shot cleanup, failure alerts, run/session retention, and plugin-level cron observability, but it does not implement the requested persisted job-level lifecycle contract for timeout, auto-cleanup, retry policy, or per-job lifecycle hooks. The related open PR #45465 may cover part of the hook story, but it is unmerged and does not cover the timeout and cleanup portions.

## What This Changes

The issue requests cron job-level lifecycle configuration for maximum runtime, retry and cleanup policies, and start/complete/failure hooks.

## Best Possible Solution

Treat this as a cron product/API design item. A complete solution should add an additive lifecycle contract across Gateway protocol schemas, stored job normalization/types, scheduler timeout and cleanup semantics, CLI/UI/docs, tests, and security boundaries while preserving existing payload.timeoutSeconds, deleteAfterRun, retry/backoff, failureAlert, session retention, run-log pruning, and plugin cron_changed behavior.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is directly verifiable in current main by inspecting the cron protocol schema, stored job types, timeout policy, cleanup path, and docs.

## Solution Assessment

No automated narrow fix is the best path yet. The requested shape spans protocol, persistence, scheduler semantics, hook execution security, CLI/UI/docs, tests, and an overlapping unmerged hook PR, so maintainers should settle the API and trust model first.
