## Summary

Keep this open. Current main still has the Discord voice autoJoin path wired only to READY, still lacks a RESUMED listener/export/registration, and still has unguarded `connection.destroy()` calls that can throw after a connection is already destroyed. The issue is also paired with open PR #40676 using closing syntax, so it should stay open until that PR is reviewed, ported/replaced, merged, or closed.

## What This Changes

The issue reports that Discord voice autoJoin fails to rejoin after health-monitor restarts because RESUMED events are not handled and stale or already-destroyed voice connections can make cleanup throw.

## Best Possible Solution

Keep this as the canonical Discord voice reliability bug. The fix should live in `extensions/discord`: trigger autoJoin on READY and RESUMED, export/register the runtime listener, make voice connection cleanup idempotent around destroyed connections, validate the stale-connection handling against `@discordjs/voice` 0.19.2, and add focused regression tests for RESUMED autoJoin plus destroyed-connection cleanup.
