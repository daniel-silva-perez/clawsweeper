## Summary

Keep open. Current main still has the reported Discord voice autoJoin reliability gap: voice autoJoin is only wired to READY, RESUMED is dispatched without a voice listener, and the voice manager still calls `VoiceConnection.destroy()` directly on cleanup paths that can hit an already-destroyed connection. The issue is also paired with open PR #40676 using closing syntax, so it should remain open until that PR is reviewed, replaced, merged, or closed.

## What This Changes

The issue requests reliable Discord voice `autoJoin` recovery after health-monitor account restarts, including RESUMED-event autoJoin handling and idempotent stale voice connection cleanup.

## Best Possible Solution

Keep this as the canonical Discord voice reliability bug. The fix should stay inside `extensions/discord`: add/register a RESUMED autoJoin listener or equivalent local listener support, make voice connection cleanup safe around already-destroyed connections, and add focused regression coverage for RESUMED autoJoin plus destroyed-connection cleanup against the current `@discordjs/voice` contract.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure `channels.discord.voice.autoJoin`, trigger an account-level Discord resume or health-monitor restart, and observe that current main dispatches RESUMED without any voice autoJoin listener. The destroyed-connection side is also source-backed because `@discordjs/voice` throws on a second `destroy()`.

## Solution Assessment

Yes, the linked PR direction targets the right owner boundary by changing the Discord plugin rather than core. The issue should close only after that PR or a narrow replacement lands with regression tests and maintainer review.
