## Summary

Keep open. Current main still has the reported Discord voice reliability gap: Discord voice autoJoin is still wired only to READY, RESUMED dispatch uses exact listener matching with no voice listener, and the voice manager still calls `VoiceConnection.destroy()` directly on paths that can see an already-destroyed connection. The issue is also paired with open PR #40676 using closing syntax, so cleanup should wait until that PR is reviewed, updated/replaced, merged, or closed.

## What This Changes

The issue reports Discord voice autoJoin failing to rejoin after health-monitor account restarts because stale voice connection cleanup can double-destroy and autoJoin only runs on READY rather than RESUMED.

## Best Possible Solution

Keep this as the canonical Discord voice reliability bug. The maintainable end state is a narrow `extensions/discord` fix that triggers autoJoin on RESUMED or an equivalent account-resume path, makes stale/destroyed voice connection cleanup idempotent against `@discordjs/voice`, and adds focused regression coverage for RESUMED autoJoin plus already-destroyed cleanup.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure `channels.discord.voice.autoJoin`, trigger a Discord account-level resume or health-monitor restart, and observe that current main dispatches RESUMED without any voice autoJoin listener; the double-destroy failure is also source-backed by direct `connection.destroy()` calls and the pinned dependency behavior.

## Solution Assessment

Yes for the owner boundary: this should be solved inside the Discord plugin rather than in core. The current paired PR may be the right starting point, but it needs maintainer review or a current-main replacement before this issue can close.
