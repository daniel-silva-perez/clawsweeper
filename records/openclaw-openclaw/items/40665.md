## Summary

Keep open. Current main still has the reported Discord voice reliability gap: autoJoin is wired only to READY, RESUMED dispatches through exact listener matching without a voice listener, and voice cleanup paths still call `VoiceConnection.destroy()` directly despite the pinned dependency throwing on double-destroy. The issue is also paired with open PR #40676 using closing syntax, so it should remain open until that PR is reviewed, replaced, merged, or closed.

## What This Changes

The issue asks for Discord voice autoJoin to recover after health-monitor account restarts by handling RESUMED events and making stale VoiceConnection cleanup safe when connections are already destroyed.

## Best Possible Solution

Keep this as the canonical Discord voice reliability bug. The fix should stay inside `extensions/discord`: add/register RESUMED autoJoin handling or an equivalent local listener path, make stale/destroyed voice connection cleanup safe against the current `@discordjs/voice` registry contract, and add focused regression coverage for RESUMED autoJoin plus already-destroyed cleanup.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure `channels.discord.voice.autoJoin`, trigger an account-level Discord resume or health-monitor restart, and observe that current main dispatches RESUMED without any voice autoJoin listener; the double-destroy failure is also source-backed by direct `connection.destroy()` calls and the pinned dependency contract.

## Solution Assessment

Yes for the owner boundary: this should be solved in the Discord plugin rather than core. The exact open PR branch appears stale and unmergeable against current main, so the maintainable path is to update that PR or land a narrow current-main replacement with tests.
