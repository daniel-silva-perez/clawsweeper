## Summary

Keep this open. Current main still has the reported Discord voice autoJoin gap: voice autoJoin is only registered for READY, RESUMED is dispatched without a voice listener, and the voice manager still calls `VoiceConnection.destroy()` without guarding the destroyed state. The issue is also paired with open PR #40676 using closing syntax, so it should remain open until that PR is reviewed, replaced, merged, or closed.

## What This Changes

The issue reports that Discord voice autoJoin can fail after health-monitor account restarts because RESUMED events do not trigger autoJoin and stale or already-destroyed voice connections can throw during cleanup.

## Best Possible Solution

Keep this as the canonical Discord voice reliability bug. The fix should stay inside `extensions/discord`: add and export/register a RESUMED autoJoin listener, make voice connection cleanup idempotent around destroyed connections, and add focused regression coverage for RESUMED autoJoin plus destroyed-connection cleanup against the current `@discordjs/voice` contract.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure `channels.discord.voice.autoJoin`, trigger an account-level Discord resume or health-monitor restart, and observe that current main dispatches RESUMED but has no voice listener for it. The destroyed-connection error also has a dependency-backed reproduction because `@discordjs/voice` 0.19.2 throws on a second `destroy()`.

## Solution Assessment

Yes, the linked PR direction targets the right owner boundary by changing the Discord plugin rather than core. It still needs maintainer review and regression tests, and the issue should close only after that PR or a replacement lands.
