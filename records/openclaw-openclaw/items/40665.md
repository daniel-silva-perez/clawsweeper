## Summary

Keep this open. Current main still does not implement the reported Discord voice restart behavior: autoJoin is wired only to READY, RESUMED is dispatched without a voice listener, and the voice manager still has unguarded `connection.destroy()` calls that can throw on an already-destroyed `@discordjs/voice` connection. The issue is also paired with open PR #40676 using closing syntax, so it should remain open until that PR is reviewed, replaced, merged, or closed.

## What This Changes

The issue reports Discord voice autoJoin failing to reconnect after health-monitor account restarts because READY-only autoJoin and non-idempotent voice connection cleanup can leave the bot disconnected.

## Best Possible Solution

Keep this as the canonical Discord voice reliability bug. The fix should stay inside `extensions/discord`: add and export/register a RESUMED autoJoin listener, make voice connection cleanup idempotent around destroyed connections, validate the behavior against `@discordjs/voice` 0.19.2, and add focused regression tests for RESUMED autoJoin plus destroyed-connection cleanup.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to configure `channels.discord.voice.autoJoin`, trigger an account-level Discord resume/restart, and observe that current main has no voice listener for RESUMED; the destroyed-connection error also has a focused dependency-backed reproduction because `@discordjs/voice` 0.19.2 throws on a second `destroy()`.

## Solution Assessment

Yes, the linked PR direction appears to target the right boundary, but it needs maintainer review against the current `extensions/discord` layout and regression tests. A separate repair job should not be queued while that open implementation PR references this issue with closing syntax.
