## Summary

Keep open. Current main treats MiniMax timeout failures as failover-worthy and can walk the configured fallback chain, but it does not implement the requested retry after the MiniMax M2.7 and M2.7-highspeed candidates both time out. The only same-model idle-timeout retry is explicitly disabled when model fallbacks are configured, which matches the reported failure shape rather than resolving it.

## Best Possible Solution

Keep this issue open and implement a bounded, replay-safe timeout retry policy. The likely path is to add a conservative retry/backoff pass for timeout-class FailoverError cases in the model fallback or embedded runner path, gated by the same no-side-effects checks used by the existing idle-timeout retry. Add regression coverage for a MiniMax primary plus highspeed fallback chain where both timeout, and update the model failover/retry docs to describe the new behavior.

## Risks / Open Questions

- A retry fix must be replay-safe: retrying after a timeout can duplicate side effects if the model already emitted tool calls or outbound messages before the timeout was observed.
- MiniMax uses the generic Anthropic-compatible provider path, so a fix should either be generic fallback retry policy with provider-safe gates or a narrowly justified MiniMax plugin policy with tests.
