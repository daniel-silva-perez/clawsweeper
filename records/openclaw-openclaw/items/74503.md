## Summary

Keep open. The in-memory retry override matches the upstream SettingsManager contract, but the repeated-thinking guard still does not cover the linked Qwen completions path and can inject stale loop warnings after the active loop has ended, so the PR needs revision before merge.

## What This Changes

The PR changes prepared embedded Pi settings to disable SDK retry in memory and adds a repeated-thinking loop hint inside transport-message transformation.

## Best Possible Solution

Revise the PR so retry disabling remains an in-memory per-run override, then move or duplicate the repeated-thinking guard into a shared context preprocessing path that covers openai-completions/Qwen and only fires on the active replay tail, with focused transport and Qwen coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire the loop hint into completions replay:** `src/agents/transport-message-transform.ts:169-171`
  - body: The new detector only runs inside `transformTransportMessages`, but Qwen is configured as `api: "openai-completions"` and `buildOpenAICompletionsParams()` still calls `@mariozechner/pi-ai`'s `convertMessages()` directly. The linked Qwen replay path therefore never receives this hint; apply the guard before completions conversion or in a shared context preprocessing seam.
  - confidence: 0.94
- **[P2] Limit loop hints to the active replay tail:** `src/agents/transport-message-transform.ts:119-131`
  - body: The detector collects every assistant turn with a thinking block and ignores later user turns or assistant turns without thinking. After any three identical thinking+tool-call turns, unrelated later requests in the same session can keep getting the loop warning appended; reset on non-matching turns and require the pattern to be at the current replay tail.
  - confidence: 0.84
