## Summary

Keep open. The latest PR branch still has a blocking settings-manager bug: it wraps `SettingsManager.inMemory()` with `{ globalSettings, pluginSettings, projectSettings }`, but the pinned pi-coding-agent API expects one flat settings object, so the retry override would not take effect and should fail type-checking.

## What This Changes

The PR disables pi-coding-agent retry for embedded runs via an in-memory settings wrapper and adds repeated-thinking loop-hint injection in `transformTransportMessages`.

## Best Possible Solution

Land a narrow, tested fix for #73781 that disables pi-coding-agent's inner retry through a flat, non-persistent in-memory settings snapshot or equivalent runtime override, then add any thinking-loop mitigation through a tested transport/runtime seam that respects provider transcript shape and the existing loop-detection policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass a flat settings snapshot to inMemory:** `src/agents/pi-project-settings.ts:61-64`
  - body: `SettingsManager.inMemory()` accepts one flat `Partial<Settings>` and the 0.70.5 API has no `getPluginSettings()` method. This wrapper stores `globalSettings`, `pluginSettings`, and `projectSettings` as ordinary unknown settings keys, so `retry.enabled` remains unset and `getRetryEnabled()` still defaults to true; the branch should also fail type-checking. Build the flat snapshot first and override top-level `retry.enabled` there.
