## Summary

Keep this PR open, but do not merge it as-is. Current main does not implement the requested retry-disable or repeated-thinking guard, and the PR is aimed at a real linked replay-loop report, but the patch still introduces concrete regressions in settings snapshot merging and loop-hint lifetime.

## What This Changes

The PR disables pi-coding-agent SDK retry for prepared embedded agent settings via an in-memory settings snapshot and adds repeated-thinking loop-hint injection for transport replay, including OpenAI-compatible completions.

## Best Possible Solution

Revise this PR so the retry override is applied to the effective prepared settings without losing nested global/project settings, and make the repeated-thinking detector expire or reset once a non-loop assistant turn or new task has occurred.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve nested prepared settings:** `src/agents/pi-project-settings.ts:56-60`
  - body: The new snapshot uses a top-level spread of global and project settings. Upstream SettingsManager merges nested objects, so if global settings contain retry.provider or another nested field and project settings set a different key in the same block, this drops the global nested values before the in-memory manager is created. Build from the effective merged settings or deep-merge the scopes before patching retry.enabled.
  - confidence: 0.9
- **[P2] Reset the loop hint after non-loop turns:** `src/agents/transport-message-transform.ts:67-68`
  - body: The detector only records assistant turns that have non-empty thinking, so a later assistant answer with no thinking or no tool call cannot clear a previous three-turn repeated-thinking pattern. After the loop resolves, future unrelated user prompts will keep getting the synthetic loop warning until another thinking block appears.
  - confidence: 0.86
