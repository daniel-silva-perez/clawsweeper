## Summary

Keep this PR open. Current main overlaps with parts of the branch, especially registry-backed gateway discovery and parallel startup maintenance, but it does not implement the PR's core secrets-runtime auth-store shortcut or the async-lock timeout. The discussion also has a recent field note from OpenClaw 2026.4.26 showing startup latency still visible after disabling mDNS, so this is not safely closeable as implemented. The PR should not merge as-is without cleanup because the provided diff still includes production profiling stderr logs and touches Windows shell spawning.

## Best Possible Solution

Keep the PR open, but require a rebase and narrower implementation. The best path is to preserve current main's registry-backed discovery and gated startup trace, remove temporary profiling logs, implement the secrets-startup optimization by injecting or using a no-external-auth loader only for SecretRef snapshot resolution, preserve external overlays for OAuth refresh/recovery callers, add targeted regression tests for the auth loader split and async-lock timeout release/timer cleanup, and then validate with the gateway startup benchmark plus the changed gate.

## Risks / Open Questions

- The proposed auth-store shortcut can regress external OAuth refresh/recovery unless the secrets preflight path is split from non-startup OAuth callers that currently use `loadAuthProfileStoreForSecretsRuntime`.
- The PR head includes unconditional production stderr profiling logs in gateway, client, and TUI paths; these should be removed or routed through the existing gated startup trace facility.
- The Windows `scripts/ui.js` shell-spawn change is security-sensitive; any quoted-shell implementation must keep `assertSafeWindowsShellArgs` coverage before user-forwarded args reach `cmd.exe`.
- The PR is partly superseded by current main's discovery-registry and parallel-startup changes, so it likely needs a rebase and narrower diff before maintainer review.
