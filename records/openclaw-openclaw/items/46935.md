## Summary

Keep open. Current main has partial mitigation for unique bare model IDs and the chat header picker now uses provider-qualified values, but the `/model` status output still builds `Available:` from `ModelCatalogEntry.id` only. That means provider-native IDs containing `/` can still be displayed in a form that is parsed as the wrong provider/model ref when copied back into `/model`. The related canonicalization PR #47249 is closed unmerged, so there is no complete merged fix for the reported UX contract.

## Best Possible Solution

Implement the remaining user-facing round-trip fix in the Control UI slash command: format `Current model` and every `Available:` entry as the canonical value that `sessions.patch` accepts, including nested provider-native IDs such as `vercel-ai-gateway/anthropic/claude-sonnet-4-6`. Keep the existing unique-bare-ID backend inference for compatibility, and add regression coverage that copies an `Available:` entry back through `/model` or `sessions.patch`.
