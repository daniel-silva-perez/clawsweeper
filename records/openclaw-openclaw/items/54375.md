## Summary

Keep open for maintainer review. Current main has not implemented the PR's proposed behavior, and the existing source/tests preserve DBus user-bus failures as errors for the generic service-loaded check; the open question is whether that broader lifecycle contract should change or whether non-fatal handling should stay scoped to install/probe paths.

## What This Changes

This PR changes `src/daemon/systemd.ts` so `isSystemdServiceEnabled` returns `false` for DBus/user-bus unavailable `systemctl is-enabled` failures by adding `isSystemctlBusUnavailable(detail)` to the non-fatal branch.

## Best Possible Solution

Keep non-fatal DBus handling scoped to install/status probe surfaces unless maintainers explicitly decide that the generic service-loaded contract should treat exhausted DBus transport failures as not loaded; if that broader behavior is desired, update lifecycle callers and tests so configure, uninstall, and status semantics remain deliberate.
