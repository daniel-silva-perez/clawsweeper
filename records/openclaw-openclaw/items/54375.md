## Summary

Keep open for maintainer review. Current main intentionally keeps exhausted DBus/user-bus `systemctl is-enabled` failures on the error path, while install has a narrower non-fatal probe helper; the PR’s one-line broadening can make lifecycle callers treat an unknown transport failure as a known disabled service state.

## What This Changes

The PR changes `isSystemdServiceEnabled` so DBus/user-bus unavailable `systemctl is-enabled` failures return `false` instead of throwing.

## Best Possible Solution

Keep DBus/user-bus transport failures on the generic service-check error path and keep the install-specific non-fatal handling. If maintainers want broader non-fatal behavior, introduce an explicit unknown transport state and update install, status, reset, stop/restart, and uninstall callers with targeted tests.

## Reproduction Assessment

Yes. The current unit tests already model direct and exhausted machine-scope DBus/user-bus failures and assert rejection; caller inspection shows a `false` result is consumed as a successful not-loaded state in uninstall.

## Solution Assessment

No. The one-line change is not the narrowest maintainable fix because it collapses transport failure into disabled state; the existing install-specific helper is the safer scoped behavior unless the service-loaded API is redesigned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve DBus probe failures as errors:** `src/daemon/systemd.ts:635`
  - body: Returning `false` here turns an exhausted DBus/user-bus transport failure into “service not loaded.” Callers such as `openclaw uninstall --service` treat `false` as a successful not-loaded state and skip stop/uninstall, so an installed service can remain after an unavailable bus. Keep this as an error path or update lifecycle callers/tests to distinguish unknown from not loaded.
  - confidence: 0.9
