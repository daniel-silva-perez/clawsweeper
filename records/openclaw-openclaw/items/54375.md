## Summary

Keep open for maintainer handling; the current patch should not merge unchanged because it turns an exhausted DBus/user-bus transport failure into a normal not-loaded result for generic service checks.

## What This Changes

The PR adds `isSystemctlBusUnavailable(detail)` to the false-return condition in `isSystemdServiceEnabled`.

## Best Possible Solution

Keep DBus/user-bus failures on the generic service-check error path, and continue using the existing install-specific non-fatal probe handling. If maintainers intentionally broaden the contract, add an explicit unknown transport state and update install, status, reset, stop/restart, and uninstall callers with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve DBus probe failures as errors:** `src/daemon/systemd.ts:635`
  - body: Returning `false` here collapses an exhausted DBus/user-bus transport failure into “service not loaded.” Callers such as `openclaw uninstall --service` treat `false` as a successful not-loaded state and skip stop/uninstall, so an installed service can remain after an unavailable bus. Keep this as an error path or update lifecycle callers/tests to distinguish unknown from not loaded.
  - confidence: 0.9
