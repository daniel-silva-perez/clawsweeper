## Summary

Keep this PR open for maintainer judgment rather than auto-closing. Current main does not implement the PR's intended change in `isSystemdServiceEnabled`: DBus/user-bus failures still throw, and tests currently assert that behavior. There is also a real functional concern from the PR discussion: returning `false` from the shared `isLoaded` path would make uninstall-style flows treat a transport failure as “disabled” and skip service removal. Current main already handles bus-unavailable service probes as non-fatal in install/configure callers, which looks like the safer narrower surface.

## Best Possible Solution

Do not auto-close as implemented. The best maintainer path is to either reject/close the PR explicitly as behaviorally unsafe, or rework it so only install/probe surfaces treat DBus bus-unavailable as non-fatal while shared lifecycle operations continue surfacing transport failures; add a focused regression test for whichever behavior is chosen.
