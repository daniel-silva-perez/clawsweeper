## Summary

Keep open for maintainer review. Current main still keeps exhausted DBus/user-bus `is-enabled` failures on the error path, and the proposed broadening can hide transport failures from lifecycle callers that treat `false` as a successful not-loaded state. Existing review discussion already flagged this lifecycle regression, while current tests and install-specific handling support the narrower behavior.

## What This Changes

The PR changes `src/daemon/systemd.ts` so `isSystemdServiceEnabled` returns `false` when `systemctl is-enabled` reports DBus/user-bus unavailability.

## Best Possible Solution

Keep DBus/user-bus transport failures on the generic service-check error path and continue using the existing install-specific non-fatal probe handling. If maintainers intentionally want broader non-fatal behavior, introduce an explicit unknown transport state and update install, status, reset, stop/restart, and uninstall callers with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve DBus probe failures as errors:** `src/daemon/systemd.ts:635`
  - body: Returning `false` here turns an exhausted DBus/user-bus transport failure into “service not loaded.” Callers such as `openclaw uninstall --service` treat `false` as a successful not-loaded state and skip stop/uninstall, so an installed service can remain after an unavailable bus. Keep this as an error path or update lifecycle callers/tests to distinguish unknown from not loaded.
  - confidence: 0.9
