## Summary

Keep open for maintainer handling, but the patch is not correct as-is. Current main already treats DBus/user-bus `is-enabled` failures as non-fatal in the install-specific probe path, while the PR broadens the generic `GatewayService.isLoaded` contract and can make lifecycle callers treat an unknown transport failure as a disabled service.

## What This Changes

The PR changes `isSystemdServiceEnabled` so DBus/user-bus unavailable `systemctl is-enabled` failures return `false` instead of throwing.

## Best Possible Solution

Keep the install-specific non-fatal handling unless maintainers intentionally change the broader service-loaded contract. If they accept the broader contract change, update lifecycle callers so disabled, missing, and unknown transport states remain distinct, then add tests for install, status, reset, and uninstall behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve DBus probe failures as errors:** `src/daemon/systemd.ts:635`
  - body: Returning `false` here turns an exhausted DBus/user-bus transport failure into “service not loaded.” Callers such as `openclaw uninstall --service` treat `false` as a successful not-loaded state and skip stop/uninstall, so an installed service can remain after an unavailable bus. Keep this as an error path or update lifecycle callers/tests to distinguish unknown from not loaded.
  - confidence: 0.88
