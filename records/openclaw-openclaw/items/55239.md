## Summary

Keep this issue open. Current main supports Telegram custom command menu entries, but not localized descriptions for built-in/plugin/custom command menus: the config schema only models a single `description`, native `/help` and `/tts` descriptions come from fixed command specs, and Telegram sync writes one default command list with no `language_code` option even though the dependency supports it.

## What This Changes

The issue requests configurable localized Telegram Bot command descriptions, especially Simplified Chinese descriptions for menu commands such as `/tts` and `/help`.

## Best Possible Solution

Implement this in the Telegram plugin/config contract by adding an additive language-specific menu-description surface that can register default and `language_code` command lists while preserving existing command validation, native command routing, custom-command conflict rules, 100-command/text-budget caps, and docs/schema alignment.

## Repair Work Prompt

Implement configurable localized Telegram Bot command-menu descriptions for openclaw/openclaw#55239. Keep the work scoped to the Telegram plugin/config contract. Add an additive config surface under `channels.telegram` and per-account Telegram config that lets users provide language-specific descriptions for existing native/plugin/custom menu commands, at minimum supporting examples like Simplified Chinese descriptions for `tts` and `help`. Preserve existing behavior when the new config is absent. Do not let localized overrides change command routing, auth, native command names, or custom command conflict rules; this should only affect descriptions sent to Telegram menus. Update TypeScript config types, zod schema/generated config surface as required by repo conventions, Telegram docs, and changelog if user-facing. Add focused tests for default English behavior staying unchanged, a configured `zh`/`zh-CN` localized menu list causing `setMyCommands` to be called with language options, native `/help` and plugin/custom descriptions being overridden only for that language, invalid duplicate/unknown entries being handled consistently, and command hash caching accounting for language-specific lists. Do not broaden this into general docs i18n or Control UI localization.
