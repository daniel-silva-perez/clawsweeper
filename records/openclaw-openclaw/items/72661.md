## Summary

Keep open. The fix direction is right for the Feishu multi-bot NO_REPLY regression, but the current patch introduces a direct-message slash-command regression for mention-prefixed commands without whitespace and is missing the required user-facing changelog entry.

## What This Changes

The PR changes Feishu mention normalization to strip only the receiving bot's leading self-mention and adds regression tests for preserving inline self-mentions in multi-bot group messages.

## Best Possible Solution

Land a corrected version of this PR that strips the receiving bot's self-mention only when it is a leading addressing prefix followed by whitespace, slash, or end-of-text, while preserving inline self-mentions for multi-bot routing. Add focused tests for both the multi-bot regression and p2p no-space slash command preservation, plus the required changelog entry crediting the contributor.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction for #72504: parseFeishuMessageEvent passes botOpenId to normalizeMentions, which globally removes the receiving bot's own mention before the model sees the message. The PR regression is also reproducible by a p2p text event like @_bot_1/model, because the new guard does not strip the leading mention and direct command probing uses ctx.content as-is.

## Solution Assessment

No, not as written. The leading-prefix-only approach is the right narrow solution, but the prefix guard should include slash as a valid boundary and the PR needs the required changelog entry before merge.
