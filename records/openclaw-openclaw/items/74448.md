## Summary

Keep open. Current main still has the reported lifecycle shape: `agent.wait` treats a lifecycle terminal `end` snapshot as successful completion, while compaction is emitted on a separate stream and the subagent registry immediately freezes the latest child-session assistant text. I did not find a compaction/resume-chain contract that keeps the original subagent task pending until the resumed work actually finishes.

## What This Changes

The issue reports that subagent completion is announced during session compaction instead of after the resumed worker finishes, causing the parent to receive a truncated mid-task `frozenResultText`.

## Best Possible Solution

Add a compaction-aware lifecycle contract so internal compaction/resume does not satisfy parent-visible subagent completion. The final behavior should freeze and announce exactly once, after the resumed child run reaches a real terminal state, while preserving existing `sessions_yield` pause semantics.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction pattern with three compaction incidents and timestamps tying parent results to pre-final child history; current source inspection shows the same wait/freeze path still exists.

## Solution Assessment

Yes. The preferred fix should keep `agent.wait` or the subagent registry pending across compaction/resume; extending the 1.5s output-capture retry would only mask short cases and still treats compaction as task completion.
