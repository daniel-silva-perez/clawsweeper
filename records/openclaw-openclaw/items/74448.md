## Summary

Keep open. Current main still has the premature-completion shape described in the report: a non-pending, non-yielded `agent.wait` result is treated as terminal subagent success, completion output is frozen immediately from the child session, and the output reader does not treat compaction markers as a continuation state. There are overlapping open PRs for compaction/announce output, but they are unmerged and need maintainer reconciliation before this issue can close.

## What This Changes

The issue reports that auto-compaction during a running subagent can make the parent receive a successful completion event with a mid-task `frozenResultText` instead of the child’s actual final handoff.

## Best Possible Solution

Keep the issue open until subagent completion is compaction-aware end to end: either transfer/continue the wait across internal compaction resume, or otherwise suppress the first terminal-looking completion and freeze the final child handoff exactly once after the resumed turn actually settles.

## Reproduction Assessment

Yes. The report gives a concrete 3-of-3 production reproduction pattern with timestamps, session logs, compaction events, and frozen mid-task output; current source inspection matches that failure path without needing a live state-writing run.

## Solution Assessment

Unclear for the open PRs. Output filtering or retrying can reduce bad `frozenResultText`, but the most maintainable fix should preserve the parent-facing `agent.wait`/subagent completion contract across compaction rather than only masking one captured-output shape.
