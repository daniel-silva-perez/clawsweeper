## Summary

Keep this issue open. Current main still drops ordinary Telegram group messages from senders blocked by `groupAllowFrom` / `groupPolicy`; the existing visible auth replies are for native commands or DM pairing, not the requested group auto-reply with admin-contact instructions and the sender's Telegram user ID.

## Best Possible Solution

Keep this open for maintainer product/security judgment. A good implementation would likely add an explicit opt-in Telegram group unauthorized-reply setting, rate-limit it, send thread-aware replies via the existing Telegram API/error helpers, include the sender ID safely, document the behavior, and test ordinary group messages separately from native command auth replies and DM pairing.
