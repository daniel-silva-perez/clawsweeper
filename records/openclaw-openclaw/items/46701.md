## Summary

Keep open. Current main still drops ordinary Telegram group messages from senders rejected by groupPolicy/groupAllowFrom before reply dispatch, and the existing visible authorization message is limited to native command handling. The request is valid, but the public-reply default, Telegram user ID disclosure, admin-contact wording, and dedupe/rate-limit behavior need maintainer product/privacy direction before a safe automated fix can be queued.

## What This Changes

Request Telegram to reply to unauthorized group senders rejected by groupAllowFrom/groupPolicy with admin-contact guidance and their Telegram user ID instead of silently dropping the message.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is topic-aware, uses existing Telegram send/error helpers, avoids public user-ID disclosure unless deliberately configured, includes dedupe or rate limiting, documents the privacy tradeoff, and tests ordinary group messages separately from DM pairing and native command authorization.

## Reproduction Assessment

Yes. A groupPolicy="allowlist" Telegram group message from a sender not in allowFrom/groupAllowFrom reproduces the no-reply path in source: the access helper returns group-policy-allowlist-unauthorized, shouldSkipGroupMessage returns true, and the existing test expects zero replies.

## Solution Assessment

No, not as the unconditional sample behavior. The maintainable solution should be an explicit Telegram config surface with privacy-safe defaults, thread-aware delivery, and rate limiting rather than always posting the sender's numeric ID into the group.

## Risks / Open Questions

- Current Telegram config has no admin-contact, message-template, opt-in, or cooldown field for this flow.
- A correct implementation must preserve Telegram forum-topic thread parameters and avoid changing DM pairing, native command authorization, reaction, and callback authorization behavior.
