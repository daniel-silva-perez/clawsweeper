## Summary

Keep this open. Current main still silently skips ordinary Telegram group messages from senders rejected by groupAllowFrom/groupPolicy; the visible authorization replies that exist today are limited to native commands or DM pairing, not the requested group auto-reply with admin-contact instructions and the sender's Telegram user ID.

## What This Changes

Requests Telegram groups to reply to senders blocked by groupAllowFrom/groupPolicy with a friendly authorization/admin-contact message that includes their Telegram user ID.

## Best Possible Solution

Add this only behind an explicit Telegram opt-in design that is thread-aware, rate-limited or deduplicated, uses existing Telegram API/error helpers, includes the sender ID safely, documents the config, and tests ordinary group messages separately from DM pairing and native command auth replies.
