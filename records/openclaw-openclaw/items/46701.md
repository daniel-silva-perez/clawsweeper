## Summary

Keep open. Current main still drops ordinary Telegram group messages from senders rejected by groupAllowFrom/groupPolicy without the requested onboarding reply, and the existing visible authorization replies are limited to DM pairing or native command flows.

## What This Changes

Request a Telegram group onboarding reply for unauthorized senders that tells them they are not authorized and includes their Telegram user ID for an admin.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is topic-aware, deduplicated or rate-limited, uses existing Telegram send/error helpers, safely includes the sender ID, documents the privacy tradeoff, and tests ordinary group messages separately from DM pairing and native command auth replies.

## Risks / Open Questions

- A default public auto-reply to every unauthorized group sender could create group spam, reveal bot presence, and disclose numeric Telegram user IDs in shared chats.
- The requested admin-contact wording needs product/config design because current Telegram config does not expose an admin contact, message template, opt-in flag, or dedicated dedupe/rate-limit control for this flow.
- A correct implementation must preserve Telegram forum-topic/thread parameters and avoid changing existing DM pairing, native command authorization, reaction, and callback authorization behavior.
