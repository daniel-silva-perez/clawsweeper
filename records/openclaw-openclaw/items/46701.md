## Summary

Keep open. Current main still returns before sending any Telegram reply for ordinary group messages rejected by groupPolicy/groupAllowFrom, and the regression test still expects zero replies for that path. Existing visible authorization text is limited to native command handling, so the requested onboarding reply is still missing and needs maintainer privacy/product direction before implementation.

## What This Changes

Request that Telegram group senders rejected by groupAllowFrom/groupPolicy receive an authorization notice with admin-contact instructions and their Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is forum-topic aware, uses the existing Telegram send/error helpers, avoids public user-ID disclosure unless deliberately configured, includes dedupe or rate limiting, documents the privacy tradeoff, and tests ordinary group messages separately from DM pairing and native command authorization.

## Security Review

Status: needs_attention

Summary: No PR patch is under review, but the requested behavior itself needs privacy and spam-control review before implementation.

Concerns:

- **[medium] Avoid public Telegram user ID disclosure by default:**
  - body: The sample response would post the sender's numeric Telegram user ID into the group; that can be useful when explicitly enabled, but it should not be hard-coded as an unconditional public default.
  - confidence: 0.88
- **[medium] Rate-limit unauthorized group notices:**
  - body: Sending a bot reply for every rejected group message can spam the group and make the bot easier to probe, so an implementation should be opt-in and deduped or rate-limited.
  - confidence: 0.85
