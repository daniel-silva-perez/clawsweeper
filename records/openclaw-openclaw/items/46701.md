## Summary

Keep open. Current main still drops ordinary Telegram group messages from senders rejected by `groupPolicy`/`groupAllowFrom` before any reply path, while the visible authorization reply that exists is scoped to native command handling. The latest user comment also supports the report but reinforces that rate limiting and opt-in/privacy behavior need maintainer direction before this should become a fix PR.

## What This Changes

Request that Telegram ordinary group messages from non-allowlisted senders receive a friendly authorization reply with admin-contact instructions and the sender's Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice with per-group/topic inheritance, sender/chat cooldowns, thread-aware delivery, privacy-safe templating, documented user-ID disclosure tradeoffs, and focused tests for ordinary group messages separate from command auth and DM pairing.

## Reproduction Assessment

Yes. Static current-main reproduction is high confidence: a Telegram group message from sender `999999` under `groupPolicy: "allowlist"` with only `123456789` allowed reaches `group-policy-allowlist-unauthorized`, returns before dispatch, and the existing test expects zero replies.

## Solution Assessment

No, not as the unconditional sample behavior. The maintainable solution is an explicit opt-in Telegram config surface with privacy-safe defaults and rate limiting rather than always posting the sender's numeric ID into the group.

## Security Review

Status: needs_attention

Summary: No PR patch is under review, but the requested behavior itself needs privacy and spam-control design before implementation.

Concerns:

- **[medium] Avoid public Telegram user ID disclosure by default:**
  - body: The sample reply would post the sender's numeric Telegram user ID into a group chat. That can be useful when deliberately configured, but it should not become an unconditional public default.
  - confidence: 0.88
- **[medium] Rate-limit unauthorized group notices:**
  - body: A bot reply for every rejected group message can spam group chats and expose allowlist behavior to probing, so a safe implementation should be opt-in and deduped or rate-limited.
  - confidence: 0.86
