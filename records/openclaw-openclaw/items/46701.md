## Summary

Keep open. Current main still skips ordinary Telegram group messages from senders rejected by groupPolicy/groupAllowFrom before reply delivery, while the existing visible authorization replies are scoped to native command handling rather than the normal group-message path; the requested onboarding behavior is valid but needs maintainer privacy/product direction before choosing a default public reply.

## What This Changes

Request Telegram group messages from senders outside groupAllowFrom/groupPolicy to receive an onboarding reply with authorization instructions and their Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is topic-aware, deduped or rate-limited, uses existing Telegram send/error helpers, avoids public user-ID disclosure unless deliberately configured, documents the privacy tradeoff, and tests ordinary group messages separately from DM pairing and native command auth replies.

## Reproduction Assessment

Yes. A Telegram group message from sender ID 999999 with groupPolicy="allowlist" and only 123456789 in allowFrom follows the current source path to group-policy-allowlist-unauthorized, returns before reply dispatch, and the existing test expects zero replies.

## Solution Assessment

No, not as the unconditional sample behavior. The maintainable path is an explicit Telegram config surface with privacy-safe defaults, topic-aware delivery, and dedupe/rate limiting rather than always posting the sender's numeric ID into the group.
