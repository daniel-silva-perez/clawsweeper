## Summary

Keep open. Current main still silently drops ordinary Telegram group messages from senders rejected by groupPolicy/groupAllowFrom before reply dispatch, while the existing visible authorization reply only covers native command and DM-pairing flows. The gap is valid, but the requested public auto-reply needs maintainer product/privacy decisions before an automated fix should choose defaults.

## What This Changes

The issue requests a Telegram group onboarding reply for senders blocked by groupAllowFrom/groupPolicy, including instructions to contact an admin with their Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that preserves forum topic threading, uses the existing Telegram send/error helpers, avoids public user-ID disclosure unless deliberately configured, documents the privacy tradeoff, and covers ordinary group messages separately from DM pairing and native command authorization.

## Reproduction Assessment

Yes. A Telegram group message from sender ID 999999 with groupPolicy="allowlist" and only 123456789 in allowFrom follows the current source path to group-policy-allowlist-unauthorized, shouldSkipGroupMessage returns true, and the colocated test expects zero replies.

## Solution Assessment

No, not as the unconditional sample behavior. The maintainable solution should be an explicit Telegram config surface with privacy-safe defaults, topic-aware delivery, and dedupe/rate limiting rather than always posting the sender's numeric ID into the group.
