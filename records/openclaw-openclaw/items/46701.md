## Summary

Keep open. Current main still rejects ordinary Telegram group messages from senders who fail `groupPolicy: "allowlist"` / `groupAllowFrom` authorization without sending the requested onboarding reply, while existing visible authorization messages only cover separate DM pairing or native-command flows. The feature is localized, but its defaults need maintainer product/privacy judgment because it would post publicly, may disclose Telegram user IDs, and currently has no admin-contact/template/rate-limit config surface.

## What This Changes

The issue requests a Telegram group onboarding auto-reply for senders rejected by `channels.telegram.groupAllowFrom` or `channels.telegram.groupPolicy`, including instructions to share their Telegram user ID with a group admin.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is topic-aware, deduplicated or rate-limited, uses the existing Telegram send/error helpers, safely handles sender ID disclosure, documents the privacy tradeoff, and tests ordinary group messages separately from DM pairing and native command auth replies.

## Risks / Open Questions

- A default public reply to every rejected group sender could spam busy groups or reveal bot presence where silent denial is intentional.
- Including the numeric Telegram user ID in a shared group message is a privacy/product choice, especially when the current docs present third-party ID bots as less private.
- The requested admin-contact wording needs config/product design because current Telegram config has no admin contact, message template, opt-in, dedupe, or rate-limit field for this flow.
- A correct implementation must preserve forum-topic thread parameters and avoid changing existing DM pairing, native command authorization, callback, and reaction authorization behavior.
