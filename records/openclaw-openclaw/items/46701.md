## Summary

Current main still drops ordinary Telegram group messages from unauthorized senders before message processing. Native commands have a narrow "not authorized" reply, but that does not cover the requested onboarding response for normal group messages blocked by `groupAllowFrom` / `groupPolicy`.

## Best Possible Solution

Keep this issue open and implement the behavior in the bundled Telegram plugin. The likely path is to return a structured rejection reason from the group access gate, then send a friendly threaded hint with the sender's Telegram user ID for unauthorized sender cases. Add focused tests for mismatched `groupAllowFrom`, empty allowlists, missing sender IDs, disabled group/topic behavior, and native command behavior remaining unchanged.
