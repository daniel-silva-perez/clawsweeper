## Summary

Keep open. Current main still drops ordinary Telegram group messages from senders rejected by groupAllowFrom/groupPolicy without the requested onboarding reply; the request is valid, but choosing a default public reply, user-ID disclosure behavior, message wording, and rate limiting needs maintainer product/privacy review first.

## What This Changes

Request Telegram to send a friendly onboarding reply to unauthorized group senders rejected by groupAllowFrom/groupPolicy, including their Telegram user ID and admin-contact instructions.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is topic-aware, deduplicated or rate-limited, uses existing Telegram send/error helpers, safely includes the sender ID, documents the privacy tradeoff, and tests ordinary group messages separately from DM pairing and native command auth replies.

## Risks / Open Questions

- A default public reply to every rejected group sender could spam busy groups or reveal bot presence in places where silent denial is intentional.
- Including a numeric Telegram user ID in a group reply has privacy implications and should be opt-in or carefully documented.
- The current Telegram config has no dedicated admin-contact, unauthorized-reply template, opt-in flag, or rate-limit/dedupe setting for this flow.
- A correct implementation must preserve Telegram forum-topic thread parameters and avoid changing DM pairing, native command authorization, reaction, and callback authorization behavior.
