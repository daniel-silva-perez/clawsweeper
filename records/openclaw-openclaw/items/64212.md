## Summary

Keep this issue open. Current main has generic custom-provider image routing and configurable image-tool timeouts, but I found no current source, test, docs, changelog, history, or live proof that the NVIDIA NIM Kimi K2.5 image abort reported here is fixed. The latest NVIDIA catalog work still declares the bundled Kimi K2.5 row as text-only, and the remaining custom-provider path is not provider-specific proof.

## Best Possible Solution

Keep this issue open. Add focused regression or live/manual proof for NVIDIA NIM Kimi K2.5 through the same configured-provider image path used by Feishu and Telegram. First verify whether raising tools.media.image.timeoutSeconds or a matching per-model image timeout above the observed provider latency fixes it; if yes, document the required config or revisit the default.
