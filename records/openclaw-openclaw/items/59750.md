## Summary

Keep open. Current main still restores stored session model/provider overrides for resumed sessions and has tests/docs that describe session selections as persistent; I did not find a config key that lets users prefer the current default model on resume. This overlaps with broader model configuration cleanup in #28965, but it is narrower and not fully superseded by that item.

## What This Changes

The issue requests a backward-compatible setting that lets resumed sessions either keep the stored session model or ignore it and use the current configured default model.

## Best Possible Solution

If maintainers accept the direction, add a documented strict-schema config option that defaults to current `session` behavior, wires through chat/gateway/CLI model selection, and clearly defines whether it ignores or clears stored overrides during resume.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature proposal. The static behavior is high-confidence: tests and source show a session with `modelOverride` resolves to that stored model rather than the current default.

## Solution Assessment

Unclear until maintainer product review. A session-vs-default strategy is plausible and backward-compatible, but the exact config path and interactions with reset, parent sessions, and CLI resume should be decided before a repair PR is queued.
