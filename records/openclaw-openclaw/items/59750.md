## Summary

Keep open. Current main still treats the saved session model as durable session state and reapplies it on later/resumed turns, while the config/schema/docs do not expose a resume strategy that lets users prefer the current default model over the stored session model.

## Best Possible Solution

Keep this issue open and implement a documented, backward-compatible resume policy near the existing model-selection and session-entry plumbing. The default should preserve current `session` behavior; an opt-in `default` mode should ignore saved session model overrides when resuming according to clearly defined semantics, with tests for gateway session continuation, session listing/status display, reset interactions, and CLI backend resume args.
