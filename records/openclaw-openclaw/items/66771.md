## Summary

Keep this issue open. Current main still has a plausible MSTeams double-thread session-key path: Teams strips the `;messageid=` suffix before route resolution, then mutates the returned route by appending the current thread id to whatever `route.sessionKey` already contains. The shared thread helper is not idempotent, and the route cache still stores and returns the same miss object before later cache hits are cloned.

## Best Possible Solution

Keep this issue open. Fix MSTeams thread routing so the handler derives a thread-qualified key from an immutable clean base route, avoids mutating the object returned by `resolveAgentRoute()`, strips any existing `:thread:` suffix with the existing parser or makes `resolveThreadSessionKeys()` idempotent, and adds a regression test for two routed Teams channel-thread messages in the same normalized channel asserting the second key is `base:thread:new`, never `base:thread:old:thread:new`.
