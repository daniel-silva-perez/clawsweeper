## Summary

Keep this issue open. Current main still has a plausible MSTeams mixed thread-session key path: the handler resolves a normalized channel route, mutates `route.sessionKey` with a thread suffix, and the shared helper appends `:thread:<id>` to whatever base key it receives. Because route-cache misses store and return the same route object, a first thread can poison the cached base route and a later thread in the same Teams channel can become `base:thread:old:thread:new`.

## Best Possible Solution

Keep this issue open. Fix MSTeams thread routing so cached base routes stay immutable: derive the thread-qualified key from a clean base route, avoid mutating the object returned by `resolveAgentRoute()`, consider stripping an existing `:thread:` suffix with the existing session-key parser before appending, and add a regression test for two routed Teams channel-thread messages in the same normalized channel asserting the second key is `base:thread:new`, never `base:thread:old:thread:new`.
