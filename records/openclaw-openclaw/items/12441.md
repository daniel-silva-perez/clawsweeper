## Summary

Keep open. Current main still does not implement the requested shared-secret reverse-proxy bootstrap for Control UI WebSocket auth: bearer auth exists for HTTP Control UI reads, and trusted-proxy identity auth exists as a separate mode, but a proxy-injected Authorization or X-OpenClaw-Token header is not carried into the shared-secret WebSocket connect path. The existing maintainer review comment is consistent with the current code and this remains a security-sensitive Gateway auth design item.

## What This Changes

The issue requests Control UI authentication bootstrap from reverse-proxy-provided Authorization or X-OpenClaw-Token headers instead of URL fragments, query parameters, or manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model in Gateway auth: either validate Authorization/X-OpenClaw-Token during the WebSocket upgrade after the existing origin, proxy, and rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session credential for the WebSocket. Keep URL-fragment/manual token flows and the existing trusted-proxy identity mode backward compatible, with docs and focused Gateway/UI tests.
