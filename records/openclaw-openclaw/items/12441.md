## Summary

Current main still does not satisfy the requested reverse-proxy shared-secret bootstrap for Control UI WebSocket auth. Bearer auth is accepted for the HTTP bootstrap/config read path, but the WebSocket shared-secret path still comes from the browser connect frame, the bootstrap payload carries no short-lived WS/session credential, and the docs/tests describe the same split. This remains a valid Gateway/Control UI auth design item rather than a close candidate.

## What This Changes

Allow Control UI deployments behind reverse proxies to bootstrap gateway token auth from `Authorization` or `X-OpenClaw-Token` headers instead of requiring URL or manual token entry for the WebSocket session.

## Best Possible Solution

Add and document one explicit reverse-proxy shared-secret bootstrap model: either validate `Authorization` / `X-OpenClaw-Token` on the WebSocket upgrade and feed it into the existing shared-secret auth path after the current origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session credential for WebSocket auth. Existing URL fragment and manual token/password flows should remain backward compatible.
