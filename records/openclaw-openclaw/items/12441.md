## Summary

Keep open. Current main accepts bearer credentials for authenticated Control UI HTTP reads, but it still does not bridge a reverse-proxy-injected `Authorization` or `X-OpenClaw-Token` header into the browser WebSocket shared-secret connect path. The prior issue comments remain consistent with the inspected code, and this is a Gateway admin auth trust-boundary feature that needs maintainer/security design before implementation.

## What This Changes

Request to let the Control UI bootstrap Gateway shared-secret auth from reverse-proxy-supplied HTTP headers instead of requiring URL, stored UI settings, or manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for the Control UI: either validate `Authorization`/`X-OpenClaw-Token` during the WebSocket upgrade after existing origin, proxy, and rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session credential for WebSocket auth. Preserve URL-fragment/manual token flows and existing Tailscale/trusted-proxy identity modes, with focused Gateway, UI, and docs coverage.
