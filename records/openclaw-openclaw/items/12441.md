## Summary

Keep open. Current `origin/main` still has the same split described in the issue and in the April maintainer/ClawSweeper comments: Control UI HTTP reads can use bearer auth, but the browser WebSocket shared-secret path is still populated from UI state and the `connect` frame, not from proxy-injected `Authorization` or `X-OpenClaw-Token` headers. Because this changes Gateway admin auth trust boundaries, it needs maintainer/secops design before implementation.

## What This Changes

The issue requests Control UI authentication bootstrap from reverse-proxy-provided `Authorization` or `X-OpenClaw-Token` headers instead of URL/manual token entry.

## Best Possible Solution

Define one documented reverse-proxy shared-secret bootstrap model for Gateway admin Control UI auth: either validate `Authorization`/`X-OpenClaw-Token` during the WebSocket upgrade after explicit trusted-proxy, origin, and rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser credential for the WebSocket. Preserve URL-fragment/manual token flows plus existing Tailscale and trusted-proxy identity modes, then add focused Gateway/UI tests and docs.
