## Summary

Keep open. Current main still implements bearer auth for Control UI HTTP reads, but it does not bridge proxy-injected Authorization or X-OpenClaw-Token headers into the browser WebSocket shared-secret connect path. The prior maintainer and ClawSweeper comments match the current code, and this remains a Gateway admin auth trust-boundary design item rather than conservative cleanup.

## What This Changes

The issue requests Control UI auth bootstrap from reverse-proxy-provided Authorization or X-OpenClaw-Token headers so operators do not need tokenized URLs or manual UI token entry.

## Best Possible Solution

Define one documented reverse-proxy shared-secret bootstrap model for Gateway admin Control UI auth: either validate Authorization/X-OpenClaw-Token during the WebSocket upgrade after explicit trusted-proxy, origin, and rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser credential for the WebSocket. Preserve existing URL-fragment/manual token flows plus Tailscale and trusted-proxy identity modes, then add focused Gateway/UI tests and docs.
