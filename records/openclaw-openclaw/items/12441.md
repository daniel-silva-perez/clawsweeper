## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads, but it does not bridge a reverse-proxy-injected shared-secret header into the browser WebSocket connect path, and the requested behavior changes the Gateway admin auth trust boundary.

## What This Changes

Allow the Control UI to bootstrap Gateway auth from a reverse-proxy-supplied Authorization or X-OpenClaw-Token header instead of relying on URL/manual token entry.

## Best Possible Solution

Add a documented reverse-proxy shared-secret bootstrap model for the Control UI that validates Authorization/X-OpenClaw-Token only under a defined trusted-proxy and origin policy, then either authenticates the WebSocket upgrade directly or issues a short-lived browser session credential. Preserve URL-fragment/manual token, Tailscale identity, and trusted-proxy identity flows.
