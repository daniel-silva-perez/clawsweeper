## Summary

Keep open. Current main still has the reported split: Control UI HTTP reads can authenticate with a bearer header, but shared-secret WebSocket auth still comes from browser connect-frame credentials or the separate identity-header modes, so a proxy-injected Authorization or X-OpenClaw-Token header is not a complete Control UI bootstrap path. Because this changes the Gateway admin authentication trust boundary, it needs maintainer/secops design before implementation.

## What This Changes

Request that Control UI accept a proxy-injected Authorization or X-OpenClaw-Token header to bootstrap shared-secret Gateway auth without URL tokens or manual UI token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap contract for the Gateway admin Control UI: either validate Authorization/X-OpenClaw-Token during the WebSocket upgrade after existing proxy/origin/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser credential for the WebSocket. Preserve URL-fragment/manual token flows and existing Tailscale/trusted-proxy identity modes, then document and test the chosen model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure shared-secret gateway auth, proxy-inject Authorization for the page/config request, leave browser token/password state empty, and observe that WebSocket shared auth is still resolved from connectParams.auth rather than the injected header.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but the best fix needs maintainer/secops selection of the header trust boundary and browser credential-bridging model before code changes are safe.
