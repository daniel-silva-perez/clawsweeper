## Summary

Current main still does not implement the requested reverse-proxy-injected shared-secret token flow for Control UI. The Gateway accepts `Authorization: Bearer ...` for the Control UI runtime-config HTTP read, and identity-bearing trusted-proxy/Tailscale modes can satisfy WebSocket auth, but shared-secret WebSocket auth still comes from `connect.params.auth.token` / `connect.params.auth.password` or device/bootstrap credentials. This should stay open for Gateway/Control UI auth design.

## Best Possible Solution

Keep this open as a Gateway/Control UI auth design item. Maintainers should choose and document one model: validate `Authorization` / `X-OpenClaw-Token` on the WebSocket upgrade and feed it into the existing shared-secret path after normal origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for WebSocket auth. The change should include focused Gateway WS/auth tests, Control UI bootstrap/client tests, and public dashboard/Control UI docs.

## Risks / Open Questions

- This is auth-sensitive work: accepting reverse-proxy headers as shared-secret auth or issuing browser session credentials needs explicit trust-boundary, origin, proxy, rate-limit, cookie/session lifetime, and logging design.
- Browsers cannot attach arbitrary `Authorization` or `X-OpenClaw-Token` headers to `new WebSocket(...)`, so a fix likely needs server-side WS upgrade header validation or a same-origin HTTP bootstrap/session endpoint.
