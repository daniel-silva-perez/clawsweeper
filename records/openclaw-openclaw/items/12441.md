## Summary

Keep open. Current main supports bearer auth for some Control UI HTTP reads and documents trusted-proxy identity mode, but it still does not map a reverse-proxy-injected Authorization or X-OpenClaw-Token header into the browser WebSocket shared-secret auth path or issue a short-lived UI bootstrap credential.

## Best Possible Solution

Keep this open as a core Gateway/Control UI auth design issue. The best path is for maintainers to choose one documented model: either validate Authorization/X-OpenClaw-Token on the WebSocket upgrade and feed it into the existing shared-secret auth path after normal origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for the WebSocket. The fix should add focused gateway WS/auth tests, Control UI bootstrap/client tests, and public dashboard/Control UI docs.

## Risks / Open Questions

- Accepting Authorization or X-OpenClaw-Token from a reverse proxy into Control UI WebSocket auth is security-sensitive because the Control UI is an admin surface; the design needs clear origin checks, trusted-proxy boundaries, rate limiting, header stripping assumptions, and credential lifetime semantics.
- Closing this now would leave shared-secret reverse-proxy deployments dependent on URL fragment/sessionStorage/manual entry, Tailscale Serve, or switching to full trusted-proxy identity mode rather than the requested injected bearer-token flow.
