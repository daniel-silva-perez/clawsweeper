## Summary

Keep open. Current main still only covers HTTP Control UI read/bootstrap requests with bearer auth; it does not bridge a proxy-injected shared-secret header into the browser WebSocket connect path, and the requested change needs explicit Gateway auth trust-boundary design.

## What This Changes

Request that Control UI shared-secret auth accept a proxy-injected `Authorization: Bearer ...` or `X-OpenClaw-Token` header instead of requiring URL-token bootstrap or manual UI entry.

## Best Possible Solution

Define one documented reverse-proxy shared-secret bootstrap model for the Control UI: validate `Authorization` or `X-OpenClaw-Token` under clear trusted-proxy and browser-origin policy, then either authorize the WebSocket upgrade or issue a short-lived same-origin browser credential. Preserve URL-fragment/manual token, Tailscale identity, and trusted-proxy identity flows.
