## Summary

Keep open. Current main still supports bearer auth for Control UI HTTP reads and trusted-proxy identity mode, but it does not convert a reverse-proxy-injected Authorization or X-OpenClaw-Token header into the browser WebSocket shared-secret auth path, nor does the bootstrap endpoint issue a short-lived browser credential for that path.

## Best Possible Solution

Keep this open as a core Gateway/Control UI auth design item. Maintainers should choose one documented model: either validate Authorization/X-OpenClaw-Token on the WebSocket upgrade and feed it into the existing shared-secret path after normal origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for the WebSocket. The change should include focused Gateway WS/auth tests, Control UI bootstrap/client tests, and public dashboard/Control UI docs.
