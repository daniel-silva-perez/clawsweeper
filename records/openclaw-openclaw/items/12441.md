## Summary

Keep open. Current main still supports bearer auth for the HTTP Control UI bootstrap/config read path, but it does not bridge a reverse-proxy-injected shared secret into the browser WebSocket connect path; the WebSocket shared-secret check is still driven by `connect.params.auth`. The existing maintainer comment is consistent with current code and this remains a Gateway/Control UI auth trust-boundary design item rather than a safe cleanup close.

## What This Changes

Request Control UI support for reverse-proxy-injected `Authorization: Bearer ...` or `X-OpenClaw-Token` credentials as shared-secret bootstrap input without requiring token-in-URL or manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for Control UI: either validate `Authorization`/`X-OpenClaw-Token` on the WebSocket upgrade after the existing origin, trusted-proxy, and rate-limit checks, or validate it on a same-origin bootstrap endpoint and issue a short-lived browser session credential for WebSocket auth. Keep fragment/manual token flows and existing Tailscale/trusted-proxy identity modes backward compatible, with docs and focused Gateway/UI tests.
