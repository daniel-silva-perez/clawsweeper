## Summary

Current main still does not implement the requested reverse-proxy-injected shared-secret token flow. Control UI HTTP reads can accept `Authorization: Bearer ...`, and trusted-proxy identity mode can authenticate WebSocket sessions, but shared-secret WebSocket auth still comes from `connect.params.auth.token` / `connect.params.auth.password` or device/bootstrap tokens, not from `Authorization` or `X-OpenClaw-Token` headers on the page request or WS upgrade.

## Best Possible Solution

Keep this open as a Gateway/Control UI auth design item. Maintainers should choose and document one model: either validate `Authorization` / `X-OpenClaw-Token` on the WebSocket upgrade and feed it into the existing shared-secret path after normal origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for the WebSocket. The change should include focused Gateway WS/auth tests, Control UI bootstrap/client tests, and public Control UI/dashboard docs.
