## Summary

Keep open. Current main still only partially covers this: Control UI HTTP read/bootstrap requests accept bearer auth, but reverse-proxy-injected shared-secret headers are not bridged into the browser WebSocket auth path, and the requested feature changes the Gateway admin auth trust boundary.

## What This Changes

Allow the Control UI to bootstrap Gateway auth from a reverse-proxy-provided Authorization or X-OpenClaw-Token header instead of relying on URL or manual token entry.

## Best Possible Solution

Define and document one explicit reverse-proxy shared-secret bootstrap model for the Control UI: validate Authorization/X-OpenClaw-Token only under a clear trusted-proxy and origin policy, then either authenticate the WebSocket upgrade directly or mint a short-lived browser session credential. Keep URL-fragment/manual token, Tailscale identity, and trusted-proxy identity flows backward compatible.
