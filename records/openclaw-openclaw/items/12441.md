## Summary

Current main still does not implement reverse-proxy-injected shared-secret auth for the Control UI WebSocket path. Bearer auth is accepted for the HTTP bootstrap/config read path, and identity-bearing trusted-proxy/Tailscale modes can satisfy WebSocket auth, but token/password shared-secret WebSocket auth still comes from the browser connect payload.

## What This Changes

Allow Control UI gateway shared-secret authentication to bootstrap from reverse-proxy injected `Authorization: Bearer ...` or `X-OpenClaw-Token` headers instead of requiring URL/manual token bootstrap.

## Best Possible Solution

Keep this open as a Gateway/Control UI auth design item. Maintainers should choose and document one model: validate `Authorization` / `X-OpenClaw-Token` on the WebSocket upgrade and feed it into the existing shared-secret path after normal origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for WebSocket auth.

## Risks / Open Questions

- This is a security-sensitive auth design change: accepting proxy-injected shared secrets on WebSocket upgrades or issuing browser session credentials must preserve origin checks, trusted-proxy boundaries, rate limiting, and Control UI device-pairing semantics.
- Browser JavaScript cannot set arbitrary `Authorization` headers on `new WebSocket(...)`, so a fix needs a clear model for reverse-proxy upgrade injection or an authenticated HTTP bootstrap/session exchange rather than a client-only change.
