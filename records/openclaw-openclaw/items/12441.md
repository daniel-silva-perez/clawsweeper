## Summary

Keep open. Current main still accepts bearer auth for Control UI HTTP reads, but WebSocket shared-secret auth is still sourced from the browser connect frame rather than proxy-injected Authorization or X-OpenClaw-Token headers. The existing maintainer and ClawSweeper discussion correctly treats this as a Gateway admin auth trust-boundary design item, so conservative cleanup should not close it.

## What This Changes

Request support for bootstrapping Control UI gateway authentication from reverse-proxy-provided HTTP headers instead of requiring URL tokens or manual UI token entry.

## Best Possible Solution

Define one documented reverse-proxy shared-secret bootstrap model for Gateway admin Control UI auth: either validate Authorization/X-OpenClaw-Token during the WebSocket upgrade after explicit trusted-proxy, origin, and rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser credential for the WebSocket. Preserve URL-fragment/manual token flows plus Tailscale and trusted-proxy identity modes, then add focused Gateway/UI tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: enable shared-secret token auth, serve the Control UI behind a proxy that injects Authorization for the page/config request, leave UI settings without a token/password, and observe that the browser WebSocket connect path still sends only connect.params.auth-derived credentials.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but the best fix needs maintainer/secops selection of an explicit header trust and credential-bridging model before code is changed.
