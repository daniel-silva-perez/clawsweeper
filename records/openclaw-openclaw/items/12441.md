## Summary

Keep open. Current main still supports bearer-header auth for Control UI HTTP reads, but it does not bridge a proxy-injected Authorization or X-OpenClaw-Token header into the shared-secret browser WebSocket connect path. The request is valid, but it changes the Gateway admin auth boundary and needs maintainer/secops design before implementation.

## What This Changes

Request Control UI/Gateway auth support for bootstrapping shared-secret authentication from a reverse-proxy-injected Authorization or X-OpenClaw-Token header instead of requiring URL token bootstrap or manual UI token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for the Gateway admin Control UI. A safe end state would either validate Authorization/X-OpenClaw-Token during the WebSocket upgrade after existing origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser credential for the WebSocket, while preserving URL-fragment/manual token flows and existing trusted-proxy identity mode.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure shared-secret gateway auth, proxy-inject Authorization for the page/config request, leave browser token/password state empty, and observe that WebSocket shared auth is still resolved from connectParams.auth rather than the injected header.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but the best fix needs maintainer/secops selection of the header trust boundary and browser credential-bridging model before code changes are safe.
