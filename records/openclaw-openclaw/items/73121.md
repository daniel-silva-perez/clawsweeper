## Summary

Keep this open. Current main has partial runtime-aware reconciliation for cron, chat-backed CLI tasks, and Gateway agent promise completion, but stale CLI task rows tied to persistent main-session or blank child-session keys can still remain `running`, and the human task table still prints shortened IDs that exact lookup/cancel does not resolve.

## What This Changes

The issue requests automatic terminalization or reconciliation for dead CLI-runtime task records, plus clearer task cancellation lookup behavior.

## Best Possible Solution

Extend the existing task-registry maintenance path so CLI-runtime tasks with no active run context are not treated as alive solely because a durable main-session row or blank child key exists, while preserving explicit runtime recovery hooks for long-lived media/tool jobs. Align human and JSON task inspection plus cancel lookup so operators can copy or resolve the identifiers shown by `openclaw tasks list`.

## Reproduction Assessment

Yes. A code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no active run context, `lastEventAt` older than the reconcile grace, and `childSessionKey: "agent:main:main"` backed by a durable session row; current `hasBackingSession()` returns true and maintenance will not mark it terminal.

## Solution Assessment

Yes for a focused repair, not for closing now. The narrowest maintainable fix belongs in task-registry maintenance and task CLI lookup semantics, rather than a broad process monitor or unrelated retention/session-archive work.
