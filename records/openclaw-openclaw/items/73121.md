## Summary

Keep this open. Current main has partial runtime-aware reconciliation for cron and chat-backed CLI tasks, but stale CLI rows tied to persistent non-chat session keys or blank child session keys can still remain active, and the human list/cancel path still presents shortened IDs that exact lookup does not resolve.

## What This Changes

The issue requests automatic terminal reconciliation for dead CLI runtime tasks and clearer task cancel/list lookup behavior.

## Best Possible Solution

Keep the existing runtime-aware sweeper, but make CLI task liveness depend on an active run context or a true runtime-specific backing source instead of persistent session-row existence, then make task inspection and cancellation use copyable full IDs or deterministic unique-prefix resolution with clear ambiguity errors.

## Reproduction Assessment

Yes. A code-level reproduction is a stale `runtime: "cli"` task with no active run context and `childSessionKey: "agent:main:main"` backed by the session store, or a blank child key; current `hasBackingSession()` treats those as backed after the grace period.

## Solution Assessment

Yes for a focused repair. The narrow maintainable path is to extend task-registry maintenance plus CLI lookup/list behavior; a broad PID monitor would be less reliable across gateway restarts and durable task records.
