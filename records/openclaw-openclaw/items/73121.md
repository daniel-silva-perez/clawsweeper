## Summary

Keep this open. Current main has partial runtime-aware stale task reconciliation, but the remaining zombie-task behavior is still present for gateway CLI task rows tied to persistent non-chat session keys, and the operator-facing JSON/list/cancel paths still make stale records difficult to inspect or cancel consistently.

## What This Changes

The issue asks for terminated CLI/subagent runtime tasks to auto-transition out of `running` and for task cancellation to work reliably with the identifiers shown by the CLI.

## Best Possible Solution

Keep the current runtime-aware sweeper, but close the remaining task-ledger gaps: add fixtures for persistent main-session and blank-child CLI rows, make JSON and human inspection agree on reconciliation semantics or clearly expose raw mode, and make list/show/cancel use copyable or resolvable task identifiers consistently.

## Reproduction Assessment

Partly. There is a high-confidence code-level reproduction for a stale CLI task with `runtime: "cli"`, an expired `lastEventAt`, and `childSessionKey: "agent:main:main"` because current maintenance treats the persistent session entry as backing; the reporter's exact persisted rows are still missing.

## Solution Assessment

Yes for a focused repair, not for closing now. The narrowest maintainable path is to extend task-registry maintenance and task CLI lookup/list behavior rather than adding a broad process monitor or changing unrelated task retention/session archival policy.
