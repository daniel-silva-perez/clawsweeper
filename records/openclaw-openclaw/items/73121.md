## Summary

Keep open. Current main still only partially reconciles stale CLI-runtime tasks: chat-backed CLI rows can become `lost` after the active run context disappears, but blank child-session rows and durable `agent:<id>:main` rows can still remain treated as backed, and the human task table still displays shortened IDs that the exact lookup/cancel path does not resolve.

## What This Changes

Request automatic reconciliation of terminated CLI-runtime task records from `running` to a terminal state, plus clearer task cancel lookup feedback for IDs shown by `openclaw tasks list`.

## Best Possible Solution

Extend task-registry maintenance so stale CLI tasks without an active run context are not kept alive solely by blank or durable main-session backing, while retaining explicit keepalive/recovery behavior for long-lived media and other detached tool jobs. Also make the task list, show, and cancel surfaces use copyable or resolvable identifiers consistently.

## Reproduction Assessment

Yes. A code-level reproduction is a stale `TaskRecord` with `runtime: "cli"`, status `running`, no active run context, and either no `childSessionKey` or `childSessionKey: "agent:main:main"`; current backing-session logic keeps that row active after the reconcile grace. The cancel UX gap is reproduced by the list table printing shortened task/run IDs while lookup only accepts exact IDs or exact related session keys.

## Solution Assessment

Yes for a focused repair, not for closing now. The narrowest maintainable solution is to tighten the existing task-registry maintenance and task lookup/list semantics, rather than adding a broad OS process monitor or changing unrelated session archival behavior.

## Repair Work Prompt

Fix the remaining zombie CLI task reconciliation reported in https://github.com/openclaw/openclaw/issues/73121. Current main already handles chat-backed CLI task rows after active run context disappears, but stale `runtime: "cli"` task records with no `childSessionKey` or with persistent main-session keys like `agent:main:main` can still stay `running` forever because `hasBackingSession()` treats them as backed. Update `src/tasks/task-registry.maintenance.ts` so stale CLI tasks with no active run context are marked `lost`/terminal after the reconcile grace unless they are an explicitly supported long-lived detached task with a live keepalive/recovery path. Preserve media-generation keepalive behavior and ACP/subagent cleanup semantics. Also align `openclaw tasks list/show/cancel` UX so operators can copy or resolve the identifiers shown by the human table; do not make ambiguous prefixes silently cancel the wrong task. Add focused regression tests for stale CLI rows with blank childSessionKey and `agent:<id>:main`, plus lookup/list/cancel behavior. Changelog is expected if the fix is user-facing. Do not broaden this into session archival, task TTL, or memory-bloat work from #73114.
