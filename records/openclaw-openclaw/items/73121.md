## Summary

Keep this open: current main has strong partial coverage for stale task reconciliation, including automatic sweeps and regression coverage for chat-backed CLI tasks and subagent orphan cleanup, but the report is broader than the implemented proof. The remaining gaps are operator-facing and concrete: normal task listings still print truncated task IDs while cancel lookup is exact, JSON task listing deliberately bypasses reconciliation, and non-chat CLI rows can still be treated as backed merely because a persistent session row exists.

## What This Changes

The issue requests automatic recovery of CLI runtime tasks whose underlying processes have exited so they no longer remain listed as running indefinitely, plus better cancellation feedback for stale task rows.

## Best Possible Solution

Keep the existing runtime-aware sweeper, then close the remaining gaps with a focused task-ledger fix: add fixtures for the stale CLI/subagent row shapes from this report, make list/show/cancel expose or resolve copyable task identifiers consistently, and ensure the JSON and human list paths either both reconcile or clearly separate preview from raw storage.

## Reproduction Assessment

Partly. Current tests give a high-confidence reproduction for stale chat-backed CLI tasks losing their run context, but the exact exec-approval and old subagent rows from this issue are not reproducible without raw task records or a new fixture that models those persisted rows.

## Solution Assessment

No. The current maintenance implementation is the right foundation, but it is not enough to close the issue while cancel lookup still cannot use the IDs printed by the table and some CLI/JSON inspection paths can still leave stale running rows visible.
