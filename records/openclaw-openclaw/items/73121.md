## Summary

Keep open: current main has partial stale-task reconciliation, but gateway-backed/non-chat CLI rows can still remain `running` after their live run context disappears, and the operator-facing task ID/cancel paths are still inconsistent. The report is recent, has no protected label or maintainer-author signal, and is concrete enough for a focused repair rather than cleanup closure.

## What This Changes

Request automatic terminal reconciliation for dead CLI runtime tasks and clearer cancellation lookup for task records shown by `openclaw tasks list`.

## Best Possible Solution

Keep the runtime-aware sweeper, then close the remaining task-ledger gaps: stale CLI rows should be kept active only by authoritative live run state or a real isolated child execution, task list/show/cancel should expose or resolve copyable identifiers consistently, and JSON and human inspection paths should agree on whether they show reconciled operator state or raw storage.

## Reproduction Assessment

Yes for the remaining current-main path by code inspection: create a stale `runtime: "cli"` task with `childSessionKey: "agent:main:main"` and a persistent session entry, then run maintenance after the grace period; `hasBackingSession()` can still return true even with no active run context. The exact 13 live rows from the report are not available, so fixture coverage should be added for the reported row shapes.

## Solution Assessment

No, current main is not yet the best solution. The existing sweeper and run-context checks are the right foundation, but a narrower repair should make CLI liveness authoritative without relying on persistent parent session rows, and should align the task ID/cancel and JSON inspection behavior.
