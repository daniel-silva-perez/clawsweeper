## Summary

Keep this open. Current main has partial zombie-task reconciliation, but stale CLI rows tied to persistent non-chat session keys can still remain active after the live run context disappears, and the human task list still displays shortened task IDs that the cancel lookup path does not resolve.

## What This Changes

The issue requests automatic reconciliation of dead CLI runtime task records plus clearer task cancel/list lookup behavior.

## Best Possible Solution

Keep the runtime-aware sweeper but finish the CLI-specific gaps: stale CLI rows should not be kept running solely by a durable main/session entry or blank child key after the live run context is gone, and task inspection/cancellation should use copyable or resolvable identifiers consistently.

## Reproduction Assessment

Yes, at code level. A stale runtime="cli" task with no active run context, childSessionKey="agent:main:main", and a matching durable session entry still flows through findTaskSessionEntry() and remains backed instead of becoming lost; the exact reporter ledger is not available here.

## Solution Assessment

Yes for a focused repair, not for closing now. The narrow maintainable fix is in task-registry maintenance plus task CLI lookup/list behavior, while preserving the existing cron/subagent/ACP reconciliation and avoiding a broad process monitor or retention rewrite.
