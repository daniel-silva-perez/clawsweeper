## Summary

Keep open. Current main still models the session store as a string path to `sessions.json`, loads/saves it through direct filesystem JSON IO, and coordinates writes with a filesystem `.lock`; plugin/runtime APIs expose helpers for reading or mutating that store, but not a backend-registration seam. The prior issue discussion also explicitly kept this open for maintainer design follow-up, with related memory-backend comments validating demand but not implementing pluggable session-store backends.

## What This Changes

Request a pluggable session-store backend interface and config shape so file, SQLite, Redis, Convex, custom, or similar stores can replace the current path-backed `sessions.json` store.

## Best Possible Solution

Keep this open for maintainer design. Decide whether OpenClaw should add a generic `SessionStoreAdapter` core/SDK seam with the current file store as the default, or first land narrower storage work such as directory-backed or SQLite-backed sessions. Any accepted path should update config types/schema/help, docs, doctor/migration behavior, plugin SDK boundaries, and focused concurrency/durability tests.
