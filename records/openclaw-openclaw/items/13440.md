## Summary

Keep open. Current main still models the session store as a resolved filesystem path with JSON load/save helpers and store-path locking; it does not provide a configured backend driver, adapter interface, or plugin registration seam for Redis, Convex, SQLite, or custom stores. The discussion and adjacent storage PRs show real demand, but this remains a maintainer API/product design item rather than implemented behavior.

## What This Changes

The issue requests a pluggable session-store backend contract with the current file store as the default and future SQLite, Redis, Convex, or custom adapter implementations.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK contract with the existing file store as the compatibility default, or narrow the roadmap to built-in local storage such as directory-backed or SQLite sessions and document why external session backends are out of scope.
