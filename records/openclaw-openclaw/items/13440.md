## Summary

Keep open. Current main still exposes `session.store` as a filesystem path, resolves it to `sessions.json`, reads/writes JSON files directly, and serializes writes through the existing session write-lock path. Plugins can call path-based session helpers, but there is no merged `SessionStoreAdapter`, driver-shaped config, or plugin registration surface for Redis/SQLite/Convex/custom session persistence. Related memory-backend work and the open directory-backed session-store PR are useful prior art but do not implement the requested pluggable session-store backend contract.

## Best Possible Solution

Keep this issue open until maintainers choose and implement a backwards-compatible session persistence abstraction. A conservative path is to keep the current file backend as the default, add driver-shaped `session.store` config/types/help/docs, route load/save/update/readUpdatedAt/discovery/maintenance through a `SessionStoreAdapter` or stable plugin API, and add contract tests for the file backend plus a mocked non-file backend. Coordinate with #51946 so any directory-backed work becomes either the default file driver internals or one adapter implementation rather than a competing storage layer.
