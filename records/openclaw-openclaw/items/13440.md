## Summary

Keep open. Current main still exposes `session.store` as a string path to a filesystem JSON store, with direct `fs` JSON reads/writes and a `.lock` sidecar write lock. The existing plugin-facing helpers expose the same file-store helpers rather than a backend-driver or adapter registration contract, and the related directory-backed session-store PR is narrower than the requested Redis/Convex/SQLite/custom multi-instance backend seam.

## What This Changes

The issue requests a pluggable SessionStoreAdapter-style backend contract so session state can use file, SQLite, Redis, Convex, custom, or other shared storage drivers instead of only `sessions.json`.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK contract with the current file store as the compatibility default, or narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external session backends remain out of scope.
