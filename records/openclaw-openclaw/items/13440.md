## Summary

Keep open. Current main still has a path-only session store backed by `sessions.json` filesystem reads/writes and lock files, while the request is for a configurable backend/adapter contract for Redis, Convex, SQLite, custom modules, or similar stores. The related backend discussion and directory-backed session-store PR are relevant demand signals, but they do not implement the requested external session-store backend seam.

## What This Changes

Request a pluggable session store adapter interface and config shape so OpenClaw can support file, SQLite, Redis, Convex, or custom session store backends instead of only a file path.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK capability with the current file store as the compatibility default, or narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external session backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: inspect the config type/schema, store resolver, load/save implementation, lock implementation, plugin runtime surface, and session docs to confirm there is no adapter backend seam.

## Solution Assessment

Unclear. A backend adapter contract is a plausible direction, but the best solution depends on maintainer choices about core versus plugin ownership, migration guarantees, credential handling, and the relationship to the directory-backed store work.
