## Summary

Keep open. Current main still exposes `session.store` as a path string and routes session persistence through filesystem JSON reads/writes plus a path-scoped lock; there is no backend-driver config, adapter interface, or plugin registration seam for Redis, Convex, SQLite, or custom session stores. The related discussion and open directory-backed PR show valid demand, but the remaining work is a maintainer API/product decision rather than a conservative cleanup close.

## What This Changes

The issue requests a pluggable session-store backend contract so deployments can use alternatives to the default file-backed `sessions.json` store, such as SQLite, Redis, Convex, or custom adapters.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK contract with the current file store as the compatibility default, or narrow the roadmap to built-in local storage such as directory-backed or SQLite sessions and document why external session backends remain out of scope.
