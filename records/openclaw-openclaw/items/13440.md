## Summary

Keep open. Current main still implements session persistence as a path-based `sessions.json` store with direct filesystem reads/writes and filesystem lock semantics. The issue discussion also has a maintainer review explicitly keeping this open for design follow-up; related memory-backend work validates the demand but does not implement a pluggable session-store backend.

## Best Possible Solution

Keep this open for maintainer design. The next step is to decide whether OpenClaw should add a generic core/SDK `SessionStoreAdapter` seam with the existing file store as the default, or land narrower storage work first, such as directory-backed or SQLite-backed sessions. Any accepted implementation should update config types/schema/help, docs, migration/doctor behavior, plugin SDK boundaries, and focused concurrency/durability tests.
