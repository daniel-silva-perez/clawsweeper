## Summary

Keep open. Current main still represents `session.store` as a string path and routes session persistence through JSON file reads, atomic writes, and a path-scoped filesystem lock. The issue discussion and related memory-backend comments show real demand, but there is no merged session-store adapter or backend-registration seam yet.

## What This Changes

The issue requests replacing the path-only JSON session store with a pluggable session-store adapter/config model that could support file, SQLite, Redis, Convex, custom, or other backends.

## Best Possible Solution

Keep this as a maintainer design item: decide whether OpenClaw should add a generic `SessionStoreAdapter` core/SDK seam with the existing file store as the default, or first land a narrower storage model such as directory-backed or SQLite-backed sessions. Any accepted path should align config types/schema/help, docs, doctor/migration behavior, plugin boundaries, and focused concurrency/durability tests.
