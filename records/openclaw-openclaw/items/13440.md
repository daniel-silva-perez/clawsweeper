## Summary

Keep open. Current main still has a path-only, filesystem-backed session store and plugin helper surface; it does not implement the requested configurable session-store backend or adapter contract. The discussion shows real demand and adjacent storage work, but this remains a broad maintainer API/product decision rather than a cleanup close candidate.

## What This Changes

Request to add a configurable SessionStoreAdapter-style session persistence contract so OpenClaw can use file, SQLite, Redis, Convex, custom, or other shared backends instead of only `sessions.json`.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported, backward-compatible session-store adapter contract with the existing file store as the default, or narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: inspect the config type/schema, resolver, load/save/lock implementation, SDK surface, and docs to confirm there is no backend adapter seam.

## Solution Assessment

Unclear. A backend adapter contract is a plausible direction, but the best solution depends on maintainer choices around core versus plugin ownership, credentials, migrations, durability/recovery semantics, and overlap with #51946.
