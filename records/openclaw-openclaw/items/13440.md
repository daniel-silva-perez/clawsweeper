## Summary

Keep open. Current main still lacks a configurable session-store backend or adapter contract: session configuration is path-only, load/write paths still operate on filesystem JSON under a sidecar lock, and the plugin-facing surface exposes helper functions around that same file store rather than a backend registration seam. The related database-backed memory work and open directory-backed session-store PR show demand, but they do not implement the external Redis/Convex/SQLite/custom session backend contract requested here.

## What This Changes

The issue requests a SessionStoreAdapter-style contract and object config so session state can use file, SQLite, Redis, Convex, or custom multi-instance backends while keeping the file store as the compatibility default.

## Best Possible Solution

Make a maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK capability with the existing file store as the compatibility default, or explicitly narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external session backends remain out of scope.
