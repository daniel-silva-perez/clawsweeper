## Summary

Keep open. Current main still implements sessions as a path-only filesystem JSON store with direct JSON reads/writes and a sidecar write lock; the plugin-facing helpers expose those same file-store operations rather than a backend driver or adapter registration contract. The discussion and related PRs show real demand, including the Postgres/pgvector memory-backend comment and the open directory-backed session-store PR, but the requested Redis/Convex/SQLite/custom multi-instance session-store seam remains a maintainer product/API decision.

## What This Changes

Request a pluggable session storage contract so session data can use backends such as file, SQLite, Redis, Convex, or custom adapters instead of only the JSON file store.

## Best Possible Solution

Make a maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK contract with the current file store as the compatibility default, or explicitly narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external session backends remain out of scope.
