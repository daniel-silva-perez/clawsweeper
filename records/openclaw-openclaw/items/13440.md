## Summary

Keep open. Current main still has a path-only, file-backed session store with filesystem locking and helper-style plugin access, not a configurable backend or adapter registration seam. The issue discussion, the closed adapter PR, and the open directory-backed PR show real demand and adjacent work, but the requested external backend contract is still a maintainer design decision rather than implemented behavior.

## What This Changes

Request to add a pluggable session-store backend contract so JSON file storage can be replaced by file, SQLite, Redis, Convex, or custom backends.

## Best Possible Solution

Keep this as a maintainer design item. The end state should either define a supported SessionStoreAdapter-style core/SDK contract with the existing file store as the compatibility default, or explicitly narrow the product direction to a built-in storage model such as directory-backed or SQLite-backed sessions and document why external backends remain out of scope.
