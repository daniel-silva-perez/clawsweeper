## Summary

Keep open. Current main still has a path-only session store backed by filesystem JSON and lock files, not a configurable backend or adapter contract. The related memory-backend discussion and directory-backed session-store PR show useful demand, but they do not satisfy the requested Redis/Convex/SQLite/custom session-store backend seam.

## What This Changes

Request a pluggable session-store backend contract so session metadata can use file, SQLite, Redis, Convex, or custom adapters instead of only a `sessions.json` path.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK capability with the existing file store as the compatibility default, or narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external session backends remain out of scope.
