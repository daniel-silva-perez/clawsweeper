## Summary

Keep open. Current main still has a path-only, filesystem-backed session store and plugin helper surface, while the request is for a configurable backend/adapter contract for SQLite, Redis, Convex, custom modules, or similar stores. The discussion and adjacent directory-backed work show real demand, but the external backend seam remains a maintainer product/API decision rather than implemented behavior.

## What This Changes

The issue requests a pluggable session-store backend contract so OpenClaw can use non-file storage backends such as SQLite, Redis, Convex, or custom adapters instead of only `sessions.json`.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK capability with the current file store as the compatibility default, or narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: inspect the config type/schema, resolver, load/save/lock implementation, plugin runtime surface, and session docs to confirm there is no adapter backend seam.

## Solution Assessment

Unclear. A backend adapter contract is a plausible direction and aligns with the plugin/capability guidance, but the best fix depends on maintainer choices about core versus plugin ownership, migration guarantees, credentials, recovery semantics, and overlap with the directory-backed store work.
