## Summary

Keep open. Current main still models `session.store` as a string path to `sessions.json`, loads/saves that JSON file directly, and serializes writes through the existing session write-lock path. The plugin SDK exposes file helper functions, not a backend registration or adapter API, so the requested pluggable session store backends remain a real design item. Related memory-backend discussions and closed MongoDB PRs are useful context but do not supersede the session-store request.

## Best Possible Solution

Keep this open for maintainer design. The best path is to decide whether session storage should get a generic core/SDK `SessionStoreAdapter` seam with the existing file store as the default, or whether narrower work such as directory-backed/SQLite session storage should land first. Any implementation should update config types/schema/help, docs, migrations/doctor behavior, plugin SDK boundaries, and focused session-store concurrency tests.
