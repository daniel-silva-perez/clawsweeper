## Summary

Keep open. Current main still has a path-only, file-backed session store with JSON load/save helpers and a filesystem-backed write lock; there is no backend-driver config, SessionStoreAdapter contract, or plugin registration seam for Redis, Convex, SQLite, or custom session stores. The related discussion and open directory-backed PR are useful adjacent work, but they do not satisfy the broader external backend/multi-instance request.

## What This Changes

Request a pluggable session-store backend contract with the current file store as the default and future Convex, SQLite, Redis, or custom drivers for multi-instance/cloud-native deployments.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK contract with the current file store as the compatibility default, or narrow the roadmap to built-in local storage such as directory-backed or SQLite sessions and document why external session backends remain out of scope.
