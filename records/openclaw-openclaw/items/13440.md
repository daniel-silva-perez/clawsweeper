## Summary

Keep open. Current main still has a path-only filesystem session store with locked JSON reads/writes and helper-style plugin access, not a configurable backend adapter contract. The issue is valid, but it remains a broad maintainer API/product decision rather than conservative cleanup.

## What This Changes

Request a backward-compatible session-store adapter contract so OpenClaw can support file, SQLite, Redis, Convex, custom, or other backends instead of only a `sessions.json` path.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported, backward-compatible session-store adapter contract with the existing file store as the default, or narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: inspect the config type/schema, resolver, load/save/lock implementation, plugin SDK surface, and docs to confirm there is no backend adapter seam.

## Solution Assessment

Unclear. A backend adapter contract is plausible, but the best solution depends on maintainer choices around core versus plugin ownership, credentials, migrations, durability/recovery semantics, and overlap with #51946.
