## Summary

Keep open. Current main has the shipped read/adopt-side Claude CLI external auth sync and an existing low-level Claude CLI credential writer, but OAuth refresh still only saves refreshed credentials to OpenClaw auth stores. This PR targets a real remaining behavior gap, but it should get maintainer/security review because it writes refreshed OAuth secrets back into an external Claude CLI credential store.

## Best Possible Solution

Keep this PR open for maintainer/security review. If bidirectional Claude CLI sync is the intended direction, land a narrow Claude-only write-back hook after refresh that validates the anthropic:claude-cli/claude-cli profile, writes only through the existing Claude CLI writer, logs failures with redacted debug-level errors, and adds focused tests. If maintainers want to preserve the documented token-sink/read-side model, close this PR with that product-boundary explanation and keep relying on #70902-style managed-profile sync.

## Risks / Open Questions

- The proposed change writes OAuth access and refresh tokens into an external Claude CLI credential store, so it needs explicit maintainer/security approval of that product boundary.
- The PR patch logs String(error) in the new write-back catch path; a thrown error that embeds credential material could leak token text into logs unless it uses the repo's redacting error formatter.
- The proposed hook should guard both profileId and credential provider/identity before writing to Claude CLI storage, to avoid writing the wrong provider or account tokens if auth-profile state is malformed.
- The PR currently changes source files only and does not add regression tests for successful write-back, no-op cases, failure isolation, provider mismatch, or redacted logging.
- The PR comments mention Claude/Codex write-back, but the implementation only writes Claude CLI credentials; wording should not imply unsupported Codex CLI mutation.
