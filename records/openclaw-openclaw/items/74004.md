## Summary

Keep open. Current main has the read-side Claude CLI external auth sync and the existing low-level Claude CLI credential writer, but it does not wire any post-refresh write-back hook from the OAuth manager into the Claude CLI file or Keychain. This PR targets a remaining behavior gap rather than work already implemented on main, but it needs maintainer/security review because it would write refreshed OAuth secrets into an external credential store.

## Best Possible Solution

Keep this PR open for maintainer review. If bidirectional Claude CLI sync is the intended product direction, implement it as a narrow provider-scoped OAuth refresh hook that writes only validated `anthropic:claude-cli` / `claude-cli` credentials via the existing Claude CLI writer, with regression tests for successful write-back, safe no-op cases, and failure isolation. If maintainers want to preserve the documented token-sink model instead, close this PR with that product-boundary explanation and keep relying on #70902-style read/adopt behavior.

## Risks / Open Questions

- The proposed hook writes OAuth tokens to an external Claude CLI credential store; it should verify both the managed profile id and expected provider/identity before writing to avoid cross-provider or cross-account credential clobbering.
- The PR changes credential persistence behavior without tests in the provided changed-file list; it needs targeted unit coverage for success, failure, non-Claude profiles, provider mismatch, and Keychain/file fallback behavior.
- The current docs describe OpenClaw auth profiles as the token sink and note prior Codex decisions against external write-back, so maintainers should explicitly confirm Claude CLI is an intended exception before merging.
- The proposed helper catches failures, but existing low-level Claude CLI writers log warnings; align logging with the PR's stated best-effort/debug behavior if accepted.
