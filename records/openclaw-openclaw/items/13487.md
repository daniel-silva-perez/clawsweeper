## Summary

Keep open. Current main still lacks deterministic Discord multi-bot target selection: the Discord preflight path resolves the normal route before target signals, checks mentions and reply metadata only against the running handler bot ID, and has no suppression state that would stop a default owner from responding when another configured bot is targeted.

## What This Changes

The issue requests Discord routing precedence where explicit bot mentions win, reply-targeted bot messages are next, and the per-channel default owner is used only when no targeted signal exists.

## Best Possible Solution

Implement target selection inside the Discord plugin before shared default routing: map configured Discord bot user IDs to OpenClaw accounts/agents, select explicit mentioned bot accounts first, select the referenced-message bot author second, and fall back to the channel default owner only when no targeted signal exists. Add focused coverage proving targeted cases suppress the default owner.

## Reproduction Assessment

Yes. GTZhou provided a live guild-channel repro, and current main has a static reproduction path: route selection happens before mention/reply target evaluation, and target checks only compare against the running botUserId.

## Solution Assessment

Yes. The narrow maintainable path is a Discord-owned target-selection step plus a small generic suppression seam only if shared mention-gating needs to express default-owner suppression; broad core hardcoding of Discord bot identities would be the wrong boundary.
