## Summary

Keep open. Current main still does not implement deterministic Discord multi-bot target precedence: the Discord preflight path resolves the normal route before target-signal handling, only compares mentions/replies with the running handler bot ID, and has no shared suppression result to prevent a default owner from responding when another configured bot is targeted.

## What This Changes

The issue requests Discord routing precedence of explicit bot mention, then reply-target bot, then per-channel default owner, with the default owner suppressed whenever a targeted signal exists.

## Best Possible Solution

Implement target selection inside the Discord plugin before shared default routing: map configured Discord bot user IDs to OpenClaw accounts/agents, select explicit mentioned bot accounts first, select the referenced-message bot author second, and fall back to the channel default owner only when no targeted signal exists. Add focused preflight/routing tests proving targeted cases suppress the default owner.

## Reproduction Assessment

Yes. GTZhou provided a live guild-channel reproduction, and current-main static inspection shows the same failure path: route selection happens before target evaluation, and mention/reply checks only compare against the running botUserId.

## Solution Assessment

Yes. The narrow maintainable solution is Discord-owned target selection plus a small generic suppression seam only if shared mention-gating needs to express default-owner suppression; hardcoding Discord identities in core would cross the plugin boundary.
