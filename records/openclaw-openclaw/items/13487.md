## Summary

Keep open. Current main still resolves the Discord route before applying cross-bot target signals, compares explicit mentions and reply metadata only against the running handler's single bot user ID, and lacks a suppression result that would stop the default owner when another configured bot is targeted. Related open PRs #47393 and #67460 cover nearby owner and mention-suppression behavior, so this should remain open for maintainer coordination rather than conservative cleanup.

## What This Changes

Request deterministic Discord multi-bot routing where explicit bot mentions win over reply targets, reply targets win over the per-channel default owner, and the default owner is suppressed whenever a targeted signal exists.

## Best Possible Solution

Implement Discord-owned target selection before shared default routing: build a configured Discord bot user ID to OpenClaw account/agent map, select explicit mentioned bot accounts first, select the referenced-message bot author second, and fall back to the per-channel default owner only when no targeted signal exists. Add focused preflight/routing tests that prove targeted cases suppress the default owner, with only a small generic mention-gating seam if shared state must express that suppression.

## Reproduction Assessment

Yes. GTZhou provided a concrete live guild-channel reproduction, and current-main inspection shows the matching failure path: routing is selected before target signals and mention/reply checks compare only against the running botUserId.

## Solution Assessment

Yes. The requested precedence is the narrow maintainable direction, but it should be implemented inside the Discord plugin boundary with a generic suppression seam only if needed; hardcoding Discord bot identities into shared routing would be the wrong layer.
