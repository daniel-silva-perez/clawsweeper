## Summary

Keep this issue open. Current main still creates a separate QMD XDG cache and `index.sqlite` per agent, so the central multi-agent disk-I/O concern remains. Main has useful mitigations, including deferred startup for some agents, per-agent update serialization, an embed lock, configurable update/search modes, and read-only SQLite busy timeouts, but it does not provide a shared QMD cache, QMD WAL/checkpoint/vacuum controls, or QMD cache size/LRU limits.

## Best Possible Solution

Keep this open as a memory-core/QMD backend design issue. The best path is to decide whether OpenClaw should support a shared, agent-namespaced QMD home/index or add bounded maintenance and cache-size controls around per-agent QMD homes, then cover the chosen behavior with config docs and multi-agent startup/update tests.

## Evidence

- **Issue context is concrete and recent:** The report describes 19 persistent agents, per-agent QMD SQLite caches under `~/.openclaw/agents/<name>/qmd/xdg-cache/qmd/index.sqlite`, 2.5GB combined cache size, and 34.36GB gateway writes over 20 hours; the follow-up asks for auto-vacuum or periodic WAL checkpoint maintenance. The issue was also assigned shortly after filing, so this is not stale cleanup.

## Risks / Open Questions

- The exact macOS write-volume numbers were not live-reproduced in this read-only review, so thresholds for an eventual fix still need production-style measurement.
- QMD owns the SQLite index implementation behind the CLI, so an OpenClaw fix may need either upstream QMD support or a carefully versioned compatibility contract.
