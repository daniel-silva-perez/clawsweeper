## Summary

Keep this PR open. Current main still conflates Mattermost session routing and delivery routing for channel/group thread messages when `replyToMode` is `off`, while the later shipped DM reply-root work only addresses the direct-message side of the same area.

## What This Changes

The PR separates Mattermost delivery thread identity from session-thread identity, routes replies, typing indicators, inbound `MessageThreadId`, and tool context through that delivery id, and folds in unknown-channel-type and reset-classification fixes.

## Best Possible Solution

Land a current-main replacement or rebase that keeps the newer DM non-threading behavior, adds a separate Mattermost channel/group delivery-thread id for `replyToMode: "off"`, wires typing, draft preview, final delivery, `MessageThreadId`, and tool context through that id, and adds focused tests for reset classification and unknown channel-type handling.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: a Mattermost group/channel post with `threadRootId` and `replyToMode: "off"` produces `effectiveReplyToId: undefined`, and all current delivery call sites use that undefined value as the outbound thread root; the PR author also reported live verification on 2026.3.28.

## Solution Assessment

Unclear as currently branched. The core approach of separating session routing from delivery routing is the right shape, but the branch is stale and mixes the #57970 dependency, so the safest solution is a rebased or replacement patch against current main that preserves newer Mattermost threading behavior.
