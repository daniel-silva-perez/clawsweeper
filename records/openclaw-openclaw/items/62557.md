## Summary

Keep open. Current main still routes `openclaw update` Linux restart handling through the managed per-user systemd service path, while separate doctor/status code and docs recognize system-level OpenClaw services as a supported external lifecycle shape. The updater does not yet detect or give accurate restart guidance for those system-scope units.

## What This Changes

`openclaw update` should detect system-wide systemd OpenClaw gateway services and either restart them safely or print the correct `sudo systemctl restart <unit>` guidance.

## Best Possible Solution

Add a Linux update restart branch that reuses existing system gateway discovery to identify active system-scope OpenClaw units, preserves the managed user-service behavior, and produces either a verified restart or precise system-scope manual restart guidance.

## Repair Work Prompt

Fix the Linux `openclaw update` behavior reported in https://github.com/openclaw/openclaw/issues/62557. Current package-update restart handling only sees the managed per-user systemd service; system-wide OpenClaw gateway units under `/etc/systemd/system` can keep running the old PID/version after the global package updates. Implement a narrow fix that preserves existing managed user-service behavior, detects active system-scope OpenClaw gateway units using the existing service-discovery concepts (`findSystemGatewayServices`, `isSystemdUnitActive`, or a shared helper), and then either restarts the system unit only when already authorized or prints precise manual guidance such as `sudo systemctl restart <unit>`. Do not silently run sudo, do not migrate/overwrite system units at update time, and do not turn the generic managed service adapter into a broad system-unit controller. Cover custom names like `openclaw.service` when the unit ExecStart clearly runs `openclaw gateway`, not only `openclaw-gateway.service`. Update user-facing docs and CHANGELOG if behavior changes; credit `Thanks @robinchoice`.
