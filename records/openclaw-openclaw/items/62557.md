## Summary

Keep open. Current main still treats Linux update restarts as OpenClaw-managed systemd user-service work: the update flow reads the normal gateway service state, the detached Linux restart helper runs `systemctl --user restart`, and the service adapter resolves units under the user systemd directory. Existing docs and doctor/status code recognize system-level OpenClaw units as a supported external supervisor shape, but `openclaw update` does not use that discovery to restart or print the correct system-scope restart command.

## What This Changes

The issue asks `openclaw update` to detect system-wide systemd OpenClaw services after a package update and either restart them or print the correct `sudo systemctl restart ...` guidance instead of user-service guidance.

## Best Possible Solution

Teach the Linux update restart path to recognize active system-scope OpenClaw gateway units using the existing service-discovery concepts, then either restart only when already authorized or print precise system-scope restart guidance while preserving the existing managed user-service behavior.

## Repair Work Prompt

Fix the Linux `openclaw update` behavior reported in https://github.com/openclaw/openclaw/issues/62557. Current package-update restart handling only targets OpenClaw-managed systemd user units and the detached restart helper emits `systemctl --user restart ...`; for a supported external/system service under `/etc/systemd/system/openclaw*.service`, the updater can leave the running gateway on the old PID/version and print misleading user-service guidance. Implement a conservative fix: during update restart handling on Linux, detect active system-scope OpenClaw gateway units using existing service discovery/systemd helpers where possible, and when such a unit owns the gateway lifecycle, either restart it only when already authorized/root or print an explicit `sudo systemctl restart <unit>` command and avoid the misleading `systemctl --user`/gateway install guidance. Do not add automatic sudo prompts or broaden core service ownership beyond this update hint/restart path without maintainer approval. Keep managed user-service behavior unchanged. Add focused Vitest coverage for system-scope detection/hint behavior and update docs/changelog because this is user-facing; credit `Thanks @robinchoice` in the changelog entry.
