## Summary

Keep open. Current main still does not make `agent --json` consume a post-`before_message_write` finalized assistant message: the CLI runner builds payloads from CLI output text, the command path persists the CLI transcript afterward, and delivery serializes the unchanged payload list in JSON mode. There is also a related open PR by the reporter for the same hook/payload class of bug on the pi embedded runner, so this issue should remain available for the CLI-specific fix.

## Best Possible Solution

Keep this issue open and fix the CLI result path so `agent --json` returns the same finalized assistant text that persistence and `before_message_write` hooks make user-visible. The most direct implementation is to thread the finalized assistant message from the write-hook/persistence path back into the CLI run result, or re-read the last persisted assistant after the guarded write, then add a focused regression test for a `before_message_write` hook that rewrites or blocks the final assistant text and assert both session JSONL and JSON `payloads[].text` agree. Coordinate with #56031 only for shared helper ideas; that PR is scoped to pi embedded runner `--expect-final` and does not close the CLI surface here.
