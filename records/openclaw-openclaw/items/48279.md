## Summary

Keep open. Current main still advertises ACP session listing and pins `@agentclientprotocol/sdk` 0.21.0, but `AcpGatewayAgent` only implements `unstable_listSessions` while SDK v0.16+ dispatches `session/list` to `agent.listSessions`. The latest release tag is still affected, the direct fix PR #48292 is closed unmerged, and open PR #73458 is related metadata work rather than a replacement fix.

## What This Changes

The issue reports that ACP `session/list` fails because OpenClaw implements `unstable_listSessions` while ACP SDK v0.16+ dispatches to `listSessions`.

## Best Possible Solution

Expose `listSessions` on `AcpGatewayAgent` using the existing Gateway-backed session-list logic, keep `unstable_listSessions` as a thin alias if compatibility is desired, and add dispatcher-level regression coverage for `session/list` so future ACP SDK method-name changes fail in tests.

## Reproduction Assessment

Yes. The reporter’s stdin JSON-RPC command is a concrete reproduction path, and the current static path confirms it: `openclaw acp` constructs an SDK `AgentSideConnection`, SDK v0.21.0 dispatches `session/list` to `agent.listSessions`, and current `AcpGatewayAgent` does not define that method.

## Solution Assessment

Yes. The narrowest maintainable fix is to add the SDK-required `listSessions` method backed by the existing implementation and test it through the dispatcher; a direct rename like PR #48292 is close, but a delegating alias is safer around older callers and related in-flight ACP session work.
