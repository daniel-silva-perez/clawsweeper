## Summary

Current main already fixes the reported runtime-deps restaging loop: generated package-level runtime-deps manifests are now accepted as supersets, later lazy activation has a materialized fast path, and regression coverage guards both behaviors. The fix is present on main after v2026.4.29, but I could not prove a shipped release tag contains it yet.

## What This Changes

Report that v2026.4.29 packaged bundled plugins repeatedly re-stage runtime dependencies on activation because package-level runtime-deps manifests are compared with strict array equality.

## Best Possible Solution

Keep the current main runtime-deps materialization behavior and let the next release carry it; no additional core change is needed unless a new trace shows repeated staging on a build that includes the generated-manifest superset fast path.

## Reproduction Assessment

Yes. The high-confidence reproduction is the v2026.4.29 package-level generated runtime-deps manifest containing all bundled deps, followed by lazy activation of one plugin whose requested deps are a subset; the old strict equality path re-staged, and current regression tests model the corrected no-reinstall path.

## Solution Assessment

Yes. The current main fix is narrow and maintainable: generated manifests may be supersets, but each requested package still has to satisfy version and entry-file checks before staging is skipped.
