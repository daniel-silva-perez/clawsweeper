## Summary

Keep open: current main has partial bundled-plugin path cleanup, but the reported old-version pnpm/npm package path still appears capable of falling through to the fatal missing `plugins.load.paths` validation path. The existing repair only recognizes aliases derived from the current package root, so it does not fully cover the upgrade failure described here.

## What This Changes

The issue asks OpenClaw to stop failing after package-manager upgrades when config contains versioned absolute paths to bundled plugins, either through upgrade-safe paths or `doctor --fix` repair.

## Best Possible Solution

Recognize stale OpenClaw-packaged bundled plugin paths across prior versioned package roots by known bundled plugin leaf/id, ignore them non-mutatingly during discovery when safe, and have `doctor --fix` remove those redundant entries while preserving fatal validation for unrelated missing custom paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure `plugins.load.paths` with a nonexistent old `node_modules/openclaw/dist/extensions/acpx` path; current discovery falls through to `plugin path not found`, and validation turns it into a `plugins.load.paths` issue.

## Solution Assessment

No for an automatic startup rewrite. The maintainable route is a narrow bundled-path compatibility repair and warning/ignore path for recognizable OpenClaw-owned packaged plugin aliases, plus `doctor --fix` cleanup, not broad migration or arbitrary path rewriting at gateway start.
