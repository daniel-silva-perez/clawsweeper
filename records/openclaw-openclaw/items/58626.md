## Summary

Keep open. Current main still has the reported upgrade hole: missing configured plugin load paths remain fatal validation errors, while the existing bundled-path doctor repair only removes aliases derived from the current OpenClaw package root, not an old versioned global package directory.

## What This Changes

The issue requests recovery after upgrades when `plugins.load.paths` and legacy plugin install records contain absolute paths into an older global OpenClaw package directory.

## Best Possible Solution

Add a targeted `doctor --fix` repair that recognizes missing old-version OpenClaw bundled plugin paths by package layout and plugin id, removes only the redundant bundled load-path entry when the current bundled plugin is available, and leaves ambiguous or third-party missing paths as validation errors.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure `plugins.load.paths` with a nonexistent old versioned bundled plugin path, then plugin discovery emits `plugin path not found`, validation reports `plugins.load.paths`, and the current doctor repair does not match that old package root.

## Solution Assessment

Yes. The best maintainable direction is the narrow doctor repair path, not startup-time config mutation or weakening all missing-path validation.
