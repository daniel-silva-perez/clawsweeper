## Summary

Keep open. Current main still has only global plugin entry configuration at `plugins.entries.<id>.config`; agent entries remain strict and do not expose `plugins` or an equivalent per-agent plugin config override. The newer discussion reinforces the multi-agent need, and the related closed scoping issues do not supersede this deeper config-merge request.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The likely fix belongs in the core config and plugin-runtime contract: add an agent-scoped plugin override shape, define how it deep-merges with `plugins.entries.<id>.config`, preserve SecretRef and redaction semantics, expose the effective config consistently to plugin hooks/tools/providers, and document/test the behavior alongside existing per-agent tool policy.

## Risks / Open Questions

- Treating `agents.list[].tools.allow/deny` as a complete fix would leave the reported Mem0-style use case unresolved because all agents would still share one plugin config payload, API key, and project-level settings.
- A future implementation needs explicit design for merge precedence, SecretRef resolution/redaction, runtime cache keys, hook/tool/provider context exposure, docs, and schema-generated help.
