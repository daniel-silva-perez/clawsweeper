## Summary

Keep open. Current main still supports plugin configuration only at the global `plugins.entries.<id>.config` level, while per-agent config covers sandbox, tools, TTS, memorySearch, params, runtime, and related agent settings but not plugin config overrides. The SDK/runtime resolver and loader also pass a single plugin config without agent context, so the reported Mem0-style need for different API keys/projects per agent remains unresolved. The related closed plugin-scoping issue covered allow/deny lifecycle gating, not deep-merged per-agent plugin config payloads.

## Best Possible Solution

Keep this issue open for core config and plugin-runtime design. The likely path is to add an agent-scoped plugin override shape, define deep-merge and validation semantics against `plugins.entries.<id>.config`, preserve secret handling/redaction, expose the effective config consistently to plugin hooks/tools/providers, and document/test the behavior alongside existing per-agent tool policy.
