## Summary

Keep open. Current main still has only global plugin configuration under `plugins.entries.<id>.config`; agent entries remain strict and do not accept `plugins`, and the SDK/runtime helper still resolves plugin config without an agent dimension. The discussion adds concrete production multi-agent evidence, while the related plugin-scoping issue covered allow/deny lifecycle gating rather than deep-merged per-agent config payloads.

## What This Changes

Request to add per-agent plugin configuration overrides, deep-merged with global plugin config, so multiple agents can use different plugin credentials, projects, and settings.

## Best Possible Solution

Keep this as a core config and plugin-runtime design item. The durable end state should define an agent-scoped plugin override shape, validate and merge it against each plugin's global config consistently, preserve secret handling, expose the effective request-time config through documented SDK/runtime seams, and update docs/tests alongside existing per-agent tool policy.

## Reproduction Assessment

Not applicable as a feature request, but the config-contract check is high confidence. Current main would reject the proposed `agents.list[].plugins` key through the strict agent schema and would still resolve only global `plugins.entries.<id>.config`.

## Solution Assessment

Unclear. The proposed deep-merge direction may be the right generic core feature, but it needs maintainer design because plugin-specific agent maps can solve some cases today while a generic SDK/config seam affects arbitrary plugins and SecretRef-bearing credentials.
