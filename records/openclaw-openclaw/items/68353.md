## Summary

Keep open. Current main still rejects the proposed `author` payload on `chat.send`, does not persist `authorId` or `authorName` with user turns, and continues to derive display attribution from existing `senderLabel` or inbound metadata blocks. The request is concrete and recent, but the authoritative identity/trust semantics need maintainer review before implementation.

## What This Changes

Add an optional validated `author` object to `chat.send` and round-trip it through `chat.history` and `session.message` as `senderLabel`, `authorId`, and `authorName` for multi-user shared sessions.

## Best Possible Solution

Define an additive gateway protocol contract for per-message author metadata, including validation, storage on user transcript rows, display projection into `chat.history` and `session.message`, UI/client typing, tests, docs, and explicit trust semantics for whether the field is attested or merely client-declared.
