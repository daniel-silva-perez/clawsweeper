## Summary

Keep open. Current main still lacks a first-class `author` field on `chat.send`, does not persist author metadata with chat-send user transcript rows, and still projects `senderLabel` from stored labels or inbound metadata blocks. The request is concrete and recent, but it changes a public gateway protocol and identity-attribution trust model, so it should get maintainer/security/product review before implementation.

## What This Changes

Add optional per-message author metadata to `chat.send`, persist it with user turns, and project it as `senderLabel`, `authorId`, and `authorName` in `chat.history` and `session.message` for multi-user shared sessions.

## Best Possible Solution

Define an additive gateway protocol contract for per-message authorship that covers validation, storage on user transcript rows, projection into `chat.history` and `session.message`, client/UI typing, docs, tests, and explicit trust semantics for whether the values are attested or merely client-declared.

## Risks / Open Questions

- Accepting a client-declared author as authoritative could create or formalize spoofed attribution unless the protocol clearly distinguishes client-declared display metadata from attested roster/device identity.
- The timestamp-prefixed inbound metadata sanitizer hardening is independent from the protocol author field and may be better split into a smaller follow-up if maintainers want it sooner.
- Docs now mention Control UI browser-local identity attribution, but the inspected `chat.send` schema and UI request path do not show a first-class `author` contract, so the docs and implementation need reconciliation during design.
