## Summary

Keep open. Current main does not have the persisted tail-drop behavior, and this PR is directionally useful and addresses the earlier counter bookkeeping review, but the latest patch still has a blocking tail-detection gap: it stops at any non-message tail entry even though upstream SessionManager can use metadata entries as leaves without appending them as replay messages, leaving an errored assistant immediately below that metadata as the last replayed turn.

## What This Changes

The PR branch modifies `repairSessionFileIfNeeded` to drop trailing errored assistant entries, adds a `droppedTrailingErrorEntries` report/log counter, expands session-file-repair regression tests, and documents the transcript-hygiene rule.

## Best Possible Solution

Revise the PR so persisted repair reasons over the replay-visible tail: preserve unrelated metadata rows, treat metadata-only tail entries as transparent when finding the last replayed message, drop trailing errored assistant entries underneath them, keep mid-transcript rewrite behavior, and add regression coverage for the metadata-tail case before the changed gate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `session -> user -> errored assistant -> model_change` or `thinking_level_change`: the PR stops at the metadata row, while SessionManager uses that row as the leaf without appending it as a replay message, so the assistant remains the final replayed turn.

## Solution Assessment

No. The repair direction is the right narrow place to fix persisted poisoned transcripts, and the counter fix is solid, but stopping at all non-message tail entries is not the best solution because metadata-only entries should be preserved while scanning past them to the last replay-visible message.
