## Summary

Keep open. Current main does not implement native `nodes` file transfer, so this PR is not obsolete, but the branch is not merge-ready because it introduces security-sensitive default file reads plus requester/contract bugs in the node-host write path.

## What This Changes

The PR adds `file_pull` and `file_push` actions to the `nodes` agent tool, new `file.read`/`file.write` node commands, node-host handlers, command-policy entries, and unit coverage for the new transfer helpers and invoke paths.

## Best Possible Solution

Land a revised file-transfer implementation that keeps the useful `file_pull`/`file_push` workflow while treating node filesystem access as high-risk, preserving requester-specific approval semantics, honoring the documented write flags, supporting host-platform path rules, and adding focused tests/docs for those contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate file.read before enabling it by default:** `src/gateway/node-command-policy.ts:63`
  - body: This adds `file.read` to the host-node default command set, but the handler reads any absolute or `~` path under the node-host user's permissions. A write-scoped operator or agent can exfiltrate arbitrary node files without an explicit dangerous-command opt-in or path approval, unlike camera/screen mutations that stay gated.
  - confidence: 0.92
- **[P1] Preserve requester-specific file.write approval:** `src/node-host/invoke.ts:588-590`
  - body: The node-host permission check scans every configured agent and permits the write if any one has `file.write` allowlisted or `security: full`. Once the gateway command is enabled, an approval for one agent can unlock arbitrary file writes for other callers because the request is not resolved against the initiating agent/session.
  - confidence: 0.86
- **[P2] Honor mkdirs and overwrite in the node handler:** `src/node-host/invoke.ts:650-651`
  - body: The tool forwards `mkdirs` and `overwrite`, but the node-host handler always creates parent directories and always overwrites via `writeFileSync`. Requests with `overwrite: false` can still replace existing files, which breaks the declared contract and risks data loss.
  - confidence: 0.9
- **[P2] Accept Windows absolute paths for host file transfer:** `src/node-host/invoke.ts:524-530`
  - body: The new path validation only accepts `/` or `~` prefixes. Windows is included in the default command policy, but normal Windows absolute paths such as `C:\tmp\a.txt` are rejected before read or write, so file transfer is broken on that advertised platform.
  - confidence: 0.88
- **[P2] Reject malformed base64 before writing files:** `src/node-host/invoke.ts:636-639`
  - body: `Buffer.from(data, "base64")` is permissive and can silently ignore invalid characters. A malformed upload can be accepted as a successful `file.write` with different bytes than the caller sent, so validate base64 strictly before decoding and writing.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces node filesystem read/write capabilities and currently broadens access in ways that need security review before merge.

Concerns:

- **[high] Default-enabled arbitrary node file reads:** `src/gateway/node-command-policy.ts:63`
  - body: `file.read` is added to host-node defaults while the handler reads arbitrary absolute or home-relative paths. That creates a default exfiltration primitive for paired node hosts instead of requiring an explicit high-risk command opt-in or path-scoped approval.
  - confidence: 0.92
- **[medium] file.write approval can leak across agents:** `src/node-host/invoke.ts:588`
  - body: The write gate checks all agents in the exec-approvals file rather than the requester. If any agent has `file.write` allowlisted or `security: full`, other callers can pass the node-host gate once the gateway command is enabled.
  - confidence: 0.86
