## Summary

Current main does not implement this PR's requested `web_fetch` env-proxy opt-in, so this PR should not be closed as already implemented. The branch is small and relevant, but the proposed unconditional `useEnvProxy: true` changes a security-sensitive SSRF/proxy boundary and appears to conflict with an existing current-main regression test that says untrusted `web_fetch` should keep DNS pinning even when `HTTP_PROXY` is configured.

## What This Changes

This PR adds `useEnvProxy: true` to the `web_fetch` guarded-fetch call site and adds a regression test asserting that `createWebFetchTool` forwards that flag.

## Best Possible Solution

Keep this PR open for maintainer/security review. The best path is to decide the intended `web_fetch` contract first: keep default strict DNS pinning, add a documented/configured trusted-proxy opt-in like the broader related `#58034`, or intentionally switch default behavior and update the existing strict-mode test and public proxy/security docs together.
