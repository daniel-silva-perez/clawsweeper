## Summary

Keep this PR open. Current main still only retries `EEXIST` from `fs.open(lockPath, "wx")` in `acquireSessionWriteLock`; a Windows `EPERM` during lock acquisition would still be rethrown before the existing retry/stale-lock path. The PR is a focused implementation candidate for the reported Windows crash, not obsolete on main.

## Best Possible Solution

Keep the PR open and review it as an implementation candidate. The best fix belongs in `src/agents/session-write-lock.ts`: treat Windows `EPERM` from `fs.open("wx")` as lock contention only when it is plausibly caused by an existing contended lock, preserve immediate permission errors for non-Windows and real ACL failures, and keep focused regression tests for Windows retry plus permission-error behavior.
