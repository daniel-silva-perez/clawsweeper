## Summary

Keep open. Current main still rethrows non-EEXIST errors from session lock creation, so the Windows EPERM crash path is not implemented on main; the PR is useful but needs maintainer review because the head has failing/pending checks and a remaining lock-handoff race concern.

## What This Changes

The PR updates `acquireSessionWriteLock` to treat Windows `EPERM` from `fs.open(..., "wx")` as retryable lock contention in selected cases and adds focused tests for Windows, non-Windows, and missing-lock EPERM behavior.

## Best Possible Solution

Land a small current-main patch in `src/agents/session-write-lock.ts` that handles Windows EPERM as lock contention without weakening real permission errors, preserves the current timeout/error behavior, and keeps focused regression tests in `src/agents/session-write-lock.test.ts`.

## Reproduction Assessment

Yes at unit level: mock `fs.open` for the `.lock` path to throw `EPERM` while a lock file exists, then current main rethrows because only `EEXIST` enters the retry path. No live Windows filesystem reproduction was performed in this read-only pass.

## Solution Assessment

Unclear as-is. The Windows-scoped retry direction is maintainable, but the current PR branch needs rebase/validation and should address or consciously accept the consecutive EPERM+ENOENT handoff race.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Retry EPERM handoff races until timeout:** `src/agents/session-write-lock.ts:557-558`
  - body: The new Windows branch treats the second `EPERM` followed by `ENOENT` as a real permission error and throws immediately. A contended owner can release and remove the lock between the failed `open` and `access` checks on consecutive attempts, so bursty handoffs can fail early instead of continuing the lock retry loop until `timeoutMs`.
  - confidence: 0.78
