## Summary

Keep this issue open. Current main has shipped partial private-network support for trusted web_search provider endpoints such as SearXNG and for model-provider transport, but the broader requested capability is still missing for generic web_fetch, Gateway OpenResponses URL file/image inputs, and the public config/schema surface that would let operators opt in safely.

## Best Possible Solution

Keep this issue open as the broader tracker. The best path is a narrow, explicit private-network opt-in for tools.web.fetch.ssrfPolicy plus Gateway OpenResponses files/images URL inputs, with propagation to remote media/image references that inherit web_fetch policy. Preserve default blocking, add strict schema/types/help/docs, isolate cache keys by policy, and add SSRF tests for direct fetches, redirects, DNS/private resolution, Gateway URL inputs, and media reference loading. Coordinate with #39604/#39630 for the web_fetch subset, but keep this issue for the remaining Gateway and cross-tool work.

## Risks / Open Questions

- Private-network access is SSRF-sensitive; any implementation should stay default-off unless maintainers explicitly approve a local-mode default.
- Any new web_fetch policy bit must be included in cache keys so blocked and allowed fetches cannot share cached payloads.
- Redirects, DNS pinning, explicit proxies, hostname allowlists, Gateway input_file/input_image URL fetches, and remote media/image reference loading all need regression coverage.
