## Summary

Keep this issue open. Current main has partial private-network support for trusted web_search provider endpoints, including SearXNG, and model-provider transport, but the broad request is still not implemented for generic web_fetch or Gateway OpenResponses URL file/image inputs.

## Best Possible Solution

Keep this issue open. The best path is a narrow, explicit private-network opt-in for tools.web.fetch.ssrfPolicy and Gateway OpenResponses files/images URL inputs, plus propagation to remote media/image references that inherit web_fetch policy. Preserve default blocking, add strict schema/types/help/docs, isolate cache keys by policy, and add SSRF tests for direct fetches, redirects, Gateway URL inputs, and media/image reference loading. Treat SearXNG web_search and models.providers[].request.allowPrivateNetwork as shipped partial precedent, not full resolution.

## Risks / Open Questions

- Private-network access is SSRF-sensitive; any implementation should remain default-off unless maintainers explicitly approve local-mode auto-enable behavior.
- Any new web_fetch policy bit must be included in cache keys so blocked and allowed fetches cannot share cached payloads.
- Redirects, DNS pinning, explicit proxies, hostname allowlists, Gateway input_file/input_image URL fetches, and remote media/image references need denied-by-default and opt-in regression coverage.
