## Summary

Keep open. Current main still has the source-level behavior described by the report: bundled runtime dependency aliases are added to plugin Jiti loads, and that alias resolver prefers the dependency export `import` target over `require`, which maps `ws` to its ESM wrapper instead of the CommonJS constructor entry. The related #74312 context is useful debugging background, but this is a separate plugin loader compatibility bug rather than a duplicate.

## What This Changes

The issue asks OpenClaw to preserve Node-style CommonJS `require("ws")` constructor semantics in custom plugins, or at least document and support the namespace-object workaround.

## Best Possible Solution

Preserve CommonJS dependency semantics for plugin `require()` by making bundled runtime dependency alias resolution choose the `require` export when building sync Jiti aliases, falling back to `import` only when no require-compatible target exists, with a regression test using a `ws`-shaped conditional-export package.

## Repair Work Prompt

Fix the plugin loader compatibility bug where custom plugin CommonJS `require("ws")` can receive the ESM wrapper/namespace shape because bundled runtime dependency Jiti aliases prefer package `exports.import` over `exports.require`. Keep the fix in the plugin loader/runtime-deps alias boundary: update `src/plugins/bundled-runtime-deps-jiti-aliases.ts` so sync plugin dependency aliases preserve CommonJS `require` semantics when a package exposes a `require` condition, falling back to `node`, `default`, or `import` only when appropriate. Add focused regression coverage with a `ws`-shaped conditional-export fixture proving the alias map points bare `ws` at the CommonJS constructor entry, and if useful add a loader-level test that a plugin doing `const WebSocket = require("ws")` sees a constructable function. Do not change bundled plugin import styles, do not add a plugin-specific `ws` special case, and do not broaden core/plugin ownership boundaries. Add a changelog fix entry under `Unreleased` crediting `@aderius`.
