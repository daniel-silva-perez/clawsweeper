## Summary

Keep open. Current main still has the source-level behavior described: bundled runtime dependency aliases are collected from package conditional exports using `import` before any `require` condition, then injected into plugin Jiti loads, so a `ws`-shaped dependency can be resolved to its ESM wrapper for CommonJS plugin code.

## What This Changes

The issue reports that custom plugins using CommonJS `require("ws")` receive an ESM namespace object instead of the constructor in the plugin loader.

## Best Possible Solution

Preserve CommonJS dependency semantics for plugin `require()` by making sync Jiti runtime dependency alias resolution prefer a package `require` export when present, falling back to `import` only when no require-compatible target exists, and add a regression test using a `ws`-shaped conditional-export package.

## Repair Work Prompt

Fix the plugin loader compatibility bug from openclaw/openclaw#74547. Current main builds bundled runtime dependency Jiti aliases from conditional exports using `import`, `node`, `default` priority and never checks `require`, so a custom plugin using CommonJS `require("ws")` can receive the ESM wrapper instead of the `ws` constructor. Change the bundled runtime dependency alias resolver so sync plugin/Jiti `require()` aliases prefer a package `require` export when available, falling back to `import`/`node`/`default` only when no require-compatible target exists. Keep the fix generic; do not special-case `ws`, bundled plugin ids, or custom plugin names. Add a regression test in `src/plugins/bundled-runtime-deps-jiti-aliases.test.ts` with a `ws`-shaped conditional-export package where `require` and `import` point to different files, and update any loader test only if needed to prove the alias is applied to plugin loads. Add a user-facing changelog fix entry credited `Thanks @aderius` if the repository’s active changelog section expects one. Do not change dependency versions, pnpm patches, plugin install policy, or docs unless the implementation intentionally chooses to document a different contract instead of preserving CommonJS semantics.
