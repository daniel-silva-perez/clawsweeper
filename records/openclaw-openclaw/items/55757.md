## Summary

Keep this PR open. Current main still passes a resolved CLI no-output watchdog into stdin-fed child spawns, so the proposed fix is not obsolete; however, the diff is not merge-ready because it guards on truthy stdin payload content instead of stdin presence and can report a misleading 0s watchdog duration in an unexpected timeout path.

## What This Changes

The PR modifies `src/agents/cli-runner/execute.ts` to skip the CLI no-output watchdog when a stdin payload is present and adjusts no-output timeout message construction for an optional watchdog value.

## Best Possible Solution

Revise the PR so normal closed-stdin child spawns skip the CLI no-output watchdog based on raw stdin presence or explicit input-mode state, keep Claude live-session timers and the overall timeout intact, avoid bogus timeout-duration text, and add focused coverage for non-empty stdin, empty stdin, and args-mode runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use stdin presence for the watchdog guard:** `src/agents/cli-runner/execute.ts:169-175`
  - body: `stdinPayload` is `stdin ?? ""`, so an empty stdin-mode prompt leaves this guard false even though the child adapter still writes and closes stdin. Please guard on raw stdin presence or explicit input mode so all closed-stdin child spawns skip the no-output watchdog.
  - confidence: 0.9
- **[P3] Avoid reporting a 0s no-output timeout:** `src/agents/cli-runner/execute.ts:219-225`
  - body: When the watchdog is disabled, `noOutputTimeoutMs ?? 0` can produce a user-facing "0s" no-output timeout if an unexpected no-output result reaches this branch. Please split the disabled-watchdog message path instead of formatting a fake duration.
  - confidence: 0.78
