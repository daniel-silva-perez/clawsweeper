## Summary

Keep this PR open. Current main still passes a CLI no-output watchdog into normal child spawns even when the resolved prompt is sent through stdin that the child adapter closes, so the PR is not obsolete; however, the proposed patch still needs a rebase/revision for empty-stdin handling, timeout messaging, live-session scoping, and regression coverage before it should land.

## What This Changes

The PR changes `src/agents/cli-runner/execute.ts` to skip the CLI no-output watchdog when a stdin payload is present and adjusts no-output timeout message construction for an optional watchdog value.

## Best Possible Solution

Revise the PR so only normal closed-stdin child spawns skip the CLI no-output watchdog, using raw stdin presence or explicit input-mode state; preserve Claude live-session timers, keep the overall timeout intact, avoid misleading timeout text, and add focused CLI-runner tests for non-empty and empty stdin.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use stdin presence for the watchdog guard:** `src/agents/cli-runner/execute.ts:169-175`
  - body: The proposed guard checks `stdinPayload`, but `stdinPayload` is `stdin ?? ""`. If a stdin-mode backend sends an empty prompt, the child adapter still closes stdin after writing `""`, yet this guard remains false and the no-output watchdog stays enabled. Please guard on raw stdin presence or explicit input mode instead.
  - confidence: 0.9
- **[P3] Avoid reporting a 0s no-output timeout:** `src/agents/cli-runner/execute.ts:219-225`
  - body: When the watchdog is disabled, the fallback `noOutputTimeoutMs ?? 0` can produce a user-facing "0s" watchdog message if an unexpected no-output result reaches this branch. Please split the message so the disabled-watchdog path does not report a bogus duration.
  - confidence: 0.78
