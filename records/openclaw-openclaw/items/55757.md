## Summary

Keep this PR open. Current main still computes and passes a CLI no-output watchdog into normal child spawns even when the prompt is supplied through stdin that the child adapter immediately closes. The PR is a plausible implementation candidate, but it needs a current-main rebase, empty-stdin-safe logic, live-session scoping, and regression coverage before it can land.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase onto current main, scope the watchdog skip to normal child spawns with closed stdin, use raw stdin presence or explicit input mode instead of payload truthiness, keep Claude live-session behavior deliberate, avoid misleading timeout text, and add regression tests for stdin-fed child spawns including empty-string stdin and preservation of the overall timeout.

## Risks / Open Questions

- The PR's current guard uses stdinPayload truthiness, which would still arm the watchdog for an explicit empty stdin payload even though stdin is closed.
- The PR's current no-output error fallback can produce misleading 0s timeout messages if an unexpected no-output result reaches that branch after disabling the watchdog.
- A straight rebase could accidentally disable Claude live-session no-output protection, because live sessions use stdin mode but keep stdin open via pipe-open.
- Disabling the no-output watchdog for stdin-fed child spawns means a silent hung backend relies on the overall timeout, so regression tests should prove the overall timeout still terminates runaway processes.
