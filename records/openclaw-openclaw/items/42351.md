## Summary

Keep open. Current main still does not register a Feishu handler for `im.chat.access_event.bot_p2p_chat_entered_v1`, and the default WebSocket monitor path still passes the fixed dispatcher map into the Lark SDK client. Current main added Feishu WebSocket startup retry handling, but that does not add or ignore the reported event, and the related implementation PR was closed unmerged.

## Best Possible Solution

Keep this issue open. Add a Feishu-owned no-op or low-noise handler for `im.chat.access_event.bot_p2p_chat_entered_v1` in `extensions/feishu/src/monitor.account.ts`, plus focused regression coverage that the handler is registered on the WebSocket monitor path. During the fix, reproduce whether the restart comes from the Lark SDK dispatcher, event pressure, or Gateway supervision before changing generic Gateway restart behavior.
