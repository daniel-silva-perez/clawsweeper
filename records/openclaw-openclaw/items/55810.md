## Summary

Keep this PR open. Current main still lacks this Docker workspace mount propagation setting, but the PR is not merge-ready because browser sandbox containers would ignore the new propagation mode and the config docs incorrectly describe Docker `rslave` as read-only.

## What This Changes

This PR adds a `sandbox.docker.workspaceMountPropagation` setting to the config/type/schema/generated-schema path, threads it into main Docker sandbox creation, and updates workspace-mount/hash tests for explicit `rw` and `rprivate`/`rslave`/`rshared` modes.

## Best Possible Solution

Finish the PR by applying the selected propagation mode consistently to both main and browser sandbox workspace mounts, correct the `rslave` documentation to describe one-way propagation rather than read-only access, and add the required user docs/changelog/config-schema validation for the new public sandbox config surface.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable the sandbox browser with `sandbox.docker.workspaceMountPropagation: "rslave"`: the PR adds the option to the main sandbox create path, while the browser create path still calls the same mount helper without passing the field.

## Solution Assessment

No. The feature direction is useful, but the current patch is incomplete; the narrow maintainable fix is to pass the resolved propagation mode through the browser container path as well and correct the Docker semantics in the config docs before maintainer policy review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Propagate workspace mount mode to browser containers:** `src/agents/sandbox/docker.ts:526`
  - body: The new option is only passed when creating the main sandbox container. `ensureSandboxBrowser` still builds the same workspace mount without `workspaceMountPropagation`, so `rslave`/`rshared` setups can see host sub-mounts in the main sandbox but not in browser-assisted flows using the same configured workspace.
  - confidence: 0.89
- **[P2] Remove read-only claim from rslave docs:** `src/config/types.sandbox.ts:67`
  - body: Docker's `rslave` controls one-way recursive mount propagation; it does not make propagated sub-mounts read-only. Read-only access is controlled separately by `ro`, so this comment can lead users to expose writable host sub-mounts while believing the mode is read-only.
  - confidence: 0.82

## Security Review
