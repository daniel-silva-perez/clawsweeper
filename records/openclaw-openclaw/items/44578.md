## Summary

Keep this PR open, but do not merge it as written. Current main still has the silent-success fallback for send results without a delivery messageId, and the PR addresses a real bug, but the latest diff would also mark valid gateway-executed plugin sends as errors because those paths can return a top-level messageId rather than result.messageId.

## What This Changes

The PR adds a missing-delivery guard to the shared message tool and tests no-result, dry-run, and successful-messageId send result paths.

## Best Possible Solution

Keep the missing-delivery behavior, but revise the helper so valid plugin/gateway send successes are preserved, either by scoping the guard to core direct sends or by accepting the top-level messageId payload shape used by gateway-executed plugin sends. Then rebase or cherry-pick the focused commits onto current main and validate the message-tool plus plugin-dispatch paths.

## Reproduction Assessment

Yes. The current-main bug can be reproduced with a mocked core send MessageActionRunResult whose payload and sendResult lack result.messageId; the PR-introduced regression can be reproduced with the existing gateway plugin send shape that returns a top-level messageId payload.

## Solution Assessment

No, not as written. Treating missing core delivery as a status:error tool result is the right direction, but the implementation must not classify gateway plugin sends with a valid top-level messageId as failures.
