## Summary

Keep open. Current main still intentionally flattens remote Control UI markdown images to alt text, and the surrounding CSP, docs, tests, and changelog describe that as no-remote-fetch hardening. The open paired PR #41025 directly targets this issue, so the remaining action is maintainer product/security review rather than cleanup closure or an automated fix lane.

## What This Changes

Request to render remote http/https markdown image URLs inline in Control UI chat instead of flattening them to alt text while preserving data-image handling.

## Best Possible Solution

Make an explicit Control UI remote-media policy decision. If remote images should be allowed, implement it coherently across renderer, sanitizer expectations, CSP, docs, and tests with a reviewed safety model such as opt-in, allowlist, or managed proxy; otherwise keep the current hardening and close this as an intentional policy choice after maintainer review.

## Reproduction Assessment

Yes. The current static path is high-confidence: `toSanitizedMarkdownHtml("![Alt text](https://example.com/img.png)")` reaches the image renderer branch that returns escaped alt text, and the colocated test asserts no `<img>` output for the same class of URL.

## Solution Assessment

Unclear until maintainers decide the remote-media policy; a renderer-only fix is not the best solution because current CSP, docs, and tests intentionally block remote image fetches. The maintainable path is either a full policy-backed implementation or an explicit decision to keep the current hardening.
