## Summary

Keep open. Current main still flattens remote markdown images to alt text and the surrounding Control UI CSP, docs, tests, and changelog now describe that as deliberate no-remote-fetch hardening. The open paired PR #41025 specifically targets this issue, so the right path is maintainer product/security review of that policy rather than closing or queuing an automated fix.

## What This Changes

The issue requests that Control UI chat markdown render remote http/https image URLs inline instead of flattening them to alt text.

## Best Possible Solution

Make an explicit Control UI remote-media policy decision. If remote images are allowed, implement the change across renderer, sanitizer expectations, CSP, docs, and tests with a clear safety model such as opt-in, allowlist, or managed proxy; otherwise keep the current hardening and close the request as an intentional policy choice after maintainer review.

## Security Review

Status: needs_attention

Summary: The requested behavior is security-sensitive because rendered chat content would be able to trigger browser fetches to arbitrary remote image hosts.

Concerns:

- **[medium] Remote images would trigger browser fetches:** `docs/web/control-ui.md:348`
  - body: Allowing http(s) markdown image URLs inline would let model- or user-supplied chat content cause the operator browser to request third-party hosts, conflicting with the documented Control UI no-remote-fetch policy unless maintainers add an explicit policy control.
  - confidence: 0.93
- **[medium] Current CSP intentionally excludes remote image sources:** `src/gateway/control-ui-csp.ts:47`
  - body: The Control UI CSP currently omits https from img-src. A renderer-only change would either fail at runtime or pressure maintainers to broaden CSP without a reviewed remote-media safety model.
  - confidence: 0.91
