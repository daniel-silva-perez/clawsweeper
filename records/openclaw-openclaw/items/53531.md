## Summary

Keep open. Current main still carries compaction through a single resolved authProfileId, while the normal run path has the auth-profile ordering, cooldown skipping, failure marking, and profile advancement machinery that compaction does not reuse. The idempotent retry boundary raised in the discussion is also not present across the direct and context-engine compaction entry points.

## What This Changes

Compaction should retry or fail over across same-provider auth profiles when the active compaction profile is rate-limited, unavailable, or cooling down.

## Best Possible Solution

Add shared compaction retry support that captures one transcript/checkpoint snapshot, orders same-provider auth-profile candidates with the normal auth semantics, skips or marks cooldown/failover states consistently, retries only failover-classified compaction failures, and commits a successful summary, checkpoint, hooks, and transcript rotation at most once.
