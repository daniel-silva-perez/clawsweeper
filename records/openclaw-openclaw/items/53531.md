## Summary

Keep this issue open. Current main still has compaction paths built around one resolved authProfileId per compaction request, while the normal turn loop has separate ordered profile-candidate selection, cooldown skipping, failure marking, and profile advancement. The idempotency requirement raised in discussion also remains unimplemented for compaction retries.

## What This Changes

The issue requests same-provider auth-profile rotation and failover for compaction when the initially selected auth profile is rate-limited or unavailable.

## Best Possible Solution

Add a shared compaction auth-profile retry design that reuses the normal same-provider profile ordering, cooldown skipping/probing, failure classification, and failure marking, while compacting one captured transcript snapshot and committing only one successful summary/checkpoint/transcript rotation across all compaction entry points.
