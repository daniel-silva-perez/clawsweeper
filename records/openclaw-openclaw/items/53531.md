## Summary

Keep open. Current main still resolves compaction to one effective auth profile and returns or throws from that compaction attempt; the normal run path has the ordered same-provider profile candidates, cooldown skipping, failure marking, and profile advancement that the issue is asking compaction to reuse.

## What This Changes

Compaction should retry eligible same-provider auth profiles when the active profile is rate-limited or unavailable, while preserving an apply-once compaction result.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with the existing auth semantics, retries only classified failover failures, updates cooldown/failure state consistently, and commits exactly one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes. A focused harness can reproduce this by configuring two same-provider profiles, making the first compaction request fail with a rate-limit/auth failover error, and asserting current main does not retry the compaction against the second profile.

## Solution Assessment

Yes on direction, unclear on final design. Same-provider profile failover is the narrow maintainable scope, but it should be implemented as an idempotent compaction job rather than by wrapping the existing side-effectful `activeSession.compact()` call in a blind retry loop.
