## Summary

Keep open. Current main still resolves compaction to a single effective auth profile and returns or throws from that compaction attempt, while the normal turn path has ordered same-provider candidates, cooldown skipping, profile advancement, and failure marking. Recent timeout-recovery tests show profile rotation around repeated run attempts, but they do not implement retry/failover inside a compaction job when the compaction request itself is rate-limited or auth-fails.

## What This Changes

The issue requests same-provider auth-profile rotation and failover during compaction when the currently selected profile is rate-limited or unavailable.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles using the existing auth semantics, retries only classified failover failures, records cooldown/failure state consistently, and commits exactly one successful compaction result across direct and context-engine paths.
