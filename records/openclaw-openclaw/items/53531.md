## Summary

Keep open. Current main still resolves compaction around a single authProfileId per request and does not share the normal turn-loop profile rotation, cooldown skipping, failure marking, and profile advancement path inside the compaction operation. The idempotent retry boundary raised in the discussion is also still missing.

## What This Changes

Request compaction to retry same-provider auth profiles when the current profile is rate-limited or unavailable, while preserving at-most-one compaction semantics.

## Best Possible Solution

Add a shared compaction retry design that reuses same-provider auth-profile ordering, cooldown checks, failure classification, and failure marking from normal runs, while treating compaction as one captured job whose successful summary/checkpoint/transcript rotation can be committed only once across all entry points.
