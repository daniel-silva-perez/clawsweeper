## Summary

Keep this issue open. Current main still treats compaction as a single-profile operation: direct/manual compaction resolves one authProfileId, gets one credential, and performs one compact call. Queued/context-engine and CLI compaction forward a single runtimeContext rather than the normal run loop's ordered auth-profile candidate/failure/cooldown machinery. The issue discussion's idempotency concern also remains unresolved for any future retry design.

## Best Possible Solution

Keep this issue open. The best fix is to extract/share the normal same-provider auth-profile ordering, cooldown skipping/probing, failure classification, and failure marking for compaction; snapshot the transcript once; retry eligible same-provider profiles against that exact snapshot; and commit exactly one successful summary/checkpoint/transcript rotation. Add focused coverage for manual compaction, queued/context-engine delegation, overflow recovery, timeout recovery, CLI budget compaction, and the partial-failure idempotency case raised in discussion.

## Risks / Open Questions

- A naive retry loop could double-compact or apply a summary to a transcript that changed after the first attempt; the fix needs a stable snapshot/attempt boundary and at-most-one commit semantics.
- Manual /compact, queued/context-engine compaction, overflow recovery, timeout recovery, and CLI budget compaction enter through different paths, so a narrow fix could leave some reported surfaces on the current single-profile behavior.
