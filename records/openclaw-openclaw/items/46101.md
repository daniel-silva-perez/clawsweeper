## Summary

Keep open. Current main still implements the mixed heartbeat scoping rule that disables agents without explicit `agents.list[].heartbeat` blocks once any listed agent has one, and the doctor preview-warning path still has no targeted diagnostic for that footgun. The broader inheritance policy remains tracked separately in #41879, but this issue is a narrow diagnostics request that can be fixed without changing scheduler semantics.

## What This Changes

Add an `openclaw doctor` or startup diagnostic warning when `agents.defaults.heartbeat` is configured but explicit per-agent heartbeat blocks cause listed agents without `heartbeat` blocks to be skipped.

## Best Possible Solution

Add a narrow, non-mutating doctor config-health warning that detects mixed `agents.defaults.heartbeat` plus explicit per-agent heartbeat blocks and names listed agents without explicit heartbeat blocks as not scheduled under the current rule. Keep scheduler inheritance changes out of this issue and leave that product decision to #41879.

## Reproduction Assessment

Yes. Current source and checked-in tests reproduce the core configuration: `agents.defaults.heartbeat` plus `main` without a heartbeat block and `ops` with one makes `main` disabled and `ops` enabled.

## Solution Assessment

Yes. A doctor warning is the narrowest maintainable fix for this issue because current docs and tests treat the scoping rule as intentional; changing heartbeat inheritance belongs in #41879 instead.
