## Summary

Keep this PR open. Current main still lacks the proposed origin-aware doctor auto-enable guard, but the patch as written can leave non-bundled channel owners without the explicit trust that current startup and channel planning require, so it needs maintainer revision rather than cleanup closure.

## What This Changes

The PR changes doctor plugin auto-enable logic to skip writing `plugins.entries` for global or workspace manifest-channel plugins, extends the test registry helper with plugin origins, and adds global, workspace, and bundled channel test cases.

## Best Possible Solution

Keep the fix in core doctor/plugin activation logic, but revise it so doctor skips `plugins.entries` only when the non-bundled channel owner is already explicitly trusted and will actually be eligible at startup, or preserve trust while avoiding duplicate manifest registration. The landed version should cover global, workspace, bundled, startup-planning, and restrictive-allowlist cases.

## Reproduction Assessment

Yes. A focused reproduction can construct `applyPluginAutoEnable` with `channels.acme-chat` and a manifest record for `acme-chat` from `global` or `workspace`; current main writes `plugins.entries`/allowlist state, while the PR returns no changes, and current activation tests show that untrusted non-bundled owners are filtered from configured-channel planning.

## Solution Assessment

No. The proposed skip is narrower than a registry-level suppression, but it is not the best current fix because it uses discovery plus configured channel presence as a proxy for runtime trust; the safer fix must check or preserve explicit trust before avoiding duplicate materialization.
