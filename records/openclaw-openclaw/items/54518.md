## Summary

Keep open. Current main still treats each agent workspace as a default working directory with opt-in filesystem guards and optional sandboxing, not as a hard default confinement boundary; docs, config, implementation, and tests all show that outside-workspace reads/writes/edits remain allowed when `tools.fs.workspaceOnly` is unset or false, and runtime/plugin surfaces need broader policy coverage.

## What This Changes

The issue requests hard per-agent filesystem confinement so an agent cannot read or modify paths outside its own workspace.

## Best Possible Solution

Define a maintainer-approved per-agent confinement profile or onboarding option for agents like the PM/Feishu bot: make workspace-scoped filesystem access explicit, deny or sandbox runtime tools, cover plugin/MCP/media paths, surface the active workspace clearly to skills, and add regression coverage for cross-workspace reads/writes while preserving intentional trusted-host workflows.
