## Summary

Current main still does not implement hard per-agent workspace confinement by default. The reported behavior remains consistent with current docs and code: an agent workspace is the default working directory plus optional filesystem guards, while unrestricted host filesystem access remains the default unless `tools.fs.workspaceOnly` or sandboxing is explicitly enabled. Runtime surfaces such as `exec`, `process`, plugin tools, MCP/bundle tools, media loading, and elevated mode are separate policy domains, so the shipped opt-in guards do not fully satisfy the issue’s requested boundary.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. A complete fix should define a per-agent confinement profile or onboarding option that enables workspace-scoped filesystem tools, constrains or sandboxes runtime tools such as `exec` and `process`, covers plugin/MCP/media path surfaces, updates skill guidance to use the active per-agent workspace instead of fixed `.openclaw/workspace` paths, and adds regression coverage for per-agent workspace boundaries while preserving intentional trusted-host workflows.
