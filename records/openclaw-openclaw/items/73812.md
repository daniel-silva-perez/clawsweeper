## Summary

Keep this PR open. Current main still lacks the visible restart-continuation reply override and per-request explicit-restart deferral timeout, so the PR is useful, but the proposed timeout fix is incomplete for coalesced scheduled restarts and the linked #74424 continuation-coalescing case still needs an explicit maintainer decision or follow-up.

## What This Changes

The PR adds automatic source reply delivery for restart-sentinel continuation turns, passes a 30-second deferral timeout from explicit gateway and chat restart callers, and adds regression coverage for group-topic continuations and deferral timeouts.

## Best Possible Solution

Revise the PR so restart continuations explicitly request automatic source delivery, explicit restart timeout policy is centralized, pending restart timeout state can be updated by coalesced explicit restarts, and the continuation coalescing contract is either fixed here or clearly left to #74424 with truthful tool responses.

## Reproduction Assessment

Yes. The PR-specific blocker can be reproduced with fake timers by scheduling a pending restart without a per-request timeout, disabling the runtime deferral timeout, then coalescing an explicit restart with a timeout while the pending count never drains. The linked continuation loss can be reproduced by calling `gateway.restart` with `continuationMessage` while an unconsumed restart signal is already in flight.

## Solution Assessment

No as submitted. The automatic source-delivery override is narrow and maintainable, but the timeout needs to live in pending restart state rather than the first call's captured `opts`, and the explicit restart continuation coalescing contract needs a deliberate fix or scoping decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate deferral timeouts through coalesced restarts:** `src/infra/restart.ts:739-746`
  - body: The new timeout is read from the `opts` captured by the first scheduled restart. If a later explicit restart with `deferralTimeoutMs: 30_000` coalesces onto an already pending restart that had no timeout while the runtime timeout is disabled, the pending closure still waits indefinitely. Store the pending timeout with restart state and update it when coalescing.
  - confidence: 0.89
- **[P3] Share the explicit restart timeout constant:** `src/agents/tools/gateway-tool.ts:24-25`
  - body: The same `30_000` timeout is defined independently in the gateway tool and chat restart command paths. These paths need one explicit-restart guarantee, so export one named constant from the restart module and import it in both callers to avoid silent drift.
  - confidence: 0.78
