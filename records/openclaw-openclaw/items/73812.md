## Summary

Keep open. The PR is addressing real current-main behavior, and the automatic source-delivery override is the right direction for Telegram-style group/topic restart continuations, but the timeout change still has a coalesced-restart ordering hole and the user-facing fix lacks the required changelog entry.

## What This Changes

The PR forces restart-sentinel continuation turns to request automatic source reply delivery, adds a 30-second per-request deferral timeout to explicit gateway restart callers, and adds focused regression tests.

## Best Possible Solution

Revise the PR so restart-sentinel continuations explicitly request automatic source delivery, explicit restart deferral timeout state is centralized in the restart module and updated when scheduled restarts coalesce, and the final patch includes focused coverage for group-topic visibility, direct per-request timeout, and coalesced scheduled-restart timeout ordering.

## Reproduction Assessment

Yes. Current main can be checked by dispatching a group/topic restart continuation and observing no requested source reply mode, and the PR-specific timeout gap can be reproduced by scheduling a pending restart without a per-request timeout, then coalescing an explicit restart with a timeout while pending work never drains.

## Solution Assessment

No as submitted. The source-delivery override is narrow and maintainable, but the timeout fix should keep timeout state with pending restart state instead of capturing the first call’s options, and the duplicated explicit timeout constant should be shared.
