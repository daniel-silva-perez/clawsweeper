## Summary

Keep open. Current main still lacks both requested runtime changes, and the PR is useful but not merge-ready because the new per-request deferral timeout is captured only by the first scheduled restart and can be lost when an explicit restart coalesces onto an existing pending restart.

## What This Changes

The PR branch adds automatic reply delivery for restart-sentinel continuation turns, adds a 30-second deferral timeout to explicit gateway restart callers, and extends restart continuation and deferral tests.

## Best Possible Solution

Revise this PR so restart-sentinel continuations explicitly request automatic source delivery, while explicit restart deferral timeout state is centralized and stored with pending restart state so coalesced explicit restarts can update the pending timeout. The finished change should include regression coverage for group-topic continuation visibility, direct per-request timeout, and coalesced scheduled explicit-restart ordering.

## Reproduction Assessment

Yes. The current-main behavior can be checked by dispatching a restart continuation from a group/topic session and observing it has no requested source reply mode, then by setting `gateway.reload.deferralTimeoutMs: 0` with a nonzero pending deferral count. The PR-specific gap can be reproduced by scheduling a pending restart without a per-request timeout, then coalescing an explicit restart with a timeout while the pending count never drains.

## Solution Assessment

No as submitted. The automatic source delivery part is a narrow maintainable fix, but the timeout fix should keep per-request timeout state alongside pending restart state and share the explicit timeout constant instead of capturing duplicated caller constants in separate files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate deferral timeouts through coalesced restarts:** `src/infra/restart.ts:739-746`
  - body: The new timeout is read from the `opts` captured by the first scheduled restart. If a later explicit restart with `deferralTimeoutMs: 30_000` coalesces onto an already-pending restart that had no timeout, the coalesced branch only updates emit hooks and the pending closure still waits indefinitely when runtime config disables the timeout. Store the pending timeout in restart state and update it when coalescing.
  - confidence: 0.89
- **[P3] Share the explicit restart timeout constant:** `src/agents/tools/gateway-tool.ts:24-25`
  - body: The same `30_000` timeout is added separately in the gateway tool and chat restart command paths. These paths need the same explicit-restart guarantee, so exporting one named constant from the restart module avoids silent drift between tool and chat restarts.
  - confidence: 0.78
