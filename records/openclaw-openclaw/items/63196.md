## Summary

Keep open. Current main still only applies durable cron run-log/job-state recovery to active cron task rows, so the reported terminal `lost` row can continue to audit as `backing session missing`; the newly linked open PR is the right implementation path but has not merged.

## What This Changes

The issue reports that task audit and maintenance should reconcile a terminal `lost` cron task when durable cron run history proves the same execution finished successfully.

## Best Possible Solution

Ship a narrow task-maintenance reconciliation path that recovers terminal `lost` cron rows only when the row has a parseable `cron:<jobId>:<startedAt>` execution id and durable run log or job state proves that exact execution reached a terminal outcome.

## Reproduction Assessment

Yes. A source-level reproduction is a `runtime: "cron"`, `status: "lost"` task with matching durable finished run history; current main skips durable recovery because the row is not active, then audit still reports `lost`.

## Solution Assessment

Yes. The best fix direction is narrow durable recovery for terminal `lost` cron rows constrained to the backing-session false-positive path and exact execution matching; job-id-only inference would be too broad.
