## Summary

Current main still appears to leave the reported terminal cron lost-row case unresolved. Durable cron recovery exists and matches exact finished cron runs, but it is gated to active tasks only, so an already-terminal `lost` cron task can still be reported by audit and only cleanup-stamped/pruned by maintenance instead of being recovered from matching cron run history.

## Best Possible Solution

Keep this issue open and add a narrow task-maintenance/audit fix: when a terminal `lost` cron task has exact run identity and durable cron run logs or cron job state prove that same run finished, recover the task to the matching terminal status. Do not infer recovery from job id alone when exact identity is missing or mismatched.

## Clownfish Work Prompt

Fix the cron task-maintenance gap from this issue: `openclaw tasks audit` can keep reporting a terminal cron task as `lost` / `backing session missing` even though durable cron run history for the same exact run says it finished `ok`.

Expected boundary:
- Work primarily in `src/tasks/task-registry.maintenance.ts` and tests.
- Extend durable cron recovery so it can reconcile terminal `status: "lost"` cron task rows only when the task has exact identity: `runId` parses as `cron:<jobId>:<startedAt>` and any `sourceId` matches that job id.
- Reuse the existing run-log/job-state recovery semantics that match the same `jobId` and `runAtMs` / `lastRunAtMs`; map terminal cron status the same way current active recovery does.
- Make `reconcileTaskRecordForOperatorInspection()`, `previewTaskRegistryMaintenance()`, and `runTaskRegistryMaintenance()` agree: a recoverable terminal lost cron row should project/apply the recovered terminal state and no longer appear as a lost audit error.
- Do not guess recovery from `sourceId` or job id alone when exact run identity is absent or mismatched; those rows should keep existing cleanupAfter/prune behavior.
- Do not change the broad terminal-status precedence rules in `src/tasks/task-registry.ts` unless a test proves it is necessary for this narrow cron maintenance path.
- Add focused regression coverage, preferably in `src/tasks/task-registry.maintenance.issue-60299.test.ts` or a new issue-specific maintenance test file, for a terminal lost cron row recovered from a matching finished run log and for a no-exact-identity lost row that is not recovered.
- Add/update user-facing changelog under the active version with `Fixes #63196` and `Thanks @andyylin`; update docs only if the behavior wording needs clarification.

Related refs: this issue, #60299, #71963.
