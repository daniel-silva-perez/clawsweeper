## Summary

Keep this open. Current main still recovers durable cron run-log/job-state outcomes only for active cron task rows, while the reported record is already terminal `lost`; that means maintenance can stamp or eventually prune the row, but it does not reconcile it back to the matching successful cron execution.

## What This Changes

The issue asks task audit and maintenance to reconcile a terminal `lost` cron task row when durable cron run history shows the same logical run finished successfully.

## Best Possible Solution

Add a narrow task-maintenance reconciliation path for terminal `lost` cron records only when durable cron run logs or job state prove the exact same cron execution reached a terminal result, then update the task ledger to the matching terminal status and clear stale lost metadata without broadening cron liveness inference.
