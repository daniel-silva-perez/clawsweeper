## Summary

Keep this open. Current `main` has durable cron recovery for active cron tasks, but the reported row is already terminal `lost`; that path is still excluded before reconciliation can use the matching cron run-log/job-state evidence, while audit continues to report every `lost` row.

## What This Changes

The issue asks task audit and maintenance to reconcile a terminal cron task row marked `lost` when durable cron run history proves the same logical run finished successfully.

## Best Possible Solution

Add a narrow task-maintenance reconciliation path for terminal `lost` cron rows that have exact cron execution identity and durable cron run-log or cron job state proof of the same run's terminal outcome, then recover the task to the matching terminal status and cleanup metadata without broadening job-id-only inference.

## Repair Work Prompt

Fix the task maintenance/audit bug from openclaw/openclaw#63196. Current `main` recovers durable cron outcomes only for active `queued`/`running` tasks; terminal cron rows already marked `lost` still show in `openclaw tasks audit` as `backing session missing` even when cron run history proves the exact run finished. Add regression coverage in `src/tasks/task-registry.maintenance.issue-60299.test.ts` for an already-terminal `lost` cron task with matching finished `ok` run-log proof being recovered to `succeeded`, plus a negative case for missing or mismatched exact run identity. Add/update audit coverage if needed so `openclaw tasks audit` no longer reports the recovered row. Update docs only if the behavior text changes, and add a single-line changelog fix entry crediting `Thanks @andyylin`.
