## Summary

Current main has a partial fix for the cron/task mismatch: active cron task records are recovered from durable cron run logs or job state before maintenance marks them lost. The exact reported terminal-row case remains open because durable recovery explicitly skips non-active tasks, so a cron task already stored as `lost` can still be reported by audit until cleanup stamping/pruning handles it.

## Best Possible Solution

Keep this issue open and add a narrow maintenance/audit fix for cron rows already stored as `lost`: when a lost cron task has an exact parseable cron execution runId and durable cron run history or job state proves that same run finished, recover it to the corresponding terminal status instead of continuing to report `backing session missing`. Preserve retention cleanup for rows without exact run identity.

## Clownfish Work Prompt

Fix the remaining cron task audit mismatch for already-terminal lost rows. Current main recovers active cron tasks from durable run logs/job state before marking them lost, but `resolveDurableCronTaskRecovery()` returns early for non-active tasks, so a cron task already stored as `status: "lost"` can still be reported as `backing session missing` even when the matching cron run history says `finished` + `ok`.

Implement a narrow fix in `src/tasks/task-registry.maintenance.ts`: allow durable cron recovery for cron tasks that are active or already `lost`, but only when `runId` parses exactly as `cron:<jobId>:<startedAt>` and matches `sourceId` plus persisted cron run log/job state for that same execution. Do not infer from `sourceId` alone. Preserve behavior for non-cron tasks, tasks without a parseable runId, cancelled/failed/timed_out terminal states, and rows with no matching durable terminal evidence. If recovered, use the existing `markTaskRecovered`/projection shape so maintenance and inspect/audit agree.

Add regression coverage in `src/tasks/task-registry.maintenance.issue-60299.test.ts` for a `status: "lost"`, `error: "backing session missing"` cron task with matching `finished` + `ok` run log being projected/recovered to `succeeded` and no longer counted as a lost audit error. Add a negative test for a lost cron task with missing or mismatched runId staying lost. Update docs/changelog only if the behavior change is user-visible per repo policy; do not broaden task terminal semantics beyond exact cron durable-history recovery.
