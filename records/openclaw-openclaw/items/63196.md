## Summary

Current main still appears to miss the reported terminal cron lost-row case. Durable cron recovery exists and uses exact cron run identity, but `resolveDurableCronTaskRecovery()` only runs for `queued` or `running` tasks, so an already-terminal `lost` cron task can still fall through to cleanup stamping/pruning and continue to be reported by audit.

## What This Changes

Reconcile terminal `lost` cron task rows whose exact cron run history shows the run finished successfully so `tasks audit` and `tasks maintenance --apply` stop reporting stale `backing session missing` errors.

## Best Possible Solution

Keep this issue open and add a narrow task-maintenance/audit fix: when a terminal `lost` cron task has exact run identity and durable cron run logs or cron job state prove that same run finished, recover the task to the matching terminal status. Do not infer recovery from job id alone when exact identity is missing or mismatched.

## Repair Work Prompt

Implement a narrow fix for openclaw/openclaw#63196. Current main recovers cron tasks from durable run logs/job state only while they are `queued` or `running`; a terminal `lost` cron task with `runId` shaped as `cron:<jobId>:<startedAt>` and a matching finished cron run can still be reported by `openclaw tasks audit` as `backing session missing`. Extend task maintenance/inspection to recover terminal synthetic `lost` cron rows only when exact cron execution identity matches durable run evidence: `sourceId`/job id must match and the run log `runAtMs` or job state `lastRunAtMs` must equal the parsed startedAt. Reuse the existing status mapping (`ok`/`skipped` -> `succeeded`, error timeout -> `timed_out`, other error -> `failed`). Do not infer recovery from job id alone, do not weaken non-cron terminal precedence, and do not change general task finalization in `src/tasks/task-registry.ts`. Add regression coverage for a `lost` cron row with a matching finished ok run log proving `reconcileInspectableTasks()`, `previewTaskRegistryMaintenance()`, and `runTaskRegistryMaintenance()` recover it and audit no longer emits `lost`; add mismatch/no-run coverage that remains `lost`. Update docs/changelog only if the behavior change is user-facing; any changelog entry should credit `Thanks @andyylin`.
