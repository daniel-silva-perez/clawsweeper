## Summary

Keep this open. Current main still only applies durable cron run-log/job-state recovery to active cron tasks, while the reported row is already terminal `lost`; maintenance therefore will not reconcile that terminal row back to the matching successful cron result, and audit still reports `lost` rows.

## What This Changes

The issue requests that cron task audit/maintenance reconcile or clear stale terminal `lost` cron task rows when cron run history proves the same logical run finished successfully.

## Best Possible Solution

Add a narrow maintenance reconciliation path for terminal `lost` cron rows when durable cron run logs or job state prove the same parsed cron execution reached a terminal outcome, then recover the task to the matching terminal status and cleanup metadata without broadening inference beyond exact run identity.

## Repair Work Prompt

Fix the task-maintenance bug reported in openclaw/openclaw#63196. Current main recovers active cron tasks from durable run logs/job state, but terminal `lost` cron rows are excluded by the `isActiveTask()` guard and still surface as `backing session missing` audit findings. Add a narrow reconciliation path for terminal `lost` cron rows in `src/tasks/task-registry.maintenance.ts` that only recovers when the task has an exact parsed cron execution id (`cron:<jobId>:<startedAt>`) and durable cron run-log or cron job state proof for that same run. Recover to the matching terminal status (`succeeded`, `failed`, or `timed_out`) and clean up stale lost metadata as appropriate. Do not recover by job id alone, do not weaken audit reporting for real lost rows, and do not introduce broad task pruning. Add/extend regression tests in `src/tasks/task-registry.maintenance.issue-60299.test.ts` for a terminal `lost` cron row with matching `finished`/`ok` run-log evidence, plus a negative case that refuses mismatched run identity. Add audit coverage only if behavior changes there. Add a single-line user-facing changelog fix entry with `Thanks @andyylin` if the PR changes shipped behavior.
