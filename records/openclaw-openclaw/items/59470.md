## Summary

Do not close. Current main still has a plausible unresolved plugin gateway-method response-path gap: plugin handlers are registered as side-effecting handlers, the WebSocket path only sends a response when `respond(...)` is called, and there is no WS fallback when a plugin handler completes without producing a response. That matches the reported symptom closely enough that this needs a fix or a clearer SDK contract/test before cleanup.

## Best Possible Solution

Keep the issue open and add a focused gateway/plugin regression test around a plugin method that completes without producing a response. Maintainers should then decide whether plugin gateway handlers should support returned payloads, or whether the WS path should fail fast with a clear `completed without a response` error and the SDK docs should explicitly require `respond(...)`. If the reporter's plugin did call `respond`, the next step is to capture the plugin source/logs and trace why the response frame was not delivered.
