## Summary

Keep open. Current `main` still treats plugin gateway methods as callback-style handlers: the server passes a `respond` function into the handler and ignores any returned value, so a plugin method can finish its side effect and return normally while the CLI receives no RPC response. A later gateway teardown-timeout fix is shipped, but it does not cover the no-response plugin-handler path described here.

## What This Changes

The issue reports that `openclaw gateway call` can time out for plugin-registered gateway methods even when the plugin method completes its work successfully.

## Best Possible Solution

Add a backward-compatible plugin gateway method contract: for plugin-registered handlers only, if the handler returns a non-`undefined` JSON-compatible value and has not already called `respond`, send that value as a successful RPC response. Keep explicit `respond` behavior unchanged, guard against double responses, update the SDK type/docs, and add a regression test for both returned payloads and explicit callback responses.

## Reproduction Assessment

Yes. A plugin that registers a gateway method whose handler returns `{ ok: true }` without calling `respond` will be awaited by `handleGatewayRequest`, but no response frame is sent, so the CLI waits until its gateway timeout path fires.

## Solution Assessment

No. Current `main` has a useful teardown-timeout fix for already-successful RPC responses, but it is not the best solution for this report because plugin handler return values are still ignored; the narrower maintainable fix is to either support returned payloads for plugin handlers or explicitly reject/document that contract.
