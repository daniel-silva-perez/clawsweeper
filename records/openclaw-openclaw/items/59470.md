## Summary

Keep open. Current main still treats plugin gateway methods as callback-style handlers: the WebSocket layer only sends a response through the injected `respond` function, while `handleGatewayRequest` awaits plugin handlers without consuming returned values. That matches the report's success-side-effect plus CLI timeout shape, and no current code/docs/test evidence shows this has been fixed.

## What This Changes

The issue asks for `openclaw gateway call <plugin.method>` to return a successful plugin method result instead of timing out after the handler completes its work.

## Best Possible Solution

Add a backward-compatible plugin gateway method contract: for plugin-registered handlers only, if the handler returns a non-`undefined` JSON-compatible value and has not already called `respond`, send it as a successful RPC response. Preserve explicit `respond` behavior, prevent double responses, keep core gateway method semantics unchanged, and document the supported handler patterns.

## Reproduction Assessment

Yes. A plugin that registers a gateway method whose handler returns `{ ok: true }` without calling `respond(...)` will be awaited by current `handleGatewayRequest`, but no response frame is sent, so `openclaw gateway call` waits until its timeout path fires.

## Solution Assessment

No. Current main has timeout and teardown hardening for requests that do receive responses, but it does not solve plugin handlers that complete by returning a value; the narrow maintainable fix is in plugin gateway handler dispatch and SDK docs/tests, not in the generic CLI timeout budget.
