## Summary

Keep open. Current main still does not implement `failureAlert.mode: "agent-turn"`, so the PR is not obsolete, but the proposed patch has blocking correctness gaps: the CLI parser and gateway protocol still reject the new mode, the new wake path is untargeted, and the patch exposes `agent-turn` on `failureDestination` without a coherent dispatch path.

## What This Changes

The PR widens cron failure-alert and failure-destination mode types/config schemas to include `agent-turn`, adds a cron failure-alert branch that enqueues a system event and wakes heartbeat, and updates CLI help text.

## Best Possible Solution

Implement `failureAlert.mode: "agent-turn"` end-to-end on current main using the current cron/gateway/protocol layout, update CLI parsing plus gateway validation, target the wake to the failing job’s agent/session, and leave `failureDestination` as announce/webhook unless maintainers explicitly choose and test that broader API.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: `openclaw cron edit --failure-alert-mode agent-turn` still hits the parser rejection path, gateway cron schemas still reject the value, and the PR’s wake call lacks the target fields the heartbeat runner uses to avoid global runs.

## Solution Assessment

No. The current PR is not the best way to solve the issue because it widens public types/config before all entry points and dispatch paths agree; the safer fix is a narrower failure-alert-only implementation with protocol, CLI, docs, and regression tests kept in sync.

  - body: The help text advertises `agent-turn`, but the action handler still throws unless the mode is `announce` or `webhook`. The usage shown in the PR body therefore fails locally before `cron.update` is sent.
  - confidence: 0.96

  - confidence: 0.93
- **[P2] Target the heartbeat wake to the failing job:** `src/cron/service/timer.ts:355-356`
  - body: The new branch enqueues the event for the job agent but wakes heartbeat without `agentId` or `sessionKey`. Untargeted wakes run the non-targeted heartbeat path across configured agents, so one cron failure can wake unrelated agents and incur extra work/cost.
  - confidence: 0.9
- **[P2] Do not treat agent-turn destinations as unset:** `src/cron/delivery.ts:205-208`
  - body: Returning `null` for `failureDestination.mode: "agent-turn"` makes the gateway treat the explicit failure destination as absent and continue to primary-delivery fallback behavior. Either remove `agent-turn` from failureDestination or implement that path end-to-end.
  - confidence: 0.88
