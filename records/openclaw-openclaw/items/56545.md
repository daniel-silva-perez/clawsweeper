## Summary

Keep open. Current main does not implement `failureAlert.mode: "agent-turn"`, and this PR is useful source work but still incomplete: it advertises the CLI mode without accepting it, leaves Gateway and agent-tool schemas on `announce | webhook`, wakes heartbeat without a job target, widens failure-destination types without end-to-end dispatch semantics, and does not update the generated public config schema.

## What This Changes

The PR widens cron failure-alert and failure-destination mode types/config schemas to include `agent-turn`, adds a timer failure-alert branch that enqueues a system event and wakes heartbeat, and updates CLI help plus a gateway guard.

## Best Possible Solution

Implement `failureAlert.mode: "agent-turn"` end to end on current main across the cron service, CLI parser/help, Gateway protocol validators, agent tool schema, config types/schema/generated schema, docs/tests, and heartbeat targeting. Keep `delivery.failureDestination.mode` limited to `announce | webhook` unless maintainers explicitly choose to broaden that API and add matching dispatch behavior.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main and the PR diff show the advertised CLI value is rejected, Gateway schemas still reject `agent-turn`, and the PR wake call omits the target fields that the heartbeat runner uses to avoid broad runs.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best mergeable fix as-is because it widens public failure-destination/config types before all protocol, tool, generated-schema, and dispatch paths agree; a narrower current-main implementation is safer.
