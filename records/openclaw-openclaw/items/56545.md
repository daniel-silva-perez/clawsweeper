## Summary

Keep open. Current main still does not implement `failureAlert.mode: "agent-turn"`, so this PR is not obsolete, but the proposed patch is not safe to merge because it leaves CLI/RPC entry points inconsistent, wakes heartbeat without targeting the failing job, and broadens `failureDestination` without an end-to-end dispatch contract.

## What This Changes

The PR widens cron failure-alert and failure-destination mode types/config schemas to include `agent-turn`, adds a timer failure-alert branch that enqueues a system event and wakes heartbeat, and updates CLI help text plus a gateway guard.

## Best Possible Solution

Implement `failureAlert.mode: "agent-turn"` end-to-end on current main with the current cron service, CLI, config, Gateway protocol, and normalization layout; keep `failureDestination` announce/webhook-only unless maintainers explicitly choose a broader API and add matching dispatch tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main rejects `agent-turn` in the CLI parser and Gateway schemas, and the PR's heartbeat wake call lacks the target fields the heartbeat runner uses to avoid global runs.

## Solution Assessment

No. The PR is useful source work, but it is not the best current fix because it widens public failure-destination/config types before all entry points, runtime normalizers, protocol schemas, and dispatch paths agree; a narrower current-main implementation is safer.

  - body: The help text advertises `agent-turn`, but the action handler still throws unless the mode is `announce` or `webhook`. The usage shown in the PR body therefore fails before `cron.update` is sent.

  - confidence: 0.93
- **[P2] Target the heartbeat wake to the failing job:** `src/cron/service/timer.ts:355-356`
  - body: The new branch enqueues the event for the job agent but wakes heartbeat without `agentId` or `sessionKey`. Untargeted wakes run across configured agents, so one cron failure can wake unrelated agents and incur extra work/cost.
  - confidence: 0.9
- **[P2] Do not treat agent-turn destinations as unset:** `src/cron/delivery.ts:205-208`
  - body: Returning `null` for `failureDestination.mode: "agent-turn"` makes the gateway treat the explicit failure destination as absent and continue to primary-delivery fallback behavior. Either remove `agent-turn` from failureDestination or implement that path end-to-end.
  - confidence: 0.88
