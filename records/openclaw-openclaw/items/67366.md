## Summary

Current main still has the reported Telegram token replacement crash path: the configured Telegram setup flow still reaches the token prompt after choosing Modify settings, and the plaintext token helper still calls `.trim()` directly on the raw text prompt result. The two related fixing PRs, #67374 and #67733, are closed unmerged in the provided context.

## Best Possible Solution

Keep this issue open. Patch the shared setup text handling so no caller invokes `.trim()` on a nullish prompt result, at minimum in `promptSingleChannelToken` and the generic setup-wizard text-input path. Add a focused regression test for replacing an existing Telegram token after declining the keep prompt with a text prompter that resolves `undefined`, and verify the flow returns validation/default handling instead of throwing. The closed unmerged PRs #67374 and #67733 can be used as context, but the current fix should preserve cancel behavior and secret hygiene.

## Risks / Open Questions

- A fix should preserve cancel semantics: Clack cancel symbols should still raise `WizardCancelledError`; only nullish text results should be normalized or rejected before string methods run.
- Because this path handles Telegram bot tokens, regression tests should use fake token values and avoid logging or snapshotting secrets.
- The `WizardPrompter.text` type currently promises `string`, which can mask runtime or adapter nullish values; the fix should either harden callers or centralize normalization in the prompter boundary without weakening validation.
