## Summary

Keep open. Current main still uses exact-only silent-token matching, so a payload made only of repeated whitespace-separated `NO_REPLY` tokens is not classified as silent; the normalization path then strips only the trailing occurrence and can leave a visible `NO_REPLY`. The Quiubo plugin guard is useful mitigation and #69472 is useful prior art, but that PR is closed unmerged and neither resolves the shared core predicate on main.

## What This Changes

The issue requests that core silent-reply suppression handle doubled whitespace-separated `NO_REPLY` tokens without leaking them to channel plugins.

## Best Possible Solution

Update the shared silent-token predicate or normalization path so payloads made only of repeated whitespace-separated silent tokens are suppressed, while mixed-content replies and media-bearing payloads keep their existing behavior. Add focused regression coverage around repeated default and custom tokens plus the existing #19537 protections.

Current main treats only a single trimmed silent token as silent; `NO_REPLY\n\nNO_REPLY` falls through `isSilentReplyPayloadText`, then `normalizeReplyPayload` strips only one trailing token and can deliver visible `NO_REPLY`.
