## Summary

Keep open. Current main and the latest release still use exact-only silent-token matching, so doubled token-only payloads fall through the shared suppression predicate; normalization then strips one trailing token and can leave a visible `NO_REPLY`. The Quiubo plugin guard and the closed unmerged #69472 attempt are useful context, but neither resolves the core behavior on main.

## What This Changes

Suppress token-only replies containing repeated whitespace-separated `NO_REPLY` or silent-token values so channel plugins relying on core filtering do not receive visible control text.

## Best Possible Solution

Update the shared silent-token predicate or normalization path so payloads made only of repeated whitespace-separated silent tokens are suppressed, while mixed-content replies continue to be delivered without leaked control tokens. Add focused regression coverage around exact matching, repeated tokens, custom tokens, and the existing #19537 protections.

## Repair Work Prompt

Fix the remaining silent-token leak from this issue. In current main, `NO_REPLY\n\nNO_REPLY` is not treated as silent by `isSilentReplyText` / `isSilentReplyPayloadText`, and `normalizeReplyPayload` can strip one trailing token and leave a visible `NO_REPLY`. Update the shared core behavior in `src/auto-reply/tokens.ts` and, if needed, `src/auto-reply/reply/normalize-reply.ts` so token-only sequences made of repeated whitespace-separated silent tokens are suppressed.
