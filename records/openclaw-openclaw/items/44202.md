## Summary

Keep this open. Current main has partial QMD mitigations, but the central request remains unresolved: the built-in local GGUF embedding path still has no documented or typed CPU/GPU/Metal backend selector, local embed batches still call one node-llama-cpp embedding context concurrently, and the docs do not yet give official Apple Silicon Metal-safe guidance. The keepmyclaw comments are useful recovery context, but they are backup mitigation rather than an OpenClaw reliability fix.

## What This Changes

The issue requests a supported Apple Silicon path for local memory embeddings, including Metal/GPU guidance, safer failure behavior, and serialized or otherwise safe local embedding execution.

## Best Possible Solution

Keep this as the canonical Apple Silicon local-embedding stability tracker until OpenClaw has a documented macOS support matrix, validated node-llama-cpp CPU/GPU/backend configuration, bounded or serialized built-in GGUF embedding calls, and a gateway-safe failure boundary or fallback for native embedding crashes.

## Reproduction Assessment

Yes. The issue and follow-up comments provide a high-confidence reproduction shape using Apple Silicon, EmbeddingGemma GGUF, node-llama-cpp/ggml-metal, restart/shutdown or QMD semantic cleanup, and SIGABRT/non-zero exit behavior; source inspection confirms the requested controls are still missing.

## Solution Assessment

No. Current main has useful QMD mitigations and contextSize controls, and #60347 may address one configuration gap, but the complete solution needs coordinated config/schema, docs, node-llama-cpp backend validation, local batching behavior, and crash-boundary design.
