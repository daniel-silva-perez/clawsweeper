## Summary

Keep this open. Current main has useful QMD mitigations, but it still does not provide the requested supported Apple Silicon local-embedding path: built-in GGUF embeddings have no CPU/GPU/Metal backend selector, local batches still run concurrently against one node-llama-cpp context, and the public docs do not give Metal-safe or CPU-only guidance.

## What This Changes

The issue asks for reliable Apple Silicon local memory embeddings, including official Metal/GPU guidance or CPU-only/backend controls, safer batching, and gateway-safe handling of node-llama-cpp/ggml-metal crashes.

## Best Possible Solution

Keep this as the canonical Apple Silicon local-embedding stability tracker until OpenClaw has a documented macOS support matrix for built-in local and QMD embeddings, a validated node-llama-cpp CPU/GPU/backend config path, bounded or serialized built-in GGUF embedding calls, and a gateway-safe failure boundary or fallback for native embedding crashes.

## Reproduction Assessment

Yes. The issue and follow-up comments provide a high-confidence reproduction shape using Apple Silicon, EmbeddingGemma GGUF, node-llama-cpp/ggml-metal, gateway restart/shutdown or QMD semantic embed/vsearch cleanup, and SIGABRT/non-zero exit behavior; current-main inspection confirms the requested controls are still missing.

## Solution Assessment

No, current main is not the complete solution. The QMD defaults, locking, and backoff are useful mitigations, and #60347 may address one config gap, but the narrowest maintainable outcome needs coordinated config schema, docs, node-llama-cpp backend validation, built-in local batching changes, and crash-boundary design.
