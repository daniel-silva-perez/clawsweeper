## Summary

Keep open. Current main still lacks a supported Apple Silicon Metal-safe path for local GGUF memory embeddings: the built-in local provider still initializes node-llama-cpp inside the gateway process without a CPU/GPU/backend selector, local batch embedding still runs concurrent calls against one embedding context, and docs/config still do not define CPU-only or safe Metal guidance. Fresh reproduction comments and the open backend-selector PR show this is active product/runtime follow-up, not cleanup-ready.

## What This Changes

The issue requests official Apple Silicon guidance and safer behavior for local memory embeddings that can crash in the node-llama-cpp/ggml-metal path, including CPU/GPU selection, batch serialization, and gateway failure isolation.

## Best Possible Solution

Keep this as the canonical Apple Silicon local-embedding stability tracker until OpenClaw has a documented macOS support matrix for built-in local and QMD embeddings, a validated node-llama-cpp CPU/GPU/backend config path, bounded or serialized local GGUF embedding calls, and a gateway-safe failure boundary or fallback for native embedding crashes.
