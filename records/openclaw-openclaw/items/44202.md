## Summary

Keep this issue open. Current main still does not provide the requested supported Apple Silicon local-embedding path: built-in local GGUF embeddings still initialize node-llama-cpp without a CPU/GPU/backend selector, local batch embedding still runs concurrent calls against one context, and the public memory docs/config schema still do not give Metal-safe or CPU-only guidance. The newer reproduction comments and open PR #60347 are useful context, but they do not make the issue cleanup-ready.

## What This Changes

The issue asks OpenClaw to make Apple Silicon local GGUF memory embeddings safer by documenting or exposing CPU-only/GPU backend guidance, avoiding unsafe batching, and preventing node-llama-cpp/ggml-metal failures from destabilizing the gateway or memory path.

## Best Possible Solution

Keep this as the canonical Apple Silicon local-embedding stability tracker until OpenClaw has a documented macOS support matrix for built-in local and QMD embeddings, a validated node-llama-cpp CPU/GPU/backend config path, bounded or serialized local GGUF embedding calls, and a gateway-safe failure boundary or fallback for native embedding crashes.

## Reproduction Assessment

Yes. The issue and later comments provide a concrete Apple Silicon reproduction shape using EmbeddingGemma GGUF, node-llama-cpp/ggml-metal, gateway restart or QMD embed/vsearch cleanup, and SIGABRT/non-zero exit behavior; current-main inspection confirms the affected local embedding path and missing config/docs guidance still exist.

## Solution Assessment

Unclear. The reported workarounds and PR #60347 address useful slices, but the best maintainable fix needs maintainer design across config schema, docs, QMD behavior, native-runtime failure handling, and batching semantics rather than a narrow cleanup close.
