## Summary

Keep open. Current main still has the same core gaps: built-in local GGUF embeddings initialize node-llama-cpp in the gateway process without a CPU/GPU/backend selector, batch embedding remains concurrent against one context, and the public config/docs still lack Apple Silicon Metal-safe or CPU-only guidance. The newer reproduction comments, the keepmyclaw crash-recovery workaround, and open PR #60347 are useful context, but they do not make this cleanup-ready.

## What This Changes

The issue requests official Apple Silicon guidance and safer behavior for local memory embeddings that can crash in the node-llama-cpp/ggml-metal path, including CPU/GPU selection, batch serialization, and gateway failure isolation.

## Best Possible Solution

Keep this as the canonical Apple Silicon local-embedding stability tracker until OpenClaw has a documented macOS support matrix for built-in local and QMD embeddings, a validated node-llama-cpp CPU/GPU/backend config path, bounded or serialized local GGUF embedding calls, and a gateway-safe failure boundary or fallback for native embedding crashes.

## Reproduction Assessment

Yes. The issue and later comments provide a concrete Apple Silicon reproduction shape using EmbeddingGemma GGUF, node-llama-cpp/ggml-metal, gateway restart or QMD embed/vsearch cleanup, and SIGABRT/non-zero exit behavior; current code inspection confirms the affected in-process local path still exists.

## Solution Assessment

Unclear. The reported workarounds and backup plugin reduce blast radius, and PR #60347 addresses one backend-selection slice, but the narrow maintainable solution still needs maintainer design across config, docs, concurrency, QMD guidance, upstream native behavior, and crash isolation.
