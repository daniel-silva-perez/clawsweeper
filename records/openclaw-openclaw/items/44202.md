## Summary

Keep this open. Current main has partial QMD mitigations, but it still does not provide the requested Apple Silicon Metal-safe local embedding path: the built-in local provider loads node-llama-cpp in-process without a GPU/backend selector, its local batch path still fans out concurrent getEmbeddingFor calls inside one embedding context, and the public memory docs do not document a supported CPU-only or safe Metal configuration. Fresh comments plus open #60347 show the issue is active rather than obsolete.

## What This Changes

The issue requests official guidance and safer behavior for Apple Silicon local memory embeddings that can crash in node-llama-cpp/ggml-metal, including CPU-only/backend config, safer batching, and gateway crash isolation.

## Best Possible Solution

Keep this as the canonical Apple Silicon local-embedding stability tracker until OpenClaw has a documented macOS support matrix for built-in local and QMD embeddings, a validated node-llama-cpp CPU/GPU/backend config path, bounded or serialized local GGUF embedding calls, and a gateway-safe failure boundary or fallback for native embedding crashes.
