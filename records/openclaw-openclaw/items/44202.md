## Summary

Keep this issue open. Current main still leaves the reported Apple Silicon local-embedding stability problem unresolved: the built-in local provider runs node-llama-cpp in-process, still parallelizes batch embeddings with Promise.all, exposes only modelPath/modelCacheDir/contextSize for local config, and the docs do not provide official Metal/GPU or CPU-only guidance. Recent April 2026 comments in the issue add fresh reproductions, so stale closure is not appropriate.

## Best Possible Solution

Keep this issue open as the canonical Apple Silicon local-embedding stability and guidance tracker. The best path is to land or replace the narrow GPU/backend config work, document the supported macOS CPU/GPU modes, serialize or otherwise constrain local GGUF embedding batches, and decide whether local native embedding failures need process isolation or another gateway-safe containment boundary.

## Evidence

- **In-process local provider:** The local embedding provider lazy-loads node-llama-cpp, creates a shared Llama embedding context, and serves queries/batches directly from that provider object. There is no process boundary that would contain a native ggml/Metal abort away from the gateway/plugin runtime.
