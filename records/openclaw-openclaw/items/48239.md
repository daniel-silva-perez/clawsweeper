## Summary

Keep open. Current main has real fixes for the stale-routing class: Telegram native approvals honor agent/session filters, use turn-source/account metadata, preserve Telegram topic origins, and send an origin-chat notice when `target: "dm"` routes the actual approval card to approver DMs. That likely covers the related wrong-DM reports such as #62718. This issue is broader, though: current `target: "dm"` behavior is still approver-ID based rather than an approval prompt in the originating agent/chat, the visible Telegram approval card does not identify the owning agent/session, and typed `/approve` submits only `{id, decision}` after sender authorization, without checking the command came from the pending request's origin session/chat.

## Best Possible Solution

Keep this open and narrow it to the remaining safety gap: make Telegram approval cards visibly include the owning agent/session or origin chat/account, and add an origin/session-aware guard or warning for typed `/approve` and callbacks when the pending request has Telegram turn-source/session metadata. Add regression coverage for a multi-agent same-approver setup with `agentFilter: ["main"]` and `target: "dm"`. If maintainers intentionally want current approver-DM semantics, update the docs and issue with that product boundary instead of closing this as fully implemented.
