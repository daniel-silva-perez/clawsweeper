## Summary

Keep open. Current main still lacks Feishu recall/delete and chat-history list support, so the PR is not obsolete, but the branch should not merge as-is because it introduces tool-schema, config, typecheck, and account-routing defects around a destructive Feishu message-management surface.

## What This Changes

The PR adds a Feishu `feishu_message` tool with list/recall/delete actions and a `tools.message` toggle, plus message tests and Feishu error-message localization across several helper modules.

## Best Possible Solution

Implement Feishu recall/delete/list through the existing shared channel message action surface when possible, or through a maintainer-approved separate Feishu tool with TypeBox/JSON Schema parameters, strict config schema/help/docs alignment, account-aware routing, conservative destructive-action enablement, and focused tests for schema, config, routing, and Feishu API call shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Update merged Feishu tools config for message:** `extensions/feishu/src/types.ts:98`
  - body: Adding `message?: boolean` makes `resolveAnyEnabledFeishuToolsConfig` return `Required<FeishuToolsConfig>`, but that helper's object literal still lacks `message`. This branch should fail typechecking once the changed type is compiled; add `message` to the merged default and OR it across accounts.
  - confidence: 0.91
- **[P1] Use a JSON Schema for feishu_message parameters:** `extensions/feishu/src/message.ts:152`
  - body: `parameters` is set to a Zod object, but OpenClaw tool registration and provider normalization expect a JSON Schema/TypeBox-shaped object. This can leave providers with an invalid function schema for `feishu_message`; convert the schema before registering it.
  - confidence: 0.9
- **[P2] Use the Feishu message-list query shape:** `extensions/feishu/src/message.ts:29-34`
  - body: The list call passes `chat_id` under `path`, while the existing Feishu runtime uses `params.container_id_type` and `params.container_id` for `client.im.message.list`. Without those query parameters, the new `list` action will not query the requested chat history correctly.
  - confidence: 0.82
- **[P2] Add the message toggle to the strict config schema:** `extensions/feishu/src/tools-config.ts:15`
  - body: The branch adds `tools.message` to the type/defaults, but `FeishuToolsConfigSchema` is strict and still omits `message`. A user setting `channels.feishu.tools.message` will be rejected as an unknown key, so the advertised toggle cannot be configured.
  - confidence: 0.89
- **[P2] Route message operations through the active account:** `extensions/feishu/src/message.ts:135-144`
  - body: The new tool binds enablement and the client to `accounts[0]`, so multi-account sessions can skip the tool when the first account disables it and can execute list/recall/delete against the wrong tenant. Follow the existing Feishu tool pattern: merge enablement across accounts and resolve the client from `ctx.agentAccountId` or explicit `accountId` per call.
  - confidence: 0.87
