## Summary

Keep open. Current `main` still does not implement Feishu message recall/delete or chat-history listing, so the PR is not obsolete; however, the proposed branch has concrete correctness defects in schema registration, config merging, API call shape, and multi-account routing, plus a destructive-action default that needs maintainer approval before merge.

## What This Changes

The PR adds a Feishu `feishu_message` tool with list, recall, and delete actions, a `tools.message` toggle, tests for the new schema, and localized Feishu error messages across several helper modules.

## Best Possible Solution

Implement Feishu recall/delete/history listing through the shared channel message action surface where possible, or through a maintainer-approved separate Feishu tool with JSON Schema/TypeBox parameters, strict config schema/help/docs alignment, active-account routing, conservative destructive-action enablement, and focused tests for schema, config parsing, account routing, and Feishu API request shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Update merged Feishu tools config for message:** `extensions/feishu/src/types.ts:98`
  - body: Adding `message?: boolean` makes `Required<FeishuToolsConfig>` include `message`, but `resolveAnyEnabledFeishuToolsConfig` still builds and returns an object without that field. This should fail typechecking after the branch is compiled and also prevents message enablement from being merged across accounts.
  - confidence: 0.91
- **[P1] Use a JSON Schema for feishu_message parameters:** `extensions/feishu/src/message.ts:152`
  - body: `parameters` is set to `FeishuMessageSchema`, but that value is a Zod object. OpenClaw's tool pipeline normalizes JSON-schema-shaped objects and existing Feishu tools use TypeBox schemas, so providers can receive an invalid function schema for this new tool.
  - confidence: 0.9
- **[P2] Use the Feishu message-list query shape:** `extensions/feishu/src/message.ts:29-34`
  - body: The list call passes `chat_id` under `path`, while current Feishu message listing uses `params.container_id_type` and `params.container_id`. Without the query container shape, the new `list` action will not query the requested chat history correctly.
  - confidence: 0.84
- **[P2] Add the message toggle to the strict config schema:** `extensions/feishu/src/tools-config.ts:15`
  - body: The branch adds `tools.message` to the type/defaults, but `FeishuToolsConfigSchema` is strict and still omits `message`. A user setting `channels.feishu.tools.message` would have the config rejected as an unknown key.
  - confidence: 0.89
- **[P2] Route message operations through the active account:** `extensions/feishu/src/message.ts:135-144`
  - body: The new tool binds enablement and the client to `accounts[0]`, so multi-account sessions can skip the tool when the first account disables it and can execute list/recall/delete against the wrong tenant. Follow the existing account-aware tool pattern using merged enablement and per-call account resolution.
  - confidence: 0.87
