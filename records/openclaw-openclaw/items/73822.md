## Summary

Keep open. Current main still does not implement SecretRef support for these channel phone-number and sender-identity fields, so the PR remains useful and is paired with the open feature request it closes. The latest PR head is not mergeable as-is because it lets any SecretRef satisfy the explicit wildcard guard for `dmPolicy="open"`, moving a public-DM opt-in into hidden secret material instead of keeping it visible in reviewed config.

## What This Changes

This PR adds SecretRef-capable phone-number and sender-identifier config fields, secret target collection, UI sensitivity hints, docs/changelog, generated metadata, and tests across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, plus a Telegram test-helper cleanup.

## Best Possible Solution

Keep the SecretRef feature direction, but preserve the visible public-DM opt-in contract: SecretRefs may count as configured allowlist entries for allowlist-style policies, while only a literal `"*"` in the reviewed config should satisfy `dmPolicy="open"`. Add a negative test for SecretRef-only open policy and keep the docs/security review aligned with that boundary.

## Reproduction Assessment

Yes. On PR head 7158d93d181951690edf9d102d4db80111230d84, `requireOpenAllowFrom` returns successfully when `allowFrom` contains only a SecretRef, and the added test at `src/config/zod-schema.secretref-allowfrom.test.ts:88` codifies that behavior; current main requires a literal `"*"`.

## Solution Assessment

No. The SecretRef support is a useful narrow direction, but using `params.allowFrom?.some(isSecretRef)` for the open-policy guard is not the safest solution; the safer change is to allow SecretRefs for phone allowlists while requiring a visible wildcard for public-open DM behavior.
