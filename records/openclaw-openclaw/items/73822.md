## Summary

Keep open. Current main does not implement SecretRef-capable phone-number channel fields, and this PR is useful work for the linked feature request, but the latest PR head still weakens the existing `dmPolicy="open"` guard by treating any SecretRef as the explicit wildcard opt-in.

## What This Changes

The PR adds SecretRef support, secret target discovery, UI sensitivity hints, docs, changelog, generated metadata, and tests for phone-number and sender-identity config fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin.

## Best Possible Solution

Keep the SecretRef support for phone-number and handle fields, but preserve the existing visible wildcard contract: `dmPolicy="open"` should require a literal `*` in reviewed config, while `dmPolicy="allowlist"` can accept SecretRef-only allowlists that resolve before runtime checks.

## Reproduction Assessment

Yes. On the PR head, `requireOpenAllowFrom` accepts `{ dmPolicy: "open", allowFrom: [envRef("WILDCARD")] }`; the added `zod-schema.secretref-allowfrom.test.ts` test at line 88 is the focused reproduction.

## Solution Assessment

No. The PR is directionally the right implementation for the feature request, but accepting any SecretRef as the open-policy opt-in is not the narrowest safe solution; the safer fix is to allow SecretRefs for allowlist entries while requiring a literal `*` for open policy.
