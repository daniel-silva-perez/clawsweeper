## Summary

Keep open. Current main still lacks the requested SecretRef coverage, and this PR is the active implementation for the still-open related feature request, but the latest PR head still weakens the explicit open-DM wildcard guard by letting any SecretRef satisfy it.

## What This Changes

The PR adds SecretRef support for phone-number and handle fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, updating schemas, runtime secret collection, plugin manifest paths, UI hints, docs, changelog, generated metadata, and tests.

## Best Possible Solution

Keep the SecretRef feature on this PR branch, but preserve the visible wildcard contract: SecretRefs can count as configured allowlist entries for allowlist policy, while only a literal resolved "*" in config should enable dmPolicy="open" public DM behavior.

## Reproduction Assessment

Yes. On PR head b6de50391d48664d9a974185120b1251353ebcd0, requireOpenAllowFrom accepts dmPolicy="open" with allowFrom containing only a SecretRef, and the new test codifies that path.

## Solution Assessment

No. The feature direction is useful, but the open-policy escape hatch is too broad; the safer narrow fix is to keep SecretRefs valid for phone-number allowlists while requiring a visible literal wildcard for open policy.
