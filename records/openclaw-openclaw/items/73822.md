## Summary

Keep open. Current main still lacks the requested SecretRef support for these phone-number and sender fields, and this PR is the active implementation for the linked feature request, but the branch has a blocking access-control regression in the shared `dmPolicy="open"` allowlist refinement.

## What This Changes

The PR adds SecretRef support for phone-number and handle-like config fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, including schemas, runtime secret resolution, UI hints, docs, generated metadata, changelog, and tests.

## Best Possible Solution

Preserve the shipped explicit-open safety contract by requiring a literal or resolved `*` for `dmPolicy="open"`, while allowing SecretRefs for allowlist-style phone-number fields; after that, re-run focused schema/secret-contract tests and the appropriate changed gate before landing the implementation for #56502.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require a wildcard before accepting open policy:** `src/config/zod-schema.core.ts:652`
  - body: `params.allowFrom?.some(isSecretRef)` lets any object SecretRef satisfy the `dmPolicy="open"` opt-in. If that ref resolves to a phone number, the schema accepts a config that lacks `*`, but runtime open policy allows everyone, bypassing the existing explicit-wildcard guard. Keep the escape hatch out of the open-policy check, or only accept a value that is known to resolve to `*`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR introduces an access-control concern in the shared open-policy allowlist validation; no supply-chain or dependency change concern was found in the diff context.

Concerns:

  - body: The changed open-policy refinement accepts any object SecretRef in `allowFrom` as if it were the explicit `*` opt-in, so a SecretRef-backed phone-number allowlist can accidentally validate a world-open DM policy.
