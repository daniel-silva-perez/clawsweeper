## Summary

Keep open. Current main still lacks the requested SecretRef support for these phone-number and handle fields, and the PR is a useful active implementation, but the current branch introduces an access-control regression by letting any SecretRef satisfy the explicit open-DM wildcard guard.

## What This Changes

The PR adds SecretRef-compatible schemas, runtime secret collection, plugin manifest metadata, UI hints, docs, changelog, generated metadata, and tests for phone-number and handle fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin.

## Best Possible Solution

Keep the SecretRef feature on this PR branch, but preserve the visible open-policy contract: SecretRefs may count as configured allowlist entries for allowlist mode, while only a literal "*" in reviewed config should enable dmPolicy="open" public DM behavior.

## Reproduction Assessment

Yes. On PR head b6de50391d48664d9a974185120b1251353ebcd0, requireOpenAllowFrom accepts dmPolicy="open" when allowFrom contains only a SecretRef, and the new test at src/config/zod-schema.secretref-allowfrom.test.ts:88 codifies that path.

## Solution Assessment

No. The feature direction is useful, but the current implementation is too broad for open policy; the safer narrow fix is to allow SecretRefs for phone-number allowlists while requiring a visible literal wildcard for public-open DM opt-in.
