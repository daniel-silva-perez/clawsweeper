## Summary

Keep open. Current main still does not implement SecretRef support for the requested channel phone-number fields, and this PR is the active implementation for the linked feature request, but the latest PR head still has a blocking access-control regression in the open-DM policy guard.

## What This Changes

The PR adds SecretRef support for phone-number and sender-identity config fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, updating schemas, runtime secret resolution, UI hints, docs, generated metadata, and tests.

## Best Possible Solution

Preserve the shipped explicit-open safety contract while landing the additive SecretRef support: allow SecretRefs for phone-number allowlist and default-target fields, resolve them before runtime checks, and require a literal or resolved '*' specifically for open DM policy.

## Reproduction Assessment

Yes. At PR head f7b20ab35f09098971435ef7302ce8b388f56de2, `requireOpenAllowFrom` returns success when `allowFrom` contains any SecretRef, and the added test at `src/config/zod-schema.secretref-allowfrom.test.ts:88` locks in that behavior.

## Solution Assessment

No. The overall SecretRef direction is a reasonable solution for PII in configs, but treating any SecretRef as the open-policy wildcard is not the narrowest safe fix; the open-policy guard should stay tied to a known '*' value.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require a literal wildcard before accepting open policy:** `src/config/zod-schema.core.ts:652`
  - body: `params.allowFrom?.some(isSecretRef)` treats any SecretRef as equivalent to `*` for `dmPolicy="open"`. A ref that resolves to a phone number now validates as world-open at runtime, bypassing the explicit wildcard opt-in; keep this guard tied to a literal or resolved `*`.
  - confidence: 0.88
