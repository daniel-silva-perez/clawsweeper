## Summary

Keep open for normal PR review. Current main has a credible Telegram group command regression: group policy can allow an explicitly configured group with no sender allowlist, while the text command gate still treats the empty command authorizer as unauthorized and drops `/status` before the command layer can reply. The PR is a narrow, plausible fix, but it changes auth-sensitive `CommandAuthorized` behavior and should get targeted regression coverage or maintainer sign-off before landing.

## What This Changes

The PR changes Telegram inbound body command gating so access-group command enforcement is disabled when the effective command allowlist has no entries, preventing group slash commands from being dropped solely because no sender allowlist is configured.

## Best Possible Solution

Land a narrowly tested Telegram fix that lets text slash commands continue after the Telegram group policy layer has already authorized the group/sender, while preserving explicit sender allowlists, `commands.allowFrom`, and owner-only command checks.

## Reproduction Assessment

Yes. The static current-main path is high confidence: an explicitly configured group can pass group-policy checks with no sender allowlist, but `/status` then hits `resolveControlCommandGate` with `useAccessGroups=true` and no configured authorizer, causing `resolveTelegramInboundBody` to return null.

## Solution Assessment

Yes, with a test expectation. The PR’s direction is narrow because group policy is enforced before this gate, but it should be validated with a Telegram regression test since the changed value also becomes downstream `CommandAuthorized`.
