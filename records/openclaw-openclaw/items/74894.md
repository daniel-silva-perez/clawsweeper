## Summary

Keep this PR open, but do not merge it as written. It targets a real Telegram group `/status` drop on current main, but the patch disables access-group command enforcement for every empty effective allowlist, including open-group configurations where current tests and docs keep text commands unauthorized.

## What This Changes

The PR changes Telegram inbound body command gating to skip access-group enforcement when the effective command allowlist has no entries.

## Best Possible Solution

Narrow the Telegram fix to the configured-group case that has already passed group-policy authorization, while preserving sender allowlists, `commands.allowFrom`, owner-only command checks, and the existing per-group-open denial guard. Add regression tests for explicit groups, wildcard groups, open groups, and configured sender allowlists.

## Reproduction Assessment

Yes. Static inspection of current main shows an explicitly configured group with no effective sender allowlist can pass group policy and then be dropped by `resolveControlCommandGate`; existing tests also reproduce the PR regression risk for open groups.

## Solution Assessment

No. Keying command enforcement on `allowForCommands.hasEntries` is too broad; a safer solution should pass an explicit group-policy authorization fact into the command gate or add a dedicated authorizer for configured `groups` entries only.
