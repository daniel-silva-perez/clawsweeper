## Summary

Keep this PR open. Current main still does not implement the central Slack `file_shared` fallback for large or multi-file uploads that bypass normal `message` delivery. The related discussion is useful: prior bot review identified real implementation concerns, and the later ClawSweeper comment correctly noted that the work must be ported from the removed `src/slack` tree into the current `extensions/slack` plugin layout.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a rebase or replacement port into `extensions/slack`: add and register a `file_shared` handler, subscribe to the bot event in setup/docs, hydrate the matching message with a timestamp-anchored bounded `conversations.history` lookup, dedupe retries and normal message delivery, preserve existing media/security checks, and add focused Slack plugin tests.

## Risks / Open Questions

- The PR cannot be merged directly because it targets the removed `src/slack` layout instead of `extensions/slack`.
- A correct implementation needs Slack setup/docs changes so installed apps subscribe to `file_shared`; otherwise the handler will never receive the fallback event.
- The port should dedupe Slack retries and normal `message`/`file_share` delivery, handle multi-file uploads, and avoid a sliding latest-N history lookup that can miss busy channels.
- The fallback reads channel history and may log channel/file identifiers; the port should keep that bounded and preserve existing Slack media, file download, and SSRF protections.
