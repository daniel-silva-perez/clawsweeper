## Summary

Keep this PR open. Current `main` still lacks the central Slack `file_shared` fallback: the Slack plugin registers `message` and `app_mention` plus other metadata events, but no `file_shared` handler, subscription, docs entry, or message-handler source path. The related #65299 context corroborates the same missing Socket Mode path, while closed #40286 was not merged and does not supersede this work. This PR remains a plausible implementation candidate, but it needs to be ported from the removed `src/slack` tree into the current `extensions/slack` plugin and tightened around history lookup, dedupe, setup/docs, tests, and security-preserving media handling.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a rebase or replacement port into `extensions/slack`: add and register a `file_shared` handler under `extensions/slack/src/monitor/events`, hydrate the matching Slack message using an `event_ts`-anchored bounded `conversations.history` lookup, dedupe Slack retries and normal `message`/`file_share` delivery, preserve existing Slack authorization and media protections, update `extensions/slack/src/setup-shared.ts` and https://docs.openclaw.ai/channels/slack for the `file_shared` bot event, and add focused Slack plugin tests for large-file, multi-file, duplicate-delivery, busy-channel, and app-mention paths.

## Risks / Open Questions

- The PR cannot merge as-is because it targets the removed `src/slack` layout instead of `extensions/slack`.
- A correct implementation needs an `event_ts`-anchored bounded history lookup and synchronous retry/dedupe handling so busy channels and duplicate Slack retries do not silently drop or double-process files.
- Setup docs and generated manifest must add the `file_shared` bot event, or users will not receive the fallback event even if the handler exists.
- Tests are still needed for large-file and multi-file fallback, duplicate `message`/`file_shared` delivery, Slack retry delivery, busy-channel lookup, and app-mention interaction.
- The new handler must preserve current Slack authorization and media protections, especially team/account mismatch drops, allowlists, mention gating, Slack-host URL validation, bearer-token handling, and channel/thread scope checks for file downloads.
