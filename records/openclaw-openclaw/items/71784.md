## Summary

Keep this issue open. Current main still has the reported live-query gap: `openclaw memory search` computes a single query embedding without the embedding retry loop, and the shared remote HTTP path for OpenAI-compatible/Gemini embeddings still performs one guarded fetch. The related batch/reindex issue and PR do not supersede this live-search failure, and a fresh 2026.4.26 retest in the discussion says the same class still reproduces.

## Best Possible Solution

Keep this issue open. A conservative fix should add a narrow transient transport-error classifier and bounded retry/backoff to the live query embedding path, either in `embedQueryWithTimeout()` or the shared single-embed remote HTTP helper, while preserving immediate failures for auth/config errors. Add regression coverage for the reported transport strings and coordinate with the batch/reindex work in #44166/#44167 rather than treating it as a duplicate.
