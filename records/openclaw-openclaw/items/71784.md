## Summary

Keep open. The report is recent, provider-diverse, and still matches current main: live builtin memory search obtains the query vector through a single remote embedding call, so the reported `fetch failed`/socket reset class can still abort `openclaw memory search`. The related open PR is scoped to batch indexing/retry behavior for #44166 and does not close this live-query surface.

## What This Changes

Make live memory-search query embeddings resilient to intermittent remote embedding fetch/socket failures across providers.

## Best Possible Solution

Add a bounded retry/backoff around live query embeddings, using the existing memory embedding retry helper or a small shared variant, and expand the classifier only for narrow transient transport/socket failures while preserving immediate failures for auth, invalid config, schema errors, and non-retryable HTTP responses.

## Reproduction Assessment

Yes. The issue provides a concrete remote-provider `openclaw memory search` loop and follow-up confirmations across OpenAI, Gemini, and OpenRouter; current-main source inspection shows that the live query path remains single-shot.

## Solution Assessment

Yes. A focused live-query retry is the narrowest maintainable fix for the observed user-visible failure; broader Undici pool tuning or a new config knob should wait for evidence that the scoped retry is insufficient.

Scope the change to the memory-core live query embedding path and shared retry classifier/helper as needed.
