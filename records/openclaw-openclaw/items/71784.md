## Summary

Keep open. Current main still sends live builtin memory search through a single query embedding call with only a timeout wrapper, while the existing retry helper and classifier are used for batch paths and still do not classify the reported fetch/Undici/TLS socket failures. The issue also has two later production-style confirmations after 2026.4.26, so this is still actionable and not superseded by the open batch/reindex PR.

## What This Changes

Live builtin memory search should tolerate transient remote embedding transport/socket failures instead of intermittently failing semantic queries when upstream providers are healthy.

## Best Possible Solution

Add a conservative, shared transient transport-error classifier and bounded retry/backoff for the live query embedding path used by builtin memory search, while preserving fail-fast behavior for auth, config, schema, and HTTP 4xx errors. Keep dispatcher keep-alive tuning and new config knobs as follow-up only if the narrow retry does not cover the confirmed failure class.

## Reproduction Assessment

Yes. The issue body provides concrete provider config and a repeatable CLI loop, and later comments confirm the same failure class on upgraded installs and another provider; static inspection shows current main still has the same single-shot live query embedding path.

## Solution Assessment

Yes. A narrow retry-on-transient-transport fix around live query embeddings is the smallest maintainable first step; broader Undici dispatcher tuning or a user-facing retry config knob should wait for more targeted proof.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/71784 with a narrow memory transport resilience PR. Current main still runs live builtin memory search through `MemoryManager.search()` -> `embedQueryWithTimeout()` -> provider `embedQuery()` without retrying transient fetch/Undici/TLS socket errors. Add a conservative transient transport-error classifier for messages/codes such as `fetch failed` with network causes, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, socket hangups, `terminated`, and `Client network socket disconnected before secure TLS connection was established`, and apply a bounded retry/backoff to the live query embedding path. Preserve fail-fast behavior for auth/config/schema errors and ordinary HTTP 4xx responses. Coordinate with #44166/#44167 so batch and query paths share the classifier where practical, but do not broaden the change into dispatcher keep-alive tuning or a new config knob unless a failing test requires it. Add focused unit tests for classifier behavior and query retry behavior; update provider/host tests only if the implementation lives there. Add a single-line user-facing changelog fix entry with appropriate reporter credit, not bot credit. Do not change provider selection, embedding dimensions, cache format, SSRF policy, or batch splitting semantics beyond what is needed for this live-query reliability fix.
