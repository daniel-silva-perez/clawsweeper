## Summary

Keep open. Current main and latest release still leave live builtin memory search query embeddings as a single-shot remote call, so the reported `fetch failed` / socket-close class can still abort `openclaw memory search`; the related open PR is scoped to batch indexing and does not close this live-query surface.

## What This Changes

Make live memory-search query embeddings resilient to intermittent remote embedding transport failures across providers.

## Best Possible Solution

Add a bounded retry/backoff around live query embeddings, reuse or lightly adapt the existing memory embedding retry helper, and expand the classifier only for narrow transient transport/socket failures while preserving immediate failure for auth, invalid config, schema errors, and non-retryable HTTP responses.

## Reproduction Assessment

Yes. The issue provides concrete live-provider search loops and follow-up confirmations, and current-main source inspection shows that `memory search` still performs one live query embedding call without retry.

## Solution Assessment

Yes. A focused live-query retry is the narrowest maintainable first fix; broader Undici dispatcher tuning or a new config knob should wait for evidence that scoped retry is insufficient.

Scope the change to the memory-core live query embedding path and shared retry classifier/helper as needed. Add a bounded retry/backoff for `embedQueryWithTimeout()` or a shared query/batch embedding retry wrapper; classify only narrow transient transport/socket failures such as `fetch failed` with socket causes, `other side closed`, `ECONNRESET`, `EPIPE`, `socket hang up`, `terminated`, `UND_ERR_SOCKET`, and `Client network socket disconnected before secure TLS connection was established`.
