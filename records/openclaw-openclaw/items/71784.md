## Summary

Keep open. Current main still exposes the reported live memory-search failure path: CLI search calls `MemoryManager.search()`, search performs a single query embedding before vector lookup, and `embedQueryWithTimeout()` wraps `provider.embedQuery()` only in a timeout rather than the existing retry loop. The related open PR is useful overlap for batch reindexing, but it does not close this live-query surface.

## What This Changes

The issue asks for reliable live builtin memory search when remote embedding providers intermittently fail with transient fetch, socket, or TLS connection errors.

## Best Possible Solution

Add a narrow transient transport-error classifier and bounded retry/backoff for live query embeddings, preferably by reusing the existing memory embedding retry helper while preserving immediate failures for auth, config, schema, and non-retryable HTTP errors. Treat dispatcher keep-alive tuning and user-facing retry knobs as follow-up only if focused tests or live proof show the retry does not cover the observed failure.

## Reproduction Assessment

Yes. The issue includes a concrete remote-provider `openclaw memory search` loop and later confirmations from upgraded/provider-diverse installs; current-main source inspection verifies the live query embedding path is still single-shot.

## Solution Assessment

Yes. A focused live-query retry using the existing memory embedding policy is the narrowest maintainable first fix; broad Undici pool changes or new config should wait for proof that the scoped retry is insufficient.

## Repair Work Prompt

Fix the live builtin memory-search remote embedding reliability issue reported in https://github.com/openclaw/openclaw/issues/71784. Current main retries batch embeddings but `embedQueryWithTimeout()` is single-shot, and `isRetryableMemoryEmbeddingError()` does not classify transient transport/socket failures. Implement a narrow retry/backoff for live query embeddings, preferably reusing `runMemoryEmbeddingRetryLoop(...)` and extending the classifier for fetch/Undici/TLS socket reset cases such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, `socket hang up`, `terminated`, and `Client network socket disconnected before secure TLS connection was established`. Preserve fail-fast behavior for auth/config/schema errors and non-retryable 4xx provider responses. Coordinate with #44166/#44167 so batch retry semantics do not diverge, but do not turn this into a dispatcher/proxy rewrite or add a config knob unless targeted proof shows the narrow retry cannot work. Add focused unit coverage for live query retry and classifier behavior; add a user-facing changelog fix entry with appropriate reporter attribution. Do not touch lockfiles, provider auth, unrelated QMD lock-contention behavior, or broad Undici global dispatcher settings.
