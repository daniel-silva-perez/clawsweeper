## Summary

Keep open. Current main still rebuilds every array-backed text part and returns a new message object even when inline directive stripping makes no text change; the focused identity-preservation tests are still absent, and the related implementation PRs are closed unmerged.

## What This Changes

The issue requests preserving message and text-part object identity when display-time inline directive stripping does not actually change message content.

## Best Possible Solution

Update the directive-tag message helper so it reuses unchanged text parts, tracks whether any part changed, returns the original message when nothing was stripped, and add focused identity assertions that lock in those semantics.

## Repair Work Prompt

Fix the identity churn reported in openclaw/openclaw#37589. In `src/utils/directive-tags.ts`, update `stripInlineDirectiveTagsFromMessageForDisplay` so it only recreates text parts whose text actually changes after `stripInlineDirectiveTagsForDisplay`, preserves unchanged non-text and unchanged text-part references, and returns the original message object when no array content part changed. Keep all existing directive stripping output semantics, exported types, and plugin SDK surfaces stable; do not change the directive regexes, delivery stripping, parse behavior, or unrelated chat/media paths. Add focused tests in `src/utils/directive-tags.test.ts` proving the original message object is returned when array-backed content has no directives and proving mixed changed/unchanged text parts preserve only unchanged references. Related closed source attempts: #37588, #37642, #47205. No changelog entry is expected for this internal helper/performance fix unless maintainers request one.
