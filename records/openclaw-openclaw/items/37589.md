## Summary

Keep this open. Current main still recreates array-backed message objects and text-part objects in `stripInlineDirectiveTagsFromMessageForDisplay` even when no inline directive tags are stripped, while the linked implementation attempts are closed unmerged.

## Best Possible Solution

Implement the focused identity-preservation fix in `src/utils/directive-tags.ts`: reuse unchanged text-part objects, track whether any text part changed, and return the original message object when nothing was stripped. Add colocated Vitest coverage using `toBe` for unchanged array-backed messages and mixed changed/unchanged text parts. Keep docs, SDK exports, and directive parsing semantics unchanged unless maintainers deliberately decide this helper should not preserve identity.

## Clownfish Work Prompt

Fix the open issue about `stripInlineDirectiveTagsFromMessageForDisplay` recreating unchanged message objects. In `src/utils/directive-tags.ts`, update the helper so array-backed messages preserve identity when no inline directive tags are stripped: call `stripInlineDirectiveTagsForDisplay` for text parts, reuse each original text-part object when `changed` is false, track whether any text part changed, and return the original message object when no part changed. Preserve existing behavior for `undefined`, non-array content, null/non-object parts, non-text parts, and empty-string text when a directive is the whole content. Add focused tests in `src/utils/directive-tags.test.ts` using `toBe` for an unchanged array-backed message and for unchanged text-part references in a mixed changed/unchanged content array. Do not change directive regex semantics, SDK/package exports, docs, extension APIs, or unrelated gateway sanitizers. No changelog entry is expected for this narrow internal/plugin-helper behavior fix unless a maintainer asks for one.
