## Summary

Keep open for maintainer review. Current main still rejects empty Slack slash-command prompts and does not expose `slashCommand.contextAware`; the PR's feature direction is understandable, but the proposed patch targets the old `src/slack` layout and introduces security-sensitive history-to-command wiring that should not merge as-is.

## What This Changes

The PR adds a Slack `slashCommand.contextAware` config/schema/resolver option and changes empty Slack slash-command invocations to fetch recent channel history and use it as the agent prompt.

## Best Possible Solution

If maintainers want this feature, implement it in the current Slack plugin as an opt-in setting that checks slash-command authorization before reading history, reuses Slack supplemental-context visibility filtering for eligible messages, and passes recent history as untrusted or supplemental context while keeping command/directive fields limited to the actual slash-command text. The change should update config type/schema/generated metadata/docs and add focused tests for empty prompts, directive isolation, and history filtering.

## Reproduction Assessment

Yes. Current-main behavior is reproducible by invoking the configured Slack slash command with no text and observing the `Message required.` response; the PR-specific regression is reproducible from the provided diff because empty context-aware mode writes recent channel history into prompt/command handling fields.

## Solution Assessment

No. The feature direction may be useful, but this patch is not the safest maintainable solution because it treats recent Slack history as command text, skips the existing supplemental-context filtering model, leaves dead `contextHistory` wiring, and needs to be ported to the current plugin layout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep recent history out of command fields:** `src/slack/monitor/slash.ts:597-598`
  - body: When the empty context-aware path writes recent Slack history into the slash turn's prompt/command fields, downstream directive parsing can treat prior channel messages as command text. A channel participant could place directive-like text in history and alter the invoker's empty slash turn, so history should stay in untrusted or supplemental context and directive parsing should use only the actual slash text.
  - confidence: 0.89
- **[P2] Reuse the Slack supplemental context filter:** `src/slack/monitor/slash.ts:565-582`
  - body: This new branch formats every returned history message into the prompt. Current Slack thread history filters current-bot and non-allowlisted senders with the supplemental-context visibility policy before exposing history, so the empty slash-command path should apply the same filtering before history can influence a turn.
  - confidence: 0.78
- **[P3] Remove or wire contextHistory:** `src/slack/monitor/slash.ts:563`
  - body: `contextHistory` is assigned from the formatted Slack message history but is never read afterward. Remove it if prompt-only behavior is intended, or pass it through the chosen structured context path once the context model is fixed.
  - confidence: 0.82

## Security Review
