## Summary

Keep open. Current main still renders cron schedules as raw `Cron ...` text in the Cron Jobs list and still renders Dreaming phase rows from `nextRunAtMs` only, so this PR is not obsolete. The branch is a plausible narrow UI improvement, but it needs maintainer review because current Dreaming docs/status now describe one managed sweep schedule, and the user-facing fix is missing required changelog coverage.

## What This Changes

The PR adds a `describeCronExpression()` UI formatter with tests, uses it for Cron Jobs schedule labels, and changes Dreaming phase rows to show a cron description plus the next-run time while preserving a next-run fallback when cron text is absent.

## Best Possible Solution

Land a small, tested UI cron description formatter after adding the required changelog entry, tightening formatter edge cases, and making the Dreaming phase text match the current one-sweep Dreaming model; if that wording is misleading, scope the PR to Cron Jobs and leave Dreaming copy for a separate product decision.

## Reproduction Assessment

Yes. A high-confidence source check reproduces the current-main behavior: Cron Jobs render `formatCronSchedule(job)` and cron schedules still return raw `Cron ...` text, while Dreaming phase rows render only `nextRunAtMs`.

## Solution Assessment

Unclear. A shared formatter is the narrow maintainable shape for Cron Jobs, but applying per-phase cron descriptions in Dreaming needs maintainer confirmation because the current backend/docs model one managed sweep schedule rather than distinct phase cron jobs.
