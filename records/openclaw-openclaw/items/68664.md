## Summary

Keep open. Current main applies runtime legacy migrations before config validation, so the broad migration-order claim is partly obsolete, but the specific stale `agents.defaults.memorySearch` subkey cleanup is still missing and the linked open PR #68685 explicitly closes this issue. Per policy, the issue should remain open until that PR is reviewed and either landed or rejected.

## What This Changes

The issue reports that legacy configs with obsolete `agents.defaults.memorySearch` keys such as `chunkSize`, `chunkOverlap`, and root-level `maxResults` can fail strict schema validation before automatic cleanup makes startup safe.

## Best Possible Solution

Keep this issue open and resolve it by landing a narrow migration/doctor fix that strips only the known obsolete `agents.defaults.memorySearch` root keys before strict validation, with regression coverage for already-migrated configs and combined top-level move plus stale subkey cleanup.

## Reproduction Assessment

Yes. The source-backed reproduction is a config containing `agents.defaults.memorySearch.chunkSize`, `chunkOverlap`, or root `maxResults`; the current strict schema rejects those keys and current migrations do not remove them from an already-migrated `agents.defaults.memorySearch` object.

## Solution Assessment

Yes, the narrowest maintainable solution is a targeted legacy migration/doctor repair for the obsolete memorySearch keys, not loosening the schema globally. Open PR #68685 appears to target that boundary and should be reviewed as the implementation candidate.
