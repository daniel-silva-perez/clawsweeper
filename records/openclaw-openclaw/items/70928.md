## Summary

Keep open. Current main still routes WebChat and Gateway-backed TUI replies through `chat.send`, where final assistant delivery uses a local dispatcher, direct transcript append, and direct chat broadcast instead of the hook-aware wrapper or `before_message_write` path; local embedded TUI final events also emit directly.

## What This Changes

Ensure WebChat and openclaw-tui outbound assistant replies fire the expected outbound lifecycle hooks, especially `message_sent` and `before_message_write`.

## Best Possible Solution

Land a narrow Gateway chat and TUI hook-coverage fix before the broader outbound observer work: final assistant replies should run the existing message lifecycle semantics with stable session/run context, emit plugin and internal sent hooks exactly once for terminal delivery, run `before_message_write` before Gateway-injected assistant transcript appends, and include WebChat plus TUI regression tests.

## Reproduction Assessment

Yes for `message_sent` and `before_message_write`: a source-level reproduction is clear because WebChat and Gateway TUI both use `chat.send`, and current main shows that path bypassing sent-message and message-write hook calls. The `agent_end` claim is unclear and should be verified with a focused test before changing runner behavior.

## Solution Assessment

Yes: the best near-term fix is a local `chat.send` and TUI parity repair using existing hook seams, while leaving the broader shared outbound observer design to #50126. A safer implementation should preserve terminal final frames on cancel so UI clients do not hang.
