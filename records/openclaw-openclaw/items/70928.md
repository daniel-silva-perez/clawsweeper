## Summary

Keep open. Current main still leaves the default WebChat and Gateway-backed TUI outbound reply path outside the existing hook-aware delivery seams, so the reported `message_sending`/`message_sent`/`message:sent` and Gateway transcript `before_message_write` coverage gap remains valid and actionable.

## What This Changes

Request to make WebChat and openclaw-tui outbound assistant replies fire outbound lifecycle and transcript-write hooks consistently.

## Best Possible Solution

Add a focused Gateway chat hook coverage path: run `message_sending` before WebChat/TUI final delivery, emit plugin `message_sent` plus internal `message:sent` after final success/failure/cancel where applicable, run `before_message_write` before Gateway-injected assistant transcript appends, and cover WebChat plus Gateway TUI with regression tests while leaving the broader outbound observer architecture to #50126.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: WebChat and Gateway TUI both route through `chat.send`, and current main shows that path bypassing the hook-aware dispatcher, sent-hook emitters, and `before_message_write` before Gateway-injected assistant transcript appends. I did not run live tests because this review was read-only.

## Solution Assessment

Yes. The narrow Gateway chat fix is the best next step; the broader outbound observer design remains useful for #50126, but this default-surface bug can be repaired using existing hook seams and targeted exactly-once tests.
