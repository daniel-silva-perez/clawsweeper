## Summary

Keep this open. Current main still routes WebChat and Gateway-backed TUI sends through `chat.send`, where the reply path constructs a raw dispatcher and bypasses the existing hook-aware dispatcher wrapper; the reply dispatcher also has no `message_sent` emission, and Gateway-injected assistant transcript appends do not run the `before_message_write` guard path.

## What This Changes

Request to make WebChat and openclaw-tui outbound assistant replies fire outbound message lifecycle hooks, including `message_sending`, `message_sent`, and `before_message_write`.

## Best Possible Solution

Land a narrow Gateway chat hook coverage fix before the broader #50126 unification: route `chat.send` replies through the existing hook-aware dispatcher or equivalent shared seam, emit `message_sent`/internal `message:sent` exactly once for delivered or failed final replies, and run `before_message_write` before Gateway-injected assistant transcript appends. Cover WebChat/Gateway TUI `chat.send` in regression tests, while leaving larger cross-channel observer architecture to the umbrella issue.

## Reproduction Assessment

Yes. The source trace is high-confidence: WebChat and Gateway TUI both call `chat.send`, and current `chat.send` bypasses the hook-aware wrapper while its dispatcher and transcript helper lack the missing hook calls. I did not run a live hook because this review was read-only.

## Solution Assessment

Yes. The issue’s local-fix direction is the narrowest maintainable next step: wire the Gateway chat path into the existing hook-aware outbound behavior, then coordinate the broader channel unification separately under #50126.
