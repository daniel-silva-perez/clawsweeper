## Summary

Keep this open. Current main still matches the reported WebChat/Gateway TUI gap: both surfaces use `chat.send`, and that path still builds a raw reply dispatcher, bypasses the existing `message_sending` wrapper, lacks `message_sent` emission, and appends Gateway-injected assistant transcript messages without the `before_message_write` guard path. Related issues remain open but do not fully supersede this issue's combined WebChat/TUI and transcript-write coverage.

## What This Changes

Request to make WebChat and Gateway-backed openclaw-tui outbound assistant replies fire lifecycle hooks, including `message_sending`, `message_sent`, and `before_message_write`.

## Best Possible Solution

Land a narrow Gateway chat hook coverage fix before the broader #50126 unification: route `chat.send` replies through the existing hook-aware dispatcher or equivalent shared seam, emit plugin `message_sent` and internal `message:sent` exactly once for delivered or failed final replies, and run `before_message_write` before Gateway-injected assistant transcript appends. Cover WebChat/Gateway TUI `chat.send` in regression tests while leaving the cross-channel observer architecture to the umbrella issue.
