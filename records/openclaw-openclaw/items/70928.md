## Summary

Keep open. Current main still leaves the default WebChat and Gateway-backed TUI reply path outside the existing hook-aware outbound seams, so the reported hook coverage gap remains actionable rather than implemented or safely duplicate-closed.

## What This Changes

The issue requests that outbound assistant replies from WebChat and openclaw-tui fire `message_sending`/`message_sent` and `before_message_write` hooks consistently with other delivery paths.

## Best Possible Solution

Add a focused Gateway chat hook coverage path: apply `message_sending` before final WebChat/TUI delivery, emit plugin `message_sent` and internal `message:sent` after final success/failure/cancel, run `before_message_write` before Gateway-injected assistant transcript appends, and cover both WebChat plus Gateway TUI with regression tests while leaving the broader outbound observer architecture to #50126.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: WebChat and Gateway TUI both call `chat.send`, and current main shows that path bypassing the hook-aware dispatcher, sent-hook emitters, and before-write hook on Gateway-injected assistant transcript appends; I did not run live tests because this review was read-only.

## Solution Assessment

Yes, the issue's local Gateway chat fix direction is the best next step. The broader observer design remains useful for #50126, but this default-surface regression can be repaired with existing hook seams and targeted exactly-once tests.
