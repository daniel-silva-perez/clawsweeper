## Summary

Keep this open. Current main still has the default WebChat/Gateway TUI outbound hook gap: `chat.send` builds a raw reply dispatcher and calls the lower-level dispatch function directly, so it bypasses the hook-aware wrapper added for `message_sending`; the reply dispatcher itself has no `message_sent` emission; and Gateway assistant transcript appends still write through the injected transcript helper without the `before_message_write` guard path.

## What This Changes

The issue reports that WebChat and Gateway-backed TUI assistant replies do not fire outbound lifecycle hooks such as `message_sending`, `message_sent`, and `before_message_write`.

## Best Possible Solution

Keep this open and fix the Gateway `chat.send` path used by WebChat and Gateway TUI. The maintainable end state is one hook-aware outbound seam for assistant replies: `message_sending` can rewrite or cancel before visible delivery, `message_sent` observes success/failure once, and `before_message_write` applies before Gateway assistant transcript appends, with tests covering WebChat `chat.send` and the TUI client route.

## Reproduction Assessment

Yes for the main reported hook gap: the current source trace from WebChat/TUI to `chat.send` shows `message_sending`, `message_sent`, and `before_message_write` are not wired on that path. I did not run a live hook because this review was read-only, and `agent_end` needs a focused test because it is dispatched by the agent harness rather than the outbound dispatcher.

## Solution Assessment

Yes, the issue’s local-fix direction is the narrowest maintainable next step: wire the Gateway chat path into the existing hook-aware dispatcher or add equivalent hook handling around its current dispatcher and transcript append. A broader outbound observer can remain coordinated with #50126, but the default-surface regression does not need to wait for a full architecture rewrite.
