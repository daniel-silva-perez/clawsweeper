## Summary

Keep this open. Current main still has the actionable config-portability bug: the source does not contain a literal hardcoded `wangtao` path, but setup/onboarding/configure paths still resolve the default workspace to a host-absolute path before persisting it, and the directly related #56810 fix PR is closed and unmerged.

## What This Changes

Report that OpenClaw can persist or report a host-specific absolute default workspace path instead of a portable `~/.openclaw/workspace` default.

## Best Possible Solution

Land a focused config-serialization fix that stores the canonical default workspace as a portable `~/.openclaw/workspace` or profile-aware `~/.openclaw/workspace-<profile>` value, while continuing to resolve to absolute paths only at runtime and preserving explicitly entered custom absolute paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: follow the default setup/onboard path and inspect the value written to `agents.defaults.workspace`; current code resolves the default with `resolveUserPath()` or `resolveDefaultAgentWorkspaceDir()` before persisting it.

## Solution Assessment

Yes, the narrowest maintainable solution is to canonicalize only default workspace values at config write time. Runtime already supports portable `~` values through `resolveUserPath()`, so broad runtime changes or startup migrations are unnecessary for the core fix.
