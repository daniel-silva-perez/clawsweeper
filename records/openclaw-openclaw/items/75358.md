## Summary

Keep open. Current main still drops PDF page images when `@napi-rs/canvas` cannot be loaded, and there is no Poppler/`pdftoppm` fallback in the bundled `document-extract` plugin or docs. The report is concrete, recent, and source-backed; it is not a duplicate of the nearby PDF fallback reports provided in context.

## What This Changes

The issue asks for PDF page-image extraction to fall back to Poppler/`pdftoppm` when the bundled `document-extract` plugin cannot load optional `@napi-rs/canvas`.

## Best Possible Solution

Keep the issue open and fix the bundled `document-extract` plugin boundary: when canvas is unavailable, either render through a bounded, documented Poppler fallback or surface a clear degradation signal, while preserving existing page, pixel, size, sandbox, and model-capability limits. The PDF docs and focused regression tests should be updated with the chosen dependency contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches image rendering, `loadCanvasModule()` throws when `@napi-rs/canvas` is absent, the catch block returns an empty image list, and the agent PDF tool does not pass an error callback.

## Solution Assessment

Unclear. The requested `pdftoppm` fallback is a plausible narrow owner-module fix, but it changes the runtime dependency and external-process/security contract, so a maintainer should choose the final policy before automation implements it.
