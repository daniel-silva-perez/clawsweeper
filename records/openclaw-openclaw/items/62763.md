## Summary

Keep open. Current main still appears to reproduce the reported Signal doctor corruption: the doctor compatibility migration can still create `channels.signal.accounts.default` from top-level `cliPath`, `dmPolicy`, and `groupPolicy`, while Signal runtime account resolution treats channel-root values as inherited per-account defaults. The related systemd doctor issue is separate and does not supersede this Signal config migration bug.

## What This Changes

The issue reports that `openclaw doctor --fix` moves valid Signal channel-level defaults into a phantom `channels.signal.accounts.default` entry during multi-account repair, breaking the Signal gateway runtime.

## Best Possible Solution

Add a Signal-specific promotion contract or doctor guard that preserves valid channel-root defaults for multi-account Signal configs and prevents settings-only `accounts.default` entries, with regression coverage for the exact reported config and a docs note if Signal becomes an explicit exception to the generic migration wording.

## Reproduction Assessment

Yes. The issue provides a concrete config, and current source still routes that shape through `normalizeCompatibilityConfigValues` into `seedMissingDefaultAccountsFromSingleAccountBase`, where `cliPath`, `dmPolicy`, and `groupPolicy` are enough to synthesize `accounts.default`. I did not run tests because this review was read-only, but the source path is direct and high-confidence.

## Solution Assessment

No, the current doctor migration is not the best way to solve this issue. The safer fix is to keep Signal channel-root defaults inheritable in named-account configs and only promote real account-specific values into an existing real account when that behavior is explicitly intended.
