## Summary

Keep this issue open. Current main still has the generic doctor repair path that moves authored Signal top-level keys such as `cliPath`, `dmPolicy`, and `groupPolicy` into `channels.signal.accounts.default` when named accounts exist and no literal `default` account is present. Signal does not currently provide a config-promotion override to narrow that behavior, while docs and runtime account resolution still support base channel settings inheriting into named accounts.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the Signal-owned setup/promotion contract plus doctor regression coverage: preserve `channels.signal.cliPath`, `dmPolicy`, `groupPolicy`, and other shared base settings for named-account configs, and avoid inventing `accounts.default` when `defaultAccount` already points at an existing named Signal account. Add a focused doctor test using the reported two-account shape and verify runtime account listing does not include a fabricated default account.
