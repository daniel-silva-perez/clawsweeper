## Summary

Keep this open. Current main still preserves configured model fallback order, the docs do not define the requested latency/quality/cost fallback policy for the named active models, and config validation still accepts broad `models.providers.*.models[].input` modalities without a model-specific unsupported-modality guardrail.

## What This Changes

The issue requests benchmark-backed fallback model ordering by use case plus a model-input config guardrail that rejects unsupported modalities before deploy.

## Best Possible Solution

Keep this open for maintainer triage and split it into two reviewed tracks: a benchmark-backed fallback ordering policy for the active fallback chain, and a semantic validation design that catches impossible `models.providers.*.models[].input` declarations using provider/model capability metadata without rejecting valid audio/video-capable catalogs. Update the model failover docs once the policy is chosen.
