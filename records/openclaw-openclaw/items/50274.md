## Summary

Keep this issue open. Current main still preserves configured fallback order without a latency/quality/cost ranking policy, the docs only describe generic fallback-chain behavior, and `models.providers.*.models[*].input` still accepts broad modalities such as `audio`/`video` without a model-specific compatibility guardrail.

## Best Possible Solution

Keep this open for maintainer triage. The best path is to split the work into a benchmark-backed fallback ordering policy for the active fallback chain and a carefully scoped validation guardrail that catches impossible `models.providers.*.models[*].input` declarations without breaking valid audio/video-capable catalogs. #6599 can inform fallback diagnostics but does not supersede this issue.
