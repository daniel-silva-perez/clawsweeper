## Summary

Keep open. Current main still evaluates configured fallback chains in configured order and only schema-validates model input modality names; it does not yet provide benchmark-backed latency/quality/cost fallback policies for the named active chain or a capability-aware guardrail for unsupported configured model inputs.

## What This Changes

The issue requests benchmark-backed fallback ordering plus a capability-aware guardrail for `models.providers.*.models[*].input` values.

## Best Possible Solution

Define a repeatable benchmark artifact for the active fallback chain, document the selected fallback ordering policy in the model failover/model CLI docs, then add capability-aware validation for configured model inputs using trusted provider or catalog metadata while preserving valid custom multimodal catalog overrides.
