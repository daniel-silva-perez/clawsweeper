## Summary

Keep open. Current main still uses explicit configured fallback order, has no benchmark-backed latency/quality/cost ordering artifact for the named OpenRouter chain, and only validates configured model input modality names rather than checking whether a declared modality is supported by that provider/model. The existing discussion already reached the same conclusion, and the relevant current-main source/docs/tests still support keeping this as maintainer follow-up.

## What This Changes

The issue requests benchmark-backed fallback model ordering by use case plus a capability-aware config guardrail for invalid `models.providers.*.models[].input` modalities.

## Best Possible Solution

Define a repeatable benchmark artifact for the active fallback chain, document the chosen fallback-ordering policy in the model failover/models docs, then add capability-aware validation for configured model inputs using trusted provider or catalog metadata while preserving valid custom multimodal custom providers.

## Reproduction Assessment

Partially. Static inspection gives a high-confidence check for the missing guardrail and policy artifact, but the fallback-ordering part itself depends on benchmark data for the active provider/model chain rather than a deterministic local reproduction.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer-defined benchmark criteria and a capability source design; blanket schema tightening around `input` would not be the narrowest maintainable fix.
