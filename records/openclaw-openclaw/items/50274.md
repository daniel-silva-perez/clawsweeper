## Summary

Keep open. Current main still preserves explicitly configured fallback order, documents ordered fallback chains rather than latency/quality/cost policies, and only validates configured model input modality names rather than checking a declared modality against trusted provider/model capability metadata. The prior ClawSweeper comments reached the same conclusion, and the current source/docs/tests still support maintainer follow-up instead of cleanup closure.

## What This Changes

The issue requests benchmark-backed ordering for the active OpenRouter fallback chain, a documented latency/quality/cost fallback policy, and a capability-aware guardrail for invalid `models.providers.*.models[*].input` modalities.

## Best Possible Solution

Define a repeatable benchmark artifact for the active fallback chain, document the selected fallback-ordering policy in the model failover/models docs, then add capability-aware validation for configured model inputs using trusted provider or catalog metadata while preserving valid custom multimodal providers.

## Reproduction Assessment

Partially. Static inspection gives a high-confidence reproduction path for the missing capability-aware guardrail and policy artifact, but ranking the named fallback chain requires live benchmark data rather than a deterministic local reproduction.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation needs maintainer-defined benchmark criteria and a capability source design; blanket schema tightening around `input` is not the narrowest maintainable fix.
