## Summary

Keep open. Current main supports ordered configured fallback chains and an OpenRouter scan helper, and it accepts audio/video modality names in configured model metadata, but it still does not provide benchmark-backed latency/quality/cost fallback policies for the named active chain or a capability-aware guardrail that rejects unsupported `models.providers.*.models[].input` entries. Existing tests intentionally preserve explicit multimodal custom catalog metadata, so the guardrail needs maintainer design rather than blanket schema tightening.

## What This Changes

The issue asks for benchmark-ranked model fallback ordering policies and config validation that fails fast when a configured model declares unsupported input modalities.

## Best Possible Solution

Define a documented fallback policy artifact for the active model chain, backed by repeatable benchmark criteria, then add capability-aware validation for configured model inputs using trusted provider/catalog metadata while preserving valid custom multimodal catalogs.
