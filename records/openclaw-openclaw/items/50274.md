## Summary

Keep this open. Current main still treats model fallback as an operator-configured candidate chain, and the config schema still validates only allowed modality names rather than rejecting a modality that a specific model cannot support.

## What This Changes

The issue requests benchmark-backed fallback model ordering, use-case-specific ordering policy, model-specific `models.providers.*.models[*].input` guardrails, invalid-modality tests, and fallback-policy docs.

## Best Possible Solution

Define a benchmark-backed fallback policy for the active chain, then add semantic validation that compares configured model inputs against provider/model capabilities while preserving valid multimodal catalogs. Update the failover/config docs once the policy and validation boundary are chosen.
