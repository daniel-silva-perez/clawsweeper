## Summary

Keep open. Current main still has the Active Memory timeout-budget mismatch at the embedded recall boundary, and an open closing PR already targets that exact remaining handoff, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Active Memory should stop timing out on every run with zero summary output by applying the intended timeout/setup-grace budget consistently through the plugin hook, watchdog, and embedded recall runner.

## Best Possible Solution

Land #74480 or an equivalent narrow change that computes one effective Active Memory recall timeout and passes it consistently to the embedded runner, outer watchdog, and hook registration, with focused unit coverage and post-merge log validation for timeout/no-summary behavior.

## Reproduction Assessment

Yes. The discussion has multiple independent log-based reproductions, and current-main source/test inspection gives a focused synthetic path: configure setup grace, trigger before_prompt_build, and observe runEmbeddedPiAgent still receives only the raw configured timeout.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable solution for the timeout-budget mismatch; any remaining provider-specific empty-summary or CPU-saturation behavior should be validated and tracked separately after that handoff is fixed.
