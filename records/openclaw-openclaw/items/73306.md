## Summary

Keep open. Current main contains several Active Memory timeout mitigations, but the central timeout handoff remains incomplete: the hook/watchdog use the setup-grace-extended budget while `runEmbeddedPiAgent` still receives the raw configured timeout, and an open closing PR targets that exact remaining gap.

## What This Changes

The issue reports Active Memory timing out on every run with zero summary output after upgrading to v2026.4.26, causing per-turn recall failures and token/context bloat.

## Best Possible Solution

Land #74480 or an equivalent narrow fix that computes one effective Active Memory recall timeout and passes it consistently through the hook, watchdog, and embedded runner, then validate timeout/no-summary logs before closing this issue.

## Reproduction Assessment

Yes. The discussion has multiple independent log-based reproductions, and current-main source inspection gives a focused synthetic path: configure setup grace, trigger `before_prompt_build`, and observe the embedded runner still receives the raw timeout its own abort timer uses.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable solution for the timeout-budget mismatch still visible in current main; any remaining provider-specific empty-summary or CPU saturation behavior should be tracked separately if it still reproduces after that handoff is fixed.
