## Summary

Keep open. Current main still preserves the Active Memory timeout mismatch reported here: the hook and outer watchdog use the setup-grace-aware budget, while the embedded recall runner still receives only the raw configured timeout. Open closing PR #74480 targets that exact handoff, so the issue should stay open until that PR, or an equivalent fix, lands and the broader zero-summary symptoms are revalidated.

## What This Changes

The issue reports that the Active Memory plugin times out on each eligible chat turn after v2026.4.25/v2026.4.26, returning zero summary characters and causing blocking context/token growth.

## Best Possible Solution

Review and land #74480, or an equivalent narrow fix that computes one effective Active Memory recall timeout and passes it consistently through the embedded runner, wrapper watchdog, and prompt hook; then recheck whether remaining empty-summary behavior belongs in the broader Active Memory timeout trackers such as #65309 or #66804.

## Reproduction Assessment

Yes. The discussion has multiple independent log-based reproductions, and current-main inspection gives a focused synthetic path: configure Active Memory with setup grace, trigger recall, and observe that runEmbeddedPiAgent still receives the raw timeout that drives its own abort timer.

## Solution Assessment

Yes for the narrow timeout-budget mismatch. The linked PR direction is the smallest maintainable fix because it keeps the behavior inside the Active Memory plugin and aligns the existing hook, watchdog, and embedded-runner budgets without changing provider routing or memory search semantics.
