## Summary

Keep open. Current main has several Active Memory timeout mitigations, but the central timeout handoff is still not fully fixed: the plugin uses setup grace for its hook/watchdog while still passing the raw configured timeout into the embedded runner, and an open closing PR specifically targets that remaining gap.

## What This Changes

The issue reports that Active Memory times out with zero summary output on every run after upgrading to v2026.4.25/v2026.4.26, blocking chat turns and causing context/token bloat across several providers.

## Best Possible Solution

Land #74480 or an equivalent narrow fix that computes one effective Active Memory recall timeout and passes it consistently through the hook, watchdog, and embedded runner, then validate the timeout/no-summary logs before closing this issue.

## Reproduction Assessment

Yes. The issue has multiple independent log-based reproductions, and current-main source/test inspection gives a focused synthetic path: configure setup grace, trigger `before_prompt_build`, and observe the embedded runner still receives the raw timeout that its abort timer uses.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix for the timeout-budget mismatch shown in current main. Any remaining provider-specific empty-summary or CPU saturation behavior should be tracked separately if it still reproduces after that handoff is fixed.
