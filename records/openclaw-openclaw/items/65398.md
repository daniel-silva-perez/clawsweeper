## Summary

Keep open. Current main still lacks both Feishu fixes, and the related earlier PR #56520 is closed and unmerged, but this PR is not safe to merge as-is because its process-wide registration guard can suppress Feishu hook/tool registration after plugin reloads clear the active registry.

## What This Changes

The PR adds a Feishu `registerFull` idempotency guard, Feishu named-account promotion keys, a core Feishu promotion fallback, and regression tests for duplicate registration and top-level `groupPolicy` preservation.

## Best Possible Solution

Keep the Feishu config-preservation fix, but revise the registration fix to be scoped to the active registry/API load or otherwise make duplicate registration replacement-safe. On current main, expose Feishu named-account promotion through the current setup promotion contract, including the package metadata needed for early bundled setup loading, with focused regression coverage.

## Reproduction Assessment

Yes. The duplicate-registration path is source-reproducible by calling the Feishu entry registration twice, and the `groupPolicy` rewrite path follows from a named-account Feishu config entering the legacy normalizer without a Feishu named-account promotion filter.

## Solution Assessment

No. The named-account promotion direction is appropriate, but the process-wide registration guard is not the narrowest maintainable fix because plugin reloads deliberately clear registered state and require re-registration.
