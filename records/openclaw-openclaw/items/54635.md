## Summary

Keep open. Current main still has the reported behavior: mutating tool errors take a separate warning-policy branch that ignores `isRecoverableToolError(...)`, then final payload construction emits `⚠️ ... failed` whenever that branch returns `showWarning`. Existing tests still assert recoverable mutating errors are shown, so this is not implemented on main. The prior discussion correctly points to related #59868 and PR #60664, but that PR is open, does not reference this issue with closing syntax, and current main has not absorbed a fix for this warning-policy path.

## Best Possible Solution

Keep this issue open. The best fix belongs in the embedded runner payload/tool-error path: suppress false mutating failure notifications only when the mutation is recoverable or proven applied, clear stale `lastToolError` when a later mutation proves the same action succeeded, and add regression coverage showing false-positive edit/write failures do not emit `failed` payloads while genuine mutating failures still do. Coordinate with #59868 and PR #60664, but close only after the behavior is fixed on main.

## Risks / Open Questions

- A naive fix that suppresses every recoverable-looking mutating error could hide real failed writes, deletes, message sends, permission failures, or disk-full errors.
- The safer path likely needs a proven-applied signal or explicit recovery metadata for mutating tools, plus focused payload regression coverage, rather than reusing the broad recoverable-keyword classifier blindly.
