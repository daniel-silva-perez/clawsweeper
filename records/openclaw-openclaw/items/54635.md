## Summary

Keep open. Current main still routes mutating tool failures through a payload warning branch that does not consult the recoverable-error classifier, and the test suite still asserts recoverable mutating errors produce user-facing failed payloads. The newer mutation-state clearing and edit recovery code partially mitigates retry/post-write cases, but it does not resolve the reported warning-policy path. Related #59868 and open PR #60664 are useful context, but current main has not absorbed a fix for this issue and the PR does not close it.

## Best Possible Solution

Keep this issue open. The best fix is in the embedded runner tool-error/payload path: suppress mutating failure notifications only when the mutation is proven applied or explicitly recovered, clear stale `lastToolError` when a later same-action mutation succeeds, and add focused regression coverage showing false-positive edit/write failures do not emit `failed` payloads while genuine mutating failures still do. Coordinate with #59868 and PR #60664, but close this only after the behavior is fixed on main.

## Risks / Open Questions

- A broad one-line suppression based only on recoverable-keyword matching could hide genuine mutating failures such as failed writes, deletes, message sends, permission errors, or disk-full errors.
- PR #60664 may fix one edit recovery subcase, but landing it as-is would not necessarily address stale `lastToolError` payload rendering or recoverable mutating warnings across other tools.
