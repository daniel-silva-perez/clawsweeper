## Summary

Keep this issue open. Current main has adjacent hard-stop, interrupt, steer, collect, and BTW side-question behavior, but it still does not implement owner-priority or priority-flagged soft in-turn delivery that preserves the active run and lets the agent incorporate new owner context before final response.

## Best Possible Solution

Keep this as a core agent-turn/message-queue feature request. The best path is to add a small explicit core design for priority or owner-priority followups: a supported config/API surface, safe boundary injection without aborting the active run, preservation of authorization and routing metadata, and focused tests for default collect mode, Telegram/group routing, streaming and non-streaming runs, `/stop`, `interrupt`, and `/btw` interactions.

## Risks / Open Questions

- The eventual implementation needs crisp semantics around preserving pending tool calls versus canceling them, because existing `steer` intentionally cancels pending tool calls while `interrupt` aborts the run.
- Owner-priority handling must preserve sender authorization, routing, Telegram topic/thread metadata, and group-chat ownership checks without elevating non-owner messages.
- Streaming and non-streaming paths differ today; a soft in-turn delivery feature needs tests for both so followups are not duplicated, dropped, or delayed until finalization.
- `/stop`, abort triggers, `interrupt`, `steer`, `steer-backlog`, `collect`, and `/btw` overlap with this request, so docs and tests should define how priority messages interact with each existing control.
