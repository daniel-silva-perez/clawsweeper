## Summary

Keep open. The optimization is not implemented on current main, and the branch cannot merge as-is: current main moved the active session parser and sync code into the memory-host package and memory-core plugin paths, while this PR still edits the removed `src/memory` implementation and carries correctness blockers around prefix validation, byte offsets, and unbounded retention.

## What This Changes

The PR branch adds a previous-entry append-only fast path to `buildSessionEntry`, caches `SessionFileEntry` objects during session sync, and adds an append-only reuse regression test.

## Best Possible Solution

Replace or rebase the optimization against the active memory-host and memory-core files, with byte-level prefix validation, safe fallback to the existing full parser, a bounded cache lifecycle, and regression coverage for ASCII append, non-ASCII append, rewritten-prefix fallback, line maps, timestamps, redaction, and transcript classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify the cached prefix before append reuse:** `src/memory/session-files.ts:125-130`
  - body: The fast path runs when `stat.size >= previous.size`, but it never proves the first `previous.size` bytes still match the cached transcript. If an earlier JSONL record is rewritten or compacted and the file remains the same size or grows, this reuses stale `previous.content` and only parses the tail, leaving the session index out of sync with the transcript.
  - confidence: 0.91
- **[P1] Slice append data by byte offset:** `src/memory/session-files.ts:127-129`
  - body: `previous.size` comes from `fs.stat` in bytes, but the patch uses it as a JavaScript string index in `raw.slice(...)`. Non-ASCII transcripts can put the append boundary at the wrong character, which can drop or duplicate parsed messages and produce an incorrect `lineMap`.
  - confidence: 0.94
- **[P1] Bound the session entry cache:** `src/memory/manager-sync-ops.ts:149`
  - body: The new `sessionEntryCache` stores full `SessionFileEntry` objects, including transcript `content`, and there is no eviction or cleanup when sessions are pruned or indexes reset. A long-lived gateway that sees many transcripts can retain those strings indefinitely and grow heap usage over time.
  - confidence: 0.88
