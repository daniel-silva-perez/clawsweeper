## Summary

Keep this issue open. Current main still has the source-level Windows path-containment gap: the avatar resolver compares raw realpath/path-resolved values with `path.relative()`, so mixed Windows namespace and drive-letter forms can still resolve as outside the workspace. That maps to `kind: "none"`, which the avatar metadata route returns as `avatarUrl: null` and the image route returns as 404. Live GitHub state also shows PR #38923 is still open and unmerged as an implementation candidate for this issue.

## Best Possible Solution

Keep this issue open. Review PR #38923 or extract an equivalent narrow fix that normalizes Windows namespace and drive-letter path forms before workspace containment checks, preserves symlink/workspace-boundary protections, handles backtick-wrapped IDENTITY.md avatar values if maintainers want to cover the related report shape, and adds regression coverage for a Windows non-default agent resolving a workspace-relative IDENTITY.md avatar through both `/avatar/<agentId>?meta=1` and `/avatar/<agentId>`.
