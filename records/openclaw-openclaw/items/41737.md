## Summary

Keep this PR open. Current main still implements vLLM as one bundled provider id (`vllm`) using the generic self-hosted setup path, so the PR's central work, endpoint lifecycle management, stable multi-endpoint keys like `vllm-2`, interactive multi-model selection, and stale managed-vLLM default cleanup, is not implemented on main. The PR is useful but needs rework against the current plugin/provider setup architecture and a scope/security cleanup before maintainer review.

## Best Possible Solution

Keep this PR open, but rework it narrowly around the current bundled vLLM provider plugin and generic provider setup SDK. The best path is to implement endpoint add/update/delete with stable keys such as `vllm`, `vllm-2`, and `vllm-3`; scan `/v1/models` with multi-select during interactive setup; preserve auth profiles plus config/env/SecretRef-backed provider auth; suppress implicit `vllm` discovery when any managed vLLM key exists; prune stale default and agent-level primary/fallback refs after endpoint removal; update the vLLM docs; and split or drop unrelated secrets baseline, A2UI, zizmor, and memory-test changes.
