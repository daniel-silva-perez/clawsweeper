## Summary

Keep open. Current main still lacks the requested vLLM endpoint lifecycle flow, so the PR is not obsolete; however the current PR head still misses the production dispatch path for the new vLLM handler and includes unrelated build/security-policy changes that should not merge as-is.

## What This Changes

The PR adds a vLLM endpoint management flow with multi-endpoint keys, model discovery and selection, stale default cleanup, docs/tests, and additional unrelated A2UI bundling, zizmor, secrets-baseline, agents, and memory-test changes.

## Best Possible Solution

Keep the useful vLLM endpoint-management direction, but rebase it onto current plugin/provider setup, wire it through the actual vLLM auth choice path, keep endpoint/default cleanup focused in provider-owned code, and split unrelated A2UI, zizmor, secrets, and memory-test changes into separate maintainer-reviewed work.
