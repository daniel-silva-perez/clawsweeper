## Summary

Keep open. Current main still exposes vLLM through the bundled provider plugin's generic single-endpoint self-hosted setup and exact `models.providers.vllm` implicit-discovery suppression; it does not implement endpoint lifecycle management, `vllm-2` style managed endpoints, interactive multi-model selection, or stale managed-vLLM cleanup. The PR remains useful source work, but it is stale against the current provider-plugin setup architecture and still has blocking wiring/build/scope issues.

## What This Changes

This PR branch adds vLLM endpoint add/update/delete and multi-model selection code, stale vLLM default/override cleanup, related tests/docs, and unrelated changes to A2UI bundling, secrets/zizmor config, and memory SQLite test guards.

## Best Possible Solution

Keep the PR open for maintainer-guided rework: implement the endpoint lifecycle in the current vLLM provider plugin/setup contract, preserve generic plugin boundaries, add focused stale-default and multi-endpoint tests, update the vLLM docs, and split unrelated CI/script/security/test-infrastructure changes before review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route vLLM auth choices through the new handler:** `src/commands/auth-choice.apply.vllm.ts:114-121`
  - body: `applyAuthChoiceVllm` is added but never included in the `applyAuthChoice` handler list, while the generic plugin-provider handler runs first for `vllm`. Users selecting vLLM through auth/configure flows will bypass the new endpoint-management and stale-model cleanup behavior, so this needs to be wired before the generic handler or moved into the vLLM plugin setup method.
  - confidence: 0.93
- **[P2] Do not fail after a successful rolldown dlx fallback:** `scripts/bundle-a2ui.sh:95-103`
  - body: In the fallback branch, the script already runs `pnpm -s dlx rolldown -c ...`, then requires a persistent `.pnpm` CLI and runs rolldown again. Install layouts without a project-local `.pnpm` tree can now fail after the temporary `dlx` command succeeded, and pnpm layouts run the bundle twice. Resolve the command before execution or treat `dlx` as the terminal fallback.
  - confidence: 0.86
