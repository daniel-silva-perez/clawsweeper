## Summary

Keep open. Current main still lacks the requested vLLM endpoint-management flow, but this PR is not merge-ready because the new vLLM handler is not wired into production auth dispatch and the branch mixes unrelated build, test-infra, secret-scan, and workflow-scanner policy changes into the provider feature.

## What This Changes

The PR adds vLLM endpoint lifecycle setup, multi-endpoint provider keys, model discovery/selection, stale default cleanup, docs and tests, while also touching A2UI bundling, secret-scan baseline entries, memory SQLite test skipping, and zizmor configuration.

## Best Possible Solution

Rebase the provider work onto the current plugin-auth architecture, implement vLLM endpoint management in the bundled vLLM setup path that production auth and model-picker flows actually call, keep stale default/fallback cleanup covered by focused tests, update the public vLLM docs, and split build/security/test-infra changes into separate reviews.

## Reproduction Assessment

Yes. The main blocking behavior is reproducible by source inspection: PR head adds applyAuthChoiceVllm but the production applyAuthChoice path never registers it, so selecting vLLM still follows the generic plugin provider setup instead of the new endpoint menu.

## Solution Assessment

No. The requested vLLM behavior is a reasonable direction, but this PR is not the best merge shape until the handler is wired into the current dispatch path, the branch is rebased over the provider-auth refactor, and unrelated build/security/test policy edits are separated.
