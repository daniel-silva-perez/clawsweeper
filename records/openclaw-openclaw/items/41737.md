## Summary

Keep open. Current main still uses the generic single-endpoint vLLM setup, so the PR is not obsolete, but the branch is not merge-ready because the new vLLM handler is not wired into auth dispatch and the PR mixes unrelated build/security/test-infra changes into the provider feature.

## What This Changes

The PR adds vLLM endpoint add/update/delete flows, multi-model selection, stale vLLM default cleanup, related docs/tests, and also changes A2UI bundling, zizmor configuration, secrets baseline entries, and SQLite-gated memory tests.

## Best Possible Solution

Keep the useful vLLM endpoint-management work open for maintainer-guided rework: rebase onto the current plugin/provider setup code, wire the lifecycle flow through the actual vLLM auth path, preserve plugin boundaries, split unrelated build/security/test-infra changes, and validate the focused provider/docs/tests set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route vLLM auth choices through the new handler:** `src/commands/auth-choice.apply.vllm.ts:114-121`
  - body: The PR adds `applyAuthChoiceVllm`, but production `applyAuthChoice` never imports or invokes it before the generic plugin-provider handler. A normal `vllm` choice is therefore still handled by the generic setup path, so users skip the new lifecycle menu and stale-default cleanup.
  - confidence: 0.93
- **[P2] Treat pnpm dlx as the terminal rolldown fallback:** `scripts/bundle-a2ui.sh:95-103`
  - body: The fallback branch runs `pnpm -s dlx rolldown -c ...` and then still requires a persistent `.pnpm` rolldown CLI before running the bundle again. Layouts without `.pnpm` can fail after the temporary bundle succeeds, while pnpm layouts run the bundle twice.
  - confidence: 0.86
