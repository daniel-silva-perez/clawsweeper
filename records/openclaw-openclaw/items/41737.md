## Summary

Keep open. The vLLM product direction is still useful on current main, but the PR branch is stale against the current plugin-owned provider-auth architecture, its new vLLM handler is not wired into the production auth path, and it mixes unrelated build/security/test-policy changes into the provider feature.

## What This Changes

The PR adds vLLM endpoint lifecycle setup, multi-endpoint provider keys, model discovery/selection, stale default cleanup, docs and tests, while also touching A2UI bundling, secret-scan baseline entries, memory SQLite test skipping, and zizmor configuration.

## Best Possible Solution

Rebase the provider work onto current main, implement the endpoint-management flow through the bundled vLLM plugin/provider setup surface that production auth and model-picker flows already call, keep stale vLLM default/fallback cleanup covered by focused tests, update the public vLLM docs, and split build/security/test-infra edits into separate reviews.

## Reproduction Assessment

Yes. The main blocking behavior is reproducible by source inspection: PR head adds applyAuthChoiceVllm, but the production applyAuthChoice handler list does not register it, so selecting vLLM still follows the plugin-provider setup path instead of the new endpoint menu.

## Solution Assessment

No. The requested vLLM behavior is reasonable, but this PR is not the best merge shape until it is rebased onto the current plugin-auth architecture, wired through the actual vLLM provider setup path, and stripped of unrelated build/security/test-policy changes.
