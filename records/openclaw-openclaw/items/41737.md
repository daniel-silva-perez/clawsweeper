## Summary

Keep open. Current main still has only the generic single-provider vLLM setup path, and the PR branch is not merge-ready because the new vLLM handler is not wired into production dispatch and unrelated build/security policy changes are mixed into the provider feature branch.

## What This Changes

The PR branch adds vLLM endpoint-management setup helpers, default-model cleanup, docs and tests, while also modifying A2UI bundling, memory SQLite test skips, secret-scan baseline entries, and zizmor configuration.

## Best Possible Solution

Land the vLLM endpoint-management behavior as a focused provider/plugin setup change that is wired into the active auth-choice path, keeps stale default and fallback cleanup covered by targeted tests, updates the public vLLM docs, and moves build/security/test-infra changes into separate reviews.

## Reproduction Assessment

Yes. The blocking issues are reproducible by source and diff inspection: selecting vLLM still reaches the generic plugin-provider dispatch on current main, and the PR's A2UI fallback path can fail or run twice after pnpm dlx has already executed Rolldown.

## Solution Assessment

No. The product direction is reasonable, but this PR is not the best merge shape until the vLLM handler is actually invoked by production setup and unrelated build/security policy edits are split or separately approved.
