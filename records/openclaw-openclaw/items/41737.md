## Summary

Keep open. Current main still has the single-endpoint vLLM setup path, so the product direction remains useful, but this PR is not merge-ready because the new vLLM setup handler is not connected to production auth dispatch and the branch mixes unrelated build/test/security-policy changes into the provider feature.

## What This Changes

The PR branch adds vLLM endpoint-management setup helpers, multi-endpoint/stale-default cleanup, docs and tests, while also changing A2UI bundling, memory SQLite test skipping, secret-scan baseline entries, and zizmor configuration.

## Best Possible Solution

Land the vLLM endpoint-management behavior as a focused provider setup change: wire it into the active auth-choice path, preserve existing generic plugin contracts, keep stale default and fallback cleanup covered by targeted tests, update public vLLM docs, and split build/security/test-infra changes into separate reviews.

## Reproduction Assessment

Yes. The blocking issues are reproducible by source and diff inspection: selecting vLLM still reaches the generic plugin-provider dispatch before the new handler, and the PR's A2UI fallback can fail or run twice after pnpm dlx has already executed Rolldown.

## Solution Assessment

No. The requested provider behavior is reasonable, but this PR is not the best merge shape until the vLLM handler is actually invoked by production setup and the unrelated build/security/test-policy edits are split or explicitly approved.
