## Summary

Keep open. Current main still lacks the requested vLLM endpoint lifecycle flow, so the PR is not obsolete, but the branch should not merge as-is: its new vLLM auth handler is not wired into the production dispatch path, the branch is currently dirty against main, and it mixes the provider feature with unrelated build/security/test-infrastructure changes.

## What This Changes

The PR branch adds vLLM endpoint add/update/delete and multi-model selection code, stale vLLM default/override cleanup, related docs/tests, and also touches A2UI bundling, zizmor config, secrets baseline, and SQLite-gated memory tests.

## Best Possible Solution

Keep the useful vLLM endpoint-management work open for maintainer-guided rework: rebase onto current provider-plugin/model-picker code, wire the lifecycle flow through the actual vLLM plugin setup/auth path, preserve generic plugin boundaries, split unrelated build/security/test-infra changes, and then validate the focused provider/docs/tests change set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wire the vLLM handler into auth dispatch:** `src/commands/auth-choice.apply.vllm.ts:114-121`
  - body: The PR adds `applyAuthChoiceVllm`, but `applyAuthChoice` never imports or inserts it; the first handler is still the generic plugin-provider handler, which resolves `vllm` and returns first. Users selecting vLLM through configure/auth-choice flows therefore stay on the generic single-endpoint setup and skip the new lifecycle and stale-default cleanup behavior.
  - confidence: 0.93
- **[P2] Treat pnpm dlx as the rolldown fallback:** `scripts/bundle-a2ui.sh:95-103`
  - body: In the fallback branch, the script runs `pnpm -s dlx rolldown -c ...` and then still requires a persistent `.pnpm` rolldown CLI before running the bundle again. Install layouts without `.pnpm` can fail after the temporary dlx bundle succeeds, while pnpm layouts run the bundle twice.
  - confidence: 0.86
