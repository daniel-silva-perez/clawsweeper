## Summary

Keep this PR open. Current main still has the Discord line-count default the PR is trying to change, but the proposed patch is not mergeable as-is: the shared Markdown chunker still has concrete fence-boundary correctness bugs, the public SDK export needs baseline handling, and public Discord default docs/config hints would be stale after the behavior change.

## What This Changes

The PR changes Discord reply chunking to use character-first splitting when `maxLines` is omitted, extracts a shared balanced Markdown fence-aware chunker into `src/markdown`, re-exports it through plugin SDK reply subpaths, and adds Discord/shared regression tests.

## Best Possible Solution

Finish this PR in place by fixing the shared Markdown chunker's fence boundary handling, using the adjusted split index consistently, keeping the plugin SDK API baseline/export contract aligned, updating the public Discord default docs and config hints if the character-first default is intentional, then rebasing and running focused chunking/API checks plus the changed gate.

## Reproduction Assessment

Yes. The user-visible issue is reproducible on current main by calling `chunkDiscordText` with many lines under the character limit and no explicit `maxLines`; the helper still falls back to `DEFAULT_MAX_LINES = 17`. The PR blockers are reproducible by static inspection of the PR head source around `splitRemainingChunk`, `chooseBreakForRemaining`, and the new SDK export.

## Solution Assessment

No, not yet. The direction is maintainable because Discord keeps explicit line-mode behavior while generic fence-aware Markdown chunking moves to a shared helper, but the current patch still needs correctness, SDK baseline, docs, and rebase work before it is the best mergeable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment and return `safeBreakIdx` to consume one more character for progress, but this caller ignores the returned index and slices the emitted chunk with the original `breakIdx`. When that branch fires, the consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure the returned index and use it for `remaining.slice`.
  - confidence: 0.86
- **[P2] Keep splits out of fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: The reserve-adjustment loop can shrink `effectiveMaxChars` so the next preferred break lands inside a fence opener. `splitRemainingChunk` then suppresses carry-over and emits an unbalanced opener fragment even when an opener-plus-close chunk fits, breaking the balanced-fence helper contract for small valid limits.
  - confidence: 0.78
- **[P2] Update the SDK API baseline hash:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to the public `openclaw/plugin-sdk/reply-chunking` surface, but the PR file list does not include the generated API baseline update. The plugin SDK API check will report drift until the generated baseline hash is committed for the new export.
  - confidence: 0.9
- **[P3] Align public Discord default documentation:** `extensions/discord/src/chunk.ts:39-40`
  - body: The patch changes omitted `maxLines` to character-first splitting, but the current public config docs and UI hints still describe the Discord default as `maxLinesPerMessage: 17`. If the behavior change is intentional, update those surfaces so users are not told to expect the old line-splitting default.
  - confidence: 0.75
