## Summary

Keep open. Current main still maps the reported Anthropic/Pi SDK `api_error: Internal server error` shape into the timeout path, and timeout reasons are deliberately not persisted as auth-profile cooldowns. That means the existing same-provider cooldown/probe skip path does not reliably address the issue's central fast-fail request. The related fix PR in the provided context was closed unmerged.

## Best Possible Solution

Keep this issue open and implement a focused provider-failover fix: classify the Anthropic/Pi SDK `api_error: Internal server error` shape as a cooldown-worthy transient provider outage, likely `overloaded`, so same-provider profiles/candidates are bounded and cross-provider fallback can proceed quickly. Add regression coverage for plain SDK text, JSON api_error payloads, cooldown persistence, and cron-visible failure/notification behavior; handle broader cron catch-up or gap detection as follow-up work if needed.

## Risks / Open Questions

- The issue mixes three product behaviors: provider failover fast-fail, cron retry/catch-up, and delivery notification. Current main has some cron retry/alert surfaces, so the implementation should keep the provider classification/cooldown fix focused and avoid treating partial cron visibility work as the full resolution.
- Changing `internal server error` classification too broadly could incorrectly poison shared provider auth health. The safer fix is scoped to provider/API error shapes such as Anthropic/Pi SDK `api_error: Internal server error`, with regression tests preserving non-transient api_error and bare generic 5xx behavior.
