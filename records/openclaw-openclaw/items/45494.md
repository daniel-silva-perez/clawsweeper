## Summary

Keep open. Current main still routes the reported provider internal-server/api_error shape through the timeout path, and timeout classifications intentionally do not mark auth-profile cooldown, so the exact fast-fail behavior requested here is not implemented. Related PRs now touch the same server-error/cooldown boundary, which makes this a maintainer policy decision rather than a cleanup close.

## What This Changes

The issue requests cron agent jobs to fast-fail, retry/back off, and notify configured delivery targets when sustained LLM provider 500/api_error outages would otherwise consume the full cron timeout window.

## Best Possible Solution

Define a narrow provider-outage failover contract for Anthropic/Pi SDK `api_error: Internal server error` that bounds same-provider retries/profile rotation, reaches cross-provider fallbacks promptly, and lets the existing cron retry and failure-notification machinery run after the agent turn returns. The implementation should preserve generic proxy/CDN 5xx, transport timeout, auth, billing, format, and local-helper behavior, with focused classifier, runner-policy, and cron regression coverage.

## Reproduction Assessment

Yes for the focused failure path: inspect or exercise `classifyFailoverReason("LLM error api_error: Internal server error")`, observe it follows the current internal-server timeout lane, then observe `resolveAuthProfileFailureReason({ failoverReason: "timeout" })` returns null. The full live Anthropic outage plus 15-minute cron schedule was not reproduced here.

## Solution Assessment

No for current main: the existing implementation improves observability and cron timeout cleanup, but it still does not fast-fail the reported provider 500/api_error shape. The safer direction is a lower-level failover classification and runner/profile policy fix first, with cron scheduler changes only if the agent turn still cannot return promptly.
