## Summary

Keep open. Current main still has the reported failure shape in the failover path: provider-side internal-server `api_error` text is classified as `timeout`, and timeout failover reasons are deliberately excluded from auth-profile cooldown, so the closed unmerged fix PR remains useful context rather than proof of an implemented fix.

## What This Changes

The issue asks for cron agent jobs to fast-fail, retry/back off, and notify on sustained LLM provider HTTP 500 `api_error` outages instead of consuming the full cron timeout window.

## Best Possible Solution

Use the existing failover and auth-profile cooldown machinery for this provider-outage class: classify Anthropic/Pi SDK internal-server `api_error` responses as provider-busy/overloaded or an equivalent cooldown-eligible reason, prove same-provider retries are capped, and keep broader cron catch-up or gap-detection semantics as separate maintainer-scoped work if still desired.

## Reproduction Assessment

Yes for the core failure mode: current source shows `LLM error api_error: Internal server error` matching the timeout/server-error path, and timeout reasons skip shared auth-profile cooldown. The full 180-second cron timing still needs either a live outage or a focused fixture test.

## Solution Assessment

Yes, a narrow classifier/cooldown repair is the best first fix because it uses existing overload rotation, cooldown, and fallback behavior instead of adding cron-specific provider policy. The broader retry/backfill notification ideas should be evaluated after that path is proven.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/45494 with a narrow model-failover repair. Current main classifies Anthropic/Pi SDK `LLM error api_error: Internal server error` and JSON `api_error` internal-server payloads as `timeout`; timeout failover reasons skip auth-profile cooldown, so same-provider retry cycles can consume a cron job timeout. Adjust the failover classification/cooldown path so provider-side internal-server `api_error` outage signals use the existing overloaded/provider-busy behavior or another cooldown-eligible reason, causing same-provider retries to be capped and cross-provider fallback to proceed promptly. Preserve exclusions for billing, auth, context overflow, schema/format errors, and generic raw proxy 500s that should remain timeout. Add targeted classifier tests for both the plain SDK string and JSON form, plus focused auth-profile/model-fallback coverage proving cooldown/overloaded rotation reaches cross-provider fallback without exhausting all same-provider profiles. Do not implement broad cron catch-up, gap detection, or product notification redesign in this PR. Update user-facing docs or changelog only if the observable failover behavior changes.
