## Summary

Keep open. Current main still routes Anthropic/Pi SDK `api_error: Internal server error` through the timeout path, and timeout classifications are deliberately excluded from auth-profile cooldown, so the same-provider retry/cooldown failure described here is still plausible. The targeted fix PR #45625 is closed unmerged, and the current code/tests still preserve timeout behavior for transient `api_error` internal-server payloads.

## What This Changes

The issue asks cron agent jobs to fast-fail, retry with backoff, and surface delivery notifications when LLM provider or fallback API errors indicate a sustained outage instead of consuming the full cron timeout window.

## Best Possible Solution

End state should be a small failover-classification and regression-test change that makes Anthropic/Pi SDK `api_error: Internal server error` cooldown-worthy enough to bound same-provider profile rotation and reach cross-provider fallback, while preserving generic server-error timeout semantics and using existing cron retry/delivery behavior once the run returns.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the focused failure path: pass `LLM error api_error: Internal server error` through the current classifier, observe it lands as `timeout`, then observe timeout is dropped by auth-profile failure policy. The full provider-outage cron timing was not live-reproduced in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrowest maintainable fix is to adjust failover classification/cooldown semantics for the Anthropic/Pi SDK API-error shape and add targeted regression coverage. A cron scheduler rewrite would not address the root cause until the agent/fallback run returns quickly enough.
