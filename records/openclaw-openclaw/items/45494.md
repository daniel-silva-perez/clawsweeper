## Summary

Keep open. Current main still classifies Anthropic-style internal-server `api_error` text as a timeout, and timeout-shaped failures intentionally skip auth-profile cooldown, so the root failure mode described in the issue is still plausible. The closed related PR #45625 was not merged, and its review discussion points to a narrower safe repair than adding a broad `internal server error` overload pattern.

## What This Changes

The issue asks cron agent jobs to fail faster and recover visibly when sustained LLM provider 500 errors would otherwise consume the full job timeout.

## Best Possible Solution

Make a narrow failover-classification repair: treat Anthropic/API-shaped `api_error` internal-server responses as `overloaded` so the existing overloaded cooldown and profile-rotation cap can move cron runs through the fallback chain quickly, while preserving raw provider-agnostic HTTP 500 responses as timeout. Update focused tests and the model-failover docs to reflect the new boundary.

## Reproduction Assessment

Yes. A focused static reproduction is available: current main expects JSON `api_error` internal-server payloads to classify as `timeout`, SDK-style `LLM error api_error: Internal server error` misses the JSON helper, and timeout failures are not persisted as auth-profile cooldowns.

## Solution Assessment

Yes for the narrow root cause. The best fix is a targeted classifier/test/docs update that reaches the existing overloaded failover machinery without broadening all `internal server error` or raw HTTP 500 text into provider cooldowns.

## Repair Work Prompt

Fix the cron/provider-outage failover regression described in openclaw/openclaw#45494. Current main classifies Anthropic/API `api_error` internal-server responses as `timeout`, and timeout failures do not mark auth-profile cooldown, so cron runs can burn the full job timeout instead of reaching cross-provider fallback quickly.

Implement the narrow fix in the failover classifier. Treat API-shaped internal-server responses, including JSON `{"type":"api_error","message":"Internal server error"}` and SDK/plain `LLM error api_error: Internal server error`, as `overloaded` so existing overloaded cooldown/profile-rotation caps apply. Preserve raw provider-agnostic `500 Internal Server Error` as `timeout`; do not add a bare `internal server error` overloaded substring that affects all providers. Keep billing/auth/context-overflow exclusions intact.

Add focused regression coverage in `src/agents/pi-embedded-helpers.isbillingerrormessage.test.ts` for JSON and SDK formats returning `overloaded`, plus raw HTTP 500 returning `timeout`. Add or adjust an embedded fallback/auth-profile test if needed to prove the overloaded path records cooldown and caps same-provider rotations. Update `docs/concepts/model-failover.md` so it no longer documents these API internal-server payloads as timeout if behavior changes. Add a user-facing changelog fix entry with reporter/contributor credit, following repo changelog rules. Do not change cron scheduling/backoff semantics unless a failing test proves the classifier fix is insufficient.
