## Summary

Keep open. Current main still classifies Anthropic/Pi SDK `api_error: Internal server error` shapes as `timeout`, and timeout classifications are deliberately excluded from auth-profile cooldown, so repeated same-provider retry/cooldown stalls remain plausible. The focused fix PR #45625 is closed unmerged, while cron retry and failure delivery only help after the agent run returns.

## What This Changes

The issue asks cron agent runs to fast-fail, retry with backoff, and notify on sustained LLM provider API outages instead of consuming the full cron timeout window.

## Best Possible Solution

Make a small failover-classification and regression-test change so Anthropic/Pi SDK `api_error: Internal server error` becomes overload/profile-cooldown-worthy enough to bound same-provider rotation and reach cross-provider fallback, while preserving generic raw 500, CDN/proxy, and transport-timeout semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the focused failure path: pass `LLM error api_error: Internal server error` through the current classifier, observe `timeout`, then observe timeout is dropped by auth-profile failure policy. The full provider-outage cron timing was not live-reproduced in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable fix is to adjust failover classification/cooldown semantics for the Anthropic/Pi SDK API-error shape and add targeted regression coverage. A cron scheduler rewrite would not address the root cause until the agent/fallback run returns quickly enough.
