Keep this PR open. Current main still lacks a plugin status-provider API or TUI footer consumer, so this is not obsolete, but the latest PR head still has merge-blocking public-SDK and registry issues plus an unresolved maintainer decision about whether the API seam should land before the render path.

The PR adds a public `PluginStatusProvider`/`registerStatusProvider` API, stores status providers in the plugin registry, exports the type through the plugin SDK, updates docs and test fixtures, and adds registry tests for duplicate and blank ids.

Rebase onto current main, settle the public footer-status contract, update the current SDK/test helper and generated API baseline, store normalized provider ids with a regression test, add the required changelog entry, and then either include the TUI consumer or link a concrete follow-up for the render path.

## Solution Assessment
