## Summary

Keep open. Current main still lacks a plugin footer-status seam, so the PR remains relevant to #64086 and the ContextClaw use case, but the branch is not merge-ready: it misses at least one typed registry fixture, registers duplicate status-provider ids silently, and exposes a new public SDK method without matching docs.

## What This Changes

The PR adds a `PluginStatusProvider` type, `api.registerStatusProvider(...)`, `PluginRegistry.statusProviders` storage, SDK type re-export plumbing, and selected registry/test-helper updates.

## Best Possible Solution

Keep the PR as useful implementation input, but require updates before merge: cover every typed registry fixture, validate and de-duplicate status-provider ids like other id-bearing registrations, document the new public SDK method, and get maintainer agreement on whether the render-path contract lands here or in a clearly linked follow-up.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection on current main confirms the current gap: the TUI footer is fixed and `OpenClawPluginApi` has no status-provider registration method.

## Solution Assessment

Unclear as-is. A narrow core plugin API seam is a plausible owner boundary for footer status contributions, but this patch is not yet the best mergeable form because it has compile/validation/docs defects and open render-contract decisions.
