## Summary

Keep this PR open, but do not merge this revision as-is. Current main still lacks a plugin status-provider API and still renders a fixed TUI footer, so the work remains relevant; however, the diff has blocking registry-contract and API-quality issues that need author or maintainer follow-up.

## What This Changes

The PR adds a `PluginStatusProvider` type, a `registerStatusProvider` method on `OpenClawPluginApi`, registry storage for status providers, SDK re-export plumbing, and related test-helper registry updates.

## Best Possible Solution

Keep the PR as a useful implementation candidate, but require updates before merge: make the registry shape compile everywhere, validate and de-duplicate status provider ids like other id-bearing registrations, document the new public SDK method, and settle whether the render-path contract should land here or in a clearly linked follow-up.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection on current main confirms the current gap: the TUI footer is fixed and `OpenClawPluginApi` has no status-provider registration method.

## Solution Assessment

Unclear as-is. A core plugin API seam is a plausible owner boundary for plugin footer contributions, but this patch is not yet the best mergeable form because it has a required-registry-field breakage, no duplicate-id guard, stale public SDK docs, and unresolved render-contract questions.
