## Summary

Keep open. The reported collision still exists on current main, and removing the plugin-facing wrapper outright crosses a published plugin SDK surface, so this needs maintainer/API compatibility judgment rather than a cleanup close.

## What This Changes

The issue requests removing or deprecating colliding `normalizeChannelId` wrapper exports and updating callers to use `normalizeChatChannelId` and `normalizeAnyChannelId` directly.

## Best Possible Solution

Adopt canonical internal names first, expose `normalizeAnyChannelId` where useful, and keep or explicitly deprecate the public `normalizeChannelId` SDK alias until maintainers approve a compatibility-breaking removal with contract/docs updates.

## Reproduction Assessment

Yes for the reported source-level collision: current-main static inspection shows both exported wrappers and the aliased import site still present. This is a refactor/API clarity issue, not a runtime failure reproduction.

## Solution Assessment

Unclear as proposed: replacing internal imports is a narrow maintainable cleanup, but outright export removal is not clearly the best solution because the name is part of `openclaw/plugin-sdk/channel-targets` and `channel-runtime`. A staged deprecation is safer unless maintainers intentionally approve a breaking SDK change.
