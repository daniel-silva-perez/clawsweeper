## Summary

Keep this PR open. Current main still ignores `ANTHROPIC_BASE_URL` in both Anthropic Messages streaming and native PDF requests, while the latest PR head fixes the earlier PDF URL-normalization defects by reusing the Messages URL resolver; the remaining issue is a coverage and module-boundary review question rather than a reason to close it as obsolete.

## What This Changes

The PR exports and tests an Anthropic Messages URL resolver with explicit `baseUrl` > `ANTHROPIC_BASE_URL` > public Anthropic endpoint precedence, then reuses that resolver for native PDF Anthropic requests.

## Best Possible Solution

Land the provider-routing fix after adding native PDF regression coverage and confirming the helper boundary is acceptable. The finished behavior should keep explicit `baseUrl` precedence, honor only non-empty `ANTHROPIC_BASE_URL`, avoid duplicate `/v1`, and preserve the default public Anthropic endpoint when neither input is set.

## Reproduction Assessment

Yes. The current-main reproduction is source-level and direct: set `ANTHROPIC_BASE_URL` without `params.baseUrl` or `model.baseUrl`, then the streaming resolver and native PDF request still construct `https://api.anthropic.com/v1/messages`.

## Solution Assessment

Mostly yes, but not complete. Reusing one URL resolver is the narrow maintainable direction; adding PDF-specific tests, or extracting the resolver to a small shared helper, would make the PR ready for normal maintainer review.
