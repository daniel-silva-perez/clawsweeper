## Summary

Keep this PR open. Current main still sends outbound reply text through the shared payload plan without stripping echoed inbound metadata, so the PR’s intended fix is not implemented. The PR remains a plausible fix for #39847, but maintainers should choose a canonical implementation because #51653 is also open for the same bug and the sanitizer choice still needs refinement.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to choose a canonical PR between this one and #51653, rebase onto the current `createOutboundPayloadPlanEntry()` shape, strip echoed inbound metadata before reply directive parsing with an outbound-safe leading-only sanitizer where possible, and add regression tests proving metadata echoes are removed while timestamp-prefixed normal replies and intentionally quoted metadata examples are preserved. After one implementation lands, close the duplicate PR and #39847 as implemented.

## Risks / Open Questions

- Functional risk: applying the broad `stripInboundMetadata()` helper to all assistant outbound text can delete intentionally quoted metadata examples or debugging output because the helper strips matching blocks throughout the message, not only an echoed prompt prefix.
- Behavioral risk: this PR also changes shared `stripInboundMetadata()` timestamp behavior, which affects other call sites such as command detection, send policy parsing, session cost usage, TUI/gateway sanitization, replay history, and memory/session file helpers.
- Security review: the PR diff touches TypeScript source and tests only, with no CI workflow, dependency, lockfile, package publishing, install script, downloaded artifact, secret-handling, or permission changes. The security-relevant intent is positive because it reduces internal metadata leakage, but the exact sanitizer should be scoped carefully.
