## Summary

Keep open. Current main still parses and delivers outbound reply text without any inbound-metadata sanitizer in the shared payload plan, so the PR’s intended fix is not implemented. The PR remains a plausible implementation candidate for #39847, but it needs maintainer refinement/rebase because the proposed broad `stripInboundMetadata()` use can alter legitimate assistant output and a newer same-author PR is also open for the same bug.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to choose a canonical PR, rebase it onto the current payload-plan implementation, add outbound sanitization before reply directive parsing, prefer an assistant-safe or leading-only metadata stripper, and add regression tests that prove echoed metadata is removed while timestamp-prefixed and intentionally quoted metadata text is preserved. Once one implementation lands, close the duplicate PR and #39847 as implemented.

## Risks / Open Questions

- Applying `stripInboundMetadata()` to every assistant outbound payload can delete intentionally quoted metadata examples or debugging output because the helper strips matching blocks throughout the text, not only echoed prompt prefixes.
- Current main’s sanitizer still strips OpenClaw-format timestamp-only text; the PR’s second commit addresses that on the branch, but the shipped helper behavior shows why outbound regression coverage is needed.
- This branch appears stale against current main’s payload-plan shape: current main centralizes normalization in `createOutboundPayloadPlanEntry()`, while the provided PR diff edits an older direct `normalizeReplyPayloadsForDelivery()` loop.
- A newer same-author PR (#51653) is open for the same linked bug, so maintainers should decide which implementation to carry forward before closing either contributor thread.
