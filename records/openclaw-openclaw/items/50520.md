## Summary

Keep this PR open. Current main still parses outbound reply text without stripping echoed inbound metadata, so the PR's intended fix is not implemented. The PR is a plausible implementation candidate for the open #39847 bug, but it is also paired with same-author follow-up PR #51653 and still needs maintainer choice on the sanitizer scope.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to choose a canonical PR between this one and #51653, rebase it onto the current createOutboundPayloadPlanEntry() shape, strip echoed inbound metadata before reply directive parsing with an outbound-safe leading-only sanitizer where possible, and add regression tests proving metadata echoes are removed while timestamp-prefixed normal replies and intentionally quoted metadata examples are preserved. After one implementation lands, close the duplicate PR and #39847 as implemented.

## Risks / Open Questions

- Functional risk if this lands as-is: applying broad stripInboundMetadata() to every assistant outbound payload can remove intentionally quoted metadata examples or debugging text that contains matching sentinel blocks away from the leading echoed prompt area.
- Behavioral risk in the PR's second commit: changing stripInboundMetadata() timestamp behavior affects existing non-outbound callers such as command detection, send policy parsing, session-cost usage, TUI/history sanitization, and memory/session file helpers.
- Process risk: there are two open same-author PRs for the same #39847 bug, so maintainers should pick a canonical implementation rather than auto-closing one half of the pair.
