## Summary

Keep this PR open. Current main still builds the outbound payload plan from raw reply text, so echoed inbound metadata is not stripped before directive parsing or user-visible channel delivery. The PR is a plausible implementation candidate for the open #39847 bug, but it is also paired with same-author PR #51653 and has unresolved sanitizer-scope review concerns that need maintainer choice before one implementation is landed and the duplicate is closed.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to choose a canonical implementation between this PR and #51653, rebase it onto the current createOutboundPayloadPlanEntry() flow, strip echoed inbound metadata before reply directive parsing with the narrowest outbound-safe sanitizer, and add regression tests proving metadata echoes are removed while timestamp-prefixed normal replies and intentionally quoted metadata examples are preserved. After one patch lands, close the duplicate PR and #39847 as implemented.

## Risks / Open Questions

- The user-visible metadata leak remains present on current main until one implementation lands.
- Landing this PR as-is may strip intentionally quoted metadata examples or debugging text because stripInboundMetadata() removes matching blocks anywhere in the output, not only leading echoed prompt metadata.
- The PR's shared stripInboundMetadata() timestamp behavior change affects non-outbound callers such as command detection, send-policy parsing, history sanitization, and session-cost paths.
- There are two open same-author PRs for the same #39847 bug, so maintainers need to choose a canonical patch rather than auto-closing one side of the pair.
