## Summary

Keep open. Current main still builds the outbound delivery plan from raw reply text, while this PR and the same-author #51653 are overlapping dirty implementation candidates that need maintainer choice and a rebase onto the current payload-plan flow.

## What This Changes

The PR imports `stripInboundMetadata` into outbound payload normalization, calls it before reply directive parsing, adds an outbound regression test, and changes shared timestamp-only sanitizer expectations.

## Best Possible Solution

Choose one canonical implementation for #39847, rebase it onto `createOutboundPayloadPlanEntry()`, sanitize echoed inbound metadata before directive parsing with an outbound-safe leading-prefix helper or new narrow helper, and add regression coverage for stripped echoes, preserved timestamp-only replies, and intentionally quoted metadata examples.

## Reproduction Assessment

Yes. A `ReplyPayload` whose text starts with `Conversation info (untrusted metadata):` followed by real response text still flows through `parseReplyDirectives(payload.text ?? "")` on current main and then into channel delivery without metadata stripping.

## Solution Assessment

No, not as-is. The PR addresses the right delivery gap, but the current diff applies the broad shared sanitizer to every outbound payload and changes shared timestamp-only behavior instead of keeping the fix local to outbound delivery.
