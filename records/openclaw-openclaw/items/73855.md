## Summary

Keep open. Current main still has no hosted-installer checksum guard, but the latest PR head still contains the two blocking workflow issues already called out: it compares pull-request installer source changes against the already-deployed hosted script, and scheduled runs share a cancel-in-progress group with main pushes. The author’s follow-up comment says both were fixed, but the raw PR head and files API still show the original lines, and the new workflow is currently failing on the PR head.

## What This Changes

The PR adds a GitHub Actions workflow that fetches `https://openclaw.ai/install.sh`, compares its SHA-256 digest with `scripts/install.sh`, and runs on selected pushes, pull requests, a daily schedule, and manual dispatch.

## Best Possible Solution

Keep the drift guard idea, but make the live source-vs-host checksum run only after merge, on schedule, or by manual dispatch. Pull-request validation should be workflow lint/sanity or a PR-produced expected artifact, and the concurrency group should distinguish event names so daily drift checks are not cancelled by main pushes.

## Reproduction Assessment

Not applicable as a user-facing bug reproduction, but yes for the introduced CI failure mode: the raw PR head still has the problematic trigger/concurrency lines, the live source and hosted hashes differ, and the PR’s checksum workflow is failing in GitHub Actions.

## Solution Assessment

No. A read-only checksum guard is a reasonable mechanism for future drift, but this implementation is not the best mergeable form because it compares in-flight PR source against deployed bytes and can suppress scheduled alerts through shared concurrency.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Limit live-host comparisons on pull requests:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change `scripts/install.sh`, but a PR checkout contains the proposed installer while `openclaw.ai` still serves the last deployed installer. Because the job fails on any hash mismatch, future installer-source PRs will fail until after merge and deployment. Remove this path from `pull_request` validation, or skip the live checksum comparison for pull_request events.
  - confidence: 0.97
- **[P2] Keep scheduled runs out of push cancellation groups:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: `schedule` and `push` runs on `main` both use `refs/heads/main` for `github.ref`, so `cancel-in-progress: true` can let a main push cancel the daily drift check. Include `github.event_name` in the concurrency group or otherwise avoid cancelling scheduled checks from push runs.
  - confidence: 0.91
