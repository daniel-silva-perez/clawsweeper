## Summary

Keep this issue open. Current main still sends MCP stdio JSON-RPC frames unsigned, has no supported `mcps`/signing config surface, and the plugin/bundle path contributes MCP server definitions rather than a documented transport-wrapper hook. The related PR #49182 was closed unmerged after maintainer security concerns, and the reporter's follow-up explicitly raised that a plugin rewrite may need a new transport-level API seam.

## Best Possible Solution

Keep this issue open for maintainer security/API design. The best path is either a generic, opt-in MCP stdio signing/decorator seam with documented config and tests for replay, forgery, stripping, downgrade, key handling, and strict-mode failures, or an explicit maintainer decision that MCPS remains external plus a documented plugin API that can safely wrap OpenClaw-managed stdio frames.

## Risks / Open Questions

- Leaving the issue open does not mean the previous PR should be revived as-is; maintainer feedback on #49182 identified concrete security issues in that implementation.
