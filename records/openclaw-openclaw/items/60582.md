## Summary

Keep open. Current main has a parent-fallback resolver for Discord channel configs, but normal guild-message preflight only passes a parent ID for thread channels, not for ordinary text channels under a Discord category. The cross-referenced #8972 is also not a clean duplicate: its existing review comment explicitly keeps category inheritance scoped to this issue unless maintainers decide to combine the designs.

## Best Possible Solution

Keep this issue open and implement a documented Discord category allowlist path in the Discord plugin. The likely fix is to feed fetched non-thread channel `parentId`/parent metadata into the existing fallback resolver, decide whether the public config should be `categories`, bare category IDs in `channels`, or `category:<id>`, preserve direct channel overrides over category defaults, and add schema/docs/preflight tests proving new child channels inherit without per-channel config edits.

## Risks / Open Questions

- Implementing category-level allowlists changes Discord access-control behavior; maintainers should define precedence explicitly, likely direct channel override first, then category/parent match, then wildcard/guild fallback.
- #8972 overlaps the dynamic-channel use case, but its current discussion separates glob/pattern support from category inheritance, so closing this issue now would lose a focused tracker.
