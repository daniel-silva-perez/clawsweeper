## Summary

Keep open. Current main has a generic parent-fallback resolver and preserves Discord channel parent IDs, but normal guild-message preflight still only passes thread-parent metadata into allowlist resolution, so ordinary text channels under a Discord category do not inherit category allowlist settings. Related #8972 overlaps the dynamic-channel use case, but it remains an open pattern/glob request rather than a canonical tracker for category-specific inheritance.

## What This Changes

Request category-level Discord guild allowlist entries so settings such as `requireMention` inherit to child channels without per-channel config edits or gateway restarts.

## Best Possible Solution

Implement and document category inheritance inside the Discord plugin by feeding fetched non-thread channel parent metadata into the existing fallback resolver, preserving direct channel overrides over category defaults, and adding focused schema/docs/preflight tests for newly-created child channels under an allowed category.

## Reproduction Assessment

Yes. A current-main code-level reproduction is a guild text channel whose fetched `channelInfo.parentId` matches a configured category/parent entry while the direct channel is absent; preflight does not pass that non-thread parent ID to the resolver, so the category config is not applied.

## Solution Assessment

Unclear. Reusing the existing parent-fallback resolver is likely the narrowest implementation, but maintainers should first choose and document the public category config shape because the issue proposes `category:<id>` while current matching would naturally support bare parent IDs once supplied.
