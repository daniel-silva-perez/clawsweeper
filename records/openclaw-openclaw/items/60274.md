## Summary

Keep this issue open. Current main still does not filter imsg RPC tapback/reaction payloads before dispatch, and an open same-author implementation PR explicitly targets this issue, so conservative cleanup should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests that the bundled iMessage plugin stop delivering imsg RPC tapback reactions such as `Loved "..."` to the agent as ordinary inbound messages.

## Best Possible Solution

Land #60677 or an equivalent narrow change in `extensions/imessage`: preserve upstream imsg reaction metadata, drop tapback/reaction events before normal dispatch, add regression tests for metadata and quoted-text cases, and keep full reaction-event forwarding tracked separately by #39031.

## Reproduction Assessment

Yes. A code-level reproduction is clear: a valid inbound imsg RPC message with body text like `Loved "hello"` reaches `resolveIMessageInboundDecision()` and can dispatch when normal sender/access gates allow, because current main has no tapback/reaction branch; live iMessage testing was not run in this read-only review.

## Solution Assessment

Yes, keeping this issue open is the best cleanup action because #60677 is an active implementation candidate that explicitly closes it. The maintainable fix should stay in the bundled iMessage plugin and should not treat the broader reaction-event forwarding feature as completed.
