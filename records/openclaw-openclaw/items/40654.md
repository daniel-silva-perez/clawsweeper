## Summary

Keep open. The requested Discord ACP-thread archival behavior is still absent from current main, and the PR is a useful implementation candidate, but the patch is not mergeable as-is because it puts Discord close behavior in ACP core and adds delivery code under retired Discord paths while current main’s active Discord implementation lives in the bundled plugin.

## What This Changes

The PR records ACP-created Discord thread ownership metadata and tries to archive those threads after `/acp close` closes and unbinds the ACP session.

## Best Possible Solution

Add a small generic channel/conversation-binding close hook or typed reply-payload contract, persist ACP-owned child-thread metadata through the existing binding metadata path, and implement archive-after-reply entirely in `extensions/discord` while preserving runtime notices, partial-failure messaging, and unchanged non-Discord/manual-thread behavior.

## Reproduction Assessment

Yes. Static current-main inspection is enough: `/acp close` only closes and unbinds, docs describe that behavior, and targeted search found no active archive-after-reply implementation.

## Solution Assessment

No. The requested outcome is reasonable, but this PR is not the best implementation because it targets retired Discord paths and hardcodes Discord-specific close behavior in ACP core instead of using an additive plugin-owned seam.
