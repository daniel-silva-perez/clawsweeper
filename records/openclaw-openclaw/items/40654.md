## Summary

Keep open: the requested Discord ACP thread cleanup is still missing on current main, and this PR is useful source work but is not mergeable as-is because it targets retired Discord core paths and puts Discord-specific close behavior in ACP core instead of the active plugin boundary.

## What This Changes

The PR records ACP-created Discord thread ownership metadata, changes `/acp close` to emit an archive-after-reply Discord payload, adds a Discord archive REST helper, and expands ACP close, outbound payload, and thread-binding tests.

## Best Possible Solution

Add a small additive channel/conversation-binding close hook or typed reply-payload contract, persist ACP-owned child-thread metadata in the existing binding metadata path, and implement archive-after-reply entirely in `extensions/discord` while preserving runtime notices, partial-failure messaging, and unchanged non-Discord/manual-thread behavior.

## Reproduction Assessment

Yes. Static current-main inspection is enough: `/acp close` only closes and unbinds, docs/tests describe close as removing bindings, and targeted search finds no active archive-after-reply implementation.

## Solution Assessment

No. The requested behavior is reasonable, but this PR is not the best implementation because it hardcodes Discord behavior in ACP core and edits retired Discord paths; a plugin-owned port behind an additive core seam is the safer path.
