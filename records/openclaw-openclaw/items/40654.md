## Summary

Keep open. Current main does not implement archiving ACP-created Discord threads on `/acp close`, and the PR still represents useful behavior, but the branch is stale against the current plugin layout and needs a current-main port before merge.

## What This Changes

The PR records ACP-created Discord thread ownership metadata, emits an archive-after-reply Discord payload from `/acp close`, adds a Discord archive REST helper, and expands ACP close, outbound payload, and thread-binding tests.

## Best Possible Solution

Add an additive channel/conversation-binding close hook or equivalent typed payload contract, persist ACP-owned child-thread metadata, and implement archive-after-reply inside `extensions/discord` while preserving runtime notices, partial-failure messaging, and unchanged non-Discord/manual-thread behavior.

## Reproduction Assessment

Yes. Static current-main inspection is enough to reproduce the missing behavior: `/acp close` only closes/unbinds, and targeted grep finds no archive-after-reply implementation in active ACP or Discord plugin paths.

## Solution Assessment

No. The requested behavior is reasonable, but this PR’s current implementation is not the best way because it hardcodes Discord behavior in ACP core and edits retired Discord paths; a plugin-owned current-main port is the maintainable path.
