## Summary

Current main still does not implement this PR’s intended behavior. `/acp close` closes the ACP session and unbinds targets, but it does not detect ACP-created Discord child threads or request a post-reply archive. The PR remains a plausible implementation candidate, though it needs to be ported from the older `src/discord` / `src/channels/plugins/outbound` layout to the current `extensions/discord` plugin layout and reviewed for a guarded archive command surface.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best maintainer path is to port it onto current main by storing ACP-created Discord child-thread ownership metadata during ACP thread binding, having `/acp close` request archive-after-reply only after close/unbind succeeds for the current owned Discord thread, implementing the post-reply archive in `extensions/discord`, and adding focused tests for ACP-created threads, manual/reused threads, non-Discord bindings, archive failures, degraded close notices, and outbound send overrides.

## Risks / Open Questions

- The PR patch targets the old `src/discord` and `src/channels/plugins/outbound` layout; current main’s active Discord implementation is under `extensions/discord`, so it is not merge-ready without a port.
- Archiving is a persistent Discord side effect. The port should fail closed unless the removed binding is the current Discord thread and has reliable ACP-created ownership metadata.
- The proposed archive-after-reply payload shape should be internal or otherwise guarded. If arbitrary assistant/plugin outbound payloads can set it, they could archive Discord threads outside the intended `/acp close` flow.
