## Summary

Keep this PR open. Current main still does not archive ACP-created Discord threads on `/acp close`, so the contribution addresses a real missing behavior, and the review comments about runtime notices and send overrides were addressed in the PR branch. It is not merge-ready because it targets the removed pre-plugin Discord layout and hard-codes Discord-specific close behavior in ACP core instead of routing the side effect through the current Discord plugin boundary.

## What This Changes

The PR records ACP-created Discord thread ownership metadata and makes `/acp close` send a Discord archive-after-reply payload that archives only ACP-spawned threads after close/unbind succeeds.

## Best Possible Solution

Port the useful behavior onto current main by adding an extension-safe post-close hook or payload contract, recording ACP-created child-thread ownership in binding metadata, letting `extensions/discord` archive the current owned thread after the success reply, and preserving existing close/unbind behavior for manual threads, non-Discord bindings, and archive failures.
