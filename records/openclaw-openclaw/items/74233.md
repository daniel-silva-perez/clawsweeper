## Summary

Current main still contains the reported behavior: context-engine assembly is applied, but the embedded runner still passes a pre-assembly message snapshot into the hard overflow precheck, and the precheck still prefers the larger unwindowed estimate. Because open PR #74255 uses closing syntax and directly targets this host-side fix, this issue should stay open until that PR or an equivalent implementation lands.

## What This Changes

The issue reports that context-engine assembled prompts can be bypassed because overflow precheck still hard-gates prompt admission on larger pre-assembly history.

## Best Possible Solution

Land or adapt #74255 so successful context-engine assembly is prompt-authoritative by default, raw pre-assembly history becomes explicit opt-in or maintenance/debt telemetry, and the context-engine docs plus the locking precheck tests are updated to match the final contract.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence reproduction shape: a context engine can return a small assembled message window while the runner still supplies a larger pre-assembly snapshot to the precheck helper, whose current test asserts the larger unwindowed estimate wins.

## Solution Assessment

Yes for the requested direction, but no for current main. The maintainable solution is a generic context-engine contract path like #74255, not a lossless-claw-specific bypass or removal of overflow protection entirely.
