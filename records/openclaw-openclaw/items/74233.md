## Summary

Current main still contains the reported behavior: the embedded runner assembles context-engine messages, then gives the precheck a pre-assembly `unwindowedMessages` snapshot, and the precheck intentionally uses the larger estimate. This is source-verifiable and not implemented-on-main. It should stay open for maintainer review because the docs also describe a full-transcript overflow safeguard, so the fix needs an explicit context-engine contract decision rather than an automatic cleanup close.

## Best Possible Solution

Keep this open. Maintainers should decide the context-engine contract explicitly, then implement a generic separation between prompt-admission size and raw-history maintenance/debt. A likely fix would make assembled messages prompt-authoritative when the engine declares that capability, retain a separate raw-history pressure signal, update docs, and replace the current locking test with a regression that allows a small assembled prompt despite a large pre-assembly transcript.
