## Summary

Keep open. Current main still does not expose an ACP skill-context selector or default/auto injection path, and the ACP dispatch path still sends only the raw task text into the external harness. The existing `acp-router` skill is useful parent-side routing guidance, but it does not inject arbitrary workspace skills into spawned ACP sessions.

## What This Changes

Request to let `sessions_spawn({ runtime: "acp" })` include rendered OpenClaw skill context through explicit skill lists, automatic relevance selection, or configured defaults.

## Best Possible Solution

Add an explicit, documented ACP skill-context capability after maintainer policy is chosen: start with a typed `skills` selector, reuse `buildWorkspaceSkillSnapshot` under target-agent allowlists and prompt limits, pass it through a structured ACP prompt-context seam or tightly bounded task-prefix fallback, and cover docs/tests for explicit lists, defaults or auto behavior, truncation, isolation, and host-runtime safety.

## Risks / Open Questions

- The public API shape is still a maintainer/product decision: explicit skill lists are relatively bounded, while automatic relevance and default injection need policy for when local skills cross into external ACP harnesses.
- Naively prefixing full `SKILL.md` content into ACP task text could leak local workflows or imply OpenClaw-only tool, env, provider-key, or sandbox access unless the feature is explicit, allowlisted, bounded, and auditable.
- The implementation needs to preserve ACP's external-harness boundary while reusing existing skill prompt limits and per-agent allowlists.
