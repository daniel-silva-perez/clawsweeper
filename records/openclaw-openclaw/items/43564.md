## Summary

Keep open. Current main still lacks ACP session skill-context injection: `sessions_spawn` has no skill selector/default/auto field, the ACP spawn and runtime turn contracts have no prompt-context seam, and the ACP dispatch path sends the raw task text into the child harness. The prior ClawSweeper comment remains accurate; the bundled `acp-router` skill helps the parent choose ACP routing, but does not inject arbitrary workspace skills into spawned ACP sessions.

## What This Changes

Request to let `sessions_spawn({ runtime: "acp" })` include rendered OpenClaw skill context through explicit skill lists, automatic relevance selection, or configured defaults.

## Best Possible Solution

Add an explicit, documented ACP skill-context capability after maintainer policy is chosen: start with a typed `skills` selector, reuse `buildWorkspaceSkillSnapshot` under target-agent allowlists and prompt limits, pass it through a structured ACP prompt-context seam or tightly bounded task-prefix fallback, and cover docs/tests for explicit lists, defaults/auto behavior, truncation, isolation, and host-runtime safety.

## Risks / Open Questions

- The public API shape is still a maintainer/product decision: explicit lists are relatively bounded, while automatic relevance and default injection need policy around when local skills cross into external ACP harnesses.
- Naively prefixing full `SKILL.md` content into ACP task text could leak local workflows or imply OpenClaw-only tool, env, provider-key, or sandbox access unless the feature is explicit, allowlisted, bounded, and auditable.
- The implementation needs to preserve ACP's external-harness boundary while reusing existing skill prompt limits and per-agent allowlists.
