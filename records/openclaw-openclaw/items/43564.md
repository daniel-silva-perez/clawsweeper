## Summary

Current main still does not implement ACP session skill-context injection. The normal OpenClaw skill snapshot/prompt machinery exists, and the bundled acpx plugin includes an `acp-router` skill for the parent agent, but `sessions_spawn({ runtime: "acp" })` still has no `skills` selector/default/auto parameter and the ACP runtime path forwards only the raw task text plus existing runtime/session controls.

## Best Possible Solution

Keep this open and implement it as an additive ACP/session-spawn feature: add a typed skills selector or documented inherited default, reuse buildWorkspaceSkillSnapshot with the target agent allowlist and prompt limits, add a structured ACP prompt-context seam or carefully bounded task-prefix fallback, and cover explicit lists, defaults/auto behavior, truncation, isolation, and ACP host-security behavior in ACP-specific tests and docs.

## Risks / Open Questions

- A ClawHub-only answer is not sufficient for the requested behavior because current plugin/skill surfaces can ship skills, but they do not add a typed sessions_spawn skills parameter or an ACP runtime turn context field.
- When implemented, naively prefixing SKILL.md content into ACP task text could leak local workflows or sensitive instructions to external harnesses; the feature should be explicit, allowlisted, bounded, and auditable.
- ACP sessions run outside the OpenClaw sandbox, so injected skills must remain instructions-only and must not imply access to skill env vars, OpenClaw tools, provider keys, or host permissions.
