## Summary

Keep open. Current main still does not implement ACP session skill-context injection: the ACP `sessions_spawn` schema has no skill selector/default/auto parameter, ACP rejects the existing native subagent context knobs, and the ACP dispatch path forwards only the raw task text. The bundled `acp-router` skill helps the parent agent choose the ACP runtime path, but it does not inject arbitrary workspace skills into the spawned ACP session.

## What This Changes

Request to let `sessions_spawn({ runtime: "acp" })` include rendered OpenClaw skill content through explicit skill lists, automatic relevance selection, or configured defaults.

## Best Possible Solution

Implement this as an additive core ACP/session-spawn capability: add a typed skills selector or documented inherited default, reuse `buildWorkspaceSkillSnapshot` with target-agent allowlists and prompt limits, pass the result through a structured ACP prompt-context seam or a carefully bounded task-prefix fallback, and document/test explicit lists, defaults/auto behavior, truncation, isolation, and host-runtime safety.

## Risks / Open Questions

- The public API shape is still a product decision: explicit lists are straightforward, while automatic relevance and defaults need policy around when local skills should cross into external ACP harnesses.
- Naively prefixing full `SKILL.md` content into ACP task text could leak local workflows, secrets guidance, or OpenClaw-only tool assumptions unless it is explicit, allowlisted, bounded, and auditable.
- ACP sessions run on the host-side external harness path, so injected skills must remain instructions-only and must not imply access to OpenClaw tools, skill env vars, provider keys, or sandbox permissions.
