## Summary

Keep this PR open. Current main still lacks the proposed explicit session maintenance modes and gateway sweep, and the PR is useful source work, but the head diff still has blocking correctness issues in the test mock, provider-agnostic reset behavior, and expiry timestamp selection.

## What This Changes

The PR adds per-model `sessionCacheTtl` and `timeBasedContextCompact` settings, session activity/cache timestamps, gateway maintenance sweeps for compact/reset policies, Anthropic defaults, and related tests.

## Best Possible Solution

Land a rebased, maintainer-approved session lifecycle feature that separates prompt-cache TTL behavior from provider-agnostic reset policy, uses robust activity timestamps, documents the new config knobs, and coordinates keep-warm behavior with #62475.

## Reproduction Assessment

Yes. The Moonshot test failure is reproducible with the targeted `cache-ttl.test.ts` path by inspection, and the reset/stale-timestamp bugs have deterministic state inputs through `resolveCacheTtlMs`, `shouldResetExpiredSession`, and `shouldRunIdleCacheCompaction`.

## Solution Assessment

No, not as submitted. The direction is plausible, but the implementation should first make reset TTL resolution provider-agnostic, use the newest relevant activity timestamp for expiry, and fix the changed test mock before maintainer review considers the broader API shape.

- **[P1] Align the Moonshot mock with the new assertions:** `src/agents/pi-embedded-runner/cache-ttl.test.ts:31-36`
  - body: The mock still returns `true` for Moonshot, while the changed assertions now expect `false`. Since `isCacheTtlEligibleProvider` returns the plugin eligibility result when it is defined, these assertions fail at runtime.
  - confidence: 0.96
- **[P2] Let reset mode use configured TTLs for every provider:** `src/agents/pi-embedded-runner/cache-ttl.ts:170-172`
  - body: `resolveCacheTtlMs` returns `null` before reading `timeBasedContextCompact` or `sessionCacheTtl` when the provider is not prompt-cache eligible, and the sweep skips null TTLs. This makes configured `reset` mode a no-op for non-cache providers even though reset only needs idle timestamps.
  - confidence: 0.91
- **[P2] Use the newest activity timestamp for expiry:** `src/gateway/session-cache-maintenance.ts:53-57`
  - body: `resolveCacheTouchAt` always prefers `lastCacheTouchAt` over `lastAssistantMessageAt`. If maintenance was disabled while assistant turns continued and is later re-enabled, a stale cache-touch timestamp can make a recently active session reset or compact immediately.
  - confidence: 0.84
