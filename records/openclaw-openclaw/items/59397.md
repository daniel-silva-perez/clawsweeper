## Summary

Keep this PR open, but do not merge it as-is. Current main and the latest release still use raw `Number(...)` parsing for the nodes CLI gateway transport timeout, so the requested fix is not implemented; however, the proposed parser switch would also silently reinterpret inputs such as `1e3` or `123abc` because the shared helper currently uses `Number.parseInt`.

## What This Changes

The PR changes the nodes CLI gateway timeout path from raw `Number(opts.timeout ?? 10_000)` parsing to `parseTimeoutMsWithFallback(..., { invalidType: "error" })` and adds a focused regression test for `--timeout 0`.

## Best Possible Solution

Update the fix against current main by validating the nodes CLI transport timeout in `src/cli/nodes-cli/rpc.runtime.ts`, and either tighten the shared helper to accept only whole positive integer millisecond strings or use an equivalent parser that rejects partial/scientific input instead of reinterpreting it.

## Reproduction Assessment

Yes. The behavior is reproducible by source inspection: current main still forwards `Number(opts.timeout ?? 10_000)`, while the proposed helper path would parse `1e3` as `1` and `123abc` as `123` because the helper uses `Number.parseInt`.

## Solution Assessment

No. The intended direction is good, but the proposed implementation is not the best fix until it validates the whole timeout string and is rebased onto the current `rpc.runtime.ts` path.
