## Summary

Keep this PR open. Current main and v2026.4.27 still use raw `Number(opts.timeout ?? 10_000)` for the active nodes CLI gateway transport timeout, so the requested validation has not landed; however, the branch is now dirty against main's runtime split and retains a blocking parser-semantics concern from review.

## What This Changes

The PR changes the nodes CLI gateway timeout path to use `parseTimeoutMsWithFallback(..., { invalidType: "error" })` and adds a regression test expecting `timeout: "0"` to throw.

## Best Possible Solution

Rebase the PR onto current main, apply the validation in `src/cli/nodes-cli/rpc.runtime.ts`, keep the focused invalid-timeout test, and make timeout string parsing explicit so valid existing numeric forms are preserved or clearly rejected.

## Reproduction Assessment

Yes. Current main statically reproduces the gap by following `callGatewayCli` into `callGatewayCliRuntime`, where `timeout: "0"` is converted with `Number(...)` and forwarded instead of throwing; the PR review concern is likewise reproducible from `parseTimeoutMsWithFallback` using `Number.parseInt`.

## Solution Assessment

No. The validation direction is right, but the submitted branch is stale and the parser change is not the safest form until it either preserves prior numeric semantics or rejects partial/non-decimal input explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve or reject non-decimal timeout input:** `src/cli/nodes-cli/rpc.ts:37`

Preserve the old finite-number behavior or reject these forms clearly before wiring this helper into nodes CLI timeouts.
