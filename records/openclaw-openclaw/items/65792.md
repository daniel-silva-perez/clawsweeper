## Summary

Keep open. Current main still broadcasts the WebChat final `chat` event directly from `emitChatFinal` without checking or running `message_sending`; the hook contract and shared outbound path already support rewrite/cancel semantics, so this is a real focused gap rather than a stale or superseded request.

## What This Changes

Add `message_sending` plugin-hook support to the WebChat/WebSocket final assistant delivery path, including session/run context, rewrite support, cancel handling, and hook-failure isolation.

## Best Possible Solution

Keep this issue open and implement a focused Gateway WebChat fix in `src/gateway/server-chat.ts`: run plugin `message_sending` for final assistant WebSocket delivery, pass stable context such as `channelId: "webchat"`, `sessionKey`, and run correlation, apply rewritten content, isolate hook failures, and define/test cancel behavior. Leave broader all-channel lifecycle-hook unification to #50126/#70928.

## Risks / Open Questions

- `runMessageSending` is async while the current agent-event handler and `emitChatFinal` path are synchronous; the implementation needs to preserve final-event ordering and run-context cleanup semantics.
- WebChat streams deltas before the final event, so final-only rewrite/cancel may still expose transient unmodified text unless protocol/UI behavior is explicitly tested and accepted.
- A `cancel: true` path must terminate the active WebChat run cleanly without leaving the UI to persist the existing `chatStream` fallback as a delivered assistant message.
