## Summary

Keep open. Current main still bypasses the plugin `message_sending` hook on WebChat final delivery: the Gateway WebSocket final event is built from the buffered assistant text and broadcast directly, while the shared outbound delivery path already runs the hook with rewrite/cancel semantics. Related broader reports remain open and prior closed PRs did not land a WebChat fix.

## What This Changes

Request support for running the existing `message_sending` plugin hook before WebChat/WebSocket final assistant delivery, including content rewrite, cancel handling, and session correlation.

## Best Possible Solution

Add a focused WebChat final-delivery hook invocation using the existing plugin hook runner and message hook contract: skip work when no hooks are registered, pass `channelId: "webchat"`, `sessionKey`, and run/session correlation where available, apply rewritten content, suppress the assistant message on cancel while still ending the UI run, and isolate hook failures like `deliverOutboundPayloads`. Cover rewrite, cancel, no-hook, and hook-failure cases in Gateway chat tests; leave broader lifecycle-hook unification to the related umbrella issues.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: follow a Gateway agent lifecycle `end` event into `emitChatFinal`, where the final WebSocket chat payload is broadcast directly, and contrast that with `deliverOutboundPayloads`, which runs `message_sending`.

## Solution Assessment

Yes for the narrow issue. Reusing `message_sending` on WebChat final delivery is the smallest maintainable fix because the hook type, cancellation semantics, and `sessionKey` context already exist; creating a new WebChat-specific hook would add unnecessary API surface.
