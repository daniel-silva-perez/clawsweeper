## Summary

Current main still lacks the requested `message_sending` hook on the WebChat final WebSocket delivery path. `emitChatFinal` builds and broadcasts the final `chat` payload directly, while the shared outbound path and Telegram reply path already call `runMessageSending` and honor rewrite/cancel behavior. The issue remains valid and focused.

## Best Possible Solution

Keep this issue open and implement a focused Gateway WebChat fix. Run plugin `message_sending` for final assistant WebSocket delivery, pass `channelId: "webchat"`, `sessionKey`, and run correlation, apply rewritten content, isolate hook failures, and define/test cancel behavior so WebChat clients terminate cleanly without preserving canceled text. Leave the broader multi-channel hook unification to #50126/#70928.

## Risks / Open Questions

- `createAgentEventHandler` currently returns a synchronous listener while `runMessageSending` is async; the fix needs to preserve event ordering and avoid clearing run context before hook context is available.
- `cancel: true` needs careful WebChat semantics: current UI code appends buffered stream text when a final event has no renderable assistant message, so a cancel path must not leave clients stuck or accidentally deliver already streamed content.
- A final-only hook can rewrite the stored/final message, but users may briefly see unmodified deltas during streaming; tests should make that behavior explicit.
