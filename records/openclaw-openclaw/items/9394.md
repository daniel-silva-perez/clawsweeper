## Summary

Keep this issue open. Current main still does not provide rich iMessage style spans, text-size attributes, underline/strikethrough, or the requested Shake/Nod/Explode/Ripple/Bloom/Jitter text-effect model; legacy iMessage remains plain text, and BlueBubbles only exposes a narrower effectId action while stripping Markdown on normal sends.

## What This Changes

Request support for rich iMessage outbound formatting such as bold, italic, underline, strikethrough, text size, and Messages effects including Shake, Nod, Explode, Ripple, Bloom, and Jitter.

## Best Possible Solution

Define a typed outbound capability model for iMessage and BlueBubbles formatting/effects, implement only the upstream-backed subset per transport, preserve plain-text fallback where rich formatting is unsupported, and document legacy imsg limitations in the public channel docs.

## Reproduction Assessment

Yes for the observable current behavior: the current-main send paths can be traced to plain text/file/reply payloads for legacy imsg and stripped text plus optional effectId for BlueBubbles. A live transport smoke would still be needed before choosing exact upstream effect IDs or rich attribute serialization.

## Solution Assessment

Unclear; the request direction is valid, but a narrow code patch is not yet the best fix without maintainer API/product choices. The safer path is a capability-gated transport model with per-provider fallback rather than ad hoc flags on one send function.
