## Summary

Keep this issue open. Current main still has no outbound rich-text/span model for iMessage, no underline/strikethrough/text-size payload surface, and no support for the requested Shake/Nod/Explode/Ripple/Bloom/Jitter effects; the existing maintainer-side comments keeping the issue open remain accurate against the current tree.

## What This Changes

The issue requests iMessage outbound rich formatting and richer message effects instead of plain-text-only legacy imsg sends and narrow BlueBubbles effectId sends.

## Best Possible Solution

Define a typed outbound capability model for iMessage transports, implement only the upstream-backed rich-format/effect subset per transport, preserve plain-text fallback where unsupported, and document legacy imsg limitations alongside BlueBubbles capabilities.

## Reproduction Assessment

Yes for the observable current behavior: static inspection and focused tests show legacy imsg sends plain text/file/reply params, while BlueBubbles strips Markdown and only supports an optional effectId string. A live transport smoke is still needed before choosing exact upstream rich-attribute and effect serialization.

## Solution Assessment

Unclear; the requested direction is valid, but ad hoc imsg flags or a one-off send function patch would not be the best maintainable fix. The safer path is a capability-gated transport model with provider-specific fallbacks and docs.
