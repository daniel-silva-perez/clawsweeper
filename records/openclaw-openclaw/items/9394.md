## Summary

Keep this issue open. Current main still sends legacy iMessage through a plain imsg JSON-RPC text payload, while BlueBubbles only exposes a narrower effectId path and strips Markdown on normal sends instead of preserving rich style spans. The existing maintainer-side discussion remains accurate because the relevant iMessage and BlueBubbles send paths have not changed since that review snapshot.

## What This Changes

The issue requests outbound iMessage support for rich text formatting, text-size variations, underline/strikethrough, and named animation effects beyond plain text delivery.

## Best Possible Solution

Define a typed outbound capability model for iMessage/BlueBubbles formatting and effects, implement only the upstream-backed subset per transport, preserve plain-text fallback behavior where rich formatting is unsupported, and document legacy imsg limitations on the public channel docs.
