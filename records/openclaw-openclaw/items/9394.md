## Summary

Keep this issue open. Current main still does not provide an outbound rich-text/span model for iMessage, legacy imsg still sends a plain JSON-RPC text payload, and BlueBubbles normal sends strip Markdown while only exposing a narrower `effectId` action for classic effects. The maintainer-side discussion already kept this open for product/API direction, and the current tree has not closed the requested rich formatting or Shake/Nod/Explode/Ripple/Bloom/Jitter gap.

## What This Changes

Request to let the iMessage channel send rich text attributes, text-size variants, and iMessage text/effect animations instead of plain-text-only outbound messages.

## Best Possible Solution

Define a typed outbound capability model for iMessage transports, implement only upstream-backed formatting and effect subsets per transport, keep deterministic plain-text fallback where unsupported, and document the legacy imsg limitations alongside BlueBubbles capabilities.

## Reproduction Assessment

Yes for the observable current gap: source and focused tests show imsg sends plain text/file/reply params, while BlueBubbles strips Markdown and exposes only an optional `effectId` string. A live BlueBubbles/imsg smoke is still needed before choosing exact rich-attribute serialization.

## Solution Assessment

Unclear as an implementation question because there is no proposed patch and ad hoc flags would be brittle. The maintainable path is a capability-gated transport model with provider-specific fallbacks and docs, not a one-off send function change.
