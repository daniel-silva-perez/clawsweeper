## Summary

Keep this issue open. Current main still sends legacy iMessage through a plain imsg text payload and BlueBubbles normal sends still strip Markdown rather than translating style spans; BlueBubbles only exposes a narrower classic `effectId` path, not the requested rich text, text-size, underline/strikethrough, or Shake/Nod/Explode/Ripple/Bloom/Jitter text-effect model. The prior maintainer-side discussion remains accurate and this needs product/API direction before a safe implementation can be scoped.

## What This Changes

The issue requests rich iMessage outbound formatting and effects, including bold, italic, underline, strikethrough, text size variants, and Shake/Nod/Explode/Ripple/Bloom/Jitter effects.

## Best Possible Solution

Define a typed outbound capability model for iMessage and BlueBubbles formatting/effects, implement only the upstream-backed subset per transport, preserve plain-text fallback where rich formatting is unsupported, and document legacy imsg limitations in the public channel docs.
