## Summary

Keep this PR open. Current main and the latest release still lack a WhatsApp `readOnly` config and transport gate, so the feature is useful, but the proposed patch is not merge-ready because it leaves inconsistent blocked-send semantics, breaks pairing request creation in read-only pairing DMs, and omits required user-facing docs/changelog coverage.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` configuration, resolves it through account state, threads it into the WhatsApp Web listener, and gates several outbound WhatsApp send paths with focused tests and generated config metadata updates.

## Best Possible Solution

Land a repaired WhatsApp-owned implementation that documents `readOnly` at channel and account scope, resolves it per account, enforces one consistent hard outbound gate across message/media/poll/reaction/typing/presence/read-receipt/send-API paths, preserves inbound processing and pairing request creation without sending WhatsApp replies, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. The PR diff itself shows the remaining cases: empty-body/no-media sends return before the read-only guard, composing indicators silently no-op while other blocked helpers throw, and read-only pairing DMs skip the helper that upserts pending pairing requests. I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right direction and is narrower than relying on generic `sendPolicy`, but the current patch needs the guard ordering, composing contract, pairing persistence, and docs/changelog fixes before it is the best maintainable solution.
