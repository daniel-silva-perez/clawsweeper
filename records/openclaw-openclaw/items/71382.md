## Summary

Keep open. Current main still lacks WhatsApp `readOnly` config or a transport-level outbound gate, and this PR is useful but not merge-ready because the diff still has narrow read-only contract gaps around empty sends, composing indicators, pairing request persistence, and missing user-facing docs/changelog. The `sendPolicy` discussion is relevant, but the existing session policy does not cover all WhatsApp transport sends such as read receipts, presence, pairing replies, and explicit send APIs.

## What This Changes

This PR adds WhatsApp channel/account `readOnly` config, resolves it through account state, threads it into the WhatsApp Web listener, and gates multiple outbound WhatsApp send paths with tests and generated config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned implementation that documents `readOnly` at channel and account scope, resolves it per account, enforces a consistent hard outbound gate across message/media/poll/reaction/typing/presence/read-receipt/send-API paths, preserves inbound processing and pending pairing request creation without sending WhatsApp replies, and records the feature in the changelog.

## Reproduction Assessment

Yes. The PR head itself gives a high-confidence static reproduction: empty no-media sends return before the writable guard, composing indicators silently return while other blocked send paths throw, and read-only pairing DMs skip the helper that creates pending pairing requests. I did not run tests because this sweep is read-only.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right direction and narrower than relying on generic `sendPolicy`, but this patch needs the guard ordering, composing contract, pairing persistence, docs, and changelog fixes before it is the best maintainable solution.
