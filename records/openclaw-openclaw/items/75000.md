## Summary

Keep open. Current main still contains the Feishu probe/status path that matches the reported timeout/reset behavior, and the reporter’s follow-up adds concrete environment evidence that this is a plausible 2026.4.x regression rather than a stale or external-only network report.

## What This Changes

The issue reports Feishu plugin requests timing out or resetting on Linux in OpenClaw 2026.4.26 and 2026.4.27 while an older 2026.2.2 install works.

## Best Possible Solution

Keep this in Feishu plugin triage and isolate the HTTP probe path with a credentialed repro that compares OpenClaw’s SDK-backed client, native fetch/guarded fetch, and standalone axios from the same runtime. The eventual fix should stay in the Feishu plugin, make timeout and transport policy explicit, preserve bot-identity recovery, and add focused probe/client/status tests plus docs if a new operator workaround is supported.

## Reproduction Assessment

Unclear. The reporter now has a concrete deterministic reproduction on their Linux host, but maintainers still need redacted Feishu credentials or a live test app plus the exact startup/status command to reproduce the failure independently against current main.

## Solution Assessment
