## Summary

Keep open. The report is fresh, the added reporter evidence now ties the failure to OpenClaw 2026.4.26 and 2026.4.27, and current main still contains the same Feishu SDK-backed token/probe path with no targeted change since the latest release.

## What This Changes

The issue reports Feishu/Lark built-in bot token and `openclaw_bot/ping` probe requests timing out or resetting in OpenClaw 2026.4.26/2026.4.27 while direct requests from the same host succeed.

## Best Possible Solution

Keep the fix in the Feishu plugin. The desired end state is one consistent Feishu token/probe transport policy across startup, setup/status, and channel status that honors configured/caller timeouts, preserves proxy and SSRF safety, and has unit coverage plus a redacted live verification path for the SDK-vs-direct-request failure mode.

## Solution Assessment

Unclear. There is no proposed patch, and the narrowest maintainable fix depends on confirming whether the regression is in OpenClaw's SDK wrapper, timeout/status propagation, packaged runtime dependencies, or live Feishu transport behavior.
