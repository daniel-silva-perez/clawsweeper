## Summary

Keep open. The report is fresh, the reporter added credible 2026.4.27 field evidence, and current main still contains the same Feishu SDK-backed bot probe/token surface with no targeted fix after the latest release.

## What This Changes

The issue reports Feishu bot/token probe timeouts and socket resets in OpenClaw 2026.4.26 and 2026.4.27 while direct curl and standalone axios requests from the same host succeed.

## Best Possible Solution

Keep this in Feishu plugin triage and fix the owner module. The desired end state is one consistent Feishu probe/token transport policy across startup, setup/status, and channel status that honors caller/configured timeout, cancels or avoids orphaned SDK requests, preserves proxy/security behavior, and has regression coverage for timeout propagation plus hanging/resetting SDK request failures.

## Reproduction Assessment

Unclear. The reporter provided a strong field reproduction path, but a high-confidence local reproduction still needs the exact OpenClaw command with redacted Feishu credentials or a fixture that simulates the Lark SDK token/probe hang and `ECONNRESET` behavior.

## Solution Assessment

Unclear because there is no proposed patch. Keeping the fix inside `extensions/feishu` is the right boundary, but maintainers still need to choose whether the narrow fix is timeout propagation, SDK request cancellation, direct HTTP fallback for probe/token calls, or a broader Feishu transport wrapper change.
