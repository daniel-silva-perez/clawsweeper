## Summary

Keep open. The report is fresh, has matching Feishu probe/token evidence, and current main still contains the same SDK-backed bot-info probe path that can time out or reset without a demonstrated fix in the latest release or current main.

## What This Changes

The issue reports a regression in OpenClaw 2026.4.26/2026.4.27 where the built-in Feishu plugin times out or resets while obtaining Feishu token/bot identity during startup or status probing.

## Best Possible Solution

Keep this in Feishu plugin triage and fix the owner module, not core. The desired end state is one consistent Feishu probe/token transport policy across startup, setup/status, and channel status that honors caller/configured timeout, preserves SSRF/proxy safety, and has tests for timeout propagation plus the SDK-vs-direct-request failure mode if maintainers can reduce it without live credentials.
