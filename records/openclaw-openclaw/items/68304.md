## Summary

Keep open. Current main covers the normal Telegram delivery paths with shared silent-reply normalization, but it does not implement the adapter-level `sendMessageTelegram` guard requested here; the PR is still useful defense-in-depth, but the current patch can suppress inline-keyboard sends and should be amended before merge.

## What This Changes

This PR imports `isSilentReplyText` and adds an early `sendMessageTelegram` no-op return for exact `NO_REPLY` text when no `mediaUrl` is present.

## Best Possible Solution

Keep the shared outbound normalization as the primary fix for normal delivery, and if maintainers want the adapter defense-in-depth, land a Telegram-owned guard that suppresses exact silent text only when there is no media and no inline keyboard or other sendable payload, with focused `sendMessageTelegram` regression coverage.

## Reproduction Assessment

Yes. A direct mocked `sendMessageTelegram("123", "NO_REPLY", { cfg, token, api })` call on current main has no adapter-level guard and reaches the `api.sendMessage` text path; ordinary delivery paths are separately normalized before they call the adapter.

## Solution Assessment

No. The guard is a reasonable defense-in-depth direction, but this implementation is not the best version because it ignores `opts.buttons` and can drop inline-keyboard sends; the safer patch should preserve button-bearing payloads and add targeted tests.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve inline keyboard sends:** `extensions/telegram/src/send.ts:620`

Please also exclude button-bearing payloads from this suppression path and cover it with a focused test.
