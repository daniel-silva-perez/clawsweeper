## Summary

Keep this issue open. Current main still accepts and exposes `ui.seamColor` on the backend, but the Control UI bootstrap contract and frontend theme application path do not carry or apply that value, so the reported UI behavior remains plausible and unaddressed.

## What This Changes

The issue requests that `ui.seamColor` from `openclaw.json` drive the Control UI accent styling instead of leaving buttons, highlights, selections, and focus rings on the built-in red theme tokens.

## Best Possible Solution

Add a narrow Control UI runtime-accent path that transports the validated `ui.seamColor` to the browser, applies and removes root accent CSS variables only when configured, preserves the current default and imported-theme behavior when absent, and covers the gateway endpoint plus jsdom theme application path with regression tests.
