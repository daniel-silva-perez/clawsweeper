## Summary

Keep open. Current main has a concrete `model.usage` diagnostic payload and tests for parts of its behavior, but the public logging/OpenTelemetry docs still do not provide the exact JSON field table, a sample event, or a clear supported local parsing surface for external read-only tools.

## What This Changes

Request to document the exact JSON key names and supported local parsing surface for OpenClaw `model.usage` usage events/logs.

## Best Possible Solution

Add a focused docs/contract update that publishes the current `model.usage` JSON payload fields and an example, maps them to exported OTEL/Prometheus names, distinguishes camelCase OpenClaw fields from provider-native snake_case counters, explains `usage` versus `lastCallUsage` versus `context`, and explicitly states which local surfaces external parsers can safely consume today.

## Reproduction Assessment

Not applicable as a docs/API-contract request rather than a runtime bug. The gap is high-confidence from source and docs inspection: the event shape exists in code and tests, but the public docs do not expose the full schema or supported parser surface.

## Solution Assessment

Yes, a docs/contract update is the narrowest maintainable solution for the stated request. If maintainers want `/tmp/openclaw/openclaw-YYYY-MM-DD.log` to become a stable usage-event source, that should be a separate runtime/API decision rather than implied by the docs.
