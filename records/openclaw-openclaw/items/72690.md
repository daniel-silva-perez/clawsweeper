## Summary

Keep open. Current main has not implemented this PR, and the PR still needs maintainer review because it introduces a scoped CLI loading regression while only partially addressing the linked manifest conversation-access request.

## What This Changes

This PR adds manifest parsing and registry preservation for `permissions.conversationAccess`, documents it as declarative metadata still gated by host config, adds tests, and also appends the configured memory slot plugin during primary CLI command scoped loading.

## Best Possible Solution

Keep the manifest contract additive and explicitly documented, but separate or narrowly scope the memory-slot CLI change. Maintainers should choose the conversation-access trust model first; if request-only metadata is intended, the linked issue should not be closed as fully fixed, and if manifest-granted access is intended, the loader must merge it with an explicit host deny/override policy and tests.

## Reproduction Assessment

Yes. A code-level reproduction for the PR regression is a primary plugin command such as `googlemeet` whose activation plan returns a non-memory plugin id under default config; the PR helper would append the default `memory-core` slot because it only checks that `pluginIds` is non-empty.

## Solution Assessment

No for the memory-slot change: appending the configured memory slot should be limited to the memory/wiki command path or a plan that actually needs the memory slot. Unclear for the linked conversation-access feature because request-only metadata is safer, but it does not satisfy the issue's stated manifest-only grant semantics.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Limit memory-slot injection to memory commands:** `src/plugins/cli-registry-loader.ts:97`

Gate this append on the wiki/memory command or on a memory-specific activation result.
