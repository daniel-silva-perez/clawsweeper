## Summary

Keep open. The PR is still useful as an implementation candidate for manifest-level conversation-access request metadata, but it has a concrete scoped CLI loading regression, currently conflicts with main, and still needs maintainer judgment on whether request-only metadata is the intended answer to the linked manifest-grant RFC.

## What This Changes

This PR adds `permissions.conversationAccess` parsing and registry preservation for plugin manifests, documents it as declarative metadata gated by host config, adds manifest/loader/CLI tests, and appends the configured memory slot plugin during primary CLI command loading.

## Best Possible Solution

Keep the manifest contract additive and explicit, but split or narrow the memory-slot CLI behavior so only wiki/memory command paths load the configured memory slot. Maintainers should also decide whether `permissions.conversationAccess` is request/disclosure metadata only; if so, the linked issue should be updated or narrowed before merge, and if not, the loader needs a reviewed host-approval/deny precedence design.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction for the PR regression is a primary plugin command such as `googlemeet`: its manifest activation resolves a non-memory plugin id, then the PR helper appends the default `memory-core` slot because it checks only that `pluginIds` is non-empty.

## Solution Assessment

No for the memory-slot portion: appending the configured memory slot should be gated to the wiki/memory command path or to activation output that explicitly needs the memory slot. Unclear for the conversation-access portion until maintainers confirm request-only metadata versus manifest-granted access.
