## Summary

Keep open. Current main does not implement the PR's manifest permission support, and the PR still contains a concrete scoped CLI loading regression plus an unresolved product/security choice: the linked request asks for manifest-driven access, while this PR intentionally preserves host-config approval.

## What This Changes

This PR adds `permissions.conversationAccess` parsing and registry preservation for plugin manifests, documents it as request-only metadata gated by host config, adds related tests, and appends the configured memory slot plugin during primary CLI command scoped loading.

## Best Possible Solution

Keep the manifest contract additive and explicit, but separate or narrowly scope the memory-slot CLI change. Maintainers should decide whether `permissions.conversationAccess` is disclosure-only metadata requiring host approval, or whether manifest-granted access is acceptable with a clear deny/override policy and tests.

## Reproduction Assessment

Yes. A code-level reproduction for the PR regression is any primary plugin command whose activation plan returns a non-memory plugin id under default config; the PR helper appends default `memory-core` because it only checks that the planned id list is non-empty.

## Solution Assessment

No for the memory-slot change: appending the configured memory slot should be limited to the wiki/memory command path or a memory-specific activation result. Unclear for the conversation-access feature because request-only metadata is safer, but it intentionally does not implement the linked issue's manifest-only grant semantics.
