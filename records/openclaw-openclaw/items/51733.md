## Summary

Keep open. Current main supersedes the watcher-glob part by watching skill root directories, and merged work in #72534/#74942 covers one-level grouped layouts, but current main still does not load the repo-bundle shape this PR targets under configured skill roots. The PR is still useful source work, but it is stale against the current loader/watcher shape and has an unresolved hybrid-bundle precedence bug.

## What This Changes

The PR changes the skills watcher, skills loader, and regression tests to watch and load repo-style nested skill bundles such as `<skillsRoot>/<repo>/skills/<skill>/SKILL.md` under managed and workspace skill roots.

## Best Possible Solution

Land a small current-main loader update that recognizes repo-style bundle roots under workspace, project-agent, personal-agent, managed, and extra skill roots, while keeping the current root-directory watcher and the existing containment, bounded-scan, size-limit, and deterministic ordering safeguards. Add focused tests for repo-style roots and document the supported bundle shape if it becomes product surface.

## Reproduction Assessment

Yes. A source-level reproduction is `<workspace>/skills/kepano-obsidian-skills/skills/defuddle/SKILL.md`: current main scans `kepano-obsidian-skills`, then checks `kepano-obsidian-skills/skills/SKILL.md`, and never reaches `skills/defuddle/SKILL.md`.

## Solution Assessment

No, not as-is. The loader-side direction is useful, but the watcher glob changes are obsolete on current main and the branch still needs a rebase that preserves current bounded-scan safeguards and fixes hybrid bundle precedence.

Full review comments:

- **[P2] Preserve nested-root precedence for hybrid bundles:** `src/agents/skills/workspace.ts:441-445`
  - body: If a repo under a configured skills root has both a top-level `SKILL.md` and nested `skills/*/SKILL.md`, this branch loads the top-level skill through `loadSkillDir()` and never resolves the nested skills root. The same bundle is therefore exposed differently when installed as an extra dir versus under managed/workspace roots, so the loader should resolve nested repo roots before direct child loading or define the precedence explicitly.
