## Summary

Keep this PR open. Current main still lacks the Android chat parity behavior, so the branch is not obsolete, but the PR is not merge-ready as-is because the new Android model picker can race with Send and start the next run on the previous session model. The earlier review issues around default `operator.admin`, stale thinking state, and Kotlin raw-string regexes appear addressed in the latest PR head, and CI checks on the head SHA are green, but the branch is broad and stale relative to current main, so it needs a focused repair plus maintainer review before merge.

## What This Changes

This PR expands Android chat with richer session and agent controls, create/delete chat flows, timeline/status rendering, cleaner Telegram/thread labels, gateway session-title sanitization, explicit session mutation support, and related Android/gateway tests.

## Best Possible Solution

Fix the model picker/send race, rebase the branch onto current main, then have an Android/gateway maintainer decide whether to land this as one parity slice or split it into smaller UI, session mutation, and gateway-title cleanup pieces.

## Reproduction Assessment

Feature parity itself is not a bug report, but current-main comparison gives high-confidence proof that the requested Android chat behavior is still missing. For the patch finding, yes: select a model in the new Android composer and immediately send before `sessions.patch` resolves; the send path does not carry the selected model.

## Solution Assessment

No, not as-is. The PR targets the right missing surfaces and earlier review fixes are present, but the model-selection race should be repaired before maintainer review decides whether the full XL slice is the right landing shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Wait for model selection before sending:** `apps/android/app/src/main/java/ai/openclaw/app/ui/chat/ChatComposer.kt:306`
  - body: The model menu only fires `onSetModel(option.value)` and closes, while Send remains enabled and `onSend` calls `sendChat` with only the thinking value. Since `chat.send` reads the session model server-side and has no per-send model parameter, a user who picks a model and immediately sends can start the run on the previous model if the async `sessions.patch` has not completed yet. Track a pending model update and block/await it before sending, or add an explicit server-supported per-send model path.
  - confidence: 0.78
