## Summary

Keep open. Current main still only writes schema v1 backup manifests and performs structural archive verification, so the checksum feature is not obsolete; however, the proposed patch has blocking correctness issues around the existing plugin dependency filter, schema v2 checksum enforcement, link entries, and empty directory assets.

## What This Changes

The PR changes backup create, backup verify, docs, and backup verify tests to emit schema v2 backup manifests with per-asset SHA-256 checksums and verify archived content hashes while keeping v1 archive compatibility.

## Best Possible Solution

Land a polished archived-byte checksum implementation that preserves existing backup archive contracts, treats schema v2 checksums as mandatory for every asset, and either hashes or explicitly rejects empty-directory and link-like archive entries with targeted tests and docs.

## Reproduction Assessment

Yes. The source diff gives high-confidence reproduction paths: existing backup-create tests import the removed filter helper and assert extensions node_modules are omitted, while schema v2 manifests with no sha256 fields and empty directory assets can be reasoned directly through the new verifier branches.

## Solution Assessment

No. Per-asset archived-byte checksums are a good direction, but this patch is not the narrowest maintainable solution until it preserves current filters and defines consistent checksum semantics for v2, empty directories, and link entries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep the plugin dependency filter in backup create:** `src/infra/backup-create.ts:413-423`
  - body: This new packing path no longer applies the existing extensions/node_modules filter, and the helper export used by backup-create tests disappears. That both regresses the documented archive contents and leaves the existing real-archive test expecting those dependency trees to be omitted failing until the filter is restored in the new flow.
  - confidence: 0.95
- **[P1] Reject v2 manifests before skipping checksum verification:** `src/commands/backup-verify.ts:468-470`
  - body: `hasChecksums` is false when a schema v2 manifest has all `sha256` fields stripped, so the verifier skips `verifyAssetChecksums` and returns `ok` with no content validation. A corrupted v2 archive can therefore pass by removing every checksum; v2 should fail if any asset is missing its required digest.
  - confidence: 0.93
- **[P2] Hash link entries the same way on create and verify:** `src/infra/backup-create.ts:287-290`
  - body: Create-side hashing walks the extracted asset and only includes `Dirent.isFile()` entries, but verify-side hashing includes every non-directory tar entry. Archives containing symlinks or hardlinks can therefore compute a manifest hash that omits the link and then fail verification when the verifier includes it.
  - confidence: 0.88
- **[P2] Accept empty directory asset checksums:** `src/commands/backup-verify.ts:340-343`
  - body: A checksummed directory asset with no non-directory entries gets a valid empty Merkle hash during create, but verification throws before comparing it because `assetEntries.length` is zero. An empty state or workspace directory can make a freshly created archive fail its own checksum verification.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff does not add CI, dependency, script, network, or secret-handling changes, but the checksum verifier has a concrete integrity-bypass bug for schema v2 archives with stripped checksum fields.

Concerns:

- **[medium] Stripped v2 checksum manifests pass verification:** `src/commands/backup-verify.ts:468`
  - body: The verifier only enforces per-asset checksum presence after `hasChecksums` is true. If an attacker or corrupt process removes all `sha256` fields from a schema v2 manifest, verification skips content hashing and returns success, undermining the stated tamper-detection guarantee.
  - confidence: 0.93
