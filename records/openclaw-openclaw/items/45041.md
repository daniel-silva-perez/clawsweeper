## Summary

Keep open. Current main still lacks the requested configurable outbound filter surface, and the existing cleanup paths only cover structured reasoning payloads, reasoning tags, known scaffolding, special tokens, and tool-call wrappers. The related open PR #60551 overlaps the same plain-text reasoning-preamble leak class, so this should remain with maintainer product/API review rather than being closed or queued as a competing repair.

## What This Changes

The issue requests an optional per-agent or global output filter that strips configured regex/pattern matches from assistant messages before channel delivery.

## Best Possible Solution

Keep this as the tracker until maintainers either accept a version of #60551, add a documented opt-in config or plugin/API hook for outbound text filtering, or explicitly document that supported cleanup will remain heuristic/shared rather than user-defined regex filtering.

## Reproduction Assessment

Yes for the requested surface gap: current main has no outputFilters/outputFilter key in types, strict schema, docs, or delivery sanitization, and existing sanitizer tests preserve normal prose lead-ins. A live multi-agent Discord run is not necessary to verify that the requested configurable surface is absent.

## Solution Assessment

Unclear. A configurable regex filter is plausible, but it creates a new user-facing config contract and false-positive risk; #60551’s narrower heuristic sanitizer is also plausible, so maintainers should choose the boundary before implementation.
