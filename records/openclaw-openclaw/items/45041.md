## Summary

Keep open. Current main still does not provide the requested configurable output-filter surface; it only suppresses structured `isReasoning` payloads and strips known tagged/internal scaffolding. The open related PR #60551 is the nearest implementation candidate, so this should remain a maintainer product-boundary decision rather than being closed or queued as a competing repair.

## What This Changes

The issue requests an opt-in per-agent output filter that strips regex-matched reasoning or meta-planning text from assistant messages before channel delivery.

## Best Possible Solution

Keep this as the product/design tracker until maintainers choose the boundary. A complete solution should either land an accepted version of #60551, add a first-class opt-in config/API hook for output filtering, or document a plugin path, with regression coverage for English and CJK leak examples plus false-positive cases.
