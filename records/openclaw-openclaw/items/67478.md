## Summary

Keep open. Current main still lacks a preventative CLI-wide root-execution guard, and the root-state plus user-systemd installation path described in the report remains reachable; PR #67509 is still open as the direct implementation candidate, so this issue should not be closed before that or an equivalent fix lands.

## What This Changes

The issue requests preventing direct root execution of the OpenClaw CLI on DigitalOcean 1-Click droplets, or warning/wrapping it, to avoid root-owned state, conflicting user-systemd gateway units, and backup/config permission failures.

## Best Possible Solution

Review and land PR #67509, or replace it with an equivalent early CLI guard before config/state resolution, with an explicit override, help/version coverage, targeted tests, and DigitalOcean recovery or wrapper guidance if maintainers want image-specific mitigation.

## Reproduction Assessment

Yes. The report gives deterministic droplet steps, and the current code-level path still reproduces the prerequisites: root HOME resolves root state, doctor repair uses process.env, and systemd writes a user unit under the effective home. I did not run the destructive root/DigitalOcean scenario during this read-only review.

## Solution Assessment

Yes at the issue level. An early CLI guard with an explicit allow-root override is the narrowest repo-side fix because docs or image wrapper guidance alone would not prevent direct invocation of the installed CLI binary.
