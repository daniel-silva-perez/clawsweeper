## Summary

Keep this PR open for normal maintainer review. Current main still lacks the requested config and peer-default behavior, and the PR remains a plausible implementation path, but the proposed patch is incomplete because it only wires the TUI path while documenting WebChat behavior and leaves generated config metadata stale.

## What This Changes

The PR adds a `gateway.webchat.defaultPeerId` config, schema/docs entries, gateway snapshot `dmScope`, and TUI session-key logic/tests to route empty/default session selection through peer DM session construction.

## Best Possible Solution

Keep the PR open, rebase it onto current main, and either implement the configured peer default for both TUI and browser WebChat/Control UI or narrow the feature/docs to TUI only. The final change should update generated config schema/help/labels, preserve explicit session and global-scope behavior, and include focused tests for TUI plus the browser default-session snapshot/application path.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: configure `gateway.webchat.defaultPeerId`, open browser WebChat/Control UI with no selected session, and the current/PR browser path still derives the default from `snapshot.sessionDefaults.mainSessionKey` rather than the peer-session helper.

## Solution Assessment

No, not as-is. The direction is maintainable because it reuses `buildAgentPeerSessionKey`, but the patch is incomplete until the browser WebChat default path and generated config metadata are wired consistently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

The current generated WebChat schema still exposes only `chatHistoryMaxChars`, so users and config tooling would not see the new option correctly.
  - confidence: 0.88
