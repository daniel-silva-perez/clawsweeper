## Summary

Keep open. Current main still enforces `tool_result_persist` and `before_message_write` as sync-only hooks, and async handlers are explicitly ignored with a warning, so the reported governance/DLP use case remains unsupported. Adjacent work exists, but the open related PR is a sync text-only pre-model hook and does not cover async policy calls for this issue.

## What This Changes

Request to let `tool_result_persist`, and possibly `before_message_write`, return promises so plugins can perform async policy scanning or redaction before transcript persistence.

## Best Possible Solution

Define an additive plugin contract for async tool-result policy checks with explicit ordering, timeout/error behavior, and fail-open/fail-closed semantics, then update the hook types, runner wiring, docs, API baseline, and focused persistence/tool-result tests. Treat `before_message_write` separately unless the same async persistence abstraction cleanly covers both hooks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Registering an async `tool_result_persist` or `before_message_write` handler against current main follows the existing sync-only tests: the Promise result is ignored and the original persistence behavior continues.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Awaiting the existing sync persistence hooks directly may be too invasive for the hot transcript append path; a separate async pre-persist or pre-model hook with explicit latency and failure semantics may be the safer maintainer direction.
