## Summary

Keep this issue open. Current main still ships the bundled Notion skill with `NOTION_KEY=$(cat ...)` and curl examples that rely on shell expansion, while the exec-approval docs and parser keep shell expansion/control syntax outside a simple curl allowlist match. The two linked fix attempts were closed unmerged, so there is no shipped or canonical merged fix to close against.

## What This Changes

Update the bundled Notion skill docs so Notion API examples avoid shell-wrapped curl and variable-expansion patterns that miss strict exec allowlists.

## Best Possible Solution

Keep the core allowlist behavior conservative and fix the bundled Notion skill surface: replace or supplement the current examples with an allowlist-friendly helper-file workflow or direct invocation guidance, plus a clear note that shell variable setup and `$NOTION_KEY` expansion can miss under `security=allowlist` with `ask=off`.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the current Notion `API Basics` or search snippet under macOS `security=allowlist`, `ask=off`, with only curl allowlisted; current docs and parser behavior show the shell expansion parts are not covered by the curl executable allowlist.

## Solution Assessment

Yes. A narrow bundled-skill docs/helper update is the best fix; weakening core exec allowlist parsing to accept shell expansion would be broader and less maintainable for this user-facing docs problem.
