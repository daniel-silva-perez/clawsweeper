## Summary

Keep this open. Current main still has the reported gap: strict-agentic recovery is scoped away from the reported Kimi provider path, the planning-only retry guard skips fenced text, and the embedded block-reply delivery path still emits assistant text without requiring a real tool lifecycle. The linked fix PR was closed unmerged, so it is useful direction but not current behavior.

## What This Changes

Prevent Telegram direct replies from delivering first-person exec-looking fenced shell blocks unless the same turn has real tool activity or a truthful blocked/error state.

## Best Possible Solution

Add a channel-agnostic guard at the embedded assistant reply seam that detects strong first-person execution intent plus runnable shell fences when the turn has no real tool lifecycle, replaces that text with a truthful blocked/error reply, and preserves legitimate tool-backed replies and benign examples.

## Reproduction Assessment

Yes. The report gives a real Telegram session timeline, and current main has a focused seam reproduction path: feed an assistant message containing first-person install/check text plus a fenced bash command through the embedded subscribe/message-end path with no tool events, then observe that it is still eligible for block-reply delivery.

## Solution Assessment

Yes. Reviving the narrow #61132 direction on current main is the maintainable fix: put the guard at the embedded reply delivery boundary, make it provider/channel agnostic, bypass it for real tool activity, and cover both blocked and tool-backed cases in tests.
