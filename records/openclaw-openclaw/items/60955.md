## Summary

Keep this issue open. Current main has broader strict-agentic/no-plan-only machinery, but it does not cover the reported Telegram direct-session case: the report used a Kimi provider/model and a fenced shell block, while the current guard is scoped away from that provider and explicitly skips fenced text.

## Best Possible Solution

Revive or reimplement the narrow #61132 direction on current main: add a channel-agnostic embedded assistant reply guard that detects strong first-person execution intent plus runnable shell fences when the turn has no real tool lifecycle, replaces it with a truthful blocked/error reply, and adds focused regression coverage for both blocked no-tool and allowed tool-backed Telegram-style delivery paths.
