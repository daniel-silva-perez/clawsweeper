## Summary

Keep this open. Current main has partial visibility and delivery mitigations, but it still lacks the requested gateway-level silence watchdog and turn-level response-delivery confirmation state machine; the open related issues cover only parts of the problem.

## What This Changes

The issue requests gateway-native silence watchdog beacons plus turn-level outbound response delivery confirmation with retry, dedupe, no-response alerts, and per-channel configuration.

## Best Possible Solution

Keep this as a core/runtime design item. Align the active-run visibility work with #47382 and #65919, then add a generic delivery-confirmation contract around the shared outbound/reply pipeline: stable turn/response IDs, delivered/failed/no-response states, retry dedupe, per-channel capability/config, and regression coverage for final replies, message-tool sends, chunked delivery, preview/final transitions, and final-only channels.

## Reproduction Assessment

Not applicable as a single bug reproduction; this is a feature/design request. We do have a high-confidence implementation-gap check: exact config/hooks/state names are absent, and the current reply, typing, outbound queue, and docs paths only cover partial mitigations.

## Solution Assessment

The requested direction is broadly right, but it should not land as a small default-on timer without maintainer design review. The maintainable path is a shared runtime contract that respects channel capabilities, final-only delivery settings, and existing queue recovery semantics.
