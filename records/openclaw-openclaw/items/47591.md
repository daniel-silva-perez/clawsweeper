## Summary

Keep this issue open. Current main has useful mitigations for long-turn visibility and outbound delivery recovery, but it still does not implement the requested gateway-level silence watchdog or response-delivery confirmation state machine. The related open #65919 covers active-run presence, not the end-of-turn delivery confirmation, retry dedupe, and no-response alerting requested here.

## Best Possible Solution

Keep this open as a core/runtime design item. Align the active-run visibility part with #65919, then add a generic delivery-confirmation contract around the shared reply/outbound pipeline: turn/response IDs, delivered/failed/no-response states, retry dedupe, per-channel capability/config, and regression coverage for final replies, message-tool sends, chunked delivery, preview/final transitions, and final-only channels.

## Risks / Open Questions

- Closing now would conflate existing typing, preview-progress, diagnostics, and queue-recovery mitigations with the stronger runtime-enforced watchdog and delivery-confirmation contract requested here.
- A future implementation needs to preserve documented final-only channel semantics such as Telegram `streaming.mode: "off"` while adding visibility guarantees.
- Delivery retry/dedupe must avoid duplicate user-visible messages when an original channel ACK arrives late after a retry.
