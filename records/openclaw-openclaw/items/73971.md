## Summary

Keep open: current main still has a narrow Control UI/config validation UX mismatch. The form renderer labels array rows one-based, while config validation and save-error summaries still expose canonical zero-based Zod path segments, so the reporter's row-number confusion remains valid.

## What This Changes

The issue asks OpenClaw to make Control UI row numbering for additional model entries match the numbering shown in validation errors.

## Best Possible Solution

Keep ConfigValidationIssue.path and API details canonical for machine compatibility, but add display-layer formatting for Control UI config-save validation errors so array entries are shown with the same one-based row labels users see in the form, with the raw path preserved in details when useful.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an invalid item in a Control UI config array such as models.providers.<provider>.models or agents.defaults.model.fallbacks and save: the visible row badge is one-based while the gateway validation path is emitted with a zero-based numeric segment.

## Solution Assessment

Yes. The narrowest maintainable repair is UI/user-message formatting for config validation errors, not changing Zod path conversion or the canonical config validation path contract that existing tests assert.
