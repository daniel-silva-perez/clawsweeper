## Summary

Keep open: current main still has the reported Control UI numbering mismatch. Array rows are displayed one-based, while config validation errors are built from raw Zod path indexes and surfaced unchanged during config save/apply failures.

## What This Changes

The issue asks for Control UI config row numbering and backend validation error numbering to match so users can identify the failing model entry correctly.

## Best Possible Solution

Keep canonical ConfigValidationIssue.path values zero-based for API and test compatibility, but add Control UI-facing formatting for config validation errors so array entries are described with the same visible row labels users see in the form.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: make an invalid entry in a Control UI config array such as models.providers.<provider>.models, save it, and the visible row badge is one-based while the returned validation path is zero-based.

## Solution Assessment

Yes for a repair candidate. The narrowest maintainable solution is to adjust display-layer formatting for Control UI config-save validation errors, not the canonical Zod/config validation path contract.
