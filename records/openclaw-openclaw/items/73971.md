## Summary

Keep open. Current main still has the reported UX mismatch: the Control UI renders config array rows with one-based badges while backend validation keeps raw zero-based Zod path segments and returns those paths in save/apply errors. The safe repair is a display-layer formatter, not changing the canonical config issue path contract.

## What This Changes

The issue requests making Control UI and backend/config validation numbering consistent for array rows such as available model entries, because the UI can show row #4 while the error path points at zero-based entry 3.

## Best Possible Solution

Keep canonical config validation paths zero-based for API/tests, but make Control UI save/apply validation errors render array entries with the same one-based row labels users see in the form, with raw paths preserved in structured details or debug output.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an invalid entry in a Control UI config array such as models.providers.<provider>.models or agents.defaults.model.fallbacks, then save: the row badge is one-based while the returned validation path is formatted from zero-based path segments.

## Solution Assessment

Yes for a new repair. The narrowest maintainable solution is UI/config-save display formatting that uses the existing GatewayRequestError.details.issues, not changing Zod path mapping or ConfigValidationIssue.path semantics globally.
