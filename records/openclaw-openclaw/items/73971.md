## Summary

Keep open. Current main still exposes zero-based config validation paths while the Control UI labels array rows one-based, so the reported `#4` versus `...3...` confusion remains valid and narrow.

## What This Changes

The issue requests consistent model/config array numbering between the Control UI row labels and backend validation error messages.

## Best Possible Solution

Add a display-only formatter for human-facing config issue paths that presents array segments as the same one-based `#N` ordinals the form shows, while keeping structured `details.issues[].path`, config storage paths, and patch/update paths zero-based for API compatibility.

Yes. The source path is enough to reproduce conceptually: a visible `#4` array row maps to internal segment `3`, and current validation summaries print that raw segment in save errors.

## Solution Assessment

Yes. The best fix is a narrow presentation change in config issue formatting/UI error display, not changing schema paths or persisted config semantics.

Current Control UI config arrays display entries as one-based `#1`, `#2`, etc., but human-facing validation/save errors expose zero-based paths like `models.providers.openrouter.models.3.api`.
