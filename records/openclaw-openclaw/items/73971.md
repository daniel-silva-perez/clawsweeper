## Clownfish Work Prompt

Fix the config validation numbering mismatch reported by this issue. Current Control UI config arrays render visible rows as one-based labels like `#4`, but config validation errors are built from raw zero-based Zod path segments like `models.providers.<provider>.models.3.api`, which makes the fourth row look like item 3. Implement a narrow fix that makes user-facing config error text align with the form labels, preferably by adding a human-display formatter for config issue paths. Preserve the structured `ConfigValidationIssue.path` value and API details unless there is a strong compatibility reason and tests prove it. Cover at least model provider arrays such as `models.providers.openrouter.models.3.api`; the formatter should be generic for array segments. Avoid unrelated config-form redesigns, provider behavior changes, generated locale edits, or schema contract churn. Add/update focused tests for the formatter and the config.apply/config.patch error summary, and update the Control UI issue rendering only if it still exposes raw array numbering after the server-side display fix. Changelog expectation: this is user-facing Control UI/config UX, so add a short active-version Fixes entry crediting `Thanks @qinqing133` unless maintainers decide this cleanup is too small for changelog.
