## Summary

Keep open: current main still has a user-facing numbering mismatch. The generic Control UI config form labels array rows one-based, while config validation errors are summarized from raw dot-joined Zod paths with zero-based numeric array segments, so model-list or fallback-model array entries can be reported with a different number than the row label the user sees.

## What This Changes

The issue requests that model configuration row numbers shown in the Control UI and the backend/config-save error text use consistent numbering.

## Best Possible Solution

Keep canonical config issue paths zero-based for machine/API compatibility, but make Control UI config-save validation errors display array entries in a way that matches the visible row label, ideally by showing a human row label such as #4 while preserving the raw path in details when useful.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an invalid entry in a Control UI config array such as models.providers.<provider>.models or agents.defaults.model.fallbacks, then save: the UI row badge is one-based while the returned validation path is formatted from zero-based Zod path segments.

## Solution Assessment

Yes for a new repair. The narrowest maintainable fix is display-layer formatting for config validation errors, not changing ConfigValidationIssue.path or Zod path semantics that existing tests and callers treat as canonical.
