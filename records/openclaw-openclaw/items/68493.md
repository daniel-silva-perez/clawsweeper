## Summary

Keep this issue open. Current main has restart hardening after the report, including unmanaged Windows in-process fallback and bounded supervised lock recovery, but the reported Windows config-edit stale lock/EADDRINUSE path is not directly proven fixed. The Scheduled Task-marked Windows gateway path still uses a service-manager handoff, and the maintainer discussion explicitly said the related restart-drain PR did not cover this failure mode.

## Best Possible Solution

Keep the issue open for a targeted gateway fix or proof. The best path is to reproduce config-file restart on native Windows in both foreground/unmanaged and Scheduled Task-managed modes, then add a regression test around config reload, lock release/reacquire, port rebind, and WebSocket handshake recovery. If the newer supervisor-lock changes fully resolve it, link that proof and close only after confirming the exact Windows config-edit scenario.
