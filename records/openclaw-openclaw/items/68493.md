## Summary

Current main and the shipped v2026.4.27 release already cover the reported Windows gateway restart failure mode: unmanaged Windows SIGUSR1/config restarts no longer detach-spawn a child, the run loop releases and reacquires the gateway lock around in-process restart, and EADDRINUSE bind races are retried before surfacing a lock error.

## What This Changes

Editing restart-requiring gateway config while the gateway is running should not leave Windows users in a stale-lock/EADDRINUSE hot-reload crash loop.

## Best Possible Solution

Keep the shipped restart behavior: Windows unmanaged gateway config restarts should stay in-process, service-managed Windows restarts should go through Scheduled Task handoff, and remaining non-Windows supervisor edge cases should be tracked separately.

## Reproduction Assessment

Yes. The report gives a concrete Windows config edit/SIGUSR1 path, and source inspection traces that path through config.patch restart scheduling into the current run-loop restart handling; I did not run a live Windows gateway in this read-only review.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix for this Windows failure mode because it avoids unsupported detached respawn, preserves service-manager handoff when markers exist, releases/reacquires the lock around in-process restart, and retries transient bind races.
