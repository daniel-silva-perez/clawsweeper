## Summary

Keep this PR open. Current main and the latest release still use the existing `openclaw.mjs` npm bin, so the work is not obsolete, but the proposed patch is not merge-ready because it changes the package entrypoint without updating release preflight, gateway process detection, or preserving the existing CLI bootstrap path; the new local/live scripts also have concrete reliability bugs already visible in the diff and review discussion.

## What This Changes

This PR changes the npm `openclaw` bin to a new runCli launcher, packages that launcher, and adds a local tarball upgrade script, live memory-pro host smoke, and launcher e2e coverage.

## Best Possible Solution

Keep the PR open for maintainer review and revise it into one coherent package-entrypoint plan: either keep `openclaw.mjs` canonical and add tarball/memory-pro regression coverage around that path, or complete the new bin-target transition by preserving startup bootstrap behavior, updating release guards/tests, aligning gateway argv detection, and hardening the new scripts.

## Reproduction Assessment

Yes. A read-only diff/current-main check reproduces the release-preflight mismatch: the PR changes `package.json` to a new bin target while current `scripts/openclaw-npm-release-check.ts` still rejects anything other than `openclaw.mjs`; the diff also directly shows the unawaited async `main()` and PATH-default regression target.

## Solution Assessment

No. The proposed approach is not the best mergeable form yet because it bypasses established launcher/bootstrap contracts and only partially updates package/release consumers; the safer alternatives are to test the existing `openclaw.mjs` path or perform the full bin migration with all release, gateway, and bootstrap seams updated together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the CLI bootstrap in the new launcher:** `scripts/openclaw-runcli-launcher.mjs:14-20`
  - body: The new bin jumps straight to `runCli`, so installed `openclaw` invocations no longer execute the existing `openclaw.mjs`/`src/entry.ts` bootstrap that handles respawn planning, env normalization, warning filtering, compile cache setup, profile/container argv parsing, and fast paths before loading the CLI. That can break environments that rely on those startup adjustments.
  - confidence: 0.91
- **[P1] Load the stable run-main entry:** `scripts/openclaw-runcli-launcher.mjs:9`
  - body: The launcher scans the root `dist` directory for `run-main-*.js`, but current build metadata declares `cli/run-main` as the stable entry and repo guards/tests refer to `dist/cli/run-main.js`. In a normal package layout this can throw `missing run-main entry` before the CLI starts.
  - confidence: 0.83
- **[P1] Update the release guard with the bin transition:** `package.json:17`
  - body: Changing `bin.openclaw` here makes the package disagree with `collectReleasePackageMetadataErrors`, which still requires `openclaw.mjs`. The npm release preflight will fail until the validator and its tests are updated for the final entrypoint design.
  - confidence: 0.96
- **[P2] Recognize the new launcher in gateway argv checks:** `package.json:17`
  - body: Switching npm shims to this script can expose `scripts/openclaw-runcli-launcher.mjs` in process command lines, but `isGatewayArgv` does not recognize that path. Gateway lock/liveness checks can then misclassify a running gateway and attempt duplicate startup or lock reclamation.
  - confidence: 0.84
- **[P2] Run the regression against the installed bin:** `scripts/local-upgrade-from-tarball.sh:127-132`
  - body: The script validates `$BIN_PATH`, but then runs `openclaw --version` and the live regression through the default PATH-based command. If another OpenClaw binary appears earlier on PATH, this can pass or fail against the wrong install and miss breakage in the tarball that was just installed.
  - confidence: 0.9
- **[P2] Await the live smoke main promise:** `scripts/test-live-memory-pro-cli-host.mjs:274`
  - body: `main()` is async, but the top-level `try/catch` calls it without `await`, so asynchronous failures bypass the structured JSON error handler and surface as unhandled rejections instead of the intended diagnostic payload.
  - confidence: 0.95
- **[P2] Prefer stdout when extracting stats JSON:** `scripts/test-live-memory-pro-cli-host.mjs:82`
  - body: `combined` puts stderr before stdout, and `extractFirstJsonObject` parses the first `{` it sees. A diagnostic object on stderr can therefore be parsed instead of the `stats --json` stdout payload, making the smoke validate the wrong JSON.
  - confidence: 0.82
