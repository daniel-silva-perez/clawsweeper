## Summary

Keep this PR open. Current main does not already implement the PR's intended successful-fallback rollback: fallback candidate overrides are written before an alternate model attempt and current source only rolls them back on errors or classified failed results, while usage persistence still writes the actual runtime provider/model into the session identity. The PR is a plausible draft fix, but it changes the current tested auto-fallback persistence contract and needs normal maintainer review, validation, and a changelog entry rather than cleanup closure.

## What This Changes

The PR rolls back automatic model-fallback session overrides after successful CLI and embedded fallback runs, separates selected session model persistence from actual fallback usage attribution, and adds focused session accounting coverage.

## Best Possible Solution

If maintainers want primaries retried after every successful automatic fallback, land a narrow runner/session-accounting fix that makes fallback candidate selection ephemeral, keeps fallback notices coherent across turns, attributes usage and cost to the actual runtime model, covers CLI and embedded success paths, and documents the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Current main provides a high-confidence source-level reproduction path: a successful fallback candidate can be persisted through applyFallbackCandidateSelectionToEntry and only failure paths roll it back, while persistSessionUsageUpdate writes providerUsed/modelUsed into the session entry.

## Solution Assessment

Unclear. The PR is a narrow and plausible direction for future turns, but it changes an existing tested persistence contract and should add runner-level success rollback coverage plus a changelog entry before maintainers decide it is the best fix.
