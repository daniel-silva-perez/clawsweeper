## Summary

Keep this PR open. It targets a real sticky fallback bug in current main, but the proposed selected-model persistence can hide the active fallback from existing status/cost/context surfaces, so the patch needs one focused repair before merge.

## What This Changes

The PR rolls back automatic fallback session overrides after successful fallback attempts, separates selected session model persistence from actual fallback usage attribution, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the narrow fallback rollback/accounting direction, but preserve a distinct active fallback model for status, cost, and context reporting while keeping future session routing pointed at the selected primary model.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: a successful non-selected fallback can be persisted as an automatic session override and usage persistence writes the fallback runtime provider/model into the stored session runtime identity. The PR finding is also source-reproducible by applying the patch and observing that fallback notice state can coexist with selected `modelProvider/model`, which current status code treats as no active fallback.

## Solution Assessment

No, not yet. The patch fixes the right runner/session boundaries, but it also reuses `modelProvider/model` for selected routing even though current status code treats those fields as the active runtime model; the safer solution is to update that contract or keep active runtime fields distinct before merge.

- **[P2] Preserve the active fallback model in session status:** `src/auto-reply/reply/session-run-accounting.ts:75-76`
  - body: When fallback uses a different runtime model, `resolveRunSessionModelPersistence` now sends the selected provider/model through to `persistSessionUsageUpdate`, which writes that selected pair into `SessionEntry.modelProvider/model`. Keep active runtime fields available or teach status to read the fallback notice active model before merging.
