## Summary

Keep this PR open for maintainer review. The branch addresses a real fallback/session-accounting path, but it reverses current shipped and documented sticky fallback semantics, so the contract needs an explicit maintainer decision and docs alignment before merge.

## What This Changes

The PR rolls back automatic model-fallback session overrides after successful CLI and embedded fallback attempts, separates selected session model persistence from actual runtime usage attribution, updates focused tests, and adds an Unreleased changelog entry.

## Best Possible Solution

Decide the fallback contract explicitly. If recovered primaries should be retried after successful fallback replies, update the model-failover docs and contract tests around sticky auto overrides while preserving selected-vs-runtime accounting; if avoiding repeated known-bad-primary probes remains the intended behavior, adjust or close this PR instead of merging the rollback.

## Reproduction Assessment

Yes. A source-level reproduction exists on current main: a fallback candidate is persisted as an auto override before the attempt, current success paths return without rollback, session usage stores the runtime model, and docs/tests assert the auto override remains selected across turns.

## Solution Assessment

Unclear. The code change is narrow and plausible for retrying primaries after success, but it is not the best merge as-is because it conflicts with the current model-failover documentation and the shipped 2026.4.26 sticky-fallback contract.
