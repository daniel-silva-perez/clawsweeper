## Summary

Keep this issue open. Current main has meaningful SecretRef, runtime snapshot, host exec env sanitization, pattern redaction, and plugin-hook mitigation surfaces, but it still does not implement the native masked-secrets boundary requested here: no `{{secret:NAME}}` command substitution, no `secrets.mask`/`maskedSecrets` config contract, no exact configured mask list for arbitrary exec output, and no general policy preventing an agent from reading otherwise-readable files or generic environment variables.

## What This Changes

The issue requests a native masked-secrets system so agents can use credentials without reading raw API keys, including placeholder substitution, output redaction, command blocking, and secret-mask config.

## Best Possible Solution

Keep this open until OpenClaw ships or explicitly supersedes a native masked-secret or credential-provider design for agent-executed commands and process output. The end state should keep raw values out of agent-readable files/env/context, inject credentials only at dispatch time, redact configured secret values exactly at tool/log/transcript/message sinks, and document which plugin/SDK hooks remain available for external vault integrations.

## Reproduction Assessment

Yes for the missing native boundary, using dummy credentials: configure a readable dummy env/file secret and inspect the current exec/config code paths, which show no `{{secret:NAME}}` substitution or general configured mask policy. No live secret-bearing reproduction is needed for this read-only review.

## Solution Assessment

No current workaround is the best complete solution. User-isolation scripts and `openclaw-credential-vault` are valuable mitigations, but the maintainable core answer needs a security-owned credential boundary rather than only pattern redaction or command blocking.
