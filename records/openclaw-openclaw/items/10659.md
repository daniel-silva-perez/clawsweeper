## Summary

Keep this issue open. Current main has real partial hardening around structured SecretRefs, active runtime snapshots, command-scoped config resolution, host exec environment sanitization, transcript/log redaction, and plugin hooks that can support external mitigations. It still does not implement the native masked-secrets boundary requested here: no `{{secret:NAME}}` substitution for arbitrary agent-executed commands, no `secrets.mask` or `maskedSecrets` config contract, no exact configured secret mask list for arbitrary exec output, and no general policy preventing an agent from reading otherwise-readable env files or environment variables. The latest provided maintainer-associated review also explicitly kept this open, and the linked workarounds/plugins are mitigations rather than a core implementation.

## Best Possible Solution

Keep this issue open until OpenClaw ships or explicitly supersedes a native masked-secret or credential-provider design for agent-executed commands and process output. A strong solution likely belongs in core/security-owned seams: dispatch-time credential injection, raw values kept out of agent-readable files/env/context, exact output redaction for configured secrets, and clear plugin/SDK hooks for external credential providers. Keep acknowledging `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` as current mitigations.
