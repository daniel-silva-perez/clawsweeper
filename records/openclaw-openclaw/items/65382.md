## Summary

Keep open. Current main and the latest release still render cron-expression schedules as raw `Cron <expr>`, so this PR is not obsolete; the remaining blocker is that the patch humanizes out-of-range cron step values that Croner rejects.

## What This Changes

The PR adds a Control UI presenter helper to format common 5-field cron schedules as `Every ...` strings and adds focused presenter tests for hourly, minute, timezone, and fallback display cases.

## Best Possible Solution

Keep the formatter in `ui/src/ui/presenter.ts`, but make it Croner-range-aware for minute and hour step fields and add presenter tests proving out-of-range steps keep the raw cron fallback. No gateway or scheduler behavior change is needed.

## Reproduction Assessment

Yes. A source-level reproduction on current main is a `CronJob` with `schedule: { kind: "cron", expr: "0 */6 * * *", tz: "UTC" }`, which flows through `formatCronSchedule` to the raw `Cron ... (UTC)` fallback; the PR finding is reproducible from the diff because `parsePositiveCronStep` accepts any safe positive integer.

## Solution Assessment

No, not as written. The UI presenter is the right boundary and the PR is close, but the formatter should only humanize cron step values that are valid for Croner's minute/hour fields or it should keep the raw fallback.
