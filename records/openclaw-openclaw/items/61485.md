## Summary

Keep open. Current main still treats `llm_input` and `llm_output` as observe-only hooks, so the PR addresses a real plugin capability gap, but the proposed patch is not merge-ready because it introduces a cache-ordering bypass for system-prompt rewrites and breaks existing tests that mock the attempt module.

## What This Changes

The PR makes `llm_input` and `llm_output` awaited modifying hooks that can block LLM calls, rewrite prompt/system-prompt text, replace assistant text, and adds sentinel error handling, policy constraints, failover guards, and tests.

## Best Possible Solution

Land a maintainer-approved modifying-hook contract only after the branch is cache-safe, test-clean, documented, and clearly scoped against #20802. The end state should preserve the existing conversation-access and prompt-injection policy boundaries while making prompt/response mutation reliable across provider transports and prompt-cache modes.

## Reproduction Assessment

Yes. The requested gap is reproducible by inspection on current main: `llm_input`/`llm_output` return through `runVoidHook`, and the attempt runner fires them without awaiting or applying returned values. The PR regressions are also checkable from the diff plus existing harness and Google prompt-cache code.

## Solution Assessment

No, not as written. The proposed direction is plausible, but the branch should first rebuild or disable the Google prompt-cache wrapper after `systemPrompt` overrides, update the full-module test mock for `PluginBlockedError`, and align docs/changelog with the new public hook contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Rebuild the Google cache wrapper after overrides:** `src/agents/pi-embedded-runner/run/attempt.ts:2510-2514`
  - body: The new `llm_input` `systemPrompt` override is applied after `prepareGooglePromptCacheStreamFn` has already wrapped the stream with cached content for the old system prompt. Because that wrapper removes the runtime system prompt and forces the cached content, Google prompt-cache eligible runs can ignore a plugin's compliance or policy rewrite. Rebuild or disable the cache wrapper after applying the override.
  - confidence: 0.86
- **[P2] Update the attempt mock to export PluginBlockedError:** `src/agents/pi-embedded-runner/run.ts:1143`
  - body: `run.ts` now imports `PluginBlockedError` from `./run/attempt.js`, but the overflow-compaction harness fully mocks that module with only `runEmbeddedAttempt`. Under Vitest that import is missing, so the run-loop tests fail before exercising the code path. Add the sentinel export to the mock or switch the mock to include the actual module exports.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds an intended plugin capability, but the current ordering can bypass plugin-provided system-prompt rewrites when Google prompt caching is active.

Concerns:

- **[medium] System-prompt rewrites can be bypassed under Google cache:** `src/agents/pi-embedded-runner/run/attempt.ts:2510`
  - body: A security plugin may return a stricter `systemPrompt` from `llm_input`, but the Google prompt-cache wrapper is prepared earlier with the old prompt and then strips runtime system prompt content while sending cached content. That can silently defeat policy rewrites on cache-eligible Google runs.
  - confidence: 0.84
