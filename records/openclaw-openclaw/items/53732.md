## Summary

Keep this issue open. Current main still builds BOOT.md into a normal agent prompt, model fallback retries preserve that prompt for the fallback model, and the message tool has no boot-specific literal-echo guard before sending outbound content.

## Best Possible Solution

Keep this open. Implement a narrow internal boot-run context that tags BOOT.md executions, detects substantial literal echoes of the generated BOOT.md prompt before final reply projection and message-tool delivery, and adds deterministic coverage for primary failure followed by fallback echo while preserving legitimate BOOT.md-requested outbound messages.

## Risks / Open Questions

- The symptom is model-dependent, so a regression test should mock primary-model failure followed by fallback-model echo rather than rely on live Gemini behavior.
- A broad fix that disables all BOOT.md message-tool sends would regress the documented startup-notification workflow; the guard needs to detect substantial prompt echoes only.
