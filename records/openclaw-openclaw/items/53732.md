## Summary

Keeping this issue open. Current main still embeds BOOT.md as ordinary prompt text, sends that body through model fallback attempts, and has no boot-prompt identity or literal-echo guard before the message tool can deliver model-composed text. The prior ClawSweeper comments remain consistent with this checkout; the related PR #55016 is closed unmerged and addressed boot outbound delivery/send-policy behavior rather than suppressing echoed BOOT.md instructions.

## What This Changes

The issue requests preventing fallback models from exposing BOOT.md startup instructions verbatim to users when a primary model fails over.

## Best Possible Solution

Add a narrow internal boot-run context that follows the generated BOOT.md prompt through agent execution and delivery paths, then suppress substantial literal echoes of that generated prompt before user-visible final replies or message-tool sends while preserving legitimate BOOT.md-directed outbound notifications and existing fallback retry semantics.
