## Summary

Keep open. Current main still embeds BOOT.md as ordinary prompt text, reuses that body across model fallback attempts, and has no boot-specific literal-echo guard before final reply or message-tool delivery. The prior ClawSweeper keep-open comment remains accurate on this checkout, and the only closely related PR I found, #55016, is closed unmerged and did not add an echo-suppression guard.

## What This Changes

The issue asks OpenClaw to prevent fallback models from echoing BOOT.md startup instructions to users and to keep startup instructions model-agnostic or suppress literal prompt echoes.

## Best Possible Solution

Add a narrow internal boot-run context that follows runBootOnce through the agent attempt and tool surfaces, suppresses substantial literal echoes of the generated BOOT.md prompt before user-visible final replies and message-tool sends, and leaves normal BOOT.md-directed outbound notifications intact.

## Repair Work Prompt

Fix the BOOT.md fallback echo bug. Current main builds a literal BOOT.md prompt in src/gateway/boot.ts, fallback attempts preserve that body, and src/agents/tools/message-tool.ts has no boot-specific echo guard. Implement a narrow internal boot-run context/marker from runBootOnce through AgentCommandOpts and the relevant attempt/tool surfaces. Detect substantial literal echoes of the generated BOOT.md prompt and suppress them before both final reply projection and message-tool delivery. Do not disable the message tool for boot runs, do not suppress all BOOT.md-requested outbound sends, and do not change general model fallback selection. Add deterministic coverage that simulates a primary model failure followed by a fallback model echoing the BOOT.md prompt, plus coverage that a legitimate BOOT.md-requested message still sends and then returns NO_REPLY. Update docs/changelog only if the final change is user-facing; if adding a changelog fix entry, credit the reporter with Thanks @alvaro630 and keep the bullet single-line.
