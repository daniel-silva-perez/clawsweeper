## Summary

Keep open. Current main still routes `openclaw backup create` through a single `tar.c(...)` archive operation without an ENOENT-specific skip/retry path, while session maintenance can still rename transcript files during cleanup. The issue is also paired with open closing PR #67472, with broader related PR #72251 still open, so it should wait for a merged or deliberately superseded implementation.

## What This Changes

The issue requests that `openclaw backup create` skip session transcript files that disappear during backup creation and report them instead of aborting the whole backup.

## Best Possible Solution

Land one coherent backup archiver fix in `src/infra/backup-create.ts` that treats expected per-entry deleted-file ENOENT as non-fatal, reports skipped files clearly, and adds a regression test proving backup creation succeeds when a planned session transcript is removed or renamed during archive creation.

## Reproduction Assessment

Yes. The issue provides a concrete race path, and current source confirms backup planning can hand a state tree to node-tar while session maintenance can rename transcript files during the later archive walk; a focused regression can delete a planned transcript before tar lstat/open and assert backup creation continues.

## Solution Assessment

No, current main is not the best solution because it still lets node-tar abort the whole archive on missing entries. The reporter's per-entry ENOENT skip is a narrow maintainable fix, while #72251 may be preferable only if it explicitly covers this deleted-file case as well as EOF races.
