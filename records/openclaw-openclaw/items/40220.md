## Summary

Keep open. Current main still formats the top-level file-log `time` field with an explicit local offset while serializing the original tslog `_meta` object unchanged, so `_meta.date` remains a `Date` that becomes a UTC `Z` string in raw JSONL logs. The only directly related implementation PR, #48717, is closed and unmerged with zero changed files.

## What This Changes

The issue asks for raw JSONL log timestamps to avoid showing `_meta.date` in UTC while the top-level `time` field uses the configured or local offset timestamp.

## Best Possible Solution

Patch the file-log transport to build a copied JSON payload whose `_meta.date` is the same explicit-offset timestamp as top-level `time`, preferably derived from the original tslog timestamp. Add focused regression coverage for raw JSONL output and leave any switch from host-local transport timestamps to user timezone as a separate policy change.
