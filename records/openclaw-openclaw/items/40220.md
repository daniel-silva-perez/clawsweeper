## Summary

Keep open. Current main still rewrites only the top-level file-log `time` field while serializing the original log object, so nested `_meta.date` can still emit as UTC `Z`; the directly related PR was closed unmerged with no file changes.

## What This Changes

Request to make raw JSONL log `_meta.date` use the same local-offset timestamp semantics as the top-level `time` field.

## Best Possible Solution

Normalize `_meta.date` only in the copied JSONL file-log payload so it equals the already computed top-level `time` for that record, preserve redaction and structured/trace field behavior, avoid mutating the original tslog object, and add focused regression coverage for both fields.

## Reproduction Assessment

Yes. Static source inspection plus a minimal `JSON.stringify` check reproduces the mismatch: nested `_meta.date` remains a `Date` serialized as UTC `Z`, while sibling `time` can remain a local-offset string.

## Solution Assessment

Yes. The narrowest maintainable solution is to patch the serialized file-log payload’s copied `_meta.date` to the same computed `time` string; changing `userTimezone`, envelope timestamp policy, console formatting, or CLI rendering would broaden the fix unnecessarily.
