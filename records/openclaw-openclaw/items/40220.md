## Summary

Keep open. Current `main` still computes the file-log top-level `time` with the local-offset formatter, then serializes the original tslog object unchanged, so nested `_meta.date` can still stringify as UTC `Z`; the directly related PR #48717 is closed, unmerged, and empty, and the latest release keeps the same file transport shape.

## What This Changes

The issue asks OpenClaw to make file-log `_meta.date` and top-level `time` use a consistent timezone-aware representation instead of mixing UTC `Z` with local offset output.

## Best Possible Solution

Normalize `_meta.date` in the copied JSONL file-log payload so it equals the computed top-level `time` string for the same record, preserve the existing file-log timezone policy, avoid mutating the original tslog object, and add regression coverage that asserts both fields match.

## Reproduction Assessment

Yes. The source path plus a minimal `JSON.stringify` check reproduce the mismatch: file logging rewrites only `time`, while nested `_meta.date` remains a `Date` that serializes to UTC `Z`.

## Solution Assessment

Yes. The narrowest maintainable fix is to patch the serialized file-log payload’s `_meta.date` to the same computed `time` string; changing user timezone policy or CLI rendering defaults would conflate this with separate logging behavior.
