## Summary

Keep open. Current main still computes the top-level file-log `time` with an offset but serializes the original log object without overriding nested `_meta.date`, so raw JSONL logs can still show `_meta.date` as a UTC `Z` string while `time` shows host-local offset time. The directly related fix PR is closed, unmerged, and has no changed files, so this remains an actionable logging serialization bug.

## What This Changes

The issue requests consistent raw log timestamp fields so `_meta.date` does not remain UTC while the top-level `time` field is rendered with a local/offset timestamp.

## Best Possible Solution

Normalize `_meta.date` only in the copied JSONL file-log payload so it equals the computed top-level `time` for the same record, avoid mutating the original tslog object, preserve existing host-local transport timestamp semantics, and add regression coverage for both fields.
