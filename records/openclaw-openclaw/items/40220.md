## Summary

Current main still has the raw JSONL mismatch: the file transport adds a local-offset top-level `time` field but serializes the original tslog `_meta` object unchanged, so `_meta.date` remains a `Date` that JSON encodes as UTC. The directly related fix PR is closed, unmerged, and had no file changes, so this should stay open and is a narrow repair candidate.

## What This Changes

The issue asks for log `_meta.date` and top-level `time` timestamps to use a consistent timezone/offset instead of showing UTC and local time for the same event.

## Best Possible Solution

Normalize `_meta.date` in the copied JSONL file-log payload from the same timestamp used for top-level `time`, with regression coverage proving both fields match and keep an explicit offset. Leave broader user-timezone and envelope timestamp policy unchanged unless maintainers make a separate product decision.
