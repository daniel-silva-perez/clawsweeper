## Summary

Keep open. Current main still writes the file-log top-level `time` with the local-offset formatter while serializing the original log object unchanged, so nested `_meta.date` can still stringify as UTC `Z`; the directly related fix PR is closed, unmerged, and empty, and the latest release has the same file-transport shape.

## What This Changes

The issue asks for JSONL log timestamp fields to be consistent so `_meta.date` does not show UTC while top-level `time` shows the local-offset timestamp.

## Best Possible Solution

Normalize `_meta.date` in the copied JSONL file-log payload so it equals the computed top-level `time` string for the same record, preserve the existing file-log timezone policy, avoid mutating the original tslog object, and add regression coverage that asserts both fields match.

## Reproduction Assessment

Yes. The static source path and a minimal `JSON.stringify` check reproduce the mismatch: file logging rewrites only `time`, while nested `_meta.date` remains a `Date` that serializes to UTC `Z`.

## Solution Assessment

Yes. The narrowest maintainable fix is to patch the serialized file-log payload’s `_meta.date` to the same computed `time` string; changing user timezone policy or CLI display defaults would conflate this with separate logging issues.
