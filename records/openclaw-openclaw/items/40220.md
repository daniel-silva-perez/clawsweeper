## Summary

Keep open. Current `main` still rewrites only the file-log top-level `time` field to a local-offset timestamp and then serializes the original log object, so nested `_meta.date` can still stringify as UTC `Z`; the directly related fix PR is closed, unmerged, and empty.

## What This Changes

The issue asks for raw JSONL log metadata timestamps to align with the configured/local-offset `time` field instead of leaving `_meta.date` serialized as UTC.

## Best Possible Solution

Normalize `_meta.date` only in the copied JSONL file-log payload so it equals the computed top-level `time` for the same record, preserve the current file-log timezone policy, avoid mutating the original tslog object, and add focused regression coverage for both fields.

## Reproduction Assessment

Yes. The source path plus a minimal `JSON.stringify` check reproduce the mismatch: file logging rewrites only `time`, while nested `_meta.date` remains a `Date` that serializes to UTC `Z`.

## Solution Assessment

Yes. The narrowest maintainable solution is to patch the serialized file-log payload’s `_meta.date` to the same computed `time` string; changing user timezone policy, CLI rendering, or console formatting would conflate this with separate logging behavior.
