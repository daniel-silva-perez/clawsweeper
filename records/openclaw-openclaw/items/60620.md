## Summary

Keep this issue open. Current main still has a deferred configured-channel path that can reload the full startup plugin id set after listen, while the loader uses a distinct setup-runtime vs full cache key and calls `register()` during activation. The related shipped fix for broad runtime ensure calls addresses #69250, but it does not cover this deferred reload shape.

## Best Possible Solution

Keep the issue open and fix the deferred configured-channel startup path so non-deferred startup plugins are registered once per Gateway process. The conservative implementation is to narrow the post-listen reload to `deferredConfiguredChannelPluginIds`, or otherwise upgrade only deferred channel records without rerunning sidecar, memory, provider, context-engine, or other startup plugin `register()` calls. Add a regression test with a non-bundled startup plugin plus a deferred configured channel: startup plugin register count should stay at 1 while the deferred channel reaches full runtime.
