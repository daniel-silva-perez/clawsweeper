## Summary

Keep this issue open. Current main includes a related fix for broad runtime ensure calls after a gateway-bindable boot load, but the deferred configured-channel reload path still passes the full startup plugin id set back through an activating plugin load, so non-deferred startup plugins can still run register() twice in that boot shape.

## Best Possible Solution

Keep the issue open and fix the deferred configured-channel startup path so non-deferred startup plugins are registered once per Gateway process. A conservative fix would narrow the post-listen reload to deferredConfiguredChannelPluginIds, or merge/upgrade deferred channel records without rerunning sidecar, memory, context-engine, provider, or other startup plugin register() calls. Add a regression test with a non-bundled startup plugin plus a deferred configured channel and assert the startup plugin register count remains 1 while the deferred channel reaches full runtime.
