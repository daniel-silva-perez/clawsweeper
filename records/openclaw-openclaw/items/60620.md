## Summary

Keep this issue open. Current main still has a deferred configured-channel startup path that reloads the full startup plugin scope after request context is available, so non-deferred startup plugins can still run activation-time registration twice in that boot shape.

## What This Changes

The issue reports duplicate Gateway plugin initialization on restart and asks for each plugin's register/load cycle to run once per Gateway process boot.

## Best Possible Solution

Fix the deferred configured-channel startup path so only deferred channel plugins are upgraded to full runtime after listen, while sidecar, memory, provider, context-engine, and other non-deferred startup plugins keep their single initial registration. The regression should prove the deferred channel reaches full runtime and an unrelated startup plugin's register count remains 1.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to boot Gateway with a non-bundled startup plugin that increments a register counter plus at least one configured deferred channel plugin; current code loads startupPluginIds once in setup mode and then reloads startupPluginIds again in full mode after listen.

## Solution Assessment

Yes, the narrowest maintainable fix is to change the deferred reload scope to deferredConfiguredChannelPluginIds, or equivalently upgrade only deferred channel records without reactivating the full startup plugin set. Reworking loader-wide cache semantics alone would be broader and less direct.
