## Summary

Keep this PR open. Current main still lacks the requested outbound metadata route, so the PR is not obsolete, but the proposed patch is not ready to land because metadata is still dropped before channel plugin context, Hermes-specific routing values are hardcoded in core paths, the new trace-id test leaks global spies, and the branch mixes an unrelated credential-sync change into a delivery/protocol feature.

## What This Changes

The PR adds Hermes arbiter metadata to sendMessage, gateway send, direct delivery, channel outbound context, delivery queue persistence/recovery, task and agent notification paths, Swift gateway models, and tests, while also changing external CLI auth-profile bootstrap behavior.

## Best Possible Solution

Keep the PR open and ask for a focused rebase/split: land a generic opaque outbound metadata seam through sendMessage, gateway send, direct delivery, ChannelOutboundContext, queue persistence, and recovery, with channel-level tests proving adapters receive it. Keep Hermes metadata construction and topic/bot selection configurable or plugin-owned unless maintainers explicitly want core-owned Hermes defaults, and remove unrelated auth changes from this branch.

## Reproduction Assessment

Yes. Current main reproduces the missing capability by rejecting metadata on gateway send validation, exposing no metadata field in sendMessage or ChannelOutboundContext, and persisting no metadata in delivery queue entries; the PR-specific forwarding bug is reproducible with a channel adapter assertion because createChannelHandler is still called without metadata.

## Solution Assessment

No. The proposed direction is useful, but this patch is not the best solution as-is because it misses the final handler forwarding step, hardcodes deployment-specific Hermes routing in core, leaks test globals, and mixes in unrelated auth behavior; a narrower generic metadata seam plus plugin/config-owned Hermes construction is safer.
