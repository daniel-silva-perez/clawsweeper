## Summary

Keep open. Current main still has the reported stop-timeout path, the latest release tag contains the same behavior, and the regression test still asserts that a second start is blocked after timeout; the already-linked implementation PR should be reviewed rather than replaced by a new repair job.

## What This Changes

The issue asks for `stopChannel` timeout recovery to release stale channel task/abort state so health-monitor restarts can actually start a fresh polling or socket lifecycle.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged health-monitor restarts, clears stale task/abort bookkeeping, marks the account stopped or disconnected, lifecycle-gates late callbacks from retired tasks, and updates the gateway regression test to prove a fresh task is registered after timeout.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`: after a stop timeout and a second start attempt, the test expects the replacement start not to run and the runtime to stay `running: true`.

## Solution Assessment

Unclear until PR #72912 is reviewed, but the safest direction is broader than deleting the maps: forced retirement plus lifecycle-gated callbacks prevents late settlement from the timed-out task from corrupting the replacement lifecycle.
