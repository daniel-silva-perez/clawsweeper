## Summary

Keep open. Current main still has the reported stop-timeout path, the latest release tag still contains the same behavior, and the colocated gateway test still asserts that a replacement account task is blocked after timeout. The linked open implementation PR should be reviewed or replaced by an equivalent lifecycle fix before this issue closes.

## What This Changes

The issue reports that `stopChannel` can time out while leaving stale channel task bookkeeping behind, causing health-monitor restarts to appear successful without starting a fresh polling or socket lifecycle.

## Best Possible Solution

Implement and land a gateway lifecycle recovery path that can retire a wedged health-monitor restart, clear stale task and abort bookkeeping, mark the old runtime stopped or disconnected, guard callbacks from retired tasks, and replace the current regression test with one proving a fresh task starts after timeout.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`: after a stop timeout and second start attempt, the test expects the replacement start not to run and the runtime to remain `running: true`.

## Solution Assessment

Unclear until PR #72912 is reviewed. The safest direction is broader than only deleting map entries: forced retirement plus lifecycle-gated callbacks addresses both the blocked restart and the replacement-task race called out in the prior PR discussion.
