## Summary

Keep open. Current main still has the reported stop-timeout path: the timed-out task remains registered, the runtime is left running, and the health monitor can flow into a no-op restart. The latest release still contains the same behavior, and open PR #72912 is the current implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue requests fixing channel stop timeout handling so a health-monitor restart can recover after a channel task ignores abort and wedges in the runtime store.

## Best Possible Solution

Land #72912 or an equivalent gateway lifecycle fix that can forcibly retire a wedged health-monitor restart, clear stale task and abort bookkeeping, mark the old runtime stopped or disconnected, gate callbacks from retired tasks, and replace the current regression test with one proving a fresh task starts after timeout.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`: after a stop timeout and second start attempt, the test expects the replacement start not to run and the runtime to remain `running: true`.

## Solution Assessment

Yes for the direction, pending PR review. The maintainable fix should be broader than only deleting map entries: forced lifecycle retirement plus callback gating addresses both the blocked restart and replacement-task race.
