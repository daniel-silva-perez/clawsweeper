## Summary

Keep open. Current main still has the reported stop-timeout behavior: a stale task entry blocks a replacement start, the timeout branch returns before clearing task/abort bookkeeping while marking the account running, and the colocated gateway test still asserts that the replacement task is not started. The latest provided release tag, v2026.4.29, contains the same source behavior, and the related open implementation PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks for channel stop timeout handling to release stale task and abort bookkeeping, report the lifecycle as stopped, and allow health-monitor restarts to register a fresh channel task.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that can forcibly retire wedged health-monitor restarts, clear stale task and abort bookkeeping, mark the retired runtime stopped or disconnected, lifecycle-gate late callbacks from retired tasks, and replace the current regression test with proof that a fresh task starts after timeout.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`: after a stop timeout and second start attempt, the test expects the replacement start not to run and the runtime to remain `running: true`.

## Solution Assessment

Unclear until PR #72912 is reviewed. The safest fix direction appears broader than only deleting map entries, because lifecycle-gating retired task callbacks is needed to prevent old callbacks from corrupting a replacement task's status.
