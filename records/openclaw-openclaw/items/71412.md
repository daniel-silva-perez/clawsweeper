## Summary

Current main still contains the reported stop-timeout path: `stopChannel` returns before clearing the task/abort maps, keeps the account marked running, and `startChannelInternal` still silently returns when a task entry exists. The current test suite also still asserts that behavior. The provided GitHub context shows open PR #73648 uses closing syntax for this issue and is not merged, so maintainer cleanup should keep this issue open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review/land PR #73648 or an equivalent gateway channel lifecycle fix. The fix should clear stale `store.aborts` and `store.tasks` entries on stop timeout, mark the timed-out runtime stopped instead of running, guard replaced task callbacks from mutating the new task's status, and update the gateway regression tests to prove a fresh task starts after timeout.
