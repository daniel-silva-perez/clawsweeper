## Summary

Keep this issue open. Current main still has the reported stop-timeout failure mode in source and tests, and the relevant fix is not merged; the strongest current implementation candidate is open PR #72912, with older PRs #71456 and #73648 closed unmerged.

## What This Changes

The issue requests fixing channel stop-timeout recovery so health-monitor restarts can start a fresh channel lifecycle instead of leaving stale task/runtime state behind.

## Best Possible Solution

Review and land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged health-monitor restarts, clears stale task/abort bookkeeping, marks the runtime stopped/disconnected, lifecycle-gates late callbacks from retired tasks, and updates the channel manager and health-monitor tests before closing this issue.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`: after a stop timeout and a second start attempt, the test expects only one task start and the runtime to remain `running: true`. The live channel scenarios were not rerun.

## Solution Assessment

Unclear until PR #72912 is reviewed, but its force-retire path appears safer than the minimal unconditional cleanup because it keeps manual stop behavior conservative and gates stale lifecycle callbacks. The OP's map-cleanup change is necessary, but the race-hardening in the open PR or an equivalent is the maintainable endpoint.
