## Summary

Keep open. Current main still has the reported stop-timeout path: timeout returns before clearing the stale task/abort entries, leaves the runtime marked running, and the existing regression test still asserts that a second start is blocked. The later Slack confirmations broaden the affected surface, and open PR #73648 is the active implementation candidate, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests fixing a channel stop-timeout path that leaves stale channel task state registered, causing health-monitor restarts to no-op.

## Best Possible Solution

Land PR #73648 or an equivalent gateway lifecycle fix that clears stale task/abort entries on stop timeout, marks the runtime stopped, prevents superseded task callbacks from overwriting replacement task state, and updates the regression tests to prove a fresh restart registers.
