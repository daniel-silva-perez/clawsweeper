## Summary

Keep this issue open. Current main still has the reported timeout path: `stopChannel` returns before clearing the stale task/abort entries, marks the account as running, and `startChannelInternal` still no-ops when `store.tasks` contains the account. An open implementation PR, #73648, uses closing syntax for this issue, so it should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Review and land PR #73648 or an equivalent core gateway fix. The fix should clear stale `store.aborts` and `store.tasks` entries when `stopChannel` times out, mark the timed-out runtime stopped instead of running, guard old timed-out task callbacks from overwriting the replacement task's state, and update the gateway regression tests to prove a subsequent restart registers a fresh task.
