## Summary

Keep open. Current main still sends Feishu text/card, image/file, and initial streaming-card messages through one-shot Feishu SDK create/reply calls, and non-zero Feishu message API responses still throw immediately without rate-limit classification or bounded retry. The related #14206 is useful context for proactive throttling, but it does not supersede this retry-on-rate-limit failure path.

## Best Possible Solution

Fix this inside the bundled Feishu plugin with a bounded, Feishu-owned retry wrapper for outbound message create/reply delivery. Cover text/post, cards, image/file message sends after upload, and the initial streaming-card send; classify explicit Feishu rate-limit/flow-control failures, honor retry timing hints when available, preserve withdrawn-reply fallback behavior, and add focused tests without changing global channel retry policy.
