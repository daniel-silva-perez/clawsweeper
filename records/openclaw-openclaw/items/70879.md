## Summary

Keep open. Current main still sends Feishu text/card, media, and initial streaming-card messages through one-shot Feishu SDK calls, and non-zero message API responses are converted to immediate exceptions rather than classified for bounded retry/backoff. The related open Feishu pacing request #14206 is useful context but does not supersede this narrower retry-on-rate-limit delivery bug.

## What This Changes

Add bounded retry/backoff handling in the Feishu plugin for outbound message sends that fail with Feishu rate-limit or flow-control API errors.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry helper for outbound message create/reply delivery, covering text/post, cards, image/file sends after upload, and initial streaming-card sends while preserving withdrawn-reply fallback behavior and classifying Feishu-specific retryable responses deliberately.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by injecting a Feishu SDK response or thrown error for a rate-limit code into the current send helpers; the code performs one create/reply call and throws immediately without a second attempt.

## Solution Assessment

Yes. The best fix is a narrow Feishu plugin retry wrapper that classifies Feishu-specific retryable responses and timing hints; a global channel retry policy would be broader and less safe for non-idempotent sends.
