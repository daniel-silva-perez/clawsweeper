## Summary

Keep open. Current main still sends Feishu text, card, image, and file messages through direct SDK create/reply calls and immediately throws on non-zero Feishu message API responses. I found Feishu-specific backoff handling only for typing-indicator reactions, not outbound message delivery, and the latest release tag has the same send implementation.

## Best Possible Solution

Keep this issue open and fix it in the bundled Feishu plugin. Add a Feishu-owned bounded retry wrapper around message create/reply delivery for text, card, image, and file sends; classify Feishu flow-control/rate-limit responses such as 11232, 2200, and 429; honor retry-after hints when present; avoid broad non-idempotent retries; and cover retry/no-retry behavior with focused Feishu tests plus docs/changelog if a config knob is added.
