## Summary

Keep open. Current main still has one-shot Feishu outbound message delivery for text/post, cards, media sends, and initial streaming cards; rate-limit or flow-control responses are either thrown immediately or rethrown as ordinary errors. The related pacing request #14206 is useful context, but it does not supersede this narrower retry-on-send reliability bug.

## What This Changes

Add bounded exponential-backoff retry handling for Feishu outbound message sends when the Feishu API returns rate-limit or flow-control errors.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry helper around outbound message create/reply delivery. It should deliberately classify Feishu retryable response bodies and thrown SDK/Axios errors, preserve withdrawn-reply fallback and thread-reply safety behavior, and cover text/post, interactive cards, media sends after upload, and initial streaming-card delivery without introducing a global channel retry policy.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by injecting a Feishu SDK response or thrown error with a retryable rate-limit code into the current send helpers; the current code performs one create/reply call and throws without a second attempt.

## Solution Assessment

Yes. The narrow Feishu-owned retry wrapper is the best direction because message sends are potentially non-idempotent, so a broad channel retry regex would be more likely to duplicate deliveries or affect unrelated plugins.
