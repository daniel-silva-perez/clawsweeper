## Summary

Keep open. Current main still sends Feishu text/post messages, cards, media messages, and initial streaming cards with one SDK/API call and no bounded retry path for retryable rate-limit or flow-control failures. The broader pacing request #14206 is useful context but does not supersede this narrower send-retry reliability bug, and #64448 is a related caution that any retry fix needs idempotency or duplicate-delivery safeguards.

## What This Changes

Add Feishu-plugin retry handling for outbound message sends that fail with rate-limit or flow-control API errors such as 2200, 11232, or 429.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry helper around outbound message create/reply delivery. It should classify retryable Feishu response bodies and thrown SDK/Axios errors, preserve withdrawn-reply fallback and thread-reply safety behavior, cover text/post, cards, media sends, and initial streaming-card sends, and avoid a global core/channel retry policy for non-idempotent sends.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by injecting a retryable Feishu SDK response or thrown error into the current send helpers; current main performs one create/reply call and rejects without a second attempt.

## Solution Assessment

Yes. A narrow Feishu-owned retry wrapper is the best direction because message sends can be non-idempotent, so a broad shared retry layer would be more likely to duplicate deliveries or affect unrelated plugins.
