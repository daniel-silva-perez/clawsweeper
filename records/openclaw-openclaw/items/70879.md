## Summary

Keep open. Current main still sends Feishu text/card, media, and initial streaming-card messages through one-shot Feishu SDK create/reply calls, with non-zero message API responses converted to immediate exceptions instead of classified retry/backoff. The related open Feishu pacing request #14206 is useful context but does not supersede this narrower retry-on-rate-limit recovery bug.

## What This Changes

The issue requests bounded retry/backoff handling in the Feishu plugin when outbound message sends fail with rate-limit or flow-control errors such as 11232, 2200, or HTTP 429.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry helper for outbound message create/reply delivery, covering text/post, cards, image/file sends after upload, and initial streaming-card sends while preserving withdrawn-reply fallback behavior and using Feishu-supported idempotency where retry safety requires it.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by injecting a Feishu SDK response or thrown error for a rate-limit code into the current send helpers: the code performs one create/reply call and throws immediately without a second attempt.

## Solution Assessment

Yes. The best fix is a narrow Feishu plugin retry wrapper that classifies Feishu-specific retryable responses and timing hints; a global channel retry policy would be broader and less safe for non-idempotent sends.

## Repair Work Prompt

Fix Feishu plugin outbound send retry for https://github.com/openclaw/openclaw/issues/70879. Current main calls client.im.message.create/reply directly in extensions/feishu/src/send.ts, extensions/feishu/src/media.ts, and extensions/feishu/src/streaming-card.ts, then throws from extensions/feishu/src/send-result.ts for non-zero responses. Add a Feishu-owned bounded retry wrapper for outbound message delivery only: text/post and card create/reply through sendReplyOrFallbackDirect, image/file message create/reply after upload, and initial streaming-card message create/reply. Classify explicit rate-limit/flow-control failures from thrown SDK errors and non-zero Feishu response bodies, including 11232, 429, 99991400, and any confirmed 2200/230020-style retryable send codes after checking the current Feishu docs and @larksuiteoapi/node-sdk 1.62.0 types/source. Honor retry-after or x-ogw-ratelimit-reset style timing hints when the SDK exposes them; otherwise use bounded exponential backoff with jitter through the existing plugin SDK retry helper. Preserve withdrawn-reply fallback behavior. Do not retry ambiguous non-idempotent network failures unless Feishu message uuid/idempotent_id support makes the operation safe. Do not change global channel retry policy or core retry defaults. Add focused tests for retry, no-retry, and max-attempt exhaustion across send, media, streaming-card, and any new helper. Update Feishu/retry docs only if a config or documented behavior changes, and add a one-line active changelog Fixes entry with Thanks @AxelHu if maintainers treat the reliability fix as user-facing.
