## Summary

Keep open. Current main still routes documented Microsoft Teams conversation-id-shaped team keys through Graph team-name lookup during startup allowlist resolution, so `19:...@thread.skype` or `19:...@thread.tacv2` config can produce a false unresolved warning even while runtime routing continues to match the raw IDs.

## What This Changes

The issue reports that Microsoft Teams startup logs a false `channels unresolved` warning when `channels.msteams.teams` is configured with documented Teams URL-path IDs ending in `@thread.skype`.

## Best Possible Solution

Teach the Microsoft Teams plugin's startup allowlist resolver to treat already-configured Teams conversation IDs as resolved IDs, including both `@thread.skype` and `@thread.tacv2`, while preserving Graph name lookup for display-name entries and adding regression coverage for the no-warning startup path.
