## Summary

Keep open: current main still has the fast-abort response-prefix context gap, and open implementation PRs already target the bug with closing references. The issue should close only after maintainers choose, land, or supersede a canonical fix.

## What This Changes

Resolve `messages.responsePrefix` template variables such as `{model}` and `{thinkingLevel}` for early stop/abort replies instead of sending the raw placeholders.

## Best Possible Solution

Land or supersede the linked fix PRs with one narrow implementation that provides safe runtime/session model and thinking context to fast abort and routed command replies, preserves normal model-selection override behavior, and adds regression coverage for both paths.

## Reproduction Assessment

Yes. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` while a run is active, and the source path shows the abort reply is sent before model-selection context is populated while unresolved variables are preserved literally.

## Solution Assessment

Unclear which open PR is the best final patch without maintainer review. The safest direction is to derive prefix context from actual session/runtime metadata where available, rather than leaking placeholders or dropping useful static identity prefixes.
