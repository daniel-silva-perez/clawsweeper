## Summary

Keep open: current main still has the reported fast-abort prefix-context gap, and open PRs already reference this issue with closing syntax, so it should not be closed before maintainers pick and land a canonical fix.

## What This Changes

Dynamic `messages.responsePrefix` template variables such as `{model}` and `{thinkingLevel}` should be populated, or at least not leaked literally, when `/stop` or `stop` sends an early abort reply.

## Best Possible Solution

Keep the issue open and review or supersede the linked fix PRs with one narrow implementation that supplies a safe prefix context to early abort and command fast paths, carries that context through routed replies, and adds regression tests for dynamic abort prefixes plus normal `onModelSelected` override behavior.

## Reproduction Assessment

Yes. The source path is deterministic: configure a dynamic `responsePrefix`, trigger `/stop` while a run is active, and the fast-abort branch sends before `onModelSelected` populates model/thinking values, while unresolved placeholders are preserved literally.

## Solution Assessment

Unclear which linked implementation is best without maintainer review, but closing or stale-cleaning this is not the right solution. The safer direction is a canonical prefix-context fix that reflects actual runtime/session model metadata rather than only suppressing or leaking unresolved template tokens.
