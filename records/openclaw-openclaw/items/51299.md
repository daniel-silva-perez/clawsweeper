## Summary

Keep open. Current main and the latest release still run the legacy iMessage integration by spawning `imsg rpc` from the gateway process, while the docs still tell headless LaunchAgent users to trigger TCC prompts from the same context, which is the exact path the report says fails on macOS Sequoia/Tahoe. BlueBubbles is the recommended new iMessage path and #43440 tracks a related Full Disk Access parent-process problem, but neither resolves this issue's Automation/LaunchAgent workaround question.

## What This Changes

The issue asks for a working way to grant Full Disk Access and Automation to the legacy `imsg` iMessage channel when OpenClaw runs as a headless macOS LaunchAgent under a dedicated user with SIP enabled.

## Best Possible Solution

Decide the supported legacy `imsg` LaunchAgent story explicitly: either document that headless LaunchAgent TCC prompts are unsupported and steer users to BlueBubbles or a validated GUI-session/SSH pattern, or add a maintained macOS app-mediated/signed wrapper path for `imsg` with diagnostics and docs.

## Reproduction Assessment

Yes. The issue provides a concrete reproduction path: compare `imsg chats`/`imsg send` from Terminal.app with Gateway `imsg rpc` under `ai.openclaw.gateway` for a dedicated macOS user with SIP enabled; current source confirms that OpenClaw still uses that spawned `imsg rpc` path.

## Solution Assessment

Unclear. A docs/status clarification is the narrowest immediate fix, but a real runtime workaround through the signed macOS app, an app bundle wrapper, or MDM profile guidance needs maintainer product and macOS validation first.
