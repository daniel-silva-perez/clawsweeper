## Summary

Keep this issue open. Current main still documents the legacy iMessage/imsg path as needing Full Disk Access and Automation in the same process context, but it does not provide a Sequoia/Tahoe LaunchAgent-safe approval path or specialized recovery for `permissionDenied` / AppleEvents `-1743`. BlueBubbles is recommended for new iMessage setups, and the generic gateway `--wrapper` support exists, but neither current docs nor code prove a working answer for the reported headless LaunchAgent TCC failure.

## Best Possible Solution

Keep this issue open and resolve it with explicit maintainer guidance in the legacy iMessage docs and/or channel diagnostics. A good outcome would either document a tested Sequoia/Tahoe-safe path for imsg under launchd, including whether `gateway install --wrapper` with an app/run-as helper is supported, or state that bare LaunchAgent imsg is not reliable under TCC and recommend BlueBubbles or a GUI-user/remote-Mac setup. Adding probe/status text for `permissionDenied` and AppleEvents `-1743` would make the failure actionable.
