## Summary

Keep open. Current main has some capped surfaces, but the shared Control UI shell, Config editor, and Sessions/data-table primitives still expand across the available content column, so the requested ultrawide-aware behavior is not implemented broadly enough to close.

## What This Changes

The issue requests ultrawide-aware Control UI layouts that cap primary content width or use extra horizontal space for side panels instead of stretching forms and tables across very wide viewports.

## Best Possible Solution

Define a Control UI-wide responsive layout policy: cap readable form/detail content, preserve or intentionally redesign data-dense tables, use side panels where a view has meaningful secondary content, and document any optional full-width or density preference if maintainers want to preserve the current behavior for power users.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction path exists: open the Control UI Config or Sessions/Data Table surfaces at a viewport wider than 2560px and inspect the full-width `.content`, `.config-layout`, and `.data-table` rules shown in current main.

## Solution Assessment

Unclear. The proposed direction is reasonable, but the best maintainable solution is not a blanket CSS cap; it needs a shared layout token/pattern plus view-specific choices for forms, tables, chat, and existing side-panel surfaces.
