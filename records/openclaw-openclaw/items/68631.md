## Summary

Keep open. Current main already implements per-agent `agents.list[].tools.deny` with deny-before-allow semantics, but it still lacks the requested per-agent path allow/deny contract for write-capable filesystem tools or bash/exec path writes. The remaining work is valid but security-sensitive and overlaps adjacent open filesystem/exec policy efforts, so it needs maintainer/security design before implementation.

## What This Changes

The issue requests optional per-agent `tools.deny` plus `tools.path.allow` and `tools.path.deny` rules for fine-grained tool and filesystem write scoping.

## Best Possible Solution

Define one explicit filesystem and exec path-policy contract before implementation: additive config schema/types/docs, canonical path resolution, deny-before-allow semantics, per-agent/global precedence, write/edit/apply_patch coverage, and a separate carefully parsed exec/bash policy for redirects and shell payloads. Then route all affected tools through the shared resolver with focused regression tests and generated config-doc updates.

## Reproduction Assessment

Not applicable as a user-facing bug reproduction, but there is a high-confidence source-level verification path. Current main accepts and enforces `agents.list[].tools.deny`, while the schema and runtime only model filesystem scoping as `workspaceOnly` true/false with no path allow/deny globs.

## Solution Assessment

Unclear for the exact proposed schema. The direction is valid, but the safest solution is a maintainer-reviewed shared policy design rather than adding ad hoc path fields directly to individual tools, especially for canonicalization and bash/exec redirect detection.
