## Summary

Keep open: current main has a generic write-ahead outbound delivery queue and startup recovery, plus a WhatsApp reconnect drain, but I found no Discord-owned reconnect drain that replays pending Discord outbound entries when the gateway reaches READY after a reconnect. The current Discord retry path is still a per-request HTTP retry path, so the specific parity request remains valid.

## What This Changes

Add Discord outbound delivery queue replay/retry for messages that fail during gateway WebSocket reconnect windows, matching the WhatsApp reconnect-drain behavior.

## Best Possible Solution

Keep the generic queue in core and add the missing Discord-owned reconnect drain in the Discord plugin lifecycle, scoped by channel and account, reusing the existing plugin SDK delivery-queue runtime instead of adding Discord-specific core policy.

## Reproduction Assessment

Yes for a focused source-level regression: enqueue or fail a queued delivery with channel="discord", drive the Discord gateway lifecycle through READY/reconnect in a unit test, and observe that no drainPendingDeliveries call currently occurs. No live Discord gateway disconnect test was run.

## Solution Assessment

Yes, a Discord-plugin reconnect drain using the existing delivery-queue runtime is the narrowest maintainable direction. A core migration or broad new config surface is not needed unless implementation discovers a missing generic SDK hook.
