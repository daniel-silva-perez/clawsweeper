## Summary

Keep open. Current main does not have this docs recipe, and the PR is a focused docs-only workaround over existing SecretRef and provider baseUrl primitives, but it should not merge as-is because the container-oriented example points OpenClaw at localhost while the proxy is described as a separate host or Docker-bridge process. In Docker, localhost resolves inside the tenant container, so the documented cutover can fail or give operators a false hardening path.

## What This Changes

This PR adds `docs/gateway/external-auth-proxy.md` and registers it in `docs/docs.json` as a Gateway authentication and secrets recipe for routing model-provider requests through an external credential-injection proxy.

## Best Possible Solution

Publish the recipe only after it clearly separates host/systemd, shared-network-namespace, and multi-container deployments, with address examples that actually reach the proxy from the OpenClaw runtime. Keep the native credential-isolation work tracked in #11829 and #10659 rather than implying this recipe replaces it.

## Reproduction Assessment

Yes. The failure path is reproducible by inspection: the PR tells a tenant/container setup to call `localhost`, while current Docker docs state that localhost inside the container is the container itself, not the host-side proxy.

## Solution Assessment

No, not as written. The recipe is the right lightweight docs direction, but the container baseUrl example should use a reachable host-gateway or bridge address, or explicitly say localhost only applies when OpenClaw and the proxy share the same network namespace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use a reachable proxy address for container setups:** `docs/gateway/external-auth-proxy.md:151`
  - body: The recipe's container-oriented example points `baseUrl` at `http://localhost:18080/anthropic`, but the proxy skeleton listens on the Docker bridge and identifies tenants by bridge source IP. In a tenant container, `localhost` is the container itself rather than the host proxy, so readers following this config will not route model calls through the documented auth proxy. Use a host-gateway/bridge address for multi-container deployments, or scope the localhost example to same-host/shared-network setups.
  - confidence: 0.9
