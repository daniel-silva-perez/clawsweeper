## Summary

Current main should keep this open. Bare channel `/new` and `/reset` now short-circuit to a local acknowledgement instead of invoking the model, so the original bare-prompt reproduction path has changed. But the requested sender-envelope behavior is still not implemented in the reset prompt assembly, and restoring personalized model-generated reset greetings now needs maintainer product judgment because it conflicts with the current static-ack behavior and tests.

## Best Possible Solution

Maintainers should first decide the product behavior: keep bare `/new` and `/reset` as local acknowledgements, or allow a model-generated first reset greeting. If model-generated reset turns remain supported, update `src/auto-reply/reply/get-reply-run.ts` so reset prompts that reach the model include `inboundUserContext`, then add focused coverage for sender metadata on reset-tail or soft-reset paths and Telegram/native command context. If static acknowledgement is intended, document that personalized bare reset greetings are not supported and route customization to the broader reset-prompt follow-up.
