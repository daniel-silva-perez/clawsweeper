## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks several central behaviors the branch tries to add, including `input_length` overflow detection and mirrored-session fallback in `sendMessage`, but the PR is broad, stale against current main, and has concrete correctness plus security-sensitive routing blockers.

## What This Changes

The PR changes outbound delivery fallback and recovery, mirrored session channel selection, context-overflow matching, plain-text outbound sanitization, Argus session/lock metadata, Telegram polling logs, and related tests.

## Best Possible Solution

Land focused, owner-reviewed patches instead of this broad branch: keep the narrow overflow and sanitizer fixes if still needed, make explicit channel selection fail closed, add a real drain/retry path before deferring silent deliveries, and only add Argus metadata after configured-store lookup plus protocol/docs semantics are settled.

## Reproduction Assessment

Yes. The blocking issues have high-confidence static reproduction paths: enqueue a silent delivery while a user-visible queue entry exists, call `sendMessage` with an invalid explicit channel plus mirrored session context, and configure `session.store` away from the transcript directory before Argus lock inspection.

## Solution Assessment

No. The branch addresses real gaps, but combining unrelated recovery, protocol, sanitizer, gateway, and Telegram changes has produced correctness and routing regressions; narrower owner-reviewed patches are the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:493-501`
  - body: This returns immediately after enqueueing the silent delivery, before any send, ack, fail, or live retry path. Since recovery drains pending snapshots on startup/reconnect rather than continuously watching this queue, normal runtime overlap with a user-visible delivery can leave the follow-up pending until an unrelated later drain.
  - confidence: 0.9
- **[P2] Reject explicit channel failures before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: `resolveRequiredChannel` now passes mirrored session fallback even when `params.channel` is explicit. `resolveMessageChannelSelection` falls back when the explicit channel is unavailable, so a typo or malformed explicit channel can deliver through the mirrored session channel instead of failing fast.
  - confidence: 0.86
- **[P2] Use the configured session store for Argus checks:** `src/agents/session-write-lock.ts:313-317`
  - body: `resolveSessionStorePathForLock` hard-codes `sessions.json` beside the session file, but `session.store` supports configured paths and templates. In those setups Argus metadata is missed, so stale-lock cleanup and the watchdog can still reclaim locks the new protection is meant to preserve.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Supply-chain-sensitive surfaces are unchanged, but the diff introduces a concrete outbound data-routing risk through explicit-channel fallback.

Concerns:

- **[medium] Explicit channel fallback can misroute outbound content:** `src/infra/outbound/message.ts:113`
  - body: The proposed `sendMessage` fallback applies even when the caller supplied an explicit channel. If that channel is unavailable or malformed, message content can be routed through the mirrored session channel instead of failing, which can expose content on an unintended channel or recipient path.
