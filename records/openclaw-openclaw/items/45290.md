## Summary

Keep this PR open, but do not merge the current branch as-is. Current main still lacks several requested hardening behaviors, while the PR branch remains broad and has blocking outbound-routing, delivery-queue, and Argus session-store correctness issues.

## What This Changes

The PR changes outbound delivery fallback and recovery, session Argus metadata/lock behavior, context-overflow matching, plain-text outbound sanitization, Telegram polling conflict logs, and related unit tests.

## Best Possible Solution

Land focused, owner-reviewed patches instead of this broad branch: keep narrow overflow and sanitization fixes if still needed, rework delivery fallback around explicit-channel failure precedence and the current recovery/claim model, and only add Argus metadata after the protocol, docs, configured session-store lookup, and lock semantics are aligned.

## Reproduction Assessment

Yes. The blocking findings have static reproduction paths from the PR diff: enqueue a silent delivery while a user-visible queue entry exists, call sendMessage with an invalid explicit channel plus mirrored session delivery context, and configure session.store away from the transcript directory before Argus lock inspection.

## Solution Assessment

No. The branch addresses real gaps, but the current implementation is not the narrowest maintainable fix because it combines unrelated surfaces and introduces new routing and session-store correctness problems; a split or replacement is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:496-501`
  - body: This returns after enqueueing the silent delivery but before any send, ack, fail, or live retry path. Since outbound recovery drains pending entries on startup/reconnect rather than continuously watching the queue, normal runtime overlap with a user-visible delivery can leave the follow-up pending until a later drain.
  - confidence: 0.9
- **[P2] Reject explicit channel errors before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: Passing fallbackChannel when params.channel is explicit lets resolveMessageChannelSelection replace a bad or unavailable explicit channel with the mirrored session channel. A typo can therefore deliver to an unintended channel instead of failing fast.
  - confidence: 0.86
- **[P2] Use the configured session store for Argus checks:** `src/agents/session-write-lock.ts:313-314`
  - body: readArgusSessionContext reads sessions.json beside the session file, but session.store is configurable. In those configurations Argus metadata is missed and stale-lock cleanup/watchdog can still remove locks the new protection is meant to preserve.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Supply-chain-sensitive surfaces are unchanged, but the diff introduces a concrete outbound data-routing risk through explicit-channel fallback.

Concerns:

- **[medium] Explicit channel fallback can misroute outbound content:** `src/infra/outbound/message.ts:113`
  - body: The proposed sendMessage fallback applies even when the caller supplied an explicit channel. If that explicit channel is unavailable or malformed, message content can be routed through the mirrored session channel instead of failing, which can expose content on an unintended channel or recipient path.
  - confidence: 0.82
