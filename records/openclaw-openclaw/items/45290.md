Keep this PR open. Current main still lacks several requested behaviors, so it is not obsolete, but the branch is not merge-ready because the diff introduces concrete delivery and session-lock correctness issues across broad core/runtime/protocol paths.

## What This Changes

The PR branch changes outbound delivery fallback and recovery, session Argus metadata and lock handling, gateway sessions.patch protocol fields, context-overflow detection, plain-text sanitization, Telegram polling logs, and related tests.

## Best Possible Solution

Replace or split the branch into focused PRs: land narrow input_length and plain-text sanitization coverage if still desired, rebase delivery fallback on the current delivery-queue recovery/claim architecture, preserve explicit-channel failure precedence, and take Argus lock/protocol behavior through a smaller owner-reviewed contract change that uses the configured session store.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:496-501`
  - body: This returns after enqueueing the silent delivery but before any send, ack, fail, or live retry path. Since outbound recovery drains pending entries on startup/reconnect rather than continuously watching the queue, normal runtime overlap with a user-visible delivery can leave the follow-up pending until a later restart or drain.
  - confidence: 0.9
- **[P2] Reject explicit channel errors before using fallback:** `src/infra/outbound/message.ts:113-120`
  - body: Passing fallbackChannel even when params.channel is explicit lets resolveMessageChannelSelection replace a bad or unavailable explicit channel with the mirror session channel. A typo can therefore deliver to an unintended channel instead of failing fast.
  - confidence: 0.86
- **[P2] Use the configured session store for Argus lock checks:** `src/agents/session-write-lock.ts:313-314`
  - body: readArgusSessionContext always reads sessions.json beside the session file, but session.store is configurable. In those configurations Argus metadata is missed and stale-lock cleanup/watchdog can still remove locks the new protection is meant to preserve.
  - confidence: 0.84
