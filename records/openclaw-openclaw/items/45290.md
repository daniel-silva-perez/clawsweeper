## Summary

Keep this PR open, but do not merge it as-is. Current main does not fully implement the PR's intended delivery fallback, context-overflow, and sanitizer changes, while the current PR head still has blocking delivery-recovery and outbound-routing regressions that need maintainer-directed splitting or replacement.

## What This Changes

The PR changes outbound delivery fallback and recovery, mirrored session channel selection, context-overflow matching, plain-text outbound sanitization, Argus session metadata/protocol handling, Telegram polling logs, and related tests.

## Best Possible Solution

Land focused, owner-reviewed patches instead of this broad branch: keep narrow sanitizer and overflow-classifier fixes where still needed, make explicit channel selection fail closed before applying mirrored fallback, add a real retry/drain path for deferred silent deliveries, and only introduce Argus metadata with configured-store lookup plus protocol/docs semantics settled.

## Reproduction Assessment

Yes. The blocking issues have high-confidence static reproduction paths: enqueue a silent delivery while user-visible queue work exists, call sendMessage with an invalid explicit channel and mirrored session context, and configure session.store away from the transcript directory before Argus lock inspection.

## Solution Assessment

No. The branch addresses real reliability gaps, but this combined patch is not the best maintainable fix because it introduces correctness and data-routing regressions across several owners; narrower patches with explicit fail-closed routing and configured-store handling are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:493-501`
  - body: This returns after enqueueing the silent delivery, before any send, ack, fail, or live retry path. Since recovery drains pending snapshots on startup/reconnect rather than continuously watching this queue, normal runtime overlap with a user-visible delivery can leave the follow-up pending until an unrelated later drain.
  - confidence: 0.9
- **[P2] Reject explicit channel failures before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: resolveRequiredChannel passes mirrored session fallback even when params.channel is explicit. Because resolveMessageChannelSelection falls back when the explicit channel is unavailable, a typo or malformed explicit channel can deliver through the mirrored session channel instead of failing fast.
  - confidence: 0.86
- **[P2] Use the configured session store for Argus checks:** `src/agents/session-write-lock.ts:313-317`
  - body: resolveSessionStorePathForLock hard-codes sessions.json beside the session file, but session.store supports configured paths and templates. In those setups Argus metadata is missed, so stale-lock cleanup and the watchdog can still reclaim locks the new protection is meant to preserve.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: No dependency, workflow, or package-supply-chain changes are evident in the provided diff, but the PR introduces a concrete outbound data-routing risk.

Concerns:

- **[medium] Explicit channel fallback can misroute outbound content:** `src/infra/outbound/message.ts:113`
  - body: The proposed sendMessage fallback applies even when the caller supplied an explicit channel. If that channel is unavailable or malformed, message content can be routed through the mirrored session channel instead of failing, which can expose content on an unintended channel or recipient path.
