## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks several intended fixes, so it is not obsolete; however the branch is dirty against main and the latest diff has blocking delivery-routing, queue-drain, and session-store correctness problems.

## What This Changes

The PR changes outbound delivery fallback and recovery, session Argus metadata and lock handling, context-overflow matching, plain-text outbound sanitization, Telegram polling conflict logs, and related unit tests.

## Best Possible Solution

Land focused, owner-reviewed patches instead of this broad branch: keep narrow overflow and sanitization fixes if still needed, rework delivery fallback so explicit channel failures fail closed, and only add Argus metadata after the configured session-store lookup and protocol/docs semantics are settled.

## Reproduction Assessment

Yes. The blocking findings have static reproduction paths from the PR diff: enqueue a silent delivery while a user-visible queue entry exists, call sendMessage with an invalid explicit channel plus mirrored session context, and configure session.store away from the transcript directory before Argus lock inspection.

## Solution Assessment

No. The branch addresses real gaps, but it combines unrelated surfaces and introduces new routing and session-store correctness problems; a split or replacement with narrower owner-reviewed patches is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:493-501`
  - body: This returns after enqueueing the silent delivery but before any send, ack, fail, or live retry path. Since recovery drains pending entries on startup/reconnect snapshots rather than continuously watching the queue, normal runtime overlap with a user-visible delivery can leave the follow-up pending until a later drain.
  - confidence: 0.9
- **[P2] Reject explicit channel failures before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: Passing fallbackChannel when params.channel is explicit lets resolveMessageChannelSelection replace a bad or unavailable explicit channel with the mirrored session channel. A typo can therefore deliver to an unintended channel instead of failing fast.
  - confidence: 0.86
- **[P2] Use the configured session store for Argus checks:** `src/agents/session-write-lock.ts:313-314`
  - body: readArgusSessionContext reads sessions.json beside the session file, but session.store is configurable. In those configurations Argus metadata is missed and stale-lock cleanup/watchdog can still remove locks the new protection is meant to preserve.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Supply-chain-sensitive surfaces are unchanged, but the diff introduces a concrete outbound data-routing risk through explicit-channel fallback.

Concerns:

- **[medium] Explicit channel fallback can misroute outbound content:** `src/infra/outbound/message.ts:113`
  - body: The proposed sendMessage fallback applies even when the caller supplied an explicit channel. If that channel is unavailable or malformed, message content can be routed through the mirrored session channel instead of failing, which can expose content on an unintended channel or recipient path.
