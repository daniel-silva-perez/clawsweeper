## Summary

Keep open. Current main still writes `openclaw backup create` archives through an adjacent `<output>.UUID.tmp` file and cleans it in `finally` only when the JavaScript stack unwinds. I found no backup-owned SIGTERM/SIGINT/AbortSignal cleanup around the active tar write, so the reported external-timeout/process-termination leak remains distinct from the ordinary atomic-write cleanup already covered by #41830 and the broader large-backup policy work in #42273.

## Best Possible Solution

Keep this issue open and implement backup-owned graceful cleanup for timeout/termination: track the exact per-run temp archive path while `tar.c(...)` is active, register scoped SIGINT/SIGTERM and/or abort cleanup during that window, remove handlers in `finally`, preserve atomic publish/no-overwrite behavior, and add regression coverage for the signal/abort path. Leave broad large-backup exclusion/progress work to #42273 and live-file race handling to the existing backup PR threads.
