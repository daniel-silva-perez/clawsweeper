## Summary

Keep open. Current main still writes `openclaw backup create` output through an adjacent random `.tmp` archive and only removes it when the normal async stack reaches `finally`; the backup command path has no scoped SIGINT/SIGTERM or abort cleanup for externally timed-out runs. The issue discussion confirms a real disk-exhaustion impact and an external cleanup workaround, not a shipped OpenClaw fix.

## What This Changes

The issue reports that `openclaw backup create` can leave large adjacent `.tmp` archive files after timeout or process termination, causing disk space exhaustion.

## Best Possible Solution

Backup creation should own the active temporary archive lifecycle: track the exact temp archive path during the write window, clean it on graceful interruption or abort, preserve no-overwrite final publication, avoid deleting unrelated `*.tmp` files, and document unavoidable hard-kill limits.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the reported timeout scenario plus current-main source inspection: the backup writer creates an adjacent `.tmp` archive and relies on normal `finally` cleanup, while the CLI path has no backup-scoped signal or abort cleanup.

## Solution Assessment

Yes. A narrow backup-owned cleanup path for graceful termination is the best maintainable fix; broad startup sweeping or deleting arbitrary `*.tmp` files would risk removing unrelated user files and would not address the active archive lifecycle.
