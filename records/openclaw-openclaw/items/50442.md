## Summary

Keep open. Current main still writes `openclaw backup create` archives through an adjacent random `.tmp` file and only removes that file during normal JavaScript unwinding; the backup path has no scoped SIGINT/SIGTERM or abort cleanup for an external timeout that terminates the process. Existing coverage proves ordinary tar failures clean up, but not process interruption.

## What This Changes

The issue requests that `openclaw backup create` clean up large temporary backup archives when a backup is interrupted by timeout, exception, or process termination.

## Best Possible Solution

Add backup-owned interruption cleanup for the active archive write: track the exact temp archive path, install scoped graceful termination/abort handling only during backup creation, remove handlers after completion, preserve no-overwrite publication semantics, and cover the interrupted-write path with focused regression tests.

## Repair Work Prompt

Fix the backup temp-file leak described in #50442. Current `openclaw backup create` writes to an adjacent `<outputPath>.<uuid>.tmp` archive and removes it only from the normal `finally` path, so an external timeout/SIGTERM can strand a large temp archive. Add backup-owned graceful interruption cleanup around the active tar write: track the exact temp archive path, install scoped SIGINT/SIGTERM and abort cleanup only during backup creation, remove handlers in `finally`, and preserve the current no-overwrite publish behavior (`link` first, exclusive copy fallback). Add focused regression coverage for the interrupted-write path without broad refactors or unrelated backup size/exclusion work. Keep public wording as `plugin/plugins` if docs or changelog are touched. Add a changelog fix entry with the reporter credit if this is user-facing.
