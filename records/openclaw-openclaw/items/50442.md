## Summary

Keep open. Current main still creates `openclaw backup create` archives through an adjacent random `.tmp` file and only removes that file when the async backup path reaches its normal `finally` cleanup; I did not find a backup-scoped SIGINT, SIGTERM, timeout, or AbortSignal cleanup path for externally interrupted runs. The issue discussion also confirms the reporter is using an external cleanup workaround, not a fix in OpenClaw.

## What This Changes

The issue requests that `openclaw backup create` clean up large temporary archive files when a backup is interrupted or times out, while preserving atomic publication of successful archives.

## Best Possible Solution

Backup creation should own the active temporary archive lifecycle: track the exact temp archive path during the write window, clean it up on graceful interruption or abort, preserve existing no-overwrite final publication, avoid deleting unrelated `*.tmp` files, and document unavoidable hard-kill limits.

## Repair Work Prompt

Fix the backup temp-file leak reported in https://github.com/openclaw/openclaw/issues/50442. Current `openclaw backup create` writes the active archive to `<output>.<random>.tmp` and only removes it in normal `finally` cleanup; externally timed-out or gracefully signaled runs can leave large temp files behind. Implement scoped ownership of the active temp archive path during the archive-write window so SIGINT, SIGTERM, and explicit abort/timeout-style cancellation remove only that exact temp archive path before exit. Preserve existing no-overwrite final publication behavior, hard-link/copy fallback behavior, dry-run behavior, and current backup source selection. Do not add broad cleanup of unrelated `*.tmp` files, do not change backup restore scope, do not change dependency pins, and do not make core backup behavior plugin-specific. Add focused regression coverage for graceful interruption/abort cleanup and handler cleanup after success/failure. Update `docs/cli/backup.md` if the user-facing cleanup guarantee or unavoidable SIGKILL/crash limitation needs to be documented. Add a changelog fix entry if this repo requires one for user-facing CLI bug fixes, crediting `Thanks @hillerliao`.
