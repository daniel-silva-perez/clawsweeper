## Summary

Keep open. Current main still creates the backup archive in an adjacent random `.tmp` path and removes it only through normal JavaScript unwinding; the backup CLI does not install a scoped SIGINT/SIGTERM or abort cleanup path for externally timed-out runs, so the reported disk-exhaustion failure remains plausible and actionable.

## What This Changes

The issue requests that `openclaw backup create` clean up large temporary archive files left behind when a backup process times out or is interrupted.

## Best Possible Solution

Backup creation should own the active temporary archive lifecycle: track the exact temp archive path during the write window, clean it up on graceful interruption or abort, preserve existing no-overwrite publication semantics, avoid deleting unrelated `*.tmp` files, and document the unavoidable SIGKILL/crash limits.

## Repair Work Prompt

Fix the `openclaw backup create` temp archive leak reported in openclaw/openclaw#50442. Current main writes active archives as `<outputPath>.<uuid>.tmp` in `src/infra/backup-create.ts` and only removes them in `finally`, so default SIGTERM/SIGINT or an external timeout can terminate the process before cleanup runs. Implement a focused core backup fix that tracks the active temp archive path during the archive-write window and cleans it up on graceful interruption or abort. Preserve existing no-overwrite publication semantics and the hard-link/copy fallback. Do not delete arbitrary unrelated `*.tmp` files; only clean the active backup temp path unless the design has a very conservative stale-backup-temp policy. Restore/remove scoped signal handlers after completion and preserve normal CLI exit behavior. Document SIGKILL/host crash limits if user-facing behavior changes. Add regression coverage for signal/abort cleanup in the backup path, keep docs/changelog aligned, and add a user-facing changelog fix entry crediting `Thanks @hillerliao`. Do not change backup restore scope or broad large-workspace exclusion policy; those belong to separate backup issues.

## Risks / Open Questions

- Graceful in-process cleanup can cover SIGINT, SIGTERM, and explicit abort paths, but cannot reliably handle SIGKILL, host crashes, or power loss.
- A fix must preserve normal CLI exit semantics and remove any backup-scoped signal handlers after the backup completes.
- If the implementation relies on node-tar cancellation behavior, it should verify the pinned `tar` 7.5.13 contract before assuming an AbortSignal alone stops archive writes and closes file handles.
