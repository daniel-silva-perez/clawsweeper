## Summary

Keep this issue open. Current main has improved atomic archive publication and ordinary exception cleanup, which matches the already-closed related report #41830, but the specific timeout/process-termination failure mode here is still not fully handled or covered. `openclaw backup create` still writes a large adjacent `.tmp` archive before publishing, and I found no backup-specific SIGTERM/SIGINT/AbortSignal path that would remove that known temp file when an external timeout kills the process.

## Best Possible Solution

Keep this issue open and implement backup-owned graceful cleanup: track the per-run temp archive path, register scoped SIGINT/SIGTERM/abort cleanup while `tar.c` is active, keep the existing atomic publish behavior, and add regression coverage for normal tar failure plus a subprocess or injectable signal/abort path. Large-state exclusion/progress work can remain tracked separately under #42273.

## Risks / Open Questions

- A fix should only remove the exact temp archive path owned by the current backup run, not broad `*.tmp` globs in user-controlled directories.
- SIGKILL and host crashes cannot be cleaned in-process; the maintainable path is graceful SIGINT/SIGTERM cleanup plus a documented stale-temp cleanup strategy or doctor/fix check.
