## Summary

Keep open. Current main still writes `openclaw backup create` archives through an adjacent `<output>.<uuid>.tmp` file and only removes it in a `finally` block when the JavaScript stack unwinds; there is no backup-owned SIGINT/SIGTERM or abort cleanup for external timeout/process termination, so the reported disk-exhaustion failure mode remains valid.

## What This Changes

The issue requests that `openclaw backup create` clean up large `.tmp` archive files when the command times out or is interrupted, while preserving atomic publish/no-overwrite behavior.

## Best Possible Solution

Keep this open and implement backup-owned cleanup for graceful interruption: track the exact temp archive path while `tar.c(...)` is active, register scoped SIGINT/SIGTERM and/or abort cleanup during that window, remove handlers in `finally`, preserve atomic no-overwrite publication, and add regression tests for the interrupted write path. Leave broad large-backup progress/exclusion policy to #42273 and ordinary atomic-write coverage to the existing backup tests.
