## Summary

Keep this issue open. Current main includes DeepSeek V4 replay fixes and upgraded Pi packages, but the exact reported failure is a longer DeepSeek-origin multi-turn tool-call/subagent chain; current source and tests still prove short direct replay cases and blank fallback behavior, not a sessions_spawn-style chain preserving actual prior reasoning across several tool continuations.

## What This Changes

The issue asks OpenClaw to fix DeepSeek V4 thinking replay so long multi-turn tool-call and sessions_spawn flows retain required reasoning_content instead of failing with 400 and falling back to another model.

## Best Possible Solution

Add a focused DeepSeek V4 replay regression for a DeepSeek-origin chain with multiple assistant tool-call continuations, ideally including a sessions_spawn-shaped subagent handoff, then adjust the DeepSeek/OpenAI-compatible replay wrapper only if that test shows real reasoning_content is still dropped or replaced with an invalid blank. Keep the shipped short-replay behavior for foreign-provider history unless live DeepSeek proof shows it is unsafe.

## Reproduction Assessment

Yes. The issue gives a concrete DeepSeek V4 high-thinking sessions_spawn/multi-tool path and three independent user reports with the same 400 reasoning_content error, but this read-only review did not rerun a live DeepSeek long-chain session on current main.

## Solution Assessment

Unclear. The current fixes are appropriate for provider-switch and short assistant replay cases, but they are not yet proven to be the narrowest complete fix for long DeepSeek-origin tool-call chains; the safer path is to add a regression that captures the reported chain before changing replay semantics.
