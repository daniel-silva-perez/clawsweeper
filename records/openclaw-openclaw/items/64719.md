## Summary

Keep this PR open: current main still leaves external channel providers without a dangerous-tool deny list, so the contribution targets a real security gap. The proposed branch should not merge as-is because its default fallback is too broad for internal/trusted providers and memory-flush runs, and the `exec` owner-only hardening is not wired into the runtime owner-only policy.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, lets group allow policy override that deny set, adds `exec` to the owner-only core tool-name constant, and expands related unit tests.

## Best Possible Solution

Revise the policy so dangerous filesystem and exec tools are denied for untrusted external channel auto-reply contexts while preserving internal/trusted providers, channel memory-flush `read`/`write` behavior, and explicit operator policy semantics. If `exec` owner-only hardening stays in scope, enforce it through `src/agents/tool-policy.ts` or exec tool metadata with runtime authorization tests, and review it alongside #64718.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope the default deny list to untrusted providers:** `src/agents/pi-tools.message-provider-policy.ts:39-42`
  - body: The new default deny fallback applies to every non-empty `messageProvider`. Current main uses internal providers such as `webchat`, `heartbeat`, `cron-event`, and `exec-event`, so this strips exec/file tools from trusted owner/internal runs outside the external channel auto-reply threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush tools for channel sessions:** `src/agents/pi-tools.message-provider-policy.ts:39-42`
  - body: Memory-triggered runs are narrowed to `read` and `write` before message-provider filtering, and the memory runner carries the originating channel into the embedded run. Denying both tools for normal channel providers can leave channel memory flush with no usable tools.
  - confidence: 0.88
- **[P2] Wire exec owner-only through runtime policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to this constant does not make it owner-only at runtime. The runtime filter checks `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool still sets neither, so non-owner runs outside provider filtering can still receive `exec`.
  - confidence: 0.92
