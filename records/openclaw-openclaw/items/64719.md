## Summary

Keep open. Current main still does not deny dangerous tools for external channel message providers, so the intended security hardening is not implemented, but the PR branch has blocking policy and runtime-authorization issues that need maintainer/security review before merge.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider tool filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool-name constant, and expands related tests.

## Best Possible Solution

Revise the policy so dangerous filesystem and exec tools are denied for untrusted external channel auto-reply contexts while preserving internal/trusted providers, channel memory-flush `read`/`write` behavior, and documented operator allow/deny semantics. If `exec` owner-only hardening remains in scope, wire it through `src/agents/tool-policy.ts` or exec tool metadata with runtime authorization coverage, and review the result alongside #64718.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit default denies to untrusted external providers:** `src/agents/pi-tools.message-provider-policy.ts:41-43`
  - body: The new fallback applies to every non-empty `messageProvider`. Current main also uses provider values such as `webchat`, `heartbeat`, `cron-event`, and `exec-event` for trusted/internal runs, so this strips exec and file tools outside the channel auto-reply threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush read/write tools:** `src/agents/pi-tools.message-provider-policy.ts:41-43`
  - body: Memory flush runs are narrowed to `read` and `write` before this filter, and channel flushes carry the originating provider. The default deny then removes both tools for normal channel providers, leaving memory flush without a usable tool unless an unrelated group override happens to re-enable them.
  - confidence: 0.88
- **[P2] Enforce exec owner-only in the runtime policy:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to this constant does not make it owner-only at runtime. Owner filtering checks `tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs outside provider filtering can still receive `exec`.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The patch reduces a real channel auto-reply exposure, but it introduces unresolved access-control scope problems and does not actually enforce the new exec owner-only claim at runtime.

Concerns:

- **[medium] Default deny affects trusted/internal providers:** `src/agents/pi-tools.message-provider-policy.ts:41`
  - body: The PR applies the dangerous-tool deny baseline to every non-empty provider rather than only untrusted external channels, which can remove privileged tools from internal owner-controlled flows such as WebChat and heartbeat/event runs.
  - confidence: 0.9
- **[medium] Channel memory flush can lose read/write:** `src/agents/pi-tools.message-provider-policy.ts:41`
  - body: Channel memory flush relies on `read` and `write` after prior narrowing, but the new default deny removes both for channel providers, which can prevent memory persistence rather than only blocking unsafe user-requested file access.
  - confidence: 0.86
- **[medium] Exec owner-only change is not runtime-enforced:** `src/agents/tools/owner-only-tools.ts:1`
  - body: The changed constant is not the runtime authorization source for `exec`; current runtime filtering requires a tool-policy fallback or `ownerOnly` metadata, neither of which this PR adds for the exec tool.
  - confidence: 0.92
