## Summary

Keep open. Current main still has the reported channel auto-reply tool-policy gap, and this PR is relevant security hardening, but the latest patch still needs maintainer/security review because it appears to over-broaden the deny fallback and leaves the exec owner-only defense-in-depth path only partially wired.

## What This Changes

The PR adds a default dangerous-tool deny list for message-provider tool filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool-name constant, and expands related unit tests.

## Best Possible Solution

Keep this PR open and revise the policy so dangerous filesystem/exec tools are denied for untrusted external channel auto-reply contexts while internal/trusted providers, channel memory-flush read/write behavior, and explicit operator policy semantics are preserved. If exec owner-only hardening remains in scope, wire it through `src/agents/tool-policy.ts` or exec tool metadata and cover it with runtime authorization tests, then review it alongside #64718.
