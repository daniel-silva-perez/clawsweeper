## Summary

Keep open. Current main still does not implement or document any macOS system `crontab` handling for exec, and the exec failure path remains generic timeout/signal handling. The OpenClaw cron CLI surface still goes through Gateway RPC with scoped operator authorization, so recent pairing/device-token improvements and the related generic SIGKILL diagnostics issue (#72240) do not fully resolve this issue's crontab and cron-administration reports.

## Best Possible Solution

Keep this issue open. Reproduce `crontab -l`, `crontab -e`, and stdin `crontab -` through OpenClaw exec on a current macOS LaunchAgent install, then fix or document the exec environment, stdin/TTY, timeout, and diagnostic behavior. Separately make an explicit product/security decision for same-host cron administration: either keep admin-scoped Gateway cron mutations with clearer `openclaw devices list/approve` guidance, add least-privilege/read-only cron paths where appropriate, or design a narrow same-host token/device reuse path that preserves the Gateway trust boundary.
