## Summary

Current main still does not resolve the reported macOS system `crontab` exec failure or the separate non-local cron-administration friction. The source tree has no `crontab`-specific handling or docs, exec timeout/signal formatting remains generic, and `openclaw cron add` still goes through Gateway RPC/admin-scoped authorization. Recent device pairing improvements help `openclaw devices list/approve` on local loopback, but they do not prove raw `/usr/bin/crontab` works through exec or provide a cron-specific same-host bypass for chat-channel agents.

## Best Possible Solution

Keep this issue open. Reproduce `crontab -l`, `crontab -e`, and stdin `crontab -` through OpenClaw exec on a current macOS LaunchAgent install, then fix or document the exec environment, stdin/TTY, timeout, and diagnostic behavior. Separately make an explicit product/security decision for same-host cron administration: keep admin-scoped Gateway cron mutations with clearer `openclaw devices list/approve` guidance, add least-privilege/read-only cron paths where appropriate, or design a narrow same-host token/device reuse path that preserves the Gateway trust boundary.
