## Summary

Keep open. Current main has adjacent fixes for stale transcript pointers and compaction transcript rotation, but the silent auto-reply reset path can still persist a successor `sessionFile` before that transcript is durable; if replay cannot use the previous `sessionFile`, history lookup can still return an empty result while an older valid transcript remains elsewhere.

## What This Changes

Prevent context-overflow session reset from mapping a session key to a nonexistent transcript and preserve or recover the prior transcript continuity.

## Best Possible Solution

Enforce a generic reset/rotation invariant for auto-reply embedded/direct recovery: resolve the prior transcript through safe candidates, create or prewarm the successor JSONL before switching store and queued follow-up state, seed continuity when available, and keep or explicitly recover the prior valid mapping if successor creation fails.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with an active previous entry whose `sessionFile` is missing or has no replayable tail, then assert the persisted successor `sessionFile` exists and `readSessionMessages` does not return empty solely because the new file was never created.

## Solution Assessment

No current-main solution is sufficient. Candidate ordering and cleanup help adjacent stale or missing-entry cases, but the durable fix belongs at the reset/rotation write boundary rather than in later cleanup.
