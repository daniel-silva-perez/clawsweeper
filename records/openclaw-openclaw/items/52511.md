## Summary

Keep this issue open. Current main has useful reconnect and device-token fallback code, but I did not find a shipped fix for the reported Control UI/browser-control permission loss after gateway restart or token rotation. The linked PR #58022 was closed unmerged specifically in favor of keeping this issue as the canonical tracker.

## Best Possible Solution

Keep the issue as the canonical tracker. The best fix is a targeted Control UI/Gateway auth recovery change with regression coverage: reproduce the gateway-restart/token-rotation path, distinguish stale device-token or scope-upgrade failures from generic code-1000 closes, surface an actionable pairing/scope message when browser.request lacks operator.admin, and refresh or clear the cached device token only on the relevant auth signal.
