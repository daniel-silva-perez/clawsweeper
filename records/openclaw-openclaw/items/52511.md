## Summary

Keep this issue open. Current main still has generic reconnect and bounded device-token retry behavior, but there is no targeted current-main or shipped proof that the reported Control UI browser-control permission loss after gateway restart/token rotation is fixed. The related PR #58022 was closed unmerged in favor of this issue as the canonical tracker.

## Best Possible Solution

Keep this issue as the canonical tracker. The best fix is a targeted Control UI/Gateway auth recovery change with regression coverage for the gateway-restart/token-rotation path: distinguish stale device-token or scope-upgrade failures from generic code-1000 closes, surface actionable pairing/admin-scope guidance for browser.request, and refresh or clear cached device auth only on the relevant auth signal.
