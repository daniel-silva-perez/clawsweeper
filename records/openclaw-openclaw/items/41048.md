## Summary

Keep this PR open, but it is not ready to merge. The underlying Telegram null-media bug remains real on current main, yet the branch is stale against the Telegram plugin move and the proposed fix misses media groups while adding an unthrottled automatic reply path.

## What This Changes

The PR adds a Telegram warning reply when media resolution returns null and suppresses inbound media placeholders when no attachment was resolved.

## Best Possible Solution

Update or replace the change under the Telegram plugin so null media-resolution failures produce an accurate, rate-limited user-visible failure signal while preserving useful caption and file_id metadata for the agent, including media-group cases.

## Reproduction Assessment

Yes. A focused test can make ctx.getFile() return the Telegram file-too-big failure or exhaust retries so resolveMedia() returns null; current main then reaches allMedia=[] without a null-path warning, and the body helper behavior is covered by the captioned-video test.

## Solution Assessment

No. The single-message intent is useful, but the proposed implementation is stale and incomplete because it misses media groups, suppresses metadata now preserved by current main, and sends unthrottled replies with over-specific failure wording.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Handle all-failed media groups before dropping markers:** `src/telegram/bot-message-context.body.ts:141`
  - body: Conditioning the placeholder on resolved attachments also affects media-group processing, but media-group updates return before the new warning branch. If every album item resolves to null, allMedia stays empty and a media-only album can be dropped with no user-visible failure signal; add album-level warning or fallback handling before suppressing the marker.
  - confidence: 0.88
- **[P2] Throttle repeated media-failure replies:** `src/telegram/bot-handlers.ts:1031`
  - body: This branch sends a Telegram warning for every null media resolution in an allowed chat. A sender can repeatedly submit media that fails getFile() and make the bot spam the chat or consume Bot API send capacity; add per-chat, per-sender, or media-group cooldown before sending the automatic warning.
  - confidence: 0.78
- **[P3] Use reason-neutral fallback warning text:** `src/telegram/bot-handlers.ts:1031`
  - body: resolveTelegramFileWithRetry() returns null for both Bot API size-limit failures and exhausted transient retries, but this message points mainly at the 20 MB limit. Use wording that also covers temporary download failures unless the failure reason is carried through explicitly.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it adds an unthrottled automatic Telegram reply path for attacker-controlled media failures.

Concerns:

- **[medium] Unthrottled media-failure warnings can spam chats:** `src/telegram/bot-handlers.ts:1031`
  - body: Any permitted sender can repeatedly trigger null media resolution and cause the bot to send one warning reply per message, which can spam group chats or consume Telegram send capacity. Add cooldown, dedupe, or scope limits before merge.
