## Summary

Keep this PR open for maintainer follow-up. The Telegram null-media bug still exists on current main, but the PR branch targets the old Telegram path and the proposed patch has blocking media-group and warning-spam issues before it should merge.

## What This Changes

The PR adds a Telegram warning when media resolution returns null and suppresses media placeholders when no attachment was resolved.

## Best Possible Solution

Update or replace the PR against `extensions/telegram` so null media-resolution failures produce an accurate, rate-limited user-visible signal across single messages and media groups while preserving useful caption/file metadata without implying an attachment was downloaded.

## Reproduction Assessment

Yes. A focused test can make `ctx.getFile()` throw Telegram's file-too-big error or exhaust retries, causing `resolveMedia()` to return null; current main then enqueues `allMedia=[]` and the body helper still emits placeholder context without a null-path warning.

## Solution Assessment

No. The single-message intent is good, but the current patch is not the narrowest maintainable fix because it is stale, drops the media-group failure signal, and sends unthrottled warnings with failure wording that does not distinguish size-limit and transient failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle all-failed media groups before dropping placeholders:** `src/telegram/bot-message-context.body.ts:141`
  - body: Conditioning the placeholder on `allMedia.length > 0` makes media-only albums with all null media resolutions return no body. Media-group updates return before the new warning branch, so the user and agent get no failure signal for an all-failed album; add media-group warning or fallback handling before suppressing the marker.
  - confidence: 0.88
- **[P2] Throttle repeated media-failure replies:** `src/telegram/bot-handlers.ts:1031`
  - body: This branch sends a Telegram warning for every null media resolution in an allowed chat. A sender can repeatedly submit media that fails `getFile()` and make the bot spam the chat or burn send capacity; add per-chat, per-sender, or media-group cooldown before sending the automatic warning.
  - confidence: 0.76
- **[P3] Make the fallback warning reason-neutral:** `src/telegram/bot-handlers.ts:1031`
  - body: `resolveTelegramFileWithRetry()` returns null for both Bot API size-limit failures and exhausted transient retries, but this message points mainly at the 20 MB limit. Use wording that also covers temporary download failures unless the failure reason is carried through explicitly.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it introduces an unthrottled automatic Telegram reply path for attacker-controlled media failures.

Concerns:

- **[medium] Unthrottled media-failure warnings can spam chats:** `src/telegram/bot-handlers.ts:1031`
  - body: Any permitted sender can trigger null media resolution repeatedly and cause the bot to send a warning reply for each message, which can spam group chats or consume Telegram send capacity. Add cooldown, dedupe, or scope limits before merge.
  - confidence: 0.76
