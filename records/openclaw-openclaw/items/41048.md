## Summary

Keep this PR open for maintainer handling, but do not merge it as-is. Current main still has a real Telegram null-media gap, yet this branch changes the old `src/telegram` path, conflicts with the later unavailable-media metadata behavior, and adds an unthrottled user-triggered warning reply.

## What This Changes

The PR adds a Telegram warning reply when media resolution returns null and suppresses inbound media placeholders when no attachment was resolved.

## Best Possible Solution

Rework the fix in `extensions/telegram` around an explicit unavailable-media state: keep useful placeholder and `file_id` context for the agent, add throttled user-visible warnings where maintainers want them, cover single media plus albums, and preserve the shipped caption behavior from #59948.

## Reproduction Assessment

Yes. The code path is reproducible by making Telegram `ctx.getFile()` throw a file-too-big or exhausted-retry error: `resolveMedia()` returns null, the live handler builds `allMedia = []`, and current single-message handling has no null-return warning branch.

## Solution Assessment

No. The current PR is not the best fix because it changes a retired path and suppresses metadata that current main deliberately preserves; a safer solution should port to `extensions/telegram`, distinguish unavailable media from attached media, and throttle any warning replies.
