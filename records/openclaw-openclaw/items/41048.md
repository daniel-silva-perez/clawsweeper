## Summary

Keep this PR open. Current main still has the Telegram null-media behavior reported by #40991, and this PR is a plausible source fix, but it targets the pre-extension `src/telegram` layout and introduces remaining blocking concerns around all-failed media groups and unbounded warning replies.

## What This Changes

The PR adds a Telegram warning when `resolveMedia()` returns `null` for an inbound media message and suppresses media placeholders when no media attachment was resolved.

## Best Possible Solution

Rework the fix on current main under `extensions/telegram`: make the media-resolution failure state distinguishable, avoid misleading attachment placeholders when no file exists, send an accurate bounded user warning for single messages and all-failed media groups, and add focused regression tests for null `getFile()` video/photo paths plus successful normal attachment behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve a failure signal for all-failed albums:** `src/telegram/bot-message-context.body.ts:141`
  - body: Changing `placeholder` to depend on `allMedia.length > 0` makes media-only messages with no resolved attachments return `null`. Media-group updates return before the new single-message warning branch, so an album where every `getFile()` call returns `null` is silently dropped instead of notifying the user or the agent.
  - confidence: 0.91
- **[P2] Throttle failed-media warning replies:** `src/telegram/bot-handlers.ts:1019-1036`
  - body: This branch sends a Telegram reply for every media message whose resolution returns `null`. Any authorized sender can repeatedly trigger `getFile()` failures, causing chat spam and extra Bot API calls; add per-chat/per-sender or media-group suppression before emitting the warning.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or workflow changes, but it introduces an attacker-triggerable Telegram warning reply without suppression.

Concerns:

- **[medium] Failed-media warning lacks abuse suppression:** `src/telegram/bot-handlers.ts:1019`
  - body: The new `sendMessage` path runs for every null media-resolution result. In an allowed chat, repeated failing media can spam the conversation and consume Telegram Bot API send capacity unless the warning is rate-limited, deduplicated, or restricted.
