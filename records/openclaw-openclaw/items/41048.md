## Summary

Keep this PR open. Current main still has the reported Telegram null-media path: failed `getFile()` calls return `null`, the single-message handler only warns for thrown media errors, and the inbound body builder still emits media placeholders when no attachment was resolved. The PR remains a plausible focused fix, but it is stale against the current `extensions/telegram` layout and should address the review concerns before landing.

## Best Possible Solution

Keep this PR open as a valid implementation candidate, but rework it on current main under `extensions/telegram`. The best landing path is to add a distinguishable failed-media result, suppress or clearly mark unavailable-media placeholders when `allMedia` is empty, surface bounded user warnings for single messages and all-failed media groups, and add focused regression tests for failed video/photo `getFile` paths plus normal successful attachment behavior. The linked #40991 should remain open until this PR or an equivalent fix lands.

## Risks / Open Questions

- The PR is stale against current main: Telegram code moved from `src/telegram` into `extensions/telegram`, and `bot-handlers` is now split across runtime/helper modules.
- The proposed warning path can be spammed by repeated failed-media messages unless the landed fix adds per-chat, per-sender, or per-media-group suppression.
- The proposed warning text overemphasizes Telegram's 20 MB Bot API limit even though the same null path also covers exhausted transient `getFile` retries.
- The proposed single-message fix does not cover media groups where every attachment fails to resolve.
