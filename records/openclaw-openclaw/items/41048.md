## Summary

Keep this PR open. Current main still has the reported Telegram null-media path: `getFile()` failures return `null`, the single-message handler only warns for thrown download errors, and the inbound body builder still emits placeholder-only media text when no attachment was resolved. The PR remains a plausible implementation candidate for open issue #40991, but it is stale against the current `extensions/telegram` layout and should address the review/security concerns before landing.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but rework it on current main under `extensions/telegram`. The best landing path is to return or carry a distinguishable media-resolution failure reason, suppress or clearly mark unavailable-media placeholders when no attachment exists, send a bounded and accurate user warning for single messages and all-failed media groups, and add focused regression tests for null `getFile` video/photo paths plus successful normal attachment behavior.

## Risks / Open Questions

- The PR's automatic user warning could spam chats or consume Telegram send capacity if repeated failing media messages are not rate-limited per chat, sender, or media group.
- The proposed warning text emphasizes Telegram's 20 MB Bot API limit even though the same `null` path also covers transient retry exhaustion.
- Suppressing placeholders without a separate media-group failure signal can make all-failed media-only albums disappear silently.
- The PR needs rebasing from the old `src/telegram` paths into the current `extensions/telegram` plugin layout.
