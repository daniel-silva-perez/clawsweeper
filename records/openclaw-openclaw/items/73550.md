## Summary

Keep open. The reporter's follow-up about `deliveryContext` matches current source: native persistent sub-agent sessions can retain requester delivery context, while `sessions_send` still only suppresses the A2A/announce path for parent-owned ACP sessions. Related #53145 is useful coordination context for inter-session announce echoing, but it remains open and broader than this native persistent sub-agent duplicate-delivery path.

## What This Changes

The issue requests that persistent native sub-agent replies sent via `sessions_send` appear only once in the parent conversation instead of also being re-injected or announced through inherited delivery context.

## Best Possible Solution

Make the `sessions_send` delivery contract explicit for parent-owned native persistent sub-agent sessions: a parent send should have exactly one parent-visible result path, while peer A2A sends and unrelated channel-bound announces continue to work. Add focused regression coverage for the native session case and coordinate any shared filtering with #53145.

## Reproduction Assessment

Yes. The issue gives concrete reproduction steps, and current source shows a high-confidence code-level reproduction: waited `sessions_send` can return the child reply inline and still start the A2A/announce flow that can target the child session's stored delivery context.

## Solution Assessment

No. Current `main` only skips A2A for parent-owned background ACP sessions, so it does not solve the native persistent sub-agent path described here; the narrower maintainable fix is to detect the parent-owned native child send and choose one delivery path without changing peer A2A.
