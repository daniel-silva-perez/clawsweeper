## Summary

Keep open. Current main still has a plausible duplicate-delivery path for native persistent sub-agent `sessions_send`: the waited send can return the target reply inline while the A2A reply/announce flow remains scheduled, and the announce target can be hydrated from the child session's stored delivery context. Related #53145 overlaps on inter-session echo/control-message leakage, but it is broader and still open, so it is coordination context rather than a terminal close reason.

## What This Changes

The issue requests that `sessions_send` deliver persistent sub-agent replies to the parent exactly once and avoid `deliveryContext`/announce feedback loops.

## Best Possible Solution

Fix the sessions-send delivery contract so parent-owned persistent sub-agent replies have exactly one parent-visible path. The implementation should suppress same-run duplicate transcript/UI injection, avoid announcing back through the child session's inherited `deliveryContext`, preserve unrelated peer A2A messaging, and coordinate shared control-token filtering with #53145.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists from the issue steps plus current source: a waited `sessions_send` can return the target reply inline and still schedule A2A reply/announce work that uses the target session delivery context. I did not run a live UI reproduction in this read-only pass.

## Solution Assessment

No. Current main only skips A2A for parent-owned background ACP sessions; native persistent sub-agent sessions still follow the pending announce path. The narrow maintainable solution is to detect the parent-owned persistent sub-agent send and choose one delivery path, with regression coverage for the deliveryContext loop.

## Repair Work Prompt

Fix `sessions_send` duplicate delivery for parent-owned persistent native sub-agent sessions. The observable bug is that a parent sending to a persistent sub-agent can see the same reply more than once: inline `reply`, A2A reply injection, and/or announce delivery through the child session's inherited `deliveryContext`. Keep the change narrow to the sessions_send/sub-agent delivery contract. Detect parent-owned persistent sub-agent targets separately from the existing parent-owned ACP skip, choose exactly one parent-visible delivery path, and prevent the child session's deliveryContext from causing an announce loop back to the parent or child. Preserve ordinary peer agent-to-agent messaging and ACP `streamTo="parent"` behavior. Coordinate suppression of `NO_REPLY`, `ANNOUNCE_SKIP`, `REPLY_SKIP`, and placeholder/control text with #53145. Add regression tests for waited and fire-and-forget `sessions_send` against a parent-owned persistent sub-agent session with stored deliveryContext. Do not change unrelated channel routing, provider behavior, release metadata, or broad session visibility policy. Changelog only if the resulting behavior is user-facing enough under the active version's Fixes section with proper reporter credit.
