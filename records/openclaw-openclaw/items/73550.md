## Summary

Keep open. Current main has an ACP-only guard for parent-owned one-shot ACP children, but the reported native persistent sub-agent path still enters the normal sessions_send A2A ping-pong and announce flow. The reporter’s deliveryContext workaround is plausible because announce-target lookup still hydrates channel/thread routing from session deliveryContext and tests assert that behavior.

## Best Possible Solution

Keep this open and fix the native persistent sub-agent sessions_send path. The likely product direction is to choose one parent-visible delivery path for parent-owned sub-agent replies, skip or dedupe A2A announce injection for runtime="subagent" mode="session" when the requester owns the child, preserve legitimate thread-bound deliveryContext for channel routing, and add focused regression coverage for sessions_spawn thread/session plus sessions_send returning exactly one parent-visible reply. Coordinate shared NO_REPLY/ANNOUNCE_SKIP/REPLY_SKIP cleanup with #53145, but do not close this report as fully covered by it yet.

## Risks / Open Questions

- A broad A2A disable could break legitimate peer-agent sessions_send conversations; the fix should distinguish parent-owned native persistent sub-agents from general peer messaging.
- deliveryContext is also used for legitimate thread-bound session delivery, so clearing it globally would likely regress channel/thread routing.
- A UI-only duplicate filter would be incomplete because the extra inter-session turns are created in the agent/session delivery path before webchat rendering.
