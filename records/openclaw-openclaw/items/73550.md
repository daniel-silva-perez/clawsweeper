## Summary

Keep open. Current main still has a plausible duplicate-delivery path for the reported native persistent sub-agent `sessions_send` flow: the A2A skip is ACP-only, waited sends still return the child reply inline and then start A2A announce work, and announce targeting can hydrate from the target session's stored `deliveryContext`. Related #53145 is useful context, but it is broader and still open, so it is not a terminal duplicate close.

## What This Changes

The issue asks for `sessions_send` replies from persistent native sub-agent sessions to appear once in the parent conversation instead of being duplicated through overlapping reply, announce, and `deliveryContext` paths.

## Best Possible Solution

Fix the native persistent sub-agent/session-send contract so a parent-owned native child has exactly one parent-visible result path. The change should suppress duplicate same-run transcript/UI injection and avoid announcing through the child's inherited `deliveryContext` for parent-owned sends, while preserving unrelated peer A2A messaging and the existing ACP parent-owned skip semantics.

## Reproduction Assessment

Yes. The issue provides concrete steps, and current source gives a high-confidence code-level reproduction path: a waited native `sessions_send` can return the target reply inline and still schedule A2A reply/announce work that resolves delivery through the target session's stored `deliveryContext`.

## Solution Assessment

No. Current main only skips A2A for parent-owned background ACP sessions, while the reported flow is native persistent sub-agent session messaging. The narrower maintainable fix is to detect parent-owned native persistent sub-agent sends and choose one delivery path without disabling peer A2A globally.
