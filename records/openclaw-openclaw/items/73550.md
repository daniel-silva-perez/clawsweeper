## Summary

Keep open. Current main still leaves the reported native parent/sub-agent `sessions_send` duplication path plausible: the new skip only covers parent-owned ACP sessions, while non-ACP persistent sub-agent sessions can still return a waited reply inline and then run the A2A ping-pong plus announce delivery through stored `deliveryContext`. The related #53145 is useful coordination context, but it is broader and still open, so it does not supersede this narrower native sub-agent report.

## What This Changes

The issue asks for persistent sub-agent replies sent via `sessions_send` to appear once in the parent session instead of being duplicated through inline, A2A, and announce delivery paths.

## Best Possible Solution

Add a narrow sessions-send delivery contract for parent-owned persistent native sub-agent sessions so the parent sees exactly one reply path. The fix should suppress duplicate A2A/announce transcript or channel injection for parent-owned child sends, avoid reusing the child's inherited `deliveryContext` as a second parent-visible route, keep unrelated peer A2A behavior intact, and coordinate shared placeholder/NO_REPLY filtering with #53145.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction path exists on current main: create a parent-owned persistent native sub-agent session, call `sessions_send` with a waited reply, and the code can return that reply inline while still scheduling the A2A reply/announce path through the child session's stored `deliveryContext`.

## Solution Assessment

No. Current main's ACP-only skip is not broad enough for the reported native persistent sub-agent path; the safer solution is a narrowly scoped parent-owned native sub-agent suppression path with regression coverage, not a broad removal of A2A announce behavior.
