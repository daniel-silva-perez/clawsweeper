## Summary

Keep open. Current main has an ACP-only guard for parent-owned child sessions, but native persistent sub-agent sessions still preserve `deliveryContext` and `sessions_send` still starts the generic A2A ping-pong/announce path after returning an inline waited reply. Related #53145 overlaps on inter-session echo/control-token leakage, but it is broader and still open, so it is coordination context rather than a close reason.

## What This Changes

The issue asks for `sessions_send` replies from parent-owned persistent sub-agent sessions to appear once in the parent flow without duplicate announce or feedback-loop delivery.

## Best Possible Solution

Teach `sessions_send` to distinguish parent-owned child-session follow-ups from peer A2A messaging across the native and ACP paths, then ensure parent-owned child replies have exactly one parent-visible delivery path. Native sub-agent sessions may still keep channel/thread metadata for real thread binding, but inherited `deliveryContext` must not cause a second announce back into the parent/child loop.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: send from a requester session to its own native sub-agent session with `spawnedBy` and `deliveryContext`; current main can return the child reply inline and still schedule A2A reply/announce work that uses the child session’s delivery context.

## Solution Assessment

No, current main is not the best solution for the reported behavior. The narrow maintainable fix is to suppress the A2A/announce path only for parent-owned child sends, while preserving ordinary peer `sessions_send` behavior and explicit channel thread delivery.
