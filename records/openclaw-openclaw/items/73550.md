## Summary

Keep open. Current main still has a credible duplicate-delivery path for the reported native persistent sub-agent flow: `sessions_send` only skips A2A for parent-owned ACP sessions, while native sub-agent sessions can keep `deliveryContext` and still run the reply/announce path after returning a waited reply inline. Related #53145 overlaps on inter-session placeholder/control-token leakage, but it is broader and open, so it is coordination context rather than a close reason.

## What This Changes

The issue requests that `sessions_send` replies from persistent sub-agent sessions appear once in the parent flow and avoid announce/deliveryContext feedback loops.

## Best Possible Solution

Add a narrow parent-owned sub-agent delivery contract for `sessions_send`: a parent sending to its own persistent native child should have exactly one parent-visible reply path, should not announce back through the child session's inherited `deliveryContext`, and should leave generic peer A2A behavior intact. Add regressions for waited and fire-and-forget sends, including `NO_REPLY`, `REPLY_SKIP`, and `ANNOUNCE_SKIP` handling.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current main shows a code-level reproduction path: waited `sessions_send` can return the target reply inline while still scheduling A2A/announce work that resolves announce delivery from the target session's stored `deliveryContext`.

## Solution Assessment

No, current main is not the best solution for this report. The existing ACP-only skip is too narrow for native persistent sub-agents, and making native `streamTo` meaningful would be a product/API change; the safer fix is to suppress duplicate parent-owned delivery at the `sessions_send`/announce contract boundary.
