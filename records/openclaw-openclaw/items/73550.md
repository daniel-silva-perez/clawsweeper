## Summary

Keep open. Current main still has a plausible duplicate-delivery path for non-ACP parent/sub-agent `sessions_send`: the first target reply can be returned inline while the A2A ping-pong and announce flow remains pending, and the available tests still encode multiple inter-session agent steps plus a separate announce delivery. Related #53145 overlaps on placeholder/control-message leakage, but it is broader and open, so it is coordination context rather than a close reason.

## What This Changes

The issue requests that `sessions_send` replies from persistent sub-agent sessions be delivered to the parent conversation only once, without `deliveryContext` announce feedback loops.

## Best Possible Solution

Fix the sessions-send delivery contract so parent-owned persistent sub-agent replies have exactly one parent-visible path. The implementation should suppress same-run duplicate inter-session transcript/UI injection, avoid announcing back through the child session's inherited `deliveryContext`, preserve unrelated peer A2A messaging, and coordinate shared control-token filtering with #53145.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists in current main: a waited `sessions_send` can return the target reply inline and still schedule the A2A reply/announce path, with tests asserting multiple inter-session steps and a separate announce delivery.

## Solution Assessment

No, current main is not the best solution for the reported behavior. The narrow maintainable fix is to identify parent-owned persistent sub-agent sends and choose one delivery path, with explicit suppression for `NO_REPLY`, `ANNOUNCE_SKIP`, `REPLY_SKIP`, and same-run duplicate replies before transcript or channel delivery.
