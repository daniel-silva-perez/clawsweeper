## Summary

Keep open. Current `main` still has a plausible duplicate-delivery path for native persistent sub-agent `sessions_send`: waited sends can return the child reply inline while also scheduling the A2A reply/announce flow, and announce targeting can hydrate from the child session's stored `deliveryContext`. Related #53145 is useful coordination context, but it is broader and still open rather than a terminal duplicate close.

## What This Changes

The issue asks OpenClaw to stop delivering persistent native sub-agent replies multiple times when a parent session uses `sessions_send`.

## Best Possible Solution

Fix the sessions-send delivery contract so parent-owned persistent native sub-agent replies have exactly one parent-visible path. The implementation should suppress duplicate same-run transcript/UI injection, avoid announcing through the child session's inherited `deliveryContext` for parent-owned sends, preserve unrelated peer A2A messaging, and coordinate shared control-token filtering with #53145.

## Reproduction Assessment

Yes. The issue gives concrete reproduction steps, and current source shows a high-confidence code-level reproduction: a waited `sessions_send` can return the target reply inline and still schedule A2A reply/announce work that uses the target session delivery context. I did not run a live UI reproduction in this read-only pass.

## Solution Assessment

No. Current `main` only skips A2A for parent-owned background ACP sessions, so it does not solve the native persistent sub-agent path described here. The narrower maintainable fix is to detect parent-owned native persistent sub-agent sends and choose one delivery path while leaving peer A2A behavior intact.
