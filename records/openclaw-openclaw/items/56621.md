## Summary

Keep this issue open. The prior ClawSweeper keep-open assessment still matches current main: `openclaw pairing` exposes pending-request `list` and `approve` only, docs describe only those commands, and `channels status --probe` still renders `allow:` from the account snapshot rather than a merged config plus pairing-store allowlist. The adjacent chat `/allowlist` command and store helpers are useful implementation building blocks, but they are not the requested shell CLI sender-management surface.

## Best Possible Solution

Implement a scoped core CLI change: add `openclaw pairing list-approved <channel> [--account] [--json]` that merges config `allowFrom` with `readChannelAllowFromStore`, add `openclaw pairing revoke <channel> <sender> [--account]` using `removeChannelAllowFromStoreEntry`, update `channels status --probe` to show an effective merged allowlist or paired-store summary, and cover default-account/account-scoped cases in CLI, docs, and tests.

## Risks / Open Questions

- This is access-control state; an implementation should preserve account scoping and the default-account legacy allowFrom store behavior.
- `revoke` should remove pairing-store approvals only, or clearly report when an entry is config-owned and must be changed through config editing.
- Status output should avoid leaking more sender detail than existing CLI conventions allow, especially outside `--json`.
