## Summary

Keep this issue open. Current main still lacks the requested CLI sender-management surface: `openclaw pairing` only registers pending `list` and `approve`, while docs describe only those commands. The underlying pairing allowFrom store already has read/remove primitives, and the chat `/allowlist` command can touch the paired store, but there is no `pairing list-approved`, no `pairing revoke`, and `channels status --probe` still renders `allow:` from account snapshot `allowFrom` rather than a merged config + pairing-store view. The related notification request is separate; current `approve --notify` support does not solve this issue's list/revoke/status-management request.

## Best Possible Solution

Keep the issue open and implement a scoped pairing-management CLI in core: add `openclaw pairing list-approved <channel> [--account] [--json]` that merges config `allowFrom` with `readChannelAllowFromStore`, add `openclaw pairing revoke <channel> <sender> [--account]` using `removeChannelAllowFromStoreEntry`, update `channels status --probe` to show an effective merged allowlist or a clear paired-store summary, and cover the account-scoped/default-account cases in CLI, store, docs, and status tests.

## Risks / Open Questions

- This is access-control state, so an implementation should preserve account scoping, legacy default-account store behavior, and clear source labels for config vs paired-store entries.
- `revoke` should remove only pairing-store approvals, not silently rewrite config `allowFrom`; config-owned entries need a separate explicit config edit path or a clear error.
- Status output should avoid leaking more sender detail than existing CLI/status conventions allow, especially in non-JSON summaries.
