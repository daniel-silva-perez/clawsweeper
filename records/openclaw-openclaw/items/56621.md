## Summary

Keep this issue open. Current main still exposes `openclaw pairing list` for pending requests and `openclaw pairing approve` only; there is no shell CLI for listing approved paired senders or revoking a sender. `channels status --probe` still formats `allow:` from the account snapshot/config path rather than a merged config plus pairing-store allowlist.

## Best Possible Solution

Implement a scoped core CLI change: add `openclaw pairing list-approved <channel> [--account] [--json]` that merges config `allowFrom` with `readChannelAllowFromStore`, add `openclaw pairing revoke <channel> <sender> [--account]` using `removeChannelAllowFromStoreEntry`, update `channels status --probe` to show an effective merged allowlist or paired-store summary, and cover default-account/account-scoped cases in CLI tests and docs.

## Risks / Open Questions

- This is access-control state, so any implementation should preserve account scoping and the default-account legacy allowFrom store behavior.
- `revoke` should remove pairing-store approvals only, or clearly report when an entry is config-owned and must be changed through config.
- Status output should avoid leaking more sender detail than existing CLI conventions allow, especially outside `--json`.
