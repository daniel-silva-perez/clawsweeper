## Summary

Keep open. Current main still separates successful phase-report writes from best-effort detached Dream Diary prose, and the reported recovery symptoms map to current marker-normalization and backfill date-label behavior. The overlapping open PR #70332 and related open issue #72546 mean this should be scoped by a maintainer before a separate automated fix is queued.

## What This Changes

The issue reports that Dream Diary entries stop advancing even though dreaming phase reports continue, and that `memory rem-backfill`/rollback can leave `DREAMS.md` with inconsistent headings or managed blocks.

## Best Possible Solution

Keep this open and reconcile it with #72546 and PR #70332. The desired end state is an idempotent legacy-aware `DREAMS.md` marker migration/backfill/rollback path, plus clearer cron narrative failure reporting so successful phase reports do not make a detached diary failure look healthy.

## Reproduction Assessment

Partly yes. The rollback/header and date-only backfill symptoms have a high-confidence static reproduction from current `ensureDiarySection()`, `removeBackfillDiaryEntries()`, and `formatBackfillDiaryDate()` behavior; the multi-day stale diary symptom still needs live cron/subagent evidence to identify the exact failure cause.

## Solution Assessment

No proposed patch is attached, and current main is not sufficient. The narrow maintainable path should start with legacy-aware diary marker handling and focused rollback/backfill tests, then decide whether detached cron narrative failures need status surfacing or retry semantics.
