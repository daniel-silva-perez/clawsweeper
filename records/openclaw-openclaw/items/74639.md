## Summary

Keep this open. Current main still has a deterministic date-only grounded backfill heading path, and the scheduled dreaming path can write phase reports while Dream Diary narrative append remains best-effort detached work that may log and return without adding to `DREAMS.md`. The related #72546 report is open and adjacent, but this issue adds a distinct partial-artifact plus recovery-state symptom, so it should not be closed as a duplicate or implemented yet.

## What This Changes

Fix memory-core dreaming so phase reports and `DREAMS.md` stay in sync after scheduled sweeps, and make `rem-backfill`/rollback keep the Dream Diary structurally consistent with normal diary entries.

## Best Possible Solution

Define the intended Dream Diary invariant first: whether historical backfills should remain source-day-only or be normalized to the same timestamped style as live diary entries. Then add focused memory-core coverage for a scheduled sweep where phase reports are written but narrative append fails or times out, plus a repair/backfill path that preserves one managed diary section and cleanly removes only marked backfill entries.

## Reproduction Assessment

Partial. The date-only backfill heading is reproducible from current source and tests, but the automatic multi-day diary-stall path still needs logs or a controlled cron/narrative failure fixture to reproduce with high confidence on current main.

## Solution Assessment

Unclear. A direct automated fix would be premature because current tests encode one reported recovery symptom as expected behavior; the safer path is maintainer review of the diary/backfill format contract, then a narrow implementation with regression tests for diary append failure and rollback structure.
