## Summary

Keep open. Current `main` still has a plausible split-brain path where phase reports/events are written synchronously while Dream Diary prose is detached best-effort work, and the reported rollback/header corruption maps directly to current `DREAMS.md` marker-normalization code.

## What This Changes

The issue reports that memory-core dreaming phase reports keep advancing while `DREAMS.md` stops receiving Dream Diary entries, and that `memory rem-backfill` or rollback can leave diary formatting and managed-block structure inconsistent.

## Best Possible Solution

Keep this open and reconcile it with #72546 and open PR #70332. The desired end state is an idempotent Dream Diary migration/backfill/rollback path for legacy or partially marked `DREAMS.md`, plus clearer cron narrative failure handling so successful phase reports cannot silently imply a healthy diary when detached prose generation failed.

## Reproduction Assessment

Partly yes. The malformed rollback/header symptom has a high-confidence static reproduction from current `ensureDiarySection()` plus `removeBackfillDiaryEntries()` behavior on an unmarked non-empty diary; the multi-day stale diary symptom still needs live cron/subagent logs to prove the exact failure cause.

## Solution Assessment

No proposed patch is attached to this issue, and current `main` is not sufficient. A narrow maintainable fix should make diary marker insertion legacy-aware and add focused tests before changing the broader detached cron narrative contract.
