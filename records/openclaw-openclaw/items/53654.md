## Summary

Keep this open. Current main still has no Discord MESSAGE_UPDATE or MESSAGE_DELETE inbound path: the listener abstractions, gateway mapping, monitor startup registration, and run queue all remain create-message oriented, while the related Discord edit-only trackers remain open and do not cover delete-to-cancel.

## What This Changes

The issue requests Discord messageUpdate handling to reprocess edited user messages and messageDelete handling to cancel the in-progress run started by the deleted source message.

## Best Possible Solution

Implement this in the bundled Discord plugin: add typed MESSAGE_UPDATE and MESSAGE_DELETE listener support, register both during monitor startup, route eligible edits through the existing authorization/preflight/session path with edit-aware replay semantics, and add precise source-message-id cancellation for active runs. Cover the edit and delete paths with focused Discord tests before closing this tracker or its edit-only siblings.
