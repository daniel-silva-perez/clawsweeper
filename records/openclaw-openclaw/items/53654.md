## Summary

Keep this open. Current main still routes Discord inbound work through create-message handling only, with no typed or registered MESSAGE_UPDATE or MESSAGE_DELETE listener path, and the open related Discord items cover only the edit half rather than superseding this broader edit-and-delete request.

## What This Changes

Issue requests Discord MESSAGE_UPDATE handling for edit-to-reprocess and MESSAGE_DELETE handling for delete-to-cancel in the bundled Discord plugin.

## Best Possible Solution

Implement this inside the bundled Discord plugin by adding typed MESSAGE_UPDATE and MESSAGE_DELETE listener abstractions, registering them during monitor startup, routing eligible edits through the existing preflight/mention/session path with edit-aware replay behavior, and adding source-message-id scoped cancellation for active runs. Keep the edit-only trackers coordinated, but close this only after both edit-to-reprocess and delete-to-cancel are covered by tests and shipped.
