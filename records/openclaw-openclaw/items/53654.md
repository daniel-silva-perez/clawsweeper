## Summary

Keep this issue open. Current main at 68561a8c948b49e85ae012646de550a49d6c63a3 still lacks Discord messageUpdate/messageDelete listener registration, edit-aware replay semantics, and a source Discord message id to active-run cancel mapping. Related #38505 remains open and covers only the edit half; the open #73104 work is Telegram-only and does not supersede this Discord request.

## Best Possible Solution

Implement this in the bundled Discord plugin. Add Carbon messageUpdate and messageDelete listeners in the Discord monitor listener layer, register them during Discord startup, route eligible edited messages through the existing mention/preflight/worker path with explicit edit-aware replay or supersede semantics, and maintain a Discord source-message-id to active job/run abort map so delete events cancel only the matching in-flight work. Coordinate the edit half with #38505, but keep this broader tracker open until both edit-to-reprocess and delete-to-cancel are shipped.

## Risks / Open Questions

- Carbon messageUpdate payload handling likely needs hydration/content-change gating; treating it exactly like messageCreate could process partial or non-content edits.
- Delete-to-cancel needs a precise source-message-id to active-run abort mapping. Canceling by channel or session alone could stop newer unrelated work.
- The existing replay guard would drop edited messages with the same Discord message id unless the implementation adds explicit edit-aware dedupe or supersede semantics.
