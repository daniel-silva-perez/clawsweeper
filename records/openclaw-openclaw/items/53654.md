## Summary

Keep this open. Current main still lacks Discord MESSAGE_UPDATE and MESSAGE_DELETE inbound handling: the internal Discord listener, dispatch, monitor registration, replay guard, and run queue are still create-message oriented, and the open related Discord trackers only cover the edit half.

## What This Changes

The issue requests Discord messageUpdate handling to reprocess edited user messages and messageDelete handling to cancel the active run started by the deleted source message.

## Best Possible Solution

Implement the behavior inside the bundled Discord plugin: add typed update/delete gateway event support in the current internal Discord layer, register both monitor listeners, run qualifying edits through the existing authorization/preflight/session path with explicit replay semantics, and abort only the active run tied to the deleted source message.
