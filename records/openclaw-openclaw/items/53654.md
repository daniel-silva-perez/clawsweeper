## Summary

Keep this issue open. Current main still has no Discord messageUpdate or messageDelete listener path; inbound Discord work is only registered through create-message, interaction, reaction, thread-update, and optional presence listeners. The related Discord trackers cover only edited-message reprocessing and remain open, so they do not supersede this broader edit-and-delete request.

## What This Changes

Request Discord messageUpdate handling to reprocess edited user messages and messageDelete handling to cancel the in-progress run started by the deleted source message.

## Best Possible Solution

Implement this in the bundled Discord plugin by adding typed MESSAGE_UPDATE and MESSAGE_DELETE listener support, registering those listeners during monitor startup, hydrating and filtering eligible edits through the existing mention/preflight path, and adding exact source-message-id cancellation for active runs. Cover both edit-to-reprocess and delete-to-cancel with focused Discord tests before closing the tracker and its edit-only duplicates.
