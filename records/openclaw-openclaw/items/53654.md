## Summary

Keep this open. Current main still has no Discord messageUpdate or messageDelete listener path: inbound Discord work is wired through message-create, interaction, reaction, thread-update, and optional presence listeners only. The existing replay key is account/channel/message-id based, so a same-id edit would be dropped without explicit edit-aware handling, and the current run queue has lifecycle abort but no source Discord message-id index for delete-to-cancel. Related #20173 and #38505 cover only the edit half and remain open, so they do not supersede this broader edit-and-delete request.

## What This Changes

Requested Discord plugin support for reprocessing edited user messages and cancelling active runs when the source Discord message is deleted.

## Best Possible Solution

Implement the behavior in the bundled Discord plugin: add current-gateway MessageUpdate and MessageDelete listener abstractions, register them during Discord monitor startup, route eligible edits through the existing preflight/mention/inbound path with edit-aware replay semantics, and add a precise source Discord message-id to active run cancellation map for delete-to-cancel. Coordinate with the open edit-only trackers, but keep this broader tracker open until both edit-to-reprocess and delete-to-cancel are covered by tests and shipped.

- Discord messageUpdate payloads may be partial, so implementation likely needs hydration and content-change gating before replaying an edited command.
