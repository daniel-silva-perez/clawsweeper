## Summary

Keep open. Current main only partially overlaps through the generic Agents > Files editor for bootstrap files and root MEMORY.md; it still has no dedicated Control UI Memory tab or /memory route, and the existing Gateway file RPC does not expose daily memory notes under memory/YYYY-MM-DD.md. Related PR #73773 is useful implementation context, but it is open and unmerged rather than shipped behavior.

## What This Changes

The issue requests a dedicated Control UI memory panel that can display and edit MEMORY.md, USER.md, and today's memory/YYYY-MM-DD.md daily note.

## Best Possible Solution

Add a scoped Gateway memory-files API and a Control UI Memory surface for USER.md, root MEMORY.md, and today's memory/YYYY-MM-DD.md, with path/date validation, UI coverage, Gateway safety tests, docs, and a user-facing changelog entry.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The missing surface is high-confidence from current-main source inspection: there is no /memory route or Memory panel, and the existing files RPC only allows bootstrap files plus root MEMORY.md.

## Solution Assessment

No, the issue's claim that no backend work is required is not the safest implementation path. Current main rejects daily memory paths through the existing agent files API, so the maintainable solution is a narrow scoped Gateway API plus the UI view rather than a UI-only change.
