## Summary

Keep open. Current main has a partial Control UI file editor under Agents -> Files that can read/write allowlisted root workspace files including USER.md and MEMORY.md, but it is not a dedicated Memory panel and it does not surface or edit memory/YYYY-MM-DD.md daily logs. The central request remains unmet despite related agent-files work.

## Best Possible Solution

Keep this issue open and implement a Control UI memory surface that intentionally covers USER.md, MEMORY.md, and the relevant memory/YYYY-MM-DD.md daily log files. The likely path is to extend the existing agent-files backend/UI or add a memory-specific Gateway RPC with a tight date-file allowlist, boundary-safe reads/writes, admin scope for saves, and docs updates for the Control UI memory workflow.

## Risks / Open Questions

- The existing Agents -> Files editor partially overlaps the request for USER.md and MEMORY.md, so a future implementation should avoid duplicating editors unnecessarily and decide whether to extend that surface or add a Memory-focused view.
- Any daily-log write support should preserve the current boundary-safe, allowlisted file-access model; widening it to memory/*.md needs explicit validation for date-shaped paths and workspace containment.
