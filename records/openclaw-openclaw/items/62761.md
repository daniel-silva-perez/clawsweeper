## Summary

Keep open. Current main addresses the reporter's later `is_from_me` echo-loop subcase for normal DMs, but the durable restart replay cursor is still missing: the legacy iMessage monitor still subscribes without `since_rowid` or `start`, targeted search found no per-account cursor state, and related reports #72322 and #61445 were already consolidated into this issue as the canonical tracker.

## What This Changes

Request to persist a legacy iMessage per-account rowid cursor and resume `imsg` watching after update/restart so old chat history is not replayed as new inbound messages.

## Best Possible Solution

Add an iMessage-owned per-account replay cursor in the legacy plugin: load it before subscribing, pass the dependency-supported resume field to `watch.subscribe`, fence the first run against existing backlog, and persist the latest safe rowid after handled or intentionally skipped messages. Keep the existing self-chat guards and add restart/replay regression coverage.

## Reproduction Assessment

Yes. The issue has live logs and a source-level reproduction path: start the legacy iMessage monitor with prior Messages history, observe that OpenClaw sends no resume cursor to `watch.subscribe`, and upstream `imsg` therefore has no OpenClaw-provided rowid fence. A focused unit test can assert the current request params lack `since_rowid`/`start`.

## Solution Assessment

Yes for the remaining problem: a per-account cursor/resume fence in `extensions/imessage` is the narrowest maintainable fix. The already-landed `is_from_me` filtering is useful but does not prevent old inbound rows from replaying after restart.

## Risks / Open Questions

- Cursor advancement must happen after a message is safely handled or intentionally skipped; advancing before failed dispatch could permanently skip real inbound messages.
- First-run behavior needs a conservative fence, such as startup `start` time or a high-water rowid, so a new cursor does not replay the whole existing Messages database or drop live messages during startup.
- The current self-chat exceptions are intentional; a broad `is_from_me` drop would regress real self-chat behavior covered by existing tests.
