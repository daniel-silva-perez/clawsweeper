## Summary

Keep this issue open. Current main has the later `is_from_me` outbound-row mitigation and regression coverage, but the original legacy iMessage restart/backlog replay problem remains: `watch.subscribe` still starts without a persisted rowid/start cursor, and targeted source search found no per-account cursor persistence path. The closed related #72322 was explicitly treated as a duplicate of this canonical cursor-persistence tracker, not as proof that the cursor issue is fixed.

## Best Possible Solution

Keep this issue open and narrow it to the remaining legacy iMessage cursor/backlog replay behavior. The likely implementation belongs in `extensions/imessage/src/monitor/monitor-provider.ts`: persist a per-account last processed rowid or startup high-water marker, pass the resume point to `watch.subscribe` when supported, update the marker only after safe handling or intentional drops, and add restart/iCloud-sync replay regression coverage plus docs/config notes if new state behavior is introduced.
