## Summary

Keep open: current main and the latest release still have the reported unbounded memory-wiki compile read burst, and no paired open PR was found. The fix is narrow, plugin-owned, and suitable for a focused ClawSweeper repair PR.

## What This Changes

The issue requests bounded filesystem read concurrency for memory-wiki compile/ingest so iCloud/FileProvider-backed vaults do not hit EDEADLK, hangs, or partial zero-length reads under bursty `fs.readFile` load.

## Best Possible Solution

Add a bounded worker limit inside the memory-wiki page read path, preferably with the existing SDK concurrency helper and a conservative default such as 8, while preserving deterministic page ordering and existing compile output. Add a focused regression test proving peak concurrent reads stay under the cap.

## Reproduction Assessment

Yes for the OpenClaw trigger path: a 150+ page memory-wiki vault followed by `openclaw wiki compile` reaches `readPageSummaries`, which currently fans out every page read at once. The macOS FileProvider EDEADLK itself was not live-reproduced in this Linux review environment.

## Solution Assessment

Yes, a bounded read worker in `extensions/memory-wiki/src/compile.ts` is the narrowest maintainable first fix because it addresses the pathological burst without changing core loader behavior or adding a dependency. A config knob can be considered after the fixed conservative cap is covered by tests.
