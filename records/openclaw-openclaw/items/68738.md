## Summary

Keep open: current main still contains the reported unbounded `fs.readFile` fan-out in the bundled `memory-wiki` compile path, and the affected files/tests are narrow enough for a focused fix PR.

## What This Changes

The issue requests bounding memory-wiki page file-read concurrency during wiki compile/ingest flows to avoid iCloud/FileProvider-backed vault deadlocks and partial reads.

## Best Possible Solution

Add a bounded page-read helper inside the bundled `memory-wiki` plugin, use a conservative default cap for compile page summaries, preserve ordering/filtering semantics, and cover it with a focused regression test that observes max concurrent page reads.

## Reproduction Assessment

Yes for the OpenClaw trigger: current main has a direct code-level reproduction path by creating many wiki page files and observing `readPageSummaries` start all `fs.readFile` calls through one `Promise.all`. A live iCloud/FileProvider mount was not run, but it is not needed to verify the unbounded fan-out.

## Solution Assessment

Yes: bounding the memory-wiki page-read fan-out with the existing SDK concurrency helper is the narrowest maintainable fix. A configurable cap can be considered later, but a small internal default addresses the reported failure without adding config surface or dependencies.
