## Summary

Keep this PR open. Current main has the related fresh-session rotation fix, but it still does not archive the prior isolated heartbeat/cron transcript on rotation. The PR addresses a real remaining retention gap, but the proposed helper is not merge-ready because its full-store reference check can make the reset archive path a no-op when historical cron run rows still reference the prior session id.

## What This Changes

The PR adds archive-on-rotation helpers and heartbeat/cron wiring so prior isolated-session transcripts are renamed to `.reset` archives after forced session rotation, with unit and sandbox tests for capture timing, once-only archival, and heartbeat filesystem behavior.

## Best Possible Solution

Land a narrow archive-on-rotation implementation adapted to current main: capture the prior entry before overwriting the stable heartbeat/cron session key, persist the new fresh entry, archive the prior transcript with `reason: "reset"` and `restrictToStoreDir: true`, and build the reset reference set from active session rows so historical cron run rows cannot block the rename.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Ignore historical cron run rows during reset archival:** `src/cron/isolated-agent/session.ts:144`
  - body: `referencedSessionIds` is built from every store row, but retained cron `...:run:<id>` rows can still point at the prior `sessionId`. When that happens, `archiveRemovedSessionTranscripts` skips the just-rotated file, so the new reset archive path is a no-op for those runs. Build this reference set from active rows only, or explicitly ignore historical run rows for rotation archival.
  - confidence: 0.92
