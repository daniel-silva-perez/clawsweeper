## Summary

Keep this issue open. The existing ClawSweeper comment is still source-backed: current `main` exposes the supported `memory-wiki` shared search path and still calls `sharedMemoryManager.search(...)` without a structural guard or local fallback, so a malformed or incompatible active memory plugin can abort `wiki_search` after wiki results are already available.

## What This Changes

The issue reports that `wiki_search` can throw when `memory-wiki` shared search resolves a truthy memory manager without a callable `search` method for `corpus="memory"` or `corpus="all"`.

## Best Possible Solution

Add a narrow defensive fallback in `extensions/memory-wiki/src/query.ts`: validate that the resolved active memory manager has a callable `search`, catch shared-memory search failures, and preserve already-computed wiki results for `corpus="all"` while returning no memory results when shared memory is unavailable. Cover malformed-manager and throwing-search cases in `extensions/memory-wiki/src/query.test.ts`.

## Reproduction Assessment

Yes. A deterministic unit reproduction is to configure `memory-wiki` with `search.backend="shared"` and `corpus="all"`, mock `getActiveMemorySearchManager` to resolve a truthy manager without `.search`, and call `searchMemoryWiki`; current main reaches the unguarded call and would throw the reported TypeError shape.

## Solution Assessment

Yes. The narrowest maintainable fix is at the `memory-wiki` consumer boundary because both built-in and third-party active memory plugins flow through the SDK facade, and wiki-only results should not disappear when shared memory is malformed or temporarily unavailable.
