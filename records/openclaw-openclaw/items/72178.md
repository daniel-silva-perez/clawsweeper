## Summary

Keep open. Current main still lacks cursor-based `chat.history` pagination, so the PR is not obsolete, but the latest PR diff has a blocking server-side pagination bug: it computes a `before` window without applying that window before projecting the transcript, so older-page requests can return the latest page again.

## What This Changes

The PR adds a `before` cursor to `chat.history`, cursor/hasMore handling in the web UI, load-older rendering and scroll compensation, partial `NO_REPLY` stream suppression, protocol docs, and focused gateway/UI tests.

## Best Possible Solution

Keep the PR open and repair the gateway contract at the handler boundary: page the raw `[start, end)` transcript window before projection or carry raw indices through projection/sanitization/capping, then update generated protocol artifacts, docs, changelog, and handler-level tests before revisiting the UI portion for merge.

## Reproduction Assessment

Yes. A focused branch test with more than 200 raw messages can request the latest page, then request `chat.history` with `before` equal to the returned cursor; the current PR diff would still project the full raw transcript and can return the latest page again.

## Solution Assessment

No. The requested direction is maintainable, but the current proposed implementation is not the best fix because it computes cursor bounds without using them for server-side message selection and leaves generated protocol artifacts behind.
