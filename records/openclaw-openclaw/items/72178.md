## Summary

Keep open. Current main still does not implement cursor-based chat.history pagination or the web UI load-older flow, so the PR is not obsolete. The proposed branch is still not merge-ready because the gateway handler computes a cursor window but, per the provided latest PR diff, continues projecting the full raw history, so before-page requests can return the latest page again.

## What This Changes

The PR adds a before cursor to chat.history, cursor/hasMore response metadata, web UI load-older state and rendering, scroll compensation, protocol docs, and tests for history budgeting plus partial NO_REPLY stream handling.

## Best Possible Solution

Keep the PR open and fix the gateway contract at the handler boundary: make before select the raw [start, end) page before projection, or carry raw indices through projection/sanitization so cursor/hasMore describe the first returned raw message. Then add handler-level tests for normal pagination, sanitized NO_REPLY/commentary drops, and byte-budget leading trims before considering the UI/docs portion for merge.

## Reproduction Assessment

Yes. A focused branch test with more than limit raw messages and a request like before: 50, limit: 20 should show whether the handler returns messages 30-49; the provided PR diff indicates it would still project the latest messages from the full rawMessages array.

## Solution Assessment

No. The requested direction is reasonable, but the current proposed implementation is not the best fix because it computes cursor bounds without applying them to the server-side message selection; the safer solution is to page the raw transcript or preserve raw indices before any display projection and sanitization.
