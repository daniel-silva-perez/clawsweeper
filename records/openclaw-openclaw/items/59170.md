## Summary

Keep open. The feature request is still valid and current main does not yet expose per-agent `agents.list[].tools.sessions` or `agents.list[].tools.agentToAgent`; the PR also remains useful because it is paired with the open requester issue and has later user demand. It should not merge as-is: GitHub reports the branch dirty against current main, current main moved the session visibility implementation into the plugin SDK, and the patch only wires list/send while history/status remain on the global policy path.

## What This Changes

The PR adds per-agent session visibility and agent-to-agent override config under `agents.list[].tools`, updates session access policy/list/send wiring and tests, and also adds an unrelated session path regression test.

## Best Possible Solution

Land a rebased, narrow implementation on the current `src/plugin-sdk/session-visibility.ts` seam that keeps default global behavior, exposes the config schema/types/help/docs consistently, threads the requester agent tools through list/history/send/status, removes unrelated files, and covers per-agent visibility plus target-only A2A allow semantics in focused tests.

Yes. Source inspection on current main reproduces the gap: the schema has no per-agent `tools.sessions`/`tools.agentToAgent` keys, and the session visibility/A2A helpers plus current tool callers read only global config.

## Solution Assessment

No, this PR is not the best mergeable form as written. The direction is plausible, but the implementation needs to be rebased onto the SDK helper, completed across all session surfaces, and cleaned of unrelated test changes before maintainer review can decide the access-control semantics.
