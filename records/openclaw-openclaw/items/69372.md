## Summary

Keep open. Current main has not absorbed this PR's fail-closed canvas URL behavior or its client-side capability refresh. The PR is also not ready as-is: its timer-driven WebSocket close can interrupt active Control UI runs, and current main already has a non-teardown `node.canvas.capability.refresh` RPC that looks like the safer direction for rework.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The likely best direction is to preserve the fail-closed protected URL hardening, but replace the forced WebSocket reconnect with an idle-safe or RPC-based capability refresh using the existing `node.canvas.capability.refresh` seam, plus tests proving no raw protected iframe URL is emitted and active chat runs are not reset by refresh.
