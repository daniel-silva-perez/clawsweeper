## Summary

Keep open. Current main still has the APNs managed-proxy bypass this PR is trying to fix, and the PR is member-authored with protected `security` and `maintainer` labels, so it needs maintainer handling. The current head is also not merge-ready because it removes the `http2` binding still used in the APNs timeout path and the CONNECT timeout emits an unhandled request error instead of rejecting the tunnel attempt.

## What This Changes

The PR routes direct APNs HTTP/2 sends through the active managed proxy via a CONNECT tunnel, adds active proxy URL state, tests, docs, and raw-HTTP/2 lint/OpenGrep guardrails.

## Best Possible Solution

Keep the APNs-specific HTTP/2 wrapper direction, restore a valid `http2` constants binding or import the constant directly, replace the CONNECT timeout with a promise rejection and cleanup path, add coverage for stalled CONNECT handshakes, then have maintainers complete the security-sensitive review.

## Reproduction Assessment

Yes. The current-main gap is reproducible by source inspection: direct APNs sends still call `http2.connect(authority)` while the managed-proxy docs identify raw HTTP/2 sockets as bypass-prone. The PR-head defects are also reproducible by static inspection at the cited lines.

## Solution Assessment

No, not as currently patched. The wrapper and active managed-proxy state are the narrow maintainable direction, but the missing binding and timeout rejection bug must be fixed before this is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Restore the http2 constants binding:** `src/infra/push-apns.ts:1-3`
  - body: This module still calls `http2.constants.NGHTTP2_CANCEL` in the request timeout handler, but the branch removes the `node:http2` import. Typecheck should fail and the timeout path would throw, so keep the import or import the constant directly.
  - confidence: 0.97
- **[P2] Reject tunnel timeouts without emitting request errors:** `src/infra/net/http-connect-tunnel.ts:44-48`
  - body: The timeout callback emits `error` on a bare EventEmitter-backed request shim that has no listener, and `https-proxy-agent.connect()` does not consume that event to reject the promise. A stalled CONNECT can therefore throw from the timer instead of rejecting and cleaning up; race the connect attempt against a real timeout and cleanup path.
  - confidence: 0.88
