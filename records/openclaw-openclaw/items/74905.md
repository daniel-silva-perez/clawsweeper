## Summary

Keep open. This is a draft PR with the protected `maintainer` label, and current `main` still sends direct APNs through `http2.connect(authority)` without a proxy-aware HTTP/2 seam. The proposed direction is relevant, but the branch should address the `NO_PROXY` correctness finding before merge.

## What This Changes

The PR adds a proxy-aware APNs HTTP/2 connection helper and HTTP CONNECT tunnel helper, wires direct APNs sends through it, and adds regression tests for tunnel/proxy behavior.

## Best Possible Solution

Keep the PR open for maintainer review, preserve the narrow APNs HTTP/2 seam, and update the proxy decision path so standard `NO_PROXY` exclusions are honored before opening a CONNECT tunnel.

## Reproduction Assessment

Yes. With the PR branch, `HTTPS_PROXY=http://proxy.test:8080` plus `NO_PROXY=api.push.apple.com` still reaches the tunnel path because `src/infra/push-apns-http2.ts:37` reads the proxy URL directly instead of checking `shouldUseEnvHttpProxyForUrl`.

## Solution Assessment

No. The low-level APNs session seam is a good boundary, but the implementation should use the shared proxy decision semantics before tunneling and add an APNs regression for matching `NO_PROXY`.
