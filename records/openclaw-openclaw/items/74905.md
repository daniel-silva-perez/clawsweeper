## Summary

Keep open. Current main still has the raw APNs `http2.connect()` behavior, so the PR is relevant, but the latest head is a MEMBER-authored, security-labeled PR and the new custom CONNECT tunnel has concrete robustness issues that should be handled before merge.

## What This Changes

The branch replaces direct APNs `http2.connect()` with a proxy-aware APNs HTTP/2 wrapper, adds active managed proxy state and CONNECT tunnel helpers, wires lint/OpenGrep guardrails, and updates APNs/proxy tests, docs, and changelog text.

## Best Possible Solution

Keep the APNs-specific proxy wrapper direction, but harden the CONNECT helper with bounded response parsing and safe error handling, align the implementation or PR text around `https-proxy-agent`, then require maintainer security review and the APNs/proxy targeted tests plus changed gate before merge.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the remaining behavior: direct APNs delivery still calls raw `http2.connect(authority)` while the network proxy docs call out raw `http2` sockets as bypass-prone; the new tunnel defects can be exercised with malformed percent-encoded proxy credentials or a CONNECT response that never terminates headers.

## Solution Assessment

No, not as-is. The PR direction is narrow and appropriate, but the latest custom tunnel implementation needs bounded buffering and synchronous error handling before it is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Route CONNECT write errors through fail():** `src/infra/net/http-connect-tunnel.ts:116`
  - body: `writeConnectRequest()` can throw from `decodeURIComponent()` when the configured proxy URL contains malformed percent-encoded credentials. Because `onConnected()` calls it directly from the socket event handler, that exception bypasses `fail()`, skips cleanup, and can crash the process instead of rejecting the APNs send.
  - confidence: 0.91
- **[P2] Cap CONNECT response headers before buffering:** `src/infra/net/http-connect-tunnel.ts:119-123`
  - body: `onData()` appends every chunk to `responseBuffer` until it sees `\r\n\r\n`, with no maximum size. A bad or compromised proxy can keep sending unterminated header bytes and drive memory growth until timeout or OOM; enforce a bounded CONNECT response header size and fail the tunnel when exceeded.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No workflow, lockfile, dependency-source, or release-script supply-chain regression was found, but the new custom CONNECT tunnel has security-sensitive DoS and error-handling concerns.

Concerns:

- **[medium] Unbounded CONNECT response buffering:** `src/infra/net/http-connect-tunnel.ts:121`
  - body: The new tunnel accumulates proxy response bytes without a maximum header size, so a bad proxy response can consume memory before timeout; this matters because the PR touches managed egress security code.
  - confidence: 0.88
- **[low] Malformed proxy credentials can escape cleanup:** `src/infra/net/http-connect-tunnel.ts:116`
  - body: Malformed percent-encoded proxy credentials can throw during CONNECT request construction from an event handler, bypassing the Promise rejection path and socket cleanup.
  - confidence: 0.91
