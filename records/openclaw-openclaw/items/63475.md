## Summary

Keep this issue open. Current main still has the reported per-account Feishu startup path: the gateway starts account IDs concurrently with `Promise.all`, Feishu `startAccount` calls `monitorFeishuProvider` with a specific `accountId`, and that path calls `monitorSingleAccount`, which fetches bot identity directly without a shared startup probe queue. The sequential preflight exists only on the aggregate `monitorFeishuProvider` path, and the current regression tests cover that aggregate path rather than the gateway's concurrent per-account path. The provided GitHub context also shows open PR #63481 and fork PR #2190 use closing syntax for this issue, so the issue should stay open until a linked PR is reviewed and merged or closed.

## Best Possible Solution

Keep this issue open and review the linked implementation PR #63481 or its fork #2190. The likely fix boundary is Feishu-owned startup identity probing: serialize or otherwise throttle `fetchBotIdentityForMonitor` across concurrent gateway `startAccount` calls, add regression coverage for the actual gateway per-account startup path, and avoid changing generic gateway concurrency unless a broader channel contract decision is made.
