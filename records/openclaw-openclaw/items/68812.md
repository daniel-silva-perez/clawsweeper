## Summary

Keep open. Current main still leaves the retry max-attempt, base-delay, and max-delay values hardcoded in memory-core; only non-batch concurrency is configurable today, and the linked implementation PR #68822 is still open and unmerged.

## What This Changes

Request configurable memory embedding indexing concurrency and retry backoff parameters in `openclaw.json` for rate-limited embedding providers.

## Best Possible Solution

Keep this open while #68822 or a successor is reviewed. The end state should add validated retry max-attempt/base-delay/max-delay settings to the existing memory search config contract, wire them into the memory-core retry loop and delay clamp, preserve current `remote.nonBatchConcurrency` behavior, and update types, schema/help, docs, and focused tests before merge.

## Reproduction Assessment

Yes. Source inspection against current main is enough to reproduce the missing configuration path: retry knobs are constants in memory-core and are absent from schema/docs, so a live Gemini 429 is not required to verify the issue.

## Solution Assessment

Unclear as proposed. Configurable retry behavior is the right direction, but the maintainable fix should align with the existing `agents.defaults.memorySearch` contract rather than adding an unrelated top-level `memory.embedding` section.
