## Summary

Keep this issue open. Current main still has the reported gap for Feishu-visible recovery after an interrupted in-flight turn: managed restart paths are hardened, but direct supervisor SIGTERM is still a stop unless OpenClaw restart intent exists, restart sentinel delivery is single-session scoped, and main-session restart recovery resumes internally without same-channel delivery.

## What This Changes

The issue asks OpenClaw to notify Feishu users, or otherwise recover visibly, when a gateway restart interrupts an in-flight LLM request.

## Best Possible Solution

Implement or explicitly scope a generic channel-visible restart interruption recovery path: on startup, identify safely interrupted channel-originated sessions with route metadata, enqueue one idempotent same-channel notice or resumable delivered continuation, and cover Feishu as part of the generic contract rather than a Feishu-only special case.

## Reproduction Assessment

Yes. The issue provides concrete Feishu WebSocket and systemd restart steps, and current-main source inspection confirms the relevant stop/restart and recovery paths do not deliver a Feishu-visible notice for all interrupted sessions.

## Solution Assessment

Unclear. Same-channel user notification is the right user outcome, but the maintainable fix should be channel-agnostic and must settle direct supervisor restart semantics; a Feishu-only hook or unbounded startup scan would be riskier.
